`qa-channel` — це локальний для репозиторію синтетичний транспорт повідомлень для автоматизованого контролю якості OpenClaw (`extensions/qa-channel`, приватний пакет, виключений із пакетованих інсталяцій). Це не робочий канал — він існує для перевірки тієї самої межі Plugin каналу, яку використовують реальні транспорти, водночас забезпечуючи детермінований і повністю доступний для перевірки стан.

## Що він робить

- Граматика цілей класу Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- Спільні розмови `channel:` і `group:` подаються агентам як ходи в групових/канальних кімнатах, тому вони перевіряють ту саму політику маршрутизації видимих відповідей та інструментів повідомлень, яку використовують Discord, Slack, Telegram і подібні транспорти.
- Синтетична шина на основі HTTP для впровадження вхідних повідомлень, захоплення журналу вихідних повідомлень, створення гілок, реакцій, редагувань, видалень, а також дій пошуку й читання.
- Засіб самоперевірки на боці хоста, який записує звіт Markdown до `.artifacts/qa-e2e/`.

## Конфігурація

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

Ключі облікового запису:

- `enabled` — головний перемикач цього облікового запису.
- `name` — необов’язкова відображувана мітка.
- `baseUrl` — URL синтетичної шини. Обліковий запис вважається налаштованим після встановлення цього значення.
- `botUserId` — синтетичний ідентифікатор користувача-бота, що використовується в граматиці цілей (типово: `openclaw`).
- `botDisplayName` — відображуване ім’я для вихідних повідомлень (типово: `OpenClaw QA`).
- `pollTimeoutMs` — вікно очікування тривалого опитування. Ціле число від 100 до 30000 (типово: 1000).
- `allowFrom` — список дозволених відправників (ідентифікатори користувачів або `"*"`; типово: `["*"]`). Для особистих повідомлень
  завжди застосовується політика `open`; групова політика зі списком дозволених також використовує ці синтетичні
  ідентифікатори відправників.
- `groupPolicy` — політика спільної кімнати: `"open"` (типово), `"allowlist"` або
  `"disabled"`.
- `groupAllowFrom` — необов’язковий список дозволених відправників спільної кімнати. Якщо його не вказано для
  `"allowlist"`, канал QA використовує `allowFrom`.
- `groups.<room>.requireMention` — вимагати згадки бота перед відповіддю в
  певній груповій/канальній кімнаті (типово: false). `groups."*"` задає типове значення;
  `tools` / `toolsBySender` для окремих кімнат задають перевизначення політики інструментів.
- `defaultTo` — резервна ціль, якщо жодної не вказано.
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` — керування доступністю інструментів для окремих дій.

Ключі для кількох облікових записів на верхньому рівні:

- `accounts` — запис іменованих перевизначень для окремих облікових записів, індексованих за ідентифікатором облікового запису.
- `defaultAccount` — бажаний ідентифікатор облікового запису, коли налаштовано кілька.

## Засоби запуску

Самоперевірка на боці хоста (записує звіт Markdown до `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

Команда проходить через `qa-lab`, запускає шину QA в репозиторії, завантажує зріз середовища виконання `qa-channel` і виконує детерміновану самоперевірку.

Повний набір сценаріїв на основі репозиторію:

```bash
pnpm openclaw qa suite
```

Паралельно виконує сценарії в смузі Gateway для QA. Сценарії, профілі та режими постачальників описано в [огляді QA](/uk/concepts/qa-e2e-automation).

Сайт QA на основі Docker (Gateway + інтерфейс налагодження QA Lab в одному стеку):

```bash
pnpm qa:lab:up
```

Збирає сайт QA, запускає стек Gateway + QA Lab на основі Docker і виводить URL QA Lab. Там можна вибирати сценарії, вибирати смугу моделі, запускати окремі виконання та спостерігати за результатами наживо. Налагоджувач QA Lab відокремлений від пакета Control UI, що постачається.
