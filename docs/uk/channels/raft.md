Raft підключає агента OpenClaw до зовнішнього агента Raft через локальний
CLI Raft. Raft надсилає автентифіковані сигнали пробудження до Gateway, після чого агент
використовує CLI Raft для перевірки та надсилання повідомлень. Підтримуються лише особисті чати (без груп).

## Встановлення

Raft — офіційний зовнішній плагін. Установіть його на хості Gateway:

```bash
openclaw plugins install @openclaw/raft
openclaw gateway restart
```

Докладніше: [Плагіни](/uk/tools/plugin)

## Передумови

- Робочий простір Raft із зовнішнім агентом.
- CLI Raft, установлений на тому самому хості, що й Gateway OpenClaw, і доступний у
  `PATH` служби.
- Профіль CLI Raft, у якому уже виконано вхід і який пов’язано з цим
  зовнішнім агентом.

Плагін не зберігає облікові дані Raft; CLI Raft зберігає дані
автентифікації у власному профілі.

## Налаштування

Укажіть профіль у конфігурації:

```json5
{
  channels: {
    raft: {
      enabled: true,
      profile: "openclaw",
    },
  },
}
```

Для облікового запису за замовчуванням натомість можна задати `RAFT_PROFILE` у середовищі
Gateway:

```bash
RAFT_PROFILE=openclaw
```

Використовуйте іменований обліковий запис, якщо один Gateway підключається до кількох зовнішніх агентів Raft:

```json5
{
  channels: {
    raft: {
      accounts: {
        support: {
          profile: "support-agent",
        },
        engineering: {
          profile: "engineering-agent",
        },
      },
    },
  },
}
```

Під час інтерактивного налаштування зберігається той самий профіль:

```bash
openclaw channels add --channel raft
```

## Принцип роботи

Під час запуску Gateway плагін:

1. Відкриває HTTP-кінцеву точку пробудження, доступну лише через local loopback, на тимчасовому порту.
2. Запускає `raft --profile <profile> agent bridge` із цією кінцевою точкою та
   окремим токеном для кожного процесу.
3. Приймає від локального мосту лише автентифіковані сигнали пробудження без вмісту, що мають
   ідентифікатор для захисту від повторного відтворення.
4. Вимагає наявності одного з `eventId`, `attemptId`, `messageId`, `delivery_id`,
   `wake_id` або `id` у кожному корисному навантаженні пробудження.
5. Протягом 24 годин усуває дублікати повторно доставлених сигналів пробудження за ідентифікатором події мосту,
   зокрема після перезапусків Gateway.
6. Повертає стабільний сеанс середовища виконання для поточного мосту та порожній
   пакет отримання активності для протоколу CLI Raft.
7. Запускає один послідовний цикл агента OpenClaw для кожного прийнятого сигналу пробудження.

Міст керує повторними спробами доставки Raft і повторними підключеннями. Цикл OpenClaw
отримує лише сповіщення про пробудження, а не копію тексту повідомлення Raft. Він використовує CLI,
щоб прочитати повідомлення, які очікують на обробку, і надіслати відповідь:

```bash
raft --profile openclaw message check
raft --profile openclaw message send
```

<Note>
Raft не є транспортом для push-повідомлень. OpenClaw не надсилає автоматично остаточний текст моделі назад через міст, тому після обробки сигналу пробудження агент повинен скористатися CLI Raft.
</Note>

## Перевірка

Перевірте, чи OpenClaw може знайти CLI та чи має налаштований профіль:

```bash
openclaw channels status --probe
openclaw plugins inspect raft --runtime --json
```

Потім надішліть повідомлення зовнішньому агенту Raft. У журналі Gateway має з’явитися
запуск мосту Raft, а потім — вхідний сигнал пробудження. Агент має використати
налаштований профіль Raft для перевірки повідомлень, які очікують на обробку.

## Усунення несправностей

<AccordionGroup>
  <Accordion title="CLI Raft відсутній">
    Установіть CLI Raft на хості Gateway і забезпечте доступність `raft` у
    `PATH` служби. Перевірте його командою `raft --help`, а потім перезапустіть Gateway.
  </Accordion>
  <Accordion title="Міст негайно завершує роботу">
    Переконайтеся, що в налаштованому профілі виконано вхід і що він належить потрібному
    зовнішньому агенту Raft. Запустіть `raft --profile <profile> agent bridge` безпосередньо,
    щоб переглянути діагностичні дані CLI.
  </Accordion>
  <Accordion title="Сигнал пробудження надходить, але відповідь Raft не надсилається">
    Це очікувана поведінка, якщо агент не викликає CLI Raft. Міст
    пробудження не передає тексти повідомлень і не надсилає автоматичних остаточних відповідей. Перевірте
    політику інструментів агента та переконайтеся, що він може виконувати `raft --profile <profile>
    message check` і `message send`.
  </Accordion>
</AccordionGroup>
