## Огляд

Групи широкомовного розсилання запускають **кількох агентів** для одного вхідного повідомлення. Кожен агент обробляє повідомлення у власному ізольованому сеансі та публікує власну відповідь, тому один номер WhatsApp може обслуговувати команду спеціалізованих агентів в одному груповому чаті або приватному повідомленні.

Групи широкомовного розсилання перевіряються після списків дозволених каналів і правил активації груп. У групах WhatsApp широкомовне розсилання відбувається, коли OpenClaw зазвичай мав би відповісти (наприклад, у разі згадки — залежно від налаштувань групи). Вони змінюють лише те, **які агенти запускаються**, але не те, чи підлягає повідомлення обробці.

Активний контур контролю якості WhatsApp містить `whatsapp-broadcast-group-fanout`, який перевіряє, що одне групове повідомлення зі згадкою може спричинити окремі видимі відповіді від двох налаштованих агентів.

## Конфігурація

### Базове налаштування

Додайте розділ верхнього рівня `broadcast` (поруч із `bindings`). Ключами є ідентифікатори співрозмовників WhatsApp, а значеннями — масиви ідентифікаторів агентів:

- групові чати: груповий JID (наприклад, `120363403215116621@g.us`)
- приватні повідомлення: номер телефону відправника у форматі E.164 (наприклад, `+15551234567`)

```json
{
  "broadcast": {
    "120363403215116621@g.us": ["alfred", "baerbel", "assistant3"]
  }
}
```

**Результат:** коли OpenClaw мав би відповісти в цьому чаті, він запускає всіх трьох агентів.

Кожен зазначений ідентифікатор агента має існувати в `agents.list`: перевірка конфігурації повідомляє про невідомі ідентифікатори, а середовище виконання пропускає їх із попередженням `Broadcast agent <id> not found in agents.list; skipping`.

### Стратегія обробки

`broadcast.strategy` визначає, як агенти обробляють повідомлення:

| Стратегія             | Поведінка                                                              |
| -------------------- | --------------------------------------------------------------------- |
| `parallel` (типово) | Усі агенти обробляють одночасно; відповіді надходять у довільному порядку.       |
| `sequential`         | Агенти обробляють у порядку масиву; кожен чекає на завершення попереднього. |

```json
{
  "broadcast": {
    "strategy": "sequential",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

## Як це працює

### Потік повідомлення

<Steps>
  <Step title="Надходить вхідне повідомлення">
    Надходить повідомлення з групи WhatsApp або приватне повідомлення.
  </Step>
  <Step title="Маршрутизація та допуск">
    OpenClaw застосовує списки дозволених каналів, правила активації груп і налаштоване володіння прив’язкою ACP.
  </Step>
  <Step title="Перевірка широкомовного розсилання">
    Якщо маршрутом не володіє жодна налаштована прив’язка ACP, OpenClaw перевіряє, чи є ідентифікатор співрозмовника в `broadcast`.
  </Step>
  <Step title="Якщо застосовується широкомовне розсилання">
    - Усі зазначені агенти обробляють повідомлення.
    - Кожен агент має власний ключ сеансу та ізольований контекст.
    - Агенти обробляють повідомлення паралельно (типово) або послідовно.
    - Аудіовкладення транскрибуються один раз перед розподіленням, тому агенти використовують спільну транскрипцію замість окремих викликів STT.

  </Step>
  <Step title="Якщо широкомовне розсилання не застосовується">
    OpenClaw спрямовує повідомлення за звичайним маршрутом або за налаштованим маршрутом сеансу ACP, вибраним під час маршрутизації.
  </Step>
</Steps>

<Note>
Групи широкомовного розсилання не обходять списки дозволених каналів або правила активації груп (згадки, команди тощо). Вони змінюють лише те, _які агенти запускаються_, коли повідомлення підлягає обробці.
</Note>

### Ізоляція сеансів

Кожен агент у групі широкомовного розсилання має повністю окремі:

- **Ключі сеансів** (`agent:alfred:whatsapp:group:120363...` і `agent:baerbel:whatsapp:group:120363...`)
- **Історію розмови** (агент не бачить відповідей інших агентів)
- **Робочий простір** (окремі пісочниці, якщо налаштовано)
- **Доступ до інструментів** (різні списки дозволів і заборон)
- **Пам’ять/контекст** (окремі `IDENTITY.md`, `SOUL.md` тощо)

Є один навмисний виняток: **буфер контексту групи** (останні групові повідомлення, що використовуються для контексту) є спільним для кожного співрозмовника, тому всі агенти широкомовного розсилання бачать однаковий контекст під час запуску. Після завершення розподілення він очищається один раз.

Це дає змогу кожному агенту мати окремі особистості, моделі, Skills і доступ до інструментів (наприклад, лише для читання або для читання й запису).

## Варіанти використання

- **Команди спеціалізованих агентів**: група розробників, у якій `code-reviewer`, `security-auditor`, `test-generator` і `docs-checker` відповідають на одне повідомлення кожен зі своєї точки зору.
- **Багатомовна підтримка**: один чат підтримки з `support-en`, `support-de` і `support-es`, які відповідають відповідними мовами.
- **Контроль якості**: `support-agent` відповідає, а `qa-agent` перевіряє відповідь і реагує лише тоді, коли знаходить проблеми.
- **Автоматизація завдань**: `task-tracker`, `time-logger` і `report-generator` обробляють одне й те саме оновлення стану.

## Сумісність

### Постачальники

Наразі групи широкомовного розсилання реалізовано лише для WhatsApp (вебканалу). Інші канали ігнорують конфігурацію `broadcast`.

### Маршрутизація

Групи широкомовного розсилання працюють разом із наявною маршрутизацією:

```json
{
  "bindings": [
    {
      "match": { "channel": "whatsapp", "peer": { "kind": "group", "id": "GROUP_A" } },
      "agentId": "alfred"
    }
  ],
  "broadcast": {
    "GROUP_B": ["agent1", "agent2"]
  }
}
```

- `GROUP_A`: відповідає лише alfred (звичайна маршрутизація).
- `GROUP_B`: відповідають agent1 ТА agent2 (широкомовне розсилання).

<Note>
**Пріоритет:** `broadcast` має пріоритет над звичайними прив’язками маршрутів. Налаштовані прив’язки ACP (`bindings[].type="acp"`) є винятковими: якщо одна з них збігається, OpenClaw спрямовує повідомлення до налаштованого сеансу ACP замість широкомовного розподілення.
</Note>

## Усунення несправностей

<AccordionGroup>
  <Accordion title="Агенти не відповідають">
    **Перевірте:**

    1. Ідентифікатори агентів існують у `agents.list` (перевірка конфігурації відхиляє невідомі ідентифікатори).
    2. Формат ідентифікатора співрозмовника правильний (груповий JID на кшталт `120363403215116621@g.us` або номер у форматі E.164 на кшталт `+15551234567` для приватних повідомлень).
    3. Повідомлення пройшло звичайні перевірки допуску (правила згадок і активації досі застосовуються).

    **Налагодження:**

    ```bash
    openclaw logs --follow | grep -i broadcast
    ```

    Успішне розподілення додає до журналу запис `Broadcasting message to <n> agents (<strategy>)`.

  </Accordion>
  <Accordion title="Відповідає лише один агент">
    **Причина:** ідентифікатор співрозмовника може бути у звичайних прив’язках маршрутів, але не в `broadcast`, або він може відповідати винятковій налаштованій прив’язці ACP.

    **Виправлення:** додайте співрозмовників, прив’язаних до звичайних маршрутів, у конфігурацію широкомовного розсилання або вилучіть чи змініть налаштовану прив’язку ACP, якщо потрібне широкомовне розподілення.

  </Accordion>
  <Accordion title="Проблеми з продуктивністю">
    Якщо робота сповільнюється за великої кількості агентів, зменште кількість агентів у групі, використовуйте легші моделі та перевірте час запуску пісочниці.
  </Accordion>
</AccordionGroup>

## Довідник API

### Схема конфігурації

```typescript
interface OpenClawConfig {
  broadcast?: {
    strategy?: "parallel" | "sequential";
    [peerId: string]: string[];
  };
}
```

### Поля

<ParamField path="strategy" type='"parallel" | "sequential"' default='"parallel"'>
  Спосіб обробки агентами. `parallel` запускає всіх агентів одночасно; `sequential` запускає їх у порядку масиву.
</ParamField>
<ParamField path="[peerId]" type="string[]">
  Груповий JID WhatsApp або номер телефону у форматі E.164. Значенням є масив ідентифікаторів агентів, кожен із яких має обробляти повідомлення від цього співрозмовника.
</ParamField>

## Обмеження

1. **Максимальна кількість агентів:** жорсткого обмеження немає, але велика кількість агентів (10 і більше) може сповільнювати роботу.
2. **Спільний контекст:** агенти не бачать відповідей один одного (це передбачено архітектурою).
3. **Порядок повідомлень:** паралельні відповіді можуть надходити в довільному порядку.
4. **Обмеження частоти:** усі відповіді надходять з одного облікового запису WhatsApp, тому відповідь кожного агента враховується в межах тих самих обмежень частоти WhatsApp.
