```bash
openclaw onboard
```

Початкове налаштування через CLI — рекомендований спосіб налаштування в терміналі на macOS, Linux і
Windows (нативно або через WSL2). Типово воно виявляє вже доступний на
комп’ютері доступ до ШІ, перевіряє його реальним завершенням і запускає Crestodian для
налаштування робочого простору, Gateway та необов’язкових функцій. `openclaw setup` запускає той самий процес ([Налаштування](/uk/cli/setup) описує
варіант `--baseline`, який налаштовує лише конфігурацію). Користувачі настільної версії Windows також можуть почати
з [Windows Hub](/uk/platforms/windows).

Кероване початкове налаштування спочатку встановлює доступ до моделі логічного виведення. Воно виявляє доступний доступ до ШІ,
вимагає реального завершення й лише після цього запускає [Crestodian](/uk/cli/crestodian)
для налаштування решти OpenClaw. У керованому процесі немає запуску Crestodian до перевірки логічного виведення або
шляху з пропуском ШІ.

Класичний майстер залишається доступним для входу до постачальника, налаштування віддаленого Gateway,
підключення каналів, керування фоновою службою, Skills та імпорту. Запустіть його явно
командою `openclaw onboard --classic`; екран кандидатів для керованого логічного виведення
не передає керування йому. Після успішної перевірки логічного виведення Crestodian може використати `open channel
wizard for <channel>`, щоб передати налаштування каналу, яке потребує секретів, майстру
в терміналі з маскуванням введення. Щоб змінити постачальника моделі або спосіб автентифікації,
вийдіть із Crestodian і запустіть `openclaw onboard`; Crestodian не відкриває керовані або
класичні процеси налаштування постачальника.

<Info>
Найшвидший спосіб розпочати перший чат: завершіть кероване налаштування, запустіть `openclaw dashboard` і спілкуйтеся
у браузері через інтерфейс керування. Документація: [Панель керування](/uk/web/dashboard).
</Info>

## Локаль

Майстер локалізує фіксований текст початкового налаштування. Порядок визначення: `OPENCLAW_LOCALE`,
`LC_ALL`, `LC_MESSAGES`, `LANG`, а потім англійська. Підтримувані локалі: `en`,
`zh-CN`, `zh-TW`.

```bash
OPENCLAW_LOCALE=zh-CN openclaw onboard
```

Назви продуктів, команди, ключі конфігурації, URL-адреси, ідентифікатори постачальників, ідентифікатори моделей і
мітки плагінів/каналів залишаються англійською незалежно від локалі.

Щоб згодом змінити налаштування, не пов’язані з логічним виведенням:

```bash
openclaw configure
openclaw agents add <name>
```

<Note>
`--json` не вмикає неінтерактивний режим. Для сценаріїв використовуйте `--non-interactive` (див. [Автоматизація CLI](/uk/start/wizard-cli-automation)).
</Note>

<Tip>
Класичний майстер містить крок вебпошуку, на якому можна вибрати постачальника: Brave,
DuckDuckGo, Exa, Firecrawl, Gemini, Grok, Kimi, MiniMax Search, Ollama Web
Search, Perplexity, SearXNG або Tavily. Для деяких потрібен ключ API, інші
працюють без ключа. Налаштувати це згодом можна командою `openclaw configure --section web`. Документація:
[Вебінструменти](/uk/tools/web).
</Tip>

## Типовий керований процес

Звичайна команда `openclaw onboard` виконує такі кроки:

1. Приймаєте повідомлення про безпеку.
2. Виявляє налаштовані моделі, змінні середовища з ключами API та підтримувані локальні
   CLI для ШІ.
3. Перевіряє першого виявленого кандидата реальним завершенням. У разі помилки показує
   причину й переходить до наступного придатного кандидата.
4. Якщо всі виявлені варіанти вичерпано, пропонує повторити спробу з виявленим кандидатом або ввести ключ API постачальника
   в запиті з маскуванням введення. Кероване початкове налаштування
   не пропонує Crestodian або вихід із пропуском ШІ, доки логічне виведення не запрацює.
5. Зберігає лише перевірений маршрут моделі та всі потрібні йому стани облікових даних/плагінів.
   Налаштування робочого простору та Gateway залишаються без змін.
6. Запускає Crestodian із перевіреною моделлю, щоб він міг налаштувати робочий простір,
   Gateway, канали, агентів, плагіни та решту необов’язкових компонентів.

Повторний запуск команди в налаштованій інсталяції спочатку перевіряє поточну типову
модель, перетворюючи керований процес на перевірку та відновлення. Невдала
перевірка ніколи автоматично не замінює налаштовану модель; початкове налаштування зупиняється й
запитує, як продовжити. Запустіть `openclaw channels add` або `openclaw configure`, щоб
згодом додати компоненти, не пов’язані з логічним виведенням; для зміни постачальника або маршруту автентифікації
використовуйте `openclaw onboard`.

## Класичний майстер: швидкий запуск і розширене налаштування

Запустіть `openclaw onboard --classic`, щоб відкрити повний майстер. Спочатку він пропонує
вибрати між **Швидким запуском** (типові параметри) і **Розширеним налаштуванням** (повний контроль). Передайте
`--flow quickstart` або `--flow advanced` (псевдонім `manual`), щоб вибрати класичний
процес і пропустити цей запит.

<Tabs>
  <Tab title="Швидкий запуск (типові параметри)">
    - Локальний Gateway, прив’язка до local loopback
    - Типовий робочий простір (або наявний робочий простір)
    - Порт Gateway **18789**
    - Автентифікація Gateway **Токен** (створюється автоматично навіть для local loopback)
    - Політика інструментів: `tools.profile: "coding"` для нових налаштувань (наявний явно вказаний профіль зберігається)
    - Ізоляція особистих повідомлень: `session.dmScope: "per-channel-peer"` для нових налаштувань. Докладніше: [Довідник із налаштування CLI](/uk/start/wizard-cli-reference#outputs-and-internals)
    - Доступ через Tailscale **Вимкнено**
    - Для особистих повідомлень Telegram і WhatsApp типовим є **список дозволених**: Telegram запитує числовий ідентифікатор користувача Telegram, а WhatsApp — номер телефону

  </Tab>
  <Tab title="Розширене налаштування (повний контроль)">
    - Надає доступ до кожного кроку: режиму, робочого простору, Gateway, каналів, фонової служби та Skills

  </Tab>
</Tabs>

Віддалений режим (`--mode remote`) завжди використовує розширений процес; він лише
налаштовує цей комп’ютер для підключення до Gateway в іншому місці й ніколи нічого не встановлює
та не змінює на віддаленому вузлі.

## Що налаштовує класичне початкове налаштування

Локальний режим (типовий) послідовно виконує такі кроки:

1. **Модель/автентифікація** — вибір процесу автентифікації постачальника (ключ API, OAuth або
   ручна автентифікація, специфічна для постачальника), включно з користувацьким постачальником
   (сумісним з OpenAI, сумісним з OpenAI Responses, сумісним з Anthropic або
   автоматично визначеним невідомим типом). Виберіть типову модель.
   Нове налаштування ключа API OpenAI типово використовує `openai/gpt-5.6` (скорочений ідентифікатор прямого API
   відповідає Sol); нове налаштування ChatGPT/Codex типово використовує
   `openai/gpt-5.6-sol`. Повторний запуск налаштування зберігає наявну явно вказану модель,
   включно з `openai/gpt-5.5`. Явно виберіть `openai/gpt-5.5`, якщо
   обліковий запис не надає доступу до GPT-5.6.
   Примітка щодо безпеки: якщо цей агент запускатиме інструменти або оброблятиме вміст Webhook/перехоплювачів,
   надавайте перевагу найпотужнішій доступній моделі останнього покоління та дотримуйтеся
   суворої політики інструментів — слабші або старіші рівні легше піддаються ін’єкціям у запити.
   Для неінтерактивних запусків `--secret-input-mode ref` зберігає посилання на змінні середовища
   замість значень ключів API у відкритому тексті; відповідну змінну середовища має бути вже
   встановлено, інакше початкове налаштування негайно завершиться помилкою. Інтерактивний режим посилань на секрети може
   вказувати на змінну середовища або налаштоване посилання постачальника (`file` чи
   `exec`) зі швидкою попередньою перевіркою перед збереженням. Після налаштування моделі/автентифікації
   майстер пропонує необов’язкову перевірку реального завершення; у разі помилки можна один раз повернутися до
   налаштування моделі/автентифікації або проігнорувати її без блокування решти
   класичного майстра. Ігнорування не розблоковує Crestodian; розмовне налаштування
   усе одно потребує успішної перевірки логічного виведення.
2. **Робочий простір** — каталог для файлів агента (типово `~/.openclaw/workspace`). Створює початкові файли.
3. **Gateway** — порт, адреса прив’язки, режим автентифікації, доступ через Tailscale. В
   інтерактивному режимі токена виберіть зберігання токена у відкритому тексті (типово) або
   увімкніть SecretRef. Неінтерактивний шлях SecretRef: `--gateway-token-ref-env <ENV_VAR>`.
4. **Канали** — вбудовані канали чату й канали офіційних плагінів, зокрема
   Discord, Feishu, Google Chat, iMessage, Mattermost, Microsoft Teams,
   QQ Bot, Signal, Slack, Telegram, WhatsApp тощо.
5. **Фонова служба** — установлює LaunchAgent (macOS), користувацький модуль systemd
   (Linux/WSL2) або нативне заплановане завдання Windows із резервним варіантом
   у користувацькій папці автозапуску.
   Якщо потрібна автентифікація токеном і `gateway.auth.token` керується через SecretRef,
   встановлення фонової служби перевіряє його, але не зберігає визначений токен у
   метаданих середовища служби диспетчера; невизначений SecretRef блокує
   встановлення й надає вказівки. Якщо одночасно задано `gateway.auth.token` і
   `gateway.auth.password`, а `gateway.auth.mode` не задано, встановлення
   блокується, доки ви явно не вкажете режим.
6. **Перевірка працездатності** — запускає Gateway і перевіряє його доступність.
7. **Skills** — установлює рекомендовані Skills та їхні необов’язкові залежності.

<Note>
Повторний запуск початкового налаштування **не** видаляє нічого, якщо ви явно не виберете
**Скидання** (або не передасте `--reset`). CLI `--reset` типово скидає конфігурацію, облікові дані
та сеанси; використовуйте `--reset-scope full`, щоб також видалити робочий простір. Якщо
конфігурація недійсна або містить застарілі ключі, початкове налаштування запропонує спочатку запустити
`openclaw doctor`.
</Note>

`--flow import` запускає виявлений процес міграції (наприклад, Hermes) у
класичному майстрі замість нового налаштування; див. [Міграція](/uk/cli/migrate) і посібники з міграції в розділі
[Встановлення](/uk/install/migrating-hermes). `openclaw onboard --modern` — це
псевдонім сумісності для [Crestodian](/uk/cli/crestodian). Він використовує той самий
шлюз перевірки логічного виведення, що й `openclaw crestodian`: перевірене логічне виведення запускає
помічника, а помилка в інтерактивному режимі повертає до керованого налаштування логічного виведення.

## Додавання іншого агента

Використовуйте `openclaw agents add <name>`, щоб створити окремого агента з власними
робочим простором, сеансами та профілями автентифікації. Запуск без `--workspace` починає
інтерактивний процес вибору імені, робочого простору, автентифікації, каналів і прив’язок — це
не повний майстер `openclaw onboard`.

Що він задає:

- `agents.list[].name`
- `agents.list[].workspace`
- `agents.list[].agentDir`

Примітки:

- Типовий робочий простір: `~/.openclaw/workspace-<agentId>` (або всередині
  `agents.defaults.workspace`, якщо його задано).
- Додайте `bindings`, щоб спрямовувати вхідні повідомлення цьому агенту (початкове налаштування може зробити це за вас).
- Прапорці неінтерактивного режиму: `--model`, `--agent-dir`, `--bind`, `--non-interactive`.
