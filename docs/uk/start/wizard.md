```bash
openclaw onboard
```

Початкове налаштування через CLI — рекомендований спосіб налаштування в терміналі на macOS, Linux і
Windows (нативно або через WSL2). Типово воно виявляє вже доступний на
комп’ютері доступ до ШІ, перевіряє його реальним запитом на доповнення та запускає OpenClaw,
щоб налаштувати робочий простір, Gateway і додаткові функції. `openclaw setup` запускає той самий процес ([Налаштування](/uk/cli/setup) описує
варіант `--baseline` лише для конфігурації). Користувачі настільної версії Windows також можуть почати
з [Windows Hub](/uk/platforms/windows).

Кероване початкове налаштування спочатку налаштовує інференс. Воно виявляє доступний доступ до ШІ,
вимагає успішного реального запиту на доповнення й лише після цього запускає [OpenClaw](/cli/openclaw),
щоб налаштувати решту OpenClaw. Вибір **Skip for now** завершує початкове налаштування
без запуску OpenClaw.

Класичний майстер залишається доступним для власних постачальників, налаштування віддаленого Gateway,
підключення каналів, керування фоновою службою, навичок та імпорту. Запустіть його явно
за допомогою `openclaw onboard --classic`; керований засіб вибору інференсу не передає
керування йому. Після успішної перевірки інференсу OpenClaw може використовувати `open channel wizard for
<channel>`, щоб передати налаштування каналу, для якого потрібні секрети, майстру в терміналі з маскуванням.
Щоб змінити постачальника моделі або спосіб автентифікації, закрийте OpenClaw і виконайте
`openclaw onboard`; OpenClaw не відкриває керовані або класичні процеси налаштування постачальника.

<Info>
Найшвидший спосіб розпочати перший чат: завершіть кероване налаштування, виконайте `openclaw dashboard` і спілкуйтеся
у браузері через інтерфейс керування. Документація: [Панель керування](/uk/web/dashboard).
</Info>

## Локаль

Майстер локалізує фіксований текст початкового налаштування. Порядок визначення: `OPENCLAW_LOCALE`,
`LC_ALL`, `LC_MESSAGES`, `LANG`, потім англійська. Підтримувані локалі: `en`,
`zh-CN`, `zh-TW`.

```bash
OPENCLAW_LOCALE=zh-CN openclaw onboard
```

Назви продуктів, команди, ключі конфігурації, URL-адреси, ідентифікатори постачальників, ідентифікатори моделей і
мітки плагінів/каналів залишаються англійською незалежно від локалі.

Щоб згодом повторно налаштувати параметри, не пов’язані з інференсом:

```bash
openclaw configure
openclaw agents add <name>
```

<Note>
`--json` не означає неінтерактивний режим. Для сценаріїв використовуйте `--non-interactive` (див. [Автоматизація CLI](/uk/start/wizard-cli-automation)).
</Note>

<Tip>
Класичний майстер містить крок вебпошуку, де можна вибрати постачальника: Brave,
DuckDuckGo, Exa, Firecrawl, Gemini, Grok, Kimi, MiniMax Search, Ollama Web
Search, Perplexity, SearXNG або Tavily. Для деяких потрібен ключ API, інші
працюють без ключа. Налаштуйте це згодом за допомогою `openclaw configure --section web`. Документація:
[Вебінструменти](/uk/tools/web).
</Tip>

## Керований процес за замовчуванням

Звичайна команда `openclaw onboard` виконує такі кроки:

1. Прийняти повідомлення про безпеку.
2. Виявити налаштовані моделі, змінні середовища з ключами API, підтримувані локальні
   CLI для ШІ та вже встановлені моделі з підтримкою інструментів на доступних серверах Ollama або LM
   Studio на хості Gateway. Ця перевірка лише для читання ніколи не завантажує
   модель. Про встановлені Gemini CLI та Antigravity повідомляється, але вони не перевіряються автоматично,
   оскільки для них неможливо примусово виконати пробу без інструментів.
3. Перевірити першого виявленого кандидата реальним запитом на доповнення. У разі невдачі показати
   причину й перейти до наступного придатного кандидата.
4. Якщо виявлення не дало результатів, вибрати OpenAI, Anthropic, xAI (Grok), Google або
   OpenRouter чи вибрати **More…** для решти постачальників. Регіони, плани та підтримувані
   способи через браузер, пристрій, ключ API або токен для кожного постачальника
   відображаються в другому меню й перевіряються тим самим реальним запитом на доповнення.
   Виберіть **Skip for now**, щоб вийти без запуску OpenClaw.
5. Зберегти лише перевірений маршрут моделі та всі необхідні для нього дані
   облікових даних/плагінів. Параметри робочого простору й Gateway залишаються незмінними.
6. Запустити OpenClaw із перевіреною моделлю, щоб він міг налаштувати робочий простір,
   Gateway, канали, агентів, плагіни та решту додаткових параметрів.

Повторний запуск команди в налаштованому середовищі спочатку перевіряє поточну типову
модель, перетворюючи керований процес на перевірку та відновлення. Невдала
перевірка ніколи не замінює налаштовану модель автоматично; початкове налаштування зупиняється й
запитує, як продовжити. Виконайте `openclaw channels add` або `openclaw configure` для
подальших доповнень, не пов’язаних з інференсом; використовуйте `openclaw onboard`, щоб змінити маршрут
постачальника або автентифікації.

## Класичний майстер: QuickStart і Advanced

Виконайте `openclaw onboard --classic`, щоб відкрити повний майстер. Він починається з
вибору між **QuickStart** (типові параметри) та **Advanced** (повний контроль). Передайте
`--flow quickstart` або `--flow advanced` (псевдонім `manual`), щоб вибрати класичний
процес і пропустити цей запит.

<Tabs>
  <Tab title="QuickStart (типові параметри)">
    - Локальний Gateway, прив’язування до loopback
    - Типовий робочий простір (або наявний робочий простір)
    - Порт Gateway **18789**
    - Автентифікація Gateway **Token** (генерується автоматично навіть для loopback)
    - Політика інструментів: `tools.profile: "coding"` для нових налаштувань (наявний явно вказаний профіль зберігається)
    - Ізоляція приватних повідомлень: `session.dmScope: "per-channel-peer"` для нових налаштувань. Докладніше: [Довідник із налаштування CLI](/uk/start/wizard-cli-reference#outputs-and-internals)
    - Доступ через Tailscale **Off**
    - Приватні повідомлення Telegram і WhatsApp типово використовують **allowlist**: Telegram запитує числовий ідентифікатор користувача Telegram, а WhatsApp — номер телефону

  </Tab>
  <Tab title="Advanced (повний контроль)">
    - Надає доступ до кожного кроку: режиму, робочого простору, Gateway, каналів, фонової служби та навичок

  </Tab>
</Tabs>

Віддалений режим (`--mode remote`) завжди використовує розширений процес; він лише
налаштовує цей комп’ютер для підключення до Gateway в іншому місці та ніколи нічого не встановлює
і не змінює на віддаленому хості.

## Що налаштовує класичне початкове налаштування

Локальний режим (типовий) послідовно виконує такі кроки:

1. **Модель/автентифікація** — виберіть процес автентифікації постачальника (ключ API, OAuth або
   ручну автентифікацію для конкретного постачальника), зокрема Custom Provider
   (сумісний з OpenAI, сумісний з OpenAI Responses, сумісний з Anthropic або
   Unknown з автоматичним визначенням). Виберіть типову модель.
   Нове налаштування ключа API OpenAI типово використовує `openai/gpt-5.6` (простий ідентифікатор прямого API
   відповідає Sol); нове налаштування ChatGPT/Codex типово використовує
   `openai/gpt-5.6-sol`. Повторний запуск налаштування зберігає наявну явно вказану модель,
   зокрема `openai/gpt-5.5`. Явно виберіть `openai/gpt-5.5`, якщо
   обліковий запис не надає доступу до GPT-5.6.
   Примітка щодо безпеки: якщо цей агент запускатиме інструменти або оброблятиме вміст
   вебхуків/хуків, надавайте перевагу найпотужнішій доступній моделі останнього покоління та зберігайте
   сувору політику інструментів — слабші або старіші рівні легше піддаються ін’єкціям у промпт.
   Для неінтерактивних запусків `--secret-input-mode ref` зберігає посилання на змінні середовища
   замість значень ключів API у відкритому тексті; відповідну змінну середовища має бути вже
   встановлено, інакше початкове налаштування негайно завершується помилкою. В інтерактивному режимі посилання на секрет
   може вказувати на змінну середовища або налаштоване посилання постачальника (`file` чи
   `exec`) зі швидкою попередньою перевіркою перед збереженням. Після налаштування моделі/автентифікації
   майстер пропонує необов’язкову перевірку реальним запитом на доповнення; у разі невдачі можна один раз повернутися до
   налаштування моделі/автентифікації або проігнорувати її, не блокуючи решту
   класичного майстра. Ігнорування не розблоковує OpenClaw; для налаштування спілкування
   все одно потрібна успішна перевірка інференсу.
2. **Робочий простір** — каталог для файлів агента (типово `~/.openclaw/workspace`). Створює початкові файли.
3. **Gateway** — порт, адреса прив’язування, режим автентифікації, доступ через Tailscale. В
   інтерактивному режимі токена виберіть зберігання токена у відкритому тексті (типово) або
   погодьтеся на використання SecretRef. Шлях SecretRef у неінтерактивному режимі: `--gateway-token-ref-env <ENV_VAR>`.
4. **Канали** — вбудовані та офіційні плагінні канали чату, зокрема
   Discord, Feishu, Google Chat, iMessage, Mattermost, Microsoft Teams,
   QQ Bot, Signal, Slack, Telegram, WhatsApp та інші.
5. **Фонова служба** — встановлює LaunchAgent (macOS), користувацький модуль systemd
   (Linux/WSL2) або нативне заплановане завдання Windows із резервним варіантом
   через папку автозапуску користувача.
   Якщо потрібна автентифікація токеном і `gateway.auth.token` керується через SecretRef,
   встановлення фонової служби перевіряє його, але не зберігає визначений токен у
   метаданих середовища служби супервізора; невизначений SecretRef блокує
   встановлення й показує вказівки. Якщо одночасно встановлено `gateway.auth.token` і
   `gateway.auth.password`, а `gateway.auth.mode` не встановлено, встановлення
   блокується, доки режим не буде вказано явно.
6. **Перевірка працездатності** — запускає Gateway і перевіряє його доступність.
7. **Навички** — встановлює рекомендовані навички та їхні необов’язкові залежності.

<Note>
Повторний запуск початкового налаштування **не** видаляє нічого, якщо явно не вибрати
**Reset** (або не передати `--reset`). Команда CLI `--reset` типово скидає конфігурацію, облікові дані
та сеанси; використовуйте `--reset-scope full`, щоб також видалити робочий простір. Якщо
конфігурація недійсна або містить застарілі ключі, початкове налаштування запропонує спочатку виконати
`openclaw doctor`.
</Note>

`--flow import` запускає в класичному майстрі процес виявленої міграції (наприклад, Hermes)
замість нового налаштування; див. [Міграція](/uk/cli/migrate) та посібники з міграції в розділі
[Встановлення](/uk/install/migrating-hermes). `openclaw onboard --modern` — це
псевдонім сумісності для [OpenClaw](/cli/openclaw). Він використовує той самий
бар’єр інференсу, що й `openclaw setup`: перевірений інференс запускає
асистента, а невдача в інтерактивному режимі повертає до керованого налаштування інференсу.

## Додавання іншого агента

Використовуйте `openclaw agents add <name>`, щоб створити окремого агента з власними
робочим простором, сеансами та профілями автентифікації. Запуск без `--workspace` починає
інтерактивний процес вибору імені, робочого простору, автентифікації, каналів і прив’язувань — це
не повний майстер `openclaw onboard`.

Що він налаштовує:

- `agents.list[].name`
- `agents.list[].workspace`
- `agents.list[].agentDir`

Примітки:

- Типовий робочий простір: `~/.openclaw/workspace-<agentId>` (або в
  `agents.defaults.workspace`, якщо його встановлено).
- Додайте `bindings`, щоб спрямовувати вхідні повідомлення цьому агенту (початкове налаштування може зробити це за вас).
- Прапорці неінтерактивного режиму: `--model`, `--agent-dir`, `--bind`, `--non-interactive`.
