## Публічна модель можливостей

Можливості — це публічна модель **нативних плагінів** всередині OpenClaw. Кожен нативний плагін OpenClaw реєструється для одного або кількох типів можливостей:

| Можливість             | Метод реєстрації                              | Приклади плагінів                      |
| ---------------------- | ------------------------------------------------ | ------------------------------------ |
| Текстовий інференс         | `api.registerProvider(...)`                      | `openai`, `anthropic`                |
| Бекенд CLI-інференсу  | `api.registerCliBackend(...)`                    | `openai`, `anthropic`                |
| Мовлення                 | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`            |
| Транскрибування в реальному часі | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                             |
| Голос у реальному часі         | `api.registerRealtimeVoiceProvider(...)`         | `openai`                             |
| Розуміння медіа    | `api.registerMediaUnderstandingProvider(...)`    | `openai`, `google`                   |
| Генерація зображень       | `api.registerImageGenerationProvider(...)`       | `openai`, `google`, `fal`, `minimax` |
| Генерація музики       | `api.registerMusicGenerationProvider(...)`       | `google`, `minimax`                  |
| Генерація відео       | `api.registerVideoGenerationProvider(...)`       | `qwen`                               |
| Вебзавантаження              | `api.registerWebFetchProvider(...)`              | `firecrawl`                          |
| Вебпошук             | `api.registerWebSearchProvider(...)`             | `google`                             |
| Канал / обмін повідомленнями    | `api.registerChannel(...)`                       | `msteams`, `matrix`                  |
| Виявлення Gateway      | `api.registerGatewayDiscoveryService(...)`       | `bonjour`                            |

<Note>
Плагін, який реєструє нуль можливостей, але надає хуки, інструменти, служби виявлення або фонові служби, є **застарілим плагіном лише з хуками**. Цей шаблон усе ще повністю підтримується.
</Note>

### Позиція щодо зовнішньої сумісності

Модель можливостей уже впроваджена в ядрі й сьогодні використовується в комплектних/нативних плагінах, але сумісність зовнішніх плагінів усе ще потребує суворішого критерію, ніж «це експортовано, отже це заморожено».

| Ситуація з плагіном                                  | Рекомендації                                                                                         |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Наявні зовнішні плагіни                         | Підтримуйте працездатність інтеграцій на основі хуків; це базовий рівень сумісності.                        |
| Нові комплектні/нативні плагіни                        | Надавайте перевагу явній реєстрації можливостей замість звернень усередину, специфічних для постачальника, або нових дизайнів лише з хуками. |
| Зовнішні плагіни, що впроваджують реєстрацію можливостей | Дозволено, але вважайте допоміжні поверхні, специфічні для можливостей, такими, що розвиваються, якщо документація не позначає їх як стабільні. |

Реєстрація можливостей — це запланований напрям. Застарілі хуки залишаються найбезпечнішим шляхом без поломок для зовнішніх плагінів під час переходу. Не всі експортовані допоміжні підшляхи рівноцінні — надавайте перевагу вузьким задокументованим контрактам над випадковими допоміжними експортами.

### Форми плагінів

OpenClaw класифікує кожен завантажений плагін за формою на основі його фактичної поведінки реєстрації (а не лише статичних метаданих):

<AccordionGroup>
  <Accordion title="plain-capability">
    Реєструє рівно один тип можливості (наприклад, плагін лише для провайдера, як-от `mistral`).
  </Accordion>
  <Accordion title="hybrid-capability">
    Реєструє кілька типів можливостей (наприклад, `openai` відповідає за текстовий інференс, мовлення, розуміння медіа та генерацію зображень).
  </Accordion>
  <Accordion title="hook-only">
    Реєструє лише хуки (типізовані або користувацькі), без можливостей, інструментів, команд або служб.
  </Accordion>
  <Accordion title="non-capability">
    Реєструє інструменти, команди, служби або маршрути, але без можливостей.
  </Accordion>
</AccordionGroup>

Використовуйте `openclaw plugins inspect <id>`, щоб переглянути форму плагіна та розподіл його можливостей. Докладніше див. [довідник CLI](/uk/cli/plugins#inspect).

### Застарілі хуки

Хук `before_agent_start` залишається підтримуваним як шлях сумісності для плагінів лише з хуками. Застарілі реальні плагіни все ще залежать від нього.

Напрям:

- підтримувати його працездатність
- документувати його як застарілий
- надавати перевагу `before_model_resolve` для роботи з перевизначенням моделі/провайдера
- надавати перевагу `before_prompt_build` для роботи зі зміною prompt
- видаляти лише після зменшення реального використання та після того, як покриття фікстурами доведе безпечність міграції

### Сигнали сумісності

Коли ви запускаєте `openclaw doctor` або `openclaw plugins inspect <id>`, ви можете побачити одну з цих міток:

| Сигнал                     | Значення                                                      |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | Конфігурація коректно розбирається, а плагіни розв’язуються                       |
| **compatibility advisory** | Плагін використовує підтримуваний, але старіший шаблон (наприклад, `hook-only`) |
| **legacy warning**         | Плагін використовує `before_agent_start`, який є застарілим        |
| **hard error**             | Конфігурація недійсна або плагін не вдалося завантажити                   |

Ні `hook-only`, ні `before_agent_start` сьогодні не зламають ваш плагін: `hook-only` є рекомендаційним сигналом, а `before_agent_start` лише спричиняє попередження. Ці сигнали також з’являються в `openclaw status --all` і `openclaw plugins doctor`.

## Огляд архітектури

Система плагінів OpenClaw має чотири рівні:

<Steps>
  <Step title="Маніфест + виявлення">
    OpenClaw знаходить плагіни-кандидати з налаштованих шляхів, коренів робочих просторів, глобальних коренів плагінів і комплектних плагінів. Виявлення спочатку читає нативні маніфести `openclaw.plugin.json` плюс підтримувані маніфести bundle.
  </Step>
  <Step title="Увімкнення + валідація">
    Ядро вирішує, чи виявлений плагін увімкнений, вимкнений, заблокований або вибраний для ексклюзивного слота, такого як пам’ять.
  </Step>
  <Step title="Завантаження під час виконання">
    Нативні плагіни OpenClaw завантажуються в процесі та реєструють можливості в центральному реєстрі. Упакований JavaScript завантажується через нативний `require`; вихідний TypeScript повертається до Jiti. Сумісні bundle нормалізуються в записи реєстру без імпортування коду виконання.
  </Step>
  <Step title="Використання поверхнями">
    Решта OpenClaw читає реєстр, щоб надавати інструменти, канали, налаштування провайдерів, хуки, HTTP-маршрути, CLI-команди та служби.
  </Step>
</Steps>

Саме для CLI плагінів виявлення кореневих команд розділено на дві фази:

- метадані часу розбору надходять із `registerCli(..., { descriptors: [...] })`
- справжній CLI-модуль плагіна може залишатися лінивим і реєструватися під час першого виклику

Це утримує CLI-код, що належить плагіну, всередині плагіна, водночас даючи OpenClaw змогу резервувати імена кореневих команд до розбору.

Важлива межа дизайну:

- валідація маніфеста/конфігурації має працювати з **метаданих маніфеста/схеми** без виконання коду плагіна
- виявлення нативних можливостей може завантажувати довірений вхідний код плагіна, щоб побудувати неактивувальний знімок реєстру
- нативна поведінка під час виконання походить зі шляху `register(api)` модуля плагіна з `api.registrationMode === "full"`

Такий поділ дає OpenClaw змогу валідувати конфігурацію, пояснювати відсутні/вимкнені плагіни та будувати підказки UI/схеми до активації повного середовища виконання.

### Знімок метаданих плагінів і таблиця пошуку

Під час запуску Gateway будує один `PluginMetadataSnapshot` для поточного знімка конфігурації. Знімок містить лише метадані: він зберігає індекс установлених плагінів, реєстр маніфестів, діагностику маніфестів, мапи власників, нормалізатор id плагінів і записи маніфестів. Він не містить завантажених модулів плагінів, SDK провайдерів, вмісту пакетів або експортів часу виконання.

Валідація конфігурації з урахуванням плагінів, автоматичне увімкнення під час запуску та bootstrap плагінів Gateway споживають цей знімок замість незалежної перебудови метаданих маніфеста/індексу. `PluginLookUpTable` виводиться з того самого знімка й додає план плагінів запуску для поточної конфігурації виконання.

Після запуску Gateway зберігає поточний знімок метаданих як замінюваний продукт часу виконання. Повторне виявлення провайдерів під час виконання може запозичувати цей знімок замість реконструювання встановленого індексу та реєстру маніфестів для кожного проходу каталогу провайдерів. Знімок очищується або замінюється під час вимкнення Gateway, змін конфігурації/інвентарю плагінів і записів установленого індексу; виклики повертаються до холодного шляху маніфеста/індексу, коли сумісного поточного знімка немає. Перевірки сумісності мають включати корені виявлення плагінів, такі як `plugins.load.paths`, і стандартний робочий простір агента, оскільки плагіни робочого простору є частиною області метаданих.

Знімок і таблиця пошуку утримують повторні рішення запуску на швидкому шляху:

- володіння каналами
- відкладений запуск каналів
- id плагінів запуску
- володіння провайдерами та CLI-бекендами
- володіння setup-провайдером, псевдонімом команди, провайдером каталогу моделей і контрактом маніфеста
- валідація схеми конфігурації плагіна та схеми конфігурації каналу
- рішення автоматичного увімкнення під час запуску

Межа безпеки — це заміна знімка, а не мутація. Перебудовуйте знімок, коли змінюється конфігурація, інвентар плагінів, записи встановлення або політика збереженого індексу. Не вважайте його широким змінним глобальним реєстром і не зберігайте необмежені історичні знімки. Завантаження плагінів під час виконання залишається окремим від знімків метаданих, щоб застарілий стан виконання не міг бути прихований за кешем метаданих.

Правило кешу задокументовано в [внутрішній архітектурі плагінів](/uk/plugins/architecture-internals#plugin-cache-boundary): метадані маніфеста та виявлення є свіжими, якщо виклик не тримає явний знімок, таблицю пошуку або реєстр маніфестів для поточного потоку. Приховані кеші метаданих і TTL за настінним годинником не є частиною завантаження плагінів. Лише кеші завантажувача часу виконання, модулів і артефактів залежностей можуть зберігатися після фактичного завантаження коду або встановлених артефактів.

Деякі виклики холодного шляху все ще реконструюють реєстри маніфестів безпосередньо зі збереженого індексу встановлених плагінів замість отримання `PluginLookUpTable` Gateway. Тепер цей шлях реконструює реєстр на вимогу; надавайте перевагу передаванню поточної таблиці пошуку або явного реєстру маніфестів через потоки виконання, коли виклик уже має один із них.

### Планування активації

Планування активації є частиною площини керування. Виклики можуть запитувати, які плагіни релевантні для конкретної команди, провайдера, каналу, маршруту, agent harness або можливості, перш ніж завантажувати ширші реєстри часу виконання.

Планувальник зберігає поточну поведінку маніфеста сумісною:

- поля `activation.*` є явними підказками планувальника
- `providers`, `channels`, `commandAliases`, `setup.providers`, `contracts.tools` і хуки залишаються fallback володіння з маніфеста
- API планувальника лише з id залишається доступним для наявних викликів
- API плану повідомляє мітки причин, щоб діагностика могла відрізняти явні підказки від fallback володіння

<Warning>
Не розглядайте `activation` як lifecycle hook або заміну для `register(...)`. Це метадані, які використовуються для звуження завантаження. Надавайте перевагу полям ownership, коли вони вже описують зв’язок; використовуйте `activation` лише для додаткових підказок планувальнику.
</Warning>

### Канальні Plugin-и та спільний інструмент повідомлень

Канальним Plugin-ам не потрібно реєструвати окремий інструмент надсилання/редагування/реакцій для звичайних дій у чаті. OpenClaw зберігає один спільний інструмент `message` у core, а канальні Plugin-и володіють специфічним для каналу виявленням і виконанням за ним.

Поточна межа така:

- core володіє хостом спільного інструмента `message`, підключенням prompt, обліком session/thread і диспетчеризацією виконання
- канальні Plugin-и володіють scoped action discovery, capability discovery і будь-якими специфічними для каналу фрагментами схеми
- канальні Plugin-и володіють специфічною для провайдера граматикою розмов session, наприклад тим, як conversation ids кодують thread ids або успадковуються від parent conversations
- канальні Plugin-и виконують фінальну дію через свій action adapter

Для канальних Plugin-ів поверхня SDK — це `ChannelMessageActionAdapter.describeMessageTool(...)`. Цей уніфікований виклик discovery дає Plugin змогу повернути видимі дії, capabilities і внески до схеми разом, щоб ці частини не розходилися.

Коли специфічний для каналу параметр message-tool містить медіаджерело, як-от локальний шлях або URL віддаленого медіа, Plugin також має повертати `mediaSourceParams` із `describeMessageTool(...)`. Core використовує цей явний список, щоб застосовувати нормалізацію sandbox path і підказки outbound media-access без жорсткого кодування назв параметрів, якими володіє Plugin. Надавайте перевагу action-scoped maps, а не одному плоскому списку на весь канал, щоб параметр медіа лише для профілю не нормалізувався для не пов’язаних дій, як-от `send`.

Core передає runtime scope у цей крок discovery. Важливі поля включають:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- довірений inbound `requesterSenderId`

Це важливо для context-sensitive Plugin-ів. Канал може приховувати або показувати дії повідомлень на основі активного account, поточної room/thread/message або довіреної ідентичності requester без жорстко закодованих специфічних для каналу гілок у core-інструменті `message`.

Саме тому зміни маршрутизації embedded-runner усе ще є роботою Plugin: runner відповідає за передавання поточної ідентичності chat/session у межу discovery Plugin, щоб спільний інструмент `message` показував правильну, належну каналу поверхню для поточного turn.

Для execution helpers, якими володіє канал, bundled Plugin-и мають тримати execution runtime всередині власних extension modules. Core більше не володіє message-action runtimes Discord, Slack, Telegram або WhatsApp під `src/agents/tools`. Ми не публікуємо окремі subpaths `plugin-sdk/*-action-runtime`, а bundled Plugin-и мають імпортувати власний локальний runtime code безпосередньо зі своїх extension-owned modules.

Та сама межа загалом застосовується до provider-named SDK seams: core не має імпортувати специфічні для каналу convenience barrels для Slack, Discord, Signal, WhatsApp або подібних extensions. Якщо core потрібна поведінка, він має або споживати власний barrel bundled Plugin `api.ts` / `runtime-api.ts`, або просунути потребу в вузьку generic capability у спільному SDK.

Bundled Plugin-и дотримуються того самого правила. `runtime-api.ts` bundled Plugin не має повторно експортувати власний branded facade `openclaw/plugin-sdk/<plugin-id>`. Ці branded facades залишаються compatibility shims для зовнішніх Plugin-ів і старіших consumers, але bundled Plugin-и мають використовувати локальні exports плюс вузькі generic SDK subpaths, як-от `openclaw/plugin-sdk/channel-policy`, `openclaw/plugin-sdk/runtime-store` або `openclaw/plugin-sdk/webhook-ingress`. Новий код не має додавати plugin-id-specific SDK facades, якщо compatibility boundary для наявної зовнішньої екосистеми цього не потребує.

Для опитувань зокрема є два шляхи виконання:

- `outbound.sendPoll` — спільна базова лінія для каналів, що відповідають спільній моделі опитувань
- `actions.handleAction("poll")` — бажаний шлях для специфічної для каналу семантики опитувань або додаткових параметрів опитування

Core тепер відкладає спільний парсинг опитування до моменту після того, як dispatch опитування Plugin відхилить дію, тож обробники опитувань, якими володіє Plugin, можуть приймати специфічні для каналу поля опитування без блокування generic poll parser спочатку.

Див. [Внутрішню архітектуру Plugin](/uk/plugins/architecture-internals) для повної послідовності запуску.

## Модель володіння capability

OpenClaw розглядає native Plugin як межу володіння для **компанії** або **функції**, а не як набір не пов’язаних між собою інтеграцій.

Це означає:

- Plugin компанії зазвичай має володіти всіма поверхнями цієї компанії, зверненими до OpenClaw
- Plugin функції зазвичай має володіти повною поверхнею функції, яку він вводить
- канали мають споживати спільні capabilities core замість повторної ad hoc реалізації поведінки провайдера

<AccordionGroup>
  <Accordion title="Багатофункціональний вендор">
    `openai` володіє text inference, speech, realtime voice, media understanding і image generation. `google` володіє text inference плюс media understanding, image generation і web search. `qwen` володіє text inference плюс media understanding і video generation.
  </Accordion>
  <Accordion title="Однофункціональний вендор">
    `elevenlabs` і `microsoft` володіють speech; `firecrawl` володіє web-fetch; `minimax` / `mistral` / `moonshot` / `zai` володіють media-understanding backends.
  </Accordion>
  <Accordion title="Plugin функції">
    `voice-call` володіє call transport, tools, CLI, routes і Twilio media-stream bridging, але споживає спільні capabilities speech, realtime transcription і realtime voice замість прямого імпорту vendor Plugin-ів.
  </Accordion>
</AccordionGroup>

Бажаний кінцевий стан:

- OpenAI живе в одному Plugin, навіть якщо він охоплює text models, speech, images і майбутнє video
- інший вендор може зробити те саме для власної surface area
- каналам байдуже, який vendor Plugin володіє провайдером; вони споживають спільний capability contract, відкритий core

Ось ключова відмінність:

- **Plugin** = межа володіння
- **capability** = контракт core, який можуть реалізовувати або споживати кілька Plugin-ів

Тож якщо OpenClaw додає нову область, як-от video, перше питання не «який провайдер має жорстко закодувати обробку video?» Перше питання — «яким є core video capability contract?» Коли цей контракт існує, vendor Plugin-и можуть реєструватися під нього, а channel/feature Plugin-и можуть його споживати.

Якщо capability ще не існує, правильний крок зазвичай такий:

<Steps>
  <Step title="Визначте capability">
    Визначте відсутню capability у core.
  </Step>
  <Step title="Відкрийте через SDK">
    Відкрийте її через plugin API/runtime типізованим способом.
  </Step>
  <Step title="Підключіть consumers">
    Підключіть channels/features до цієї capability.
  </Step>
  <Step title="Реалізації вендорів">
    Дайте vendor Plugin-ам зареєструвати реалізації.
  </Step>
</Steps>

Це зберігає ownership явним і водночас уникає поведінки core, що залежить від одного вендора або одноразового plugin-specific code path.

### Шарування capability

Використовуйте цю ментальну модель, коли вирішуєте, де має бути код:

<Tabs>
  <Tab title="Шар capability core">
    Спільна оркестрація, policy, fallback, правила config merge, delivery semantics і typed contracts.
  </Tab>
  <Tab title="Шар vendor Plugin">
    Специфічні для вендора API, auth, model catalogs, speech synthesis, image generation, майбутні video backends, usage endpoints.
  </Tab>
  <Tab title="Шар channel/feature Plugin">
    Інтеграція Slack/Discord/voice-call/etc., яка споживає capabilities core і подає їх на surface.
  </Tab>
</Tabs>

Наприклад, TTS має таку форму:

- core володіє reply-time TTS policy, fallback order, prefs і channel delivery
- `openai`, `elevenlabs` і `microsoft` володіють synthesis implementations
- `voice-call` споживає telephony TTS runtime helper

Цей самий шаблон варто надавати перевагу для майбутніх capabilities.

### Приклад багатофункціонального Plugin компанії

Plugin компанії має відчуватися цілісним зовні. Якщо OpenClaw має спільні контракти для models, speech, realtime transcription, realtime voice, media understanding, image generation, video generation, web fetch і web search, вендор може володіти всіма своїми surfaces в одному місці:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

Важливі не точні назви helper. Важлива форма:

- один Plugin володіє vendor surface
- core усе ще володіє capability contracts
- channels і feature Plugin-и споживають helpers `api.runtime.*`, а не vendor code
- contract tests можуть перевіряти, що Plugin зареєстрував capabilities, якими заявляє, що володіє

### Приклад capability: розуміння video

OpenClaw уже розглядає розуміння image/audio/video як одну спільну capability. Та сама модель ownership застосовується і тут:

<Steps>
  <Step title="Core визначає контракт">
    Core визначає контракт media-understanding.
  </Step>
  <Step title="Vendor Plugin-и реєструються">
    Vendor Plugin-и реєструють `describeImage`, `transcribeAudio` і `describeVideo`, коли це застосовно.
  </Step>
  <Step title="Consumers використовують спільну поведінку">
    Channels і feature Plugin-и споживають спільну поведінку core замість прямого підключення до vendor code.
  </Step>
</Steps>

Це уникає вшивання припущень одного провайдера про video в core. Plugin володіє vendor surface; core володіє capability contract і fallback behavior.

Video generation уже використовує ту саму послідовність: core володіє typed capability contract і runtime helper, а vendor Plugin-и реєструють реалізації `api.registerVideoGenerationProvider(...)` під нього.

Потрібен конкретний checklist розгортання? Див. [Capability Cookbook](/uk/plugins/architecture).

## Контракти та примусове дотримання

Поверхня plugin API навмисно типізована й централізована в `OpenClawPluginApi`. Цей контракт визначає підтримувані registration points і runtime helpers, на які Plugin може покладатися.

Чому це важливо:

- автори Plugin-ів отримують один стабільний внутрішній стандарт
- core може відхиляти duplicate ownership, як-от коли два Plugin-и реєструють той самий provider id
- startup може показувати actionable diagnostics для malformed registration
- contract tests можуть забезпечувати ownership bundled-plugin і запобігати silent drift

Є два шари примусового дотримання:

<AccordionGroup>
  <Accordion title="Примусове застосування реєстрації під час виконання">
    Реєстр плагінів перевіряє реєстрації під час завантаження плагінів. Приклади: дубльовані ідентифікатори постачальників, дубльовані ідентифікатори постачальників мовлення та неправильно сформовані реєстрації створюють діагностику плагінів замість невизначеної поведінки.
  </Accordion>
  <Accordion title="Тести контрактів">
    Вбудовані плагіни фіксуються в реєстрах контрактів під час тестових запусків, щоб OpenClaw міг явно перевіряти власність. Нині це використовується для постачальників моделей, постачальників мовлення, постачальників вебпошуку та власності вбудованих реєстрацій.
  </Accordion>
</AccordionGroup>

Практичний ефект полягає в тому, що OpenClaw наперед знає, який плагін володіє якою поверхнею. Це дає змогу ядру й каналам безперешкодно компонуватися, адже власність оголошена, типізована й придатна для тестування, а не неявна.

    - типізовані
    - малі
    - специфічні для можливості
    - належать ядру
    - придатні для повторного використання кількома плагінами

## Модель виконання

Нативні плагіни OpenClaw працюють **у процесі** разом із Gateway. Вони не ізольовані в sandbox. Завантажений нативний плагін має ту саму межу довіри на рівні процесу, що й код ядра.

<Warning>
Наслідки нативних плагінів: плагін може реєструвати інструменти, мережеві обробники, хуки та служби; помилка плагіна може призвести до аварійного завершення або дестабілізації Gateway; а зловмисний нативний плагін еквівалентний виконанню довільного коду всередині процесу OpenClaw.
</Warning>

Сумісні пакети безпечніші за замовчуванням, оскільки OpenClaw наразі розглядає їх як пакети метаданих/вмісту. У поточних випусках це переважно означає вбудовані Skills.

Використовуйте списки дозволених і явні шляхи встановлення/завантаження для невбудованих плагінів. Розглядайте плагіни робочого простору як код для часу розробки, а не як виробничі значення за замовчуванням.

Для імен пакетів вбудованого робочого простору тримайте ідентифікатор плагіна прив’язаним до npm-імені: `@openclaw/<id>` за замовчуванням або схвалений типізований суфікс, як-от `-provider`, `-plugin`, `-speech`, `-sandbox` чи `-media-understanding`, коли пакет навмисно надає вужчу роль плагіна.

<Note>
**Примітка щодо довіри:** `plugins.allow` довіряє **ідентифікаторам плагінів**, а не походженню джерела. Плагін робочого простору з тим самим ідентифікатором, що й вбудований плагін, навмисно затіняє вбудовану копію, коли цей плагін робочого простору ввімкнено/додано до списку дозволених. Це нормально й корисно для локальної розробки, тестування патчів і термінових виправлень. Довіра до вбудованого плагіна визначається зі знімка джерела — маніфесту й коду на диску під час завантаження, — а не з метаданих встановлення. Пошкоджений або підмінений запис встановлення не може непомітно розширити поверхню довіри вбудованого плагіна поза межі того, що заявляє фактичне джерело.
</Note>

## Межа експорту

OpenClaw експортує можливості, а не зручності реалізації.

Тримайте реєстрацію можливостей публічною. Скорочуйте допоміжні експорти, що не є контрактами:

- допоміжні підшляхи, специфічні для вбудованого плагіна
- підшляхи інфраструктури часу виконання, не призначені як публічний API
- специфічні для постачальника допоміжні зручності
- допоміжні засоби налаштування/онбордингу, які є деталями реалізації

Зарезервовані допоміжні підшляхи вбудованих плагінів вилучено зі згенерованої карти експорту SDK. Тримайте специфічні для власника допоміжні засоби всередині пакета плагіна-власника; просувайте лише повторно використовувану поведінку хоста до загальних контрактів SDK, таких як `plugin-sdk/gateway-runtime`, `plugin-sdk/security-runtime` і `plugin-sdk/plugin-config-runtime`.
