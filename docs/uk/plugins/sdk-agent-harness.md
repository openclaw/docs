**Обв’язка агента** — це низькорівневий виконавець для одного підготовленого ходу агента OpenClaw. Це не постачальник моделі, не канал і не реєстр інструментів. Для користувацької ментальної моделі див. [Середовища виконання агентів](/uk/concepts/agent-runtimes).

Використовуйте цю поверхню лише для вбудованих або довірених нативних Plugin. Контракт досі експериментальний, бо типи параметрів навмисно віддзеркалюють поточний вбудований runner.

## Коли використовувати обв’язку

Зареєструйте обв’язку агента, коли сімейство моделей має власне нативне середовище виконання сесій, а звичайний транспорт постачальника OpenClaw є неправильною абстракцією.

Приклади:

- нативний сервер агента для кодування, який володіє потоками й compaction
- локальний CLI або daemon, який має транслювати нативні події плану/міркування/інструментів
- середовище виконання моделі, якому потрібен власний ідентифікатор відновлення на додачу до стенограми сесії OpenClaw

Не реєструйте обв’язку лише для додавання нового LLM API. Для звичайних HTTP- або WebSocket-API моделей створіть [Plugin постачальника](/uk/plugins/sdk-provider-plugins).

## Чим досі володіє ядро

Перш ніж буде вибрано обв’язку, OpenClaw уже розв’язав:

- постачальника і модель
- стан авторизації середовища виконання
- рівень мислення і бюджет контексту
- файл стенограми/сесії OpenClaw
- робочу область, sandbox і політику інструментів
- callback-и відповіді каналу та callback-и streaming
- політику fallback моделі й live перемикання моделей

Цей поділ навмисний. Обв’язка виконує підготовлену спробу; вона не вибирає постачальників, не замінює доставлення каналом і не перемикає моделі непомітно.

Підготовлена спроба також містить `params.runtimePlan`, пакет політик, яким володіє OpenClaw, для рішень середовища виконання, що мають залишатися спільними для PI та нативних обв’язок:

- `runtimePlan.tools.normalize(...)` і
  `runtimePlan.tools.logDiagnostics(...)` для політики схем інструментів з урахуванням постачальника
- `runtimePlan.transcript.resolvePolicy(...)` для санітизації стенограми й політики відновлення викликів інструментів
- `runtimePlan.delivery.isSilentPayload(...)` для спільного `NO_REPLY` і приглушення доставлення медіа
- `runtimePlan.outcome.classifyRunResult(...)` для класифікації fallback моделі
- `runtimePlan.observability` для розв’язаних метаданих постачальника/моделі/обв’язки

Обв’язки можуть використовувати план для рішень, які мають відповідати поведінці PI, але все одно повинні трактувати його як стан спроби, яким володіє host. Не змінюйте його й не використовуйте для перемикання постачальників/моделей усередині ходу.

## Зареєструвати обв’язку

**Імпорт:** `openclaw/plugin-sdk/agent-harness`

```typescript
import type { AgentHarness } from "openclaw/plugin-sdk/agent-harness";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const myHarness: AgentHarness = {
  id: "my-harness",
  label: "My native agent harness",

  supports(ctx) {
    return ctx.provider === "my-provider"
      ? { supported: true, priority: 100 }
      : { supported: false };
  },

  async runAttempt(params) {
    // Start or resume your native thread.
    // Use params.prompt, params.tools, params.images, params.onPartialReply,
    // params.onAgentEvent, and the other prepared attempt fields.
    return await runMyNativeTurn(params);
  },
};

export default definePluginEntry({
  id: "my-native-agent",
  name: "My Native Agent",
  description: "Runs selected models through a native agent daemon.",
  register(api) {
    api.registerAgentHarness(myHarness);
  },
});
```

## Політика вибору

OpenClaw вибирає обв’язку після розв’язання постачальника/моделі:

1. Перемагає записаний ідентифікатор обв’язки наявної сесії, тому зміни config/env не перемикають цю стенограму гаряче на інше середовище виконання.
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово вибирає зареєстровану обв’язку з цим ідентифікатором для сесій, які ще не закріплено.
3. `OPENCLAW_AGENT_RUNTIME=pi` примусово вибирає вбудовану обв’язку PI.
4. `OPENCLAW_AGENT_RUNTIME=auto` запитує зареєстровані обв’язки, чи підтримують вони розв’язаного постачальника/модель.
5. Якщо жодна зареєстрована обв’язка не підходить, OpenClaw використовує PI, якщо fallback PI не вимкнено.

Збої обв’язок Plugin відображаються як збої запуску. У режимі `auto` fallback PI використовується лише тоді, коли жодна зареєстрована обв’язка Plugin не підтримує розв’язаного постачальника/модель. Щойно обв’язка Plugin заявила запуск, OpenClaw не відтворює той самий хід через PI, бо це може змінити семантику авторизації/середовища виконання або дублювати побічні ефекти.

Ідентифікатор вибраної обв’язки зберігається разом з ідентифікатором сесії після вбудованого запуску. Застарілі сесії, створені до закріплення обв’язок, вважаються закріпленими за PI, щойно вони мають історію стенограми. Використовуйте нову/скинуту сесію під час перемикання між PI і нативною обв’язкою Plugin. `/status` показує ідентифікатори обв’язок, відмінні від типових, наприклад `codex`, поруч із `Fast`; PI приховано, бо це типовий шлях сумісності. Якщо вибрана обв’язка несподівана, увімкніть debug-логування `agents/harness` і перегляньте структурований запис Gateway `agent harness selected`. Він містить ідентифікатор вибраної обв’язки, причину вибору, політику середовища виконання/fallback і, у режимі `auto`, результат підтримки кожного кандидата Plugin.

Вбудований Plugin Codex реєструє `codex` як свій ідентифікатор обв’язки. Ядро трактує це як звичайний ідентифікатор обв’язки Plugin; специфічні для Codex alias-и належать до Plugin або операторського config, а не до спільного селектора середовища виконання.

## Пара постачальника й обв’язки

Більшість обв’язок також мають реєструвати постачальника. Постачальник робить посилання моделей, стан авторизації, метадані моделей і вибір `/model` видимими для решти OpenClaw. Потім обв’язка заявляє цього постачальника в `supports(...)`.

Вбудований Plugin Codex дотримується цього шаблону:

- бажані користувацькі посилання моделей: `openai/gpt-5.5` плюс
  `agentRuntime.id: "codex"`
- посилання сумісності: застарілі посилання `codex/gpt-*` залишаються прийнятними, але нові config не повинні використовувати їх як звичайні посилання постачальника/моделі
- ідентифікатор обв’язки: `codex`
- авторизація: синтетична доступність постачальника, бо обв’язка Codex володіє нативним login/сесією Codex
- запит app-server: OpenClaw надсилає Codex чистий ідентифікатор моделі й дозволяє обв’язці говорити нативним протоколом app-server

Plugin Codex є додатковим. Звичайні посилання `openai/gpt-*` продовжують використовувати нормальний шлях постачальника OpenClaw, якщо ви не примусите обв’язку Codex через `agentRuntime.id: "codex"`. Старіші посилання `codex/gpt-*` досі вибирають постачальника й обв’язку Codex для сумісності.

Для операторського налаштування, прикладів префіксів моделей і config лише для Codex див. [Обв’язка Codex](/uk/plugins/codex-harness).

OpenClaw потребує Codex app-server `0.125.0` або новішого. Plugin Codex перевіряє initialize handshake app-server і блокує старіші або неверсійовані сервери, щоб OpenClaw запускався лише проти тієї поверхні протоколу, з якою його протестовано. Мінімальна версія `0.125.0` містить підтримку нативного payload MCP hook, що з’явилася в Codex `0.124.0`, водночас закріплюючи OpenClaw за новішою протестованою стабільною лінією.

### Middleware результатів інструментів

Вбудовані Plugin можуть під’єднувати runtime-neutral middleware результатів інструментів через `api.registerAgentToolResultMiddleware(...)`, коли їхній manifest оголошує цільові ідентифікатори середовища виконання в `contracts.agentToolResultMiddleware`. Ця довірена точка розширення призначена для async-перетворень результатів інструментів, які мають виконуватися до того, як PI або Codex передає вихід інструмента назад у модель.

Застарілі вбудовані Plugin досі можуть використовувати `api.registerCodexAppServerExtensionFactory(...)` для middleware лише Codex app-server, але нові перетворення результатів повинні використовувати runtime-neutral API. Hook лише для Pi `api.registerEmbeddedExtensionFactory(...)` вилучено; перетворення результатів інструментів Pi мають використовувати runtime-neutral middleware.

### Класифікація кінцевого результату

Нативні обв’язки, які володіють власною проєкцією протоколу, можуть використовувати `classifyAgentHarnessTerminalOutcome(...)` з `openclaw/plugin-sdk/agent-harness-runtime`, коли завершений хід не створив видимого тексту assistant. Помічник повертає `empty`, `reasoning-only` або `planning-only`, щоб політика fallback OpenClaw могла вирішити, чи повторювати спробу на іншій моделі. Він навмисно не класифікує помилки prompt, ходи в процесі та навмисні тихі відповіді, як-от `NO_REPLY`.

### Нативний режим обв’язки Codex

Вбудована обв’язка `codex` — це нативний режим Codex для вбудованих ходів агента OpenClaw. Спочатку увімкніть вбудований Plugin `codex` і додайте `codex` до `plugins.allow`, якщо ваш config використовує обмежувальний allowlist. Нативні config app-server повинні використовувати `openai/gpt-*` з `agentRuntime.id: "codex"`. Використовуйте `openai-codex/*` для Codex OAuth через PI натомість. Застарілі посилання моделей `codex/*` залишаються alias-ами сумісності для нативної обв’язки.

Коли цей режим виконується, Codex володіє нативним ідентифікатором потоку, поведінкою відновлення, compaction і виконанням app-server. OpenClaw досі володіє чат-каналом, видимим дзеркалом стенограми, політикою інструментів, approvals, доставленням медіа та вибором сесії. Використовуйте `agentRuntime.id: "codex"` без override `fallback`, коли потрібно довести, що лише шлях Codex app-server може заявити запуск. Явні середовища виконання Plugin уже типово fail closed. Установлюйте `fallback: "pi"` лише тоді, коли навмисно хочете, щоб PI обробляв відсутній вибір обв’язки. Збої Codex app-server уже завершуються напряму, а не повторюються через PI.

## Вимкнути fallback PI

Типово OpenClaw запускає вбудованих агентів з `agents.defaults.agentRuntime`, установленим на `{ id: "auto", fallback: "pi" }`. У режимі `auto` зареєстровані обв’язки Plugin можуть заявити пару постачальник/модель. Якщо жодна не підходить, OpenClaw fallback-иться до PI.

У режимі `auto` установіть `fallback: "none"`, коли потрібно, щоб відсутній вибір обв’язки Plugin завершувався збоєм замість використання PI. Явні середовища виконання Plugin, як-от `agentRuntime.id: "codex"`, уже типово fail closed, якщо `fallback: "pi"` не встановлено в тому самому config або області env override. Збої вибраних обв’язок Plugin завжди завершуються жорстким збоєм. Це не блокує явний `agentRuntime.id: "pi"` або `OPENCLAW_AGENT_RUNTIME=pi`.

Для вбудованих запусків лише Codex:

```json
{
  "agents": {
    "defaults": {
      "model": "openai/gpt-5.5",
      "agentRuntime": {
        "id": "codex"
      }
    }
  }
}
```

Якщо ви хочете, щоб будь-яка зареєстрована обв’язка Plugin заявляла відповідні моделі, але не хочете, щоб OpenClaw непомітно fallback-ився до PI, залиште `runtime: "auto"` і вимкніть fallback:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto",
        "fallback": "none"
      }
    }
  }
}
```

Per-agent override-и використовують ту саму форму:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto",
        "fallback": "pi"
      }
    },
    "list": [
      {
        "id": "codex-only",
        "model": "openai/gpt-5.5",
        "agentRuntime": {
          "id": "codex",
          "fallback": "none"
        }
      }
    ]
  }
}
```

`OPENCLAW_AGENT_RUNTIME` досі override-ить налаштоване середовище виконання. Використовуйте `OPENCLAW_AGENT_HARNESS_FALLBACK=none`, щоб вимкнути fallback PI із середовища.

```bash
OPENCLAW_AGENT_RUNTIME=codex \
OPENCLAW_AGENT_HARNESS_FALLBACK=none \
openclaw gateway run
```

Коли fallback вимкнено, сесія завершується рано збоєм, якщо запитану обв’язку не зареєстровано, вона не підтримує розв’язаного постачальника/модель або завершується збоєм до створення побічних ефектів ходу. Це навмисно для розгортань лише Codex і для live-тестів, які мають довести, що шлях Codex app-server справді використовується.

Це налаштування керує лише обв’язкою вбудованого агента. Воно не вимикає маршрутизацію моделей для зображень, відео, музики, TTS, PDF або інших можливостей, специфічних для постачальника.

## Нативні сесії та дзеркало стенограми

Обв’язка може зберігати нативний ідентифікатор сесії, ідентифікатор потоку або resume token на боці daemon. Тримайте цю прив’язку явно пов’язаною із сесією OpenClaw і продовжуйте дзеркалити видимий для користувача вихід assistant/інструментів у стенограму OpenClaw.

Стенограма OpenClaw залишається шаром сумісності для:

- видимої в каналі історії сесії
- пошуку й індексації стенограм
- перемикання назад на вбудовану обв’язку PI в пізнішому ході
- загальної поведінки `/new`, `/reset` і видалення сесій

Якщо ваша обв’язка зберігає sidecar-прив’язку, реалізуйте `reset(...)`, щоб OpenClaw міг очистити її під час скидання власної сесії OpenClaw.

## Результати інструментів і медіа

Core формує список інструментів OpenClaw і передає його в підготовлену спробу.
Коли тестовий середовищний виконувач запускає динамічний виклик інструмента, повертайте результат інструмента через
форму результату цього середовищного виконувача, а не надсилайте медіа каналу самостійно.

Це зберігає текстові, графічні, відео-, музичні, TTS-, approval- та messaging-tool-виводи
на тому самому шляху доставки, що й запуски на базі PI.

## Поточні обмеження

- Публічний шлях імпорту є узагальненим, але деякі псевдоніми типів спроби/результату досі
  містять назви `Pi` для сумісності.
- Встановлення сторонніх середовищних виконувачів є експериментальним. Надавайте перевагу provider plugins,
  доки вам не знадобиться нативний runtime сеансу.
- Перемикання середовищних виконувачів підтримується між ходами. Не перемикайте середовищні виконувачі
  посеред ходу після того, як уже почалися native tools, approvals, текст асистента або надсилання повідомлень.
