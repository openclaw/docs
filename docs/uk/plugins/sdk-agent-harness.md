**Обв’язка агента** — це низькорівневий виконавець для одного підготовленого
ходу агента OpenClaw. Це не провайдер моделі, не канал і не реєстр інструментів.
Для користувацької ментальної моделі див. [Середовища виконання агентів](/uk/concepts/agent-runtimes).

Використовуйте цю поверхню лише для вбудованих або довірених нативних plugins. Контракт
усе ще експериментальний, бо типи параметрів навмисно віддзеркалюють поточний
вбудований виконавець.

## Коли використовувати обв’язку

Реєструйте обв’язку агента, коли сімейство моделей має власне нативне середовище
сеансу, а звичайний транспорт провайдера OpenClaw є неправильною абстракцією.

Приклади:

- нативний сервер агента для програмування, який володіє потоками й compaction
- локальний CLI або демон, який має потоково передавати нативні події плану/міркувань/інструментів
- середовище виконання моделі, якому потрібен власний ідентифікатор відновлення на додачу до
  транскрипту сеансу OpenClaw

**Не** реєструйте обв’язку лише для додавання нового LLM API. Для звичайних HTTP- або
WebSocket API моделей створюйте [Plugin провайдера](/uk/plugins/sdk-provider-plugins).

## Чим досі володіє ядро

До вибору обв’язки OpenClaw уже визначив:

- провайдера й модель
- стан автентифікації середовища виконання
- рівень міркування та бюджет контексту
- файл транскрипту/сеансу OpenClaw
- робочий простір, sandbox і політику інструментів
- колбеки відповідей каналу та колбеки потокової передачі
- політику fallback моделі та живого перемикання моделей

Цей поділ навмисний. Обв’язка виконує підготовлену спробу; вона не вибирає
провайдерів, не замінює доставку каналом і не перемикає моделі непомітно.

Підготовлена спроба також містить `params.runtimePlan`, набір політик, яким володіє OpenClaw,
для рішень середовища виконання, що мають лишатися спільними для PI й нативних
обв’язок:

- `runtimePlan.tools.normalize(...)` і
  `runtimePlan.tools.logDiagnostics(...)` для політики схем інструментів з урахуванням провайдера
- `runtimePlan.transcript.resolvePolicy(...)` для очищення транскрипту та
  політики виправлення викликів інструментів
- `runtimePlan.delivery.isSilentPayload(...)` для спільного `NO_REPLY` і приглушення
  доставки медіа
- `runtimePlan.outcome.classifyRunResult(...)` для класифікації fallback моделі
- `runtimePlan.observability` для визначених метаданих провайдера/моделі/обв’язки

Обв’язки можуть використовувати план для рішень, які мають збігатися з поведінкою PI, але
все одно повинні сприймати його як стан спроби, яким володіє хост. Не змінюйте його й не використовуйте його для
перемикання провайдерів/моделей усередині ходу.

## Реєстрація обв’язки

**Імпорт:** `openclaw/plugin-sdk/agent-harness`

```typescript
import type { AgentHarness } from "openclaw/plugin-sdk/agent-harness";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const myHarness: AgentHarness = {
  id: "my-harness",
  label: "My native agent harness",

  supports(ctx) {
    return ctx.provider === "my-provider"
      ? { supported: true, priority: 100 }
      : { supported: false };
  },

  async runAttempt(params) {
    // Start or resume your native thread.
    // Use params.prompt, params.tools, params.images, params.onPartialReply,
    // params.onAgentEvent, and the other prepared attempt fields.
    return await runMyNativeTurn(params);
  },
};

export default definePluginEntry({
  id: "my-native-agent",
  name: "My Native Agent",
  description: "Runs selected models through a native agent daemon.",
  register(api) {
    api.registerAgentHarness(myHarness);
  },
});
```

## Політика вибору

OpenClaw вибирає обв’язку після визначення провайдера/моделі:

1. Перемагає записаний ідентифікатор обв’язки наявного сеансу, щоб зміни config/env не
   перемикали цей транскрипт «на гарячу» на інше середовище виконання.
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово вибирає зареєстровану обв’язку з цим ідентифікатором для
   сеансів, які ще не закріплені.
3. `OPENCLAW_AGENT_RUNTIME=pi` примусово вибирає вбудовану обв’язку PI.
4. `OPENCLAW_AGENT_RUNTIME=auto` запитує зареєстровані обв’язки, чи підтримують вони
   визначеного провайдера/модель.
5. Якщо жодна зареєстрована обв’язка не збігається, OpenClaw використовує PI, якщо fallback PI
   не вимкнено.

Збої обв’язок plugin відображаються як збої запуску. У режимі `auto` fallback PI
використовується лише тоді, коли жодна зареєстрована обв’язка plugin не підтримує визначену
пару провайдер/модель. Щойно обв’язка plugin взяла запуск на себе, OpenClaw не
відтворює той самий хід через PI, бо це може змінити семантику автентифікації/середовища виконання
або продублювати побічні ефекти.

Ідентифікатор вибраної обв’язки зберігається разом з ідентифікатором сеансу після вбудованого запуску.
Застарілі сеанси, створені до закріплення обв’язок, вважаються закріпленими за PI після того, як
у них з’являється історія транскрипту. Використовуйте новий/скинутий сеанс, коли перемикаєтеся між PI і
нативною обв’язкою plugin. `/status` показує нестандартні ідентифікатори обв’язок, як-от `codex`,
поруч із `Fast`; PI прихований, бо це стандартний шлях сумісності.
Якщо вибрана обв’язка неочікувана, увімкніть debug-журналювання `agents/harness` і
перегляньте структурований запис `agent harness selected` у gateway. Він містить
ідентифікатор вибраної обв’язки, причину вибору, політику середовища виконання/fallback і, у
режимі `auto`, результат підтримки кожного кандидата plugin.

Вбудований plugin Codex реєструє `codex` як свій ідентифікатор обв’язки. Ядро сприймає його
як звичайний ідентифікатор обв’язки plugin; специфічні для Codex псевдоніми належать до plugin
або конфігурації оператора, а не до спільного селектора середовища виконання.

## Поєднання провайдера й обв’язки

Більшість обв’язок також мають реєструвати провайдера. Провайдер робить посилання на моделі,
стан автентифікації, метадані моделей і вибір `/model` видимими для решти
OpenClaw. Потім обв’язка заявляє цей провайдер у `supports(...)`.

Вбудований plugin Codex дотримується цього шаблону:

- бажані користувацькі посилання на моделі: `openai/gpt-5.5` плюс
  `agentRuntime.id: "codex"`
- сумісні посилання: застарілі посилання `codex/gpt-*` досі приймаються, але нові
  configs не повинні використовувати їх як звичайні посилання провайдер/модель
- ідентифікатор обв’язки: `codex`
- автентифікація: синтетична доступність провайдера, бо обв’язка Codex володіє
  нативним входом/сеансом Codex
- запит до app-server: OpenClaw надсилає Codex простий ідентифікатор моделі й дозволяє
  обв’язці говорити нативним протоколом app-server

Plugin Codex є додатковим. Звичайні посилання `openai/gpt-*` і далі використовують
стандартний шлях провайдера OpenClaw, якщо ви не примусите обв’язку Codex через
`agentRuntime.id: "codex"`. Старіші посилання `codex/gpt-*` досі вибирають
провайдера й обв’язку Codex для сумісності.

Для налаштування оператора, прикладів префіксів моделей і конфігурацій лише для Codex див.
[Обв’язка Codex](/uk/plugins/codex-harness).

OpenClaw потребує Codex app-server `0.125.0` або новішого. Plugin Codex перевіряє
рукостискання ініціалізації app-server і блокує старіші або неверсіоновані сервери, щоб
OpenClaw працював лише з поверхнею протоколу, з якою його протестовано. Мінімум
`0.125.0` містить підтримку payload нативного MCP hook, що з’явилася в
Codex `0.124.0`, водночас закріплюючи OpenClaw за новішою протестованою стабільною лінією.

### Посередник результатів інструментів

Вбудовані plugins можуть підключати нейтральний до середовища виконання посередник результатів інструментів через
`api.registerAgentToolResultMiddleware(...)`, коли їхній маніфест оголошує
цільові ідентифікатори середовищ виконання в `contracts.agentToolResultMiddleware`. Ця довірена
поверхня призначена для асинхронних перетворень результатів інструментів, які мають виконатися до того, як PI або Codex передасть
вивід інструмента назад у модель.

Застарілі вбудовані plugins досі можуть використовувати
`api.registerCodexAppServerExtensionFactory(...)` для посередника лише Codex app-server,
але нові перетворення результатів повинні використовувати нейтральний до середовища виконання API.
Hook `api.registerEmbeddedExtensionFactory(...)` лише для Pi було вилучено;
перетворення результатів інструментів Pi мають використовувати нейтральний до середовища виконання посередник.

### Класифікація кінцевого результату

Нативні обв’язки, які володіють власною проєкцією протоколу, можуть використовувати
`classifyAgentHarnessTerminalOutcome(...)` з
`openclaw/plugin-sdk/agent-harness-runtime`, коли завершений хід не створив
видимого тексту асистента. Допоміжна функція повертає `empty`, `reasoning-only` або
`planning-only`, щоб політика fallback OpenClaw могла вирішити, чи повторювати спробу на
іншій моделі. Вона навмисно залишає некласифікованими помилки промпта, ходи в процесі й
навмисні тихі відповіді, як-от `NO_REPLY`.

### Нативний режим обв’язки Codex

Вбудована обв’язка `codex` — це нативний режим Codex для вбудованих ходів агента
OpenClaw. Спочатку ввімкніть вбудований plugin `codex` і додайте `codex` до
`plugins.allow`, якщо ваша конфігурація використовує обмежувальний allowlist. Нативні конфігурації app-server
мають використовувати `openai/gpt-*`; ходи агента OpenAI за замовчуванням вибирають обв’язку Codex.
Застарілі маршрути `openai-codex/*` слід виправляти за допомогою
`openclaw doctor --fix`, а застарілі посилання на моделі `codex/*` лишаються сумісними
псевдонімами для нативної обв’язки.

Коли цей режим працює, Codex володіє нативним ідентифікатором потоку, поведінкою відновлення,
compaction і виконанням app-server. OpenClaw досі володіє каналом чату,
видимим дзеркалом транскрипту, політикою інструментів, approvals, доставкою медіа й вибором сеансу.
Використовуйте `agentRuntime.id: "codex"`, коли потрібно довести, що лише шлях
Codex app-server може взяти запуск на себе. Явні середовища виконання plugin завершуються закритою помилкою;
збої вибору Codex app-server і збої середовища виконання не повторюються через
PI.

## Суворість середовища виконання

За замовчуванням OpenClaw запускає вбудованих агентів з OpenClaw Pi. У режимі `auto`
зареєстровані обв’язки plugin можуть взяти пару провайдер/модель, а PI обробляє
хід, коли збігів немає. Використовуйте явне середовище виконання plugin, як-от
`agentRuntime.id: "codex"`, коли відсутність вибору обв’язки має завершуватися помилкою, а не
маршрутизуватися через PI. Збої вибраних обв’язок plugin завжди є жорсткими збоями. Це
не блокує явне `agentRuntime.id: "pi"` або
`OPENCLAW_AGENT_RUNTIME=pi`.

Для вбудованих запусків лише Codex:

```json
{
  "agents": {
    "defaults": {
      "model": "openai/gpt-5.5",
      "agentRuntime": {
        "id": "codex"
      }
    }
  }
}
```

Якщо ви хочете, щоб будь-яка зареєстрована обв’язка plugin брала відповідні моделі, а інакше
використовувався PI, задайте `id: "auto"`:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto"
      }
    }
  }
}
```

Перевизначення для окремих агентів використовують ту саму форму:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": { "id": "auto" }
    },
    "list": [
      {
        "id": "codex-only",
        "model": "openai/gpt-5.5",
        "agentRuntime": { "id": "codex" }
      }
    ]
  }
}
```

`OPENCLAW_AGENT_RUNTIME` досі перевизначає налаштоване середовище виконання.

```bash
OPENCLAW_AGENT_RUNTIME=codex openclaw gateway run
```

З явним середовищем виконання plugin сеанс завершується рано, коли запитана
обв’язка не зареєстрована, не підтримує визначену пару провайдер/модель або
завершується збоєм до створення побічних ефектів ходу. Це навмисно для розгортань
лише Codex і для live-тестів, які мають довести, що шлях Codex app-server
справді використовується.

Це налаштування керує лише вбудованою обв’язкою агента. Воно не вимикає
маршрутизацію моделей для зображень, відео, музики, TTS, PDF або інших специфічних для провайдера можливостей.

## Нативні сеанси й дзеркало транскрипту

Обв’язка може зберігати нативний ідентифікатор сеансу, ідентифікатор потоку або токен відновлення на боці демона.
Тримайте цю прив’язку явно пов’язаною із сеансом OpenClaw і продовжуйте
дзеркалити видимий користувачу вивід асистента/інструментів у транскрипт OpenClaw.

Транскрипт OpenClaw лишається шаром сумісності для:

- видимої в каналі історії сеансу
- пошуку та індексації транскриптів
- повернення до вбудованої обв’язки PI на пізнішому ході
- загальної поведінки `/new`, `/reset` і видалення сеансу

Якщо ваша обв’язка зберігає побічну прив’язку, реалізуйте `reset(...)`, щоб OpenClaw міг
очистити її під час скидання відповідного сеансу OpenClaw.

## Результати інструментів і медіа

Ядро формує список інструментів OpenClaw і передає його в підготовлену спробу.
Коли обв’язка виконує динамічний виклик інструмента, повертайте результат інструмента назад через
форму результату обв’язки, а не надсилайте медіа каналу самостійно.

Це утримує текст, зображення, відео, музику, TTS, approvals і виводи messaging-tool
на тому самому шляху доставки, що й запуски на базі PI.

## Поточні обмеження

- Публічний шлях імпорту є загальним, але деякі псевдоніми типів спроб/результатів досі
  містять назви `Pi` для сумісності.
- Встановлення стороннього harness є експериментальним. Віддавайте перевагу Plugin провайдерів,
  доки вам не знадобиться нативне середовище виконання сеансу.
- Перемикання harness підтримується між ходами. Не перемикайте harness у
  середині ходу після того, як почалися нативні інструменти, затвердження, текст асистента або надсилання
  повідомлень.
