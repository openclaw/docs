Нативна підтримка плагінів Codex дає агенту OpenClaw у режимі Codex змогу використовувати власні можливості застосунків і плагінів Codex
app-server у тому самому потоці Codex, який
обробляє хід OpenClaw.

OpenClaw не перетворює плагіни Codex на синтетичні динамічні інструменти `codex_plugin_*`
OpenClaw. Виклики плагінів залишаються в нативній стенограмі Codex, а
Codex app-server відповідає за виконання MCP, підкріплене застосунками.

Використовуйте цю сторінку після того, як базовий [harness Codex](/uk/plugins/codex-harness) уже працює.

## Вимоги

- Вибране середовище виконання агента OpenClaw має бути нативним harness Codex.
- `plugins.entries.codex.enabled` має бути true.
- `plugins.entries.codex.config.codexPlugins.enabled` має бути true.
- V1 підтримує лише плагіни `openai-curated`, які міграція виявила як
  установлені з джерела в домашньому каталозі Codex.
- Цільовий Codex app-server має бачити очікуваний маркетплейс,
  плагін та інвентар застосунків.

`codexPlugins` не впливає на запуски OpenClaw, звичайні запуски провайдера OpenAI, прив’язки розмов ACP
або інші harness, оскільки ці шляхи не створюють
потоки Codex app-server з нативною конфігурацією `apps`.

Доступ до Codex на боці OpenAI, доступність застосунків і робочі елементи керування застосунками/плагінами
походять з облікового запису Codex, у який виконано вхід. Про обліковий запис OpenAI та модель адміністрування
див. [Використання Codex із вашим планом ChatGPT](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan).

## Швидкий старт

Попередньо перегляньте міграцію з вихідного домашнього каталогу Codex:

```bash
openclaw migrate codex --dry-run
```

Використовуйте сувору перевірку вихідного застосунку, коли потрібно, щоб міграція перевірила доступність вихідного застосунку
перед плануванням нативної активації плагіна:

```bash
openclaw migrate codex --dry-run --verify-plugin-apps
```

Застосуйте міграцію, коли план виглядає правильно:

```bash
openclaw migrate apply codex --yes
```

Міграція записує явні записи `codexPlugins` для придатних плагінів і викликає
`plugin/install` Codex app-server для вибраних плагінів. Типова перенесена
конфігурація виглядає так:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

Після зміни `codexPlugins` нові розмови Codex автоматично підхоплюють оновлений набір застосунків.
Використовуйте `/new` або `/reset`, щоб оновити поточну розмову.
Перезапуск gateway не потрібен для змін увімкнення або вимкнення плагіна.

## Керування плагінами з чату

Використовуйте `/codex plugins`, коли потрібно переглянути або змінити налаштовані нативні плагіни Codex
з того самого чату, де ви використовуєте harness Codex:

```text
/codex plugins
/codex plugins list
/codex plugins disable google-calendar
/codex plugins enable google-calendar
```

`/codex plugins` є псевдонімом для `/codex plugins list`. Вивід списку показує
налаштовані ключі плагінів, стан увімкнено/вимкнено, назву плагіна Codex і маркетплейс
із `plugins.entries.codex.config.codexPlugins.plugins`.

`enable` і `disable` записують лише в конфігурацію OpenClaw за адресою
`~/.openclaw/openclaw.json`; вони не редагують `~/.codex/config.toml` і не встановлюють
нові плагіни Codex. Змінювати стан плагіна може лише власник або клієнт gateway з областю
`operator.admin`.

Увімкнення налаштованого плагіна також вмикає глобальний перемикач
`codexPlugins.enabled`. Якщо плагін було записано вимкненим, оскільки
міграція повернула `auth_required`, повторно авторизуйте застосунок у Codex перед увімкненням
його в OpenClaw.

## Як працює налаштування нативних плагінів

Інтеграція має три окремі стани:

- Установлено: Codex має локальний пакет плагіна в цільовому середовищі виконання app-server.
- Увімкнено: конфігурація OpenClaw дозволяє зробити плагін доступним для ходів
  harness Codex.
- Доступно: Codex app-server підтверджує, що записи застосунків плагіна доступні
  для активного облікового запису та можуть бути зіставлені з перенесеною ідентичністю плагіна.

Міграція є довговічним кроком установлення/визначення придатності. Під час планування OpenClaw
читає деталі `plugin/read` вихідного Codex і перевіряє, що відповідь облікового запису вихідного Codex
app-server є обліковим записом із підпискою ChatGPT. Відповіді без ChatGPT або
відсутні відповіді облікового запису пропускають плагіни, підкріплені застосунками, з
`codex_subscription_required`. За замовчуванням міграція не викликає вихідний
`app/list`; підкріплені застосунками вихідні плагіни, які проходять перевірку облікового запису, плануються
без перевірки доступності вихідного застосунку, а транспортні збої пошуку облікового запису
пропускаються з `codex_account_unavailable`. З `--verify-plugin-apps`
міграція робить свіжий знімок вихідного `app/list` і вимагає, щоб кожен належний застосунок
був присутній, увімкнений і доступний перед плануванням нативної активації. У
цьому режимі транспортні збої пошуку облікового запису переходять до перевірки інвентарю вихідних
застосунків. Інвентар застосунків часу виконання є перевіркою доступності цільового сеансу
після міграції. Потім налаштування сеансу harness Codex обчислює обмежувальну
конфігурацію застосунків потоку для увімкнених і доступних застосунків плагінів.

Конфігурація застосунків потоку обчислюється, коли OpenClaw встановлює сеанс harness Codex
або замінює застарілу прив’язку потоку Codex. Вона не переобчислюється на кожному ході, тому
`/codex plugins enable` і `/codex plugins disable` впливають на нові розмови Codex.
Використовуйте `/new` або `/reset`, коли поточна розмова має підхопити
оновлений набір застосунків.

## Межі підтримки V1

V1 навмисно вузька:

- Лише плагіни `openai-curated`, які вже були встановлені в інвентарі вихідного Codex
  app-server, придатні для міграції.
- Підкріплені застосунками вихідні плагіни мають пройти перевірку підписки під час міграції.
  `--verify-plugin-apps` додає перевірку інвентарю вихідних застосунків. Облікові записи, обмежені підпискою,
  а в режимі перевірки також недоступні, вимкнені, відсутні вихідні
  застосунки або збої оновлення інвентарю вихідних застосунків повідомляються як пропущені ручні
  елементи замість увімкнених записів конфігурації. Нечитабельні деталі плагіна пропускаються
  перед перевіркою інвентарю вихідних застосунків.
- Міграція записує явні ідентичності плагінів із `marketplaceName` і
  `pluginName`; вона не записує локальні шляхи кешу `marketplacePath`.
- `codexPlugins.enabled` є глобальним перемикачем увімкнення.
- Немає wildcard `plugins["*"]` і немає ключа конфігурації, який надає довільні
  повноваження на встановлення.
- Непідтримувані маркетплейси, кешовані пакети плагінів, хуки та файли конфігурації Codex
  зберігаються у звіті міграції для ручного перегляду.

## Інвентар застосунків і власність

OpenClaw читає інвентар застосунків Codex через `app/list` app-server, кешує його на
одну годину та асинхронно оновлює застарілі або відсутні записи. Кеш
існує лише в пам’яті; перезапуск CLI або gateway скидає його, і OpenClaw відбудовує його
з наступного читання `app/list`.

Міграція і час виконання використовують окремі ключі кешу:

- Перевірка вихідної міграції використовує вихідний домашній каталог Codex і параметри запуску вихідного app-server.
  Це виконується лише коли задано `--verify-plugin-apps`, і примусово
  запускає свіжий обхід вихідного `app/list` для цього планування.
- Налаштування цільового часу виконання використовує ідентичність Codex app-server цільового агента, коли
  будує конфігурацію застосунків потоку Codex. Активація плагіна інвалідовує цей цільовий
  ключ кешу, а потім примусово оновлює його після `plugin/install`.

Застосунок плагіна відкривається лише тоді, коли OpenClaw може зіставити його назад із перенесеним
плагіном через стабільну власність:

- точний ідентифікатор застосунку з деталей плагіна
- відома назва сервера MCP
- унікальні стабільні метадані

Власність лише за відображуваною назвою або неоднозначна власність виключається, доки наступне оновлення інвентарю
не доведе власність.

## Конфігурація застосунків потоку

OpenClaw ін’єктує обмежувальний патч `config.apps` для потоку Codex:
`_default` вимкнено, і ввімкнено лише застосунки, що належать увімкненим перенесеним плагінам.

OpenClaw задає `destructive_enabled` на рівні застосунку з ефективної глобальної або
поплагінної політики `allow_destructive_actions` і дозволяє Codex забезпечувати виконання
метаданих деструктивних інструментів із його нативних анотацій інструментів застосунку. `true`,
`"auto"` і `"always"` задають `destructive_enabled: true`; `false` задає
false. Конфігурація застосунку `_default` вимкнена з `open_world_enabled: false`.
Увімкнені застосунки плагінів виводяться з `open_world_enabled: true`; OpenClaw не
надає окремий перемикач політики open-world для плагіна і не підтримує
поплагінні списки заборони назв деструктивних інструментів.

Режим затвердження інструментів за замовчуванням автоматичний для застосунків плагінів, щоб недеструктивні
інструменти читання могли працювати без UI затвердження в тому самому потоці. Деструктивні інструменти залишаються
під контролем політики `destructive_enabled` кожного застосунку.

## Політика деструктивних дій

Деструктивні elicitations плагінів дозволені за замовчуванням для перенесених плагінів Codex,
водночас небезпечні схеми й неоднозначна власність усе ще завершуються відмовою:

- Глобальне `allow_destructive_actions` за замовчуванням має значення `true`.
- Поплагінне `allow_destructive_actions` перевизначає глобальну політику для цього
  плагіна.
- Коли політика дорівнює `false`, OpenClaw повертає детерміновану відмову.
- Коли політика дорівнює `true`, OpenClaw автоматично приймає лише безпечні схеми, які може зіставити
  з відповіддю затвердження, наприклад булеве поле approve.
- Коли політика дорівнює `"auto"`, OpenClaw відкриває деструктивні дії плагінів для
  Codex, але перетворює підтверджені власністю elicitations затвердження MCP на затвердження плагінів OpenClaw
  перед поверненням відповіді затвердження Codex.
- Коли політика дорівнює `"always"`, OpenClaw використовує той самий write/destructive
  gate Codex, що й `"auto"`, очищає довговічні перевизначення затвердження Codex для кожного інструмента
  застосунку перед запуском потоку та пропонує лише одноразове затвердження або відхилення, щоб
  довговічні затвердження не могли приглушити подальші запити дій запису.
- Відсутня ідентичність плагіна, неоднозначна власність, відсутній ідентифікатор ходу, неправильний ідентифікатор ходу
  або небезпечна схема elicitation призводить до відмови замість запиту.

## Усунення несправностей

**`auth_required`:** міграція встановила плагін, але один із його застосунків усе ще
потребує автентифікації. Явний запис плагіна записується вимкненим, доки ви
не авторизуєтеся повторно та не ввімкнете його.

**`app_inaccessible`, `app_disabled` або `app_missing`:**
міграція не встановила плагін, оскільки інвентар вихідних застосунків Codex не
показав, що всі належні застосунки присутні, увімкнені та доступні, коли
було задано `--verify-plugin-apps`. Повторно авторизуйте або ввімкніть застосунок у Codex, потім
повторно запустіть міграцію з `--verify-plugin-apps`.

**`app_inventory_unavailable`:** міграція не встановила плагін, оскільки
було запитано сувору перевірку вихідного застосунку, а оновлення інвентарю вихідних застосунків Codex
завершилося збоєм. Виправте доступ до вихідного Codex app-server або повторіть спробу без
`--verify-plugin-apps`, якщо приймаєте швидший план, обмежений перевіркою облікового запису.

**`codex_subscription_required`:** міграція не встановила підкріплений застосунком
плагін, оскільки в обліковий запис вихідного Codex app-server не було виконано вхід з
обліковим записом із підпискою ChatGPT. Увійдіть у застосунок Codex з автентифікацією підписки,
потім повторно запустіть міграцію.

**`codex_account_unavailable`:** міграція не встановила підкріплений застосунком плагін,
оскільки обліковий запис вихідного Codex app-server не вдалося прочитати. Виправте автентифікацію вихідного Codex
app-server або повторно запустіть із `--verify-plugin-apps`, якщо хочете, щоб інвентар вихідних застосунків
визначав придатність, коли пошук облікового запису завершується збоєм.

**`marketplace_missing` або `plugin_missing`:** цільовий Codex app-server
не бачить очікуваний маркетплейс або плагін `openai-curated`. Повторно запустіть міграцію
для цільового середовища виконання або перевірте стан плагінів Codex app-server.

**`app_inventory_missing` або `app_inventory_stale`:** готовність застосунку походила з
порожнього або застарілого кешу. OpenClaw планує асинхронне оновлення і виключає застосунки плагінів,
доки власність і готовність не стануть відомі.

**`app_ownership_ambiguous`:** інвентар застосунків збігся лише за відображуваною назвою, тому
застосунок не відкривається для потоку Codex.

**Конфігурацію змінено, але агент не бачить плагін:** використовуйте `/codex plugins
list`, щоб підтвердити налаштований стан, потім використовуйте `/new` або `/reset`. Наявні
прив’язки потоків Codex зберігають конфігурацію застосунків, з якою вони стартували, доки OpenClaw
не встановить новий сеанс harness або не замінить застарілу прив’язку.

**Деструктивну дію відхилено:** перевірте глобальне значення та значення для кожного Plugin
`allow_destructive_actions`. Навіть коли політика має значення true, `"auto"` або
`"always"`, небезпечні схеми elicitation і неоднозначна ідентичність Plugin усе одно завершуються
закритою відмовою.
