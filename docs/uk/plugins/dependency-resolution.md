OpenClaw виконує роботу із залежностями Plugin під час встановлення/оновлення. Завантаження під час виконання
не запускає менеджери пакетів, не відновлює дерева залежностей і не змінює каталог
пакета OpenClaw.

## Розподіл відповідальності

Пакети Plugin відповідають за власний граф залежностей:

- залежності середовища виконання містяться в `dependencies` або
  `optionalDependencies` пакета Plugin
- імпорти SDK/ядра є peer-залежностями або імпортами, наданими OpenClaw
- локальні Plugin для розробки надають власні вже встановлені залежності
- npm- і git-Plugin встановлюються в корені пакетів, якими керує OpenClaw

OpenClaw відповідає лише за життєвий цикл Plugin:

- виявити джерело Plugin
- встановити або оновити пакет за явним запитом
- записати метадані встановлення
- завантажити точку входу Plugin
- завершитися з дієвою помилкою, коли залежностей бракує

## Корені встановлення

OpenClaw використовує стабільні корені для кожного джерела:

- npm-пакети встановлюються в окремі проєкти для кожного Plugin у
  `~/.openclaw/npm/projects/<encoded-package>`
- git-пакети клонуються в `~/.openclaw/git`
- локальні/path/archive-встановлення копіюються або посилаються без відновлення залежностей

npm-встановлення запускаються в цьому корені проєкту окремого Plugin за допомогою:

```bash
cd ~/.openclaw/npm/projects/<encoded-package>
npm install --omit=dev --omit=peer --legacy-peer-deps --ignore-scripts --no-audit --no-fund
```

`openclaw plugins install npm-pack:<path.tgz>` використовує той самий корінь npm-проєкту
окремого Plugin для локального tarball npm-pack. OpenClaw читає npm-метадані tarball,
додає його до керованого проєкту як скопійовану залежність `file:`, запускає
звичайне npm-встановлення, а потім перевіряє метадані встановленого lockfile, перш ніж
довіряти Plugin.
Це призначено для перевірки прийнятності пакета та кандидатів на випуск, коли
локальний артефакт pack має поводитися як артефакт реєстру, який він імітує.

npm може підіймати транзитивні залежності до `node_modules` проєкту окремого Plugin
поруч із пакетом Plugin. OpenClaw сканує корінь керованого проєкту перед тим, як
довіряти встановленню, і видаляє цей проєкт під час видалення, тому підняті
залежності середовища виконання залишаються в межах очищення цього Plugin.

Опубліковані npm-пакети Plugin можуть постачати `npm-shrinkwrap.json`. npm використовує цей
публіковний lockfile під час встановлення, а корінь керованого npm-проєкту OpenClaw
підтримує його через звичайний шлях npm-встановлення. Публіковні пакети Plugin,
якими володіє OpenClaw, повинні містити локальний для пакета shrinkwrap, згенерований із
опублікованого графа залежностей цього пакета Plugin:

```bash
pnpm deps:shrinkwrap:generate
pnpm deps:shrinkwrap:check
```

Генератор вилучає `devDependencies` Plugin, застосовує політику перевизначень робочої області
та записує `extensions/<id>/npm-shrinkwrap.json` для кожного Plugin
`publishToNpm`. Сторонні пакети Plugin також можуть постачати shrinkwrap;
OpenClaw не вимагає цього для пакетів спільноти, але npm врахує його
за наявності.

npm-пакети Plugin, якими володіє OpenClaw, також можуть публікуватися з явними
`bundledDependencies`. Шлях npm-публікації накладає список назв залежностей
середовища виконання, вилучає dev-only метадані робочої області з опублікованого
маніфесту пакета, запускає npm-встановлення без скриптів для локальних для пакета
залежностей середовища виконання, а потім пакує або публікує tarball Plugin із цими
файлами залежностей усередині. Пакети з великою кількістю нативних компонентів, зокрема
середовища виконання Codex і ACP, відмовляються від цього через
`openclaw.release.bundleRuntimeDependencies: false`; ці пакети все одно
постачають свій shrinkwrap, але npm розв'язує залежності середовища виконання під час встановлення,
замість того щоб вбудовувати кожен платформний бінарний файл у tarball Plugin. Кореневий
пакет `openclaw` не вбудовує повне дерево своїх залежностей.

Plugin, які імпортують `openclaw/plugin-sdk/*`, оголошують `openclaw` як peer-залежність.
OpenClaw не дозволяє npm встановлювати окрему копію пакета хоста з реєстру
в керований проєкт, бо застарілі пакети хоста можуть впливати на npm-розв'язання
peer-залежностей усередині цього Plugin. Керовані npm-встановлення пропускають npm
розв'язання/матеріалізацію peer-залежностей, а OpenClaw після встановлення або оновлення
повторно закріплює локальні для Plugin посилання `node_modules/openclaw` для встановлених
пакетів, які оголошують peer-залежність від хоста.

git-встановлення клонують або оновлюють репозиторій, а потім запускають:

```bash
npm install --omit=dev --ignore-scripts --no-audit --no-fund
```

Після цього встановлений Plugin завантажується з каталогу цього пакета, тож розв'язання
локальних для пакета та батьківських `node_modules` працює так само, як і для звичайного
пакета Node.

## Локальні Plugin

Локальні Plugin вважаються каталогами, керованими розробником. OpenClaw не запускає
`npm install`, `pnpm install` або відновлення залежностей для них. Якщо локальний
Plugin має залежності, встановіть їх у цьому Plugin перед завантаженням.

Сторонні локальні Plugin TypeScript можуть використовувати аварійний шлях Jiti. Запаковані
JavaScript Plugin і вбудовані внутрішні Plugin завантажуються через нативний
import/require замість Jiti.

## Запуск і перезавантаження

Запуск Gateway і перезавантаження конфігурації ніколи не встановлюють залежності Plugin. Вони читають
записи встановлення Plugin, обчислюють точку входу та завантажують її.

Якщо залежність відсутня під час виконання, Plugin не завантажується, а помилка
має вказати оператору явне виправлення:

```bash
openclaw plugins update <id>
openclaw plugins install <source>
openclaw doctor --fix
```

`doctor --fix` може очистити застарілий стан залежностей, згенерований OpenClaw, і відновити
завантажувані Plugin, яких бракує в локальних записах встановлення, коли конфігурація
посилається на них. Doctor не відновлює залежності для вже встановленого
локального Plugin.

## Вбудовані Plugin

Легкі та критично важливі для ядра вбудовані Plugin постачаються як частина OpenClaw.
Вони або не повинні мати важкого дерева залежностей середовища виконання, або мають бути винесені в
завантажуваний пакет на ClawHub/npm.

Поточний згенерований список Plugin, які постачаються в пакеті ядра, встановлюються
зовнішньо або залишаються лише у вихідному коді, див. в [інвентарі Plugin](/uk/plugins/plugin-inventory).

Маніфести вбудованих Plugin не повинні запитувати підготовку залежностей. Велику або необов'язкову
функціональність Plugin слід пакувати як звичайний Plugin і встановлювати через
той самий шлях npm/git/ClawHub, що й сторонні Plugin.

У checkout вихідного коду OpenClaw розглядає репозиторій як pnpm-монорепозиторій. Після
`pnpm install` вбудовані Plugin завантажуються з `extensions/<id>`, тож локальні для пакета
залежності робочої області доступні, а зміни підхоплюються напряму. Розробка в
checkout вихідного коду підтримує лише pnpm; звичайний `npm install` у корені репозиторію
не є підтримуваним способом підготувати залежності вбудованих Plugin.

| Форма встановлення              | Розташування вбудованого Plugin       | Власник залежностей                                                  |
| -------------------------------- | ------------------------------------- | -------------------------------------------------------------------- |
| `npm install -g openclaw`        | Побудоване дерево середовища виконання всередині пакета | Пакет OpenClaw і явні потоки встановлення/оновлення/doctor для Plugin |
| Git checkout плюс `pnpm install` | Пакети робочої області `extensions/<id>` | Робоча область pnpm, включно з власними залежностями кожного пакета Plugin |
| `openclaw plugins install ...`   | Керований корінь npm-проєкту/git/ClawHub | Потік встановлення/оновлення Plugin                                  |

## Очищення застарілого стану

Старіші версії OpenClaw генерували корені залежностей вбудованих Plugin під час запуску або
під час відновлення doctor. Поточне очищення doctor видаляє ці застарілі каталоги та
симлінки, коли використовується `--fix`, включно зі старими коренями `plugin-runtime-deps`, глобальними
симлінками пакетів Node-prefix, які вказують на обрізані цілі `plugin-runtime-deps`,
маніфестами `.openclaw-runtime-deps*`, згенерованими `node_modules` Plugin, каталогами
етапу встановлення та локальними для пакета сховищами pnpm. Запакований postinstall також
видаляє ці глобальні симлінки перед обрізанням застарілих цільових коренів, щоб оновлення
не залишали пошкоджених імпортів пакетів ESM.

Старіші npm-встановлення також використовували спільний корінь `~/.openclaw/npm/node_modules`.
Поточні потоки встановлення, оновлення, видалення та doctor досі розпізнають цей застарілий
плаский корінь лише для відновлення та очищення. Нові npm-встановлення мають створювати
корені проєктів для окремих Plugin.
