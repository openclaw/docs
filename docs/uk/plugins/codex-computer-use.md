Computer Use — це нативний для Codex MCP-плагін для локального керування робочим столом. OpenClaw
не постачає застосунок для робочого столу, не виконує дії на робочому столі самостійно й не обходить
дозволи Codex. Вбудований плагін `codex` лише готує сервер застосунку Codex:
він вмикає підтримку плагінів Codex, знаходить або встановлює налаштований плагін Computer Use,
перевіряє доступність MCP-сервера `computer-use`, а потім дозволяє
Codex керувати нативними викликами інструментів MCP під час ходів у режимі Codex.

Використовуйте цю сторінку, коли OpenClaw уже використовує нативне середовище Codex. Налаштування
самого середовища описано в розділі [Середовище Codex](/uk/plugins/codex-harness).

Це відрізняється від вбудованого в OpenClaw [інструмента керування комп’ютером на основі вузла](/uk/nodes/computer-use). Використовуйте вбудований інструмент, коли той самий контракт агента має керувати спареним Mac незалежно від того, чи працює агент на Gateway або іншому вузлі. Використовуйте Codex Computer Use, коли сервер застосунку Codex має керувати локальним встановленням MCP, дозволами та нативними викликами інструментів.

## OpenClaw.app і Peekaboo

Інтеграція Peekaboo в OpenClaw.app не пов’язана з Codex Computer Use. Застосунок
macOS може розміщувати сокет PeekabooBridge, щоб CLI `peekaboo` міг повторно використовувати
локальні дозволи застосунку на Accessibility і Screen Recording для власних
засобів автоматизації Peekaboo. Цей міст не встановлює та не проксіює Codex Computer Use, а
Codex Computer Use не виконує виклики через сокет PeekabooBridge.

Використовуйте [міст Peekaboo](/uk/platforms/mac/peekaboo), коли потрібно, щоб OpenClaw.app був
хостом для автоматизації CLI Peekaboo з урахуванням дозволів. Використовуйте цю сторінку, коли
агент OpenClaw у режимі Codex повинен мати доступ до нативного MCP-плагіна `computer-use` від Codex
до початку ходу.

## Застосунок iOS

Застосунок iOS не пов’язаний із Codex Computer Use. Він не встановлює та не проксіює
MCP-сервер Codex `computer-use` і не є бекендом керування робочим столом.
Натомість застосунок iOS підключається як вузол OpenClaw і надає мобільні
можливості через команди вузла, як-от `canvas.*`, `camera.*`, `screen.*`,
`location.*` і `talk.*`.

Використовуйте [iOS](/uk/platforms/ios), коли потрібно, щоб агент керував вузлом iPhone
через Gateway. Використовуйте цю сторінку, коли агент у режимі Codex має керувати
локальним робочим столом macOS через нативний плагін Computer Use від Codex.

## Прямий MCP cua-driver

Codex Computer Use — не єдиний спосіб надати керування робочим столом. Якщо потрібно,
щоб середовища виконання під керуванням OpenClaw безпосередньо викликали драйвер TryCua, використовуйте вхідний
сервер `cua-driver mcp` через реєстр MCP OpenClaw замість
потоку маркетплейсу, специфічного для Codex.

Після встановлення `cua-driver` або попросіть його надати команду OpenClaw:

```bash
cua-driver mcp-config --client openclaw
```

або зареєструйте stdio-сервер безпосередньо:

```bash
openclaw mcp set cua-driver '{"command":"cua-driver","args":["mcp"]}'
```

Цей шлях зберігає незмінною поверхню інструментів вхідного MCP, зокрема схеми драйвера
та структуровані відповіді MCP. Використовуйте його, коли потрібно, щоб драйвер CUA
був доступний як звичайний MCP-сервер OpenClaw. Використовуйте налаштування Codex Computer Use
на цій сторінці, коли сервер застосунку Codex має керувати встановленням плагіна, перезавантаженням MCP
і нативними викликами інструментів усередині ходів у режимі Codex.

Драйвер CUA призначений лише для macOS і все одно потребує локальних дозволів macOS,
які запитує його застосунок, як-от Accessibility і Screen Recording. OpenClaw не
встановлює `cua-driver`, не надає ці дозволи й не обходить модель безпеки
вхідного драйвера.

## Швидке налаштування

Задайте `plugins.entries.codex.config.computerUse`, якщо для ходів у режимі Codex
Computer Use має бути доступний до початку потоку. `autoInstall: true` вмикає
Computer Use і дозволяє OpenClaw встановити або повторно ввімкнути його перед ходом:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          computerUse: {
            autoInstall: true,
          },
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

За цієї конфігурації OpenClaw перевіряє сервер застосунку Codex перед кожним ходом
у режимі Codex. Якщо Computer Use відсутній, але сервер застосунку Codex уже виявив
маркетплейс, із якого можна встановити плагін, OpenClaw просить сервер застосунку Codex встановити або
повторно ввімкнути плагін і перезавантажити MCP-сервери. У macOS, якщо відповідний
маркетплейс не зареєстровано, але існує стандартний пакет застосунку для робочого столу, OpenClaw
також намагається зареєструвати вбудований маркетплейс Codex із
`/Applications/ChatGPT.app/Contents/Resources/plugins/openai-bundled`, зберігаючи
`/Applications/Codex.app/Contents/Resources/plugins/openai-bundled`
як резервний варіант для застарілих окремих інсталяцій. Якщо після налаштування
MCP-сервер усе ще недоступний, хід завершується помилкою до початку потоку.

Після зміни конфігурації Computer Use скористайтеся `/new` або `/reset` у відповідному
чаті перед тестуванням, якщо наявний потік Codex уже запущено.

У macOS керований запуск Computer Use надає перевагу виконуваному файлу застосунку для робочого столу за шляхом
`/Applications/ChatGPT.app/Contents/Resources/codex`, а потім
переходить до `/Applications/Codex.app/Contents/Resources/codex` для застарілих
окремих інсталяцій. Це також стосується одноразових команд перевірки стану та
встановлення Computer Use, які запускають власний клієнт. Завдяки цьому керування робочим столом
залишається під контролем пакета застосунку, якому належать локальні дозволи macOS. Якщо застосунок для робочого столу
не встановлено, OpenClaw переходить до керованого виконуваного файлу Codex, установленого поруч із
плагіном. Звичайні керовані ходи Codex із типовою ізольованою домашньою директорією агента спочатку надають перевагу
цьому закріпленому пакету, щоб старіший застосунок для робочого столу не міг перекрити актуальну підтримку
моделей. Домашні директорії користувача й надалі надають перевагу застосунку для робочого столу, оскільки можуть завантажувати нативний
стан Computer Use. Ізольована домашня директорія агента, в ефективній конфігурації Codex якої ввімкнено
Computer Use, також надає перевагу застосунку для робочого столу. Явна конфігурація
`appServer.command` або `OPENCLAW_CODEX_APP_SERVER_BIN` усе одно має пріоритет над
цим керованим вибором.

OpenClaw послідовно виконує читання нативної конфігурації Codex і встановлення Computer Use
в межах одного запущеного Gateway. Окремий процес Codex або інший Gateway не
охоплюється цим блокуванням. Після зміни нативної конфігурації плагіна Codex поза
Gateway перезапустіть Gateway і почніть новий чат, перш ніж покладатися на новий
вибір.

## Команди

Використовуйте команди `/codex computer-use` з будь-якої поверхні чату, де
доступна поверхня команд плагіна `codex`. Це команди чату/середовища виконання
OpenClaw, а не підкоманди CLI `openclaw codex ...`:

```text
/codex computer-use status
/codex computer-use install
/codex computer-use install --source <marketplace-source>
/codex computer-use install --marketplace-path <path>
/codex computer-use install --marketplace <name>
```

`status` — типова дія лише для читання: вона не додає джерела маркетплейсів,
не встановлює плагіни та не вмикає підтримку плагінів Codex. Якщо жодна конфігурація не
вмикає Computer Use, `status` може повідомляти про вимкнений стан навіть після одноразової команди
встановлення.

`install` вмикає підтримку плагінів сервера застосунку Codex, за потреби додає
налаштоване джерело маркетплейсу, встановлює або повторно вмикає налаштований плагін
через сервер застосунку Codex, перезавантажує MCP-сервери та перевіряє, що MCP-сервер
надає інструменти. Оскільки встановлення змінює довірені ресурси хоста,
запускати `install` може лише власник або клієнт Gateway `operator.admin`. Інші
авторизовані відправники можуть і надалі використовувати команду `status` лише для читання,
зокрема з перевизначеннями.

Старіші випуски приймали одноразові перевизначення ідентичності `--plugin`, `--server` і `--mcp-server`.
Натомість налаштуйте `computerUse.pluginName` і
`computerUse.mcpServerName` постійно. Коли використовується застарілий прапорець ідентичності,
команда вказує точний параметр, який потрібно зберегти, і повторює
запитану дію разом з усіма підтримуваними прапорцями маркетплейсу у своїх інструкціях із міграції.

## Варіанти маркетплейсу

OpenClaw використовує той самий API сервера застосунку, який надає сам Codex.
Поля маркетплейсу визначають, де Codex має шукати `computer-use`.

| Поле                 | Коли використовувати                                              | Підтримка встановлення                                   |
| -------------------- | ----------------------------------------------------------------- | -------------------------------------------------------- |
| Без поля маркетплейсу | Потрібно, щоб сервер застосунку Codex використовував уже відомі йому маркетплейси. | Так, коли сервер застосунку повертає локальний маркетплейс. |
| `marketplaceSource`  | Є джерело маркетплейсу Codex, яке сервер застосунку може додати.  | Так, для явної команди `/codex computer-use install`.    |
| `marketplacePath`    | Локальний шлях до файлу маркетплейсу на хості вже відомий.        | Так, для явного встановлення й автоматичного встановлення на початку ходу. |
| `marketplaceName`    | Потрібно вибрати за назвою вже зареєстрований маркетплейс.        | Так, лише коли вибраний маркетплейс має локальний шлях.   |

Новим домашнім директоріям Codex може знадобитися трохи часу для початкового заповнення офіційних
маркетплейсів. Під час установлення OpenClaw опитує `plugin/list` протягом щонайбільше
`marketplaceDiscoveryTimeoutMs` мілісекунд (типово 60 секунд).

Якщо Computer Use міститься в кількох відомих маркетплейсах, OpenClaw надає перевагу
`openai-bundled`, потім `openai-curated`, а потім `local`. Невідомі неоднозначні
збіги призводять до безпечної відмови із запитом задати `marketplaceName` або
`marketplacePath`.

## Вбудований маркетплейс macOS

Поточні збірки ChatGPT для робочого столу містять Computer Use за цим шляхом; застарілі окремі
збірки Codex для робочого столу використовують таку саму структуру в `Codex.app`:

```text
/Applications/ChatGPT.app/Contents/Resources/plugins/openai-bundled/plugins/computer-use
/Applications/Codex.app/Contents/Resources/plugins/openai-bundled/plugins/computer-use
```

Коли `computerUse.autoInstall` має значення true і жоден маркетплейс, що містить
`computer-use`, не зареєстровано, OpenClaw намагається додати перший наявний стандартний
корінь вбудованого маркетплейсу:

```text
/Applications/ChatGPT.app/Contents/Resources/plugins/openai-bundled
/Applications/Codex.app/Contents/Resources/plugins/openai-bundled
```

Його також можна явно зареєструвати з оболонки за допомогою Codex:

```bash
codex plugin marketplace add /Applications/ChatGPT.app/Contents/Resources/plugins/openai-bundled
```

Якщо використовується нестандартний шлях до застосунку Codex, один раз виконайте `/codex computer-use install
--source <marketplace-root>` або задайте в `computerUse.marketplacePath`
локальний шлях до файлу маркетплейсу. Використовуйте `--marketplace-path` лише за наявності
шляху до JSON-файлу маркетплейсу, а не кореня вбудованого маркетплейсу.

### Спільний кеш плагінів

Типове значення `pluginCacheMode: "independent"` залишає кожну домашню директорію Codex та її
кеш плагінів некерованими. Задайте `pluginCacheMode: "shared"`, щоб скопіювати вбудований
плагін Computer Use до доступного для виявлення кешу плагінів активної домашньої директорії Codex
перед запуском сервера застосунку. Спільний режим зберігає старіші кешовані версії, оскільки
запущені клієнти Codex усе ще можуть посилатися на свої каталогізовані за версіями директорії плагінів;
у разі невдалого копіювання заміни активний кеш також зберігається. Явна
конфігурація `marketplaceName` або `marketplacePath` вимикає цю
синхронізацію, щоб OpenClaw не перевизначав цей вибір.

## Обмеження віддаленого каталогу

Сервер застосунку Codex може перелічувати й читати записи каталогу, доступні лише віддалено, але наразі
не підтримує віддалений `plugin/install`. Це означає, що `marketplaceName`
може вибрати маркетплейс, доступний лише віддалено, для перевірок стану, але для встановлення й
повторного ввімкнення все одно потрібен локальний маркетплейс через `marketplaceSource` або
`marketplacePath`.

Якщо стан повідомляє, що плагін доступний у віддаленому маркетплейсі Codex, але
віддалене встановлення не підтримується, запустіть встановлення з локальним джерелом або шляхом:

```text
/codex computer-use install --source <marketplace-source>
/codex computer-use install --marketplace-path <path>
```

## Довідник із конфігурації

| Поле                            | Типове значення | Значення                                                                                                 |
| ------------------------------- | --------------- | -------------------------------------------------------------------------------------------------------- |
| `enabled`                       | визначається    | Вимагати Computer Use. Типово має значення true, якщо задано інше поле Computer Use.                     |
| `autoInstall`                   | false           | Установити або повторно ввімкнути з уже виявлених маркетплейсів на початку ходу.                          |
| `marketplaceDiscoveryTimeoutMs` | 60000           | Як довго встановлення очікує виявлення маркетплейсу сервером застосунку Codex.                            |
| `liveTestTimeoutMs`             | 60000           | Час очікування тимчасового потоку перевірки готовності та його запитів на очищення.                       |
| `toolCallTimeoutMs`             | 60000           | Час очікування виклику інструмента готовності Computer Use `list_apps`.                                  |
| `healthCheckEnabled`            | false           | Виконувати періодичні перевірки готовності, поки активний клієнт сервера застосунку-власника.             |
| `healthCheckIntervalMinutes`    | 60              | Періодичність перевірок; допустимі значення: 30, 60, 120 або 240 хвилин.                                  |
| `pluginCacheMode`               | `independent`   | Використовуйте `shared`, щоб оновлювати кеш домашнього каталогу Codex із вбудованого настільного Plugin.  |
| `strictReadiness`               | false           | Зупинити запуск після невдалої оперативної перевірки замість продовження з попередженням.                 |
| `autoRepair`                    | false           | Завершити застарілі дочірні процеси MCP Computer Use у відповідній області та один раз повторити перевірку. |
| `marketplaceSource`             | не задано       | Рядок джерела, переданий у `marketplace/add` сервера застосунку Codex.                                   |
| `marketplacePath`               | не задано       | Локальний шлях до файлу маркетплейсу Codex, що містить Plugin.                                           |
| `marketplaceName`               | не задано       | Назва зареєстрованого маркетплейсу Codex для вибору.                                                     |
| `pluginName`                    | `computer-use`  | Назва Plugin у маркетплейсі Codex.                                                                       |
| `mcpServerName`                 | `computer-use`  | Назва сервера MCP, яку надає встановлений Plugin.                                                        |

Автоматичне встановлення на початку ходу навмисно відхиляє налаштовані
значення `marketplaceSource`. Додавання нового джерела є явною операцією
налаштування, тому один раз виконайте
`/codex computer-use install --source <marketplace-source>`, а надалі дозвольте
`autoInstall` виконувати повторне ввімкнення з виявлених локальних маркетплейсів.
Автоматичне встановлення на початку ходу може використовувати налаштований
`marketplacePath`, оскільки це вже локальний шлях на хості.

Для кожного поля також можна задати перевизначення змінною середовища, яке
перевіряється, якщо відповідний ключ конфігурації не задано:

| Поле                            | Змінна середовища                                              |
| ------------------------------- | -------------------------------------------------------------- |
| `enabled`                       | `OPENCLAW_CODEX_COMPUTER_USE`                                  |
| `autoInstall`                   | `OPENCLAW_CODEX_COMPUTER_USE_AUTO_INSTALL`                     |

## Що перевіряє OpenClaw

OpenClaw внутрішньо повідомляє стабільну причину стану налаштування та форматує
видимий користувачеві стан для чату:

| Причина                      | Значення                                                        | Наступний крок                                      |
| ---------------------------- | --------------------------------------------------------------- | -------------------------------------------------- |
| `disabled`                   | `computerUse.enabled` визначено як false.                        | Задайте `enabled` або інше поле Computer Use.      |
| `marketplace_missing`        | Відповідний маркетплейс недоступний.                             | Налаштуйте джерело, шлях або назву маркетплейсу.   |
| `plugin_not_installed`       | Маркетплейс існує, але Plugin не встановлено.                    | Запустіть установлення або ввімкніть `autoInstall`. |
| `plugin_disabled`            | Plugin установлено, але вимкнено в конфігурації Codex.          | Запустіть установлення, щоб повторно його ввімкнути. |
| `remote_install_unsupported` | Вибраний маркетплейс доступний лише віддалено.                   | Використайте `marketplaceSource` або `marketplacePath`. |
| `mcp_missing`                | Plugin увімкнено, але сервер MCP недоступний.                    | Перевірте Computer Use у Codex і дозволи ОС.       |
| `ready`                      | Plugin та інструменти MCP доступні.                              | Почніть хід у режимі Codex.                        |
| `check_failed`               | Запит сервера застосунку Codex завершився помилкою під час перевірки стану. | Перевірте підключення до сервера застосунку та журнали. |
| `auto_install_blocked`       | Налаштування на початку ходу потребувало б додавання нового джерела. | Спочатку запустіть явне встановлення.              |

Вивід чату містить стан Plugin, стан сервера MCP, маркетплейс, доступні
інструменти та конкретне повідомлення для невдалого етапу налаштування.

## Дозволи macOS

Computer Use працює лише в macOS. Серверу MCP, яким керує Codex, можуть
знадобитися локальні дозволи ОС, перш ніж він зможе перевіряти застосунки або
керувати ними. Якщо OpenClaw повідомляє, що Computer Use установлено, але
сервер MCP недоступний, спочатку перевірте налаштування Computer Use на боці
Codex:

- Сервер застосунку Codex працює на тому самому хості, де має відбуватися
  керування робочим столом.
- Plugin Computer Use увімкнено в конфігурації Codex.
- Сервер MCP `computer-use` відображається у стані MCP сервера застосунку Codex.
- macOS надала потрібні дозволи застосунку для керування робочим столом.
- Поточний сеанс хоста має доступ до робочого столу, яким здійснюється керування.

OpenClaw навмисно припиняє роботу в разі помилки, коли `computerUse.enabled`
має значення true. Хід у режимі Codex не повинен непомітно продовжуватися без
нативних інструментів робочого столу, яких вимагає конфігурація.

## Усунення несправностей

**Стан повідомляє, що Plugin не встановлено.** Запустіть
`/codex computer-use install`. Якщо маркетплейс не виявлено, передайте
`--source` або `--marketplace-path`.

**Стан повідомляє, що Plugin установлено, але вимкнено.** Знову запустіть
`/codex computer-use install`. Установлення через сервер застосунку Codex
записує в конфігурацію Plugin увімкнений стан.

**Стан повідомляє, що віддалене встановлення не підтримується.** Використайте
локальне джерело або шлях маркетплейсу. Записи каталогу, доступні лише
віддалено, можна переглядати, але не встановлювати через поточний API сервера
застосунку.

**Стан повідомляє, що сервер MCP недоступний.** Один раз повторно запустіть
установлення, щоб сервери MCP перезавантажилися. Якщо він залишається
недоступним, виправте застосунок Codex Computer Use, стан MCP сервера
застосунку Codex або дозволи macOS.

**Перевірка стану або готовності перевищує час очікування на
`computer-use.list_apps`.** Plugin і сервер MCP наявні, але локальний міст
Computer Use не відповів. Завершіть роботу або перезапустіть Codex Computer
Use, за потреби перезапустіть Codex Desktop, а потім повторіть спробу в новому
сеансі OpenClaw. Якщо раніше на хості Computer Use працював через старіший
керований сервер застосунку Codex, оновіть установлений Plugin із вбудованого
в настільний застосунок маркетплейсу (для окремого встановлення настільного
Codex використовуйте шлях `Codex.app`):

```text
/codex computer-use install --source /Applications/ChatGPT.app/Contents/Resources/plugins/openai-bundled
```

**Інструмент Computer Use повідомляє `Native hook relay unavailable`.**
Нативному обробнику інструментів Codex не вдалося підключитися до активного
ретранслятора OpenClaw через локальний міст або резервний Gateway. Почніть
новий сеанс OpenClaw за допомогою `/new` або `/reset`. Якщо це спрацьовує один
раз, а під час наступного виклику інструмента помилка виникає знову, `/new`
лише очищає поточну спробу; перезапустіть сервер застосунку Codex або Gateway
OpenClaw, щоб видалити старі потоки й реєстрації обробників, а потім повторіть
спробу в новому сеансі.

**Автоматичне встановлення на початку ходу відхиляє джерело.** Це навмисна
поведінка. Спочатку додайте джерело явною командою
`/codex computer-use install --source <marketplace-source>`, після чого
автоматичне встановлення на початку наступних ходів зможе використовувати
виявлений локальний маркетплейс.
