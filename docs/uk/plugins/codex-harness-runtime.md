Ця сторінка документує контракт середовища виконання для ходів Codex harness. Для налаштування й
маршрутизації почніть із [Codex harness](/uk/plugins/codex-harness). Для полів конфігурації
див. [довідник Codex harness](/uk/plugins/codex-harness-reference).

## Огляд

Режим Codex — це не OpenClaw з іншим викликом моделі під капотом. Codex володіє більшою частиною
нативного циклу моделі, а OpenClaw адаптує свої поверхні Plugin, інструментів, сеансів і
діагностики навколо цієї межі.

OpenClaw усе ще володіє маршрутизацією каналів, файлами сеансів, доставленням видимих повідомлень,
динамічними інструментами OpenClaw, схваленнями, доставленням медіа та дзеркалом транскрипту.
Codex володіє канонічним нативним потоком, нативним циклом моделі, нативним продовженням інструментів
і нативною Compaction.

Маршрутизація промптів відповідає вибраному середовищу виконання, а не лише рядку провайдера. Нативний
хід Codex отримує інструкції розробника app-server Codex, тоді як явний маршрут сумісності OpenClaw
зберігає звичайний системний промпт OpenClaw навіть тоді, коли використовує автентифікацію або транспорт
OpenAI у стилі Codex.

Нативний Codex зберігає базові/модельні інструкції та поведінку проєктної документації, що належать Codex,
відповідно до активної конфігурації потоку Codex. OpenClaw запускає й відновлює нативні потоки
Codex із вимкненою вбудованою особистістю Codex, щоб файли особистості робочого простору й ідентичність
агента OpenClaw залишалися авторитетними. Легкі запуски OpenClaw усе ще зберігають своє наявне
пригнічення проєктної документації. Інструкції розробника OpenClaw охоплюють питання середовища виконання
OpenClaw, як-от доставлення в початковий канал, динамічні інструменти OpenClaw, делегування ACP,
контекст адаптера та файли профілю активного робочого простору агента. Каталоги Skills OpenClaw і
маршрутизовані інструментами вказівники `MEMORY.md` проєктуються як обмежені ходом інструкції розробника
для співпраці в нативному Codex. Активний вміст `BOOTSTRAP.md` і повне резервне впровадження
`MEMORY.md` усе ще використовують довідковий контекст вхідних даних ходу.

## Прив’язки потоків і зміни моделі

Коли сеанс OpenClaw прикріплено до наявного потоку Codex, наступний хід знову надсилає app-server
поточну вибрану модель OpenAI, політику схвалення, пісочницю та рівень сервісу. Перемикання з
`openai/gpt-5.5` на `openai/gpt-5.2` зберігає прив’язку потоку, але просить Codex продовжити
з нововибраною моделлю.

## Видимі відповіді та Heartbeat

Коли прямий/початковий чат-хід виконується через Codex harness, видимі відповіді за замовчуванням
автоматично доставляються як фінальна відповідь асистента для внутрішніх поверхонь WebChat.
Це узгоджує Codex із контрактом промпта Pi harness: агенти відповідають звичайно, а OpenClaw публікує
фінальний текст у початкову розмову. Установіть `messages.visibleReplies: "message_tool"`, коли
прямий/початковий чат має навмисно зберігати фінальний текст асистента приватним, якщо агент не викличе
`message(action="send")`.

Ходи Codex Heartbeat також за замовчуванням отримують `heartbeat_respond` у пошуковому каталозі
інструментів OpenClaw, щоб агент міг зафіксувати, чи пробудження має залишитися тихим, чи надіслати
сповіщення, не кодувавши цей потік керування у фінальному тексті.

Спеціальні для Heartbeat вказівки щодо ініціативи надсилаються як інструкція розробника режиму співпраці
Codex у самому ході Heartbeat. Звичайні чат-ходи відновлюють стандартний режим Codex замість перенесення
філософії Heartbeat у свій звичайний промпт середовища виконання. Коли існує непорожній `HEARTBEAT.md`,
інструкції режиму співпраці Heartbeat спрямовують Codex до файла замість вбудовування його вмісту.

## Межі хуків

Codex harness має три рівні хуків:

| Рівень                                | Власник                  | Призначення                                                        |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------ |
| Хуки Plugin OpenClaw                  | OpenClaw                 | Сумісність продукту/Plugin між harness OpenClaw і Codex.           |
| Middleware розширення app-server Codex | Вбудовані Plugins OpenClaw | Поведінка адаптера на кожен хід навколо динамічних інструментів OpenClaw. |
| Нативні хуки Codex                    | Codex                    | Низькорівневий життєвий цикл Codex і політика нативних інструментів із конфігурації Codex. |

OpenClaw не використовує проєктні або глобальні файли Codex `hooks.json` для маршрутизації
поведінки Plugin OpenClaw. Для підтримуваного моста нативних інструментів і дозволів
OpenClaw впроваджує конфігурацію Codex на рівні потоку для `PreToolUse`, `PostToolUse`,
`PermissionRequest` і `Stop`.

Коли схвалення app-server Codex увімкнено, тобто `approvalPolicy` не дорівнює
`"never"`, стандартна впроваджена конфігурація нативних хуків пропускає `PermissionRequest`, щоб
рев’юер app-server Codex і міст схвалень OpenClaw обробляли реальні ескалації після перевірки.
Оператори можуть явно додати `permission_request` до `nativeHookRelay.events`, коли їм потрібен
релей сумісності.

Інші хуки Codex, як-от `SessionStart` і `UserPromptSubmit`, залишаються засобами керування рівня
Codex. Вони не надаються як хуки Plugin OpenClaw у контракті v1.

Для динамічних інструментів OpenClaw виконує інструмент після того, як Codex запитує виклик, тому
OpenClaw запускає поведінку Plugin і middleware, якою він володіє в адаптері harness. Для нативних
інструментів Codex канонічним записом інструмента володіє Codex. OpenClaw може дзеркалити вибрані
події, але не може переписувати нативний потік Codex, якщо Codex не надає цю операцію через app-server
або callback-и нативних хуків.

Події `PreToolUse` у режимі звіту app-server Codex відкладають запити схвалення Plugin до відповідного
схвалення app-server. Якщо хук OpenClaw `before_tool_call` повертає `requireApproval`, тоді як нативне
навантаження задає режим схвалення звіту (`openclaw_approval_mode` дорівнює `"report"`), релей нативних
хуків записує вимогу схвалення Plugin і не повертає нативного рішення. Коли Codex надсилає запит
схвалення app-server для того самого використання інструмента, OpenClaw відкриває промпт схвалення
Plugin і відображає рішення назад у Codex. Події Codex `PermissionRequest` є окремим шляхом схвалення
і все ще можуть маршрутизуватися через схвалення OpenClaw, коли середовище виконання налаштовано для
цього моста.

Сповіщення елементів app-server Codex також надають асинхронні спостереження `after_tool_call` для
завершень нативних інструментів, які вже не покриті релеєм нативного `PostToolUse`. Ці спостереження
призначені лише для телеметрії та сумісності Plugin; вони не можуть блокувати, затримувати або змінювати
нативний виклик інструмента.

Проєкції Compaction і життєвого циклу LLM надходять зі сповіщень app-server Codex і стану адаптера
OpenClaw, а не з команд нативних хуків Codex. Події OpenClaw `before_compaction`, `after_compaction`,
`llm_input` і `llm_output` є спостереженнями рівня адаптера, а не побайтовими захопленнями внутрішнього
запиту Codex або навантажень Compaction.

Нативні сповіщення app-server Codex `hook/started` і `hook/completed` проєктуються як події агента
`codex_app_server.hook` для траєкторії та налагодження. Вони не викликають хуки Plugin OpenClaw.

## Нативні дозволи та elicitation MCP

Для `PermissionRequest` OpenClaw повертає лише явні рішення дозволити або заборонити,
коли політика ухвалює рішення. Результат без рішення не є дозволом. Codex трактує його як відсутність
рішення хуку та переходить до власного захисника або шляху схвалення користувачем.

Режими схвалення app-server Codex типово пропускають цей нативний хук. Така поведінка
застосовується, коли `permission_request` явно включено до
`nativeHookRelay.events` або коли середовище виконання сумісності встановлює його.

Коли оператор вибирає `allow-always` для запиту нативного дозволу Codex,
OpenClaw запам'ятовує точний відбиток провайдера/сесії/вводу інструмента/cwd для
обмеженого вікна сесії. Запам'ятане рішення навмисно працює лише за точним збігом:
змінена команда, аргументи, корисне навантаження інструмента або cwd створює нове
схвалення.

Elicitations схвалення інструментів MCP Codex маршрутизуються через потік схвалення
плагінів OpenClaw, коли Codex позначає `_meta.codex_approval_kind` як
`"mcp_tool_call"`. Промпти Codex `request_user_input` надсилаються назад до
початкового чату, а наступне повідомлення-відповідь у черзі відповідає на цей нативний
запит сервера замість того, щоб спрямовуватися як додатковий контекст. Інші запити elicitation MCP
завершуються закритою відмовою.

Загальний потік схвалення плагінів, який переносить ці промпти, див.
[Запити дозволів Plugin](/uk/plugins/plugin-permission-requests).

## Керування чергою

Керування чергою активного запуску відображається на `turn/steer` app-server Codex. З
типовим `messages.queue.mode: "steer"` OpenClaw пакетно групує повідомлення чату в режимі steer
для налаштованого тихого вікна та надсилає їх як один запит `turn/steer`
у порядку надходження.

Перевірка Codex і ручні ходи Compaction можуть відхиляти керування в тому самому ході. У такому
разі OpenClaw чекає завершення активного запуску, перш ніж почати запит.
Використовуйте `/queue followup` або `/queue collect`, коли повідомлення мають ставати в чергу за замовчуванням
замість керування. Див. [Черга керування](/uk/concepts/queue-steering).

## Завантаження відгуку Codex

Коли `/diagnostics [note]` схвалено для сеансу, що використовує нативну обв'язку Codex,
OpenClaw також викликає `feedback/upload` сервера застосунку Codex для відповідних
гілок Codex. Завантаження просить сервер застосунку включити журнали для кожної зазначеної гілки
та породжених підгілок Codex, коли вони доступні.

Завантаження проходить через звичайний шлях відгуків Codex до серверів OpenAI. Якщо відгуки Codex
вимкнено на цьому сервері застосунку, команда повертає помилку сервера застосунку.
Завершена відповідь діагностики перелічує канали, ідентифікатори сеансів OpenClaw,
ідентифікатори гілок Codex і локальні команди `codex resume <thread-id>` для гілок,
які було надіслано.

Якщо ви відхилите або проігноруєте схвалення, OpenClaw не виводить ці ідентифікатори Codex і
не надсилає відгук Codex. Завантаження не замінює локальний експорт діагностики Gateway.
Див. [Експорт діагностики](/uk/gateway/diagnostics) щодо
схвалення, приватності, локального пакета та поведінки в груповому чаті.

Використовуйте `/codex diagnostics [note]` лише тоді, коли вам потрібне саме завантаження
відгуку Codex для поточної приєднаної гілки без повного пакета діагностики Gateway.

## Compaction і дзеркало транскрипту

Коли вибрана модель використовує обв'язку Codex, нативний Compaction гілки належить
серверу застосунку Codex. OpenClaw не запускає передпольотний Compaction для ходів Codex,
не замінює Compaction Codex на Compaction рушія контексту і не
відступає до підсумовування OpenClaw або публічного OpenAI, коли нативний Compaction Codex
не вдається запустити. OpenClaw зберігає дзеркало транскрипту для історії каналу,
пошуку, `/new`, `/reset` і майбутнього перемикання моделі або обв'язки.

Явні запити Compaction, як-от `/compact` або запитана Plugin операція ручного
compact, запускають нативний Compaction Codex через `thread/compact/start`.
OpenClaw повертається після запуску цієї нативної операції. Він не чекає
завершення, не накладає окремий тайм-аут OpenClaw, не перезапускає спільний сервер
застосунку Codex і не записує операцію як завершений OpenClaw Compaction.

Коли рушій контексту запитує проєкцію початкового завантаження гілки Codex, OpenClaw
проєктує назви та ідентифікатори викликів інструментів, форми вхідних даних і відредагований вміст
результатів інструментів у свіжу гілку Codex. Він не копіює сирі значення аргументів
викликів інструментів у цю проєкцію.

Дзеркало включає запит користувача, фінальний текст асистента та легкі записи міркувань
або плану Codex, коли сервер застосунку їх випускає. Наразі OpenClaw записує лише
явні нативні сигнали запуску Compaction, коли він запитує Compaction. Він не
надає зрозумілого для людини підсумку Compaction або придатного для аудиту списку
записів, які Codex зберіг після Compaction.

Оскільки Codex володіє канонічною нативною гілкою, `tool_result_persist` наразі не
перезаписує нативні записи результатів інструментів Codex. Він застосовується лише тоді,
коли OpenClaw записує результат інструмента транскрипту сеансу, що належить OpenClaw.

## Медіа та доставка

OpenClaw і надалі володіє доставкою медіа та вибором постачальника медіа. Зображення,
відео, музика, PDF, TTS і розуміння медіа використовують відповідні налаштування постачальника/моделі,
такі як `agents.defaults.imageGenerationModel`, `videoGenerationModel`,
`pdfModel` і `messages.tts`.

Текст, зображення, відео, музика, TTS, схвалення та вивід інструментів повідомлень і надалі
проходять через звичайний шлях доставки OpenClaw. Генерація медіа не потребує застарілого середовища виконання.
Коли Codex випускає нативний елемент генерації зображення з `savedPath`, OpenClaw
пересилає саме цей файл через звичайний шлях медіа відповіді, навіть якщо хід Codex
не має тексту асистента.
