OpenClaw організовує розмови в **сесії**. Кожне повідомлення маршрутизується до
сесії залежно від того, звідки воно надійшло -- особисті повідомлення, групові чати, завдання cron тощо.

## Як маршрутизуються повідомлення

| Джерело            | Поведінка                         |
| ------------------ | --------------------------------- |
| Особисті повідомлення | Спільна сесія за замовчуванням |
| Групові чати       | Ізольовано для кожної групи       |
| Кімнати/канали     | Ізольовано для кожної кімнати     |
| Завдання Cron      | Нова сесія для кожного запуску    |
| Webhooks           | Ізольовано для кожного hook       |

## Ізоляція особистих повідомлень

За замовчуванням усі особисті повідомлення використовують одну сесію для безперервності. Це підходить для
налаштувань з одним користувачем.

<Warning>
Якщо кілька людей можуть писати вашому агенту, увімкніть ізоляцію особистих повідомлень. Без неї всі
користувачі спільно використовують той самий контекст розмови -- приватні повідомлення Аліси були б
видимі Бобу.
</Warning>

**Виправлення:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // isolate by channel + sender
  },
}
```

Інші параметри:

- `main` (за замовчуванням) -- усі особисті повідомлення використовують одну сесію.
- `per-peer` -- ізолювати за відправником (між каналами).
- `per-channel-peer` -- ізолювати за каналом + відправником (рекомендовано).
- `per-account-channel-peer` -- ізолювати за обліковим записом + каналом + відправником.

<Tip>
Якщо та сама людина зв’язується з вами з кількох каналів, використовуйте
`session.identityLinks`, щоб пов’язати її ідентичності, аби вони використовували одну сесію.
</Tip>

### Пристикування пов’язаних каналів

Команди пристикування дають користувачу змогу перенести маршрут відповіді поточної сесії прямого чату до
іншого пов’язаного каналу без запуску нової сесії. Див.
[Пристикування каналів](/uk/concepts/channel-docking) для прикладів, конфігурації та
усунення несправностей.

Перевірте своє налаштування за допомогою `openclaw security audit`.

## Життєвий цикл сесії

Сесії повторно використовуються, доки не завершиться їхній строк дії:

- **Щоденне скидання** (за замовчуванням) -- нова сесія о 4:00 за місцевим часом на хості
  Gateway. Щоденна актуальність базується на часі запуску поточного `sessionId`, а не
  на пізніших записах метаданих.
- **Скидання за бездіяльністю** (необов’язково) -- нова сесія після періоду бездіяльності. Задайте
  `session.reset.idleMinutes`. Актуальність бездіяльності базується на останній реальній
  взаємодії користувача/каналу, тому Heartbeat, Cron і системні події exec не
  підтримують сесію активною.
- **Ручне скидання** -- введіть `/new` або `/reset` у чаті. `/new <model>` також
  перемикає модель.

Коли налаштовано і щоденне скидання, і скидання за бездіяльністю, спрацьовує те, строк якого завершується першим.
Heartbeat, Cron, exec та інші ходи системних подій можуть записувати метадані сесії,
але ці записи не подовжують актуальність щоденного скидання чи скидання за бездіяльністю. Коли скидання
переводить сесію, поставлені в чергу сповіщення системних подій для старої сесії
відкидаються, щоб застарілі фонові оновлення не додавалися перед першим prompt у
новій сесії.

Сесії з активною CLI-сесією, що належить провайдеру, не перериваються неявним
щоденним значенням за замовчуванням. Використовуйте `/reset` або явно налаштуйте `session.reset`, коли такі
сесії мають завершуватися за таймером.

## Де зберігається стан

Увесь стан сесій належить **gateway**. UI-клієнти запитують у Gateway
дані сесій.

- **Сховище:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Транскрипти:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

`sessions.json` зберігає окремі часові мітки життєвого циклу:

- `sessionStartedAt`: коли почався поточний `sessionId`; щоденне скидання використовує це.
- `lastInteractionAt`: остання взаємодія користувача/каналу, яка подовжує час життя за бездіяльністю.
- `updatedAt`: остання зміна рядка сховища; корисно для списків і очищення, але не
  є авторитетним джерелом для актуальності щоденного скидання/скидання за бездіяльністю.

Старіші рядки без `sessionStartedAt` визначаються із заголовка сесії JSONL-транскрипта,
коли він доступний. Якщо старіший рядок також не має `lastInteractionAt`,
актуальність бездіяльності повертається до часу початку цієї сесії, а не до пізніших службових
записів.

## Обслуговування сесій

OpenClaw автоматично обмежує обсяг сховища сесій із часом. За замовчуванням він працює
в режимі `warn` (повідомляє, що було б очищено). Задайте `session.maintenance.mode`
як `"enforce"` для автоматичного очищення:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

Для виробничих лімітів `maxEntries` записи під час виконання Gateway використовують невеликий буфер верхньої межі й пакетно очищаються назад до налаштованого обмеження. Читання сховища сесій не очищають і не обмежують записи під час запуску Gateway. Це дає змогу уникнути повного очищення сховища під час кожного запуску або ізольованої сесії cron. `openclaw sessions cleanup --enforce` застосовує обмеження негайно.

Обслуговування зберігає довговічні зовнішні вказівники розмов, зокрема групові
сесії та сесії чатів у межах thread, водночас дозволяючи синтетичним записам Cron,
hook, Heartbeat, ACP і під-агентів застарівати.

Перегляньте попередній результат за допомогою `openclaw sessions cleanup --dry-run`.

## Перегляд сесій

- `openclaw status` -- шлях до сховища сесій і нещодавня активність.
- `openclaw sessions --json` -- усі сесії (фільтруйте за допомогою `--active <minutes>`).
- `/status` у чаті -- використання контексту, модель і перемикачі.
- `/context list` -- що міститься в системному prompt.
