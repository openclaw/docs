OpenClaw організовує розмови в **сесії**. Кожне повідомлення маршрутизується до
сесії залежно від того, звідки воно надійшло -- приватні повідомлення, групові чати, cron-завдання тощо.

## Як маршрутизуються повідомлення

| Джерело            | Поведінка                            |
| ------------------ | ------------------------------------ |
| Приватні повідомлення | Спільна сесія за замовчуванням     |
| Групові чати       | Ізольовано для кожної групи          |
| Кімнати/канали     | Ізольовано для кожної кімнати        |
| Cron-завдання      | Нова сесія для кожного запуску       |
| Webhook-и          | Ізольовано для кожного hook          |

## Ізоляція приватних повідомлень

За замовчуванням усі приватні повідомлення мають спільну сесію для безперервності. Це підходить для
налаштувань з одним користувачем.

<Warning>
Якщо кілька людей можуть надсилати повідомлення вашому агенту, увімкніть ізоляцію приватних повідомлень. Без цього всі
користувачі мають спільний контекст розмови -- приватні повідомлення Аліси були б
видимі Бобу.
</Warning>

**Виправлення:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // isolate by channel + sender
  },
}
```

Інші варіанти:

- `main` (за замовчуванням) -- усі приватні повідомлення мають одну спільну сесію.
- `per-peer` -- ізолювати за відправником (між каналами).
- `per-channel-peer` -- ізолювати за каналом + відправником (рекомендовано).
- `per-account-channel-peer` -- ізолювати за обліковим записом + каналом + відправником.

<Tip>
Якщо та сама людина зв'язується з вами з кількох каналів, використовуйте
`session.identityLinks`, щоб зв'язати її ідентичності, аби вони мали одну спільну сесію.
</Tip>

### Закріплення пов'язаних каналів

Команди dock дають змогу користувачу перемістити маршрут відповіді поточної сесії прямого чату до
іншого пов'язаного каналу, не починаючи нову сесію. Див.
[Закріплення каналів](/uk/concepts/channel-docking) для прикладів, конфігурації та
усунення несправностей.

Перевірте своє налаштування за допомогою `openclaw security audit`.

## Життєвий цикл сесії

Сесії використовуються повторно, доки не спливе їхній термін:

- **Щоденне скидання** (за замовчуванням) -- нова сесія о 4:00 ранку за локальним часом на хості Gateway.
  Щоденна свіжість базується на тому, коли поточний `sessionId` почався, а не
  на пізніших записах метаданих.
- **Скидання через бездіяльність** (необов'язково) -- нова сесія після періоду бездіяльності. Задайте
  `session.reset.idleMinutes`. Свіжість за бездіяльністю базується на останній реальній
  взаємодії користувача/каналу, тому Heartbeat, Cron та системні події exec не
  підтримують сесію активною.
- **Ручне скидання** -- введіть `/new` або `/reset` у чаті. `/new <model>` також
  перемикає модель.

Коли налаштовано і щоденне скидання, і скидання через бездіяльність, спрацьовує те, чий термін спливає першим.
Heartbeat, Cron, exec та інші ходи системних подій можуть записувати метадані сесії,
але ці записи не подовжують свіжість щоденного скидання або скидання через бездіяльність. Коли скидання
переводить сесію, поставлені в чергу сповіщення системних подій для старої сесії
відкидаються, щоб застарілі фонові оновлення не додавалися на початок першого prompt у
новій сесії.

Сесії з активною CLI-сесією, що належить провайдеру, не перериваються неявним
щоденним значенням за замовчуванням. Використовуйте `/reset` або явно налаштуйте `session.reset`, коли такі
сесії мають спливати за таймером.

## Де зберігається стан

Усім станом сесій володіє **gateway**. UI-клієнти запитують дані сесій у gateway.

- **Сховище:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Транскрипти:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

`sessions.json` зберігає окремі часові мітки життєвого циклу:

- `sessionStartedAt`: коли почався поточний `sessionId`; щоденне скидання використовує це.
- `lastInteractionAt`: остання взаємодія користувача/каналу, яка подовжує час життя за бездіяльністю.
- `updatedAt`: остання зміна рядка сховища; корисно для списків і очищення, але не
  є авторитетним джерелом для свіжості щоденного скидання або скидання через бездіяльність.

Старіші рядки без `sessionStartedAt` визначаються з заголовка сесії в транскрипті JSONL,
коли він доступний. Якщо старішому рядку також бракує `lastInteractionAt`,
свіжість за бездіяльністю повертається до часу початку цієї сесії, а не до пізніших
службових записів.

## Обслуговування сесій

OpenClaw автоматично обмежує сховище сесій із часом. За замовчуванням він працює
в режимі `warn` (повідомляє, що було б очищено). Задайте `session.maintenance.mode`
як `"enforce"` для автоматичного очищення:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

Для виробничих лімітів `maxEntries` середовище виконання Gateway під час запису використовує невеликий буфер верхньої межі та очищує сховище пакетами до налаштованого ліміту. Читання сховища сесій не очищують і не обмежують записи під час запуску Gateway. Це уникає повного очищення сховища під час кожного запуску або ізольованої cron-сесії. `openclaw sessions cleanup --enforce` застосовує ліміт негайно.

Попередній перегляд за допомогою `openclaw sessions cleanup --dry-run`.

## Перегляд сесій

- `openclaw status` -- шлях до сховища сесій і нещодавня активність.
- `openclaw sessions --json` -- усі сесії (фільтруйте за допомогою `--active <minutes>`).
- `/status` у чаті -- використання контексту, модель і перемикачі.
- `/context list` -- що міститься в системному prompt.
