OpenClaw організовує розмови в **сеанси**. Кожне повідомлення маршрутизується до
сеансу залежно від того, звідки воно надійшло — DM, групові чати, Cron-завдання тощо.

## Як маршрутизуються повідомлення

| Джерело         | Поведінка                   |
| --------------- | --------------------------- |
| Прямі повідомлення | Спільний сеанс за замовчуванням |
| Групові чати    | Ізоляція для кожної групи   |
| Кімнати/канали  | Ізоляція для кожної кімнати |
| Cron-завдання   | Новий сеанс для кожного запуску |
| Webhook         | Ізоляція для кожного хука   |

## Ізоляція DM

За замовчуванням усі DM використовують один сеанс для безперервності. Це добре
підходить для однокористувацьких конфігурацій.

<Warning>
Якщо вашому агенту можуть писати кілька людей, увімкніть ізоляцію DM. Без неї
всі користувачі спільно використовують той самий контекст розмови — приватні
повідомлення Alice будуть видимі Bob.
</Warning>

**Виправлення:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // isolate by channel + sender
  },
}
```

Інші варіанти:

- `main` (за замовчуванням) — усі DM використовують один сеанс.
- `per-peer` — ізоляція за відправником (між каналами).
- `per-channel-peer` — ізоляція за каналом + відправником (рекомендовано).
- `per-account-channel-peer` — ізоляція за обліковим записом + каналом + відправником.

<Tip>
Якщо та сама людина звертається до вас із кількох каналів, використовуйте
`session.identityLinks`, щоб пов’язати її ідентичності, і тоді вона матиме один спільний сеанс.
</Tip>

Перевірте свою конфігурацію за допомогою `openclaw security audit`.

## Життєвий цикл сеансу

Сеанси використовуються повторно, доки не завершиться їхній термін дії:

- **Щоденне скидання** (за замовчуванням) — новий сеанс о 4:00 ранку за місцевим часом на хості Gateway.
  Щоденна свіжість визначається тим, коли почався поточний `sessionId`, а не
  пізнішими записами метаданих.
- **Скидання через бездіяльність** (необов’язково) — новий сеанс після періоду бездіяльності. Установіть
  `session.reset.idleMinutes`. Свіжість для бездіяльності визначається за останньою реальною
  взаємодією користувача/каналу, тому системні події Heartbeat, Cron і exec не
  підтримують сеанс активним.
- **Ручне скидання** — введіть у чаті `/new` або `/reset`. `/new <model>` також
  перемикає модель.

Коли налаштовано і щоденне скидання, і скидання через бездіяльність, спрацьовує те,
термін дії якого завершується раніше. Heartbeat, Cron, exec та інші ходи системних подій
можуть записувати метадані сеансу, але ці записи не подовжують свіжість для щоденного
скидання або скидання через бездіяльність. Коли скидання переводить сеанс на новий,
поставлені в чергу сповіщення системних подій для старого сеансу відкидаються, щоб
застарілі фонові оновлення не додавалися на початок першого запиту в новому сеансі.

Сеанси з активним CLI-сеансом, яким керує провайдер, не перериваються неявним
щоденним скиданням за замовчуванням. Використовуйте `/reset` або явно налаштуйте
`session.reset`, якщо такі сеанси мають завершуватися за таймером.

## Де зберігається стан

Увесь стан сеансу належить **Gateway**. UI-клієнти звертаються до Gateway по
дані сеансу.

- **Сховище:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Транскрипти:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

`sessions.json` зберігає окремі часові мітки життєвого циклу:

- `sessionStartedAt`: коли почався поточний `sessionId`; щоденне скидання використовує саме це значення.
- `lastInteractionAt`: остання взаємодія користувача/каналу, що подовжує час життя до скидання через бездіяльність.
- `updatedAt`: остання мутація рядка сховища; корисно для перегляду й очищення, але
  не є авторитетним значенням для свіжості щоденного скидання/скидання через бездіяльність.

Для старіших рядків без `sessionStartedAt` значення визначається з заголовка сеансу
в транскрипті JSONL, коли це можливо. Якщо в старішому рядку також немає `lastInteractionAt`,
свіжість для бездіяльності повертається до часу початку цього сеансу, а не до пізніших
службових записів.

## Обслуговування сеансів

OpenClaw автоматично обмежує обсяг сховища сеансів із часом. За замовчуванням він працює
в режимі `warn` (повідомляє, що було б очищено). Установіть `session.maintenance.mode`
в `"enforce"` для автоматичного очищення:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

Для лімітів `maxEntries` виробничого масштабу записи середовища виконання Gateway використовують
невеликий буфер верхнього порога й пакетно очищають сховище назад до налаштованого ліміту. Це
дозволяє уникнути повного очищення сховища під час кожного ізольованого Cron-сеансу. `openclaw sessions cleanup --enforce`
застосовує ліміт негайно.

Попередній перегляд: `openclaw sessions cleanup --dry-run`.

## Перегляд сеансів

- `openclaw status` — шлях до сховища сеансів і нещодавня активність.
- `openclaw sessions --json` — усі сеанси (фільтрація за допомогою `--active <minutes>`).
- `/status` у чаті — використання контексту, модель і перемикачі.
- `/context list` — що входить до системного запиту.
