OpenClaw організовує розмови в **сеанси**. Кожне повідомлення маршрутизується до
сеансу залежно від того, звідки воно надійшло — DM, групові чати, Cron-завдання тощо.

## Як маршрутизуються повідомлення

| Джерело         | Поведінка                     |
| --------------- | ----------------------------- |
| Прямі повідомлення | Спільний сеанс за замовчуванням |
| Групові чати    | Ізольовано для кожної групи   |
| Кімнати/канали  | Ізольовано для кожної кімнати |
| Cron-завдання   | Новий сеанс для кожного запуску |
| Webhook         | Ізольовано для кожного hook   |

## Ізоляція DM

За замовчуванням усі DM використовують один сеанс для безперервності. Це добре
підходить для однокористувацьких конфігурацій.

<Warning>
Якщо вашому агенту можуть писати кілька людей, увімкніть ізоляцію DM. Без неї
всі користувачі спільно використовують той самий контекст розмови — приватні
повідомлення Alice будуть видимі Bob.
</Warning>

**Рішення:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // ізоляція за каналом + відправником
  },
}
```

Інші варіанти:

- `main` (за замовчуванням) — усі DM використовують один сеанс.
- `per-peer` — ізоляція за відправником (між каналами).
- `per-channel-peer` — ізоляція за каналом + відправником (рекомендовано).
- `per-account-channel-peer` — ізоляція за обліковим записом + каналом + відправником.

<Tip>
Якщо одна й та сама людина звертається до вас із кількох каналів, використовуйте
`session.identityLinks`, щоб пов’язати її ідентичності й дати їй спільний сеанс.
</Tip>

Перевірте свою конфігурацію за допомогою `openclaw security audit`.

## Життєвий цикл сеансу

Сеанси використовуються повторно, доки не завершиться строк їх дії:

- **Щоденне скидання** (за замовчуванням) — новий сеанс о 4:00 ранку за місцевим часом на хості
  Gateway. Щоденна актуальність визначається моментом початку поточного `sessionId`, а не
  пізнішими записами метаданих.
- **Скидання через неактивність** (необов’язково) — новий сеанс після періоду бездіяльності. Установіть
  `session.reset.idleMinutes`. Актуальність при неактивності визначається останньою реальною
  взаємодією користувача/каналу, тому системні події heartbeat, Cron і exec не
  підтримують сеанс активним.
- **Ручне скидання** — введіть `/new` або `/reset` у чаті. `/new <model>` також
  перемикає модель.

Якщо налаштовано і щоденне скидання, і скидання через неактивність, спрацьовує те,
у якого строк дії завершиться раніше.
Системні події Heartbeat, Cron, exec та інші можуть записувати метадані сеансу,
але ці записи не подовжують актуальність для щоденного скидання чи скидання через неактивність.
Коли скидання переводить сеанс на новий, відкладені сповіщення системних подій для старого сеансу
відкидаються, щоб застарілі фонові оновлення не додавалися на початок першого запиту в
новому сеансі.

Сеанси з активним CLI-сеансом, яким керує провайдер, не перериваються неявним
щоденним скиданням за замовчуванням. Використовуйте `/reset` або явно налаштуйте `session.reset`,
коли такі сеанси мають завершуватися за таймером.

## Де зберігається стан

Увесь стан сеансу належить **Gateway**. UI-клієнти запитують у Gateway дані
сеансу.

- **Сховище:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Транскрипти:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

У `sessions.json` зберігаються окремі часові позначки життєвого циклу:

- `sessionStartedAt`: коли почався поточний `sessionId`; це використовує щоденне скидання.
- `lastInteractionAt`: остання взаємодія користувача/каналу, яка подовжує строк життя при неактивності.
- `updatedAt`: остання мутація рядка в сховищі; корисно для перегляду списків і очищення, але
  не є авторитетним значенням для актуальності щоденного скидання чи скидання через неактивність.

Старіші рядки без `sessionStartedAt` визначаються з заголовка сеансу в транскрипті JSONL,
якщо він доступний. Якщо в старішому рядку також немає `lastInteractionAt`,
актуальність при неактивності визначається за часом початку того сеансу, а не за
пізнішими службовими записами.

## Обслуговування сеансів

OpenClaw автоматично обмежує обсяг сховища сеансів з часом. За замовчуванням він працює
в режимі `warn` (повідомляє, що було б очищено). Установіть `session.maintenance.mode`
у значення `"enforce"` для автоматичного очищення:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

Попередній перегляд: `openclaw sessions cleanup --dry-run`.

## Перегляд сеансів

- `openclaw status` — шлях до сховища сеансів і нещодавня активність.
- `openclaw sessions --json` — усі сеанси (фільтр через `--active <minutes>`).
- `/status` у чаті — використання контексту, модель і перемикачі.
- `/context list` — що входить до системного запиту.
