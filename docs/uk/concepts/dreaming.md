Dreaming — це фонова система консолідації пам’яті в `memory-core`. Вона допомагає OpenClaw переносити сильні короткострокові сигнали в довготривалу пам’ять, зберігаючи процес пояснюваним і придатним до перегляду.

<Note>
Dreaming є **opt-in** і вимкнена за замовчуванням.
</Note>

## Що записує Dreaming

Dreaming зберігає два типи виводу:

- **Машинний стан** у `memory/.dreams/` (сховище пригадування, фазові сигнали, контрольні точки ingestion, блокування).
- **Людиночитний вивід** у `DREAMS.md` (або наявному `dreams.md`) і необов’язкових файлах фазових звітів у `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Довгострокове просування й надалі записує лише в `MEMORY.md`.

## Фазова модель

Dreaming використовує три кооперативні фази:

| Фаза | Призначення                                   | Довготривалий запис     |
| ----- | ----------------------------------------- | ----------------- |
| Light | Сортувати й підготувати нещодавній короткостроковий матеріал | Ні                |
| Deep  | Оцінити й просунути довготривалих кандидатів      | Так (`MEMORY.md`) |
| REM   | Осмислити теми й повторювані ідеї     | Ні                |

Ці фази є внутрішніми деталями реалізації, а не окремими користувацько налаштовуваними «режимами».

<AccordionGroup>
  <Accordion title="Фаза Light">
    Фаза Light приймає нещодавні щоденні сигнали пам’яті та сліди пригадування, усуває дублікати й готує рядки-кандидати.

    - Читає з короткострокового стану пригадування, нещодавніх щоденних файлів пам’яті та відредагованих транскриптів сеансів, коли вони доступні.
    - Записує керований блок `## Light Sleep`, коли сховище включає inline-вивід.
    - Записує сигнали підкріплення для подальшого ранжування Deep.
    - Ніколи не записує в `MEMORY.md`.

  </Accordion>
  <Accordion title="Фаза Deep">
    Фаза Deep вирішує, що стане довгостроковою пам’яттю.

    - Ранжує кандидатів за допомогою зваженого оцінювання та порогових перевірок.
    - Вимагає проходження `minScore`, `minRecallCount` і `minUniqueQueries`.
    - Повторно гідрує фрагменти з live щоденних файлів перед записом, тому застарілі або видалені фрагменти пропускаються.
    - Додає просунуті записи до `MEMORY.md`.
    - Записує підсумок `## Deep Sleep` у `DREAMS.md` і за потреби записує `memory/dreaming/deep/YYYY-MM-DD.md`.

  </Accordion>
  <Accordion title="Фаза REM">
    Фаза REM витягує патерни та рефлексивні сигнали.

    - Будує підсумки тем і рефлексій з нещодавніх короткострокових слідів.
    - Записує керований блок `## REM Sleep`, коли сховище включає inline-вивід.
    - Записує сигнали підкріплення REM, які використовуються ранжуванням Deep.
    - Ніколи не записує в `MEMORY.md`.

  </Accordion>
</AccordionGroup>

## Ingestion транскриптів сеансів

Dreaming може приймати відредаговані транскрипти сеансів у корпус Dreaming. Коли транскрипти доступні, вони подаються у фазу Light разом із щоденними сигналами пам’яті та слідами пригадування. Особистий і чутливий вміст редагується перед ingestion.

## Щоденник сновидінь

Dreaming також веде наративний **Щоденник сновидінь** у `DREAMS.md`. Після того як кожна фаза має достатньо матеріалу, `memory-core` запускає best-effort фоновий хід субагента й додає короткий запис у щоденник. Використовується стандартна модель runtime, якщо не налаштовано `dreaming.model`. Якщо налаштована модель недоступна, Щоденник сновидінь повторює спробу один раз із типовою моделлю сеансу.

<Note>
Цей щоденник призначений для читання людиною в UI Dreams, а не як джерело просування. Створені Dreaming артефакти щоденника/звіту виключаються з короткострокового просування. Лише обґрунтовані фрагменти пам’яті можуть бути просунуті в `MEMORY.md`.
</Note>

Також є обґрунтована історична лінія backfill для роботи з переглядом і відновленням:

<AccordionGroup>
  <Accordion title="Команди backfill">
    - `memory rem-harness --path ... --grounded` попередньо переглядає обґрунтований вивід щоденника з історичних нотаток `YYYY-MM-DD.md`.
    - `memory rem-backfill --path ...` записує оборотні обґрунтовані записи щоденника в `DREAMS.md`.
    - `memory rem-backfill --path ... --stage-short-term` готує обґрунтованих довготривалих кандидатів у тому самому короткостроковому сховищі доказів, яке вже використовує звичайна фаза Deep.
    - `memory rem-backfill --rollback` і `--rollback-short-term` видаляють ці підготовлені артефакти backfill, не торкаючись звичайних записів щоденника або live короткострокового пригадування.

  </Accordion>
</AccordionGroup>

Control UI надає той самий потік backfill/reset для щоденника, щоб ви могли перевірити результати в сцені Dreams перед рішенням, чи заслуговують обґрунтовані кандидати на просування. Сцена також показує окрему обґрунтовану лінію, щоб ви могли бачити, які підготовлені короткострокові записи походять з історичного replay, які просунуті елементи були led by grounded, і очищати лише підготовлені записи grounded-only, не торкаючись звичайного live короткострокового стану.

## Сигнали ранжування Deep

Ранжування Deep використовує шість зважених базових сигналів плюс фазове підкріплення:

| Сигнал              | Вага | Опис                                       |
| ------------------- | ------ | ------------------------------------------------- |
| Частота           | 0.24   | Скільки короткострокових сигналів накопичив запис |
| Релевантність           | 0.30   | Середня якість retrieval для запису           |
| Різноманітність запитів     | 0.15   | Різні контексти запитів/днів, у яких він з’являвся      |
| Свіжість             | 0.15   | Оцінка свіжості з урахуванням згасання з часом                      |
| Консолідація       | 0.10   | Сила повторення протягом кількох днів                     |
| Концептуальна насиченість | 0.06   | Щільність концептуальних тегів із фрагмента/шляху             |

Збіги фаз Light і REM додають невеликий boost зі згасанням за свіжістю з `memory/.dreams/phase-signals.json`.

## Планування

Коли ввімкнено, `memory-core` автоматично керує одним завданням cron для повного проходу Dreaming. Кожне проходження запускає фази по порядку: light → REM → deep.

Поведінка каденції за замовчуванням:

| Налаштування              | За замовчуванням       |
| -------------------- | ------------- |
| `dreaming.frequency` | `0 3 * * *`   |
| `dreaming.model`     | типова модель |

## Ключові значення за замовчуванням

Усі налаштування розміщені в `plugins.entries.memory-core.config.dreaming`.

<ParamField path="enabled" type="boolean" default="false">
  Увімкнути або вимкнути проходження Dreaming.
</ParamField>
<ParamField path="frequency" type="string" default="0 3 * * *">
  Каденція Cron для повного проходу Dreaming.
</ParamField>
<ParamField path="model" type="string">
  Необов’язкове перевизначення моделі субагента Щоденника сновидінь. Використовуйте канонічне значення `provider/model`, коли також задаєте allowlist `allowedModels` для субагента.
</ParamField>

<Warning>
`dreaming.model` вимагає `plugins.entries.memory-core.subagent.allowModelOverride: true`. Щоб обмежити його, також задайте `plugins.entries.memory-core.subagent.allowedModels`. Помилки довіри або allowlist залишаються видимими замість тихого fallback; повторна спроба покриває лише помилки недоступності моделі.
</Warning>

<Note>
Політика фаз, пороги та поведінка сховища є внутрішніми деталями реалізації (не користувацькою конфігурацією). Повний список ключів див. у [довіднику конфігурації пам’яті](/uk/reference/memory-config#dreaming).
</Note>

## UI Dreams

Коли ввімкнено, вкладка **Dreams** у Gateway показує:

- поточний стан увімкнення Dreaming
- статус на рівні фаз і наявність керованого проходу
- кількість короткострокових, обґрунтованих, сигнальних і просунутих сьогодні елементів
- час наступного запланованого запуску
- окрему обґрунтовану лінію сцени для підготовлених записів історичного replay
- розгортаний читач Щоденника сновидінь на основі `doctor.memory.dreamDiary`
