Dreaming — це фонова система ущільнення пам’яті в `memory-core`. Вона допомагає OpenClaw переносити сильні короткострокові сигнали в довготривалу пам’ять, зберігаючи процес зрозумілим і придатним для перевірки.

<Note>
Dreaming є **опціональним** і за замовчуванням вимкнено.
</Note>

## Що записує Dreaming

Dreaming зберігає два типи результатів:

- **Машинний стан** у `memory/.dreams/` (сховище recall, сигнали фаз, контрольні точки інгесту, блокування).
- **Зрозумілий людині вивід** у `DREAMS.md` (або наявному `dreams.md`) і необов’язкових файлах звітів фаз у `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Довгострокове просування, як і раніше, записується лише в `MEMORY.md`.

## Модель фаз

Dreaming використовує три кооперативні фази:

| Фаза | Призначення | Довготривалий запис |
| ----- | ----------- | ------------------- |
| Light | Сортування й підготовка нещодавнього короткострокового матеріалу | Ні |
| Deep  | Оцінювання та просування довготривалих кандидатів | Так (`MEMORY.md`) |
| REM   | Осмислення тем і повторюваних ідей | Ні |

Ці фази є внутрішніми деталями реалізації, а не окремими користувацькими «режимами».

<AccordionGroup>
  <Accordion title="Фаза Light">
    Фаза Light інгестує недавні щоденні сигнали пам’яті та сліди recall, дедуплікує їх і готує рядки-кандидати.

    - Читає зі стану короткострокового recall, нещодавніх щоденних файлів пам’яті та відредагованих транскриптів сесій, коли вони доступні.
    - Записує керований блок `## Light Sleep`, коли сховище містить вбудований вивід.
    - Фіксує сигнали підкріплення для подальшого ранжування Deep.
    - Ніколи не записує в `MEMORY.md`.

  </Accordion>
  <Accordion title="Фаза Deep">
    Фаза Deep вирішує, що стане довготривалою пам’яттю.

    - Ранжує кандидатів за допомогою зваженого оцінювання та порогових перевірок.
    - Вимагає проходження `minScore`, `minRecallCount` і `minUniqueQueries`.
    - Перед записом повторно гідратує фрагменти з актуальних щоденних файлів, тому застарілі/видалені фрагменти пропускаються.
    - Додає просунуті записи до `MEMORY.md`.
    - Записує підсумок `## Deep Sleep` у `DREAMS.md` і за потреби записує `memory/dreaming/deep/YYYY-MM-DD.md`.

  </Accordion>
  <Accordion title="Фаза REM">
    Фаза REM витягує шаблони та рефлексивні сигнали.

    - Створює підсумки тем і рефлексій із нещодавніх короткострокових слідів.
    - Записує керований блок `## REM Sleep`, коли сховище містить вбудований вивід.
    - Фіксує сигнали підкріплення REM, що використовуються в ранжуванні Deep.
    - Ніколи не записує в `MEMORY.md`.

  </Accordion>
</AccordionGroup>

## Інгест транскриптів сесій

Dreaming може інгестувати відредаговані транскрипти сесій до корпусу dreaming. Коли транскрипти доступні, вони подаються до фази Light разом із щоденними сигналами пам’яті та слідами recall. Особистий і чутливий вміст редагується перед інгестом.

## Dream Diary

Dreaming також веде наративний **Dream Diary** у `DREAMS.md`. Після того як для кожної фази накопичується достатньо матеріалу, `memory-core` запускає фоновий підхід субагента в режимі best-effort і додає короткий запис щоденника. Використовується модель runtime за замовчуванням, якщо не налаштовано `dreaming.model`.

<Note>
Цей щоденник призначений для читання людиною в Dreams UI, а не є джерелом просування. Артефакти щоденника/звітів, згенеровані Dreaming, виключаються з короткострокового просування. До просування в `MEMORY.md` допускаються лише обґрунтовані фрагменти пам’яті.
</Note>

Також є окремий обґрунтований шлях історичного backfill для перевірки та відновлення:

<AccordionGroup>
  <Accordion title="Команди backfill">
    - `memory rem-harness --path ... --grounded` показує попередній перегляд обґрунтованого виводу щоденника з історичних нотаток `YYYY-MM-DD.md`.
    - `memory rem-backfill --path ...` записує оборотні обґрунтовані записи щоденника в `DREAMS.md`.
    - `memory rem-backfill --path ... --stage-short-term` готує обґрунтованих довготривалих кандидатів у тому самому сховищі короткострокових доказів, яке вже використовує звичайна фаза Deep.
    - `memory rem-backfill --rollback` і `--rollback-short-term` видаляють ці підготовлені артефакти backfill, не торкаючись звичайних записів щоденника чи активного короткострокового recall.
  </Accordion>
</AccordionGroup>

Control UI надає той самий процес backfill/скидання щоденника, щоб ви могли перевірити результати в сцені Dreams, перш ніж вирішувати, чи заслуговують обґрунтовані кандидати на просування. Сцена також показує окрему обґрунтовану смугу, щоб ви могли бачити, які підготовлені короткострокові записи надійшли з історичного відтворення, які просунуті елементи були спричинені grounded, і очищати лише підготовлені записи, що належать тільки grounded, не торкаючись звичайного активного короткострокового стану.

## Сигнали ранжування Deep

Ранжування Deep використовує шість зважених базових сигналів плюс підкріплення фаз:

| Сигнал | Вага | Опис |
| ------ | ---- | ---- |
| Частота | 0.24 | Скільки короткострокових сигналів накопичив запис |
| Релевантність | 0.30 | Середня якість витягування для запису |
| Різноманітність запитів | 0.15 | Різні контексти запитів/днів, у яких він з’являвся |
| Нещодавність | 0.15 | Оцінка свіжості з часовим затуханням |
| Ущільнення | 0.10 | Сила повторення впродовж кількох днів |
| Концептуальна насиченість | 0.06 | Щільність тегів концептів із фрагмента/шляху |

Влучання фаз Light і REM додають невелике підсилення з часовим затуханням із `memory/.dreams/phase-signals.json`.

## Планування

Коли увімкнено, `memory-core` автоматично керує одним Cron-завданням для повного проходу dreaming. Кожен прохід запускає фази по порядку: light → REM → deep.

Поведінка стандартної періодичності:

| Налаштування | Значення за замовчуванням |
| ------------ | ------------------------- |
| `dreaming.frequency` | `0 3 * * *` |
| `dreaming.model` | модель за замовчуванням |

## Основні значення за замовчуванням

Усі налаштування розміщено в `plugins.entries.memory-core.config.dreaming`.

<ParamField path="enabled" type="boolean" default="false">
  Увімкнути або вимкнути прохід dreaming.
</ParamField>
<ParamField path="frequency" type="string" default="0 3 * * *">
  Періодичність Cron для повного проходу dreaming.
</ParamField>
<ParamField path="model" type="string">
  Необов’язкове перевизначення моделі субагента Dream Diary. Використовуйте канонічне значення `provider/model`, якщо також задаєте allowlist `allowedModels` для субагента.
</ParamField>

<Warning>
`dreaming.model` вимагає `plugins.entries.memory-core.subagent.allowModelOverride: true`. Щоб обмежити його, також задайте `plugins.entries.memory-core.subagent.allowedModels`.
</Warning>

<Note>
Політика фаз, пороги та поведінка сховища є внутрішніми деталями реалізації (а не користувацькою конфігурацією). Повний список ключів див. у [Довіднику з конфігурації Memory](/uk/reference/memory-config#dreaming).
</Note>

## Dreams UI

Коли увімкнено, вкладка **Dreams** у Gateway показує:

- поточний стан увімкнення dreaming
- статус на рівні фаз і наявність керованого проходу
- кількість короткострокових, grounded, сигнальних і просунутих сьогодні записів
- час наступного запланованого запуску
- окрему смугу grounded Scene для підготовлених записів історичного відтворення
- розгортаний читач Dream Diary на основі `doctor.memory.dreamDiary`
