**Середовище виконання агента** — це компонент, який володіє одним підготовленим циклом моделі: він
отримує промпт, керує виведенням моделі, обробляє нативні виклики інструментів і повертає
завершений хід до OpenClaw.

Середовища виконання легко сплутати з провайдерами, бо обидва з'являються поруч із
конфігурацією моделі. Це різні рівні:

| Рівень                  | Приклади                              | Що це означає                                                       |
| ------------- | ------------------------------------- | ------------------------------------------------------------------- |
| Провайдер      | `openai`, `anthropic`, `openai-codex` | Як OpenClaw автентифікується, виявляє моделі та іменує посилання на моделі. |
| Модель         | `gpt-5.5`, `claude-opus-4-6`          | Модель, вибрана для ходу агента.                              |
| Середовище виконання агента | `pi`, `codex`, `claude-cli`           | Низькорівневий цикл або бекенд, який виконує підготовлений хід.      |
| Канал       | Telegram, Discord, Slack, WhatsApp    | Де повідомлення входять в OpenClaw і виходять із нього.                            |

У коді ви також побачите слово **обв'язка**. Обв'язка — це реалізація,
яка надає середовище виконання агента. Наприклад, вбудована обв'язка Codex
реалізує середовище виконання `codex`. Публічна конфігурація використовує `agentRuntime.id`; `openclaw
doctor --fix` переписує старіші ключі політики середовища виконання до цієї форми.

Є два сімейства середовищ виконання:

- **Вбудовані обв'язки** працюють усередині підготовленого агентського циклу OpenClaw. Сьогодні це
  вбудоване середовище виконання `pi` плюс зареєстровані обв'язки plugin, як-от
  `codex`.
- **CLI-бекенди** запускають локальний процес CLI, зберігаючи канонічне посилання на модель.
  Наприклад, `anthropic/claude-opus-4-7` з
  `agentRuntime.id: "claude-cli"` означає "вибрати модель Anthropic, виконати
  через Claude CLI." `claude-cli` не є id вбудованої обв'язки, і його не можна
  передавати до вибору AgentHarness.

## Поверхні Codex

Найбільше плутанини виникає через кілька різних поверхонь, які мають назву Codex:

| Поверхня                                              | Назва/конфігурація OpenClaw                       | Що вона робить                                                                                               |
| ---------------------------------------------------- | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| Нативне середовище виконання app-server Codex                      | `openai/*` плюс `agentRuntime.id: "codex"` | Запускає вбудований агентський хід через app-server Codex. Це звичайне налаштування підписки ChatGPT/Codex. |
| Маршрут провайдера Codex OAuth                           | Посилання на моделі `openai-codex/*`                | Використовує OAuth підписки ChatGPT/Codex через звичайний раннер OpenClaw PI.                               |
| Адаптер Codex ACP                                    | `runtime: "acp"`, `agentId: "codex"`       | Запускає Codex через зовнішню площину керування ACP/acpx. Використовуйте лише тоді, коли ACP/acpx явно запитано.        |
| Нативний набір команд керування чатом Codex                | `/codex ...`                               | Прив'язує, відновлює, спрямовує, зупиняє та інспектує потоки app-server Codex із чату.                            |
| Маршрут OpenAI Platform API для моделей у стилі GPT/Codex | Посилання на моделі `openai/*`                      | Використовує автентифікацію OpenAI API-ключем, якщо перевизначення середовища виконання, як-от `agentRuntime.id: "codex"`, не запускає хід.     |

Ці поверхні навмисно незалежні. Увімкнення plugin `codex` робить
нативні можливості app-server доступними; воно не переписує
`openai-codex/*` на `openai/*`, не змінює наявні сесії та не
робить ACP типовим варіантом Codex. Вибір `openai-codex/*` означає "використовувати маршрут
провайдера Codex OAuth", якщо окремо не примусити середовище виконання.

Поширене налаштування підписки ChatGPT/Codex використовує Codex OAuth для автентифікації, але зберігає
посилання на модель як `openai/*` і вибирає середовище виконання `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

Це означає, що OpenClaw вибирає посилання на модель OpenAI, а потім просить середовище виконання app-server
Codex запустити вбудований агентський хід. Це не означає "використовувати API-білінг",
і не означає, що канал, каталог провайдера моделей або сховище сесій OpenClaw
стає Codex.

Коли вбудований plugin `codex` увімкнено, керування Codex природною мовою
має використовувати нативну командну поверхню `/codex` (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`) замість ACP. Використовуйте ACP для
Codex лише тоді, коли користувач явно просить ACP/acpx або тестує шлях адаптера ACP.
Claude Code, Gemini CLI, OpenCode, Cursor та подібні зовнішні обв'язки
далі використовують ACP.

Це дерево рішень для агента:

1. Якщо користувач просить **прив'язати/керувати/потік/відновити/спрямувати/зупинити Codex**, використовуйте
   нативну командну поверхню `/codex`, коли вбудований plugin `codex` увімкнено.
2. Якщо користувач просить **Codex як вбудоване середовище виконання** або хоче звичайний
   агентський досвід Codex із підтримкою підписки, використовуйте
   `openai/<model>` з `agentRuntime.id: "codex"`.
3. Якщо користувач просить **Codex OAuth/автентифікацію підписки на звичайному раннері OpenClaw**,
   використовуйте `openai-codex/<model>` і залиште середовище виконання як PI.
4. Якщо користувач явно каже **ACP**, **acpx** або **адаптер Codex ACP**, використовуйте
   ACP з `runtime: "acp"` і `agentId: "codex"`.
5. Якщо запит стосується **Claude Code, Gemini CLI, OpenCode, Cursor, Droid або
   іншої зовнішньої обв'язки**, використовуйте ACP/acpx, а не нативне середовище виконання під-агента.

| Ви маєте на увазі...                             | Використовуйте...                                       |
| --------------------------------------- | -------------------------------------------- |
| Керування чатом/потоками app-server Codex    | `/codex ...` з вбудованого plugin `codex` |
| Вбудоване середовище виконання агента app-server Codex | `agentRuntime.id: "codex"`                   |
| OpenAI Codex OAuth на раннері PI     | Посилання на моделі `openai-codex/*`                  |
| Claude Code або інша зовнішня обв'язка   | ACP/acpx                                     |

Про розділення префіксів сімейства OpenAI див. [OpenAI](/uk/providers/openai) і
[Провайдери моделей](/uk/concepts/model-providers). Про контракт підтримки середовища виконання Codex
див. [Обв'язка Codex](/uk/plugins/codex-harness#v1-support-contract).

## Відповідальність середовища виконання

Різні середовища виконання володіють різними частинами циклу.

| Поверхня                     | Вбудований PI OpenClaw                    | app-server Codex                                                            |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| Власник циклу моделі            | OpenClaw через вбудований раннер PI | app-server Codex                                                            |
| Канонічний стан потоку      | Транскрипт OpenClaw                     | Потік Codex, плюс дзеркало транскрипту OpenClaw                               |
| Динамічні інструменти OpenClaw      | Нативний цикл інструментів OpenClaw               | Передаються через адаптер Codex                                           |
| Нативні інструменти оболонки та файлів | Шлях PI/OpenClaw                        | Нативні інструменти Codex, передані через нативні хуки там, де підтримується            |
| Контекстний рушій              | Нативне складання контексту OpenClaw        | OpenClaw проєктує зібраний контекст у хід Codex                     |
| Compaction                  | OpenClaw або вибраний контекстний рушій     | Нативна Compaction Codex, зі сповіщеннями OpenClaw і підтримкою дзеркала |
| Доставка каналом            | OpenClaw                                | OpenClaw                                                                    |

Цей розподіл відповідальності є головним правилом проєктування:

- Якщо OpenClaw володіє поверхнею, OpenClaw може надати звичайну поведінку хуків plugin.
- Якщо нативне середовище виконання володіє поверхнею, OpenClaw потребує подій середовища виконання або нативних хуків.
- Якщо нативне середовище виконання володіє канонічним станом потоку, OpenClaw має дзеркалити і проєктувати контекст, а не переписувати непідтримувані внутрішні механізми.

## Вибір середовища виконання

OpenClaw вибирає вбудоване середовище виконання після визначення провайдера та моделі:

1. Записане середовище виконання сесії має пріоритет. Зміни конфігурації не перемикають на льоту
   наявний транскрипт на іншу нативну систему потоків.
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово встановлює це середовище виконання для нових або скинутих сесій.
3. `agents.defaults.agentRuntime.id` або `agents.list[].agentRuntime.id` може встановити
   `auto`, `pi`, зареєстрований id вбудованої обв'язки, як-от `codex`, або
   підтримуваний псевдонім CLI-бекенду, як-от `claude-cli`.
4. У режимі `auto` зареєстровані середовища виконання plugin можуть заявляти підтримувані пари
   провайдер/модель.
5. Якщо жодне середовище виконання не заявляє хід у режимі `auto` і встановлено `fallback: "pi"`
   (типово), OpenClaw використовує PI як резерв сумісності. Установіть
   `fallback: "none"`, щоб натомість зробити невідповідний вибір у режимі `auto` помилкою.

Явні середовища виконання plugin типово завершуються закритою помилкою. Наприклад,
`agentRuntime.id: "codex"` означає Codex або зрозумілу помилку вибору, якщо ви не встановите
`fallback: "pi"` у тій самій області перевизначення. Перевизначення середовища виконання не успадковує
ширше резервне налаштування, тому `agentRuntime.id: "codex"` на рівні агента не буде
непомітно перенаправлено назад до PI лише тому, що типові значення використовували `fallback: "pi"`.

Псевдоніми CLI-бекендів відрізняються від id вбудованих обв'язок. Бажана
форма Claude CLI така:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

Застарілі посилання, як-от `claude-cli/claude-opus-4-7`, залишаються підтримуваними для
сумісності, але нова конфігурація має зберігати провайдера/модель канонічними і розміщувати
бекенд виконання в `agentRuntime.id`.

Режим `auto` навмисно консервативний. Середовища виконання plugin можуть заявляти
пари провайдер/модель, які вони розуміють, але plugin Codex не заявляє
провайдера `openai-codex` у режимі `auto`. Це зберігає
`openai-codex/*` як явний маршрут PI Codex OAuth і не дає непомітно
перемістити конфігурації автентифікації підписки на нативну обв'язку app-server.

Якщо `openclaw doctor` попереджає, що plugin `codex` увімкнено, тоді як
`openai-codex/*` усе ще маршрутизується через PI, сприймайте це як діагностику, а не як
міграцію. Залиште конфігурацію без змін, коли PI Codex OAuth — це те, що вам потрібно.
Перемикайтеся на `openai/<model>` плюс `agentRuntime.id: "codex"` лише тоді, коли вам потрібне нативне
виконання app-server Codex.

## Контракт сумісності

Коли середовище виконання не є PI, воно має документувати, які поверхні OpenClaw підтримує.
Використовуйте цю форму для документації середовищ виконання:

| Запитання                              | Чому це важливо                                                                                         |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| Хто керує циклом моделі?               | Визначає, де відбуваються повторні спроби, продовження інструментів і рішення щодо фінальної відповіді. |
| Хто керує канонічною історією потоку?  | Визначає, чи OpenClaw може редагувати історію, чи лише віддзеркалювати її.                              |
| Чи працюють динамічні інструменти OpenClaw? | Обмін повідомленнями, сеанси, cron і інструменти, що належать OpenClaw, залежать від цього.             |
| Чи працюють хуки динамічних інструментів? | Plugins очікують `before_tool_call`, `after_tool_call` і middleware навколо інструментів, що належать OpenClaw. |
| Чи працюють хуки нативних інструментів? | Shell, patch та інструментам, що належать runtime, потрібна нативна підтримка хуків для політик і спостереження. |
| Чи запускається життєвий цикл контекстного рушія? | Plugins пам’яті й контексту залежать від життєвого циклу assemble, ingest, after-turn і compaction.      |
| Які дані compaction відкриваються?      | Деяким plugins потрібні лише сповіщення, тоді як іншим потрібні метадані збереженого/відкинутого.        |
| Що навмисно не підтримується?           | Користувачі не повинні припускати еквівалентність PI там, де нативний runtime керує більшою кількістю стану. |

Контракт підтримки runtime Codex задокументовано в
[Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Мітки стану

Вивід стану може показувати обидві мітки: `Execution` і `Runtime`. Сприймайте їх як
діагностику, а не як назви провайдерів.

- Посилання на модель, як-от `openai/gpt-5.5`, показує вибраного провайдера/модель.
- Ідентифікатор runtime, як-от `codex`, показує, який цикл виконує хід.
- Мітка каналу, як-от Telegram або Discord, показує, де відбувається розмова.

Якщо сеанс усе ще показує PI після зміни конфігурації runtime, почніть новий сеанс
за допомогою `/new` або очистьте поточний за допомогою `/reset`. Наявні сеанси зберігають свій
записаний runtime, щоб transcript не відтворювався через дві несумісні нативні
системи сеансів.
