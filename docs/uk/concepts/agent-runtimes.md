**Середовище виконання агента** - це компонент, який володіє одним підготовленим циклом моделі: воно
отримує prompt, керує виводом моделі, обробляє нативні виклики інструментів і повертає
завершений хід в OpenClaw.

Середовища виконання легко сплутати з провайдерами, бо обидва з'являються поруч із
конфігурацією моделі. Це різні рівні:

| Рівень        | Приклади                              | Що це означає                                                     |
| ------------- | ------------------------------------- | ----------------------------------------------------------------- |
| Провайдер     | `openai`, `anthropic`, `openai-codex` | Як OpenClaw автентифікується, знаходить моделі та називає посилання на моделі. |
| Модель        | `gpt-5.5`, `claude-opus-4-6`          | Модель, вибрана для ходу агента.                                  |
| Середовище виконання агента | `pi`, `codex`, `claude-cli`           | Низькорівневий цикл або бекенд, який виконує підготовлений хід.   |
| Канал         | Telegram, Discord, Slack, WhatsApp    | Де повідомлення входять в OpenClaw і виходять із нього.           |

У коді ви також побачите слово **harness**. Harness - це реалізація,
яка надає середовище виконання агента. Наприклад, вбудований Codex harness
реалізує середовище виконання `codex`. Публічна конфігурація використовує `agentRuntime.id`; `openclaw
doctor --fix` переписує старіші ключі політики середовища виконання в цю форму.

Є дві родини середовищ виконання:

- **Вбудовані harnesses** працюють усередині підготовленого циклу агента OpenClaw. Сьогодні це
  вбудоване середовище виконання `pi` плюс зареєстровані plugin harnesses, такі як
  `codex`.
- **CLI-бекенди** запускають локальний CLI-процес, зберігаючи канонічне посилання на модель.
  Наприклад, `anthropic/claude-opus-4-7` з
  `agentRuntime.id: "claude-cli"` означає "вибрати модель Anthropic, виконати
  через Claude CLI." `claude-cli` не є ідентифікатором вбудованого harness і не має
  передаватися у вибір AgentHarness.

## Поверхні Codex

Найбільше плутанини виникає через кілька різних поверхонь, які використовують назву Codex:

| Поверхня                                              | Назва/конфігурація OpenClaw                       | Що вона робить                                                                                               |
| ----------------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| Нативне середовище виконання Codex app-server         | `openai/*` плюс `agentRuntime.id: "codex"`        | Запускає вбудований хід агента через Codex app-server. Це звичайне налаштування підписки ChatGPT/Codex. |
| Маршрут провайдера Codex OAuth                        | `openai-codex/*` model refs                       | Використовує OAuth підписки ChatGPT/Codex через звичайний OpenClaw PI runner.                               |
| Адаптер Codex ACP                                     | `runtime: "acp"`, `agentId: "codex"`              | Запускає Codex через зовнішню площину керування ACP/acpx. Використовуйте лише тоді, коли ACP/acpx явно запитано. |
| Нативний набір команд керування чатом Codex           | `/codex ...`                                      | Прив'язує, відновлює, спрямовує, зупиняє та інспектує потоки Codex app-server із чату.                      |
| Маршрут OpenAI Platform API для моделей у стилі GPT/Codex | `openai/*` model refs                          | Використовує автентифікацію API-ключем OpenAI, якщо перевизначення середовища виконання, таке як `agentRuntime.id: "codex"`, не запускає хід. |

Ці поверхні навмисно незалежні. Увімкнення Plugin `codex` робить
доступними нативні функції app-server; воно не переписує
`openai-codex/*` у `openai/*`, не змінює наявні сесії та не робить
ACP типовим варіантом Codex. Вибір `openai-codex/*` означає "використати маршрут
провайдера Codex OAuth", якщо ви окремо не примусите середовище виконання.

Поширене налаштування підписки ChatGPT/Codex використовує Codex OAuth для автентифікації, але зберігає
посилання на модель як `openai/*` і вибирає середовище виконання `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

Це означає, що OpenClaw вибирає посилання на модель OpenAI, а потім просить середовище виконання
Codex app-server запустити вбудований хід агента. Це не означає "використовувати оплату через API", і
це не означає, що канал, каталог провайдера моделей або сховище сесій OpenClaw
стає Codex.

Коли вбудований Plugin `codex` увімкнено, керування Codex природною мовою
має використовувати нативну поверхню команд `/codex` (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`) замість ACP. Використовуйте ACP для
Codex лише тоді, коли користувач явно просить ACP/acpx або тестує шлях
адаптера ACP. Claude Code, Gemini CLI, OpenCode, Cursor і подібні зовнішні
harnesses і далі використовують ACP.

Це дерево рішень для агента:

1. Якщо користувач просить **Codex bind/control/thread/resume/steer/stop**, використовуйте
   нативну поверхню команд `/codex`, коли вбудований Plugin `codex` увімкнено.
2. Якщо користувач просить **Codex як вбудоване середовище виконання** або хоче звичайний
   агентський досвід Codex із підтримкою підписки, використовуйте
   `openai/<model>` з `agentRuntime.id: "codex"`.
3. Якщо користувач просить **Codex OAuth/автентифікацію підписки у звичайному OpenClaw
   runner**, використовуйте `openai-codex/<model>` і залиште середовище виконання як PI.
4. Якщо користувач явно каже **ACP**, **acpx** або **адаптер Codex ACP**, використовуйте
   ACP з `runtime: "acp"` і `agentId: "codex"`.
5. Якщо запит стосується **Claude Code, Gemini CLI, OpenCode, Cursor, Droid або
   іншого зовнішнього harness**, використовуйте ACP/acpx, а не нативне середовище виконання під-агента.

| Ви маєте на увазі...                    | Використовуйте...                            |
| --------------------------------------- | -------------------------------------------- |
| Керування чатом/потоком Codex app-server | `/codex ...` із вбудованого Plugin `codex` |
| Вбудоване середовище виконання агента Codex app-server | `agentRuntime.id: "codex"`                   |
| OpenAI Codex OAuth на PI runner         | `openai-codex/*` model refs                  |
| Claude Code або інший зовнішній harness | ACP/acpx                                     |

Про поділ префіксів родини OpenAI див. [OpenAI](/uk/providers/openai) і
[Провайдери моделей](/uk/concepts/model-providers). Про контракт підтримки середовища виконання Codex
див. [Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Власність середовища виконання

Різні середовища виконання володіють різними частинами циклу.

| Поверхня                    | Вбудований OpenClaw PI                  | Codex app-server                                                            |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| Власник циклу моделі        | OpenClaw через вбудований PI runner     | Codex app-server                                                            |
| Канонічний стан потоку      | Transcript OpenClaw                     | Потік Codex плюс дзеркало transcript OpenClaw                               |
| Динамічні інструменти OpenClaw | Нативний цикл інструментів OpenClaw  | Передаються через адаптер Codex                                             |
| Нативні інструменти shell і файлів | Шлях PI/OpenClaw                 | Нативні інструменти Codex, передані через нативні hooks там, де підтримується |
| Двигун контексту            | Нативне складання контексту OpenClaw    | OpenClaw проектує зібраний контекст у хід Codex                             |
| Compaction                  | OpenClaw або вибраний двигун контексту  | Нативна compaction Codex, зі сповіщеннями OpenClaw і підтриманням дзеркала |
| Доставка каналом            | OpenClaw                                | OpenClaw                                                                    |

Цей поділ власності є головним правилом проєктування:

- Якщо OpenClaw володіє поверхнею, OpenClaw може надавати звичайну поведінку hook Plugin.
- Якщо нативне середовище виконання володіє поверхнею, OpenClaw потребує подій середовища виконання або нативних hooks.
- Якщо нативне середовище виконання володіє канонічним станом потоку, OpenClaw має дзеркалити і проєктувати контекст, а не переписувати непідтримувані внутрішні дані.

## Вибір середовища виконання

OpenClaw вибирає вбудоване середовище виконання після визначення провайдера і моделі:

1. Записане середовище виконання сесії має пріоритет. Зміни конфігурації не перемикають гарячо
   наявний transcript на іншу нативну систему потоків.
2. `OPENCLAW_AGENT_RUNTIME=<id>` примусово задає це середовище виконання для нових або скинутих сесій.
3. `agents.defaults.agentRuntime.id` або `agents.list[].agentRuntime.id` можуть задавати
   `auto`, `pi`, зареєстрований ідентифікатор вбудованого harness, такий як `codex`, або
   підтримуваний alias CLI-бекенду, такий як `claude-cli`.
4. У режимі `auto` зареєстровані середовища виконання Plugin можуть заявляти підтримувані пари провайдер/модель.
5. Якщо жодне середовище виконання не заявляє хід у режимі `auto`, OpenClaw використовує PI як
   середовище виконання сумісності. Використовуйте явний ідентифікатор середовища виконання, коли запуск має бути
   строгим.

Явні середовища виконання Plugin завершуються закрито. Наприклад, `agentRuntime.id: "codex"`
означає Codex або чітку помилку вибору/середовища виконання; воно ніколи не маршрутизується мовчки назад
до PI.

Alias CLI-бекендів відрізняються від ідентифікаторів вбудованих harness. Бажана
форма Claude CLI така:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

Застарілі посилання, такі як `claude-cli/claude-opus-4-7`, залишаються підтримуваними для
сумісності, але нова конфігурація має зберігати провайдер/модель канонічними і поміщати
бекенд виконання в `agentRuntime.id`.

Режим `auto` навмисно консервативний. Середовища виконання Plugin можуть заявляти
пари провайдер/модель, які вони розуміють, але Plugin Codex не заявляє
провайдера `openai-codex` у режимі `auto`. Це зберігає
`openai-codex/*` як явний маршрут PI Codex OAuth і уникає мовчазного
перенесення конфігурацій автентифікації підписки на нативний app-server harness.

Якщо `openclaw doctor` попереджає, що Plugin `codex` увімкнено, тоді як
`openai-codex/*` усе ще маршрутизується через PI, сприймайте це як діагноз, а не
міграцію. Залиште конфігурацію без змін, коли PI Codex OAuth - це те, що вам потрібно.
Перемикайтеся на `openai/<model>` плюс `agentRuntime.id: "codex"` лише тоді, коли хочете нативне
виконання Codex app-server.

## Контракт сумісності

Коли середовище виконання не є PI, воно має документувати, які поверхні OpenClaw підтримує.
Використовуйте цю форму для документації середовища виконання:

| Питання                               | Чому це важливо                                                                                    |
| ------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Хто володіє циклом моделі?            | Визначає, де відбуваються повтори, продовження інструментів і рішення про фінальну відповідь.      |
| Хто володіє канонічною історією потоку? | Визначає, чи OpenClaw може редагувати історію, чи лише дзеркалити її.                             |
| Чи працюють динамічні інструменти OpenClaw? | Messaging, сесії, cron і інструменти, якими володіє OpenClaw, покладаються на це.             |
| Чи працюють hooks динамічних інструментів? | Plugins очікують `before_tool_call`, `after_tool_call` і middleware навколо інструментів, якими володіє OpenClaw. |
| Чи працюють hooks нативних інструментів? | Shell, patch та інструменти, якими володіє середовище виконання, потребують підтримки нативних hook для політики й спостереження. |
| Чи запускається життєвий цикл двигуна контексту? | Memory і context plugins залежать від життєвого циклу assemble, ingest, after-turn і compaction. |
| Які дані compaction відкриваються?    | Деяким plugins потрібні лише сповіщення, тоді як іншим потрібні метадані kept/dropped.            |
| Що навмисно не підтримується?         | Користувачі не повинні припускати еквівалентність PI там, де нативне середовище виконання володіє більшим станом. |

Контракт підтримки середовища виконання Codex задокументовано в
[Codex harness](/uk/plugins/codex-harness#v1-support-contract).

## Мітки стану

Вивід стану може показувати обидві мітки: `Execution` і `Runtime`. Сприймайте їх як
діагностику, а не як назви провайдерів.

- Посилання на модель, як-от `openai/gpt-5.5`, повідомляє вибраного провайдера/модель.
- Ідентифікатор середовища виконання, як-от `codex`, повідомляє, який цикл виконує хід.
- Мітка каналу, як-от Telegram або Discord, повідомляє, де відбувається розмова.

Якщо після зміни конфігурації середовища виконання сеанс усе ще показує PI, почніть новий сеанс
за допомогою `/new` або очистьте поточний за допомогою `/reset`. Наявні сеанси зберігають своє
записане середовище виконання, щоб transcript не відтворювався через дві несумісні нативні
системи сеансів.
