# Streaming + chunking

OpenClaw має два окремі шари streaming:

- **Блоковий streaming (канали):** надсилає завершені **блоки**, коли асистент їх пише. Це звичайні повідомлення каналу (не дельти токенів).
- **Попередній перегляд streaming (Telegram/Discord/Slack):** оновлює тимчасове **повідомлення попереднього перегляду** під час генерації.

Справжнього streaming дельт токенів до повідомлень каналу наразі **немає**. Streaming попереднього перегляду базується на повідомленнях (надсилання + редагування/додавання).

## Блоковий streaming (повідомлення каналу)

Блоковий streaming надсилає вивід асистента великими фрагментами в міру його появи.

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

Легенда:

- `text_delta/events`: події потокового виводу моделі (можуть бути рідкими для моделей без streaming).
- `chunker`: `EmbeddedBlockChunker`, який застосовує мінімальні/максимальні межі + пріоритет розриву.
- `channel send`: фактичні вихідні повідомлення (блокові відповіді).

**Керування:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (типово вимкнено).
- Перевизначення каналів: `*.blockStreaming` (і варіанти для окремих облікових записів), щоб примусово ввімкнути `"on"`/вимкнути `"off"` для кожного каналу.
- `agents.defaults.blockStreamingBreak`: `"text_end"` або `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (об’єднання потокових блоків перед надсиланням).
- Жорстке обмеження каналу: `*.textChunkLimit` (наприклад, `channels.whatsapp.textChunkLimit`).
- Режим chunking каналу: `*.chunkMode` (`length` типово, `newline` розбиває за порожніми рядками (межі абзаців) перед chunking за довжиною).
- М’яке обмеження Discord: `channels.discord.maxLinesPerMessage` (типово 17) розбиває високі відповіді, щоб уникнути обрізання в UI.

**Семантика меж:**

- `text_end`: надсилати блоки в потоці, щойно їх видає chunker; flush на кожному `text_end`.
- `message_end`: чекати, поки повідомлення асистента завершиться, а потім flush накопиченого виводу.

`message_end` усе одно використовує chunker, якщо накопичений текст перевищує `maxChars`, тому наприкінці він може видати кілька chunk.

### Доставка медіа з блоковим streaming

Директиви `MEDIA:` — це звичайні метадані доставки. Коли блоковий streaming рано надсилає
медіаблок, OpenClaw запам’ятовує цю доставку для поточного ходу. Якщо фінальний
payload асистента повторює ту саму URL-адресу медіа, фінальна доставка прибирає
дубльоване медіа замість повторного надсилання вкладення.

Точні дублікати фінального payload пригнічуються. Якщо фінальний payload додає
окремий текст навколо медіа, яке вже було передано потоком, OpenClaw усе одно надсилає
новий текст, зберігаючи одноразову доставку медіа. Це запобігає дублюванню
голосових повідомлень або файлів у каналах на кшталт Telegram, коли агент надсилає `MEDIA:` під час
streaming, а провайдер також включає його в завершену відповідь.

## Алгоритм chunking (нижня/верхня межі)

Блоковий chunking реалізований у `EmbeddedBlockChunker`:

- **Нижня межа:** не видавати, доки буфер < `minChars` (якщо не примусово).
- **Верхня межа:** віддавати перевагу розбиттю до `maxChars`; якщо примусово, розбивати на `maxChars`.
- **Пріоритет розриву:** `paragraph` → `newline` → `sentence` → `whitespace` → жорсткий розрив.
- **Огородження коду:** ніколи не розбивати всередині fenced-блоків; якщо примусове розбиття на `maxChars`, закрити й знову відкрити блок, щоб Markdown залишався коректним.

`maxChars` обмежується значенням `textChunkLimit` каналу, тому перевищити обмеження конкретного каналу не можна.

## Coalescing (об’єднання потокових блоків)

Коли блоковий streaming увімкнено, OpenClaw може **об’єднувати послідовні chunk блоків**
перед їх надсиланням. Це зменшує «спам із одного рядка», водночас зберігаючи
поступовий вивід.

- Coalescing чекає **пауз без активності** (`idleMs`) перед flush.
- Буфери обмежені `maxChars` і будуть flush, якщо перевищать це значення.
- `minChars` не дає надсилати крихітні фрагменти, доки не накопичиться достатньо тексту
  (фінальний flush завжди надсилає залишковий текст).
- Joiner визначається з `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`, `newline` → `\n`, `sentence` → пробіл).
- Перевизначення на рівні каналу доступні через `*.blockStreamingCoalesce` (зокрема для конфігурацій окремих облікових записів).
- Типове значення coalesce `minChars` збільшується до 1500 для Signal/Slack/Discord, якщо не перевизначено.

## Пауза в людському стилі між блоками

Коли блоковий streaming увімкнено, можна додати **випадкову паузу** між
блоковими відповідями (після першого блоку). Це робить відповіді з кількох бульбашок
природнішими.

- Конфігурація: `agents.defaults.humanDelay` (перевизначення для агента через `agents.list[].humanDelay`).
- Режими: `off` (типово), `natural` (800–2500 мс), `custom` (`minMs`/`maxMs`).
- Застосовується лише до **блокових відповідей**, а не до фінальних відповідей чи підсумків інструментів.

## "Транслювати chunk або все"

Це відповідає таким налаштуванням:

- **Транслювати chunk:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (надсилати в міру появи). Для каналів, відмінних від Telegram, також потрібно `*.blockStreaming: true`.
- **Транслювати все наприкінці:** `blockStreamingBreak: "message_end"` (один flush, але, можливо, кілька chunk, якщо текст дуже довгий).
- **Без блокового streaming:** `blockStreamingDefault: "off"` (лише фінальна відповідь).

**Примітка щодо каналів:** Блоковий streaming **вимкнено, доки**
`*.blockStreaming` не встановлено явно в `true`. Канали можуть транслювати живий попередній перегляд
(`channels.<channel>.streaming`) без блокових відповідей.

Нагадування про розташування конфігурації: типові значення `blockStreaming*` розміщені в `agents.defaults`, а не в кореневій конфігурації.

## Режими streaming попереднього перегляду

Канонічний ключ: `channels.<channel>.streaming`

Режими:

- `off`: вимкнути streaming попереднього перегляду.
- `partial`: один попередній перегляд, який замінюється найновішим текстом.
- `block`: оновлення попереднього перегляду chunk-ами/додаваннями.
- `progress`: попередній перегляд прогресу/статусу під час генерації, фінальна відповідь після завершення.

### Відповідність каналів

| Канал      | `off` | `partial` | `block` | `progress`        |
| ---------- | ----- | --------- | ------- | ----------------- |
| Telegram   | ✅    | ✅        | ✅      | зіставляється з `partial` |
| Discord    | ✅    | ✅        | ✅      | зіставляється з `partial` |
| Slack      | ✅    | ✅        | ✅      | ✅                |
| Mattermost | ✅    | ✅        | ✅      | ✅                |

Лише для Slack:

- `channels.slack.streaming.nativeTransport` перемикає виклики нативного API streaming Slack, коли `channels.slack.streaming.mode="partial"` (типово: `true`).
- Нативний streaming Slack і статус потоку помічника Slack потребують цільового потоку відповіді; DM верхнього рівня не показують такий попередній перегляд у стилі потоку.

Міграція застарілих ключів:

- Telegram: `streamMode` і булеве `streaming` автоматично мігрують у enum `streaming`.
- Discord: `streamMode` і булеве `streaming` автоматично мігрують у enum `streaming`.
- Slack: `streamMode` автоматично мігрує до `streaming.mode`; булеве `streaming` автоматично мігрує до `streaming.mode` плюс `streaming.nativeTransport`; застаріле `nativeStreaming` автоматично мігрує до `streaming.nativeTransport`.

### Поведінка під час виконання

Telegram:

- Використовує `sendMessage` + `editMessageText` для оновлення попереднього перегляду в DM, групах і темах.
- Streaming попереднього перегляду пропускається, якщо блоковий streaming Telegram явно увімкнено (щоб уникнути подвійного streaming).
- `/reasoning stream` може записувати reasoning у попередній перегляд.

Discord:

- Використовує надсилання + редагування повідомлень попереднього перегляду.
- Режим `block` використовує draft chunking (`draftChunk`).
- Streaming попереднього перегляду пропускається, якщо блоковий streaming Discord явно увімкнено.
- Фінальні payload медіа, помилок і явних відповідей скасовують очікувані попередні перегляди без flush нового чернеткового повідомлення, а потім використовують звичайну доставку.

Slack:

- `partial` може використовувати нативний streaming Slack (`chat.startStream`/`append`/`stop`), коли він доступний.
- `block` використовує попередні перегляди чернеток у стилі додавання.
- `progress` використовує текст попереднього перегляду статусу, а потім фінальну відповідь.
- Нативний і чернетковий streaming попереднього перегляду пригнічують блокові відповіді для цього ходу, тому відповідь Slack передається потоком лише одним шляхом доставки.
- Фінальні payload медіа/помилок і фінали progress не створюють тимчасові чернеткові повідомлення; лише текстові/блокові фінали, які можуть редагувати попередній перегляд, виконують flush очікуваного тексту чернетки.

Mattermost:

- Транслює thinking, активність інструментів і частковий текст відповіді в один чернетковий допис попереднього перегляду, який фіналізується на місці, коли фінальну відповідь уже безпечно надсилати.
- Повертається до надсилання нового фінального допису, якщо допис попереднього перегляду було видалено або він недоступний на момент фіналізації.
- Фінальні payload медіа/помилок скасовують очікувані оновлення попереднього перегляду перед звичайною доставкою, замість flush тимчасового допису попереднього перегляду.

Matrix:

- Чернеткові попередні перегляди фіналізуються на місці, коли фінальний текст може повторно використати подію попереднього перегляду.
- Фінали лише з медіа, помилки та фінали з невідповідністю цілі відповіді скасовують очікувані оновлення попереднього перегляду перед звичайною доставкою; уже видимий застарілий попередній перегляд редагується.

### Оновлення попереднього перегляду прогресу інструментів

Streaming попереднього перегляду також може включати оновлення **tool-progress** — короткі рядки стану на кшталт «пошук у вебі», «читання файлу» або «виклик інструмента», які з’являються в тому самому повідомленні попереднього перегляду під час роботи інструментів, ще до фінальної відповіді. Це підтримує візуальну активність багатоетапних ходів із інструментами, замість тиші між першим попереднім переглядом thinking і фінальною відповіддю.

Підтримувані поверхні:

- **Discord**, **Slack** і **Telegram** типово транслюють tool-progress у живе редагування попереднього перегляду, коли streaming попереднього перегляду активний.
- У Telegram оновлення попереднього перегляду tool-progress уже постачаються ввімкненими з `v2026.4.22`; збереження їх увімкненими підтримує цю вже випущену поведінку.
- **Mattermost** уже включає активність інструментів у свій єдиний чернетковий допис попереднього перегляду (див. вище).
- Редагування tool-progress дотримуються активного режиму streaming попереднього перегляду; вони пропускаються, коли streaming попереднього перегляду встановлено в `off` або коли керування повідомленням перейняв блоковий streaming.
- Щоб зберегти streaming попереднього перегляду, але приховати рядки tool-progress, встановіть `streaming.preview.toolProgress` у `false` для цього каналу. Щоб повністю вимкнути редагування попереднього перегляду, встановіть `streaming.mode` у `off`.

Приклад:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": false
        }
      }
    }
  }
}
```
