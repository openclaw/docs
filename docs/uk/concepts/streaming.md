# Streaming + chunking

OpenClaw має два окремі рівні streaming:

- **Блокове streaming (канали):** надсилає завершені **блоки**, поки асистент пише. Це звичайні повідомлення каналу (не дельти токенів).
- **Streaming попереднього перегляду (Telegram/Discord/Slack):** оновлює тимчасове **повідомлення попереднього перегляду** під час генерації.

Справжнього streaming дельт токенів до повідомлень каналу сьогодні **немає**. Streaming попереднього перегляду є повідомлення-орієнтованим (надсилання + редагування/додавання).

## Блокове streaming (повідомлення каналу)

Блокове streaming надсилає вивід асистента грубими порціями, щойно він стає доступним.

```
Вивід моделі
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker надсилає блоки, коли буфер зростає
       └─ (blockStreamingBreak=message_end)
            └─ chunker скидає вміст на message_end
                   └─ надсилання каналу (блокові відповіді)
```

Легенда:

- `text_delta/events`: події потоку моделі (можуть бути нечастими для моделей без streaming).
- `chunker`: `EmbeddedBlockChunker`, що застосовує мінімальні/максимальні межі + пріоритет розриву.
- `channel send`: фактичні вихідні повідомлення (блокові відповіді).

**Керування:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (типово вимкнено).
- Перевизначення каналу: `*.blockStreaming` (і варіанти для окремого акаунта), щоб примусово встановити `"on"`/`"off"` для кожного каналу.
- `agents.defaults.blockStreamingBreak`: `"text_end"` або `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (об’єднує streamed-блоки перед надсиланням).
- Жорстке обмеження каналу: `*.textChunkLimit` (наприклад, `channels.whatsapp.textChunkLimit`).
- Режим chunking каналу: `*.chunkMode` (`length` типово, `newline` розбиває за порожніми рядками (межами абзаців) перед chunking за довжиною).
- М’яке обмеження Discord: `channels.discord.maxLinesPerMessage` (типово 17) розбиває високі відповіді, щоб уникнути обрізання в UI.

**Семантика меж:**

- `text_end`: передавати блоки, щойно їх надсилає chunker; скидати на кожному `text_end`.
- `message_end`: чекати, доки повідомлення асистента завершиться, а потім скидати буферизований вивід.

`message_end` усе одно використовує chunker, якщо буферизований текст перевищує `maxChars`, тому наприкінці він може надсилати кілька chunk.

## Алгоритм chunking (нижня/верхня межі)

Блоковий chunking реалізовано в `EmbeddedBlockChunker`:

- **Нижня межа:** не надсилати, доки буфер < `minChars` (якщо не примусово).
- **Верхня межа:** віддавати перевагу розривам перед `maxChars`; якщо примусово, розбивати на `maxChars`.
- **Пріоритет розриву:** `paragraph` → `newline` → `sentence` → `whitespace` → жорсткий розрив.
- **Огорожі коду:** ніколи не розбивати всередині огорож; якщо примусово на `maxChars`, закривати + знову відкривати огорожу, щоб Markdown залишався валідним.

`maxChars` обмежується значенням `textChunkLimit` каналу, тому перевищити ліміти конкретного каналу не можна.

## Coalescing (об’єднання streamed-блоків)

Коли блокове streaming увімкнено, OpenClaw може **об’єднувати послідовні блокові chunk**
перед їх надсиланням. Це зменшує «спам із одного рядка», водночас зберігаючи
поступове виведення.

- Coalescing чекає **пауз бездіяльності** (`idleMs`) перед скиданням.
- Буфери обмежуються `maxChars` і скидаються, якщо перевищують його.
- `minChars` запобігає надсиланню крихітних фрагментів, доки не накопичиться достатньо тексту
  (остаточне скидання завжди надсилає решту тексту).
- Об’єднувач визначається з `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`, `newline` → `\n`, `sentence` → пробіл).
- Перевизначення для каналу доступні через `*.blockStreamingCoalesce` (включно з конфігураціями для окремих акаунтів).
- Типове значення coalesce `minChars` підвищується до 1500 для Signal/Slack/Discord, якщо не перевизначено.

## Природне сповільнення між блоками

Коли блокове streaming увімкнено, можна додати **випадкову паузу** між
блоковими відповідями (після першого блока). Це робить відповіді з кількох бульбашок
природнішими.

- Конфігурація: `agents.defaults.humanDelay` (перевизначається для агента через `agents.list[].humanDelay`).
- Режими: `off` (типово), `natural` (800–2500 мс), `custom` (`minMs`/`maxMs`).
- Застосовується лише до **блокових відповідей**, а не до фінальних відповідей чи зведень інструментів.

## «Stream chunks або все одразу»

Це зіставляється так:

- **Stream chunks:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (надсилати в процесі). Для каналів, окрім Telegram, також потрібне `*.blockStreaming: true`.
- **Stream everything at end:** `blockStreamingBreak: "message_end"` (скинути один раз, можливо, кількома chunk, якщо дуже довго).
- **Без блокового streaming:** `blockStreamingDefault: "off"` (лише фінальна відповідь).

**Примітка щодо каналів:** Блокове streaming **вимкнено, якщо тільки**
`*.blockStreaming` явно не встановлено в `true`. Канали можуть передавати живий попередній перегляд
(`channels.<channel>.streaming`) без блокових відповідей.

Нагадування щодо розташування конфігурації: типові значення `blockStreaming*` містяться в `agents.defaults`, а не в корені конфігурації.

## Режими streaming попереднього перегляду

Канонічний ключ: `channels.<channel>.streaming`

Режими:

- `off`: вимкнути streaming попереднього перегляду.
- `partial`: один попередній перегляд, який замінюється найновішим текстом.
- `block`: попередній перегляд оновлюється chunk-ами або кроками додавання.
- `progress`: попередній перегляд прогресу/статусу під час генерації, фінальна відповідь після завершення.

### Зіставлення каналів

| Канал      | `off` | `partial` | `block` | `progress`        |
| ---------- | ----- | --------- | ------- | ----------------- |
| Telegram   | ✅    | ✅        | ✅      | зіставляється з `partial` |
| Discord    | ✅    | ✅        | ✅      | зіставляється з `partial` |
| Slack      | ✅    | ✅        | ✅      | ✅                |
| Mattermost | ✅    | ✅        | ✅      | ✅                |

Лише для Slack:

- `channels.slack.streaming.nativeTransport` перемикає нативні виклики Slack streaming API, коли `channels.slack.streaming.mode="partial"` (типово: `true`).
- Нативне streaming Slack і статус потоків помічника Slack потребують цільового потоку відповіді; DM верхнього рівня не показують такий попередній перегляд у стилі потоку.

Міграція застарілих ключів:

- Telegram: `streamMode` + булеве `streaming` автоматично мігрують до enum `streaming`.
- Discord: `streamMode` + булеве `streaming` автоматично мігрують до enum `streaming`.
- Slack: `streamMode` автоматично мігрує до `streaming.mode`; булеве `streaming` автоматично мігрує до `streaming.mode` плюс `streaming.nativeTransport`; застаріле `nativeStreaming` автоматично мігрує до `streaming.nativeTransport`.

### Поведінка під час виконання

Telegram:

- Використовує оновлення попереднього перегляду через `sendMessage` + `editMessageText` у DM, групах і темах.
- Streaming попереднього перегляду пропускається, коли блокове streaming Telegram явно увімкнено (щоб уникнути подвійного streaming).
- `/reasoning stream` може записувати міркування в попередній перегляд.

Discord:

- Використовує повідомлення попереднього перегляду через надсилання + редагування.
- Режим `block` використовує draft chunking (`draftChunk`).
- Streaming попереднього перегляду пропускається, коли блокове streaming Discord явно увімкнено.

Slack:

- `partial` може використовувати нативне streaming Slack (`chat.startStream`/`append`/`stop`), коли воно доступне.
- `block` використовує попередні перегляди чернетки у стилі додавання.
- `progress` використовує текст попереднього перегляду статусу, а потім фінальну відповідь.

Mattermost:

- Передає thinking, активність інструментів і частковий текст відповіді в один чернетковий допис попереднього перегляду, який фіналізується на місці, коли фінальну відповідь безпечно надсилати.
- Повертається до надсилання нового фінального допису, якщо допис попереднього перегляду було видалено або він інакше недоступний під час фіналізації.

### Оновлення попереднього перегляду прогресу інструментів

Streaming попереднього перегляду також може включати оновлення **tool-progress** — короткі рядки стану на кшталт "searching the web", "reading file" або "calling tool", — які з’являються в тому самому повідомленні попереднього перегляду, поки інструменти працюють, ще до фінальної відповіді. Це робить багатокрокові ходи з інструментами візуально активними, а не мовчазними між першим thinking-попереднім переглядом і фінальною відповіддю.

Підтримувані поверхні:

- **Discord**, **Slack** і **Telegram** передають tool-progress у редагування живого попереднього перегляду.
- **Mattermost** уже вбудовує активність інструментів у свій єдиний чернетковий допис попереднього перегляду (див. вище).
- Редагування tool-progress дотримуються активного режиму streaming попереднього перегляду; вони пропускаються, коли streaming попереднього перегляду має значення `off` або коли блокове streaming уже перебрало керування повідомленням.
