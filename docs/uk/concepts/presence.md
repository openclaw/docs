  Присутність OpenClaw (“presence”) — це полегшене, best-effort представлення:

  - самого **Gateway**, і
  - **клієнтів, підключених до Gateway** (mac app, WebChat, CLI тощо)

  Присутність використовується насамперед для відображення вкладки **Instances** в застосунку macOS і для
  швидкої видимості стану для оператора.

  ## Поля присутності (що відображається)

  Записи присутності — це структуровані об’єкти з такими полями:

  - `instanceId` (необов’язкове, але наполегливо рекомендоване): стабільна ідентичність клієнта (зазвичай `connect.client.instanceId`)
  - `host`: зрозуміла людині назва хоста
  - `ip`: IP-адреса в режимі best-effort
  - `version`: рядок версії клієнта
  - `deviceFamily` / `modelIdentifier`: підказки про апаратне забезпечення
  - `mode`: `ui`, `webchat`, `cli`, `backend`, `probe`, `test`, `node`, ...
  - `lastInputSeconds`: “секунди з часу останнього введення користувача” (якщо відомо)
  - `reason`: `self`, `connect`, `node-connected`, `periodic`, ...
  - `ts`: часова позначка останнього оновлення (мс від епохи)

  ## Джерела (звідки походить присутність)

  Записи присутності створюються кількома джерелами й **об’єднуються**.

  ### 1) Власний запис Gateway

  Gateway завжди створює початковий власний запис “self” під час запуску, щоб UI показував хост gateway
  ще до підключення будь-яких клієнтів.

  ### 2) Підключення WebSocket

  Кожен клієнт WS починає з запиту `connect`. Після успішного handshake
  Gateway виконує upsert запису присутності для цього підключення.

  #### Чому одноразові команди CLI не відображаються

  CLI часто підключається для коротких одноразових команд. Щоб не засмічувати
  список Instances, `client.mode === "cli"` **не** перетворюється на запис присутності.

  ### 3) Beacon-и `system-event`

  Клієнти можуть надсилати багатші періодичні beacon-и через метод `system-event`. mac
  app використовує це, щоб повідомляти назву хоста, IP та `lastInputSeconds`.

  ### 4) Підключення Node (role: node)

  Коли Node підключається через WebSocket Gateway з `role: node`, Gateway
  виконує upsert запису присутності для цього Node (той самий процес, що й для інших клієнтів WS).

  ## Правила об’єднання + дедуплікації (чому важливий `instanceId`)

  Записи присутності зберігаються в одному map у пам’яті:

  - Записи мають ключ у вигляді **ключа присутності**.
  - Найкращий ключ — стабільний `instanceId` (з `connect.client.instanceId`), який переживає перезапуски.
  - Ключі нечутливі до регістру.

  Якщо клієнт перепідключається без стабільного `instanceId`, він може відображатися
  **дубльованим** рядком.

  ## TTL і обмежений розмір

  Присутність навмисно є ефемерною:

  - **TTL:** записи, старші за 5 хвилин, видаляються
  - **Максимум записів:** 200 (найстаріші видаляються першими)

  Це підтримує список свіжим і запобігає необмеженому зростанню використання пам’яті.

  ## Застереження щодо remote/tunnel (loopback IP-адреси)

  Коли клієнт підключається через SSH tunnel / локальне перенаправлення порту, Gateway може
  бачити віддалену адресу як `127.0.0.1`. Щоб не перезаписати коректну IP-адресу, повідомлену клієнтом,
  віддалені loopback-адреси ігноруються.

  ## Споживачі

  ### Вкладка Instances у macOS

  macOS app відображає результат `system-presence` і застосовує невеликий індикатор
  стану (Active/Idle/Stale) на основі віку останнього оновлення.

  ## Поради з налагодження

  - Щоб побачити сирий список, викличте `system-presence` до Gateway.
  - Якщо ви бачите дублікати:
  - переконайтеся, що клієнти надсилають стабільний `client.instanceId` під час handshake
  - переконайтеся, що періодичні beacon-и використовують той самий `instanceId`
  - перевірте, чи в записі, похідному від підключення, не відсутній `instanceId` (у такому разі дублікати очікувані)
