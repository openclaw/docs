OpenClaw запускає єдиний вбудований runtime агента.

## Workspace (обов’язково)

OpenClaw використовує єдиний каталог workspace агента (`agents.defaults.workspace`) як **єдиний** робочий каталог агента (`cwd`) для інструментів і контексту.

Рекомендовано: використовуйте `openclaw setup`, щоб створити `~/.openclaw/openclaw.json`, якщо його немає, і ініціалізувати файли workspace.

Повний макет workspace + посібник із резервного копіювання: [Workspace агента](/uk/concepts/agent-workspace)

Якщо ввімкнено `agents.defaults.sandbox`, не-main session можуть перевизначати це через
workspace для кожної session у `agents.defaults.sandbox.workspaceRoot` (див.
[Конфігурація Gateway](/uk/gateway/configuration)).

## Bootstrap-файли (впроваджуються)

Усередині `agents.defaults.workspace` OpenClaw очікує такі редаговані користувачем файли:

- `AGENTS.md` — інструкції з роботи + «memory»
- `SOUL.md` — persona, межі, тон
- `TOOLS.md` — нотатки про інструменти, які підтримує користувач (наприклад, `imsg`, `sag`, conventions)
- `BOOTSTRAP.md` — одноразовий ритуал першого запуску (видаляється після завершення)
- `IDENTITY.md` — ім’я/вайб/emoji агента
- `USER.md` — профіль користувача + бажана форма звертання

На першому ході нової session OpenClaw безпосередньо впроваджує вміст цих файлів у контекст агента.

Порожні файли пропускаються. Великі файли обрізаються та усікаються з маркером, щоб prompt-и залишалися компактними (прочитайте файл, щоб побачити повний вміст).

Якщо файлу немає, OpenClaw впроваджує один рядок-маркер «файл відсутній» (а `openclaw setup` створить безпечний типовий шаблон).

`BOOTSTRAP.md` створюється лише для **абсолютно нового workspace** (коли відсутні інші bootstrap-файли). Якщо ви видалите його після завершення ритуалу, він не повинен створюватися знову під час наступних перезапусків.

Щоб повністю вимкнути створення bootstrap-файлів (для попередньо підготовлених workspace), задайте:

```json5
{ agent: { skipBootstrap: true } }
```

## Вбудовані інструменти

Базові інструменти (read/exec/edit/write та пов’язані системні інструменти) завжди доступні,
з урахуванням політики інструментів. `apply_patch` — необов’язковий інструмент, який шлюзується через
`tools.exec.applyPatch`. `TOOLS.md` **не** керує тим, які інструменти існують; це
вказівки щодо того, як _ви_ хочете, щоб їх використовували.

## Skills

OpenClaw завантажує Skills із таких розташувань (від найвищого пріоритету до нижчого):

- Workspace: `<workspace>/skills`
- Skills агента проєкту: `<workspace>/.agents/skills`
- Особисті Skills агента: `~/.agents/skills`
- Керовані/локальні: `~/.openclaw/skills`
- Вбудовані (постачаються разом із встановленням)
- Додаткові папки Skills: `skills.load.extraDirs`

Skills можуть шлюзуватися через config/env (див. `skills` у [Конфігурація Gateway](/uk/gateway/configuration)).

## Межі runtime

Вбудований runtime агента побудований на ядрі агента Pi (моделі, інструменти та
pipeline prompt-ів). Керування session, виявлення, підключення інструментів і доставка через канали —
це шари, якими володіє OpenClaw поверх цього ядра.

## Sessions

Транскрипти session зберігаються як JSONL у:

- `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

Ідентифікатор session є сталим і вибирається OpenClaw.
Застарілі папки session з інших інструментів не читаються.

## Керування під час streaming

Коли режим черги — `steer`, вхідні повідомлення впроваджуються в поточний запуск.
Поставлене в чергу керування доставляється **після того, як поточний хід асистента завершить
виконання своїх викликів інструментів**, перед наступним викликом LLM. Керування більше не пропускає
решту викликів інструментів із поточного повідомлення асистента; натомість воно впроваджує поставлене в чергу
повідомлення на наступній межі моделі.

Коли режим черги — `followup` або `collect`, вхідні повідомлення утримуються, доки
поточний хід не завершиться, а потім починається новий хід агента з поставленими в чергу payload-ами. Див.
[Черга](/uk/concepts/queue) щодо режимів і поведінки debounce/cap.

Block streaming надсилає завершені блоки асистента щойно вони закінчуються; він
**типово вимкнений** (`agents.defaults.blockStreamingDefault: "off"`).
Налаштовуйте межу через `agents.defaults.blockStreamingBreak` (`text_end` або `message_end`; типово `text_end`).
Керуйте м’яким розбиттям блоків на фрагменти через `agents.defaults.blockStreamingChunk` (типово
800–1200 символів; надає перевагу розривам абзаців, потім новим рядкам; речення — в останню чергу).
Об’єднуйте streamed-фрагменти через `agents.defaults.blockStreamingCoalesce`, щоб зменшити
спам з одиночних рядків (об’єднання перед надсиланням на основі idle). Канали, відмінні від Telegram, вимагають
явного `*.blockStreaming: true`, щоб увімкнути block replies.
Докладні summary інструментів публікуються на старті інструмента (без debounce); UI Control
stream-ить вивід інструментів через події агента, коли це доступно.
Докладніше: [Streaming + chunking](/uk/concepts/streaming).

## Посилання на моделі

Посилання на моделі в конфігурації (наприклад, `agents.defaults.model` і `agents.defaults.models`) розбираються шляхом поділу за **першим** `/`.

- Використовуйте `provider/model` під час налаштування моделей.
- Якщо сам ідентифікатор моделі містить `/` (у стилі OpenRouter), додавайте префікс provider (приклад: `openrouter/moonshotai/kimi-k2`).
- Якщо ви пропускаєте provider, OpenClaw спочатку пробує alias, потім унікальний
  збіг налаштованого provider для цього точного model id, і лише потім повертається до налаштованого типового provider. Якщо цей provider більше не надає
  налаштовану типову модель, OpenClaw повертається до першого налаштованого
  provider/model замість того, щоб показувати застаріле типове значення видаленого provider.
