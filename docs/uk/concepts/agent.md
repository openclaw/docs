OpenClaw запускає **єдине вбудоване середовище виконання агента** — один процес агента на
Gateway, із власним робочим простором, bootstrap-файлами та сховищем сеансів. Ця сторінка
описує контракт цього середовища виконання: що має містити робочий простір, які файли
впроваджуються та як сеанси bootstrap-ініціалізуються відносно нього.

## Робочий простір (обов’язково)

OpenClaw використовує єдиний каталог робочого простору агента (`agents.defaults.workspace`) як **єдиний** робочий каталог (`cwd`) агента для інструментів і контексту.

Рекомендовано: використайте `openclaw setup`, щоб створити `~/.openclaw/openclaw.json`, якщо його немає, і ініціалізувати файли робочого простору.

Повна структура робочого простору + посібник із резервного копіювання: [Робочий простір агента](/uk/concepts/agent-workspace)

Якщо `agents.defaults.sandbox` увімкнено, неосновні сеанси можуть перевизначити це
робочими просторами для окремих сеансів у `agents.defaults.sandbox.workspaceRoot` (див.
[конфігурацію Gateway](/uk/gateway/configuration)).

## Bootstrap-файли (впроваджуються)

Усередині `agents.defaults.workspace` OpenClaw очікує такі файли, які може редагувати користувач:

- `AGENTS.md` — операційні інструкції + «пам’ять»
- `SOUL.md` — персона, межі, тон
- `TOOLS.md` — нотатки про інструменти, які підтримує користувач (наприклад, `imsg`, `sag`, домовленості)
- `BOOTSTRAP.md` — одноразовий ритуал першого запуску (видаляється після завершення)
- `IDENTITY.md` — ім’я/вайб/емодзі агента
- `USER.md` — профіль користувача + бажана форма звертання

На першому ході нового сеансу OpenClaw впроваджує вміст цих файлів у Project Context системного prompt.

Порожні файли пропускаються. Великі файли обрізаються й скорочуються з маркером, щоб prompt залишався компактним (прочитайте файл, щоб отримати повний вміст).

Якщо файл відсутній, OpenClaw впроваджує один рядок-маркер «файл відсутній» (а `openclaw setup` створить безпечний шаблон за замовчуванням).

`BOOTSTRAP.md` створюється лише для **цілком нового робочого простору** (коли немає інших bootstrap-файлів). Поки він очікує на виконання, OpenClaw тримає його в Project Context і додає bootstrap-настанови системного prompt для початкового ритуалу замість копіювання його в повідомлення користувача. Якщо ви видалите його після завершення ритуалу, він не має створюватися повторно під час наступних перезапусків.

Щоб повністю вимкнути створення bootstrap-файлів (для попередньо наповнених робочих просторів), задайте:

```json5
{ agents: { defaults: { skipBootstrap: true } } }
```

## Вбудовані інструменти

Основні інструменти (read/exec/edit/write і пов’язані системні інструменти) завжди доступні
з урахуванням політики інструментів. `apply_patch` є необов’язковим і обмежується
`tools.exec.applyPatch`. `TOOLS.md` **не** керує тим, які інструменти існують; це
настанови щодо того, як _ви_ хочете, щоб їх використовували.

## Skills

OpenClaw завантажує Skills із цих розташувань (від найвищого пріоритету):

- Робочий простір: `<workspace>/skills`
- Skills агента проєкту: `<workspace>/.agents/skills`
- Особисті Skills агента: `~/.agents/skills`
- Керовані/локальні: `~/.openclaw/skills`
- Вбудовані (постачаються з інсталяцією)
- Додаткові папки Skills: `skills.load.extraDirs`

Skills можуть обмежуватися конфігурацією/env (див. `skills` у [конфігурації Gateway](/uk/gateway/configuration)).

## Межі середовища виконання

Вбудоване середовище виконання агента побудоване на ядрі агента Pi (моделі, інструменти та
prompt-конвеєр). Керування сеансами, виявлення, підключення інструментів і доставлення в канали
є шарами OpenClaw поверх цього ядра.

## Сеанси

Транскрипти сеансів зберігаються як JSONL тут:

- `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

Ідентифікатор сеансу стабільний і вибирається OpenClaw.
Застарілі папки сеансів з інших інструментів не читаються.

## Керування під час streaming

Коли режим черги — `steer`, вхідні повідомлення впроваджуються в поточний запуск.
Керування з черги доставляється **після того, як поточний хід асистента завершить
виконання своїх викликів інструментів**, перед наступним викликом LLM. Pi зчитує всі очікувані
повідомлення керування разом для `steer`; застарілий `queue` зчитує одне повідомлення на
межу моделі. Керування більше не пропускає решту викликів інструментів із поточного
повідомлення асистента.

Коли режим черги — `followup` або `collect`, вхідні повідомлення утримуються до завершення
поточного ходу, потім починається новий хід агента з payload із черги. Див.
[Черга](/uk/concepts/queue) і [Черга керування](/uk/concepts/queue-steering), щоб дізнатися про поведінку
режимів і меж.

Block streaming надсилає завершені блоки асистента одразу після їх завершення; за замовчуванням він
**вимкнений** (`agents.defaults.blockStreamingDefault: "off"`).
Налаштуйте межу через `agents.defaults.blockStreamingBreak` (`text_end` проти `message_end`; за замовчуванням text_end).
Керуйте м’яким розбиттям блоків на фрагменти за допомогою `agents.defaults.blockStreamingChunk` (за замовчуванням
800–1200 символів; надає перевагу розривам абзаців, потім новим рядкам; речення — останні).
Об’єднуйте streamed-фрагменти за допомогою `agents.defaults.blockStreamingCoalesce`, щоб зменшити
спам з окремих рядків (об’єднання на основі простою перед надсиланням). Канали не-Telegram потребують
явного `*.blockStreaming: true`, щоб увімкнути блокові відповіді.
Докладні підсумки інструментів надсилаються на старті інструмента (без debounce); Control UI
транслює вивід інструмента через події агента, коли доступно.
Більше деталей: [Streaming + розбиття на фрагменти](/uk/concepts/streaming).

## Посилання на моделі

Посилання на моделі в конфігурації (наприклад, `agents.defaults.model` і `agents.defaults.models`) розбираються шляхом поділу за **першим** `/`.

- Використовуйте `provider/model` під час налаштування моделей.
- Якщо сам ідентифікатор моделі містить `/` (у стилі OpenRouter), додайте префікс провайдера (приклад: `openrouter/moonshotai/kimi-k2`).
- Якщо ви пропускаєте провайдера, OpenClaw спершу пробує псевдонім, потім унікальний
  збіг налаштованого провайдера для цього точного ідентифікатора моделі, і лише після цього повертається
  до налаштованого провайдера за замовчуванням. Якщо цей провайдер більше не надає
  налаштовану модель за замовчуванням, OpenClaw повертається до першої налаштованої
  пари провайдер/модель замість показу застарілого стандартного значення для видаленого провайдера.
