## Що це таке

- Отримує дані про використання/квоту постачальника безпосередньо з їхніх кінцевих точок використання.
- Жодних оцінених витрат; лише вікна, повідомлені постачальником.
- Зручний для читання вивід стану нормалізовано до `X% left`, навіть коли
  зовнішній API повідомляє спожиту квоту, залишок квоти або лише необроблені лічильники.
- `/status` на рівні сеансу та `session_status` можуть повертатися до останнього
  запису використання в транскрипті, коли поточний знімок сеансу містить мало даних. Цей
  резервний варіант заповнює відсутні лічильники токенів/кешу, може відновити мітку активної
  моделі runtime і віддає перевагу більшому загальному значенню, орієнтованому на промпт, коли метадані
  сеансу відсутні або менші. Наявні ненульові поточні значення все одно мають пріоритет.

## Де це відображається

- `/status` у чатах: насичена емодзі картка стану з токенами сеансу + оціненою вартістю (лише ключ API). Використання постачальника показується для **поточного постачальника моделі**, коли доступне, як нормалізоване вікно `X% left`.
- `/usage off|tokens|full` у чатах: нижній колонтитул використання для кожної відповіді (OAuth показує лише токени).
- `/usage cost` у чатах: локальний підсумок витрат, агрегований із журналів сеансів OpenClaw.
- CLI: `openclaw status --usage` виводить повну деталізацію для кожного постачальника.
- CLI: `openclaw channels list` виводить той самий знімок використання поряд із конфігурацією постачальника (використовуйте `--no-usage`, щоб пропустити).
- Панель меню macOS: розділ “Використання” в Context (лише якщо доступно).

## Постачальники + облікові дані

- **Anthropic (Claude)**: токени OAuth у профілях автентифікації.
- **GitHub Copilot**: токени OAuth у профілях автентифікації.
- **Gemini CLI**: токени OAuth у профілях автентифікації.
  - Використання JSON повертається до `stats`; `stats.cached` нормалізується в
    `cacheRead`.
- **OpenAI Codex**: токени OAuth у профілях автентифікації (accountId використовується, коли присутній).
- **MiniMax**: ключ API або профіль автентифікації MiniMax OAuth. OpenClaw розглядає
  `minimax`, `minimax-cn` і `minimax-portal` як одну й ту саму поверхню квоти MiniMax,
  віддає перевагу збереженому MiniMax OAuth, коли він присутній, а інакше повертається
  до `MINIMAX_CODE_PLAN_KEY`, `MINIMAX_CODING_API_KEY` або `MINIMAX_API_KEY`.
  Опитування використання виводить хост Coding Plan з `models.providers.minimax-portal.baseUrl`
  або `models.providers.minimax.baseUrl`, коли налаштовано, а інакше використовує
  хост MiniMax CN.
  Необроблені поля MiniMax `usage_percent` / `usagePercent` означають **залишок**
  квоти, тому OpenClaw інвертує їх перед відображенням; поля на основі лічильників мають
  пріоритет, коли присутні.
  - Мітки вікна coding-plan беруться з полів годин/хвилин постачальника, коли
    присутні, а потім повертаються до проміжку `start_time` / `end_time`.
  - Якщо кінцева точка coding-plan повертає `model_remains`, OpenClaw віддає перевагу
    запису моделі чату, виводить мітку вікна з часових міток, коли явні
    поля `window_hours` / `window_minutes` відсутні, і включає назву моделі
    в мітку плану.
- **Xiaomi MiMo**: ключ API через env/config/auth store (`XIAOMI_API_KEY`).
- **z.ai**: ключ API через env/config/auth store.

Використання приховується, коли не вдається визначити придатну автентифікацію використання постачальника. Постачальники
можуть надавати plugin-специфічну логіку автентифікації використання; інакше OpenClaw повертається до
відповідних облікових даних OAuth/API-ключів із профілів автентифікації, змінних середовища
або конфігурації.
