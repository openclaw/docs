Кожна модель має вікно контексту: максимальну кількість токенів, яку вона може обробити. Коли розмова наближається до цього ліміту, OpenClaw виконує **Compaction** старих повідомлень у підсумок, щоб чат міг продовжуватися.

## Як це працює

1. Старіші ходи розмови підсумовуються в компактний запис.
2. Підсумок зберігається в транскрипті сесії.
3. Останні повідомлення залишаються без змін.

Коли OpenClaw ділить історію на фрагменти Compaction, він зберігає виклики інструментів асистента в парі з відповідними записами `toolResult`. Якщо точка поділу потрапляє всередину блоку інструмента, OpenClaw зсуває межу так, щоб пара залишалася разом, а поточний непідсумований хвіст зберігався.

Повна історія розмови залишається на диску. Compaction змінює лише те, що модель бачить на наступному ході.

## Автоматична Compaction

Автоматична Compaction увімкнена типово. Вона запускається, коли сесія наближається до ліміту контексту або коли модель повертає помилку переповнення контексту (у такому разі OpenClaw виконує Compaction і повторює спробу).

Ви побачите:

- `🧹 Auto-compaction complete` у докладному режимі.
- `/status`, що показує `🧹 Compactions: <count>`.

<Info>
Перед виконанням Compaction OpenClaw автоматично нагадує агенту зберегти важливі нотатки у файли [memory](/uk/concepts/memory). Це запобігає втраті контексту.
</Info>

<AccordionGroup>
  <Accordion title="Розпізнавані сигнатури переповнення">
    OpenClaw виявляє переповнення контексту за такими шаблонами помилок провайдера:

    - `request_too_large`
    - `context length exceeded`
    - `input exceeds the maximum number of tokens`
    - `input token count exceeds the maximum number of input tokens`
    - `input is too long for the model`
    - `ollama error: context length exceeded`

  </Accordion>
</AccordionGroup>

## Ручна Compaction

Введіть `/compact` у будь-якому чаті, щоб примусово виконати Compaction. Додайте інструкції, щоб спрямувати підсумок:

```text
/compact Зосередься на рішеннях щодо дизайну API
```

Коли задано `agents.defaults.compaction.keepRecentTokens`, ручна Compaction враховує цю точку відсікання Pi і зберігає недавній хвіст у відновленому контексті. Без явного бюджету збереження ручна Compaction поводиться як жорстка контрольна точка і продовжується лише з нового підсумку.

## Конфігурація

Налаштуйте Compaction у `agents.defaults.compaction` у вашому `openclaw.json`. Нижче наведено найпоширеніші параметри; повний довідник див. у [Поглибленому огляді керування сесіями](/uk/reference/session-management-compaction).

### Використання іншої моделі

Типово Compaction використовує основну модель агента. Установіть `agents.defaults.compaction.model`, щоб делегувати підсумовування більш потужній або спеціалізованій моделі. Перевизначення приймає будь-який рядок `provider/model-id`:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "openrouter/anthropic/claude-sonnet-4-6"
      }
    }
  }
}
```

Це також працює з локальними моделями, наприклад із другою моделлю Ollama, виділеною для підсумовування:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "ollama/llama3.1:8b"
      }
    }
  }
}
```

Якщо значення не задано, Compaction використовує основну модель агента.

### Збереження ідентифікаторів

Підсумовування Compaction типово зберігає непрозорі ідентифікатори (`identifierPolicy: "strict"`). Перевизначте на `identifierPolicy: "off"`, щоб вимкнути це, або задайте `identifierPolicy: "custom"` разом з `identifierInstructions` для власних вказівок.

### Захист за байтами активного транскрипту

Коли задано `agents.defaults.compaction.maxActiveTranscriptBytes`, OpenClaw запускає звичайну локальну Compaction перед виконанням, якщо активний JSONL досягає цього розміру. Це корисно для довготривалих сесій, де керування контекстом на боці провайдера може підтримувати здоровий контекст моделі, тоді як локальний транскрипт продовжує зростати. Це не ділить сирі байти JSONL; натомість звичайний конвеєр Compaction створює семантичний підсумок.

<Warning>
Захист за байтами потребує `truncateAfterCompaction: true`. Без ротації транскрипту активний файл не зменшиться, і захист залишиться неактивним.
</Warning>

### Наступні транскрипти

Коли ввімкнено `agents.defaults.compaction.truncateAfterCompaction`, OpenClaw не переписує наявний транскрипт на місці. Він створює новий активний наступний транскрипт на основі підсумку Compaction, збереженого стану й непідсумованого хвоста, а попередній JSONL зберігає як джерело архівної контрольної точки.
Наступні транскрипти також відкидають точні дублікати довгих ходів користувача, які надходять у межах короткого вікна повторної спроби, тож шторми повторів каналу не переносяться до наступного активного транскрипту після Compaction.

Контрольні точки до Compaction зберігаються лише доти, доки не перевищують ліміт розміру контрольних точок OpenClaw; надто великі активні транскрипти все одно проходять Compaction, але OpenClaw пропускає великий знімок для налагодження, щоб не подвоювати використання диска.

### Сповіщення про Compaction

Типово Compaction виконується безшумно. Установіть `notifyUser`, щоб показувати короткі повідомлення про стан, коли Compaction починається й завершується:

```json5
{
  agents: {
    defaults: {
      compaction: {
        notifyUser: true,
      },
    },
  },
}
```

### Вивантаження memory

Перед Compaction OpenClaw може виконати **тихе вивантаження memory**, щоб зберегти довготривалі нотатки на диск. Докладніше та конфігурацію див. у [Memory](/uk/concepts/memory).

## Підключувані провайдери Compaction

Плагіни можуть реєструвати власного провайдера Compaction через `registerCompactionProvider()` у API плагіна. Коли провайдера зареєстровано й налаштовано, OpenClaw делегує йому підсумовування замість вбудованого конвеєра LLM.

Щоб використовувати зареєстрованого провайдера, задайте його id у конфігурації:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "provider": "my-provider"
      }
    }
  }
}
```

Установлення `provider` автоматично примусово задає `mode: "safeguard"`. Провайдери отримують ті самі інструкції для Compaction і політику збереження ідентифікаторів, що й вбудований шлях, а OpenClaw усе одно зберігає контекст нещодавніх ходів і розділених ходів після результату провайдера.

<Note>
Якщо провайдер завершується з помилкою або повертає порожній результат, OpenClaw повертається до вбудованого підсумовування через LLM.
</Note>

## Compaction проти pruning

|                  | Compaction                    | Pruning                              |
| ---------------- | ----------------------------- | ------------------------------------ |
| **Що робить**    | Підсумовує старішу розмову    | Обрізає старі результати інструментів |
| **Зберігається?** | Так (у транскрипті сесії)    | Ні (лише в пам’яті, для кожного запиту) |
| **Обсяг**        | Уся розмова                   | Лише результати інструментів         |

[Session pruning](/uk/concepts/session-pruning) — це легше доповнення, яке обрізає вивід інструментів без підсумовування.

## Усунення несправностей

**Compaction виконується надто часто?** Вікно контексту моделі може бути малим, або виводи інструментів можуть бути великими. Спробуйте ввімкнути [session pruning](/uk/concepts/session-pruning).

**Контекст здається застарілим після Compaction?** Використовуйте `/compact Зосередься на <topic>`, щоб спрямувати підсумок, або ввімкніть [вивантаження memory](/uk/concepts/memory), щоб нотатки зберігалися.

**Потрібен чистий старт?** `/new` починає нову сесію без Compaction.

Для розширеної конфігурації (резерв токенів, збереження ідентифікаторів, власні рушії контексту, серверна Compaction OpenAI) див. [Поглиблений огляд керування сесіями](/uk/reference/session-management-compaction).
