# Автоматизація QA E2E

Приватний стек QA призначений для тестування OpenClaw у реалістичніший,
орієнтований на канали спосіб, ніж це може зробити один unit-тест.

Поточні складові:

- `extensions/qa-channel`: синтетичний канал повідомлень із поверхнями DM, каналу, треду,
  реакцій, редагування та видалення.
- `extensions/qa-lab`: UI для налагодження і QA bus для спостереження за транскриптом,
  впровадження вхідних повідомлень та експорту звіту у Markdown.
- `qa/`: початкові ресурси в репозиторії для стартового завдання та базових QA-
  сценаріїв.

Поточний потік роботи оператора QA — це двопанельний QA-сайт:

- Ліворуч: панель керування Gateway (Control UI) з агентом.
- Праворуч: QA Lab, що показує транскрипт у стилі Slack і план сценарію.

Запустіть так:

```bash
pnpm qa:lab:up
```

Це збирає QA-сайт, запускає lane Gateway на базі Docker і відкриває
сторінку QA Lab, де оператор або цикл автоматизації може дати агенту QA-
місію, спостерігати реальну поведінку каналу та фіксувати, що спрацювало, що
не вдалося або що залишилося заблокованим.

Щоб швидше ітеруватися над UI QA Lab без перебудови Docker-образу щоразу,
запустіть стек із bundle QA Lab, змонтованим через bind-mount:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

`qa:lab:up:fast` залишає Docker-сервіси на попередньо зібраному образі та bind-mount-ить
`extensions/qa-lab/web/dist` у контейнер `qa-lab`. `qa:lab:watch`
перебудовує цей bundle при змінах, а браузер автоматично перезавантажується, коли
змінюється хеш ресурсів QA Lab.

Для transport-real lane перевірки Matrix виконайте:

```bash
pnpm openclaw qa matrix
```

Цей lane розгортає одноразовий homeserver Tuwunel у Docker, реєструє
тимчасових користувачів driver, SUT і observer, створює одну приватну кімнату, а
потім запускає справжній Plugin Matrix всередині дочірнього процесу QA gateway. Lane live transport зберігає
конфігурацію дочірнього процесу обмеженою транспортом, який тестується, тому Matrix
працює без `qa-channel` у конфігурації дочірнього процесу. Він записує структуровані
артефакти звіту та комбінований stdout/stderr log до вибраного вихідного каталогу Matrix QA.
Щоб також захопити зовнішній вивід збирання/запуску `scripts/run-node.mjs`,
встановіть `OPENCLAW_RUN_NODE_OUTPUT_LOG=<path>` на файл log у межах репозиторію.

Для transport-real lane перевірки Telegram виконайте:

```bash
pnpm openclaw qa telegram
```

Цей lane націлюється на одну реальну приватну групу Telegram замість
розгортання одноразового сервера. Для нього потрібні `OPENCLAW_QA_TELEGRAM_GROUP_ID`,
`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN` і
`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`, а також два різні боти в одній
приватній групі. Бот SUT повинен мати ім’я користувача Telegram, а
спостереження bot-to-bot працює найкраще, коли для обох ботів увімкнено режим
Bot-to-Bot Communication Mode у `@BotFather`.

Тепер lane live transport використовують один менший контракт замість того, щоб кожен
вигадував власну форму списку сценаріїв:

`qa-channel` залишається широким набором синтетичних перевірок поведінки продукту і не входить
до матриці покриття live transport.

| Lane     | Canary | Гейтінг згадок | Блокування allowlist | Відповідь верхнього рівня | Відновлення після перезапуску | Подальша відповідь у треді | Ізоляція треду | Спостереження за реакціями | Команда help |
| -------- | ------ | -------------- | -------------------- | ------------------------- | ----------------------------- | -------------------------- | -------------- | -------------------------- | ------------ |
| Matrix   | x      | x              | x                    | x                         | x                             | x                          | x              | x                          |              |
| Telegram | x      |                |                      |                           |                               |                            |                |                            | x            |

Це зберігає `qa-channel` як широкий набір перевірок поведінки продукту, тоді як Matrix,
Telegram та майбутні live transport спільно використовують один явний контрольний список транспортного контракту.

Для lane з одноразовою Linux VM без залучення Docker у шлях QA виконайте:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

Це завантажує новий гостьовий екземпляр Multipass, встановлює залежності, збирає OpenClaw
всередині гостя, запускає `qa suite`, а потім копіює звичайний QA-звіт і
підсумок назад у `.artifacts/qa-e2e/...` на хості.
Він повторно використовує ту саму поведінку вибору сценаріїв, що й `qa suite` на хості.
Запуски suite на хості та в Multipass за замовчуванням виконують кілька вибраних сценаріїв паралельно
з ізольованими worker-процесами gateway, до 64 worker-ів або до кількості вибраних
сценаріїв. Використовуйте `--concurrency <count>`, щоб налаштувати кількість worker-ів, або
`--concurrency 1` для послідовного виконання.
Live-запуски передають підтримувані вхідні дані автентифікації QA, практичні для
гостя: ключі провайдерів через env, шлях до конфігурації QA live provider і
`CODEX_HOME`, якщо він присутній. Тримайте `--output-dir` у межах кореня репозиторію, щоб гість
міг записувати назад через змонтований workspace.

## Початкові ресурси в репозиторії

Початкові ресурси розміщені в `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/*.md`

Вони навмисно зберігаються в git, щоб QA-план був видимий і людям, і
агенту.

`qa-lab` має залишатися універсальним виконавцем markdown. Кожен markdown-файл сценарію є
джерелом істини для одного тестового запуску і повинен визначати:

- метадані сценарію
- посилання на документацію та код
- необов’язкові вимоги до Plugin
- необов’язковий patch конфігурації gateway
- виконуваний `qa-flow`

Багаторазову runtime-поверхню, яка лежить в основі `qa-flow`, дозволено залишати універсальною
та наскрізною. Наприклад, markdown-сценарії можуть поєднувати
допоміжні засоби на стороні транспорту з допоміжними засобами на стороні браузера, які керують вбудованим Control UI через
шов Gateway `browser.request`, не додаючи спеціального виконавця для конкретного випадку.

Базовий список має залишатися достатньо широким, щоб охоплювати:

- DM і спілкування в каналі
- поведінку тредів
- життєвий цикл дій із повідомленнями
- зворотні виклики Cron
- виклик пам’яті
- перемикання моделей
- передавання підзадач субагенту
- читання репозиторію та документації
- одне невелике завдання зі збірки, наприклад Lobster Invaders

## Lanes із mock-провайдерами

`qa suite` має два локальні lanes із mock-провайдерами:

- `mock-openai` — це сценарійно-орієнтований mock OpenClaw. Він залишається типовим
  детермінованим mock-lane для QA, що спирається на репозиторій, і parity gate.
- `aimock` запускає сервер провайдера на базі AIMock для експериментального покриття протоколів,
  фікстур, запису/відтворення та chaos-перевірок. Це доповнення, а не заміна
  диспетчера сценаріїв `mock-openai`.

Реалізація lane провайдера розміщена в `extensions/qa-lab/src/providers/`.
Кожен провайдер володіє своїми типовими значеннями, запуском локального сервера,
конфігурацією моделей gateway, потребами staging auth-profile, а також прапорами можливостей live/mock. Спільний код suite і gateway
має маршрутизуватися через реєстр провайдерів, а не гілкуватися за назвами провайдерів.

## Адаптери транспорту

`qa-lab` володіє універсальним транспортним швом для markdown QA-сценаріїв.
`qa-channel` — перший адаптер на цьому шві, але ціль дизайну ширша:
майбутні реальні або синтетичні канали мають підключатися до того самого виконавця suite
замість додавання окремого виконавця QA для конкретного транспорту.

На рівні архітектури поділ такий:

- `qa-lab` володіє універсальним виконанням сценаріїв, паралелізмом worker-ів, записом артефактів і звітністю.
- адаптер транспорту володіє конфігурацією gateway, готовністю, спостереженням за вхідними та вихідними подіями, транспортними діями й нормалізованим станом транспорту.
- markdown-файли сценаріїв у `qa/scenarios/` визначають тестовий запуск; `qa-lab` надає багаторазову runtime-поверхню, яка їх виконує.

Орієнтовані на мейнтейнерів настанови щодо впровадження нових адаптерів каналів розміщені в
[Тестуванні](/uk/help/testing#adding-a-channel-to-qa).

## Звітність

`qa-lab` експортує звіт протоколу у Markdown зі спостережуваної часової шкали bus.
Звіт має відповідати на такі запитання:

- Що спрацювало
- Що не спрацювало
- Що залишилося заблокованим
- Які сценарії подальших перевірок варто додати

Для перевірок характеру та стилю запустіть той самий сценарій на кількох live model
refs і запишіть оцінений звіт у Markdown:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.4,thinking=xhigh \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.4,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

Команда запускає локальні дочірні процеси QA gateway, а не Docker. Сценарії оцінювання характеру
мають задавати persona через `SOUL.md`, а потім виконувати звичайні користувацькі ходи,
такі як спілкування, допомога з workspace і невеликі файлові завдання. Кандидатній моделі
не слід повідомляти, що її оцінюють. Команда зберігає кожен повний
транскрипт, записує базову статистику запуску, а потім просить моделі-оцінювачі у fast mode з
міркуванням `xhigh` ранжувати запуски за природністю, вайбом і гумором.
Використовуйте `--blind-judge-models` при порівнянні провайдерів: prompt оцінювача все одно отримує
кожен транскрипт і статус запуску, але refs кандидатів замінюються на нейтральні
мітки на кшталт `candidate-01`; звіт зіставляє ранжування назад із реальними refs після
парсингу.
Для кандидатних запусків за замовчуванням використовується рівень thinking `high`, а для моделей OpenAI, які
його підтримують, — `xhigh`. Перевизначте конкретного кандидата inline через
`--model provider/model,thinking=<level>`. `--thinking <level>` усе ще задає
глобальне резервне значення, а старіша форма `--model-thinking <provider/model=level>` збережена
для сумісності.
Для refs кандидатів OpenAI за замовчуванням використовується fast mode, щоб там, де провайдер це підтримує,
застосовувалася пріоритетна обробка. Додайте `,fast`, `,no-fast` або `,fast=false` inline, коли
потрібне перевизначення для окремого кандидата або оцінювача. Передавайте `--fast` лише тоді, коли
хочете примусово ввімкнути fast mode для кожної кандидатної моделі. Тривалість запусків кандидатів і оцінювачів
записується у звіт для аналізу продуктивності, але в prompt-ах оцінювачів прямо вказується
не ранжувати за швидкістю.
Для запусків кандидатних моделей та моделей-оцінювачів за замовчуванням використовується concurrency 16. Зменшуйте
`--concurrency` або `--judge-concurrency`, коли обмеження провайдера або навантаження на локальний gateway
роблять запуск надто шумним.
Якщо не передано жодного кандидатного `--model`, character eval за замовчуванням використовує
`openai/gpt-5.4`, `openai/gpt-5.2`, `openai/gpt-5`, `anthropic/claude-opus-4-6`,
`anthropic/claude-sonnet-4-6`, `zai/glm-5.1`,
`moonshot/kimi-k2.5` і
`google/gemini-3.1-pro-preview`, якщо не передано `--model`.
Якщо не передано жодного `--judge-model`, оцінювачами за замовчуванням є
`openai/gpt-5.4,thinking=xhigh,fast` і
`anthropic/claude-opus-4-6,thinking=high`.
