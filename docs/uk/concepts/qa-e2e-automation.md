Приватний стек QA призначений для перевірки OpenClaw у реалістичніший,
каналоподібний спосіб, ніж це може зробити один unit-тест.

Поточні складові:

- `extensions/qa-channel`: синтетичний канал повідомлень із поверхнями DM, channel, thread,
  reaction, edit і delete.
- `extensions/qa-lab`: UI налагодження та QA bus для спостереження за транскриптом,
  ін’єкції вхідних повідомлень і експорту Markdown-звіту.
- `qa/`: seed-ресурси, що підтримуються репозиторієм, для стартового завдання та базових QA
  сценаріїв.

Поточний операторський потік QA — це двопанельний сайт QA:

- Ліворуч: панель Gateway (Control UI) з агентом.
- Праворуч: QA Lab, який показує Slack-подібний транскрипт і план сценарію.

Запуск:

```bash
pnpm qa:lab:up
```

Це збирає сайт QA, запускає lane gateway на базі Docker і відкриває
сторінку QA Lab, де оператор або цикл автоматизації може дати агенту QA-місію,
спостерігати реальну поведінку каналу й зафіксувати, що спрацювало, що зламалося або
що залишилося заблокованим.

Для швидшої ітерації UI QA Lab без повторного збирання Docker-образу щоразу,
запускайте стек із bind-mounted набором QA Lab:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

`qa:lab:up:fast` залишає сервіси Docker на попередньо зібраному образі й bind-mount-ить
`extensions/qa-lab/web/dist` у контейнер `qa-lab`. `qa:lab:watch`
перезбирає цей набір при змінах, а браузер автоматично перезавантажується, коли змінюється хеш ресурсів QA Lab.

Для smoke-lane Matrix із реальним transport запустіть:

```bash
pnpm openclaw qa matrix
```

Цей lane розгортає тимчасовий homeserver Tuwunel у Docker, реєструє
тимчасових користувачів driver, SUT і observer, створює одну приватну кімнату, а потім запускає
реальний Plugin Matrix усередині дочірнього QA gateway. Lane живого transport зберігає конфігурацію дочірнього процесу обмеженою transport, що тестується, тож Matrix запускається без
`qa-channel` у конфігурації дочірнього процесу. Він записує структуровані артефакти звіту та
об’єднаний лог stdout/stderr у вибраний вихідний каталог Matrix QA. Щоб
також захопити зовнішній вивід збирання/запуску `scripts/run-node.mjs`, установіть
`OPENCLAW_RUN_NODE_OUTPUT_LOG=<path>` на repo-local файл журналу.

Для smoke-lane Telegram із реальним transport запустіть:

```bash
pnpm openclaw qa telegram
```

Цей lane націлений на одну реальну приватну групу Telegram замість розгортання
тимчасового сервера. Для нього потрібні `OPENCLAW_QA_TELEGRAM_GROUP_ID`,
`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN` і
`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`, а також два різні боти в тій самій
приватній групі. Бот SUT повинен мати ім’я користувача Telegram, а спостереження bot-to-bot
найкраще працює, коли в обох ботів увімкнено Bot-to-Bot Communication Mode
у `@BotFather`.
Команда завершується з ненульовим кодом, якщо будь-який сценарій завершується невдачею. Використовуйте `--allow-failures`, якщо
вам потрібні артефакти без аварійного коду завершення.
Звіт і підсумок Telegram включають RTT для кожної відповіді — від запиту
надсилання повідомлення driver до спостережуваної відповіді SUT, починаючи з canary.

Тепер живі transport-lane використовують один менший спільний контракт замість того, щоб
кожен вигадував власну форму списку сценаріїв:

`qa-channel` залишається широким синтетичним набором перевірок поведінки продукту й не входить до матриці покриття живого transport.

| Lane     | Canary | Шлюз згадок | Блокування allowlist | Відповідь верхнього рівня | Відновлення після перезапуску | Follow-up у thread | Ізоляція thread | Спостереження реакцій | Команда help |
| -------- | ------ | ----------- | -------------------- | ------------------------- | ----------------------------- | ------------------ | --------------- | --------------------- | ------------ |
| Matrix   | x      | x           | x                    | x                         | x                             | x                  | x               | x                     |              |
| Telegram | x      |             |                      |                           |                               |                    |                 |                       | x            |

Це зберігає `qa-channel` як широкий набір перевірок поведінки продукту, тоді як Matrix,
Telegram і майбутні живі transport спільно використовують один явний контрольний список transport-контракту.

Для lane із тимчасовою Linux VM без залучення Docker у шлях QA запустіть:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

Це запускає свіжу гостьову Multipass-машину, встановлює залежності, збирає
OpenClaw усередині гостя, запускає `qa suite`, а потім копіює звичайний QA-звіт і
підсумок назад у `.artifacts/qa-e2e/...` на хості.
Він повторно використовує ту саму поведінку вибору сценаріїв, що й `qa suite` на хості.
Хостові та Multipass-запуски suite типово виконують кілька вибраних сценаріїв паралельно
з ізольованими працівниками gateway. Для `qa-channel` типова паралельність — 4,
обмежена кількістю вибраних сценаріїв. Використовуйте `--concurrency <count>`, щоб налаштувати
кількість працівників, або `--concurrency 1` для послідовного виконання.
Команда завершується з ненульовим кодом, якщо будь-який сценарій завершується невдачею. Використовуйте `--allow-failures`, якщо
вам потрібні артефакти без аварійного коду завершення.
Живі запуски передають підтримувані вхідні дані автентифікації QA, практичні для
гостя: ключі провайдера через env, шлях до конфігурації QA live provider і
`CODEX_HOME`, якщо він присутній. Тримайте `--output-dir` у межах кореня repo, щоб гість
міг записувати назад через змонтований робочий простір.

## Seed-ресурси, що підтримуються репозиторієм

Seed-ресурси живуть у `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/<theme>/*.md`

Вони навмисно зберігаються в git, щоб план QA був видимий і людям, і
агенту.

`qa-lab` має залишатися загальним runner для markdown. Кожен markdown-файл сценарію є
джерелом істини для одного тестового запуску і має визначати:

- метадані сценарію
- необов’язкові метадані category, capability, lane і risk
- посилання на docs і code
- необов’язкові вимоги до Plugin
- необов’язковий patch конфігурації gateway
- виконуваний `qa-flow`

Повторно використовувана поверхня runtime, що лежить в основі `qa-flow`, може залишатися загальною
і наскрізною. Наприклад, markdown-сценарії можуть поєднувати transport-орієнтовані
допоміжні засоби з browser-орієнтованими допоміжними засобами, які керують вбудованим Control UI через
seam `browser.request` Gateway без додавання special-case runner.

Файли сценаріїв слід групувати за можливістю продукту, а не за папкою дерева source.
Зберігайте стабільність ID сценаріїв під час переміщення файлів; використовуйте `docsRefs` і `codeRefs` для трасованості реалізації.

Базовий список має залишатися досить широким, щоб покривати:

- DM і chat у channel
- поведінку thread
- життєвий цикл дій із повідомленнями
- зворотні виклики cron
- відновлення memory
- перемикання моделей
- передачу субагенту
- читання repo і docs
- одне невелике завдання зі збирання, наприклад Lobster Invaders

## Mock-lane провайдерів

`qa suite` має два локальні mock-lane провайдерів:

- `mock-openai` — це сценарно-орієнтований mock OpenClaw. Він залишається типовим
  детермінованим mock-lane для QA, що підтримується репозиторієм, і перевірок паритету.
- `aimock` запускає сервер провайдера на базі AIMock для експериментального покриття
  протоколу, фікстур, record/replay і chaos. Він є додатковим і не замінює dispatcher сценаріїв `mock-openai`.

Реалізація provider-lane розміщена в `extensions/qa-lab/src/providers/`.
Кожен провайдер володіє своїми типовими значеннями, запуском локального сервера,
конфігурацією моделі gateway, потребами staging auth-profile і прапорцями live/mock capability. Спільний код suite і gateway має маршрутизувати через реєстр провайдерів, а не розгалужуватися за назвами провайдерів.

## Адаптери transport

`qa-lab` володіє загальним seam transport для markdown QA-сценаріїв.
`qa-channel` — перший адаптер на цьому seam, але ціль дизайну ширша:
майбутні реальні або синтетичні канали мають підключатися до того самого runner suite замість додавання transport-специфічного QA-runner.

На рівні архітектури розподіл такий:

- `qa-lab` відповідає за загальне виконання сценаріїв, паралельність працівників, запис артефактів і звітність.
- адаптер transport відповідає за конфігурацію gateway, готовність, спостереження за вхідним і вихідним трафіком, дії transport і нормалізований стан transport.
- markdown-файли сценаріїв у `qa/scenarios/` визначають тестовий запуск; `qa-lab` надає повторно використовувану поверхню runtime, яка їх виконує.

Інструкції для мейнтейнерів щодо підключення нових адаптерів каналів розміщені в
[Testing](/uk/help/testing#adding-a-channel-to-qa).

## Звітність

`qa-lab` експортує Markdown-звіт протоколу на основі спостережуваної часової шкали bus.
Звіт має відповідати на запитання:

- Що спрацювало
- Що зламалося
- Що залишилося заблокованим
- Які додаткові сценарії варто додати

Для перевірок характеру й стилю запускайте той самий сценарій на кількох живих refs моделей
і записуйте Markdown-звіт з оцінюванням:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.5,thinking=xhigh \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.5,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

Команда запускає локальні дочірні процеси QA gateway, а не Docker. Сценарії
character eval мають задавати persona через `SOUL.md`, а потім виконувати звичайні ходи користувача,
такі як chat, допомога з робочим простором і невеликі файлові завдання. Моделі-кандидату
не слід повідомляти, що її оцінюють. Команда зберігає кожен повний
транскрипт, записує базову статистику запуску, а потім просить judge-моделі в режимі fast з
міркуванням `xhigh` ранжувати запуски за природністю, vibe і гумором.
Використовуйте `--blind-judge-models` під час порівняння провайдерів: prompt для judge, як і раніше, отримує
кожен транскрипт і статус запуску, але refs кандидатів замінюються нейтральними
мітками на кшталт `candidate-01`; звіт зіставляє рейтинги назад із реальними refs після
розбору.
Для запусків кандидатів типово використовується `high` thinking, а для моделей OpenAI, які це
підтримують, — `xhigh`. Перевизначайте конкретного кандидата inline через
`--model provider/model,thinking=<level>`. `--thinking <level>` як і раніше задає
глобальне резервне значення, а старіша форма `--model-thinking <provider/model=level>` збережена
для сумісності.
refs кандидатів OpenAI типово використовують режим fast, щоб застосовувалася пріоритетна обробка там,
де провайдер це підтримує. Додайте `,fast`, `,no-fast` або `,fast=false` inline, коли
окремому кандидату або judge потрібне перевизначення. Передавайте `--fast` лише тоді, коли хочете
примусово ввімкнути режим fast для кожної моделі-кандидата. Тривалість запусків кандидатів і judge
записується у звіт для бенчмарк-аналізу, але prompts для judge прямо вказують
не ранжувати за швидкістю.
І candidate, і judge model runs типово використовують паралельність 16. Зменшуйте
`--concurrency` або `--judge-concurrency`, коли обмеження провайдера або навантаження локального gateway
роблять запуск надто шумним.
Якщо не передано жодного кандидатського `--model`, character eval типово використовує
`openai/gpt-5.5`, `openai/gpt-5.2`, `openai/gpt-5`, `anthropic/claude-opus-4-6`,
`anthropic/claude-sonnet-4-6`, `zai/glm-5.1`,
`moonshot/kimi-k2.5` і
`google/gemini-3.1-pro-preview`, коли `--model` не передано.
Якщо не передано `--judge-model`, суддівськими моделями типово є
`openai/gpt-5.5,thinking=xhigh,fast` і
`anthropic/claude-opus-4-6,thinking=high`.
