OpenClaw запам’ятовує речі, записуючи **звичайні Markdown-файли** в
workspace вашого агента. Модель "пам’ятає" лише те, що збережено на диск — жодного
прихованого стану немає.

## Як це працює

Ваш агент має три файли, пов’язані з пам’яттю:

- **`MEMORY.md`** — довготривала пам’ять. Стійкі факти, уподобання та
  рішення. Завантажується на початку кожного DM-сеансу.
- **`memory/YYYY-MM-DD.md`** — щоденні нотатки. Поточний контекст і спостереження.
  Нотатки за сьогодні та вчора завантажуються автоматично.
- **`DREAMS.md`** (необов’язково) — щоденник Dream Diary і підсумки проходів
  Dreaming для людського перегляду, включно з grounded історичними backfill-записами.

Ці файли розташовані у workspace агента (за замовчуванням `~/.openclaw/workspace`).

<Tip>
Якщо хочете, щоб ваш агент щось запам’ятав, просто попросіть його: "Запам’ятай, що я
віддаю перевагу TypeScript." Він запише це у відповідний файл.
</Tip>

## Інструменти пам’яті

Агент має два інструменти для роботи з пам’яттю:

- **`memory_search`** — знаходить релевантні нотатки за допомогою семантичного пошуку, навіть коли
  формулювання відрізняється від оригінального.
- **`memory_get`** — читає конкретний файл пам’яті або діапазон рядків.

Обидва інструменти надає активний Plugin пам’яті (за замовчуванням: `memory-core`).

## Супровідний Plugin Memory Wiki

Якщо ви хочете, щоб стійка пам’ять поводилася радше як підтримувана база знань, ніж
просто сирі нотатки, використовуйте вбудований Plugin `memory-wiki`.

`memory-wiki` компілює стійкі знання у wiki-сховище з:

- детермінованою структурою сторінок
- структурованими твердженнями та доказами
- відстеженням суперечностей і свіжості
- згенерованими панелями
- скомпільованими дайджестами для споживачів агента/runtime
- wiki-native інструментами, як-от `wiki_search`, `wiki_get`, `wiki_apply` і `wiki_lint`

Він не замінює активний Plugin пам’яті. Активний Plugin пам’яті й далі
відповідає за пригадування, просування і Dreaming. `memory-wiki` додає поруч шар
знань із багатою provenance-інформацією.

Див. [Memory Wiki](/uk/plugins/memory-wiki).

## Пошук у пам’яті

Коли налаштовано embedding-провайдера, `memory_search` використовує **гібридний
пошук** — поєднує векторну схожість (семантичне значення) із пошуком за ключовими словами
(точні терміни, як-от ID і символи коду). Це працює одразу після того, як у вас є
API-ключ для будь-якого підтримуваного провайдера.

<Info>
OpenClaw автоматично визначає вашого embedding-провайдера за доступними API-ключами. Якщо у вас
налаштовано ключ OpenAI, Gemini, Voyage або Mistral, пошук у пам’яті
вмикається автоматично.
</Info>

Докладніше про те, як працює пошук, параметри налаштування та налаштування провайдера, див.
[Пошук у пам’яті](/uk/concepts/memory-search).

## Автоматичне скидання пам’яті

Перед тим як [Compaction](/uk/concepts/compaction) підсумує вашу розмову, OpenClaw
запускає тихий хід, який нагадує агенту зберегти важливий контекст у файли
пам’яті. Це ввімкнено за замовчуванням — вам не потрібно нічого налаштовувати.

Щоб цей housekeeping-хід виконувався на локальній моделі, задайте точне перевизначення моделі
скидання пам’яті:

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "memoryFlush": {
          "model": "ollama/qwen3:8b"
        }
      }
    }
  }
}
```

Перевизначення застосовується лише до ходу скидання пам’яті й не успадковує
fallback-ланцюжок активного сеансу.

<Tip>
Скидання пам’яті запобігає втраті контексту під час Compaction. Якщо ваш агент має
важливі факти в розмові, які ще не записано у файл, вони
будуть автоматично збережені перед створенням підсумку.
</Tip>

## Dreaming

Dreaming — це необов’язковий фоновий прохід консолідації пам’яті. Він збирає
короткострокові сигнали, оцінює кандидатів і просуває лише кваліфіковані елементи в
довготривалу пам’ять (`MEMORY.md`).

Він спроєктований так, щоб довготривала пам’ять мала високу корисність:

- **Увімкнення вручну**: вимкнено за замовчуванням.
- **За розкладом**: коли ввімкнено, `memory-core` автоматично керує одним повторюваним cron-завданням
  для повного проходу Dreaming.
- **З порогами**: просування мають пройти фільтри за оцінкою, частотою пригадування та різноманітністю
  запитів.
- **Доступно для перегляду**: підсумки фаз і записи щоденника записуються в `DREAMS.md`
  для людського перегляду.

Про поведінку фаз, сигнали оцінювання та деталі Dream Diary див.
[Dreaming](/uk/concepts/dreaming).

## Grounded backfill і live promotion

Система Dreaming тепер має дві тісно пов’язані лінії перегляду:

- **Live dreaming** працює з короткострокового сховища Dreaming у
  `memory/.dreams/` і є тим, що звичайна deep phase використовує, вирішуючи, що
  може перейти в `MEMORY.md`.
- **Grounded backfill** читає історичні нотатки `memory/YYYY-MM-DD.md` як
  окремі денні файли та записує структурований результат перегляду в `DREAMS.md`.

Grounded backfill корисний, коли ви хочете повторно програти старіші нотатки й перевірити, що
система вважає стійким, без ручного редагування `MEMORY.md`.

Коли ви використовуєте:

```bash
openclaw memory rem-backfill --path ./memory --stage-short-term
```

grounded стійкі кандидати не просуваються напряму. Вони ставляться в чергу в
те саме короткострокове сховище Dreaming, яке вже використовує звичайна deep phase. Це
означає:

- `DREAMS.md` залишається поверхнею людського перегляду.
- короткострокове сховище залишається machine-facing поверхнею ранжування.
- `MEMORY.md` як і раніше записується лише deep promotion.

Якщо ви вирішите, що replay не був корисним, можна видалити staged-артефакти,
не торкаючись звичайних записів щоденника або звичайного стану пригадування:

```bash
openclaw memory rem-backfill --rollback
openclaw memory rem-backfill --rollback-short-term
```
