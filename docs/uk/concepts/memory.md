OpenClaw запам’ятовує дані, записуючи **звичайні Markdown-файли** в робочому
просторі вашого агента. Модель “пам’ятає” лише те, що було збережено на диск —
жодного прихованого стану немає.

## Як це працює

Ваш агент має три файли, пов’язані з пам’яттю:

- **`MEMORY.md`** — довготривала пам’ять. Стійкі факти, уподобання та
  рішення. Завантажується на початку кожної сесії особистих повідомлень.
- **`memory/YYYY-MM-DD.md`** — щоденні нотатки. Поточний контекст і спостереження.
  Нотатки за сьогодні та вчора завантажуються автоматично.
- **`DREAMS.md`** (необов’язково) — Dream Diary і підсумки проходів Dreaming
  для перегляду людиною, включно з обґрунтованими записами історичного заповнення.

Ці файли знаходяться в робочому просторі агента (типово `~/.openclaw/workspace`).

<Tip>
Якщо ви хочете, щоб агент щось запам’ятав, просто скажіть йому: “Запам’ятай, що я
віддаю перевагу TypeScript.” Він запише це у відповідний файл.
</Tip>

## Інструменти пам’яті

Агент має два інструменти для роботи з пам’яттю:

- **`memory_search`** — знаходить релевантні нотатки за допомогою семантичного пошуку, навіть коли
  формулювання відрізняється від оригінального.
- **`memory_get`** — читає конкретний файл пам’яті або діапазон рядків.

Обидва інструменти надаються активним Plugin пам’яті (типово: `memory-core`).

## Супутній Plugin Memory Wiki

Якщо ви хочете, щоб стійка пам’ять поводилася радше як підтримувана база знань,
а не просто як сирі нотатки, використовуйте вбудований Plugin `memory-wiki`.

`memory-wiki` компілює стійкі знання у wiki vault з такими можливостями:

- детермінована структура сторінок
- структуровані твердження та докази
- відстеження суперечностей і свіжості
- згенеровані інформаційні панелі
- скомпільовані дайджести для споживачів agent/runtime
- wiki-native інструменти на кшталт `wiki_search`, `wiki_get`, `wiki_apply` і `wiki_lint`

Він не замінює активний Plugin пам’яті. Активний Plugin пам’яті, як і раніше,
відповідає за recall, підвищення та Dreaming. `memory-wiki` додає поруч
із ним шар знань із багатим provenance.

Див. [Memory Wiki](/uk/plugins/memory-wiki).

## Пошук у пам’яті

Коли налаштовано провайдера embeddings, `memory_search` використовує **гібридний
пошук** — поєднуючи векторну схожість (семантичне значення) з пошуком за ключовими словами
(точні терміни, такі як ID та символи коду). Це працює з коробки, щойно у вас є
API-ключ для будь-якого підтримуваного провайдера.

<Info>
OpenClaw автоматично визначає вашого провайдера embeddings із доступних API-ключів. Якщо у
вас налаштовано ключ OpenAI, Gemini, Voyage або Mistral, пошук у пам’яті
вмикається автоматично.
</Info>

Докладніше про те, як працює пошук, параметри налаштування та конфігурацію провайдера див. у
[Пошук у пам’яті](/uk/concepts/memory-search).

## Автоматичне скидання пам’яті

Перед тим як [Compaction](/uk/concepts/compaction) підсумує вашу розмову, OpenClaw
запускає тихий хід, який нагадує агенту зберегти важливий контекст у файли
пам’яті. Це ввімкнено типово — нічого налаштовувати не потрібно.

<Tip>
Скидання пам’яті запобігає втраті контексту під час Compaction. Якщо у розмові
агента є важливі факти, які ще не записані у файл, їх буде автоматично
збережено до того, як відбудеться підсумовування.
</Tip>

## Dreaming

Dreaming — це необов’язковий фоновий прохід консолідації пам’яті. Він збирає
короткострокові сигнали, оцінює кандидатів і підвищує лише кваліфіковані елементи в
довготривалу пам’ять (`MEMORY.md`).

Він створений для того, щоб підтримувати високу сигналізованість довготривалої пам’яті:

- **За бажанням**: типово вимкнено.
- **За розкладом**: коли ввімкнено, `memory-core` автоматично керує одним повторюваним завданням Cron
  для повного проходу Dreaming.
- **З порогами**: підвищення мають проходити перевірки за оцінкою, частотою recall і
  різноманітністю запитів.
- **Придатний для перегляду**: підсумки фаз і записи щоденника записуються в `DREAMS.md`
  для перегляду людиною.

Поведінку фаз, сигнали оцінювання та деталі Dream Diary див. у
[Dreaming](/uk/concepts/dreaming).

## Обґрунтоване заповнення історії та живе підвищення

Система Dreaming тепер має дві тісно пов’язані смуги перегляду:

- **Живе Dreaming** працює з короткострокового сховища Dreaming у
  `memory/.dreams/` і саме його звичайна глибока фаза використовує, вирішуючи, що
  може перейти до `MEMORY.md`.
- **Обґрунтоване заповнення історії** читає історичні нотатки `memory/YYYY-MM-DD.md` як
  окремі файли днів і записує структурований результат перегляду в `DREAMS.md`.

Обґрунтоване заповнення історії корисне, коли ви хочете програти старі нотатки й перевірити, що
система вважає стійким, не редагуючи вручну `MEMORY.md`.

Коли ви використовуєте:

```bash
openclaw memory rem-backfill --path ./memory --stage-short-term
```

обґрунтовані довготривалі кандидати не підвищуються безпосередньо. Вони поміщаються в
те саме короткострокове сховище Dreaming, яке вже використовує звичайна глибока фаза. Це
означає, що:

- `DREAMS.md` залишається поверхнею перегляду для людини.
- короткострокове сховище залишається поверхнею ранжування для машини.
- `MEMORY.md` усе ще записується лише через глибоке підвищення.

Якщо ви вирішите, що повторний прогін був не корисним, ви можете видалити проміжні артефакти,
не торкаючись звичайних записів щоденника або звичайного стану recall:

```bash
openclaw memory rem-backfill --rollback
openclaw memory rem-backfill --rollback-short-term
```
