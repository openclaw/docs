OpenClaw створює власний системний промпт для кожного запуску агента. Промпт **належить OpenClaw** і не використовує стандартний промпт pi-coding-agent.

Промпт збирається OpenClaw і впроваджується в кожен запуск агента.

Provider plugins можуть додавати cache-aware вказівки для промпта, не замінюючи
повний промпт, що належить OpenClaw. Runtime провайдера може:

- замінювати невеликий набір іменованих основних секцій (`interaction_style`,
  `tool_call_style`, `execution_bias`)
- впроваджувати **стабільний префікс** над межею кешу промпта
- впроваджувати **динамічний суфікс** під межею кешу промпта

Використовуйте внески, що належать провайдеру, для налаштування під конкретні сімейства моделей. Залишайте застарілу
мутацію промпта `before_prompt_build` для сумісності або справді глобальних змін промпта,
а не для звичайної поведінки провайдера.

Оверлей сімейства OpenAI GPT-5 зберігає основне правило виконання невеликим і додає
специфічні для моделі вказівки щодо закріплення persona, стислого виводу, дисципліни інструментів,
паралельного пошуку, покриття deliverable, перевірки, відсутнього контексту та
гігієни terminal-tool.

## Структура

Промпт навмисно компактний і використовує фіксовані секції:

- **Інструменти**: нагадування про structured-tool джерело істини та вказівки щодо використання інструментів під час runtime.
- **Ухил виконання**: компактні вказівки щодо доведення роботи до кінця: діяти в межах поточного ходу на
  actionable запити, продовжувати до завершення або блокування, відновлюватися після слабких результатів інструментів,
  перевіряти змінний стан наживо та перевіряти перед фіналізацією.
- **Безпека**: коротке нагадування-обмеження уникати прагнення до влади або обходу нагляду.
- **Skills** (коли доступні): пояснює моделі, як завантажувати інструкції навичок на вимогу.
- **Самооновлення OpenClaw**: як безпечно інспектувати конфігурацію за допомогою
  `config.schema.lookup`, виправляти конфігурацію через `config.patch`, замінювати повну
  конфігурацію через `config.apply` і запускати `update.run` лише на явний запит користувача.
  Інструмент лише для власника `gateway` також відмовляється перезаписувати
  `tools.exec.ask` / `tools.exec.security`, включно із застарілими псевдонімами `tools.bash.*`,
  які нормалізуються до цих захищених exec шляхів.
- **Робоча область**: робочий каталог (`agents.defaults.workspace`).
- **Документація**: локальний шлях до документації OpenClaw (репозиторій або npm-пакет) і коли її читати.
- **Файли робочої області (впроваджені)**: вказує, що bootstrap-файли включені нижче.
- **Пісочниця** (коли увімкнено): вказує runtime у пісочниці, шляхи пісочниці та чи доступний elevated exec.
- **Поточна дата й час**: локальний час користувача, часовий пояс і формат часу.
- **Теги відповіді**: необов’язковий синтаксис тегів відповіді для підтримуваних провайдерів.
- **Heartbeats**: промпт Heartbeat і поведінка ack, коли Heartbeats увімкнено для стандартного агента.
- **Runtime**: хост, ОС, Node, модель, корінь репозиторію (коли виявлено), рівень thinking (один рядок).
- **Міркування**: поточний рівень видимості + підказка про перемикач /reasoning.

OpenClaw тримає великий стабільний вміст, включно з **Контекстом проєкту**, над
внутрішньою межею кешу промпта. Нестабільні секції каналу/сеансу, як-от
вказівки для вбудовування Control UI, **Обмін повідомленнями**, **Голос**, **Контекст групового чату**,
**Реакції**, **Heartbeats** і **Runtime**, додаються під цією межею,
щоб локальні бекенди з prefix caches могли повторно використовувати стабільний префікс робочої області
між ходами каналу. Описи інструментів так само мають уникати вбудовування поточних
назв каналів, коли прийнята схема вже містить цю runtime деталь.

Секція Інструментів також містить runtime-вказівки для довготривалої роботи:

- використовуйте cron для майбутнього follow-up (`check back later`, нагадування, повторювана робота)
  замість циклів `exec` sleep, трюків із затримкою `yieldMs` або повторного
  polling `process`
- використовуйте `exec` / `process` лише для команд, які запускаються зараз і продовжують працювати
  у фоновому режимі
- коли автоматичне пробудження після завершення увімкнено, запустіть команду один раз і покладайтеся на
  push-based wake path, коли вона виводить дані або завершується помилкою
- використовуйте `process` для журналів, статусу, вводу або втручання, коли потрібно
  інспектувати запущену команду
- якщо завдання більше, віддавайте перевагу `sessions_spawn`; завершення sub-agent є
  push-based і автоматично повідомляє requester
- не опитуйте `subagents list` / `sessions_list` у циклі лише для очікування
  завершення

Коли експериментальний інструмент `update_plan` увімкнено, Інструменти також вказують
моделі використовувати його лише для нетривіальної багатокрокової роботи, тримати рівно один
крок `in_progress` і не повторювати весь план після кожного оновлення.

Запобіжні обмеження безпеки в системному промпті є рекомендаційними. Вони спрямовують поведінку моделі, але не забезпечують примусове застосування політики. Для жорсткого примусового застосування використовуйте політику інструментів, exec approvals, пісочницю та allowlists каналів; оператори можуть вимикати їх за задумом.

У каналах із нативними approval картками/кнопками runtime-промпт тепер вказує
агенту спершу покладатися на цей нативний approval UI. Він має включати ручну
команду `/approve` лише тоді, коли результат інструмента каже, що chat approvals недоступні або
ручне approval є єдиним шляхом.

## Режими промпта

OpenClaw може рендерити менші системні промпти для sub-agents. Runtime встановлює
`promptMode` для кожного запуску (це не user-facing конфігурація):

- `full` (стандартно): включає всі секції вище.
- `minimal`: використовується для sub-agents; пропускає **Skills**, **Memory Recall**, **Самооновлення OpenClaw**,
  **Псевдоніми моделей**, **Ідентичність користувача**, **Теги відповіді**,
  **Обмін повідомленнями**, **Тихі відповіді** та **Heartbeats**. Інструменти, **Безпека**,
  Робоча область, Пісочниця, Поточна дата й час (коли відомо), Runtime та впроваджений
  контекст залишаються доступними.
- `none`: повертає лише базовий рядок ідентичності.

Коли `promptMode=minimal`, додатково впроваджені промпти позначаються як **Контекст subagent**
замість **Контекст групового чату**.

Для запусків channel auto-reply OpenClaw може пропускати загальну секцію **Тихі відповіді**,
коли контекст direct/group chat уже містить розв’язану
специфічну для розмови поведінку `NO_REPLY`. Це уникає повторення token mechanics
і в глобальному системному промпті, і в контексті каналу.

## Знімки промптів

OpenClaw зберігає закомічені знімки промптів для happy path runtime Codex у
`test/fixtures/agents/prompt-snapshots/codex-runtime-happy-path/`. Вони рендерять
вибрані параметри thread/turn app-server плюс реконструйований стек шарів промпта,
прив’язаного до моделі, для прямих ходів Telegram, груп Discord і Heartbeat. Цей стек
містить pinned Codex `gpt-5.5` fixture промпта моделі, згенерований із форми
каталогу/кешу моделей Codex, developer text дозволів happy path Codex,
інструкції developer OpenClaw, інструкції collaboration-mode у межах ходу,
коли OpenClaw їх надає, ввід ходу користувача та посилання на динамічні tool
specs.

Оновіть pinned Codex fixture промпта моделі за допомогою
`pnpm prompt:snapshots:sync-codex-model`. Стандартно скрипт шукає
runtime-кеш Codex у `$CODEX_HOME/models_cache.json`, потім у
`~/.codex/models_cache.json`, і лише після цього fallback до maintainer Codex
checkout convention за `~/code/codex/codex-rs/models-manager/models.json`. Якщо
жодне з цих джерел не існує, команда завершується без зміни закоміченого
fixture. Передайте `--catalog <path>`, щоб оновити з конкретного файлу `models_cache.json`
або `models.json`.

Ці знімки все ще не є byte-for-byte сирим захопленням запиту OpenAI. Codex
може додавати runtime-owned контекст робочої області, як-от `AGENTS.md`, environment
context, memories, інструкції app/plugin і вбудовані стандартні
інструкції collaboration-mode всередині runtime Codex після того, як OpenClaw надішле
параметри thread і turn.

Перегенеруйте їх за допомогою `pnpm prompt:snapshots:gen` і перевірте drift через
`pnpm prompt:snapshots:check`. CI запускає drift check у додатковому
boundary shard, щоб зміни промпта й оновлення знімків залишалися прив’язаними до того самого
PR.

## Впровадження bootstrap у робочу область

Bootstrap-файли обрізаються й додаються під **Контекст проєкту**, щоб модель бачила identity і profile context без потреби в явному читанні:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md` (лише в зовсім нових робочих областях)
- `MEMORY.md`, коли наявний

Усі ці файли **впроваджуються у context window** на кожному ході, якщо
не застосовується gate для конкретного файлу. `HEARTBEAT.md` пропускається у звичайних запусках, коли
Heartbeats вимкнено для стандартного агента або
`agents.defaults.heartbeat.includeSystemPromptSection` має значення false. Тримайте впроваджені
файли стислими — особливо `MEMORY.md`, який може зростати з часом і призводити до
неочікувано високого використання контексту та частішої Compaction.

Коли сеанс працює на нативному harness Codex, Codex завантажує `AGENTS.md`
через власне project-doc discovery. OpenClaw усе одно розв’язує решту
bootstrap-файлів і передає їх як config instructions Codex, тож `SOUL.md`,
`TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md` і
`MEMORY.md` зберігають ту саму роль контексту робочої області без дублювання
`AGENTS.md`.

<Note>
Щоденні файли `memory/*.md` **не** є частиною звичайного bootstrap Project Context. У звичайних ходах до них звертаються на вимогу через інструменти `memory_search` і `memory_get`, тому вони не враховуються в context window, якщо модель явно їх не читає. Виняток — порожні ходи `/new` і `/reset`: runtime може додати recent daily memory на початок як одноразовий startup-context блок для цього першого ходу.
</Note>

Великі файли обрізаються з маркером. Максимальний розмір на файл контролюється
`agents.defaults.bootstrapMaxChars` (стандартно: 12000). Загальний впроваджений bootstrap
вміст у всіх файлах обмежений `agents.defaults.bootstrapTotalMaxChars`
(стандартно: 60000). Відсутні файли впроваджують короткий missing-file marker. Коли відбувається обрізання,
OpenClaw може впровадити стислий warning notice у системний промпт; керуйте цим через
`agents.defaults.bootstrapPromptTruncationWarning` (`off`, `once`, `always`;
стандартно: `once`). Детальні raw/injected лічильники залишаються в діагностиці, як-от
`/context`, `/status`, doctor і logs.

Сеанси sub-agent впроваджують лише `AGENTS.md` і `TOOLS.md` (інші bootstrap-файли
фільтруються, щоб зберегти контекст sub-agent малим).

Внутрішні hooks можуть перехоплювати цей крок через `agent:bootstrap`, щоб мутувати або замінювати
впроваджені bootstrap-файли (наприклад, замінювати `SOUL.md` на альтернативну persona).

Якщо хочете зробити звучання агента менш загальним, почніть із
[Посібника з особистості SOUL.md](/uk/concepts/soul).

Щоб перевірити, скільки додає кожен впроваджений файл (raw vs injected, truncation, плюс overhead схеми інструментів), використовуйте `/context list` або `/context detail`. Див. [Контекст](/uk/concepts/context).

## Обробка часу

Системний промпт містить окрему секцію **Поточна дата й час**, коли
часовий пояс користувача відомий. Щоб зберегти промпт cache-stable, тепер вона містить лише
**часовий пояс** (без динамічного годинника або формату часу).

Використовуйте `session_status`, коли агенту потрібен поточний час; status card
містить рядок timestamp. Той самий інструмент може за потреби встановити per-session model
override (`model=default` очищає його).

Налаштуйте через:

- `agents.defaults.userTimezone`
- `agents.defaults.timeFormat` (`auto` | `12` | `24`)

Див. [Дата й час](/uk/date-time) для повних деталей поведінки.

## Skills

Коли існують придатні Skills, OpenClaw впроваджує компактний **список доступних skills**
(`formatSkillsForPrompt`), який містить **шлях до файлу** для кожного skill. Промпт
інструктує модель використовувати `read`, щоб завантажити SKILL.md у вказаному
розташуванні (workspace, managed або bundled). Якщо жоден skill не придатний, секція
Skills пропускається.

Придатність включає gates метаданих skill, перевірки runtime environment/config
і ефективний allowlist skills агента, коли налаштовано `agents.defaults.skills` або
`agents.list[].skills`.

Skills, bundled із Plugin, придатні лише тоді, коли їхній Plugin-власник увімкнено.
Це дозволяє tool plugins надавати глибші операційні посібники без вбудовування всіх
цих вказівок безпосередньо в кожен опис інструмента.

```
<available_skills>
  <skill>
    <name>...</name>
    <description>...</description>
    <location>...</location>
  </skill>
</available_skills>
```

Це зберігає базовий промпт малим і водночас дає змогу цільово використовувати skills.

Бюджет списку skills належить підсистемі skills:

- Глобальне значення за замовчуванням: `skills.limits.maxSkillsPromptChars`
- Перевизначення для окремого агента: `agents.list[].skillsLimits.maxSkillsPromptChars`

Загальні обмежені фрагменти середовища виконання використовують іншу поверхню:

- `agents.defaults.contextLimits.*`
- `agents.list[].contextLimits.*`

Цей поділ відокремлює визначення розміру Skills від визначення розміру читання/інʼєкції середовища виконання, як-от `memory_get`, результатів живих інструментів і оновлень AGENTS.md після Compaction.

## Документація

Системний промпт містить розділ **Документація**. Коли локальна документація доступна, він
указує на локальний каталог документації OpenClaw (`docs/` у Git checkout або документацію
пакета npm у комплекті). Якщо локальна документація недоступна, він повертається до
[https://docs.openclaw.ai](https://docs.openclaw.ai).

Цей самий розділ також містить розташування вихідного коду OpenClaw. Git checkout відкривають локальний
корінь вихідного коду, щоб агент міг безпосередньо перевіряти код. Встановлення пакета містять URL
вихідного коду на GitHub і повідомляють агенту переглядати вихідний код там, коли документація неповна або
застаріла. Промпт також згадує публічне дзеркало документації, спільноту Discord і ClawHub
([https://clawhub.ai](https://clawhub.ai)) для пошуку Skills. Він повідомляє моделі, що
для поведінки, команд, конфігурації або архітектури OpenClaw спершу слід звертатися до документації, а також
за можливості самостійно запускати `openclaw status` (звертаючись до користувача лише тоді, коли їй бракує доступу).
Спеціально для конфігурації він спрямовує агентів до дії інструмента `gateway`
`config.schema.lookup`, щоб отримати точну документацію та обмеження на рівні полів, а потім до
`docs/gateway/configuration.md` і `docs/gateway/configuration-reference.md`
для ширших настанов.
