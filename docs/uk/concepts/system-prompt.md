OpenClaw створює власний системний промпт для кожного запуску агента. Промпт **належить OpenClaw** і не використовує типовий промпт pi-coding-agent.

Промпт збирається OpenClaw і впроваджується в кожен запуск агента.

Provider plugins можуть додавати cache-aware настанови для промпта без заміни
повного промпта, що належить OpenClaw. Provider runtime може:

- замінювати невеликий набір іменованих основних розділів (`interaction_style`,
  `tool_call_style`, `execution_bias`)
- вставляти **стабільний префікс** над межею кешу промпта
- вставляти **динамічний суфікс** під межею кешу промпта

Використовуйте внески, що належать provider, для налаштування під конкретні сімейства моделей. Залишайте застарілу
мутацію промпта `before_prompt_build` для сумісності або справді глобальних змін
промпта, а не для звичайної поведінки provider.

Оверлей сімейства OpenAI GPT-5 зберігає основне правило виконання невеликим і додає
настанови, специфічні для моделі, щодо закріплення persona, стислого виводу, дисципліни інструментів,
паралельного пошуку, покриття результатів, перевірки, відсутнього контексту та
гігієни термінальних інструментів.

## Структура

Промпт навмисно компактний і використовує фіксовані розділи:

- **Інструменти**: нагадування про структуровані інструменти як джерело істини плюс runtime-настанови щодо використання інструментів.
- **Схильність до виконання**: компактні настанови для доведення задач до кінця: діяти в межах поточного ходу для
  actionable-запитів, продовжувати до завершення або блокування, відновлюватися після слабких результатів
  інструментів, перевіряти змінний стан наживо та перевіряти перед фіналізацією.
- **Безпека**: коротке нагадування про запобіжники, щоб уникати поведінки, спрямованої на здобуття влади, або обходу нагляду.
- **Skills** (коли доступні): пояснює моделі, як завантажувати інструкції skill на вимогу.
- **Самооновлення OpenClaw**: як безпечно перевіряти конфігурацію через
  `config.schema.lookup`, застосовувати патчі до конфігурації через `config.patch`, замінювати повну
  конфігурацію через `config.apply` і запускати `update.run` лише за явним запитом
  користувача. Owner-only інструмент `gateway` також відмовляється переписувати
  `tools.exec.ask` / `tools.exec.security`, включно із застарілими псевдонімами `tools.bash.*`,
  які нормалізуються до цих захищених exec-шляхів.
- **Робоча область**: робочий каталог (`agents.defaults.workspace`).
- **Документація**: локальний шлях до документації OpenClaw (репозиторій або npm-пакет) і коли її читати.
- **Файли робочої області (впроваджені)**: вказує, що bootstrap-файли включені нижче.
- **Пісочниця** (коли ввімкнено): вказує sandboxed runtime, шляхи пісочниці та чи доступний exec з підвищеними правами.
- **Поточні дата й час**: локальний час користувача, часовий пояс і формат часу.
- **Теги відповіді**: необов’язковий синтаксис тегів відповіді для підтримуваних providers.
- **Heartbeats**: Heartbeat-промпт і поведінка підтвердження, коли Heartbeats увімкнені для типового агента.
- **Runtime**: хост, ОС, node, модель, корінь репозиторію (коли виявлено), рівень мислення (один рядок).
- **Міркування**: поточний рівень видимості + підказка про перемикач /reasoning.

OpenClaw тримає великий стабільний вміст, включно з **Контекстом проєкту**, над
внутрішньою межею кешу промпта. Нестабільні розділи каналу/сесії, як-от
настанови для вбудовування Control UI, **Повідомлення**, **Голос**, **Контекст групового чату**,
**Реакції**, **Heartbeats** і **Runtime**, додаються нижче цієї межі,
щоб локальні бекенди з префіксними кешами могли повторно використовувати стабільний префікс робочої області
між ходами каналу. Описи інструментів так само мають уникати вбудовування поточних
назв каналів, коли прийнята схема вже містить цю runtime-деталь.

Розділ «Інструменти» також включає runtime-настанови для довготривалої роботи:

- використовуйте cron для майбутніх follow-up (`check back later`, нагадування, повторювана робота)
  замість циклів сну в `exec`, трюків із затримкою `yieldMs` або повторного
  опитування `process`
- використовуйте `exec` / `process` лише для команд, які стартують зараз і продовжують працювати
  у фоновому режимі
- коли ввімкнене автоматичне пробудження після завершення, запустіть команду один раз і покладайтеся на
  push-based шлях пробудження, коли вона виводить дані або завершується з помилкою
- використовуйте `process` для логів, статусу, вводу або втручання, коли потрібно
  перевірити запущену команду
- якщо задача більша, віддавайте перевагу `sessions_spawn`; завершення sub-agent є
  push-based і автоматично оголошується назад requester
- не опитуйте `subagents list` / `sessions_list` у циклі лише для очікування
  завершення

Коли експериментальний інструмент `update_plan` увімкнено, розділ «Інструменти» також вказує
моделі використовувати його лише для нетривіальної багатоетапної роботи, тримати рівно один
крок `in_progress` і уникати повторення всього плану після кожного оновлення.

Запобіжники безпеки в системному промпті мають рекомендаційний характер. Вони спрямовують поведінку моделі, але не забезпечують примусове виконання політик. Для жорсткого примусу використовуйте політики інструментів, exec-схвалення, пісочницю та allowlists каналів; оператори можуть вимикати їх за задумом.

У каналах із нативними картками/кнопками схвалення runtime-промпт тепер каже
агенту спершу покладатися на цей нативний UI схвалення. Він має включати ручну
команду `/approve` лише тоді, коли результат інструмента каже, що схвалення в чаті недоступні або
ручне схвалення є єдиним шляхом.

## Режими промпта

OpenClaw може рендерити менші системні промпти для sub-agents. Runtime встановлює
`promptMode` для кожного запуску (це не user-facing конфігурація):

- `full` (типово): включає всі розділи вище.
- `minimal`: використовується для sub-agents; пропускає **Skills**, **Відновлення пам’яті**, **Самооновлення OpenClaw**,
  **Псевдоніми моделей**, **Ідентичність користувача**, **Теги відповіді**,
  **Повідомлення**, **Мовчазні відповіді** та **Heartbeats**. Інструменти, **Безпека**,
  робоча область, пісочниця, поточні дата й час (коли відомі), Runtime і впроваджений
  контекст залишаються доступними.
- `none`: повертає лише базовий рядок ідентичності.

Коли `promptMode=minimal`, додаткові впроваджені промпти позначаються як **Контекст subagent**
замість **Контекст групового чату**.

Для запусків авто-відповіді каналу OpenClaw може пропускати загальний розділ **Мовчазні відповіді**,
коли контекст прямого/групового чату вже включає розв’язану
conversation-specific поведінку `NO_REPLY`. Це уникає повторення механіки токенів
і в глобальному системному промпті, і в контексті каналу.

## Знімки промптів

OpenClaw зберігає закомічені happy-path знімки промптів для Codex/message-tool
runtime у `test/fixtures/agents/prompt-snapshots/happy-path/`. Вони рендерять
вибрані app-server параметри thread/turn плюс реконструйований стек шарів промпта, прив’язаний до моделі,
для прямих ходів Telegram, групових ходів Discord і heartbeat-ходів. Цей стек
включає pinned фікстуру промпта моделі Codex `gpt-5.5`, згенеровану з форми
каталогу/кешу моделей Codex, developer-текст дозволів Codex happy-path,
інструкції розробника OpenClaw, ввід ходу користувача та посилання на динамічні
специфікації інструментів.

Оновіть pinned фікстуру промпта моделі Codex за допомогою
`pnpm prompt:snapshots:sync-codex-model`. Типово скрипт шукає
runtime-кеш Codex у `$CODEX_HOME/models_cache.json`, потім
`~/.codex/models_cache.json`, і лише після цього повертається до maintainer-конвенції checkout Codex
у `~/code/codex/codex-rs/models-manager/models.json`. Якщо
жодне з цих джерел не існує, команда завершується без зміни закоміченої
фікстури. Передайте `--catalog <path>`, щоб оновити з конкретного файлу `models_cache.json`
або `models.json`.

Ці знімки все ще не є побайтовим raw-захопленням запиту OpenAI. Codex
може додавати runtime-owned контекст робочої області, як-от `AGENTS.md`, контекст
середовища, memories, інструкції app/plugin і майбутні інструкції collaboration-mode
всередині Codex runtime після того, як OpenClaw надсилає параметри thread і turn.

Перегенеруйте їх за допомогою `pnpm prompt:snapshots:gen` і перевірте drift за допомогою
`pnpm prompt:snapshots:check`. CI запускає перевірку drift у додатковому
boundary shard, щоб зміни промптів і оновлення знімків залишалися прив’язаними до того самого
PR.

## Впровадження bootstrap робочої області

Bootstrap-файли обрізаються та додаються під **Контекстом проєкту**, щоб модель бачила контекст ідентичності й профілю без потреби в явному читанні:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md` (лише для зовсім нових робочих областей)
- `MEMORY.md`, коли присутній

Усі ці файли **впроваджуються у вікно контексту** на кожному ході, якщо
не застосовується file-specific gate. `HEARTBEAT.md` пропускається у звичайних запусках, коли
heartbeats вимкнені для типового агента або
`agents.defaults.heartbeat.includeSystemPromptSection` має значення false. Тримайте впроваджені
файли стислими — особливо `MEMORY.md`, який може з часом зростати й призводити до
неочікувано високого використання контексту та частішої Compaction.

Коли сесія працює на нативному Codex harness, Codex завантажує `AGENTS.md`
через власне виявлення проєктної документації. OpenClaw все одно розв’язує решту
bootstrap-файлів і передає їх як інструкції конфігурації Codex, тож `SOUL.md`,
`TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md` і
`MEMORY.md` зберігають ту саму роль контексту робочої області без дублювання
`AGENTS.md`.

<Note>
Щоденні файли `memory/*.md` **не** є частиною звичайного bootstrap Контексту проєкту. У звичайних ходах до них звертаються на вимогу через інструменти `memory_search` і `memory_get`, тож вони не враховуються у вікні контексту, якщо модель явно їх не читає. Голі ходи `/new` і `/reset` є винятком: runtime може додати останню щоденну пам’ять на початок як одноразовий блок startup-context для цього першого ходу.
</Note>

Великі файли обрізаються з маркером. Максимальний розмір на файл контролюється
`agents.defaults.bootstrapMaxChars` (типово: 12000). Загальний впроваджений bootstrap-вміст
усіх файлів обмежується `agents.defaults.bootstrapTotalMaxChars`
(типово: 60000). Відсутні файли впроваджують короткий маркер missing-file. Коли відбувається обрізання,
OpenClaw може впровадити warning-блок у Контекст проєкту; керуйте цим через
`agents.defaults.bootstrapPromptTruncationWarning` (`off`, `once`, `always`;
типово: `once`).

Сесії sub-agent впроваджують лише `AGENTS.md` і `TOOLS.md` (інші bootstrap-файли
відфільтровуються, щоб зберегти контекст sub-agent малим).

Внутрішні hooks можуть перехоплювати цей крок через `agent:bootstrap`, щоб змінити або замінити
впроваджені bootstrap-файли (наприклад, замінити `SOUL.md` на альтернативну persona).

Якщо ви хочете, щоб агент звучав менш шаблонно, почніть із
[Посібника з особистості SOUL.md](/uk/concepts/soul).

Щоб перевірити, скільки додає кожен впроваджений файл (raw проти впровадженого, обрізання, плюс накладні витрати схеми інструментів), використовуйте `/context list` або `/context detail`. Див. [Контекст](/uk/concepts/context).

## Обробка часу

Системний промпт включає окремий розділ **Поточні дата й час**, коли
часовий пояс користувача відомий. Щоб зберегти cache-stable промпт, тепер він включає
лише **часовий пояс** (без динамічного годинника або формату часу).

Використовуйте `session_status`, коли агенту потрібен поточний час; картка статусу
містить рядок із timestamp. Той самий інструмент може необов’язково встановити per-session перевизначення моделі
(`model=default` очищає його).

Налаштовується через:

- `agents.defaults.userTimezone`
- `agents.defaults.timeFormat` (`auto` | `12` | `24`)

Див. [Дата й час](/uk/date-time), щоб отримати повні подробиці поведінки.

## Skills

Коли існують придатні skills, OpenClaw впроваджує компактний **список доступних skills**
(`formatSkillsForPrompt`), який включає **шлях до файлу** для кожного skill. Промпт
інструктує модель використовувати `read`, щоб завантажити SKILL.md за вказаним
розташуванням (робоча область, managed або bundled). Якщо немає придатних skills,
розділ Skills пропускається.

Придатність включає metadata gates skill, перевірки runtime-середовища/конфігурації
та ефективний allowlist skill агента, коли налаштовано `agents.defaults.skills` або
`agents.list[].skills`.

Skills, bundled у plugin, придатні лише тоді, коли їхній owning plugin увімкнений.
Це дозволяє tool plugins надавати глибші операційні посібники без вбудовування всіх
цих настанов безпосередньо в кожен опис інструмента.

```
<available_skills>
  <skill>
    <name>...</name>
    <description>...</description>
    <location>...</location>
  </skill>
</available_skills>
```

Це зберігає базовий промпт малим, водночас усе ще даючи змогу цільово використовувати skills.

Бюджет списку skills належить підсистемі skills:

- Глобальне типове значення: `skills.limits.maxSkillsPromptChars`
- Per-agent перевизначення: `agents.list[].skillsLimits.maxSkillsPromptChars`

Узагальнені обмежені runtime-уривки використовують іншу поверхню:

- `agents.defaults.contextLimits.*`
- `agents.list[].contextLimits.*`

Таке розділення відокремлює налаштування розмірів Skills від налаштування розмірів читання/ін’єкції під час виконання, зокрема `memory_get`, результатів live tool і оновлень AGENTS.md після Compaction.

## Документація

Системний prompt містить розділ **Документація**. Коли локальна документація доступна, він указує на локальний каталог документації OpenClaw (`docs/` у робочій копії Git або документацію з комплекту npm-пакета). Якщо локальна документація недоступна, використовується резервне джерело [https://docs.openclaw.ai](https://docs.openclaw.ai).

У цьому самому розділі також указано розташування вихідного коду OpenClaw. Робочі копії Git відкривають локальний корінь вихідного коду, щоб агент міг безпосередньо переглядати код. Інсталяції пакета містять URL вихідного коду на GitHub і вказують агенту переглядати вихідний код там, коли документація неповна або застаріла. Prompt також зазначає публічне дзеркало документації, спільноту Discord і ClawHub ([https://clawhub.ai](https://clawhub.ai)) для пошуку Skills. Він указує моделі спочатку звертатися до документації щодо поведінки, команд, конфігурації або архітектури OpenClaw, а також самостійно запускати `openclaw status`, коли це можливо (запитуючи користувача лише тоді, коли доступу немає). Зокрема для конфігурації він скеровує агентів до дії інструмента `gateway` `config.schema.lookup`, щоб отримати точну документацію й обмеження на рівні полів, а потім до `docs/gateway/configuration.md` і `docs/gateway/configuration-reference.md` для ширших рекомендацій.
