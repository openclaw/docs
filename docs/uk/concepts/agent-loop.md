Агентний цикл — це повний «реальний» запуск агента: прийом → збирання контексту → інференс моделі →
виконання інструментів → потокове передавання відповідей → збереження стану. Це авторитетний шлях, який перетворює повідомлення
на дії та фінальну відповідь, водночас зберігаючи узгодженість стану сесії.

В OpenClaw цикл — це один серіалізований запуск на сесію, який видає події життєвого циклу й потокові події,
поки модель міркує, викликає інструменти та передає вивід потоком. У цьому документі пояснюється, як цей справжній цикл
з’єднано наскрізно.

## Точки входу

- Gateway RPC: `agent` і `agent.wait`.
- CLI: команда `agent`.

## Як це працює (високий рівень)

1. RPC `agent` перевіряє параметри, визначає сесію (sessionKey/sessionId), зберігає метадані сесії та одразу повертає `{ runId, acceptedAt }`.
2. `agentCommand` запускає агента:
   - визначає типові значення model + thinking/verbose/trace
   - завантажує знімок Skills
   - викликає `runEmbeddedPiAgent` (runtime pi-agent-core)
   - видає **lifecycle end/error**, якщо вбудований цикл цього не зробив
3. `runEmbeddedPiAgent`:
   - серіалізує запуски через черги на рівні сесії та глобальні черги
   - визначає model + auth profile і будує сесію Pi
   - підписується на події Pi та передає дельти помічника/інструментів потоком
   - застосовує timeout -> перериває запуск, якщо ліміт перевищено
   - повертає payload-и та метадані usage
4. `subscribeEmbeddedPiSession` з’єднує події pi-agent-core з потоком OpenClaw `agent`:
   - події інструментів => `stream: "tool"`
   - дельти помічника => `stream: "assistant"`
   - події життєвого циклу => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` використовує `waitForAgentRun`:
   - очікує на **lifecycle end/error** для `runId`
   - повертає `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## Постановка в чергу + паралелізм

- Запуски серіалізуються для кожного ключа сесії (смуга сесії) й, за потреби, через глобальну смугу.
- Це запобігає гонкам інструментів/сесії та зберігає узгодженість історії сесії.
- Канали повідомлень можуть вибирати режими черги (collect/steer/followup), які подаються в цю систему смуг.
  Див. [Command Queue](/uk/concepts/queue).
- Записи транскрипту також захищені блокуванням запису сесії у файлі сесії. Блокування
  враховує процеси й базується на файлі, тому перехоплює записувачі, які оминають внутрішньопроцесну чергу або надходять
  з іншого процесу.
- Блокування запису сесії за замовчуванням не є реентрантними. Якщо допоміжний код навмисно вкладає захоплення
  того самого блокування, зберігаючи одного логічного записувача, він має явно ввімкнути це через
  `allowReentrant: true`.

## Підготовка сесії + робочого простору

- Робочий простір визначається та створюється; у sandboxed-запусках він може бути перенаправлений до кореня sandbox-робочого простору.
- Skills завантажуються (або перевикористовуються зі знімка) й інжектуються в env і prompt.
- Bootstrap/context-файли визначаються та інжектуються у звіт системного prompt.
- Захоплюється блокування запису сесії; `SessionManager` відкривається й готується до початку потокового передавання. Будь-який
  пізніший шлях перезапису, Compaction або обрізання транскрипту має взяти те саме блокування перед відкриттям або
  зміною файлу транскрипту.

## Збирання prompt + системний prompt

- Системний prompt будується з базового prompt OpenClaw, prompt Skills, bootstrap-контексту та перевизначень для окремого запуску.
- Застосовуються обмеження, специфічні для моделі, і резервні токени для Compaction.
- Див. [System prompt](/uk/concepts/system-prompt), щоб побачити, що саме бачить модель.

## Точки hook-ів (де можна перехопити)

OpenClaw має дві системи hook-ів:

- **Внутрішні hook-и** (Gateway hooks): скрипти, керовані подіями, для команд і подій життєвого циклу.
- **Plugin hooks**: точки розширення всередині життєвого циклу агента/інструментів і конвеєра gateway.

### Внутрішні hook-и (Gateway hooks)

- **`agent:bootstrap`**: запускається під час побудови bootstrap-файлів до остаточного формування системного prompt.
  Використовуйте це, щоб додавати/видаляти bootstrap-файли контексту.
- **Command hooks**: `/new`, `/reset`, `/stop` та інші події команд (див. документ Hooks).

Налаштування та приклади див. у [Hooks](/uk/automation/hooks).

### Plugin hooks (життєвий цикл агента + gateway)

Вони запускаються всередині циклу агента або конвеєра gateway:

- **`before_model_resolve`**: запускається до сесії (без `messages`), щоб детерміновано перевизначити provider/model до визначення моделі.
- **`before_prompt_build`**: запускається після завантаження сесії (з `messages`), щоб інжектувати `prependContext`, `systemPrompt`, `prependSystemContext` або `appendSystemContext` перед надсиланням prompt. Використовуйте `prependContext` для динамічного тексту на кожному ході, а поля system-context — для стабільних вказівок, які мають розміщуватися в просторі системного prompt.
- **`before_agent_start`**: застарілий hook сумісності, який може запускатися в будь-якій фазі; надавайте перевагу явним hook-ам вище.
- **`before_agent_reply`**: запускається після вбудованих дій і перед викликом LLM, дозволяючи Plugin перехопити хід і повернути синтетичну відповідь або повністю приглушити цей хід.
- **`agent_end`**: дає змогу перевірити фінальний список повідомлень і метадані запуску після завершення.
- **`before_compaction` / `after_compaction`**: дають змогу спостерігати за циклами Compaction або додавати до них анотації.
- **`before_tool_call` / `after_tool_call`**: перехоплюють параметри/результати інструментів.
- **`before_install`**: дає змогу перевірити вбудовані результати сканування й за потреби заблокувати встановлення Skill або Plugin.
- **`tool_result_persist`**: синхронно трансформує результати інструментів до їх запису в транскрипт сесії, що належить OpenClaw.
- **`message_received` / `message_sending` / `message_sent`**: hook-и для вхідних і вихідних повідомлень.
- **`session_start` / `session_end`**: межі життєвого циклу сесії.
- **`gateway_start` / `gateway_stop`**: події життєвого циклу gateway.

Правила ухвалення рішень hook-ами для захисту вихідних повідомлень/інструментів:

- `before_tool_call`: `{ block: true }` є термінальним і зупиняє обробники з нижчим пріоритетом.
- `before_tool_call`: `{ block: false }` не виконує жодної дії та не скасовує попереднє блокування.
- `before_install`: `{ block: true }` є термінальним і зупиняє обробники з нижчим пріоритетом.
- `before_install`: `{ block: false }` не виконує жодної дії та не скасовує попереднє блокування.
- `message_sending`: `{ cancel: true }` є термінальним і зупиняє обробники з нижчим пріоритетом.
- `message_sending`: `{ cancel: false }` не виконує жодної дії та не скасовує попереднє скасування.

API hook-ів і подробиці реєстрації див. у [Plugin hooks](/uk/plugins/hooks).

Harness-обгортки можуть адаптувати ці hook-и по-різному. Harness app-server у Codex зберігає
Plugin hooks OpenClaw як контракт сумісності для задокументованих дзеркальних
поверхонь, тоді як native hooks Codex залишаються окремим нижчорівневим механізмом Codex.

## Потокове передавання + часткові відповіді

- Дельти помічника передаються потоком із pi-agent-core і видаються як події `assistant`.
- Блокове потокове передавання може видавати часткові відповіді або на `text_end`, або на `message_end`.
- Потокове передавання reasoning може видаватися як окремий потік або як блокові відповіді.
- Див. [Streaming](/uk/concepts/streaming) щодо чанкування та поведінки блокових відповідей.

## Виконання інструментів + інструменти повідомлень

- Події початку/оновлення/завершення інструментів видаються у потік `tool`.
- Результати інструментів очищуються з урахуванням розміру та image-payload-ів перед логуванням/видачею.
- Надсилання через інструменти повідомлень відстежуються, щоб пригнічувати дубльовані підтвердження помічника.

## Формування відповіді + пригнічення

- Фінальні payload-и збираються з:
  - тексту помічника (і за потреби reasoning)
  - вбудованих підсумків інструментів (коли verbose увімкнено й це дозволено)
  - тексту помилки помічника, якщо модель завершується помилкою
- Точний silent-токен `NO_REPLY` / `no_reply` відфільтровується з вихідних
  payload-ів.
- Дублікати інструментів повідомлень видаляються з фінального списку payload-ів.
- Якщо не залишається payload-ів, придатних для рендерингу, і інструмент завершився помилкою, видається
  резервна відповідь про помилку інструмента
  (якщо тільки інструмент повідомлень уже не надіслав видиму користувачу відповідь).

## Compaction + повторні спроби

- Автоматичний Compaction видає події потоку `compaction` і може спричинити повторну спробу.
- Під час повторної спроби буфери в пам’яті та підсумки інструментів скидаються, щоб уникнути дублювання виводу.
- Див. [Compaction](/uk/concepts/compaction) щодо конвеєра Compaction.

## Потоки подій (сьогодні)

- `lifecycle`: видається `subscribeEmbeddedPiSession` (і як резервний варіант — `agentCommand`)
- `assistant`: потокові дельти з pi-agent-core
- `tool`: потокові події інструментів із pi-agent-core

## Обробка chat-каналу

- Дельти помічника буферизуються в chat-повідомлення `delta`.
- Chat `final` видається на **lifecycle end/error**.

## Тайм-аути

- Типове значення `agent.wait`: 30 с (лише очікування). Перевизначається параметром `timeoutMs`.
- Runtime агента: типове значення `agents.defaults.timeoutSeconds` — 172800 с (48 годин); застосовується в таймері переривання `runEmbeddedPiAgent`.
- Тайм-аут бездіяльності LLM: `agents.defaults.llm.idleTimeoutSeconds` перериває запит до моделі, якщо до завершення вікна бездіяльності не надходять чанки відповіді. Установіть його явно для повільних локальних моделей або provider-ів reasoning/tool-call; установіть `0`, щоб вимкнути. Якщо його не задано, OpenClaw використовує `agents.defaults.timeoutSeconds`, якщо налаштовано, інакше — 120 с. Для запусків, ініційованих Cron, без явного тайм-ауту LLM або агента, idle-watchdog вимикається, і використовується зовнішній тайм-аут Cron.
- Тайм-аут HTTP-запиту provider-а: `models.providers.<id>.timeoutSeconds` застосовується лише до HTTP-fetch-запитів моделі цього provider-а, включно з підключенням, заголовками, тілом і загальною обробкою guarded-fetch abort. Використовуйте це для повільних локальних/self-hosted provider-ів, таких як Ollama, перш ніж збільшувати тайм-аут усього runtime агента.
