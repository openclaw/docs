OpenClaw надає агентам інструменти для роботи між сесіями, перевірки статусу та
оркестрації субагентів.

## Доступні інструменти

| Інструмент         | Що він робить                                                               |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | Показує список сесій з необов’язковими фільтрами (тип, мітка, агент, давність, попередній перегляд) |
| `sessions_history` | Читає стенограму конкретної сесії                                           |
| `sessions_send`    | Надсилає повідомлення до іншої сесії та за потреби очікує                    |
| `sessions_spawn`   | Запускає ізольовану сесію субагента для фонової роботи                      |
| `sessions_yield`   | Завершує поточний хід і очікує подальших результатів субагента              |
| `subagents`        | Показує список, керує або завершує запущених субагентів для цієї сесії      |
| `session_status`   | Показує картку у стилі `/status` і за потреби задає перевизначення моделі для сесії |

Ці інструменти, як і раніше, підпорядковуються активному профілю інструментів і
політиці allow/deny. `tools.profile: "coding"` включає повний набір
оркестрації сесій, зокрема `sessions_spawn`, `sessions_yield` і `subagents`.
`tools.profile: "messaging"` включає інструменти міжсесійного обміну
повідомленнями (`sessions_list`, `sessions_history`, `sessions_send`, `session_status`), але
не включає запуск субагентів. Щоб зберегти профіль messaging і водночас
дозволити нативне делегування, додайте:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

Політики групи, провайдера, пісочниці та окремого агента все ще можуть
прибрати ці інструменти після етапу профілю. Використовуйте `/tools` із
відповідної сесії, щоб перевірити фактичний список інструментів.

## Перелік і читання сесій

`sessions_list` повертає сесії з їхнім ключем, agentId, типом, каналом, моделлю,
кількістю токенів і часовими позначками. Можна фільтрувати за типом (`main`, `group`, `cron`, `hook`,
`node`), точною `label`, точним `agentId`, текстом пошуку або давністю
(`activeMinutes`). Якщо вам потрібне сортування у стилі поштової скриньки, інструмент також
може запитувати похідний заголовок в межах видимості, фрагмент попереднього перегляду
останнього повідомлення або обмежену кількість недавніх повідомлень у кожному рядку.
Похідні заголовки й попередні перегляди створюються лише для тих сесій,
які викликаючий агент уже може бачити згідно з налаштованою політикою
видимості інструментів сесії, тож непов’язані сесії залишаються прихованими.

`sessions_history` отримує стенограму розмови для конкретної сесії.
За замовчуванням результати інструментів виключаються — передайте `includeTools: true`, щоб їх побачити.
Повернуте представлення навмисно обмежене й відфільтроване з міркувань безпеки:

- текст помічника нормалізується перед відновленням:
  - теги thinking видаляються
  - блоки каркаса `<relevant-memories>` / `<relevant_memories>` видаляються
  - текстові XML-блоки викликів інструментів, такі як `<tool_call>...</tool_call>`,
    `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` і
    `<function_calls>...</function_calls>`, видаляються, включно з
    обрізаними payload, які так і не закриваються коректно
  - спрощений каркас викликів/результатів інструментів, такий як `[Tool Call: ...]`,
    `[Tool Result ...]` і `[Historical context ...]`, видаляється
  - витоки керівних токенів моделі, такі як `<|assistant|>`, інші ASCII
    токени `<|...|>` і повноширинні варіанти `<｜...｜>`, видаляються
  - некоректний XML викликів інструментів MiniMax, такий як `<invoke ...>` /
    `</minimax:tool_call>`, видаляється
- текст, схожий на облікові дані/токени, редагується перед поверненням
- довгі текстові блоки обрізаються
- дуже великі історії можуть відкидати старіші рядки або замінювати
  надто великий рядок на `[sessions_history omitted: message too large]`
- інструмент повідомляє підсумкові прапорці, такі як `truncated`, `droppedMessages`,
  `contentTruncated`, `contentRedacted` і `bytes`

Обидва інструменти приймають або **ключ сесії** (наприклад, `"main"`), або **ID сесії**
з попереднього виклику списку.

Якщо вам потрібна точна стенограма побайтно, перевіряйте файл стенограми на
диску, а не використовуйте `sessions_history` як необроблений дамп.

## Надсилання міжсесійних повідомлень

`sessions_send` доставляє повідомлення до іншої сесії та за потреби очікує
на відповідь:

- **Fire-and-forget:** задайте `timeoutSeconds: 0`, щоб поставити в чергу й
  одразу повернутися.
- **Очікування відповіді:** задайте тайм-аут і отримайте відповідь безпосередньо.

Після відповіді цільової сесії OpenClaw може запустити **цикл зворотних відповідей**,
у якому агенти чергують повідомлення (до 5 ходів). Цільовий агент може відповісти
`REPLY_SKIP`, щоб зупинити це раніше.

## Допоміжні інструменти статусу й оркестрації

`session_status` — це полегшений еквівалент `/status` для поточної
або іншої видимої сесії. Він повідомляє про використання, час, стан моделі/середовища виконання
та пов’язаний контекст фонового завдання, якщо він є. Подібно до `/status`, він може
дозаповнювати розріджені лічильники токенів/кешу з останнього запису використання у стенограмі, а
`model=default` очищає перевизначення для сесії. Використовуйте `sessionKey="current"` для
поточної сесії викликача; видимі клієнтські мітки, такі як `openclaw-tui`, не є ключами сесії.

`sessions_yield` навмисно завершує поточний хід, щоб наступним повідомленням
стала подія продовження, на яку ви чекаєте. Використовуйте це після запуску субагентів, коли
хочете, щоб результати завершення прийшли наступним повідомленням, а не через цикли опитування.

`subagents` — це допоміжний інструмент площини керування для вже запущених субагентів
OpenClaw. Він підтримує:

- `action: "list"` для перевірки активних/нещодавніх запусків
- `action: "steer"` для надсилання додаткових вказівок запущеному дочірньому агенту
- `action: "kill"` для зупинки одного дочірнього агента або `all`

## Запуск субагентів

`sessions_spawn` за замовчуванням створює ізольовану сесію для фонового завдання.
Він завжди неблокувальний — одразу повертає `runId` і
`childSessionKey`.

Ключові параметри:

- `runtime: "subagent"` (типово) або `"acp"` для зовнішніх агентів harness.
- перевизначення `model` і `thinking` для дочірньої сесії.
- `thread: true`, щоб прив’язати запуск до потоку чату (Discord, Slack тощо).
- `sandbox: "require"`, щоб примусово використовувати пісочницю для дочірнього агента.
- `context: "fork"` для нативних субагентів, коли дочірньому агенту потрібна стенограма
  поточного запитувача; пропустіть цей параметр або використайте `context: "isolated"` для чистої дочірньої сесії.

Листові субагенти за замовчуванням не отримують інструментів сесії. Якщо
`maxSpawnDepth >= 2`, то субагенти-оркестратори рівня глибини 1 додатково отримують
`sessions_spawn`, `subagents`, `sessions_list` і `sessions_history`, щоб вони
могли керувати власними дочірніми агентами. Листові запуски все одно не отримують рекурсивних
інструментів оркестрації.

Після завершення крок сповіщення публікує результат у канал запитувача.
Доставка завершення зберігає прив’язану маршрутизацію потоку/теми, коли це можливо, а якщо
джерело завершення ідентифікує лише канал, OpenClaw усе одно може повторно використати
збережений маршрут сесії запитувача (`lastChannel` / `lastTo`) для прямої
доставки.

Для поведінки, специфічної для ACP, див. [ACP Agents](/uk/tools/acp-agents).

## Видимість

Інструменти сесії мають обмежену область видимості, щоб обмежити те, що агент може бачити:

| Рівень  | Область видимості                        |
| ------- | ---------------------------------------- |
| `self`  | Лише поточна сесія                       |
| `tree`  | Поточна сесія + запущені субагенти       |
| `agent` | Усі сесії для цього агента               |
| `all`   | Усі сесії (між агентами, якщо налаштовано) |

За замовчуванням використовується `tree`. Сесії в пісочниці примусово обмежуються до `tree`
незалежно від конфігурації.
