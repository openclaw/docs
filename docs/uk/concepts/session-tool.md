OpenClaw надає агентам інструменти для роботи між сеансами, перевірки статусу та
оркестрації субагентів.

## Доступні інструменти

| Інструмент        | Що робить                                                                 |
| ----------------- | ------------------------------------------------------------------------- |
| `sessions_list`   | Перелічує сеанси з необов’язковими фільтрами (вид, мітка, агент, давність, попередній перегляд) |
| `sessions_history` | Читає транскрипт певного сеансу                                           |
| `sessions_send`   | Надсилає повідомлення в інший сеанс і необов’язково чекає                 |
| `sessions_spawn`  | Запускає ізольований сеанс субагента для фонової роботи                   |
| `sessions_yield`  | Завершує поточний хід і чекає на подальші результати субагентів           |
| `subagents`       | Перелічує статус запущених субагентів для цього сеансу                    |
| `session_status`  | Показує картку у стилі `/status` і необов’язково задає перевизначення моделі для окремого сеансу |

На ці інструменти й надалі поширюються активний профіль інструментів і політика
дозволів/заборон. `tools.profile: "coding"` містить повний набір оркестрації
сеансів, зокрема `sessions_spawn`, `sessions_yield` і `subagents`.
`tools.profile: "messaging"` містить інструменти міжсеансових повідомлень
(`sessions_list`, `sessions_history`, `sessions_send`, `session_status`), але
не містить запуску субагентів. Щоб зберегти профіль повідомлень і все одно
дозволити нативне делегування, додайте:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

Політики групи, постачальника, пісочниці та окремого агента все ще можуть
вилучити ці інструменти після етапу профілю. Використовуйте `/tools` з
відповідного сеансу, щоб перевірити фактичний список інструментів.

## Перелік і читання сеансів

`sessions_list` повертає сеанси з їхнім ключем, agentId, видом, каналом, моделлю,
лічильниками токенів і часовими позначками. Фільтруйте за видом (`main`, `group`, `cron`, `hook`,
`node`), точним `label`, точним `agentId`, текстом пошуку або давністю
(`activeMinutes`). Коли потрібне сортування у стилі поштової скриньки, інструмент
також може запросити похідний заголовок з урахуванням видимості, фрагмент
попереднього перегляду останнього повідомлення або обмежені нещодавні повідомлення
в кожному рядку. Похідні заголовки й попередні перегляди створюються лише для
сеансів, які викликач уже може бачити відповідно до налаштованої політики видимості
інструментів сеансу, тому непов’язані сеанси лишаються прихованими. Коли видимість
обмежена, `sessions_list` повертає необов’язкові метадані `visibility`, що
показують фактичний режим і попередження, що результати можуть бути обмежені
областю видимості.

`sessions_history` отримує транскрипт розмови для певного сеансу.
За замовчуванням результати інструментів вилучено -- передайте `includeTools: true`, щоб побачити їх.
Повернене подання навмисно обмежене й відфільтроване з міркувань безпеки:

- текст асистента нормалізується перед пригадуванням:
  - теги мислення вилучаються
  - службові блоки `<relevant-memories>` / `<relevant_memories>` вилучаються
  - текстові XML-блоки викликів інструментів, як-от `<tool_call>...</tool_call>`,
    `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` і
    `<function_calls>...</function_calls>`, вилучаються, включно з обрізаними
    корисними навантаженнями, які ніколи коректно не закриваються
  - понижені службові блоки викликів/результатів інструментів, як-от `[Tool Call: ...]`,
    `[Tool Result ...]` і `[Historical context ...]`, вилучаються
  - витеклі керівні токени моделі, як-от `<|assistant|>`, інші ASCII-токени
    `<|...|>` і повноширинні варіанти `<｜...｜>`, вилучаються
  - некоректний XML викликів інструментів MiniMax, як-от `<invoke ...>` /
    `</minimax:tool_call>`, вилучається
- текст, схожий на облікові дані або токени, редагується перед поверненням
- довгі текстові блоки обрізаються
- дуже великі історії можуть відкидати старіші рядки або замінювати завеликий рядок на
  `[sessions_history omitted: message too large]`
- інструмент повідомляє прапорці зведення, як-от `truncated`, `droppedMessages`,
  `contentTruncated`, `contentRedacted` і `bytes`

Обидва інструменти приймають або **ключ сеансу** (наприклад, `"main"`), або **ID сеансу**
з попереднього виклику списку.

Якщо потрібен точний побайтовий транскрипт, перевіряйте файл транскрипта на
диску, а не трактуйте `sessions_history` як сирий дамп.

## Надсилання міжсеансових повідомлень

`sessions_send` доставляє повідомлення в інший сеанс і необов’язково чекає на
відповідь:

- **Надіслати й не чекати:** задайте `timeoutSeconds: 0`, щоб поставити в чергу й
  повернутися негайно.
- **Чекати відповіді:** задайте тайм-аут і отримайте відповідь вбудовано.

Чатові сеанси з областю потоку, як-от ключі Slack або Discord, що закінчуються на
`:thread:<id>`, не є допустимими цілями `sessions_send`. Використовуйте ключ
сеансу батьківського каналу для координації між агентами, щоб повідомлення,
маршрутизовані інструментом, не з’являлися всередині активного потоку, зверненого
до людини.

Повідомлення та подальші відповіді A2A позначаються як міжсеансові дані в
промпті отримувача (`[Inter-session message ... isUser=false]`) і в походженні
транскрипта. Агент-отримувач має трактувати їх як дані, маршрутизовані інструментом,
а не як пряму інструкцію, написану кінцевим користувачем.

Після відповіді цілі OpenClaw може запустити **цикл відповіді назад**, у якому
агенти чергують повідомлення (до `session.agentToAgent.maxPingPongTurns`, діапазон
0-20, стандартно 5). Цільовий агент може відповісти
`REPLY_SKIP`, щоб зупинитися раніше.

## Помічники статусу та оркестрації

`session_status` — це легкий інструмент, еквівалентний `/status`, для поточного
або іншого видимого сеансу. Він повідомляє використання, час, стан моделі/середовища
виконання та пов’язаний контекст фонового завдання, якщо він є. Як і `/status`,
він може дозаповнювати розріджені лічильники токенів/кешу з останнього запису
використання транскрипта, а `model=default` очищає перевизначення для окремого
сеансу. Використовуйте `sessionKey="current"` для поточного сеансу викликача;
видимі мітки клієнта, як-от `openclaw-tui`, не є ключами сеансів.

Коли доступні метадані маршруту, `session_status` також містить видимий JSON-блок
`Route context` і відповідні структуровані поля `details`. Ці поля розрізняють
ключ сеансу та маршрут, який наразі обробляє живий запуск:

- `origin` — це місце, де було створено сеанс, або постачальник, виведений із
  префікса ключа сеансу, придатного для доставки, коли старішому стану бракує
  збережених метаданих походження.
- `active` — це поточний маршрут живого запуску. Він повідомляється лише для
  живого або поточного сеансу, який обробляється зараз.
- `deliveryContext` — це збережений маршрут доставки, записаний у сеансі,
  який OpenClaw може повторно використати для пізнішої доставки, навіть коли
  активна поверхня відрізняється.

`sessions_yield` навмисно завершує поточний хід, щоб наступним повідомленням
могла бути подальша подія, на яку ви чекаєте. Використовуйте його після запуску
субагентів, коли хочете, щоб результати завершення надійшли як наступне повідомлення,
а не будувати цикли опитування.

`subagents` — це помічник видимості для вже запущених субагентів OpenClaw.
Він підтримує `action: "list"` для перевірки активних/нещодавніх запусків.

## Запуск субагентів

`sessions_spawn` за замовчуванням створює ізольований сеанс для фонового завдання.
Він завжди неблокувальний -- він негайно повертає `runId` і
`childSessionKey`. Нативні запуски субагентів отримують делеговане завдання в
першому видимому повідомленні дочірнього сеансу `[Subagent Task]`, тоді як
системний промпт містить лише правила середовища виконання субагента й контекст
маршрутизації.

Ключові параметри:

- `runtime: "subagent"` (стандартно) або `"acp"` для зовнішніх агентів-обгорток.
- Перевизначення `model` і `thinking` для дочірнього сеансу.
- `thread: true`, щоб прив’язати запуск до чатового потоку (Discord, Slack тощо).
- `sandbox: "require"`, щоб примусово застосувати пісочницю до дочірнього сеансу.
- `context: "fork"` для нативних субагентів, коли дочірньому сеансу потрібен поточний
  транскрипт запитувача; пропустіть це або використайте `context: "isolated"` для чистого дочірнього сеансу.
  Нативні субагенти, прив’язані до потоку, стандартно використовують `context: "fork"`, якщо
  `threadBindings.defaultSpawnContext` не вказує інакше.

Стандартні листові субагенти не отримують інструменти сеансу. Коли
`maxSpawnDepth >= 2`, субагенти-оркестратори глибини 1 додатково отримують
`sessions_spawn`, `subagents`, `sessions_list` і `sessions_history`, щоб вони
могли керувати власними дочірніми агентами. Листові запуски все одно не отримують
рекурсивні інструменти оркестрації.

Після завершення крок оголошення публікує результат у канал запитувача.
Доставка завершення зберігає прив’язану маршрутизацію потоку/теми, коли вона
доступна, а якщо походження завершення визначає лише канал, OpenClaw все одно
може повторно використати збережений маршрут сеансу запитувача (`lastChannel` / `lastTo`) для прямої
доставки.

Про поведінку, специфічну для ACP, див. [Агенти ACP](/uk/tools/acp-agents).

## Видимість

Інструменти сеансу обмежуються областю, щоб контролювати, що агент може бачити:

| Рівень  | Область                                  |
| ------- | ---------------------------------------- |
| `self`  | Лише поточний сеанс                      |
| `tree`  | Поточний сеанс + запущені субагенти      |
| `agent` | Усі сеанси цього агента                  |
| `all`   | Усі сеанси (між агентами, якщо налаштовано) |

Стандартне значення — `tree`. Сеанси в пісочниці обмежуються до `tree` незалежно
від конфігурації.
