OpenClaw надає агентам інструменти для роботи між сесіями, перегляду стану та
оркестрації субагентів.

## Доступні інструменти

| Інструмент        | Що він робить                                                              |
| ----------------- | -------------------------------------------------------------------------- |
| `sessions_list`   | Перелічує сесії з необов’язковими фільтрами (kind, label, agent, recency, preview) |
| `sessions_history`| Читає transcript конкретної сесії                                          |
| `sessions_send`   | Надсилає повідомлення в іншу сесію та за потреби очікує                    |
| `sessions_spawn`  | Створює ізольовану сесію субагента для фонової роботи                      |
| `sessions_yield`  | Завершує поточний хід і чекає на follow-up результати субагента            |
| `subagents`       | Перелічує, спрямовує або зупиняє створених субагентів для цієї сесії       |
| `session_status`  | Показує картку у стилі `/status` і за потреби встановлює перевизначення моделі для сесії |

## Перелік і читання сесій

`sessions_list` повертає сесії з їхніми ключем, agentId, kind, channel, model,
кількістю токенів і часовими мітками. Можна фільтрувати за kind (`main`, `group`, `cron`, `hook`,
`node`), точним `label`, точним `agentId`, текстом пошуку або давністю
(`activeMinutes`). Якщо вам потрібне сортування в стилі поштової скриньки, інструмент також може запитувати
похідний заголовок з урахуванням області видимості, фрагмент попереднього перегляду останнього повідомлення або обмежені
останні повідомлення в кожному рядку. Похідні заголовки й попередні перегляди створюються лише для
сесій, які викликач уже може бачити згідно з налаштованою політикою
видимості інструментів сесії, тож нерелевантні сесії залишаються прихованими.

`sessions_history` отримує transcript розмови для конкретної сесії.
Типово результати інструментів виключено — передайте `includeTools: true`, щоб їх побачити.
Повернутий вигляд навмисно обмежений і відфільтрований для безпеки:

- текст помічника нормалізується перед recall:
  - теґи thinking видаляються
  - блоки каркаса `<relevant-memories>` / `<relevant_memories>` видаляються
  - блоки XML-навантаження викликів інструментів у звичайному тексті, такі як `<tool_call>...</tool_call>`,
    `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` і
    `<function_calls>...</function_calls>`, видаляються, включно з усіченими
    навантаженнями, які так і не закриваються коректно
  - понижений каркас викликів/результатів інструментів, такий як `[Tool Call: ...]`,
    `[Tool Result ...]` і `[Historical context ...]`, видаляється
  - витеклі токени керування моделлю, такі як `<|assistant|>`, інші ASCII
    токени `<|...|>` і повноширинні варіанти `<｜...｜>`, видаляються
  - некоректний XML викликів інструментів MiniMax, такий як `<invoke ...>` /
    `</minimax:tool_call>`, видаляється
- текст, схожий на облікові дані/токени, редагується до повернення
- довгі текстові блоки усікаються
- дуже великі історії можуть відкидати старіші рядки або замінювати надто великий рядок на
  `[sessions_history omitted: message too large]`
- інструмент повідомляє підсумкові прапорці, такі як `truncated`, `droppedMessages`,
  `contentTruncated`, `contentRedacted` і `bytes`

Обидва інструменти приймають або **ключ сесії** (наприклад `"main"`), або **ID сесії**
з попереднього виклику list.

Якщо вам потрібен точний transcript побайтно, перегляньте файл transcript на
диску, а не використовуйте `sessions_history` як сирий дамп.

## Надсилання міжсесійних повідомлень

`sessions_send` доставляє повідомлення до іншої сесії та за потреби чекає на
відповідь:

- **Fire-and-forget:** установіть `timeoutSeconds: 0`, щоб поставити в чергу й
  одразу повернутися.
- **Чекати відповіді:** установіть тайм-аут і отримайте відповідь inline.

Після відповіді цільової сесії OpenClaw може запустити **цикл відповіді у відповідь**, де
агенти чергують повідомлення (до 5 ходів). Цільовий агент може відповісти
`REPLY_SKIP`, щоб зупинитися раніше.

## Допоміжні інструменти стану й оркестрації

`session_status` — це легковаговий еквівалент `/status` для поточної
або іншої видимої сесії. Він повідомляє про використання, час, стан моделі/runtime і
пов’язаний контекст фонового завдання, якщо він є. Як і `/status`, він може дозаповнювати
неповні лічильники токенів/кешу з останнього запису використання transcript, а
`model=default` очищає перевизначення для окремої сесії.

`sessions_yield` навмисно завершує поточний хід, щоб наступним повідомленням могла стати
follow-up подія, на яку ви чекаєте. Використовуйте його після створення субагентів, коли
хочете, щоб результати завершення надійшли як наступне повідомлення замість побудови
циклів опитування.

`subagents` — це допоміжний інструмент control plane для вже створених OpenClaw
субагентів. Він підтримує:

- `action: "list"` для перегляду активних/нещодавніх запусків
- `action: "steer"` для надсилання follow-up вказівок запущеному дочірньому процесу
- `action: "kill"` для зупинки одного дочірнього процесу або `all`

## Створення субагентів

`sessions_spawn` типово створює ізольовану сесію для фонового завдання.
Він завжди є неблокувальним — одразу повертає `runId` і
`childSessionKey`.

Ключові параметри:

- `runtime: "subagent"` (типово) або `"acp"` для зовнішніх harness-агентів.
- Перевизначення `model` і `thinking` для дочірньої сесії.
- `thread: true` для прив’язки створення до потоку чату (Discord, Slack тощо).
- `sandbox: "require"` для примусового sandboxing дочірньої сесії.
- `context: "fork"` для нативних субагентів, коли дочірній процес потребує
  transcript поточного запитувача; не вказуйте або використовуйте `context: "isolated"` для чистого дочірнього процесу.

Типові кінцеві субагенти не отримують інструменти сесії. Коли
`maxSpawnDepth >= 2`, субагенти-оркестратори глибини 1 додатково отримують
`sessions_spawn`, `subagents`, `sessions_list` і `sessions_history`, щоб вони
могли керувати власними дочірніми процесами. Кінцеві запуски, як і раніше, не отримують рекурсивних
інструментів оркестрації.

Після завершення крок оголошення публікує результат у каналі запитувача.
Доставка після завершення зберігає прив’язану маршрутизацію потоку/теми, коли це доступно, а якщо
джерело завершення ідентифікує лише канал, OpenClaw усе одно може повторно використати збережений маршрут
сесії запитувача (`lastChannel` / `lastTo`) для прямої
доставки.

Специфічну для ACP поведінку див. у [ACP Agents](/uk/tools/acp-agents).

## Видимість

Інструменти сесії мають область дії, щоб обмежити те, що агент може бачити:

| Рівень  | Область дії                              |
| ------- | ---------------------------------------- |
| `self`  | Лише поточна сесія                       |
| `tree`  | Поточна сесія + створені субагенти       |
| `agent` | Усі сесії для цього агента               |
| `all`   | Усі сесії (між агентами, якщо налаштовано) |

Типове значення — `tree`. Сесії в sandbox примусово обмежуються до `tree`
незалежно від конфігурації.
