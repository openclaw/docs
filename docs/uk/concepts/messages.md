Ця сторінка поєднує інформацію про те, як OpenClaw обробляє вхідні повідомлення, сеанси, постановку в чергу, потокову передачу та видимість міркувань.

## Потік повідомлень (загальний огляд)

```
Inbound message
  -> routing/bindings -> session key
  -> queue (if a run is active)
  -> agent run (streaming + tools)
  -> outbound replies (channel limits + chunking)
```

Ключові параметри розташовані в конфігурації:

- `messages.*` для префіксів, постановки в чергу та поведінки груп.
- `agents.defaults.*` для типових значень блокової потокової передачі та поділу на частини.
- Перевизначення каналів (`channels.whatsapp.*`, `channels.telegram.*` тощо) для обмежень і перемикачів потокової передачі.

Див. [Конфігурація](/uk/gateway/configuration) для повної схеми.

## Усунення дублікатів вхідних повідомлень

Канали можуть повторно доставляти те саме повідомлення після повторних підключень. OpenClaw зберігає короткочасний кеш із ключем на основі каналу/облікового запису/співрозмовника/сеансу/ідентифікатора повідомлення, щоб повторні доставки не запускали ще один запуск агента.

## Усунення дрібних інтервалів між вхідними повідомленнями

Швидкі послідовні повідомлення від **того самого відправника** можна об’єднати в один хід агента через `messages.inbound`. Усунення дрібних інтервалів застосовується в межах каналу + розмови та використовує найновіше повідомлення для прив’язки відповіді/ідентифікаторів.

Конфігурація (глобальне типове значення + перевизначення для окремих каналів):

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

Примітки:

- Усунення дрібних інтервалів застосовується до повідомлень **лише з текстом**; медіа/вкладення скидають буфер негайно.
- Керівні команди оминають це об’єднання, щоб залишатися окремими — **окрім** випадків, коли канал явно вмикає об’єднання DM від того самого відправника (наприклад, [BlueBubbles `coalesceSameSenderDms`](/uk/channels/bluebubbles#coalescing-split-send-dms-command--url-in-one-composition)), де DM-команди очікують у межах вікна debounce, щоб розділене навантаження надсилання могло приєднатися до того самого ходу агента.

## Сеанси та пристрої

Сеанси належать Gateway, а не клієнтам.

- Прямі чати згортаються до основного ключа сеансу агента.
- Групи/канали отримують власні ключі сеансів.
- Сховище сеансів і стенограми зберігаються на хості Gateway.

Кілька пристроїв/каналів можуть відображатися на той самий сеанс, але історія не синхронізується повністю назад до кожного клієнта. Рекомендація: використовуйте один основний пристрій для довгих розмов, щоб уникнути розбіжного контексту. Control UI і TUI завжди показують стенограму сеансу, що підтримується Gateway, тож саме вони є джерелом істини.

Докладніше: [Керування сеансами](/uk/concepts/session).

## Метадані результату інструмента

`content` результату інструмента — це видимий моделі результат. `details` результату інструмента — це метадані середовища виконання для відображення в UI, діагностики, доставки медіа та Plugin.

OpenClaw чітко підтримує цю межу:

- `toolResult.details` вилучається перед повторним відтворенням у провайдері та перед входом для Compaction.
- У збережених стенограмах сеансів лишаються лише обмежені `details`; завеликі метадані замінюються компактним підсумком, позначеним як `persistedDetailsTruncated: true`.
- Plugins та інструменти мають поміщати текст, який модель повинна прочитати, у `content`, а не лише в `details`.

## Вхідні тіла повідомлень та контекст історії

OpenClaw відокремлює **тіло запиту** від **тіла команди**:

- `Body`: текст запиту, надісланий агенту. Він може містити оболонки каналу та необов’язкові обгортки історії.
- `CommandBody`: сирий текст користувача для розбору директив/команд.
- `RawBody`: застарілий псевдонім для `CommandBody` (збережений для сумісності).

Коли канал надає історію, використовується спільна обгортка:

- `[Chat messages since your last reply - for context]`
- `[Current message - respond to this]`

Для **непрямих чатів** (груп/каналів/кімнат) **тіло поточного повідомлення** має префікс із міткою відправника (той самий стиль, що й для записів історії). Це забезпечує узгодженість повідомлень у реальному часі та повідомлень із черги/історії в запиті агента.

Буфери історії є **лише для очікування**: вони включають групові повідомлення, які _не_ запускали виконання (наприклад, повідомлення, що вимагають згадки), і **не включають** повідомлення, які вже є в стенограмі сеансу.

Вилучення директив застосовується лише до розділу **поточного повідомлення**, щоб історія залишалася недоторканою. Канали, які обгортають історію, мають встановлювати `CommandBody` (або `RawBody`) в оригінальний текст повідомлення та зберігати `Body` як об’єднаний запит. Буфери історії налаштовуються через `messages.groupChat.historyLimit` (глобальне типове значення) і перевизначення для окремих каналів, такі як `channels.slack.historyLimit` або `channels.telegram.accounts.<id>.historyLimit` (встановіть `0`, щоб вимкнути).

## Постановка в чергу та подальші дії

Якщо виконання вже активне, вхідні повідомлення можна поставити в чергу, спрямувати до поточного виконання або зібрати для подальшого ходу.

- Налаштовується через `messages.queue` (і `messages.queue.byChannel`).
- Режими: `interrupt`, `steer`, `followup`, `collect`, а також варіанти з беклогом.

Докладніше: [Постановка в чергу](/uk/concepts/queue).

## Потокова передача, поділ на частини та пакетування

Блокова потокова передача надсилає часткові відповіді, коли модель створює текстові блоки.
Поділ на частини враховує текстові обмеження каналу та уникає розриву огороджених блоків коду.

Ключові налаштування:

- `agents.defaults.blockStreamingDefault` (`on|off`, типово off)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (пакетування на основі простою)
- `agents.defaults.humanDelay` (людиноподібна пауза між блоковими відповідями)
- Перевизначення каналів: `*.blockStreaming` і `*.blockStreamingCoalesce` (канали, відмінні від Telegram, вимагають явного `*.blockStreaming: true`)

Докладніше: [Потокова передача + поділ на частини](/uk/concepts/streaming).

## Видимість міркувань і токени

OpenClaw може показувати або приховувати міркування моделі:

- `/reasoning on|off|stream` керує видимістю.
- Вміст міркувань усе одно враховується у використанні токенів, коли його створює модель.
- Telegram підтримує потокову передачу міркувань у чернетку-бульбашку.

Докладніше: [Директиви thinking + reasoning](/uk/tools/thinking) і [Використання токенів](/uk/reference/token-use).

## Префікси, прив’язка до гілок і відповіді

Форматування вихідних повідомлень централізовано в `messages`:

- `messages.responsePrefix`, `channels.<channel>.responsePrefix` і `channels.<channel>.accounts.<id>.responsePrefix` (каскад префіксів вихідних повідомлень), а також `channels.whatsapp.messagePrefix` (вхідний префікс WhatsApp)
- Прив’язка відповідей до гілок через `replyToMode` і типові значення для окремих каналів

Докладніше: [Конфігурація](/uk/gateway/config-agents#messages) і документацію каналів.

## Тихі відповіді

Точний тихий токен `NO_REPLY` / `no_reply` означає «не доставляти видиму для користувача відповідь».
Коли хід також має очікувані медіа від інструмента, наприклад згенероване аудіо TTS, OpenClaw
видаляє тихий текст, але все одно доставляє медіавкладення.
OpenClaw визначає цю поведінку за типом розмови:

- Прямі розмови типово забороняють тишу та переписують просту тиху відповідь на короткий видимий запасний варіант.
- Групи/канали типово дозволяють тишу.
- Внутрішня оркестрація типово дозволяє тишу.

Типові значення розташовані в `agents.defaults.silentReply` і
`agents.defaults.silentReplyRewrite`; `surfaces.<id>.silentReply` і
`surfaces.<id>.silentReplyRewrite` можуть перевизначати їх для окремої поверхні.

Коли батьківський сеанс має один або більше очікуваних породжених виконань підагентів, прості
тихі відповіді відкидаються на всіх поверхнях замість переписування, щоб
батьківський процес залишався тихим, доки подія завершення дочірнього процесу не доставить справжню відповідь.
