OpenClaw обробляє вхідні повідомлення через конвеєр визначення сеансу, постановки в чергу, стримінгу, виконання інструментів і видимості міркувань. Ця сторінка показує шлях від вхідного повідомлення до відповіді.

## Потік повідомлень (загальний рівень)

```
Inbound message
  -> routing/bindings -> session key
  -> queue (if a run is active)
  -> agent run (streaming + tools)
  -> outbound replies (channel limits + chunking)
```

Ключові параметри містяться в конфігурації:

- `messages.*` для префіксів, чергування та поведінки груп.
- `agents.defaults.*` для стандартних налаштувань блокового стримінгу та поділу на фрагменти.
- Перевизначення каналів (`channels.whatsapp.*`, `channels.telegram.*` тощо) для обмежень і перемикачів стримінгу.

Повну схему див. у [Конфігурації](/uk/gateway/configuration).

## Дедуплікація вхідних повідомлень

Канали можуть повторно доставити те саме повідомлення після перепідключення. OpenClaw зберігає короткочасний кеш за ключем канал/акаунт/учасник/сеанс/id повідомлення, щоб дублікати доставлення не запускали ще один прогін агента.

## Debouncing вхідних повідомлень

Швидкі послідовні повідомлення від **того самого відправника** можна об’єднувати в один хід агента через `messages.inbound`. Debouncing обмежений парою канал + розмова й використовує найновіше повідомлення для тредингу відповідей/ID.

Конфігурація (глобальне значення за замовчуванням + перевизначення для каналів):

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

Примітки:

- Debounce застосовується до повідомлень **лише з текстом**; медіа/вкладення надсилаються негайно.
- Керівні команди обходять debouncing, щоб залишатися самостійними — **крім** випадків, коли канал явно вмикає об’єднання DM від того самого відправника (наприклад, [BlueBubbles `coalesceSameSenderDms`](/uk/channels/bluebubbles#coalescing-split-send-dms-command--url-in-one-composition)), де команди DM очікують у межах вікна debounce, щоб payload із розділеного надсилання міг приєднатися до того самого ходу агента.

## Сеанси та пристрої

Сеансами володіє Gateway, а не клієнти.

- Прямі чати згортаються в ключ основного сеансу агента.
- Групи/канали отримують власні ключі сеансів.
- Сховище сеансів і транскрипти розташовані на хості Gateway.

Кілька пристроїв/каналів можуть відповідати одному сеансу, але історія не повністю синхронізується назад до кожного клієнта. Рекомендація: використовуйте один основний пристрій для довгих розмов, щоб уникати розходження контексту. Control UI і TUI завжди показують транскрипт сеансу з Gateway, тому вони є джерелом істини.

Докладніше: [Керування сеансами](/uk/concepts/session).

## Метадані результатів інструментів

`content` результату інструмента — це видимий для моделі результат. `details` результату інструмента — це runtime-метадані для відображення в UI, діагностики, доставлення медіа та plugins.

OpenClaw явно підтримує цю межу:

- `toolResult.details` вилучається перед повторним відтворенням провайдером і вхідними даними Compaction.
- Збережені транскрипти сеансів утримують лише обмежені `details`; надмірно великі метадані замінюються компактним підсумком із позначкою `persistedDetailsTruncated: true`.
- Plugins та інструменти мають класти текст, який модель повинна прочитати, у `content`, а не лише в `details`.

## Тіла вхідних повідомлень і контекст історії

OpenClaw розділяє **тіло промпта** та **тіло команди**:

- `Body`: текст промпта, надісланий агенту. Він може містити оболонки каналу та необов’язкові обгортки історії.
- `CommandBody`: сирий текст користувача для розбору директив/команд.
- `RawBody`: застарілий псевдонім для `CommandBody` (збережено для сумісності).

Коли канал надає історію, він використовує спільну обгортку:

- `[Chat messages since your last reply - for context]`
- `[Current message - respond to this]`

Для **непрямих чатів** (груп/каналів/кімнат) **тіло поточного повідомлення** має префікс із міткою відправника (у тому самому стилі, що використовується для записів історії). Це робить повідомлення в реальному часі та повідомлення з черги/історії узгодженими в промпті агента.

Буфери історії є **лише pending**: вони містять групові повідомлення, які _не_ запускали прогін (наприклад, повідомлення, обмежені згадкою), і **виключають** повідомлення, які вже є в транскрипті сеансу.

Вилучення директив застосовується лише до розділу **поточного повідомлення**, тому історія залишається неушкодженою. Канали, які обгортають історію, мають задавати `CommandBody` (або `RawBody`) як оригінальний текст повідомлення та залишати `Body` як об’єднаний промпт. Буфери історії налаштовуються через `messages.groupChat.historyLimit` (глобальне значення за замовчуванням) і перевизначення для каналів, як-от `channels.slack.historyLimit` або `channels.telegram.accounts.<id>.historyLimit` (встановіть `0`, щоб вимкнути).

## Чергування та followup-повідомлення

Якщо прогін уже активний, вхідні повідомлення можна поставити в чергу, спрямувати в поточний прогін або зібрати для наступного ходу followup.

- Налаштовується через `messages.queue` (і `messages.queue.byChannel`).
- Режим за замовчуванням — `steer`, із 500 мс followup debounce, коли steering повертається до доставлення followup із черги.
- Режими: `steer`, `followup`, `collect`, `steer-backlog`, `interrupt` і застарілий псевдонім `queue`.

Докладніше: [Черга команд](/uk/concepts/queue).

## Володіння прогоном каналу

Channel plugins можуть зберігати порядок, застосовувати debounce до введення та транспортний backpressure перед тим, як повідомлення потрапить у чергу сеансу. Вони не повинні накладати окремий таймаут навколо самого ходу агента. Щойно повідомлення маршрутизовано до сеансу, довготривалу роботу контролюють життєві цикли сеансу, інструмента та runtime, щоб усі канали послідовно повідомляли про повільні ходи й відновлювалися після них.

## Стримінг, поділ на фрагменти та пакетування

Блоковий стримінг надсилає часткові відповіді, коли модель створює текстові блоки. Поділ на фрагменти враховує текстові обмеження каналу та не розбиває fenced code.

Ключові налаштування:

- `agents.defaults.blockStreamingDefault` (`on|off`, за замовчуванням off)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (пакетування на основі idle)
- `agents.defaults.humanDelay` (людиноподібна пауза між блоками відповідей)
- Перевизначення каналів: `*.blockStreaming` і `*.blockStreamingCoalesce` (канали, окрім Telegram, потребують явного `*.blockStreaming: true`)

Докладніше: [Стримінг + поділ на фрагменти](/uk/concepts/streaming).

## Видимість міркувань і токени

OpenClaw може показувати або приховувати міркування моделі:

- `/reasoning on|off|stream` керує видимістю.
- Вміст міркувань усе одно враховується у використанні токенів, коли його створює модель.
- Telegram підтримує стрим міркувань у draft bubble.

Докладніше: [Директиви thinking + reasoning](/uk/tools/thinking) і [Використання токенів](/uk/reference/token-use).

## Префікси, трединг і відповіді

Форматування вихідних повідомлень централізовано в `messages`:

- `messages.responsePrefix`, `channels.<channel>.responsePrefix` і `channels.<channel>.accounts.<id>.responsePrefix` (каскад вихідних префіксів), а також `channels.whatsapp.messagePrefix` (вхідний префікс WhatsApp)
- Трединг відповідей через `replyToMode` і стандартні значення для каналів

Докладніше: [Конфігурація](/uk/gateway/config-agents#messages) і документація каналів.

## Тихі відповіді

Точний тихий токен `NO_REPLY` / `no_reply` означає “не доставляти видиму для користувача відповідь”.
Коли хід також має pending-медіа інструмента, як-от згенероване TTS-аудіо, OpenClaw вилучає тихий текст, але все одно доставляє медіавкладення.
OpenClaw визначає цю поведінку за типом розмови:

- Прямі розмови за замовчуванням забороняють тишу й переписують голу тиху відповідь у короткий видимий fallback.
- Групи/канали за замовчуванням дозволяють тишу.
- Внутрішня оркестрація за замовчуванням дозволяє тишу.

OpenClaw також використовує тихі відповіді для внутрішніх помилок runner, що трапляються до будь-якої відповіді асистента в непрямих чатах, щоб групи/канали не бачили шаблонний текст помилки Gateway. Прямі чати за замовчуванням показують компактний текст помилки; сирі деталі runner показуються лише коли `/verbose` має значення `on` або `full`.

Стандартні значення містяться в `agents.defaults.silentReply` і `agents.defaults.silentReplyRewrite`; `surfaces.<id>.silentReply` і `surfaces.<id>.silentReplyRewrite` можуть перевизначати їх для кожної поверхні.

Коли батьківський сеанс має один або кілька pending запусків породжених subagent, голі тихі відповіді відкидаються на всіх поверхнях замість переписування, тож батьківський сеанс мовчить, доки подія завершення дочірнього процесу не доставить справжню відповідь.
