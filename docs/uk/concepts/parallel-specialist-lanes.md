Паралельні спеціалізовані напрямки дають змогу одному Gateway спрямовувати різні чати або кімнати до різних агентів, водночас зберігаючи швидку взаємодію для користувача. Розглядайте паралелізм як завдання проєктування з обмеженими ресурсами, а не просто як «більше агентів».

## Основні принципи

Спеціалізований напрямок підвищує пропускну здатність лише тоді, коли зменшує конкуренцію за реальні вузькі місця:

- **Блокування сеансів**: одночасно змінювати певний сеанс має лише один запуск.
- **Загальна пропускна здатність моделі**: усі видимі запуски чатів і надалі спільно використовують ліміти постачальника.
- **Пропускна здатність інструментів**: робота з оболонкою, браузером, мережею та репозиторієм може бути повільнішою за сам хід моделі.
- **Бюджет контексту**: довгі журнали розмов уповільнюють кожен наступний хід і знижують його зосередженість.
- **Неоднозначність відповідальності**: агенти, які дублюють ту саму роботу, марнують ресурси.

OpenClaw уже послідовно виконує запуски в межах кожного сеансу та обмежує загальний паралелізм за допомогою [черги команд](/uk/concepts/queue). Спеціалізовані напрямки додають поверх цього правила: який агент відповідає за певну роботу, що залишається в чаті, а що стає фоновою роботою.

## Рекомендований порядок упровадження

### Етап 1: контракти напрямків і ресурсомістка робота у фоновому режимі

Надайте кожному напрямку письмовий контракт у його робочому просторі та системному запиті:

- **Призначення**: робота, за яку відповідає цей напрямок.
- **Що не є метою**: робота, яку слід передати замість спроби виконати її самостійно.
- **Бюджет чату**: швидкі відповіді залишаються в чаті; для тривалих завдань надається коротке підтвердження, після чого вони виконуються фоновим підлеглим агентом або фоновим завданням.
- **Правило передавання**: коли за роботу відповідає інший напрямок, зазначте, куди її слід передати, і надайте стислий опис для передавання.
- **Правило ризиків інструментів**: віддавайте перевагу найменшому набору інструментів, достатньому для виконання роботи.

Це найдешевший етап, який усуває більшість заторів: одне завдання з програмування більше не перетворює дослідницький напрямок на вкрай повільний, а кожен чат зберігає власний контекст чистим.

### Етап 2: керування пріоритетами та паралельністю

Налаштуйте пропускну здатність черги та моделі відповідно до цінності кожного напрямку для бізнесу:

```json5
{
  agents: {
    defaults: {
      maxConcurrent: 4,
      subagents: { maxConcurrent: 8, delegationMode: "prefer" },
    },
  },
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
    },
  },
}
```

Використовуйте прямі й особисті чати та агентів виробничих операцій для високопріоритетної роботи. Коли система зайнята, переносіть дослідження, підготовку чернеток і пакетне програмування у фонові завдання.

### Етап 3: координатор / диспетчер трафіку

Коли активними стануть кілька напрямків, додайте просту модель координатора:

- Відстежуйте активні завдання напрямків і відповідальних за них.
- Виявляйте дублікати запитів у різних групах.
- Спрямовуйте стислі описи для передавання між напрямками.
- Показуйте лише блокувальні проблеми, завершені результати та рішення, які має ухвалити людина.

Не починайте з цього. Координатор без контрактів напрямків лише координує хаос.

## Мінімальний шаблон контракту напрямку

```md
# Контракт напрямку

## Відповідає за

- <робота, за яку відповідає цей напрямок>

## Не відповідає за

- <робота, яку слід передати>

## Бюджет чату

- Відповідайте на швидкі запитання безпосередньо.
- Для багатоетапної, повільної або насиченої використанням інструментів роботи: коротко підтвердьте запит, запустіть роботу як підлеглого агента або у фоновому режимі, а після завершення поверніть результат.

## Передавання

Якщо за запит відповідає інший напрямок, надайте відповідь із такими відомостями:

- цільовий напрямок
- мета
- відповідний контекст
- точна наступна дія

## Підхід до інструментів

Використовуйте найменший набір інструментів, достатній для виконання завдання. Уникайте широкого використання оболонки або мережі, якщо цей напрямок явно не відповідає за таку роботу.
```
