# Черга команд (2026-01-16)

Ми серіалізуємо вхідні запуски автоматичних відповідей (усі канали) через невелику вбудовану в процес чергу, щоб запобігти конфліктам між кількома запусками агента, водночас зберігаючи безпечний паралелізм між сесіями.

## Чому

- Запуски автоматичних відповідей можуть бути дорогими (виклики LLM) і можуть конфліктувати, коли кілька вхідних повідомлень надходять майже одночасно.
- Серіалізація запобігає конкуренції за спільні ресурси (файли сесій, журнали, stdin CLI) і зменшує ймовірність обмежень частоти з боку зовнішніх сервісів.

## Як це працює

- Черга FIFO з урахуванням lane обробляє кожен lane з налаштовуваним обмеженням паралелізму (типово 1 для неналаштованих lane; для main типово 4, для subagent — 8).
- `runEmbeddedPiAgent` ставить у чергу за **ключем сесії** (lane `session:<key>`), щоб гарантувати лише один активний запуск на сесію.
- Потім кожен запуск сесії ставиться в чергу до **глобального lane** (`main` типово), щоб загальний паралелізм обмежувався через `agents.defaults.maxConcurrent`.
- Коли ввімкнено докладне журналювання, запуски в черзі видають коротке повідомлення, якщо вони чекали понад ~2 с до старту.
- Індикатори набору тексту, як і раніше, спрацьовують одразу під час постановки в чергу (якщо це підтримує канал), тому взаємодія для користувача не змінюється, поки ми чекаємо своєї черги.

## Режими черги (для кожного каналу)

Вхідні повідомлення можуть спрямовувати поточний запуск, чекати наступного ходу або робити обидва варіанти:

- `steer`: негайно вставити в поточний запуск (скасовує очікувані виклики інструментів після наступної межі інструмента). Якщо потокова передача вимкнена, використовується запасний варіант followup.
- `followup`: поставити в чергу на наступний хід агента після завершення поточного запуску.
- `collect`: об’єднати всі повідомлення в черзі в **один** followup-хід (типово). Якщо повідомлення націлені на різні канали/потоки, вони обробляються окремо, щоб зберегти маршрутизацію.
- `steer-backlog` (також `steer+backlog`): спрямувати зараз **і** зберегти повідомлення для followup-ходу.
- `interrupt` (застарілий): перервати активний запуск для цієї сесії, а потім запустити найновіше повідомлення.
- `queue` (застарілий псевдонім): те саме, що `steer`.

Steer-backlog означає, що ви можете отримати followup-відповідь після спрямованого запуску, тож
поверхні з потоковою передачею можуть виглядати як дублікати. Надавайте перевагу `collect`/`steer`, якщо хочете
одну відповідь на кожне вхідне повідомлення.
Надішліть `/queue collect` як окрему команду (для сесії) або встановіть `messages.queue.byChannel.discord: "collect"`.

Типові значення (якщо не встановлено в конфігурації):

- Усі поверхні → `collect`

Налаштовуйте глобально або для кожного каналу через `messages.queue`:

```json5
{
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## Параметри черги

Параметри застосовуються до `followup`, `collect` і `steer-backlog` (а також до `steer`, коли він використовує запасний варіант followup):

- `debounceMs`: чекати тиші перед запуском followup-ходу (запобігає “продовжуй, продовжуй”).
- `cap`: максимальна кількість повідомлень у черзі на сесію.
- `drop`: політика переповнення (`old`, `new`, `summarize`).

Summarize зберігає короткий список із маркерами для відкинутих повідомлень і вставляє його як синтетичний followup-запит.
Типові значення: `debounceMs: 1000`, `cap: 20`, `drop: summarize`.

## Перевизначення для окремої сесії

- Надішліть `/queue <mode>` як окрему команду, щоб зберегти режим для поточної сесії.
- Параметри можна комбінувати: `/queue collect debounce:2s cap:25 drop:summarize`
- `/queue default` або `/queue reset` очищає перевизначення сесії.

## Область дії та гарантії

- Застосовується до запусків агента з автоматичними відповідями в усіх вхідних каналах, які використовують конвеєр відповідей gateway (WhatsApp web, Telegram, Slack, Discord, Signal, iMessage, webchat тощо).
- Типовий lane (`main`) діє на рівні всього процесу для вхідних повідомлень + основних Heartbeat; установіть `agents.defaults.maxConcurrent`, щоб дозволити паралельну роботу кількох сесій.
- Можуть існувати додаткові lane (наприклад, `cron`, `subagent`), щоб фонові завдання могли виконуватися паралельно, не блокуючи вхідні відповіді. Ці відокремлені запуски відстежуються як [фонові завдання](/uk/automation/tasks).
- Lane на рівні сесії гарантують, що лише один запуск агента торкається конкретної сесії в один момент часу.
- Без зовнішніх залежностей чи фонових worker threads; лише TypeScript + promises.

## Усунення неполадок

- Якщо команди здаються завислими, увімкніть докладні журнали й шукайте рядки “queued for …ms”, щоб підтвердити, що черга обробляється.
- Якщо вам потрібна глибина черги, увімкніть докладні журнали й стежте за рядками з таймінгами черги.
