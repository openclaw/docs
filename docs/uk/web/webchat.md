Стан: macOS/iOS SwiftUI-інтерфейс чату напряму взаємодіє з Gateway WebSocket.

## Що це таке

- Нативний інтерфейс чату для Gateway (без вбудованого браузера і без локального статичного сервера).
- Використовує ті самі сеанси та правила маршрутизації, що й інші канали.
- Детермінована маршрутизація: відповіді завжди повертаються до WebChat.

## Швидкий старт

1. Запустіть Gateway.
2. Відкрийте інтерфейс WebChat (застосунок macOS/iOS) або вкладку чату Інтерфейсу керування.
3. Переконайтеся, що налаштовано дійсний шлях автентифікації Gateway (за замовчуванням shared-secret,
   навіть на loopback).

## Як це працює (поведінка)

- Інтерфейс підключається до Gateway WebSocket і використовує `chat.history`, `chat.send`, `chat.inject` та `chat.transcribeAudio`.
- `chat.history` обмежено для стабільності: Gateway може обрізати довгі текстові поля, пропускати важкі метадані та замінювати надмірно великі записи на `[chat.history omitted: message too large]`.
- `chat.history` використовує активну гілку транскрипту для сучасних append-only файлів сеансів, тому покинуті гілки перезапису та замінені копії промптів не відображаються у WebChat.
- Інтерфейс керування запам’ятовує базовий Gateway `sessionId`, повернений `chat.history`, і додає його до наступних викликів `chat.send`, тому повторні підключення та оновлення сторінки продовжують ту саму збережену розмову, якщо користувач не почне або не скине сеанс.
- Інтерфейс керування об’єднує дублікати надісланих запитів, що ще виконуються, для того самого сеансу, повідомлення та вкладень до створення нового ідентифікатора запуску `chat.send`; Gateway все одно дедуплікує повторні запити, які повторно використовують той самий ключ ідемпотентності.
- `chat.history` також нормалізується для відображення: runtime-only контекст OpenClaw,
  вхідні обгортки envelope, вбудовані теги директив доставки,
  як-от `[[reply_to_*]]` і `[[audio_as_voice]]`, plain-text XML-корисні навантаження викликів інструментів
  (зокрема `<tool_call>...</tool_call>`,
  `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`,
  `<function_calls>...</function_calls>` та обрізані блоки викликів інструментів), а також
  просочені ASCII/full-width керівні токени моделі вилучаються з видимого тексту,
  а записи асистента, увесь видимий текст яких складається лише з точного тихого
  токена `NO_REPLY` / `no_reply`, пропускаються.
- Корисні навантаження відповідей, позначені прапорцем reasoning (`isReasoning: true`), виключаються з вмісту асистента у WebChat, тексту повторного відтворення транскрипту та аудіоблоків вмісту, тому payload-и лише для міркування не з’являються як видимі повідомлення асистента або відтворюване аудіо.
- `chat.transcribeAudio` забезпечує серверне диктування в композері чату Інтерфейсу керування. Браузер записує аудіо з мікрофона, надсилає його як base64 до Gateway, а Gateway запускає налаштований конвеєр `tools.media.audio`. Повернений транскрипт вставляється в чернетку; запуск агента не починається, доки користувач не надішле її.
- `chat.inject` додає нотатку асистента безпосередньо до транскрипту та транслює її в інтерфейс (без запуску агента).
- Перервані запуски можуть залишати частковий вивід асистента видимим в інтерфейсі.
- Gateway зберігає перерваний частковий текст асистента в історію транскрипту, коли існує буферизований вивід, і позначає ці записи метаданими переривання.
- Історію завжди отримують із Gateway (без локального спостереження за файлами).
- Якщо Gateway недоступний, WebChat працює лише для читання.

## Панель інструментів агентів Інтерфейсу керування

- Панель Tools Інтерфейсу керування `/agents` має два окремі подання:
  - **Доступні прямо зараз** використовує `tools.effective(sessionKey=...)` і показує, що поточний
    сеанс справді може використовувати під час виконання, включно з інструментами ядра, Plugin та канальними інструментами.
  - **Конфігурація інструментів** використовує `tools.catalog` і зосереджується на профілях, перевизначеннях та
    семантиці каталогу.
- Доступність під час виконання обмежена сеансом. Перемикання сеансів для того самого агента може змінити
  список **Доступні прямо зараз**.
- Редактор конфігурації не означає доступність під час виконання; ефективний доступ і далі дотримується пріоритету політик
  (`allow`/`deny`, перевизначень для окремого агента та provider/channel).

## Віддалене використання

- Віддалений режим тунелює Gateway WebSocket через SSH/Tailscale.
- Не потрібно запускати окремий сервер WebChat.

## Довідник конфігурації (WebChat)

Повна конфігурація: [Конфігурація](/uk/gateway/configuration)

Параметри WebChat:

- `gateway.webchat.chatHistoryMaxChars`: максимальна кількість символів для текстових полів у відповідях `chat.history`. Коли запис транскрипту перевищує цей ліміт, Gateway обрізає довгі текстові поля та може замінювати надмірно великі повідомлення заповнювачем. Клієнт також може надіслати `maxChars` для окремого запиту, щоб перевизначити це значення за замовчуванням для одного виклику `chat.history`.

Пов’язані глобальні параметри:

- `gateway.port`, `gateway.bind`: хост/порт WebSocket.
- `gateway.auth.mode`, `gateway.auth.token`, `gateway.auth.password`:
  автентифікація WebSocket через shared-secret.
- `gateway.auth.allowTailscale`: вкладка чату браузерного Інтерфейсу керування може використовувати
  заголовки ідентичності Tailscale Serve, коли це ввімкнено.
- `gateway.auth.mode: "trusted-proxy"`: автентифікація через reverse-proxy для браузерних клієнтів за identity-aware **non-loopback** джерелом proxy (див. [Автентифікація через довірений Proxy](/uk/gateway/trusted-proxy-auth)).
- `gateway.remote.url`, `gateway.remote.token`, `gateway.remote.password`: ціль віддаленого Gateway.
- `session.*`: сховище сеансів і типові значення основного ключа.
