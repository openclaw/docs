# Віддалений OpenClaw (macOS ⇄ віддалений хост)

Цей сценарій дає змогу застосунку macOS працювати як повноцінний пульт віддаленого керування для OpenClaw gateway, що працює на іншому хості (настільному комп’ютері/сервері). Це функція застосунку **Віддалено через SSH** (віддалений запуск). Усі функції — перевірки стану, пересилання голосового пробудження та веб-чат — повторно використовують ту саму віддалену SSH-конфігурацію з _Налаштування → Загальні_.

## Режими

- **Локально (цей Mac)**: усе працює на ноутбуці. SSH не використовується.
- **Віддалено через SSH (типово)**: команди OpenClaw виконуються на віддаленому хості. Застосунок Mac відкриває SSH-з’єднання з `-o BatchMode`, вибраним вами ідентифікатором/ключем і локальним перенаправленням порту.
- **Віддалено напряму (ws/wss)**: без SSH-тунелю. Застосунок Mac підключається безпосередньо до URL gateway (наприклад, через Tailscale Serve або публічний зворотний HTTPS-проксі).

## Віддалені транспорти

Віддалений режим підтримує два транспорти:

- **SSH-тунель** (типово): використовує `ssh -N -L ...`, щоб перенаправити порт gateway на localhost. Gateway бачитиме IP вузла як `127.0.0.1`, бо тунель є loopback.
- **Пряме підключення (ws/wss)**: підключається прямо до URL gateway. Gateway бачить реальний IP клієнта.

У режимі SSH-тунелю виявлені імена хостів LAN/tailnet зберігаються як
`gateway.remote.sshTarget`. Застосунок залишає `gateway.remote.url` на локальній
кінцевій точці тунелю, наприклад `ws://127.0.0.1:18789`, щоб CLI, веб-чат і
локальна служба хоста вузла використовували той самий безпечний loopback-транспорт.

Автоматизацією браузера у віддаленому режимі керує хост вузла CLI, а не
рідний вузол застосунку macOS. Застосунок запускає встановлену службу хоста вузла,
коли це можливо; якщо вам потрібне керування браузером із цього Mac, установіть/запустіть її за допомогою
`openclaw node install ...` і `openclaw node start` (або запустіть
`openclaw node run ...` у передньому плані), а потім спрямуйте роботу на цей
вузол із підтримкою браузера.

## Передумови на віддаленому хості

1. Установіть Node + pnpm і зберіть/установіть OpenClaw CLI (`pnpm install && pnpm build && pnpm link --global`).
2. Переконайтеся, що `openclaw` є в PATH для неінтерактивних оболонок (за потреби створіть symlink у `/usr/local/bin` або `/opt/homebrew/bin`).
3. Відкрийте SSH з автентифікацією за ключем. Ми рекомендуємо IP-адреси **Tailscale** для стабільної доступності поза LAN.

## Налаштування застосунку macOS

1. Відкрийте _Налаштування → Загальні_.
2. У розділі **OpenClaw runs** виберіть **Віддалено через SSH** і задайте:
   - **Транспорт**: **SSH-тунель** або **Пряме підключення (ws/wss)**.
   - **SSH-ціль**: `user@host` (необов’язково `:port`).
     - Якщо gateway у тій самій LAN і рекламує Bonjour, виберіть його зі списку виявлених, щоб автоматично заповнити це поле.
   - **URL Gateway** (лише для прямого підключення): `wss://gateway.example.ts.net` (або `ws://...` для локального/LAN).
   - **Файл ідентичності** (розширено): шлях до вашого ключа.
   - **Корінь проєкту** (розширено): шлях віддаленого checkout, що використовується для команд.
   - **Шлях CLI** (розширено): необов’язковий шлях до виконуваної точки входу/бінарного файлу `openclaw` (автоматично заповнюється, коли рекламується).
3. Натисніть **Перевірити віддалене підключення**. Успіх означає, що віддалений `openclaw status --json` виконується правильно. Збої зазвичай означають проблеми з PATH/CLI; код виходу 127 означає, що CLI не знайдено віддалено.
4. Перевірки стану й веб-чат тепер автоматично працюватимуть через цей SSH-тунель.

## Веб-чат

- **SSH-тунель**: веб-чат підключається до gateway через перенаправлений керівний порт WebSocket (типово 18789).
- **Пряме підключення (ws/wss)**: веб-чат підключається прямо до налаштованого URL gateway.
- Окремого HTTP-сервера WebChat більше немає.

## Дозволи

- Віддаленому хосту потрібні ті самі дозволи TCC, що й локальному (Automation, Accessibility, Screen Recording, Microphone, Speech Recognition, Notifications). Запустіть onboarding на цій машині, щоб надати їх один раз.
- Вузли рекламують свій стан дозволів через `node.list` / `node.describe`, щоб агенти знали, що доступно.

## Примітки щодо безпеки

- Надавайте перевагу loopback-прив’язкам на віддаленому хості та підключайтеся через SSH або Tailscale.
- SSH-тунелювання використовує сувору перевірку ключа хоста; спочатку довірте ключ хоста, щоб він існував у `~/.ssh/known_hosts`.
- Якщо ви прив’язуєте Gateway до інтерфейсу, що не є loopback, вимагайте дійсну автентифікацію Gateway: токен, пароль або зворотний проксі з урахуванням ідентичності з `gateway.auth.mode: "trusted-proxy"`.
- Див. [Безпека](/uk/gateway/security) і [Tailscale](/uk/gateway/tailscale).

## Сценарій входу WhatsApp (віддалено)

- Запустіть `openclaw channels login --verbose` **на віддаленому хості**. Проскануйте QR-код у WhatsApp на телефоні.
- Повторно запустіть вхід на цьому хості, якщо термін автентифікації мине. Перевірка стану покаже проблеми з прив’язкою.

## Усунення несправностей

- **код виходу 127 / не знайдено**: `openclaw` не в PATH для оболонок без входу. Додайте його до `/etc/paths`, rc-файлу вашої оболонки або створіть symlink у `/usr/local/bin`/`/opt/homebrew/bin`.
- **Проба стану не вдалася**: перевірте доступність SSH, PATH і те, що в Baileys виконано вхід (`openclaw status --json`).
- **Веб-чат завис**: підтвердьте, що gateway працює на віддаленому хості, а перенаправлений порт збігається з WS-портом gateway; UI потребує справного WS-з’єднання.
- **IP вузла показує 127.0.0.1**: очікувано для SSH-тунелю. Змініть **Транспорт** на **Пряме підключення (ws/wss)**, якщо хочете, щоб gateway бачив реальний IP клієнта.
- **Dashboard працює, але можливості Mac офлайн**: це означає, що операторське/керівне з’єднання застосунку справне, але з’єднання супутнього вузла не підключене або не має поверхні команд. Відкрийте розділ пристрою в рядку меню й перевірте, чи Mac має стан `paired · disconnected`. Для кінцевих точок Tailscale Serve `wss://*.ts.net` застосунок виявляє застарілі legacy TLS leaf pins після ротації сертифіката, очищає застарілий pin, коли macOS довіряє новому сертифікату, і автоматично повторює спробу. Якщо сертифікат не є довіреним у системі або хост не є іменем Tailscale Serve, перегляньте сертифікат або перейдіть на **Віддалено через SSH**.
- **Голосове пробудження**: тригерні фрази автоматично пересилаються у віддаленому режимі; окремий пересилач не потрібен.

## Звуки сповіщень

Вибирайте звуки для кожного сповіщення зі скриптів за допомогою `openclaw` і `node.invoke`, наприклад:

```bash
openclaw nodes notify --node <id> --title "Ping" --body "Remote gateway ready" --sound Glass
```

Глобального перемикача «типовий звук» у застосунку більше немає; виклики вибирають звук (або жодного) для кожного запиту.
