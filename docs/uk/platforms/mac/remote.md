Цей потік дає змогу застосунку macOS діяти як повний пульт дистанційного керування для OpenClaw Gateway, що працює на іншому хості (настільному комп’ютері/сервері). Застосунок може напряму підключатися до довірених URL-адрес Gateway у LAN/Tailnet або керувати SSH-тунелем, коли віддалений Gateway доступний лише через loopback. Перевірки стану, пересилання голосового пробудження та вебчат повторно використовують ту саму віддалену конфігурацію з _Налаштування → Загальні_.

## Режими

- **Локально (цей Mac)**: Усе працює на ноутбуці. SSH не використовується.
- **Віддалено через SSH (типово)**: Команди OpenClaw виконуються на віддаленому хості. Застосунок Mac відкриває SSH-з’єднання з `-o BatchMode`, вибраною ідентичністю/ключем і локальним перенаправленням порту.
- **Віддалено напряму (ws/wss)**: Без SSH-тунелю. Застосунок Mac напряму підключається до URL-адреси Gateway (наприклад, через LAN, Tailscale, Tailscale Serve або публічний зворотний HTTPS-проксі).

## Віддалені транспорти

Віддалений режим підтримує два транспорти:

- **SSH-тунель** (типово): Використовує `ssh -N -L ...`, щоб перенаправити порт Gateway на localhost. Gateway бачитиме IP-адресу вузла як `127.0.0.1`, оскільки тунель є loopback.
- **Напряму (ws/wss)**: Підключається безпосередньо до URL-адреси Gateway. Gateway бачить реальну IP-адресу клієнта.

У режимі SSH-тунелю виявлені імена хостів LAN/tailnet зберігаються як
`gateway.remote.sshTarget`. Застосунок тримає `gateway.remote.url` на локальній
кінцевій точці тунелю, наприклад `ws://127.0.0.1:18789`, щоб CLI, вебчат і
локальна служба хоста вузла використовували той самий безпечний loopback-транспорт.
Коли виявлення повертає і сирі IP-адреси Tailnet, і стабільні імена хостів,
застосунок надає перевагу Tailscale MagicDNS або іменам LAN, щоб віддалені
підключення краще переживали зміни адрес.
Якщо локальний порт тунелю відрізняється від віддаленого порту Gateway, задайте
`gateway.remote.remotePort` як порт на віддаленому хості.

Автоматизацією браузера у віддаленому режимі володіє хост вузла CLI, а не
нативний вузол застосунку macOS. Застосунок запускає встановлену службу хоста вузла,
коли це можливо; якщо вам потрібне керування браузером із цього Mac, установіть/запустіть її за допомогою
`openclaw node install ...` і `openclaw node start` (або запустіть
`openclaw node run ...` на передньому плані), а потім націльтеся на цей
вузол із підтримкою браузера.

## Передумови на віддаленому хості

1. Установіть Node + pnpm і зберіть/установіть OpenClaw CLI (`pnpm install && pnpm build && pnpm link --global`).
2. Переконайтеся, що `openclaw` є в PATH для неінтерактивних оболонок (за потреби створіть символічне посилання в `/usr/local/bin` або `/opt/homebrew/bin`).
3. Лише для SSH-транспорту: відкрийте SSH з автентифікацією за ключем. Ми рекомендуємо IP-адреси **Tailscale** для стабільної доступності поза LAN.

## Налаштування застосунку macOS

Щоб попередньо налаштувати застосунок без вітального потоку:

```bash
openclaw-mac configure-remote \
  --ssh-target user@gateway.local \
  --local-port 18789 \
  --remote-port 18789 \
  --token "$OPENCLAW_GATEWAY_TOKEN"
```

Для Gateway, який уже доступний у довіреній LAN або Tailnet, повністю пропустіть SSH:

```bash
openclaw-mac configure-remote \
  --direct-url ws://192.168.0.202:18789 \
  --token "$OPENCLAW_GATEWAY_TOKEN"
```

Це записує віддалену конфігурацію, позначає онбординг завершеним і дає застосунку змогу володіти
вибраним транспортом під час запуску.

1. Відкрийте _Налаштування → Загальні_.
2. У розділі **OpenClaw працює** виберіть **Віддалено** і задайте:
   - **Транспорт**: **SSH-тунель** або **Напряму (ws/wss)**.
   - **SSH-ціль**: `user@host` (необов’язково `:port`).
     - Якщо Gateway у тій самій LAN і рекламує Bonjour, виберіть його зі списку виявлених, щоб автоматично заповнити це поле.
   - **URL-адреса Gateway** (лише напряму): `wss://gateway.example.ts.net` (або `ws://...` для локального/LAN).
   - **Файл ідентичності** (додатково): шлях до вашого ключа.
   - **Корінь проєкту** (додатково): шлях до віддаленої копії репозиторію, що використовується для команд.
   - **Шлях CLI** (додатково): необов’язковий шлях до виконуваної точки входу/бінарного файлу `openclaw` (автозаповнюється, коли рекламується).
3. Натисніть **Тестувати віддалено**. Успіх означає, що віддалена команда `openclaw status --json` виконується коректно. Помилки зазвичай означають проблеми PATH/CLI; код виходу 127 означає, що CLI не знайдено віддалено.
4. Перевірки стану та вебчат тепер автоматично працюватимуть через вибраний транспорт.

## Вебчат

- **SSH-тунель**: вебчат підключається до Gateway через перенаправлений керівний порт WebSocket (типово 18789).
- **Напряму (ws/wss)**: вебчат підключається безпосередньо до налаштованої URL-адреси Gateway.
- Окремого HTTP-сервера WebChat більше немає.

## Дозволи

- Віддаленому хосту потрібні ті самі схвалення TCC, що й локальному (Автоматизація, Доступність, Запис екрана, Мікрофон, Розпізнавання мовлення, Сповіщення). Запустіть онбординг на цій машині, щоб надати їх один раз.
- Вузли рекламують свій стан дозволів через `node.list` / `node.describe`, щоб агенти знали, що доступно.

## Примітки з безпеки

- Віддавайте перевагу loopback-прив’язкам на віддаленому хості й підключайтеся через SSH, Tailscale Serve або довірену пряму URL-адресу Tailnet/LAN.
- SSH-тунелювання використовує сувору перевірку ключа хоста; спочатку довірте ключу хоста, щоб він існував у `~/.ssh/known_hosts`.
- Якщо ви прив’язуєте Gateway до інтерфейсу не loopback, вимагайте чинну автентифікацію Gateway: токен, пароль або зворотний проксі з урахуванням ідентичності з `gateway.auth.mode: "trusted-proxy"`.
- Див. [Безпека](/uk/gateway/security) і [Tailscale](/uk/gateway/tailscale).

## Потік входу WhatsApp (віддалено)

- Запустіть `openclaw channels login --verbose` **на віддаленому хості**. Відскануйте QR-код за допомогою WhatsApp на телефоні.
- Повторно запустіть вхід на цьому хості, якщо автентифікація спливе. Перевірка стану покаже проблеми зі зв’язком.

## Усунення несправностей

- **код виходу 127 / не знайдено**: `openclaw` не в PATH для оболонок без входу. Додайте його до `/etc/paths`, rc-файлу вашої оболонки або створіть символічне посилання в `/usr/local/bin`/`/opt/homebrew/bin`.
- **Перевірка стану не вдалася**: перевірте доступність SSH, PATH і те, що Baileys увійшов у систему (`openclaw status --json`).
- **Вебчат завис**: підтвердьте, що Gateway працює на віддаленому хості, а перенаправлений порт відповідає WS-порту Gateway; UI потребує справного WS-з’єднання.
- **IP вузла показує 127.0.0.1**: очікувано з SSH-тунелем. Перемкніть **Транспорт** на **Напряму (ws/wss)**, якщо хочете, щоб Gateway бачив реальну IP-адресу клієнта.
- **Панель працює, але можливості Mac офлайн**: це означає, що операторське/керівне з’єднання застосунку справне, але супутнє з’єднання вузла не підключене або не має своєї командної поверхні. Відкрийте розділ пристрою в рядку меню й перевірте, чи Mac має стан `paired · disconnected`. Для кінцевих точок Tailscale Serve `wss://*.ts.net` застосунок виявляє застарілі legacy TLS leaf pins після ротації сертифіката, очищає застарілий pin, коли macOS довіряє новому сертифікату, і автоматично повторює спробу. Якщо сертифікат не довірений системою або хост не є іменем Tailscale Serve, задайте `gateway.remote.tlsFingerprint` як очікуваний відбиток сертифіката, перевірте сертифікат або перемкніться на **Віддалено через SSH**.
- **Голосове пробудження**: тригерні фрази автоматично пересилаються у віддаленому режимі; окремий пересилач не потрібен.

## Звуки сповіщень

Вибирайте звуки для кожного сповіщення зі скриптів за допомогою `openclaw` і `node.invoke`, наприклад:

```bash
openclaw nodes notify --node <id> --title "Ping" --body "Remote gateway ready" --sound Glass
```

У застосунку більше немає глобального перемикача "звук за замовчуванням"; викликачі вибирають звук (або відсутність звуку) для кожного запиту.
