Доступність: збірки застосунку для iPhone поширюються через канали Apple, коли це увімкнено для випуску. Локальні розробницькі збірки також можна запускати з вихідного коду.

## Що він робить

- Підключається до Gateway через WebSocket (LAN або tailnet).
- Надає можливості вузла: Canvas, знімок екрана, зйомка камерою, геолокація, режим Talk, голосове пробудження.
- Отримує команди `node.invoke` і повідомляє події стану вузла.

## Вимоги

- Gateway, запущений на іншому пристрої (macOS, Linux або Windows через WSL2).
- Мережевий шлях:
  - Та сама LAN через Bonjour, **або**
  - Tailnet через одноадресний DNS-SD (приклад домену: `openclaw.internal.`), **або**
  - Ручний host/port (резервний варіант).

## Швидкий старт (спарювання + підключення)

1. Запустіть Gateway:

```bash
openclaw gateway --port 18789
```

2. У застосунку iOS відкрийте Settings і виберіть знайдений gateway (або увімкніть Manual Host і введіть host/port).

3. Схваліть запит на спарювання на хості gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

Якщо застосунок повторює спарювання зі зміненими даними автентифікації (роль/scopes/публічний ключ),
попередній запит в очікуванні замінюється, і створюється новий `requestId`.
Перед схваленням знову запустіть `openclaw devices list`.

Необов’язково: якщо вузол iOS завжди підключається з жорстко контрольованої підмережі, ви
можете явно ввімкнути автоматичне схвалення вузла під час першого підключення за допомогою явних CIDR або точних IP:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

За замовчуванням це вимкнено. Це застосовується лише до нового спарювання `role: node`
без запитаних scopes. Спарювання оператора/браузера та будь-яка зміна ролі, scope, метаданих або
публічного ключа все одно потребує ручного схвалення.

4. Перевірте підключення:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## Push через ретранслятор для офіційних збірок

Офіційні поширювані збірки iOS використовують зовнішній push-ретранслятор замість публікації необробленого APNs
токена в gateway.

Офіційні збірки App Store з публічної гілки випуску використовують розміщений ретранслятор за адресою `https://ios-push-relay.openclaw.ai`.

Користувацькі розгортання ретранслятора потребують навмисно окремого шляху збірки/розгортання iOS, URL ретранслятора якого збігається з URL ретранслятора gateway. Публічна гілка випуску App Store не приймає перевизначення користувацького URL ретранслятора. Якщо ви використовуєте користувацьку збірку з ретранслятором, задайте відповідний URL ретранслятора gateway:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

Як працює потік:

- Застосунок iOS реєструється в ретрансляторі за допомогою App Attest і JWS транзакції застосунку StoreKit.
- Ретранслятор повертає непрозорий дескриптор ретранслятора плюс дозвіл на надсилання, обмежений цією реєстрацією.
- Застосунок iOS отримує ідентичність спареного gateway і включає її в реєстрацію ретранслятора, тож реєстрація через ретранслятор делегується цьому конкретному gateway.
- Застосунок пересилає цю реєстрацію через ретранслятор до спареного gateway за допомогою `push.apns.register`.
- Gateway використовує збережений дескриптор ретранслятора для `push.test`, фонових пробуджень і підштовхувань до пробудження.
- Користувацькі URL ретранслятора gateway мають збігатися з URL ретранслятора, вбудованим у збірку iOS.
- Якщо застосунок пізніше підключається до іншого gateway або до збірки з іншим базовим URL ретранслятора, він оновлює реєстрацію ретранслятора замість повторного використання старої прив’язки.

Що **не** потрібно gateway для цього шляху:

- Жодного токена ретранслятора на рівні всього розгортання.
- Жодного прямого ключа APNs для офіційних надсилань через ретранслятор з App Store.

Очікуваний потік оператора:

1. Установіть офіційний застосунок iOS.
2. Необов’язково: задайте `gateway.push.apns.relay.baseUrl` на gateway лише коли використовуєте навмисно окрему користувацьку збірку з ретранслятором.
3. Спаруйте застосунок із gateway і дайте йому завершити підключення.
4. Застосунок автоматично публікує `push.apns.register` після того, як має APNs token, сесію оператора підключено, а реєстрація ретранслятора успішна.
5. Після цього `push.test`, пробудження для повторного підключення та підштовхування до пробудження можуть використовувати збережену реєстрацію через ретранслятор.

## Фонові сигнали alive

Коли iOS пробуджує застосунок для silent push, фонового оновлення або події значної зміни геолокації, застосунок
намагається коротко перепідключити вузол, а потім викликає `node.event` з `event: "node.presence.alive"`.
Gateway записує це як `lastSeenAtMs`/`lastSeenReason` у метаданих спареного вузла/пристрою лише
після того, як відома автентифікована ідентичність пристрою вузла.

Застосунок вважає фонове пробудження успішно записаним лише тоді, коли відповідь gateway містить
`handled: true`. Старіші gateway можуть підтверджувати `node.event` через `{ "ok": true }`; така відповідь
сумісна, але не рахується як надійне оновлення last-seen.

Примітка щодо сумісності:

- `OPENCLAW_APNS_RELAY_BASE_URL` досі працює як тимчасове env-перевизначення для gateway.
- Публічна гілка випуску App Store відхиляє `OPENCLAW_PUSH_RELAY_BASE_URL` для збірок iOS.

## Потік автентифікації та довіри

Ретранслятор існує, щоб забезпечити два обмеження, яких прямий APNs-на-gateway не може надати для
офіційних збірок iOS:

- Лише справжні збірки OpenClaw iOS, поширені через Apple, можуть використовувати розміщений ретранслятор.
- Gateway може надсилати push через ретранслятор лише для пристроїв iOS, спарених із цим конкретним
  gateway.

Крок за кроком:

1. `iOS app -> gateway`
   - Застосунок спочатку спарюється з gateway через звичайний потік автентифікації Gateway.
   - Це дає застосунку автентифіковану сесію вузла плюс автентифіковану сесію оператора.
   - Сесія оператора використовується для виклику `gateway.identity.get`.

2. `iOS app -> relay`
   - Застосунок викликає кінцеві точки реєстрації ретранслятора через HTTPS.
   - Реєстрація включає доказ App Attest плюс JWS транзакції застосунку StoreKit.
   - Ретранслятор перевіряє bundle ID, доказ App Attest і доказ поширення Apple, а також вимагає
     офіційний/production шлях поширення.
   - Саме це блокує локальні Xcode/dev збірки від використання розміщеного ретранслятора. Локальна збірка може бути
     підписана, але вона не задовольняє офіційний доказ поширення Apple, якого очікує ретранслятор.

3. `gateway identity delegation`
   - Перед реєстрацією в ретрансляторі застосунок отримує ідентичність спареного gateway з
     `gateway.identity.get`.
   - Застосунок включає цю ідентичність gateway у payload реєстрації ретранслятора.
   - Ретранслятор повертає дескриптор ретранслятора і дозвіл на надсилання, обмежений реєстрацією, які делеговані
     цій ідентичності gateway.

4. `gateway -> relay`
   - Gateway зберігає дескриптор ретранслятора і дозвіл на надсилання з `push.apns.register`.
   - Під час `push.test`, пробуджень для повторного підключення і підштовхувань до пробудження gateway підписує запит на надсилання своєю
     власною ідентичністю пристрою.
   - Ретранслятор перевіряє і збережений дозвіл на надсилання, і підпис gateway відносно делегованої
     ідентичності gateway з реєстрації.
   - Інший gateway не може повторно використати цю збережену реєстрацію, навіть якщо якимось чином отримає дескриптор.

5. `relay -> APNs`
   - Ретранслятор володіє production обліковими даними APNs і необробленим APNs token для офіційної збірки.
   - Gateway ніколи не зберігає необроблений APNs token для офіційних збірок через ретранслятор.
   - Ретранслятор надсилає фінальний push до APNs від імені спареного gateway.

Чому було створено цей дизайн:

- Щоб не зберігати production облікові дані APNs на gateway користувачів.
- Щоб уникнути зберігання необроблених APNs token офіційних збірок на gateway.
- Щоб дозволити використання розміщеного ретранслятора лише для офіційних збірок OpenClaw iOS.
- Щоб запобігти надсиланню wake push одним gateway на пристрої iOS, що належать іншому gateway.

Локальні/ручні збірки залишаються на прямому APNs. Якщо ви тестуєте такі збірки без ретранслятора,
gateway усе одно потребує прямих облікових даних APNs:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

Це runtime env vars хоста gateway, а не налаштування Fastlane. `apps/ios/fastlane/.env` зберігає лише
автентифікацію App Store Connect, як-от `APP_STORE_CONNECT_KEY_ID` і
`APP_STORE_CONNECT_ISSUER_ID`; він не налаштовує пряму доставку APNs для локальних збірок iOS.

Рекомендоване зберігання на хості gateway:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

Не комітьте файл `.p8` і не розміщуйте його в checkout репозиторію.

## Шляхи виявлення

### Bonjour (LAN)

Застосунок iOS переглядає `_openclaw-gw._tcp` на `local.` і, коли налаштовано, той самий
wide-area домен виявлення DNS-SD. Gateway в тій самій LAN з’являються автоматично з `local.`;
міжмережеве виявлення може використовувати налаштований wide-area домен без зміни типу beacon.

### Tailnet (міжмережево)

Якщо mDNS заблоковано, використовуйте одноадресну зону DNS-SD (виберіть домен; приклад:
`openclaw.internal.`) і Tailscale split DNS.
Див. [Bonjour](/uk/gateway/bonjour) для прикладу CoreDNS.

### Ручний host/port

У Settings увімкніть **Manual Host** і введіть хост gateway + порт (за замовчуванням `18789`).

## Canvas + A2UI

Вузол iOS рендерить canvas WKWebView. Використовуйте `node.invoke`, щоб керувати ним:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

Примітки:

- Хост canvas Gateway обслуговує `/__openclaw__/canvas/` і `/__openclaw__/a2ui/`.
- Він обслуговується з HTTP-сервера Gateway (той самий порт, що й `gateway.port`, за замовчуванням `18789`).
- Вузол iOS зберігає вбудований scaffold як підключене подання за замовчуванням. `canvas.a2ui.push` і `canvas.a2ui.reset` використовують bundled сторінку A2UI, що належить застосунку.
- Віддалені сторінки Gateway A2UI на iOS доступні лише для рендерингу; нативні дії кнопок A2UI приймаються лише з bundled сторінок, що належать застосунку.
- Поверніться до вбудованого scaffold за допомогою `canvas.navigate` і `{"url":""}`.

## Зв’язок із Computer Use

Застосунок iOS є мобільною поверхнею вузла, а не backend Codex Computer Use. Codex
Computer Use і `cua-driver mcp` керують локальним робочим столом macOS через інструменти MCP;
застосунок iOS надає можливості iPhone через команди вузла OpenClaw,
як-от `canvas.*`, `camera.*`, `screen.*`, `location.*` і `talk.*`.

Агенти все ще можуть керувати застосунком iOS через OpenClaw, викликаючи команди вузла,
але ці виклики проходять через протокол вузла gateway і дотримуються обмежень iOS
для foreground/background. Використовуйте [Codex Computer Use](/uk/plugins/codex-computer-use)
для керування локальним робочим столом, а цю сторінку — для можливостей вузла iOS.

## Голосове пробудження + режим talk

- Голосове пробудження і режим talk доступні в Settings.
- OpenAI realtime Talk використовує WebRTC, що належить клієнту, коли `talk.realtime.transport` дорівнює `webrtc`; явна конфігурація `gateway-relay` залишається у власності Gateway. Див. [Режим Talk](/uk/nodes/talk).
- Вузли iOS з підтримкою Talk оголошують можливість `talk` і можуть декларувати
  `talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` і `talk.ptt.once`;
  Gateway за замовчуванням дозволяє ці команди push-to-talk для довірених
  вузлів з підтримкою Talk.
- iOS може призупиняти фонове аудіо; розглядайте голосові функції як best-effort, коли застосунок не активний.

## Поширені помилки

- `NODE_BACKGROUND_UNAVAILABLE`: виведіть застосунок iOS на передній план (команди canvas/camera/screen цього потребують).
- `A2UI_HOST_UNAVAILABLE`: bundled сторінка A2UI була недоступна у WebView застосунку; тримайте застосунок на передньому плані на вкладці Screen і повторіть спробу.
- Запит на спарювання ніколи не з’являється: запустіть `openclaw devices list` і схваліть вручну.
- Повторне підключення не вдається після перевстановлення: токен спарювання Keychain було очищено; повторно спаруйте вузол.
