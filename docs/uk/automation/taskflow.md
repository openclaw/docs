TaskFlow — це субстрат оркестрації потоків, який розташований над [фоновими завданнями](/uk/automation/tasks). Він керує довговічними багатокроковими потоками з власним станом, відстеженням ревізій і семантикою синхронізації, тоді як окремі завдання залишаються одиницею відокремленої роботи.

## Коли використовувати TaskFlow

Використовуйте TaskFlow, коли робота охоплює кілька послідовних або розгалужених кроків і вам потрібне довговічне відстеження прогресу між перезапусками Gateway. Для одиночних фонових операцій достатньо звичайного [завдання](/uk/automation/tasks).

| Сценарій                             | Використання        |
| ------------------------------------ | ------------------- |
| Одиночне фонове завдання             | Звичайне завдання   |
| Багатокроковий конвеєр (A, потім B, потім C) | TaskFlow (керований) |
| Спостереження за зовнішньо створеними завданнями | TaskFlow (дзеркалений) |
| Одноразове нагадування               | Cron-завдання       |

## Надійний шаблон запланованого робочого процесу

Для повторюваних робочих процесів, як-от брифінги ринкової аналітики, розглядайте розклад, оркестрацію та перевірки надійності як окремі рівні:

1. Використовуйте [Scheduled Tasks](/uk/automation/cron-jobs) для таймінгу.
2. Використовуйте постійну cron-сесію, коли робочий процес має спиратися на попередній контекст.
3. Використовуйте [Lobster](/uk/tools/lobster) для детермінованих кроків, шлюзів затвердження та токенів відновлення.
4. Використовуйте TaskFlow для відстеження багатокрокового запуску між дочірніми завданнями, очікуваннями, повторними спробами та перезапусками Gateway.

Приклад форми cron:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

Використовуйте `session:<id>` замість `isolated`, коли повторюваному робочому процесу потрібна навмисна історія, підсумки попередніх запусків або постійний контекст. Використовуйте `isolated`, коли кожен запуск має починатися з чистого стану, а весь потрібний стан явно заданий у робочому процесі.

Усередині робочого процесу розміщуйте перевірки надійності перед кроком підсумовування LLM:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

Рекомендовані попередні перевірки:

- Доступність браузера та вибір профілю, наприклад `openclaw` для керованого стану або `user`, коли потрібна сесія Chrome з виконаним входом. Див. [Browser](/uk/tools/browser).
- Облікові дані API та квота для кожного джерела.
- Мережева доступність потрібних кінцевих точок.
- Потрібні інструменти, увімкнені для агента, як-от `lobster`, `browser` і `llm-task`.
- Налаштоване місце призначення для помилок cron, щоб помилки попередніх перевірок були видимими. Див. [Scheduled Tasks](/uk/automation/cron-jobs#delivery-and-output).

Рекомендовані поля походження даних для кожного зібраного елемента:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

Налаштуйте робочий процес так, щоб він відхиляв або позначав застарілі елементи перед підсумовуванням. Крок LLM має отримувати лише структурований JSON, і його слід попросити зберігати `sourceUrl`, `retrievedAt` і `asOf` у вихідних даних. Використовуйте [LLM Task](/uk/tools/llm-task), коли всередині робочого процесу потрібен крок моделі з перевіркою за схемою.

Для багаторазових командних або спільнотних робочих процесів запакуйте CLI, файли `.lobster` і будь-які нотатки з налаштування як skill або Plugin і опублікуйте через [ClawHub](/clawhub). Тримайте захисні обмеження, специфічні для робочого процесу, у цьому пакеті, якщо API Plugin не має потрібної загальної можливості.

## Режими синхронізації

### Керований режим

TaskFlow володіє життєвим циклом від початку до кінця. Він створює завдання як кроки потоку, доводить їх до завершення та автоматично просуває стан потоку.

Приклад: потік щотижневого звіту, який (1) збирає дані, (2) генерує звіт і (3) доставляє його. TaskFlow створює кожен крок як фонове завдання, чекає завершення, а потім переходить до наступного кроку.

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### Дзеркалений режим

TaskFlow спостерігає за зовнішньо створеними завданнями та підтримує стан потоку синхронізованим, не беручи на себе володіння створенням завдань. Це корисно, коли завдання походять із cron-завдань, команд CLI або інших джерел, і вам потрібне єдине подання їхнього прогресу як потоку.

Приклад: три незалежні cron-завдання, які разом формують рутину "ранкових операцій". Дзеркалений потік відстежує їхній спільний прогрес, не контролюючи, коли або як вони виконуються.

## Довговічний стан і відстеження ревізій

Кожен потік зберігає власний стан і відстежує ревізії, щоб прогрес переживав перезапуски Gateway. Відстеження ревізій дає змогу виявляти конфлікти, коли кілька джерел намагаються одночасно просунути той самий потік.
Реєстр потоків використовує SQLite з обмеженим обслуговуванням журналу випереджального запису, включно з
періодичними контрольними точками та контрольними точками під час завершення роботи, тому довготривалі Gateway не зберігають
необмежені побічні файли `registry.sqlite-wal`.

## Поведінка скасування

`openclaw tasks flow cancel` встановлює стійкий намір скасування для потоку. Активні завдання в межах потоку скасовуються, і нові кроки не запускаються. Намір скасування зберігається між перезапусками, тому скасований потік залишається скасованим, навіть якщо Gateway перезапуститься до завершення всіх дочірніх завдань.

## Команди CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| Команда                           | Опис                                          |
| --------------------------------- | --------------------------------------------- |
| `openclaw tasks flow list`        | Показує відстежувані потоки зі статусом і режимом синхронізації |
| `openclaw tasks flow show <id>`   | Переглядає один потік за ідентифікатором потоку або ключем пошуку |
| `openclaw tasks flow cancel <id>` | Скасовує запущений потік і його активні завдання |

## Як потоки пов’язані із завданнями

Потоки координують завдання, а не замінюють їх. Один потік може запускати кілька фонових завдань протягом свого життєвого циклу. Використовуйте `openclaw tasks`, щоб переглядати окремі записи завдань, і `openclaw tasks flow`, щоб переглядати оркеструвальний потік.
