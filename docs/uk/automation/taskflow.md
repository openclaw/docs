Task Flow — це субстрат оркестрації потоків, що розташований над [фоновими завданнями](/uk/automation/tasks). Він керує довговічними багатокроковими потоками з власним станом, відстеженням ревізій і семантикою синхронізації, тоді як окремі завдання залишаються одиницею відокремленої роботи.

## Коли використовувати Task Flow

Використовуйте Task Flow, коли робота охоплює кілька послідовних або розгалужених кроків і вам потрібне довговічне відстеження прогресу між перезапусками gateway. Для одиничних фонових операцій достатньо звичайного [завдання](/uk/automation/tasks).

| Сценарій                             | Використання          |
| ------------------------------------ | --------------------- |
| Одиничне фонове завдання             | Звичайне завдання     |
| Багатокроковий конвеєр (A, потім B, потім C) | Task Flow (керований) |
| Спостереження за зовнішньо створеними завданнями | Task Flow (дзеркальний) |
| Одноразове нагадування               | Завдання Cron         |

## Надійний шаблон запланованого робочого процесу

Для повторюваних робочих процесів, як-от аналітичні брифінги щодо ринку, розглядайте розклад, оркестрацію та перевірки надійності як окремі шари:

1. Використовуйте [заплановані завдання](/uk/automation/cron-jobs) для визначення часу.
2. Зберігайте попередній контекст у власних файлах, базі даних або стані інструментів робочого процесу.
3. Використовуйте [Lobster](/uk/tools/lobster) для детермінованих кроків, шлюзів затвердження та токенів відновлення.
4. Використовуйте Task Flow для відстеження багатокрокового запуску через дочірні завдання, очікування, повторні спроби та перезапуски gateway.

Приклад форми cron:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

Використовуйте `session:<id>`, коли завдання має спрямовуватися до відомого чату/сеансу для контексту доставки або безпечного початкового заповнення налаштувань. Cron все одно виконує кожен запуск у відокремленому сеансі, тому розміщуйте підсумки попередніх запусків і постійний стан робочого процесу в явному сховищі, яке завдання може прочитати.

Усередині робочого процесу розмістіть перевірки надійності перед кроком підсумовування LLM:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

Рекомендовані попередні перевірки:

- Доступність браузера та вибір профілю, наприклад `openclaw` для керованого стану або `user`, коли потрібен сеанс Chrome із виконаним входом. Див. [Браузер](/uk/tools/browser).
- Облікові дані API та квота для кожного джерела.
- Мережева доступність потрібних кінцевих точок.
- Потрібні інструменти, увімкнені для агента, як-от `lobster`, `browser` і `llm-task`.
- Налаштоване місце призначення для збоїв cron, щоб помилки попередньої перевірки були видимі. Див. [Заплановані завдання](/uk/automation/cron-jobs#delivery-and-output).

Рекомендовані поля походження даних для кожного зібраного елемента:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

Зробіть так, щоб робочий процес відхиляв або позначав застарілі елементи перед підсумовуванням. Крок LLM має отримувати лише структурований JSON, і його слід попросити зберігати `sourceUrl`, `retrievedAt` і `asOf` у своєму виводі. Використовуйте [LLM Task](/uk/tools/llm-task), коли вам потрібен крок моделі з перевіркою схеми всередині робочого процесу.

Для багаторазових командних або спільнотних робочих процесів упакуйте CLI, файли `.lobster` і будь-які нотатки з налаштування як skill або plugin та опублікуйте через [ClawHub](/clawhub). Тримайте специфічні для робочого процесу захисні правила в цьому пакеті, якщо API plugin не має потрібної загальної можливості.

## Режими синхронізації

### Керований режим

Task Flow володіє життєвим циклом від початку до кінця. Він створює завдання як кроки потоку, доводить їх до завершення та автоматично просуває стан потоку.

Приклад: потік щотижневого звіту, який (1) збирає дані, (2) генерує звіт і (3) доставляє його. Task Flow створює кожен крок як фонове завдання, чекає на завершення, а потім переходить до наступного кроку.

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### Дзеркальний режим

Task Flow спостерігає за зовнішньо створеними завданнями та підтримує стан потоку синхронізованим, не беручи на себе володіння створенням завдань. Це корисно, коли завдання походять із cron-завдань, команд CLI або інших джерел, і ви хочете мати уніфікований перегляд їхнього прогресу як потоку.

Приклад: три незалежні cron-завдання, які разом утворюють рутину "morning ops". Дзеркальний потік відстежує їхній сукупний прогрес, не керуючи тим, коли або як вони виконуються.

## Довговічний стан і відстеження ревізій

Кожен потік зберігає власний стан і відстежує ревізії, щоб прогрес переживав перезапуски gateway. Відстеження ревізій дає змогу виявляти конфлікти, коли кілька джерел намагаються одночасно просунути той самий потік.
Реєстр потоків використовує SQLite з обмеженим обслуговуванням журналу попереднього запису, зокрема
періодичними контрольними точками та контрольними точками під час завершення роботи, тому довготривалі gateway не зберігають
необмежені допоміжні файли `registry.sqlite-wal`.

## Поведінка скасування

`openclaw tasks flow cancel` встановлює для потоку стійкий намір скасування. Активні завдання в потоці скасовуються, і нові кроки не запускаються. Намір скасування зберігається між перезапусками, тому скасований потік залишається скасованим, навіть якщо gateway перезапускається до завершення всіх дочірніх завдань.

## Команди CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| Команда                           | Опис                                           |
| --------------------------------- | ---------------------------------------------- |
| `openclaw tasks flow list`        | Показує відстежувані потоки зі статусом і режимом синхронізації |
| `openclaw tasks flow show <id>`   | Переглянути один потік за id потоку або ключем пошуку |
| `openclaw tasks flow cancel <id>` | Скасувати запущений потік і його активні завдання |

## Як потоки пов'язані із завданнями

Потоки координують завдання, а не замінюють їх. Один потік може запускати кілька фонових завдань протягом свого життєвого циклу. Використовуйте `openclaw tasks`, щоб переглядати записи окремих завдань, і `openclaw tasks flow`, щоб переглядати оркеструвальний потік.
