OpenClaw застосовує **виправлення, специфічні для провайдера**, до транскриптів перед запуском (під час побудови контексту моделі). Більшість із них є коригуваннями **в пам’яті**, потрібними для виконання суворих вимог провайдера. Окремий прохід ремонту файлу сесії також може переписати збережений JSONL до завантаження сесії, але лише для некоректно сформованих рядків або збережених ходів, які є недійсними довготривалими записами. Доставлені відповіді assistant зберігаються на диску; специфічне для провайдера вилучення assistant-prefill відбувається лише під час формування вихідних payload. Коли відбувається ремонт, початковий файл резервно копіюється поряд із файлом сесії.

Область охоплює:

- Runtime-контекст промпта, який не потрапляє до видимих користувачу ходів транскрипту
- Очищення ідентифікаторів викликів інструментів
- Валідацію вхідних даних викликів інструментів
- Ремонт парності результатів інструментів
- Валідацію / впорядкування ходів
- Очищення сигнатур thought
- Очищення сигнатур thinking
- Очищення payload зображень
- Очищення порожніх текстових блоків перед повторним відтворенням для провайдера
- Позначення походження користувацького введення (для промптів, маршрутизованих між сесіями)
- Ремонт порожнього ходу помилки assistant для повторного відтворення Bedrock Converse

Якщо вам потрібні деталі зберігання транскриптів, дивіться:

- [Поглиблений огляд керування сесіями](/uk/reference/session-management-compaction)

## Глобальне правило: runtime-контекст не є транскриптом користувача

Runtime/system-контекст може бути додано до промпта моделі для ходу, але це
не вміст, створений кінцевим користувачем. OpenClaw зберігає окреме тіло
промпта, орієнтоване на транскрипт, для відповідей Gateway, поставлених у чергу подальших дій, ACP, CLI та вбудованих запусків Pi.
Збережені видимі користувацькі ходи використовують це тіло транскрипту замість
промпта, збагаченого runtime-контекстом.

Для застарілих сесій, у яких уже було збережено runtime-обгортки, поверхні історії Gateway
застосовують проєкцію відображення перед поверненням повідомлень клієнтам
WebChat, TUI, REST або SSE.

## Де це виконується

Уся гігієна транскриптів централізована у вбудованому runner:

- Вибір політики: `src/agents/transcript-policy.ts`
- Застосування очищення/ремонту: `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

Політика використовує `provider`, `modelApi` і `modelId`, щоб визначити, що застосовувати.

Окремо від гігієни транскриптів файли сесій ремонтуються (за потреби) перед завантаженням:

- `repairSessionFileIfNeeded` у `src/agents/session-file-repair.ts`
- Викликається з `run/attempt.ts` і `compact.ts` (вбудований runner)

## Глобальне правило: очищення зображень

Payload зображень завжди очищується, щоб запобігти відхиленню на боці провайдера через обмеження
розміру (зменшення масштабу/повторне стиснення завеликих base64-зображень).

Це також допомагає контролювати токенне навантаження від зображень для моделей із підтримкою vision.
Менші максимальні розміри зазвичай зменшують використання токенів; більші розміри зберігають деталізацію.

Реалізація:

- `sanitizeSessionMessagesImages` у `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` у `src/agents/tool-images.ts`
- Максимальна сторона зображення налаштовується через `agents.defaults.imageMaxDimensionPx` (типово: `1200`).
- Порожні текстові блоки видаляються, поки цей прохід обходить вміст повторного відтворення. Ходи assistant,
  які стають порожніми, вилучаються з копії повторного відтворення; користувацькі ходи та ходи результатів інструментів,
  які стають порожніми, отримують непорожній placeholder пропущеного вмісту.

## Глобальне правило: некоректно сформовані виклики інструментів

Блоки викликів інструментів assistant, у яких відсутні і `input`, і `arguments`, вилучаються
до побудови контексту моделі. Це запобігає відхиленням провайдера через частково
збережені виклики інструментів (наприклад, після збою через обмеження швидкості).

Реалізація:

- `sanitizeToolCallInputs` у `src/agents/session-transcript-repair.ts`
- Застосовується в `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

## Глобальне правило: походження міжсесійного введення

Коли агент надсилає промпт в іншу сесію через `sessions_send` (включно з кроками
відповіді/оголошення від агента до агента), OpenClaw зберігає створений користувацький хід із:

- `message.provenance.kind = "inter_session"`

OpenClaw також додає перед текстом маршрутизованого промпта в тому самому ході маркер
`[Inter-session message ... isUser=false]`, щоб активний виклик моделі міг відрізнити
вивід сторонньої сесії від зовнішніх інструкцій кінцевого користувача. Цей маркер містить
джерельну сесію, канал та інструмент, коли вони доступні. Транскрипт усе ще використовує
`role: "user"` для сумісності з провайдерами, але видимий текст і метадані походження
позначають цей хід як міжсесійні дані.

Під час повторної побудови контексту OpenClaw застосовує той самий маркер до старіших збережених
міжсесійних користувацьких ходів, які мають лише метадані походження.

## Матриця провайдерів (поточна поведінка)

**OpenAI / OpenAI Codex**

- Лише очищення зображень.
- Вилучення осиротілих reasoning-сигнатур (окремих reasoning-елементів без наступного content-блоку) для транскриптів OpenAI Responses/Codex, а також вилучення придатного до повторного відтворення reasoning OpenAI після перемикання маршруту моделі.
- Збереження payload reasoning-елементів OpenAI Responses, придатних до повторного відтворення, включно із зашифрованими елементами з порожнім summary, щоб ручне/WebSocket-відтворення зберігало потрібний стан `rs_*` у парі з елементами виводу assistant.
- Без очищення ідентифікаторів викликів інструментів.
- Ремонт парності результатів інструментів може переміщувати справжні зіставлені виводи та синтезувати Codex-style виводи `aborted` для відсутніх викликів інструментів.
- Без валідації чи перевпорядкування ходів.
- Відсутні виводи інструментів сімейства OpenAI Responses синтезуються як `aborted`, щоб відповідати нормалізації повторного відтворення Codex.
- Без вилучення сигнатур thought.

**OpenAI-сумісна Gemma 4**

- Історичні thinking/reasoning-блоки assistant вилучаються перед повторним відтворенням, щоб локальні
  OpenAI-сумісні сервери Gemma 4 не отримували reasoning-вміст із попередніх ходів.
- Поточні продовження викликів інструментів у межах того самого ходу зберігають reasoning-блок assistant,
  прикріплений до виклику інструмента, доки результат інструмента не буде повторно відтворено.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Очищення ідентифікаторів викликів інструментів: суворо літерно-цифрові.
- Ремонт парності результатів інструментів і синтетичні результати інструментів.
- Валідація ходів (чергування ходів у стилі Gemini).
- Виправлення порядку ходів Google (додавання крихітного користувацького bootstrap на початок, якщо історія починається з assistant).
- Antigravity Claude: нормалізація thinking-сигнатур; вилучення непідписаних thinking-блоків.

**Anthropic / Minimax (Anthropic-сумісний)**

- Ремонт парності результатів інструментів і синтетичні результати інструментів.
- Валідація ходів (об’єднання послідовних користувацьких ходів для дотримання суворого чергування).
- Кінцеві ходи assistant prefill вилучаються з вихідних payload Anthropic Messages,
  коли thinking увімкнено, включно з маршрутами Cloudflare AI Gateway.
- Thinking-блоки з відсутніми, порожніми або blank replay-сигнатурами вилучаються
  перед перетворенням для провайдера. Якщо це спорожнює хід assistant, OpenClaw зберігає
  форму ходу з непорожнім текстом omitted-reasoning.
- Старіші ходи assistant лише з thinking, які потрібно вилучити, замінюються
  непорожнім текстом omitted-reasoning, щоб адаптери провайдера не відкидали хід
  повторного відтворення.

**Amazon Bedrock (Converse API)**

- Порожні ходи помилки потоку assistant ремонтуються до непорожнього fallback-текстового блоку
  перед повторним відтворенням. Bedrock Converse відхиляє повідомлення assistant із `content: []`, тому
  збережені ходи assistant із `stopReason: "error"` і порожнім вмістом також
  ремонтуються на диску перед завантаженням.
- Ходи помилки потоку assistant, які містять лише порожні текстові блоки, вилучаються
  з копії повторного відтворення в пам’яті замість відтворення недійсного порожнього блоку.
- Claude thinking-блоки з відсутніми, порожніми або blank replay-сигнатурами
  вилучаються перед повторним відтворенням Converse. Якщо це спорожнює хід assistant, OpenClaw
  зберігає форму ходу з непорожнім текстом omitted-reasoning.
- Старіші ходи assistant лише з thinking, які потрібно вилучити, замінюються
  непорожнім текстом omitted-reasoning, щоб повторне відтворення Converse зберігало сувору форму ходів.
- Повторне відтворення фільтрує delivery-mirror OpenClaw і gateway-injected ходи assistant.
- Очищення зображень застосовується через глобальне правило.

**Mistral (включно з виявленням на основі model-id)**

- Очищення ідентифікаторів викликів інструментів: strict9 (літерно-цифрові довжиною 9).

**OpenRouter Gemini**

- Очищення сигнатур thought: вилучення не-base64 значень `thought_signature` (base64 зберігається).

**OpenRouter Anthropic**

- Кінцеві ходи assistant prefill вилучаються з перевірених OpenRouter
  OpenAI-сумісних payload моделей Anthropic, коли reasoning увімкнено, відповідно до
  поведінки повторного відтворення direct Anthropic і Cloudflare Anthropic.

**Усе інше**

- Лише очищення зображень.

## Історична поведінка (до 2026.1.22)

До релізу 2026.1.22 OpenClaw застосовував кілька шарів гігієни транскриптів:

- **transcript-sanitize extension** запускався під час кожної побудови контексту та міг:
  - Ремонтувати парність tool use/result.
  - Очищувати ідентифікатори викликів інструментів (включно з несуворим режимом, який зберігав `_`/`-`).
- Runner також виконував очищення, специфічне для провайдера, що дублювало роботу.
- Додаткові мутації відбувалися поза політикою провайдера, зокрема:
  - Вилучення тегів `<final>` із тексту assistant перед збереженням.
  - Відкидання порожніх ходів помилок assistant.
  - Обрізання вмісту assistant після викликів інструментів.

Ця складність спричиняла регресії між провайдерами (зокрема парність `openai-responses`
`call_id|fc_id`). Очищення 2026.1.22 прибрало extension, централізувало
логіку в runner і зробило OpenAI **no-touch** поза очищенням зображень.
