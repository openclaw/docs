# Гігієна транскриптів (fixups для provider)

Цей документ описує **специфічні для provider виправлення**, які застосовуються до транскриптів перед запуском
(побудовою контексту моделі). Це коригування **в пам’яті**, які використовуються для задоволення суворих
вимог provider. Ці кроки гігієни **не** переписують збережений JSONL-транскрипт
на диску; однак окремий прохід відновлення файла сесії може переписати некоректні JSONL-файли,
відкинувши невалідні рядки до завантаження сесії. Коли відновлення відбувається, оригінальний
файл резервно копіюється поруч із файлом сесії.

Область охоплює:

- Санітизацію id викликів tools
- Валідацію input викликів tools
- Відновлення pairing result викликів tools
- Валідацію / упорядкування ходів
- Очищення thought signature
- Санітизацію payload зображень
- Позначення походження user-input (для prompt, маршрутизованих між сесіями)

Якщо вам потрібні деталі зберігання транскриптів, див.:

- [/reference/session-management-compaction](/uk/reference/session-management-compaction)

## Де це виконується

Уся гігієна транскриптів централізована у вбудованому runner:

- Вибір policy: `src/agents/transcript-policy.ts`
- Застосування санітизації/відновлення: `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

Policy використовує `provider`, `modelApi` і `modelId`, щоб вирішити, що саме застосовувати.

Окремо від гігієни транскриптів, файли сесій відновлюються (за потреби) до завантаження:

- `repairSessionFileIfNeeded` у `src/agents/session-file-repair.ts`
- Викликається з `run/attempt.ts` і `compact.ts` (вбудований runner)

## Глобальне правило: санітизація зображень

Payload зображень завжди санітизуються, щоб запобігти відхиленню на боці provider через
обмеження розміру (зменшення/повторне стиснення надто великих base64-зображень).

Це також допомагає контролювати тиск на токени, що створюється зображеннями, для моделей із підтримкою vision.
Менші максимальні розміри зазвичай зменшують використання токенів; більші зберігають деталі.

Реалізація:

- `sanitizeSessionMessagesImages` у `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` у `src/agents/tool-images.ts`
- Максимальний розмір сторони зображення налаштовується через `agents.defaults.imageMaxDimensionPx` (типово: `1200`).

## Глобальне правило: некоректні виклики tools

Блоки викликів tools асистента, у яких відсутні і `input`, і `arguments`, відкидаються
до побудови контексту моделі. Це запобігає відхиленню provider через частково
збережені виклики tools (наприклад, після збою через rate limit).

Реалізація:

- `sanitizeToolCallInputs` у `src/agents/session-transcript-repair.ts`
- Застосовується в `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

## Глобальне правило: походження вводу між сесіями

Коли агент надсилає prompt в іншу сесію через `sessions_send` (включно з
кроками відповіді/оголошення agent-to-agent), OpenClaw зберігає створений user turn з:

- `message.provenance.kind = "inter_session"`

Ці метадані записуються під час додавання до транскрипту і не змінюють роль
(`role: "user"` залишається для сумісності з provider). Читачі транскриптів можуть використовувати
це, щоб не трактувати маршрутизовані внутрішні prompts як інструкції, створені кінцевим користувачем.

Під час відбудови контексту OpenClaw також у пам’яті додає короткий маркер `[Inter-session message]`
до таких user turns, щоб модель могла відрізняти їх від
інструкцій зовнішнього кінцевого користувача.

## Матриця provider (поточна поведінка)

**OpenAI / OpenAI Codex**

- Лише санітизація зображень.
- Відкидання осиротілих reasoning signature (окремі reasoning items без наступного content block) для транскриптів OpenAI Responses/Codex.
- Без санітизації id викликів tools.
- Без відновлення pairing result tools.
- Без валідації або перестановки ходів.
- Без синтетичних result tools.
- Без видалення thought signature.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Санітизація id викликів tools: суворо алфавітно-цифрові.
- Відновлення pairing result tools і синтетичні result tools.
- Валідація ходів (чергування ходів у стилі Gemini).
- Виправлення порядку ходів Google (додавання крихітного bootstrap user на початку, якщо історія починається з assistant).
- Antigravity Claude: нормалізація thinking signature; відкидання thinking blocks без підпису.

**Anthropic / Minimax (Anthropic-compatible)**

- Відновлення pairing result tools і синтетичні result tools.
- Валідація ходів (злиття послідовних user turns для задоволення суворого чергування).

**Mistral (включно з визначенням за model-id)**

- Санітизація id викликів tools: strict9 (алфавітно-цифрові, довжина 9).

**OpenRouter Gemini**

- Очищення thought signature: видалення значень `thought_signature`, які не є base64 (base64 зберігаються).

**Усе інше**

- Лише санітизація зображень.

## Історична поведінка (до 2026.1.22)

До випуску 2026.1.22 OpenClaw застосовував кілька шарів гігієни транскриптів:

- **Розширення transcript-sanitize** запускалося під час кожної побудови контексту і могло:
  - Відновлювати pairing tool use/result.
  - Санітизувати id викликів tools (включно з нестрогим режимом, який зберігав `_`/`-`).
- Runner також виконував специфічну для provider санітизацію, що дублювало роботу.
- Додаткові мутації відбувалися поза policy provider, зокрема:
  - Видалення тегів `<final>` з тексту assistant до збереження.
  - Відкидання порожніх assistant turns із помилками.
  - Обрізання вмісту assistant після викликів tools.

Ця складність спричиняла міжprovider-ні регресії (зокрема для pairing
`call_id|fc_id` у `openai-responses`). Очищення у 2026.1.22 прибрало розширення, централізувало
логіку в runner і зробило OpenAI **недоторканим** понад санітизацію зображень.
