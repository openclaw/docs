OpenClaw застосовує **виправлення для конкретних провайдерів** до транскриптів перед запуском (побудовою контексту моделі). Більшість із них — це коригування **в пам’яті**, які потрібні для відповідності суворим вимогам провайдерів. Окремий прохід відновлення файлу сесії також може переписувати збережений JSONL перед завантаженням сесії: або видаляючи некоректні рядки JSONL, або відновлюючи збережені turns, які синтаксично валідні, але відомо, що вони відхиляються
провайдером під час повторного відтворення. Коли відбувається відновлення, оригінальний файл резервується поруч
із файлом сесії.

Обсяг охоплює:

- Контекст запиту лише для часу виконання, який не потрапляє до видимих користувачу turns транскрипту
- Очищення ідентифікаторів викликів інструментів
- Валідацію вхідних даних викликів інструментів
- Відновлення парності результатів інструментів
- Валідацію / впорядкування turns
- Очищення сигнатур думок
- Очищення thinking-сигнатур
- Очищення корисного навантаження зображень
- Очищення порожніх текстових блоків перед повторним відтворенням у провайдера
- Позначення походження введення користувача (для запитів, маршрутизованих між сесіями)
- Відновлення порожніх assistant error-turns для повторного відтворення Bedrock Converse

Якщо вам потрібні деталі зберігання транскриптів, див.:

- [Поглиблений огляд керування сесіями](/uk/reference/session-management-compaction)

## Глобальне правило: контекст часу виконання не є користувацьким транскриптом

Контекст часу виконання/системний контекст може додаватися до запиту моделі для turn, але він
не є вмістом, створеним кінцевим користувачем. OpenClaw зберігає окреме тіло запиту,
призначене для транскрипту, для відповідей Gateway, чергових followups, ACP, CLI та вбудованих запусків Pi.
Збережені видимі користувацькі turns використовують це тіло транскрипту замість
запиту, збагаченого контекстом часу виконання.

Для застарілих сесій, у яких уже збережено runtime-обгортки, поверхні історії Gateway
застосовують проєкцію відображення перед поверненням повідомлень клієнтам WebChat,
TUI, REST або SSE.

## Де це виконується

Уся гігієна транскриптів централізована у вбудованому runner:

- Вибір політики: `src/agents/transcript-policy.ts`
- Застосування очищення/відновлення: `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

Політика використовує `provider`, `modelApi` і `modelId`, щоб вирішити, що застосовувати.

Окремо від гігієни транскриптів, файли сесій відновлюються (за потреби) перед завантаженням:

- `repairSessionFileIfNeeded` у `src/agents/session-file-repair.ts`
- Викликається з `run/attempt.ts` і `compact.ts` (вбудований runner)

## Глобальне правило: очищення зображень

Корисні навантаження зображень завжди очищуються, щоб запобігти відхиленню на боці провайдера через обмеження
розміру (зменшення масштабу/повторне стиснення надто великих base64-зображень).

Це також допомагає контролювати тиск на токени, спричинений зображеннями, для моделей із підтримкою бачення.
Менші максимальні розміри зазвичай зменшують використання токенів; більші розміри зберігають деталізацію.

Реалізація:

- `sanitizeSessionMessagesImages` у `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` у `src/agents/tool-images.ts`
- Максимальна сторона зображення налаштовується через `agents.defaults.imageMaxDimensionPx` (типово: `1200`).
- Порожні текстові блоки видаляються, поки цей прохід обходить вміст повторного відтворення. Assistant
  turns, які стають порожніми, вилучаються з копії повторного відтворення; user і tool-result
  turns, які стають порожніми, отримують непорожній placeholder для пропущеного вмісту.

## Глобальне правило: некоректні виклики інструментів

Блоки assistant tool-call, у яких відсутні і `input`, і `arguments`, видаляються
до побудови контексту моделі. Це запобігає відхиленням провайдерами через частково
збережені виклики інструментів (наприклад, після збою через ліміт частоти).

Реалізація:

- `sanitizeToolCallInputs` у `src/agents/session-transcript-repair.ts`
- Застосовується в `sanitizeSessionHistory` у `src/agents/pi-embedded-runner/replay-history.ts`

## Глобальне правило: походження міжсесійного введення

Коли агент надсилає запит в іншу сесію через `sessions_send` (включно з
кроками reply/announce між агентами), OpenClaw зберігає створений user turn з:

- `message.provenance.kind = "inter_session"`

OpenClaw також додає на початок того самого turn маркер `[Inter-session message ... isUser=false]`
перед текстом маршрутизованого запиту, щоб активний виклик моделі міг відрізнити
вивід чужої сесії від зовнішніх інструкцій кінцевого користувача. Цей маркер включає
джерельну сесію, канал та інструмент, коли вони доступні. Транскрипт усе одно використовує
`role: "user"` для сумісності з провайдерами, але і видимий текст, і метадані
походження позначають turn як міжсесійні дані.

Під час перебудови контексту OpenClaw застосовує такий самий маркер до старіших збережених
міжсесійних user turns, які мають лише метадані походження.

## Матриця провайдерів (поточна поведінка)

**OpenAI / OpenAI Codex**

- Лише очищення зображень.
- Видаляє осиротілі reasoning-сигнатури (окремі reasoning-елементи без наступного content-блоку) для транскриптів OpenAI Responses/Codex і видаляє придатний до повторного відтворення OpenAI reasoning після перемикання маршруту моделі.
- Зберігає корисні навантаження reasoning-елементів OpenAI Responses, придатні до повторного відтворення, включно із зашифрованими елементами з порожнім summary, щоб ручне/WebSocket повторне відтворення зберігало потрібний стан `rs_*` у парі з елементами виводу assistant.
- Без очищення ідентифікаторів викликів інструментів.
- Відновлення парності результатів інструментів може переміщувати реальні зіставлені outputs і синтезувати outputs у стилі Codex зі станом `aborted` для відсутніх викликів інструментів.
- Без валідації або перевпорядкування turns.
- Відсутні outputs інструментів сімейства OpenAI Responses синтезуються як `aborted`, щоб відповідати нормалізації повторного відтворення Codex.
- Без вилучення сигнатур думок.

**OpenAI-сумісна Gemma 4**

- Історичні блоки thinking/reasoning assistant вилучаються перед повторним відтворенням, щоб локальні
  OpenAI-сумісні сервери Gemma 4 не отримували reasoning-вміст із попередніх turns.
- Поточні продовження tool-call у тому самому turn зберігають reasoning-блок assistant,
  прикріплений до виклику інструмента, доки результат інструмента не буде повторно відтворено.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Очищення ідентифікаторів викликів інструментів: суворо буквено-цифрові.
- Відновлення парності результатів інструментів і синтетичні результати інструментів.
- Валідація turns (чергування turns у стилі Gemini).
- Виправлення порядку turns для Google (додає на початок крихітний користувацький bootstrap, якщо історія починається з assistant).
- Antigravity Claude: нормалізує thinking-сигнатури; видаляє непідписані thinking-блоки.

**Anthropic / Minimax (Anthropic-сумісний)**

- Відновлення парності результатів інструментів і синтетичні результати інструментів.
- Валідація turns (об’єднує послідовні user turns, щоб задовольнити суворе чергування).
- Кінцеві assistant prefill turns вилучаються з вихідних корисних навантажень Anthropic Messages,
  коли thinking увімкнено, включно з маршрутами Cloudflare AI Gateway.
- Thinking-блоки з відсутніми, порожніми або blank replay-сигнатурами вилучаються
  перед конвертацією для провайдера. Якщо через це assistant turn стає порожнім, OpenClaw зберігає
  форму turn з непорожнім текстом omitted-reasoning.
- Старіші thinking-only assistant turns, які потрібно вилучити, замінюються
  непорожнім текстом omitted-reasoning, щоб адаптери провайдерів не відкидали replay
  turn.

**Amazon Bedrock (Converse API)**

- Порожні assistant stream-error turns відновлюються до непорожнього fallback-текстового блоку
  перед повторним відтворенням. Bedrock Converse відхиляє повідомлення assistant з `content: []`, тому
  збережені assistant turns з `stopReason: "error"` і порожнім вмістом також
  відновлюються на диску перед завантаженням.
- Assistant stream-error turns, які містять лише порожні текстові блоки, вилучаються
  з копії повторного відтворення в пам’яті замість відтворення некоректного порожнього блоку.
- Claude thinking-блоки з відсутніми, порожніми або blank replay-сигнатурами
  вилучаються перед повторним відтворенням Converse. Якщо через це assistant turn стає порожнім, OpenClaw
  зберігає форму turn з непорожнім текстом omitted-reasoning.
- Старіші thinking-only assistant turns, які потрібно вилучити, замінюються
  непорожнім текстом omitted-reasoning, щоб replay Converse зберігав сувору форму turns.
- Replay фільтрує delivery-mirror OpenClaw і gateway-injected assistant turns.
- Очищення зображень застосовується через глобальне правило.

**Mistral (включно з виявленням на основі model-id)**

- Очищення ідентифікаторів викликів інструментів: strict9 (буквено-цифрові, довжина 9).

**OpenRouter Gemini**

- Очищення сигнатур думок: вилучає не-base64 значення `thought_signature` (залишає base64).

**OpenRouter Anthropic**

- Кінцеві assistant prefill turns вилучаються з перевірених корисних навантажень OpenRouter
  OpenAI-сумісних моделей Anthropic, коли reasoning увімкнено, відповідно до
  поведінки повторного відтворення прямого Anthropic і Cloudflare Anthropic.

## Історична поведінка (до 2026.1.22)

До випуску 2026.1.22 OpenClaw застосовував кілька шарів гігієни транскриптів:

- **transcript-sanitize extension** виконувався під час кожної побудови контексту і міг:
  - Відновлювати парність tool use/result.
  - Очищати ідентифікатори викликів інструментів (включно з несуворим режимом, який зберігав `_`/`-`).
- Runner також виконував очищення для конкретних провайдерів, що дублювало роботу.
- Додаткові мутації відбувалися поза політикою провайдера, включно з:
  - Вилученням тегів `<final>` з тексту assistant перед збереженням.
  - Видаленням порожніх assistant error turns.
  - Обрізанням вмісту assistant після викликів інструментів.

Ця складність спричиняла регресії між провайдерами (зокрема парність `openai-responses`
`call_id|fc_id`). Очищення 2026.1.22 прибрало extension, централізувало
логіку в runner і зробило OpenAI **недоторканним** за межами очищення зображень.
