OpenClaw наскрізно керує сесіями в таких областях:

- **Маршрутизація сесій** (як вхідні повідомлення зіставляються із `sessionKey`)
- **Сховище сесій** (`sessions.json`) і те, що воно відстежує
- **Збереження transcript** (`*.jsonl`) і його структура
- **Гігієна transcript** (виправлення для конкретних провайдерів перед запусками)
- **Обмеження контексту** (контекстне вікно проти відстежуваних токенів)
- **Compaction** (ручний і автоматичний Compaction) і де підключати підготовчу роботу перед Compaction
- **Тихе службове обслуговування** (записи в пам’ять, які не повинні створювати видимий для користувача вивід)

## Джерело істини: Gateway

OpenClaw спроєктовано навколо єдиного **процесу Gateway**, який володіє станом сесій.

- UI (macOS app, вебінтерфейс Control UI, TUI) мають запитувати Gateway для отримання списків сесій і кількості токенів.
- У віддаленому режимі файли сесій знаходяться на віддаленому хості; «перевірка локальних файлів на вашому Mac» не відображатиме те, що використовує Gateway.

---

## Два шари збереження

OpenClaw зберігає сесії у двох шарах:

1. **Сховище сесій (`sessions.json`)**
   - Мапа ключ/значення: `sessionKey -> SessionEntry`
   - Невелике, змінюване, безпечне для редагування (або видалення записів)
   - Відстежує метадані сесії (поточний id сесії, останню активність, перемикачі, лічильники токенів тощо)

2. **Transcript (`<sessionId>.jsonl`)**
   - Transcript із додаванням у кінець, із деревоподібною структурою (записи мають `id` + `parentId`)
   - Зберігає фактичну розмову + виклики інструментів + підсумки Compaction
   - Використовується для відновлення контексту моделі для майбутніх ходів

## Розташування на диску

Для кожного агента, на хості Gateway:

- Сховище: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- Transcripts: `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
  - Сесії тем Telegram: `.../<sessionId>-topic-<threadId>.jsonl`

OpenClaw визначає ці шляхи через `src/config/sessions.ts`.

## Обслуговування сховища й контроль диска

Збереження сесій має автоматичні механізми обслуговування (`session.maintenance`) для `sessions.json` і артефактів transcript:

- `mode`: `warn` (типово) або `enforce`
- `pruneAfter`: поріг віку застарілих записів (типово `30d`)
- `maxEntries`: максимальна кількість записів у `sessions.json` (типово `500`)
- `rotateBytes`: ротація `sessions.json`, коли файл стає завеликим (типово `10mb`)
- `resetArchiveRetention`: строк зберігання архівів transcript `*.reset.<timestamp>` (типово: такий самий, як `pruneAfter`; `false` вимикає очищення)
- `maxDiskBytes`: необов’язковий бюджет каталогу сесій
- `highWaterBytes`: необов’язкова ціль після очищення (типово `80%` від `maxDiskBytes`)

Порядок застосування очищення за бюджетом диска (`mode: "enforce"`):

1. Спочатку видалити найстаріші архівні або осиротілі артефакти transcript.
2. Якщо все ще перевищено ціль, витіснити найстаріші записи сесій і їхні файли transcript.
3. Продовжувати, доки використання не стане меншим або рівним `highWaterBytes`.

У режимі `mode: "warn"` OpenClaw повідомляє про потенційні витіснення, але не змінює сховище/файли.

Запуск обслуговування на вимогу:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

## Сесії Cron і журнали запусків

Ізольовані запуски Cron також створюють записи сесій/transcripts, і для них є окремі налаштування зберігання:

- `cron.sessionRetention` (типово `24h`) очищає старі сесії ізольованих запусків Cron зі сховища сесій (`false` вимикає це).
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` очищають файли `~/.openclaw/cron/runs/<jobId>.jsonl` (типові значення: `2_000_000` байтів і `2000` рядків).

Коли Cron примусово створює нову сесію ізольованого запуску, він санітизує попередній
запис сесії `cron:<jobId>` перед записом нового рядка. Він переносить безпечні
налаштування, як-от thinking/fast/verbose, мітки та явні
перевизначення model/auth, вибрані користувачем. Він відкидає фоновий контекст розмови, такий
як маршрутизація channel/group, політика send або queue, elevation, origin і
runtime-прив’язка ACP, щоб новий ізольований запуск не міг успадкувати застаріле доставлення або
повноваження runtime від старішого запуску.

## Ключі сесій (`sessionKey`)

`sessionKey` ідентифікує, _у якому кошику розмови_ ви перебуваєте (маршрутизація + ізоляція).

Поширені шаблони:

- Основний/прямий чат (для агента): `agent:<agentId>:<mainKey>` (типово `main`)
- Група: `agent:<agentId>:<channel>:group:<id>`
- Кімната/канал (Discord/Slack): `agent:<agentId>:<channel>:channel:<id>` або `...:room:<id>`
- Cron: `cron:<job.id>`
- Webhook: `hook:<uuid>` (якщо не перевизначено)

Канонічні правила задокументовано в [/concepts/session](/uk/concepts/session).

---

## Id сесій (`sessionId`)

Кожен `sessionKey` вказує на поточний `sessionId` (файл transcript, який продовжує розмову).

Основні правила:

- **Скидання** (`/new`, `/reset`) створює новий `sessionId` для цього `sessionKey`.
- **Щоденне скидання** (типово о 4:00 ранку за локальним часом на хості gateway) створює новий `sessionId` на наступному повідомленні після межі скидання.
- **Спливання через неактивність** (`session.reset.idleMinutes` або застарілий `session.idleMinutes`) створює новий `sessionId`, коли повідомлення надходить після вікна неактивності. Коли налаштовано і щоденне скидання, і неактивність, спрацьовує те, що спливає раніше.
- **Системні події** (пробудження heartbeat, Cron, сповіщення exec, службові дії gateway) можуть змінювати рядок сесії, але не подовжують свіжість щоденного/неактивного скидання. Під час перенесення через скидання повідомлення системних подій у черзі для попередньої сесії відкидаються до побудови нового prompt.
- **Захист форку батьківської гілки** (`session.parentForkMaxTokens`, типово `100000`) пропускає форкування transcript батьківської сесії, коли батьківська сесія вже надто велика; нова гілка починається з чистого стану. Установіть `0`, щоб вимкнути.

Деталь реалізації: рішення ухвалюється в `initSessionState()` у `src/auto-reply/reply/session.ts`.

## Схема сховища сесій (`sessions.json`)

Тип значення сховища — `SessionEntry` у `src/config/sessions.ts`.

Ключові поля (не вичерпно):

- `sessionId`: id поточного transcript (ім’я файлу виводиться з цього значення, якщо не задано `sessionFile`)
- `sessionStartedAt`: час початку для поточного `sessionId`; свіжість
  щоденного скидання використовує саме його. Для застарілих рядків це значення може визначатися із заголовка сесії JSONL.
- `lastInteractionAt`: час останньої реальної взаємодії користувача/каналу; свіжість скидання
  через неактивність використовує це поле, тому події heartbeat, Cron і exec не підтримують
  життя сесії. Для застарілих рядків без цього поля використовується відновлений час початку сесії
  як базове значення для неактивності.
- `updatedAt`: час останньої зміни рядка сховища; використовується для списків, очищення та
  службових дій. Це не авторитетне поле для свіжості щоденного/неактивного скидання.
- `sessionFile`: необов’язкове явне перевизначення шляху transcript
- `chatType`: `direct | group | room` (допомагає UI і політиці надсилання)
- `provider`, `subject`, `room`, `space`, `displayName`: метадані для міток груп/каналів
- Перемикачі:
  - `thinkingLevel`, `verboseLevel`, `reasoningLevel`, `elevatedLevel`
  - `sendPolicy` (перевизначення для конкретної сесії)
- Вибір моделі:
  - `providerOverride`, `modelOverride`, `authProfileOverride`
- Лічильники токенів (best-effort / залежать від провайдера):
  - `inputTokens`, `outputTokens`, `totalTokens`, `contextTokens`
- `compactionCount`: скільки разів автоматичний Compaction завершувався для цього ключа сесії
- `memoryFlushAt`: час останнього скидання пам’яті перед Compaction
- `memoryFlushCompactionCount`: лічильник Compaction на момент останнього скидання

Сховище можна безпечно редагувати, але авторитетом є Gateway: він може переписувати або відновлювати записи під час роботи сесій.

## Структура transcript (`*.jsonl`)

Transcripts керуються `SessionManager` із `@mariozechner/pi-coding-agent`.

Файл має формат JSONL:

- Перший рядок: заголовок сесії (`type: "session"`, містить `id`, `cwd`, `timestamp`, необов’язковий `parentSession`)
- Далі: записи сесії з `id` + `parentId` (дерево)

Помітні типи записів:

- `message`: повідомлення user/assistant/toolResult
- `custom_message`: повідомлення, ін’єктовані розширенням, які _потрапляють_ у контекст моделі (можуть бути приховані від UI)
- `custom`: стан розширення, який _не потрапляє_ в контекст моделі
- `compaction`: збережений підсумок Compaction з `firstKeptEntryId` і `tokensBefore`
- `branch_summary`: збережений підсумок під час навігації гілкою дерева

OpenClaw навмисно **не** «виправляє» transcripts; Gateway використовує `SessionManager` для їх читання/запису.

## Контекстні вікна проти відстежуваних токенів

Важливі два різні поняття:

1. **Контекстне вікно моделі**: жорстка межа для моделі (токени, видимі моделі)
2. **Лічильники сховища сесій**: ковзні статистики, записані в `sessions.json` (використовуються для `/status` і панелей)

Якщо ви налаштовуєте обмеження:

- Контекстне вікно походить із каталогу моделей (і може бути перевизначене через конфігурацію).
- `contextTokens` у сховищі — це оцінка/значення звітності під час runtime; не сприймайте його як сувору гарантію.

Докладніше див. у [/token-use](/uk/reference/token-use).

## Compaction: що це таке

Compaction підсумовує старішу частину розмови в збережений запис `compaction` у transcript і залишає недоторканими недавні повідомлення.

Після Compaction майбутні ходи бачать:

- Підсумок Compaction
- Повідомлення після `firstKeptEntryId`

Compaction є **стійким** (на відміну від очищення сесії). Див. [/concepts/session-pruning](/uk/concepts/session-pruning).

## Межі фрагментів Compaction і поєднання інструментів

Коли OpenClaw розбиває довгий transcript на фрагменти для Compaction, він зберігає
пари викликів інструментів асистента з відповідними записами `toolResult`.

- Якщо розділення за часткою токенів припадає між викликом інструмента та його результатом, OpenClaw
  зсуває межу до повідомлення асистента з викликом інструмента, а не розділяє пару.
- Якщо кінцевий блок результату інструмента інакше виштовхнув би фрагмент за цільовий розмір,
  OpenClaw зберігає цей очікувальний блок інструмента й залишає непідсумований хвіст недоторканим.
- Перервані/помилкові блоки викликів інструментів не утримують очікувальний поділ відкритим.

## Коли відбувається автоматичний Compaction (runtime Pi)

У вбудованому агенті Pi автоматичний Compaction спрацьовує у двох випадках:

1. **Відновлення після переповнення**: модель повертає помилку переповнення контексту
   (`request_too_large`, `context length exceeded`, `input exceeds the maximum
number of tokens`, `input token count exceeds the maximum number of input
tokens`, `input is too long for the model`, `ollama error: context length
exceeded` і подібні варіанти для конкретних провайдерів) → Compaction → повторна спроба.
2. **Порогове обслуговування**: після успішного ходу, коли:

`contextTokens > contextWindow - reserveTokens`

Де:

- `contextWindow` — це контекстне вікно моделі
- `reserveTokens` — це запас токенів для prompt і наступного виводу моделі

Це семантика runtime Pi (OpenClaw споживає події, але коли саме виконувати Compaction, вирішує Pi).

OpenClaw також може запускати локальний попередній Compaction перед відкриттям наступного
запуску, коли задано `agents.defaults.compaction.maxActiveTranscriptBytes` і
активний файл transcript досягає цього розміру. Це захист за розміром файлу для локальної
вартості повторного відкриття, а не сире архівування: OpenClaw усе одно виконує звичайний
семантичний Compaction, і для цього потрібен `truncateAfterCompaction`, щоб підсумок
Compaction міг стати новим наступним transcript.

## Налаштування Compaction (`reserveTokens`, `keepRecentTokens`)

Налаштування Compaction Pi знаходяться в налаштуваннях Pi:

```json5
{
  compaction: {
    enabled: true,
    reserveTokens: 16384,
    keepRecentTokens: 20000,
  },
}
```

OpenClaw також застосовує мінімальний поріг безпеки для вбудованих запусків:

- Якщо `compaction.reserveTokens < reserveTokensFloor`, OpenClaw збільшує це значення.
- Типовий мінімальний поріг — `20000` токенів.
- Установіть `agents.defaults.compaction.reserveTokensFloor: 0`, щоб вимкнути цей поріг.
- Якщо значення вже вище, OpenClaw його не змінює.
- Ручний `/compact` дотримується явного `agents.defaults.compaction.keepRecentTokens`
  і зберігає точку відсікання недавнього хвоста Pi. Без явного бюджету збереження
  ручний Compaction залишається жорсткою контрольною точкою, а відновлений контекст починається
  з нового підсумку.
- Установіть `agents.defaults.compaction.maxActiveTranscriptBytes` у значення байтів або
  рядок на кшталт `"20mb"`, щоб запускати локальний Compaction перед ходом, коли активний
  transcript стає великим. Цей захист активний лише тоді, коли
  також увімкнено `truncateAfterCompaction`. Не задавайте значення або встановіть `0`, щоб
  вимкнути.
- Коли увімкнено `agents.defaults.compaction.truncateAfterCompaction`,
  OpenClaw після Compaction ротує активний transcript у compacted successor JSONL.
  Старий повний transcript залишається заархівованим і зв’язаним із контрольною точкою
  Compaction, замість того щоб переписуватися на місці.

Чому: щоб залишити достатньо запасу для багатокрокового «службового обслуговування» (наприклад, записів у пам’ять) до того, як Compaction стане неминучим.

Реалізація: `ensurePiCompactionReserveTokens()` у `src/agents/pi-settings.ts`
(викликається з `src/agents/pi-embedded-runner.ts`).

## Підключувані провайдери Compaction

Plugins можуть реєструвати провайдера Compaction через `registerCompactionProvider()` в API plugin. Коли `agents.defaults.compaction.provider` встановлено в id зареєстрованого провайдера, розширення safeguard делегує підсумовування цьому провайдеру замість вбудованого конвеєра `summarizeInStages`.

- `provider`: id зареєстрованого plugin-провайдера Compaction. Не задавайте, щоб використовувати типове підсумовування LLM.
- Установлення `provider` примусово задає `mode: "safeguard"`.
- Провайдери отримують ті самі інструкції Compaction і політику збереження ідентифікаторів, що й вбудований шлях.
- Safeguard, як і раніше, зберігає контекст суфікса недавніх ходів і розділених ходів після виводу провайдера.
- Вбудоване підсумовування safeguard повторно дистилює попередні підсумки разом із новими повідомленнями,
  а не зберігає повний попередній підсумок дослівно.
- Режим safeguard типово вмикає аудити якості підсумків; установіть
  `qualityGuard.enabled: false`, щоб пропустити поведінку повторних спроб при некоректному виводі.
- Якщо провайдер завершується помилкою або повертає порожній результат, OpenClaw автоматично повертається до вбудованого підсумовування LLM.
- Сигнали abort/timeout перевикидаються далі (не поглинаються), щоб поважати скасування викликувачем.

Джерело: `src/plugins/compaction-provider.ts`, `src/agents/pi-hooks/compaction-safeguard.ts`.

## Поверхні, видимі користувачеві

## Тихе службове обслуговування (`NO_REPLY`)

OpenClaw підтримує «тихі» ходи для фонових завдань, де користувач не повинен бачити проміжний вивід.

Домовленість:

- Асистент починає свій вивід із точного тихого токена `NO_REPLY` /
  `no_reply`, щоб позначити «не доставляти відповідь користувачеві».
- OpenClaw вилучає/пригнічує це на рівні доставлення.
- Пригнічення точного тихого токена нечутливе до регістру, тож `NO_REPLY` і
  `no_reply` зараховуються, коли весь payload — це лише тихий токен.
- Це призначено лише для справді фонових ходів/ходів без доставлення; це не скорочення для
  звичайних користувацьких запитів, які вимагають дії.

Станом на `2026.1.10` OpenClaw також пригнічує **чернеткове/індикаторне потокове виведення**, коли
частковий chunk починається з `NO_REPLY`, щоб тихі операції не розкривали частковий
вивід посеред ходу.

## Підготовче «скидання пам’яті» перед Compaction (реалізовано)

Мета: перед тим, як відбудеться автоматичний Compaction, виконати тихий агентний хід, який записує стійкий
стан на диск (наприклад, `memory/YYYY-MM-DD.md` у робочому просторі агента), щоб Compaction не міг
стерти критичний контекст.

OpenClaw використовує підхід **скидання перед порогом**:

1. Відстежувати використання контексту сесії.
2. Коли воно перетинає «м’який поріг» (нижче порогу Compaction Pi), запускати тиху
   директиву «запиши пам’ять зараз» для агента.
3. Використовувати точний тихий токен `NO_REPLY` / `no_reply`, щоб користувач
   нічого не бачив.

Конфігурація (`agents.defaults.compaction.memoryFlush`):

- `enabled` (типово: `true`)
- `softThresholdTokens` (типово: `4000`)
- `prompt` (повідомлення користувача для ходу скидання)
- `systemPrompt` (додатковий system prompt, доданий для ходу скидання)

Примітки:

- Типові `prompt`/`systemPrompt` містять підказку `NO_REPLY` для пригнічення
  доставлення.
- Скидання виконується один раз за цикл Compaction (відстежується в `sessions.json`).
- Скидання виконується лише для вбудованих сесій Pi (бекенди CLI його пропускають).
- Скидання пропускається, коли робочий простір сесії доступний лише для читання (`workspaceAccess: "ro"` або `"none"`).
- Див. [Пам’ять](/uk/concepts/memory) щодо структури файлів робочого простору та шаблонів запису.

Pi також надає hook `session_before_compact` в API розширень, але логіка
скидання OpenClaw сьогодні знаходиться на боці Gateway.

## Контрольний список для усунення проблем

- Неправильний ключ сесії? Почніть із [/concepts/session](/uk/concepts/session) і підтвердьте `sessionKey` у `/status`.
- Не збігаються сховище й transcript? Перевірте хост Gateway і шлях до сховища з `openclaw status`.
- Забагато Compaction? Перевірте:
  - контекстне вікно моделі (надто мале)
  - налаштування Compaction (`reserveTokens`, занадто велике для вікна моделі, може спричиняти ранніший Compaction)
  - роздування `toolResult`: увімкніть/налаштуйте очищення сесій
- Тихі ходи «протікають»? Переконайтеся, що відповідь починається з `NO_REPLY` (нечутливий до регістру точний токен) і що ви використовуєте збірку, яка містить виправлення пригнічення потокового виведення.
