OpenClaw керує сесіями наскрізно в таких областях:

- **Маршрутизація сесій** (як вхідні повідомлення зіставляються з `sessionKey`)
- **Сховище сесій** (`sessions.json`) і що воно відстежує
- **Збереження стенограм** (`*.jsonl`) та їхня структура
- **Гігієна стенограм** (специфічні для провайдера виправлення перед запусками)
- **Ліміти контексту** (контекстне вікно проти відстежуваних токенів)
- **Compaction** (ручна й автоматична Compaction) і де підключати роботу перед Compaction
- **Тихе обслуговування** (записи пам’яті, які не мають створювати видимого для користувача виводу)

## Джерело істини: Gateway

OpenClaw спроєктовано навколо єдиного **процесу Gateway**, який володіє станом сесій.

- UI (застосунок macOS, веб Control UI, TUI) мають запитувати в Gateway списки сесій і кількість токенів.
- У віддаленому режимі файли сесій розташовані на віддаленому хості; «перевірка локальних файлів на Mac» не відображатиме того, що використовує Gateway.

---

## Два рівні збереження

OpenClaw зберігає сесії на двох рівнях:

1. **Сховище сесій (`sessions.json`)**
   - Мапа ключ/значення: `sessionKey -> SessionEntry`
   - Невелике, змінюване, безпечне для редагування (або видалення записів)
   - Відстежує метадані сесії (поточний ідентифікатор сесії, останню активність, перемикачі, лічильники токенів тощо)

2. **Стенограма (`<sessionId>.jsonl`)**
   - Стенограма лише з додаванням із деревоподібною структурою (записи мають `id` + `parentId`)
   - Зберігає фактичну розмову + виклики інструментів + підсумки Compaction
   - Використовується для відновлення контексту моделі для майбутніх ходів
   - Великі налагоджувальні контрольні точки перед Compaction пропускаються, щойно активна
     стенограма перевищує обмеження розміру контрольної точки, щоб уникнути другої гігантської
     копії `.checkpoint.*.jsonl`.

Зчитувачі історії Gateway мають уникати матеріалізації всієї стенограми, якщо
поверхні явно не потрібен довільний історичний доступ. Історія першої сторінки,
вбудована історія чату, відновлення після перезапуску та перевірки токенів/використання
використовують обмежене читання хвоста. Повні сканування стенограми проходять через асинхронний індекс стенограм, який
кешується за шляхом файлу плюс `mtimeMs`/`size` і спільно використовується паралельними зчитувачами.

## Розташування на диску

Для кожного агента, на хості Gateway:

- Сховище: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- Стенограми: `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
  - Сесії тем Telegram: `.../<sessionId>-topic-<threadId>.jsonl`

OpenClaw визначає ці шляхи через `src/config/sessions.ts`.

## Обслуговування сховища та керування диском

Збереження сесій має автоматичні засоби обслуговування (`session.maintenance`) для `sessions.json`, артефактів стенограм і побічних файлів траєкторій:

- `mode`: `warn` (типово) або `enforce`
- `pruneAfter`: віковий поріг для застарілих записів (типово `30d`)
- `maxEntries`: обмеження кількості записів у `sessions.json` (типово `500`)
- `resetArchiveRetention`: термін зберігання архівів стенограм `*.reset.<timestamp>` (типово: такий самий, як `pruneAfter`; `false` вимикає очищення)
- `maxDiskBytes`: необов’язковий бюджет каталогу сесій
- `highWaterBytes`: необов’язкова ціль після очищення (типово `80%` від `maxDiskBytes`)

Звичайні записи Gateway проходять через окремий для сховища записувач сесій, який серіалізує внутрішньопроцесні мутації без отримання файлового блокування під час виконання. Допоміжні засоби виправлення гарячого шляху позичають перевірений змінюваний кеш, поки утримують цей слот записувача, тому великі файли `sessions.json` не клонуються й не перечитуються для кожного оновлення метаданих. Код під час виконання має віддавати перевагу `updateSessionStore(...)` або `updateSessionStoreEntry(...)`; прямі збереження всього сховища є інструментами сумісності та офлайн-обслуговування. Коли Gateway доступний, `openclaw sessions cleanup --enforce` делегує обслуговування Gateway, щоб очищення приєдналося до тієї самої черги записувача; `--store <path>` — явний офлайн-шлях ремонту для прямого обслуговування файлів. Очищення `maxEntries` досі пакетне для обмежень виробничого розміру, тож сховище може ненадовго перевищити налаштоване обмеження, перш ніж наступне очищення до верхньої межі перепише його назад до меншого розміру. Читання сховища сесій не очищує й не обмежує записи під час запуску Gateway; для очищення використовуйте записи або `openclaw sessions cleanup --enforce`. `openclaw sessions cleanup --enforce` усе одно негайно застосовує налаштоване обмеження.

Обслуговування зберігає довговічні зовнішні вказівники розмов, як-от групові сесії
та сесії чату в межах потоків, але синтетичні записи виконання для cron, хуків,
heartbeat, ACP і субагентів усе ще можуть видалятися, коли вони перевищують
налаштований вік, кількість або дисковий бюджет.

OpenClaw більше не створює автоматичних ротаційних резервних копій `sessions.json.bak.*` під час записів Gateway. Застарілий ключ `session.maintenance.rotateBytes` ігнорується, а `openclaw doctor --fix` видаляє його зі старіших конфігурацій.

Порядок примусового застосування для очищення дискового бюджету (`mode: "enforce"`):

1. Спершу видалити найстаріші архівні, осиротілі артефакти стенограм або осиротілі артефакти траєкторій.
2. Якщо розмір усе ще перевищує ціль, вилучити найстаріші записи сесій і їхні файли стенограм/траєкторій.
3. Продовжувати, доки використання не стане на рівні `highWaterBytes` або нижче.

У `mode: "warn"` OpenClaw повідомляє про потенційні вилучення, але не змінює сховище/файли.

Запуск обслуговування на вимогу:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

## Сесії Cron і журнали запусків

Ізольовані запуски cron також створюють записи сесій/стенограми та мають окремі засоби керування терміном зберігання:

- `cron.sessionRetention` (типово `24h`) очищує старі ізольовані сесії запусків cron зі сховища сесій (`false` вимикає).
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` очищують файли `~/.openclaw/cron/runs/<jobId>.jsonl` (типово: `2_000_000` байтів і `2000` рядків).

Коли cron примусово створює нову ізольовану сесію запуску, він очищує попередній
запис сесії `cron:<jobId>` перед записом нового рядка. Він переносить безпечні
параметри, як-от налаштування мислення/швидкості/детальності, мітки та явні
перевизначення моделі/автентифікації, вибрані користувачем. Він відкидає навколишній контекст розмови, як-от маршрутизацію каналу/групи, політику надсилання або черги, підвищення прав, походження та
прив’язку виконання ACP, щоб новий ізольований запуск не міг успадкувати застарілу доставку або
повноваження виконання від старішого запуску.

## Ключі сесій (`sessionKey`)

`sessionKey` визначає, _у якому контейнері розмови_ ви перебуваєте (маршрутизація + ізоляція).

Поширені шаблони:

- Основний/прямий чат (для кожного агента): `agent:<agentId>:<mainKey>` (типово `main`)
- Група: `agent:<agentId>:<channel>:group:<id>`
- Кімната/канал (Discord/Slack): `agent:<agentId>:<channel>:channel:<id>` або `...:room:<id>`
- Cron: `cron:<job.id>`
- Webhook: `hook:<uuid>` (якщо не перевизначено)

Канонічні правила задокументовано в [/concepts/session](/uk/concepts/session).

## Ідентифікатори сесій (`sessionId`)

Кожен `sessionKey` вказує на поточний `sessionId` (файл стенограми, який продовжує розмову).

Практичні правила:

- **Скидання** (`/new`, `/reset`) створює новий `sessionId` для цього `sessionKey`.
- **Щоденне скидання** (типово 4:00 AM за місцевим часом на хості Gateway) створює новий `sessionId` під час наступного повідомлення після межі скидання.
- **Закінчення строку через бездіяльність** (`session.reset.idleMinutes` або застарілий `session.idleMinutes`) створює новий `sessionId`, коли повідомлення надходить після вікна бездіяльності. Коли налаштовано і щоденне скидання, і бездіяльність, спрацьовує те, що завершується першим.
- **Системні події** (heartbeat, пробудження cron, сповіщення exec, службові дії gateway) можуть змінювати рядок сесії, але не подовжують актуальність щоденного/бездіяльного скидання. Перехід скидання відкидає поставлені в чергу сповіщення системних подій для попередньої сесії перед побудовою нового промпта.
- **Політика відгалуження від батька** використовує активну гілку PI під час створення потоку або відгалуження субагента. Якщо ця гілка занадто велика, OpenClaw запускає дочірній об’єкт з ізольованим контекстом замість помилки або успадкування непридатної історії. Політика визначення розміру автоматична; застарілу конфігурацію `session.parentForkMaxTokens` видаляє `openclaw doctor --fix`.

Деталь реалізації: рішення ухвалюється в `initSessionState()` у `src/auto-reply/reply/session.ts`.

## Схема сховища сесій (`sessions.json`)

Тип значення сховища — `SessionEntry` у `src/config/sessions.ts`.

Ключові поля (не вичерпно):

- `sessionId`: поточний ідентифікатор стенограми (назва файлу виводиться з нього, якщо не задано `sessionFile`)
- `sessionStartedAt`: часова позначка початку для поточного `sessionId`; актуальність щоденного скидання
  використовує її. Застарілі рядки можуть виводити її із заголовка сесії JSONL.
- `lastInteractionAt`: часова позначка останньої реальної взаємодії користувача/каналу; актуальність скидання через бездіяльність
  використовує її, тому heartbeat, cron і події exec не підтримують сесії
  активними. Застарілі рядки без цього поля для актуальності бездіяльності повертаються до відновленого часу початку
  сесії.
- `updatedAt`: часова позначка останньої мутації рядка сховища, використовується для списків, очищення та
  службового обліку. Вона не є джерелом істини для актуальності щоденного/бездіяльного скидання.
- `sessionFile`: необов’язкове явне перевизначення шляху стенограми
- `chatType`: `direct | group | room` (допомагає UI та політиці надсилання)
- `provider`, `subject`, `room`, `space`, `displayName`: метадані для маркування груп/каналів
- Перемикачі:
  - `thinkingLevel`, `verboseLevel`, `reasoningLevel`, `elevatedLevel`
  - `sendPolicy` (перевизначення для окремої сесії)
- Вибір моделі:
  - `providerOverride`, `modelOverride`, `authProfileOverride`
- Лічильники токенів (best-effort / залежать від провайдера):
  - `inputTokens`, `outputTokens`, `totalTokens`, `contextTokens`
- `compactionCount`: як часто автоматична Compaction завершувалась для цього ключа сесії
- `memoryFlushAt`: часова позначка останнього скидання пам’яті перед Compaction
- `memoryFlushCompactionCount`: кількість Compaction, коли востаннє запускалося скидання

Сховище безпечно редагувати, але Gateway є джерелом істини: він може переписувати або повторно гідратувати записи під час роботи сесій.

## Структура стенограми (`*.jsonl`)

Стенограмами керує `SessionManager` з `@mariozechner/pi-coding-agent`.

Файл має формат JSONL:

- Перший рядок: заголовок сесії (`type: "session"`, містить `id`, `cwd`, `timestamp`, необов’язковий `parentSession`)
- Далі: записи сесії з `id` + `parentId` (дерево)

Важливі типи записів:

- `message`: повідомлення користувача/асистента/toolResult
- `custom_message`: повідомлення, вставлені розширенням, які _потрапляють_ у контекст моделі (можуть бути приховані від UI)
- `custom`: стан розширення, який _не потрапляє_ у контекст моделі
- `compaction`: збережений підсумок Compaction із `firstKeptEntryId` і `tokensBefore`
- `branch_summary`: збережений підсумок під час навігації гілкою дерева

OpenClaw навмисно **не** «виправляє» стенограми; Gateway використовує `SessionManager` для їх читання/запису.

## Контекстні вікна проти відстежуваних токенів

Важливі два різні поняття:

1. **Контекстне вікно моделі**: жорстке обмеження для кожної моделі (токени, видимі моделі)
2. **Лічильники сховища сесій**: ковзна статистика, записана в `sessions.json` (використовується для /status і панелей)

Якщо ви налаштовуєте обмеження:

- Контекстне вікно походить із каталогу моделей (і може бути перевизначене через конфігурацію).
- `contextTokens` у сховищі — це оцінка/звітне значення під час виконання; не вважайте його суворою гарантією.

Докладніше див. [/token-use](/uk/reference/token-use).

## Compaction: що це таке

Compaction підсумовує старішу розмову в збережений запис `compaction` у стенограмі та залишає останні повідомлення без змін.

Після Compaction майбутні ходи бачать:

- Підсумок Compaction
- Повідомлення після `firstKeptEntryId`

Compaction є **сталою** (на відміну від очищення сесій). Див. [/concepts/session-pruning](/uk/concepts/session-pruning).

## Межі фрагментів Compaction і поєднання інструментів

Коли OpenClaw розбиває довгу стенограму на фрагменти Compaction, він зберігає
виклики інструментів асистента в парі з відповідними записами `toolResult`.

- Якщо поділ за часткою токенів припадає між викликом інструмента та його результатом, OpenClaw
  зміщує межу до повідомлення асистента з викликом інструмента замість того, щоб розділяти
  цю пару.
- Якщо кінцевий блок результату інструмента інакше виштовхнув би фрагмент за цільовий розмір,
  OpenClaw зберігає цей очікуваний блок інструмента й залишає нескорочений хвіст
  неушкодженим.
- Перервані/помилкові блоки виклику інструмента не утримують очікуваний поділ відкритим.

## Коли відбувається автоматичне ущільнення (середовище виконання Pi)

У вбудованому агенті Pi автоматичне ущільнення запускається у двох випадках:

1. **Відновлення після переповнення**: модель повертає помилку переповнення контексту
   (`request_too_large`, `context length exceeded`, `input exceeds the maximum
number of tokens`, `input token count exceeds the maximum number of input
tokens`, `input is too long for the model`, `ollama error: context length
exceeded` та подібні варіанти, сформовані провайдерами) → ущільнити → повторити.
2. **Підтримання порога**: після успішного ходу, коли:

`contextTokens > contextWindow - reserveTokens`

Де:

- `contextWindow` — контекстне вікно моделі
- `reserveTokens` — запас, зарезервований для промптів + наступного виводу моделі

Це семантика середовища виконання Pi (OpenClaw споживає події, але Pi вирішує, коли виконувати ущільнення).

OpenClaw також може запускати попереднє локальне ущільнення перед відкриттям наступного
запуску, коли задано `agents.defaults.compaction.maxActiveTranscriptBytes` і
активний файл транскрипту досягає цього розміру. Це обмежувач розміру файлу для вартості
локального повторного відкриття, а не сире архівування: OpenClaw усе одно запускає звичайне семантичне ущільнення,
і для цього потрібен `truncateAfterCompaction`, щоб ущільнений підсумок міг стати
новим наступним транскриптом.

Для вбудованих запусків Pi `agents.defaults.compaction.midTurnPrecheck.enabled: true`
додає опційний захист циклу інструментів. Після додавання результату інструмента й перед
наступним викликом моделі OpenClaw оцінює навантаження на промпт за тією самою логікою
попереднього бюджету, що використовується на початку ходу. Якщо контекст більше не вміщується, захист
не виконує ущільнення всередині хука Pi `transformContext`. Він піднімає структурований
сигнал попередньої перевірки посеред ходу, зупиняє поточне подання промпта й дає
зовнішньому циклу запуску використати наявний шлях відновлення: обрізати завеликі результати інструментів,
коли цього достатньо, або запустити налаштований режим ущільнення й повторити. Ця
опція вимкнена за замовчуванням і працює з режимами ущільнення `default` і `safeguard`,
зокрема з ущільненням safeguard на основі провайдера.
Це не залежить від `maxActiveTranscriptBytes`: обмежувач за розміром у байтах запускається
перед відкриттям ходу, а попередня перевірка посеред ходу запускається пізніше у вбудованому циклі інструментів Pi
після додавання нових результатів інструментів.

## Налаштування ущільнення (`reserveTokens`, `keepRecentTokens`)

Налаштування ущільнення Pi містяться в налаштуваннях Pi:

```json5
{
  compaction: {
    enabled: true,
    reserveTokens: 16384,
    keepRecentTokens: 20000,
  },
}
```

OpenClaw також застосовує мінімальний безпечний поріг для вбудованих запусків:

- Якщо `compaction.reserveTokens < reserveTokensFloor`, OpenClaw підвищує його.
- Стандартний мінімум — `20000` токенів.
- Задайте `agents.defaults.compaction.reserveTokensFloor: 0`, щоб вимкнути цей мінімум.
- Якщо він уже вищий, OpenClaw залишає його без змін.
- Ручний `/compact` враховує явне `agents.defaults.compaction.keepRecentTokens`
  і зберігає точку відсікання нещодавнього хвоста Pi. Без явного бюджету збереження
  ручне ущільнення залишається жорсткою контрольною точкою, а перебудований контекст починається з
  нового підсумку.
- Задайте `agents.defaults.compaction.midTurnPrecheck.enabled: true`, щоб запускати
  опційну попередню перевірку циклу інструментів після нових результатів інструментів і перед наступним викликом
  моделі. Це лише тригер; створення підсумку все одно використовує налаштований
  шлях ущільнення. Він не залежить від `maxActiveTranscriptBytes`, який є
  обмежувачем розміру активного транскрипту в байтах на початку ходу.
- Задайте `agents.defaults.compaction.maxActiveTranscriptBytes` як значення в байтах або
  рядок на кшталт `"20mb"`, щоб запускати локальне ущільнення перед ходом, коли активний
  транскрипт стає великим. Цей захист активний лише коли
  `truncateAfterCompaction` також увімкнено. Не задавайте його або задайте `0`, щоб
  вимкнути.
- Коли `agents.defaults.compaction.truncateAfterCompaction` увімкнено,
  OpenClaw після ущільнення ротуватиме активний транскрипт у ущільнений наступний JSONL.
  Старий повний транскрипт лишається в архіві й посилається з контрольної точки
  ущільнення, а не переписується на місці.

Причина: залишити достатній запас для багатоходового “обслуговування” (наприклад, записів пам’яті) до того, як ущільнення стане неминучим.

Реалізація: `ensurePiCompactionReserveTokens()` у `src/agents/pi-settings.ts`
(викликається з `src/agents/pi-embedded-runner.ts`).

## Підключувані провайдери ущільнення

Плагіни можуть реєструвати провайдера ущільнення через `registerCompactionProvider()` в API плагіна. Коли `agents.defaults.compaction.provider` задано як ідентифікатор зареєстрованого провайдера, розширення safeguard делегує підсумовування цьому провайдеру замість вбудованого конвеєра `summarizeInStages`.

- `provider`: ідентифікатор зареєстрованого плагіна провайдера ущільнення. Не задавайте, щоб використовувати стандартне LLM-підсумовування.
- Задання `provider` примусово встановлює `mode: "safeguard"`.
- Провайдери отримують ті самі інструкції з ущільнення та політику збереження ідентифікаторів, що й вбудований шлях.
- Safeguard усе одно зберігає контекст суфікса нещодавнього ходу та розділеного ходу після виводу провайдера.
- Вбудоване підсумовування safeguard повторно дистилює попередні підсумки з новими повідомленнями
  замість дослівного збереження повного попереднього підсумку.
- Режим Safeguard за замовчуванням вмикає аудити якості підсумків; задайте
  `qualityGuard.enabled: false`, щоб пропустити поведінку повтору в разі некоректно сформованого виводу.
- Якщо провайдер зазнає невдачі або повертає порожній результат, OpenClaw автоматично повертається до вбудованого LLM-підсумовування.
- Сигнали переривання/тайм-ауту повторно пробрасываются (не поглинаються), щоб поважати скасування викликача.

Джерело: `src/plugins/compaction-provider.ts`, `src/agents/pi-hooks/compaction-safeguard.ts`.

## Тихе обслуговування (`NO_REPLY`)

OpenClaw підтримує “тихі” ходи для фонових завдань, де користувач не має бачити проміжний вивід.

Угода:

- Асистент починає свій вивід з точного тихого токена `NO_REPLY` /
  `no_reply`, щоб позначити “не доставляти відповідь користувачу”.
- OpenClaw видаляє/пригнічує це на рівні доставки.
- Пригнічення точного тихого токена нечутливе до регістру, тому `NO_REPLY` і
  `no_reply` обидва зараховуються, коли весь payload є лише тихим токеном.
- Це лише для справді фонових ходів без доставки; це не скорочення для
  звичайних дієвих запитів користувача.

Станом на `2026.1.10`, OpenClaw також пригнічує **потокову передачу чернетки/набору** коли
частковий фрагмент починається з `NO_REPLY`, тому тихі операції не витікають частковим
виводом посеред ходу.

## Попереднє перед ущільненням "скидання пам’яті" (реалізовано)

Мета: перед автоматичним ущільненням запустити тихий агентний хід, який записує довговічний
стан на диск (наприклад, `memory/YYYY-MM-DD.md` у робочій області агента), щоб ущільнення не могло
стерти критичний контекст.

OpenClaw використовує підхід **скидання перед порогом**:

1. Відстежувати використання контексту сесії.
2. Коли воно перетинає “м’який поріг” (нижче порога ущільнення Pi), запустити тиху
   директиву “запиши пам’ять зараз” для агента.
3. Використати точний тихий токен `NO_REPLY` / `no_reply`, щоб користувач нічого
   не бачив.

Конфігурація (`agents.defaults.compaction.memoryFlush`):

- `enabled` (за замовчуванням: `true`)
- `model` (опційне точне перевизначення провайдера/моделі для ходу скидання, наприклад `ollama/qwen3:8b`)
- `softThresholdTokens` (за замовчуванням: `4000`)
- `prompt` (повідомлення користувача для ходу скидання)
- `systemPrompt` (додатковий системний промпт, доданий для ходу скидання)

Примітки:

- Стандартні prompt/system prompt містять підказку `NO_REPLY`, щоб пригнітити
  доставку.
- Коли `model` задано, хід скидання використовує цю модель без успадкування
  ланцюга fallback активної сесії, тому локальне лише обслуговування не буде непомітно
  переходити на платну розмовну модель.
- Скидання запускається один раз за цикл ущільнення (відстежується в `sessions.json`).
- Скидання запускається лише для вбудованих сесій Pi (CLI-бекенди пропускають його).
- Скидання пропускається, коли робоча область сесії доступна лише для читання (`workspaceAccess: "ro"` або `"none"`).
- Див. [Пам’ять](/uk/concepts/memory) щодо структури файлів робочої області та шаблонів запису.

Pi також надає хук `session_before_compact` в API розширень, але логіка скидання OpenClaw
сьогодні живе на боці Gateway.

## Контрольний список усунення несправностей

- Неправильний ключ сесії? Почніть з [/concepts/session](/uk/concepts/session) і підтвердьте `sessionKey` у `/status`.
- Невідповідність сховища й транскрипту? Підтвердьте хост Gateway і шлях до сховища з `openclaw status`.
- Спам ущільнення? Перевірте:
  - контекстне вікно моделі (занадто мале)
  - налаштування ущільнення (`reserveTokens` завеликий для вікна моделі може спричиняти раніше ущільнення)
  - роздування результатів інструментів: увімкніть/налаштуйте обрізання сесії
- Тихі ходи витікають? Переконайтеся, що відповідь починається з `NO_REPLY` (точний токен без урахування регістру) і ви використовуєте збірку, яка містить виправлення пригнічення потокової передачі.
