Кешування промптів означає, що провайдер моделі може повторно використовувати незмінені префікси промптів (зазвичай системні/розробницькі інструкції та інший стабільний контекст) між ходами замість повторної обробки їх щоразу. OpenClaw нормалізує використання провайдера в `cacheRead` і `cacheWrite`, коли upstream API напряму надає ці лічильники.

Поверхні статусу також можуть відновлювати лічильники кешу з найновішого журналу
використання transcript, коли їх немає в live-знімку сесії, тож `/status` може й надалі
показувати рядок кешу після часткової втрати метаданих сесії. Наявні ненульові live
значення кешу все ще мають пріоритет над резервними значеннями з transcript.

Чому це важливо: нижча вартість токенів, швидші відповіді та передбачуваніша продуктивність для довготривалих сесій. Без кешування повторювані промпти оплачують повну вартість промпта на кожному ході, навіть коли більшість вхідних даних не змінилася.

Розділи нижче охоплюють усі пов’язані з кешем параметри, що впливають на повторне використання промптів і вартість токенів.

Посилання на провайдерів:

- Кешування промптів Anthropic: [https://platform.claude.com/docs/en/build-with-claude/prompt-caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)
- Кешування промптів OpenAI: [https://developers.openai.com/api/docs/guides/prompt-caching](https://developers.openai.com/api/docs/guides/prompt-caching)
- Заголовки OpenAI API та ідентифікатори запитів: [https://developers.openai.com/api/reference/overview](https://developers.openai.com/api/reference/overview)
- Ідентифікатори запитів і помилки Anthropic: [https://platform.claude.com/docs/en/api/errors](https://platform.claude.com/docs/en/api/errors)

## Основні параметри

### `cacheRetention` (глобальне значення за замовчуванням, модель і окремий агент)

Задайте збереження кешу як глобальне значення за замовчуванням для всіх моделей:

```yaml
agents:
  defaults:
    params:
      cacheRetention: "long" # none | short | long
```

Перевизначення для окремої моделі:

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "short" # none | short | long
```

Перевизначення для окремого агента:

```yaml
agents:
  list:
    - id: "alerts"
      params:
        cacheRetention: "none"
```

Порядок злиття конфігурації:

1. `agents.defaults.params` (глобальне значення за замовчуванням — застосовується до всіх моделей)
2. `agents.defaults.models["provider/model"].params` (перевизначення для окремої моделі)
3. `agents.list[].params` (відповідний id агента; перевизначає за ключем)

### `contextPruning.mode: "cache-ttl"`

Обрізає старий контекст результатів інструментів після вікон TTL кешу, щоб запити після простою не кешували повторно надмірно велику історію.

```yaml
agents:
  defaults:
    contextPruning:
      mode: "cache-ttl"
      ttl: "1h"
```

Повну поведінку див. у [Обрізання сесії](/uk/concepts/session-pruning).

### Підігрів через Heartbeat

Heartbeat може підтримувати вікна кешу теплими та зменшувати повторні записи в кеш після періодів простою.

```yaml
agents:
  defaults:
    heartbeat:
      every: "55m"
```

Heartbeat для окремого агента підтримується в `agents.list[].heartbeat`.

## Поведінка провайдерів

### Anthropic (прямий API)

- `cacheRetention` підтримується.
- Для профілів автентифікації Anthropic API key OpenClaw задає `cacheRetention: "short"` для посилань на моделі Anthropic, якщо значення не встановлено.
- Нативні відповіді Anthropic Messages надають і `cache_read_input_tokens`, і `cache_creation_input_tokens`, тому OpenClaw може показувати і `cacheRead`, і `cacheWrite`.
- Для нативних запитів Anthropic `cacheRetention: "short"` відповідає типовому 5-хвилинному ефемерному кешу, а `cacheRetention: "long"` підвищує TTL до 1 години лише на прямих хостах `api.anthropic.com`.

### OpenAI (прямий API)

- Кешування промптів автоматичне на підтримуваних сучасних моделях. OpenClaw не потрібно вставляти маркери кешу на рівні блоків.
- OpenClaw використовує `prompt_cache_key`, щоб маршрутизація кешу залишалася стабільною між ходами. Прямі хости OpenAI використовують `prompt_cache_retention: "24h"`, коли вибрано `cacheRetention: "long"`.
- OpenAI-сумісні провайдери Completions отримують `prompt_cache_key` лише тоді, коли їхня конфігурація моделі явно задає `compat.supportsPromptCacheKey: true`. Передавання довгого збереження є окремою можливістю: явне `cacheRetention: "long"` надсилає `prompt_cache_retention: "24h"` лише тоді, коли цей запис compat також підтримує довге збереження кешу. Такі провайдери, як Mistral, можуть увімкнути ключі кешу, встановивши `compat.supportsLongCacheRetention: false`, щоб придушити поле довгого збереження. `cacheRetention: "none"` придушує обидва поля.
- Відповіді OpenAI надають кешовані токени промпта через `usage.prompt_tokens_details.cached_tokens` (або `input_tokens_details.cached_tokens` у подіях Responses API). OpenClaw відображає це як `cacheRead`.
- OpenAI не надає окремий лічильник токенів запису кешу, тому `cacheWrite` залишається `0` на шляхах OpenAI, навіть коли провайдер прогріває кеш.
- OpenAI повертає корисні заголовки трасування та обмеження частоти, як-от `x-request-id`, `openai-processing-ms` і `x-ratelimit-*`, але облік cache hit має надходити з payload використання, а не із заголовків.
- На практиці OpenAI часто поводиться як кеш початкового префікса, а не як Anthropic-стиль повторного використання всієї історії, що рухається. Стабільні ходи з довгим префіксним текстом можуть досягати плато приблизно `4864` кешованих токенів у поточних live-пробах, тоді як transcript із великою кількістю інструментів або MCP-стилем часто виходять на плато близько `4608` кешованих токенів навіть за точних повторів.

### Anthropic Vertex

- Моделі Anthropic на Vertex AI (`anthropic-vertex/*`) підтримують `cacheRetention` так само, як прямий Anthropic.
- `cacheRetention: "long"` відповідає реальному 1-годинному TTL кешу промптів на endpoint Vertex AI.
- Типове збереження кешу для `anthropic-vertex` збігається з типовими значеннями прямого Anthropic.
- Запити Vertex маршрутизуються через формування кешу з урахуванням меж, щоб повторне використання кешу залишалося узгодженим із тим, що провайдери фактично отримують.

### Amazon Bedrock

- Посилання на моделі Anthropic Claude (`amazon-bedrock/*anthropic.claude*`) підтримують явне наскрізне передавання `cacheRetention`.
- Не-Anthropic моделі Bedrock примусово отримують `cacheRetention: "none"` під час виконання.

### Моделі OpenRouter

Для посилань на моделі `openrouter/anthropic/*` OpenClaw вставляє Anthropic
`cache_control` у системні/розробницькі блоки промпта, щоб покращити повторне
використання кешу промптів лише тоді, коли запит усе ще націлений на перевірений маршрут OpenRouter
(`openrouter` на типовому endpoint або будь-який provider/base URL, що резолвиться
до `openrouter.ai`).

Для посилань на моделі `openrouter/deepseek/*`, `openrouter/moonshot*/*` і `openrouter/zai/*`
дозволено `contextPruning.mode: "cache-ttl"`, оскільки OpenRouter
автоматично обробляє кешування промптів на стороні провайдера. OpenClaw не вставляє
Anthropic-маркери `cache_control` у ці запити.

Побудова кешу DeepSeek працює за принципом best-effort і може тривати кілька секунд. Негайний
наступний запит усе ще може показати `cached_tokens: 0`; перевіряйте повторним
запитом із тим самим префіксом після короткої затримки та використовуйте `usage.prompt_tokens_details.cached_tokens`
як сигнал cache hit.

Якщо ви переналаштуєте модель на довільний OpenAI-сумісний proxy URL, OpenClaw
припинить вставляти ці специфічні для OpenRouter Anthropic-маркери кешу.

### Інші провайдери

Якщо провайдер не підтримує цей режим кешу, `cacheRetention` не має ефекту.

### Прямий API Google Gemini

- Прямий транспорт Gemini (`api: "google-generative-ai"`) повідомляє cache hit
  через upstream `cachedContentTokenCount`; OpenClaw відображає це як `cacheRead`.
- Коли `cacheRetention` задано для прямої моделі Gemini, OpenClaw автоматично
  створює, повторно використовує та оновлює ресурси `cachedContents` для системних промптів
  під час запусків Google AI Studio. Це означає, що більше не потрібно попередньо створювати
  handle cached-content вручну.
- Ви все ще можете передати наявний Gemini handle cached-content як
  `params.cachedContent` (або legacy `params.cached_content`) у налаштованій
  моделі.
- Це окремо від кешування префіксів промптів Anthropic/OpenAI. Для Gemini
  OpenClaw керує provider-native ресурсом `cachedContents`, а не
  вставляє маркери кешу в запит.

### Використання Gemini CLI

- Вивід Gemini CLI `stream-json` може показувати cache hit через `stats.cached`;
  OpenClaw відображає це як `cacheRead`. Legacy-перевизначення `--output-format json` використовують
  ту саму нормалізацію використання.
- Якщо CLI не надає прямого значення `stats.input`, OpenClaw виводить вхідні токени
  з `stats.input_tokens - stats.cached`.
- Це лише нормалізація використання. Це не означає, що OpenClaw створює
  Anthropic/OpenAI-стиль маркерів кешу промптів для Gemini CLI.

## Межа кешу системного промпта

OpenClaw розділяє системний промпт на **стабільний префікс** і **мінливий
суфікс**, розділені внутрішньою межею префікса кешу. Контент над
межею (визначення інструментів, метадані Skills, файли workspace та інший
відносно статичний контекст) упорядковується так, щоб залишатися byte-identical між ходами.
Контент нижче межі (наприклад `HEARTBEAT.md`, часові мітки runtime та
інші метадані окремого ходу) може змінюватися без інвалідації кешованого
префікса.

Ключові проєктні рішення:

- Стабільні файли project-context workspace упорядковуються перед `HEARTBEAT.md`, щоб
  зміни Heartbeat не руйнували стабільний префікс.
- Межа застосовується до формування транспорту сімейств Anthropic, OpenAI, Google і
  CLI, щоб усі підтримувані провайдери отримували користь від тієї самої стабільності префікса.
- Запити Codex Responses і Anthropic Vertex маршрутизуються через
  формування кешу з урахуванням меж, щоб повторне використання кешу залишалося узгодженим із тим, що провайдери
  фактично отримують.
- Відбитки системного промпта нормалізуються (пробіли, закінчення рядків,
  контекст, доданий hook, упорядкування runtime capabilities), щоб семантично незмінені
  промпти спільно використовували KV/кеш між ходами.

Якщо ви бачите неочікувані сплески `cacheWrite` після зміни конфігурації або workspace,
перевірте, чи зміна потрапляє вище або нижче межі кешу. Переміщення
мінливого контенту нижче межі (або його стабілізація) часто розв’язує
проблему.

## Запобіжники стабільності кешу OpenClaw

OpenClaw також зберігає кілька чутливих до кешу форм payload детермінованими до того,
як запит доходить до провайдера:

- Каталоги інструментів Bundle MCP сортуються детерміновано перед
  реєстрацією інструментів, тож зміни порядку `listTools()` не змінюють блок інструментів і
  не руйнують префікси кешу промпта.
- Legacy-сесії зі збереженими блоками зображень залишають **3 найновіші
  завершені ходи** неушкодженими; старіші вже оброблені блоки зображень можуть бути
  замінені маркером, щоб подальші запити з великою кількістю зображень не надсилали повторно великі
  застарілі payload.

## Патерни налаштування

### Змішаний трафік (рекомендоване значення за замовчуванням)

Тримайте довготривалу базову лінію на головному агенті, вимкніть кешування для bursty агентів-сповіщувачів:

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m"
    - id: "alerts"
      params:
        cacheRetention: "none"
```

### Базова лінія з пріоритетом вартості

- Задайте базове `cacheRetention: "short"`.
- Увімкніть `contextPruning.mode: "cache-ttl"`.
- Тримайте Heartbeat нижче вашого TTL лише для агентів, яким корисні теплі кеші.

## Діагностика кешу

OpenClaw надає спеціалізовану діагностику cache-trace для вбудованих запусків агентів.

Для звичайної діагностики, видимої користувачу, `/status` та інші підсумки використання можуть використовувати
найновіший запис використання transcript як резервне джерело для `cacheRead` /
`cacheWrite`, коли live-запис сесії не має цих лічильників.

## Live-регресійні тести

OpenClaw має один об’єднаний live-регресійний gate кешу для повторюваних префіксів, ходів інструментів, ходів із зображеннями, transcript інструментів MCP-стилю та Anthropic no-cache control.

- `src/agents/live-cache-regression.live.test.ts`
- `src/agents/live-cache-regression-baseline.ts`

Запустіть вузький live gate так:

```sh
OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache
```

Файл baseline зберігає найновіші спостережені live-числа плюс provider-specific regression floors, які використовує тест.
Runner також використовує свіжі для кожного запуску ідентифікатори сесій і простори імен промптів, щоб попередній стан кешу не забруднював поточний regression sample.

Ці тести навмисно не використовують однакові критерії успіху для всіх постачальників.

### Anthropic live очікування

- Очікуйте явних warmup-записів через `cacheWrite`.
- Очікуйте майже повного повторного використання історії під час повторних ходів, бо керування кешем Anthropic просуває точку розриву кешу крізь розмову.
- Поточні live-перевірки все ще використовують високі пороги частки влучань для стабільних шляхів, інструментів і зображень.

### OpenAI live очікування

- Очікуйте лише `cacheRead`. `cacheWrite` залишається `0`.
- Розглядайте повторне використання кешу під час повторних ходів як специфічне для постачальника плато, а не як рухоме повторне використання повної історії у стилі Anthropic.
- Поточні live-перевірки використовують консервативні мінімальні перевірки, виведені зі спостережуваної live-поведінки на `gpt-5.4-mini`:
  - стабільний префікс: `cacheRead >= 4608`, частка влучань `>= 0.90`
  - стенограма інструментів: `cacheRead >= 4096`, частка влучань `>= 0.85`
  - стенограма зображень: `cacheRead >= 3840`, частка влучань `>= 0.82`
  - стенограма у стилі MCP: `cacheRead >= 4096`, частка влучань `>= 0.85`

Свіжа об’єднана live-перевірка 2026-04-04 дала:

- стабільний префікс: `cacheRead=4864`, частка влучань `0.966`
- стенограма інструментів: `cacheRead=4608`, частка влучань `0.896`
- стенограма зображень: `cacheRead=4864`, частка влучань `0.954`
- стенограма у стилі MCP: `cacheRead=4608`, частка влучань `0.891`

Нещодавній локальний реальний час виконання для об’єднаної перевірки становив приблизно `88s`.

Чому перевірки відрізняються:

- Anthropic надає явні точки розриву кешу й рухоме повторне використання історії розмови.
- Кешування підказок OpenAI досі чутливе до точного префікса, але ефективний повторно використовуваний префікс у live-трафіку Responses може виходити на плато раніше, ніж повна підказка.
- Через це порівняння Anthropic і OpenAI за єдиним міжпостачальницьким відсотковим порогом створює хибні регресії.

### Конфігурація `diagnostics.cacheTrace`

```yaml
diagnostics:
  cacheTrace:
    enabled: true
    filePath: "~/.openclaw/logs/cache-trace.jsonl" # optional
    includeMessages: false # default true
    includePrompt: false # default true
    includeSystem: false # default true
```

Типові значення:

- `filePath`: `$OPENCLAW_STATE_DIR/logs/cache-trace.jsonl`
- `includeMessages`: `true`
- `includePrompt`: `true`
- `includeSystem`: `true`

### Перемикачі середовища (одноразове налагодження)

- `OPENCLAW_CACHE_TRACE=1` вмикає трасування кешу.
- `OPENCLAW_CACHE_TRACE_FILE=/path/to/cache-trace.jsonl` перевизначає шлях виводу.
- `OPENCLAW_CACHE_TRACE_MESSAGES=0|1` перемикає захоплення повного вмісту повідомлень.
- `OPENCLAW_CACHE_TRACE_PROMPT=0|1` перемикає захоплення тексту підказки.
- `OPENCLAW_CACHE_TRACE_SYSTEM=0|1` перемикає захоплення системної підказки.

### Що перевіряти

- Події трасування кешу мають формат JSONL і містять поетапні знімки на кшталт `session:loaded`, `prompt:before`, `stream:context` і `session:after`.
- Вплив токенів кешу за хід видно у звичайних поверхнях використання через `cacheRead` і `cacheWrite` (наприклад, `/usage full` і підсумки використання сеансу).
- Для Anthropic очікуйте і `cacheRead`, і `cacheWrite`, коли кешування активне.
- Для OpenAI очікуйте `cacheRead` під час влучань у кеш, а `cacheWrite` має залишатися `0`; OpenAI не публікує окреме поле токенів запису в кеш.
- Якщо вам потрібне трасування запитів, журналюйте ідентифікатори запитів і заголовки обмеження частоти окремо від метрик кешу. Поточний вивід трасування кешу OpenClaw зосереджений на формі підказки/сеансу та нормалізованому використанні токенів, а не на сирих заголовках відповідей постачальника.

## Швидке усунення несправностей

- Високий `cacheWrite` на більшості ходів: перевірте мінливі вхідні дані системної підказки й переконайтеся, що модель/постачальник підтримує ваші налаштування кешу.
- Високий `cacheWrite` на Anthropic: часто означає, що точка розриву кешу потрапляє на вміст, який змінюється в кожному запиті.
- Низький OpenAI `cacheRead`: переконайтеся, що стабільний префікс розташований на початку, повторюваний префікс має щонайменше 1024 токени, а той самий `prompt_cache_key` повторно використовується для ходів, які мають спільно використовувати кеш.
- Немає ефекту від `cacheRetention`: підтвердьте, що ключ моделі відповідає `agents.defaults.models["provider/model"]`.
- Запити Bedrock Nova/Mistral з налаштуваннями кешу: очікуване примусове встановлення runtime у `none`.
