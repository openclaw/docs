Кешування промптів означає, що провайдер моделі може повторно використовувати незмінені префікси промптів (зазвичай системні/розробницькі інструкції та інший стабільний контекст) між зверненнями замість повторної обробки щоразу. OpenClaw нормалізує використання провайдера в `cacheRead` і `cacheWrite`, коли upstream API напряму надає ці лічильники.

Поверхні статусу також можуть відновлювати лічильники кешу з останнього журналу
використання в транскрипті, коли вони відсутні в актуальному знімку сесії, тож `/status` може й надалі
показувати рядок кешу після часткової втрати метаданих сесії. Наявні ненульові актуальні
значення кешу все одно мають пріоритет над fallback-значеннями з транскрипту.

Чому це важливо: нижча вартість токенів, швидші відповіді та більш передбачувана продуктивність для довготривалих сесій. Без кешування повторювані промпти оплачують повну вартість промпту на кожному зверненні, навіть коли більшість вхідних даних не змінилася.

Розділи нижче охоплюють усі пов’язані з кешем параметри, які впливають на повторне використання промптів і вартість токенів.

Посилання на провайдерів:

- Кешування промптів Anthropic: [https://platform.claude.com/docs/en/build-with-claude/prompt-caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)
- Кешування промптів OpenAI: [https://developers.openai.com/api/docs/guides/prompt-caching](https://developers.openai.com/api/docs/guides/prompt-caching)
- Заголовки OpenAI API та ідентифікатори запитів: [https://developers.openai.com/api/reference/overview](https://developers.openai.com/api/reference/overview)
- Ідентифікатори запитів Anthropic і помилки: [https://platform.claude.com/docs/en/api/errors](https://platform.claude.com/docs/en/api/errors)

## Основні параметри

### `cacheRetention` (глобальне значення за замовчуванням, модель і окремий агент)

Задайте утримання кешу як глобальне значення за замовчуванням для всіх моделей:

```yaml
agents:
  defaults:
    params:
      cacheRetention: "long" # none | short | long
```

Перевизначення для окремої моделі:

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "short" # none | short | long
```

Перевизначення для окремого агента:

```yaml
agents:
  list:
    - id: "alerts"
      params:
        cacheRetention: "none"
```

Порядок об’єднання конфігурації:

1. `agents.defaults.params` (глобальне значення за замовчуванням — застосовується до всіх моделей)
2. `agents.defaults.models["provider/model"].params` (перевизначення для окремої моделі)
3. `agents.list[].params` (відповідний ідентифікатор агента; перевизначає за ключем)

### `contextPruning.mode: "cache-ttl"`

Обрізає старий контекст результатів інструментів після вікон TTL кешу, щоб запити після простою не кешували повторно надмірно велику історію.

```yaml
agents:
  defaults:
    contextPruning:
      mode: "cache-ttl"
      ttl: "1h"
```

Повну поведінку див. у [Обрізанні сесії](/uk/concepts/session-pruning).

### Підтримання тепла через Heartbeat

Heartbeat може підтримувати теплими вікна кешу та зменшувати повторні записи кешу після періодів простою.

```yaml
agents:
  defaults:
    heartbeat:
      every: "55m"
```

Heartbeat для окремого агента підтримується в `agents.list[].heartbeat`.

## Поведінка провайдерів

### Anthropic (прямий API)

- `cacheRetention` підтримується.
- Для профілів автентифікації Anthropic з API-ключем OpenClaw задає `cacheRetention: "short"` для посилань на моделі Anthropic, якщо значення не встановлено.
- Нативні відповіді Messages Anthropic надають і `cache_read_input_tokens`, і `cache_creation_input_tokens`, тож OpenClaw може показувати як `cacheRead`, так і `cacheWrite`.
- Для нативних запитів Anthropic `cacheRetention: "short"` відповідає стандартному 5-хвилинному ефемерному кешу, а `cacheRetention: "long"` підвищує TTL до 1 години лише на прямих хостах `api.anthropic.com`.

### OpenAI (прямий API)

- Кешування промптів автоматичне для підтримуваних нових моделей. OpenClaw не потрібно вставляти маркери кешу на рівні блоків.
- OpenClaw використовує `prompt_cache_key`, щоб маршрутизація кешу залишалася стабільною між зверненнями. Прямі хости OpenAI використовують `prompt_cache_retention: "24h"`, коли вибрано `cacheRetention: "long"`.
- OpenAI-сумісні провайдери Completions отримують `prompt_cache_key` лише тоді, коли їхня конфігурація моделі явно задає `compat.supportsPromptCacheKey: true`. Передавання довгого утримання є окремою можливістю: явне `cacheRetention: "long"` надсилає `prompt_cache_retention: "24h"` лише тоді, коли цей compat-запис також підтримує довге утримання кешу. Провайдери, як-от Mistral, можуть увімкнути кеш-ключі й водночас задати `compat.supportsLongCacheRetention: false`, щоб приглушити поле довгого утримання. `cacheRetention: "none"` приглушує обидва поля.
- Відповіді OpenAI надають кешовані токени промпту через `usage.prompt_tokens_details.cached_tokens` (або `input_tokens_details.cached_tokens` у подіях Responses API). OpenClaw зіставляє це з `cacheRead`.
- Використання GPT-5.6 Responses також може надавати `input_tokens_details.cache_write_tokens`. OpenClaw зіставляє це з `cacheWrite` і тарифікує за ставкою запису кешу моделі; Responses, які не містять цього поля, залишають `cacheWrite` на `0`.
- OpenAI повертає корисні заголовки трасування й обмеження швидкості, як-от `x-request-id`, `openai-processing-ms` і `x-ratelimit-*`, але облік cache-hit має братися з payload використання, а не із заголовків.
- На практиці OpenAI часто поводиться як кеш початкового префікса, а не як повторне використання рухомої повної історії у стилі Anthropic. Стабільні звернення з довгим префіксним текстом можуть виходити на плато близько `4864` кешованих токенів у поточних live-зондуваннях, тоді як насичені інструментами або MCP-подібні транскрипти часто виходять на плато близько `4608` кешованих токенів навіть за точних повторів.

### Anthropic Vertex

- Моделі Anthropic у Vertex AI (`anthropic-vertex/*`) підтримують `cacheRetention` так само, як прямий Anthropic.
- `cacheRetention: "long"` відповідає справжньому 1-годинному TTL кешу промптів на endpoints Vertex AI.
- Стандартне утримання кешу для `anthropic-vertex` відповідає стандартним значенням прямого Anthropic.
- Запити Vertex маршрутизуються через boundary-aware формування кешу, щоб повторне використання кешу залишалося узгодженим із тим, що провайдери фактично отримують.

### Amazon Bedrock

- Посилання на моделі Anthropic Claude (`amazon-bedrock/*anthropic.claude*`) підтримують явне pass-through для `cacheRetention`.
- Для не-Anthropic моделей Bedrock під час виконання примусово встановлюється `cacheRetention: "none"`.

### Моделі OpenRouter

Для посилань на моделі `openrouter/anthropic/*` OpenClaw вставляє Anthropic
`cache_control` у блоки системних/розробницьких промптів, щоб покращити повторне
використання кешу промптів лише тоді, коли запит усе ще націлений на перевірений маршрут OpenRouter
(`openrouter` на його стандартному endpoint або будь-який провайдер/base URL, що resolves
до `openrouter.ai`).

Для посилань на моделі `openrouter/deepseek/*`, `openrouter/moonshot*/*` і `openrouter/zai/*`
дозволено `contextPruning.mode: "cache-ttl"`, тому що OpenRouter
автоматично обробляє кешування промптів на боці провайдера. OpenClaw не вставляє
маркери Anthropic `cache_control` у ці запити.

Побудова кешу DeepSeek виконується best-effort і може тривати кілька секунд. Негайний
наступний запит усе ще може показати `cached_tokens: 0`; перевіряйте повторним
запитом із тим самим префіксом після короткої затримки й використовуйте `usage.prompt_tokens_details.cached_tokens`
як сигнал cache-hit.

Якщо ви перенаправите модель на довільний OpenAI-сумісний proxy URL, OpenClaw
припинить вставляти ці специфічні для OpenRouter маркери кешу Anthropic.

### Інші провайдери

Якщо провайдер не підтримує цей режим кешу, `cacheRetention` не має ефекту.

### Прямий API Google Gemini

- Прямий транспорт Gemini (`api: "google-generative-ai"`) повідомляє cache hits
  через upstream `cachedContentTokenCount`; OpenClaw зіставляє це з `cacheRead`.
- Коли `cacheRetention` задано для прямої моделі Gemini, OpenClaw автоматично
  створює, повторно використовує й оновлює ресурси `cachedContents` для системних промптів
  у запусках Google AI Studio. Це означає, що вам більше не потрібно попередньо створювати
  handle cached-content вручну.
- Ви все ще можете передати наявний handle cached-content Gemini як
  `params.cachedContent` (або legacy `params.cached_content`) у налаштованій
  моделі.
- Це окремо від кешування префіксів промптів Anthropic/OpenAI. Для Gemini
  OpenClaw керує provider-native ресурсом `cachedContents`, а не
  вставляє маркери кешу в запит.

### Використання Gemini CLI

- Вивід Gemini CLI `stream-json` може показувати cache hits через `stats.cached`;
  OpenClaw зіставляє це з `cacheRead`. Legacy перевизначення `--output-format json` використовують
  таку саму нормалізацію використання.
- Якщо CLI пропускає пряме значення `stats.input`, OpenClaw виводить вхідні токени
  з `stats.input_tokens - stats.cached`.
- Це лише нормалізація використання. Це не означає, що OpenClaw створює
  маркери кешу промптів у стилі Anthropic/OpenAI для Gemini CLI.

## Межа кешу системного промпту

OpenClaw розділяє системний промпт на **стабільний префікс** і **мінливий
суфікс**, розділені внутрішньою межею префікса кешу. Вміст над
межею (визначення інструментів, метадані Skills, файли робочого простору та інший
відносно статичний контекст) упорядковується так, щоб залишатися байт-ідентичним між зверненнями.
Вміст під межею (наприклад `HEARTBEAT.md`, часові мітки runtime та
інші метадані окремого звернення) може змінюватися без інвалідації кешованого
префікса.

Ключові проєктні рішення:

- Стабільні файли project-context робочого простору впорядковуються перед `HEARTBEAT.md`, щоб
  зміни heartbeat не ламали стабільний префікс.
- Межа застосовується до формування транспорту Anthropic-family, OpenAI-family, Google і
  CLI, щоб усі підтримувані провайдери отримували користь від тієї самої стабільності префікса.
- Запити Codex Responses і Anthropic Vertex маршрутизуються через
  boundary-aware формування кешу, щоб повторне використання кешу залишалося узгодженим із тим, що провайдери
  фактично отримують.
- Відбитки системних промптів нормалізуються (пробіли, закінчення рядків,
  контекст, доданий hook, порядок runtime-можливостей), щоб семантично незмінені
  промпти спільно використовували KV/кеш між зверненнями.

Якщо ви бачите неочікувані сплески `cacheWrite` після зміни конфігурації або робочого простору,
перевірте, чи ця зміна потрапляє над або під межу кешу. Переміщення
мінливого вмісту під межу (або його стабілізація) часто розв’язує
проблему.

## Захисні механізми стабільності кешу OpenClaw

OpenClaw також зберігає кілька чутливих до кешу форм payload детермінованими до того,
як запит досягне провайдера:

- Каталоги інструментів Bundle MCP сортуються детерміновано перед реєстрацією інструментів,
  тож зміни порядку `listTools()` не створюють churn у блоці інструментів і
  не ламають префікси кешу промптів.
- Legacy сесії зі збереженими блоками зображень залишають **3 найновіші
  завершені звернення** недоторканими; старіші вже оброблені блоки зображень можуть бути
  замінені маркером, щоб image-heavy наступні запити не надсилали повторно великі
  застарілі payload.

## Шаблони налаштування

### Змішаний трафік (рекомендоване значення за замовчуванням)

Тримайте довгоживучий baseline на головному агенті, вимкніть кешування на bursty агентах-сповіщувачах:

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m"
    - id: "alerts"
      params:
        cacheRetention: "none"
```

### Baseline з пріоритетом вартості

- Задайте baseline `cacheRetention: "short"`.
- Увімкніть `contextPruning.mode: "cache-ttl"`.
- Тримайте heartbeat нижче вашого TTL лише для агентів, яким корисні теплі кеші.

## Діагностика кешу

OpenClaw надає спеціальні діагностичні засоби cache-trace для вбудованих запусків агентів.

Для звичайної користувацької діагностики `/status` та інші підсумки використання можуть використовувати
останній запис використання з транскрипту як fallback-джерело для `cacheRead` /
`cacheWrite`, коли актуальний запис сесії не має цих лічильників.

## Live регресійні тести

OpenClaw підтримує один комбінований live regression gate для повторюваних префіксів, звернень з інструментами, звернень із зображеннями, MCP-подібних транскриптів інструментів і Anthropic no-cache control.

- `src/agents/live-cache-regression.live.test.ts`
- `src/agents/live-cache-regression-baseline.ts`

Запустіть вузький live gate за допомогою:

```sh
OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache
```

Базовий файл зберігає найновіші спостережені live-числа, а також специфічні для провайдера нижні межі регресії, які використовує тест.
Runner також використовує свіжі ідентифікатори сесій і простори імен промптів для кожного запуску, щоб попередній стан кешу не забруднював поточний регресійний зразок.

Ці тести навмисно не використовують однакові критерії успіху для всіх провайдерів.

### Live-очікування Anthropic

- Очікуйте явні warmup-записи через `cacheWrite`.
- Очікуйте майже повне повторне використання історії на повторних ходах, бо cache control Anthropic просуває контрольну точку кешу через розмову.
- Поточні live-твердження все ще використовують високі пороги частки влучань для стабільних шляхів, шляхів інструментів і зображень.

### Live-очікування OpenAI

- Очікуйте лише `cacheRead`. `cacheWrite` залишається `0`.
- Розглядайте повторне використання кешу на повторних ходах як специфічне для провайдера плато, а не як повторне використання всієї історії зі зсувом у стилі Anthropic.
- Поточні live-твердження використовують консервативні перевірки нижньої межі, виведені зі спостереженої live-поведінки на `gpt-5.4-mini`:
  - стабільний префікс: `cacheRead >= 4608`, частка влучань `>= 0.90`
  - транскрипт інструмента: `cacheRead >= 4096`, частка влучань `>= 0.85`
  - транскрипт зображення: `cacheRead >= 3840`, частка влучань `>= 0.82`
  - транскрипт у стилі MCP: `cacheRead >= 4096`, частка влучань `>= 0.85`

Свіжа об’єднана live-перевірка 2026-04-04 дала такі результати:

- стабільний префікс: `cacheRead=4864`, частка влучань `0.966`
- транскрипт інструмента: `cacheRead=4608`, частка влучань `0.896`
- транскрипт зображення: `cacheRead=4864`, частка влучань `0.954`
- транскрипт у стилі MCP: `cacheRead=4608`, частка влучань `0.891`

Нещодавній локальний фактичний час виконання об’єднаної перевірки становив близько `88s`.

Чому твердження відрізняються:

- Anthropic надає явні контрольні точки кешу й повторне використання історії розмови зі зсувом.
- Кешування промптів OpenAI все ще чутливе до точного префікса, але ефективний префікс, який можна повторно використати в live-трафіку Responses, може виходити на плато раніше, ніж повний промпт.
- Через це порівняння Anthropic і OpenAI за одним міжпровайдерним відсотковим порогом створює хибні регресії.

### Конфігурація `diagnostics.cacheTrace`

```yaml
diagnostics:
  cacheTrace:
    enabled: true
    filePath: "~/.openclaw/logs/cache-trace.jsonl" # optional
    includeMessages: false # default true
    includePrompt: false # default true
    includeSystem: false # default true
```

Типові значення:

- `filePath`: `$OPENCLAW_STATE_DIR/logs/cache-trace.jsonl`
- `includeMessages`: `true`
- `includePrompt`: `true`
- `includeSystem`: `true`

### Перемикачі середовища (одноразове налагодження)

- `OPENCLAW_CACHE_TRACE=1` вмикає трасування кешу.
- `OPENCLAW_CACHE_TRACE_FILE=/path/to/cache-trace.jsonl` перевизначає шлях виводу.
- `OPENCLAW_CACHE_TRACE_MESSAGES=0|1` перемикає захоплення повного payload повідомлень.
- `OPENCLAW_CACHE_TRACE_PROMPT=0|1` перемикає захоплення тексту промпта.
- `OPENCLAW_CACHE_TRACE_SYSTEM=0|1` перемикає захоплення системного промпта.

### Що перевіряти

- Події трасування кешу мають формат JSONL і містять поетапні знімки, як-от `session:loaded`, `prompt:before`, `stream:context` і `session:after`.
- Вплив токенів кешу для кожного ходу видно у звичайних поверхнях використання через `cacheRead` і `cacheWrite` (наприклад, `/usage full` і підсумки використання сесії).
- Для Anthropic очікуйте і `cacheRead`, і `cacheWrite`, коли кешування активне.
- Для OpenAI очікуйте `cacheRead` під час влучань у кеш. GPT-5.6 Responses також може повідомляти `cacheWrite`, коли сегменти промпта записуються; інші payload Responses, які не містять лічильник запису, залишають його на `0`.
- Якщо потрібне трасування запитів, логувати ідентифікатори запитів і заголовки обмеження швидкості потрібно окремо від метрик кешу. Поточний вивід cache-trace OpenClaw зосереджений на формі промпта/сесії та нормалізованому використанні токенів, а не на сирих заголовках відповіді провайдера.

## Швидке усунення неполадок

- Високий `cacheWrite` на більшості ходів: перевірте змінні вхідні дані системного промпта й переконайтеся, що модель/провайдер підтримує ваші налаштування кешу.
- Високий `cacheWrite` на Anthropic: часто означає, що контрольна точка кешу потрапляє на вміст, який змінюється в кожному запиті.
- Низький `cacheRead` OpenAI: переконайтеся, що стабільний префікс стоїть на початку, повторюваний префікс має щонайменше 1024 токени, а той самий `prompt_cache_key` повторно використовується для ходів, які мають спільно використовувати кеш.
- Немає ефекту від `cacheRetention`: підтвердьте, що ключ моделі збігається з `agents.defaults.models["provider/model"]`.
- Запити Bedrock Nova/Mistral з налаштуваннями кешу: очікуване примусове встановлення runtime на `none`.
