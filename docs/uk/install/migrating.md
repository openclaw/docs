OpenClaw підтримує три шляхи міграції: імпорт з іншої системи агентів, перенесення наявної інсталяції на нову машину та оновлення Plugin на місці.

## Імпорт з іншої системи агентів

Використовуйте вбудовані провайдери міграції, щоб перенести інструкції, MCP-сервери, Skills, конфігурацію моделей і (за бажанням) API-ключі до OpenClaw. Плани переглядаються перед будь-якими змінами, секрети редагуються у звітах, а apply захищено перевіреною резервною копією.

Точка входу CLI — [`openclaw migrate`](/uk/cli/migrate). Онбординг також може запропонувати міграцію, коли виявляє відоме джерело (`openclaw onboard --flow import`).

## Перенесення OpenClaw на нову машину

Скопіюйте **каталог стану** (типово `~/.openclaw/`) і ваш **робочий простір**, щоб зберегти:

- **Конфігурацію** — `openclaw.json` і всі налаштування Gateway.
- **Автентифікацію** — `auth-profiles.json` для кожного агента (API-ключі та OAuth), а також будь-який стан каналу або provider-а в `credentials/`.
- **Сесії** — історію розмов і стан агента.
- **Стан каналу** — вхід WhatsApp, сесію Telegram тощо.
- **Файли робочого простору** — `MEMORY.md`, `USER.md`, Skills і промпти.

<Tip>
Виконайте `openclaw status` на старій машині, щоб підтвердити шлях до вашого каталогу стану. Користувацькі профілі використовують `~/.openclaw-<profile>/` або шлях, заданий через `OPENCLAW_STATE_DIR`.
</Tip>

### Кроки міграції

<Steps>
  <Step title="Зупиніть gateway і створіть резервну копію">
    На **старій** машині зупиніть gateway, щоб файли не змінювалися під час копіювання, а потім архівуйте:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    Якщо ви використовуєте кілька профілів (наприклад, `~/.openclaw-work`), архівуйте кожен окремо.

  </Step>

  <Step title="Установіть OpenClaw на новій машині">
    [Установіть](/uk/install) CLI (і Node, якщо потрібно) на новій машині. Нічого страшного, якщо онбординг створить новий `~/.openclaw/`. Ви перезапишете його на наступному кроці.
  </Step>

  <Step title="Скопіюйте каталог стану та робочий простір">
    Передайте архів через `scp`, `rsync -a` або зовнішній диск, а потім розпакуйте:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    Переконайтеся, що приховані каталоги було включено і що власник файлів збігається з користувачем, який запускатиме gateway.

  </Step>

  <Step title="Запустіть doctor і перевірте">
    На новій машині запустіть [Doctor](/uk/gateway/doctor), щоб застосувати міграції конфігурації та відновити сервіси:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

### Поширені помилки

<AccordionGroup>
  <Accordion title="Невідповідність профілю або state-dir">
    Якщо старий gateway використовував `--profile` або `OPENCLAW_STATE_DIR`, а новий — ні, канали виглядатимуть як розлогінені, а сесії будуть порожніми. Запустіть gateway з **тим самим** профілем або state-dir, який ви перенесли, а потім знову виконайте `openclaw doctor`.
  </Accordion>

  <Accordion title="Копіювання лише openclaw.json">
    Одного лише файла конфігурації недостатньо. Профілі автентифікації моделей зберігаються в `agents/<agentId>/agent/auth-profiles.json`, а стан каналів і provider-ів — у `credentials/`. Завжди переносіть **увесь** каталог стану.
  </Accordion>

  <Accordion title="Права доступу та власник">
    Якщо ви копіювали від root або змінили користувача, gateway може не змогти прочитати облікові дані. Переконайтеся, що каталог стану й робочий простір належать користувачеві, який запускає gateway.
  </Accordion>

  <Accordion title="Віддалений режим">
    Якщо ваш UI вказує на **віддалений** gateway, віддалений хост володіє сесіями та робочим простором. Переносьте сам хост gateway, а не свій локальний ноутбук. Див. [FAQ](/uk/help/faq#where-things-live-on-disk).
  </Accordion>

  <Accordion title="Секрети в резервних копіях">
    Каталог стану містить профілі автентифікації, облікові дані каналів та інший стан provider-ів. Зберігайте резервні копії у зашифрованому вигляді, уникайте незахищених каналів передавання та змініть ключі, якщо підозрюєте компрометацію.
  </Accordion>
</AccordionGroup>

### Контрольний список перевірки

На новій машині переконайтеся, що:

- [ ] `openclaw status` показує, що gateway запущено.
- [ ] Канали все ще підключені (повторне сполучення не потрібне).
- [ ] Інформаційна панель відкривається і показує наявні сесії.
- [ ] Файли робочого простору (пам’ять, конфігурації) присутні.

## Оновлення Plugin на місці

Оновлення Plugin на місці зберігають той самий id плагіна та ключі конфігурації, але можуть переміщувати стан на диску до поточного компонування. Посібники з оновлення для конкретних плагінів розміщені поруч із їхніми каналами:

- [Міграція Matrix](/uk/channels/matrix-migration): обмеження відновлення зашифрованого стану, автоматична поведінка знімків і команди ручного відновлення.
