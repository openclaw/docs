OpenClaw підтримує три шляхи міграції: імпорт з іншої агентної системи, перенесення наявної інсталяції на нову машину та оновлення плагіна на місці.

## Імпорт з іншої агентної системи

Вбудовані постачальники міграції переносять до OpenClaw інструкції, сервери MCP, Skills, конфігурацію моделей і, за згодою, ключі API. Плани можна попередньо переглянути до внесення будь-яких змін, секрети в звітах приховуються, а застосування змін захищене перевіреною резервною копією.

Точка входу CLI — [`openclaw migrate`](/uk/cli/migrate). Початкове налаштування також може запропонувати міграцію, коли виявить відоме джерело (`openclaw onboard --flow import`).

## Перенесення OpenClaw на нову машину

Скопіюйте **каталог стану** (типово `~/.openclaw/`) і свій **робочий простір**, щоб зберегти:

- **Конфігурацію** — `openclaw.json` і всі налаштування Gateway.
- **Автентифікацію** — окремий для кожного агента файл `auth-profiles.json` (ключі API та OAuth), а також будь-який стан каналів або постачальників у `credentials/`.
- **Сеанси** — історію розмов і стан агента.
- **Стан каналів** — дані входу WhatsApp, сеанс Telegram тощо.
- **Файли робочого простору** — `MEMORY.md`, `USER.md`, Skills і запити.

<Tip>
Виконайте `openclaw status` на старій машині, щоб підтвердити шлях до каталогу стану. Власні профілі використовують `~/.openclaw-<profile>/` або шлях, заданий через `OPENCLAW_STATE_DIR`.
</Tip>

### Кроки міграції

<Steps>
  <Step title="Зупиніть Gateway і створіть резервну копію">
    На **старій** машині зупиніть Gateway, щоб файли не змінювалися під час копіювання, а потім створіть архів:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    Якщо ви використовуєте кілька профілів (наприклад, `~/.openclaw-work`), архівуйте кожен окремо.

  </Step>

  <Step title="Установіть OpenClaw на новій машині">
    [Установіть](/uk/install) CLI (і Node, якщо потрібно) на новій машині. Нічого страшного, якщо початкове налаштування створить новий каталог `~/.openclaw/` — на наступному кроці ви його перезапишете.
  </Step>

  <Step title="Скопіюйте каталог стану та робочий простір">
    Передайте архів за допомогою `scp`, `rsync -a` або зовнішнього накопичувача, а потім розпакуйте:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    Переконайтеся, що приховані каталоги також скопійовано, а власником файлів є користувач, від імені якого працюватиме Gateway.

  </Step>

  <Step title="Запустіть діагностику та перевірте стан">
    На новій машині запустіть [діагностику](/uk/gateway/doctor), щоб застосувати міграції конфігурації та відновити служби:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

Якщо Telegram або Discord використовує типовий резервний варіант зі змінною середовища (`TELEGRAM_BOT_TOKEN` або `DISCORD_BOT_TOKEN`), переконайтеся, що перенесений файл `.env` у каталозі стану містить ці ключі, не виводячи значення секретів:

```bash
awk -F= '/^(TELEGRAM_BOT_TOKEN|DISCORD_BOT_TOKEN)=/ { print $1 "=present" }' ~/.openclaw/.env
```

`openclaw doctor` також попереджає, якщо для активованого типового облікового запису Telegram або Discord не налаштовано токен, а відповідна змінна середовища недоступна процесу діагностики.

### Поширені помилки

<AccordionGroup>
  <Accordion title="Невідповідність профілю або каталогу стану">
    Якщо старий Gateway використовував `--profile` або `OPENCLAW_STATE_DIR`, а новий їх не використовує, канали виглядатимуть так, ніби вхід не виконано, а сеанси будуть порожніми. Запустіть Gateway із **тим самим** профілем або каталогом стану, який ви перенесли, а потім знову виконайте `openclaw doctor`.
  </Accordion>

  <Accordion title="Копіювання лише openclaw.json">
    Одного файлу конфігурації недостатньо. Профілі автентифікації моделей зберігаються в `agents/<agentId>/agent/auth-profiles.json`, а стан каналів і постачальників — у `credentials/`. Завжди переносьте **весь** каталог стану.
  </Accordion>

  <Accordion title="Дозволи та власник">
    Якщо ви копіювали файли від імені root або змінили користувача, Gateway може не мати змоги прочитати облікові дані. Переконайтеся, що каталог стану й робочий простір належать користувачеві, від імені якого працює Gateway.
  </Accordion>

  <Accordion title="Віддалений режим">
    Якщо ваш інтерфейс підключений до **віддаленого** Gateway, сеанси й робочий простір належать віддаленому хосту. Перенесіть сам хост Gateway, а не локальний ноутбук. Див. [поширені запитання](/uk/help/faq#where-things-live-on-disk).
  </Accordion>

  <Accordion title="Секрети в резервних копіях">
    Каталог стану містить профілі автентифікації, облікові дані каналів та інший стан постачальників. Зберігайте резервні копії в зашифрованому вигляді, уникайте незахищених каналів передавання та змініть ключі, якщо підозрюєте витік.
  </Accordion>
</AccordionGroup>

### Контрольний список перевірки

На новій машині переконайтеся, що:

- [ ] `openclaw status` показує, що Gateway працює.
- [ ] Канали й надалі підключені (повторне сполучення не потрібне).
- [ ] Панель керування відкривається та показує наявні сеанси.
- [ ] Файли робочого простору (пам’ять, конфігурації) наявні.

## Оновлення плагіна на місці

Оновлення плагіна на місці зберігає той самий ідентифікатор плагіна та ключі конфігурації, але може перемістити стан на диску до поточної структури. Посібники з оновлення окремих плагінів розміщені поруч із документацією їхніх каналів:

- [Міграція Matrix](/uk/channels/matrix-migration): обмеження відновлення зашифрованого стану, поведінка автоматичних знімків і команди ручного відновлення.
