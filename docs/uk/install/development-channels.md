OpenClaw постачається з чотирма каналами оновлень:

- **стабільний**: dist-tag npm `latest`. Рекомендовано для більшості користувачів.
- **розширено стабільний**: dist-tag npm `extended-stable`. Цілком новий канал пакетів
  за останній підтримуваний місяць. Він доступний лише для пакетів, а встановлення
  виконується лише у передньому плані. Збережений вибір отримує підказки про
  оновлення лише для читання, коли ввімкнено `update.checkOnStart`, але вони
  ніколи не застосовуються автоматично.
- **бета**: dist-tag npm `beta`. Повертається до `latest`, коли `beta` відсутній
  або старіший за поточний стабільний випуск.
- **для розробки**: рухома вершина `main` (git). dist-tag npm `dev`, коли опубліковано. `main`
  призначено для експериментів і активної розробки; він може містити незавершені
  функції або несумісні зміни. Не використовуйте його для робочих Gateway.

Стабільні збірки зазвичай спочатку випускають у канал **бета**, перевіряють там, а потім
переводять у **latest** без зміни версії. Супровідники також можуть публікувати
безпосередньо в `latest`. Dist-tag є джерелом істини для встановлень через npm.

## Перемикання каналів

```bash
openclaw update --channel stable
openclaw update --channel extended-stable
openclaw update --channel beta
openclaw update --channel dev
```

`--channel` зберігає вибір у `update.channel` у конфігурації та керує обома
шляхами встановлення:

| Канал             | Встановлення через npm/пакет                                                                                                                                                           | Встановлення через git                                                                                                                                             |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `stable`          | dist-tag `latest`                                                                                                                                                                      | найновіший стабільний тег git (за винятком `-alpha.N`, `-beta.N`, `-rc.N`, `-dev.N`, `-next.N`, `-preview.N`, `-canary.N`, `-nightly.N` та інших іменованих суфіксів попередніх випусків) |
| `extended-stable` | визначає загальнодоступний селектор npm `extended-stable`, перевіряє точно вибраний пакет і встановлює саме цю версію. У разі помилки припиняє роботу без повернення до `latest`, `beta` або `dev`. | не підтримується: OpenClaw залишає робочу копію без змін і пропонує скористатися встановленням пакета                                                                |
| `beta`            | dist-tag `beta` із поверненням до `latest`, коли `beta` відсутній або старіший                                                                                                         | найновіший бета-тег git із поверненням до найновішого стабільного тегу git, коли бета-тег відсутній або старіший                                                     |
| `dev`             | dist-tag `dev` (використовується рідко; більшість розробників використовують встановлення через git)                                                                                   | отримує зміни, перебазовує робочу копію на висхідну гілку `main`, збирає та перевстановлює глобальний CLI                                                            |

Для встановлень `dev` через git типова робоча копія розташована в `~/openclaw` (або
`$OPENCLAW_HOME/openclaw`, коли задано `OPENCLAW_HOME`); перевизначте її за допомогою
`OPENCLAW_GIT_DIR`.

<Tip>
Щоб паралельно використовувати стабільну версію та версію для розробки, створіть дві окремі робочі копії та спрямуйте кожен Gateway на власну.
</Tip>

## Одноразове націлювання на версію або тег

Використовуйте `--tag`, щоб націлитися на певний dist-tag, версію або специфікацію
пакета для одного оновлення **без** зміни збереженого каналу:

```bash
# Встановити певну версію
openclaw update --tag 2026.4.1-beta.1

# Встановити з dist-tag beta (одноразово, не зберігається)
openclaw update --tag beta

# Перемкнутися на рухому робочу копію main з GitHub (зберігається)
openclaw update --channel dev

# Встановити певну специфікацію пакета npm
openclaw update --tag openclaw@2026.4.1-beta.1

# Одноразово встановити з main на GitHub без збереження каналу
openclaw update --tag main
```

Примітки:

- `--tag` застосовується **лише до встановлень пакетів (npm)**; встановлення через git його ігнорують.
- Тег не зберігається; наступний запуск `openclaw update` використовує налаштований
  канал.
- `--tag main` зіставляється із сумісною з npm специфікацією `github:openclaw/openclaw#main`
  для цього одного запуску. Для постійного встановлення з рухомої гілки `main` використовуйте
  `openclaw update --channel dev` (встановлення пакетів перемикаються на робочу копію git)
  або повторно встановіть за допомогою git-методу інсталятора:
  `curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method git --version main`.
  Шлях встановлення через npm повністю відхиляє цілі з джерел GitHub/git і натомість
  спрямовує вас до git-методу.
- Захист від пониження версії: якщо цільова версія старіша за поточну,
  OpenClaw запитує підтвердження (пропустіть за допомогою `--yes`).
- Розширено стабільний канал завжди використовує свою перевірену точну ціль пакета. Він не є
  одноразовим псевдонімом для `--tag extended-stable`, а `--tag` не можна поєднувати
  з фактичним розширено стабільним каналом.
- `--channel beta` відрізняється від `--tag beta`: потік каналу може повертатися
  до stable/latest, коли бета-версія відсутня або старіша, тоді як `--tag beta` завжди
  націлюється на необроблений dist-tag `beta` для цього одного запуску.

## Пробний запуск

Перегляньте, що зробить `openclaw update`, без внесення змін:

```bash
openclaw update --dry-run
openclaw update --channel beta --dry-run
openclaw update --tag 2026.4.1-beta.1 --dry-run
openclaw update --dry-run --json
```

Пробний запуск повідомляє фактичний канал, цільову версію, заплановані дії
та чи буде потрібне підтвердження пониження версії.

## Плагіни та канали

Перемикання каналів за допомогою `openclaw update` також синхронізує джерела плагінів:

- `dev` перемикає встановлені плагіни, які мають вбудований відповідник, назад на
  їхнє вбудоване джерело (робочу копію git).
- `stable` і `beta` відновлюють пакети плагінів, установлені через npm або ClawHub.
- `extended-stable` визначає придатні офіційні плагіни npm із порожнім/типовим
  наміром або наміром `latest` до точної встановленої версії ядра. Під час виконання він не опитує
  теги плагінів `@extended-stable`.
- Плагіни, установлені через npm, оновлюються після завершення оновлення ядра.

## Перевірка поточного стану

```bash
openclaw update status
```

Показує активний канал (із джерелом, що його визначило: конфігурація, тег git,
гілка git, установлена версія або типове значення), тип установлення (git або пакет),
поточну версію та доступність оновлення.

## Рекомендації щодо тегування

- Позначайте тегами випуски, на яких мають зупинятися робочі копії git: `vYYYY.M.PATCH` для стабільних,
  `vYYYY.M.PATCH-beta.N` для бета-версій. Іменовані суфікси попередніх випусків, як-от
  `-alpha.N`, `-rc.N` і `-next.N`, не є цільовими для стабільного або бета-каналу.
- Застарілі числові стабільні теги, як-от `vYYYY.M.PATCH-1` і `v1.0.1-1`, досі
  розпізнаються як стабільні теги git для сумісності.
- `vYYYY.M.PATCH.beta.N` (розділений крапками) також розпізнається для сумісності;
  віддавайте перевагу `-beta.N`.
- Зберігайте теги незмінними: ніколи не переміщуйте й не використовуйте тег повторно.
- Dist-tag npm залишаються джерелом істини для встановлень через npm:
  - `latest` -> стабільний
  - `extended-stable` -> випуск пакета за останній підтримуваний місяць
  - `beta` -> збірка-кандидат або стабільна збірка, спочатку випущена як бета
  - `dev` -> знімок main (необов'язково)

## Доступність застосунку macOS

Бета-збірки та збірки для розробки можуть **не** містити випуск застосунку macOS. Це нормально:

- Тег git і dist-tag npm усе одно можна опублікувати окремо.
- Зазначте «немає збірки macOS для цієї бета-версії» у примітках до випуску або журналі змін.
