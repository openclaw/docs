OpenClaw постачається з трьома каналами оновлень:

- **stable**: npm dist-tag `latest`. Рекомендовано для більшості користувачів.
- **beta**: npm dist-tag `beta`, коли він актуальний; якщо beta відсутній або старіший за
  останній стабільний реліз, процес оновлення повертається до `latest`.
- **dev**: рухома вершина `main` (git). npm dist-tag: `dev` (коли опубліковано).
  Гілка `main` призначена для експериментів і активної розробки. Вона може містити
  незавершені функції або несумісні зміни. Не використовуйте її для виробничих Gateway.

Зазвичай ми спочатку випускаємо стабільні збірки в **beta**, тестуємо їх там, а потім виконуємо
явний крок просування, який переміщує перевірену збірку до `latest` без
зміни номера версії. За потреби мейнтейнери також можуть опублікувати стабільний реліз
безпосередньо в `latest`. Dist-tags є джерелом істини для встановлень через npm.

## Заплановані щомісячні лінії підтримки

OpenClaw ще не постачає LTS або щомісячний канал підтримки. Ми працюємо
над сумісними із SemVer щомісячними лініями підтримки, щоб користувачі могли залишатися на спокійнішій
лінії, поки `latest` продовжує швидко рухатися.

Запланований формат версії: `YYYY.M.PATCH`:

- `YYYY` — це рік.
- `M` — це щомісячна лінія релізу без початкового нуля.
- `PATCH` збільшується в межах цієї щомісячної лінії та за потреби може перевищувати 100.

Приклади майбутніх тегів:

- `v2026.6.0`, `v2026.6.1`, `v2026.6.2` для червневої лінії.
- `v2026.6.3-beta.1` для попереднього релізу у швидкому/latest потоці.
- Майбутній dist-tag лінії підтримки, наприклад `stable-2026-6` або `lts-2026-6`, може
  вказувати на щомісячну лінію, але сьогодні такого каналу немає.

Поки цю міграцію не завершено, публічними каналами оновлень залишаються `stable`, `beta`
і `dev`.

## Перемикання каналів

```bash
openclaw update --channel stable
openclaw update --channel beta
openclaw update --channel dev
```

`--channel` зберігає ваш вибір у конфігурації (`update.channel`) і узгоджує
метод встановлення:

- **`stable`** (пакетні встановлення): оновлюється через npm dist-tag `latest`.
- **`beta`** (пакетні встановлення): віддає перевагу npm dist-tag `beta`, але повертається до
  `latest`, коли `beta` відсутній або старіший за поточний стабільний тег.
- **`stable`** (git-встановлення): переходить на останній стабільний git-тег.
- **`beta`** (git-встановлення): віддає перевагу останньому beta git-тегу, але повертається до
  останнього стабільного git-тегу, коли beta відсутній або старіший.
- **`dev`**: забезпечує git checkout (типово `~/openclaw`, можна перевизначити через
  `OPENCLAW_GIT_DIR`), перемикається на `main`, виконує rebase на upstream, збирає та
  встановлює глобальний CLI з цього checkout.

<Tip>
Якщо ви хочете мати stable і dev паралельно, тримайте два клони й спрямуйте свій gateway на стабільний.
</Tip>

## Одноразове націлювання на версію або тег

Використовуйте `--tag`, щоб націлитися на конкретний dist-tag, версію або специфікацію пакета для одного
оновлення **без** зміни вашого збереженого каналу:

```bash
# Install a specific version
openclaw update --tag 2026.4.1-beta.1

# Install from the beta dist-tag (one-off, does not persist)
openclaw update --tag beta

# Install from GitHub main branch (npm tarball)
openclaw update --tag main

# Install a specific npm package spec
openclaw update --tag openclaw@2026.4.1-beta.1
```

Примітки:

- `--tag` застосовується **лише до пакетних (npm) встановлень**. Git-встановлення його ігнорують.
- Тег не зберігається. Наступний `openclaw update` як зазвичай використовує ваш налаштований
  канал.
- Захист від пониження версії: якщо цільова версія старіша за поточну,
  OpenClaw запитує підтвердження (пропустити можна через `--yes`).
- `--channel beta` відрізняється від `--tag beta`: потік каналу може повернутися
  до stable/latest, коли beta відсутній або старіший, тоді як `--tag beta` націлюється на
  сирий dist-tag `beta` лише для цього одного запуску.

## Пробний запуск

Перегляньте, що зробив би `openclaw update`, без внесення змін:

```bash
openclaw update --dry-run
openclaw update --channel beta --dry-run
openclaw update --tag 2026.4.1-beta.1 --dry-run
openclaw update --dry-run --json
```

Пробний запуск показує ефективний канал, цільову версію, заплановані дії та
чи потрібне підтвердження пониження версії.

## Plugin та канали

Коли ви перемикаєте канали за допомогою `openclaw update`, OpenClaw також синхронізує джерела Plugin:

- `dev` віддає перевагу вбудованим Plugin із git checkout.
- `stable` і `beta` відновлюють пакети Plugin, встановлені через npm.
- Plugin, встановлені через npm, оновлюються після завершення оновлення ядра.

## Перевірка поточного стану

```bash
openclaw update status
```

Показує активний канал, тип встановлення (git або пакет), поточну версію та
джерело (конфігурація, git-тег, git-гілка або значення за замовчуванням).

## Найкращі практики тегування

- Позначайте тегами релізи, на які мають переходити git checkout (`vYYYY.M.D` для поточних
  стабільних релізів, `vYYYY.M.D-beta.N` для поточних beta-релізів).
- `vYYYY.M.D.beta.N` також розпізнається для сумісності, але надавайте перевагу `-beta.N`.
- Застарілі теги `vYYYY.M.D-<patch>` усе ще розпізнаються як стабільні (не-beta),
  але запланована модель щомісячної підтримки використовуватиме звичайні номери патчів
  (`vYYYY.M.PATCH`) замість суфікса виправлення через дефіс.
- Зберігайте теги незмінними: ніколи не переміщуйте й не використовуйте тег повторно.
- npm dist-tags залишаються джерелом істини для встановлень через npm:
  - `latest` -> stable
  - `beta` -> збірка-кандидат або stable-збірка, спочатку випущена в beta
  - `dev` -> знімок main (необов’язково)

## Доступність застосунку macOS

Збірки beta і dev можуть **не** містити реліз застосунку macOS. Це нормально:

- Git-тег і npm dist-tag усе одно можна опублікувати.
- Зазначте "немає збірки macOS для цієї beta" у примітках до релізу або changelog.
