GitHub Copilot — це ШІ-помічник GitHub для програмування. Він надає доступ до моделей
Copilot для вашого облікового запису й тарифного плану GitHub. OpenClaw може використовувати Copilot як
постачальника моделей або середовище виконання агента трьома різними способами.

## Три способи використання Copilot в OpenClaw

<Tabs>
  <Tab title="Вбудований постачальник (github-copilot)">
    Скористайтеся вбудованим процесом входу через пристрій, щоб отримати токен GitHub, а потім обмінюйте його на
    токени API Copilot під час роботи OpenClaw. Це **стандартний** і найпростіший спосіб,
    оскільки він не потребує VS Code.

    <Steps>
      <Step title="Виконайте команду входу">
        ```bash
        openclaw models auth login-github-copilot
        ```

        Вам буде запропоновано перейти за URL-адресою та ввести одноразовий код. Не закривайте
        термінал до завершення процесу.
      </Step>
      <Step title="Установіть модель за замовчуванням">
        ```bash
        openclaw models set github-copilot/claude-opus-4.7
        ```

        Або в конфігурації:

        ```json5
        {
          agents: {
            defaults: { model: { primary: "github-copilot/claude-opus-4.7" } },
          },
        }
        ```
      </Step>
    </Steps>

  </Tab>

  <Tab title="Plugin середовища Copilot SDK (copilot)">
    Установіть зовнішній Plugin `@openclaw/copilot`, якщо потрібно, щоб GitHub
    Copilot CLI та SDK керували низькорівневим циклом агента для вибраних
    моделей `github-copilot/*`.

    ```bash
    openclaw plugins install @openclaw/copilot
    ```

    Потім підключіть модель або постачальника до середовища виконання:

    ```json5
    {
      agents: {
        defaults: {
          model: "github-copilot/gpt-5.5",
          models: {
            "github-copilot/gpt-5.5": {
              agentRuntime: { id: "copilot" },
            },
          },
        },
      },
    }
    ```

    Виберіть цей варіант, якщо потрібні нативні сеанси Copilot CLI, стан потоків
    під керуванням SDK і керована Copilot операція Compaction для цих ходів агента. Без
    явного підключення через `agentRuntime` моделі `github-copilot/*` і надалі використовують
    вбудованого постачальника. Повний контракт середовища виконання див. у розділі
    [Середовище Copilot SDK](/uk/plugins/copilot).

  </Tab>

  <Tab title="Plugin Copilot Proxy (copilot-proxy)">
    Використовуйте розширення VS Code **Copilot Proxy** як локальний міст. OpenClaw звертається до
    кінцевої точки `/v1` проксі-сервера (за замовчуванням `http://localhost:3000/v1`) і використовує
    налаштований вами список моделей.

    Plugin `copilot-proxy` постачається з OpenClaw і ввімкнений за замовчуванням.
    Налаштуйте базову URL-адресу та ідентифікатори моделей за допомогою:

    ```bash
    openclaw models auth login --provider copilot-proxy --set-default
    ```

    <Note>
    Виберіть цей варіант, якщо ви вже використовуєте Copilot Proxy у VS Code або потребуєте маршрутизації
    через нього. Розширення VS Code має залишатися запущеним.
    </Note>

  </Tab>
</Tabs>

## GitHub Enterprise (резидентність даних)

Якщо ваша організація використовує клієнт GitHub Enterprise із резидентністю даних (вузол
`*.ghe.com`, наприклад `your-org.ghe.com`), Copilot працює на локальних для клієнта
кінцевих точках, а не на загальнодоступному `github.com`. OpenClaw надає це як
повноцінний варіант автентифікації, щоб вам не доводилося вручну редагувати URL-адреси.

<Steps>
  <Step title="Виберіть варіант автентифікації Enterprise">
    Під час початкового налаштування або в `openclaw models auth` виберіть
    **GitHub Copilot (Enterprise / data residency)**. Вам буде запропоновано вказати
    домен Enterprise (наприклад, `your-org.ghe.com`), після чого вхід через
    пристрій буде виконано для цього клієнта.

    Вводьте лише кореневий домен клієнта (`your-org.ghe.com`). Похідні вузли служб, як-от
    `api.your-org.ghe.com` або `copilot-api.your-org.ghe.com`, не приймаються;
    OpenClaw автоматично формує ці кінцеві точки з кореневого домену клієнта.

    ```bash
    openclaw models auth login --provider github-copilot --method device-enterprise
    ```

  </Step>
  <Step title="Домен зберігається в конфігурації">
    Вибраний вузол зберігається в параметрах постачальника, тому подальші оновлення токена
    й завершення автоматично спрямовуються до клієнта:

    ```json5
    {
      models: {
        providers: {
          "github-copilot": { params: { githubDomain: "your-org.ghe.com" } },
        },
      },
    }
    ```

  </Step>
</Steps>

Процес входу через пристрій, обмін токена та завершення відповідно використовують
`https://your-org.ghe.com/login/device/code`,
`https://api.your-org.ghe.com/copilot_internal/v2/token` і
`https://copilot-api.your-org.ghe.com`. Токени з резидентністю даних містять
позначку клієнта й не містять підказки щодо проксі-сервера, тому базова URL-адреса завершень повертається до
вузла Copilot клієнта замість загальнодоступної кінцевої точки.

<Note>
Перемикання доменів завжди повторно запускає вхід через пристрій. Якщо у вас уже збережено
токен Copilot і ви виберете інший домен (загальнодоступний `github.com` ↔ клієнт
`*.ghe.com` або один клієнт ↔ інший), OpenClaw не використовуватиме наявний токен повторно —
він примусово виконає новий вхід, щоб область дії токена відповідала домену, який записується до
конфігурації. Повторний вхід для *того самого* домену, як і раніше, пропонує повторно використати поточний
токен. Повернення до загальнодоступного `github.com` очищає збережений
`githubDomain`, щоб конфігурація повернулася до стандартного стану.
</Note>

<Note>
Змінна середовища `COPILOT_GITHUB_DOMAIN` перевизначає визначений домен
для кожного процесу Copilot, який його визначає: входу через пристрій Enterprise
(`--method device-enterprise`), окремої скороченої команди
`openclaw models auth login-github-copilot`, оновлення токена, векторних представлень
і завершень. Установіть для неї значення вашого вузла `*.ghe.com` для повністю безінтерфейсних середовищ або
CI. Не встановлюйте її (і не додавайте параметр конфігурації), щоб використовувати загальнодоступний `github.com`.
Під час входу зберігається домен, для якого було створено токен (і очищається під час входу
через загальнодоступний `github.com`), тому маршрутизація залишається правильною навіть після
видалення змінної середовища.
</Note>

## Необов’язкові прапорці

| Команда                                                                | Прапорець       | Опис                                                        |
| ---------------------------------------------------------------------- | --------------- | ----------------------------------------------------------- |
| `openclaw models auth login-github-copilot`                            | `--yes`         | Перезаписати наявний профіль автентифікації без підтвердження |
| `openclaw models auth login --provider github-copilot --method device` | `--set-default` | Також застосувати рекомендовану постачальником стандартну модель |

```bash
# Пропустити підтвердження повторного входу
openclaw models auth login-github-copilot --yes

# Увійти й установити модель за замовчуванням за один крок
openclaw models auth login --provider github-copilot --method device --set-default
```

## Неінтерактивне початкове налаштування

Процес входу через пристрій потребує інтерактивного TTY. Для налаштування без інтерфейсу імпортуйте
наявний токен доступу GitHub OAuth за допомогою `openclaw onboard --non-interactive`:

```bash
openclaw onboard --non-interactive --accept-risk \
  --auth-choice github-copilot \
  --github-copilot-token "$COPILOT_GITHUB_TOKEN" \
  --skip-channels --skip-health
```

Також можна не вказувати `--auth-choice`: передавання `--github-copilot-token` автоматично визначає
варіант автентифікації постачальника GitHub Copilot. Якщо прапорець не вказано, початкове налаштування
послідовно перевіряє `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, а потім `GITHUB_TOKEN`. Використовуйте
`--secret-input-mode ref` із установленою змінною `COPILOT_GITHUB_TOKEN`, щоб зберігати
`tokenRef` із посиланням на змінну середовища замість відкритого тексту в `auth-profiles.json`.

    позначає наступні ходи після результату інструмента як ініційовані агентом і встановлює заголовок
    візуального введення Copilot, коли хід містить вхідне зображення.
  </Accordion>

  <Accordion title="Порядок визначення змінних середовища">
    OpenClaw визначає автентифікацію Copilot зі змінних середовища в такому
    порядку пріоритетності:

    | Пріоритет | Змінна                 | Примітки                                      |
    | --------- | ---------------------- | --------------------------------------------- |
    | 1         | `COPILOT_GITHUB_TOKEN` | Найвищий пріоритет, призначена для Copilot    |
    | 2         | `GH_TOKEN`             | Токен GitHub CLI (резервний варіант)          |
    | 3         | `GITHUB_TOKEN`         | Стандартний токен GitHub (найнижчий пріоритет) |

    Якщо встановлено кілька змінних, OpenClaw використовує змінну з найвищим пріоритетом.
    Процес входу через пристрій (`openclaw models auth login-github-copilot`) зберігає
    свій токен у сховищі профілів автентифікації та має пріоритет над усіма змінними
    середовища.

  </Accordion>

## Векторні представлення для пошуку в пам’яті

GitHub Copilot також можна використовувати як постачальника векторних представлень для
[пошуку в пам’яті](/uk/concepts/memory-search). Якщо у вас є передплата Copilot і
ви ввійшли в систему, OpenClaw може використовувати його для векторних представлень без окремого ключа API.

### Конфігурація

Щоб використовувати векторні представлення GitHub Copilot, явно встановіть `memorySearch.provider`. Якщо
токен GitHub доступний, OpenClaw виявляє доступні моделі векторних представлень через
API Copilot і автоматично вибирає найкращу.

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "github-copilot",
        // Необов’язково: перевизначити автоматично виявлену модель
        model: "text-embedding-3-small",
      },
    },
  },
}
```

### Принцип роботи

1. OpenClaw визначає ваш токен GitHub (зі змінних середовища або профілю автентифікації).
2. Обмінює його на короткостроковий токен API Copilot.
3. Надсилає запит до кінцевої точки Copilot `/models`, щоб виявити доступні моделі векторних представлень.
4. Вибирає найкращу модель (порядок переваги: `text-embedding-3-small`,
   `text-embedding-3-large`, `text-embedding-ada-002`).
5. Надсилає запити на створення векторних представлень до кінцевої точки Copilot `/embeddings`.

Доступність моделей залежить від вашого тарифного плану GitHub. Якщо моделі векторних представлень
недоступні, OpenClaw пропускає Copilot і пробує наступного постачальника.
