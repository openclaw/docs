OpenClaw використовує один ідентифікатор провайдера, `openai`, як для автентифікації безпосередньо за допомогою ключа API, так і для
автентифікації за передплатою ChatGPT/Codex. `openai/*` — канонічний маршрут моделі.
Для ходів вбудованого агента, коли політику середовища виконання не задано або встановлено значення `auto`, характеристики маршруту OpenAI
визначають, чи може OpenClaw неявно вибрати вбудоване середовище виконання сервера застосунку Codex.
Сам по собі префікс `openai/*` не вибирає середовище виконання.

- **Моделі агентів** — `openai/*` через середовище виконання, вибране явною
  конфігурацією `agentRuntime` або неявною політикою маршруту OpenAI. Увійдіть за допомогою автентифікації Codex,
  щоб використовувати передплату ChatGPT/Codex, або налаштуйте профіль автентифікації
  за ключем API, якщо потрібна тарифікація за ключем.
- **Неагентні API OpenAI** — прямий доступ до OpenAI Platform із тарифікацією за використання,
  через `OPENAI_API_KEY` або профіль автентифікації за ключем API `openai`.
- **Застаріла конфігурація** — посилання `codex/*` і `openai-codex/*` виправляються на
  `openai/*` разом з обмеженим моделлю `agentRuntime.id: "codex"` за допомогою
  `openclaw doctor --fix`.

OpenAI явно підтримує використання OAuth передплати в зовнішніх інструментах і
робочих процесах, як-от OpenClaw.

## Відстеження використання та вартості

OpenClaw розділяє квоту передплати й тарифікацію API Platform:

- OAuth ChatGPT/Codex показує план передплати, вікна квот і баланс кредитів.
- `OPENAI_ADMIN_KEY` показує в розділі **Використання** Control UI дані за 30 днів про повідомлену провайдером вартість організації та використання завершень, зокрема щоденні витрати, загальну кількість запитів і токенів, найпопулярніші моделі та категорії витрат.
- `OPENAI_PROJECT_ID` за потреби обмежує історію Admin API одним проєктом.
- OpenClaw ніколи не надсилає `OPENAI_API_KEY` або профіль виведення `openai` до API організації; ці облікові дані можуть належати спеціальним кінцевим точкам, Azure або локальним кінцевим точкам агента.

Явно заданий ключ адміністратора має пріоритет над OAuth. Повідомлена провайдером історія не об’єднується з оцінкою вартості OpenClaw, отриманою із сеансів; вона може містити активність API інших клієнтів і коригування тарифікації на боці провайдера.

Документація OpenAI щодо [панелі використання API](https://help.openai.com/en/articles/10478918) описує вимоги щодо статусу власника організації та явного дозволу Usage Dashboard для доступу до даних про використання.

Провайдер, модель, середовище виконання та канал — це окремі рівні. Якщо ці позначення
плутаються, прочитайте [Середовища виконання агентів](/uk/concepts/agent-runtimes), перш ніж
змінювати конфігурацію.

| Тарифікація безпосередньо за ключем API, явне середовище виконання OpenClaw | `openai/gpt-5.6` разом із `agentRuntime.id: "openclaw"` провайдера/моделі | Виберіть звичайний профіль ключа API `openai`.                           |

## Неявне середовище виконання агента

Коли політику `agentRuntime` провайдера/моделі не задано або встановлено значення `auto`, належна провайдеру
політика маршруту OpenAI вибирає неявне середовище виконання на основі фактичної
кінцевої точки й адаптера:

| Характеристики фактичного маршруту                                                                                                                                                  | Неявне середовище виконання      |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| Точна офіційна кінцева точка Platform HTTPS із `openai-responses` або точна офіційна кінцева точка ChatGPT HTTPS із `openai-chatgpt-responses`; без заданого перевизначення запиту | Можна вибрати Codex |
| Заданий адаптер `openai-completions`                                                                                                                                  | OpenClaw              |
| Спеціальна кінцева точка                                                                                                                                                        | OpenClaw              |
| Явно задана точна офіційна кінцева точка, що використовує HTTP                                                                                                                            | Відхилено              |
| Маршрут із заданим перевизначенням запиту провайдера/моделі                                                                                                                 | OpenClaw              |

Явна нестандартна політика `agentRuntime.id` провайдера/моделі залишається визначальною.
Наприклад, `agentRuntime.id: "openclaw"` залишає в OpenClaw маршрут, який за інших умов
відповідав би вимогам Codex, тоді як `agentRuntime.id: "codex"` вимагає Codex і завершується
відмовою, якщо фактичний маршрут не оголошено сумісним із Codex.
Вибір середовища виконання не змінює тип облікових даних або тарифікацію: автентифікація
за ключем API Platform та автентифікація за передплатою ChatGPT/Codex залишаються окремими.

`openclaw doctor --fix` переносить застарілі посилання на моделі `codex/*` і `openai-codex/*`,
застарілі ідентифікатори профілів автентифікації Codex і застарілі записи порядку автентифікації Codex до
канонічного маршруту `openai`. Перенесені посилання на моделі отримують обмежений моделлю
`agentRuntime.id: "codex"`; використовуйте `auth.order.openai` для нової конфігурації порядку автентифікації.

<Note>
Нове налаштування OpenAI застосовує основну модель GPT-5.6 лише тоді, коли основну модель
не налаштовано. Додавання або оновлення автентифікації OpenAI зберігає наявний явний
вибір, зокрема `openai/gpt-5.5`, якщо явно не використати
`models auth login --set-default` або `models set`. Використовуйте профіль автентифікації
за ключем API лише тоді, коли потрібна автентифікація моделі агента за ключем API.
</Note>

## Обмежений попередній доступ до GPT-5.6

OpenClaw розпізнає точні ідентифікатори моделей `openai/gpt-5.6-sol`,
`openai/gpt-5.6-terra` і `openai/gpt-5.6-luna`. Усі три підтримують рівні міркування
`xhigh` і `max` у поточному каталозі. OpenAI описує Sol як
флагманський рівень, Terra — як збалансований, а Luna — як швидкий рівень
із нижчою вартістю. Див.
[оголошення про запуск GPT-5.6](https://openai.com/index/previewing-gpt-5-6-sol/)
і [посібник із доступу](https://help.openai.com/en/articles/20001325-a-preview-of-gpt-5-6-sol-terra-and-luna).

За безпосередньої автентифікації OpenAI за ключем API базовий ідентифікатор `openai/gpt-5.6` є псевдонімом
Sol і типовим значенням для нового налаштування. Нативний каталог Codex не застосовує
цей псевдонім прямого API на боці клієнта; залежно від доступу робочого простору він може показувати
точні ідентифікатори Sol, Terra й Luna. Тому нове налаштування OAuth ChatGPT/Codex
використовує `openai/gpt-5.6-sol`. Перевірте поточний обліковий запис за допомогою:

```bash
openclaw models list --provider openai
```

Доступ організації API та робочого простору Codex може відрізнятися. Якщо GPT-5.6
недоступна, явно виберіть GPT-5.5:

```bash
openclaw models set openai/gpt-5.5
```

OpenClaw показує помилку доступу від зовнішньої системи й не замінює без повідомлення
вибір GPT-5.6 на GPT-5.5.

<Note>
Для відповідних точних офіційних маршрутів HTTPS може бути вибрано вбудований Plugin сервера застосунку
Codex, коли політику середовища виконання не задано або встановлено значення `auto`; задані маршрути Completions,
спеціальні кінцеві точки й перевизначення транспорту запитів залишаються в OpenClaw. Незашифровані
офіційні кінцеві точки HTTP відхиляються. Явна конфігурація середовища виконання провайдера/моделі залишається
визначальною. Запустіть `openclaw doctor --fix`, щоб виправити застарілі посилання на моделі Codex,
посилання `codex-cli/*` або старі закріплення сеансів середовища виконання, які не були задані
явною конфігурацією середовища виконання.
</Note>

<Note>
Голос OpenAI Realtime працює через загальнодоступний **OpenAI Platform Realtime
API** і потребує ключа API платформи. Натомість токени Codex OAuth автентифікують
серверну частину ChatGPT Codex; вони не взаємозамінні з ключами API платформи
для загальнодоступних кінцевих точок Realtime.

Якщо автентифікація за ключем API повідомляє про відсутність коштів, поповніть кредити платформи на
[platform.openai.com/account/billing](https://platform.openai.com/account/billing)
для організації, що забезпечує ваші облікові дані реального часу, коли використовується автентифікація
за ключем API. Голос у реальному часі приймає профіль автентифікації за ключем API `openai`, створений
за допомогою `openclaw onboard --auth-choice openai-api-key`, ключ API платформи, установлений через
`talk.realtime.providers.openai.apiKey` для розмови в Control UI, або
`plugins.entries.voice-call.config.realtime.providers.openai.apiKey` для Voice
Call, або змінну середовища `OPENAI_API_KEY`.
</Note>

## Векторні представлення пам’яті

OpenClaw може використовувати OpenAI або сумісну з OpenAI кінцеву точку векторних представлень для
індексування `memory_search` і векторних представлень запитів:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
        model: "text-embedding-3-small",
      },
    },
  },
}
```

Для сумісних з OpenAI кінцевих точок, які потребують асиметричних міток векторних представлень, установіть
`queryInputType` і `documentInputType` у `memorySearch`. OpenClaw
передає їх як специфічні для постачальника поля запиту `input_type`: векторні представлення
запитів використовують `queryInputType`; проіндексовані фрагменти пам’яті й пакетне індексування використовують
`documentInputType`. Повний приклад див. у
[довіднику з налаштування пам’яті](/uk/reference/memory-config#provider-specific-config).

    ```json5
    {

    }
    ```

    Неповний ідентифікатор прямого API `gpt-5.6` відповідає рівню Sol. Якщо ця організація API
    не надає GPT-5.6, явно встановіть основну модель на
    `openai/gpt-5.5`.

    Щоб спробувати поточну модель Instant ChatGPT через OpenAI API, установіть модель
    на `openai/chat-latest`:

    ```json5
    {
      env: { OPENAI_API_KEY: "example-openai-key-not-real" },
      agents: { defaults: { model: { primary: "openai/chat-latest" } } },
    }
    ```

    `chat-latest` — змінний псевдонім. Натомість нове налаштування ключа API OpenAI використовує
    `openai/gpt-5.6`, чий неповний ідентифікатор прямого API відповідає Sol. Наявні
    явно задані основні моделі, зокрема `openai/gpt-5.5`, залишаються незмінними. Псевдонім
    `chat-latest` приймає лише текстову деталізацію `medium`; для цієї моделі OpenClaw примусово
    змінює будь-яку іншу запитану деталізацію на `medium`.

    <Warning>
    OpenClaw **не** надає `gpt-5.3-codex-spark` через прямий маршрут
    ключа API OpenAI. Вона доступна лише через записи каталогу підписки Codex,
    коли вона доступна для вашого облікового запису, у який виконано вхід.
    </Warning>

  </Tab>

  <Tab title="Підписка Codex">
    **Найкраще підходить для:** використання підписки ChatGPT/Codex із вбудованим виконанням
    app-server Codex замість окремого ключа API. Для хмарного Codex потрібен
    вхід у ChatGPT.

    <Steps>

        за допомогою потоку коду пристрою ChatGPT замість зворотного виклику браузера

    ### Приклад конфігурації

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
    }
    ```

    За наявності резервного ключа API залиште вибрану модель у `openai/*`, а
    порядок автентифікації розмістіть у `openai`. OpenClaw спочатку використовує підписку, а потім
    ключ API, залишаючись у середовищі Codex:

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
      auth: {
        order: {
          openai: [
            "openai:user@example.com",
            "openai:api-key-backup",
          ],
        },
      },
    }
    ```

    <Note>
    Початкове налаштування більше не імпортує дані OAuth з `~/.codex`. Увійдіть за допомогою
    OAuth у браузері (типово) або описаного вище потоку коду пристрою; OpenClaw керує
    отриманими обліковими даними у власному сховищі автентифікації агента.
    </Note>

    ### Перевірка та відновлення маршрутизації OAuth Codex

    ```bash
    openclaw models status
    openclaw models auth list --provider openai
    openclaw config get agents.defaults.model --json
    openclaw config get models.providers.openai.agentRuntime --json
    ```

    Для конкретного агента додайте `--agent <id>`:

    ```bash
    openclaw models status --agent <id>
    openclaw models auth list --agent <id> --provider openai
    ```

    Якщо стара конфігурація все ще містить застарілі посилання Codex GPT або неактуальне
    закріплення сеансу середовища виконання OpenAI без явної конфігурації середовища виконання, виправте її:

    ```bash
    openclaw doctor --fix
    openclaw config validate
    ```

    Якщо `models auth list --provider openai` не показує придатного профілю, увійдіть
    знову:

    ```bash
    openclaw models auth login --provider openai
    openclaw models status --probe --probe-provider openai
    ```

    Використовуйте `--profile-id` для кількох входів через OAuth Codex в одному агенті, а потім
    керуйте ними за допомогою порядку автентифікації або `/model ...@<profileId>`:

    ```bash
    openclaw models auth login --provider openai --profile-id openai:ritsuko
    openclaw models auth login --provider openai --profile-id openai:lain
    ```

    Виконайте `openclaw doctor --fix`, щоб перенести старі застарілі ідентифікатори профілів
    із префіксом OpenAI Codex і записи порядку, перш ніж покладатися на впорядкування профілів.

    ### Індикатор стану

    Команда чату `/status` показує, яке середовище виконання моделі активне для поточного
    сеансу. Вбудоване середовище сервера застосунку Codex відображається як
    `Runtime: OpenAI Codex`, коли його вибирає придатний неявний маршрут або явна
    політика середовища виконання постачальника чи моделі.

    ### Попередження doctor

    Якщо в конфігурації або стані сеансу залишаються застарілі посилання на моделі Codex
    чи неактуальні закріплення середовища виконання OpenAI, `openclaw doctor --fix` переписує їх на `openai/*`
    із середовищем виконання Codex, якщо OpenClaw не налаштовано явно.

    ### Обмеження контекстного вікна

    OpenClaw розглядає метадані моделі та обмеження контексту середовища виконання як окремі
    значення. Для `openai/gpt-5.5` через каталог OAuth Codex:

    - Нативне `contextWindow`: `400000`
    - Типове обмеження `contextTokens` середовища виконання: `272000`

    Менше типове обмеження на практиці забезпечує кращі характеристики
    затримки та якості. Перевизначте його за допомогою `contextTokens`:

    ```json5
    {
      models: {
        providers: {
          openai: {
            models: [{ id: "gpt-5.5", contextTokens: 160000 }],
          },
        },
      },
    }
    ```

    <Note>
    Використовуйте `contextWindow`, щоб оголосити нативні метадані моделі. Використовуйте `contextTokens`,
    щоб обмежити бюджет контексту середовища виконання. Прямий маршрут через ключ API OpenAI
    повідомляє про більший нативний `contextWindow` (`1000000`) для `gpt-5.5`; ці два
    маршрути відстежуються окремо, оскільки каталоги вищого рівня відрізняються.
    </Note>

    ### Відновлення каталогу

    OpenClaw використовує метадані каталогу Codex вищого рівня для `gpt-5.5`, якщо вони
    наявні. Якщо під час поточного виявлення Codex відсутній рядок `gpt-5.5`, хоча обліковий запис
    автентифіковано, OpenClaw створює цей рядок моделі OAuth, щоб запуски Cron,
    субагента та налаштованої типової моделі не завершувалися помилкою
    `Unknown model`.

## Автентифікація нативного сервера застосунку Codex

Нативне середовище сервера застосунку Codex використовує посилання на моделі `openai/*`, коли його неявно
вибирає придатний точний офіційний маршрут HTTPS або коли його явно вибирає
`agentRuntime.id: "codex"` постачальника чи моделі. Його автентифікація все одно
пов’язана з обліковим записом. OpenClaw вибирає автентифікацію в такому порядку:

1. Упорядковані профілі автентифікації OpenAI для агента, бажано в
   `auth.order.openai`. Виконайте `openclaw doctor --fix`, щоб перенести старі застарілі
   ідентифікатори профілів автентифікації Codex і порядок автентифікації.
2. Наявний обліковий запис сервера застосунку, наприклад локальний вхід у ChatGPT
   через Codex CLI. Для типового ізольованого домашнього каталогу агента OpenClaw передає цей нативний
   обліковий запис CLI до сервера застосунку через його RPC входу; він не надає спільного доступу до
   конфігурації, плагінів або сховища гілок CLI.
3. Лише для локальних запусків сервера застосунку через stdio й лише коли сервер застосунку
   повідомляє про відсутність облікового запису: `CODEX_API_KEY`, потім `OPENAI_API_KEY`.

Локальний вхід через підписку ChatGPT/Codex не замінюється лише через те, що
процес Gateway також має `OPENAI_API_KEY` для прямих моделей OpenAI або
вбудовувань. Резервне використання ключа API зі змінної середовища застосовується лише до локального шляху stdio
без облікового запису; він ніколи не надсилається через з’єднання WebSocket із сервером застосунку. Коли
вибрано профіль Codex із підпискою, OpenClaw також не передає
`CODEX_API_KEY` і `OPENAI_API_KEY` дочірньому процесу сервера застосунку stdio,
а натомість надсилає вибрані облікові дані через RPC входу сервера застосунку.

Коли цей профіль підписки заблоковано через обмеження використання Codex, OpenClaw
позначає профіль заблокованим до оголошеного Codex часу скидання та дає змогу порядку
автентифікації перейти до наступного профілю `openai:*`, не змінюючи вибрану
модель і не виходячи із середовища Codex. Після настання часу скидання
профіль підписки знову стає придатним.

## Генерування зображень

Вбудований Plugin `openai` реєструє генерування зображень за допомогою
інструмента `image_generate`. Він підтримує генерування зображень як через ключ API OpenAI, так і через OAuth Codex
за допомогою того самого посилання на модель `openai/gpt-image-2`.

| Можливість                | Ключ API OpenAI                     | OAuth Codex                          |
| ------------------------- | ---------------------------------- | ------------------------------------ |
| Посилання на модель       | `openai/gpt-image-2`               | `openai/gpt-image-2`                 |
| Автентифікація            | `OPENAI_API_KEY`                   | Вхід через OAuth OpenAI Codex        |
| Транспорт                 | API зображень OpenAI                  | Серверна частина Codex Responses     |
| Макс. зображень на запит  | 4                                  | 4                                    |
| Режим редагування         | Увімкнено (до 5 еталонних зображень) | Увімкнено (до 5 еталонних зображень) |
| Перевизначення розміру    | Підтримується, зокрема розміри 2K/4K | Підтримується, зокрема розміри 2K/4K |
| Співвідношення сторін / роздільність | Не передається до API зображень OpenAI | За можливості безпечно зіставляється з підтримуваним розміром |

```json5
{
  agents: {
    defaults: {
      imageGenerationModel: { primary: "openai/gpt-image-2" },
    },
  },
}
```

<Note>
Спільні параметри інструмента, вибір постачальника та поведінку резервного перемикання див. у розділі
[Генерування зображень](/uk/tools/image-generation).
</Note>

`gpt-image-2` є типовою моделлю для генерування зображень із тексту та редагування зображень
в OpenAI. `gpt-image-1.5`, `gpt-image-1` і `gpt-image-1-mini` залишаються доступними
як явні перевизначення моделі. Використовуйте `openai/gpt-image-1.5` для
виведення PNG/WebP із прозорим тлом; поточний API `gpt-image-2` відхиляє
`background: "transparent"`.

Для запиту з прозорим тлом викличте `image_generate` із
`model: "openai/gpt-image-1.5"`, `outputFormat: "png"` або `"webp"`, а також
`background: "transparent"`; старіший параметр постачальника `openai.background`
усе ще приймається. OpenClaw також захищає публічні маршрути OpenAI та OAuth OpenAI Codex,
переписуючи типові прозорі запити `openai/gpt-image-2` на
`gpt-image-1.5`; Azure і власні кінцеві точки, сумісні з OpenAI, зберігають
налаштовані назви розгортань і моделей.

Те саме налаштування доступне для безголових запусків CLI:

```bash
openclaw infer image generate \
  --model openai/gpt-image-1.5 \
  --output-format png \
  --background transparent \
  --prompt "Проста наліпка у вигляді червоного кола на прозорому тлі" \
  --json
```

Використовуйте ті самі прапорці `--output-format` і `--background` разом із
`openclaw infer image edit`, починаючи з вхідного файла.
`--openai-background` залишається доступним як псевдонім, специфічний для OpenAI. Використовуйте
`--quality low|medium|high|auto`, щоб керувати якістю та вартістю зображень OpenAI.
Використовуйте `--openai-moderation low|auto`, щоб передати підказку модерації OpenAI з
`image generate` або `image edit`.

Для встановлень з OAuth ChatGPT/Codex зберігайте те саме посилання `openai/gpt-image-2`. Коли
налаштовано профіль OAuth `openai`, OpenClaw визначає збережений токен доступу OAuth
і надсилає запити зображень через серверну частину Codex Responses; він
не намагається спочатку використати `OPENAI_API_KEY` і не переходить непомітно на ключ API.
Явно налаштуйте `models.providers.openai` із ключем API, власною базовою
URL-адресою або кінцевою точкою Azure, коли натомість потрібен прямий маршрут
API зображень OpenAI. Якщо ця власна кінцева точка зображень розташована в довіреній LAN або за приватною адресою,
також установіть `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true`; OpenClaw
блокує приватні та внутрішні кінцеві точки зображень, сумісні з OpenAI, якщо немає цієї
явної згоди.

Згенерувати:

```
/tool image_generate model=openai/gpt-image-2 prompt="Професійний плакат запуску OpenClaw на macOS" size=3840x2160 count=1
```

Згенерувати прозорий PNG:

```
/tool image_generate model=openai/gpt-image-1.5 prompt="Проста наліпка у вигляді червоного кола на прозорому тлі" outputFormat=png background=transparent
```

Відредагувати:

```
/tool image_generate model=openai/gpt-image-2 prompt="Збережи форму об’єкта, зміни матеріал на напівпрозоре скло" image=/path/to/reference.png size=1024x1536
```

## Генерування відео

Вбудований Plugin `openai` реєструє генерування відео за допомогою
інструмента `video_generate`.

| Можливість        | Значення                                                                           |
| ---------------- | ---------------------------------------------------------------------------------- |
| Типова модель     | `openai/sora-2`                                                                    |
| Режими            | Перетворення тексту на відео, зображення на відео, редагування одного відео        |
| Еталонні вхідні дані | 1 зображення або 1 відео                                                        |
| Перевизначення розміру | Підтримується для перетворення тексту на відео та зображення на відео          |
| Співвідношення сторін | Перетворюється на найближчий підтримуваний розмір, без передавання необробленого значення |
| Інші перевизначення | `resolution`, `audio`, `watermark` не підтримуються та відкидаються з попередженням інструмента |

Запити OpenAI на перетворення зображення на відео використовують `POST /v1/videos` із зображенням
`input_reference`. Для редагування одного відео використовується `POST /v1/videos/edits` із
завантаженим відео в полі `video`.

```json5
{
  agents: {
    defaults: {
      videoGenerationModel: { primary: "openai/sora-2" },
    },
  },
}
```

<Note>
Відомості про спільні параметри інструмента, вибір провайдера та поведінку
аварійного перемикання див. в розділі [Генерування відео](/uk/tools/video-generation).

Провайдер OpenAI оголошує `supportsSize`, але не `supportsAspectRatio` чи
`supportsResolution`. Спільний шар нормалізації OpenClaw перетворює запитаний
`aspectRatio` на найближчий відповідний OpenAI `size` до того, як
запит надійде до провайдера, тому запити зі співвідношенням сторін зазвичай усе одно працюють.
Для `resolution` немає резервного розміру, тому цей параметр відкидається, а викликачеві
повідомляється `Ignored unsupported overrides for openai/<model>: resolution=<value>`.
</Note>

## Внесок у промпт GPT-5

OpenClaw додає спільний внесок у промпт GPT-5 для моделей сімейства GPT-5 у
провайдера `openai` (зокрема для застарілих посилань Codex до виправлення, які нормалізуються
до `openai/*`). Інші провайдери, які також надають ідентифікатори моделей сімейства GPT-5,
як-от маршрути OpenRouter або opencode, не отримують цього накладання; воно обмежується
ідентифікатором провайдера `openai`, а не лише ідентифікатором моделі. Старіші моделі GPT-4.x
ніколи його не отримують.

Нативний стенд app-server Codex не отримує через інструкції
розробника контракт поведінки щодо персони й дисципліни використання інструментів або накладання дружнього
стилю взаємодії; нативний Codex зберігає базову поведінку, поведінку моделі та
проєктної документації, що належать Codex, а OpenClaw вимикає вбудовану особистість Codex для
нативних потоків, щоб файли особистості робочого простору агента залишалися авторитетними.
OpenClaw додає до нативних потоків Codex лише контекст середовища виконання: доставлення
через канал, динамічні інструменти OpenClaw, делегування ACP, контекст робочого простору та
Skills OpenClaw. Текст рекомендацій щодо Heartbeat із цього самого внеску є
єдиним винятком: нативні виклики Heartbeat у Codex його отримують як окремі
інструкції зі співпраці, а не через спільний гачок внеску
в промпт.

Внесок GPT-5 додає позначений тегами контракт поведінки щодо збереження
персони, безпеки виконання, дисципліни використання інструментів, форми виведення, перевірок
завершення та верифікації для відповідних промптів, сформованих OpenClaw. Специфічна для каналу
поведінка щодо відповідей і беззвучних повідомлень залишається в спільному системному
промпті OpenClaw і політиці вихідного доставлення. Шар дружнього стилю взаємодії
є окремим і налаштовуваним.

| Значення                  | Ефект                                      |
| ---------------------- | ------------------------------------------- |
| `"friendly"` (за замовчуванням) | Увімкнути шар дружнього стилю взаємодії |
| `"on"`                 | Псевдонім для `"friendly"`                      |
| `"off"`                | Вимкнути лише шар дружнього стилю       |

<Tabs>
  <Tab title="Конфігурація">
    ```json5
    {
      agents: {
        defaults: {
          promptOverlays: {
            gpt5: { personality: "friendly" },
          },
        },
      },
    }
    ```
  </Tab>
  <Tab title="CLI">
    ```bash
    openclaw config set agents.defaults.promptOverlays.gpt5.personality off
    ```
  </Tab>
</Tabs>

<Tip>
Під час виконання значення нечутливі до регістру, тому і `"Off"`, і `"off"`
вимикають шар дружнього стилю.
</Tip>

<Note>
Застарілий `plugins.entries.openai.config.personality` усе ще зчитується як
резервний варіант сумісності, коли спільне налаштування
`agents.defaults.promptOverlays.gpt5.personality` не задано.
</Note>

## Кінцеві точки Azure OpenAI

Вбудований провайдер `openai` може використовувати ресурс Azure OpenAI для генерування
зображень через перевизначення базової URL-адреси. На шляху генерування зображень OpenClaw
виявляє імена хостів Azure у `models.providers.openai.baseUrl` та автоматично переходить на
формат запитів Azure.

<Note>
Голос у реальному часі використовує окремий шлях конфігурації
(`plugins.entries.voice-call.config.realtime.providers.openai.azureEndpoint`)
і не залежить від `models.providers.openai.baseUrl`. Налаштування Azure наведено в розділі **Голос у
реальному часі** панелі-акордеона [Голос і мовлення](#voice-and-speech).
</Note>

Використовуйте Azure OpenAI, якщо:

- У вас уже є передплата Azure OpenAI, квота або корпоративна
  угода
- Вам потрібні регіональне розміщення даних або засоби контролю відповідності, які надає Azure
- Ви хочете зберігати трафік у межах наявного клієнтського середовища Azure

### Конфігурація

Для генерування зображень в Azure через вбудований провайдер `openai` спрямуйте
`models.providers.openai.baseUrl` на свій ресурс Azure та задайте в `apiKey`
ключ Azure OpenAI (не ключ OpenAI Platform):

```json5
{
  models: {
    providers: {
      openai: {
        baseUrl: "https://<your-resource>.openai.azure.com",
        apiKey: "<azure-openai-api-key>",
      },
    },
  },
}
```

OpenClaw розпізнає такі суфікси хостів Azure для маршруту генерування
зображень Azure:

- `*.openai.azure.com`
- `*.services.ai.azure.com`
- `*.cognitiveservices.azure.com`

Для запитів генерування зображень на розпізнаному хості Azure OpenClaw:

- Надсилає заголовок `api-key` замість `Authorization: Bearer`
- Використовує шляхи, прив’язані до розгортання (`/openai/deployments/{deployment}/...`)
- Додає `?api-version=...` до кожного запиту
- Використовує стандартний час очікування запиту 600 с для викликів генерування зображень Azure.
  Значення `timeoutMs` для окремих викликів і надалі перевизначають це стандартне значення.

Інші базові URL-адреси (публічний OpenAI, проксі, сумісні з OpenAI) зберігають стандартний
формат запиту зображень OpenAI.

<Note>
Маршрутизація Azure для шляху генерування зображень провайдера `openai` потребує
OpenClaw 2026.4.22 або новішої версії. Попередні версії обробляють будь-яке користувацьке
значення `openai.baseUrl` як публічну кінцеву точку OpenAI, тому запити до розгортань зображень
Azure завершуються помилкою.
</Note>

### Версія API

Задайте `AZURE_OPENAI_API_VERSION`, щоб закріпити певну попередню або загальнодоступну версію Azure
для шляху генерування зображень Azure:

```bash
export AZURE_OPENAI_API_VERSION="2024-12-01-preview"
```

Якщо змінну не задано, стандартним значенням є `2024-12-01-preview`.

### Назви моделей — це назви розгортань

Azure OpenAI пов’язує моделі з розгортаннями. Для запитів генерування зображень Azure,
маршрутизованих через вбудований провайдер `openai`, поле `model` в OpenClaw
має містити **назву розгортання Azure**, налаштовану на порталі Azure, а не
ідентифікатор публічної моделі OpenAI.

Якщо створити розгортання з назвою `gpt-image-2-prod`, яке обслуговує `gpt-image-2`:

```
/tool image_generate model=openai/gpt-image-2-prod prompt="Чистий плакат" size=1024x1024 count=1
```

Те саме правило щодо назви розгортання застосовується до будь-якого виклику генерування зображень,
маршрутизованого через вбудований провайдер `openai`.

### Регіональна доступність

Наразі генерування зображень Azure доступне лише в частині регіонів
(наприклад, `eastus2`, `swedencentral`, `polandcentral`, `westus3`,
`uaenorth`). Перед створенням розгортання перевірте актуальний список регіонів Microsoft
і переконайтеся, що потрібна модель пропонується у вашому регіоні.

### Відмінності параметрів

Azure OpenAI і публічний OpenAI не завжди приймають однакові параметри зображень.
Azure може відхиляти параметри, дозволені публічним OpenAI (наприклад, певні
значення `background` у `gpt-image-2`), або надавати їх лише для певних версій
моделей. Ці відмінності походять від Azure та базової моделі, а не від
OpenClaw. Якщо запит Azure завершується помилкою перевірки, перегляньте
набір параметрів, підтримуваний конкретним розгортанням і версією API, на
порталі Azure.

<Note>
Azure OpenAI використовує нативний транспорт і поведінку сумісності, але не отримує
прихованих заголовків атрибуції OpenClaw — див. панель-акордеон **Нативні та сумісні з OpenAI
маршрути** в розділі [Розширена конфігурація](#advanced-configuration).

Для трафіку чату або Responses в Azure (окрім генерування зображень) використовуйте
процес початкового налаштування або окрему конфігурацію провайдера Azure; одного лише `openai.baseUrl`
недостатньо для застосування формату API й автентифікації Azure. Існує окремий
провайдер `azure-openai-responses/*`; див. панель-акордеон «Серверне ущільнення»
нижче.
</Note>

## Розширена конфігурація

Наведені нижче приклади `params` для окремих моделей формують запит вбудованого провайдера
OpenClaw. Їх налаштування є явно заданою поведінкою запиту, тому маршрут
`auto`, який за інших умов відповідає вимогам, залишається в OpenClaw замість неявного вибору Codex. Нативний
серверний каркас застосунку Codex керує власним транспортом і налаштуваннями запитів; явне
значення `agentRuntime.id: "codex"` призводить до відмови, якщо фактичний маршрут не оголошено
сумісним із Codex.

<AccordionGroup>
  <Accordion title="Транспорт (WebSocket або SSE)">
    OpenClaw спочатку використовує WebSocket із резервним переходом на SSE (`"auto"`) для `openai/*`.

    У режимі `"auto"` OpenClaw:
    - Повторює одну ранню невдалу спробу WebSocket перед переходом на SSE
    - Після помилки позначає WebSocket як деградований на 60 секунд і використовує SSE
      протягом періоду відновлення
    - Додає стабільні заголовки ідентифікації сеансу та ходу для повторних спроб і
      повторних підключень
    - Нормалізує лічильники використання (`input_tokens` / `prompt_tokens`) між
      варіантами транспорту

    | Значення                | Поведінка                          |
    | ---------------------- | ------------------------------------ |
    | `"auto"` (стандартне)   | Спочатку WebSocket, резервний перехід на SSE     |
    | `"sse"`              | Примусово використовувати лише SSE                    |
    | `"websocket"`        | Примусово використовувати лише WebSocket              |

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": {
              params: { transport: "auto" },
            },
          },
        },
      },
    }
    ```

    Пов’язана документація OpenAI:
    - [Realtime API через WebSocket](https://platform.openai.com/docs/guides/realtime-websocket)
    - [Потокові відповіді API (SSE)](https://platform.openai.com/docs/guides/streaming-responses)

  </Accordion>

  <Accordion title="Швидкий режим">
    OpenClaw надає спільний перемикач швидкого режиму для `openai/*`:

    - **Чат/UI:** `/fast status|auto|on|off`
    - **Конфігурація:** `agents.defaults.models["<provider>/<model>"].params.fastMode`

    Коли його ввімкнено, OpenClaw зіставляє швидкий режим із пріоритетною обробкою OpenAI
    (`service_tier = "priority"`). Наявні значення `service_tier`
    зберігаються, а швидкий режим не перезаписує `reasoning` або
    `text.verbosity`. `fastMode: "auto"` запускає нові виклики моделі у швидкому режимі до
    автоматичного порогового часу, а потім запускає наступні повторні, резервні виклики, виклики з результатами інструментів або
    виклики продовження без швидкого режиму. Стандартний пороговий час становить 60 секунд;
    задайте `params.fastAutoOnSeconds` для активної моделі, щоб змінити його.

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { fastMode: "auto", fastAutoOnSeconds: 30 } },
          },
        },
      },
    }
    ```

    <Note>
    Перевизначення сеансу мають пріоритет над конфігурацією. Очищення перевизначення сеансу в
    UI сеансів повертає сеанс до налаштованого стандартного значення.
    </Note>

  </Accordion>

  <Accordion title="Пріоритетна обробка (service_tier)">
    API OpenAI надає пріоритетну обробку через `service_tier`. Налаштуйте її для кожної
    моделі в OpenClaw:

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { serviceTier: "priority" } },
          },
        },
      },
    }
    ```

    Підтримувані значення: `auto`, `default`, `flex`, `priority`.

    <Warning>
    `serviceTier` передається лише нативним кінцевим точкам OpenAI
    (`api.openai.com`) і нативним кінцевим точкам Codex (`chatgpt.com/backend-api`).
    Якщо маршрутизувати будь-якого з цих провайдерів через проксі, OpenClaw залишає
    `service_tier` без змін.
    </Warning>

  </Accordion>

  <Accordion title="Серверне ущільнення (Responses API)">
    Для прямих моделей OpenAI Responses (`openai/*` у `api.openai.com`) потокова обгортка
    OpenClaw у плагіні OpenAI автоматично вмикає серверне
    ущільнення:

    - Примусово встановлює `store: true` (якщо сумісність моделі не задає `supportsStore: false`)
    - Вставляє `context_management: [{ type: "compaction", compact_threshold: ... }]`
    - Стандартне значення `compact_threshold`: 70% від `contextWindow` (або `80000`, коли
      воно недоступне)

    Це застосовується до вбудованого шляху середовища виконання OpenClaw і до хуків провайдера OpenAI,
    які використовують вбудовані запуски. Нативний серверний каркас застосунку Codex керує
    власним контекстом через Codex, і це налаштування на нього не впливає.

    <Tabs>
      <Tab title="Увімкнути явно">
        Корисно для сумісних кінцевих точок, як-от Azure OpenAI Responses:

        ```json5
        {
          agents: {
            defaults: {
              models: {
                "azure-openai-responses/gpt-5.5": {
                  params: { responsesServerCompaction: true },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="Власне порогове значення">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: {
                    responsesServerCompaction: true,
                    responsesCompactThreshold: 120000,
                  },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="Вимкнути">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: { responsesServerCompaction: false },
                },
              },
            },
          },
        }
        ```
      </Tab>
    </Tabs>

    <Note>
    `responsesServerCompaction` керує лише вставленням `context_management`.
    Прямі моделі OpenAI Responses і надалі примусово встановлюють `store: true`, якщо сумісність
    не задає `supportsStore: false`.
    </Note>

  </Accordion>

  <Accordion title="Суворий агентний режим GPT">
    Для моделей сімейства GPT-5 провайдера `openai`, запущених через вбудоване
    середовище виконання OpenClaw, OpenClaw уже стандартно застосовує суворіший контракт виконання під назвою
    `strict-agentic`. Він автоматично активується, коли визначеним провайдером є
    `openai`, а ідентифікатор моделі відповідає сімейству GPT-5, якщо конфігурація
    явно не вимикає його:

    ```json5
    {
      agents: {
        defaults: {
          embeddedAgent: { executionContract: "default" },
        },
      },
    }
    ```

    Явне встановлення `"strict-agentic"` не має ефекту в підтримуваному каналі (це
    вже значення за замовчуванням) і не діє для непідтримуваних пар провайдера/моделі.

    Коли `strict-agentic` активний, OpenClaw:
    - Автоматично вмикає `update_plan` для значного обсягу роботи
    - Повторює структурно порожні ходи або ходи лише з міркуваннями з продовженням,
      що містить видиму відповідь
    - Використовує явні події плану засобу виконання, коли вибраний засіб виконання
      їх надає

    OpenClaw не класифікує текст асистента, щоб визначити, чи є хід
    планом, оновленням прогресу або остаточною відповіддю.

    <Note>
    Цей контракт повністю реалізовано у вбудованому засобі запуску агента OpenClaw. Він
    не застосовується до нативного засобу виконання app-server Codex, який сам керує
    поведінкою ходів і планів; для нативних запусків Codex вибір засобу виконання має
    більше значення, ніж налаштування контракту виконання.
    </Note>

  </Accordion>

  <Accordion title="Нативні та OpenAI-сумісні маршрути">
    OpenClaw обробляє прямі кінцеві точки OpenAI, Codex і Azure OpenAI
    інакше, ніж універсальні OpenAI-сумісні проксі `/v1`:

    **Нативні маршрути** (`openai/*`, Azure OpenAI):
    - Зберігають `reasoning: { effort: "none" }` лише для моделей, які підтримують
      інтенсивність OpenAI `none`
    - Не додають вимкнені міркування для моделей або проксі, які відхиляють
      `reasoning.effort: "none"`
    - За замовчуванням використовують строгий режим для схем інструментів
    - Додають приховані заголовки атрибуції лише на перевірених нативних хостах (Azure
      OpenAI не отримує ці заголовки, хоча є нативним маршрутом)
    - Зберігають формування запитів, специфічне для OpenAI (`service_tier`, `store`,
      сумісність міркувань, підказки кешу промптів)

    **Проксі-/сумісні маршрути:**
    - Використовують менш строгу поведінку сумісності
    - Вилучають Completions `store` із ненативних корисних навантажень `openai-completions`
    - Приймають розширений наскрізний JSON `params.extra_body`/`params.extraBody`
      для OpenAI-сумісних проксі Completions
    - Приймають `params.chat_template_kwargs` для OpenAI-сумісних проксі Completions,
      як-от vLLM
    - Не примусово застосовують строгі схеми інструментів або заголовки лише для нативних маршрутів

  </Accordion>
</AccordionGroup>
