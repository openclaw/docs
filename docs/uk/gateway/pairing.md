У сполученні, керованому Gateway, **Gateway** є джерелом істини щодо того, яким вузлам дозволено приєднуватися. Інтерфейси користувача (застосунок macOS, майбутні клієнти) — це лише фронтенди, які підтверджують або відхиляють запити, що очікують на розгляд.

**Важливо:** вузли WS використовують **сполучення пристрою** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище сполучення, і воно **не** контролює WS-handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Запит, що очікує на розгляд**: вузол попросив приєднатися; потрібне підтвердження.
- **Сполучений вузол**: підтверджений вузол із виданим токеном автентифікації.
- **Транспорт**: кінцева точка Gateway WS пересилає запити, але не вирішує
  питання членства. (Підтримку застарілого TCP bridge було видалено.)

## Як працює сполучення

1. Вузол підключається до Gateway WS і запитує сполучення.
2. Gateway зберігає **запит, що очікує на розгляд**, і генерує `node.pair.requested`.
3. Ви підтверджуєте або відхиляєте запит (CLI або UI).
4. Після підтвердження Gateway видає **новий токен** (токени ротуються при повторному сполученні).
5. Вузол перепідключається, використовуючи токен, і тепер є «сполученим».

Запити, що очікують на розгляд, автоматично спливають через **5 хвилин**.

## Робочий процес CLI (зручний для headless)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує сполучені/підключені вузли та їхні можливості.

## Поверхня API (протокол gateway)

Події:

- `node.pair.requested` — генерується, коли створено новий запит, що очікує на розгляд.
- `node.pair.resolved` — генерується, коли запит підтверджено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати запит, що очікує на розгляд.
- `node.pair.list` — показати список вузлів, що очікують на розгляд, і сполучених вузлів (`operator.pairing`).
- `node.pair.approve` — підтвердити запит, що очікує на розгляд (видає токен).
- `node.pair.reject` — відхилити запит, що очікує на розгляд.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного вузла: повторні виклики повертають той самий
  запит, що очікує на розгляд.
- Повторні запити для того самого вузла, що очікує на розгляд, також оновлюють збережені метадані вузла
  та останній знімок задекларованих команд із allowlist для видимості оператору.
- Підтвердження **завжди** генерує свіжий токен; токен ніколи не повертається з
  `node.pair.request`.
- Запити можуть містити `silent: true` як підказку для потоків автопідтвердження.
- `node.pair.approve` використовує задекларовані команди запиту, що очікує на розгляд, для застосування
  додаткових областей підтвердження:
  - запит без команд: `operator.pairing`
  - запит команд без exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

Важливо:

- Сполучення вузлів — це потік довіри/ідентичності плюс видача токена.
- Воно **не** закріплює живу поверхню команд вузла для кожного вузла.
- Живі команди вузла надходять із того, що вузол оголошує під час підключення після застосування
  глобальної політики команд вузлів gateway (`gateway.nodes.allowCommands` /
  `denyCommands`).
- Політика allow/ask для `system.run` на рівні окремого вузла зберігається на вузлі в
  `exec.approvals.node.*`, а не в записі сполучення.

## Контроль команд вузла (2026.3.31+)

<Warning>
**Несумісна зміна:** починаючи з `2026.3.31`, команди вузла вимкнено, доки не буде підтверджено сполучення вузла. Самого лише сполучення пристрою більше недостатньо для відкриття задекларованих команд вузла.
</Warning>

Коли вузол підключається вперше, запит на сполучення створюється автоматично. Доки запит на сполучення не буде підтверджено, усі відкладені команди вузла від цього вузла фільтруються й не виконуються. Щойно довіру встановлено через підтвердження сполучення, задекларовані команди вузла стають доступними відповідно до звичайної політики команд.

Це означає:

- Вузли, які раніше покладалися лише на сполучення пристрою для відкриття команд, тепер мають завершити сполучення вузла.
- Команди, поставлені в чергу до підтвердження сполучення, відкидаються, а не відкладаються.

## Межі довіри для подій вузла (2026.3.31+)

<Warning>
**Несумісна зміна:** запуски, ініційовані вузлом, тепер залишаються в межах звуженої довіреної поверхні.
</Warning>

Зведення, ініційовані вузлом, і пов’язані події сеансу обмежені призначеною довіреною поверхнею. Потоки, керовані сповіщеннями або ініційовані вузлом, які раніше покладалися на ширший доступ до інструментів хоста або сеансу, можуть потребувати коригування. Це посилення захисту гарантує, що події вузла не зможуть ескалювати до доступу до інструментів рівня хоста понад межі довіри, дозволені для вузла.

## Автопідтвердження (застосунок macOS)

Застосунок macOS може за бажанням спробувати **тихе підтвердження**, якщо:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-з’єднання з хостом gateway, використовуючи того самого користувача.

Якщо тихе підтвердження не вдається, використовується звичайний запит «Підтвердити/Відхилити».

## Автопідтвердження пристроїв із Trusted-CIDR

WS-сполучення пристроїв для `role: node` типово лишається ручним. Для приватних
мереж вузлів, де Gateway вже довіряє мережевому шляху, оператори можуть
увімкнути це явно за допомогою CIDR або точних IP-адрес:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Межа безпеки:

- Вимкнено, якщо `gateway.nodes.pairing.autoApproveCidrs` не задано.
- Режиму загального автопідтвердження для LAN або приватної мережі не існує.
- Підходить лише свіже сполучення пристрою `role: node` без запитаних областей.
- Клієнти Operator, browser, Control UI і WebChat лишаються ручними.
- Оновлення ролі, області, метаданих і публічного ключа лишаються ручними.
- Шляхи trusted-proxy header loopback того самого хоста не підходять, оскільки
  цей шлях може бути підроблений локальними викликачами.

## Автопідтвердження оновлення метаданих

Коли вже сполучений пристрій перепідключається лише з несуттєвими змінами
метаданих (наприклад, відображуваної назви або підказок щодо клієнтської платформи), OpenClaw трактує
це як `metadata-upgrade`. Тихе автопідтвердження є вузьким: воно застосовується лише
до довірених небраузерних локальних перепідключень, які вже довели володіння локальними
або спільними обліковими даними, зокрема до повторних підключень рідних застосунків на тому самому хості після змін
метаданих версії ОС. Клієнти browser/Control UI та віддалені клієнти, як і раніше,
використовують явний потік повторного підтвердження. Підвищення області (з read до write/admin) і зміни
публічного ключа **не** підходять для автопідтвердження `metadata-upgrade` —
вони залишаються явними запитами на повторне підтвердження.

## Допоміжні засоби QR-сполучення

`/pair qr` відтворює payload сполучення як структуроване медіа, щоб мобільні та
браузерні клієнти могли сканувати його безпосередньо.

Видалення пристрою також очищає будь-які застарілі запити на сполучення, що очікують на розгляд, для цього
ідентифікатора пристрою, тож `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і переслані заголовки

Під час сполучення Gateway вважає з’єднання loopback лише тоді, коли і сирий сокет,
і будь-які свідчення від проксі вище за потоком узгоджуються. Якщо запит надходить через loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`,
які вказують на нелокальне джерело, ці свідчення пересланих заголовків скасовують
твердження про loopback-локальність. Тоді шлях сполучення вимагає явного підтвердження
замість того, щоб тихо трактувати запит як підключення з того самого хоста. Див.
[Trusted Proxy Auth](/uk/gateway/trusted-proxy-auth) для еквівалентного правила в
автентифікації оператора.

## Зберігання (локальне, приватне)

Стан сполучення зберігається в каталозі стану Gateway (типово `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначите `OPENCLAW_STATE_DIR`, папка `nodes/` переміститься разом із ним.

Примітки щодо безпеки:

- Токени є секретами; ставтеся до `paired.json` як до чутливого файлу.
- Ротація токена вимагає повторного підтвердження (або видалення запису вузла).

## Поведінка транспорту

- Транспорт є **безстановим**; він не зберігає відомості про членство.
- Якщо Gateway недоступний або сполучення вимкнено, вузли не можуть виконати сполучення.
- Якщо Gateway працює у віддаленому режимі, сполучення все одно відбувається зі сховищем віддаленого Gateway.
