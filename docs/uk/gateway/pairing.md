У сполученні під керуванням Gateway саме **Gateway** є джерелом істини щодо того, яким Node
дозволено приєднуватися. UI (застосунок macOS, майбутні клієнти) — це лише фронтенди,
які схвалюють або відхиляють запити, що очікують на розгляд.

**Важливо:** WS Node використовують **сполучення пристроїв** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище сполучення, і воно **не** керує WS handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Запит, що очікує на розгляд**: Node попросив приєднатися; потрібне схвалення.
- **Сполучений Node**: схвалений Node з виданим токеном автентифікації.
- **Транспорт**: кінцева точка Gateway WS пересилає запити, але не вирішує
  питання членства. (Підтримку застарілого TCP bridge видалено.)

## Як працює сполучення

1. Node підключається до Gateway WS і запитує сполучення.
2. Gateway зберігає **запит, що очікує на розгляд**, і генерує `node.pair.requested`.
3. Ви схвалюєте або відхиляєте запит (CLI або UI).
4. Після схвалення Gateway видає **новий токен** (токени ротуються під час повторного сполучення).
5. Node перепідключається з використанням токена й тепер вважається «сполученим».

Запити, що очікують на розгляд, автоматично спливають через **5 хвилин**.

## Потік CLI (зручно для headless-режиму)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує сполучені/підключені Node та їхні можливості.

## Поверхня API (протокол gateway)

Події:

- `node.pair.requested` — генерується, коли створюється новий запит, що очікує на розгляд.
- `node.pair.resolved` — генерується, коли запит схвалено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати запит, що очікує на розгляд.
- `node.pair.list` — показати запити, що очікують на розгляд, + сполучені Node (`operator.pairing`).
- `node.pair.approve` — схвалити запит, що очікує на розгляд (видає токен).
- `node.pair.reject` — відхилити запит, що очікує на розгляд.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного Node: повторні виклики повертають той самий
  запит, що очікує на розгляд.
- Повторні запити для того самого Node, що очікує на розгляд, також оновлюють збережені метадані Node
  і найновіший allowlist-знімок оголошених команд для видимості оператора.
- Схвалення **завжди** генерує свіжий токен; токен ніколи не повертається з
  `node.pair.request`.
- Запити можуть містити `silent: true` як підказку для потоків автоматичного схвалення.
- `node.pair.approve` використовує оголошені команди із запиту, що очікує на розгляд, щоб застосувати
  додаткові області схвалення:
  - запит без команд: `operator.pairing`
  - запит із командами, не пов’язаними з exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
Сполучення Node — це потік довіри та ідентичності плюс видача токена. Воно **не** закріплює живу поверхню команд Node для кожного Node.

- Живі команди Node надходять із того, що Node оголошує під час connect після застосування глобальної політики команд Node gateway (`gateway.nodes.allowCommands` і `denyCommands`).
- Політика дозволу та запиту для `system.run` на рівні окремого Node зберігається на самому Node в `exec.approvals.node.*`, а не в записі сполучення.
  </Warning>

## Обмеження команд Node (2026.3.31+)

<Warning>
**Зламна зміна:** починаючи з `2026.3.31`, команди Node вимкнено, доки сполучення Node не буде схвалено. Самого лише сполучення пристрою більше недостатньо для відкриття оголошених команд Node.
</Warning>

Коли Node підключається вперше, запит на сполучення створюється автоматично. Доки цей запит не буде схвалено, усі відкладені команди Node від цього Node фільтруються й не виконуються. Після встановлення довіри через схвалення сполучення оголошені команди Node стають доступними згідно зі звичайною політикою команд.

Це означає:

- Node, які раніше покладалися лише на сполучення пристрою для відкриття команд, тепер повинні завершити сполучення Node.
- Команди, поставлені в чергу до схвалення сполучення, відкидаються, а не відкладаються.

## Межі довіри для подій Node (2026.3.31+)

<Warning>
**Зламна зміна:** запуски, ініційовані Node, тепер залишаються в межах зменшеної довіреної поверхні.
</Warning>

Підсумки, ініційовані Node, і пов’язані події сесії обмежуються призначеною довіреною поверхнею. Потоки, керовані сповіщеннями або запуском із Node, які раніше покладалися на ширший доступ до інструментів хоста чи сесії, можуть потребувати коригування. Це посилення захисту гарантує, що події Node не можуть підвищити привілеї до доступу до інструментів на рівні хоста понад межі довіри, дозволені для цього Node.

## Автосхвалення (застосунок macOS)

Застосунок macOS за потреби може спробувати **тихе схвалення**, якщо:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-з’єднання з хостом gateway під тим самим користувачем.

Якщо тихе схвалення не вдається, використовується звичайне запрошення «Approve/Reject».

## Автосхвалення пристроїв із довірених CIDR

WS-сполучення пристроїв для `role: node` за замовчуванням залишається ручним. Для приватних
мереж Node, де Gateway уже довіряє мережевому шляху, оператори можуть
увімкнути це явно через конкретні CIDR або точні IP-адреси:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Межа безпеки:

- Вимкнено, коли `gateway.nodes.pairing.autoApproveCidrs` не задано.
- Не існує загального режиму автосхвалення для LAN або приватної мережі.
- Підходить лише свіже сполучення пристрою `role: node` без запитаних областей.
- Клієнти operator, browser, Control UI і WebChat залишаються ручними.
- Оновлення ролі, області, метаданих і публічного ключа залишаються ручними.
- Шляхи trusted-proxy headers для loopback на тому самому хості не підходять, тому що
  цей шлях може бути підроблений локальними викликами.

## Автосхвалення оновлення метаданих

Коли вже сполучений пристрій перепідключається лише з несуттєвими змінами
метаданих (наприклад, відображуване ім’я або підказки платформи клієнта), OpenClaw розглядає
це як `metadata-upgrade`. Тихе автосхвалення має вузьку сферу дії: воно застосовується лише до
довірених не-browser локальних перепідключень, які вже довели володіння локальними
або спільними обліковими даними, включно з перепідключеннями нативних застосунків на тому самому хості після змін метаданих версії ОС. Browser/Control UI клієнти та віддалені клієнти, як і раніше, використовують явний потік повторного схвалення. Підвищення області (з читання до write/admin) і зміни публічного ключа **не** підходять для автосхвалення як metadata-upgrade —
вони залишаються явними запитами на повторне схвалення.

## Допоміжні засоби QR-сполучення

`/pair qr` відтворює payload сполучення як структуроване медіа, щоб мобільні та
browser-клієнти могли сканувати його безпосередньо.

Видалення пристрою також очищає будь-які застарілі запити на сполучення для цього
ідентифікатора пристрою, тому `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і переслані заголовки

Gateway pairing вважає з’єднання loopback лише тоді, коли і сирий сокет,
і будь-які дані від upstream proxy узгоджуються. Якщо запит надходить на loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`, які
вказують на нелокальне джерело, ці дані пересланих заголовків анулюють заяву про loopback-локальність. Тоді шлях сполучення вимагає явного схвалення замість
тихого трактування запиту як підключення з того самого хоста. Див.
[Trusted Proxy Auth](/uk/gateway/trusted-proxy-auth) для еквівалентного правила щодо
автентифікації operator.

## Сховище (локальне, приватне)

Стан сполучення зберігається в каталозі стану Gateway (за замовчуванням `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначили `OPENCLAW_STATE_DIR`, тека `nodes/` переміщується разом із ним.

Примітки щодо безпеки:

- Токени є секретами; вважайте `paired.json` чутливим файлом.
- Ротація токена потребує повторного схвалення (або видалення запису Node).

## Поведінка транспорту

- Транспорт є **безстановим**; він не зберігає членство.
- Якщо Gateway офлайн або сполучення вимкнено, Node не можуть виконати сполучення.
- Якщо Gateway працює у віддаленому режимі, сполучення все одно відбувається зі сховищем віддаленого Gateway.
