# Pairing під керуванням Gateway (варіант B)

У pairингy під керуванням **Gateway** саме **Gateway** є джерелом істини щодо того, яким вузлам
дозволено приєднуватися. UI (застосунок macOS, майбутні клієнти) — це лише фронтенди,
які підтверджують або відхиляють очікувальні запити.

**Важливо:** WS-вузли використовують **device pairing** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище pairing і воно **не** керує WS-handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Очікувальний запит**: вузол попросився приєднатися; потребує підтвердження.
- **Сpairений вузол**: підтверджений вузол із виданим токеном автентифікації.
- **Транспорт**: кінцева точка Gateway WS пересилає запити, але не вирішує,
  хто має членство. (Підтримку застарілого TCP bridge було видалено.)

## Як працює pairing

1. Вузол підключається до Gateway WS і запитує pairing.
2. Gateway зберігає **очікувальний запит** і надсилає подію `node.pair.requested`.
3. Ви підтверджуєте або відхиляєте запит (через CLI або UI).
4. Після підтвердження Gateway видає **новий токен** (токени ротуються під час повторного pairing).
5. Вузол перепідключається з використанням токена і тепер є «спареним».

Очікувальні запити автоматично спливають через **5 хвилин**.

## Потік CLI (зручний для headless)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує спарені/підключені вузли та їхні можливості.

## Поверхня API (протокол gateway)

Події:

- `node.pair.requested` — надсилається, коли створено новий очікувальний запит.
- `node.pair.resolved` — надсилається, коли запит підтверджено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати очікувальний запит.
- `node.pair.list` — показати очікувальні + спарені вузли (`operator.pairing`).
- `node.pair.approve` — підтвердити очікувальний запит (видає токен).
- `node.pair.reject` — відхилити очікувальний запит.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного вузла: повторні виклики повертають той самий
  очікувальний запит.
- Повторні запити для того самого очікувального вузла також оновлюють збережені метадані вузла
  і найсвіжіший знімок оголошених команд зі списку дозволених для видимості оператору.
- Підтвердження **завжди** генерує свіжий токен; жоден токен ніколи не повертається з
  `node.pair.request`.
- Запити можуть включати `silent: true` як підказку для потоків автопідтвердження.
- `node.pair.approve` використовує оголошені команди очікувального запиту, щоб застосувати
  додаткові області підтвердження:
  - запит без команд: `operator.pairing`
  - запит команди без exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

Важливо:

- Pairing вузла — це потік довіри/ідентичності плюс видача токена.
- Він **не** фіксує активну поверхню команд вузла для кожного вузла.
- Активні команди вузла беруться з того, що вузол оголошує під час підключення після застосування
  глобальної політики команд вузлів gateway (`gateway.nodes.allowCommands` /
  `denyCommands`).
- Політика дозволу/підтвердження `system.run` для кожного вузла живе на самому вузлі в
  `exec.approvals.node.*`, а не в записі pairing.

## Керування командами вузлів (2026.3.31+)

<Warning>
**Несумісна зміна:** починаючи з `2026.3.31`, команди вузлів вимкнено, доки не буде підтверджено pairing вузла. Одного лише device pairing більше недостатньо, щоб відкрити оголошені команди вузла.
</Warning>

Коли вузол підключається вперше, pairing запитується автоматично. Доки запит на pairing не буде підтверджено, усі очікувальні команди вузла від цього вузла фільтруються і не виконуються. Після встановлення довіри через підтвердження pairing оголошені команди вузла стають доступними відповідно до звичайної політики команд.

Це означає:

- Вузли, які раніше покладалися лише на device pairing для відкриття команд, тепер мають завершити pairing вузла.
- Команди, поставлені в чергу до підтвердження pairing, відкидаються, а не відкладаються.

## Межі довіри для подій вузлів (2026.3.31+)

<Warning>
**Несумісна зміна:** запуски, що походять від вузлів, тепер залишаються в межах скороченої довіреної поверхні.
</Warning>

Зведення, що походять від вузлів, і пов’язані події сесій обмежені призначеною довіреною поверхнею. Потоки, ініційовані сповіщеннями або вузлами, які раніше покладалися на ширший доступ до інструментів хоста або сесії, можуть потребувати коригування. Це посилення безпеки гарантує, що події вузлів не можуть підвищити рівень доступу до інструментів хоста понад те, що дозволяє межа довіри вузла.

## Автопідтвердження (застосунок macOS)

Застосунок macOS може за бажанням спробувати **тихе підтвердження**, якщо:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-підключення до хоста gateway від імені того самого користувача.

Якщо тихе підтвердження не вдається, виконується повернення до звичайного запиту «Підтвердити/Відхилити».

## Автопідтвердження оновлення метаданих

Коли вже спарений пристрій перепідключається лише зі змінами несутливих метаданих
(наприклад, відображуваної назви або підказок щодо платформи клієнта), OpenClaw розглядає
це як `metadata-upgrade`. Тихе автопідтвердження є вузьким: воно застосовується лише
до довірених локальних перепідключень CLI/допоміжних засобів, які вже довели володіння
спільним токеном або паролем через loopback. Клієнти браузера/Control UI і віддалені
клієнти, як і раніше, використовують явний потік повторного підтвердження. Підвищення рівня доступу
(read до write/admin) і зміни відкритих ключів **не** підпадають під автопідтвердження
`metadata-upgrade` — для них і далі потрібен явний запит на повторне підтвердження.

## Допоміжні засоби QR pairing

`/pair qr` відтворює корисне навантаження pairing як структуровані медіадані, щоб мобільні й
браузерні клієнти могли сканувати його безпосередньо.

Видалення пристрою також очищає будь-які застарілі очікувальні запити pairing для цього
ідентифікатора пристрою, тому `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і forwarded headers

Gateway pairing вважає підключення loopback лише тоді, коли збігаються і сирий сокет,
і будь-які докази від проксі вище по ланцюгу. Якщо запит надходить через loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`,
які вказують на нелокальне джерело, ці докази через forwarded headers скасовують
твердження про loopback-локальність. Тоді шлях pairing вимагає явного підтвердження,
замість того щоб мовчки вважати запит підключенням з того самого хоста. Див.
[Trusted Proxy Auth](/uk/gateway/trusted-proxy-auth) для еквівалентного правила щодо
автентифікації оператора.

## Сховище (локальне, приватне)

Стан pairing зберігається в каталозі стану Gateway (типово `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначите `OPENCLAW_STATE_DIR`, тека `nodes/` переміститься разом із ним.

Примітки щодо безпеки:

- Токени є секретами; ставтеся до `paired.json` як до чутливого файлу.
- Ротація токена вимагає повторного підтвердження (або видалення запису вузла).

## Поведінка транспорту

- Транспорт є **безстановим**; він не зберігає інформацію про членство.
- Якщо Gateway офлайн або pairing вимкнено, вузли не можуть виконати pairing.
- Якщо Gateway працює у віддаленому режимі, pairing усе одно відбувається відносно сховища віддаленого Gateway.
