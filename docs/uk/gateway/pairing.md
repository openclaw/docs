У зіставленні, яким володіє Gateway, **Gateway** є джерелом істини щодо того, яким Node
дозволено приєднуватися. UI (застосунок macOS, майбутні клієнти) — це лише фронтенди, які
підтверджують або відхиляють очікувальні запити.

**Важливо:** WS Node використовують **зіставлення пристроїв** (role `node`) під час `connect`.
`node.pair.*` — це окреме сховище зіставлення і воно **не** керує WS handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Очікувальний запит**: Node запитав приєднання; потрібне підтвердження.
- **Зіставлений Node**: підтверджений Node із виданим auth-токеном.
- **Транспорт**: WS endpoint Gateway пересилає запити, але не вирішує
  членство. (Підтримку застарілого TCP bridge видалено.)

## Як працює зіставлення

1. Node підключається до WS Gateway і запитує зіставлення.
2. Gateway зберігає **очікувальний запит** і генерує `node.pair.requested`.
3. Ви підтверджуєте або відхиляєте запит (CLI або UI).
4. Після підтвердження Gateway видає **новий токен** (токени ротуються під час повторного зіставлення).
5. Node повторно підключається, використовуючи токен, і тепер вважається «зіставленим».

Очікувальні запити автоматично спливають через **5 хвилин**.

## Потік CLI (дружній до headless)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує зіставлені/підключені Node і їхні можливості.

## Поверхня API (протокол Gateway)

Події:

- `node.pair.requested` — генерується, коли створюється новий очікувальний запит.
- `node.pair.resolved` — генерується, коли запит підтверджено/відхилено/його строк дії сплив.

Методи:

- `node.pair.request` — створити або повторно використати очікувальний запит.
- `node.pair.list` — перелічити очікувальні + зіставлені Node (`operator.pairing`).
- `node.pair.approve` — підтвердити очікувальний запит (видає токен).
- `node.pair.reject` — відхилити очікувальний запит.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного Node: повторні виклики повертають той самий
  очікувальний запит.
- Повторні запити для того самого очікувального Node також оновлюють збережені метадані Node
  і найновіший знімок заявлених команд із allowlist для видимості оператора.
- Підтвердження **завжди** генерує новий токен; токен ніколи не повертається з
  `node.pair.request`.
- Запити можуть містити `silent: true` як підказку для потоків автоматичного підтвердження.
- `node.pair.approve` використовує заявлені команди очікувального запиту, щоб застосувати
  додаткові області підтвердження:
  - запит без команд: `operator.pairing`
  - запит команд без exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

Важливо:

- Зіставлення Node — це потік довіри/ідентичності плюс видача токена.
- Воно **не** фіксує поточну поверхню команд Node для кожного Node.
- Поточні команди Node надходять із того, що Node заявляє під час connect після
  застосування глобальної політики команд Node gateway (`gateway.nodes.allowCommands` /
  `denyCommands`).
- Політика allow/ask для `system.run` на рівні окремого Node живе на самому Node в
  `exec.approvals.node.*`, а не в записі зіставлення.

## Керування командами Node (2026.3.31+)

<Warning>
**Несумісна зміна:** Починаючи з `2026.3.31`, команди Node вимкнені, доки не буде підтверджено зіставлення Node. Самого лише зіставлення пристрою більше недостатньо, щоб відкрити заявлені команди Node.
</Warning>

Коли Node підключається вперше, запит на зіставлення створюється автоматично. Доки запит на зіставлення не буде підтверджено, усі очікувальні команди Node від цього Node фільтруються і не виконуються. Щойно довіру встановлено через підтвердження зіставлення, заявлені команди Node стають доступними за звичайною політикою команд.

Це означає:

- Node, які раніше покладалися лише на зіставлення пристрою для відкриття команд, тепер мають завершити зіставлення Node.
- Команди, поставлені в чергу до підтвердження зіставлення, відкидаються, а не відкладаються.

## Межі довіри подій Node (2026.3.31+)

<Warning>
**Несумісна зміна:** Запуски, ініційовані Node, тепер залишаються на зменшеній довіреній поверхні.
</Warning>

Зведення, ініційовані Node, і пов’язані події сесії обмежені призначеною довіреною поверхнею. Потоки, керовані сповіщеннями або тригерені Node, які раніше покладалися на ширший доступ до host або session tools, можуть потребувати коригування. Це посилення гарантує, що події Node не можуть підвищити привілеї до доступу до tools рівня host понад межі довіри, дозволені для Node.

## Автоматичне підтвердження (застосунок macOS)

Застосунок macOS за бажання може спробувати **тихе підтвердження**, коли:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-з’єднання з host Gateway, використовуючи того самого користувача.

Якщо тихе підтвердження не вдається, відбувається повернення до звичайного запиту «Підтвердити/Відхилити».

## Автоматичне підтвердження пристроїв для довірених CIDR

WS-зіставлення пристроїв для `role: node`, як і раніше, типово виконується вручну. Для приватних
мереж Node, де Gateway уже довіряє мережевому шляху, оператори можуть
увімкнути це явно через CIDR або точні IP:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Межа безпеки:

- Вимкнено, коли `gateway.nodes.pairing.autoApproveCidrs` не задано.
- Не існує загального режиму автоматичного підтвердження для LAN або приватної мережі.
- Право мають лише нові зіставлення пристроїв `role: node` без запитаних scopes.
- Клієнти Operator, browser, Control UI і WebChat залишаються ручними.
- Оновлення role, scope, metadata і public key залишаються ручними.
- Шляхи заголовків trusted proxy через loopback на тому самому host не мають права на це, бо цей
  шлях може бути підроблений локальними викликачами.

## Автоматичне підтвердження оновлення metadata

Коли вже зіставлений пристрій підключається знову лише зі змінами нечутливих
metadata (наприклад, display name або підказок платформи клієнта), OpenClaw розглядає
це як `metadata-upgrade`. Тихе автоматичне підтвердження є вузьким: воно застосовується лише до
довірених локальних CLI/helper повторних підключень, які вже довели володіння
спільним токеном або паролем через loopback. Клієнти browser/Control UI і віддалені
клієнти, як і раніше, використовують явний потік повторного підтвердження. Оновлення scope (з read до
write/admin) і зміни public key **не** мають права на автоматичне підтвердження `metadata-upgrade` —
вони залишаються явними запитами на повторне підтвердження.

## Допоміжні засоби QR-зіставлення

`/pair qr` відтворює payload зіставлення як структуровані медіадані, щоб мобільні й
browser-клієнти могли сканувати його безпосередньо.

Видалення пристрою також очищає всі застарілі очікувальні запити на зіставлення для цього
id пристрою, тож `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і переслані заголовки

Під час зіставлення Gateway вважає з’єднання loopback лише тоді, коли і raw socket,
і будь-які свідчення проксі вище за ланцюжком збігаються. Якщо запит надходить через loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`, які вказують
на нелокальне джерело, ці свідчення пересланих заголовків скасовують твердження про локальність loopback.
Тоді шлях зіставлення вимагає явного підтвердження замість
тихого трактування запиту як підключення з того самого host. Див.
[Автентифікація Trusted Proxy](/uk/gateway/trusted-proxy-auth) для еквівалентного правила щодо
auth оператора.

## Зберігання (локальне, приватне)

Стан зіставлення зберігається в каталозі стану Gateway (типово `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначаєте `OPENCLAW_STATE_DIR`, теку `nodes/` буде перенесено разом із ним.

Примітки щодо безпеки:

- Токени — це секрети; ставтеся до `paired.json` як до чутливого файлу.
- Ротація токена вимагає повторного підтвердження (або видалення запису Node).

## Поведінка транспорту

- Транспорт є **безстановим**; він не зберігає членство.
- Якщо Gateway офлайн або зіставлення вимкнено, Node не можуть зіставлятися.
- Якщо Gateway працює у віддаленому режимі, зіставлення все одно відбувається зі сховищем віддаленого Gateway.
