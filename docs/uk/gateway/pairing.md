У сполученні під керуванням Gateway саме **Gateway** є джерелом істини щодо того, яким Node дозволено приєднуватися. UI (застосунок macOS, майбутні клієнти) — це лише фронтенди, які схвалюють або відхиляють запити, що очікують на розгляд.

**Важливо:** WS Node використовують **сполучення пристроїв** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище сполучення, і воно **не** керує WS-handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Запит, що очікує на розгляд**: Node надіслав запит на приєднання; потрібне схвалення.
- **Сполучений Node**: схвалений Node із виданим токеном автентифікації.
- **Транспорт**: кінцева точка Gateway WS пересилає запити, але не вирішує
  питання членства. (Підтримку застарілого мосту TCP видалено.)

## Як працює сполучення

1. Node підключається до Gateway WS і запитує сполучення.
2. Gateway зберігає **запит, що очікує на розгляд**, і надсилає `node.pair.requested`.
3. Ви схвалюєте або відхиляєте запит (через CLI або UI).
4. Після схвалення Gateway видає **новий токен** (токени ротуються при повторному сполученні).
5. Node перепідключається, використовуючи токен, і тепер вважається “сполученим”.

Запити, що очікують на розгляд, автоматично спливають через **5 хвилин**.

## Потік CLI (зручний для headless-середовищ)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує сполучені/підключені Node та їхні можливості.

## Поверхня API (протокол gateway)

Події:

- `node.pair.requested` — надсилається, коли створюється новий запит, що очікує на розгляд.
- `node.pair.resolved` — надсилається, коли запит схвалено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати запит, що очікує на розгляд.
- `node.pair.list` — показати список Node, що очікують на розгляд + сполучених Node (`operator.pairing`).
- `node.pair.approve` — схвалити запит, що очікує на розгляд (видає токен).
- `node.pair.reject` — відхилити запит, що очікує на розгляд.
- `node.pair.remove` — видалити застарілий запис сполученого Node.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного Node: повторні виклики повертають той самий запит, що очікує на розгляд.
- Повторні запити для того самого Node, що очікує на розгляд, також оновлюють збережені метадані Node
  та останній знімок оголошених команд зі списку дозволених для видимості оператору.
- Схвалення **завжди** генерує новий токен; `node.pair.request` ніколи не повертає токен.
- Запити можуть містити `silent: true` як підказку для потоків авто-схвалення.
- `node.pair.approve` використовує оголошені команди запиту, що очікує на розгляд, для застосування
  додаткових областей схвалення:
  - запит без команд: `operator.pairing`
  - запит команди без exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
Сполучення Node — це потік довіри й ідентичності плюс видача токенів. Воно **не** закріплює live-поверхню команд Node для кожного Node.

- Live-команди Node надходять із того, що Node оголошує під час connect після застосування глобальної політики команд Node Gateway (`gateway.nodes.allowCommands` і `denyCommands`).
- Політика allow і ask для `system.run` на рівні окремого Node зберігається на Node у `exec.approvals.node.*`, а не в записі сполучення.
  </Warning>

## Керування доступом до команд Node (2026.3.31+)

<Warning>
**Несумісна зміна:** Починаючи з `2026.3.31`, команди Node вимкнені, доки не схвалено сполучення Node. Самого лише сполучення пристрою більше недостатньо, щоб показувати оголошені команди Node.
</Warning>

Коли Node підключається вперше, запит на сполучення створюється автоматично. Доки запит на сполучення не схвалено, усі незавершені команди Node від цього Node відфільтровуються і не виконуються. Щойно довіру встановлено через схвалення сполучення, оголошені команди Node стають доступними відповідно до звичайної політики команд.

Це означає:

- Node, які раніше покладалися лише на сполучення пристрою для відкриття команд, тепер мають завершити сполучення Node.
- Команди, поставлені в чергу до схвалення сполучення, відкидаються, а не відкладаються.

## Межі довіри для подій Node (2026.3.31+)

<Warning>
**Несумісна зміна:** Запуски, що походять від Node, тепер залишаються в межах зменшеної довіреної поверхні.
</Warning>

Підсумки, що походять від Node, і пов’язані події сесії обмежені призначеною довіреною поверхнею. Потоки, керовані сповіщеннями або тригерами Node, які раніше покладалися на ширший доступ до інструментів хоста чи сесії, можуть потребувати коригування. Це посилення гарантує, що події Node не можуть підвищити привілеї до доступу до інструментів рівня хоста понад межі довіри, дозволені для цього Node.

## Авто-схвалення (застосунок macOS)

Застосунок macOS за бажанням може спробувати **тихе схвалення**, коли:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-з’єднання з хостом gateway, використовуючи того самого користувача.

Якщо тихе схвалення не вдається, використовується звичайний запит “Схвалити/Відхилити”.

## Авто-схвалення пристроїв із довірених CIDR

WS-сполучення пристроїв для `role: node` типово лишається ручним. Для приватних
мереж Node, де Gateway уже довіряє мережевому шляху, оператори можуть
увімкнути це явно за допомогою CIDR або точних IP:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Межа безпеки:

- Вимкнено, коли `gateway.nodes.pairing.autoApproveCidrs` не задано.
- Загального режиму авто-схвалення для LAN або приватної мережі не існує.
- Право мають лише нові сполучення пристроїв `role: node` без запитаних областей.
- Клієнти Operator, Browser, Control UI і WebChat лишаються ручними.
- Оновлення ролі, області, метаданих і публічного ключа лишаються ручними.
- Шляхи trusted-proxy header з loopback на тому самому хості не мають права,
  оскільки цей шлях може бути підроблено локальними викликами.

## Авто-схвалення оновлення метаданих

Коли вже сполучений пристрій перепідключається лише зі змінами нечутливих
метаданих (наприклад, відображуваного імені або підказок про платформу клієнта), OpenClaw вважає
це `metadata-upgrade`. Тихе авто-схвалення є вузьким: воно застосовується лише
до довірених локальних перепідключень не з браузера, які вже довели володіння локальними
або спільними обліковими даними, включно з повторними підключеннями native-застосунків на тому самому хості після змін метаданих версії ОС. Клієнти Browser/Control UI та віддалені клієнти, як і раніше,
використовують явний потік повторного схвалення. Оновлення області (від read до write/admin) і
зміни публічного ключа **не** мають права на авто-схвалення оновлення метаданих —
вони лишаються явними запитами на повторне схвалення.

## Допоміжні засоби QR-сполучення

`/pair qr` відображає payload сполучення як структуровані медіадані, щоб мобільні та
браузерні клієнти могли сканувати його безпосередньо.

Видалення пристрою також прибирає всі застарілі запити на сполучення, що очікують на розгляд, для цього
ідентифікатора пристрою, тому `nodes pending` не показує осиротілих рядків після відкликання.

## Локальність і переслані заголовки

Під час сполучення Gateway вважає з’єднання loopback лише тоді, коли і сирий сокет,
і будь-які докази від проксі вище за ланцюгом узгоджуються. Якщо запит надходить через loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`,
які вказують на нелокальне джерело, ці докази з пересланих заголовків скасовують
твердження про loopback-локальність. Тоді шлях сполучення вимагає явного схвалення
замість того, щоб неявно вважати запит підключенням із того самого хоста. Див.
[Trusted Proxy Auth](/uk/gateway/trusted-proxy-auth) для еквівалентного правила щодо
автентифікації оператора.

## Сховище (локальне, приватне)

Стан сполучення зберігається в каталозі стану Gateway (типово `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначили `OPENCLAW_STATE_DIR`, папка `nodes/` переміщується разом із ним.

Примітки щодо безпеки:

- Токени — це секрети; вважайте `paired.json` чутливим.
- Ротація токена вимагає повторного схвалення (або видалення запису Node).

## Поведінка транспорту

- Транспорт **не має стану**; він не зберігає членство.
- Якщо Gateway офлайн або сполучення вимкнено, Node не можуть виконати сполучення.
- Якщо Gateway працює у віддаленому режимі, сполучення все одно відбувається зі сховищем віддаленого Gateway.
