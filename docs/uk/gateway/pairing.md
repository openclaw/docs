У паруванні, яким володіє Gateway, **Gateway** є джерелом істини щодо того, яким вузлам
дозволено приєднуватися. Інтерфейси користувача (застосунок macOS, майбутні клієнти) — це лише фронтенди, які
схвалюють або відхиляють очікувані запити.

**Важливо:** вузли WS використовують **парування пристрою** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище парування, і воно **не** блокує рукостискання WS.
Лише клієнти, які явно викликають `node.pair.*`, використовують цей потік.

## Поняття

- **Очікуваний запит**: вузол попросив приєднатися; потребує схвалення.
- **Спарований вузол**: схвалений вузол з виданим токеном автентифікації.
- **Транспорт**: кінцева точка WS Gateway пересилає запити, але не вирішує
  членство. (Підтримку застарілого TCP-моста вилучено.)

## Як працює парування

1. Вузол підключається до WS Gateway і запитує парування.
2. Gateway зберігає **очікуваний запит** і надсилає `node.pair.requested`.
3. Ви схвалюєте або відхиляєте запит (CLI або UI).
4. Після схвалення Gateway видає **новий токен** (токени ротуються під час повторного парування).
5. Вузол повторно підключається з токеном і тепер є «спарованим».

Очікувані запити автоматично спливають через **5 хвилин**.

## Робочий процес CLI (зручний для headless-середовищ)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує спаровані/підключені вузли та їхні можливості.

## Поверхня API (протокол Gateway)

Події:

- `node.pair.requested` — надсилається, коли створюється новий очікуваний запит.
- `node.pair.resolved` — надсилається, коли запит схвалено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати очікуваний запит.
- `node.pair.list` — перелічити очікувані + спаровані вузли (`operator.pairing`).
- `node.pair.approve` — схвалити очікуваний запит (видає токен).
- `node.pair.reject` — відхилити очікуваний запит.
- `node.pair.remove` — вилучити застарілий запис спарованого вузла.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного вузла: повторні виклики повертають той самий
  очікуваний запит.
- Повторні запити для того самого очікуваного вузла також оновлюють збережені метадані вузла
  і найновіший знімок дозволених оголошених команд для видимості оператору.
- Схвалення **завжди** генерує свіжий токен; `node.pair.request` ніколи не повертає токен.
- Запити можуть містити `silent: true` як підказку для потоків автоматичного схвалення.
- `node.pair.approve` використовує оголошені команди очікуваного запиту, щоб застосувати
  додаткові області схвалення:
  - запит без команд: `operator.pairing`
  - запит команди не exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
Парування вузла — це потік довіри та ідентичності плюс видача токена. Воно **не** закріплює живу поверхню команд вузла за кожним вузлом.

- Живі команди вузла походять із того, що вузол оголошує під час підключення після застосування глобальної політики команд вузлів gateway (`gateway.nodes.allowCommands` і `denyCommands`).
- Політика дозволу та запиту для `system.run` на рівні вузла живе на вузлі в `exec.approvals.node.*`, а не в записі парування.

</Warning>

## Обмеження команд вузлів (2026.3.31+)

<Warning>
**Критична зміна:** Починаючи з `2026.3.31`, команди вузлів вимкнені, доки парування вузла не буде схвалено. Самого парування пристрою більше недостатньо, щоб відкрити оголошені команди вузла.
</Warning>

Коли вузол підключається вперше, парування запитується автоматично. Доки запит парування не схвалено, усі очікувані команди вузла з цього вузла фільтруються і не виконуватимуться. Після встановлення довіри через схвалення парування оголошені вузлом команди стають доступними з урахуванням звичайної політики команд.

Це означає:

- Вузли, які раніше покладалися лише на парування пристрою для відкриття команд, тепер мають завершити парування вузла.
- Команди, поставлені в чергу до схвалення парування, відкидаються, а не відкладаються.

## Межі довіри подій вузлів (2026.3.31+)

<Warning>
**Критична зміна:** Запуски, ініційовані вузлом, тепер залишаються на зменшеній довіреній поверхні.
</Warning>

Зведення, ініційовані вузлом, і пов’язані події сеансу обмежені призначеною довіреною поверхнею. Потоки, керовані сповіщеннями або ініційовані вузлом, які раніше покладалися на ширший доступ до інструментів хоста чи сеансу, можуть потребувати коригування. Це посилення гарантує, що події вузла не можуть підвищити доступ до інструментів рівня хоста понад те, що дозволяє межа довіри вузла.

Стійкі оновлення присутності вузла дотримуються тієї самої межі ідентичності. Подія `node.presence.alive`
приймається лише від автентифікованих сеансів пристрою вузла і оновлює метадані парування лише тоді, коли
ідентичність пристрою/вузла вже спарована. Самостійно оголошених значень `client.id` недостатньо, щоб записувати
стан останньої активності.

## Автоматичне схвалення (застосунок macOS)

Застосунок macOS може необов’язково спробувати **тихе схвалення**, коли:

- запит позначено як `silent`, і
- застосунок може перевірити SSH-підключення до хоста gateway з використанням того самого користувача.

Якщо тихе схвалення не вдається, він повертається до звичайного запиту «Схвалити/Відхилити».

## Автоматичне схвалення пристрою за довіреним CIDR

Парування WS-пристрою для `role: node` за замовчуванням залишається ручним. Для приватних
мереж вузлів, де Gateway уже довіряє мережевому шляху, оператори можуть
увімкнути явні CIDR або точні IP-адреси:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Межа безпеки:

- Вимкнено, коли `gateway.nodes.pairing.autoApproveCidrs` не задано.
- Немає режиму автоматичного схвалення для всієї LAN або приватної мережі.
- Придатне лише свіже парування пристрою `role: node` без запитаних областей.
- Клієнти оператора, браузера, Control UI і WebChat залишаються ручними.
- Роль, область, метадані та оновлення публічного ключа залишаються ручними.
- Шляхи заголовків довіреного проксі same-host loopback не придатні, тому що цей
  шлях може бути підроблений локальними викликачами.

## Автоматичне схвалення оновлення метаданих

Коли вже спарований пристрій повторно підключається лише зі змінами нечутливих метаданих
(наприклад, відображуваного імені або підказок платформи клієнта), OpenClaw трактує
це як `metadata-upgrade`. Тихе автоматичне схвалення вузьке: воно застосовується лише
до довірених небраузерних локальних повторних підключень, які вже довели володіння локальними
або спільними обліковими даними, включно з повторними підключеннями нативного застосунку same-host після змін
метаданих версії ОС. Клієнти браузера/Control UI та віддалені клієнти все ще
використовують явний потік повторного схвалення. Підвищення областей (з читання до запису/адміністрування) і
зміни публічного ключа **не** придатні для автоматичного схвалення metadata-upgrade —
вони залишаються явними запитами повторного схвалення.

## Допоміжні засоби QR-парування

`/pair qr` відображає корисне навантаження парування як структуровані медіа, щоб мобільні та
браузерні клієнти могли сканувати його напряму.

Видалення пристрою також прибирає будь-які застарілі очікувані запити парування для цього
ідентифікатора пристрою, тому `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і переслані заголовки

Парування Gateway трактує підключення як loopback лише тоді, коли і необроблений сокет,
і будь-які докази від upstream-проксі узгоджуються. Якщо запит надходить на loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`,
які вказують на нелокальне походження, такі докази з пересланих заголовків дискваліфікують
твердження про локальність loopback. Тоді шлях парування потребує явного схвалення,
замість того щоб тихо трактувати запит як підключення same-host. Див.
[Автентифікація довіреного проксі](/uk/gateway/trusted-proxy-auth) для еквівалентного правила щодо
автентифікації оператора.

## Зберігання (локальне, приватне)

Стан парування зберігається в каталозі стану Gateway (за замовчуванням `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначите `OPENCLAW_STATE_DIR`, папка `nodes/` переміститься разом із ним.

Примітки щодо безпеки:

- Токени є секретами; вважайте `paired.json` чутливим.
- Ротація токена потребує повторного схвалення (або видалення запису вузла).

## Поведінка транспорту

- Транспорт є **безстанним**; він не зберігає членство.
- Якщо Gateway офлайн або парування вимкнено, вузли не можуть паруватися.
- Якщо Gateway перебуває у віддаленому режимі, парування все одно відбувається зі сховищем віддаленого Gateway.
