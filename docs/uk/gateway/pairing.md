У pairing під керуванням Gateway саме **Gateway** є джерелом істини щодо того,
яким вузлам дозволено приєднуватися. UI (macOS app, майбутні клієнти) — це лише фронтенди,
які підтверджують або відхиляють запити, що очікують на розгляд.

**Важливо:** WS-вузли використовують **pairing пристрою** (роль `node`) під час `connect`.
`node.pair.*` — це окреме сховище pairing і воно **не** керує WS-handshake.
Цей потік використовують лише клієнти, які явно викликають `node.pair.*`.

## Поняття

- **Запит, що очікує на розгляд**: вузол попросив приєднатися; потрібне підтвердження.
- **Спарений вузол**: підтверджений вузол із виданим токеном автентифікації.
- **Транспорт**: WS endpoint Gateway пересилає запити, але не вирішує
  членство. (Підтримку застарілого TCP bridge було вилучено.)

## Як працює pairing

1. Вузол підключається до Gateway WS і запитує pairing.
2. Gateway зберігає **запит, що очікує на розгляд**, і надсилає `node.pair.requested`.
3. Ви підтверджуєте або відхиляєте запит (через CLI або UI).
4. Після підтвердження Gateway видає **новий токен** (токени обертаються під час повторного pairing).
5. Вузол перепідключається з використанням токена і тепер вважається “спареним”.

Запити, що очікують на розгляд, автоматично спливають через **5 хвилин**.

## Потік CLI (зручно для headless)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показує спарені/підключені вузли та їхні можливості.

## API surface (протокол gateway)

Події:

- `node.pair.requested` — надсилається, коли створюється новий запит, що очікує на розгляд.
- `node.pair.resolved` — надсилається, коли запит підтверджено/відхилено/прострочено.

Методи:

- `node.pair.request` — створити або повторно використати запит, що очікує на розгляд.
- `node.pair.list` — показати список вузлів, що очікують на розгляд, і спарених вузлів (`operator.pairing`).
- `node.pair.approve` — підтвердити запит, що очікує на розгляд (видає токен).
- `node.pair.reject` — відхилити запит, що очікує на розгляд.
- `node.pair.verify` — перевірити `{ nodeId, token }`.

Примітки:

- `node.pair.request` є ідемпотентним для кожного вузла: повторні виклики повертають той самий
  запит, що очікує на розгляд.
- Повторні запити для того самого вузла, що очікує на розгляд, також оновлюють збережені метадані вузла
  і останній знімок оголошених команд з allowlist для видимості оператору.
- Підтвердження **завжди** генерує новий токен; жоден токен ніколи не повертається з
  `node.pair.request`.
- Запити можуть включати `silent: true` як підказку для потоків автоматичного підтвердження.
- `node.pair.approve` використовує оголошені команди із запиту, що очікує на розгляд, для застосування
  додаткових областей підтвердження:
  - запит без команд: `operator.pairing`
  - запит команд без exec: `operator.pairing` + `operator.write`
  - запит `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

Важливо:

- Pairing вузла — це потік довіри/ідентичності плюс видача токена.
- Він **не** закріплює живу surface команд вузла для кожного вузла.
- Живі команди вузла надходять із того, що вузол оголошує під час connect після
  застосування глобальної політики команд вузлів gateway (`gateway.nodes.allowCommands` /
  `denyCommands`).
- Політика allow/ask для `system.run` на рівні окремого вузла розміщується на самому вузлі в
  `exec.approvals.node.*`, а не в записі pairing.

## Обмеження команд вузла (2026.3.31+)

<Warning>
**Несумісна зміна:** Починаючи з `2026.3.31`, команди вузлів вимикаються, доки pairing вузла не буде підтверджено. Лише pairing пристрою більше недостатньо для відкриття оголошених команд вузла.
</Warning>

Коли вузол підключається вперше, pairing запитується автоматично. Поки запит pairing не буде підтверджено, усі очікувані команди вузла з цього вузла фільтруються і не виконуються. Щойно довіру буде встановлено через підтвердження pairing, оголошені команди вузла стануть доступними відповідно до звичайної політики команд.

Це означає:

- Вузли, які раніше покладалися лише на pairing пристрою для відкриття команд, тепер мають завершити pairing вузла.
- Команди, поставлені в чергу до підтвердження pairing, відкидаються, а не відкладаються.

## Межі довіри подій вузла (2026.3.31+)

<Warning>
**Несумісна зміна:** Запуски, ініційовані вузлом, тепер залишаються в межах зменшеної довіреної surface.
</Warning>

Підсумки, ініційовані вузлом, і пов’язані події сесії обмежуються призначеною довіреною surface. Потоки, що запускаються сповіщеннями або вузлом і раніше покладалися на ширший доступ до інструментів хоста чи сесії, можуть потребувати коригування. Це посилення захисту гарантує, що події вузла не зможуть підвищити привілеї до доступу до інструментів рівня хоста понад межі довіри, дозволені для цього вузла.

## Автоматичне підтвердження (macOS app)

macOS app за бажанням може спробувати **тихе підтвердження**, якщо:

- запит позначено як `silent`, і
- app може перевірити SSH-підключення до хоста gateway від імені того самого користувача.

Якщо тихе підтвердження не вдається, виконується резервний перехід до звичайного запиту “Підтвердити/Відхилити”.

## Автоматичне підтвердження оновлення метаданих

Коли вже спарений пристрій перепідключається лише з несуттєвими змінами метаданих
(наприклад, відображувана назва або підказки щодо платформи клієнта), OpenClaw розглядає
це як `metadata-upgrade`. Тихе автоматичне підтвердження є вузьким: воно застосовується лише до
довірених локальних перепідключень CLI/допоміжних інструментів, які вже довели володіння
спільним токеном або паролем через local loopback. Browser/Control UI клієнти та віддалені
клієнти й надалі використовують явний потік повторного підтвердження. Розширення scope (від read до
write/admin) і зміни відкритого ключа **не** підпадають під автоматичне підтвердження
`metadata-upgrade` — вони залишаються явними запитами на повторне підтвердження.

## Допоміжні засоби QR pairing

`/pair qr` візуалізує payload pairing як структуроване медіа, щоб мобільні та
browser-клієнти могли сканувати його безпосередньо.

Видалення пристрою також очищає всі застарілі запити pairing, що очікують на розгляд, для цього id
пристрою, тому `nodes pending` не показує осиротілі рядки після відкликання.

## Локальність і forwarded headers

Gateway pairing розглядає з’єднання як loopback лише тоді, коли і raw socket,
і будь-які докази від проксі вище за ланцюжком узгоджуються між собою. Якщо запит надходить через loopback, але
містить заголовки `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`,
що вказують на нелокальне джерело, ці докази через forwarded headers спростовують
твердження про loopback-локальність. Тоді потік pairing вимагає явного підтвердження
замість того, щоб мовчки трактувати запит як підключення з того самого хоста. Див.
[Trusted Proxy Auth](/uk/gateway/trusted-proxy-auth) для еквівалентного правила щодо
автентифікації оператора.

## Сховище (локальне, приватне)

Стан pairing зберігається в каталозі стану Gateway (типово `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

Якщо ви перевизначаєте `OPENCLAW_STATE_DIR`, папка `nodes/` переміщується разом із ним.

Примітки щодо безпеки:

- Токени є секретами; вважайте `paired.json` чутливим.
- Ротація токена вимагає повторного підтвердження (або видалення запису вузла).

## Поведінка транспорту

- Транспорт є **безстановим**; він не зберігає дані про членство.
- Якщо Gateway офлайн або pairing вимкнено, вузли не можуть виконати pairing.
- Якщо Gateway працює у віддаленому режимі, pairing усе одно відбувається щодо сховища віддаленого Gateway.
