# Виявлення та транспорти

OpenClaw має дві окремі проблеми, які на поверхні виглядають схоже:

1. **Віддалене керування оператором**: застосунок у рядку меню macOS керує Gateway, що працює в іншому місці.
2. **Сполучення Node**: iOS/Android (і майбутні nodes) знаходять Gateway і безпечно сполучаються.

Мета дизайну — тримати все мережеве виявлення/оголошення в **Node Gateway** (`openclaw gateway`), а клієнтів (застосунок для Mac, iOS) залишати споживачами.

## Терміни

- **Gateway**: один довготривалий процес Gateway, який володіє станом (сеанси, сполучення, реєстр nodes) і запускає канали. Більшість конфігурацій використовує один на хост; ізольовані конфігурації з кількома Gateway можливі.
- **Gateway WS (площина керування)**: кінцева точка WebSocket на `127.0.0.1:18789` за замовчуванням; може бути прив’язана до LAN/tailnet через `gateway.bind`.
- **Прямий WS-транспорт**: кінцева точка Gateway WS, доступна з LAN/tailnet (без SSH).
- **SSH-транспорт (резервний)**: віддалене керування через переспрямування `127.0.0.1:18789` через SSH.
- **Застарілий TCP-міст (видалено)**: старіший транспорт node (див.
  [Протокол мосту](/uk/gateway/bridge-protocol)); більше не оголошується для
  виявлення і більше не є частиною поточних збірок.

Деталі протоколів:

- [Протокол Gateway](/uk/gateway/protocol)
- [Протокол мосту (застарілий)](/uk/gateway/bridge-protocol)

## Чому ми зберігаємо і "direct", і SSH

- **Direct WS** забезпечує найкращий UX у тій самій мережі та всередині tailnet:
  - автовиявлення в LAN через Bonjour
  - токени сполучення + ACL, якими володіє Gateway
  - доступ до shell не потрібен; поверхня протоколу може залишатися вузькою та придатною для аудиту
- **SSH** залишається універсальним резервним варіантом:
  - працює всюди, де у вас є SSH-доступ (навіть між непов’язаними мережами)
  - витримує проблеми multicast/mDNS
  - не потребує нових вхідних портів, окрім SSH

## Вхідні дані виявлення (як клієнти дізнаються, де Gateway)

### 1) Виявлення Bonjour / DNS-SD

Multicast Bonjour працює за принципом найкращої спроби й не перетинає мережі. OpenClaw також може переглядати
той самий маяк Gateway через налаштований домен wide-area DNS-SD, тож виявлення може охоплювати:

- `local.` у тій самій LAN
- налаштований домен unicast DNS-SD для виявлення між мережами

Цільовий напрямок:

- **Gateway** оголошує свою кінцеву точку WS через Bonjour, коли ввімкнено вбудований
  Plugin `bonjour`. Plugin автоматично запускається на хостах macOS і є
  опційним в інших середовищах.
- Клієнти переглядають і показують список “виберіть Gateway”, а потім зберігають вибрану кінцеву точку.

Усунення несправностей і деталі маяка: [Bonjour](/uk/gateway/bonjour).

#### Деталі службового маяка

- Типи служб:
  - `_openclaw-gw._tcp` (маяк транспорту Gateway)
- TXT-ключі (не секретні):
  - `role=gateway`
  - `transport=gateway`
  - `displayName=<friendly name>` (зручне ім’я, налаштоване оператором)
  - `lanHost=<hostname>.local`
  - `gatewayPort=18789` (Gateway WS + HTTP)
  - `gatewayTls=1` (лише коли TLS увімкнено)
  - `gatewayTlsSha256=<sha256>` (лише коли TLS увімкнено і відбиток доступний)
  - `canvasPort=<port>` (порт canvas host; наразі такий самий, як `gatewayPort`, коли canvas host увімкнено)
  - `tailnetDns=<magicdns>` (необов’язкова підказка; автоматично визначається, коли доступний Tailscale)
  - `sshPort=<port>` (лише повний режим mDNS; wide-area DNS-SD може його пропускати, у такому разі стандартні значення SSH залишаються `22`)
  - `cliPath=<path>` (лише повний режим mDNS; wide-area DNS-SD усе одно записує його як підказку для віддаленого встановлення)

Нотатки щодо безпеки:

- TXT-записи Bonjour/mDNS **не автентифіковані**. Клієнти мають сприймати TXT-значення лише як UX-підказки.
- Маршрутизація (хост/порт) має віддавати перевагу **розв’язаній кінцевій точці служби** (SRV + A/AAAA) над наданими через TXT `lanHost`, `tailnetDns` або `gatewayPort`.
- Закріплення TLS ніколи не має дозволяти оголошеному `gatewayTlsSha256` перевизначати раніше збережений pin.
- Nodes iOS/Android мають вимагати явного підтвердження “довіряти цьому відбитку” перед збереженням першого pin (позасмугова перевірка), коли вибраний маршрут є захищеним/базованим на TLS.

Увімкнення/вимкнення/перевизначення:

- `openclaw plugins enable bonjour` вмикає multicast-оголошення в LAN.
- `OPENCLAW_DISABLE_BONJOUR=1` вимикає оголошення.
- Коли Plugin Bonjour увімкнено, а `OPENCLAW_DISABLE_BONJOUR` не задано,
  Bonjour оголошується на звичайних хостах і автоматично вимикається всередині виявлених контейнерів.
  Запуск macOS Gateway з порожньою конфігурацією автоматично вмикає Plugin; Linux,
  Windows і контейнеризовані розгортання потребують явного ввімкнення.
  Використовуйте `0` лише на host, macvlan або іншій мережі з підтримкою mDNS; використовуйте `1`, щоб
  примусово вимкнути.
- `gateway.bind` у `~/.openclaw/openclaw.json` керує режимом прив’язки Gateway.
- `OPENCLAW_SSH_PORT` перевизначає SSH-порт, який оголошується, коли випускається `sshPort`.
- `OPENCLAW_TAILNET_DNS` публікує підказку `tailnetDns` (MagicDNS).
- `OPENCLAW_CLI_PATH` перевизначає оголошений шлях CLI.

### 2) Tailnet (між мережами)

Для конфігурацій у стилі Лондон/Відень Bonjour не допоможе. Рекомендована “direct” ціль:

- ім’я Tailscale MagicDNS (бажано) або стабільна IP-адреса tailnet.

Якщо Gateway може визначити, що працює під Tailscale, він публікує `tailnetDns` як необов’язкову підказку для клієнтів (включно з wide-area маяками).

Застосунок macOS тепер надає перевагу іменам MagicDNS над сирими IP-адресами Tailscale для виявлення Gateway. Це підвищує надійність, коли IP-адреси tailnet змінюються (наприклад, після перезапусків node або перепризначення CGNAT), оскільки імена MagicDNS автоматично розв’язуються в поточну IP-адресу.

Для сполучення mobile node підказки виявлення не послаблюють безпеку транспорту на маршрутах tailnet/public:

- iOS/Android усе ще вимагають захищений шлях першого підключення tailnet/public (`wss://` або Tailscale Serve/Funnel).
- Виявлена сира IP-адреса tailnet — це підказка маршрутизації, а не дозвіл використовувати plaintext віддалений `ws://`.
- Пряме підключення через приватну LAN з `ws://` залишається підтримуваним.
- Якщо вам потрібен найпростіший шлях Tailscale для mobile nodes, використовуйте Tailscale Serve, щоб і виявлення, і код налаштування розв’язувалися до тієї самої захищеної кінцевої точки MagicDNS.

### 3) Ручна / SSH-ціль

Коли прямого маршруту немає (або direct вимкнено), клієнти завжди можуть підключитися через SSH, переспрямувавши порт Gateway на loopback.

Див. [Віддалений доступ](/uk/gateway/remote).

## Вибір транспорту (політика клієнта)

Рекомендована поведінка клієнта:

1. Якщо налаштована і доступна сполучена пряма кінцева точка, використовуйте її.
2. Інакше, якщо виявлення знаходить Gateway на `local.` або налаштованому wide-area домені, запропонуйте вибір “Використати цей Gateway” одним дотиком і збережіть його як пряму кінцеву точку.
3. Інакше, якщо налаштовано DNS/IP tailnet, спробуйте direct.
   Для mobile nodes на маршрутах tailnet/public direct означає захищену кінцеву точку, а не plaintext віддалений `ws://`.
4. Інакше перейдіть на SSH.

## Сполучення + автентифікація (прямий транспорт)

Gateway є джерелом істини для допуску node/клієнта.

- Запити на сполучення створюються/схвалюються/відхиляються в Gateway (див. [Сполучення Gateway](/uk/gateway/pairing)).
- Gateway забезпечує:
  - автентифікацію (токен / пара ключів)
  - scopes/ACL (Gateway не є сирим проксі до кожного методу)
  - обмеження швидкості

## Відповідальність за компонентами

- **Gateway**: оголошує маяки виявлення, володіє рішеннями щодо сполучення і розміщує кінцеву точку WS.
- **Застосунок macOS**: допомагає вибрати Gateway, показує запити на сполучення і використовує SSH лише як резервний варіант.
- **Nodes iOS/Android**: переглядають Bonjour для зручності й підключаються до сполученого Gateway WS.
