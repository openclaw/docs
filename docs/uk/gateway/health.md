# Перевірки стану (CLI)

Короткий посібник, як перевіряти підключення каналів без здогадок.

## Швидкі перевірки

- `openclaw status` — локальне зведення: доступність/режим gateway, підказка щодо оновлення, вік автентифікації прив’язаних каналів, sessions + нещодавня активність.
- `openclaw status --all` — повна локальна діагностика (лише читання, кольори, безпечно вставляти для налагодження).
- `openclaw status --deep` — запитує в запущеного gateway живу перевірку стану (`health` з `probe:true`), включно з перевірками каналів для кожного акаунта, якщо це підтримується.
- `openclaw health` — запитує у запущеного gateway його snapshot стану (лише WS; жодних прямих сокетів каналів із CLI).
- `openclaw health --verbose` — примусово запускає живу перевірку стану і виводить деталі підключення gateway.
- `openclaw health --json` — машинозчитуваний вивід snapshot-а стану.
- Надішліть `/status` як окреме повідомлення у WhatsApp/WebChat, щоб отримати відповідь зі станом без виклику агента.
- Логи: переглядайте `/tmp/openclaw/openclaw-*.log` і фільтруйте за `web-heartbeat`, `web-reconnect`, `web-auto-reply`, `web-inbound`.

## Глибока діагностика

- Creds на диску: `ls -l ~/.openclaw/credentials/whatsapp/<accountId>/creds.json` (mtime має бути нещодавнім).
- Сховище session: `ls -l ~/.openclaw/agents/<agentId>/sessions/sessions.json` (шлях можна перевизначити в конфігурації). Кількість і нещодавні отримувачі показуються через `status`.
- Потік переприв’язки: `openclaw channels logout && openclaw channels login --verbose`, коли в логах з’являються коди стану 409–515 або `loggedOut`. (Примітка: потік входу через QR автоматично перезапускається один раз для статусу 515 після pairing.)
- Діагностика типово ввімкнена. Gateway записує операційні факти, якщо не задано `diagnostics.enabled: false`. Події memory фіксують кількість байтів RSS/heap, тиск порогів і тиск зростання. Події надто великих payload-ів фіксують, що було відхилено, обрізано або розбито на фрагменти, а також розміри й ліміти, коли вони доступні. Вони не записують текст повідомлення, вміст вкладень, тіло Webhook, сире тіло запиту чи відповіді, токени, cookies або секретні значення. Той самий Heartbeat запускає обмежений recorder стабільності, доступний через `openclaw gateway stability` або Gateway RPC `diagnostics.stability`. Фатальні виходи Gateway, timeout-и вимкнення та збої запуску після перезапуску зберігають останній snapshot recorder-а в `~/.openclaw/logs/stability/`, якщо події існують; перевіряйте найновіший збережений bundle через `openclaw gateway stability --bundle latest`.
- Для bug report-ів запустіть `openclaw gateway diagnostics export` і прикріпіть згенерований zip. Експорт об’єднує Markdown-зведення, найновіший bundle стабільності, очищені метадані логів, очищені snapshot-и status/health Gateway і форму конфігурації. Він призначений для поширення: текст чату, тіла Webhook, вивід інструментів, облікові дані, cookies, ідентифікатори акаунтів/повідомлень і секретні значення пропускаються або редагуються. Див. [Експорт діагностики](/uk/gateway/diagnostics).

## Конфігурація монітора стану

- `gateway.channelHealthCheckMinutes`: як часто gateway перевіряє стан каналу. Типове значення: `5`. Задайте `0`, щоб глобально вимкнути перезапуски монітора стану.
- `gateway.channelStaleEventThresholdMinutes`: як довго підключений канал може залишатися бездіяльним, перш ніж монітор стану вважатиме його застарілим і перезапустить. Типове значення: `30`. Тримайте це значення більшим або рівним `gateway.channelHealthCheckMinutes`.
- `gateway.channelMaxRestartsPerHour`: ковзне обмеження на одну годину для перезапусків монітора стану на канал/акаунт. Типове значення: `10`.
- `channels.<provider>.healthMonitor.enabled`: вимкнути перезапуски монітора стану для конкретного каналу, залишивши глобальний моніторинг увімкненим.
- `channels.<provider>.accounts.<accountId>.healthMonitor.enabled`: перевизначення для кількох акаунтів, яке має пріоритет над налаштуванням рівня каналу.
- Ці перевизначення для окремих каналів застосовуються до вбудованих моніторів каналів, які сьогодні їх підтримують: Discord, Google Chat, iMessage, Microsoft Teams, Signal, Slack, Telegram і WhatsApp.

## Коли щось не працює

- `logged out` або статус 409–515 → переприв’яжіть через `openclaw channels logout`, потім `openclaw channels login`.
- Gateway недоступний → запустіть його: `openclaw gateway --port 18789` (використовуйте `--force`, якщо порт зайнятий).
- Немає вхідних повідомлень → переконайтеся, що прив’язаний телефон онлайн і відправник дозволений (`channels.whatsapp.allowFrom`); для групових чатів переконайтеся, що правила allowlist і згадок збігаються (`channels.whatsapp.groups`, `agents.list[].groupChat.mentionPatterns`).

## Окрема команда "health"

`openclaw health` запитує у запущеного gateway snapshot його стану (без прямих сокетів
каналів із CLI). Типово вона може повертати свіжий кешований snapshot gateway; після цього
gateway оновлює цей кеш у фоновому режимі. `openclaw health --verbose` натомість примусово запускає
живу перевірку. Команда показує вік прив’язаних creds/auth, коли це доступно,
зведення перевірок каналів для кожного каналу, зведення сховища session і тривалість перевірки. Вона завершується
з ненульовим кодом, якщо gateway недоступний або перевірка не вдалася/спрацював timeout.

Параметри:

- `--json`: машинозчитуваний вивід JSON
- `--timeout <ms>`: перевизначити типовий timeout перевірки 10s
- `--verbose`: примусово запустити живу перевірку та вивести деталі підключення gateway
- `--debug`: псевдонім для `--verbose`

Snapshot стану включає: `ok` (boolean), `ts` (timestamp), `durationMs` (час перевірки), стан для кожного каналу, доступність агента та зведення сховища session.
