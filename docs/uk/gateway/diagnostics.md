OpenClaw може створити локальний діагностичний архів `.zip` для звітів про помилки: очищені дані про
стан і працездатність Gateway, журнали, структуру конфігурації та нещодавні події стабільності без корисного навантаження.

Вважайте діагностичні пакети секретними даними, доки їх не буде перевірено. Корисні навантаження та облікові дані
приховуються за задумом, але пакет усе одно містить зведення локальних журналів Gateway і
стану середовища виконання на рівні хоста.

## Швидкий початок

```bash
openclaw gateway diagnostics export
```

Виводить шлях до записаного zip-архіву. Щоб вибрати шлях виведення:

```bash
openclaw gateway diagnostics export --output openclaw-diagnostics.zip
```

Для автоматизації:

```bash
openclaw gateway diagnostics export --json
```

## Команда чату

Власники можуть виконати `/diagnostics [note]` у будь-якій розмові, щоб запросити локальний
експорт Gateway як єдиний зручний для копіювання звіт для служби підтримки:

1. Надішліть `/diagnostics`, за бажанням додавши коротку примітку (`/diagnostics bad tool choice`).
2. OpenClaw надсилає вступне повідомлення та запитує одне явне схвалення виконання, яке запускає
   `openclaw gateway diagnostics export --json`. Не схвалюйте діагностику за допомогою
   правила, що дозволяє все.
3. Після схвалення OpenClaw відповідає локальним шляхом до пакета, зведенням
   маніфесту, примітками щодо конфіденційності та відповідними ідентифікаторами сеансів.

У групових чатах власник також може виконати `/diagnostics`, але OpenClaw надсилає
результат експорту, запити на схвалення та розподіл сеансів/гілок Codex
власнику приватно. Група бачить лише коротке сповіщення про те, що діагностичні дані надіслано
приватно. Якщо приватного маршруту до власника немає, команда безпечно завершується відмовою та просить
власника виконати її в особистих повідомленнях.

Коли активний сеанс використовує нативне середовище OpenAI Codex, те саме схвалення
виконання також охоплює надсилання відгуку OpenAI для гілок Codex, про які
відомо OpenClaw. Це надсилання відбувається окремо від локального zip-архіву Gateway і лише
для сеансів середовища Codex. У запиті на схвалення зазначено, що схвалення
також надсилає відгук Codex, без переліку ідентифікаторів сеансів або гілок Codex. Після
схвалення відповідь містить канали, ідентифікатори сеансів OpenClaw, ідентифікатори гілок Codex і
локальні команди відновлення для гілок, надісланих до OpenAI. Відмова або
ігнорування схвалення пропускає експорт, надсилання відгуку Codex і
список ідентифікаторів Codex.

Це скорочує цикл налагодження Codex: помітьте некоректну поведінку в каналі,
виконайте `/diagnostics`, один раз схваліть, поділіться звітом, а потім локально виконайте виведену
команду `codex resume <thread-id>`, якщо хочете самостійно перевірити гілку.
Див. [середовище Codex](/uk/plugins/codex-harness#inspect-codex-threads-locally).

## Що містить експорт

- `summary.md`: зручний для читання огляд для служби підтримки.
- `diagnostics.json`: машинозчитуване зведення конфігурації, журналів, стану, працездатності
  та даних стабільності.
- `manifest.json`: метадані експорту та список файлів.
- Очищена структура конфігурації та несекретні відомості про конфігурацію.
- Очищені зведення журналів і нещодавні рядки журналів із прихованими даними.
- Знімки стану та працездатності Gateway, отримані в міру можливості.
- `stability/latest.json`: найновіший збережений пакет стабільності, якщо доступний.

Експорт залишається корисним, навіть коли Gateway несправний: якщо запити
стану або працездатності завершуються помилкою, локальні журнали, структура конфігурації та останній пакет стабільності
все одно збираються, якщо доступні.

## Модель конфіденційності

Зберігаються: назви підсистем, ідентифікатори плагінів, ідентифікатори постачальників, ідентифікатори каналів, налаштовані
режими, коди стану, тривалості, кількість байтів, стан черги, показники пам’яті,
очищені метадані журналів, операційні повідомлення з прихованими даними, структура конфігурації та
несекретні налаштування функцій.

Пропускаються або приховуються: текст чатів, запити, інструкції, тіла webhook-запитів, результати
інструментів, облікові дані, ключі API, токени, файли cookie, секретні значення, необроблені
тіла запитів/відповідей, ідентифікатори облікових записів, ідентифікатори повідомлень, необроблені ідентифікатори сеансів,
імена хостів і локальні імена користувачів.

Коли повідомлення журналу схоже на текст користувацького, чатового, запитового або інструментального корисного навантаження,
експорт зберігає лише факт пропуску повідомлення та кількість його байтів.

## Реєстратор стабільності

Gateway за замовчуванням записує обмежений потік подій стабільності без корисного навантаження, коли
діагностику ввімкнено. Він фіксує операційні факти, а не вміст.

Той самий Heartbeat також перевіряє працездатність, коли цикл подій або ЦП здається
перевантаженим, створюючи події `diagnostic.liveness.warning` із затримкою циклу подій,
використанням циклу подій, співвідношенням навантаження до кількості ядер ЦП, кількістю активних/очікувальних/поставлених у чергу сеансів,
поточною фазою запуску/виконання (якщо відома), нещодавніми інтервалами фаз і
обмеженими мітками роботи. Вони перетворюються на рядки журналу Gateway рівня `warn`, лише коли
робота очікує або перебуває в черзі, чи коли активна робота збігається з тривалою затримкою
циклу подій; інакше вони записуються з рівнем `debug`. Вибірки працездатності під час простою також записуються
як діагностичні події, але самі собою ніколи не підвищуються до попередження.

Фази запуску створюють події `diagnostic.phase.completed` із часом за настінним годинником і
часом ЦП. Діагностика завислого вбудованого запуску встановлює `terminalProgressStale=true`,
коли останній прогрес мосту виглядав завершальним (наприклад, необроблений елемент відповіді
або подія завершення відповіді), але Gateway усе ще вважає
вбудований запуск активним.

Перевірте активний реєстратор:

```bash
openclaw gateway stability
openclaw gateway stability --type payload.large
openclaw gateway stability --json
```

Перевірте найновіший збережений пакет після фатального завершення, перевищення часу очікування вимкнення або
помилки запуску після перезапуску:

```bash
openclaw gateway stability --bundle latest
```

Створіть діагностичний zip-архів із найновішого збереженого пакета:

```bash
openclaw gateway stability --bundle latest --export
```

Збережені пакети розміщуються в `~/.openclaw/logs/stability/`, якщо події існують.

## Корисні параметри

```bash
openclaw gateway diagnostics export \
  --output openclaw-diagnostics.zip \
  --log-lines 5000 \
  --log-bytes 1000000
```

| Прапорець               | Значення за замовчуванням                                                    | Опис                                                         |
| ----------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------------------ |
| `--output <path>`       | `$OPENCLAW_STATE_DIR/logs/support/openclaw-diagnostics-<timestamp>-<pid>.zip` | Записати за вказаним шляхом до zip-архіву (або каталогу).     |
| `--log-lines <count>`   | `5000`                                                                        | Максимальна кількість очищених рядків журналу для включення.  |
| `--log-bytes <bytes>`   | `1000000`                                                                     | Максимальна кількість байтів журналу для перевірки.           |
| `--url <url>`           | -                                                                             | URL WebSocket Gateway для знімків стану/працездатності.       |
| `--token <token>`       | -                                                                             | Токен Gateway для знімків стану/працездатності.               |
| `--password <password>` | -                                                                             | Пароль Gateway для знімків стану/працездатності.              |
| `--timeout <ms>`        | `3000`                                                                        | Час очікування знімка стану/працездатності.                   |
| `--no-stability-bundle` | вимкнено                                                                      | Пропустити пошук збереженого пакета стабільності.              |
| `--json`                | вимкнено                                                                      | Вивести машинозчитувані метадані експорту.                    |

## Вимкнення діагностики

Діагностику ввімкнено за замовчуванням. Щоб вимкнути реєстратор стабільності та
збирання діагностичних подій:

```json5
{
  diagnostics: {
    enabled: false,
  },
}
```

Вимкнення діагностики зменшує докладність звітів про помилки, але не впливає на звичайне
журналювання Gateway.

Знімки за критичного тиску на пам’ять за замовчуванням вимкнено. Щоб створювати
знімок стабільності перед OOM на додаток до звичайних діагностичних подій:

```json5
{
  diagnostics: {
    memoryPressureSnapshot: true,
  },
}
```

Використовуйте це лише на хостах, здатних витримати додаткове сканування файлової системи та
запис знімка під час критичного тиску на пам’ять. Звичайні події тиску на пам’ять
усе одно записують дані про RSS, купу, порогове значення та зростання (`rss_threshold`,
`heap_threshold`, `rss_growth`), коли створення знімка вимкнено.
