Більшості конфігурацій потрібен один Gateway — один Gateway обслуговує кілька підключень до месенджерів і агентів. Запускайте окремі Gateway з ізольованими профілями/портами лише тоді, коли потрібна надійніша ізоляція або резервування (наприклад, аварійний бот).

## Швидкий запуск аварійного бота

Найпростіша конфігурація аварійного бота:

- Залиште основного бота в профілі за замовчуванням.
- Запустіть аварійного бота з `--profile rescue` та окремим токеном бота Telegram.
- Призначте аварійному боту інший базовий порт, наприклад `19789`.

Завдяки цьому аварійний бот зможе виконувати діагностику або змінювати конфігурацію, якщо основний бот не працює. Залишайте між базовими портами щонайменше 20 портів, щоб похідні порти браузера/CDP ніколи не конфліктували.

```bash
# Аварійний бот (окремий бот Telegram, окремий профіль, порт 19789)
openclaw --profile rescue onboard
openclaw --profile rescue gateway install --port 19789
```

Якщо основний бот уже працює, зазвичай цього достатньо. Якщо під час початкового налаштування аварійну службу вже встановлено, пропустіть останню команду `gateway install`.

Під час виконання `openclaw --profile rescue onboard`:

- Використовуйте окремий токен бота Telegram, призначений для аварійного облікового запису (його легко зробити доступним лише операторам, він не залежить від встановлення каналу/застосунку основного бота й забезпечує простий шлях відновлення через особисті повідомлення).
- Залиште назву профілю `rescue`.
- Використовуйте базовий порт, щонайменше на 20 більший за порт основного бота.
- Прийміть стандартний робочий простір аварійного бота, якщо ви ще не керуєте власним.

### Що змінює `--profile rescue onboard`

`--profile rescue onboard` виконує звичайний процес початкового налаштування, але записує все в окремий профіль, тому аварійний бот отримує власні:

- Файл профілю/конфігурації
- Каталог стану
- Робочий простір (за замовчуванням: `~/.openclaw/workspace-rescue`)
- Назву керованої служби
- Базовий порт (і похідні порти)
- Токен бота Telegram

В іншому запити не відрізняються від звичайного початкового налаштування.

## Загальне налаштування кількох Gateway

Та сама схема ізоляції працює для будь-якої пари або групи Gateway на одному хості — надайте кожному додатковому Gateway окремий іменований профіль і базовий порт:

```bash
# основний (профіль за замовчуванням)
openclaw setup
openclaw gateway --port 18789

# додатковий Gateway
openclaw --profile ops setup
openclaw --profile ops gateway --port 19789
```

Іменовані профілі також можна використовувати для обох екземплярів:

```bash
openclaw --profile main setup
openclaw --profile main gateway --port 18789

openclaw --profile ops setup
openclaw --profile ops gateway --port 19789
```

Для служб діє та сама схема:

```bash
openclaw gateway install
openclaw --profile ops gateway install --port 19789
```

Використовуйте швидкий запуск аварійного бота як резервний операторський канал, а загальну схему профілів — для кількох довготривалих Gateway, що працюють із різними каналами, орендарями, робочими просторами або операційними ролями.

## Контрольний список ізоляції

Ці параметри мають бути унікальними для кожного екземпляра Gateway:

| Параметр                     | Призначення                                      |
| ---------------------------- | ------------------------------------------------ |
| `OPENCLAW_CONFIG_PATH`       | Окремий файл конфігурації для кожного екземпляра |
| `OPENCLAW_STATE_DIR`         | Окремі сеанси, облікові дані та кеші             |
| `agents.defaults.workspace`  | Корінь робочого простору окремого екземпляра      |
| `gateway.port` (або `--port`) | Унікальний для кожного екземпляра               |
| Похідні порти браузера/CDP   | Див. нижче                                       |

Спільне використання будь-якого з них спричиняє конфлікти конфігурації та портів.

## Зіставлення портів (похідних)

Базовий порт = `gateway.port` (або `OPENCLAW_GATEWAY_PORT` / `--port`).

- Порт служби керування браузером = базовий + 2 (лише local loopback).
- Хост Canvas обслуговується безпосередньо HTTP-сервером Gateway (на тому самому порту, що й `gateway.port`).
- Порти CDP профілів браузера автоматично виділяються від `browser control port + 9` до `+ 108`.

Якщо ви перевизначаєте будь-які з них у конфігурації або змінних середовища, вони мають залишатися унікальними для кожного екземпляра.

## Примітки щодо браузера/CDP (поширена пастка)

- **Не** задавайте для `browser.cdpUrl` однакове значення в кількох екземплярах.
- Кожному екземпляру потрібні власний порт керування браузером і діапазон CDP (похідні від його порту Gateway).
- Щоб явно задати порти CDP, налаштуйте `browser.profiles.<name>.cdpPort` окремо для кожного екземпляра.
- Для віддаленого Chrome використовуйте `browser.profiles.<name>.cdpUrl` (окремо для кожного профілю й екземпляра).

## Приклад ручного налаштування змінних середовища

```bash
OPENCLAW_CONFIG_PATH=~/.openclaw/main.json \
OPENCLAW_STATE_DIR=~/.openclaw \
openclaw gateway --port 18789

OPENCLAW_CONFIG_PATH=~/.openclaw/rescue.json \
OPENCLAW_STATE_DIR=~/.openclaw-rescue \
openclaw gateway --port 19789
```

## Швидкі перевірки

```bash
openclaw gateway status --deep
openclaw --profile rescue gateway status --deep
openclaw --profile rescue gateway probe
openclaw status
openclaw --profile rescue status
openclaw --profile rescue browser status
```

- `gateway status --deep` виявляє застарілі служби launchd/systemd/schtasks від попередніх установлень.
- Текст попередження `gateway probe`, наприклад `multiple reachable gateway identities detected`, очікуваний лише тоді, коли ви навмисно запускаєте кілька ізольованих Gateway або коли OpenClaw не може підтвердити, що доступні цілі перевірки є тим самим Gateway. Тунель SSH, URL проксі або налаштований віддалений URL до того самого Gateway — це один Gateway із кількома способами транспортування, навіть якщо їхні порти відрізняються.
