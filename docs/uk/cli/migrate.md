# `openclaw migrate`

Імпортуйте стан з іншої агентної системи через постачальника міграції, яким керує Plugin. Вбудовані постачальники підтримують Claude, Codex CLI та [Hermes](/uk/install/migrating-hermes); плагіни можуть реєструвати додаткових постачальників.

<Tip>
Покрокові інструкції для користувачів дивіться в розділах [Міграція з Claude](/uk/install/migrating-claude) та [Міграція з Hermes](/uk/install/migrating-hermes). У [центрі міграції](/uk/install/migrating) перелічено всі шляхи.
</Tip>

## Команди

```bash
openclaw migrate list
openclaw migrate claude --dry-run
openclaw migrate codex --dry-run
openclaw migrate codex --skill gog-vault77-google-workspace
openclaw migrate codex --plugin google-calendar --dry-run
openclaw migrate codex --plugin google-calendar --verify-plugin-apps --dry-run
openclaw migrate hermes --dry-run
openclaw migrate hermes
openclaw migrate apply codex --yes --skill gog-vault77-google-workspace
openclaw migrate apply codex --yes --plugin google-calendar
openclaw migrate apply codex --yes
openclaw migrate apply claude --yes
openclaw migrate apply hermes --yes
openclaw migrate apply hermes --include-secrets --yes
openclaw onboard --flow import
openclaw onboard --import-from claude --import-source ~/.claude
openclaw onboard --import-from hermes --import-source ~/.hermes
```

Запуск `openclaw migrate <provider>` без інших прапорців створює план, показує попередній перегляд і (у TTY) запитує підтвердження перед застосуванням. `openclaw migrate plan <provider>` та `openclaw migrate apply <provider>` розділяють попередній перегляд і застосування на окремі підкоманди з тими самими прапорцями.

<ParamField path="<provider>" type="string">
  Ім’я зареєстрованого постачальника міграції, наприклад `hermes`. Виконайте `openclaw migrate list`, щоб переглянути встановлених постачальників.
</ParamField>
<ParamField path="--dry-run" type="boolean">
  Створити план і завершити роботу без зміни стану.
</ParamField>
<ParamField path="--from <path>" type="string">
  Перевизначити каталог вихідного стану. Типове значення для Hermes — `~/.hermes`, для Codex — `~/.codex` (або `$CODEX_HOME`), для Claude — `~/.claude`.
</ParamField>
<ParamField path="--include-secrets" type="boolean">
  Імпортувати підтримувані облікові дані без запиту. Під час інтерактивного застосування перед імпортом виявлених облікових даних автентифікації з’являється запит, у якому типово вибрано «так»; для їх імпорту в неінтерактивному режимі з `--yes` потрібен прапорець `--include-secrets`.
</ParamField>
<ParamField path="--no-auth-credentials" type="boolean">
  Пропустити імпорт облікових даних автентифікації, включно з інтерактивним запитом.
</ParamField>
<ParamField path="--overwrite" type="boolean">
  Дозволити під час застосування замінювати наявні цільові об’єкти, якщо план повідомляє про конфлікти.
</ParamField>
<ParamField path="--yes" type="boolean">
  Пропустити запит підтвердження. Обов’язковий у неінтерактивному режимі.
</ParamField>
<ParamField path="--skill <name>" type="string">
  Вибрати один елемент копіювання Skills за його назвою або ідентифікатором елемента. Повторіть прапорець, щоб перенести кілька Skills. Якщо прапорець не вказано, інтерактивні міграції Codex показують засіб вибору з прапорцями, а неінтерактивні міграції зберігають усі заплановані Skills.
</ParamField>
<ParamField path="--plugin <name>" type="string">
  Вибрати один елемент установлення плагіна Codex за назвою плагіна або ідентифікатором елемента. Повторіть прапорець, щоб перенести кілька плагінів Codex. Якщо прапорець не вказано, інтерактивні міграції Codex показують вбудований засіб вибору плагінів Codex із прапорцями, а неінтерактивні міграції зберігають усі заплановані плагіни. Застосовується лише до встановлених із вихідного коду плагінів Codex `openai-curated`, виявлених інвентаризацією сервера застосунків Codex.
</ParamField>
<ParamField path="--verify-plugin-apps" type="boolean">
  Лише для Codex. Примусово виконує свіжий обхід `app/list` вихідного сервера застосунків Codex перед плануванням активації вбудованих плагінів. Типово вимкнено, щоб планування міграції залишалося швидким.
</ParamField>
<ParamField path="--backup-output <path>" type="string">
  Шлях до архіву або каталогу резервної копії перед міграцією. Передається без змін до `openclaw backup create`.
</ParamField>
<ParamField path="--no-backup" type="boolean">
  Пропустити резервне копіювання перед застосуванням. Потребує `--force`, якщо існує локальний стан OpenClaw.
</ParamField>
<ParamField path="--force" type="boolean">
  Обов’язковий разом із `--no-backup`, якщо інакше застосування відмовилося б пропустити резервне копіювання.
</ParamField>
<ParamField path="--json" type="boolean">
  Вивести план або результат застосування у форматі JSON. Якщо вказано `--json` без `--yes`, команда застосування виводить план і не змінює стан.
</ParamField>

## Постачальник Claude

Вбудований постачальник Claude типово виявляє стан Claude Code у `~/.claude`. Використовуйте `--from <path>`, щоб імпортувати конкретний домашній каталог або кореневий каталог проєкту Claude Code.

<Tip>
Покрокову інструкцію для користувачів дивіться в розділі [Міграція з Claude](/uk/install/migrating-claude).
</Tip>

### Що імпортує Claude

- Файли проєкту `CLAUDE.md` і `.claude/CLAUDE.md` до робочого простору агента OpenClaw (`AGENTS.md`).
- Вміст користувацького `~/.claude/CLAUDE.md` додається до `USER.md` у робочому просторі.
- Визначення серверів MCP із проєктного `.mcp.json`, Claude Code `~/.claude.json` (включно з окремими записами для кожного проєкту) і Claude Desktop `claude_desktop_config.json`.
- Каталоги Skills Claude, що містять `SKILL.md` (користувацький `~/.claude/skills` і проєктний `.claude/skills`).
- Файли команд Claude у форматі Markdown (користувацькі `~/.claude/commands` і проєктні `.claude/commands`), перетворені на Skills OpenClaw, доступні лише для ручного виклику.

### Архів і стан для ручної перевірки

Хуки Claude, дозволи, типові значення середовища, проєктний `CLAUDE.local.md`, `.claude/rules`, користувацькі та проєктні каталоги `agents/`, а також історія проєктів (`projects`, `cache`, `plans` у `~/.claude`) зберігаються у звіті про міграцію або позначаються як елементи для ручної перевірки. OpenClaw не виконує хуки, не копіює широкі списки дозволів і не імпортує автоматично стан облікових даних OAuth/Desktop.

## Постачальник Codex

Вбудований постачальник Codex типово виявляє стан Codex CLI у `~/.codex` або в `CODEX_HOME`, якщо цю змінну середовища встановлено. Використовуйте `--from <path>`, щоб інвентаризувати конкретний домашній каталог Codex.

Використовуйте цього постачальника під час переходу на середовище виконання Codex в OpenClaw, якщо хочете свідомо перенести корисні особисті ресурси Codex CLI. Локальні запуски сервера застосунків Codex використовують окремий для кожного агента `CODEX_HOME`, тому типово не читають ваш особистий `~/.codex`. Звичайний процес усе одно успадковує `HOME`, тому Codex бачить спільні Skills у `$HOME/.agents/*` і записи каталогу плагінів, а підпроцеси можуть знаходити конфігурацію та токени в домашньому каталозі користувача.

Запуск `openclaw migrate codex` в інтерактивному терміналі показує попередній перегляд повного плану, а потім відкриває засоби вибору з прапорцями перед остаточним підтвердженням застосування. Спочатку пропонується вибрати елементи копіювання Skills. Використовуйте `Toggle all on` або `Toggle all off` для масового вибору. Натисніть пробіл, щоб перемикати рядки, або Enter, щоб активувати виділений рядок і продовжити. Заплановані Skills спочатку позначені, конфліктні Skills — не позначені, а `Skip for now` пропускає копіювання Skills для цього запуску, але продовжує перехід до вибору плагінів. Якщо встановлені з вихідного коду рекомендовані плагіни Codex можна перенести й не вказано `--plugin`, міграція далі пропонує активувати вбудовані плагіни Codex за назвою плагіна. Елементи плагінів спочатку позначені, якщо тільки цільова конфігурація плагінів Codex в OpenClaw уже не містить цей плагін. Наявні цільові плагіни спочатку не позначені й показують підказку про конфлікт, наприклад `conflict: plugin exists`; виберіть `Toggle all off`, щоб не переносити жодних вбудованих плагінів Codex під час цього запуску, або `Skip for now`, щоб зупинитися перед застосуванням.

Для сценарних або точних запусків явно виберіть один чи кілька Skills або плагінів:

```bash
openclaw migrate codex --dry-run --skill gog-vault77-google-workspace
openclaw migrate apply codex --yes --skill gog-vault77-google-workspace
openclaw migrate codex --dry-run --plugin google-calendar
openclaw migrate apply codex --yes --plugin google-calendar
```

### Що імпортує Codex

- Каталоги Skills Codex CLI у `$CODEX_HOME/skills`, за винятком системного кешу Codex `.system`.
- Особисті AgentSkills у `$HOME/.agents/skills`, скопійовані до поточного робочого простору агента OpenClaw для окремого володіння кожним агентом.
- Встановлені з вихідного коду плагіни Codex `openai-curated`, виявлені через `plugin/list` сервера застосунків Codex. Під час планування виконується читання `plugin/read` для кожного ввімкненого встановленого плагіна.

Міграція плагінів із підтримкою застосунків має додаткові обмеження:

- Для плагінів із підтримкою застосунків обліковий запис вихідного сервера застосунків Codex має бути обліковим записом із підпискою ChatGPT. Відповіді для облікових записів, що не належать до ChatGPT, або за відсутності облікового запису пропускаються з причиною `codex_subscription_required`.
- Типово міграція не викликає `app/list` джерела, тому плагіни з підтримкою застосунків, які проходять перевірку облікового запису, плануються без перевірки доступності вихідного застосунку, а помилки транспорту під час пошуку облікового запису призводять до пропуску з причиною `codex_account_unavailable`.
- Передайте `--verify-plugin-apps`, щоб примусово отримати свіжий знімок `app/list` джерела та вимагати, щоб кожен власний застосунок був наявний, увімкнений і доступний до планування вбудованої активації. У цьому режимі після помилок транспорту під час пошуку облікового запису виконується перевірка інвентаризації вихідних застосунків. Знімок зберігається в пам’яті лише для поточного процесу; він ніколи не записується у вихідні дані міграції або цільову конфігурацію.

Вимкнені плагіни, недоступні для читання відомості про плагіни, вихідні облікові записи з обмеженням за підпискою, а також (якщо встановлено `--verify-plugin-apps`) відсутні, вимкнені або недоступні застосунки стають пропущеними елементами для ручної перевірки з типізованими причинами замість записів цільової конфігурації. Для кожного вибраного придатного плагіна команда застосування викликає `plugin/install` сервера застосунків, навіть якщо цільовий сервер застосунків уже повідомляє, що цей плагін установлено й увімкнено. Перенесені плагіни Codex можна використовувати лише в сеансах, які вибирають вбудоване середовище виконання Codex; вони недоступні для запусків постачальника OpenClaw, прив’язок розмов ACP або інших середовищ виконання.

### Стан Codex для ручної перевірки

Codex `config.toml`, вбудовані `hooks/hooks.json`, нерекомендовані каталоги, кешовані пакети плагінів, які не є встановленими з вихідного коду рекомендованими плагінами, та встановлені з вихідного коду плагіни, які не проходять вихідну перевірку підписки, не активуються автоматично. Якщо встановлено `--verify-plugin-apps`, плагіни, які не проходять перевірку інвентаризації вихідних застосунків, також пропускаються. Усе це копіюється або зазначається у звіті про міграцію для ручної перевірки.

Для перенесених рекомендованих плагінів, установлених із вихідного коду, команда застосування записує:

- `plugins.entries.codex.enabled: true`
- `plugins.entries.codex.config.codexPlugins.enabled: true`
- `plugins.entries.codex.config.codexPlugins.allow_destructive_actions: true`
- один явний запис плагіна з `marketplaceName: "openai-curated"` і `pluginName` для кожного вибраного плагіна

Міграція ніколи не записує `plugins["*"]` і ніколи не зберігає локальні шляхи кешу каталогу плагінів.

Пропущені плагіни не записуються до цільової конфігурації. Помилки підписок на стороні джерела зазначаються для елементів, що потребують ручного втручання, із типізованими причинами: `codex_subscription_required`, `codex_account_unavailable`, `plugin_disabled` або `plugin_read_unavailable`. З параметром `--verify-plugin-apps` помилки інвентаризації застосунків джерела також можуть мати значення `app_inaccessible`, `app_disabled`, `app_missing` або `app_inventory_unavailable`. Інсталяції на цільовій стороні, що потребують автентифікації, зазначаються для відповідного елемента плагіна зі `status: "skipped"`, `reason: "auth_required"` і знеособленими ідентифікаторами застосунків; відповідні явні записи конфігурації записуються як вимкнені, доки ви повторно не авторизуєте та не ввімкнете їх. Інші помилки інсталяції повертаються як результати `error`, прив’язані до окремих елементів.

Якщо під час планування інвентар плагінів сервера застосунків Codex недоступний, міграція замість збою всієї процедури використовує резервні рекомендаційні елементи з кешованого пакета.

## Провайдер Hermes

Вбудований провайдер Hermes за замовчуванням виявляє стан у `~/.hermes`. Якщо Hermes розташований в іншому місці, використовуйте `--from <path>`.

### Що імпортує Hermes

- Конфігурацію моделі за замовчуванням із `config.yaml`.
- Налаштованих провайдерів моделей і власні кінцеві точки, сумісні з OpenAI, із `providers` та `custom_providers`.
- Визначення серверів MCP із `mcp_servers` або `mcp.servers`.
- `SOUL.md` і `AGENTS.md` до робочого простору агента OpenClaw.
- `memories/MEMORY.md` і `memories/USER.md`, додаючи їх до файлів пам’яті робочого простору.

## Контракт плагіна

Джерела міграції є плагінами. Плагін оголошує ідентифікатори своїх провайдерів у `openclaw.plugin.json`:

```json
{
  "contracts": {
    "migrationProviders": ["hermes"]
  }
}
```

Під час виконання плагін викликає `api.registerMigrationProvider(...)`. Провайдер реалізує `detect`, `plan` і `apply`. Ядро відповідає за оркестрацію CLI, політику резервного копіювання, запити до користувача, виведення JSON і попередню перевірку конфліктів. Ядро передає перевірений план до `apply(ctx, plan)`, а для сумісності провайдери можуть перебудувати план лише за відсутності цього аргументу.

Плагіни провайдерів можуть використовувати `openclaw/plugin-sdk/migration` для створення елементів і підрахунку підсумків, а також `openclaw/plugin-sdk/migration-runtime` для копіювання файлів з урахуванням конфліктів, копіювання до звітів лише для архівування, кешованих обгорток середовища виконання конфігурації та звітів про міграцію.

## Інтеграція з початковим налаштуванням

Початкове налаштування може запропонувати міграцію, коли провайдер виявляє відоме джерело. І `openclaw onboard --flow import`, і `openclaw setup --wizard --import-from hermes` використовують того самого провайдера міграції плагіна та показують попередній перегляд перед застосуванням.

<Note>
Для імпорту під час початкового налаштування потрібне нове налаштування OpenClaw. Якщо у вас уже є локальний стан, спочатку скиньте конфігурацію, облікові дані, сеанси та робочий простір. Імпорт із резервним копіюванням і перезаписом або зі злиттям для наявних налаштувань доступний лише за ввімкнення відповідної функції.
</Note>
