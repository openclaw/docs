# `openclaw models`

Виявлення, сканування та налаштування моделей (модель за замовчуванням, резервні варіанти, профілі автентифікації).

Пов’язане:

- Провайдери + моделі: [Моделі](/uk/providers/models)
- Поняття вибору моделей + slash-команда `/models`: [Поняття моделей](/uk/concepts/models)
- Налаштування автентифікації провайдера: [Початок роботи](/uk/start/getting-started)

## Поширені команди

```bash
openclaw models status
openclaw models list
openclaw models set <model-or-alias>
openclaw models scan
```

`openclaw models status` показує розв’язані модель за замовчуванням/резервні варіанти, а також огляд автентифікації.
Коли доступні знімки використання провайдера, розділ стану OAuth/API-ключів містить
вікна використання провайдера та знімки квот.
Поточні провайдери вікон використання: Anthropic, GitHub Copilot, Gemini CLI, OpenAI,
MiniMax, Xiaomi і z.ai. Автентифікація використання надходить із провайдер-специфічних хуків,
коли вони доступні; інакше OpenClaw повертається до відповідних облікових даних OAuth/API-ключа
з профілів автентифікації, env або конфігурації.
У виводі `--json` `auth.providers` — це огляд провайдерів з урахуванням env/конфігурації/сховища,
а `auth.oauth` — лише стан профілів сховища автентифікації.
Додайте `--probe`, щоб запустити live-перевірки автентифікації для кожного налаштованого профілю провайдера.
Перевірки — це реальні запити (можуть споживати токени й спричиняти обмеження частоти).
Використовуйте `--agent <id>`, щоб переглянути стан моделі/автентифікації налаштованого агента. Якщо його опущено,
команда використовує `OPENCLAW_AGENT_DIR`, якщо встановлено, інакше —
налаштованого агента за замовчуванням.
Рядки перевірок можуть походити з профілів автентифікації, облікових даних env або `models.json`.
Для усунення проблем OAuth OpenAI ChatGPT/Codex `openclaw models status`,
`openclaw models auth list --provider openai` і
`openclaw config get agents.defaults.model --json` — найшвидший спосіб
підтвердити, чи має агент придатний профіль OAuth `openai` для
`openai/*` через нативне середовище виконання Codex. Див. [Налаштування провайдера OpenAI](/uk/providers/openai#check-and-recover-codex-oauth-routing).

Примітки:

- `models set <model-or-alias>` приймає `provider/model` або псевдонім.
- `models list` працює лише для читання: вона читає конфігурацію, профілі автентифікації, наявний стан каталогу
  і рядки каталогу, якими володіють провайдери, але не перезаписує
  `models.json`.
- Стовпець `Auth` має рівень провайдера й доступний лише для читання. Він обчислюється з локальних
  метаданих профілів автентифікації, маркерів env, налаштованих ключів провайдерів, маркерів локальних провайдерів,
  маркерів env/профілю AWS Bedrock і синтетичних метаданих автентифікації Plugin;
  він не завантажує середовище виконання провайдера, не читає секрети keychain, не викликає
  API провайдера й не доводить точну готовність виконання для кожної моделі.
- `models list --all --provider <id>` може містити статичні рядки каталогу, якими володіє провайдер,
  з маніфестів Plugin або метаданих каталогу вбудованого провайдера, навіть коли ви
  ще не автентифікувалися в цього провайдера. Такі рядки все одно відображаються як
  недоступні, доки не буде налаштовано відповідну автентифікацію.
- `models list` зберігає площину керування чутливою, коли виявлення каталогу провайдера
  повільне. Подання за замовчуванням і налаштовані подання після короткого очікування повертаються до налаштованих або
  синтетичних рядків моделей і дають виявленню завершитися у
  фоновому режимі. Використовуйте `--all`, коли потрібен точний повний виявлений каталог і
  ви готові чекати на виявлення провайдера.
- Широкий `models list --all` об’єднує рядки каталогу маніфесту поверх рядків реєстру,
  не завантажуючи додаткові runtime-хуки провайдера. Швидкі шляхи маніфесту з фільтром за провайдером
  використовують лише провайдерів, позначених `static`; провайдери, позначені `refreshable`,
  залишаються підкріпленими реєстром/кешем і додають рядки маніфесту як доповнення, тоді як
  провайдери, позначені `runtime`, залишаються на виявленні через реєстр/runtime.
- `models list` тримає нативні метадані моделі й runtime-обмеження окремо. У табличному
  виводі `Ctx` показує `contextTokens/contextWindow`, коли ефективне runtime-обмеження
  відрізняється від нативного контекстного вікна; рядки JSON містять `contextTokens`,
  коли провайдер відкриває це обмеження.
- `models list --provider <id>` фільтрує за id провайдера, наприклад `moonshot` або
  `openai`. Вона не приймає відображувані назви з інтерактивних
  засобів вибору провайдера, як-от `Moonshot AI`.
- Посилання на моделі розбираються поділом за **першим** `/`. Якщо ID моделі містить `/` (у стилі OpenRouter), додайте префікс провайдера (приклад: `openrouter/moonshotai/kimi-k2`).
- Якщо ви опускаєте провайдера, OpenClaw спершу розв’язує введення як псевдонім, потім
  як унікальний збіг налаштованого провайдера для цього точного id моделі, і лише після цього
  повертається до налаштованого провайдера за замовчуванням із попередженням про застарілість.
  Якщо цей провайдер більше не відкриває налаштовану модель за замовчуванням, OpenClaw
  повертається до першої налаштованої пари провайдер/модель замість показу
  застарілого значення за замовчуванням для вилученого провайдера.
- `models status` може показувати `marker(<value>)` у виводі автентифікації для несекретних заповнювачів (наприклад `OPENAI_API_KEY`, `secretref-managed`, `minimax-oauth`, `oauth:chutes`, `ollama-local`) замість маскування їх як секретів.

### Сканування моделей

`models scan` читає публічний каталог OpenRouter `:free` і ранжує кандидатів для
використання як резервні варіанти. Сам каталог є публічним, тому сканування лише метаданих не потребує
ключа OpenRouter.

За замовчуванням OpenClaw намагається перевірити підтримку інструментів і зображень live-викликами моделей.
Якщо ключ OpenRouter не налаштовано, команда повертається до виводу лише метаданих
і пояснює, що моделі `:free` все одно потребують `OPENROUTER_API_KEY` для
перевірок і інференсу.

Параметри:

- `--no-probe` (лише метадані; без пошуку конфігурації/секретів)
- `--min-params <b>`
- `--max-age-days <days>`
- `--provider <name>`
- `--max-candidates <n>`
- `--timeout <ms>` (запит каталогу та тайм-аут для кожної перевірки)
- `--concurrency <n>`
- `--yes`
- `--no-input`
- `--set-default`
- `--set-image`
- `--json`

`--set-default` і `--set-image` потребують live-перевірок; результати сканування лише метаданих
є інформаційними й не застосовуються до конфігурації.

### Стан моделей

Параметри:

- `--json`
- `--plain`
- `--check` (код виходу 1=прострочено/відсутнє, 2=спливає)
- `--probe` (live-перевірка налаштованих профілів автентифікації)
- `--probe-provider <name>` (перевірити одного провайдера)
- `--probe-profile <id>` (повторювані або розділені комами id профілів)
- `--probe-timeout <ms>`
- `--probe-concurrency <n>`
- `--probe-max-tokens <n>`
- `--agent <id>` (id налаштованого агента; перевизначає `OPENCLAW_AGENT_DIR`)

`--json` залишає stdout зарезервованим для JSON-навантаження. Діагностика профілів автентифікації, провайдера
і запуску спрямовується до stderr, щоб скрипти могли передавати stdout напряму
в інструменти на кшталт `jq`.

Кошики станів перевірки:

- `ok`
- `auth`
- `rate_limit`
- `billing`
- `timeout`
- `format`
- `unknown`
- `no_model`

Очікувані випадки деталей/кодів причин перевірки:

- `excluded_by_auth_order`: збережений профіль існує, але явний
  `auth.order.<provider>` його пропустив, тому перевірка повідомляє про виключення замість
  спроби використати його.
- `missing_credential`, `invalid_expires`, `expired`, `unresolved_ref`:
  профіль наявний, але не придатний/не розв’язуваний.
- `no_model`: автентифікація провайдера існує, але OpenClaw не зміг розв’язати придатного до перевірки
  кандидата моделі для цього провайдера.

## Профілі автентифікації

```bash
openclaw models auth add
openclaw models auth list [--provider <id>] [--json]
openclaw models auth login --provider <id>
openclaw models auth login --provider openai --profile-id openai:work
openclaw models auth paste-api-key --provider <id>
openclaw models auth setup-token --provider <id>
openclaw models auth paste-token
```

`models auth add` — інтерактивний помічник автентифікації. Він може запустити потік автентифікації провайдера
(OAuth/API-ключ) або провести вас через ручне вставлення токена, залежно від
вибраного провайдера.

`models auth list` перелічує збережені профілі автентифікації для вибраного агента без
виведення токена, API-ключа або секретного матеріалу OAuth. Використовуйте `--provider <id>`, щоб
відфільтрувати одного провайдера, наприклад `openai`, і `--json` для скриптів.

`models auth login` запускає потік автентифікації Plugin провайдера (OAuth/API-ключ). Використовуйте
`openclaw plugins list`, щоб побачити, які провайдери встановлено.
Використовуйте `openclaw models auth --agent <id> <subcommand>`, щоб записати результати автентифікації до
сховища конкретного налаштованого агента. Батьківський прапорець `--agent` підтримується командами
`add`, `list`, `login`, `paste-api-key`, `setup-token`, `paste-token` і
`login-github-copilot`.

Для моделей OpenAI `--provider openai` за замовчуванням означає вхід в обліковий запис ChatGPT/Codex.
Використовуйте `--method api-key` лише коли хочете додати профіль API-ключа OpenAI,
зазвичай як резерв для лімітів підписки Codex. Запустіть `openclaw doctor --fix`,
щоб перенести старі legacy-стани автентифікації/профілів із префіксом OpenAI Codex до `openai`.

Приклади:

```bash
openclaw models auth login --provider openai --set-default
openclaw models auth login --provider openai --method api-key
openclaw models auth paste-api-key --provider openai
openclaw models auth list --provider openai
```

Примітки:

- `login` приймає `--profile-id <id>` для провайдерів, які підтримують іменовані
  профілі під час входу. Використовуйте це, щоб тримати кілька входів для того самого
  провайдера окремо.
- `paste-api-key` приймає API-ключі, згенеровані деінде, запитує значення ключа
  і записує його до id профілю за замовчуванням `<provider>:manual`, якщо ви не
  передасте `--profile-id`. В автоматизації передавайте ключ через stdin, наприклад
  `printf "%s\n" "$OPENAI_API_KEY" | openclaw models auth paste-api-key --provider openai`.
- `setup-token` і `paste-token` залишаються загальними командами токенів для провайдерів,
  які відкривають методи автентифікації токеном.
- `setup-token` потребує інтерактивного TTY і запускає метод token-auth провайдера
  (за замовчуванням метод `setup-token` цього провайдера, коли він його відкриває).
- `paste-token` приймає рядок токена, згенерований деінде або з автоматизації.
- `paste-token` потребує `--provider`, за замовчуванням запитує значення токена
  і записує його до id профілю за замовчуванням `<provider>:manual`, якщо ви не передасте
  `--profile-id`.
- В автоматизації передавайте токен через stdin замість передавання його як аргумент, щоб
  облікові дані провайдера не з’являлися в історії shell або списках процесів.
- `paste-token --expires-in <duration>` зберігає абсолютний час спливу токена з
  відносної тривалості, як-от `365d` або `12h`.
- Для `openai` API-ключі OpenAI і матеріал токенів ChatGPT/OAuth —
  різні форми автентифікації. Використовуйте `paste-api-key` для API-ключів OpenAI `sk-...` і
  `paste-token` лише для матеріалу автентифікації токеном.
- Примітка щодо Anthropic: співробітники Anthropic повідомили нам, що використання Claude CLI у стилі OpenClaw знову дозволене, тому OpenClaw вважає повторне використання Claude CLI і використання `claude -p` санкціонованими для цієї інтеграції, якщо Anthropic не опублікує нову політику.
- Anthropic `setup-token` / `paste-token` залишаються доступними як підтримуваний шлях токена OpenClaw, але OpenClaw тепер віддає перевагу повторному використанню Claude CLI і `claude -p`, коли вони доступні.
