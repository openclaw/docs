# `openclaw models`

Виявлення, сканування та налаштування моделей (модель за замовчуванням, резервні варіанти, профілі автентифікації).

Пов’язано:

- Постачальники + моделі: [Моделі](/uk/providers/models)
- Концепції вибору моделей + slash-команда `/models`: [Концепція моделей](/uk/concepts/models)
- Налаштування автентифікації постачальника: [Початок роботи](/uk/start/getting-started)

## Поширені команди

```bash
openclaw models status
openclaw models list
openclaw models set <model-or-alias>
openclaw models scan
```

`openclaw models status` показує визначені значення за замовчуванням/резервні варіанти, а також огляд автентифікації.
Коли доступні знімки використання постачальників, розділ стану OAuth/API-ключа містить
вікна використання постачальників і знімки квот.
Поточні постачальники вікон використання: Anthropic, GitHub Copilot, Gemini CLI, OpenAI
Codex, MiniMax, Xiaomi та z.ai. Автентифікація використання надходить із хуків, специфічних для постачальника,
коли вони доступні; інакше OpenClaw повертається до відповідних облікових даних OAuth/API-ключа
з профілів автентифікації, env або конфігурації.
У виводі `--json` `auth.providers` є оглядом постачальників із урахуванням env/конфігурації/сховища,
тоді як `auth.oauth` є лише станом профілів сховища автентифікації.
Додайте `--probe`, щоб виконати live-перевірки автентифікації для кожного налаштованого профілю постачальника.
Перевірки є справжніми запитами (можуть споживати токени й запускати обмеження швидкості).
Використовуйте `--agent <id>`, щоб переглянути стан моделі/автентифікації налаштованого агента. Якщо пропущено,
команда використовує `OPENCLAW_AGENT_DIR`/`PI_CODING_AGENT_DIR`, якщо задано, інакше
налаштованого агента за замовчуванням.
Рядки перевірок можуть походити з профілів автентифікації, облікових даних env або `models.json`.

Примітки:

- `models set <model-or-alias>` приймає `provider/model` або псевдонім.
- `models list` доступна лише для читання: вона читає конфігурацію, профілі автентифікації, наявний стан каталогу
  та рядки каталогу, якими володіє постачальник, але не перезаписує
  `models.json`.
- Стовпець `Auth` є рівнем постачальника й доступний лише для читання. Він обчислюється з локальних
  метаданих профілю автентифікації, маркерів env, налаштованих ключів постачальників, маркерів локальних постачальників,
  маркерів env/профілю AWS Bedrock і синтетичних метаданих автентифікації Plugin;
  він не завантажує runtime постачальника, не читає секрети keychain, не викликає API постачальника
  й не доводить точну готовність виконання для окремої моделі.
- `models list --all --provider <id>` може включати статичні рядки каталогу, якими володіє постачальник,
  із маніфестів Plugin або метаданих каталогу вбудованого постачальника, навіть якщо ви
  ще не автентифікувалися в цього постачальника. Такі рядки все одно показуються як
  недоступні, доки не буде налаштовано відповідну автентифікацію.
- `models list` зберігає responsiveness control plane, поки виявлення каталогу постачальника
  повільне. Подання за замовчуванням і налаштовані подання після короткого очікування повертаються до налаштованих або
  синтетичних рядків моделей і дають виявленню завершитися у
  фоновому режимі. Використовуйте `--all`, коли вам потрібен точний повний виявлений каталог і
  ви готові чекати на виявлення постачальника.
- Широкий `models list --all` об’єднує рядки каталогу маніфесту поверх рядків реєстру
  без завантаження додаткових хуків runtime постачальника. Швидкі шляхи маніфесту з фільтром за постачальником
  використовують лише постачальників, позначених `static`; постачальники, позначені `refreshable`,
  залишаються на основі реєстру/кешу й додають рядки маніфесту як доповнення, тоді як
  постачальники, позначені `runtime`, залишаються на виявленні реєстру/runtime.
- `models list` зберігає нативні метадані моделі й caps runtime окремо. У табличному
  виводі `Ctx` показує `contextTokens/contextWindow`, коли ефективний cap runtime
  відрізняється від нативного контекстного вікна; JSON-рядки містять `contextTokens`,
  коли постачальник надає цей cap.
- `models list --provider <id>` фільтрує за id постачальника, наприклад `moonshot` або
  `openai-codex`. Вона не приймає відображувані назви з інтерактивних вибірників постачальників,
  наприклад `Moonshot AI`.
- Посилання на моделі розбираються поділом за **першим** `/`. Якщо ID моделі містить `/` (у стилі OpenRouter), додайте префікс постачальника (приклад: `openrouter/moonshotai/kimi-k2`).
- Якщо ви пропустите постачальника, OpenClaw спершу визначає введене значення як псевдонім, потім
  як унікальний збіг налаштованого постачальника для цього точного id моделі, і лише тоді
  повертається до налаштованого постачальника за замовчуванням із попередженням про застарівання.
  Якщо цей постачальник більше не надає налаштовану модель за замовчуванням, OpenClaw
  повертається до першої налаштованої пари постачальник/модель замість показу
  застарілого значення за замовчуванням для видаленого постачальника.
- `models status` може показувати `marker(<value>)` у виводі автентифікації для несекретних placeholders (наприклад `OPENAI_API_KEY`, `secretref-managed`, `minimax-oauth`, `oauth:chutes`, `ollama-local`) замість маскування їх як секретів.

### Сканування моделей

`models scan` читає публічний каталог OpenRouter `:free` і ранжує кандидатів для
резервного використання. Сам каталог є публічним, тому сканування лише метаданих не потребують
ключа OpenRouter.

За замовчуванням OpenClaw намагається перевірити підтримку інструментів і зображень live-викликами моделей.
Якщо ключ OpenRouter не налаштовано, команда повертається до виводу лише метаданих
і пояснює, що моделі `:free` все одно потребують `OPENROUTER_API_KEY` для
перевірок і inference.

Параметри:

- `--no-probe` (лише метадані; без пошуку конфігурації/секретів)
- `--min-params <b>`
- `--max-age-days <days>`
- `--provider <name>`
- `--max-candidates <n>`
- `--timeout <ms>` (запит каталогу й тайм-аут для кожної перевірки)
- `--concurrency <n>`
- `--yes`
- `--no-input`
- `--set-default`
- `--set-image`
- `--json`

`--set-default` і `--set-image` потребують live-перевірок; результати сканування лише метаданих
є інформаційними й не застосовуються до конфігурації.

### Стан моделей

Параметри:

- `--json`
- `--plain`
- `--check` (exit 1=expired/missing, 2=expiring)
- `--probe` (live-перевірка налаштованих профілів автентифікації)
- `--probe-provider <name>` (перевірити одного постачальника)
- `--probe-profile <id>` (повторювані або розділені комами id профілів)
- `--probe-timeout <ms>`
- `--probe-concurrency <n>`
- `--probe-max-tokens <n>`
- `--agent <id>` (id налаштованого агента; перевизначає `OPENCLAW_AGENT_DIR`/`PI_CODING_AGENT_DIR`)

`--json` залишає stdout зарезервованим для JSON payload. Діагностика профілів автентифікації, постачальника
та запуску спрямовується до stderr, щоб scripts могли передавати stdout безпосередньо
в інструменти, такі як `jq`.

Кошики стану перевірок:

- `ok`
- `auth`
- `rate_limit`
- `billing`
- `timeout`
- `format`
- `unknown`
- `no_model`

Очікувані випадки деталей/кодів причин перевірки:

- `excluded_by_auth_order`: збережений профіль існує, але явний
  `auth.order.<provider>` пропустив його, тому перевірка повідомляє про виключення замість
  спроби його використати.
- `missing_credential`, `invalid_expires`, `expired`, `unresolved_ref`:
  профіль присутній, але не придатний/не розв’язується.
- `no_model`: автентифікація постачальника існує, але OpenClaw не зміг визначити кандидата моделі,
  придатного для перевірки для цього постачальника.

## Профілі автентифікації

```bash
openclaw models auth add
openclaw models auth list [--provider <id>] [--json]
openclaw models auth login --provider <id>
openclaw models auth setup-token --provider <id>
openclaw models auth paste-token
```

`models auth add` є інтерактивним помічником автентифікації. Він може запускати flow автентифікації постачальника
(OAuth/API key) або провести вас до ручного вставлення токена, залежно від
обраного постачальника.

`models auth list` перелічує збережені профілі автентифікації для вибраного агента без
виведення токена, API-ключа або секретного матеріалу OAuth. Використовуйте `--provider <id>`, щоб
відфільтрувати до одного постачальника, наприклад `openai-codex`, і `--json` для scripting.

`models auth login` запускає flow автентифікації Plugin постачальника (OAuth/API key). Використовуйте
`openclaw plugins list`, щоб побачити, які постачальники встановлені.
Використовуйте `openclaw models auth --agent <id> <subcommand>`, щоб записати результати автентифікації до
сховища конкретного налаштованого агента. Батьківський прапорець `--agent` враховується командами
`add`, `list`, `login`, `setup-token`, `paste-token` і
`login-github-copilot`.

Приклади:

```bash
openclaw models auth login --provider openai-codex --set-default
openclaw models auth list --provider openai-codex
```

Примітки:

- `setup-token` і `paste-token` залишаються загальними командами токенів для постачальників,
  які надають методи автентифікації через токен.
- `setup-token` потребує інтерактивного TTY і запускає метод автентифікації токена постачальника
  (за замовчуванням метод `setup-token` цього постачальника, коли він його надає).
- `paste-token` приймає рядок токена, створений деінде або автоматизацією.
- `paste-token` потребує `--provider`, запитує значення токена й записує
  його до id профілю за замовчуванням `<provider>:manual`, якщо ви не передасте
  `--profile-id`.
- `paste-token --expires-in <duration>` зберігає абсолютний строк дії токена з
  відносної тривалості, наприклад `365d` або `12h`.
- Примітка щодо Anthropic: співробітники Anthropic повідомили нам, що використання Claude CLI у стилі OpenClaw знову дозволене, тому OpenClaw вважає повторне використання Claude CLI і використання `claude -p` санкціонованими для цієї інтеграції, якщо Anthropic не опублікує нову політику.
- Anthropic `setup-token` / `paste-token` залишаються доступними як підтримуваний шлях токенів OpenClaw, але OpenClaw тепер віддає перевагу повторному використанню Claude CLI і `claude -p`, коли вони доступні.
