# `openclaw models`

Виявлення, сканування та конфігурація моделей (модель за замовчуванням, резервні варіанти, профілі автентифікації).

Пов’язано:

- Провайдери + моделі: [Моделі](/uk/providers/models)
- Концепції вибору моделей + slash-команда `/models`: [Концепція моделей](/uk/concepts/models)
- Налаштування автентифікації провайдера: [Початок роботи](/uk/start/getting-started)

## Поширені команди

```bash
openclaw models status
openclaw models list
openclaw models set <model-or-alias>
openclaw models scan
```

`openclaw models status` показує визначені значення за замовчуванням/резервні варіанти, а також огляд автентифікації.
Коли доступні знімки використання провайдерів, розділ стану OAuth/API-ключів містить
вікна використання провайдерів і знімки квот.
Поточні провайдери вікон використання: Anthropic, GitHub Copilot, Gemini CLI, OpenAI
Codex, MiniMax, Xiaomi і z.ai. Автентифікація використання надходить із провайдерських хуків,
коли вони доступні; інакше OpenClaw повертається до відповідних облікових даних OAuth/API-ключів
із профілів автентифікації, env або конфігурації.
У виводі `--json` `auth.providers` — це огляд провайдерів з урахуванням env/config/store,
тоді як `auth.oauth` — лише стан профілів auth-store.
Додайте `--probe`, щоб запустити живі перевірки автентифікації для кожного налаштованого профілю провайдера.
Перевірки — це реальні запити (можуть споживати токени й запускати обмеження швидкості).
Використовуйте `--agent <id>`, щоб перевірити стан моделі/автентифікації налаштованого агента. Якщо опущено,
команда використовує `OPENCLAW_AGENT_DIR`/`PI_CODING_AGENT_DIR`, якщо задано, інакше
налаштованого агента за замовчуванням.
Рядки перевірок можуть надходити з профілів автентифікації, облікових даних env або `models.json`.

Примітки:

- `models set <model-or-alias>` приймає `provider/model` або псевдонім.
- `models list` лише читає дані: вона читає конфігурацію, профілі автентифікації, наявний стан каталогу
  і рядки каталогу, якими володіє провайдер, але не переписує
  `models.json`.
- Стовпець `Auth` є рівнем провайдера й доступний лише для читання. Він обчислюється з локальних
  метаданих профілів автентифікації, маркерів env, налаштованих ключів провайдерів, маркерів локальних провайдерів,
  маркерів AWS Bedrock env/profile і синтетичних метаданих автентифікації Plugin;
  він не завантажує runtime провайдера, не читає секрети keychain, не викликає API провайдерів
  і не доводить точну готовність виконання для кожної моделі.
- `models list --all --provider <id>` може містити статичні рядки каталогу, якими володіє провайдер,
  із маніфестів Plugin або вбудованих метаданих каталогу провайдера, навіть якщо ви
  ще не автентифікувалися в цього провайдера. Такі рядки все одно відображаються як
  недоступні, доки не буде налаштовано відповідну автентифікацію.
- Широка команда `models list --all` об’єднує рядки каталогу з маніфестів поверх рядків реєстру
  без завантаження провайдерських хуків runtime supplement. Швидкі шляхи маніфестів із фільтром за провайдером
  використовують лише провайдерів, позначених як `static`; провайдери, позначені як `refreshable`,
  залишаються на основі реєстру/кешу й додають рядки маніфесту як supplement, тоді як
  провайдери, позначені як `runtime`, залишаються на виявленні через реєстр/runtime.
- `models list` тримає власні метадані моделі й runtime-обмеження окремо. У табличному
  виводі `Ctx` показує `contextTokens/contextWindow`, коли ефективне runtime-обмеження
  відрізняється від власного контекстного вікна; JSON-рядки містять `contextTokens`,
  коли провайдер надає таке обмеження.
- `models list --provider <id>` фільтрує за id провайдера, наприклад `moonshot` або
  `openai-codex`. Вона не приймає відображувані мітки з інтерактивних вибірників провайдера,
  наприклад `Moonshot AI`.
- Посилання на моделі розбираються поділом за **першим** `/`. Якщо ID моделі містить `/` (у стилі OpenRouter), додайте префікс провайдера (приклад: `openrouter/moonshotai/kimi-k2`).
- Якщо ви опустите провайдера, OpenClaw спершу визначає введення як псевдонім, потім
  як унікальний збіг налаштованого провайдера для цього точного id моделі, і лише потім
  повертається до налаштованого провайдера за замовчуванням із попередженням про застарівання.
  Якщо цей провайдер більше не надає налаштовану модель за замовчуванням, OpenClaw
  повертається до першого налаштованого провайдера/моделі замість показу
  застарілого значення за замовчуванням для вилученого провайдера.
- `models status` може показувати `marker(<value>)` у виводі автентифікації для несекретних заповнювачів (наприклад `OPENAI_API_KEY`, `secretref-managed`, `minimax-oauth`, `oauth:chutes`, `ollama-local`) замість маскування їх як секретів.

### Сканування моделей

`models scan` читає публічний каталог OpenRouter `:free` і ранжує кандидатів для
використання як резервних. Сам каталог є публічним, тому сканування лише метаданих не потребує
ключа OpenRouter.

За замовчуванням OpenClaw намагається перевірити підтримку інструментів і зображень живими викликами моделі.
Якщо ключ OpenRouter не налаштовано, команда повертається до виводу лише метаданих
і пояснює, що моделі `:free` усе одно потребують `OPENROUTER_API_KEY` для
перевірок і inference.

Параметри:

- `--no-probe` (лише метадані; без пошуку конфігурації/секретів)
- `--min-params <b>`
- `--max-age-days <days>`
- `--provider <name>`
- `--max-candidates <n>`
- `--timeout <ms>` (запит каталогу й тайм-аут для кожної перевірки)
- `--concurrency <n>`
- `--yes`
- `--no-input`
- `--set-default`
- `--set-image`
- `--json`

`--set-default` і `--set-image` потребують живих перевірок; результати сканування
лише метаданих є інформаційними й не застосовуються до конфігурації.

### Стан моделей

Параметри:

- `--json`
- `--plain`
- `--check` (код виходу 1=прострочено/відсутнє, 2=спливає)
- `--probe` (жива перевірка налаштованих профілів автентифікації)
- `--probe-provider <name>` (перевірити одного провайдера)
- `--probe-profile <id>` (повторювані або розділені комами id профілів)
- `--probe-timeout <ms>`
- `--probe-concurrency <n>`
- `--probe-max-tokens <n>`
- `--agent <id>` (id налаштованого агента; перевизначає `OPENCLAW_AGENT_DIR`/`PI_CODING_AGENT_DIR`)

`--json` залишає stdout зарезервованим для JSON-навантаження. Діагностика профілів автентифікації, провайдера
і запуску спрямовується до stderr, щоб скрипти могли напряму передавати stdout
в інструменти на кшталт `jq`.

Групи станів перевірки:

- `ok`
- `auth`
- `rate_limit`
- `billing`
- `timeout`
- `format`
- `unknown`
- `no_model`

Очікувані випадки деталей/кодів причин перевірки:

- `excluded_by_auth_order`: збережений профіль існує, але явний
  `auth.order.<provider>` його пропустив, тому перевірка повідомляє про виключення замість
  спроби використати його.
- `missing_credential`, `invalid_expires`, `expired`, `unresolved_ref`:
  профіль присутній, але не придатний/не може бути визначений.
- `no_model`: автентифікація провайдера існує, але OpenClaw не зміг визначити придатного до перевірки
  кандидата моделі для цього провайдера.

## Профілі автентифікації

```bash
openclaw models auth add
openclaw models auth login --provider <id>
openclaw models auth setup-token --provider <id>
openclaw models auth paste-token
```

`models auth add` — це інтерактивний помічник автентифікації. Він може запустити потік автентифікації провайдера
(OAuth/API-ключ) або провести вас до ручного вставлення токена, залежно від
вибраного провайдера.

`models auth login` запускає потік автентифікації провайдерського Plugin (OAuth/API-ключ). Використовуйте
`openclaw plugins list`, щоб побачити, які провайдери встановлено.
Використовуйте `openclaw models auth --agent <id> <subcommand>`, щоб записати результати автентифікації до
сховища конкретного налаштованого агента. Батьківський прапорець `--agent` враховується командами
`add`, `login`, `setup-token`, `paste-token` і `login-github-copilot`.

Приклади:

```bash
openclaw models auth login --provider openai-codex --set-default
```

Примітки:

- `setup-token` і `paste-token` залишаються універсальними командами токенів для провайдерів,
  які надають методи автентифікації токеном.
- `setup-token` потребує інтерактивного TTY і запускає метод token-auth провайдера
  (за замовчуванням використовуючи метод `setup-token` цього провайдера, коли він його надає).
- `paste-token` приймає рядок токена, згенерований деінде або з автоматизації.
- `paste-token` потребує `--provider`, запитує значення токена й записує
  його до id профілю за замовчуванням `<provider>:manual`, якщо ви не передасте
  `--profile-id`.
- `paste-token --expires-in <duration>` зберігає абсолютний час завершення дії токена з
  відносної тривалості, наприклад `365d` або `12h`.
- Примітка Anthropic: співробітники Anthropic повідомили нам, що використання Claude CLI у стилі OpenClaw знову дозволене, тому OpenClaw вважає повторне використання Claude CLI і `claude -p` санкціонованими для цієї інтеграції, якщо Anthropic не опублікує нову політику.
- Anthropic `setup-token` / `paste-token` залишаються доступними як підтримуваний шлях токенів OpenClaw, але OpenClaw тепер надає перевагу повторному використанню Claude CLI і `claude -p`, коли вони доступні.
