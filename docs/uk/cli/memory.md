# `openclaw memory`

Керуйте індексуванням і пошуком семантичної пам’яті, а також перенесенням даних до `MEMORY.md`.
Функціональність надає вбудований Plugin `memory-core`, доступний, коли
`plugins.slots.memory` вибирає `memory-core` (за замовчуванням). Інші Plugin
пам’яті мають власні простори імен CLI.

Пов’язані матеріали: концепції [Пам’ять](/uk/concepts/memory) і [Dreaming](/uk/concepts/dreaming),
[Довідник із конфігурації пам’яті](/uk/reference/memory-config), [Вікі пам’яті](/uk/plugins/memory-wiki),
[вікі](/uk/cli/wiki), [Plugin](/uk/tools/plugin).

## `memory status`

```bash
openclaw memory status [--agent <id>] [--deep] [--index] [--fix] [--json] [--verbose]
```

Без `--agent` команда виконується для кожного агента в `agents.list`; якщо список
агентів не налаштовано, використовується агент за замовчуванням.

| Прапорець   | Дія                                                                                                                                                                                                                                                                                                                                 |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--deep`    | Перевіряє готовність векторного сховища, постачальника вбудовувань і семантичного пошуку (передбачає додаткові виклики постачальника). Звичайна команда `memory status` працює швидко й пропускає цю перевірку; невідомий стан векторного або семантичного пошуку означає, що його не перевіряли. Лексичний режим QMD `searchMode: "search"` завжди пропускає семантичні векторні перевірки, навіть із `--deep`. |
| `--index`   | Повторно індексує дані, якщо сховище має незбережені зміни. Передбачає `--deep`.                                                                                                                                                                                                                                                      |
| `--fix`     | Виправляє застарілі блокування пригадування та нормалізує метадані перенесення.                                                                                                                                                                                                                                                       |
| `--json`    | Виводить JSON.                                                                                                                                                                                                                                                                                                                       |
| `--verbose` | Виводить докладні журнали для кожної фази.                                                                                                                                                                                                                                                                                           |

Якщо рядок `Dreaming` залишається в стані `off` навіть із
`dreaming.enabled: true` або заплановані проходи не виконуються, керований Cron
Dreaming залежить від спрацювання Heartbeat агента за замовчуванням, яке запускає
узгодження. Докладніше про планування див. у розділі
[Dreaming](/uk/concepts/dreaming).

У стані також перелічено всі додаткові шляхи пошуку з `agents.defaults.memorySearch.extraPaths`.

## `memory index`

```bash
openclaw memory index [--agent <id>] [--force] [--verbose]
```

Область дії для кожного агента така сама, як у `status`. `--force` запускає повне
повторне індексування замість інкрементального. `--verbose` виводить відомості про
постачальника, модель, джерела й додаткові шляхи кожного агента перед відображенням
перебігу індексування.

## `memory search`

```bash
openclaw memory search [query] [--query <text>] [--agent <id>] [--max-results <n>] [--min-score <n>] [--json]
```

- Запит: позиційний `[query]` або `--query <text>`. Якщо вказано обидва,
  пріоритет має `--query`. Якщо не вказано жодного, команда завершується помилкою.
- `--agent <id>`: за замовчуванням використовується агент за замовчуванням
  (не повний список агентів).
- `--max-results <n>`: обмежує кількість результатів (додатне ціле число).
- `--min-score <n>`: відфільтровує збіги з оцінкою нижче вказаної.

## `memory promote`

Ранжує короткострокові кандидатні записи з `memory/YYYY-MM-DD.md` і за потреби
додає найкращі записи до `MEMORY.md`.

```bash
openclaw memory promote [--agent <id>] [--limit <n>] [--min-score <n>] \
  [--min-recall-count <n>] [--min-unique-queries <n>] [--apply] [--include-promoted] [--json]
```

| Прапорець                  | Значення за замовчуванням | Дія                                                               |
| -------------------------- | ------------------------- | ----------------------------------------------------------------- |
| `--limit <n>`              |                           | Максимальна кількість кандидатів для повернення або застосування. |
| `--min-score <n>`          | `0.75`                    | Мінімальна зважена оцінка перенесення.                             |
| `--min-recall-count <n>`   | `3`                       | Мінімальна необхідна кількість пригадувань.                        |
| `--min-unique-queries <n>` | `2`                       | Мінімальна необхідна кількість різних запитів.                     |
| `--apply`                  | лише попередній перегляд  | Додає вибраних кандидатів до `MEMORY.md` і позначає їх перенесеними. |
| `--include-promoted`       |                           | Включає кандидатів, уже перенесених у попередніх циклах.           |
| `--json`                   |                           | Виводить JSON.                                                     |

Ці стандартні значення CLI відрізняються від порогових значень глибокої фази
запланованого проходу Dreaming (див. [Dreaming](#dreaming) нижче); щоб відтворити
поведінку проходу під час одноразового ручного запуску, явно передайте прапорці.

Сигнали ранжування: частота пригадування, релевантність отримання, різноманітність
запитів, часова актуальність, консолідація між днями та насиченість похідними
концепціями. Вони надходять як із пригадувань пам’яті, так і зі щоденних проходів
завантаження даних, із додатковим невеликим посиленням на легкій фазі та фазі REM
для повторних переглядів Dreaming. Перед записуванням процес перенесення повторно
читає поточну щоденну нотатку, тому враховує редагування або видалення
короткострокових фрагментів після ранжування, замість перенесення із застарілого
знімка.

## `memory promote-explain`

Пояснює складові оцінки одного кандидата на перенесення.

```bash
openclaw memory promote-explain <selector> [--agent <id>] [--include-promoted] [--json]
```

`<selector>` зіставляється з ключем кандидата (точно або за підрядком), шляхом чи
текстом фрагмента.

## `memory rem-harness`

Попередньо показує роздуми REM, кандидатні істини та результати перенесення
глибокої фази, нічого не записуючи.

```bash
openclaw memory rem-harness [--agent <id>] [--path <file-or-dir>] [--grounded] [--include-promoted] [--json]
```

- `--path <file-or-dir>`: наповнює тестове середовище даними з історичних
  щоденних файлів `YYYY-MM-DD.md` замість поточного робочого простору.
- `--grounded`: також формує на основі історичних нотаток обґрунтований попередній
  перегляд розділів `Що сталося` / `Роздуми` / `Можливі довготривалі оновлення`.

## `memory rem-backfill`

Записує обґрунтовані історичні підсумки REM до `DREAMS.md` для перегляду в
інтерфейсі. Зміни можна скасувати.

```bash
openclaw memory rem-backfill --path <file-or-dir> [--agent <id>] [--stage-short-term] [--json]
openclaw memory rem-backfill --rollback [--rollback-short-term] [--json]
```

- `--path <file-or-dir>`: обов’язковий, якщо не вказано
  `--rollback`/`--rollback-short-term`. Історичні щоденні файли пам’яті або каталог,
  з яких потрібно дозаповнити дані.
- `--stage-short-term`: також додає обґрунтованих кандидатів на довготривале
  зберігання до поточного сховища короткострокового перенесення, щоб звичайна
  глибока фаза могла їх ранжувати.
- `--rollback`: видаляє раніше записані обґрунтовані записи щоденника з
  `DREAMS.md`.
- `--rollback-short-term`: видаляє раніше підготовлених обґрунтованих
  короткострокових кандидатів.

## Dreaming

Dreaming — фонова система консолідації пам’яті з трьома узгодженими фазами, які
виконуються послідовно за одним розкладом: **легка** (сортування й підготовка
короткострокового матеріалу), **REM** (осмислення й виявлення тем), **глибока**
(перенесення довготривалих фактів до `MEMORY.md`). Лише глибока фаза записує до
`MEMORY.md`.

- Увімкніть за допомогою
  `plugins.entries.memory-core.config.dreaming.enabled: true` (за замовчуванням
  `false`); `memory-core` автоматично керує завданням Cron для проходів, тому
  вручну виконувати `openclaw cron add` не потрібно.
- Перемикайте в чаті за допомогою `/dreaming on|off`; перевіряйте стан через
  `/dreaming status` (або `/dreaming`/`/dreaming help`). Для `on`/`off` потрібен
  статус власника каналу або `operator.admin` у Gateway; стан і довідка доступні
  всім, хто може викликати команду.
- Зрозумілий для людини результат фаз записується до `DREAMS.md` (або наявного
  `dreams.md`). За замовчуванням (`dreaming.storage.mode: "separate"`) кожна фаза
  також записує окремий звіт до `memory/dreaming/<phase>/YYYY-MM-DD.md`; установіть
  `mode: "inline"`, щоб натомість додавати звіти до щоденного файлу пам’яті, або
  `"both"`, щоб використовувати обидва варіанти.
- Заплановані й ручні запуски `memory promote` використовують однакові сигнали
  ранжування глибокої фази; відрізняються лише стандартні порогові значення
  (порівняйте таблицю вище зі стандартними значеннями для запланованих запусків
  нижче).
- Заплановані запуски розгалужуються між робочими просторами пам’яті всіх
  налаштованих агентів.

Стандартні значення для запланованих запусків
(`plugins.entries.memory-core.config.dreaming`):

| Ключ                                   | Значення за замовчуванням |
| -------------------------------------- | ------------------------- |
| `frequency`                            | `0 3 * * *`               |
| `phases.deep.minScore`                 | `0.8`                     |
| `phases.deep.minRecallCount`           | `3`                       |
| `phases.deep.minUniqueQueries`         | `3`                       |
| `phases.deep.recencyHalfLifeDays`      | `14`                      |
| `phases.deep.maxAgeDays`               | `30`                      |
| `phases.deep.maxPromotedSnippetTokens` | `160`                     |

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Повний список ключів і докладний опис фаз: [Dreaming](/uk/concepts/dreaming),
[Довідник із конфігурації пам’яті](/uk/reference/memory-config#dreaming).

## Залежність Gateway для SecretRef

Якщо поля ключів віддаленого API активної пам’яті налаштовано як SecretRef,
команди `memory` отримують їх з активного знімка Gateway; якщо Gateway
недоступний, команда негайно завершується помилкою. Для цього потрібен Gateway,
який підтримує метод `secrets.resolve`; старіші Gateway повертають помилку
невідомого методу.
