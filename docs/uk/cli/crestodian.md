# `openclaw crestodian`

Crestodian — це локальний помічник OpenClaw для налаштування, відновлення та конфігурації. Його
створено так, щоб він залишався доступним, коли звичайний шлях агента зламано.

Запуск `openclaw` без команди відкриває Crestodian в інтерактивному терміналі.
Запуск `openclaw crestodian` явно запускає того самого помічника.

## Що показує Crestodian

Під час запуску Crestodian виводить стислий огляд системи:

- шлях до конфігурації та її валідність
- налаштовані агенти й типовий агент
- типова модель
- доступність локальних CLI Codex і Claude Code
- наявність API-ключів OpenAI та Anthropic
- режим планувальника (`deterministic` або з підтримкою моделі через налаштовану модель)
- шлях до локальної документації або URL публічної документації
- шлях до локального вихідного коду для Git checkout, або URL вихідного коду OpenClaw на GitHub
- доступність Gateway
- негайно рекомендований наступний крок

Він не вивантажує секрети й не завантажує CLI-команди Plugin лише для запуску.

Crestodian використовує те саме виявлення довідкових ресурсів OpenClaw, що й звичайні агенти. У Git checkout
він спрямовує себе на локальний `docs/` і локальне дерево вихідного коду. В npm-встановленні пакета він
використовує вбудовану документацію пакета та посилання на
[https://github.com/openclaw/openclaw](https://github.com/openclaw/openclaw), із явною
рекомендацією переглядати вихідний код, коли документації недостатньо.

## Приклади

```bash
openclaw
openclaw crestodian
openclaw crestodian --json
openclaw crestodian --message "models"
openclaw crestodian --message "validate config"
openclaw crestodian --message "setup workspace ~/Projects/work model openai/gpt-5.5" --yes
openclaw crestodian --message "set default model openai/gpt-5.5" --yes
openclaw onboard --modern
```

Усередині інтерактивного запиту:

```text
status
health
doctor
doctor fix
validate config
setup
setup workspace ~/Projects/work model openai/gpt-5.5
config set gateway.port 19001
config set-ref gateway.auth.token env OPENCLAW_GATEWAY_TOKEN
gateway status
restart gateway
agents
create agent work workspace ~/Projects/work
models
set default model openai/gpt-5.5
talk to work agent
talk to agent for ~/Projects/work
audit
quit
```

## Безпечний запуск

Шлях запуску Crestodian навмисно мінімальний. Він може працювати, коли:

- `openclaw.json` відсутній
- `openclaw.json` невалідний
- Gateway недоступний
- реєстрація команд Plugin недоступна
- ще не налаштовано жодного агента

`openclaw --help` і `openclaw --version` як і раніше використовують звичайні швидкі шляхи.
Неінтерактивний `openclaw` завершується коротким повідомленням замість виведення кореневої
довідки, оскільки продукт без указання команди — це Crestodian.

## Операції та схвалення

Crestodian використовує типізовані операції замість довільного редагування конфігурації.

Операції лише для читання можуть виконуватися відразу:

- показати огляд
- перелічити агентів
- показати стан моделі/бекенда
- виконати перевірки status або health
- перевірити доступність Gateway
- запустити doctor без інтерактивних виправлень
- перевірити конфігурацію
- показати шлях до журналу аудиту

Постійні операції вимагають схвалення в діалозі в інтерактивному режимі, якщо
ви не передасте `--yes` для прямої команди:

- записати конфігурацію
- виконати `config set`
- встановити підтримувані значення SecretRef через `config set-ref`
- запустити bootstrap налаштування/онбордингу
- змінити типову модель
- запустити, зупинити або перезапустити Gateway
- створити агентів
- запустити відновлення doctor, які переписують конфігурацію або стан

Застосовані записи фіксуються в:

```text
~/.openclaw/audit/crestodian.jsonl
```

Виявлення не аудитується. Журналюються лише застосовані операції та записи.

`openclaw onboard --modern` запускає Crestodian як сучасний попередній перегляд онбордингу.
Звичайний `openclaw onboard` і далі запускає класичний онбординг.

## Bootstrap налаштування

`setup` — це bootstrap онбордингу з підходом chat-first. Він записує дані лише через типізовані
операції конфігурації й спочатку запитує схвалення.

```text
setup
setup workspace ~/Projects/work
setup workspace ~/Projects/work model openai/gpt-5.5
```

Коли модель не налаштована, setup вибирає перший придатний бекенд у такому
порядку та повідомляє, що саме він вибрав:

- наявна явно вказана модель, якщо вже налаштована
- `OPENAI_API_KEY` -> `openai/gpt-5.5`
- `ANTHROPIC_API_KEY` -> `anthropic/claude-opus-4-7`
- CLI Claude Code -> `claude-cli/claude-opus-4-7`
- CLI Codex -> `codex-cli/gpt-5.5`

Якщо жоден не доступний, setup однаково записує типове робоче середовище й залишає
модель невстановленою. Установіть або увійдіть у Codex/Claude Code, або надайте
`OPENAI_API_KEY`/`ANTHROPIC_API_KEY`, а потім знову запустіть setup.

## Планувальник із підтримкою моделі

Crestodian завжди запускається в режимі deterministic. Для нечітких команд, які
детермінований парсер не розуміє, локальний Crestodian може зробити один обмежений
хід планувальника через звичайні шляхи виконання OpenClaw. Спочатку він використовує
налаштовану модель OpenClaw. Якщо жодна налаштована модель ще не придатна до використання,
він може перейти до локальних runtime, які вже є на машині:

- CLI Claude Code: `claude-cli/claude-opus-4-7`
- harness app-server Codex: `openai/gpt-5.5` з `embeddedHarness.runtime: "codex"`
- CLI Codex: `codex-cli/gpt-5.5`

Планувальник із підтримкою моделі не може безпосередньо змінювати конфігурацію. Він має
перетворити запит на одну з типізованих команд Crestodian, після чого застосовуються
звичайні правила схвалення та аудиту. Crestodian виводить модель, яку він використав, і
інтерпретовану команду до виконання будь-яких дій. Ходи резервного планувальника без конфігурації
є тимчасовими, з вимкненими інструментами там, де це підтримує runtime, і використовують
тимчасове робоче середовище/сеанс.

Режим порятунку каналу повідомлень не використовує планувальник із підтримкою моделі. Віддалений
режим порятунку залишається deterministic, щоб зламаний або скомпрометований звичайний шлях агента
не можна було використати як редактор конфігурації.

## Перемикання до агента

Використовуйте селектор природною мовою, щоб вийти з Crestodian і відкрити звичайний TUI:

```text
talk to agent
talk to work agent
switch to main agent
```

`openclaw tui`, `openclaw chat` і `openclaw terminal` як і раніше безпосередньо відкривають звичайний
TUI агента. Вони не запускають Crestodian.

Після перемикання до звичайного TUI використовуйте `/crestodian`, щоб повернутися до Crestodian.
Ви можете додати наступний запит:

```text
/crestodian
/crestodian restart gateway
```

Перемикання агента всередині TUI залишає підказку про те, що доступний `/crestodian`.

## Режим порятунку повідомлень

Режим порятунку повідомлень — це точка входу каналу повідомлень для Crestodian. Він призначений
для випадків, коли ваш звичайний агент не працює, але довірений канал, такий як WhatsApp,
усе ще приймає команди.

Підтримувана текстова команда:

- `/crestodian <request>`

Потік дій оператора:

```text
You, in a trusted owner DM: /crestodian status
OpenClaw: Crestodian rescue mode. Gateway reachable: no. Config valid: no.
You: /crestodian restart gateway
OpenClaw: Plan: restart the Gateway. Reply /crestodian yes to apply.
You: /crestodian yes
OpenClaw: Applied. Audit entry written.
```

Створення агента також можна поставити в чергу з локального запиту або з режиму порятунку:

```text
create agent work workspace ~/Projects/work model openai/gpt-5.5
/crestodian create agent work workspace ~/Projects/work
```

Віддалений режим порятунку — це адміністративна поверхня. До нього треба ставитися як до
віддаленого відновлення конфігурації, а не як до звичайного чату.

Контракт безпеки для віддаленого режиму порятунку:

- Вимкнений, коли активне sandboxing. Якщо агент/сеанс у sandbox, 
  Crestodian має відмовити у віддаленому режимі порятунку й пояснити, що потрібне локальне відновлення через CLI.
- Типовий ефективний стан — `auto`: дозволяти віддалений режим порятунку лише в довіреному режимі роботи YOLO,
  де runtime уже має незасandboxену локальну владу.
- Вимагати явну ідентичність власника. Режим порятунку не повинен приймати шаблонні правила відправника,
  політику відкритих груп, неавтентифіковані Webhook або анонімні канали.
- Типово лише owner DM. Порятунок у групі/каналі вимагає явної згоди й
  усе одно має спрямовувати запити на схвалення до owner DM.
- Віддалений режим порятунку не може відкрити локальний TUI або перемкнутися до інтерактивного сеансу агента.
  Для передачі керування агенту використовуйте локальний `openclaw`.
- Постійні записи все одно вимагають схвалення, навіть у режимі порятунку.
- Аудитувати кожну застосовану операцію порятунку, включно з каналом, обліковим записом, відправником,
  ключем сеансу, операцією, хешем конфігурації до та хешем конфігурації після.
- Ніколи не повторювати секрети. Перевірка SecretRef має повідомляти про доступність, а не значення.
- Якщо Gateway працює, віддавайте перевагу типізованим операціям Gateway. Якщо Gateway не працює,
  використовуйте лише мінімальну локальну поверхню відновлення, яка не залежить від звичайного циклу агента.

Форма конфігурації:

```jsonc
{
  "crestodian": {
    "rescue": {
      "enabled": "auto",
      "ownerDmOnly": true,
    },
  },
}
```

`enabled` має приймати:

- `"auto"`: типово. Дозволяти лише тоді, коли ефективний runtime — YOLO і
  sandboxing вимкнено.
- `false`: ніколи не дозволяти режим порятунку через канал повідомлень.
- `true`: явно дозволяти режим порятунку, коли проходять перевірки власника/каналу. Це
  все одно не повинно обходити заборону sandboxing.

Типова поза YOLO для `"auto"` така:

- режим sandbox вирішується як `off`
- `tools.exec.security` вирішується як `full`
- `tools.exec.ask` вирішується як `off`

Віддалений режим порятунку покривається Docker lane:

```bash
pnpm test:docker:crestodian-rescue
```

Локальний резервний планувальник без конфігурації покривається:

```bash
pnpm test:docker:crestodian-planner
```

Опційна жива перевірка поверхні команд каналу перевіряє `/crestodian status`, а також
постійний цикл схвалення через обробник режиму порятунку:

```bash
pnpm test:live:crestodian-rescue-channel
```

Свіже налаштування без конфігурації через Crestodian покривається:

```bash
pnpm test:docker:crestodian-first-run
```
