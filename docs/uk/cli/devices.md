# `openclaw devices`

Керування запитами на сполучення пристроїв і токенами в межах пристрою.

## Команди

### `openclaw devices list`

Показати запити на сполучення, що очікують підтвердження, і сполучені пристрої.

```
openclaw devices list
openclaw devices list --json
```

У виводі запитів, що очікують підтвердження, запитуваний доступ показується поруч із поточним підтвердженим доступом пристрою, якщо пристрій уже сполучено. Це робить підвищення scope/ролі явними, замість того щоб створювати враження, ніби сполучення було втрачено.

### `openclaw devices remove <deviceId>`

Видалити один запис сполученого пристрою.

Коли ви автентифіковані токеном сполученого пристрою, виклики без прав адміністратора можуть видаляти лише **власний** запис пристрою. Видалення іншого пристрою потребує `operator.admin`.

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

Масово очистити сполучені пристрої.

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

Підтвердити запит на сполучення пристрою, що очікує підтвердження, за точним `requestId`. Якщо `requestId` пропущено або передано `--latest`, OpenClaw лише виводить вибраний запит, що очікує підтвердження, і завершує роботу; повторно запустіть підтвердження з точним ID запиту після перевірки деталей.

<Note>
Якщо пристрій повторює спробу сполучення зі зміненими даними автентифікації (роль, scopes або публічний ключ), OpenClaw замінює попередній запис, що очікує підтвердження, і видає новий `requestId`. Виконайте `openclaw devices list` безпосередньо перед підтвердженням, щоб використати актуальний ID.
</Note>

Якщо пристрій уже сполучений і запитує ширші scopes або ширшу роль, OpenClaw зберігає наявне підтвердження та створює новий запит на підвищення, що очікує підтвердження. Перегляньте стовпці `Requested` і `Approved` у `openclaw devices list` або скористайтеся `openclaw devices approve --latest`, щоб попередньо переглянути точне підвищення перед підтвердженням.

Якщо для Gateway явно налаштовано `gateway.nodes.pairing.autoApproveCidrs`, запити першого сполучення з `role: node` від клієнтських IP-адрес, що збігаються, можуть бути підтверджені до того, як з’являться в цьому списку. Ця політика вимкнена за замовчуванням і ніколи не застосовується до клієнтів operator/browser або до запитів на підвищення.

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

Відхилити запит на сполучення пристрою, що очікує підтвердження.

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

Виконати ротацію токена пристрою для певної ролі (за потреби також оновивши scopes).
Цільова роль уже має існувати в підтвердженому контракті сполучення цього пристрою;
ротація не може створити нову непідтверджену роль.
Якщо ви не вкажете `--scope`, подальші повторні підключення зі збереженим токеном після ротації використовуватимуть кешовані підтверджені scopes цього токена. Якщо ви передасте явні значення `--scope`, вони стануть збереженим набором scopes для майбутніх повторних підключень із кешованим токеном.
Виклики від сполучених пристроїв без прав адміністратора можуть виконувати ротацію лише **власного** токена пристрою.
Цільовий набір scopes токена має залишатися в межах власних операторських scopes сесії виклику; ротація не може створити або зберегти ширший операторський токен, ніж той, який уже має виклик.

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

Повертає нове корисне навантаження токена у форматі JSON.

### `openclaw devices revoke --device <id> --role <role>`

Відкликати токен пристрою для певної ролі.

Виклики від сполучених пристроїв без прав адміністратора можуть відкликати лише **власний** токен пристрою.
Відкликання токена іншого пристрою потребує `operator.admin`.
Цільовий набір scopes токена також має вкладатися у власні операторські scopes сесії виклику; виклики лише зі сполученням не можуть відкликати адміністративні або записувальні операторські токени.

```
openclaw devices revoke --device <deviceId> --role node
```

Повертає результат відкликання у форматі JSON.

## Поширені параметри

- `--url <url>`: URL WebSocket Gateway (типово `gateway.remote.url`, якщо налаштовано).
- `--token <token>`: токен Gateway (якщо потрібен).
- `--password <password>`: пароль Gateway (автентифікація за паролем).
- `--timeout <ms>`: тайм-аут RPC.
- `--json`: вивід у форматі JSON (рекомендовано для сценаріїв).

<Warning>
Коли ви задаєте `--url`, CLI не повертається до облікових даних із конфігурації чи середовища. Явно передайте `--token` або `--password`. Відсутність явно заданих облікових даних є помилкою.
</Warning>

## Примітки

- Ротація токена повертає новий токен (чутливі дані). Поводьтеся з ним як із секретом.
- Ці команди потребують scope `operator.pairing` (або `operator.admin`).
- `gateway.nodes.pairing.autoApproveCidrs` — це політика Gateway з явним увімкненням лише для нового сполучення пристроїв node; вона не змінює повноваження CLI на підтвердження.
- Ротація та відкликання токенів залишаються в межах підтвердженого набору ролей сполучення та базового підтвердженого набору scopes для цього пристрою. Випадковий запис кешованого токена не надає ціль для керування токеном.
- Для сесій токенів сполучених пристроїв керування іншими пристроями доступне лише адміністратору: `remove`, `rotate` і `revoke` дозволені лише для себе, якщо виклик не має `operator.admin`.
- Зміна токенів також обмежена scopes виклику: сесія лише зі сполученням не може виконати ротацію або відкликання токена, який наразі має `operator.admin` або `operator.write`.
- `devices clear` навмисно захищено прапорцем `--yes`.
- Якщо scope сполучення недоступний на local loopback (і явний `--url` не передано), list/approve можуть використовувати локальний резервний механізм сполучення.
- `devices approve` потребує явного ID запиту перед створенням токенів; якщо не вказати `requestId` або передати `--latest`, буде лише попередньо показано найновіший запит, що очікує підтвердження.

## Контрольний список відновлення при розсинхронізації токенів

Використовуйте це, коли Control UI або інші клієнти постійно завершуються з `AUTH_TOKEN_MISMATCH` або `AUTH_DEVICE_TOKEN_MISMATCH`.

1. Підтвердьте поточне джерело токена gateway:

```bash
openclaw config get gateway.auth.token
```

2. Перелічіть сполучені пристрої та визначте ID проблемного пристрою:

```bash
openclaw devices list
```

3. Виконайте ротацію операторського токена для проблемного пристрою:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. Якщо ротації недостатньо, видаліть застаріле сполучення та підтвердьте знову:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. Повторіть спробу підключення клієнта з поточним спільним токеном/паролем.

Примітки:

- Звичайний пріоритет автентифікації при повторному підключенні: спочатку явний спільний токен/пароль, потім явний `deviceToken`, потім збережений токен пристрою, потім bootstrap-токен.
- Для довіреного відновлення `AUTH_TOKEN_MISMATCH` тимчасово можна надіслати разом і спільний токен, і збережений токен пристрою для однієї обмеженої повторної спроби.
