# `openclaw devices`

Керуйте запитами на сполучення пристроїв і токенами, прив’язаними до пристроїв.

## Команди

### `openclaw devices list`

Показати запити на сполучення, що очікують на розгляд, і сполучені пристрої.

```
openclaw devices list
openclaw devices list --json
```

Вивід запитів, що очікують на розгляд, показує запитуваний доступ поруч із поточним
схваленим доступом пристрою, якщо цей пристрій уже сполучений. Це робить
підвищення області дії/ролі явними, замість того щоб створювати враження, ніби
сполучення було втрачено.

### `openclaw devices remove <deviceId>`

Видалити один запис про сполучений пристрій.

Якщо ви автентифіковані за допомогою токена сполученого пристрою, виклики від
неадміністраторів можуть видаляти лише запис **власного** пристрою. Видалення
будь-якого іншого пристрою вимагає `operator.admin`.

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

Масово очистити сполучені пристрої.

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

Схвалити запит на сполучення пристрою, що очікує на розгляд, за точним `requestId`. Якщо `requestId`
не вказано або передано `--latest`, OpenClaw лише виводить вибраний запит, що очікує на розгляд,
і завершує роботу; повторно запустіть схвалення з точним ID запиту після перевірки
деталей.

<Note>
Якщо пристрій повторює спробу сполучення зі зміненими даними автентифікації (роль, області дії або відкритий ключ), OpenClaw замінює попередній запис, що очікує на розгляд, і видає новий `requestId`. Запускайте `openclaw devices list` безпосередньо перед схваленням, щоб використати поточний ID.
</Note>

Якщо пристрій уже сполучений і запитує ширші області дії або ширшу роль,
OpenClaw зберігає чинне схвалення та створює новий запит на підвищення,
що очікує на розгляд. Перегляньте стовпці `Requested` і `Approved` у `openclaw devices list`
або скористайтеся `openclaw devices approve --latest`, щоб попередньо переглянути точне підвищення
перед його схваленням.

Якщо Gateway явно налаштовано з
`gateway.nodes.pairing.autoApproveCidrs`, перші запити `role: node` від
клієнтських IP-адрес, що відповідають умовам, можуть бути схвалені ще до того, як вони з’являться в цьому списку. Ця політика
за замовчуванням вимкнена і ніколи не застосовується до клієнтів operator/browser або запитів на підвищення.

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

Відхилити запит на сполучення пристрою, що очікує на розгляд.

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

Ротувати токен пристрою для конкретної ролі (за потреби оновивши області дії).
Цільова роль уже має існувати в схваленому контракті сполучення цього пристрою;
ротація не може випустити нову несхвалену роль.
Якщо ви не вкажете `--scope`, пізніші повторні підключення зі збереженим ротованим токеном повторно використовуватимуть
закешовані схвалені області дії цього токена. Якщо ви передасте явні значення `--scope`, вони
стануть збереженим набором областей дії для майбутніх повторних підключень із кешованим токеном.
Виклики від неадміністраторів зі сполученого пристрою можуть ротувати лише **власний** токен пристрою.
Набір областей дії цільового токена має залишатися в межах власних областей дії operator
сеансу виклику; ротація не може випустити або зберегти ширший токен operator, ніж той,
який уже має викликач.

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

Повертає метадані ротації у форматі JSON. Якщо викликач ротуватиме власний токен, будучи
автентифікованим за допомогою токена цього пристрою, відповідь також міститиме замінний
токен, щоб клієнт міг зберегти його перед повторним підключенням. Спільні/адміністраторські ротації
не повертають bearer-токен.

### `openclaw devices revoke --device <id> --role <role>`

Відкликати токен пристрою для конкретної ролі.

Виклики від неадміністраторів зі сполученого пристрою можуть відкликати лише **власний** токен пристрою.
Відкликання токена будь-якого іншого пристрою вимагає `operator.admin`.
Набір областей дії цільового токена також має вкладатися у власні
області дії operator сеансу виклику; виклики лише зі сполученням не можуть відкликати токени operator з правами admin/write.

```
openclaw devices revoke --device <deviceId> --role node
```

Повертає результат відкликання у форматі JSON.

## Загальні параметри

- `--url <url>`: URL WebSocket Gateway (за замовчуванням використовується `gateway.remote.url`, якщо налаштовано).
- `--token <token>`: токен Gateway (якщо потрібен).
- `--password <password>`: пароль Gateway (автентифікація паролем).
- `--timeout <ms>`: тайм-аут RPC.
- `--json`: вивід JSON (рекомендовано для скриптів).

<Warning>
Коли ви вказуєте `--url`, CLI не повертається до облікових даних із конфігурації або середовища. Явно передайте `--token` або `--password`. Відсутність явно переданих облікових даних є помилкою.
</Warning>

## Примітки

- Ротація токена повертає новий токен (чутливі дані). Поводьтеся з ним як із секретом.
- Для цих команд потрібна область дії `operator.pairing` (або `operator.admin`).
- `gateway.nodes.pairing.autoApproveCidrs` — це політика Gateway, яку потрібно явно увімкнути,
  лише для нового сполучення пристроїв node; вона не змінює повноваження схвалення в CLI.
- Ротація та відкликання токенів залишаються в межах схваленого набору ролей сполучення і
  базового рівня схвалених областей дії для цього пристрою. Випадковий запис кешованого токена не
  надає цілі для керування токеном.
- Для сеансів із токенами сполучених пристроїв керування іншими пристроями доступне лише адміністратору:
  `remove`, `rotate` і `revoke` дозволені лише для власного пристрою, якщо викликач не має
  `operator.admin`.
- Зміна токена також обмежена областями дії викликача: сеанс лише зі сполученням не може
  ротувати або відкликати токен, який наразі має `operator.admin` або
  `operator.write`.
- `devices clear` навмисно захищено параметром `--yes`.
- Якщо область дії сполучення недоступна на local loopback (і явний `--url` не передано), list/approve можуть використовувати локальний резервний механізм сполучення.
- `devices approve` вимагає явного ID запиту перед випуском токенів; якщо не вказати `requestId` або передати `--latest`, виконується лише попередній перегляд найновішого запиту, що очікує на розгляд.

## Контрольний список відновлення при розсинхронізації токенів

Використовуйте це, коли Control UI або інші клієнти постійно завершуються помилками `AUTH_TOKEN_MISMATCH` або `AUTH_DEVICE_TOKEN_MISMATCH`.

1. Підтвердьте поточне джерело токена gateway:

```bash
openclaw config get gateway.auth.token
```

2. Перелічіть сполучені пристрої та визначте ID проблемного пристрою:

```bash
openclaw devices list
```

3. Ротуйте токен operator для проблемного пристрою:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. Якщо ротації недостатньо, видаліть застаріле сполучення і схваліть знову:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. Повторіть спробу підключення клієнта з поточним спільним токеном/паролем.

Примітки:

- Звичайний пріоритет автентифікації при повторному підключенні: спочатку явний спільний токен/пароль, потім явний `deviceToken`, далі збережений токен пристрою, а потім bootstrap-токен.
- Довірене відновлення після `AUTH_TOKEN_MISMATCH` може тимчасово надіслати одночасно і спільний токен, і збережений токен пристрою для однієї обмеженої повторної спроби.
