# `openclaw configure`

Інтерактивні запити для цільових змін наявного налаштування: облікових даних, пристроїв, типових параметрів агента, Gateway, каналів, плагінів, Skills і перевірок справності.

Використовуйте `openclaw onboard` або `openclaw setup` для повного керованого процесу першого запуску, `openclaw setup --baseline` — лише для базової конфігурації та робочого простору, а `openclaw channels add` — коли потрібно лише налаштувати обліковий запис каналу.

<Tip>
Команда `openclaw config` без підкоманди відкриває той самий майстер. Для неінтерактивного редагування використовуйте `openclaw config get|set|unset`.
</Tip>

## Параметри

`--section <section>`: повторюваний фільтр розділів. Доступні розділи:

`workspace`, `model`, `web`, `gateway`, `daemon`, `channels`, `plugins`, `skills`, `health`

```bash
openclaw configure
openclaw configure --section web
openclaw configure --section model --section channels
openclaw configure --section gateway --section daemon
```

Вибір `gateway`, `daemon` або `health` (чи запуск повного майстра без `--section`) запитує, де працює Gateway, і оновлює `gateway.mode`. Фільтри розділів, які пропускають усі три, одразу переходять до запитаного налаштування без запиту режиму Gateway. Вибір віддаленого режиму Gateway записує віддалену конфігурацію та негайно завершує роботу; локальні кроки, як-от установлення плагінів, не виконуються.

<Note>
Для `openclaw configure` потрібен інтерактивний термінал (і stdin, і stdout мають бути TTY). Без нього команда виводить еквівалентні неінтерактивні команди `openclaw config get|set|patch|validate` та завершується з помилкою замість часткового виконання.
</Note>

## Розділ моделі

<Note>
**Модель** містить множинний вибір для списку дозволених моделей `agents.defaults.models` (того, що відображається в `/model` і засобі вибору моделі). Варіанти налаштування для окремих провайдерів додають вибрані моделі до наявного списку дозволених моделей замість заміни не пов’язаних із ними провайдерів, які вже є в конфігурації.

Повторна автентифікація провайдера через configure зберігає наявне значення `agents.defaults.model.primary`, навіть коли крок автентифікації провайдера повертає виправлення конфігурації з власною рекомендованою типовою моделлю. Додавання або повторна автентифікація провайдера робить його моделі доступними, не замінюючи вашу поточну основну модель. Щоб навмисно змінити типову модель, використовуйте `openclaw models auth login --provider <id> --set-default` або `openclaw models set <model>`.
</Note>

Коли configure запускається з варіанта автентифікації провайдера, засоби вибору типової моделі та списку дозволених моделей автоматично надають перевагу цьому провайдеру. Для парних провайдерів, як-от Volcengine і BytePlus, така сама перевага також охоплює їхні варіанти планів для програмування (`volcengine-plan/*`, `byteplus-plan/*`). Якщо фільтр бажаного провайдера дає порожній список, configure повертається до нефільтрованого каталогу замість відображення порожнього засобу вибору.

## Веброзділ

`openclaw configure --section web` вибирає провайдера вебпошуку та налаштовує його облікові дані. Деякі провайдери показують додаткові запити, специфічні для них:

- **Grok** може запропонувати необов’язкове налаштування `x_search` із тим самим профілем xAI OAuth або ключем API та дати змогу вибрати модель `x_search`.
- **Kimi** може запитати регіон API Moonshot (`api.moonshot.ai` або `api.moonshot.cn`) і типову модель вебпошуку Kimi.

## Інші примітки

- Після локального запису конфігурації configure встановлює вибрані завантажувані плагіни, коли цього потребує вибраний шлях налаштування. Конфігурація віддаленого Gateway не встановлює локальні пакети плагінів.
- Орієнтовані на канали служби (Slack/Discord/Matrix/Microsoft Teams) під час налаштування запитують списки дозволених каналів або кімнат. Можна вводити назви чи ідентифікатори; за можливості майстер перетворює назви на ідентифікатори.
- Якщо ви запускаєте крок установлення демона, для автентифікації за токеном потрібен токен. Якщо `gateway.auth.token` керується через SecretRef, configure перевіряє SecretRef, але не зберігає визначені значення токена у вигляді відкритого тексту в метаданих середовища служби супервізора; якщо SecretRef не визначено, configure блокує встановлення демона та надає практичні вказівки щодо усунення проблеми.
- Якщо налаштовано і `gateway.auth.token`, і `gateway.auth.password`, а `gateway.auth.mode` не задано, configure блокує встановлення демона, доки ви явно не задасте режим.
