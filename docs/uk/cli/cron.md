# `openclaw cron`

Керуйте завданнями Cron для планувальника Gateway.

<Tip>
Запустіть `openclaw cron --help`, щоб побачити повний набір команд. Дивіться [Завдання Cron](/uk/automation/cron-jobs) для концептуального посібника.
</Tip>

## Сесії

`--session` приймає `main`, `isolated`, `current` або `session:<id>`.

<AccordionGroup>
  <Accordion title="Ключі сесій">
    - `main` прив’язується до основної сесії агента.
    - `isolated` створює новий транскрипт і ідентифікатор сесії для кожного запуску.
    - `current` прив’язується до активної сесії на момент створення.
    - `session:<id>` закріплює за явним ключем постійної сесії.
  </Accordion>
  <Accordion title="Семантика ізольованої сесії">
    Ізольовані запуски скидають фоновий контекст розмови. Маршрутизація каналу та групи, політика надсилання/черги, підвищення привілеїв, походження та прив’язка середовища виконання ACP скидаються для нового запуску. Безпечні налаштування та явні перевизначення моделі або автентифікації, вибрані користувачем, можуть переноситися між запусками.
  </Accordion>
</AccordionGroup>

## Доставка

`openclaw cron list` і `openclaw cron show <job-id>` показують попередній перегляд визначеного маршруту доставки. Для `channel: "last"` у перегляді буде показано, чи маршрут визначено з основної або поточної сесії, чи він завершиться безпечною відмовою.

<Note>
Ізольовані завдання `cron add` за замовчуванням використовують доставку `--announce`. Щоб зберегти вивід внутрішнім, використовуйте `--no-deliver`. `--deliver` залишається застарілим псевдонімом для `--announce`.
</Note>

### Володіння доставкою

У ізольованих чат-доставках Cron відповідальність за доставку спільна між агентом і виконавцем:

- Агент може надсилати напряму за допомогою інструмента `message`, коли маршрут чату доступний.
- `announce` резервно доставляє лише фінальну відповідь, коли агент не надіслав її безпосередньо до визначеної цілі.
- `webhook` надсилає завершене корисне навантаження на URL.
- `none` вимикає резервну доставку виконавця.

`--announce` — це резервна доставка фінальної відповіді виконавцем. `--no-deliver` вимикає цю резервну доставку, але не прибирає в агента інструмент `message`, коли маршрут чату доступний.

Нагадування, створені з активного чату, зберігають ціль доставки поточного чату для резервної доставки announce. Внутрішні ключі сесій можуть бути в нижньому регістрі; не використовуйте їх як джерело істини для чутливих до регістру ідентифікаторів провайдерів, таких як ідентифікатори кімнат Matrix.

### Доставка помилок

Сповіщення про помилки визначаються в такому порядку:

1. `delivery.failureDestination` у завданні.
2. Глобальний `cron.failureDestination`.
3. Основна ціль announce для завдання (коли явну ціль для помилок не задано).

<Note>
Завдання основної сесії можуть використовувати `delivery.failureDestination` лише тоді, коли основний режим доставки — `webhook`. Ізольовані завдання приймають його в усіх режимах.
</Note>

Примітка: ізольовані запуски cron трактують збої агента на рівні запуску як помилки завдання, навіть коли корисне навантаження відповіді не створено, тому збої моделі/провайдера все одно збільшують лічильники помилок і запускають сповіщення про збої.

## Планування

### Одноразові завдання

`--at <datetime>` планує одноразовий запуск. Дати й час без зсуву трактуються як UTC, якщо ви також не передасте `--tz <iana>`, який інтерпретує час настінного годинника в заданому часовому поясі.

<Note>
Одноразові завдання після успішного виконання за замовчуванням видаляються. Використовуйте `--keep-after-run`, щоб зберегти їх.
</Note>

### Періодичні завдання

Після послідовних помилок періодичні завдання використовують експоненційне збільшення затримки повторних спроб: 30 с, 1 хв, 5 хв, 15 хв, 60 хв. Після наступного успішного запуску розклад повертається до звичайного режиму.

Пропущені запуски відстежуються окремо від помилок виконання. Вони не впливають на затримку повторних спроб, але `openclaw cron edit <job-id> --failure-alert-include-skipped` може додати повторювані сповіщення про пропущені запуски до сповіщень про збої.

Примітка: визначення завдань cron зберігаються в `jobs.json`, а стан виконання, що очікує, — у `jobs-state.json`. Якщо `jobs.json` редагується зовні, Gateway перезавантажує змінені розклади й очищає застарілі слоти очікування; перезаписи лише з форматуванням не очищають слот очікування.

### Ручні запуски

`openclaw cron run` повертається, щойно ручний запуск поставлено в чергу. Успішні відповіді містять `{ ok: true, enqueued: true, runId }`. Використовуйте `openclaw cron runs --id <job-id>`, щоб відстежити остаточний результат.

<Note>
`openclaw cron run <job-id>` за замовчуванням примусово запускає завдання. Використовуйте `--due`, щоб зберегти старішу поведінку «запускати лише якщо настав час».
</Note>

## Моделі

`cron add|edit --model <ref>` вибирає дозволену модель для завдання.

<Warning>
Якщо модель не дозволена, cron покаже попередження й повернеться до агента завдання або вибору моделі за замовчуванням. Налаштовані ланцюжки резервного переходу все ще застосовуються, але звичайне перевизначення моделі без явного списку резервних варіантів для завдання більше не додає основну модель агента як приховану додаткову ціль повторної спроби.
</Warning>

### Пріоритет моделі в ізольованому cron

Ізольований cron визначає активну модель у такому порядку:

1. Перевизначення Gmail-hook.
2. `--model` для окремого завдання.
3. Збережене перевизначення моделі cron-session (коли користувач його вибрав).
4. Вибір моделі агента або модель за замовчуванням.

### Швидкий режим

Швидкий режим ізольованого cron слідує за визначеним вибором моделі в реальному часі. Конфігурація моделі `params.fastMode` застосовується за замовчуванням, але збережене перевизначення `fastMode` для сесії все одно має вищий пріоритет над конфігурацією.

### Повторні спроби під час живого перемикання моделі

Якщо ізольований запуск породжує `LiveSessionModelSwitchError`, cron перед повторною спробою зберігає перемкнені провайдер і модель (а також перевизначення профілю автентифікації після перемикання, якщо воно є) для активного запуску. Зовнішній цикл повторних спроб обмежений двома повторними спробами перемикання після початкової спроби, після чого переривається замість нескінченного циклу.

## Вивід запуску та відмови

### Придушення застарілих підтверджень

Для ізольованого cron придушуються відповіді, що містять лише застаріле підтвердження. Якщо перший результат — лише проміжне оновлення стану і жоден дочірній запуск субагента не відповідає за остаточну відповідь, cron один раз повторно надсилає запит для отримання реального результату перед доставкою.

### Придушення тихих токенів

Якщо ізольований запуск cron повертає лише тихий токен (`NO_REPLY` або `no_reply`), cron придушує як пряме вихідне надсилання, так і резервний шлях підсумку через чергу, тож назад у чат нічого не публікується.

### Структуровані відмови

Ізольовані запуски cron надають перевагу структурованим метаданим відмови у виконанні з вбудованого запуску, а потім повертаються до відомих маркерів відмови у фінальному виводі, таких як `SYSTEM_RUN_DENIED`, `INVALID_REQUEST` і фрази відмови, пов’язані з прив’язкою до схвалення.

`cron list` та історія запусків показують причину відмови замість того, щоб позначати заблоковану команду як `ok`.

## Зберігання

Зберігання та очищення налаштовуються в конфігурації:

- `cron.sessionRetention` (типово `24h`) очищає завершені сесії ізольованих запусків.
- `cron.runLog.maxBytes` і `cron.runLog.keepLines` очищають `~/.openclaw/cron/runs/<jobId>.jsonl`.

## Міграція старіших завдань

<Note>
Якщо у вас є завдання cron, створені до поточного формату доставки та зберігання, запустіть `openclaw doctor --fix`. Doctor нормалізує застарілі поля cron (`jobId`, `schedule.cron`, поля доставки верхнього рівня, включно із застарілим `threadId`, псевдоніми доставки `provider` у корисному навантаженні) і мігрує прості резервні webhook-завдання `notify: true` до явної доставки webhook, коли налаштовано `cron.webhook`.
</Note>

## Поширені зміни

Оновіть налаштування доставки, не змінюючи повідомлення:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

Вимкніть доставку для ізольованого завдання:

```bash
openclaw cron edit <job-id> --no-deliver
```

Увімкніть полегшений контекст початкового завантаження для ізольованого завдання:

```bash
openclaw cron edit <job-id> --light-context
```

Оголосіть у конкретний канал:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

Створіть ізольоване завдання з полегшеним контекстом початкового завантаження:

```bash
openclaw cron add \
  --name "Короткий ранковий звіт" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Підсумуй нічні оновлення." \
  --light-context \
  --no-deliver
```

`--light-context` застосовується лише до ізольованих завдань типу agent-turn. Для запусків cron полегшений режим залишає контекст початкового завантаження порожнім замість ін’єкції повного набору bootstrap-контексту робочого простору.

## Поширені команди адміністратора

Ручний запуск і перевірка:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

Записи `cron runs` містять діагностику доставки з наміченою ціллю cron, визначеною ціллю, надсиланнями через інструмент message, використанням резервного варіанта та станом доставки.

Перенацілення агента й сесії:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

Налаштування доставки:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
