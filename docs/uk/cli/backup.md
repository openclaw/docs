# `openclaw backup`

Створити локальний архів резервної копії для стану OpenClaw, конфігурації, auth profiles, облікових даних каналів/provider-ів, сесій і, за потреби, робочих просторів.

```bash
openclaw backup create
openclaw backup create --output ~/Backups
openclaw backup create --dry-run --json
openclaw backup create --verify
openclaw backup create --no-include-workspace
openclaw backup create --only-config
openclaw backup verify ./2026-03-09T00-00-00.000Z-openclaw-backup.tar.gz
```

## Примітки

- Архів містить файл `manifest.json` із розв’язаними вихідними шляхами та схемою архіву.
- Типовий результат — архів `.tar.gz` з часовою позначкою в поточному робочому каталозі.
- Якщо поточний робочий каталог знаходиться всередині дерева вихідного стану/робочого простору, що резервується, OpenClaw використовує домашній каталог як типове розташування архіву.
- Наявні файли архівів ніколи не перезаписуються.
- Вихідні шляхи всередині вихідних дерев стану/робочого простору відхиляються, щоб уникнути самовключення.
- `openclaw backup verify <archive>` перевіряє, що архів містить рівно один кореневий `manifest`, відхиляє шляхи архіву зі стилем traversal і перевіряє, що кожен payload, оголошений у маніфесті, існує в tarball.
- `openclaw backup create --verify` запускає цю перевірку одразу після запису архіву.
- `openclaw backup create --only-config` резервує лише активний файл конфігурації JSON.

## Що входить до резервної копії

`openclaw backup create` планує джерела резервної копії з вашого локального встановлення OpenClaw:

- Каталог стану, який повертає локальний резолвер стану OpenClaw, зазвичай `~/.openclaw`
- Шлях до активного файлу конфігурації
- Розв’язаний каталог `credentials/`, якщо він існує поза каталогом стану
- Каталоги робочих просторів, виявлені з поточної конфігурації, якщо не передано `--no-include-workspace`

Модельні auth profiles уже є частиною каталогу стану в
`agents/<agentId>/agent/auth-profiles.json`, тому зазвичай вони покриваються
елементом резервування стану.

Якщо ви використовуєте `--only-config`, OpenClaw пропускає виявлення стану, каталогу облікових даних і робочих просторів та архівує лише шлях до активного файлу конфігурації.

OpenClaw канонікалізує шляхи перед побудовою архіву. Якщо конфігурація,
каталог облікових даних або робочий простір уже знаходяться всередині каталогу стану,
вони не дублюються як окремі джерела резервної копії верхнього рівня. Відсутні шляхи
пропускаються.

Payload архіву зберігає вміст файлів із цих вихідних дерев, а вбудований `manifest.json` фіксує розв’язані абсолютні вихідні шляхи та схему архіву, використану для кожного ресурсу.

## Поведінка при невалідній конфігурації

`openclaw backup` навмисно обходить звичайну попередню перевірку конфігурації, щоб усе одно допомогти під час відновлення. Оскільки виявлення робочих просторів залежить від валідної конфігурації, `openclaw backup create` тепер одразу завершується з помилкою, якщо файл конфігурації існує, але є невалідним, а резервування робочих просторів іще ввімкнене.

Якщо в такій ситуації вам усе ж потрібна часткова резервна копія, запустіть повторно:

```bash
openclaw backup create --no-include-workspace
```

Це залишає в області охоплення стан, конфігурацію та зовнішній каталог облікових даних,
повністю пропускаючи виявлення робочих просторів.

Якщо вам потрібна лише копія самого файлу конфігурації, `--only-config` також працює, коли конфігурація пошкоджена, оскільки не покладається на її парсинг для виявлення робочих просторів.

## Розмір і продуктивність

OpenClaw не накладає вбудованого максимального розміру резервної копії чи обмеження на розмір окремого файлу.

Практичні обмеження визначаються локальною машиною та цільовою файловою системою:

- Доступний простір для тимчасового запису архіву плюс фінального архіву
- Час на обхід великих дерев робочих просторів і їх стиснення в `.tar.gz`
- Час на повторне сканування архіву, якщо ви використовуєте `openclaw backup create --verify` або запускаєте `openclaw backup verify`
- Поведінка файлової системи за цільовим шляхом. OpenClaw надає перевагу кроку публікації через hard link без перезапису й переходить до ексклюзивного копіювання, якщо hard links не підтримуються

Великі робочі простори зазвичай найбільше впливають на розмір архіву. Якщо вам потрібна менша або швидша резервна копія, використовуйте `--no-include-workspace`.

Для найменшого архіву використовуйте `--only-config`.
