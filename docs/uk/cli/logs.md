# `openclaw logs`

Виводить хвіст файлових журналів Gateway через RPC (працює у віддаленому режимі).

Пов’язано:

- Огляд журналювання: [Журналювання](/uk/logging)
- CLI Gateway: [gateway](/uk/cli/gateway)

## Параметри

- `--limit <n>`: максимальна кількість рядків журналу для повернення (за замовчуванням `200`)
- `--max-bytes <n>`: максимальна кількість байтів для читання з файлу журналу (за замовчуванням `250000`)
- `--follow`: стежити за потоком журналу
- `--interval <ms>`: інтервал опитування під час стеження (за замовчуванням `1000`)
- `--json`: виводити JSON-події, розділені рядками
- `--plain`: виведення простим текстом без стилізованого форматування
- `--no-color`: вимкнути ANSI-кольори
- `--local-time`: відображати часові позначки у вашому локальному часовому поясі (за замовчуванням)
- `--utc`: відображати часові позначки в UTC

## Спільні параметри RPC Gateway

`openclaw logs` також приймає стандартні прапорці клієнта Gateway:

- `--url <url>`: URL WebSocket Gateway
- `--token <token>`: токен Gateway
- `--timeout <ms>`: тайм-аут у мс (за замовчуванням `30000`)
- `--expect-final`: чекати на фінальну відповідь, коли виклик Gateway підтримується агентом

Коли ви передаєте `--url`, CLI не застосовує автоматично конфігураційні або середовищні облікові дані. Додайте `--token` явно, якщо цільовий Gateway вимагає автентифікації.

## Приклади

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --limit 500
openclaw logs --local-time
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## Примітки

- Часові позначки за замовчуванням відображаються у вашому локальному часовому поясі. Використовуйте `--utc` для виведення в UTC.
- Якщо неявний local loopback Gateway запитує спарювання, закривається під час підключення або перевищує час очікування до відповіді `logs.tail`, `openclaw logs` автоматично повертається до налаштованого файлового журналу Gateway. Явні цілі `--url` не використовують цей резервний варіант.
- `openclaw logs --follow` не стежить за резервними налаштованими файлами після збоїв неявного локального RPC Gateway. У Linux він використовує активний журнал Gateway user-systemd за PID, коли він доступний, і друкує вибране джерело журналу; інакше продовжує повторювати спроби з live Gateway замість читання хвоста потенційно застарілого сусіднього файлу.
- Під час використання `--follow` тимчасові розриви з’єднання з gateway (закриття WebSocket, тайм-аут, обрив з’єднання) запускають автоматичне повторне підключення з експоненційною затримкою (до 8 повторних спроб, обмежено 30 с між спробами). Під час кожної повторної спроби в stderr друкується попередження, а після успішного опитування друкується сповіщення `[logs] gateway reconnected`. У режимі `--json` і попередження про повторну спробу, і перехід повторного підключення виводяться в stderr як записи `{"type":"notice"}`. Невідновлювані помилки (збій автентифікації, неправильна конфігурація) усе ще завершують виконання негайно.
- У режимі `--follow --json` переходи джерела журналу виводяться як записи `{"type":"meta"}`. Споживачі мають відстежувати курсори для кожного `sourceKind`: потік може перейти від файлового виведення Gateway (`sourceKind: "file"`) до резервного локального журналу (`sourceKind: "journal"`, `localFallback: true`, з `service.pid`/`service.unit`) і назад до файлового виведення Gateway після відновлення. Не припускайте одне стабільне джерело або курсор для всього сеансу стеження, і допускайте перекриття рядків, коли відновлення повторно відтворює курсор файлу Gateway.
