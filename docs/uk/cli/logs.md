# `openclaw logs`

Виводить останні рядки файлових журналів Gateway через RPC. Працює у віддаленому режимі.

## Параметри

- `--limit <n>`: максимальна кількість рядків журналу у відповіді (за замовчуванням `200`)
- `--max-bytes <n>`: максимальна кількість байтів для читання з файлу журналу (за замовчуванням `250000`)
- `--follow`: стежити за потоком журналу
- `--interval <ms>`: інтервал опитування під час стеження (за замовчуванням `1000`)
- `--json`: виводити події JSON, розділені рядками
- `--plain`: виводити звичайний текст без стилізованого форматування
- `--no-color`: вимкнути кольори ANSI
- `--local-time`: відображати часові позначки у вашому локальному часовому поясі (за замовчуванням)
- `--utc`: відображати часові позначки в UTC

## Спільні параметри RPC Gateway

- `--url <url>`: URL WebSocket для Gateway
- `--token <token>`: токен Gateway
- `--timeout <ms>`: час очікування в мілісекундах (за замовчуванням `30000`)
- `--expect-final`: чекати на остаточну відповідь, якщо виклик Gateway обслуговує агент

Передавання `--url` пропускає автоматичне застосування облікових даних із конфігурації; явно вкажіть `--token`, якщо цільовий Gateway вимагає автентифікації.

## Приклади

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## Резервна поведінка та відновлення

- Якщо неявний Gateway через local loopback запитує сполучення, закривається під час підключення або час очікування спливає до відповіді `logs.tail`, `openclaw logs` автоматично переходить до налаштованого файлового журналу Gateway. Для явно заданих цілей `--url` цей резервний механізм ніколи не використовується.
- Після невдалого неявного локального виклику RPC Gateway параметр `--follow` не переходить до цього налаштованого файлу — застарілий паралельний файл може створити хибне уявлення про поточний потік. Натомість у Linux команда використовує активний журнал Gateway користувацької служби systemd за PID, якщо він доступний, і виводить вибране джерело; інакше вона продовжує повторювати спроби підключення до активного Gateway.
- Під час `--follow` тимчасові розриви з’єднання (закриття WebSocket, спливання часу очікування, втрата з’єднання) запускають автоматичне повторне підключення з експоненційною затримкою: до 8 спроб з інтервалом не більш ніж 30 с. Під час кожної повторної спроби до stderr виводиться попередження, а після успішного опитування один раз виводиться сповіщення `[logs] gateway reconnected`. У режимі `--json` обидва повідомлення виводяться до stderr як записи `{"type":"notice"}`. У разі помилок, які неможливо виправити (помилка автентифікації, неправильна конфігурація), команда однаково негайно завершує роботу.
- У режимі `--follow --json` переходи між джерелами журналу виводяться як записи `{"type":"meta"}`. Відстежуйте курсори окремо для кожного `sourceKind`: потік може перейти від виведення з файлу Gateway (`sourceKind: "file"`) до резервного локального журналу (`sourceKind: "journal"`, `localFallback: true`, із `service.pid`/`service.unit`), а після відновлення — назад до виведення з файлу Gateway. Не припускайте, що протягом усього сеансу джерело або курсор залишатимуться незмінними, і допускайте повторення рядків, коли під час відновлення курсор файлу Gateway відтворюється повторно.
