Діагностика для каналів + сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

Примітки:

- `--deep` запускає живі перевірки (WhatsApp Web + Telegram + Discord + Slack + Signal).
- Звичайний `openclaw status` працює швидким шляхом лише для читання й позначає пам’ять як `not checked`, а не недоступну, коли пропускає перевірку пам’яті. Важкий аудит безпеки, сумісність plugin і перевірки векторів пам’яті залишено для `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` і `openclaw memory status --deep`.
- `status --json --all` повідомляє подробиці пам’яті з активного середовища виконання memory plugin, вибраного через `plugins.slots.memory`. Користувацькі memory plugins можуть залишати вбудований `agents.defaults.memorySearch.enabled` вимкненим і все одно повідомляти власний стан файлів, фрагментів, векторів і FTS.
- `--usage` виводить нормалізовані вікна використання провайдера у форматі `X% left`.
- Вивід стану сеансу розділяє `Execution:` і `Runtime:`. `Execution` — це шлях пісочниці (`direct`, `docker/*`), а `Runtime` показує, чи сеанс використовує `OpenClaw Pi Default`, `OpenAI Codex`, CLI-бекенд або ACP-бекенд, як-от `codex (acp/acpx)`. Див. [середовища виконання агентів](/uk/concepts/agent-runtimes), щоб розрізняти провайдера, модель і середовище виконання.
- Необроблені поля MiniMax `usage_percent` / `usagePercent` означають залишок квоти, тому OpenClaw інвертує їх перед показом; поля на основі лічильників мають пріоритет, коли вони присутні. Відповіді `model_remains` надають перевагу запису chat-моделі, за потреби виводять мітку вікна з часових позначок і включають назву моделі в мітку плану.
- Коли поточний знімок сеансу розріджений, `/status` може доповнити лічильники токенів і кешу з найновішого журналу використання transcript. Наявні ненульові живі значення все одно мають пріоритет над резервними значеннями transcript.
- `/status` включає компактний час роботи процесу Gateway і час роботи системи хоста.
- Резервний transcript також може відновити мітку активної моделі середовища виконання, коли її бракує в записі живого сеансу. Якщо ця модель transcript відрізняється від вибраної моделі, status визначає вікно контексту за відновленою моделлю середовища виконання, а не за вибраною.
- Для обліку розміру prompt резервний transcript надає перевагу більшій prompt-орієнтованій сумі, коли метаданих сеансу немає або вони менші, тому сеанси користувацьких провайдерів не згортаються до показу `0` токенів.
- Вивід включає сховища сеансів для кожного агента, коли налаштовано кілька агентів.
- Огляд включає стан встановлення й виконання Gateway + служби хоста node, коли доступно.
- Огляд включає канал оновлення + git SHA (для source checkouts).
- Інформація про оновлення показується в Огляді; якщо оновлення доступне, status виводить підказку запустити `openclaw update` (див. [Оновлення](/uk/install/updating)).
- Поверхні стану лише для читання (`status`, `status --json`, `status --all`) за можливості розв’язують підтримувані SecretRefs для цільових шляхів конфігурації.
- Якщо налаштовано підтримуваний SecretRef каналу, але він недоступний у поточному шляху команди, status залишається лише для читання й повідомляє погіршений вивід замість аварійного завершення. Людський вивід показує попередження на кшталт "налаштований токен недоступний у цьому шляху команди", а JSON-вивід включає `secretDiagnostics`.
- Коли command-local розв’язання SecretRef успішне, status надає перевагу розв’язаному знімку й очищає тимчасові маркери каналу "secret unavailable" з фінального виводу.
- `status --all` включає рядок огляду Secrets і розділ діагностики, який підсумовує діагностику секретів (обрізану для читабельності), не зупиняючи створення звіту.
