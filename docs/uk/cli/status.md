Діагностика каналів + сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

Примітки:

- `--deep` виконує живі перевірки (WhatsApp Web + Telegram + Discord + Slack + Signal).
- Звичайний `openclaw status` залишається на швидкому шляху лише для читання і позначає пам’ять як `not checked`, а не недоступну, коли пропускає перевірку пам’яті. Важкий аудит безпеки, сумісність плагінів і перевірки векторів пам’яті залишаються для `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` і `openclaw memory status --deep`.
- `status --json --all` повідомляє подробиці пам’яті з активного середовища виконання плагіна пам’яті, вибраного через `plugins.slots.memory`. Користувацькі плагіни пам’яті можуть залишати вбудований `agents.defaults.memorySearch.enabled` вимкненим і все одно повідомляти власний стан файлів, фрагментів, векторів і FTS.
- `--usage` виводить нормалізовані вікна використання провайдера як `X% left`.
- Вивід стану сеансу розділяє `Execution:` і `Runtime:`. `Execution` — це шлях пісочниці (`direct`, `docker/*`), тоді як `Runtime` показує, чи сеанс використовує `OpenClaw Pi Default`, `OpenAI Codex`, CLI-бекенд або ACP-бекенд, наприклад `codex (acp/acpx)`. Див. [середовища виконання агентів](/uk/concepts/agent-runtimes), щоб розрізняти провайдера, модель і середовище виконання.
- Сирі поля MiniMax `usage_percent` / `usagePercent` є залишком квоти, тому OpenClaw інвертує їх перед показом; поля на основі лічильників мають пріоритет, коли вони наявні. Відповіді `model_remains` надають перевагу запису чат-моделі, за потреби виводять мітку вікна з часових позначок і включають назву моделі до мітки плану.
- Коли поточний знімок сеансу розріджений, `/status` може дозаповнити лічильники токенів і кешу з найновішого журналу використання транскрипту. Наявні ненульові живі значення все одно мають пріоритет над резервними значеннями з транскрипту.
- `/status` включає стислий час роботи процесу Gateway і час роботи системи хоста.
- Резервне відновлення з транскрипту також може відновити мітку активної моделі середовища виконання, коли її немає в записі живого сеансу. Якщо ця модель транскрипту відрізняється від вибраної моделі, status визначає вікно контексту за відновленою моделлю середовища виконання, а не за вибраною.
- Для обліку розміру підказки резервне відновлення з транскрипту надає перевагу більшій загальній сумі, орієнтованій на підказку, коли метадані сеансу відсутні або менші, тож сеанси користувацьких провайдерів не згортаються до показу `0` токенів.
- Вивід включає сховища сеансів для кожного агента, коли налаштовано кілька агентів.
- Огляд включає стан встановлення та виконання служби Gateway + вузла хоста, коли це доступно.
- Огляд включає канал оновлень + git SHA (для вихідних checkout).
- Інформація про оновлення відображається в Огляді; якщо доступне оновлення, status виводить підказку виконати `openclaw update` (див. [Оновлення](/uk/install/updating)).
- Збої оновлення цін моделей показуються як необов’язкові попередження щодо цін. Вони не
  означають, що Gateway або канали несправні.
- Поверхні стану лише для читання (`status`, `status --json`, `status --all`) за можливості розв’язують підтримувані SecretRefs для цільових шляхів конфігурації.
- Якщо підтримуваний SecretRef каналу налаштований, але недоступний у поточному шляху команди, status залишається лише для читання і повідомляє погіршений вивід замість аварійного завершення. Людський вивід показує попередження, як-от «налаштований токен недоступний у цьому шляху команди», а JSON-вивід включає `secretDiagnostics`.
- Коли розв’язання SecretRef у межах команди успішне, status надає перевагу розв’язаному знімку й очищає тимчасові маркери каналу «секрет недоступний» з фінального виводу.
- `status --all` включає рядок огляду секретів і розділ діагностики, який підсумовує діагностику секретів (обрізану для читабельності), не зупиняючи створення звіту.
