# `openclaw status`

Діагностика для каналів і сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

Примітки:

- `--deep` запускає живі зондування (WhatsApp Web + Telegram + Discord + Slack + Signal).
- Звичайний `openclaw status` лишається на швидкому шляху лише для читання й позначає пам’ять як `not checked` замість unavailable, коли пропускає перевірку пам’яті. Важку перевірку безпеки, сумісності Plugin і зондування memory-vector залишено для `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` і `openclaw memory status --deep`.
- `status --json --all` повідомляє деталі пам’яті з активного середовища виконання Plugin пам’яті, вибраного через `plugins.slots.memory`. Власні Plugin пам’яті можуть залишати вбудований `agents.defaults.memorySearch.enabled` вимкненим і все одно повідомляти власний стан файлів, фрагментів, vector і FTS.
- `--usage` виводить нормалізовані вікна використання провайдера як `X% left`.
- Вивід стану сеансу розділяє `Execution:` і `Runtime:`. `Execution` — це шлях sandbox (`direct`, `docker/*`), тоді як `Runtime` повідомляє, чи використовує сеанс `OpenClaw Pi Default`, `OpenAI Codex`, бекенд CLI або бекенд ACP, як-от `codex (acp/acpx)`. Перегляньте [Середовища виконання агентів](/uk/concepts/agent-runtimes), щоб зрозуміти різницю між провайдером, моделлю та середовищем виконання.
- Сирі поля MiniMax `usage_percent` / `usagePercent` означають залишкову квоту, тому OpenClaw інвертує їх перед показом; поля на основі лічильників мають пріоритет, коли вони присутні. Відповіді `model_remains` надають перевагу запису моделі чату, за потреби виводять мітку вікна з часових позначок і включають назву моделі до мітки плану.
- Коли поточний знімок сеансу неповний, `/status` може дозаповнювати лічильники токенів і кешу з найновішого журналу використання транскрипту. Наявні ненульові живі значення все одно мають пріоритет над резервними значеннями з транскрипту.
- Резервне заповнення з транскрипту також може відновити мітку активної runtime-моделі, коли її немає в записі живого сеансу. Якщо ця модель із транскрипту відрізняється від вибраної моделі, статус визначає контекстне вікно за відновленою runtime-моделлю, а не за вибраною.
- Для обліку розміру підказки резервне заповнення з транскрипту надає перевагу більшому загальному значенню, орієнтованому на підказку, коли метадані сеансу відсутні або менші, щоб сеанси з власними провайдерами не зводилися до показу `0` токенів.
- Вивід включає сховища сеансів для кожного агента, коли налаштовано кілька агентів.
- Огляд включає стан встановлення/виконання сервісу Gateway + хоста node, коли доступно.
- Огляд включає канал оновлень + git SHA (для вихідних checkout).
- Інформація про оновлення відображається в розділі Overview; якщо оновлення доступне, status виводить підказку запустити `openclaw update` (див. [Оновлення](/uk/install/updating)).
- Поверхні стану лише для читання (`status`, `status --json`, `status --all`) за можливості визначають підтримувані SecretRef для цільових шляхів конфігурації.
- Якщо підтримуваний SecretRef каналу налаштований, але недоступний у поточному шляху команди, status лишається лише для читання й повідомляє про деградований вивід замість аварійного завершення. У виводі для людей показуються попередження на кшталт “configured token unavailable in this command path”, а JSON-вивід містить `secretDiagnostics`.
- Коли локальне для команди визначення SecretRef успішне, status надає перевагу визначеному знімку й прибирає тимчасові позначки каналу “secret unavailable” з фінального виводу.
- `status --all` включає рядок огляду Secrets і розділ діагностики, який підсумовує діагностику секретів (усічену для зручності читання), не зупиняючи генерацію звіту.
