Діагностика каналів і сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

| Прапорець               | Опис                                                                                                                           |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `--all`                 | Повна діагностика (лише для читання, придатна для вставлення). Включає аудит безпеки, сумісність плагінів і перевірки векторної пам’яті. |
| `--deep`                | Виконує перевірки в реальному часі (WhatsApp Web + Telegram + Discord + Slack + Signal). Також вмикає аудит безпеки.             |
| `--usage`               | Виводить нормалізовані вікна використання постачальника у форматі `X% left`.                                                     |
| `--json`                | Машиночитаний вивід.                                                                                                            |
| `--verbose` / `--debug` | Також виводить необроблений результат визначення цілі Gateway перед звітом.                                                      |

Звичайна команда `openclaw status` використовує швидкий шлях лише для читання й позначає пам’ять як
`not checked`, а не як недоступну, коли пропускає її перевірку. Ресурсомісткі
перевірки аудиту безпеки, сумісності плагінів і векторної пам’яті виконують
`openclaw status --all`, `openclaw status --deep`, `openclaw security audit`
і `openclaw memory status --deep`.

## Визначення сеансу та моделі

- Вивід стану сеансу розділяє `Execution:` і `Runtime:`. `Execution`
  — це шлях пісочниці (`direct`, `docker/*`), а `Runtime` вказує,
  чи використовує сеанс `OpenClaw Default`, `OpenAI Codex`, серверну частину
  CLI або серверну частину ACP, наприклад `codex (acp/acpx)`. Відмінності між
  постачальником, моделлю та середовищем виконання описано в розділі
  [Середовища виконання агента](/uk/concepts/agent-runtimes).
- Якщо поточний знімок сеансу містить мало даних, `/status` може доповнити лічильники
  токенів і кешу з найновішого журналу використання стенограми. Наявні
  ненульові актуальні значення все одно мають перевагу над резервними значеннями зі стенограми.
- Резервні дані зі стенограми також можуть відновити мітку активної моделі середовища виконання, якщо
  її немає в актуальному записі сеансу. Якщо ця модель зі стенограми відрізняється
  від вибраної моделі, стан визначає контекстне вікно за
  відновленою моделлю середовища виконання, а не за вибраною.
- Для обліку розміру запиту резервні дані зі стенограми віддають перевагу більшому
  підсумковому значенню, орієнтованому на запит, якщо метадані сеансу відсутні або містять менше значення, щоб
  сеанси з власними постачальниками не відображали `0` токенів.
- Якщо сеанс закріплено за моделлю, що відрізняється від налаштованої
  основної моделі, стан виводить обидва значення, причину (`session override`) і
  підказку `/model default`. Налаштована основна модель застосовується до нових або
  незакріплених сеансів; наявні закріплені сеанси зберігають вибір моделі для сеансу,
  доки його не буде скинуто.
- Якщо налаштовано кілька агентів, вивід містить сховища сеансів для кожного агента.

## Використання та квота

- `--usage` виводить нормалізовані вікна використання постачальника у форматі `X% left`.
- Необроблені поля MiniMax `usage_percent` / `usagePercent` містять залишок квоти,
  тому OpenClaw інвертує їх перед відображенням; поля на основі кількості мають перевагу, якщо
  вони наявні. Для відповідей `model_remains` перевага надається запису моделі чату, мітка
  вікна за потреби визначається з позначок часу, а назва моделі додається
  до мітки плану.
- Помилки оновлення цін моделей відображаються як необов’язкові попередження про ціни.
  Вони не означають, що Gateway або канали несправні.

## Огляд і стан оновлення

- Огляд містить стан встановлення й виконання служби хоста Gateway + Node, якщо
  він доступний, а також стислий час роботи процесу Gateway і системи хоста.
- Огляд містить канал оновлення та SHA git (для робочих копій вихідного коду).
- Відомості про оновлення відображаються в огляді; якщо оновлення доступне, стан
  виводить підказку виконати `openclaw update` (див. [Оновлення](/uk/install/updating)).

## Секрети

- Поверхні стану лише для читання (`status`, `status --json`, `status --all`)
  за можливості визначають підтримувані SecretRef для цільових шляхів конфігурації.
- Якщо для підтримуваного каналу налаштовано SecretRef, але він недоступний у
  поточному шляху команди, стан залишається лише для читання та повідомляє про неповний вивід
  замість аварійного завершення. У зрозумілому для людини виводі відображаються попередження на кшталт «налаштований токен
  недоступний у цьому шляху команди», а JSON-вивід містить
  `secretDiagnostics`.
- Якщо локальне для команди визначення SecretRef успішне, стан віддає перевагу
  визначеному знімку й вилучає тимчасові позначки каналу «секрет недоступний»
  з остаточного виводу.
- `status --all` містить рядок огляду секретів і розділ діагностики,
  який узагальнює діагностику секретів (зі скороченням для зручності читання), не
  зупиняючи створення звіту.

## Пам’ять

`status --json --all` повідомляє відомості про пам’ять з активного середовища виконання плагіна пам’яті,
вибраного через `plugins.slots.memory`. Власні плагіни пам’яті можуть залишати
вбудований параметр `agents.defaults.memorySearch.enabled` вимкненим і все одно повідомляти
про власні файли, фрагменти, вектори та стан FTS.
