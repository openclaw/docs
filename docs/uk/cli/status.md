Діагностика каналів + сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

Примітки:

- `--deep` запускає live-перевірки (WhatsApp Web + Telegram + Discord + Slack + Signal).
- Звичайний `openclaw status` залишається на швидкому шляху лише для читання й позначає пам’ять як `not checked`, а не як недоступну, коли пропускає інспекцію пам’яті. Важкий аудит безпеки, сумісність плагінів і перевірки векторів пам’яті залишаються для `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` і `openclaw memory status --deep`.
- `status --json --all` повідомляє деталі пам’яті з активного середовища виконання плагіна пам’яті, вибраного через `plugins.slots.memory`. Користувацькі плагіни пам’яті можуть залишати вбудований `agents.defaults.memorySearch.enabled` вимкненим і все одно повідомляти власний стан файлів, фрагментів, векторів і FTS.
- `--usage` друкує нормалізовані вікна використання провайдера як `X% left`.
- Вивід стану сеансу відокремлює `Execution:` від `Runtime:`. `Execution` — це шлях sandbox (`direct`, `docker/*`), а `Runtime` показує, чи сеанс використовує `OpenClaw Default`, `OpenAI Codex`, бекенд CLI або бекенд ACP, як-от `codex (acp/acpx)`. Див. [Середовища виконання агентів](/uk/concepts/agent-runtimes), щоб зрозуміти відмінність між провайдером, моделлю і середовищем виконання.
- Сирі поля MiniMax `usage_percent` / `usagePercent` означають залишок квоти, тому OpenClaw інвертує їх перед відображенням; поля на основі лічильників мають пріоритет, коли вони наявні. Відповіді `model_remains` надають перевагу запису чат-моделі, за потреби виводять мітку вікна з часових позначок і додають назву моделі до мітки плану.
- Коли поточний знімок сеансу розріджений, `/status` може дозаповнити лічильники токенів і кешу з найсвіжішого журналу використання транскрипту. Наявні ненульові live-значення все одно мають пріоритет над fallback-значеннями з транскрипту.
- `/status` включає стислий час безвідмовної роботи процесу Gateway і час безвідмовної роботи системи хоста.
- Fallback із транскрипту також може відновити мітку моделі активного середовища виконання, коли вона відсутня в live-записі сеансу. Якщо ця модель із транскрипту відрізняється від вибраної моделі, status визначає контекстне вікно за відновленою моделлю середовища виконання, а не за вибраною.
- Коли сеанс закріплено за моделлю, що відрізняється від налаштованої основної, status друкує обидва значення, причину (`session override`) і чітку підказку (`/model default`). Налаштована основна модель застосовується до нових або незакріплених сеансів; наявні закріплені сеанси зберігають свій вибір сеансу, доки його не буде очищено.
- Для обліку розміру prompt fallback із транскрипту віддає перевагу більшій prompt-орієнтованій сумі, коли метадані сеансу відсутні або менші, тому сеанси користувацьких провайдерів не згортаються до відображення `0` токенів.
- Вивід включає сховища сеансів для кожного агента, коли налаштовано кількох агентів.
- Огляд включає стан інсталяції та виконання Gateway + служби хоста node, коли він доступний.
- Огляд включає канал оновлень + git SHA (для вихідних checkout).
- Інформація про оновлення з’являється в Огляді; якщо оновлення доступне, status друкує підказку запустити `openclaw update` (див. [Оновлення](/uk/install/updating)).
- Помилки оновлення цін моделей показуються як необов’язкові попередження про ціни. Вони не
  означають, що Gateway або канали несправні.
- Поверхні стану лише для читання (`status`, `status --json`, `status --all`) за можливості розв’язують підтримувані SecretRefs для своїх цільових шляхів конфігурації.
- Якщо підтримуваний SecretRef каналу налаштовано, але він недоступний у поточному шляху команди, status залишається лише для читання й повідомляє погіршений вивід замість аварійного завершення. Людинозчитуваний вивід показує попередження на кшталт «налаштований токен недоступний у цьому шляху команди», а JSON-вивід включає `secretDiagnostics`.
- Коли локальне для команди розв’язання SecretRef успішне, status надає перевагу розв’язаному знімку й очищає тимчасові маркери каналу «секрет недоступний» з фінального виводу.
- `status --all` включає рядок огляду Secrets і розділ діагностики, який підсумовує діагностику секретів (обрізану для читабельності), не зупиняючи створення звіту.
