# `openclaw status`

Діагностика каналів + сеансів.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

Примітки:

- `--deep` запускає живі перевірки (WhatsApp Web + Telegram + Discord + Slack + Signal).
- Звичайний `openclaw status` залишається на швидкому шляху лише для читання й позначає пам’ять як `not checked`, а не як недоступну, коли пропускає інспекцію пам’яті. Важкий аудит безпеки, сумісність Plugin і перевірки векторів пам’яті залишаються для `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` і `openclaw memory status --deep`.
- `status --json --all` повідомляє деталі пам’яті з runtime активного Plugin пам’яті, вибраного через `plugins.slots.memory`. Користувацькі Plugin пам’яті можуть залишати вбудований `agents.defaults.memorySearch.enabled` вимкненим і все одно повідомляти власний стан файлів, фрагментів, вектора та FTS.
- `--usage` виводить нормалізовані вікна використання провайдера як `X% left`.
- Вивід стану сеансу відокремлює `Execution:` від `Runtime:`. `Execution` — це шлях sandbox (`direct`, `docker/*`), тоді як `Runtime` показує, чи сеанс використовує `OpenClaw Pi Default`, `OpenAI Codex`, бекенд CLI або бекенд ACP, як-от `codex (acp/acpx)`. Див. [Середовища виконання агентів](/uk/concepts/agent-runtimes), щоб розрізняти провайдера, модель і runtime.
- Сирі поля MiniMax `usage_percent` / `usagePercent` означають залишок квоти, тому OpenClaw інвертує їх перед відображенням; поля на основі лічильників мають пріоритет, коли вони наявні. Відповіді `model_remains` надають перевагу запису чат-моделі, за потреби виводять мітку вікна з часових позначок і включають назву моделі в мітку плану.
- Коли поточний знімок сеансу розріджений, `/status` може дозаповнити лічильники токенів і кешу з найновішого журналу використання транскрипту. Наявні ненульові живі значення все одно мають пріоритет над резервними значеннями з транскрипту.
- `/status` включає стислий час роботи процесу Gateway і час роботи системи хоста.
- Резервний механізм транскрипту також може відновити мітку активної runtime-моделі, коли її бракує в живому записі сеансу. Якщо ця модель транскрипту відрізняється від вибраної моделі, status визначає контекстне вікно за відновленою runtime-моделлю, а не за вибраною.
- Для обліку розміру промпта резервний механізм транскрипту надає перевагу більшому підсумку, орієнтованому на промпт, коли метадані сеансу відсутні або менші, щоб сеанси користувацьких провайдерів не згорталися до відображення `0` токенів.
- Вивід включає сховища сеансів для кожного агента, коли налаштовано кілька агентів.
- Огляд включає стан встановлення та runtime служби хоста Gateway + Node, коли він доступний.
- Огляд включає канал оновлень + SHA git (для вихідних checkout).
- Інформація про оновлення відображається в огляді; якщо оновлення доступне, status виводить підказку запустити `openclaw update` (див. [Оновлення](/uk/install/updating)).
- Поверхні стану лише для читання (`status`, `status --json`, `status --all`) за можливості розв’язують підтримувані SecretRefs для цільових шляхів конфігурації.
- Якщо підтримуваний SecretRef каналу налаштовано, але він недоступний у поточному шляху команди, status залишається лише для читання й повідомляє деградований вивід замість аварійного завершення. Людський вивід показує попередження на кшталт “налаштований токен недоступний у цьому шляху команди”, а JSON-вивід включає `secretDiagnostics`.
- Коли локальне для команди розв’язання SecretRef успішне, status надає перевагу розв’язаному знімку й очищає тимчасові маркери каналу “секрет недоступний” з фінального виводу.
- `status --all` включає рядок огляду Secrets і розділ діагностики, який підсумовує діагностику секретів (скорочено для читабельності), не зупиняючи створення звіту.
