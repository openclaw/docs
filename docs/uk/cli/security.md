# `openclaw security`

Інструменти безпеки (аудит + необов’язкові виправлення).

Пов’язане:

- Посібник із безпеки: [Безпека](/uk/gateway/security)

## Аудит

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --deep --password <password>
openclaw security audit --deep --token <token>
openclaw security audit --fix
openclaw security audit --json
```

Звичайна команда `security audit` залишається на холодному шляху конфігурації/файлової системи/тільки читання. За замовчуванням вона не виявляє runtime-збирачі аудиту безпеки, що належать plugins, тому регулярні аудити не завантажують runtime кожного встановленого plugin. Використовуйте `--deep`, щоб включити best-effort живі перевірки Gateway і збирачі аудиту безпеки, що належать plugins; явні внутрішні викликачі також можуть увімкнути ці збирачі, що належать plugins, коли вже мають відповідну runtime-область.

Аудит попереджає, коли кілька відправників DM спільно використовують основну сесію, і рекомендує **безпечний режим DM**: `session.dmScope="per-channel-peer"` (або `per-account-channel-peer` для каналів із кількома обліковими записами) для спільних вхідних скриньок.
Це призначено для посилення захисту кооперативних/спільних вхідних скриньок. Один Gateway, спільний для взаємно недовірених/ворожих операторів, не є рекомендованою конфігурацією; розділяйте межі довіри окремими шлюзами (або окремими користувачами ОС/хостами).
Він також видає `security.trust_model.multi_user_heuristic`, коли конфігурація вказує на ймовірний вхідний доступ для спільного користування (наприклад, відкрита політика DM/груп, налаштовані групові цілі або wildcard-правила відправників), і нагадує, що OpenClaw за замовчуванням має модель довіри персонального асистента.
Для навмисних конфігурацій зі спільними користувачами рекомендація аудиту: запускати всі сесії в пісочниці, обмежувати доступ до файлової системи межами workspace і не розміщувати персональні/приватні ідентичності або облікові дані в цьому runtime.
Він також попереджає, коли малі моделі (`<=300B`) використовуються без пісочниці та з увімкненими web/browser tools.
Для вхідного доступу через webhook він попереджає, коли `hooks.token` повторно використовує токен Gateway, коли `hooks.token` короткий, коли `hooks.path="/"`, коли `hooks.defaultSessionKey` не задано, коли `hooks.allowedAgentIds` необмежений, коли дозволені перевизначення `sessionKey` у запитах і коли перевизначення ввімкнені без `hooks.allowedSessionKeyPrefixes`.
Він також попереджає, коли параметри Docker для пісочниці налаштовані, але режим пісочниці вимкнений; коли `gateway.nodes.denyCommands` використовує неефективні шаблоноподібні/невідомі записи (лише точний збіг імен команд node, а не фільтрація shell-тексту); коли `gateway.nodes.allowCommands` явно вмикає небезпечні команди node; коли глобальний `tools.profile="minimal"` перевизначається профілями інструментів agent; коли відкриті групи відкривають runtime/інструменти файлової системи без захисних обмежень пісочниці/workspace; і коли інструменти встановлених plugins можуть бути доступними за дозвільної політики інструментів.
Він також позначає `gateway.allowRealIpFallback=true` (ризик підміни заголовків, якщо проксі налаштовані неправильно) і `discovery.mdns.mode="full"` (витік метаданих через записи mDNS TXT).
Він також попереджає, коли браузер у пісочниці використовує Docker-мережу `bridge` без `sandbox.browser.cdpSourceRange`.
Він також позначає небезпечні мережеві режими Docker для пісочниці (зокрема `host` і приєднання до просторів імен `container:*`).
Він також попереджає, коли наявні Docker-контейнери браузера в пісочниці мають відсутні/застарілі hash-мітки (наприклад, контейнери до міграції без `openclaw.browserConfigEpoch`) і рекомендує `openclaw sandbox recreate --browser --all`.
Він також попереджає, коли записи встановлення plugins/hooks на основі npm не зафіксовані за версією, не мають метаданих integrity або розходяться з поточними встановленими версіями пакетів.
Він попереджає, коли allowlist каналів покладаються на змінювані імена/електронні адреси/теги замість стабільних ID (Discord, Slack, Google Chat, Microsoft Teams, Mattermost, IRC-області, де застосовно).
Він попереджає, коли `gateway.auth.mode="none"` залишає HTTP API Gateway доступними без спільного секрету (`/tools/invoke` плюс будь-який увімкнений endpoint `/v1/*`).
Параметри з префіксом `dangerous`/`dangerously` є явними аварійними перевизначеннями оператора; увімкнення одного з них саме по собі не є звітом про вразливість безпеки.
Повний перелік небезпечних параметрів див. у розділі «Зведення небезпечних або небезпечних для безпеки прапорів» у [Безпека](/uk/gateway/security).

Поведінка SecretRef:

- `security audit` розв’язує підтримувані SecretRefs у режимі тільки читання для своїх цільових шляхів.
- Якщо SecretRef недоступний у поточному шляху команди, аудит продовжується і повідомляє `secretDiagnostics` (замість аварійного завершення).
- `--token` і `--password` лише перевизначають автентифікацію deep-probe для цього виклику команди; вони не переписують конфігурацію або зіставлення SecretRef.

## JSON-вивід

Використовуйте `--json` для перевірок CI/політик:

```bash
openclaw security audit --json | jq '.summary'
openclaw security audit --deep --json | jq '.findings[] | select(.severity=="critical") | .checkId'
```

Якщо `--fix` і `--json` поєднано, вивід містить і дії виправлення, і фінальний звіт:

```bash
openclaw security audit --fix --json | jq '{fix: .fix.ok, summary: .report.summary}'
```

## Що змінює `--fix`

`--fix` застосовує безпечні, детерміновані remediation-дії:

- перемикає поширене `groupPolicy="open"` на `groupPolicy="allowlist"` (включно з варіантами облікових записів у підтримуваних каналах)
- коли політика груп WhatsApp перемикається на `allowlist`, заповнює `groupAllowFrom` зі
  збереженого файлу `allowFrom`, якщо такий список існує, а конфігурація ще не
  визначає `allowFrom`
- змінює `logging.redactSensitive` з `"off"` на `"tools"`
- посилює дозволи для стану/конфігурації та поширених чутливих файлів
  (`credentials/*.json`, `auth-profiles.json`, `sessions.json`, сесійні
  `*.jsonl`)
- також посилює дозволи для config include-файлів, на які посилається `openclaw.json`
- використовує `chmod` на POSIX-хостах і скидання `icacls` у Windows

`--fix` **не**:

- ротирує токени/паролі/API-ключі
- вимикає інструменти (`gateway`, `cron`, `exec` тощо)
- змінює вибір bind/auth/network exposure для gateway
- видаляє або переписує plugins/skills
