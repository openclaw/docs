Підагенти — це фонові виконання агента, запущені з наявного виконання агента.
Вони працюють у власній сесії (`agent:<agentId>:subagent:<uuid>`) і,
після завершення, **повідомляють** свій результат назад у канал чату
запитувача. Кожне виконання підагента відстежується як
[фонове завдання](/uk/automation/tasks).

Основні цілі:

- Паралелізувати роботу типу «дослідження / довге завдання / повільний інструмент», не блокуючи основне виконання.
- За замовчуванням тримати підагентів ізольованими (розділення сесій + необов’язкова ізоляція sandbox).
- Зробити поверхню інструментів стійкою до неправильного використання: підагенти **не** отримують session tools за замовчуванням.
- Підтримувати налаштовувану глибину вкладеності для шаблонів оркестрації.

<Note>
**Примітка про вартість:** кожен підагент за замовчуванням має власний контекст і споживання токенів.
Для важких або повторюваних завдань задайте для підагентів дешевшу модель,
а основного агента залиште на якіснішій моделі. Налаштування виконуються через
`agents.defaults.subagents.model` або перевизначення для окремого агента. Коли дочірньому агенту
справді потрібен поточний transcript запитувача, агент може запросити
`context: "fork"` лише для цього запуску.
</Note>

## Слеш-команда

Використовуйте `/subagents`, щоб переглядати або керувати виконаннями підагентів для **поточної
сесії**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

`/subagents info` показує метадані виконання (стан, часові мітки, session id,
шлях до transcript, очищення). Використовуйте `sessions_history` для обмеженого,
відфільтрованого з міркувань безпеки представлення згадування; переглядайте шлях transcript на диску, коли
вам потрібен сирий повний transcript.

### Елементи керування прив’язкою до гілок

Ці команди працюють у каналах, які підтримують постійні прив’язки до гілок.
Див. [Канали з підтримкою гілок](#thread-supporting-channels) нижче.

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### Поведінка запуску

`/subagents spawn` запускає фонового підагента як команду користувача (а не
внутрішнє relay) і надсилає одне фінальне оновлення про завершення назад у
чат запитувача, коли виконання закінчується.

<AccordionGroup>
  <Accordion title="Неблокувальне завершення на основі push">
    - Команда spawn є неблокувальною; вона одразу повертає id виконання.
    - Після завершення підагент надсилає повідомлення зі зведенням/результатом назад у канал чату запитувача.
    - Доставка завершення працює за моделлю push. Після запуску **не** опитуйте в циклі `/subagents list`, `sessions_list` або `sessions_history` лише для того, щоб дочекатися завершення; перевіряйте стан лише за потреби для налагодження або втручання.
    - Після завершення OpenClaw у межах best effort закриває відстежувані вкладки/процеси браузера, відкриті цією сесією підагента, перш ніж продовжиться потік очищення announce.

  </Accordion>
  <Accordion title="Стійкість доставки при ручному запуску">
    - OpenClaw спочатку намагається виконати пряму доставку `agent` зі стабільним ключем ідемпотентності.
    - Якщо пряма доставка не вдається, виконується резервний перехід до маршрутизації через чергу.
    - Якщо маршрутизація через чергу все ще недоступна, announce повторюється з коротким експоненційним backoff до остаточної відмови.
    - Доставка завершення зберігає визначений маршрут запитувача: маршрути завершення, прив’язані до thread або conversation, мають пріоритет, коли доступні; якщо джерело завершення надає лише канал, OpenClaw доповнює відсутні target/account із визначеного маршруту сесії запитувача (`lastChannel` / `lastTo` / `lastAccountId`), щоб пряма доставка все одно працювала.

  </Accordion>
  <Accordion title="Метадані передачі завершення">
    Передача завершення до сесії запитувача — це внутрішній контекст,
    згенерований під час виконання (а не текст, написаний користувачем), і він містить:

    - `Result` — останній видимий текст відповіді `assistant`, або, якщо його немає, очищений останній текст `tool`/`toolResult`. Для невдалих термінальних виконань захоплений текст відповіді повторно не використовується.
    - `Status` — `completed successfully` / `failed` / `timed out` / `unknown`.
    - Компактну статистику runtime/токенів.
    - Інструкцію доставки, яка вказує агенту запитувача переписати повідомлення звичайним голосом асистента (а не пересилати сирі внутрішні метадані).

  </Accordion>
  <Accordion title="Режими та ACP runtime">
    - `--model` і `--thinking` перевизначають типові значення для конкретного виконання.
    - Використовуйте `info`/`log`, щоб переглядати деталі та вивід після завершення.
    - `/subagents spawn` — це одноразовий режим (`mode: "run"`). Для постійних сесій, прив’язаних до thread, використовуйте `sessions_spawn` із `thread: true` і `mode: "session"`.
    - Для сесій ACP harness (Claude Code, Gemini CLI, OpenCode або явно Codex ACP/acpx) використовуйте `sessions_spawn` із `runtime: "acp"`, коли інструмент рекламує цей runtime. Див. [Модель доставки ACP](/uk/tools/acp-agents#delivery-model), коли налагоджуєте завершення або цикли агент-до-агента. Коли Plugin `codex` увімкнено, для керування чатом/thread Codex слід віддавати перевагу `/codex ...` перед ACP, якщо тільки користувач явно не просить ACP/acpx.
    - OpenClaw приховує `runtime: "acp"`, доки ACP не увімкнено, запитувач не ізольований sandbox-ом і не завантажено backend Plugin на кшталт `acpx`. `runtime: "acp"` очікує зовнішній id ACP harness або запис `agents.list[]` із `runtime.type="acp"`; для звичайних агентів конфігурації OpenClaw з `agents_list` використовуйте типовий runtime підагента.

  </Accordion>
</AccordionGroup>

## Режими контексту

Нативні підагенти запускаються ізольовано, якщо викликач явно не просить fork
поточного transcript.

| Режим      | Коли його використовувати                                                                                                              | Поведінка                                                                        |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `isolated` | Нове дослідження, незалежна реалізація, повільна робота інструментів або будь-що, що можна коротко описати в тексті завдання         | Створює чистий дочірній transcript. Це типове значення й воно знижує споживання токенів. |
| `fork`     | Робота, що залежить від поточної розмови, попередніх результатів інструментів або тонких інструкцій, уже наявних у transcript запитувача | Розгалужує transcript запитувача в дочірню сесію до запуску дочірнього агента. |

Використовуйте `fork` помірковано. Це для делегування, чутливого до контексту, а не
заміна чіткого формулювання prompt завдання.

## Інструмент: `sessions_spawn`

Запускає виконання підагента з `deliver: false` у глобальній смузі `subagent`,
потім виконує крок announce і публікує відповідь announce у канал чату
запитувача.

Доступність залежить від ефективної політики інструментів викликача. Профілі `coding` і
`full` відкривають `sessions_spawn` за замовчуванням. Профіль `messaging`
не відкриває; додайте `tools.alsoAllow: ["sessions_spawn", "sessions_yield",
"subagents"]` або використовуйте `tools.profile: "coding"` для агентів, які мають делегувати
роботу. Політики дозволу/заборони на рівні каналу/групи, провайдера, sandbox і окремого агента
все одно можуть вилучити інструмент після етапу профілю. Використовуйте `/tools` у тій самій
сесії, щоб підтвердити ефективний список інструментів.

**Типові значення:**

- **Модель:** успадковується від викликача, якщо ви не задасте `agents.defaults.subagents.model` (або `agents.list[].subagents.model` для окремого агента); явне `sessions_spawn.model` все одно має пріоритет.
- **Thinking:** успадковується від викликача, якщо ви не задасте `agents.defaults.subagents.thinking` (або `agents.list[].subagents.thinking` для окремого агента); явне `sessions_spawn.thinking` все одно має пріоритет.
- **Тайм-аут виконання:** якщо `sessions_spawn.runTimeoutSeconds` не задано, OpenClaw використовує `agents.defaults.subagents.runTimeoutSeconds`, якщо воно задане; інакше використовується `0` (без тайм-ауту).

</ParamField>
<ParamField path="resumeSessionId" type="string">

<ParamField path="streamTo" type='"parent"'>

</ParamField>
<ParamField path="thinking" type="string">
  Перевизначає рівень thinking для виконання підагента.
</ParamField>
<ParamField path="runTimeoutSeconds" type="number">
  За замовчуванням дорівнює `agents.defaults.subagents.runTimeoutSeconds`, якщо задано, інакше `0`. Якщо задано, виконання підагента переривається через N секунд.
</ParamField>
<ParamField path="thread" type="boolean" default="false">
  Якщо `true`, запитує прив’язку thread каналу для цієї сесії підагента.
</ParamField>
<ParamField path="mode" type='"run" | "session"' default="run">
  Якщо `thread: true`, а `mode` не вказано, типове значення стає `session`. `mode: "session"` потребує `thread: true`.
</ParamField>
<ParamField path="cleanup" type='"delete" | "keep"' default="keep">
  `"delete"` архівує одразу після announce (при цьому transcript все одно зберігається через перейменування).
</ParamField>
<ParamField path="sandbox" type='"inherit" | "require"' default="inherit">
  `require` відхиляє запуск, якщо цільовий дочірній runtime не ізольований sandbox-ом.
</ParamField>
<ParamField path="context" type='"isolated" | "fork"' default="isolated">
  `fork` розгалужує поточний transcript запитувача в дочірню сесію. Лише для нативних підагентів. Використовуйте `fork` лише тоді, коли дочірньому агенту потрібен поточний transcript.
</ParamField>

<Warning>
`sessions_spawn` **не** приймає параметри доставки каналу (`target`,
`channel`, `to`, `threadId`, `replyTo`, `transport`). Для доставки використовуйте
`message`/`sessions_send` із запущеного виконання.
</Warning>

## Сесії, прив’язані до гілок

Коли для каналу увімкнено прив’язки до гілок, підагент може залишатися прив’язаним
до гілки, щоб подальші повідомлення користувача в цій гілці й далі маршрутизувалися до
тієї самої сесії підагента.

### Канали з підтримкою гілок

**Discord** наразі є єдиним підтримуваним каналом. Він підтримує
постійні сесії підагентів, прив’язані до гілок (`sessions_spawn` із
`thread: true`), ручне керування гілками (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`) і ключі адаптера
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours` і
`channels.discord.threadBindings.spawnSubagentSessions`.

### Швидкий сценарій

<Steps>
  <Step title="Запуск">
    `sessions_spawn` із `thread: true` (і, за потреби, `mode: "session"`).
  </Step>
  <Step title="Прив’язка">
    OpenClaw створює або прив’язує гілку до цілі цієї сесії в активному каналі.
  </Step>
  <Step title="Маршрутизація подальших повідомлень">
    Відповіді та подальші повідомлення в цій гілці маршрутизуються до прив’язаної сесії.
  </Step>
  <Step title="Перевірка тайм-аутів">
    Використовуйте `/session idle`, щоб переглядати/оновлювати автоматичне відв’язування через неактивність, і
    `/session max-age`, щоб керувати жорстким обмеженням.
  </Step>
  <Step title="Від’єднання">
    Використовуйте `/unfocus` для ручного від’єднання.
  </Step>
</Steps>

### Ручні елементи керування

| Команда            | Дія                                                                    |
| ------------------ | ---------------------------------------------------------------------- |
| `/focus <target>`  | Прив’язує поточну гілку (або створює її) до цілі підагента/сесії       |
| `/unfocus`         | Видаляє прив’язку для поточної прив’язаної гілки                       |
| `/agents`          | Показує список активних виконань і стан прив’язки (`thread:<id>` або `unbound`) |
| `/session idle`    | Перегляд/оновлення автоматичного відв’язування через неактивність (лише для прив’язаних гілок у фокусі) |
| `/session max-age` | Перегляд/оновлення жорсткого обмеження (лише для прив’язаних гілок у фокусі) |

### Перемикачі конфігурації

- **Глобальне типове значення:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **Ключі перевизначення каналу та автоматичної прив’язки під час spawn** залежать від адаптера. Див. [Канали з підтримкою гілок](#thread-supporting-channels) вище.

Див. [Довідник із конфігурації](/uk/gateway/configuration-reference) і
[Слеш-команди](/uk/tools/slash-commands) для актуальних деталей адаптера.

### Allowlist

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  Список id агентів, на які можна націлюватися через явний `agentId` (`["*"]` дозволяє будь-який). Типово: лише агент запитувача. Якщо ви задаєте список і все одно хочете, щоб запитувач запускав самого себе через `agentId`, включіть id запитувача до списку.
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  Типовий allowlist цільових агентів, що використовується, коли агент запитувача не задає власний `subagents.allowAgents`.
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  Блокує виклики `sessions_spawn`, у яких пропущено `agentId` (примушує до явного вибору профілю). Перевизначення для окремого агента: `agents.list[].subagents.requireAgentId`.
</ParamField>

Якщо сесію запитувача ізольовано sandbox-ом, `sessions_spawn` відхиляє цілі,
які працювали б без sandbox.

### Виявлення

Використовуйте `agents_list`, щоб побачити, які id агентів зараз дозволені для
`sessions_spawn`. Відповідь включає ефективну
модель кожного переліченого агента та метадані вбудованого runtime, щоб викликачі могли розрізняти PI, Codex
app-server та інші налаштовані нативні runtime.

### Автоархівація

- Сесії підагентів автоматично архівуються після `agents.defaults.subagents.archiveAfterMinutes` (типово `60`).
- Архівація використовує `sessions.delete` і перейменовує transcript на `*.deleted.<timestamp>` (у тій самій теці).
- `cleanup: "delete"` архівує одразу після announce (при цьому transcript усе одно зберігається через перейменування).
- Автоархівація виконується в режимі best effort; очікувані таймери втрачаються, якщо gateway перезапускається.
- `runTimeoutSeconds` **не** виконує автоархівацію; він лише зупиняє виконання. Сесія залишається до автоархівації.
- Автоархівація однаково застосовується до сесій глибини 1 і 2.
- Очищення браузера відокремлене від очищення архіву: відстежувані вкладки/процеси браузера закриваються в режимі best effort після завершення виконання, навіть якщо запис transcript/сесії зберігається.

## Вкладені підагенти

За замовчуванням підагенти не можуть запускати власних підагентів
(`maxSpawnDepth: 1`). Установіть `maxSpawnDepth: 2`, щоб увімкнути один рівень
вкладеності — **шаблон оркестратора**: main → підагент-оркестратор →
дочірні підагенти-воркери.

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // дозволяє підагентам запускати дочірніх агентів (типово: 1)
        maxChildrenPerAgent: 5, // максимальна кількість активних дочірніх агентів на сесію агента (типово: 5)
        maxConcurrent: 8, // глобальне обмеження паралельності lane (типово: 8)
        runTimeoutSeconds: 900, // типовий тайм-аут для sessions_spawn, якщо його пропущено (0 = без тайм-ауту)
      },
    },
  },
}
```

### Рівні глибини

| Глибина | Форма session key                           | Роль                                          | Може запускати?              |
| ------- | ------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0       | `agent:<id>:main`                           | Основний агент                                | Завжди                       |
| 1       | `agent:<id>:subagent:<uuid>`                | Підагент (оркестратор, коли дозволено глибину 2) | Лише якщо `maxSpawnDepth >= 2` |
| 2       | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | Дочірній підагент-воркер (листовий вузол)     | Ніколи                       |

### Ланцюг announce

Результати повертаються вгору по ланцюгу:

1. Воркер глибини 2 завершується → надсилає announce своєму батьківському агенту (оркестратору глибини 1).
2. Оркестратор глибини 1 отримує announce, синтезує результати, завершується → надсилає announce до main.
3. Основний агент отримує announce і доставляє його користувачу.

Кожен рівень бачить announce лише від своїх прямих дочірніх агентів.

<Note>
**Операційна рекомендація:** запускайте дочірню роботу один раз і чекайте на події
завершення, а не будуйте цикли опитування навколо `sessions_list`,
`sessions_history`, `/subagents list` або команд `exec` зі сном.
`sessions_list` і `/subagents list` тримають зв’язки дочірніх сесій
сфокусованими на живій роботі — активні дочірні сесії залишаються прив’язаними, завершені лишаються
видимими короткий недавній проміжок, а застарілі посилання на дочірні сесії лише зі сховища
ігноруються після завершення вікна свіжості. Це запобігає тому, щоб старі метадані `spawnedBy` /
`parentSessionKey` відновлювали примарних дочірніх агентів після
перезапуску. Якщо подія завершення дочірнього агента надходить після того, як ви вже надіслали
фінальну відповідь, правильним продовженням є точний тихий токен
`NO_REPLY` / `no_reply`.
</Note>

### Політика інструментів за глибиною

- Роль і межі керування записуються в метадані сесії під час запуску. Це не дає змоги плоским або відновленим session key випадково повернути привілеї оркестратора.
- **Глибина 1 (оркестратор, коли `maxSpawnDepth >= 2`):** отримує `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history`, щоб керувати своїми дочірніми агентами. Інші session tools і system tools залишаються забороненими.
- **Глибина 1 (листовий вузол, коли `maxSpawnDepth == 1`):** без session tools (поточна типова поведінка).
- **Глибина 2 (листовий воркер):** без session tools — `sessions_spawn` завжди заборонено на глибині 2. Не може запускати подальших дочірніх агентів.

### Ліміт запуску для окремого агента

Кожна сесія агента (на будь-якій глибині) може мати щонайбільше `maxChildrenPerAgent`
(типово `5`) активних дочірніх агентів одночасно. Це запобігає неконтрольованому fan-out
від одного оркестратора.

### Каскадна зупинка

Зупинка оркестратора глибини 1 автоматично зупиняє всіх його дочірніх агентів
глибини 2:

- `/stop` в основному чаті зупиняє всіх агентів глибини 1 і каскадно їхніх дочірніх агентів глибини 2.
- `/subagents kill <id>` зупиняє конкретного підагента і каскадно його дочірніх агентів.
- `/subagents kill all` зупиняє всіх підагентів для запитувача і виконує каскадну зупинку.

## Автентифікація

Автентифікація підагента визначається за **id агента**, а не за типом сесії:

- Session key підагента: `agent:<agentId>:subagent:<uuid>`.
- Сховище auth завантажується з `agentDir` цього агента.
- Профілі auth основного агента додаються як **резервний варіант**; профілі агента мають пріоритет над профілями main у разі конфліктів.

Злиття є додатковим, тому профілі main завжди доступні як
резервні. Повністю ізольована auth для кожного агента поки що не підтримується.

## Announce

Підагенти звітують назад через крок announce:

- Крок announce виконується всередині сесії підагента (а не сесії запитувача).
- Якщо підагент відповідає точно `ANNOUNCE_SKIP`, нічого не публікується.
- Якщо останній текст assistant є точним тихим токеном `NO_REPLY` / `no_reply`, вивід announce пригнічується, навіть якщо раніше був видимий прогрес.

Доставка залежить від глибини запитувача:

- Сесії запитувача верхнього рівня використовують подальший виклик `agent` із зовнішньою доставкою (`deliver=true`).
- Вкладені сесії підагентів запитувача отримують внутрішнє подальше впровадження (`deliver=false`), щоб оркестратор міг синтезувати результати дочірніх агентів у межах сесії.
- Якщо вкладена сесія підагента запитувача зникла, OpenClaw використовує резервний перехід до запитувача цієї сесії, якщо він доступний.

Для сесій запитувача верхнього рівня пряма доставка в режимі завершення
спочатку визначає будь-який прив’язаний маршрут conversation/thread і перевизначення hook, а потім заповнює
відсутні поля цілі каналу збереженим маршрутом сесії запитувача.
Це зберігає завершення в правильному чаті/темі, навіть коли джерело завершення
ідентифікує лише канал.

Агрегація завершень дочірніх агентів обмежується поточним виконанням запитувача під час
побудови вкладених результатів завершення, що запобігає витіканню застарілих виводів дочірніх агентів із попередніх виконань
у поточний announce. Відповіді announce зберігають
маршрутизацію thread/topic, коли вона доступна в адаптерах каналу.

### Контекст announce

Контекст announce нормалізується до стабільного блоку внутрішньої події:

| Поле           | Джерело                                                                                                        |
| -------------- | -------------------------------------------------------------------------------------------------------------- |
| Джерело        | `subagent` або `cron`                                                                                          |
| Ідентифікатори сесії | Session key/id дочірньої сесії                                                                           |
| Тип            | Тип announce + мітка завдання                                                                                  |
| Статус         | Визначається з результату runtime (`success`, `error`, `timeout` або `unknown`) — **не** виводиться з тексту моделі |
| Вміст результату | Останній видимий текст assistant, або, якщо його немає, очищений останній текст `tool`/`toolResult`          |
| Подальша дія   | Інструкція, що описує, коли відповідати, а коли залишатися безмовним                                           |

Невдалі термінальні виконання повідомляють про статус помилки без повторного відтворення
захопленого тексту відповіді. У разі тайм-ауту, якщо дочірній агент встиг лише до викликів інструментів, announce
може згорнути цю історію до короткого зведення часткового прогресу
замість повторного відтворення сирого виводу інструмента.

### Рядок статистики

Payload announce містять рядок статистики наприкінці (навіть якщо вони обгорнуті):

- Runtime (наприклад, `runtime 5m12s`).
- Використання токенів (вхідні/вихідні/загалом).
- Оцінна вартість, коли налаштовано ціни моделей (`models.providers.*.models[].cost`).
- `sessionKey`, `sessionId` і шлях до transcript, щоб основний агент міг отримати історію через `sessions_history` або перевірити файл на диску.

Внутрішні метадані призначені лише для оркестрації; відповіді для користувача
слід переписувати звичайним голосом асистента.

### Чому краще використовувати `sessions_history`

`sessions_history` — безпечніший шлях оркестрації:

- Згадування assistant спочатку нормалізується: теги thinking видаляються; каркас `<relevant-memories>` / `<relevant_memories>` видаляється; блоки XML payload викликів інструментів у простому тексті (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) видаляються, включно з обрізаними payload, які так і не закриваються коректно; знижується і видаляється каркас викликів/результатів інструментів і маркери історичного контексту; видаляються витеклі токени керування моделі (`<|assistant|>`, інші ASCII `<|...|>`, повноширинні `<｜...｜>`); видаляється некоректний XML викликів інструментів MiniMax.
- Текст, схожий на облікові дані/токени, редагується.
- Довгі блоки можуть обрізатися.
- Дуже великі історії можуть скидати старіші рядки або замінювати надто великий рядок на `[sessions_history omitted: message too large]`.
- Перегляд сирого transcript на диску — це резервний варіант, коли вам потрібен повний transcript байт у байт.

## Політика інструментів

Підагенти спочатку використовують той самий профіль і конвеєр політики інструментів, що й батьківський
або цільовий агент. Після цього OpenClaw застосовує шар обмежень підагента.

Без обмежувального `tools.profile` підагенти отримують **усі інструменти, крім
session tools** і system tools:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

Тут `sessions_history` також залишається обмеженим і очищеним представленням згадування —
це не сирий дамп transcript.

Коли `maxSpawnDepth >= 2`, підагенти-оркестратори глибини 1 додатково
отримують `sessions_spawn`, `subagents`, `sessions_list` і
`sessions_history`, щоб керувати своїми дочірніми агентами.

### Перевизначення через конфігурацію

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // заборона має пріоритет
        deny: ["gateway", "cron"],
        // якщо задано allow, він стає фільтром лише-дозволити (deny усе одно має пріоритет)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` — це фінальний фільтр лише-дозволити. Він може звузити
вже визначений набір інструментів, але не може **повернути**
інструмент, вилучений `tools.profile`. Наприклад, `tools.profile: "coding"` включає
`web_search`/`web_fetch`, але не інструмент `browser`. Щоб дозволити
підагентам із профілем coding використовувати автоматизацію браузера, додайте browser на
етапі профілю:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

Використовуйте `agents.list[].tools.alsoAllow: ["browser"]` для окремого агента, коли лише один
агент має отримати автоматизацію браузера.

## Паралельність

Підагенти використовують окрему lane черги в межах процесу:

- **Назва lane:** `subagent`
- **Паралельність:** `agents.defaults.subagents.maxConcurrent` (типово `8`)

## Доступність і відновлення

OpenClaw не вважає відсутність `endedAt` постійним доказом того, що
підагент усе ще живий. Незавершені виконання, старіші за вікно застарілого виконання,
перестають враховуватися як активні/очікувані в `/subagents list`, зведеннях статусу,
контролі завершення нащадків і перевірках паралельності для окремої сесії.

Після перезапуску gateway застарілі відновлені незавершені виконання відсікаються, якщо
їхня дочірня сесія не позначена `abortedLastRun: true`. Такі дочірні сесії,
перервані перезапуском, залишаються відновлюваними через потік відновлення orphan підагентів,
який надсилає синтетичне повідомлення відновлення перед
очищенням маркера переривання.

<Note>
Якщо запуск підагента завершується помилкою Gateway `PAIRING_REQUIRED` /
`scope-upgrade`, перевірте викликач RPC, перш ніж редагувати стан pairing.
Внутрішня координація `sessions_spawn` повинна підключатися як
`client.id: "gateway-client"` з `client.mode: "backend"` через пряму
автентифікацію shared-token/password у loopback; цей шлях не залежить від
базового рівня scope парного пристрою CLI. Віддалені викликачі, явний
`deviceIdentity`, явні шляхи device-token і browser/node клієнти
усе ще потребують звичайного схвалення пристрою для підвищення scope.
</Note>

## Зупинка

- Надсилання `/stop` у чаті запитувача перериває сесію запитувача та зупиняє будь-які активні виконання підагентів, запущені з неї, з каскадним поширенням на вкладених дочірніх агентів.
- `/subagents kill <id>` зупиняє конкретного підагента і каскадно його дочірніх агентів.

## Обмеження

- Announce підагента — це **best-effort**. Якщо gateway перезапускається, відкладена робота «повідомити назад» втрачається.
- Підагенти все ще спільно використовують ресурси того самого процесу gateway; розглядайте `maxConcurrent` як запобіжний клапан.
- `sessions_spawn` завжди неблокувальний: він одразу повертає `{ status: "accepted", runId, childSessionKey }`.
- Контекст підагента впроваджує лише `AGENTS.md` + `TOOLS.md` (без `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` або `BOOTSTRAP.md`).
- Максимальна глибина вкладеності — 5 (діапазон `maxSpawnDepth`: 1–5). Для більшості випадків використання рекомендовано глибину 2.
- `maxChildrenPerAgent` обмежує кількість активних дочірніх агентів на сесію (типово `5`, діапазон `1–20`).
