Sub-agents — це фонові запуски агентів, породжені з наявного запуску агента.
Вони працюють у власній сесії (`agent:<agentId>:subagent:<uuid>`) і,
після завершення, **повідомляють** про свій результат назад у канал чату
запитувача. Кожен запуск sub-agent відстежується як
[фонове завдання](/uk/automation/tasks).

Основні цілі:

- Розпаралелити роботу типу "дослідження / довге завдання / повільний інструмент", не блокуючи основний запуск.
- За замовчуванням тримати sub-agent ізольованими (розділення сесій + необов’язкова пісочниця).
- Утримувати поверхню інструментів складною для неправильного використання: sub-agents **не** отримують інструменти сесій за замовчуванням.
- Підтримувати налаштовувану глибину вкладеності для шаблонів оркестрації.

<Note>
**Примітка про вартість:** кожен sub-agent за замовчуванням має власний контекст і власне використання токенів.
Для важких або повторюваних завдань задайте дешевшу модель для sub-agents,
а основного агента залиште на якіснішій моделі. Налаштовується через
`agents.defaults.subagents.model` або перевизначення для окремого агента. Коли дочірньому агенту
справді потрібен поточний transcript запитувача, агент може запитати
`context: "fork"` для цього конкретного spawn.
</Note>

## Slash-команда

Використовуйте `/subagents`, щоб переглядати або керувати запусками sub-agent для **поточної
сесії**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

`/subagents info` показує metadata запуску (статус, часові мітки, id сесії,
шлях до transcript, cleanup). Використовуйте `sessions_history` для обмеженого,
відфільтрованого з погляду безпеки перегляду recall; переглядайте шлях до transcript на диску, коли
вам потрібен сирий повний transcript.

### Елементи керування прив’язкою до треду

Ці команди працюють у каналах, що підтримують постійні прив’язки до тредів.
Див. [Канали з підтримкою тредів](#thread-supporting-channels) нижче.

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### Поведінка spawn

`/subagents spawn` запускає фоновий sub-agent як команду користувача (а не
внутрішнє relay) і надсилає одне фінальне оновлення про завершення назад у
чат запитувача, коли запуск завершується.

<AccordionGroup>
  <Accordion title="Неблокувальне завершення на основі push">
    - Команда spawn не блокує; вона одразу повертає id запуску.
    - Після завершення sub-agent повідомляє коротке зведення/результат назад у канал чату запитувача.
    - Доставка завершення відбувається через push. Після spawn **не** потрібно опитувати `/subagents list`, `sessions_list` або `sessions_history` у циклі лише для очікування завершення; перевіряйте статус лише за потреби для налагодження або втручання.
    - Після завершення OpenClaw best-effort закриває відстежувані вкладки/процеси браузера, відкриті цією сесією sub-agent, перш ніж продовжиться потік cleanup для announce.
  </Accordion>
  <Accordion title="Стійкість доставки при ручному spawn">
    - OpenClaw спочатку намагається доставити напряму в `agent` зі стабільним ключем ідемпотентності.
    - Якщо пряма доставка не вдається, використовується резервна маршрутизація через queue.
    - Якщо маршрутизація через queue усе ще недоступна, announce повторюється з коротким exponential backoff перед остаточною відмовою.
    - Доставка завершення зберігає розв’язаний маршрут запитувача: маршрути завершення, прив’язані до треду або розмови, мають пріоритет, коли доступні; якщо джерело завершення надає лише канал, OpenClaw заповнює відсутні target/account із розв’язаного маршруту сесії запитувача (`lastChannel` / `lastTo` / `lastAccountId`), щоб пряма доставка однаково працювала.
  </Accordion>
  <Accordion title="Metadata для передачі завершення">
    Передача завершення назад у сесію запитувача — це внутрішній контекст,
    згенерований під час виконання (а не текст, створений користувачем), і він містить:

    - `Result` — останній видимий текст відповіді `assistant`, або, якщо його немає, санітизований останній текст `tool`/`toolResult`. Для термінально невдалих запусків захоплений текст відповіді не використовується повторно.
    - `Status` — `completed successfully` / `failed` / `timed out` / `unknown`.
    - Компактну статистику runtime/token.
    - Інструкцію доставки, яка каже агенту запитувача переписати це нормальним голосом assistant (а не пересилати сирі внутрішні metadata).

  </Accordion>
  <Accordion title="Режими та середовище виконання ACP">
    - `--model` і `--thinking` перевизначають типові значення для цього конкретного запуску.
    - Використовуйте `info`/`log`, щоб переглянути деталі та вивід після завершення.
    - `/subagents spawn` — це одноразовий режим (`mode: "run"`). Для постійних сесій, прив’язаних до треду, використовуйте `sessions_spawn` з `thread: true` і `mode: "session"`.
    - Для сесій harness ACP (Claude Code, Gemini CLI, OpenCode або явний Codex ACP/acpx) використовуйте `sessions_spawn` з `runtime: "acp"`, коли інструмент рекламує це середовище виконання. Див. [Модель доставки ACP](/uk/tools/acp-agents#delivery-model), якщо налагоджуєте завершення або цикли агент-до-агента. Коли Plugin `codex` увімкнено, для керування чатом/тредом Codex слід надавати перевагу `/codex ...` замість ACP, якщо користувач явно не просить ACP/acpx.
    - OpenClaw приховує `runtime: "acp"`, доки ACP не буде увімкнено, запитувач не буде в пісочниці, а plugin бекенда, як-от `acpx`, не буде завантажено. `runtime: "acp"` очікує зовнішній id harness ACP або запис `agents.list[]` з `runtime.type="acp"`; для звичайних агентів конфігурації OpenClaw з `agents_list` використовуйте типове середовище виконання sub-agent.
  </Accordion>
</AccordionGroup>

## Режими контексту

Нативні sub-agents запускаються ізольовано, якщо викликач явно не просить зробити fork
поточного transcript.

| Mode       | Коли його використовувати                                                                                                                         | Поведінка                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | Нове дослідження, незалежна реалізація, повільна робота інструментів або будь-що, що можна коротко описати в тексті завдання                           | Створює чистий дочірній transcript. Це типове значення й утримує використання токенів на нижчому рівні.  |
| `fork`     | Робота, що залежить від поточної розмови, попередніх результатів інструментів або нюансованих інструкцій, уже наявних у transcript запитувача | Розгалужує transcript запитувача в дочірню сесію до того, як дочірня сесія почне роботу. |

Використовуйте `fork` ощадливо. Це варіант для делегування, чутливого до
контексту, а не заміна чітко написаному prompt завдання.

## Інструмент: `sessions_spawn`

Запускає sub-agent із `deliver: false` на глобальній смузі `subagent`,
потім виконує крок announce і публікує announce-відповідь у
канал чату запитувача.

**Типові значення:**

- **Модель:** успадковується від викликача, якщо ви не задали `agents.defaults.subagents.model` (або `agents.list[].subagents.model` для окремого агента); явне `sessions_spawn.model` усе одно має пріоритет.
- **Thinking:** успадковується від викликача, якщо ви не задали `agents.defaults.subagents.thinking` (або `agents.list[].subagents.thinking` для окремого агента); явне `sessions_spawn.thinking` усе одно має пріоритет.
- **Тайм-аут запуску:** якщо `sessions_spawn.runTimeoutSeconds` пропущено, OpenClaw використовує `agents.defaults.subagents.runTimeoutSeconds`, якщо його задано; інакше використовується `0` (без тайм-ауту).

### Параметри інструмента

<ParamField path="task" type="string" required>
  Опис завдання для sub-agent.
</ParamField>
<ParamField path="label" type="string">
  Необов’язкова зручна для людини мітка.
</ParamField>
<ParamField path="agentId" type="string">
  Spawn під іншим id агента, якщо це дозволено через `subagents.allowAgents`.
</ParamField>
<ParamField path="runtime" type='"subagent" | "acp"' default="subagent">
  `acp` призначено лише для зовнішніх harness ACP (`claude`, `droid`, `gemini`, `opencode` або явно запитаний Codex ACP/acpx) і для записів `agents.list[]`, у яких `runtime.type` дорівнює `acp`.
</ParamField>
<ParamField path="resumeSessionId" type="string">
  Лише ACP. Відновлює наявну сесію harness ACP, коли `runtime: "acp"`; ігнорується для нативних spawn sub-agent.
</ParamField>
<ParamField path="streamTo" type='"parent"'>
  Лише ACP. Потоково передає вивід запуску ACP в батьківську сесію, коли `runtime: "acp"`; для нативних spawn sub-agent не вказуйте.
</ParamField>
<ParamField path="model" type="string">
  Перевизначає модель sub-agent. Некоректні значення пропускаються, і sub-agent працює на типовій моделі з попередженням у результаті інструмента.
</ParamField>
<ParamField path="thinking" type="string">
  Перевизначає рівень thinking для запуску sub-agent.
</ParamField>
<ParamField path="runTimeoutSeconds" type="number">
  За замовчуванням використовується `agents.defaults.subagents.runTimeoutSeconds`, якщо його задано, інакше `0`. Якщо задано, запуск sub-agent переривається через N секунд.
</ParamField>
<ParamField path="thread" type="boolean" default="false">
  Якщо `true`, запитує прив’язку до треду каналу для цієї сесії sub-agent.
</ParamField>
<ParamField path="mode" type='"run" | "session"' default="run">
  Якщо `thread: true` і `mode` пропущено, типовим стає `session`. `mode: "session"` потребує `thread: true`.
</ParamField>
<ParamField path="cleanup" type='"delete" | "keep"' default="keep">
  `"delete"` архівує одразу після announce (усе одно зберігає transcript через перейменування).
</ParamField>
<ParamField path="sandbox" type='"inherit" | "require"' default="inherit">
  `require` відхиляє spawn, якщо цільове дочірнє середовище виконання не перебуває в пісочниці.
</ParamField>
<ParamField path="context" type='"isolated" | "fork"' default="isolated">
  `fork` розгалужує поточний transcript запитувача в дочірню сесію. Лише нативні sub-agents. Використовуйте `fork` лише тоді, коли дочірній агент потребує поточного transcript.
</ParamField>

<Warning>
`sessions_spawn` **не** приймає параметри доставки каналу (`target`,
`channel`, `to`, `threadId`, `replyTo`, `transport`). Для доставки використовуйте
`message`/`sessions_send` із породженого запуску.
</Warning>

## Сесії, прив’язані до треду

Коли прив’язки до треду увімкнено для каналу, sub-agent може залишатися прив’язаним
до треду, щоб подальші повідомлення користувача в цьому треді й далі маршрутизувалися до
тієї самої сесії sub-agent.

### Канали з підтримкою тредів

**Discord** наразі є єдиним підтримуваним каналом. Він підтримує
постійні сесії subagent, прив’язані до треду (`sessions_spawn` з
`thread: true`), ручне керування тредами (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`) і ключі адаптера
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours` і
`channels.discord.threadBindings.spawnSubagentSessions`.

### Швидкий потік

<Steps>
  <Step title="Spawn">
    `sessions_spawn` з `thread: true` (і необов’язково `mode: "session"`).
  </Step>
  <Step title="Прив’язка">
    OpenClaw створює тред або прив’язує його до цільової сесії в активному каналі.
  </Step>
  <Step title="Маршрутизація подальших повідомлень">
    Відповіді й подальші повідомлення в цьому треді маршрутизуються до прив’язаної сесії.
  </Step>
  <Step title="Перевірка тайм-аутів">
    Використовуйте `/session idle`, щоб переглядати/оновлювати автоматичне зняття фокуса через неактивність, і
    `/session max-age`, щоб керувати жорстким лімітом.
  </Step>
  <Step title="Відв’язка">
    Використовуйте `/unfocus`, щоб відв’язати вручну.
  </Step>
</Steps>

### Ручні елементи керування

| Command            | Effect                                                                |
| ------------------ | --------------------------------------------------------------------- |
| `/focus <target>`  | Прив’язати поточний тред (або створити новий) до цілі sub-agent/session |
| `/unfocus`         | Видалити прив’язку для поточного прив’язаного треду                       |
| `/agents`          | Перелічити активні запуски та стан прив’язки (`thread:<id>` або `unbound`)       |
| `/session idle`    | Переглянути/оновити автоматичне зняття фокуса через неактивність (лише для прив’язаних тредів у фокусі)         |
| `/session max-age` | Переглянути/оновити жорсткий ліміт (лише для прив’язаних тредів у фокусі)                  |

### Перемикачі конфігурації

- **Глобальне типове значення:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **Ключі перевизначення каналу та автоматичної прив’язки під час spawn** залежать від адаптера. Див. [Канали з підтримкою тредів](#thread-supporting-channels) вище.

Див. [Довідник із конфігурації](/uk/gateway/configuration-reference) і
[Slash-команди](/uk/tools/slash-commands) для актуальних деталей адаптерів.

### Список дозволених

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  Список id агентів, які можна вказувати через `agentId` (`["*"]` дозволяє будь-який). Типово: лише агент запитувача.
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  Типовий список дозволених цільових агентів, який використовується, коли агент запитувача не задає власний `subagents.allowAgents`.
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  Блокувати виклики `sessions_spawn`, у яких пропущено `agentId` (примушує до явного вибору профілю). Перевизначення для окремого агента: `agents.list[].subagents.requireAgentId`.
</ParamField>

Якщо сесія запитувача перебуває в пісочниці, `sessions_spawn` відхиляє цілі,
які запускалися б без пісочниці.

### Виявлення

Використовуйте `agents_list`, щоб побачити, які id агентів зараз дозволені для
`sessions_spawn`. Відповідь містить ефективну
модель кожного переліченого агента та вбудовані metadata середовища виконання, щоб викликачі могли відрізняти PI, Codex
app-server та інші налаштовані нативні середовища виконання.

### Автоархівація

- Сесії sub-agent автоматично архівуються через `agents.defaults.subagents.archiveAfterMinutes` (типово `60`).
- Архівація використовує `sessions.delete` і перейменовує transcript на `*.deleted.<timestamp>` (у тій самій папці).
- `cleanup: "delete"` архівує одразу після announce (усе одно зберігає transcript через перейменування).
- Автоархівація виконується best-effort; очікувані таймери втрачаються після перезапуску gateway.
- `runTimeoutSeconds` **не** виконує автоархівацію; він лише зупиняє запуск. Сесія залишається до автоархівації.
- Автоархівація однаково застосовується до сесій глибини 1 і глибини 2.
- Очищення браузера відокремлене від очищення архіву: відстежувані вкладки/процеси браузера best-effort закриваються, коли запуск завершується, навіть якщо запис transcript/сесії зберігається.

## Вкладені sub-agents

За замовчуванням sub-agents не можуть породжувати власних sub-agents
(`maxSpawnDepth: 1`). Задайте `maxSpawnDepth: 2`, щоб увімкнути один рівень
вкладеності — **шаблон оркестратора**: main → sub-agent-оркестратор →
sub-sub-agents-воркери.

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // дозволити sub-agents породжувати дочірніх агентів (типово: 1)
        maxChildrenPerAgent: 5, // максимальна кількість активних дочірніх агентів на сесію агента (типово: 5)
        maxConcurrent: 8, // глобальне обмеження паралельності lane (типово: 8)
        runTimeoutSeconds: 900, // типовий тайм-аут для sessions_spawn, якщо пропущено (0 = без тайм-ауту)
      },
    },
  },
}
```

### Рівні глибини

| Depth | Session key shape                            | Role                                          | Can spawn?                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | Основний агент                                    | Завжди                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | Sub-agent (оркестратор, коли дозволено depth 2) | Лише якщо `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | Sub-sub-agent (листовий воркер)                   | Ніколи                        |

### Ланцюжок announce

Результати повертаються вгору по ланцюжку:

1. Воркер глибини 2 завершується → повідомляє батьківський агент (оркестратор глибини 1).
2. Оркестратор глибини 1 отримує announce, синтезує результати, завершується → повідомляє main.
3. Основний агент отримує announce і доставляє його користувачу.

Кожен рівень бачить announce лише від своїх прямих дочірніх агентів.

<Note>
**Операційна рекомендація:** запускайте дочірню роботу один раз і чекайте
подій завершення замість побудови циклів опитування навколо `sessions_list`,
`sessions_history`, `/subagents list` або команд сну `exec`.
`sessions_list` і `/subagents list` зосереджують зв’язки дочірніх сесій
на живій роботі — живі дочірні сесії залишаються прикріпленими, завершені дочірні сесії лишаються
видимими коротке недавнє вікно, а застарілі посилання на дочірні сесії лише зі сховища
ігноруються після вікна їх свіжості. Це запобігає тому, щоб старі metadata `spawnedBy` /
`parentSessionKey` відроджували примарних дочірніх агентів після
перезапуску. Якщо подія завершення дочірньої сесії приходить після того, як ви вже надіслали
остаточну відповідь, правильна подальша дія — це точний тихий токен
`NO_REPLY` / `no_reply`.
</Note>

### Політика інструментів за глибиною

- Роль і область керування записуються в metadata сесії під час spawn. Це не дає плоским або відновленим ключам сесій випадково повернути собі привілеї оркестратора.
- **Глибина 1 (оркестратор, коли `maxSpawnDepth >= 2`):** отримує `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history`, щоб керувати своїми дочірніми агентами. Інші інструменти сесії/системи лишаються забороненими.
- **Глибина 1 (листовий агент, коли `maxSpawnDepth == 1`):** інструменти сесії відсутні (поточна типова поведінка).
- **Глибина 2 (листовий воркер):** інструменти сесії відсутні — `sessions_spawn` завжди заборонено на глибині 2. Не може породжувати подальших дочірніх агентів.

### Ліміт spawn для окремого агента

Кожна сесія агента (на будь-якій глибині) може мати не більше `maxChildrenPerAgent`
(типово `5`) активних дочірніх агентів одночасно. Це запобігає неконтрольованому fan-out
від одного оркестратора.

### Каскадна зупинка

Зупинка оркестратора глибини 1 автоматично зупиняє всіх його дочірніх агентів
глибини 2:

- `/stop` в основному чаті зупиняє всіх агентів глибини 1 і каскадно — їхніх дочірніх агентів глибини 2.
- `/subagents kill <id>` зупиняє конкретного sub-agent і каскадно — його дочірніх агентів.
- `/subagents kill all` зупиняє всіх sub-agents для запитувача й виконує каскадну зупинку.

## Автентифікація

Auth sub-agent розв’язується за **id агента**, а не за типом сесії:

- Ключ сесії sub-agent — `agent:<agentId>:subagent:<uuid>`.
- Сховище auth завантажується з `agentDir` цього агента.
- Профілі auth основного агента додаються як **резервний варіант**; у разі конфліктів профілі агента мають пріоритет над профілями main.

Злиття є адитивним, тому профілі main завжди доступні як
резервні. Повністю ізольований auth для кожного агента поки не підтримується.

## Announce

Sub-agents звітують назад через крок announce:

- Крок announce виконується всередині сесії sub-agent (а не сесії запитувача).
- Якщо sub-agent відповідає рівно `ANNOUNCE_SKIP`, нічого не публікується.
- Якщо останній текст `assistant` є точним тихим токеном `NO_REPLY` / `no_reply`, вивід announce пригнічується, навіть якщо раніше був видимий прогрес.

Доставка залежить від глибини запитувача:

- Сесії запитувача верхнього рівня використовують подальший виклик `agent` із зовнішньою доставкою (`deliver=true`).
- Вкладені сесії subagent запитувача отримують внутрішню ін’єкцію подальшого виклику (`deliver=false`), щоб оркестратор міг синтезувати результати дочірніх агентів у межах сесії.
- Якщо вкладена сесія subagent запитувача вже відсутня, OpenClaw використовує резервне повернення до запитувача цієї сесії, якщо це можливо.

Для сесій запитувача верхнього рівня пряма доставка в режимі завершення
спочатку розв’язує будь-який прив’язаний маршрут розмови/треду та перевизначення hook, а потім заповнює
відсутні поля channel-target із збереженого маршруту сесії запитувача.
Це утримує завершення в правильному чаті/темі, навіть коли джерело завершення
ідентифікує лише канал.

Агрегація завершення дочірніх агентів обмежується поточним запуском запитувача під час
побудови вкладених результатів завершення, що не дає застарілим виводам дочірніх агентів із попередніх запусків
просочуватися в поточний announce. Відповіді announce зберігають
маршрутизацію треду/теми, коли вона доступна в адаптерах каналів.

### Контекст announce

Контекст announce нормалізується до стабільного внутрішнього блоку події:

| Field          | Source                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| Джерело         | `subagent` або `cron`                                                                                          |
| Id сесії    | Ключ/id дочірньої сесії                                                                                          |
| Тип           | Тип announce + мітка завдання                                                                                    |
| Статус         | Похідний від результату runtime (`success`, `error`, `timeout` або `unknown`) — **не** виводиться з тексту моделі |
| Вміст результату | Останній видимий текст `assistant`, або, якщо його немає, санітизований останній текст `tool`/`toolResult`                                |
| Подальша дія      | Інструкція, що описує, коли відповідати, а коли мовчати                                                           |

Термінально невдалі запуски повідомляють статус невдачі без повторного
відтворення захопленого тексту відповіді. У разі тайм-ауту, якщо дочірня сесія встигла
лише до викликів інструментів, announce може згорнути цю історію в коротке зведення часткового прогресу
замість відтворення сирого виводу інструментів.

### Рядок статистики

Payload announce містять рядок статистики в кінці (навіть якщо вони загорнуті):

- Runtime (наприклад, `runtime 5m12s`).
- Використання токенів (вхідні/вихідні/загалом).
- Орієнтовну вартість, коли налаштовано ціни моделі (`models.providers.*.models[].cost`).
- `sessionKey`, `sessionId` і шлях до transcript, щоб основний агент міг отримати історію через `sessions_history` або перевірити файл на диску.

Внутрішні metadata призначені лише для оркестрації; відповіді для користувача
слід переписувати нормальним голосом assistant.

### Чому краще `sessions_history`

`sessions_history` — безпечніший шлях оркестрації:

- Recall assistant спочатку нормалізується: теги thinking видаляються; scaffolding `<relevant-memories>` / `<relevant_memories>` видаляється; блоки XML payload викликів інструментів у звичайному тексті (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) видаляються, включно з обрізаними payload, які ніколи не закриваються коректно; понижений scaffolding викликів/результатів інструментів і маркери історичного контексту видаляються; витоки контрольних токенів моделі (`<|assistant|>`, інші ASCII `<|...|>`, повноширинні `<｜...｜>`) видаляються; некоректний XML викликів інструментів MiniMax видаляється.
- Текст, схожий на облікові дані/токени, редагується.
- Довгі блоки можуть обрізатися.
- Дуже великі історії можуть відкидати старіші рядки або замінювати надто великий рядок на `[sessions_history omitted: message too large]`.
- Перегляд сирого transcript на диску — резервний варіант, коли вам потрібен повний transcript байт-у-байт.

## Політика інструментів

Sub-agents спочатку використовують той самий профіль і pipeline політики інструментів, що й батьківський або
цільовий агент. Після цього OpenClaw застосовує шар обмежень sub-agent.

Якщо обмежувальний `tools.profile` відсутній, sub-agents отримують **усі інструменти, крім
інструментів сесії** та системних інструментів:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

Тут `sessions_history` також лишається обмеженим, санітизованим переглядом recall —
це не сирий дамп transcript.

Коли `maxSpawnDepth >= 2`, sub-agents-оркестратори глибини 1 додатково
отримують `sessions_spawn`, `subagents`, `sessions_list` і
`sessions_history`, щоб керувати своїми дочірніми агентами.

### Перевизначення через конфігурацію

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny має пріоритет
        deny: ["gateway", "cron"],
        // якщо задано allow, він стає фільтром лише дозволених (deny усе одно має пріоритет)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` — це фінальний фільтр лише дозволених інструментів. Він може звузити
вже розв’язаний набір інструментів, але не може **повернути**
інструмент, прибраний через `tools.profile`. Наприклад, `tools.profile: "coding"` включає
`web_search`/`web_fetch`, але не включає інструмент `browser`. Щоб дозволити
sub-agents із профілем coding використовувати автоматизацію браузера, додайте browser на
етапі профілю:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

Використовуйте `agents.list[].tools.alsoAllow: ["browser"]` для окремого агента, коли лише один
агент повинен отримати автоматизацію браузера.

## Паралельність

Sub-agents використовують окрему внутрішньопроцесну lane черги:

- **Назва lane:** `subagent`
- **Паралельність:** `agents.defaults.subagents.maxConcurrent` (типово `8`)

## Живучість і відновлення

OpenClaw не вважає відсутність `endedAt` постійним доказом того, що
sub-agent усе ще живий. Незавершені запуски, старші за вікно застарілого запуску,
перестають враховуватися як активні/очікувані в `/subagents list`, зведеннях стану,
контролі завершення нащадків і перевірках паралельності для окремої сесії.

Після перезапуску gateway застарілі відновлені незавершені запуски відсікаються, якщо
їхня дочірня сесія не позначена як `abortedLastRun: true`. Такі
дочірні сесії, перервані перезапуском, залишаються відновлюваними через потік відновлення сиріт sub-agent, який надсилає синтетичне повідомлення відновлення перед
очищенням маркера переривання.

<Note>
Якщо spawn sub-agent завершується помилкою Gateway `PAIRING_REQUIRED` /
`scope-upgrade`, перевірте RPC-викликач, перш ніж редагувати стан pairing.
Внутрішня координація `sessions_spawn` повинна підключатися як
`client.id: "gateway-client"` з `client.mode: "backend"` через пряму
loopback-аутентифікацію зі спільним токеном/паролем; цей шлях не залежить від базового рівня scope paired-device у CLI. Віддалені викликачі, явний
`deviceIdentity`, явні шляхи device-token і клієнти браузера/Node
усе ще потребують звичайного схвалення пристрою для підвищення scope.
</Note>

## Зупинка

- Надсилання `/stop` у чаті запитувача перериває сесію запитувача та зупиняє всі активні запуски sub-agent, породжені з неї, каскадно зупиняючи вкладених дочірніх агентів.
- `/subagents kill <id>` зупиняє конкретного sub-agent і каскадно — його дочірніх агентів.

## Обмеження

- Announce sub-agent виконується в режимі **best-effort**. Якщо gateway перезапуститься, очікувана робота "повідомити назад" буде втрачена.
- Sub-agents усе ще ділять ті самі ресурси процесу gateway; розглядайте `maxConcurrent` як запобіжний клапан.
- `sessions_spawn` завжди неблокувальний: він одразу повертає `{ status: "accepted", runId, childSessionKey }`.
- Контекст sub-agent інжектує лише `AGENTS.md` + `TOOLS.md` (без `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` або `BOOTSTRAP.md`).
- Максимальна глибина вкладеності — 5 (діапазон `maxSpawnDepth`: 1–5). Для більшості випадків використання рекомендовано глибину 2.
- `maxChildrenPerAgent` обмежує кількість активних дочірніх агентів на сесію (типово `5`, діапазон `1–20`).
