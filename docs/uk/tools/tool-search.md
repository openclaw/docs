Пошук інструментів є експериментальною функцією середовища виконання агента OpenClaw. Він дає агентам один
компактний спосіб знаходити й викликати великі каталоги інструментів. Це корисно, коли запуск
має багато доступних інструментів, але модель, імовірно, потребуватиме лише кількох із них.

Ця сторінка документує Пошук інструментів OpenClaw. Це не власна для Codex поверхня
пошуку інструментів або динамічних інструментів. Власний для Codex режим коду, пошук інструментів, відкладені
динамічні інструменти та вкладені виклики інструментів є стабільними поверхнями оснастки Codex і
не залежать від `tools.toolSearch`.

Коли це ввімкнено для запусків OpenClaw, модель за замовчуванням отримує один інструмент `tool_search_code`.
Цей інструмент запускає коротке тіло JavaScript в ізольованому підпроцесі Node
з мостом `openclaw.tools`:

```js
const hits = await openclaw.tools.search("create a GitHub issue");
const tool = await openclaw.tools.describe(hits[0].id);
return await openclaw.tools.call(tool.id, {
  title: "Crash on startup",
  body: "Steps to reproduce...",
});
```

Каталог може містити інструменти OpenClaw, інструменти Plugin, інструменти MCP та
інструменти, надані клієнтом. Модель не бачить кожну повну схему наперед.
Натомість вона шукає компактні дескриптори, описує один вибраний інструмент, коли їй
потрібна точна схема, і викликає цей інструмент через OpenClaw.

Запуски оснастки Codex не отримують ці експериментальні засоби керування Пошуком інструментів OpenClaw.
OpenClaw передає можливості продукту до Codex як динамічні інструменти, а
Codex володіє стабільним власним режимом коду, власним пошуком інструментів, відкладеними динамічними
інструментами та вкладеними викликами інструментів.

## Як виконується хід

Під час планування вбудований runner OpenClaw будує ефективний каталог для
запуску:

1. Визначає активну політику інструментів для агента, профілю, пісочниці та сесії.
2. Перелічує придатні інструменти OpenClaw і Plugin.
3. Перелічує придатні інструменти MCP через середовище виконання MCP сесії.
4. Додає придатні клієнтські інструменти, надані для поточного запуску.
5. Індексує компактні дескриптори для пошуку.
6. Надає моделі кодовий міст OpenClaw, структуровані резервні інструменти або
   компактну поверхню каталогу.

Під час виконання кожен реальний виклик інструмента повертається до OpenClaw. Ізольоване середовище виконання Node
не містить реалізацій Plugin, клієнтських об’єктів MCP або секретів.
`openclaw.tools.call(...)` переходить мостом назад у Gateway, де
й далі застосовуються звичайні політики, затвердження, хуки, журналювання та обробка результатів.

## Режими

`tools.toolSearch` має три режими, видимі моделі:

- `code`: надає `tool_search_code`, стандартний компактний міст JavaScript.
- `tools`: надає `tool_search`, `tool_describe` і `tool_call` як прості
  структуровані інструменти для провайдерів, які не повинні отримувати код.
- `directory`: надає `tool_search`, `tool_describe` і `tool_call`, а також
  обмежений prompt-каталог доступних назв і описів інструментів для
  провайдерів, які мають бачити назви інструментів без кожної повної схеми. OpenClaw також може
  напряму надавати невеликий обмежений набір імовірних або потрібних схем інструментів
  для поточного ходу.

Усі режими використовують той самий каталог, відфільтрований політикою, і звичайний шлях виконання OpenClaw.
Якщо поточне середовище виконання не може запустити ізольований дочірній процес Node для режиму коду,
стандартний режим `code` переходить на `tools` перед
Compaction каталогу. У режимі `directory` інструменти, надані клієнтом, залишаються напряму видимими
для поточного запуску, тоді як інструменти OpenClaw, інструменти Plugin та інструменти MCP можуть бути
скомпактовані за каталогом директорії. Прямий виклик точної прихованої
назви з директорії гідратується з того самого авторизованого каталогу перед виконанням.

Усі режими експериментальні. Для малих каталогів інструментів OpenClaw віддавайте перевагу прямому наданню інструментів,
а для запусків оснастки Codex — власним стабільним поверхням Codex.

Окремої конфігурації вибору джерел немає. Коли Пошук інструментів увімкнено,
каталог містить придатні інструменти OpenClaw, MCP і клієнтські інструменти після звичайної
фільтрації політиками.

## Навіщо це існує

Великі каталоги корисні, але дорогі. Передавання кожної схеми інструмента моделі
збільшує запит, уповільнює планування та підвищує ризик випадкового
вибору інструмента.

Пошук інструментів змінює форму:

- прямі інструменти: модель бачить кожну вибрану схему до першого токена
- режим коду Пошуку інструментів: модель бачить один компактний кодовий інструмент і короткий контракт API
- режим інструментів Пошуку інструментів: модель бачить три компактні структуровані резервні
  інструменти
- режим директорії Пошуку інструментів: модель бачить обмежену директорію плюс
  засоби search/describe/call і невеликий обмежений набір імовірних або потрібних
  схем
- під час ходу: модель може завантажувати решту схем за потреби

Пряме надання інструментів і далі є правильним стандартним варіантом для малих каталогів. Пошук інструментів
найкраще підходить, коли один запуск може бачити багато інструментів, особливо з серверів MCP або
інструментів застосунку, наданих клієнтом.

## API

`openclaw.tools.search(query, options?)`

Шукає в ефективному каталозі для поточного запуску. Результати компактні й безпечні
для повернення в контекст prompt.

```js
const hits = await openclaw.tools.search("calendar event", { limit: 5 });
```

`openclaw.tools.describe(id)`

Завантажує повні метадані для одного результату пошуку, включно з точною вхідною схемою.

```js
const calendarCreate = await openclaw.tools.describe("mcp:calendar:create_event");
```

`openclaw.tools.call(id, args)`

Викликає вибраний інструмент через OpenClaw.

```js
await openclaw.tools.call(calendarCreate.id, {
  summary: "Planning",
  start: "2026-05-09T14:00:00Z",
});
```

Структурований резервний режим надає ті самі операції як інструменти:

- `tool_search`
- `tool_describe`
- `tool_call`

Режим директорії надає:

- `tool_search`
- `tool_describe`
- `tool_call`

Він також залишає інструменти, надані клієнтом, напряму видимими та може напряму надавати невеликий
обмежений набір імовірних або потрібних схем інструментів каталогу для поточного
ходу. Якщо обмежена директорія пропускає записи, використовуйте `tool_search`, щоб їх знайти. Якщо
модель напряму запитує точну назву прихованого інструмента директорії, OpenClaw
гідратує його з авторизованого каталогу перед звичайним виконанням.
Назви клієнтських інструментів у режимі директорії не повинні конфліктувати з назвами інструментів OpenClaw, Plugin або MCP,
оскільки точна відкладена диспетчеризація використовує ці назви.

## Межа середовища виконання

Кодовий міст працює в короткоживучому підпроцесі Node. Підпроцес запускається
з увімкненим режимом дозволів Node, порожнім середовищем, без дозволів на файлову систему або
мережу, а також без дозволів на дочірні процеси чи worker-и. OpenClaw забезпечує
тайм-аут за настінним часом у батьківському процесі та вбиває підпроцес у разі тайм-ауту, зокрема
після асинхронних продовжень.

Середовище виконання надає лише:

- `console.log`, `console.warn` і `console.error`
- `openclaw.tools.search`
- `openclaw.tools.describe`
- `openclaw.tools.call`

Звичайна поведінка OpenClaw і далі застосовується до фінальних викликів:

- політики дозволу й заборони інструментів
- обмеження інструментів на рівні агента та пісочниці
- політика інструментів каналу/середовища виконання
- хуки затвердження
- хуки Plugin `before_tool_call`
- ідентичність сесії, журнали та телеметрія

## Конфігурація

Увімкніть Пошук інструментів для запусків OpenClaw зі стандартним кодовим мостом:

```bash
openclaw config set tools.toolSearch true
```

Еквівалентний JSON:

```json5
{
  tools: {
    toolSearch: true,
  },
}
```

Натомість використовуйте структуровані резервні інструменти для запусків OpenClaw:

```json5
{
  tools: {
    toolSearch: {
      mode: "tools",
    },
  },
}
```

Натомість використовуйте компактну поверхню директорії для запусків OpenClaw:

```json5
{
  tools: {
    toolSearch: {
      mode: "directory",
    },
  },
}
```

Налаштуйте тайм-аут режиму коду та ліміти результатів пошуку:

```json5
{
  tools: {
    toolSearch: {
      mode: "code",
      codeTimeoutMs: 10000,
      searchDefaultLimit: 8,
      maxSearchLimit: 20,
    },
  },
}
```

Вимкніть це:

```json5
{
  tools: {
    toolSearch: false,
  },
}
```

## Prompt і телеметрія

Пошук інструментів записує достатньо телеметрії, щоб порівнювати його з прямим наданням інструментів:

- загальна кількість серіалізованих байтів інструментів і prompt, надісланих до оснастки
- розмір каталогу та розподіл за джерелами
- кількість операцій search, describe і call
- фінальні виклики інструментів, виконані через OpenClaw
- вибрані ідентифікатори інструментів і джерела

Журнали сесії мають давати змогу відповісти:

- скільки схем інструментів модель побачила наперед
- скільки операцій search і describe вона виконала
- який фінальний інструмент було викликано
- чи результат надійшов від OpenClaw, MCP або клієнтського інструмента

## E2E-валідація

E2E-runner Gateway перевіряє обидва шляхи з середовищем виконання OpenClaw:

```bash
node --import tsx scripts/tool-search-gateway-e2e.ts
```

Він створює тимчасовий фальшивий Plugin із великим каталогом інструментів, запускає мок-провайдер
OpenAI, запускає Gateway один раз у прямому режимі й один раз із увімкненим Пошуком інструментів,
а потім порівнює payload-и запитів до провайдера та журнали сесій.

Регресія доводить:

1. Прямий режим може викликати інструмент фальшивого Plugin.
2. Пошук інструментів може викликати той самий інструмент фальшивого Plugin.
3. Прямий режим напряму надає схеми інструментів фальшивого Plugin провайдеру.
4. Пошук інструментів надає лише компактний міст.
5. Payload запиту Пошуку інструментів менший для великого фальшивого каталогу.
6. Журнали сесії показують очікувану кількість викликів інструментів і телеметрію мостових викликів.

## Поведінка в разі помилок

Пошук інструментів має відмовляти закрито:

- якщо інструмент не входить до ефективної політики, пошук не повинен його повертати
- якщо вибраний інструмент стає недоступним, `tool_call` має завершитися помилкою
- якщо політика або затвердження блокує виконання, результат виклику має повідомити про цей
  блок, а не обходити його
- якщо кодовий міст не може створити ізольоване середовище виконання, використовуйте `mode: "tools"` або
  вимкніть Пошук інструментів для цього розгортання
