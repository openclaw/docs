OpenClaw підтримує Grok як провайдера `web_search`, використовуючи відповіді xAI із прив’язкою до вебджерел для створення синтезованих ШІ відповідей на основі актуальних результатів пошуку з посиланнями на джерела.

Вебпошук Grok надає перевагу наявному входу через xAI OAuth, якщо він доступний. Якщо профілю OAuth немає, той самий API-ключ xAI також забезпечує роботу вбудованого інструмента `x_search` для пошуку дописів у X (раніше Twitter) та інструмента `code_execution`. Збереження ключа в `plugins.entries.xai.config.webSearch.apiKey` також дає OpenClaw змогу повторно використовувати його як резервний варіант для вбудованого провайдера моделей xAI.

Для отримання показників окремого допису X (репостів, відповідей, закладок, переглядів) використовуйте [`x_search`](/uk/tools/web#x_search) із точною URL-адресою допису або ідентифікатором статусу замість загального пошукового запиту.

## Початкове налаштування та конфігурація

Вибір **Grok** під час `openclaw onboard` або `openclaw configure --section
web` дає OpenClaw змогу повторно використати наявний профіль xAI OAuth без запиту окремого ключа для вебпошуку. За відсутності OAuth використовується налаштування API-ключа xAI як резервний варіант.

Після цього OpenClaw пропонує додатковий крок для ввімкнення `x_search` із тими самими обліковими даними xAI. Цей додатковий крок:

- з’являється лише після вибору Grok для `web_search`
- не є окремим варіантом вибору провайдера вебпошуку верхнього рівня
- дає змогу за бажанням установити модель `x_search` у тому самому процесі

Пропустіть його, щоб увімкнути або змінити `x_search` пізніше в конфігурації.

## Увійдіть або отримайте API-ключ

<Steps>
  <Step title="Використання xAI OAuth">
    Якщо ви вже ввійшли через xAI під час початкового налаштування або автентифікації моделі, виберіть Grok як провайдера `web_search`. Окремий API-ключ не потрібен:

    ```bash
    openclaw onboard --auth-choice xai-oauth
    openclaw config set tools.web.search.provider grok
    ```

  </Step>
  <Step title="Використання API-ключа як резервного варіанта">
    Отримайте API-ключ у [xAI](https://console.x.ai/), якщо OAuth недоступний або ви свідомо хочете використовувати конфігурацію вебпошуку на основі ключа.
  </Step>
  <Step title="Збереження ключа">
    Установіть `XAI_API_KEY` у середовищі Gateway або виконайте конфігурацію за допомогою:

    ```bash
    openclaw configure --section web
    ```

  </Step>
</Steps>

## Конфігурація

```json5
{
  plugins: {
    entries: {
      xai: {
        config: {
          webSearch: {
            apiKey: "xai-...", // необов’язково, якщо доступні xAI OAuth або XAI_API_KEY
            baseUrl: "https://api.x.ai/v1", // необов’язкове перевизначення проксі або базової URL-адреси Responses API
          },
        },
      },
    },
  },
  tools: {
    web: {
      search: {
        provider: "grok",
      },
    },
  },
}
```

**Альтернативні варіанти облікових даних:** `openclaw models auth login --provider xai
--method oauth`, `XAI_API_KEY` у середовищі Gateway або
`plugins.entries.xai.config.webSearch.apiKey`. Для інсталяції Gateway додайте змінні середовища до `~/.openclaw/.env`.

## Як це працює

Grok використовує відповіді xAI із прив’язкою до вебджерел для синтезу відповідей із вбудованими посиланнями на джерела, подібно до підходу Gemini із прив’язкою до Google Search.

## Підтримувані параметри

Пошук Grok підтримує `query`. Параметр `count` приймається для сумісності зі спільним інтерфейсом `web_search`, але Grok завжди повертає одну синтезовану відповідь із посиланнями на джерела, а не список із N результатів. Фільтри, специфічні для провайдера, не підтримуються.

Стандартний час очікування Grok становить 60 секунд, оскільки вебпошуки xAI Responses із прив’язкою до джерел можуть тривати довше, ніж стандартний час очікування спільного `web_search`. Перевизначте його за допомогою `tools.web.search.timeoutSeconds`.

## Перевизначення базової URL-адреси

Установіть `plugins.entries.xai.config.webSearch.baseUrl`, щоб спрямувати вебпошук Grok через операторський проксі-сервер або сумісну з xAI кінцеву точку Responses. OpenClaw надсилає POST-запити до `<baseUrl>/responses` після видалення кінцевих скісних рисок. `x_search` використовує той самий `webSearch.baseUrl` як резервний варіант, якщо не встановлено `plugins.entries.xai.config.xSearch.baseUrl`.
