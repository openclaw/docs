<Info>
  Це **посібник для учасників** для розробників ядра OpenClaw. Якщо ви
  створюєте зовнішній plugin, натомість дивіться [Building Plugins](/uk/plugins/building-plugins).
</Info>

Використовуйте це, коли OpenClaw потребує нового домену, такого як генерація зображень, генерація відео
або якась майбутня функціональна область, що підтримується постачальником.

Правило:

- plugin = межа відповідальності
- capability = спільний контракт ядра

Це означає, що не слід починати з прямого підключення постачальника до каналу або
інструмента. Починайте з визначення capability.

## Коли створювати capability

Створюйте нову capability, коли все наведене нижче є правдою:

1. її потенційно може реалізувати більше ніж один постачальник
2. канали, інструменти або функціональні pluginи повинні використовувати її, не зважаючи
   на постачальника
3. ядро має володіти поведінкою резервування, політики, конфігурації або доставки

Якщо робота стосується лише постачальника і спільний контракт ще не існує, зупиніться й спочатку визначте
контракт.

## Стандартна послідовність

1. Визначте типізований контракт ядра.
2. Додайте реєстрацію plugin для цього контракту.
3. Додайте спільний допоміжний компонент середовища виконання.
4. Підключіть один реальний plugin постачальника як підтвердження.
5. Переведіть споживачів функцій/каналів на допоміжний компонент середовища виконання.
6. Додайте тести контракту.
7. Задокументуйте операторську конфігурацію та модель відповідальності.

## Що куди належить

Ядро:

- типи запитів/відповідей
- реєстр провайдерів + резолюція
- поведінка резервування
- схема конфігурації плюс передані метадані документації `title` / `description` на вкладених об’єктах, wildcard, елементах масиву та вузлах композиції
- поверхня допоміжного компонента середовища виконання

Plugin постачальника:

- виклики API постачальника
- обробка автентифікації постачальника
- нормалізація запитів, специфічна для постачальника
- реєстрація реалізації capability

Функціональний plugin/plugin каналу:

- викликає `api.runtime.*` або відповідний допоміжний компонент `plugin-sdk/*-runtime`
- ніколи не викликає реалізацію постачальника безпосередньо

## Межі провайдера та harness

Використовуйте хуки провайдера, коли поведінка належить до контракту провайдера моделі,
а не до загального циклу агента. Приклади включають параметри запиту, специфічні для провайдера, після вибору транспорту,
перевагу auth-profile, накладання prompt і маршрутизацію подальшого резервування після перемикання
моделі/профілю.

Використовуйте хуки harness агента, коли поведінка належить до середовища виконання, яке
виконує хід. Harness можуть класифікувати успішні, але непридатні результати спроб,
такі як порожні відповіді, відповіді лише з міркуваннями або лише з плануванням, щоб зовнішня політика
резервування моделі могла прийняти рішення про повторну спробу.

Зберігайте обидві межі вузькими:

- ядро володіє політикою повторних спроб/резервування
- pluginи провайдерів володіють підказками для запитів/автентифікації/маршрутизації, специфічними для провайдера
- pluginи harness володіють класифікацією спроб, специфічною для середовища виконання
- сторонні pluginи повертають підказки, а не прямі мутації стану ядра

## Контрольний список файлів

Для нової capability очікуйте зміни в таких областях:

- `src/<capability>/types.ts`
- `src/<capability>/...registry/runtime.ts`
- `src/plugins/types.ts`
- `src/plugins/registry.ts`
- `src/plugins/captured-registration.ts`
- `src/plugins/contracts/registry.ts`
- `src/plugins/runtime/types-core.ts`
- `src/plugins/runtime/index.ts`
- `src/plugin-sdk/<capability>.ts`
- `src/plugin-sdk/<capability>-runtime.ts`
- один або кілька пакетів вбудованих pluginів
- config/docs/tests

## Приклад: генерація зображень

Генерація зображень дотримується стандартної форми:

1. ядро визначає `ImageGenerationProvider`
2. ядро надає `registerImageGenerationProvider(...)`
3. ядро надає `runtime.imageGeneration.generate(...)`
4. pluginи `openai`, `google`, `fal` і `minimax` реєструють реалізації, що підтримуються постачальниками
5. майбутні постачальники можуть реєструвати той самий контракт без змін у каналах/інструментах

Ключ конфігурації є окремим від маршрутизації аналізу зору:

- `agents.defaults.imageModel` = аналізувати зображення
- `agents.defaults.imageGenerationModel` = генерувати зображення

Зберігайте це розділення, щоб резервування та політика залишалися явними.

## Контрольний список перевірки

Перш ніж випускати нову capability, перевірте:

- жоден канал/інструмент не імпортує код постачальника безпосередньо
- допоміжний компонент середовища виконання є спільним шляхом
- принаймні один тест контракту підтверджує вбудовану відповідальність
- документація конфігурації називає новий ключ моделі/конфігурації
- документація pluginів пояснює межу відповідальності

Якщо PR пропускає шар capability і жорстко вбудовує поведінку постачальника в
канал/інструмент, поверніть його назад і спочатку визначте контракт.
