Plugins розширюють OpenClaw, додаючи канали, постачальників моделей, середовища виконання агентів, інструменти,
навички, мовлення, транскрибування в реальному часі, голосові функції, розуміння медіа, генерування,
отримання даних із вебу, вебпошук та інші можливості середовища виконання.

Скористайтеся цією сторінкою, щоб установити Plugin, перезапустити Gateway, перевірити, чи середовище виконання
завантажило його, і усунути поширені помилки налаштування. Приклади лише з командами див. у розділі
[Керування плагінами](/uk/plugins/manage-plugins). Згенерований перелік
вбудованих, офіційних зовнішніх і доступних лише у вихідному коді плагінів див. у розділі
[Перелік плагінів](/uk/plugins/plugin-inventory).

## Вимоги

- робоча копія або інсталяція OpenClaw із доступним CLI `openclaw`
- доступ до мережі для вибраного джерела (ClawHub, npm або git-хост)
- усі облікові дані, ключі конфігурації або інструменти ОС, потрібні конкретному плагіну й зазначені
  в документації з його налаштування
- дозвіл для Gateway, який обслуговує ваші канали, на перезавантаження або перезапуск

## Швидкий початок

<Steps>
  <Step title="Знайдіть Plugin">
    Знайдіть загальнодоступні пакети плагінів у [ClawHub](/clawhub):

    ```bash
    openclaw plugins search "calendar"
    ```

    ClawHub — основне місце пошуку плагінів спільноти. Під час
    переходу після запуску звичайні специфікації пакетів без префікса й надалі встановлюються з npm, якщо
    вони не відповідають ідентифікатору офіційного плагіна. Необроблені специфікації `@openclaw/*`, що відповідають
    вбудованому плагіну, вказують на його вбудовану копію. Використовуйте явний префікс джерела,
    коли потрібне конкретне джерело.

  </Step>

  <Step title="Установіть Plugin">
    ```bash
    # Із ClawHub.
    openclaw plugins install clawhub:<package>

    # Із npm.
    openclaw plugins install npm:<package>

    # Із git.
    openclaw plugins install git:github.com/<owner>/<repo>@<ref>

    # Із локальної робочої копії для розробки.
    openclaw plugins install ./my-plugin
    openclaw plugins install --link ./my-plugin
    ```

    Ставтеся до встановлення плагінів як до запуску коду. Для
    відтворюваних виробничих інсталяцій віддавайте перевагу зафіксованим версіям. Пакети ClawHub і
    вбудований/офіційний каталог OpenClaw є довіреними джерелами. Нові довільні джерела npm, git,
    локального шляху/архіву, `npm-pack:` або маркетплейсу потребують
    `--force` під час неінтерактивного встановлення після того, як ви
    перевірите джерело й переконаєтеся, що йому можна довіряти.

  </Step>

  <Step title="Налаштуйте й увімкніть його">
    Налаштуйте специфічні для плагіна параметри в `plugins.entries.<id>.config`.
    Увімкніть Plugin, якщо його ще не ввімкнено:

    ```bash
    openclaw plugins enable <plugin-id>
    ```

    Якщо задано `plugins.allow`, ідентифікатор установленого плагіна має бути в цьому списку,
    перш ніж Plugin зможе завантажитися. `openclaw plugins install` додає встановлений
    ідентифікатор до наявного списку `plugins.allow` і вилучає той самий ідентифікатор із
    `plugins.deny`, щоб явно встановлений Plugin міг завантажитися після перезапуску.

  </Step>

  <Step title="Дозвольте Gateway перезавантажитися">
    Установлення, оновлення або видалення коду плагіна потребує перезапуску Gateway.
    Керований Gateway з увімкненим перезавантаженням конфігурації виявляє зміну
    запису про встановлення плагіна й автоматично перезапускається. В іншому разі перезапустіть його
    самостійно:

    ```bash
    openclaw gateway restart
    ```

    Увімкнення або вимкнення оновлює конфігурацію та холодний реєстр. Перевірка середовища виконання
    усе ще є найнаочнішим підтвердженням активних поверхонь середовища виконання.

  </Step>

  <Step title="Перевірте реєстрацію в середовищі виконання">
    ```bash
    openclaw plugins inspect <plugin-id> --runtime --json
    ```

    Використовуйте `--runtime`, щоб підтвердити реєстрацію інструментів, обробників, служб, методів Gateway
    або команд CLI, які належать плагіну. Звичайна команда `inspect` виконує лише холодну перевірку маніфесту
    та реєстру.

  </Step>
</Steps>

## Конфігурація

### Виберіть джерело встановлення

| Джерело      | Коли використовувати                                                                       | Приклад                                                        |
| ----------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| ClawHub     | Потрібні нативний для OpenClaw пошук, сканування, метадані версій і підказки щодо встановлення | `openclaw plugins install clawhub:<package>`                   |
| npm         | Потрібні безпосередні робочі процеси з реєстром npm або dist-tag                             | `openclaw plugins install npm:<package>`                       |
| git         | Потрібна гілка, тег або коміт із репозиторію                            | `openclaw plugins install git:github.com/<owner>/<repo>@<ref>` |
| локальний шлях  | Ви розробляєте або тестуєте плагін на тому самому комп’ютері                     | `openclaw plugins install --link ./my-plugin`                  |
| маркетплейс | Ви встановлюєте сумісний із Claude плагін маркетплейсу                      | `openclaw plugins install <plugin> --marketplace <source>`     |

Специфікації пакетів без уточнень мають особливу поведінку сумісності: ім’я без уточнень, яке
збігається з ідентифікатором вбудованого плагіна, використовує це вбудоване джерело; ім’я без уточнень, яке збігається
з ідентифікатором офіційного зовнішнього плагіна, використовує офіційний каталог пакетів; будь-яка інша
специфікація без уточнень під час переходу на запуск установлюється через npm. Необроблені специфікації `@openclaw/*`,
які збігаються з вбудованими плагінами, також перед резервним переходом до npm
зіставляються з вбудованою копією. Використовуйте `npm:@openclaw/<plugin>@<version>`, щоб навмисно встановити
зовнішній пакет npm замість вбудованої копії. Використовуйте `clawhub:`, `npm:`,
`git:` або `npm-pack:` для детермінованого вибору джерела. Повний контракт команди див.
у розділі [`openclaw plugins`](/uk/cli/plugins#install).

Для встановлень із npm незакріплені специфікації та `@latest` вибирають найновіший стабільний
пакет, який заявляє сумісність із цією збіркою OpenClaw. Якщо поточний
найновіший випуск npm оголошує новіший `openclaw.compat.pluginApi` або
`openclaw.install.minHostVersion`, ніж підтримує ця збірка, OpenClaw сканує
старіші стабільні версії та встановлює найновішу відповідну. Точні версії
та явні теги каналів, як-от `@beta`, залишаються закріпленими за вибраним пакетом
і завершуються помилкою в разі несумісності.

### Політика встановлення оператора

Налаштуйте `security.installPolicy`, щоб перед продовженням встановлення або оновлення плагіна
запускалася довірена локальна команда політики. Політика отримує метадані разом
зі шляхом до підготовленого джерела й може дозволити або заблокувати встановлення. Вона охоплює
як CLI, так і шляхи встановлення та оновлення через Gateway. Хуки плагіна `before_install` запускаються
пізніше й лише в процесах OpenClaw, де завантажено хуки плагінів, тому для
рішень щодо встановлення, якими керує оператор, натомість використовуйте
`security.installPolicy`. Застарілий прапорець `--dangerously-force-unsafe-install`
приймається для сумісності, але нічого не робить: він не обходить політику встановлення чи
вбудований у OpenClaw список заборонених залежностей плагінів.

Спільну схему виконання `security.installPolicy`, яку використовують і Skills, і
плагіни, див. у розділі [Налаштування Skills](/uk/tools/skills-config#operator-install-policy-securityinstallpolicy).

### Налаштуйте політику плагінів

Загальна структура конфігурації плагінів:

```json5
{
  plugins: {
    enabled: true,
    allow: ["voice-call"],
    deny: ["untrusted-plugin"],
    load: { paths: ["~/Projects/oss/voice-call-plugin"] },
    slots: { memory: "memory-core" },
    entries: {
      "voice-call": { enabled: true, config: { provider: "twilio" } },
    },
  },
}
```

Основні правила політики:

- `plugins.enabled: false` вимикає всі плагіни та пропускає роботу з виявлення й завантаження.
  Застарілі посилання на плагіни залишаються неактивними, доки цей параметр діє; знову ввімкніть
  плагіни перед очищенням за допомогою doctor, якщо потрібно видалити застарілі ідентифікатори.
- `plugins.deny` має пріоритет над списком дозволів і ввімкненням окремого плагіна.
- `plugins.allow` — це винятковий список дозволів. Інструменти, що належать плагінам поза
  списком дозволів, залишаються недоступними, навіть коли `tools.allow` містить `"*"`.
- `plugins.entries.<id>.enabled: false` вимикає один плагін, зберігаючи його
  конфігурацію.
- `plugins.load.paths` додає явні локальні файли або каталоги плагінів.
  Керовані локальні шляхи `plugins install` мають бути каталогами або
  архівами плагінів; для окремих файлів плагінів використовуйте `plugins.load.paths`.
- Плагіни з робочого простору за замовчуванням вимкнені; явно ввімкніть або
  додайте їх до списку дозволів перед використанням локального коду робочого простору.
- Вбудовані плагіни дотримуються вбудованих метаданих про ввімкнення або вимкнення
  за замовчуванням, якщо конфігурація явно не перевизначає їх.
- `plugins.slots.<slot>` (`memory` або `contextEngine`) вибирає один плагін для
  виняткової категорії. Вибір слота вважається явною активацією та
  примусово вмикає вибраний плагін для цього слота, навіть якщо інакше
  він потребував би явного ввімкнення. `plugins.deny` і `plugins.entries.<id>.enabled: false` усе одно
  блокують його.
- Вбудовані плагіни з явним ввімкненням можуть активуватися автоматично, коли конфігурація називає одну з
  належних їм поверхонь, як-от посилання на постачальника або модель, конфігурацію каналу, серверну частину CLI
  чи середовище виконання каркаса агента.
- Маршрутизація Codex родини OpenAI зберігає межі плагінів постачальника й середовища виконання
  окремими: застарілі посилання на моделі Codex є застарілою конфігурацією, яку виправляє doctor,
  тоді як вбудований плагін `codex` керує середовищем виконання сервера застосунку Codex для
  канонічних посилань агентів `openai/*`, явного `agentRuntime.id: "codex"` і
  застарілих посилань `codex/*`.

Коли `plugins.allow` не задано, а невбудовані плагіни автоматично виявляються в
робочому просторі або глобальних кореневих каталогах плагінів, під час запуску журналюється
`plugins.allow is empty; discovered non-bundled plugins may auto-load: ...`
з ідентифікаторами виявлених плагінів і, для коротких списків, мінімальним фрагментом `plugins.allow`.
Запустіть [`openclaw plugins list --enabled --verbose`](/uk/cli/plugins#list)
або [`openclaw plugins inspect <id>`](/uk/cli/plugins#inspect) для вказаного
ідентифікатора плагіна, перш ніж копіювати довірені плагіни до `openclaw.json`. Таке саме
закріплення довіри застосовується, коли діагностика повідомляє, що плагін завантажено
`without install/load-path provenance`: перевірте цей ідентифікатор плагіна, а потім закріпіть його в
`plugins.allow` або повторно встановіть із довіреного джерела, щоб OpenClaw записав
походження встановлення.

Запустіть `openclaw doctor` або `openclaw doctor --fix`, коли перевірка конфігурації
повідомляє про застарілі ідентифікатори плагінів, невідповідності списку дозволів та інструментів або застарілі шляхи
вбудованих плагінів.

## Формати плагінів

OpenClaw розпізнає два формати плагінів:

| Формат                 | Як завантажується                                                                 | Коли використовувати                                                               |
| ---------------------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Нативний плагін OpenClaw | `openclaw.plugin.json` разом із модулем середовища виконання, завантаженим у процес               | Ви встановлюєте або створюєте специфічні для OpenClaw можливості середовища виконання  |
| Сумісний пакет      | Макет плагіна Codex, Claude або Cursor, зіставлений з інвентарем плагінів OpenClaw | Ви повторно використовуєте сумісні навички, команди, хуки або метадані пакетів |

Обидва формати відображаються в `openclaw plugins list`, `openclaw plugins inspect`,
`openclaw plugins enable` і `openclaw plugins disable`. Межі сумісності пакетів див.
у розділі [Пакети плагінів](/uk/plugins/bundles), а створення нативних плагінів —
у розділі [Створення плагінів](/uk/plugins/building-plugins).

## Хуки плагінів

Плагіни можуть реєструвати хуки під час виконання через два різні API:

- `api.on(...)` — типізовані хуки для подій життєвого циклу середовища виконання. Це
  рекомендована поверхня для проміжного ПЗ, політик, переписування повідомлень, формування
  запитів та керування інструментами.
- `api.registerHook(...)` — для внутрішньої системи хуків, описаної в розділі
  [Хуки](/uk/automation/hooks). Вона призначена переважно для грубих побічних ефектів команд і життєвого циклу
  та сумісності з наявною автоматизацією у стилі HOOK.

Коротке правило: якщо обробнику потрібні пріоритет, семантика об’єднання або
поведінка блокування чи скасування, використовуйте типізовані хуки. Якщо він лише реагує на `command:new`,
`command:reset`, `message:sent` або подібні грубі події, `api.registerHook`
цілком підходить.

Внутрішні хуки, якими керують плагіни, відображаються в `openclaw hooks list` з
`plugin:<id>`. Їх неможливо ввімкнути або вимкнути через `openclaw hooks`;
натомість увімкніть або вимкніть плагін.

## Перевірте активний Gateway

`openclaw plugins list` і звичайна команда `openclaw plugins inspect` зчитують холодний стан конфігурації,
маніфесту та реєстру. Вони не підтверджують, що вже запущений
Gateway імпортував той самий код плагіна.

Якщо плагін видається встановленим, але трафік активних чатів його не використовує:

```bash
openclaw gateway status --deep --require-rpc
openclaw plugins inspect <plugin-id> --runtime --json
openclaw gateway restart
```

Керовані Gateway автоматично перезапускаються після встановлення, оновлення та
видалення плагінів, якщо ці зміни модифікують вихідний код плагіна. Для встановлень на VPS або в контейнері
переконайтеся, що будь-який ручний перезапуск спрямовано на фактичний дочірній процес `openclaw gateway run`,
який обслуговує ваші канали, а не лише на оболонку чи супервізор.

## Усунення несправностей

| Ознака                                                        | Перевірка                                                                                                                                      | Виправлення                                                                                                     |
| -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------- |
| Плагін відображається в `plugins list`, але обробники середовища виконання не запускаються  | Скористайтеся `openclaw plugins inspect <id> --runtime --json` і підтвердьте активний Gateway за допомогою `gateway status --deep --require-rpc`             | Перезапустіть активний Gateway після встановлення, оновлення, зміни конфігурації або вихідного коду                               |
| З’являється діагностика дублювання власників каналів або інструментів         | Запустіть `openclaw plugins list --enabled --verbose`, перевірте кожен підозрілий плагін за допомогою `--runtime --json` і порівняйте власників каналів та інструментів | Вимкніть одного власника, видаліть застарілі встановлення або використайте `preferOver` у маніфесті для навмисної заміни      |
| У конфігурації вказано, що плагін відсутній                                | Перевірте в [переліку плагінів](/uk/plugins/plugin-inventory), чи є він вбудованим, офіційним зовнішнім або доступним лише як вихідний код                           | Установіть зовнішній пакет, увімкніть вбудований плагін або видаліть застарілу конфігурацію                         |
| Конфігурація недійсна під час встановлення                               | Прочитайте повідомлення перевірки та запустіть `openclaw doctor --fix`, якщо воно вказує на застарілий стан плагіна                                             | Doctor може ізолювати недійсну конфігурацію плагіна, вимкнувши запис і видаливши недійсне корисне навантаження     |
| Шлях до плагіна заблоковано через підозрілого власника або дозволи | Перевірте діагностичне повідомлення перед помилкою конфігурації                                                                                             | Виправте власника й дозволи файлової системи, а потім запустіть `openclaw plugins registry --refresh`                    |
| `OPENCLAW_NIX_MODE=1` блокує команди життєвого циклу                | Переконайтеся, що встановленням керує Nix                                                                                                      | Змініть вибір плагіна у вихідному коді Nix замість використання команд-мутаторів плагіна                      |
| Імпорт залежності завершується помилкою під час виконання                             | Перевірте, чи плагін було встановлено через npm/git/ClawHub або завантажено з локального шляху                                                 | Запустіть `openclaw plugins update <id>`, повторно встановіть джерело або самостійно встановіть залежності локального плагіна |

Якщо застаріла конфігурація плагіна все ще містить назву плагіна каналу, який більше неможливо виявити,
перевірка конфігурації понижує помилку цього ключа каналу до попередження замість критичної
помилки, тому під час запуску Gateway може й надалі обслуговувати всі інші канали. Запустіть
`openclaw doctor --fix`, щоб видалити застарілі записи плагінів і каналів. Невідомі
ключі каналів без ознак застарілого плагіна все одно не проходять перевірку, щоб
друкарські помилки залишалися помітними.

Для навмисної заміни каналу бажаний плагін має оголосити
`channelConfigs.<channel-id>.preferOver` з ідентифікатором застарілого плагіна або плагіна з нижчим пріоритетом.
Якщо обидва плагіни явно ввімкнено, OpenClaw зберігає цей запит
і повідомляє про дублювання власників каналів або інструментів, а не вибирає
одного власника без попередження.

Якщо встановлений пакет повідомляє, що він `requires compiled runtime output for
TypeScript entry ...`, пакет опубліковано без файлів JavaScript,
необхідних OpenClaw під час виконання. Оновіть або повторно встановіть його після того, як видавець надасть
скомпільований JavaScript, або до того часу вимкніть чи видаліть плагін.

### Заблокований власник шляху до плагіна

Якщо діагностика повідомляє
`blocked plugin candidate: suspicious ownership (... uid=1000, expected uid=0 or root)`,
а потім перевірка видає `plugin present but blocked`, OpenClaw знайшов
файли плагіна, що належать іншому користувачеві Unix, а не процесу, який їх завантажує.
Залиште конфігурацію плагіна без змін; виправте власника файлової системи або запускайте OpenClaw
від імені того самого користувача, якому належить каталог стану.

Для встановлень Docker офіційний образ працює від імені `node` (uid `1000`), тому
примонтовані з хоста каталоги конфігурації та робочого простору OpenClaw зазвичай мають
належати uid `1000`:

```bash
sudo chown -R 1000:1000 /path/to/openclaw-config /path/to/openclaw-workspace
```

Якщо ви навмисно запускаєте OpenClaw від імені root, натомість змініть
власника кореневого каталогу керованих плагінів на root:

```bash
sudo chown -R root:root /path/to/openclaw-config/npm
```

Після виправлення власника повторно запустіть `openclaw doctor --fix` або
`openclaw plugins registry --refresh`, щоб збережений реєстр плагінів
відповідав виправленим файлам.

### Повільне налаштування інструментів плагіна

Якщо під час підготовки інструментів виконання агента ніби призупиняється, увімкніть журналювання трасування
та перевірте рядки тривалості роботи фабрик інструментів плагінів:

```bash
openclaw config set logging.level trace
openclaw logs --follow
```

Знайдіть:

```text
[trace:plugin-tools] тривалість роботи фабрик ...
```

У зведенні наведено загальний час роботи фабрик і найповільніші фабрики інструментів плагінів,
зокрема ідентифікатор плагіна, оголошені назви інструментів, форму результату й те, чи є інструмент
необов’язковим. Повільні рядки підвищуються до рівня попереджень, якщо одна фабрика працює
принаймні 1s або загальна підготовка фабрик інструментів плагінів триває принаймні 5s.

OpenClaw кешує успішні результати фабрик інструментів плагінів для повторних
визначень з однаковим фактичним контекстом запиту. Ключ кешу містить
фактичну конфігурацію середовища виконання, робочий простір та ідентифікатор агента, політику пісочниці, налаштування
браузера, контекст доставки, ідентичність запитувача й стан власності, тому
фабрики, що залежать від цих довірених полів, запускаються повторно, коли контекст
змінюється. Якщо тривалість залишається високою, плагін може виконувати витратну роботу до
повернення визначень своїх інструментів.

Якщо один плагін займає більшу частину часу, перевірте його реєстрації в середовищі виконання:

```bash
openclaw plugins inspect <plugin-id> --runtime --json
```

Потім оновіть, повторно встановіть або вимкніть цей плагін. Автори плагінів мають перенести
витратне завантаження залежностей у шлях виконання інструмента, а не виконувати
його всередині фабрики інструментів.

Докладніше про кореневі каталоги залежностей, перевірку метаданих пакетів, записи реєстру, поведінку
перезавантаження під час запуску й очищення застарілих даних див. у розділі
[Визначення залежностей плагінів](/uk/plugins/dependency-resolution).
