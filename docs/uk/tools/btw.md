`/btw` дає змогу поставити швидке побічне запитання про **поточну сесію** без
перетворення цього запитання на звичайну історію розмови.

Цю функцію змодельовано за зразком поведінки `/btw` у Claude Code, але адаптовано до
Gateway OpenClaw і багатоканальної архітектури.

## Що це робить

Коли ви надсилаєте:

```text
/btw what changed?
```

OpenClaw:

1. робить snapshot поточного контексту сесії,
2. запускає окремий **безінструментальний** виклик моделі,
3. відповідає лише на побічне запитання,
4. не змінює основний запуск,
5. **не** записує запитання або відповідь BTW в історію сесії,
6. надсилає відповідь як **живий побічний результат**, а не як звичайне повідомлення assistant.

Важлива ментальна модель:

- той самий контекст сесії
- окремий одноразовий побічний запит
- без викликів інструментів
- без забруднення майбутнього контексту
- без збереження transcript

## Чого це не робить

`/btw` **не**:

- створює нову довготривалу сесію,
- продовжує незавершене основне завдання,
- запускає інструменти або цикли інструментів агента,
- записує дані запитання/відповіді BTW в історію transcript,
- з’являється в `chat.history`,
- переживає перезавантаження.

Це навмисно **епемерна** функція.

## Як працює контекст

BTW використовує поточну сесію **лише як фоновий контекст**.

Якщо основний запуск зараз активний, OpenClaw робить snapshot поточного стану
повідомлень і включає незавершений основний prompt як фоновий контекст, водночас
явно повідомляючи моделі:

- відповідати лише на побічне запитання,
- не відновлювати й не завершувати незавершене основне завдання,
- не надсилати виклики інструментів або псевдовиклики інструментів.

Це зберігає ізоляцію BTW від основного запуску, водночас даючи йому розуміння
того, про що ця сесія.

## Модель доставки

BTW **не** доставляється як звичайне transcript-повідомлення assistant.

На рівні протоколу Gateway:

- звичайний assistant chat використовує подію `chat`
- BTW використовує подію `chat.side_result`

Це розділення навмисне. Якби BTW повторно використовував звичайний шлях подій `chat`,
клієнти трактували б його як звичайну історію розмови.

Оскільки BTW використовує окрему live-подію і не відтворюється з
`chat.history`, після перезавантаження він зникає.

## Поведінка на поверхнях

### TUI

У TUI BTW рендериться в поточному поданні сесії inline, але залишається
епемерним:

- візуально відрізняється від звичайної відповіді assistant
- можна закрити клавішами `Enter` або `Esc`
- не відтворюється після перезавантаження

### Зовнішні канали

У таких каналах, як Telegram, WhatsApp і Discord, BTW доставляється як
чітко позначена одноразова відповідь, оскільки ці поверхні не мають локальної
концепції епемерного накладання.

Відповідь усе одно розглядається як побічний результат, а не як звичайна історія сесії.

### Control UI / web

Gateway правильно надсилає BTW як `chat.side_result`, і BTW не включається
в `chat.history`, тож контракт збереження для web уже правильний.

Поточний Control UI все ще потребує окремого споживача `chat.side_result`, щоб
рендерити BTW у браузері в live-режимі. Поки ця підтримка на стороні клієнта не з’явиться, BTW є функцією рівня Gateway з повною поведінкою в TUI та зовнішніх каналах, але ще не має повноцінного UX у браузері.

## Коли використовувати BTW

Використовуйте `/btw`, коли вам потрібні:

- швидке уточнення щодо поточної роботи,
- фактична побічна відповідь, поки довгий запуск усе ще триває,
- тимчасова відповідь, яка не повинна ставати частиною майбутнього контексту сесії.

Приклади:

```text
/btw what file are we editing?
/btw what does this error mean?
/btw summarize the current task in one sentence
/btw what is 17 * 19?
```

## Коли не варто використовувати BTW

Не використовуйте `/btw`, якщо ви хочете, щоб відповідь стала частиною
майбутнього робочого контексту сесії.

У такому разі ставте запитання звичайним способом у main session замість BTW.
