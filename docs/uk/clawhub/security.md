# Безпека + модерація

ClawHub відкритий до публікації, але публічні списки все одно проходять через засоби контролю довіри,
сканування, звітування та модерації. Мета практична: допомогти користувачам
перевіряти те, що вони встановлюють, дати видавцям шлях відновлення після хибних спрацювань
і не допускати зловживальних пакетів до публічного пошуку.

Див. також [Прийнятне використання](/uk/clawhub/acceptable-usage).

## Що користувачі можуть перевірити

Перед встановленням skill або plugin перевірте його список у ClawHub на наявність:

- власника та атрибуції джерела
- останньої версії та журналу змін
- потрібних змінних середовища або дозволів
- метаданих сумісності для plugins
- статусу сканування або модерації
- звітів, коментарів, зірок, завантажень і сигналів встановлення, де вони показані

Встановлюйте лише той вміст, який ви розумієте і якому довіряєте.

## Стани сканування

ClawHub може показувати результати сканування або модерації на публічних сторінках і в діагностиці,
видимій власнику.

Поширені результати включають:

- `clean`: блокувальної проблеми не знайдено.
- `suspicious`: реліз потребує обережності або перевірки.
- `malicious`: реліз вважається небезпечним.
- `pending`: перевірки ще не завершено.
- `held`, `quarantined`, `revoked` або `hidden`: реліз не повністю
  доступний на публічних поверхнях встановлення.

Точне формулювання може відрізнятися залежно від поверхні, але практичний зміст той самий: якщо
реліз утримано або заблоковано, користувачам не слід встановлювати його, доки власник не вирішить
проблему або модерація не відновить його.

## Skills

Сканування Skills перевіряє опублікований пакет skill, метадані, заявлені
вимоги та підозрілі інструкції.

ClawHub приділяє особливу увагу невідповідностям між тим, що skill заявляє, і
тим, що він, схоже, робить. Наприклад, skill, який посилається на потрібний ключ API,
має заявити цю вимогу в `SKILL.md`, щоб користувачі могли побачити її перед
встановленням.

Висновки сканування ґрунтуються на артефактах. Очікувана поведінка провайдера, така як заявлені
облікові дані API, OAuth-зворотні виклики localhost, очищення під час видалення в межах scope, кодування Basic Auth
або вибрані користувачем завантаження файлів до заявленого провайдера, розглядається
інакше, ніж приховане пересилання облікових даних, широкий доступ до приватних файлів,
непов’язані мережеві призначення або приховане зловживання браузером.

Див. [Формат skill](/uk/clawhub/skill-format).

## Plugins

Релізи Plugin включають метадані пакета, атрибуцію джерела, поля сумісності
та інформацію про цілісність артефактів.

OpenClaw перевіряє сумісність перед встановленням plugins, розміщених у ClawHub. Записи пакетів
також можуть надавати метадані digest, щоб OpenClaw міг перевіряти завантажені
артефакти. ClawScan включає заявлені метадані env/config пакета `openclaw.environment`
під час перевірки релізів plugin, щоб заявлені вимоги runtime
порівнювалися зі спостережуваною поведінкою.

## Звіти

Користувачі, які ввійшли в систему, можуть повідомляти про skills, пакети та коментарі.

Звіти мають бути конкретними й придатними до дії. Зловживання звітуванням саме по собі може призвести до
дій щодо облікового запису.

Приклади звітів:

- оманливі метадані
- незаявлені вимоги до облікових даних або дозволів
- підозрілі інструкції зі встановлення
- шахрайські коментарі або видавання себе за іншу особу
- недобросовісні реєстрації або неправомірне використання торговельних марок
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

## Примітки ClawScan від видавця

Видавці можуть надати необов’язкову примітку ClawScan під час публікації skill або
plugin. Ця примітка дає ClawScan контекст для поведінки, яка інакше може виглядати
незвичною, наприклад мережевого доступу, доступу до native host або специфічних для провайдера
облікових даних.

## Модераційні утримання

Коли статичний сканер позначає завантажений skill як зловмисний, видавець
автоматично потрапляє під модераційне утримання (`requiresModerationAt`, встановлене для
користувача). Це приховує всі skills видавця, змушує майбутні публікації
починатися прихованими та створює запис журналу аудиту `user.moderation.auto`.

Статичні підозрілі висновки зберігаються як докази файл/рядок для модераторів,
але самі по собі вони не приховують вміст і не визначають публічний вердикт сканування.
Нові завантаження залишаються у стані review/pending, доки не завершиться LLM-перевірка. Статичне
сканування негайно блокує лише зловмисні сигнатури. Спрацювання рушія VirusTotal
залишаються видимими доказами безпеки, але вердикти VirusTotal Code Insight/Palm
є рекомендаційними й самі по собі не приховують skills. LLM-перевірки ClawScan
зберігають примітки, узгоджені з призначенням, як настанови. Висновки перевірки середнього рівня залишаються видимими на
артефакті, тоді як фільтр підозрілості зарезервований для LLM-занепокоєнь із великим впливом,
зловмисних висновків або підтверджених виявлень AV-рушієм.

Адміністратори можуть зняти утримання через хибне спрацювання:

```bash
npx convex run users:liftModerationHold '{"userId": "<user-id>", "reason": "False positive from security tool scanning"}'
```

Це очищає `requiresModerationAt` і `requiresModerationReason`, відновлює
skills, приховані утриманням на рівні користувача, і записує запис журналу аудиту `user.moderation.lift`.
Skills, приховані з інших причин, або ті, чий власний статичний scan залишається
зловмисним, залишаються прихованими.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання
можуть призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених
списків.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збої після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан облікового
запису. Якщо вхід або звичайний доступ CLI заблоковано, зверніться до
security@openclaw.ai для перевірки відновлення.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, підсумків, тегів і журналів змін
- заявляйте потрібні змінні середовища та дозволи
- додавайте примітку ClawScan від видавця, коли реліз має незвичну, але навмисну поведінку
- уникайте обфускованих команд встановлення
- посилайтеся на джерело, коли можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку пакета
