# Безпека + модерація

ClawHub відкритий для публікації, але публічні списки все одно проходять через перевірки довіри,
сканування, звітування та модераційні контролі. Мета практична: допомогти користувачам
перевіряти те, що вони встановлюють, дати видавцям шлях відновлення після хибних спрацювань
і не допускати зловживальні пакунки до публічного пошуку.

Див. також [Прийнятне використання](/uk/clawhub/acceptable-usage).

## Що можуть перевірити користувачі

Перед встановленням skill або plugin перевірте його список у ClawHub на наявність:

- власника та атрибуції джерела
- останньої версії та журналу змін
- потрібних змінних середовища або дозволів
- метаданих сумісності для plugins
- статусу сканування або модерації
- звітів, коментарів, зірок, завантажень і сигналів встановлення, якщо вони показані

Встановлюйте лише той вміст, який ви розумієте і якому довіряєте.

## Стани сканування

ClawHub може показувати результати сканування або модерації на публічних сторінках і в
діагностиці, видимій власнику.

Поширені результати:

- `clean`: блокувальної проблеми не знайдено.
- `suspicious`: реліз потребує обережності або перевірки.
- `malicious`: реліз вважається небезпечним.
- `pending`: перевірки ще не завершилися.
- `held`, `quarantined`, `revoked` або `hidden`: реліз не повністю
  доступний на публічних поверхнях встановлення.

Точне формулювання може відрізнятися залежно від поверхні, але практичний зміст той самий: якщо
реліз утримано або заблоковано, користувачам не слід встановлювати його, доки власник не розв’яже
проблему або модерація не відновить реліз.

## Skills

Сканування skill перевіряє опублікований пакет skill, метадані, заявлені
вимоги та підозрілі інструкції.

ClawHub приділяє особливу увагу невідповідностям між тим, що skill декларує, і
тим, що він, схоже, робить. Наприклад, skill, який посилається на потрібний API-ключ,
має задекларувати цю вимогу в `SKILL.md`, щоб користувачі могли побачити її перед
встановленням.

Висновки сканування базуються на артефактах. Очікувана поведінка провайдера, як-от задекларовані
API-облікові дані, localhost OAuth callbacks, обмежене очищення під час видалення, кодування Basic Auth
або вибрані користувачем завантаження файлів до заявленого провайдера, розглядається
інакше, ніж приховане пересилання облікових даних, широкий доступ до приватних файлів,
непов’язані мережеві призначення або приховане зловживання браузером.

Див. [Формат skill](/uk/clawhub/skill-format).

## Plugins

Релізи Plugin містять метадані пакунка, атрибуцію джерела, поля сумісності
та інформацію про цілісність артефактів.

OpenClaw перевіряє сумісність перед встановленням розміщених у ClawHub plugins. Записи пакунків
також можуть надавати метадані дайджесту, щоб OpenClaw міг перевіряти завантажені
артефакти. ClawScan включає задекларовані метадані env/config пакунка `openclaw.environment`
під час перевірки релізів plugin, щоб заявлені runtime-вимоги
порівнювалися зі спостережуваною поведінкою.

## Звіти

Користувачі, які ввійшли в систему, можуть повідомляти про skills, пакунки та коментарі.

Звіти мають бути конкретними й придатними до дії. Зловживання звітуванням саме по собі може призвести до
дій щодо облікового запису.

Приклади звітів:

- оманливі метадані
- незадекларовані вимоги до облікових даних або дозволів
- підозрілі інструкції встановлення
- шахрайські коментарі або видавання себе за іншу особу
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

## Нотатки ClawScan від видавця

Видавці можуть надати необов’язкову нотатку ClawScan під час публікації skill або
plugin. Ця нотатка дає ClawScan контекст для поведінки, яка інакше може виглядати
незвичною, як-от мережевий доступ, доступ до нативного хоста або специфічні для провайдера
облікові дані.

## Модераційні утримання

Коли статичний сканер позначає завантажений skill як malicious, видавець
автоматично потрапляє під модераційне утримання (`requiresModerationAt` встановлюється для
користувача). Це приховує всі skills видавця, змушує майбутні публікації
починатися прихованими та створює запис журналу аудиту `user.moderation.auto`.

Статичні підозрілі висновки зберігаються як докази файл/рядок для модераторів,
але самі по собі вони не приховують вміст і не визначають публічний вердикт сканування.
Нові завантаження залишаються у стані review/pending, доки не завершиться LLM-перевірка. Статичне
сканування негайно блокує лише malicious signatures. Спрацювання рушіїв VirusTotal
залишаються видимими доказами безпеки, але вердикти VirusTotal Code Insight/Palm
є консультативними й самі по собі не приховують skills. LLM-перевірки ClawScan
зберігають purpose-aligned нотатки як настанови. Medium-висновки перевірки залишаються видимими на
артефакті, тоді як suspicious filter зарезервований для високовпливових LLM
занепокоєнь, malicious findings або підтверджених AV-engine detections.

Адміністратори можуть зняти хибнопозитивне утримання:

```bash
npx convex run users:liftModerationHold '{"userId": "<user-id>", "reason": "False positive from security tool scanning"}'
```

Це очищає `requiresModerationAt` і `requiresModerationReason`, відновлює
skills, приховані утриманням на рівні користувача, і записує запис журналу аудиту `user.moderation.lift`.
Skills, приховані з інших причин, або ті, чий власний статичний скан усе ще залишається
malicious, залишаються прихованими.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання
можуть призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених
списків.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо CLI auth
починає давати збої після дії щодо облікового запису, увійдіть у web UI, щоб переглянути
стан облікового запису. Якщо вхід або звичайний CLI-доступ заблоковано, зверніться на
security@openclaw.ai для перевірки відновлення.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- додавайте нотатку ClawScan від видавця, коли реліз має незвичну, але навмисну поведінку
- уникайте обфускованих команд встановлення
- посилайтеся на джерело, коли це можливо
- використовуйте dry runs перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку пакунка
