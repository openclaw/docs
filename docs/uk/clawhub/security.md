# Безпека + модерація

ClawHub відкритий для публікації, але публічні списки все одно проходять через механізми довіри,
сканування, звітів і модерації. Мета практична: допомогти користувачам
перевіряти те, що вони встановлюють, дати видавцям шлях відновлення після хибних спрацювань
і не допускати зловживальні пакети до публічного виявлення.

Див. також [Прийнятне використання](/uk/clawhub/acceptable-usage).

## Що можуть перевірити користувачі

Перед установленням skill або plugin перевірте його список у ClawHub на:

- власника й атрибуцію джерела
- останню версію та журнал змін
- потрібні змінні середовища або дозволи
- метадані сумісності для plugins
- статус сканування або модерації
- звіти, коментарі, зірки, завантаження та сигнали встановлення, де вони показані

Встановлюйте лише вміст, який ви розумієте і якому довіряєте.

## Стани сканування

ClawHub може показувати результати сканування або модерації на публічних сторінках і в діагностиці,
видимій власнику.

Поширені результати:

- `clean`: блокувальної проблеми не знайдено.
- `suspicious`: реліз потребує обережності або перегляду.
- `malicious`: реліз вважається небезпечним.
- `pending`: перевірки ще не завершилися.
- `held`, `quarantined`, `revoked` або `hidden`: реліз не повністю
  доступний на публічних поверхнях встановлення.

Точне формулювання може відрізнятися залежно від поверхні, але практичний зміст той самий: якщо
реліз утримано або заблоковано, користувачам не слід встановлювати його, доки власник не вирішить
проблему або модерація не відновить його.

## Skills

Сканування Skills перевіряє опублікований пакет skill, метадані, заявлені
вимоги та підозрілі інструкції.

ClawHub приділяє особливу увагу невідповідностям між тим, що skill заявляє, і
тим, що він, схоже, робить. Наприклад, skill, який посилається на потрібний API-ключ,
має заявити цю вимогу в `SKILL.md`, щоб користувачі могли побачити її перед
встановленням.

Результати сканування базуються на артефактах. Очікувана поведінка провайдера, як-от заявлені
API-облікові дані, OAuth-зворотні виклики localhost, scoped uninstall cleanup, кодування Basic Auth
або вибрані користувачем завантаження файлів до заявленого провайдера, розглядається
інакше, ніж приховане пересилання облікових даних, широкий доступ до приватних файлів,
непов’язані мережеві призначення або приховане зловживання браузером.

Див. [Формат skill](/uk/clawhub/skill-format).

## Plugins

Релізи Plugin містять метадані пакета, атрибуцію джерела, поля сумісності
та інформацію про цілісність артефактів.

OpenClaw перевіряє сумісність перед установленням plugins, розміщених у ClawHub. Записи пакетів
також можуть надавати метадані digest, щоб OpenClaw міг перевіряти завантажені
артефакти. ClawScan включає заявлені метадані env/config пакета `openclaw.environment`
під час перегляду релізів plugin, щоб заявлені вимоги часу виконання
порівнювалися зі спостережуваною поведінкою.

## Звіти

Користувачі, які ввійшли в систему, можуть повідомляти про skills, пакети та коментарі.

Звіти мають бути конкретними й придатними до дії. Зловживання звітуванням саме по собі може призвести до
дій щодо облікового запису.

Приклади звітів:

- оманливі метадані
- незаявлені вимоги до облікових даних або дозволів
- підозрілі інструкції встановлення
- шахрайські коментарі або видавання себе за іншу особу
- недобросовісні реєстрації або неправомірне використання торговельних марок
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

## Звіти про недобросовісність або торговельні марки

ClawHub використовує той самий конвеєр звітів і модерації персоналом для недобросовісних
реєстрацій, видавання себе за іншу особу та спорів, пов’язаних із торговельними марками. Такі звіти потребують
достатнього контексту, щоб персонал міг ідентифікувати заявника, спірний список і
запитану дію.

Додайте:

- канонічну URL-адресу skill або пакета ClawHub і handle власника
- торговельну марку, проєкт, компанію або назву продукту, яких стосується питання
- публічні докази права власності або повноважень заявника
- чому поточний власник не має права публікувати під цією назвою
- запитану дію, як-от приховати до завершення перегляду, передати право власності, перейменувати
  або видалити

Не розміщуйте приватні секрети або чутливі юридичні документи в публічних звітах. Відкрийте
GitHub issue з нечутливими доказами й попросіть мейнтейнерів надати приватний
шлях передавання, коли це потрібно.

## Апеляції та повторні сканування

Власники можуть запросити повторне сканування, якщо вважають, що skill або пакет було помилково
утримано або позначено. Модератори й адміністратори платформи можуть запитувати повторні сканування для будь-якого
skill або пакета під час обробки звітів чи запитів підтримки:

```bash
clawhub skill rescan <slug>
clawhub package rescan <name>
```

Для модерованого вмісту власники можуть мати змогу подати апеляцію з поверхонь ClawHub,
видимих власнику. Апеляції мають пояснювати, що змінилося або чому
позначка є неправильною.

## Модераційні утримання

Коли статичний сканер позначає завантажений skill як malicious, видавець
автоматично потрапляє під модераційне утримання (`requiresModerationAt` встановлюється для
користувача). Це приховує всі skills видавця, змушує майбутні публікації
починатися прихованими та створює запис журналу аудиту `user.moderation.auto`.

Статичні підозрілі знахідки зберігаються як докази файл/рядок для модераторів,
але самі по собі вони не приховують вміст і не визначають публічний вердикт сканування.
Нові завантаження залишаються у стані review/pending, доки не завершиться LLM-перегляд. Статичне
сканування блокує негайно лише за malicious-сигнатурами. Спрацювання рушія VirusTotal
залишаються видимими доказами безпеки, але вердикти VirusTotal Code Insight/Palm
є рекомендаційними й самі по собі не приховують skills. LLM-перегляди ClawScan
зберігають purpose-aligned нотатки як guidance. Знахідки середнього рівня під час review залишаються видимими на
артефакті, тоді як фільтр suspicious зарезервований для високовпливових LLM-занепокоєнь,
malicious-знахідок або підтверджених виявлень AV-рушіями.

Адміністратори можуть зняти хибнопозитивне утримання:

```bash
npx convex run users:liftModerationHold '{"userId": "<user-id>", "reason": "False positive from security tool scanning"}'
```

Це очищає `requiresModerationAt` і `requiresModerationReason`, відновлює
skills, приховані утриманням на рівні користувача, і записує запис журналу аудиту `user.moderation.lift`.
Skills, приховані з інших причин, або ті, чиє власне статичне сканування залишається
malicious, лишаються прихованими.

## Бани та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання
можуть призвести до банів облікових записів, відкликання токенів, прихованого вмісту або видалених
списків.

Видалені, забанені або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо CLI auth
починає давати збій після дій щодо облікового запису, увійдіть у web UI, щоб переглянути стан облікового
запису, або зв’яжіться з мейнтейнерами через очікуваний канал підтримки проєкту.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- заявляйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на джерело, коли можливо
- використовуйте dry runs перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку пакета
