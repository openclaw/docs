# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але поверхні публічного пошуку та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані оголошення та дії щодо
облікових записів допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що не відповідає
політиці.

Ця сторінка охоплює модерацію та статус облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) та
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту marketplace, наприклад:

- шкідливі оголошення
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про навичку** на сторінці навички або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або до репозиторію вихідного коду,
пов’язаного з оголошенням. ClawHub не підтримує і не виправляє
код сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх навичках або плагінах.

Хороші скарги конкретні та придатні до дії. Зловживання системою скарг саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо організації, бренду, області пакета, імені власника або володіння простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований потік скарг або форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub переглянули неконфіденційні докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримав псевдонім
або був інакше переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge tokens до
публічного issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного утримання
видавця або оголошення. Коли це трапляється, відповідний вміст може бути прихований від публічного
пошуку або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки з високим ризиком.
Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані оголошення

Оголошення може бути утримане, приховане, поміщене в карантин, відкликане або інакше недоступне на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих оголошень. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
оголошення зможе повернутися на публічні поверхні.

## Бани та статус облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до банів облікових записів, відкликання токенів, прихованого вмісту або видалених оголошень.
Сигнали тиску зловживань видавців перевіряються щодня. Сигнали, які досягають
порога потенційного бану ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
допустиме сканування після крайнього строку попередження все ще поміщає видавця в
поріг потенційного бану, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого часовими рамками перегляду не входять до автоматичного
застосування.

Видалені, забанені або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо CLI-автентифікація
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан облікового запису.
Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію навички або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте оголошення, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, описів, тегів і changelogs
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
