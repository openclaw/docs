# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захистити користувачів, коли випуск або обліковий запис здається небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших
прав на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, Plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або неправомірне використання торговельних марок
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на Skill** на сторінці Skill або команду/API
для повідомлення про пакет.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього Skill або
Plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
пов’язаний зі списком. ClawHub не супроводжує й не виправляє код сторонніх Skill або Plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх Skills або Plugins.

Хороші скарги конкретні й придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, дескриптором власника або простором імен мають
використовувати процес [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не
внутрішній потік скарг у продукті чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли потрібно, щоб команда ClawHub переглянула неконфіденційний доказ того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, пов’язаний псевдонімом
або інакше переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або токени DNS challenge у
публічний issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть помістити видавця чи список під
модераційне утримання. Коли це відбувається, відповідний вміст може бути приховано з публічного
виявлення або майбутні публікації можуть одразу ставати прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки з високим ризиком.
Їх також можуть зняти, коли підтверджено хибнопозитивний результат.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте випуск, доки власник
не усуне проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого терміну попередження все ще розміщує видавця в
порозі потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали перегляду з нижчою впевненістю та обмежені в часі залишаються поза автоматичним
застосуванням санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування чи вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію Skill або Plugin шкідливою,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для Plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибнопозитивні результати та підвищити довіру користувачів:

- підтримуйте точність назв, резюме, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією Plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку випуску
