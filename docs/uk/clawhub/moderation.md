# Модерація та безпека облікових записів

ClawHub відкритий до публікації, але публічні поверхні виявлення та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані записи та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/uk/clawhub/security-audits).

Див. також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/uk/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- шкідливі записи
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за когось іншого
- недобросовісні реєстрації або зловживання торговельною маркою
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на навичку** на сторінці навички або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в записі. ClawHub не підтримує й не виправляє
код сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisory ClawHub
для вразливостей у сторонніх навичках або плагінах.

Добрі скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
використовувати процес [Претензії щодо організацій і просторів імен](/uk/clawhub/namespace-claims), а не
вбудований потік скарг у продукті чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub перевірила нечутливі докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або бути іншим чином переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge-токени в
публічну issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного утримання
видавця або запису. Коли це трапляється, відповідний вміст може бути приховано від публічного
виявлення або майбутні публікації можуть спершу бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки з високим ризиком.
Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані записи

Запис може бути утриманий, прихований, поміщений у карантин, відкликаний або іншим чином недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих записів. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
запис зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених записів.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть запустити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого терміну попередження все ще розміщує видавця в
пороговій зоні потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені в часі сигнали перевірки не входять до автоматичного
застосування санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування чи вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію навички або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте запис, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- оголошуйте необхідні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори питають про поведінку релізу
