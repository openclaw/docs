# Модерація та безпека облікового запису

ClawHub відкритий до публікації, але публічні поверхні пошуку та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані записи та дії з
обліковими записами допомагають захистити користувачів, коли реліз або обліковий запис
здається небезпечним, оманливим або таким, що порушує політики.

Ця сторінка описує модерацію та статус облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань авторського права або інших
прав на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, plugins і пакунки.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- шкідливі записи
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за інших
- недобросовісні реєстрації або зловживання торговельною маркою
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на навичку** на сторінці навички або команду/API
для скарг на пакунки.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої
навички або plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій
вихідного коду, посилання на який є в записі. ClawHub не підтримує й не виправляє
код сторонніх навичок або plugins.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте
advisories ClawHub для вразливостей у сторонніх навичках або plugins.

Добрі скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може
призвести до дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакунків, ідентифікатором власника
або простором імен мають використовувати процес
[Претензій щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований потік скарг у продукті чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub розглянули
неконфіденційні докази того, що простір імен слід зарезервувати, передати,
перейменувати, приховати, помістити в карантин, зробити псевдонімом або інакше
переглянути. Не додавайте секрети, приватні документи, приватні юридичні файли,
документи, що посвідчують особу, API-токени або токени DNS-виклику до публічної issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного
утримання видавця або запису. Коли це стається, відповідний вміст може бути прихований
від публічного пошуку, або майбутні публікації можуть одразу ставати прихованими, доки
проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує
високоризикові випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані записи

Запис може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше
недоступний на публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник не розв’яже
проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих записів.
Ця діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж запис
зможе повернутися на публічні поверхні.

## Блокування та статус облікового запису

Облікові записи, що порушують політики ClawHub, можуть втратити доступ до публікації.
Серйозні зловживання можуть призвести до блокування облікового запису, відкликання
токенів, прихованого вмісту або видалених записів. Сигнали тиску зловживань видавця
перевіряються щодня. Сигнали, що досягають порога ClawHub для потенційного блокування,
можуть спричинити автоматичне попередження. Якщо наступне придатне сканування після
кінцевого строку попередження все ще поміщає видавця в поріг потенційного блокування,
ClawHub може застосувати дію щодо облікового запису автоматично. Сигнали з нижчою
достовірністю та обмежені в часі сигнали перевірки не входять до автоматичного
застосування санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени
ClawHub. Якщо автентифікація CLI починає давати збій після дії щодо облікового запису,
увійдіть у вебінтерфейс, щоб переглянути стан облікового запису. Якщо вхід або звичайний
доступ CLI заблоковано через блокування або вимкнений обліковий запис, використовуйте
[форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію навички або plugin шкідливою,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте запис, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
