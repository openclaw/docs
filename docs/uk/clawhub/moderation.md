# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та статус облікового запису. Для міток аудиту, таких як
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику, див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливих списків
- оманливих метаданих
- незадекларованих облікових даних або вимог до дозволів
- підозрілих інструкцій зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісних реєстрацій або зловживання торговельними марками
- вмісту, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
пов’язаний зі списком. ClawHub не підтримує і не виправляє код сторонніх skill або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте advisory ClawHub
для вразливостей у сторонніх Skills або плагінах.

Хороші скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен слід
вирішувати через процес [Претензії на організації та простори імен](/clawhub/namespace-claims), а не через
вбудований у продукт потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub переглянула нечутливі докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримав псевдонім
або був інакше переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge tokens до
публічного issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть поставити видавця чи список під
модераційне утримання. Коли це трапляється, відповідний вміст може бути прихований від публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Заборони та статус облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до заборони облікового запису, відкликання токенів, приховання вмісту або видалення списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогового значення ClawHub для потенційної заборони, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого строку попередження все ще поміщає видавця в
порогове значення потенційної заборони, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені часові сигнали перевірки не беруть участі в автоматичному
застосуванні заходів.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає не працювати після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через заборону або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо email, спричинений сканером, називає версію skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вихід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
