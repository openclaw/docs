# Модерація та безпека облікових записів

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки й дії щодо облікових записів
допомагають захистити користувачів, коли випуск або обліковий запис виглядає небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка охоплює модерацію та стан облікового запису. Для міток аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на skills, plugins і packages.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або неправомірне використання торговельних марок
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на skill** на сторінці skill або команду/API
для скарг на packages.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який є в списку. ClawHub не супроводжує й не виправляє
код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, registry, auth,
scanning, moderation або межах довіри download/install. Не використовуйте advisories ClawHub
для вразливостей у сторонніх skills або plugins.

Якісні скарги конкретні й придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо org і namespace

Спори щодо володіння org, brand, package-scope, owner-handle або namespace слід
вирішувати через процес [Претензії щодо org і namespace](/clawhub/namespace-claims), а не через
вбудований у продукт потік скарг чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub переглянули нечутливий доказ того, що
namespace слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити alias
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API tokens або DNS challenge tokens у
публічний issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного утримання
видавця або списку. Коли це стається, зачеплений вміст може бути прихований від публічного
виявлення або майбутні публікації можуть спершу ставати прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибнопозитивний результат.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте випуск, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, які досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого строку попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого часового перегляду не входять до автоматичного
застосування заходів.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API tokens ClawHub. Якщо CLI auth
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування чи вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для розгляду відновлення.

Якщо email, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибнопозитивних результатів і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- оголошуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку випуску
