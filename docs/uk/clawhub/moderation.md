# Модерація та безпека облікового запису

ClawHub відкритий для публікацій, але публічні поверхні виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані сторінки та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка охоплює модерацію та статус облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику, див.
[Аудити безпеки](/uk/clawhub/security-audits).

Див. також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав
на контент використовуйте [Запити щодо прав на контент](/uk/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, плагіни та пакунки.

Використовуйте скарги ClawHub лише для небезпечного контенту маркетплейсу, наприклад:

- шкідливі сторінки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за інших
- недобросовісні реєстрації або зловживання торговельними марками
- контент, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на навичку** на сторінці навички або команду/API
для скарг на пакунки.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено на сторінці. ClawHub не підтримує і не виправляє код
сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте рекомендації
ClawHub для вразливостей у сторонніх навичках або плагінах.

Якісні скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакунків, ідентифікатором власника або простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/uk/clawhub/namespace-claims), а не
внутрішній потік скарг у продукті чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub перевірили неконфіденційний доказ того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше переглянути. Не додавайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або токени DNS-перевірки до
публічного issue.

## Модераційні утримання

Деякі серйозні знахідки або порушення політики можуть призвести до модераційного утримання
видавця або сторінки. Коли це трапляється, уражений контент може бути прихований від публічного
виявлення або майбутні публікації можуть одразу створюватися прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані сторінки

Сторінка може бути утримана, прихована, поміщена в карантин, відкликана або інакше недоступна на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих сторінок. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
сторінка зможе повернутися на публічні поверхні.

## Блокування та статус облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого контенту або видалених сторінок.
Сигнали тиску через зловживання видавця перевіряються щодня. Сигнали, що досягають
порога потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після дедлайну попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може автоматично застосувати дію щодо облікового запису.
Сигнали з нижчою впевненістю та обмежені часові сигнали перевірки не входять до автоматичного
застосування.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан облікового запису.
Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо лист, спричинений сканером, називає версію навички або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте сторінку, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
