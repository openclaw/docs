# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії з обліковими записами
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику дивіться в
[Аудитах безпеки](/uk/clawhub/security-audits).

Дивіться також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав
на вміст використовуйте [Запити щодо прав на вміст](/uk/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про Skill** на сторінці Skill або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього Skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
пов’язаний зі списком. ClawHub не підтримує і не виправляє код сторонніх Skills або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх Skills або плагінах.

Хороші скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, дескриптором власника або простором імен мають
використовувати процес [Претензії щодо організацій і просторів імен](/uk/clawhub/namespace-claims), а не
вбудований у продукт потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub переглянули неконфіденційний доказ того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або іншим чином переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge-токени в
публічний issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть поставити видавця або список під
модераційне утримання. Коли це відбувається, зачеплений вміст може бути прихований із публічного
виявлення або майбутні публікації можуть починатися прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або іншим чином недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, якщо власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживань з боку видавців перевіряються щодня. Сигнали, що досягають
порогу ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне
відповідне сканування після дедлайну попередження все ще розміщує видавця в
пороговому діапазоні потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені часові сигнали перевірки не потрапляють до автоматичного
застосування санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає не вдаватися після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію Skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вихід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і changelog
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
