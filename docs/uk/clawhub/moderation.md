# Модерація та безпека облікових записів

ClawHub відкритий для публікації, але публічне виявлення та поверхні встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії з обліковими записами
допомагають захистити користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка описує модерацію та статус облікового запису. Про аудиторські мітки, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику дивіться в
[Аудитах безпеки](/clawhub/security-audits).

Дивіться також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Щодо авторського права або інших питань
прав на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на Skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- зловмисні списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який є у списку. ClawHub не підтримує і не виправляє
код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади охоплюють помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх skills або plugins.

Хороші скарги конкретні й придатні до виконання. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
проходити через процес [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не через
вбудований у продукт потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли потрібно, щоб співробітники ClawHub переглянули нечутливий доказ того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, призначити псевдонім
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API tokens або DNS challenge tokens у
публічне issue.

## Модераційні утримання

Деякі серйозні виявлені проблеми або порушення політики можуть призвести до модераційного утримання
видавця або списку. Коли це трапляється, зачеплений вміст може бути прихований від публічного
виявлення, або майбутні публікації можуть спочатку ставати прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Заборони та статус облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до заборон облікового запису, відкликання token, прихованого вмісту або видалених списків.
Сигнали тиску зловживань з боку видавців перевіряються щодня. Сигнали, що досягають
порогу потенційної заборони ClawHub, можуть запустити автоматичне попередження. Якщо наступне
допустиме сканування після кінцевого строку попередження все ще ставить видавця в
поріг потенційної заборони, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмежені в часі сигнали перегляду не беруть участі в автоматичному
застосуванні.

Видалені, забанені або вимкнені облікові записи не можуть використовувати API tokens ClawHub. Якщо автентифікація CLI
починає не працювати після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо email, спричинений сканером, називає версію skill або plugin зловмисною,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер
версії та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори питають про поведінку релізу
