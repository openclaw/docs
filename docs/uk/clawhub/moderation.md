# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії з обліковими записами
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Щодо питань авторського права або інших прав
на контент використовуйте [Запити щодо прав на контент](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за інших
- недобросовісні реєстрації або неправомірне використання торговельних марок
- контент, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на навичку** на сторінці навички або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який указано в списку. ClawHub не супроводжує і не виправляє код
сторонніх навичок або плагінів.

Рекомендації GitHub із безпеки для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте рекомендації
ClawHub для вразливостей у сторонніх навичках або плагінах.

Хороші скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен слід
вирішувати через процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не через
вбудований у продукт потік скарг або форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub переглянула нечутливі докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений на карантин, отримав псевдонім
або був інакше переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або токени DNS-перевірки в
публічну проблему.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть помістити видавця чи список під
модераційне утримання. Коли це відбувається, зачеплений контент може бути прихований від публічного
виявлення, або майбутні публікації можуть одразу створюватися прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки з високим ризиком.
Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений на карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до блокування облікового запису, відкликання токенів, приховання контенту або видалення списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого строку попередження все ще розміщує видавця в
порозі потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого часовими рамками перегляду не беруть участі в автоматичному
застосуванні санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію навички або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Поради видавцям

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
