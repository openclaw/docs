# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення й інсталяції все ще
потребують запобіжників. Скарги, модераційні утримання, приховані публікації та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис виглядає небезпечним, оманливим чи таким,
що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про аудиторські мітки, такі як
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань авторського права або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту marketplace, наприклад:

- шкідливих публікацій
- оманливих метаданих
- незадекларованих облікових даних або вимог до дозволів
- підозрілих інструкцій з інсталяції
- видавання себе за когось іншого
- недобросовісних реєстрацій або неправомірного використання торговельної марки
- вмісту, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про Skill** на сторінці Skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього Skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в публікації. ClawHub не підтримує і не виправляє код сторонніх
Skills або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади охоплюють помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/інсталяції. Не використовуйте рекомендації ClawHub
для вразливостей у сторонніх Skills або плагінах.

Хороші скарги є конкретними та придатними до дії. Зловживання поданням скарг саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організації та namespace

Спори щодо володіння організацією, брендом, областю пакета, дескриптором власника або namespace слід
вирішувати через процес [Претензії щодо організації та Namespace](/clawhub/namespace-claims), а не через
вбудований у продукт процес скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб працівники ClawHub переглянули неконфіденційні докази того, що
namespace має бути зарезервовано, передано, перейменовано, приховано, поміщено в карантин, пов’язано псевдонімом
або інакше переглянуто. Не додавайте секрети, приватні документи, приватні юридичні
файли, особисті документи, що посвідчують особу, API-токени або DNS challenge-токени до
публічного issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми політики можуть помістити видавця чи публікацію під
модераційне утримання. Коли це відбувається, зачеплений вміст може бути прихований від публічного
виявлення або майбутні публікації можуть спершу ставати прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані публікації

Публікація може бути утримана, прихована, поміщена в карантин, відкликана або інакше недоступна на
публічних поверхнях інсталяції.

Якщо ви бачите один із цих станів, не інсталюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих публікацій. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
публікація зможе повернутися на публічні поверхні.

## Бани та стан облікового запису

Облікові записи, які порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до банів облікового запису, відкликання токенів, прихованого вмісту або видалених публікацій.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного бану ClawHub, можуть запустити автоматичне попередження. Якщо наступне
придатне сканування після дедлайну попередження все ще поміщає видавця в
поріг потенційного бану, ClawHub може автоматично застосувати дію щодо облікового запису.
Сигнали нижчої впевненості та обмеженого в часі перегляду не беруть участі в автоматичному
застосуванні санкцій.

Видалені, забанені або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збої після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан облікового
запису. Якщо вхід або звичайний доступ CLI заблоковано баном або вимкненим обліковим записом,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію Skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте публікацію, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд інсталяції
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
