# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але поверхні публічного виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії з обліковими записами
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка охоплює модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, що ввійшли в систему, можуть скаржитися на навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- зловмисні списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за інших
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на навичку** на сторінці навички або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який є у списку. ClawHub не підтримує й не виправляє код сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisory ClawHub
для вразливостей у сторонніх навичках або плагінах.

Хороші скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо організацій, брендів, областей пакетів, імен власників або володіння просторами імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не
внутрішній потік скарг у продукті чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub перевірили неконфіденційний доказ того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або іншим чином переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge-токени в
публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного утримання
видавця або списку. Коли це стається, відповідний вміст може бути прихований від публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Заборони та стан облікового запису

Облікові записи, які порушують політику ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до заборон облікового запису, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживання видавця перевіряються щодня. Сигнали, що досягають
порога ClawHub для потенційної заборони, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього строку попередження все ще поміщає видавця в
поріг потенційної заборони, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені в часі сигнали перевірки не потрапляють до автоматичного
застосування.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає завершуватися помилкою після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб перевірити стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через заборону або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо лист, спричинений сканером, називає версію навички або плагіна зловмисною,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вихідні дані сканування, виправте список, збільште номер
версії та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
