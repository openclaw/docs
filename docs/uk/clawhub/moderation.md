# Модерація та безпека облікових записів

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захистити користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка охоплює модерацію та стан облікового запису. Про аудиторські мітки, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Щодо авторського права або інших питань
прав на контент використовуйте [Запити щодо прав на контент](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного контенту маркетплейсу, зокрема:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу
- недобросовісні реєстрації або зловживання торговельними марками
- контент, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або у вихідний репозиторій,
пов’язаний зі списком. ClawHub не підтримує й не виправляє код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх Skills або plugins.

Добрі скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
проходити через процес [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не через
потік скарг у продукті чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub переглянула неконфіденційні докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або інакше переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, що посвідчують особу, API-токени або DNS challenge-токени в
публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть помістити видавця чи список під
модераційне утримання. Коли це відбувається, відповідний контент може бути прихований від публічного
виявлення, або майбутні публікації можуть спочатку бути прихованими, доки проблему не переглянуть.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки з високим ризиком.
Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, які порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, приховання контенту або видалення списків.
Сигнали тиску через зловживання видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього терміну попередження все ще розміщує видавця в
порозі потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали перегляду з нижчою впевненістю та обмежені в часі залишаються поза автоматичним
примусовим застосуванням.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо email, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
