# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічне виявлення та поверхні встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує правила.

Ця сторінка описує модерацію та стан облікового запису. Для аудиторських міток, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Допустиме використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за інших
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Допустиме використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API для
повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в списку. ClawHub не підтримує й не виправляє код
сторонніх skill або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у самому
ClawHub. Приклади: помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх Skills або плагінах.

Добрі скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
використовувати процес [Заявки на організації та простори імен](/clawhub/namespace-claims), а не
вбудований у продукт потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub переглянули неконфіденційні докази того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або DNS challenge-токени в
публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або порушення правил можуть призвести до модераційного утримання
видавця або списку. Коли це трапляється, відповідний вміст може бути приховано з публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують правила ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до блокування облікового запису, відкликання токенів, приховання вмісту або видалення списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порога потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього строку попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені часові сигнали перегляду не входять до автоматичного
застосування заходів.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер
версії та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і changelog
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте dry runs перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
