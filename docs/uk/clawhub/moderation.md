# Модерація та безпека облікових записів

ClawHub відкритий для публікації, але поверхні публічного пошуку й інсталяції все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захистити користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції з інсталяції
- видавання себе за іншу особу
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або до репозиторію вихідного коду,
посилання на який є у списку. ClawHub не підтримує й не виправляє код
сторонніх skills або plugins.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/інсталяції. Не використовуйте advisories
ClawHub для вразливостей у сторонніх skills або plugins.

Добрі скарги є конкретними й придатними до дії. Зловживання скаргами саме собою може призвести до
дій щодо облікового запису.

## Заявки на org і простори імен

Спори щодо org, бренду, області пакета, handle власника або володіння простором імен мають
проходити через процес [Заявки на org і простори імен](/clawhub/namespace-claims), а не через
вбудований потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub розглянули неконфіденційний доказ того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або токени DNS challenge у
публічне issue.

## Модераційні утримання

Деякі серйозні висновки або проблеми з політикою можуть помістити видавця або список під
модераційне утримання. Коли це відбувається, зачеплений вміст може бути прихований із публічного
пошуку або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях інсталяції.

Якщо ви бачите один із цих станів, не інсталюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, які порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікових записів, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порога ClawHub для потенційного блокування, можуть запустити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого строку попередження все ще розміщує видавця в
порозі потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені в часі сигнали перегляду залишаються поза автоматичним
застосуванням.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо CLI-автентифікація
починає відмовляти після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо email, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер
версії та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і changelogs
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд інсталяції
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
