# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що
порушує політики.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/uk/clawhub/security-audits).

Див. також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). З питань авторського права чи інших прав на вміст
використовуйте [Запити щодо прав на вміст](/uk/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за інших
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який є у списку. ClawHub не підтримує й не виправляє код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх skills або plugins.

Якісні скарги конкретні й придатні до дії. Зловживання повідомленнями може саме по собі призвести до
дій щодо облікового запису.

## Заявки на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакетів, ідентифікатором власника або простором імен мають
використовувати процес [Заявки на організації та простори імен](/uk/clawhub/namespace-claims), а не
вбудований у продукт потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібен перегляд співробітниками ClawHub неконфіденційних доказів того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати як псевдонім
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge tokens у
публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політиками можуть призвести до модераційного утримання
видавця або списку. Коли це трапляється, відповідний вміст може бути прихований від публічного
виявлення, або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники можуть і надалі бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого строку попередження все ще розміщує видавця в межах
порогу потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмежені часові сигнали перевірки не беруть участі в автоматичному
застосуванні санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає не вдаватися після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
і завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і changelogs
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори питають про поведінку релізу
