# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але поверхні публічного виявлення та встановлення все ще потребують захисних обмежень. Скарги, модераційні утримання, приховані сторінки та дії щодо облікових записів допомагають захищати користувачів, коли реліз або обліковий запис виглядає небезпечним, оманливим або таким, що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику дивіться в
[Аудитах безпеки](/clawhub/security-audits).

Дивіться також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, як-от:

- шкідливі сторінки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу або організацію
- недобросовісні реєстрації або неправомірне використання торговельних марок
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про навичку** на сторінці навички або команду/API для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду, посилання на який є на сторінці. ClawHub не супроводжує й не виправляє код сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації, скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте advisory ClawHub для вразливостей у сторонніх навичках або плагінах.

Добрі скарги є конкретними та придатними до дії. Зловживання системою скарг саме по собі може призвести до дій щодо облікового запису.

## Заявки на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакетів, ідентифікатором власника або простором імен мають використовувати процес [Заявки на організації та простори імен](/clawhub/namespace-claims), а не вбудований у продукт потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub розглянули неконфіденційні докази того, що простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримав псевдонім або був інакше переглянутий. Не додавайте секрети, приватні документи, приватні юридичні файли, документи, що посвідчують особу, API-токени або токени DNS-викликів у публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або порушення політики можуть помістити видавця чи сторінку під модераційне утримання. Коли це трапляється, уражений вміст може бути приховано від публічного виявлення, або майбутні публікації можуть одразу ставати прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки високого ризику. Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані сторінки

Сторінка може бути утримана, прихована, поміщена в карантин, відкликана або інакше недоступна на публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих сторінок. Ця діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж сторінка зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених сторінок. Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають порога ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне придатне сканування після дедлайну попередження все ще поміщає видавця в поріг потенційного блокування, ClawHub може автоматично застосувати дію щодо облікового запису. Сигнали з нижчою впевненістю та обмежені часові сигнали перевірки не потрапляють до автоматичного застосування.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис, використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для розгляду відновлення.

Якщо лист, спричинений сканером, називає версію навички або плагіна шкідливою, завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте сторінку, збільште номер версії та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
