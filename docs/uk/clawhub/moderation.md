# Модерація та безпека акаунта

ClawHub відкритий для публікації, але публічні поверхні пошуку та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані публікації та дії щодо акаунтів
допомагають захистити користувачів, коли реліз або акаунт виглядає небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та статус акаунта. Про аудиторські мітки, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших
прав на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі публікації
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для надсилання скарг щодо пакетів.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в публікації. ClawHub не підтримує й не виправляє код
сторонніх skill або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте
advisories ClawHub для вразливостей у сторонніх Skills або плагінах.

Добрі скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо акаунта.

## Заявки на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакета, дескриптором власника або простором імен
мають використовувати процес [Заявки на організації та простори імен](/clawhub/namespace-claims), а не
потік скарг у продукті чи форму апеляції акаунта.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub розглянули нечутливі докази того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити для нього псевдонім
або інакше переглянути. Не додавайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або токени DNS-перевірки в
публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть помістити видавця чи публікацію під
модераційне утримання. Коли це трапляється, відповідний вміст може бути прихований від публічного
пошуку або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибне спрацьовування.

## Приховані або заблоковані публікації

Публікація може бути утримана, прихована, поміщена в карантин, відкликана або інакше недоступна на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих публікацій. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
публікація зможе повернутися на публічні поверхні.

## Блокування та статус акаунта

Акаунти, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування акаунта, відкликання токенів, приховування вмісту або видалення публікацій.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порога потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після дедлайну попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може застосувати дію щодо акаунта автоматично.
Сигнали нижчої впевненості та обмеженого в часі перегляду не потрапляють до автоматичного
застосування заходів.

Видалені, заблоковані або вимкнені акаунти не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає завершуватися помилкою після дії щодо акаунта, увійдіть у вебінтерфейс, щоб переглянути
стан акаунта. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений акаунт,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо лист, спричинений сканером, називає версію skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте публікацію, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацьовувань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
