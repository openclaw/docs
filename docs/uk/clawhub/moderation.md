# Модерація та безпека облікових записів

ClawHub відкритий для публікації, але механізми загальнодоступного пошуку та встановлення все одно потребують захисних обмежень. Скарги, блокування на час модерації, приховані сторінки та заходи щодо облікових записів допомагають захистити користувачів, коли випуск або обліковий запис видається небезпечним, оманливим чи таким, що порушує правила.

На цій сторінці описано модерацію та статус облікового запису. Відомості про мітки аудиту, як-от `Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див. у розділі [Аудити безпеки](/clawhub/security-audits).

Див. також розділи [Безпека](/uk/clawhub/security) та [Допустиме використання](/clawhub/acceptable-usage). Із питань авторського права чи інших прав на вміст скористайтеся розділом [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть подавати скарги на Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише щодо небезпечного вмісту торговельного майданчика, зокрема:

- шкідливих сторінок
- оманливих метаданих
- незадекларованих облікових даних або вимог до дозволів
- підозрілих інструкцій зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісних реєстрацій або неправомірного використання торговельних марок
- вмісту, що порушує правила [допустимого використання](/clawhub/acceptable-usage)

Скористайтеся кнопкою **Повідомити про Skill** на сторінці Skill або командою/API для подання скарги на пакет.

Не використовуйте скарги ClawHub для повідомлення про вразливості у власному вихідному коді стороннього Skill або плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторії вихідного коду, посилання на який наведено на сторінці. ClawHub не супроводжує та не виправляє код сторонніх Skills або плагінів.

Рекомендації GitHub із безпеки для `openclaw/clawhub` призначені для вразливостей у самому ClawHub. До них належать, наприклад, помилки на вебсайті, в API, CLI, реєстрі, автентифікації, скануванні, модерації або на межах довіри під час завантаження чи встановлення. Не використовуйте рекомендації ClawHub для повідомлення про вразливості у сторонніх Skills або плагінах.

Якісні скарги мають бути конкретними та придатними для вжиття заходів. Зловживання системою скарг саме по собі може призвести до заходів щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо права власності на організацію, бренд, область пакетів, ідентифікатор власника або простір імен слід вирішувати за процедурою [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не через вбудований механізм скарг або форму оскарження заходів щодо облікового запису.

Використовуйте цю процедуру, коли потрібно, щоб працівники ClawHub перевірили нечутливі докази того, що простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, призначити йому псевдонім або перевірити в інший спосіб. Не додавайте до загальнодоступного звернення секрети, приватні документи, конфіденційні юридичні файли, документи, що посвідчують особу, токени API або токени перевірки DNS.

## Блокування на час модерації

Деякі серйозні знахідки або порушення правил можуть призвести до блокування видавця чи сторінки на час модерації. У такому разі відповідний вміст може бути приховано із загальнодоступного пошуку, а майбутні публікації можуть спочатку бути прихованими до завершення перевірки проблеми.

Блокування на час модерації призначені для захисту користувачів, поки ClawHub розглядає випадки з високим ризиком. Їх також можна зняти після підтвердження хибнопозитивного результату.

## Приховані або заблоковані сторінки

Сторінку може бути затримано, приховано, поміщено в карантин, відкликано або в інший спосіб зроблено недоступною в загальнодоступних механізмах встановлення.

Якщо ви бачите один із цих станів, не встановлюйте випуск, доки власник не усуне проблему або модерація не відновить його доступність.

Власники все одно можуть бачити діагностичні дані для власних затриманих або прихованих сторінок. Ці дані допомагають пояснити, що сталося та що потрібно змінити, перш ніж сторінка зможе повернутися до загальнодоступних механізмів.

## Блокування та статус облікового запису

Облікові записи, які порушують правила ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть призвести до блокування облікового запису, відкликання токенів, приховування вмісту або видалення сторінок. Сигнали ризику зловживань із боку видавців перевіряються щодня. Сигнали, які досягають порогового рівня ClawHub для можливого блокування, можуть спричинити автоматичне попередження. Якщо наступне планове сканування після завершення строку, зазначеного в попередженні, усе ще відносить видавця до порогового рівня можливого блокування, ClawHub може автоматично застосувати захід щодо облікового запису. Сигнали з нижчою достовірністю та сигнали обмеженої в часі перевірки не призводять до автоматичного застосування заходів.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати токени API ClawHub. Якщо після застосування заходу щодо облікового запису автентифікація в CLI починає завершуватися помилкою, увійдіть у вебінтерфейс, щоб перевірити стан облікового запису. Якщо блокування або вимкнення облікового запису перешкоджає входу чи звичайному доступу через CLI, скористайтеся [формою оскарження ClawHub](https://appeals.openclaw.ai/), щоб подати запит на перевірку з метою відновлення доступу.

Якщо в електронному листі, надісланому за результатами сканування, версію Skill або плагіна названо шкідливою, завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте результати сканування, виправте сторінку, збільште номер версії та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибнопозитивних результатів і підвищити довіру користувачів:

- підтримуйте точність назв, описів, тегів і журналів змін
- декларуйте необхідні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості додавайте посилання на вихідний код
- виконуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку випуску
