# Модерація та безпека облікового запису

ClawHub відкритий до публікації, але публічні поверхні виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані оголошення та дії з обліковими записами
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка охоплює модерацію та стан облікового запису. Для аудиторських міток, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/uk/clawhub/security-audits).

Див. також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). Щодо авторського права або інших питань
прав на контент використовуйте [Запити щодо прав на контент](/uk/clawhub/content-rights).

## Скарги

Користувачі, що ввійшли в систему, можуть скаржитися на Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного контенту marketplace, зокрема:

- шкідливі оголошення
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельними марками
- контент, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в оголошенні. ClawHub не підтримує й не виправляє
код сторонніх skills або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories
ClawHub для вразливостей у сторонніх skills або плагінах.

Добрі скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакета, handle власника або простором імен слід
вирішувати через процес [Заявки на організації та простори імен](/uk/clawhub/namespace-claims), а не через
вбудований потік скарг або форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub переглянули неконфіденційні докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або бути інакше переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або DNS challenge-токени до
публічного issue.

## Модераційні утримання

Деякі серйозні знахідки або порушення політики можуть помістити видавця чи оголошення під
модераційне утримання. Коли це відбувається, відповідний контент може бути прихований від публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не переглянуть.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв'язує випадки
високого ризику. Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані оголошення

Оголошення може бути утримане, приховане, поміщене в карантин, відкликане або інакше недоступне на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв'яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих оголошень. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
оголошення зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, приховання контенту або видалення оголошень.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після дедлайну попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмежені в часі сигнали перевірки не беруть участі в автоматичному
застосуванні заходів.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо CLI auth
починає не працювати після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо лист, спричинений сканером, називає версію skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте оголошення, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і changelogs
- оголошуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
