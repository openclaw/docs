# Модерація та безпека облікового запису

ClawHub відкритий до публікації, але поверхні публічного виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії з обліковими записами
допомагають захистити користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що виходить
за межі політики.

Ця сторінка описує модерацію та стан облікового запису. Для міток аудиту, таких як
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав
на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, що ввійшли в систему, можуть скаржитися на Skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на skill** на сторінці skill або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено у списку. ClawHub не підтримує й не виправляє
код сторонніх skills або plugins.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх skills або plugins.

Добрі скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки на організацію та простір імен

Спори щодо володіння організацією, брендом, областю пакета, handle власника або простором імен мають
використовувати процес [Заявки на організацію та простір імен](/clawhub/namespace-claims), а не
вбудований у продукт потік скарг або форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб персонал ClawHub розглянув неконфіденційний доказ того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити для нього псевдонім
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, що посвідчують особу, токени API або токени DNS challenge у
публічну issue.

## Модераційні утримання

Деякі серйозні висновки або проблеми з політикою можуть помістити видавця або список під
модераційне утримання. Коли це трапляється, уражений вміст може бути прихований від публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибнопозитивний результат.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
поверхнях публічного встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, якщо власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токена, прихованого вмісту або вилучених списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього терміну попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого часового перегляду не беруть участі в автоматичному
застосуванні.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для розгляду відновлення.

Якщо електронний лист, ініційований сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте результат сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибнопозитивні результати та підвищити довіру користувачів:

- підтримуйте точність назв, підсумків, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
