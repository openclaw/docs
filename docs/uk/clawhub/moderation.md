# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічне виявлення та поверхні встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані оголошення та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка охоплює модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав
на контент використовуйте [Запити щодо прав на контент](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного контенту маркетплейсу, такого як:

- шкідливі оголошення
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за когось іншого
- недобросовісні реєстрації або неправомірне використання торговельної марки
- контент, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про навичку** на сторінці навички або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який є в оголошенні. ClawHub не підтримує й не виправляє код
сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх навичках або плагінах.

Хороші скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований у продукт потік скарг або форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub переглянула нечутливі докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримав псевдонім
або був інакше переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або DNS challenge tokens у
публічне питання.

## Модераційні утримання

Деякі серйозні висновки або проблеми з політикою можуть призвести до модераційного утримання
видавця або оголошення. Коли це трапляється, уражений контент може бути прихований із публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані оголошення

Оголошення може бути утримане, приховане, поміщене в карантин, відкликане або інакше недоступне на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих оголошень. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
оголошення зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до блокування облікового запису, відкликання токенів, прихованого контенту або видалених оголошень.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогового значення ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього строку попередження все ще розміщує видавця в
пороговому значенні потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого в часі перегляду не потрапляють до автоматичного
застосування.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо лист, спричинений сканером, називає версію навички або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вихід сканування, виправте оголошення, збільште номер
версії та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити хибні спрацювання та покращити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
