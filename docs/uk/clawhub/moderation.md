# Модерація та безпека облікового запису

ClawHub відкритий для публікацій, але публічні поверхні виявлення та встановлення все ще
потребують захисних обмежень. Скарги, модераційні утримання, приховані записи та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Для аудиторських міток, таких як
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Щодо авторських прав або інших питань
прав на контент використовуйте [Запити щодо прав на контент](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, зокрема:

- зловмисні записи
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або неправомірне використання торговельної марки
- контент, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на skill** на сторінці skill або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в записі. ClawHub не підтримує й не виправляє
код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisory ClawHub
для вразливостей у сторонніх Skills або plugins.

Добрі скарги конкретні й придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо організації, бренду, області пакета, імені власника або володіння простором імен мають
проходити через процес [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не через
вбудований у продукт потік скарг або форму апеляції щодо облікового запису.

Використовуйте цей процес, коли потрібно, щоб співробітники ClawHub переглянули неконфіденційні докази того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити для нього псевдонім
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, токени API або токени DNS-перевірки в
публічну issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного утримання
видавця або запису. Коли це стається, відповідний контент може бути прихований від публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не переглянуть.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані записи

Запис може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих записів. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
запис зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого контенту або видалених записів.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порога потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після строку попередження все ще розміщує видавця в межах
порога потенційного блокування, ClawHub може автоматично застосувати дію щодо облікового запису.
Сигнали нижчої впевненості та обмежені в часі сигнали перегляду не входять до автоматичного
застосування.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати токени API ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо лист, спричинений сканером, називає версію skill або plugin зловмисною,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте запис, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості посилайтеся на вихідний код
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори питають про поведінку релізу
