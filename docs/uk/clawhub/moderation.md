# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але інтерфейси публічного пошуку та встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані записи та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та статус облікового запису. Для міток аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань авторського права або інших прав
на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про Skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі записи
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за когось іншого
- недобросовісні реєстрації або неправомірне використання торговельних марок
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про Skill** на сторінці Skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього Skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який указано в записі. ClawHub не підтримує й не виправляє код сторонніх
Skills або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте
advisories ClawHub для вразливостей у сторонніх Skills або плагінах.

Хороші скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, псевдонімом власника або простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований потік скарг чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub перевірила нечутливі докази того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або іншим чином переглянути. Не додавайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або DNS challenge tokens у публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного утримання
видавця або запису. Коли це відбувається, відповідний вміст може бути прихований від публічного
пошуку, або майбутні публікації можуть одразу ставати прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує високоризикові
випадки. Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані записи

Запис може бути утриманий, прихований, поміщений у карантин, відкликаний або іншим чином недоступний
в інтерфейсах публічного встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник не розв’яже
проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих записів. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
запис зможе повернутися до публічних інтерфейсів.

## Блокування та статус облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, приховання вмісту або видалення записів.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогового значення ClawHub для можливого блокування, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після дедлайну попередження все ще ставить видавця в
порогове значення можливого блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого в часі перегляду не беруть участі в автоматичному
застосуванні санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо CLI auth
починає давати збої після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ через CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію Skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої надісланої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте запис, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і changelog
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте dry runs перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
