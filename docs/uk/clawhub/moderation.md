# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але поверхні публічного пошуку та встановлення все одно
потребують захисних обмежень. Скарги, модераційні утримання, приховані записи та дії з обліковими записами
допомагають захищати користувачів, коли реліз або обліковий запис видається небезпечним, оманливим або таким,
що порушує правила.

Ця сторінка охоплює модерацію та стан облікового запису. Для аудиторських міток, таких як
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших прав
на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть повідомляти про skills, plugins і пакунки.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі записи
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельною маркою
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для повідомлення про пакунки.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
пов’язаний із записом. ClawHub не підтримує і не виправляє код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте advisories
ClawHub для вразливостей у сторонніх skills або plugins.

Добрі скарги є конкретними й придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакунка, handle власника або простором імен мають
використовувати процес [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований у продукт потік скарг чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли потрібно, щоб співробітники ClawHub переглянули нечутливі докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або бути інакше переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або DNS challenge tokens у публічне issue.

## Модераційні утримання

Деякі серйозні висновки або порушення правил можуть поставити видавця чи запис під
модераційне утримання. Коли це стається, відповідний вміст може бути прихований від публічного
пошуку, або майбутні публікації можуть спершу ставати прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує випадки з високим ризиком.
Їх також можуть зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані записи

Запис може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих записів. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
запис зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують правила ClawHub, можуть втратити доступ до публікації. Серйозне зловживання може
призвести до блокування облікового запису, відкликання токена, прихованого вмісту або видалених записів.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порога ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього строку попередження все ще відносить видавця до
порога потенційного блокування, ClawHub може застосувати дію до облікового запису автоматично.
Сигнали з нижчою впевненістю та обмежені часові сигнали перевірки не беруть участі в автоматичному
застосуванні санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збої після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте запис, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
