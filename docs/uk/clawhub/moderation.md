# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все одно
потребують захисних обмежень. Скарги, модераційні утримання, приховані списки та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка описує модерацію та статус облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/uk/clawhub/security-audits).

Див. також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). Для питань щодо авторських прав або інших прав на вміст
використовуйте [Запити щодо прав на вміст](/uk/clawhub/content-rights).

## Скарги

Користувачі, що ввійшли в систему, можуть повідомляти про skills, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або неправомірне використання торговельної марки
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено у списку. ClawHub не підтримує й не виправляє
код сторонніх skills або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх skills або плагінах.

Добрі скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, handle власника або простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/uk/clawhub/namespace-claims), а не
вбудований у продукт потік скарг чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб персонал ClawHub переглянув неконфіденційний доказ того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, призначити йому alias
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, документи, що посвідчують особу, API-токени або токени DNS challenge у
публічне issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть помістити видавця чи список під
модераційне утримання. Коли це відбувається, відповідний вміст може бути прихований із публічного
виявлення або майбутні публікації можуть спершу бути прихованими, доки проблему не буде переглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує високоризикові
випадки. Їх також можуть зняти, коли підтверджено хибнопозитивний результат.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або іншим чином недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та статус облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених списків.
Сигнали тиску зловживань видавцем перевіряються щодня. Сигнали, що досягають
порогу ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне
допустиме сканування після кінцевого строку попередження все ще розміщує видавця в
порозі потенційного блокування, ClawHub може автоматично застосувати дію щодо облікового запису.
Сигнали нижчої впевненості та обмежені часові сигнали перевірки не потрапляють до автоматичного
застосування.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування чи вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію skill або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити хибнопозитивні результати та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- за можливості посилайтеся на вихідний код
- використовуйте dry runs перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
