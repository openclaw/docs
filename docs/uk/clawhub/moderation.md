# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічне виявлення та поверхні встановлення все одно
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо облікового запису
допомагають захищати користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким, що
порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Для міток аудиту, таких як
`Pass`, `Review`, `Warn`, `Malicious`, і рівня ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Щодо питань авторських прав або інших
прав на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливих списків
- оманливих метаданих
- незадекларованих облікових даних або вимог до дозволів
- підозрілих інструкцій зі встановлення
- видавання себе за іншу особу
- недобросовісних реєстрацій або неправомірного використання торговельної марки
- вмісту, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на skill** на сторінці skill або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в списку. ClawHub не підтримує й не виправляє
код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisory ClawHub
для вразливостей у сторонніх skills або plugins.

Хороші скарги конкретні та придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований у продукт потік скарг чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub розглянула неконфіденційний доказ того, що
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити для нього псевдонім
або інакше переглянути. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, що посвідчують особу, токени API або токени DNS-перевірки в
публічну issue.

## Модераційні утримання

Деякі серйозні висновки або порушення політики можуть помістити видавця чи список під
модераційне утримання. Коли це трапляється, відповідний вміст може бути приховано з публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки питання не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, якщо власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токена, приховання вмісту або видалення списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
допустиме сканування після кінцевого строку попередження все ще розміщує видавця в
порозі потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмеженого часового перегляду не включаються до автоматичного
застосування санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у веб-інтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через блокування чи вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для розгляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
і завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
