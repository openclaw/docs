# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але поверхні публічного пошуку й установлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані оголошення та дії щодо облікових записів
допомагають захищати користувачів, коли реліз або обліковий запис видається небезпечним, оманливим або таким,
що порушує політики.

На цій сторінці описано модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Щодо авторського права або інших питань
прав на контент використовуйте [Запити щодо прав на контент](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на skills, plugins і пакети.

Використовуйте скарги ClawHub лише для небезпечного контенту маркетплейсу, наприклад:

- шкідливі оголошення
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції з установлення
- видавання себе за інших
- недобросовісні реєстрації або неправомірне використання торговельних марок
- контент, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді стороннього skill або
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в оголошенні. ClawHub не підтримує й не виправляє
код сторонніх skill або plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/встановлення. Не використовуйте advisories
ClawHub для вразливостей у сторонніх skills або plugins.

Хороші скарги конкретні й придатні до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Заявки щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника або простором імен мають
використовувати процес [Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований потік скарг чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібен перегляд співробітниками ClawHub нечутливих доказів того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або інакше переглянутий. Не додавайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API tokens або DNS challenge tokens до
публічного issue.

## Модераційні утримання

Деякі серйозні висновки або проблеми з політиками можуть призвести до модераційного утримання
видавця або оголошення. У такому разі відповідний контент може бути приховано з публічного
пошуку, або майбутні публікації можуть спершу ставати прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані оголошення

Оголошення може бути утримане, приховане, поміщене в карантин, відкликане або інакше недоступне на
публічних поверхнях установлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих оголошень. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
оголошення зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політики ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого контенту або видалених оголошень.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після крайнього строку попередження все ще відносить видавця до
порогу потенційного блокування, ClawHub може автоматично застосувати дію щодо облікового запису.
Сигнали перевірки з нижчою впевненістю та обмежені в часі залишаються поза автоматичним
застосуванням санкцій.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API tokens ClawHub. Якщо автентифікація CLI
починає давати збої після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
скористайтеся [формою апеляції ClawHub](https://appeals.openclaw.ai/) для розгляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вивід сканування, виправте оголошення, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацювань і підвищити довіру користувачів:

- підтримуйте точність назв, резюме, тегів і журналів змін
- декларуйте потрібні змінні середовища й дозволи
- уникайте обфускованих команд установлення
- за можливості додавайте посилання на вихідний код
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
