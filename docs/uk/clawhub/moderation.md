# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні пошуку й установлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо
облікових записів допомагають захистити користувачів, коли реліз або обліковий запис
здається небезпечним, оманливим або таким, що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику дивіться в
[Аудити безпеки](/uk/clawhub/security-audits).

Дивіться також [Безпека](/uk/clawhub/security) і
[Прийнятне використання](/uk/clawhub/acceptable-usage). Для питань щодо авторського права
або інших прав на вміст використовуйте [Запити щодо прав на вміст](/uk/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції з установлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельною маркою
- вміст, що порушує [Прийнятне використання](/uk/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про навичку** на сторінці навички або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої
навички або плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій
джерела, посилання на який є в списку. ClawHub не підтримує й не виправляє код
сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у самому
ClawHub. Приклади: помилки у вебсайті, API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри для завантаження/установлення. Не використовуйте
advisories ClawHub для вразливостей у сторонніх навичках або плагінах.

Добрі скарги є конкретними й придатними до дії. Зловживання скаргами саме по собі може
призвести до дій щодо облікового запису.

## Заявки на організації та простори імен

Спори щодо володіння організацією, брендом, областю пакета, ідентифікатором власника
або простором імен мають використовувати процес
[Заявки на організації та простори імен](/uk/clawhub/namespace-claims), а не потік скарг
у продукті чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб персонал ClawHub розглянув
неконфіденційні докази того, що простір імен має бути зарезервований, переданий,
перейменований, прихований, поміщений у карантин, отримав псевдонім або був інакше
переглянутий. Не додавайте секрети, приватні документи, приватні юридичні файли,
документи, що посвідчують особу, API-токени або токени DNS-виклику в публічну issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть призвести до модераційного
утримання видавця чи списку. Коли це відбувається, відповідний вміст може бути
прихований із публічного пошуку або майбутні публікації можуть спочатку ставати
прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує
високоризикові випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше
недоступний на публічних поверхнях установлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник не вирішить
проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків.
Ця діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж список
зможе повернутися на публічні поверхні.

## Бани та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації.
Серйозні зловживання можуть призвести до банів облікового запису, відкликання токенів,
прихованого вмісту або видалених списків. Сигнали тиску зловживань видавця
перевіряються щодня. Сигнали, що досягають порогу потенційного бану ClawHub, можуть
спричинити автоматичне попередження. Якщо наступне придатне сканування після кінцевого
терміну попередження все ще розміщує видавця в порозі потенційного бану, ClawHub може
застосувати дію щодо облікового запису автоматично. Сигнали з нижчою впевненістю та
обмежені в часі сигнали перегляду не потрапляють до автоматичного застосування.

Видалені, забанені або вимкнені облікові записи не можуть використовувати API-токени
ClawHub. Якщо автентифікація CLI починає не вдаватися після дії щодо облікового запису,
увійдіть у вебінтерфейс, щоб переглянути стан облікового запису. Якщо вхід або
звичайний доступ CLI заблоковано баном чи вимкненим обліковим записом, використовуйте
[форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію навички або плагіна
шкідливою, завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, коротких описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд установлення
- посилайтеся на джерело, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
