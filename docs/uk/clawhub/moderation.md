# Модерація та безпека облікового запису

ClawHub відкритий для публікації, але публічні поверхні виявлення та встановлення все ще
потребують запобіжників. Скарги, модераційні утримання, приховані списки та дії щодо
облікового запису допомагають захистити користувачів, коли реліз або обліковий запис
виглядає небезпечним, оманливим або таким, що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику дивіться
[Аудити безпеки](/clawhub/security-audits).

Дивіться також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). Для питань щодо авторського права або інших
прав на контент використовуйте [Запити щодо прав на контент](/clawhub/content-rights).

## Скарги

Користувачі, які ввійшли в систему, можуть скаржитися на навички, плагіни та пакети.

Використовуйте скарги ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі списки
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції зі встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельними марками
- контент, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Поскаржитися на навичку** на сторінці навички або команду/API
для скарг на пакети.

Не використовуйте скарги ClawHub для вразливостей у власному вихідному коді сторонньої навички або
плагіна. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який наведено в списку. ClawHub не підтримує й не виправляє
код сторонніх навичок або плагінів.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади включають помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх навичках або плагінах.

Хороші скарги є конкретними та придатними до дії. Зловживання скаргами саме по собі може призвести до
дій щодо облікового запису.

## Претензії щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, handle власника або простором імен мають
використовувати процес [Претензії щодо організацій і просторів імен](/clawhub/namespace-claims), а не
вбудований у продукт потік скарг чи форму апеляції облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб команда ClawHub переглянула неконфіденційні докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, пов’язаний псевдонімом
або інакше переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, API-токени або токени DNS challenge у
публічний issue.

## Модераційні утримання

Деякі серйозні знахідки або проблеми з політикою можуть помістити видавця або список під
модераційне утримання. Коли це стається, зачеплений контент може бути прихований від публічного
виявлення або майбутні публікації можуть спочатку бути прихованими, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub вирішує високоризикові
випадки. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані списки

Список може бути утриманий, прихований, поміщений у карантин, відкликаний або інакше недоступний на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не вирішить проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих списків. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
список зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого контенту або видалених списків.
Сигнали тиску зловживань видавця перевіряються щодня. Сигнали, що досягають
порогу ClawHub для потенційного блокування, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після дедлайну попередження все ще відносить видавця до
порогу потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали перегляду з нижчою впевненістю та обмежені в часі залишаються поза автоматичним
застосуванням.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збій після дії щодо облікового запису, увійдіть у вебінтерфейс, щоб переглянути стан
облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через бан або вимкнений обліковий запис,
використовуйте [форму апеляції ClawHub](https://appeals.openclaw.ai/) для перегляду відновлення.

Якщо електронний лист, спричинений сканером, називає версію навички або плагіна шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для плагінів додайте
`--kind plugin`. Перегляньте вивід сканування, виправте список, збільште номер версії
та завантажте виправлену версію.

## Рекомендації для видавців

Щоб зменшити хибні спрацювання та підвищити довіру користувачів:

- підтримуйте точність назв, описів, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- посилайтеся на вихідний код, коли це можливо
- використовуйте пробні запуски перед публікацією плагінів
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
