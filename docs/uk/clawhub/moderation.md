# Модерація та безпека облікових записів

ClawHub відкритий до публікації, але публічне виявлення та поверхні встановлення все одно
потребують запобіжників. Звіти, модераційні утримання, приховані оголошення та дії щодо облікових записів
допомагають захистити користувачів, коли реліз або обліковий запис здається небезпечним, оманливим або таким,
що порушує політику.

Ця сторінка описує модерацію та стан облікового запису. Про мітки аудиту, як-от
`Pass`, `Review`, `Warn`, `Malicious`, і рівень ризику див.
[Аудити безпеки](/clawhub/security-audits).

Див. також [Безпека](/clawhub/security) і
[Прийнятне використання](/clawhub/acceptable-usage). З питань авторського права або інших прав
на вміст використовуйте [Запити щодо прав на вміст](/clawhub/content-rights).

## Звіти

Користувачі, що ввійшли в систему, можуть повідомляти про skills, plugins і пакети.

Використовуйте звіти ClawHub лише для небезпечного вмісту маркетплейсу, наприклад:

- шкідливі оголошення
- оманливі метадані
- незадекларовані облікові дані або вимоги до дозволів
- підозрілі інструкції встановлення
- видавання себе за іншу особу чи організацію
- недобросовісні реєстрації або зловживання торговельними марками
- вміст, що порушує [Прийнятне використання](/clawhub/acceptable-usage)

Використовуйте кнопку **Повідомити про skill** на сторінці skill або команду/API
для повідомлення про пакети.

Не використовуйте звіти ClawHub для вразливостей у власному вихідному коді стороннього skill чи
plugin. Повідомляйте про них безпосередньо видавцю або в репозиторій вихідного коду,
посилання на який є в оголошенні. ClawHub не підтримує й не виправляє
код сторонніх skill чи plugin.

GitHub Security Advisories для `openclaw/clawhub` призначені для вразливостей у
самому ClawHub. Приклади: помилки на вебсайті, в API, CLI, реєстрі, автентифікації,
скануванні, модерації або межах довіри завантаження/встановлення. Не використовуйте advisories ClawHub
для вразливостей у сторонніх skills чи plugins.

Добрі звіти є конкретними й придатними до дії. Зловживання повідомленнями саме по собі може призвести до
дій щодо облікового запису.

## Заяви щодо організацій і просторів імен

Спори щодо володіння організацією, брендом, областю пакета, дескриптором власника або простором імен мають
використовувати процес [Заяви щодо організацій і просторів імен](/clawhub/namespace-claims), а не
внутрішній потік повідомлень у продукті чи форму апеляції щодо облікового запису.

Використовуйте цей процес, коли вам потрібно, щоб співробітники ClawHub переглянули неконфіденційні докази того, що
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або інакше переглянутий. Не включайте секрети, приватні документи, приватні юридичні
файли, особисті документи, що посвідчують особу, токени API або токени DNS-перевірки в
публічне issue.

## Модераційні утримання

Деякі серйозні висновки або порушення політики можуть помістити видавця чи оголошення під
модераційне утримання. Коли це трапляється, відповідний вміст може бути приховано від публічного
виявлення або майбутні публікації можуть починатися як приховані, доки проблему не буде розглянуто.

Модераційні утримання призначені для захисту користувачів, поки ClawHub розв’язує випадки
високого ризику. Їх також можна зняти, коли підтверджено хибне спрацювання.

## Приховані або заблоковані оголошення

Оголошення може бути утримане, приховане, поміщене в карантин, відкликане або інакше недоступне на
публічних поверхнях встановлення.

Якщо ви бачите один із цих станів, не встановлюйте реліз, доки власник
не розв’яже проблему або модерація не відновить його.

Власники все ще можуть бачити діагностику для власних утриманих або прихованих оголошень. Ця
діагностика допомагає пояснити, що сталося і що потрібно змінити, перш ніж
оголошення зможе повернутися на публічні поверхні.

## Блокування та стан облікового запису

Облікові записи, що порушують політику ClawHub, можуть втратити доступ до публікації. Серйозні зловживання можуть
призвести до блокування облікового запису, відкликання токенів, прихованого вмісту або видалених оголошень.
Сигнали тиску зловживань видавців перевіряються щодня. Сигнали, що досягають
порогу потенційного блокування ClawHub, можуть спричинити автоматичне попередження. Якщо наступне
придатне сканування після кінцевого строку попередження все ще поміщає видавця в
поріг потенційного блокування, ClawHub може застосувати дію щодо облікового запису автоматично.
Сигнали нижчої впевненості та обмежені часові сигнали перевірки не входять до автоматичного
застосування заходів.

Видалені, заблоковані або вимкнені облікові записи не можуть використовувати API-токени ClawHub. Якщо автентифікація CLI
починає давати збої після дії щодо облікового запису, увійдіть у веб-UI, щоб переглянути
стан облікового запису. Якщо вхід або звичайний доступ CLI заблоковано через заблокований або вимкнений обліковий запис,
скористайтеся [формою апеляції ClawHub](https://appeals.openclaw.ai/) для розгляду відновлення.

Якщо електронний лист, ініційований сканером, називає версію skill або plugin шкідливою,
завантажте збережені результати сканування для заблокованої поданої версії:
`clawhub scan download <slug> --version <version>`. Для plugins додайте
`--kind plugin`. Перегляньте вихідні дані сканування, виправте оголошення, збільште номер версії
та завантажте виправлену версію.

## Настанови для видавців

Щоб зменшити кількість хибних спрацьовувань і підвищити довіру користувачів:

- підтримуйте точність назв, підсумків, тегів і журналів змін
- декларуйте потрібні змінні середовища та дозволи
- уникайте обфускованих команд встановлення
- додавайте посилання на вихідний код, коли можливо
- використовуйте пробні запуски перед публікацією plugins
- відповідайте чітко, якщо користувачі або модератори запитують про поведінку релізу
