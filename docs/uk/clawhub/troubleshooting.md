# Усунення несправностей

## `clawhub login` відкриває браузер, але ніколи не завершується

CLI запускає короткочасний локальний сервер зворотного виклику під час входу через браузер.

- Переконайтеся, що ваш браузер може відкрити `http://127.0.0.1:<port>/callback`.
- Перевірте локальний брандмауер, VPN і правила проксі, якщо зворотний виклик не надходить.
- У безголових середовищах створіть API-токен у вебінтерфейсі ClawHub і виконайте:

```bash
clawhub login --token clh_...
```

## `whoami` або `publish` повертає `Unauthorized` (401)

- Увійдіть знову за допомогою `clawhub login`.
- Якщо ви використовуєте власний шлях конфігурації, переконайтеся, що `CLAWHUB_CONFIG_PATH` вказує на
  файл, який містить ваш поточний токен.
- Якщо ви використовуєте API-токен, переконайтеся, що його не було відкликано у вебінтерфейсі.

## Пошук або встановлення повертає `Rate limit exceeded` (429)

Прочитайте інформацію про повторну спробу у відповіді:

- `Retry-After`: кількість секунд, які потрібно зачекати перед повторною спробою.
- `RateLimit-Limit`: ліміт, застосований до цього запиту.
- `RateLimit-Remaining`: ваш точний залишковий бюджет, коли заголовок присутній. На `429` він дорівнює `0`.
- `RateLimit-Reset` або `X-RateLimit-Reset`: час скидання.

Якщо багато користувачів спільно використовують одну вихідну IP-адресу, анонімні IP-ліміти можуть спрацювати, навіть коли кожна
особа надсилає лише кілька запитів. Увійдіть, де це можливо, і повторіть спробу після
вказаної затримки.

## Пошук або встановлення не працює за проксі

CLI враховує стандартні змінні проксі:

```bash
export HTTPS_PROXY=http://proxy.example.com:3128
clawhub search "my query"
```

Підтримувані імена включають `HTTPS_PROXY`, `HTTP_PROXY`, `https_proxy` і
`http_proxy`.

## Skill не відображається в пошуку

- Перевірте точний слаг або сторінку власника, якщо ви її знаєте.
- Переконайтеся, що реліз публічний і не утримується через сканування або модерацію.
- Якщо ви власник Skill, увійдіть і перевірте його:

```bash
clawhub inspect @openclaw/demo
```

Видима для власника діагностика може пояснити стан сканування, шлюзу завантаження або модерації.

## Публікація не вдається, бо бракує обов’язкових метаданих

Для Skills перевірте frontmatter у `SKILL.md`. Обов’язкові змінні середовища та
інструменти слід оголошувати, щоб користувачі й сканери могли зрозуміти пакет.

Для plugins перевірте метадані сумісності в `package.json`. Публікації code-plugin
потребують полів сумісності OpenClaw, як-от `openclaw.compat.pluginApi` і
`openclaw.build.openclawVersion`.

Спочатку перегляньте попередній варіант корисного навантаження публікації:

```bash
clawhub package publish <source> --family code-plugin --dry-run
```

## Публікація не вдається через помилку власника GitHub або джерела

ClawHub використовує ідентичність GitHub і атрибуцію джерела, щоб пов’язувати пакети з їхніми
публікаторами.

- Переконайтеся, що ви ввійшли за допомогою облікового запису GitHub, який володіє пакетом або може його публікувати.
- Перевірте, що URL джерела є публічним або доступним для ClawHub.
- Для джерел GitHub використовуйте `owner/repo`, `owner/repo@ref` або повний URL GitHub.

## Публікація не вдається, бо простір імен заявлений або зарезервований

Якщо публікація не вдається, бо дескриптор власника, простір імен організації, scope пакета, слаг Skill
або назва пакета вже заявлені чи зарезервовані, спершу переконайтеся, що ви
публікуєте від власника, який відповідає простору імен. Для plugin-пакетів
імена зі scope, як-от `@example-org/example-plugin`, мають публікуватися від
відповідного власника `example-org`.

Якщо ви вважаєте, що ваша організація, проєкт або бренд є правомірним власником простору імен, але
ви не можете керувати поточним власником ClawHub, відкрийте
[звернення щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
з публічними, неконфіденційними доказами. Див.
[Заявки щодо організацій і просторів імен](/clawhub/namespace-claims), щоб дізнатися, які докази надати та що
не слід додавати до публічних звернень.

## `sync` повідомляє, що Skills не знайдено

`sync` шукає папки, які містять `SKILL.md` або `skill.md`.

Спрямуйте його на кореневі каталоги, які хочете сканувати:

```bash
clawhub sync --root /path/to/skills
```

Спочатку перегляньте попередній варіант, якщо не впевнені, що буде опубліковано:

```bash
clawhub sync --all --dry-run --no-input
```

## `update` відмовляється через локальні зміни

Локальні файли не відповідають жодній версії, відомій ClawHub. Виберіть один варіант:

- Зберегти локальні зміни й пропустити оновлення.
- Перезаписати опублікованою версією:

```bash
clawhub update @openclaw/demo --force
```

- Опублікувати свою відредаговану копію як новий слаг або fork.

## Встановлення plugin не вдається в OpenClaw

- Використайте явне джерело ClawHub:

```bash
openclaw plugins install clawhub:<package>
```

- Перевірте сторінку деталей пакета щодо статусу сканування та метаданих сумісності.
- Переконайтеся, що ваша версія OpenClaw задовольняє оголошений пакетом
  діапазон сумісності.
- Якщо пакет прихований, утримується або заблокований, його може бути неможливо встановити, доки
  власник не розв’яже проблему.

## Запити до публічного API не вдаються

- Дотримуйтеся заголовків повторної спроби `429` і кешуйте публічні відповіді списків/пошуку.
- Направляйте користувачів назад до канонічного запису ClawHub.
- Не дзеркальте прихований, приватний, утримуваний або заблокований модерацією вміст поза
  поверхнею публічного API.

Див. [HTTP API](/clawhub/http-api) для деталей кінцевих точок.
