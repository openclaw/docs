# Усунення несправностей

## `clawhub login` відкриває браузер, але процес не завершується

Під час входу через браузер CLI запускає короткочасний локальний сервер зворотного виклику.

- Переконайтеся, що браузер може підключитися до `http://127.0.0.1:<port>/callback`.
- Якщо зворотний виклик не надходить, перевірте правила локального брандмауера, VPN і проксі.
- У середовищах без графічного інтерфейсу створіть токен API у вебінтерфейсі ClawHub і виконайте:

```bash
clawhub login --token clh_...
```

## `whoami` або `publish` повертає `Unauthorized` (401)

- Увійдіть знову за допомогою `clawhub login`.
- Якщо використовується власний шлях до конфігурації, переконайтеся, що `CLAWHUB_CONFIG_PATH` указує на
  файл із поточним токеном.
- Якщо використовується токен API, переконайтеся, що його не було відкликано у вебінтерфейсі.

## Пошук або встановлення повертає `Rate limit exceeded` (429)

Перегляньте відомості про повторну спробу у відповіді:

- `Retry-After`: кількість секунд очікування перед повторною спробою.
- `RateLimit-Limit`: обмеження, застосоване до цього запиту.
- `RateLimit-Remaining`: точний залишок ліміту, якщо цей заголовок наявний. Для `429` це `0`.
- `RateLimit-Reset` або `X-RateLimit-Reset`: час скидання.

Якщо багато користувачів спільно використовують одну вихідну IP-адресу, анонімні обмеження за IP-адресою можуть бути досягнуті, навіть коли кожен
користувач надсилає лише кілька запитів. За можливості увійдіть у систему та повторіть спробу після
зазначеної затримки.

## Пошук або встановлення не працює через проксі

CLI враховує стандартні змінні проксі:

```bash
export HTTPS_PROXY=http://proxy.example.com:3128
clawhub search "my query"
```

Підтримувані назви включають `HTTPS_PROXY`, `HTTP_PROXY`, `https_proxy` і
`http_proxy`.

## Навичка не відображається в пошуку

- Якщо відомий точний slug або сторінка власника, перевірте їх.
- Переконайтеся, що випуск загальнодоступний і не затриманий скануванням або модерацією.
- Якщо навичка належить вам, увійдіть у систему та перевірте її:

```bash
clawhub inspect @openclaw/demo
```

Діагностичні відомості, доступні власнику, можуть пояснити стан сканування, шлюзу завантаження або модерації.

## Публікація завершується помилкою через відсутність обов’язкових метаданих

Для навичок перевірте frontmatter у `SKILL.md`. Обов’язкові змінні середовища та
інструменти слід оголосити, щоб користувачі й сканери могли зрозуміти вміст пакета.

Для плагінів перевірте метадані сумісності в `package.json`. Для публікації плагінів із кодом
потрібні поля сумісності з OpenClaw, як-от `openclaw.compat.pluginApi` і
`openclaw.build.openclawVersion`.

Спочатку перегляньте дані публікації:

```bash
clawhub package publish <source> --family code-plugin --dry-run
```

## Публікація завершується помилкою власника або джерела GitHub

ClawHub використовує ідентичність GitHub і зазначення джерела, щоб пов’язувати пакети з їхніми
видавцями.

- Переконайтеся, що ви ввійшли за допомогою облікового запису GitHub, який володіє пакетом або має право його публікувати.
- Перевірте, чи URL-адреса джерела загальнодоступна або доступна для ClawHub.
- Для джерел GitHub використовуйте `owner/repo`, `owner/repo@ref` або повну URL-адресу GitHub.

## Публікація завершується помилкою, оскільки простір імен зайнятий або зарезервований

Якщо публікація завершується помилкою через те, що ідентифікатор власника, простір імен організації, область пакета, slug навички
або назва пакета вже зайняті чи зарезервовані, спочатку переконайтеся, що
публікація виконується від імені власника, який відповідає простору імен. Для пакетів плагінів
імена з областю, як-от `@example-org/example-plugin`, потрібно публікувати від імені
відповідного власника `example-org`.

Якщо ви вважаєте, що ваша організація, проєкт або бренд є законним власником простору імен, але
не можете керувати поточним власником ClawHub, створіть
[звернення щодо права на простір імен організації](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
із загальнодоступними доказами, що не містять конфіденційної інформації. Рекомендації щодо доказів і відомостей,
які не слід публікувати у відкритих зверненнях, див. у розділі
[Права на простори імен організацій](/clawhub/namespace-claims).

## `sync` повідомляє, що навичок не знайдено

`sync` шукає папки, що містять `SKILL.md` або `skill.md`.

Укажіть кореневі каталоги, які потрібно просканувати:

```bash
clawhub sync --root /path/to/skills
```

Якщо невідомо, що саме буде опубліковано, спочатку перегляньте результат:

```bash
clawhub sync --all --dry-run --no-input
```

## `update` відмовляється працювати через локальні зміни

Локальні файли не відповідають жодній версії, відомій ClawHub. Виберіть один із варіантів:

- Збережіть локальні зміни та пропустіть оновлення.
- Перезапишіть файли опублікованою версією:

```bash
clawhub update @openclaw/demo --force
```

- Опублікуйте змінену копію з новим slug або як форк.

## Не вдається встановити плагін в OpenClaw

- Укажіть джерело ClawHub явно:

```bash
openclaw plugins install clawhub:<package>
```

- Перевірте на сторінці відомостей про пакет стан сканування та метадані сумісності.
- Переконайтеся, що версія OpenClaw відповідає заявленому пакетом
  діапазону сумісності.
- Якщо пакет прихований, затриманий або заблокований, його може бути неможливо встановити, доки
  власник не усуне проблему.

## Запити до загальнодоступного API завершуються помилкою

- Дотримуйтеся вказівок заголовків повторної спроби `429` і кешуйте загальнодоступні відповіді зі списками та результатами пошуку.
- Спрямовуйте користувачів на канонічну сторінку ClawHub.
- Не дублюйте прихований, приватний, затриманий або заблокований модерацією вміст поза межами
  загальнодоступної поверхні API.

Докладні відомості про кінцеві точки див. у розділі [HTTP API](/clawhub/http-api).
