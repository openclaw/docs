# Усунення несправностей

## `clawhub login` відкриває браузер, але ніколи не завершується

CLI запускає короткочасний локальний callback-сервер під час входу через браузер.

- Переконайтеся, що ваш браузер може відкрити `http://127.0.0.1:<port>/callback`.
- Перевірте правила локального фаєрвола, VPN і проксі, якщо callback не надходить.
- У headless-середовищах створіть API-токен у вебінтерфейсі ClawHub і виконайте:

```bash
clawhub login --token clh_...
```

## `whoami` або `publish` повертає `Unauthorized` (401)

- Увійдіть знову за допомогою `clawhub login`.
- Якщо ви використовуєте власний шлях до конфігурації, переконайтеся, що `CLAWHUB_CONFIG_PATH` вказує на
  файл, який містить ваш поточний токен.
- Якщо ви використовуєте API-токен, переконайтеся, що його не було відкликано у вебінтерфейсі.

## Пошук або встановлення повертає `Rate limit exceeded` (429)

Прочитайте інформацію про повторну спробу у відповіді:

- `Retry-After`: скільки секунд чекати перед повторною спробою.
- `RateLimit-Limit`: ліміт, застосований до цього запиту.
- `RateLimit-Remaining`: ваш точний залишковий бюджет, коли цей заголовок присутній. Для `429` це `0`.
- `RateLimit-Reset` або `X-RateLimit-Reset`: час скидання.

Якщо багато користувачів використовують одну вихідну IP-адресу, анонімні IP-ліміти можуть спрацювати, навіть коли кожна
людина надсилає лише кілька запитів. Увійдіть, якщо можливо, і повторіть спробу після
повідомленої затримки.

## Пошук або встановлення не працює за проксі

CLI враховує стандартні змінні проксі:

```bash
export HTTPS_PROXY=http://proxy.example.com:3128
clawhub search "my query"
```

Підтримувані назви включають `HTTPS_PROXY`, `HTTP_PROXY`, `https_proxy` і
`http_proxy`.

## Навичка не з’являється в пошуку

- Перевірте точний slug або сторінку власника, якщо ви її знаєте.
- Переконайтеся, що реліз є публічним і не затриманий через сканування або модерацію.
- Якщо ви володієте навичкою, увійдіть і перегляньте її:

```bash
clawhub inspect @openclaw/demo
```

Діагностика, видима власнику, може пояснити стан сканування, upload-gate або модерації.

## Публікація не вдається, бо бракує обов’язкових метаданих

Для навичок перевірте frontmatter у `SKILL.md`. Обов’язкові змінні середовища та
інструменти мають бути оголошені, щоб користувачі й сканери могли зрозуміти пакет.

Для plugins перевірте метадані сумісності в `package.json`. Публікації code-plugin
потребують полів сумісності OpenClaw, як-от `openclaw.compat.pluginApi` і
`openclaw.build.openclawVersion`.

Спочатку перегляньте payload публікації:

```bash
clawhub package publish <source> --family code-plugin --dry-run
```

## Публікація не вдається через помилку власника GitHub або джерела

ClawHub використовує ідентичність GitHub та атрибуцію джерела, щоб пов’язувати пакети з їхніми
публікаторами.

- Переконайтеся, що ви увійшли з GitHub-акаунтом, який володіє пакетом або може його публікувати.
- Перевірте, що URL джерела є публічним або доступним для ClawHub.
- Для джерел GitHub використовуйте `owner/repo`, `owner/repo@ref` або повну URL-адресу GitHub.

## Публікація не вдається, бо namespace заявлений або зарезервований

Якщо публікація не вдається, бо owner handle, namespace організації, scope пакета, slug навички
або назва пакета вже заявлені чи зарезервовані, спочатку переконайтеся, що ви
публікуєте від імені власника, який відповідає namespace. Для пакетів Plugin
scoped-назви, як-от `@example-org/example-plugin`, мають публікуватися від імені
відповідного власника `example-org`.

Якщо ви вважаєте, що ваша організація, проєкт або бренд є законним власником namespace, але
ви не можете керувати поточним власником ClawHub, відкрийте
[заявку Org / Namespace Claim](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
з публічними, нечутливими доказами. Див.
[Заявки на організації та namespaces](/clawhub/namespace-claims), щоб дізнатися, які докази надавати і що
не варто включати в публічні issue.

## `sync` повідомляє, що Skills не знайдено

`sync` шукає папки, що містять `SKILL.md` або `skill.md`.

Вкажіть корені, які потрібно просканувати:

```bash
clawhub sync --root /path/to/skills
```

Спочатку перегляньте результат, якщо не впевнені, що буде опубліковано:

```bash
clawhub sync --all --dry-run --no-input
```

## `update` відмовляється виконуватися через локальні зміни

Локальні файли не відповідають жодній версії, відомій ClawHub. Виберіть один варіант:

- Збережіть локальні редагування й пропустіть оновлення.
- Перезапишіть опублікованою версією:

```bash
clawhub update @openclaw/demo --force
```

- Опублікуйте свою відредаговану копію як новий slug або fork.

## Встановлення Plugin не вдається в OpenClaw

- Використайте явне джерело ClawHub:

```bash
openclaw plugins install clawhub:<package>
```

- Перевірте сторінку деталей пакета щодо стану сканування та метаданих сумісності.
- Переконайтеся, що ваша версія OpenClaw відповідає заявленому пакетом
  діапазону сумісності.
- Якщо пакет прихований, затриманий або заблокований, його може бути неможливо встановити, доки
  власник не вирішить проблему.

## Запити до публічного API не вдаються

- Дотримуйтеся заголовків повторної спроби для `429` і кешуйте публічні відповіді списку/пошуку.
- Спрямовуйте користувачів назад до канонічного списингу ClawHub.
- Не дзеркальте прихований, приватний, затриманий або заблокований модерацією вміст поза
  поверхнею публічного API.

Див. [HTTP API](/clawhub/http-api), щоб дізнатися подробиці про endpoint.
