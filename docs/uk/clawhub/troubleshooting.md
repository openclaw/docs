# Усунення неполадок

## `clawhub login` відкриває браузер, але ніколи не завершується

CLI запускає короткочасний локальний сервер зворотного виклику під час входу через браузер.

- Переконайтеся, що ваш браузер може відкрити `http://127.0.0.1:<port>/callback`.
- Перевірте правила локального фаєрвола, VPN і проксі, якщо зворотний виклик не надходить.
- У середовищах без графічного інтерфейсу створіть API-токен у вебінтерфейсі ClawHub і виконайте:

```bash
clawhub login --token clh_...
```

## `whoami` або `publish` повертає `Unauthorized` (401)

- Увійдіть знову за допомогою `clawhub login`.
- Якщо ви використовуєте власний шлях конфігурації, переконайтеся, що `CLAWHUB_CONFIG_PATH` вказує на
  файл, який містить ваш поточний токен.
- Якщо ви використовуєте API-токен, переконайтеся, що його не відкликано у вебінтерфейсі.

## Пошук або встановлення повертає `Rate limit exceeded` (429)

Прочитайте інформацію про повторну спробу у відповіді:

- `Retry-After`: скільки секунд чекати перед повторною спробою.
- `RateLimit-Limit`: ліміт, застосований до цього запиту.
- `RateLimit-Remaining`: ваш точний залишковий ліміт, коли заголовок присутній. Для `429` він дорівнює `0`.
- `RateLimit-Reset` або `X-RateLimit-Reset`: час скидання.

Якщо багато користувачів використовують одну вихідну IP-адресу, анонімні IP-ліміти можуть спрацювати навіть тоді, коли кожна
людина надсилає лише кілька запитів. Увійдіть, де це можливо, і повторіть спробу після
повідомленої затримки.

## Пошук або встановлення не працює за проксі

CLI враховує стандартні змінні проксі:

```bash
export HTTPS_PROXY=http://proxy.example.com:3128
clawhub search "my query"
```

Підтримувані назви включають `HTTPS_PROXY`, `HTTP_PROXY`, `https_proxy` і
`http_proxy`.

## Навичка не з’являється в пошуку

- Перевірте точний slug або сторінку власника, якщо ви її знаєте.
- Переконайтеся, що реліз публічний і не затриманий скануванням або модерацією.
- Якщо ви є власником навички, увійдіть і перевірте її:

```bash
clawhub inspect @openclaw/demo
```

Діагностика, видима власнику, може пояснити стан сканування, блокування завантаження або модерації.

## Публікація не вдається, бо бракує обов’язкових метаданих

Для навичок перевірте frontmatter у `SKILL.md`. Обов’язкові змінні середовища й
інструменти мають бути оголошені, щоб користувачі та сканери могли зрозуміти пакет.

Для плагінів перевірте метадані сумісності в `package.json`. Публікації code-plugin
потребують полів сумісності OpenClaw, як-от `openclaw.compat.pluginApi` і
`openclaw.build.openclawVersion`.

Спершу перегляньте payload публікації:

```bash
clawhub package publish <source> --family code-plugin --dry-run
```

## Публікація не вдається через помилку власника GitHub або джерела

ClawHub використовує ідентичність GitHub і атрибуцію джерела, щоб пов’язати пакети з їхніми
публікаторами.

- Переконайтеся, що ви ввійшли з обліковим записом GitHub, який володіє пакетом або може його публікувати.
- Перевірте, що URL джерела публічний або доступний для ClawHub.
- Для джерел GitHub використовуйте `owner/repo`, `owner/repo@ref` або повний URL GitHub.

## Публікація не вдається, бо namespace уже заявлений або зарезервований

Якщо публікація не вдається через те, що handle власника, namespace організації, scope пакета, slug навички
або назва пакета вже заявлені або зарезервовані, спершу переконайтеся, що ви
публікуєте від імені власника, який відповідає namespace. Для пакетів плагінів
scoped-назви, як-от `@example-org/example-plugin`, мають публікуватися від імені
відповідного власника `example-org`.

Якщо ви вважаєте, що ваша організація, проєкт або бренд є законним власником namespace, але
ви не можете керувати поточним власником ClawHub, відкрийте
[issue для заявки на організацію / namespace](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
з публічними, неконфіденційними доказами. Див.
[Заявки на організації та namespace](/clawhub/namespace-claims), щоб дізнатися, які докази надати й що
не слід додавати до публічних issue.

## `sync` каже, що навичок не знайдено

`sync` шукає папки, що містять `SKILL.md` або `skill.md`.

Вкажіть кореневі папки, які потрібно просканувати:

```bash
clawhub sync --root /path/to/skills
```

Спершу перегляньте результат, якщо не впевнені, що буде опубліковано:

```bash
clawhub sync --all --dry-run --no-input
```

## `update` відмовляється через локальні зміни

Локальні файли не відповідають жодній версії, відомій ClawHub. Виберіть один варіант:

- Залишити локальні зміни й пропустити оновлення.
- Перезаписати опублікованою версією:

```bash
clawhub update @openclaw/demo --force
```

- Опублікувати вашу відредаговану копію як новий slug або fork.

## Встановлення плагіна в OpenClaw не вдається

- Використовуйте явне джерело ClawHub:

```bash
openclaw plugins install clawhub:<package>
```

- Перевірте сторінку деталей пакета щодо статусу сканування та метаданих сумісності.
- Переконайтеся, що ваша версія OpenClaw відповідає заявленому пакетом
  діапазону сумісності.
- Якщо пакет прихований, затриманий або заблокований, його може бути неможливо встановити, доки
  власник не вирішить проблему.

## Публічні API-запити не виконуються

- Дотримуйтеся заголовків повторної спроби `429` і кешуйте публічні відповіді списку/пошуку.
- Спрямовуйте користувачів назад до канонічного списку ClawHub.
- Не віддзеркалюйте прихований, приватний, затриманий або заблокований модерацією вміст поза межами
  публічної поверхні API.

Див. [HTTP API](/clawhub/http-api), щоб дізнатися подробиці про endpoint.
