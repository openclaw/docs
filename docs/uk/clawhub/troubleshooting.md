# Усунення несправностей

## `clawhub login` відкриває браузер, але ніколи не завершується

CLI запускає короткочасний локальний сервер зворотного виклику під час входу через браузер.

- Переконайтеся, що ваш браузер може відкрити `http://127.0.0.1:<port>/callback`.
- Перевірте локальний брандмауер, VPN і правила проксі, якщо зворотний виклик не надходить.
- У безголових середовищах створіть API-токен у вебінтерфейсі ClawHub і виконайте:

```bash
clawhub login --token clh_...
```

## `whoami` або `publish` повертає `Unauthorized` (401)

- Увійдіть знову за допомогою `clawhub login`.
- Якщо ви використовуєте власний шлях до конфігурації, переконайтеся, що `CLAWHUB_CONFIG_PATH` вказує на
  файл, який містить ваш поточний токен.
- Якщо ви використовуєте API-токен, переконайтеся, що його не було відкликано у вебінтерфейсі.

## Пошук або встановлення повертає `Rate limit exceeded` (429)

Прочитайте інформацію про повторну спробу у відповіді:

- `Retry-After`: кількість секунд очікування перед повторною спробою.
- `RateLimit-Limit`: ліміт, застосований до цього запиту.
- `RateLimit-Remaining`: ваш точний залишковий бюджет, коли заголовок присутній. На `429` це `0`.
- `RateLimit-Reset` або `X-RateLimit-Reset`: час скидання.

Якщо багато користувачів спільно використовують одну вихідну IP-адресу, анонімні IP-ліміти можуть спрацювати навіть тоді, коли кожна
людина надсилає лише кілька запитів. Увійдіть, де це можливо, і повторіть спробу після
повідомленої затримки.

## Пошук або встановлення не працює за проксі

CLI враховує стандартні змінні проксі:

```bash
export HTTPS_PROXY=http://proxy.example.com:3128
clawhub search "my query"
```

Підтримувані назви включають `HTTPS_PROXY`, `HTTP_PROXY`, `https_proxy` і
`http_proxy`.

## Навичка не з’являється в пошуку

- Перевірте точний slug або сторінку власника, якщо ви їх знаєте.
- Переконайтеся, що випуск публічний і не затриманий скануванням або модерацією.
- Якщо ви є власником навички, увійдіть і перевірте її:

```bash
clawhub inspect @openclaw/demo
```

Діагностика, видима власнику, може пояснити стан сканування, шлюзу завантаження або модерації.

## Публікація не вдається, бо бракує обов’язкових метаданих

Для навичок перевірте frontmatter у `SKILL.md`. Обов’язкові змінні середовища й
інструменти мають бути оголошені, щоб користувачі та сканери могли зрозуміти пакет.

Для плагінів перевірте метадані сумісності в `package.json`. Для публікації кодових плагінів
потрібні поля сумісності OpenClaw, як-от `openclaw.compat.pluginApi` і
`openclaw.build.openclawVersion`.

Спочатку перегляньте попередній варіант payload публікації:

```bash
clawhub package publish <source> --family code-plugin --dry-run
```

## Публікація не вдається через помилку власника GitHub або джерела

ClawHub використовує ідентичність GitHub і атрибуцію джерела, щоб пов’язувати пакети з їхніми
видавцями.

- Переконайтеся, що ви ввійшли з обліковим записом GitHub, який володіє пакетом або може його публікувати.
- Перевірте, що URL джерела є публічним або доступним для ClawHub.
- Для джерел GitHub використовуйте `owner/repo`, `owner/repo@ref` або повну URL-адресу GitHub.

## Публікація не вдається, бо namespace зайнятий або зарезервований

Якщо публікація не вдається, бо handle власника, namespace організації, scope пакета, slug навички
або назва пакета вже зайняті чи зарезервовані, спочатку переконайтеся, що ви
публікуєте від імені власника, який відповідає namespace. Для пакетів плагінів
scoped-назви, як-от `@example-org/example-plugin`, мають публікуватися від імені
відповідного власника `example-org`.

Якщо ви вважаєте, що ваша організація, проєкт або бренд є законним власником namespace, але
ви не можете керувати поточним власником ClawHub, відкрийте
[заявку на організацію / namespace](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
із публічними, неконфіденційними доказами. Див.
[Заявки на організації та namespace](/clawhub/namespace-claims), щоб отримати вказівки щодо доказів і того, що
не слід додавати до публічних issues.

## `sync` повідомляє, що навички не знайдено

`sync` шукає папки, які містять `SKILL.md` або `skill.md`.

Вкажіть корені, які потрібно просканувати:

```bash
clawhub sync --root /path/to/skills
```

Спочатку перегляньте попередній результат, якщо ви не впевнені, що буде опубліковано:

```bash
clawhub sync --all --dry-run --no-input
```

## `update` відмовляється через локальні зміни

Локальні файли не відповідають жодній версії, відомій ClawHub. Виберіть один варіант:

- Залишити локальні правки й пропустити оновлення.
- Перезаписати опублікованою версією:

```bash
clawhub update @openclaw/demo --force
```

- Опублікувати відредаговану копію як новий slug або fork.

## Встановлення плагіна в OpenClaw не вдається

- Використайте явне джерело ClawHub:

```bash
openclaw plugins install clawhub:<package>
```

- Перевірте сторінку деталей пакета щодо стану сканування й метаданих сумісності.
- Переконайтеся, що ваша версія OpenClaw відповідає заявленому пакетом
  діапазону сумісності.
- Якщо пакет прихований, затриманий або заблокований, він може бути недоступним для встановлення, доки
  власник не усуне проблему.

## Публічні API-запити не працюють

- Дотримуйтеся заголовків повторної спроби `429` і кешуйте публічні відповіді списку/пошуку.
- Посилайте користувачів назад на канонічний список ClawHub.
- Не дзеркальте прихований, приватний, затриманий або заблокований модерацією вміст поза
  публічною поверхнею API.

Див. [HTTP API](/clawhub/http-api), щоб отримати подробиці про endpoints.
