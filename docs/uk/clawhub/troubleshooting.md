# Усунення несправностей

## `clawhub login` відкриває браузер, але ніколи не завершується

CLI запускає короткочасний локальний сервер зворотного виклику під час входу через браузер.

- Переконайтеся, що ваш браузер може відкрити `http://127.0.0.1:<port>/callback`.
- Перевірте правила локального брандмауера, VPN і проксі, якщо зворотний виклик не надходить.
- У середовищах без графічного інтерфейсу створіть API-токен у веб-інтерфейсі ClawHub і виконайте:

```bash
clawhub login --token clh_...
```

## `whoami` або `publish` повертає `Unauthorized` (401)

- Увійдіть знову за допомогою `clawhub login`.
- Якщо ви використовуєте власний шлях до конфігурації, переконайтеся, що `CLAWHUB_CONFIG_PATH` вказує на
  файл, який містить ваш поточний токен.
- Якщо ви використовуєте API-токен, переконайтеся, що його не було відкликано у веб-інтерфейсі.

## Пошук або встановлення повертає `Rate limit exceeded` (429)

Прочитайте інформацію про повторну спробу у відповіді:

- `Retry-After`: секунди очікування перед повторною спробою.
- `RateLimit-Limit`: ліміт, застосований до цього запиту.
- `RateLimit-Remaining`: ваш точний залишковий бюджет, коли цей заголовок присутній. Для `429` він дорівнює `0`.
- `RateLimit-Reset` або `X-RateLimit-Reset`: час скидання.

Якщо багато користувачів мають одну вихідну IP-адресу, анонімні IP-ліміти можуть спрацьовувати навіть тоді, коли кожна
людина надсилає лише кілька запитів. Увійдіть, де це можливо, і повторіть спробу після
повідомленої затримки.

## Пошук або встановлення не працює за проксі

CLI враховує стандартні змінні проксі:

```bash
export HTTPS_PROXY=http://proxy.example.com:3128
clawhub search "my query"
```

Підтримувані назви включають `HTTPS_PROXY`, `HTTP_PROXY`, `https_proxy` і
`http_proxy`.

## Навичка не з’являється в пошуку

- Перевірте точний slug або сторінку власника, якщо ви її знаєте.
- Переконайтеся, що реліз публічний і не затриманий скануванням або модерацією.
- Якщо ви є власником навички, увійдіть і перевірте її:

```bash
clawhub inspect @openclaw/demo
```

Діагностика, видима власнику, може пояснити стан сканування, обмеження завантаження або модерації.

## Публікація не вдається, бо бракує обов’язкових метаданих

Для навичок перевірте frontmatter у `SKILL.md`. Обов’язкові змінні середовища й
інструменти мають бути оголошені, щоб користувачі та сканери могли зрозуміти пакет.

Для plugin перевірте метадані сумісності в `package.json`. Публікації code-plugin
потребують полів сумісності з OpenClaw, таких як `openclaw.compat.pluginApi` і
`openclaw.build.openclawVersion`.

Спершу перегляньте payload публікації:

```bash
clawhub package publish <source> --family code-plugin --dry-run
```

## Публікація не вдається через помилку власника GitHub або джерела

ClawHub використовує ідентичність GitHub і атрибуцію джерела, щоб пов’язувати пакети з їхніми
публікаторами.

- Переконайтеся, що ви увійшли з обліковим записом GitHub, який володіє пакетом або може його публікувати.
- Перевірте, що URL джерела є публічним або доступним для ClawHub.
- Для джерел GitHub використовуйте `owner/repo`, `owner/repo@ref` або повний URL GitHub.

## Публікація не вдається, бо простір імен заявлений або зарезервований

Якщо публікація не вдається, бо handle власника, простір імен org, scope пакета, slug навички
або назва пакета вже заявлені чи зарезервовані, спершу переконайтеся, що ви
публікуєте від імені власника, який відповідає простору імен. Для пакетів plugin
scoped-імена, такі як `@example-org/example-plugin`, мають публікуватися від імені
відповідного власника `example-org`.

Якщо ви вважаєте, що ваша org, проєкт або бренд є законним власником простору імен, але
ви не можете керувати поточним власником ClawHub, відкрийте
[issue для заявки на org / простір імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
з публічними, неконфіденційними доказами. Див.
[Заявки на org і простір імен](/uk/clawhub/namespace-claims), щоб дізнатися, які докази потрібні та що
не слід публікувати в публічних issue.

## `sync` каже, що навичок не знайдено

`sync` шукає папки, які містять `SKILL.md` або `skill.md`.

Спрямуйте його на корені, які потрібно просканувати:

```bash
clawhub sync --root /path/to/skills
```

Спершу перегляньте результат, якщо не впевнені, що буде опубліковано:

```bash
clawhub sync --all --dry-run --no-input
```

## `update` відмовляється через локальні зміни

Локальні файли не відповідають жодній версії, про яку знає ClawHub. Виберіть один варіант:

- Залишити локальні зміни та пропустити оновлення.
- Перезаписати опублікованою версією:

```bash
clawhub update @openclaw/demo --force
```

- Опублікувати вашу відредаговану копію як новий slug або fork.

## Встановлення plugin не вдається в OpenClaw

- Використайте явне джерело ClawHub:

```bash
openclaw plugins install clawhub:<package>
```

- Перевірте сторінку деталей пакета щодо стану сканування та метаданих сумісності.
- Переконайтеся, що ваша версія OpenClaw відповідає заявленому пакетом
  діапазону сумісності.
- Якщо пакет прихований, затриманий або заблокований, він може бути недоступний для встановлення, доки
  власник не вирішить проблему.

## Публічні API-запити не вдаються

- Дотримуйтеся заголовків повторної спроби `429` і кешуйте відповіді публічних списків/пошуку.
- Посилайте користувачів назад на канонічний запис ClawHub.
- Не дзеркальте прихований, приватний, затриманий або заблокований модерацією вміст поза
  публічною поверхнею API.

Див. [HTTP API](/uk/clawhub/http-api) для подробиць про endpoint.
