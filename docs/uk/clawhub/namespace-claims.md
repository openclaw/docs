# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug-и Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен здається таким,
що належить реальному проєкту, бренду, екосистемі пакетів або організації, але
вже заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть персонал переглянути його через
[форму звернення щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного перегляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо вважаєте, що персонал ClawHub має перевірити, чи
простір імен потрібно зарезервувати, передати, перейменувати, приховати, ізолювати, додати до нього псевдонім
або іншим чином змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише від імені
  відповідного власника ClawHub
- slug Skills або назва пакета Plugin, що здається імітацією проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, зловмисним або оманливим поза межами спору про право власності,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для перегляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо через публікацію,
перейменування, передачу, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли персонал має вирішити
спір.

## Докази, які варто додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- доказ домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, Skills, Plugin, пакет або звернення ClawHub

Поясніть, що доводить кожне посилання. Персонал має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Що не додавати

Не розміщуйте секрети або приватні докази в публічному зверненні GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або контракти
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти про безпеку або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал персоналу для чутливих доказів.
Скористайтеся цим варіантом замість публікації чутливих матеріалів у відкритому доступі.

## Можливі результати

Залежно від доказів і ризику, персонал ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або ізолювати наявний запис,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Персонал зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
