# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржений у ClawHub, попросіть команду розглянути його
через
[форму заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте вбудовані в продукт
скарги або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкривайте заявку щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен потрібно зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, наприклад `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- slug Skills або назва пакета Plugin, що схожа на спробу видати себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує правомірного власника
  простору імен

Якщо запис небезпечний, шкідливий або оманливий не лише в межах спору про право власності,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin scoped-назви, такі як `@example-org/example-plugin`, мають бути
опубліковані від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли команді потрібно розв’язати
спір.

## Докази, які варто додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або мейнтейнерів
- офіційну документацію проєкту, де названо простір імен
- доказ домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, Skills, Plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має розуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не слід додавати

Не розміщуйте секрети або приватні докази в публічному GitHub issue. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або контракти
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал команди для конфіденційних доказів.
Використовуйте цю опцію замість публічного розміщення конфіденційних матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або переспрямування, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
