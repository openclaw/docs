# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug навичок, імена пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен схожий на такий, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду перевірити його
за допомогою
[форми issue для заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічної, неконфіденційної перевірки власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо вважаєте, що команда ClawHub має перевірити, чи
простір імен потрібно зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити через реальну власність.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, наприклад `@example-org/*`, у якій має публікувати лише
  відповідний власник ClawHub
- slug навички або ім’я пакета Plugin, що виглядає як спроба видати себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника простору імен

Якщо запис небезпечний, шкідливий або оманливий поза межами спору про власність,
також дотримуйтеся відповідних рекомендацій щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для перевірки власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму: опублікуйте,
перейменуйте, передайте, приховайте або видаліть відповідний ресурс. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли команда має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або мейнтейнерів
- офіційну документацію проєкту, у якій названо простір імен
- доказ домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази торговельної марки, бренду або власності на проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, навичку, Plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має змогти зрозуміти
зв’язок без приватних облікових даних або секретів.

## Чого не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або контракти
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал із командою для конфіденційних доказів.
Використовуйте цей варіант замість публічного розміщення конфіденційних матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати власність, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перевірка простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
