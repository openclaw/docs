# Претензії щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги Skills, назви пакетів Plugin і
області видимості пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду розглянути його
через
[форму issue для претензії щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції щодо облікового запису для претензій щодо просторів імен.

## Коли відкривати претензію

Відкрийте претензію щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи слід
зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити простір імен через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область видимості пакета, як-от `@example-org/*`, у якій має публікуватися лише
  відповідний власник ClawHub
- слаг Skills або назва пакета Plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, зловмисний або оманливий не лише в межах спору про право власності,
також дотримуйтеся відповідних настанов із модерації або безпеки. Форма претензії щодо простору імен
призначена для розгляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спочатку переконайтеся, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin назви з областю видимості, як-от `@example-org/example-plugin`, мають
публікуватися від відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму: опублікуйте,
перейменуйте, передайте, приховайте або видаліть відповідний ресурс. Використовуйте претензію,
коли ви не можете керувати поточним власником або коли команда має вирішити
спір.

## Які докази додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або мейнтейнерів
- офіційну документацію проєкту, у якій названо простір імен
- доказ домену або офіційного домену електронної пошти
- контроль області видимості в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника ClawHub, Skills, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Команда має зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Чого не додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цю опцію замість публікації чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
