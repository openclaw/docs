# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги навичок, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, вводить в оману або оскаржується на ClawHub, попросіть співробітників переглянути його
за допомогою
[форми issue для заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічної, неконфіденційної перевірки права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для заявок щодо простору імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо вважаєте, що співробітники ClawHub мають перевірити, чи слід
зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити псевдонім
або інакше змінити простір імен через право власності в реальному світі.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- слаг навички або назва пакета Plugin, що, схоже, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, шкідливий або вводить в оману не лише в межах спору про власність,
також дотримуйтеся відповідних рекомендацій щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для перевірки власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте з власником, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли співробітникам потрібно розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, що називає простір імен
- доказ домену або офіційного поштового домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, навичку, Plugin, пакет або issue ClawHub

Поясніть, що доводить кожне посилання. Співробітники мають зрозуміти
зв’язок без приватних облікових даних або секретів.

## Чого не слід додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не включайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен для чутливих доказів приватний канал зі співробітниками.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику співробітники ClawHub можуть зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний запис,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перевірка простору імен не гарантує передавання кожного відповідного імені.
Співробітники зважують публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
