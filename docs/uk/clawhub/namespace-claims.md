# Заявки на організації та простори імен

ClawHub використовує імена власників, імена організацій, слаги Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, вводить в оману або оскаржується в ClawHub, попросіть команду переглянути його
за допомогою
[форми заявки на організацію / простір імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного перегляду права власності. Не використовуйте внутрішні
звіти в продукті або форму апеляції облікового запису для заявок на простори імен.

## Коли відкривати заявку

Відкрийте заявку на простір імен, коли вважаєте, що команда ClawHub має переглянути, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити через реальне право власності.

Приклади:

- ім’я організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, наприклад `@example-org/*`, яка має публікуватися лише від імені
  відповідного власника ClawHub
- слаг Skills або назва пакета Plugin, що виглядає як спроба видати себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує правомірного власника
  простору імен

Якщо запис є небезпечним, шкідливим або вводить в оману не лише в межах спору про право власності,
також дотримуйтеся відповідних рекомендацій щодо модерації або безпеки. Форма заявки на простір імен
призначена для перегляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу переконайтеся, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin scoped-імена, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте заявку,
коли не можете керувати поточним власником або коли команді потрібно розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, Skills, Plugin, пакет або issue в ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Чого не додавати

Не розміщуйте секрети або приватні докази в публічному GitHub issue. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти про безпеку або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публікації чутливих матеріалів публічно.

## Можливі результати

Залежно від доказів і ризику команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
