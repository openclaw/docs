# Претензії щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, здається, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду переглянути його
за допомогою
[форми issue для претензії щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного перегляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції щодо облікового запису для претензій щодо просторів імен.

## Коли відкривати претензію

Відкрийте претензію щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- слаг Skill або назва пакета Plugin, що, здається, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, зловмисний або оманливий не лише в межах спору про право власності,
також дотримуйтесь відповідних вказівок щодо модерації або безпеки. Форма претензії щодо простору імен
призначена для перегляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу переконайтеся, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin назви з областями, як-от `@example-org/example-plugin`, мають бути
опубліковані від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо: опублікуйте,
перейменуйте, передайте, приховайте або видаліть відповідний ресурс. Використовуйте претензію,
коли ви не можете керувати поточним власником або коли команда має вирішити
спір.

## Докази, які варто додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, що називає простір імен
- доказ домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, Skill, Plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Що не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- токени API, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний запис,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
