# Претензії щодо організації та простору імен

ClawHub використовує handles власників, handles організацій, слаги Skills, назви пакетів plugin та
області пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржуваний у ClawHub, попросіть команду перевірити його
за допомогою
[форми issue для претензії щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічної, неконфіденційної перевірки права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для претензій щодо простору імен.

## Коли відкривати претензію

Відкривайте претензію щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- handle організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише від імені
  відповідного власника ClawHub
- слаг Skills або назва пакета plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, шкідливий або оманливий не лише в межах спору про право власності,
також дотримуйтеся відповідних настанов із модерації або безпеки. Форма претензії щодо простору імен
призначена для перевірки права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що публікуєте від імені власника, який відповідає простору імен.
Для пакетів plugin scoped-назви, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте претензію,
коли не можете керувати поточним власником або коли команда має вирішити
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного email-домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, Skills, plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Чого не слід додавати

Не додавайте секрети або приватні докази до публічного GitHub issue. Не включайте:

- API-токени, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перевірка простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
