# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги навичок, назви пакетів плагінів і
області пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржений у ClawHub, попросіть команду переглянути його
за допомогою
[форми issue для заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного перегляду власності. Не використовуйте внутрішньопродуктові
звіти або форму оскарження облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо вважаєте, що команда ClawHub має перевірити, чи
простір імен потрібно зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, який збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета на кшталт `@example-org/*`, яка має публікуватися лише від імені
  відповідного власника ClawHub
- слаг навички або назва пакета плагіна, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, шкідливим або оманливим поза межами спору про власність,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для перегляду власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів плагінів назви з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли команді потрібно вирішити
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або мейнтейнерів
- офіційну документацію проєкту, де названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, навичку, плагін, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має бути здатна зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не слід додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- DNS-токени для перевірки
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публікації чутливих матеріалів у відкритому доступі.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
