# Заявки щодо організацій і просторів імен

ClawHub використовує хендли власників, хендли організацій, слаги Skills, імена пакетів Plugin і
області видимості пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржується в ClawHub, попросіть команду розглянути його
через
[форму питання про заявку щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкривайте заявку щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- хендл організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область видимості пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- слаг Skills або ім’я пакета Plugin, що, схоже, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, зловмисний або оманливий не лише в межах спору про власність,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю видимості, як-от `@example-org/example-plugin`, мають бути
опубліковані від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму шляхом публікації,
перейменування, передавання, приховування або видалення зачепленого ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли команді потрібно розв’язати
спір.

## Які докази додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, що називає простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль над областю видимості в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, Skills, Plugin, пакет або issue в ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Чого не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- токени API, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цю опцію замість публікації чутливих матеріалів публічно.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожне відповідне ім’я буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
