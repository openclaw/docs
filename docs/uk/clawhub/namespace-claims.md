# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду розглянути його
за допомогою
[форми issue для заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду власності. Не використовуйте внутрішні
звіти продукту або форму оскарження облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо ви вважаєте, що команда ClawHub має розглянути, чи потрібно
зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити простір імен через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, у якій має публікуватися лише
  відповідний власник ClawHub
- slug Skills або назва пакета Plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недосяжний власник, який блокує законного власника
  простору імен

Якщо listing небезпечний, зловмисний або оманливий поза межами спору про власність,
також дотримуйтеся відповідних інструкцій із модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin scoped names, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму: опублікуйте,
перейменуйте, передайте, приховайте або видаліть відповідний ресурс. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли команді потрібно розв’язати
спір.

## Докази, які варто додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, release або maintainer
- офіційну документацію проєкту, де названо простір імен
- доказ домену або офіційного email-домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника ClawHub, Skills, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Чого не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний listing у карантин,
додати псевдонім або redirect, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
