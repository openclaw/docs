# Заяви щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржується в ClawHub, попросіть персонал переглянути його
за допомогою
[форми issue для заяви щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного перегляду права власності. Не використовуйте звіти у продукті
або форму оскарження облікового запису для заяв щодо просторів імен.

## Коли відкривати заяву

Відкрийте заяву щодо простору імен, коли вважаєте, що персонал ClawHub має переглянути, чи
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або бути інакше змінений через право власності в реальному світі.

Приклади:

- ідентифікатор організації, який збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише від імені
  відповідного власника ClawHub
- слаг Skills або назва пакета Plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, зловмисним або оманливим поза межами спору про право власності,
також дотримуйтеся відповідних рекомендацій щодо модерації або безпеки. Форма заяви щодо простору імен
призначена для перегляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте заяву,
коли не можете керувати поточним власником або коли персонал має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або мейнтейнерів
- офіційну документацію проєкту, у якій названо простір імен
- доказ домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, Skills, Plugin, пакет або issue в ClawHub

Поясніть, що доводить кожне посилання. Персонал має мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не слід додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або контракти
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заяви запитує, чи потрібен приватний канал персоналу для чутливих доказів.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, персонал ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити додаткові докази або відхилити запит.

Перегляд простору імен не гарантує, що кожне відповідне ім’я буде передано.
Персонал зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
