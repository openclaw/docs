# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги навичок, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржується в ClawHub, попросіть персонал переглянути його
за допомогою
[форми issue для заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, нечутливого перегляду права власності. Не використовуйте продуктові
звіти або форму оскарження облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкривайте заявку щодо простору імен, коли вважаєте, що персонал ClawHub має переглянути, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, у якій публікацію має виконувати лише
  відповідний власник ClawHub
- слаг навички або назва пакета Plugin, що, схоже, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо список небезпечний, зловмисний або оманливий не лише в межах спору щодо власності,
також дотримуйтеся відповідних вказівок із модерації або безпеки. Форма заявки щодо простору імен
призначена для перегляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу переконайтеся, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте заявку,
коли не можете керувати поточним власником або коли персонал має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, нечутливі докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізу або мейнтейнера
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, навичку, Plugin, пакет або issue в ClawHub

Поясніть, що доводить кожне посилання. Персонал має мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Чого не слід додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти про безпеку або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал персоналу для чутливих доказів.
Скористайтеся цією опцією замість публікації чутливих матеріалів публічно.

## Можливі результати

Залежно від доказів і ризику, персонал ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний список,
додати псевдонім або перенаправлення, попросити додаткові докази або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Персонал зважує публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
