# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги Skills, імена пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду розглянути його
за допомогою
[форми заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте вбудовані в продукт
скарги або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо вважаєте, що команда ClawHub має перевірити, чи потрібно
зарезервувати, передати, перейменувати, приховати, помістити на карантин, додати псевдонім
або інакше змінити простір імен через право власності в реальному світі.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- слаг Skills або ім’я пакета Plugin, що, схоже, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, зловмисним або оманливим поза межами спору щодо власності,
також дотримуйтеся відповідних настанов із модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли команді потрібно розв’язати
спір.

## Які докази додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, де названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, Skills, Plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

У формі заявки запитується, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публікації чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити на карантин наявний запис,
додати псевдонім або переспрямування, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожне відповідне ім’я буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
