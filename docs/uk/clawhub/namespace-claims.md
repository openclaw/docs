# Претензії щодо організацій і просторів імен

ClawHub використовує імена власників, імена організацій, слаги навичок, назви пакетів плагінів і
області видимості пакетів як публічні простори імен. Якщо простір імен здається таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду розглянути його
через
[форму issue для претензії щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічної, неконфіденційної перевірки права власності. Не використовуйте звіти
в продукті або форму апеляції щодо облікового запису для претензій щодо просторів імен.

## Коли відкривати претензію

Відкрийте претензію щодо простору імен, якщо вважаєте, що команда ClawHub має перевірити, чи
простір імен потрібно зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або інакше змінити через право власності в реальному світі.

Приклади:

- ім’я організації, яке збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область видимості пакета, як-от `@example-org/*`, у якій публікацію має виконувати лише
  відповідний власник ClawHub
- слаг навички або назва пакета плагіна, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недосяжний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, зловмисний або оманливий поза межами спору щодо права власності,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма претензії щодо простору імен
призначена для перевірки права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів плагінів імена з областю видимості, як-от `@example-org/example-plugin`, мають бути
опубліковані відповідним власником `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте претензію,
коли ви не можете керувати поточним власником або коли команді потрібно розв’язати
спір.

## Які докази додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, де названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області видимості в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію репозиторію з вихідним кодом, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, навичку, плагін, пакет або issue ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цю опцію замість публікації чутливих матеріалів публічно.

## Можливі результати

Залежно від доказів і ризику команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити додаткові докази або відхилити запит.

Перевірка простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
