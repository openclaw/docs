# Претензії щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду переглянути його
через
[форму issue щодо претензії організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, нечутливого перегляду права власності. Не використовуйте звіти
в продукті або форму апеляції облікового запису для претензій щодо просторів імен.

## Коли відкривати претензію

Відкрийте претензію щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити псевдонім
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- слаг Skills або назва пакета Plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо список є небезпечним, шкідливим або оманливим поза межами спору про право власності,
також дотримуйтеся відповідних інструкцій з модерації або безпеки. Форма претензії щодо простору імен
призначена для перегляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте з власником, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися як відповідний власник `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте претензію,
коли ви не можете керувати поточним власником або коли команді потрібно вирішити
спір.

## Докази, які слід додати

Використовуйте публічні, нечутливі докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізу або мейнтейнера
- офіційну документацію проєкту, яка називає простір імен
- підтвердження домену або офіційного поштового домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію репозиторію джерельного коду, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника ClawHub, Skills, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Команда має бути здатна зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не слід додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або контракти
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний список у карантин,
додати псевдонім або перенаправлення, попросити додаткові докази або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
