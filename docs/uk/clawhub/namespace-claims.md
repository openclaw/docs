# Заяви щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги навичок, імена пакетів плагінів і
області пакетів як публічні простори імен. Якщо простір імен здається таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржений у ClawHub, попросіть співробітників переглянути його
за допомогою
[форми issue для заяв щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного перегляду власності. Не використовуйте внутрішні
звіти продукту або форму апеляції щодо облікового запису для заяв щодо простору імен.

## Коли відкривати заяву

Відкрийте заяву щодо простору імен, якщо вважаєте, що співробітники ClawHub мають перевірити, чи
простір імен має бути зарезервований, переданий, перейменований, прихований, поміщений у карантин, отримати псевдонім
або бути змінений іншим чином через реальну власність.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, наприклад `@example-org/*`, у якій має публікувати лише
  відповідний власник ClawHub
- слаг навички або ім’я пакета плагіна, що схоже на імітацію проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, зловмисний або оманливий поза межами спору про власність,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заяви щодо простору імен
призначена для перегляду власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів плагінів імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заяву,
коли не можете керувати поточним власником або коли співробітники мають розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскарженого власника ClawHub, навичку, плагін, пакет або issue

Поясніть, що доводить кожне посилання. Співробітники мають мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Чого не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не включайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заяви запитує, чи потрібен приватний канал для співробітників для конфіденційних доказів.
Використайте цю опцію замість публічного розміщення конфіденційних матеріалів.

## Можливі результати

Залежно від доказів і ризику, співробітники ClawHub можуть зарезервувати простір імен,
передати власність, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або переспрямування, попросити додаткові докази або відхилити запит.

Перегляд простору імен не гарантує, що кожне відповідне ім’я буде передане.
Співробітники зважують публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
