# Претензії щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slugs Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, вводить в оману або оскаржується на ClawHub, попросіть персонал розглянути його
за допомогою
[форми issue для претензії щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, нечутливого розгляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції щодо облікового запису для претензій щодо просторів імен.

## Коли відкривати претензію

Відкривайте претензію щодо простору імен, коли вважаєте, що персонал ClawHub має перевірити, чи слід
простір імен зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або іншим чином змінити через реальне право власності.

Приклади:

- ідентифікатор організації, який збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, наприклад `@example-org/*`, яка має публікуватися лише від імені
  відповідного власника ClawHub
- slug Skills або назва пакета Plugin, що виглядає як спроба видавати себе за проєкт
- суперечка щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недосяжний власник, який блокує законного власника
  простору імен

Якщо listing є небезпечним, зловмисним або вводить в оману поза межами спору про право власності,
також дотримуйтеся відповідних рекомендацій щодо модерації або безпеки. Форма претензії щодо простору імен
призначена для розгляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спочатку переконайтеся, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin scoped-імена, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо через публікацію,
перейменування, передання, приховування або видалення відповідного ресурсу. Використовуйте претензію,
коли ви не можете керувати поточним власником або коли персонал має вирішити
спір.

## Докази, які слід додати

Використовуйте публічні, нечутливі докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізу або maintainer
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного email-домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника ClawHub, Skills, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Персонал має бути здатний зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не слід додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен для чутливих доказів приватний канал персоналу.
Скористайтеся цим варіантом замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, персонал ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний listing у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожну відповідну назву буде передано.
Персонал зважує публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
