# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug навичок, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду розглянути його
за допомогою
[форми заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте вбудовані в продукт
звіти або форму апеляції щодо облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкривайте заявку щодо простору імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яку слід публікувати лише від імені
  відповідного власника ClawHub
- slug навички або назва пакета Plugin, що, схоже, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, шкідливим або оманливим поза межами спору про право власності,
також дотримуйтеся відповідних рекомендацій щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу переконайтеся, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли команда має вирішити
спір.

## Докази, які варто додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, де названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль над областю в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника ClawHub, навичку, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Команда має бути здатна зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не варто додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- токени API, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

У формі заявки запитується, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний запис,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожне відповідне ім’я буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
