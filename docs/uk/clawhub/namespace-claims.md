# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги Skills, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, здається, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржується в ClawHub, попросіть персонал переглянути його
за допомогою
[форми заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічної, нечутливої перевірки права власності. Не використовуйте внутрішні
звіти продукту або форму оскарження облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкривайте заявку щодо простору імен, коли вважаєте, що персонал ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, створити для нього псевдонім
або іншим чином змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета на кшталт `@example-org/*`, у якій публікація має виконуватися лише від імені
  відповідного власника ClawHub
- слаг Skills або назва пакета Plugin, що, здається, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис небезпечний, зловмисний або оманливий поза межами спору про право власності,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для перевірки права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin імена з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли персонал має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, нечутливі докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізу або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль над областю в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника, Skills, Plugin, пакет або issue в ClawHub

Поясніть, що доводить кожне посилання. Персонал має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Що не слід додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал персоналу для чутливих доказів.
Використовуйте цю опцію замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику персонал ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перевірка простору імен не гарантує, що кожну відповідну назву буде передано.
Персонал зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
