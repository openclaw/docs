# Претензії на організації та простори імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug навичок, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен, здається, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже є
заявленим, зарезервованим, оманливим або спірним у ClawHub, попросіть команду переглянути його
через
[форму issue для претензії на організацію / простір імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, нечутливого перегляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для претензій на простір імен.

## Коли відкривати претензію

Відкрийте претензію на простір імен, якщо вважаєте, що команда ClawHub має переглянути, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- slug навички або назва пакета Plugin, що, здається, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, шкідливим або оманливим поза межами спору про право власності,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма претензії на простір імен
призначена для перегляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin scoped-назви, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму шляхом публікації,
перейменування, передавання, приховування або видалення зачепленого ресурсу. Використовуйте претензію,
коли ви не можете керувати поточним власником або коли команда має вирішити
спір.

## Докази, які слід додати

Використовуйте публічні, нечутливі докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, яка називає простір імен
- доказ домену або офіційного поштового домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, навичку, Plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Чого не додавати

Не розміщуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або контракти
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма претензії запитує, чи потрібен приватний канал команди для чутливих доказів.
Використовуйте цей варіант замість публікації чутливих матеріалів публічно.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний запис,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик безпеки та вплив на користувачів.
