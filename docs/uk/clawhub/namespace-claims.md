# Заявки на організації та простори імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги навичок, назви пакетів плагінів і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або оскаржується в ClawHub, попросіть команду переглянути його
через
[форму issue для заявки на організацію / простір імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте вбудовані в продукт
звіти або форму апеляції щодо облікового запису для заявок на простір імен.

## Коли відкривати заявку

Відкрийте заявку на простір імен, коли вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, наприклад `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- слаг навички або назва пакета плагіна, що, схоже, видає себе за проєкт
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо listing є небезпечним, шкідливим або оманливим не лише в межах спору про право власності,
також дотримуйтеся відповідних настанов із модерації або безпеки. Форма заявки на простір імен
призначена для розгляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спочатку підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів плагінів scoped-назви, як-от `@example-org/example-plugin`, мають
публікуватися від відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли команда має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або maintainer
- офіційну документацію проєкту, де зазначено простір імен
- доказ домену або офіційного email-домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, навичку, плагін, пакет або issue в ClawHub

Поясніть, що доводить кожне посилання. Команда має мати змогу зрозуміти
зв’язок без приватних облікових даних або секретів.

## Що не слід додавати

Не розміщуйте секрети або приватні докази в публічному GitHub issue. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- особисті документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал команди для чутливих доказів.
Скористайтеся цією опцією замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику, команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний listing,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
