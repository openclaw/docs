# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, слаги навичок, назви пакетів plugin і
області пакетів як публічні простори імен. Якщо простір імен, схоже, належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть співробітників розглянути його
за допомогою
[форми issue для заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте внутрішні
звіти продукту або форму оскарження облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, коли вважаєте, що співробітники ClawHub мають розглянути, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати до нього псевдонім
або іншим чином змінити через реальне право власності.

Приклади:

- ідентифікатор організації, який відповідає вашій організації GitHub, проєкту, компанії або спільноті
- область пакета, як-от `@example-org/*`, у якій публікація має відбуватися лише від
  відповідного власника ClawHub
- слаг навички або назва пакета plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недосяжний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, зловмисним або оманливим поза межами спору про право власності,
також дотримуйтеся відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду права власності, а не для екстреного розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від власника, який відповідає простору імен.
Для пакетів plugin назви з областю, як-от `@example-org/example-plugin`, мають бути
опубліковані від відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен напряму шляхом публікації,
перейменування, передавання, приховування або видалення відповідного ресурсу. Використовуйте заявку,
коли не можете керувати поточним власником або коли співробітники мають вирішити
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізу або мейнтейнера
- офіційну документацію проєкту, де названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію репозиторію вихідного коду, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника, навичку, plugin, пакет або issue у ClawHub

Поясніть, що доводить кожне посилання. Співробітники мають розуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не слід додавати

Не публікуйте секрети або приватні докази в публічному GitHub issue. Не додавайте:

- токени API, ключі підпису або облікові дані
- токени DNS challenge
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал співробітників для конфіденційних доказів.
Використовуйте цей варіант замість публікації конфіденційних матеріалів.

## Можливі результати

Залежно від доказів і ризику співробітники ClawHub можуть зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або переспрямування, попросити більше доказів або відхилити запит.

Розгляд простору імен не гарантує, що кожну відповідну назву буде передано.
Співробітники зважують публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
