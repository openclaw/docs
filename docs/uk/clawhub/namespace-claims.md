# Заявки щодо організацій і просторів імен

ClawHub використовує ідентифікатори власників, ідентифікатори організацій, slug-и Skills, назви пакетів Plugin і
простори пакетів як публічні простори імен. Якщо простір імен здається таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
заявлений, зарезервований, оманливий або спірний у ClawHub, попросіть команду перевірити його
через
[форму заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічної, неконфіденційної перевірки права власності. Не використовуйте внутрішні
скарги в продукті або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкрийте заявку щодо простору імен, якщо ви вважаєте, що команда ClawHub має перевірити, чи
простір імен слід зарезервувати, передати, перейменувати, приховати, помістити в карантин, зробити псевдонімом
або інакше змінити через реальне право власності.

Приклади:

- ідентифікатор організації, що збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- простір пакета, як-от `@example-org/*`, який має публікуватися лише від імені
  відповідного власника ClawHub
- slug Skills або назва пакета Plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недоступний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, шкідливим або оманливим поза межами спору про право власності,
також дотримуйтесь відповідних настанов щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для перевірки права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin назви з просторами, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо: опублікуйте,
перейменуйте, передайте, приховайте або видаліть відповідний ресурс. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли команда має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні підтвердження включають:

- історію організації GitHub, репозиторію, релізів або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного домену електронної пошти
- контроль простору в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на спірного власника ClawHub, Skills, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Команда має розуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Що не додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- токени DNS-перевірки
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

У формі заявки запитується, чи потрібен приватний канал команди для конфіденційних доказів.
Скористайтеся цією опцією замість публікації конфіденційних матеріалів.

## Можливі результати

Залежно від доказів і ризику команда ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити в карантин наявний запис,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перевірка простору імен не гарантує, що кожну відповідну назву буде передано.
Команда зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
