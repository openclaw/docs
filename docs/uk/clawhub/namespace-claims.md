# Заявки щодо організацій і просторів імен

ClawHub використовує дескриптори власників, дескриптори організацій, slug навичок, назви пакетів Plugin і
області пакетів як публічні простори імен. Якщо простір імен виглядає таким, що належить
реальному проєкту, бренду, екосистемі пакетів або організації, але вже
зайнятий, зарезервований, оманливий або оскаржуваний у ClawHub, попросіть персонал переглянути його
за допомогою
[форми заявки щодо організації / простору імен](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml).

Використовуйте цей шлях для публічного, неконфіденційного розгляду права власності. Не використовуйте внутрішні
звіти продукту або форму апеляції облікового запису для заявок щодо просторів імен.

## Коли відкривати заявку

Відкривайте заявку щодо простору імен, коли вважаєте, що персонал ClawHub має перевірити, чи
потрібно зарезервувати, передати, перейменувати, приховати, помістити в карантин, додати псевдонім
або іншим чином змінити простір імен через реальне право власності.

Приклади:

- дескриптор організації, який збігається з вашою організацією GitHub, проєктом, компанією або спільнотою
- область пакета, як-от `@example-org/*`, яка має публікуватися лише під
  відповідним власником ClawHub
- slug навички або назва пакета Plugin, що виглядає як імітація проєкту
- спір щодо бренду, торговельної марки, перейменування проєкту або історії пакета
- видалений, неактивний або недосяжний власник, який блокує законного власника
  простору імен

Якщо запис є небезпечним, шкідливим або оманливим поза межами спору про власність,
також дотримуйтеся відповідних інструкцій щодо модерації або безпеки. Форма заявки щодо простору імен
призначена для розгляду права власності, а не для термінового розкриття вразливостей.

## Перед поданням

Спершу підтвердьте, що ви публікуєте від імені власника, який відповідає простору імен.
Для пакетів Plugin назви з областю, як-от `@example-org/example-plugin`, мають
публікуватися від імені відповідного власника `example-org`.

Якщо ви можете керувати поточним власником, виправте простір імен безпосередньо, опублікувавши,
перейменувавши, передавши, приховавши або видаливши відповідний ресурс. Використовуйте заявку,
коли ви не можете керувати поточним власником або коли персонал має розв’язати
спір.

## Докази, які слід додати

Використовуйте публічні, неконфіденційні докази. Корисні докази включають:

- історію організації GitHub, репозиторію, релізу або супровідників
- офіційну документацію проєкту, у якій названо простір імен
- підтвердження домену або офіційного поштового домену
- контроль області в npm, PyPI, crates.io або іншому реєстрі пакетів
- докази права власності на торговельну марку, бренд або проєкт, які безпечно обговорювати
  публічно
- історію вихідного репозиторію, історію пакета або публічні повідомлення про перейменування
- посилання на оскаржуваного власника ClawHub, навичку, Plugin, пакет або issue

Поясніть, що доводить кожне посилання. Персонал має зрозуміти
зв’язок без потреби в приватних облікових даних або секретах.

## Чого не слід додавати

Не публікуйте секрети або приватні докази в публічному issue GitHub. Не додавайте:

- API-токени, ключі підпису або облікові дані
- DNS-токени перевірки
- приватні юридичні файли або договори
- документи, що посвідчують особу
- приватні електронні листи, приватні звіти з безпеки або конфіденційні дані клієнтів

Форма заявки запитує, чи потрібен приватний канал зв’язку з персоналом для чутливих доказів.
Використайте цю опцію замість публічного розміщення чутливих матеріалів.

## Можливі результати

Залежно від доказів і ризику персонал ClawHub може зарезервувати простір імен,
передати право власності, перейменувати ресурс, приховати або помістити наявний запис у карантин,
додати псевдонім або перенаправлення, попросити більше доказів або відхилити запит.

Перегляд простору імен не гарантує, що кожну відповідну назву буде передано.
Персонал зважує публічні докази, наявне використання, ризик для безпеки та вплив на користувачів.
