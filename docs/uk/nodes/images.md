# Підтримка зображень і медіа (2025-12-05)

Канал WhatsApp працює через **Baileys Web**. Цей документ фіксує поточні правила обробки медіа для send, gateway і відповідей агента.

## Цілі

- Надсилати медіа з необов’язковими captions через `openclaw message send --media`.
- Дозволити авто-відповідям із web inbox включати медіа разом із текстом.
- Зберігати адекватні та передбачувані ліміти для кожного типу.

## Поверхня CLI

- `openclaw message send --media <path-or-url> [--message <caption>]`
  - `--media` необов’язковий; caption може бути порожнім для надсилання лише медіа.
  - `--dry-run` друкує визначений payload; `--json` виводить `{ channel, to, messageId, mediaUrl, caption }`.

## Поведінка каналу WhatsApp Web

- Вхід: локальний шлях до файла **або** HTTP(S) URL.
- Потік: завантажити в Buffer, визначити тип медіа та побудувати правильний payload:
  - **Зображення:** змінити розмір і повторно стиснути до JPEG (максимальна сторона 2048px) з орієнтацією на `channels.whatsapp.mediaMaxMb` (типово: 50 MB).
  - **Audio/Voice/Video:** pass-through до 16 MB; audio надсилається як voice note (`ptt: true`).
  - **Документи:** усе інше, до 100 MB, зі збереженням filename, якщо він доступний.
- Відтворення у стилі GIF для WhatsApp: надсилайте MP4 з `gifPlayback: true` (CLI: `--gif-playback`), щоб мобільні клієнти зациклювали його inline.
- Визначення MIME надає перевагу magic bytes, потім заголовкам, а потім розширенню файла.
- Caption береться з `--message` або `reply.text`; порожній caption дозволений.
- Логування: у не-verbose режимі показує `↩️`/`✅`; у verbose включає розмір і шлях/URL джерела.

## Pipeline авто-відповіді

- `getReplyFromConfig` повертає `{ text?, mediaUrl?, mediaUrls? }`.
- Коли медіа присутнє, web sender визначає локальні шляхи або URL за тим самим pipeline, що й `openclaw message send`.
- Якщо надано кілька записів медіа, вони надсилаються послідовно.

## Вхідні медіа для команд (Pi)

- Коли вхідні web-повідомлення містять медіа, OpenClaw завантажує їх у тимчасовий файл і надає змінні шаблонізації:
  - `{{MediaUrl}}` pseudo-URL для вхідного медіа.
  - `{{MediaPath}}` локальний тимчасовий шлях, записаний перед запуском команди.
- Коли ввімкнено Docker sandbox для кожної session, вхідне медіа копіюється в workspace пісочниці, а `MediaPath`/`MediaUrl` переписуються на відносний шлях на кшталт `media/inbound/<filename>`.
- Розуміння медіа (якщо налаштовано через `tools.media.*` або спільне `tools.media.models`) запускається до шаблонізації і може вставляти блоки `[Image]`, `[Audio]` і `[Video]` у `Body`.
  - Для audio задається `{{Transcript}}`, а сам transcript використовується для розбору команд, щоб slash-команди все одно працювали.
  - Опис video і image зберігає будь-який текст caption для розбору команд.
  - Якщо активна основна image-модель уже нативно підтримує vision, OpenClaw пропускає summary-блок `[Image]` і натомість передає моделі оригінальне зображення.
- Типово обробляється лише перше відповідне вкладення image/audio/video; задайте `tools.media.<cap>.attachments`, щоб обробляти кілька вкладень.

## Ліміти й помилки

**Ліміти вихідного надсилання (надсилання через WhatsApp Web)**

- Зображення: до `channels.whatsapp.mediaMaxMb` (типово: 50 MB) після повторного стиснення.
- Audio/voice/video: ліміт 16 MB; документи: 100 MB.
- Завеликі або непридатні до читання медіа → чітка помилка в логах, а відповідь пропускається.

**Ліміти для розуміння медіа (транскрибування/опис)**

- Типове значення для зображень: 10 MB (`tools.media.image.maxBytes`).
- Типове значення для audio: 20 MB (`tools.media.audio.maxBytes`).
- Типове значення для video: 50 MB (`tools.media.video.maxBytes`).
- Завеликі медіа пропускають етап розуміння, але відповіді все одно проходять з оригінальним body.

## Примітки для тестів

- Покривайте потоки send + reply для випадків із image/audio/document.
- Перевіряйте повторне стиснення для зображень (обмеження розміру) і прапорець voice-note для audio.
- Переконайтеся, що multi-media replies розгортаються в послідовні надсилання.
