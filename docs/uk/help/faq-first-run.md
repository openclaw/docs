  <Accordion title="Не можу продовжити роботу — як найшвидше розв’язати проблему">
    Скористайтеся локальним ШІ-агентом, який може **бачити ваш комп’ютер**. Більшість випадків «не можу продовжити»
    спричинені **проблемами локальної конфігурації або середовища**, які віддалений помічник не може перевірити, тому це ефективніше,
    ніж запитувати в Discord.

    - **Claude Code**: [https://www.anthropic.com/claude-code/](https://www.anthropic.com/claude-code/)
    - **OpenAI Codex**: [https://openai.com/codex/](https://openai.com/codex/)

    Надайте агенту повну копію вихідного коду за допомогою модифікованого встановлення (git), щоб він міг прочитати
    код і документацію та врахувати точну версію, яку ви використовуєте:

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method git
    ```

    Попросіть агента покроково спланувати й проконтролювати виправлення, а потім виконати лише
    необхідні команди — менші відмінності легше перевіряти.

    Надайте ці результати, коли звертаєтеся по допомогу (у Discord або в задачі GitHub):

    | Команда | Що показує |
    | --- | --- |
    | `openclaw status` | Стан Gateway/агента й базовий знімок конфігурації |
    | `openclaw status --all` | Повна діагностика лише для читання, яку можна вставити |
    | `openclaw models status` | Автентифікація постачальника й доступність моделей |
    | `openclaw doctor` | Перевіряє та виправляє поширені проблеми конфігурації/стану |
    | `openclaw logs --follow` | Потоковий перегляд журналу в реальному часі |
    | `openclaw gateway status --deep` | Поглиблена перевірка стану Gateway/конфігурації/plugin |
    | `openclaw health --verbose` | Докладний звіт про стан |

    Знайшли справжню помилку або виправлення? Створіть задачу або надішліть PR:
    [Задачі](https://github.com/openclaw/openclaw/issues) /
    [Запити на злиття](https://github.com/openclaw/openclaw/pulls).

    Короткий цикл налагодження: [Перші 60 секунд, якщо щось не працює](/uk/help/faq#first-60-seconds-if-something-is-broken).
    Документація зі встановлення: [Встановлення](/uk/install), [Прапорці інсталятора](/uk/install/installer), [Оновлення](/uk/install/updating).

    | --- | --- |
    | `quiet-hours` | Поза межами налаштованого проміжку активних годин |

  <Accordion title="Як відкрити панель керування після початкового налаштування?">
    Одразу після налаштування відкривається браузер із чистою (без токена) URL-адресою панелі керування,
    а посилання виводиться в підсумку. Не закривайте цю вкладку; якщо вона не відкрилася,
    скопіюйте та вставте виведену URL-адресу на тому самому комп’ютері.
  </Accordion>

  <Accordion title="Як автентифікувати панель керування на localhost і віддалено?">
    **Localhost (той самий комп’ютер):**

    - Відкрийте `http://127.0.0.1:18789/`.
    - Якщо з’явиться запит на автентифікацію за допомогою спільного секрету, вставте налаштований токен або пароль у налаштування Control UI.
    - Джерело токена: `gateway.auth.token` (або `OPENCLAW_GATEWAY_TOKEN`).
    - Джерело пароля: `gateway.auth.password` (або `OPENCLAW_GATEWAY_PASSWORD`).
    - Спільний секрет ще не налаштовано? Запустіть `openclaw doctor --generate-gateway-token` (або `openclaw doctor --fix --generate-gateway-token`).

    **Не на localhost:**

    - **Tailscale Serve** (рекомендовано): залиште прив’язку до loopback, запустіть `openclaw gateway --tailscale serve`, відкрийте `https://<magicdns>/`. З `gateway.auth.allowTailscale: true` заголовки ідентичності задовольняють вимоги автентифікації Control UI/WebSocket (не потрібно вставляти спільний секрет; передбачається довірений хост Gateway); API HTTP усе одно потребують автентифікації за допомогою спільного секрету, якщо навмисно не використовується приватний вхідний трафік `none` або HTTP-автентифікація довіреного проксі.
      Одночасні спроби Serve з неправильною автентифікацією від одного клієнта серіалізуються до того, як обмежувач невдалих автентифікацій їх зареєструє, тому вже під час другої невдалої повторної спроби може з’явитися `retry later`.
    - **Прив’язка Tailnet**: запустіть `openclaw gateway --bind tailnet --token "<token>"` (або налаштуйте автентифікацію паролем), відкрийте `http://<tailscale-ip>:18789/`, вставте відповідний спільний секрет у налаштування панелі керування.
    - **Зворотний проксі з урахуванням ідентичності**: залиште Gateway за довіреним проксі, установіть `gateway.auth.mode: "trusted-proxy"`, відкрийте URL-адресу проксі. Для loopback-проксі на тому самому хості потрібно явно вказати `gateway.auth.trustedProxy.allowLoopback: true`.
    - **Тунель SSH**: `ssh -N -L 18789:127.0.0.1:18789 user@gateway-host`, потім відкрийте `http://127.0.0.1:18789/`. Автентифікація за допомогою спільного секрету також застосовується через тунель; якщо з’явиться запит, вставте налаштований токен або пароль.

    Відомості про режими прив’язки та автентифікацію див. в розділах [Панель керування](/uk/web/dashboard) і [Вебінтерфейси](/uk/web).

  </Accordion>

  <Accordion title="Чому для схвалень у чаті є дві конфігурації схвалення exec?">
    Вони керують різними рівнями:

    - `approvals.exec` — пересилає запити на схвалення до чатів призначення.
    - `channels.<channel>.execApprovals` — робить цей канал нативним клієнтом схвалення для схвалень exec.

    Політика exec хоста все одно залишається справжнім бар’єром схвалення; конфігурація чату визначає лише те,
    де з’являються запити та як люди на них відповідають.

    Обидві конфігурації рідко потрібні одночасно:

    - Якщо чат уже підтримує команди та відповіді, `/approve` у тому самому чаті працює через спільний шлях.
    - Коли підтримуваний нативний канал може безпечно визначити осіб, які схвалюють, OpenClaw автоматично вмикає нативні схвалення з пріоритетом особистих повідомлень, якщо `channels.<channel>.execApprovals.enabled` не задано або має значення `"auto"`.
    - Коли доступні нативні картки/кнопки схвалення, цей інтерфейс є основним; згадуйте ручну команду `/approve` лише тоді, коли результат інструмента повідомляє, що схвалення через чат недоступні.
    - Використовуйте `approvals.exec` лише тоді, коли запити також мають надходити до інших чатів або явно визначених операційних кімнат.
    - Використовуйте `channels.<channel>.execApprovals.target: "channel"` або `"both"` лише тоді, коли запити на схвалення потрібно публікувати назад у вихідній кімнаті/темі.
    - Схвалення plugin є окремими: за замовчуванням `/approve` у тому самому чаті, необов’язкове пересилання `approvals.plugin`, і лише деякі нативні канали зберігають нативне опрацювання також для них.

    Коротко: пересилання призначене для маршрутизації, а конфігурація нативного клієнта — для розширеного UX, специфічного для каналу.
    Див. [Схвалення exec](/uk/tools/exec-approvals).

  </Accordion>

  <Accordion title="Яке середовище виконання потрібне?">
    Потрібен Node **22.22.3+**, **24.15+** або **25.9+** (рекомендовано Node 24). `pnpm` — менеджер пакетів репозиторію.
    Bun може встановлювати залежності та запускати пакетні скрипти, але не може запускати CLI OpenClaw або Gateway, оскільки в ньому немає `node:sqlite`.
  </Accordion>

  <Accordion title="Чи працює це на Raspberry Pi?">
    Так, але спочатку перевірте обсяг оперативної пам’яті: Pi 5 і Pi 4 (2 GB+) — оптимальні варіанти; Pi 3B+ (1 GB) працює, але повільно; Pi Zero 2 W (512 MB) не рекомендовано.

    | Модель | Оперативна пам’ять | Придатність |
    | --- | --- | --- |
    | Pi 5 | 4/8 GB | Найкраще |
    | Pi 4 | 4 GB | Добре |
    | Pi 4 | 2 GB | Прийнятно, додайте swap |
    | Pi 4 | 1 GB | Обмежено |
    | Pi 3B+ | 1 GB | Повільно |
    | Pi Zero 2 W | 512 MB | Не рекомендовано |

    Абсолютний мінімум: 1 GB оперативної пам’яті, 1 ядро, 500 MB вільного місця на диску, 64-розрядна ОС. Оскільки на Pi працює лише
    Gateway (моделі звертаються до хмарних API), навіть Pi зі скромними характеристиками витримує навантаження.

    Невеликий Pi/VPS також може розміщувати лише Gateway, а **вузли** можна сполучити
    на ноутбуці/телефоні для локального доступу до екрана/камери/полотна або виконання команд. Див. [Вузли](/uk/nodes).

    Повна покрокова інструкція з налаштування: [Raspberry Pi](/uk/install/raspberry-pi).

  </Accordion>

  <Accordion title="Поради щодо встановлення на Raspberry Pi">
    - Використовуйте **64-розрядну** ОС; не використовуйте 32-розрядну Raspberry Pi OS.
    - Додайте swap на платах із 2 GB або меншим обсягом пам’яті.
    - Для кращої швидкодії та довговічності віддавайте перевагу **USB SSD**, а не SD-картці.
    - Віддавайте перевагу модифікованому встановленню (git), щоб переглядати журнали й швидко оновлювати систему.
    - Почніть без каналів/навичок і додавайте їх по одному.
    - Дивні помилки бінарних файлів («exec format error») зазвичай означають відсутність збірки ARM64 для необов’язкового інструмента навички.

    Повний посібник: [Raspberry Pi](/uk/install/raspberry-pi). Також див. [Linux](/uk/platforms/linux).

  </Accordion>

  <Accordion title="Система застрягла на wake up my friend / початкове налаштування не завершується. Що робити?">
    Цей екран залежить від доступності й автентифікації Gateway. Під час першого запуску TUI також автоматично надсилає
    «Прокинься, друже!», якщо налаштовано постачальника моделей. Якщо
    налаштування моделі/автентифікації пропущено, під час початкового налаштування з’являється примітка «Відсутня автентифікація моделі» й відкривається
    TUI без надсилання повідомлень — додайте постачальника за допомогою `openclaw configure --section model`.
    Якщо відображається рядок пробудження **без відповіді**, а кількість токенів залишається 0, агент не запускався.

    1. Перезапустіть Gateway:

    ```bash
    openclaw gateway restart
    ```

    2. Перевірте стан і автентифікацію:

    ```bash
    openclaw status
    openclaw models status
    openclaw logs --follow
    ```

    3. Усе ще зависає? Запустіть:

    ```bash
    openclaw doctor
    ```

    Якщо Gateway віддалений, переконайтеся, що з’єднання через тунель/Tailscale активне, а UI
    вказує на правильний Gateway. Див. [Віддалений доступ](/uk/gateway/remote).

  </Accordion>

  <Accordion title="Чи можна перенести налаштування на новий комп’ютер без повторного початкового налаштування?">
    Так. Скопіюйте **каталог стану** та **робочий простір**, а потім один раз запустіть Doctor:

    1. Установіть OpenClaw на новому комп’ютері.
    2. Скопіюйте `$OPENCLAW_STATE_DIR` (за замовчуванням: `~/.openclaw`) зі старого комп’ютера.
    3. Скопіюйте робочий простір (за замовчуванням: `~/.openclaw/workspace`).
    4. Запустіть `openclaw doctor` і перезапустіть службу Gateway.

    Це збереже конфігурацію, профілі автентифікації, облікові дані WhatsApp, сеанси та пам’ять — ваш
    бот залишиться без змін, якщо скопіювати **обидва** розташування. У віддаленому режимі
    сховище сеансів і робочий простір належать хосту Gateway.

    **Важливо:** якщо лише зафіксувати/надіслати робочий простір до GitHub, буде створено резервну копію
    **пам’яті та файлів початкового завантаження**, але не історії сеансів чи даних автентифікації. Вони зберігаються в
    `~/.openclaw/` (наприклад, `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`).

    Пов’язані розділи: [Перенесення](/uk/install/migrating), [Розташування даних на диску](/uk/help/faq#where-things-live-on-disk),
    [Робочий простір агента](/uk/concepts/agent-workspace), [Doctor](/uk/gateway/doctor),
    [Віддалений режим](/uk/gateway/remote).

  </Accordion>

  <Accordion title="Різниця між стабільною та бета-версією">
    **Стабільна** та **бета-версія** — це **npm dist-tags**, а не окремі гілки коду:

    - `latest` = стабільна версія
    - `beta` = рання збірка для тестування (повертається до `latest`, якщо бета-версія відсутня або старіша за поточний стабільний випуск)

    Стабільний випуск зазвичай спочатку потрапляє до **бета-версії**, а потім окремий крок просування
    переміщує ту саму версію до `latest` без зміни номера версії. Супровідники
    також можуть публікувати безпосередньо до `latest`. Саме тому після просування бета- й стабільна версії можуть указувати на
    **ту саму версію**.

    Перегляньте зміни: [CHANGELOG.md](https://github.com/openclaw/openclaw/blob/main/CHANGELOG.md).

    Однорядкові команди встановлення та відмінності між бета- й dev-версіями наведено в наступному розділі.

  </Accordion>

  <Accordion title="Як установити бета-версію та чим вона відрізняється від dev-версії?">
    **Бета-версія** — це npm dist-tag `beta` (після просування може збігатися з `latest`).
    **Dev-версія** — це рухома верхівка `main` (git); у разі публікації в npm вона використовує dist-tag `dev`.

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --beta
    ```

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method git
    ```

    Інсталятор для Windows (PowerShell): `iwr -useb https://openclaw.ai/install.ps1 | iex`

    Докладніше: [Канали розробки](/uk/install/development-channels) та [Прапорці інсталятора](/uk/install/installer).

  <Accordion title="Під час установлення у Windows повідомляється, що git не знайдено або openclaw не розпізнано">
    Дві поширені проблеми у Windows:

    **1) помилка npm spawn git / git не знайдено**

    - Установіть **Git for Windows** і переконайтеся, що `git` додано до PATH.
    - Закрийте й знову відкрийте PowerShell, а потім повторно запустіть інсталятор.

    **2) openclaw не розпізнано після встановлення**

    - Глобальної папки бінарних файлів npm немає в PATH.
    - Перевірте її: `npm config get prefix`.
    - Додайте цей каталог до користувацького PATH (суфікс `\bin` не потрібен; у більшості систем це `%AppData%\npm`).
    - Закрийте й знову відкрийте PowerShell.

    Віддаєте перевагу настільному застосунку? Використовуйте **Windows Hub**. Для налаштування лише через термінал
    підтримуються як інсталятор PowerShell, так і шляхи Gateway у WSL2. Документація: [Windows](/uk/platforms/windows).

  </Accordion>

  <Accordion title="Виведення exec у Windows показує спотворений китайський текст — що робити?">
    Зазвичай причина полягає в невідповідності кодової сторінки консолі у власних оболонках Windows.

    Ознаки: виведення `system.run`/`exec` відображає китайський текст у вигляді пошкоджених символів; та сама команда
    має правильний вигляд в іншому профілі термінала.

    Обхідне рішення в PowerShell:

    ```powershell
    chcp 65001
    [Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
    [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
    $OutputEncoding = [System.Text.UTF8Encoding]::new($false)
    ```

    Потім перезапустіть Gateway і повторіть спробу:

    ```powershell
    openclaw gateway restart
    ```

    Проблема все ще відтворюється в останній версії OpenClaw? Стежте за нею або повідомте про неї: [Проблема №30640](https://github.com/openclaw/openclaw/issues/30640).

  </Accordion>

  <Accordion title="Що саме робить початкове налаштування?">
    `openclaw onboard` — рекомендований спосіб налаштування. У **локальному режимі** він проводить через такі етапи:

    1. **Модель/автентифікація** — OAuth постачальника, ключі API або ручна автентифікація (зокрема локальні варіанти на кшталт LM Studio); вибір стандартної моделі.
    2. **Робочий простір** — розташування та початкові файли.
    3. **Gateway** — порт, адреса прив’язування, режим автентифікації, доступ через Tailscale.
    4. **Канали** — вбудовані канали чатів і канали офіційних plugin: iMessage, Discord, Feishu, Google Chat, Mattermost, Microsoft Teams, QQ Bot, Signal, Slack, Telegram, WhatsApp тощо.
    5. **Демон** — LaunchAgent (macOS), користувацький модуль systemd (Linux/WSL2) або власне заплановане завдання Windows.
    6. **Перевірка справності** — запускає Gateway і перевіряє, що він працює.
    7. **Skills** — установлює рекомендовані навички та необов’язкові залежності.

    Він заздалегідь повідомляє очікувану тривалість і попереджає, якщо налаштована модель невідома
    або дані автентифікації відсутні. Повний опис: [Початкове налаштування (CLI)](/uk/start/wizard).

  <Accordion title="Чому я отримую HTTP 429 rate_limit_error від Anthropic?">
    Вашу **квоту/ліміт частоти запитів Anthropic** вичерпано для поточного часового вікна. У **Claude
    CLI** зачекайте, доки вікно скинеться, або перейдіть на вищий тарифний план. Для **API-ключа Anthropic**
    перевірте використання й оплату в Anthropic Console та за потреби збільште ліміти.

    Якщо повідомлення має конкретно вигляд `Extra usage is required for long context requests`,
    запит намагається використати контекстне вікно Anthropic розміром 1M (модель Claude 4.x із підтримкою 1M,
    доступну на рівні GA, або застарілу конфігурацію `params.context1m: true`), а ваші поточні облікові дані
    не дають права на оплату довгого контексту.

    Установіть **резервну модель**, щоб OpenClaw продовжував відповідати, поки провайдер обмежує частоту запитів.
    Див. [Моделі](/uk/cli/models), [OAuth](/uk/concepts/oauth) і
    [Для довгого контексту помилка Anthropic 429 вимагає додаткового використання](/uk/gateway/troubleshooting#anthropic-429-extra-usage-required-for-long-context).

  </Accordion>

  <Accordion title="Чи підтримується AWS Bedrock?">
    Так. OpenClaw має вбудованого провайдера **Amazon Bedrock (Converse)**. За наявності маркерів
    середовища AWS (`AWS_ACCESS_KEY_ID`, `AWS_PROFILE`, `AWS_BEARER_TOKEN_BEDROCK`)
    OpenClaw автоматично вмикає неявного провайдера Bedrock для виявлення моделей; інакше
    встановіть `plugins.entries.amazon-bedrock.config.discovery.enabled: true` або додайте запис
    провайдера вручну. Див. [Amazon Bedrock](/uk/providers/bedrock) і [Провайдери моделей](/uk/providers/models).
    OpenAI-сумісний проксі перед Bedrock також залишається прийнятним варіантом, якщо ви віддаєте перевагу керованому процесу роботи з ключами.
  </Accordion>

  <Accordion title="Як працює автентифікація Codex?">
    OpenClaw підтримує **OpenAI Codex** через OAuth (вхід у ChatGPT). Нове
    налаштування без основної моделі використовує саме `openai/gpt-5.6-sol` для
    автентифікації за підпискою ChatGPT/Codex разом із нативним виконанням через сервер застосунку Codex.
    Повторна автентифікація зберігає наявну явно вказану модель, зокрема
    `openai/gpt-5.5`. Якщо робочий простір Codex не надає GPT-5.6, явно виберіть
    `openai/gpt-5.5`; OpenClaw не переходить непомітно на слабшу модель. Застарілі
    посилання на моделі з префіксом Codex є застарілою конфігурацією, яку виправляє `openclaw doctor
    --fix`. Прямий доступ за API-ключем OpenAI залишається доступним для неагентних
    поверхонь API OpenAI, а через упорядкований профіль API-ключа `openai` — також для агентних
    моделей. Див. [Провайдери моделей](/uk/concepts/model-providers) і
    [Початкове налаштування (CLI)](/uk/start/wizard).
  </Accordion>

  <Accordion title="Чому OpenClaw досі згадує застарілий префікс OpenAI Codex?">
    `openai` — це поточний ідентифікатор провайдера та профілю автентифікації як для API-ключів OpenAI, так і для
    OAuth ChatGPT/Codex — OpenAI Codex інтегровано в нього. У старіших конфігураціях і попередженнях про міграцію
    ще може траплятися застарілий префікс `openai-codex`:

    - `openai/gpt-5.6-sol` = нове налаштування підписки ChatGPT/Codex із нативним середовищем виконання Codex для звернень агента.
    - `openai/gpt-5.5` = явно підтримуваний вибір для наявної конфігурації або облікових записів без доступу до GPT-5.6.
    - Застарілі посилання на моделі `openai-codex/*` = застарілий маршрут, який виправляє `openclaw doctor --fix`.
    - `openai/gpt-5.5` разом з упорядкованим профілем API-ключа `openai` = автентифікація за API-ключем для агентної моделі OpenAI.
    - Застарілі ідентифікатори профілів автентифікації `openai-codex` = застарілі ідентифікатори, які мігрує `openclaw doctor --fix`.

    Потрібна пряма оплата через OpenAI Platform? Установіть `OPENAI_API_KEY`. Потрібна автентифікація за підпискою
    ChatGPT/Codex? Виконайте `openclaw models auth login --provider openai`. Зберігайте
    посилання на моделі в канонічному провайдері `openai/*`. Нове налаштування підписки
    використовує саме `openai/gpt-5.6-sol`; doctor виправляє застарілі посилання з префіксом Codex,
    не змінюючи явно вибрану модель `openai/gpt-5.5` на новішу.

  </Accordion>

  <Accordion title="Чому ліміти Codex OAuth можуть відрізнятися від вебверсії ChatGPT?">
    Codex OAuth використовує керовані OpenAI вікна квот, що залежать від тарифного плану й можуть відрізнятися від
    доступних у вебсайті або застосунку ChatGPT навіть для того самого облікового запису.

    `openclaw models status` показує видимі зараз вікна використання/квот провайдера, але
    не вигадує й не перетворює права доступу у вебверсії ChatGPT на прямий доступ до API. Для
    прямої оплати та лімітів OpenAI Platform використовуйте `openai/*` з API-ключем.

  </Accordion>

  <Accordion title="Чи підтримується автентифікація за підпискою OpenAI (Codex OAuth)?">
    Так, повністю. OpenAI явно дозволяє використовувати OAuth підписки в зовнішніх
    інструментах і робочих процесах, як-от OpenClaw. Початкове налаштування може виконати процес OAuth за вас.

    Див. [OAuth](/uk/concepts/oauth), [Провайдери моделей](/uk/concepts/model-providers) і [Початкове налаштування (CLI)](/uk/start/wizard).

  </Accordion>

  <Accordion title="Як налаштувати OAuth Gemini CLI?">
    Gemini CLI використовує **процес автентифікації Plugin**, а не ідентифікатор клієнта чи секрет у `openclaw.json`.

    1. Установіть Gemini CLI локально, щоб `gemini` був у `PATH`:
       - Homebrew: `brew install gemini-cli`
       - npm: `npm install -g @google/gemini-cli`
    2. Увімкніть Plugin: `openclaw plugins enable google`
    3. Увійдіть: `openclaw models auth login --provider google-gemini-cli --set-default`
    4. Модель за замовчуванням після входу: `google/gemini-3.1-pro-preview` (середовище виконання `google-gemini-cli`)
    5. Запити не виконуються після входу? Установіть `GOOGLE_CLOUD_PROJECT` або `GOOGLE_CLOUD_PROJECT_ID` на хості Gateway і повторіть спробу.

    Токени OAuth зберігаються у профілях автентифікації на хості Gateway. Докладніше: [Google](/uk/providers/google), [Провайдери моделей](/uk/concepts/model-providers).

  </Accordion>

  <Accordion title="Чи підходить локальна модель для звичайних розмов?">
    Зазвичай ні. OpenClaw потребує великого контексту й надійного захисту; моделі для малих відеокарт обрізають контекст
    і оминають фільтри безпеки на боці провайдера. Якщо це необхідно, запускайте локально **найбільшу** збірку моделі,
    яку можете (LM Studio), — див. [Локальні моделі](/uk/gateway/local-models). Менші/квантовані
    моделі підвищують ризик ін'єкцій у запити — див. [Безпека](/uk/gateway/security).
  </Accordion>

  <Accordion title="Як зберігати трафік розміщеної моделі в певному регіоні?">
    Вибирайте кінцеві точки, закріплені за регіоном. OpenRouter пропонує розміщені у США варіанти для MiniMax, Kimi
    та GLM; виберіть розміщений у США варіант, щоб дані залишалися в регіоні. Anthropic/OpenAI
    можна й надалі вказувати разом із ними через `models.mode: "merge"`, щоб резервні варіанти
    залишалися доступними зі збереженням вимог вибраного регіонального провайдера.
  </Accordion>

  <Accordion title="Чи потрібно купувати Mac Mini, щоб установити OpenClaw?">
    Ні. OpenClaw працює на macOS або Linux (у Windows через WSL2). Mac mini — популярний
    вибір постійно ввімкненого хоста, але також підійде невеликий VPS, домашній сервер або пристрій класу Raspberry Pi.

    Mac потрібен лише **для інструментів, доступних тільки на macOS**. Для iMessage використовуйте [iMessage](/uk/channels/imessage)
    з `imsg` на будь-якому Mac, на якому виконано вхід у Messages. Якщо Gateway працює на Linux або деінде,
    задайте для `channels.imessage.cliPath` SSH-обгортку, яка запускає `imsg` на цьому Mac. Для інших
    інструментів, доступних лише на macOS, запускайте Gateway на Mac або під'єднайте Node macOS.

    Документація: [iMessage](/uk/channels/imessage), [Node](/uk/nodes), [Віддалений режим Mac](/uk/platforms/mac/remote).

  </Accordion>

  <Accordion title="Чи потрібен Mac mini для підтримки iMessage?">
    Потрібен **будь-який пристрій macOS**, на якому виконано вхід у Messages, — не обов'язково Mac mini,
    підійде будь-який Mac. Використовуйте [iMessage](/uk/channels/imessage) з `imsg`; Gateway може працювати на цьому
    Mac або деінде через SSH-обгортку `cliPath`.

    Поширені конфігурації:

    - Gateway на Linux/VPS, для `channels.imessage.cliPath` задано SSH-обгортку, яка запускає `imsg` на Mac, де виконано вхід у Messages.
    - Усе на одному Mac для найпростішої конфігурації на одній машині.

    Документація: [iMessage](/uk/channels/imessage), [Node](/uk/nodes), [Віддалений режим Mac](/uk/platforms/mac/remote).

  </Accordion>

  <Accordion title="Якщо я придбаю Mac mini для запуску OpenClaw, чи зможу під'єднати його до MacBook Pro?">
    Так. **Mac mini може запускати Gateway**, а MacBook Pro під'єднується як **Node**
    (пристрій-компаньйон). Node не запускають Gateway — вони додають можливості на кшталт
    екрана, камери, полотна та `system.run` на цьому пристрої.

    Поширена схема: Gateway працює на постійно ввімкненому Mac mini; на MacBook Pro працює застосунок macOS або
    хост Node, спарений із Gateway. Перевірте за допомогою `openclaw nodes status` / `openclaw nodes list`.

    Документація: [Node](/uk/nodes), [CLI Node](/uk/cli/nodes).

  </Accordion>

  <Accordion title="Чи можна використовувати Bun?">
    Bun можна використовувати для встановлення залежностей або запуску скриптів пакета. CLI OpenClaw і
    Gateway потребують **Node**, оскільки канонічне сховище стану використовує `node:sqlite`; Bun
    не надає цього API.
  </Accordion>

  <Accordion title="Telegram: що вказувати в allowFrom?">
    `channels.telegram.allowFrom` — це **ідентифікатор користувача Telegram відправника-людини** (числовий),
    а не ім'я користувача бота. Під час налаштування запитуються лише числові ідентифікатори користувачів; `openclaw doctor --fix`
    може спробувати розпізнати застарілі записи `@username`.

    Безпечніший спосіб (без стороннього бота): надішліть своєму боту приватне повідомлення, виконайте `openclaw logs --follow`, прочитайте `from.id`.

    Офіційний Bot API: надішліть своєму боту приватне повідомлення, викличте `https://api.telegram.org/bot<bot_token>/getUpdates`, прочитайте `message.from.id`.

    Сторонній спосіб (менш приватний): надішліть приватне повідомлення `@userinfobot` або `@getidsbot`.

    Див. [Керування доступом у Telegram](/uk/channels/telegram#access-control-and-activation).

  </Accordion>

  <Accordion title="Чи можуть кілька людей використовувати один номер WhatsApp із різними екземплярами OpenClaw?">
    Так, завдяки **маршрутизації між кількома агентами**. Прив'яжіть приватне повідомлення WhatsApp кожного відправника (`peer: { kind: "direct", id: "+15551234567" }`) до іншого `agentId`, надавши кожній особі власний робочий простір і сховище сеансів. Відповіді все одно надходитимуть з **того самого облікового запису WhatsApp**; керування доступом до приватних повідомлень (`channels.whatsapp.dmPolicy` / `channels.whatsapp.allowFrom`) є глобальним для кожного облікового запису. Див. [Маршрутизація між кількома агентами](/uk/concepts/multi-agent) і [WhatsApp](/uk/channels/whatsapp).
  </Accordion>

  <Accordion title='Чи можна запустити агента для «швидкого чату» й агента «Opus для програмування»?'>
    Так. Використовуйте маршрутизацію між кількома агентами: задайте кожному агенту власну модель за замовчуванням, а потім прив'яжіть вхідні
    маршрути (обліковий запис провайдера або конкретних співрозмовників) до відповідного агента. Приклад конфігурації:
    [Маршрутизація між кількома агентами](/uk/concepts/multi-agent). Див. також [Моделі](/uk/concepts/models) і
    [Конфігурація](/uk/gateway/configuration).
  </Accordion>

  <Accordion title="Чи працює Homebrew у Linux?">
    Так, через Linuxbrew:

    ```bash
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
    echo 'eval "$(/home/linuxbrew/.linuxbrew/bin/brew shellenv)"' >> ~/.profile
    eval "$(/home/linuxbrew/.linuxbrew/bin/brew shellenv)"
    brew install <formula>
    ```

    Під час запуску OpenClaw через systemd переконайтеся, що PATH служби містить
    `/home/linuxbrew/.linuxbrew/bin` (або ваш префікс brew), щоб інструменти, встановлені через `brew`,
    розпізнавалися в оболонках без входу в систему. Останні збірки також додають на початок типові користувацькі каталоги бінарних файлів у службах
    systemd Linux (наприклад, `~/.local/bin`, `~/.npm-global/bin`,
    `~/.local/share/pnpm`, `~/.bun/bin`) і враховують `PNPM_HOME`, `NPM_CONFIG_PREFIX`,
    `BUN_INSTALL`, `VOLTA_HOME`, `ASDF_DATA_DIR`, `NVM_DIR` та `FNM_DIR`, якщо їх задано.

  </Accordion>

  <Accordion title="Відмінність між придатним до модифікації встановленням із git і встановленням із npm">
    - **Придатне до модифікації встановлення (git):** повне отримання вихідного коду, який можна редагувати; найкраще підходить для учасників розробки. Збірка виконується локально, а код і документацію можна змінювати.
    - **Встановлення через npm:** глобальне встановлення CLI без репозиторію; найкраще підходить, щоб «просто запустити». Оновлення надходять через dist-теги npm.

    Документація: [Початок роботи](/uk/start/getting-started), [Оновлення](/uk/install/updating).

  </Accordion>

  <Accordion title="Чи можна згодом перемикатися між установленням через npm і git?">
    Так, за допомогою `openclaw update --channel ...` у наявному встановленні. Це **не
    видаляє ваші дані** — змінюється лише спосіб установлення коду OpenClaw. Стан (`~/.openclaw`) і
    робочий простір (`~/.openclaw/workspace`) залишаються без змін.

    З npm на git:

    ```bash
    openclaw update --channel dev
    ```

    З git на npm:

    ```bash
    openclaw update --channel stable
    ```

    Додайте `--dry-run`, щоб спочатку переглянути заплановану зміну режиму. Засіб оновлення виконує
    подальші дії Doctor, оновлює джерела плагінів для цільового каналу та перезапускає Gateway,
    якщо не передано `--no-restart`.

    Інсталятор також може примусово вибрати будь-який із цих режимів:

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method git
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method npm
    ```

    Поради щодо резервного копіювання: [Де дані зберігаються на диску](/uk/help/faq#where-things-live-on-disk).

  </Accordion>

  <Accordion title="Чи варто запускати Gateway на ноутбуці або VPS?">
    Потрібна надійна робота 24/7? Використовуйте **VPS**. Хочете якнайпростішого налаштування й вас влаштовують
    режим сну та перезапуски? Запускайте локально.

    **Ноутбук (локальний Gateway)**

    - **Переваги:** немає витрат на сервер, прямий доступ до локальних файлів, видиме вікно браузера.
    - **Недоліки:** режим сну та перебої мережі розривають з’єднання, оновлення й перезавантаження ОС переривають роботу, ноутбук має залишатися активним.

    **VPS / хмара**

    - **Переваги:** постійна робота, стабільна мережа, немає проблем із режимом сну ноутбука, простіше підтримувати безперервну роботу.
    - **Недоліки:** часто без графічного інтерфейсу (використовуйте знімки екрана), лише віддалений доступ до файлів, для оновлень потрібен SSH.

    WhatsApp/Telegram/Slack/Mattermost/Discord добре працюють із VPS — справжній
    компроміс полягає у виборі між браузером без графічного інтерфейсу та видимим вікном. Див. [Браузер](/uk/tools/browser).

    Рекомендація за замовчуванням: VPS, якщо раніше виникали розриви з’єднання з Gateway; локальне розгортання чудово підходить,
    коли Mac активно використовується й потрібен доступ до локальних файлів або автоматизація
    інтерфейсу у видимому браузері.

  </Accordion>

  <Accordion title="Наскільки важливо запускати OpenClaw на окремому комп’ютері?">
    Це не обов’язково, але рекомендовано для надійності та ізоляції.

    - **Окремий хост (VPS/Mac mini/Raspberry Pi):** постійно ввімкнений, менше переривань через режим сну чи перезавантаження, простіші дозволи, легше забезпечити безперервну роботу.
    - **Спільний ноутбук або настільний комп’ютер:** підходить для тестування й активного використання, але слід очікувати пауз, коли комп’ютер переходить у режим сну або оновлюється.

    Найкраще з обох варіантів: розмістіть Gateway на окремому хості та під’єднайте ноутбук як
    **вузол** для локальних інструментів роботи з екраном, камерою та виконанням команд. Див. [Вузли](/uk/nodes) і [Безпека](/uk/gateway/security).

  </Accordion>

  <Accordion title="Які мінімальні вимоги до VPS і рекомендована ОС?">
    - **Абсолютний мінімум:** 1 vCPU, 1 ГБ оперативної пам’яті, ~500 МБ дискового простору.
    - **Рекомендовано:** 1-2 vCPU, 2 ГБ+ оперативної пам’яті для запасу ресурсів (журнали, медіафайли, кілька каналів). Інструменти Node й автоматизація браузера можуть потребувати значних ресурсів.

    ОС: **Ubuntu LTS** (або будь-яка сучасна Debian/Ubuntu) — найкраще протестований шлях установлення в Linux.

    Документація: [Linux](/uk/platforms/linux), [Хостинг на VPS](/uk/vps).

  </Accordion>

  <Accordion title="Чи можна запускати OpenClaw у віртуальній машині та які вимоги?">
    Так. Віртуальну машину слід розглядати як VPS: вона має бути постійно ввімкненою, доступною та мати достатньо оперативної пам’яті
    для Gateway і всіх увімкнених каналів.

    - **Абсолютний мінімум:** 1 vCPU, 1 ГБ оперативної пам’яті.
    - **Рекомендовано:** 2 ГБ+ оперативної пам’яті для кількох каналів, автоматизації браузера або інструментів роботи з медіафайлами.
    - **ОС:** Ubuntu LTS або інша сучасна Debian/Ubuntu.

    У Windows використовуйте **Windows Hub** для налаштування настільного середовища або WSL2 для віртуальної машини Gateway у стилі Linux
    із широкою сумісністю з інструментами. Див. [Windows](/uk/platforms/windows), [Хостинг на VPS](/uk/vps).
    Запуск macOS у віртуальній машині: див. [Віртуальна машина macOS](/uk/install/macos-vm).

  </Accordion>
