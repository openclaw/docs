## ภาพรวมการรองรับ

## แอปคู่หู Wear OS

แอปคู่หู Wear OS ใช้การเชื่อมต่อ Gateway ที่ผ่านการยืนยันตัวตนของโทรศัพท์ Android ที่จับคู่ไว้ นาฬิกาจะไม่ได้รับหรือจัดเก็บข้อมูลประจำตัวของ Gateway แอปสามารถเลือกเอเจนต์และเซสชัน อ่านบทสนทนาแบบจำกัดขอบเขต ส่งข้อความหรือคำตอบที่ป้อนด้วยเสียง ยกเลิกการทำงานที่กำลังดำเนินอยู่ เริ่มการสนทนาแบบเรียลไทม์ภายในเซสชันที่เลือก และเชื่อมต่อหรือตัดการเชื่อมต่อ Gateway ของโทรศัพท์ที่จับคู่ไว้ นอกจากนี้ยังมีการแจ้งเตือนคำตอบภายในเครื่อง รูปลักษณ์แบบมืดหรือสว่าง และตัวเลือกการอ่านคำตอบออกเสียงโดยอัตโนมัติ การควบคุมเอเจนต์และ Gateway จะเจรจาความสามารถเพื่อรองรับการอัปเดตโทรศัพท์และนาฬิกาที่ไม่พร้อมกัน การสนทนาแบบเรียลไทม์จะสตรีมเสียงไมโครโฟนและเสียงที่เล่นผ่านช่อง Wear OS Data Layer ชั่วคราว และหยุดเมื่อขาดการเชื่อมต่อกับโทรศัพท์ที่เลือก การเชื่อมต่อ Gateway หรือช่องเสียง

## ติดตั้งจากภายนอก Google Play

GitHub Releases รุ่นสมบูรณ์และรุ่นแก้ไขตามปกติประกอบด้วย `OpenClaw-Android.apk` แบบสากลและ `OpenClaw-Android-SHA256SUMS.txt` APK นี้บิลด์จากแท็กรีลีส ลงนามด้วยคีย์รีลีส OpenClaw Android และมีข้อมูลยืนยันแหล่งที่มาจาก GitHub Actions

เลือก [รีลีส](https://github.com/openclaw/openclaw/releases) ที่แสดงแอสเซ็ตทั้งสองรายการ จากนั้นดาวน์โหลดและตรวจสอบแท็กนั้นให้ตรงกันก่อนติดตั้งจากภายนอก:

```bash
release_tag=vYYYY.M.PATCH
gh release download "$release_tag" \
  --repo openclaw/openclaw \
  --pattern OpenClaw-Android.apk \
  --pattern OpenClaw-Android-SHA256SUMS.txt
sha256sum --check OpenClaw-Android-SHA256SUMS.txt
gh attestation verify OpenClaw-Android.apk \
  --repo openclaw/openclaw \
  --signer-workflow openclaw/openclaw/.github/workflows/android-release.yml \
  --source-ref "refs/tags/${release_tag}" \
  --deny-self-hosted-runners
```

<Warning>
การติดตั้งจาก Google Play และ APK แบบติดตั้งแยกใช้ช่องทางการอัปเดตที่แตกต่างกันและอาจมีข้อมูลประจำตัวการลงนามต่างกัน Android อาจกำหนดให้ถอนการติดตั้งแอปที่มีอยู่ก่อนสลับช่องทาง ซึ่งจะลบข้อมูลแอปภายในเครื่อง ควรใช้ช่องทางเดียวสำหรับการอัปเดตตามปกติ
</Warning>

## มิเรอร์และควบคุม Android จาก Mac ระยะไกล

[scrcpy](https://github.com/Genymobile/scrcpy) มิเรอร์หน้าจอ Android ในหน้าต่าง macOS และ
ส่งต่ออินพุตจากแป้นพิมพ์และตัวชี้ผ่าน Android Debug Bridge (ADB) ขั้นตอนการทำงานนี้ดำเนินการ
ฝั่งผู้ควบคุมและแยกจากการเชื่อมต่อ Node ของ OpenClaw ซึ่งมีประโยชน์เมื่ออุปกรณ์ Android และ
Mac อยู่คนละตำแหน่งแต่ใช้เครือข่าย Tailscale ส่วนตัวร่วมกัน

### ก่อนเริ่มต้น

- ติดตั้ง Tailscale บนอุปกรณ์ Android และ Mac แล้วเชื่อมต่อทั้งสองเครื่องกับ tailnet เดียวกัน
- บน Android ให้เปิดใช้ **Developer options** และ **USB debugging** Android 16 วาง **Wireless
  debugging** ไว้ใต้ **Settings > System > Developer options** ดู [ตัวเลือกสำหรับนักพัฒนา
  Android](https://developer.android.com/studio/debug/dev-options)
- ติดตั้ง scrcpy และ ADB บน Mac:

  ```bash
  brew install scrcpy
  brew install --cask android-platform-tools
  ```

- เตรียมอุปกรณ์ Android ให้พร้อมสำหรับการเชื่อมต่อครั้งแรก Android ต้องอนุมัติคีย์ ADB
  ของ Mac แต่ละเครื่องก่อนที่ Mac เครื่องนั้นจะควบคุมอุปกรณ์ได้

### เปิดใช้ ADB ผ่าน TCP

สำหรับการตั้งค่าครั้งแรก ให้เชื่อมต่ออุปกรณ์ Android กับคอมพิวเตอร์ที่เชื่อถือได้ผ่าน USB และอนุมัติ
ข้อความแจ้งการดีบัก จากนั้นเรียกใช้:

```bash
adb devices
adb tcpip 5555
```

จากนั้นสามารถถอด USB ได้ หากพอร์ต 5555 หยุดรับฟังหลังรีบูตอุปกรณ์หรือรีเซ็ตการดีบัก
ให้ทำขั้นตอนการตั้งค่าภายในเครื่องนี้ซ้ำ Android 11 และใหม่กว่ายังสามารถสร้างความเชื่อถือครั้งแรกด้วย
**Wireless debugging > Pair device with pairing code** และ `adb pair`

### อนุญาตเฉพาะ Mac ที่ใช้ควบคุม

tailnet ที่ใช้การให้สิทธิ์แบบจำกัดต้องอนุญาตให้ Mac ที่ใช้ควบคุมเข้าถึงพอร์ต TCP 5555
บนอุปกรณ์ Android อย่างชัดเจน เพิ่มกฎที่จำกัดขอบเขตลงในนโยบาย tailnet โดยแทนที่ที่อยู่ตัวอย่าง
ด้วย IP Tailscale แบบคงที่ของอุปกรณ์ทั้งสอง:

```json5
{
  grants: [
    {
      src: ["<remote-mac-tailnet-ip>"],
      dst: ["<android-tailnet-ip>"],
      ip: ["tcp:5555"],
    },
  ],
}
```

ดู [การให้สิทธิ์ของ Tailscale](https://tailscale.com/docs/reference/syntax/grants) สำหรับนามแฝงโฮสต์และ
ตัวเลือกอื่นๆ อย่าเปิดพอร์ตนี้ให้แก่อินเทอร์เน็ตสาธารณะหรือเปิดเผยผ่าน Funnel เนื่องจากไคลเอนต์ ADB
ที่ได้รับอนุญาตสามารถควบคุมอุปกรณ์ได้อย่างกว้างขวาง

### เชื่อมต่อและเริ่มมิเรอร์

บน Mac ระยะไกล:

```bash
adb connect <android-tailnet-ip>:5555
adb devices
scrcpy --serial <android-tailnet-ip>:5555
```

`adb connect` ครั้งแรกจาก Mac เครื่องนี้จะแสดงกล่องโต้ตอบการอนุญาตบน Android ปลดล็อกอุปกรณ์
ยืนยันลายนิ้วมือของคีย์ และเลือก **Always allow from this computer** เฉพาะเมื่อเชื่อถือ
Mac เครื่องนั้น รายการ `adb devices` ที่สำเร็จจะลงท้ายด้วย `device`; `unauthorized` หมายความว่ายังไม่ได้อนุมัติข้อความแจ้ง
บนอุปกรณ์

เมื่อหน้าต่าง scrcpy เปิดขึ้น ให้ใช้งานโดยตรงหรือกำหนดเป็นเป้าหมายของเครื่องมือทำงานอัตโนมัติบนหน้าจอ macOS เช่น
[Peekaboo](https://peekaboo.sh/) scrcpy รับส่งภาพหน้าจอและอินพุต ส่วน Tailscale ทำหน้าที่เพียงจัดเตรียม
เส้นทางเครือข่ายส่วนตัว

### การแก้ไขปัญหา

- `Connection timed out`: ตรวจสอบการให้สิทธิ์ของ tailnet สำหรับ TCP 5555 การทำงานสำเร็จของ `tailscale ping` พิสูจน์ได้เพียง
  ว่าเพียร์เข้าถึงกันได้ ไม่ได้พิสูจน์ว่านโยบายอนุญาตพอร์ต TCP นี้ ให้ทดสอบด้วย
  `nc -vz <android-tailnet-ip> 5555` จาก Mac
- `unauthorized`: ปลดล็อก Android และอนุมัติคีย์ ADB ของ Mac ระยะไกล หรือลบเวิร์กสเตชันเก่า
  ใต้ **Wireless debugging > Paired devices** แล้วจับคู่ใหม่
- `Connection refused`: เชื่อมต่อภายในเครื่องอีกครั้งแล้วเรียกใช้ `adb tcpip 5555` ซ้ำ
- มีอุปกรณ์มากกว่าหนึ่งเครื่องในรายการ: ให้คงอาร์กิวเมนต์ `--serial <android-tailnet-ip>:5555` ที่ระบุไว้อย่างชัดเจน

เมื่อเสร็จสิ้น ให้ปิด scrcpy และตัดการเชื่อมต่อ ADB:

```bash
adb disconnect <android-tailnet-ip>:5555
```

## คู่มือปฏิบัติงานการเชื่อมต่อ

แอป Node บน Android ⇄ (mDNS/NSD + WebSocket) ⇄ **Gateway**

Android เชื่อมต่อโดยตรงกับ WebSocket ของ Gateway และใช้การจับคู่อุปกรณ์ (`role: node`)

สำหรับ Tailscale หรือโฮสต์สาธารณะ Android ต้องใช้ปลายทางที่ปลอดภัย:

- แนะนำ: Tailscale Serve / Funnel พร้อม `https://<magicdns>` / `wss://<magicdns>`
- รองรับเช่นกัน: URL ของ Gateway แบบ `wss://` อื่นใดก็ตามที่มีปลายทาง TLS จริง
- ยังคงรองรับ `ws://` แบบข้อความธรรมดาบนที่อยู่ LAN ส่วนตัว / โฮสต์ `.local` รวมถึง `localhost`, `127.0.0.1` และบริดจ์ของโปรแกรมจำลอง Android (`10.0.2.2`); การตั้งค่าที่ไม่ใช่ลูปแบ็กจะใช้สิทธิ์ผู้ควบคุมแบบจำกัดโดยอัตโนมัติ

### ข้อกำหนดเบื้องต้น

- Gateway กำลังทำงานบนเครื่องอื่น (หรือเข้าถึงได้ผ่าน SSH)
- อุปกรณ์/โปรแกรมจำลอง Android สามารถเข้าถึง WebSocket ของ Gateway:
  - อยู่บน LAN เดียวกันพร้อม mDNS/NSD **หรือ**
  - อยู่บน tailnet ของ Tailscale เดียวกันโดยใช้ Wide-Area Bonjour / unicast DNS-SD (ดูด้านล่าง) **หรือ**
  - ระบุโฮสต์/พอร์ตของ Gateway ด้วยตนเอง (ทางเลือกสำรอง)
- การจับคู่ผ่าน tailnet/เครือข่ายมือถือสาธารณะ **ไม่** ใช้ปลายทาง IP ของ tailnet แบบดิบ `ws://` ให้ใช้ Tailscale Serve หรือ URL แบบ `wss://` อื่นแทน
- มี CLI `openclaw` บนเครื่อง Gateway (หรือผ่าน SSH) เพื่ออนุมัติคำขอจับคู่

### 1. เริ่ม Gateway

```bash
openclaw gateway --port 18789 --verbose
```

ยืนยันว่าในบันทึกมีข้อความลักษณะดังนี้:

- `listening on ws://0.0.0.0:18789`

สำหรับการเข้าถึง Android ระยะไกลผ่าน Tailscale ให้เลือกใช้ Serve/Funnel แทนการผูกกับ tailnet IP แบบดิบ:

```bash
openclaw gateway --tailscale serve
```

วิธีนี้ทำให้ Android ได้ปลายทาง `wss://` / `https://` ที่ปลอดภัย การตั้งค่า `gateway.bind: "tailnet"` แบบธรรมดาไม่เพียงพอสำหรับการจับคู่ Android ระยะไกลครั้งแรก เว้นแต่จะยุติ TLS แยกต่างหากด้วย

### 2. ตรวจสอบการค้นพบ (ไม่บังคับ)

จากเครื่อง Gateway:

```bash
dns-sd -B _openclaw-gw._tcp local.
```

หมายเหตุการดีบักเพิ่มเติม: [Bonjour](/th/gateway/bonjour)

หากกำหนดค่าโดเมนการค้นพบแบบ Wide-Area ไว้ด้วย ให้เปรียบเทียบกับ:

```bash
openclaw gateway discover --json
```

คำสั่งนี้จะแสดง `local.` พร้อมโดเมน Wide-Area ที่กำหนดค่าไว้ในครั้งเดียว โดยใช้ปลายทางบริการที่แก้ไขแล้วแทนคำใบ้จาก TXT เพียงอย่างเดียว

#### การค้นพบข้ามเครือข่ายผ่าน unicast DNS-SD

การค้นพบด้วย Android NSD/mDNS ไม่สามารถข้ามเครือข่ายได้ หาก Node Android และ Gateway อยู่คนละเครือข่ายแต่เชื่อมต่อผ่าน Tailscale ให้ใช้ Wide-Area Bonjour / unicast DNS-SD แทน การค้นพบเพียงอย่างเดียวไม่เพียงพอสำหรับการจับคู่ Android ผ่าน tailnet/เครือข่ายสาธารณะ — เส้นทางที่ค้นพบยังคงต้องมีปลายทางที่ปลอดภัย (`wss://` หรือ Tailscale Serve):

1. ตั้งค่าโซน DNS-SD (ตัวอย่างเช่น `openclaw.internal.`) บนโฮสต์ Gateway และเผยแพร่ระเบียน `_openclaw-gw._tcp`
2. กำหนดค่า split DNS ของ Tailscale สำหรับโดเมนที่เลือกให้ชี้ไปยังเซิร์ฟเวอร์ DNS นั้น

รายละเอียดและตัวอย่างการกำหนดค่า CoreDNS: [Bonjour](/th/gateway/bonjour)

### 3. เชื่อมต่อจาก Android

ในแอป Android:

- แอปจะรักษาการเชื่อมต่อ Gateway ให้ทำงานอยู่ผ่าน **foreground service** (การแจ้งเตือนถาวร)
- เปิดแท็บ **Connect**
- ใช้โหมด **Setup Code** หรือ **Manual**
- หากการค้นพบถูกบล็อก ให้ใช้โฮสต์/พอร์ตแบบกำหนดเองใน **Advanced controls** สำหรับโฮสต์ LAN ส่วนตัว `ws://` ยังคงใช้งานได้ สำหรับโฮสต์ Tailscale/สาธารณะ ให้เปิด TLS และใช้ปลายทาง `wss://` / Tailscale Serve

หลังการจับคู่ครั้งแรกสำเร็จ Android จะเชื่อมต่อใหม่โดยอัตโนมัติเมื่อเปิดแอปไปยัง Gateway ที่จับคู่และเปิดใช้งานอยู่ (พยายามอย่างดีที่สุดสำหรับ Gateway ที่ค้นพบ ซึ่งต้องมองเห็นได้บนเครือข่าย)

รหัสตั้งค่าอย่างเป็นทางการจะเชื่อมต่อ Android เป็น Node และให้สิทธิ์ผู้ควบคุม Gateway
แบบเต็มตามค่าเริ่มต้นผ่าน `wss://` การตั้งค่า `ws://` ที่ไม่ใช่ลูปแบ็กแบบข้อความธรรมดา
จะใช้สิทธิ์แบบจำกัดโดยอัตโนมัติเพื่อความปลอดภัยของ bearer token **Settings → Gateway**
จะแสดงสิทธิ์ **Full** หรือ **Limited** สำหรับการเชื่อมต่อแบบจำกัด ให้กำหนดค่า
`wss://` หรือ Tailscale Serve สร้างรหัสสิทธิ์เต็มใหม่ใน Control UI หรือ
ด้วย `openclaw qr` จากนั้นสแกนหรือวางรหัสในหน้านั้นแล้วเชื่อมต่อใหม่ ผู้ควบคุม
ที่ต้องการโปรไฟล์แบบลดสิทธิ์สามารถเลือก **Limited access** ใน Control UI หรือเรียกใช้
`openclaw qr --limited`

### Gateway หลายรายการ

แอปจะเก็บรีจิสทรีของ Gateway ทุกตัวที่เคยจับคู่ไว้ จึงสามารถสลับระหว่าง Gateway เหล่านั้นได้โดยไม่ต้องจับคู่ใหม่:

- **Settings -> Gateways** แสดงรายการ Gateway ที่จับคู่แล้ว โดยทำเครื่องหมาย Gateway ที่ใช้งานอยู่ แตะรายการเพื่อสลับ แอปจะยุติเซสชันปัจจุบันและเชื่อมต่อกับ Gateway ที่เลือกใหม่
- แท็บ **Connect** จะแสดงตัวสลับด่วนเมื่อจับคู่ Gateway ไว้มากกว่าหนึ่งรายการ
- ข้อมูลรับรอง โทเค็นอุปกรณ์ ความเชื่อถือ TLS ประวัติแชต และข้อความออฟไลน์ที่เข้าคิวจะจัดเก็บแยกตาม Gateway การสลับจะไม่ผสมสถานะระหว่าง Gateway และข้อความที่เข้าคิวขณะออฟไลน์จะส่งไปยัง Gateway ปลายทางที่ข้อความนั้นเขียนไว้เท่านั้น
- **Forget** จะลบรายการรีจิสทรีของ Gateway พร้อมข้อมูลรับรอง โทเค็นอุปกรณ์ พิน TLS และแชตที่แคชไว้

### บีคอนแสดงว่ายังออนไลน์

หลังจากเชื่อมต่อเซสชัน Node ที่ผ่านการยืนยันตัวตนแล้ว และเมื่อแอปย้ายไปทำงานเบื้องหลังขณะที่บริการเบื้องหน้ายังคงเชื่อมต่ออยู่ Android จะเรียก `node.event` พร้อม `event: "node.presence.alive"` โดย Gateway จะบันทึกข้อมูลนี้เป็น `lastSeenAtMs`/`lastSeenReason` ในเมทาดาทาของ Node/อุปกรณ์ที่จับคู่เท่านั้น หลังจากทราบข้อมูลประจำตัวของอุปกรณ์ Node ที่ผ่านการยืนยันตัวตนแล้ว

แอปจะถือว่าบันทึกบีคอนสำเร็จเฉพาะเมื่อการตอบกลับจาก Gateway มี `handled: true` เท่านั้น Gateway รุ่นเก่าอาจตอบรับ `node.event` ด้วย `{ "ok": true }` การตอบกลับดังกล่าวใช้งานร่วมกันได้ แต่ไม่นับเป็นการอัปเดตเวลาที่พบล่าสุดแบบคงทน

### 4. อนุมัติการจับคู่ (CLI)

บนเครื่อง Gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

รายละเอียดการจับคู่: [การจับคู่](/th/channels/pairing)

ทางเลือก: หาก Node Android เชื่อมต่อจากซับเน็ตที่ควบคุมอย่างเข้มงวดเสมอ คุณสามารถเลือกใช้การอนุมัติ Node อัตโนมัติในการจับคู่ครั้งแรก โดยระบุ CIDR หรือ IP ที่แน่นอนอย่างชัดเจน:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ฟีเจอร์นี้ปิดใช้งานโดยค่าเริ่มต้น และใช้เฉพาะกับการจับคู่ `role: node` ใหม่ที่ไม่มีขอบเขตที่ร้องขอ การจับคู่ของผู้ดำเนินการ/เบราว์เซอร์ รวมถึงการเปลี่ยนแปลงบทบาท ขอบเขต เมทาดาทา หรือกุญแจสาธารณะใด ๆ ยังคงต้องได้รับการอนุมัติด้วยตนเอง

### 6. แชตและประวัติ

แท็บ Chat ของ Android รองรับการเลือกเซสชัน (ค่าเริ่มต้น `main` รวมถึงเซสชันอื่นที่มีอยู่):

- ประวัติ: `chat.history` (ปรับรูปแบบเพื่อการแสดงผลแล้ว โดยจะตัดแท็กคำสั่งแบบอินไลน์ เพย์โหลด XML ของการเรียกใช้เครื่องมือในรูปแบบข้อความธรรมดา (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>` และรูปแบบที่ถูกตัดทอน) รวมถึงโทเค็นควบคุมโมเดลแบบ ASCII/ความกว้างเต็มที่รั่วไหลออกมา แถวผู้ช่วยที่มีเฉพาะโทเค็นเงียบ เช่น `NO_REPLY` / `no_reply` ที่ตรงกันทุกประการจะถูกละเว้น และแถวที่มีขนาดใหญ่เกินไปอาจถูกแทนที่ด้วยตัวแทนข้อความ)
- ส่ง: `chat.send`
- การส่งแบบคงทน: การส่งทุกครั้ง (ข้อความ รูปภาพที่เลือก และข้อความเสียง) จะถูกบันทึกลงในกล่องขาออกบนอุปกรณ์ที่แยกตาม Gateway ก่อนพยายามเชื่อมต่อเครือข่าย เพื่อให้การปิดแอปไม่ทำให้ข้อมูลที่ส่งแล้วสูญหาย รายการส่งที่เข้าคิวขณะออฟไลน์จะถูกส่งตามลำดับเมื่อเชื่อมต่อใหม่โดยใช้คีย์การทำงานซ้ำอย่างปลอดภัยที่คงที่ และรายการส่งจะถูกนำออกก็ต่อเมื่อเทิร์นนั้นปรากฏใน `chat.history` ที่เป็นข้อมูลหลักแล้วเท่านั้น การตอบรับเพียงอย่างเดียวจะไม่ถือเป็นหลักฐานการส่งสำเร็จ ผลลัพธ์ที่ไม่ชัดเจน (การตอบรับสูญหาย แอปถูกปิดระหว่างส่ง หรือ Gateway รีสตาร์ตก่อนเขียนทรานสคริปต์) จะแสดงเป็นแถวที่มองเห็นได้พร้อม **Retry**/**Delete** อย่างชัดเจน แทนการส่งซ้ำอัตโนมัติ คำสั่งแบบสแลชจะไม่เล่นซ้ำอัตโนมัติหลังเชื่อมต่อใหม่ แต่จะพักไว้เพื่อให้ลองใหม่อย่างชัดเจน คิวมีขีดจำกัด (50 ข้อความและข้อมูลไฟล์แนบ 48 MB ต่อ Gateway) และแถวที่ยังไม่ได้ส่งจะหมดอายุหลังจาก 48 ชั่วโมง แบบร่างในช่องเขียนที่ไม่เคยส่งจะไม่คงอยู่หลังสิ้นสุดโปรเซส
- การอัปเดตแบบพุช (ตามความพยายามสูงสุด): `chat.subscribe` -> `event:"chat"`
- ฟัง: กดข้อความของผู้ช่วยค้างไว้แล้วเลือก **Listen** เพื่อฟัง ระบบจะเรนเดอร์เสียงผ่าน `tts.speak` ของ Gateway โดยใช้สายโซ่ผู้ให้บริการ TTS ที่กำหนดค่าไว้ และใช้ TTS ของระบบบนอุปกรณ์เมื่อ Gateway ไม่สามารถเรนเดอร์เสียงได้ การเล่นจะหยุดเมื่อสลับเซสชัน เริ่มแชตใหม่ แอปทำงานเบื้องหลัง หรือปิดแชต

### 7. Canvas และกล้อง

#### โฮสต์ Canvas ของ Gateway (แนะนำสำหรับเนื้อหาเว็บ)

หากต้องการให้ Node แสดง HTML/CSS/JS จริงที่เอเจนต์สามารถแก้ไขบนดิสก์ได้ ให้กำหนด Node ไปยังโฮสต์ Canvas ของ Gateway

<Note>
Node จะโหลด Canvas จากเซิร์ฟเวอร์ HTTP ของ Gateway (พอร์ตเดียวกับ `gateway.port` ค่าเริ่มต้น `18789`)
</Note>

1. สร้าง `~/.openclaw/workspace/canvas/index.html` บนโฮสต์ Gateway
2. นำทาง Node ไปยังตำแหน่งดังกล่าว (LAN):

```bash
openclaw nodes invoke --node "<Android Node>" --command canvas.navigate --params '{"url":"http://<gateway-hostname>.local:18789/__openclaw__/canvas/"}'
```

Tailnet (ไม่บังคับ): หากอุปกรณ์ทั้งสองอยู่บน Tailscale ให้ใช้ชื่อ MagicDNS หรือ IP ของ Tailnet แทน `.local` เช่น `http://<gateway-magicdns>:18789/__openclaw__/canvas/`

เซิร์ฟเวอร์นี้จะแทรกไคลเอนต์โหลดใหม่แบบสดลงใน HTML และโหลดใหม่เมื่อไฟล์เปลี่ยนแปลง Gateway ยังให้บริการ `/__openclaw__/a2ui/` ด้วย แต่แอป Android จะถือว่าหน้า A2UI ระยะไกลใช้สำหรับการเรนเดอร์เท่านั้น คำสั่ง A2UI ที่ดำเนินการได้จะใช้หน้า A2UI ที่เป็นของแอปและรวมมาในชุด

คำสั่ง Canvas (เฉพาะเบื้องหน้า):

- `canvas.eval`, `canvas.snapshot`, `canvas.navigate` (ใช้ `{"url":""}` หรือ `{"url":"/"}` เพื่อกลับไปยังโครงตั้งต้นเริ่มต้น) `canvas.snapshot` จะส่งคืน `{ format, base64 }` (ค่าเริ่มต้น `format="jpeg"`)
- A2UI: `canvas.a2ui.push`, `canvas.a2ui.reset` (`canvas.a2ui.pushJSONL` เป็นชื่อแทนแบบเก่า) คำสั่งเหล่านี้ใช้หน้า A2UI ที่เป็นของแอปและรวมมาในชุดสำหรับการเรนเดอร์ที่ดำเนินการได้

คำสั่งกล้อง (เฉพาะเบื้องหน้า และควบคุมด้วยสิทธิ์): `camera.snap` (jpg), `camera.clip` (mp4) ดูพารามิเตอร์และตัวช่วย CLI ที่ [Node กล้อง](/th/nodes/camera)

### 8. เสียงและพื้นผิวคำสั่ง Android ที่ขยายเพิ่มเติม

- การนำทางหลักของ Android คือ **Home**, **Chat** และ **Settings** การป้อนข้อมูลด้วยเสียง
  อยู่ในช่องเขียนของ Chat และไม่มีแท็บ Voice แยกต่างหาก
- แตะไมโครโฟนในช่องเขียนเพื่อใช้การรู้จำเสียงพูดบนอุปกรณ์ ซึ่งจะแทรก
  ทรานสคริปต์ลงในแบบร่าง กดไมโครโฟนค้างไว้เพื่อบันทึกไฟล์แนบข้อความเสียง
  UI จะแจ้งเมื่อใช้การรู้จำไม่ได้ ไม่มีสิทธิ์
  เกิดข้อผิดพลาดเนื่องจากระบบไม่ว่าง/เครือข่าย และไม่มีเสียงพูด แทนที่จะละทิ้ง
  ความพยายามโดยไม่แจ้ง
- เริ่ม **Talk** แบบต่อเนื่องจากรูปคลื่นใน Chat การป้อนตามคำบอก การบันทึกข้อความเสียง
  และ Talk เป็นเส้นทางไมโครโฟนที่ไม่สามารถทำงานพร้อมกันได้
- Talk Mode จะเลื่อนระดับบริการเบื้องหน้าที่มีอยู่จาก `connectedDevice` เป็น `connectedDevice|microphone` ก่อนเริ่มบันทึก แล้วลดระดับเมื่อ Talk Mode หยุด บริการ Node ประกาศ `FOREGROUND_SERVICE_CONNECTED_DEVICE` พร้อม `CHANGE_NETWORK_STATE` ส่วน Android 14+ ยังต้องมีการประกาศ `FOREGROUND_SERVICE_MICROPHONE` การให้สิทธิ์รันไทม์ `RECORD_AUDIO` และประเภทบริการไมโครโฟนขณะรันไทม์
- โดยค่าเริ่มต้น Android Talk จะใช้การรู้จำเสียงพูดแบบเนทีฟ แชตผ่าน Gateway และ `talk.speak` ผ่านผู้ให้บริการ Talk ของ Gateway ที่กำหนดค่าไว้ ระบบจะใช้ TTS ภายในเครื่องเฉพาะเมื่อ `talk.speak` ไม่พร้อมใช้งาน
- Android Talk จะใช้รีเลย์ Gateway แบบเรียลไทม์เฉพาะเมื่อ `talk.realtime.mode` เป็น `realtime` และ `talk.realtime.transport` เป็น `gateway-relay`
- Android ไม่ประกาศความสามารถ `voiceWake` ใช้การป้อนตามคำบอกใน Chat
  ข้อความเสียง หรือ Talk สำหรับการป้อนข้อมูลด้วยเสียง
- กลุ่มคำสั่ง Android เพิ่มเติม (ความพร้อมใช้งานขึ้นอยู่กับอุปกรณ์ สิทธิ์ และการตั้งค่าของผู้ใช้):
  - `device.status`, `device.info`, `device.permissions`, `device.health`
  - `device.apps` เฉพาะเมื่อเปิดใช้ **Settings > Phone Capabilities > Installed Apps** โดยค่าเริ่มต้นจะแสดงรายการแอปที่มองเห็นได้ในตัวเรียกใช้งาน (ส่ง `includeNonLaunchable` เพื่อดูรายการทั้งหมด)
  - `notifications.list`, `notifications.actions` (ดู[การส่งต่อการแจ้งเตือน](#notification-forwarding)ด้านล่าง)
  - `photos.latest`
  - `contacts.search`, `contacts.add`
  - `calendar.events`, `calendar.add`
  - `callLog.search`
  - `sms.search`
  - `motion.activity`, `motion.pedometer`

### 9. ไฟล์พื้นที่ทำงาน (อ่านอย่างเดียว)

ภาพรวมหน้า Home มีการ์ด **Files** สำหรับเรียกดูพื้นที่ทำงานของเอเจนต์ที่ใช้งานอยู่ผ่าน RPC ของ Gateway แบบอ่านอย่างเดียว `agents.workspace.list` / `agents.workspace.get` ซึ่งรองรับการเจาะดูไดเรกทอรี การแสดงตัวอย่างข้อความและรูปภาพ และการส่งออกผ่านแผ่นงานแชร์ของ Android ไม่มีการดำเนินการเขียน และ Gateway จะจำกัดขนาดของตัวอย่าง

`operator.approvals` ที่จับคู่แล้วและถูก Gateway กำหนดเป้าหมายอย่างชัดเจน สามารถตรวจสอบ
คำขอ exec ที่รออยู่ภายใต้ **Settings -> Approvals** แอปจะโหลด
ระเบียนการอนุมัติที่ผ่านการกรองความปลอดภัยของ Gateway ก่อนเปิดใช้งานปุ่มต่าง ๆ แสดง
คำเตือนด้านความปลอดภัยและการตัดสินใจที่คำขอนั้นเสนออย่างครบถ้วน และส่ง
ID การอนุมัติและประเภทเจ้าของกลับไปยัง Gateway

สถานะการอนุมัติใช้ร่วมกับ Control UI และพื้นผิวแชตที่รองรับ
คำตอบแรกที่ยืนยันแล้วจะมีผล Android จะแสดงผลลัพธ์หลักดังกล่าวแม้ว่า
พื้นผิวอื่นจะตอบก่อน หากการตอบกลับการแก้ไขสูญหายหรือ Gateway
ตัดการเชื่อมต่อ แอปจะคงการล็อกการดำเนินการไว้และอ่านข้อมูลการอนุมัติอีกครั้ง
ก่อนเสนอการตัดสินใจอื่น

Gateway ที่เก่ากว่าวิธีการอนุมัติแบบรวมจะถอยกลับไปใช้
วิธีการเฉพาะสำหรับ exec ที่จัดส่งมาแล้ว การตรวจสอบรายการที่รออยู่ยังคงทำงานได้ แต่สถานะเทอร์มินัล
ที่เก็บรักษาไว้และผลลัพธ์ข้ามพื้นผิวที่สมบูรณ์ยิ่งขึ้นจำเป็นต้องใช้ Gateway ที่อัปเดตแล้ว

## ตอบคำถามของเอเจนต์

Chat จะแสดงคำถามจาก Gateway ที่รออยู่เป็นการ์ดแบบเนทีฟสำหรับการเชื่อมต่อของผู้ดำเนินการ
ที่มี `operator.questions` (หรือ `operator.admin`) การ์ดรองรับตัวเลือกแบบเลือกเดียวและ
หลายตัวเลือก คำอธิบายตัวเลือก คำตอบแบบข้อความอิสระ **Other** และ
การนับถอยหลังจนหมดอายุ การเชื่อมต่อใหม่จะโหลดคำถามที่รออยู่จาก Gateway อีกครั้ง การ์ด
จะล็อกเมื่ออุปกรณ์นี้ตอบ พื้นผิวอื่นตอบก่อน หรือ
คำถามหมดอายุหรือถูกยกเลิก

## จุดเริ่มต้นของผู้ช่วย

Android รองรับการเปิด OpenClaw จากทริกเกอร์ผู้ช่วยระบบ (Google Assistant) การกดปุ่ม Home ค้างไว้ (หรือทริกเกอร์ `ACTION_ASSIST` อื่น) จะเปิดแอป ส่วนการพูดว่า "Hey Google, ask OpenClaw `<prompt>`" จะตรงกับรูปแบบคำค้นหา App Actions ที่แอปประกาศไว้ และส่งพรอมต์ไปยังช่องเขียนแชตโดยไม่ส่งอัตโนมัติ

ฟีเจอร์นี้ใช้ **App Actions** ของ Android (ความสามารถ `shortcuts.xml`) ที่ประกาศไว้ในไฟล์แมนิเฟสต์ของแอป ไม่จำเป็นต้องกำหนดค่าฝั่ง Gateway เนื่องจากแอป Android จะจัดการเจตนาของผู้ช่วยทั้งหมด

<Note>
ความพร้อมใช้งานของ App Actions ขึ้นอยู่กับอุปกรณ์ เวอร์ชัน Google Play Services และผู้ใช้ได้ตั้ง OpenClaw เป็นแอปผู้ช่วยเริ่มต้นหรือไม่
</Note>

## การส่งต่อการแจ้งเตือน

Android สามารถส่งต่อการแจ้งเตือนของอุปกรณ์ไปยัง Gateway เป็นรายการ `node.event` ได้ ฟีเจอร์นี้กำหนดค่า **บนอุปกรณ์** ในแผ่นงาน Settings ของแอป ไม่ใช่ในการกำหนดค่า gateway/`openclaw.json`

| การตั้งค่า                     | คำอธิบาย                                                                                                                                                                                            |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ส่งต่อเหตุการณ์การแจ้งเตือน | สวิตช์หลัก ปิดไว้ตามค่าเริ่มต้น และต้องให้สิทธิ์เข้าถึงตัวฟังการแจ้งเตือนก่อน                                                                                                              |
| ตัวกรองแพ็กเกจ              | **รายการที่อนุญาต** (ส่งต่อเฉพาะ ID แพ็กเกจที่ระบุไว้) หรือ **รายการที่บล็อก** (ค่าเริ่มต้น: ทุกแพ็กเกจยกเว้น ID ที่ระบุไว้) แพ็กเกจของ OpenClaw จะถูกยกเว้นเสมอในโหมดรายการที่บล็อก เพื่อป้องกันการส่งต่อวนซ้ำ |
| ช่วงเวลางดรบกวน                 | ช่วงเวลาเริ่มต้น/สิ้นสุดตามเวลาท้องถิ่นในรูปแบบ HH:mm ซึ่งระงับการส่งต่อ ปิดใช้งานตามค่าเริ่มต้น และเมื่อเปิดใช้งานจะมีค่าเริ่มต้นเป็น `22:00`-`07:00`                                                                                |
| จำนวนเหตุการณ์สูงสุด / นาที         | ขีดจำกัดอัตราการแจ้งเตือนที่ส่งต่อต่ออุปกรณ์ ค่าเริ่มต้นคือ 20                                                                                                                                          |
| คีย์เซสชันเส้นทาง           | ไม่บังคับ กำหนดให้เหตุการณ์การแจ้งเตือนที่ส่งต่อเข้าสู่เซสชันที่ระบุ แทนเส้นทางการแจ้งเตือนเริ่มต้นของอุปกรณ์                                                                               |

<Note>
การส่งต่อการแจ้งเตือนต้องได้รับสิทธิ์ตัวฟังการแจ้งเตือนของ Android แอปจะแจ้งให้ให้สิทธิ์นี้ระหว่างการตั้งค่า
</Note>

การแจ้งเตือนจาก WhatsApp, WhatsApp Business, Telegram, Telegram X, Discord และ Signal จะถูกยกเว้นเสมอ ข้อความเหล่านี้ได้รับการจัดการโดยเซสชันช่องทางดั้งเดิมของ OpenClaw อยู่แล้ว การส่งต่อการแจ้งเตือนของ Android เป็นเหตุการณ์ Node แยกต่างหากอาจทำให้การตอบกลับถูกส่งไปยังการสนทนาที่ไม่ถูกต้อง
