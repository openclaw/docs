สถานะพร้อมใช้งาน: รุ่นพรีวิวภายใน แอป iOS ยังไม่ได้เผยแพร่สู่สาธารณะ

## สิ่งที่ทำได้

- เชื่อมต่อกับ Gateway ผ่าน WebSocket (LAN หรือ tailnet)
- เปิดเผยความสามารถของโหนด: แคนวาส, สแนปช็อตหน้าจอ, การจับภาพจากกล้อง, ตำแหน่งที่ตั้ง, โหมดพูดคุย, การปลุกด้วยเสียง
- รับคำสั่ง `node.invoke` และรายงานเหตุการณ์สถานะของโหนด

## ข้อกำหนด

- Gateway ที่ทำงานอยู่บนอุปกรณ์อีกเครื่องหนึ่ง (macOS, Linux หรือ Windows ผ่าน WSL2)
- เส้นทางเครือข่าย:
  - LAN เดียวกันผ่าน Bonjour, **หรือ**
  - Tailnet ผ่าน DNS-SD แบบยูนิคาสต์ (โดเมนตัวอย่าง: `openclaw.internal.`), **หรือ**
  - โฮสต์/พอร์ตแบบกำหนดเอง (ทางเลือกสำรอง)

## เริ่มต้นอย่างรวดเร็ว (จับคู่ + เชื่อมต่อ)

1. เริ่ม Gateway:

```bash
openclaw gateway --port 18789
```

2. ในแอป iOS ให้เปิดการตั้งค่าแล้วเลือก Gateway ที่ค้นพบ (หรือเปิดใช้โฮสต์แบบกำหนดเองและป้อนโฮสต์/พอร์ต)

3. อนุมัติคำขอจับคู่บนโฮสต์ของ Gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

หากแอปลองจับคู่อีกครั้งโดยมีรายละเอียดการยืนยันตัวตนที่เปลี่ยนไป (บทบาท/ขอบเขต/กุญแจสาธารณะ)
คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่ และจะสร้าง `requestId` ใหม่
ให้เรียกใช้ `openclaw devices list` อีกครั้งก่อนอนุมัติ

ไม่บังคับ: หากโหนด iOS เชื่อมต่อจากซับเน็ตที่ควบคุมอย่างเข้มงวดเสมอ คุณ
สามารถเลือกใช้การอนุมัติโหนดอัตโนมัติในครั้งแรกด้วย CIDR หรือ IP แบบระบุชัดเจน:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ค่านี้ปิดไว้โดยค่าเริ่มต้น ใช้เฉพาะกับการจับคู่ `role: node` ใหม่ที่
ไม่ได้ขอขอบเขตใดๆ การจับคู่ของผู้ควบคุม/เบราว์เซอร์ และการเปลี่ยนแปลงบทบาท ขอบเขต เมตาดาตา หรือ
กุญแจสาธารณะใดๆ ยังคงต้องอนุมัติด้วยตนเอง

4. ตรวจสอบการเชื่อมต่อ:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## พุชที่ใช้รีเลย์รองรับสำหรับบิลด์ทางการ

บิลด์ iOS ที่เผยแพร่อย่างเป็นทางการใช้รีเลย์พุชภายนอกแทนการเผยแพร่โทเค็น APNs ดิบ
ไปยัง Gateway

ข้อกำหนดฝั่ง Gateway:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

ลำดับการทำงานเป็นดังนี้:

- แอป iOS ลงทะเบียนกับรีเลย์โดยใช้ App Attest และ StoreKit app transaction JWS
- รีเลย์ส่งคืนแฮนเดิลรีเลย์แบบทึบและสิทธิ์ส่งที่ผูกกับการลงทะเบียน
- แอป iOS ดึงข้อมูลตัวตนของ Gateway ที่จับคู่แล้วและใส่ไว้ในการลงทะเบียนรีเลย์ ดังนั้นการลงทะเบียนที่ใช้รีเลย์รองรับจึงถูกมอบหมายให้ Gateway นั้นโดยเฉพาะ
- แอปส่งต่อการลงทะเบียนที่ใช้รีเลย์รองรับนั้นไปยัง Gateway ที่จับคู่แล้วด้วย `push.apns.register`
- Gateway ใช้แฮนเดิลรีเลย์ที่จัดเก็บไว้นั้นสำหรับ `push.test`, การปลุกเพื่อเชื่อมต่อใหม่ และการสะกิดให้ตื่น
- URL ฐานของรีเลย์ Gateway ต้องตรงกับ URL รีเลย์ที่ฝังอยู่ในบิลด์ iOS ทางการ/TestFlight
- หากแอปเชื่อมต่อกับ Gateway อื่นในภายหลัง หรือกับบิลด์ที่มี URL ฐานของรีเลย์ต่างกัน แอปจะรีเฟรชการลงทะเบียนรีเลย์แทนการใช้การผูกเดิมซ้ำ

สิ่งที่ Gateway **ไม่** ต้องใช้สำหรับเส้นทางนี้:

- ไม่มีโทเค็นรีเลย์สำหรับทั้งการติดตั้งใช้งาน
- ไม่มีกุญแจ APNs โดยตรงสำหรับการส่งแบบใช้รีเลย์รองรับของบิลด์ทางการ/TestFlight

ลำดับการทำงานที่คาดหวังสำหรับผู้ควบคุม:

1. ติดตั้งบิลด์ iOS ทางการ/TestFlight
2. ตั้งค่า `gateway.push.apns.relay.baseUrl` บน Gateway
3. จับคู่แอปกับ Gateway แล้วปล่อยให้เชื่อมต่อจนเสร็จ
4. แอปเผยแพร่ `push.apns.register` โดยอัตโนมัติหลังจากมีโทเค็น APNs, เซสชันผู้ควบคุมเชื่อมต่อแล้ว และการลงทะเบียนรีเลย์สำเร็จ
5. หลังจากนั้น `push.test`, การปลุกเพื่อเชื่อมต่อใหม่ และการสะกิดให้ตื่นจะใช้การลงทะเบียนที่จัดเก็บไว้ซึ่งใช้รีเลย์รองรับได้

## บีคอนสถานะมีชีวิตในเบื้องหลัง

เมื่อ iOS ปลุกแอปจากพุชแบบเงียบ, การรีเฟรชเบื้องหลัง หรือเหตุการณ์ตำแหน่งสำคัญ แอป
จะพยายามเชื่อมต่อโหนดใหม่ช่วงสั้นๆ แล้วเรียก `node.event` ด้วย `event: "node.presence.alive"`
Gateway บันทึกสิ่งนี้เป็น `lastSeenAtMs`/`lastSeenReason` บนเมตาดาตาของโหนด/อุปกรณ์ที่จับคู่แล้วเท่านั้น
หลังจากรู้ตัวตนอุปกรณ์โหนดที่ผ่านการยืนยันตัวตนแล้ว

แอปถือว่าการปลุกในเบื้องหลังถูกบันทึกสำเร็จเฉพาะเมื่อการตอบกลับของ Gateway มี
`handled: true` Gateway รุ่นเก่าอาจตอบรับ `node.event` ด้วย `{ "ok": true }`; การตอบกลับนั้น
เข้ากันได้ แต่ไม่นับเป็นการอัปเดตเวลาที่เห็นล่าสุดแบบถาวร

หมายเหตุความเข้ากันได้:

- `OPENCLAW_APNS_RELAY_BASE_URL` ยังคงใช้งานได้ในฐานะการแทนที่ชั่วคราวผ่านตัวแปรสภาพแวดล้อมสำหรับ Gateway

## การยืนยันตัวตนและลำดับความไว้วางใจ

รีเลย์มีไว้เพื่อบังคับใช้ข้อจำกัดสองอย่างที่ APNs โดยตรงบน Gateway ไม่สามารถให้ได้สำหรับ
บิลด์ iOS ทางการ:

- เฉพาะบิลด์ iOS ของ OpenClaw ของแท้ที่เผยแพร่ผ่าน Apple เท่านั้นที่ใช้รีเลย์แบบโฮสต์ได้
- Gateway สามารถส่งพุชที่ใช้รีเลย์รองรับได้เฉพาะสำหรับอุปกรณ์ iOS ที่จับคู่กับ Gateway นั้นโดยเฉพาะ

ทีละช่วงการเชื่อมต่อ:

1. `iOS app -> gateway`
   - แอปจับคู่กับ Gateway ก่อนผ่านลำดับการยืนยันตัวตน Gateway ตามปกติ
   - สิ่งนั้นให้เซสชันโหนดที่ผ่านการยืนยันตัวตนแก่แอป พร้อมกับเซสชันผู้ควบคุมที่ผ่านการยืนยันตัวตน
   - เซสชันผู้ควบคุมใช้สำหรับเรียก `gateway.identity.get`

2. `iOS app -> relay`
   - แอปเรียกปลายทางลงทะเบียนของรีเลย์ผ่าน HTTPS
   - การลงทะเบียนมีหลักฐาน App Attest พร้อมกับ StoreKit app transaction JWS
   - รีเลย์ตรวจสอบ ID บันเดิล, หลักฐาน App Attest และหลักฐานการเผยแพร่ของ Apple และกำหนดให้ใช้
     เส้นทางการเผยแพร่ทางการ/โปรดักชัน
   - นี่คือสิ่งที่บล็อกบิลด์ Xcode/นักพัฒนาในเครื่องไม่ให้ใช้รีเลย์แบบโฮสต์ บิลด์ในเครื่องอาจ
     ลงนามแล้ว แต่ไม่ผ่านหลักฐานการเผยแพร่ทางการของ Apple ตามที่รีเลย์คาดหวัง

3. `gateway identity delegation`
   - ก่อนลงทะเบียนรีเลย์ แอปดึงข้อมูลตัวตนของ Gateway ที่จับคู่แล้วจาก
     `gateway.identity.get`
   - แอปใส่ตัวตนของ Gateway นั้นไว้ในเพย์โหลดการลงทะเบียนรีเลย์
   - รีเลย์ส่งคืนแฮนเดิลรีเลย์และสิทธิ์ส่งที่ผูกกับการลงทะเบียน ซึ่งถูกมอบหมายให้
     ตัวตนของ Gateway นั้น

4. `gateway -> relay`
   - Gateway จัดเก็บแฮนเดิลรีเลย์และสิทธิ์ส่งจาก `push.apns.register`
   - เมื่อใช้ `push.test`, การปลุกเพื่อเชื่อมต่อใหม่ และการสะกิดให้ตื่น Gateway จะลงนามคำขอส่งด้วย
     ตัวตนอุปกรณ์ของตัวเอง
   - รีเลย์ตรวจสอบทั้งสิทธิ์ส่งที่จัดเก็บไว้และลายเซ็นของ Gateway เทียบกับตัวตน
     Gateway ที่ได้รับมอบหมายจากการลงทะเบียน
   - Gateway อื่นไม่สามารถใช้การลงทะเบียนที่จัดเก็บไว้นั้นซ้ำได้ แม้จะได้แฮนเดิลมาด้วยวิธีใดก็ตาม

5. `relay -> APNs`
   - รีเลย์เป็นเจ้าของข้อมูลประจำตัว APNs โปรดักชันและโทเค็น APNs ดิบสำหรับบิลด์ทางการ
   - Gateway ไม่เคยจัดเก็บโทเค็น APNs ดิบสำหรับบิลด์ทางการที่ใช้รีเลย์รองรับ
   - รีเลย์ส่งพุชสุดท้ายไปยัง APNs ในนามของ Gateway ที่จับคู่แล้ว

เหตุผลที่สร้างการออกแบบนี้:

- เพื่อเก็บข้อมูลประจำตัว APNs โปรดักชันให้อยู่นอก Gateway ของผู้ใช้
- เพื่อหลีกเลี่ยงการจัดเก็บโทเค็น APNs ดิบของบิลด์ทางการบน Gateway
- เพื่ออนุญาตให้ใช้รีเลย์แบบโฮสต์เฉพาะกับบิลด์ OpenClaw ทางการ/TestFlight เท่านั้น
- เพื่อป้องกันไม่ให้ Gateway หนึ่งส่งพุชปลุกไปยังอุปกรณ์ iOS ที่เป็นของ Gateway อื่น

บิลด์ในเครื่อง/แบบกำหนดเองยังคงใช้ APNs โดยตรง หากคุณกำลังทดสอบบิลด์เหล่านั้นโดยไม่มีรีเลย์
Gateway ยังคงต้องใช้ข้อมูลประจำตัว APNs โดยตรง:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

สิ่งเหล่านี้เป็นตัวแปรสภาพแวดล้อมรันไทม์ของโฮสต์ Gateway ไม่ใช่การตั้งค่า Fastlane `apps/ios/fastlane/.env` เก็บเฉพาะ
การยืนยันตัวตน App Store Connect / TestFlight เช่น `ASC_KEY_ID` และ `ASC_ISSUER_ID`; ไม่ได้กำหนดค่า
การส่ง APNs โดยตรงสำหรับบิลด์ iOS ในเครื่อง

การจัดเก็บที่แนะนำบนโฮสต์ Gateway:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

อย่าคอมมิตไฟล์ `.p8` หรือวางไว้ใต้เช็กเอาต์ของรีโป

## เส้นทางการค้นพบ

### Bonjour (LAN)

แอป iOS ค้นหา `_openclaw-gw._tcp` บน `local.` และเมื่อกำหนดค่าไว้ จะค้นหาโดเมน
การค้นพบ DNS-SD แบบไวด์แอเรียเดียวกันด้วย Gateway ใน LAN เดียวกันจะปรากฏโดยอัตโนมัติจาก `local.`;
การค้นพบข้ามเครือข่ายใช้โดเมนไวด์แอเรียที่กำหนดค่าไว้ได้โดยไม่ต้องเปลี่ยนชนิดบีคอน

### Tailnet (ข้ามเครือข่าย)

หาก mDNS ถูกบล็อก ให้ใช้โซน DNS-SD แบบยูนิคาสต์ (เลือกโดเมนหนึ่งรายการ; ตัวอย่าง:
`openclaw.internal.`) และ Tailscale split DNS
ดูตัวอย่าง CoreDNS ได้ที่ [Bonjour](/th/gateway/bonjour)

### โฮสต์/พอร์ตแบบกำหนดเอง

ในการตั้งค่า ให้เปิดใช้ **โฮสต์แบบกำหนดเอง** แล้วป้อนโฮสต์ + พอร์ตของ Gateway (ค่าเริ่มต้น `18789`)

## แคนวาส + A2UI

โหนด iOS เรนเดอร์แคนวาส WKWebView ใช้ `node.invoke` เพื่อควบคุม:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

หมายเหตุ:

- โฮสต์แคนวาสของ Gateway ให้บริการ `/__openclaw__/canvas/` และ `/__openclaw__/a2ui/`
- ให้บริการจากเซิร์ฟเวอร์ HTTP ของ Gateway (พอร์ตเดียวกับ `gateway.port`, ค่าเริ่มต้น `18789`)
- โหนด iOS นำทางไปยัง A2UI อัตโนมัติเมื่อเชื่อมต่อ หากมีการประกาศ URL โฮสต์แคนวาส
- กลับไปยังโครงตั้งต้นในตัวด้วย `canvas.navigate` และ `{"url":""}`

## ความสัมพันธ์กับ Computer Use

แอป iOS เป็นพื้นผิวโหนดบนมือถือ ไม่ใช่แบ็กเอนด์ Codex Computer Use Codex
Computer Use และ `cua-driver mcp` ควบคุมเดสก์ท็อป macOS ในเครื่องผ่านเครื่องมือ MCP;
แอป iOS เปิดเผยความสามารถของ iPhone ผ่านคำสั่งโหนด OpenClaw
เช่น `canvas.*`, `camera.*`, `screen.*`, `location.*` และ `talk.*`

เอเจนต์ยังสามารถใช้งานแอป iOS ผ่าน OpenClaw ได้โดยเรียกคำสั่งโหนด
แต่การเรียกเหล่านั้นจะผ่านโปรโตคอลโหนดของ Gateway และเป็นไปตามข้อจำกัดเบื้องหน้า/เบื้องหลังของ iOS
ใช้ [Codex Computer Use](/th/plugins/codex-computer-use)
สำหรับการควบคุมเดสก์ท็อปในเครื่อง และใช้หน้านี้สำหรับความสามารถโหนด iOS

### การประเมินผล / สแนปช็อตแคนวาส

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.eval --params '{"javaScript":"(() => { const {ctx} = window.__openclaw; ctx.clearRect(0,0,innerWidth,innerHeight); ctx.lineWidth=6; ctx.strokeStyle=\"#ff2d55\"; ctx.beginPath(); ctx.moveTo(40,40); ctx.lineTo(innerWidth-40, innerHeight-40); ctx.stroke(); return \"ok\"; })()"}'
```

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.snapshot --params '{"maxWidth":900,"format":"jpeg"}'
```

## การปลุกด้วยเสียง + โหมดพูดคุย

- การปลุกด้วยเสียงและโหมดพูดคุยมีให้ใช้งานในการตั้งค่า
- โหนด iOS ที่รองรับการพูดคุยประกาศความสามารถ `talk` และสามารถประกาศ
  `talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` และ `talk.ptt.once`;
  Gateway อนุญาตคำสั่งกดเพื่อพูดเหล่านั้นโดยค่าเริ่มต้นสำหรับโหนดที่รองรับ
  การพูดคุยและเชื่อถือได้
- iOS อาจระงับเสียงเบื้องหลัง; ให้ถือว่าฟีเจอร์เสียงเป็นแบบพยายามให้ดีที่สุดเมื่อแอปไม่ได้ทำงานอยู่

## ข้อผิดพลาดที่พบบ่อย

- `NODE_BACKGROUND_UNAVAILABLE`: นำแอป iOS มาไว้เบื้องหน้า (คำสั่งแคนวาส/กล้อง/หน้าจอต้องใช้สิ่งนี้)
- `A2UI_HOST_NOT_CONFIGURED`: Gateway ไม่ได้ประกาศ URL โฮสต์แคนวาส; ตรวจสอบ `canvasHost` ใน [การกำหนดค่า Gateway](/th/gateway/configuration)
- พรอมต์จับคู่ไม่ปรากฏ: เรียกใช้ `openclaw devices list` แล้วอนุมัติด้วยตนเอง
- เชื่อมต่อใหม่ล้มเหลวหลังติดตั้งใหม่: โทเค็นการจับคู่ใน Keychain ถูกล้างแล้ว; จับคู่โหนดใหม่
