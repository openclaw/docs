ความพร้อมใช้งาน: บิลด์แอป iPhone เผยแพร่ผ่านช่องทางของ Apple เมื่อเปิดใช้งานสำหรับรีลีส นอกจากนี้ยังสามารถเรียกใช้บิลด์สำหรับการพัฒนาในเครื่องจากซอร์สได้

## การทำงาน

- เชื่อมต่อกับ Gateway ผ่าน WebSocket (LAN หรือ tailnet)
- เปิดให้ใช้ความสามารถของ Node ได้แก่ Canvas, สแนปช็อตหน้าจอ, การบันทึกภาพจากกล้อง, ตำแหน่งที่ตั้ง, โหมดสนทนา, การปลุกด้วยเสียง และข้อมูลสรุปสุขภาพที่เลือกเปิดใช้
- รับคำสั่ง `node.invoke` และรายงานเหตุการณ์สถานะของ Node
- เรียกดูพื้นที่ทำงานของเอเจนต์ที่เลือกแบบอ่านอย่างเดียวจากหน้าจอเอเจนต์ (ไฟล์): ไล่ดูไดเรกทอรีลงไปทีละระดับ แสดงตัวอย่างข้อความพร้อมไฮไลต์ไวยากรณ์ แสดงตัวอย่างรูปภาพ และส่งออกผ่านแผ่นแบ่งปัน ไม่มีการดำเนินการเขียน โดย Gateway จำกัดขนาดของตัวอย่าง
- เก็บแคชออฟไลน์ขนาดเล็กแบบอ่านอย่างเดียวของเซสชันแชตและทรานสคริปต์ล่าสุดสำหรับ Gateway ที่จับคู่แต่ละรายการ: เมื่อเปิดแบบ cold open จะแสดงทรานสคริปต์ล่าสุดที่ทราบทันทีและรีเฟรชเมื่อ Gateway ตอบกลับ แชตล่าสุดยังเรียกดูได้ขณะตัดการเชื่อมต่อ และการรีเซ็ต/ลืมจะล้างแคชภายในเครื่องที่ได้รับการป้องกัน
- จัดคิวข้อความที่ส่งขณะตัดการเชื่อมต่อไว้ในกล่องขาออกถาวรแยกตาม Gateway (สูงสุด 50 รายการ): บับเบิลที่เข้าคิวจะแสดงในทรานสคริปต์ ส่งออกตามลำดับเมื่อเชื่อมต่อใหม่พร้อมการลองซ้ำแบบ idempotent คงอยู่จนกว่าประวัติมาตรฐานจะยืนยันการส่ง ลองซ้ำโดยเพิ่มช่วงหน่วงก่อนแสดงการดำเนินการลองซ้ำ/ลบ และหมดอายุแทนการส่งหลังออฟไลน์ 48 ชั่วโมง การรีเซ็ต/ลืมจะล้างคิวพร้อมกับแคช
- แชตเป็นหน้าจอเดียวสำหรับข้อความและเสียง การดำเนินการในแชตสามารถเปิดหน้าจอเซสชันแบบเต็มได้โดยไม่ออกจากแชต และสามารถแสดงหรือซ่อนการให้เหตุผลของผู้ช่วยและกิจกรรมของเครื่องมือได้ แตะไมโครโฟนเพื่อป้อนข้อความร่างด้วยเสียง เปิดเมนูเพื่อบันทึกข้อความเสียง หรือใช้ตัวควบคุมโหมดสนทนาแบบอินไลน์สำหรับเสียงแบบเรียลไทม์ ตัวควบคุมโหมดสนทนาจะแสดงภาพเคลื่อนไหวตามระดับไมโครโฟนสดหรือระดับการเล่นเสียงขณะฟังหรือพูด
- อ่านออกเสียงข้อความของผู้ช่วยตามต้องการ: กดข้อความในแชตค้างไว้แล้วเลือก **Listen** แอปจะเล่นคลิป `tts.speak` จาก Gateway ที่รองรับด้วยผู้ให้บริการ TTS ที่กำหนดค่าไว้ และใช้เสียงพูดบนอุปกรณ์เป็นทางเลือกเมื่อเสียงจาก Gateway ไม่พร้อมใช้งานหรือเล่นไม่ได้ การเล่นจะหยุดเมื่อสลับเซสชันหรือเมื่อแอปเข้าสู่เบื้องหลัง

## ข้อกำหนด

- Gateway ทำงานอยู่บนอุปกรณ์อีกเครื่องหนึ่ง (macOS, Linux หรือ Windows ผ่าน WSL2)
- เส้นทางเครือข่าย:
  - LAN เดียวกันผ่าน Bonjour **หรือ**
  - Tailnet ผ่าน unicast DNS-SD (โดเมนตัวอย่าง: `openclaw.internal.`) **หรือ**
  - โฮสต์/พอร์ตที่ระบุด้วยตนเอง (ทางเลือกสำรอง)

## เริ่มต้นอย่างรวดเร็ว (จับคู่ + เชื่อมต่อ)

เมื่อเปิดครั้งแรก แอปจะแนะนำคำอธิบายสั้นๆ เกี่ยวกับการจับคู่และหน้า
สิทธิ์ (การแจ้งเตือน กล้อง ไมโครโฟน รูปภาพ รายชื่อ
ปฏิทิน การเตือนความจำ ตำแหน่งที่ตั้ง) สิทธิ์ทุกอย่างเป็นทางเลือกและเปลี่ยนแปลงได้
ภายหลังใน **Settings** -> **Permissions** หรือในแอป Settings ของ iOS

1. เริ่ม Gateway ที่ผ่านการตรวจสอบสิทธิ์ด้วยเส้นทางที่โทรศัพท์เข้าถึงได้ Tailscale
   Serve เป็นเส้นทางระยะไกลที่แนะนำ:

```bash
openclaw gateway --port 18789 --tailscale serve
```

สำหรับการตั้งค่าบน LAN เดียวกันที่เชื่อถือได้ ให้ใช้ `gateway.bind: "lan"` ที่ผ่านการตรวจสอบสิทธิ์
แทน การ bind กับ loopback ตามค่าเริ่มต้นไม่สามารถเข้าถึงได้จากโทรศัพท์ หากยังไม่ได้
กำหนดค่า Gateway ให้เรียกใช้ `openclaw onboard` ก่อน เพื่อให้การสร้าง
รหัสตั้งค่ามีเส้นทางการตรวจสอบสิทธิ์ด้วยโทเค็นหรือรหัสผ่าน

2. เปิด [Control UI](/th/web/control-ui) เลือก **Nodes** แล้วคลิก
   **Pair mobile device** บนหน้า **Devices** แนะนำให้ใช้สิทธิ์เข้าถึงแบบเต็ม
   และระบบจะเลือกไว้เป็นค่าเริ่มต้น เลือก Limited access เฉพาะเมื่อต้องการละเว้น
   ตัวควบคุมการดูแล Gateway จากนั้นคลิก **Create setup code**

3. ในแอป iOS ให้เปิด **Settings** -> **Gateway** สแกนคิวอาร์โค้ด (หรือวาง
   รหัสตั้งค่า) แล้วเชื่อมต่อ

   หากรหัสตั้งค่ามีทั้งเส้นทาง LAN และ Tailscale Serve แอปจะ
   ตรวจสอบตามลำดับและบันทึกปลายทางแรกที่เข้าถึงได้

4. แอปอย่างเป็นทางการจะเชื่อมต่อโดยอัตโนมัติ หาก **Pending approval** แสดง
   คำขอ ให้ตรวจสอบบทบาทและขอบเขตของคำขอก่อนอนุมัติ

   **Settings → Gateway** จะแสดงว่าการเชื่อมต่อของผู้ปฏิบัติการที่บันทึกไว้มี
   สิทธิ์เข้าถึงแบบ **Full** หรือ **Limited** การตั้งค่า `ws://` บน LAN แบบข้อความธรรมดาจะถูก
   จำกัดโดยอัตโนมัติเพื่อความปลอดภัยของ bearer token หากถูกจำกัด ให้กำหนดค่า `wss://` หรือ
   Tailscale Serve สแกนรหัสสิทธิ์เข้าถึงแบบเต็มใหม่จาก Control UI หรือ `openclaw qr`
   แล้วเชื่อมต่อใหม่เพื่อเปิดใช้การตั้งค่าและการอัปเกรด

ปุ่ม Control UI ต้องใช้เซสชันที่จับคู่แล้วกับ `operator.admin`
สำหรับทางเลือกสำรองผ่านเทอร์มินัล ให้เลือก Gateway ที่ค้นพบในแอป iOS (หรือเปิดใช้
Manual Host แล้วป้อนโฮสต์/พอร์ต) จากนั้นอนุมัติคำขอบนโฮสต์ Gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

หากแอปลองจับคู่ใหม่ด้วยรายละเอียดการตรวจสอบสิทธิ์ที่เปลี่ยนไป (บทบาท/ขอบเขต/กุญแจสาธารณะ) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่และสร้าง `requestId` ใหม่ เรียกใช้ `openclaw devices list` อีกครั้งก่อนอนุมัติ

ทางเลือก: หาก Node ของ iOS เชื่อมต่อจากซับเน็ตที่ควบคุมอย่างเข้มงวดเสมอ สามารถเลือกเปิดใช้การอนุมัติ Node ครั้งแรกโดยอัตโนมัติด้วย CIDR หรือ IP ที่ระบุอย่างชัดเจน:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

คุณลักษณะนี้ปิดใช้งานเป็นค่าเริ่มต้น และใช้เฉพาะกับการจับคู่ `role: node` ใหม่ที่ไม่มีขอบเขตที่ร้องขอ การจับคู่ของผู้ปฏิบัติการ/เบราว์เซอร์ รวมถึงการเปลี่ยนแปลงบทบาท ขอบเขต เมทาดาทา หรือกุญแจสาธารณะ ยังคงต้องได้รับการอนุมัติด้วยตนเอง

5. ตรวจสอบการเชื่อมต่อ:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## ข้อมูลสรุปสุขภาพ

Node ของ iOS สามารถส่งคืนข้อมูลรวม HealthKit แบบอ่านอย่างเดียวที่เลือกเปิดใช้สำหรับ
วันตามปฏิทินปัจจุบัน การยินยอมบนอุปกรณ์ iOS และการอนุญาตคำสั่ง Gateway อย่างชัดแจ้งเป็น
ด่านที่แยกจากกัน ดู [ข้อมูลสรุป HealthKit](/th/platforms/ios-healthkit) สำหรับ
การตั้งค่า การเรียกใช้ ฟิลด์เพย์โหลด ลักษณะการทำงานด้านความเป็นส่วนตัว และการแก้ไขปัญหา

ตามค่าเริ่มต้น แอปคู่หูบน Apple Watch จะยังคงใช้รีเลย์ iPhone ที่มีอยู่ และ
ไม่ต้องจับคู่กับ Gateway แยกต่างหาก จับคู่ Watch กับ iPhone ใน
แอป Watch ของ Apple ติดตั้ง OpenClaw จาก **Watch app -> My Watch -> Available
Apps** จากนั้นเปิด OpenClaw หนึ่งครั้งบนอุปกรณ์ทั้งสอง

## ตรวจสอบการอนุมัติคำสั่ง

การเชื่อมต่อของผู้ปฏิบัติการที่มี `operator.admin` หรือการเชื่อมต่อ
`operator.approvals` ที่จับคู่แล้วและ Gateway ระบุเป็นเป้าหมายอย่างชัดเจน สามารถตรวจสอบ
คำขอ exec ที่รอดำเนินการบน iPhone ได้ การ์ดอนุมัติจะแสดงตัวอย่างคำสั่งที่ผ่านการปรับให้ปลอดภัยจาก Gateway
คำเตือน บริบทโฮสต์ เวลาหมดอายุ และเฉพาะ
ตัวเลือกการตัดสินใจที่คำขอนั้นเสนอ Apple Watch ที่จับคู่จะได้รับ
พรอมต์ที่ปลอดภัยสำหรับผู้ตรวจสอบแบบเดียวกันผ่านรีเลย์ iPhone ที่มีอยู่ และมีชุดตัวเลือกแบบย่อ
สำหรับอนุญาตครั้งเดียว/ปฏิเสธ โหมด Gateway โดยตรงของ Watch ไม่รองรับ
พรอมต์การอนุมัติ

สถานะการอนุมัติใช้ร่วมกับ Control UI และหน้าจอแชตที่รองรับ
คำตอบแรกที่ยืนยันแล้วมีผล iPhone และ Watch จะดึงข้อมูลระเบียน
สถานะปลายทางมาตรฐานของ Gateway หลังจากหน้าจออื่นแก้ไขคำขอแล้ว หลังจากได้รับการแจ้งเตือน
ว่าแก้ไขจากระยะไกลแล้ว และเมื่อใดก็ตามที่อาจสูญเสียการตอบรับการแก้ไข
การดำเนินการจะยังไม่พร้อมใช้งานจนกว่าการอ่านกลับนั้นจะยืนยันว่า
คำขอยังคงรอดำเนินการอยู่หรือไม่

ความเป็นเจ้าของการอนุมัติผูกกับ Gateway ที่เลือก การสลับ Gateway ไม่สามารถ
นำพรอมต์เก่าไปใช้กับการเชื่อมต่อใหม่ได้ Gateway ที่เก่ากว่า
เมธอดการอนุมัติแบบรวมจะใช้เมธอดเฉพาะสำหรับ exec ที่เผยแพร่แล้วเป็นทางเลือกสำรอง
สถานะปลายทางที่เก็บรักษาไว้และผลลัพธ์ข้ามหน้าจอที่มีรายละเอียดมากขึ้นต้องใช้
Gateway ที่อัปเดตแล้ว

## ตอบคำถามของเอเจนต์

แชตจะแสดงคำถามจาก Gateway ที่รอดำเนินการเป็นการ์ดแบบเนทีฟสำหรับการเชื่อมต่อของผู้ปฏิบัติการ
ที่มี `operator.questions` (หรือ `operator.admin`) การ์ดรองรับตัวเลือกแบบเลือกเดียวและ
หลายตัวเลือก คำอธิบายตัวเลือก คำตอบ **Other** แบบข้อความอิสระ และ
การนับถอยหลังจนหมดอายุ การเชื่อมต่อใหม่จะโหลดคำถามที่รอดำเนินการจาก Gateway อีกครั้ง การ์ด
จะล็อกเมื่ออุปกรณ์นี้ตอบ เมื่อหน้าจออื่นตอบก่อน หรือเมื่อ
คำถามหมดอายุหรือถูกยกเลิก

## Node โดยตรงบน Apple Watch ที่เป็นทางเลือก

โหมดโดยตรงทำให้นาฬิกามีข้อมูลประจำตัว Node ที่ลงนามและการเชื่อมต่อ Gateway เป็นของตนเอง
คำสั่ง Node ที่รองรับจะยังคงทำงานผ่าน Wi-Fi หรือเซลลูลาร์ของนาฬิกาขณะที่
OpenClaw ทำงานอยู่ แม้ iPhone ที่จับคู่จะไม่พร้อมใช้งาน

ข้อกำหนด:

- iPhone เชื่อมต่อกับ Gateway ด้วยขอบเขต `operator.admin`
- รหัสตั้งค่าประกาศปลายทาง Gateway แบบ `wss://` พร้อมใบรับรองที่ watchOS เชื่อถือ
  นาฬิกาจะสำรวจ origin `https://` ที่เกี่ยวข้อง ไม่รองรับ HTTP แบบข้อความธรรมดาและ
  ความเชื่อถือแบบลงนามเองหรือใช้เฉพาะลายนิ้วมือ ดู [การจับคู่ที่ Gateway เป็นเจ้าของ
  ](/th/gateway/pairing) สำหรับการกำหนดค่าปลายทาง เส้นทาง loopback, เฉพาะ iPhone
  และเฉพาะ tailnet ไม่สามารถเข้าถึงจากนาฬิกาได้โดยอิสระ
- การใช้งานเซลลูลาร์ต้องใช้ Apple Watch ที่รองรับเซลลูลาร์และมีบริการที่ใช้งานอยู่
- OpenClaw ทำงานอยู่บนนาฬิกา Apple ไม่อนุญาตให้แอป watchOS ทั่วไป
  รักษาการเชื่อมต่อ WebSocket/TCP แบบทั่วไปไว้ ดังนั้น Node โดยตรงจึงใช้การสำรวจ HTTPS
  แบบสั้นและเชื่อมต่อใหม่เมื่อแอปกลับมาทำงานเบื้องหน้า ดู
  [คำแนะนำด้านเครือข่ายระดับต่ำของ watchOS](https://developer.apple.com/documentation/technotes/tn3135-low-level-networking-on-watchOS) จาก Apple

การตั้งค่า:

1. บน iPhone ให้เปิด **Settings -> Apple Watch**
2. แตะ **Enable Direct Gateway Connection**
3. เปิด OpenClaw บนนาฬิกาก่อนรหัสตั้งค่าอายุสั้นจะหมดอายุ
4. ตรวจสอบแถว Apple Watch แยกต่างหากด้วย `openclaw nodes status`

รหัสตั้งค่ามีข้อมูลรับรองบูตสแตรปสำหรับ Node เท่านั้นและมีอายุสั้น ให้ปฏิบัติต่อข้อมูลนี้
เช่นเดียวกับรหัสผ่านจนกว่าจะหมดอายุ รหัสดังกล่าวไม่มีรหัสผ่าน
หรือโทเค็น Gateway ที่บันทึกไว้ของ iPhone หลังจากจับคู่ นาฬิกาจะจัดเก็บโทเค็นอุปกรณ์ของตนเองและ
ลบข้อมูลรับรองบูตสแตรป โหมดโดยตรงครอบคลุมเฉพาะคำสั่งด้านล่าง
แชต โหมดสนทนา การอนุมัติ และขั้นตอนการแจ้งเตือน `watch.*` ที่มีอยู่ยังคงเป็น
คุณลักษณะผ่านรีเลย์ iPhone และยังต้องใช้ iPhone ที่จับคู่

คำสั่ง Node โดยตรงของ watchOS:

| พื้นที่ใช้งาน       | คำสั่ง                       | หมายเหตุ                                                   |
| ------------- | ------------------------------ | ------------------------------------------------------- |
| อุปกรณ์        | `device.info`, `device.status` | ข้อมูลประจำตัวของ Watch แบตเตอรี่ อุณหภูมิ พื้นที่จัดเก็บ และเครือข่าย |
| การแจ้งเตือน | `system.notify`                | ขณะที่แอปทำงานอยู่ ต้องได้รับสิทธิ์จาก Watch     |

watchOS ไม่เปิดให้แอปของบุคคลที่สามใช้ WebKit ดังนั้น Node โดยตรงบน Watch
จึงไม่ประกาศคำสั่ง Canvas

## พุชที่รองรับด้วยรีเลย์สำหรับบิลด์อย่างเป็นทางการ

บิลด์ iOS อย่างเป็นทางการที่เผยแพร่จะใช้รีเลย์พุชภายนอกแทนการเผยแพร่โทเค็น APNs ดิบไปยัง Gateway บิลด์ App Store อย่างเป็นทางการจากช่องทางรีลีสสาธารณะใช้รีเลย์ที่โฮสต์ไว้ที่ `https://ios-push-relay.openclaw.ai` URL ฐานนี้ฝังไว้ตายตัวสำหรับการเผยแพร่ผ่าน App Store และไม่อ่านค่าที่ใช้แทน

การติดตั้งใช้งานรีเลย์แบบกำหนดเองต้องใช้เส้นทางบิลด์/การติดตั้งใช้งาน iOS ที่แยกออกมาโดยเจตนา ซึ่ง URL ของรีเลย์ตรงกับ URL รีเลย์ของ Gateway ช่องทางรีลีส App Store ไม่ยอมรับ URL รีเลย์แบบกำหนดเอง หากใช้บิลด์รีเลย์แบบกำหนดเอง ให้ตั้งค่า URL รีเลย์ของ Gateway ที่ตรงกัน:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

ขั้นตอนการทำงาน:

- แอป iOS ลงทะเบียนกับรีเลย์โดยใช้ App Attest และ JWS ของธุรกรรมแอป StoreKit
- รีเลย์ส่งคืนแฮนเดิลรีเลย์แบบทึบพร้อมสิทธิ์อนุญาตการส่งที่จำกัดขอบเขตเฉพาะการลงทะเบียน
- แอป iOS ดึงข้อมูลอัตลักษณ์ของ Gateway ที่จับคู่ไว้ (`gateway.identity.get`) และรวมข้อมูลดังกล่าวในการลงทะเบียนรีเลย์ เพื่อมอบสิทธิ์การลงทะเบียนที่รองรับด้วยรีเลย์ให้แก่ Gateway นั้นโดยเฉพาะ
- แอปส่งต่อการลงทะเบียนที่รองรับด้วยรีเลย์ดังกล่าวไปยัง Gateway ที่จับคู่ไว้ด้วย `push.apns.register`
- Gateway ใช้แฮนเดิลรีเลย์ที่จัดเก็บไว้นั้นสำหรับ `push.test` การปลุกในเบื้องหลัง และสัญญาณกระตุ้นการปลุก
- หากภายหลังแอปเชื่อมต่อกับ Gateway อื่นหรือบิลด์ที่ใช้ URL ฐานของรีเลย์ต่างกัน แอปจะรีเฟรชการลงทะเบียนรีเลย์แทนการใช้การผูกเดิมซ้ำ

สิ่งที่ Gateway **ไม่** จำเป็นต้องมีสำหรับเส้นทางนี้: ไม่ต้องมีโทเค็นรีเลย์ที่ใช้ทั่วทั้งการปรับใช้ และไม่ต้องมีคีย์ APNs โดยตรงสำหรับการส่งอย่างเป็นทางการจาก App Store ที่รองรับด้วยรีเลย์

ขั้นตอนที่คาดหวังสำหรับผู้ปฏิบัติงาน:

1. ติดตั้งแอป iOS อย่างเป็นทางการ
2. ไม่บังคับ: ตั้งค่า `gateway.push.apns.relay.baseUrl` บน Gateway เฉพาะเมื่อใช้บิลด์รีเลย์แบบกำหนดเองที่แยกออกมาต่างหากโดยตั้งใจ
3. จับคู่แอปกับ Gateway และรอให้เชื่อมต่อเสร็จสมบูรณ์
4. แอปเผยแพร่ `push.apns.register` เมื่อได้รับโทเค็น APNs เซสชันผู้ปฏิบัติงานเชื่อมต่อแล้ว และการลงทะเบียนรีเลย์สำเร็จ
5. หลังจากนั้น `push.test` การปลุกเมื่อเชื่อมต่อใหม่ และสัญญาณกระตุ้นการปลุกจะสามารถใช้การลงทะเบียนที่รองรับด้วยรีเลย์ซึ่งจัดเก็บไว้ได้

## บีคอนแสดงการทำงานในเบื้องหลัง

เมื่อ iOS ปลุกแอปด้วยพุชแบบเงียบ การรีเฟรชในเบื้องหลัง หรือเหตุการณ์ตำแหน่งที่มีการเปลี่ยนแปลงอย่างมีนัยสำคัญ แอปจะพยายามเชื่อมต่อ Node ใหม่เป็นช่วงสั้น ๆ แล้วเรียก `node.event` พร้อม `event: "node.presence.alive"` Gateway จะบันทึกข้อมูลนี้เป็น `lastSeenAtMs`/`lastSeenReason` ในข้อมูลเมตาของ Node/อุปกรณ์ที่จับคู่ไว้ หลังจากทราบอัตลักษณ์ของอุปกรณ์ Node ที่ผ่านการตรวจสอบสิทธิ์แล้วเท่านั้น

แอปจะถือว่าการปลุกในเบื้องหลังได้รับการบันทึกสำเร็จเฉพาะเมื่อการตอบกลับจาก Gateway มี `handled: true` Gateway รุ่นเก่าอาจตอบรับ `node.event` ด้วย `{ "ok": true }` การตอบกลับดังกล่าวเข้ากันได้ แต่ไม่นับเป็นการอัปเดตเวลาที่พบเห็นล่าสุดแบบคงทน

หมายเหตุเกี่ยวกับความเข้ากันได้:

- `OPENCLAW_APNS_RELAY_BASE_URL` ยังคงใช้เป็นการแทนที่ชั่วคราวผ่านตัวแปรสภาพแวดล้อมสำหรับ Gateway ได้ (`gateway.push.apns.relay.baseUrl` คือเส้นทางที่ให้ความสำคัญกับการกำหนดค่าก่อน)
- โหมดพุชของบิลด์รุ่น App Store กำหนดโฮสต์รีเลย์ที่ให้บริการไว้ตายตัวและจะไม่อ่านค่าการแทนที่ URL ของรีเลย์ — ตัวแปรสภาพแวดล้อมขณะบิลด์ `OPENCLAW_PUSH_RELAY_BASE_URL` มีผลเฉพาะกับโหมดบิลด์ iOS แบบโลคัล/แซนด์บ็อกซ์เท่านั้น

## ขั้นตอนการตรวจสอบสิทธิ์และความเชื่อถือ

รีเลย์มีไว้เพื่อบังคับใช้ข้อจำกัดสองประการที่การใช้ APNs โดยตรงบน Gateway ไม่สามารถทำได้สำหรับบิลด์ iOS อย่างเป็นทางการ:

- เฉพาะบิลด์ OpenClaw สำหรับ iOS ของแท้ที่เผยแพร่ผ่าน Apple เท่านั้นที่สามารถใช้รีเลย์ที่ให้บริการได้
- Gateway สามารถส่งพุชที่รองรับด้วยรีเลย์ได้เฉพาะไปยังอุปกรณ์ iOS ที่จับคู่กับ Gateway นั้นโดยเฉพาะ

ทีละช่วงการส่งต่อ:

1. `iOS app -> gateway`: แอปจับคู่กับ Gateway ผ่านขั้นตอนการตรวจสอบสิทธิ์ตามปกติของ Gateway ซึ่งทำให้ได้รับทั้งเซสชัน Node ที่ผ่านการตรวจสอบสิทธิ์และเซสชันผู้ปฏิบัติงานที่ผ่านการตรวจสอบสิทธิ์ เซสชันผู้ปฏิบัติงานเรียก `gateway.identity.get`
2. `iOS app -> relay`: แอปเรียกปลายทางการลงทะเบียนของรีเลย์ผ่าน HTTPS โดยใช้หลักฐาน App Attest พร้อม JWS ของธุรกรรมแอป StoreKit รีเลย์ตรวจสอบ bundle ID หลักฐาน App Attest และหลักฐานการเผยแพร่ของ Apple รวมทั้งกำหนดให้ใช้เส้นทางการเผยแพร่อย่างเป็นทางการ/สำหรับการใช้งานจริง — นี่คือสิ่งที่ป้องกันไม่ให้บิลด์ Xcode/สำหรับการพัฒนาในเครื่องใช้รีเลย์ที่ให้บริการ เนื่องจากบิลด์ในเครื่องไม่สามารถแสดงหลักฐานการเผยแพร่อย่างเป็นทางการของ Apple ได้
3. `gateway identity delegation`: ก่อนลงทะเบียนรีเลย์ แอปจะดึงข้อมูลอัตลักษณ์ของ Gateway ที่จับคู่ไว้จาก `gateway.identity.get` และรวมไว้ในเพย์โหลดการลงทะเบียนรีเลย์ รีเลย์ส่งคืนแฮนเดิลรีเลย์และสิทธิ์อนุญาตการส่งที่จำกัดขอบเขตเฉพาะการลงทะเบียน ซึ่งมอบสิทธิ์ให้แก่อัตลักษณ์ของ Gateway นั้น
4. `gateway -> relay`: Gateway จัดเก็บแฮนเดิลรีเลย์และสิทธิ์อนุญาตการส่งจาก `push.apns.register` เมื่อเกิด `push.test` การปลุกเมื่อเชื่อมต่อใหม่ และสัญญาณกระตุ้นการปลุก Gateway จะลงนามคำขอส่งด้วยอัตลักษณ์อุปกรณ์ของตนเอง รีเลย์จะตรวจสอบทั้งสิทธิ์อนุญาตการส่งที่จัดเก็บไว้และลายเซ็นของ Gateway เทียบกับอัตลักษณ์ Gateway ที่ได้รับมอบสิทธิ์จากการลงทะเบียน Gateway อื่นไม่สามารถนำการลงทะเบียนที่จัดเก็บไว้นั้นไปใช้ซ้ำได้ แม้จะได้แฮนเดิลมาด้วยวิธีใดก็ตาม
5. `relay -> APNs`: รีเลย์เป็นผู้ถือครองข้อมูลประจำตัว APNs สำหรับการใช้งานจริงและโทเค็น APNs ดิบของบิลด์อย่างเป็นทางการ Gateway จะไม่จัดเก็บโทเค็น APNs ดิบสำหรับบิลด์อย่างเป็นทางการที่รองรับด้วยรีเลย์ รีเลย์จะส่งพุชสุดท้ายไปยัง APNs ในนามของ Gateway ที่จับคู่ไว้

เหตุผลที่สร้างการออกแบบนี้: เพื่อเก็บข้อมูลประจำตัว APNs สำหรับการใช้งานจริงไว้นอก Gateway ของผู้ใช้ หลีกเลี่ยงการจัดเก็บโทเค็น APNs ดิบของบิลด์อย่างเป็นทางการบน Gateway อนุญาตให้เฉพาะบิลด์ OpenClaw สำหรับ iOS อย่างเป็นทางการใช้รีเลย์ที่ให้บริการ และป้องกันไม่ให้ Gateway หนึ่งส่งพุชปลุกไปยังอุปกรณ์ iOS ที่เป็นของ Gateway อื่น

บิลด์ในเครื่อง/แบบทำเองยังคงใช้ APNs โดยตรง หากกำลังทดสอบบิลด์เหล่านั้นโดยไม่ใช้รีเลย์ Gateway ยังคงต้องมีข้อมูลประจำตัว APNs โดยตรง:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

ตัวแปรเหล่านี้เป็นตัวแปรสภาพแวดล้อมรันไทม์ของโฮสต์ Gateway ไม่ใช่การตั้งค่า Fastlane `apps/ios/fastlane/.env` จัดเก็บเฉพาะข้อมูลการตรวจสอบสิทธิ์ App Store Connect เช่น `APP_STORE_CONNECT_KEY_ID` และ `APP_STORE_CONNECT_ISSUER_ID` เท่านั้น โดยไม่ได้กำหนดค่าการนำส่ง APNs โดยตรงสำหรับบิลด์ iOS ในเครื่อง

ตำแหน่งจัดเก็บที่แนะนำบนโฮสต์ Gateway ซึ่งสอดคล้องกับข้อมูลประจำตัวของผู้ให้บริการรายอื่นภายใต้ `~/.openclaw/credentials/`:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

อย่าคอมมิตไฟล์ `.p8` หรือวางไว้ภายใต้เช็กเอาต์ของรีโพ

## เส้นทางการค้นหา

### Bonjour (LAN)

แอป iOS เรียกดู `_openclaw-gw._tcp` บน `local.` และเมื่อกำหนดค่าไว้ จะเรียกดูโดเมนการค้นหา DNS-SD แบบบริเวณกว้างเดียวกันด้วย Gateway ที่อยู่ใน LAN เดียวกันจะปรากฏโดยอัตโนมัติจาก `local.` ส่วนการค้นหาข้ามเครือข่ายสามารถใช้โดเมนบริเวณกว้างที่กำหนดค่าไว้โดยไม่ต้องเปลี่ยนชนิดบีคอน

### Tailnet (ข้ามเครือข่าย)

หาก mDNS ถูกบล็อก ให้ใช้โซน DNS-SD แบบยูนิแคสต์ (เลือกโดเมน เช่น `openclaw.internal.`) และ DNS แบบแยกของ Tailscale ดูตัวอย่าง CoreDNS ได้ที่ [Bonjour](/th/gateway/bonjour)

### โฮสต์/พอร์ตแบบกำหนดเอง

ใน Settings ให้เปิดใช้งาน **Manual Host** แล้วป้อนโฮสต์และพอร์ตของ Gateway (ค่าเริ่มต้น `18789`)

## Gateway หลายรายการ

แอปเก็บรีจิสทรีของ Gateway ทุกแห่งที่เคยจับคู่ไว้ จึงสามารถสลับระหว่างกันได้โดยไม่ต้องจับคู่ใหม่:

- **Settings -> Gateway** แสดงรายการ **Paired Gateways** พร้อมทำเครื่องหมาย Gateway ที่ใช้งานอยู่ แตะรายการเพื่อสลับ แอปจะยุติเซสชันปัจจุบันและเชื่อมต่อใหม่กับ Gateway ที่เลือก เมนูสลับด่วนจะปรากฏถัดจากแถวการเชื่อมต่อเมื่อมี Gateway ที่จับคู่ไว้มากกว่าหนึ่งรายการ
- ข้อมูลประจำตัว การตัดสินใจเกี่ยวกับความเชื่อถือ TLS การตั้งค่าเฉพาะแต่ละ Gateway และประวัติแชตที่แคชไว้จะถูกจัดเก็บแยกตาม Gateway การสลับจะไม่ผสมสถานะระหว่าง Gateway และการลงทะเบียนพุชจะติดตาม Gateway ที่ใช้งานอยู่
- ปัด Gateway ที่จับคู่ไว้ (หรือใช้เมนูบริบทของรายการนั้น) เพื่อเลือก **Forget** ซึ่งจะลบข้อมูลประจำตัว โทเค็นอุปกรณ์ พิน TLS และแชตที่แคชไว้
- Gateway ที่ค้นพบต้องมองเห็นได้บนเครือข่ายจึงจะสลับไปใช้งานได้ ส่วน Gateway แบบกำหนดเองจะเชื่อมต่อใหม่โดยใช้โฮสต์และพอร์ตที่บันทึกไว้

## Canvas + A2UI

Node iOS แสดงผล Canvas ด้วย WKWebView ใช้ `node.invoke` เพื่อควบคุม:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

หมายเหตุ:

- โฮสต์ Canvas ของ Gateway ให้บริการ `/__openclaw__/canvas/` และ `/__openclaw__/a2ui/` จากเซิร์ฟเวอร์ HTTP ของ Gateway (พอร์ตเดียวกับ `gateway.port` ค่าเริ่มต้น `18789`)
- Node iOS ใช้โครงร่างในตัวเป็นมุมมองเริ่มต้นเมื่อเชื่อมต่อแล้ว `canvas.a2ui.push` และ `canvas.a2ui.reset` ใช้หน้า A2UI ที่รวมมากับแอปและเป็นของแอป
- หน้า A2UI ของ Gateway ระยะไกลใช้แสดงผลอย่างเดียวบน iOS ส่วนการทำงานของปุ่ม A2UI แบบเนทีฟจะได้รับการยอมรับเฉพาะจากหน้าที่รวมมากับแอปและเป็นของแอปเท่านั้น
- กลับไปยังโครงร่างในตัวด้วย `canvas.navigate` และ `{"url":""}`

## ความสัมพันธ์กับ Computer Use

แอป iOS เป็นพื้นผิว Node บนอุปกรณ์เคลื่อนที่ ไม่ใช่แบ็กเอนด์ Codex Computer Use โดย Codex Computer Use และ `cua-driver mcp` ควบคุมเดสก์ท็อป macOS ในเครื่องผ่านเครื่องมือ MCP ส่วนแอป iOS เปิดเผยความสามารถของ iPhone ผ่านคำสั่ง Node ของ OpenClaw เช่น `canvas.*`, `camera.*`, `screen.*`, `location.*` และ `talk.*`

เอเจนต์ยังคงควบคุมแอป iOS ผ่าน OpenClaw ได้โดยเรียกคำสั่ง Node แต่การเรียกเหล่านั้นจะผ่านโปรโตคอล Node ของ Gateway และอยู่ภายใต้ข้อจำกัดการทำงานเบื้องหน้า/เบื้องหลังของ iOS ใช้ [Codex Computer Use](/th/plugins/codex-computer-use) สำหรับการควบคุมเดสก์ท็อปในเครื่อง และใช้หน้านี้สำหรับความสามารถของ Node iOS

### การประเมินผล / สแนปช็อต Canvas

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.eval --params '{"javaScript":"(() => { const {ctx} = window.__openclaw; ctx.clearRect(0,0,innerWidth,innerHeight); ctx.lineWidth=6; ctx.strokeStyle=\"#ff2d55\"; ctx.beginPath(); ctx.moveTo(40,40); ctx.lineTo(innerWidth-40, innerHeight-40); ctx.stroke(); return \"ok\"; })()"}'
```

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.snapshot --params '{"maxWidth":900,"format":"jpeg"}'
```

## การปลุกด้วยเสียง + โหมดสนทนา

- การปลุกด้วยเสียงและโหมดสนทนามีให้ใช้งานใน Settings
- Talk แบบเรียลไทม์ของ OpenAI ใช้ WebRTC ที่ไคลเอนต์เป็นผู้ถือครองเมื่อ `talk.realtime.transport` เป็น `webrtc` ส่วนการกำหนดค่า `gateway-relay` อย่างชัดเจนยังคงเป็นของ Gateway ดู [โหมด Talk](/th/nodes/talk)
- Node iOS ที่รองรับ Talk จะประกาศความสามารถ `talk` และสามารถประกาศ `talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` และ `talk.ptt.once` ได้ โดยค่าเริ่มต้น Gateway จะอนุญาตคำสั่งกดเพื่อพูดเหล่านั้นสำหรับ Node ที่เชื่อถือได้และรองรับ Talk
- iOS อาจระงับเสียงในเบื้องหลัง ให้ถือว่าฟีเจอร์เสียงทำงานแบบพยายามให้ดีที่สุดเมื่อแอปไม่ได้ทำงานอยู่

## ข้อผิดพลาดที่พบบ่อย

- `NODE_BACKGROUND_UNAVAILABLE`: นำแอป iOS มาไว้เบื้องหน้า (คำสั่ง Canvas/กล้อง/หน้าจอจำเป็นต้องใช้แอปในเบื้องหน้า)
- `A2UI_HOST_UNAVAILABLE`: ไม่สามารถเข้าถึงหน้า A2UI ที่รวมมาใน WebView ของแอปได้ ให้เปิดแอปไว้เบื้องหน้าที่แท็บ Screen แล้วลองอีกครั้ง
- ข้อความแจ้งให้จับคู่ไม่ปรากฏขึ้น: เรียกใช้ `openclaw devices list` และอนุมัติด้วยตนเอง
- Watch ไม่แสดงสถานะของ iPhone: ยืนยันว่า iPhone รายงาน `watchPaired: true`
  และ `watchAppInstalled: true` ใน `watch.status` หากการจับคู่เป็น false ให้จับคู่
  Watch ในแอป Watch ของ Apple หากการติดตั้งเป็น false ให้ติดตั้งแอปคู่หู
  จาก **My Watch -> Available Apps** หลังจากเปลี่ยนแปลงอย่างใดอย่างหนึ่งแล้ว ให้เปิด OpenClaw บน
  Watch หนึ่งครั้ง การเข้าถึงได้ทันทีจะยังคงต้องให้ทั้งสองแอปทำงานอยู่
  ส่วนการอัปเดตที่เข้าคิวไว้สามารถมาถึงภายหลังในเบื้องหลังได้
- การเชื่อมต่อใหม่ล้มเหลวหลังติดตั้งใหม่: โทเค็นการจับคู่ใน Keychain ถูกล้างแล้ว ให้จับคู่ Node ใหม่
