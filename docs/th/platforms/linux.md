Gateway รองรับ Linux อย่างสมบูรณ์และต้องใช้ Node ส่วน Bun ยังใช้เป็น
ตัวติดตั้ง dependency หรือตัวรันสคริปต์แพ็กเกจได้ แต่ไม่สามารถรัน OpenClaw
ได้เนื่องจากไม่มี `node:sqlite`

## แอปเดสก์ท็อปคู่หู

แอปคู่หู OpenClaw สำหรับ Linux เป็นแอปเดสก์ท็อป Tauri สำหรับ Gateway ภายในเครื่อง โดยแอปจะ:

- ติดตั้ง OpenClaw CLI และรันไทม์ Node ที่จัดการให้เมื่อยังไม่มี โดยบิลด์รุ่นเผยแพร่จะติดตั้งช่องทางเสถียรโดยอัตโนมัติ ส่วนบิลด์สำหรับการพัฒนาจะถามช่องทางก่อน
- เชื่อมต่อกับ Gateway ที่ทำงานปกติก่อนพยายามเปลี่ยนแปลงบริการ
- มอบหมายการติดตั้ง การเริ่ม หยุด และรีสตาร์ตให้แก่บริการผู้ใช้ systemd ที่จัดการโดย CLI
- ค้นหา Bonjour Gateway ใกล้เคียงและเปิด Control UI จากปลายทางบริการที่แก้ไขแล้ว
- เปิด Control UI ที่ให้บริการโดย Gateway ด้วย URL การยืนยันตัวตนที่แก้ไขแล้ว
- เปิด Control UI ในโหมดเริ่มต้นใช้งานหลังการติดตั้งครั้งแรก ซึ่ง
  เสนอให้นำเข้าหน่วยความจำ Claude Code, Codex หรือ Hermes ที่ตรวจพบไปยัง
  พื้นที่ทำงานของเอเจนต์ (การนำเข้าเดียวกันนี้ยังใช้งานได้ภายหลังที่
  Settings → Import Memory)
- แสดงผล Canvas ที่ขับเคลื่อนโดยเอเจนต์และเนื้อหา A2UI ที่รวมมาให้สำหรับโฮสต์ Node ของ CLI ที่อยู่ร่วมกัน
- ยังคงพร้อมใช้งานจากถาดระบบเมื่อปิดหน้าต่างแล้ว

รุ่นเสถียรที่สร้างจาก `main` จะจัดส่งบันเดิล `.deb` และ AppImage เป็นแอสเซ็ตใน
[รุ่นเผยแพร่บน GitHub](https://github.com/openclaw/openclaw/releases) สำหรับแท็กนั้น
โดยใช้ชื่อ `OpenClaw-<version>-amd64.deb` และ `OpenClaw-<version>-amd64.AppImage`
พร้อมไฟล์ checksum `SHA256SUMS.linux-app.txt` อยู่ข้างกัน ดาวน์โหลด
`.deb` และติดตั้งด้วย `sudo apt install ./OpenClaw-<version>-amd64.deb`
หรือกำหนดให้ AppImage เรียกทำงานได้แล้วรันโดยตรง รันไทม์ AppImage
ต้องใช้ FUSE 2 (`sudo apt install libfuse2` หรือ `libfuse2t64` บน Ubuntu 24.04+)
หากไม่มี ให้รัน AppImage ด้วย `APPIMAGE_EXTRACT_AND_RUN=1`

คุณยังสามารถสร้างบันเดิลเดียวกันจาก checkout ของซอร์สได้:

```bash
cd apps/linux/src-tauri
pnpm dlx @tauri-apps/cli@2.11.4 build --bundles deb,appimage
```

เวิร์กโฟลว์ CI `Linux App` จะอัปโหลดบันเดิลเดียวกันเป็น
อาร์ติแฟกต์ `openclaw-linux-companion` สำหรับ pull request ที่แก้ไขแอปและสำหรับ
การรันด้วยตนเอง โปรดดู `apps/linux/README.md` ในรีโพซิทอรีสำหรับ dependency
ในการบิลด์บน Linux และคำสั่งสำหรับการพัฒนา

### แชตด่วน

เปิดแชตด่วนด้วย `Ctrl+Shift+Space` หรือรายการ **Quick Chat** ในถาดระบบ ชิปเอเจนต์
จะแสดงอวตาร อีโมจิ หรืออักษรย่อที่กำหนดไว้ เลือกชิปเพื่อสลับเอเจนต์
ข้อความจะใช้เซสชันหลักของเอเจนต์ที่เลือกและปฏิบัติตามขอบเขตเซสชันส่วนกลาง
ไคลเอนต์ Rust แบบเนทีฟเป็นเจ้าของข้อมูลประจำตัวอุปกรณ์ Ed25519 แบบถาวร โดยใช้
โทเค็นหรือรหัสผ่านที่ใช้ร่วมกันจากการส่งมอบของ CLI เพียงเพื่อเริ่มต้นการจับคู่ จากนั้นจึงจัดเก็บและ
เลือกใช้โทเค็นอุปกรณ์ที่ Gateway ออกให้ในการเชื่อมต่อครั้งต่อไป ข้อมูลประจำตัวและ
โทเค็นอุปกรณ์อยู่ในไดเรกทอรีการกำหนดค่าแอปในไฟล์โหมด `0600`; WebView ของแชตด่วน
จะไม่ได้รับทั้งข้อมูลรับรองและ WebSocket

เมื่อการเชื่อมต่อแบบเนทีฟไม่พร้อมใช้งาน แชตด่วนจะแสดง **Gateway
unreachable — retrying** และปิดการส่งจนกว่าจะเชื่อมต่อใหม่ อุปกรณ์ระยะไกล
ที่เข้าสู่ขั้นตอนการจับคู่แล้วจะแสดง **Approve this device in the dashboard
(Nodes)** แทน พร้อม ID อุปกรณ์แบบสั้นเมื่อ Gateway ระบุให้ ส่วน
Gateway ที่ต้องใช้ข้อมูลรับรองร่วมซึ่งขาดหายไปจะแสดง **Gateway requires a
credential — open the dashboard on the gateway host**; ในสถานะดังกล่าวจะไม่มีคำขอจับคู่
รอการอนุมัติ คำแนะนำการแก้ไขที่เซิร์ฟเวอร์ระบุ
จะแทนที่ข้อความสำรองเหล่านี้เมื่อมีรายละเอียดเฉพาะเจาะจงกว่า
สำหรับ Gateway แบบ TLS นั้น CLI จะส่งลายนิ้วมือ SHA-256 ของใบรับรอง
Gateway ให้แก่แอป ไคลเอนต์แบบเนทีฟจะตรึงใบรับรองนั้นและรายงาน **Gateway TLS
trust failed — check the certificate fingerprint** แยกจากกรณีระบบหยุดทำงาน
Gateway ที่กำหนดค่าความลับร่วมผ่าน SecretRef จะไม่นำความลับนั้นรวมไว้ในการ
ส่งมอบของ CLI การติดตั้งเดิมที่จับคู่แล้วจะยังคงทำงานผ่านโทเค็นอุปกรณ์ที่จัดเก็บไว้
แต่การติดตั้งใหม่ไม่สามารถสร้างคำขอจับคู่ที่รอดำเนินการภายใต้การยืนยันตัวตน
ด้วยความลับร่วมได้หากไม่มีข้อมูลรับรองเริ่มต้นดังกล่าว
การแลกใช้รหัสตั้งค่าและ `bootstrapToken` ต้องมี UI ผลิตภัณฑ์เฉพาะและยังคงเป็น
งานติดตามผล แชตด่วนจะไม่พยายามดำเนินขั้นตอนใดทั้งสองแบบ

บน X11 ให้ใช้ไอคอนเฟืองในแชตด่วนเพื่อบันทึกหรือรีเซ็ตแป้นพิมพ์ลัดแบบกำหนดเอง สวิตช์
**Quick Chat shortcut** ในถาดระบบจะเปิดหรือปิดแป้นพิมพ์ลัดโดยไม่ปิดใช้งาน
รายการ **Quick Chat** ปกติในถาดระบบ แป้นพิมพ์ลัดส่วนกลางใช้ไม่ได้บน Wayland ดังนั้น
การตั้งค่าแป้นพิมพ์ลัดจึงถูกซ่อนและรายการในถาดระบบยังคงเป็นจุดเข้าใช้งาน
หลังจากยอมรับการส่งแล้ว แชตด่วนจะยังเปิดอยู่และสตรีมคำตอบข้อความล้วนของเอเจนต์ที่เลือก
ไว้ใต้ช่องเขียนข้อความ กด `Esc` เพื่อปิดแถบและคำตอบของแถบ
ส่วน `Ctrl+Enter` ยังคงใช้เปิดแดชบอร์ด

### Canvas

Canvas บน Linux ใช้สองโปรเซสที่ทำงานร่วมกัน `openclaw node run` ยังคงเป็นการเชื่อมต่อ Node เดียวของ Gateway ส่วน Plugin `linux-canvas` ที่รวมมาให้จะส่งต่อการเรียก `canvas.*` ไปยังแอปเดสก์ท็อปที่กำลังทำงานผ่าน Unix socket สำหรับผู้ใช้เท่านั้น แอปเป็นเจ้าของหน้าต่าง WebView แบบสร้างเมื่อจำเป็นหนึ่งหน้าต่าง ซึ่งรวมตัวเรนเดอร์ A2UI ที่ให้มาด้วยและบริดจ์การดำเนินการกลับไปยังเอเจนต์

Plugin เปิดใช้งานเป็นค่าเริ่มต้น โดยจะประกาศ Canvas เฉพาะเมื่อมีซ็อกเก็ตเดสก์ท็อปที่ `$XDG_RUNTIME_DIR/openclaw-canvas.sock` หรือ `/tmp/openclaw-canvas-$UID.sock` เมื่อ `XDG_RUNTIME_DIR` ใช้งานไม่ได้ ปิดใช้งานด้วย `plugins.entries.linux-canvas.enabled: false` บนเซิร์ฟเวอร์ Linux แบบไม่มีส่วนติดต่อซึ่งไม่มีแอปเดสก์ท็อป จะไม่มีการประกาศ Canvas

Linux v1 ใช้หน้าต่าง Canvas หนึ่งหน้าต่าง สามารถแสดงผลหน้า HTTP และ HTTPS ได้ แต่จะยอมรับการดำเนินการ A2UI จากตัวเรนเดอร์ที่รวมมาให้เท่านั้น

## ทางเลือก CLI และ SSH

CLI ยังคงเป็นตัวเลือกที่ง่ายที่สุดสำหรับเซิร์ฟเวอร์แบบไม่มีส่วนติดต่อ, VPS หรือ Gateway ระยะไกล:

1. ติดตั้ง Node 24.15+ (แนะนำ), Node 22.22.3+ (LTS) หรือ Node 25.9+
2. `npm i -g openclaw@latest`
3. `openclaw onboard --install-daemon`
4. จากแล็ปท็อป: `ssh -N -L 18789:127.0.0.1:18789 <user>@<host>`
5. เปิด `http://127.0.0.1:18789/` และยืนยันตัวตนด้วยความลับร่วมที่กำหนดค่าไว้
   (ค่าเริ่มต้นคือโทเค็น หรือรหัสผ่านหาก `gateway.auth.mode` เป็น `"password"`)

คู่มือเซิร์ฟเวอร์ฉบับเต็ม: [เซิร์ฟเวอร์ Linux](/th/vps) ตัวอย่าง VPS แบบทีละขั้นตอน:
[exe.dev](/th/install/exe-dev)

## ความสามารถของ Node

Plugin สำหรับ Node บน Linux ที่รวมมาให้จะมอบความสามารถของอุปกรณ์บริการ `openclaw node` แก่ CLI โดยไม่ต้องใช้แอปเดสก์ท็อป คำสั่งจะถูกประกาศต่อ Gateway เฉพาะเมื่อเปิดใช้งานความสามารถและมีเครื่องมือภายในเครื่องที่จำเป็นเท่านั้น

| ความสามารถ                              | ค่าเริ่มต้น | ข้อกำหนด                                                           |
| --------------------------------------- | ------- | --------------------------------------------------------------------- |
| การแจ้งเตือนบนเดสก์ท็อป (`system.notify`) | เปิด      | `notify-send` จาก libnotify และเซสชันการแจ้งเตือนบนเดสก์ท็อป       |
| ภาพถ่ายและคลิปจากกล้อง (`camera.*`)    | ปิด     | FFmpeg, สิทธิ์เข้าถึงกล้อง V4L2 และ PulseAudio หรือ PipeWire สำหรับเสียงในคลิป |
| ตำแหน่ง (`location.get`)               | ปิด     | GeoClue2 และตัวอย่าง `where-am-i` ของโปรแกรม                                    |

กำหนดค่า Plugin ใน `openclaw.json`:

```json5
{
  plugins: {
    entries: {
      "linux-node": {
        config: {
          notify: { enabled: true },
          camera: { enabled: true },
          location: { enabled: true },
        },
      },
    },
  },
}
```

รีสตาร์ตบริการ Node หลังจากเปลี่ยนการตั้งค่าเหล่านี้ ระบบจะตรวจสอบความพร้อมใช้งานหนึ่งครั้งต่อโปรเซส และสร้างการประกาศ Node ใหม่เมื่อรีสตาร์ต

Gateway อนุมัติพื้นผิวคำสั่งและความสามารถของ Node แยกจากการจับคู่อุปกรณ์ เมื่อเริ่มทำงานครั้งแรกหรือหลังจากเปิดใช้ความสามารถเพิ่มเติม ให้อนุมัติพื้นผิวที่รอดำเนินการ:

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
```

Node สามารถเชื่อมต่อและจับคู่กับอุปกรณ์แล้ว ขณะที่ `caps` และ `commands` ที่มีผลยังคงว่างเปล่าจนกว่าการอนุมัตินี้จะเสร็จสมบูรณ์

ผู้ใช้บริการต้องมีสิทธิ์อ่านอุปกรณ์กล้อง ซึ่งโดยทั่วไปทำผ่านกลุ่ม `video` คลิปจากกล้องจะใช้แหล่ง PulseAudio หรือ PipeWire เริ่มต้นเมื่อ `includeAudio` เป็น true โดยเสียงไมโครโฟนจะมีอยู่เฉพาะเป็นแทร็กของคลิปนั้น ไม่ใช่คำสั่งแบบแยกต่างหาก ตำแหน่งต้องให้ผู้ใช้บริการ Node ได้รับอนุญาตตามนโยบาย GeoClue ของโฮสต์

`camera.snap` และ `camera.clip` ยังต้องเปิดใช้งานอย่างชัดเจนผ่าน Gateway โดยใช้ `gateway.nodes.allowCommands` โปรดดู [การบันทึกภาพจากกล้อง](/th/nodes/camera) และ [คำสั่งตำแหน่ง](/th/nodes/location-command) สำหรับเพย์โหลด ขีดจำกัด และข้อผิดพลาด

## บริการ Gateway (systemd)

ติดตั้งด้วยวิธีใดวิธีหนึ่งต่อไปนี้:

```bash
openclaw onboard --install-daemon
openclaw gateway install
openclaw configure   # เลือก "Gateway service" เมื่อระบบถาม
```

ซ่อมแซมหรือย้ายการติดตั้งที่มีอยู่:

```bash
openclaw doctor
```

`openclaw gateway install` จะสร้างยูนิต **ผู้ใช้** ของ systemd เป็นค่าเริ่มต้น คำแนะนำเกี่ยวกับ
บริการฉบับเต็ม รวมถึงยูนิตระดับ **ระบบ** สำหรับโฮสต์ที่ใช้ร่วมกันหรือ
เปิดทำงานตลอดเวลา อยู่ใน [คู่มือการดำเนินงาน Gateway](/th/gateway#supervision-and-service-lifecycle)

เขียนยูนิตด้วยตนเองเฉพาะสำหรับการตั้งค่าแบบกำหนดเอง ตัวอย่างยูนิตผู้ใช้ขั้นต่ำ
(`~/.config/systemd/user/openclaw-gateway[-<profile>].service`):

```ini
[Unit]
Description=OpenClaw Gateway (โปรไฟล์: <profile>, v<version>)
After=network-online.target
Wants=network-online.target
StartLimitBurst=5
StartLimitIntervalSec=60

[Service]
ExecStart=/usr/local/bin/openclaw gateway --port 18789
Restart=always
RestartSec=5
RestartPreventExitStatus=78
TimeoutStopSec=30
TimeoutStartSec=30
SuccessExitStatus=0 143
OOMPolicy=continue
KillMode=control-group

[Install]
WantedBy=default.target
```

ยูนิตที่เขียนด้วยตนเองจะไม่ได้รับการปรับขนาดฮีปแบบปรับตามสภาวะที่ `openclaw gateway install` เขียนให้บริการ Gateway ที่จัดการโดยระบบ ควรใช้ตัวติดตั้งที่จัดการให้ หรือกำหนดขีดจำกัดฮีปอย่างชัดเจนในตัวควบคุมแบบกำหนดเองหลังจากเผื่อพื้นที่หน่วยความจำเนทีฟแล้ว

เปิดใช้งาน:

```bash
systemctl --user enable --now openclaw-gateway[-<profile>].service
```

## ภาวะกดดันของหน่วยความจำและการยุติโดย OOM

บน Linux เคอร์เนลจะเลือกโปรเซสเป้าหมาย OOM เมื่อโฮสต์, VM หรือ cgroup ของคอนเทนเนอร์
มีหน่วยความจำไม่เพียงพอ Gateway ไม่เหมาะเป็นเป้าหมายเพราะเป็นเจ้าของเซสชัน
และการเชื่อมต่อช่องทางที่มีอายุยาวนาน ดังนั้น OpenClaw จึงให้น้ำหนักกับการยุติ
โปรเซสลูกชั่วคราวก่อนเมื่อทำได้

สำหรับการสร้างโปรเซสลูกบน Linux ที่เข้าเกณฑ์ OpenClaw จะครอบคำสั่งด้วยชิม
`/bin/sh` ขนาดสั้น ซึ่งเพิ่ม `oom_score_adj` ของโปรเซสลูกเองเป็น `1000` จากนั้น
`exec` คำสั่งจริง การดำเนินการนี้ไม่ต้องใช้สิทธิ์พิเศษ เพราะโปรเซสสามารถเพิ่ม
คะแนน OOM ของตนเองได้เสมอ

พื้นผิวโปรเซสลูกที่ครอบคลุม:

- โปรเซสลูกของคำสั่งที่จัดการโดยตัวควบคุม
- โปรเซสลูกของเชลล์ PTY
- โปรเซสลูกของเซิร์ฟเวอร์ MCP stdio
- โปรเซสเบราว์เซอร์/Chrome ที่ OpenClaw เริ่มทำงาน (ผ่านรันไทม์โปรเซสของ plugin SDK)

ตัวครอบนี้ใช้เฉพาะ Linux และจะข้ามเมื่อ `/bin/sh` ใช้งานไม่ได้ หรือเมื่อ
สภาพแวดล้อมของโปรเซสลูกตั้งค่า `OPENCLAW_CHILD_OOM_SCORE_ADJ` เป็น `0`, `false`, `no` หรือ
`off`

ตรวจสอบโปรเซสลูก:

```bash
cat /proc/<child-pid>/oom_score_adj
```

ค่าที่คาดไว้สำหรับโปรเซสลูกที่ครอบคลุมคือ `1000` ส่วนโปรเซส Gateway เอง
จะคงคะแนนปกติไว้ (โดยทั่วไปคือ `0`)

`OOMPolicy=continue` ของยูนิต systemd จะทำให้บริการ Gateway ทำงานต่อไปเมื่อ
โปรเซสลูกชั่วคราวถูก OOM killer เลือก แทนที่จะทำเครื่องหมายว่ายูนิตทั้งหมด
ล้มเหลวและรีสตาร์ตทุกช่องทาง โดยโปรเซสลูก/เซสชันที่ล้มเหลวจะรายงาน
ข้อผิดพลาดของตนเอง

สิ่งนี้ไม่สามารถแทนที่การปรับแต่งหน่วยความจำตามปกติได้ หาก VPS หรือคอนเทนเนอร์
ยุติโปรเซสลูกซ้ำ ๆ ให้เพิ่มขีดจำกัดหน่วยความจำ ลดการทำงานพร้อมกัน หรือเพิ่ม
การควบคุมทรัพยากรที่เข้มงวดยิ่งขึ้น (systemd `MemoryMax=`, ขีดจำกัดหน่วยความจำของคอนเทนเนอร์)
