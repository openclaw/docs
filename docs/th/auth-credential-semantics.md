เอกสารนี้กำหนดความหมายมาตรฐานสำหรับคุณสมบัติการมีสิทธิ์ใช้งานและการแก้ไขข้อมูลประจำตัวที่ใช้ทั่วทั้ง:

- `resolveAuthProfileOrder`
- `resolveApiKeyForProfile`
- `models status --probe`
- `doctor-auth`

เป้าหมายคือทำให้พฤติกรรมตอนเลือกและตอนรันสอดคล้องกัน

## รหัสเหตุผลของการ probe ที่เสถียร

- `ok`
- `excluded_by_auth_order`
- `missing_credential`
- `invalid_expires`
- `expired`
- `unresolved_ref`
- `no_model`

## ข้อมูลประจำตัวแบบโทเค็น

ข้อมูลประจำตัวแบบโทเค็น (`type: "token"`) รองรับ `token` แบบอินไลน์ และ/หรือ `tokenRef`

### กฎการมีสิทธิ์ใช้งาน

1. โปรไฟล์โทเค็นไม่มีสิทธิ์ใช้งานเมื่อไม่มีทั้ง `token` และ `tokenRef`
2. `expires` เป็นค่าไม่บังคับ
3. หากมี `expires` ต้องเป็นตัวเลขจำกัดที่มากกว่า `0`
4. หาก `expires` ไม่ถูกต้อง (`NaN`, `0`, ค่าลบ, ไม่จำกัด, หรือชนิดผิด) โปรไฟล์จะไม่มีสิทธิ์ใช้งานด้วย `invalid_expires`
5. หาก `expires` อยู่ในอดีต โปรไฟล์จะไม่มีสิทธิ์ใช้งานด้วย `expired`
6. `tokenRef` ไม่ข้ามการตรวจสอบ `expires`

### กฎการแก้ไข

1. ความหมายของตัวแก้ไขตรงกับความหมายของการมีสิทธิ์ใช้งานสำหรับ `expires`
2. สำหรับโปรไฟล์ที่มีสิทธิ์ใช้งาน วัสดุโทเค็นอาจถูกแก้ไขจากค่าอินไลน์หรือ `tokenRef`
3. ref ที่แก้ไขไม่ได้จะสร้าง `unresolved_ref` ในเอาต์พุต `models status --probe`

## การพกพาการคัดลอกของเอเจนต์

การสืบทอด auth ของเอเจนต์เป็นแบบอ่านผ่าน เมื่อเอเจนต์ไม่มีโปรไฟล์ในเครื่อง เอเจนต์สามารถแก้ไขโปรไฟล์จากที่เก็บเอเจนต์เริ่มต้น/หลักได้ขณะรันโดยไม่ต้องคัดลอกวัสดุลับไปยัง `auth-profiles.json` ของตัวเอง

โฟลว์การคัดลอกแบบชัดเจน เช่น `openclaw agents add` ใช้นโยบายการพกพานี้:

- โปรไฟล์ `api_key` พกพาได้ เว้นแต่ `copyToAgents: false`
- โปรไฟล์ `token` พกพาได้ เว้นแต่ `copyToAgents: false`
- โปรไฟล์ `oauth` ไม่พกพาโดยค่าเริ่มต้น เนื่องจาก refresh token อาจใช้ได้ครั้งเดียวหรือไวต่อการหมุนเวียน
- โฟลว์ OAuth ที่ผู้ให้บริการเป็นเจ้าของอาจเลือกเข้าร่วมด้วย `copyToAgents: true` เฉพาะเมื่อทราบว่าการคัดลอกวัสดุ refresh ข้ามเอเจนต์ปลอดภัย

โปรไฟล์ที่พกพาไม่ได้ยังคงใช้งานได้ผ่านการสืบทอดแบบอ่านผ่าน เว้นแต่เอเจนต์เป้าหมายจะลงชื่อเข้าใช้แยกต่างหากและสร้างโปรไฟล์ในเครื่องของตัวเอง

## การกรองลำดับ auth แบบชัดเจน

- เมื่อมีการตั้งค่า `auth.order.<provider>` หรือการแทนที่ลำดับ auth-store สำหรับผู้ให้บริการหนึ่ง `models status --probe` จะ probe เฉพาะ id โปรไฟล์ที่ยังอยู่ในลำดับ auth ที่แก้ไขแล้วสำหรับผู้ให้บริการนั้น
- โปรไฟล์ที่จัดเก็บไว้สำหรับผู้ให้บริการนั้นซึ่งถูกละไว้จากลำดับแบบชัดเจนจะไม่ถูกลองอย่างเงียบ ๆ ในภายหลัง เอาต์พุตการ probe จะรายงานด้วย `reasonCode: excluded_by_auth_order` และรายละเอียด `Excluded by auth.order for this provider.`

## การแก้ไขเป้าหมายการ probe

- เป้าหมายการ probe อาจมาจากโปรไฟล์ auth, ข้อมูลประจำตัวสภาพแวดล้อม, หรือ `models.json`
- หากผู้ให้บริการมีข้อมูลประจำตัวแต่ OpenClaw ไม่สามารถแก้ไขตัวเลือกโมเดลที่ probe ได้สำหรับผู้ให้บริการนั้น `models status --probe` จะรายงาน `status: no_model` พร้อม `reasonCode: no_model`

## การค้นพบข้อมูลประจำตัว CLI ภายนอก

- ข้อมูลประจำตัวที่ใช้เฉพาะตอนรันซึ่ง CLI ภายนอกเป็นเจ้าของจะถูกค้นพบเฉพาะเมื่อผู้ให้บริการ, runtime, หรือโปรไฟล์ auth อยู่ในขอบเขตของการดำเนินการปัจจุบัน หรือเมื่อมีโปรไฟล์ในเครื่องที่จัดเก็บไว้สำหรับแหล่งภายนอกนั้นอยู่แล้ว
- เส้นทางอ่านอย่างเดียว/สถานะส่งผ่าน `allowKeychainPrompt: false`; เส้นทางเหล่านี้ใช้เฉพาะข้อมูลประจำตัว CLI ภายนอกที่มีไฟล์รองรับ และไม่อ่านหรือนำผลลัพธ์ macOS Keychain กลับมาใช้

## ตัวคุ้มนโยบาย OAuth SecretRef

- อินพุต SecretRef ใช้สำหรับข้อมูลประจำตัวแบบคงที่เท่านั้น
- หากข้อมูลประจำตัวของโปรไฟล์เป็น `type: "oauth"` จะไม่รองรับอ็อบเจ็กต์ SecretRef สำหรับวัสดุข้อมูลประจำตัวของโปรไฟล์นั้น
- หาก `auth.profiles.<id>.mode` เป็น `"oauth"` อินพุต `keyRef`/`tokenRef` ที่รองรับด้วย SecretRef สำหรับโปรไฟล์นั้นจะถูกปฏิเสธ
- การละเมิดเป็นความล้มเหลวแบบเด็ดขาดในเส้นทางการแก้ไข auth ตอนเริ่มต้น/โหลดใหม่

## ข้อความที่เข้ากันได้กับระบบเดิม

เพื่อความเข้ากันได้ของสคริปต์ ข้อผิดพลาดการ probe จะคงบรรทัดแรกนี้ไว้ไม่เปลี่ยนแปลง:

`Auth profile credentials are missing or expired.`

อาจเพิ่มรายละเอียดที่เป็นมิตรต่อมนุษย์และรหัสเหตุผลที่เสถียรในบรรทัดถัด ๆ ไป
