กระบวนการ **Gateway เพียงกระบวนการเดียว** เป็นเจ้าของสถานะเซสชันตั้งแต่ต้นจนจบ UI ต่างๆ (แอป macOS, Control UI บนเว็บ, TUI) สอบถามรายการเซสชันและจำนวนโทเค็นจาก Gateway ในโหมดระยะไกล ไฟล์เซสชันจะอยู่บนโฮสต์ระยะไกล ดังนั้นการตรวจสอบไฟล์ใน Mac ภายในเครื่องจะไม่สะท้อนสิ่งที่ Gateway กำลังใช้งาน

เริ่มจากเอกสารภาพรวม: [การจัดการเซสชัน](/th/concepts/session), [Compaction](/th/concepts/compaction), [ภาพรวมหน่วยความจำ](/th/concepts/memory), [การค้นหาหน่วยความจำ](/th/concepts/memory-search), [การตัดแต่งเซสชัน](/th/concepts/session-pruning), [สุขอนามัยของทรานสคริปต์](/th/reference/transcript-hygiene), ข้อมูลอ้างอิงการกำหนดค่าฉบับเต็มที่ [การกำหนดค่าเอเจนต์](/th/gateway/config-agents)

## เลเยอร์การคงอยู่สองชั้น

1. **แถวเซสชัน (SQLite ต่อเอเจนต์)** - แมปคีย์/ค่า `sessionKey -> SessionEntry` สถานะรันไทม์ที่เปลี่ยนแปลงได้ซึ่ง Gateway เป็นเจ้าของ ใช้ติดตามเมตาดาต้า ได้แก่ รหัสเซสชันปัจจุบัน กิจกรรมล่าสุด ตัวสลับ และตัวนับโทเค็น
2. **เหตุการณ์ทรานสคริปต์ (SQLite ต่อเอเจนต์)** - เพิ่มต่อท้ายเท่านั้นและมีโครงสร้างแบบต้นไม้ (รายการมี `id` + `parentId`) จัดเก็บบทสนทนา การเรียกใช้เครื่องมือ และข้อมูลสรุป Compaction พร้อมสร้างบริบทโมเดลใหม่สำหรับรอบถัดไป จุดตรวจ Compaction เป็นเมตาดาต้าเหนือทรานสคริปต์ตัวสืบทอดที่ถูกบีบอัด โดย Compaction ใหม่จะไม่เขียนสำเนา `.checkpoint.*.jsonl` ชุดที่สอง

การติดตั้งรุ่นเก่าอาจยังมีไฟล์ `sessions.json` อยู่ภายใต้ไดเรกทอรี `sessions/`
ของเอเจนต์ ให้ถือว่าไฟล์เหล่านั้นเป็นอินพุตสำหรับย้ายแถวเซสชันแบบเดิม หรือเป็น
เป้าหมายการบำรุงรักษาแบบออฟไลน์โดยชัดแจ้ง การเริ่มต้น Gateway และ `openclaw doctor --fix` จะนำเข้า
แถวเดิมที่ใช้งานอยู่และประวัติทรานสคริปต์ไปยังที่เก็บ SQLite ต่อเอเจนต์
โดยอัตโนมัติ เรียกใช้ `openclaw doctor --session-sqlite inspect
--session-sqlite-all-agents` แล้วทำตาม [ลำดับการย้ายข้อมูลของ
Doctor](/th/cli/doctor#session-sqlite-migration) เมื่อต้องการหลักฐานการตรวจสอบ
หรือการตรวจยืนยันอย่างชัดแจ้ง หากการย้ายข้อมูลล้มเหลวหลังจากเก็บถาวรอาร์ติแฟกต์
ทรานสคริปต์แบบเดิมแล้ว ให้ใช้โหมดกู้คืนของ Doctor จากลำดับดังกล่าว
การกู้คืนใช้ไฟล์กำกับการย้ายข้อมูล คืนค่าเฉพาะอาร์ติแฟกต์สนับสนุนที่เก็บถาวร
ซึ่งได้รับผลกระทบ จัดเตรียมรายงานปัญหา GitHub ที่ผ่านการล้างข้อมูลเมื่อมีการร้องขอ
และไม่ทำให้รันไทม์ที่ใช้งานอยู่กลับมาอ่านไฟล์ JSONL อีกครั้ง

ตัวอ่านประวัติของ Gateway หลีกเลี่ยงการโหลดทรานสคริปต์ทั้งหมดไว้ในหน่วยความจำ เว้นแต่พื้นผิวนั้นต้องเข้าถึงประวัติส่วนใดก็ได้ ประวัติหน้าแรก ประวัติแชตแบบฝัง การกู้คืนหลังรีสตาร์ต และการตรวจสอบโทเค็น/การใช้งาน ใช้การอ่านส่วนท้ายแบบจำกัดจาก SQLite การสแกนทรานสคริปต์ทั้งหมดจะผ่านดัชนีทรานสคริปต์แบบอะซิงโครนัสและใช้ร่วมกันระหว่างตัวอ่านที่ทำงานพร้อมกัน

## ตำแหน่งบนดิสก์

สำหรับแต่ละเอเจนต์ บนโฮสต์ Gateway (แก้ไขตำแหน่งผ่าน `src/config/sessions.ts`):

- ที่เก็บแถวเซสชันรันไทม์: `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- แถวทรานสคริปต์รันไทม์: `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- อาร์ติแฟกต์ทรานสคริปต์แบบเดิม/ที่เก็บถาวร: `~/.openclaw/agents/<agentId>/sessions/`
- อินพุตการย้ายแถวแบบเดิม: `~/.openclaw/agents/<agentId>/sessions/sessions.json`

## การบำรุงรักษาที่เก็บและการควบคุมดิสก์

`session.maintenance` ควบคุมการบำรุงรักษาอัตโนมัติสำหรับแถวเซสชัน SQLite แถวทรานสคริปต์ SQLite อาร์ติแฟกต์ที่เก็บถาวร และไฟล์เสริมวิถีการทำงาน:

| คีย์                     | ค่าเริ่มต้น               | หมายเหตุ                                                                                       |
| ----------------------- | --------------------- | ------------------------------------------------------------------------------------------- |
| `mode`                  | `"enforce"`           | หรือ `"warn"` (รายงานเท่านั้น ไม่มีการแก้ไข)                                                      |
| `pruneAfter`            | `"30d"`               | เกณฑ์อายุสำหรับรายการที่ล้าสมัย                                                                      |
| `maxEntries`            | `500`                 | ขีดจำกัดจำนวนรายการเซสชัน                                                                      |
| `resetArchiveRetention` | เก็บไว้ (ไม่มีเกณฑ์อายุ)  | เกณฑ์อายุสำหรับทรานสคริปต์ที่เก็บถาวร `*.reset.*`/`*.deleted.*`; การระบุระยะเวลาจะเปิดใช้การลบ |
| `maxDiskBytes`          | `2gb`                 | งบประมาณดิสก์สำหรับเซสชันต่อเอเจนต์; `false` ปิดใช้งาน                                            |
| `highWaterBytes`        | 80% ของ `maxDiskBytes` | เป้าหมายหลังการล้างตามงบประมาณ                                                                 |

ทรานสคริปต์ที่เก็บถาวรจะถูกเก็บไว้ตามค่าเริ่มต้นและบีบอัดด้วย zstd (`*.jsonl.<reason>.<timestamp>.zst`) เมื่อรันไทม์รองรับ ดังนั้นการลบหรือรีเซ็ตเซสชันจะไม่ทิ้งประวัติการสนทนาโดยไม่แจ้งให้ทราบ งบประมาณดิสก์จะขับทรานสคริปต์ที่เก็บถาวรเก่าที่สุดออกก่อน แล้วจึงแตะต้องเซสชันที่ใช้งานอยู่

การบังคับใช้ `maxDiskBytes` ใน SQLite ที่ใช้งานอยู่จะวัดจำนวนไบต์ของ JSON แถวเซสชันรวมกับ JSON เหตุการณ์ทรานสคริปต์ต่อเซสชัน ส่วนการบังคับใช้สำหรับการบำรุงรักษาแบบออฟไลน์รุ่นเดิมจะวัดไฟล์ในไดเรกทอรีเซสชันที่เลือก

เซสชันโพรบการเรียกใช้โมเดลของ Gateway (คีย์ที่ตรงกับ `agent:*:explicit:model-run-<uuid>`) มีระยะเวลาเก็บรักษาคงที่แยกต่างหากที่ `24h` การตัดแต่งนี้ขึ้นอยู่กับแรงกดดัน โดยจะทำงานเฉพาะเมื่อการบำรุงรักษา/ขีดจำกัดจำนวนรายการเซสชันถึงระดับกดดัน และเฉพาะก่อนขั้นตอนล้าง/จำกัดรายการล้าสมัยส่วนกลางเท่านั้น เซสชันแบบชัดแจ้งอื่นๆ ไม่ใช้ระยะเวลาเก็บรักษานี้

ลำดับการบังคับใช้สำหรับการล้างตามงบประมาณดิสก์ (`mode: "enforce"`):

1. นำอาร์ติแฟกต์ทรานสคริปต์ที่เก็บถาวรที่เก่าที่สุด อาร์ติแฟกต์แบบเดิมที่ไม่มีเจ้าของ หรืออาร์ติแฟกต์วิถีการทำงานที่ไม่มีเจ้าของออกก่อน
2. หากยังเกินเป้าหมาย ให้ขับรายการเซสชันที่เก่าที่สุดและแถวทรานสคริปต์หรืออาร์ติแฟกต์วิถีการทำงานที่เกี่ยวข้องออก
3. ทำซ้ำจนกว่าการใช้งานจะเท่ากับหรือต่ำกว่า `highWaterBytes`

`mode: "warn"` รายงานรายการที่อาจถูกขับออกโดยไม่แก้ไขที่เก็บหรือไฟล์

เรียกใช้การบำรุงรักษาตามต้องการ:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

การบำรุงรักษาจะเก็บตัวชี้การสนทนาภายนอกที่คงทน เช่น เซสชันกลุ่มและเซสชันแชตที่มีขอบเขตเป็นเธรด แต่รายการรันไทม์สังเคราะห์ (Cron, ฮุก, Heartbeat, ACP, เอเจนต์ย่อย) ยังสามารถถูกนำออกได้เมื่อเกินอายุ จำนวน หรืองบประมาณดิสก์ที่กำหนด การทำงาน Cron แบบแยกใช้การควบคุม `cron.sessionRetention` แยกต่างหาก โดยไม่ขึ้นกับระยะเวลาเก็บรักษาโพรบการเรียกใช้โมเดล

การเขียนปกติของ Gateway จะผ่านตัวเข้าถึงเซสชัน ซึ่งจัดลำดับการเปลี่ยนแปลง SQLite ต่อเอเจนต์ผ่านเส้นทางตัวเขียนรันไทม์ โค้ดรันไทม์ควรเลือกใช้ตัวช่วยของตัวเข้าถึงใน `src/config/sessions/session-accessor.ts`; ตัวช่วย `sessions.json` แบบเดิมเป็นเครื่องมือสำหรับการย้ายข้อมูลและการบำรุงรักษาแบบออฟไลน์ เมื่อเข้าถึง Gateway ได้ คำสั่ง `openclaw sessions cleanup` และ `openclaw agents delete` ที่ไม่ใช่โหมดทดลองจะมอบหมายการเปลี่ยนแปลงที่เก็บให้ Gateway เพื่อให้การล้างเข้าร่วมคิวตัวเขียนเดียวกัน ส่วน `--store <path>` เป็นเส้นทางซ่อมแซมแบบออฟไลน์อย่างชัดแจ้งสำหรับที่เก็บแบบเดิมที่เลือกและจะทำงานภายในเครื่องเสมอ (เช่นเดียวกับ `--dry-run`) การล้าง `maxEntries` จะประมวลผลเป็นชุดสำหรับที่เก็บขนาดใช้งานจริง ดังนั้นที่เก็บอาจเกินขีดจำกัดที่กำหนดชั่วครู่ก่อนการล้างเมื่อถึงระดับสูงครั้งถัดไปจะเขียนใหม่ให้ลดลง การอ่านจะไม่ตัดแต่งหรือจำกัดรายการระหว่างการเริ่มต้น Gateway โดยจะมีเพียงการเขียนหรือ `openclaw sessions cleanup --enforce` เท่านั้นที่ทำเช่นนั้น และรายการหลังจะบังคับใช้ขีดจำกัดทันที รวมทั้งตัดแต่งอาร์ติแฟกต์ทรานสคริปต์แบบเดิม จุดตรวจ และวิถีการทำงานที่เก่าและไม่มีการอ้างอิง แม้ไม่ได้กำหนดงบประมาณดิสก์ไว้

OpenClaw จะไม่สร้างข้อมูลสำรองหมุนเวียน `sessions.json.bak.*` โดยอัตโนมัติระหว่างการเขียนของ Gateway อีกต่อไป สคีมาปัจจุบันปฏิเสธคีย์ `session.maintenance.rotateBytes` แบบเดิม และ `openclaw doctor --fix` จะนำคีย์ดังกล่าวออกจากการกำหนดค่ารุ่นเก่า

การเปลี่ยนแปลงทรานสคริปต์ใช้คิวเขียนเซสชันสำหรับเป้าหมายทรานสคริปต์ SQLite:

| การตั้งค่า                              | ค่าเริ่มต้น   | การแทนที่ด้วยตัวแปรสภาพแวดล้อม                                     |
| ------------------------------------ | --------- | ------------------------------------------------ |
| `session.writeLock.acquireTimeoutMs` | `60000`   | `OPENCLAW_SESSION_WRITE_LOCK_ACQUIRE_TIMEOUT_MS` |
| `session.writeLock.staleMs`          | `1800000` | `OPENCLAW_SESSION_WRITE_LOCK_STALE_MS`           |
| `session.writeLock.maxHoldMs`        | `300000`  | `OPENCLAW_SESSION_WRITE_LOCK_MAX_HOLD_MS`        |

`acquireTimeoutMs` คือระยะเวลาที่การรอล็อกจะแสดงข้อผิดพลาดเซสชันไม่ว่างก่อนยกเลิก ให้เพิ่มค่านี้เฉพาะเมื่อการเตรียม การล้าง Compaction หรืองานมิเรอร์ทรานสคริปต์ที่ถูกต้องแข่งขันกันนานกว่านั้นบนเครื่องที่ช้า `staleMs` คือระยะเวลาที่ล็อกเดิมจะสามารถถูกเรียกคืนในฐานะล็อกล้าสมัย `maxHoldMs` คือเกณฑ์การปล่อยล็อกของตัวเฝ้าระวังภายในกระบวนการ

### การดาวน์เกรดหลังเปลี่ยนมาใช้ SQLite

คืนค่าอาร์ติแฟกต์ทรานสคริปต์แบบเดิมที่เก็บถาวรก่อนเรียกใช้ OpenClaw
เวอร์ชันเก่าที่ใช้ไฟล์เป็นแบ็กเอนด์:

```bash
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

การย้ายข้อมูลจะคงไฟล์ `sessions.json` แบบเดิมไว้เพื่อการสนับสนุนและ
การย้อนกลับ แต่ไฟล์ JSONL ทรานสคริปต์ที่ใช้งานอยู่ซึ่งถูกนำเข้าไปยัง SQLite จะถูก
เปลี่ยนชื่อเป็น `session-sqlite-import-archive/` รันไทม์รุ่นเก่าที่ใช้ไฟล์เป็นแบ็กเอนด์จะติดตาม
เส้นทาง `sessionFile` ใน `sessions.json` ดังนั้นจึงต้องคืนค่าอาร์ติแฟกต์เหล่านั้น
ก่อนเริ่มต้น การคืนค่าใช้ไฟล์กำกับการย้ายข้อมูล ย้ายเฉพาะอาร์ติแฟกต์ที่เก็บถาวร
ซึ่งมีการบันทึกไว้และเส้นทางเดิมหายไป และคงฐานข้อมูล SQLite ไว้
เพื่อการกู้คืนไปข้างหน้า

เซสชันที่สร้างหลังเปลี่ยนมาใช้ SQLite จะมีเฉพาะใน SQLite และจะไม่ปรากฏต่อ
รันไทม์รุ่นเก่าที่ใช้ไฟล์เป็นแบ็กเอนด์ หากอัปเกรดอีกครั้งหลังจากดาวน์เกรด ให้เรียกใช้
ลำดับการตรวจสอบและตรวจยืนยันของ Doctor อีกครั้ง เพื่อให้ OpenClaw สามารถตรวจยืนยัน
อาร์ติแฟกต์แบบเดิมที่คืนค่าก่อนนำเข้า

## เซสชัน Cron และบันทึกการทำงาน

การทำงาน Cron แบบแยกจะสร้างรายการเซสชัน/ทรานสคริปต์ของตนเองพร้อมระยะเวลาเก็บรักษาเฉพาะ:

- `cron.sessionRetention` (ค่าเริ่มต้น `"24h"`) ตัดเซสชันการทำงาน Cron แบบแยกที่เก่าออกจากที่เก็บ; `false` ปิดใช้งาน
- ประวัติการทำงานเก็บแถวปลายทางล่าสุด 2000 แถวต่องาน Cron แถวที่สูญหายยังคงมีช่วงเวลาล้าง 24 ชั่วโมง

เมื่อ Cron บังคับสร้างเซสชันการทำงานแบบแยกใหม่ ระบบจะล้างข้อมูลรายการเซสชัน `cron:<jobId>` ก่อนหน้าก่อนเขียนแถวใหม่ โดยคงค่ากำหนดที่ปลอดภัย (การตั้งค่าการคิด/เร็ว/ละเอียด/การให้เหตุผล ป้ายกำกับ ชื่อที่แสดง) และการแทนที่โมเดล/การยืนยันตัวตนที่ผู้ใช้เลือกไว้อย่างชัดแจ้ง แต่จะทิ้งบริบทการสนทนารอบข้าง (การกำหนดเส้นทางช่องทาง/กลุ่ม นโยบายการส่ง/คิว การยกระดับ ต้นทาง การผูกกับรันไทม์ ACP) เพื่อให้การทำงานแบบแยกใหม่ไม่สามารถสืบทอดสิทธิ์การจัดส่งหรือรันไทม์ที่ล้าสมัยจากการทำงานเก่า

## คีย์เซสชัน (`sessionKey`)

`sessionKey` ระบุว่ากำลังอยู่ในบักเก็ตการสนทนาใด (การกำหนดเส้นทาง + การแยก) กฎมาตรฐาน: [/concepts/session](/th/concepts/session)

| รูปแบบ                      | ตัวอย่าง                                                     |
| ---------------------------- | ----------------------------------------------------------- |
| แชตหลัก/โดยตรง (ต่อเอเจนต์) | `agent:<agentId>:<mainKey>` (ค่าเริ่มต้น `main`)                |
| กลุ่ม                        | `agent:<agentId>:<channel>:group:<id>`                      |
| ห้อง/ช่องทาง (Discord/Slack) | `agent:<agentId>:<channel>:channel:<id>` หรือ `...:room:<id>` |
| Cron                         | `cron:<job.id>`                                             |
| Webhook                      | `hook:<uuid>` (เว้นแต่ถูกแทนที่)                           |

## รหัสเซสชัน (`sessionId`)

แต่ละ `sessionKey` ชี้ไปยัง `sessionId` ปัจจุบัน (ข้อมูลประจำตัวทรานสคริปต์ SQLite ที่ดำเนินบทสนทนาต่อ) ตรรกะการตัดสินใจอยู่ใน `initSessionState()` ภายใน `src/auto-reply/reply/session.ts`

- **รีเซ็ต** (`/new`, `/reset`) สร้าง `sessionId` ใหม่สำหรับ `sessionKey` นั้น
- **รีเซ็ตรายวัน** (ค่าเริ่มต้นคือ 4:00 AM ตามเวลาท้องถิ่นบนโฮสต์ Gateway) สร้าง `sessionId` ใหม่เมื่อมีข้อความถัดไปหลังขอบเขตการรีเซ็ต
- **การหมดอายุเมื่อไม่มีการใช้งาน** (`session.reset.idleMinutes` หรือ `session.idleMinutes` แบบเดิม) สร้าง `sessionId` ใหม่เมื่อมีข้อความเข้ามาหลังพ้นช่วงไม่มีการใช้งาน หากกำหนดค่าทั้งแบบรายวันและแบบไม่มีการใช้งาน รายการใดหมดอายุก่อนจะมีผล
- **การดำเนินเซสชันต่อเมื่อ Control UI เชื่อมต่อใหม่** รักษาเซสชันที่มองเห็นอยู่ในขณะนั้นไว้สำหรับการส่งหนึ่งครั้งหลังเชื่อมต่อใหม่ เมื่อ Gateway ได้รับ `sessionId` ที่ตรงกันจากไคลเอนต์ UI ของผู้ปฏิบัติงาน นี่เป็นสัญญาณแบบใช้ครั้งเดียว การส่งทั่วไปจากสถานะเก่ายังคงสร้าง `sessionId` ใหม่
- **เหตุการณ์ระบบ** (Heartbeat, การปลุกจาก Cron, การแจ้งเตือน exec, การทำบัญชีของ Gateway) อาจแก้ไขแถวเซสชัน แต่จะไม่ยืดความสดใหม่สำหรับการรีเซ็ตรายวัน/เมื่อไม่มีการใช้งาน เมื่อเปลี่ยนรอบการรีเซ็ต ระบบจะทิ้งการแจ้งเหตุการณ์ระบบที่อยู่ในคิวของเซสชันก่อนหน้า ก่อนสร้างพรอมต์ใหม่
- **นโยบายการ fork จากพาเรนต์** ใช้สาขาที่ใช้งานอยู่ของ OpenClaw เมื่อสร้างเธรดหรือ fork ของเอเจนต์ย่อย หากสาขานั้นมีขนาดใหญ่เกินไป (เกินขีดจำกัดภายในแบบคงที่ ซึ่งปัจจุบันคือ 100K โทเค็น) OpenClaw จะเริ่มต้นตัวลูกด้วยบริบทที่แยกออกมา แทนที่จะล้มเหลวหรือสืบทอดประวัติที่ใช้งานไม่ได้ การกำหนดขนาดเป็นไปโดยอัตโนมัติและกำหนดค่าไม่ได้ โดย `openclaw doctor --fix` จะนำการกำหนดค่า `session.parentForkMaxTokens` แบบเดิมออก
- **การ fork โดยผู้ปฏิบัติงาน**: `sessions.create { parentSessionKey, fork: true }` สร้างเซสชันใหม่ที่ทรานสคริปต์แตกสาขาจากสถานะปัจจุบันของพาเรนต์ (ใช้กลไก fork เดียวกับการสร้างเอเจนต์ย่อย รวมถึงขีดจำกัดขนาดข้างต้น) ระบบจะปฏิเสธการ fork ขณะที่พาเรนต์มีการรันที่ใช้งานอยู่ สืบทอดการเลือกโมเดลของพาเรนต์เว้นแต่จะส่งโมเดลมาอย่างชัดเจน และทำเครื่องหมายตัวลูกเป็น `forkedFromParent` พร้อมตัวนับโทเค็นใหม่

## สคีมาที่เก็บเซสชัน

ที่เก็บรันไทม์เก็บค่า `SessionEntry` ไว้ใน SQLite แยกตามเอเจนต์ ชนิดค่าคือ `SessionEntry` ใน `src/config/sessions.ts` ฟิลด์สำคัญ (ไม่ครบทั้งหมด):

- `sessionId`: รหัสทรานสคริปต์ปัจจุบันที่ใช้ระบุแถวทรานสคริปต์ใน SQLite
- `sessionStartedAt`: การประทับเวลาเริ่มต้นของ `sessionId` ปัจจุบัน การตรวจสอบความสดใหม่สำหรับการรีเซ็ตรายวันใช้ค่านี้ แถวแบบเดิมอาจอนุมานค่านี้จากส่วนหัวเซสชัน JSONL
- `lastInteractionAt`: การประทับเวลาของการโต้ตอบจริงล่าสุดจากผู้ใช้/ช่องทาง การตรวจสอบความสดใหม่สำหรับการรีเซ็ตเมื่อไม่มีการใช้งานใช้ค่านี้ เพื่อไม่ให้เหตุการณ์ Heartbeat, Cron และ exec ทำให้เซสชันยังคงใช้งานอยู่ แถวแบบเดิมที่ไม่มีฟิลด์นี้จะใช้เวลาเริ่มต้นเซสชันที่กู้คืนมาแทน
- `updatedAt`: การประทับเวลาการแก้ไขแถวในที่เก็บครั้งล่าสุด ใช้สำหรับการแสดงรายการ/ตัดทิ้ง/ทำบัญชี ไม่ใช่แหล่งข้อมูลหลักสำหรับความสดใหม่แบบรายวัน/เมื่อไม่มีการใช้งาน
- `archivedAt`: การประทับเวลาการเก็บถาวรที่ไม่บังคับ เซสชันที่เก็บถาวรยังคงอยู่ในที่เก็บพร้อมทรานสคริปต์ครบถ้วน และไม่รวมอยู่ในรายการเซสชันที่ใช้งานอยู่ตามปกติ
- `pinnedAt`: การประทับเวลาการปักหมุดที่ไม่บังคับ เซสชันที่ใช้งานอยู่และถูกปักหมุดจะเรียงก่อนเซสชันที่ไม่ถูกปักหมุด การเก็บเซสชันถาวรจะล้างการปักหมุด
- การทำงานร่วมกันกับเธรด Codex: ทั้งสองฟิลด์เป็นไปตามรูปแบบการจัดการเธรดของ Codex โดยค่าบูลีน `archived`/`pinned` บนสายสัญญาณจะอนุมานจากการประทับเวลาและประทับโดยฝั่งเซิร์ฟเวอร์เสมอ ซึ่งตรงกับความหมายของ `threads.archived_at` ใน Codex และการทำซีเรียลไลซ์แบบ camelCase การประทับเวลาของ OpenClaw ใช้มิลลิวินาทีนับจาก epoch ส่วน Codex ใช้วินาทีนับจาก epoch ดังนั้นบริดจ์จึงแปลงค่าที่จุดเชื่อมต่อ Plugin `codex` ขณะนี้ Codex ยังไม่มี API สำหรับปักหมุด (มีเพียง `thread/archive`/`thread/unarchive`) สถานะการปักหมุดจึงยังคงอยู่ฝั่ง OpenClaw จนกว่าจะมี API ดังกล่าว ซึ่งเมื่อมีแล้ว รูปแบบที่ตรงกันจะทำให้เซสชันที่ผูกไว้สามารถส่งสถานะการปักหมุดไป-กลับได้โดยอัตโนมัติ
- การกำกับดูแลของ Codex แสดงเฉพาะเธรดเนทีฟที่ไม่ได้เก็บถาวร เธรด `idle` หรือ `notLoaded` ภายใน Gateway ซึ่งไม่ทราบสถานะกิจกรรม สามารถเก็บถาวรผ่าน `thread/archive` แบบเนทีฟได้ก็ต่อเมื่อผู้ปฏิบัติงานยืนยันอย่างชัดเจนว่าไม่มีกระบวนการ Codex อื่นเป็นเจ้าของ Plugin จะอ่านสถานะภายในกระบวนการล่าสุดก่อน จากนั้นเธรดจะหายไปจากแค็ตตาล็อก การอ่านดังกล่าวไม่สามารถพิสูจน์ได้ว่ากระบวนการ App Server อื่นไม่ได้ใช้เธรดนี้อยู่ OpenClaw ปฏิเสธการเก็บแถวที่ใช้งานอยู่และแถวข้อผิดพลาดเป็นข้อมูลถาวร และยังไม่รองรับการเก็บถาวรผ่าน Node ที่จับคู่ จนกว่าบริดจ์ของ Node จะสามารถเป็นเจ้าของวงจรชีวิตทั้งหมดของเธรดแบบสตรีมได้ การยกเลิกการเก็บถาวรในไคลเอนต์ Codex แบบเนทีฟทำให้เธรดมีสิทธิ์ปรากฏอีกครั้ง
- `lastReadAt` / `markedUnreadAt`: การประทับเวลาสถานะการอ่านที่ประทับโดยฝั่งเซิร์ฟเวอร์ผ่าน `sessions.patch { unread }` โดย `unread: false` บันทึกการอ่าน (ตั้งค่า `lastReadAt` และล้าง `markedUnreadAt`) ส่วน `unread: true` ทำเครื่องหมายเซสชันว่าไม่ได้อ่านจนกว่าจะมีการอ่านครั้งถัดไป แถวเซสชันแสดงค่าบูลีน `unread` ที่อนุมานได้: ถูกทำเครื่องหมายว่าไม่ได้อ่านอย่างชัดเจน หรือถูกอ่านก่อนกิจกรรมล่าสุด เซสชันที่ไม่เคยถูกทำเครื่องหมายว่าอ่านแล้วจะยังคงเป็น `unread: false` เพื่อไม่ให้การติดตั้งเดิมแสดงสถานะใหม่ขึ้นมาเมื่ออัปเกรด
- `lastActivityAt`: การประทับเวลาของการรันเอเจนต์ที่เสร็จสมบูรณ์ครั้งล่าสุดซึ่งนับเป็นกิจกรรมที่ควรถือว่ายังไม่ได้อ่าน (การรันจากผู้ใช้ ช่องทาง และ Cron) เทิร์นจาก Heartbeat และเหตุการณ์ภายใน รวมถึงแพตช์ข้อมูลเมตา จะไม่อัปเดตค่านี้ โดย `updatedAt` ไม่ใช่สัญญาณกิจกรรม
- `sessionFile`: เครื่องหมายแบบเดิมที่คงไว้เพื่อความเข้ากันได้กับการย้ายข้อมูล/การเก็บถาวร รันไทม์ที่ใช้งานอยู่ใช้ข้อมูลประจำตัวของ SQLite
- `chatType`: `direct | group | room`
- `provider`, `subject`, `room`, `space`, `displayName`: ข้อมูลเมตาสำหรับป้ายกำกับกลุ่ม/ช่องทาง
- ตัวสลับ: `thinkingLevel`, `verboseLevel`, `reasoningLevel`, `elevatedLevel`, `sendPolicy` (การแทนที่รายเซสชัน)
- การเลือกโมเดล: `providerOverride`, `modelOverride`, `authProfileOverride`
- ตัวนับโทเค็น (ใช้ความพยายามสูงสุด/ขึ้นอยู่กับผู้ให้บริการ): `inputTokens`, `outputTokens`, `totalTokens`, `contextTokens`
- `compactionCount`: จำนวนครั้งที่การทำ Compaction อัตโนมัติเสร็จสมบูรณ์สำหรับคีย์เซสชันนี้
- `memoryFlushAt` / `memoryFlushCompactionCount`: การประทับเวลาและจำนวน Compaction ของการฟลัชหน่วยความจำก่อนทำ Compaction ครั้งล่าสุด

Gateway เป็นแหล่งข้อมูลหลัก โดยอาจเขียนใหม่หรือเติมข้อมูลรายการอีกครั้งขณะที่เซสชัน
ทำงานอยู่ สำหรับการติดตั้งแบบเดิมที่ใช้ไฟล์เป็นที่เก็บ ให้ย้ายข้อมูลด้วย
`openclaw doctor --session-sqlite import --session-sqlite-all-agents` แทนการ
แก้ไข `sessions.json` แล้วคาดหวังให้รันไทม์อ่านไฟล์นั้นต่อไป

## โครงสร้างเหตุการณ์ทรานสคริปต์

ทรานสคริปต์ได้รับการจัดการโดยตัวเข้าถึงเซสชันของ OpenClaw และเปิดให้โค้ดรันไทม์เข้าถึงผ่านตัวช่วยที่อิงข้อมูลประจำตัว สตรีมเหตุการณ์เป็นแบบเพิ่มต่อท้ายเท่านั้น:

- รายการแรก: ส่วนหัวเซสชัน - `type: "session"`, `id`, `cwd`, `timestamp`, และ `parentSession` ที่ไม่บังคับ
- จากนั้น: รายการที่มี `id` + `parentId` (โครงสร้างแบบต้นไม้)

ชนิดรายการที่สำคัญ:

- `message`: ข้อความจากผู้ใช้/ผู้ช่วย/toolResult
- `custom_message`: ข้อความที่ส่วนขยายแทรกซึ่ง _เข้าสู่_ บริบทของโมเดล (แสดงผลใน TUI เมื่อเป็น `display: true` และซ่อนทั้งหมดเมื่อเป็น `display: false`)
- `custom`: สถานะส่วนขยายที่ _ไม่เข้าสู่_ บริบทของโมเดล (สำหรับคงสถานะส่วนขยายไว้ข้ามการโหลดใหม่)
- `compaction`: สรุป Compaction ที่คงไว้ พร้อม `firstKeptEntryId` และ `tokensBefore`
- `branch_summary`: สรุปที่คงไว้เมื่อนำทางสาขาของต้นไม้

OpenClaw ตั้งใจไม่ "แก้ไขปรับแต่ง" ทรานสคริปต์ โดย Gateway ใช้ `SessionManager` เพื่ออ่าน/เขียนทรานสคริปต์

## หน้าต่างบริบทเทียบกับโทเค็นที่ติดตาม

แนวคิดที่แตกต่างกันสองอย่าง:

1. **หน้าต่างบริบทของโมเดล**: ขีดจำกัดสูงสุดแบบตายตัวต่อโมเดล (โทเค็นที่โมเดลมองเห็น) มาจากแค็ตตาล็อกโมเดลและสามารถแทนที่ผ่านการกำหนดค่า
2. **ตัวนับในที่เก็บเซสชัน**: สถิติแบบต่อเนื่องที่เขียนลงในแถวเซสชัน (ใช้สำหรับ `/status` และแดชบอร์ด) `contextTokens` เป็นค่าประมาณ/ค่ารายงานขณะรันไทม์ อย่าถือเป็นการรับประกันที่เคร่งครัด

ข้อมูลเพิ่มเติมเกี่ยวกับขีดจำกัด: [/reference/token-use](/th/reference/token-use)

## Compaction คืออะไร

Compaction สรุปการสนทนาส่วนเก่าเป็นรายการ `compaction` ที่คงไว้ในทรานสคริปต์ และรักษาข้อความล่าสุดไว้โดยไม่เปลี่ยนแปลง หลังทำ Compaction เทิร์นในอนาคตจะเห็นสรุป Compaction พร้อมข้อความหลัง `firstKeptEntryId` Compaction เป็นแบบ **คงอยู่ถาวร** ซึ่งต่างจากการตัดเซสชัน โปรดดู [/concepts/session-pruning](/th/concepts/session-pruning)

การแทรกส่วน AGENTS.md กลับเข้าไปอีกครั้งหลังทำ Compaction ต้องเลือกเปิดใช้ผ่าน `agents.defaults.compaction.postCompactionSections` เมื่อไม่ได้ตั้งค่าหรือเป็น `[]` OpenClaw จะไม่เพิ่มข้อความคัดย่อจาก AGENTS.md ต่อท้ายสรุป Compaction

### ขอบเขตส่วนและการจับคู่เครื่องมือ

เมื่อแบ่งทรานสคริปต์ยาวเป็นส่วนสำหรับ Compaction OpenClaw จะเก็บการเรียกเครื่องมือของผู้ช่วยไว้คู่กับรายการ `toolResult` ที่ตรงกัน:

- หากการแบ่งตามสัดส่วนโทเค็นจะตกอยู่ระหว่างการเรียกเครื่องมือกับผลลัพธ์ OpenClaw จะเลื่อนขอบเขตไปยังข้อความการเรียกเครื่องมือของผู้ช่วย แทนการแยกคู่ออกจากกัน
- หากบล็อกผลลัพธ์เครื่องมือท้ายสุดอาจทำให้ส่วนเกินเป้าหมาย OpenClaw จะรักษาบล็อกเครื่องมือที่รอดำเนินการนั้นไว้ และคงส่วนท้ายที่ยังไม่ได้สรุปไว้อย่างครบถ้วน
- บล็อกการเรียกเครื่องมือที่ถูกยกเลิก/เกิดข้อผิดพลาดจะไม่ทำให้จุดแบ่งที่รอดำเนินการยังคงเปิดอยู่

## การทำ Compaction อัตโนมัติเกิดขึ้นเมื่อใด

ทริกเกอร์สองรายการในเอเจนต์ OpenClaw แบบฝังตัว:

1. **การกู้คืนจากข้อมูลล้น**: โมเดลส่งคืนข้อผิดพลาดบริบทล้น (`request_too_large`, `context length exceeded`, `input exceeds the maximum number of tokens`, `input token count exceeds the maximum number of input tokens`, `input is too long for the model`, `ollama error: context length exceeded` และรูปแบบอื่นที่ขึ้นกับผู้ให้บริการ) ให้ทำ Compaction แล้วลองใหม่ เมื่อผู้ให้บริการรายงานจำนวนโทเค็นที่พยายามใช้ OpenClaw จะส่งต่อจำนวนที่สังเกตได้นั้นไปยัง Compaction สำหรับการกู้คืนจากข้อมูลล้น หากผู้ให้บริการยืนยันว่าข้อมูลล้นแต่ไม่แสดงจำนวนที่แยกวิเคราะห์ได้ OpenClaw จะส่งจำนวนสังเคราะห์ที่เกินงบประมาณเพียงเล็กน้อยไปยังเอนจิน Compaction และการวินิจฉัย หากการกู้คืนจากข้อมูลล้นยังคงล้มเหลว OpenClaw จะแสดงคำแนะนำอย่างชัดเจนและรักษาการแมปเซสชันปัจจุบันไว้ แทนการหมุนไปใช้รหัสเซสชันใหม่โดยไม่แจ้ง ให้ลองส่งข้อความอีกครั้ง เรียกใช้ `/compact` หรือเรียกใช้ `/new`
2. **การบำรุงรักษาตามเกณฑ์**: หลังจากเทิร์นสำเร็จ เมื่อ `contextTokens > contextWindow - reserveTokens` โดย `contextWindow` คือหน้าต่างบริบทของโมเดล และ `reserveTokens` คือพื้นที่สำรองสำหรับพรอมต์รวมถึงเอาต์พุตถัดไปของโมเดล

ตัวป้องกันเพิ่มเติมอีกสองรายการทำงานนอกทริกเกอร์ทั้งสองนี้:

- **Compaction ภายในเครื่องก่อนเริ่มงาน**: ตั้งค่า `agents.defaults.compaction.maxActiveTranscriptBytes` (ไบต์หรือสตริง เช่น `"20mb"`) เพื่อทริกเกอร์ Compaction ภายในเครื่องก่อนเปิดการรันครั้งถัดไป เมื่อทรานสคริปต์ที่ใช้งานอยู่มีขนาดถึงค่าดังกล่าว นี่เป็นตัวป้องกันด้านขนาดสำหรับต้นทุนการเปิดใหม่ภายในเครื่อง ไม่ใช่การเก็บถาวรแบบดิบ โดย Compaction เชิงความหมายตามปกติยังคงทำงาน และต้องใช้ `truncateAfterCompaction` เพื่อให้สรุปที่ผ่าน Compaction กลายเป็นทรานสคริปต์ตัวสืบทอดใหม่
- **การตรวจสอบล่วงหน้าระหว่างเทิร์น**: ตั้งค่า `agents.defaults.compaction.midTurnPrecheck.enabled: true` (ค่าเริ่มต้น `false`) เพื่อเพิ่มตัวป้องกันลูปเครื่องมือ หลังเพิ่มผลลัพธ์เครื่องมือและก่อนเรียกโมเดลครั้งถัดไป OpenClaw จะประเมินแรงกดดันของพรอมต์โดยใช้ตรรกะงบประมาณก่อนเริ่มงานเดียวกับที่ใช้ตอนเริ่มเทิร์น หากบริบทไม่พอดีอีกต่อไป ตัวป้องกันจะไม่ทำ Compaction ในตำแหน่งนั้น แต่จะส่งสัญญาณการตรวจสอบล่วงหน้าระหว่างเทิร์นแบบมีโครงสร้าง หยุดการส่งพรอมต์ปัจจุบัน และให้ลูปการรันชั้นนอกใช้เส้นทางการกู้คืนที่มีอยู่ (ตัดผลลัพธ์เครื่องมือที่มีขนาดใหญ่เกินไปเมื่อเพียงพอ หรือทริกเกอร์โหมด Compaction ที่กำหนดค่าไว้แล้วลองใหม่) ใช้ได้กับโหมด Compaction ทั้ง `default` และ `safeguard` รวมถึง Compaction ป้องกันที่รองรับโดยผู้ให้บริการ ทำงานแยกจาก `maxActiveTranscriptBytes`: ตัวป้องกันขนาดไบต์ทำงานก่อนเปิดเทิร์น ส่วนการตรวจสอบล่วงหน้าระหว่างเทิร์นทำงานภายหลัง หลังเพิ่มผลลัพธ์เครื่องมือใหม่แล้ว

## การตั้งค่า Compaction

```json5
{
  agents: {
    defaults: {
      compaction: {
        enabled: true,
        reserveTokens: 16384,
        keepRecentTokens: 20000,
      },
    },
  },
}
```

OpenClaw ยังบังคับใช้ค่าความปลอดภัยขั้นต่ำสำหรับการรันแบบฝังด้วย: หาก `compaction.reserveTokens` ต่ำกว่า `reserveTokensFloor` (ค่าเริ่มต้น `20000`) OpenClaw จะเพิ่มค่าให้ถึงขั้นต่ำ ตั้งค่า `agents.defaults.compaction.reserveTokensFloor: 0` เพื่อปิดใช้ค่าขั้นต่ำนี้ เมื่อทราบขนาดหน้าต่างบริบทของโมเดลที่ใช้งานอยู่ ระบบจะจำกัดทั้งค่าขั้นต่ำและค่าสำรองที่มีผลจริงขั้นสุดท้าย เพื่อไม่ให้ค่าสำรองใช้โควตาพรอมต์ทั้งหมด วิธีนี้ป้องกันไม่ให้โมเดลที่มีบริบทขนาดเล็ก (เช่น โมเดลภายในเครื่องขนาด 16K โทเค็น) เข้าสู่ Compaction ตั้งแต่โทเค็นแรก หากไม่ทราบขนาดหน้าต่างบริบท งบประมาณสำรองที่กำหนดค่าไว้และที่ใช้อยู่ในปัจจุบันจะไม่ถูกจำกัด เหตุผลที่ต้องมีค่าขั้นต่ำ: เพื่อเหลือพื้นที่ส่วนเกินเพียงพอสำหรับ "งานดูแลระบบ" แบบหลายรอบ (เช่น การฟลัชหน่วยความจำด้านล่าง) ก่อนที่ Compaction จะหลีกเลี่ยงไม่ได้ การทำงาน: `applyAgentCompactionSettingsFromConfig()` ใน `src/agents/agent-settings.ts` ซึ่งเรียกจากเส้นทางการตั้งค่ารอบของตัวรันแบบฝังและ Compaction

การใช้ `/compact` ด้วยตนเองจะใช้ค่า `agents.defaults.compaction.keepRecentTokens` ที่ระบุไว้อย่างชัดเจน และคงจุดตัดส่วนท้ายล่าสุดของรันไทม์ไว้ หากไม่ได้ระบุงบประมาณที่จะเก็บไว้ Compaction แบบดำเนินการด้วยตนเองจะเป็นจุดตรวจสอบแบบตายตัว และบริบทที่สร้างใหม่จะเริ่มจากข้อมูลสรุปใหม่

เมื่อเปิดใช้ `truncateAfterCompaction` OpenClaw จะหมุนเวียนทรานสคริปต์ที่ใช้งานอยู่ไปยังทรานสคริปต์ลำดับถัดไปที่ผ่านการ Compaction หลังจาก Compaction การดำเนินการกับจุดตรวจสอบสำหรับแตกแขนง/คืนค่าจะใช้ทรานสคริปต์ลำดับถัดไปที่ผ่านการ Compaction นั้น ส่วนไฟล์จุดตรวจสอบรุ่นเก่าก่อน Compaction จะยังคงอ่านได้ขณะที่มีการอ้างอิงอยู่

## ผู้ให้บริการ Compaction แบบเสียบใช้ได้

Plugin ลงทะเบียนผู้ให้บริการ Compaction ผ่าน `registerCompactionProvider()` บน API ของ Plugin เมื่อกำหนด `agents.defaults.compaction.provider` เป็นรหัสผู้ให้บริการที่ลงทะเบียนแล้ว ส่วนขยายป้องกันจะมอบหมายการสรุปให้ผู้ให้บริการนั้นแทนไปป์ไลน์ `summarizeInStages` ในตัว

- `provider`: รหัสของ Plugin ผู้ให้บริการ Compaction ที่ลงทะเบียนไว้ เว้นว่างไว้เพื่อใช้การสรุปด้วย LLM เริ่มต้น การตั้งค่า `provider` จะบังคับใช้ `mode: "safeguard"`
- ผู้ให้บริการจะได้รับคำสั่ง Compaction และนโยบายการรักษาตัวระบุแบบเดียวกับเส้นทางในตัว และระบบป้องกันจะยังคงรักษาบริบทส่วนต่อท้ายของรอบล่าสุดและรอบที่ถูกแบ่งไว้หลังผลลัพธ์จากผู้ให้บริการ
- การสรุปของระบบป้องกันในตัวจะกลั่นข้อมูลสรุปก่อนหน้าใหม่ร่วมกับข้อความใหม่ แทนที่จะเก็บข้อมูลสรุปก่อนหน้าทั้งหมดไว้ตามข้อความเดิมทุกประการ
- โหมดป้องกันจะเปิดใช้การตรวจสอบคุณภาพข้อมูลสรุปโดยค่าเริ่มต้น ตั้งค่า `qualityGuard.enabled: false` เพื่อข้ามพฤติกรรมลองใหม่เมื่อผลลัพธ์มีรูปแบบไม่ถูกต้อง
- หากผู้ให้บริการล้มเหลวหรือส่งคืนผลลัพธ์ว่าง OpenClaw จะย้อนกลับไปใช้การสรุปด้วย LLM ในตัวโดยอัตโนมัติ สัญญาณยกเลิก/หมดเวลาที่ผู้เรียกเป็นผู้เรียกใช้อย่างชัดเจนจะถูกโยนซ้ำแทนที่จะถูกกลืนทิ้ง เพื่อให้ระบบเคารพการยกเลิกเสมอ

แหล่งที่มา: `src/plugins/compaction-provider.ts`, `src/agents/agent-hooks/compaction-safeguard.ts`

## ส่วนที่ผู้ใช้มองเห็น

- `/status` ในเซสชันแชตใดก็ได้
- `openclaw status` (CLI)
- `openclaw sessions` / `openclaw sessions --json`
- บันทึกของ Gateway (`pnpm gateway:watch` หรือ `openclaw logs --follow`): `embedded run auto-compaction start` + `complete`
- โหมดแบบละเอียด: `🧹 Auto-compaction complete` พร้อมจำนวนครั้งของ Compaction

## งานดูแลระบบแบบเงียบ (`NO_REPLY`)

OpenClaw รองรับรอบการทำงานแบบ "เงียบ" สำหรับงานเบื้องหลังที่ไม่ควรแสดงผลลัพธ์ระหว่างทางให้ผู้ใช้เห็น

- ผู้ช่วยเริ่มผลลัพธ์ด้วยโทเค็นเงียบที่ตรงกันทุกประการ `NO_REPLY` / `no_reply` เพื่อหมายความว่า "ไม่ต้องส่งคำตอบให้ผู้ใช้" OpenClaw จะลบ/ระงับข้อความนี้ในชั้นการนำส่ง
- การระงับโทเค็นเงียบที่ตรงกันทุกประการไม่คำนึงถึงตัวพิมพ์เล็กและตัวพิมพ์ใหญ่: ทั้ง `NO_REPLY` และ `no_reply` จะถือว่าใช้ได้เมื่อเพย์โหลดทั้งหมดมีเพียงโทเค็นเงียบ
- ตั้งแต่ `2026.1.10` เป็นต้นมา OpenClaw จะระงับการสตรีมฉบับร่าง/สถานะกำลังพิมพ์ด้วยเมื่อชิ้นส่วนบางส่วนเริ่มต้นด้วย `NO_REPLY` เพื่อไม่ให้การดำเนินการแบบเงียบเผยผลลัพธ์บางส่วนระหว่างรอบ
- สิ่งนี้มีไว้สำหรับรอบเบื้องหลัง/ไม่ส่งผลลัพธ์อย่างแท้จริงเท่านั้น ไม่ใช่ทางลัดสำหรับคำขอทั่วไปจากผู้ใช้ที่ต้องดำเนินการ

## การฟลัชหน่วยความจำก่อน Compaction

ก่อนเกิด Compaction อัตโนมัติ OpenClaw สามารถเรียกใช้รอบแบบเอเจนต์ที่ทำงานเงียบเพื่อเขียนสถานะถาวรลงดิสก์ (เช่น `memory/YYYY-MM-DD.md` ในพื้นที่ทำงานของเอเจนต์) เพื่อไม่ให้ Compaction ลบบริบทสำคัญ ระบบจะตรวจสอบการใช้บริบทของเซสชัน และเมื่อการใช้เกินเกณฑ์แบบผ่อนปรนที่ต่ำกว่าเกณฑ์ Compaction ระบบจะส่งคำสั่ง "เขียนหน่วยความจำตอนนี้" แบบเงียบโดยใช้โทเค็นเงียบที่ตรงกันทุกประการ `NO_REPLY` / `no_reply` เพื่อไม่ให้ผู้ใช้เห็นสิ่งใด

การกำหนดค่า (`agents.defaults.compaction.memoryFlush`) ดูข้อมูลอ้างอิงฉบับเต็มที่ [/gateway/config-agents](/th/gateway/config-agents#agentsdefaultscompaction):

| คีย์                         | ค่าเริ่มต้น          | หมายเหตุ                                                                                                                                  |
| --------------------------- | ---------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`                   | `true`           |                                                                                                                                        |
| `model`                     | ไม่ได้ตั้งค่า            | การแทนที่ผู้ให้บริการ/โมเดลอย่างเจาะจงสำหรับรอบการฟลัชเท่านั้น เช่น `ollama/qwen3:8b`                                                   |
| `softThresholdTokens`       | `4000`           | ระยะห่างต่ำกว่าเกณฑ์ Compaction ที่กระตุ้นให้เกิดการฟลัช                                                                               |
| `forceFlushTranscriptBytes` | ไม่ได้ตั้งค่า (ปิดใช้) | บังคับให้ฟลัชเมื่อไฟล์ทรานสคริปต์มีขนาดถึงจำนวนไบต์นี้ (หรือสตริง เช่น `"2mb"`) แม้ว่าตัวนับโทเค็นจะล้าสมัย โดย `0` จะปิดใช้ |
| `prompt`                    | ในตัว         | ข้อความผู้ใช้สำหรับรอบการฟลัช                                                                                                        |
| `systemPrompt`              | ในตัว         | พรอมต์ระบบเพิ่มเติมที่ต่อท้ายสำหรับรอบการฟลัช                                                                                        |

หมายเหตุ:

- พรอมต์/พรอมต์ระบบเริ่มต้นมีคำแนะนำ `NO_REPLY` เพื่อระงับการนำส่ง
- เมื่อตั้งค่า `model` รอบการฟลัชจะใช้โมเดลนั้นโดยไม่สืบทอดสายโซ่สำรองของเซสชันที่ใช้งานอยู่ เพื่อไม่ให้งานดูแลระบบเฉพาะภายในเครื่องย้อนกลับไปใช้โมเดลสนทนาแบบเสียค่าใช้จ่ายโดยไม่แจ้งให้ทราบเมื่อล้มเหลว
- การฟลัชจะทำงานหนึ่งครั้งต่อรอบ Compaction (ติดตามในแถวของเซสชัน)
- การฟลัชจะทำงานเฉพาะเซสชัน OpenClaw แบบฝัง ส่วนแบ็กเอนด์ CLI และรอบ Heartbeat จะข้ามขั้นตอนนี้
- ระบบจะข้ามการฟลัชเมื่อพื้นที่ทำงานของเซสชันเป็นแบบอ่านอย่างเดียว (`workspaceAccess: "ro"` หรือ `"none"`)
- ดู [หน่วยความจำ](/th/concepts/memory) สำหรับโครงสร้างไฟล์ในพื้นที่ทำงานและรูปแบบการเขียน

OpenClaw เปิดเผยฮุก `session_before_compact` ใน API ส่วนขยาย แต่ตรรกะการฟลัชด้านบนอยู่ฝั่ง Gateway (`src/auto-reply/reply/memory-flush.ts`, `src/auto-reply/reply/agent-runner-memory.ts`) ไม่ได้อยู่บนฮุกดังกล่าว

## รายการตรวจสอบการแก้ไขปัญหา

- **คีย์เซสชันไม่ถูกต้องหรือไม่** เริ่มจาก [/concepts/session](/th/concepts/session) และตรวจสอบ `sessionKey` ใน `/status`
- **ที่เก็บข้อมูลกับทรานสคริปต์ไม่ตรงกันหรือไม่** ตรวจสอบโฮสต์ Gateway และเส้นทางที่เก็บข้อมูลจาก `openclaw status`
- **เกิด Compaction ถี่เกินไปหรือไม่** ตรวจสอบขนาดหน้าต่างบริบทของโมเดล (หากเล็กเกินไปจะบังคับให้เกิด Compaction บ่อยครั้ง), `reserveTokens` (หากสูงเกินไปสำหรับหน้าต่างของโมเดลจะทำให้เกิด Compaction เร็วขึ้น) และผลลัพธ์จากเครื่องมือที่มีขนาดใหญ่เกินไป (ปรับแต่งการตัดแต่งเซสชัน)
- **ทุกพรอมต์ดูเหมือนจะล้นเมื่อใช้โมเดลภายในเครื่องขนาดเล็กหรือไม่** ตรวจสอบว่าผู้ให้บริการรายงานขนาดหน้าต่างบริบทของโมเดลอย่างถูกต้อง OpenClaw สามารถจำกัดค่าสำรองที่มีผลจริงได้เฉพาะเมื่อทราบขนาดหน้าต่างดังกล่าว
- **รอบแบบเงียบมีข้อมูลรั่วไหลหรือไม่** ตรวจสอบว่าคำตอบเริ่มต้นด้วยโทเค็นเงียบที่ตรงกันทุกประการ `NO_REPLY` (ไม่คำนึงถึงตัวพิมพ์เล็กและตัวพิมพ์ใหญ่) และกำลังใช้บิลด์ที่มีการแก้ไขการระงับการสตรีม (`2026.1.10`+)
