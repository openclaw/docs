OpenClaw จัดการเซสชันแบบครบวงจรในพื้นที่เหล่านี้:

- **การกำหนดเส้นทางเซสชัน** (วิธีที่ข้อความขาเข้าถูกแมปกับ `sessionKey`)
- **ที่เก็บเซสชัน** (`sessions.json`) และสิ่งที่มันติดตาม
- **การคงอยู่ของทรานสคริปต์** (`*.jsonl`) และโครงสร้างของมัน
- **สุขอนามัยของทรานสคริปต์** (การแก้ไขเฉพาะผู้ให้บริการก่อนการรัน)
- **ขีดจำกัดบริบท** (หน้าต่างบริบทเทียบกับโทเคนที่ติดตาม)
- **Compaction** (การ Compaction แบบแมนนวลและอัตโนมัติ) และตำแหน่งสำหรับเชื่อมงานก่อน Compaction
- **งานดูแลเบื้องหลังแบบเงียบ** (การเขียนหน่วยความจำที่ไม่ควรสร้างเอาต์พุตที่ผู้ใช้มองเห็น)

## แหล่งข้อมูลจริง: Gateway

OpenClaw ถูกออกแบบรอบ **กระบวนการ Gateway** เพียงหนึ่งเดียวที่เป็นเจ้าของสถานะเซสชัน

- UI (แอป macOS, Control UI บนเว็บ, TUI) ควรสอบถาม Gateway เพื่อดูรายการเซสชันและจำนวนโทเคน
- ในโหมดรีโมต ไฟล์เซสชันอยู่บนโฮสต์รีโมต; “การตรวจไฟล์ Mac ในเครื่องของคุณ” จะไม่สะท้อนสิ่งที่ Gateway กำลังใช้

## เลเยอร์การคงอยู่สองชั้น

OpenClaw คงอยู่เซสชันไว้ในสองเลเยอร์:

1. **ที่เก็บเซสชัน (`sessions.json`)**
   - แมปคีย์/ค่า: `sessionKey -> SessionEntry`
   - ขนาดเล็ก เปลี่ยนแปลงได้ แก้ไขได้อย่างปลอดภัย (หรือลบรายการได้)
   - ติดตามเมทาดาทาของเซสชัน (รหัสเซสชันปัจจุบัน กิจกรรมล่าสุด ตัวสลับ ตัวนับโทเคน ฯลฯ)

2. **ทรานสคริปต์ (`<sessionId>.jsonl`)**
   - ทรานสคริปต์แบบต่อท้ายเท่านั้นพร้อมโครงสร้างแบบต้นไม้ (รายการมี `id` + `parentId`)
   - เก็บบทสนทนาจริง + การเรียกเครื่องมือ + สรุป Compaction
   - ใช้สร้างบริบทของโมเดลใหม่สำหรับเทิร์นในอนาคต
   - จุดตรวจดีบักก่อน Compaction ขนาดใหญ่จะถูกข้ามเมื่อทรานสคริปต์ที่ใช้งานอยู่
     เกินเพดานขนาดจุดตรวจแล้ว เพื่อหลีกเลี่ยงการคัดลอกเป็นไฟล์
     `.checkpoint.*.jsonl` ขนาดใหญ่อีกชุด

## ตำแหน่งบนดิสก์

ต่อเอเจนต์ บนโฮสต์ Gateway:

- ที่เก็บ: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- ทรานสคริปต์: `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
  - เซสชันหัวข้อ Telegram: `.../<sessionId>-topic-<threadId>.jsonl`

OpenClaw แก้ตำแหน่งเหล่านี้ผ่าน `src/config/sessions.ts`

## การบำรุงรักษาที่เก็บและการควบคุมดิสก์

การคงอยู่ของเซสชันมีการควบคุมการบำรุงรักษาอัตโนมัติ (`session.maintenance`) สำหรับ `sessions.json`, อาร์ติแฟกต์ทรานสคริปต์ และไฟล์ข้างเคียงของ trajectory:

- `mode`: `warn` (ค่าเริ่มต้น) หรือ `enforce`
- `pruneAfter`: จุดตัดอายุของรายการที่เก่า (ค่าเริ่มต้น `30d`)
- `maxEntries`: เพดานจำนวนรายการใน `sessions.json` (ค่าเริ่มต้น `500`)
- `resetArchiveRetention`: ระยะเวลาเก็บถาวรสำหรับอาร์ไคฟ์ทรานสคริปต์ `*.reset.<timestamp>` (ค่าเริ่มต้น: เหมือน `pruneAfter`; `false` ปิดการล้างข้อมูล)
- `maxDiskBytes`: งบประมาณไดเรกทอรีเซสชันแบบไม่บังคับ
- `highWaterBytes`: เป้าหมายแบบไม่บังคับหลังการล้างข้อมูล (ค่าเริ่มต้น `80%` ของ `maxDiskBytes`)

การเขียนตามปกติของ Gateway จะรวมชุดการล้างข้อมูล `maxEntries` สำหรับเพดานขนาดระดับโปรดักชัน ดังนั้นที่เก็บอาจเกินเพดานที่กำหนดไว้ชั่วครู่ก่อนที่การล้างข้อมูลตาม high-water ครั้งถัดไปจะเขียนลดกลับลงมา `openclaw sessions cleanup --enforce` ยังคงใช้เพดานที่กำหนดไว้ทันที

OpenClaw ไม่สร้างแบ็กอัปหมุนเวียน `sessions.json.bak.*` อัตโนมัติระหว่างการเขียนของ Gateway อีกต่อไป คีย์เดิม `session.maintenance.rotateBytes` จะถูกละเว้น และ `openclaw doctor --fix` จะลบออกจากคอนฟิกเก่า

ลำดับการบังคับใช้สำหรับการล้างข้อมูลงบประมาณดิสก์ (`mode: "enforce"`):

1. ลบอาร์ติแฟกต์ที่เก็บถาวรเก่าที่สุด ทรานสคริปต์กำพร้า หรือ trajectory กำพร้าก่อน
2. หากยังเกินเป้าหมาย ให้ขับรายการเซสชันเก่าที่สุดและไฟล์ทรานสคริปต์/trajectory ของรายการนั้นออก
3. ทำต่อจนกว่าการใช้งานจะอยู่ที่หรือต่ำกว่า `highWaterBytes`

ใน `mode: "warn"` OpenClaw จะรายงานการขับออกที่อาจเกิดขึ้น แต่จะไม่แก้ไขที่เก็บ/ไฟล์

รันการบำรุงรักษาตามต้องการ:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

## เซสชัน Cron และบันทึกการรัน

การรัน Cron แบบแยกยังสร้างรายการเซสชัน/ทรานสคริปต์ด้วย และมีการควบคุมการเก็บรักษาเฉพาะ:

- `cron.sessionRetention` (ค่าเริ่มต้น `24h`) ตัดแต่งเซสชันการรัน Cron แบบแยกเก่าออกจากที่เก็บเซสชัน (`false` ปิดใช้งาน)
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` ตัดแต่งไฟล์ `~/.openclaw/cron/runs/<jobId>.jsonl` (ค่าเริ่มต้น: `2_000_000` ไบต์และ `2000` บรรทัด)

เมื่อ Cron บังคับสร้างเซสชันการรันแบบแยกใหม่ มันจะทำความสะอาดรายการเซสชัน
`cron:<jobId>` ก่อนหน้าก่อนเขียนแถวใหม่ มันนำค่ากำหนดที่ปลอดภัยติดไปด้วย
เช่น การตั้งค่า thinking/fast/verbose, ป้ายกำกับ และการแทนที่โมเดล/auth
ที่ผู้ใช้เลือกอย่างชัดเจน มันตัดบริบทบทสนทนาแวดล้อมทิ้ง เช่น การกำหนดเส้นทาง
channel/group, นโยบายการส่งหรือคิว, การยกระดับ, ต้นทาง และการผูก ACP
runtime เพื่อให้การรันแบบแยกใหม่ไม่สามารถสืบทอดการส่งหรืออำนาจ runtime
ที่ค้างจากการรันเก่าได้

## คีย์เซสชัน (`sessionKey`)

`sessionKey` ระบุว่า _คุณอยู่ในกลุ่มบทสนทนาใด_ (การกำหนดเส้นทาง + การแยก)

รูปแบบทั่วไป:

- แชตหลัก/โดยตรง (ต่อเอเจนต์): `agent:<agentId>:<mainKey>` (ค่าเริ่มต้น `main`)
- กลุ่ม: `agent:<agentId>:<channel>:group:<id>`
- ห้อง/channel (Discord/Slack): `agent:<agentId>:<channel>:channel:<id>` หรือ `...:room:<id>`
- Cron: `cron:<job.id>`
- Webhook: `hook:<uuid>` (เว้นแต่ถูกแทนที่)

กฎมาตรฐานมีเอกสารที่ [/concepts/session](/th/concepts/session)

---

## รหัสเซสชัน (`sessionId`)

แต่ละ `sessionKey` ชี้ไปยัง `sessionId` ปัจจุบัน (ไฟล์ทรานสคริปต์ที่ดำเนินบทสนทนาต่อ)

หลักทั่วไป:

- **รีเซ็ต** (`/new`, `/reset`) สร้าง `sessionId` ใหม่สำหรับ `sessionKey` นั้น
- **รีเซ็ตรายวัน** (ค่าเริ่มต้น 4:00 น. เวลาท้องถิ่นบนโฮสต์ Gateway) สร้าง `sessionId` ใหม่ในข้อความถัดไปหลังขอบเขตการรีเซ็ต
- **หมดอายุจากการว่าง** (`session.reset.idleMinutes` หรือเดิม `session.idleMinutes`) สร้าง `sessionId` ใหม่เมื่อมีข้อความมาถึงหลังช่วงเวลาว่าง เมื่อกำหนดค่าทั้งรายวัน + ว่างไว้ ตัวที่หมดอายุก่อนจะมีผล
- **เหตุการณ์ระบบ** (Heartbeat, การปลุก Cron, การแจ้งเตือน exec, งานบันทึกบัญชีของ Gateway) อาจแก้ไขแถวเซสชัน แต่จะไม่ต่ออายุความสดของการรีเซ็ตรายวัน/ว่าง การเปลี่ยนรอบรีเซ็ตจะทิ้งประกาศเหตุการณ์ระบบที่อยู่ในคิวสำหรับเซสชันก่อนหน้า ก่อนสร้างพรอมป์ใหม่
- **การ์ดป้องกันการ fork จาก parent ของเธรด** (`session.parentForkMaxTokens`, ค่าเริ่มต้น `100000`) ข้ามการ fork ทรานสคริปต์ parent เมื่อเซสชัน parent มีขนาดใหญ่เกินไปแล้ว; เธรดใหม่เริ่มใหม่ ตั้งค่า `0` เพื่อปิดใช้งาน

รายละเอียดการใช้งาน: การตัดสินใจเกิดขึ้นใน `initSessionState()` ใน `src/auto-reply/reply/session.ts`

## สคีมาที่เก็บเซสชัน (`sessions.json`)

ชนิดค่าของที่เก็บคือ `SessionEntry` ใน `src/config/sessions.ts`

ฟิลด์สำคัญ (ไม่ครบทั้งหมด):

- `sessionId`: รหัสทรานสคริปต์ปัจจุบัน (ชื่อไฟล์ได้มาจากค่านี้ เว้นแต่ตั้งค่า `sessionFile`)
- `sessionStartedAt`: ประทับเวลาเริ่มต้นสำหรับ `sessionId` ปัจจุบัน; ความสดของการรีเซ็ตรายวัน
  ใช้ค่านี้ แถวเดิมอาจได้ค่านี้มาจากส่วนหัวเซสชัน JSONL
- `lastInteractionAt`: ประทับเวลาการโต้ตอบจริงล่าสุดของผู้ใช้/channel; ความสดของการรีเซ็ตจากการว่าง
  ใช้ค่านี้ เพื่อให้ Heartbeat, Cron และเหตุการณ์ exec ไม่ทำให้เซสชัน
  ยังมีชีวิตอยู่ แถวเดิมที่ไม่มีฟิลด์นี้จะ fallback ไปใช้เวลาเริ่มต้นเซสชันที่กู้คืนได้
  สำหรับความสดจากการว่าง
- `updatedAt`: ประทับเวลาการแก้ไขแถวที่เก็บล่าสุด ใช้สำหรับการแสดงรายการ การตัดแต่ง และ
  งานบันทึกบัญชี ค่านี้ไม่ใช่อำนาจสำหรับความสดของการรีเซ็ตรายวัน/ว่าง
- `sessionFile`: การแทนที่เส้นทางทรานสคริปต์แบบชัดเจนที่ไม่บังคับ
- `chatType`: `direct | group | room` (ช่วย UI และนโยบายการส่ง)
- `provider`, `subject`, `room`, `space`, `displayName`: เมทาดาทาสำหรับการติดป้ายกลุ่ม/channel
- ตัวสลับ:
  - `thinkingLevel`, `verboseLevel`, `reasoningLevel`, `elevatedLevel`
  - `sendPolicy` (การแทนที่ต่อเซสชัน)
- การเลือกโมเดล:
  - `providerOverride`, `modelOverride`, `authProfileOverride`
- ตัวนับโทเคน (พยายามดีที่สุด / ขึ้นกับผู้ให้บริการ):
  - `inputTokens`, `outputTokens`, `totalTokens`, `contextTokens`
- `compactionCount`: จำนวนครั้งที่ auto-compaction เสร็จสมบูรณ์สำหรับคีย์เซสชันนี้
- `memoryFlushAt`: ประทับเวลาสำหรับการ flush หน่วยความจำก่อน Compaction ครั้งล่าสุด
- `memoryFlushCompactionCount`: จำนวน Compaction เมื่อการ flush ครั้งล่าสุดรัน

ที่เก็บแก้ไขได้อย่างปลอดภัย แต่ Gateway เป็นอำนาจหลัก: มันอาจเขียนใหม่หรือเติมรายการกลับระหว่างที่เซสชันรัน

## โครงสร้างทรานสคริปต์ (`*.jsonl`)

ทรานสคริปต์ถูกจัดการโดย `SessionManager` ของ `@mariozechner/pi-coding-agent`

ไฟล์เป็น JSONL:

- บรรทัดแรก: ส่วนหัวเซสชัน (`type: "session"`, มี `id`, `cwd`, `timestamp`, `parentSession` แบบไม่บังคับ)
- จากนั้น: รายการเซสชันที่มี `id` + `parentId` (ต้นไม้)

ชนิดรายการที่น่าสังเกต:

- `message`: ข้อความ user/assistant/toolResult
- `custom_message`: ข้อความที่ส่วนขยายแทรกเข้ามาและ _เข้า_ บริบทโมเดล (สามารถซ่อนจาก UI ได้)
- `custom`: สถานะส่วนขยายที่ _ไม่เข้า_ บริบทโมเดล
- `compaction`: สรุป Compaction ที่คงอยู่พร้อม `firstKeptEntryId` และ `tokensBefore`
- `branch_summary`: สรุปที่คงอยู่เมื่อไปยังสาขาต้นไม้

OpenClaw ตั้งใจ **ไม่** “แก้ไข” ทรานสคริปต์; Gateway ใช้ `SessionManager` เพื่ออ่าน/เขียนมัน

## หน้าต่างบริบทเทียบกับโทเคนที่ติดตาม

มีสองแนวคิดที่ต่างกันและสำคัญ:

1. **หน้าต่างบริบทของโมเดล**: เพดานตายตัวต่อโมเดล (โทเคนที่โมเดลมองเห็น)
2. **ตัวนับในที่เก็บเซสชัน**: สถิติแบบต่อเนื่องที่เขียนลง `sessions.json` (ใช้สำหรับ /status และแดชบอร์ด)

หากคุณกำลังปรับขีดจำกัด:

- หน้าต่างบริบทมาจากแคตตาล็อกโมเดล (และสามารถแทนที่ผ่านคอนฟิกได้)
- `contextTokens` ในที่เก็บเป็นค่าประมาณ/ค่ารายงานขณะรัน; อย่าถือว่าเป็นการรับประกันแบบเข้มงวด

ดูเพิ่มเติมที่ [/token-use](/th/reference/token-use)

## Compaction: คืออะไร

Compaction สรุปบทสนทนาเก่าเป็นรายการ `compaction` ที่คงอยู่ในทรานสคริปต์ และเก็บข้อความล่าสุดไว้เหมือนเดิม

หลัง Compaction เทิร์นในอนาคตจะเห็น:

- สรุป Compaction
- ข้อความหลัง `firstKeptEntryId`

Compaction เป็นแบบ **คงอยู่** (ต่างจากการตัดแต่งเซสชัน) ดู [/concepts/session-pruning](/th/concepts/session-pruning)

## ขอบเขตชังก์ Compaction และการจับคู่เครื่องมือ

เมื่อ OpenClaw แยกทรานสคริปต์ยาวออกเป็นชังก์ Compaction มันจะเก็บ
การเรียกเครื่องมือของ assistant ให้จับคู่กับรายการ `toolResult` ที่ตรงกัน

- หากการแบ่งตามสัดส่วนโทเคนตกอยู่ระหว่างการเรียกเครื่องมือกับผลลัพธ์ OpenClaw
  จะเลื่อนขอบเขตไปที่ข้อความการเรียกเครื่องมือของ assistant แทนการแยก
  คู่ออกจากกัน
- หากบล็อกผลลัพธ์เครื่องมือท้ายสุดจะทำให้ชังก์เกินเป้าหมาย มิฉะนั้น
  OpenClaw จะคงบล็อกเครื่องมือที่รออยู่นั้นไว้ และเก็บส่วนท้ายที่ยังไม่ได้สรุป
  ไว้เหมือนเดิม
- บล็อกการเรียกเครื่องมือที่ถูกยกเลิก/มีข้อผิดพลาดจะไม่เปิดการแบ่งที่รออยู่ค้างไว้

## เมื่อ auto-compaction เกิดขึ้น (Pi runtime)

ในเอเจนต์ Pi ที่ฝังอยู่ auto-compaction จะถูกทริกเกอร์ในสองกรณี:

1. **การกู้คืนจากการล้น**: โมเดลส่งคืนข้อผิดพลาดบริบทล้น
   (`request_too_large`, `context length exceeded`, `input exceeds the maximum
number of tokens`, `input token count exceeds the maximum number of input
tokens`, `input is too long for the model`, `ollama error: context length
exceeded` และรูปแบบคล้ายกันตามผู้ให้บริการ) → compact → retry
2. **การบำรุงรักษาตามเกณฑ์**: หลังเทิร์นสำเร็จ เมื่อ:

`contextTokens > contextWindow - reserveTokens`

โดยที่:

- `contextWindow` คือหน้าต่างบริบทของโมเดล
- `reserveTokens` คือพื้นที่เผื่อที่สงวนไว้สำหรับพรอมป์ + เอาต์พุตโมเดลถัดไป

สิ่งเหล่านี้เป็น semantics ของ Pi runtime (OpenClaw ใช้เหตุการณ์เหล่านี้ แต่ Pi ตัดสินใจว่าจะ compact เมื่อใด)

OpenClaw ยังสามารถทริกเกอร์ Compaction ภายในแบบ preflight ก่อนเปิดการรันถัดไป
เมื่อมีการตั้งค่า `agents.defaults.compaction.maxActiveTranscriptBytes` และไฟล์
ทรานสคริปต์ที่ใช้งานอยู่ถึงขนาดนั้น นี่เป็นการ์ดขนาดไฟล์สำหรับต้นทุนการเปิดใหม่
ในเครื่อง ไม่ใช่การอาร์ไคฟ์ดิบ: OpenClaw ยังคงรัน Compaction เชิงความหมายตามปกติ
และต้องใช้ `truncateAfterCompaction` เพื่อให้สรุปที่ผ่าน Compaction แล้วกลายเป็น
ทรานสคริปต์ตัวสืบทอดใหม่ได้

## การตั้งค่า Compaction (`reserveTokens`, `keepRecentTokens`)

การตั้งค่า Compaction ของ Pi อยู่ในการตั้งค่า Pi:

```json5
{
  compaction: {
    enabled: true,
    reserveTokens: 16384,
    keepRecentTokens: 20000,
  },
}
```

OpenClaw ยังบังคับใช้ค่าพื้นความปลอดภัยสำหรับการรันแบบฝัง:

- หาก `compaction.reserveTokens < reserveTokensFloor` OpenClaw จะปรับเพิ่มให้
- ค่าพื้นเริ่มต้นคือ `20000` โทเค็น
- ตั้งค่า `agents.defaults.compaction.reserveTokensFloor: 0` เพื่อปิดใช้ค่าพื้น
- หากค่าสูงกว่าอยู่แล้ว OpenClaw จะปล่อยไว้เหมือนเดิม
- `/compact` แบบแมนนวลจะเคารพ `agents.defaults.compaction.keepRecentTokens` ที่ระบุไว้อย่างชัดเจน
  และคงจุดตัดส่วนท้ายล่าสุดของ Pi ไว้ หากไม่มีการระบุงบประมาณการคงเก็บไว้อย่างชัดเจน
  Compaction แบบแมนนวลจะยังเป็นเช็กพอยต์แบบแข็ง และบริบทที่สร้างใหม่จะเริ่มจาก
  สรุปใหม่
- ตั้งค่า `agents.defaults.compaction.maxActiveTranscriptBytes` เป็นค่าไบต์หรือ
  สตริง เช่น `"20mb"` เพื่อรัน Compaction ภายในเครื่องก่อนรอบหนึ่งเมื่อทรานสคริปต์ที่ใช้งานอยู่
  มีขนาดใหญ่ การป้องกันนี้จะทำงานเฉพาะเมื่อเปิดใช้
  `truncateAfterCompaction` ด้วยเท่านั้น ปล่อยไว้ไม่ตั้งค่าหรือตั้งเป็น `0` เพื่อ
  ปิดใช้
- เมื่อเปิดใช้ `agents.defaults.compaction.truncateAfterCompaction`
  OpenClaw จะหมุนทรานสคริปต์ที่ใช้งานอยู่ไปยัง JSONL ตัวต่อจาก Compaction หลังจาก
  Compaction ทรานสคริปต์ฉบับเต็มเก่าจะยังถูกเก็บถาวรและลิงก์จาก
  เช็กพอยต์ Compaction แทนที่จะถูกเขียนทับในที่เดิม

เหตุผล: เว้นพื้นที่เผื่อให้เพียงพอสำหรับ “งานดูแลระบบ” หลายรอบ เช่น การเขียนหน่วยความจำ ก่อนที่ Compaction จะหลีกเลี่ยงไม่ได้

การใช้งานจริง: `ensurePiCompactionReserveTokens()` ใน `src/agents/pi-settings.ts`
(เรียกจาก `src/agents/pi-embedded-runner.ts`)

## ผู้ให้บริการ Compaction แบบเสียบเปลี่ยนได้

Plugin สามารถลงทะเบียนผู้ให้บริการ Compaction ผ่าน `registerCompactionProvider()` บน API ของ Plugin ได้ เมื่อ `agents.defaults.compaction.provider` ถูกตั้งเป็น id ของผู้ให้บริการที่ลงทะเบียนไว้ ส่วนขยาย safeguard จะมอบหมายการสรุปให้ผู้ให้บริการนั้นแทนไปป์ไลน์ `summarizeInStages` ในตัว

- `provider`: id ของ Plugin ผู้ให้บริการ Compaction ที่ลงทะเบียนไว้ ปล่อยว่างไว้เพื่อใช้การสรุปด้วย LLM เริ่มต้น
- การตั้งค่า `provider` จะบังคับ `mode: "safeguard"`
- ผู้ให้บริการจะได้รับคำสั่ง Compaction และนโยบายการรักษาตัวระบุแบบเดียวกับเส้นทางในตัว
- safeguard ยังรักษาบริบทส่วนท้ายของรอบล่าสุดและรอบที่ถูกแยกไว้หลังเอาต์พุตจากผู้ให้บริการ
- การสรุป safeguard ในตัวจะกลั่นสรุปเดิมซ้ำพร้อมข้อความใหม่
  แทนการคงสรุปก่อนหน้าแบบเต็มไว้ตามตัวอักษร
- โหมด safeguard เปิดใช้การตรวจสอบคุณภาพสรุปโดยค่าเริ่มต้น; ตั้งค่า
  `qualityGuard.enabled: false` เพื่อข้ามพฤติกรรมลองใหม่เมื่อเอาต์พุตมีรูปแบบผิด
- หากผู้ให้บริการล้มเหลวหรือส่งผลลัพธ์ว่าง OpenClaw จะถอยกลับไปใช้การสรุปด้วย LLM ในตัวโดยอัตโนมัติ
- สัญญาณยกเลิก/หมดเวลาจะถูกโยนต่ออีกครั้ง (ไม่ถูกกลืน) เพื่อเคารพการยกเลิกของผู้เรียก

แหล่งที่มา: `src/plugins/compaction-provider.ts`, `src/agents/pi-hooks/compaction-safeguard.ts`

## งานดูแลระบบแบบเงียบ (`NO_REPLY`)

OpenClaw รองรับรอบแบบ “เงียบ” สำหรับงานเบื้องหลังที่ผู้ใช้ไม่ควรเห็นเอาต์พุตระหว่างทาง

แนวปฏิบัติ:

- ผู้ช่วยเริ่มเอาต์พุตด้วยโทเค็นเงียบที่ตรงกันทุกตัวอักษร `NO_REPLY` /
  `no_reply` เพื่อระบุว่า “ไม่ต้องส่งคำตอบให้ผู้ใช้”
- OpenClaw จะตัด/ระงับสิ่งนี้ในเลเยอร์การส่ง
- การระงับโทเค็นเงียบที่ตรงกันทุกตัวอักษรไม่สนตัวพิมพ์เล็กใหญ่ ดังนั้น `NO_REPLY` และ
  `no_reply` จะนับทั้งคู่เมื่อเพย์โหลดทั้งหมดเป็นเพียงโทเค็นเงียบ
- สิ่งนี้มีไว้สำหรับรอบเบื้องหลัง/ไม่ส่งมอบจริงเท่านั้น ไม่ใช่ทางลัดสำหรับ
  คำขอผู้ใช้ทั่วไปที่ต้องดำเนินการ

ตั้งแต่ `2026.1.10` เป็นต้นมา OpenClaw ยังระงับ **การสตรีมแบบร่าง/กำลังพิมพ์** เมื่อ
ชังก์บางส่วนเริ่มด้วย `NO_REPLY` เพื่อไม่ให้การทำงานแบบเงียบรั่วเอาต์พุตบางส่วน
กลางรอบ

## "การล้างหน่วยความจำ" ก่อน Compaction (ใช้งานแล้ว)

เป้าหมาย: ก่อนเกิด Compaction อัตโนมัติ ให้รันรอบแบบ agentic เงียบที่เขียนสถานะถาวร
ลงดิสก์ เช่น `memory/YYYY-MM-DD.md` ในพื้นที่ทำงานของเอเจนต์ เพื่อให้ Compaction ไม่สามารถ
ลบบริบทสำคัญได้

OpenClaw ใช้แนวทาง **การล้างก่อนถึงเกณฑ์**:

1. ติดตามการใช้บริบทของเซสชัน
2. เมื่อข้าม “เกณฑ์แบบอ่อน” (ต่ำกว่าเกณฑ์ Compaction ของ Pi) ให้รันคำสั่ง
   “เขียนหน่วยความจำตอนนี้” แบบเงียบให้เอเจนต์
3. ใช้โทเค็นเงียบที่ตรงกันทุกตัวอักษร `NO_REPLY` / `no_reply` เพื่อให้ผู้ใช้ไม่เห็น
   อะไรเลย

การกำหนดค่า (`agents.defaults.compaction.memoryFlush`):

- `enabled` (ค่าเริ่มต้น: `true`)
- `model` (การ override provider/model ที่ตรงกันแบบเลือกได้สำหรับรอบล้าง เช่น `ollama/qwen3:8b`)
- `softThresholdTokens` (ค่าเริ่มต้น: `4000`)
- `prompt` (ข้อความผู้ใช้สำหรับรอบล้าง)
- `systemPrompt` (พรอมป์ระบบเพิ่มเติมที่ต่อท้ายสำหรับรอบล้าง)

หมายเหตุ:

- พรอมป์/พรอมป์ระบบเริ่มต้นมีคำใบ้ `NO_REPLY` เพื่อระงับ
  การส่งมอบ
- เมื่อตั้งค่า `model` รอบล้างจะใช้โมเดลนั้นโดยไม่สืบทอดเชน fallback ของ
  เซสชันที่ใช้งานอยู่ ดังนั้นงานดูแลระบบเฉพาะภายในเครื่องจะไม่ถอยกลับแบบเงียบๆ
  ไปยังโมเดลสนทนาแบบมีค่าใช้จ่าย
- การล้างจะรันหนึ่งครั้งต่อรอบ Compaction (ติดตามใน `sessions.json`)
- การล้างจะรันเฉพาะสำหรับเซสชัน Pi แบบฝัง (แบ็กเอนด์ CLI จะข้าม)
- การล้างจะถูกข้ามเมื่อพื้นที่ทำงานของเซสชันเป็นแบบอ่านอย่างเดียว (`workspaceAccess: "ro"` หรือ `"none"`)
- ดู [หน่วยความจำ](/th/concepts/memory) สำหรับโครงร่างไฟล์พื้นที่ทำงานและรูปแบบการเขียน

Pi ยังเปิดเผยฮุก `session_before_compact` ใน API ของส่วนขยายด้วย แต่ตรรกะการล้างของ OpenClaw
อยู่ฝั่ง Gateway ในปัจจุบัน

## รายการตรวจสอบการแก้ไขปัญหา

- คีย์เซสชันผิดหรือไม่ เริ่มจาก [/concepts/session](/th/concepts/session) และยืนยัน `sessionKey` ใน `/status`
- ที่เก็บกับทรานสคริปต์ไม่ตรงกันหรือไม่ ยืนยันโฮสต์ Gateway และเส้นทางที่เก็บจาก `openclaw status`
- Compaction ถี่เกินไปหรือไม่ ตรวจสอบ:
  - หน้าต่างบริบทของโมเดล (เล็กเกินไป)
  - การตั้งค่า Compaction (`reserveTokens` สูงเกินไปสำหรับหน้าต่างของโมเดลอาจทำให้เกิด Compaction เร็วขึ้น)
  - ผลลัพธ์เครื่องมือบวม: เปิดใช้/ปรับแต่งการตัดแต่งเซสชัน
- รอบเงียบรั่วหรือไม่ ยืนยันว่าคำตอบเริ่มด้วย `NO_REPLY` (โทเค็นที่ตรงกันทุกตัวอักษรแบบไม่สนตัวพิมพ์เล็กใหญ่) และคุณอยู่บนบิลด์ที่มีการแก้ไขการระงับการสตรีมแล้ว
