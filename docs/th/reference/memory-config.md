การตั้งค่าการค้นหาหน่วยความจำทั้งหมดอยู่ภายใต้ `agents.defaults.memorySearch` ใน `openclaw.json` (หรือการแทนที่ `agents.list[].memorySearch` สำหรับแต่ละเอเจนต์) เว้นแต่จะระบุไว้เป็นอย่างอื่น

<Note>
หากกำลังมองหาสวิตช์เปิดใช้ฟีเจอร์ **Active Memory** และการกำหนดค่าเอเจนต์ย่อย การตั้งค่านั้นจะอยู่ภายใต้ `plugins.entries.active-memory` แทน `memorySearch`

Active Memory ใช้โมเดลแบบสองด่าน:

1. ต้องเปิดใช้ Plugin และกำหนดเป้าหมายเป็น ID ของเอเจนต์ปัจจุบัน
2. คำขอต้องเป็นเซสชันแชตถาวรแบบโต้ตอบที่มีสิทธิ์

ดูโมเดลการเปิดใช้งาน การกำหนดค่าที่ Plugin เป็นเจ้าของ การเก็บรักษาทรานสคริปต์ และรูปแบบการทยอยเปิดใช้อย่างปลอดภัยได้ที่ [Active Memory](/th/concepts/active-memory)
</Note>

## การเลือกผู้ให้บริการ

| คีย์        | ชนิด      | ค่าเริ่มต้น          | คำอธิบาย                                                                                                                                                                                                                                                                                 |
| ---------- | --------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`  | `boolean` | `true`           | เปิดหรือปิดการค้นหาหน่วยความจำ                                                                                                                                                                                                                                                             |
| `provider` | `string`  | `"openai"`       | ID อะแดปเตอร์การฝังเวกเตอร์ เช่น `bedrock`, `deepinfra`, `gemini`, `github-copilot`, `local`, `mistral`, `ollama`, `openai`, `openai-compatible` หรือ `voyage`; นอกจากนี้ยังอาจเป็น `models.providers.<id>` ที่กำหนดค่าไว้ โดยมี `api` ชี้ไปยังอะแดปเตอร์การฝังเวกเตอร์ของหน่วยความจำหรือ API โมเดลที่เข้ากันได้กับ OpenAI |
| `model`    | `string`  | ค่าเริ่มต้นของผู้ให้บริการ | ชื่อโมเดลการฝังเวกเตอร์                                                                                                                                                                                                                                                                        |
| `fallback` | `string`  | `"none"`         | ID อะแดปเตอร์สำรองเมื่ออะแดปเตอร์หลักล้มเหลว                                                                                                                                                                                                                                                  |

เมื่อไม่ได้ตั้งค่า `provider` OpenClaw จะใช้การฝังเวกเตอร์ของ OpenAI ตั้งค่า `provider`
อย่างชัดเจนเพื่อใช้ Bedrock, DeepInfra, Gemini, GitHub Copilot, Mistral, Ollama,
Voyage, โมเดล GGUF ภายในเครื่อง หรือเอนด์พอยต์ `/v1/embeddings` ที่เข้ากันได้กับ OpenAI
การกำหนดค่าแบบเดิมที่ยังระบุ `provider: "auto"` จะถูกแก้ไขเป็น `openai`

<Warning>
การเปลี่ยนผู้ให้บริการการฝังเวกเตอร์ โมเดล การตั้งค่าผู้ให้บริการ แหล่งข้อมูล ขอบเขต
การแบ่งส่วน หรือโทเคไนเซอร์ อาจทำให้ดัชนีเวกเตอร์ SQLite ที่มีอยู่เข้ากันไม่ได้
OpenClaw จะหยุดการค้นหาเวกเตอร์ชั่วคราวและรายงานคำเตือนเกี่ยวกับอัตลักษณ์ของดัชนี แทนการ
ฝังเวกเตอร์ทุกอย่างใหม่โดยอัตโนมัติ สร้างดัชนีใหม่เมื่อพร้อมด้วย
`openclaw memory status --index --agent <id>` หรือ
`openclaw memory index --force --agent <id>`
</Warning>

เมื่อไม่ได้ตั้งค่า `provider`, มี `provider: "auto"` แบบเดิมอยู่ หรือ
`provider: "none"` เลือกโหมด FTS เท่านั้นโดยเจตนา การเรียกคืนหน่วยความจำยังสามารถ
ใช้การจัดอันดับ FTS เชิงคำศัพท์ได้เมื่อการฝังเวกเตอร์ไม่พร้อมใช้งาน

ผู้ให้บริการที่ไม่ใช่ภายในเครื่องซึ่งระบุอย่างชัดเจนจะปฏิเสธการทำงานเมื่อไม่พร้อมใช้งาน หากตั้งค่า `memorySearch.provider` เป็น
ผู้ให้บริการที่มีแบ็กเอนด์ระยะไกลโดยเฉพาะ เช่น Bedrock, DeepInfra, Gemini, GitHub
Copilot, LM Studio, Mistral, Ollama, OpenAI, Voyage หรือผู้ให้บริการแบบกำหนดเอง
ที่เข้ากันได้กับ OpenAI และผู้ให้บริการนั้นไม่พร้อมใช้งานขณะรันไทม์ `memory_search`
จะส่งคืนผลลัพธ์ว่าไม่พร้อมใช้งาน แทนการใช้การเรียกคืนแบบ FTS เท่านั้นโดยไม่แจ้งให้ทราบ แก้ไข
การกำหนดค่าผู้ให้บริการ/การยืนยันตัวตน เปลี่ยนไปใช้ผู้ให้บริการที่เข้าถึงได้ หรือตั้งค่า
`provider: "none"` หากต้องการใช้การเรียกคืนแบบ FTS เท่านั้นโดยเจตนา

### ID ผู้ให้บริการแบบกำหนดเอง

`memorySearch.provider` สามารถชี้ไปยังรายการ `models.providers.<id>` แบบกำหนดเองสำหรับอะแดปเตอร์ผู้ให้บริการเฉพาะหน่วยความจำ เช่น `ollama` หรือสำหรับ API โมเดลที่เข้ากันได้กับ OpenAI เช่น `openai-responses` / `openai-completions` OpenClaw จะแก้ไขเจ้าของ `api` ของผู้ให้บริการนั้นสำหรับอะแดปเตอร์การฝังเวกเตอร์ พร้อมคง ID ผู้ให้บริการแบบกำหนดเองไว้สำหรับการจัดการเอนด์พอยต์ การยืนยันตัวตน และคำนำหน้าโมเดล ซึ่งช่วยให้การติดตั้งแบบหลาย GPU หรือหลายโฮสต์สามารถจัดสรรการฝังเวกเตอร์ของหน่วยความจำให้กับเอนด์พอยต์ภายในเครื่องที่ระบุได้:

```json5
{
  models: {
    providers: {
      "ollama-5080": {
        api: "ollama",
        baseUrl: "http://gpu-box.local:11435",
        apiKey: "ollama-local",
        models: [{ id: "qwen3-embedding:0.6b", name: "Qwen3 Embedding 0.6B" }],
      },
    },
  },
  agents: {
    defaults: {
      memorySearch: {
        provider: "ollama-5080",
        model: "qwen3-embedding:0.6b",
      },
    },
  },
}
```

### การแก้ไขคีย์ API

การฝังเวกเตอร์ระยะไกลต้องใช้คีย์ API ส่วน Bedrock ใช้สายโซ่ข้อมูลประจำตัวเริ่มต้นของ AWS SDK แทน (บทบาทอินสแตนซ์, SSO, คีย์การเข้าถึง หรือคีย์ API ของ Bedrock)

| ผู้ให้บริการ       | ตัวแปรสภาพแวดล้อม                                             | คีย์การกำหนดค่า                          |
| -------------- | --------------------------------------------------- | ----------------------------------- |
| Bedrock        | สายโซ่ข้อมูลประจำตัว AWS หรือ `AWS_BEARER_TOKEN_BEDROCK` | ไม่ต้องใช้คีย์ API                   |
| DeepInfra      | `DEEPINFRA_API_KEY`                                 | `models.providers.deepinfra.apiKey` |
| Gemini         | `GEMINI_API_KEY`                                    | `models.providers.google.apiKey`    |
| GitHub Copilot | `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, `GITHUB_TOKEN`  | โปรไฟล์การยืนยันตัวตนผ่านการเข้าสู่ระบบด้วยอุปกรณ์       |
| Mistral        | `MISTRAL_API_KEY`                                   | `models.providers.mistral.apiKey`   |
| Ollama         | `OLLAMA_API_KEY` (ตัวยึดตำแหน่ง)                      | --                                  |
| OpenAI         | `OPENAI_API_KEY`                                    | `models.providers.openai.apiKey`    |
| Voyage         | `VOYAGE_API_KEY`                                    | `models.providers.voyage.apiKey`    |

<Note>
Codex OAuth ครอบคลุมเฉพาะแชต/การเติมข้อความให้สมบูรณ์ และไม่รองรับคำขอการฝังเวกเตอร์
</Note>

## การกำหนดค่าเอนด์พอยต์ระยะไกล

ใช้ `provider: "openai-compatible"` สำหรับเซิร์ฟเวอร์ `/v1/embeddings`
ทั่วไปที่เข้ากันได้กับ OpenAI ซึ่งไม่ควรรับช่วงข้อมูลประจำตัวสำหรับแชต OpenAI ส่วนกลาง

<ParamField path="remote.baseUrl" type="string">
  URL ฐาน API แบบกำหนดเอง
</ParamField>
<ParamField path="remote.apiKey" type="string">
  แทนที่คีย์ API
</ParamField>
<ParamField path="remote.headers" type="object">
  ส่วนหัว HTTP เพิ่มเติม (ผสานกับค่าเริ่มต้นของผู้ให้บริการ)
</ParamField>

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai-compatible",
        model: "text-embedding-3-small",
        remote: {
          baseUrl: "https://api.example.com/v1/",
          apiKey: "YOUR_KEY",
        },
      },
    },
  },
}
```

## การกำหนดค่าเฉพาะผู้ให้บริการ

<AccordionGroup>
  <Accordion title="Gemini">
    | คีย์                    | ชนิด     | ค่าเริ่มต้น                | คำอธิบาย                                |
    | ---------------------- | -------- | ---------------------- | ------------------------------------------- |
    | `model`                | `string` | `gemini-embedding-001` | รองรับ `gemini-embedding-2-preview` ด้วย |
    | `outputDimensionality` | `number` | `3072`                 | สำหรับ Embedding 2: 768, 1536 หรือ 3072        |

    <Warning>
    การเปลี่ยนโมเดลหรือ `outputDimensionality` จะเปลี่ยนอัตลักษณ์ของดัชนี OpenClaw
    จะหยุดการค้นหาเวกเตอร์ชั่วคราวจนกว่าจะสร้างดัชนีหน่วยความจำใหม่อย่างชัดเจน
    </Warning>

  </Accordion>
  <Accordion title="ชนิดอินพุตที่เข้ากันได้กับ OpenAI">
    เอนด์พอยต์การฝังเวกเตอร์ที่เข้ากันได้กับ OpenAI สามารถเลือกใช้ฟิลด์คำขอ `input_type` เฉพาะผู้ให้บริการได้ ซึ่งมีประโยชน์สำหรับโมเดลการฝังเวกเตอร์แบบอสมมาตรที่ต้องใช้ป้ายกำกับต่างกันสำหรับการฝังเวกเตอร์คำค้นหาและเอกสาร

    | คีย์                 | ชนิด     | ค่าเริ่มต้น | คำอธิบาย                                             |
    | ------------------- | -------- | ------- | -------------------------------------------------------- |
    | `inputType`         | `string` | ไม่ได้ตั้งค่า   | `input_type` ที่ใช้ร่วมกันสำหรับการฝังเวกเตอร์คำค้นหาและเอกสาร   |
    | `queryInputType`    | `string` | ไม่ได้ตั้งค่า   | `input_type` ขณะค้นหา; แทนที่ `inputType`          |
    | `documentInputType` | `string` | ไม่ได้ตั้งค่า   | `input_type` สำหรับดัชนี/เอกสาร; แทนที่ `inputType`      |

    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            provider: "openai-compatible",
            remote: {
              baseUrl: "https://embeddings.example/v1",
              apiKey: "${EMBEDDINGS_API_KEY}",
            },
            model: "asymmetric-embedder",
            queryInputType: "query",
            documentInputType: "passage",
          },
        },
      },
    }
    ```

    การเปลี่ยนค่าเหล่านี้จะส่งผลต่ออัตลักษณ์แคชการฝังเวกเตอร์สำหรับการสร้างดัชนีแบบแบตช์ของผู้ให้บริการ และควรสร้างดัชนีหน่วยความจำใหม่หลังจากนั้น เมื่อโมเดลต้นทางจัดการป้ายกำกับแตกต่างกัน

  </Accordion>
  <Accordion title="Bedrock">
    ### การกำหนดค่าการฝังเวกเตอร์ของ Bedrock

    Bedrock ใช้สายโซ่ข้อมูลประจำตัวเริ่มต้นของ AWS SDK ร่วมกับโทเค็นผู้ถือที่ OpenClaw ตรวจสอบแล้ว จึงไม่มีการจัดเก็บคีย์ API ไว้ในการกำหนดค่า หาก OpenClaw ทำงานบน EC2 ที่มีบทบาทอินสแตนซ์ซึ่งเปิดใช้ Bedrock เพียงตั้งค่าผู้ให้บริการและโมเดล:

    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            provider: "bedrock",
            model: "amazon.titan-embed-text-v2:0",
          },
        },
      },
    }
    ```

    | คีย์                    | ชนิด     | ค่าเริ่มต้น                        | คำอธิบาย                     |
    | ---------------------- | -------- | ------------------------------- | -------------------------------- |
    | `model`                | `string` | `amazon.titan-embed-text-v2:0` | ID โมเดลการฝังเวกเตอร์ Bedrock ใดก็ได้  |
    | `outputDimensionality` | `number` | ค่าเริ่มต้นของโมเดล                  | สำหรับ Titan V2: 256, 512 หรือ 1024 |

    **โมเดลที่รองรับ** (พร้อมการตรวจหาตระกูลและค่าเริ่มต้นของมิติ):

    | รหัสโมเดล                                   | ผู้ให้บริการ   | มิติเริ่มต้น | มิติที่กำหนดค่าได้          |
    | ------------------------------------------- | ---------- | ------------- | -------------------------- |
    | `amazon.titan-embed-text-v2:0`             | Amazon     | 1024         | 256, 512, 1024             |
    | `amazon.titan-embed-text-v1`               | Amazon     | 1536         | --                          |
    | `amazon.titan-embed-g1-text-02`            | Amazon     | 1536         | --                          |
    | `amazon.titan-embed-image-v1`              | Amazon     | 1024         | --                          |
    | `amazon.nova-2-multimodal-embeddings-v1:0` | Amazon     | 1024         | 256, 384, 1024, 3072       |
    | `cohere.embed-english-v3`                  | Cohere     | 1024         | --                          |
    | `cohere.embed-multilingual-v3`             | Cohere     | 1024         | --                          |
    | `cohere.embed-v4:0`                        | Cohere     | 1536         | 256, 384, 512, 768, 1024, 1536 |
    | `twelvelabs.marengo-embed-3-0-v1:0`        | TwelveLabs | 512          | --                          |
    | `twelvelabs.marengo-embed-2-7-v1:0`        | TwelveLabs | 1024         | --                          |

    ตัวแปรที่มีคำต่อท้ายระบุปริมาณงาน (เช่น `amazon.titan-embed-text-v1:2:8k`) และรหัสโปรไฟล์การอนุมานที่มีคำนำหน้าภูมิภาค (เช่น `us.amazon.titan-embed-text-v2:0`) จะสืบทอดการกำหนดค่าของโมเดลพื้นฐาน

    **ภูมิภาค:** แก้ไขค่าตามลำดับดังนี้: ค่าที่เขียนทับด้วย `memorySearch.remote.baseUrl`, การกำหนดค่า `models.providers.amazon-bedrock.baseUrl`, `AWS_REGION`, `AWS_DEFAULT_REGION` แล้วจึงใช้ค่าเริ่มต้นเป็น `us-east-1`

    **การยืนยันตัวตน:** OpenClaw จะตรวจหา `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` หรือ `AWS_BEARER_TOKEN_BEDROCK` ก่อน จากนั้นจึงใช้สายโซ่ผู้ให้บริการข้อมูลประจำตัวเริ่มต้นมาตรฐานของ AWS SDK ตามลำดับ:

    1. ตัวแปรสภาพแวดล้อม (`AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY`) เว้นแต่จะตั้งค่า `AWS_PROFILE` ไว้ด้วย
    2. SSO (เฉพาะเมื่อกำหนดค่าฟิลด์ SSO แล้ว)
    3. ไฟล์ข้อมูลประจำตัวและไฟล์การกำหนดค่าที่ใช้ร่วมกัน (`fromIni` ซึ่งรวม `AWS_PROFILE`)
    4. กระบวนการข้อมูลประจำตัว (`credential_process` ในไฟล์การกำหนดค่า AWS)
    5. ข้อมูลประจำตัวโทเค็นข้อมูลประจำตัวบนเว็บ
    6. ข้อมูลประจำตัวจากข้อมูลเมตาของอินสแตนซ์ ECS หรือ EC2

    **สิทธิ์ IAM:** บทบาทหรือผู้ใช้ IAM ต้องมี:

    ```json
    {
      "Effect": "Allow",
      "Action": "bedrock:InvokeModel",
      "Resource": "*"
    }
    ```

    เพื่อให้มีสิทธิ์เท่าที่จำเป็น ให้จำกัดขอบเขต `InvokeModel` ไว้ที่โมเดลที่ระบุ:

    ```text
    arn:aws:bedrock:*::foundation-model/amazon.titan-embed-text-v2:0
    ```

## พาธหน่วยความจำเพิ่มเติม

| คีย์          | ชนิด       | คำอธิบาย                              |
| ------------ | ---------- | ---------------------------------------- |
| `extraPaths` | `string[]` | ไดเรกทอรีหรือไฟล์เพิ่มเติมที่จะทำดัชนี |

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        extraPaths: ["../team-docs", "/srv/shared-notes"],
      },
    },
  },
}
```

พาธอาจเป็นพาธแบบสัมบูรณ์หรือสัมพันธ์กับเวิร์กสเปซ ไดเรกทอรีจะถูกสแกนแบบเรียกซ้ำเพื่อค้นหาไฟล์ `.md` การจัดการลิงก์สัญลักษณ์ขึ้นอยู่กับแบ็กเอนด์ที่ใช้งานอยู่: เอนจินในตัวจะข้ามลิงก์สัญลักษณ์ ส่วน QMD จะทำตามพฤติกรรมของตัวสแกน QMD ที่อยู่เบื้องหลัง

สำหรับการค้นหาบทสนทนาข้ามเอเจนต์แบบจำกัดขอบเขตตามเอเจนต์ ให้ใช้ `agents.list[].memorySearch.qmd.extraCollections` แทน `memory.qmd.paths` คอลเลกชันเพิ่มเติมเหล่านั้นใช้รูปแบบ `{ path, name, pattern? }` เดียวกัน แต่จะผสานแยกตามเอเจนต์และสามารถรักษาชื่อที่ใช้ร่วมกันซึ่งระบุไว้อย่างชัดเจนได้เมื่อพาธชี้ออกนอกเวิร์กสเปซปัจจุบัน หากพาธที่แก้ไขแล้วเดียวกันปรากฏทั้งใน `memory.qmd.paths` และ `memorySearch.qmd.extraCollections` QMD จะเก็บรายการแรกและข้ามรายการที่ซ้ำกัน

## แคช Embedding

| คีย์                | ชนิด      | ค่าเริ่มต้น | คำอธิบาย                                  |
| ------------------ | --------- | ------- | -------------------------------------------- |
| `cache.enabled`    | `boolean` | `true`  | แคช Embedding ของส่วนข้อความใน SQLite             |
| `cache.maxEntries` | `number`  | ไม่ได้ตั้งค่า   | ขีดจำกัดสูงสุดโดยประมาณของ Embedding ที่แคชไว้ |

ช่วยป้องกันไม่ให้สร้าง Embedding ของข้อความที่ไม่มีการเปลี่ยนแปลงซ้ำระหว่างการทำดัชนีใหม่หรือการอัปเดตบทถอดเสียง ปล่อย `maxEntries` ไว้โดยไม่ตั้งค่าสำหรับแคชที่ไม่มีขีดจำกัด ให้ตั้งค่าเมื่อการเติบโตของพื้นที่ดิสก์สำคัญกว่าความเร็วสูงสุดในการทำดัชนีใหม่ เมื่อตั้งค่าแล้ว รายการที่เก่าที่สุด (ตามเวลาที่อัปเดตล่าสุด) จะถูกลบออกก่อนเมื่อแคชเกินขีดจำกัด

| `remote.batch.timeoutMinutes` | `number`  | `60`    | ระยะหมดเวลาของแบตช์              |

ใช้ได้กับ `gemini`, `openai` และ `voyage` โดยทั่วไปแบตช์ของ OpenAI จะเร็วที่สุดและประหยัดที่สุดสำหรับการเติมข้อมูลย้อนหลังจำนวนมาก

`remote.nonBatchConcurrency` ควบคุมการเรียกใช้ Embedding แบบอินไลน์ที่ใช้โดยผู้ให้บริการในเครื่อง/โฮสต์เอง และผู้ให้บริการแบบโฮสต์เมื่อ API แบตช์ของผู้ให้บริการไม่ได้ทำงานอยู่ Ollama ใช้ค่าเริ่มต้นเป็น `1` สำหรับการทำดัชนีที่ไม่ใช่แบบแบตช์ เพื่อหลีกเลี่ยงการใช้งานโฮสต์ในเครื่องขนาดเล็กหนักเกินไป ให้ตั้งค่าที่สูงขึ้นบนเครื่องขนาดใหญ่

การตั้งค่านี้แยกจาก `sync.embeddingBatchTimeoutSeconds` ซึ่งควบคุมระยะหมดเวลาสำหรับการเรียกใช้ Embedding แบบอินไลน์

## การค้นหาหน่วยความจำเซสชัน (ขั้นทดลอง)

ทำดัชนีบทถอดเสียงของเซสชันและแสดงผลผ่าน `memory_search`:

| คีย์                           | ชนิด       | ค่าเริ่มต้น      | คำอธิบาย                             |
| ----------------------------- | ---------- | ------------ | --------------------------------------- |
| `experimental.sessionMemory`  | `boolean`  | `false`      | เปิดใช้การทำดัชนีเซสชัน                 |
| `sources`                     | `string[]` | `["memory"]` | เพิ่ม `"sessions"` เพื่อรวมบทถอดเสียง |
| `sync.sessions.deltaBytes`    | `number`   | `100000`     | เกณฑ์จำนวนไบต์สำหรับการทำดัชนีใหม่              |
| `sync.sessions.deltaMessages` | `number`   | `50`         | เกณฑ์จำนวนข้อความสำหรับการทำดัชนีใหม่           |

<Warning>
การทำดัชนีเซสชันต้องเปิดใช้งานโดยสมัครใจและทำงานแบบอะซิงโครนัส ผลลัพธ์อาจไม่เป็นปัจจุบันเล็กน้อย บันทึกเซสชันจัดเก็บอยู่บนดิสก์ ดังนั้นให้ถือว่าการเข้าถึงระบบไฟล์เป็นขอบเขตความเชื่อถือ
</Warning>

ผลลัพธ์ที่พบในบทถอดเสียงของเซสชันยังอยู่ภายใต้
[`tools.sessions.visibility`](/th/gateway/config-tools#toolssessions) การมองเห็นเริ่มต้น
`tree` จะแสดงเฉพาะเซสชันปัจจุบันและเซสชันที่เซสชันนี้สร้างขึ้น หากต้องการ
เรียกคืนเซสชันที่ส่งผ่าน Gateway ซึ่งเป็นของเอเจนต์เดียวกันแต่ไม่เกี่ยวข้องกัน จาก
เซสชันอื่น เช่น DM ให้ขยายการมองเห็นเป็น `agent` โดยตั้งใจ (หรือ `all` เฉพาะ
เมื่อต้องเรียกคืนข้ามเอเจนต์ด้วย และนโยบายระหว่างเอเจนต์อนุญาต)

ตัวอย่างด้านล่างวางการตั้งค่าเหล่านี้ไว้ภายใต้ `agents.defaults` นอกจากนี้ยังสามารถ
ใช้การตั้งค่า `memorySearch` ที่เทียบเท่ากันในการกำหนดค่าแทนที่เฉพาะเอเจนต์ เมื่อมีเพียงเอเจนต์เดียว
ที่ควรทำดัชนีและค้นหาบทถอดเสียงของเซสชัน

สำหรับการเรียกคืนจาก Gateway ไปยัง DM ภายในเอเจนต์เดียวกัน:

<Tabs>
  <Tab title="แบ็กเอนด์ในตัว">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
  <Tab title="แบ็กเอนด์ QMD">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      memory: {
        backend: "qmd",
        qmd: {
          sessions: { enabled: true },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
</Tabs>

เมื่อใช้ QMD ค่า `agents.defaults.memorySearch.experimental.sessionMemory` และ
`sources: ["sessions"]` เพียงอย่างเดียวจะไม่ส่งออกบทถอดเสียงไปยัง QMD ต้องตั้งค่า
`memory.qmd.sessions.enabled: true` ด้วย

## การกำหนดค่าแบ็กเอนด์ QMD

ตั้งค่า `memory.backend = "qmd"` เพื่อเปิดใช้ การตั้งค่า QMD ทั้งหมดอยู่ภายใต้ `memory.qmd`:

| คีย์                      | ชนิด      | ค่าเริ่มต้น  | คำอธิบาย                                                                           |
| ------------------------ | --------- | -------- | ------------------------------------------------------------------------------------- |
| `command`                | `string`  | `qmd`    | พาธไฟล์ปฏิบัติการ QMD; ตั้งเป็นพาธสัมบูรณ์เมื่อ `PATH` ของบริการแตกต่างจากเชลล์ของคุณ |
| `searchMode`             | `string`  | `search` | คำสั่งค้นหา: `search`, `vsearch`, `query`                                          |
| `rerank`                 | `boolean` | --       | ตั้งเป็น `false` พร้อมกับ `searchMode: "query"` และ QMD 2.1+ เพื่อข้ามการจัดอันดับใหม่ของ QMD          |
| `includeDefaultMemory`   | `boolean` | `true`   | ทำดัชนี `MEMORY.md` + `memory/**/*.md` โดยอัตโนมัติ                                             |
| `paths[]`                | `array`   | --       | พาธเพิ่มเติม: `{ name, path, pattern? }`                                               |
| `sessions.enabled`       | `boolean` | `false`  | ส่งออกบทถอดเสียงของเซสชันไปยัง QMD                                                   |
| `sessions.retentionDays` | `number`  | --       | ระยะเวลาการเก็บรักษาบทถอดเสียง                                                                  |
| `sessions.exportDir`     | `string`  | --       | ไดเรกทอรีส่งออก                                                                      |

`searchMode: "search"` ใช้เฉพาะการค้นหาเชิงคำศัพท์/BM25 เท่านั้น OpenClaw จะไม่เรียกใช้การตรวจสอบความพร้อมของเวกเตอร์เชิงความหมายหรือการบำรุงรักษา Embedding ของ QMD สำหรับโหมดดังกล่าว รวมถึงระหว่าง `memory status --deep`; ส่วน `vsearch` และ `query` ยังคงต้องใช้ความพร้อมของเวกเตอร์และ Embedding ของ QMD

`rerank: false` เปลี่ยนเฉพาะโหมด `query` ของ QMD และต้องใช้ QMD 2.1 หรือใหม่กว่า ในโหมด CLI โดยตรง OpenClaw จะส่ง `--no-rerank`; ในโหมด MCP ที่รองรับด้วย mcporter จะส่ง `rerank: false` ไปยังเครื่องมือค้นหาแบบรวมของ QMD ปล่อยไว้โดยไม่ตั้งค่าเพื่อใช้พฤติกรรมการจัดอันดับผลการค้นหาใหม่ตามค่าเริ่มต้นของ QMD

OpenClaw เลือกใช้รูปแบบคอลเลกชันและการค้นหา MCP ของ QMD เวอร์ชันปัจจุบัน แต่ยังคงรองรับ QMD รุ่นเก่าโดยลองใช้แฟล็กรูปแบบคอลเลกชันที่เข้ากันได้และชื่อเครื่องมือ MCP รุ่นเก่าเมื่อจำเป็น เมื่อ QMD ระบุว่ารองรับตัวกรองหลายคอลเลกชัน ระบบจะค้นหาคอลเลกชันจากแหล่งเดียวกันด้วยกระบวนการ QMD เพียงกระบวนการเดียว ส่วน QMD รุ่นเก่ายังคงใช้เส้นทางความเข้ากันได้แบบแยกตามคอลเลกชัน แหล่งเดียวกันหมายถึงคอลเลกชันหน่วยความจำถาวร (ไฟล์หน่วยความจำเริ่มต้นรวมกับพาธที่กำหนดเอง) จะถูกจัดกลุ่มร่วมกัน ขณะที่คอลเลกชันบทถอดเสียงของเซสชันจะยังคงเป็นอีกกลุ่มหนึ่ง เพื่อให้การกระจายแหล่งข้อมูลยังคงมีข้อมูลนำเข้าจากทั้งสองแหล่ง

<Note>
การกำหนดโมเดลแทนค่าเริ่มต้นของ QMD จะอยู่ฝั่ง QMD ไม่ใช่ในการกำหนดค่าของ OpenClaw หากต้องการกำหนดโมเดลของ QMD แทนค่าเริ่มต้นแบบทั่วทั้งระบบ ให้ตั้งค่าตัวแปรสภาพแวดล้อม เช่น `QMD_EMBED_MODEL`, `QMD_RERANK_MODEL` และ `QMD_GENERATE_MODEL` ในสภาพแวดล้อมรันไทม์ของ Gateway
</Note>

### การผสานรวม mcporter

ทั้งหมดอยู่ภายใต้ `memory.qmd.mcporter` โดยกำหนดเส้นทางการค้นหา QMD ผ่านดีมอน MCP `mcporter` ที่ทำงานระยะยาว แทนการสร้าง `qmd` ต่อการค้นหาแต่ละครั้ง ซึ่งช่วยลดภาระจากการเริ่มต้นแบบเย็นสำหรับโมเดลขนาดใหญ่

| คีย์           | ชนิด      | ค่าเริ่มต้น | คำอธิบาย                                                            |
| ------------- | --------- | ------- | ---------------------------------------------------------------------- |
| `enabled`     | `boolean` | `false` | กำหนดเส้นทางการเรียก QMD ผ่าน mcporter แทนการสร้าง `qmd` ต่อคำขอ |
| `serverName`  | `string`  | `qmd`   | ชื่อเซิร์ฟเวอร์ mcporter ที่เรียกใช้ `qmd mcp` พร้อม `lifecycle: keep-alive`  |
| `startDaemon` | `boolean` | `true`  | เริ่มดีมอน mcporter โดยอัตโนมัติเมื่อ `enabled` เป็น true         |

ต้องติดตั้ง `mcporter` และให้อยู่ใน PATH รวมถึงกำหนดค่าเซิร์ฟเวอร์ mcporter ที่เรียกใช้ `qmd mcp` ปิดใช้งานไว้สำหรับการตั้งค่าในเครื่องแบบเรียบง่าย ซึ่งยอมรับต้นทุนการสร้างกระบวนการต่อการค้นหาได้

  </Accordion>
</AccordionGroup>

เมื่อเปิดใช้การเริ่มต้น QMD ขณะ Gateway เริ่มทำงาน OpenClaw จะเริ่ม QMD เฉพาะสำหรับเอเจนต์ที่มีสิทธิ์เท่านั้น หาก `update.onBoot` เป็น true และไม่ได้กำหนดช่วงเวลาการบำรุงรักษาสำหรับการอัปเดต/การฝังข้อมูล การเริ่มทำงานจะใช้ตัวจัดการแบบครั้งเดียวเพื่อรีเฟรชขณะบูตแล้วปิดตัวจัดการนั้น หากกำหนดช่วงเวลาการอัปเดตหรือการฝังข้อมูลไว้ การเริ่มทำงานจะเปิดตัวจัดการ QMD แบบทำงานระยะยาว เพื่อให้ตัวจัดการเป็นเจ้าของตัวเฝ้าดูและตัวจับเวลาตามช่วงเวลา ส่วน `update.onBoot: false` จะข้ามเฉพาะการรีเฟรชทันทีขณะบูต

### ตัวอย่าง QMD ฉบับเต็ม

```json5
{
  memory: {
    backend: "qmd",
    citations: "auto",
    qmd: {
      includeDefaultMemory: true,
      update: { interval: "5m", debounceMs: 15000 },
      limits: { maxResults: 4, timeoutMs: 4000 },
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

## Dreaming

กำหนดค่า Dreaming ภายใต้ `plugins.entries.memory-core.config.dreaming` ไม่ใช่ภายใต้ `agents.defaults.memorySearch`

Dreaming ทำงานเป็นการกวาดตามกำหนดการหนึ่งครั้ง และใช้ระยะ light/deep/REM ภายในเป็นรายละเอียดการนำไปใช้

สำหรับลักษณะการทำงานเชิงแนวคิดและคำสั่งแบบสแลช โปรดดู [Dreaming](/th/concepts/dreaming)

### การตั้งค่าของผู้ใช้

| คีย์                                    | ชนิด      | ค่าเริ่มต้น       | คำอธิบาย                                                                                                                      |
| -------------------------------------- | --------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`                              | `boolean` | `false`       | เปิดหรือปิด Dreaming ทั้งหมด                                                                                              |
| `frequency`                            | `string`  | `0 3 * * *`   | รอบเวลา Cron ที่เป็นทางเลือกสำหรับการกวาด Dreaming แบบเต็ม                                                                                |
| `model`                                | `string`  | โมเดลเริ่มต้น | การแทนที่โมเดลซับเอเจนต์ Dream Diary ซึ่งเป็นทางเลือก                                                                                     |
| `phases.deep.maxPromotedSnippetTokens` | `number`  | `160`         | จำนวนโทเค็นโดยประมาณสูงสุดที่เก็บไว้จากข้อความการเรียกคืนระยะสั้นแต่ละส่วนซึ่งได้รับการเลื่อนระดับเข้าสู่ `MEMORY.md`; เมทาดาทาแหล่งที่มายังคงมองเห็นได้ |

### ตัวอย่าง

```json5
{
  plugins: {
    entries: {
      "memory-core": {
        subagent: {
          allowModelOverride: true,
          allowedModels: ["anthropic/claude-sonnet-4-6"],
        },
        config: {
          dreaming: {
            enabled: true,
            frequency: "0 3 * * *",
            model: "anthropic/claude-sonnet-4-6",
          },
        },
      },
    },
  },
}
```

<Note>
- Dreaming เขียนสถานะของเครื่องลงใน `memory/.dreams/`
- Dreaming เขียนผลลัพธ์การบรรยายที่มนุษย์อ่านได้ลงใน `DREAMS.md` (หรือ `dreams.md` ที่มีอยู่)
- `dreaming.model` ใช้เกตความเชื่อถือของซับเอเจนต์ Plugin ที่มีอยู่ โปรดตั้งค่า `plugins.entries.memory-core.subagent.allowModelOverride: true` ก่อนเปิดใช้งาน
- Dream Diary จะลองอีกครั้งหนึ่งครั้งด้วยโมเดลเริ่มต้นของเซสชัน เมื่อโมเดลที่กำหนดค่าไว้ไม่พร้อมใช้งาน ความล้มเหลวเกี่ยวกับความเชื่อถือหรือรายการที่อนุญาตจะถูกบันทึก และจะไม่มีการลองใหม่โดยไม่แจ้ง
- นโยบายและเกณฑ์ของระยะ light/deep/REM เป็นลักษณะการทำงานภายใน ไม่ใช่การกำหนดค่าที่ผู้ใช้เข้าถึงได้

</Note>
