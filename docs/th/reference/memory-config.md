การตั้งค่าการค้นหาหน่วยความจำทั้งหมดอยู่ภายใต้ `agents.defaults.memorySearch` ใน `openclaw.json` (หรือการแทนที่เฉพาะเอเจนต์ด้วย `agents.list[].memorySearch`) เว้นแต่จะระบุไว้เป็นอย่างอื่น

<Note>
สำหรับเวิร์กโฟลว์เอเจนต์ส่วนบุคคลที่แนะนำ ให้ใช้
`memorySearch.rememberAcrossConversations` การควบคุมเป้าหมาย
โมเดล พรอมต์ และเวลาแฝงขั้นสูงของ Active Memory อยู่ภายใต้ `plugins.entries.active-memory`

โปรดดู [Active Memory](/th/concepts/active-memory) สำหรับทั้งสองวิธีการเปิดใช้งาน
การเก็บทรานสคริปต์ไว้ และแนวทางการทยอยเปิดใช้งานอย่างปลอดภัย
</Note>

## จดจำข้ามการสนทนา

| คีย์                           | ชนิด      | ค่าเริ่มต้น                                                    | คำอธิบาย                                                                    |
| ----------------------------- | --------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `rememberAcrossConversations` | `boolean` | เปิดสำหรับการติดตั้งส่วนบุคคล; ปิดเมื่อกำหนดการแยก DM | ใช้บริบทที่เกี่ยวข้องจากการสนทนาส่วนตัวอื่นที่เอเจนต์นี้รู้จัก |

กำหนดค่าแยกตามเอเจนต์ เมื่อควรมีเพียงเอเจนต์ส่วนบุคคลที่เชื่อถือได้เท่านั้นที่ใช้
การเรียกคืนทรานสคริปต์ข้ามการสนทนา:

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        memorySearch: {
          rememberAcrossConversations: true,
        },
      },
    ],
  },
}
```

ค่านี้เป็นไปตามการสืบทอด `agents.defaults.memorySearch` ตามปกติ โดยมี
การแทนที่เฉพาะเอเจนต์ เมื่อไม่ได้ตั้งค่า ค่าเริ่มต้นจะเปิดเฉพาะเมื่อ
ไม่ได้ตั้งค่า `session.dmScope` ส่วนกลาง หรือเป็น `"main"` และไม่มีการผูกใดมีการแทนที่
`session.dmScope` การกำหนดค่าการแยก DM ใดๆ จะทำให้ค่าเริ่มต้นปิด การระบุ `true` หรือ
`false` อย่างชัดเจนจะมีผลเหนือกว่าเสมอ การเปิดใช้งานหมายรวมถึงการทำดัชนีทรานสคริปต์เซสชันและ
เพิ่ม `sessions` ไปยังแหล่งหน่วยความจำที่ได้รับการแก้ไขของเอเจนต์ เมื่อใช้ QMD จะ
เปิดใช้งานการส่งออกเซสชันของเอเจนต์นั้นด้วย โดยไม่จำเป็นต้องตั้งค่า
`memory.qmd.sessions.enabled` แยกต่างหากสำหรับโหมดนี้

ผู้ให้บริการหน่วยความจำในตัวของ OpenClaw รองรับเส้นทางที่ได้รับการป้องกันนี้ทั้งกับ
แบ็กเอนด์ในตัวและ QMD ผู้ให้บริการหน่วยความจำอื่นยังสามารถใช้
ฮุกการเรียกคืนของตนเองและเครื่องมือ Active Memory ขั้นสูงต่อไปได้ แต่การตั้งค่านี้จะถูกข้าม
เว้นแต่ผู้ให้บริการปัจจุบันรองรับการเรียกคืนทรานสคริปต์ส่วนตัวที่ได้รับการป้องกัน
`openclaw doctor` รายงานผู้ให้บริการที่ไม่รองรับ หรือรายการ `toolsAllow`
ของ Active Memory ที่ระบุไว้อย่างชัดเจนแต่ไม่มี `memory_search`

ขอบเขตการเรียกคืนแคบกว่าการค้นหาเซสชันทั่วไป:

- มีสิทธิ์เฉพาะการสนทนาส่วนตัวที่รู้จักของเอเจนต์เดียวกัน
- ไม่รวมการสนทนาที่กำลังตอบอยู่
- ไม่รวมกลุ่มและช่องทั้งในฐานะแหล่งที่มาและปลายทาง
- ชนิดการสนทนาที่ไม่รู้จักจะปฏิเสธโดยค่าเริ่มต้น
- การเรียกคืนในแซนด์บ็อกซ์ไม่สามารถใช้การให้สิทธิ์พิเศษข้ามการสนทนาได้

การตั้งค่านี้ไม่เปลี่ยนแปลง `tools.sessions.visibility`, คีย์เซสชัน,
พื้นที่จัดเก็บทรานสคริปต์ การกำหนดเส้นทางการส่ง หรือสิทธิ์ของ `sessions_list`,
`sessions_history` และ `sessions_send` Active Memory ดำเนินการ
เรียกคืนแบบอ่านอย่างเดียวภายในขอบเขตที่จำกัด การเรียกคืนที่ใช้ไม่ได้หรือหมดเวลาไม่ขัดขวาง
การตอบกลับ

## การเลือกผู้ให้บริการ

| คีย์        | ชนิด      | ค่าเริ่มต้น          | คำอธิบาย                                                                                                                                                                                                                                                                                 |
| ---------- | --------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`  | `boolean` | `true`           | เปิดหรือปิดการค้นหาหน่วยความจำ                                                                                                                                                                                                                                                             |
| `provider` | `string`  | `"openai"`       | ID อะแดปเตอร์การฝัง เช่น `bedrock`, `deepinfra`, `gemini`, `github-copilot`, `local`, `mistral`, `ollama`, `openai`, `openai-compatible` หรือ `voyage`; นอกจากนี้ยังอาจเป็น `models.providers.<id>` ที่กำหนดค่าไว้ ซึ่ง `api` ชี้ไปยังอะแดปเตอร์การฝังหน่วยความจำหรือ API โมเดลที่เข้ากันได้กับ OpenAI |
| `model`    | `string`  | ค่าเริ่มต้นของผู้ให้บริการ | ชื่อโมเดลการฝัง                                                                                                                                                                                                                                                                        |
| `fallback` | `string`  | `"none"`         | ID อะแดปเตอร์สำรองเมื่ออะแดปเตอร์หลักล้มเหลว                                                                                                                                                                                                                                                  |

เมื่อไม่ได้ตั้งค่า `provider` OpenClaw จะใช้การฝังของ OpenAI ตั้งค่า `provider`
อย่างชัดเจนเพื่อใช้ Bedrock, DeepInfra, Gemini, GitHub Copilot, Mistral, Ollama,
Voyage, โมเดล GGUF ภายในเครื่อง หรือปลายทาง `/v1/embeddings` ที่เข้ากันได้กับ OpenAI
การกำหนดค่าแบบเก่าที่ยังคงระบุ `provider: "auto"` จะถูกแก้ไขเป็น `openai`

<Warning>
การเปลี่ยนผู้ให้บริการการฝัง โมเดล การตั้งค่าผู้ให้บริการ แหล่งที่มา ขอบเขต
การแบ่งส่วน หรือโทเคไนเซอร์ อาจทำให้ดัชนีเวกเตอร์ SQLite ที่มีอยู่เข้ากันไม่ได้
OpenClaw จะหยุดการค้นหาเวกเตอร์ชั่วคราวและรายงานคำเตือนเกี่ยวกับข้อมูลประจำตัวของดัชนี แทนที่จะ
ฝังข้อมูลทั้งหมดใหม่โดยอัตโนมัติ สร้างใหม่เมื่อพร้อมด้วย
`openclaw memory status --index --agent <id>` หรือ
`openclaw memory index --force --agent <id>`
</Warning>

เมื่อไม่ได้ตั้งค่า `provider`, มี `provider: "auto"` แบบเก่า หรือ
`provider: "none"` เลือกโหมด FTS เท่านั้นโดยเจตนา การเรียกคืนหน่วยความจำยังคง
ใช้การจัดอันดับ FTS ตามคำศัพท์ได้เมื่อการฝังไม่พร้อมใช้งาน

ผู้ให้บริการที่ไม่ใช่ภายในเครื่องซึ่งระบุอย่างชัดเจนจะปฏิเสธโดยค่าเริ่มต้น หากตั้งค่า `memorySearch.provider` เป็น
ผู้ให้บริการระยะไกลที่เจาะจง เช่น Bedrock, DeepInfra, Gemini, GitHub
Copilot, LM Studio, Mistral, Ollama, OpenAI, Voyage หรือผู้ให้บริการแบบกำหนดเอง
ที่เข้ากันได้กับ OpenAI และผู้ให้บริการนั้นไม่พร้อมใช้งานขณะรันไทม์ `memory_search`
จะส่งคืนผลลัพธ์ว่าไม่พร้อมใช้งาน แทนที่จะใช้การเรียกคืนแบบ FTS เท่านั้นโดยไม่มีการแจ้งเตือน แก้ไข
การกำหนดค่าผู้ให้บริการ/การตรวจสอบสิทธิ์ เปลี่ยนไปใช้ผู้ให้บริการที่เข้าถึงได้ หรือตั้งค่า
`provider: "none"` หากต้องการใช้การเรียกคืนแบบ FTS เท่านั้นโดยเจตนา

### ID ผู้ให้บริการแบบกำหนดเอง

`memorySearch.provider` สามารถชี้ไปยังรายการ `models.providers.<id>` แบบกำหนดเองสำหรับอะแดปเตอร์ผู้ให้บริการเฉพาะหน่วยความจำ เช่น `ollama` หรือสำหรับ API โมเดลที่เข้ากันได้กับ OpenAI เช่น `openai-responses` / `openai-completions` OpenClaw จะแก้ไขเจ้าของ `api` ของผู้ให้บริการนั้นสำหรับอะแดปเตอร์การฝัง โดยคง ID ผู้ให้บริการแบบกำหนดเองไว้สำหรับการจัดการปลายทาง การตรวจสอบสิทธิ์ และคำนำหน้าโมเดล ซึ่งช่วยให้การตั้งค่าหลาย GPU หรือหลายโฮสต์สามารถจัดสรรการฝังหน่วยความจำให้กับปลายทางภายในเครื่องเฉพาะแห่งได้:

```json5
{
  models: {
    providers: {
      "ollama-5080": {
        api: "ollama",
        baseUrl: "http://gpu-box.local:11435",
        apiKey: "ollama-local",
        models: [{ id: "qwen3-embedding:0.6b", name: "Qwen3 Embedding 0.6B" }],
      },
    },
  },
  agents: {
    defaults: {
      memorySearch: {
        provider: "ollama-5080",
        model: "qwen3-embedding:0.6b",
      },
    },
  },
}
```

| ผู้ให้บริการ       | ตัวแปรสภาพแวดล้อม                                             | คีย์การกำหนดค่า                          |

## การกำหนดค่าปลายทางระยะไกล

ใช้ `provider: "openai-compatible"` สำหรับเซิร์ฟเวอร์ `/v1/embeddings`
ทั่วไปที่เข้ากันได้กับ OpenAI ซึ่งไม่ควรสืบทอดข้อมูลรับรองการแชต OpenAI ส่วนกลาง

<ParamField path="remote.baseUrl" type="string">
  URL ฐาน API แบบกำหนดเอง
</ParamField>
<ParamField path="remote.apiKey" type="string">
  แทนที่คีย์ API
</ParamField>
<ParamField path="remote.headers" type="object">
  ส่วนหัว HTTP เพิ่มเติม (ผสานกับค่าเริ่มต้นของผู้ให้บริการ)
</ParamField>

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai-compatible",
        model: "text-embedding-3-small",
        remote: {
          baseUrl: "https://api.example.com/v1/",
          apiKey: "YOUR_KEY",
        },
      },
    },
  },
}
```

    | `queryInputType`    | `string` | ไม่ได้ตั้งค่า   | `input_type` ขณะค้นหา; แทนที่ `inputType`          |
    | `documentInputType` | `string` | ไม่ได้ตั้งค่า   | `input_type` ของดัชนี/เอกสาร; แทนที่ `inputType`      |

    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            provider: "openai-compatible",
            remote: {
              baseUrl: "https://embeddings.example/v1",
              apiKey: "${EMBEDDINGS_API_KEY}",
            },
            model: "asymmetric-embedder",
            queryInputType: "query",
            documentInputType: "passage",
          },
        },
      },
    }
    ```

    การเปลี่ยนค่าเหล่านี้ส่งผลต่ออัตลักษณ์ของแคช embedding สำหรับการทำดัชนีแบบแบตช์ของผู้ให้บริการ และควรทำดัชนีหน่วยความจำใหม่เมื่อโมเดลต้นทางตีความป้ายกำกับแตกต่างกัน

  </Accordion>

    ### การกำหนดค่า embedding ของ Bedrock

    Bedrock ใช้สายโซ่ข้อมูลประจำตัวเริ่มต้นของ AWS SDK ร่วมกับ bearer token ที่ OpenClaw ตรวจสอบ จึงไม่มีการจัดเก็บคีย์ API ในการกำหนดค่า หาก OpenClaw ทำงานบน EC2 ซึ่งมีบทบาทอินสแตนซ์ที่เปิดใช้ Bedrock เพียงตั้งค่าผู้ให้บริการและโมเดล:

    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            provider: "bedrock",
            model: "amazon.titan-embed-text-v2:0",
          },
        },
      },
    }
    ```

    | คีย์                    | ชนิด     | ค่าเริ่มต้น                        | คำอธิบาย                     |
    | ---------------------- | -------- | ------------------------------- | -------------------------------- |
    | `model`                | `string` | `amazon.titan-embed-text-v2:0` | ID โมเดล embedding ของ Bedrock ใดก็ได้  |
    | `outputDimensionality` | `number` | ค่าเริ่มต้นของโมเดล                  | สำหรับ Titan V2: 256, 512 หรือ 1024 |

    **โมเดลที่รองรับ** (พร้อมการตรวจหาตระกูลและมิติเริ่มต้น):

    | ID โมเดล                                   | ผู้ให้บริการ   | มิติเริ่มต้น | มิติที่กำหนดค่าได้          |
    | ------------------------------------------- | ---------- | ------------- | -------------------------- |
    | `amazon.titan-embed-text-v2:0`             | Amazon     | 1024         | 256, 512, 1024             |
    | `amazon.titan-embed-text-v1`               | Amazon     | 1536         | --                          |
    | `amazon.titan-embed-g1-text-02`            | Amazon     | 1536         | --                          |
    | `amazon.titan-embed-image-v1`              | Amazon     | 1024         | --                          |
    | `amazon.nova-2-multimodal-embeddings-v1:0` | Amazon     | 1024         | 256, 384, 1024, 3072       |
    | `cohere.embed-english-v3`                  | Cohere     | 1024         | --                          |
    | `cohere.embed-multilingual-v3`             | Cohere     | 1024         | --                          |
    | `cohere.embed-v4:0`                        | Cohere     | 1536         | 256, 384, 512, 768, 1024, 1536 |
    | `twelvelabs.marengo-embed-3-0-v1:0`        | TwelveLabs | 512          | --                          |
    | `twelvelabs.marengo-embed-2-7-v1:0`        | TwelveLabs | 1024         | --                          |

    ตัวแปรที่มีส่วนต่อท้ายระบุอัตรารับส่งข้อมูล (เช่น `amazon.titan-embed-text-v1:2:8k`) และ ID โปรไฟล์การอนุมานที่มีคำนำหน้าภูมิภาค (เช่น `us.amazon.titan-embed-text-v2:0`) จะสืบทอดการกำหนดค่าของโมเดลฐาน

    **ภูมิภาค:** กำหนดตามลำดับต่อไปนี้: ค่าที่แทนที่ด้วย `memorySearch.remote.baseUrl`, การกำหนดค่า `models.providers.amazon-bedrock.baseUrl`, `AWS_REGION`, `AWS_DEFAULT_REGION` แล้วจึงใช้ค่าเริ่มต้น `us-east-1`

    **การยืนยันตัวตน:** OpenClaw ตรวจหา `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` หรือ `AWS_BEARER_TOKEN_BEDROCK` ก่อน แล้วจึงใช้สายโซ่ผู้ให้บริการข้อมูลประจำตัวเริ่มต้นมาตรฐานของ AWS SDK ตามลำดับ:

    1. ตัวแปรสภาพแวดล้อม (`AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY`) เว้นแต่จะตั้งค่า `AWS_PROFILE` ไว้ด้วย
    2. SSO (เฉพาะเมื่อกำหนดค่าฟิลด์ SSO แล้ว)
    3. ไฟล์ข้อมูลประจำตัวและการกำหนดค่าที่ใช้ร่วมกัน (`fromIni` ซึ่งรวม `AWS_PROFILE`)
    4. กระบวนการข้อมูลประจำตัว (`credential_process` ในไฟล์การกำหนดค่า AWS)
    5. ข้อมูลประจำตัวจากโทเค็นอัตลักษณ์เว็บ
    6. ข้อมูลประจำตัวจากข้อมูลเมตาของอินสแตนซ์ ECS หรือ EC2

    **สิทธิ์ IAM:** บทบาทหรือผู้ใช้ IAM ต้องมี:

    ```json
    {
      "Effect": "Allow",
      "Action": "bedrock:InvokeModel",
      "Resource": "*"
    }
    ```

    เพื่อให้มีสิทธิ์น้อยที่สุด ให้จำกัดขอบเขต `InvokeModel` ไว้เฉพาะโมเดลที่ระบุ:

    ```text
    arn:aws:bedrock:*::foundation-model/amazon.titan-embed-text-v2:0
    ```

    ```bash
    openclaw memory index --force --agent main
    ```

    ค่า `local.contextSize` แบบตัวเลขยังใช้กำหนดการจัดวางเลเยอร์ GPU อัตโนมัติของ node-llama-cpp เพื่อให้น้ำหนักโมเดลและบริบท embedding ที่ร้องขอพอดีกัน `openclaw memory status --deep` รายงานแบ็กเอนด์ llama.cpp, อุปกรณ์, การถ่ายโอนภาระ, บริบทที่ร้องขอ และข้อมูลหน่วยความจำพร้อมการประทับเวลาที่ทราบล่าสุดหลังจากรันไทม์โหลดแล้ว; สถานะแบบพาสซีฟจะไม่โหลดโมเดล

## พฤติกรรมการทำดัชนี

ทั้งหมดอยู่ภายใต้ `memorySearch.sync` เว้นแต่จะระบุไว้:

| คีย์                            | ชนิด      | ค่าเริ่มต้น | คำอธิบาย                                                           |
| ------------------------------ | --------- | ------- | --------------------------------------------------------------------- |
| `onSessionStart`               | `boolean` | `true`  | ซิงค์ดัชนีหน่วยความจำเมื่อเซสชันเริ่มต้น                           |
| `onSearch`                     | `boolean` | `true`  | ซิงค์แบบหน่วงเวลาเมื่อค้นหาหลังจากตรวจพบการเปลี่ยนแปลงเนื้อหา                 |
| `watch`                        | `boolean` | `true`  | เฝ้าดูไฟล์หน่วยความจำ (chokidar) และกำหนดเวลาทำดัชนีใหม่เมื่อมีการเปลี่ยนแปลง         |
| `watchDebounceMs`              | `number`  | `1500`  | ช่วง debounce สำหรับรวมเหตุการณ์เฝ้าดูไฟล์ที่เกิดขึ้นอย่างรวดเร็ว                |
| `intervalMinutes`              | `number`  | `0`     | ช่วงเวลาทำดัชนีใหม่เป็นระยะในหน่วยนาที (`0` ปิดใช้งาน)                   |
| `sessions.postCompactionForce` | `boolean` | `true`  | บังคับทำดัชนีเซสชันใหม่หลังการอัปเดตบทถอดเสียงที่ถูกกระตุ้นโดย Compaction |

<ParamField path="chunking.tokens" type="number">
  ขนาดชังก์เป็นจำนวนโทเค็นที่ใช้เมื่อแบ่งแหล่งหน่วยความจำก่อนทำ embedding (ค่าเริ่มต้น: 400)
</ParamField>
<ParamField path="chunking.overlap" type="number">
  จำนวนโทเค็นที่ซ้อนทับกันระหว่างชังก์ที่อยู่ติดกันเพื่อรักษาบริบทใกล้ขอบเขตการแบ่ง (ค่าเริ่มต้น: 80)
</ParamField>

<Note>
การเปลี่ยน `chunking.tokens` หรือ `chunking.overlap` จะเปลี่ยนขอบเขตของชังก์และทำให้อัตลักษณ์ดัชนีที่มีอยู่ใช้ไม่ได้ (ดูคำเตือนใต้หัวข้อการเลือกผู้ให้บริการ)
</Note>

## พาธหน่วยความจำเพิ่มเติม

| คีย์          | ชนิด       | คำอธิบาย                              |
| ------------ | ---------- | ---------------------------------------- |
| `extraPaths` | `string[]` | ไดเรกทอรีหรือไฟล์เพิ่มเติมที่จะสร้างดัชนี |

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        extraPaths: ["../team-docs", "/srv/shared-notes"],
      },
    },
  },
}
```

พาธอาจเป็นพาธแบบสัมบูรณ์หรือแบบสัมพัทธ์กับเวิร์กสเปซ ระบบจะสแกนไดเรกทอรีแบบเรียกซ้ำเพื่อค้นหาไฟล์ `.md` การจัดการลิงก์สัญลักษณ์ขึ้นอยู่กับแบ็กเอนด์ที่ใช้งานอยู่ โดยเอนจินในตัวจะข้ามลิงก์สัญลักษณ์ ส่วน QMD จะทำตามพฤติกรรมของตัวสแกน QMD ที่อยู่เบื้องหลัง

สำหรับการค้นหาทรานสคริปต์ข้ามเอเจนต์ที่จำกัดขอบเขตตามเอเจนต์ ให้ใช้ `agents.list[].memorySearch.qmd.extraCollections` แทน `memory.qmd.paths` คอลเลกชันเพิ่มเติมเหล่านั้นใช้รูปแบบ `{ path, name, pattern? }` เดียวกัน แต่จะผสานแยกตามเอเจนต์และสามารถคงชื่อที่แชร์ซึ่งระบุไว้อย่างชัดเจนได้เมื่อพาธชี้ออกนอกเวิร์กสเปซปัจจุบัน หากพาธที่แก้ไขแล้วเดียวกันปรากฏทั้งใน `memory.qmd.paths` และ `memorySearch.qmd.extraCollections` QMD จะเก็บรายการแรกและข้ามรายการซ้ำ

ใช้กับไฟล์ใน `extraPaths` เท่านั้น รากหน่วยความจำเริ่มต้นยังคงรองรับเฉพาะ Markdown ต้องใช้ `gemini-embedding-2-preview` โดย `fallback` ต้องเป็น `"none"`
</Note>

รูปแบบที่รองรับ: `.jpg`, `.jpeg`, `.png`, `.webp`, `.gif`, `.heic`, `.heif` (รูปภาพ); `.mp3`, `.wav`, `.ogg`, `.opus`, `.m4a`, `.aac`, `.flac` (เสียง)

## แคช Embedding

| คีย์                | ชนิด      | ค่าเริ่มต้น | คำอธิบาย                                  |
| ------------------ | --------- | ------- | -------------------------------------------- |
| `cache.enabled`    | `boolean` | `true`  | แคช Embedding ของส่วนข้อความใน SQLite             |
| `cache.maxEntries` | `number`  | ไม่ได้ตั้งค่า   | ขีดจำกัดบนโดยประมาณสำหรับ Embedding ที่แคชไว้ |

ป้องกันการสร้าง Embedding ซ้ำสำหรับข้อความที่ไม่เปลี่ยนแปลงระหว่างการสร้างดัชนีใหม่หรือการอัปเดตทรานสคริปต์ ปล่อย `maxEntries` ไว้โดยไม่ตั้งค่าเพื่อใช้แคชแบบไม่จำกัด หรือตั้งค่าเมื่อการเติบโตของพื้นที่ดิสก์สำคัญกว่าความเร็วสูงสุดในการสร้างดัชนีใหม่ เมื่อตั้งค่าแล้ว ระบบจะล้างรายการที่เก่าที่สุดก่อน (ตามเวลาที่อัปเดตล่าสุด) เมื่อแคชเกินขีดจำกัด

## การสร้างดัชนีแบบแบตช์

| คีย์                           | ชนิด      | ค่าเริ่มต้น | คำอธิบาย                |
| ----------------------------- | --------- | ------- | -------------------------- |
| `remote.nonBatchConcurrency`  | `number`  | `4`     | Embedding แบบอินไลน์ที่ทำงานขนานกัน |
| `remote.batch.enabled`        | `boolean` | `false` | เปิดใช้ API การสร้าง Embedding แบบแบตช์ |
| `remote.batch.concurrency`    | `number`  | `2`     | งานแบตช์ที่ทำงานขนานกัน        |
| `remote.batch.wait`           | `boolean` | `true`  | รอให้แบตช์เสร็จสมบูรณ์  |
| `remote.batch.pollIntervalMs` | `number`  | `2000`  | ช่วงเวลาการสำรวจสถานะ              |
| `remote.batch.timeoutMinutes` | `number`  | `60`    | ระยะหมดเวลาของแบตช์              |

ใช้ได้กับ `gemini`, `openai` และ `voyage` โดยทั่วไปแบตช์ของ OpenAI เร็วที่สุดและมีค่าใช้จ่ายต่ำที่สุดสำหรับการเติมข้อมูลย้อนหลังขนาดใหญ่

`remote.nonBatchConcurrency` ควบคุมการเรียก Embedding แบบอินไลน์ที่ผู้ให้บริการในเครื่อง/โฮสต์เองใช้ รวมถึงผู้ให้บริการแบบโฮสต์เมื่อ API แบบแบตช์ของผู้ให้บริการไม่ได้เปิดใช้งาน Ollama ใช้ค่าเริ่มต้นเป็น `1` สำหรับการสร้างดัชนีที่ไม่ใช่แบบแบตช์ เพื่อหลีกเลี่ยงการสร้างภาระมากเกินไปแก่โฮสต์ภายในเครื่องขนาดเล็ก ให้ตั้งค่าที่สูงขึ้นบนเครื่องขนาดใหญ่

ค่านี้แยกจาก `sync.embeddingBatchTimeoutSeconds` ซึ่งควบคุมระยะหมดเวลาสำหรับการเรียก Embedding แบบอินไลน์

## การค้นหาหน่วยความจำเซสชัน (ขั้นทดลอง)

สร้างดัชนีทรานสคริปต์ของเซสชันและแสดงผลผ่าน `memory_search`:

| คีย์                           | ชนิด       | ค่าเริ่มต้น      | คำอธิบาย                             |
| ----------------------------- | ---------- | ------------ | --------------------------------------- |
| `experimental.sessionMemory`  | `boolean`  | `false`      | เปิดใช้การสร้างดัชนีเซสชัน                 |
| `sources`                     | `string[]` | `["memory"]` | เพิ่ม `"sessions"` เพื่อรวมทรานสคริปต์ |
| `sync.sessions.deltaBytes`    | `number`   | `100000`     | เกณฑ์จำนวนไบต์สำหรับการสร้างดัชนีใหม่              |
| `sync.sessions.deltaMessages` | `number`   | `50`         | เกณฑ์จำนวนข้อความสำหรับการสร้างดัชนีใหม่           |

<Warning>
การสร้างดัชนีเซสชันเป็นแบบเลือกเปิดใช้และทำงานแบบอะซิงโครนัส ผลลัพธ์อาจล้าสมัยเล็กน้อย บันทึกเซสชันจัดเก็บอยู่บนดิสก์ ดังนั้นให้ถือว่าการเข้าถึงระบบไฟล์เป็นขอบเขตความเชื่อถือ
</Warning>

การค้นหาทรานสคริปต์เซสชันตามปกติที่โมเดลเรียกใช้จะปฏิบัติตาม
[`tools.sessions.visibility`](/th/gateway/config-tools#toolssessions) การมองเห็นเริ่มต้น
`tree` เปิดเผยเซสชันปัจจุบัน เซสชันที่เซสชันนี้สร้างขึ้น และ
เซสชันกลุ่มของเอเจนต์เดียวกันที่เฝ้าดูผ่านการรับรู้กลุ่มโดยรอบ ส่วนเซสชันอื่นที่
ไม่เกี่ยวข้องต้องใช้การมองเห็น `agent` (หรือ `all` เฉพาะเมื่อจำเป็นต้อง
เรียกคืนข้อมูลข้ามเอเจนต์ด้วยและนโยบายระหว่างเอเจนต์อนุญาต)

`rememberAcrossConversations` ไม่ได้ขยายการตั้งค่านั้น แต่ให้
การอนุญาตแยกต่างหากที่ใช้เฉพาะขณะรันไทม์ โดยจำกัดอยู่ที่ทรานสคริปต์ส่วนตัว
ของเอเจนต์เดียวกันระหว่างการทำงาน Active Memory ที่มีขอบเขตจำกัด

ตัวอย่างด้านล่างวางการตั้งค่าเหล่านี้ไว้ภายใต้ `agents.defaults` และยังสามารถ
ใช้การตั้งค่า `memorySearch` ที่เทียบเท่ากันในการกำหนดค่าทับรายเอเจนต์ เมื่อควรให้เอเจนต์เพียงตัวเดียว
สร้างดัชนีและค้นหาทรานสคริปต์เซสชัน

สำหรับการเรียกคืนข้อมูลจาก Gateway ไปยังข้อความส่วนตัวภายในเอเจนต์เดียวกัน:

<Tabs>
  <Tab title="แบ็กเอนด์ในตัว">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
  <Tab title="แบ็กเอนด์ QMD">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      memory: {
        backend: "qmd",
        qmd: {
          sessions: { enabled: true },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
</Tabs>

เมื่อใช้ QMD ค่า `agents.defaults.memorySearch.experimental.sessionMemory` และ
`sources: ["sessions"]` เพียงอย่างเดียวจะไม่ส่งออกทรานสคริปต์ไปยัง QMD ต้องตั้งค่า
`memory.qmd.sessions.enabled: true` ด้วยเช่นกัน การตั้งค่าระดับสูงกว่า
`rememberAcrossConversations: true` เป็นข้อยกเว้น โดยจะเปิดใช้การส่งออกเซสชัน
QMD ที่จำเป็นสำหรับเอเจนต์นั้นโดยนัย การส่งออกโดยนัยยังคงเป็นส่วนตัว:
จะใช้ตำแหน่งส่งออกภายในเริ่มต้นเสมอ (`sessions.exportDir` ที่กำหนดค่าไว้
ใช้กับการส่งออกโดยชัดแจ้งเท่านั้น) จะถูกค้นหาเฉพาะ
ระหว่างการเรียกคืนข้อมูลข้ามบทสนทนาของเอเจนต์นั้น และ `memory_get`
ตามปกติจะไม่สามารถอ่านได้ ส่วน `memory.qmd.sessions.enabled: true`
แบบชัดแจ้งจะคงพฤติกรรมเดิมและทำให้ทรานสคริปต์ที่ส่งออก
เป็นส่วนหนึ่งของคลังหน่วยความจำตามปกติ

## การกำหนดค่าแบ็กเอนด์ QMD

ตั้งค่า `memory.backend = "qmd"` เพื่อเปิดใช้ การตั้งค่า QMD ทั้งหมดอยู่ภายใต้ `memory.qmd`:

| คีย์                      | ชนิด      | ค่าเริ่มต้น  | คำอธิบาย                                                                           |
| ------------------------ | --------- | -------- | ------------------------------------------------------------------------------------- |
| `command`                | `string`  | `qmd`    | พาธไฟล์ปฏิบัติการ QMD; ตั้งเป็นพาธแบบสัมบูรณ์เมื่อ `PATH` ของบริการแตกต่างจากเชลล์ของคุณ |
| `searchMode`             | `string`  | `search` | คำสั่งค้นหา: `search`, `vsearch`, `query`                                          |
| `rerank`                 | `boolean` | --       | ตั้งเป็น `false` ร่วมกับ `searchMode: "query"` และ QMD 2.1+ เพื่อข้ามการจัดอันดับใหม่ของ QMD          |
| `includeDefaultMemory`   | `boolean` | `true`   | สร้างดัชนี `MEMORY.md` + `memory/**/*.md` โดยอัตโนมัติ                                             |
| `paths[]`                | `array`   | --       | พาธเพิ่มเติม: `{ name, path, pattern? }`                                               |
| `sessions.enabled`       | `boolean` | `false`  | ส่งออกทรานสคริปต์เซสชันไปยัง QMD                                                   |
| `sessions.retentionDays` | `number`  | --       | ระยะเวลาการเก็บรักษาทรานสคริปต์                                                                  |
| `sessions.exportDir`     | `string`  | --       | ไดเรกทอรีส่งออก                                                                      |

`searchMode: "search"` ใช้เฉพาะการค้นหาเชิงคำศัพท์/BM25 เท่านั้น OpenClaw จะไม่เรียกใช้การตรวจสอบความพร้อมของเวกเตอร์เชิงความหมายหรือการบำรุงรักษา Embedding ของ QMD สำหรับโหมดนั้น รวมถึงระหว่าง `memory status --deep`; `vsearch` และ `query` ยังคงต้องใช้ความพร้อมของเวกเตอร์และ Embedding ของ QMD

`rerank: false` เปลี่ยนเฉพาะโหมด QMD `query` และต้องใช้ QMD 2.1 หรือใหม่กว่า ในโหมด CLI โดยตรง OpenClaw จะส่ง `--no-rerank`; ในโหมด MCP ที่ใช้ mcporter เป็นแบ็กเอนด์ จะส่ง `rerank: false` ไปยังเครื่องมือค้นหาแบบรวมของ QMD เว้นค่านี้ไว้เพื่อใช้พฤติกรรมการจัดอันดับผลการค้นหาใหม่ตามค่าเริ่มต้นของ QMD

OpenClaw เลือกใช้รูปแบบคอลเลกชันและคำค้น MCP ปัจจุบันของ QMD แต่ยังคงรองรับ QMD รุ่นเก่าด้วยการลองใช้แฟล็กรูปแบบคอลเลกชันที่เข้ากันได้และชื่อเครื่องมือ MCP รุ่นเก่าเมื่อจำเป็น เมื่อ QMD ระบุว่ารองรับตัวกรองหลายคอลเลกชัน ระบบจะค้นหาคอลเลกชันจากแหล่งเดียวกันด้วยกระบวนการ QMD เดียว ส่วน QMD รุ่นเก่าจะยังคงใช้เส้นทางความเข้ากันได้แบบแยกต่อคอลเลกชัน แหล่งเดียวกันหมายถึงคอลเลกชันหน่วยความจำถาวร (ไฟล์หน่วยความจำเริ่มต้นรวมถึงพาธที่กำหนดเอง) จะถูกจัดกลุ่มเข้าด้วยกัน ขณะที่คอลเลกชันบทสนทนาของเซสชันยังคงเป็นอีกกลุ่มหนึ่ง เพื่อให้การกระจายแหล่งข้อมูลยังคงมีอินพุตทั้งสองประเภท

<Note>
การแทนที่โมเดล QMD ต้องกำหนดที่ฝั่ง QMD ไม่ใช่ในการกำหนดค่า OpenClaw หากต้องการแทนที่โมเดลของ QMD แบบส่วนกลาง ให้ตั้งค่าตัวแปรสภาพแวดล้อม เช่น `QMD_EMBED_MODEL`, `QMD_RERANK_MODEL` และ `QMD_GENERATE_MODEL` ในสภาพแวดล้อมรันไทม์ของ Gateway
</Note>

### การผสานรวม mcporter

ทั้งหมดอยู่ภายใต้ `memory.qmd.mcporter` กำหนดเส้นทางการค้นหา QMD ผ่านดีมอน MCP `mcporter` ที่ทำงานต่อเนื่อง แทนการสร้าง `qmd` ใหม่ต่อคำค้น ซึ่งช่วยลดค่าใช้จ่ายในการเริ่มต้นแบบเย็นสำหรับโมเดลขนาดใหญ่

| คีย์           | ประเภท      | ค่าเริ่มต้น | คำอธิบาย                                                            |
| ------------- | --------- | ------- | ---------------------------------------------------------------------- |
| `enabled`     | `boolean` | `false` | กำหนดเส้นทางการเรียก QMD ผ่าน mcporter แทนการสร้าง `qmd` ใหม่ต่อคำขอ |
| `serverName`  | `string`  | `qmd`   | ชื่อเซิร์ฟเวอร์ mcporter ที่เรียกใช้ `qmd mcp` พร้อม `lifecycle: keep-alive`  |
| `startDaemon` | `boolean` | `true`  | เริ่มดีมอน mcporter โดยอัตโนมัติเมื่อ `enabled` เป็น true         |

ต้องติดตั้ง `mcporter` และให้เข้าถึงได้ผ่าน PATH รวมถึงกำหนดค่าเซิร์ฟเวอร์ mcporter ที่เรียกใช้ `qmd mcp` ไว้แล้ว คงสถานะปิดใช้งานไว้สำหรับการตั้งค่าภายในเครื่องแบบเรียบง่ายที่ยอมรับต้นทุนการสร้างกระบวนการต่อคำค้นได้

    | `update.waitForBootSync`  | `boolean` | `false` | ระงับการเปิดตัวจัดการไว้จนกว่าการรีเฟรชครั้งแรกจะเสร็จสมบูรณ์ |
    | `update.embedInterval`    | `string`  | `60m`   | รอบเวลาการฝังข้อมูลแยกต่างหาก                |
    | `update.commandTimeoutMs` | `number`  | `30000` | ระยะหมดเวลาสำหรับคำสั่งบำรุงรักษา QMD (แสดงรายการ/เพิ่มคอลเลกชัน) |
    | `update.updateTimeoutMs`  | `number`  | `120000` | ระยะหมดเวลาสำหรับแต่ละรอบ `qmd update`   |
    | `update.embedTimeoutMs`   | `number`  | `120000` | ระยะหมดเวลาสำหรับแต่ละรอบ `qmd embed`    |
  </Accordion>
  <Accordion title="ขีดจำกัด">
    | คีย์                       | ประเภท     | ค่าเริ่มต้น | คำอธิบาย                |
    | --------------------------- | -------- | ------- | ------------------------------ |
    | `limits.maxResults`       | `number` | `4`     | จำนวนผลการค้นหาสูงสุด         |
    | `limits.maxSnippetChars`  | `number` | `450`   | จำกัดความยาวข้อความที่ตัดมา       |
    | `limits.maxInjectedChars` | `number` | `2200`  | จำกัดจำนวนอักขระที่แทรกทั้งหมด |
    | `limits.timeoutMs`        | `number` | `4000`  | ระยะหมดเวลาของคำสั่ง QMD ระหว่างการค้นหาที่ใช้ QMD เป็นแบ็กเอนด์ รวมถึง `memory_search`; งานตั้งค่า ซิงค์ การใช้ทางเลือกสำรองในตัว และงานเสริมยังคงใช้กำหนดเวลาสิ้นสุดเริ่มต้นของเครื่องมือ |
  </Accordion>
  <Accordion title="ขอบเขต">
    ควบคุมว่าเซสชันใดสามารถรับผลการค้นหา QMD ได้ ใช้สคีมาเดียวกับ [`session.sendPolicy`](/th/gateway/config-agents#session):

    ```json5
    {
      memory: {
        qmd: {
          scope: {
            default: "deny",
            rules: [{ action: "allow", match: { chatType: "direct" } }],
          },
        },
      },
    }
    ```

    ค่าเริ่มต้นที่จัดส่งรองรับเฉพาะ DM/ข้อความโดยตรง โดยปฏิเสธกลุ่มและประเภทช่องทางอื่น `match.keyPrefix` ตรงกับคีย์เซสชันที่ปรับให้อยู่ในรูปแบบมาตรฐานแล้ว; `match.rawKeyPrefix` ตรงกับคีย์ดิบซึ่งรวม `agent:<id>:`

  </Accordion>
  <Accordion title="การอ้างอิง">
    `memory.citations` ใช้กับแบ็กเอนด์ทั้งหมด:

    | ค่า            | พฤติกรรม                                            |
    | ------------------ | ------------------------------------------------------ |
    | `auto` (ค่าเริ่มต้น) | รวมส่วนท้าย `Source: <path#line>` ในข้อความที่ตัดมา    |
    | `on`             | รวมส่วนท้ายเสมอ                               |
    | `off`            | ละเว้นส่วนท้าย (ยังคงส่งพาธให้เอเจนต์ภายใน) |

  </Accordion>
</AccordionGroup>

เมื่อเปิดใช้การเริ่มต้น QMD ขณะ Gateway เริ่มทำงาน OpenClaw จะเริ่ม QMD เฉพาะสำหรับเอเจนต์ที่มีสิทธิ์เท่านั้น หาก `update.onBoot` เป็น true และไม่ได้กำหนดการบำรุงรักษาตามช่วงเวลา/การฝังข้อมูล การเริ่มทำงานจะใช้ตัวจัดการแบบครั้งเดียวสำหรับการรีเฟรชขณะบูตแล้วปิดตัวจัดการนั้น หากกำหนดช่วงเวลาการอัปเดตหรือการฝังข้อมูลไว้ การเริ่มทำงานจะเปิดตัวจัดการ QMD ที่ทำงานต่อเนื่อง เพื่อให้เป็นเจ้าของตัวเฝ้าดูและตัวจับเวลาตามช่วงเวลา; `update.onBoot: false` จะข้ามเฉพาะการรีเฟรชทันทีขณะบูต

### ตัวอย่าง QMD แบบเต็ม

```json5
{
  memory: {
    backend: "qmd",
    citations: "auto",
    qmd: {
      includeDefaultMemory: true,
      update: { interval: "5m", debounceMs: 15000 },
      limits: { maxResults: 4, timeoutMs: 4000 },
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

## Dreaming

Dreaming กำหนดค่าภายใต้ `plugins.entries.memory-core.config.dreaming` ไม่ใช่ภายใต้ `agents.defaults.memorySearch`

Dreaming ทำงานเป็นการกวาดตรวจตามกำหนดการหนึ่งครั้ง และใช้ระยะ light/deep/REM ภายในเป็นรายละเอียดการนำไปใช้

สำหรับพฤติกรรมเชิงแนวคิดและคำสั่งแบบสแลช โปรดดู [Dreaming](/th/concepts/dreaming)

### การตั้งค่าของผู้ใช้

| คีย์                                    | ประเภท      | ค่าเริ่มต้น       | คำอธิบาย                                                                                                                      |
| -------------------------------------- | --------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`                              | `boolean` | `false`       | เปิดหรือปิด Dreaming ทั้งหมด                                                                                              |
| `frequency`                            | `string`  | `0 3 * * *`   | รอบเวลา cron ที่เลือกกำหนดได้สำหรับการกวาดตรวจ Dreaming แบบเต็ม                                                                                |
| `model`                                | `string`  | โมเดลเริ่มต้น | การแทนที่โมเดลซับเอเจนต์ Dream Diary ที่เลือกกำหนดได้                                                                                     |
| `phases.deep.maxPromotedSnippetTokens` | `number`  | `160`         | จำนวนโทเค็นโดยประมาณสูงสุดที่เก็บไว้จากข้อความเรียกคืนระยะสั้นแต่ละส่วนซึ่งถูกยกระดับไปยัง `MEMORY.md`; เมทาดาทาแหล่งที่มายังคงมองเห็นได้ |

### ตัวอย่าง

```json5
{
  plugins: {
    entries: {
      "memory-core": {
        subagent: {
          allowModelOverride: true,
          allowedModels: ["anthropic/claude-sonnet-4-6"],
        },
        config: {
          dreaming: {
            enabled: true,
            frequency: "0 3 * * *",
            model: "anthropic/claude-sonnet-4-6",
          },
        },
      },
    },
  },
}
```

<Note>
- Dreaming เขียนสถานะเครื่องลงใน `memory/.dreams/`
- Dreaming เขียนผลลัพธ์เชิงบรรยายที่มนุษย์อ่านได้ลงใน `DREAMS.md` (หรือ `dreams.md` ที่มีอยู่)
- `dreaming.model` ใช้เกตความเชื่อถือของซับเอเจนต์ Plugin ที่มีอยู่; ตั้งค่า `plugins.entries.memory-core.subagent.allowModelOverride: true` ก่อนเปิดใช้งาน
- Dream Diary จะลองอีกครั้งหนึ่งครั้งด้วยโมเดลเริ่มต้นของเซสชันเมื่อโมเดลที่กำหนดค่าไว้ไม่พร้อมใช้งาน ความล้มเหลวด้านความเชื่อถือหรือรายการอนุญาตจะถูกบันทึก และระบบจะไม่ลองใหม่โดยไม่แจ้ง
- นโยบายและเกณฑ์ของระยะ light/deep/REM เป็นพฤติกรรมภายใน ไม่ใช่การกำหนดค่าที่ผู้ใช้เข้าถึงได้

</Note>
