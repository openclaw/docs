# แผนการปรับปรุงแอปพลิเคชันให้ทันสมัย

## เป้าหมาย

พัฒนาแอปพลิเคชันไปสู่ผลิตภัณฑ์ที่สะอาดขึ้น เร็วขึ้น และดูแลรักษาได้ง่ายขึ้น โดยไม่
ทำให้เวิร์กโฟลว์ปัจจุบันเสียหายหรือซ่อนความเสี่ยงไว้ในการรีแฟกเตอร์ขนาดใหญ่ งานควร
ถูกส่งมอบเป็นส่วนย่อยที่เล็กและตรวจรีวิวได้ พร้อมหลักฐานสำหรับทุกพื้นผิวที่มีการแก้ไข

## หลักการ

- คงสถาปัตยกรรมปัจจุบันไว้ เว้นแต่จะพิสูจน์ได้ชัดเจนว่าขอบเขตใดก่อให้เกิดความปั่นป่วน
  ต้นทุนด้านประสิทธิภาพ หรือบั๊กที่ผู้ใช้มองเห็นได้
- ให้ความสำคัญกับแพตช์ที่เล็กที่สุดและถูกต้องสำหรับแต่ละปัญหาก่อน แล้วจึงทำซ้ำ
- แยกการแก้ไขที่จำเป็นออกจากงานเก็บรายละเอียดเพิ่มเติม เพื่อให้ผู้ดูแลสามารถส่งมอบงาน
  ที่มีมูลค่าสูงได้โดยไม่ต้องรอการตัดสินใจเชิงอัตวิสัย
- คงพฤติกรรมที่กระทบต่อ Plugin ให้มีเอกสารกำกับและเข้ากันได้ย้อนหลัง
- ตรวจสอบพฤติกรรมที่ใช้งานจริง สัญญาของ dependency และการทดสอบ ก่อนอ้างว่า
  แก้ regression ได้แล้ว
- ปรับปรุงเส้นทางหลักของผู้ใช้ก่อน: onboarding, auth, แชต, การตั้งค่าผู้ให้บริการ,
  การจัดการ Plugin และการวินิจฉัยปัญหา

## ระยะที่ 1: ตรวจสอบสถานะพื้นฐาน

สำรวจแอปพลิเคชันปัจจุบันก่อนทำการเปลี่ยนแปลง

- ระบุเวิร์กโฟลว์หลักของผู้ใช้และพื้นผิวโค้ดที่เป็นเจ้าของเวิร์กโฟลว์เหล่านั้น
- แสดงรายการ affordance ที่ตายแล้ว การตั้งค่าที่ซ้ำซ้อน สถานะข้อผิดพลาดที่ไม่ชัดเจน และ
  เส้นทางการเรนเดอร์ที่มีต้นทุนสูง
- บันทึกคำสั่งตรวจสอบปัจจุบันสำหรับแต่ละพื้นผิว
- จัดหมวดปัญหาเป็น จำเป็น แนะนำ หรือไม่บังคับ
- บันทึก blocker ที่ทราบแล้วซึ่งต้องมีการทบทวนจาก owner โดยเฉพาะการเปลี่ยนแปลงด้าน API,
  ความปลอดภัย การรีลีส และสัญญา Plugin

นิยามของงานเสร็จสมบูรณ์:

- มีรายการประเด็นเดียวพร้อมการอ้างอิงไฟล์แบบ repo-root
- แต่ละประเด็นมีระดับความรุนแรง พื้นผิวเจ้าของ ผลกระทบที่คาดต่อผู้ใช้ และแนวทางตรวจสอบที่เสนอ
- ไม่มีรายการเก็บกวาดเชิงคาดเดาปะปนอยู่กับการแก้ไขที่จำเป็น

## ระยะที่ 2: ปรับปรุงผลิตภัณฑ์และ UX

จัดลำดับความสำคัญให้เวิร์กโฟลว์ที่ผู้ใช้มองเห็นได้ และลดความสับสน

- ปรับถ้อยคำ onboarding และสถานะว่างให้ชัดเจนขึ้นรอบการยืนยันตัวตนโมเดล สถานะ Gateway
  และการตั้งค่า Plugin
- นำ affordance ที่ตายแล้วออกหรือปิดใช้งานเมื่อไม่มีการดำเนินการใดที่ทำได้
- คงการกระทำที่สำคัญให้มองเห็นได้ในทุกความกว้างที่รองรับ แทนการซ่อนไว้หลังสมมติฐาน
  ด้านเลย์เอาต์ที่เปราะบาง
- รวมภาษาสถานะที่ซ้ำกันให้เหลือแหล่งความจริงเดียว เพื่อให้ข้อผิดพลาดมีต้นทางเดียว
- เพิ่ม progressive disclosure สำหรับการตั้งค่าขั้นสูง โดยยังคงให้การตั้งค่าหลักทำได้รวดเร็ว

การตรวจสอบที่แนะนำ:

- ทดสอบ happy path แบบแมนนวลสำหรับการตั้งค่าครั้งแรกและการเริ่มใช้งานของผู้ใช้เดิม
- การทดสอบแบบเจาะจงสำหรับตรรกะเกี่ยวกับ routing, การคงอยู่ของ config หรือการคำนวณสถานะ
- ภาพหน้าจอเบราว์เซอร์สำหรับพื้นผิว responsive ที่เปลี่ยนแปลง

## ระยะที่ 3: กระชับสถาปัตยกรรมฝั่งฟรอนต์เอนด์

เพิ่มความสามารถในการดูแลรักษาโดยไม่เขียนใหม่ในวงกว้าง

- ย้ายการแปลงสถานะ UI ที่ทำซ้ำ ไปไว้ใน helper แบบมี type ที่แคบและชัดเจน
- แยกความรับผิดชอบด้านการดึงข้อมูล การคงอยู่ของข้อมูล และการแสดงผลออกจากกัน
- ใช้ hooks, stores และรูปแบบคอมโพเนนต์ที่มีอยู่แล้วก่อนเพิ่ม abstraction ใหม่
- แยกคอมโพเนนต์ที่ใหญ่เกินไปเฉพาะเมื่อช่วยลด coupling หรือทำให้การทดสอบชัดเจนขึ้น
- หลีกเลี่ยงการเพิ่ม global state ขนาดใหญ่สำหรับการโต้ตอบเฉพาะภายใน panel

แนวป้องกันที่จำเป็น:

- ห้ามเปลี่ยนพฤติกรรมสาธารณะโดยเป็นผลข้างเคียงจากการแยกไฟล์
- คงพฤติกรรมด้าน accessibility สำหรับเมนู ไดอะล็อก แท็บ และการนำทางด้วยคีย์บอร์ด
- ตรวจสอบว่าสถานะ loading, empty, error และ optimistic ยังคงแสดงผลได้

## ระยะที่ 4: ประสิทธิภาพและความเชื่อถือได้

มุ่งแก้ปัญหาที่วัดได้ แทนการเพิ่มประสิทธิภาพเชิงทฤษฎีแบบกว้าง

- วัดต้นทุนของการเริ่มระบบ การเปลี่ยน route รายการขนาดใหญ่ และทรานสคริปต์แชต
- แทนที่ข้อมูลอนุพันธ์ที่มีต้นทุนสูงและคำนวณซ้ำด้วย memoized selectors หรือ cached
  helpers เมื่อการทำ profiling พิสูจน์ว่าคุ้มค่า
- ลดการสแกนเครือข่ายหรือระบบไฟล์ที่เลี่ยงได้บน hot path
- คงลำดับที่กำหนดแน่นอนสำหรับอินพุตของ prompt, registry, ไฟล์, Plugin และเครือข่าย
  ก่อนสร้าง payload ของโมเดล
- เพิ่มการทดสอบ regression แบบเบาสำหรับ hot helpers และขอบเขตสัญญา

นิยามของงานเสร็จสมบูรณ์:

- ทุกการเปลี่ยนแปลงด้านประสิทธิภาพต้องบันทึก baseline, ผลกระทบที่คาด, ผลกระทบจริง และ
  ช่องว่างที่ยังเหลือ
- ไม่มีแพตช์ด้านประสิทธิภาพใดถูกส่งมอบโดยอาศัยสัญชาตญาณล้วน เมื่อมีการวัดแบบประหยัดให้ใช้ได้

## ระยะที่ 5: เสริมความแข็งแรงด้าน type, contract และการทดสอบ

ยกระดับความถูกต้องในจุดขอบเขตที่ผู้ใช้และผู้เขียน Plugin ต้องพึ่งพา

- แทนที่สตริง runtime แบบหลวมด้วย discriminated unions หรือรายการโค้ดแบบปิด
- ตรวจสอบอินพุตภายนอกด้วย schema helpers ที่มีอยู่หรือ zod
- เพิ่ม contract tests รอบ manifest ของ Plugin, catalog ของผู้ให้บริการ, ข้อความใน
  gateway protocol และพฤติกรรมการย้าย config
- คงเส้นทางความเข้ากันได้ไว้ในโฟลว์ doctor หรือ repair แทนการย้ายข้อมูลแบบซ่อนเร้นขณะเริ่มระบบ
- หลีกเลี่ยงการผูกกับภายในของ Plugin เฉพาะเพื่อการทดสอบ ใช้ SDK facades และ
  barrels ที่มีเอกสารกำกับ

การตรวจสอบที่แนะนำ:

- `pnpm check:changed`
- การทดสอบแบบเจาะจงสำหรับทุกขอบเขตที่เปลี่ยนแปลง
- `pnpm build` เมื่อมีการเปลี่ยน lazy boundaries, การแพ็กเกจ หรือพื้นผิวที่เผยแพร่

## ระยะที่ 6: ความพร้อมด้านเอกสารและการรีลีส

รักษาเอกสารที่ผู้ใช้เห็นให้สอดคล้องกับพฤติกรรม

- อัปเดตเอกสารเมื่อมีการเปลี่ยนพฤติกรรม API config onboarding หรือ Plugin
- เพิ่มรายการใน changelog เฉพาะเมื่อมีการเปลี่ยนแปลงที่ผู้ใช้มองเห็นได้
- ใช้คำว่า Plugin ในเนื้อหาที่ผู้ใช้เห็น; ใช้ชื่อ package ภายในเท่าที่จำเป็นสำหรับผู้มีส่วนร่วม
- ยืนยันว่าคำแนะนำการรีลีสและการติดตั้งยังคงตรงกับพื้นผิวคำสั่งปัจจุบัน

นิยามของงานเสร็จสมบูรณ์:

- เอกสารที่เกี่ยวข้องได้รับการอัปเดตใน branch เดียวกับการเปลี่ยนพฤติกรรม
- การตรวจสอบ generated docs หรือ API drift ผ่านเมื่อมีการแตะส่วนนั้น
- การส่งต่องานระบุการตรวจสอบที่ข้ามไปและเหตุผลที่ข้ามอย่างชัดเจน

## ส่วนเริ่มต้นที่แนะนำ

เริ่มจากการปรับปรุง Control UI และ onboarding แบบจำกัดขอบเขต:

- ตรวจสอบพื้นผิวการตั้งค่าครั้งแรก ความพร้อมของการยืนยันตัวตนผู้ให้บริการ สถานะ Gateway และ
  การตั้งค่า Plugin
- นำการกระทำที่ตายแล้วออก และทำให้สถานะล้มเหลวชัดเจนขึ้น
- เพิ่มหรืออัปเดตการทดสอบแบบเจาะจงสำหรับการคำนวณสถานะและการคงอยู่ของ config
- รัน `pnpm check:changed`

แนวทางนี้ให้คุณค่าสูงกับผู้ใช้โดยมีความเสี่ยงด้านสถาปัตยกรรมจำกัด

## การอัปเดต skill ฝั่งฟรอนต์เอนด์

ใช้ส่วนนี้เพื่ออัปเดต `SKILL.md` ที่เน้นงานฟรอนต์เอนด์ซึ่งมาพร้อมกับงาน
modernization หากนำแนวทางนี้ไปใช้เป็น skill แบบ repo-local ของ OpenClaw
ให้สร้าง `.agents/skills/openclaw-frontend/SKILL.md` ก่อน คง frontmatter
ที่ควรอยู่ใน skill เป้าหมายนั้นไว้ จากนั้นเพิ่มหรือแทนที่คำแนะนำใน body ด้วย
เนื้อหาต่อไปนี้

```markdown
# มาตรฐานการส่งมอบฝั่งฟรอนต์เอนด์

ใช้ skill นี้เมื่อพัฒนา หรือตรวจทานงาน UI ที่ผู้ใช้มองเห็นได้บน React, Next.js,
desktop webview หรือแอป

## กฎการทำงาน

- เริ่มจากเวิร์กโฟลว์ของผลิตภัณฑ์และรูปแบบโค้ดที่มีอยู่
- ให้ความสำคัญกับแพตช์ที่เล็กที่สุดและถูกต้องซึ่งช่วยปรับปรุงเส้นทางผู้ใช้ปัจจุบัน
- แยกการแก้ไขที่จำเป็นออกจากงานเก็บรายละเอียดเพิ่มเติมใน handoff
- อย่าสร้างหน้า marketing เมื่อคำขอเป็นงานบนพื้นผิวของแอปพลิเคชัน
- คงการกระทำให้มองเห็นและใช้งานได้ในทุกขนาด viewport ที่รองรับ
- นำ affordance ที่ตายแล้วออก แทนการปล่อยคอนโทรลที่ไม่สามารถทำงานได้ไว้
- คงสถานะ loading, empty, error, success และ permission
- ใช้คอมโพเนนต์ hooks stores และไอคอนของ design system ที่มีอยู่ก่อน
  เพิ่ม primitive ใหม่

## เช็กลิสต์การพัฒนา

1. ระบุงานหลักของผู้ใช้และคอมโพเนนต์หรือ route ที่เป็นเจ้าของงานนั้น
2. อ่านรูปแบบคอมโพเนนต์เฉพาะที่นั้นก่อนแก้ไข
3. แพตช์พื้นผิวที่แคบที่สุดซึ่งแก้ปัญหาได้
4. เพิ่มข้อจำกัดด้าน responsive สำหรับคอนโทรลแบบรูปแบบคงที่ แถบเครื่องมือ กริด และ
   ตัวนับ เพื่อให้ข้อความและสถานะ hover ไม่ทำให้เลย์เอาต์เปลี่ยนขนาดโดยไม่คาดคิด
5. รักษาความชัดเจนของความรับผิดชอบด้านการโหลดข้อมูล การคำนวณสถานะ และการเรนเดอร์
6. เพิ่มการทดสอบเมื่อมีการเปลี่ยนตรรกะ การคงอยู่ การกำหนดเส้นทาง permissions หรือ shared helpers
7. ตรวจสอบ happy path หลักและ edge case ที่เกี่ยวข้องที่สุด

## เกณฑ์คุณภาพด้านภาพ

- ข้อความต้องพอดีภายในคอนเทนเนอร์ทั้งบนมือถือและเดสก์ท็อป
- แถบเครื่องมือสามารถตัดบรรทัดได้ แต่คอนโทรลต้องยังเข้าถึงได้
- ปุ่มควรใช้ไอคอนที่คุ้นเคยเมื่อไอคอนสื่อความหมายได้ชัดกว่าข้อความ
```
