OpenClaw ใช้ **การแก้ไขเฉพาะผู้ให้บริการ** กับบันทึกบทสนทนาก่อนการรัน (การสร้างบริบทโมเดล) การปรับส่วนใหญ่นี้เป็นการปรับ **ในหน่วยความจำ** เพื่อให้ตรงตามข้อกำหนดที่เข้มงวดของผู้ให้บริการ ขั้นตอนซ่อมแซมไฟล์เซสชันแยกต่างหากอาจเขียน JSONL ที่จัดเก็บไว้ใหม่ก่อนโหลดเซสชันด้วย แต่ทำเฉพาะบรรทัดที่ผิดรูปแบบหรือเทิร์นที่คงอยู่ซึ่งเป็นเรกคอร์ดถาวรที่ไม่ถูกต้องเท่านั้น คำตอบของผู้ช่วยที่ส่งแล้วจะถูกเก็บรักษาไว้บนดิสก์ การตัด assistant-prefill เฉพาะผู้ให้บริการเกิดขึ้นเฉพาะขณะสร้างเพย์โหลดขาออกเท่านั้น เมื่อมีการซ่อมแซม ไฟล์ต้นฉบับจะถูกสำรองไว้ข้างไฟล์เซสชัน

ขอบเขตรวมถึง:

- บริบทพรอมป์เฉพาะรันไทม์ที่ไม่เข้าไปอยู่ในเทิร์นบันทึกบทสนทนาที่ผู้ใช้มองเห็น
- การทำความสะอาด id ของการเรียกเครื่องมือ
- การตรวจสอบอินพุตของการเรียกเครื่องมือ
- การซ่อมแซมการจับคู่ผลลัพธ์เครื่องมือ
- การตรวจสอบ / การจัดลำดับเทิร์น
- การล้าง signature ของความคิด
- การล้าง signature ของ thinking
- การทำความสะอาดเพย์โหลดรูปภาพ
- การล้างบล็อกข้อความว่างก่อน replay ของผู้ให้บริการ
- การติดแท็กที่มาของอินพุตผู้ใช้ (สำหรับพรอมป์ที่ถูกส่งต่อระหว่างเซสชัน)
- การซ่อมแซมเทิร์นข้อผิดพลาดของผู้ช่วยที่ว่างเปล่าสำหรับ replay ของ Bedrock Converse

หากคุณต้องการรายละเอียดการจัดเก็บบันทึกบทสนทนา โปรดดู:

- [เจาะลึกการจัดการเซสชัน](/th/reference/session-management-compaction)

## กฎสากล: บริบทรันไทม์ไม่ใช่บันทึกบทสนทนาของผู้ใช้

สามารถเพิ่มบริบทรันไทม์/ระบบลงในพรอมป์โมเดลสำหรับเทิร์นหนึ่งได้ แต่ไม่ใช่เนื้อหาที่ผู้ใช้ปลายทางเขียนขึ้น OpenClaw เก็บเนื้อหาพรอมป์ที่ใช้สำหรับบันทึกบทสนทนาแยกต่างหากสำหรับคำตอบของ Gateway, followup ที่เข้าคิว, ACP, CLI และการรัน Pi แบบฝัง เทิร์นผู้ใช้ที่มองเห็นได้ซึ่งจัดเก็บไว้จะใช้เนื้อหาบันทึกบทสนทนานั้นแทนพรอมป์ที่เสริมด้วยรันไทม์

สำหรับเซสชันเดิมที่เคยบันทึก wrapper ของรันไทม์ไว้แล้ว พื้นผิวประวัติของ Gateway จะใช้การฉายผลสำหรับแสดงผลก่อนส่งข้อความกลับไปยัง WebChat, TUI, REST หรือไคลเอนต์ SSE

- การเลือกนโยบาย: `src/agents/transcript-policy.ts`

แยกจากสุขอนามัยของบันทึกบทสนทนา ไฟล์เซสชันจะถูกซ่อมแซม (หากจำเป็น) ก่อนโหลด:

- `repairSessionFileIfNeeded` ใน `src/agents/session-file-repair.ts`
- เรียกจาก `run/attempt.ts` และ `compact.ts` (runner แบบฝัง)

## กฎสากล: การทำความสะอาดรูปภาพ

เพย์โหลดรูปภาพจะถูกทำความสะอาดเสมอเพื่อป้องกันการปฏิเสธฝั่งผู้ให้บริการจากข้อจำกัดด้านขนาด (ลดขนาด/บีบอัดรูปภาพ base64 ที่ใหญ่เกินไปใหม่)

สิ่งนี้ยังช่วยควบคุมแรงกดดันด้านโทเคนจากรูปภาพสำหรับโมเดลที่รองรับการมองเห็น โดยทั่วไปขนาดสูงสุดที่ต่ำกว่าจะลดการใช้โทเคน ส่วนขนาดที่สูงกว่าจะรักษารายละเอียดไว้

การใช้งาน:

- `sanitizeSessionMessagesImages` ใน `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` ใน `src/agents/tool-images.ts`
- ด้านสูงสุดของรูปภาพกำหนดค่าได้ผ่าน `agents.defaults.imageMaxDimensionPx` (ค่าเริ่มต้น: `1200`)
- บล็อกข้อความว่างจะถูกลบออกขณะขั้นตอนนี้เดินผ่านเนื้อหา replay เทิร์นผู้ช่วยที่กลายเป็นว่างเปล่าจะถูกตัดออกจากสำเนา replay ส่วนเทิร์นผู้ใช้และผลลัพธ์เครื่องมือที่กลายเป็นว่างเปล่าจะได้รับ placeholder เนื้อหาที่ถูกละไว้ซึ่งไม่ว่างเปล่า

## กฎสากล: การเรียกเครื่องมือผิดรูปแบบ

บล็อกการเรียกเครื่องมือของผู้ช่วยที่ไม่มีทั้ง `input` และ `arguments` จะถูกตัดออกก่อนสร้างบริบทโมเดล สิ่งนี้ป้องกันการปฏิเสธจากผู้ให้บริการจากการเรียกเครื่องมือที่คงอยู่แบบไม่สมบูรณ์ (เช่น หลังเกิดความล้มเหลวจาก rate limit)

การใช้งาน:

- `sanitizeToolCallInputs` ใน `src/agents/session-transcript-repair.ts`
- ใช้ใน `sanitizeSessionHistory` ใน `src/agents/pi-embedded-runner/replay-history.ts`

## กฎสากล: ที่มาของอินพุตระหว่างเซสชัน

เมื่อ agent ส่งพรอมป์เข้าไปยังอีกเซสชันผ่าน `sessions_send` (รวมถึงขั้นตอนตอบกลับ/ประกาศจาก agent ถึง agent) OpenClaw จะคงอยู่เทิร์นผู้ใช้ที่สร้างขึ้นพร้อมกับ:

- `message.provenance.kind = "inter_session"`

OpenClaw ยังเติม marker `[Inter-session message ... isUser=false]` ในเทิร์นเดียวกันไว้หน้าข้อความพรอมป์ที่ถูกส่งต่อ เพื่อให้การเรียกโมเดลที่ทำงานอยู่แยกแยะเอาต์พุตจากเซสชันอื่นออกจากคำสั่งของผู้ใช้ปลายทางภายนอกได้ marker นี้รวมเซสชันต้นทาง ช่องทาง และเครื่องมือเมื่อมีให้ใช้ บันทึกบทสนทนายังคงใช้ `role: "user"` เพื่อความเข้ากันได้กับผู้ให้บริการ แต่ทั้งข้อความที่มองเห็นได้และ metadata ที่มาจะระบุว่าเทิร์นนั้นเป็นข้อมูลระหว่างเซสชัน

ระหว่างการสร้างบริบทใหม่ OpenClaw ใช้ marker เดียวกันกับเทิร์นผู้ใช้ระหว่างเซสชันที่คงอยู่เก่ากว่าซึ่งมีเพียง metadata ที่มา

## เมทริกซ์ผู้ให้บริการ (พฤติกรรมปัจจุบัน)

**OpenAI / OpenAI Codex**

- ทำความสะอาดรูปภาพเท่านั้น
- ตัด signature การให้เหตุผลที่เป็นกำพร้า (รายการ reasoning เดี่ยวที่ไม่มีบล็อกเนื้อหาตามมา) สำหรับบันทึกบทสนทนา OpenAI Responses/Codex และตัด reasoning ของ OpenAI ที่ replay ได้หลังการสลับเส้นทางโมเดล
- เก็บรักษาเพย์โหลดรายการ reasoning ของ OpenAI Responses ที่ replay ได้ รวมถึงรายการ summary ว่างแบบเข้ารหัส เพื่อให้ replay แบบ manual/WebSocket ยังคงมีสถานะ `rs_*` ที่จำเป็นจับคู่กับรายการเอาต์พุตของผู้ช่วย
- ไม่มีการทำความสะอาด id ของการเรียกเครื่องมือ
- การซ่อมแซมการจับคู่ผลลัพธ์เครื่องมืออาจย้ายเอาต์พุตจริงที่จับคู่ได้และสังเคราะห์เอาต์พุต `aborted` แบบ Codex สำหรับการเรียกเครื่องมือที่ขาดหาย
- ไม่มีการตรวจสอบหรือจัดลำดับเทิร์นใหม่
- เอาต์พุตเครื่องมือของตระกูล OpenAI Responses ที่ขาดหายจะถูกสังเคราะห์เป็น `aborted` เพื่อให้ตรงกับการปรับ replay ของ Codex ให้เป็นมาตรฐาน
- ไม่มีการตัด signature ของความคิด

**Gemma 4 ที่เข้ากันได้กับ OpenAI**

- บล็อก thinking/reasoning ของผู้ช่วยในอดีตจะถูกตัดออกก่อน replay เพื่อให้เซิร์ฟเวอร์ Gemma 4 ในเครื่องที่เข้ากันได้กับ OpenAI ไม่ได้รับเนื้อหา reasoning จากเทิร์นก่อนหน้า
- การต่อเนื่องของการเรียกเครื่องมือในเทิร์นเดียวกันปัจจุบันจะเก็บบล็อก reasoning ของผู้ช่วยไว้กับการเรียกเครื่องมือจนกว่าจะ replay ผลลัพธ์เครื่องมือแล้ว

**Google (Generative AI / Gemini CLI / Antigravity)**

- การทำความสะอาด id ของการเรียกเครื่องมือ: อักษรและตัวเลขเท่านั้นแบบเข้มงวด
- การซ่อมแซมการจับคู่ผลลัพธ์เครื่องมือและผลลัพธ์เครื่องมือสังเคราะห์
- การตรวจสอบเทิร์น (การสลับเทิร์นแบบ Gemini)
- การแก้ไขลำดับเทิร์นของ Google (เติม bootstrap ของผู้ใช้ขนาดเล็กไว้ข้างหน้าหากประวัติเริ่มต้นด้วยผู้ช่วย)
- Antigravity Claude: ปรับ signature ของ thinking ให้เป็นมาตรฐาน; ตัดบล็อก thinking ที่ไม่มี signature

**Anthropic / Minimax (เข้ากันได้กับ Anthropic)**

- การซ่อมแซมการจับคู่ผลลัพธ์เครื่องมือและผลลัพธ์เครื่องมือสังเคราะห์
- การตรวจสอบเทิร์น (รวมเทิร์นผู้ใช้ที่ต่อเนื่องกันเพื่อให้เป็นไปตามการสลับแบบเข้มงวด)
- เทิร์น prefill ของผู้ช่วยที่ท้ายสุดจะถูกตัดออกจากเพย์โหลด Anthropic Messages ขาออกเมื่อเปิดใช้ thinking รวมถึงเส้นทาง Cloudflare AI Gateway
- บล็อก thinking ที่มี signature สำหรับ replay ขาดหาย ว่างเปล่า หรือเป็นช่องว่าง จะถูกตัดออกก่อนแปลงเป็นรูปแบบผู้ให้บริการ หากทำให้เทิร์นผู้ช่วยว่างเปล่า OpenClaw จะคงรูปทรงเทิร์นไว้ด้วยข้อความ omitted-reasoning ที่ไม่ว่างเปล่า
- เทิร์นผู้ช่วยเก่าที่มีเฉพาะ thinking ซึ่งต้องถูกตัด จะถูกแทนที่ด้วยข้อความ omitted-reasoning ที่ไม่ว่างเปล่า เพื่อให้อะแดปเตอร์ผู้ให้บริการไม่ตัดเทิร์น replay ทิ้ง

**Amazon Bedrock (Converse API)**

- เทิร์นข้อผิดพลาดสตรีมของผู้ช่วยที่ว่างเปล่าจะถูกซ่อมแซมเป็นบล็อกข้อความ fallback ที่ไม่ว่างเปล่าก่อน replay Bedrock Converse ปฏิเสธข้อความผู้ช่วยที่มี `content: []` ดังนั้นเทิร์นผู้ช่วยที่คงอยู่ซึ่งมี `stopReason: "error"` และเนื้อหาว่างเปล่าจะถูกซ่อมแซมบนดิสก์ก่อนโหลดด้วย
- เทิร์นข้อผิดพลาดสตรีมของผู้ช่วยที่มีเฉพาะบล็อกข้อความว่างจะถูกตัดออกจากสำเนา replay ในหน่วยความจำแทนการ replay บล็อกว่างที่ไม่ถูกต้อง
- บล็อก thinking ของ Claude ที่มี signature สำหรับ replay ขาดหาย ว่างเปล่า หรือเป็นช่องว่าง จะถูกตัดออกก่อน replay ของ Converse หากทำให้เทิร์นผู้ช่วยว่างเปล่า OpenClaw จะคงรูปทรงเทิร์นไว้ด้วยข้อความ omitted-reasoning ที่ไม่ว่างเปล่า
- เทิร์นผู้ช่วยเก่าที่มีเฉพาะ thinking ซึ่งต้องถูกตัด จะถูกแทนที่ด้วยข้อความ omitted-reasoning ที่ไม่ว่างเปล่า เพื่อให้ replay ของ Converse คงรูปทรงเทิร์นแบบเข้มงวด
- replay กรองเทิร์นผู้ช่วยที่เป็น delivery-mirror ของ OpenClaw และที่ Gateway ฉีดเข้ามา
- การทำความสะอาดรูปภาพใช้ตามกฎสากล

**Mistral (รวมถึงการตรวจจับตาม model-id)**

- การทำความสะอาด id ของการเรียกเครื่องมือ: strict9 (อักษรและตัวเลขความยาว 9)

**OpenRouter Gemini**

- การล้าง signature ของความคิด: ตัดค่า `thought_signature` ที่ไม่ใช่ base64 (เก็บ base64 ไว้)

**OpenRouter Anthropic**

- เทิร์น prefill ของผู้ช่วยที่ท้ายสุดจะถูกตัดออกจากเพย์โหลดโมเดล Anthropic ของ OpenRouter ที่ตรวจสอบแล้วและเข้ากันได้กับ OpenAI เมื่อเปิดใช้ reasoning ให้ตรงกับพฤติกรรม replay ของ Anthropic โดยตรงและ Cloudflare Anthropic

**อย่างอื่นทั้งหมด**

- ทำความสะอาดรูปภาพเท่านั้น

## พฤติกรรมในอดีต (ก่อน 2026.1.22)

ก่อนรุ่น 2026.1.22 OpenClaw ใช้สุขอนามัยของบันทึกบทสนทนาหลายชั้น:

- **transcript-sanitize extension** รันทุกครั้งที่สร้างบริบทและสามารถ:
  - ซ่อมแซมการจับคู่ tool use/result
  - ทำความสะอาด id ของการเรียกเครื่องมือ (รวมถึงโหมดไม่เข้มงวดที่เก็บ `_`/`-` ไว้)
- runner ยังทำการทำความสะอาดเฉพาะผู้ให้บริการ ซึ่งทำงานซ้ำซ้อน
- มีการกลายพันธุ์เพิ่มเติมเกิดขึ้นนอกนโยบายผู้ให้บริการ รวมถึง:
  - ตัดแท็ก `<final>` ออกจากข้อความผู้ช่วยก่อนคงอยู่
  - ตัดเทิร์นข้อผิดพลาดของผู้ช่วยที่ว่างเปล่า
  - ตัดเนื้อหาผู้ช่วยหลังการเรียกเครื่องมือ

ความซับซ้อนนี้ทำให้เกิด regression ข้ามผู้ให้บริการ (โดยเฉพาะการจับคู่ `call_id|fc_id` ของ `openai-responses`) การล้างใน 2026.1.22 ลบ extension ออก รวมตรรกะไว้ใน runner และทำให้ OpenAI **ไม่แตะต้อง** นอกเหนือจากการทำความสะอาดรูปภาพ
