OpenClaw ใช้ **การแก้ไขเฉพาะผู้ให้บริการ** กับทรานสคริปต์ก่อนการรันหนึ่งครั้ง (การสร้างบริบทของโมเดล) ส่วนใหญ่เป็นการปรับแต่ง **ในหน่วยความจำ** ที่ใช้เพื่อให้เป็นไปตามข้อกำหนดที่เข้มงวดของผู้ให้บริการ นอกจากนี้ยังอาจมีรอบการซ่อมแซมไฟล์เซสชันแยกต่างหากที่เขียน JSONL ที่จัดเก็บไว้ใหม่ก่อนโหลดเซสชัน ไม่ว่าจะด้วยการทิ้งบรรทัด JSONL ที่มีรูปแบบผิด หรือซ่อมแซมเทิร์นที่ถูกจัดเก็บไว้ซึ่งถูกต้องตามไวยากรณ์แต่ทราบว่าจะถูกปฏิเสธโดย
ผู้ให้บริการระหว่างการเล่นซ้ำ เมื่อเกิดการซ่อมแซม ไฟล์ต้นฉบับจะถูกสำรองไว้ข้าง
ไฟล์เซสชัน

ขอบเขตรวมถึง:

- บริบทพรอมป์เฉพาะรันไทม์ที่ไม่ปะปนกับเทิร์นทรานสคริปต์ที่ผู้ใช้มองเห็น
- การทำให้ id ของ tool call ปลอดภัย
- การตรวจสอบ input ของ tool call
- การซ่อมแซมการจับคู่ tool result
- การตรวจสอบ / การจัดลำดับเทิร์น
- การล้าง thought signature
- การล้าง thinking signature
- การทำให้ payload รูปภาพปลอดภัย
- การล้างบล็อกข้อความว่างก่อนเล่นซ้ำกับผู้ให้บริการ
- การติดแท็กที่มาของ input จากผู้ใช้ (สำหรับพรอมป์ที่ถูกส่งข้ามเซสชัน)
- การซ่อมแซมเทิร์นข้อผิดพลาดของ assistant ที่ว่างสำหรับการเล่นซ้ำ Bedrock Converse

หากคุณต้องการรายละเอียดการจัดเก็บทรานสคริปต์ โปรดดู:

- [เจาะลึกการจัดการเซสชัน](/th/reference/session-management-compaction)

## กฎทั่วโลก: บริบทรันไทม์ไม่ใช่ทรานสคริปต์ของผู้ใช้

บริบท runtime/system สามารถเพิ่มลงในพรอมป์ของโมเดลสำหรับเทิร์นหนึ่งได้ แต่ไม่ใช่
เนื้อหาที่ผู้ใช้ปลายทางเขียนขึ้น OpenClaw เก็บเนื้อหาพรอมป์ฝั่งทรานสคริปต์
แยกต่างหากสำหรับการตอบกลับของ Gateway, followup ที่เข้าคิว, ACP, CLI และการรัน Pi
แบบฝัง เทิร์นผู้ใช้ที่มองเห็นได้ซึ่งจัดเก็บไว้จะใช้เนื้อหาทรานสคริปต์นั้นแทน
พรอมป์ที่เสริมด้วยรันไทม์

สำหรับเซสชันเดิมที่มี wrapper รันไทม์ถูกจัดเก็บไว้แล้ว พื้นผิวประวัติของ Gateway
จะใช้การฉายผลเพื่อแสดงผลก่อนส่งข้อความคืนให้ไคลเอนต์ WebChat,
TUI, REST หรือ SSE

## ตำแหน่งที่สิ่งนี้ทำงาน

สุขอนามัยของทรานสคริปต์ทั้งหมดรวมศูนย์อยู่ใน runner แบบฝัง:

- การเลือกนโยบาย: `src/agents/transcript-policy.ts`
- การใช้การทำให้ปลอดภัย/การซ่อมแซม: `sanitizeSessionHistory` ใน `src/agents/pi-embedded-runner/replay-history.ts`

นโยบายใช้ `provider`, `modelApi` และ `modelId` เพื่อตัดสินใจว่าจะใช้อะไร

แยกจากสุขอนามัยของทรานสคริปต์ ไฟล์เซสชันจะถูกซ่อมแซม (หากจำเป็น) ก่อนโหลด:

- `repairSessionFileIfNeeded` ใน `src/agents/session-file-repair.ts`
- ถูกเรียกจาก `run/attempt.ts` และ `compact.ts` (runner แบบฝัง)

## กฎทั่วโลก: การทำให้รูปภาพปลอดภัย

payload รูปภาพจะถูกทำให้ปลอดภัยเสมอเพื่อป้องกันการถูกปฏิเสธฝั่งผู้ให้บริการเนื่องจาก
ข้อจำกัดด้านขนาด (ลดสเกล/บีบอัดรูปภาพ base64 ที่ใหญ่เกินอีกครั้ง)

สิ่งนี้ยังช่วยควบคุมแรงกดดันด้านโทเคนที่เกิดจากรูปภาพสำหรับโมเดลที่รองรับ vision
โดยทั่วไป มิติสูงสุดที่ต่ำลงจะลดการใช้โทเคน ส่วนมิติที่สูงขึ้นจะรักษารายละเอียดไว้

การนำไปใช้:

- `sanitizeSessionMessagesImages` ใน `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` ใน `src/agents/tool-images.ts`
- ด้านสูงสุดของรูปภาพกำหนดค่าได้ผ่าน `agents.defaults.imageMaxDimensionPx` (ค่าเริ่มต้น: `1200`)
- บล็อกข้อความว่างจะถูกลบขณะที่รอบนี้ไล่ผ่านเนื้อหาสำหรับเล่นซ้ำ เทิร์น assistant
  ที่กลายเป็นว่างจะถูกทิ้งจากสำเนาสำหรับเล่นซ้ำ ส่วนเทิร์น user และ tool-result
  ที่กลายเป็นว่างจะได้รับ placeholder ของเนื้อหาที่ถูกละไว้ซึ่งไม่ว่าง

## กฎทั่วโลก: tool call ที่มีรูปแบบผิด

บล็อก tool-call ของ assistant ที่ไม่มีทั้ง `input` และ `arguments` จะถูกทิ้ง
ก่อนสร้างบริบทของโมเดล สิ่งนี้ป้องกันการถูกปฏิเสธจากผู้ให้บริการเนื่องจาก tool call
ที่ถูกจัดเก็บไว้เพียงบางส่วน (เช่น หลังจากเกิดความล้มเหลวจากการจำกัดอัตรา)

การนำไปใช้:

- `sanitizeToolCallInputs` ใน `src/agents/session-transcript-repair.ts`
- ใช้ใน `sanitizeSessionHistory` ใน `src/agents/pi-embedded-runner/replay-history.ts`

## กฎทั่วโลก: ที่มาของ input ข้ามเซสชัน

เมื่อ agent ส่งพรอมป์เข้าไปยังเซสชันอื่นผ่าน `sessions_send` (รวมถึง
ขั้นตอนตอบกลับ/ประกาศระหว่าง agent) OpenClaw จะจัดเก็บเทิร์นผู้ใช้ที่สร้างขึ้นพร้อมกับ:

- `message.provenance.kind = "inter_session"`

OpenClaw ยังเติม marker `[Inter-session message ... isUser=false]`
ไว้ก่อนข้อความพรอมป์ที่ถูกส่งในเทิร์นเดียวกัน เพื่อให้การเรียกโมเดลที่ใช้งานอยู่แยกแยะ
output จากเซสชันอื่นออกจากคำสั่งของผู้ใช้ปลายทางภายนอกได้ marker นี้รวม
เซสชันต้นทาง ช่องทาง และเครื่องมือเมื่อมี ทรานสคริปต์ยังคงใช้
`role: "user"` เพื่อความเข้ากันได้กับผู้ให้บริการ แต่ทั้งข้อความที่มองเห็นได้และ metadata
ที่มาจะทำเครื่องหมายเทิร์นว่าเป็นข้อมูลข้ามเซสชัน

ระหว่างการสร้างบริบทใหม่ OpenClaw จะใช้ marker เดียวกันกับเทิร์นผู้ใช้ข้ามเซสชัน
ที่จัดเก็บไว้เก่ากว่าซึ่งมีเพียง metadata ที่มา

## เมทริกซ์ผู้ให้บริการ (พฤติกรรมปัจจุบัน)

**OpenAI / OpenAI Codex**

- ทำให้รูปภาพปลอดภัยเท่านั้น
- ทิ้ง reasoning signature กำพร้า (รายการ reasoning แบบเดี่ยวที่ไม่มีบล็อกเนื้อหาตามมา) สำหรับทรานสคริปต์ OpenAI Responses/Codex และทิ้ง reasoning ของ OpenAI ที่เล่นซ้ำได้หลังจากสลับเส้นทางโมเดล
- เก็บ payload รายการ reasoning ของ OpenAI Responses ที่เล่นซ้ำได้ รวมถึงรายการสรุปว่างที่เข้ารหัสไว้ เพื่อให้การเล่นซ้ำแบบ manual/WebSocket คงสถานะ `rs_*` ที่จำเป็นไว้คู่กับรายการ output ของ assistant
- ไม่มีการทำให้ id ของ tool call ปลอดภัย
- การซ่อมแซมการจับคู่ tool result อาจย้าย output จริงที่จับคู่แล้วและสร้าง output แบบ Codex ที่เป็น `aborted` สำหรับ tool call ที่ขาดหาย
- ไม่มีการตรวจสอบหรือจัดลำดับเทิร์นใหม่
- output ของเครื่องมือตระกูล OpenAI Responses ที่ขาดหายจะถูกสร้างเป็น `aborted` เพื่อให้ตรงกับการปรับมาตรฐานการเล่นซ้ำของ Codex
- ไม่มีการลบ thought signature

**Gemma 4 ที่เข้ากันได้กับ OpenAI**

- บล็อก thinking/reasoning ของ assistant ในอดีตจะถูกลบก่อนเล่นซ้ำ เพื่อให้เซิร์ฟเวอร์
  Gemma 4 ในเครื่องที่เข้ากันได้กับ OpenAI ไม่ได้รับเนื้อหา reasoning จากเทิร์นก่อนหน้า
- continuation ของ tool-call ในเทิร์นเดียวกันปัจจุบันจะเก็บบล็อก reasoning ของ assistant
  ที่แนบกับ tool call ไว้จนกว่า tool result จะถูกเล่นซ้ำแล้ว

**Google (Generative AI / Gemini CLI / Antigravity)**

- การทำให้ id ของ tool call ปลอดภัย: ตัวอักษรและตัวเลขเท่านั้นแบบเข้มงวด
- การซ่อมแซมการจับคู่ tool result และ tool result สังเคราะห์
- การตรวจสอบเทิร์น (การสลับเทิร์นแบบ Gemini)
- การแก้ไขการจัดลำดับเทิร์นของ Google (เติม user bootstrap ขนาดเล็กไว้ด้านหน้าหากประวัติเริ่มด้วย assistant)
- Antigravity Claude: ปรับ thinking signature ให้เป็นมาตรฐาน; ทิ้งบล็อก thinking ที่ไม่มีลายเซ็น

**Anthropic / Minimax (เข้ากันได้กับ Anthropic)**

- การซ่อมแซมการจับคู่ tool result และ tool result สังเคราะห์
- การตรวจสอบเทิร์น (รวมเทิร์นผู้ใช้ที่ต่อเนื่องกันเพื่อให้เป็นไปตามการสลับแบบเข้มงวด)
- เทิร์น prefill ของ assistant ท้ายสุดจะถูกลบจาก payload Anthropic Messages
  ขาออกเมื่อเปิดใช้ thinking รวมถึงเส้นทาง Cloudflare AI Gateway
- บล็อก thinking ที่มี replay signature ขาดหาย ว่าง หรือเป็นช่องว่าง จะถูกลบ
  ก่อนแปลงเป็นรูปแบบของผู้ให้บริการ หากสิ่งนั้นทำให้เทิร์น assistant ว่าง OpenClaw จะรักษา
  รูปร่างเทิร์นไว้ด้วยข้อความ omitted-reasoning ที่ไม่ว่าง
- เทิร์น assistant เก่าที่มีแต่ thinking และต้องถูกลบ จะถูกแทนที่ด้วย
  ข้อความ omitted-reasoning ที่ไม่ว่าง เพื่อให้ adapter ของผู้ให้บริการไม่ทิ้งเทิร์น
  สำหรับเล่นซ้ำ

**Amazon Bedrock (Converse API)**

- เทิร์นข้อผิดพลาดสตรีมของ assistant ที่ว่างจะถูกซ่อมแซมเป็นบล็อกข้อความ fallback ที่ไม่ว่าง
  ก่อนเล่นซ้ำ Bedrock Converse ปฏิเสธข้อความ assistant ที่มี `content: []` ดังนั้น
  เทิร์น assistant ที่จัดเก็บไว้ซึ่งมี `stopReason: "error"` และเนื้อหาว่างจะถูก
  ซ่อมแซมบนดิสก์ก่อนโหลดด้วย
- เทิร์นข้อผิดพลาดสตรีมของ assistant ที่มีเพียงบล็อกข้อความว่างจะถูกทิ้ง
  จากสำเนาเล่นซ้ำในหน่วยความจำ แทนที่จะเล่นซ้ำบล็อกว่างที่ไม่ถูกต้อง
- บล็อก thinking ของ Claude ที่มี replay signature ขาดหาย ว่าง หรือเป็นช่องว่าง จะถูก
  ลบก่อนการเล่นซ้ำ Converse หากสิ่งนั้นทำให้เทิร์น assistant ว่าง OpenClaw
  จะรักษารูปร่างเทิร์นไว้ด้วยข้อความ omitted-reasoning ที่ไม่ว่าง
- เทิร์น assistant เก่าที่มีแต่ thinking และต้องถูกลบ จะถูกแทนที่ด้วย
  ข้อความ omitted-reasoning ที่ไม่ว่าง เพื่อให้การเล่นซ้ำ Converse รักษารูปร่างเทิร์นแบบเข้มงวด
- การเล่นซ้ำกรองเทิร์น assistant ที่เป็น delivery-mirror ของ OpenClaw และที่ Gateway แทรกเข้ามา
- การทำให้รูปภาพปลอดภัยใช้ผ่านกฎทั่วโลก

**Mistral (รวมถึงการตรวจจับตาม model-id)**

- การทำให้ id ของ tool call ปลอดภัย: strict9 (ตัวอักษรและตัวเลข ความยาว 9)

**OpenRouter Gemini**

- การล้าง thought signature: ลบค่า `thought_signature` ที่ไม่ใช่ base64 (เก็บ base64 ไว้)

**อื่นๆ ทั้งหมด**

- ทำให้รูปภาพปลอดภัยเท่านั้น

## พฤติกรรมในอดีต (ก่อน 2026.1.22)

ก่อนรุ่น 2026.1.22 OpenClaw ใช้สุขอนามัยของทรานสคริปต์หลายชั้น:

- **transcript-sanitize extension** ทำงานในทุกการสร้างบริบท และสามารถ:
  - ซ่อมแซมการจับคู่ tool use/result
  - ทำให้ id ของ tool call ปลอดภัย (รวมถึงโหมดไม่เข้มงวดที่คง `_`/`-` ไว้)
- runner ยังทำการทำให้ปลอดภัยเฉพาะผู้ให้บริการด้วย ซึ่งทำงานซ้ำซ้อน
- การกลายพันธุ์เพิ่มเติมเกิดขึ้นนอกนโยบายผู้ให้บริการ รวมถึง:
  - ลบแท็ก `<final>` ออกจากข้อความ assistant ก่อนจัดเก็บถาวร
  - ทิ้งเทิร์นข้อผิดพลาดของ assistant ที่ว่าง
  - ตัดเนื้อหา assistant หลังจาก tool call

ความซับซ้อนนี้ทำให้เกิด regression ข้ามผู้ให้บริการ (โดยเฉพาะการจับคู่ `call_id|fc_id`
ของ `openai-responses`) การล้างข้อมูลใน 2026.1.22 ลบ extension ออก รวมศูนย์
ตรรกะใน runner และทำให้ OpenAI **ไม่ถูกแตะต้อง** นอกเหนือจากการทำให้รูปภาพปลอดภัย
