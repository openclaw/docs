OpenClaw ใช้ **การแก้ไขเฉพาะผู้ให้บริการ** กับทรานสคริปต์ก่อนเริ่มการทำงาน
(ขณะสร้างบริบทของโมเดล) การแก้ไขส่วนใหญ่เป็นการปรับเปลี่ยน **ในหน่วยความจำ** ที่ใช้เพื่อ
ให้เป็นไปตามข้อกำหนดที่เข้มงวดของผู้ให้บริการ ขั้นตอนซ่อมแซมไฟล์เซสชันที่แยกต่างหากอาจ
เขียน JSONL ที่จัดเก็บไว้ใหม่ก่อนโหลดเซสชันด้วย แต่เฉพาะสำหรับ
บรรทัดที่มีรูปแบบไม่ถูกต้องหรือเทิร์นที่บันทึกถาวรซึ่งไม่ใช่ระเบียนถาวรที่ถูกต้องเท่านั้น
การตอบกลับของผู้ช่วยที่ส่งมอบแล้วจะยังคงเก็บรักษาไว้บนดิสก์ ส่วนการลบข้อมูลป้อนล่วงหน้าของผู้ช่วย
เฉพาะผู้ให้บริการจะเกิดขึ้นเฉพาะขณะสร้างเพย์โหลดขาออกเท่านั้น

เมื่อมีการซ่อมแซม ระบบจะเขียนไฟล์ต้นฉบับไปยังไฟล์พี่น้องชั่วคราว
`*.bak-<pid>-<ts>` ก่อนแทนที่แบบอะตอมมิก แล้วนำออกเมื่อ
การแทนที่สำเร็จ ระบบจะเก็บข้อมูลสำรองไว้เฉพาะเมื่อการล้างข้อมูลล้มเหลว
ซึ่งในกรณีนั้นจะรายงานพาธกลับมา

ขอบเขตประกอบด้วย:

- การป้องกันไม่ให้บริบทพรอมต์เฉพาะรันไทม์ปรากฏในเทิร์นทรานสคริปต์ที่ผู้ใช้มองเห็น
- การทำความสะอาด ID การเรียกใช้เครื่องมือ
- การตรวจสอบอินพุตของการเรียกใช้เครื่องมือ
- การซ่อมแซมการจับคู่ผลลัพธ์ของเครื่องมือ
- การตรวจสอบ / การจัดลำดับเทิร์น
- การล้างลายเซ็นความคิด
- การล้างลายเซ็นกระบวนการคิด
- การทำความสะอาดเพย์โหลดรูปภาพ
- การล้างบล็อกข้อความว่างก่อนเล่นซ้ำให้ผู้ให้บริการ
- การล้างเทิร์นที่มีเฉพาะการให้เหตุผลซึ่งไม่สมบูรณ์และถึงขีดจำกัดความยาวก่อนเล่นซ้ำให้ผู้ให้บริการ
- การติดแท็กแหล่งที่มาของอินพุตผู้ใช้ (สำหรับพรอมต์ที่กำหนดเส้นทางระหว่างเซสชัน)
- การซ่อมแซมเทิร์นข้อผิดพลาดของผู้ช่วยที่ว่างเปล่าสำหรับการเล่นซ้ำ Bedrock Converse

หากต้องการรายละเอียดเกี่ยวกับพื้นที่จัดเก็บทรานสคริปต์ โปรดดู
[เจาะลึกการจัดการเซสชัน](/th/reference/session-management-compaction)

## กฎส่วนกลาง: บริบทรันไทม์ไม่ใช่ทรานสคริปต์ของผู้ใช้

สามารถเพิ่มบริบทรันไทม์/ระบบลงในพรอมต์ของโมเดลสำหรับแต่ละเทิร์นได้ แต่บริบทดังกล่าว
ไม่ใช่เนื้อหาที่ผู้ใช้ปลายทางเป็นผู้เขียน OpenClaw เก็บเนื้อหาพรอมต์สำหรับ
ทรานสคริปต์แยกต่างหากสำหรับการตอบกลับของ Gateway การติดตามผลในคิว ACP, CLI และการทำงาน
OpenClaw แบบฝัง เทิร์นของผู้ใช้ที่มองเห็นและจัดเก็บไว้จะใช้เนื้อหาทรานสคริปต์นั้นแทน
พรอมต์ที่เสริมด้วยบริบทรันไทม์

สำหรับเซสชันแบบเก่าที่บันทึกตัวห่อหุ้มรันไทม์ไว้แล้ว พื้นผิวประวัติของ Gateway
จะใช้การฉายภาพสำหรับการแสดงผลก่อนส่งคืนข้อความไปยังไคลเอนต์ WebChat,
TUI, REST หรือ SSE

## กฎส่วนกลาง: การทำความสะอาดรูปภาพ

เพย์โหลดรูปภาพจะได้รับการทำความสะอาดเสมอเพื่อป้องกันไม่ให้ผู้ให้บริการปฏิเสธเนื่องจาก
ขีดจำกัดขนาด (ลดขนาด/บีบอัดรูปภาพ base64 ที่มีขนาดใหญ่เกินไปใหม่) การดำเนินการนี้ยังช่วย
ควบคุมภาระโทเค็นที่เกิดจากรูปภาพสำหรับโมเดลที่รองรับการมองเห็นด้วย โดยขนาดสูงสุดที่
เล็กลงจะลดการใช้โทเค็น ส่วนขนาดที่ใหญ่ขึ้นจะรักษารายละเอียดไว้

การนำไปใช้:

- `sanitizeSessionMessagesImages` ใน
  `src/agents/embedded-agent-helpers/images.ts`
- `sanitizeContentBlocksImages` ใน `src/agents/tool-images.ts`
- กำหนดค่าความยาวด้านสูงสุดของรูปภาพได้ผ่าน `agents.defaults.imageMaxDimensionPx`
  (ค่าเริ่มต้น: `1200`)
- บล็อกข้อความว่างจะถูกนำออกขณะที่ขั้นตอนนี้ไล่ตรวจเนื้อหาสำหรับเล่นซ้ำ
  เทิร์นของผู้ช่วยที่กลายเป็นว่างเปล่าจะถูกตัดออกจากสำเนาสำหรับเล่นซ้ำ ส่วนเทิร์นของผู้ใช้
  และผลลัพธ์ของเครื่องมือที่กลายเป็นว่างเปล่าจะได้รับตัวยึดตำแหน่งเนื้อหาที่ละไว้
  ซึ่งไม่ว่างเปล่า

## กฎส่วนกลาง: การเรียกใช้เครื่องมือที่มีรูปแบบไม่ถูกต้อง

บล็อกการเรียกใช้เครื่องมือของผู้ช่วยที่ไม่มีทั้ง `input` และ `arguments` จะถูกตัดออก
ก่อนสร้างบริบทของโมเดล ซึ่งป้องกันไม่ให้ผู้ให้บริการปฏิเสธ
การเรียกใช้เครื่องมือที่บันทึกไว้เพียงบางส่วน (ตัวอย่างเช่น หลังจากเกิดข้อผิดพลาดจากขีดจำกัดอัตรา)

การนำไปใช้:

- `sanitizeToolCallInputs` ใน `src/agents/session-transcript-repair.ts`
- ใช้ใน `sanitizeSessionHistory`
  (`src/agents/embedded-agent-runner/replay-history.ts`)

## กฎส่วนกลาง: การจับคู่ผลลัพธ์ของเครื่องมือ

ผลลัพธ์ของเครื่องมือจะถูกจับคู่กับแต่ละเหตุการณ์การเรียกใช้เครื่องมือภายในแต่ละเทิร์นของผู้ช่วยก่อน
เขียน ID การเรียกใช้เฉพาะผู้ให้บริการใหม่ ID ที่ผู้ให้บริการสร้างอาจซ้ำกันในเทิร์นภายหลัง
ดังนั้นผลลัพธ์ที่อยู่ติดกับการเรียกใช้ซ้ำจะยังคงอยู่กับเหตุการณ์นั้น ผลลัพธ์ที่อยู่ผิดตำแหน่ง
จะถูกย้ายเฉพาะเมื่อมีเหตุการณ์ที่ยังไม่คลี่คลายเพียงหนึ่งเหตุการณ์ซึ่งสามารถเป็นเจ้าของผลลัพธ์นั้นได้
ส่วนรายการส่วนเกินที่กำกวมจะถูกตัดออก และเหตุการณ์ที่ขาดหายจะได้รับผลลัพธ์ข้อผิดพลาดสังเคราะห์

การนำไปใช้: `sanitizeToolUseResultPairing` ใน
`src/agents/session-transcript-repair.ts`

## กฎส่วนกลาง: เทิร์นที่มีเฉพาะการให้เหตุผลซึ่งไม่สมบูรณ์

เทิร์นของผู้ช่วยที่ถึงขีดจำกัดเอาต์พุตของผู้ให้บริการโดยมีเพียงเนื้อหากระบวนการคิดหรือ
กระบวนการคิดที่ปกปิดไว้ จะถูกละออกจากสำเนาสำหรับเล่นซ้ำในหน่วยความจำ เทิร์นดังกล่าว
มีสถานะผู้ให้บริการที่ไม่สมบูรณ์และอาจมีลายเซ็นกระบวนการคิดเพียงบางส่วน

เทิร์นที่ว่างเปล่าและถึงขีดจำกัดความยาวจะยังคงไม่เปลี่ยนแปลง เช่นเดียวกับเทิร์นที่ถึงขีดจำกัดความยาว
ซึ่งมีข้อความที่มองเห็น การเรียกใช้เครื่องมือ หรือบล็อกเนื้อหาที่ไม่รู้จัก ทรานสคริปต์ที่จัดเก็บไว้
จะไม่ถูกเขียนใหม่

การนำไปใช้: `normalizeAssistantReplayContent` ใน
`src/agents/embedded-agent-runner/replay-history.ts`

## กฎส่วนกลาง: แหล่งที่มาของอินพุตระหว่างเซสชัน

เมื่อเอเจนต์ส่งพรอมต์ไปยังอีกเซสชันผ่าน `sessions_send`
(รวมถึงขั้นตอนการตอบกลับ/ประกาศระหว่างเอเจนต์) OpenClaw จะบันทึก
เทิร์นผู้ใช้ที่สร้างขึ้นพร้อม `message.provenance.kind = "inter_session"`

OpenClaw ยังเติมเครื่องหมาย `[Inter-session message] ... isUser=false` ในเทิร์นเดียวกันไว้ข้างหน้า
ข้อความพรอมต์ที่กำหนดเส้นทาง เพื่อให้การเรียกใช้โมเดลที่ทำงานอยู่สามารถ
แยกแยะเอาต์พุตจากเซสชันอื่นออกจากคำสั่งภายนอกของผู้ใช้ปลายทาง เครื่องหมายนี้
ประกอบด้วยเซสชันต้นทาง ช่องทาง และเครื่องมือเมื่อมีข้อมูล ทรานสคริปต์ยังคงใช้
`role: "user"` เพื่อความเข้ากันได้กับผู้ให้บริการ แต่ทั้งข้อความที่มองเห็น
และข้อมูลเมตาของแหล่งที่มาต่างระบุว่าเทิร์นดังกล่าวเป็นข้อมูลระหว่างเซสชัน

ระหว่างการสร้างบริบทใหม่ OpenClaw จะใช้เครื่องหมายเดียวกันกับเทิร์นผู้ใช้ระหว่างเซสชัน
ที่บันทึกไว้ก่อนหน้านี้ซึ่งมีเพียงข้อมูลเมตาของแหล่งที่มา

## เมทริกซ์ผู้ให้บริการ (ลักษณะการทำงานปัจจุบัน)

**OpenAI / OpenAI Codex**

- ทำความสะอาดรูปภาพเท่านั้น
- ตัดลายเซ็นการให้เหตุผลที่ไม่มีคู่ (รายการการให้เหตุผลแบบเดี่ยวที่ไม่มี
  บล็อกเนื้อหาตามหลัง) สำหรับทรานสคริปต์ OpenAI Responses/Codex และตัด
  การให้เหตุผล OpenAI ที่เล่นซ้ำได้หลังสลับเส้นทางโมเดล
- รักษาเพย์โหลดรายการการให้เหตุผล OpenAI Responses ที่เล่นซ้ำได้ รวมถึง
  รายการสรุปว่างเปล่าที่เข้ารหัส เพื่อให้การเล่นซ้ำแบบกำหนดเอง/WebSocket ยังคงจับคู่
  สถานะ `rs_*` ที่จำเป็นกับรายการเอาต์พุตของผู้ช่วย
- Native ChatGPT Codex Responses ทำตามความสอดคล้องของข้อมูลบนสายสื่อสาร Codex โดยเล่นซ้ำ
  เพย์โหลดการให้เหตุผล/ข้อความ/ฟังก์ชันของ Responses ก่อนหน้าโดยไม่มี ID รายการก่อนหน้า
  พร้อมรักษา `prompt_cache_key` ของเซสชันไว้
- การเล่นซ้ำในตระกูล OpenAI Responses รักษาคู่การให้เหตุผลในโมเดลเดียวกัน
  `call_*|fc_*` ตามรูปแบบมาตรฐาน แต่ปรับ ID รายการ `call_id`/การเรียกใช้ฟังก์ชัน
  ที่มีรูปแบบไม่ถูกต้องหรือยาวเกินไปให้เป็นมาตรฐานแบบกำหนดผลแน่นอนก่อนแปลงเพย์โหลด pi-ai
- การซ่อมแซมการจับคู่ผลลัพธ์ของเครื่องมืออาจย้ายเอาต์พุตจริงที่ตรงกันและสังเคราะห์
  เอาต์พุต `aborted` แบบ Codex สำหรับการเรียกใช้เครื่องมือที่ขาดหาย
- ไม่มีการตรวจสอบหรือจัดลำดับเทิร์นใหม่ และไม่มีการลบลายเซ็นความคิด

**Chat Completions ที่เข้ากันได้กับ OpenAI**

- บล็อกกระบวนการคิด/การให้เหตุผลในอดีตของผู้ช่วยจะถูกลบออกก่อนเล่นซ้ำ
  เพื่อไม่ให้เซิร์ฟเวอร์ภายในและเซิร์ฟเวอร์แบบพร็อกซีที่เข้ากันได้กับ OpenAI ได้รับ
  ฟิลด์การให้เหตุผลจากเทิร์นก่อนหน้า เช่น `reasoning` หรือ `reasoning_content`
- การต่อเนื่องของการเรียกใช้เครื่องมือในเทิร์นเดียวกันปัจจุบันจะรักษาบล็อกการให้เหตุผล
  ของผู้ช่วยให้แนบอยู่กับการเรียกใช้เครื่องมือจนกว่าจะเล่นซ้ำผลลัพธ์ของเครื่องมือแล้ว
- รายการโมเดลแบบกำหนดเอง/โฮสต์เองที่มี `reasoning: true` จะรักษา
  ข้อมูลเมตาการให้เหตุผลที่เล่นซ้ำไว้
- ข้อยกเว้นที่ผู้ให้บริการเป็นเจ้าของสามารถเลือกไม่ใช้ได้เมื่อโปรโตคอลบนสายสื่อสาร
  กำหนดให้ต้องเล่นซ้ำข้อมูลเมตาการให้เหตุผล

**Google (Generative AI / Gemini CLI / Antigravity)**

- การทำความสะอาด ID การเรียกใช้เครื่องมือ: เฉพาะอักขระตัวอักษรและตัวเลขอย่างเคร่งครัด
- การซ่อมแซมการจับคู่ผลลัพธ์ของเครื่องมือและผลลัพธ์เครื่องมือสังเคราะห์
- การตรวจสอบเทิร์น (การสลับเทิร์นแบบ Gemini)
- การแก้ไขลำดับเทิร์นของ Google (เติมเทิร์นเริ่มต้นของผู้ใช้ขนาดเล็กไว้ข้างหน้าหากประวัติ
  เริ่มต้นด้วยผู้ช่วย)
- Antigravity Claude: ปรับลายเซ็นกระบวนการคิดให้เป็นมาตรฐาน และตัดบล็อกกระบวนการคิด
  ที่ไม่มีลายเซ็น

**Anthropic / Minimax (เข้ากันได้กับ Anthropic)**

- การซ่อมแซมการจับคู่ผลลัพธ์ของเครื่องมือและผลลัพธ์เครื่องมือสังเคราะห์
- การตรวจสอบเทิร์น (รวมเทิร์นผู้ใช้ที่ต่อเนื่องกันเพื่อให้เป็นไปตาม
  การสลับเทิร์นอย่างเคร่งครัด)
- เทิร์นป้อนล่วงหน้าของผู้ช่วยที่อยู่ท้ายสุดจะถูกลบออกจากเพย์โหลด Anthropic
  Messages ขาออกเมื่อเปิดใช้กระบวนการคิด รวมถึงเส้นทาง Cloudflare AI
  Gateway
- ลายเซ็นกระบวนการคิดของผู้ช่วยก่อน Compaction จะถูกลบออกก่อนเล่นซ้ำให้ผู้ให้บริการ
  เมื่อเซสชันผ่าน Compaction แล้ว ลายเซ็นกระบวนการคิดถูกผูกด้วยการเข้ารหัสลับ
  กับส่วนนำหน้าของบทสนทนา ณ เวลาที่สร้าง หลัง Compaction ส่วนนำหน้าจะเปลี่ยนไป
  (เนื้อหาที่สรุปจะแทนที่ต้นฉบับ) ดังนั้นการเล่นซ้ำลายเซ็นเดิมจะทำให้ Anthropic
  ปฏิเสธคำขอด้วยข้อความ "Invalid signature in thinking block" ข้อความ
  กระบวนการคิดจะถูกรักษาไว้เป็นบล็อกที่ไม่มีลายเซ็น แล้วจึงจัดการโดย
  กฎด้านล่าง
- บล็อกกระบวนการคิดที่มีลายเซ็นสำหรับเล่นซ้ำหายไป ว่างเปล่า หรือมีแต่ช่องว่าง
  จะถูกลบออกก่อนแปลงสำหรับผู้ให้บริการ หากทำให้เทิร์นของผู้ช่วยว่างเปล่า
  OpenClaw จะรักษารูปทรงของเทิร์นไว้ด้วยข้อความการให้เหตุผลที่ละไว้ซึ่งไม่ว่างเปล่า
- เทิร์นผู้ช่วยแบบเก่าที่มีเฉพาะกระบวนการคิดซึ่งต้องถูกลบออกจะถูกแทนที่
  ด้วยข้อความการให้เหตุผลที่ละไว้ซึ่งไม่ว่างเปล่า เพื่อไม่ให้อะแดปเตอร์ของผู้ให้บริการตัด
  เทิร์นสำหรับเล่นซ้ำออก

**Amazon Bedrock (Converse API)**

- เทิร์นข้อผิดพลาดสตรีมของผู้ช่วยที่ว่างเปล่าจะถูกซ่อมแซมเป็นบล็อก
  ข้อความสำรองที่ไม่ว่างเปล่าก่อนเล่นซ้ำ Bedrock Converse ปฏิเสธข้อความของผู้ช่วย
  ที่มี `content: []` ดังนั้นเทิร์นผู้ช่วยที่บันทึกไว้ซึ่งมี `stopReason:
"error"` และเนื้อหาว่างเปล่า
  จะถูกซ่อมแซมบนดิสก์ก่อนโหลดด้วย
- เทิร์นข้อผิดพลาดสตรีมของผู้ช่วยที่มีเพียงบล็อกข้อความว่างจะถูกตัดออกจาก
  สำเนาสำหรับเล่นซ้ำในหน่วยความจำ แทนที่จะเล่นซ้ำบล็อกว่างที่ไม่ถูกต้อง
- ลายเซ็นกระบวนการคิดของผู้ช่วยก่อน Compaction จะถูกลบออกก่อนเล่นซ้ำผ่าน Converse
  เมื่อเซสชันผ่าน Compaction แล้ว ด้วยเหตุผลเดียวกับ Anthropic ข้างต้น
- บล็อกกระบวนการคิด Claude ที่มีลายเซ็นสำหรับเล่นซ้ำหายไป ว่างเปล่า หรือมีแต่ช่องว่าง
  จะถูกลบออกก่อนเล่นซ้ำผ่าน Converse หากทำให้เทิร์นของผู้ช่วยว่างเปล่า
  OpenClaw จะรักษารูปทรงของเทิร์นไว้ด้วยข้อความการให้เหตุผลที่ละไว้ซึ่งไม่ว่างเปล่า
- เทิร์นผู้ช่วยแบบเก่าที่มีเฉพาะกระบวนการคิดซึ่งต้องถูกลบออกจะถูกแทนที่
  ด้วยข้อความการให้เหตุผลที่ละไว้ซึ่งไม่ว่างเปล่า เพื่อให้การเล่นซ้ำผ่าน Converse
  รักษารูปทรงของเทิร์นอย่างเคร่งครัด
- การเล่นซ้ำจะกรองเทิร์นผู้ช่วยที่เป็นสำเนาสะท้อนการส่งมอบของ OpenClaw และที่ Gateway แทรก
- ใช้การทำความสะอาดรูปภาพตามกฎส่วนกลาง

**Mistral (รวมถึงการตรวจจับตาม ID โมเดล)**

- การทำความสะอาด ID การเรียกใช้เครื่องมือ: strict9 (อักขระตัวอักษรและตัวเลข ความยาว 9)

**OpenRouter Gemini**

- การล้างลายเซ็นความคิด: ลบค่า `thought_signature` ที่ไม่ใช่ base64
  (เก็บ base64 ไว้)

**OpenRouter Anthropic**

- เทิร์นป้อนล่วงหน้าของผู้ช่วยที่อยู่ท้ายสุดจะถูกลบออกจากเพย์โหลดโมเดล Anthropic
  ที่เข้ากันได้กับ OpenAI และผ่านการยืนยันของ OpenRouter เมื่อเปิดใช้การให้เหตุผล
  ให้สอดคล้องกับลักษณะการเล่นซ้ำของ Anthropic โดยตรงและ Cloudflare Anthropic

**อื่น ๆ ทั้งหมด**

- ทำความสะอาดรูปภาพเท่านั้น

## ลักษณะการทำงานในอดีต (ก่อน 2026.1.22)

ก่อนรุ่น 2026.1.22 OpenClaw ใช้สุขอนามัยของทรานสคริปต์
หลายชั้น:

- **ส่วนขยาย transcript-sanitize** ทำงานในทุกครั้งที่สร้างบริบท และสามารถ:
  - ซ่อมแซมการจับคู่การใช้เครื่องมือกับผลลัพธ์
  - ปรับรหัสการเรียกเครื่องมือให้ปลอดภัย (รวมถึงโหมดที่ไม่เข้มงวดซึ่งคง
    `_`/`-` ไว้)
- ตัวรันยังดำเนินการปรับให้ปลอดภัยเฉพาะสำหรับผู้ให้บริการ ซึ่ง
  ทำให้เกิดการทำงานซ้ำซ้อน
- มีการแก้ไขเพิ่มเติมนอกนโยบายของผู้ให้บริการ รวมถึง
  การลบแท็ก `<final>` ออกจากข้อความของผู้ช่วยก่อนบันทึกถาวร การละทิ้ง
  เทิร์นข้อผิดพลาดของผู้ช่วยที่ว่างเปล่า และการตัดเนื้อหาของผู้ช่วยหลังจาก
  เรียกใช้เครื่องมือ

ความซับซ้อนนี้ทำให้เกิดการถดถอยข้ามผู้ให้บริการ (โดยเฉพาะ
การจับคู่ `openai-responses` `call_id|fc_id`) การปรับปรุงเมื่อ 2026.1.22 ได้ลบ
ส่วนขยายดังกล่าว รวมตรรกะไว้ที่ตัวรัน และกำหนดให้ OpenAI **ไม่แตะต้อง**
นอกเหนือจากการปรับรูปภาพให้ปลอดภัย
