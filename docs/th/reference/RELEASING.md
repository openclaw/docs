OpenClaw เปิดให้ผู้ใช้ใช้งานช่องทางอัปเดตสามช่องทางในปัจจุบัน:

- stable: ช่องทางรีลีสที่โปรโมตอยู่เดิม ซึ่งยังคงแก้ปลายทางผ่าน npm `latest` จนกว่าหมุดหมาย CLI/ช่องทางแบบแยกจะเสร็จสมบูรณ์
- beta: แท็กพรีรีลีสที่เผยแพร่ไปยัง npm `beta`
- dev: HEAD ที่เคลื่อนไหวของ `main`

นอกจากนี้ ผู้ดำเนินการรีลีสสามารถเผยแพร่แพ็กเกจหลักของเดือนที่เสร็จสิ้นล่าสุด
ไปยัง npm `extended-stable` โดยเริ่มที่แพตช์ `33` ส่วนสายรีลีสสุดท้ายตามปกติ
ของเดือนปัจจุบันยังคงอยู่บน npm `latest`; การแยกการเผยแพร่
ฝั่งผู้ดำเนินการนี้ไม่ได้เปลี่ยนการแก้ปลายทางช่องทางอัปเดตของ CLI ด้วยตัวมันเอง

บิลด์อัลฟาของ Tideclaw เป็นสายพรีรีลีสภายในที่แยกต่างหาก (npm dist-tag `alpha`) ซึ่งอธิบายไว้ใน [อินพุตของเวิร์กโฟลว์ NPM](#npm-workflow-inputs) และ [กล่องทดสอบรีลีส](#release-test-boxes)

## การตั้งชื่อเวอร์ชัน

- เวอร์ชันรีลีส extended-stable รายเดือนของ npm: `YYYY.M.PATCH` พร้อม `PATCH >= 33`, แท็ก git `vYYYY.M.PATCH`
- เวอร์ชันรีลีสสุดท้ายรายวัน/ตามปกติ: `YYYY.M.PATCH` พร้อม `PATCH < 33`, แท็ก git `vYYYY.M.PATCH`
- เวอร์ชันรีลีสแก้ไขสำรองตามปกติ: `YYYY.M.PATCH-N`, แท็ก git `vYYYY.M.PATCH-N`
- เวอร์ชันพรีรีลีสเบตา: `YYYY.M.PATCH-beta.N`, แท็ก git `vYYYY.M.PATCH-beta.N`
- เวอร์ชันพรีรีลีสอัลฟา: `YYYY.M.PATCH-alpha.N`, แท็ก git `vYYYY.M.PATCH-alpha.N`
- ห้ามเติมเลขศูนย์นำหน้าเดือนหรือแพตช์
- `PATCH` เป็นหมายเลขลำดับสายรีลีสรายเดือน ไม่ใช่วันที่ในปฏิทิน รีลีสสุดท้ายตามปกติและรีลีสเบตาจะเลื่อนสายปัจจุบัน ส่วนแท็กเฉพาะอัลฟาจะไม่ใช้หรือเลื่อนหมายเลขแพตช์ของเบตา/รีลีสตามปกติ ดังนั้นให้ข้ามแท็กเฉพาะอัลฟารุ่นเก่าที่มีหมายเลขแพตช์สูงกว่าเมื่อเลือกสายเบตาหรือสายตามปกติ
- บิลด์อัลฟา/ไนต์ลีใช้สายแพตช์ถัดไปที่ยังไม่รีลีส และเพิ่มเฉพาะ `alpha.N` สำหรับบิลด์ที่ทำซ้ำ เมื่อแพตช์นั้นมีเบตาแล้ว บิลด์อัลฟาใหม่จะย้ายไปยังแพตช์ถัดไป
- เวอร์ชัน npm เปลี่ยนแปลงไม่ได้: ห้ามลบ เผยแพร่ซ้ำ หรือนำแท็กที่เผยแพร่แล้วกลับมาใช้ใหม่ ให้สร้างหมายเลขพรีรีลีสถัดไปหรือแพตช์รายเดือนถัดไปแทน
- `latest` ยังคงเป็นไปตามสาย npm รายวัน/ตามปกติในปัจจุบัน; `beta` คือเป้าหมายการติดตั้งเบตาปัจจุบัน
- `extended-stable` หมายถึงแพ็กเกจ npm ของเดือนล่าสุดที่รองรับ โดยเริ่มที่แพตช์ `33`; แพตช์ `34` และหลังจากนั้นเป็นรีลีสบำรุงรักษาบนสายรายเดือนนั้น
- รีลีสสุดท้ายตามปกติและรีลีสแก้ไขตามปกติจะเผยแพร่ไปยัง npm `beta` โดยค่าเริ่มต้น; ผู้ดำเนินการรีลีสสามารถกำหนดเป้าหมายเป็น `latest` อย่างชัดเจน หรือโปรโมตบิลด์เบตาที่ผ่านการตรวจสอบในภายหลัง
- เส้นทาง extended-stable รายเดือนโดยเฉพาะจะเผยแพร่แพ็กเกจหลักของ npm และ Plugin ทางการทุกตัวที่เผยแพร่ไปยัง npm ได้ด้วยเวอร์ชันเดียวกันทุกประการ เส้นทางนี้ไม่เผยแพร่ Plugin ไปยัง ClawHub และไม่เผยแพร่อาร์ติแฟกต์ macOS หรือ Windows, GitHub Release, dist-tag ของรีโพซิทอรีส่วนตัว, อิมเมจ Docker, อาร์ติแฟกต์อุปกรณ์เคลื่อนที่ หรือไฟล์ดาวน์โหลดจากเว็บไซต์
- รีลีสสุดท้ายตามปกติทุกรีลีสจะจัดส่งแพ็กเกจ npm, แอป macOS, Android APK แบบสแตนด์อโลนที่ลงนามแล้ว และโปรแกรมติดตั้ง Windows Hub ที่ลงนามแล้วพร้อมกัน โดยปกติรีลีสเบตาจะตรวจสอบและเผยแพร่เส้นทาง npm/แพ็กเกจก่อน ส่วนการบิลด์/ลงนาม/รับรองโดยโนตารี/โปรโมตแอปเนทีฟจะสงวนไว้สำหรับรีลีสสุดท้ายตามปกติ เว้นแต่จะมีการร้องขออย่างชัดเจน

## รอบการรีลีส

- รีลีสจะดำเนินไปโดยเริ่มจากเบตาก่อน; stable จะตามมาหลังจากตรวจสอบเบตาล่าสุดแล้วเท่านั้น
- โดยปกติผู้ดูแลจะสร้างรีลีสจากแบรนช์ `release/YYYY.M.PATCH` ที่สร้างจาก `main` ปัจจุบัน เพื่อให้การตรวจสอบและแก้ไขรีลีสไม่ขัดขวางการพัฒนาใหม่บน `main`
- หากแท็กเบตาถูกพุชหรือเผยแพร่แล้วและจำเป็นต้องแก้ไข ผู้ดูแลจะสร้างแท็ก `-beta.N` ถัดไปแทนการลบหรือสร้างแท็กเดิมใหม่
- ขั้นตอนรีลีสโดยละเอียด การอนุมัติ ข้อมูลรับรอง และบันทึกการกู้คืนมีไว้สำหรับผู้ดูแลเท่านั้น

## การเผยแพร่ extended-stable รายเดือนเฉพาะ npm

นี่เป็นข้อยกเว้นเฉพาะจากขั้นตอนรีลีสตามปกติด้านล่าง สำหรับเดือน
ที่เสร็จสิ้นแล้ว `YYYY.M` ให้สร้าง `extended-stable/YYYY.M.33`; เผยแพร่
`vYYYY.M.33` และแพตช์บำรุงรักษาหลังจากนั้นจากแบรนช์เดียวกัน แท็กรีลีส
ปลายแบรนช์ เช็กเอาต์ เวอร์ชันแพ็กเกจ การตรวจสอบเบื้องต้นของ npm และการรัน Full Release
Validation ต้องระบุคอมมิตเดียวกันทั้งหมด `main` ที่ได้รับการป้องกันต้อง
มีเวอร์ชันสุดท้ายของเดือนปฏิทินที่ใหม่กว่าอย่างเคร่งครัดและต่ำกว่าแพตช์
`33` อยู่แล้ว; แพตช์บำรุงรักษายังคงมีสิทธิ์หลังจาก `main` เลื่อนไปมากกว่า
หนึ่งเดือน

บนแบรนช์ extended-stable ที่ตรงกันทุกประการ ให้เพิ่มเวอร์ชันแพ็กเกจรูทเป็น `YYYY.M.P`, รัน
`pnpm release:prep` และตรวจสอบว่าแพ็กเกจส่วนขยายทุกแพ็กเกจที่เผยแพร่ได้มี
เวอร์ชันเดียวกัน คอมมิตและพุชการเปลี่ยนแปลงที่สร้างขึ้นทั้งหมด สร้างและพุช
แท็ก `vYYYY.M.P` ที่เปลี่ยนแปลงไม่ได้ ณ คอมมิตนั้น และบันทึก SHA แบบเต็มที่ได้
เวิร์กโฟลว์จะใช้ต้นไม้ที่เตรียมไว้นี้; เวิร์กโฟลว์จะไม่เพิ่มหรือซิงโครไนซ์
เวอร์ชันให้

รันการตรวจสอบเบื้องต้นของ npm และ Full Release Validation จากปลายแบรนช์
ที่เตรียมไว้ตรงกันทุกประการ จากนั้นบันทึก ID ของทั้งสองรันและความพยายามรัน
Full Release Validation ที่สำเร็จ:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=true \
  -f npm_dist_tag=extended-stable

gh workflow run full-release-validation.yml \
  --ref extended-stable/YYYY.M.33 \
  -f ref=extended-stable/YYYY.M.33 \
  -f release_profile=stable
```

`release_profile=stable` คือโปรไฟล์ระดับความลึกในการตรวจสอบที่มีอยู่;
แยกต่างหากจาก dist-tag `extended-stable` ของ npm และตั้งใจให้
คงเดิม

หลังจากทั้งสองรันสำเร็จ ให้เผยแพร่ Plugin ทางการทุกตัวที่เผยแพร่ไปยัง npm ได้จาก
ปลายแบรนช์เดียวกันที่ตรงกันทุกประการ แพตช์ `P` ต้องเป็น `33` หรือสูงกว่า ส่ง SHA รีลีสแบบเต็ม
เป็น `ref`, รอให้เมทริกซ์ทั้งหมดและการอ่านกลับจากรีจิสทรีเสร็จสิ้น จากนั้นบันทึก
ID การรัน Plugin NPM Release ที่สำเร็จ:

```bash
RELEASE_SHA="$(git rev-parse HEAD)"
gh workflow run plugin-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f publish_scope=all-publishable \
  -f ref="$RELEASE_SHA" \
  -f npm_dist_tag=extended-stable
```

เวิร์กโฟลว์ใช้รายการแพ็กเกจ `all-publishable` ที่เตรียมไว้ตามปกติ
รวมถึงแพ็กเกจที่ซอร์สไม่ได้เปลี่ยนแปลง โดยจะตรวจสอบแพ็กเกจทุกแพ็กเกจที่ตรงกันทุกประการ
และแท็ก `extended-stable` ของ Plugin ทุกแท็กก่อนสำเร็จ หากการรันบางส่วน
ล้มเหลว ให้รันคำสั่งเดิมอีกครั้ง: แพ็กเกจที่เผยแพร่แล้วจะถูกนำกลับมาใช้ แท็ก Plugin
ที่ขาดหายหรือล้าสมัยจะได้รับการปรับให้สอดคล้องภายใต้สภาพแวดล้อมรีลีสของ npm และ
การอ่านกลับขั้นสุดท้ายยังคงครอบคลุมชุดแพ็กเกจทั้งหมด

หลังจากเวิร์กโฟลว์ Plugin สำเร็จและสภาพแวดล้อมรีลีสของ npm พร้อมแล้ว
ให้เผยแพร่ทาร์บอลหลักจากการตรวจสอบเบื้องต้นที่ตรงกันทุกประการ การเผยแพร่แพ็กเกจหลักจะตรวจสอบว่า
การรัน Plugin ที่อ้างอิงเป็น `completed/success` บนแบรนช์มาตรฐานเดียวกันและ
SHA ของซอร์สที่ตรงกันทุกประการ:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=false \
  -f npm_dist_tag=extended-stable \
  -f preflight_run_id=<npm-preflight-run-id> \
  -f full_release_validation_run_id=<full-validation-run-id> \
  -f full_release_validation_run_attempt=<full-validation-run-attempt> \
  -f plugin_npm_run_id=<plugin-npm-run-id>
```

สำหรับฟอร์กหรือการซ้อมในสภาพแวดล้อมที่ไม่ใช่การผลิตซึ่งตั้งใจให้ไม่สามารถเป็นไปตาม
นโยบายเดือนของ `.33` หรือ `main` ที่ได้รับการป้องกัน ให้เพิ่ม
`-f bypass_extended_stable_guard=true` ในการสั่งงานทั้งการตรวจสอบเบื้องต้นและการเผยแพร่
ของ npm ค่าเริ่มต้นคือ `false` การข้ามจะได้รับการยอมรับเฉพาะเมื่อใช้กับ
`npm_dist_tag=extended-stable` และจะถูกบันทึกในสรุปเวิร์กโฟลว์ การข้ามนี้
ไม่ข้าม ref เวิร์กโฟลว์ `extended-stable/YYYY.M.33` มาตรฐาน,
ความเท่ากันของปลายแบรนช์/แท็ก/เช็กเอาต์, ไวยากรณ์แท็กสุดท้าย, ความเท่ากันของเวอร์ชัน
แพ็กเกจ/แท็ก, ข้อมูลประจำตัวของรันและแมนิเฟสต์ที่อ้างอิง, ที่มาของทาร์บอล,
การอนุมัติสภาพแวดล้อม, การอ่านกลับจากรีจิสทรี หรือหลักฐานการซ่อมแซมตัวเลือก

เวิร์กโฟลว์การเผยแพร่จะตรวจสอบข้อมูลประจำตัวของการตรวจสอบเบื้องต้น การตรวจสอบ และการรัน Plugin
ที่อ้างอิง ไดเจสต์ของทาร์บอลที่เตรียมไว้ และตัวเลือกรีจิสทรีหลัก
ให้ยืนยันผลลัพธ์โดยอิสระหลังจากเวิร์กโฟลว์สำเร็จ:

```bash
npm view openclaw@YYYY.M.P version --userconfig "$(mktemp)"
npm view openclaw@extended-stable version --userconfig "$(mktemp)"
```

ทั้งสองคำสั่งต้องส่งคืน `YYYY.M.P` หากการเผยแพร่สำเร็จแต่การอ่านกลับ
ตัวเลือกล้มเหลว ห้ามเผยแพร่เวอร์ชันแพ็กเกจที่เปลี่ยนแปลงไม่ได้ซ้ำ ให้ใช้
คำสั่งซ่อมแซม `npm dist-tag add openclaw@YYYY.M.P extended-stable` เพียงคำสั่งเดียว
ที่พิมพ์ในสรุปแบบรันเสมอของเวิร์กโฟลว์ที่ล้มเหลว จากนั้นทำการอ่านกลับ
โดยอิสระทั้งสองรายการซ้ำ การย้อนกลับไปยังตัวเลือกก่อนหน้าเป็นการตัดสินใจของผู้ดำเนินการ
ที่แยกต่างหาก ไม่ใช่เส้นทางซ่อมแซมการอ่านกลับ

เอกสารการสนับสนุนสาธารณะกำหนดให้ Slack, Discord และ Codex เป็น
พื้นผิว Plugin extended-stable ที่ครอบคลุมในช่วงแรก รายการดังกล่าวเป็นคำประกาศการสนับสนุน ไม่ใช่
รายการอนุญาตของโค้ดรีลีส: Plugin ทางการทุกตัวที่เผยแพร่ไปยัง npm ได้จะใช้
เส้นทางการเผยแพร่เวอร์ชันเดียวกันทุกประการ

รายการตรวจสอบตามปกติด้านล่างยังคงรับผิดชอบเบตา, `latest`, GitHub Release,
Plugin, macOS, Windows และการเผยแพร่แพลตฟอร์มอื่น ๆ ห้ามรัน

## รายการตรวจสอบสำหรับผู้ดำเนินการรีลีสตามปกติ

รายการตรวจสอบนี้แสดงโครงสร้างสาธารณะของกระบวนการรีลีส ข้อมูลรับรองส่วนตัว การลงนาม การรับรองโดยโนตารี การกู้คืน dist-tag และรายละเอียดการย้อนกลับฉุกเฉินจะอยู่ในคู่มือการดำเนินการรีลีสสำหรับผู้ดูแลเท่านั้น

1. เริ่มจาก `main` ปัจจุบัน: ดึงข้อมูลล่าสุด ยืนยันว่าคอมมิตเป้าหมายถูกพุชแล้ว และยืนยันว่า CI ของ `main` อยู่ในสถานะเขียวเพียงพอสำหรับสร้างแบรนช์
2. สร้าง `release/YYYY.M.PATCH` จากคอมมิตนั้น การแบ็กพอร์ตเป็นทางเลือก; ใช้เฉพาะชุดที่ผู้ดำเนินการเลือก เพิ่มทุกตำแหน่งเวอร์ชันที่จำเป็น รัน `pnpm release:prep`, ดำเนินการแก้ไขรีลีสและฟอร์เวิร์ดพอร์ตที่จำเป็นให้เสร็จ แล้วตรวจสอบ `src/plugins/compat/registry.ts` พร้อม `src/commands/doctor/shared/deprecation-compat.ts`
3. ตรึงคอมมิตก่อนบันทึกการเปลี่ยนแปลงที่ผลิตภัณฑ์เสร็จสมบูรณ์เป็น **Code SHA** รันการตรวจสอบเบื้องต้นของซอร์สแบบกำหนดผลได้ จากนั้นใช้ `node scripts/full-release-validation-at-sha.mjs --sha <code-sha> --target-ref release/YYYY.M.PATCH` วิธีนี้จะตรึงเครื่องมือเวิร์กโฟลว์ที่เชื่อถือได้ ขณะที่เมทริกซ์ Vitest, Docker, QA, แพ็กเกจ และประสิทธิภาพแบบเต็มกำหนดเป้าหมายไปยัง Code SHA ที่ตรงกันทุกประการ
4. จำแนกความล้มเหลวก่อนแก้ไข ความล้มเหลวของผลิตภัณฑ์/โค้ดจะสร้าง Code SHA ใหม่และต้องผ่านการตรวจสอบแบบเต็มเป็นสถานะเขียวสำหรับ SHA นั้น ความล้มเหลวของเวิร์กโฟลว์ ชุดทดสอบ ข้อมูลรับรอง การอนุมัติ หรือโครงสร้างพื้นฐานจะได้รับการซ่อมแซมในพื้นผิวที่เป็นเจ้าของและรันซ้ำกับ Code SHA เดิม
5. หลังจาก Code SHA เป็นสถานะเขียวแล้วเท่านั้น ให้สร้างส่วนบนสุดของ `CHANGELOG.md` จาก PR ที่ผสานแล้วและคอมมิตโดยตรงนับตั้งแต่แท็กที่จัดส่งล่าสุดซึ่งเข้าถึงได้ ทำให้รายการมุ่งเน้นผู้ใช้และไม่ซ้ำกัน เมื่อแท็กที่จัดส่งซึ่งแตกแขนงออกไปหรือฟอร์เวิร์ดพอร์ตภายหลังเชื่อมโยง PR ที่รีลีสแล้วอีกครั้ง ให้ส่งแท็กนั้นอย่างชัดเจนเป็น `--shipped-ref`
6. คอมมิตเฉพาะ `CHANGELOG.md` คอมมิตนี้คือ **Release SHA** ดิฟทั้งหมดจาก Code SHA ถึง Release SHA ต้องเป็น `CHANGELOG.md` เท่านั้น; หากมีพาธอื่นเปลี่ยนแปลง ให้ย้อนกระบวนการรีลีสกลับไปยังขั้นตอนที่ 2
7. รัน Full Release Validation ที่ตรึงด้วย SHA สำหรับ Release SHA โดยเปิดใช้การนำหลักฐานกลับมาใช้ โหนดแม่แบบน้ำหนักเบาต้องบันทึก `changelog-only-release-v1`, ชี้ไปยัง Code SHA ที่เป็นสถานะเขียว และไม่สั่งงานเลนลูกของผลิตภัณฑ์ วิธีนี้นำหลักฐานผลิตภัณฑ์กลับมาใช้; ไม่ได้นำไบต์แพ็กเกจกลับมาใช้
8. รัน `OpenClaw NPM Release` ด้วย `preflight_only=true` กับ Release SHA/แท็ก บันทึก `preflight_run_id` ที่สำเร็จ ขั้นตอนนี้จะบิลด์และตรวจสอบไบต์แพ็กเกจที่ตรงกันทุกประการซึ่งรวมบันทึกการเปลี่ยนแปลงขั้นสุดท้าย
9. ติดแท็ก Release SHA จากนั้นรันตัวช่วยแคนดิเดตด้วยโหนดแม่สำหรับการตรวจสอบ Release-SHA ที่สำเร็จและการตรวจสอบเบื้องต้นของ npm แทนการสั่งงานรายการใดรายการหนึ่งซ้ำ:

   ```bash
   pnpm release:candidate -- \
     --tag vYYYY.M.PATCH-beta.N \
     --full-release-run <release-sha-validation-run-id> \
     --npm-preflight-run <preflight-run-id> \
     --skip-dispatch
   ```

   สำหรับ stable ให้ส่ง `--windows-node-tag vX.Y.Z` เพิ่มด้วย ตัวช่วยจะตรวจสอบที่มาของบันทึกประจำรุ่น ไบต์จากการตรวจสอบล่วงหน้า npm หลักฐานการติดตั้ง/อัปเดต Parallels หลักฐานแพ็กเกจ Telegram และแผนการเผยแพร่ Plugin จากนั้นจะแสดงคำสั่งเผยแพร่

   `OpenClaw Release Publish` จะส่งแพ็กเกจ Plugin ที่เลือกหรือทั้งหมดที่เผยแพร่ได้ไปยัง npm และส่งชุดเดียวกันไปยัง ClawHub แบบขนาน จากนั้นโปรโมตอาร์ติแฟกต์การตรวจสอบล่วงหน้า npm ของ OpenClaw ที่เตรียมไว้ด้วย dist-tag ที่ตรงกัน เมื่อการเผยแพร่ Plugin ไปยัง npm สำเร็จ เช็กเอาต์รุ่นยังคงเป็นรูทของผลิตภัณฑ์/ข้อมูล ขณะที่การวางแผนและการตรวจสอบขั้นสุดท้ายดำเนินการจากเช็กเอาต์ต้นทางเวิร์กโฟลว์ที่เชื่อถือได้ตรงตามเวอร์ชัน เพื่อไม่ให้คอมมิตรุ่นเก่าใช้เครื่องมือออกรุ่นที่ล้าสมัยโดยไม่แจ้งให้ทราบ ก่อนเริ่มโพรเซสลูกสำหรับการเผยแพร่ใดๆ ระบบจะเรนเดอร์และแคชเนื้อหาของรุ่นบน GitHub ที่แน่นอน เมื่อส่วน `CHANGELOG.md` ที่ตรงกันทั้งหมดมีขนาดไม่เกินขีดจำกัด 125,000 อักขระของ GitHub และเพดานความปลอดภัย 125,000 ไบต์ที่ตรงกันของตัวเรนเดอร์ หน้าเว็บจะมีส่วน `## YYYY.M.PATCH` นั้นอย่างครบถ้วนรวมถึงหัวข้อ เมื่อส่วนต้นทางมีขนาดเกิน หน้าเว็บจะเก็บบันทึกบรรณาธิการแบบจัดกลุ่มไว้อย่างครบถ้วน และแทนที่บันทึกการมีส่วนร่วมที่มีขนาดใหญ่เกินด้วยลิงก์แบบคงที่ไปยังบันทึกฉบับเต็มใน `CHANGELOG.md` ที่ตรึงกับแท็ก โดยจะไม่มีการเผยแพร่บันทึกเพียงบางส่วนหรือรายการหัวข้อย่อยที่ถูกตัด เวิร์กโฟลว์จะเลือกเนื้อหาฉบับเต็มหรือฉบับย่อก่อนเพิ่ม `### Release verification`; หากส่วนท้ายของหลักฐานทำให้เกินขีดจำกัด ระบบจะคงเนื้อหามาตรฐานไว้และใช้หลักฐานแนบที่เปลี่ยนแปลงไม่ได้แทน รุ่น stable ที่เผยแพร่ไปยัง npm `latest` จะกลายเป็นรุ่นล่าสุดบน GitHub ส่วนรุ่นบำรุงรักษา stable ที่คงไว้บน npm `beta` จะถูกสร้างด้วย GitHub `latest=false` เวิร์กโฟลว์ยังอัปโหลดหลักฐานการพึ่งพาจากการตรวจสอบล่วงหน้า แมนิเฟสต์การตรวจสอบแบบเต็ม และหลักฐานการตรวจสอบรีจิสทรีหลังเผยแพร่ไปยังรุ่นบน GitHub เพื่อใช้ตอบสนองต่อเหตุขัดข้องหลังออกรุ่น โดยจะแสดง ID ของรันลูกทันที อนุมัติเกตสภาพแวดล้อมการออกรุ่นโดยอัตโนมัติในส่วนที่โทเค็นเวิร์กโฟลว์ได้รับอนุญาตให้อนุมัติ สรุปงานลูกที่ล้มเหลวพร้อมส่วนท้ายของบันทึก สร้างหน้ารุ่นฉบับร่างบน GitHub ไว้ล่วงหน้า และโปรโมตอาร์ติแฟกต์ Windows และ Android พร้อมกับการเผยแพร่ OpenClaw ไปยัง npm ปิดงานหน้ารุ่นและหลักฐานการพึ่งพาเมื่อขั้นตอนเหล่านั้นสำเร็จ รอ ClawHub ทุกครั้งที่กำลังเผยแพร่ OpenClaw ไปยัง npm จากนั้นเรียกใช้ตัวตรวจสอบ beta จาก main ที่เชื่อถือได้ และอัปโหลดหลักฐานหลังเผยแพร่สำหรับรุ่นบน GitHub, แพ็กเกจ npm, แพ็กเกจ Plugin บน npm ที่เลือก, แพ็กเกจ ClawHub ที่เลือก, ID รันของเวิร์กโฟลว์ลูก และ ID รัน NPM Telegram ซึ่งเป็นตัวเลือก ตัวตรวจสอบการบูตสแตรป ClawHub กำหนดให้มีพาธและ SHA ของเวิร์กโฟลว์จาก main ที่เชื่อถือได้ตรงตามค่า ความพยายามรันของผู้ผลิตและรันปลายทาง SHA ของรุ่น ชุดแพ็กเกจที่ร้องขอ ทูเพิลอาร์ติแฟกต์แพ็กเกจที่เปลี่ยนแปลงไม่ได้ และอาร์ติแฟกต์การอ่านค่ากลับจากรีจิสทรีปลายทาง โดยจะไม่ยอมรับรันจาก ref รุ่นแบบเก่าที่สำเร็จ

   จากนั้นเรียกใช้การยอมรับแพ็กเกจหลังเผยแพร่กับแพ็กเกจ `openclaw@YYYY.M.PATCH-beta.N` หรือ `openclaw@beta` ที่เผยแพร่แล้ว หาก prerelease ที่พุชหรือเผยแพร่แล้วต้องแก้ไข ให้สร้างหมายเลข prerelease ถัดไปที่ตรงกัน ห้ามลบหรือเขียนทับหมายเลขเดิม

10. เมื่อความพยายามเผยแพร่ล้มเหลว ให้คง Release SHA เดิมไว้ เว้นแต่ความล้มเหลวนั้นพิสูจน์ว่ามีข้อบกพร่องในผลิตภัณฑ์หรือบันทึกการเปลี่ยนแปลง ให้ทำต่อจากโพรเซสลูกและอาร์ติแฟกต์ที่สำเร็จและเปลี่ยนแปลงไม่ได้ ห้ามสร้างใหม่หรือเผยแพร่ซ้ำแพ็กเกจเวอร์ชันที่สำเร็จแล้ว
11. สำหรับ stable ให้ดำเนินการต่อเมื่อ beta หรือ release candidate ที่ผ่านการตรวจสอบมีหลักฐานการตรวจสอบที่กำหนดเท่านั้น การเผยแพร่ stable ไปยัง npm จะผ่าน `OpenClaw Release Publish` เช่นกัน โดยนำอาร์ติแฟกต์การตรวจสอบล่วงหน้าที่สำเร็จแล้วกลับมาใช้ผ่าน `preflight_run_id` ความพร้อมสำหรับการออกรุ่น stable บน macOS ยังกำหนดให้มี `.zip`, `.dmg`, `.dSYM.zip` ที่จัดแพ็กเกจแล้ว และ `appcast.xml` ที่อัปเดตแล้วบน `main`; เวิร์กโฟลว์เผยแพร่ macOS จะเผยแพร่ appcast ที่ลงนามแล้วไปยัง `main` สาธารณะโดยอัตโนมัติหลังตรวจสอบอาร์ติแฟกต์รุ่นแล้ว หรือเปิด/อัปเดต PR ของ appcast หากการป้องกันสาขาขัดขวางการพุชโดยตรง ความพร้อมสำหรับ Windows Hub รุ่น stable กำหนดให้มีอาร์ติแฟกต์ `OpenClawCompanion-Setup-x64.exe`, `OpenClawCompanion-Setup-arm64.exe` และ `OpenClawCompanion-SHA256SUMS.txt` ที่ลงนามแล้วในรุ่น OpenClaw บน GitHub ส่งแท็กรุ่น `openclaw/openclaw-windows-node` ที่ลงนามแล้วตรงตามค่าเป็น `windows_node_tag` และแผนที่ไดเจสต์ของตัวติดตั้งที่ candidate อนุมัติแล้วเป็น `windows_node_installer_digests`; `OpenClaw Release Publish` จะคงฉบับร่างของรุ่นไว้ ส่ง `Windows Node Release` และตรวจสอบอาร์ติแฟกต์ทั้งสามก่อนเผยแพร่
12. หลังเผยแพร่ ให้เรียกใช้ตัวตรวจสอบหลังเผยแพร่ของ npm, E2E Telegram แบบสแตนด์อโลนที่ใช้ npm ซึ่งเผยแพร่แล้ว (ตัวเลือก) เมื่อต้องการหลักฐานช่องทางหลังเผยแพร่, การโปรโมต dist-tag เมื่อจำเป็น, ตรวจสอบหน้ารุ่นบน GitHub ที่สร้างขึ้น, ดำเนินขั้นตอนประกาศรุ่น จากนั้นทำ [การปิดงาน stable บน main](#stable-main-closeout) ให้เสร็จก่อนถือว่าการออกรุ่น stable เสร็จสมบูรณ์

## การปิดงาน stable บน main

การเผยแพร่ stable จะยังไม่สมบูรณ์จนกว่า `main` จะมีสถานะของรุ่นที่จัดส่งจริง

1. เริ่มจาก `main` ล่าสุดที่สดใหม่ ตรวจสอบ `release/YYYY.M.PATCH` เทียบกับรายการนั้น และพอร์ตการแก้ไขจริงที่ไม่มีอยู่ใน `main` ไปข้างหน้า ห้ามผสานอะแดปเตอร์ด้านความเข้ากันได้ การทดสอบ หรือการตรวจสอบที่ใช้เฉพาะรุ่นเข้าไปใน `main` ที่ใหม่กว่าโดยไม่พิจารณา
2. สำหรับเส้นทางปกติ ให้ตั้ง `main` เป็นเวอร์ชัน stable ที่จัดส่ง การปิดงานล่าช้าอาจใช้ `main` หลังจากที่ค่านั้นเลื่อนไปเป็น OpenClaw CalVer รุ่น stable ที่ใหม่กว่าแล้ว ห้ามดาวน์เกรดขบวนการออกรุ่นที่เริ่มไปแล้วเพียงเพื่อปิดงานรุ่นก่อนหน้า ตัวตรวจสอบยังคงกำหนดให้มีส่วนบันทึกการเปลี่ยนแปลงและรายการ appcast ของรุ่นที่จัดส่งตรงตามค่า และบันทึกเวอร์ชันกับ SHA ของ `main` ที่ใช้จริง เรียกใช้ `pnpm release:prep` หลังการเปลี่ยนแปลงเวอร์ชันรูททุกครั้ง จากนั้นเรียกใช้ `pnpm deps:shrinkwrap:generate`
3. ทำให้ส่วน `## YYYY.M.PATCH` ของ `CHANGELOG.md` บน `main` ตรงกับสาขารุ่นที่แท็กทุกประการ รวมการอัปเดต stable ของ `appcast.xml` เมื่อมีการเผยแพร่รุ่น Mac
4. ห้ามเพิ่ม `YYYY.M.PATCH+1`, เวอร์ชัน beta หรือส่วนบันทึกการเปลี่ยนแปลงในอนาคตที่ว่างเปล่าลงใน `main` จนกว่าผู้ปฏิบัติงานจะเริ่มขบวนการออกรุ่นนั้นอย่างชัดเจน
5. เรียกใช้ `pnpm release:generated:check`, `pnpm deps:shrinkwrap:check` และ `OPENCLAW_TESTBOX=1 pnpm check:changed` พุช แล้วตรวจสอบว่า `origin/main` มีเวอร์ชันและบันทึกการเปลี่ยนแปลงที่จัดส่งก่อนถือว่าการออกรุ่น stable เสร็จสิ้น
6. ปรับตัวแปรของที่เก็บ `RELEASE_ROLLBACK_DRILL_ID` และ `RELEASE_ROLLBACK_DRILL_DATE` ให้เป็นปัจจุบันหลังการซ้อมย้อนกลับแบบส่วนตัวแต่ละครั้ง

`OpenClaw Stable Main Closeout` เริ่มจากการพุช `main` ที่มีเวอร์ชัน บันทึกการเปลี่ยนแปลง และ appcast ที่จัดส่งหลังเผยแพร่ stable โดยจะอ่านหลักฐานหลังเผยแพร่ที่เปลี่ยนแปลงไม่ได้เพื่อผูกแท็กที่จัดส่งกับรัน Full Release Validation และ Publish จากนั้นตรวจสอบสถานะ stable บน main รุ่น ช่วงเฝ้าระวัง stable ที่บังคับ และหลักฐานประสิทธิภาพที่ใช้บล็อก โดยจะแนบแมนิเฟสต์การปิดงานและเช็กซัมที่เปลี่ยนแปลงไม่ได้ไปยังรุ่นบน GitHub ทริกเกอร์การพุชอัตโนมัติจะข้ามรุ่นแบบเก่าที่มีมาก่อนหลักฐานหลังเผยแพร่แบบเปลี่ยนแปลงไม่ได้ และจะไม่ถือว่าการข้ามนั้นเป็นการปิดงานที่เสร็จสมบูรณ์

การปิดงานที่สมบูรณ์ต้องมีทั้งอาร์ติแฟกต์และเช็กซัมที่ตรงกัน แมนิเฟสต์บางส่วนจะเล่นซ้ำ SHA ของ `main` และการซ้อมย้อนกลับที่บันทึกไว้เพื่อสร้างไบต์ที่เหมือนกัน จากนั้นแนบเช็กซัมที่ขาดหายไป คู่ที่ไม่ถูกต้องหรือเช็กซัมที่ไม่มีแมนิเฟสต์จะยังคงบล็อกอยู่ รันที่ทริกเกอร์ด้วยการพุชแต่ไม่มีตัวแปรของที่เก็บสำหรับการซ้อมย้อนกลับจะข้ามโดยไม่ทำให้การปิดงานเสร็จสมบูรณ์ ส่วนบันทึกการซ้อมที่ขาดหายไปหรือมีอายุเกิน 90 วันจะยังคงบล็อกการปิดงานแบบแมนนวลที่มีหลักฐานรองรับ คำสั่งกู้คืนแบบส่วนตัวยังคงอยู่ในคู่มือปฏิบัติการสำหรับผู้ดูแลเท่านั้น ใช้การส่งแบบแมนนวลเฉพาะเพื่อซ่อมแซมหรือเล่นซ้ำการปิดงาน stable ที่มีหลักฐานรองรับ

หากพาเรนต์ Release Publish ล้มเหลวหลังจากแนบหลักฐาน npm/Plugin ที่เปลี่ยนแปลงไม่ได้แล้วเท่านั้น ให้ซ่อมแซมและเผยแพร่อาร์ติแฟกต์แพลตฟอร์ม stable ทั้งหมดก่อน จากนั้นผู้ดูแลอาจส่งการปิดงานแบบแมนนวลด้วย `allow_failed_publish_recovery=true`; โหมดนั้นยอมรับเฉพาะพาเรนต์ที่ล้มเหลวแต่ทำงานเสร็จแล้ว และยังต้องมีสัญญาอาร์ติแฟกต์ Android และ Windows ที่ตรงตามค่า ไดเจสต์ SHA-256 ของ GitHub การตรวจสอบเช็กซัม ที่มาของ Android และการโปรโมต Windows ที่พาเรนต์ส่งและสำเร็จ ซึ่งการตรวจสอบ Authenticode และไดเจสต์ที่ candidate อนุมัติตรงกับตัวติดตั้งที่เผยแพร่แล้ว พร้อมกับการตรวจสอบ macOS/appcast ตามปกติ การปิดงานอัตโนมัติจากการพุชจะไม่เปิดใช้โหมดกู้คืนนี้

แท็กการแก้ไขสำรองแบบเก่าอาจนำหลักฐานแพ็กเกจฐานกลับมาใช้ได้เฉพาะเมื่อแท็กการแก้ไขชี้ไปยังคอมมิตต้นทางเดียวกับแท็ก stable ฐาน รุ่น Android ของแท็กนั้นจะนำ APK ที่ตรวจสอบแล้วจากแท็กฐานกลับมาใช้ และเพิ่มที่มาสำหรับแท็กการแก้ไข การแก้ไขที่มีต้นทางต่างกันต้องเผยแพร่และตรวจสอบหลักฐานแพ็กเกจของตนเอง และใช้ `versionCode` ของ Android ที่สูงกว่า

## การตรวจสอบล่วงหน้าก่อนออกรุ่น

- เรียกใช้ `pnpm check:test-types` ก่อนการตรวจสอบล่วงหน้าก่อนออกรุ่น เพื่อให้ TypeScript สำหรับการทดสอบยังได้รับการครอบคลุมนอกเกต `pnpm check` ในเครื่องที่เร็วกว่า
- เรียกใช้ `pnpm check:architecture` ก่อนการตรวจสอบล่วงหน้าก่อนออกรุ่น เพื่อให้การตรวจสอบวงจรการนำเข้าและขอบเขตสถาปัตยกรรมที่กว้างกว่าผ่านนอกเกตในเครื่องที่เร็วกว่า
- เรียกใช้ `pnpm build && pnpm ui:build` ก่อน `pnpm release:check` เพื่อให้อาร์ติแฟกต์รุ่น `dist/*` ที่คาดไว้และบันเดิล Control UI มีอยู่สำหรับขั้นตอนตรวจสอบแพ็ก
- เรียกใช้ `pnpm release:prep` หลังเพิ่มเวอร์ชันรูทและก่อนแท็ก คำสั่งนี้เรียกใช้ตัวสร้างรุ่นแบบกำหนดผลลัพธ์แน่นอนทุกตัวที่มักคลาดเคลื่อนหลังการเปลี่ยนแปลงเวอร์ชัน/การกำหนดค่า/API ได้แก่ เวอร์ชัน Plugin, shrinkwrap ของ npm, รายการ Plugin, สคีมาการกำหนดค่าพื้นฐาน, เมทาดาทาการกำหนดค่าช่องทางที่รวมมาให้, ค่าพื้นฐานของเอกสารการกำหนดค่า, การส่งออก Plugin SDK และค่าพื้นฐาน API ของ Plugin SDK โดย `pnpm release:check` จะเรียกใช้การป้องกันเหล่านั้นซ้ำในโหมดตรวจสอบ (รวมถึงการตรวจสอบงบประมาณพื้นผิว Plugin SDK) และรายงานความล้มเหลวจากความคลาดเคลื่อนของสิ่งที่สร้างขึ้นทั้งหมดในรอบเดียว ก่อนเรียกใช้การตรวจสอบการออกรุ่นของแพ็กเกจ
- การซิงค์เวอร์ชัน Plugin จะอัปเดตแพ็กเกจรันไทม์ `@openclaw/ai` ที่เผยแพร่ได้ เวอร์ชันแพ็กเกจ Plugin ทางการ และค่าขั้นต่ำ `openclaw.compat.pluginApi` ที่มีอยู่ให้เป็นเวอร์ชันรุ่นของ OpenClaw โดยค่าเริ่มต้น ให้ถือว่าฟิลด์นั้นเป็นค่าขั้นต่ำของ API Plugin SDK/รันไทม์ ไม่ใช่เพียงสำเนาเวอร์ชันแพ็กเกจ สำหรับการออกรุ่นเฉพาะ Plugin ที่ตั้งใจให้ยังเข้ากันได้กับโฮสต์ OpenClaw รุ่นเก่า ให้คงค่าขั้นต่ำไว้ที่ API โฮสต์เก่าสุดที่รองรับ และบันทึกตัวเลือกนั้นไว้ในหลักฐานการออกรุ่น Plugin
- เรียกใช้เวิร์กโฟลว์ `Full Release Validation` แบบแมนนวลก่อนอนุมัติการออกรุ่น เพื่อเริ่มกล่องทดสอบก่อนออกรุ่นทั้งหมดจากจุดเริ่มต้นเดียว เวิร์กโฟลว์นี้รับสาขา แท็ก หรือ SHA คอมมิตแบบเต็ม ส่ง `CI` แบบแมนนวล และส่ง `OpenClaw Release Checks` สำหรับการทดสอบควันการติดตั้ง การยอมรับแพ็กเกจ การตรวจสอบแพ็กเกจข้ามระบบปฏิบัติการ ความสอดคล้องของ QA Lab, Matrix และเลน Telegram รัน stable และรันแบบเต็มจะรวมการเฝ้าระวังเส้นทางออกรุ่นแบบ live/E2E และ Docker อย่างละเอียดครบถ้วนเสมอ ส่วน `run_release_soak=true` จะคงไว้สำหรับการเฝ้าระวัง beta ที่ระบุอย่างชัดเจน Package Acceptance ให้ E2E Telegram ของแพ็กเกจมาตรฐานระหว่างการตรวจสอบ candidate เพื่อหลีกเลี่ยงการมีตัวโพลแบบ live ตัวที่สองทำงานพร้อมกัน

  ระบุ `release_package_spec` หลังเผยแพร่ beta เพื่อนำแพ็กเกจ npm ที่จัดส่งแล้วกลับมาใช้ในการตรวจสอบรุ่น, Package Acceptance และ E2E Telegram ของแพ็กเกจ โดยไม่สร้าง tarball รุ่นใหม่ ระบุ `npm_telegram_package_spec` เฉพาะเมื่อ Telegram ควรใช้แพ็กเกจที่เผยแพร่แล้วคนละแพ็กเกจกับส่วนอื่นของการตรวจสอบรุ่น ระบุ `package_acceptance_package_spec` เมื่อ Package Acceptance ควรใช้แพ็กเกจที่เผยแพร่แล้วคนละแพ็กเกจกับข้อกำหนดแพ็กเกจรุ่น ระบุ `evidence_package_spec` เมื่อรายงานหลักฐานรุ่นควรพิสูจน์ว่าการตรวจสอบตรงกับแพ็กเกจ npm ที่เผยแพร่แล้ว โดยไม่บังคับใช้ E2E Telegram

  ```bash
  node scripts/full-release-validation-at-sha.mjs \
    --sha <code-sha> \
    --target-ref release/YYYY.M.PATCH
  ```

- เรียกใช้เวิร์กโฟลว์ `Package Acceptance` แบบแมนนวลเมื่อต้องการหลักฐานจากช่องทางเสริมสำหรับแพ็กเกจที่เป็นตัวเลือก ขณะที่งานรีลีสยังดำเนินต่อไป ใช้ `source=npm` สำหรับ `openclaw@beta`, `openclaw@latest` หรือเวอร์ชันรีลีสที่ระบุแน่นอน; ใช้ `source=ref` เพื่อแพ็ก branch/tag/SHA ของ `package_ref` ที่เชื่อถือได้ด้วยชุดทดสอบ `workflow_ref` ปัจจุบัน; ใช้ `source=url` สำหรับ tarball HTTPS สาธารณะที่ต้องระบุ SHA-256 และอยู่ภายใต้นโยบาย URL สาธารณะที่เข้มงวด; ใช้ `source=trusted-url` สำหรับนโยบายแหล่งที่เชื่อถือได้ที่มีชื่อกำกับ โดยต้องใช้ `trusted_source_id` และ SHA-256; หรือใช้ `source=artifact` สำหรับ tarball ที่อัปโหลดโดยการเรียกใช้ GitHub Actions อื่น

  เวิร์กโฟลว์จะแปลงตัวเลือกเป็น `package-under-test` ใช้ตัวจัดกำหนดการรีลีส Docker E2E ซ้ำกับ tarball นั้น และสามารถเรียกใช้ QA ของ Telegram กับ tarball เดียวกันด้วย `telegram_mode=mock-openai` หรือ `telegram_mode=live-frontier` เมื่อเลน Docker ที่เลือกมี `published-upgrade-survivor` อาร์ติแฟกต์แพ็กเกจจะเป็นตัวเลือก และ `published_upgrade_survivor_baseline` จะเลือกค่าอ้างอิงที่เผยแพร่แล้ว `update-restart-auth` ใช้แพ็กเกจตัวเลือกเป็นทั้ง CLI ที่ติดตั้งและแพ็กเกจภายใต้การทดสอบ เพื่อให้ทดสอบเส้นทางการรีสตาร์ตที่มีการจัดการของคำสั่งอัปเดตจากตัวเลือก

  ตัวอย่าง:

  ```bash
  gh workflow run package-acceptance.yml --ref main -f workflow_ref=main -f source=npm -f package_spec=openclaw@beta -f suite_profile=product -f published_upgrade_survivor_baseline=openclaw@2026.4.26 -f telegram_mode=mock-openai
  ```

  โปรไฟล์ที่ใช้ทั่วไป:
  - `smoke`: เลนการติดตั้ง/ช่องทาง/เอเจนต์ เครือข่าย Gateway และการโหลดการกำหนดค่าใหม่
  - `package`: เลนแพ็กเกจ/อัปเดต/รีสตาร์ต/Plugin ที่ทำงานโดยตรงกับอาร์ติแฟกต์ โดยไม่มี OpenWebUI หรือ ClawHub แบบใช้งานจริง
  - `product`: โปรไฟล์แพ็กเกจ รวมถึงช่องทาง MCP การล้างข้อมูล cron/เอเจนต์ย่อย การค้นหาเว็บของ OpenAI และ OpenWebUI
  - `full`: ส่วนย่อยของเส้นทางรีลีส Docker พร้อม OpenWebUI
  - `custom`: การเลือก `docker_lanes` แบบระบุแน่นอนสำหรับการเรียกใช้ซ้ำแบบเจาะจง

- เรียกใช้เวิร์กโฟลว์ `CI` แบบแมนนวลโดยตรง เมื่อต้องการเพียงความครอบคลุมของ CI ปกติที่ทำซ้ำได้แน่นอนสำหรับตัวเลือกรีลีส การสั่งรัน CI แบบแมนนวลจะข้ามการจำกัดขอบเขตตามการเปลี่ยนแปลง และบังคับใช้ชาร์ด Linux Node, ชาร์ด Plugin ที่รวมมาให้, ชาร์ดสัญญาของ Plugin และช่องทาง, ความเข้ากันได้กับ Node 22, `check-*`, `check-additional-*`, การตรวจสอบเบื้องต้นของอาร์ติแฟกต์ที่บิลด์แล้ว, การตรวจสอบเอกสาร, Skills ของ Python, Windows, macOS และเลน i18n ของ Control UI การรัน CI แบบแมนนวลแยกเดี่ยวจะเรียกใช้ Android เฉพาะเมื่อสั่งรันด้วย `include_android=true`; `Full Release Validation` จะส่งอินพุตดังกล่าวให้ CI ลูก

  ```bash
  gh workflow run ci.yml --ref release/YYYY.M.PATCH -f include_android=true
  ```

- ระบบอัตโนมัติสำหรับรีลีสของผู้ดูแลใช้กระบวนการตรวจสอบเบื้องต้นแล้วจึงเลื่อนขั้น:
  - การเผยแพร่ npm จริงต้องผ่าน npm `preflight_run_id` ที่สำเร็จ
  - การประสานงานและการตรวจสอบเบื้องต้นสำหรับการเผยแพร่เบต้าและเสถียรตามปกติใช้ `main` ที่เชื่อถือได้กับแท็กเป้าหมายที่ตรงกัน การเผยแพร่และการตรวจสอบเบื้องต้นสำหรับ Tideclaw alpha ใช้บรานช์ alpha ที่ตรงกัน
  - รีลีส npm แบบเสถียรมีค่าเริ่มต้นเป็น `beta`; การเผยแพร่ npm แบบเสถียรสามารถกำหนดเป้าหมายเป็น `latest` อย่างชัดเจนผ่านอินพุตของเวิร์กโฟลว์
  - การเปลี่ยนแปลง dist-tag ของ npm โดยใช้โทเค็นอยู่ใน `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml` เนื่องจาก `npm dist-tag add` ยังคงต้องใช้ `NPM_TOKEN` ขณะที่รีโพซิทอรีต้นทางคงการเผยแพร่แบบ OIDC เท่านั้น
  - `macOS Release` สาธารณะใช้สำหรับการตรวจสอบเท่านั้น เมื่อแท็กอยู่บนบรานช์รีลีสเท่านั้นแต่เวิร์กโฟลว์ถูกสั่งเรียกจาก `main` ให้ตั้งค่า `public_release_branch=release/YYYY.M.PATCH`
  - การเผยแพร่ macOS จริงต้องผ่าน macOS `preflight_run_id` และ `validate_run_id` ที่สำเร็จ
  - เส้นทางเผยแพร่จริงจะเลื่อนขั้นอาร์ติแฟกต์ที่เตรียมไว้แทนการบิลด์ใหม่อีกครั้ง
- สำหรับรีลีสแก้ไขแบบเสถียร เช่น `YYYY.M.PATCH-N` ตัวตรวจสอบหลังเผยแพร่ยังตรวจสอบเส้นทางอัปเกรดในคำนำหน้าชั่วคราวเดียวกันจาก `YYYY.M.PATCH` เป็น `YYYY.M.PATCH-N` เพื่อไม่ให้การแก้ไขรีลีสทิ้งการติดตั้งแบบส่วนกลางรุ่นเก่าไว้บนเพย์โหลดเสถียรพื้นฐานโดยไม่มีการแจ้งเตือน
- การตรวจสอบเบื้องต้นของรีลีส npm จะปฏิเสธโดยค่าเริ่มต้น เว้นแต่ tarball จะมีทั้ง `dist/control-ui/index.html` และเพย์โหลด `dist/control-ui/assets/` ที่ไม่ว่างเปล่า เพื่อไม่ให้เราจัดส่งแดชบอร์ดเบราว์เซอร์ที่ว่างเปล่าอีก
- การตรวจสอบหลังเผยแพร่ยังตรวจสอบว่าเอนทรีพอยต์ของ Plugin และข้อมูลเมตาของแพ็กเกจที่เผยแพร่แล้วมีอยู่ในเลย์เอาต์รีจิสทรีที่ติดตั้งแล้ว รีลีสที่จัดส่งโดยไม่มีเพย์โหลดรันไทม์ของ Plugin จะไม่ผ่านตัวตรวจสอบหลังเผยแพร่และไม่สามารถเลื่อนขั้นเป็น `latest` ได้
- `pnpm test:install:smoke` ยังบังคับใช้งบประมาณ `unpackedSize` ของ npm pack กับ tarball อัปเดตตัวเลือกด้วย เพื่อให้ E2E ของตัวติดตั้งตรวจพบการขยายขนาดแพ็กโดยไม่ตั้งใจก่อนถึงเส้นทางเผยแพร่รีลีส
- หากงานรีลีสแตะการวางแผน CI ไฟล์กำหนดเวลาของส่วนขยาย หรือเมทริกซ์การทดสอบส่วนขยาย ให้สร้างเอาต์พุตเมทริกซ์ `plugin-prerelease-extension-shard` ที่ตัววางแผนเป็นเจ้าของขึ้นใหม่จาก `.github/workflows/plugin-prerelease.yml` และตรวจสอบก่อนอนุมัติ เพื่อไม่ให้บันทึกประจำรุ่นอธิบายเลย์เอาต์ CI ที่ล้าสมัย
- ความพร้อมของรีลีส macOS แบบเสถียรยังรวมพื้นผิวของตัวอัปเดตด้วย: รีลีส GitHub ต้องมี `.zip`, `.dmg` และ `.dSYM.zip` ที่จัดแพ็กแล้ว; `appcast.xml` บน `main` ต้องชี้ไปยังไฟล์ zip แบบเสถียรใหม่หลังเผยแพร่ (เวิร์กโฟลว์เผยแพร่ macOS จะคอมมิตให้โดยอัตโนมัติ หรือเปิด PR ของ appcast เมื่อการพุชโดยตรงถูกบล็อก); แอปที่จัดแพ็กแล้วต้องคงรหัสบันเดิลที่ไม่ใช่แบบดีบัก URL ฟีด Sparkle ที่ไม่ว่างเปล่า และ `CFBundleVersion` ที่ไม่น้อยกว่าค่าขั้นต่ำบิลด์ Sparkle มาตรฐานสำหรับเวอร์ชันรีลีสนั้น

## กล่องทดสอบรีลีส

`Full Release Validation` คือวิธีที่ผู้ปฏิบัติงานเริ่มเมทริกซ์ผลิตภัณฑ์แบบเต็มจากเอนทรีพอยต์เดียว ใช้ตัวช่วยเพื่อให้ทุกเวิร์กโฟลว์ลูกทำงานจากบรานช์ชั่วคราวที่ตรึงไว้กับ SHA ของเวิร์กโฟลว์ `main` ที่เชื่อถือได้หนึ่งรายการ ขณะที่คอมมิตที่ร้องขอยังคงเป็นตัวเลือกที่กำลังทดสอบ:

```bash
pnpm ci:full-release \
  --sha <code-sha> \
  --target-ref release/YYYY.M.PATCH
```

ตัวช่วยจะดึงข้อมูล `origin/main` ปัจจุบัน พุช `release-ci/<workflow-sha>-...` ที่คอมมิตเวิร์กโฟลว์ที่เชื่อถือได้นั้น อนุมาน `beta` จากเวอร์ชันแพ็กเกจ alpha/beta และใช้ `stable` ในกรณีอื่น สั่งเรียก `Full Release Validation` จากบรานช์ชั่วคราวพร้อม `ref=<target-sha>` ตรวจสอบว่า `headSha` ของทุกเวิร์กโฟลว์ลูกตรงกับ SHA ของเวิร์กโฟลว์แม่ที่ตรึงไว้ แล้วลบบรานช์ชั่วคราว ส่ง `-f reuse_evidence=false` เพื่อบังคับให้เรียกใช้ใหม่, `-f release_profile=full` สำหรับการกวาดตรวจเชิงคำแนะนำแบบกว้าง หรือ `--workflow-sha <trusted-main-sha>` เพื่อตรึงคอมมิตที่เก่ากว่าซึ่งยังเข้าถึงได้จาก `origin/main` ปัจจุบัน ตัวเวิร์กโฟลว์เองจะไม่เขียนรีเฟอเรนซ์ของรีโพซิทอรี วิธีนี้ทำให้เครื่องมือรีลีสที่ใช้เฉพาะบน main ยังพร้อมใช้งานโดยไม่เพิ่มคอมมิตเครื่องมือลงในตัวเลือก และหลีกเลี่ยงการพิสูจน์การเรียกใช้ลูก `main` ที่ใหม่กว่าโดยไม่ตั้งใจ

หลังจาก Code SHA ผ่านแล้ว ให้คอมมิตเฉพาะ `CHANGELOG.md` และเรียกใช้ตัวช่วยเดิมด้วย Release SHA:

```bash
pnpm ci:full-release \
  --sha <release-sha> \
  --target-ref release/YYYY.M.PATCH
```

เวิร์กโฟลว์แม่ครั้งที่สองจะนำหลักฐานผลิตภัณฑ์กลับมาใช้ซ้ำเฉพาะเมื่อ GitHub พิสูจน์ว่า Release SHA สืบต่อจาก Code SHA และชุดพาธที่เปลี่ยนแปลงทั้งหมดมีเพียง `CHANGELOG.md` เท่านั้น โดยจะบันทึก `changelog-only-release-v1` และไม่สั่งเรียกเวิร์กโฟลว์ลูกของผลิตภัณฑ์ การตรวจสอบเบื้องต้นของ npm และการยอมรับแพ็กเกจ/การติดตั้งยังคงทำงานบน Release SHA เนื่องจากไบต์ของ tarball เปลี่ยนไป

สำหรับ Code SHA ใหม่ เวิร์กโฟลว์จะแก้ไขเป้าหมาย สั่งเรียก `CI` แบบเรียกใช้ด้วยตนเอง แล้วจึงสั่งเรียก `OpenClaw Release Checks` โดย `OpenClaw Release Checks` จะกระจายงานไปยังการทดสอบควันการติดตั้ง การตรวจสอบรีลีสข้ามระบบปฏิบัติการ ความครอบคลุมเส้นทางรีลีส Docker แบบไลฟ์/E2E เมื่อเปิดใช้ soak การยอมรับแพ็กเกจพร้อม E2E แพ็กเกจ Telegram มาตรฐาน ความสอดคล้องของ QA Lab, Matrix แบบไลฟ์ และ Telegram แบบไลฟ์ การเรียกใช้แบบเต็ม/ทั้งหมดจะยอมรับได้เฉพาะเมื่อสรุป `Full Release Validation` แสดงว่า `normal_ci`, `plugin_prerelease` และ `release_checks` สำเร็จ เว้นแต่การเรียกใช้ซ้ำแบบเจาะจงจะจงใจข้ามลูก `Plugin Prerelease` ที่แยกต่างหาก ใช้ลูก `npm-telegram` แบบสแตนด์อโลนเฉพาะสำหรับการเรียกใช้ซ้ำแบบเจาะจงของแพ็กเกจที่เผยแพร่แล้วพร้อม `release_package_spec` หรือ `npm_telegram_package_spec` สรุปตัวตรวจสอบขั้นสุดท้ายมีตารางงานที่ช้าที่สุดสำหรับการเรียกใช้ลูกแต่ละครั้ง เพื่อให้ผู้จัดการรีลีสเห็นเส้นทางวิกฤตปัจจุบันโดยไม่ต้องดาวน์โหลดบันทึก

เวิร์กโฟลว์ลูกด้านประสิทธิภาพผลิตภัณฑ์ใช้เฉพาะอาร์ติแฟกต์ในเส้นทางรีลีสนี้ เวิร์กโฟลว์
หลักจะสั่งเรียกพร้อม `publish_reports=false` และการตรวจสอบจะถูกปฏิเสธ
เว้นแต่มาตรการป้องกันแบบใช้อาร์ติแฟกต์เท่านั้นจะพิสูจน์ว่าตัวเผยแพร่รายงาน Clawgrit ยังคง
ถูกข้าม

ดู [การตรวจสอบรีลีสแบบเต็ม](/th/reference/full-release-validation) สำหรับเมทริกซ์ขั้นตอนทั้งหมด ชื่องานเวิร์กโฟลว์ที่ตรงกัน ความแตกต่างระหว่างโปรไฟล์เสถียรกับแบบเต็ม อาร์ติแฟกต์ และตัวควบคุมการเรียกใช้ซ้ำแบบเจาะจง

เวิร์กโฟลว์ลูกถูกสั่งเรียกจากรีเฟอเรนซ์ที่เชื่อถือได้ซึ่งตรึงด้วย SHA และเรียกใช้ `Full Release Validation` การเรียกใช้ลูกทุกครั้งต้องใช้ SHA ของเวิร์กโฟลว์แม่ที่ตรงกัน ห้ามใช้การสั่งเรียก `--ref main -f ref=<sha>` โดยตรงเป็นหลักฐานรีลีส ให้ใช้ `pnpm ci:full-release --sha <target-sha> --target-ref release/YYYY.M.PATCH`

ใช้ `release_profile` เพื่อเลือกขอบเขตความครอบคลุมแบบไลฟ์/ผู้ให้บริการ:

- `beta`: เส้นทางไลฟ์และ Docker ของ OpenAI/แกนหลักที่สำคัญต่อรีลีสและเร็วที่สุด
- `stable`: ความครอบคลุมผู้ให้บริการ/แบ็กเอนด์แบบเบต้ารวมถึงแบบเสถียรสำหรับการอนุมัติรีลีส
- `full`: แบบเสถียรรวมถึงความครอบคลุมผู้ให้บริการ/สื่อเชิงคำแนะนำแบบกว้าง

การตรวจสอบแบบเสถียรและแบบเต็มจะเรียกใช้การทดสอบไลฟ์/E2E อย่างครอบคลุม เส้นทางรีลีส Docker และการกวาดตรวจความอยู่รอดหลังอัปเกรดจากแพ็กเกจที่เผยแพร่แล้วแบบจำกัดขอบเขตก่อนเลื่อนขั้นเสมอ ใช้ `run_release_soak=true` เพื่อขอการกวาดตรวจแบบเดียวกันสำหรับเบต้า การกวาดตรวจนั้นครอบคลุมแพ็กเกจเสถียรล่าสุดสี่รายการ รวมถึงค่าอ้างอิง `2026.4.23` และ `2026.5.2` ที่ตรึงไว้ ตลอดจนความครอบคลุม `2026.4.15` ที่เก่ากว่า โดยนำค่าอ้างอิงที่ซ้ำกันออกและแบ่งแต่ละค่าอ้างอิงเป็นงานรันเนอร์ Docker ของตนเอง

`OpenClaw Release Checks` ใช้รีเฟอเรนซ์เวิร์กโฟลว์ที่เชื่อถือได้เพื่อแก้ไขรีเฟอเรนซ์เป้าหมายหนึ่งครั้งเป็น `release-package-under-test` และนำอาร์ติแฟกต์นั้นกลับมาใช้ซ้ำในการตรวจสอบข้ามระบบปฏิบัติการ การยอมรับแพ็กเกจ และการตรวจสอบเส้นทางรีลีส Docker เมื่อ soak ทำงาน วิธีนี้ทำให้กล่องทั้งหมดที่เกี่ยวข้องกับแพ็กเกจใช้ไบต์ชุดเดียวกันและหลีกเลี่ยงการบิลด์แพ็กเกจซ้ำ หลังจากเบต้าอยู่บน npm แล้ว ให้ตั้งค่า `release_package_spec=openclaw@YYYY.M.PATCH-beta.N` เพื่อให้การตรวจสอบรีลีสดาวน์โหลดแพ็กเกจที่จัดส่งแล้วหนึ่งครั้ง แยก SHA ของซอร์สบิลด์จาก `dist/build-info.json` และนำอาร์ติแฟกต์นั้นกลับมาใช้ซ้ำสำหรับเลนข้ามระบบปฏิบัติการ การยอมรับแพ็กเกจ เส้นทางรีลีส Docker และ Telegram ของแพ็กเกจ

การทดสอบควันการติดตั้ง OpenAI ข้ามระบบปฏิบัติการใช้ `OPENCLAW_CROSS_OS_OPENAI_MODEL` เมื่อตั้งค่าตัวแปรของรีโพซิทอรี/องค์กร มิฉะนั้นใช้ `openai/gpt-5.6-luna` เนื่องจากเลนนี้พิสูจน์การติดตั้งแพ็กเกจ การเริ่มต้นใช้งาน การเริ่มต้น Gateway และการทำงานของเอเจนต์แบบไลฟ์หนึ่งครั้ง แทนที่จะวัดประสิทธิภาพโมเดลที่มีความสามารถสูงสุด เมทริกซ์ผู้ให้บริการแบบไลฟ์ที่กว้างกว่ายังคงเป็นพื้นที่สำหรับความครอบคลุมเฉพาะโมเดล

ใช้รูปแบบต่อไปนี้ตามขั้นตอนของรีลีส:

```bash
# ตรวจสอบ Code SHA ที่ผลิตภัณฑ์สมบูรณ์
pnpm ci:full-release \
  --sha <code-sha> \
  --target-ref release/YYYY.M.PATCH

# ตรวจสอบ Release SHA ที่เปลี่ยนเฉพาะบันทึกการเปลี่ยนแปลง โดยนำหลักฐานผลิตภัณฑ์จาก Code SHA มาใช้ซ้ำ
pnpm ci:full-release \
  --sha <release-sha> \
  --target-ref release/YYYY.M.PATCH

# หลังเผยแพร่รุ่นเบตา ให้เพิ่ม Telegram E2E สำหรับแพ็กเกจที่เผยแพร่แล้ว
pnpm ci:full-release \
  --sha <release-sha> \
  --target-ref release/YYYY.M.PATCH \
  -f release_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f evidence_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f npm_telegram_provider_mode=mock-openai
```

อย่าใช้ชุดรวมทั้งหมดเป็นการรันซ้ำครั้งแรกหลังแก้ไขเฉพาะจุด หากกล่องใดกล่องหนึ่งล้มเหลว ให้ใช้เวิร์กโฟลว์ลูก งาน เลน Docker โปรไฟล์แพ็กเกจ ผู้ให้บริการโมเดล หรือเลน QA ที่ล้มเหลวสำหรับการพิสูจน์ครั้งถัดไป รันชุดรวมทั้งหมดอีกครั้งเฉพาะเมื่อการแก้ไขเปลี่ยนการประสานงานการเผยแพร่ที่ใช้ร่วมกัน หรือทำให้หลักฐานจากทุกกล่องก่อนหน้านี้ล้าสมัย ตัวตรวจสอบขั้นสุดท้ายของชุดรวมจะตรวจสอบรหัสการรันเวิร์กโฟลว์ลูกที่บันทึกไว้อีกครั้ง ดังนั้นหลังจากรันเวิร์กโฟลว์ลูกซ้ำสำเร็จแล้ว ให้รันซ้ำเฉพาะงานแม่ `Verify full validation` ที่ล้มเหลว

`rerun_group=all` อาจนำการรันชุดรวมที่ผ่านก่อนหน้านี้มาใช้ซ้ำได้ เมื่อโปรไฟล์การเผยแพร่
การตั้งค่าระยะเฝ้าดูที่มีผล และอินพุตการตรวจสอบตรงกัน และ SHA เป้าหมาย
เหมือนกัน หรือเป้าหมายใหม่เป็นลูกหลานซึ่งชุดพาธที่เปลี่ยนแปลงทั้งหมด
เป็น `CHANGELOG.md` เท่านั้น การใช้ซ้ำสำหรับเป้าหมายที่ตรงกันทุกประการจะบันทึก
`exact-target-full-validation-v1`; ส่วน Release SHA หลังการตรวจสอบจะบันทึก
`changelog-only-release-v1` กรณีหลังจะนำเฉพาะการตรวจสอบผลิตภัณฑ์มาใช้ซ้ำ การตรวจสอบล่วงหน้าของ npm
ไบต์ของแพ็กเกจ ที่มาของบันทึกประจำรุ่น และการยอมรับการติดตั้ง/อัปเดต
ยังคงต้องทำกับ Release SHA การเปลี่ยนแปลงใด ๆ ที่เกี่ยวกับเวอร์ชัน ซอร์ส ไฟล์ที่สร้างขึ้น
การขึ้นต่อกัน แพ็กเกจ หรือเป้าหมายที่เวิร์กโฟลว์เป็นเจ้าของ จำเป็นต้องมี Code SHA ใหม่
และการตรวจสอบแบบเต็มใหม่ การรันชุดรวมที่ใหม่กว่าสำหรับ ref `release/*` เดียวกันและ
กลุ่มการรันซ้ำเดียวกัน จะแทนที่รายการที่กำลังดำเนินอยู่โดยอัตโนมัติ ส่ง
`reuse_evidence=false` เพื่อบังคับให้รันแบบเต็มใหม่

สำหรับการกู้คืนแบบจำกัดขอบเขต ให้ส่ง `rerun_group` ไปยังชุดรวม `all` คือการรันผู้สมัครเผยแพร่จริง `ci` รันเฉพาะเวิร์กโฟลว์ลูก CI ปกติ `plugin-prerelease` รันเฉพาะเวิร์กโฟลว์ลูกของ Plugin ที่ใช้เฉพาะการเผยแพร่ `release-checks` รันทุกกล่องการเผยแพร่ และกลุ่มการเผยแพร่ที่แคบกว่าคือ `install-smoke`, `cross-os`, `live-e2e`, `package`, `qa`, `qa-parity`, `qa-live` และ `npm-telegram` การรันซ้ำแบบเฉพาะจุดด้วย `npm-telegram` ต้องใช้ `release_package_spec` หรือ `npm_telegram_package_spec`; การรันแบบเต็ม/ทั้งหมดใช้ Telegram E2E มาตรฐานของแพ็กเกจภายใน Package Acceptance การรันข้ามระบบปฏิบัติการซ้ำแบบเฉพาะจุดสามารถเพิ่ม `cross_os_suite_filter=windows/packaged-upgrade` หรือตัวกรองระบบปฏิบัติการ/ชุดทดสอบอื่นได้ ความล้มเหลวในการตรวจสอบการเผยแพร่ของ QA จะขัดขวางการตรวจสอบการเผยแพร่ตามปกติ รวมถึงการคลาดเคลื่อนของเครื่องมือแบบไดนามิกของ OpenClaw ที่จำเป็นในระดับมาตรฐาน การรัน Tideclaw alpha ยังคงอาจถือว่าเลนตรวจสอบการเผยแพร่ที่ไม่เกี่ยวกับความปลอดภัยของแพ็กเกจเป็นเพียงคำแนะนำ เมื่อใช้ `release_profile=beta` ชุดทดสอบผู้ให้บริการจริง `Run repo/live E2E validation` จะเป็นเพียงคำแนะนำ (แสดงคำเตือน ไม่ใช่ตัวขัดขวาง) ส่วนโปรไฟล์เสถียรและแบบเต็มยังคงถือเป็นตัวขัดขวาง เมื่อ `live_suite_filter` ขอเลน QA แบบสดที่มีเกตอย่างชัดเจน เช่น Discord, WhatsApp หรือ Slack ต้องเปิดใช้ตัวแปรของรีโพ `OPENCLAW_RELEASE_QA_*_LIVE_CI_ENABLED` ที่ตรงกัน มิฉะนั้นการรับอินพุตจะล้มเหลวแทนที่จะข้ามเลนโดยไม่แจ้ง

### Vitest

กล่อง Vitest คือเวิร์กโฟลว์ลูกแบบสั่งด้วยตนเอง `CI` CI แบบสั่งด้วยตนเองจงใจข้ามการจำกัดขอบเขตตามการเปลี่ยนแปลง และบังคับใช้กราฟการทดสอบปกติสำหรับผู้สมัครเผยแพร่ ได้แก่ ชาร์ด Linux Node, ชาร์ด Plugin ที่รวมมาให้, ชาร์ดสัญญาของ Plugin และช่องทาง, ความเข้ากันได้กับ Node 22, `check-*`, `check-additional-*`, การตรวจสอบเบื้องต้นของอาร์ติแฟกต์ที่สร้างแล้ว, การตรวจสอบเอกสาร, Skills ของ Python, Windows, macOS และ i18n ของ Control UI Android จะรวมอยู่ด้วยเมื่อ `Full Release Validation` รันกล่องนี้ เนื่องจากชุดรวมส่ง `include_android=true`; CI แบบสั่งด้วยตนเองที่ทำงานเดี่ยว ๆ ต้องใช้ `include_android=true` เพื่อให้ครอบคลุม Android

ใช้กล่องนี้เพื่อตอบว่า "ซอร์สทรีผ่านชุดทดสอบปกติทั้งหมดหรือไม่" ซึ่งไม่เหมือนกับการตรวจสอบผลิตภัณฑ์ตามเส้นทางการเผยแพร่ หลักฐานที่ต้องเก็บไว้:

- สรุป `Full Release Validation` ที่แสดง URL การรัน `CI` ซึ่งถูกสั่งให้ทำงาน
- การรัน `CI` ผ่านบน SHA เป้าหมายที่ตรงกันทุกประการ
- ชื่อชาร์ดที่ล้มเหลวหรือทำงานช้าจากงาน CI เมื่อตรวจสอบการถดถอย
- อาร์ติแฟกต์การจับเวลาของ Vitest เช่น `.artifacts/vitest-shard-timings.json` เมื่อการรันต้องวิเคราะห์ประสิทธิภาพ

รัน CI แบบสั่งด้วยตนเองโดยตรงเฉพาะเมื่อการเผยแพร่ต้องการ CI ปกติที่ให้ผลแน่นอน แต่ไม่ต้องการกล่อง Docker, QA Lab, แบบสด, ข้ามระบบปฏิบัติการ หรือแพ็กเกจ ใช้คำสั่งแรกสำหรับ CI โดยตรงที่ไม่รวม Android เพิ่ม `include_android=true` เมื่อ CI ของผู้สมัครเผยแพร่ที่รันโดยตรงต้องครอบคลุม Android:

```bash
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH -f include_android=true
```

### Docker

กล่อง Docker อยู่ใน `OpenClaw Release Checks` ถึง `openclaw-live-and-e2e-checks-reusable.yml` รวมถึงเวิร์กโฟลว์โหมดเผยแพร่ `install-smoke` กล่องนี้ตรวจสอบผู้สมัครเผยแพร่ผ่านสภาพแวดล้อม Docker ที่จัดแพ็กเกจแล้ว แทนที่จะใช้เฉพาะการทดสอบระดับซอร์ส

ความครอบคลุมของ Docker สำหรับการเผยแพร่ประกอบด้วย:

- การตรวจสอบเบื้องต้นของการติดตั้งทั้งหมด โดยเปิดใช้การตรวจสอบเบื้องต้นแบบช้าสำหรับการติดตั้ง Bun แบบส่วนกลาง
- การเตรียม/นำอิมเมจตรวจสอบเบื้องต้นจาก Dockerfile รากมาใช้ซ้ำตาม SHA เป้าหมาย โดยงานตรวจสอบเบื้องต้น QR, ราก/Gateway และตัวติดตั้ง/Bun ทำงานเป็นชาร์ดตรวจสอบการติดตั้งแยกกัน
- เลน E2E ของรีโพ
- ส่วนย่อย Docker ตามเส้นทางการเผยแพร่: `core`, `package-update-openai`, `package-update-anthropic`, `package-update-core`, `plugins-runtime-plugins`, `plugins-runtime-services`, `plugins-runtime-install-a` ถึง `plugins-runtime-install-h` และ `openwebui`
- ความครอบคลุม OpenWebUI บนรันเนอร์ดิสก์ขนาดใหญ่โดยเฉพาะเมื่อมีการร้องขอ
- เลนติดตั้ง/ถอนการติดตั้ง Plugin ที่รวมมาให้แบบแยก `bundled-plugin-install-uninstall-0` ถึง `bundled-plugin-install-uninstall-23`
- ชุดทดสอบผู้ให้บริการแบบสด/E2E และความครอบคลุมโมเดลแบบสดใน Docker เมื่อการตรวจสอบการเผยแพร่รวมชุดทดสอบแบบสด

ใช้อาร์ติแฟกต์ Docker ก่อนรันซ้ำ ตัวจัดกำหนดการตามเส้นทางการเผยแพร่จะอัปโหลด `.artifacts/docker-tests/` พร้อมบันทึกของเลน, `summary.json`, `failures.json`, เวลาของแต่ละเฟส, JSON แผนของตัวจัดกำหนดการ และคำสั่งรันซ้ำ สำหรับการกู้คืนแบบเฉพาะจุด ให้ใช้ `docker_lanes=<lane[,lane]>` กับเวิร์กโฟลว์แบบสด/E2E ที่นำกลับมาใช้ได้ แทนการรันทุกส่วนย่อยของการเผยแพร่ซ้ำ คำสั่งรันซ้ำที่สร้างขึ้นจะรวม `package_artifact_run_id` ก่อนหน้าและอินพุตอิมเมจ Docker ที่เตรียมไว้เมื่อมี เพื่อให้เลนที่ล้มเหลวนำ tarball และอิมเมจ GHCR เดิมมาใช้ซ้ำได้

### QA Lab

กล่อง QA Lab เป็นส่วนหนึ่งของ `OpenClaw Release Checks` เช่นกัน กล่องนี้เป็นเกตการเผยแพร่สำหรับพฤติกรรมแบบเอเจนต์และระดับช่องทาง ซึ่งแยกจาก Vitest และกลไกแพ็กเกจ Docker

ความครอบคลุมของ QA Lab สำหรับการเผยแพร่ประกอบด้วย:

- เลนความเท่าเทียมแบบจำลองที่เปรียบเทียบเลนผู้สมัคร OpenAI กับเกณฑ์ฐาน `anthropic/claude-opus-4-8` โดยใช้ชุดความเท่าเทียมแบบเอเจนต์
- โปรไฟล์การเผยแพร่อะแดปเตอร์แบบสดของ Matrix ที่ใช้สภาพแวดล้อม `qa-live-shared`
- เลน QA แบบสดของ Telegram ที่ใช้สัญญาเช่าข้อมูลประจำตัว Convex CI
- `pnpm qa:otel:smoke`, `pnpm qa:otel:collector-smoke`, `pnpm qa:prometheus:smoke` หรือ `pnpm qa:observability:smoke` เมื่อข้อมูลวัดและส่งทางไกลของการเผยแพร่ต้องการหลักฐานภายในเครื่องอย่างชัดเจน

ใช้กล่องนี้เพื่อตอบว่า "การเผยแพร่ทำงานอย่างถูกต้องในสถานการณ์ QA และโฟลว์ช่องทางแบบสดหรือไม่" เก็บ URL อาร์ติแฟกต์ของเลนความเท่าเทียม Matrix และ Telegram ไว้เมื่ออนุมัติการเผยแพร่ ความครอบคลุม Matrix แบบเต็มยังคงใช้ได้ในรูปแบบการรัน QA Lab แบบแบ่งชาร์ดด้วยตนเอง แทนที่จะเป็นเลนสำคัญต่อการเผยแพร่โดยปริยาย

### แพ็กเกจ

กล่องแพ็กเกจคือเกตของผลิตภัณฑ์ที่ติดตั้งได้ รองรับโดย `Package Acceptance` และตัวแก้ไข `scripts/resolve-openclaw-package-candidate.mjs` ตัวแก้ไขจะทำให้ผู้สมัครอยู่ในรูปแบบ tarball `package-under-test` มาตรฐานที่ Docker E2E ใช้ ตรวจสอบรายการภายในแพ็กเกจ บันทึกเวอร์ชันและ SHA-256 ของแพ็กเกจ และแยก ref ของชุดควบคุมเวิร์กโฟลว์ออกจาก ref ซอร์สของแพ็กเกจ

ซอร์สผู้สมัครที่รองรับ:

- `source=npm`: `openclaw@beta`, `openclaw@latest` หรือเวอร์ชันเผยแพร่ OpenClaw ที่ระบุแน่นอน
- `source=ref`: แพ็ก branch, tag หรือ SHA คอมมิตแบบเต็ม `package_ref` ที่เชื่อถือได้ด้วยชุดควบคุม `workflow_ref` ที่เลือก
- `source=url`: ดาวน์โหลด `.tgz` ผ่าน HTTPS สาธารณะโดยต้องมี `package_sha256`; ข้อมูลประจำตัวใน URL, พอร์ต HTTPS ที่ไม่ใช่ค่าเริ่มต้น, ชื่อโฮสต์หรือที่อยู่ที่แก้ไขแล้วซึ่งเป็นแบบส่วนตัว/ภายใน/ใช้งานเฉพาะ และการเปลี่ยนเส้นทางที่ไม่ปลอดภัยจะถูกปฏิเสธ
- `source=trusted-url`: ดาวน์โหลด `.tgz` ผ่าน HTTPS โดยต้องมี `package_sha256` และ `trusted_source_id` จากนโยบายที่ระบุชื่อใน `.github/package-trusted-sources.json`; ใช้ตัวเลือกนี้สำหรับมิเรอร์ระดับองค์กรหรือรีโพแพ็กเกจส่วนตัวที่ผู้ดูแลเป็นเจ้าของ แทนการเพิ่มทางเลี่ยงเครือข่ายส่วนตัวระดับอินพุตให้กับ `source=url`
- `source=artifact`: นำ `.tgz` ที่อัปโหลดโดยการรัน GitHub Actions อื่นมาใช้ซ้ำ

`OpenClaw Release Checks` รัน Package Acceptance ด้วย `source=artifact`, อาร์ติแฟกต์แพ็กเกจการเผยแพร่ที่เตรียมไว้, `suite_profile=custom`, `docker_lanes=doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor root-managed-vps-upgrade update-restart-auth plugins-offline plugin-update plugin-binding-command-escape`, `telegram_mode=mock-openai` Package Acceptance จะคงการย้ายข้อมูล การอัปเดต การอัปเกรด VPS ที่จัดการโดยราก การรีสตาร์ตหลังอัปเดตเมื่อกำหนดค่าการตรวจสอบสิทธิ์แล้ว การติดตั้ง Skills แบบสดจาก ClawHub การล้างการขึ้นต่อกันของ Plugin ที่ล้าสมัย ฟิกซ์เจอร์ Plugin แบบออฟไลน์ การอัปเดต Plugin การเสริมความแข็งแกร่งต่อการหลบอักขระของการผูกคำสั่ง Plugin และ QA ของแพ็กเกจ Telegram โดยใช้ tarball ที่แก้ไขแล้วชุดเดียวกัน การตรวจสอบการเผยแพร่ที่เป็นตัวขัดขวางจะใช้เกณฑ์ฐานแพ็กเกจเผยแพร่ล่าสุดตามค่าเริ่มต้น ส่วนโปรไฟล์เบตาที่มี `run_release_soak=true`, `release_profile=stable` หรือ `release_profile=full` จะขยายการกวาดตรวจผู้รอดจากการอัปเกรดแพ็กเกจที่เผยแพร่แล้วไปยัง `last-stable-4` รวมถึงเกณฑ์ฐานที่ตรึงไว้ `2026.4.23`, `2026.5.2` และ `2026.4.15` พร้อมสถานการณ์ `reported-issues` ใช้ Package Acceptance กับ `source=npm` สำหรับผู้สมัครที่เผยแพร่แล้ว, `source=ref` สำหรับ tarball npm ภายในเครื่องที่อ้างอิง SHA ก่อนเผยแพร่, `source=trusted-url` สำหรับมิเรอร์ระดับองค์กร/ส่วนตัวที่ผู้ดูแลเป็นเจ้าของ หรือ `source=artifact` สำหรับ tarball ที่เตรียมไว้และอัปโหลดโดยการรัน GitHub Actions อื่น

กล่องนี้เป็นสิ่งทดแทนที่ทำงานบน GitHub โดยตรงสำหรับความครอบคลุมด้านแพ็กเกจ/การอัปเดตส่วนใหญ่ที่ก่อนหน้านี้ต้องใช้ Parallels การตรวจสอบการเผยแพร่ข้ามระบบปฏิบัติการยังคงสำคัญสำหรับการเริ่มต้นใช้งาน ตัวติดตั้ง และพฤติกรรมเฉพาะแพลตฟอร์ม แต่การตรวจสอบผลิตภัณฑ์ด้านแพ็กเกจ/การอัปเดตควรเลือกใช้ Package Acceptance

รายการตรวจสอบมาตรฐานสำหรับการอัปเดตและการตรวจสอบ Plugin คือ [การทดสอบการอัปเดตและ Plugin](/th/help/testing-updates-plugins) ใช้รายการนี้เมื่อตัดสินใจว่าเลนภายในเครื่อง, Docker, Package Acceptance หรือการตรวจสอบการเผยแพร่ใดพิสูจน์การติดตั้ง/อัปเดต Plugin, การล้างข้อมูลด้วย doctor หรือการเปลี่ยนแปลงการย้ายข้อมูลของแพ็กเกจที่เผยแพร่แล้ว การย้ายข้อมูลการอัปเดตที่เผยแพร่แล้วอย่างครอบคลุมจากแพ็กเกจเสถียร `2026.4.23+` ทุกแพ็กเกจ เป็นเวิร์กโฟลว์แบบสั่งด้วยตนเอง `Update Migration` แยกต่างหาก ไม่ใช่ส่วนหนึ่งของ Full Release CI

การผ่อนปรน Package Acceptance แบบเดิมถูกจำกัดเวลาโดยเจตนา แพ็กเกจจนถึง `2026.4.25` อาจใช้เส้นทางความเข้ากันได้สำหรับช่องว่างของข้อมูลเมตาที่เผยแพร่ไปยัง npm แล้ว ได้แก่ รายการสินค้าคงคลัง QA ส่วนตัวที่ไม่มีใน tarball, ไม่มี `gateway install --wrapper`, ไม่มีไฟล์แพตช์ในฟิกซ์เจอร์ git ที่ได้จาก tarball, ไม่มี `update.channel` ที่คงอยู่, ตำแหน่งบันทึกการติดตั้ง Plugin แบบเดิม, ไม่มีการคงอยู่ของบันทึกการติดตั้งจากมาร์เก็ตเพลส และการย้ายข้อมูลเมตาการกำหนดค่าระหว่าง `plugins update` แพ็กเกจ `2026.4.26` ที่เผยแพร่แล้วอาจเตือนเกี่ยวกับไฟล์ประทับข้อมูลเมตาของบิลด์ภายในเครื่องที่ส่งมอบไปแล้ว แพ็กเกจหลังจากนั้นต้องเป็นไปตามสัญญาแพ็กเกจสมัยใหม่ ช่องว่างเดียวกันเหล่านั้นจะทำให้การตรวจสอบการเผยแพร่ล้มเหลว

ใช้โปรไฟล์ Package Acceptance ที่กว้างขึ้นเมื่อคำถามเกี่ยวกับการเผยแพร่เป็นเรื่องของแพ็กเกจที่ติดตั้งได้จริง:

```bash
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f published_upgrade_survivor_baseline=openclaw@2026.4.26
```

โปรไฟล์แพ็กเกจที่ใช้ทั่วไป:

- `smoke`: เลนติดตั้งแพ็กเกจ/ช่องทาง/เอเจนต์แบบรวดเร็ว เครือข่าย Gateway และการโหลดการกำหนดค่าใหม่
- `package`: สัญญาการติดตั้ง/อัปเดต/รีสตาร์ต/แพ็กเกจ Plugin พร้อมหลักฐานการติดตั้ง Skills จาก ClawHub แบบสด ซึ่งเป็นค่าเริ่มต้นของการตรวจสอบรีลีส
- `product`: `package` พร้อมช่องทาง MCP, การล้าง cron/เอเจนต์ย่อย, การค้นหาเว็บด้วย OpenAI และ OpenWebUI
- `full`: ส่วนย่อยของเส้นทางรีลีส Docker พร้อม OpenWebUI
- `custom`: รายการ `docker_lanes` ที่ตรงกันทุกประการสำหรับการรันซ้ำแบบเจาะจง

สำหรับหลักฐาน Telegram ของแพ็กเกจตัวเลือก ให้เปิดใช้ `telegram_mode=mock-openai` หรือ `telegram_mode=live-frontier` ใน Package Acceptance เวิร์กโฟลว์จะส่ง tarball ของ `package-under-test` ที่แก้ไขแล้วไปยังเลน Telegram ส่วนเวิร์กโฟลว์ Telegram แบบสแตนด์อโลนยังคงยอมรับข้อกำหนด npm ที่เผยแพร่แล้วสำหรับการตรวจสอบหลังเผยแพร่

## ระบบอัตโนมัติสำหรับเผยแพร่รีลีสตามปกติ

สำหรับการเผยแพร่เบตา, `latest`, Plugin, GitHub Release และแพลตฟอร์ม
`OpenClaw Release Publish` คือจุดเริ่มต้นมาตรฐานที่มีการเปลี่ยนแปลงสถานะ เส้นทาง extended-stable แบบ npm เท่านั้นรายเดือน
`.33+` ไม่ใช้ตัวควบคุมการทำงานนี้
เวิร์กโฟลว์ปกติจะควบคุมเวิร์กโฟลว์ trusted-publisher ตามลำดับที่
รีลีสต้องใช้:

1. เช็กเอาต์แท็กรีลีสและแก้ไข SHA ของคอมมิต
2. ตรวจสอบว่าเข้าถึงแท็กได้จาก `main` หรือ `release/*` (หรือจากสาขาอัลฟา Tideclaw สำหรับอัลฟาพรีรีลีส)
3. รัน `pnpm plugins:sync:check`
4. ส่ง `Plugin NPM Release` โดยใช้ `publish_scope=all-publishable` และ `ref=<release-sha>`
5. ส่ง `Plugin ClawHub Release` โดยใช้ขอบเขตและ SHA เดียวกัน
6. ส่ง `OpenClaw NPM Release` โดยใช้แท็กรีลีส, npm dist-tag และ `preflight_run_id` ที่บันทึกไว้ หลังจากตรวจสอบ `full_release_validation_run_id` ที่บันทึกไว้และครั้งที่รันที่ตรงกันทุกประการ
7. สำหรับรีลีสเสถียร ให้สร้างหรืออัปเดต GitHub release เป็นฉบับร่าง ส่ง `Windows Node Release` โดยใช้ `windows_node_tag` ที่ระบุอย่างชัดเจนและ `windows_node_installer_digests` ที่ตัวเลือกได้รับอนุมัติ แล้วตรวจสอบแอสเซ็ตตัวติดตั้ง/เช็กซัม Windows มาตรฐาน นอกจากนี้ ให้ส่ง `Android Release` เพื่อสร้าง APK ที่ลงนามจากแท็กที่ตรงกันทุกประการ พร้อมเช็กซัมและที่มา ตรวจสอบสัญญาแอสเซ็ตเนทีฟทั้งสองก่อนเผยแพร่ฉบับร่าง

ตัวอย่างการเผยแพร่เบตา:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH-beta.N \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

การเผยแพร่เวอร์ชันเสถียรไปยัง beta dist-tag เริ่มต้น:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

การเลื่อนขั้นเวอร์ชันเสถียรไปยัง `latest` โดยตรงต้องระบุอย่างชัดเจน:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=latest
```

ใช้เวิร์กโฟลว์ระดับล่าง `Plugin NPM Release` และ `Plugin ClawHub Release` สำหรับงานซ่อมแซมหรือเผยแพร่ซ้ำแบบเจาะจงเท่านั้น `OpenClaw Release Publish` จะปฏิเสธ `plugin_publish_scope=selected` เมื่อ `publish_openclaw_npm=true` เพื่อไม่ให้แพ็กเกจแกนหลักเผยแพร่โดยไม่มี Plugin ทางการที่เผยแพร่ได้ครบทุกตัว รวมถึง `@openclaw/diffs-language-pack` สำหรับการซ่อมแซม Plugin ที่เลือกไว้ ให้ตั้งค่า `publish_openclaw_npm=false` พร้อม `plugin_publish_scope=selected` และ `plugins=@openclaw/name` หรือส่งเวิร์กโฟลว์ลูกโดยตรง

การบูตสแตรป ClawHub สำหรับการเผยแพร่ครั้งแรกเป็นข้อยกเว้น: ให้ส่ง `Plugin ClawHub New`
จาก `main` ที่เชื่อถือได้ และส่ง SHA ของรีลีสเป้าหมายฉบับเต็มผ่าน `ref`
ห้ามรันเวิร์กโฟลว์บูตสแตรปจากแท็กหรือสาขารีลีสโดยตรง:

```bash
gh workflow run plugin-clawhub-new.yml \
  --ref main \
  -f plugins=@openclaw/name \
  -f ref=<full-40-character-release-sha> \
  -f pretag_validation=true \
  -f dry_run=true
```

การตรวจสอบก่อนติดแท็กต้องใช้ `dry_run=true` ปฏิเสธอินพุตแท็กรีลีสและการรันหลัก
และยอมรับเฉพาะเป้าหมายที่ตรงกันทุกประการซึ่งเข้าถึงได้จาก `main` หรือ `release/*`
การดำเนินการนี้จะไม่โหลดข้อมูลประจำตัว ClawHub, เผยแพร่ไบต์แพ็กเกจ หรือเปลี่ยนการกำหนดค่า
trusted publisher เวิร์กโฟลว์ยังคงแก้ไขแผนรีจิสทรีแบบสด
เช็กเอาต์และแพ็กเป้าหมายเฉพาะในงานที่ไม่มีข้อมูลลับ จัดเตรียม
ชุดเครื่องมือ ClawHub ที่ล็อกไว้ และตรวจสอบอาร์ติแฟกต์ที่เปลี่ยนแปลงไม่ได้ รวมถึง
slug/ข้อมูลประจำตัวของแพ็กเกจ ก่อนที่แท็กรีลีสจะมีอยู่ อนุมัติสภาพแวดล้อม
`clawhub-plugin-bootstrap` หลังจากงานแพ็กที่ไม่มีข้อมูลลับ
เสร็จสิ้นเท่านั้น งานตรวจสอบที่ได้รับการป้องกันนี้ไม่มีข้อมูลประจำตัวหรือคำสั่งที่ทำให้เกิดการเปลี่ยนแปลง

การทดลองรันที่อนุมัติแล้วหรือการบูตสแตรปจริงหลังติดแท็กต้องระบุ
แท็กรีลีสที่ตรงกันทุกประการ พร้อมรหัสการรัน ครั้งที่รัน และ
สาขาของ `OpenClaw Release Publish` หลัก ตัวหลักรับรอง SHA ของเวิร์กโฟลว์ตนเองและ SHA ของ `main` ที่เชื่อถือได้ซึ่งแยกต่างหากและตรงกันทุกประการสำหรับ `Plugin ClawHub New` การรันลูกและการอนุมัติ
สภาพแวดล้อมที่ได้รับการป้องกันทุกรายการต้องตรงกับ SHA ของเวิร์กโฟลว์ลูกที่อนุมัติ แท็กรีลีสจะถูก
ตรวจสอบซ้ำก่อนทุกความพยายามเผยแพร่และการเปลี่ยนแปลง trusted-publisher

งานแพ็ก
อัปโหลดอาร์ติแฟกต์ที่เปลี่ยนแปลงไม่ได้หนึ่งรายการ โดยชื่อ, Actions artifact ID/digest,
การรัน/ครั้งที่รันของผู้สร้าง, SHA เป้าหมาย และ SHA-256/ขนาดของ tarball ต่อแพ็กเกจ
จะถูกส่งต่อไปยังงานตรวจสอบและงานที่ได้รับการป้องกัน งานที่ได้รับการป้องกันจะเช็กเอาต์เฉพาะเครื่องมือ `main`
ที่เชื่อถือได้ ตรวจสอบทูเพิลของอาร์ติแฟกต์ผ่าน GitHub API ดาวน์โหลด
ด้วย artifact ID ที่ตรงกันทุกประการ แฮช tarball ทุกรายการใหม่ และตรวจสอบพาธ TAR ภายในเครื่องกับ
ข้อมูลประจำตัวของแพ็กเกจตามกฎ USTAR canonicalization ของ CLI ที่ปักหมุดไว้ จากนั้น
ตัวเลือกทุกรายการต้องผ่านการทดลองเผยแพร่ด้วย CLI ที่ปักหมุดไว้ ซึ่งจะคืนค่าก่อน
ค้นหารีจิสทรีหรือยืนยันตัวตน ตัวกรองล่วงหน้าของงานข้อมูลประจำตัวจำกัด ClawPack ที่บีบอัดไว้
ที่ 120 MiB, เพย์โหลดไฟล์รวมที่ 50 MiB, ข้อมูล TAR ที่ขยายแล้วที่ 64 MiB และ
จำนวนรายการ TAR ที่ 10,000 การซ่อมแซม trusted-publisher สำหรับแพ็กเกจที่มีอยู่
ยังคงเป็นแบบกำหนดค่าเท่านั้น แต่จะยังคงแพ็กเป้าหมายและต้องให้แท็กที่ร้องขอ
รวมถึงไบต์และข้อมูลเมตาของรีจิสทรีตรงกันทุกประการก่อนเปลี่ยนการกำหนดค่า trusted-publisher
การตรวจสอบหลังเผยแพร่จะดาวน์โหลดอาร์ติแฟกต์ ClawHub และ
กำหนดให้ SHA-256 และขนาดตรงกัน การกู้คืนด้วยการรันซ้ำเฉพาะงานที่ล้มเหลวอาจใช้อาร์ติแฟกต์แพ็กเกจจาก
ครั้งที่รันก่อนหน้าได้ เฉพาะเมื่องานผู้สร้างที่ตรงกันทุกประการเสร็จสมบูรณ์
สำเร็จ หลักฐานสุดท้ายยังผูกเวอร์ชัน ClawHub ที่ล็อกไว้, SHA-256 ของล็อก
และความสมบูรณ์ของ npm หากไม่ตรงกัน ต้องใช้แพ็กเกจเวอร์ชันใหม่

## อินพุตเวิร์กโฟลว์ NPM

`OpenClaw NPM Release` ยอมรับอินพุตที่ผู้ดำเนินการควบคุมต่อไปนี้:

- `tag`: แท็กรีลีสที่จำเป็น เช่น `v2026.4.2`, `v2026.4.2-1`, `v2026.4.2-beta.1` หรือ `v2026.4.2-alpha.1`; เมื่อ `preflight_only=true` ค่านี้อาจเป็น SHA ของคอมมิตปัจจุบันจากสาขาเวิร์กโฟลว์แบบเต็ม 40 อักขระสำหรับการตรวจสอบก่อนดำเนินการเท่านั้น
- `preflight_only`: `true` สำหรับการตรวจสอบ/บิลด์/แพ็กเกจเท่านั้น, `false` สำหรับเส้นทางเผยแพร่จริง
- `preflight_run_id`: รหัสการรันก่อนดำเนินการที่สำเร็จและมีอยู่แล้ว ซึ่งจำเป็นสำหรับเส้นทางเผยแพร่จริง เพื่อให้เวิร์กโฟลว์นำ tarball ที่เตรียมไว้กลับมาใช้แทนการสร้างใหม่
- `full_release_validation_run_id`: รหัสการรัน `Full Release Validation` ที่สำเร็จสำหรับแท็ก/SHA นี้ ซึ่งจำเป็นสำหรับการเผยแพร่จริง การเผยแพร่เบตาอาจดำเนินต่อโดยใช้เพียงการตรวจสอบก่อนดำเนินการพร้อมคำเตือน แต่การเลื่อนขั้นเวอร์ชันเสถียร/`latest` ยังคงต้องใช้ค่านี้
- `full_release_validation_run_attempt`: ครั้งที่รันที่เป็นจำนวนบวกและตรงกันทุกประการ ซึ่งจับคู่กับ `full_release_validation_run_id`; จำเป็นทุกครั้งที่ระบุรหัสการรัน เพื่อไม่ให้การรันซ้ำเปลี่ยนหลักฐานการอนุญาตระหว่างการเผยแพร่
- `release_publish_run_id`: รหัสการรัน `OpenClaw Release Publish` ที่อนุมัติแล้ว; จำเป็นเมื่อเวิร์กโฟลว์นี้ถูกส่งโดยตัวหลักนั้น (การเรียกเผยแพร่จริงโดยตัวดำเนินการบอต)
- `plugin_npm_run_id`: รหัสการรัน `Plugin NPM Release` ที่สำเร็จและตรงกับส่วนหัวทุกประการ; จำเป็นสำหรับการเผยแพร่แกนหลัก `extended-stable` จริง
- `npm_dist_tag`: แท็กเป้าหมาย npm สำหรับเส้นทางเผยแพร่; ยอมรับ `alpha`, `beta`, `latest` หรือ `extended-stable` และมีค่าเริ่มต้นเป็น `beta` แพตช์สุดท้าย `33` และใหม่กว่าต้องใช้ `extended-stable`; โดยค่าเริ่มต้น `extended-stable` จะปฏิเสธแพตช์ก่อนหน้า และจะปฏิเสธแท็กที่ไม่ใช่แท็กสุดท้ายเสมอ
- `bypass_extended_stable_guard`: บูลีนสำหรับการทดสอบเท่านั้น ค่าเริ่มต้นคือ `false`; เมื่อใช้กับ `npm_dist_tag=extended-stable` จะข้ามคุณสมบัติการมีสิทธิ์ extended-stable รายเดือน โดยยังคงการตรวจสอบข้อมูลประจำตัวของรีลีส, อาร์ติแฟกต์, การอนุมัติ และการอ่านกลับ

`Plugin NPM Release` ยอมรับ `npm_dist_tag=default` สำหรับลักษณะการทำงานของรีลีสที่มีอยู่
หรือ `npm_dist_tag=extended-stable` สำหรับเส้นทางรายเดือนที่มีการป้องกัน ตัวเลือก
extended-stable ต้องใช้ `publish_scope=all-publishable`, อินพุต
`plugins` ที่ว่างเปล่า, แพตช์สุดท้ายที่มีค่าตั้งแต่ `33` ขึ้นไป และสาขามาตรฐาน
`extended-stable/YYYY.M.33` ที่ส่วนปลายตรงกันทุกประการ ตัวเลือกนี้จะไม่ย้าย
`latest` หรือ `beta` ของ Plugin แพ็กเกจเวอร์ชันใหม่จะได้รับ `extended-stable` แบบอะตอม
ผ่านการเผยแพร่ที่เชื่อถือได้ด้วย OIDC (`npm publish --tag extended-stable`);
เวิร์กโฟลว์ต้นทางนี้ไม่ใช้ `npm dist-tag add` ที่ยืนยันตัวตนด้วยโทเค็น การลองใหม่
จะข้ามเวอร์ชันที่ตรงกันทุกประการซึ่งมีอยู่ใน npm แล้ว จากนั้นหยุดแบบปิดเว้นแต่การอ่านกลับอย่างครบถ้วน
จะยืนยันว่าแพ็กเกจทุกตัวที่ตรงกันทุกประการและแท็ก `extended-stable` มาบรรจบกันแล้ว

`OpenClaw Release Publish` ยอมรับอินพุตที่ผู้ดำเนินการควบคุมต่อไปนี้:

- `tag`: แท็กรีลีสที่จำเป็น; ต้องมีอยู่แล้ว
- `preflight_run_id`: รหัสการรันก่อนดำเนินการ `OpenClaw NPM Release` ที่สำเร็จ; จำเป็นเมื่อ `publish_openclaw_npm=true` หรือ `plugin_publish_scope=all-publishable`
- `full_release_validation_run_id`: รหัสการรัน `Full Release Validation` ที่สำเร็จ; จำเป็นเมื่อ `publish_openclaw_npm=true` หรือ `plugin_publish_scope=all-publishable`
- `full_release_validation_run_attempt`: ครั้งที่รันที่เป็นจำนวนบวกและตรงกันทุกประการ ซึ่งจับคู่กับ `full_release_validation_run_id`; จำเป็นทุกครั้งที่ระบุรหัสการรัน
- `windows_node_tag`: แท็กรีลีส `openclaw/openclaw-windows-node` ที่ไม่ใช่พรีรีลีสและตรงกันทุกประการ; จำเป็นสำหรับการเผยแพร่ OpenClaw เวอร์ชันเสถียร
- `windows_node_installer_digests`: แมป JSON แบบกะทัดรัดที่ตัวเลือกได้รับอนุมัติ ซึ่งแมปชื่อตัวติดตั้ง Windows ปัจจุบันไปยังไดเจสต์ `sha256:` ที่ปักหมุดไว้; จำเป็นสำหรับการเผยแพร่ OpenClaw เวอร์ชันเสถียร
- `npm_telegram_run_id`: รหัสการรัน `NPM Telegram Beta E2E` ที่สำเร็จซึ่งระบุหรือไม่ก็ได้ เพื่อรวมไว้ในหลักฐานรีลีสสุดท้าย
- `npm_dist_tag`: แท็กเป้าหมาย npm สำหรับแพ็กเกจ OpenClaw ซึ่งเป็นหนึ่งใน `alpha`, `beta` หรือ `latest`
- `plugin_publish_scope`: ค่าเริ่มต้นคือ `all-publishable`; ใช้ `selected` เฉพาะสำหรับงานซ่อมแซมที่เจาะจงเฉพาะ Plugin ด้วย `publish_openclaw_npm=false`
- `plugins`: ชื่อแพ็กเกจ `@openclaw/*` ที่คั่นด้วยจุลภาค เมื่อ `plugin_publish_scope=selected`
- `publish_openclaw_npm`: ค่าเริ่มต้นคือ `true`; ตั้งค่า `false` เฉพาะเมื่อใช้เวิร์กโฟลว์เป็นตัวควบคุมการซ่อมแซมเฉพาะ Plugin
- `release_profile`: โปรไฟล์ความครอบคลุมของรีลีสที่ใช้สำหรับสรุปหลักฐานรีลีส; ค่าเริ่มต้นคือ `from-validation` ซึ่งอ่านค่าจากแมนิเฟสต์การตรวจสอบ หรือแทนที่ด้วย `beta`, `stable` หรือ `full`
- `wait_for_clawhub`: ค่าเริ่มต้นคือ `false` เพื่อไม่ให้ความพร้อมใช้งานของ npm ถูกขัดขวางโดยไซด์คาร์ ClawHub; ตั้งค่า `true` เฉพาะเมื่อการเสร็จสิ้นของเวิร์กโฟลว์ต้องรวมการเสร็จสิ้นของ ClawHub

`OpenClaw Release Checks` ยอมรับอินพุตที่ผู้ดำเนินการควบคุมต่อไปนี้:

- `ref`: แบรนช์ แท็ก หรือ SHA ของคอมมิตแบบเต็มที่จะตรวจสอบ การตรวจสอบที่ใช้ข้อมูลลับกำหนดให้คอมมิตที่แก้ไขแล้วต้องเข้าถึงได้จากแบรนช์หรือแท็กรีลีสของ OpenClaw
- `run_release_soak`: เลือกใช้การทดสอบ live/E2E แบบครบถ้วน เส้นทางรีลีส Docker และการทดสอบ soak ของ upgrade-survivor สำหรับทุกเวอร์ชันที่ผ่านมาในการตรวจสอบรีลีสเบตา ตัวเลือกนี้จะถูกบังคับเปิดโดย `release_profile=stable` และ `release_profile=full`

กฎ:

- เวอร์ชันสุดท้ายและเวอร์ชันแก้ไขปกติที่ต่ำกว่าแพตช์ `33` อาจเผยแพร่ไปยัง `beta` หรือ `latest` ก็ได้ เวอร์ชันสุดท้ายที่แพตช์ `33` ขึ้นไปต้องเผยแพร่ไปยัง `extended-stable` และเวอร์ชันที่มีส่วนต่อท้ายการแก้ไข ณ ขอบเขตดังกล่าวจะถูกปฏิเสธ
- แท็กก่อนรีลีสแบบเบตาเผยแพร่ได้เฉพาะไปยัง `beta`; แท็กก่อนรีลีสแบบแอลฟาเผยแพร่ได้เฉพาะไปยัง `alpha`
- สำหรับ `OpenClaw NPM Release` อนุญาตให้อินพุตเป็น SHA ของคอมมิตแบบเต็มได้เฉพาะเมื่อ `preflight_only=true`
- `OpenClaw Release Checks` และ `Full Release Validation` ใช้เพื่อตรวจสอบเท่านั้นเสมอ
- เส้นทางเผยแพร่จริงต้องใช้ `npm_dist_tag` เดียวกับที่ใช้ระหว่างการตรวจสอบเบื้องต้น เวิร์กโฟลว์จะตรวจสอบเมทาดาทาดังกล่าวก่อนดำเนินการเผยแพร่ต่อ

## ลำดับการรีลีสเบตาปกติ/เสถียรล่าสุด

ลำดับแบบเดิมนี้ใช้สำหรับรีลีสที่จัดการแบบประสานงานตามปกติ ซึ่งยังครอบคลุม Plugin, GitHub Release, Windows และงานบนแพลตฟอร์มอื่นด้วย ไม่ใช่เส้นทาง extended-stable รายเดือนแบบ npm เท่านั้นของ `.33+` ที่อธิบายไว้ด้านบนของหน้านี้

เมื่อจัดทำรีลีสเสถียรที่จัดการแบบประสานงานตามปกติ:

1. เรียกใช้ `OpenClaw NPM Release` ด้วย `preflight_only=true` ก่อนจะมีแท็ก คุณอาจใช้ SHA ของคอมมิตแบบเต็มปัจจุบันจากแบรนช์เวิร์กโฟลว์เพื่อทดลองรันเวิร์กโฟลว์ตรวจสอบเบื้องต้นแบบตรวจสอบเท่านั้น
2. เลือก `npm_dist_tag=beta` สำหรับโฟลว์ปกติที่เริ่มจากเบตา หรือเลือก `latest` เฉพาะเมื่อตั้งใจเผยแพร่เวอร์ชันเสถียรโดยตรง
3. เรียกใช้ `Full Release Validation` บนแบรนช์รีลีส แท็กรีลีส หรือ SHA ของคอมมิตแบบเต็ม เมื่อต้องการ CI ปกติพร้อมการครอบคลุมแคชพรอมป์แบบ live, Docker, QA Lab, Matrix และ Telegram จากเวิร์กโฟลว์แบบแมนนวลเดียว หากตั้งใจต้องการเฉพาะกราฟการทดสอบปกติที่ให้ผลแน่นอน ให้เรียกใช้เวิร์กโฟลว์ `CI` แบบแมนนวลบนรีลีส ref แทน
4. เลือกแท็กรีลีส `openclaw/openclaw-windows-node` ที่ไม่ใช่ก่อนรีลีสให้ตรงกับตัวติดตั้ง x64 และ ARM64 ที่ลงนามแล้วซึ่งต้องการจัดส่ง บันทึกเป็น `windows_node_tag` และบันทึกแมปไดเจสต์ที่ผ่านการตรวจสอบเป็น `windows_node_installer_digests` ตัวช่วย release-candidate จะบันทึกทั้งสองค่าและรวมไว้ในคำสั่งเผยแพร่ที่สร้างขึ้น
5. บันทึก `preflight_run_id`, `full_release_validation_run_id` ที่สำเร็จ และ `full_release_validation_run_attempt` ที่ตรงกัน
6. เรียกใช้ `OpenClaw Release Publish` จาก `main` ที่เชื่อถือได้ โดยใช้ `tag` เดิม, `npm_dist_tag` เดิม, `windows_node_tag` ที่เลือก, `windows_node_installer_digests` ที่บันทึกไว้ของรายการนั้น, `preflight_run_id` ที่บันทึกไว้, `full_release_validation_run_id` และ `full_release_validation_run_attempt` ระบบจะเผยแพร่ Plugin ที่แยกออกไปยัง npm และ ClawHub ก่อนโปรโมตแพ็กเกจ npm ของ OpenClaw
7. หากรีลีสเผยแพร่ลงบน `beta` ให้ใช้เวิร์กโฟลว์ `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml` เพื่อโปรโมตเวอร์ชันเสถียรนั้นจาก `beta` ไปยัง `latest`
8. หากรีลีสตั้งใจเผยแพร่โดยตรงไปยัง `latest` และ `beta` ควรตามบิลด์เสถียรเดียวกันทันที ให้ใช้เวิร์กโฟลว์รีลีสเดียวกันนั้นเพื่อชี้ dist-tag ทั้งสองไปยังเวอร์ชันเสถียร หรือปล่อยให้การซิงค์แก้ไขตนเองตามกำหนดเวลาย้าย `beta` ในภายหลัง

การเปลี่ยนแปลง dist-tag อยู่ในรีโพซิทอรีบัญชีแยกประเภทรีลีส เนื่องจากยังคงต้องใช้ `NPM_TOKEN` ขณะที่รีโพซิทอรีซอร์สยังคงเผยแพร่ผ่าน OIDC เท่านั้น วิธีนี้ทำให้ทั้งเส้นทางเผยแพร่โดยตรงและเส้นทางโปรโมตที่เริ่มจากเบตาได้รับการบันทึกไว้และผู้ปฏิบัติงานมองเห็นได้

หากผู้ดูแลจำเป็นต้องย้อนกลับไปใช้การยืนยันตัวตน npm ในเครื่อง ให้เรียกใช้คำสั่ง 1Password CLI (`op`) เฉพาะภายในเซสชัน tmux ที่จัดไว้โดยเฉพาะเท่านั้น ห้ามเรียก `op` โดยตรงจากเชลล์หลักของเอเจนต์ การเก็บไว้ภายใน tmux ทำให้พรอมป์ การแจ้งเตือน และการจัดการ OTP สังเกตการณ์ได้ และป้องกันการแจ้งเตือนจากโฮสต์ซ้ำ
