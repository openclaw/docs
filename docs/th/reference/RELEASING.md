OpenClaw มีช่องทางการอัปเดตที่ผู้ใช้มองเห็นอยู่สามช่องทางในปัจจุบัน:

- stable: ช่องทางรุ่นเผยแพร่ที่โปรโมตอยู่เดิม ซึ่งยังคงชี้ผ่าน npm `latest` จนกว่าหมุดหมาย CLI/ช่องทางแยกจะเสร็จสมบูรณ์
- beta: แท็กก่อนเผยแพร่ที่เผยแพร่ไปยัง npm `beta`
- dev: ปลายล่าสุดที่เปลี่ยนแปลงต่อเนื่องของ `main`

นอกจากนี้ ผู้ดำเนินการเผยแพร่สามารถเผยแพร่แพ็กเกจแกนหลักของเดือนก่อนหน้าที่เสร็จสมบูรณ์
ไปยัง npm `extended-stable` โดยเริ่มที่แพตช์ `33` ส่วนสายรุ่นสุดท้ายปกติ
ของเดือนปัจจุบันยังคงอยู่บน npm `latest`; การแยกการเผยแพร่ฝั่งผู้ดำเนินการนี้
ไม่ได้เปลี่ยนการกำหนดช่องทางอัปเดตของ CLI ด้วยตัวมันเอง

บิลด์อัลฟาของ Tideclaw เป็นสายก่อนเผยแพร่ภายในที่แยกต่างหาก (npm dist-tag `alpha`) ซึ่งอธิบายไว้ใน [อินพุตเวิร์กโฟลว์ NPM](#npm-workflow-inputs) และ [กล่องทดสอบรุ่นเผยแพร่](#release-test-boxes)

## การตั้งชื่อเวอร์ชัน

- เวอร์ชันรุ่นเผยแพร่ extended-stable รายเดือนบน npm: `YYYY.M.PATCH` โดยมี `PATCH >= 33`, แท็ก git `vYYYY.M.PATCH`
- เวอร์ชันรุ่นเผยแพร่สุดท้ายรายวัน/ปกติ: `YYYY.M.PATCH` โดยมี `PATCH < 33`, แท็ก git `vYYYY.M.PATCH`
- เวอร์ชันรุ่นเผยแพร่แก้ไขสำรองแบบปกติ: `YYYY.M.PATCH-N`, แท็ก git `vYYYY.M.PATCH-N`
- เวอร์ชันก่อนเผยแพร่เบตา: `YYYY.M.PATCH-beta.N`, แท็ก git `vYYYY.M.PATCH-beta.N`
- เวอร์ชันก่อนเผยแพร่อัลฟา: `YYYY.M.PATCH-alpha.N`, แท็ก git `vYYYY.M.PATCH-alpha.N`
- ห้ามเติมเลขศูนย์นำหน้าเดือนหรือแพตช์
- `PATCH` คือหมายเลขขบวนรุ่นเผยแพร่รายเดือนแบบลำดับต่อเนื่อง ไม่ใช่วันตามปฏิทิน รุ่นสุดท้ายปกติและรุ่นเบตาจะเลื่อนขบวนปัจจุบัน ส่วนแท็กที่เป็นอัลฟาเท่านั้นจะไม่ใช้หรือเลื่อนหมายเลขแพตช์เบตา/ปกติ ดังนั้นให้ละเว้นแท็กเดิมที่เป็นอัลฟาเท่านั้นซึ่งมีหมายเลขแพตช์สูงกว่าเมื่อเลือกขบวนเบตาหรือปกติ
- บิลด์อัลฟา/รายคืนใช้ขบวนแพตช์ถัดไปที่ยังไม่เผยแพร่ และเพิ่มเฉพาะ `alpha.N` สำหรับบิลด์ซ้ำ เมื่อแพตช์นั้นมีรุ่นเบตาแล้ว บิลด์อัลฟาใหม่จะย้ายไปยังแพตช์ถัดไป
- เวอร์ชัน npm เปลี่ยนแปลงไม่ได้: ห้ามลบ เผยแพร่ซ้ำ หรือนำแท็กที่เผยแพร่แล้วกลับมาใช้ใหม่ ให้สร้างหมายเลขก่อนเผยแพร่ถัดไปหรือแพตช์รายเดือนถัดไปแทน
- `latest` ยังคงติดตามสาย npm ปกติ/รายวันปัจจุบัน; `beta` คือเป้าหมายติดตั้งเบตาปัจจุบัน
- `extended-stable` หมายถึงแพ็กเกจ npm ของเดือนก่อนหน้าที่ยังรองรับ โดยเริ่มที่แพตช์ `33`; แพตช์ `34` และหลังจากนั้นเป็นรุ่นบำรุงรักษาบนสายรายเดือนนั้น
- รุ่นสุดท้ายปกติและรุ่นแก้ไขปกติจะเผยแพร่ไปยัง npm `beta` โดยค่าเริ่มต้น; ผู้ดำเนินการเผยแพร่สามารถกำหนดเป้าหมาย `latest` อย่างชัดเจน หรือโปรโมตบิลด์เบตาที่ผ่านการตรวจสอบแล้วภายหลัง
- เส้นทาง extended-stable รายเดือนโดยเฉพาะจะเผยแพร่แพ็กเกจแกนหลักบน npm และ Plugin ทางการทุกรายการที่สามารถเผยแพร่บน npm ได้ด้วยเวอร์ชันเดียวกันทุกประการ เส้นทางนี้จะไม่เผยแพร่ Plugin ไปยัง ClawHub หรือเผยแพร่อาร์ติแฟกต์ macOS หรือ Windows, GitHub Release, dist-tag ของรีโพซิทอรีส่วนตัว, อิมเมจ Docker, อาร์ติแฟกต์อุปกรณ์เคลื่อนที่ หรือไฟล์ดาวน์โหลดจากเว็บไซต์
- รุ่นเผยแพร่สุดท้ายปกติทุกครั้งจะจัดส่งแพ็กเกจ npm, แอป macOS, Android APK แบบสแตนด์อโลนที่ลงนามแล้ว และตัวติดตั้ง Windows Hub ที่ลงนามแล้วพร้อมกัน โดยปกติรุ่นเบตาจะตรวจสอบและเผยแพร่เส้นทาง npm/แพ็กเกจก่อน ส่วนการบิลด์/ลงนาม/รับรองเอกสาร/โปรโมตแอปเนทีฟจะสงวนไว้สำหรับรุ่นสุดท้ายปกติ เว้นแต่มีการร้องขออย่างชัดเจน

## รอบการเผยแพร่

- รุ่นเผยแพร่ดำเนินตามลำดับเบตาก่อน; stable จะตามมาหลังจากตรวจสอบเบตาล่าสุดแล้วเท่านั้น
- โดยปกติผู้ดูแลจะสร้างรุ่นเผยแพร่จากสาขา `release/YYYY.M.PATCH` ที่สร้างจาก `main` ปัจจุบัน เพื่อให้การตรวจสอบและการแก้ไขรุ่นเผยแพร่ไม่ขัดขวางการพัฒนาใหม่บน `main`
- หากมีการพุชหรือเผยแพร่แท็กเบตาแล้วและจำเป็นต้องแก้ไข ผู้ดูแลจะสร้างแท็ก `-beta.N` ถัดไปแทนการลบหรือสร้างแท็กเดิมใหม่
- ขั้นตอนการเผยแพร่โดยละเอียด การอนุมัติ ข้อมูลประจำตัว และบันทึกการกู้คืนมีไว้สำหรับผู้ดูแลเท่านั้น

## การเผยแพร่ extended-stable รายเดือนเฉพาะ npm

นี่เป็นข้อยกเว้นเฉพาะจากขั้นตอนการเผยแพร่ปกติด้านล่าง สำหรับเดือนที่เสร็จสมบูรณ์
`YYYY.M` ให้สร้าง `extended-stable/YYYY.M.33`; เผยแพร่
`vYYYY.M.33` และแพตช์บำรุงรักษาหลังจากนั้นจากสาขาเดียวกัน แท็กรุ่นเผยแพร่
ปลายสาขา เช็กเอาต์ เวอร์ชันแพ็กเกจ การตรวจสอบล่วงหน้าของ npm และการรัน Full Release
Validation ต้องระบุคอมมิตเดียวกันทั้งหมด `main` ที่ได้รับการป้องกันต้อง
มีเวอร์ชันสุดท้ายของเดือนตามปฏิทินที่ใหม่กว่าอย่างเคร่งครัดและต่ำกว่าแพตช์
`33` อยู่แล้ว; แพตช์บำรุงรักษายังคงมีสิทธิ์หลังจาก `main` เลื่อนไปมากกว่า
หนึ่งเดือน

บนสาขา extended-stable ที่ตรงกันทุกประการ ให้เพิ่มเวอร์ชันแพ็กเกจรากเป็น `YYYY.M.P`, รัน
`pnpm release:prep` และตรวจสอบว่าแพ็กเกจส่วนขยายที่เผยแพร่ได้ทุกรายการมี
เวอร์ชันเดียวกัน คอมมิตและพุชการเปลี่ยนแปลงที่สร้างขึ้นทั้งหมด สร้างและพุช
แท็ก `vYYYY.M.P` ที่เปลี่ยนแปลงไม่ได้ ณ คอมมิตนั้น และบันทึก SHA แบบเต็มที่ได้
เวิร์กโฟลว์จะใช้โครงสร้างที่เตรียมไว้นี้; เวิร์กโฟลว์จะไม่เพิ่มหรือซิงโครไนซ์
เวอร์ชันให้โดยอัตโนมัติ

รันการตรวจสอบล่วงหน้าของ npm และ Full Release Validation จากปลายสาขา
ที่เตรียมไว้ตรงกันทุกประการ จากนั้นบันทึก ID ของการรันทั้งสองและครั้งที่รัน
Full Release Validation ที่สำเร็จ:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=true \
  -f npm_dist_tag=extended-stable

gh workflow run full-release-validation.yml \
  --ref extended-stable/YYYY.M.33 \
  -f ref=extended-stable/YYYY.M.33 \
  -f release_profile=stable
```

`release_profile=stable` คือโปรไฟล์ระดับความลึกของการตรวจสอบที่มีอยู่
ซึ่งแยกจาก dist-tag `extended-stable` ของ npm และตั้งใจให้
ไม่เปลี่ยนแปลง

หลังจากการรันทั้งสองสำเร็จ ให้เผยแพร่ Plugin ทางการทุกรายการที่เผยแพร่บน npm ได้จาก
ปลายสาขาเดียวกันทุกประการ แพตช์ `P` ต้องเป็น `33` หรือสูงกว่า ส่ง SHA รุ่นเผยแพร่แบบเต็ม
เป็น `ref` รอให้เมทริกซ์ทั้งหมดและการอ่านกลับจากรีจิสทรีเสร็จสมบูรณ์ จากนั้นบันทึก
ID การรัน Plugin NPM Release ที่สำเร็จ:

```bash
RELEASE_SHA="$(git rev-parse HEAD)"
gh workflow run plugin-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f publish_scope=all-publishable \
  -f ref="$RELEASE_SHA" \
  -f npm_dist_tag=extended-stable
```

เวิร์กโฟลว์ใช้รายการแพ็กเกจ `all-publishable` ที่เตรียมไว้ตามปกติ
รวมถึงแพ็กเกจที่ซอร์สไม่ได้เปลี่ยนแปลง เวิร์กโฟลว์จะตรวจสอบแพ็กเกจทุกแพ็กเกจที่ตรงกัน
ทุกประการและแท็ก `extended-stable` ของ Plugin ทุกแท็กก่อนที่จะสำเร็จ หากการรันบางส่วน
ล้มเหลว ให้รันคำสั่งเดิมซ้ำ: แพ็กเกจที่เผยแพร่แล้วจะถูกนำกลับมาใช้ใหม่ แท็ก Plugin
ที่ขาดหายหรือล้าสมัยจะถูกปรับให้สอดคล้องกันภายใต้สภาพแวดล้อมการเผยแพร่ npm และ
การอ่านกลับขั้นสุดท้ายยังคงครอบคลุมชุดแพ็กเกจทั้งหมด

หลังจากเวิร์กโฟลว์ Plugin สำเร็จและสภาพแวดล้อมการเผยแพร่ npm พร้อมแล้ว
ให้เผยแพร่ tarball แกนหลักจากการตรวจสอบล่วงหน้าที่ตรงกันทุกประการ การเผยแพร่แกนหลักจะตรวจสอบว่า
การรัน Plugin ที่อ้างอิงมีสถานะ `completed/success` บนสาขามาตรฐานเดียวกันและ
ใช้ SHA ซอร์สเดียวกันทุกประการ:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=false \
  -f npm_dist_tag=extended-stable \
  -f preflight_run_id=<npm-preflight-run-id> \
  -f full_release_validation_run_id=<full-validation-run-id> \
  -f full_release_validation_run_attempt=<full-validation-run-attempt> \
  -f plugin_npm_run_id=<plugin-npm-run-id>
```

สำหรับ fork หรือการซ้อมที่ไม่ใช่การใช้งานจริงซึ่งตั้งใจให้ไม่สามารถปฏิบัติตาม
นโยบายเดือนของ `.33` หรือ `main` ที่ได้รับการป้องกันได้ ให้เพิ่ม
`-f bypass_extended_stable_guard=true` ลงในการส่งงานทั้งการตรวจสอบล่วงหน้าและการเผยแพร่
ของ npm ค่าเริ่มต้นคือ `false` การข้ามนโยบายจะได้รับการยอมรับเฉพาะเมื่อใช้กับ
`npm_dist_tag=extended-stable` และจะถูกบันทึกไว้ในสรุปเวิร์กโฟลว์ การข้ามนี้
ไม่ข้าม ref เวิร์กโฟลว์ `extended-stable/YYYY.M.33` มาตรฐาน
ความเท่ากันของปลายสาขา/แท็ก/เช็กเอาต์ ไวยากรณ์แท็กสุดท้าย ความเท่ากันของเวอร์ชัน
แพ็กเกจ/แท็ก ข้อมูลประจำตัวของการรันและแมนิเฟสต์ที่อ้างอิง ที่มาของ tarball
การอนุมัติสภาพแวดล้อม การอ่านกลับจากรีจิสทรี หรือหลักฐานการซ่อมแซมตัวเลือก

เวิร์กโฟลว์เผยแพร่จะตรวจสอบข้อมูลประจำตัวของการตรวจสอบล่วงหน้า การตรวจสอบ และการรัน Plugin
ที่อ้างอิง ไดเจสต์ของ tarball ที่เตรียมไว้ และตัวเลือกรีจิสทรีของแกนหลัก
หลังจากเวิร์กโฟลว์สำเร็จ ให้ยืนยันผลแยกต่างหาก:

```bash
npm view openclaw@YYYY.M.P version --userconfig "$(mktemp)"
npm view openclaw@extended-stable version --userconfig "$(mktemp)"
```

ทั้งสองคำสั่งต้องคืนค่า `YYYY.M.P` หากการเผยแพร่สำเร็จแต่การอ่านกลับตัวเลือก
ล้มเหลว ห้ามเผยแพร่เวอร์ชันแพ็กเกจที่เปลี่ยนแปลงไม่ได้ซ้ำ ให้ใช้
คำสั่งซ่อมแซม `npm dist-tag add openclaw@YYYY.M.P extended-stable` เพียงคำสั่งเดียว
ที่พิมพ์ไว้ในสรุปแบบรันเสมอของเวิร์กโฟลว์ที่ล้มเหลว จากนั้นทำการอ่านกลับ
แยกต่างหากทั้งสองรายการซ้ำ การย้อนกลับไปยังตัวเลือกก่อนหน้าเป็นการตัดสินใจ
ของผู้ดำเนินการแยกต่างหาก ไม่ใช่เส้นทางซ่อมแซมการอ่านกลับ

## รายการตรวจสอบสำหรับผู้ดำเนินการเผยแพร่ปกติ

รายการตรวจสอบนี้แสดงโครงสร้างสาธารณะของขั้นตอนการเผยแพร่ ข้อมูลประจำตัวส่วนตัว การลงนาม การรับรองเอกสาร การกู้คืน dist-tag และรายละเอียดการย้อนกลับฉุกเฉินจะอยู่ในคู่มือการรันรุ่นเผยแพร่สำหรับผู้ดูแลเท่านั้น

1. เริ่มจาก `main` ปัจจุบัน: ดึงข้อมูลล่าสุด ยืนยันว่าคอมมิตเป้าหมายถูกพุชแล้ว และยืนยันว่า CI ของ `main` ผ่านมากเพียงพอสำหรับการสร้างสาขา
2. สร้าง `release/YYYY.M.PATCH` จากคอมมิตนั้น การ backport เป็นทางเลือก; ใช้เฉพาะชุดที่ผู้ดำเนินการเลือก เพิ่มทุกตำแหน่งเวอร์ชันที่จำเป็น รัน `pnpm release:prep` ดำเนินการแก้ไขรุ่นเผยแพร่และ forward-port ที่จำเป็นให้เสร็จ และตรวจสอบ `src/plugins/compat/registry.ts` พร้อมกับ `src/commands/doctor/shared/deprecation-compat.ts`
3. ตรึงคอมมิตก่อนบันทึกการเปลี่ยนแปลงที่ผลิตภัณฑ์เสร็จสมบูรณ์เป็น **Code SHA** รันการตรวจสอบล่วงหน้าของซอร์สแบบกำหนดผลได้ จากนั้นใช้ `node scripts/full-release-validation-at-sha.mjs --sha <code-sha> --target-ref release/YYYY.M.PATCH` วิธีนี้จะตรึงเครื่องมือเวิร์กโฟลว์ที่เชื่อถือได้ ขณะที่เมทริกซ์ Vitest, Docker, QA, แพ็กเกจ และประสิทธิภาพแบบเต็มกำหนดเป้าหมายไปยัง Code SHA ที่ตรงกันทุกประการ
4. จำแนกความล้มเหลวก่อนแก้ไข ความล้มเหลวของผลิตภัณฑ์/โค้ดจะสร้าง Code SHA ใหม่และต้องมีการตรวจสอบแบบเต็มที่ผ่านสำหรับ SHA นั้น ความล้มเหลวของเวิร์กโฟลว์ ชุดทดสอบ ข้อมูลประจำตัว การอนุมัติ หรือโครงสร้างพื้นฐานจะได้รับการซ่อมแซมในพื้นผิวเจ้าของและรันซ้ำกับ Code SHA เดิม
5. หลังจาก Code SHA ผ่านแล้วเท่านั้น ให้สร้างส่วน `CHANGELOG.md` ด้านบนจาก PR ที่ผสานแล้วและคอมมิตโดยตรงนับจากแท็กรุ่นที่จัดส่งแล้วล่าสุดซึ่งเข้าถึงได้ จัดทำรายการสำหรับผู้ใช้และไม่ซ้ำกัน เมื่อแท็กที่จัดส่งแล้วซึ่งแยกออกไปหรือ forward-port ภายหลังเชื่อมโยง PR ที่เผยแพร่แล้วอีกครั้ง ให้ส่งแท็กนั้นอย่างชัดเจนเป็น `--shipped-ref`
6. คอมมิตเฉพาะ `CHANGELOG.md` คอมมิตนี้คือ **Release SHA** diff ทั้งหมดจาก Code SHA ถึง Release SHA ต้องเป็น `CHANGELOG.md` เท่านั้น; หากมีเส้นทางอื่นเปลี่ยนแปลง รุ่นเผยแพร่จะกลับไปยังขั้นตอนที่ 2
7. รัน Full Release Validation ที่ตรึงด้วย SHA สำหรับ Release SHA โดยเปิดใช้การนำหลักฐานกลับมาใช้ใหม่ งานแม่แบบเบาต้องบันทึก `changelog-only-release-v1` ชี้ไปยัง Code SHA ที่ผ่าน และไม่ส่งงานลูกของผลิตภัณฑ์ใดๆ วิธีนี้นำหลักฐานผลิตภัณฑ์กลับมาใช้ใหม่; ไม่ได้นำไบต์แพ็กเกจกลับมาใช้ใหม่
8. รัน `OpenClaw NPM Release` โดยใช้ `preflight_only=true` กับ Release SHA/แท็ก บันทึก `preflight_run_id` ที่สำเร็จ ขั้นตอนนี้จะบิลด์และตรวจสอบไบต์แพ็กเกจที่ตรงกันทุกประการซึ่งรวมบันทึกการเปลี่ยนแปลงขั้นสุดท้าย
9. แท็ก Release SHA จากนั้นรันตัวช่วยรุ่นที่เสนอโดยใช้การตรวจสอบงานแม่ Release-SHA ที่สำเร็จและการตรวจสอบล่วงหน้าของ npm แทนการส่งงานใดงานหนึ่งอีกครั้ง:

   ```bash
   pnpm release:candidate -- \
     --tag vYYYY.M.PATCH-beta.N \
     --full-release-run <release-sha-validation-run-id> \
     --npm-preflight-run <preflight-run-id> \
     --skip-dispatch
   ```

   สำหรับ stable ให้ส่ง `--windows-node-tag vX.Y.Z` ด้วย ตัวช่วยจะตรวจสอบที่มาของบันทึกประจำรุ่น ไบต์พรีไฟลต์ของ npm หลักฐานการติดตั้ง/อัปเดต Parallels หลักฐานแพ็กเกจ Telegram และแผนเผยแพร่ Plugin จากนั้นจะแสดงคำสั่งเผยแพร่

   `OpenClaw Release Publish` จะส่งแพ็กเกจ Plugin ที่เลือกหรือแพ็กเกจ Plugin ที่เผยแพร่ได้ทั้งหมดไปยัง npm และส่งชุดเดียวกันไปยัง ClawHub แบบขนาน จากนั้นเลื่อนสถานะอาร์ติแฟกต์พรีไฟลต์ npm ของ OpenClaw ที่เตรียมไว้ด้วย dist-tag ที่ตรงกันเมื่อการเผยแพร่ Plugin ไปยัง npm สำเร็จ เช็กเอาต์ของรุ่นยังคงเป็นรากของผลิตภัณฑ์/ข้อมูล ส่วนการวางแผนและการตรวจสอบขั้นสุดท้ายจะดำเนินการจากเช็กเอาต์ซอร์สเวิร์กโฟลว์ที่เชื่อถือได้และตรงกันทุกประการ เพื่อไม่ให้คอมมิตรุ่นเก่าใช้เครื่องมือออกรุ่นที่ล้าสมัยโดยไม่มีการแจ้งเตือน ก่อนเริ่มกระบวนการย่อยสำหรับเผยแพร่ใดๆ ระบบจะเรนเดอร์และแคชเนื้อหา GitHub release ที่แน่นอน เมื่อส่วน `CHANGELOG.md` ที่ตรงกันและครบถ้วนอยู่ภายในขีดจำกัด 125,000 อักขระของ GitHub และเพดานความปลอดภัย 125,000 ไบต์ที่ตรงกันของตัวเรนเดอร์ หน้าจะมีส่วน `## YYYY.M.PATCH` ดังกล่าวทุกประการ รวมถึงหัวข้อของส่วนนั้น เมื่อส่วนต้นฉบับมีขนาดเกินขีดจำกัด หน้าจะเก็บบันทึกเชิงบรรณาธิการแบบจัดกลุ่มไว้ทุกประการ และแทนที่ระเบียนการมีส่วนร่วมที่มีขนาดใหญ่เกินไปด้วยลิงก์คงที่ไปยังระเบียนฉบับเต็มใน `CHANGELOG.md` ที่ตรึงกับแท็ก โดยจะไม่เผยแพร่ระเบียนบางส่วนหรือรายการหัวข้อย่อยที่ถูกตัดทอนเด็ดขาด เวิร์กโฟลว์จะเลือกเนื้อหาแบบเต็มหรือแบบกะทัดรัดก่อนเพิ่ม `### Release verification`; หากส่วนท้ายของหลักฐานทำให้เกินขีดจำกัด ระบบจะคงเนื้อหามาตรฐานไว้และอาศัยหลักฐานแนบที่เปลี่ยนแปลงไม่ได้แทน รุ่น stable ที่เผยแพร่ไปยัง npm `latest` จะกลายเป็น GitHub release ล่าสุด ส่วนรุ่นบำรุงรักษา stable ที่คงไว้บน npm `beta` จะถูกสร้างด้วย GitHub `latest=false` เวิร์กโฟลว์ยังอัปโหลดหลักฐานการพึ่งพาจากพรีไฟลต์ แมนิเฟสต์การตรวจสอบแบบเต็ม และหลักฐานการตรวจสอบรีจิสทรีหลังเผยแพร่ไปยัง GitHub release เพื่อรองรับการตอบสนองต่อเหตุขัดข้องหลังออกรุ่น เวิร์กโฟลว์จะแสดง ID ของรันย่อยทันที อนุมัติเกตสภาพแวดล้อมการออกรุ่นที่โทเค็นเวิร์กโฟลว์ได้รับอนุญาตให้อนุมัติโดยอัตโนมัติ สรุปงานย่อยที่ล้มเหลวพร้อมส่วนท้ายของล็อก สร้างหน้าร่าง GitHub release ล่วงหน้า และเลื่อนสถานะอาร์ติแฟกต์ Windows และ Android พร้อมกับการเผยแพร่ OpenClaw ไปยัง npm ปิดงานหน้า release และหลักฐานการพึ่งพาเมื่อขั้นตอนเหล่านั้นสำเร็จ รอ ClawHub ทุกครั้งที่มีการเผยแพร่ OpenClaw ไปยัง npm จากนั้นเรียกใช้ตัวตรวจสอบ beta จาก main ที่เชื่อถือได้ และอัปโหลดหลักฐานหลังเผยแพร่สำหรับ GitHub release, แพ็กเกจ npm, แพ็กเกจ Plugin บน npm ที่เลือก, แพ็กเกจ ClawHub ที่เลือก, ID ของรันเวิร์กโฟลว์ย่อย และ ID ของรัน NPM Telegram ที่ไม่บังคับ ตัวตรวจสอบบูตสแตรป ClawHub ต้องการพาธและ SHA ของเวิร์กโฟลว์จาก main ที่เชื่อถือได้ซึ่งตรงกันทุกประการ ความพยายามรันของผู้สร้างและรันปลายทาง SHA ของรุ่น ชุดแพ็กเกจที่ร้องขอ ทูเพิลอาร์ติแฟกต์แพ็กเกจที่เปลี่ยนแปลงไม่ได้ และอาร์ติแฟกต์การอ่านค่ากลับจากรีจิสทรีปลายทาง โดยจะไม่ยอมรับรันแบบเดิมจาก ref ของรุ่นแม้ว่าจะสำเร็จก็ตาม

   จากนั้นเรียกใช้การยอมรับแพ็กเกจหลังเผยแพร่กับแพ็กเกจ `openclaw@YYYY.M.PATCH-beta.N` หรือ `openclaw@beta` ที่เผยแพร่แล้ว หาก prerelease ที่พุชหรือเผยแพร่แล้วต้องแก้ไข ให้สร้างหมายเลข prerelease ถัดไปที่ตรงกัน ห้ามลบหรือเขียนทับรายการเดิมเด็ดขาด

10. เมื่อความพยายามเผยแพร่ล้มเหลว ให้คง Release SHA เดิมไว้ เว้นแต่ความล้มเหลวนั้นพิสูจน์ว่ามีข้อบกพร่องในผลิตภัณฑ์หรือ changelog ให้ทำงานต่อจากกระบวนการย่อยและอาร์ติแฟกต์ที่เปลี่ยนแปลงไม่ได้ซึ่งสำเร็จแล้ว ห้ามสร้างใหม่หรือเผยแพร่ซ้ำแพ็กเกจเวอร์ชันที่สำเร็จแล้วเด็ดขาด
11. สำหรับ stable ให้ดำเนินการต่อเมื่อ beta หรือ release candidate ที่ผ่านการตรวจสอบมีหลักฐานการตรวจสอบที่จำเป็นเท่านั้น การเผยแพร่ stable ไปยัง npm ต้องผ่าน `OpenClaw Release Publish` เช่นกัน โดยนำอาร์ติแฟกต์พรีไฟลต์ที่สำเร็จแล้วกลับมาใช้ผ่าน `preflight_run_id` ความพร้อมในการออกรุ่น stable สำหรับ macOS ยังต้องมี `.zip`, `.dmg`, `.dSYM.zip` ที่จัดแพ็กเกจแล้ว และ `appcast.xml` ที่อัปเดตแล้วบน `main`; เวิร์กโฟลว์เผยแพร่ macOS จะเผยแพร่ appcast ที่ลงนามแล้วไปยัง `main` สาธารณะโดยอัตโนมัติหลังจากตรวจสอบอาร์ติแฟกต์รุ่นแล้ว หรือเปิด/อัปเดต appcast PR หากการป้องกันสาขาขัดขวางการพุชโดยตรง ความพร้อมของ Windows Hub สำหรับ stable ต้องมีอาร์ติแฟกต์ `OpenClawCompanion-Setup-x64.exe`, `OpenClawCompanion-Setup-arm64.exe` และ `OpenClawCompanion-SHA256SUMS.txt` ที่ลงนามแล้วบน GitHub release ของ OpenClaw ส่งแท็กออกรุ่น `openclaw/openclaw-windows-node` ที่ลงนามแล้วและตรงกันทุกประการเป็น `windows_node_tag` และแมปไดเจสต์ของโปรแกรมติดตั้งที่ candidate อนุมัติแล้วเป็น `windows_node_installer_digests`; `OpenClaw Release Publish` จะเก็บร่าง release ไว้ ส่ง `Windows Node Release` และตรวจสอบอาร์ติแฟกต์ทั้งสามก่อนเผยแพร่
12. หลังเผยแพร่ ให้เรียกใช้ตัวตรวจสอบ npm หลังเผยแพร่, Telegram E2E แบบสแตนด์อโลนที่ไม่บังคับกับ npm ที่เผยแพร่แล้วเมื่อต้องการหลักฐานช่องทางหลังเผยแพร่, การเลื่อนสถานะ dist-tag เมื่อจำเป็น, ตรวจสอบหน้า GitHub release ที่สร้างขึ้น, ดำเนินขั้นตอนประกาศออกรุ่น แล้วทำ [การปิดงาน stable บน main](#stable-main-closeout) ให้เสร็จก่อนถือว่าการออกรุ่น stable เสร็จสมบูรณ์

## การปิดงาน stable บน main

การเผยแพร่ stable ยังไม่เสร็จสมบูรณ์จนกว่า `main` จะมีสถานะของรุ่นที่จัดส่งจริง

1. เริ่มจาก `main` ล่าสุดที่เป็นปัจจุบัน ตรวจสอบ `release/YYYY.M.PATCH` เทียบกับรายการดังกล่าว และ forward-port การแก้ไขจริงที่ไม่มีใน `main` ห้ามผสานอะแดปเตอร์ด้านความเข้ากันได้ การทดสอบ หรือการตรวจสอบที่มีเฉพาะรุ่นเข้าไปใน `main` ที่ใหม่กว่าโดยไม่พิจารณา
2. สำหรับเส้นทางปกติ ให้ตั้ง `main` เป็นเวอร์ชัน stable ที่จัดส่ง การปิดงานล่าช้าอาจใช้ `main` หลังจากที่รายการดังกล่าวเลื่อนไปเป็น OpenClaw CalVer รุ่น stable ที่ใหม่กว่าแล้ว ห้ามดาวน์เกรดขบวนการออกรุ่นที่เริ่มไปแล้วเพียงเพื่อปิดงานรุ่นก่อนหน้า ตัวตรวจสอบยังคงต้องการส่วน changelog และรายการ appcast ของรุ่นที่จัดส่งซึ่งตรงกันทุกประการ และจะบันทึกเวอร์ชันกับ SHA จริงของ `main` เรียกใช้ `pnpm release:prep` หลังจากเปลี่ยนเวอร์ชันรากใดๆ แล้วจึงเรียกใช้ `pnpm deps:shrinkwrap:generate`
3. ทำให้ส่วน `## YYYY.M.PATCH` ของ `CHANGELOG.md` บน `main` ตรงกับสาขารุ่นที่ติดแท็กทุกประการ รวมการอัปเดต stable ของ `appcast.xml` เมื่อรุ่น mac เผยแพร่รายการดังกล่าว
4. ห้ามเพิ่ม `YYYY.M.PATCH+1`, เวอร์ชัน beta หรือส่วน changelog ในอนาคตที่ว่างเปล่าลงใน `main` จนกว่าผู้ดำเนินการจะเริ่มขบวนการออกรุ่นนั้นอย่างชัดเจน
5. เรียกใช้ `pnpm release:generated:check`, `pnpm deps:shrinkwrap:check` และ `OPENCLAW_TESTBOX=1 pnpm check:changed` พุชแล้วตรวจสอบว่า `origin/main` มีเวอร์ชันและ changelog ที่จัดส่ง ก่อนถือว่าการออกรุ่น stable เสร็จสิ้น
6. ดูแลตัวแปรที่เก็บของ `RELEASE_ROLLBACK_DRILL_ID` และ `RELEASE_ROLLBACK_DRILL_DATE` ให้เป็นปัจจุบันหลังการซ้อมย้อนกลับแบบส่วนตัวแต่ละครั้ง

`OpenClaw Stable Main Closeout` เริ่มจากการพุช `main` ที่มีเวอร์ชัน changelog และ appcast ที่จัดส่งหลังการเผยแพร่ stable ระบบจะอ่านหลักฐานหลังเผยแพร่ที่เปลี่ยนแปลงไม่ได้เพื่อผูกแท็กที่จัดส่งเข้ากับรัน Full Release Validation และ Publish จากนั้นตรวจสอบสถานะ stable บน main, release, ช่วง soak สำหรับ stable ที่บังคับ และหลักฐานประสิทธิภาพที่มีผลบล็อก ระบบจะแนบแมนิเฟสต์การปิดงานและ checksum ที่เปลี่ยนแปลงไม่ได้ไปยัง GitHub release ทริกเกอร์พุชอัตโนมัติจะข้ามรุ่นแบบเดิมที่เก่ากว่าหลักฐานหลังเผยแพร่ที่เปลี่ยนแปลงไม่ได้ และจะไม่ถือว่าการข้ามนั้นเป็นการปิดงานที่เสร็จสมบูรณ์

การปิดงานที่สมบูรณ์ต้องมีทั้งอาร์ติแฟกต์และ checksum ที่ตรงกัน แมนิเฟสต์บางส่วนจะเล่นซ้ำ SHA ของ `main` และการซ้อมย้อนกลับที่บันทึกไว้เพื่อสร้างไบต์ที่เหมือนเดิม จากนั้นแนบ checksum ที่ขาดหายไป คู่ที่ไม่ถูกต้อง หรือ checksum ที่ไม่มีแมนิเฟสต์ จะยังคงมีผลบล็อก รันที่ทริกเกอร์ด้วยการพุชแต่ไม่มีตัวแปรที่เก็บสำหรับการซ้อมย้อนกลับจะข้ามโดยไม่ทำให้การปิดงานเสร็จสมบูรณ์ ระเบียนการซ้อมที่ขาดหายไปหรือเก่ากว่า 90 วันยังคงบล็อกการปิดงานแบบดำเนินการเองที่มีหลักฐานรองรับ คำสั่งกู้คืนแบบส่วนตัวยังคงอยู่ในคู่มือปฏิบัติการสำหรับผู้ดูแลเท่านั้น ใช้การส่งงานด้วยตนเองเฉพาะเพื่อซ่อมแซมหรือเล่นซ้ำการปิดงาน stable ที่มีหลักฐานรองรับ

หาก Release Publish ระดับแม่ล้มเหลวหลังจากแนบหลักฐาน npm/Plugin ที่เปลี่ยนแปลงไม่ได้แล้วเท่านั้น ให้ซ่อมแซมและเผยแพร่อาร์ติแฟกต์แพลตฟอร์ม stable ทั้งหมดก่อน จากนั้นผู้ดูแลจึงอาจส่งงานปิดงานด้วยตนเองโดยใช้ `allow_failed_publish_recovery=true`; โหมดดังกล่าวยอมรับเฉพาะรันระดับแม่ที่ล้มเหลวและเสร็จสิ้นแล้ว และยังต้องการสัญญาอาร์ติแฟกต์ Android และ Windows ที่ตรงกันทุกประการ ไดเจสต์ SHA-256 ของ GitHub การตรวจสอบ checksum ที่มาอาร์ติแฟกต์ของ Android และการเลื่อนสถานะ Windows ที่สำเร็จซึ่งส่งจากรันระดับแม่ โดยการตรวจสอบ Authenticode และไดเจสต์ที่ candidate อนุมัติต้องตรงกับโปรแกรมติดตั้งที่เผยแพร่ พร้อมกับการตรวจสอบ macOS/appcast ตามปกติ การปิดงานอัตโนมัติจากการพุชจะไม่เปิดใช้โหมดกู้คืนนี้เด็ดขาด

แท็กแก้ไขสำรองแบบเดิมอาจนำหลักฐานแพ็กเกจฐานกลับมาใช้ได้เฉพาะเมื่อแท็กแก้ไขชี้ไปยังคอมมิตซอร์สเดียวกับแท็ก stable ฐาน รุ่น Android ของแท็กดังกล่าวจะนำ APK ที่ตรวจสอบแล้วของแท็กฐานกลับมาใช้ และเพิ่มที่มาอาร์ติแฟกต์สำหรับแท็กแก้ไข การแก้ไขที่ใช้ซอร์สต่างกันต้องเผยแพร่และตรวจสอบหลักฐานแพ็กเกจของตนเอง และใช้ `versionCode` ของ Android ที่สูงกว่า

## พรีไฟลต์การออกรุ่น

- เรียกใช้ `pnpm check:test-types` ก่อนพรีไฟลต์การออกรุ่น เพื่อให้ TypeScript ของการทดสอบยังคงครอบคลุมนอกเกต `pnpm check` ภายในเครื่องที่เร็วกว่า
- เรียกใช้ `pnpm check:architecture` ก่อนพรีไฟลต์การออกรุ่น เพื่อให้การตรวจสอบวงจรการนำเข้าและขอบเขตสถาปัตยกรรมที่กว้างขึ้นผ่านนอกเกตภายในเครื่องที่เร็วกว่า
- เรียกใช้ `pnpm build && pnpm ui:build` ก่อน `pnpm release:check` เพื่อให้อาร์ติแฟกต์รุ่น `dist/*` ที่คาดไว้และบันเดิล Control UI มีอยู่สำหรับขั้นตอนตรวจสอบแพ็ก
- เรียกใช้ `pnpm release:prep` หลังเพิ่มเวอร์ชันรากและก่อนติดแท็ก คำสั่งนี้เรียกใช้ตัวสร้างรุ่นแบบกำหนดผลลัพธ์ได้ทุกตัวที่มักคลาดเคลื่อนหลังการเปลี่ยนเวอร์ชัน/การกำหนดค่า/API ได้แก่ เวอร์ชัน Plugin, shrinkwrap ของ npm, บัญชีรายการ Plugin, สคีมาการกำหนดค่าฐาน, เมทาดาทาการกำหนดค่าช่องทางที่รวมมาให้, baseline เอกสารการกำหนดค่า, การส่งออก SDK ของ Plugin, แมนิเฟสต์สัญญา API ของ SDK สำหรับ Plugin และบันเดิลภาษาของ Control UI `pnpm release:check` จะเรียกใช้ตัวป้องกันเหล่านั้นซ้ำในโหมดตรวจสอบ (รวมถึงเกตภาษาที่เข้มงวดแบบไม่มี fallback และงบประมาณพื้นผิว SDK ของ Plugin) และรายงานความล้มเหลวจากความคลาดเคลื่อนของสิ่งที่สร้างขึ้นทั้งหมดในรอบเดียว ก่อนเรียกใช้การตรวจสอบการออกรุ่นแพ็กเกจ
- การซิงค์เวอร์ชัน Plugin จะอัปเดตแพ็กเกจรันไทม์ `@openclaw/ai` ที่เผยแพร่ได้ เวอร์ชันแพ็กเกจ Plugin อย่างเป็นทางการ และค่าขั้นต่ำ `openclaw.compat.pluginApi` ที่มีอยู่ ให้เป็นเวอร์ชันรุ่นของ OpenClaw โดยค่าเริ่มต้น ให้ถือว่าฟิลด์ดังกล่าวเป็นค่าขั้นต่ำของ API สำหรับ SDK/รันไทม์ของ Plugin ไม่ใช่เพียงสำเนาเวอร์ชันแพ็กเกจ สำหรับรุ่นที่มีเฉพาะ Plugin ซึ่งตั้งใจให้ยังเข้ากันได้กับโฮสต์ OpenClaw รุ่นเก่า ให้คงค่าขั้นต่ำไว้ที่ API ของโฮสต์รุ่นเก่าสุดที่รองรับ และบันทึกเหตุผลของการเลือกดังกล่าวไว้ในหลักฐานการออกรุ่น Plugin
- เรียกใช้เวิร์กโฟลว์ `Full Release Validation` แบบดำเนินการเองก่อนอนุมัติรุ่น เพื่อเริ่มกล่องทดสอบก่อนออกรุ่นทั้งหมดจากจุดเริ่มต้นเดียว เวิร์กโฟลว์นี้รับสาขา แท็ก หรือ SHA ของคอมมิตแบบเต็ม ส่ง `CI` แบบดำเนินการเอง และส่ง `OpenClaw Release Checks` สำหรับการทดสอบเบื้องต้นการติดตั้ง การยอมรับแพ็กเกจ การตรวจสอบแพ็กเกจข้ามระบบปฏิบัติการ ความสอดคล้องของ QA Lab และเลน Matrix กับ Telegram รัน stable และรันแบบเต็มจะรวมการทดสอบ live/E2E แบบครบถ้วนและการ soak เส้นทางออกรุ่นผ่าน Docker เสมอ ส่วน `run_release_soak=true` จะเก็บไว้สำหรับการ soak beta ที่ร้องขออย่างชัดเจน การยอมรับแพ็กเกจจะให้ Telegram E2E มาตรฐานของแพ็กเกจระหว่างการตรวจสอบ candidate เพื่อหลีกเลี่ยงตัวโพล live ตัวที่สองซึ่งทำงานพร้อมกัน

  ระบุ `release_package_spec` หลังเผยแพร่ beta เพื่อนำแพ็กเกจ npm ที่จัดส่งแล้วกลับมาใช้ในการตรวจสอบรุ่น การยอมรับแพ็กเกจ และ Telegram E2E ของแพ็กเกจ โดยไม่สร้าง tarball รุ่นใหม่ ระบุ `npm_telegram_package_spec` เฉพาะเมื่อ Telegram ควรใช้แพ็กเกจที่เผยแพร่ต่างจากส่วนอื่นของการตรวจสอบรุ่น ระบุ `package_acceptance_package_spec` เมื่อการยอมรับแพ็กเกจควรใช้แพ็กเกจที่เผยแพร่ต่างจากข้อกำหนดแพ็กเกจของรุ่น ระบุ `evidence_package_spec` เมื่อรายงานหลักฐานการออกรุ่นควรพิสูจน์ว่าการตรวจสอบตรงกับแพ็กเกจ npm ที่เผยแพร่แล้ว โดยไม่บังคับใช้ Telegram E2E

  ```bash
  node scripts/full-release-validation-at-sha.mjs \
    --sha <code-sha> \
    --target-ref release/YYYY.M.PATCH
  ```

- เรียกใช้เวิร์กโฟลว์ `Package Acceptance` แบบแมนนวลเมื่อต้องการหลักฐานจากช่องทางเสริมสำหรับแพ็กเกจตัวเลือกในระหว่างที่งานรีลีสดำเนินต่อไป ใช้ `source=npm` สำหรับ `openclaw@beta`, `openclaw@latest` หรือเวอร์ชันรีลีสที่ระบุแน่นอน; ใช้ `source=ref` เพื่อแพ็ก branch/tag/SHA ของ `package_ref` ที่เชื่อถือได้ด้วยชุดทดสอบ `workflow_ref` ปัจจุบัน; ใช้ `source=url` สำหรับ tarball HTTPS สาธารณะที่ต้องระบุ SHA-256 และใช้นโยบาย URL สาธารณะที่เข้มงวด; ใช้ `source=trusted-url` สำหรับนโยบายแหล่งที่เชื่อถือได้แบบมีชื่อซึ่งกำหนดให้ใช้ `trusted_source_id` และ SHA-256; หรือใช้ `source=artifact` สำหรับ tarball ที่อัปโหลดโดยการรัน GitHub Actions อื่น

  เวิร์กโฟลว์จะแปลงตัวเลือกเป็น `package-under-test` นำตัวจัดกำหนดการรีลีส Docker E2E มาใช้กับ tarball นั้นอีกครั้ง และสามารถเรียกใช้ QA ของ Telegram กับ tarball เดียวกันโดยใช้ `telegram_mode=mock-openai` หรือ `telegram_mode=live-frontier` เมื่อเลน Docker ที่เลือกมี `published-upgrade-survivor` อาร์ติแฟกต์แพ็กเกจจะเป็นตัวเลือก และ `published_upgrade_survivor_baseline` จะเลือกข้อมูลพื้นฐานที่เผยแพร่แล้ว `update-restart-auth` ใช้แพ็กเกจตัวเลือกทั้งเป็น CLI ที่ติดตั้งและแพ็กเกจภายใต้การทดสอบ เพื่อให้ทดสอบเส้นทางรีสตาร์ตที่มีการจัดการของคำสั่งอัปเดตตัวเลือก

  ตัวอย่าง:

  ```bash
  gh workflow run package-acceptance.yml --ref main -f workflow_ref=main -f source=npm -f package_spec=openclaw@beta -f suite_profile=product -f published_upgrade_survivor_baseline=openclaw@2026.4.26 -f telegram_mode=mock-openai
  ```

  โปรไฟล์ที่ใช้ทั่วไป:
  - `smoke`: เลนการติดตั้ง/ช่องทาง/เอเจนต์ เครือข่าย Gateway และการโหลดการกำหนดค่าใหม่
  - `package`: เลนแพ็กเกจ/อัปเดต/รีสตาร์ต/Plugin ที่ทำงานกับอาร์ติแฟกต์โดยตรง โดยไม่มี OpenWebUI หรือ ClawHub แบบสด
  - `product`: โปรไฟล์แพ็กเกจ รวมถึงช่องทาง MCP, การล้างข้อมูล cron/เอเจนต์ย่อย, การค้นหาเว็บของ OpenAI และ OpenWebUI
  - `full`: ส่วนย่อยของเส้นทางรีลีส Docker พร้อม OpenWebUI
  - `custom`: การเลือก `docker_lanes` ที่ระบุแน่นอนสำหรับการเรียกใช้ซ้ำแบบเจาะจง

- เรียกใช้เวิร์กโฟลว์ `CI` แบบแมนนวลโดยตรงเมื่อต้องการเฉพาะความครอบคลุมของ CI ปกติที่ให้ผลลัพธ์แน่นอนสำหรับตัวเลือกรีลีส การสั่งรัน CI แบบแมนนวลจะข้ามการจำกัดขอบเขตตามการเปลี่ยนแปลง และบังคับใช้ชาร์ด Linux Node, ชาร์ด Plugin ที่รวมมาให้, ชาร์ดสัญญา Plugin และช่องทาง, ความเข้ากันได้กับ Node 22, `check-*`, `check-additional-*`, การตรวจสอบเบื้องต้นของอาร์ติแฟกต์ที่บิลด์แล้ว, การตรวจสอบเอกสาร, Skills ของ Python, Windows, macOS และเลน i18n ของ Control UI การรัน CI แบบแมนนวลแยกเดี่ยวจะเรียกใช้ Android เฉพาะเมื่อสั่งรันด้วย `include_android=true`; `Full Release Validation` จะส่งอินพุตนั้นให้ CI ลูก

  ```bash
  gh workflow run ci.yml --ref release/YYYY.M.PATCH -f include_android=true
  ```

- ระบบอัตโนมัติสำหรับรีลีสของผู้ดูแลใช้การตรวจสอบล่วงหน้าแล้วจึงเลื่อนขั้น:
  - การเผยแพร่ npm จริงต้องผ่าน npm `preflight_run_id` ที่สำเร็จ
  - การควบคุมลำดับและการตรวจสอบล่วงหน้าสำหรับการเผยแพร่เบต้าและเสถียรตามปกติใช้ `main` ที่เชื่อถือได้กับแท็กเป้าหมายที่ตรงกันทุกประการ การเผยแพร่และการตรวจสอบล่วงหน้าแบบแอลฟาของ Tideclaw ใช้สาขาแอลฟาที่ตรงกัน
  - รีลีส npm แบบเสถียรมีค่าเริ่มต้นเป็น `beta`; การเผยแพร่ npm แบบเสถียรสามารถกำหนดเป้าหมายเป็น `latest` อย่างชัดเจนผ่านอินพุตของเวิร์กโฟลว์
  - การเปลี่ยนแปลง dist-tag ของ npm ที่ใช้โทเค็นอยู่ใน `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml` เนื่องจาก `npm dist-tag add` ยังคงต้องใช้ `NPM_TOKEN` ขณะที่รีโพซิทอรีต้นทางคงการเผยแพร่แบบ OIDC เท่านั้น
  - `macOS Release` แบบสาธารณะใช้สำหรับการตรวจสอบเท่านั้น เมื่อแท็กอยู่เฉพาะบนสาขารีลีสแต่เวิร์กโฟลว์ถูกสั่งเริ่มจาก `main` ให้ตั้งค่า `public_release_branch=release/YYYY.M.PATCH`
  - การเผยแพร่ macOS จริงต้องผ่าน macOS `preflight_run_id` และ `validate_run_id` ที่สำเร็จ
  - เส้นทางการเผยแพร่จริงจะเลื่อนขั้นอาร์ติแฟกต์ที่เตรียมไว้แทนการบิลด์ใหม่อีกครั้ง
- สำหรับรีลีสแก้ไขแบบเสถียร เช่น `YYYY.M.PATCH-N` ตัวตรวจสอบหลังเผยแพร่ยังตรวจสอบเส้นทางอัปเกรดด้วยคำนำหน้าชั่วคราวเดียวกันจาก `YYYY.M.PATCH` ไปยัง `YYYY.M.PATCH-N` เพื่อไม่ให้การแก้ไขรีลีสปล่อยให้การติดตั้งส่วนกลางรุ่นเก่าคงอยู่บนเพย์โหลดเสถียรพื้นฐานโดยไม่มีการแจ้งเตือน
- การตรวจสอบล่วงหน้าของรีลีส npm จะปฏิเสธโดยค่าเริ่มต้น เว้นแต่ tarball จะมีทั้ง `dist/control-ui/index.html` และเพย์โหลด `dist/control-ui/assets/` ที่ไม่ว่างเปล่า เพื่อไม่ให้เราจัดส่งแดชบอร์ดเบราว์เซอร์ที่ว่างเปล่าอีก
- การตรวจสอบหลังเผยแพร่ยังตรวจสอบว่า entrypoint ของ Plugin ที่เผยแพร่และข้อมูลเมตาของแพ็กเกจมีอยู่ในเลย์เอาต์รีจิสทรีที่ติดตั้งแล้ว รีลีสที่จัดส่งโดยไม่มีเพย์โหลดรันไทม์ของ Plugin จะไม่ผ่านตัวตรวจสอบหลังเผยแพร่และไม่สามารถเลื่อนขั้นเป็น `latest` ได้
- `pnpm test:install:smoke` ยังบังคับใช้งบประมาณ `unpackedSize` ของ npm pack กับ tarball อัปเดตแคนดิเดต เพื่อให้ E2E ของตัวติดตั้งตรวจพบการเพิ่มขนาดแพ็กเกจโดยไม่ตั้งใจก่อนเข้าสู่เส้นทางเผยแพร่รีลีส
- หากงานรีลีสมีการเปลี่ยนแปลงการวางแผน CI ไฟล์แมนิเฟสต์เวลาการทำงานของส่วนขยาย หรือเมทริกซ์การทดสอบส่วนขยาย ให้สร้างใหม่และตรวจสอบเอาต์พุตเมทริกซ์ `plugin-prerelease-extension-shard` ที่ตัววางแผนเป็นเจ้าของจาก `.github/workflows/plugin-prerelease.yml` ก่อนอนุมัติ เพื่อไม่ให้บันทึกประจำรุ่นอธิบายเลย์เอาต์ CI ที่ล้าสมัย
- ความพร้อมสำหรับรีลีส macOS แบบเสถียรยังรวมถึงพื้นผิวของตัวอัปเดตด้วย: รีลีส GitHub ต้องมี `.zip`, `.dmg` และ `.dSYM.zip` ที่แพ็กเกจแล้ว; `appcast.xml` บน `main` ต้องชี้ไปยังไฟล์ zip เสถียรใหม่หลังเผยแพร่ (เวิร์กโฟลว์เผยแพร่ macOS จะคอมมิตให้โดยอัตโนมัติ หรือเปิด PR ของ appcast เมื่อการพุชโดยตรงถูกบล็อก); แอปที่แพ็กเกจแล้วต้องคง bundle id ที่ไม่ใช่แบบดีบัก, URL ฟีด Sparkle ที่ไม่ว่างเปล่า และ `CFBundleVersion` ที่เท่ากับหรือสูงกว่าค่าขั้นต่ำของบิลด์ Sparkle มาตรฐานสำหรับเวอร์ชันรีลีสนั้น

## กล่องทดสอบรีลีส

`Full Release Validation` คือวิธีที่ผู้ปฏิบัติงานใช้เริ่มเมทริกซ์ผลิตภัณฑ์แบบเต็มจาก entrypoint เดียว ใช้ตัวช่วยเพื่อให้ทุกเวิร์กโฟลว์ลูกทำงานจากสาขาชั่วคราวที่ตรึงไว้กับ SHA ของเวิร์กโฟลว์ `main` ที่เชื่อถือได้หนึ่งรายการ ขณะที่คอมมิตที่ร้องขอยังคงเป็นแคนดิเดตภายใต้การทดสอบ:

```bash
pnpm ci:full-release \
  --sha <code-sha> \
  --target-ref release/YYYY.M.PATCH
```

ตัวช่วยจะดึงข้อมูล `origin/main` ปัจจุบัน พุช `release-ci/<workflow-sha>-...` ที่คอมมิตเวิร์กโฟลว์ที่เชื่อถือได้นั้น อนุมาน `beta` จากเวอร์ชันแพ็กเกจแอลฟา/เบต้าและใช้ `stable` ในกรณีอื่น สั่งเริ่ม `Full Release Validation` จากสาขาชั่วคราวพร้อม `ref=<target-sha>` ตรวจสอบว่า `headSha` ของเวิร์กโฟลว์ลูกทุกตัวตรงกับ SHA ของเวิร์กโฟลว์แม่ที่ตรึงไว้ แล้วลบสาขาชั่วคราว ส่ง `-f reuse_evidence=false` เพื่อบังคับให้เรียกใช้ใหม่, `-f release_profile=full` สำหรับการตรวจสอบคำแนะนำแบบกว้าง หรือ `--workflow-sha <trusted-main-sha>` เพื่อตรึงคอมมิตที่เก่ากว่าซึ่งยังสามารถเข้าถึงได้จาก `origin/main` ปัจจุบัน ตัวเวิร์กโฟลว์เองจะไม่เขียน ref ของรีโพซิทอรี วิธีนี้ทำให้เครื่องมือรีลีสที่มีเฉพาะบน main ยังคงพร้อมใช้งานโดยไม่เพิ่มคอมมิตเครื่องมือลงในแคนดิเดต และหลีกเลี่ยงการพิสูจน์รอบการทำงานลูก `main` ที่ใหม่กว่าโดยไม่ตั้งใจ

หลังจาก Code SHA ผ่านเป็นสีเขียว ให้คอมมิตเฉพาะ `CHANGELOG.md` และเรียกใช้ตัวช่วยเดิมด้วย Release SHA:

```bash
pnpm ci:full-release \
  --sha <release-sha> \
  --target-ref release/YYYY.M.PATCH
```

เวิร์กโฟลว์แม่รอบที่สองจะนำหลักฐานผลิตภัณฑ์กลับมาใช้ใหม่เฉพาะเมื่อ GitHub พิสูจน์ว่า Release SHA สืบทอดจาก Code SHA และชุดพาธที่เปลี่ยนแปลงทั้งหมดเป็น `CHANGELOG.md` อย่างตรงกันทุกประการ โดยจะบันทึก `changelog-only-release-v1` และไม่สั่งเริ่มเวิร์กโฟลว์ลูกด้านผลิตภัณฑ์ การตรวจสอบล่วงหน้าของ npm และการยอมรับแพ็กเกจ/การติดตั้งยังคงทำงานกับ Release SHA เนื่องจากไบต์ของ tarball เปลี่ยนแปลง

สำหรับ Code SHA ใหม่ เวิร์กโฟลว์จะแก้ไขเป้าหมาย สั่งเริ่ม `CI` แบบดำเนินการด้วยตนเอง แล้วจึงสั่งเริ่ม `OpenClaw Release Checks` จากนั้น `OpenClaw Release Checks` จะกระจายงานไปยังการทดสอบติดตั้งอย่างรวดเร็ว การตรวจสอบรีลีสข้ามระบบปฏิบัติการ ความครอบคลุมเส้นทางรีลีส Docker แบบสด/E2E เมื่อเปิดใช้การเฝ้าระยะยาว การยอมรับแพ็กเกจพร้อม E2E แพ็กเกจ Telegram มาตรฐาน การตรวจสอบความสอดคล้องของ QA Lab, Matrix แบบสด และ Telegram แบบสด การเรียกใช้แบบเต็ม/ทั้งหมดจะยอมรับได้เฉพาะเมื่อสรุป `Full Release Validation` แสดงว่า `normal_ci`, `plugin_prerelease` และ `release_checks` สำเร็จ เว้นแต่การเรียกใช้ซ้ำแบบเจาะจงจะข้ามเวิร์กโฟลว์ลูก `Plugin Prerelease` ที่แยกต่างหากโดยเจตนา ใช้เวิร์กโฟลว์ลูก `npm-telegram` แบบเดี่ยวเฉพาะสำหรับการเรียกใช้แพ็กเกจที่เผยแพร่แล้วซ้ำแบบเจาะจงด้วย `release_package_spec` หรือ `npm_telegram_package_spec` สรุปของตัวตรวจสอบขั้นสุดท้ายมีตารางงานที่ช้าที่สุดสำหรับรอบการทำงานลูกแต่ละรอบ เพื่อให้ผู้จัดการรีลีสดูเส้นทางวิกฤตปัจจุบันได้โดยไม่ต้องดาวน์โหลดบันทึก

เวิร์กโฟลว์ลูกด้านประสิทธิภาพผลิตภัณฑ์ใช้เฉพาะอาร์ติแฟกต์ในเส้นทางรีลีสนี้ เวิร์กโฟลว์
ครอบจะสั่งเริ่มด้วย `publish_reports=false` และการตรวจสอบจะถูกปฏิเสธ
เว้นแต่มาตรการป้องกันแบบเฉพาะอาร์ติแฟกต์จะพิสูจน์ว่าตัวเผยแพร่รายงาน Clawgrit ยังคง
ถูกข้าม

ดู [การตรวจสอบรีลีสแบบเต็ม](/th/reference/full-release-validation) สำหรับเมทริกซ์ขั้นตอนทั้งหมด ชื่องานเวิร์กโฟลว์ที่ตรงกัน ความแตกต่างระหว่างโปรไฟล์เสถียรกับโปรไฟล์เต็ม อาร์ติแฟกต์ และตัวจัดการการเรียกใช้ซ้ำแบบเจาะจง

เวิร์กโฟลว์ลูกจะถูกสั่งเริ่มจาก ref ที่เชื่อถือได้และตรึงด้วย SHA ซึ่งเรียกใช้ `Full Release Validation` รอบการทำงานลูกทุกตัวต้องใช้ SHA ของเวิร์กโฟลว์แม่ที่ตรงกันทุกประการ อย่าใช้การสั่งเริ่ม `--ref main -f ref=<sha>` โดยตรงสำหรับการพิสูจน์รีลีส ให้ใช้ `pnpm ci:full-release --sha <target-sha> --target-ref release/YYYY.M.PATCH`

ใช้ `release_profile` เพื่อเลือกขอบเขตความครอบคลุมแบบสด/ผู้ให้บริการ:

- `beta`: เส้นทาง OpenAI/แกนหลักแบบสดและ Docker ที่สำคัญต่อรีลีสและเร็วที่สุด
- `stable`: ความครอบคลุมผู้ให้บริการ/แบ็กเอนด์สำหรับเบต้าและเสถียรเพื่ออนุมัติรีลีส
- `full`: แบบเสถียรพร้อมความครอบคลุมผู้ให้บริการ/สื่อเชิงคำแนะนำในวงกว้าง

การตรวจสอบแบบเสถียรและแบบเต็มจะเรียกใช้การทดสอบแบบสด/E2E ที่ครอบคลุมทั้งหมด เส้นทางรีลีส Docker และการตรวจสอบการคงอยู่หลังอัปเกรดของแพ็กเกจที่เผยแพร่แบบมีขอบเขตก่อนการเลื่อนขั้นเสมอ ใช้ `run_release_soak=true` เพื่อร้องขอการตรวจสอบชุดเดียวกันสำหรับเบต้า การตรวจสอบชุดนี้ครอบคลุมแพ็กเกจเสถียรล่าสุดสี่รายการ รวมถึงค่าพื้นฐาน `2026.4.23` และ `2026.5.2` ที่ตรึงไว้ ตลอดจนความครอบคลุม `2026.4.15` ที่เก่ากว่า โดยตัดค่าพื้นฐานที่ซ้ำกันออกและแบ่งแต่ละค่าพื้นฐานเป็นงานรันเนอร์ Docker แยกกัน

`OpenClaw Release Checks` ใช้ ref ของเวิร์กโฟลว์ที่เชื่อถือได้เพื่อแก้ไข ref เป้าหมายหนึ่งครั้งเป็น `release-package-under-test` และนำอาร์ติแฟกต์นั้นกลับมาใช้ในการตรวจสอบข้ามระบบปฏิบัติการ การยอมรับแพ็กเกจ และการตรวจสอบ Docker ตามเส้นทางรีลีสเมื่อมีการเฝ้าระยะยาว วิธีนี้ทำให้กล่องทั้งหมดที่เกี่ยวข้องกับแพ็กเกจใช้ไบต์ชุดเดียวกันและหลีกเลี่ยงการบิลด์แพ็กเกจซ้ำ หลังจากเบต้าอยู่บน npm แล้ว ให้ตั้งค่า `release_package_spec=openclaw@YYYY.M.PATCH-beta.N` เพื่อให้การตรวจสอบรีลีสดาวน์โหลดแพ็กเกจที่จัดส่งแล้วหนึ่งครั้ง ดึง SHA ของต้นทางบิลด์จาก `dist/build-info.json` และนำอาร์ติแฟกต์นั้นกลับมาใช้สำหรับเลนข้ามระบบปฏิบัติการ การยอมรับแพ็กเกจ Docker ตามเส้นทางรีลีส และ Telegram ของแพ็กเกจ

การทดสอบติดตั้ง OpenAI ข้ามระบบปฏิบัติการอย่างรวดเร็วใช้ `OPENCLAW_CROSS_OS_OPENAI_MODEL` เมื่อตั้งค่าตัวแปรของรีโพซิทอรี/องค์กร มิฉะนั้นจะใช้ `openai/gpt-5.6-luna` เนื่องจากเลนนี้พิสูจน์การติดตั้งแพ็กเกจ การเริ่มต้นใช้งาน การเริ่มทำงานของ Gateway และการทำงานของเอเจนต์แบบสดหนึ่งรอบ ไม่ใช่การเปรียบเทียบประสิทธิภาพของโมเดลที่มีความสามารถสูงสุด เมทริกซ์ผู้ให้บริการแบบสดที่กว้างกว่ายังคงเป็นตำแหน่งสำหรับความครอบคลุมเฉพาะโมเดล

ใช้รูปแบบเหล่านี้ตามขั้นตอนของรีลีส:

```bash
# ตรวจสอบ Code SHA ที่ผลิตภัณฑ์เสร็จสมบูรณ์
pnpm ci:full-release \
  --sha <code-sha> \
  --target-ref release/YYYY.M.PATCH

# ตรวจสอบ Release SHA ที่เปลี่ยนเฉพาะ changelog โดยใช้หลักฐานผลิตภัณฑ์จาก Code SHA ซ้ำ
pnpm ci:full-release \
  --sha <release-sha> \
  --target-ref release/YYYY.M.PATCH

# หลังเผยแพร่เบต้า ให้เพิ่ม Telegram E2E สำหรับแพ็กเกจที่เผยแพร่แล้ว
pnpm ci:full-release \
  --sha <release-sha> \
  --target-ref release/YYYY.M.PATCH \
  -f release_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f evidence_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f npm_telegram_provider_mode=mock-openai
```

อย่าใช้ umbrella แบบเต็มเป็นการรันซ้ำครั้งแรกหลังการแก้ไขเฉพาะจุด หากกล่องใดกล่องหนึ่งล้มเหลว ให้ใช้เวิร์กโฟลว์ลูก งาน เลน Docker โปรไฟล์แพ็กเกจ ผู้ให้บริการโมเดล หรือเลน QA ที่ล้มเหลวสำหรับการพิสูจน์ครั้งถัดไป รัน umbrella แบบเต็มอีกครั้งเฉพาะเมื่อการแก้ไขเปลี่ยนการประสานงานรีลีสร่วมกัน หรือทำให้หลักฐานจากทุกกล่องก่อนหน้านี้ล้าสมัย ตัวตรวจสอบขั้นสุดท้ายของ umbrella จะตรวจสอบรหัสการรันของเวิร์กโฟลว์ลูกที่บันทึกไว้อีกครั้ง ดังนั้นหลังจากรันเวิร์กโฟลว์ลูกซ้ำสำเร็จแล้ว ให้รันซ้ำเฉพาะงานแม่ `Verify full validation` ที่ล้มเหลว

`rerun_group=all` อาจใช้การรัน umbrella สีเขียวก่อนหน้านี้ซ้ำได้ เมื่อโปรไฟล์รีลีส

### Vitest

กล่อง Vitest คือเวิร์กโฟลว์ลูกแบบแมนนวล `CI` CI แบบแมนนวลจงใจข้ามการกำหนดขอบเขตตามการเปลี่ยนแปลง และบังคับกราฟการทดสอบปกติสำหรับ release candidate ได้แก่ shard ของ Linux Node, shard ของ Plugin ที่รวมมาด้วย, shard สัญญาของ Plugin และช่องทาง, ความเข้ากันได้กับ Node 22, `check-*`, `check-additional-*`, การตรวจสอบ smoke ของอาร์ติแฟกต์ที่บิลด์แล้ว, การตรวจสอบเอกสาร, Skills ของ Python, Windows, macOS และ i18n ของ Control UI Android จะรวมอยู่เมื่อ `Full Release Validation` รันกล่องนี้ เนื่องจาก umbrella ส่ง `include_android=true`; CI แบบแมนนวลที่รันแยกต้องใช้ `include_android=true` เพื่อให้ครอบคลุม Android

ใช้กล่องนี้เพื่อตอบว่า "ซอร์สทรีผ่านชุดทดสอบปกติแบบเต็มหรือไม่" ซึ่งไม่เหมือนกับการตรวจสอบผลิตภัณฑ์ตามพาธรีลีส หลักฐานที่ต้องเก็บไว้:

- สรุป `Full Release Validation` ที่แสดง URL การรัน `CI` ที่ถูกสั่งให้ทำงาน
- การรัน `CI` เป็นสีเขียวบน target SHA ที่ตรงกันทุกประการ
- ชื่อ shard ที่ล้มเหลวหรือทำงานช้าจากงาน CI เมื่อตรวจสอบรีเกรสชัน
- อาร์ติแฟกต์การจับเวลาของ Vitest เช่น `.artifacts/vitest-shard-timings.json` เมื่อจำเป็นต้องวิเคราะห์ประสิทธิภาพของการรัน

รัน CI แบบแมนนวลโดยตรงเฉพาะเมื่อรีลีสต้องการ CI ปกติที่ให้ผลลัพธ์แน่นอน แต่ไม่ต้องใช้กล่อง Docker, QA Lab, แบบใช้งานจริง, ข้ามระบบปฏิบัติการ หรือแพ็กเกจ ใช้คำสั่งแรกสำหรับ CI โดยตรงที่ไม่รวม Android เพิ่ม `include_android=true` เมื่อ CI ของ release candidate ที่รันโดยตรงต้องครอบคลุม Android:

```bash
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH -f include_android=true
```

### Docker

กล่อง Docker อยู่ใน `OpenClaw Release Checks` ผ่าน `openclaw-live-and-e2e-checks-reusable.yml` รวมถึงเวิร์กโฟลว์โหมดรีลีส `install-smoke` กล่องนี้ตรวจสอบ release candidate ผ่านสภาพแวดล้อม Docker ที่จัดแพ็กเกจแล้ว แทนที่จะใช้เฉพาะการทดสอบระดับซอร์ส

ความครอบคลุม Docker สำหรับรีลีสประกอบด้วย:

- การทดสอบ smoke การติดตั้งแบบเต็ม โดยเปิดใช้การทดสอบ smoke การติดตั้ง Bun แบบส่วนกลางที่ทำงานช้า
- การเตรียม/ใช้ซ้ำอิมเมจ smoke ของ Dockerfile ระดับรากตาม target SHA โดยงาน smoke สำหรับ QR, root/gateway และตัวติดตั้ง/Bun ทำงานเป็น shard การทดสอบ smoke การติดตั้งแยกกัน
- เลน E2E ของรีโพ
- ส่วน Docker ตามพาธรีลีส: `core`, `package-update-openai`, `package-update-anthropic`, `package-update-core`, `plugins-runtime-plugins`, `plugins-runtime-services`, `plugins-runtime-install-a` ถึง `plugins-runtime-install-h` และ `openwebui`
- ความครอบคลุม OpenWebUI บนรันเนอร์ดิสก์ขนาดใหญ่โดยเฉพาะเมื่อมีการร้องขอ
- เลนติดตั้ง/ถอนการติดตั้ง Plugin ที่รวมมาด้วยซึ่งแยกเป็น `bundled-plugin-install-uninstall-0` ถึง `bundled-plugin-install-uninstall-23`
- ชุดทดสอบผู้ให้บริการแบบใช้งานจริง/E2E และความครอบคลุมโมเดลแบบใช้งานจริงใน Docker เมื่อการตรวจสอบรีลีสรวมชุดทดสอบแบบใช้งานจริง

ใช้อาร์ติแฟกต์ Docker ก่อนรันซ้ำ ตัวกำหนดตารางเวลาตามพาธรีลีสจะอัปโหลด `.artifacts/docker-tests/` พร้อมบันทึกของเลน, `summary.json`, `failures.json`, เวลาของแต่ละเฟส, JSON แผนตัวกำหนดตารางเวลา และคำสั่งรันซ้ำ สำหรับการกู้คืนแบบเฉพาะจุด ให้ใช้ `docker_lanes=<lane[,lane]>` บนเวิร์กโฟลว์แบบใช้งานจริง/E2E ที่นำกลับมาใช้ซ้ำได้ แทนการรันส่วนรีลีสทั้งหมดซ้ำ คำสั่งรันซ้ำที่สร้างขึ้นจะรวม `package_artifact_run_id` ก่อนหน้าและอินพุตอิมเมจ Docker ที่เตรียมไว้เมื่อมี เพื่อให้เลนที่ล้มเหลวสามารถใช้ tarball และอิมเมจ GHCR เดิมซ้ำได้

### QA Lab

กล่อง QA Lab เป็นส่วนหนึ่งของ `OpenClaw Release Checks` เช่นกัน กล่องนี้เป็นเกตรีลีสสำหรับพฤติกรรมแบบเอเจนต์และระดับช่องทาง ซึ่งแยกจากกลไกแพ็กเกจของ Vitest และ Docker

ความครอบคลุม QA Lab สำหรับรีลีสประกอบด้วย:

- เลนตรวจสอบความเท่าเทียมแบบจำลอง ซึ่งเปรียบเทียบเลน candidate ของ OpenAI กับ baseline `anthropic/claude-opus-4-8` โดยใช้ชุดตรวจสอบความเท่าเทียมแบบเอเจนต์
- โปรไฟล์รีลีสอะแดปเตอร์แบบใช้งานจริงของ Matrix โดยใช้สภาพแวดล้อม `qa-live-shared`
- เลน QA ของ Telegram แบบใช้งานจริงโดยใช้สัญญาเช่าข้อมูลประจำตัว Convex CI
- `pnpm qa:otel:smoke`, `pnpm qa:otel:collector-smoke`, `pnpm qa:prometheus:smoke` หรือ `pnpm qa:observability:smoke` เมื่อเทเลเมทรีของรีลีสต้องการหลักฐานภายในเครื่องอย่างชัดเจน

ใช้กล่องนี้เพื่อตอบว่า "รีลีสทำงานอย่างถูกต้องในสถานการณ์ QA และโฟลว์ช่องทางแบบใช้งานจริงหรือไม่" เก็บ URL ของอาร์ติแฟกต์สำหรับเลนตรวจสอบความเท่าเทียม, Matrix และ Telegram ไว้เมื่ออนุมัติรีลีส ความครอบคลุม Matrix แบบเต็มยังคงมีให้ใช้งานในรูปแบบการรัน QA-Lab แบบแบ่ง shard ด้วยตนเอง แทนที่จะเป็นเลนสำคัญต่อรีลีสโดยค่าเริ่มต้น

### แพ็กเกจ

กล่อง Package คือเกตสำหรับผลิตภัณฑ์ที่ติดตั้งได้ รองรับโดย `Package Acceptance` และตัวแก้ไข `scripts/resolve-openclaw-package-candidate.mjs` ตัวแก้ไขจะทำให้ candidate อยู่ในรูป tarball `package-under-test` ที่ Docker E2E ใช้ ตรวจสอบรายการภายในแพ็กเกจ บันทึกเวอร์ชันแพ็กเกจและ SHA-256 และแยก ref ของชุดควบคุมเวิร์กโฟลว์ออกจาก ref ซอร์สของแพ็กเกจ

ซอร์ส candidate ที่รองรับ:

- `source=npm`: `openclaw@beta`, `openclaw@latest` หรือเวอร์ชันรีลีส OpenClaw ที่ตรงกันทุกประการ
- `source=ref`: แพ็ก branch, tag หรือ commit SHA แบบเต็มของ `package_ref` ที่เชื่อถือได้ด้วยชุดควบคุม `workflow_ref` ที่เลือก
- `source=url`: ดาวน์โหลด `.tgz` ผ่าน HTTPS สาธารณะ โดยต้องมี `package_sha256`; ข้อมูลประจำตัวใน URL, พอร์ต HTTPS ที่ไม่ใช่ค่าเริ่มต้น, ชื่อโฮสต์หรือแอดเดรสที่แก้ไขแล้วซึ่งเป็นแบบส่วนตัว/ภายใน/ใช้งานเฉพาะ และการเปลี่ยนเส้นทางที่ไม่ปลอดภัยจะถูกปฏิเสธ
- `source=trusted-url`: ดาวน์โหลด `.tgz` ผ่าน HTTPS โดยต้องมี `package_sha256` และ `trusted_source_id` จากนโยบายที่ระบุชื่อใน `.github/package-trusted-sources.json`; ใช้วิธีนี้สำหรับมิเรอร์ระดับองค์กรหรือรีโพแพ็กเกจส่วนตัวที่ผู้ดูแลเป็นเจ้าของ แทนการเพิ่มช่องทางข้ามเครือข่ายส่วนตัวระดับอินพุตให้กับ `source=url`
- `source=artifact`: ใช้ `.tgz` ที่อัปโหลดโดยการรัน GitHub Actions อื่นซ้ำ

`OpenClaw Release Checks` รัน Package Acceptance ด้วย `source=artifact`, อาร์ติแฟกต์แพ็กเกจรีลีสที่เตรียมไว้, `suite_profile=custom`, `docker_lanes=doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor root-managed-vps-upgrade update-restart-auth plugins-offline plugin-update plugin-binding-command-escape`, `telegram_mode=mock-openai` Package Acceptance จะคงการย้ายข้อมูล การอัปเดต การอัปเกรด VPS ที่จัดการโดย root การรีสตาร์ตหลังอัปเดตโดยมีการยืนยันตัวตนที่กำหนดค่าไว้ การติดตั้ง Skill จาก ClawHub แบบใช้งานจริง การล้างการขึ้นต่อกันของ Plugin ที่ล้าสมัย ฟิกซ์เจอร์ Plugin แบบออฟไลน์ การอัปเดต Plugin การเสริมความแข็งแกร่งต่อการหลบหนีการผูกคำสั่ง Plugin และ QA ของแพ็กเกจ Telegram ไว้กับ tarball ที่แก้ไขแล้วรายการเดียวกัน การตรวจสอบรีลีสแบบบล็อกใช้ baseline ของแพ็กเกจที่เผยแพร่ล่าสุดโดยค่าเริ่มต้น; โปรไฟล์เบต้าที่มี `run_release_soak=true`, `release_profile=stable` หรือ `release_profile=full` จะขยายการตรวจสอบผู้รอดจากการอัปเกรดแพ็กเกจที่เผยแพร่แล้วไปยัง `last-stable-4` รวมถึง baseline ที่ปักหมุด `2026.4.23`, `2026.5.2` และ `2026.4.15` พร้อมสถานการณ์ `reported-issues` ใช้ Package Acceptance กับ `source=npm` สำหรับ candidate ที่เผยแพร่แล้ว, `source=ref` สำหรับ tarball npm ภายในเครื่องที่อ้างอิง SHA ก่อนเผยแพร่, `source=trusted-url` สำหรับมิเรอร์ระดับองค์กร/ส่วนตัวที่ผู้ดูแลเป็นเจ้าของ หรือ `source=artifact` สำหรับ tarball ที่เตรียมไว้และอัปโหลดโดยการรัน GitHub Actions อื่น

นี่คือสิ่งทดแทนแบบ GitHub-native สำหรับความครอบคลุมด้านแพ็กเกจ/การอัปเดตส่วนใหญ่ที่ก่อนหน้านี้ต้องใช้ Parallels การตรวจสอบรีลีสข้ามระบบปฏิบัติการยังคงสำคัญสำหรับการเริ่มต้นใช้งาน ตัวติดตั้ง และพฤติกรรมเฉพาะแพลตฟอร์ม แต่การตรวจสอบผลิตภัณฑ์ด้านแพ็กเกจ/การอัปเดตควรเลือกใช้ Package Acceptance

รายการตรวจสอบมาตรฐานสำหรับการตรวจสอบการอัปเดตและ Plugin คือ [การทดสอบการอัปเดตและ Plugin](/th/help/testing-updates-plugins) ใช้รายการนี้เมื่อตัดสินใจว่าเลนภายในเครื่อง, Docker, Package Acceptance หรือการตรวจสอบรีลีสใดพิสูจน์การติดตั้ง/อัปเดต Plugin, การล้างโดย doctor หรือการเปลี่ยนแปลงการย้ายข้อมูลแพ็กเกจที่เผยแพร่แล้ว การย้ายข้อมูลการอัปเดตที่เผยแพร่แล้วแบบครบถ้วนจากทุกแพ็กเกจ stable `2026.4.23+` เป็นเวิร์กโฟลว์แบบแมนนวล `Update Migration` แยกต่างหาก ไม่ใช่ส่วนหนึ่งของ Full Release CI

การผ่อนปรนสำหรับ package-acceptance แบบเดิมถูกจำกัดเวลาโดยเจตนา แพ็กเกจจนถึง `2026.4.25` อาจใช้พาธความเข้ากันได้สำหรับช่องว่างของเมทาดาทาที่เผยแพร่ไปยัง npm แล้ว ได้แก่ รายการคลัง QA ส่วนตัวที่ขาดหายไปจาก tarball, การขาด `gateway install --wrapper`, ไฟล์แพตช์ที่ขาดหายไปในฟิกซ์เจอร์ git ที่สร้างจาก tarball, การขาด `update.channel` ที่คงอยู่, ตำแหน่งบันทึกการติดตั้ง Plugin แบบเดิม, การไม่คงบันทึกการติดตั้ง marketplace และการย้ายข้อมูลเมทาดาทาการกำหนดค่าระหว่าง `plugins update` แพ็กเกจ `2026.4.26` ที่เผยแพร่แล้วอาจเตือนเกี่ยวกับไฟล์ตราประทับเมทาดาทาของบิลด์ภายในเครื่องที่จัดส่งไปแล้ว แพ็กเกจหลังจากนั้นต้องเป็นไปตามสัญญาแพ็กเกจสมัยใหม่ ช่องว่างเดียวกันเหล่านั้นจะทำให้การตรวจสอบรีลีสล้มเหลว

ใช้โปรไฟล์ Package Acceptance ที่กว้างขึ้นเมื่อคำถามเกี่ยวกับรีลีสเป็นเรื่องแพ็กเกจที่ติดตั้งได้จริง:

```bash
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f published_upgrade_survivor_baseline=openclaw@2026.4.26
```

โปรไฟล์แพ็กเกจที่ใช้บ่อย:

- `smoke`: ช่องทางตรวจสอบด่วนสำหรับการติดตั้งแพ็กเกจ/ช่องทาง/เอเจนต์ เครือข่าย Gateway และการโหลดการกำหนดค่าใหม่
- `package`: สัญญาการติดตั้ง/อัปเดต/รีสตาร์ต/แพ็กเกจ Plugin พร้อมหลักฐานการติดตั้ง Skills จาก ClawHub แบบใช้งานจริง; นี่เป็นค่าเริ่มต้นสำหรับการตรวจสอบรีลีส
- `product`: `package` รวมถึงช่องทาง MCP, การล้างข้อมูล cron/เอเจนต์ย่อย, การค้นหาเว็บด้วย OpenAI และ OpenWebUI
- `full`: ส่วนย่อยของเส้นทางรีลีส Docker พร้อม OpenWebUI
- `custom`: รายการ `docker_lanes` ที่ตรงกันทุกประการสำหรับการเรียกใช้งานซ้ำแบบเจาะจง

สำหรับหลักฐาน Telegram ของแพ็กเกจตัวเลือก ให้เปิดใช้ `telegram_mode=mock-openai` หรือ `telegram_mode=live-frontier` ใน Package Acceptance เวิร์กโฟลว์จะส่ง tarball `package-under-test` ที่แก้ค่าแล้วไปยังช่องทางตรวจสอบ Telegram; เวิร์กโฟลว์ Telegram แบบสแตนด์อโลนยังคงรับสเปก npm ที่เผยแพร่แล้วสำหรับการตรวจสอบหลังเผยแพร่

## ระบบอัตโนมัติสำหรับเผยแพร่รีลีสปกติ

สำหรับการเผยแพร่เบตา, `latest`, Plugin, GitHub Release และแพลตฟอร์ม
`OpenClaw Release Publish` คือจุดเริ่มต้นปกติที่มีการเปลี่ยนแปลงข้อมูล เส้นทาง extended-stable รายเดือนแบบ
npm เท่านั้นของ `.33+` ไม่ใช้ตัวประสานงานนี้
เวิร์กโฟลว์ปกติจะประสานเวิร์กโฟลว์ trusted publisher ตามลำดับที่
รีลีสต้องใช้:

1. เช็กเอาต์แท็กรีลีสและหาค่า SHA ของคอมมิต
2. ตรวจสอบว่าสามารถเข้าถึงแท็กได้จาก `main` หรือ `release/*` (หรือสาขาอัลฟาของ Tideclaw สำหรับอัลฟาพรีรีลีส)
3. เรียกใช้ `pnpm plugins:sync:check`
4. ส่งงานไปยัง `Plugin NPM Release` พร้อม `publish_scope=all-publishable` และ `ref=<release-sha>`
5. ส่งงานไปยัง `Plugin ClawHub Release` โดยใช้ขอบเขตและ SHA เดียวกัน
6. ส่งงานไปยัง `OpenClaw NPM Release` พร้อมแท็กรีลีส, npm dist-tag และ `preflight_run_id` ที่บันทึกไว้ หลังตรวจสอบ `full_release_validation_run_id` ที่บันทึกไว้และครั้งที่เรียกใช้งานที่ตรงกันทุกประการ
7. สำหรับรีลีสเสถียร ให้สร้างหรืออัปเดต GitHub release เป็นฉบับร่าง ส่งงานไปยัง `Windows Node Release` พร้อม `windows_node_tag` ที่ระบุชัดเจนและ `windows_node_installer_digests` ที่ตัวเลือกได้รับอนุมัติแล้ว และตรวจสอบแอสเซ็ตตัวติดตั้ง/เช็กซัม Windows ที่เป็นมาตรฐาน นอกจากนี้ ให้ส่งงานไปยัง `Android Release` เพื่อสร้าง APK ที่ลงนามแล้วจากแท็กที่ตรงกันทุกประการ พร้อมเช็กซัมและที่มาที่ตรวจสอบได้ ตรวจสอบสัญญาของแอสเซ็ตเนทีฟทั้งสองก่อนเผยแพร่ฉบับร่าง

ตัวอย่างการเผยแพร่เบตา:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH-beta.N \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

เผยแพร่รีลีสเสถียรไปยัง dist-tag เบตาเริ่มต้น:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

การโปรโมตรีลีสเสถียรไปยัง `latest` โดยตรงต้องระบุอย่างชัดเจน:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=latest
```

ใช้เวิร์กโฟลว์ระดับล่าง `Plugin NPM Release` และ `Plugin ClawHub Release` เฉพาะสำหรับงานซ่อมแซมหรือเผยแพร่ซ้ำแบบเจาะจงเท่านั้น `OpenClaw Release Publish` จะปฏิเสธ `plugin_publish_scope=selected` เมื่อ `publish_openclaw_npm=true` เพื่อไม่ให้แพ็กเกจหลักเผยแพร่โดยขาด Plugin อย่างเป็นทางการที่เผยแพร่ได้แม้แต่รายการเดียว รวมถึง `@openclaw/diffs-language-pack` สำหรับการซ่อมแซม Plugin ที่เลือก ให้ตั้งค่า `publish_openclaw_npm=false` พร้อม `plugin_publish_scope=selected` และ `plugins=@openclaw/name` หรือส่งงานไปยังเวิร์กโฟลว์ลูกโดยตรง

การบูตสแตรป ClawHub สำหรับการเผยแพร่ครั้งแรกเป็นข้อยกเว้น: ส่งงานไปยัง `Plugin ClawHub New`
จาก `main` ที่เชื่อถือได้ และส่ง SHA ของรีลีสเป้าหมายแบบเต็มผ่าน `ref`
ห้ามเรียกใช้เวิร์กโฟลว์บูตสแตรปจากแท็กหรือสาขารีลีสโดยตรง:

```bash
gh workflow run plugin-clawhub-new.yml \
  --ref main \
  -f plugins=@openclaw/name \
  -f ref=<full-40-character-release-sha> \
  -f pretag_validation=true \
  -f dry_run=true
```

การตรวจสอบก่อนติดแท็กต้องใช้ `dry_run=true`, ปฏิเสธอินพุตแท็กรีลีสและการเรียกใช้งานแม่
และยอมรับเฉพาะเป้าหมายที่ตรงกันทุกประการซึ่งเข้าถึงได้จาก `main` หรือ `release/*`
กระบวนการนี้จะไม่โหลดข้อมูลประจำตัว ClawHub, เผยแพร่ไบต์ของแพ็กเกจ หรือเปลี่ยนการกำหนดค่า
trusted publisher เวิร์กโฟลว์ยังคงแก้ค่าแผนรีจิสทรีที่ใช้งานจริง
เช็กเอาต์และแพ็กเป้าหมายเฉพาะในงานที่ไม่มีข้อมูลลับ จัดเตรียม
ชุดเครื่องมือ ClawHub ที่ล็อกไว้ และตรวจสอบอาร์ติแฟกต์ที่เปลี่ยนแปลงไม่ได้ รวมถึง
slug/ข้อมูลระบุตัวตนของแพ็กเกจก่อนที่แท็กรีลีสจะมีอยู่ อนุมัติสภาพแวดล้อม
`clawhub-plugin-bootstrap` หลังจากงานแพ็กที่ไม่มีข้อมูลลับ
เสร็จสิ้นแล้วเท่านั้น; งานตรวจสอบที่มีการป้องกันนี้ไม่มีข้อมูลประจำตัวหรือคำสั่งเปลี่ยนแปลงข้อมูล

การทดลองทำงานที่ได้รับอนุมัติหรือการบูตสแตรปจริงหลังติดแท็กต้องระบุ
แท็กรีลีสที่ตรงกันทุกประการ พร้อมรหัสการเรียกใช้งาน `OpenClaw Release Publish`, ครั้งที่เรียกใช้งาน และ
สาขาของงานแม่ งานแม่รับรอง SHA ของเวิร์กโฟลว์ตนเองและ SHA ของ `main` ที่เชื่อถือได้และตรงกันทุกประการอีกค่าหนึ่ง
สำหรับ `Plugin ClawHub New`; การเรียกใช้งานลูกและการอนุมัติ
สภาพแวดล้อมที่มีการป้องกันทุกรายการต้องตรงกับ SHA ของงานลูกที่ได้รับอนุมัตินั้น แท็กรีลีสจะถูก
ตรวจสอบซ้ำก่อนการพยายามเผยแพร่และเปลี่ยนแปลง trusted publisher ทุกครั้ง

งานแพ็ก
อัปโหลดอาร์ติแฟกต์ที่เปลี่ยนแปลงไม่ได้หนึ่งรายการ โดยชื่อ, ID/ไดเจสต์อาร์ติแฟกต์ Actions,
การเรียกใช้งาน/ครั้งที่เรียกใช้งานของผู้สร้าง, SHA เป้าหมาย และ SHA-256/ขนาดของ tarball
ต่อแพ็กเกจ จะถูกส่งต่อไปยังงานตรวจสอบและงานที่มีการป้องกัน งานที่มีการป้องกันจะเช็กเอาต์เฉพาะ
เครื่องมือ `main` ที่เชื่อถือได้ ตรวจสอบทูเพิลอาร์ติแฟกต์ผ่าน GitHub API ดาวน์โหลด
ด้วย ID อาร์ติแฟกต์ที่ตรงกันทุกประการ คำนวณแฮช tarball ทุกไฟล์ใหม่ และตรวจสอบเส้นทาง TAR ภายใน
รวมถึงข้อมูลระบุตัวตนของแพ็กเกจตามกฎการทำให้ USTAR เป็นมาตรฐานของ CLI ที่ตรึงเวอร์ชันไว้ จากนั้น
ตัวเลือกทุกรายการต้องผ่านการทดลองเผยแพร่ของ CLI ที่ตรึงเวอร์ชันไว้ ซึ่งจะคืนค่าก่อน
การค้นหารีจิสทรีหรือการยืนยันตัวตน ตัวกรองเบื้องต้นของงานข้อมูลประจำตัวจำกัด ClawPacks ที่บีบอัดแล้ว
ไว้ที่ 120 MiB, เพย์โหลดไฟล์รวมที่ 50 MiB, ข้อมูล TAR ที่ขยายแล้วที่ 64 MiB และ
จำนวนรายการ TAR ที่ 10,000 การซ่อมแซม trusted publisher ของแพ็กเกจที่มีอยู่แล้ว
ยังคงเป็นการกำหนดค่าเท่านั้น แต่ยังคงแพ็กเป้าหมายและกำหนดให้แท็กที่ร้องขอ
ตรงกับไบต์และข้อมูลเมตาของรีจิสทรีทุกประการก่อนเปลี่ยนการกำหนดค่า trusted publisher
การตรวจสอบหลังเผยแพร่จะดาวน์โหลดอาร์ติแฟกต์ ClawHub และ
กำหนดให้ SHA-256 และขนาดตรงกัน การกู้คืนด้วยการเรียกใช้ซ้ำเฉพาะงานที่ล้มเหลวอาจนำอาร์ติแฟกต์แพ็กเกจ
จากครั้งก่อนกลับมาใช้ได้เฉพาะเมื่อ งานผู้สร้างที่ตรงกันทุกประการเสร็จสมบูรณ์
สำเร็จแล้ว หลักฐานสุดท้ายยังผูกกับเวอร์ชัน ClawHub ที่ล็อกไว้, SHA-256 ของล็อก
และความสมบูรณ์ของ npm หากไม่ตรงกันต้องใช้แพ็กเกจเวอร์ชันใหม่

## อินพุตของเวิร์กโฟลว์ NPM

`OpenClaw NPM Release` รับอินพุตที่ผู้ปฏิบัติงานควบคุมดังต่อไปนี้:

- `tag`: แท็กรีลีสที่จำเป็น เช่น `v2026.4.2`, `v2026.4.2-1`, `v2026.4.2-beta.1` หรือ `v2026.4.2-alpha.1`; เมื่อ `preflight_only=true` ค่านี้อาจเป็น SHA คอมมิต 40 อักขระแบบเต็มปัจจุบันของสาขาเวิร์กโฟลว์สำหรับการตรวจสอบก่อนดำเนินการเท่านั้น
- `preflight_only`: `true` สำหรับการตรวจสอบ/บิลด์/แพ็กเกจเท่านั้น, `false` สำหรับเส้นทางเผยแพร่จริง
- `preflight_run_id`: รหัสการเรียกใช้งานก่อนดำเนินการที่มีอยู่และสำเร็จแล้ว ซึ่งจำเป็นในเส้นทางเผยแพร่จริง เพื่อให้เวิร์กโฟลว์นำ tarball ที่เตรียมไว้กลับมาใช้แทนการสร้างใหม่
- `full_release_validation_run_id`: รหัสการเรียกใช้ `Full Release Validation` ที่สำเร็จสำหรับแท็ก/SHA นี้ ซึ่งจำเป็นสำหรับการเผยแพร่จริง การเผยแพร่เบตาอาจดำเนินการต่อโดยมีเพียงขั้นตอนก่อนดำเนินการพร้อมคำเตือน แต่การโปรโมตรีลีสเสถียร/`latest` ยังคงต้องใช้ค่านี้
- `full_release_validation_run_attempt`: ครั้งที่เรียกใช้งานซึ่งเป็นจำนวนบวกและตรงกันทุกประการที่จับคู่กับ `full_release_validation_run_id`; จำเป็นเมื่อมีการระบุรหัสการเรียกใช้งาน เพื่อไม่ให้การเรียกใช้ซ้ำเปลี่ยนหลักฐานการอนุญาตระหว่างการเผยแพร่
- `release_publish_run_id`: รหัสการเรียกใช้ `OpenClaw Release Publish` ที่ได้รับอนุมัติ; จำเป็นเมื่อเวิร์กโฟลว์นี้ถูกส่งงานโดยงานแม่นั้น (การเรียกเผยแพร่จริงโดยแอ็กเตอร์บอต)
- `plugin_npm_run_id`: รหัสการเรียกใช้ `Plugin NPM Release` แบบตรงกับ HEAD ทุกประการที่สำเร็จ; จำเป็นสำหรับการเผยแพร่แพ็กเกจหลัก `extended-stable` จริง
- `npm_dist_tag`: แท็กเป้าหมาย npm สำหรับเส้นทางเผยแพร่; รับค่า `alpha`, `beta`, `latest` หรือ `extended-stable` และมีค่าเริ่มต้นเป็น `beta` แพตช์สุดท้าย `33` และใหม่กว่าต้องใช้ `extended-stable`; โดยค่าเริ่มต้น `extended-stable` จะปฏิเสธแพตช์ก่อนหน้า และจะปฏิเสธแท็กที่ไม่ใช่แท็กสุดท้ายเสมอ
- `bypass_extended_stable_guard`: บูลีนสำหรับการทดสอบเท่านั้น มีค่าเริ่มต้นเป็น `false`; เมื่อใช้กับ `npm_dist_tag=extended-stable` จะข้ามเงื่อนไขสิทธิ์ extended-stable รายเดือน โดยยังคงการตรวจสอบข้อมูลระบุรีลีส, อาร์ติแฟกต์, การอนุมัติ และการอ่านค่ากลับไว้

`Plugin NPM Release` รับ `npm_dist_tag=default` สำหรับพฤติกรรมรีลีส
ที่มีอยู่ หรือ `npm_dist_tag=extended-stable` สำหรับเส้นทางรายเดือนที่มีการป้องกัน ตัวเลือก
extended-stable ต้องใช้ `publish_scope=all-publishable`, อินพุต
`plugins` ที่ว่างเปล่า, แพตช์สุดท้ายที่มีค่าตั้งแต่ `33` ขึ้นไป และสาขามาตรฐาน
`extended-stable/YYYY.M.33` ที่ตำแหน่งปลายตรงกันทุกประการ เส้นทางนี้จะไม่ย้าย
`latest` หรือ `beta` ของ Plugin แพ็กเกจเวอร์ชันใหม่จะได้รับ `extended-stable` แบบอะตอม
ผ่านการเผยแพร่แบบเชื่อถือได้ด้วย OIDC (`npm publish --tag extended-stable`);
เวิร์กโฟลว์ต้นทางนี้ไม่ใช้ `npm dist-tag add` ที่ยืนยันตัวตนด้วยโทเค็น การลองใหม่
จะข้ามเวอร์ชันที่ตรงกันทุกประการซึ่งมีอยู่แล้วใน npm จากนั้นหยุดแบบปิด เว้นแต่
การอ่านค่ากลับอย่างสมบูรณ์จะยืนยันว่าแพ็กเกจที่ตรงกันทุกประการทั้งหมดและแท็ก `extended-stable`
สอดคล้องกันแล้ว

`OpenClaw Release Publish` รับอินพุตที่ผู้ปฏิบัติงานควบคุมดังต่อไปนี้:

- `tag`: แท็กรีลีสที่จำเป็น; ต้องมีอยู่แล้ว
- `preflight_run_id`: รหัสการเรียกใช้ก่อนดำเนินการ `OpenClaw NPM Release` ที่สำเร็จ; จำเป็นเมื่อ `publish_openclaw_npm=true` หรือ `plugin_publish_scope=all-publishable`
- `full_release_validation_run_id`: รหัสการเรียกใช้ `Full Release Validation` ที่สำเร็จ; จำเป็นเมื่อ `publish_openclaw_npm=true` หรือ `plugin_publish_scope=all-publishable`
- `full_release_validation_run_attempt`: ครั้งที่เรียกใช้งานซึ่งเป็นจำนวนบวกและตรงกันทุกประการที่จับคู่กับ `full_release_validation_run_id`; จำเป็นเมื่อมีการระบุรหัสการเรียกใช้งาน
- `windows_node_tag`: แท็กรีลีส `openclaw/openclaw-windows-node` ที่ไม่ใช่พรีรีลีสและตรงกันทุกประการ; จำเป็นสำหรับการเผยแพร่ OpenClaw แบบเสถียร
- `windows_node_installer_digests`: แมป JSON แบบกะทัดรัดที่ตัวเลือกได้รับอนุมัติแล้ว ซึ่งแมปชื่อตัวติดตั้ง Windows ปัจจุบันกับไดเจสต์ `sha256:` ที่ตรึงไว้; จำเป็นสำหรับการเผยแพร่ OpenClaw แบบเสถียร
- `npm_telegram_run_id`: รหัสการเรียกใช้ `NPM Telegram Beta E2E` ที่สำเร็จซึ่งระบุหรือไม่ก็ได้ เพื่อนำไปรวมในหลักฐานรีลีสสุดท้าย
- `npm_dist_tag`: แท็กเป้าหมาย npm สำหรับแพ็กเกจ OpenClaw โดยเป็นหนึ่งใน `alpha`, `beta` หรือ `latest`
- `plugin_publish_scope`: มีค่าเริ่มต้นเป็น `all-publishable`; ใช้ `selected` เฉพาะสำหรับงานซ่อมแซมที่จำกัดเฉพาะ Plugin แบบเจาะจงพร้อม `publish_openclaw_npm=false`
- `plugins`: ชื่อแพ็กเกจ `@openclaw/*` ที่คั่นด้วยจุลภาคเมื่อ `plugin_publish_scope=selected`
- `publish_openclaw_npm`: มีค่าเริ่มต้นเป็น `true`; ตั้งค่า `false` เฉพาะเมื่อใช้เวิร์กโฟลว์เป็นตัวประสานงานซ่อมแซมเฉพาะ Plugin
- `release_profile`: โปรไฟล์ความครอบคลุมของรีลีสที่ใช้สำหรับสรุปหลักฐานรีลีส; มีค่าเริ่มต้นเป็น `from-validation` ซึ่งอ่านค่าจากแมนิเฟสต์การตรวจสอบ หรือแทนที่ด้วย `beta`, `stable` หรือ `full`
- `wait_for_clawhub`: มีค่าเริ่มต้นเป็น `false` เพื่อไม่ให้ความพร้อมใช้งานของ npm ถูกบล็อกโดยไซด์คาร์ ClawHub; ตั้งค่า `true` เฉพาะเมื่อการเสร็จสิ้นเวิร์กโฟลว์ต้องรวมการเสร็จสิ้นของ ClawHub

`OpenClaw Release Checks` รับอินพุตที่ผู้ปฏิบัติงานควบคุมดังต่อไปนี้:

- `ref`: แบรนช์ แท็ก หรือ commit SHA แบบเต็มที่จะตรวจสอบ การตรวจสอบที่ใช้ข้อมูลลับกำหนดให้ commit ที่แก้ค่าแล้วต้องเข้าถึงได้จากแบรนช์หรือแท็กรีลีสของ OpenClaw
- `run_release_soak`: เลือกใช้การตรวจสอบแบบ live/E2E อย่างครอบคลุม เส้นทางรีลีส Docker และการทดสอบแบบ soak ของ upgrade-survivor ทั้งหมดนับตั้งแต่ต้น สำหรับการตรวจสอบรีลีสเบต้า ตัวเลือกนี้จะถูกบังคับเปิดโดย `release_profile=stable` และ `release_profile=full`

กฎ:

- เวอร์ชันสุดท้ายปกติและเวอร์ชันแก้ไขที่มีค่า patch ต่ำกว่า `33` อาจเผยแพร่ไปยัง `beta` หรือ `latest` ก็ได้ เวอร์ชันสุดท้ายที่มีค่า patch ตั้งแต่ `33` ขึ้นไปต้องเผยแพร่ไปยัง `extended-stable` และเวอร์ชันที่มีส่วนต่อท้ายสำหรับการแก้ไข ณ ขอบเขตดังกล่าวจะถูกปฏิเสธ
- แท็ก prerelease แบบเบต้าเผยแพร่ได้เฉพาะไปยัง `beta`; แท็ก prerelease แบบแอลฟาเผยแพร่ได้เฉพาะไปยัง `alpha`
- สำหรับ `OpenClaw NPM Release` อนุญาตให้ป้อน commit SHA แบบเต็มเฉพาะเมื่อ `preflight_only=true`
- `OpenClaw Release Checks` และ `Full Release Validation` ใช้สำหรับการตรวจสอบเท่านั้นเสมอ
- เส้นทางเผยแพร่จริงต้องใช้ `npm_dist_tag` เดียวกับที่ใช้ระหว่าง preflight โดยเวิร์กโฟลว์จะตรวจสอบ metadata ดังกล่าวก่อนดำเนินการเผยแพร่ต่อ

## ลำดับการรีลีสเบต้าปกติ/เสถียรล่าสุด

ลำดับแบบเดิมนี้ใช้สำหรับรีลีสปกติที่จัดการแบบประสานงาน ซึ่งรับผิดชอบ Plugin, GitHub Release, Windows และงานแพลตฟอร์มอื่นๆ ด้วย ไม่ใช่เส้นทาง extended-stable แบบ npm-only รายเดือน `.33+` ที่อธิบายไว้ด้านบนของหน้านี้

เมื่อสร้างรีลีสเสถียรปกติที่จัดการแบบประสานงาน:

1. เรียกใช้ `OpenClaw NPM Release` พร้อม `preflight_only=true` ก่อนที่จะมีแท็ก คุณอาจใช้ commit SHA ปัจจุบันแบบเต็มของแบรนช์เวิร์กโฟลว์เพื่อ dry run เวิร์กโฟลว์ preflight สำหรับการตรวจสอบเท่านั้น
2. เลือก `npm_dist_tag=beta` สำหรับขั้นตอนปกติที่เริ่มจากเบต้าก่อน หรือเลือก `latest` เฉพาะเมื่อตั้งใจเผยแพร่เวอร์ชันเสถียรโดยตรง
3. เรียกใช้ `Full Release Validation` บนแบรนช์รีลีส แท็กรีลีส หรือ commit SHA แบบเต็ม เมื่อต้องการ CI ปกติพร้อมการครอบคลุม live prompt cache, Docker, QA Lab, Matrix และ Telegram จากเวิร์กโฟลว์แบบแมนนวลเดียว หากตั้งใจต้องการเฉพาะกราฟการทดสอบปกติแบบกำหนดผลลัพธ์แน่นอน ให้เรียกใช้เวิร์กโฟลว์ `CI` แบบแมนนวลบน ref ของรีลีสแทน
4. เลือกแท็กรีลีส `openclaw/openclaw-windows-node` ที่ไม่ใช่ prerelease ให้ตรงกับตัวติดตั้ง x64 และ ARM64 ที่ลงนามแล้วซึ่งควรเผยแพร่ บันทึกเป็น `windows_node_tag` และบันทึกแผนที่ digest ที่ผ่านการตรวจสอบของตัวติดตั้งเหล่านั้นเป็น `windows_node_installer_digests` ตัวช่วย release-candidate จะบันทึกทั้งสองค่าและรวมไว้ในคำสั่งเผยแพร่ที่สร้างขึ้น
5. บันทึก `preflight_run_id`, `full_release_validation_run_id` ที่สำเร็จ และ `full_release_validation_run_attempt` ที่ตรงกันทุกประการ
6. เรียกใช้ `OpenClaw Release Publish` จาก `main` ที่เชื่อถือได้ โดยใช้ `tag` เดิม `npm_dist_tag` เดิม `windows_node_tag` ที่เลือก `windows_node_installer_digests` ที่บันทึกไว้ของรายการนั้น `preflight_run_id` ที่บันทึกไว้ `full_release_validation_run_id` และ `full_release_validation_run_attempt` ขั้นตอนนี้จะเผยแพร่ Plugin ที่แยกออกเป็นภายนอกไปยัง npm และ ClawHub ก่อนโปรโมตแพ็กเกจ npm ของ OpenClaw
7. หากรีลีสเผยแพร่ไปยัง `beta` ให้ใช้เวิร์กโฟลว์ `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml` เพื่อโปรโมตเวอร์ชันเสถียรนั้นจาก `beta` ไปยัง `latest`
8. หากตั้งใจเผยแพร่รีลีสโดยตรงไปยัง `latest` และต้องการให้ `beta` ชี้ไปยังบิลด์เสถียรเดียวกันทันที ให้ใช้เวิร์กโฟลว์รีลีสเดียวกันเพื่อชี้ dist-tag ทั้งสองไปยังเวอร์ชันเสถียร หรือปล่อยให้การซิงค์ซ่อมแซมตัวเองตามกำหนดการย้าย `beta` ในภายหลัง

การแก้ไข dist-tag อยู่ใน repo บัญชีแยกประเภทรีลีส เนื่องจากยังคงต้องใช้ `NPM_TOKEN` ขณะที่ repo ซอร์สยังคงใช้การเผยแพร่แบบ OIDC-only วิธีนี้ทำให้ทั้งเส้นทางเผยแพร่โดยตรงและเส้นทางโปรโมตที่เริ่มจากเบต้าก่อนได้รับการบันทึกไว้ในเอกสารและมองเห็นได้โดยผู้ปฏิบัติงาน

หากผู้ดูแลต้องเปลี่ยนไปใช้การยืนยันตัวตน npm ภายในเครื่อง ให้เรียกใช้คำสั่ง 1Password CLI (`op`) ภายในเซสชัน tmux เฉพาะเท่านั้น อย่าเรียก `op` โดยตรงจากเชลล์หลักของเอเจนต์ การเก็บไว้ภายใน tmux ทำให้สามารถสังเกต prompt, การแจ้งเตือน และการจัดการ OTP ได้ และป้องกันการแจ้งเตือนบนโฮสต์ซ้ำๆ
