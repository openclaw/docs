หน้านี้รวบรวมหลักฐานเบื้องหลังการปรับปรุงประสิทธิภาพ ขนาดแพ็กเกจ การขึ้นต่อกัน และ shrinkwrap ของ OpenClaw ในเดือนพฤษภาคม 2026 โดยเป็นเอกสารประกอบเชิงเทคนิค
สำหรับบล็อกโพสต์สาธารณะ

การตรวจสอบสองชุดถูกรวมไว้ที่นี่:

- **การตรวจสอบประสิทธิภาพของรีลีส:** GitHub Releases ตั้งแต่ `v2026.5.28` ย้อนกลับไปถึง
  รุ่นเสถียร `v2026.4.23` โดยใช้เวิร์กโฟลว์ `OpenClaw Performance`,
  `profile=smoke` และเลนผู้ให้บริการจำลอง แถวของแท็กส่วนใหญ่ใช้ตัวอย่างหนึ่งครั้ง ส่วนแถว
  `v2026.5.27` และ `v2026.5.28` ใช้อาร์ติแฟกต์ล่าสุดจากสาขารีลีสที่ทำซ้ำ 3 ครั้ง
- **บริบทก่อนหน้าในเดือนเมษายน:** ค่าฐานของผู้ให้บริการจำลองจาก `clawgrit-reports`
  ที่เผยแพร่แล้ว ตั้งแต่ `v2026.4.1` ถึง `v2026.5.2` ใช้เพียงเพื่อหลีกเลี่ยงการนำ
  รีลีสปลายเดือนเมษายนที่เสียหายมาเป็นค่าฐานประสิทธิภาพสาธารณะ
- **การตรวจสอบพื้นที่ติดตั้ง:** ติดตั้งใหม่ด้วย `npm install --ignore-scripts`
  ลงในแพ็กเกจชั่วคราว โดยใช้ `du -sk node_modules` สำหรับขนาด และสำรวจ
  `node_modules` เพื่อนับอินสแตนซ์แพ็กเกจ
- **การตรวจสอบขนาดแพ็กเกจ npm:** ใช้ `npm pack openclaw@<version> --dry-run --json`
  กับรีลีสที่เผยแพร่แล้ว โดยบันทึกขนาด tarball ที่บีบอัดแล้ว ขนาดเมื่อแตกแพ็กเกจ และ
  จำนวนไฟล์

<Warning>
การตรวจสอบประสิทธิภาพหลักใช้ตัวอย่าง smoke หนึ่งครั้งต่อแท็ก ยกเว้นแถว
`v2026.5.27` และ `v2026.5.28` ซึ่งใช้อาร์ติแฟกต์ล่าสุดจากสาขารีลีส

### ขอบเขตของ Shrinkwrap

`2026.5.20` เผยแพร่โดยไม่มี Shrinkwrap ระดับรากและไม่มีแผนผังการพึ่งพา OpenClaw
แบบซ้อนขนาดใหญ่ `2026.5.22` เพิ่ม Shrinkwrap ระดับรากและติดตั้งข้อมูล 911.8MB
ภายใต้ `openclaw/node_modules` ที่ซ้อนอยู่ `2026.5.28` ยังคงใช้ Shrinkwrap และยังคง
ติดตั้งข้อมูล 259.7MiB ภายใต้ `openclaw/node_modules` ที่ซ้อนอยู่ แต่ไม่ติดตั้ง
แพ็กเกจ `@napi-rs/canvas` ใด ๆ ในการตรวจสอบการติดตั้งใหม่ภายในเครื่องอีกต่อไป

ความแตกต่างที่สำคัญคือ: **ตัว Shrinkwrap เองไม่ใช่ปัญหา**

ภายในเครื่องอีกต่อไป

สำหรับคำอธิบาย Shrinkwrap ที่เข้าใจง่ายและการตรวจสอบแพ็กเกจระดับผู้ดูแล

## การตีความด้านห่วงโซ่อุปทาน

จำนวนการพึ่งพาเป็นตัวชี้วัดความปลอดภัยในการปฏิบัติงาน ไม่ใช่เพียงตัวชี้วัด
ขนาดการติดตั้ง แต่ละแพ็กเกจเพิ่มจำนวนผู้ดูแล, Tarball, การอัปเดตแบบสืบทอด,
ไบนารีเนทีฟที่เป็นทางเลือก และพฤติกรรมระหว่างการติดตั้งที่ผู้ปฏิบัติงาน
ต้องให้ความไว้วางใจ

แนวทางการปรับปรุงคือ:

- แยกความสามารถที่มีขนาดใหญ่และเป็นทางเลือกออกจากการติดตั้งแกนหลักเริ่มต้น
- ให้แพ็กเกจ Plugin เป็นเจ้าของกราฟการพึ่งพาขณะทำงานของตนเอง
- หลีกเลี่ยงการซ่อมแซมด้วยตัวจัดการแพ็กเกจขณะทำงานระหว่างการเริ่มต้น Gateway
- รักษาการติดตั้งให้กำหนดผลลัพธ์ได้แน่นอนโดยไม่ทำให้แพ็กเกจเนทีฟสำหรับทุกแพลตฟอร์ม
  ถูกสร้างขึ้นจริง
- ปิดใช้งานสคริปต์การติดตั้งในเส้นทางการยอมรับและการวัดผลแพ็กเกจ
- ตรวจจับแผนผังการพึ่งพาแบบซ้อนและการเพิ่มขึ้นอย่างมากของการพึ่งพาเนทีฟที่เป็นทางเลือกก่อน
  เผยแพร่
