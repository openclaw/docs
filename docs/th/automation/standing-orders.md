คำสั่งถาวรมอบ**อำนาจการปฏิบัติงานถาวร**ให้เอเจนต์ของคุณสำหรับโปรแกรมที่กำหนดไว้ แทนที่จะต้องให้คำสั่งงานรายครั้งทุกครั้ง คุณจะกำหนดโปรแกรมพร้อมขอบเขต ทริกเกอร์ และกฎการยกระดับที่ชัดเจน — แล้วเอเจนต์จะดำเนินการโดยอัตโนมัติภายในขอบเขตเหล่านั้น

นี่คือความแตกต่างระหว่างการบอกผู้ช่วยของคุณว่า "ส่งรายงานประจำสัปดาห์" ทุกวันศุกร์ กับการมอบอำนาจแบบคำสั่งถาวรว่า "คุณรับผิดชอบรายงานประจำสัปดาห์ รวบรวมทุกวันศุกร์ ส่งออกไป และยกระดับมาเฉพาะเมื่อมีบางอย่างดูผิดปกติ"

## ทำไมต้องมีคำสั่งถาวร?

**หากไม่มีคำสั่งถาวร:**

- คุณต้องพรอมต์เอเจนต์สำหรับทุกงาน
- เอเจนต์จะอยู่นิ่งระหว่างคำขอ
- งานประจำอาจถูกลืมหรือล่าช้า
- คุณกลายเป็นคอขวด

**หากมีคำสั่งถาวร:**

- เอเจนต์จะดำเนินการโดยอัตโนมัติภายในขอบเขตที่กำหนด
- งานประจำเกิดขึ้นตามกำหนดโดยไม่ต้องพรอมต์
- คุณเข้าไปเกี่ยวข้องเฉพาะเมื่อมีข้อยกเว้นและการอนุมัติ
- เอเจนต์ใช้เวลาว่างให้เกิดประโยชน์

## วิธีการทำงาน

คำสั่งถาวรถูกกำหนดไว้ในไฟล์ [พื้นที่ทำงานของเอเจนต์](/th/concepts/agent-workspace) ของคุณ วิธีที่แนะนำคือใส่ไว้โดยตรงใน `AGENTS.md` (ซึ่งจะถูกแทรกอัตโนมัติในทุกเซสชัน) เพื่อให้เอเจนต์มีบริบทนี้อยู่เสมอ สำหรับการตั้งค่าที่มีขนาดใหญ่ขึ้น คุณยังสามารถแยกไว้ในไฟล์เฉพาะ เช่น `standing-orders.md` แล้วอ้างอิงจาก `AGENTS.md`

แต่ละโปรแกรมจะระบุสิ่งต่อไปนี้:

1. **ขอบเขต** — สิ่งที่เอเจนต์ได้รับอนุญาตให้ทำ
2. **ทริกเกอร์** — ให้ทำงานเมื่อใด (ตามกำหนดการ เหตุการณ์ หรือเงื่อนไข)
3. **จุดอนุมัติ** — อะไรบ้างที่ต้องได้รับการลงนามอนุมัติจากมนุษย์ก่อนดำเนินการ
4. **กฎการยกระดับ** — เมื่อใดควรหยุดและขอความช่วยเหลือ

เอเจนต์จะโหลดคำสั่งเหล่านี้ในทุกเซสชันผ่านไฟล์บูตสแตรปของพื้นที่ทำงาน (ดู [Agent Workspace](/th/concepts/agent-workspace) สำหรับรายการไฟล์ที่ถูกแทรกอัตโนมัติทั้งหมด) และดำเนินการตามคำสั่งเหล่านั้นร่วมกับ [Cron jobs](/th/automation/cron-jobs) สำหรับการบังคับใช้ตามเวลา

<Tip>
ใส่คำสั่งถาวรไว้ใน `AGENTS.md` เพื่อให้มั่นใจว่าจะถูกโหลดในทุกเซสชัน ระบบบูตสแตรปของพื้นที่ทำงานจะแทรก `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md` และ `MEMORY.md` โดยอัตโนมัติ — แต่จะไม่แทรกไฟล์ทั่วไปในไดเรกทอรีย่อยโดยอัตโนมัติ
</Tip>

## โครงสร้างของคำสั่งถาวร

```markdown
## Program: รายงานสถานะประจำสัปดาห์

**Authority:** รวบรวมข้อมูล สร้างรายงาน ส่งให้ผู้มีส่วนได้ส่วนเสีย
**Trigger:** ทุกวันศุกร์ เวลา 16:00 น. (บังคับใช้ผ่าน Cron job)
**Approval gate:** ไม่มีสำหรับรายงานมาตรฐาน แจ้งสิ่งผิดปกติเพื่อให้มนุษย์ตรวจสอบ
**Escalation:** หากแหล่งข้อมูลไม่พร้อมใช้งาน หรือเมตริกดูผิดปกติ (>2σ จากค่าปกติ)

### ขั้นตอนการดำเนินการ

1. ดึงเมตริกจากแหล่งข้อมูลที่กำหนดไว้
2. เปรียบเทียบกับสัปดาห์ก่อนและเป้าหมาย
3. สร้างรายงานใน Reports/weekly/YYYY-MM-DD.md
4. ส่งสรุปผ่านช่องทางที่กำหนด
5. บันทึกการเสร็จสิ้นลงใน Agent/Logs/

### สิ่งที่ห้ามทำ

- ห้ามส่งรายงานให้บุคคลภายนอก
- ห้ามแก้ไขข้อมูลต้นทาง
- ห้ามข้ามการส่งเพียงเพราะเมตริกออกมาดูไม่ดี — ให้รายงานอย่างถูกต้อง
```

## คำสั่งถาวร + Cron Jobs

คำสั่งถาวรกำหนดว่าเอเจนต์ได้รับอนุญาตให้ทำ**อะไร** ส่วน [Cron jobs](/th/automation/cron-jobs) กำหนดว่าเหตุการณ์นั้นเกิดขึ้น**เมื่อใด** ทั้งสองทำงานร่วมกัน:

```
คำสั่งถาวร: "คุณรับผิดชอบการคัดแยกกล่องข้อความเข้าประจำวัน"
    ↓
Cron Job (ทุกวันเวลา 8:00 น.): "ดำเนินการคัดแยกกล่องข้อความเข้าตามคำสั่งถาวร"
    ↓
เอเจนต์: อ่านคำสั่งถาวร → ดำเนินการตามขั้นตอน → รายงานผลลัพธ์
```

พรอมต์ของ Cron job ควรอ้างอิงคำสั่งถาวรแทนการทำซ้ำเนื้อหาเดิม:

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+1XXXXXXXXXX" \
  --message "ดำเนินการคัดแยกกล่องข้อความเข้าประจำวันตามคำสั่งถาวร ตรวจสอบอีเมลเพื่อหาการแจ้งเตือนใหม่ แยกวิเคราะห์ จัดหมวดหมู่ และบันทึกแต่ละรายการอย่างถาวร รายงานสรุปให้เจ้าของทราบ ยกระดับกรณีที่ไม่ทราบแน่ชัด"
```

## ตัวอย่าง

### ตัวอย่างที่ 1: เนื้อหาและโซเชียลมีเดีย (รอบรายสัปดาห์)

```markdown
## Program: เนื้อหาและโซเชียลมีเดีย

**Authority:** ร่างเนื้อหา ตั้งเวลาโพสต์ รวบรวมรายงานการมีส่วนร่วม
**Approval gate:** ทุกโพสต์ต้องได้รับการตรวจทานจากเจ้าของในช่วง 30 วันแรก จากนั้นใช้การอนุมัติแบบถาวร
**Trigger:** รอบรายสัปดาห์ (ทบทวนวันจันทร์ → ร่างช่วงกลางสัปดาห์ → สรุปวันศุกร์)

### รอบรายสัปดาห์

- **วันจันทร์:** ทบทวนเมตริกของแพลตฟอร์มและการมีส่วนร่วมของผู้ชม
- **วันอังคาร–พฤหัสบดี:** ร่างโพสต์โซเชียล สร้างเนื้อหาบล็อก
- **วันศุกร์:** รวบรวมสรุปการตลาดประจำสัปดาห์ → ส่งให้เจ้าของ

### กฎของเนื้อหา

- น้ำเสียงต้องสอดคล้องกับแบรนด์ (ดู SOUL.md หรือคู่มือน้ำเสียงแบรนด์)
- ห้ามระบุว่าเป็น AI ในเนื้อหาที่เผยแพร่สู่สาธารณะ
- ใส่เมตริกเมื่อมีให้ใช้
- มุ่งเน้นคุณค่าต่อผู้ชม ไม่ใช่การโปรโมตตัวเอง
```

### ตัวอย่างที่ 2: การดำเนินงานทางการเงิน (ทริกเกอร์ตามเหตุการณ์)

```markdown
## Program: การประมวลผลทางการเงิน

**Authority:** ประมวลผลข้อมูลธุรกรรม สร้างรายงาน ส่งสรุป
**Approval gate:** ไม่มีสำหรับการวิเคราะห์ คำแนะนำต้องได้รับการอนุมัติจากเจ้าของ
**Trigger:** ตรวจพบไฟล์ข้อมูลใหม่ หรือถึงรอบรายเดือนตามกำหนด

### เมื่อมีข้อมูลใหม่เข้ามา

1. ตรวจจับไฟล์ใหม่ในไดเรกทอรีรับเข้าที่กำหนด
2. แยกวิเคราะห์และจัดหมวดหมู่ธุรกรรมทั้งหมด
3. เปรียบเทียบกับเป้าหมายงบประมาณ
4. แจ้งเตือน: รายการผิดปกติ การเกินเกณฑ์ รายจ่ายประจำใหม่
5. สร้างรายงานในไดเรกทอรีส่งออกที่กำหนด
6. ส่งสรุปให้เจ้าของผ่านช่องทางที่กำหนด

### กฎการยกระดับ

- รายการเดียว > $500: แจ้งเตือนทันที
- หมวดหมู่ > งบประมาณ 20%: ทำเครื่องหมายไว้ในรายงาน
- ธุรกรรมที่ไม่สามารถระบุได้: ขอให้เจ้าของช่วยจัดหมวดหมู่
- ประมวลผลล้มเหลวหลังลองใหม่ 2 ครั้ง: รายงานความล้มเหลว ห้ามเดา
```

### ตัวอย่างที่ 3: การติดตามระบบและการแจ้งเตือน (ต่อเนื่อง)

```markdown
## Program: การติดตามระบบ

**Authority:** ตรวจสอบสถานะระบบ รีสตาร์ตบริการ ส่งการแจ้งเตือน
**Approval gate:** รีสตาร์ตบริการโดยอัตโนมัติ ยกระดับหากรีสตาร์ตล้มเหลวสองครั้ง
**Trigger:** ทุก Heartbeat cycle

### การตรวจสอบ

- health endpoint ของบริการตอบสนอง
- พื้นที่ดิสก์สูงกว่าเกณฑ์
- งานที่ค้างอยู่ไม่ล้าสมัย (>24 ชั่วโมง)
- ช่องทางการส่งมอบทำงานได้

### เมทริกซ์การตอบสนอง

| เงื่อนไข | การดำเนินการ | ยกระดับหรือไม่? |
| ---------------- | ------------------------ | ------------------------ |
| บริการล่ม | รีสตาร์ตอัตโนมัติ | เฉพาะเมื่อรีสตาร์ตล้มเหลว 2 ครั้ง |
| พื้นที่ดิสก์ < 10% | แจ้งเจ้าของ | ใช่ |
| งานค้าง > 24 ชม. | เตือนเจ้าของ | ไม่ |
| ช่องทางออฟไลน์ | บันทึกและลองใหม่ในรอบถัดไป | หากออฟไลน์เกิน 2 ชั่วโมง |
```

## รูปแบบ Execute-Verify-Report

คำสั่งถาวรทำงานได้ดีที่สุดเมื่อใช้ร่วมกับวินัยการปฏิบัติงานที่เข้มงวด ทุกงานในคำสั่งถาวรควรทำตามลูปนี้:

1. **Execute** — ทำงานจริงให้เสร็จ (ไม่ใช่แค่รับทราบคำสั่ง)
2. **Verify** — ยืนยันว่าผลลัพธ์ถูกต้อง (มีไฟล์อยู่จริง ส่งข้อความสำเร็จ แยกวิเคราะห์ข้อมูลสำเร็จ)
3. **Report** — แจ้งเจ้าของว่าทำอะไรไปแล้ว และยืนยันอะไรแล้วบ้าง

```markdown
### กฎการดำเนินการ

- ทุกงานต้องเป็นไปตาม Execute-Verify-Report ไม่มีข้อยกเว้น
- "เดี๋ยวฉันทำให้" ไม่ถือว่าเป็นการดำเนินการ ต้องทำก่อนแล้วจึงรายงาน
- "เสร็จแล้ว" โดยไม่มีการยืนยันไม่ถือว่ายอมรับได้ ต้องพิสูจน์ได้
- หากการดำเนินการล้มเหลว: ลองใหม่หนึ่งครั้งด้วยวิธีที่ปรับแล้ว
- หากยังล้มเหลว: รายงานความล้มเหลวพร้อมการวินิจฉัย ห้ามล้มเหลวแบบเงียบ ๆ
- ห้ามลองซ้ำไม่สิ้นสุด — สูงสุด 3 ครั้ง จากนั้นยกระดับ
```

รูปแบบนี้ช่วยป้องกันโหมดความล้มเหลวที่พบบ่อยที่สุดของเอเจนต์: รับทราบงานแต่ไม่ได้ทำให้เสร็จจริง

## สถาปัตยกรรมหลายโปรแกรม

สำหรับเอเจนต์ที่ดูแลหลายด้าน ให้จัดคำสั่งถาวรเป็นโปรแกรมแยกกันพร้อมขอบเขตที่ชัดเจน:

```markdown
## Program 1: [โดเมน A] (รายสัปดาห์)

...

## Program 2: [โดเมน B] (รายเดือน + ตามต้องการ)

...

## Program 3: [โดเมน C] (ตามความจำเป็น)

...

## กฎการยกระดับ (ทุกโปรแกรม)

- [เกณฑ์การยกระดับร่วม]
- [จุดอนุมัติที่ใช้กับทุกโปรแกรม]
```

แต่ละโปรแกรมควรมี:

- **รอบทริกเกอร์** ของตัวเอง (รายสัปดาห์ รายเดือน ขับเคลื่อนด้วยเหตุการณ์ หรือต่อเนื่อง)
- **จุดอนุมัติ** ของตัวเอง (บางโปรแกรมต้องการการกำกับดูแลมากกว่าโปรแกรมอื่น)
- **ขอบเขต** ที่ชัดเจน (เอเจนต์ควรรู้ว่าจุดสิ้นสุดของโปรแกรมหนึ่งและจุดเริ่มต้นของอีกโปรแกรมหนึ่งอยู่ตรงไหน)

## แนวทางปฏิบัติที่ดี

### ควรทำ

- เริ่มต้นด้วยอำนาจที่แคบ แล้วค่อยขยายเมื่อความไว้วางใจเพิ่มขึ้น
- กำหนดจุดอนุมัติอย่างชัดเจนสำหรับการกระทำที่มีความเสี่ยงสูง
- ใส่ส่วน "สิ่งที่ห้ามทำ" — ขอบเขตสำคัญพอ ๆ กับสิทธิ์ที่อนุญาต
- ใช้ร่วมกับ Cron jobs เพื่อให้การดำเนินการตามเวลาเชื่อถือได้
- ตรวจสอบบันทึกของเอเจนต์ทุกสัปดาห์เพื่อตรวจยืนยันว่ามีการปฏิบัติตามคำสั่งถาวร
- อัปเดตคำสั่งถาวรเมื่อความต้องการของคุณเปลี่ยนไป — เอกสารเหล่านี้เป็นเอกสารที่มีชีวิต

### หลีกเลี่ยง

- มอบอำนาจกว้างเกินไปตั้งแต่วันแรก ("ทำอะไรก็ได้ที่คุณคิดว่าดีที่สุด")
- ข้ามกฎการยกระดับ — ทุกโปรแกรมต้องมีเงื่อนไข "เมื่อใดควรหยุดและถาม"
- คิดว่าเอเจนต์จะจำคำสั่งด้วยวาจาได้ — ใส่ทุกอย่างไว้ในไฟล์
- ปะปนหลายเรื่องไว้ในโปรแกรมเดียว — แยกโปรแกรมตามแต่ละโดเมน
- ลืมบังคับใช้ด้วย Cron jobs — คำสั่งถาวรที่ไม่มีทริกเกอร์จะกลายเป็นเพียงคำแนะนำ
