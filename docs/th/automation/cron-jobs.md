Cron เป็นตัวจัดตารางเวลาในตัวของ Gateway โดยจะคงงานไว้ ปลุกเอเจนต์ในเวลาที่เหมาะสม และสามารถส่งผลลัพธ์กลับไปยังช่องแชตหรือปลายทาง Webhook ได้

## เริ่มต้นอย่างรวดเร็ว

<Steps>
  <Step title="เพิ่มการเตือนแบบครั้งเดียว">
    ```bash
    openclaw cron add \
      --name "Reminder" \
      --at "2026-02-01T16:00:00Z" \
      --session main \
      --system-event "Reminder: check the cron docs draft" \
      --wake now \
      --delete-after-run
    ```
  </Step>
  <Step title="ตรวจสอบงานของคุณ">
    ```bash
    openclaw cron list
    openclaw cron show <job-id>
    ```
  </Step>
  <Step title="ดูประวัติการรัน">
    ```bash
    openclaw cron runs --id <job-id>
    ```
  </Step>
</Steps>

## Cron ทำงานอย่างไร

- Cron รัน **ภายในโปรเซส Gateway** (ไม่ใช่ภายในโมเดล)
- นิยามงานจะคงอยู่ที่ `~/.openclaw/cron/jobs.json` ดังนั้นการรีสตาร์ตจะไม่ทำให้ตารางเวลาหายไป
- สถานะการดำเนินการขณะรันจะคงอยู่ข้างกันใน `~/.openclaw/cron/jobs-state.json` หากคุณติดตามนิยาม Cron ใน git ให้ติดตาม `jobs.json` และเพิ่ม `jobs-state.json` ลงใน gitignore
- หลังจากการแยก เวอร์ชัน OpenClaw ที่เก่ากว่าสามารถอ่าน `jobs.json` ได้ แต่อาจถือว่างานเป็นงานใหม่ เพราะตอนนี้ฟิลด์ขณะรันอยู่ใน `jobs-state.json`
- เมื่อมีการแก้ไข `jobs.json` ขณะที่ Gateway กำลังทำงานหรือหยุดอยู่ OpenClaw จะเปรียบเทียบฟิลด์ตารางเวลาที่เปลี่ยนกับเมทาดาต้าช่องเวลาขณะรันที่ค้างอยู่ และล้างค่า `nextRunAtMs` ที่ล้าสมัย การเขียนใหม่ที่เปลี่ยนเฉพาะรูปแบบหรือเฉพาะลำดับคีย์จะคงช่องเวลาที่ค้างอยู่ไว้
- การดำเนินการ Cron ทั้งหมดจะสร้างระเบียน [งานเบื้องหลัง](/th/automation/tasks)
- เมื่อ Gateway เริ่มทำงาน งานแบบรอบเอเจนต์แยกที่เลยกำหนดจะถูกจัดตารางใหม่ออกจากช่วงเชื่อมต่อช่องทางแทนที่จะเล่นซ้ำทันที เพื่อให้การเริ่มต้น Discord/Telegram และการตั้งค่าคำสั่งเนทีฟยังตอบสนองได้ดีหลังรีสตาร์ต
- งานแบบครั้งเดียว (`--at`) จะลบอัตโนมัติหลังสำเร็จโดยค่าเริ่มต้น
- การรัน Cron แบบแยกจะพยายามปิดแท็บ/โปรเซสเบราว์เซอร์ที่ติดตามไว้สำหรับเซสชัน `cron:<jobId>` ของตนเมื่อรันเสร็จ เพื่อให้ออโตเมชันเบราว์เซอร์ที่แยกออกมาไม่ทิ้งโปรเซสกำพร้าไว้
- การรัน Cron แบบแยกยังป้องกันการตอบรับที่ล้าสมัยด้วย หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะชั่วคราว (`on it`, `pulling everything together` และคำใบ้ที่คล้ายกัน) และไม่มีการรันซับเอเจนต์ลูกหลานใดยังรับผิดชอบคำตอบสุดท้ายอยู่ OpenClaw จะพรอมป์ซ้ำหนึ่งครั้งเพื่อขอผลลัพธ์จริงก่อนส่งมอบ
- การรัน Cron แบบแยกจะเลือกใช้เมทาดาต้าการปฏิเสธการดำเนินการแบบมีโครงสร้างจากการรันแบบฝังตัวก่อน แล้วจึงถอยกลับไปใช้เครื่องหมายสรุป/เอาต์พุตสุดท้ายที่รู้จัก เช่น `SYSTEM_RUN_DENIED` และ `INVALID_REQUEST` เพื่อไม่ให้รายงานคำสั่งที่ถูกบล็อกว่าเป็นการรันสำเร็จ
- การรัน Cron แบบแยกยังถือว่าความล้มเหลวของเอเจนต์ระดับการรันเป็นข้อผิดพลาดของงาน แม้จะไม่มีเพย์โหลดตอบกลับถูกสร้างขึ้นก็ตาม ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการจะเพิ่มตัวนับข้อผิดพลาดและทริกเกอร์การแจ้งเตือนความล้มเหลว แทนที่จะล้างงานว่าเสร็จสำเร็จ
- เมื่องานแบบรอบเอเจนต์แยกถึง `timeoutSeconds` Cron จะยกเลิกการรันเอเจนต์ที่อยู่เบื้องหลังและให้ช่วงเวลาสั้น ๆ สำหรับล้างทรัพยากร หากการรันไม่ระบายจนจบ การล้างทรัพยากรที่ Gateway เป็นเจ้าของจะบังคับล้างความเป็นเจ้าของเซสชันของการรันนั้นก่อนที่ Cron จะบันทึกการหมดเวลา เพื่อไม่ให้งานแชตที่เข้าคิวถูกทิ้งไว้หลังเซสชันประมวลผลที่ล้าสมัย

<a id="maintenance"></a>

<Note>
การกระทบยอดงานสำหรับ Cron ให้สิทธิ์ความเป็นเจ้าของขณะรันเป็นอันดับแรก และใช้ประวัติที่คงทนเป็นข้อมูลสำรอง: งาน Cron ที่ใช้งานอยู่จะยังคงอยู่ขณะที่รันไทม์ Cron ยังติดตามว่างานนั้นกำลังทำงานอยู่ แม้จะยังมีแถวเซสชันลูกเก่าอยู่ก็ตาม เมื่อรันไทม์หยุดเป็นเจ้าของงานและช่วงผ่อนผัน 5 นาทีหมดลง การบำรุงรักษาจะตรวจสอบบันทึกการรันที่คงอยู่และสถานะงานสำหรับการรัน `cron:<jobId>:<startedAt>` ที่ตรงกัน หากประวัติที่คงทนนั้นแสดงผลลัพธ์ปลายทาง บัญชีแยกประเภทงานจะถูกสรุปจากประวัตินั้น มิฉะนั้นการบำรุงรักษาที่ Gateway เป็นเจ้าของสามารถทำเครื่องหมายงานเป็น `lost` ได้ การตรวจสอบ CLI แบบออฟไลน์สามารถกู้คืนจากประวัติที่คงทนได้ แต่จะไม่ถือว่าชุดงานที่ใช้งานอยู่ในโปรเซสของตนเองที่ว่างเปล่าเป็นหลักฐานว่าการรัน Cron ที่ Gateway เป็นเจ้าของหายไป
</Note>

## ประเภทตารางเวลา

| ชนิด    | แฟล็ก CLI  | คำอธิบาย                                             |
| ------- | --------- | ------------------------------------------------------- |
| `at`    | `--at`    | เวลาประทับแบบครั้งเดียว (ISO 8601 หรือแบบสัมพัทธ์ เช่น `20m`)    |
| `every` | `--every` | ช่วงเวลาคงที่                                          |
| `cron`  | `--cron`  | นิพจน์ Cron แบบ 5 ฟิลด์หรือ 6 ฟิลด์ พร้อม `--tz` แบบไม่บังคับ |

เวลาประทับที่ไม่มีเขตเวลาจะถือเป็น UTC เพิ่ม `--tz America/New_York` สำหรับการจัดตารางเวลาตามเวลานาฬิกาท้องถิ่น

นิพจน์ที่เกิดซ้ำตรงต้นชั่วโมงจะถูกกระจายเวลาโดยอัตโนมัติสูงสุด 5 นาทีเพื่อลดโหลดพุ่ง ใช้ `--exact` เพื่อบังคับเวลาที่แม่นยำ หรือ `--stagger 30s` สำหรับหน้าต่างเวลาที่ระบุชัดเจน

### วันของเดือนและวันของสัปดาห์ใช้ตรรกะ OR

นิพจน์ Cron ถูกแยกวิเคราะห์โดย [croner](https://github.com/Hexagon/croner) เมื่อทั้งฟิลด์วันของเดือนและวันของสัปดาห์ไม่ใช่ไวลด์การ์ด croner จะจับคู่เมื่อฟิลด์ **ใดฟิลด์หนึ่ง** ตรงกัน ไม่ใช่ทั้งสองฟิลด์ นี่คือพฤติกรรมมาตรฐานของ Vixie cron

```
# Intended: "9 AM on the 15th, only if it's a Monday"
# Actual:   "9 AM on every 15th, AND 9 AM on every Monday"
0 9 15 * 1
```

สิ่งนี้จะทำงานประมาณ 5-6 ครั้งต่อเดือนแทนที่จะเป็น 0-1 ครั้งต่อเดือน OpenClaw ใช้พฤติกรรม OR เริ่มต้นของ Croner ที่นี่ หากต้องการบังคับให้ทั้งสองเงื่อนไขตรงกัน ให้ใช้ตัวปรับวันของสัปดาห์ `+` ของ Croner (`0 9 15 * +1`) หรือจัดตารางบนฟิลด์หนึ่งและตรวจอีกฟิลด์ในพรอมป์หรือคำสั่งของงาน

## รูปแบบการดำเนินการ

| รูปแบบ           | ค่า `--session`   | รันใน                  | เหมาะที่สุดสำหรับ                        |
| --------------- | ------------------- | ------------------------ | ------------------------------- |
| เซสชันหลัก    | `main`              | รอบ Heartbeat ถัดไป      | การเตือน เหตุการณ์ระบบ        |
| แบบแยก        | `isolated`          | `cron:<jobId>` เฉพาะ | รายงาน งานเบื้องหลัง      |
| เซสชันปัจจุบัน | `current`           | ผูกตอนสร้าง   | งานที่เกิดซ้ำและรับรู้บริบท    |
| เซสชันกำหนดเอง  | `session:custom-id` | เซสชันชื่อถาวร | เวิร์กโฟลว์ที่ต่อยอดจากประวัติ |

<AccordionGroup>
  <Accordion title="เซสชันหลักเทียบกับแบบแยกเทียบกับแบบกำหนดเอง">
    งาน **เซสชันหลัก** จะนำเหตุการณ์ระบบเข้าคิว และปลุก Heartbeat ได้แบบไม่บังคับ (`--wake now` หรือ `--wake next-heartbeat`) เหตุการณ์ระบบเหล่านั้นจะไม่ขยายความสดใหม่ของการรีเซ็ตรายวัน/เมื่อว่างสำหรับเซสชันเป้าหมาย งาน **แบบแยก** จะรันรอบเอเจนต์เฉพาะด้วยเซสชันใหม่ งาน **เซสชันกำหนดเอง** (`session:xxx`) จะคงบริบทข้ามการรัน ทำให้ทำเวิร์กโฟลว์อย่างสแตนด์อัปรายวันที่ต่อยอดจากสรุปก่อนหน้าได้
  </Accordion>
  <Accordion title="ความหมายของ 'เซสชันใหม่' สำหรับงานแบบแยก">
    สำหรับงานแบบแยก "เซสชันใหม่" หมายถึงทรานสคริปต์/รหัสเซสชันใหม่สำหรับการรันแต่ละครั้ง OpenClaw อาจนำค่ากำหนดที่ปลอดภัยติดไปด้วย เช่น การตั้งค่า thinking/fast/verbose, ป้ายกำกับ และการเลือกโมเดล/การยืนยันตัวตนที่ผู้ใช้เลือกไว้อย่างชัดเจน แต่จะไม่สืบทอดบริบทการสนทนาแวดล้อมจากแถว Cron ที่เก่ากว่า: การกำหนดเส้นทางช่องทาง/กลุ่ม นโยบายส่งหรือเข้าคิว การยกระดับ แหล่งที่มา หรือการผูก ACP runtime ใช้ `current` หรือ `session:<id>` เมื่องานที่เกิดซ้ำควรจงใจต่อยอดจากบริบทการสนทนาเดียวกัน
  </Accordion>
  <Accordion title="การล้างทรัพยากรขณะรัน">
    สำหรับงานแบบแยก ตอนนี้การรื้อถอนรันไทม์รวมถึงการล้างเบราว์เซอร์แบบพยายามเต็มที่สำหรับเซสชัน Cron นั้น ความล้มเหลวในการล้างจะถูกละเว้นเพื่อให้ผลลัพธ์ Cron จริงยังเป็นตัวตัดสิน

    การรัน Cron แบบแยกยัง dispose อินสแตนซ์ MCP runtime ที่รวมมาด้วยซึ่งสร้างขึ้นสำหรับงานผ่านเส้นทางการล้างรันไทม์ที่ใช้ร่วมกัน สิ่งนี้ตรงกับวิธีรื้อถอน MCP client ของเซสชันหลักและเซสชันกำหนดเอง ดังนั้นงาน Cron แบบแยกจะไม่รั่วโปรเซสลูก stdio หรือการเชื่อมต่อ MCP ที่มีอายุยาวข้ามการรัน

  </Accordion>
  <Accordion title="ซับเอเจนต์และการส่งมอบ Discord">
    เมื่อการรัน Cron แบบแยกประสานงานซับเอเจนต์ การส่งมอบจะเลือกเอาต์พุตสุดท้ายของลูกหลานมากกว่าข้อความชั่วคราวของพาเรนต์ที่ล้าสมัยด้วย หากลูกหลานยังคงทำงานอยู่ OpenClaw จะระงับการอัปเดตพาเรนต์บางส่วนนั้นแทนที่จะประกาศออกไป

    สำหรับเป้าหมายประกาศ Discord แบบข้อความเท่านั้น OpenClaw จะส่งข้อความผู้ช่วยสุดท้ายตามมาตรฐานหนึ่งครั้ง แทนที่จะเล่นซ้ำทั้งเพย์โหลดข้อความแบบสตรีม/ระหว่างกลางและคำตอบสุดท้าย เพย์โหลด Discord แบบสื่อและแบบมีโครงสร้างยังคงถูกส่งเป็นเพย์โหลดแยกต่างหาก เพื่อไม่ให้ไฟล์แนบและคอมโพเนนต์ถูกทิ้ง

  </Accordion>

`--model` ใช้โมเดลที่อนุญาตซึ่งเลือกไว้เป็นโมเดลหลักของงานนั้น ไม่เหมือนกับการแทนที่ `/model` ของเซสชันแชต: เชน fallback ที่กำหนดค่าไว้ยังคงใช้เมื่อโมเดลหลักของงานล้มเหลว หากโมเดลที่ร้องขอไม่ได้รับอนุญาตหรือไม่สามารถแปลงเป็นผลลัพธ์ได้ Cron จะทำให้การรันล้มเหลวพร้อมข้อผิดพลาดการตรวจสอบที่ชัดเจน แทนที่จะถอยกลับไปใช้การเลือกโมเดลของเอเจนต์/ค่าเริ่มต้นของงานอย่างเงียบ ๆ

งาน Cron สามารถมี `fallbacks` ระดับเพย์โหลดได้ด้วย เมื่อมีอยู่ รายการนั้นจะแทนที่เชน fallback ที่กำหนดค่าไว้สำหรับงาน ใช้ `fallbacks: []` ในเพย์โหลด/API ของงานเมื่อต้องการการรัน Cron แบบเข้มงวดที่ลองเฉพาะโมเดลที่เลือกไว้ หากงานมี `--model` แต่ไม่มี fallback ทั้งในเพย์โหลดหรือการกำหนดค่า OpenClaw จะส่งการแทนที่ fallback ว่างอย่างชัดเจน เพื่อไม่ให้โมเดลหลักของเอเจนต์ถูกต่อท้ายเป็นเป้าหมายการลองซ้ำแฝง

ลำดับความสำคัญในการเลือกโมเดลสำหรับงานแบบแยกคือ:

1. การแทนที่โมเดลของฮุก Gmail (เมื่อการรันมาจาก Gmail และอนุญาตให้ใช้การแทนที่นั้น)
2. `model` ระดับเพย์โหลดต่องาน
3. การแทนที่โมเดลเซสชัน Cron ที่ผู้ใช้เลือกและจัดเก็บไว้
4. การเลือกโมเดลของเอเจนต์/ค่าเริ่มต้น

โหมดเร็วจะตามการเลือกสดที่แก้ไขแล้วด้วย หากคอนฟิกโมเดลที่เลือกมี `params.fastMode` Cron แบบแยกจะใช้ค่านั้นโดยค่าเริ่มต้น การแทนที่ `fastMode` ของเซสชันที่จัดเก็บไว้ยังคงชนะคอนฟิกได้ทั้งสองทิศทาง

หากการรันแบบแยกพบการส่งต่อแบบสลับโมเดลสด Cron จะลองซ้ำด้วยผู้ให้บริการ/โมเดลที่สลับไป และคงการเลือกสดนั้นไว้สำหรับการรันที่ใช้งานอยู่ก่อนลองซ้ำ เมื่อการสลับยังมีโปรไฟล์การยืนยันตัวตนใหม่มาด้วย Cron จะคงการแทนที่โปรไฟล์การยืนยันตัวตนนั้นไว้สำหรับการรันที่ใช้งานอยู่ด้วย การลองซ้ำมีขอบเขต: หลังจากความพยายามแรกบวกการลองซ้ำจากการสลับ 2 ครั้ง Cron จะยกเลิกแทนที่จะวนตลอดไป

ก่อนที่การรัน Cron แบบแยกจะเข้าสู่ตัวรันเอเจนต์ OpenClaw จะตรวจสอบปลายทางผู้ให้บริการในเครื่องที่เข้าถึงได้สำหรับผู้ให้บริการ `api: "ollama"` และ `api: "openai-completions"` ที่กำหนดค่าไว้ ซึ่ง `baseUrl` เป็น local loopback, เครือข่ายส่วนตัว หรือ `.local` หากปลายทางนั้นหยุดทำงาน การรันจะถูกบันทึกเป็น `skipped` พร้อมข้อผิดพลาดผู้ให้บริการ/โมเดลที่ชัดเจน แทนที่จะเริ่มการเรียกโมเดล ผลลัพธ์ปลายทางจะถูกแคชเป็นเวลา 5 นาที ดังนั้นงานจำนวนมากที่ถึงกำหนดและใช้เซิร์ฟเวอร์ Ollama, vLLM, SGLang หรือ LM Studio ในเครื่องตัวเดียวกันที่หยุดทำงาน จะใช้การตรวจสอบขนาดเล็กครั้งเดียวร่วมกัน แทนที่จะสร้างพายุคำขอ การรันที่ถูกข้ามจากการตรวจล่วงหน้าผู้ให้บริการจะไม่เพิ่ม backoff ข้อผิดพลาดการดำเนินการ เปิดใช้ `failureAlert.includeSkipped` เมื่อต้องการการแจ้งเตือนการข้ามซ้ำ ๆ

## การส่งมอบและเอาต์พุต

| โหมด       | สิ่งที่เกิดขึ้น                                                        |
| ---------- | ------------------------------------------------------------------- |
| `announce` | ส่งข้อความสุดท้ายแบบ fallback ไปยังเป้าหมายหากเอเจนต์ไม่ได้ส่ง |
| `webhook`  | POST เพย์โหลดเหตุการณ์ที่เสร็จแล้วไปยัง URL                                |
| `none`     | ไม่มีการส่งมอบ fallback ของตัวรัน                                         |

ใช้ `--announce --channel telegram --to "-1001234567890"` สำหรับการส่งไปยังช่องทาง สำหรับหัวข้อฟอรัมของ Telegram ให้ใช้ `-1001234567890:topic:123`; ผู้เรียก RPC/config โดยตรงอาจส่ง `delivery.threadId` เป็นสตริงหรือตัวเลขได้เช่นกัน เป้าหมาย Slack/Discord/Mattermost ควรใช้คำนำหน้าที่ชัดเจน (`channel:<id>`, `user:<id>`) ID ห้องของ Matrix แยกตัวพิมพ์เล็กใหญ่ ให้ใช้ ID ห้องที่ตรงเป๊ะหรือรูปแบบ `room:!room:server` จาก Matrix

สำหรับงานแบบแยก การส่งแชทจะใช้ร่วมกัน หากมีเส้นทางแชท เอเจนต์สามารถใช้เครื่องมือ `message` ได้แม้งานจะใช้ `--no-deliver` หากเอเจนต์ส่งไปยังเป้าหมายที่กำหนดค่าไว้/ปัจจุบัน OpenClaw จะข้ามการประกาศสำรอง มิฉะนั้น `announce`, `webhook` และ `none` จะควบคุมเฉพาะสิ่งที่ตัวรันทำกับคำตอบสุดท้ายหลังจากรอบของเอเจนต์เท่านั้น

เมื่อเอเจนต์สร้างการแจ้งเตือนแบบแยกจากแชทที่ใช้งานอยู่ OpenClaw จะจัดเก็บเป้าหมายการส่งแบบสดที่เก็บรักษาไว้สำหรับเส้นทางการประกาศสำรอง คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็ก เป้าหมายการส่งของผู้ให้บริการจะไม่ถูกสร้างใหม่จากคีย์เหล่านั้นเมื่อมีบริบทแชทปัจจุบันให้ใช้

การแจ้งเตือนความล้มเหลวใช้เส้นทางปลายทางแยกต่างหาก:

- `cron.failureDestination` ตั้งค่าเริ่มต้นส่วนกลางสำหรับการแจ้งเตือนความล้มเหลว
- `job.delivery.failureDestination` แทนที่ค่านั้นต่อหนึ่งงาน
- หากไม่ได้ตั้งค่าทั้งสองอย่างและงานส่งผ่าน `announce` อยู่แล้ว ตอนนี้การแจ้งเตือนความล้มเหลวจะถอยกลับไปยังเป้าหมายประกาศหลักนั้น
- `delivery.failureDestination` รองรับเฉพาะงาน `sessionTarget="isolated"` เท่านั้น เว้นแต่โหมดการส่งหลักจะเป็น `webhook`
- `failureAlert.includeSkipped: true` เลือกให้งานหรือนโยบายแจ้งเตือน cron ส่วนกลางรวมการแจ้งเตือนรอบที่ถูกข้ามซ้ำ ๆ รอบที่ถูกข้ามจะรักษาตัวนับการข้ามต่อเนื่องแยกต่างหาก จึงไม่ส่งผลต่อการถอยหลังเมื่อเกิดข้อผิดพลาดในการดำเนินการ

## Webhook

Gateway สามารถเปิดเผยปลายทาง HTTP Webhook สำหรับตัวกระตุ้นภายนอก เปิดใช้ในการกำหนดค่า:

```json5
{
  hooks: {
    enabled: true,
    token: "shared-secret",
    path: "/hooks",
  },
}
```

### การตรวจสอบสิทธิ์

ทุกคำขอต้องมีโทเค็น hook ผ่านส่วนหัว:

- `Authorization: Bearer <token>` (แนะนำ)
- `x-openclaw-token: <token>`

โทเค็นในสตริงคิวรีจะถูกปฏิเสธ

<AccordionGroup>
  <Accordion title="POST /hooks/wake">
    เข้าคิวเหตุการณ์ระบบสำหรับเซสชันหลัก:

    ```bash
    curl -X POST http://127.0.0.1:18789/hooks/wake \
      -H 'Authorization: Bearer SECRET' \
      -H 'Content-Type: application/json' \
      -d '{"text":"New email received","mode":"now"}'
    ```

    <ParamField path="text" type="string" required>
      คำอธิบายเหตุการณ์
    </ParamField>
    <ParamField path="mode" type="string" default="now">
      `now` หรือ `next-heartbeat`
    </ParamField>

  </Accordion>
  <Accordion title="POST /hooks/agent">
    รันรอบเอเจนต์แบบแยก:

    ```bash
    curl -X POST http://127.0.0.1:18789/hooks/agent \
      -H 'Authorization: Bearer SECRET' \
      -H 'Content-Type: application/json' \
      -d '{"message":"Summarize inbox","name":"Email","model":"openai/gpt-5.4"}'
    ```

    ฟิลด์: `message` (จำเป็น), `name`, `agentId`, `wakeMode`, `deliver`, `channel`, `to`, `model`, `fallbacks`, `thinking`, `timeoutSeconds`

  </Accordion>
  <Accordion title="Hook ที่แมปแล้ว (POST /hooks/<name>)">
    ชื่อ hook แบบกำหนดเองจะถูกแก้ไขผ่าน `hooks.mappings` ในการกำหนดค่า การแมปสามารถแปลงเพย์โหลดใด ๆ ให้เป็นการทำงาน `wake` หรือ `agent` ด้วยเทมเพลตหรือการแปลงด้วยโค้ด
  </Accordion>
</AccordionGroup>

<Warning>
เก็บปลายทาง hook ไว้หลัง loopback, tailnet หรือพร็อกซีแบบย้อนกลับที่เชื่อถือได้

- ใช้โทเค็น hook โดยเฉพาะ อย่านำโทเค็นการตรวจสอบสิทธิ์ของ gateway มาใช้ซ้ำ
- เก็บ `hooks.path` ไว้บนเส้นทางย่อยโดยเฉพาะ; `/` จะถูกปฏิเสธ
- ตั้งค่า `hooks.allowedAgentIds` เพื่อจำกัดการกำหนดเส้นทาง `agentId` ที่ระบุชัดเจน
- เก็บ `hooks.allowRequestSessionKey=false` ไว้ เว้นแต่คุณต้องการให้ผู้เรียกเลือกเซสชันได้
- หากเปิดใช้ `hooks.allowRequestSessionKey` ให้ตั้งค่า `hooks.allowedSessionKeyPrefixes` ด้วยเพื่อจำกัดรูปแบบคีย์เซสชันที่อนุญาต
- เพย์โหลด hook จะถูกห่อด้วยขอบเขตความปลอดภัยโดยค่าเริ่มต้น

</Warning>

## การผสานรวม Gmail PubSub

เชื่อมตัวกระตุ้นกล่องจดหมาย Gmail เข้ากับ OpenClaw ผ่าน Google PubSub

<Note>
**ข้อกำหนดเบื้องต้น:** `gcloud` CLI, `gog` (gogcli), เปิดใช้ hook ของ OpenClaw, Tailscale สำหรับปลายทาง HTTPS สาธารณะ
</Note>

### การตั้งค่าด้วยตัวช่วย (แนะนำ)

```bash
openclaw webhooks gmail setup --account openclaw@gmail.com
```

คำสั่งนี้เขียนการกำหนดค่า `hooks.gmail`, เปิดใช้พรีเซ็ต Gmail และใช้ Tailscale Funnel สำหรับปลายทาง push

### การเริ่ม Gateway อัตโนมัติ

เมื่อ `hooks.enabled=true` และตั้งค่า `hooks.gmail.account` แล้ว Gateway จะเริ่ม `gog gmail watch serve` ตอนบูตและต่ออายุการ watch อัตโนมัติ ตั้งค่า `OPENCLAW_SKIP_GMAIL_WATCHER=1` เพื่อเลือกไม่ใช้

### การตั้งค่าแบบครั้งเดียวด้วยตนเอง

<Steps>
  <Step title="เลือกโปรเจกต์ GCP">
    เลือกโปรเจกต์ GCP ที่เป็นเจ้าของไคลเอนต์ OAuth ที่ `gog` ใช้:

    ```bash
    gcloud auth login
    gcloud config set project <project-id>
    gcloud services enable gmail.googleapis.com pubsub.googleapis.com
    ```

  </Step>
  <Step title="สร้าง topic และให้สิทธิ์ Gmail push">
    ```bash
    gcloud pubsub topics create gog-gmail-watch
    gcloud pubsub topics add-iam-policy-binding gog-gmail-watch \
      --member=serviceAccount:gmail-api-push@system.gserviceaccount.com \
      --role=roles/pubsub.publisher
    ```
  </Step>
  <Step title="เริ่ม watch">
    ```bash
    gog gmail watch start \
      --account openclaw@gmail.com \
      --label INBOX \
      --topic projects/<project-id>/topics/gog-gmail-watch
    ```
  </Step>
</Steps>

## การจัดการงาน

```bash
# List all jobs
openclaw cron list

# Show one job, including resolved delivery route
openclaw cron show <jobId>

# Edit a job
openclaw cron edit <jobId> --message "Updated prompt" --model "opus"

# Force run a job now
openclaw cron run <jobId>

# Run only if due
openclaw cron run <jobId> --due

# View run history
openclaw cron runs --id <jobId> --limit 50

# Delete a job
openclaw cron remove <jobId>

# Agent selection (multi-agent setups)
openclaw cron add --name "Ops sweep" --cron "0 6 * * *" --session isolated --message "Check ops queue" --agent ops
openclaw cron edit <jobId> --clear-agent
```

<Note>
หมายเหตุการแทนที่โมเดล:

- `openclaw cron add|edit --model ...` เปลี่ยนโมเดลที่เลือกของงาน
- หากอนุญาตโมเดลนั้น provider/model ที่ตรงเป๊ะจะไปถึงการรันเอเจนต์แบบแยก
- หากไม่ได้รับอนุญาตหรือไม่สามารถแก้ไขได้ cron จะทำให้การรันล้มเหลวพร้อมข้อผิดพลาดการตรวจสอบความถูกต้องที่ชัดเจน
- เชนสำรองที่กำหนดค่าไว้ยังคงมีผล เพราะ `--model` ของ cron เป็นค่าหลักของงาน ไม่ใช่การแทนที่ `/model` ของเซสชัน
- เพย์โหลด `fallbacks` จะแทนที่ fallback ที่กำหนดค่าไว้สำหรับงานนั้น; `fallbacks: []` ปิดใช้ fallback และทำให้การรันเข้มงวด
- `--model` เปล่า ๆ ที่ไม่มีรายการ fallback ที่ระบุชัดเจนหรือกำหนดค่าไว้จะไม่ตกผ่านไปยังค่าหลักของเอเจนต์เป็นเป้าหมายลองใหม่เงียบ ๆ เพิ่มเติม

</Note>

```json5
}
```

`maxConcurrentRuns` จำกัดทั้งการส่ง cron ตามกำหนดเวลาและการดำเนินการรอบเอเจนต์แบบแยก รอบเอเจนต์ cron แบบแยกใช้เลนการดำเนินการ `cron-nested` เฉพาะของคิวภายใน ดังนั้นการเพิ่มค่านี้จะทำให้การรัน LLM ของ cron ที่เป็นอิสระต่อกันคืบหน้าแบบขนาน แทนที่จะเริ่มได้เฉพาะตัวห่อ cron ชั้นนอก เลน `nested` ที่ไม่ใช่ cron แบบใช้ร่วมกันจะไม่ถูกขยายด้วยการตั้งค่านี้

ไฟล์ประกอบสถานะรันไทม์ได้มาจาก `cron.store`: store `.json` เช่น `~/clawd/cron/jobs.json` ใช้ `~/clawd/cron/jobs-state.json` ส่วนเส้นทาง store ที่ไม่มีส่วนต่อท้าย `.json` จะเติม `-state.json`

หากคุณแก้ไข `jobs.json` ด้วยมือ อย่านำ `jobs-state.json` เข้า source control OpenClaw ใช้ไฟล์ประกอบนั้นสำหรับสล็อตที่รอดำเนินการ เครื่องหมายที่กำลังใช้งาน เมตาดาต้าการรันล่าสุด และตัวตนของกำหนดการที่บอก scheduler ว่างานที่แก้ไขจากภายนอกต้องมี `nextRunAtMs` ใหม่

ปิดใช้ cron: `cron.enabled: false` หรือ `OPENCLAW_SKIP_CRON=1`

<AccordionGroup>
  <Accordion title="พฤติกรรมการลองใหม่">
    **การลองใหม่แบบครั้งเดียว**: ข้อผิดพลาดชั่วคราว (rate limit, overload, network, server error) จะลองใหม่ได้สูงสุด 3 ครั้งพร้อม exponential backoff ข้อผิดพลาดถาวรจะปิดใช้ทันที

    **การลองใหม่แบบเกิดซ้ำ**: exponential backoff (30 วินาทีถึง 60 นาที) ระหว่างการลองใหม่ Backoff จะรีเซ็ตหลังจากการรันสำเร็จครั้งถัดไป

  </Accordion>
  <Accordion title="การบำรุงรักษา">
    `cron.sessionRetention` (ค่าเริ่มต้น `24h`) ตัดรายการเซสชันรันแบบแยกออก `cron.runLog.maxBytes` / `cron.runLog.keepLines` ตัดไฟล์บันทึกการรันอัตโนมัติ
  </Accordion>
</AccordionGroup>

## การแก้ไขปัญหา

### ลำดับคำสั่ง

```bash
openclaw status
openclaw gateway status
openclaw cron status
openclaw cron list
openclaw cron runs --id <jobId> --limit 20
openclaw system heartbeat last
openclaw logs --follow
openclaw doctor
```

<AccordionGroup>
  <Accordion title="Cron ไม่ทำงาน">
    - ตรวจสอบ `cron.enabled` และตัวแปรสภาพแวดล้อม `OPENCLAW_SKIP_CRON`
    - ยืนยันว่า Gateway ทำงานต่อเนื่อง
    - สำหรับกำหนดการ `cron` ให้ตรวจสอบเขตเวลา (`--tz`) เทียบกับเขตเวลาของโฮสต์
    - `reason: not-due` ในผลลัพธ์การรันหมายความว่าการรันด้วยตนเองถูกตรวจด้วย `openclaw cron run <jobId> --due` และงานยังไม่ถึงกำหนด

  </Accordion>
  <Accordion title="Cron ทำงานแล้วแต่ไม่มีการส่ง">
    - โหมดการส่ง `none` หมายความว่าไม่คาดว่าจะมีการส่งสำรองจากตัวรัน เอเจนต์ยังสามารถส่งโดยตรงด้วยเครื่องมือ `message` ได้เมื่อมีเส้นทางแชท
    - เป้าหมายการส่งหายไป/ไม่ถูกต้อง (`channel`/`to`) หมายความว่าการส่งออกถูกข้าม
    - สำหรับ Matrix งานที่คัดลอกมาหรือเป็นงานเดิมที่มี ID ห้อง `delivery.to` เป็นตัวพิมพ์เล็กอาจล้มเหลวได้ เพราะ ID ห้องของ Matrix แยกตัวพิมพ์เล็กใหญ่ แก้ไขงานให้เป็นค่า `!room:server` หรือ `room:!room:server` ที่ตรงเป๊ะจาก Matrix
    - ข้อผิดพลาดการตรวจสอบสิทธิ์ช่องทาง (`unauthorized`, `Forbidden`) หมายความว่าการส่งถูกบล็อกโดยข้อมูลประจำตัว
    - หากการรันแบบแยกส่งคืนเฉพาะโทเค็นเงียบ (`NO_REPLY` / `no_reply`) OpenClaw จะระงับการส่งออกโดยตรงและระงับเส้นทางสรุปสำรองที่เข้าคิวด้วย ดังนั้นจึงไม่มีสิ่งใดถูกโพสต์กลับไปยังแชท
    - หากเอเจนต์ควรส่งข้อความถึงผู้ใช้เอง ให้ตรวจสอบว่างานมีเส้นทางที่ใช้ได้ (`channel: "last"` พร้อมแชทก่อนหน้า หรือช่องทาง/เป้าหมายที่ระบุชัดเจน)

  </Accordion>
  <Accordion title="Cron หรือ Heartbeat ดูเหมือนจะป้องกันการ rollover แบบ /new-style">
    - ความสดใหม่ของการรีเซ็ตรายวันและเมื่อไม่ได้ใช้งานไม่ได้อิงตาม `updatedAt`; ดู [การจัดการเซสชัน](/th/concepts/session#session-lifecycle)
    - การปลุกของ Cron, การรัน Heartbeat, การแจ้งเตือน exec และงานบัญชีของ Gateway อาจอัปเดตแถวเซสชันสำหรับการกำหนดเส้นทาง/สถานะ แต่จะไม่ขยาย `sessionStartedAt` หรือ `lastInteractionAt`
    - สำหรับแถวแบบเดิมที่สร้างขึ้นก่อนมีฟิลด์เหล่านั้น OpenClaw สามารถกู้คืน `sessionStartedAt` จากส่วนหัวเซสชันใน transcript JSONL ได้เมื่อไฟล์ยังพร้อมใช้งานอยู่ แถวว่างแบบเดิมที่ไม่มี `lastInteractionAt` จะใช้เวลาเริ่มต้นที่กู้คืนนั้นเป็นค่าพื้นฐานของการไม่ได้ใช้งาน

  </Accordion>
  <Accordion title="ข้อควรระวังเกี่ยวกับเขตเวลา">
    - Cron ที่ไม่มี `--tz` จะใช้เขตเวลาของโฮสต์ Gateway
    - กำหนดการ `at` ที่ไม่มีเขตเวลาจะถูกถือว่าเป็น UTC
    - `activeHours` ของ Heartbeat ใช้การแก้ค่าเขตเวลาที่กำหนดไว้

  </Accordion>
</AccordionGroup>
