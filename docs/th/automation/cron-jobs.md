Cron คือเครื่องมือตั้งเวลาที่มีในตัวของ Gateway โดยจะคงข้อมูลงาน ปลุกเอเจนต์ตามเวลาที่ถูกต้อง และสามารถส่งผลลัพธ์กลับไปยังช่องทางแชตหรือปลายทาง webhook ได้

## เริ่มต้นอย่างรวดเร็ว

<Steps>
  <Step title="เพิ่มการเตือนแบบครั้งเดียว">
    ```bash
    openclaw cron add \
      --name "Reminder" \
      --at "2026-02-01T16:00:00Z" \
      --session main \
      --system-event "Reminder: check the cron docs draft" \
      --wake now \
      --delete-after-run
    ```
  </Step>
  <Step title="ตรวจสอบงานของคุณ">
    ```bash
    openclaw cron list
    openclaw cron show <job-id>
    ```
  </Step>
  <Step title="ดูประวัติการรัน">
    ```bash
    openclaw cron runs --id <job-id>
    ```
  </Step>
</Steps>

## วิธีการทำงานของ cron

- Cron ทำงาน **ภายในโปรเซส Gateway** (ไม่ใช่ภายในโมเดล)
- นิยามงานจะถูกคงไว้ที่ `~/.openclaw/cron/jobs.json` ดังนั้นการรีสตาร์ทจะไม่ทำให้ตารางเวลาหายไป
- สถานะการดำเนินงานขณะรันจะถูกคงไว้ข้างกันใน `~/.openclaw/cron/jobs-state.json` หากคุณติดตามนิยาม cron ใน git ให้ติดตาม `jobs.json` และเพิ่ม `jobs-state.json` ใน gitignore
- หลังจากแยกไฟล์แล้ว OpenClaw เวอร์ชันเก่าสามารถอ่าน `jobs.json` ได้ แต่อาจถือว่างานเป็นงานใหม่ เพราะตอนนี้ฟิลด์ขณะรันอยู่ใน `jobs-state.json`
- เมื่อแก้ไข `jobs.json` ระหว่างที่ Gateway กำลังทำงานหรือหยุดอยู่ OpenClaw จะเปรียบเทียบฟิลด์ตารางเวลาที่เปลี่ยนกับเมตาดาต้าช่วงเวลารันไทม์ที่ค้างอยู่ และล้างค่า `nextRunAtMs` ที่เก่าแล้ว การเขียนใหม่ที่เป็นเพียงการจัดรูปแบบหรือเปลี่ยนลำดับคีย์เท่านั้นจะยังคงช่วงเวลาที่ค้างอยู่ไว้
- การดำเนินการ cron ทั้งหมดจะสร้างระเบียน [งานเบื้องหลัง](/th/automation/tasks)
- เมื่อ Gateway เริ่มทำงาน งาน agent-turn แบบแยกที่เลยกำหนดจะถูกจัดตารางใหม่ให้อยู่นอกช่วงเชื่อมต่อช่องทางแทนการเล่นซ้ำทันที เพื่อให้การเริ่มต้น Discord/Telegram และการตั้งค่าคำสั่งเนทีฟยังตอบสนองได้ดีหลังรีสตาร์ท
- งานแบบครั้งเดียว (`--at`) จะลบตัวเองอัตโนมัติหลังสำเร็จโดยค่าเริ่มต้น
- การรัน cron แบบแยกจะพยายามปิดแท็บ/โปรเซสเบราว์เซอร์ที่ติดตามไว้สำหรับเซสชัน `cron:<jobId>` เมื่อการรันเสร็จสิ้น ดังนั้นระบบอัตโนมัติของเบราว์เซอร์ที่แยกออกมาจะไม่ทิ้งโปรเซสกำพร้าไว้
- การรัน cron แบบแยกที่ได้รับสิทธิ์การล้างตัวเองของ cron แบบจำกัดยังสามารถอ่านสถานะตัวตั้งเวลา รายการที่กรองเฉพาะงานปัจจุบันของตัวเอง และประวัติการรันของงานนั้นได้ เพื่อให้การตรวจสอบสถานะ/heartbeat สามารถตรวจตารางเวลาของตัวเองได้โดยไม่ต้องได้สิทธิ์แก้ไข cron ที่กว้างกว่า
- การรัน cron แบบแยกยังป้องกันการตอบรับที่เก่าแล้วด้วย หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะชั่วคราว (`on it`, `pulling everything together` และคำใบ้คล้ายกัน) และไม่มีการรันซับเอเจนต์สืบทอดใดที่ยังรับผิดชอบคำตอบสุดท้าย OpenClaw จะพร้อมต์ซ้ำหนึ่งครั้งเพื่อขอผลลัพธ์จริงก่อนส่งมอบ
- การรัน cron แบบแยกจะเลือกใช้เมตาดาต้าการปฏิเสธการดำเนินการแบบมีโครงสร้างจากการรันแบบฝังตัวก่อน จากนั้นจึงย้อนกลับไปใช้มาร์กเกอร์สรุป/เอาต์พุตสุดท้ายที่รู้จัก เช่น `SYSTEM_RUN_DENIED` และ `INVALID_REQUEST` เพื่อไม่ให้คำสั่งที่ถูกบล็อกรายงานเป็นการรันสีเขียว
- การรัน cron แบบแยกยังถือว่าความล้มเหลวของเอเจนต์ระดับการรันเป็นข้อผิดพลาดของงาน แม้จะไม่มีเพย์โหลดตอบกลับก็ตาม ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการจะเพิ่มตัวนับข้อผิดพลาดและทริกเกอร์การแจ้งเตือนความล้มเหลว แทนที่จะล้างงานว่าเสร็จสำเร็จ
- เมื่องาน agent-turn แบบแยกถึง `timeoutSeconds` cron จะยกเลิกการรันเอเจนต์ที่อยู่ข้างใต้และให้ช่วงเวลาสั้น ๆ สำหรับการล้างข้อมูล หากการรันไม่ระบายจนเสร็จ การล้างข้อมูลที่ Gateway เป็นเจ้าของจะบังคับล้างความเป็นเจ้าของเซสชันของการรันนั้นก่อนที่ cron จะบันทึกการหมดเวลา เพื่อไม่ให้งานแชตที่เข้าคิวค้างอยู่หลังเซสชันประมวลผลที่เก่าแล้ว
- หาก agent-turn แบบแยกหยุดค้างก่อนที่รันเนอร์จะเริ่มหรือก่อนการเรียกโมเดลครั้งแรก cron จะบันทึกการหมดเวลาตามเฟส เช่น `setup timed out before runner start` หรือ `stalled before first model call (last phase: context-engine)` ตัวเฝ้าระวังเหล่านี้ครอบคลุมผู้ให้บริการแบบฝังตัวและผู้ให้บริการที่มี CLI รองรับก่อนที่โปรเซส CLI ภายนอกจะเริ่มจริง และถูกจำกัดแยกจากค่า `timeoutSeconds` ที่ยาว เพื่อให้ความล้มเหลวจากการเริ่มต้นแบบเย็น/การยืนยันตัวตน/บริบทปรากฏอย่างรวดเร็ว แทนที่จะรอครบงบเวลาทั้งหมดของงาน

<Note>
การกระทบยอดงานสำหรับ cron ให้รันไทม์เป็นเจ้าของก่อน และใช้ประวัติถาวรรองลงมา: งาน cron ที่ทำงานอยู่จะยังคงมีชีวิตตราบใดที่รันไทม์ cron ยังติดตามว่างานนั้นกำลังรันอยู่ แม้แถวเซสชันลูกเก่าจะยังมีอยู่ก็ตาม เมื่อรันไทม์หยุดเป็นเจ้าของงานและช่วงผ่อนผัน 5 นาทีหมดลง การบำรุงรักษาจะตรวจบันทึกการรันที่คงไว้และสถานะงานสำหรับการรัน `cron:<jobId>:<startedAt>` ที่ตรงกัน หากประวัติถาวรนั้นแสดงผลลัพธ์ปลายทาง บัญชีแยกประเภทงานจะถูกสรุปจากผลลัพธ์นั้น มิฉะนั้นการบำรุงรักษาที่ Gateway เป็นเจ้าของสามารถทำเครื่องหมายงานเป็น `lost` ได้ การตรวจสอบแบบออฟไลน์ของ CLI สามารถกู้คืนจากประวัติถาวรได้ แต่จะไม่ถือว่าชุดงานที่ทำงานอยู่ในโปรเซสของตัวเองซึ่งว่างเปล่าเป็นหลักฐานว่าการรัน cron ที่ Gateway เป็นเจ้าของหายไปแล้ว
</Note>

## ประเภทตารางเวลา

| ชนิด    | แฟล็ก CLI  | คำอธิบาย                                             |
| ------- | --------- | ------------------------------------------------------- |
| `at`    | `--at`    | เวลาประทับแบบครั้งเดียว (ISO 8601 หรือแบบสัมพัทธ์ เช่น `20m`)    |
| `every` | `--every` | ช่วงเวลาคงที่                                          |
| `cron`  | `--cron`  | นิพจน์ cron แบบ 5 ฟิลด์หรือ 6 ฟิลด์ พร้อม `--tz` ที่ไม่บังคับ |

เวลาประทับที่ไม่มีเขตเวลาจะถือเป็น UTC เพิ่ม `--tz America/New_York` สำหรับการตั้งเวลาตามนาฬิกาท้องถิ่น

นิพจน์ที่เกิดซ้ำตรงต้นชั่วโมงจะถูกกระจายเวลาโดยอัตโนมัติสูงสุด 5 นาทีเพื่อลดโหลดพุ่งสูง ใช้ `--exact` เพื่อบังคับเวลาแบบแม่นยำ หรือ `--stagger 30s` สำหรับกรอบเวลาที่ระบุชัดเจน

### วันของเดือนและวันของสัปดาห์ใช้ตรรกะ OR

นิพจน์ cron ถูกแยกวิเคราะห์โดย [croner](https://github.com/Hexagon/croner) เมื่อทั้งฟิลด์วันของเดือนและวันของสัปดาห์ไม่ใช่ไวลด์การ์ด croner จะจับคู่เมื่อฟิลด์ **ใดฟิลด์หนึ่ง** ตรงกัน ไม่ใช่ทั้งสองฟิลด์ นี่คือพฤติกรรมมาตรฐานของ Vixie cron

```
# Intended: "9 AM on the 15th, only if it's a Monday"
# Actual:   "9 AM on every 15th, AND 9 AM on every Monday"
0 9 15 * 1
```

สิ่งนี้จะทำงานประมาณ 5-6 ครั้งต่อเดือน แทนที่จะเป็น 0-1 ครั้งต่อเดือน OpenClaw ใช้พฤติกรรม OR เริ่มต้นของ Croner ที่นี่ หากต้องการบังคับให้ตรงทั้งสองเงื่อนไข ให้ใช้ตัวปรับวันของสัปดาห์ `+` ของ Croner (`0 9 15 * +1`) หรือกำหนดตารางตามฟิลด์หนึ่งแล้วตรวจอีกฟิลด์ในพร้อมต์หรือคำสั่งของงาน

## รูปแบบการดำเนินการ

| รูปแบบ           | ค่า `--session`   | รันใน                  | เหมาะที่สุดสำหรับ                        |
| --------------- | ------------------- | ------------------------ | ------------------------------- |
| เซสชันหลัก    | `main`              | รอบ Heartbeat ถัดไป      | การเตือน เหตุการณ์ระบบ        |
| แบบแยก        | `isolated`          | `cron:<jobId>` เฉพาะ | รายงาน งานเบื้องหลัง      |
| เซสชันปัจจุบัน | `current`           | ผูกไว้ ณ เวลาสร้าง   | งานที่เกิดซ้ำซึ่งรับรู้บริบท    |
| เซสชันกำหนดเอง  | `session:custom-id` | เซสชันมีชื่อแบบถาวร | เวิร์กโฟลว์ที่ต่อยอดจากประวัติ |

### ตัวเลือกเพย์โหลดสำหรับงานแบบแยก

<ParamField path="--message" type="string" required>
  ข้อความพร้อมต์ (จำเป็นสำหรับแบบแยก)
</ParamField>
<ParamField path="--model" type="string">
  การแทนที่โมเดล ใช้โมเดลที่อนุญาตซึ่งเลือกไว้สำหรับงาน
</ParamField>
<ParamField path="--thinking" type="string">
  การแทนที่ระดับ thinking
</ParamField>
<ParamField path="--light-context" type="boolean">
  ข้ามการฉีดไฟล์ bootstrap ของเวิร์กสเปซ
</ParamField>
<ParamField path="--tools" type="string">
  จำกัดเครื่องมือที่งานสามารถใช้ได้ เช่น `--tools exec,read`
</ParamField>

`--model` ใช้โมเดลที่อนุญาตซึ่งเลือกไว้เป็นโมเดลหลักของงานนั้น ไม่เหมือนกับการแทนที่ `/model` ของเซสชันแชต: เชน fallback ที่กำหนดค่าไว้ยังคงมีผลเมื่อโมเดลหลักของงานล้มเหลว หากโมเดลที่ขอไม่ได้รับอนุญาตหรือแก้ไขไม่ได้ cron จะทำให้การรันล้มเหลวพร้อมข้อผิดพลาดการตรวจสอบที่ชัดเจน แทนที่จะ fallback ไปยังการเลือกโมเดลของเอเจนต์/ค่าเริ่มต้นของงานอย่างเงียบ ๆ

งาน Cron ยังสามารถมี `fallbacks` ระดับเพย์โหลดได้ด้วย เมื่อมีอยู่ รายการนั้นจะแทนที่เชน fallback ที่กำหนดค่าไว้สำหรับงาน ใช้ `fallbacks: []` ในเพย์โหลด/API ของงานเมื่อคุณต้องการการรัน cron แบบเข้มงวดที่ลองเฉพาะโมเดลที่เลือก หากงานมี `--model` แต่ไม่มี fallback ทั้งระดับเพย์โหลดหรือที่กำหนดค่าไว้ OpenClaw จะส่งการแทนที่ fallback ว่างอย่างชัดเจน เพื่อไม่ให้โมเดลหลักของเอเจนต์ถูกผนวกเป็นเป้าหมายลองซ้ำเพิ่มเติมแบบซ่อนอยู่

ลำดับความสำคัญของการเลือกโมเดลสำหรับงานแบบแยกคือ:

1. การแทนที่โมเดลของฮุค Gmail (เมื่อการรันมาจาก Gmail และอนุญาตให้ใช้การแทนที่นั้น)
2. `model` ในเพย์โหลดรายงาน
3. การแทนที่โมเดลเซสชัน cron ที่ผู้ใช้เลือกและจัดเก็บไว้
4. การเลือกโมเดลของเอเจนต์/ค่าเริ่มต้น

โหมดเร็วจะตามการเลือกสดที่แก้ไขแล้วเช่นกัน หากคอนฟิกของโมเดลที่เลือกมี `params.fastMode` cron แบบแยกจะใช้ค่านั้นโดยค่าเริ่มต้น การแทนที่ `fastMode` ของเซสชันที่จัดเก็บไว้ยังคงชนะคอนฟิกได้ทั้งสองทิศทาง

หากการรันแบบแยกเจอการส่งต่อจากการสลับโมเดลสด cron จะลองซ้ำด้วยผู้ให้บริการ/โมเดลที่สลับแล้ว และคงการเลือกสดนั้นไว้สำหรับการรันที่ทำงานอยู่ก่อนลองซ้ำ เมื่อการสลับนั้นมีโปรไฟล์การยืนยันตัวตนใหม่มาด้วย cron จะคงการแทนที่โปรไฟล์การยืนยันตัวตนนั้นไว้สำหรับการรันที่ทำงานอยู่ด้วย การลองซ้ำมีขอบเขตจำกัด: หลังจากความพยายามครั้งแรกบวกการลองซ้ำจากการสลับ 2 ครั้ง cron จะยกเลิกแทนที่จะวนซ้ำตลอดไป

ก่อนที่การรัน Cron แบบแยกส่วนจะเข้าสู่ตัวรันเอเจนต์ OpenClaw จะตรวจสอบเอนด์พอยต์ผู้ให้บริการภายในเครื่องที่เข้าถึงได้สำหรับผู้ให้บริการ `api: "ollama"` และ `api: "openai-completions"` ที่กำหนดค่าไว้ ซึ่งมี `baseUrl` เป็น loopback, เครือข่ายส่วนตัว หรือ `.local` หากเอนด์พอยต์นั้นล่ม การรันจะถูกบันทึกเป็น `skipped` พร้อมข้อผิดพลาดผู้ให้บริการ/โมเดลที่ชัดเจน แทนที่จะเริ่มการเรียกโมเดล ผลลัพธ์ของเอนด์พอยต์จะถูกแคชไว้ 5 นาที ดังนั้นงานจำนวนมากที่ถึงกำหนดและใช้เซิร์ฟเวอร์ Ollama, vLLM, SGLang หรือ LM Studio ภายในเครื่องเดียวกันที่ใช้งานไม่ได้ จะใช้โพรบขนาดเล็กครั้งเดียวร่วมกันแทนที่จะสร้างคำขอจำนวนมาก การรันที่ถูกข้ามจากการตรวจสอบผู้ให้บริการล่วงหน้าจะไม่เพิ่ม backoff ของข้อผิดพลาดในการดำเนินการ ให้เปิดใช้ `failureAlert.includeSkipped` เมื่อต้องการการแจ้งเตือนการข้ามซ้ำ

## การส่งมอบและเอาต์พุต

| โหมด       | สิ่งที่เกิดขึ้น                                                        |
| ---------- | ------------------------------------------------------------------- |
| `announce` | ส่งข้อความสุดท้ายแบบสำรองไปยังเป้าหมายหากเอเจนต์ไม่ได้ส่ง |
| `webhook`  | POST เพย์โหลดเหตุการณ์ที่เสร็จแล้วไปยัง URL                                |
| `none`     | ไม่มีการส่งมอบสำรองของตัวรัน                                         |

ใช้ `--announce --channel telegram --to "-1001234567890"` สำหรับการส่งมอบไปยังช่อง สำหรับหัวข้อฟอรัม Telegram ให้ใช้ `-1001234567890:topic:123`; ผู้เรียก RPC/การกำหนดค่าโดยตรงสามารถส่ง `delivery.threadId` เป็นสตริงหรือตัวเลขได้เช่นกัน เป้าหมาย Slack/Discord/Mattermost ควรใช้คำนำหน้าแบบชัดเจน (`channel:<id>`, `user:<id>`) ID ห้อง Matrix แยกตัวพิมพ์เล็กใหญ่ ให้ใช้ ID ห้องที่ตรงกันทุกประการหรือรูปแบบ `room:!room:server` จาก Matrix

เมื่อการส่งมอบแบบประกาศใช้ `channel: "last"` หรือไม่ได้ระบุ `channel` เป้าหมายที่มีคำนำหน้าผู้ให้บริการ เช่น `telegram:123` สามารถเลือกช่องก่อนที่ Cron จะย้อนกลับไปใช้ประวัติเซสชันหรือช่องที่กำหนดค่าไว้ช่องเดียว เฉพาะคำนำหน้าที่ Plugin ที่โหลดประกาศไว้เท่านั้นที่เป็นตัวเลือกผู้ให้บริการ หาก `delivery.channel` ระบุไว้อย่างชัดเจน คำนำหน้าเป้าหมายต้องระบุชื่อผู้ให้บริการเดียวกัน ตัวอย่างเช่น `channel: "whatsapp"` พร้อม `to: "telegram:123"` จะถูกปฏิเสธ แทนที่จะให้ WhatsApp ตีความ ID Telegram เป็นหมายเลขโทรศัพท์ คำนำหน้าชนิดเป้าหมายและบริการ เช่น `channel:<id>`, `user:<id>`, `imessage:<handle>` และ `sms:<number>` ยังคงเป็นไวยากรณ์เป้าหมายที่ช่องเป็นเจ้าของ ไม่ใช่ตัวเลือกผู้ให้บริการ

สำหรับงานแบบแยกส่วน การส่งมอบแชตจะใช้ร่วมกัน หากมีเส้นทางแชต เอเจนต์สามารถใช้เครื่องมือ `message` ได้แม้ว่างานจะใช้ `--no-deliver` หากเอเจนต์ส่งไปยังเป้าหมายที่กำหนดค่าไว้/ปัจจุบัน OpenClaw จะข้ามประกาศสำรอง มิฉะนั้น `announce`, `webhook` และ `none` จะควบคุมเฉพาะสิ่งที่ตัวรันทำกับคำตอบสุดท้ายหลังจากเทิร์นของเอเจนต์เท่านั้น

เมื่อเอเจนต์สร้างการเตือนความจำแบบแยกส่วนจากแชตที่ใช้งานอยู่ OpenClaw จะจัดเก็บเป้าหมายการส่งมอบสดที่เก็บรักษาไว้สำหรับเส้นทางประกาศสำรอง คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็ก เป้าหมายการส่งมอบของผู้ให้บริการจะไม่ถูกสร้างใหม่จากคีย์เหล่านั้นเมื่อมีบริบทแชตปัจจุบัน

การส่งมอบประกาศโดยนัยใช้ allowlist ของช่องที่กำหนดค่าไว้เพื่อตรวจสอบและเปลี่ยนเส้นทางเป้าหมายที่ล้าสมัย การอนุมัติจาก pairing-store ของ DM ไม่ใช่ผู้รับระบบอัตโนมัติสำรอง ให้ตั้งค่า `delivery.to` หรือกำหนดค่ารายการ `allowFrom` ของช่องเมื่องานตามกำหนดควรส่งไปยัง DM เชิงรุก

การแจ้งเตือนความล้มเหลวใช้เส้นทางปลายทางแยกต่างหาก:

- `cron.failureDestination` ตั้งค่าเริ่มต้นส่วนกลางสำหรับการแจ้งเตือนความล้มเหลว
- `job.delivery.failureDestination` แทนที่ค่านั้นเป็นรายงาน
- หากไม่ได้ตั้งค่าทั้งสองค่าและงานส่งมอบผ่าน `announce` อยู่แล้ว ตอนนี้การแจ้งเตือนความล้มเหลวจะย้อนกลับไปยังเป้าหมายประกาศหลักนั้น
- `delivery.failureDestination` รองรับเฉพาะงาน `sessionTarget="isolated"` เว้นแต่โหมดการส่งมอบหลักจะเป็น `webhook`
- `failureAlert.includeSkipped: true` เลือกให้งานหรือนโยบายการแจ้งเตือน Cron ส่วนกลางเข้าร่วมการแจ้งเตือนการรันที่ถูกข้ามซ้ำ การรันที่ถูกข้ามจะมีตัวนับการข้ามต่อเนื่องแยกต่างหาก จึงไม่กระทบ backoff ของข้อผิดพลาดในการดำเนินการ

## Webhook

Gateway สามารถเปิดเผยเอนด์พอยต์ HTTP Webhook สำหรับทริกเกอร์ภายนอก เปิดใช้ในการกำหนดค่า:

```json5
{
  hooks: {
    enabled: true,
    token: "shared-secret",
    path: "/hooks",
  },
}
```

### การยืนยันตัวตน

ทุกคำขอต้องรวมโทเคนของ hook ผ่านส่วนหัว:

- `Authorization: Bearer <token>` (แนะนำ)
- `x-openclaw-token: <token>`

โทเคนใน query string จะถูกปฏิเสธ

<AccordionGroup>
  <Accordion title="POST /hooks/wake">
    เข้าคิวเหตุการณ์ระบบสำหรับเซสชันหลัก:

    ```bash
    curl -X POST http://127.0.0.1:18789/hooks/wake \
      -H 'Authorization: Bearer SECRET' \
      -H 'Content-Type: application/json' \
      -d '{"text":"New email received","mode":"now"}'
    ```

    <ParamField path="text" type="string" required>
      คำอธิบายเหตุการณ์
    </ParamField>
    <ParamField path="mode" type="string" default="now">
      `now` หรือ `next-heartbeat`
    </ParamField>

  </Accordion>
  <Accordion title="POST /hooks/agent">
    รันเทิร์นเอเจนต์แบบแยกส่วน:

    ```bash
    curl -X POST http://127.0.0.1:18789/hooks/agent \
      -H 'Authorization: Bearer SECRET' \
      -H 'Content-Type: application/json' \
      -d '{"message":"Summarize inbox","name":"Email","model":"openai/gpt-5.4"}'
    ```

    ฟิลด์: `message` (จำเป็น), `name`, `agentId`, `wakeMode`, `deliver`, `channel`, `to`, `model`, `fallbacks`, `thinking`, `timeoutSeconds`

  </Accordion>
  <Accordion title="Hook ที่แมปไว้ (POST /hooks/<name>)">
    ชื่อ hook แบบกำหนดเองจะถูกแก้ผ่าน `hooks.mappings` ในการกำหนดค่า การแมปสามารถแปลงเพย์โหลดใดๆ ให้เป็นการทำงาน `wake` หรือ `agent` ด้วยเทมเพลตหรือการแปลงโค้ด
  </Accordion>
</AccordionGroup>

<Warning>
เก็บเอนด์พอยต์ hook ไว้หลัง loopback, tailnet หรือ reverse proxy ที่เชื่อถือได้

- ใช้โทเคน hook เฉพาะ อย่าใช้โทเคนยืนยันตัวตนของ Gateway ซ้ำ
- เก็บ `hooks.path` ไว้บนเส้นทางย่อยเฉพาะ; `/` จะถูกปฏิเสธ
- ตั้งค่า `hooks.allowedAgentIds` เพื่อจำกัดการกำหนดเส้นทาง `agentId` แบบชัดเจน
- เก็บ `hooks.allowRequestSessionKey=false` เว้นแต่คุณต้องการเซสชันที่ผู้เรียกเลือกเอง
- หากคุณเปิดใช้ `hooks.allowRequestSessionKey` ให้ตั้งค่า `hooks.allowedSessionKeyPrefixes` ด้วยเพื่อจำกัดรูปแบบคีย์เซสชันที่อนุญาต
- เพย์โหลด hook จะถูกครอบด้วยขอบเขตความปลอดภัยตามค่าเริ่มต้น

</Warning>

## การผสานรวม Gmail PubSub

เชื่อมทริกเกอร์กล่องขาเข้า Gmail เข้ากับ OpenClaw ผ่าน Google PubSub

<Note>
**ข้อกำหนดเบื้องต้น:** CLI `gcloud`, `gog` (gogcli), เปิดใช้ hook ของ OpenClaw แล้ว, Tailscale สำหรับเอนด์พอยต์ HTTPS สาธารณะ
</Note>

### การตั้งค่าด้วยตัวช่วย (แนะนำ)

```bash
openclaw webhooks gmail setup --account openclaw@gmail.com
```

คำสั่งนี้เขียนการกำหนดค่า `hooks.gmail` เปิดใช้พรีเซ็ต Gmail และใช้ Tailscale Funnel สำหรับเอนด์พอยต์ push

### การเริ่ม Gateway อัตโนมัติ

เมื่อ `hooks.enabled=true` และตั้งค่า `hooks.gmail.account` แล้ว Gateway จะเริ่ม `gog gmail watch serve` ตอนบูตและต่ออายุ watch โดยอัตโนมัติ ตั้งค่า `OPENCLAW_SKIP_GMAIL_WATCHER=1` เพื่อไม่เข้าร่วม

### การตั้งค่าแบบครั้งเดียวด้วยตนเอง

<Steps>
  <Step title="เลือกโปรเจกต์ GCP">
    เลือกโปรเจกต์ GCP ที่เป็นเจ้าของไคลเอนต์ OAuth ที่ `gog` ใช้:

    ```bash
    gcloud auth login
    gcloud config set project <project-id>
    gcloud services enable gmail.googleapis.com pubsub.googleapis.com
    ```

  </Step>
  <Step title="สร้าง topic และให้สิทธิ์การเข้าถึง Gmail push">
    ```bash
    gcloud pubsub topics create gog-gmail-watch
    gcloud pubsub topics add-iam-policy-binding gog-gmail-watch \
      --member=serviceAccount:gmail-api-push@system.gserviceaccount.com \
      --role=roles/pubsub.publisher
    ```
  </Step>
  <Step title="เริ่ม watch">
    ```bash
    gog gmail watch start \
      --account openclaw@gmail.com \
      --label INBOX \
      --topic projects/<project-id>/topics/gog-gmail-watch
    ```
  </Step>
</Steps>

```json5
      model: "openrouter/meta-llama/llama-3.3-70b-instruct:free",
      thinking: "off",
    },
  },
}
```

## การจัดการงาน

```bash
# List all jobs
openclaw cron list

# Show one job, including resolved delivery route
openclaw cron show <jobId>

# Edit a job
openclaw cron edit <jobId> --message "Updated prompt" --model "opus"

# Force run a job now
openclaw cron run <jobId>

# Run only if due
openclaw cron run <jobId> --due

# View run history
openclaw cron runs --id <jobId> --limit 50

# Delete a job
openclaw cron remove <jobId>

# Agent selection (multi-agent setups)
openclaw cron add --name "Ops sweep" --cron "0 6 * * *" --session isolated --message "Check ops queue" --agent ops
openclaw cron edit <jobId> --clear-agent
```

<Note>
หมายเหตุการแทนที่โมเดล:

- `openclaw cron add|edit --model ...` เปลี่ยนโมเดลที่งานเลือกไว้
- หากอนุญาตโมเดลนั้น ผู้ให้บริการ/โมเดลที่ตรงกันจะไปถึงการรันเอเจนต์แบบแยกส่วน
- หากไม่อนุญาตหรือไม่สามารถแก้ค่าได้ Cron จะทำให้การรันล้มเหลวด้วยข้อผิดพลาดการตรวจสอบความถูกต้องที่ชัดเจน
- เชน fallback ที่กำหนดค่าไว้ยังคงมีผล เพราะ `--model` ของ Cron เป็นตัวหลักของงาน ไม่ใช่การแทนที่ `/model` ของเซสชัน
- เพย์โหลด `fallbacks` แทนที่ fallback ที่กำหนดค่าไว้สำหรับงานนั้น; `fallbacks: []` ปิดใช้ fallback และทำให้การรันเข้มงวด
- `--model` ธรรมดาที่ไม่มีรายการ fallback แบบชัดเจนหรือที่กำหนดค่าไว้ จะไม่ตกผ่านไปยังตัวหลักของเอเจนต์เป็นเป้าหมายลองซ้ำพิเศษแบบเงียบ

</Note>

## การกำหนดค่า

```json5
{
  cron: {
    enabled: true,
    store: "~/.openclaw/cron/jobs.json",
    maxConcurrentRuns: 1,
    retry: {
      maxAttempts: 3,
      backoffMs: [60000, 120000, 300000],
      retryOn: ["rate_limit", "overloaded", "network", "server_error"],
    },
    webhookToken: "replace-with-dedicated-webhook-token",
    sessionRetention: "24h",
    runLog: { maxBytes: "2mb", keepLines: 2000 },
  },
}
```

`maxConcurrentRuns` จำกัดทั้งการส่งงาน Cron ตามกำหนดเวลาและการดำเนินการเทิร์นเอเจนต์แบบแยกส่วน เทิร์นเอเจนต์ Cron แบบแยกส่วนใช้เลนการดำเนินการ `cron-nested` เฉพาะของคิวภายใน ดังนั้นการเพิ่มค่านี้ทำให้การรัน LLM ของ Cron ที่เป็นอิสระต่อกันคืบหน้าแบบขนานได้ แทนที่จะเริ่มได้เฉพาะ wrapper Cron ชั้นนอก เลน `nested` ที่ใช้ร่วมกันและไม่ใช่ Cron จะไม่ถูกขยายด้วยการตั้งค่านี้

sidecar สถานะรันไทม์ได้มาจาก `cron.store`: store `.json` เช่น `~/clawd/cron/jobs.json` จะใช้ `~/clawd/cron/jobs-state.json` ในขณะที่เส้นทาง store ที่ไม่มีส่วนท้าย `.json` จะเติม `-state.json`

หากคุณแก้ไข `jobs.json` ด้วยมือ ให้กัน `jobs-state.json` ออกจากการควบคุมซอร์ส OpenClaw ใช้ sidecar นั้นสำหรับสล็อตที่รอดำเนินการ เครื่องหมายที่ใช้งานอยู่ เมตาดาต้าการรันล่าสุด และข้อมูลประจำตัวของกำหนดการที่บอกตัวจัดกำหนดการว่าเมื่อใดงานที่แก้ไขจากภายนอกต้องการ `nextRunAtMs` ใหม่

ปิดใช้ Cron: `cron.enabled: false` หรือ `OPENCLAW_SKIP_CRON=1`

<AccordionGroup>
  <Accordion title="พฤติกรรมการลองซ้ำ">
    **การลองซ้ำแบบครั้งเดียว**: ข้อผิดพลาดชั่วคราว (ขีดจำกัดอัตรา, โอเวอร์โหลด, เครือข่าย, ข้อผิดพลาดเซิร์ฟเวอร์) จะลองซ้ำสูงสุด 3 ครั้งด้วย exponential backoff ข้อผิดพลาดถาวรจะปิดใช้ทันที

    **การลองซ้ำแบบเกิดซ้ำ**: exponential backoff (30 วินาทีถึง 60 นาที) ระหว่างการลองซ้ำ Backoff จะรีเซ็ตหลังจากการรันสำเร็จครั้งถัดไป

  </Accordion>
  <Accordion title="การบำรุงรักษา">
    `cron.sessionRetention` (ค่าเริ่มต้น `24h`) ตัดรายการเซสชันการรันแบบแยกส่วน `cron.runLog.maxBytes` / `cron.runLog.keepLines` ตัดไฟล์ run-log โดยอัตโนมัติ
  </Accordion>
</AccordionGroup>

<AccordionGroup>
  <Accordion title="Cron ไม่ทำงาน">
    - ตรวจสอบตัวแปรสภาพแวดล้อม `cron.enabled` และ `OPENCLAW_SKIP_CRON`
    - ยืนยันว่า Gateway กำลังทำงานอย่างต่อเนื่อง
    - สำหรับกำหนดการ `cron` ให้ตรวจสอบเขตเวลา (`--tz`) เทียบกับเขตเวลาของโฮสต์
    - `reason: not-due` ในเอาต์พุตการรันหมายความว่าการรันด้วยตนเองถูกตรวจสอบด้วย `openclaw cron run <jobId> --due` และงานยังไม่ถึงกำหนด

  </Accordion>
  <Accordion title="Cron ทำงานแล้วแต่ไม่มีการส่ง">
    - โหมดการส่ง `none` หมายความว่าไม่คาดว่าจะมีการส่งสำรองจาก runner เอเจนต์ยังคงส่งโดยตรงด้วยเครื่องมือ `message` ได้เมื่อมีเส้นทางแชตพร้อมใช้งาน
    - เป้าหมายการส่งขาดหายหรือไม่ถูกต้อง (`channel`/`to`) หมายความว่าการส่งออกถูกข้าม
    - สำหรับ Matrix งานที่คัดลอกมาหรืองานเดิมที่มี ID ห้อง `delivery.to` เป็นตัวพิมพ์เล็กอาจล้มเหลวได้ เพราะ ID ห้องของ Matrix แยกแยะตัวพิมพ์ใหญ่และตัวพิมพ์เล็ก แก้ไขงานให้เป็นค่า `!room:server` หรือ `room:!room:server` ที่ตรงกันจาก Matrix
    - ข้อผิดพลาดการยืนยันตัวตนของช่องทาง (`unauthorized`, `Forbidden`) หมายความว่าการส่งถูกบล็อกด้วยข้อมูลรับรอง
    - หากการรันแบบแยกคืนค่าเฉพาะโทเค็นเงียบ (`NO_REPLY` / `no_reply`) OpenClaw จะระงับการส่งออกโดยตรงและระงับเส้นทางสรุปสำรองที่เข้าคิวไว้ด้วย ดังนั้นจะไม่มีสิ่งใดถูกโพสต์กลับไปยังแชต
    - หากเอเจนต์ควรส่งข้อความถึงผู้ใช้เอง ให้ตรวจสอบว่างานมีเส้นทางที่ใช้งานได้ (`channel: "last"` พร้อมแชตก่อนหน้า หรือช่องทาง/เป้าหมายที่ระบุชัดเจน)

  </Accordion>
  <Accordion title="Cron หรือ Heartbeat ดูเหมือนจะขัดขวางการ rollover แบบ /new-style">
    - ความสดใหม่ของการรีเซ็ตรายวันและเมื่อไม่ได้ใช้งานไม่ได้อิงตาม `updatedAt`; ดู [การจัดการเซสชัน](/th/concepts/session#session-lifecycle)
    - การปลุกของ Cron, การรัน Heartbeat, การแจ้งเตือน exec และงานบันทึกสถานะของ Gateway อาจอัปเดตแถวเซสชันสำหรับการกำหนดเส้นทาง/สถานะ แต่จะไม่ขยาย `sessionStartedAt` หรือ `lastInteractionAt`
    - สำหรับแถวเดิมที่สร้างก่อนมีฟิลด์เหล่านั้น OpenClaw สามารถกู้คืน `sessionStartedAt` จากส่วนหัวเซสชันของ transcript JSONL ได้เมื่อไฟล์ยังพร้อมใช้งาน แถว idle เดิมที่ไม่มี `lastInteractionAt` จะใช้เวลาเริ่มต้นที่กู้คืนได้นั้นเป็นค่าฐาน idle

  </Accordion>
  <Accordion title="ข้อควรระวังเกี่ยวกับเขตเวลา">
    - Cron ที่ไม่มี `--tz` จะใช้เขตเวลาของโฮสต์ Gateway
    - กำหนดการ `at` ที่ไม่มีเขตเวลาจะถือว่าเป็น UTC
    - `activeHours` ของ Heartbeat ใช้การแก้ค่าตามเขตเวลาที่กำหนดค่าไว้

  </Accordion>
</AccordionGroup>
