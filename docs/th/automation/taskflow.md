TaskFlow คือซับสเตรตการประสานลำดับการไหลที่อยู่เหนือ [งานเบื้องหลัง](/th/automation/tasks) โดยจัดการโฟลว์หลายขั้นตอนแบบคงทนที่มีสถานะ การติดตามรีวิชัน และความหมายด้านการซิงค์ของตัวเอง ในขณะที่งานแต่ละรายการยังคงเป็นหน่วยของงานที่แยกออกไปทำงานเอง

## ควรใช้ TaskFlow เมื่อใด

ใช้ TaskFlow เมื่องานครอบคลุมหลายขั้นตอนแบบลำดับต่อเนื่องหรือแตกแขนง และคุณต้องการติดตามความคืบหน้าแบบคงทนข้ามการรีสตาร์ท Gateway สำหรับการทำงานเบื้องหลังแบบเดี่ยว [task](/th/automation/tasks) ธรรมดาก็เพียงพอ

| สถานการณ์                              | ใช้                  |
| ------------------------------------- | -------------------- |
| งานเบื้องหลังรายการเดียว                 | task ธรรมดา           |
| ไปป์ไลน์หลายขั้นตอน (A แล้ว B แล้ว C) | TaskFlow (จัดการให้)  |
| สังเกตงานที่สร้างจากภายนอก      | TaskFlow (สะท้อนสถานะ) |
| การเตือนแบบครั้งเดียว                     | งาน Cron             |

## รูปแบบเวิร์กโฟลว์ตามกำหนดเวลาที่เชื่อถือได้

สำหรับเวิร์กโฟลว์ที่เกิดซ้ำ เช่น สรุปข่าวกรองตลาด ให้แยกกำหนดเวลา การประสานงาน และการตรวจสอบความน่าเชื่อถือออกเป็นคนละชั้น:

1. ใช้ [งานตามกำหนดเวลา](/th/automation/cron-jobs) สำหรับการกำหนดเวลา
2. เก็บบริบทก่อนหน้าไว้ในไฟล์ ฐานข้อมูล หรือสถานะเครื่องมือของเวิร์กโฟลว์เอง
3. ใช้ [Lobster](/th/tools/lobster) สำหรับขั้นตอนที่กำหนดผลได้แน่นอน เกตการอนุมัติ และโทเค็นสำหรับดำเนินต่อ
4. ใช้ TaskFlow เพื่อติดตามการรันหลายขั้นตอนข้ามงานย่อย การรอ การลองใหม่ และการรีสตาร์ท Gateway

รูปแบบ cron ตัวอย่าง:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

ใช้ `session:<id>` เมื่องานควรมุ่งไปยังแชต/เซสชันที่ทราบอยู่แล้วสำหรับบริบทการส่งมอบหรือการตั้งค่าความชอบเริ่มต้นอย่างปลอดภัย Cron ยังคงดำเนินการแต่ละรันในเซสชันที่แยกออกไป ดังนั้นให้ใส่สรุปจากรันก่อนหน้าและสถานะเวิร์กโฟลว์ถาวรไว้ในที่เก็บข้อมูลแบบชัดเจนที่งานอ่านได้

ภายในเวิร์กโฟลว์ ให้วางการตรวจสอบความน่าเชื่อถือไว้ก่อนขั้นตอนสรุปด้วย LLM:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

การตรวจสอบล่วงหน้าที่แนะนำ:

- ความพร้อมใช้งานของเบราว์เซอร์และการเลือกโปรไฟล์ เช่น `openclaw` สำหรับสถานะที่จัดการให้ หรือ `user` เมื่อต้องใช้เซสชัน Chrome ที่ลงชื่อเข้าใช้แล้ว ดู [เบราว์เซอร์](/th/tools/browser)
- ข้อมูลรับรอง API และโควตาสำหรับแต่ละแหล่งข้อมูล
- การเข้าถึงเครือข่ายสำหรับปลายทางที่จำเป็น
- เครื่องมือที่จำเป็นเปิดใช้งานสำหรับเอเจนต์แล้ว เช่น `lobster`, `browser` และ `llm-task`
- ปลายทางสำหรับความล้มเหลวถูกกำหนดค่าไว้สำหรับ cron เพื่อให้เห็นความล้มเหลวของการตรวจสอบล่วงหน้า ดู [งานตามกำหนดเวลา](/th/automation/cron-jobs#delivery-and-output)

ฟิลด์แหล่งที่มาของข้อมูลที่แนะนำสำหรับทุกรายการที่รวบรวม:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

ให้เวิร์กโฟลว์ปฏิเสธหรือทำเครื่องหมายรายการที่ล้าสมัยก่อนการสรุป ขั้นตอน LLM ควรได้รับเฉพาะ JSON ที่มีโครงสร้าง และควรถูกขอให้คง `sourceUrl`, `retrievedAt` และ `asOf` ไว้ในผลลัพธ์ ใช้ [LLM Task](/th/tools/llm-task) เมื่อคุณต้องการขั้นตอนโมเดลที่ตรวจสอบกับสคีมาได้ภายในเวิร์กโฟลว์

สำหรับเวิร์กโฟลว์ที่ทีมนํากลับมาใช้ซ้ำได้หรือชุมชนใช้ร่วมกัน ให้แพ็กเกจ CLI, ไฟล์ `.lobster` และบันทึกการตั้งค่าใด ๆ เป็น Skills หรือ Plugin แล้วเผยแพร่ผ่าน [ClawHub](/clawhub) เก็บการ์ดเรลเฉพาะเวิร์กโฟลว์ไว้ในแพ็กเกจนั้น เว้นแต่ว่า API ของ Plugin จะขาดความสามารถทั่วไปที่จำเป็น

## โหมดการซิงค์

### โหมดที่จัดการให้

TaskFlow เป็นเจ้าของวงจรชีวิตตั้งแต่ต้นจนจบ โดยสร้างงานเป็นขั้นตอนของโฟลว์ ขับเคลื่อนให้เสร็จสิ้น และเลื่อนสถานะของโฟลว์โดยอัตโนมัติ

ตัวอย่าง: โฟลว์รายงานรายสัปดาห์ที่ (1) รวบรวมข้อมูล (2) สร้างรายงาน และ (3) ส่งมอบรายงาน TaskFlow สร้างแต่ละขั้นตอนเป็นงานเบื้องหลัง รอจนเสร็จสิ้น แล้วจึงย้ายไปขั้นตอนถัดไป

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### โหมดสะท้อนสถานะ

TaskFlow สังเกตงานที่สร้างจากภายนอกและทำให้สถานะโฟลว์ซิงค์อยู่เสมอโดยไม่รับเป็นเจ้าของการสร้างงาน โหมดนี้มีประโยชน์เมื่องานมีต้นทางจากงาน cron, คำสั่ง CLI หรือแหล่งอื่น ๆ และคุณต้องการมุมมองรวมของความคืบหน้าในฐานะโฟลว์

ตัวอย่าง: งาน cron อิสระสามรายการที่รวมกันเป็นกิจวัตร "morning ops" โฟลว์แบบสะท้อนสถานะติดตามความคืบหน้าโดยรวมของงานเหล่านี้โดยไม่ควบคุมว่าจะรันเมื่อใดหรืออย่างไร

## สถานะคงทนและการติดตามรีวิชัน

แต่ละโฟลว์คงสถานะของตัวเองและติดตามรีวิชัน เพื่อให้ความคืบหน้าอยู่รอดจากการรีสตาร์ท Gateway การติดตามรีวิชันช่วยตรวจจับความขัดแย้งเมื่อหลายแหล่งพยายามเลื่อนโฟลว์เดียวกันพร้อมกัน
รีจิสทรีของโฟลว์ใช้ SQLite พร้อมการดูแล write-ahead log แบบมีขอบเขต รวมถึง
checkpoint เป็นระยะและตอนปิดระบบ เพื่อให้ Gateway ที่ทำงานระยะยาวไม่เก็บ
ไฟล์ sidecar `registry.sqlite-wal` แบบไร้ขอบเขต

## พฤติกรรมการยกเลิก

`openclaw tasks flow cancel` ตั้งเจตนายกเลิกแบบติดค้างไว้บนโฟลว์ งานที่กำลังทำงานอยู่ภายในโฟลว์จะถูกยกเลิก และจะไม่เริ่มขั้นตอนใหม่ เจตนายกเลิกจะคงอยู่ข้ามการรีสตาร์ท ดังนั้นโฟลว์ที่ถูกยกเลิกจะยังคงถูกยกเลิก แม้ว่า Gateway จะรีสตาร์ทก่อนที่งานย่อยทั้งหมดจะสิ้นสุด

## คำสั่ง CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| คำสั่ง                           | คำอธิบาย                                   |
| --------------------------------- | --------------------------------------------- |
| `openclaw tasks flow list`        | แสดงโฟลว์ที่ติดตามพร้อมสถานะและโหมดการซิงค์ |
| `openclaw tasks flow show <id>`   | ตรวจสอบโฟลว์หนึ่งรายการตามรหัสโฟลว์หรือคีย์ค้นหา     |
| `openclaw tasks flow cancel <id>` | ยกเลิกโฟลว์ที่กำลังทำงานและงานที่กำลังทำงานอยู่ของโฟลว์    |

## โฟลว์สัมพันธ์กับงานอย่างไร

โฟลว์ประสานงาน ไม่ได้แทนที่งาน โฟลว์หนึ่งรายการอาจขับเคลื่อนงานเบื้องหลังหลายรายการตลอดช่วงอายุของมัน ใช้ `openclaw tasks` เพื่อตรวจสอบเรคคอร์ดงานแต่ละรายการ และใช้ `openclaw tasks flow` เพื่อตรวจสอบโฟลว์ที่ทำหน้าที่ประสานงาน
