OpenClaw ใช้รหัสผู้ให้บริการเดียวคือ `openai` สำหรับทั้งการยืนยันตัวตนด้วยคีย์ API โดยตรงและ
การยืนยันตัวตนด้วยการสมัครสมาชิก ChatGPT/Codex โดย `openai/*` คือเส้นทางโมเดลมาตรฐาน
สำหรับรอบการทำงานของเอเจนต์แบบฝังตัวที่ไม่ได้กำหนดนโยบายรันไทม์หรือกำหนดเป็น `auto` ข้อมูลเส้นทาง
ของ OpenAI จะเป็นตัวตัดสินว่า OpenClaw สามารถเลือกรันไทม์เซิร์ฟเวอร์แอป Codex ที่ให้มาพร้อมกัน
โดยปริยายได้หรือไม่ คำนำหน้า `openai/*` เพียงอย่างเดียวไม่ได้เลือกรันไทม์

- **โมเดลเอเจนต์** - `openai/*` ผ่านรันไทม์ที่เลือกโดยการกำหนดค่า
  `agentRuntime` อย่างชัดเจนหรือนโยบายเส้นทางโดยปริยายของ OpenAI ลงชื่อเข้าใช้ด้วยการยืนยันตัวตน Codex
  เพื่อใช้การสมัครสมาชิก ChatGPT/Codex หรือกำหนดค่าโปรไฟล์การยืนยันตัวตนด้วยคีย์ API
  เมื่อต้องการเรียกเก็บเงินตามคีย์
- **API ของ OpenAI ที่ไม่ใช่เอเจนต์** - เข้าถึง OpenAI Platform โดยตรง โดยเรียกเก็บเงินตามการใช้งาน
  ผ่าน `OPENAI_API_KEY` หรือโปรไฟล์การยืนยันตัวตนด้วยคีย์ API `openai`
- **การกำหนดค่าแบบเก่า** - การอ้างอิง `codex/*` และ `openai-codex/*` จะได้รับการแก้ไขเป็น
  `openai/*` พร้อม `agentRuntime.id: "codex"` ที่มีขอบเขตระดับโมเดลโดย
  `openclaw doctor --fix`

OpenAI รองรับการใช้ OAuth ของการสมัครสมาชิกในเครื่องมือภายนอกและ
เวิร์กโฟลว์อย่าง OpenClaw อย่างชัดเจน

## การติดตามการใช้งานและค่าใช้จ่าย

OpenClaw แยกโควตาการสมัครสมาชิกออกจากการเรียกเก็บเงิน API ของ Platform:

- OAuth ของ ChatGPT/Codex แสดงแผนการสมัครสมาชิก ช่วงเวลาโควตา และยอดเครดิตคงเหลือ
- `OPENAI_ADMIN_KEY` แสดงค่าใช้จ่ายระดับองค์กรและการใช้งานคำขอให้โมเดลสร้างผลลัพธ์ที่ผู้ให้บริการรายงานย้อนหลัง 30 วันใน **การใช้งาน** ของ Control UI รวมถึงค่าใช้จ่ายรายวัน ยอดรวมคำขอ/โทเค็น โมเดลที่ใช้มากที่สุด และหมวดหมู่ค่าใช้จ่าย
- `OPENAI_PROJECT_ID` สามารถจำกัดขอบเขตประวัติ Admin API ให้เหลือเพียงหนึ่งโปรเจกต์ได้
- OpenClaw จะไม่ส่ง `OPENAI_API_KEY` หรือโปรไฟล์การอนุมาน `openai` ไปยัง API ระดับองค์กรโดยเด็ดขาด ข้อมูลรับรองเหล่านั้นอาจเป็นของปลายทางแบบกำหนดเอง, Azure หรือปลายทางภายในเอเจนต์

คีย์ Admin ที่ระบุอย่างชัดเจนมีลำดับความสำคัญเหนือ OAuth ประวัติที่ผู้ให้บริการรายงานจะไม่ถูกรวมเข้ากับค่าใช้จ่ายโดยประมาณที่ OpenClaw คำนวณจากเซสชัน และอาจรวมกิจกรรม API จากไคลเอนต์อื่นรวมถึงการปรับยอดเรียกเก็บเงินโดยผู้ให้บริการ

เอกสาร [แดชบอร์ดการใช้งาน API](https://help.openai.com/en/articles/10478918) ของ OpenAI อธิบายข้อกำหนดเกี่ยวกับการเป็นเจ้าขององค์กรและสิทธิ์ Usage Dashboard ที่ต้องกำหนดอย่างชัดเจนเพื่อเข้าถึงข้อมูลการใช้งาน

ผู้ให้บริการ โมเดล รันไทม์ และช่องทางเป็นคนละชั้นกัน หากป้ายกำกับเหล่านี้
เริ่มปะปนกัน โปรดอ่าน [รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes) ก่อน
เปลี่ยนการกำหนดค่า

## รันไทม์เอเจนต์โดยปริยาย

เมื่อนโยบาย `agentRuntime` ของผู้ให้บริการ/โมเดลไม่ได้กำหนดไว้หรือเป็น `auto` นโยบายเส้นทาง
ที่ผู้ให้บริการ OpenAI เป็นเจ้าของจะเลือกรันไทม์โดยปริยายจาก
ปลายทางและอะแดปเตอร์ที่มีผล:

| ข้อมูลเส้นทางที่มีผล                                                                                                                                                  | รันไทม์โดยปริยาย      |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| ปลายทาง HTTPS อย่างเป็นทางการของ Platform ที่ตรงกันทุกประการพร้อม `openai-responses` หรือปลายทาง HTTPS อย่างเป็นทางการของ ChatGPT ที่ตรงกันทุกประการพร้อม `openai-chatgpt-responses`; ไม่มีการแทนที่คำขอที่ผู้ใช้กำหนด | อาจเลือก Codex |
| อะแดปเตอร์ `openai-completions` ที่ผู้ใช้กำหนด                                                                                                                                  | OpenClaw              |
| ปลายทางแบบกำหนดเอง                                                                                                                                                        | OpenClaw              |
| ปลายทางอย่างเป็นทางการที่ตรงกันทุกประการซึ่งระบุอย่างชัดเจนโดยใช้ HTTP                                                                                                                            | ถูกปฏิเสธ              |
| เส้นทางที่มีการแทนที่คำขอระดับผู้ให้บริการ/โมเดลซึ่งผู้ใช้กำหนด                                                                                                                 | OpenClaw              |

การกำหนด `agentRuntime.id` ของผู้ให้บริการ/โมเดลอย่างชัดเจนเป็นค่าที่ไม่ใช่ค่าเริ่มต้นยังคงมีอำนาจสูงสุด
ตัวอย่างเช่น `agentRuntime.id: "openclaw"` จะคงเส้นทางที่มีสิทธิ์ใช้ Codex ไว้
บน OpenClaw ขณะที่ `agentRuntime.id: "codex"` กำหนดให้ต้องใช้ Codex และจะล้มเหลว
แบบปิดเมื่อเส้นทางที่มีผลไม่ได้รับการประกาศว่ารองรับ Codex
การเลือกรันไทม์ไม่เปลี่ยนประเภทข้อมูลรับรองหรือการเรียกเก็บเงิน: การยืนยันตัวตนด้วยคีย์ API
ของ Platform และการยืนยันตัวตนด้วยการสมัครสมาชิก ChatGPT/Codex ยังคงแยกจากกัน

`openclaw doctor --fix` ย้ายการอ้างอิงโมเดล `codex/*` และ `openai-codex/*`
แบบเก่า รหัสโปรไฟล์การยืนยันตัวตน Codex แบบเก่า และรายการลำดับการยืนยันตัวตน Codex แบบเก่าไปยัง
เส้นทางมาตรฐาน `openai` การอ้างอิงโมเดลที่ย้ายแล้วจะได้รับ
`agentRuntime.id: "codex"` ที่มีขอบเขตระดับโมเดล ใช้ `auth.order.openai` สำหรับการกำหนดค่าลำดับการยืนยันตัวตนใหม่

<Note>
การตั้งค่า OpenAI ใหม่จะใช้โมเดลหลัก GPT-5.6 เฉพาะเมื่อยังไม่ได้กำหนด
โมเดลหลัก การเพิ่มหรือรีเฟรชการยืนยันตัวตน OpenAI จะคงการเลือกที่ระบุอย่างชัดเจนเดิมไว้
รวมถึง `openai/gpt-5.5` เว้นแต่จะใช้
`models auth login --set-default` หรือ `models set` อย่างชัดเจน ใช้โปรไฟล์การยืนยันตัวตนด้วยคีย์ API
เฉพาะเมื่อต้องการใช้การยืนยันตัวตนด้วยคีย์ API สำหรับโมเดลเอเจนต์
</Note>

## ตัวอย่างจำกัดของ GPT-5.6

OpenClaw รู้จักรหัสโมเดล `openai/gpt-5.6-sol`,
`openai/gpt-5.6-terra` และ `openai/gpt-5.6-luna` ที่ตรงกันทุกประการ ทั้งสามรายการรองรับ
การใช้เหตุผลแบบ `xhigh` และ `max` ในแค็ตตาล็อกปัจจุบัน OpenAI อธิบายว่า Sol เป็น
ระดับเรือธง Terra เป็นระดับสมดุล และ Luna เป็นระดับที่รวดเร็ว
และมีต้นทุนต่ำกว่า ดู
[ประกาศเปิดตัว GPT-5.6](https://openai.com/index/previewing-gpt-5-6-sol/)
และ [คู่มือการเข้าถึง](https://help.openai.com/en/articles/20001325-a-preview-of-gpt-5-6-sol-terra-and-luna)

เมื่อใช้การยืนยันตัวตนด้วยคีย์ API ของ OpenAI โดยตรง รหัส `openai/gpt-5.6` แบบไม่ระบุระดับเป็นนามแฝงของ
Sol และเป็นค่าเริ่มต้นสำหรับการตั้งค่าใหม่ แค็ตตาล็อก Codex แบบเนทีฟจะไม่ใช้
นามแฝง API โดยตรงนั้นที่ฝั่งไคลเอนต์ และอาจแสดง
รหัส Sol, Terra และ Luna ที่ตรงกันทุกประการตามสิทธิ์เข้าถึงของเวิร์กสเปซ ดังนั้นการตั้งค่า OAuth ของ ChatGPT/Codex ใหม่
จึงใช้ `openai/gpt-5.6-sol` ตรวจสอบบัญชีปัจจุบันด้วย:

```bash
openclaw models list --provider openai
```

สิทธิ์เข้าถึงขององค์กร API และเวิร์กสเปซ Codex อาจแตกต่างกัน หาก GPT-5.6
ไม่พร้อมใช้งาน ให้เลือก GPT-5.5 อย่างชัดเจน:

```bash
openclaw models set openai/gpt-5.5
```

OpenClaw จะแสดงข้อผิดพลาดการเข้าถึงจากต้นทางและจะไม่แทนที่
การเลือก GPT-5.6 ด้วย GPT-5.5 โดยปริยาย

<Note>
เส้นทาง HTTPS อย่างเป็นทางการที่ตรงกันทุกประการและมีสิทธิ์อาจเลือก Plugin
เซิร์ฟเวอร์แอป Codex ที่ให้มาพร้อมกัน เมื่อนโยบายรันไทม์ไม่ได้กำหนดไว้หรือเป็น `auto`; เส้นทาง Completions ที่ผู้ใช้กำหนด
ปลายทางแบบกำหนดเอง และการแทนที่การขนส่งคำขอยังคงใช้ OpenClaw ปลายทาง
HTTP อย่างเป็นทางการแบบข้อความธรรมดาจะถูกปฏิเสธ การกำหนดค่ารันไทม์ระดับผู้ให้บริการ/โมเดลอย่างชัดเจนยังคง
มีอำนาจสูงสุด เรียกใช้ `openclaw doctor --fix` เพื่อแก้ไขการอ้างอิงโมเดล Codex แบบเก่าที่ยังคงค้างอยู่
การอ้างอิง `codex-cli/*` หรือการตรึงเซสชันรันไทม์แบบเก่าที่ไม่ได้กำหนดโดย
การกำหนดค่ารันไทม์อย่างชัดเจน
</Note>

## การฝังเวกเตอร์หน่วยความจำ

OpenClaw สามารถใช้ OpenAI หรือปลายทางการฝังเวกเตอร์ที่เข้ากันได้กับ OpenAI สำหรับ
การทำดัชนีและการฝังเวกเตอร์คำค้นของ `memory_search`:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
        model: "text-embedding-3-small",
      },
    },
  },
}
```

สำหรับปลายทางที่เข้ากันได้กับ OpenAI ซึ่งต้องใช้ป้ายกำกับการฝังเวกเตอร์แบบไม่สมมาตร ให้ตั้งค่า
`queryInputType` และ `documentInputType` ภายใต้ `memorySearch` โดย OpenClaw
จะส่งต่อค่าเหล่านี้เป็นฟิลด์คำขอ `input_type` เฉพาะของผู้ให้บริการ: การฝังเวกเตอร์
คำค้นใช้ `queryInputType`; ส่วนชิ้นส่วนหน่วยความจำที่จัดทำดัชนีและการจัดทำดัชนีแบบแบตช์ใช้
`documentInputType` ดูตัวอย่างฉบับเต็มได้ที่
[ข้อมูลอ้างอิงการกำหนดค่าหน่วยความจำ](/th/reference/memory-config#provider-specific-config)

    ```json5

    ```

    หากมีคีย์ API สำรอง ให้เก็บโมเดลที่เลือกไว้ภายใต้ `openai/*` และกำหนด
    ลำดับการตรวจสอบสิทธิ์ไว้ภายใต้ `openai` OpenClaw จะลองใช้การสมัครใช้บริการก่อน แล้วจึง
    ใช้คีย์ API โดยยังคงทำงานบนชุดควบคุม Codex:

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
      auth: {
        order: {
          openai: [
            "openai:user@example.com",
            "openai:api-key-backup",
          ],
        },
      },
    }
    ```

    <Note>
    การเริ่มต้นใช้งานจะไม่นำเข้าข้อมูล OAuth จาก `~/.codex` อีกต่อไป ให้ลงชื่อเข้าใช้ด้วย
    OAuth ผ่านเบราว์เซอร์ (ค่าเริ่มต้น) หรือโฟลว์รหัสอุปกรณ์ข้างต้น OpenClaw จะจัดการ
    ข้อมูลประจำตัวที่ได้ในที่เก็บการตรวจสอบสิทธิ์ของเอเจนต์ของตนเอง
    </Note>

    ### ตรวจสอบและกู้คืนการกำหนดเส้นทาง OAuth ของ Codex

    ```bash
    openclaw models status
    openclaw models auth list --provider openai
    openclaw config get agents.defaults.model --json
    openclaw config get models.providers.openai.agentRuntime --json
    ```

    สำหรับเอเจนต์ที่ระบุ ให้เพิ่ม `--agent <id>`:

    ```bash
    openclaw models status --agent <id>
    openclaw models auth list --agent <id> --provider openai
    ```

    หากการกำหนดค่าเก่ายังมีรายการอ้างอิง GPT ของ Codex แบบเดิม หรือมีการตรึงเซสชัน
    รันไทม์ OpenAI ที่ล้าสมัยโดยไม่มีการกำหนดค่ารันไทม์อย่างชัดเจน ให้ซ่อมแซมดังนี้:

    ```bash
    openclaw doctor --fix
    openclaw config validate
    ```

    หาก `models auth list --provider openai` ไม่แสดงโปรไฟล์ที่ใช้งานได้ ให้ลงชื่อเข้าใช้
    อีกครั้ง:

    ```bash
    openclaw models auth login --provider openai
    openclaw models status --probe --probe-provider openai
    ```

    ใช้ `--profile-id` สำหรับการลงชื่อเข้าใช้ OAuth ของ Codex หลายรายการในเอเจนต์เดียวกัน จากนั้น
    ควบคุมรายการเหล่านั้นผ่านลำดับการตรวจสอบสิทธิ์หรือ `/model ...@<profileId>`:

    ```bash
    openclaw models auth login --provider openai --profile-id openai:ritsuko
    openclaw models auth login --provider openai --profile-id openai:lain
    ```

    เรียกใช้ `openclaw doctor --fix` เพื่อย้าย ID โปรไฟล์และรายการลำดับ
    ที่ใช้คำนำหน้า OpenAI Codex แบบเดิม ก่อนอาศัยการจัดลำดับโปรไฟล์

    ### ตัวบ่งชี้สถานะ

    `/status` ในแชตจะแสดงว่ารันไทม์โมเดลใดกำลังใช้งานอยู่สำหรับ
    เซสชันปัจจุบัน ชุดควบคุม app-server ของ Codex ที่มาพร้อมระบบจะแสดงเป็น
    `Runtime: OpenAI Codex` เมื่อเส้นทางโดยนัยที่เข้าเกณฑ์หรือนโยบาย
    รันไทม์ของผู้ให้บริการ/โมเดลแบบชัดเจนเลือกใช้

    ### คำเตือนจาก Doctor

    หากรายการอ้างอิงโมเดล Codex แบบเดิมหรือการตรึงรันไทม์ OpenAI ที่ล้าสมัยยังคงอยู่ในการกำหนดค่า
    หรือสถานะเซสชัน `openclaw doctor --fix` จะเขียนรายการเหล่านั้นใหม่เป็น `openai/*` พร้อม
    รันไทม์ Codex เว้นแต่ OpenClaw จะได้รับการกำหนดค่าไว้อย่างชัดเจน

    ### ขีดจำกัดหน้าต่างบริบท

    OpenClaw ถือว่าข้อมูลเมตาของโมเดลและขีดจำกัดบริบทของรันไทม์เป็น
    ค่าที่แยกจากกัน สำหรับ `openai/gpt-5.5` ผ่านแค็ตตาล็อก OAuth ของ Codex:

    - `contextWindow` ดั้งเดิม: `400000`
    - ขีดจำกัด `contextTokens` เริ่มต้นของรันไทม์: `272000`

    ในทางปฏิบัติ ขีดจำกัดเริ่มต้นที่เล็กกว่าจะมีคุณลักษณะด้านเวลาแฝงและคุณภาพที่ดีกว่า
    ลบล้างค่านี้ด้วย `contextTokens`:

    ```json5
    {
      models: {
        providers: {
          openai: {
            models: [{ id: "gpt-5.5", contextTokens: 160000 }],
          },
        },
      },
    }
    ```

    <Note>
    ใช้ `contextWindow` เพื่อประกาศข้อมูลเมตาของโมเดลดั้งเดิม ใช้ `contextTokens`
    เพื่อจำกัดงบประมาณบริบทของรันไทม์ เส้นทางคีย์ API ของ OpenAI โดยตรง
    จะรายงาน `contextWindow` ดั้งเดิมที่ใหญ่กว่า (`1000000`) สำหรับ `gpt-5.5`; ระบบจะ
    ติดตามทั้งสองเส้นทางแยกกันเนื่องจากแค็ตตาล็อกต้นทางแตกต่างกัน
    </Note>

    ### การกู้คืนแค็ตตาล็อก

    OpenClaw ใช้ข้อมูลเมตาของแค็ตตาล็อก Codex ต้นทางสำหรับ `gpt-5.5` เมื่อมี
    ข้อมูลดังกล่าว หากการค้นหา Codex แบบสดไม่มีแถว `gpt-5.5` ทั้งที่บัญชี
    ผ่านการตรวจสอบสิทธิ์แล้ว OpenClaw จะสังเคราะห์แถวโมเดล OAuth ดังกล่าว เพื่อให้การทำงานของ cron,
    เอเจนต์ย่อย และโมเดลเริ่มต้นที่กำหนดค่าไว้ไม่ล้มเหลวด้วย
    `Unknown model`

## การตรวจสอบสิทธิ์ app-server ของ Codex ดั้งเดิม

ชุดควบคุม app-server ของ Codex ดั้งเดิมใช้รายการอ้างอิงโมเดล `openai/*` เมื่อเส้นทาง
HTTPS ทางการแบบตรงกันทุกประการที่เข้าเกณฑ์เลือกใช้โดยนัย หรือเมื่อ `agentRuntime.id: "codex"`
ของผู้ให้บริการ/โมเดลเลือกใช้อย่างชัดเจน การตรวจสอบสิทธิ์ยังคง
อิงตามบัญชี OpenClaw เลือกการตรวจสอบสิทธิ์ตามลำดับดังนี้:

1. โปรไฟล์การตรวจสอบสิทธิ์ OpenAI ที่เรียงลำดับแล้วสำหรับเอเจนต์ โดยควรอยู่ภายใต้
   `auth.order.openai` เรียกใช้ `openclaw doctor --fix` เพื่อย้าย ID โปรไฟล์
   การตรวจสอบสิทธิ์ Codex และลำดับการตรวจสอบสิทธิ์แบบเดิมที่เก่ากว่า
2. บัญชีที่มีอยู่ของ app-server เช่น การลงชื่อเข้าใช้ ChatGPT
   ของ Codex CLI ในเครื่อง สำหรับโฮมเอเจนต์แบบแยกส่วนเริ่มต้น OpenClaw จะเชื่อมบัญชี CLI
   ดั้งเดิมดังกล่าวเข้ากับ app-server ผ่าน RPC สำหรับเข้าสู่ระบบ โดยจะไม่แชร์
   การกำหนดค่า Plugin หรือที่เก็บเธรดของ CLI
3. เฉพาะการเปิด app-server ผ่าน stdio ในเครื่อง และเฉพาะเมื่อ app-server
   รายงานว่าไม่มีบัญชี: `CODEX_API_KEY` แล้วจึง `OPENAI_API_KEY`

การลงชื่อเข้าใช้การสมัครใช้บริการ ChatGPT/Codex ในเครื่องจะไม่ถูกแทนที่เพียงเพราะ
กระบวนการ Gateway มี `OPENAI_API_KEY` สำหรับโมเดล OpenAI โดยตรงหรือ
การฝังข้อมูลด้วย ทางเลือกสำรองคีย์ API จากสภาพแวดล้อมใช้กับเส้นทาง stdio ในเครื่องที่ไม่มีบัญชี
เท่านั้น และจะไม่ถูกส่งผ่านการเชื่อมต่อ app-server แบบ WebSocket เมื่อเลือก
โปรไฟล์ Codex แบบการสมัครใช้บริการ OpenClaw จะไม่ส่ง
`CODEX_API_KEY` และ `OPENAI_API_KEY` ไปยังกระบวนการลูก app-server แบบ stdio ที่เปิดขึ้น
และจะส่งข้อมูลประจำตัวที่เลือกผ่าน RPC สำหรับเข้าสู่ระบบของ app-server แทน

เมื่อโปรไฟล์การสมัครใช้บริการดังกล่าวถูกบล็อกเนื่องจากขีดจำกัดการใช้งาน Codex OpenClaw
จะทำเครื่องหมายโปรไฟล์ว่าถูกบล็อกจนถึงเวลารีเซ็ตที่ Codex ประกาศ และอนุญาตให้ลำดับ
การตรวจสอบสิทธิ์หมุนไปยังโปรไฟล์ `openai:*` ถัดไป โดยไม่เปลี่ยนโมเดลที่เลือก
หรือออกจากชุดควบคุม Codex เมื่อพ้นเวลารีเซ็ตแล้ว
โปรไฟล์การสมัครใช้บริการจะเข้าเกณฑ์อีกครั้ง

## การสร้างรูปภาพ

Plugin `openai` ที่มาพร้อมระบบจะลงทะเบียนการสร้างรูปภาพผ่านเครื่องมือ
`image_generate` โดยรองรับทั้งการสร้างรูปภาพด้วยคีย์ API ของ OpenAI และ OAuth ของ Codex
ผ่านรายการอ้างอิงโมเดล `openai/gpt-image-2` เดียวกัน

| ความสามารถ                | คีย์ API ของ OpenAI                     | OAuth ของ Codex                          |
| ------------------------- | ---------------------------------- | ------------------------------------ |
| รายการอ้างอิงโมเดล                 | `openai/gpt-image-2`               | `openai/gpt-image-2`                 |
| การตรวจสอบสิทธิ์                      | `OPENAI_API_KEY`                   | การลงชื่อเข้าใช้ OAuth ของ OpenAI Codex           |
| การรับส่งข้อมูล                 | OpenAI Images API                  | แบ็กเอนด์ Codex Responses              |
| จำนวนรูปภาพสูงสุดต่อคำขอ    | 4                                  | 4                                    |
| โหมดแก้ไข                 | เปิดใช้งาน (รูปภาพอ้างอิงสูงสุด 5 รูป) | เปิดใช้งาน (รูปภาพอ้างอิงสูงสุด 5 รูป)   |
| การลบล้างขนาด            | รองรับ รวมถึงขนาด 2K/4K   | รองรับ รวมถึงขนาด 2K/4K     |
| อัตราส่วนภาพ / ความละเอียด | ไม่ส่งต่อไปยัง OpenAI Images API | จับคู่กับขนาดที่รองรับเมื่อปลอดภัย |

```json5
{
  agents: {
    defaults: {
      imageGenerationModel: { primary: "openai/gpt-image-2" },
    },
  },
}
```

<Note>
ดู[การสร้างรูปภาพ](/th/tools/image-generation)สำหรับพารามิเตอร์เครื่องมือที่ใช้ร่วมกัน
การเลือกผู้ให้บริการ และพฤติกรรมการสลับเมื่อเกิดข้อผิดพลาด
</Note>

`gpt-image-2` เป็นค่าเริ่มต้นสำหรับการสร้างรูปภาพจากข้อความและการแก้ไขรูปภาพ
ของ OpenAI ส่วน `gpt-image-1.5`, `gpt-image-1` และ `gpt-image-1-mini` ยังคงใช้งานได้
ในฐานะการลบล้างโมเดลแบบชัดเจน ใช้ `openai/gpt-image-1.5` สำหรับ
เอาต์พุต PNG/WebP ที่มีพื้นหลังโปร่งใส โดย API `gpt-image-2` ปัจจุบันจะปฏิเสธ
`background: "transparent"`

สำหรับคำขอพื้นหลังโปร่งใส ให้เรียก `image_generate` พร้อม
`model: "openai/gpt-image-1.5"`, `outputFormat: "png"` หรือ `"webp"` และ
`background: "transparent"` โดยตัวเลือกผู้ให้บริการ `openai.background` รุ่นเก่า
ยังคงรองรับ OpenClaw ยังปกป้องเส้นทาง OpenAI สาธารณะและ OAuth ของ OpenAI Codex
ด้วยการเขียนคำขอโปร่งใส `openai/gpt-image-2` เริ่มต้นใหม่เป็น
`gpt-image-1.5` ส่วน Azure และปลายทางที่เข้ากันได้กับ OpenAI แบบกำหนดเองจะคง
ชื่อการปรับใช้/โมเดลที่กำหนดค่าไว้

การตั้งค่าเดียวกันนี้พร้อมใช้งานสำหรับการเรียกใช้ CLI แบบไม่มีส่วนติดต่อด้วย:

```bash
openclaw infer image generate \
  --model openai/gpt-image-1.5 \
  --output-format png \
  --background transparent \
  --prompt "A simple red circle sticker on a transparent background" \
  --json
```

ใช้แฟล็ก `--output-format` และ `--background` เดียวกันกับ
`openclaw infer image edit` เมื่อเริ่มจากไฟล์อินพุต
`--openai-background` ยังคงพร้อมใช้งานเป็นนามแฝงเฉพาะของ OpenAI ใช้
`--quality low|medium|high|auto` เพื่อควบคุมคุณภาพและต้นทุนของ OpenAI Images
ใช้ `--openai-moderation low|auto` เพื่อส่งคำใบ้การกลั่นกรองของ OpenAI จาก
`image generate` หรือ `image edit`

สำหรับการติดตั้ง OAuth ของ ChatGPT/Codex ให้ใช้รายการอ้างอิง `openai/gpt-image-2` เดิม เมื่อ
มีการกำหนดค่าโปรไฟล์ OAuth `openai` OpenClaw จะค้นหาโทเค็นการเข้าถึง OAuth
ที่จัดเก็บไว้ และส่งคำขอรูปภาพผ่านแบ็กเอนด์ Codex Responses โดยจะ
ไม่ลองใช้ `OPENAI_API_KEY` ก่อนหรือสลับไปใช้คีย์ API โดยไม่แจ้ง
กำหนดค่า `models.providers.openai` อย่างชัดเจนด้วยคีย์ API, URL ฐานแบบกำหนดเอง
หรือปลายทาง Azure เมื่อต้องการใช้เส้นทาง OpenAI Images API โดยตรง
แทน หากปลายทางรูปภาพแบบกำหนดเองดังกล่าวอยู่บน LAN/ที่อยู่ส่วนตัวที่เชื่อถือได้
ให้ตั้งค่า `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` ด้วย OpenClaw
จะบล็อกปลายทางรูปภาพภายใน/ส่วนตัวที่เข้ากันได้กับ OpenAI ต่อไป เว้นแต่จะมี
การเลือกใช้อย่างชัดเจนนี้

สร้าง:

```
/tool image_generate model=openai/gpt-image-2 prompt="โปสเตอร์เปิดตัว OpenClaw บน macOS ที่ขัดเกลาอย่างดี" size=3840x2160 count=1
```

สร้าง PNG แบบโปร่งใส:

```
/tool image_generate model=openai/gpt-image-1.5 prompt="สติกเกอร์วงกลมสีแดงเรียบง่ายบนพื้นหลังโปร่งใส" outputFormat=png background=transparent
```

แก้ไข:

```
/tool image_generate model=openai/gpt-image-2 prompt="คงรูปทรงของวัตถุไว้ และเปลี่ยนวัสดุเป็นแก้วโปร่งแสง" image=/path/to/reference.png size=1024x1536
```

| อินพุตอ้างอิง | รูปภาพ 1 รูปหรือวิดีโอ 1 รายการ                                                                 |
| การลบล้างขนาด   | รองรับสำหรับข้อความเป็นวิดีโอและรูปภาพเป็นวิดีโอ                                     |
| อัตราส่วนภาพ     | แปลงเป็นขนาดที่รองรับซึ่งใกล้เคียงที่สุด โดยไม่ส่งค่าดิบ                         |
| การลบล้างอื่นๆ  | ไม่รองรับ `resolution`, `audio`, `watermark` และจะละทิ้งพร้อมคำเตือนจากเครื่องมือ |

คำขอแปลงรูปภาพเป็นวิดีโอของ OpenAI ใช้ `POST /v1/videos` พร้อมรูปภาพ
`input_reference` การแก้ไขวิดีโอเดียวใช้ `POST /v1/videos/edits` โดยใส่วิดีโอ
ที่อัปโหลดไว้ในฟิลด์ `video`

```json5
{
  agents: {
    defaults: {
      videoGenerationModel: { primary: "openai/sora-2" },
    },
  },
}
```

<Note>
ดูพารามิเตอร์เครื่องมือที่ใช้ร่วมกัน การเลือกผู้ให้บริการ และพฤติกรรมการสลับไปใช้ระบบสำรองได้ที่ [การสร้างวิดีโอ](/th/tools/video-generation)

ผู้ให้บริการ OpenAI ประกาศ `supportsSize` แต่ไม่ได้ประกาศ `supportsAspectRatio` หรือ
`supportsResolution` เลเยอร์การปรับให้เป็นมาตรฐานที่ใช้ร่วมกันของ OpenClaw จะแปลง
`aspectRatio` ที่ร้องขอให้เป็น `size` ของ OpenAI ที่ตรงกันมากที่สุดก่อนที่
คำขอจะไปถึงผู้ให้บริการ ดังนั้นโดยทั่วไปคำขออัตราส่วนภาพจึงยังคงใช้งานได้
`resolution` ไม่มีค่าขนาดสำรองและจะถูกตัดทิ้ง โดยแจ้งแก่ผู้เรียกเป็น
`Ignored unsupported overrides for openai/<model>: resolution=<value>`
</Note>

## การเสริมพรอมต์สำหรับ GPT-5

OpenClaw เพิ่มการเสริมพรอมต์ GPT-5 ที่ใช้ร่วมกันสำหรับโมเดลตระกูล GPT-5 บน
ผู้ให้บริการ `openai` (รวมถึงการอ้างอิง Codex แบบเก่าก่อนซ่อมแซม ซึ่งจะถูกปรับ
ให้เป็น `openai/*`) ผู้ให้บริการอื่นที่ให้บริการรหัสโมเดลตระกูล GPT-5 เช่น
เส้นทาง OpenRouter หรือ opencode จะไม่ได้รับโอเวอร์เลย์นี้ เนื่องจากกำหนดเงื่อนไขตาม
รหัสผู้ให้บริการ `openai` ไม่ใช่เฉพาะรหัสโมเดล โมเดล GPT-4.x รุ่นเก่าจะ
ไม่ได้รับการเสริมนี้

ชุดทดสอบ app-server แบบเนทีฟของ Codex จะไม่ได้รับสัญญาพฤติกรรมด้านตัวตน/วินัย
การใช้เครื่องมือ หรือโอเวอร์เลย์รูปแบบการโต้ตอบที่เป็นมิตรผ่านคำสั่งสำหรับนักพัฒนา
Codex แบบเนทีฟจะคงพฤติกรรมพื้นฐาน โมเดล และเอกสารโครงการที่ Codex เป็นเจ้าของไว้
และ OpenClaw จะปิดใช้บุคลิกภาพในตัวของ Codex สำหรับเธรดแบบเนทีฟ เพื่อให้ไฟล์บุคลิกภาพ
ในพื้นที่ทำงานของเอเจนต์ยังคงเป็นแหล่งข้อมูลหลัก OpenClaw จะเพิ่มเฉพาะบริบทรันไทม์
ให้เธรด Codex แบบเนทีฟ ได้แก่ การส่งผ่านช่องทาง เครื่องมือแบบไดนามิกของ OpenClaw
การมอบหมายงานผ่าน ACP บริบทพื้นที่ทำงาน และ Skills ของ OpenClaw ข้อความคำแนะนำ
Heartbeat จากการเสริมเดียวกันนี้เป็นข้อยกเว้นเพียงอย่างเดียว กล่าวคือ รอบ Heartbeat
ของ Codex แบบเนทีฟจะได้รับข้อความนี้ โดยแทรกเป็นคำสั่งการทำงานร่วมกันเฉพาะแทนการส่ง
ผ่านฮุกการเสริมพรอมต์ที่ใช้ร่วมกัน

การเสริม GPT-5 เพิ่มสัญญาพฤติกรรมแบบมีแท็กสำหรับการคงตัวตน ความปลอดภัยในการดำเนินการ
วินัยในการใช้เครื่องมือ รูปแบบผลลัพธ์ การตรวจสอบความสมบูรณ์ และการยืนยันผลบนพรอมต์
ที่ OpenClaw ประกอบขึ้นและตรงตามเงื่อนไข พฤติกรรมการตอบกลับเฉพาะช่องทางและข้อความ
แบบเงียบยังคงอยู่ในพรอมต์ระบบที่ใช้ร่วมกันของ OpenClaw และนโยบายการส่งออก
เลเยอร์รูปแบบการโต้ตอบที่เป็นมิตรแยกจากกันและกำหนดค่าได้

| ค่า                  | ผล                                      |
| ---------------------- | ------------------------------------------- |
| `"friendly"` (ค่าเริ่มต้น) | เปิดใช้เลเยอร์รูปแบบการโต้ตอบที่เป็นมิตร |
| `"on"`                 | นามแฝงสำหรับ `"friendly"`                      |
| `"off"`                | ปิดใช้เฉพาะเลเยอร์รูปแบบที่เป็นมิตร       |

<Tabs>
  <Tab title="การกำหนดค่า">
    ```json5
    {
      agents: {
        defaults: {
          promptOverlays: {
            gpt5: { personality: "friendly" },
          },
        },
      },
    }
    ```
  </Tab>
  <Tab title="CLI">
    ```bash
    openclaw config set agents.defaults.promptOverlays.gpt5.personality off
    ```
  </Tab>
</Tabs>

<Tip>
ค่าจะไม่แยกตัวพิมพ์เล็กและตัวพิมพ์ใหญ่ขณะรันไทม์ ดังนั้นทั้ง `"Off"` และ `"off"`
จึงปิดใช้เลเยอร์รูปแบบที่เป็นมิตร
</Tip>

<Note>
ระบบยังคงอ่าน `plugins.entries.openai.config.personality` แบบเก่าเป็น
ค่าความเข้ากันได้สำรอง เมื่อไม่ได้ตั้งค่าการกำหนดค่า
`agents.defaults.promptOverlays.gpt5.personality` ที่ใช้ร่วมกัน
</Note>

<AccordionGroup>
  <Accordion title="การสังเคราะห์เสียงพูด (TTS)">
    Plugin `openai` ที่รวมมาให้จะลงทะเบียนการสังเคราะห์เสียงพูดสำหรับ
    พื้นผิว `messages.tts`

    | การตั้งค่า      | เส้นทางการกำหนดค่า                                            | ค่าเริ่มต้น                          |
    | ------------- | --------------------------------------------------------- | ----------------------------------- |
    | โมเดล        | `messages.tts.providers.openai.model`                  | `gpt-4o-mini-tts`                |
    | เสียง        | `messages.tts.providers.openai.speakerVoice`           | `coral`                          |
    | ความเร็ว        | `messages.tts.providers.openai.speed`                  | (ไม่ได้ตั้งค่า)                          |
    | คำสั่ง | `messages.tts.providers.openai.instructions`           | (ไม่ได้ตั้งค่า, เฉพาะ `gpt-4o-mini-tts`)  |
    | รูปแบบ       | `messages.tts.providers.openai.responseFormat`         | `opus` สำหรับข้อความเสียง, `mp3` สำหรับไฟล์ |
    | คีย์ API      | `messages.tts.providers.openai.apiKey`                 | ใช้ `OPENAI_API_KEY` เป็นค่าสำรอง   |
    | URL ฐาน     | `messages.tts.providers.openai.baseUrl`                | `https://api.openai.com/v1`      |
    | เนื้อหาคำขอเพิ่มเติม   | `messages.tts.providers.openai.extraBody` / `extra_body` | (ไม่ได้ตั้งค่า)                        |

    โมเดลที่ใช้ได้: `gpt-4o-mini-tts`, `tts-1`, `tts-1-hd` เสียงที่ใช้ได้:
    `alloy`, `ash`, `ballad`, `cedar`, `coral`, `echo`, `fable`, `juniper`,
    `marin`, `onyx`, `nova`, `sage`, `shimmer`, `verse`

    `extraBody` จะถูกผสานเข้าใน JSON คำขอ `/audio/speech` หลังฟิลด์ที่ OpenClaw
    สร้างขึ้น ดังนั้นให้ใช้กับปลายทางที่เข้ากันได้กับ OpenAI ซึ่งต้องใช้
    คีย์เพิ่มเติม เช่น `lang` ระบบจะละเว้นคีย์ต้นแบบ

    ```json5
    {
      messages: {
        tts: {
          providers: {
            openai: { model: "gpt-4o-mini-tts", speakerVoice: "coral" },
          },
        },
      },
    }
    ```

    <Note>
    ตั้งค่า `OPENAI_TTS_BASE_URL` เพื่อแทนที่ URL ฐานของ TTS โดยไม่กระทบ
    ปลายทาง API แชต ทั้ง OpenAI TTS และเสียง Realtime กำหนดค่าผ่าน
    คีย์ API ของ OpenAI Platform การติดตั้งที่ใช้เฉพาะ OAuth ยังคงใช้
    โมเดลแชตที่สนับสนุนโดย Codex ได้ แต่ใช้การโต้ตอบด้วยเสียงสดของ OpenAI ไม่ได้
    </Note>

  </Accordion>

    ```json5
        },
      },
    }
    ```

    โดย API สำหรับนักพัฒนากำลังทยอยเปิดให้แก่องค์กรที่ได้รับสิทธิ์เข้าถึงล่วงหน้า
    OpenClaw รู้จักตระกูลโมเดลนี้แต่ยังไม่สามารถใช้งานได้ เนื่องจากเซสชัน GPT-Live
    รองรับเฉพาะ WebRTC จัดการการผลัดกันพูดด้วยตัวเอง (ไม่มี VAD) และมอบหมายงาน

    ยังไม่ได้รองรับ การกำหนดค่าโมเดล `gpt-live-*` จะล้มเหลวแบบปิด พร้อม
    คำแนะนำสำหรับทั้งบริดจ์ WebSocket และเซสชันเบราว์เซอร์ Talk แทนการเชื่อมต่อ
    เสียงอย่างเงียบๆ โดยไม่มีสิทธิ์เข้าถึงเอเจนต์ นอกจากนี้ ในช่วงการเข้าถึงล่วงหน้า
    สิทธิ์ใช้ API ยังถูกจำกัดตามองค์กร OpenAI ให้ใช้ `gpt-realtime-2.1` (ค่าเริ่มต้น)
    ต่อไปจนกว่าจะรองรับ GPT-Live
    </Note>

    <Note>
    บริดจ์ OpenAI realtime ฝั่งแบ็กเอนด์ใช้รูปแบบเซสชัน Realtime WebSocket
    แบบ GA ซึ่งไม่ยอมรับ `session.temperature` การปรับใช้ Azure OpenAI
    ยังคงใช้งานได้ผ่าน `azureEndpoint` และ `azureDeployment` และ
    คงรูปแบบเซสชันที่เข้ากันได้กับการปรับใช้ไว้ (รวมถึง `temperature`)
    รองรับการเรียกใช้เครื่องมือแบบสองทิศทางและเสียง G.711 u-law
    </Note>

    <Note>
    ระบบจะเลือกเสียงแบบเรียลไทม์เมื่อสร้างเซสชัน OpenAI อนุญาตให้เปลี่ยนฟิลด์ส่วนใหญ่
    ของเซสชันได้ในภายหลัง แต่ไม่สามารถเปลี่ยนเสียงได้หลังจากโมเดล
    ส่งสัญญาณเสียงออกมาในเซสชันนั้นแล้ว ปัจจุบัน OpenClaw เปิดให้ใช้
    รหัสเสียง Realtime ในตัวในรูปแบบสตริง
    </Note>

    <Note>
    ฟีเจอร์ Talk ของ Control UI ใช้เซสชันแบบเรียลไทม์บนเบราว์เซอร์ของ OpenAI โดยมีข้อมูลลับไคลเอนต์ชั่วคราว
    ที่ออกโดย Gateway และแลกเปลี่ยน SDP ของ WebRTC โดยตรงจากเบราว์เซอร์
    กับ OpenAI Realtime API โดย Gateway จะออกข้อมูลลับไคลเอนต์ดังกล่าวด้วย
    ข้อมูลรับรอง `openai` ที่เลือกไว้ คีย์ที่กำหนดค่าไว้ โปรไฟล์คีย์ API และ
    `OPENAI_API_KEY` จะมีลำดับความสำคัญสูงกว่า ส่วนโปรไฟล์ OAuth `openai` หรือการเข้าสู่ระบบ
    Codex ภายนอกจะใช้เป็นทางเลือกสำรอง รีเลย์ของ Gateway และบริดจ์ WebSocket แบบเรียลไทม์
    ของแบ็กเอนด์ Voice Call ใช้ลำดับข้อมูลรับรองเดียวกันสำหรับปลายทาง OpenAI แบบเนทีฟ
    ผู้ดูแลสามารถตรวจสอบการทำงานจริงได้ด้วย
    `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts`;
    โดยส่วนที่เชื่อมต่อกับ OpenAI จะตรวจสอบทั้งบริดจ์ WebSocket ของแบ็กเอนด์และการแลกเปลี่ยน
    SDP ของ WebRTC บนเบราว์เซอร์โดยไม่บันทึกข้อมูลลับ
    ส่ง `--openai-only` เพื่อเรียกใช้สองส่วนนี้โดยไม่มีข้อมูลรับรอง Google
    </Note>

  </Accordion>

## ปลายทาง Azure OpenAI

ผู้ให้บริการ `openai` ที่รวมมาให้สามารถกำหนดเป้าหมายไปยังทรัพยากร Azure OpenAI เพื่อสร้าง
รูปภาพได้ด้วยการแทนที่ URL ฐาน ในเส้นทางการสร้างรูปภาพ OpenClaw
จะตรวจพบชื่อโฮสต์ Azure ใน `models.providers.openai.baseUrl` และเปลี่ยนไปใช้
รูปแบบคำขอของ Azure โดยอัตโนมัติ

<Note>
เสียงแบบเรียลไทม์ใช้เส้นทางการกำหนดค่าแยกต่างหาก
(`plugins.entries.voice-call.config.realtime.providers.openai.azureEndpoint`)
และไม่ได้รับผลกระทบจาก `models.providers.openai.baseUrl` โปรดดูแอคคอร์เดียน **เสียงแบบ
เรียลไทม์** ใน [เสียงและคำพูด](#voice-and-speech) สำหรับการตั้งค่า Azure
</Note>

ใช้ Azure OpenAI เมื่อ:

- มีการสมัครใช้งาน Azure OpenAI โควตา หรือข้อตกลง
  ระดับองค์กรอยู่แล้ว
- ต้องการการจัดเก็บข้อมูลตามภูมิภาคหรือการควบคุมการปฏิบัติตามข้อกำหนดที่ Azure มีให้
- ต้องการให้ทราฟฟิกอยู่ภายในเทนแนนซี Azure ที่มีอยู่

### การกำหนดค่า

สำหรับการสร้างรูปภาพผ่าน Azure โดยใช้ผู้ให้บริการ `openai` ที่รวมมาให้ ให้ชี้
`models.providers.openai.baseUrl` ไปยังทรัพยากร Azure และตั้งค่า `apiKey` เป็น
คีย์ Azure OpenAI (ไม่ใช่คีย์ OpenAI Platform):

```json5
{
  models: {
    providers: {
      openai: {
        baseUrl: "https://<your-resource>.openai.azure.com",
        apiKey: "<azure-openai-api-key>",
      },
    },
  },
}
```

OpenClaw รองรับส่วนต่อท้ายโฮสต์ Azure ต่อไปนี้สำหรับเส้นทางการสร้างรูปภาพ
ของ Azure:

- `*.openai.azure.com`
- `*.services.ai.azure.com`
- `*.cognitiveservices.azure.com`

สำหรับคำขอสร้างรูปภาพบนโฮสต์ Azure ที่รองรับ OpenClaw จะ:

- ส่งส่วนหัว `api-key` แทน `Authorization: Bearer`
- ใช้พาธที่จำกัดขอบเขตตามการปรับใช้ (`/openai/deployments/{deployment}/...`)
- เพิ่ม `?api-version=...` ต่อท้ายแต่ละคำขอ
- ใช้ระยะหมดเวลาคำขอเริ่มต้น 600 วินาทีสำหรับการเรียกสร้างรูปภาพผ่าน Azure
  ค่า `timeoutMs` ของแต่ละการเรียกยังคงแทนที่ค่าเริ่มต้นนี้

URL ฐานอื่นๆ (OpenAI สาธารณะและพร็อกซีที่เข้ากันได้กับ OpenAI) จะยังคงใช้
รูปแบบคำขอรูปภาพมาตรฐานของ OpenAI

<Note>
การกำหนดเส้นทาง Azure สำหรับเส้นทางการสร้างรูปภาพของผู้ให้บริการ `openai` ต้องใช้
OpenClaw 2026.4.22 หรือใหม่กว่า เวอร์ชันก่อนหน้าจะถือว่า
`openai.baseUrl` แบบกำหนดเองเป็นปลายทาง OpenAI สาธารณะและจะล้มเหลวเมื่อใช้กับการปรับใช้
รูปภาพของ Azure
</Note>

### เวอร์ชัน API

ตั้งค่า `AZURE_OPENAI_API_VERSION` เพื่อตรึงเวอร์ชัน Azure รุ่นตัวอย่างหรือ GA ที่ระบุ
สำหรับเส้นทางการสร้างรูปภาพผ่าน Azure:

```bash
export AZURE_OPENAI_API_VERSION="2024-12-01-preview"
```

ค่าเริ่มต้นคือ `2024-12-01-preview` เมื่อไม่ได้ตั้งค่าตัวแปร

### ชื่อโมเดลคือชื่อการปรับใช้

Azure OpenAI ผูกโมเดลเข้ากับการปรับใช้ สำหรับคำขอสร้างรูปภาพผ่าน Azure
ที่กำหนดเส้นทางผ่านผู้ให้บริการ `openai` ที่รวมมาให้ ฟิลด์ `model` ใน OpenClaw
ต้องเป็น **ชื่อการปรับใช้ Azure** ที่กำหนดค่าไว้ในพอร์ทัล Azure ไม่ใช่
รหัสโมเดล OpenAI สาธารณะ

หากสร้างการปรับใช้ชื่อ `gpt-image-2-prod` ที่ให้บริการ `gpt-image-2`:

```
/tool image_generate model=openai/gpt-image-2-prod prompt="โปสเตอร์ที่ดูสะอาดตา" size=1024x1024 count=1
```

กฎชื่อการปรับใช้เดียวกันนี้ใช้กับการเรียกสร้างรูปภาพทั้งหมดที่กำหนดเส้นทาง
ผ่านผู้ให้บริการ `openai` ที่รวมมาให้

### ความพร้อมให้บริการตามภูมิภาค

ปัจจุบันการสร้างรูปภาพผ่าน Azure พร้อมให้บริการเฉพาะบางภูมิภาค
(ตัวอย่างเช่น `eastus2`, `swedencentral`, `polandcentral`, `westus3`,
`uaenorth`) ตรวจสอบรายการภูมิภาคปัจจุบันของ Microsoft ก่อนสร้าง
การปรับใช้ และยืนยันว่าโมเดลที่ระบุพร้อมให้บริการในภูมิภาคของคุณ

### ความแตกต่างของพารามิเตอร์

Azure OpenAI และ OpenAI สาธารณะอาจไม่ยอมรับพารามิเตอร์รูปภาพชุดเดียวกันเสมอไป
Azure อาจปฏิเสธตัวเลือกที่ OpenAI สาธารณะอนุญาต (เช่นค่า
`background` บางค่าบน `gpt-image-2`) หรือเปิดให้ใช้เฉพาะในโมเดล
บางเวอร์ชัน ความแตกต่างเหล่านี้มาจาก Azure และโมเดลพื้นฐาน ไม่ใช่
OpenClaw หากคำขอ Azure ล้มเหลวด้วยข้อผิดพลาดการตรวจสอบความถูกต้อง ให้ตรวจสอบ
ชุดพารามิเตอร์ที่การปรับใช้และเวอร์ชัน API ของคุณรองรับใน
พอร์ทัล Azure

<Note>
Azure OpenAI ใช้การรับส่งข้อมูลแบบเนทีฟและพฤติกรรมความเข้ากันได้ แต่จะไม่ได้รับ
ส่วนหัวการระบุแหล่งที่มาแบบซ่อนของ OpenClaw โปรดดูแอคคอร์เดียน **เส้นทางแบบเนทีฟเทียบกับเส้นทางที่เข้ากันได้กับ OpenAI**
ใน [การกำหนดค่าขั้นสูง](#advanced-configuration)

สำหรับทราฟฟิกแชตหรือ Responses บน Azure (นอกเหนือจากการสร้างรูปภาพ) ให้ใช้
ขั้นตอนเริ่มต้นใช้งานหรือการกำหนดค่าผู้ให้บริการ Azure โดยเฉพาะ การใช้ `openai.baseUrl` เพียงอย่างเดียว
จะไม่รับรูปแบบ API/การยืนยันตัวตนของ Azure มีผู้ให้บริการ
`azure-openai-responses/*` แยกต่างหาก โปรดดูแอคคอร์เดียน Compaction
ฝั่งเซิร์ฟเวอร์ด้านล่าง
</Note>

## การกำหนดค่าขั้นสูง

ตัวอย่าง `params` รายโมเดลด้านล่างกำหนดรูปแบบคำขอผู้ให้บริการแบบฝังของ OpenClaw
การกำหนดค่าเหล่านี้เป็นพฤติกรรมคำขอที่ผู้ใช้กำหนด ดังนั้นเส้นทาง
`auto` ที่มีคุณสมบัติเหมาะสมจะยังคงทำงานบน OpenClaw แทนที่จะเลือก Codex โดยปริยาย ชุดทดสอบ app-server
แบบเนทีฟของ Codex จะจัดการการรับส่งข้อมูลและการตั้งค่าคำขอของตนเอง ส่วน
`agentRuntime.id: "codex"` ที่ระบุอย่างชัดเจนจะหยุดทำงานแบบปิดเมื่อเส้นทางที่มีผลไม่ได้ประกาศว่า
เข้ากันได้กับ Codex

<AccordionGroup>
  <Accordion title="การรับส่งข้อมูล (WebSocket เทียบกับ SSE)">
    OpenClaw ใช้ WebSocket ก่อนและใช้ SSE เป็นทางเลือกสำรอง (`"auto"`) สำหรับ `openai/*`

    ในโหมด `"auto"` OpenClaw จะ:
    - ลอง WebSocket ใหม่หนึ่งครั้งเมื่อเกิดความล้มเหลวช่วงต้น ก่อนเปลี่ยนไปใช้ SSE
    - หลังเกิดความล้มเหลว จะทำเครื่องหมาย WebSocket ว่าประสิทธิภาพลดลงเป็นเวลา 60 วินาทีและใช้ SSE
      ระหว่างช่วงพักฟื้น
    - แนบส่วนหัวระบุตัวตนของเซสชันและเทิร์นที่คงที่สำหรับการลองใหม่และ
      การเชื่อมต่อใหม่
    - ปรับตัวนับการใช้งาน (`input_tokens` / `prompt_tokens`) ให้เป็นรูปแบบเดียวกันใน
      ตัวเลือกการรับส่งข้อมูลต่างๆ

    | ค่า                | พฤติกรรม                          |
    | ---------------------- | ------------------------------------ |
    | `"auto"` (ค่าเริ่มต้น)   | ใช้ WebSocket ก่อนและใช้ SSE เป็นทางเลือกสำรอง     |
    | `"sse"`              | บังคับใช้เฉพาะ SSE                    |
    | `"websocket"`        | บังคับใช้เฉพาะ WebSocket              |

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": {
              params: { transport: "auto" },
            },
          },
        },
      },
    }
    ```

    เอกสาร OpenAI ที่เกี่ยวข้อง:
    - [Realtime API ผ่าน WebSocket](https://platform.openai.com/docs/guides/realtime-websocket)
    - [การสตรีมการตอบกลับ API (SSE)](https://platform.openai.com/docs/guides/streaming-responses)

  </Accordion>

  <Accordion title="โหมดเร็ว">
    OpenClaw มีตัวสลับโหมดเร็วที่ใช้ร่วมกันสำหรับ `openai/*`:

    - **แชต/UI:** `/fast status|auto|on|off`
    - **การกำหนดค่า:** `agents.defaults.models["<provider>/<model>"].params.fastMode`

    เมื่อเปิดใช้งาน OpenClaw จะแมปโหมดเร็วกับการประมวลผลแบบมีลำดับความสำคัญของ OpenAI
    (`service_tier = "priority"`) ค่า `service_tier` ที่มีอยู่จะยังคงเดิม
    และโหมดเร็วจะไม่เขียน `reasoning` หรือ
    `text.verbosity` ใหม่ `fastMode: "auto"` จะเริ่มการเรียกโมเดลใหม่ในโหมดเร็วจนถึง
    จุดตัดอัตโนมัติ จากนั้นจะเริ่มการลองใหม่ ทางเลือกสำรอง ผลลัพธ์เครื่องมือ หรือ
    การเรียกต่อเนื่องภายหลังโดยไม่ใช้โหมดเร็ว จุดตัดมีค่าเริ่มต้นที่ 60 วินาที
    ให้ตั้งค่า `params.fastAutoOnSeconds` บนโมเดลที่ใช้งานอยู่เพื่อเปลี่ยนค่านี้

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { fastMode: "auto", fastAutoOnSeconds: 30 } },
          },
        },
      },
    }
    ```

    <Note>
    การแทนที่ระดับเซสชันมีลำดับความสำคัญเหนือการกำหนดค่า การล้างการแทนที่เซสชันใน
    UI ของ Sessions จะทำให้เซสชันกลับไปใช้ค่าเริ่มต้นที่กำหนดไว้
    </Note>

  </Accordion>

  <Accordion title="การประมวลผลแบบมีลำดับความสำคัญ (service_tier)">
    API ของ OpenAI เปิดให้ใช้การประมวลผลแบบมีลำดับความสำคัญผ่าน `service_tier` ให้ตั้งค่าต่อ
    โมเดลใน OpenClaw:

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { serviceTier: "priority" } },
          },
        },
      },
    }
    ```

    ค่าที่รองรับ: `auto`, `default`, `flex`, `priority`

    <Warning>
    `serviceTier` จะถูกส่งต่อไปยังปลายทาง OpenAI แบบเนทีฟเท่านั้น
    (`api.openai.com`) และปลายทาง Codex แบบเนทีฟ (`chatgpt.com/backend-api`)
    หากกำหนดเส้นทางผู้ให้บริการใดผู้ให้บริการหนึ่งผ่านพร็อกซี OpenClaw จะคง
    `service_tier` ไว้โดยไม่เปลี่ยนแปลง
    </Warning>

  </Accordion>

  <Accordion title="Compaction ฝั่งเซิร์ฟเวอร์ (Responses API)">
    สำหรับโมเดล OpenAI Responses โดยตรง (`openai/*` บน `api.openai.com`)
    ตัวห่อหุ้มสตรีม OpenClaw ของ Plugin OpenAI จะเปิดใช้ Compaction ฝั่งเซิร์ฟเวอร์
    โดยอัตโนมัติ:

    - บังคับใช้ `store: true` (เว้นแต่ความเข้ากันได้ของโมเดลจะตั้งค่า `supportsStore: false`)
    - แทรก `context_management: [{ type: "compaction", compact_threshold: ... }]`
    - ค่าเริ่มต้นของ `compact_threshold`: 70% ของ `contextWindow` (หรือ `80000` เมื่อ
      ไม่มีค่า)

    การตั้งค่านี้ใช้กับเส้นทางรันไทม์ OpenClaw ในตัวและฮุกของผู้ให้บริการ OpenAI
    ที่ใช้โดยการทำงานแบบฝัง ชุดทดสอบ app-server แบบเนทีฟของ Codex จัดการ
    บริบทของตนเองผ่าน Codex และไม่ได้รับผลกระทบจากการตั้งค่านี้

    <Tabs>
      <Tab title="เปิดใช้อย่างชัดเจน">
        มีประโยชน์สำหรับปลายทางที่เข้ากันได้ เช่น Azure OpenAI Responses:

        ```json5
        {
          agents: {
            defaults: {
              models: {
                "azure-openai-responses/gpt-5.5": {
                  params: { responsesServerCompaction: true },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="เกณฑ์แบบกำหนดเอง">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: {
                    responsesServerCompaction: true,
                    responsesCompactThreshold: 120000,
                  },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="ปิดใช้งาน">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: { responsesServerCompaction: false },
                },
              },
            },
          },
        }
        ```
      </Tab>
    </Tabs>

    <Note>
    `responsesServerCompaction` ควบคุมเฉพาะการแทรก `context_management`
    โมเดล OpenAI Responses โดยตรงยังคงบังคับใช้ `store: true` เว้นแต่ความเข้ากันได้
    จะตั้งค่า `supportsStore: false`
    </Note>

  </Accordion>

  <Accordion title="โหมด GPT แบบเอเจนต์เข้มงวด">
    สำหรับโมเดลตระกูล GPT-5 ของผู้ให้บริการ `openai` ที่ทำงานผ่านรันไทม์แบบฝัง
    ของ OpenClaw นั้น OpenClaw ใช้สัญญาการดำเนินงานที่เข้มงวดกว่าซึ่งเรียกว่า
    `strict-agentic` เป็นค่าเริ่มต้นอยู่แล้ว โดยจะเปิดใช้อัตโนมัติเมื่อผู้ให้บริการที่ได้จากการแก้ไขค่าเป็น
    `openai` และรหัสโมเดลตรงกับตระกูล GPT-5 เว้นแต่การกำหนดค่า
    จะเลือกยกเลิกอย่างชัดเจน:

    ```json5
    {
      agents: {
        defaults: {
          embeddedAgent: { executionContract: "default" },
        },
      },
    }
    ```

    การตั้งค่า `"strict-agentic"` อย่างชัดเจนจะไม่ทำให้เกิดการเปลี่ยนแปลงใดๆ ในเลนที่รองรับ (เนื่องจากเป็นค่าเริ่มต้นอยู่แล้ว) และไม่มีผลกับคู่ผู้ให้บริการ/โมเดลที่ไม่รองรับ

    เมื่อเปิดใช้ `strict-agentic` OpenClaw จะ:
    - เปิดใช้ `update_plan` โดยอัตโนมัติสำหรับงานขนาดใหญ่
    - ลองดำเนินเทิร์นที่ว่างเปล่าในเชิงโครงสร้างหรือมีเพียงการให้เหตุผลอีกครั้ง โดยดำเนินการต่อด้วยคำตอบที่มองเห็นได้
    - ใช้เหตุการณ์แผนของฮาร์เนสอย่างชัดเจนเมื่อฮาร์เนสที่เลือกมีเหตุการณ์ดังกล่าว

    OpenClaw ไม่จำแนกข้อความร้อยแก้วของผู้ช่วยเพื่อตัดสินว่าเทิร์นหนึ่งเป็นแผน การอัปเดตความคืบหน้า หรือคำตอบสุดท้าย

    <Note>
    สัญญานี้อยู่ในตัวเรียกใช้เอเจนต์แบบฝังของ OpenClaw ทั้งหมด โดยไม่มีผลกับฮาร์เนส app-server แบบเนทีฟของ Codex ซึ่งจัดการลักษณะการทำงานของเทิร์นและแผนด้วยตนเอง สำหรับการเรียกใช้ Codex แบบเนทีฟ การเลือกฮาร์เนสมีความสำคัญมากกว่าการตั้งค่าสัญญาการดำเนินการ
    </Note>

  </Accordion>

  <Accordion title="เส้นทางแบบเนทีฟเทียบกับเส้นทางที่เข้ากันได้กับ OpenAI">
    OpenClaw ปฏิบัติต่อปลายทางโดยตรงของ OpenAI, Codex และ Azure OpenAI แตกต่างจากพร็อกซี `/v1` ทั่วไปที่เข้ากันได้กับ OpenAI:

    **เส้นทางแบบเนทีฟ** (`openai/*`, Azure OpenAI):
    - คง `reasoning: { effort: "none" }` ไว้เฉพาะสำหรับโมเดลที่รองรับระดับความพยายาม `none` ของ OpenAI
    - ละเว้นการให้เหตุผลที่ปิดใช้งานสำหรับโมเดลหรือพร็อกซีที่ปฏิเสธ `reasoning.effort: "none"`
    - กำหนดให้สคีมาของเครื่องมือใช้โหมดเข้มงวดเป็นค่าเริ่มต้น
    - แนบส่วนหัวการระบุแหล่งที่มาแบบซ่อนเฉพาะบนโฮสต์แบบเนทีฟที่ผ่านการตรวจสอบแล้ว (Azure OpenAI จะไม่ได้รับส่วนหัวเหล่านี้ แม้ว่าจะเป็นเส้นทางแบบเนทีฟก็ตาม)
    - คงการปรับรูปแบบคำขอเฉพาะ OpenAI ไว้ (`service_tier`, `store`, ความเข้ากันได้ด้านการให้เหตุผล, คำใบ้แคชพรอมต์)

    **เส้นทางพร็อกซี/เส้นทางที่เข้ากันได้:**
    - ใช้ลักษณะการทำงานด้านความเข้ากันได้ที่ยืดหยุ่นกว่า
    - นำ `store` ของ Completions ออกจากเพย์โหลด `openai-completions` ที่ไม่ใช่แบบเนทีฟ
    - ยอมรับ JSON แบบส่งผ่านขั้นสูงของ `params.extra_body`/`params.extraBody` สำหรับพร็อกซี Completions ที่เข้ากันได้กับ OpenAI
    - ยอมรับ `params.chat_template_kwargs` สำหรับพร็อกซี Completions ที่เข้ากันได้กับ OpenAI เช่น vLLM
    - ไม่บังคับใช้สคีมาของเครื่องมือแบบเข้มงวดหรือส่วนหัวเฉพาะแบบเนทีฟ

  </Accordion>
</AccordionGroup>
