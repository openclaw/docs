OpenClaw ใช้ ID ผู้ให้บริการเดียวคือ `openai` สำหรับทั้งการยืนยันตัวตนด้วยคีย์ API โดยตรงและ
การยืนยันตัวตนด้วยการสมัครสมาชิก ChatGPT/Codex โดย `openai/*` คือเส้นทางโมเดลมาตรฐาน
สำหรับรอบการทำงานของเอเจนต์แบบฝังตัวที่ไม่ได้กำหนดนโยบายรันไทม์หรือกำหนดเป็น `auto` ข้อมูลเส้นทาง
ของ OpenAI จะกำหนดว่า OpenClaw สามารถเลือกรันไทม์เซิร์ฟเวอร์แอป Codex ที่ให้มาพร้อมกัน
โดยปริยายได้หรือไม่ คำนำหน้า `openai/*` เพียงอย่างเดียวไม่ได้เลือกรันไทม์

- **โมเดลเอเจนต์** - `openai/*` ผ่านรันไทม์ที่เลือกโดยการกำหนดค่า
  `agentRuntime` อย่างชัดเจนหรือนโยบายเส้นทางโดยปริยายของ OpenAI ลงชื่อเข้าใช้ด้วยการยืนยันตัวตน Codex
  เพื่อใช้การสมัครสมาชิก ChatGPT/Codex หรือกำหนดค่าโปรไฟล์การยืนยันตัวตนด้วยคีย์ API
  เมื่อต้องการการเรียกเก็บเงินตามคีย์
- **API ของ OpenAI ที่ไม่ใช่เอเจนต์** - การเข้าถึง OpenAI Platform โดยตรง โดยคิดค่าบริการตามการใช้งาน
  ผ่าน `OPENAI_API_KEY` หรือโปรไฟล์การยืนยันตัวตนด้วยคีย์ API `openai`
- **การกำหนดค่าเดิม** - การอ้างอิง `codex/*` และ `openai-codex/*` จะได้รับการซ่อมแซมเป็น
  `openai/*` พร้อม `agentRuntime.id: "codex"` ที่มีขอบเขตระดับโมเดลโดย
  `openclaw doctor --fix`

OpenAI รองรับการใช้ OAuth ของการสมัครสมาชิกในเครื่องมือภายนอกและ
เวิร์กโฟลว์อย่าง OpenClaw อย่างชัดเจน

## การติดตามการใช้งานและค่าใช้จ่าย

OpenClaw แยกโควตาการสมัครสมาชิกออกจากการเรียกเก็บเงินของ Platform API:

- OAuth ของ ChatGPT/Codex แสดงแผนการสมัครสมาชิก ช่วงโควตา และยอดเครดิตคงเหลือ
- `OPENAI_ADMIN_KEY` แสดงค่าใช้จ่ายขององค์กรและการใช้งาน completions ที่ผู้ให้บริการรายงานในช่วง 30 วันใน **Usage** ของ Control UI ซึ่งรวมถึงค่าใช้จ่ายรายวัน ยอดรวมคำขอ/โทเค็น โมเดลที่ใช้มากที่สุด และหมวดหมู่ค่าใช้จ่าย
- `OPENAI_PROJECT_ID` สามารถจำกัดขอบเขตประวัติ Admin API ให้เหลือโครงการเดียวได้
- OpenClaw จะไม่ส่ง `OPENAI_API_KEY` หรือโปรไฟล์การอนุมาน `openai` ไปยัง API ระดับองค์กร เนื่องจากข้อมูลประจำตัวเหล่านั้นอาจเป็นของปลายทางแบบกำหนดเอง, Azure หรือปลายทางภายในเอเจนต์

คีย์ Admin ที่ระบุอย่างชัดเจนมีลำดับความสำคัญเหนือ OAuth ประวัติที่ผู้ให้บริการรายงานจะไม่ถูกรวมกับค่าใช้จ่ายโดยประมาณที่ OpenClaw คำนวณจากเซสชัน โดยอาจรวมกิจกรรม API จากไคลเอนต์อื่นและการปรับยอดเรียกเก็บเงินฝั่งผู้ให้บริการ

เอกสาร [แดชบอร์ดการใช้งาน API](https://help.openai.com/en/articles/10478918) ของ OpenAI อธิบายข้อกำหนดเกี่ยวกับการเป็นเจ้าขององค์กรและสิทธิ์ Usage Dashboard ที่ต้องกำหนดอย่างชัดเจนเพื่อเข้าถึงข้อมูลการใช้งาน

ผู้ให้บริการ โมเดล รันไทม์ และช่องทางเป็นคนละชั้นกัน หากป้ายกำกับเหล่านี้
ปะปนกัน โปรดอ่าน [รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes) ก่อน
เปลี่ยนการกำหนดค่า

## รันไทม์เอเจนต์โดยปริยาย

เมื่อนโยบาย `agentRuntime` ของผู้ให้บริการ/โมเดลไม่ได้กำหนดหรือเป็น `auto` นโยบายเส้นทาง
ที่ผู้ให้บริการ OpenAI เป็นเจ้าของจะเลือกรันไทม์โดยปริยายจาก
ปลายทางและอะแดปเตอร์ที่มีผล:

| ข้อมูลเส้นทางที่มีผล                                                                                                                                                  | รันไทม์โดยปริยาย      |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| ปลายทาง HTTPS ของ Platform อย่างเป็นทางการที่ตรงกันทุกประการพร้อม `openai-responses` หรือปลายทาง HTTPS ของ ChatGPT อย่างเป็นทางการที่ตรงกันทุกประการพร้อม `openai-chatgpt-responses`; ไม่มีการแทนที่คำขอที่ผู้ใช้กำหนด | อาจเลือก Codex ได้ |
| อะแดปเตอร์ `openai-completions` ที่ผู้ใช้กำหนด                                                                                                                                  | OpenClaw              |
| ปลายทางแบบกำหนดเอง                                                                                                                                                        | OpenClaw              |
| ปลายทางอย่างเป็นทางการที่ระบุชัดเจนและตรงกันทุกประการโดยใช้ HTTP                                                                                                                            | ถูกปฏิเสธ              |
| เส้นทางที่มีการแทนที่คำขอระดับผู้ให้บริการ/โมเดลที่ผู้ใช้กำหนด                                                                                                                 | OpenClaw              |

`agentRuntime.id` ของผู้ให้บริการ/โมเดลที่ระบุชัดเจนและไม่ใช่ค่าเริ่มต้นยังคงมีผลสูงสุด
ตัวอย่างเช่น `agentRuntime.id: "openclaw"` จะคงเส้นทางที่มีสิทธิ์ใช้ Codex
ไว้บน OpenClaw ขณะที่ `agentRuntime.id: "codex"` กำหนดให้ใช้ Codex และจะหยุดทำงานโดยปฏิเสธ
เมื่อเส้นทางที่มีผลไม่ได้รับการประกาศว่ารองรับ Codex
การเลือกรันไทม์ไม่เปลี่ยนประเภทข้อมูลประจำตัวหรือการเรียกเก็บเงิน การยืนยันตัวตนด้วยคีย์ API ของ Platform
และการยืนยันตัวตนด้วยการสมัครสมาชิก ChatGPT/Codex ยังคงแยกจากกัน

`openclaw doctor --fix` ย้ายการอ้างอิงโมเดล `codex/*` และ `openai-codex/*` แบบเดิม
ID โปรไฟล์การยืนยันตัวตน Codex แบบเดิม และรายการลำดับการยืนยันตัวตน Codex แบบเดิมไปยัง
เส้นทางมาตรฐาน `openai` การอ้างอิงโมเดลที่ย้ายแล้วจะได้รับ
`agentRuntime.id: "codex"` ที่มีขอบเขตระดับโมเดล ใช้ `auth.order.openai` สำหรับการกำหนดค่าลำดับการยืนยันตัวตนใหม่

<Note>
การตั้งค่า OpenAI ใหม่จะใช้ GPT-5.6 เป็นโมเดลหลักเฉพาะเมื่อยังไม่ได้กำหนด
โมเดลหลัก การเพิ่มหรือรีเฟรชการยืนยันตัวตน OpenAI จะรักษาตัวเลือกที่ระบุไว้อย่างชัดเจน
ซึ่งรวมถึง `openai/gpt-5.5` เว้นแต่จะใช้
`models auth login --set-default` หรือ `models set` อย่างชัดเจน ใช้โปรไฟล์การยืนยันตัวตนด้วยคีย์ API
เฉพาะเมื่อต้องการใช้การยืนยันตัวตนด้วยคีย์ API สำหรับโมเดลเอเจนต์
</Note>

## GPT-5.6 รุ่นตัวอย่างแบบจำกัด

OpenClaw รู้จัก ID โมเดล `openai/gpt-5.6-sol`,
`openai/gpt-5.6-terra` และ `openai/gpt-5.6-luna` ที่ตรงกันทุกประการ ทั้งสามรายการรองรับ
การให้เหตุผลแบบ `xhigh` และ `max` ในแค็ตตาล็อกปัจจุบัน OpenAI อธิบายว่า Sol เป็น
ระดับเรือธง Terra เป็นระดับสมดุล และ Luna เป็นระดับที่รวดเร็ว
และมีต้นทุนต่ำกว่า ดู
[ประกาศเปิดตัว GPT-5.6](https://openai.com/index/previewing-gpt-5-6-sol/)
และ[คู่มือการเข้าถึง](https://help.openai.com/en/articles/20001325-a-preview-of-gpt-5-6-sol-terra-and-luna)

เมื่อใช้การยืนยันตัวตนด้วยคีย์ API ของ OpenAI โดยตรง ID `openai/gpt-5.6` แบบไม่ระบุส่วนต่อท้ายเป็นนามแฝงของ
Sol และเป็นค่าเริ่มต้นสำหรับการตั้งค่าใหม่ แค็ตตาล็อก Codex แบบเนทีฟจะไม่นำนามแฝง API โดยตรงนั้น
มาใช้ฝั่งไคลเอนต์ โดยอาจแสดง ID ของ Sol, Terra และ Luna ที่ตรงกันทุกประการ
ตามสิทธิ์เข้าถึงของเวิร์กสเปซ ดังนั้นการตั้งค่า OAuth ของ ChatGPT/Codex ใหม่
จึงใช้ `openai/gpt-5.6-sol` ตรวจสอบบัญชีปัจจุบันด้วย:

```bash
openclaw models list --provider openai
```

สิทธิ์เข้าถึงขององค์กร API และเวิร์กสเปซ Codex อาจแตกต่างกัน หาก GPT-5.6
ไม่พร้อมใช้งาน ให้เลือก GPT-5.5 อย่างชัดเจน:

```bash
openclaw models set openai/gpt-5.5
```

OpenClaw จะแสดงข้อผิดพลาดการเข้าถึงจากต้นทางและจะไม่แทนที่
ตัวเลือก GPT-5.6 ด้วย GPT-5.5 โดยไม่แจ้ง

<Note>
เส้นทาง HTTPS อย่างเป็นทางการที่ตรงกันทุกประการและมีสิทธิ์อาจเลือก Plugin เซิร์ฟเวอร์แอป Codex
ที่ให้มาพร้อมกันได้เมื่อนโยบายรันไทม์ไม่ได้กำหนดหรือเป็น `auto`; เส้นทาง Completions ที่ผู้ใช้กำหนด
ปลายทางแบบกำหนดเอง และการแทนที่การขนส่งคำขอจะยังคงใช้ OpenClaw ปลายทาง HTTP
อย่างเป็นทางการแบบข้อความธรรมดาจะถูกปฏิเสธ การกำหนดค่ารันไทม์ระดับผู้ให้บริการ/โมเดลที่ระบุชัดเจนยังคง
มีผลสูงสุด เรียกใช้ `openclaw doctor --fix` เพื่อซ่อมแซมการอ้างอิงโมเดล Codex แบบเดิมที่ค้างอยู่
การอ้างอิง `codex-cli/*` หรือการปักหมุดเซสชันรันไทม์เก่าที่ไม่ได้ตั้งค่าโดย
การกำหนดค่ารันไทม์อย่างชัดเจน
</Note>

เข้ากับเซสชัน OpenAI

## เอ็มเบดดิงหน่วยความจำ

OpenClaw สามารถใช้ OpenAI หรือเอนด์พอยต์เอ็มเบดดิงที่เข้ากันได้กับ OpenAI สำหรับ
การจัดทำดัชนี `memory_search` และเอ็มเบดดิงคำค้น:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
        model: "text-embedding-3-small",
      },
    },
  },
}
```

สำหรับเอนด์พอยต์ที่เข้ากันได้กับ OpenAI ซึ่งต้องใช้ป้ายกำกับเอ็มเบดดิงแบบอสมมาตร ให้ตั้งค่า
`queryInputType` และ `documentInputType` ภายใต้ `memorySearch` OpenClaw
จะส่งต่อค่าเหล่านี้เป็นฟิลด์คำขอ `input_type` เฉพาะผู้ให้บริการ: เอ็มเบดดิง
คำค้นใช้ `queryInputType`; ส่วนชิ้นข้อมูลหน่วยความจำที่จัดทำดัชนีและการจัดทำดัชนีแบบแบตช์ใช้
`documentInputType` ดูตัวอย่างฉบับเต็มได้ใน
[ข้อมูลอ้างอิงการกำหนดค่าหน่วยความจำ](/th/reference/memory-config#provider-specific-config)

    ### ตัวอย่างการกำหนดค่า

    ```json5
    {
      env: { OPENAI_API_KEY: "example-openai-key-not-real" },

    }
    ```

    รหัส `gpt-5.6` ของ API โดยตรงที่ไม่ระบุเทียร์จะแก้ไขเป็นเทียร์ Sol หากองค์กร API นี้

    `openai/gpt-5.5` อย่างชัดเจน

    หากต้องการลองใช้โมเดล Instant ปัจจุบันของ ChatGPT จาก OpenAI API ให้ตั้งค่าโมเดล

    ### ตัวอย่างการกำหนดค่า

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
    }
    ```

    เมื่อมีคีย์ API สำรอง ให้คงโมเดลที่เลือกไว้ภายใต้ `openai/*` และกำหนด
    ลำดับการยืนยันตัวตนไว้ภายใต้ `openai` OpenClaw จะลองใช้การสมัครสมาชิกก่อน แล้วจึง
    ใช้คีย์ API โดยยังคงทำงานบนชุดควบคุม Codex:

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
      auth: {
        order: {
          openai: [
            "openai:user@example.com",
            "openai:api-key-backup",
          ],
        },
      },
    }
    ```

    <Note>
    การเริ่มต้นใช้งานจะไม่นำเข้าข้อมูล OAuth จาก `~/.codex` อีกต่อไป ลงชื่อเข้าใช้ด้วย
    OAuth ผ่านเบราว์เซอร์ (ค่าเริ่มต้น) หรือขั้นตอนรหัสอุปกรณ์ข้างต้น OpenClaw จะจัดการ
    ข้อมูลประจำตัวที่ได้ในที่จัดเก็บการยืนยันตัวตนของเอเจนต์ของตนเอง
    </Note>

    ### ตรวจสอบและกู้คืนการกำหนดเส้นทาง OAuth ของ Codex

    ```bash
    openclaw models status
    openclaw models auth list --provider openai
    openclaw config get agents.defaults.model --json
    openclaw config get models.providers.openai.agentRuntime --json
    ```

    สำหรับเอเจนต์ที่ระบุ ให้เพิ่ม `--agent <id>`:

    ```bash
    openclaw models status --agent <id>
    openclaw models auth list --agent <id> --provider openai
    ```

    หากการกำหนดค่ารุ่นเก่ายังคงมีการอ้างอิง GPT ของ Codex แบบเดิม หรือมีการปักหมุด
    เซสชันรันไทม์ OpenAI ที่ล้าสมัยโดยไม่มีการกำหนดค่ารันไทม์อย่างชัดเจน ให้ซ่อมแซมดังนี้:

    ```bash
    openclaw doctor --fix
    openclaw config validate
    ```

    หาก `models auth list --provider openai` ไม่แสดงโปรไฟล์ที่ใช้งานได้ ให้ลงชื่อเข้าใช้
    อีกครั้ง:

    ```bash
    openclaw models auth login --provider openai
    openclaw models status --probe --probe-provider openai
    ```

    ใช้ `--profile-id` สำหรับการลงชื่อเข้าใช้ OAuth ของ Codex หลายรายการในเอเจนต์เดียวกัน จากนั้น
    ควบคุมรายการเหล่านั้นผ่านลำดับการยืนยันตัวตนหรือ `/model ...@<profileId>`:

    ```bash
    openclaw models auth login --provider openai --profile-id openai:ritsuko
    openclaw models auth login --provider openai --profile-id openai:lain
    ```

    เรียกใช้ `openclaw doctor --fix` เพื่อย้าย ID โปรไฟล์และรายการลำดับ
    ที่ใช้คำนำหน้า OpenAI Codex แบบเดิม ก่อนใช้ลำดับโปรไฟล์

    ### ตัวบ่งชี้สถานะ

    `/status` ในแชตจะแสดงว่ารันไทม์โมเดลใดกำลังใช้งานอยู่สำหรับ
    เซสชันปัจจุบัน ชุดควบคุม app-server ของ Codex ที่รวมมาให้จะแสดงเป็น
    `Runtime: OpenAI Codex` เมื่อเส้นทางโดยนัยที่มีสิทธิ์หรือนโยบายรันไทม์
    ผู้ให้บริการ/โมเดลที่ระบุอย่างชัดเจนเลือกใช้งาน

    ### คำเตือนจาก Doctor

    หากการอ้างอิงโมเดล Codex แบบเดิมหรือการปักหมุดรันไทม์ OpenAI ที่ล้าสมัยยังคงอยู่ในการกำหนดค่า
    หรือสถานะเซสชัน `openclaw doctor --fix` จะเขียนรายการเหล่านั้นใหม่เป็น `openai/*` พร้อม
    รันไทม์ Codex เว้นแต่ OpenClaw จะได้รับการกำหนดค่าไว้อย่างชัดเจน

    ### ขีดจำกัดหน้าต่างบริบท

    OpenClaw ถือว่าข้อมูลเมตาของโมเดลและขีดจำกัดบริบทของรันไทม์เป็น
    ค่าที่แยกจากกัน สำหรับ `openai/gpt-5.5` ผ่านแค็ตตาล็อก OAuth ของ Codex:

    - `contextWindow` แบบเนทีฟ: `400000`
    - ขีดจำกัด `contextTokens` เริ่มต้นของรันไทม์: `272000`

    ในทางปฏิบัติ ขีดจำกัดเริ่มต้นที่เล็กกว่าจะให้คุณลักษณะด้านเวลาแฝงและคุณภาพ
    ที่ดีกว่า เขียนทับได้ด้วย `contextTokens`:

    ```json5
    {
      models: {
        providers: {
          openai: {
            models: [{ id: "gpt-5.5", contextTokens: 160000 }],
          },
        },
      },
    }
    ```

    <Note>
    ใช้ `contextWindow` เพื่อประกาศข้อมูลเมตาของโมเดลแบบเนทีฟ ใช้ `contextTokens`
    เพื่อจำกัดงบประมาณบริบทของรันไทม์ เส้นทางคีย์ API โดยตรงของ OpenAI
    รายงาน `contextWindow` แบบเนทีฟที่ใหญ่กว่า (`1000000`) สำหรับ `gpt-5.5`; ทั้งสอง
    เส้นทางได้รับการติดตามแยกกันเนื่องจากแค็ตตาล็อกต้นทางแตกต่างกัน
    </Note>

    ### การกู้คืนแค็ตตาล็อก

    OpenClaw ใช้ข้อมูลเมตาแค็ตตาล็อก Codex จากต้นทางสำหรับ `gpt-5.5` เมื่อมี
    ข้อมูลดังกล่าว หากการค้นหา Codex แบบสดไม่มีแถว `gpt-5.5` ขณะที่บัญชี
    ได้รับการยืนยันตัวตนแล้ว OpenClaw จะสังเคราะห์แถวโมเดล OAuth ดังกล่าวเพื่อให้การทำงาน
    ของ cron, เอเจนต์ย่อย และโมเดลเริ่มต้นที่กำหนดค่าไว้ไม่ล้มเหลวด้วย
    `Unknown model`

## การยืนยันตัวตนของ app-server Codex แบบเนทีฟ

ชุดควบคุม app-server Codex แบบเนทีฟใช้การอ้างอิงโมเดล `openai/*` เมื่อเส้นทาง HTTPS
ทางการแบบตรงกันทุกประการที่มีสิทธิ์เลือกใช้งานโดยนัย หรือเมื่อ `agentRuntime.id: "codex"`
ของผู้ให้บริการ/โมเดลเลือกใช้งานอย่างชัดเจน การยืนยันตัวตนยังคง
อิงตามบัญชี OpenClaw เลือกการยืนยันตัวตนตามลำดับนี้:

1. โปรไฟล์การยืนยันตัวตน OpenAI ที่จัดลำดับไว้สำหรับเอเจนต์ โดยควรอยู่ภายใต้
   `auth.order.openai` เรียกใช้ `openclaw doctor --fix` เพื่อย้าย ID โปรไฟล์
   การยืนยันตัวตน Codex แบบเดิมและลำดับการยืนยันตัวตน
2. บัญชีที่มีอยู่ของ app-server เช่น การลงชื่อเข้าใช้ ChatGPT
   ใน Codex CLI ภายในเครื่อง สำหรับโฮมเอเจนต์แบบแยกที่เป็นค่าเริ่มต้น OpenClaw จะเชื่อมบัญชี CLI
   แบบเนทีฟนั้นเข้าสู่ app-server ผ่าน RPC การเข้าสู่ระบบ โดยจะไม่ใช้การกำหนดค่า
   Plugin หรือที่จัดเก็บเธรดของ CLI ร่วมกัน
3. เฉพาะการเปิด app-server ผ่าน stdio ภายในเครื่อง และเฉพาะเมื่อ app-server
   รายงานว่าไม่มีบัญชี: `CODEX_API_KEY` แล้วจึง `OPENAI_API_KEY`

การลงชื่อเข้าใช้การสมัครสมาชิก ChatGPT/Codex ภายในเครื่องจะไม่ถูกแทนที่เพียงเพราะ
กระบวนการ Gateway มี `OPENAI_API_KEY` สำหรับโมเดล OpenAI โดยตรงหรือ
การฝังด้วย การใช้คีย์ API จากสภาพแวดล้อมเป็นทางเลือกสำรองจะใช้เฉพาะเส้นทาง stdio ภายในเครื่อง
ที่ไม่มีบัญชีเท่านั้น และจะไม่ถูกส่งผ่านการเชื่อมต่อ app-server แบบ WebSocket เมื่อเลือก
โปรไฟล์ Codex แบบการสมัครสมาชิก OpenClaw จะกัน
`CODEX_API_KEY` และ `OPENAI_API_KEY` ออกจากโปรเซสลูก app-server แบบ stdio ที่เรียกขึ้นมา
และส่งข้อมูลประจำตัวที่เลือกผ่าน RPC การเข้าสู่ระบบของ app-server แทน

เมื่อโปรไฟล์การสมัครสมาชิกดังกล่าวถูกบล็อกโดยขีดจำกัดการใช้งาน Codex OpenClaw
จะทำเครื่องหมายโปรไฟล์ว่าถูกบล็อกจนถึงเวลารีเซ็ตที่ Codex ประกาศ และให้ลำดับ
การยืนยันตัวตนหมุนไปยังโปรไฟล์ `openai:*` ถัดไป โดยไม่เปลี่ยนโมเดลที่เลือก
หรือออกจากชุดควบคุม Codex เมื่อพ้นเวลารีเซ็ตแล้ว
โปรไฟล์การสมัครสมาชิกจะมีสิทธิ์ใช้งานอีกครั้ง

## การสร้างรูปภาพ

Plugin `openai` ที่รวมมาให้จะลงทะเบียนการสร้างรูปภาพผ่านเครื่องมือ
`image_generate` โดยรองรับทั้งการสร้างรูปภาพด้วยคีย์ API ของ OpenAI และ OAuth ของ Codex
ผ่านการอ้างอิงโมเดล `openai/gpt-image-2` เดียวกัน

| ความสามารถ                | คีย์ API ของ OpenAI                     | OAuth ของ Codex                          |
| ------------------------- | ---------------------------------- | ------------------------------------ |
| การอ้างอิงโมเดล                 | `openai/gpt-image-2`               | `openai/gpt-image-2`                 |
| การยืนยันตัวตน                      | `OPENAI_API_KEY`                   | การลงชื่อเข้าใช้ OAuth ของ OpenAI Codex           |
| การรับส่งข้อมูล                 | OpenAI Images API                  | แบ็กเอนด์ Codex Responses              |
| จำนวนรูปภาพสูงสุดต่อคำขอ    | 4                                  | 4                                    |
| โหมดแก้ไข                 | เปิดใช้งาน (รูปภาพอ้างอิงสูงสุด 5 รูป) | เปิดใช้งาน (รูปภาพอ้างอิงสูงสุด 5 รูป)   |
| การเขียนทับขนาด            | รองรับ รวมถึงขนาด 2K/4K   | รองรับ รวมถึงขนาด 2K/4K     |
| อัตราส่วนภาพ / ความละเอียด | ไม่ส่งต่อไปยัง OpenAI Images API | จับคู่กับขนาดที่รองรับเมื่อทำได้อย่างปลอดภัย |

```json5
{
  agents: {
    defaults: {
      imageGenerationModel: { primary: "openai/gpt-image-2" },
    },
  },
}
```

<Note>
ดู[การสร้างรูปภาพ](/th/tools/image-generation)สำหรับพารามิเตอร์เครื่องมือที่ใช้ร่วมกัน
การเลือกผู้ให้บริการ และลักษณะการทำงานเมื่อสลับไปใช้ตัวสำรอง
</Note>

`gpt-image-2` เป็นค่าเริ่มต้นสำหรับการสร้างรูปภาพจากข้อความและการแก้ไข
รูปภาพของ OpenAI ส่วน `gpt-image-1.5`, `gpt-image-1` และ `gpt-image-1-mini` ยังคงใช้งานได้
โดยระบุเป็นการเขียนทับโมเดลอย่างชัดเจน ใช้ `openai/gpt-image-1.5` สำหรับ
เอาต์พุต PNG/WebP ที่มีพื้นหลังโปร่งใส โดย API `gpt-image-2` ปัจจุบันปฏิเสธ
`background: "transparent"`

สำหรับคำขอพื้นหลังโปร่งใส ให้เรียก `image_generate` โดยใช้
`model: "openai/gpt-image-1.5"`, `outputFormat: "png"` หรือ `"webp"` และ
`background: "transparent"` ทั้งนี้ตัวเลือกผู้ให้บริการ `openai.background` แบบเก่า
ยังคงได้รับการยอมรับ OpenClaw ยังปกป้องเส้นทางสาธารณะของ OpenAI และ OAuth ของ OpenAI Codex
โดยเขียนคำขอโปร่งใส `openai/gpt-image-2` เริ่มต้นใหม่เป็น
`gpt-image-1.5` ส่วน Azure และปลายทางที่เข้ากันได้กับ OpenAI แบบกำหนดเองจะคง
ชื่อดีพลอยเมนต์/โมเดลที่กำหนดค่าไว้

การตั้งค่าเดียวกันนี้มีให้ใช้กับการทำงาน CLI แบบไม่มีส่วนติดต่อด้วย:

```bash
openclaw infer image generate \
  --model openai/gpt-image-1.5 \
  --output-format png \
  --background transparent \
  --prompt "วงกลมสีแดงเรียบง่ายในรูปแบบสติกเกอร์บนพื้นหลังโปร่งใส" \
  --json
```

ใช้แฟล็ก `--output-format` และ `--background` เดียวกันกับ
`openclaw infer image edit` เมื่อเริ่มจากไฟล์อินพุต
`--openai-background` ยังคงใช้ได้ในฐานะชื่อแทนเฉพาะของ OpenAI ใช้
`--quality low|medium|high|auto` เพื่อควบคุมคุณภาพและค่าใช้จ่ายของ OpenAI Images
ใช้ `--openai-moderation low|auto` เพื่อส่งคำใบ้การกลั่นกรองของ OpenAI จาก
`image generate` หรือ `image edit`

สำหรับการติดตั้งด้วย OAuth ของ ChatGPT/Codex ให้คงการอ้างอิง `openai/gpt-image-2` เดิมไว้ เมื่อ
กำหนดค่าโปรไฟล์ OAuth `openai` แล้ว OpenClaw จะแก้ไขโทเค็นการเข้าถึง OAuth
ที่จัดเก็บไว้และส่งคำขอรูปภาพผ่านแบ็กเอนด์ Codex Responses โดยจะ
ไม่ลองใช้ `OPENAI_API_KEY` ก่อนหรือสลับไปใช้คีย์ API โดยไม่แจ้ง
กำหนดค่า `models.providers.openai` อย่างชัดเจนด้วยคีย์ API, URL ฐานแบบกำหนดเอง
หรือปลายทาง Azure เมื่อต้องการใช้เส้นทาง OpenAI Images API โดยตรง
แทน หากปลายทางรูปภาพแบบกำหนดเองดังกล่าวอยู่ใน LAN/ที่อยู่ส่วนตัวที่เชื่อถือได้
ให้ตั้งค่า `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` ด้วย OpenClaw
จะบล็อกปลายทางรูปภาพที่เข้ากันได้กับ OpenAI แบบส่วนตัว/ภายใน เว้นแต่จะมี
การเลือกเข้าร่วมนี้

สร้าง:

```
/tool image_generate model=openai/gpt-image-2 prompt="โปสเตอร์เปิดตัว OpenClaw บน macOS ที่สวยงามประณีต" size=3840x2160 count=1
```

สร้าง PNG โปร่งใส:

```
/tool image_generate model=openai/gpt-image-1.5 prompt="วงกลมสีแดงเรียบง่ายในรูปแบบสติกเกอร์บนพื้นหลังโปร่งใส" outputFormat=png background=transparent
```

แก้ไข:

```
/tool image_generate model=openai/gpt-image-2 prompt="คงรูปร่างของวัตถุไว้ เปลี่ยนวัสดุเป็นแก้วโปร่งแสง" image=/path/to/reference.png size=1024x1536
```

| ---------------- | ---------------------------------------------------------------------------------- |
| โมเดลเริ่มต้น    | `openai/sora-2`                                                                    |
| โหมด            | ข้อความเป็นวิดีโอ, รูปภาพเป็นวิดีโอ, การแก้ไขวิดีโอเดียว                                   |
| อินพุตอ้างอิง | รูปภาพ 1 รูปหรือวิดีโอ 1 รายการ                                                                 |
| การเขียนทับขนาด   | รองรับสำหรับข้อความเป็นวิดีโอและรูปภาพเป็นวิดีโอ                                     |
| อัตราส่วนภาพ     | แปลงเป็นขนาดที่รองรับซึ่งใกล้เคียงที่สุด โดยไม่ส่งต่อค่าดิบ                         |
| การเขียนทับอื่น ๆ  | ไม่รองรับ `resolution`, `audio`, `watermark` และจะละทิ้งพร้อมคำเตือนจากเครื่องมือ |

คำขอแปลงรูปภาพเป็นวิดีโอของ OpenAI ใช้ `POST /v1/videos` พร้อมรูปภาพ
`input_reference` การแก้ไขวิดีโอรายการเดียวใช้ `POST /v1/videos/edits` พร้อมวิดีโอที่
อัปโหลดในฟิลด์ `video`

```json5
{
  agents: {
    defaults: {
      videoGenerationModel: { primary: "openai/sora-2" },
    },
  },
}
```

<Note>
ดูพารามิเตอร์เครื่องมือที่ใช้ร่วมกัน การเลือกผู้ให้บริการ และลักษณะการทำงานเมื่อเกิดการสลับระบบที่ [การสร้างวิดีโอ](/th/tools/video-generation)

ผู้ให้บริการ OpenAI ประกาศ `supportsSize` แต่ไม่ประกาศ `supportsAspectRatio` หรือ
`supportsResolution` เลเยอร์การทำให้เป็นมาตรฐานร่วมของ OpenClaw จะแปลง
`aspectRatio` ที่ร้องขอให้เป็น `size` ของ OpenAI ที่ตรงกันมากที่สุดก่อนที่
คำขอจะไปถึงผู้ให้บริการ ดังนั้นโดยทั่วไปคำขออัตราส่วนภาพจึงยังทำงานได้
`resolution` ไม่มีขนาดสำรองและจะถูกตัดออก โดยแจ้งแก่ผู้เรียกเป็น
`Ignored unsupported overrides for openai/<model>: resolution=<value>`
</Note>

## การเพิ่มเนื้อหาในพรอมต์ GPT-5

OpenClaw เพิ่มเนื้อหาในพรอมต์ GPT-5 ที่ใช้ร่วมกันสำหรับโมเดลตระกูล GPT-5 บน
ผู้ให้บริการ `openai` (รวมถึงการอ้างอิง Codex แบบเดิมก่อนการซ่อมแซม ซึ่งจะถูกทำให้เป็นมาตรฐาน
เป็น `openai/*`) ผู้ให้บริการอื่นที่ให้บริการรหัสโมเดลตระกูล GPT-5 เช่นกัน เช่น
OpenRouter หรือเส้นทาง opencode จะไม่ได้รับโอเวอร์เลย์นี้ เนื่องจากจะตรวจสอบ
รหัสผู้ให้บริการ `openai` ไม่ใช่เฉพาะรหัสโมเดล โมเดล GPT-4.x รุ่นเก่า
จะไม่ได้รับเนื้อหานี้เลย

ชุดทดสอบ app-server ของแอป Codex แบบเนทีฟจะไม่ได้รับสัญญาพฤติกรรมด้านบุคลิก/ระเบียบวินัย
การใช้เครื่องมือหรือโอเวอร์เลย์รูปแบบการโต้ตอบที่เป็นมิตรผ่าน
คำสั่งสำหรับนักพัฒนา โดย Codex แบบเนทีฟจะคงพฤติกรรมพื้นฐาน โมเดล และ
เอกสารโครงการที่ Codex เป็นเจ้าของไว้ และ OpenClaw จะปิดใช้งานบุคลิกภาพในตัวของ Codex
สำหรับเธรดแบบเนทีฟ เพื่อให้ไฟล์บุคลิกภาพในพื้นที่ทำงานของเอเจนต์ยังคงเป็นแหล่งข้อมูลหลัก
OpenClaw เพิ่มเฉพาะบริบทรันไทม์ให้เธรด Codex แบบเนทีฟ ได้แก่ การส่งผ่านช่องทาง
เครื่องมือแบบไดนามิกของ OpenClaw การมอบหมายผ่าน ACP บริบทพื้นที่ทำงาน และ
Skills ของ OpenClaw ข้อความแนะแนว Heartbeat จากเนื้อหาที่เพิ่มชุดเดียวกันนี้เป็น
ข้อยกเว้นเพียงอย่างเดียว กล่าวคือ รอบ Heartbeat ของ Codex แบบเนทีฟจะได้รับข้อความนี้
โดยแทรกเป็นคำสั่งการทำงานร่วมกันโดยเฉพาะ แทนที่จะผ่านฮุก
การเพิ่มเนื้อหาในพรอมต์ที่ใช้ร่วมกัน

เนื้อหาที่เพิ่มสำหรับ GPT-5 จะเพิ่มสัญญาพฤติกรรมที่มีแท็กสำหรับการคงอยู่ของบุคลิก
ความปลอดภัยในการดำเนินงาน ระเบียบวินัยในการใช้เครื่องมือ รูปแบบผลลัพธ์ การตรวจสอบ
ความสมบูรณ์ และการยืนยันผลในพรอมต์ที่ OpenClaw ประกอบขึ้นและตรงตามเงื่อนไข พฤติกรรม
การตอบกลับเฉพาะช่องทางและข้อความแบบเงียบยังคงอยู่ในพรอมต์ระบบ OpenClaw ที่ใช้ร่วมกัน
และนโยบายการส่งออก เลเยอร์รูปแบบการโต้ตอบที่เป็นมิตรแยกออกมาต่างหากและกำหนดค่าได้

| ค่า                  | ผล                                      |
| ---------------------- | ------------------------------------------- |
| `"friendly"` (ค่าเริ่มต้น) | เปิดใช้งานเลเยอร์รูปแบบการโต้ตอบที่เป็นมิตร |
| `"on"`                 | นามแฝงสำหรับ `"friendly"`                      |
| `"off"`                | ปิดใช้งานเฉพาะเลเยอร์รูปแบบที่เป็นมิตร       |

<Tabs>
  <Tab title="การกำหนดค่า">
    ```json5
    {
      agents: {
        defaults: {
          promptOverlays: {
            gpt5: { personality: "friendly" },
          },
        },
      },
    }
    ```
  </Tab>
  <Tab title="CLI">
    ```bash
    openclaw config set agents.defaults.promptOverlays.gpt5.personality off
    ```
  </Tab>
</Tabs>

<Tip>
ค่าที่รันไทม์ไม่คำนึงถึงตัวพิมพ์เล็กและตัวพิมพ์ใหญ่ ดังนั้นทั้ง `"Off"` และ `"off"` จึงปิดใช้งาน
เลเยอร์รูปแบบที่เป็นมิตร
</Tip>

<Note>
ระบบยังคงอ่าน `plugins.entries.openai.config.personality` แบบเดิมเพื่อเป็น
ทางเลือกสำรองสำหรับความเข้ากันได้ เมื่อไม่ได้ตั้งค่าการตั้งค่า
`agents.defaults.promptOverlays.gpt5.personality` ที่ใช้ร่วมกัน
</Note>

<AccordionGroup>
  <Accordion title="การสังเคราะห์เสียงพูด (TTS)">
    Plugin `openai` ที่รวมมาให้จะลงทะเบียนการสังเคราะห์เสียงพูดสำหรับ
    พื้นผิว `messages.tts`

    | การตั้งค่า      | พาธการกำหนดค่า                                            | ค่าเริ่มต้น                          |
    | ------------- | --------------------------------------------------------- | ----------------------------------- |
    | โมเดล        | `messages.tts.providers.openai.model`                  | `gpt-4o-mini-tts`                |
    | เสียง        | `messages.tts.providers.openai.speakerVoice`           | `coral`                          |
    | ความเร็ว        | `messages.tts.providers.openai.speed`                  | (ไม่ได้ตั้งค่า)                          |
    | คำสั่ง | `messages.tts.providers.openai.instructions`           | (ไม่ได้ตั้งค่า, เฉพาะ `gpt-4o-mini-tts`)  |
    | รูปแบบ       | `messages.tts.providers.openai.responseFormat`         | `opus` สำหรับข้อความเสียง, `mp3` สำหรับไฟล์ |
    | คีย์ API      | `messages.tts.providers.openai.apiKey`                 | ใช้ `OPENAI_API_KEY` เป็นค่าทดแทน   |
    | URL ฐาน     | `messages.tts.providers.openai.baseUrl`                | `https://api.openai.com/v1`      |
    | เนื้อหาคำขอเพิ่มเติม   | `messages.tts.providers.openai.extraBody` / `extra_body` | (ไม่ได้ตั้งค่า)                        |

    โมเดลที่ใช้ได้: `gpt-4o-mini-tts`, `tts-1`, `tts-1-hd` เสียงที่ใช้ได้:
    `alloy`, `ash`, `ballad`, `cedar`, `coral`, `echo`, `fable`, `juniper`,
    `marin`, `onyx`, `nova`, `sage`, `shimmer`, `verse`

    ระบบจะผสาน `extraBody` ลงใน JSON คำขอ `/audio/speech` หลังฟิลด์ที่
    OpenClaw สร้างขึ้น ดังนั้นให้ใช้สำหรับปลายทางที่เข้ากันได้กับ OpenAI ซึ่งต้องใช้
    คีย์เพิ่มเติม เช่น `lang` โดยจะไม่สนใจคีย์ต้นแบบ

    ```json5
    {
      messages: {
        tts: {
          providers: {
            openai: { model: "gpt-4o-mini-tts", speakerVoice: "coral" },
          },
        },
      },
    }
    ```

    <Note>
    ตั้งค่า `OPENAI_TTS_BASE_URL` เพื่อแทนที่ URL ฐานของ TTS โดยไม่กระทบ
    ปลายทาง API แชต ทั้ง OpenAI TTS และเสียง Realtime กำหนดค่าผ่าน
    คีย์ API ของ OpenAI Platform การติดตั้งที่ใช้ OAuth เท่านั้นยังคงใช้
    โมเดลแชตที่รองรับโดย Codex ได้ แต่ใช้การสนทนาโต้ตอบสดของ OpenAI ไม่ได้
    </Note>

  </Accordion>

    - ใช้ทุกตำแหน่งที่การถอดเสียงจากเสียงขาเข้าอ่าน `tools.media.audio`

    ```json5
          },
        },
      },
    }
    ```

    เชื่อมต่อเสียงแบบเงียบ ๆ โดยไม่มีสิทธิ์เข้าถึงเอเจนต์ นอกจากนี้ การเข้าถึง API ยังถูกจำกัด
    แยกตามองค์กร OpenAI ในช่วงการเข้าถึงล่วงหน้า ให้ใช้ `gpt-realtime-2.1` (ค่าเริ่มต้น)
    ต่อไปจนกว่าจะรองรับ GPT-Live
    </Note>

    <Note>
    บริดจ์เรียลไทม์ของ OpenAI ฝั่งแบ็กเอนด์ใช้รูปแบบเซสชัน Realtime WebSocket
    แบบ GA ซึ่งไม่ยอมรับ `session.temperature` การปรับใช้ Azure OpenAI
    ยังคงใช้ได้ผ่าน `azureEndpoint` และ `azureDeployment` และ
    คงรูปแบบเซสชันที่เข้ากันได้กับการปรับใช้ไว้ (รวมถึง `temperature`)
    รองรับการเรียกใช้เครื่องมือแบบสองทิศทางและเสียง G.711 u-law
    </Note>

    <Note>
    ระบบจะเลือกเสียงแบบเรียลไทม์เมื่อสร้างเซสชัน OpenAI อนุญาตให้เปลี่ยนฟิลด์ส่วนใหญ่
    ของเซสชันได้ในภายหลัง แต่จะไม่สามารถเปลี่ยนเสียงได้หลังจากที่
    โมเดลส่งเสียงออกมาแล้วในเซสชันนั้น ปัจจุบัน OpenClaw เปิดให้ใช้
    ID เสียง Realtime ในตัวในรูปแบบสตริง
    </Note>

    <Note>
    ฟีเจอร์ Talk ใน Control UI ใช้เซสชันแบบเรียลไทม์บนเบราว์เซอร์ของ OpenAI โดยมี
    ข้อมูลลับไคลเอนต์ชั่วคราวที่ Gateway ออกให้ และแลกเปลี่ยน WebRTC SDP โดยตรง
    จากเบราว์เซอร์กับ OpenAI Realtime API โดย Gateway จะออกข้อมูลลับไคลเอนต์ดังกล่าวด้วย
    ข้อมูลประจำตัว `openai` ที่เลือกไว้ คีย์ที่กำหนดค่า โปรไฟล์ API key และ
    `OPENAI_API_KEY` มีลำดับความสำคัญสูงกว่า ส่วนโปรไฟล์ OAuth `openai` หรือการเข้าสู่ระบบ
    Codex ภายนอกจะใช้เป็นทางเลือกสำรอง รีเลย์ของ Gateway และบริดจ์ WebSocket แบบเรียลไทม์
    ของแบ็กเอนด์ Voice Call ใช้ลำดับข้อมูลประจำตัวเดียวกันสำหรับปลายทาง OpenAI แบบเนทีฟ
    ผู้ดูแลสามารถตรวจสอบแบบสดได้ด้วย
    `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts`;
    ส่วนที่เชื่อมต่อกับ OpenAI จะตรวจสอบทั้งบริดจ์ WebSocket ของแบ็กเอนด์และการแลกเปลี่ยน
    WebRTC SDP ของเบราว์เซอร์โดยไม่บันทึกข้อมูลลับ
    ส่ง `--openai-only` เพื่อเรียกใช้ทั้งสองส่วนดังกล่าวโดยไม่มีข้อมูลประจำตัว Google
    </Note>

## ปลายทาง Azure OpenAI

ผู้ให้บริการ `openai` ที่รวมมาให้สามารถกำหนดเป้าหมายไปยังทรัพยากร Azure OpenAI สำหรับการสร้าง
รูปภาพได้โดยแทนที่ URL ฐาน ในเส้นทางการสร้างรูปภาพ OpenClaw
จะตรวจจับชื่อโฮสต์ Azure ใน `models.providers.openai.baseUrl` และเปลี่ยนไปใช้
รูปแบบคำขอของ Azure โดยอัตโนมัติ

<Note>
เสียงแบบเรียลไทม์ใช้เส้นทางการกำหนดค่าแยกต่างหาก
(`plugins.entries.voice-call.config.realtime.providers.openai.azureEndpoint`)
และไม่ได้รับผลกระทบจาก `models.providers.openai.baseUrl` โปรดดูแอคคอร์เดียน **เสียงแบบ
เรียลไทม์** ใต้หัวข้อ [เสียงพูดและเสียง](#voice-and-speech) สำหรับการตั้งค่า Azure
</Note>

ใช้ Azure OpenAI เมื่อ:

- มีการสมัครใช้งาน Azure OpenAI โควตา หรือข้อตกลง
  ระดับองค์กรอยู่แล้ว
- ต้องการการจัดเก็บข้อมูลตามภูมิภาคหรือการควบคุมการปฏิบัติตามข้อกำหนดที่ Azure มีให้
- ต้องการให้ทราฟฟิกอยู่ภายใน Azure tenancy ที่มีอยู่

### การกำหนดค่า

สำหรับการสร้างรูปภาพผ่าน Azure โดยใช้ผู้ให้บริการ `openai` ที่รวมมาให้ ให้ชี้
`models.providers.openai.baseUrl` ไปยังทรัพยากร Azure และตั้งค่า `apiKey` เป็น
คีย์ Azure OpenAI (ไม่ใช่คีย์ OpenAI Platform):

```json5
{
  models: {
    providers: {
      openai: {
        baseUrl: "https://<your-resource>.openai.azure.com",
        apiKey: "<azure-openai-api-key>",
      },
    },
  },
}
```

OpenClaw รู้จักส่วนต่อท้ายโฮสต์ Azure ต่อไปนี้สำหรับเส้นทางการสร้างรูปภาพ
ของ Azure:

- `*.openai.azure.com`
- `*.services.ai.azure.com`
- `*.cognitiveservices.azure.com`

สำหรับคำขอสร้างรูปภาพบนโฮสต์ Azure ที่รู้จัก OpenClaw จะ:

- ส่งส่วนหัว `api-key` แทน `Authorization: Bearer`
- ใช้พาธที่กำหนดขอบเขตตาม deployment (`/openai/deployments/{deployment}/...`)
- ต่อท้าย `?api-version=...` ในแต่ละคำขอ
- ใช้ค่า timeout เริ่มต้นของคำขอที่ 600s สำหรับการเรียกสร้างรูปภาพผ่าน Azure
  ค่า `timeoutMs` ของแต่ละการเรียกยังคงแทนที่ค่าเริ่มต้นนี้ได้

URL ฐานอื่นๆ (OpenAI สาธารณะและพร็อกซีที่เข้ากันได้กับ OpenAI) จะยังคงใช้
รูปแบบคำขอรูปภาพมาตรฐานของ OpenAI

<Note>
การกำหนดเส้นทาง Azure สำหรับเส้นทางการสร้างรูปภาพของผู้ให้บริการ `openai` ต้องใช้
OpenClaw 2026.4.22 หรือใหม่กว่า เวอร์ชันก่อนหน้านี้จะถือว่า
`openai.baseUrl` แบบกำหนดเองทั้งหมดเหมือนกับปลายทาง OpenAI สาธารณะ และจะล้มเหลวเมื่อใช้กับ
deployment รูปภาพของ Azure
</Note>

### เวอร์ชัน API

ตั้งค่า `AZURE_OPENAI_API_VERSION` เพื่อตรึงเวอร์ชัน Azure แบบพรีวิวหรือ GA ที่ระบุ
สำหรับเส้นทางการสร้างรูปภาพผ่าน Azure:

```bash
export AZURE_OPENAI_API_VERSION="2024-12-01-preview"
```

ค่าเริ่มต้นคือ `2024-12-01-preview` เมื่อตัวแปรไม่ได้ถูกตั้งค่า

### ชื่อโมเดลคือชื่อ deployment

Azure OpenAI ผูกโมเดลเข้ากับ deployment สำหรับคำขอสร้างรูปภาพผ่าน Azure
ที่กำหนดเส้นทางผ่านผู้ให้บริการ `openai` ที่รวมมาให้ ฟิลด์ `model` ใน OpenClaw
ต้องเป็น **ชื่อ deployment ของ Azure** ที่กำหนดค่าไว้ในพอร์ทัล Azure ไม่ใช่
ID โมเดล OpenAI สาธารณะ

หากสร้าง deployment ชื่อ `gpt-image-2-prod` ที่ให้บริการ `gpt-image-2`:

```
/tool image_generate model=openai/gpt-image-2-prod prompt="โปสเตอร์เรียบสะอาด" size=1024x1024 count=1
```

กฎการใช้ชื่อ deployment เดียวกันนี้มีผลกับการเรียกสร้างรูปภาพทั้งหมดที่กำหนดเส้นทาง
ผ่านผู้ให้บริการ `openai` ที่รวมมาให้

### ความพร้อมใช้งานตามภูมิภาค

ขณะนี้การสร้างรูปภาพผ่าน Azure ใช้งานได้เฉพาะในบางภูมิภาค
(เช่น `eastus2`, `swedencentral`, `polandcentral`, `westus3`,
`uaenorth`) ตรวจสอบรายการภูมิภาคล่าสุดของ Microsoft ก่อนสร้าง
deployment และยืนยันว่าโมเดลที่ระบุมีให้บริการในภูมิภาคของคุณ

### ความแตกต่างของพารามิเตอร์

Azure OpenAI และ OpenAI สาธารณะไม่ได้ยอมรับพารามิเตอร์รูปภาพชุดเดียวกันเสมอไป
Azure อาจปฏิเสธตัวเลือกที่ OpenAI สาธารณะอนุญาต (เช่น ค่า
`background` บางค่าบน `gpt-image-2`) หรือเปิดให้ใช้เฉพาะในโมเดล
บางเวอร์ชัน ความแตกต่างเหล่านี้มาจาก Azure และโมเดลพื้นฐาน ไม่ใช่
OpenClaw หากคำขอ Azure ล้มเหลวด้วยข้อผิดพลาดในการตรวจสอบความถูกต้อง ให้ตรวจสอบ
ชุดพารามิเตอร์ที่ deployment และเวอร์ชัน API ของคุณรองรับใน
พอร์ทัล Azure

<Note>
Azure OpenAI ใช้การรับส่งข้อมูลแบบเนทีฟและพฤติกรรมความเข้ากันได้ แต่จะไม่ได้รับ
ส่วนหัวการระบุแหล่งที่มาแบบซ่อนของ OpenClaw โปรดดูแอคคอร์เดียน **เส้นทางแบบเนทีฟเทียบกับเส้นทางที่เข้ากันได้กับ OpenAI**
ใต้หัวข้อ [การกำหนดค่าขั้นสูง](#advanced-configuration)

สำหรับทราฟฟิกแชตหรือ Responses บน Azure (นอกเหนือจากการสร้างรูปภาพ) ให้ใช้
ขั้นตอน onboarding หรือการกำหนดค่าผู้ให้บริการ Azure โดยเฉพาะ การตั้งค่า `openai.baseUrl` เพียงอย่างเดียว
จะไม่ใช้รูปแบบ API/การยืนยันตัวตนของ Azure มีผู้ให้บริการ
`azure-openai-responses/*` แยกต่างหาก โปรดดูแอคคอร์เดียน Compaction
ฝั่งเซิร์ฟเวอร์ด้านล่าง
</Note>

## การกำหนดค่าขั้นสูง

ตัวอย่าง `params` แยกตามโมเดลด้านล่างกำหนดรูปแบบคำขอของผู้ให้บริการแบบฝังของ OpenClaw
การกำหนดค่าตัวอย่างเหล่านี้ถือเป็นพฤติกรรมของคำขอที่ผู้ใช้กำหนด ดังนั้นเส้นทาง
`auto` ที่เข้าเกณฑ์ในด้านอื่นจะยังคงทำงานบน OpenClaw แทนที่จะเลือก Codex โดยปริยาย ชุดทดสอบ
app-server แบบเนทีฟของ Codex จัดการการรับส่งข้อมูลและการตั้งค่าคำขอของตนเอง ส่วน
`agentRuntime.id: "codex"` แบบระบุชัดเจนจะหยุดอย่างปลอดภัยเมื่อเส้นทางที่มีผลไม่ได้ประกาศว่า
เข้ากันได้กับ Codex

<AccordionGroup>
  <Accordion title="การรับส่งข้อมูล (WebSocket เทียบกับ SSE)">
    OpenClaw ใช้ WebSocket เป็นอันดับแรกและใช้ SSE เป็นทางเลือกสำรอง (`"auto"`) สำหรับ `openai/*`

    ในโหมด `"auto"` OpenClaw จะ:
    - ลองใหม่หนึ่งครั้งเมื่อ WebSocket ล้มเหลวในช่วงต้น ก่อนเปลี่ยนไปใช้ SSE
    - หลังเกิดความล้มเหลว จะทำเครื่องหมายว่า WebSocket มีประสิทธิภาพลดลงเป็นเวลา 60 วินาทีและใช้ SSE
      ระหว่างช่วงพัก
    - แนบส่วนหัวระบุตัวตนของเซสชันและรอบการสนทนาที่คงที่สำหรับการลองใหม่และ
      การเชื่อมต่อใหม่
    - ปรับตัวนับการใช้งาน (`input_tokens` / `prompt_tokens`) ให้เป็นมาตรฐานเดียวกันใน
      รูปแบบการรับส่งข้อมูลต่างๆ

    | ค่า                | พฤติกรรม                          |
    | ---------------------- | ------------------------------------ |
    | `"auto"` (ค่าเริ่มต้น)   | ใช้ WebSocket ก่อน และใช้ SSE เป็นทางเลือกสำรอง     |
    | `"sse"`              | บังคับใช้ SSE เท่านั้น                    |
    | `"websocket"`        | บังคับใช้ WebSocket เท่านั้น              |

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": {
              params: { transport: "auto" },
            },
          },
        },
      },
    }
    ```

    เอกสาร OpenAI ที่เกี่ยวข้อง:
    - [Realtime API ผ่าน WebSocket](https://platform.openai.com/docs/guides/realtime-websocket)
    - [การสตรีมการตอบกลับ API (SSE)](https://platform.openai.com/docs/guides/streaming-responses)

  </Accordion>

  <Accordion title="โหมดเร็ว">
    OpenClaw มีสวิตช์โหมดเร็วที่ใช้ร่วมกันสำหรับ `openai/*`:

    - **แชต/UI:** `/fast status|auto|on|off`
    - **การกำหนดค่า:** `agents.defaults.models["<provider>/<model>"].params.fastMode`

    เมื่อเปิดใช้งาน OpenClaw จะแมปโหมดเร็วไปยังการประมวลผลแบบมีลำดับความสำคัญของ OpenAI
    (`service_tier = "priority"`) ค่า `service_tier` ที่มีอยู่จะถูก
    รักษาไว้ และโหมดเร็วจะไม่เขียน `reasoning` หรือ
    `text.verbosity` ใหม่ `fastMode: "auto"` จะเริ่มการเรียกโมเดลใหม่ด้วยโหมดเร็วจนถึง
    จุดตัดอัตโนมัติ จากนั้นจะเริ่มการลองใหม่ การใช้ทางเลือกสำรอง การส่งผลลัพธ์เครื่องมือ หรือ
    การเรียกต่อเนื่องในภายหลังโดยไม่ใช้โหมดเร็ว จุดตัดมีค่าเริ่มต้นเป็น 60 วินาที
    ให้ตั้งค่า `params.fastAutoOnSeconds` บนโมเดลที่ใช้งานอยู่เพื่อเปลี่ยนค่านี้

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { fastMode: "auto", fastAutoOnSeconds: 30 } },
          },
        },
      },
    }
    ```

    <Note>
    ค่าที่แทนที่ในเซสชันมีลำดับความสำคัญเหนือกว่าการกำหนดค่า การล้างค่าที่แทนที่ของเซสชันใน
    UI เซสชันจะทำให้เซสชันกลับไปใช้ค่าเริ่มต้นที่กำหนดไว้
    </Note>

  </Accordion>

  <Accordion title="การประมวลผลแบบมีลำดับความสำคัญ (service_tier)">
    API ของ OpenAI เปิดให้ใช้การประมวลผลแบบมีลำดับความสำคัญผ่าน `service_tier` ให้ตั้งค่าแยกตาม
    โมเดลใน OpenClaw:

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { serviceTier: "priority" } },
          },
        },
      },
    }
    ```

    ค่าที่รองรับ: `auto`, `default`, `flex`, `priority`

    <Warning>
    `serviceTier` จะถูกส่งต่อไปยังปลายทาง OpenAI แบบเนทีฟ
    (`api.openai.com`) และปลายทาง Codex แบบเนทีฟ (`chatgpt.com/backend-api`) เท่านั้น
    หากกำหนดเส้นทางผู้ให้บริการใดผู้ให้บริการหนึ่งผ่านพร็อกซี OpenClaw จะคง
    `service_tier` ไว้โดยไม่แก้ไข
    </Warning>

  </Accordion>

  <Accordion title="Compaction ฝั่งเซิร์ฟเวอร์ (Responses API)">
    สำหรับโมเดล OpenAI Responses โดยตรง (`openai/*` บน `api.openai.com`)
    ตัวครอบสตรีม OpenClaw ของ Plugin OpenAI จะเปิดใช้ Compaction ฝั่งเซิร์ฟเวอร์
    โดยอัตโนมัติ:

    - บังคับใช้ `store: true` (เว้นแต่ความเข้ากันได้ของโมเดลจะตั้งค่า `supportsStore: false`)
    - แทรก `context_management: [{ type: "compaction", compact_threshold: ... }]`
    - ค่าเริ่มต้นของ `compact_threshold`: 70% ของ `contextWindow` (หรือ `80000` เมื่อ
      ไม่มีค่า)

    การตั้งค่านี้มีผลกับเส้นทางรันไทม์ OpenClaw ในตัวและ hook ของผู้ให้บริการ OpenAI
    ที่ใช้โดยการทำงานแบบฝัง ชุดทดสอบ app-server แบบเนทีฟของ Codex จัดการ
    บริบทของตนเองผ่าน Codex และไม่ได้รับผลกระทบจากการตั้งค่านี้

    <Tabs>
      <Tab title="เปิดใช้อย่างชัดเจน">
        มีประโยชน์สำหรับปลายทางที่เข้ากันได้ เช่น Azure OpenAI Responses:

        ```json5
        {
          agents: {
            defaults: {
              models: {
                "azure-openai-responses/gpt-5.5": {
                  params: { responsesServerCompaction: true },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="เกณฑ์กำหนดเอง">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: {
                    responsesServerCompaction: true,
                    responsesCompactThreshold: 120000,
                  },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="ปิดใช้งาน">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: { responsesServerCompaction: false },
                },
              },
            },
          },
        }
        ```
      </Tab>
    </Tabs>

    <Note>
    `responsesServerCompaction` ควบคุมเฉพาะการแทรก `context_management`
    โมเดล OpenAI Responses โดยตรงยังคงบังคับใช้ `store: true` เว้นแต่การตั้งค่าความเข้ากันได้
    จะกำหนด `supportsStore: false`
    </Note>

  </Accordion>

  <Accordion title="โหมด GPT แบบ agentic ที่เข้มงวด">
    สำหรับโมเดลตระกูล GPT-5 ของผู้ให้บริการ `openai` ที่ทำงานผ่านรันไทม์แบบฝัง
    ของ OpenClaw นั้น OpenClaw ใช้สัญญาการทำงานที่เข้มงวดยิ่งขึ้นซึ่งเรียกว่า
    `strict-agentic` เป็นค่าเริ่มต้นอยู่แล้ว โดยจะเปิดใช้งานโดยอัตโนมัติเมื่อผู้ให้บริการที่แก้ไขแล้วคือ
    `openai` และ ID โมเดลตรงกับตระกูล GPT-5 เว้นแต่การกำหนดค่า
    จะเลือกไม่ใช้ไว้อย่างชัดเจน:

    ```json5
    {
      agents: {
        defaults: {
          embeddedAgent: { executionContract: "default" },
        },
      },
    }
    ```

    การตั้งค่า `"strict-agentic"` อย่างชัดเจนไม่มีผลใดๆ บนเส้นทางที่รองรับ (เนื่องจาก
    เป็นค่าเริ่มต้นอยู่แล้ว) และไม่มีผลกับคู่ผู้ให้บริการ/โมเดลที่ไม่รองรับ

    เมื่อเปิดใช้ `strict-agentic` OpenClaw จะ:
    - เปิดใช้ `update_plan` โดยอัตโนมัติสำหรับงานขนาดใหญ่
    - ลองทำเทิร์นที่ว่างเปล่าในเชิงโครงสร้างหรือมีเฉพาะการให้เหตุผลอีกครั้ง โดยดำเนินการต่อ
      เพื่อให้ได้คำตอบที่มองเห็นได้
    - ใช้เหตุการณ์แผนของชุดควบคุมอย่างชัดเจน เมื่อชุดควบคุมที่เลือก
      มีเหตุการณ์ดังกล่าว

    OpenClaw ไม่จำแนกข้อความร้อยแก้วของผู้ช่วยเพื่อตัดสินว่าเทิร์นนั้นเป็น
    แผน การอัปเดตความคืบหน้า หรือคำตอบสุดท้าย

    <Note>
    สัญญานี้อยู่ภายในตัวเรียกใช้เอเจนต์แบบฝังของ OpenClaw ทั้งหมด โดย
    ไม่ใช้กับชุดควบคุม app-server ของ Codex แบบเนทีฟ ซึ่งจัดการ
    พฤติกรรมของเทิร์นและแผนด้วยตนเอง สำหรับการเรียกใช้ Codex แบบเนทีฟ การเลือกชุดควบคุม
    มีความสำคัญมากกว่าการตั้งค่าสัญญาการดำเนินการ
    </Note>

  </Accordion>

  <Accordion title="เส้นทางแบบเนทีฟเทียบกับเส้นทางที่เข้ากันได้กับ OpenAI">
    OpenClaw จัดการปลายทาง OpenAI โดยตรง, Codex และ Azure OpenAI
    แตกต่างจากพร็อกซี `/v1` ทั่วไปที่เข้ากันได้กับ OpenAI:

    **เส้นทางแบบเนทีฟ** (`openai/*`, Azure OpenAI):
    - คง `reasoning: { effort: "none" }` ไว้เฉพาะสำหรับโมเดลที่รองรับระดับ
      ความพยายาม `none` ของ OpenAI
    - ละเว้นการให้เหตุผลที่ปิดใช้งานสำหรับโมเดลหรือพร็อกซีที่ปฏิเสธ
      `reasoning.effort: "none"`
    - ตั้งค่าเริ่มต้นให้สคีมาเครื่องมือใช้โหมดเข้มงวด
    - แนบส่วนหัวการระบุที่มาแบบซ่อนเฉพาะบนโฮสต์แบบเนทีฟที่ผ่านการตรวจสอบแล้ว (Azure
      OpenAI จะไม่ได้รับส่วนหัวเหล่านี้ แม้ว่าจะเป็นเส้นทางแบบเนทีฟ)
    - คงการปรับรูปแบบคำขอเฉพาะของ OpenAI (`service_tier`, `store`,
      ความเข้ากันได้ของการให้เหตุผล, คำแนะนำแคชพรอมต์)

    **เส้นทางพร็อกซี/ที่เข้ากันได้:**
    - ใช้พฤติกรรมความเข้ากันได้ที่ผ่อนปรนกว่า
    - ตัด `store` ของ Completions ออกจากเพย์โหลด `openai-completions` ที่ไม่ใช่แบบเนทีฟ
    - ยอมรับ JSON แบบส่งผ่านขั้นสูงของ `params.extra_body`/`params.extraBody`
      สำหรับพร็อกซี Completions ที่เข้ากันได้กับ OpenAI
    - ยอมรับ `params.chat_template_kwargs` สำหรับพร็อกซี Completions
      ที่เข้ากันได้กับ OpenAI เช่น vLLM
    - ไม่บังคับใช้สคีมาเครื่องมือแบบเข้มงวดหรือส่วนหัวเฉพาะแบบเนทีฟ

  </Accordion>
</AccordionGroup>
