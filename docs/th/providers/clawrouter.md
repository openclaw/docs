ClawRouter มอบคีย์เดียวที่กำหนดขอบเขตด้วยนโยบายให้ OpenClaw สำหรับผู้ให้บริการโมเดล
ต้นทางหลายราย Plugin `clawrouter` ที่รวมมาด้วยจะค้นหาเฉพาะโมเดลที่ได้รับอนุญาต
สำหรับคีย์นั้น กำหนดเส้นทางแต่ละโมเดลผ่านโปรโตคอลที่ประกาศไว้ และรายงาน
งบประมาณของคีย์และการใช้งานรวมบนส่วนแสดงการใช้งานของ OpenClaw

ข้อมูลประจำตัวต้นทางและการส่งต่อตามผู้ให้บริการจะคงอยู่ใน ClawRouter ดังนั้น
จึงไม่ต้องติดตั้งหรือยืนยันตัวตน Plugin ของผู้ให้บริการต้นทางแต่ละรายบน
โฮสต์ OpenClaw Plugin นี้รวมมากับ OpenClaw (`enabledByDefault: true`);
ต้องมีเพียงข้อมูลประจำตัว ClawRouter ที่ออกให้แล้ว

| คุณสมบัติ      | ค่า                                    |
| ------------- | ---------------------------------------- |
| ผู้ให้บริการ      | `clawrouter`                             |
| Plugin        | รวมมาด้วย (รวมอยู่ใน OpenClaw)           |
| การยืนยันตัวตน          | `CLAWROUTER_API_KEY`                     |
| URL เริ่มต้น   | `https://clawrouter.openclaw.ai`         |
| แค็ตตาล็อกโมเดล | กำหนดขอบเขตตามข้อมูลประจำตัวผ่าน `/v1/catalog`      |
| โควตา        | งบประมาณและการใช้งานรายเดือนผ่าน `/v1/usage` |

## เริ่มต้นใช้งาน

<Steps>
  <Step title="รับข้อมูลประจำตัวที่กำหนดขอบเขต">
    ขอข้อมูลประจำตัวจากผู้ดูแลระบบ ClawRouter โดยนโยบายของข้อมูลประจำตัวนั้นต้องครอบคลุม
    ผู้ให้บริการ โมเดล และงบประมาณรายเดือนที่ควรใช้ ระบบจะแสดงข้อมูลประจำตัว
    เพียงครั้งเดียวเมื่อออกให้
  </Step>
  <Step title="กำหนดค่า OpenClaw">
    ```bash
    export CLAWROUTER_API_KEY="..."
    openclaw onboard --auth-choice clawrouter-api-key
    openclaw plugins enable clawrouter
    ```

    `clawrouter` รวมมาด้วยและเปิดใช้งานเป็นค่าเริ่มต้น หากการกำหนดค่าตั้งค่า
    `plugins.allow` ให้เพิ่ม `clawrouter` ลงในรายการนั้นก่อนเปิดใช้งาน สำหรับการ
    ติดตั้งใช้งานแบบกำหนดเอง ให้ตั้งค่า `models.providers.clawrouter.baseUrl` เป็น
    ต้นทาง ClawRouter โดยค่าเริ่มต้นคือ `https://clawrouter.openclaw.ai`

  </Step>
  <Step title="แสดงรายการโมเดลที่ได้รับสิทธิ์">
    ```bash
    openclaw models list --all --provider clawrouter
    ```

    ใช้การอ้างอิงโมเดลที่ส่งคืนมาตามที่แสดงทุกประการ โดยจะคงเนมสเปซ
    ต้นทางไว้ เช่น `clawrouter/openai/gpt-5.5`,
    `clawrouter/anthropic/claude-sonnet-4-6` หรือ
    `clawrouter/google/gemini-3.5-flash` หาก `agents.defaults.models` เป็น
    รายการอนุญาตในการกำหนดค่า ให้เพิ่มการอ้างอิง ClawRouter แต่ละรายการที่เลือกไว้ลงไป

  </Step>
  <Step title="เลือกโมเดล">
    ```bash
    openclaw models set clawrouter/<provider>/<model>
    ```

    `openclaw agent --model clawrouter/<provider>/<model> --message "..."`

  </Step>
</Steps>

## การติดตั้งใช้งานแบบไม่โต้ตอบที่มีการจัดการ

เก็บคีย์พร็อกซีไว้ในการแทรกข้อมูลลับของเวิร์กโหลด และจัดเก็บเฉพาะ
SecretRef ใน `openclaw.json` ฟิลด์มาตรฐานสำหรับการจัดการมีดังนี้:

| วัตถุประสงค์       | ฟิลด์การกำหนดค่าหรือสภาพแวดล้อม                                              |
| ------------- | ------------------------------------------------------------------------ |
| ต้นทางเราเตอร์ | `models.providers.clawrouter.baseUrl`                                    |
| ข้อมูลประจำตัว    | `models.providers.clawrouter.apiKey` -> SecretRef ของ env                    |
| ค่าข้อมูลลับ  | `CLAWROUTER_API_KEY` ในสภาพแวดล้อมของกระบวนการ Gateway                  |
| โมเดลเริ่มต้น | `agents.defaults.model.primary` -> `clawrouter/<provider>/<model>`       |
| แท็กเวิร์กโหลด  | `models.providers.clawrouter.headers.X-ClawRouter-Project-Id` (ไม่บังคับ) |

ตัวอย่างเช่น ตัวควบคุมการติดตั้งใช้งานสามารถเป็นเจ้าของแพตช์ JSON5 นี้:

```json5
{
  plugins: {
    entries: { clawrouter: { enabled: true } },
  },
  models: {
    providers: {
      clawrouter: {
        baseUrl: "https://clawrouter.internal.example",
        apiKey: {
          source: "env",
          provider: "default",
          id: "CLAWROUTER_API_KEY",
        },
        headers: {
          "X-ClawRouter-Project-Id": "fakeco",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: { primary: "clawrouter/openai/gpt-5.5" },
    },
  },
}
```

หากการติดตั้งใช้งานตั้งค่า `plugins.allow` ให้คงรายการเดิมไว้และเพิ่ม
`clawrouter` ตรวจสอบและนำไปใช้โดยไม่ใช้ตัวช่วยแบบโต้ตอบ:

```bash
openclaw config patch --file ./clawrouter.patch.json5 --dry-run --json
openclaw config patch --file ./clawrouter.patch.json5
```

การทดลองรันจะแก้ค่า SecretRef แต่จะไม่พิมพ์ค่าของข้อมูลลับออกมา หากต้องการหมุนเวียน
ข้อมูลประจำตัว ให้อัปเดต Secret ภายนอกที่จัดหา `CLAWROUTER_API_KEY` และ
รีสตาร์ตเวิร์กโหลด Gateway เพื่อโหลดสภาพแวดล้อมของกระบวนการใหม่
ไฟล์การกำหนดค่าและการอ้างอิงโมเดลจะไม่เปลี่ยนแปลง

สำหรับ Gateway Docker แบบสแตนด์อโลนที่สร้างจากซอร์ส ClawRouter รวมอยู่ใน
รันไทม์รากแล้ว เลือกเฉพาะ Plugin ช่องทางที่ต้องจัดแพ็กเกจแยก
เช่น `OPENCLAW_EXTENSIONS=clickclack`, `slack` หรือ `msteams`; ดู
[อิมเมจที่สร้างจากซอร์สพร้อม Plugin ที่เลือก](/th/install/docker#source-built-images-with-selected-plugins)
การติดตั้งใช้งานแบบอาร์ไคฟ์/อุปกรณ์ต้องจัดแพ็กเกจซอร์สเดียวกันที่รวมแล้วผ่าน
ไปป์ไลน์อาร์ติแฟกต์ของตนเอง แทนการใช้อิมเมจ OCI

## ความพร้อมและหลักฐานจากระบบจริง

การตรวจสอบเหล่านี้พิสูจน์ขอบเขตที่ต่างกัน ห้ามใช้แทนกัน:

```bash
# เฉพาะสถานะกระบวนการ ClawRouter โดยไม่มีการใช้ข้อมูลประจำตัวหรือโมเดลต้นทาง
curl -fsS https://clawrouter.internal.example/v1/health

# เฉพาะความพร้อมเมื่อเริ่มต้น Gateway ของ OpenClaw โดยไม่มีการเรียกโมเดล
curl -fsS http://127.0.0.1:18789/readyz

# การค้นพบแค็ตตาล็อกที่กำหนดขอบเขตตามข้อมูลประจำตัว
openclaw models list --all --provider clawrouter --json

# โพรบการอนุมานจริงขั้นต่ำผ่านผู้ให้บริการ ClawRouter ที่กำหนดค่าไว้
openclaw models status --probe --probe-provider clawrouter --probe-max-tokens 8 --json

# คาแนรีของเวิร์กโหลดโดยใช้การอ้างอิงโมเดลที่ได้รับสิทธิ์อย่างแม่นยำ
openclaw agent --agent main \
  --model clawrouter/openai/gpt-5.5 \
  --message "ตอบข้อความนี้ทุกประการ: CLAWROUTER_CANARY_OK" \
  --json
```

ใช้โมเดลที่ส่งคืนจากแค็ตตาล็อกที่กำหนดขอบเขต แทนการคัดลอกโมเดลตัวอย่าง
โดยไม่ตรวจสอบ การตอบกลับ `/readyz` ที่สำเร็จหมายความว่า Gateway สามารถให้บริการ
คำขอได้ แต่ไม่ได้ยืนยันว่า ClawRouter ข้อมูลประจำตัวของ ClawRouter หรือผู้ให้บริการ
ต้นทางพร้อมใช้งาน โพรบโมเดลและคาแนรีเอเจนต์คือหลักฐานการอนุมาน

สำหรับการวินิจฉัยระบบจริง ให้ส่งคาแนรีและตรวจสอบบันทึกมาตรฐานของ Gateway
การวินิจฉัยการรับส่งข้อมูลโมเดลที่มีเฉพาะเมทาดาทาเดิมจะส่งบรรทัดที่มีรูปแบบดังนี้:

```text
[model-fetch] เริ่ม provider=clawrouter api=openai-responses model=openai/gpt-5.5 method=POST url=https://clawrouter.internal.example/v1/responses
[model-fetch] การตอบกลับ provider=clawrouter api=openai-responses model=openai/gpt-5.5 status=200
```

Plugin จะส่งส่วนหัว `X-ClawRouter-Client`, `X-ClawRouter-Agent-Id` และ
`X-ClawRouter-Session-Id` แบบจำกัดขนาดเมื่อมีตัวระบุเหล่านั้น นอกจากนี้ยัง
แมป `callId` (`<run-id>:model:<n>`) สำหรับการวินิจฉัยของการเรียกโมเดลไปยัง
`X-Request-ID` เพื่อให้เชื่อมเหตุการณ์การเรียกโมเดลของ OpenClaw เข้ากับ
เส้นทางการตรวจสอบที่มีเฉพาะเมทาดาทาของ ClawRouter ได้ ค่าที่อยู่ภายในงบประมาณ
รหัสคำขอ 128 อักขระจะเหมือนกัน ค่าที่ยาวกว่านั้นจะคงส่วนต่อท้าย
`:model:<n>` และแฮชแบบกำหนดแน่นอนไว้ เพื่อให้การเรียกที่ต่างกันยังคงมีขนาดจำกัด
และเชื่อมโยงกันได้ เมทาดาทาการติดตั้งใช้งานแบบคงที่ เช่น `X-ClawRouter-Project-Id`
สามารถตั้งค่าในแมป `headers` ของผู้ให้บริการได้ ส่วนหัวการระบุแหล่งที่มาของ
เอเจนต์และเซสชันยังคงมีขีดจำกัดแยกที่ 256 อักขระ รหัสคำขออัตโนมัติที่มี
อักขระนอกชุดตัวระบุ ASCII ของ ClawRouter จะใช้รูปแบบจำกัดขนาดแบบกำหนดแน่นอนเดียวกัน
ส่วนหัวที่กำหนดไว้อย่างชัดเจน รวมถึง `X-Request-ID` ในรูปแบบตัวพิมพ์ใดก็ตาม จะมี
ลำดับความสำคัญเหนือค่าอัตโนมัติ การวินิจฉัยการรับส่งข้อมูลจะบันทึกเมทาดาทาการกำหนด
เส้นทางและการตอบกลับ แต่จะไม่บันทึกข้อมูลประจำตัว รหัสคำขอ พรอมต์ หรือผลลัพธ์
เหตุการณ์การตรวจสอบของ ClawRouter เองจะแสดงผู้ให้บริการต้นทางที่เลือกและ
สถานะการเก็บรักษาเนื้อหา

## การค้นพบโมเดล

`GET /v1/catalog` ส่งคืน `{ providers: [...] }` โดยรายการผู้ให้บริการแต่ละรายการ
จะแสดง `models[]` ของตนเอง (พร้อมรหัสต้นทาง ความสามารถ และราคา) และ
เส้นทางคำขอที่รองรับ OpenClaw ไม่มีรายการโมเดล ClawRouter แบบตายตัวชุดที่สอง
โมเดลในแค็ตตาล็อกจะได้รับการประกาศเป็นโมเดล OpenClaw เมื่อ:

- นโยบายของข้อมูลประจำตัวให้สิทธิ์ผู้ให้บริการนั้น;
- โมเดลในแค็ตตาล็อกประกาศความสามารถ LLM ที่รองรับ (`llm.responses`,
  `llm.chat`, `llm.messages` หรือ `llm.stream` พร้อมเส้นทางสตรีมมิง
  ที่ตรงกัน); และ
- ผู้ให้บริการเปิดเผยเส้นทางที่ตรงกันสำหรับการรับส่งข้อมูลรายการใดรายการหนึ่งด้านล่าง

การเพิ่มโมเดลไปยังผู้ให้บริการ ClawRouter ที่รองรับไม่ต้องใช้ OpenClaw รุ่นใหม่:
การรีเฟรชแค็ตตาล็อกครั้งถัดไป (แคช 60 วินาทีต่อขอบเขตข้อมูลประจำตัว) จะค้นพบ
โมเดลนั้น โมเดลที่ต้องใช้โปรโตคอลการรับส่งใหม่ต้องมีการรองรับจาก Plugin ก่อน

## โปรโตคอลและ Plugin ผู้ให้บริการ

ClawRouter เป็นเจ้าของข้อมูลประจำตัวต้นทาง แค็ตตาล็อกจะบอก OpenClaw ว่าต้องใช้
การรับส่งข้อมูลใด จึงไม่ต้องติดตั้ง Plugin การยืนยันตัวตนของบริษัทต้นทางทุกแห่ง

| ความสามารถ/เส้นทางของแค็ตตาล็อก                               | การรับส่งข้อมูลของ OpenClaw     |
| -------------------------------------------------------- | ---------------------- |
| `llm.responses` (ผู้ให้บริการที่เข้ากันได้กับ OpenAI)             | `openai-responses`     |
| `llm.chat` (ผู้ให้บริการที่เข้ากันได้กับ OpenAI)                  | `openai-completions`   |
| `llm.messages` + เส้นทาง `anthropic.messages`              | `anthropic-messages`   |
| `llm.stream` + เส้นทาง `google.generate_content` แบบสตรีมมิง | `google-generative-ai` |

Plugin ยังใช้นโยบายการเล่นซ้ำและสคีมาเครื่องมือที่ตรงกันสำหรับ
ตระกูลเหล่านั้นด้วย (ความเข้ากันได้ของสคีมาเครื่องมือ OpenAI/DeepSeek/Gemini/Perplexity;
นโยบายการเล่นซ้ำแบบเนทีฟของ Anthropic และ Google Gemini) โมเดล Perplexity จะได้รับ
การเขียนสคีมาใหม่อย่างเข้มงวด: ระบบจะลบ `patternProperties` และ `additionalProperties`
และทุกสคีมาวัตถุจะประกาศ `properties` เนื่องจาก Perplexity ปฏิเสธ
สคีมาเครื่องมือที่ไม่มีรายการเหล่านี้ ผู้ให้บริการในแค็ตตาล็อกที่เปิดเผยเฉพาะ
รูปแบบคำขอที่ไม่รองรับจะไม่ได้รับการประกาศเป็นโมเดลข้อความของ OpenClaw โดยเจตนา
ปรับผู้ให้บริการเหล่านั้นให้เป็นหนึ่งในสัญญาที่รองรับใน ClawRouter แทนการส่ง
เพย์โหลดที่เข้ากันไม่ได้

## โควตาและการใช้งาน

การตอบกลับ `/v1/usage` ของ ClawRouter จะป้อนข้อมูลเข้าสู่ส่วนแสดงการใช้งาน
ผู้ให้บริการตามปกติของ OpenClaw ได้แก่ ยอดรวมคำขอ โทเค็น และค่าใช้จ่าย รวมถึง
ช่วงงบประมาณรายเดือนเมื่อคีย์มีขีดจำกัด คีย์ที่ไม่มีการวัดปริมาณยังคงแสดง
การใช้งานรวมโดยไม่มีช่วงเปอร์เซ็นต์

การค้นหาโควตาใช้คีย์ที่กำหนดขอบเขตเดียวกับการค้นพบโมเดล หากการค้นหา
โควตาล้มเหลว จะไม่ขัดขวางการเรียกใช้โมเดล

ตรวจสอบสแนปช็อตปัจจุบันด้วย:

```bash
openclaw status --usage
openclaw models status
```

สแนปช็อตผู้ให้บริการเดียวกันนี้พร้อมใช้งานสำหรับ `/status` ในแชตและ UI
การใช้งานของ OpenClaw งบประมาณมีผลทั่วทั้งนโยบาย ดังนั้นคำขอที่ส่งจากไคลเอนต์อื่น
ซึ่งใช้นโยบาย ClawRouter เดียวกันอาจเปลี่ยนเปอร์เซ็นต์ที่เหลือได้

## การแก้ไขปัญหา

| อาการ                                  | ตรวจสอบ                                                                                                                                          |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| ไม่มีโมเดล ClawRouter                     | ยืนยันว่า Plugin เปิดใช้งานและได้รับอนุญาตโดย `plugins.allow` จากนั้นตรวจสอบว่าข้อมูลประจำตัวยังใช้งานอยู่และให้สิทธิ์ผู้ให้บริการที่พร้อมอย่างน้อยหนึ่งราย |
| โมเดล ClawRouter ที่กำหนดค่าหายไป | ตรวจสอบความสามารถ `/v1/catalog` และการรองรับเส้นทาง สัญญาการรับส่งข้อมูลที่ไม่รองรับจะถูกกรองออกโดยเจตนา                            |
| `Unknown model: clawrouter/...`          | เพิ่มการอ้างอิงแค็ตตาล็อกที่ตรงกันทุกประการลงใน `agents.defaults.models` เมื่อใช้แมปการกำหนดค่านั้นเป็นรายการอนุญาต                               |
| `401` หรือ `403` จากแค็ตตาล็อกหรือการใช้งาน     | ออกข้อมูลประจำตัว ClawRouter ใหม่หรือปรับขอบเขตใหม่ OpenClaw จะไม่ย้อนกลับไปใช้คีย์ของผู้ให้บริการต้นทาง                                          |
| การเรียกโมเดลล้มเหลวหลังการค้นพบ         | ตรวจสอบการเชื่อมต่อผู้ให้บริการและสถานะต้นทางใน ClawRouter แล้วลองอีกครั้งหลังสถานะความพร้อมฟื้นตัว                                |
| การใช้งานมียอดรวมแต่ไม่มีเปอร์เซ็นต์       | นโยบายไม่มีการวัดปริมาณ ให้เพิ่มงบประมาณรายเดือนใน ClawRouter เพื่อแสดงช่วงเปอร์เซ็นต์                                                     |

## พฤติกรรมด้านความปลอดภัย

- การค้นหาแค็ตตาล็อกจำกัดขอบเขตตามคีย์พร็อกซีที่กำหนดค่าไว้ และแคชแยกตามขอบเขตข้อมูลประจำตัว (ไดเรกทอรีเอเจนต์ ไดเรกทอรีเวิร์กสเปซ รหัสโปรไฟล์การยืนยันตัวตน และ URL ฐาน)
- คีย์พร็อกซีจะถูกแนบเฉพาะเมื่อส่งคำขอเท่านั้น และไม่ถูกจัดเก็บในเมทาดาทาของโมเดล
- ค่าการระบุแหล่งที่มาและค่าความสัมพันธ์ของคำขอแบบอัตโนมัติจะถูกตัดช่องว่าง และปฏิเสธอักขระควบคุมก่อนส่ง ค่าการระบุแหล่งที่มาจำกัดไว้ที่ 256 อักขระ ส่วนรหัสคำขอจำกัดไว้ที่ 128 อักขระ
- ข้อมูลวินิจฉัยการรับส่งข้อมูลของโมเดลมีเฉพาะเมทาดาทา และจะไม่มีคีย์พร็อกซีหรือเนื้อหาของโมเดลโดยเด็ดขาด
- รหัสโมเดลแบบเนทีฟของ Anthropic และ Gemini จะถูกเขียนใหม่เป็นรหัสต้นทางเฉพาะเมื่อส่งคำขอเท่านั้น
- แถวในแค็ตตาล็อกที่ไม่รองรับหรือไม่ได้รับสิทธิ์จะปฏิเสธการทำงานโดยปริยายและไม่สามารถเลือกได้
