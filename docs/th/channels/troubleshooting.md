ใช้หน้านี้เมื่อช่องทางเชื่อมต่อได้ แต่ทำงานไม่ถูกต้อง

## ลำดับคำสั่ง

เริ่มต้นด้วยการเรียกใช้คำสั่งเหล่านี้ตามลำดับ:

```bash
openclaw status
openclaw gateway status
openclaw logs --follow
openclaw doctor
openclaw channels status --probe
```

ค่าพื้นฐานเมื่อระบบทำงานปกติ:

- `Runtime: running`
- `Connectivity probe: ok`
- `Capability: read-only`, `write-capable` หรือ `admin-capable`
- การตรวจสอบช่องทางแสดงว่าการรับส่งข้อมูลเชื่อมต่อแล้ว และในกรณีที่รองรับจะแสดง `works` หรือ `audit ok`

## หลังการอัปเดต

ใช้ขั้นตอนนี้เมื่อ Telegram, iMessage, การกำหนดค่าจากยุค BlueBubbles หรือช่องทาง Plugin อื่นหายไปหลังการอัปเดต

```bash
openclaw status --all
openclaw doctor --fix
openclaw gateway restart
openclaw status --all
```

มองหา `plugin load failed: dependency tree corrupted; run openclaw doctor --fix` ใน `openclaw
status --all` ข้อความนี้หมายความว่าช่องทางได้รับการกำหนดค่าแล้ว แต่การตั้งค่าหรือโหลด Plugin พบโครงสร้างลำดับชั้นการขึ้นต่อกันที่เสียหาย แทนที่จะลงทะเบียนช่องทาง `openclaw doctor --fix` จะล้าง symlink ของการขึ้นต่อกันในรันไทม์ Plugin ที่ล้าสมัยและเงาข้อมูลยืนยันตัวตนที่ล้าสมัย จากนั้น `openclaw gateway restart` จะโหลดสถานะที่สะอาดใหม่

## WhatsApp

### ลักษณะอาการขัดข้องของ WhatsApp

| อาการ                                | วิธีตรวจสอบที่เร็วที่สุด                             | วิธีแก้ไข                                                                                                                                       |
| ----------------------------------- | --------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| เชื่อมต่อแล้วแต่ไม่ตอบกลับ DM         | `openclaw pairing list whatsapp`                    | อนุมัติผู้ส่งหรือเปลี่ยนนโยบาย DM/รายการอนุญาต                                                                                                  |
| ละเว้นข้อความกลุ่ม                    | ตรวจสอบ `requireMention` และรูปแบบการกล่าวถึงในการกำหนดค่า | กล่าวถึงบอตหรือผ่อนปรนนโยบายการกล่าวถึงสำหรับกลุ่มนั้น                                                                                           |
| การเข้าสู่ระบบด้วย QR หมดเวลาด้วย 408 | ตรวจสอบตัวแปรสภาพแวดล้อม `HTTPS_PROXY` / `HTTP_PROXY` ของ Gateway | ตั้งค่าพร็อกซีที่เข้าถึงได้ ใช้ `NO_PROXY` เฉพาะเมื่อต้องการข้ามพร็อกซี                                                                         |
| วนตัดการเชื่อมต่อ/เข้าสู่ระบบใหม่แบบสุ่ม | `openclaw channels status --probe` และบันทึกเหตุการณ์ | ระบบจะแจ้งเตือนการเชื่อมต่อใหม่ที่เพิ่งเกิดขึ้น แม้ขณะนี้จะเชื่อมต่ออยู่ ให้เฝ้าดูบันทึกเหตุการณ์ รีสตาร์ต Gateway แล้วเชื่อมโยงใหม่หากยังตัดต่อซ้ำ |
| วน `status=408 Request Time-out`     | ตรวจสอบช่องทาง บันทึกเหตุการณ์ doctor แล้วตรวจสอบสถานะ Gateway | แก้ไขการเชื่อมต่อ/การกำหนดเวลาของโฮสต์ก่อน สำรองข้อมูลยืนยันตัวตนและเชื่อมโยงบัญชีใหม่หากยังเกิดซ้ำ                                              |
| การตอบกลับมาถึงช้าหลายวินาที/นาที    | `openclaw doctor --fix`                             | Doctor จะหยุดไคลเอนต์ TUI ภายในเครื่องที่ตรวจสอบแล้วว่าล้าสมัย เมื่อไคลเอนต์เหล่านั้นทำให้ลูปเหตุการณ์ของ Gateway มีประสิทธิภาพลดลง                 |

การแก้ไขปัญหาแบบครบถ้วน: [การแก้ไขปัญหา WhatsApp](/th/channels/whatsapp#troubleshooting)

| `setMyCommands` ถูกปฏิเสธเมื่อเริ่มต้นระบบ | ตรวจสอบบันทึกเหตุการณ์เพื่อหา `BOT_COMMANDS_TOO_MUCH` | ลดจำนวนคำสั่ง Telegram จาก Plugin/Skills/คำสั่งที่กำหนดเอง หรือปิดใช้งานเมนูแบบเนทีฟ                                                                  |
| หลังอัปเกรดแล้วรายการอนุญาตบล็อกคุณ      | `openclaw security audit` และรายการอนุญาตในการกำหนดค่า | เรียกใช้ `openclaw doctor --fix` หรือแทนที่ `@username` ด้วย ID ตัวเลขของผู้ส่ง                                                                          |
