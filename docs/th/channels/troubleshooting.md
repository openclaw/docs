ใช้หน้านี้เมื่อช่องทางเชื่อมต่อได้ แต่ทำงานไม่ถูกต้อง

## ลำดับคำสั่ง

ให้เรียกใช้คำสั่งเหล่านี้ตามลำดับก่อน:

```bash
openclaw status
openclaw gateway status
openclaw logs --follow
openclaw doctor
openclaw channels status --probe
```

สถานะพื้นฐานที่ปกติ:

- `Runtime: running`
- `Connectivity probe: ok`
- `Capability: read-only`, `write-capable` หรือ `admin-capable`
- การตรวจสอบช่องทางแสดงว่าการรับส่งข้อมูลเชื่อมต่อแล้ว และแสดง `works` หรือ `audit ok` ในกรณีที่รองรับ

## หลังการอัปเดต

ใช้ส่วนนี้เมื่อ Telegram, iMessage, การกำหนดค่าจากยุค BlueBubbles หรือช่องทาง Plugin อื่นหายไป
หลังจากอัปเดต

```bash
openclaw status --all
openclaw doctor --fix
openclaw gateway restart
openclaw status --all
```

มองหา `plugin load failed: dependency tree corrupted; run openclaw doctor --fix` ใน `openclaw
status --all` ซึ่งหมายความว่ามีการกำหนดค่าช่องทางแล้ว แต่การตั้งค่า/โหลด Plugin พบโครงสร้าง
การขึ้นต่อกันที่เสียหายแทนที่จะลงทะเบียนช่องทาง `openclaw doctor --fix` จะล้าง symlink
การขึ้นต่อกันของรันไทม์ Plugin ที่ค้างอยู่และเงาการยืนยันตัวตนที่ค้างอยู่ จากนั้น `openclaw gateway restart` จะโหลด
สถานะที่สะอาดอีกครั้ง

## WhatsApp

### ลักษณะอาการขัดข้องของ WhatsApp

| อาการ                             | วิธีตรวจสอบที่เร็วที่สุด                                       | วิธีแก้ไข                                                                                                                              |
| ----------------------------------- | --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| เชื่อมต่อแล้วแต่ไม่ตอบกลับ DM         | `openclaw pairing list whatsapp`                    | อนุมัติผู้ส่งหรือเปลี่ยนนโยบาย DM/รายการที่อนุญาต                                                                                    |
| ไม่สนใจข้อความกลุ่ม              | ตรวจสอบ `requireMention` และรูปแบบการกล่าวถึงในการกำหนดค่า | กล่าวถึงบอตหรือผ่อนปรนนโยบายการกล่าวถึงสำหรับกลุ่มนั้น                                                                          |
| การเข้าสู่ระบบด้วย QR หมดเวลาด้วย 408         | ตรวจสอบตัวแปรสภาพแวดล้อม `HTTPS_PROXY` / `HTTP_PROXY` ของ Gateway      | ตั้งค่าพร็อกซีที่เข้าถึงได้ ใช้ `NO_PROXY` สำหรับการข้ามเท่านั้น                                                                         |
| วนตัดการเชื่อมต่อ/เข้าสู่ระบบใหม่แบบสุ่ม     | `openclaw channels status --probe` และบันทึกเหตุการณ์           | ระบบจะแจ้งเตือนการเชื่อมต่อใหม่ล่าสุดแม้ปัจจุบันจะเชื่อมต่ออยู่ ให้เฝ้าดูบันทึกเหตุการณ์ รีสตาร์ต Gateway แล้วเชื่อมโยงใหม่หากยังคงเชื่อมต่อสลับไปมา |
| วน `status=408 Request Time-out`  | ตรวจสอบเชิงรุก บันทึกเหตุการณ์ doctor แล้วตรวจสอบสถานะ Gateway            | แก้ไขการเชื่อมต่อ/การจับเวลาของโฮสต์ก่อน สำรองข้อมูลการยืนยันตัวตนแล้วเชื่อมโยงบัญชีใหม่หากยังคงเกิดการวนซ้ำ                                   |
| การตอบกลับมาถึงช้าหลายวินาที/นาที | `openclaw doctor --fix`                             | Doctor จะหยุดไคลเอ็นต์ TUI ภายในเครื่องที่ยืนยันแล้วว่าค้างอยู่ เมื่อไคลเอ็นต์เหล่านั้นทำให้ลูปเหตุการณ์ของ Gateway ทำงานด้อยลง                                    |
