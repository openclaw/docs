**สถานะ:** Plugin ที่รวมมาให้แบบเลือกใช้ได้ (ปิดใช้งานโดยค่าเริ่มต้นจนกว่าจะกำหนดค่า)

Nostr เป็นโปรโตคอลแบบกระจายศูนย์สำหรับเครือข่ายสังคม ช่องทางนี้ช่วยให้ OpenClaw รับและตอบกลับข้อความโดยตรง (DM) ที่เข้ารหัสผ่าน NIP-04 ได้

## Plugin ที่รวมมาให้

OpenClaw รุ่นปัจจุบันจัดส่ง Nostr มาเป็น Plugin ที่รวมมาให้ ดังนั้นบิลด์แบบแพ็กเกจปกติ
ไม่ต้องติดตั้งแยกต่างหาก

### การติดตั้งรุ่นเก่า/แบบกำหนดเอง

- การเริ่มใช้งาน (`openclaw onboard`) และ `openclaw channels add` ยังแสดง
  Nostr จากแค็ตตาล็อกช่องทางที่ใช้ร่วมกัน
- หากบิลด์ของคุณไม่รวม Nostr ที่รวมมาให้ ให้ติดตั้งแพ็กเกจ npm ปัจจุบันเมื่อมี
  การเผยแพร่แล้ว

```bash
openclaw plugins install @openclaw/nostr
```

หาก npm รายงานว่าแพ็กเกจที่ OpenClaw เป็นเจ้าของเลิกใช้แล้ว ให้ใช้บิลด์ OpenClaw
แบบแพ็กเกจปัจจุบัน หรือเช็กเอาต์ในเครื่องจนกว่าจะมีการเผยแพร่แพ็กเกจ npm รุ่นใหม่กว่า

ใช้เช็กเอาต์ในเครื่อง (เวิร์กโฟลว์สำหรับการพัฒนา):

```bash
openclaw plugins install --link <path-to-local-nostr-plugin>
```

รีสตาร์ท Gateway หลังติดตั้งหรือเปิดใช้งาน Plugin

### การตั้งค่าแบบไม่โต้ตอบ

```bash
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY"
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY" --relay-urls "wss://relay.damus.io,wss://relay.primal.net"
```

ใช้ `--use-env` เพื่อเก็บ `NOSTR_PRIVATE_KEY` ไว้ในสภาพแวดล้อมแทนการเก็บคีย์ไว้ในการกำหนดค่า

## การตั้งค่าอย่างรวดเร็ว

1. สร้างคู่คีย์ Nostr (หากจำเป็น):

```bash
# Using nak
nak key generate
```

2. เพิ่มลงในการกำหนดค่า:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
    },
  },
}
```

3. ส่งออกคีย์:

```bash
export NOSTR_PRIVATE_KEY="nsec1..."
```

4. รีสตาร์ท Gateway

## ข้อมูลอ้างอิงการกำหนดค่า

| คีย์         | ประเภท   | ค่าเริ่มต้น                                | คำอธิบาย                             |
| ------------ | -------- | ------------------------------------------- | ----------------------------------- |
| `privateKey` | string   | จำเป็น                                     | คีย์ส่วนตัวในรูปแบบ `nsec` หรือ hex |
| `relays`     | string[] | `['wss://relay.damus.io', 'wss://nos.lol']` | URL ของรีเลย์ (WebSocket)           |
| `dmPolicy`   | string   | `pairing`                                   | นโยบายการเข้าถึง DM                 |
| `allowFrom`  | string[] | `[]`                                        | pubkey ของผู้ส่งที่อนุญาต           |
| `enabled`    | boolean  | `true`                                      | เปิด/ปิดใช้งานช่องทาง               |
| `name`       | string   | -                                           | ชื่อที่แสดง                         |
| `profile`    | object   | -                                           | เมตาดาต้าโปรไฟล์ NIP-01             |

## เมตาดาต้าโปรไฟล์

ข้อมูลโปรไฟล์จะเผยแพร่เป็นเหตุการณ์ NIP-01 `kind:0` คุณสามารถจัดการได้จาก Control UI (Channels -> Nostr -> Profile) หรือตั้งค่าโดยตรงในการกำหนดค่า

ตัวอย่าง:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      profile: {
        name: "openclaw",
        displayName: "OpenClaw",
        about: "Personal assistant DM bot",
        picture: "https://example.com/avatar.png",
        banner: "https://example.com/banner.png",
        website: "https://example.com",
        nip05: "openclaw@example.com",
        lud16: "openclaw@example.com",
      },
    },
  },
}
```

หมายเหตุ:

- URL ของโปรไฟล์ต้องใช้ `https://`
- การนำเข้าจากรีเลย์จะผสานฟิลด์และคงการแทนที่ในเครื่องไว้

## การควบคุมการเข้าถึง

### นโยบาย DM

- **pairing** (ค่าเริ่มต้น): ผู้ส่งที่ไม่รู้จักจะได้รับรหัสจับคู่
- **allowlist**: เฉพาะ pubkey ใน `allowFrom` เท่านั้นที่สามารถส่ง DM ได้
- **open**: DM ขาเข้าสาธารณะ (ต้องมี `allowFrom: ["*"]`)
- **disabled**: เพิกเฉยต่อ DM ขาเข้า

หมายเหตุการบังคับใช้:

- ลายเซ็นเหตุการณ์ขาเข้าจะถูกตรวจสอบก่อนนโยบายผู้ส่งและการถอดรหัส NIP-04 ดังนั้นเหตุการณ์ปลอมจะถูกปฏิเสธตั้งแต่เนิ่นๆ
- การตอบกลับการจับคู่จะถูกส่งโดยไม่ประมวลผลเนื้อหา DM เดิม
- DM ขาเข้าจะถูกจำกัดอัตรา และเพย์โหลดที่ใหญ่เกินจะถูกทิ้งก่อนถอดรหัส

### ตัวอย่างรายการอนุญาต

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      dmPolicy: "allowlist",
      allowFrom: ["npub1abc...", "npub1xyz..."],
    },
  },
}
```

## รูปแบบคีย์

รูปแบบที่รองรับ:

- **คีย์ส่วนตัว:** `nsec...` หรือ hex 64 อักขระ
- **Pubkey (`allowFrom`):** `npub...` หรือ hex

## รีเลย์

ค่าเริ่มต้น: `relay.damus.io` และ `nos.lol`

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["wss://relay.damus.io", "wss://relay.primal.net", "wss://nostr.wine"],
    },
  },
}
```

เคล็ดลับ:

- ใช้รีเลย์ 2-3 รายการเพื่อความสำรอง
- หลีกเลี่ยงการใช้รีเลย์มากเกินไป (เวลาแฝง, การซ้ำซ้อน)
- รีเลย์แบบชำระเงินสามารถปรับปรุงความน่าเชื่อถือได้
- รีเลย์ในเครื่องเหมาะสำหรับการทดสอบ (`ws://localhost:7777`)

## การรองรับโปรโตคอล

| NIP    | สถานะ    | คำอธิบาย                              |
| ------ | --------- | ------------------------------------- |
| NIP-01 | รองรับแล้ว | รูปแบบเหตุการณ์พื้นฐาน + เมตาดาต้าโปรไฟล์ |
| NIP-04 | รองรับแล้ว | DM ที่เข้ารหัส (`kind:4`)             |
| NIP-17 | วางแผนไว้ | DM แบบ gift-wrapped                   |
| NIP-44 | วางแผนไว้ | การเข้ารหัสแบบมีเวอร์ชัน              |

## การทดสอบ

### รีเลย์ในเครื่อง

```bash
# Start strfry
docker run -p 7777:7777 ghcr.io/hoytech/strfry
```

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["ws://localhost:7777"],
    },
  },
}
```

### การทดสอบด้วยตนเอง

1. จด pubkey ของบอต (npub) จากบันทึก
2. เปิดไคลเอนต์ Nostr (Damus, Amethyst ฯลฯ)
3. ส่ง DM ไปยัง pubkey ของบอต
4. ตรวจสอบการตอบกลับ

## การแก้ไขปัญหา

### ไม่ได้รับข้อความ

- ตรวจสอบว่าคีย์ส่วนตัวถูกต้อง
- ตรวจสอบให้แน่ใจว่า URL ของรีเลย์เข้าถึงได้และใช้ `wss://` (หรือ `ws://` สำหรับในเครื่อง)
- ยืนยันว่า `enabled` ไม่ใช่ `false`
- ตรวจสอบบันทึก Gateway เพื่อหาข้อผิดพลาดการเชื่อมต่อรีเลย์

### ไม่ส่งการตอบกลับ

- ตรวจสอบว่ารีเลย์ยอมรับการเขียน
- ตรวจสอบการเชื่อมต่อขาออก
- เฝ้าดูการจำกัดอัตราของรีเลย์

### การตอบกลับซ้ำ

- เป็นสิ่งที่คาดไว้เมื่อใช้หลายรีเลย์
- ข้อความจะถูกขจัดรายการซ้ำตาม ID เหตุการณ์ เฉพาะการส่งมอบครั้งแรกเท่านั้นที่ทริกเกอร์การตอบกลับ

## ความปลอดภัย

- อย่าคอมมิตคีย์ส่วนตัว
- ใช้ตัวแปรสภาพแวดล้อมสำหรับคีย์
- พิจารณาใช้ `allowlist` สำหรับบอตในระบบจริง
- ลายเซ็นจะถูกตรวจสอบก่อนนโยบายผู้ส่ง และนโยบายผู้ส่งจะถูกบังคับใช้ก่อนถอดรหัส ดังนั้นเหตุการณ์ปลอมจะถูกปฏิเสธตั้งแต่เนิ่นๆ และผู้ส่งที่ไม่รู้จักไม่สามารถบังคับให้ทำงานเข้ารหัสเต็มรูปแบบได้

## ข้อจำกัด (MVP)

- เฉพาะข้อความโดยตรงเท่านั้น (ไม่มีแชตกลุ่ม)
- ไม่มีไฟล์แนบสื่อ
- เฉพาะ NIP-04 เท่านั้น (มีแผนสำหรับ NIP-17 gift-wrap)
