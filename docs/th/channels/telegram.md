## Mini App ของแดชบอร์ด

เรียกใช้ `/dashboard` ใน DM กับบอต เพื่อเปิดแดชบอร์ด OpenClaw ภายใน Telegram

ข้อกำหนด:

- `gateway.tailscale.mode: "serve"` หรือ `"funnel"` สำหรับ URL HTTPS ของ Mini App ที่เผยแพร่แล้ว
- ID ผู้ใช้ Telegram แบบตัวเลขของคุณต้องอยู่ใน `allowFrom` ที่มีผลของบัญชีที่เลือก หรืออยู่ใน `commands.ownerAllowFrom`
- ใช้ DM ในกลุ่ม `/dashboard` จะตอบกลับด้วย `open this in a DM with the bot` และไม่ส่งปุ่ม
- การติดตั้งด้วย Docker: โหมด Serve/Funnel กำหนดให้ Gateway ผูกกับ loopback ข้าง `tailscaled` ซึ่งระบบเครือข่ายแบบบริดจ์ที่เผยแพร่พอร์ตไม่สามารถรองรับได้ ให้เรียกใช้คอนเทนเนอร์ Gateway ด้วย `network_mode: host` และเมานต์ซ็อกเก็ต `tailscaled` ของโฮสต์ (`/var/run/tailscale`) พร้อม CLI `tailscale` เข้าในคอนเทนเนอร์

Mini App เป็นเส้นทาง v1 ที่ใช้ได้เฉพาะ Tailscale และไม่รองรับ iframe ของ Telegram Web

### ข้อมูลประจำตัวของบอตในกลุ่ม

ในกลุ่มและหัวข้อฟอรัม การกล่าวถึงแฮนเดิลบอตที่กำหนดไว้อย่างชัดเจน (เช่น `@my_bot`) จะเป็นการส่งถึงเอเจนต์ OpenClaw ที่เลือก แม้ว่าชื่อบุคลิกของเอเจนต์จะแตกต่างจากชื่อผู้ใช้ Telegram ก็ตาม นโยบายไม่ตอบในกลุ่มยังคงใช้กับการรับส่งข้อมูลที่ไม่เกี่ยวข้อง แต่แฮนเดิลของบอตเองจะไม่ถือเป็น "บุคคลอื่น"

<Tabs>
  <Tab title="นโยบาย DM">
    `channels.telegram.dmPolicy` ควบคุมการเข้าถึงข้อความโดยตรง:

    - `pairing` (ค่าเริ่มต้น)
    - `allowlist` (ต้องมี ID ผู้ส่งอย่างน้อยหนึ่งรายการใน `allowFrom`)
    - `open` (กำหนดให้ `allowFrom` มี `"*"`)
    - `disabled`

    `dmPolicy: "open"` ร่วมกับ `allowFrom: ["*"]` ทำให้บัญชี Telegram ใดก็ตามที่พบหรือคาดเดาชื่อผู้ใช้ของบอตได้ สามารถสั่งงานบอตได้ ใช้เฉพาะกับบอตที่ตั้งใจเปิดเป็นสาธารณะและจำกัดเครื่องมืออย่างเข้มงวดเท่านั้น บอตที่มีเจ้าของคนเดียวควรใช้ `allowlist` พร้อม ID ผู้ใช้แบบตัวเลข

    `channels.telegram.allowFrom` รองรับ ID ผู้ใช้ Telegram แบบตัวเลข คำนำหน้า `telegram:` / `tg:` รองรับและจะถูกปรับเป็นรูปแบบมาตรฐาน
    ใน config หลายบัญชี `channels.telegram.allowFrom` ระดับบนสุดที่เข้มงวดเป็นขอบเขตความปลอดภัย โดย `allowFrom: ["*"]` ระดับบัญชีจะไม่ทำให้บัญชีนั้นเป็นสาธารณะ เว้นแต่รายการอนุญาตที่มีผลหลังรวมแล้วยังมีไวลด์การ์ดอย่างชัดเจน
    `dmPolicy: "allowlist"` ที่มี `allowFrom` ว่างจะบล็อก DM ทั้งหมด และการตรวจสอบ config จะปฏิเสธ
    การตั้งค่าจะขอเฉพาะ ID ผู้ใช้แบบตัวเลข หาก config มีรายการในรายการอนุญาต `@username` จากการตั้งค่าเก่า ให้เรียกใช้ `openclaw doctor --fix` เพื่อแปลงเป็น ID แบบตัวเลข (ดำเนินการอย่างเต็มความสามารถ ต้องใช้โทเค็นบอต Telegram)
    หากก่อนหน้านี้อาศัยไฟล์รายการอนุญาตของที่เก็บการจับคู่ `openclaw doctor --fix` สามารถกู้คืนรายการเข้าไปใน `channels.telegram.allowFrom` สำหรับขั้นตอนที่ใช้รายการอนุญาตได้ (เช่น เมื่อ `dmPolicy: "allowlist"` ยังไม่มี ID ระบุไว้อย่างชัดเจน)

    สำหรับบอตที่มีเจ้าของคนเดียว ควรใช้ `dmPolicy: "allowlist"` พร้อม ID `allowFrom` แบบตัวเลขที่ระบุชัดเจน แทนการพึ่งพาการอนุมัติการจับคู่ก่อนหน้า

    ความสับสนที่พบบ่อย: การอนุมัติการจับคู่ DM ไม่ได้หมายความว่า "ผู้ส่งรายนี้ได้รับอนุญาตทุกที่" การจับคู่ให้สิทธิ์เข้าถึง DM เท่านั้น หากยังไม่มีเจ้าของคำสั่ง การจับคู่รายการแรกที่อนุมัติจะตั้งค่า `commands.ownerAllowFrom` ด้วย ทำให้คำสั่งสำหรับเจ้าของเท่านั้นและการอนุมัติการดำเนินการมีบัญชีผู้ควบคุมอย่างชัดเจน ส่วนการอนุญาตผู้ส่งในกลุ่มยังคงมาจากรายการอนุญาตที่กำหนดไว้อย่างชัดเจนใน config
    หากต้องการให้ข้อมูลประจำตัวเดียวได้รับอนุญาตทั้งสำหรับ DM และคำสั่งในกลุ่ม: ใส่ ID ผู้ใช้ Telegram แบบตัวเลขของคุณใน `channels.telegram.allowFrom` และสำหรับคำสั่งเฉพาะเจ้าของ ให้ตรวจสอบว่า `commands.ownerAllowFrom` มี `telegram:<your user id>`

    การควบคุมสองรายการต่อไปนี้ทำงานร่วมกัน:

    1. **กลุ่มใดได้รับอนุญาต** (`channels.telegram.groups`)
       - ไม่มี config `groups`, `groupPolicy: "open"`: ทุกกลุ่มจะผ่านการตรวจสอบ ID กลุ่ม
       - ไม่มี config `groups`, `groupPolicy: "allowlist"` (ค่าเริ่มต้น): บล็อกทุกกลุ่มจนกว่าจะเพิ่มรายการ `groups` (หรือ `"*"`)
       - มีการกำหนดค่า `groups`: ทำหน้าที่เป็นรายการอนุญาต (ID ที่ระบุชัดเจนหรือ `"*"`)

    2. **ผู้ส่งรายใดได้รับอนุญาตในกลุ่ม** (`channels.telegram.groupPolicy`)
       - `open` / `allowlist` (ค่าเริ่มต้น) / `disabled`

    `groupAllowFrom` กรองผู้ส่งในกลุ่ม หากไม่ได้ตั้งค่า Telegram จะใช้ `allowFrom` เป็นค่าสำรอง (ไม่ใช่ที่เก็บการจับคู่ — การอนุญาตผู้ส่งในกลุ่มจะไม่สืบทอดการอนุมัติจากที่เก็บการจับคู่ DM ซึ่งเป็นขอบเขตความปลอดภัยนับตั้งแต่ `2026.2.25`)
    รายการ `groupAllowFrom` ควรเป็น ID ผู้ใช้ Telegram แบบตัวเลข (คำนำหน้า `telegram:` / `tg:` จะถูกปรับเป็นรูปแบบมาตรฐาน) โดยจะละเว้นรายการที่ไม่ใช่ตัวเลข อย่าใส่ ID แชตกลุ่มหรือซูเปอร์กรุ๊ปที่นี่ เพราะ ID แชตที่เป็นค่าลบต้องอยู่ภายใต้ `channels.telegram.groups`
    รูปแบบที่ใช้ได้จริงสำหรับบอตที่มีเจ้าของคนเดียว: ตั้งค่า ID ผู้ใช้ของคุณใน `channels.telegram.allowFrom` ไม่ต้องตั้งค่า `groupAllowFrom` และอนุญาตกลุ่มเป้าหมายภายใต้ `channels.telegram.groups`
    หาก `channels.telegram` ไม่มีอยู่ใน config เลย รันไทม์จะใช้ค่าเริ่มต้นแบบปิดเมื่อเกิดข้อผิดพลาดเป็น `groupPolicy="allowlist"` เว้นแต่จะตั้งค่า `channels.defaults.groupPolicy` อย่างชัดเจน

    การตั้งค่ากลุ่มสำหรับเจ้าของเท่านั้น:

```json5
{
  channels: {
    telegram: {
      enabled: true,
      dmPolicy: "pairing",
      allowFrom: ["<YOUR_TELEGRAM_USER_ID>"],
      groupPolicy: "allowlist",
      groups: {
        "<GROUP_CHAT_ID>": {
          requireMention: true,
        },
      },
    },
  },
}
```

    ทดสอบจากกลุ่มด้วย `@<bot_username> ping` ข้อความทั่วไปในกลุ่มจะไม่เรียกใช้บอตขณะที่ `requireMention: true`

    อนุญาตสมาชิกทุกคนในกลุ่มที่ระบุหนึ่งกลุ่ม:

```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          groupPolicy: "open",
          requireMention: false,
        },
      },
    },
  },
}
```

    อนุญาตเฉพาะผู้ใช้ที่ระบุภายในกลุ่มที่ระบุหนึ่งกลุ่ม:

```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          requireMention: true,
          allowFrom: ["8734062810", "745123456"],
        },
      },
    },
  },
}
```

    <Warning>
      ข้อผิดพลาดที่พบบ่อย: `groupAllowFrom` ไม่ใช่รายการอนุญาตของกลุ่ม

      - ID แชตกลุ่ม/ซูเปอร์กรุ๊ป Telegram ที่เป็นค่าลบ (`-1001234567890`) ต้องอยู่ภายใต้ `channels.telegram.groups`
      - ID ผู้ใช้ Telegram (`8734062810`) ต้องอยู่ภายใต้ `groupAllowFrom` เพื่อจำกัดว่าบุคคลใดภายในกลุ่มที่อนุญาตสามารถเรียกใช้บอตได้
      - ใช้ `groupAllowFrom: ["*"]` เฉพาะเมื่อต้องการให้สมาชิกทุกคนในกลุ่มที่อนุญาตพูดคุยกับบอตได้

    </Warning>

  </Tab>

  <Tab title="ลักษณะการกล่าวถึง">

    - การกล่าวถึง `@botusername` แบบเนทีฟ หรือ
    - รูปแบบการกล่าวถึงใน `agents.list[].groupChat.mentionPatterns` หรือ `messages.groupChat.mentionPatterns`

    ตัวสลับระดับเซสชัน (เฉพาะสถานะ ไม่คงอยู่ถาวร): `/activation always`, `/activation mention` ใช้ config เพื่อให้คงอยู่ถาวร:

```json5

    telegram: {
      groups: {
        "*": { requireMention: false },
      },
    },

```

    บริบทประวัติกลุ่มเปิดอยู่เสมอและจำกัดด้วย `historyLimit` ตั้งค่า `channels.telegram.historyLimit: 0` เพื่อปิดหน้าต่างประวัติกลุ่ม `openclaw doctor --fix` จะลบคีย์ `includeGroupHistoryContext` ที่เลิกใช้แล้ว

    การรับ ID แชตกลุ่ม: ส่งต่อข้อความกลุ่มไปยัง `@userinfobot` / `@getidsbot` อ่าน `chat.id` จาก `openclaw logs --follow` ตรวจสอบ `getUpdates` ของ Bot API หรือเรียกใช้ `/whoami@<bot_username>` เมื่อกลุ่มได้รับอนุญาตแล้ว

## ลักษณะการทำงานขณะรันไทม์

- Telegram ทำงานภายในโพรเซส Gateway
- การกำหนดเส้นทางเป็นแบบตายตัว: การตอบกลับข้อความขาเข้าจาก Telegram จะส่งกลับไปยัง Telegram (โมเดลไม่ได้เป็นผู้เลือกช่องทาง)
- ข้อความขาเข้าจะถูกปรับให้อยู่ในรูปแบบเอนเวโลปช่องทางที่ใช้ร่วมกัน พร้อมข้อมูลเมตาของการตอบกลับ ตัวยึดตำแหน่งสื่อ และบริบทสายการตอบกลับที่บันทึกไว้สำหรับข้อความตอบกลับที่ Gateway ตรวจพบ
- เซสชันกลุ่มจะแยกจากกันตาม ID กลุ่ม หัวข้อฟอรัมจะต่อท้ายด้วย `:topic:<threadId>`
- ข้อความ DM สามารถมี `message_thread_id` ได้ โดย OpenClaw จะเก็บค่านี้ไว้สำหรับการตอบกลับ เซสชันหัวข้อ DM จะแยกเฉพาะเมื่อ Telegram `getMe` รายงาน `has_topics_enabled: true` สำหรับบอต มิฉะนั้น DM จะยังคงใช้เซสชันแบบแบน
- การทำ long polling ใช้ตัวรัน grammY โดยจัดลำดับแยกตามแชต/เธรด ภาวะพร้อมกันของ sink ในตัวรันใช้ `agents.defaults.maxConcurrent`
- การเริ่มต้นหลายบัญชีจะจำกัดจำนวนโพรบ `getMe` ที่ทำงานพร้อมกัน เพื่อไม่ให้กลุ่มบอตขนาดใหญ่กระจายโพรบของทุกบัญชีพร้อมกัน
- แต่ละโพรเซส Gateway ป้องกันการทำ long polling เพื่อให้มี poller ที่ทำงานอยู่เพียงตัวเดียวซึ่งใช้โทเค็นบอตได้ในแต่ละครั้ง ข้อขัดแย้ง 409 ของ `getUpdates` ที่เกิดขึ้นต่อเนื่องบ่งชี้ว่ามี Gateway ของ OpenClaw, สคริปต์ หรือ poller ภายนอกอื่นกำลังใช้โทเค็นเดียวกัน
- โดยค่าเริ่มต้น watchdog ของการ polling จะเริ่มใหม่เมื่อไม่มีการยืนยันความพร้อมทำงานของ `getUpdates` ที่เสร็จสมบูรณ์เป็นเวลา 120 วินาที เพิ่ม `channels.telegram.pollingStallThresholdMs` (30000-600000 รองรับการกำหนดทับแยกตามบัญชี) เฉพาะเมื่อการปรับใช้ของคุณพบการเริ่มใหม่จากการตรวจพบว่าการ polling หยุดชะงักโดยผิดพลาดระหว่างงานที่ใช้เวลานาน
- Telegram Bot API ไม่รองรับการยืนยันว่าอ่านข้อความแล้ว (`sendReadReceipts` ไม่มีผล)

<Note>
  `channels.telegram.dm.threadReplies` และ `channels.telegram.direct.<chatId>.threadReplies` ถูกนำออกแล้ว หากการกำหนดค่าของคุณยังมีคีย์เหล่านี้ ให้เรียกใช้ `openclaw doctor --fix` หลังอัปเกรด ขณะนี้การกำหนดเส้นทางหัวข้อ DM เป็นไปตาม Telegram `getMe.has_topics_enabled` (ควบคุมโดยโหมดเธรดของ BotFather): บอตที่เปิดใช้หัวข้อจะใช้เซสชัน DM ที่จำกัดขอบเขตตามเธรดเมื่อ Telegram ส่ง `message_thread_id`; DM อื่นจะยังคงใช้เซสชันแบบแบน
</Note>

## ข้อมูลอ้างอิงฟีเจอร์

<AccordionGroup>
  <Accordion title="ตัวอย่างสตรีมสด (การแก้ไขข้อความ)">
    OpenClaw สตรีมคำตอบบางส่วนแบบเรียลไทม์ในแชตส่วนตัว กลุ่ม และหัวข้อ โดยส่งข้อความตัวอย่าง จากนั้นเรียก `editMessageText` ซ้ำ ๆ และทำให้เสร็จสมบูรณ์ในตำแหน่งเดิม

    - `channels.telegram.streaming` คือ `off | partial | block | progress` (ค่าเริ่มต้น: `partial`)
    - ตัวอย่างคำตอบเริ่มต้นแบบสั้นจะถูกหน่วงเพื่อลดการเรียกถี่ จากนั้นจึงแสดงเป็นข้อความหลังจากระยะเวลาที่จำกัด หากการทำงานยังดำเนินอยู่
    - `progress` จะเก็บฉบับร่างสถานะที่แก้ไขได้หนึ่งรายการสำหรับความคืบหน้าของเครื่องมือ แสดงป้ายสถานะคงที่เมื่อคำตอบเริ่มทำงานก่อนความคืบหน้าของเครื่องมือ ล้างฉบับร่างเมื่อเสร็จสิ้น และส่งคำตอบสุดท้ายเป็นข้อความปกติ
    - `streaming.preview.toolProgress` ควบคุมว่าการอัปเดตเครื่องมือ/ความคืบหน้าจะใช้ข้อความตัวอย่างที่แก้ไขเดิมซ้ำหรือไม่ (ค่าเริ่มต้น: `true` เมื่อการสตรีมตัวอย่างทำงานอยู่)
    - `streaming.preview.commandText` ควบคุมรายละเอียดคำสั่ง/การเรียกใช้ภายในบรรทัดเหล่านั้น: `raw` (ค่าเริ่มต้น) หรือ `status` (เฉพาะป้ายเครื่องมือ)
    - `streaming.progress.commentary` (ค่าเริ่มต้น: `false`) เลือกเปิดใช้ข้อความคำอธิบาย/เกริ่นนำของผู้ช่วยในฉบับร่างความคืบหน้าชั่วคราว
    - ระบบจะตรวจพบ `channels.telegram.streamMode` แบบเดิม ค่า `streaming` ชนิดบูลีน และคีย์ตัวอย่างฉบับร่างแบบเนทีฟที่เลิกใช้แล้ว ให้เรียกใช้ `openclaw doctor --fix` เพื่อย้ายข้อมูล

    บรรทัดความคืบหน้าของเครื่องมือคือการอัปเดตสถานะแบบสั้นที่แสดงระหว่างเครื่องมือทำงาน (การเรียกใช้คำสั่ง การอ่านไฟล์ การอัปเดตการวางแผน สรุปแพตช์ และคำเกริ่นนำ/คำอธิบายของ Codex ในโหมด app-server) Telegram เปิดใช้รายการเหล่านี้โดยค่าเริ่มต้น (ตรงกับพฤติกรรมที่เผยแพร่ตั้งแต่ `v2026.4.22`+)

    คงการแก้ไขตัวอย่างคำตอบไว้ แต่ซ่อนบรรทัดความคืบหน้าของเครื่องมือ:

    ```json
    {
      "channels": {
        "telegram": {
          "streaming": {
            "mode": "partial",
            "preview": { "toolProgress": false }
          }
        }
      }
    }
    ```

    คงการแสดงความคืบหน้าของเครื่องมือไว้ แต่ซ่อนข้อความคำสั่ง/การเรียกใช้:

    ```json
    {
      "channels": {
        "telegram": {
          "streaming": {
            "mode": "partial",
            "preview": { "commandText": "status" }
          }
        }
      }
    }
    ```

    โหมด `progress` จะแสดงความคืบหน้าของเครื่องมือโดยไม่แก้ไขคำตอบสุดท้ายลงในข้อความนั้น ให้กำหนดนโยบายข้อความคำสั่งภายใต้ `streaming.progress`:

    ```json
    {
      "channels": {
        "telegram": {
          "streaming": {
            "mode": "progress",
            "progress": {
              "toolProgress": true,
              "commandText": "status"
            }
          }
        }
      }
    }
    ```

    `streaming.mode: "off"` ปิดการแก้ไขตัวอย่างและระงับข้อความทั่วไปเกี่ยวกับเครื่องมือ/ความคืบหน้า แทนที่จะส่งเป็นข้อความสถานะแยกต่างหาก ส่วนข้อความขออนุมัติ สื่อ และข้อผิดพลาดจะยังส่งผ่านการนำส่งขั้นสุดท้ายตามปกติ `streaming.preview.toolProgress: false` จะคงไว้เฉพาะการแก้ไขตัวอย่างคำตอบ

    <Note>
      การตอบกลับด้วยข้อความอ้างอิงที่เลือกเป็นข้อยกเว้น เมื่อ `replyToMode` เป็น `first`, `all` หรือ `batched` และข้อความขาเข้ามีข้อความอ้างอิงที่เลือก OpenClaw จะส่งคำตอบสุดท้ายผ่านเส้นทางตอบกลับด้วยข้อความอ้างอิงแบบเนทีฟของ Telegram แทนการแก้ไขตัวอย่างคำตอบ ดังนั้น `streaming.preview.toolProgress` จึงไม่สามารถแสดงบรรทัดสถานะในการโต้ตอบครั้งนั้นได้ การตอบกลับข้อความปัจจุบันที่ไม่มีข้อความอ้างอิงที่เลือกจะยังคงสตรีมตามปกติ ตั้งค่า `replyToMode: "off"` เมื่อการมองเห็นความคืบหน้าของเครื่องมือสำคัญกว่าการตอบกลับด้วยข้อความอ้างอิงแบบเนทีฟ หรือตั้งค่า `streaming.preview.toolProgress: false` เพื่อยอมรับข้อแลกเปลี่ยนดังกล่าว
    </Note>

    สำหรับคำตอบที่มีเฉพาะข้อความ: ตัวอย่างแบบสั้นจะได้รับการแก้ไขขั้นสุดท้ายในตำแหน่งเดิม คำตอบสุดท้ายแบบยาวที่แบ่งเป็นหลายข้อความจะใช้ตัวอย่างเป็นส่วนแรกซ้ำ จากนั้นส่งเฉพาะส่วนที่เหลือ คำตอบสุดท้ายในโหมดความคืบหน้าจะล้างฉบับร่างสถานะและใช้การนำส่งขั้นสุดท้ายตามปกติ หากการแก้ไขขั้นสุดท้ายล้มเหลวก่อนยืนยันว่าเสร็จสมบูรณ์ OpenClaw จะย้อนกลับไปใช้การนำส่งขั้นสุดท้ายตามปกติและล้างตัวอย่างที่ค้างอยู่ สำหรับคำตอบที่ซับซ้อน (เพย์โหลดสื่อ) OpenClaw จะย้อนกลับไปใช้การนำส่งขั้นสุดท้ายตามปกติและล้างตัวอย่างเสมอ

    การสตรีมตัวอย่างและการสตรีมแบบบล็อกไม่สามารถใช้พร้อมกันได้ — เมื่อเปิดใช้การสตรีมแบบบล็อกอย่างชัดเจน OpenClaw จะข้ามสตรีมตัวอย่างเพื่อหลีกเลี่ยงการสตรีมซ้ำซ้อน

    การให้เหตุผล: `/reasoning stream` จะสตรีมการให้เหตุผลลงในตัวอย่างสดระหว่างสร้างคำตอบ จากนั้นลบตัวอย่างการให้เหตุผลหลังการนำส่งขั้นสุดท้าย (ใช้ `/reasoning on` เพื่อให้ยังมองเห็นได้) คำตอบสุดท้ายจะถูกส่งโดยไม่มีข้อความการให้เหตุผล

  </Accordion>

  <Accordion title="การจัดรูปแบบข้อความแบบสมบูรณ์">
    โดยค่าเริ่มต้น ข้อความขาออกใช้ข้อความ HTML มาตรฐานของ Telegram ซึ่งอ่านได้ในไคลเอนต์ปัจจุบัน: ตัวหนา ตัวเอียง ลิงก์ โค้ด สปอยเลอร์ และข้อความอ้างอิง — ไม่ใช่บล็อกแบบสมบูรณ์ที่มีเฉพาะใน Bot API 10.2 (ตารางแบบเนทีฟ รายละเอียด สื่อแบบสมบูรณ์ และสูตร)

    เลือกเปิดใช้ข้อความแบบสมบูรณ์ของ Bot API 10.2:

```json5
{
  channels: {
    telegram: {
      richMessages: true,
    },
  },
}
```

    เมื่อเปิดใช้: เอเจนต์จะได้รับแจ้งว่าบอต/บัญชีนี้รองรับข้อความแบบสมบูรณ์ (พร้อมข้อกำหนดการเขียน Markdown + HTML island ที่รองรับ) ข้อความ Markdown จะแสดงผลผ่าน Markdown IR ของ OpenClaw เป็นบล็อกแบบสมบูรณ์ที่มีชนิดตาม Bot API 10.2 (หัวข้อ ตาราง รายละเอียด รายการตรวจสอบ สื่อแบบสมบูรณ์ สูตร แผนที่ และภาพปะติด) คำบรรยายสื่อยังคงใช้คำบรรยาย HTML ของ Telegram (ข้อความแบบสมบูรณ์ไม่แทนที่คำบรรยาย และคำบรรยายจำกัดที่ 1024 อักขระ)

    วิธีนี้ช่วยไม่ให้ข้อความของโมเดลสัมผัสกับสัญลักษณ์ Markdown แบบสมบูรณ์ของ Telegram ดังนั้นสกุลเงินอย่าง `$400-600K` จะไม่ถูกแยกวิเคราะห์เป็นคณิตศาสตร์ ข้อความแบบสมบูรณ์ที่ยาวจะแบ่งโดยอัตโนมัติตามขีดจำกัดของ Telegram ตารางที่มีเกินขีดจำกัด 20 คอลัมน์จะย้อนกลับไปใช้บล็อกโค้ด

    ค่าเริ่มต้น: ปิด เพื่อความเข้ากันได้กับไคลเอนต์ — ไคลเอนต์ Desktop, Web, Android และไคลเอนต์จากบุคคลที่สามบางรายการในปัจจุบันแสดงข้อความแบบสมบูรณ์ที่ยอมรับแล้วว่าไม่รองรับ ให้ปิดตัวเลือกนี้ไว้ เว้นแต่ไคลเอนต์ทุกตัวที่ใช้กับบอตจะแสดงผลข้อความเหล่านี้ได้ `/status` จะแสดงว่าเซสชันปัจจุบันเปิดหรือปิดข้อความแบบสมบูรณ์อยู่

    ตัวอย่างลิงก์เปิดอยู่โดยค่าเริ่มต้น `channels.telegram.linkPreview: false` จะปิดการตรวจหาเอนทิตีโดยอัตโนมัติสำหรับข้อความแบบสมบูรณ์

  <Accordion title="คำสั่งแบบเนทีฟและคำสั่งกำหนดเอง">
    เมนูคำสั่งของ Telegram จะลงทะเบียนเมื่อเริ่มต้นด้วย `setMyCommands` โดย `commands.native: "auto"` จะเปิดใช้คำสั่งแบบเนทีฟสำหรับ Telegram

    เพิ่มรายการเมนูคำสั่งกำหนดเอง:

```json5
{
  channels: {
    telegram: {
      customCommands: [
        { command: "backup", description: "สำรองข้อมูล Git" },
        { command: "generate", description: "สร้างรูปภาพ" },
      ],
    },
  },
}
```

    กฎ: ชื่อจะถูกปรับให้อยู่ในรูปแบบมาตรฐาน (ตัด `/` ที่นำหน้าออกและเปลี่ยนเป็นตัวพิมพ์เล็ก) รูปแบบที่ใช้ได้คือ `a-z`, `0-9`, `_` ความยาว 1-32 อักขระ คำสั่งกำหนดเองไม่สามารถเขียนทับคำสั่งแบบเนทีฟได้ รายการที่ขัดแย้ง/ซ้ำกันจะถูกข้ามและบันทึกลงในล็อก

    คำสั่งกำหนดเองเป็นเพียงรายการในเมนู — ไม่ได้สร้างพฤติกรรมให้โดยอัตโนมัติ คำสั่งของ Plugin/Skill ยังคงทำงานได้เมื่อพิมพ์ แม้จะไม่แสดงในเมนู Telegram หากปิดใช้คำสั่งแบบเนทีฟ คำสั่งในตัวจะถูกนำออก แต่คำสั่งกำหนดเอง/Plugin อาจยังลงทะเบียนได้หากกำหนดค่าไว้

    ปัญหาที่พบบ่อยในการตั้งค่า:

    - `setMyCommands failed` พร้อม `BOT_COMMANDS_TOO_MUCH` หลังลองตัดรายการใหม่ หมายความว่าเมนูยังคงเกินขีดจำกัด ให้ลดจำนวนคำสั่งของ Plugin/Skill/คำสั่งกำหนดเอง หรือปิดใช้ `channels.telegram.commands.native`
    - หาก `deleteWebhook`, `deleteMyCommands` หรือ `setMyCommands` ล้มเหลวด้วย `404: Not Found` ขณะที่คำสั่ง curl ไปยัง Bot API โดยตรงทำงานได้ โดยทั่วไปหมายความว่า `channels.telegram.apiRoot` ถูกตั้งค่าเป็นปลายทาง `/bot<TOKEN>` แบบเต็ม `apiRoot` ต้องเป็นเฉพาะรากของ Bot API เท่านั้น โดย `openclaw doctor --fix` จะลบ `/bot<TOKEN>` ที่ต่อท้ายโดยไม่ตั้งใจ
    - `getMe returned 401` หมายความว่า Telegram ปฏิเสธโทเค็นบอตที่กำหนดค่าไว้ ให้อัปเดต `botToken`, `tokenFile` หรือ `TELEGRAM_BOT_TOKEN` (บัญชีเริ่มต้น) ด้วยโทเค็น BotFather ปัจจุบัน OpenClaw จะหยุดก่อนเริ่ม polling ดังนั้นจึงไม่รายงานปัญหานี้เป็นความล้มเหลวในการล้าง Webhook
    - `setMyCommands failed` พร้อมข้อผิดพลาดเครือข่าย/fetch โดยทั่วไปหมายความว่า DNS/HTTPS ขาออกไปยัง `api.telegram.org` ถูกบล็อก

    ### คำสั่งจับคู่อุปกรณ์ (Plugin `device-pair`)

    เมื่อติดตั้งแล้ว:

    1. `/pair` สร้างรหัสตั้งค่า
    2. วางรหัสในแอป iOS
    3. `/pair pending` แสดงรายการคำขอที่รอดำเนินการ (รวมบทบาท/ขอบเขต)
    4. อนุมัติ: `/pair approve <requestId>`, `/pair approve` (เฉพาะคำขอที่รอดำเนินการ) หรือ `/pair approve latest`

    หากอุปกรณ์ลองใหม่โดยเปลี่ยนรายละเอียดการยืนยันตัวตน (บทบาท ขอบเขต หรือกุญแจสาธารณะ) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่ด้วย `requestId` ใหม่ ให้เรียกใช้ `/pair pending` อีกครั้งก่อนอนุมัติ

    รายละเอียดเพิ่มเติม: [การจับคู่](/th/channels/pairing#pair-via-telegram)

  <Accordion title="หัวข้อฟอรัมและลักษณะการทำงานของเธรด">
    ซูเปอร์กรุ๊ปแบบฟอรัม: คีย์เซสชันของหัวข้อจะต่อท้าย `:topic:<threadId>`; การตอบกลับและสถานะกำลังพิมพ์จะกำหนดเป้าหมายไปยังเธรดของหัวข้อ; เส้นทางการกำหนดค่าหัวข้อคือ `channels.telegram.groups.<chatId>.topics.<threadId>`

    หัวข้อทั่วไป (`threadId=1`) เป็นกรณีพิเศษ: การส่งข้อความจะละเว้น `message_thread_id` (Telegram ปฏิเสธ `sendMessage(...thread_id=1)` ด้วยข้อความ "ไม่พบเธรด") แต่การดำเนินการแสดงสถานะกำลังพิมพ์ยังคงรวม `message_thread_id` (จากการทดสอบจริง จำเป็นเพื่อให้ตัวบ่งชี้สถานะกำลังพิมพ์ปรากฏ)

    รายการหัวข้อจะสืบทอดการตั้งค่ากลุ่ม เว้นแต่จะถูกเขียนทับ (`requireMention`, `allowFrom`, `skills`, `systemPrompt`, `enabled`, `groupPolicy`) ส่วน `agentId` ใช้เฉพาะหัวข้อและไม่สืบทอดจากค่าเริ่มต้นของกลุ่ม `topics."*"` กำหนดค่าเริ่มต้นสำหรับทุกหัวข้อในกลุ่มนั้น; ID หัวข้อที่ตรงกันทุกประการยังคงมีลำดับความสำคัญเหนือ `"*"`

    **การกำหนดเส้นทางเอเจนต์รายหัวข้อ**: แต่ละหัวข้อสามารถกำหนดเส้นทางไปยังเอเจนต์ต่างกันผ่าน `agentId` ในการกำหนดค่าหัวข้อ ทำให้แต่ละหัวข้อมีพื้นที่ทำงาน หน่วยความจำ และเซสชันของตนเอง:

    ```json5
    {
      channels: {
        telegram: {
          groups: {
            "-1001234567890": {
              topics: {
                "1": { agentId: "main" },      // หัวข้อทั่วไป -> เอเจนต์หลัก
                "3": { agentId: "zu" },        // หัวข้อการพัฒนา -> เอเจนต์ zu
                "5": { agentId: "coder" }      // การรีวิวโค้ด -> เอเจนต์ coder
              }
            }
          }
        }
      }
    }
    ```

    จากนั้นแต่ละหัวข้อจะมีคีย์เซสชันของตนเอง เช่น `agent:zu:telegram:group:-1001234567890:topic:3`

    **การผูกหัวข้อ ACP แบบถาวร**: หัวข้อฟอรัมสามารถปักหมุดเซสชันฮาร์เนส ACP ผ่านการผูกที่มีชนิดในระดับบนสุด (`bindings[]` พร้อม `type: "acp"`, `match.channel: "telegram"`, `peer.kind: "group"` และ ID ที่ระบุหัวข้อ เช่น `-1001234567890:topic:42`) ปัจจุบันจำกัดขอบเขตไว้เฉพาะหัวข้อฟอรัมในกลุ่ม/ซูเปอร์กรุ๊ป ดู [เอเจนต์ ACP](/th/tools/acp-agents)

    **การสร้าง ACP ที่ผูกกับเธรดจากแชต**: `/acp spawn <agent> --thread here|auto` ผูกหัวข้อปัจจุบันเข้ากับเซสชัน ACP ใหม่; ข้อความติดตามผลจะถูกกำหนดเส้นทางไปยังเซสชันนั้นโดยตรง และ OpenClaw จะปักหมุดการยืนยันการสร้างไว้ในหัวข้อ ต้องใช้ `channels.telegram.threadBindings.spawnSessions` (ค่าเริ่มต้น: `true`)

    บริบทเทมเพลตเปิดเผย `MessageThreadId` และ `IsForum` แชต DM ที่มี `message_thread_id` จะเก็บเมทาดาทาการตอบกลับไว้ แต่จะใช้คีย์เซสชันที่รับรู้เธรดเฉพาะเมื่อ `getMe` ของ Telegram รายงาน `has_topics_enabled: true`
    การเขียนทับ `dm.threadReplies` และ `direct.*.threadReplies` ที่เลิกใช้แล้วถูกนำออกไป; โหมดเธรดของ BotFather เป็นแหล่งข้อมูลจริงเพียงแหล่งเดียว เรียกใช้ `openclaw doctor --fix` เพื่อนำคีย์การกำหนดค่าที่ล้าสมัยออก

  </Accordion>

    ### ข้อความเสียง

    Telegram แยกบันทึกเสียงออกจากไฟล์เสียง ค่าเริ่มต้น: ทำงานแบบไฟล์เสียง; ใส่แท็ก `[[audio_as_voice]]` ในคำตอบของเอเจนต์เพื่อบังคับให้ส่งเป็นบันทึกเสียง ข้อความถอดเสียงจากบันทึกเสียงขาเข้าจะถูกจัดกรอบเป็นข้อความที่เครื่องสร้างขึ้นและไม่น่าเชื่อถือในบริบทของเอเจนต์ แต่การตรวจจับการกล่าวถึงยังคงใช้ข้อความถอดเสียงดิบ เพื่อให้ข้อความเสียงที่กำหนดให้ต้องกล่าวถึงยังคงทำงานได้

```json5
{
  action: "send",
  channel: "telegram",
  to: "123456789",
  media: "https://example.com/voice.ogg",
  asVoice: true,
}
```

    ### ข้อความวิดีโอ

    Telegram แยกไฟล์วิดีโอออกจากบันทึกวิดีโอ บันทึกวิดีโอไม่รองรับคำบรรยาย; ข้อความที่ระบุจะถูกส่งแยกต่างหาก

```json5
{
  action: "send",
  channel: "telegram",
  to: "123456789",
  media: "https://example.com/video.mp4",
  asVideoNote: true,
}
```

    ### ตำแหน่งที่ตั้งและสถานที่

    ใช้การดำเนินการ `send` ที่มีอยู่กับออบเจ็กต์ `location` แบบเดี่ยวหนึ่งรายการ พิกัดจะส่งหมุดแบบเนทีฟ; การเพิ่มทั้ง `name` และ `address` จะส่งการ์ดสถานที่แบบเนทีฟ การส่งตำแหน่งที่ตั้งไม่สามารถใช้ร่วมกับข้อความหรือสื่อได้

```json5
{
  action: "send",
  channel: "telegram",
  to: "123456789",
  location: {
    latitude: 48.858844,
    longitude: 2.294351,
    accuracy: 12,
```

    รีแอ็กชันของ Telegram เข้ามาเป็นการอัปเดต `message_reaction` ซึ่งแยกจากเพย์โหลดข้อความ เมื่อเปิดใช้งาน OpenClaw จะจัดคิวเหตุการณ์ระบบ เช่น `Telegram reaction added: 👍 by Alice (@alice) on msg 42`

    - `channels.telegram.reactionNotifications`: `off | own | all` (ค่าเริ่มต้น: `own`)
    - `channels.telegram.reactionLevel`: `off | ack | minimal | extensive` (ค่าเริ่มต้น: `minimal`)

    `own` หมายถึงเฉพาะรีแอ็กชันของผู้ใช้ต่อข้อความที่บอตส่งเท่านั้น (พยายามให้ดีที่สุดผ่านแคชข้อความที่ส่งแล้ว) เหตุการณ์รีแอ็กชันยังคงปฏิบัติตามการควบคุมการเข้าถึงของ Telegram (`dmPolicy`, `allowFrom`, `groupPolicy`, `groupAllowFrom`); ผู้ส่งที่ไม่ได้รับอนุญาตจะถูกละทิ้ง

    Telegram ไม่ได้ให้ ID เธรดในการอัปเดตรีแอ็กชัน: กลุ่มที่ไม่ใช่ฟอรัมจะกำหนดเส้นทางไปยังเซสชันแชตของกลุ่ม; กลุ่มแบบฟอรัมจะกำหนดเส้นทางไปยังเซสชันหัวข้อทั่วไป (`:topic:1`) ไม่ใช่หัวข้อต้นทางที่ตรงกัน

    `allowed_updates` สำหรับการโพล/Webhook จะรวม `message_reaction` โดยอัตโนมัติ

    ค่าเริ่มต้นคือการโพลระยะยาว สำหรับโหมด Webhook ให้ตั้งค่า `channels.telegram.webhookUrl` และ `channels.telegram.webhookSecret`; `webhookPath` ซึ่งระบุหรือไม่ก็ได้ (ค่าเริ่มต้น `/telegram-webhook`), `webhookHost` (ค่าเริ่มต้น `127.0.0.1`), `webhookPort` (ค่าเริ่มต้น `8787`), `webhookCertPath` (PEM ของใบรับรองที่ลงนามด้วยตนเองสำหรับการตั้งค่าแบบใช้ IP โดยตรงหรือไม่มีโดเมน)

    ในโหมดการโพลระยะยาว OpenClaw จะบันทึกวอเตอร์มาร์กการรีสตาร์ตอย่างถาวรหลังจากส่งต่อการอัปเดตสำเร็จเท่านั้น; ตัวจัดการที่ล้มเหลวจะปล่อยให้การอัปเดตนั้นลองใหม่ได้ในโปรเซสเดิม แทนที่จะทำเครื่องหมายว่าเสร็จสมบูรณ์

    ลิสเนอร์ภายในเครื่องจะผูกกับ `127.0.0.1:8787` โดยค่าเริ่มต้น สำหรับทางเข้าจากสาธารณะ ให้ตั้งรีเวิร์สพร็อกซีไว้หน้าพอร์ตภายในเครื่อง หรือตั้งค่า `webhookHost: "0.0.0.0"` โดยตั้งใจ

    โหมด Webhook จะตรวจสอบตัวป้องกันคำขอ โทเค็นลับของ Telegram และเนื้อหา JSON จากนั้นคอมมิตการอัปเดตไปยังคิวทางเข้าที่คงทนก่อนส่งคืน `200` ว่าง การรับเข้าอย่างคงทนที่สำเร็จจะมี `x-openclaw-delivery-accepted: durable`; การตอบกลับเกี่ยวกับสถานะระบบ การกำหนดเส้นทาง การยืนยันตัวตน การตรวจสอบความถูกต้อง และข้อผิดพลาดด้านพื้นที่จัดเก็บจะละเว้นส่วนหัวนี้ รีเวิร์สพร็อกซีและตัวควบคุมโฮสต์สามารถกำหนดให้ต้องมีส่วนหัวนี้ เพื่อแยกการรับเข้าของ OpenClaw ออกจาก `200` ว่างทั่วไป โดยไม่ต้องอนุมานการยอมรับจากเวลาตอบสนอง

    หลังจากการเขียนแบบคงทน OpenClaw จะรับสิทธิ์และประมวลผลการอัปเดตผ่านการระบายข้อมูลขาเข้าของช่องทางหลัก (เลนรายแชต/รายหัวข้อ, เสร็จสมบูรณ์เมื่อรับช่วงการทำงาน, หมดเวลาค้างก่อนการรับช่วง) รอบการทำงานของเอเจนต์ที่ช้าจะไม่ค้าง ACK การนำส่งของ Telegram

    - `channels.telegram.pollingStallThresholdMs` มีค่าเริ่มต้นเป็น 120000; ปรับค่าให้อยู่ระหว่าง 30000 ถึง 600000 เฉพาะเมื่อมีการรีสตาร์ตจากการตรวจพบการโพลค้างที่เป็นผลบวกลวง

    - บริบทเสริมจากการตอบกลับ/อ้างอิง/ส่งต่อจะถูกทำให้เป็นมาตรฐานภายในหน้าต่างบริบทการสนทนาที่เลือกเพียงหนึ่งหน้าต่าง เมื่อ Gateway เคยพบข้อความต้นทางแล้ว; แคชข้อความที่พบจะอยู่ในสถานะ Plugin ของ OpenClaw SQLite และ `openclaw doctor --fix` จะนำเข้าไฟล์ข้างเคียงแบบเดิม Telegram มี `reply_to_message` แบบตื้นเพียงหนึ่งรายการต่อการอัปเดต ดังนั้นสายข้อความที่เก่ากว่าแคชจึงถูกจำกัดไว้ตามเพย์โหลดนั้น
    - รายการอนุญาตของ Telegram ใช้ควบคุมเป็นหลักว่าใครสามารถเรียกใช้เอเจนต์ได้ ไม่ใช่ขอบเขตการปกปิดข้อมูลบริบทเสริมอย่างสมบูรณ์
    - ประวัติ DM: `channels.telegram.dmHistoryLimit`, `channels.telegram.dms["<user_id>"].historyLimit`
    - `channels.telegram.retry` ใช้กับตัวช่วยส่งของ Telegram (CLI/เครื่องมือ/การดำเนินการ) สำหรับข้อผิดพลาด API ขาออกที่กู้คืนได้ การส่งคำตอบสุดท้ายขาเข้าจะใช้การลองส่งอย่างปลอดภัยซ้ำโดยมีขอบเขตสำหรับความล้มเหลวก่อนเชื่อมต่อ แต่จะไม่ลองซ้ำสำหรับเอนเวโลปเครือข่ายหลังส่งที่มีความกำกวม ซึ่งอาจทำให้เกิดข้อความที่ผู้ใช้มองเห็นซ้ำกัน

    เป้าหมายการส่งของ CLI และเครื่องมือข้อความรองรับ ID แชตแบบตัวเลข ชื่อผู้ใช้ หรือเป้าหมายหัวข้อฟอรัม:

```bash
openclaw message send --channel telegram --target 123456789 --message "hi"
openclaw message send --channel telegram --target @name --message "hi"
openclaw message send --channel telegram --target -1001234567890:topic:42 --message "hi topic"
```

    แบบสำรวจใช้ `openclaw message poll` และรองรับหัวข้อฟอรัม:

```bash
openclaw message poll --channel telegram --target 123456789 \
  --poll-question "Ship it?" --poll-option "Yes" --poll-option "No"
openclaw message poll --channel telegram --target -1001234567890:topic:42 \
  --poll-question "Pick a time" --poll-option "10am" --poll-option "2pm" \
  --poll-duration-seconds 300 --poll-public
```

    แฟล็กแบบสำรวจเฉพาะ Telegram: `--poll-duration-seconds` (5-600), `--poll-anonymous`, `--poll-public`, `--thread-id` (หรือเป้าหมาย `:topic:`) `--poll-option` ทำซ้ำ 2-12 ครั้ง (ขีดจำกัดตัวเลือกของ Telegram)

    การส่งผ่าน Telegram ยังรองรับ `--presentation` พร้อมบล็อก `buttons` สำหรับแป้นพิมพ์แบบอินไลน์ (เมื่อ `channels.telegram.capabilities.inlineButtons` อนุญาต), `--pin` หรือ `--delivery '{"pin":true}'` เพื่อขอให้ส่งแบบปักหมุดเมื่อบอตสามารถปักหมุดในแชตนั้นได้ และ `--force-document` เพื่อส่งรูปภาพ GIF และวิดีโอขาออกเป็นเอกสารแทนการอัปโหลดแบบบีบอัด/ภาพเคลื่อนไหว/วิดีโอ

    การควบคุมการดำเนินการ: `channels.telegram.actions.sendMessage=false` ปิดใช้งานข้อความขาออกทั้งหมดรวมถึงแบบสำรวจ; `channels.telegram.actions.poll=false` ปิดใช้งานการสร้างแบบสำรวจโดยยังคงเปิดใช้งานการส่งตามปกติ

  <Accordion title="การอนุมัติ Exec ใน Telegram">
    Telegram รองรับการอนุมัติ Exec ใน DM ของผู้อนุมัติ และสามารถเลือกโพสต์คำขอในแชตหรือหัวข้อต้นทางได้ ผู้อนุมัติต้องเป็น ID ผู้ใช้ Telegram แบบตัวเลข

    - `channels.telegram.execApprovals.enabled` (`"auto"` เปิดใช้งานเมื่อสามารถระบุผู้อนุมัติได้อย่างน้อยหนึ่งราย)
    - `channels.telegram.execApprovals.approvers` (ใช้ ID เจ้าของแบบตัวเลขจาก `commands.ownerAllowFrom` เป็นค่าทดแทน)
    - `channels.telegram.execApprovals.target`: `dm` (ค่าเริ่มต้น) | `channel` | `both`
    - `agentFilter`, `sessionFilter`

    `channels.telegram.allowFrom`, `groupAllowFrom` และ `defaultTo` ควบคุมว่าใครสามารถสื่อสารกับบอตได้และบอตจะส่งคำตอบปกติไปที่ใด — ค่าเหล่านี้ไม่ได้ทำให้บุคคลใดเป็นผู้อนุมัติ Exec การจับคู่ DM ที่ได้รับอนุมัติครั้งแรกจะเริ่มต้น `commands.ownerAllowFrom` เมื่อยังไม่มีเจ้าของคำสั่ง ทำให้การตั้งค่าที่มีเจ้าของหนึ่งรายทำงานได้โดยไม่ต้องทำ ID ซ้ำภายใต้ `execApprovals.approvers`

    การส่งไปยังช่องทางจะแสดงข้อความคำสั่งในแชต; เปิดใช้ `channel` หรือ `both` เฉพาะในกลุ่ม/หัวข้อที่เชื่อถือได้ เมื่อคำขอปรากฏในหัวข้อฟอรัม OpenClaw จะคงหัวข้อนั้นไว้สำหรับคำขออนุมัติและข้อความติดตามผล การอนุมัติ Exec จะหมดอายุหลังจาก 30 นาทีโดยค่าเริ่มต้น

    ปุ่มอนุมัติแบบอินไลน์ยังกำหนดให้ `channels.telegram.capabilities.inlineButtons` อนุญาตพื้นผิวเป้าหมาย (`dm`, `group` หรือ `all`) ID การอนุมัติที่ขึ้นต้นด้วย `plugin:` จะถูกแก้ไขผ่านการอนุมัติของ Plugin; รายการอื่นจะถูกแก้ไขผ่านการอนุมัติ Exec ก่อน

    ดู [การอนุมัติ Exec](/th/tools/exec-approvals)

  </Accordion>
</AccordionGroup>

## การควบคุมการตอบกลับข้อผิดพลาด

เมื่อเอเจนต์พบข้อผิดพลาดในการส่งหรือข้อผิดพลาดจากผู้ให้บริการ นโยบายข้อผิดพลาดจะควบคุมว่าข้อความข้อผิดพลาดจะไปถึงแชต Telegram หรือไม่:

| คีย์                                 | ค่า                     | ค่าเริ่มต้น         | คำอธิบาย                                                                                                                                                                                              |
| ----------------------------------- | -------------------------- | --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `channels.telegram.errorPolicy`     | `always`, `once`, `silent` | `always`        | `always` ส่งข้อความข้อผิดพลาดทุกรายการไปยังแชต `once` ส่งข้อความข้อผิดพลาดที่ไม่ซ้ำแต่ละรายการหนึ่งครั้งต่อช่วงคูลดาวน์ (ระงับข้อผิดพลาดเหมือนกันที่เกิดซ้ำ) `silent` ไม่ส่งข้อความข้อผิดพลาดไปยังแชต |
| `channels.telegram.errorCooldownMs` | ตัวเลข (มิลลิวินาที)                | `14400000` (4 ชม.) | ช่วงคูลดาวน์สำหรับนโยบาย `once` หลังจากส่งข้อผิดพลาดแล้ว ข้อความเดียวกันจะถูกระงับจนกว่าช่วงเวลานี้จะผ่านไป ป้องกันข้อความข้อผิดพลาดจำนวนมากระหว่างระบบขัดข้อง                                           |

รองรับการเขียนทับค่าแยกตามบัญชี กลุ่ม และหัวข้อ (ใช้การสืบทอดแบบเดียวกับคีย์การกำหนดค่า Telegram อื่น ๆ)

```json5
{
  channels: {
    telegram: {
      errorPolicy: "always",
      errorCooldownMs: 120000,
      groups: {
        "-1001234567890": {
          errorPolicy: "silent", // ระงับข้อผิดพลาดในกลุ่มนี้
        },
      },
    },
  },
}
```

<AccordionGroup>
  <Accordion title="บอตไม่ตอบสนองต่อข้อความกลุ่มที่ไม่ได้กล่าวถึง">

    - หาก `requireMention=false` โหมดความเป็นส่วนตัวของ Telegram ต้องอนุญาตให้มองเห็นได้ทั้งหมด: BotFather `/setprivacy` -> Disable จากนั้นนำบอตออกแล้วเพิ่มกลับเข้าไปในกลุ่ม
    - `openclaw channels status` จะแจ้งเตือนเมื่อการกำหนดค่าคาดว่าจะรับข้อความกลุ่มที่ไม่ได้กล่าวถึง
    - `openclaw channels status --probe` ตรวจสอบ ID กลุ่มแบบตัวเลขที่ระบุชัดเจน; ไม่สามารถตรวจสอบการเป็นสมาชิกสำหรับไวลด์การ์ด `"*"` ได้
    - การทดสอบเซสชันอย่างรวดเร็ว: `/activation always`

  </Accordion>

  <Accordion title="บอตไม่เห็นข้อความกลุ่มเลย">

    - เมื่อมี `channels.telegram.groups` ต้องระบุกลุ่มไว้ในรายการ (หรือรวม `"*"`)
    - ตรวจสอบว่าบอตเป็นสมาชิกของกลุ่ม
    - ตรวจสอบ `openclaw logs --follow` เพื่อดูเหตุผลที่ข้าม

  </Accordion>

  <Accordion title="คำสั่งทำงานบางส่วนหรือไม่ทำงานเลย">

    - อนุญาตข้อมูลประจำตัวผู้ส่งของคุณ (การจับคู่และ/หรือ `allowFrom` แบบตัวเลข); การอนุญาตคำสั่งยังคงมีผลแม้ว่านโยบายกลุ่มจะเป็น `open`
    - `setMyCommands failed` พร้อม `BOT_COMMANDS_TOO_MUCH` หมายความว่าเมนูเนทีฟมีรายการมากเกินไป; ลดคำสั่ง Plugin/Skills/แบบกำหนดเอง หรือปิดใช้งานเมนูเนทีฟ
    - การเรียกเริ่มต้น `deleteMyCommands` / `setMyCommands` และการเรียกการพิมพ์ `sendChatAction` มีขอบเขตจำกัดและจะลองใหม่หนึ่งครั้งผ่านกลไกสำรองการขนส่งของ Telegram เมื่อคำขอหมดเวลา ข้อผิดพลาดเครือข่าย/fetch ที่เกิดอย่างต่อเนื่องมักหมายความว่าไม่สามารถเข้าถึง `api.telegram.org` ผ่าน DNS/HTTPS ได้

  </Accordion>

  <Accordion title="การเริ่มต้นรายงานว่าโทเค็นไม่ได้รับอนุญาต">

    - `getMe returned 401` คือความล้มเหลวในการตรวจสอบสิทธิ์ Telegram สำหรับโทเค็นบอตที่กำหนดค่าไว้ คัดลอกหรือสร้างโทเค็นใหม่ใน BotFather จากนั้นอัปเดต `channels.telegram.botToken`, `tokenFile`, `accounts.<id>.botToken` หรือ `TELEGRAM_BOT_TOKEN` (บัญชีเริ่มต้น)
    - `deleteWebhook 401 Unauthorized` ระหว่างการเริ่มต้นก็เป็นความล้มเหลวในการตรวจสอบสิทธิ์เช่นกัน; การตีความว่าเป็น "ไม่มี Webhook" จะเพียงเลื่อนความล้มเหลวจากโทเค็นที่ไม่ถูกต้องเดียวกันไปยังการเรียก API ในภายหลัง

  </Accordion>

  <Accordion title="ความไม่เสถียรของการโพลหรือเครือข่าย">

    - Node 22+ ที่ใช้ fetch/พร็อกซีแบบกำหนดเองอาจทำให้เกิดพฤติกรรมยกเลิกทันที หากประเภท `AbortSignal` ไม่ตรงกัน
    - โฮสต์บางแห่งแปลง `api.telegram.org` เป็น IPv6 ก่อน; การเชื่อมต่อขาออก IPv6 ที่ขัดข้องทำให้เกิดความล้มเหลวของ API เป็นระยะ
    - บันทึกที่มี `TypeError: fetch failed` หรือ `Network request for 'getUpdates' failed!` จะถูกลองใหม่ในฐานะข้อผิดพลาดเครือข่ายที่กู้คืนได้
    - ระหว่างการเริ่มต้นการโพล OpenClaw จะนำโพรบ `getMe` ที่สำเร็จตอนเริ่มต้นมาใช้ซ้ำสำหรับ grammY เพื่อให้ตัวรันไม่ต้องเรียก `getMe` ครั้งที่สองก่อน `getUpdates` ครั้งแรก
    - หาก `deleteWebhook` ล้มเหลวด้วยข้อผิดพลาดเครือข่ายชั่วคราวระหว่างการเริ่มต้นการโพล OpenClaw จะดำเนินการต่อไปยัง long polling แทนที่จะเรียก control plane ก่อนโพลอีกครั้ง จากนั้น Webhook ที่ยังทำงานอยู่จะปรากฏเป็นข้อขัดแย้ง `getUpdates`; OpenClaw จะสร้างการขนส่งใหม่และลองล้าง Webhook อีกครั้ง
    - หากซ็อกเก็ต Telegram ถูกนำกลับมาใช้ใหม่ตามรอบคงที่สั้น ๆ ให้ตรวจสอบว่า `channels.telegram.timeoutSeconds` ต่ำเกินไปหรือไม่ — ไคลเอนต์บอตจะปรับค่าที่กำหนดซึ่งต่ำกว่าตัวป้องกันคำขอขาออกและ `getUpdates` แต่รุ่นเก่าอาจยกเลิกทุกการโพลหรือคำตอบเมื่อกำหนดค่านี้ต่ำกว่าตัวป้องกันเหล่านั้น
    - `Polling stall detected` ในบันทึกหมายความว่า OpenClaw รีสตาร์ตการโพลและสร้างการขนส่งใหม่ หลังไม่มี long-poll ที่เสร็จสมบูรณ์เพื่อยืนยันการทำงานเป็นเวลา 120 วินาทีโดยค่าเริ่มต้น
    - `openclaw channels status --probe` และ `openclaw doctor` จะแจ้งเตือนเมื่อบัญชีการโพลที่กำลังทำงานยังดำเนินการ `getUpdates` ไม่เสร็จหลังช่วงผ่อนผันการเริ่มต้น, บัญชี Webhook ที่กำลังทำงานยังดำเนินการ `setWebhook` ไม่เสร็จหลังช่วงผ่อนผันการเริ่มต้น หรือกิจกรรมการขนส่งการโพลที่สำเร็จครั้งล่าสุดเก่าเกินไป
    - เพิ่ม `channels.telegram.pollingStallThresholdMs` เฉพาะเมื่อการเรียก `getUpdates` ที่ทำงานเป็นเวลานานยังเป็นปกติ แต่โฮสต์ยังรายงานการรีสตาร์ตจากการโพลค้างที่เป็นผลบวกลวง การค้างอย่างต่อเนื่องมักบ่งชี้ถึงปัญหาพร็อกซี DNS, IPv6 หรือการเชื่อมต่อ TLS ขาออกไปยัง `api.telegram.org`
    - Telegram ใช้ตัวแปรสภาพแวดล้อมพร็อกซีของโปรเซสสำหรับการขนส่ง Bot API: `HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY` และรูปแบบตัวพิมพ์เล็ก `NO_PROXY` / `no_proxy` ยังสามารถข้าม `api.telegram.org` ได้
    - หากกำหนด `OPENCLAW_PROXY_URL` สำหรับสภาพแวดล้อมบริการและไม่มีตัวแปรสภาพแวดล้อมพร็อกซีมาตรฐาน Telegram จะใช้ URL นั้นสำหรับการขนส่ง Bot API ด้วย
    - บนโฮสต์ VPS ที่การเชื่อมต่อขาออกโดยตรง/TLS ไม่เสถียร ให้กำหนดเส้นทางการเรียก Telegram API ผ่านพร็อกซี:

```yaml
channels:
  telegram:
    proxy: socks5://<user>:<password>@proxy-host:1080
```

    - Node 22+ ใช้ค่าเริ่มต้นเป็น `autoSelectFamily=true` (ยกเว้น WSL2) ลำดับผลลัพธ์ DNS ของ Telegram จะใช้ `OPENCLAW_TELEGRAM_DNS_RESULT_ORDER` ก่อน ตามด้วย `channels.telegram.network.dnsResultOrder` แล้วจึงใช้ค่าเริ่มต้นของโปรเซส (เช่น `NODE_OPTIONS=--dns-result-order=ipv4first`) และจะย้อนกลับไปใช้ `ipv4first` บน Node 22+ หากไม่มีค่าใดใช้ได้
    - บน WSL2 หรือเมื่อการทำงานแบบ IPv4 เท่านั้นให้ผลดีกว่า ให้บังคับการเลือกตระกูลแอดเดรส:

```yaml
channels:
  telegram:
    network:
      autoSelectFamily: false
```

    - คำตอบในช่วงสำหรับการทดสอบประสิทธิภาพตาม RFC 2544 (`198.18.0.0/15`) ได้รับอนุญาตสำหรับการดาวน์โหลดสื่อของ Telegram ตามค่าเริ่มต้นอยู่แล้ว หากพร็อกซี fake-IP หรือพร็อกซีโปร่งใสที่เชื่อถือได้เขียน `api.telegram.org` ใหม่เป็นแอดเดรสส่วนตัว/ภายใน/สำหรับการใช้งานพิเศษอื่นระหว่างดาวน์โหลดสื่อ ให้เลือกใช้การข้ามการป้องกันสำหรับ Telegram เท่านั้น:

```yaml
channels:
  telegram:
    network:
      dangerouslyAllowPrivateNetwork: true
```

    - สามารถเลือกใช้แบบเดียวกันแยกตามบัญชีได้ที่ `channels.telegram.accounts.<accountId>.network.dangerouslyAllowPrivateNetwork`
    - หากพร็อกซีของคุณแก้ชื่อโฮสต์สื่อของ Telegram เป็น `198.18.x.x` ให้ปิดแฟล็กอันตรายไว้ก่อน เนื่องจากช่วงดังกล่าวได้รับอนุญาตตามค่าเริ่มต้นอยู่แล้ว

    <Warning>
      `channels.telegram.network.dangerouslyAllowPrivateNetwork` ลดความเข้มงวดของการป้องกัน SSRF สำหรับสื่อของ Telegram ใช้เฉพาะในสภาพแวดล้อมพร็อกซีที่เชื่อถือได้และควบคุมโดยผู้ดำเนินการ (การกำหนดเส้นทาง fake-IP ของ Clash, Mihomo, Surge) ซึ่งสร้างคำตอบที่เป็นแอดเดรสส่วนตัวหรือสำหรับการใช้งานพิเศษนอกช่วงทดสอบประสิทธิภาพตาม RFC 2544 เท่านั้น ปิดไว้สำหรับการเข้าถึง Telegram ผ่านอินเทอร์เน็ตสาธารณะตามปกติ
    </Warning>

    - การแทนค่าชั่วคราวผ่านตัวแปรสภาพแวดล้อม: `OPENCLAW_TELEGRAM_DISABLE_AUTO_SELECT_FAMILY=1`, `OPENCLAW_TELEGRAM_ENABLE_AUTO_SELECT_FAMILY=1`, `OPENCLAW_TELEGRAM_DNS_RESULT_ORDER=ipv4first`
    - ตรวจสอบคำตอบ DNS:

```bash
dig +short api.telegram.org A
dig +short api.telegram.org AAAA
```

  </Accordion>
</AccordionGroup>

- Webhook: `webhookUrl`, `webhookSecret`, `webhookPath`, `webhookHost`, `webhookPort`, `webhookCertPath`
- การดำเนินการ/ความสามารถ: `capabilities.inlineButtons`, `actions.sendMessage|editMessage|deleteMessage|reactions|sticker|createForumTopic|editForumTopic`
- รีแอ็กชัน: `reactionNotifications`, `reactionLevel`
- ข้อผิดพลาด: `errorPolicy`, `errorCooldownMs`, `silentErrorReplies`
- การเขียน/ประวัติ: `configWrites`, `historyLimit`, `dmHistoryLimit`, `dms.*.historyLimit`

</Accordion>

<Note>
ลำดับความสำคัญสำหรับหลายบัญชี: เมื่อกำหนดค่า ID บัญชีตั้งแต่สองรายการขึ้นไป ให้ตั้งค่า `channels.telegram.defaultAccount` (หรือรวม `channels.telegram.accounts.default`) เพื่อระบุการกำหนดเส้นทางเริ่มต้นอย่างชัดเจน มิฉะนั้น OpenClaw จะย้อนกลับไปใช้ ID บัญชีแรกหลังการปรับให้อยู่ในรูปแบบมาตรฐาน และ `openclaw doctor` จะแสดงคำเตือน บัญชีที่มีชื่อจะสืบทอด `channels.telegram.allowFrom` / `groupAllowFrom` แต่จะไม่สืบทอดค่าของ `accounts.default.*`
</Note>
