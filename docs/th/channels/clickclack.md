ClickClack เชื่อมต่อ OpenClaw กับเวิร์กสเปซ ClickClack ที่โฮสต์เองผ่านโทเค็นบอต ClickClack ที่รองรับโดยตรง

ใช้วิธีนี้เมื่อต้องการให้เอเจนต์ OpenClaw ปรากฏเป็นผู้ใช้บอต ClickClack โดย ClickClack รองรับทั้งบอตบริการอิสระและบอตที่ผู้ใช้เป็นเจ้าของ บอตที่ผู้ใช้เป็นเจ้าของจะคง `owner_user_id` ไว้และได้รับเฉพาะขอบเขตโทเค็นที่คุณอนุญาต

## การตั้งค่าอย่างรวดเร็ว

ใน ClickClack ให้เปิด **Workspace settings → Integrations → OpenClaw** สร้าง
บอต แล้วคัดลอกโทเค็น จากนั้นกำหนดค่าช่องทาง:

```bash
openclaw channels add clickclack --base-url https://clickclack.example.com --token ccb_... --workspace default
```

`workspace` ยอมรับ ID เวิร์กสเปซ (`wsp_...`), slug หรือชื่อที่แสดง
`channels add` จะตรวจสอบเซิร์ฟเวอร์ โทเค็น และเวิร์กสเปซหลังบันทึก จากนั้น
รายงานว่า Gateway ที่กำลังทำงานตรวจพบบัญชีใหม่หรือไม่ หาก OpenClaw
กำลังทำงานอยู่แล้ว ClickClack จะเชื่อมต่อโดยอัตโนมัติและไม่จำเป็นต้องใช้
คำสั่งที่สอง มิฉะนั้นให้เริ่มด้วย:

```bash
openclaw gateway
```

สำหรับการตั้งค่าแบบมีคำแนะนำ ให้เรียกใช้:

```bash
openclaw onboard
```

เลือก ClickClack จากนั้นป้อน URL เซิร์ฟเวอร์ โทเค็นบอต และเวิร์กสเปซเมื่อ
ระบบแจ้ง การตั้งค่าแบบมีคำแนะนำจะตรวจสอบเซิร์ฟเวอร์ โทเค็น และเวิร์กสเปซหลังบันทึก
หากตรวจสอบไม่ผ่าน ระบบจะไม่ทิ้งการกำหนดค่า

### ทางเลือก: โทเค็นจากตัวแปรสภาพแวดล้อม

บัญชีเริ่มต้นสามารถอ่าน `CLICKCLACK_BOT_TOKEN` แทนการจัดเก็บโทเค็น
ในการกำหนดค่า:

```bash
export CLICKCLACK_BOT_TOKEN="ccb_..."
openclaw channels add clickclack --base-url https://clickclack.example.com --workspace default --use-env
openclaw gateway
```

บัญชีที่มีชื่อต้องใช้โทเค็นที่กำหนดค่าหรือไฟล์โทเค็น ตัวแปรสภาพแวดล้อม
ที่ใช้ร่วมกันถูกจำกัดไว้สำหรับบัญชีเริ่มต้นโดยเจตนา

### ข้อมูลอ้างอิง JSON5

รูปแบบการกำหนดค่าที่เทียบเท่าคือ:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      defaultTo: "channel:general",
    },
  },
}
```

บัญชีจะถือว่าได้รับการกำหนดค่าแล้วก็ต่อเมื่อกำหนด `baseUrl` แหล่งโทเค็น และ
`workspace` ครบทั้งหมด แหล่งโทเค็นอาจเป็น `token`, `tokenFile` หรือ
`CLICKCLACK_BOT_TOKEN` สำหรับบัญชีเริ่มต้น `workspace` ยอมรับ ID เวิร์กสเปซ
(`wsp_...`), slug หรือชื่อ โดย Gateway จะแปลงเป็น ID เมื่อเริ่มต้น

### คีย์การกำหนดค่าบัญชี

| คีย์                     | ค่าเริ่มต้น             | หมายเหตุ                                                                                   |
| ----------------------- | ------------------- | --------------------------------------------------------------------------------------- |
| `baseUrl`               | ไม่มี (จำเป็น)     | URL เซิร์ฟเวอร์ ClickClack                                                                  |
| `token`                 | ไม่มี                | โทเค็นบอตในรูปแบบสตริงธรรมดาหรือการอ้างอิงข้อมูลลับ (`source: "env" \| "file" \| "exec"`)        |
| `tokenFile`             | ไม่มี                | พาธไปยังไฟล์โทเค็นบอต มีลำดับความสำคัญเหนือ `token`                                |
| `workspace`             | ไม่มี (จำเป็น)     | ID, slug หรือชื่อเวิร์กสเปซ                                                            |
| `replyMode`             | `"agent"`           | `"agent"` เรียกใช้ไปป์ไลน์เอเจนต์ทั้งหมด ส่วน `"model"` ส่งการเติมข้อความจากโมเดลโดยตรงแบบสั้น |
| `defaultTo`             | `"channel:general"` | เป้าหมายที่ใช้เมื่อเส้นทางขาออกไม่ได้ระบุเป้าหมาย                                      |
| `allowFrom`             | `["*"]`             | รายการอนุญาต ID ผู้ใช้สำหรับ DM และข้อความช่องทางขาเข้า                                 |
| `botUserId`             | ตรวจพบโดยอัตโนมัติ       | แปลงจากข้อมูลประจำตัวของโทเค็นบอตเมื่อเริ่มต้น                                        |
| `agentId`               | ค่าเริ่มต้นของเส้นทาง       | ตรึงข้อความขาเข้าของบัญชีนี้ไว้กับเอเจนต์หนึ่งตัว                                       |
| `toolsAllow`            | ไม่มี                | รายการอนุญาตเครื่องมือสำหรับการตอบกลับของเอเจนต์จากบัญชีนี้                                     |
| `model`, `systemPrompt` | ไม่มี                | ใช้โดยการเติมข้อความ `replyMode: "model"`                                               |
| `commandMenu`           | `true`              | เผยแพร่คำสั่งเนทีฟไปยังการเติมข้อความอัตโนมัติของช่องเขียน ClickClack                            |
| `reconnectMs`           | `1500`              | ระยะหน่วงการเชื่อมต่อแบบเรียลไทม์อีกครั้ง (100 ถึง 60000)                                                |

หาก `plugins.allow` เป็นรายการจำกัดที่ไม่ว่าง การเลือก
ClickClack อย่างชัดเจนในการตั้งค่าช่องทางหรือการเรียกใช้ `openclaw plugins enable clickclack`
จะเพิ่ม `clickclack` ต่อท้ายรายการนั้น การติดตั้งระหว่างการเริ่มต้นใช้งานใช้พฤติกรรม
การเลือกอย่างชัดเจนแบบเดียวกัน เส้นทางเหล่านี้จะไม่เขียนทับ `plugins.deny` หรือการตั้งค่า
`plugins.enabled: false` ส่วนกลาง การใช้
`openclaw plugins install @openclaw/clickclack` โดยตรงจะเป็นไปตามนโยบาย
การติดตั้ง Plugin ตามปกติ และบันทึก ClickClack ลงในรายการอนุญาตที่มีอยู่ด้วย

## บอตหลายตัว

แต่ละบัญชีจะเปิดการเชื่อมต่อแบบเรียลไทม์กับ ClickClack ของตนเองและใช้โทเค็นบอตของตนเอง

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      defaultAccount: "service",
      accounts: {
        service: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SERVICE_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "channel:general",
          agentId: "service-bot",
        },
        support: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SUPPORT_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "dm:usr_...",
          agentId: "support-bot",
        },
      },
    },
  },
}
```

## โหมดการตอบกลับ

- `replyMode: "agent"` (ค่าเริ่มต้น) ส่งข้อความขาเข้าผ่านไปป์ไลน์เอเจนต์ตามปกติ รวมถึงการบันทึกเซสชันและนโยบายเครื่องมือ
- `replyMode: "model"` ข้ามไปป์ไลน์เอเจนต์และใช้ `llm.complete` ของรันไทม์ Plugin สำหรับการตอบกลับโดยตรงของบอต โดยอาจปรับรูปแบบด้วย `model` และ `systemPrompt` ผู้ให้บริการและโมเดลที่เลือกเป็นผู้ควบคุมโควตาการเติมข้อความ

โหมดโมเดลเรียกใช้การเติมข้อความกับ ID เอเจนต์บอตที่แปลงแล้ว ซึ่งต้องใช้
บิตความเชื่อถือ `plugins.entries.clickclack.llm.allowAgentIdOverride: true` อย่างชัดเจน:

```json5
{
  plugins: {
    entries: {
      clickclack: {
        llm: {
          allowAgentIdOverride: true,
        },
      },
    },
  },
}
```

ปิดบิตความเชื่อถือไว้หากใช้เฉพาะโหมดตอบกลับ `agent` เริ่มต้น
เนื่องจากโหมดดังกล่าวไม่จำเป็นต้องใช้บิตนี้

## เมนูคำสั่ง

เมื่อ Gateway เริ่มต้น แต่ละบัญชีที่กำหนดค่าจะเผยแพร่คำสั่งเนทีฟ
ของ OpenClaw ไปยัง ClickClack คำสั่งเหล่านี้จะปรากฏในการเติมข้อความอัตโนมัติของช่องเขียน
พร้อมป้ายกำกับเป็นชื่อเรียกของบอต ชุดคำสั่งที่เผยแพร่จะถูกแทนที่ทั้งหมดทุกครั้งที่เริ่มต้น
รวมถึงการล้างเมนูเก่าเมื่อแค็ตตาล็อกคำสั่งเนทีฟว่างเปล่า

การซิงค์เมนูคำสั่งเปิดใช้งานโดยค่าเริ่มต้น ตั้งค่า `commandMenu: false` ในบัญชี
เพื่อไม่เข้าร่วม:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      commandMenu: false,
    },
  },
}
```

โทเค็นต้องมี `commands:write` ชุด `bot:write` และ
`bot:admin` ของ ClickClack ปัจจุบันมีขอบเขตดังกล่าวรวมอยู่ และยังสามารถอนุญาต
แยกต่างหากได้ โทเค็นที่สร้างก่อนมีการเปิดตัวเมนูคำสั่งอาจต้องเพิ่ม
ขอบเขตหรือเปลี่ยนโทเค็นใหม่

การซิงค์ทำแบบพยายามให้ดีที่สุดและทำงานหนึ่งครั้งต่อการเริ่ม Gateway แต่ละครั้ง หากขาดขอบเขตหรือเกิด
ความล้มเหลวของเครือข่าย ระบบจะบันทึกคำเตือน ส่วนเซิร์ฟเวอร์ ClickClack รุ่นเก่าที่ไม่มีเอนด์พอยต์
จะบันทึกที่ระดับดีบัก ความล้มเหลวเหล่านี้ไม่ขัดขวางการเริ่มต้นแบบเรียลไทม์ เมนูยังคง
ใช้งานได้ขณะที่เอเจนต์ออฟไลน์ และจะถูกลบเมื่อบอตออกจาก
เวิร์กสเปซ

รีลีสนี้เผยแพร่เฉพาะข้อกำหนดคำสั่งเนทีฟเท่านั้น ระบบจะไม่เพิ่มชื่อเรียกอื่นและ
แค็ตตาล็อกคำสั่งของ Skills, Plugin หรือคำสั่งกำหนดเองลงในเมนู หาก
ชื่อหนึ่งได้รับการลงทะเบียนเป็นคำสั่งเครื่องหมายทับ HTTP ด้วย ClickClack จะส่งต่อ
การลงทะเบียนนั้นก่อน ส่วนคำสั่งเมนูอื่นจะดำเนินต่อผ่านการส่งข้อความ
ตามปกติ

ใช้โหมด `agent` สำหรับหลักฐานสหสัมพันธ์ข้ามบริการ สำหรับ
ID ข้อความ ClickClack ที่เชื่อถือได้ในรูปแบบมาตรฐาน `msg_<ulid>` ช่องทางจะสร้าง
ID การเรียกใช้ OpenClaw แบบกำหนดแน่นอน `clickclack:<message-id>` จากนั้นการเรียกโมเดลแต่ละครั้ง
จะแสดงในการวินิจฉัยเป็น `clickclack:<message-id>:model:<n>` เมื่อ
รอบนั้นใช้ ClawRouter ระบบจะส่ง ID การเรียกโมเดลเดียวกันเป็น `X-Request-ID`
โหมด `model` ข้ามการวินิจฉัยการเรียกใช้เอเจนต์/เซสชันตามปกติ จึง
ไม่เหมาะสำหรับเส้นทางหลักฐานนี้

เมื่อเหตุการณ์แบบเรียลไทม์มี `payload.correlation_id` ที่ผ่านการตรวจสอบ
ช่องทางจะส่งต่อค่านั้นเป็น `X-Correlation-ID` ในการดึงข้อความที่เชื่อถือได้และ
คำขอตอบกลับ ClickClack ที่เกิดขึ้น ค่าใช้ชุดอักขระปลอดภัย
128 อักขระของ ClickClack (`A-Z`, `a-z`, `0-9`, `.`, `_`, `:` และ `-`) โดยจะละเว้นค่าที่ไม่ถูกต้อง
การเชื่อมโยงเหล่านี้มีเฉพาะตัวระบุเท่านั้น ไม่รวมเนื้อหาข้อความ
พรอมต์ การเติมข้อความ ข้อมูลประจำตัว หรือเอาต์พุตเครื่องมือ

## การส่งสื่อแบบคงทน

การตอบกลับของเอเจนต์ที่มีสื่อใช้การส่งแบบคงทนที่จำเป็น OpenClaw กำหนด
นอนซ์ข้อความและการอัปโหลดที่เสถียรสำหรับแต่ละส่วนก่อนเขียนไปยัง ClickClack ครั้งแรก ดังนั้น
การลองใหม่จะใช้การอัปโหลดและข้อความเดิมแทนการใช้โควตาพื้นที่จัดเก็บ
หรือเผยแพร่รายการซ้ำ หากมีการอัปโหลดอยู่แล้วหลังเริ่มระบบใหม่
OpenClaw จะไม่อ่านพาธภายในเครื่องเดิมหรือ URL สื่อระยะไกลซ้ำ

สัญญาการกู้คืนนี้ต้องใช้เซิร์ฟเวอร์ ClickClack ที่รองรับ:

- `GET /api/uploads/by-nonce` พร้อม
  `X-ClickClack-Upload-Nonce: supported` สำหรับผลลัพธ์ทั้งที่พบและไม่พบ
- `GET /api/messages/by-nonce` พร้อม
  `X-ClickClack-Message-Nonce: supported` สำหรับผลลัพธ์ทั้งที่พบและไม่พบ
- การสร้างข้อความและการเชื่อมโยงไฟล์แนบแบบไอดอมโพเทนต์สำหรับ
  นอนซ์และการอัปโหลดเดียวกันที่มีขอบเขตตามเจ้าของ

ระบบจะไม่ถือว่า 404 ทั่วไปจากเซิร์ฟเวอร์รุ่นเก่าเป็นหลักฐานว่าไม่มีการส่ง
OpenClaw จะปล่อยให้สถานะการส่งยังไม่ยุติแทนที่จะเสี่ยงสร้างรายการซ้ำ โปรดอัปเดต
ClickClack ก่อนเปิดใช้งานการตอบกลับของเอเจนต์ที่สร้างสื่อ

## แถวกิจกรรมเอเจนต์

โดยค่าเริ่มต้น ช่องทาง ClickClack จะไม่แสดงสิ่งใดขณะที่รอบการทำงานของเอเจนต์กำลังดำเนินอยู่ โดยจะแสดงเฉพาะคำตอบสุดท้าย ตั้งค่า `agentActivity: true` ในบัญชีเพื่อเผยแพร่แถวข้อความ `agent_commentary` และ `agent_tool` แบบคงทนขณะที่รอบกำลังดำเนินอยู่:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      agentActivity: true,
    },
  },
}
```

ข้อกำหนดและพฤติกรรม:

- **ปิดโดยค่าเริ่มต้น** การตั้งค่ามาตรฐานและเซิร์ฟเวอร์ ClickClack รุ่นเก่าจะไม่ได้รับผลกระทบ
- **ต้องใช้ขอบเขตโทเค็น `agent_activity:write`** ขอบเขตนี้แยกจาก `bot:write` และไม่ได้รับการสืบทอดจากขอบเขตดังกล่าว ให้สร้างโทเค็นบอตด้วย `--scopes bot:write,agent_activity:write` (หรืออนุญาตขอบเขตแก่โทเค็นที่มีอยู่) ก่อนเปิดใช้งานตัวเลือก
- **ลดระดับการทำงานแบบพยายามให้ดีที่สุด** หากโทเค็นไม่มี `agent_activity:write` หรือเซิร์ฟเวอร์ปฏิเสธการเขียนกิจกรรม ระบบจะบันทึกความล้มเหลวและยังคงส่งคำตอบสุดท้ายได้ตามปกติ โดยจะไม่มีแถวกิจกรรมปรากฏ
- แถวจะถูกจัดกลุ่มตามแต่ละรอบ (`turn_id`) และรวมเข้าด้วยกันเพื่อให้หนึ่งขั้นตอนเชิงตรรกะเป็นหนึ่งแถว โดยแถวเครื่องมือใช้รูปแบบความคืบหน้าเดียวกับ Discord/Slack/Telegram (ชื่อเครื่องมือพร้อมรายละเอียดคำสั่ง)
- **ข้อมูลเมตาการระบุแหล่งที่มา** โพสต์ที่สร้างโดยเอเจนต์ (แถวกิจกรรมและคำตอบสุดท้าย) มีฟิลด์ `author_model` และ `author_thinking` ซึ่งแปลงจากโมเดลที่ใช้จริงสำหรับรอบนั้น (รวมถึงหลังการใช้โมเดลสำรอง) เซิร์ฟเวอร์ที่ไม่ได้กำหนดคอลัมน์เหล่านี้จะละเว้นฟิลด์ JSON ที่ไม่รู้จัก ส่วนเซิร์ฟเวอร์ที่จัดเก็บฟิลด์เหล่านี้สามารถตอบได้ว่า "โมเดลใดกล่าวบรรทัดนี้ ที่ระดับการคิดใด" สำหรับแต่ละข้อความ

## เป้าหมาย

- `channel:<name-or-id>` ส่งไปยังช่องของพื้นที่ทำงาน เป้าหมายที่ไม่ระบุประเภทจะใช้ `channel:` เป็นค่าเริ่มต้น
- `dm:<user_id>` สร้างหรือใช้การสนทนาโดยตรงที่มีกับผู้ใช้นั้นอีกครั้ง
- `thread:<message_id>` ตอบกลับในเธรดที่เริ่มต้นจากข้อความนั้น

เป้าหมายขาออกที่ระบุอย่างชัดเจนอาจมีคำนำหน้าผู้ให้บริการ `clickclack:` หรือ `cc:` ด้วย

สื่อขาออกใช้ API อัปโหลดของ ClickClack แล้วแนบไฟล์อัปโหลดแบบคงทน
ไปกับข้อความในช่อง การตอบกลับในเธรด หรือ DM ที่สร้างขึ้น ไฟล์ในเครื่องและ
URL สื่อระยะไกลที่รองรับจะเป็นไปตามนโยบายการเข้าถึงสื่อตามปกติของ OpenClaw โดยจำกัด
ขนาดไว้ที่ 64 MiB ต่อไฟล์ การส่งแบบคงทนที่อยู่ในคิวจะใช้ nonce ซึ่งกำหนดขอบเขตตามเจ้าของแยกกันสำหรับ
การอัปโหลดและแต่ละส่วนของข้อความ จากนั้นลองเชื่อมโยงไฟล์แนบอีกครั้งด้วยออบเจ็กต์
เดิมเหล่านั้น ดูสัญญาฝั่งเซิร์ฟเวอร์และพฤติกรรมการกู้คืนได้ที่ [การส่งมอบสื่อแบบคงทน](#durable-media-delivery)

ตัวอย่าง:

```bash
openclaw message send --channel clickclack --target channel:general --message "hello"
openclaw message send --channel clickclack --target dm:usr_123 --message "hello"
openclaw message send --channel clickclack --target thread:msg_123 --message "following up"
```

## สิทธิ์

ขอบเขตโทเค็น ClickClack บังคับใช้โดย ClickClack API

- `bot:read`: อ่านข้อมูลพื้นที่ทำงาน/ช่อง/ข้อความ/เธรด/DM/เรียลไทม์/โปรไฟล์
- `bot:write`: `bot:read` รวมถึงข้อความในช่อง การตอบกลับในเธรด DM การอัปโหลด และการเผยแพร่เมนูคำสั่ง
- `bot:admin`: `bot:write` รวมถึงการสร้างช่อง
- `commands:write`: เผยแพร่เมนูคำสั่งของบอต รวมอยู่ในชุด `bot:write` และ `bot:admin` ปัจจุบัน และสามารถให้สิทธิ์แยกต่างหากได้
- `agent_activity:write`: แถวกิจกรรมของเอเจนต์แบบคงทน (`agent_commentary` / `agent_tool`) ไม่สืบทอดโดย `bot:write` หรือ `bot:admin`; จำเป็นเฉพาะเมื่อตั้งค่า `agentActivity: true`

OpenClaw ต้องใช้เพียง `bot:write` ปัจจุบันสำหรับการแชตกับเอเจนต์ตามปกติและการซิงค์เมนูคำสั่ง เพิ่ม `agent_activity:write` เมื่อเปิดใช้งาน [แถวกิจกรรมของเอเจนต์](#agent-activity-rows)

## การแก้ไขปัญหา

- `ClickClack is not configured for account "<id>"`: ตั้งค่า `baseUrl`, `token` (ตัวอย่างเช่น ผ่าน `CLICKCLACK_BOT_TOKEN`) และ `workspace` สำหรับบัญชีนั้น
- `ClickClack workspace not found: <value>`: ตั้งค่า `workspace` เป็น ID, slug หรือชื่อของพื้นที่ทำงานที่ ClickClack ส่งคืน
- ไม่มีการตอบกลับขาเข้า: ยืนยันว่าโทเค็นมีสิทธิ์อ่านแบบเรียลไทม์ และโปรดทราบว่าบอตจะไม่สนใจข้อความของตนเองและข้อความจากบอตอื่น
- การส่งไปยังช่องล้มเหลว: ตรวจสอบว่าบอตเป็นสมาชิกของพื้นที่ทำงานและมี `bot:write`
- ไม่มีเมนูคำสั่ง: ยืนยันว่า `commandMenu` ไม่ใช่ `false`, เซิร์ฟเวอร์ ClickClack รองรับ `PUT /api/bots/self/commands` และโทเค็นมี `commands:write`
