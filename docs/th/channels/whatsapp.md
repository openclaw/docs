สถานะ: พร้อมใช้งานจริงผ่าน WhatsApp Web (Baileys) Gateway เป็นผู้ดูแลเซสชันที่เชื่อมโยงไว้ โดยไม่มีช่องทาง Twilio WhatsApp แยกต่างหาก

## การติดตั้ง

`openclaw onboard` และ `openclaw channels add --channel whatsapp` จะแจ้งให้ติดตั้ง Plugin เมื่อเลือกเป็นครั้งแรก ส่วน `openclaw channels login --channel whatsapp` จะเสนอกระบวนการติดตั้งเดียวกันหากไม่มี Plugin เช็กเอาต์สำหรับการพัฒนาจะใช้พาธ Plugin ภายในเครื่อง ส่วนการติดตั้ง stable/beta จะติดตั้ง `@openclaw/whatsapp` จาก ClawHub ก่อน และใช้ npm เป็นทางเลือกสำรอง รันไทม์ WhatsApp จัดส่งแยกจากแพ็กเกจ npm หลักของ OpenClaw ดังนั้นการขึ้นต่อกันของรันไทม์จึงอยู่กับ Plugin ภายนอก การติดตั้งด้วยตนเอง:

```bash
openclaw plugins install clawhub:@openclaw/whatsapp
```

ใช้แพ็กเกจ npm โดยตรง (`@openclaw/whatsapp`) สำหรับการสำรองผ่านรีจิสทรีเท่านั้น และตรึงเวอร์ชันที่แน่นอนเฉพาะเมื่อต้องการให้การติดตั้งทำซ้ำได้เหมือนเดิม

## การตั้งค่าด่วน

<Steps>
  <Step title="กำหนดค่านโยบายการเข้าถึง">

```json5
{
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      allowFrom: ["+15551234567"],
      groupPolicy: "allowlist",
      groupAllowFrom: ["+15551234567"],
    },
  },
}
```

  </Step>

  <Step title="เชื่อมโยง WhatsApp (QR)">

```bash
openclaw channels login --channel whatsapp
```

    การเข้าสู่ระบบทำได้ผ่าน QR เท่านั้น สำหรับโฮสต์ระยะไกลหรือแบบไม่มีหน้าจอ ให้เตรียมช่องทางที่เชื่อถือได้สำหรับส่ง QR ที่ยังใช้งานได้ไปยังโทรศัพท์ก่อนเริ่มเข้าสู่ระบบ เนื่องจาก QR ที่แสดงในเทอร์มินัล ภาพหน้าจอ หรือไฟล์แนบในแชตอาจหมดอายุระหว่างการส่ง

    สำหรับบัญชีที่ระบุ:

```bash
openclaw channels login --channel whatsapp --account work
```

    หากต้องการแนบไดเรกทอรีการยืนยันตัวตนที่มีอยู่หรือกำหนดเองก่อนเข้าสู่ระบบ:

```bash
openclaw channels add --channel whatsapp --account work --auth-dir /path/to/wa-auth
openclaw channels login --channel whatsapp --account work
```

  </Step>

  <Step title="เริ่ม Gateway">

```bash
openclaw gateway
```

  </Step>

  <Step title="อนุมัติคำขอจับคู่ครั้งแรก (โหมดจับคู่)">

```bash
openclaw pairing approve whatsapp <CODE>
```

    คำขอจับคู่จะหมดอายุหลังจาก 1 ชั่วโมง และจำกัดคำขอที่รอดำเนินการไว้ที่ 3 รายการต่อบัญชี

  </Step>

## โมเดลรันไทม์

- Gateway เป็นผู้ดูแลซ็อกเก็ต WhatsApp และลูปการเชื่อมต่อใหม่
- ตัวเฝ้าระวังติดตามสัญญาณสองประเภทแยกจากกัน ได้แก่ กิจกรรมทรานสปอร์ตดิบของ WhatsApp Web และกิจกรรมข้อความของแอปพลิเคชัน เซสชันที่ไม่มีการใช้งานแต่ยังเชื่อมต่ออยู่จะไม่ถูกเริ่มใหม่เพียงเพราะไม่มีข้อความเข้ามาเมื่อไม่นานมานี้ ระบบจะบังคับให้เชื่อมต่อใหม่เฉพาะเมื่อเฟรมทรานสปอร์ตหยุดเข้ามาเกินช่วงเวลาภายในคงที่ (ผู้ใช้กำหนดค่าไม่ได้) หรือไม่มีข้อความของแอปพลิเคชันนานเกิน 4 เท่าของระยะหมดเวลาข้อความปกติ หลังจากเชื่อมต่อเซสชันที่เพิ่งมีการใช้งานใหม่ทันที หน้าต่างเวลาแรกจะใช้ระยะหมดเวลาข้อความปกติที่สั้นกว่าแทนหน้าต่างเวลา 4 เท่า OpenClaw สามารถตอบกลับข้อความออฟไลน์ที่ Baileys ส่งมาในช่วงต้นของการเชื่อมต่อใหม่นั้นโดยอัตโนมัติ ภายใต้ขอบเขตอายุการขจัดข้อความซ้ำตาม ID ข้อความขาเข้า ส่วนการเริ่มต้นระบบครั้งแรกยังคงใช้ตัวป้องกันประวัติเก่าระยะสั้น
- ระยะเวลาของซ็อกเก็ต Baileys ระบุไว้อย่างชัดเจนภายใต้ `web.whatsapp.*`: `keepAliveIntervalMs` (ช่วงเวลาการ ping ของแอปพลิเคชัน), `connectTimeoutMs` (ระยะหมดเวลาของแฮนด์เชกเริ่มต้น), `defaultQueryTimeoutMs` (การรอคิวรีของ Baileys รวมถึงระยะหมดเวลาสำหรับการส่งขาออก/สถานะการแสดงตัวและใบตอบรับการอ่านขาเข้าของ OpenClaw)
- การส่งขาออกต้องมีตัวรับฟัง WhatsApp ที่ทำงานอยู่สำหรับบัญชีเป้าหมาย มิฉะนั้นการส่งจะล้มเหลวทันที
- การส่งไปยังกลุ่มจะแนบข้อมูลเมตาการกล่าวถึงแบบเนทีฟสำหรับโทเค็น `@+<digits>` และ `@<digits>` (ทั้งในข้อความและคำบรรยายสื่อ) เมื่อโทเค็นตรงกับข้อมูลเมตาของผู้เข้าร่วมปัจจุบัน รวมถึงกลุ่มที่ใช้ LID
- ระบบจะเพิกเฉยต่อแชตสถานะและแชตบรอดแคสต์ (`@status`, `@broadcast`)
- แชตโดยตรงใช้กฎเซสชัน DM (`session.dmScope`; ค่าเริ่มต้น `main` จะรวม DM ไว้ในเซสชันหลักของเอเจนต์) เซสชันกลุ่มจะแยกจากกันตาม JID (`agent:<agentId>:whatsapp:group:<jid>`)
- สามารถกำหนด WhatsApp Channels/Newsletters เป็นเป้าหมายขาออกอย่างชัดเจนผ่าน JID แบบเนทีฟ `@newsletter` โดยใช้ข้อมูลเมตาเซสชันช่องทาง (`agent:<agentId>:whatsapp:channel:<jid>`) แทนความหมายแบบ DM
- ทรานสปอร์ต WhatsApp Web ใช้ตัวแปรสภาพแวดล้อมพร็อกซีมาตรฐานบนโฮสต์ Gateway (`HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY` และรูปแบบตัวพิมพ์เล็ก) ควรใช้การกำหนดค่าพร็อกซีระดับโฮสต์แทนการตั้งค่าแยกตามช่องทาง
- เมื่อเปิดใช้ `messages.removeAckAfterReply` OpenClaw จะล้างรีแอ็กชันตอบรับเมื่อส่งคำตอบที่มองเห็นได้สำเร็จแล้ว

## โทรหาผู้ร้องขอปัจจุบันด้วย MeowCaller (ทดลอง)

Plugin สามารถเปิดใช้ `whatsapp_call` ในรอบการทำงานของเอเจนต์ที่เริ่มจาก WhatsApp โดยใช้ [MeowCaller](https://github.com/purpshell/meowcaller) เพื่อโทรด้วยเสียงผ่าน WhatsApp ไปยังผู้ร้องขอที่ได้รับอนุญาตในปัจจุบัน และเล่นข้อความ TTS ของ OpenClaw หลังจากรับสาย เครื่องมือนี้ไม่มีพารามิเตอร์หมายเลขปลายทาง ดังนั้นพรอมต์จึงไม่สามารถเปลี่ยนเส้นทางสายได้ ปิดใช้งานโดยค่าเริ่มต้น

<Warning>
MeowCaller ยังอยู่ในขั้นทดลอง ไม่มีรุ่นที่ติดแท็ก และใช้เซสชันอุปกรณ์ที่เชื่อมโยงผ่าน whatsmeow ซึ่งจับคู่แยกต่างหาก จึงไม่สามารถใช้ข้อมูลประจำตัว Baileys ของ Plugin ซ้ำได้ การจับคู่จะเพิ่มอุปกรณ์ที่เชื่อมโยงอีกหนึ่งเครื่องไปยังบัญชี WhatsApp เดียวกัน ให้สแกนด้วยข้อมูลประจำตัวที่ OpenClaw ใช้ โหมดหมายเลขส่วนตัว/แชตกับตนเองไม่สามารถโทรหาตนเองได้ ให้ใช้หมายเลข OpenClaw เฉพาะเพื่อโทรไปยังหมายเลขส่วนตัว
</Warning>

<Steps>
  <Step title="เปิดใช้การโทรแบบทดลอง">

    เพิ่ม `actions.calls: true` ในการกำหนดค่าช่องทาง WhatsApp แล้วเริ่ม Gateway ใหม่:

```json
{
  "channels": {
    "whatsapp": {
      "actions": {
        "calls": true
      }
    }
  }
}
```

    เมื่อไม่มีค่าหรือเป็น `false` OpenClaw จะไม่เปิดเผยเครื่องมือ `whatsapp_call`

  </Step>

  <Step title="ติดตั้ง MeowCaller CLI ที่ผ่านการตรวจสอบแล้ว">

    อะแดปเตอร์ต้องการไฟล์ปฏิบัติการ `meowcaller` ใน `PATH` ของโฮสต์ Gateway จนกว่า [MeowCaller PR #7](https://github.com/purpshell/meowcaller/pull/7) จะผสาน ให้สร้างจากสาขาที่ผ่านการตรวจสอบแล้ว:

```bash
git clone --branch feat/send-only-notify https://github.com/steipete/meowcaller.git
cd meowcaller
git checkout 752050471fc2bf7a8cdfbf7dbd3cd4e865d85d3f
mkdir -p "$HOME/.local/bin"
go build -o "$HOME/.local/bin/meowcaller" ./cmd/meowcaller
```

    ตรวจสอบว่า `$HOME/.local/bin` อยู่ใน `PATH` ของบริการ Gateway รีวิชันนี้มีคำสั่ง `pair` และคำสั่งส่งอย่างเดียว `notify` อย่างชัดเจน โดย `notify` จะไม่เปิดไมโครโฟน ลำโพง อุปกรณ์วิดีโอ หรือการบันทึกเพื่อวินิจฉัย ห้ามใช้คำสั่ง `play` ของ CLI ตัวอย่างต้นทางแทน

  </Step>

  <Step title="จับคู่อุปกรณ์ที่เชื่อมโยงสำหรับ MeowCaller">

    ขอให้เอเจนต์ WhatsApp ตรวจสอบการตั้งค่าการโทร (การดำเนินการสถานะ `whatsapp_call` จะรายงานไดเรกทอรีสถานะเฉพาะบัญชีและคำสั่งจับคู่) สำหรับบัญชีเริ่มต้น:

```bash
state_dir="$HOME/.openclaw/credentials/whatsapp-calls/default"
mkdir -p "$state_dir"
chmod 700 "$state_dir"
meowcaller pair --store "$state_dir/wa-voip.db"
```

    เรียกใช้แบบโต้ตอบ สแกน QR จาก **WhatsApp > Linked devices** และรอ `MeowCaller linked device ready` เก็บ `wa-voip.db` เป็นความลับ เนื่องจากเป็นเซสชัน MeowCaller บัญชีที่ไม่ใช่บัญชีเริ่มต้นจะมีพาธที่เก็บของตนเองจากการดำเนินการสถานะ ส่วนบน Windows ให้เรียกใช้คำสั่ง PowerShell ที่ระบุ

  </Step>

  <Step title="กำหนดค่า TTS และโทรจาก WhatsApp">

    กำหนดค่า [ผู้ให้บริการ TTS](/th/tools/tts) ที่รองรับระบบโทรศัพท์ เริ่ม Gateway ใหม่ แล้วส่งคำขอ เช่น `Call me and say the build finished.` เครื่องมือจะระบุผู้ส่งจากบริบทขาเข้าที่เชื่อถือได้ สังเคราะห์ไฟล์ WAV ส่วนตัวชั่วคราว เรียกใช้ MeowCaller ภายในช่วงเวลาการโทรที่จำกัด และลบไฟล์เสียงหลังจากนั้น OpenClaw จะส่งที่เก็บของบัญชีอย่างชัดเจน รอสถานะออกเป็นศูนย์หลังการรับสาย/เล่นเสียง/วางสาย และถือว่าการหมดเวลาหรือสถานะออกที่ไม่เป็นศูนย์เป็นการเรียกเครื่องมือที่ล้มเหลว

  </Step>
</Steps>

ข้อจำกัด: รองรับเฉพาะการโทรด้วยเสียงขาออกแบบหนึ่งต่อหนึ่ง ไม่มีหมายเลขปลายทางตามอำเภอใจ ไม่ใช้การยืนยันตัวตนร่วมกับการเชื่อมต่อแชต ไม่รองรับการโทรหาตนเองจากโหมดหมายเลขส่วนตัว/แชตกับตนเอง เสียงสังเคราะห์จำกัดไว้ที่ 60 วินาที ไม่มีใบตอบรับจากฝั่งเครื่องโทรศัพท์ว่าได้ยินเสียงนอกเหนือจากการเสร็จสิ้นขั้นตอนรับสาย/เล่นเสียง/วางสายของ MeowCaller และ OpenClaw จะหยุดกระบวนการประกอบหลังจากช่วงเวลาจำกัด 115-175 วินาที (ครอบคลุมขั้นตอนการเชื่อมต่อ การรับสาย การเล่นเสียง และการปิดระบบของ MeowCaller)

## พรอมต์การอนุมัติ

WhatsApp สามารถแสดงพรอมต์การอนุมัติ exec และ Plugin เป็นรีแอ็กชัน `👍`/`👎` โดยควบคุมผ่านการกำหนดค่าการส่งต่อการอนุมัติระดับบนสุด:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "session",
    },
    plugin: {
      enabled: true,
      mode: "targets",
      targets: [{ channel: "whatsapp", to: "+15551234567" }],
    },
  },
}
```

`approvals.exec` และ `approvals.plugin` เป็นอิสระต่อกัน การเปิดใช้ WhatsApp เป็นช่องทางเพียงเชื่อมโยงทรานสปอร์ต และจะไม่ส่งสิ่งใด เว้นแต่จะเปิดใช้ตระกูลการอนุมัติที่ตรงกันและกำหนดเส้นทางมายังช่องทางนี้ โหมดเซสชันจะส่งการอนุมัติด้วยอีโมจิแบบเนทีฟเฉพาะการอนุมัติที่เริ่มจาก WhatsApp เท่านั้น โหมดเป้าหมายใช้ไปป์ไลน์การส่งต่อร่วมสำหรับเป้าหมายที่ระบุไว้อย่างชัดเจน และไม่สร้างการกระจาย DM แยกไปยังผู้อนุมัติ

รีแอ็กชันการอนุมัติของ WhatsApp ต้องระบุผู้อนุมัติอย่างชัดเจนใน `allowFrom` (หรือ `"*"`) `defaultTo` กำหนดเป้าหมายข้อความเริ่มต้นทั่วไป ไม่ใช่รายชื่อผู้อนุมัติ คำสั่ง `/approve` ที่เรียกด้วยตนเองยังต้องผ่านเส้นทางการให้สิทธิ์ผู้ส่งตามปกติของ WhatsApp ก่อนดำเนินการอนุมัติ

## ฮุกของ Plugin และความเป็นส่วนตัว

ข้อความ WhatsApp ขาเข้าอาจมีเนื้อหาส่วนบุคคล หมายเลขโทรศัพท์ ตัวระบุกลุ่ม ชื่อผู้ส่ง และฟิลด์เชื่อมโยงเซสชัน WhatsApp จะไม่บรอดแคสต์เพย์โหลดฮุก `message_received` ขาเข้าไปยัง Plugin เว้นแต่จะเลือกเข้าร่วม:

```json5
{
  channels: {
    whatsapp: {
      pluginHooks: {
        messageReceived: true,
      },
    },
  },
}
```

จำกัดการเลือกเข้าร่วมไว้ที่บัญชีเดียวภายใต้ `channels.whatsapp.accounts.<id>.pluginHooks.messageReceived` เปิดใช้เฉพาะกับ Plugin ที่ไว้วางใจให้เข้าถึงเนื้อหาและตัวระบุ WhatsApp ขาเข้าเท่านั้น

## การควบคุมการเข้าถึงและการเปิดใช้งาน

<Tabs>
  <Tab title="นโยบาย DM">
    `channels.whatsapp.dmPolicy`:

    | ค่า | ลักษณะการทำงาน |
    | --- | --- |
    | `pairing` (ค่าเริ่มต้น) | ผู้ส่งที่ไม่รู้จักร้องขอการจับคู่ โดยเจ้าของเป็นผู้อนุมัติ |
    | `allowlist` | อนุญาตเฉพาะผู้ส่ง `allowFrom` |
    | `open` | กำหนดให้ `allowFrom` มี `"*"` |
    | `disabled` | บล็อก DM ทั้งหมด |

    `allowFrom` รองรับหมายเลขในรูปแบบ E.164 (ปรับรูปแบบภายใน) โดยเป็นเพียงรายการควบคุมการเข้าถึงสำหรับผู้ส่ง DM เท่านั้น ไม่ได้ควบคุมการส่งขาออกที่ระบุอย่างชัดเจนไปยัง JID ของกลุ่มหรือ JID ช่องทาง `@newsletter`

    การแทนที่สำหรับหลายบัญชี: `channels.whatsapp.accounts.<id>.dmPolicy` (และ `.allowFrom`) มีลำดับความสำคัญเหนือค่าเริ่มต้นระดับช่องทางสำหรับบัญชีนั้น

    หมายเหตุเกี่ยวกับรันไทม์:

    - การจับคู่จะคงอยู่ในที่เก็บรายการอนุญาตของช่องและผสานกับ `allowFrom` ที่กำหนดค่าไว้
    - ระบบอัตโนมัติตามกำหนดเวลาและการใช้ผู้รับสำรองของ Heartbeat จะใช้เป้าหมายการส่งที่ระบุอย่างชัดเจนหรือ `allowFrom` ที่กำหนดค่าไว้ การอนุมัติการจับคู่ DM จะไม่ถือเป็นผู้รับ Cron/Heartbeat โดยปริยาย
    - หากไม่ได้กำหนดค่ารายการอนุญาต หมายเลขของตนเองที่เชื่อมโยงไว้จะได้รับอนุญาตโดยค่าเริ่มต้น
    - OpenClaw จะไม่จับคู่ DM `fromMe` ขาออกโดยอัตโนมัติเด็ดขาด (ข้อความที่คุณส่งถึงตัวเองจากอุปกรณ์ที่เชื่อมโยงไว้)

  </Tab>

  <Tab title="นโยบายกลุ่มและรายการอนุญาต">
    การเข้าถึงกลุ่มมีสองชั้น:

    1. **รายการอนุญาตสมาชิกกลุ่ม** (`channels.whatsapp.groups`): หากละ `groups` ไว้ ทุกกลุ่มจะมีสิทธิ์ใช้งาน หากระบุไว้ ค่านี้จะทำหน้าที่เป็นรายการอนุญาตของกลุ่ม (`"*"` อนุญาตทั้งหมด)
    2. **นโยบายผู้ส่งในกลุ่ม** (`channels.whatsapp.groupPolicy` + `groupAllowFrom`): `open` จะข้ามรายการอนุญาตผู้ส่ง, `allowlist` กำหนดให้ต้องตรงกับ `groupAllowFrom` (หรือ `*`) และ `disabled` จะบล็อกข้อความขาเข้าจากกลุ่มทั้งหมด

    หากไม่ได้ตั้งค่า `groupAllowFrom` การตรวจสอบผู้ส่งจะย้อนกลับไปใช้ `allowFrom` เมื่อมีรายการอยู่ ระบบจะประเมินรายการอนุญาตผู้ส่งก่อนการเปิดใช้งานด้วยการกล่าวถึง/การตอบกลับ

    หากไม่มีบล็อก `channels.whatsapp` อยู่เลย รันไทม์จะย้อนกลับไปใช้ `groupPolicy: "allowlist"` (พร้อมบันทึกคำเตือน) แม้จะตั้งค่า `channels.defaults.groupPolicy` เป็นค่าอื่นก็ตาม

    <Note>
    การระบุสมาชิกกลุ่มมีกลไกป้องกันสำหรับบัญชีเดียว: หากกำหนดค่าบัญชี WhatsApp เพียงบัญชีเดียวและ `accounts.<id>.groups` ของบัญชีนั้นเป็นออบเจ็กต์ว่างที่ระบุไว้อย่างชัดเจน (`{}`) ระบบจะถือว่า "ไม่ได้ตั้งค่า" และย้อนกลับไปใช้แมป `channels.whatsapp.groups` ระดับราก แทนที่จะบล็อกทุกกลุ่มโดยไม่มีการแจ้งเตือน เมื่อกำหนดค่า 2 บัญชีขึ้นไป แมปบัญชีว่างที่ระบุไว้อย่างชัดเจนจะยังคงว่างและไม่ย้อนกลับไปใช้ค่าอื่น ซึ่งช่วยให้บัญชีหนึ่งสามารถปิดใช้งานทุกกลุ่มโดยตั้งใจได้โดยไม่กระทบบัญชีอื่น
    </Note>

  </Tab>

  <Tab title="การกล่าวถึงและ /activation">
    โดยค่าเริ่มต้น การตอบกลับในกลุ่มต้องมีการกล่าวถึง การตรวจจับการกล่าวถึงประกอบด้วย:

    - การกล่าวถึงตัวตนของบอตใน WhatsApp อย่างชัดเจน
    - รูปแบบนิพจน์ทั่วไปสำหรับการกล่าวถึงที่กำหนดค่าไว้ (`agents.list[].groupChat.mentionPatterns`, ใช้ `messages.groupChat.mentionPatterns` เป็นค่าสำรอง)
    - ข้อความถอดเสียงจากบันทึกเสียงขาเข้าสำหรับข้อความกลุ่มที่ได้รับอนุญาต
    - การตรวจจับการตอบกลับถึงบอตโดยปริยาย (ผู้ส่งข้อความตอบกลับตรงกับตัวตนของบอต)

    ความปลอดภัย: การอ้างข้อความ/ตอบกลับทำให้ผ่านเกณฑ์การกล่าวถึงเท่านั้น แต่ **ไม่ได้** ให้สิทธิ์แก่ผู้ส่ง เมื่อใช้ `groupPolicy: "allowlist"` ผู้ส่งที่ไม่อยู่ในรายการอนุญาตจะยังคงถูกบล็อก แม้จะตอบกลับข้อความของผู้ใช้ที่อยู่ในรายการอนุญาตก็ตาม

    คำสั่งเปิดใช้งานระดับเซสชัน: `/activation mention` หรือ `/activation always` คำสั่งนี้จะอัปเดตสถานะเซสชัน (ไม่ใช่การกำหนดค่าส่วนกลาง) และจำกัดให้เจ้าของใช้งานเท่านั้น

  </Tab>
</Tabs>

## การเชื่อมโยง ACP ที่กำหนดค่าไว้

WhatsApp รองรับการเชื่อมโยง ACP แบบถาวรผ่าน `bindings[]` ระดับบนสุด:

```json5
{
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "direct", id: "+15555550123" },
      },
    },
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "group", id: "120363424282127706@g.us" },
      },
    },
  ],
}
```

แชตโดยตรงจะจับคู่กับหมายเลข E.164 ส่วนกลุ่มจะจับคู่กับ JID กลุ่มของ WhatsApp รายการอนุญาตของกลุ่ม นโยบายผู้ส่ง และเกณฑ์การกล่าวถึง/การเปิดใช้งานจะทำงานก่อนที่ OpenClaw จะตรวจสอบว่ามีเซสชัน ACP ที่เชื่อมโยงอยู่ การเชื่อมโยงที่ตรงกันจะเป็นเจ้าของเส้นทาง กลุ่มกระจายข้อความจะไม่กระจายรอบนั้นไปยังเซสชัน WhatsApp ปกติ

## ลักษณะการทำงานของหมายเลขส่วนตัวและแชตกับตนเอง

เมื่อหมายเลขของตนเองที่เชื่อมโยงไว้ปรากฏอยู่ใน `allowFrom` ด้วย ระบบป้องกันสำหรับแชตกับตนเองจะเปิดใช้งาน: ข้ามใบตอบรับการอ่านสำหรับรอบแชตกับตนเอง, ละเว้นพฤติกรรมทริกเกอร์อัตโนมัติจาก JID การกล่าวถึงที่อาจแจ้งเตือนตัวคุณเอง และกำหนดให้การตอบกลับใช้ `[{identity.name}]` (หรือ `[openclaw]`) โดยค่าเริ่มต้นเมื่อไม่ได้ตั้งค่า `messages.responsePrefix`

    - การกำหนดค่า: `channels.whatsapp.historyLimit`, ใช้ `messages.groupChat.historyLimit` เป็นค่าสำรอง

    - `channels.whatsapp.streaming.chunkMode = "length" | "newline"`; `newline` จะเลือกขอบเขตย่อหน้าก่อน (บรรทัดว่าง) แล้วจึงย้อนกลับไปใช้การแบ่งส่วนที่ปลอดภัยตามความยาว

  <Accordion title="ลักษณะการทำงานของสื่อขาออก">
    - รองรับเพย์โหลดรูปภาพ วิดีโอ เสียง (บันทึกเสียง PTT) และเอกสาร
    - ระบบจะส่งเสียงเป็นเพย์โหลด `audio` ของ Baileys พร้อม `ptt: true` โดยแสดงผลเป็นบันทึกเสียงแบบกดเพื่อพูด ระบบจะรักษา `audioAsVoice` ไว้ในเพย์โหลดการตอบกลับ เพื่อให้เอาต์พุตบันทึกเสียงจาก TTS ยังคงใช้เส้นทางนี้โดยไม่ขึ้นกับรูปแบบต้นทางของผู้ให้บริการ
    - เสียง Ogg/Opus แบบเนทีฟจะส่งเป็น `audio/ogg; codecs=opus` ส่วนรูปแบบอื่นทั้งหมด (รวมถึงเอาต์พุต MP3/WebM จาก Microsoft Edge TTS) จะถูกแปลงด้วย `ffmpeg` เป็น Ogg/Opus โมโน 48 kHz ก่อนส่งแบบ PTT
    - `/tts latest` จะส่งการตอบกลับล่าสุดของผู้ช่วยเป็นบันทึกเสียงหนึ่งรายการและระงับการส่งซ้ำสำหรับการตอบกลับเดียวกัน ส่วน `/tts chat on|off|default` ควบคุม TTS อัตโนมัติสำหรับแชตปัจจุบัน
    - การใช้ `gifPlayback: true` กับวิดีโอขาออกจะเปิดใช้การเล่น GIF แบบเคลื่อนไหว
    - `forceDocument`/`asDocument` จะกำหนดเส้นทางรูปภาพ GIF และวิดีโอขาออกผ่านเพย์โหลดเอกสารของ Baileys เพื่อหลีกเลี่ยงการบีบอัดสื่อของ WhatsApp โดยคงชื่อไฟล์และประเภท MIME ที่ระบุได้ไว้
    - คำบรรยายจะใช้กับสื่อรายการแรกในการตอบกลับที่มีหลายสื่อ ยกเว้นบันทึกเสียง PTT: ระบบจะส่งเสียงก่อนโดยไม่มีคำบรรยาย จากนั้นส่งคำบรรยายเป็นข้อความแยกต่างหาก (ไคลเอนต์ WhatsApp แสดงผลคำบรรยายของบันทึกเสียงไม่สม่ำเสมอ)
    - แหล่งที่มาของสื่ออาจเป็น HTTP(S), `file://` หรือพาธภายในเครื่อง

  </Accordion>

  <Accordion title="ขีดจำกัดขนาดสื่อและลักษณะการทำงานของค่าสำรอง">
    - ขีดจำกัดการบันทึกขาเข้าและการส่งขาออก: `channels.whatsapp.mediaMaxMb` (ค่าเริ่มต้น `50`)
    - การแทนค่าระดับบัญชี: `channels.whatsapp.accounts.<id>.mediaMaxMb`
    - รูปภาพจะปรับให้เหมาะสมโดยอัตโนมัติ (ปรับขนาด/ไล่ระดับคุณภาพ) เพื่อให้พอดีกับขีดจำกัด เว้นแต่ `forceDocument`/`asDocument` จะร้องขอการส่งแบบเอกสาร
    - เมื่อส่งสื่อล้มเหลว ค่าสำรองสำหรับรายการแรกจะส่งคำเตือนเป็นข้อความแทนการทิ้งการตอบกลับโดยไม่มีการแจ้งเตือน

  </Accordion>
</AccordionGroup>

## การอ้างข้อความในการตอบกลับ

`channels.whatsapp.replyToMode` ควบคุมการอ้างข้อความตอบกลับแบบเนทีฟ (การตอบกลับขาออกจะแสดงการอ้างข้อความขาเข้า):

| ค่า             | ลักษณะการทำงาน                                                       |
| ----------------- | -------------------------------------------------------------- |
| `"off"` (ค่าเริ่มต้น) | ไม่อ้างข้อความ ส่งเป็นข้อความธรรมดา                           |
| `"first"`         | อ้างเฉพาะส่วนแรกของการตอบกลับขาออก                      |
| `"all"`           | อ้างทุกส่วนของการตอบกลับขาออก                               |
| `"batched"`       | อ้างการตอบกลับแบบชุดที่อยู่ในคิว และไม่อ้างการตอบกลับทันที |

การแทนค่าระดับบัญชี: `channels.whatsapp.accounts.<id>.replyToMode`

```json5
{ channels: { whatsapp: { replyToMode: "first" } } }
```

## ระดับการแสดงปฏิกิริยา

`channels.whatsapp.reactionLevel` ควบคุมขอบเขตการใช้ปฏิกิริยาอีโมจิของเอเจนต์:

| ระดับ                 | ปฏิกิริยาตอบรับ | ปฏิกิริยาที่เอเจนต์เป็นผู้เริ่ม  |
| --------------------- | ------------- | -------------------------- |
| `"off"`               | ไม่            | ไม่                         |
| `"ack"`               | ใช่           | ไม่                         |
| `"minimal"` (ค่าเริ่มต้น) | ใช่           | ใช่ แนวทางแบบระมัดระวัง |
| `"extensive"`         | ใช่           | ใช่ แนวทางที่สนับสนุนให้ใช้   |

การแทนค่าระดับบัญชี: `channels.whatsapp.accounts.<id>.reactionLevel`

```json5
{ channels: { whatsapp: { reactionLevel: "ack" } } }
```

## ปฏิกิริยาตอบรับ

`channels.whatsapp.ackReaction` จะส่งปฏิกิริยาทันทีเมื่อได้รับข้อความขาเข้า โดยมี `reactionLevel` เป็นเกณฑ์ควบคุม (ระงับเมื่อ `"off"`):

```json5
{
  channels: {
    whatsapp: {
      ackReaction: {
        emoji: "👀",
        direct: true,
        group: "mentions", // เสมอ | การกล่าวถึง | ไม่เลย
      },
    },
  },
}
```

หมายเหตุ: ส่งทันทีหลังจากยอมรับข้อความขาเข้า (ก่อนตอบกลับ) หากมี `ackReaction` แต่ไม่มี `emoji` WhatsApp จะใช้อีโมจิตัวตนของเอเจนต์ที่รับเส้นทาง โดยใช้อีโมจิ "👀" เป็นค่าสำรอง (ละ `ackReaction` ไว้หรือตั้งค่า `emoji: ""` เพื่อไม่ให้มีการตอบรับ) ระบบจะบันทึกความล้มเหลวแต่จะไม่บล็อกการส่งคำตอบ โหมดกลุ่ม `mentions` จะแสดงปฏิกิริยาเฉพาะรอบที่ทริกเกอร์ด้วยการกล่าวถึง ขณะที่การเปิดใช้งานกลุ่ม `always` จะข้ามการตรวจสอบนั้น WhatsApp ใช้เฉพาะ `channels.whatsapp.ackReaction` เท่านั้น (`messages.ackReaction` แบบเดิมไม่มีผลที่นี่)

## ปฏิกิริยาสถานะวงจรชีวิต

ตั้งค่า `messages.statusReactions.enabled: true` เพื่อให้ WhatsApp แทนที่ปฏิกิริยาตอบรับระหว่างรอบแทนการคงอีโมจิรับทราบแบบคงที่ โดยหมุนเวียนผ่านสถานะต่าง ๆ เช่น อยู่ในคิว กำลังคิด กิจกรรมเครื่องมือ Compaction เสร็จสิ้น และข้อผิดพลาด:

```json5
{
  messages: {
    statusReactions: {
      enabled: true,
      emojis: {
        deploy: "🛫",
        build: "🏗️",
        concierge: "💁",
      },
    },
  },
}
```

หมายเหตุ: `channels.whatsapp.ackReaction` ยังคงควบคุมสิทธิ์สำหรับข้อความโดยตรงและกลุ่ม สถานะอยู่ในคิวใช้อีโมจิที่มีผลเดียวกับปฏิกิริยาตอบรับปกติ WhatsApp มีช่องปฏิกิริยาของบอตหนึ่งช่องต่อข้อความ ดังนั้นการอัปเดตวงจรชีวิตจะแทนที่ปฏิกิริยาปัจจุบัน ณ ตำแหน่งเดิม `messages.removeAckAfterReply: true` จะล้างปฏิกิริยาสถานะสุดท้ายหลังจากระยะเวลาคงสถานะเสร็จสิ้น/ข้อผิดพลาดที่กำหนดไว้ หมวดหมู่อีโมจิของเครื่องมือประกอบด้วย `tool`, `coding`, `web`, `deploy`, `build` และ `concierge`

## เครื่องมือ การดำเนินการ และการเขียนการกำหนดค่า

- การรองรับเครื่องมือ Agent รวมถึงการดำเนินการแสดงความรู้สึกใน WhatsApp (`react`)
- เกตควบคุมการดำเนินการ: `channels.whatsapp.actions.reactions`, `channels.whatsapp.actions.polls` (การดำเนินการที่มีอยู่ใช้ค่าเริ่มต้นเป็น `true`), `channels.whatsapp.actions.calls` (ค่าเริ่มต้น `false` โปรดดู MeowCaller ด้านบน)
- การเขียนการกำหนดค่าที่เริ่มต้นโดยช่องทางจะเปิดใช้งานโดยค่าเริ่มต้น ปิดใช้งานผ่าน `channels.whatsapp.configWrites: false`

```bash
openclaw channels login --channel whatsapp
```

  <Accordion title="เชื่อมโยงแล้วแต่ตัดการเชื่อมต่อ / วนเชื่อมต่อใหม่">
    อาการ: บัญชีที่เชื่อมโยงแล้วตัดการเชื่อมต่อหรือพยายามเชื่อมต่อใหม่ซ้ำๆ

    บัญชีที่ไม่ค่อยมีกิจกรรมสามารถคงการเชื่อมต่อไว้เกินระยะหมดเวลาของข้อความตามปกติได้ ตัวเฝ้าระวังจะรีสตาร์ตเฉพาะเมื่อกิจกรรมของการรับส่งข้อมูล WhatsApp Web หยุดลง ซ็อกเก็ตปิด หรือกิจกรรมระดับแอปพลิเคชันเงียบเกินกรอบเวลาความปลอดภัยที่ยาวกว่า (โปรดดูโมเดลรันไทม์ด้านบน)

    หากบันทึกแสดง `status=408 Request Time-out Connection was lost` ซ้ำๆ ให้ปรับเวลาของซ็อกเก็ต Baileys ภายใต้ `web.whatsapp` เริ่มโดยลด `keepAliveIntervalMs` ให้ต่ำกว่าระยะหมดเวลาเมื่อไม่มีการใช้งานของเครือข่าย และเพิ่ม `connectTimeoutMs` สำหรับการเชื่อมต่อที่ช้าหรือมีการสูญเสียข้อมูล:

    ```json5
    {
      web: {
        whatsapp: {
          keepAliveIntervalMs: 15000,
          connectTimeoutMs: 60000,
          defaultQueryTimeoutMs: 60000,
        },
      },
    }
    ```

    วิธีแก้ไข:

    ```bash
    openclaw channels status --probe
    openclaw doctor
    openclaw logs --follow
    openclaw gateway status
    ```

    หากยังคงเกิดการวนซ้ำหลังแก้ไขการเชื่อมต่อของโฮสต์และการตั้งเวลาแล้ว ให้สำรองไดเรกทอรีการยืนยันตัวตนของบัญชีและเชื่อมโยงใหม่:

    ```bash
    cp -a ~/.openclaw/credentials/whatsapp/<accountId> \
      ~/.openclaw/credentials/whatsapp/<accountId>.bak
    openclaw channels logout --channel whatsapp --account <accountId>
    openclaw channels login --channel whatsapp --account <accountId>
    ```

    หาก `~/.openclaw/logs/whatsapp-health.log` ระบุว่า `Gateway inactive` แต่ทั้ง `openclaw gateway status` และ `openclaw channels status --probe` แสดงว่าปกติ ให้เรียกใช้ `openclaw doctor` บน Linux คำสั่ง doctor จะเตือนเกี่ยวกับรายการ crontab แบบเดิมที่เรียกใช้สคริปต์ `~/.openclaw/bin/ensure-whatsapp.sh` ซึ่งเลิกใช้แล้ว ให้ลบรายการเหล่านั้นด้วย `crontab -e` เนื่องจาก cron อาจไม่มีสภาพแวดล้อมบัสผู้ใช้ของ systemd และทำให้สคริปต์เก่านั้นรายงานสถานะ Gateway ผิดพลาด

  </Accordion>

  <Accordion title="การเข้าสู่ระบบด้วย QR หมดเวลาหลังพร็อกซี">
    อาการ: `openclaw channels login --channel whatsapp` ล้มเหลวก่อนแสดง QR ที่ใช้งานได้ พร้อม `status=408 Request Time-out` หรือการตัดการเชื่อมต่อซ็อกเก็ต TLS

    การเข้าสู่ระบบ WhatsApp Web ใช้สภาพแวดล้อมพร็อกซีมาตรฐานของโฮสต์ Gateway (`HTTPS_PROXY`, `HTTP_PROXY`, รูปแบบตัวพิมพ์เล็ก, `NO_PROXY`) ตรวจสอบว่าโปรเซส Gateway สืบทอดตัวแปรสภาพแวดล้อมพร็อกซี และ `NO_PROXY` ไม่ตรงกับ `mmg.whatsapp.net`

  </Accordion>

  </Accordion>

  <Accordion title="การตอบกลับปรากฏในข้อความถอดเสียงแต่ไม่ปรากฏใน WhatsApp">
    แถวข้อความถอดเสียงบันทึกสิ่งที่ Agent สร้างขึ้น ส่วนการส่งผ่าน WhatsApp จะได้รับการตรวจสอบแยกต่างหาก OpenClaw จะถือว่าการตอบกลับอัตโนมัติถูกส่งแล้วก็ต่อเมื่อ Baileys ส่งคืนรหัสข้อความขาออกสำหรับการส่งข้อความหรือสื่อที่มองเห็นได้อย่างน้อยหนึ่งรายการ

    การแสดงความรู้สึกเพื่อรับทราบเป็นใบรับก่อนตอบกลับที่ทำงานอย่างเป็นอิสระ การแสดงความรู้สึกสำเร็จไม่ได้พิสูจน์ว่าการตอบกลับด้วยข้อความ/สื่อในภายหลังได้รับการยอมรับ ตรวจสอบบันทึก Gateway สำหรับ `auto-reply delivery failed` หรือ `auto-reply was not accepted by WhatsApp provider`

  </Accordion>

  <Accordion title="ข้อความกลุ่มถูกเพิกเฉยโดยไม่คาดคิด">
    ตรวจสอบตามลำดับนี้: `groupPolicy`, `groupAllowFrom`/`allowFrom`, รายการอนุญาต `groups`, เกตควบคุมการกล่าวถึง (`requireMention` + รูปแบบการกล่าวถึง) และคีย์ที่ซ้ำกันใน `openclaw.json` (รายการ JSON5 ที่อยู่ภายหลังจะแทนที่รายการก่อนหน้า ให้มี `groupPolicy` เพียงรายการเดียวต่อขอบเขต)

    หากมี `channels.whatsapp.groups` WhatsApp ยังคงสามารถสังเกตข้อความจากกลุ่มอื่นได้ แต่ OpenClaw จะทิ้งข้อความเหล่านั้นก่อนกำหนดเส้นทางไปยังเซสชัน เพิ่ม JID ของกลุ่มลงใน `channels.whatsapp.groups` หรือเพิ่ม `groups["*"]` เพื่อยอมรับทุกกลุ่ม โดยยังคงควบคุมการอนุญาตผู้ส่งภายใต้ `groupPolicy`/`groupAllowFrom`

  </Accordion>

  <Accordion title="คำเตือนเกี่ยวกับรันไทม์ Bun">
    Gateway ของ OpenClaw ต้องใช้ Node เนื่องจาก Bun ไม่มี API `node:sqlite` ที่พื้นที่จัดเก็บสถานะมาตรฐานใช้ และคำสั่ง doctor จะย้ายบริการ Bun แบบเดิมไปยัง Node
  </Accordion>
</AccordionGroup>

## พรอมต์ระบบ

WhatsApp รองรับพรอมต์ระบบแบบ Telegram สำหรับกลุ่มและแชตโดยตรงผ่านแมป `groups` และ `direct`

การกำหนดค่าสำหรับข้อความกลุ่ม: ระบบจะกำหนดแมป `groups` ที่มีผลก่อน หากบัญชีกำหนดคีย์ `groups` ของตนเอง ไม่ว่าจะมีค่าใด คีย์ดังกล่าวจะแทนที่แมปราก `groups` ทั้งหมด (ไม่มีการผสานแบบลึก) จากนั้นการค้นหาพรอมต์จะทำงานบนแมปผลลัพธ์เพียงแมปเดียวนั้น:

1. **พรอมต์เฉพาะกลุ่ม** (`groups["<groupId>"].systemPrompt`): ใช้เมื่อมีรายการกลุ่มอยู่ **และ** มีการกำหนดคีย์ `systemPrompt` สตริงว่าง (`""`) จะระงับไวลด์การ์ดและไม่ใช้พรอมต์ใดๆ
2. **พรอมต์ไวลด์การ์ดของกลุ่ม** (`groups["*"].systemPrompt`): ใช้เมื่อไม่มีรายการกลุ่มเฉพาะ หรือมีรายการอยู่แต่ไม่มีคีย์ `systemPrompt`

การกำหนดค่าสำหรับข้อความโดยตรงใช้รูปแบบเดียวกันกับแมป `direct` และ `direct["*"]`

<Note>
`dms` ยังคงเป็นบักเก็ตขนาดเล็กสำหรับแทนที่ประวัติต่อ DM (`dms.<id>.historyLimit`) การแทนที่พรอมต์อยู่ภายใต้ `direct`
</Note>

<Note>
ลักษณะการทำงานที่บัญชีแทนที่ค่ารากสำหรับการกำหนดพรอมต์นี้เป็นการแทนที่แบบตื้นทั่วไป: คีย์ `groups`/`direct` ใดๆ ของบัญชี รวมถึงออบเจ็กต์ว่างที่ระบุไว้อย่างชัดเจน จะแทนที่แมปราก ซึ่งแตกต่างจากการตรวจสอบรายการอนุญาตสมาชิกกลุ่มที่อธิบายไว้ด้านบน ซึ่งมีกลไกป้องกันสำหรับบัญชีเดียวในกรณีที่ `groups: {}` ว่างโดยไม่ตั้งใจ
</Note>

**ความแตกต่างจาก Telegram:** Telegram จะระงับ `groups` ระดับรากสำหรับทุกบัญชีในการตั้งค่าแบบหลายบัญชี (แม้แต่บัญชีที่ไม่มี `groups` ของตนเอง) เพื่อป้องกันไม่ให้บอตรับข้อความกลุ่มจากกลุ่มที่บอตไม่ได้เป็นสมาชิก WhatsApp ไม่ใช้กลไกป้องกันดังกล่าว โดยบัญชีใดก็ตามที่ไม่มีการแทนที่ของตนเองจะสืบทอด `groups`/`direct` ระดับราก ไม่ว่าจะมีจำนวนบัญชีเท่าใด ในการตั้งค่า WhatsApp แบบหลายบัญชี ให้กำหนดแมปทั้งหมดภายใต้แต่ละบัญชีอย่างชัดเจนหากต้องการพรอมต์แยกตามบัญชี

ลักษณะการทำงานที่สำคัญ:

- `channels.whatsapp.groups` เป็นทั้งแมปการกำหนดค่าต่อกลุ่มและรายการอนุญาตกลุ่มระดับแชต ไม่ว่าจะอยู่ในขอบเขตรากหรือบัญชี `groups["*"]` หมายถึง "ยอมรับทุกกลุ่ม" สำหรับขอบเขตนั้น
- เพิ่มไวลด์การ์ด `systemPrompt` เฉพาะเมื่อขอบเขตนั้นควรยอมรับทุกกลุ่มอยู่แล้ว หากต้องการให้เฉพาะชุดรหัสกลุ่มคงที่มีสิทธิ์ ให้ระบุพรอมต์ซ้ำในแต่ละรายการที่อนุญาตไว้อย่างชัดเจนแทนการใช้ `groups["*"]`
- การยอมรับกลุ่มและการอนุญาตผู้ส่งเป็นการตรวจสอบคนละส่วนกัน `groups["*"]` จะขยายว่ากลุ่มใดเข้าถึงการจัดการกลุ่มได้ แต่ไม่ได้อนุญาตผู้ส่งทุกคนในกลุ่มเหล่านั้น ซึ่งยังคงควบคุมโดย `groupPolicy`/`groupAllowFrom`
- `channels.whatsapp.direct` ไม่มีผลข้างเคียงที่เทียบเท่าสำหรับ DM: `direct["*"]` เพียงระบุการกำหนดค่าเริ่มต้นหลังจากที่ DM ได้รับการยอมรับโดย `dmPolicy` ร่วมกับ `allowFrom` หรือกฎของพื้นที่จัดเก็บการจับคู่แล้วเท่านั้น

ตัวอย่าง:

```json5
{
  channels: {
    whatsapp: {
      groups: {
        // ใช้เฉพาะเมื่อควรยอมรับทุกกลุ่มในขอบเขตราก
        // ใช้กับทุกบัญชีที่ไม่ได้กำหนดแมป groups ของตนเอง
        "*": { systemPrompt: "พรอมต์เริ่มต้นสำหรับทุกกลุ่ม" },
      },
      direct: {
        // ใช้กับทุกบัญชีที่ไม่ได้กำหนดแมป direct ของตนเอง
        "*": { systemPrompt: "พรอมต์เริ่มต้นสำหรับแชตโดยตรงทั้งหมด" },
      },
      accounts: {
        work: {
          groups: {
            // บัญชีนี้กำหนด groups ของตนเอง ดังนั้น groups ระดับรากจะถูก
            // แทนที่ทั้งหมด หากต้องการคงไวลด์การ์ดไว้ ให้กำหนด "*" ที่นี่อย่างชัดเจนด้วย
            "120363406415684625@g.us": {
              requireMention: false,
              systemPrompt: "เน้นการจัดการโครงการ",
            },
            // ใช้เฉพาะเมื่อควรยอมรับทุกกลุ่มในบัญชีนี้
            "*": { systemPrompt: "พรอมต์เริ่มต้นสำหรับกลุ่มงาน" },
          },
          direct: {
            // บัญชีนี้กำหนดแมป direct ของตนเอง ดังนั้นรายการ direct ระดับรากจะถูก
            // แทนที่ทั้งหมด หากต้องการคงไวลด์การ์ดไว้ ให้กำหนด "*" ที่นี่อย่างชัดเจนด้วย
            "+15551234567": { systemPrompt: "พรอมต์สำหรับแชตโดยตรงของงานที่ระบุ" },
            "*": { systemPrompt: "พรอมต์เริ่มต้นสำหรับแชตโดยตรงของงาน" },
          },
        },
      },
    },
  },
}
```
