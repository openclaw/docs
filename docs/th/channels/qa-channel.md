`qa-channel` คือการขนส่งข้อความสังเคราะห์ภายในรีโพสำหรับ QA อัตโนมัติของ OpenClaw (`extensions/qa-channel` เป็นแพ็กเกจส่วนตัวและไม่รวมอยู่ในการติดตั้งแบบแพ็กเกจ) ไม่ใช่ช่องทางสำหรับการใช้งานจริง แต่มีไว้เพื่อทดสอบขอบเขต Plugin ช่องทางแบบเดียวกับที่การขนส่งจริงใช้ โดยรักษาสถานะให้กำหนดแน่นอนและตรวจสอบได้อย่างสมบูรณ์

## การทำงาน

- ไวยากรณ์เป้าหมายระดับเดียวกับ Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- การสนทนาแบบใช้ร่วมกันใน `channel:` และ `group:` จะแสดงแก่เอเจนต์เป็นรอบการสนทนาในห้องกลุ่ม/ช่องทาง เพื่อทดสอบนโยบายการกำหนดเส้นทางสำหรับการตอบกลับที่มองเห็นได้และเครื่องมือข้อความแบบเดียวกับที่ Discord, Slack, Telegram และการขนส่งลักษณะเดียวกันใช้
- บัสสังเคราะห์ที่ใช้ HTTP สำหรับการฉีดข้อความขาเข้า การบันทึกทรานสคริปต์ขาออก การสร้างเธรด การแสดงความรู้สึก การแก้ไข การลบ และการดำเนินการค้นหา/อ่าน
- ตัวรันตรวจสอบตนเองฝั่งโฮสต์ที่เขียนรายงาน Markdown ไปยัง `.artifacts/qa-e2e/`

## การกำหนดค่า

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

คีย์บัญชี:

- `enabled` - สวิตช์หลักสำหรับบัญชีนี้
- `name` - ป้ายชื่อที่แสดงซึ่งระบุหรือไม่ก็ได้
- `baseUrl` - URL ของบัสสังเคราะห์ บัญชีจะถือว่าได้รับการกำหนดค่าเมื่อตั้งค่านี้แล้ว
- `botUserId` - รหัสผู้ใช้บอตสังเคราะห์ที่ใช้ในไวยากรณ์เป้าหมาย (ค่าเริ่มต้น: `openclaw`)
- `botDisplayName` - ชื่อที่แสดงสำหรับข้อความขาออก (ค่าเริ่มต้น: `OpenClaw QA`)
- `pollTimeoutMs` - ช่วงเวลารอของการโพลระยะยาว ต้องเป็นจำนวนเต็มระหว่าง 100 ถึง 30000 (ค่าเริ่มต้น: 1000)
- `allowFrom` - รายการอนุญาตผู้ส่ง (รหัสผู้ใช้หรือ `"*"`; ค่าเริ่มต้น: `["*"]`) ข้อความส่วนตัวใช้
  นโยบาย `open` เสมอ ส่วนนโยบายกลุ่มแบบรายการอนุญาตจะใช้รหัสผู้ส่ง
  สังเคราะห์เหล่านี้ด้วย
- `groupPolicy` - นโยบายห้องที่ใช้ร่วมกัน: `"open"` (ค่าเริ่มต้น), `"allowlist"` หรือ
  `"disabled"`
- `groupAllowFrom` - รายการอนุญาตผู้ส่งสำหรับห้องที่ใช้ร่วมกันซึ่งระบุหรือไม่ก็ได้ หากละไว้ภายใต้
  `"allowlist"` ช่องทาง QA จะย้อนกลับไปใช้ `allowFrom`
- `groups.<room>.requireMention` - กำหนดให้ต้องกล่าวถึงบอตก่อนตอบกลับใน
  ห้องกลุ่ม/ช่องทางที่ระบุ (ค่าเริ่มต้น: false) `groups."*"` กำหนดค่าเริ่มต้น
  ส่วน `tools` / `toolsBySender` รายห้องใช้กำหนดค่าทับนโยบายเครื่องมือ
- `defaultTo` - เป้าหมายสำรองเมื่อไม่ได้ระบุเป้าหมาย
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` - การควบคุมสิทธิ์เครื่องมือแยกตามการดำเนินการ

คีย์หลายบัญชีที่ระดับบนสุด:

- `accounts` - ระเบียนค่าทับรายบัญชีที่มีชื่อ โดยใช้รหัสบัญชีเป็นคีย์
- `defaultAccount` - รหัสบัญชีที่ต้องการใช้เมื่อกำหนดค่าไว้หลายบัญชี

## ตัวรัน

การตรวจสอบตนเองฝั่งโฮสต์ (เขียนรายงาน Markdown ไว้ภายใต้ `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

คำสั่งนี้กำหนดเส้นทางผ่าน `qa-lab` เริ่มบัส QA ภายในรีโพ บูตส่วนรันไทม์ `qa-channel` และเรียกใช้การตรวจสอบตนเองที่ให้ผลลัพธ์แน่นอน

ชุดสถานการณ์แบบเต็มที่อาศัยรีโพ:

```bash
pnpm openclaw qa suite
```

เรียกใช้สถานการณ์แบบขนานกับเลน Gateway สำหรับ QA ดูสถานการณ์ โปรไฟล์ และโหมดผู้ให้บริการได้ที่ [ภาพรวม QA](/th/concepts/qa-e2e-automation)

ไซต์ QA ที่ใช้ Docker (Gateway + UI ตัวดีบัก QA Lab ในสแต็กเดียว):

```bash
pnpm qa:lab:up
```

สร้างไซต์ QA เริ่มสแต็ก Gateway + QA Lab ที่ใช้ Docker และแสดง URL ของ QA Lab จากนั้นคุณสามารถเลือกสถานการณ์ เลือกเลนโมเดล เริ่มการทำงานแต่ละรายการ และดูผลลัพธ์แบบสดได้ ตัวดีบัก QA Lab แยกจากชุด Control UI ที่เผยแพร่
