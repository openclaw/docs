`qa-channel` เป็นระบบขนส่งข้อความจำลองที่บันเดิลมาเพื่อการ QA อัตโนมัติของ OpenClaw ไม่ใช่ช่องทางสำหรับใช้งานจริง — มีไว้เพื่อทดสอบขอบเขต Plugin ช่องทางเดียวกับที่ระบบขนส่งจริงใช้ พร้อมทำให้สถานะกำหนดซ้ำได้และตรวจสอบได้ครบถ้วน

## สิ่งที่ทำ

- ไวยากรณ์เป้าหมายระดับ Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- การสนทนา `channel:` และ `group:` แบบแชร์จะแสดงต่อเอเจนต์เป็นเทิร์นของห้องกลุ่ม/ช่องทาง จึงทดสอบนโยบายการตอบกลับที่มองเห็นได้และการกำหนดเส้นทางเครื่องมือข้อความแบบเดียวกับที่ Discord, Slack, Telegram และระบบขนส่งที่คล้ายกันใช้
- บัสจำลองที่รองรับด้วย HTTP สำหรับการฉีดข้อความขาเข้า การบันทึกทรานสคริปต์ขาออก การสร้างเธรด รีแอ็กชัน การแก้ไข การลบ และการค้นหา/อ่าน
- ตัวรันตรวจสอบตนเองฝั่งโฮสต์ที่เขียนรายงาน Markdown ไปยัง `.artifacts/qa-e2e/`

## การกำหนดค่า

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

คีย์บัญชี:

- `enabled` — สวิตช์หลักสำหรับบัญชีนี้
- `name` — ป้ายชื่อที่แสดงผลแบบไม่บังคับ
- `baseUrl` — URL ของบัสจำลอง
- `botUserId` — รหัสผู้ใช้บอตแบบ Matrix ที่ใช้ในไวยากรณ์เป้าหมาย
- `botDisplayName` — ชื่อที่แสดงสำหรับข้อความขาออก
- `pollTimeoutMs` — หน้าต่างรอแบบ long-poll จำนวนเต็มระหว่าง 100 ถึง 30000
- `allowFrom` — รายการอนุญาตผู้ส่ง (รหัสผู้ใช้หรือ `"*"`)
- `defaultTo` — เป้าหมายสำรองเมื่อไม่ได้ระบุไว้
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` — การควบคุมการเปิดใช้เครื่องมือรายแอ็กชัน

คีย์หลายบัญชีที่ระดับบนสุด:

- `accounts` — ระเบียนของการแทนที่รายบัญชีที่ตั้งชื่อ โดยใช้รหัสบัญชีเป็นคีย์
- `defaultAccount` — รหัสบัญชีที่ต้องการใช้เมื่อกำหนดค่าไว้หลายบัญชี

## ตัวรัน

การตรวจสอบตนเองฝั่งโฮสต์ (เขียนรายงาน Markdown ใต้ `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

คำสั่งนี้กำหนดเส้นทางผ่าน `qa-lab` เริ่มบัส QA ในรีโป บูตสไลซ์รันไทม์ `qa-channel` ที่บันเดิลมา และรันการตรวจสอบตนเองแบบกำหนดซ้ำได้

ชุดสถานการณ์แบบอิงรีโปเต็มรูปแบบ:

```bash
pnpm openclaw qa suite
```

รันสถานการณ์แบบขนานกับเลน Gateway สำหรับ QA ดู [ภาพรวม QA](/th/concepts/qa-e2e-automation) สำหรับสถานการณ์ โปรไฟล์ และโหมดผู้ให้บริการ

ไซต์ QA ที่รองรับด้วย Docker (Gateway + UI ดีบักเกอร์ QA Lab ในสแต็กเดียว):

```bash
pnpm qa:lab:up
```

สร้างไซต์ QA เริ่มสแต็ก Gateway + QA Lab ที่รองรับด้วย Docker และพิมพ์ URL ของ QA Lab จากนั้นคุณสามารถเลือกสถานการณ์ เลือกเลนโมเดล เปิดการรันแต่ละรายการ และดูผลลัพธ์แบบสดได้ ดีบักเกอร์ QA Lab แยกจากบันเดิล Control UI ที่จัดส่ง
