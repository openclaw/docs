`qa-channel` เป็นระบบขนส่งข้อความสังเคราะห์เฉพาะภายใน repo สำหรับ QA อัตโนมัติของ OpenClaw (`extensions/qa-channel`, แพ็กเกจส่วนตัว ซึ่งไม่รวมอยู่ในการติดตั้งแบบแพ็กเกจ) ไม่ใช่ช่องทางสำหรับการใช้งานจริง แต่มีไว้เพื่อทดสอบขอบเขต Plugin ช่องทางเดียวกับที่ระบบขนส่งจริงใช้ พร้อมทั้งทำให้สถานะกำหนดได้แน่นอนและตรวจสอบได้อย่างครบถ้วน

## ทำอะไรได้บ้าง

- ไวยากรณ์เป้าหมายระดับเดียวกับ Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- การสนทนา `channel:` และ `group:` ที่ใช้ร่วมกันจะแสดงต่อเอเจนต์เป็นเทิร์นของห้องกลุ่ม/ช่องทาง เพื่อให้ทดสอบนโยบายการกำหนดเส้นทางการตอบกลับที่มองเห็นได้และเครื่องมือข้อความแบบเดียวกับที่ Discord, Slack, Telegram และระบบขนส่งที่คล้ายกันใช้
- บัสสังเคราะห์ที่รองรับด้วย HTTP สำหรับแทรกข้อความขาเข้า บันทึกทรานสคริปต์ขาออก สร้างเธรด แสดงความรู้สึก แก้ไข ลบ และดำเนินการค้นหา/อ่าน
- ตัวรันตรวจสอบตนเองฝั่งโฮสต์ที่เขียนรายงาน Markdown ไปยัง `.artifacts/qa-e2e/`

## การกำหนดค่า

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

คีย์บัญชี:

- `enabled` - สวิตช์หลักสำหรับบัญชีนี้
- `name` - ป้ายกำกับการแสดงผลที่ระบุหรือไม่ก็ได้
- `baseUrl` - URL ของบัสสังเคราะห์ ระบบจะถือว่าบัญชีได้รับการกำหนดค่าเมื่อกำหนดค่านี้แล้ว
- `botUserId` - รหัสผู้ใช้บอตสังเคราะห์ที่ใช้ในไวยากรณ์เป้าหมาย (ค่าเริ่มต้น: `openclaw`)
- `botDisplayName` - ชื่อที่แสดงสำหรับข้อความขาออก (ค่าเริ่มต้น: `OpenClaw QA`)
- `pollTimeoutMs` - ช่วงเวลารอของ long-poll เป็นจำนวนเต็มระหว่าง 100 ถึง 30000 (ค่าเริ่มต้น: 1000)
- `allowFrom` - รายการอนุญาตผู้ส่ง (รหัสผู้ใช้หรือ `"*"`; ค่าเริ่มต้น: `["*"]`) DM จะใช้นโยบาย
  `open` เสมอ และนโยบายกลุ่มที่ใช้รายการอนุญาตจะใช้รหัสผู้ส่งสังเคราะห์
  เหล่านี้ด้วย
- `groupPolicy` - นโยบายห้องที่ใช้ร่วมกัน: `"open"` (ค่าเริ่มต้น), `"allowlist"` หรือ
  `"disabled"`
- `groupAllowFrom` - รายการอนุญาตผู้ส่งของห้องที่ใช้ร่วมกันซึ่งระบุหรือไม่ก็ได้ เมื่อไม่ได้ระบุภายใต้
  `"allowlist"` QA Channel จะใช้ `allowFrom` แทน
- `groups.<room>.requireMention` - กำหนดให้ต้องกล่าวถึงบอตก่อนตอบกลับใน
  ห้องกลุ่ม/ช่องทางที่ระบุ (ค่าเริ่มต้น: false) `groups."*"` กำหนดค่าเริ่มต้น
  ส่วน `tools` / `toolsBySender` ของแต่ละห้องกำหนดค่าลบล้างนโยบายเครื่องมือ
- `defaultTo` - เป้าหมายสำรองเมื่อไม่ได้ระบุเป้าหมาย
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` - การควบคุมสิทธิ์ใช้เครื่องมือแยกตามการดำเนินการ

คีย์หลายบัญชีที่ระดับบนสุด:

- `accounts` - ระเบียนค่าลบล้างแยกตามบัญชีที่มีชื่อ โดยใช้รหัสบัญชีเป็นคีย์
- `defaultAccount` - รหัสบัญชีที่ต้องการใช้เมื่อกำหนดค่าไว้หลายบัญชี

## ตัวรัน

การตรวจสอบตนเองฝั่งโฮสต์ (เขียนรายงาน Markdown ภายใต้ `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

คำสั่งนี้กำหนดเส้นทางผ่าน `qa-lab` เริ่มบัส QA ภายใน repo บูตส่วนรันไทม์ `qa-channel` และเรียกใช้การตรวจสอบตนเองแบบกำหนดผลลัพธ์ได้แน่นอน

ชุดสถานการณ์เต็มรูปแบบที่รองรับด้วย repo:

```bash
pnpm openclaw qa suite
```

เรียกใช้สถานการณ์แบบขนานกับเลน Gateway สำหรับ QA ดูสถานการณ์ โปรไฟล์ และโหมดผู้ให้บริการได้ที่[ภาพรวม QA](/th/concepts/qa-e2e-automation)

ไซต์ QA ที่รองรับด้วย Docker (Gateway + UI ดีบักเกอร์ QA Lab ในสแต็กเดียว):

```bash
pnpm qa:lab:up
```

สร้างไซต์ QA เริ่มสแต็ก Gateway + QA Lab ที่รองรับด้วย Docker และแสดง URL ของ QA Lab จากจุดนั้นสามารถเลือกสถานการณ์ เลือกเลนโมเดล เริ่มรันแต่ละรายการ และดูผลลัพธ์แบบสดได้ ดีบักเกอร์ QA Lab แยกจากชุด Control UI ที่เผยแพร่
