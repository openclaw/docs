`qa-channel` เป็นตัวขนส่งข้อความสังเคราะห์ที่บันเดิลมาให้สำหรับ QA อัตโนมัติของ OpenClaw ไม่ใช่ช่องทางสำหรับใช้งานจริง — มีไว้เพื่อทดสอบขอบเขต Plugin ของช่องทางเดียวกับที่ตัวขนส่งจริงใช้ ขณะเดียวกันก็ทำให้สถานะกำหนดซ้ำได้และตรวจสอบได้ครบถ้วน

## ทำอะไรได้บ้าง

- ไวยากรณ์เป้าหมายระดับ Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `thread:<room>/<thread>`
- บัสสังเคราะห์ที่มี HTTP รองรับสำหรับฉีดข้อความขาเข้า จับทรานสคริปต์ขาออก สร้างเธรด รีแอ็กชัน แก้ไข ลบ และการทำงานค้นหา/อ่าน
- ตัวรันตรวจสอบตัวเองฝั่งโฮสต์ที่เขียนรายงาน Markdown ไปยัง `.artifacts/qa-e2e/`

## การกำหนดค่า

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

คีย์บัญชี:

- `enabled` — สวิตช์หลักสำหรับบัญชีนี้
- `name` — ป้ายชื่อที่แสดงผลแบบไม่บังคับ
- `baseUrl` — URL ของบัสสังเคราะห์
- `botUserId` — id ผู้ใช้บอตแบบ Matrix ที่ใช้ในไวยากรณ์เป้าหมาย
- `botDisplayName` — ชื่อที่แสดงสำหรับข้อความขาออก
- `pollTimeoutMs` — ช่วงเวลารอของ long-poll จำนวนเต็มระหว่าง 100 ถึง 30000
- `allowFrom` — รายการอนุญาตผู้ส่ง (id ผู้ใช้หรือ `"*"`)
- `defaultTo` — เป้าหมายสำรองเมื่อไม่ได้ระบุ
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` — การกั้นเครื่องมือแยกตามการทำงาน

คีย์หลายบัญชีที่ระดับบนสุด:

- `accounts` — เรคคอร์ดของการเขียนทับค่ารายบัญชีที่ตั้งชื่อไว้ โดยมีคีย์เป็น id บัญชี
- `defaultAccount` — id บัญชีที่ต้องการใช้เมื่อกำหนดค่าไว้หลายบัญชี

## ตัวรัน

การตรวจสอบตัวเองฝั่งโฮสต์ (เขียนรายงาน Markdown ใต้ `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

คำสั่งนี้จะส่งผ่าน `qa-lab` เริ่มบัส QA ภายในรีโป บูตสไลซ์รันไทม์ `qa-channel` ที่บันเดิลมาให้ และรันการตรวจสอบตัวเองแบบกำหนดซ้ำได้

ชุดสถานการณ์เต็มรูปแบบที่มีรีโปเป็นฐาน:

```bash
pnpm openclaw qa suite
```

รันสถานการณ์แบบขนานกับเลน QA gateway ดู [ภาพรวม QA](/th/concepts/qa-e2e-automation) สำหรับสถานการณ์ โปรไฟล์ และโหมดผู้ให้บริการ

ไซต์ QA ที่มี Docker รองรับ (Gateway + UI ดีบักเกอร์ QA Lab ในสแตกเดียว):

```bash
pnpm qa:lab:up
```

สร้างไซต์ QA เริ่มสแตก Gateway + QA Lab ที่มี Docker รองรับ และพิมพ์ URL ของ QA Lab จากตรงนั้นคุณสามารถเลือกสถานการณ์ เลือกเลนโมเดล เปิดการรันแต่ละรายการ และดูผลลัพธ์แบบสดได้ ดีบักเกอร์ QA Lab แยกจากบันเดิล Control UI ที่จัดส่ง
