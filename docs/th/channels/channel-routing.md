# ช่องทางและการกำหนดเส้นทาง

OpenClaw กำหนดเส้นทางการตอบกลับ **กลับไปยังช่องทางที่ข้อความมาจาก** ตัว
โมเดลไม่ได้เลือกช่องทาง การกำหนดเส้นทางเป็นแบบกำหนดแน่นอนและควบคุมโดยการกำหนดค่าของโฮสต์

## คำสำคัญ

- **ช่องทาง**: `telegram`, `whatsapp`, `discord`, `irc`, `googlechat`, `slack`, `signal`, `imessage`, `line` รวมถึงช่องทาง Plugin `webchat` คือช่องทาง WebChat UI ภายใน และไม่ใช่ช่องทางขาออกที่กำหนดค่าได้
- **AccountId**: อินสแตนซ์บัญชีต่อช่องทาง (เมื่อรองรับ)
- บัญชีเริ่มต้นของช่องทางแบบไม่บังคับ: `channels.<channel>.defaultAccount` เลือก
  ว่าจะใช้บัญชีใดเมื่อเส้นทางขาออกไม่ได้ระบุ `accountId`
  - ในการตั้งค่าหลายบัญชี ให้ตั้งค่าเริ่มต้นอย่างชัดเจน (`defaultAccount` หรือ `accounts.default`) เมื่อมีการกำหนดค่าบัญชีตั้งแต่สองบัญชีขึ้นไป หากไม่มี การกำหนดเส้นทางสำรองอาจเลือก ID บัญชีที่ถูกปรับรูปแบบแล้วตัวแรก
- **AgentId**: พื้นที่ทำงาน + ที่เก็บเซสชันที่แยกกัน (“สมอง”)
- **SessionKey**: คีย์บัคเก็ตที่ใช้เก็บบริบทและควบคุมการทำงานพร้อมกัน

## รูปแบบคีย์เซสชัน (ตัวอย่าง)

โดยค่าเริ่มต้น ข้อความโดยตรงจะถูกรวมเข้ากับเซสชัน **หลัก** ของเอเจนต์:

- `agent:<agentId>:<mainKey>` (ค่าเริ่มต้น: `agent:main:main`)

แม้เมื่อประวัติการสนทนาแบบข้อความโดยตรงถูกแชร์กับหลัก sandbox และ
นโยบายเครื่องมือจะใช้คีย์รันไทม์แชทโดยตรงต่อบัญชีที่สืบทอดมาสำหรับ DM ภายนอก
เพื่อไม่ให้ข้อความที่มาจากช่องทางถูกปฏิบัติเหมือนการรันเซสชันหลักในเครื่อง

กลุ่มและช่องทางยังคงแยกกันตามช่องทาง:

- กลุ่ม: `agent:<agentId>:<channel>:group:<id>`
- ช่องทาง/ห้อง: `agent:<agentId>:<channel>:channel:<id>`

เธรด:

- เธรด Slack/Discord ต่อท้าย `:thread:<threadId>` เข้ากับคีย์ฐาน
- หัวข้อฟอรัม Telegram ฝัง `:topic:<topicId>` ในคีย์กลุ่ม

ตัวอย่าง:

- `agent:main:telegram:group:-1001234567890:topic:42`
- `agent:main:discord:channel:123456:thread:987654`

## การปักหมุดเส้นทาง DM หลัก

เมื่อ `session.dmScope` เป็น `main` ข้อความโดยตรงอาจแชร์เซสชันหลักเดียวกัน
เพื่อป้องกันไม่ให้ `lastRoute` ของเซสชันถูกเขียนทับโดย DM ที่ไม่ใช่เจ้าของ
OpenClaw จะอนุมานเจ้าของที่ถูกปักหมุดจาก `allowFrom` เมื่อเงื่อนไขทั้งหมดนี้เป็นจริง:

- `allowFrom` มีรายการที่ไม่ใช่ไวลด์การ์ดเพียงรายการเดียวพอดี
- รายการนั้นสามารถปรับรูปแบบเป็น ID ผู้ส่งที่เป็นรูปธรรมสำหรับช่องทางนั้นได้
- ผู้ส่ง DM ขาเข้าไม่ตรงกับเจ้าของที่ถูกปักหมุดนั้น

ในกรณีที่ไม่ตรงกันนั้น OpenClaw ยังคงบันทึกเมตาดาต้าเซสชันขาเข้า แต่จะ
ข้ามการอัปเดต `lastRoute` ของเซสชันหลัก

## การบันทึกขาเข้าที่มีการป้องกัน

Plugin ช่องทางสามารถทำเครื่องหมายระเบียนเซสชันขาเข้าเป็น `createIfMissing: false`
เมื่อเส้นทางที่มีการป้องกันต้องไม่สร้างเซสชัน OpenClaw ใหม่ ในโหมดนั้น
OpenClaw อาจอัปเดตเมตาดาต้าและ `lastRoute` สำหรับเซสชันที่มีอยู่ แต่จะ
ไม่สร้างรายการเซสชันที่มีแต่เส้นทางเพียงเพราะพบเห็นข้อความหนึ่ง

## กฎการกำหนดเส้นทาง (วิธีเลือกเอเจนต์)

การกำหนดเส้นทางจะเลือก **หนึ่งเอเจนต์** สำหรับข้อความขาเข้าแต่ละรายการ:

1. **การตรงกันของเพียร์แบบตรงตัว** (`bindings` ที่มี `peer.kind` + `peer.id`)
2. **การตรงกันของเพียร์หลัก** (การสืบทอดของเธรด)
3. **การตรงกันของกิลด์ + บทบาท** (Discord) ผ่าน `guildId` + `roles`
4. **การตรงกันของกิลด์** (Discord) ผ่าน `guildId`
5. **การตรงกันของทีม** (Slack) ผ่าน `teamId`
6. **การตรงกันของบัญชี** (`accountId` บนช่องทาง)
7. **การตรงกันของช่องทาง** (บัญชีใดก็ได้บนช่องทางนั้น, `accountId: "*"`)
8. **เอเจนต์เริ่มต้น** (`agents.list[].default` มิฉะนั้นใช้รายการแรกในลิสต์ และสำรองเป็น `main`)

เมื่อ binding มีฟิลด์การจับคู่หลายฟิลด์ (`peer`, `guildId`, `teamId`, `roles`) **ฟิลด์ที่ระบุทั้งหมดต้องตรงกัน** เพื่อให้ binding นั้นมีผล

เอเจนต์ที่ตรงกันจะกำหนดว่าจะใช้พื้นที่ทำงานและที่เก็บเซสชันใด

## กลุ่มบรอดแคสต์ (รันหลายเอเจนต์)

กลุ่มบรอดแคสต์ช่วยให้คุณรัน **หลายเอเจนต์** สำหรับเพียร์เดียวกัน **เมื่อปกติแล้ว OpenClaw จะตอบกลับ** (ตัวอย่างเช่น: ในกลุ่ม WhatsApp หลังผ่านการ gating ด้วยการกล่าวถึง/การเปิดใช้งาน)

การกำหนดค่า:

```json5
{
  broadcast: {
    strategy: "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"],
    "+15555550123": ["support", "logger"],
  },
}
```

ดู: [กลุ่มบรอดแคสต์](/th/channels/broadcast-groups)

## ภาพรวมการกำหนดค่า

- `agents.list`: คำจำกัดความเอเจนต์ที่มีชื่อ (พื้นที่ทำงาน, โมเดล ฯลฯ)
- `bindings`: แมปช่องทาง/บัญชี/เพียร์ขาเข้าไปยังเอเจนต์

ตัวอย่าง:

```json5
{
  agents: {
    list: [{ id: "support", name: "Support", workspace: "~/.openclaw/workspace-support" }],
  },
  bindings: [
    { match: { channel: "slack", teamId: "T123" }, agentId: "support" },
    { match: { channel: "telegram", peer: { kind: "group", id: "-100123" } }, agentId: "support" },
  ],
}
```

## ที่เก็บเซสชัน

ที่เก็บเซสชันอยู่ใต้ไดเรกทอรีสถานะ (ค่าเริ่มต้น `~/.openclaw`):

- `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- ทรานสคริปต์ JSONL อยู่ข้างที่เก็บ

คุณสามารถแทนที่พาธที่เก็บผ่าน `session.store` และการทำเทมเพลต `{agentId}`

การค้นหาเซสชันของ Gateway และ ACP ยังสแกนที่เก็บเอเจนต์ที่อยู่บนดิสก์ภายใต้
ราก `agents/` เริ่มต้น และภายใต้ราก `session.store` ที่ใช้เทมเพลต ที่เก็บที่ค้นพบ
ต้องอยู่ภายในรากเอเจนต์ที่ resolve แล้วนั้น และใช้ไฟล์ `sessions.json`
ปกติ Symlink และพาธที่อยู่นอกรากจะถูกละเว้น

## พฤติกรรม WebChat

WebChat แนบเข้ากับ **เอเจนต์ที่เลือก** และค่าเริ่มต้นคือเซสชันหลักของเอเจนต์
ด้วยเหตุนี้ WebChat จึงช่วยให้คุณเห็นบริบทข้ามช่องทางของเอเจนต์นั้นได้ในที่เดียว

## บริบทการตอบกลับ

การตอบกลับขาเข้ามี:

- `ReplyToId`, `ReplyToBody` และ `ReplyToSender` เมื่อมีให้ใช้
- บริบทที่อ้างอิงจะถูกต่อท้ายเข้าใน `Body` เป็นบล็อก `[Replying to ...]`

สิ่งนี้สอดคล้องกันในทุกช่องทาง
