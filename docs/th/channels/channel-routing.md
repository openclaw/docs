# ช่องทางและการกำหนดเส้นทาง

OpenClaw จะกำหนดเส้นทางการตอบกลับ **กลับไปยังช่องทางที่ข้อความนั้นส่งมาจาก** โดยโมเดลจะไม่เลือกช่องทางเอง การกำหนดเส้นทางเป็นแบบกำหนดแน่นอนและควบคุมโดยการตั้งค่าโฮสต์

## คำสำคัญ

- **ช่องทาง**: `telegram`, `whatsapp`, `discord`, `irc`, `googlechat`, `slack`, `signal`, `imessage`, `line` รวมถึงช่องทางของ Plugin โดย `webchat` เป็นช่องทาง WebChat UI ภายใน และไม่ใช่ช่องทางขาออกที่ตั้งค่าได้
- **AccountId**: อินสแตนซ์บัญชีแยกตามช่องทาง (เมื่อรองรับ)
- ค่าเริ่มต้นของบัญชีในช่องทางแบบเลือกได้: `channels.<channel>.defaultAccount` ใช้เลือกว่าจะใช้บัญชีใดเมื่อเส้นทางขาออกไม่ได้ระบุ `accountId`
  - ในการตั้งค่าแบบหลายบัญชี ให้กำหนดค่าเริ่มต้นอย่างชัดเจน (`defaultAccount` หรือ `accounts.default`) เมื่อมีการตั้งค่าบัญชีตั้งแต่สองบัญชีขึ้นไป หากไม่กำหนด การกำหนดเส้นทางสำรองอาจเลือก Account ID แรกที่ถูกทำให้เป็นมาตรฐาน
- **AgentId**: พื้นที่ทำงาน + ที่เก็บเซสชันแบบแยกขาดจากกัน (“สมอง”)
- **SessionKey**: คีย์บัคเก็ตที่ใช้เก็บบริบทและควบคุมการทำงานพร้อมกัน

## รูปแบบคีย์เซสชัน (ตัวอย่าง)

ข้อความส่วนตัวจะถูกรวมเข้ากับเซสชัน **main** ของเอเจนต์โดยค่าเริ่มต้น:

- `agent:<agentId>:<mainKey>` (ค่าเริ่มต้น: `agent:main:main`)

แม้ว่าประวัติการสนทนาในข้อความส่วนตัวจะถูกใช้ร่วมกับ main แต่ sandbox และนโยบายเครื่องมือจะใช้คีย์รันไทม์ direct-chat แยกตามบัญชีที่สร้างต่อยอดสำหรับ DM ภายนอก เพื่อไม่ให้ข้อความที่มาจากช่องทางถูกปฏิบัติเสมือนการรันเซสชัน main ในเครื่อง

กลุ่มและช่องทางจะยังคงแยกกันตามแต่ละช่องทาง:

- กลุ่ม: `agent:<agentId>:<channel>:group:<id>`
- ช่อง/ห้อง: `agent:<agentId>:<channel>:channel:<id>`

เธรด:

- เธรดของ Slack/Discord จะเติม `:thread:<threadId>` ต่อท้ายคีย์ฐาน
- หัวข้อฟอรัมของ Telegram จะฝัง `:topic:<topicId>` ไว้ในคีย์กลุ่ม

ตัวอย่าง:

- `agent:main:telegram:group:-1001234567890:topic:42`
- `agent:main:discord:channel:123456:thread:987654`

## การปักหมุดเส้นทาง DM ของ main

เมื่อ `session.dmScope` เป็น `main` ข้อความส่วนตัวอาจใช้เซสชัน main ร่วมกันได้ เพื่อป้องกันไม่ให้ `lastRoute` ของเซสชันถูกเขียนทับโดย DM จากผู้ที่ไม่ใช่เจ้าของ OpenClaw จะอนุมานเจ้าของที่ถูกปักหมุดจาก `allowFrom` เมื่อเงื่อนไขทั้งหมดต่อไปนี้เป็นจริง:

- `allowFrom` มีรายการที่ไม่ใช่ไวลด์การ์ดอยู่เพียงรายการเดียว
- รายการนั้นสามารถทำให้เป็นมาตรฐานเป็น ID ผู้ส่งแบบเจาะจงสำหรับช่องทางนั้นได้
- ผู้ส่ง DM ขาเข้าไม่ตรงกับเจ้าของที่ถูกปักหมุดนั้น

ในกรณีที่ไม่ตรงกัน OpenClaw จะยังคงบันทึกเมทาดาทาของเซสชันขาเข้า แต่จะข้ามการอัปเดต `lastRoute` ของเซสชัน main

## กฎการกำหนดเส้นทาง (เลือกเอเจนต์อย่างไร)

การกำหนดเส้นทางจะเลือก **เอเจนต์หนึ่งตัว** สำหรับข้อความขาเข้าแต่ละข้อความ:

1. **ตรงกับ peer แบบตรงตัว** (`bindings` พร้อม `peer.kind` + `peer.id`)
2. **ตรงกับ peer แม่** (สืบทอดจากเธรด)
3. **ตรงกับ guild + roles** (Discord) ผ่าน `guildId` + `roles`
4. **ตรงกับ guild** (Discord) ผ่าน `guildId`
5. **ตรงกับทีม** (Slack) ผ่าน `teamId`
6. **ตรงกับบัญชี** (`accountId` บนช่องทาง)
7. **ตรงกับช่องทาง** (ทุกบัญชีบนช่องทางนั้น, `accountId: "*"` )
8. **เอเจนต์เริ่มต้น** (`agents.list[].default`, มิฉะนั้นใช้รายการแรก และสำรองเป็น `main`)

เมื่อ binding มีหลายฟิลด์สำหรับการจับคู่ (`peer`, `guildId`, `teamId`, `roles`) **ทุกฟิลด์ที่ระบุต้องตรงกันทั้งหมด** จึงจะใช้ binding นั้นได้

เอเจนต์ที่ตรงกันจะเป็นตัวกำหนดว่าจะใช้พื้นที่ทำงานและที่เก็บเซสชันใด

## กลุ่ม Broadcast (รันหลายเอเจนต์)

กลุ่ม Broadcast ช่วยให้คุณรัน **หลายเอเจนต์** สำหรับ peer เดียวกันได้ **เมื่อ OpenClaw ปกติจะตอบกลับอยู่แล้ว** (ตัวอย่างเช่น: ในกลุ่ม WhatsApp หลังจากผ่านเกณฑ์ mention/activation แล้ว)

การตั้งค่า:

```json5
{
  broadcast: {
    strategy: "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"],
    "+15555550123": ["support", "logger"],
  },
}
```

ดูเพิ่มเติม: [กลุ่ม Broadcast](/th/channels/broadcast-groups)

## ภาพรวมการตั้งค่า

- `agents.list`: คำนิยามเอเจนต์แบบมีชื่อ (workspace, model ฯลฯ)
- `bindings`: แมปช่องทาง/บัญชี/peer ขาเข้าไปยังเอเจนต์

ตัวอย่าง:

```json5
{
  agents: {
    list: [{ id: "support", name: "Support", workspace: "~/.openclaw/workspace-support" }],
  },
  bindings: [
    { match: { channel: "slack", teamId: "T123" }, agentId: "support" },
    { match: { channel: "telegram", peer: { kind: "group", id: "-100123" } }, agentId: "support" },
  ],
}
```

## ที่เก็บเซสชัน

ที่เก็บเซสชันจะอยู่ภายใต้ไดเรกทอรีสถานะ (ค่าเริ่มต้น `~/.openclaw`):

- `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- ทรานสคริปต์ JSONL จะอยู่ข้างๆ ที่เก็บนี้

คุณสามารถแทนที่พาธที่เก็บได้ผ่าน `session.store` และเทมเพลต `{agentId}`

การค้นหาเซสชันของ Gateway และ ACP จะสแกนที่เก็บเอเจนต์แบบ disk-backed ภายใต้ราก `agents/` ค่าเริ่มต้น และภายใต้ราก `session.store` ที่ใช้เทมเพลตด้วย ที่เก็บที่ค้นพบต้องอยู่ภายในรากเอเจนต์ที่ resolve แล้วนั้น และต้องใช้ไฟล์ `sessions.json` แบบไฟล์ปกติ โดยจะละเว้น symlink และพาธที่อยู่นอกราก

## พฤติกรรมของ WebChat

WebChat จะเชื่อมต่อกับ **เอเจนต์ที่เลือกไว้** และใช้เซสชัน main ของเอเจนต์นั้นโดยค่าเริ่มต้น ด้วยเหตุนี้ WebChat จึงช่วยให้คุณเห็นบริบทข้ามช่องทางของเอเจนต์นั้นได้ในที่เดียว

## บริบทการตอบกลับ

การตอบกลับขาเข้าจะมีข้อมูลดังนี้:

- `ReplyToId`, `ReplyToBody` และ `ReplyToSender` เมื่อมีข้อมูล
- บริบทที่อ้างอิงจะถูกต่อท้ายไปยัง `Body` เป็นบล็อก `[Replying to ...]`

ลักษณะนี้สอดคล้องกันในทุกช่องทาง
