“การจับคู่” คือขั้นตอนอนุมัติการเข้าถึงอย่างชัดเจนของ OpenClaw
ใช้ในสองตำแหน่ง:

1. **การจับคู่ DM** (ใครได้รับอนุญาตให้คุยกับบอต)
2. **การจับคู่ Node** (อุปกรณ์/Node ใดได้รับอนุญาตให้เข้าร่วมเครือข่าย Gateway)

บริบทด้านความปลอดภัย: [ความปลอดภัย](/th/gateway/security)

## 1) การจับคู่ DM (การเข้าถึงแชทขาเข้า)

เมื่อกำหนดค่าช่องด้วยนโยบาย DM `pairing` ผู้ส่งที่ไม่รู้จักจะได้รับรหัสสั้น และข้อความของพวกเขาจะ **ไม่ถูกประมวลผล** จนกว่าคุณจะอนุมัติ

นโยบาย DM เริ่มต้นมีเอกสารอยู่ที่: [ความปลอดภัย](/th/gateway/security)

`dmPolicy: "open"` จะเป็นสาธารณะก็ต่อเมื่อ allowlist ของ DM ที่มีผลรวม `"*"` เท่านั้น
การตั้งค่าและการตรวจสอบความถูกต้องต้องใช้ wildcard นั้นสำหรับคอนฟิกแบบ public-open หากสถานะที่มีอยู่
มี `open` พร้อมรายการ `allowFrom` ที่ระบุเจาะจง runtime จะยังคงอนุญาต
เฉพาะผู้ส่งเหล่านั้น และการอนุมัติใน pairing-store จะไม่ขยายการเข้าถึง `open`

รหัสจับคู่:

- 8 อักขระ ตัวพิมพ์ใหญ่ ไม่มีอักขระที่สับสนได้ง่าย (`0O1I`)
- **หมดอายุหลัง 1 ชั่วโมง** บอตจะส่งข้อความจับคู่เฉพาะเมื่อมีการสร้างคำขอใหม่ (ประมาณหนึ่งครั้งต่อชั่วโมงต่อผู้ส่ง)
- คำขอจับคู่ DM ที่รอดำเนินการถูกจำกัดไว้ที่ **3 รายการต่อช่อง** ตามค่าเริ่มต้น คำขอเพิ่มเติมจะถูกละเว้นจนกว่ารายการหนึ่งจะหมดอายุหรือได้รับการอนุมัติ

### อนุมัติผู้ส่ง

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

หากยังไม่ได้กำหนดค่าเจ้าของคำสั่ง การอนุมัติรหัสจับคู่ DM จะ bootstrap
`commands.ownerAllowFrom` ไปยังผู้ส่งที่ได้รับอนุมัติด้วย เช่น `telegram:123456789`
สิ่งนี้ทำให้การตั้งค่าครั้งแรกมีเจ้าของที่ชัดเจนสำหรับคำสั่งที่มีสิทธิ์สูงและพรอมป์อนุมัติ
exec หลังจากมีเจ้าของแล้ว การอนุมัติการจับคู่ครั้งถัดไปจะให้สิทธิ์เข้าถึง DM
เท่านั้น และจะไม่เพิ่มเจ้าของเพิ่มเติม

ช่องที่รองรับ: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### กลุ่มผู้ส่งที่ใช้ซ้ำได้

ใช้ `accessGroups` ระดับบนสุดเมื่อชุดผู้ส่งที่เชื่อถือได้ชุดเดียวกันควรใช้กับ
หลายช่องข้อความ หรือใช้กับทั้ง allowlist ของ DM และกลุ่ม

กลุ่มแบบคงที่ใช้ `type: "message.senders"` และถูกอ้างอิงด้วย
`accessGroup:<name>` จาก allowlist ของช่อง:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

กลุ่มการเข้าถึงมีเอกสารอย่างละเอียดที่นี่: [กลุ่มการเข้าถึง](/th/channels/access-groups)

### ตำแหน่งที่เก็บสถานะ

เก็บไว้ใต้ `~/.openclaw/credentials/`:

- คำขอที่รอดำเนินการ: `<channel>-pairing.json`
- ที่เก็บ allowlist ที่อนุมัติแล้ว:
  - บัญชีเริ่มต้น: `<channel>-allowFrom.json`
  - บัญชีที่ไม่ใช่ค่าเริ่มต้น: `<channel>-<accountId>-allowFrom.json`

พฤติกรรมการกำหนดขอบเขตตามบัญชี:

- บัญชีที่ไม่ใช่ค่าเริ่มต้นอ่าน/เขียนเฉพาะไฟล์ allowlist ตามขอบเขตของตน
- บัญชีเริ่มต้นใช้ไฟล์ allowlist แบบไม่กำหนดขอบเขตตามบัญชีของช่อง

ให้ถือว่าสิ่งเหล่านี้เป็นข้อมูลอ่อนไหว (เพราะเป็นตัวควบคุมการเข้าถึงผู้ช่วยของคุณ)

<Note>
ที่เก็บ allowlist สำหรับการจับคู่มีไว้สำหรับการเข้าถึง DM การอนุญาตของกลุ่มแยกต่างหาก
การอนุมัติรหัสจับคู่ DM จะไม่อนุญาตให้ผู้ส่งนั้นเรียกใช้คำสั่งกลุ่ม
หรือควบคุมบอตในกลุ่มโดยอัตโนมัติ การ bootstrap เจ้าของคนแรกเป็นสถานะคอนฟิกที่แยกต่างหาก
ใน `commands.ownerAllowFrom` และการส่งมอบแชทกลุ่มยังคงเป็นไปตาม
allowlist ของกลุ่มของช่อง (เช่น `groupAllowFrom`, `groups` หรือการแทนที่รายกลุ่ม
หรือรายหัวข้อ ขึ้นอยู่กับช่อง)
</Note>

## 2) การจับคู่อุปกรณ์ Node (iOS/Android/macOS/Node แบบ headless)

Node เชื่อมต่อกับ Gateway ในฐานะ **อุปกรณ์** ด้วย `role: node` Gateway
จะสร้างคำขอจับคู่อุปกรณ์ที่ต้องได้รับการอนุมัติ

### จับคู่ผ่าน Telegram (แนะนำสำหรับ iOS)

หากคุณใช้ Plugin `device-pair` คุณสามารถจับคู่อุปกรณ์ครั้งแรกทั้งหมดจาก Telegram ได้:

1. ใน Telegram ส่งข้อความหาบอตของคุณ: `/pair`
2. บอตตอบกลับด้วยสองข้อความ: ข้อความคำแนะนำและข้อความ **รหัสตั้งค่า** แยกต่างหาก (คัดลอก/วางใน Telegram ได้ง่าย)
3. บนโทรศัพท์ของคุณ เปิดแอป OpenClaw iOS → Settings → Gateway
4. สแกนคิวอาร์โค้ดหรือวางรหัสตั้งค่า แล้วเชื่อมต่อ
5. กลับไปที่ Telegram: `/pair pending` (ตรวจสอบ ID คำขอ บทบาท และขอบเขต) จากนั้นอนุมัติ

รหัสตั้งค่าเป็น payload JSON ที่เข้ารหัส base64 ซึ่งมี:

- `url`: URL WebSocket ของ Gateway (`ws://...` หรือ `wss://...`)
- `bootstrapToken`: โทเคน bootstrap แบบอุปกรณ์เดียวที่มีอายุสั้น ใช้สำหรับ handshake การจับคู่ครั้งแรก

โทเคน bootstrap นั้นมีโปรไฟล์ bootstrap สำหรับการจับคู่ในตัว:

- โทเคน `node` หลักที่ส่งต่อให้จะยังคงเป็น `scopes: []`
- โทเคน `operator` ใด ๆ ที่ส่งต่อให้จะยังถูกจำกัดอยู่ใน bootstrap allowlist:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- การตรวจสอบขอบเขต bootstrap มี prefix ตามบทบาท ไม่ใช่ pool ขอบเขตแบบแบนชุดเดียว:
  รายการขอบเขตของ operator ใช้ได้เฉพาะกับคำขอของ operator และบทบาทที่ไม่ใช่ operator
  ยังคงต้องขอขอบเขตใต้ prefix บทบาทของตนเอง
- การหมุนเวียน/เพิกถอนโทเคนในภายหลังยังคงถูกจำกัดด้วยทั้งสัญญาบทบาทที่อุปกรณ์ได้รับอนุมัติ
  และขอบเขต operator ของเซสชันผู้เรียก

ให้ปฏิบัติต่อรหัสตั้งค่าเหมือนรหัสผ่านในขณะที่ยังใช้ได้

สำหรับ Tailscale การจับคู่อุปกรณ์เคลื่อนที่แบบสาธารณะ หรือแบบรีโมตอื่น ๆ ให้ใช้ Tailscale Serve/Funnel
หรือ URL Gateway แบบ `wss://` อื่น รหัสตั้งค่าแบบ plaintext `ws://` จะยอมรับเฉพาะ
สำหรับ loopback, ที่อยู่ LAN ส่วนตัว, โฮสต์ Bonjour `.local` และโฮสต์
Android emulator เท่านั้น ที่อยู่ Tailnet CGNAT, ชื่อ `.ts.net` และโฮสต์สาธารณะยังคง
ปิดกั้นก่อนออก QR/รหัสตั้งค่า

### อนุมัติอุปกรณ์ Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

เมื่อการอนุมัติอย่างชัดเจนถูกปฏิเสธเพราะเซสชันอุปกรณ์ที่จับคู่ซึ่งใช้อนุมัติ
ถูกเปิดด้วยขอบเขตสำหรับการจับคู่เท่านั้น CLI จะลองคำขอเดิมอีกครั้งด้วย
`operator.admin` สิ่งนี้ช่วยให้อุปกรณ์ที่จับคู่แล้วซึ่งมีความสามารถระดับแอดมินกู้คืนการจับคู่
Control UI/เบราว์เซอร์ใหม่ได้โดยไม่ต้องแก้ไข `devices/paired.json` ด้วยมือ
Gateway ยังคงตรวจสอบการเชื่อมต่อที่ลองใหม่ โทเคนที่ไม่สามารถยืนยันตัวตน
ด้วย `operator.admin` จะยังถูกบล็อก

หากอุปกรณ์เดิมลองใหม่ด้วยรายละเอียดการยืนยันตัวตนที่แตกต่างกัน (เช่น
บทบาท/ขอบเขต/กุญแจสาธารณะที่แตกต่างกัน) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่และสร้าง
`requestId` ใหม่

<Note>
อุปกรณ์ที่จับคู่แล้วจะไม่ได้รับการเข้าถึงที่กว้างขึ้นโดยเงียบ ๆ หากเชื่อมต่อใหม่โดยขอขอบเขตมากขึ้นหรือบทบาทที่กว้างขึ้น OpenClaw จะคงการอนุมัติที่มีอยู่ไว้ตามเดิมและสร้างคำขออัปเกรดใหม่ที่รอดำเนินการ ใช้ `openclaw devices list` เพื่อเปรียบเทียบการเข้าถึงที่อนุมัติอยู่ในปัจจุบันกับการเข้าถึงที่ขอใหม่ก่อนที่คุณจะอนุมัติ
</Note>

### การอนุมัติอัตโนมัติของ Node จาก trusted-CIDR ที่เป็นตัวเลือก

การจับคู่อุปกรณ์ยังคงเป็นแบบแมนนวลตามค่าเริ่มต้น สำหรับเครือข่าย Node ที่ควบคุมอย่างเข้มงวด
คุณสามารถเลือกใช้การอนุมัติอัตโนมัติของ Node ครั้งแรกด้วย CIDR หรือ IP ที่ระบุชัดเจน:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

สิ่งนี้ใช้เฉพาะกับคำขอจับคู่ `role: node` ใหม่ที่ไม่มีการขอ
ขอบเขตเท่านั้น ไคลเอนต์ Operator, เบราว์เซอร์, Control UI และ WebChat ยังคงต้องได้รับการ
อนุมัติด้วยตนเอง การเปลี่ยนแปลงบทบาท ขอบเขต metadata และ public-key ยังคงต้องได้รับการ
อนุมัติด้วยตนเอง

### ที่เก็บสถานะการจับคู่ Node

เก็บไว้ใต้ `~/.openclaw/devices/`:

- `pending.json` (อายุสั้น; คำขอที่รอดำเนินการจะหมดอายุ)
- `paired.json` (อุปกรณ์ที่จับคู่แล้ว + โทเคน)

### หมายเหตุ

- API ดั้งเดิม `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) เป็น
  ที่เก็บการจับคู่แยกต่างหากที่ Gateway เป็นเจ้าของ Node แบบ WS ยังคงต้องใช้การจับคู่อุปกรณ์
- ระเบียนการจับคู่เป็นแหล่งความจริงถาวรสำหรับบทบาทที่อนุมัติ โทเคนอุปกรณ์ที่ใช้งานอยู่
  ยังคงถูกจำกัดอยู่ในชุดบทบาทที่อนุมัตินั้น รายการโทเคนที่หลุดออกไป
  นอกบทบาทที่อนุมัติจะไม่สร้างการเข้าถึงใหม่
