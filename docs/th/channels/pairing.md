"การจับคู่" คือขั้นตอนอนุมัติการเข้าถึงอย่างชัดเจนของ OpenClaw
ใช้ในสองตำแหน่ง:

1. **การจับคู่ DM** (ใครได้รับอนุญาตให้คุยกับบอท)
2. **การจับคู่ Node** (อุปกรณ์/โหนดใดได้รับอนุญาตให้เข้าร่วมเครือข่าย Gateway)

บริบทด้านความปลอดภัย: [ความปลอดภัย](/th/gateway/security)

## 1) การจับคู่ DM (การเข้าถึงแชทขาเข้า)

เมื่อกำหนดค่าช่องทางด้วยนโยบาย DM `pairing` ผู้ส่งที่ไม่รู้จักจะได้รับรหัสสั้น และข้อความของพวกเขาจะ **ไม่ถูกประมวลผล** จนกว่าคุณจะอนุมัติ

นโยบาย DM เริ่มต้นมีเอกสารอยู่ใน: [ความปลอดภัย](/th/gateway/security)

`dmPolicy: "open"` เป็นสาธารณะเฉพาะเมื่อรายการอนุญาต DM ที่มีผลรวม `"*"` อยู่ด้วย
การตั้งค่าและการตรวจสอบความถูกต้องต้องใช้ไวลด์การ์ดนั้นสำหรับค่ากำหนดแบบเปิดสาธารณะ หากสถานะเดิม
มี `open` พร้อมรายการ `allowFrom` ที่ระบุไว้ชัดเจน รันไทม์ยังคงรับเฉพาะ
ผู้ส่งเหล่านั้น และการอนุมัติในที่เก็บการจับคู่จะไม่ขยายการเข้าถึง `open`

รหัสการจับคู่:

- 8 ตัวอักษร ตัวพิมพ์ใหญ่ ไม่มีอักขระกำกวม (`0O1I`)
- **หมดอายุหลัง 1 ชั่วโมง** บอทจะส่งข้อความการจับคู่เฉพาะเมื่อสร้างคำขอใหม่ (โดยประมาณหนึ่งครั้งต่อชั่วโมงต่อผู้ส่ง)
- คำขอจับคู่ DM ที่รอดำเนินการถูกจำกัดไว้ที่ **3 รายการต่อช่องทาง** โดยค่าเริ่มต้น คำขอเพิ่มเติมจะถูกละเว้นจนกว่ารายการหนึ่งจะหมดอายุหรือได้รับการอนุมัติ

### อนุมัติผู้ส่ง

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

หากยังไม่ได้กำหนดค่าเจ้าของคำสั่ง การอนุมัติรหัสจับคู่ DM จะบูตสแตรป
`commands.ownerAllowFrom` ไปยังผู้ส่งที่ได้รับอนุมัติด้วย เช่น `telegram:123456789`
สิ่งนี้ทำให้การตั้งค่าครั้งแรกมีเจ้าของที่ชัดเจนสำหรับคำสั่งที่มีสิทธิ์สูงและพรอมป์อนุมัติ
การรันคำสั่ง หลังจากมีเจ้าของแล้ว การอนุมัติการจับคู่ภายหลังจะให้สิทธิ์เข้าถึง DM
เท่านั้น และจะไม่เพิ่มเจ้าของเพิ่มเติม

ช่องทางที่รองรับ: `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`

### กลุ่มผู้ส่งที่ใช้ซ้ำได้

ใช้ `accessGroups` ระดับบนเมื่อชุดผู้ส่งที่เชื่อถือได้ชุดเดียวกันควรใช้กับ
หลายช่องทางข้อความ หรือใช้กับทั้งรายการอนุญาต DM และกลุ่ม

กลุ่มแบบคงที่ใช้ `type: "message.senders"` และอ้างอิงด้วย
`accessGroup:<name>` จากรายการอนุญาตของช่องทาง:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

กลุ่มการเข้าถึงมีเอกสารอย่างละเอียดที่นี่: [กลุ่มการเข้าถึง](/th/channels/access-groups)

### สถานะถูกเก็บไว้ที่ใด

เก็บไว้ใต้ `~/.openclaw/credentials/`:

- คำขอที่รอดำเนินการ: `<channel>-pairing.json`
- ที่เก็บรายการอนุญาตที่ได้รับอนุมัติ:
  - บัญชีเริ่มต้น: `<channel>-allowFrom.json`
  - บัญชีที่ไม่ใช่ค่าเริ่มต้น: `<channel>-<accountId>-allowFrom.json`

พฤติกรรมการกำหนดขอบเขตบัญชี:

- บัญชีที่ไม่ใช่ค่าเริ่มต้นอ่าน/เขียนเฉพาะไฟล์รายการอนุญาตตามขอบเขตของตน
- บัญชีเริ่มต้นใช้ไฟล์รายการอนุญาตแบบไม่มีขอบเขตระดับช่องทาง

ให้ถือว่าสิ่งเหล่านี้เป็นข้อมูลอ่อนไหว (เพราะควบคุมการเข้าถึงผู้ช่วยของคุณ)

<Note>
ที่เก็บรายการอนุญาตการจับคู่ใช้สำหรับการเข้าถึง DM การอนุญาตกลุ่มแยกต่างหาก
การอนุมัติรหัสจับคู่ DM ไม่ได้อนุญาตให้ผู้ส่งนั้นรันคำสั่งกลุ่ม
หรือควบคุมบอทในกลุ่มโดยอัตโนมัติ การบูตสแตรปเจ้าของคนแรกเป็นสถานะค่ากำหนด
แยกต่างหากใน `commands.ownerAllowFrom` และการส่งในแชทกลุ่มยังคงเป็นไปตาม
รายการอนุญาตกลุ่มของช่องทาง (เช่น `groupAllowFrom`, `groups` หรือการแทนที่ต่อกลุ่ม
หรือต่อหัวข้อ ขึ้นอยู่กับช่องทาง)
</Note>

## 2) การจับคู่อุปกรณ์ Node (Node บน iOS/Android/macOS/แบบไม่มีส่วนหัว)

Node เชื่อมต่อกับ Gateway ในฐานะ **อุปกรณ์** ที่มี `role: node` Gateway
จะสร้างคำขอจับคู่อุปกรณ์ที่ต้องได้รับการอนุมัติ

### จับคู่ผ่าน Telegram (แนะนำสำหรับ iOS)

หากคุณใช้ Plugin `device-pair` คุณสามารถจับคู่อุปกรณ์ครั้งแรกทั้งหมดจาก Telegram ได้:

1. ใน Telegram ส่งข้อความถึงบอทของคุณ: `/pair`
2. บอทตอบกลับด้วยสองข้อความ: ข้อความคำแนะนำและข้อความ **รหัสตั้งค่า** แยกต่างหาก (คัดลอก/วางใน Telegram ได้ง่าย)
3. บนโทรศัพท์ของคุณ เปิดแอป OpenClaw iOS → Settings → Gateway
4. สแกนรหัส QR หรือวางรหัสตั้งค่าแล้วเชื่อมต่อ
5. กลับไปที่ Telegram: `/pair pending` (ตรวจสอบรหัสคำขอ บทบาท และขอบเขต) จากนั้นอนุมัติ

รหัสตั้งค่าเป็นเพย์โหลด JSON ที่เข้ารหัสแบบ base64 ซึ่งมี:

- `url`: URL WebSocket ของ Gateway (`ws://...` หรือ `wss://...`)
- `bootstrapToken`: โทเค็นบูตสแตรปอุปกรณ์เดียวอายุสั้นที่ใช้สำหรับแฮนด์เชกการจับคู่ครั้งแรก

โทเค็นบูตสแตรปนั้นมีโปรไฟล์บูตสแตรปการจับคู่ในตัว:

- โปรไฟล์ตั้งค่าในตัวอนุญาตเฉพาะพื้นฐาน QR/รหัสตั้งค่าใหม่:
  `node` พร้อมการส่งต่อ `operator` แบบมีขอบเขต
- โทเค็น `node` ที่ส่งต่อยังคงเป็น `scopes: []`
- โทเค็น `operator` ที่ส่งต่อถูกจำกัดไว้ที่ `operator.approvals`,
  `operator.read`, `operator.talk.secrets` และ `operator.write`
- `operator.admin` ไม่ได้รับจากบูตสแตรป QR/รหัสตั้งค่า ต้องใช้
  การจับคู่ operator หรือโฟลว์โทเค็นที่ได้รับอนุมัติแยกต่างหาก
- การหมุนเวียน/เพิกถอนโทเค็นภายหลังยังคงถูกจำกัดทั้งด้วยสัญญาบทบาทที่ได้รับอนุมัติ
  ของอุปกรณ์และขอบเขต operator ของเซสชันผู้เรียก

ให้ถือว่ารหัสตั้งค่าเหมือนรหัสผ่านในขณะที่ยังใช้ได้

สำหรับ Tailscale, สาธารณะ หรือการจับคู่มือถือระยะไกลแบบอื่น ให้ใช้ Tailscale Serve/Funnel
หรือ URL Gateway แบบ `wss://` อื่น รหัสตั้งค่าแบบข้อความธรรมดา `ws://` จะยอมรับเฉพาะ
สำหรับลูปแบ็ก, ที่อยู่ LAN ส่วนตัว, โฮสต์ Bonjour `.local` และโฮสต์อีมูเลเตอร์
Android ที่อยู่ Tailnet CGNAT, ชื่อ `.ts.net` และโฮสต์สาธารณะยังคง
ปิดกั้นก่อนการออก QR/รหัสตั้งค่า

### อนุมัติอุปกรณ์ Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

เมื่อการอนุมัติแบบชัดเจนถูกปฏิเสธเพราะเซสชันอุปกรณ์ที่จับคู่ซึ่งกำลังอนุมัติ
ถูกเปิดด้วยขอบเขตเฉพาะการจับคู่ CLI จะลองคำขอเดียวกันอีกครั้งด้วย
`operator.admin` สิ่งนี้ช่วยให้อุปกรณ์ที่จับคู่แล้วซึ่งมีความสามารถระดับผู้ดูแลสามารถกู้คืนการจับคู่
Control UI/เบราว์เซอร์ใหม่ได้โดยไม่ต้องแก้ไข `devices/paired.json` ด้วยมือ
Gateway ยังคงตรวจสอบการเชื่อมต่อที่ลองใหม่ โทเค็นที่ไม่สามารถรับรองตัวตน
ด้วย `operator.admin` ยังคงถูกบล็อก

หากอุปกรณ์เดียวกันลองใหม่ด้วยรายละเอียดการรับรองตัวตนที่แตกต่างกัน (เช่น
บทบาท/ขอบเขต/กุญแจสาธารณะที่แตกต่างกัน) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่ และจะสร้าง
`requestId` ใหม่

<Note>
อุปกรณ์ที่จับคู่แล้วจะไม่ได้รับสิทธิ์เข้าถึงที่กว้างขึ้นอย่างเงียบ ๆ หากเชื่อมต่อใหม่โดยขอขอบเขตเพิ่มเติมหรือบทบาทที่กว้างกว่า OpenClaw จะคงการอนุมัติเดิมไว้ตามเดิมและสร้างคำขออัปเกรดใหม่ที่รอดำเนินการ ใช้ `openclaw devices list` เพื่อเปรียบเทียบสิทธิ์เข้าถึงที่ได้รับอนุมัติในปัจจุบันกับสิทธิ์เข้าถึงใหม่ที่ขอก่อนที่คุณจะอนุมัติ
</Note>

### การอนุมัติอัตโนมัติ Node ตาม CIDR ที่เชื่อถือได้แบบเลือกใช้ได้

การจับคู่อุปกรณ์ยังคงเป็นแบบแมนนวลโดยค่าเริ่มต้น สำหรับเครือข่าย Node ที่ควบคุมอย่างเข้มงวด
คุณสามารถเลือกใช้การอนุมัติอัตโนมัติของ Node ครั้งแรกด้วย CIDR หรือ IP ที่ระบุชัดเจน:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

สิ่งนี้ใช้เฉพาะกับคำขอจับคู่ `role: node` ใหม่ที่ไม่มี
ขอบเขตที่ร้องขอ ไคลเอนต์ operator, เบราว์เซอร์, Control UI และ WebChat ยังคงต้องได้รับการ
อนุมัติแบบแมนนวล การเปลี่ยนแปลงบทบาท ขอบเขต เมตาดาตา และกุญแจสาธารณะยังคงต้องได้รับการ
อนุมัติแบบแมนนวล

### การจัดเก็บสถานะการจับคู่ Node

เก็บไว้ใต้ `~/.openclaw/devices/`:

- `pending.json` (อายุสั้น; คำขอที่รอดำเนินการจะหมดอายุ)
- `paired.json` (อุปกรณ์ที่จับคู่แล้ว + โทเค็น)

### หมายเหตุ

- API เดิม `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) เป็น
  ที่เก็บการจับคู่แยกต่างหากที่ Gateway เป็นเจ้าของ Node แบบ WS ยังคงต้องใช้การจับคู่อุปกรณ์
- ระเบียนการจับคู่เป็นแหล่งความจริงถาวรสำหรับบทบาทที่ได้รับอนุมัติ โทเค็นอุปกรณ์
  ที่ใช้งานอยู่ยังคงถูกจำกัดไว้กับชุดบทบาทที่ได้รับอนุมัตินั้น รายการโทเค็นหลงเหลือ
  นอกบทบาทที่ได้รับอนุมัติจะไม่สร้างการเข้าถึงใหม่
