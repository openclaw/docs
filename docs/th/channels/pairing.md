“การจับคู่” คือขั้นตอนอนุมัติการเข้าถึงอย่างชัดเจนของ OpenClaw.
ใช้ในสองตำแหน่ง:

1. **การจับคู่ DM** (ใครได้รับอนุญาตให้คุยกับบอต)
2. **การจับคู่ Node** (อุปกรณ์/Node ใดได้รับอนุญาตให้เข้าร่วมเครือข่าย Gateway)

บริบทด้านความปลอดภัย: [ความปลอดภัย](/th/gateway/security)

## 1) การจับคู่ DM (การเข้าถึงแชทขาเข้า)

เมื่อกำหนดค่าช่องทางด้วยนโยบาย DM `pairing` ผู้ส่งที่ไม่รู้จักจะได้รับโค้ดสั้น และข้อความของพวกเขาจะ **ไม่ถูกประมวลผล** จนกว่าคุณจะอนุมัติ.

นโยบาย DM เริ่มต้นมีเอกสารอยู่ที่: [ความปลอดภัย](/th/gateway/security)

`dmPolicy: "open"` จะเป็นสาธารณะเฉพาะเมื่อ allowlist DM ที่มีผลมี `"*"`.
การตั้งค่าและการตรวจสอบความถูกต้องต้องใช้ไวลด์การ์ดนั้นสำหรับค่ากำหนด public-open. หากสถานะที่มีอยู่
มี `open` พร้อมรายการ `allowFrom` แบบเจาะจง runtime จะยังคงอนุญาตเฉพาะ
ผู้ส่งเหล่านั้น และการอนุมัติใน pairing-store จะไม่ขยายการเข้าถึงแบบ `open`.

โค้ดการจับคู่:

- 8 อักขระ ตัวพิมพ์ใหญ่ ไม่มีอักขระที่สับสนได้ (`0O1I`).
- **หมดอายุหลัง 1 ชั่วโมง**. บอตจะส่งข้อความการจับคู่เฉพาะเมื่อมีการสร้างคำขอใหม่ (ประมาณหนึ่งครั้งต่อชั่วโมงต่อผู้ส่ง).
- คำขอจับคู่ DM ที่รอดำเนินการถูกจำกัดไว้ที่ **3 ต่อช่องทาง** โดยค่าเริ่มต้น; คำขอเพิ่มเติมจะถูกละเว้นจนกว่ารายการหนึ่งจะหมดอายุหรือได้รับอนุมัติ.

### อนุมัติผู้ส่ง

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

หากยังไม่ได้กำหนดค่าเจ้าของคำสั่ง การอนุมัติโค้ดการจับคู่ DM จะเริ่มตั้งค่า
`commands.ownerAllowFrom` เป็นผู้ส่งที่ได้รับอนุมัติด้วย เช่น `telegram:123456789`.
สิ่งนี้ทำให้การตั้งค่าครั้งแรกมีเจ้าของที่ชัดเจนสำหรับคำสั่งที่มีสิทธิ์สูงและพรอมป์อนุมัติ
การ exec. หลังจากมีเจ้าของแล้ว การอนุมัติการจับคู่ภายหลังจะให้เฉพาะการเข้าถึง DM;
จะไม่เพิ่มเจ้าของเพิ่มเติม.

ช่องทางที่รองรับ: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### กลุ่มผู้ส่งที่ใช้ซ้ำได้

ใช้ `accessGroups` ระดับบนเมื่อชุดผู้ส่งที่เชื่อถือได้ชุดเดียวกันควรใช้กับ
หลายช่องทางข้อความ หรือใช้กับทั้ง allowlist ของ DM และกลุ่ม.

กลุ่มแบบคงที่ใช้ `type: "message.senders"` และอ้างอิงด้วย
`accessGroup:<name>` จาก allowlist ของช่องทาง:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

มีเอกสารรายละเอียดเกี่ยวกับกลุ่มการเข้าถึงอยู่ที่นี่: [กลุ่มการเข้าถึง](/th/channels/access-groups)

### ตำแหน่งที่เก็บสถานะ

จัดเก็บภายใต้ `~/.openclaw/credentials/`:

- คำขอที่รอดำเนินการ: `<channel>-pairing.json`
- ที่เก็บ allowlist ที่อนุมัติแล้ว:
  - บัญชีเริ่มต้น: `<channel>-allowFrom.json`
  - บัญชีที่ไม่ใช่ค่าเริ่มต้น: `<channel>-<accountId>-allowFrom.json`

พฤติกรรมการกำหนดขอบเขตบัญชี:

- บัญชีที่ไม่ใช่ค่าเริ่มต้นอ่าน/เขียนเฉพาะไฟล์ allowlist ที่อยู่ในขอบเขตของตน.
- บัญชีเริ่มต้นใช้ไฟล์ allowlist แบบไม่กำหนดขอบเขตที่อิงตามช่องทาง.

ถือว่าสิ่งเหล่านี้เป็นข้อมูลอ่อนไหว (เพราะควบคุมการเข้าถึงผู้ช่วยของคุณ).

<Note>
ที่เก็บ allowlist สำหรับการจับคู่มีไว้สำหรับการเข้าถึง DM. การอนุญาตของกลุ่มแยกต่างหาก.
การอนุมัติโค้ดการจับคู่ DM จะไม่อนุญาตให้ผู้ส่งนั้นเรียกใช้คำสั่งกลุ่ม
หรือควบคุมบอตในกลุ่มโดยอัตโนมัติ. การเริ่มตั้งค่าเจ้าของคนแรกเป็นสถานะ config
แยกต่างหากใน `commands.ownerAllowFrom` และการส่งมอบแชทกลุ่มยังคงเป็นไปตาม
allowlist ของกลุ่มในช่องทางนั้น (เช่น `groupAllowFrom`, `groups` หรือการ override
รายกลุ่มหรือรายหัวข้อ ขึ้นอยู่กับช่องทาง).
</Note>

## 2) การจับคู่อุปกรณ์ Node (Node บน iOS/Android/macOS/แบบ headless)

Node เชื่อมต่อกับ Gateway ในฐานะ **อุปกรณ์** ด้วย `role: node`. Gateway
จะสร้างคำขอจับคู่อุปกรณ์ที่ต้องได้รับการอนุมัติ.

### จับคู่ผ่าน Telegram (แนะนำสำหรับ iOS)

หากคุณใช้ Plugin `device-pair` คุณสามารถทำการจับคู่อุปกรณ์ครั้งแรกทั้งหมดจาก Telegram ได้:

1. ใน Telegram ให้ส่งข้อความถึงบอตของคุณ: `/pair`
2. บอตตอบกลับด้วยสองข้อความ: ข้อความคำแนะนำ และข้อความ **โค้ดตั้งค่า** แยกต่างหาก (คัดลอก/วางใน Telegram ได้ง่าย).
3. บนโทรศัพท์ของคุณ เปิดแอป OpenClaw iOS → Settings → Gateway.
4. สแกนคิวอาร์โค้ดหรือวางโค้ดตั้งค่าแล้วเชื่อมต่อ.
5. กลับไปที่ Telegram: `/pair pending` (ตรวจสอบรหัสคำขอ บทบาท และขอบเขต) จากนั้นอนุมัติ.

โค้ดตั้งค่าเป็น payload JSON ที่เข้ารหัส base64 ซึ่งมี:

- `url`: URL WebSocket ของ Gateway (`ws://...` หรือ `wss://...`)
- `bootstrapToken`: โทเค็น bootstrap อายุสั้นสำหรับอุปกรณ์เดียวที่ใช้ในการ handshake การจับคู่ครั้งแรก

โทเค็น bootstrap นั้นมีโปรไฟล์ bootstrap การจับคู่ในตัว:

- โทเค็น `node` หลักที่ส่งต่อยังคงเป็น `scopes: []`
- โทเค็น `operator` ใด ๆ ที่ส่งต่อยังคงถูกจำกัดไว้กับ allowlist ของ bootstrap:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- การตรวจสอบขอบเขต bootstrap ใช้คำนำหน้าตามบทบาท ไม่ใช่พูลขอบเขตแบบแบนชุดเดียว:
  รายการขอบเขตของ operator จะตอบสนองเฉพาะคำขอของ operator และบทบาทที่ไม่ใช่ operator
  ยังคงต้องร้องขอขอบเขตภายใต้คำนำหน้าบทบาทของตนเอง
- การหมุนเวียน/เพิกถอนโทเค็นในภายหลังยังคงถูกจำกัดโดยทั้งสัญญาบทบาทที่ได้รับอนุมัติของอุปกรณ์
  และขอบเขต operator ของเซสชันผู้เรียก

ถือว่าโค้ดตั้งค่าเหมือนรหัสผ่านขณะที่ยังใช้ได้.

สำหรับ Tailscale, สาธารณะ หรือการจับคู่มือถือแบบ non-loopback อื่น ๆ ให้ใช้ Tailscale
Serve/Funnel หรือ URL Gateway `wss://` อื่น. URL ตั้งค่า `ws://` แบบ direct non-loopback
จะถูกปฏิเสธก่อนออก QR/โค้ดตั้งค่า. โค้ดตั้งค่า `ws://` แบบ plaintext
จำกัดไว้เฉพาะ URL loopback; ไคลเอนต์ `ws://` บนเครือข่ายส่วนตัวยังคงต้องใช้
break-glass `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` อย่างชัดเจนตามที่อธิบายในคู่มือ
Gateway ระยะไกล.

### อนุมัติอุปกรณ์ Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

เมื่อการอนุมัติแบบชัดเจนถูกปฏิเสธเพราะเซสชันอุปกรณ์ที่จับคู่แล้วซึ่งใช้อนุมัติ
ถูกเปิดด้วยขอบเขตเฉพาะการจับคู่ CLI จะลองคำขอเดิมอีกครั้งด้วย
`operator.admin`. สิ่งนี้ช่วยให้อุปกรณ์ที่จับคู่แล้วซึ่งมีความสามารถระดับ admin สามารถกู้คืนการจับคู่
Control UI/เบราว์เซอร์ใหม่ได้โดยไม่ต้องแก้ไข `devices/paired.json` ด้วยมือ. Gateway
ยังคงตรวจสอบการเชื่อมต่อที่ลองใหม่; โทเค็นที่ไม่สามารถยืนยันตัวตนด้วย
`operator.admin` จะยังคงถูกบล็อก.

หากอุปกรณ์เดียวกันลองใหม่ด้วยรายละเอียด auth ที่ต่างออกไป (เช่น
บทบาท/ขอบเขต/คีย์สาธารณะที่ต่างกัน) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่ และมีการสร้าง
`requestId` ใหม่.

<Note>
อุปกรณ์ที่จับคู่แล้วจะไม่ได้รับการเข้าถึงที่กว้างขึ้นแบบเงียบ ๆ. หากเชื่อมต่อใหม่โดยขอขอบเขตเพิ่มหรือบทบาทที่กว้างขึ้น OpenClaw จะคงการอนุมัติเดิมไว้ตามเดิมและสร้างคำขออัปเกรดที่รอดำเนินการใหม่. ใช้ `openclaw devices list` เพื่อเปรียบเทียบการเข้าถึงที่อนุมัติอยู่ปัจจุบันกับการเข้าถึงที่ขอใหม่ก่อนที่คุณจะอนุมัติ.
</Note>

### การอนุมัติอัตโนมัติสำหรับ Node แบบ trusted-CIDR ที่เป็นทางเลือก

การจับคู่อุปกรณ์ยังคงเป็นแบบ manual โดยค่าเริ่มต้น. สำหรับเครือข่าย Node ที่ควบคุมอย่างเข้มงวด
คุณสามารถเลือกใช้การอนุมัติ Node ครั้งแรกโดยอัตโนมัติด้วย CIDR หรือ IP แบบเจาะจงที่ระบุชัดเจน:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

สิ่งนี้ใช้เฉพาะกับคำขอจับคู่ `role: node` ใหม่ที่ไม่มีขอบเขตที่ร้องขอ.
ไคลเอนต์ Operator, เบราว์เซอร์, Control UI และ WebChat ยังคงต้องได้รับการอนุมัติ
ด้วยตนเอง. การเปลี่ยนแปลงบทบาท ขอบเขต metadata และคีย์สาธารณะยังคงต้องได้รับการอนุมัติ
ด้วยตนเอง.

### ที่เก็บสถานะการจับคู่ Node

จัดเก็บภายใต้ `~/.openclaw/devices/`:

- `pending.json` (อายุสั้น; คำขอที่รอดำเนินการจะหมดอายุ)
- `paired.json` (อุปกรณ์ที่จับคู่แล้ว + โทเค็น)

### หมายเหตุ

- API เดิม `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) เป็น
  ที่เก็บการจับคู่ที่ Gateway เป็นเจ้าของแบบแยกต่างหาก. WS Node ยังคงต้องใช้การจับคู่อุปกรณ์.
- ระเบียนการจับคู่คือแหล่งความจริงถาวรสำหรับบทบาทที่ได้รับอนุมัติ. โทเค็นอุปกรณ์ที่ใช้งานอยู่
  ยังคงถูกจำกัดไว้กับชุดบทบาทที่ได้รับอนุมัตินั้น; รายการโทเค็นที่หลุดอยู่นอก
  บทบาทที่ได้รับอนุมัติจะไม่สร้างการเข้าถึงใหม่.
