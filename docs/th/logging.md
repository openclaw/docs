OpenClaw มีพื้นผิวบันทึกหลักสองแบบ:

- **บันทึกไฟล์** (JSON lines) ที่เขียนโดย Gateway
- **เอาต์พุตคอนโซล** ที่แสดงในเทอร์มินัลและ Gateway Debug UI

แท็บ **Logs** ใน Control UI จะติดตามบันทึกไฟล์ของ gateway หน้านี้อธิบายว่าบันทึก
อยู่ที่ไหน วิธีอ่านบันทึก และวิธีกำหนดค่าระดับและรูปแบบบันทึก

## ตำแหน่งที่เก็บบันทึก

โดยค่าเริ่มต้น Gateway จะเขียนไฟล์บันทึกแบบหมุนเวียนไว้ใต้:

`/tmp/openclaw/openclaw-YYYY-MM-DD.log`

วันที่ใช้เขตเวลาท้องถิ่นของโฮสต์ gateway

แต่ละไฟล์จะหมุนเวียนเมื่อมีขนาดถึง `logging.maxFileBytes` (ค่าเริ่มต้น: 100 MB)
OpenClaw เก็บไฟล์เก็บถาวรแบบมีหมายเลขไว้ข้างไฟล์ที่ใช้งานอยู่ได้สูงสุดห้าไฟล์ เช่น
`openclaw-YYYY-MM-DD.1.log` และเขียนต่อไปยังบันทึกที่ใช้งานอยู่ไฟล์ใหม่แทนการ
ระงับข้อมูลวินิจฉัย

คุณสามารถแทนที่ค่านี้ได้ใน `~/.openclaw/openclaw.json`:

```json
{
  "logging": {
    "file": "/path/to/openclaw.log"
  }
}
```

## วิธีอ่านบันทึก

### CLI: ติดตามสด (แนะนำ)

ใช้ CLI เพื่อติดตามไฟล์บันทึก gateway ผ่าน RPC:

```bash
openclaw logs --follow
```

ตัวเลือกปัจจุบันที่มีประโยชน์:

- `--local-time`: แสดงเวลาในเขตเวลาท้องถิ่นของคุณ
- `--url <url>` / `--token <token>` / `--timeout <ms>`: แฟล็ก Gateway RPC มาตรฐาน
- `--expect-final`: แฟล็กรอคำตอบสุดท้ายของ RPC ที่มี agent รองรับ (ยอมรับที่นี่ผ่านเลเยอร์ไคลเอนต์ร่วม)

โหมดเอาต์พุต:

- **เซสชัน TTY**: บรรทัดบันทึกแบบสวยงาม มีสี และมีโครงสร้าง
- **เซสชันที่ไม่ใช่ TTY**: ข้อความล้วน
- `--json`: JSON แบบแยกบรรทัด (หนึ่งเหตุการณ์บันทึกต่อหนึ่งบรรทัด)
- `--plain`: บังคับใช้ข้อความล้วนในเซสชัน TTY
- `--no-color`: ปิดใช้งานสี ANSI

เมื่อคุณส่ง `--url` แบบชัดเจน CLI จะไม่ใช้การกำหนดค่าหรือข้อมูลรับรองจาก
สภาพแวดล้อมโดยอัตโนมัติ ให้ใส่ `--token` เองหาก Gateway เป้าหมาย
ต้องการการยืนยันตัวตน

ในโหมด JSON, CLI จะปล่อยอ็อบเจกต์ที่ติดแท็ก `type`:

- `meta`: เมทาดาทาของสตรีม (ไฟล์, เคอร์เซอร์, ขนาด)
- `log`: รายการบันทึกที่แยกวิเคราะห์แล้ว
- `notice`: คำใบ้การตัดทอน / การหมุนเวียน
- `raw`: บรรทัดบันทึกที่ยังไม่ได้แยกวิเคราะห์

หาก Gateway local loopback โดยนัยขอการจับคู่ ปิดระหว่างเชื่อมต่อ
หรือหมดเวลาก่อนที่ `logs.tail` จะตอบ `openclaw logs` จะถอยกลับไปใช้
บันทึกไฟล์ Gateway ที่กำหนดค่าไว้โดยอัตโนมัติ เป้าหมาย `--url` แบบชัดเจนจะไม่ใช้
การถอยกลับนี้

หากติดต่อ Gateway ไม่ได้ CLI จะแสดงคำใบ้สั้น ๆ ให้เรียกใช้:

```bash
openclaw doctor
```

### Control UI (เว็บ)

แท็บ **Logs** ของ Control UI จะติดตามไฟล์เดียวกันโดยใช้ `logs.tail`
ดู [/web/control-ui](/th/web/control-ui) สำหรับวิธีเปิดใช้งาน

### บันทึกเฉพาะช่องทาง

หากต้องการกรองกิจกรรมช่องทาง (WhatsApp/Telegram/ฯลฯ) ให้ใช้:

```bash
openclaw channels logs --channel whatsapp
```

## รูปแบบบันทึก

### บันทึกไฟล์ (JSONL)

แต่ละบรรทัดในไฟล์บันทึกเป็นอ็อบเจกต์ JSON CLI และ Control UI จะแยกวิเคราะห์
รายการเหล่านี้เพื่อแสดงเอาต์พุตแบบมีโครงสร้าง (เวลา, ระดับ, ระบบย่อย, ข้อความ)

ระเบียน JSONL ของบันทึกไฟล์ยังมีฟิลด์ระดับบนสุดที่เครื่องกรองได้เมื่อ
พร้อมใช้งาน:

- `hostname`: ชื่อโฮสต์ gateway
- `message`: ข้อความบันทึกแบบแบนสำหรับการค้นหาข้อความเต็ม
- `agent_id`: id ของ agent ที่ใช้งานอยู่เมื่อการเรียกบันทึกมีบริบท agent
- `session_id`: id/key ของเซสชันที่ใช้งานอยู่เมื่อการเรียกบันทึกมีบริบทเซสชัน
- `channel`: ช่องทางที่ใช้งานอยู่เมื่อการเรียกบันทึกมีบริบทช่องทาง

OpenClaw เก็บอาร์กิวเมนต์บันทึกแบบมีโครงสร้างต้นฉบับไว้ข้างฟิลด์เหล่านี้
เพื่อให้ตัวแยกวิเคราะห์เดิมที่อ่านคีย์อาร์กิวเมนต์ tslog แบบมีหมายเลขยังทำงานได้

### เอาต์พุตคอนโซล

บันทึกคอนโซล **รับรู้ TTY** และจัดรูปแบบเพื่อให้อ่านง่าย:

- คำนำหน้าระบบย่อย (เช่น `gateway/channels/whatsapp`)
- การใส่สีระดับ (info/warn/error)
- โหมดแบบย่อหรือ JSON ที่เป็นตัวเลือก

การจัดรูปแบบคอนโซลควบคุมโดย `logging.consoleStyle`

### บันทึก WebSocket ของ Gateway

`openclaw gateway` ยังมีการบันทึกโปรโตคอล WebSocket สำหรับทราฟฟิก RPC:

- โหมดปกติ: เฉพาะผลลัพธ์ที่น่าสนใจ (ข้อผิดพลาด, ข้อผิดพลาดการแยกวิเคราะห์, การเรียกที่ช้า)
- `--verbose`: ทราฟฟิกคำขอ/คำตอบทั้งหมด
- `--ws-log auto|compact|full`: เลือกรูปแบบการแสดงผลแบบละเอียด
- `--compact`: นามแฝงของ `--ws-log compact`

ตัวอย่าง:

```bash
openclaw gateway
openclaw gateway --verbose --ws-log compact
openclaw gateway --verbose --ws-log full
```

## การกำหนดค่าการบันทึก

การกำหนดค่าการบันทึกทั้งหมดอยู่ใต้ `logging` ใน `~/.openclaw/openclaw.json`

```json
{
  "logging": {
    "level": "info",
    "file": "/tmp/openclaw/openclaw-YYYY-MM-DD.log",
    "consoleLevel": "info",
    "consoleStyle": "pretty",
    "redactSensitive": "tools",
    "redactPatterns": ["sk-.*"]
  }
}
```

### ระดับบันทึก

- `logging.level`: ระดับของ **บันทึกไฟล์** (JSONL)
- `logging.consoleLevel`: ระดับความละเอียดของ **คอนโซล**

คุณสามารถแทนที่ทั้งสองค่านี้ได้ผ่านตัวแปรสภาพแวดล้อม **`OPENCLAW_LOG_LEVEL`** (เช่น `OPENCLAW_LOG_LEVEL=debug`) ตัวแปรสภาพแวดล้อมจะมีลำดับความสำคัญเหนือไฟล์กำหนดค่า ดังนั้นคุณจึงเพิ่มความละเอียดสำหรับการเรียกใช้ครั้งเดียวได้โดยไม่ต้องแก้ไข `openclaw.json` คุณยังสามารถส่งตัวเลือก CLI ส่วนกลาง **`--log-level <level>`** (เช่น `openclaw --log-level debug gateway run`) ซึ่งจะแทนที่ตัวแปรสภาพแวดล้อมสำหรับคำสั่งนั้น

`--verbose` มีผลเฉพาะกับเอาต์พุตคอนโซลและความละเอียดของบันทึก WS เท่านั้น และไม่เปลี่ยน
ระดับบันทึกไฟล์

### การเชื่อมโยง trace

บันทึกไฟล์เป็น JSONL เมื่อการเรียกบันทึกมีบริบท trace การวินิจฉัยที่ถูกต้อง
OpenClaw จะเขียนฟิลด์ trace เป็นคีย์ JSON ระดับบนสุด (`traceId`, `spanId`,
`parentSpanId`, `traceFlags`) เพื่อให้ตัวประมวลผลบันทึกภายนอกสามารถเชื่อมโยงบรรทัด
กับ span ของ OTEL และการส่งต่อ `traceparent` ของผู้ให้บริการได้

คำขอ HTTP ของ Gateway และเฟรม WebSocket ของ Gateway จะสร้างขอบเขต trace คำขอ
ภายใน บันทึกและเหตุการณ์วินิจฉัยที่ปล่อยภายในขอบเขต async นั้นจะสืบทอด
trace คำขอเมื่อไม่ได้ส่งบริบท trace แบบชัดเจน trace ของการรัน agent และ
การเรียกโมเดลจะกลายเป็นลูกของ trace คำขอที่ใช้งานอยู่ ดังนั้นบันทึกท้องถิ่น
สแนปช็อตวินิจฉัย span ของ OTEL และเฮดเดอร์ `traceparent` ของผู้ให้บริการที่เชื่อถือได้
จึงสามารถเชื่อมกันได้ด้วย `traceId` โดยไม่ต้องบันทึกเนื้อหาคำขอหรือโมเดลดิบ

### ขนาดและเวลาของการเรียกโมเดล

ข้อมูลวินิจฉัยการเรียกโมเดลบันทึกการวัดคำขอ/คำตอบที่มีขอบเขตโดยไม่
จับเนื้อหา prompt หรือคำตอบดิบ:

- `requestPayloadBytes`: ขนาดไบต์ UTF-8 ของ payload คำขอโมเดลสุดท้าย
- `responseStreamBytes`: ขนาดไบต์ UTF-8 ของเหตุการณ์คำตอบโมเดลแบบสตรีม
- `timeToFirstByteMs`: เวลาที่ผ่านไปก่อนเหตุการณ์คำตอบแบบสตรีมแรก
- `durationMs`: ระยะเวลารวมของการเรียกโมเดล

ฟิลด์เหล่านี้พร้อมใช้งานกับสแนปช็อตวินิจฉัย hook ของ Plugin การเรียกโมเดล และ
span/เมตริกการเรียกโมเดลของ OTEL เมื่อเปิดใช้งานการส่งออกข้อมูลวินิจฉัย

### รูปแบบคอนโซล

`logging.consoleStyle`:

- `pretty`: เป็นมิตรกับมนุษย์ มีสี พร้อมเวลา
- `compact`: เอาต์พุตกระชับขึ้น (เหมาะที่สุดสำหรับเซสชันยาว)
- `json`: JSON ต่อบรรทัด (สำหรับตัวประมวลผลบันทึก)

### การปกปิดข้อมูล

OpenClaw สามารถปกปิดโทเค็นที่ละเอียดอ่อนก่อนที่จะไปถึงเอาต์พุตคอนโซล บันทึกไฟล์
ระเบียนบันทึก OTLP ข้อความ transcript เซสชันที่คงไว้ หรือ payload เหตุการณ์เครื่องมือของ Control UI
(args เริ่มเครื่องมือ, payload ผลลัพธ์บางส่วน/สุดท้าย, เอาต์พุต exec ที่ได้มา
และสรุป patch):

- `logging.redactSensitive`: `off` | `tools` (ค่าเริ่มต้น: `tools`)
- `logging.redactPatterns`: รายการสตริง regex เพื่อแทนที่ชุดค่าเริ่มต้น รูปแบบกำหนดเองจะใช้ทับค่าเริ่มต้นในตัวสำหรับ payload เครื่องมือของ Control UI ดังนั้นการเพิ่มรูปแบบจะไม่ทำให้การปกปิดค่าที่ค่าเริ่มต้นตรวจจับได้อยู่แล้วอ่อนลง

บันทึกไฟล์และ transcript เซสชันยังคงเป็น JSONL แต่ค่าความลับที่ตรงกันจะถูก
ปิดบังก่อนเขียนบรรทัดหรือข้อความลงดิสก์ การปกปิดข้อมูลเป็นแบบพยายามอย่างดีที่สุด:
ใช้กับเนื้อหาข้อความที่มีตัวอักษรและสตริงบันทึก ไม่ใช่ทุก
ตัวระบุหรือฟิลด์ payload ไบนารี

ค่าเริ่มต้นในตัวครอบคลุมข้อมูลรับรอง API ทั่วไปและชื่อฟิลด์ข้อมูลรับรองการชำระเงิน
เช่น หมายเลขบัตร, CVC/CVV, โทเค็นการชำระเงินที่ใช้ร่วมกัน และข้อมูลรับรองการชำระเงิน
เมื่อปรากฏเป็นฟิลด์ JSON, พารามิเตอร์ URL, แฟล็ก CLI หรือการกำหนดค่า

`logging.redactSensitive: "off"` จะปิดใช้งานเฉพาะนโยบายบันทึก/transcript
ทั่วไปนี้เท่านั้น OpenClaw ยังปกปิด payload ขอบเขตความปลอดภัยที่สามารถแสดงต่อไคลเอนต์ UI
บันเดิลสนับสนุน ผู้สังเกตการณ์วินิจฉัย prompt การอนุมัติ หรือเครื่องมือ agent ได้
ตัวอย่างรวมถึงเหตุการณ์การเรียกเครื่องมือของ Control UI, เอาต์พุต `sessions_history`,
การส่งออกการสนับสนุนวินิจฉัย, การสังเกตข้อผิดพลาดของผู้ให้บริการ, การแสดงคำสั่งอนุมัติ exec
และบันทึกโปรโตคอล WebSocket ของ Gateway `logging.redactPatterns` แบบกำหนดเอง
ยังสามารถเพิ่มรูปแบบเฉพาะโปรเจกต์บนพื้นผิวเหล่านั้นได้

## ข้อมูลวินิจฉัยและ OpenTelemetry

ข้อมูลวินิจฉัยเป็นเหตุการณ์แบบมีโครงสร้างที่เครื่องอ่านได้สำหรับการรันโมเดลและ
telemetry ของลำดับข้อความ (webhooks, การเข้าคิว, สถานะเซสชัน) ข้อมูลเหล่านี้ **ไม่**
แทนที่บันทึก แต่ส่งต่อไปยังเมตริก, trace และตัวส่งออก เหตุการณ์จะถูกปล่อย
ภายในโปรเซสไม่ว่าคุณจะส่งออกหรือไม่ก็ตาม

พื้นผิวใกล้เคียงสองแบบ:

- **การส่งออก OpenTelemetry** — ส่งเมตริก, trace และบันทึกผ่าน OTLP/HTTP ไปยัง
  collector หรือ backend ที่เข้ากันได้กับ OpenTelemetry ใด ๆ (Grafana, Datadog,
  Honeycomb, New Relic, Tempo, ฯลฯ) การกำหนดค่าแบบเต็ม แค็ตตาล็อกสัญญาณ
  ชื่อเมตริก/span, ตัวแปรสภาพแวดล้อม และโมเดลความเป็นส่วนตัวอยู่ในหน้าเฉพาะ:
  [การส่งออก OpenTelemetry](/th/gateway/opentelemetry)
- **แฟล็กข้อมูลวินิจฉัย** — แฟล็กบันทึกดีบักแบบเจาะจงที่กำหนดเส้นทางบันทึกเพิ่มเติมไปยัง
  `logging.file` โดยไม่เพิ่ม `logging.level` แฟล็กไม่สนใจตัวพิมพ์เล็กใหญ่
  และรองรับ wildcard (`telegram.*`, `*`) กำหนดค่าใต้ `diagnostics.flags`
  หรือผ่านการแทนที่สภาพแวดล้อม `OPENCLAW_DIAGNOSTICS=...` คู่มือฉบับเต็ม:
  [แฟล็กข้อมูลวินิจฉัย](/th/diagnostics/flags)

หากต้องการเปิดใช้งานเหตุการณ์ข้อมูลวินิจฉัยสำหรับ plugins หรือ sink แบบกำหนดเองโดยไม่มีการส่งออก OTLP:

```json5
{
  diagnostics: { enabled: true },
}
```

สำหรับการส่งออก OTLP ไปยัง collector โปรดดู [การส่งออก OpenTelemetry](/th/gateway/opentelemetry)

## เคล็ดลับการแก้ไขปัญหา

- **ติดต่อ Gateway ไม่ได้?** เรียกใช้ `openclaw doctor` ก่อน
- **บันทึกว่างเปล่า?** ตรวจสอบว่า Gateway กำลังทำงานและเขียนไปยังเส้นทางไฟล์
  ใน `logging.file`
- **ต้องการรายละเอียดเพิ่ม?** ตั้งค่า `logging.level` เป็น `debug` หรือ `trace` แล้วลองใหม่
