# การรับมือเหตุการณ์

## 1. การตรวจจับและการคัดแยก

เราตรวจสอบสัญญาณด้านความปลอดภัยจาก:

- GitHub Security Advisories (GHSA) และรายงานช่องโหว่แบบส่วนตัว
- GitHub issues/discussions สาธารณะเมื่อรายงานไม่ใช่เรื่องละเอียดอ่อน
- สัญญาณอัตโนมัติ (เช่น Dependabot, CodeQL, npm advisories และ secret scanning)

การคัดแยกเบื้องต้น:

1. ยืนยันคอมโพเนนต์ เวอร์ชัน และผลกระทบต่อขอบเขตความเชื่อถือที่ได้รับผลกระทบ
2. จำแนกว่าเป็นปัญหาความปลอดภัยหรือเป็นการเสริมความแข็งแกร่ง/ไม่ต้องดำเนินการ โดยใช้ขอบเขตและกฎสิ่งที่อยู่นอกขอบเขตใน `SECURITY.md` ของ repository
3. เจ้าของเหตุการณ์ตอบสนองตามความเหมาะสม

## 2. การประเมิน

แนวทางระดับความรุนแรง:

- **วิกฤต:** การถูกบุกรุกของแพ็กเกจ/รีลีส/repository, การถูกใช้โจมตีอยู่จริง หรือการข้ามขอบเขตความเชื่อถือโดยไม่ต้องยืนยันตัวตนซึ่งมีผลกระทบสูงต่อการควบคุมหรือการเปิดเผยข้อมูล
- **สูง:** การข้ามขอบเขตความเชื่อถือที่ยืนยันแล้วและต้องมีเงื่อนไขเบื้องต้นจำกัด (เช่น ยืนยันตัวตนแล้วแต่ไม่มีสิทธิ์ดำเนินการที่มีผลกระทบสูง) หรือการเปิดเผยข้อมูลรับรองละเอียดอ่อนที่ OpenClaw เป็นเจ้าของ
- **ปานกลาง:** จุดอ่อนด้านความปลอดภัยที่สำคัญและมีผลกระทบในทางปฏิบัติ แต่มีความสามารถในการใช้โจมตีที่จำกัดหรือต้องมีเงื่อนไขเบื้องต้นมาก
- **ต่ำ:** ข้อค้นพบด้านการป้องกันหลายชั้น, การปฏิเสธการให้บริการที่มีขอบเขตแคบ หรือช่องว่างด้านการเสริมความแข็งแกร่ง/ความเท่าเทียมที่ยังไม่มีการแสดงให้เห็นว่าสามารถข้ามขอบเขตความเชื่อถือได้

## 3. การตอบสนอง

1. ยืนยันการได้รับรายงานกับผู้รายงาน (แบบส่วนตัวเมื่อเป็นเรื่องละเอียดอ่อน)
2. ทำซ้ำปัญหาบนรีลีสที่รองรับและ `main` ล่าสุด จากนั้นดำเนินการและตรวจสอบแพตช์พร้อมความครอบคลุมสำหรับการป้องกันการถดถอย
3. สำหรับเหตุการณ์ระดับวิกฤต/สูง ให้เตรียมรีลีสที่มีแพตช์ให้เร็วที่สุดเท่าที่ทำได้ในทางปฏิบัติ
4. สำหรับเหตุการณ์ระดับปานกลาง/ต่ำ ให้แพตช์ในกระบวนการรีลีสปกติและจัดทำเอกสารคำแนะนำการบรรเทาผลกระทบ

## 4. การสื่อสาร

เราสื่อสารผ่าน:

- GitHub Security Advisories ใน repository ที่ได้รับผลกระทบ
- บันทึกรีลีส/รายการ changelog สำหรับเวอร์ชันที่แก้ไขแล้ว
- การติดตามผลโดยตรงกับผู้รายงานเกี่ยวกับสถานะและการแก้ไข

นโยบายการเปิดเผย:

- เหตุการณ์ระดับวิกฤต/สูงควรได้รับการเปิดเผยแบบประสานงาน พร้อมการออก CVE เมื่อเหมาะสม
- ข้อค้นพบด้านการเสริมความแข็งแกร่งที่มีความเสี่ยงต่ำอาจถูกบันทึกไว้ในบันทึกรีลีสหรือ advisories โดยไม่มี CVE ขึ้นอยู่กับผลกระทบและการเปิดเผยต่อผู้ใช้

## 5. การกู้คืนและการติดตามผล

หลังจากส่งมอบการแก้ไขแล้ว:

1. ตรวจสอบการแก้ไขใน CI และอาร์ติแฟกต์ของรีลีส
2. ดำเนินการทบทวนหลังเหตุการณ์แบบสั้น (ไทม์ไลน์ สาเหตุราก ช่องว่างในการตรวจจับ แผนการป้องกัน)
3. เพิ่มงานติดตามผลสำหรับการเสริมความแข็งแกร่ง/การทดสอบ/เอกสาร และติดตามจนเสร็จสมบูรณ์
