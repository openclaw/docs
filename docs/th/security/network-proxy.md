# พร็อกซีเครือข่าย

OpenClaw สามารถกำหนดเส้นทางทราฟฟิก HTTP และ WebSocket ขณะรันผ่านพร็อกซีส่งต่อที่ผู้ปฏิบัติการจัดการได้ นี่เป็นการป้องกันเชิงลึกแบบเลือกใช้ได้สำหรับการปรับใช้ที่ต้องการควบคุมขาออกจากศูนย์กลาง การป้องกัน SSRF ที่แข็งแรงขึ้น และความสามารถในการตรวจสอบเครือข่ายที่ดีขึ้น

OpenClaw ไม่ได้จัดส่ง ดาวน์โหลด เริ่มต้น กำหนดค่า หรือรับรองพร็อกซี คุณรันเทคโนโลยีพร็อกซีที่เหมาะกับสภาพแวดล้อมของคุณ และ OpenClaw จะกำหนดเส้นทางไคลเอนต์ HTTP และ WebSocket ภายในกระบวนการตามปกติผ่านพร็อกซีนั้น

## ทำไมต้องใช้พร็อกซี?

พร็อกซีให้จุดควบคุมเครือข่ายเดียวแก่ผู้ปฏิบัติการสำหรับทราฟฟิก HTTP และ WebSocket ขาออก ซึ่งมีประโยชน์ได้แม้นอกเหนือจากการเสริมความแข็งแรงต่อ SSRF:

- นโยบายกลาง: ดูแลนโยบายขาออกเพียงชุดเดียว แทนการพึ่งพาให้ทุกจุดเรียก HTTP ของแอปพลิเคชันตั้งกฎเครือข่ายให้ถูกต้อง
- การตรวจสอบขณะเชื่อมต่อ: ประเมินปลายทางหลังการแก้ไข DNS และทันทีก่อนที่พร็อกซีจะเปิดการเชื่อมต่อต้นทาง
- การป้องกัน DNS rebinding: ลดช่องว่างระหว่างการตรวจสอบ DNS ระดับแอปพลิเคชันกับการเชื่อมต่อขาออกจริง
- ครอบคลุม JavaScript กว้างขึ้น: กำหนดเส้นทาง `fetch`, `node:http`, `node:https`, WebSocket, axios, got, node-fetch และไคลเอนต์ที่คล้ายกันผ่านเส้นทางเดียวกัน
- ความสามารถในการตรวจสอบ: บันทึกปลายทางที่อนุญาตและถูกปฏิเสธที่ขอบเขตขาออก
- การควบคุมเชิงปฏิบัติการ: บังคับใช้กฎปลายทาง การแบ่งส่วนเครือข่าย ขีดจำกัดอัตรา หรือรายการอนุญาตขาออกโดยไม่ต้องสร้าง OpenClaw ใหม่

การกำหนดเส้นทางผ่านพร็อกซีเป็นราวกันตกระดับกระบวนการสำหรับ HTTP และ WebSocket ขาออกตามปกติ ให้เส้นทางแบบปิดเมื่อผิดพลาดแก่ผู้ปฏิบัติการสำหรับกำหนดเส้นทางไคลเอนต์ HTTP ของ JavaScript ที่รองรับผ่านพร็อกซีกรองของตนเอง แต่ไม่ใช่แซนด์บ็อกซ์เครือข่ายระดับ OS และไม่ได้ทำให้ OpenClaw รับรองนโยบายปลายทางของพร็อกซี

## OpenClaw กำหนดเส้นทางทราฟฟิกอย่างไร

เมื่อ `proxy.enabled=true` และกำหนดค่า URL พร็อกซีแล้ว กระบวนการรันไทม์ที่ได้รับการป้องกัน เช่น `openclaw gateway run`, `openclaw node run` และ `openclaw agent --local` จะกำหนดเส้นทาง HTTP และ WebSocket ขาออกตามปกติผ่านพร็อกซีที่กำหนดค่าไว้:

```text
OpenClaw process
  fetch                  -> operator-managed filtering proxy -> public internet
  node:http and https    -> operator-managed filtering proxy -> public internet
  WebSocket clients      -> operator-managed filtering proxy -> public internet
```

สัญญาสาธารณะคือพฤติกรรมการกำหนดเส้นทาง ไม่ใช่ hook ภายในของ Node ที่ใช้ในการติดตั้งใช้งาน OpenClaw Gateway control-plane WebSocket clients ใช้เส้นทางตรงแบบแคบสำหรับทราฟฟิก Gateway RPC แบบ local loopback เมื่อ URL ของ Gateway ใช้ `localhost` หรือ IP loopback แบบตัวอักษร เช่น `127.0.0.1` หรือ `[::1]` เส้นทาง control-plane นั้นต้องสามารถเข้าถึง Gateway แบบ loopback ได้แม้พร็อกซีของผู้ปฏิบัติการบล็อกปลายทาง loopback อยู่ คำขอ HTTP และ WebSocket รันไทม์ตามปกติยังคงใช้พร็อกซีที่กำหนดค่าไว้

ภายใน OpenClaw ใช้ hook การกำหนดเส้นทางระดับกระบวนการสองตัวสำหรับฟีเจอร์นี้:

- การกำหนดเส้นทางด้วย dispatcher ของ Undici ครอบคลุม `fetch`, ไคลเอนต์ที่ใช้ undici เป็นฐาน และ transport ที่ให้ dispatcher ของ undici เอง
- การกำหนดเส้นทางด้วย `global-agent` ครอบคลุมผู้เรียก Node core `node:http` และ `node:https` รวมถึงไลบรารีจำนวนมากที่วางซ้อนบน `http.request`, `https.request`, `http.get` และ `https.get` โหมดพร็อกซีที่มีการจัดการจะบังคับใช้ global agent นั้น เพื่อไม่ให้ agent ของ Node HTTP ที่ระบุอย่างชัดเจนเลี่ยงผ่านพร็อกซีของผู้ปฏิบัติการโดยไม่ตั้งใจ

Plugin บางตัวเป็นเจ้าของ transport แบบกำหนดเองที่ต้องมีการต่อสายพร็อกซีอย่างชัดเจนแม้มีการกำหนดเส้นทางระดับกระบวนการอยู่แล้ว ตัวอย่างเช่น transport ของ Bot API ของ Telegram ใช้ dispatcher HTTP/1 ของ undici เอง จึงเคารพ env พร็อกซีของกระบวนการพร้อม fallback `OPENCLAW_PROXY_URL` ที่มีการจัดการในเส้นทาง transport เฉพาะเจ้าของนั้น

URL พร็อกซีเองต้องใช้ `http://` ปลายทาง HTTPS ยังคงรองรับผ่านพร็อกซีด้วย HTTP `CONNECT`; นี่หมายความเพียงว่า OpenClaw คาดหวัง listener ของ forward-proxy แบบ HTTP ธรรมดา เช่น `http://127.0.0.1:3128`

ขณะที่พร็อกซีทำงานอยู่ OpenClaw จะล้าง `no_proxy`, `NO_PROXY` และ `GLOBAL_AGENT_NO_PROXY` รายการเลี่ยงผ่านเหล่านั้นอิงตามปลายทาง ดังนั้นหากปล่อย `localhost` หรือ `127.0.0.1` ไว้ในนั้น จะทำให้เป้าหมาย SSRF ที่มีความเสี่ยงสูงข้ามพร็อกซีกรองได้

เมื่อปิดระบบ OpenClaw จะกู้คืนสภาพแวดล้อมพร็อกซีก่อนหน้าและรีเซ็ตสถานะการกำหนดเส้นทางของกระบวนการที่แคชไว้

## คำที่เกี่ยวข้องกับพร็อกซี

- `proxy.enabled` / `proxy.proxyUrl`: การกำหนดเส้นทาง forward-proxy ขาออกสำหรับทราฟฟิกขาออกของรันไทม์ OpenClaw หน้านี้บันทึกฟีเจอร์นั้น
- `gateway.auth.mode: "trusted-proxy"`: การยืนยันตัวตน reverse-proxy ขาเข้าที่รับรู้ตัวตนสำหรับการเข้าถึง Gateway ดู [การยืนยันตัวตนด้วยพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth)
- `openclaw proxy`: พร็อกซีดีบักภายในเครื่องและตัวตรวจสอบการจับข้อมูลสำหรับการพัฒนาและการสนับสนุน ดู [openclaw proxy](/th/cli/proxy)
- `tools.web.fetch.useTrustedEnvProxy`: การเลือกใช้สำหรับ `web_fetch` เพื่อให้พร็อกซี env HTTP(S) ที่ผู้ปฏิบัติการควบคุมแก้ไข DNS ได้ ขณะยังคงการตรึง DNS แบบเข้มงวดและนโยบายชื่อโฮสต์ตามค่าเริ่มต้น ดู [Web fetch](/th/tools/web-fetch#trusted-env-proxy)
- การตั้งค่าพร็อกซีเฉพาะช่องทางหรือผู้ให้บริการ: การ override เฉพาะเจ้าของสำหรับ transport เฉพาะรายการหนึ่ง ให้ใช้พร็อกซีเครือข่ายที่มีการจัดการเป็นหลักเมื่อเป้าหมายคือการควบคุมขาออกจากศูนย์กลางทั่วทั้งรันไทม์

## การกำหนดค่า

```yaml
proxy:
  enabled: true
  proxyUrl: http://127.0.0.1:3128
```

คุณยังสามารถให้ URL ผ่านสภาพแวดล้อมได้ โดยยังคง `proxy.enabled=true` ใน config:

```bash
OPENCLAW_PROXY_URL=http://127.0.0.1:3128 openclaw gateway run
```

`proxy.proxyUrl` มีลำดับความสำคัญเหนือ `OPENCLAW_PROXY_URL`

### โหมด loopback ของ Gateway

ไคลเอนต์ control-plane ของ Gateway ภายในเครื่องมักเชื่อมต่อกับ WebSocket แบบ loopback เช่น `ws://127.0.0.1:18789` ใช้ `proxy.loopbackMode` เพื่อเลือกว่าทราฟฟิกนั้นจะทำงานอย่างไรขณะที่พร็อกซีที่มีการจัดการทำงานอยู่:

```yaml
proxy:
  enabled: true
  proxyUrl: http://127.0.0.1:3128
  loopbackMode: gateway-only # gateway-only, proxy, or block
```

- `gateway-only` (ค่าเริ่มต้น): OpenClaw ลงทะเบียน authority ของ Gateway loopback ในตัวควบคุม `NO_PROXY` ของ `global-agent` ที่ทำงานอยู่ เพื่อให้ทราฟฟิก WebSocket ของ Gateway ภายในเครื่องเชื่อมต่อโดยตรงได้ พอร์ต Gateway แบบ loopback ที่กำหนดเองทำงานได้เพราะมีการลงทะเบียน host และ port ของ URL Gateway ที่ทำงานอยู่
- `proxy`: OpenClaw ไม่ลงทะเบียน authority ของ Gateway loopback ใน `NO_PROXY` ดังนั้นทราฟฟิก Gateway ภายในเครื่องจะถูกส่งผ่านพร็อกซีที่มีการจัดการ หากพร็อกซีอยู่ระยะไกล พร็อกซีนั้นต้องให้การกำหนดเส้นทางพิเศษสำหรับบริการ loopback ของโฮสต์ OpenClaw เช่น การแมปไปยังชื่อโฮสต์ IP หรือ tunnel ที่พร็อกซีเข้าถึงได้ พร็อกซีระยะไกลมาตรฐานแก้ไข `127.0.0.1` และ `localhost` จากโฮสต์พร็อกซี ไม่ใช่จากโฮสต์ OpenClaw
- `block`: OpenClaw ปฏิเสธการเชื่อมต่อ control-plane ของ Gateway แบบ loopback ก่อนเปิด socket

หาก `enabled=true` แต่ไม่มีการกำหนดค่า URL พร็อกซีที่ถูกต้อง คำสั่งที่ได้รับการป้องกันจะเริ่มต้นไม่สำเร็จแทนที่จะ fallback ไปใช้การเข้าถึงเครือข่ายโดยตรง

สำหรับบริการ gateway ที่มีการจัดการซึ่งเริ่มด้วย `openclaw gateway start` ให้เก็บ URL ใน config เป็นหลัก:

```bash
openclaw config set proxy.enabled true
openclaw config set proxy.proxyUrl http://127.0.0.1:3128
openclaw gateway install --force
openclaw gateway start
```

fallback ผ่านสภาพแวดล้อมเหมาะที่สุดสำหรับการรันแบบ foreground หากคุณใช้กับบริการที่ติดตั้งไว้ ให้ใส่ `OPENCLAW_PROXY_URL` ในสภาพแวดล้อมถาวรของบริการ เช่น `$OPENCLAW_STATE_DIR/.env` หรือ `~/.openclaw/.env` จากนั้นติดตั้งบริการใหม่เพื่อให้ launchd, systemd หรือ Scheduled Tasks เริ่ม gateway พร้อมค่านั้น

สำหรับคำสั่ง `openclaw --container ...` OpenClaw จะส่งต่อ `OPENCLAW_PROXY_URL` ไปยัง child CLI ที่กำหนดเป้าหมายไปยังคอนเทนเนอร์เมื่อมีการตั้งค่า URL ต้องเข้าถึงได้จากภายในคอนเทนเนอร์; `127.0.0.1` หมายถึงตัวคอนเทนเนอร์เอง ไม่ใช่โฮสต์ OpenClaw ปฏิเสธ URL พร็อกซีแบบ loopback สำหรับคำสั่งที่กำหนดเป้าหมายไปยังคอนเทนเนอร์ เว้นแต่คุณจะ override การตรวจสอบความปลอดภัยนั้นอย่างชัดเจน

## ข้อกำหนดของพร็อกซี

นโยบายพร็อกซีคือขอบเขตความปลอดภัย OpenClaw ไม่สามารถตรวจสอบได้ว่าพร็อกซีบล็อกเป้าหมายที่ถูกต้องหรือไม่

กำหนดค่าพร็อกซีให้:

- Bind เฉพาะกับ loopback หรืออินเทอร์เฟซส่วนตัวที่เชื่อถือได้
- จำกัดการเข้าถึงเพื่อให้เฉพาะกระบวนการ โฮสต์ คอนเทนเนอร์ หรือบัญชีบริการของ OpenClaw ใช้งานได้
- แก้ไขปลายทางด้วยตัวเองและบล็อก IP ปลายทางหลังการแก้ไข DNS
- ใช้นโยบายขณะเชื่อมต่อทั้งสำหรับคำขอ HTTP ธรรมดาและ tunnel HTTPS `CONNECT`
- ปฏิเสธการเลี่ยงผ่านตามปลายทางสำหรับ loopback, private, link-local, metadata, multicast, reserved หรือช่วง documentation
- หลีกเลี่ยงรายการอนุญาตชื่อโฮสต์ เว้นแต่คุณจะเชื่อถือเส้นทางการแก้ไข DNS อย่างเต็มที่
- บันทึกปลายทาง การตัดสินใจ สถานะ และเหตุผล โดยไม่บันทึกเนื้อหาคำขอ header การอนุญาต cookie หรือความลับอื่น
- เก็บนโยบายพร็อกซีไว้ภายใต้การควบคุมเวอร์ชันและตรวจทานการเปลี่ยนแปลงเหมือนการกำหนดค่าที่อ่อนไหวต่อความปลอดภัย

## ปลายทางที่แนะนำให้บล็อก

ใช้รายการปฏิเสธนี้เป็นจุดเริ่มต้นสำหรับ forward proxy, firewall หรือนโยบายขาออกใดๆ

ตรรกะ classifier ระดับแอปพลิเคชันของ OpenClaw อยู่ใน `src/infra/net/ssrf.ts` และ `src/shared/net/ip.ts` hook parity ที่เกี่ยวข้องคือ `BLOCKED_HOSTNAMES`, `BLOCKED_IPV4_SPECIAL_USE_RANGES`, `BLOCKED_IPV6_SPECIAL_USE_RANGES`, `RFC2544_BENCHMARK_PREFIX` และการจัดการ sentinel IPv4 แบบฝังสำหรับ NAT64, 6to4, Teredo, ISATAP และรูปแบบ IPv4-mapped ไฟล์เหล่านั้นเป็นเอกสารอ้างอิงที่มีประโยชน์เมื่อดูแลนโยบายพร็อกซีภายนอก แต่ OpenClaw ไม่ได้ export หรือบังคับใช้กฎเหล่านั้นในพร็อกซีของคุณโดยอัตโนมัติ

| ช่วงหรือโฮสต์                                                                        | เหตุผลที่ต้องบล็อก                                    |
| ------------------------------------------------------------------------------------ | ---------------------------------------------------- |
| `127.0.0.0/8`, `localhost`, `localhost.localdomain`                                  | IPv4 loopback                                        |

## การตรวจสอบความถูกต้อง

ตรวจสอบพร็อกซีจากโฮสต์ คอนเทนเนอร์ หรือบัญชีบริการเดียวกับที่รัน OpenClaw:

```bash
openclaw proxy validate --proxy-url http://127.0.0.1:3128
```

ตามค่าเริ่มต้น เมื่อไม่ได้ระบุปลายทางแบบกำหนดเอง คำสั่งจะตรวจสอบว่า `https://example.com/` สำเร็จ และเริ่มคานารี loopback ชั่วคราวที่พร็อกซีต้องเข้าถึงไม่ได้ การตรวจสอบรายการที่ถูกปฏิเสธตามค่าเริ่มต้นจะผ่านเมื่อพร็อกซีส่งคืนการตอบสนองปฏิเสธที่ไม่ใช่ 2xx หรือบล็อกคานารีด้วยความล้มเหลวของการขนส่งข้อมูล และจะล้มเหลวหากการตอบสนองที่สำเร็จไปถึงคานารี หากไม่มีการเปิดใช้และกำหนดค่าพร็อกซี การตรวจสอบจะรายงานปัญหาการกำหนดค่า ใช้ `--proxy-url` สำหรับการตรวจสอบล่วงหน้าแบบครั้งเดียวก่อนเปลี่ยนการกำหนดค่า ใช้ `--allowed-url` และ `--denied-url` เพื่อทดสอบความคาดหวังเฉพาะการปรับใช้ เพิ่ม `--apns-reachable` เพื่อยืนยันด้วยว่าการส่ง APNs HTTP/2 โดยตรงสามารถเปิดอุโมงค์ CONNECT ผ่านพร็อกซีและได้รับการตอบสนองจาก APNs sandbox ได้ โพรบใช้โทเค็นผู้ให้บริการที่ตั้งใจให้ไม่ถูกต้อง ดังนั้น `403 InvalidProviderToken` จึงเป็นผลที่คาดไว้และนับว่าเข้าถึงได้ ปลายทางที่ถูกปฏิเสธแบบกำหนดเองเป็นแบบ fail-closed: การตอบสนอง HTTP ใดๆ หมายความว่าปลายทางเข้าถึงได้ผ่านพร็อกซี และข้อผิดพลาดการขนส่งข้อมูลใดๆ จะถูกรายงานว่าไม่สามารถสรุปผลได้ เพราะ OpenClaw ไม่สามารถพิสูจน์ได้ว่าพร็อกซีบล็อกต้นทางที่เข้าถึงได้ เมื่อการตรวจสอบล้มเหลว คำสั่งจะออกด้วยรหัส 1

ใช้ `--json` สำหรับระบบอัตโนมัติ เอาต์พุต JSON มีผลลัพธ์โดยรวม แหล่งที่มาของการกำหนดค่าพร็อกซีที่มีผลใช้งาน ข้อผิดพลาดการกำหนดค่าใดๆ และการตรวจสอบปลายทางแต่ละรายการ ข้อมูลรับรองใน URL พร็อกซีจะถูกปกปิดในเอาต์พุตข้อความและ JSON:

```json
{
  "ok": true,
  "config": {
    "enabled": true,
    "proxyUrl": "http://127.0.0.1:3128/",
    "source": "override",
    "errors": []
  },
  "checks": [
    {
      "kind": "allowed",
      "url": "https://example.com/",
      "ok": true,
      "status": 200
    },
    {
      "kind": "apns",
      "url": "https://api.sandbox.push.apple.com",
      "ok": true,
      "status": 403
    }
  ]
}
```

คุณยังสามารถตรวจสอบด้วยตนเองโดยใช้ `curl`:

```bash
curl -x http://127.0.0.1:3128 https://example.com/
curl -x http://127.0.0.1:3128 http://127.0.0.1/
curl -x http://127.0.0.1:3128 http://169.254.169.254/
```

คำขอสาธารณะควรสำเร็จ คำขอ loopback และ metadata ควรถูกพร็อกซีบล็อก สำหรับ `openclaw proxy validate` คานารี loopback ในตัวสามารถแยกการปฏิเสธของพร็อกซีออกจากต้นทางที่เข้าถึงได้ การตรวจสอบ `--denied-url` แบบกำหนดเองไม่มีคานารีนั้น ดังนั้นให้ถือว่าทั้งการตอบสนอง HTTP และความล้มเหลวของการขนส่งข้อมูลที่กำกวมเป็นความล้มเหลวในการตรวจสอบ เว้นแต่พร็อกซีของคุณจะเปิดเผยสัญญาณการปฏิเสธเฉพาะการปรับใช้ที่คุณสามารถยืนยันแยกต่างหากได้

จากนั้นเปิดใช้การกำหนดเส้นทางพร็อกซีของ OpenClaw:

```bash
openclaw config set proxy.enabled true
openclaw config set proxy.proxyUrl http://127.0.0.1:3128
openclaw gateway run
```

หรือกำหนด:

```yaml
proxy:
  enabled: true
  proxyUrl: http://127.0.0.1:3128
```

## ข้อจำกัด

- พร็อกซีช่วยเพิ่มความครอบคลุมสำหรับไคลเอนต์ HTTP และ WebSocket ของ JavaScript ภายในกระบวนการ แต่ไม่ใช่ sandbox เครือข่ายระดับระบบปฏิบัติการ
- การรับส่งข้อมูล control-plane ของ Gateway loopback มีค่าเริ่มต้นเป็นการข้ามผ่านภายในโดยตรงผ่าน `proxy.loopbackMode: "gateway-only"` OpenClaw ดำเนินการข้ามผ่านนี้โดยลงทะเบียน authority ของ Gateway loopback ที่ใช้งานอยู่ในคอนโทรลเลอร์ `NO_PROXY` ของ `global-agent` ที่จัดการอยู่ ผู้ปฏิบัติงานสามารถตั้งค่า `proxy.loopbackMode: "proxy"` เพื่อส่งการรับส่งข้อมูล Gateway loopback ผ่านพร็อกซีที่จัดการอยู่ หรือ `proxy.loopbackMode: "block"` เพื่อปฏิเสธการเชื่อมต่อ Gateway แบบ loopback ดู [โหมด Gateway Loopback](#gateway-loopback-mode) สำหรับข้อควรระวังเกี่ยวกับพร็อกซีระยะไกล
- ซ็อกเก็ต `net`, `tls` และ `http2` แบบดิบ native addons และกระบวนการลูกที่ไม่ใช่ OpenClaw อาจข้ามการกำหนดเส้นทางพร็อกซีระดับ Node เว้นแต่จะสืบทอดและเคารพตัวแปรสภาพแวดล้อมของพร็อกซี CLI ลูกของ OpenClaw ที่ถูก fork จะสืบทอด URL พร็อกซีที่จัดการอยู่และสถานะ `proxy.loopbackMode`
- IRC เป็นช่องทาง TCP/TLS แบบดิบที่อยู่นอกการกำหนดเส้นทางพร็อกซีส่งต่อที่ผู้ปฏิบัติงานจัดการ ในการปรับใช้ที่กำหนดให้การรับส่งข้อมูลขาออกทั้งหมดต้องผ่านพร็อกซีส่งต่อนั้น ให้ตั้งค่า `channels.irc.enabled=false` เว้นแต่การรับส่งข้อมูลขาออก IRC โดยตรงจะได้รับอนุมัติอย่างชัดเจน
- พร็อกซีดีบักภายในเป็นเครื่องมือวินิจฉัย และการส่งต่อ upstream โดยตรงสำหรับคำขอพร็อกซีและอุโมงค์ CONNECT จะถูกปิดใช้งานตามค่าเริ่มต้นขณะที่โหมดพร็อกซีที่จัดการอยู่ทำงานอยู่ เปิดใช้การส่งต่อโดยตรงเฉพาะสำหรับการวินิจฉัยภายในที่ได้รับอนุมัติแล้วเท่านั้น
- WebUI ภายในของผู้ใช้และเซิร์ฟเวอร์โมเดลภายในควรถูกเพิ่มในรายการอนุญาตในนโยบายพร็อกซีของผู้ปฏิบัติงานเมื่อจำเป็น OpenClaw ไม่เปิดเผยการข้ามผ่านเครือข่ายภายในแบบทั่วไปสำหรับสิ่งเหล่านี้
- การข้ามพร็อกซีของ control-plane ของ Gateway ถูกจำกัดไว้เฉพาะ `localhost` และ URL IP loopback แบบตรงตัวโดยตั้งใจ ใช้ `ws://127.0.0.1:18789`, `ws://[::1]:18789` หรือ `ws://localhost:18789` สำหรับการเชื่อมต่อ control-plane ของ Gateway ภายในโดยตรง ชื่อโฮสต์อื่นจะถูกกำหนดเส้นทางเหมือนการรับส่งข้อมูลแบบอิงชื่อโฮสต์ทั่วไป
- OpenClaw ไม่ตรวจสอบ ทดสอบ หรือรับรองนโยบายพร็อกซีของคุณ
- ให้ถือว่าการเปลี่ยนแปลงนโยบายพร็อกซีเป็นการเปลี่ยนแปลงการปฏิบัติการที่อ่อนไหวด้านความปลอดภัย
