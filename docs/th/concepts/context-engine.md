**กลไกบริบท**ควบคุมวิธีที่ OpenClaw สร้างบริบทของโมเดลสำหรับการทำงานแต่ละครั้ง ได้แก่ ข้อความที่จะรวม วิธีสรุปประวัติเก่า และวิธีจัดการบริบทข้ามขอบเขตของ subagent

OpenClaw มาพร้อมกับกลไก `legacy` ในตัวและใช้กลไกนี้เป็นค่าเริ่มต้น ติดตั้งและเลือกกลไก Plugin เฉพาะเมื่อต้องการลักษณะการประกอบบริบท, Compaction หรือการเรียกคืนข้อมูลข้ามเซสชันที่แตกต่างออกไป

## เริ่มต้นอย่างรวดเร็ว

<Steps>
  <Step title="ตรวจสอบว่ากลไกใดกำลังทำงานอยู่">
    ```bash
    openclaw doctor
    # หรือตรวจสอบการกำหนดค่าโดยตรง:
    cat ~/.openclaw/openclaw.json | jq '.plugins.slots.contextEngine'
    ```
  </Step>
  <Step title="ติดตั้งกลไก Plugin">
    Plugin กลไกบริบทติดตั้งได้เช่นเดียวกับ Plugin อื่นๆ ของ OpenClaw

    <Tabs>
      <Tab title="จาก npm">
        ```bash
        openclaw plugins install @martian-engineering/lossless-claw
        ```
      </Tab>
      <Tab title="จากพาธภายในเครื่อง">
        ```bash
        openclaw plugins install -l ./my-context-engine
        ```
      </Tab>
    </Tabs>

  </Step>
  <Step title="เปิดใช้งานและเลือกกลไก">
    ```json5
    // openclaw.json
    {
      plugins: {
        slots: {
          contextEngine: "lossless-claw", // ต้องตรงกับรหัสกลไกที่ Plugin ลงทะเบียนไว้
        },
        entries: {
          "lossless-claw": {
            enabled: true,
            // ใส่การกำหนดค่าเฉพาะของ Plugin ที่นี่ (ดูเอกสารของ Plugin)
          },
        },
      },
    }
    ```

    รีสตาร์ต Gateway หลังจากติดตั้งและกำหนดค่า

  </Step>
  <Step title="เปลี่ยนกลับไปใช้แบบดั้งเดิม (ไม่บังคับ)">
    ตั้งค่า `contextEngine` เป็น `"legacy"` (หรือลบคีย์ออกทั้งหมด โดย `"legacy"` เป็นค่าเริ่มต้น)
  </Step>
</Steps>

## วิธีการทำงาน

ทุกครั้งที่ OpenClaw เรียกใช้พรอมต์ของโมเดล กลไกบริบทจะมีส่วนร่วมใน 4 จุดของวงจรชีวิต:

<AccordionGroup>
  <Accordion title="1. รับเข้า">
    เรียกใช้เมื่อมีการเพิ่มข้อความใหม่ลงในเซสชัน กลไกสามารถจัดเก็บหรือสร้างดัชนีข้อความไว้ในแหล่งเก็บข้อมูลของตนเอง
  </Accordion>
  <Accordion title="2. ประกอบ">
    เรียกใช้ก่อนการทำงานของโมเดลแต่ละครั้ง กลไกจะส่งคืนชุดข้อความตามลำดับ (และ `systemPromptAddition` ซึ่งไม่บังคับ) ที่อยู่ภายในงบประมาณโทเค็น
  </Accordion>
  <Accordion title="3. ทำ Compaction">
    เรียกใช้เมื่อหน้าต่างบริบทเต็ม หรือเมื่อผู้ใช้เรียกใช้ `/compact` กลไกจะสรุปประวัติเก่าเพื่อเพิ่มพื้นที่ว่าง
  </Accordion>
  <Accordion title="4. หลังจบรอบ">
    เรียกใช้หลังจากการทำงานเสร็จสิ้น กลไกสามารถคงสถานะไว้ เริ่ม Compaction ในเบื้องหลัง หรืออัปเดตดัชนี
  </Accordion>
</AccordionGroup>

กลไกยังสามารถใช้งานเมธอด `maintain()` ซึ่งไม่บังคับเพื่อบำรุงรักษาทรานสคริปต์ (เขียนใหม่อย่างปลอดภัยผ่าน `runtimeContext.rewriteTranscriptEntries()`) หลังการเริ่มต้นระบบ หลังจบรอบที่สำเร็จ หรือหลัง Compaction ตั้งค่า `info.turnMaintenanceMode: "background"` เพื่อเรียกใช้เป็นงานที่เลื่อนออกไปแทนการบล็อกการตอบกลับ

สำหรับชุดทดสอบ Codex ที่ไม่ใช่ ACP ซึ่งรวมมาให้ OpenClaw ใช้วงจรชีวิตเดียวกันโดยฉายบริบทที่ประกอบแล้วไปยังคำสั่งสำหรับนักพัฒนาของ Codex และพรอมต์ของรอบปัจจุบัน Codex ยังคงจัดการประวัติเธรดและตัวทำ Compaction แบบเนทีฟของตนเอง

### วงจรชีวิตของ subagent (ไม่บังคับ)

OpenClaw เรียกใช้ฮุกวงจรชีวิตของ subagent ที่ไม่บังคับ 2 รายการ:

<ParamField path="prepareSubagentSpawn" type="method">
  เตรียมสถานะบริบทที่ใช้ร่วมกันก่อนเริ่มการทำงานของลูก ฮุกจะได้รับคีย์เซสชันแม่/ลูก, `contextMode` (`isolated` หรือ `fork`), รหัส/ไฟล์ทรานสคริปต์ที่พร้อมใช้งาน และ TTL ซึ่งไม่บังคับ หากส่งคืนแฮนเดิลสำหรับย้อนกลับ OpenClaw จะเรียกใช้แฮนเดิลนั้นเมื่อการสร้างล้มเหลวหลังจากเตรียมการสำเร็จ การสร้าง subagent แบบเนทีฟที่ร้องขอ `lightContext` และแปลงค่าเป็น `contextMode="isolated"` จะข้ามฮุกนี้โดยเจตนา เพื่อให้ลูกเริ่มต้นจากบริบทการเริ่มระบบขนาดเล็กโดยไม่มีสถานะก่อนการสร้างที่จัดการโดยกลไกบริบท
</ParamField>
<ParamField path="onSubagentEnded" type="method">
  ล้างข้อมูลเมื่อเซสชันของ subagent เสร็จสิ้นหรือถูกกวาดล้าง
</ParamField>

### ส่วนเพิ่มเติมของพรอมต์ระบบ

เมธอด `assemble` สามารถส่งคืนสตริง `systemPromptAddition` ได้ OpenClaw จะเติมสตริงนี้ไว้หน้าพรอมต์ระบบสำหรับการทำงานครั้งนั้น ซึ่งช่วยให้กลไกแทรกคำแนะนำการเรียกคืนข้อมูลแบบไดนามิก คำสั่งการดึงข้อมูล หรือคำใบ้ที่รับรู้บริบทได้โดยไม่ต้องใช้ไฟล์เวิร์กสเปซแบบคงที่

## กลไกแบบดั้งเดิม

กลไก `legacy` ในตัวจะคงลักษณะการทำงานเดิมของ OpenClaw ไว้:

- **รับเข้า**: ไม่ดำเนินการ (ตัวจัดการเซสชันจัดการการคงข้อความไว้โดยตรง)
- **ประกอบ**: ส่งผ่านโดยไม่เปลี่ยนแปลง (ไปป์ไลน์ sanitize → validate → limit ที่มีอยู่ในรันไทม์จัดการการประกอบบริบท)
- **ทำ Compaction**: มอบหมายให้ Compaction สำหรับการสรุปในตัว ซึ่งสร้างข้อมูลสรุปเดียวจากข้อความเก่าและคงข้อความล่าสุดไว้โดยไม่เปลี่ยนแปลง
- **หลังจบรอบ**: ไม่ดำเนินการ

กลไกแบบดั้งเดิมไม่ลงทะเบียนเครื่องมือหรือให้ `systemPromptAddition`

เมื่อไม่ได้ตั้งค่า `plugins.slots.contextEngine` (หรือตั้งค่าเป็น `"legacy"`) ระบบจะใช้กลไกนี้โดยอัตโนมัติ

## กลไก Plugin

Plugin สามารถลงทะเบียนกลไกบริบทโดยใช้ API ของ Plugin:

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";
import { resolveSessionAgentId } from "openclaw/plugin-sdk/memory-host-core";

export default function register(api) {
  api.registerContextEngine("my-engine", (ctx) => ({
    info: {
      id: "my-engine",
      name: "My Context Engine",
      ownsCompaction: true,
    },

    async ingest({ sessionId, message, isHeartbeat }) {
      // จัดเก็บข้อความในแหล่งเก็บข้อมูลของคุณ
      return { ingested: true };
    },

    async assemble({
      sessionId,
      sessionKey,
      messages,
      tokenBudget,
      availableTools,
      citationsMode,
    }) {
      // ส่งคืนข้อความที่อยู่ภายในงบประมาณ
      return {
        messages: buildContext(messages, tokenBudget),
        estimatedTokens: countTokens(messages),
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentId: resolveSessionAgentId({ config: ctx.config, sessionKey }),
          agentSessionKey: sessionKey,
        }),
      };
    },

    async compact({ sessionId, force }) {
      // สรุปบริบทเก่า
      return { ok: true, compacted: true };
    },
  }));
}
```

แฟกทอรี `ctx` มีค่า `config`, `agentDir` และ `workspaceDir`
ซึ่งไม่บังคับ เพื่อให้ Plugin เริ่มต้นสถานะต่อเอเจนต์หรือต่อเวิร์กสเปซได้ก่อน
การเรียกวงจรชีวิตครั้งแรก ก่อนการเรียก `assemble()` ที่ไม่ใช่แบบดั้งเดิม โฮสต์จะดำเนินการ
เตรียมพรอมต์หน่วยความจำแบบอะซิงโครนัสที่ลงทะเบียนไว้จนเสร็จสมบูรณ์ ตัวช่วยแบบซิงโครนัส
`buildMemorySystemPromptAddition(...)` จะอ่านสแนปช็อตการทำงานที่เปลี่ยนแปลงไม่ได้ดังกล่าว
โดยส่งผ่านบริบทของเครื่องมือ การอ้างอิง เอเจนต์ และเซสชันที่ได้รับมาโดยไม่เปลี่ยนแปลง

จากนั้นเปิดใช้งานในการกำหนดค่า:

```json5
{
  plugins: {
    slots: {
      contextEngine: "my-engine",
    },
    entries: {
      "my-engine": {
        enabled: true,
      },
    },
  },
}
```

### อินเทอร์เฟซ ContextEngine

สมาชิกที่จำเป็น:

| สมาชิก             | ชนิด     | วัตถุประสงค์                                                  |
| ------------------ | -------- | -------------------------------------------------------- |
| `info`             | คุณสมบัติ | รหัส ชื่อ เวอร์ชันของกลไก และกลไกเป็นผู้จัดการ Compaction หรือไม่ |
| `ingest(params)`   | เมธอด   | จัดเก็บข้อความหนึ่งรายการ                                   |
| `assemble(params)` | เมธอด   | สร้างบริบทสำหรับการทำงานของโมเดล (ส่งคืน `AssembleResult`) |
| `compact(params)`  | เมธอด   | สรุป/ลดบริบท                                 |

`assemble` ส่งคืน `AssembleResult` ซึ่งประกอบด้วย:

<ParamField path="messages" type="Message[]" required>
  ข้อความตามลำดับที่จะส่งไปยังโมเดล
</ParamField>
<ParamField path="estimatedTokens" type="number" required>
  ค่าประมาณจำนวนโทเค็นทั้งหมดในบริบทที่ประกอบแล้วของกลไก OpenClaw ใช้ค่านี้ในการตัดสินใจเกี่ยวกับเกณฑ์ Compaction และการรายงานเพื่อวินิจฉัย
</ParamField>
<ParamField path="systemPromptAddition" type="string">
  เติมไว้หน้าพรอมต์ระบบ
</ParamField>
<ParamField path="promptAuthority" type='"assembled" | "preassembly_may_overflow"'>
  ควบคุมค่าประมาณโทเค็นที่ตัวเรียกใช้ใช้สำหรับการตรวจสอบล่วงหน้าก่อนเกิดการล้น
  ค่าเริ่มต้นคือ `"assembled"` ซึ่งหมายความว่าจะตรวจสอบเฉพาะค่าประมาณของ
  พรอมต์ที่ประกอบแล้วสำหรับกลไกที่ไม่ได้จัดการ Compaction
  กลไกที่ตั้งค่า `ownsCompaction: true` จะจัดการการรับพรอมต์ด้วยตนเอง
  ดังนั้น OpenClaw จึงข้ามการตรวจสอบล่วงหน้าก่อนพรอมต์แบบทั่วไปโดยค่าเริ่มต้น ตั้งค่า
  `"preassembly_may_overflow"` เฉพาะเมื่อมุมมองที่ประกอบแล้วสามารถซ่อนความเสี่ยง
  ที่จะเกิดการล้นในทรานสคริปต์พื้นฐานได้ จากนั้นตัวเรียกใช้จะเปิดการตรวจสอบล่วงหน้าแบบทั่วไป
  ไว้และใช้ค่าสูงสุดระหว่างค่าประมาณที่ประกอบแล้วกับค่าประมาณประวัติเซสชัน
  ก่อนการประกอบ (ที่ไม่จำกัดหน้าต่าง) เมื่อตัดสินใจว่าจะทำ Compaction
  ล่วงหน้าหรือไม่ ไม่ว่าจะเลือกแบบใด ข้อความที่ส่งคืนยังคงเป็นสิ่งที่
  โมเดลมองเห็น โดย `promptAuthority` มีผลเฉพาะต่อการตรวจสอบล่วงหน้า
</ParamField>
<ParamField path="contextProjection" type="ContextEngineProjection">
  วงจรชีวิตการฉายบริบทซึ่งไม่บังคับสำหรับโฮสต์ที่มีเธรดแบ็กเอนด์แบบถาวร (เช่น Codex app-server) `mode: "thread_bootstrap"` พร้อม `epoch` ที่คงที่ จะขอให้โฮสต์แทรกบริบทที่ประกอบแล้วหนึ่งครั้งต่อยุคและใช้เธรดแบ็กเอนด์ซ้ำจนกว่ายุคจะเปลี่ยน แทนที่จะฉายใหม่ทุกรอบ ละเว้นฟิลด์นี้สำหรับการฉายแบบปกติในแต่ละรอบ
</ParamField>

`compact` ส่งคืน `CompactResult` เมื่อ Compaction เปลี่ยนข้อมูลประจำตัวของเซสชันที่ใช้งานอยู่
`result.sessionTarget` (`ContextEngineSessionTarget` ที่มีชนิดชัดเจนซึ่งบรรจุ
ข้อมูลประจำตัวของเซสชันและขอบเขตแหล่งเก็บข้อมูล) จะระบุเซสชันถัดไปที่
การลองใหม่หรือรอบถัดไปต้องใช้ โดย `result.sessionId` สะท้อนรหัสของเซสชันถัดไป

สมาชิกที่ไม่บังคับ:

| สมาชิก                         | ชนิด   | วัตถุประสงค์                                                                                                                                      |
| ------------------------------ | ------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `bootstrap(params)`            | เมธอด | เริ่มต้นสถานะกลไกสำหรับเซสชัน เรียกหนึ่งครั้งเมื่อกลไกพบเซสชันเป็นครั้งแรก (เช่น นำเข้าประวัติ)                              |
| `maintain(params)`             | เมธอด | บำรุงรักษาทรานสคริปต์หลังการเริ่มต้นระบบ หลังจบรอบที่สำเร็จ หรือหลัง Compaction ใช้ `runtimeContext.rewriteTranscriptEntries()` สำหรับการเขียนใหม่อย่างปลอดภัย |
| `ingestBatch(params)`          | เมธอด | รับเข้ารอบที่เสร็จสมบูรณ์เป็นชุด เรียกหลังจากการทำงานเสร็จสิ้น พร้อมข้อความทั้งหมดจากรอบนั้นในคราวเดียว                                  |
| `afterTurn(params)`            | เมธอด | งานวงจรชีวิตหลังการทำงาน (คงสถานะไว้ เริ่ม Compaction ในเบื้องหลัง)                                                                      |
| `prepareSubagentSpawn(params)` | เมธอด | ตั้งค่าสถานะที่ใช้ร่วมกันสำหรับเซสชันลูกก่อนเริ่มทำงาน                                                                                    |
| `onSubagentEnded(params)`      | เมธอด | ล้างข้อมูลหลังจาก subagent สิ้นสุด                                                                                                              |
| `dispose()`                    | เมธอด | ปล่อยทรัพยากร เรียกระหว่างการปิด Gateway หรือการโหลด Plugin ใหม่ ไม่ใช่ต่อเซสชัน                                                        |

### การตั้งค่ารันไทม์

ฮุกวงจรชีวิตที่ทำงานภายใน OpenClaw จะได้รับออบเจ็กต์
`runtimeSettings` ซึ่งไม่บังคับ ออบเจ็กต์นี้เป็นพื้นผิว API ภายในแบบอ่านอย่างเดียว
ที่มีการกำหนดเวอร์ชันสำหรับผู้ผลิต/ผู้บริโภค โดย OpenClaw สร้างออบเจ็กต์นี้ให้กับกลไกบริบท
ที่เลือก และกลไกบริบทจะใช้ออบเจ็กต์นี้ภายในฮุกวงจรชีวิต ระบบจะไม่แสดงออบเจ็กต์นี้
แก่ผู้ใช้โดยตรง และจะไม่สร้างพื้นผิวการรายงานเฉพาะ

- `schemaVersion`: ปัจจุบันคือ `1`
- `runtime`: โฮสต์ OpenClaw, โหมดรันไทม์ (`normal`, `fallback` หรือ
  `degraded`) และรหัสชุดทดสอบ/รันไทม์ที่ระบุหรือไม่ก็ได้
- `contextEngineSelection`: รหัสเอนจินบริบทที่เลือกและแหล่งที่มาของการเลือก
- `executionHost`: รหัสและป้ายกำกับโฮสต์สำหรับพื้นผิวที่เรียกใช้ฮุก
- `model`: โมเดลที่ร้องขอ โมเดลที่แก้ไขแล้ว ผู้ให้บริการ และตระกูลโมเดลที่ระบุหรือไม่ก็ได้
- `limits`: งบประมาณโทเค็นพรอมต์และจำนวนโทเค็นเอาต์พุตสูงสุดเมื่อทราบ
- `diagnostics`: รหัสเหตุผลของการสำรองแบบปิดและการลดระดับเมื่อทราบ

ฟิลด์ที่อาจไม่ทราบจะแสดงเป็น `null`; ฟิลด์ตัวจำแนก เช่น
โหมดรันไทม์และแหล่งที่มาของการเลือก ยังคงไม่อนุญาตให้เป็นค่าว่าง เอนจินรุ่นเก่ายังคง
เข้ากันได้: หากเอนจินแบบเดิมที่เข้มงวดปฏิเสธ `runtimeSettings` เนื่องจากเป็น
พร็อพเพอร์ตีที่ไม่รู้จัก OpenClaw จะลองเรียกวงจรชีวิตอีกครั้งโดยไม่มีพร็อพเพอร์ตีดังกล่าว
แทนที่จะกักกันเอนจิน

### ข้อกำหนดของโฮสต์

เอนจินบริบทสามารถประกาศข้อกำหนดด้านความสามารถของโฮสต์ใน `info.hostRequirements`
OpenClaw จะตรวจสอบข้อกำหนดเหล่านี้ก่อนเริ่มการดำเนินการ และปฏิเสธการทำงาน
พร้อมข้อผิดพลาดที่อธิบายชัดเจนเมื่อรันไทม์ที่เลือกไม่สามารถตอบสนองข้อกำหนดได้

สำหรับการเรียกใช้เอเจนต์ ให้ประกาศ `assemble-before-prompt` เมื่อเอนจินต้องควบคุม
พรอมต์โมเดลจริงผ่าน `assemble()`:

```ts
info: {
  id: "my-context-engine",
  name: "My Context Engine",
  hostRequirements: {
    "agent-run": {
      requiredCapabilities: ["assemble-before-prompt"],
      unsupportedMessage:
        "ใช้รันไทม์ Codex แบบเนทีฟหรือรันไทม์แบบฝังของ OpenClaw หรือเลือกเอนจินบริบทแบบเดิม",
    },
  },
}
```

การเรียกใช้เอเจนต์ด้วย Codex แบบเนทีฟและ OpenClaw แบบฝังรองรับ `assemble-before-prompt`
แบ็กเอนด์ CLI ทั่วไปไม่รองรับ ดังนั้นเอนจินที่ต้องใช้ความสามารถนี้จะถูกปฏิเสธก่อน
เริ่มกระบวนการ CLI

### การแยกความล้มเหลว

OpenClaw แยกเอนจิน Plugin ที่เลือกออกจากเส้นทางการตอบกลับหลัก หากเอนจิน
ที่ไม่ใช่แบบเดิมหายไป ไม่ผ่านการตรวจสอบความถูกต้องตามสัญญา เกิดข้อยกเว้นระหว่าง
การสร้างแฟกทอรี หรือเกิดข้อยกเว้นจากเมธอดวงจรชีวิต OpenClaw จะกักกันเอนจินนั้น
สำหรับกระบวนการ Gateway ปัจจุบัน และลดระดับงานของเอนจินบริบทไปใช้เอนจิน
`legacy` ในตัว ข้อผิดพลาดจะถูกบันทึกพร้อมการดำเนินการที่ล้มเหลว เพื่อให้
ผู้ปฏิบัติงานสามารถซ่อมแซม อัปเดต หรือปิดใช้งาน Plugin ได้โดยที่เอเจนต์ไม่หยุด
ตอบสนอง

ความล้มเหลวของข้อกำหนดโฮสต์แตกต่างออกไป: เมื่อเอนจินประกาศว่ารันไทม์
ขาดความสามารถที่จำเป็น OpenClaw จะปฏิเสธการทำงานก่อนเริ่มการเรียกใช้ ซึ่ง
ช่วยปกป้องเอนจินที่อาจทำให้สถานะเสียหายหากทำงานในโฮสต์ที่ไม่รองรับ

### ownsCompaction

`ownsCompaction` ควบคุมว่าจะเปิดใช้ Compaction อัตโนมัติภายในความพยายามที่มีมาให้ในรันไทม์ OpenClaw สำหรับการเรียกใช้นั้นต่อไปหรือไม่:

<AccordionGroup>
  <Accordion title="ownsCompaction: true">
    เอนจินเป็นผู้ควบคุมพฤติกรรม Compaction โดย OpenClaw จะปิดใช้ Compaction อัตโนมัติที่มีมาให้ในรันไทม์ OpenClaw และการตรวจสอบโอเวอร์โฟลว์ทั่วไปก่อนพรอมต์สำหรับการเรียกใช้นั้น และการใช้งาน `compact()` ของเอนจินจะรับผิดชอบ `/compact`, Compaction เพื่อกู้คืนจากโอเวอร์โฟลว์ของผู้ให้บริการ และ Compaction เชิงรุกใดๆ ที่ต้องการดำเนินการใน `afterTurn()` ทั้งนี้ OpenClaw ยังคงเรียกใช้มาตรการป้องกันโอเวอร์โฟลว์ก่อนพรอมต์เมื่อเอนจินส่งคืน `promptAuthority: "preassembly_may_overflow"` จาก `assemble()`
  </Accordion>
  <Accordion title="ownsCompaction: false หรือไม่ได้ตั้งค่า">
    Compaction อัตโนมัติที่มีมาให้ในรันไทม์ OpenClaw อาจยังทำงานระหว่างการประมวลผลพรอมต์ แต่เมธอด `compact()` ของเอนจินที่ใช้งานอยู่จะยังคงถูกเรียกสำหรับ `/compact` และการกู้คืนจากโอเวอร์โฟลว์
  </Accordion>
</AccordionGroup>

<Warning>
`ownsCompaction: false` **ไม่ได้** หมายความว่า OpenClaw จะย้อนกลับไปใช้เส้นทาง Compaction ของเอนจินแบบเดิมโดยอัตโนมัติ
</Warning>

ดังนั้นจึงมีรูปแบบ Plugin ที่ถูกต้องสองรูปแบบ:

<Tabs>
  <Tab title="โหมดควบคุมเอง">
    ใช้อัลกอริทึม Compaction ของคุณเองและตั้งค่า `ownsCompaction: true`
  </Tab>
  <Tab title="โหมดมอบหมาย">
    ตั้งค่า `ownsCompaction: false` และให้ `compact()` เรียก `delegateCompactionToRuntime(...)` จาก `openclaw/plugin-sdk/core` เพื่อใช้พฤติกรรม Compaction ที่มีมาให้ใน OpenClaw
  </Tab>
</Tabs>

`compact()` ที่ไม่ดำเนินการใดๆ ไม่ปลอดภัยสำหรับเอนจินที่ใช้งานอยู่และไม่ได้ควบคุม Compaction เอง เพราะจะปิดใช้เส้นทาง Compaction ปกติสำหรับ `/compact` และการกู้คืนจากโอเวอร์โฟลว์สำหรับสล็อตเอนจินนั้น

## เอกสารอ้างอิงการกำหนดค่า

```json5
{
  plugins: {
    slots: {
      // เลือกเอนจินบริบทที่ใช้งานอยู่ ค่าเริ่มต้น: "legacy"
      // ตั้งเป็นรหัส Plugin เพื่อใช้เอนจิน Plugin
      contextEngine: "legacy",
    },
  },
}
```

<Note>
สล็อตนี้ใช้ได้เพียงรายการเดียวขณะรัน โดยจะมีการแก้ไขเอนจินบริบทที่ลงทะเบียนเพียงหนึ่งรายการสำหรับการเรียกใช้หรือการดำเนินการ Compaction แต่ละครั้ง Plugin `kind: "context-engine"` อื่นๆ ที่เปิดใช้งานยังคงโหลดและเรียกใช้โค้ดการลงทะเบียนได้; `plugins.slots.contextEngine` เพียงเลือกว่ารหัสเอนจินที่ลงทะเบียนใดที่ OpenClaw จะแก้ไขเมื่อต้องการเอนจินบริบท
</Note>

<Note>
**การถอนการติดตั้ง Plugin:** เมื่อถอนการติดตั้ง Plugin ที่เลือกเป็น `plugins.slots.contextEngine` อยู่ในขณะนั้น OpenClaw จะรีเซ็ตสล็อตกลับเป็นค่าเริ่มต้น (`legacy`) พฤติกรรมการรีเซ็ตแบบเดียวกันนี้ใช้กับ `plugins.slots.memory` โดยไม่จำเป็นต้องแก้ไขการกำหนดค่าด้วยตนเอง
</Note>

## ความสัมพันธ์กับ Compaction และหน่วยความจำ

<AccordionGroup>
  <Accordion title="Compaction">
    Compaction เป็นหนึ่งในความรับผิดชอบของเอนจินบริบท เอนจินแบบเดิมมอบหมายให้ระบบสรุปข้อมูลในตัวของ OpenClaw ส่วนเอนจิน Plugin สามารถใช้กลยุทธ์ Compaction ใดก็ได้ (สรุปแบบ DAG, การเรียกคืนด้วยเวกเตอร์ ฯลฯ)
  </Accordion>
  <Accordion title="Plugin หน่วยความจำ">
    Plugin หน่วยความจำ (`plugins.slots.memory`) แยกจากเอนจินบริบท Plugin หน่วยความจำให้บริการค้นหา/เรียกคืน ส่วนเอนจินบริบทควบคุมสิ่งที่โมเดลมองเห็น ทั้งสองสามารถทำงานร่วมกันได้ โดยเอนจินบริบทอาจใช้ข้อมูลจาก Plugin หน่วยความจำระหว่างการประกอบ เอนจิน Plugin ที่ต้องการเส้นทางพรอมต์หน่วยความจำที่ใช้งานอยู่ควรเลือกใช้ `buildMemorySystemPromptAddition(...)` จาก `openclaw/plugin-sdk/core` ซึ่งจะแปลงส่วนพรอมต์หน่วยความจำที่ใช้งานอยู่ให้เป็น `systemPromptAddition` ที่พร้อมนำไปต่อไว้ด้านหน้า หากเอนจินต้องการควบคุมในระดับต่ำกว่า ก็ยังสามารถดึงบรรทัดดิบจาก `openclaw/plugin-sdk/memory-host-core` ผ่าน `buildActiveMemoryPromptSection(...)` ได้
  </Accordion>
  <Accordion title="การตัดแต่งเซสชัน">
    การตัดผลลัพธ์เครื่องมือเก่าในหน่วยความจำยังคงทำงานไม่ว่าเอนจินบริบทใดจะใช้งานอยู่
  </Accordion>
</AccordionGroup>

## เคล็ดลับ

- ใช้ `openclaw doctor` เพื่อตรวจสอบว่าเอนจินโหลดอย่างถูกต้อง
- หากสลับเอนจิน เซสชันที่มีอยู่จะดำเนินต่อด้วยประวัติปัจจุบัน เอนจินใหม่จะเข้าควบคุมการเรียกใช้ในอนาคต
- ข้อผิดพลาดของเอนจินจะถูกบันทึก และเอนจิน Plugin ที่เลือกจะถูกกักกันสำหรับกระบวนการ Gateway ปัจจุบัน OpenClaw จะย้อนกลับไปใช้ `legacy` สำหรับรอบการโต้ตอบของผู้ใช้เพื่อให้ตอบกลับต่อไปได้ แต่ยังควรซ่อมแซม อัปเดต ปิดใช้งาน หรือถอนการติดตั้ง Plugin ที่เสียหาย
- สำหรับการพัฒนา ให้ใช้ `openclaw plugins install -l ./my-engine` เพื่อเชื่อมโยงไดเรกทอรี Plugin ในเครื่องโดยไม่ต้องคัดลอก
