เวิร์กทรีที่มีการจัดการช่วยให้งานของเอเจนต์มีแบรนช์ git และเช็กเอาต์ของตนเอง โดยไม่สร้างไดเรกทอรีชั่วคราวภายในที่เก็บซอร์สโค้ด OpenClaw สร้างเวิร์กทรีเหล่านี้ภายใต้ไดเรกทอรีสถานะ บันทึกไว้ในฐานข้อมูลสถานะที่ใช้ร่วมกัน และสร้างสแนปช็อตเนื้อหาที่ติดตามและเนื้อหาที่ไม่ได้ติดตามซึ่งไม่ถูกละเว้นก่อนนำออก

## โครงสร้างและชื่อ

เวิร์กทรีแต่ละรายการอยู่ที่:

```text
<openclaw-state-dir>/worktrees/<repo-fingerprint>/<name>
```

ลายนิ้วมือของที่เก็บคืออักขระเลขฐานสิบหก 16 ตัวแรกของแฮช SHA-256 ที่คำนวณจากไดเรกทอรี git ส่วนกลางแบบมาตรฐานและ URL ต้นทาง ชื่อที่ระบุต้องตรงกับ `[a-z0-9][a-z0-9-]{0,63}` หากไม่ระบุชื่อ OpenClaw จะสร้าง `wt-` ตามด้วยอักขระเลขฐานสิบหกแบบสุ่ม 8 ตัว

OpenClaw สร้างแบรนช์ `openclaw/<name>` ที่ base ref ที่ร้องขอ หากไม่มี base ref ระบบจะดึงข้อมูล `origin` ใช้แบรนช์เริ่มต้นของรีโมตเมื่อพร้อมใช้งาน และย้อนกลับไปใช้ `HEAD` ในเครื่องเมื่อที่เก็บออฟไลน์หรือไม่มีรีโมตที่ใช้งานได้

## จัดเตรียมไฟล์ที่ถูกละเว้น

เพิ่ม `.worktreeinclude` ที่รากของที่เก็บซอร์สเพื่อคัดลอกไฟล์ที่ถูกละเว้นและไม่ได้ติดตามซึ่งเลือกไว้ไปยังเวิร์กทรีใหม่ ไฟล์นี้ใช้ไวยากรณ์รูปแบบ gitignore โดยหนึ่งรูปแบบต่อบรรทัด พร้อมความคิดเห็น `#`:

```gitignore
.env.local
fixtures/generated/**
```

เฉพาะไฟล์ที่ git รายงานว่าทั้งถูกละเว้นและไม่ได้ติดตามเท่านั้นที่มีสิทธิ์ ไฟล์ที่ติดตามมีอยู่แล้วผ่าน git และขั้นตอนนี้จะไม่คัดลอกไฟล์เหล่านั้น OpenClaw จะไม่เขียนทับหรือเปลี่ยนไฟล์ปลายทางที่มีอยู่แล้ว ไม่ติดตามไดเรกทอรีที่เป็น symlink และคงโหมดของไฟล์ที่คัดลอกไว้ ระบบบันทึกเฉพาะพาธที่สร้างขึ้นจริง ดังนั้นการแก้ไขไฟล์ manifest ในภายหลังจะไม่ทำให้ไฟล์เหล่านั้นหายไปจากการคุ้มครองระหว่างการล้างข้อมูล

## เรียกใช้การตั้งค่าที่เก็บ

หาก `.openclaw/worktree-setup.sh` มีอยู่ในที่เก็บซอร์สและเรียกทำงานได้ OpenClaw จะเรียกใช้โดยกำหนดให้เวิร์กทรีใหม่เป็นไดเรกทอรีปัจจุบัน สคริปต์จะได้รับ:

```text
OPENCLAW_SOURCE_TREE_PATH=<source checkout>
OPENCLAW_WORKTREE_PATH=<managed worktree>
```

การออกด้วยค่าที่ไม่ใช่ศูนย์จะยกเลิกการสร้างและนำเวิร์กทรีกับแบรนช์ใหม่ออก นี่เป็นสัญญาภายในที่เก็บ และไม่มีคีย์การกำหนดค่า OpenClaw สำหรับสิ่งนี้

## เวิร์กทรีของเซสชัน

เริ่มแชตแบบแยกจากพื้นที่ทำงาน git ของเอเจนต์ที่ใช้งานอยู่ด้วยเซสชันที่ใช้เวิร์กทรี: เปิดใช้ **เวิร์กทรี** ในหน้าเซสชันใหม่ของ Control UI (ซึ่งมีตัวเลือกแบรนช์ฐานและชื่อเวิร์กทรีที่ระบุหรือไม่ก็ได้ด้วย) หรือใช้เมนูการดำเนินการของแชตบน iOS หรือการดำเนินการเพิ่มเติมข้างแชตใหม่บน Android ตัวเลือกนี้พร้อมใช้งานเฉพาะสำหรับเอเจนต์ที่มี git รองรับและไคลเอนต์มีความสามารถดังกล่าวเท่านั้น ไคลเอนต์ที่ไม่สามารถตรวจสอบล่วงหน้าได้จะแสดงข้อผิดพลาดจาก Gateway แทน

เอเจนต์เขียนโค้ดยังสามารถเรียก `spawn_task` เมื่อพบงานติดตามผลที่ยืนยันแล้วซึ่งอยู่นอกงานปัจจุบัน Control UI จะแสดงชิปคำแนะนำโดยไม่เริ่มดำเนินการใด ๆ ส่วน TUI ที่ใช้ Gateway จะแสดงพรอมต์แบบโต้ตอบพร้อมการดำเนินการเดียวกัน การเลือก **เริ่มในเวิร์กทรี** จะสร้างเวิร์กทรีใหม่ที่เซสชันเป็นเจ้าของจากโปรเจกต์ที่แนะนำ และส่งพรอมต์ที่มีข้อมูลครบถ้วนในตัวเองเป็นเทิร์นแรก การปิดคำแนะนำจะไม่แตะต้องที่เก็บ คำแนะนำและ ID ของคำแนะนำเป็นข้อมูลชั่วคราวและจะไม่คงอยู่หลัง Gateway รีสตาร์ต

OpenClaw เปิดเผยเครื่องมือเหล่านี้เฉพาะแก่เซสชันผู้ปฏิบัติงานที่มี UI ของ Gateway ซึ่งดำเนินการได้ เซสชันช่องทางและเซสชัน TUI แบบภายในเครื่อง/ฝังตัวจะไม่ได้รับเครื่องมือเหล่านี้จนกว่าพื้นผิวเหล่านั้นจะมีสัญญาการดำเนินการกับงานแบบมีชนิดที่พกพาได้

เวิร์กทรีที่มีการจัดการซึ่งได้มาจะเป็นของเซสชัน และการเรียกใช้เอเจนต์ทุกครั้งในเซสชันนั้นจะใช้เช็กเอาต์ดังกล่าว เมื่อพื้นที่ทำงานเป็นไดเรกทอรีย่อยของที่เก็บ เวิร์กทรีจะยึดที่รากของที่เก็บ และเซสชันจะทำงานจากไดเรกทอรีย่อยที่ตรงกันภายในเวิร์กทรี การสร้างเวิร์กทรีของเซสชันใช้ขอบเขต `operator.write` ของเมธอด แต่ฮุกเช็กเอาต์ของที่เก็บและขั้นตอน `.openclaw/worktree-setup.sh` จะทำงานเฉพาะสำหรับผู้เรียก `operator.admin` เนื่องจากสิ่งเหล่านี้เรียกใช้โค้ดของที่เก็บ ส่วนการจัดเตรียม `.worktreeinclude` ยังคงใช้กับผู้เรียกทุกราย การลบเซสชันจะนำเวิร์กทรีออกเฉพาะเมื่อทำได้โดยไม่สูญเสียข้อมูล เวิร์กทรีที่มีการเปลี่ยนแปลงหรือแบรนช์ที่มีคอมมิตซึ่งยังไม่ได้พุชจะยังคงพร้อมใช้งาน การล้างข้อมูลรายชั่วโมงจะสร้างสแนปช็อตเวิร์กทรีของเซสชันหลังไม่มีการใช้งานเป็นเวลา 7 วัน โดยถือว่ากิจกรรมล่าสุดของเซสชันเป็นกิจกรรมของเวิร์กทรี เวิร์กทรีที่นำออกแล้วยังคงกู้คืนได้จากสแนปช็อตตามที่อธิบายด้านล่าง

`sessions.create` อาจมี `cwd` แบบสัมบูรณ์ร่วมกับ `worktree: true` เมื่องานกำหนดเป้าหมายไปยังโปรเจกต์อื่นนอกเหนือจากพื้นที่ทำงานของเอเจนต์ที่กำหนดค่าไว้ พาธโฮสต์ที่ระบุอย่างชัดเจนนั้นต้องใช้ `operator.admin` ส่วนการสร้างแชตเวิร์กทรีทั่วไปยังคงเป็น `operator.write` และยึดกับพื้นที่ทำงานที่กำหนดค่าไว้

`sessions.create` ยังยอมรับ `worktreeBaseRef` และ `worktreeName` ร่วมกับ `worktree: true` เพื่อเลือก base ref และชื่อเวิร์กทรี (แบรนช์จะกลายเป็น `openclaw/<name>`) โดยทั้งสองยังคงอยู่ที่ `operator.write` เวิร์กทรีที่สร้างจะถูกส่งคืนในผลลัพธ์การสร้างและจัดเก็บถาวรในแถวเซสชันเป็น `worktree: { id, branch, repoRoot }` เพื่อให้รายการเซสชันแสดงเช็กเอาต์และแบรนช์ได้ การลบเซสชันจะรายงานเช็กเอาต์ที่มีการเปลี่ยนแปลงซึ่งถูกเก็บรักษาไว้เป็น `worktreePreserved` แทนที่จะปล่อยทิ้งไว้อย่างเงียบ ๆ

## สแนปช็อต การล้างข้อมูล และการกู้คืน

การนำออกจะสร้างคอมมิตสังเคราะห์ที่มีไฟล์ที่ติดตามและไฟล์ที่ไม่ได้ติดตามซึ่งไม่ถูกละเว้นก่อน จากนั้นจึงปักหมุดไว้ที่ `refs/openclaw/snapshots/<id>` ไฟล์ที่ถูกละเว้นจะไม่เข้าสู่ฐานข้อมูลอ็อบเจ็กต์ของที่เก็บ OpenClaw จัดเก็บเฉพาะไฟล์ที่ถูกละเว้นซึ่งระบบจัดเตรียมจริงไว้ในแถวฐานข้อมูลสถานะที่ใช้ร่วมกันแบบแบ่งส่วน ชุดพาธที่บันทึกไว้ยังคงเป็นแหล่งข้อมูลที่มีอำนาจ แม้ว่า `.worktreeinclude` จะเปลี่ยนแปลงหรือหายไปในภายหลัง การกู้คืนจะอ่านไบต์เหล่านั้นจากสแนปช็อตที่เปลี่ยนแปลงไม่ได้และนำโหมดทั้งหมดกลับมาใช้ การล้างข้อมูลอัตโนมัติจะเก็บรักษาเวิร์กทรีที่ใช้งานอยู่เมื่อไม่สามารถสร้างสแนปช็อตของพาธที่บันทึกไว้ได้อย่างปลอดภัยอีกต่อไป หากการสร้างสแนปช็อตล้มเหลว การนำออกจะหยุดลง การบังคับลบอย่างชัดเจนสามารถดำเนินการต่อโดยไม่มีสแนปช็อตได้

OpenClaw ใช้กฎการล้างข้อมูลต่อไปนี้:

- เมื่อสิ้นสุดการเรียกใช้ ระบบจะนำเวิร์กทรีออกเฉพาะเมื่อ `git status --porcelain` ว่างเปล่าและ `git log HEAD --not --remotes --oneline` ไม่พบคอมมิตที่ยังไม่ได้พุช มิฉะนั้นระบบจะเพียงปลดล็อกกิจกรรม
- การล้างข้อมูลรายชั่วโมงจะสร้างสแนปช็อตและนำเวิร์กทรีที่ Workboard และเซสชันเป็นเจ้าของซึ่งไม่ได้ใช้งานเกิน 7 วันและไม่ได้ล็อกออก แม้จะมีการเปลี่ยนแปลงอยู่ก็ตาม เวิร์กทรีแบบกำหนดเองจะไม่ถูกนำออกโดยอัตโนมัติ
- เมื่อกำหนดค่า `worktrees.cleanup.maxCount` หรือ `worktrees.cleanup.maxTotalSizeGb` การล้างข้อมูลจะสร้างสแนปช็อตและนำเวิร์กทรีที่ Workboard และเซสชันเป็นเจ้าของซึ่งมีกิจกรรมล่าสุดเก่าที่สุดออกด้วย จนกว่าจำนวนรวมและขนาดดิสก์จะอยู่ภายในขีดจำกัด เวิร์กทรีที่มีการจัดการทั้งหมดจะถูกนับในยอดรวม แต่เวิร์กทรีแบบกำหนดเองและเวิร์กทรีที่ได้รับการคุ้มครองด้วยเหตุผลอื่นจะไม่ถูกขับออกเนื่องจากขีดจำกัด ดังนั้นขีดจำกัดอาจยังคงเกินอยู่จนกว่าจะมีเวิร์กทรีที่มีสิทธิ์ ค่า 0 หรือไม่ได้กำหนดจะปิดใช้ขีดจำกัด
- ระเบียนสแนปช็อตจะกู้คืนได้เป็นเวลา 30 วัน จากนั้นการล้างข้อมูลจะลบ ref ของสแนปช็อตและแถวรีจิสทรี
- ล็อกของกระบวนการ OpenClaw ที่กำลังทำงานอยู่ รวมถึงล็อกเวิร์กทรี git จากภายนอกหรือที่ไม่รู้จัก จะปกป้องเวิร์กทรีจากการเก็บกวาดขยะ

การกู้คืนจะสร้าง `openclaw/<name>` ขึ้นใหม่ที่คอมมิตเดิมก่อนสร้างสแนปช็อต จากนั้นสร้างความแตกต่างของสแนปช็อตกลับมาเป็นการแก้ไขที่ยังไม่ได้ stage และไฟล์ที่ไม่ได้ติดตาม วิธีนี้ช่วยไม่ให้คอมมิตสแนปช็อตสังเคราะห์เข้าสู่ประวัติแบรนช์ ref ของสแนปช็อตยังคงถูกบันทึกไว้เป็นหลักฐานที่มา

## CLI

```bash
openclaw worktrees list [--json]
openclaw worktrees create <repo-root> [--name <name>] [--base-ref <ref>] [--json]
openclaw worktrees remove <id> [--force] [--json]
openclaw worktrees restore <id> [--json]
openclaw worktrees gc [--json]
```

หน้า **เวิร์กทรี** ของ Control UI ภายใต้การตั้งค่ามีการดำเนินการเดียวกัน พร้อมทั้งการสร้างโดยใช้ตัวเลือกแบรนช์ฐาน แสดงเจ้าของเวิร์กทรีแต่ละรายการ (กำหนดเอง, Workboard หรือเซสชันเจ้าของพร้อมลิงก์ไปยังแชต) และมีตัวเลือกบังคับลองใหม่เมื่อการนำออกรายงานว่าสร้างสแนปช็อตล้มเหลว ส่วน **การล้างข้อมูล** ใช้แก้ไขขีดจำกัดการเก็บรักษา `worktrees.cleanup` ที่อธิบายไว้ใน[เอกสารอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference#worktrees)

| `worktrees.list`     | แสดงระเบียนเวิร์กทรีที่ใช้งานอยู่และกู้คืนได้                            |
| `worktrees.branches` | แสดงแบรนช์ภายในเครื่องและรีโมตของที่เก็บสำหรับตัวเลือก base ref    |
| `worktrees.create`   | สร้างหรือนำเวิร์กทรีที่มีการจัดการตามชื่อกลับมาใช้                               |
| `worktrees.remove`   | สร้างสแนปช็อตและนำเวิร์กทรีออก การบังคับนำออกจะรายงาน `snapshotError` |
| `worktrees.restore`  | กู้คืนเวิร์กทรีที่นำออกแล้วจากสแนปช็อต                           |
| `worktrees.gc`       | เรียกใช้การล้างข้อมูลรายการที่ไม่ได้ใช้งาน รายการกำพร้า และตามการเก็บรักษาทันที                            |

`worktrees.list` ต้องใช้ `operator.read` และเมธอดที่เปลี่ยนแปลงข้อมูลต้องใช้ `operator.admin` ส่วน `worktrees.branches` ต้องใช้ `operator.write` สำหรับพื้นที่ทำงานของเอเจนต์ที่กำหนดค่าไว้ ขณะที่พาธโฮสต์อื่นต้องใช้ `operator.admin` (ตรงกับเกณฑ์ cwd ของ `sessions.create`) เมธอดนี้อ่านเฉพาะ ref ที่มีอยู่และไม่ดึงข้อมูล และแบรนช์ที่มีเฉพาะบนรีโมตจะถูกส่งคืนโดยมีคุณสมบัติรีโมตกำกับ (`origin/feature-a`) เพื่อให้ชื่อที่ส่งคืนทุกชื่อแปลงเป็น base ref ได้

## พื้นที่ทำงานของ Workboard

[Plugin Workboard](/th/plugins/workboard) ที่รวมมาให้สามารถสร้างพื้นที่ทำงานของการ์ดเป็นเวิร์กทรีที่มีการจัดการได้:

```json
{
  "kind": "worktree",
  "path": "/absolute/path/to/source-checkout",
  "branch": "main"
}
```

`path` ระบุเช็กเอาต์ git ต้นทาง ส่วน `branch` ระบุหรือไม่ก็ได้และจะกลายเป็น base ref สำหรับผู้เรียกที่เข้าถึงโฮสต์เต็มรูปแบบ Workboard จะสร้างหรือนำ `wb-<card-id>` กลับมาใช้ เรียกใช้เอเจนต์ย่อยโดยกำหนดให้เช็กเอาต์ที่มีการจัดการเป็นไดเรกทอรีทำงาน และเขียนพาธกับแบรนช์ที่แปลงแล้วกลับไปยังการ์ด ไคลเอนต์ Gateway ต้องใช้ `operator.admin` สำหรับการสร้างบนโฮสต์เต็มรูปแบบ เมื่อสิ้นสุดการเรียกใช้ Workboard จะนำเช็กเอาต์ออกเฉพาะเมื่อพิสูจน์ได้ว่าจะไม่สูญเสียข้อมูล งานที่มีการเปลี่ยนแปลงหรือคอมมิตที่ยังไม่ได้พุชจะยังคงพร้อมใช้งาน

สำหรับผู้เรียกที่ผูกกับพื้นที่ทำงาน `path` และรากของที่เก็บต้องตรงกับพื้นที่ทำงานของเอเจนต์เป้าหมายทุกประการ จากนั้น Workboard จะทำงานโดยตรงในไดเรกทอรีนั้นและบันทึกพื้นที่ทำงานแบบไดเรกทอรีแทนการสร้างเวิร์กทรีที่มีการจัดการบนโฮสต์ เป้าหมายต้องใช้ sandbox Docker ที่เขียนได้และไม่ได้ใช้ร่วมกันสำหรับพื้นที่ทำงานเดียวกัน แฮชคอนเทนเนอร์ที่ใช้งานอยู่ต้องตรงกับเมานต์และนโยบายที่ร้องขอ และต้องไม่เปิดเผยการดำเนินการแบบยกระดับ การควบคุมโฮสต์ เซสชันทั่วทั้งโฮสต์ การดำเนินการบนโฮสต์/Node ที่จัดเก็บถาวร หรือ Plugin และเครื่องมือ MCP ที่ยังไม่ได้จัดประเภท หากนโยบายเป้าหมายหรือคอนเทนเนอร์ที่ใช้งานอยู่มีขอบเขตกว้างกว่า การส่งงานจะปล่อยการ์ดไว้โดยไม่มีผู้รับและรายงานสถานะที่เข้ากันไม่ได้
