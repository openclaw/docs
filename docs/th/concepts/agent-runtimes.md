**รันไทม์เอเจนต์** คือคอมโพเนนต์ที่เป็นเจ้าของลูปโมเดลที่เตรียมไว้หนึ่งลูป โดย
รับพรอมป์ ขับเคลื่อนเอาต์พุตของโมเดล จัดการการเรียกใช้เครื่องมือเนทีฟ และส่งคืน
เทิร์นที่เสร็จสมบูรณ์ให้ OpenClaw

รันไทม์มักทำให้สับสนกับผู้ให้บริการได้ง่าย เพราะทั้งสองอย่างปรากฏอยู่ใกล้กับการกำหนดค่า
โมเดล ทั้งสองเป็นเลเยอร์ที่ต่างกัน:

| เลเยอร์       | ตัวอย่าง                              | ความหมาย                                                              |
| ------------- | ------------------------------------- | --------------------------------------------------------------------- |
| ผู้ให้บริการ  | `openai`, `anthropic`, `openai-codex` | วิธีที่ OpenClaw รับรองความถูกต้อง ค้นพบโมเดล และตั้งชื่อการอ้างอิงโมเดล |
| โมเดล         | `gpt-5.5`, `claude-opus-4-6`          | โมเดลที่เลือกสำหรับเทิร์นของเอเจนต์                                  |
| รันไทม์เอเจนต์ | `pi`, `codex`, `claude-cli`           | ลูประดับล่างหรือแบ็กเอนด์ที่ดำเนินการเทิร์นที่เตรียมไว้               |
| ช่องทาง       | Telegram, Discord, Slack, WhatsApp    | ที่ที่ข้อความเข้าและออกจาก OpenClaw                                  |

คุณจะเห็นคำว่า **ฮาร์เนส** ในโค้ดด้วย ฮาร์เนสคือการใช้งานจริง
ที่ให้รันไทม์เอเจนต์ ตัวอย่างเช่น ฮาร์เนส Codex ที่รวมมาให้
ใช้งานรันไทม์ `codex` การกำหนดค่าสาธารณะใช้ `agentRuntime.id`; `openclaw
doctor --fix` เขียนคีย์ runtime-policy รุ่นเก่าใหม่ให้อยู่ในรูปแบบนั้น

รันไทม์มีสองตระกูล:

- **ฮาร์เนสแบบฝังตัว** ทำงานภายในลูปเอเจนต์ที่ OpenClaw เตรียมไว้ ปัจจุบันนี้
  คือรันไทม์ `pi` ในตัว รวมถึงฮาร์เนส Plugin ที่ลงทะเบียนไว้ เช่น
  `codex`
- **แบ็กเอนด์ CLI** เรียกใช้โปรเซส CLI ภายในเครื่องพร้อมกับคงการอ้างอิงโมเดลไว้
  เป็นแบบแคนนอนิคัล ตัวอย่างเช่น `anthropic/claude-opus-4-7` พร้อม
  `agentRuntime.id: "claude-cli"` หมายถึง "เลือกโมเดล Anthropic แล้วดำเนินการ
  ผ่าน Claude CLI" `claude-cli` ไม่ใช่รหัสฮาร์เนสแบบฝังตัว และต้องไม่
  ส่งให้การเลือก AgentHarness

## พื้นผิว Codex

ความสับสนส่วนใหญ่มาจากพื้นผิวหลายแบบที่ใช้ชื่อ Codex ร่วมกัน:

| พื้นผิว                                               | ชื่อ/การกำหนดค่าใน OpenClaw               | สิ่งที่ทำ                                                                                                  |
| ---------------------------------------------------- | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| รันไทม์ app-server ของ Codex แบบเนทีฟ                | `openai/*` plus `agentRuntime.id: "codex"` | เรียกใช้เทิร์นเอเจนต์แบบฝังตัวผ่าน Codex app-server นี่คือการตั้งค่าการสมัครใช้งาน ChatGPT/Codex โดยทั่วไป |
| เส้นทางผู้ให้บริการ OAuth ของ Codex                  | `openai-codex/*` การอ้างอิงโมเดล           | ใช้ OAuth การสมัครใช้งาน ChatGPT/Codex ผ่านตัวรัน OpenClaw PI ปกติ                                      |
| อะแดปเตอร์ ACP ของ Codex                             | `runtime: "acp"`, `agentId: "codex"`       | เรียกใช้ Codex ผ่านระนาบควบคุม ACP/acpx ภายนอก ใช้เฉพาะเมื่อมีการขอ ACP/acpx อย่างชัดเจน                 |
| ชุดคำสั่งควบคุมแชตของ Codex แบบเนทีฟ                 | `/codex ...`                               | ผูก ทำงานต่อ ชี้นำ หยุด และตรวจสอบเธรด Codex app-server จากแชต                                           |
| เส้นทาง OpenAI Platform API สำหรับโมเดลแบบ GPT/Codex | `openai/*` การอ้างอิงโมเดล                | ใช้การรับรองความถูกต้องด้วยคีย์ API ของ OpenAI เว้นแต่การแทนที่รันไทม์ เช่น `agentRuntime.id: "codex"` จะเรียกใช้เทิร์น |

พื้นผิวเหล่านั้นแยกจากกันโดยตั้งใจ การเปิดใช้ Plugin `codex` ทำให้
ฟีเจอร์ app-server แบบเนทีฟพร้อมใช้งาน แต่ไม่ได้เขียน
`openai-codex/*` ใหม่เป็น `openai/*` ไม่เปลี่ยนเซสชันที่มีอยู่ และไม่ได้
ทำให้ ACP เป็นค่าเริ่มต้นของ Codex การเลือก `openai-codex/*` หมายถึง "ใช้เส้นทาง
ผู้ให้บริการ OAuth ของ Codex" เว้นแต่คุณจะบังคับรันไทม์แยกต่างหาก

การตั้งค่าการสมัครใช้งาน ChatGPT/Codex ที่พบบ่อยใช้ OAuth ของ Codex สำหรับการรับรองความถูกต้อง แต่ยังคง
การอ้างอิงโมเดลเป็น `openai/*` และเลือกรันไทม์ `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

นั่นหมายความว่า OpenClaw เลือกการอ้างอิงโมเดล OpenAI แล้วขอให้รันไทม์ Codex app-server
เรียกใช้เทิร์นเอเจนต์แบบฝังตัว ไม่ได้หมายความว่า "ใช้การคิดค่าบริการผ่าน API" และ
ไม่ได้หมายความว่าช่องทาง แค็ตตาล็อกผู้ให้บริการโมเดล หรือที่เก็บเซสชันของ OpenClaw
จะกลายเป็น Codex

เมื่อเปิดใช้ Plugin `codex` ที่รวมมาให้ การควบคุม Codex ด้วยภาษาธรรมชาติ
ควรใช้พื้นผิวคำสั่ง `/codex` แบบเนทีฟ (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`) แทน ACP ใช้ ACP สำหรับ
Codex เฉพาะเมื่อผู้ใช้ขอ ACP/acpx อย่างชัดเจน หรือกำลังทดสอบเส้นทางอะแดปเตอร์
ACP เท่านั้น Claude Code, Gemini CLI, OpenCode, Cursor และฮาร์เนสภายนอกที่คล้ายกัน
ยังคงใช้ ACP

นี่คือต้นไม้การตัดสินใจสำหรับเอเจนต์:

1. หากผู้ใช้ขอ **Codex bind/control/thread/resume/steer/stop** ให้ใช้
   พื้นผิวคำสั่ง `/codex` แบบเนทีฟเมื่อเปิดใช้ Plugin `codex` ที่รวมมาให้
2. หากผู้ใช้ขอ **Codex เป็นรันไทม์แบบฝังตัว** หรือต้องการประสบการณ์เอเจนต์ Codex
   แบบปกติที่รองรับด้วยการสมัครใช้งาน ให้ใช้
   `openai/<model>` พร้อม `agentRuntime.id: "codex"`
3. หากผู้ใช้ขอ **การรับรองความถูกต้องด้วย OAuth/การสมัครใช้งานของ Codex บนตัวรัน OpenClaw
   ปกติ** ให้ใช้ `openai-codex/<model>` และปล่อยรันไทม์เป็น PI
4. หากผู้ใช้พูดอย่างชัดเจนว่า **ACP**, **acpx** หรือ **อะแดปเตอร์ ACP ของ Codex** ให้ใช้
   ACP พร้อม `runtime: "acp"` และ `agentId: "codex"`
5. หากคำขอเป็นของ **Claude Code, Gemini CLI, OpenCode, Cursor, Droid หรือ
   ฮาร์เนสภายนอกอื่น** ให้ใช้ ACP/acpx ไม่ใช่รันไทม์ซับเอเจนต์แบบเนทีฟ

| คุณหมายถึง...                         | ให้ใช้...                                     |
| --------------------------------------- | -------------------------------------------- |
| การควบคุมแชต/เธรด Codex app-server     | `/codex ...` จาก Plugin `codex` ที่รวมมาให้ |
| รันไทม์เอเจนต์แบบฝังตัว Codex app-server | `agentRuntime.id: "codex"`                   |
| OpenAI Codex OAuth บนตัวรัน PI          | `openai-codex/*` การอ้างอิงโมเดล             |
| Claude Code หรือฮาร์เนสภายนอกอื่น      | ACP/acpx                                     |

สำหรับการแยกคำนำหน้าตระกูล OpenAI โปรดดู [OpenAI](/th/providers/openai) และ
[ผู้ให้บริการโมเดล](/th/concepts/model-providers) สำหรับสัญญาการรองรับรันไทม์ Codex
โปรดดู [ฮาร์เนส Codex](/th/plugins/codex-harness#v1-support-contract)

## ความเป็นเจ้าของรันไทม์

รันไทม์ต่างกันเป็นเจ้าของลูปในปริมาณที่ต่างกัน

| พื้นผิว                    | OpenClaw PI แบบฝังตัว                  | Codex app-server                                                            |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| เจ้าของลูปโมเดล             | OpenClaw ผ่านตัวรัน PI แบบฝังตัว       | Codex app-server                                                            |
| สถานะเธรดแคนนอนิคัล        | ทรานสคริปต์ OpenClaw                   | เธรด Codex พร้อมกระจกทรานสคริปต์ OpenClaw                                  |
| เครื่องมือไดนามิกของ OpenClaw | ลูปเครื่องมือ OpenClaw แบบเนทีฟ        | เชื่อมผ่านอะแดปเตอร์ Codex                                                  |
| เครื่องมือเชลล์และไฟล์แบบเนทีฟ | เส้นทาง PI/OpenClaw                    | เครื่องมือเนทีฟของ Codex เชื่อมผ่านฮุกเนทีฟเมื่อรองรับ                     |
| เอนจินบริบท                 | การประกอบบริบท OpenClaw แบบเนทีฟ       | OpenClaw โปรเจกต์บริบทที่ประกอบแล้วเข้าสู่เทิร์น Codex                     |
| Compaction                  | OpenClaw หรือเอนจินบริบทที่เลือก       | Compaction แบบเนทีฟของ Codex พร้อมการแจ้งเตือนและการดูแลกระจกโดย OpenClaw |
| การส่งผ่านช่องทาง           | OpenClaw                                | OpenClaw                                                                    |

การแบ่งความเป็นเจ้าของนี้คือกฎการออกแบบหลัก:

- หาก OpenClaw เป็นเจ้าของพื้นผิว OpenClaw สามารถให้พฤติกรรมฮุก Plugin ตามปกติได้
- หากรันไทม์เนทีฟเป็นเจ้าของพื้นผิว OpenClaw ต้องมีอีเวนต์รันไทม์หรือฮุกเนทีฟ
- หากรันไทม์เนทีฟเป็นเจ้าของสถานะเธรดแคนนอนิคัล OpenClaw ควรทำกระจกและฉายบริบท ไม่ใช่เขียนภายในที่ไม่รองรับใหม่

## การเลือกรันไทม์

OpenClaw เลือกรันไทม์แบบฝังตัวหลังจากแก้ไขผู้ให้บริการและโมเดลแล้ว:

1. รันไทม์ที่บันทึกไว้ของเซสชันชนะ การเปลี่ยนการกำหนดค่าจะไม่สลับทรานสคริปต์
   ที่มีอยู่แบบทันทีไปยังระบบเธรดเนทีฟอื่น
2. `OPENCLAW_AGENT_RUNTIME=<id>` บังคับรันไทม์นั้นสำหรับเซสชันใหม่หรือเซสชันที่รีเซ็ต
3. `agents.defaults.agentRuntime.id` หรือ `agents.list[].agentRuntime.id` สามารถตั้งเป็น
   `auto`, `pi`, รหัสฮาร์เนสแบบฝังตัวที่ลงทะเบียนไว้ เช่น `codex` หรือ
   นามแฝงแบ็กเอนด์ CLI ที่รองรับ เช่น `claude-cli`
4. ในโหมด `auto` รันไทม์ Plugin ที่ลงทะเบียนไว้สามารถอ้างสิทธิ์คู่ผู้ให้บริการ/โมเดล
   ที่รองรับได้
5. หากไม่มีรันไทม์ใดอ้างสิทธิ์เทิร์นในโหมด `auto` OpenClaw จะใช้ PI เป็น
   รันไทม์ความเข้ากันได้ ใช้รหัสรันไทม์แบบชัดเจนเมื่อการรันต้องเข้มงวด

รันไทม์ Plugin แบบชัดเจนจะล้มเหลวแบบปิด ตัวอย่างเช่น `agentRuntime.id: "codex"`
หมายถึง Codex หรือข้อผิดพลาดการเลือก/รันไทม์ที่ชัดเจน โดยจะไม่มีวันถูกส่งต่อกลับ
ไปยัง PI แบบเงียบๆ

นามแฝงแบ็กเอนด์ CLI แตกต่างจากรหัสฮาร์เนสแบบฝังตัว รูปแบบ Claude CLI
ที่แนะนำคือ:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

การอ้างอิงรุ่นเก่า เช่น `claude-cli/claude-opus-4-7` ยังคงรองรับเพื่อ
ความเข้ากันได้ แต่การกำหนดค่าใหม่ควรคงผู้ให้บริการ/โมเดลไว้เป็นแบบแคนนอนิคัล และใส่
แบ็กเอนด์การดำเนินการไว้ใน `agentRuntime.id`

โหมด `auto` ตั้งใจให้ระมัดระวัง รันไทม์ Plugin สามารถอ้างสิทธิ์
คู่ผู้ให้บริการ/โมเดลที่ตนเข้าใจได้ แต่ Plugin Codex จะไม่อ้างสิทธิ์
ผู้ให้บริการ `openai-codex` ในโหมด `auto` นั่นทำให้
`openai-codex/*` ยังคงเป็นเส้นทาง OAuth ของ Codex แบบชัดเจนบน PI และหลีกเลี่ยงการ
ย้ายการกำหนดค่าการรับรองความถูกต้องด้วยการสมัครใช้งานไปยังฮาร์เนส app-server แบบเนทีฟ
อย่างเงียบๆ

หาก `openclaw doctor` เตือนว่าเปิดใช้ Plugin `codex` อยู่ขณะที่
`openai-codex/*` ยังส่งผ่าน PI ให้ถือว่านั่นเป็นการวินิจฉัย ไม่ใช่การ
ย้ายข้อมูล คงการกำหนดค่าไว้ตามเดิมเมื่อ PI Codex OAuth คือสิ่งที่คุณต้องการ
เปลี่ยนเป็น `openai/<model>` พร้อม `agentRuntime.id: "codex"` เฉพาะเมื่อคุณต้องการการดำเนินการ
Codex app-server แบบเนทีฟ

## สัญญาความเข้ากันได้

เมื่อรันไทม์ไม่ใช่ PI ควรมีเอกสารว่ารองรับพื้นผิวใดของ OpenClaw บ้าง
ใช้รูปแบบนี้สำหรับเอกสารรันไทม์:

## ป้ายกำกับสถานะ

เอาต์พุตสถานะอาจแสดงทั้งป้ายกำกับ `Execution` และ `Runtime` ให้อ่านเป็น
ข้อมูลวินิจฉัย ไม่ใช่ชื่อผู้ให้บริการ

- การอ้างอิงโมเดล เช่น `openai/gpt-5.5` จะบอกผู้ให้บริการ/โมเดลที่เลือก
- รหัส Runtime เช่น `codex` จะบอกว่าลูปใดกำลังดำเนินการเทิร์นนี้
- ป้ายกำกับช่องทาง เช่น Telegram หรือ Discord จะบอกว่าบทสนทนากำลังเกิดขึ้นที่ใด

หากเซสชันยังคงแสดง PI หลังจากเปลี่ยนการกำหนดค่า Runtime ให้เริ่มเซสชันใหม่
ด้วย `/new` หรือล้างเซสชันปัจจุบันด้วย `/reset` เซสชันที่มีอยู่จะเก็บ
Runtime ที่บันทึกไว้ เพื่อไม่ให้ทรานสคริปต์ถูกเล่นซ้ำผ่านระบบเซสชันเนทีฟ
สองระบบที่เข้ากันไม่ได้
