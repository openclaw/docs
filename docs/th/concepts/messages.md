ข้อความขาเข้าจะผ่านการกำหนดเส้นทาง การขจัดรายการซ้ำ/การหน่วงรวม การทำงานของเอเจนต์ และการส่งออก:

```text
ข้อความขาเข้า
  -> การกำหนดเส้นทาง/การผูก -> คีย์เซสชัน
  -> ขจัดรายการซ้ำ + หน่วงรวม
  -> คิว (หากมีการทำงานที่กำลังใช้งานอยู่แล้ว)
  -> การทำงานของเอเจนต์ (สตรีมมิง + เครื่องมือ)
  -> การตอบกลับขาออก (ขีดจำกัดของช่องทาง + การแบ่งส่วน)
```

พื้นผิวการกำหนดค่าหลัก:

- `messages.*` สำหรับคำนำหน้า การเข้าคิว การหน่วงรวมขาเข้า และพฤติกรรมของกลุ่ม
- `agents.defaults.*` สำหรับสตรีมมิงแบบบล็อก การแบ่งส่วน และค่าเริ่มต้นของการตอบกลับแบบเงียบ
- การแทนค่าระดับช่องทาง (`channels.telegram.*`, `channels.whatsapp.*` เป็นต้น) สำหรับขีดจำกัดและตัวสลับสตรีมมิงของแต่ละช่องทาง

ดูสคีมาทั้งหมดที่ [การกำหนดค่า](/th/gateway/configuration)

## การขจัดข้อความขาเข้าซ้ำ

ช่องทางอาจส่งข้อความเดิมซ้ำหลังจากเชื่อมต่อใหม่ OpenClaw เก็บแคชในหน่วยความจำซึ่งกำหนดคีย์ตามขอบเขตของเอเจนต์ เส้นทางช่องทาง (ช่องทาง + คู่สนทนา + บัญชี + เธรด) และรหัสข้อความ เพื่อไม่ให้ข้อความที่ส่งซ้ำเรียกใช้เอเจนต์เป็นครั้งที่สอง รายการแคชจะหมดอายุหลังจาก 20 นาทีหรือเมื่อมีการติดตามครบ 5000 รายการ แล้วแต่ว่าเงื่อนไขใดจะเกิดขึ้นก่อน

## การหน่วงรวมข้อความขาเข้า

ข้อความตัวอักษรที่ส่งต่อเนื่องอย่างรวดเร็วจากผู้ส่งรายเดียวกันสามารถรวมเป็นเทิร์นเดียวของเอเจนต์ผ่าน `messages.inbound` การหน่วงรวมมีขอบเขตแยกตามช่องทาง + การสนทนา และใช้ข้อความล่าสุดสำหรับเธรด/รหัสการตอบกลับ

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        discord: 1500,
        slack: 1500,
        whatsapp: 5000,
      },
    },
  },
}
```

- การหน่วงรวมใช้กับข้อความที่มีเฉพาะตัวอักษร ส่วนสื่อ/ไฟล์แนบจะส่งออกทันที
- คำสั่งควบคุม (หยุด/ยกเลิก/สถานะ เป็นต้น) จะข้ามการหน่วงรวมเพื่อให้ส่งต่อทันที
- ปิดใช้งานโดยค่าเริ่มต้น: `messages.inbound.debounceMs` ไม่มีค่าเริ่มต้นในตัว ดังนั้นการหน่วงรวมจะเริ่มทำงานต่อเมื่อกำหนดค่าแล้วเท่านั้น (ทั่วทั้งระบบหรือรายช่องทาง)
- การเลือกใช้ `coalesceSameSenderDms` ของ iMessage เป็นข้อยกเว้นเพียงกรณีเดียว โดยจะพักข้อความ DM ทั้งหมดจากผู้ส่งรายเดียวกัน (รวมถึงคำสั่ง) ไว้นานพอให้การส่งคำสั่ง+URL แบบแยกข้อความของ Apple มาถึงเป็นเทิร์นเดียว แชตกลุ่มจะส่งต่อทันทีเสมอโดยไม่ขึ้นกับการตั้งค่านี้

## เซสชันและอุปกรณ์

เซสชันอยู่ภายใต้การดูแลของ Gateway ไม่ใช่ไคลเอนต์

- แชตโดยตรงจะถูกรวมไว้ในคีย์เซสชันหลักของเอเจนต์
- กลุ่ม/ช่องทางจะมีคีย์เซสชันของตนเอง
- ที่เก็บเซสชันและบทถอดความอยู่บนโฮสต์ของ Gateway

อุปกรณ์/ช่องทางหลายรายการสามารถแมปไปยังเซสชันเดียวกันได้ แต่ประวัติจะไม่ถูกซิงค์กลับไปยังไคลเอนต์ทุกตัวอย่างสมบูรณ์ ใช้อุปกรณ์หลักเพียงเครื่องเดียวสำหรับการสนทนาที่ยาวนานเพื่อหลีกเลี่ยงบริบทที่แตกต่างกัน Control UI และ TUI จะแสดงบทถอดความของเซสชันที่มี Gateway รองรับเสมอ จึงเป็นแหล่งข้อมูลจริงที่เชื่อถือได้

รายละเอียด: [การจัดการเซสชัน](/th/concepts/session)

## เนื้อหาพรอมต์และบริบทประวัติ

Plugin ช่องทางจะเติมข้อมูลในช่องข้อความหลายช่องของบริบทขาเข้า โดยเรียงจากที่ควรใช้มากที่สุดไปน้อยที่สุด:

| ฟิลด์             | วัตถุประสงค์                                                                                                     |
| ----------------- | ----------------------------------------------------------------------------------------------------------- |
| `BodyForAgent`    | ข้อความสำหรับโมเดลในเทิร์นปัจจุบัน หากไม่ได้ตั้งค่าจะใช้ `CommandBody` / `RawBody` / `Body` แทน        |
| `BodyForCommands` | ข้อความสะอาดที่ใช้แยกวิเคราะห์ไดเรกทีฟ/คำสั่ง หากไม่ได้ตั้งค่าจะใช้ `CommandBody` / `RawBody` / `Body` แทน |
| `CommandBody`     | เนื้อหาระหว่างกลางแบบเดิม ควรใช้ `BodyForCommands`                                                         |
| `RawBody`         | นามแฝงที่เลิกใช้แล้วสำหรับ `CommandBody`                                                                         |
| `Body`            | เนื้อหาพรอมต์แบบเดิม อาจมีซองข้อมูลของช่องทางและตัวครอบประวัติ                                     |

เมื่อช่องทางส่งประวัติมา จะครอบประวัติด้วย:

- `[Chat messages since your last reply - for context]`
- `[Current message - respond to this]`

สำหรับแชตที่ไม่ใช่แชตโดยตรง (กลุ่ม/ช่องทาง/ห้อง) เนื้อหาข้อความปัจจุบันจะมีป้ายชื่อผู้ส่งนำหน้า โดยใช้รูปแบบเดียวกับรายการประวัติ การตัดไดเรกทีฟจะใช้กับส่วนข้อความปัจจุบันเท่านั้น ประวัติจึงยังคงอยู่ครบถ้วน ช่องทางที่ครอบประวัติควรตั้งค่า `BodyForCommands` (หรือ `CommandBody` / `RawBody` แบบเดิม) เป็นข้อความต้นฉบับ และเก็บ `Body` เป็นพรอมต์ที่รวมแล้ว

บัฟเฟอร์ประวัติประกอบด้วยเฉพาะรายการที่รอดำเนินการ โดยรวมข้อความกลุ่มที่ไม่ได้เรียกให้เกิดการทำงาน (เช่น ข้อความที่ต้องมีการกล่าวถึง) และไม่รวมข้อความที่อยู่ในบทถอดความของเซสชันแล้ว ประวัติแบบมีโครงสร้าง ข้อมูลการตอบกลับ การส่งต่อ และเมทาดาทาของช่องทางจะถูกเรนเดอร์เป็นบล็อกบริบทบทบาทผู้ใช้ที่ไม่น่าเชื่อถือระหว่างประกอบพรอมต์

กำหนดขนาดประวัติด้วย `messages.groupChat.historyLimit` (ค่าเริ่มต้นทั่วทั้งระบบ) หรือการแทนค่ารายช่องทาง เช่น `channels.slack.historyLimit` และ `channels.telegram.accounts.<id>.historyLimit` (ตั้งค่า `0` เพื่อปิดใช้งาน)

## เมทาดาทาผลลัพธ์ของเครื่องมือ

`content` ของผลลัพธ์เครื่องมือคือผลลัพธ์ที่โมเดลมองเห็น ส่วน `details` คือเมทาดาทารันไทม์สำหรับการเรนเดอร์ UI การวินิจฉัย การส่งสื่อ และ Plugin

- `toolResult.details` จะถูกตัดออกก่อนส่งซ้ำไปยังผู้ให้บริการและก่อนป้อนข้อมูลเข้า Compaction
- บทถอดความเซสชันที่จัดเก็บถาวรจะเก็บเฉพาะ `details` ที่มีขนาดจำกัด เมทาดาทาที่มีขนาดใหญ่เกินไปจะถูกแทนที่ด้วยสรุปแบบกระชับที่ทำเครื่องหมายว่า `persistedDetailsTruncated: true`
- Plugin และเครื่องมือควรใส่ข้อความที่โมเดลต้องอ่านไว้ใน `content` ไม่ใช่เฉพาะใน `details`

## การเข้าคิวและรายการติดตามผล

เมื่อมีการทำงานที่กำลังใช้งานอยู่ ข้อความขาเข้าจะถูกนำไปกำกับการทำงานนั้นโดยค่าเริ่มต้น `messages.queue` ใช้ควบคุมโหมด:

| โหมด              | พฤติกรรม                                            |
| ----------------- | --------------------------------------------------- |
| `steer` (ค่าเริ่มต้น) | แทรกพรอมต์ใหม่เข้าในการทำงานที่กำลังใช้งานอยู่          |
| `followup`        | เรียกใช้ข้อความหลังจากการทำงานที่กำลังใช้งานอยู่เสร็จสิ้น      |
| `collect`         | รวมข้อความที่เข้ากันได้เป็นเทิร์นเดียวในภายหลัง      |
| `interrupt`       | ยกเลิกการทำงานที่กำลังใช้งานอยู่ แล้วเริ่มพรอมต์ล่าสุด |

ค่าเริ่มต้น: `messages.queue.debounceMs` คือ 500ms (ใช้กับการกำกับ การติดตามผล และการรวมชุดแบบรวบรวมเหมือนกัน) `messages.queue.cap` คือข้อความในคิว 20 ข้อความ และ `messages.queue.drop` คือ `summarize` (มี `old` และ `new` ให้ใช้เช่นกัน) กำหนดการแทนค่ารายช่องทางผ่าน `messages.queue.byChannel` และ `messages.queue.debounceMsByChannel`

รายละเอียด: [คิวคำสั่ง](/th/concepts/queue) และ [คิวการกำกับ](/th/concepts/queue-steering)

## ความเป็นเจ้าของการทำงานของช่องทาง

Plugin ช่องทางอาจรักษาลำดับ หน่วงรวมอินพุต และใช้แรงกดกลับของการขนส่งก่อนที่ข้อความจะเข้าสู่คิวเซสชัน แต่ไม่ควรกำหนดระยะหมดเวลาแยกต่างหากครอบเทิร์นของเอเจนต์ เมื่อข้อความถูกกำหนดเส้นทางไปยังเซสชันแล้ว วงจรชีวิตของเซสชัน เครื่องมือ และรันไทม์จะควบคุมงานที่ใช้เวลานาน เพื่อให้ทุกช่องทางรายงานและกู้คืนจากเทิร์นที่ล่าช้าได้อย่างสอดคล้องกัน

## สตรีมมิง การแบ่งส่วน และการรวมชุด

สตรีมมิงแบบบล็อกจะส่งคำตอบบางส่วนเมื่อโมเดลสร้างบล็อกข้อความ ส่วนการแบ่งส่วนจะเคารพขีดจำกัดข้อความของช่องทางและหลีกเลี่ยงการแยกโค้ดที่มีรั้วล้อม

- `agents.defaults.blockStreamingDefault` (`on|off`, ค่าเริ่มต้น `off`)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (การรวมชุดตามช่วงที่ไม่มีการใช้งาน)
- `agents.defaults.humanDelay` (การหยุดพักแบบมนุษย์ระหว่างคำตอบแบบบล็อก)
- การแทนค่าของช่องทาง: `*.streaming.block.enabled` และ `*.streaming.block.coalesce` ในช่องทางที่รวมมาให้ คีย์แบบแบนที่ล้าสมัยจะถูกย้ายข้อมูลโดย `openclaw doctor --fix` สตรีมมิงแบบบล็อกจะปิดอยู่ เว้นแต่จะเปิดใช้งานอย่างชัดเจนในทุกช่องทางรวมถึง Telegram QQ Bot เป็นข้อยกเว้น โดยไม่มีคีย์ `streaming.block` และจะสตรีมคำตอบแบบบล็อก เว้นแต่ `channels.qqbot.streaming.mode` จะเป็น `"off"`

รายละเอียด: [สตรีมมิง + การแบ่งส่วน](/th/concepts/streaming)

## การมองเห็นเหตุผลและโทเค็น

- `/reasoning on|off|stream` ควบคุมการมองเห็น
- เนื้อหาการให้เหตุผลยังคงนับรวมในการใช้โทเค็นเมื่อโมเดลสร้างเนื้อหานั้น
- Telegram รองรับการสตรีมเหตุผลไปยังฟองฉบับร่างชั่วคราว ซึ่งจะถูกลบหลังการส่งขั้นสุดท้าย ใช้ `/reasoning on` สำหรับเอาต์พุตเหตุผลแบบถาวร

รายละเอียด: [ไดเรกทีฟการคิด + การให้เหตุผล](/th/tools/thinking) และ [การใช้โทเค็น](/th/reference/token-use)

## คำนำหน้า เธรด และการตอบกลับ

- ลำดับการใช้คำนำหน้าขาออก: `messages.responsePrefix`, `channels.<channel>.responsePrefix`, `channels.<channel>.accounts.<id>.responsePrefix` WhatsApp ยังมี `channels.whatsapp.messagePrefix` สำหรับคำนำหน้าขาเข้า
- เธรดการตอบกลับผ่าน `replyToMode` และค่าเริ่มต้นรายช่องทาง

## การตอบกลับแบบเงียบ

โทเค็นแบบเงียบ `NO_REPLY` (ไม่คำนึงถึงตัวพิมพ์เล็ก-ใหญ่ ดังนั้น `no_reply` จึงตรงกันด้วย) หมายถึง "ไม่ส่งคำตอบที่ผู้ใช้มองเห็น" เมื่อเทิร์นนั้นยังมีสื่อจากเครื่องมือที่รอส่ง เช่น เสียง TTS ที่สร้างขึ้น OpenClaw จะตัดข้อความแบบเงียบออกแต่ยังคงส่งไฟล์แนบสื่อ

นโยบายความเงียบจะกำหนดตามประเภทการสนทนา:

- การสนทนาโดยตรงจะไม่ได้รับคำแนะนำพรอมต์ `NO_REPLY` หากการทำงานโดยตรงส่งคืนเพียงโทเค็นแบบเงียบโดยไม่ตั้งใจ OpenClaw จะระงับโทเค็นนั้นแทนการเขียนใหม่หรือส่งออก
- กลุ่ม/ช่องทางอนุญาตให้เงียบโดยค่าเริ่มต้น ในโหมดการตอบกลับที่มองเห็นได้ `message_tool` ความเงียบหมายความว่าโมเดลไม่เรียก `message(action=send)`
- การประสานงานภายในอนุญาตให้เงียบโดยค่าเริ่มต้น

ค่าเริ่มต้นอยู่ภายใต้ `agents.defaults.silentReply` ส่วน `surfaces.<id>.silentReply` สามารถแทนค่านโยบายกลุ่ม/ภายในแยกตามพื้นผิวได้

OpenClaw ยังใช้การตอบกลับแบบเงียบสำหรับความล้มเหลวทั่วไปของตัวเรียกใช้ภายในในแชตที่ไม่ใช่แชตโดยตรง เพื่อไม่ให้กลุ่ม/ช่องทางเห็นข้อความข้อผิดพลาดสำเร็จรูปของ Gateway ความล้มเหลวที่ได้รับการจำแนกและมีข้อความกู้คืนสำหรับผู้ใช้ เช่น การแจ้งเตือนว่าขาดการยืนยันตัวตน ถึงขีดจำกัดอัตรา หรือระบบทำงานหนักเกินไป ยังคงส่งออกได้ แชตโดยตรงจะแสดงข้อความความล้มเหลวแบบกระชับโดยค่าเริ่มต้น ส่วนรายละเอียดดิบของตัวเรียกใช้จะแสดงเฉพาะเมื่อเปิดใช้งาน `/verbose full`

คำตอบที่มีเพียงโทเค็นแบบเงียบจะถูกละทิ้งในทุกพื้นผิว เพื่อให้เซสชันแม่ยังคงเงียบแทนที่จะเขียนข้อความเซนทิเนลใหม่เป็นข้อความสำรอง
