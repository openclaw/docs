OpenClaw สร้าง system prompt แบบกำหนดเองสำหรับการรัน agent ทุกครั้ง prompt นี้เป็น **ของ OpenClaw** และไม่ได้ใช้ prompt เริ่มต้นของ pi-coding-agent

prompt จะถูกประกอบโดย OpenClaw และฉีดเข้าไปในการรัน agent แต่ละครั้ง

Plugin ของผู้ให้บริการสามารถเพิ่มคำแนะนำ prompt ที่รับรู้แคชได้โดยไม่ต้องแทนที่
prompt เต็มรูปแบบที่ OpenClaw เป็นเจ้าของ runtime ของผู้ให้บริการสามารถ:

- แทนที่ชุดเล็กๆ ของส่วนแกนหลักที่มีชื่อ (`interaction_style`,
  `tool_call_style`, `execution_bias`)
- ฉีด **คำนำหน้าแบบเสถียร** ไว้เหนือขอบเขต prompt cache
- ฉีด **คำต่อท้ายแบบไดนามิก** ไว้ใต้ขอบเขต prompt cache

ใช้การมีส่วนร่วมที่ผู้ให้บริการเป็นเจ้าของสำหรับการปรับแต่งเฉพาะตระกูลโมเดล เก็บการกลายพันธุ์ prompt แบบเดิม
`before_prompt_build` ไว้สำหรับความเข้ากันได้หรือการเปลี่ยนแปลง prompt
แบบรวมทั้งระบบจริงๆ ไม่ใช่พฤติกรรมปกติของผู้ให้บริการ

โอเวอร์เลย์ตระกูล OpenAI GPT-5 ทำให้กฎการดำเนินการแกนหลักมีขนาดเล็ก และเพิ่ม
คำแนะนำเฉพาะโมเดลสำหรับการยึด persona, ผลลัพธ์ที่กระชับ, วินัยการใช้เครื่องมือ,
การค้นหาแบบขนาน, ความครอบคลุมของ deliverable, การตรวจสอบยืนยัน, บริบทที่ขาดหาย และ
สุขอนามัยของเครื่องมือ terminal

## โครงสร้าง

prompt ถูกทำให้กะทัดรัดโดยตั้งใจและใช้ส่วนที่คงที่:

- **เครื่องมือ**: ตัวเตือน source-of-truth ของ structured-tool พร้อมคำแนะนำการใช้เครื่องมือใน runtime
- **อคติการดำเนินการ**: คำแนะนำการทำต่อให้จบแบบกะทัดรัด: ลงมือภายในเทิร์นเมื่อ
  คำขอดำเนินการได้, ทำต่อจนเสร็จหรือถูกบล็อก, กู้คืนจากผลลัพธ์เครื่องมือที่อ่อน,
  ตรวจสอบสถานะที่เปลี่ยนแปลงได้แบบสด และตรวจสอบยืนยันก่อนสรุปขั้นสุดท้าย
- **ความปลอดภัย**: ตัวเตือน guardrail สั้นๆ เพื่อหลีกเลี่ยงพฤติกรรมแสวงหาอำนาจหรือการเลี่ยงการกำกับดูแล
- **Skills** (เมื่อพร้อมใช้งาน): บอกโมเดลว่าจะโหลดคำแนะนำ skill ตามต้องการได้อย่างไร
- **การอัปเดตตัวเองของ OpenClaw**: วิธีตรวจสอบ config อย่างปลอดภัยด้วย
  `config.schema.lookup`, patch config ด้วย `config.patch`, แทนที่ config ทั้งหมดด้วย
  `config.apply` และรัน `update.run` เฉพาะเมื่อผู้ใช้ร้องขออย่างชัดเจนเท่านั้น
  เครื่องมือ `gateway` สำหรับเจ้าของเท่านั้นยังปฏิเสธการเขียนทับ
  `tools.exec.ask` / `tools.exec.security` รวมถึง alias เดิม `tools.bash.*`
  ที่ normalize ไปยัง exec path ที่ได้รับการป้องกันเหล่านั้น
- **พื้นที่ทำงาน**: ไดเรกทอรีทำงาน (`agents.defaults.workspace`)
- **เอกสารประกอบ**: path ภายในเครื่องไปยังเอกสาร OpenClaw (repo หรือ npm package) และเวลาที่ควรอ่าน
- **ไฟล์พื้นที่ทำงาน (ฉีดเข้าไป)**: ระบุว่าไฟล์ bootstrap ถูกรวมไว้ด้านล่าง
- **แซนด์บ็อกซ์** (เมื่อเปิดใช้งาน): ระบุ runtime ที่อยู่ในแซนด์บ็อกซ์, path ของแซนด์บ็อกซ์ และ exec แบบยกระดับพร้อมใช้งานหรือไม่
- **วันที่และเวลาปัจจุบัน**: เวลาท้องถิ่นของผู้ใช้, เขตเวลา และรูปแบบเวลา
- **แท็กตอบกลับ**: ไวยากรณ์แท็กตอบกลับแบบเลือกได้สำหรับผู้ให้บริการที่รองรับ
- **Heartbeat**: prompt ของ Heartbeat และพฤติกรรม ack เมื่อเปิดใช้งาน Heartbeat สำหรับ agent เริ่มต้น
- **Runtime**: host, OS, node, model, repo root (เมื่อตรวจพบ), ระดับ thinking (หนึ่งบรรทัด)
- **การให้เหตุผล**: ระดับการมองเห็นปัจจุบัน + คำใบ้ toggle /reasoning

OpenClaw เก็บเนื้อหาเสถียรขนาดใหญ่ รวมถึง **บริบทโครงการ** ไว้เหนือ
ขอบเขต prompt cache ภายใน ส่วน channel/session ที่เปลี่ยนแปลงง่าย เช่น
คำแนะนำการฝัง Control UI, **การรับส่งข้อความ**, **เสียง**, **บริบทแชตกลุ่ม**,
**รีแอ็กชัน**, **Heartbeat** และ **Runtime** จะถูกต่อท้ายไว้ใต้ขอบเขตนั้น
เพื่อให้ backend ภายในเครื่องที่มี prefix cache สามารถใช้คำนำหน้าพื้นที่ทำงานที่เสถียรซ้ำ
ข้ามเทิร์นของ channel ได้ คำอธิบายเครื่องมือก็ควรหลีกเลี่ยงการฝังชื่อ
channel ปัจจุบันเช่นกัน เมื่อ schema ที่ยอมรับมีรายละเอียด runtime นั้นอยู่แล้ว

ส่วนเครื่องมือยังรวมคำแนะนำ runtime สำหรับงานที่ใช้เวลานาน:

- ใช้ cron สำหรับการติดตามผลในอนาคต (`check back later`, ตัวเตือน, งานที่เกิดซ้ำ)
  แทน loop sleep ของ `exec`, เทคนิคหน่วงเวลา `yieldMs` หรือการ polling `process`
  ซ้ำๆ
- ใช้ `exec` / `process` เฉพาะกับคำสั่งที่เริ่มตอนนี้และทำงานต่อ
  ในพื้นหลัง
- เมื่อเปิดใช้งาน completion wake อัตโนมัติ ให้เริ่มคำสั่งเพียงครั้งเดียวและพึ่งพา
  เส้นทาง wake แบบ push-based เมื่อมี output หรือ fail
- ใช้ `process` สำหรับ log, status, input หรือการแทรกแซงเมื่อคุณต้อง
  ตรวจสอบคำสั่งที่กำลังทำงานอยู่
- หากงานมีขนาดใหญ่กว่า ให้เลือกใช้ `sessions_spawn`; การเสร็จสิ้นของ sub-agent เป็นแบบ
  push-based และประกาศกลับไปยังผู้ร้องขอโดยอัตโนมัติ
- อย่า poll `subagents list` / `sessions_list` ใน loop เพียงเพื่อรอ
  ให้เสร็จสิ้น

เมื่อเปิดใช้งานเครื่องมือทดลอง `update_plan` ส่วนเครื่องมือยังบอก
โมเดลให้ใช้เฉพาะกับงานหลายขั้นตอนที่ไม่ใช่งานง่าย, คงให้มีขั้นตอน
`in_progress` เพียงหนึ่งขั้นตอน และหลีกเลี่ยงการทำซ้ำแผนทั้งหมดหลังการอัปเดตแต่ละครั้ง

guardrail ด้านความปลอดภัยใน system prompt เป็นคำแนะนำเชิงที่ปรึกษา สิ่งเหล่านี้ชี้นำพฤติกรรมของโมเดลแต่ไม่ได้บังคับใช้นโยบาย ใช้นโยบายเครื่องมือ, การอนุมัติ exec, sandboxing และ allowlist ของ channel สำหรับการบังคับใช้แบบเข้มงวด; operator สามารถปิดสิ่งเหล่านี้ได้ตามการออกแบบ

บน channel ที่มีการ์ด/ปุ่มอนุมัติแบบ native ตอนนี้ runtime prompt จะบอก
agent ให้พึ่งพา UI อนุมัติแบบ native นั้นก่อน ควรรวมคำสั่ง
`/approve` แบบ manual เฉพาะเมื่อผลลัพธ์เครื่องมือระบุว่า chat approval ไม่พร้อมใช้งาน หรือ
การอนุมัติแบบ manual เป็นเส้นทางเดียวเท่านั้น

## โหมด prompt

OpenClaw สามารถเรนเดอร์ system prompt ที่เล็กลงสำหรับ sub-agent ได้ runtime ตั้งค่า
`promptMode` สำหรับแต่ละการรัน (ไม่ใช่ config ที่แสดงต่อผู้ใช้):

- `full` (ค่าเริ่มต้น): รวมทุกส่วนข้างต้น
- `minimal`: ใช้สำหรับ sub-agent; ละเว้น **Skills**, **Memory Recall**, **การอัปเดตตัวเองของ OpenClaw**,
  **Model Aliases**, **User Identity**, **แท็กตอบกลับ**,
  **การรับส่งข้อความ**, **การตอบกลับแบบเงียบ** และ **Heartbeat** ส่วนเครื่องมือ, **ความปลอดภัย**,
  พื้นที่ทำงาน, แซนด์บ็อกซ์, วันที่และเวลาปัจจุบัน (เมื่อทราบ), Runtime และบริบทที่ฉีดเข้าไป
  ยังคงพร้อมใช้งาน
- `none`: ส่งคืนเฉพาะบรรทัดตัวตนพื้นฐาน

เมื่อ `promptMode=minimal` prompt เพิ่มเติมที่ฉีดเข้าไปจะถูกติดป้ายเป็น **บริบท Subagent**
แทน **บริบทแชตกลุ่ม**

สำหรับการรันตอบกลับอัตโนมัติของ channel OpenClaw สามารถละเว้นส่วน **การตอบกลับแบบเงียบ**
ทั่วไปเมื่อบริบทแชตตรง/กลุ่มมีพฤติกรรม `NO_REPLY` เฉพาะบทสนทนาที่ resolve แล้ว
อยู่แล้ว สิ่งนี้หลีกเลี่ยงการทำซ้ำกลไก token
ทั้งใน global system prompt และบริบท channel

## การฉีด bootstrap ของพื้นที่ทำงาน

ไฟล์ Bootstrap จะถูกตัดให้สั้นและต่อท้ายไว้ภายใต้ **บริบทโครงการ** เพื่อให้โมเดลเห็นบริบทตัวตนและ profile โดยไม่ต้องอ่านอย่างชัดเจน:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md` (เฉพาะในพื้นที่ทำงานที่สร้างใหม่)
- `MEMORY.md` เมื่อมีอยู่

ไฟล์ทั้งหมดเหล่านี้จะถูก **ฉีดเข้าไปใน context window** ในทุกเทิร์น เว้นแต่
มี gate เฉพาะไฟล์ใช้อยู่ `HEARTBEAT.md` จะถูกละเว้นในการรันปกติเมื่อ
ปิดใช้งาน Heartbeat สำหรับ agent เริ่มต้น หรือ
`agents.defaults.heartbeat.includeSystemPromptSection` เป็น false ทำให้ไฟล์ที่ฉีดเข้าไป
กระชับ โดยเฉพาะ `MEMORY.md` ซึ่งสามารถเติบโตเมื่อเวลาผ่านไปและนำไปสู่
การใช้บริบทสูงโดยไม่คาดคิดและการ Compaction ที่ถี่ขึ้น

<Note>
ไฟล์รายวัน `memory/*.md` **ไม่ใช่** ส่วนหนึ่งของ Project Context bootstrap ปกติ ในเทิร์นทั่วไป ไฟล์เหล่านี้จะถูกเข้าถึงตามต้องการผ่านเครื่องมือ `memory_search` และ `memory_get` ดังนั้นจึงไม่ถูกนับรวมกับ context window เว้นแต่โมเดลจะอ่านอย่างชัดเจน เทิร์น `/new` และ `/reset` แบบเปล่าเป็นข้อยกเว้น: runtime สามารถ prepend memory รายวันที่เพิ่งเกิดขึ้นเป็นบล็อก startup-context แบบใช้ครั้งเดียวสำหรับเทิร์นแรกนั้นได้
</Note>

ไฟล์ขนาดใหญ่จะถูกตัดทอนพร้อม marker ขนาดสูงสุดต่อไฟล์ควบคุมโดย
`agents.defaults.bootstrapMaxChars` (ค่าเริ่มต้น: 12000) เนื้อหา bootstrap ทั้งหมดที่ฉีด
รวมทุกไฟล์ถูกจำกัดโดย `agents.defaults.bootstrapTotalMaxChars`
(ค่าเริ่มต้น: 60000) ไฟล์ที่หายไปจะฉีด marker สั้นๆ ว่าไฟล์หาย เมื่อเกิดการตัดทอน
OpenClaw สามารถฉีดบล็อกคำเตือนใน Project Context ได้; ควบคุมสิ่งนี้ด้วย
`agents.defaults.bootstrapPromptTruncationWarning` (`off`, `once`, `always`;
ค่าเริ่มต้น: `once`)

เซสชัน Sub-agent จะฉีดเฉพาะ `AGENTS.md` และ `TOOLS.md` (ไฟล์ bootstrap อื่นๆ
จะถูกกรองออกเพื่อให้บริบท sub-agent มีขนาดเล็ก)

hook ภายในสามารถดักขั้นตอนนี้ผ่าน `agent:bootstrap` เพื่อกลายพันธุ์หรือแทนที่
ไฟล์ bootstrap ที่ฉีดเข้าไป (เช่น สลับ `SOUL.md` เป็น persona ทางเลือก)

หากคุณต้องการทำให้ agent ฟังดูเป็นแบบทั่วไปน้อยลง ให้เริ่มจาก
[คู่มือบุคลิกภาพ SOUL.md](/th/concepts/soul)

หากต้องการตรวจสอบว่าไฟล์ที่ฉีดเข้าไปแต่ละไฟล์มีส่วนร่วมมากเท่าใด (raw เทียบกับ injected, การตัดทอน รวมถึง overhead ของ tool schema) ให้ใช้ `/context list` หรือ `/context detail` ดู [บริบท](/th/concepts/context)

## การจัดการเวลา

system prompt จะรวมส่วน **วันที่และเวลาปัจจุบัน** โดยเฉพาะเมื่อทราบ
เขตเวลาของผู้ใช้ เพื่อให้ prompt cache เสถียร ตอนนี้จึงรวมเฉพาะ
**เขตเวลา** (ไม่มีนาฬิกาไดนามิกหรือรูปแบบเวลา)

ใช้ `session_status` เมื่อ agent ต้องการเวลาปัจจุบัน; การ์ดสถานะ
มีบรรทัด timestamp เครื่องมือเดียวกันยังสามารถตั้งค่า model override ต่อเซสชันแบบเลือกได้
(`model=default` จะล้างค่า)

กำหนดค่าด้วย:

- `agents.defaults.userTimezone`
- `agents.defaults.timeFormat` (`auto` | `12` | `24`)

ดู [วันที่และเวลา](/th/date-time) สำหรับรายละเอียดพฤติกรรมทั้งหมด

## Skills

เมื่อมี skills ที่มีสิทธิ์ OpenClaw จะฉีด **รายการ skills ที่พร้อมใช้งาน** แบบกะทัดรัด
(`formatSkillsForPrompt`) ที่รวม **path ไฟล์** สำหรับแต่ละ skill
prompt จะสั่งให้โมเดลใช้ `read` เพื่อโหลด SKILL.md ที่ตำแหน่งที่ระบุ
(พื้นที่ทำงาน, managed หรือ bundled) หากไม่มี skills ที่มีสิทธิ์
ส่วน Skills จะถูกละเว้น

สิทธิ์รวมถึง gate metadata ของ skill, การตรวจสอบ environment/config ของ runtime
และ allowlist skill ของ agent ที่มีผล เมื่อกำหนดค่า `agents.defaults.skills` หรือ
`agents.list[].skills`

Skills ที่ bundle มากับ Plugin จะมีสิทธิ์เฉพาะเมื่อ Plugin เจ้าของถูกเปิดใช้งาน
สิ่งนี้ช่วยให้ tool Plugin เปิดเผยคู่มือการปฏิบัติงานที่ลึกขึ้นได้โดยไม่ต้องฝัง
คำแนะนำทั้งหมดนั้นโดยตรงในคำอธิบายเครื่องมือทุกตัว

```
<available_skills>
  <skill>
    <name>...</name>
    <description>...</description>
    <location>...</location>
  </skill>
</available_skills>
```

สิ่งนี้ทำให้ prompt พื้นฐานมีขนาดเล็ก ขณะยังเปิดใช้งานการใช้ skill แบบเจาะจงได้

งบประมาณรายการ skills เป็นของ subsystem skills:

- ค่าเริ่มต้นระดับ global: `skills.limits.maxSkillsPromptChars`
- override ต่อ agent: `agents.list[].skillsLimits.maxSkillsPromptChars`

excerpt runtime แบบจำกัดทั่วไปใช้ surface อื่น:

- `agents.defaults.contextLimits.*`
- `agents.list[].contextLimits.*`

การแยกนี้ทำให้การกำหนดขนาด skills แยกจากการกำหนดขนาดการอ่าน/ฉีดของ runtime เช่น
`memory_get`, ผลลัพธ์เครื่องมือแบบสด และการ refresh AGENTS.md หลัง Compaction

## เอกสารประกอบ

system prompt รวมส่วน **เอกสารประกอบ** เมื่อมีเอกสารภายในเครื่องพร้อมใช้งาน ส่วนนี้จะ
ชี้ไปยังไดเรกทอรีเอกสาร OpenClaw ภายในเครื่อง (`docs/` ใน Git checkout หรือเอกสาร npm
package ที่ bundled มา) หากเอกสารภายในเครื่องไม่พร้อมใช้งาน จะ fallback ไปที่
[https://docs.openclaw.ai](https://docs.openclaw.ai)

ส่วนเดียวกันยังรวมตำแหน่งซอร์สของ OpenClaw ด้วย Git checkout จะเปิดเผย source root ภายในเครื่อง
เพื่อให้ agent ตรวจสอบโค้ดโดยตรงได้ การติดตั้งแบบ package จะรวม URL ซอร์ส GitHub
และบอกให้ agent ตรวจสอบซอร์สที่นั่นเมื่อเอกสารไม่สมบูรณ์หรือ
ล้าสมัย prompt ยังระบุ mirror เอกสารสาธารณะ, Discord ชุมชน และ ClawHub
([https://clawhub.ai](https://clawhub.ai)) สำหรับการค้นพบ skills และบอกโมเดลให้
ปรึกษาเอกสารก่อนสำหรับพฤติกรรม, คำสั่ง, การกำหนดค่า หรือสถาปัตยกรรมของ OpenClaw และให้
รัน `openclaw status` เองเมื่อทำได้ (ถามผู้ใช้เฉพาะเมื่อไม่มีสิทธิ์เข้าถึง)
สำหรับการกำหนดค่าโดยเฉพาะ จะชี้ agent ไปยัง action เครื่องมือ `gateway`
`config.schema.lookup` เพื่อดูเอกสารและข้อจำกัดระดับ field ที่แม่นยำ จากนั้นไปที่
`docs/gateway/configuration.md` และ `docs/gateway/configuration-reference.md`
สำหรับคำแนะนำที่กว้างขึ้น
