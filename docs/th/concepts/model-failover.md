OpenClaw จัดการความล้มเหลวเป็นสองขั้นตอน:

1. **การหมุนเวียนโปรไฟล์การยืนยันตัวตน** ภายในผู้ให้บริการปัจจุบัน
2. **การสำรองโมเดล** ไปยังโมเดลถัดไปใน `agents.defaults.model.fallbacks`

เอกสารนี้อธิบายกฎของรันไทม์และข้อมูลที่รองรับกฎเหล่านั้น

## โฟลว์รันไทม์

สำหรับการรันข้อความปกติ OpenClaw จะประเมินตัวเลือกตามลำดับนี้:

<Steps>
  <Step title="แก้ไขสถานะเซสชัน">
    แก้ไขโมเดลเซสชันที่ใช้งานอยู่และการตั้งค่าโปรไฟล์การยืนยันตัวตน
  </Step>
  <Step title="สร้างสายโซ่ตัวเลือก">
    สร้างสายโซ่ตัวเลือกโมเดลจากการเลือกโมเดลปัจจุบันและนโยบายสำรองสำหรับแหล่งที่มาของการเลือกนั้น ค่าเริ่มต้นที่กำหนดไว้ โมเดลหลักของงาน cron และโมเดลสำรองที่เลือกอัตโนมัติสามารถใช้รายการสำรองที่กำหนดไว้ได้ ส่วนการเลือกเซสชันโดยผู้ใช้อย่างชัดเจนจะเข้มงวด
  </Step>
  <Step title="ลองผู้ให้บริการปัจจุบัน">
    ลองผู้ให้บริการปัจจุบันด้วยกฎการหมุนเวียน/คูลดาวน์โปรไฟล์การยืนยันตัวตน
  </Step>
  <Step title="ขยับต่อเมื่อเกิดข้อผิดพลาดที่สมควร failover">
    หากผู้ให้บริการนั้นถูกใช้จนหมดพร้อมข้อผิดพลาดที่สมควร failover ให้ย้ายไปยังตัวเลือกโมเดลถัดไป
  </Step>
  <Step title="คง override สำรองไว้">
    คง override สำรองที่เลือกไว้ก่อนเริ่มลองใหม่ เพื่อให้ตัวอ่านเซสชันอื่นเห็นผู้ให้บริการ/โมเดลเดียวกับที่ runner กำลังจะใช้ override โมเดลที่คงไว้จะถูกทำเครื่องหมาย `modelOverrideSource: "auto"`
  </Step>
  <Step title="ย้อนกลับอย่างจำกัดเมื่อล้มเหลว">
    หากตัวเลือกสำรองล้มเหลว ให้ย้อนกลับเฉพาะฟิลด์ override ของเซสชันที่ fallback เป็นเจ้าของ เมื่อฟิลด์เหล่านั้นยังตรงกับตัวเลือกที่ล้มเหลวนั้นอยู่
  </Step>
  <Step title="โยน FallbackSummaryError หากใช้หมดแล้ว">
    หากตัวเลือกทั้งหมดล้มเหลว ให้โยน `FallbackSummaryError` พร้อมรายละเอียดรายความพยายามและเวลาหมดคูลดาวน์ที่เร็วที่สุดเมื่อทราบ
  </Step>
</Steps>

สิ่งนี้ตั้งใจให้แคบกว่า “บันทึกและกู้คืนทั้งเซสชัน” reply runner จะคงไว้เฉพาะฟิลด์การเลือกโมเดลที่ตนเป็นเจ้าของสำหรับ fallback:

- `providerOverride`
- `modelOverride`
- `modelOverrideSource`
- `authProfileOverride`
- `authProfileOverrideSource`
- `authProfileOverrideCompactionCount`

สิ่งนี้ป้องกันไม่ให้การลอง fallback ซ้ำที่ล้มเหลวเขียนทับการเปลี่ยนแปลงเซสชันอื่นที่ใหม่กว่าและไม่เกี่ยวข้อง เช่น การเปลี่ยน `/model` ด้วยตนเอง หรือการอัปเดตการหมุนเวียนเซสชันที่เกิดขึ้นระหว่างที่ความพยายามกำลังทำงานอยู่

## นโยบายแหล่งที่มาของการเลือก

OpenClaw แยกผู้ให้บริการ/โมเดลที่เลือกออกจากเหตุผลที่เลือก แหล่งที่มานั้นควบคุมว่าสายโซ่ fallback ได้รับอนุญาตหรือไม่:

- **ค่าเริ่มต้นที่กำหนดไว้**: `agents.defaults.model.primary` ใช้ `agents.defaults.model.fallbacks`
- **โมเดลหลักของเอเจนต์**: `agents.list[].model` จะเข้มงวด เว้นแต่วัตถุโมเดลของเอเจนต์นั้นจะมี `fallbacks` ของตัวเอง ใช้ `fallbacks: []` เพื่อระบุพฤติกรรมเข้มงวดอย่างชัดเจน หรือให้รายการที่ไม่ว่างเพื่อเลือกให้เอเจนต์นั้นใช้ model fallback
- **Override fallback อัตโนมัติ**: runtime fallback จะเขียน `providerOverride`, `modelOverride`, `modelOverrideSource: "auto"` และโมเดลต้นทางที่เลือกก่อนลองใหม่ override อัตโนมัตินั้นสามารถเดินต่อไปตามสายโซ่ fallback ที่กำหนดไว้ได้โดยไม่ต้อง probe โมเดลหลักในทุกข้อความ แต่ OpenClaw จะ probe ต้นทางที่กำหนดไว้อีกครั้งเป็นระยะ และล้าง override อัตโนมัติเมื่อกู้คืนได้ `/new`, `/reset` และ `sessions.reset` จะล้าง override ที่มีแหล่งที่มาเป็น auto ด้วย Heartbeat ที่รันโดยไม่มี `heartbeat.model` ชัดเจนจะล้าง override อัตโนมัติโดยตรงเมื่อ origin ของมันไม่ตรงกับค่าเริ่มต้นที่กำหนดไว้ในปัจจุบันอีกต่อไป
- **Override เซสชันของผู้ใช้**: `/model`, ตัวเลือกโมเดล, `session_status(model=...)` และ `sessions.patch` จะเขียน `modelOverrideSource: "user"` นั่นคือการเลือกเซสชันแบบเจาะจง หากผู้ให้บริการ/โมเดลที่เลือกล้มเหลวก่อนสร้างคำตอบ OpenClaw จะรายงานความล้มเหลวแทนที่จะตอบจาก fallback ที่กำหนดไว้แต่ไม่เกี่ยวข้อง
- **Override เซสชันเดิม**: รายการเซสชันเก่าอาจมี `modelOverride` โดยไม่มี `modelOverrideSource` OpenClaw ถือว่าสิ่งเหล่านั้นเป็น override ของผู้ใช้ เพื่อไม่ให้การเลือกเก่าที่ชัดเจนถูกแปลงเป็นพฤติกรรม fallback อย่างเงียบ ๆ
- **โมเดลใน payload ของ Cron**: งาน cron `payload.model` / `--model` เป็นโมเดลหลักของงาน ไม่ใช่ override เซสชันของผู้ใช้ มันใช้ fallback ที่กำหนดไว้ เว้นแต่งานจะให้ `payload.fallbacks`; `payload.fallbacks: []` ทำให้การรัน cron เข้มงวด

ช่วงเวลา probe โมเดลหลักของ auto fallback คือห้านาทีและกำหนดค่าไม่ได้ OpenClaw จดจำ probe ล่าสุดต่อเซสชันและโมเดลหลัก เพื่อไม่ให้ลองโมเดลหลักที่ล้มเหลวซ้ำในทุก turn OpenClaw ส่งประกาศที่มองเห็นได้เมื่อเซสชันย้ายไป fallback และส่งประกาศอีกครั้งเมื่อกลับสู่โมเดลหลักที่เลือกไว้ โดยจะไม่ส่งประกาศซ้ำในทุก turn ของ sticky fallback

## แคชข้ามความล้มเหลวของการยืนยันตัวตน

โดยค่าเริ่มต้น turn ใหม่ทุกครั้งจะคงพฤติกรรมการลอง fallback ซ้ำเดิมไว้: OpenClaw
จะลองตัวเลือก fallback ที่กำหนดไว้แต่ละตัวอีกครั้ง รวมถึงตัวเลือกที่ไม่ใช่โมเดลหลัก
ซึ่งเพิ่งล้มเหลวด้วย `auth` หรือ `auth_permanent`

ผู้ปฏิบัติงานที่ต้องการระงับความล้มเหลวของการยืนยันตัวตนซ้ำเหล่านั้นสามารถเลือกเปิดใช้ด้วย:

```bash
OPENCLAW_FALLBACK_SKIP_TTL_MS=60000
```

เมื่อเปิดใช้ OpenClaw จะบันทึก marker สำหรับข้ามแบบ in-memory และจำกัดตามเซสชันสำหรับ
ตัวเลือก fallback ที่ไม่ใช่โมเดลหลักหลังจากความล้มเหลวระดับ auth marker จะถูก key
ด้วย session id, provider และ model ตัวเลือกโมเดลหลักจะไม่ถูกข้ามเลย ดังนั้น
การเลือกโมเดลผู้ใช้อย่างชัดเจนยังคงแสดงข้อผิดพลาด auth จริง แคชนี้เป็นแบบ
process-local และจะถูกล้างเมื่อ Gateway รีสตาร์ท

ค่านี้เป็น TTL หน่วยมิลลิวินาที `0` หรือค่าที่ไม่ได้ตั้งไว้จะปิดใช้แคช
ค่าบวกจะถูก clamp ให้อยู่ระหว่าง 1 วินาทีถึง 10 นาที

## ประกาศ fallback ที่ผู้ใช้มองเห็น

เมื่อเซสชันย้ายไปยัง fallback ที่เลือกอัตโนมัติ OpenClaw จะส่งประกาศสถานะในพื้นที่ตอบกลับเดียวกัน:

```text
↪️ Model Fallback: <fallback> (selected <primary>; <reason>)
```

เมื่อ probe ภายหลังสำเร็จและเซสชันกลับสู่โมเดลหลักที่เลือก OpenClaw จะส่ง:

```text
↪️ Model Fallback cleared: <primary> (was <fallback>)
```

ประกาศเหล่านี้เป็นข้อความปฏิบัติการ ไม่ใช่เนื้อหาของ assistant ข้อความเหล่านี้จะถูกส่งหนึ่งครั้งต่อการเปลี่ยนสถานะ รวมถึง turn ที่มีเฉพาะผลข้างเคียงเมื่อทำได้ แต่ turn ของ sticky fallback จะไม่ส่งซ้ำ การส่งจะข้ามการระงับ source-reply ปกติ ประกาศจะไม่ใช้ช่องตอบกลับ assistant แรกสำหรับช่องทางแบบ thread และถูกแยกออกจาก text-to-speech และการดึง commitment

## พื้นที่จัดเก็บการยืนยันตัวตน (คีย์ + OAuth)

OpenClaw ใช้ **โปรไฟล์การยืนยันตัวตน** สำหรับทั้ง API key และ OAuth token

- ความลับและสถานะ runtime auth-routing อยู่ใน `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- Config `auth.profiles` / `auth.order` เป็น **metadata + routing เท่านั้น** (ไม่มีความลับ)
- ไฟล์ OAuth เดิมสำหรับนำเข้าเท่านั้น: `~/.openclaw/credentials/oauth.json` (นำเข้าไปยัง auth store ต่อเอเจนต์เมื่อใช้ครั้งแรก)
- ไฟล์เดิม `auth-profiles.json`, `auth-state.json` และไฟล์ `auth.json` ต่อเอเจนต์จะถูกนำเข้าโดย `openclaw doctor --fix`

รายละเอียดเพิ่มเติม: [OAuth](/th/concepts/oauth)

ชนิดข้อมูลรับรอง:

- `type: "api_key"` → `{ provider, key }`
- `type: "oauth"` → `{ provider, access, refresh, expires, email? }` (+ `projectId`/`enterpriseUrl` สำหรับผู้ให้บริการบางราย)

## ID โปรไฟล์

การเข้าสู่ระบบ OAuth จะสร้างโปรไฟล์แยกกันเพื่อให้หลายบัญชีอยู่ร่วมกันได้

- ค่าเริ่มต้น: `provider:default` เมื่อไม่มีอีเมล
- OAuth ที่มีอีเมล: `provider:<email>` (เช่น `google-antigravity:user@gmail.com`)

โปรไฟล์อยู่ใน auth profile store ของ `openclaw-agent.sqlite` ต่อเอเจนต์

## ลำดับการหมุนเวียน

เมื่อผู้ให้บริการมีหลายโปรไฟล์ OpenClaw จะเลือกลำดับดังนี้:

<Steps>
  <Step title="Config ที่ระบุชัดเจน">
    `auth.order[provider]` (ถ้าตั้งไว้)
  </Step>
  <Step title="โปรไฟล์ที่กำหนดไว้">
    `auth.profiles` ที่กรองตามผู้ให้บริการ
  </Step>
  <Step title="โปรไฟล์ที่จัดเก็บไว้">
    รายการ auth profile ใน SQLite ต่อเอเจนต์สำหรับผู้ให้บริการ
  </Step>
</Steps>

หากไม่ได้กำหนดลำดับอย่างชัดเจน OpenClaw จะใช้ลำดับแบบ round-robin:

- **คีย์หลัก:** ชนิดโปรไฟล์ (**OAuth ก่อน API key**)
- **คีย์รอง:** `usageStats.lastUsed` (เก่าที่สุดก่อน ภายในแต่ละชนิด)
- **โปรไฟล์ที่อยู่ในคูลดาวน์/ถูกปิดใช้** จะถูกย้ายไปท้ายสุด โดยเรียงตามเวลาหมดอายุที่เร็วที่สุด

### ความเหนียวแน่นของเซสชัน (เป็นมิตรกับแคช)

OpenClaw **ตรึงโปรไฟล์การยืนยันตัวตนที่เลือกไว้ต่อเซสชัน** เพื่อให้แคชของผู้ให้บริการยังอุ่นอยู่ มัน **ไม่** หมุนเวียนทุกคำขอ โปรไฟล์ที่ตรึงไว้จะถูกใช้ซ้ำจนกว่า:

- เซสชันถูกรีเซ็ต (`/new` / `/reset`)
- compaction เสร็จสมบูรณ์ (จำนวน compaction เพิ่มขึ้น)
- โปรไฟล์อยู่ในคูลดาวน์/ถูกปิดใช้

การเลือกด้วยตนเองผ่าน `/model …@<profileId>` ตั้งค่า **override ของผู้ใช้** สำหรับเซสชันนั้น และจะไม่ถูกหมุนเวียนอัตโนมัติจนกว่าจะเริ่มเซสชันใหม่

<Note>
โปรไฟล์ที่ตรึงอัตโนมัติ (เลือกโดย session router) จะถูกถือเป็น **ความต้องการ**: จะถูกลองก่อน แต่ OpenClaw อาจหมุนเวียนไปยังโปรไฟล์อื่นเมื่อเกิด rate limit/timeout เมื่อโปรไฟล์เดิมพร้อมใช้งานอีกครั้ง การรันใหม่สามารถเลือกใช้โปรไฟล์นั้นอีกครั้งได้โดยไม่เปลี่ยนโมเดลหรือรันไทม์ที่เลือกไว้ โปรไฟล์ที่ผู้ใช้ตรึงไว้จะล็อกอยู่กับโปรไฟล์นั้น หากมันล้มเหลวและมีการกำหนด model fallback ไว้ OpenClaw จะย้ายไปโมเดลถัดไปแทนการสลับโปรไฟล์
</Note>

### การสมัครสมาชิก OpenAI Codex พร้อม API-key สำรอง

สำหรับโมเดลเอเจนต์ OpenAI การยืนยันตัวตนและรันไทม์แยกจากกัน `openai/gpt-*` ยังคงอยู่บน
Codex harness ขณะที่การยืนยันตัวตนสามารถหมุนเวียนระหว่างโปรไฟล์การสมัครสมาชิก Codex และ
OpenAI API-key สำรองได้

ใช้ `auth.order.openai` สำหรับลำดับที่ผู้ใช้เห็น:

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

ใช้ `openai:*` สำหรับทั้งโปรไฟล์ OAuth ของ ChatGPT/Codex และโปรไฟล์ OpenAI API-key
เมื่อการสมัครสมาชิกชนขีดจำกัดการใช้งาน Codex
OpenClaw จะบันทึกเวลารีเซ็ตที่แน่นอนเมื่อ Codex ให้มา ลองโปรไฟล์การยืนยันตัวตนถัดไป
ตามลำดับ และคงการรันไว้ภายใน Codex harness เมื่อเวลารีเซ็ตผ่านไป
โปรไฟล์การสมัครสมาชิกจะมีสิทธิ์อีกครั้ง และการเลือกอัตโนมัติครั้งถัดไป
สามารถกลับไปใช้โปรไฟล์นั้นได้

ใช้โปรไฟล์ที่ผู้ใช้ตรึงไว้เฉพาะเมื่อคุณต้องการบังคับใช้บัญชี/คีย์หนึ่งสำหรับ
เซสชันนั้น โปรไฟล์ที่ผู้ใช้ตรึงไว้ตั้งใจให้เข้มงวดและจะไม่ข้ามไป

## คูลดาวน์

เมื่อโปรไฟล์ล้มเหลวจากข้อผิดพลาด auth/rate-limit (หรือ timeout ที่ดูเหมือน rate limiting) OpenClaw จะทำเครื่องหมายว่าอยู่ในคูลดาวน์และย้ายไปยังโปรไฟล์ถัดไป

<AccordionGroup>
  <Accordion title="สิ่งที่เข้าสู่ bucket rate-limit / timeout">
    bucket rate-limit นั้นกว้างกว่า `429` ธรรมดา: ยังรวมข้อความจากผู้ให้บริการ เช่น `Too many concurrent requests`, `ThrottlingException`, `concurrency limit reached`, `workers_ai ... quota limit exceeded`, `throttled`, `resource exhausted` และขีดจำกัดหน้าต่างการใช้งานเป็นระยะ เช่น `weekly/monthly limit reached`

    ข้อผิดพลาด format/invalid-request มักเป็น terminal เพราะการลอง payload เดิมซ้ำจะล้มเหลวแบบเดิม ดังนั้น OpenClaw จะแสดงข้อผิดพลาดเหล่านั้นแทนการหมุนเวียนโปรไฟล์การยืนยันตัวตน เส้นทาง retry-repair ที่รู้จักสามารถเลือกเปิดใช้อย่างชัดเจนได้: เช่น ความล้มเหลวในการตรวจสอบ ID ของ Cloud Code Assist tool call จะถูกทำให้ปลอดภัยและลองซ้ำหนึ่งครั้งผ่านนโยบาย `allowFormatRetry` ข้อผิดพลาด stop-reason ที่เข้ากันได้กับ OpenAI เช่น `Unhandled stop reason: error`, `stop reason: error` และ `reason: error` จะถูกจัดประเภทเป็นสัญญาณ timeout/failover

    ข้อความเซิร์ฟเวอร์ทั่วไปอาจเข้าสู่ bucket timeout นั้นได้เช่นกันเมื่อแหล่งที่มาตรงกับรูปแบบ transient ที่รู้จัก ตัวอย่างเช่น ข้อความเปล่า ๆ จาก stream-wrapper ของรันไทม์โมเดล `An unknown error occurred` จะถูกถือว่าสมควร failover สำหรับผู้ให้บริการทุกเจ้า เพราะรันไทม์โมเดลร่วมปล่อยข้อความนี้เมื่อ stream ของผู้ให้บริการจบด้วย `stopReason: "aborted"` หรือ `stopReason: "error"` โดยไม่มีรายละเอียดเฉพาะ payload JSON `api_error` ที่มีข้อความเซิร์ฟเวอร์ transient เช่น `internal server error`, `unknown error, 520`, `upstream error` หรือ `backend error` จะถูกถือว่าเป็น timeout ที่สมควร failover เช่นกัน

    ข้อความ upstream ทั่วไปเฉพาะ OpenRouter เช่น `Provider returned error` แบบเปล่า ๆ จะถูกถือว่าเป็น timeout เฉพาะเมื่อบริบทผู้ให้บริการเป็น OpenRouter จริง ๆ ข้อความ fallback ภายในทั่วไป เช่น `LLM request failed with an unknown error.` จะยังคงใช้แนวทางระมัดระวังและไม่กระตุ้น failover ด้วยตัวเอง

  </Accordion>
  <Accordion title="ขีดจำกัด retry-after ของ SDK">
    SDK ของผู้ให้บริการบางรายอาจพักเป็นเวลานานตามช่วง `Retry-After` ก่อนส่งการควบคุมกลับให้ OpenClaw สำหรับ SDK ที่ใช้ Stainless เช่น Anthropic และ OpenAI โดยค่าเริ่มต้น OpenClaw จะจำกัดเวลารอ `retry-after-ms` / `retry-after` ภายใน SDK ไว้ที่ 60 วินาที และแสดงผลการตอบกลับที่ลองใหม่ได้ซึ่งนานกว่านั้นทันที เพื่อให้เส้นทางเฟลโอเวอร์นี้ทำงานได้ ปรับแต่งหรือปิดขีดจำกัดนี้ด้วย `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS`; ดู [พฤติกรรมการลองใหม่](/th/concepts/retry)
  </Accordion>
  <Accordion title="คูลดาวน์ตามขอบเขตโมเดล">
    คูลดาวน์จากการจำกัดอัตราสามารถจำกัดขอบเขตตามโมเดลได้เช่นกัน:

    - OpenClaw บันทึก `cooldownModel` สำหรับความล้มเหลวจากการจำกัดอัตราเมื่อทราบ id ของโมเดลที่ล้มเหลว
    - ยังสามารถลองใช้โมเดลพี่น้องในผู้ให้บริการเดียวกันได้เมื่อคูลดาวน์ถูกจำกัดขอบเขตไว้ที่โมเดลอื่น
    - ช่วงเวลาที่เกี่ยวกับการเรียกเก็บเงิน/การปิดใช้งานยังคงบล็อกทั้งโปรไฟล์ข้ามโมเดลทั้งหมด

  </Accordion>
</AccordionGroup>

คูลดาวน์ใช้ exponential backoff:

- 1 นาที
- 5 นาที
- 25 นาที
- 1 ชั่วโมง (ขีดจำกัด)

สถานะถูกเก็บไว้ในสถานะการยืนยันตัวตน SQLite ต่อเอเจนต์ภายใต้ `usageStats`:

```json
{
  "usageStats": {
    "provider:profile": {
      "lastUsed": 1736160000000,
      "cooldownUntil": 1736160600000,
      "errorCount": 2
    }
  }
}
```

## การปิดใช้งานจากการเรียกเก็บเงิน

ความล้มเหลวด้านการเรียกเก็บเงิน/เครดิต (เช่น "insufficient credits" / "credit balance too low") ถือว่าเหมาะสำหรับเฟลโอเวอร์ แต่โดยปกติไม่ใช่ข้อผิดพลาดชั่วคราว แทนที่จะใช้คูลดาวน์สั้น ๆ OpenClaw จะทำเครื่องหมายโปรไฟล์เป็น **ปิดใช้งาน** (พร้อม backoff ที่นานขึ้น) และหมุนไปยังโปรไฟล์/ผู้ให้บริการถัดไป

<Note>
ไม่ใช่ทุกการตอบกลับที่มีลักษณะเกี่ยวกับการเรียกเก็บเงินจะเป็น `402` และไม่ใช่ทุก HTTP `402` จะมาถึงเส้นทางนี้ OpenClaw เก็บข้อความการเรียกเก็บเงินที่ชัดเจนไว้ในช่องทางการเรียกเก็บเงิน แม้ผู้ให้บริการจะส่งคืน `401` หรือ `403` แทน แต่ตัวจับคู่เฉพาะผู้ให้บริการจะยังคงจำกัดขอบเขตอยู่กับผู้ให้บริการที่เป็นเจ้าของเท่านั้น (เช่น OpenRouter `403 Key limit exceeded`)

ในขณะเดียวกัน ข้อผิดพลาด `402` ชั่วคราวที่เกี่ยวกับช่วงเวลาการใช้งานและขีดจำกัดค่าใช้จ่ายขององค์กร/เวิร์กสเปซจะถูกจัดประเภทเป็น `rate_limit` เมื่อข้อความดูเหมือนลองใหม่ได้ (เช่น `weekly usage limit exhausted`, `daily limit reached, resets tomorrow` หรือ `organization spending limit exceeded`) สิ่งเหล่านี้จะอยู่บนเส้นทางคูลดาวน์สั้น/เฟลโอเวอร์ แทนเส้นทางปิดใช้งานจากการเรียกเก็บเงินที่ยาวกว่า
</Note>

สถานะถูกเก็บไว้ในสถานะการยืนยันตัวตน SQLite ต่อเอเจนต์:

```json
{
  "usageStats": {
    "provider:profile": {
      "disabledUntil": 1736178000000,
      "disabledReason": "billing"
    }
  }
}
```

ค่าเริ่มต้น:

- backoff การเรียกเก็บเงินเริ่มที่ **5 ชั่วโมง** เพิ่มเป็นสองเท่าต่อความล้มเหลวด้านการเรียกเก็บเงินแต่ละครั้ง และจำกัดสูงสุดที่ **24 ชั่วโมง**
- ตัวนับ backoff จะรีเซ็ตหากโปรไฟล์ไม่ล้มเหลวเป็นเวลา **24 ชั่วโมง** (กำหนดค่าได้)
- การลองใหม่จากภาวะโอเวอร์โหลดอนุญาตให้ **หมุนโปรไฟล์ในผู้ให้บริการเดียวกัน 1 ครั้ง** ก่อน fallback โมเดล
- การลองใหม่จากภาวะโอเวอร์โหลดใช้ **backoff 0 ms** โดยค่าเริ่มต้น

## fallback ของโมเดล

หากโปรไฟล์ทั้งหมดของผู้ให้บริการล้มเหลว OpenClaw จะย้ายไปยังโมเดลถัดไปใน `agents.defaults.model.fallbacks` สิ่งนี้ใช้กับความล้มเหลวด้านการยืนยันตัวตน การจำกัดอัตรา และการหมดเวลาที่ใช้การหมุนโปรไฟล์จนหมดแล้ว (ข้อผิดพลาดอื่นจะไม่เลื่อน fallback) ข้อผิดพลาดของผู้ให้บริการที่ไม่เปิดเผยรายละเอียดเพียงพอยังคงถูกติดป้ายอย่างแม่นยำในสถานะ fallback: `empty_response` หมายถึงผู้ให้บริการไม่ได้ส่งคืนข้อความหรือสถานะที่ใช้งานได้, `no_error_details` หมายถึงผู้ให้บริการส่งคืน `Unknown error (no error details in response)` อย่างชัดเจน และ `unclassified` หมายถึง OpenClaw เก็บตัวอย่างดิบไว้แล้ว แต่ยังไม่มีตัวจัดประเภทใดตรงกัน

ข้อผิดพลาดจากภาวะโอเวอร์โหลดและการจำกัดอัตราถูกจัดการอย่างเชิงรุกกว่าคูลดาวน์จากการเรียกเก็บเงิน โดยค่าเริ่มต้น OpenClaw อนุญาตให้ลองโปรไฟล์การยืนยันตัวตนในผู้ให้บริการเดียวกันหนึ่งครั้ง จากนั้นสลับไปยัง fallback โมเดลถัดไปที่กำหนดค่าไว้โดยไม่ต้องรอ สัญญาณผู้ให้บริการไม่ว่าง เช่น `ModelNotReadyException` จะอยู่ในกลุ่มโอเวอร์โหลดนั้น ปรับแต่งสิ่งนี้ด้วย `auth.cooldowns.overloadedProfileRotations`, `auth.cooldowns.overloadedBackoffMs` และ `auth.cooldowns.rateLimitedProfileRotations`

เมื่อการรันเริ่มจาก primary เริ่มต้นที่กำหนดค่าไว้, primary ของงาน Cron, primary ของเอเจนต์ที่มี fallback ชัดเจน หรือ override fallback ที่เลือกอัตโนมัติ OpenClaw สามารถเดินตามสาย fallback ที่กำหนดค่าไว้ซึ่งตรงกันได้ primary ของเอเจนต์ที่ไม่มี fallback ชัดเจนและการเลือกของผู้ใช้อย่างชัดเจน (เช่น `/model ollama/qwen3.5:27b`, ตัวเลือกโมเดล, `sessions.patch` หรือ override ผู้ให้บริการ/โมเดลแบบครั้งเดียวของ CLI) จะเข้มงวด: หากผู้ให้บริการ/โมเดลนั้นเข้าถึงไม่ได้หรือล้มเหลวก่อนสร้างคำตอบ OpenClaw จะรายงานความล้มเหลวแทนการตอบจาก fallback ที่ไม่เกี่ยวข้อง

### กฎของสายตัวเลือก

OpenClaw สร้างรายการตัวเลือกจาก `provider/model` ที่ร้องขออยู่ปัจจุบันบวกกับ fallback ที่กำหนดค่าไว้

<AccordionGroup>
  <Accordion title="กฎ">
    - โมเดลที่ร้องขอจะอยู่เป็นอันดับแรกเสมอ
    - fallback ที่กำหนดค่าไว้อย่างชัดเจนจะถูกตัดรายการซ้ำ แต่ไม่ถูกกรองด้วยรายการโมเดลที่อนุญาต ถือว่าเป็นเจตนาชัดเจนของผู้ปฏิบัติการ
    - หากการรันปัจจุบันอยู่บน fallback ที่กำหนดค่าไว้แล้วในตระกูลผู้ให้บริการเดียวกัน OpenClaw จะยังใช้สายที่กำหนดค่าไว้ทั้งหมดต่อไป
    - เมื่อไม่ได้ระบุ override fallback อย่างชัดเจน จะลอง fallback ที่กำหนดค่าไว้ก่อน primary ที่กำหนดค่าไว้ แม้โมเดลที่ร้องขอจะใช้ผู้ให้บริการอื่นก็ตาม
    - เมื่อไม่ได้ระบุ override fallback อย่างชัดเจนให้ตัวรัน fallback จะเพิ่ม primary ที่กำหนดค่าไว้ต่อท้าย เพื่อให้สายสามารถกลับไปอยู่บนค่าเริ่มต้นปกติได้เมื่อใช้ตัวเลือกก่อนหน้าจนหมดแล้ว
    - เมื่อผู้เรียกส่ง `fallbacksOverride` ตัวรันจะใช้เฉพาะโมเดลที่ร้องขอบวกกับรายการ override นั้นเท่านั้น รายการว่างจะปิด fallback โมเดลและป้องกันไม่ให้เพิ่ม primary ที่กำหนดค่าไว้เป็นเป้าหมายลองใหม่แบบซ่อน

  </Accordion>
</AccordionGroup>

### พฤติกรรมการข้ามคูลดาวน์เทียบกับการ probe

เมื่อทุกโปรไฟล์การยืนยันตัวตนของผู้ให้บริการอยู่ในคูลดาวน์แล้ว OpenClaw จะไม่ข้ามผู้ให้บริการนั้นตลอดไปโดยอัตโนมัติ แต่จะตัดสินใจรายตัวเลือก:

<AccordionGroup>
  <Accordion title="การตัดสินใจรายตัวเลือก">
    - ความล้มเหลวด้านการยืนยันตัวตนแบบต่อเนื่องจะข้ามทั้งผู้ให้บริการทันที
    - การปิดใช้งานจากการเรียกเก็บเงินมักจะข้าม แต่ยังสามารถ probe ตัวเลือก primary ตาม throttle ได้ เพื่อให้กู้คืนได้โดยไม่ต้องรีสตาร์ท
    - อาจ probe ตัวเลือก primary ใกล้เวลาหมดอายุคูลดาวน์ โดยมี throttle ต่อผู้ให้บริการ
    - fallback พี่น้องในผู้ให้บริการเดียวกันสามารถถูกลองได้แม้อยู่ในคูลดาวน์เมื่อความล้มเหลวดูลักษณะชั่วคราว (`rate_limit`, `overloaded` หรือไม่ทราบชนิด) สิ่งนี้สำคัญเป็นพิเศษเมื่อการจำกัดอัตรามีขอบเขตตามโมเดล และโมเดลพี่น้องอาจกู้คืนได้ทันที
    - การ probe คูลดาวน์ชั่วคราวถูกจำกัดไว้ที่หนึ่งครั้งต่อผู้ให้บริการต่อการรัน fallback เพื่อไม่ให้ผู้ให้บริการเดียวถ่วง fallback ข้ามผู้ให้บริการ

  </Accordion>
</AccordionGroup>

## override เซสชันและการสลับโมเดลแบบสด

การเปลี่ยนโมเดลของเซสชันเป็นสถานะที่ใช้ร่วมกัน ตัวรันที่ทำงานอยู่ คำสั่ง `/model` การอัปเดต Compaction/เซสชัน และการปรับให้ตรงกันของเซสชันสด ล้วนอ่านหรือเขียนบางส่วนของรายการเซสชันเดียวกัน

นั่นหมายความว่าการลอง fallback ต้องประสานกับการสลับโมเดลแบบสด:

- เฉพาะการเปลี่ยนโมเดลที่ผู้ใช้ขับเคลื่อนอย่างชัดเจนเท่านั้นที่ทำเครื่องหมายการสลับแบบสดที่ค้างอยู่ ซึ่งรวมถึง `/model`, `session_status(model=...)` และ `sessions.patch`
- การเปลี่ยนโมเดลที่ระบบขับเคลื่อน เช่น การหมุน fallback, override ของ Heartbeat หรือ Compaction จะไม่ทำเครื่องหมายการสลับแบบสดที่ค้างอยู่ด้วยตัวเอง
- override โมเดลที่ผู้ใช้ขับเคลื่อนถือเป็นการเลือกที่แน่นอนสำหรับนโยบาย fallback ดังนั้นผู้ให้บริการที่เลือกไว้แต่เข้าถึงไม่ได้จะแสดงเป็นความล้มเหลวแทนที่จะถูกซ่อนด้วย `agents.defaults.model.fallbacks`
- ก่อนเริ่มการลอง fallback ตัวรันคำตอบจะคงค่าฟิลด์ override fallback ที่เลือกไว้ลงในรายการเซสชัน
- override fallback อัตโนมัติจะยังคงถูกเลือกไว้ในเทิร์นถัด ๆ ไป เพื่อให้ OpenClaw ไม่ probe primary ที่ทราบว่าไม่ดีในทุกข้อความ OpenClaw จะ probe ต้นทางที่กำหนดค่าไว้อีกครั้งเป็นระยะ และล้าง override อัตโนมัติเมื่อกู้คืนได้; `/new`, `/reset` และ `sessions.reset` จะล้าง override ที่มีแหล่งที่มาอัตโนมัติทันที
- คำตอบของผู้ใช้จะประกาศการเปลี่ยนผ่าน fallback และการกู้คืนเมื่อล้าง fallback หนึ่งครั้งต่อการเปลี่ยนสถานะแต่ละครั้ง เทิร์น fallback แบบติดค้างจะไม่แสดงประกาศซ้ำ
- `/status` แสดงโมเดลที่เลือกไว้ และเมื่อสถานะ fallback แตกต่าง จะแสดงโมเดล fallback ที่ใช้งานอยู่และเหตุผล
- การปรับให้ตรงกันของเซสชันสดจะให้ความสำคัญกับ override เซสชันที่คงอยู่มากกว่าฟิลด์โมเดลรันไทม์ที่เก่าเกินไป
- หากข้อผิดพลาดการสลับแบบสดชี้ไปยังตัวเลือกถัดไปในสาย fallback ที่ใช้งานอยู่ OpenClaw จะกระโดดตรงไปยังโมเดลที่เลือกนั้นแทนการเดินผ่านตัวเลือกที่ไม่เกี่ยวข้องก่อน
- หากความพยายาม fallback ล้มเหลว ตัวรันจะย้อนกลับเฉพาะฟิลด์ override ที่ตัวเองเขียน และเฉพาะเมื่อฟิลด์เหล่านั้นยังตรงกับตัวเลือกที่ล้มเหลวนั้น

สิ่งนี้ป้องกันการแข่งขันแบบคลาสสิก:

<Steps>
  <Step title="Primary ล้มเหลว">
    โมเดล primary ที่เลือกไว้ล้มเหลว
  </Step>
  <Step title="เลือก fallback ในหน่วยความจำ">
    ตัวเลือก fallback ถูกเลือกในหน่วยความจำ
  </Step>
  <Step title="ที่เก็บเซสชันยังบอกว่าเป็น primary เดิม">
    ที่เก็บเซสชันยังสะท้อน primary เดิม
  </Step>
  <Step title="การปรับให้ตรงกันแบบสดอ่านสถานะเก่า">
    การปรับให้ตรงกันของเซสชันสดอ่านสถานะเซสชันที่เก่าเกินไป
  </Step>
  <Step title="การลองใหม่ถูกดึงกลับ">
    การลองใหม่ถูกดึงกลับไปยังโมเดลเดิมก่อนที่ความพยายาม fallback จะเริ่ม
  </Step>
</Steps>

override fallback ที่คงอยู่จะปิดช่องว่างนั้น และการย้อนกลับแบบแคบจะคงการเปลี่ยนแปลงเซสชันแบบแมนนวลหรือรันไทม์ที่ใหม่กว่าไว้เหมือนเดิม

## การสังเกตการณ์และสรุปความล้มเหลว

`runWithModelFallback(...)` บันทึกรายละเอียดต่อความพยายามซึ่งป้อนเข้าสู่บันทึกและข้อความคูลดาวน์ที่แสดงต่อผู้ใช้:

- ผู้ให้บริการ/โมเดลที่ลอง
- เหตุผล (`rate_limit`, `overloaded`, `billing`, `auth`, `model_not_found` และเหตุผลเฟลโอเวอร์ที่คล้ายกัน)
- สถานะ/โค้ดที่ไม่บังคับ
- สรุปข้อผิดพลาดที่มนุษย์อ่านได้

บันทึก `model_fallback_decision` แบบมีโครงสร้างยังรวมฟิลด์แบน `fallbackStep*` เมื่อผู้สมัครล้มเหลว ถูกข้าม หรือ fallback ถัดไปสำเร็จ ฟิลด์เหล่านี้ทำให้การเปลี่ยนผ่านที่พยายามชัดเจน (`fallbackStepFromModel`, `fallbackStepToModel`, `fallbackStepFromFailureReason`, `fallbackStepFromFailureDetail`, `fallbackStepFinalOutcome`) เพื่อให้ตัวส่งออกบันทึกและการวินิจฉัยสามารถสร้างความล้มเหลวของ primary ขึ้นใหม่ได้ แม้ fallback สุดท้ายจะล้มเหลวด้วยก็ตาม

เมื่อทุกตัวเลือกล้มเหลว OpenClaw จะโยน `FallbackSummaryError` ตัวรันคำตอบชั้นนอกสามารถใช้สิ่งนั้นเพื่อสร้างข้อความที่เฉพาะเจาะจงมากขึ้น เช่น "ทุกโมเดลถูกจำกัดอัตราชั่วคราว" และรวมเวลาหมดอายุคูลดาวน์ที่เร็วที่สุดเมื่อทราบ

สรุปคูลดาวน์นั้นรับรู้โมเดล:

- การจำกัดอัตราตามขอบเขตโมเดลที่ไม่เกี่ยวข้องจะถูกละเว้นสำหรับสายผู้ให้บริการ/โมเดลที่พยายาม
- หากบล็อกที่เหลืออยู่เป็นการจำกัดอัตราตามขอบเขตโมเดลที่ตรงกัน OpenClaw จะรายงานเวลาหมดอายุล่าสุดที่ตรงกันซึ่งยังบล็อกโมเดลนั้นอยู่
