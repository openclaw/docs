OpenClaw จัดการความล้มเหลวเป็นสองขั้นตอน:

1. **การหมุนเวียนโปรไฟล์การยืนยันตัวตน** ภายในผู้ให้บริการปัจจุบัน
2. **การถอยกลับไปใช้โมเดลสำรอง** ไปยังโมเดลถัดไปใน `agents.defaults.model.fallbacks`

เอกสารนี้อธิบายกฎขณะรันและข้อมูลที่รองรับกฎเหล่านั้น

## โฟลว์ขณะรัน

สำหรับการรันข้อความปกติ OpenClaw จะประเมินตัวเลือกตามลำดับนี้:

<Steps>
  <Step title="แก้ไขสถานะเซสชัน">
    แก้ไขโมเดลเซสชันที่ใช้งานอยู่และค่ากำหนดโปรไฟล์การยืนยันตัวตน
  </Step>
  <Step title="สร้างลำดับตัวเลือก">
    สร้างลำดับตัวเลือกโมเดลจากการเลือกโมเดลปัจจุบันและนโยบายการถอยกลับสำหรับแหล่งที่มาของการเลือกนั้น ค่าเริ่มต้นที่ตั้งค่าไว้ โมเดลหลักของงาน Cron และโมเดลสำรองที่เลือกอัตโนมัติสามารถใช้รายการสำรองที่ตั้งค่าไว้ได้ ส่วนการเลือกเซสชันโดยผู้ใช้อย่างชัดเจนจะเข้มงวด
  </Step>
  <Step title="ลองผู้ให้บริการปัจจุบัน">
    ลองผู้ให้บริการปัจจุบันด้วยกฎการหมุนเวียน/คูลดาวน์ของโปรไฟล์การยืนยันตัวตน
  </Step>
  <Step title="เดินหน้าต่อเมื่อเกิดข้อผิดพลาดที่ควรย้ายไปใช้ตัวสำรอง">
    หากผู้ให้บริการนั้นถูกใช้จนหมดพร้อมข้อผิดพลาดที่ควรย้ายไปใช้ตัวสำรอง ให้ย้ายไปยังตัวเลือกโมเดลถัดไป
  </Step>
  <Step title="บันทึกการเขียนทับสำรอง">
    บันทึกการเขียนทับสำรองที่เลือกไว้ก่อนเริ่มลองใหม่ เพื่อให้ตัวอ่านเซสชันอื่นเห็นผู้ให้บริการ/โมเดลเดียวกับที่ runner กำลังจะใช้ การเขียนทับโมเดลที่บันทึกไว้จะถูกทำเครื่องหมายเป็น `modelOverrideSource: "auto"`
  </Step>
  <Step title="ย้อนกลับแบบจำกัดเมื่อเกิดความล้มเหลว">
    หากตัวเลือกสำรองล้มเหลว ให้ย้อนกลับเฉพาะฟิลด์การเขียนทับเซสชันที่เป็นของ fallback เมื่อฟิลด์เหล่านั้นยังตรงกับตัวเลือกที่ล้มเหลวนั้น
  </Step>
  <Step title="โยน FallbackSummaryError หากใช้จนหมด">
    หากทุกตัวเลือกล้มเหลว ให้โยน `FallbackSummaryError` พร้อมรายละเอียดรายครั้งที่ลองและเวลาสิ้นสุดคูลดาวน์ที่เร็วที่สุดเมื่อทราบ
  </Step>
</Steps>

เจตนาของสิ่งนี้แคบกว่า "บันทึกและกู้คืนทั้งเซสชัน" reply runner จะบันทึกเฉพาะฟิลด์การเลือกโมเดลที่ตนเป็นเจ้าของสำหรับ fallback:

- `providerOverride`
- `modelOverride`
- `modelOverrideSource`
- `authProfileOverride`
- `authProfileOverrideSource`
- `authProfileOverrideCompactionCount`

สิ่งนี้ป้องกันไม่ให้การลอง fallback ใหม่ที่ล้มเหลวเขียนทับการเปลี่ยนแปลงเซสชันอื่นที่ใหม่กว่าและไม่เกี่ยวข้อง เช่น การเปลี่ยน `/model` ด้วยตนเอง หรือการอัปเดตการหมุนเวียนเซสชันที่เกิดขึ้นขณะกำลังลองอยู่

## นโยบายแหล่งที่มาของการเลือก

OpenClaw แยกผู้ให้บริการ/โมเดลที่เลือกออกจากเหตุผลที่เลือก แหล่งที่มานั้นควบคุมว่าจะอนุญาตให้ใช้ลำดับ fallback หรือไม่:

- **ค่าเริ่มต้นที่ตั้งค่าไว้**: `agents.defaults.model.primary` ใช้ `agents.defaults.model.fallbacks`
- **โมเดลหลักของเอเจนต์**: `agents.list[].model` จะเข้มงวด เว้นแต่ออบเจ็กต์โมเดลของเอเจนต์นั้นจะมี `fallbacks` ของตัวเอง ใช้ `fallbacks: []` เพื่อทำให้พฤติกรรมเข้มงวดชัดเจน หรือระบุรายการที่ไม่ว่างเพื่อให้เอเจนต์นั้นเลือกใช้การ fallback โมเดล
- **การเขียนทับ fallback อัตโนมัติ**: fallback ขณะรันจะเขียน `providerOverride`, `modelOverride`, `modelOverrideSource: "auto"` และโมเดลต้นทางที่เลือกก่อนลองใหม่ การเขียนทับอัตโนมัตินั้นสามารถเดินต่อไปตามลำดับ fallback ที่ตั้งค่าไว้ และจะถูกล้างโดย `/new`, `/reset` และ `sessions.reset` การรัน Heartbeat ที่ไม่มี `heartbeat.model` ชัดเจนจะล้างการเขียนทับอัตโนมัติโดยตรงด้วยเมื่อต้นทางของมันไม่ตรงกับค่าเริ่มต้นที่ตั้งค่าไว้ปัจจุบันอีกต่อไป
- **การเขียนทับเซสชันโดยผู้ใช้**: `/model`, ตัวเลือกโมเดล, `session_status(model=...)` และ `sessions.patch` จะเขียน `modelOverrideSource: "user"` นี่คือการเลือกเซสชันที่แน่นอน หากผู้ให้บริการ/โมเดลที่เลือกล้มเหลวก่อนสร้างคำตอบ OpenClaw จะรายงานความล้มเหลวแทนที่จะตอบจาก fallback ที่ตั้งค่าไว้ซึ่งไม่เกี่ยวข้อง
- **การเขียนทับเซสชันรุ่นเก่า**: รายการเซสชันรุ่นเก่าอาจมี `modelOverride` โดยไม่มี `modelOverrideSource` OpenClaw จะถือว่าสิ่งเหล่านั้นเป็นการเขียนทับโดยผู้ใช้ เพื่อไม่ให้การเลือกเก่าอย่างชัดเจนถูกเปลี่ยนเป็นพฤติกรรม fallback อย่างเงียบๆ
- **โมเดล payload ของ Cron**: งาน Cron ที่มี `payload.model` / `--model` คือโมเดลหลักของงาน ไม่ใช่การเขียนทับเซสชันโดยผู้ใช้ โดยจะใช้ fallback ที่ตั้งค่าไว้ เว้นแต่งานจะระบุ `payload.fallbacks`; `payload.fallbacks: []` ทำให้การรัน Cron เข้มงวด

## ที่เก็บการยืนยันตัวตน (คีย์ + OAuth)

OpenClaw ใช้ **โปรไฟล์การยืนยันตัวตน** สำหรับทั้งคีย์ API และโทเค็น OAuth

- ข้อมูลลับอยู่ใน `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` (เดิม: `~/.openclaw/agent/auth-profiles.json`)
- สถานะการกำหนดเส้นทางการยืนยันตัวตนขณะรันอยู่ใน `~/.openclaw/agents/<agentId>/agent/auth-state.json`
- Config `auth.profiles` / `auth.order` เป็น **ข้อมูลเมตา + การกำหนดเส้นทางเท่านั้น** (ไม่มีข้อมูลลับ)
- ไฟล์ OAuth เดิมสำหรับนำเข้าเท่านั้น: `~/.openclaw/credentials/oauth.json` (นำเข้าไปยัง `auth-profiles.json` เมื่อใช้งานครั้งแรก)

รายละเอียดเพิ่มเติม: [OAuth](/th/concepts/oauth)

ประเภทข้อมูลรับรอง:

- `type: "api_key"` → `{ provider, key }`
- `type: "oauth"` → `{ provider, access, refresh, expires, email? }` (+ `projectId`/`enterpriseUrl` สำหรับผู้ให้บริการบางราย)

## ID โปรไฟล์

การเข้าสู่ระบบ OAuth จะสร้างโปรไฟล์แยกกันเพื่อให้หลายบัญชีอยู่ร่วมกันได้

- ค่าเริ่มต้น: `provider:default` เมื่อไม่มีอีเมล
- OAuth พร้อมอีเมล: `provider:<email>` (เช่น `google-antigravity:user@gmail.com`)

โปรไฟล์อยู่ใน `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` ภายใต้ `profiles`

## ลำดับการหมุนเวียน

เมื่อผู้ให้บริการมีหลายโปรไฟล์ OpenClaw จะเลือกลำดับดังนี้:

<Steps>
  <Step title="Config ชัดเจน">
    `auth.order[provider]` (หากตั้งค่าไว้)
  </Step>
  <Step title="โปรไฟล์ที่ตั้งค่าไว้">
    `auth.profiles` ที่กรองตามผู้ให้บริการ
  </Step>
  <Step title="โปรไฟล์ที่จัดเก็บไว้">
    รายการใน `auth-profiles.json` สำหรับผู้ให้บริการ
  </Step>
</Steps>

หากไม่ได้ตั้งค่าลำดับอย่างชัดเจน OpenClaw จะใช้ลำดับแบบวนรอบ:

- **คีย์หลัก:** ประเภทโปรไฟล์ (**OAuth ก่อนคีย์ API**)
- **คีย์รอง:** `usageStats.lastUsed` (เก่าที่สุดก่อน ภายในแต่ละประเภท)
- **โปรไฟล์ที่อยู่ในคูลดาวน์/ถูกปิดใช้งาน** จะถูกย้ายไปท้ายสุด เรียงตามเวลาสิ้นสุดที่เร็วที่สุด

### การยึดติดกับเซสชัน (เป็นมิตรกับแคช)

OpenClaw **ตรึงโปรไฟล์การยืนยันตัวตนที่เลือกต่อเซสชัน** เพื่อให้แคชของผู้ให้บริการยังอุ่นอยู่ ระบบ **จะไม่** หมุนเวียนทุกคำขอ โปรไฟล์ที่ตรึงไว้จะถูกใช้ซ้ำจนกว่า:

- เซสชันจะถูกรีเซ็ต (`/new` / `/reset`)
- Compaction เสร็จสมบูรณ์ (จำนวน Compaction เพิ่มขึ้น)
- โปรไฟล์อยู่ในคูลดาวน์/ถูกปิดใช้งาน

การเลือกด้วยตนเองผ่าน `/model …@<profileId>` จะตั้งค่า **การเขียนทับโดยผู้ใช้** สำหรับเซสชันนั้น และจะไม่ถูกหมุนเวียนอัตโนมัติจนกว่าจะเริ่มเซสชันใหม่

<Note>
โปรไฟล์ที่ตรึงอัตโนมัติ (เลือกโดยเราเตอร์เซสชัน) จะถูกถือเป็น **ค่ากำหนด**: จะถูกลองก่อน แต่ OpenClaw อาจหมุนเวียนไปยังโปรไฟล์อื่นเมื่อเกิด rate limit/timeout โปรไฟล์ที่ผู้ใช้ตรึงจะยังล็อกอยู่กับโปรไฟล์นั้น หากล้มเหลวและมีการตั้งค่า fallback ของโมเดล OpenClaw จะย้ายไปยังโมเดลถัดไปแทนการสลับโปรไฟล์
</Note>

### เหตุใด OAuth จึงอาจ "ดูเหมือนหายไป"

หากคุณมีทั้งโปรไฟล์ OAuth และโปรไฟล์คีย์ API สำหรับผู้ให้บริการเดียวกัน การวนรอบอาจสลับระหว่างโปรไฟล์เหล่านั้นข้ามข้อความ เว้นแต่จะถูกตรึงไว้ หากต้องการบังคับใช้โปรไฟล์เดียว:

- ตรึงด้วย `auth.order[provider] = ["provider:profileId"]`, หรือ
- ใช้การเขียนทับต่อเซสชันผ่าน `/model …` พร้อมการเขียนทับโปรไฟล์ (เมื่อ UI/พื้นผิวแชตของคุณรองรับ)

## คูลดาวน์

เมื่อโปรไฟล์ล้มเหลวจากข้อผิดพลาดการยืนยันตัวตน/rate-limit (หรือ timeout ที่ดูเหมือน rate limiting) OpenClaw จะทำเครื่องหมายให้โปรไฟล์นั้นอยู่ในคูลดาวน์และย้ายไปยังโปรไฟล์ถัดไป

<AccordionGroup>
  <Accordion title="สิ่งที่อยู่ในกลุ่ม rate-limit / timeout">
    กลุ่ม rate-limit นั้นกว้างกว่า `429` ธรรมดา: รวมถึงข้อความจากผู้ให้บริการ เช่น `Too many concurrent requests`, `ThrottlingException`, `concurrency limit reached`, `workers_ai ... quota limit exceeded`, `throttled`, `resource exhausted` และขีดจำกัดหน้าต่างการใช้งานเป็นช่วงๆ เช่น `weekly/monthly limit reached`

    ข้อผิดพลาดรูปแบบ/คำขอไม่ถูกต้องมักเป็นจุดสิ้นสุด เพราะการลอง payload เดิมใหม่จะล้มเหลวแบบเดิม ดังนั้น OpenClaw จะแสดงข้อผิดพลาดเหล่านั้นแทนการหมุนเวียนโปรไฟล์การยืนยันตัวตน เส้นทางลองซ่อมแซมที่รู้จักสามารถเลือกใช้ได้อย่างชัดเจน: เช่น ความล้มเหลวในการตรวจสอบ ID การเรียกเครื่องมือ Cloud Code Assist จะถูกปรับให้ปลอดภัยและลองใหม่หนึ่งครั้งผ่านนโยบาย `allowFormatRetry` ข้อผิดพลาด stop-reason ที่เข้ากันได้กับ OpenAI เช่น `Unhandled stop reason: error`, `stop reason: error` และ `reason: error` จะถูกจัดประเภทเป็นสัญญาณ timeout/failover

    ข้อความเซิร์ฟเวอร์ทั่วไปก็อาจอยู่ในกลุ่ม timeout นั้นได้เช่นกันเมื่อแหล่งที่มาตรงกับรูปแบบชั่วคราวที่รู้จัก ตัวอย่างเช่น ข้อความ stream-wrapper ของ pi-ai แบบไม่มีรายละเอียด `An unknown error occurred` จะถูกถือว่าควรย้ายไปใช้ตัวสำรองสำหรับทุกผู้ให้บริการ เพราะ pi-ai ส่งข้อความนี้เมื่อสตรีมของผู้ให้บริการจบด้วย `stopReason: "aborted"` หรือ `stopReason: "error"` โดยไม่มีรายละเอียดเฉพาะ payload JSON `api_error` ที่มีข้อความเซิร์ฟเวอร์ชั่วคราว เช่น `internal server error`, `unknown error, 520`, `upstream error` หรือ `backend error` จะถูกถือเป็น timeout ที่ควรย้ายไปใช้ตัวสำรองเช่นกัน

    ข้อความ upstream ทั่วไปเฉพาะ OpenRouter เช่น `Provider returned error` แบบไม่มีรายละเอียด จะถูกถือเป็น timeout เฉพาะเมื่อบริบทผู้ให้บริการเป็น OpenRouter จริงๆ ข้อความ fallback ภายในทั่วไป เช่น `LLM request failed with an unknown error.` จะคงความระมัดระวังและไม่ทริกเกอร์ failover ด้วยตัวมันเอง

  </Accordion>
  <Accordion title="เพดาน retry-after ของ SDK">
    SDK ของผู้ให้บริการบางรายอาจพักเป็นช่วง `Retry-After` ที่ยาวก่อนคืนการควบคุมให้ OpenClaw สำหรับ SDK ที่ใช้ Stainless เช่น Anthropic และ OpenAI นั้น OpenClaw จะจำกัดการรอ `retry-after-ms` / `retry-after` ภายใน SDK ไว้ที่ 60 วินาทีโดยค่าเริ่มต้น และแสดงผลตอบกลับที่ลองใหม่ได้ซึ่งยาวกว่านั้นทันที เพื่อให้เส้นทาง failover นี้ทำงานได้ ปรับแต่งหรือปิดเพดานด้วย `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS`; ดู [พฤติกรรมการลองใหม่](/th/concepts/retry)
  </Accordion>
  <Accordion title="คูลดาวน์เฉพาะโมเดล">
    คูลดาวน์ rate-limit สามารถจำกัดเฉพาะโมเดลได้ด้วย:

    - OpenClaw บันทึก `cooldownModel` สำหรับความล้มเหลวแบบ rate-limit เมื่อทราบ ID โมเดลที่ล้มเหลว
    - โมเดลพี่น้องบนผู้ให้บริการเดียวกันยังสามารถถูกลองได้เมื่อคูลดาวน์จำกัดอยู่กับโมเดลอื่น
    - หน้าต่าง billing/disabled ยังบล็อกทั้งโปรไฟล์ข้ามโมเดล

  </Accordion>
</AccordionGroup>

คูลดาวน์ใช้ exponential backoff:

- 1 นาที
- 5 นาที
- 25 นาที
- 1 ชั่วโมง (เพดาน)

```json
      "lastUsed": 1736160000000,
      "cooldownUntil": 1736160600000,
      "errorCount": 2
    }
  }
}
```

## การปิดใช้งานจาก billing

ความล้มเหลวด้าน billing/เครดิต (เช่น "insufficient credits" / "credit balance too low") จะถูกถือว่าควรย้ายไปใช้ตัวสำรอง แต่โดยปกติไม่ใช่ข้อผิดพลาดชั่วคราว แทนที่จะใช้คูลดาวน์สั้นๆ OpenClaw จะทำเครื่องหมายโปรไฟล์เป็น **disabled** (พร้อม backoff ที่ยาวกว่า) และหมุนเวียนไปยังโปรไฟล์/ผู้ให้บริการถัดไป

<Note>
ไม่ใช่ทุกผลตอบกลับที่มีลักษณะ billing จะเป็น `402` และไม่ใช่ทุก HTTP `402` ที่จะเข้าทางนี้ OpenClaw จะเก็บข้อความ billing ที่ชัดเจนไว้ในเส้นทาง billing แม้ผู้ให้บริการจะคืน `401` หรือ `403` แทน แต่ตัวจับคู่เฉพาะผู้ให้บริการจะยังจำกัดอยู่กับผู้ให้บริการที่เป็นเจ้าของมัน (เช่น OpenRouter `403 Key limit exceeded`)

ในขณะเดียวกัน ข้อผิดพลาดชั่วคราวแบบ `402` สำหรับหน้าต่างการใช้งานและขีดจำกัดค่าใช้จ่ายขององค์กร/เวิร์กสเปซจะถูกจัดประเภทเป็น `rate_limit` เมื่อข้อความดูเหมือนลองใหม่ได้ (เช่น `weekly usage limit exhausted`, `daily limit reached, resets tomorrow` หรือ `organization spending limit exceeded`) สิ่งเหล่านั้นจะอยู่บนเส้นทางคูลดาวน์/ย้ายไปใช้ตัวสำรองระยะสั้นแทนเส้นทางปิดใช้งานจาก billing ระยะยาว
</Note>

สถานะถูกจัดเก็บใน `auth-state.json`:

```json
{
  "usageStats": {
    "provider:profile": {
      "disabledUntil": 1736178000000,
      "disabledReason": "billing"
    }
  }
}
```

ค่าเริ่มต้น:

- backoff ของ billing เริ่มที่ **5 ชั่วโมง**, เพิ่มเป็นสองเท่าต่อความล้มเหลวด้าน billing แต่ละครั้ง และมีเพดานที่ **24 ชั่วโมง**
- ตัวนับ backoff จะรีเซ็ตหากโปรไฟล์ไม่ล้มเหลวเป็นเวลา **24 ชั่วโมง** (ตั้งค่าได้)
- การลองใหม่เมื่อ overload อนุญาตให้ **หมุนเวียนโปรไฟล์ผู้ให้บริการเดียวกัน 1 ครั้ง** ก่อน fallback โมเดล
- การลองใหม่เมื่อ overload ใช้ **0 ms backoff** โดยค่าเริ่มต้น

## การ fallback โมเดล

หากโปรไฟล์ทั้งหมดสำหรับผู้ให้บริการล้มเหลว OpenClaw จะไปยังโมเดลถัดไปใน `agents.defaults.model.fallbacks` การทำงานนี้ใช้กับความล้มเหลวของการยืนยันตัวตน การจำกัดอัตรา และการหมดเวลาที่ใช้การหมุนเวียนโปรไฟล์จนหมดแล้ว (ข้อผิดพลาดอื่นจะไม่เลื่อนไปยังตัวสำรอง) ข้อผิดพลาดของผู้ให้บริการที่ไม่ได้เปิดเผยรายละเอียดเพียงพอยังคงถูกติดป้ายอย่างแม่นยำในสถานะตัวสำรอง: `empty_response` หมายถึงผู้ให้บริการไม่ได้ส่งคืนข้อความหรือสถานะที่ใช้งานได้, `no_error_details` หมายถึงผู้ให้บริการส่งคืน `Unknown error (no error details in response)` อย่างชัดเจน และ `unclassified` หมายถึง OpenClaw เก็บตัวอย่างดิบไว้แล้ว แต่ยังไม่มีตัวจัดประเภทใดตรงกับข้อผิดพลาดนั้น

ข้อผิดพลาดแบบโอเวอร์โหลดและการจำกัดอัตราจะถูกจัดการเชิงรุกกว่าคูลดาวน์ด้านการเรียกเก็บเงิน โดยค่าเริ่มต้น OpenClaw อนุญาตให้ลองโปรไฟล์ยืนยันตัวตนของผู้ให้บริการเดิมซ้ำหนึ่งครั้ง จากนั้นจึงสลับไปยังตัวสำรองโมเดลถัดไปที่กำหนดค่าไว้โดยไม่รอ สัญญาณว่าผู้ให้บริการไม่ว่าง เช่น `ModelNotReadyException` จะอยู่ในกลุ่มโอเวอร์โหลดนี้ ปรับแต่งได้ด้วย `auth.cooldowns.overloadedProfileRotations`, `auth.cooldowns.overloadedBackoffMs` และ `auth.cooldowns.rateLimitedProfileRotations`

เมื่อการรันเริ่มจากโมเดลหลักค่าเริ่มต้นที่กำหนดค่าไว้, โมเดลหลักของงาน Cron, โมเดลหลักของเอเจนต์ที่มีตัวสำรองชัดเจน หรือการแทนที่ตัวสำรองที่เลือกโดยอัตโนมัติ OpenClaw สามารถไล่ตามสายตัวสำรองที่กำหนดค่าไว้ให้ตรงกันได้ โมเดลหลักของเอเจนต์ที่ไม่มีตัวสำรองชัดเจนและการเลือกของผู้ใช้โดยตรง (เช่น `/model ollama/qwen3.5:27b`, ตัวเลือกโมเดล, `sessions.patch` หรือการแทนที่ผู้ให้บริการ/โมเดลแบบครั้งเดียวผ่าน CLI) จะเข้มงวด: หากผู้ให้บริการ/โมเดลนั้นเข้าถึงไม่ได้หรือล้มเหลวก่อนสร้างคำตอบ OpenClaw จะรายงานความล้มเหลวแทนการตอบจากตัวสำรองที่ไม่เกี่ยวข้อง

<AccordionGroup>
  <Accordion title="กฎ">

    - ตัวสำรองที่กำหนดค่าไว้อย่างชัดเจนจะถูกลบค่าซ้ำ แต่จะไม่ถูกกรองด้วยรายการอนุญาตของโมเดล รายการเหล่านี้ถือเป็นเจตนาชัดเจนของผู้ปฏิบัติงาน
    - หากการรันปัจจุบันอยู่บนตัวสำรองที่กำหนดค่าไว้แล้วในตระกูลผู้ให้บริการเดียวกัน OpenClaw จะใช้สายที่กำหนดค่าไว้ทั้งหมดต่อไป
    - หากการรันปัจจุบันอยู่บนผู้ให้บริการคนละรายกับการกำหนดค่า และโมเดลปัจจุบันนั้นยังไม่ได้เป็นส่วนหนึ่งของสายตัวสำรองที่กำหนดค่าไว้ OpenClaw จะไม่ต่อท้ายตัวสำรองที่กำหนดค่าไว้ซึ่งไม่เกี่ยวข้องจากผู้ให้บริการอีกราย
    - เมื่อไม่ได้ส่งการแทนที่ตัวสำรองอย่างชัดเจนให้ตัวรันตัวสำรอง โมเดลหลักที่กำหนดค่าไว้จะถูกต่อท้ายไว้ตอนท้าย เพื่อให้สายสามารถกลับไปอยู่บนค่าเริ่มต้นปกติได้หลังจากตัวเลือกก่อนหน้าถูกใช้จนหมด
    - เมื่อผู้เรียกส่ง `fallbacksOverride` ตัวรันจะใช้เฉพาะโมเดลที่ร้องขอพร้อมกับรายการแทนที่นั้นเท่านั้น รายการว่างจะปิดใช้ตัวสำรองโมเดลและป้องกันไม่ให้โมเดลหลักที่กำหนดค่าไว้ถูกต่อท้ายเป็นเป้าหมายลองซ้ำแบบซ่อน

### พฤติกรรมการข้ามคูลดาวน์เทียบกับการตรวจสอบ

เมื่อโปรไฟล์ยืนยันตัวตนทุกโปรไฟล์สำหรับผู้ให้บริการอยู่ในคูลดาวน์แล้ว OpenClaw จะไม่ข้ามผู้ให้บริการนั้นตลอดไปโดยอัตโนมัติ แต่จะตัดสินใจแยกตามตัวเลือก:

    - ความล้มเหลวของการยืนยันตัวตนแบบถาวรจะข้ามผู้ให้บริการทั้งหมดทันที
    - การปิดใช้งานด้านการเรียกเก็บเงินมักจะถูกข้าม แต่ตัวเลือกหลักยังคงถูกตรวจสอบแบบจำกัดความถี่ได้ เพื่อให้กู้คืนได้โดยไม่ต้องเริ่มใหม่
    - ตัวเลือกหลักอาจถูกตรวจสอบใกล้เวลาคูลดาวน์หมดอายุ โดยมีการจำกัดความถี่แยกตามผู้ให้บริการ

</AccordionGroup>

## การแทนที่เซสชันและการสลับโมเดลแบบสด

การเปลี่ยนโมเดลของเซสชันเป็นสถานะที่ใช้ร่วมกัน ตัวรันที่ใช้งานอยู่, คำสั่ง `/model`, การอัปเดต Compaction/เซสชัน และการปรับเทียบเซสชันสด ต่างอ่านหรือเขียนบางส่วนของรายการเซสชันเดียวกัน

นั่นหมายความว่าการลองซ้ำด้วยตัวสำรองต้องประสานกับการสลับโมเดลแบบสด:

- เฉพาะการเปลี่ยนโมเดลที่ผู้ใช้สั่งอย่างชัดเจนเท่านั้นที่จะทำเครื่องหมายว่ามีการสลับแบบสดที่รอดำเนินการ ซึ่งรวมถึง `/model`, `session_status(model=...)` และ `sessions.patch`
- การเปลี่ยนโมเดลที่ระบบขับเคลื่อน เช่น การหมุนเวียนตัวสำรอง, การแทนที่จาก Heartbeat หรือ Compaction จะไม่ทำเครื่องหมายว่ามีการสลับแบบสดที่รอดำเนินการด้วยตัวเอง
- การแทนที่โมเดลที่ผู้ใช้สั่งจะถือเป็นการเลือกแบบตรงตัวสำหรับนโยบายตัวสำรอง ดังนั้นผู้ให้บริการที่เลือกไว้แต่เข้าถึงไม่ได้จะแสดงเป็นความล้มเหลวแทนการถูกปิดบังด้วย `agents.defaults.model.fallbacks`
- ก่อนการลองซ้ำด้วยตัวสำรองเริ่มขึ้น ตัวรันคำตอบจะบันทึกฟิลด์การแทนที่ตัวสำรองที่เลือกลงในรายการเซสชัน
- การแทนที่ตัวสำรองอัตโนมัติจะยังคงถูกเลือกในเทิร์นถัดไป เพื่อให้ OpenClaw ไม่ตรวจสอบโมเดลหลักที่ทราบว่าเสียในทุกข้อความ `/new`, `/reset` และ `sessions.reset` จะล้างการแทนที่ที่มาจากอัตโนมัติและคืนเซสชันกลับสู่ค่าเริ่มต้นที่กำหนดค่าไว้
- `/status` แสดงโมเดลที่เลือก และเมื่อสถานะตัวสำรองแตกต่างกัน จะแสดงโมเดลตัวสำรองที่ใช้งานอยู่พร้อมเหตุผล
- การปรับเทียบเซสชันสดให้ความสำคัญกับการแทนที่เซสชันที่บันทึกไว้มากกว่าฟิลด์โมเดลรันไทม์ที่ค้างอยู่
- หากข้อผิดพลาดการสลับแบบสดชี้ไปยังตัวเลือกถัดไปในสายตัวสำรองที่ใช้งานอยู่ OpenClaw จะกระโดดไปยังโมเดลที่เลือกนั้นโดยตรง แทนที่จะไล่ตัวเลือกที่ไม่เกี่ยวข้องก่อน
- หากความพยายามใช้ตัวสำรองล้มเหลว ตัวรันจะย้อนกลับเฉพาะฟิลด์การแทนที่ที่เขียนไว้ และเฉพาะเมื่อฟิลด์เหล่านั้นยังตรงกับตัวเลือกที่ล้มเหลวนั้น

## การสังเกตการณ์และสรุปความล้มเหลว

`runWithModelFallback(...)` บันทึกรายละเอียดรายความพยายามที่ป้อนให้บันทึกและข้อความคูลดาวน์ที่แสดงต่อผู้ใช้:

- ผู้ให้บริการ/โมเดลที่ลองใช้
- เหตุผล (`rate_limit`, `overloaded`, `billing`, `auth`, `model_not_found` และเหตุผลการสลับไปสำรองที่คล้ายกัน)
- สถานะ/โค้ดที่เป็นตัวเลือก
- สรุปข้อผิดพลาดที่มนุษย์อ่านได้

บันทึก `model_fallback_decision` แบบมีโครงสร้างยังมีฟิลด์แบน `fallbackStep*` เมื่อมีตัวเลือกล้มเหลว ถูกข้าม หรือมีตัวสำรองถัดไปสำเร็จ ฟิลด์เหล่านี้ทำให้การเปลี่ยนผ่านที่ลองทำชัดเจน (`fallbackStepFromModel`, `fallbackStepToModel`, `fallbackStepFromFailureReason`, `fallbackStepFromFailureDetail`, `fallbackStepFinalOutcome`) เพื่อให้ตัวส่งออกบันทึกและข้อมูลวินิจฉัยสามารถสร้างความล้มเหลวของโมเดลหลักขึ้นใหม่ได้ แม้เมื่อตัวสำรองสุดท้ายล้มเหลวด้วย

เมื่อทุกตัวเลือกล้มเหลว OpenClaw จะโยน `FallbackSummaryError` ตัวรันคำตอบชั้นนอกสามารถใช้สิ่งนั้นเพื่อสร้างข้อความที่เจาะจงขึ้น เช่น "ทุกโมเดลถูกจำกัดอัตราชั่วคราว" และรวมเวลาคูลดาวน์ที่เร็วที่สุดเมื่อทราบ

สรุปคูลดาวน์นั้นรับรู้ระดับโมเดล:

- การจำกัดอัตราแบบผูกกับโมเดลที่ไม่เกี่ยวข้องจะถูกละเว้นสำหรับสายผู้ให้บริการ/โมเดลที่พยายามใช้
- หากการบล็อกที่เหลือเป็นการจำกัดอัตราแบบผูกกับโมเดลที่ตรงกัน OpenClaw จะรายงานเวลาหมดอายุที่ตรงกันล่าสุดซึ่งยังบล็อกโมเดลนั้นอยู่
