เอ็นจินในตัวคือแบ็กเอนด์หน่วยความจำเริ่มต้น โดยจะจัดเก็บดัชนีหน่วยความจำของคุณไว้ใน
ฐานข้อมูล SQLite แยกตามเอเจนต์ และไม่ต้องมี dependency เพิ่มเติมเพื่อเริ่มใช้งาน

## สิ่งที่มีให้

- **การค้นหาด้วยคีย์เวิร์ด** ผ่านการทำดัชนีข้อความเต็ม FTS5 (การให้คะแนน BM25)
- **การค้นหาเวกเตอร์** ผ่าน embeddings จากผู้ให้บริการที่รองรับ
- **การค้นหาแบบไฮบริด** ที่รวมทั้งสองแบบเพื่อให้ได้ผลลัพธ์ดีที่สุด
- **การรองรับ CJK** ผ่านการตัดคำแบบ trigram สำหรับภาษาจีน ญี่ปุ่น และเกาหลี
- **การเร่งความเร็วด้วย sqlite-vec** สำหรับคิวรีเวกเตอร์ภายในฐานข้อมูล (ไม่บังคับ)

## เริ่มต้นใช้งาน

หากคุณมีคีย์ API สำหรับ OpenAI, Gemini, Voyage, Mistral หรือ DeepInfra เอ็นจินในตัว
จะตรวจหาโดยอัตโนมัติและเปิดใช้การค้นหาเวกเตอร์ ไม่ต้องตั้งค่าใด ๆ

หากต้องการตั้งค่าผู้ให้บริการอย่างชัดเจน:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
      },
    },
  },
}
```

หากไม่มีผู้ให้บริการ embedding จะใช้งานได้เฉพาะการค้นหาด้วยคีย์เวิร์ดเท่านั้น

หากต้องการบังคับใช้ผู้ให้บริการ embedding ในเครื่องแบบในตัว ให้ติดตั้งแพ็กเกจรันไทม์
`node-llama-cpp` แบบไม่บังคับไว้ข้าง OpenClaw จากนั้นชี้ `local.modelPath`
ไปยังไฟล์ GGUF:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "local",
        fallback: "none",
        local: {
          modelPath: "~/.node-llama-cpp/models/embeddinggemma-300m-qat-Q8_0.gguf",
        },
      },
    },
  },
}
```

## ผู้ให้บริการ embedding ที่รองรับ

| ผู้ให้บริการ | ID          | ตรวจพบอัตโนมัติ | หมายเหตุ                                  |
| --------- | ----------- | ------------- | ----------------------------------- |
| OpenAI    | `openai`    | ใช่           | ค่าเริ่มต้น: `text-embedding-3-small`   |
| Gemini    | `gemini`    | ใช่           | รองรับหลายรูปแบบ (รูปภาพ + เสียง) |
| Voyage    | `voyage`    | ใช่           |                                     |
| Mistral   | `mistral`   | ใช่           |                                     |
| DeepInfra | `deepinfra` | ใช่           | ค่าเริ่มต้น: `BAAI/bge-m3`              |
| Ollama    | `ollama`    | ไม่            | ในเครื่อง ตั้งค่าอย่างชัดเจน               |
| ในเครื่อง     | `local`     | ใช่ (ลำดับแรก)   | รันไทม์ `node-llama-cpp` แบบไม่บังคับ   |

การตรวจหาอัตโนมัติจะเลือกผู้ให้บริการรายแรกที่สามารถ resolve คีย์ API ได้ ตาม
ลำดับที่แสดง ตั้งค่า `memorySearch.provider` เพื่อแทนที่

## การทำดัชนีทำงานอย่างไร

OpenClaw ทำดัชนี `MEMORY.md` และ `memory/*.md` เป็นชิ้นส่วน (~400 tokens พร้อม
ส่วนซ้อนทับ 80-token) และจัดเก็บไว้ในฐานข้อมูล SQLite แยกตามเอเจนต์

- **ตำแหน่งดัชนี:** `~/.openclaw/memory/<agentId>.sqlite`
- **การบำรุงรักษาพื้นที่จัดเก็บ:** ไฟล์ข้างเคียง SQLite WAL จะถูกจำกัดขนาดด้วย checkpoint ตามรอบและ
  ตอนปิดระบบ
- **การเฝ้าดูไฟล์:** การเปลี่ยนแปลงไฟล์หน่วยความจำจะกระตุ้นการทำดัชนีใหม่แบบ debounce (1.5 วินาที)
- **การทำดัชนีใหม่อัตโนมัติ:** เมื่อผู้ให้บริการ embedding, โมเดล หรือค่าการแบ่งชิ้นส่วน
  เปลี่ยนแปลง ดัชนีทั้งหมดจะถูกสร้างใหม่โดยอัตโนมัติ
- **ทำดัชนีใหม่ตามต้องการ:** `openclaw memory index --force`

<Info>
คุณยังสามารถทำดัชนีไฟล์ Markdown นอก workspace ได้ด้วย
`memorySearch.extraPaths` ดู
[ข้อมูลอ้างอิงการตั้งค่า](/th/reference/memory-config#additional-memory-paths)
</Info>

## ควรใช้เมื่อใด

เอ็นจินในตัวเป็นตัวเลือกที่เหมาะสำหรับผู้ใช้ส่วนใหญ่:

- ใช้งานได้ทันทีโดยไม่ต้องมี dependency เพิ่มเติม
- จัดการการค้นหาด้วยคีย์เวิร์ดและเวกเตอร์ได้ดี
- รองรับผู้ให้บริการ embedding ทั้งหมด
- การค้นหาแบบไฮบริดรวมข้อดีที่สุดของวิธีดึงข้อมูลทั้งสองแบบ

พิจารณาเปลี่ยนไปใช้ [QMD](/th/concepts/memory-qmd) หากคุณต้องการ reranking, การขยายคิวรี
หรือต้องการทำดัชนีไดเรกทอรีนอก workspace

พิจารณา [Honcho](/th/concepts/memory-honcho) หากคุณต้องการหน่วยความจำข้ามเซสชันพร้อม
การสร้างโมเดลผู้ใช้โดยอัตโนมัติ

## การแก้ไขปัญหา

**การค้นหาหน่วยความจำถูกปิดใช้งานหรือไม่** ตรวจสอบ `openclaw memory status` หากตรวจไม่พบผู้ให้บริการ
ให้ตั้งค่าอย่างชัดเจนหรือเพิ่มคีย์ API

**ตรวจไม่พบผู้ให้บริการในเครื่องหรือไม่** ยืนยันว่าพาธในเครื่องมีอยู่จริง แล้วรัน:

```bash
openclaw memory status --deep --agent main
openclaw memory index --force --agent main
```

ทั้งคำสั่ง CLI แบบสแตนด์อโลนและ Gateway ใช้ ID ผู้ให้บริการ `local` เดียวกัน
หากผู้ให้บริการถูกตั้งเป็น `auto` ระบบจะพิจารณา embeddings ในเครื่องก่อน
ก็ต่อเมื่อ `memorySearch.local.modelPath` ชี้ไปยังไฟล์ในเครื่องที่มีอยู่จริง

**ผลลัพธ์เก่าหรือไม่** รัน `openclaw memory index --force` เพื่อสร้างใหม่ ตัวเฝ้าดู
อาจพลาดการเปลี่ยนแปลงได้ในกรณีพิเศษที่พบได้น้อย

**sqlite-vec ไม่โหลดหรือไม่** OpenClaw จะ fallback ไปใช้ cosine similarity ภายในโปรเซส
โดยอัตโนมัติ ตรวจสอบบันทึกเพื่อดูข้อผิดพลาดการโหลดที่เฉพาะเจาะจง

## การตั้งค่า

สำหรับการตั้งค่าผู้ให้บริการ embedding, การปรับแต่งการค้นหาแบบไฮบริด (น้ำหนัก, MMR, temporal
decay), การทำดัชนีแบบแบตช์, หน่วยความจำหลายรูปแบบ, sqlite-vec, พาธเพิ่มเติม และ
ตัวเลือกการตั้งค่าอื่น ๆ ทั้งหมด ดู
[ข้อมูลอ้างอิงการตั้งค่าหน่วยความจำ](/th/reference/memory-config)
