OpenClaw จัดระเบียบการสนทนาเป็น **เซสชัน** แต่ละข้อความจะถูกกำหนดเส้นทางไปยัง
เซสชันตามแหล่งที่มา เช่น DM, แชตกลุ่ม, งาน cron เป็นต้น

## วิธีการกำหนดเส้นทางข้อความ

| แหล่งที่มา       | พฤติกรรม                  |
| ---------------- | ------------------------- |
| ข้อความส่วนตัว   | ใช้เซสชันร่วมกันเป็นค่าเริ่มต้น |
| แชตกลุ่ม         | แยกต่อกลุ่ม               |
| ห้อง/ช่อง        | แยกต่อห้อง                |
| งาน Cron         | เซสชันใหม่ต่อการรันหนึ่งครั้ง |
| Webhook          | แยกต่อ hook               |

## การแยก DM

โดยค่าเริ่มต้น DM ทั้งหมดจะใช้เซสชันเดียวกันเพื่อความต่อเนื่อง ซึ่งเหมาะกับ
การตั้งค่าแบบผู้ใช้คนเดียว

<Warning>
หากมีหลายคนที่ส่งข้อความหาเอเจนต์ของคุณได้ ให้เปิดใช้การแยก DM มิฉะนั้นผู้ใช้ทุกคน
จะใช้บริบทการสนทนาเดียวกัน — ข้อความส่วนตัวของ Alice จะมองเห็นได้โดย Bob
</Warning>

**วิธีแก้:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // แยกตาม channel + ผู้ส่ง
  },
}
```

ตัวเลือกอื่น:

- `main` (ค่าเริ่มต้น) -- DM ทั้งหมดใช้เซสชันเดียวกัน
- `per-peer` -- แยกตามผู้ส่ง (ข้าม channel)
- `per-channel-peer` -- แยกตาม channel + ผู้ส่ง (แนะนำ)
- `per-account-channel-peer` -- แยกตามบัญชี + channel + ผู้ส่ง

<Tip>
หากคนเดียวกันติดต่อคุณจากหลาย channel ให้ใช้
`session.identityLinks` เพื่อเชื่อม identity ของบุคคลนั้นเข้าด้วยกันเพื่อให้ใช้เซสชันเดียวกัน
</Tip>

ตรวจสอบการตั้งค่าของคุณด้วย `openclaw security audit`

## วงจรชีวิตของเซสชัน

เซสชันจะถูกนำกลับมาใช้ซ้ำจนกว่าจะหมดอายุ:

- **รีเซ็ตรายวัน** (ค่าเริ่มต้น) -- เซสชันใหม่เวลา 4:00 น. ตามเวลาท้องถิ่นของโฮสต์
  gateway ความใหม่ล่าสุดรายวันอิงจากเวลาที่ `sessionId` ปัจจุบันเริ่มต้น ไม่ใช่
  การเขียน metadata ในภายหลัง
- **รีเซ็ตเมื่อว่างงาน** (เลือกได้) -- เซสชันใหม่หลังไม่มีการใช้งานช่วงหนึ่ง ตั้งค่า
  `session.reset.idleMinutes` ความใหม่ล่าสุดเมื่อว่างงานอิงจากการโต้ตอบจริงครั้งล่าสุดของ
  ผู้ใช้/channel ดังนั้น system event จาก heartbeat, cron และ exec จะไม่ทำให้เซสชันยังมีชีวิตอยู่
- **รีเซ็ตด้วยตนเอง** -- พิมพ์ `/new` หรือ `/reset` ในแชต `/new <model>` จะ
  สลับโมเดลด้วย

เมื่อมีการตั้งค่าทั้งรีเซ็ตรายวันและรีเซ็ตเมื่อว่างงาน เงื่อนไขที่หมดอายุก่อนจะมีผล
Heartbeat, cron, exec และเทิร์น system event อื่น ๆ อาจเขียน metadata ของเซสชัน
แต่การเขียนเหล่านั้นจะไม่ยืดความใหม่ล่าสุดของการรีเซ็ตรายวันหรือเมื่อว่างงาน เมื่อการรีเซ็ต
หมุนเซสชัน ข้อความแจ้งเตือน system event ที่เข้าคิวไว้สำหรับเซสชันเก่า
จะถูกทิ้ง เพื่อไม่ให้ข้อมูลอัปเดตเบื้องหลังที่ค้างเก่าถูกเติมไว้ต้นพรอมป์แรกใน
เซสชันใหม่

เซสชันที่มี CLI session ซึ่ง provider เป็นเจ้าของและยังใช้งานอยู่ จะไม่ถูกตัดด้วยค่าเริ่มต้นรายวันแบบปริยาย
ให้ใช้ `/reset` หรือกำหนด `session.reset` อย่างชัดเจนเมื่อคุณต้องการให้เซสชันเหล่านั้นหมดอายุตามเวลา

## สถานะถูกเก็บไว้ที่ไหน

สถานะทั้งหมดของเซสชันเป็นของ **gateway** ไคลเอนต์ UI จะสอบถามข้อมูลเซสชันจาก gateway

- **Store:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Transcript:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

`sessions.json` เก็บ timestamp ของวงจรชีวิตแยกกัน:

- `sessionStartedAt`: เวลาที่ `sessionId` ปัจจุบันเริ่มต้น; การรีเซ็ตรายวันใช้ค่านี้
- `lastInteractionAt`: การโต้ตอบผู้ใช้/channel ครั้งล่าสุดที่ยืดอายุเมื่อว่างงาน
- `updatedAt`: การเปลี่ยนแปลงแถวใน store ครั้งล่าสุด; มีประโยชน์สำหรับการแสดงรายการและการลบข้อมูลเก่า แต่ไม่ใช่
  แหล่งอ้างอิงสิทธิ์ขาดสำหรับความใหม่ล่าสุดของการรีเซ็ตรายวัน/เมื่อว่างงาน

แถวเก่าที่ไม่มี `sessionStartedAt` จะ resolve จากส่วนหัว session ใน transcript JSONL
เมื่อมีให้ใช้ หากแถวเก่านั้นไม่มี `lastInteractionAt` ด้วย
ความใหม่ล่าสุดเมื่อว่างงานจะ fallback ไปใช้เวลาเริ่มต้นเซสชันนั้น ไม่ใช่ใช้การเขียนข้อมูลประกอบในภายหลัง

## การดูแลเซสชัน

OpenClaw จะจำกัดขนาดการจัดเก็บเซสชันโดยอัตโนมัติตามเวลา โดยค่าเริ่มต้นจะทำงาน
ในโหมด `warn` (รายงานว่าจะล้างอะไรบ้าง) ตั้งค่า `session.maintenance.mode`
เป็น `"enforce"` เพื่อให้ล้างอัตโนมัติ:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

ดูตัวอย่างล่วงหน้าด้วย `openclaw sessions cleanup --dry-run`

## การตรวจสอบเซสชัน

- `openclaw status` -- path ของ session store และกิจกรรมล่าสุด
- `openclaw sessions --json` -- เซสชันทั้งหมด (กรองด้วย `--active <minutes>`)
- `/status` ในแชต -- การใช้บริบท โมเดล และตัวสลับต่าง ๆ
- `/context list` -- สิ่งที่อยู่ใน system prompt
