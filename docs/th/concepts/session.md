OpenClaw จัดระเบียบการสนทนาเป็น **เซสชัน** แต่ละข้อความจะถูกส่งไปยัง
เซสชันตามที่มาของข้อความนั้น -- DM, แชทกลุ่ม, งาน cron เป็นต้น

## วิธีการกำหนดเส้นทางข้อความ

| แหล่งที่มา | พฤติกรรม |
| --------------- | ------------------------- |
| ข้อความส่วนตัว | ใช้เซสชันร่วมกันตามค่าเริ่มต้น |
| แชทกลุ่ม | แยกตามแต่ละกลุ่ม |
| ห้อง/ช่อง | แยกตามแต่ละห้อง |
| งาน cron | เซสชันใหม่ต่อการรันแต่ละครั้ง |
| Webhook | แยกตามแต่ละ hook |

## การแยก DM

ตามค่าเริ่มต้น DM ทั้งหมดจะใช้เซสชันเดียวร่วมกันเพื่อความต่อเนื่อง ซึ่งเหมาะสำหรับ
การตั้งค่าแบบผู้ใช้คนเดียว

<Warning>
หากหลายคนสามารถส่งข้อความถึงเอเจนต์ของคุณได้ ให้เปิดใช้การแยก DM หากไม่เปิดใช้
ผู้ใช้ทั้งหมดจะใช้บริบทการสนทนาเดียวกัน -- ข้อความส่วนตัวของ Alice จะมองเห็นได้โดย
Bob
</Warning>

**วิธีแก้ไข:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // isolate by channel + sender
  },
}
```

ตัวเลือกอื่น:

- `main` (ค่าเริ่มต้น) -- DM ทั้งหมดใช้เซสชันเดียวร่วมกัน
- `per-peer` -- แยกตามผู้ส่ง (ข้ามช่องทาง)
- `per-channel-peer` -- แยกตามช่องทาง + ผู้ส่ง (แนะนำ)
- `per-account-channel-peer` -- แยกตามบัญชี + ช่องทาง + ผู้ส่ง

<Tip>
หากคนเดียวกันติดต่อคุณจากหลายช่องทาง ให้ใช้
`session.identityLinks` เพื่อเชื่อมโยงตัวตนของเขา เพื่อให้ใช้เซสชันเดียวกัน
</Tip>

### Dock ช่องทางที่เชื่อมโยง

คำสั่ง Dock ช่วยให้ผู้ใช้ย้ายเส้นทางการตอบกลับของเซสชันแชทส่วนตัวปัจจุบันไปยัง
ช่องทางที่เชื่อมโยงอีกช่องหนึ่งได้โดยไม่ต้องเริ่มเซสชันใหม่ ดู
[การ dock ช่องทาง](/th/concepts/channel-docking) สำหรับตัวอย่าง การกำหนดค่า และ
การแก้ไขปัญหา

ตรวจสอบการตั้งค่าของคุณด้วย `openclaw security audit`

## วงจรชีวิตของเซสชัน

เซสชันจะถูกนำกลับมาใช้ซ้ำจนกว่าจะหมดอายุ:

- **รีเซ็ตรายวัน** (ค่าเริ่มต้น) -- เซสชันใหม่เวลา 4:00 น. ตามเวลาท้องถิ่นบนโฮสต์
  Gateway ความใหม่รายวันอิงตามเวลาที่ `sessionId` ปัจจุบันเริ่มต้น ไม่ใช่
  การเขียน metadata ในภายหลัง
- **รีเซ็ตเมื่อไม่ได้ใช้งาน** (ไม่บังคับ) -- เซสชันใหม่หลังจากไม่มีความเคลื่อนไหวเป็นระยะเวลาหนึ่ง ตั้งค่า
  `session.reset.idleMinutes` ความใหม่จากการไม่ได้ใช้งานอิงตามปฏิสัมพันธ์จริงล่าสุดของ
  ผู้ใช้/ช่องทาง ดังนั้นเหตุการณ์ระบบ heartbeat, cron และ exec จะไม่
  ทำให้เซสชันยังคงอยู่
- **รีเซ็ตด้วยตนเอง** -- พิมพ์ `/new` หรือ `/reset` ในแชท `/new <model>` ยัง
  เปลี่ยนโมเดลด้วย

เมื่อกำหนดค่าทั้งการรีเซ็ตรายวันและการรีเซ็ตเมื่อไม่ได้ใช้งาน ตัวใดหมดอายุก่อนจะมีผลก่อน
เทิร์นของเหตุการณ์ระบบ เช่น heartbeat, cron, exec และอื่นๆ อาจเขียน metadata ของเซสชัน
แต่การเขียนเหล่านั้นจะไม่ยืดความใหม่ของการรีเซ็ตรายวันหรือการรีเซ็ตเมื่อไม่ได้ใช้งาน เมื่อการรีเซ็ต
เปลี่ยนเซสชัน การแจ้งเตือนเหตุการณ์ระบบที่ค้างคิวสำหรับเซสชันเก่าจะถูก
ทิ้ง เพื่อไม่ให้อัปเดตเบื้องหลังที่ล้าสมัยถูกนำหน้า prompt แรกใน
เซสชันใหม่

เซสชันที่มีเซสชัน CLI ที่ provider เป็นเจ้าของและยังทำงานอยู่จะไม่ถูกตัดด้วยค่าเริ่มต้นรายวัน
โดยปริยาย ใช้ `/reset` หรือกำหนดค่า `session.reset` อย่างชัดเจนเมื่อเซสชันเหล่านั้น
ควรหมดอายุตามตัวจับเวลา

## ตำแหน่งที่เก็บสถานะ

สถานะเซสชันทั้งหมดเป็นของ **Gateway** ไคลเอนต์ UI จะสอบถาม Gateway เพื่อรับ
ข้อมูลเซสชัน

- **ที่เก็บ:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **ทรานสคริปต์:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

`sessions.json` เก็บ timestamp ของวงจรชีวิตแยกกัน:

- `sessionStartedAt`: เวลาที่ `sessionId` ปัจจุบันเริ่มต้น การรีเซ็ตรายวันใช้ค่านี้
- `lastInteractionAt`: ปฏิสัมพันธ์ล่าสุดของผู้ใช้/ช่องทางที่ยืดอายุเมื่อไม่ได้ใช้งาน
- `updatedAt`: การเปลี่ยนแปลงแถวใน store ล่าสุด มีประโยชน์สำหรับการแสดงรายการและการตัดทิ้ง แต่ไม่ใช่
  แหล่งอ้างอิงสำหรับความใหม่ของการรีเซ็ตรายวัน/เมื่อไม่ได้ใช้งาน

แถวเก่าที่ไม่มี `sessionStartedAt` จะถูก resolve จากส่วนหัวเซสชันในทรานสคริปต์ JSONL
เมื่อมีให้ใช้ หากแถวเก่ายังไม่มี `lastInteractionAt` ด้วย
ความใหม่เมื่อไม่ได้ใช้งานจะ fallback ไปยังเวลาเริ่มต้นเซสชันนั้น ไม่ใช่ไปยังการเขียน bookkeeping
ในภายหลัง

## การบำรุงรักษาเซสชัน

OpenClaw จำกัดพื้นที่จัดเก็บเซสชันโดยอัตโนมัติตามเวลา ตามค่าเริ่มต้นจะทำงาน
ในโหมด `warn` (รายงานสิ่งที่จะถูกล้าง) ตั้งค่า `session.maintenance.mode`
เป็น `"enforce"` เพื่อให้ล้างข้อมูลโดยอัตโนมัติ:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

สำหรับขีดจำกัด `maxEntries` ขนาด production การเขียนของ runtime Gateway จะใช้บัฟเฟอร์ high-water ขนาดเล็กและล้างกลับลงมาถึงเพดานที่กำหนดค่าไว้เป็นชุด การอ่าน session store จะไม่ prune หรือจำกัดรายการระหว่างการเริ่มต้น Gateway วิธีนี้หลีกเลี่ยงการล้าง store ทั้งหมดทุกครั้งที่เริ่มต้นหรือในเซสชัน cron ที่แยกไว้ `openclaw sessions cleanup --enforce` ใช้เพดานทันที

การบำรุงรักษาจะรักษาตัวชี้การสนทนาภายนอกแบบถาวร รวมถึงเซสชันกลุ่ม
และเซสชันแชทที่ผูกกับ thread ในขณะที่ยังอนุญาตให้รายการ cron สังเคราะห์,
hook, heartbeat, ACP และ sub-agent หมดอายุได้

ดูตัวอย่างด้วย `openclaw sessions cleanup --dry-run`

## การตรวจสอบเซสชัน

- `openclaw status` -- path ของ session store และกิจกรรมล่าสุด
- `openclaw sessions --json` -- เซสชันทั้งหมด (กรองด้วย `--active <minutes>`)
- `/status` ในแชท -- การใช้บริบท โมเดล และ toggle
- `/context list` -- สิ่งที่อยู่ใน system prompt
