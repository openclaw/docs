OpenClaw จัดระเบียบการสนทนาเป็น **เซสชัน** แต่ละข้อความจะถูกกำหนดเส้นทางไปยัง
เซสชันตามแหล่งที่มา เช่น DM, แชตกลุ่ม, งาน Cron เป็นต้น

## วิธีการกำหนดเส้นทางข้อความ

| แหล่งที่มา       | พฤติกรรม                         |
| --------------- | ------------------------- |
| ข้อความโดยตรง | ใช้เซสชันร่วมกันตามค่าเริ่มต้น |
| แชตกลุ่ม     | แยกตามแต่ละกลุ่ม        |
| ห้อง/ช่อง  | แยกตามแต่ละห้อง         |
| งาน Cron       | เซสชันใหม่ต่อการรันแต่ละครั้ง     |
| Webhook        | แยกตามแต่ละ hook         |

## การแยก DM

ตามค่าเริ่มต้น DM ทั้งหมดจะใช้เซสชันเดียวร่วมกันเพื่อความต่อเนื่อง วิธีนี้เหมาะสำหรับ
การตั้งค่าที่มีผู้ใช้คนเดียว

<Warning>
หากมีหลายคนที่สามารถส่งข้อความถึงเอเจนต์ของคุณได้ ให้เปิดใช้การแยก DM หากไม่เปิดใช้
ผู้ใช้ทั้งหมดจะใช้บริบทการสนทนาเดียวกัน ข้อความส่วนตัวของ Alice จะมองเห็นได้สำหรับ Bob
</Warning>

**วิธีแก้:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // isolate by channel + sender
  },
}
```

ตัวเลือกอื่น:

- `main` (ค่าเริ่มต้น) -- DM ทั้งหมดใช้เซสชันเดียวร่วมกัน
- `per-peer` -- แยกตามผู้ส่ง (ข้ามช่อง)
- `per-channel-peer` -- แยกตามช่อง + ผู้ส่ง (แนะนำ)
- `per-account-channel-peer` -- แยกตามบัญชี + ช่อง + ผู้ส่ง

<Tip>
หากบุคคลเดียวกันติดต่อคุณจากหลายช่อง ให้ใช้
`session.identityLinks` เพื่อเชื่อมโยงตัวตนของพวกเขาเพื่อให้ใช้เซสชันเดียวกัน
</Tip>

### เชื่อมต่อช่องที่ลิงก์ไว้

คำสั่ง Dock ช่วยให้ผู้ใช้ย้ายเส้นทางการตอบกลับของเซสชันแชตโดยตรงปัจจุบันไปยัง
ช่องอื่นที่ลิงก์ไว้ได้โดยไม่ต้องเริ่มเซสชันใหม่ ดู
[การ dock ช่อง](/th/concepts/channel-docking) สำหรับตัวอย่าง การกำหนดค่า และ
การแก้ไขปัญหา

ตรวจสอบการตั้งค่าของคุณด้วย `openclaw security audit`

## วงจรชีวิตของเซสชัน

เซสชันจะถูกใช้ซ้ำจนกว่าจะหมดอายุ:

- **รีเซ็ตรายวัน** (ค่าเริ่มต้น) -- เซสชันใหม่เวลา 4:00 น. ตามเวลาท้องถิ่นบนโฮสต์
  Gateway ความสดใหม่รายวันอิงตามเวลาที่ `sessionId` ปัจจุบันเริ่มต้น ไม่ใช่
  การเขียนเมทาดาทาในภายหลัง
- **รีเซ็ตเมื่อไม่ได้ใช้งาน** (ไม่บังคับ) -- เซสชันใหม่หลังจากไม่มีการใช้งานช่วงหนึ่ง ตั้งค่า
  `session.reset.idleMinutes` ความสดใหม่เมื่อไม่ได้ใช้งานอิงตามการโต้ตอบจริงล่าสุดของ
  ผู้ใช้/ช่อง ดังนั้นเหตุการณ์ระบบ Heartbeat, Cron และ exec จะไม่
  ทำให้เซสชันยังคงอยู่
- **รีเซ็ตด้วยตนเอง** -- พิมพ์ `/new` หรือ `/reset` ในแชต `/new <model>` ยัง
  เปลี่ยนโมเดลด้วย

เมื่อกำหนดค่าทั้งรีเซ็ตรายวันและรีเซ็ตเมื่อไม่ได้ใช้งานไว้ ตัวใดหมดอายุก่อนจะมีผลก่อน
เทิร์นของเหตุการณ์ระบบอย่าง Heartbeat, Cron, exec และอื่นๆ อาจเขียนเมทาดาทาเซสชัน
แต่การเขียนเหล่านั้นจะไม่ขยายความสดใหม่ของการรีเซ็ตรายวันหรือเมื่อไม่ได้ใช้งาน เมื่อการรีเซ็ต
หมุนเซสชัน การแจ้งเตือนเหตุการณ์ระบบที่อยู่ในคิวสำหรับเซสชันเก่าจะถูก
ทิ้ง เพื่อไม่ให้อัปเดตเบื้องหลังที่ล้าสมัยถูกเติมไว้หน้าพรอมป์แรกใน
เซสชันใหม่

เซสชันที่มีเซสชัน CLI ที่ผู้ให้บริการเป็นเจ้าของและยังทำงานอยู่จะไม่ถูกตัดโดยค่าเริ่มต้นรายวันแบบแฝง
ใช้ `/reset` หรือกำหนดค่า `session.reset` อย่างชัดเจนเมื่อเซสชันเหล่านั้น
ควรหมดอายุตามตัวจับเวลา

## สถานะอยู่ที่ไหน

สถานะเซสชันทั้งหมดเป็นของ **Gateway** ไคลเอนต์ UI จะสอบถาม Gateway เพื่อ
ข้อมูลเซสชัน

- **ที่จัดเก็บ:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **ทรานสคริปต์:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

`sessions.json` เก็บ timestamp วงจรชีวิตแยกกัน:

- `sessionStartedAt`: เวลาที่ `sessionId` ปัจจุบันเริ่มต้น การรีเซ็ตรายวันใช้ค่านี้
- `lastInteractionAt`: การโต้ตอบของผู้ใช้/ช่องล่าสุดที่ขยายอายุเมื่อไม่ได้ใช้งาน
- `updatedAt`: การเปลี่ยนแปลงแถวในที่จัดเก็บล่าสุด มีประโยชน์สำหรับการแสดงรายการและการตัดออก แต่ไม่ใช่
  แหล่งอ้างอิงหลักสำหรับความสดใหม่ของการรีเซ็ตรายวัน/เมื่อไม่ได้ใช้งาน

แถวเก่าที่ไม่มี `sessionStartedAt` จะถูกแก้จาก header เซสชัน JSONL ของทรานสคริปต์
เมื่อมี หากแถวเก่ายังไม่มี `lastInteractionAt` ด้วย
ความสดใหม่เมื่อไม่ได้ใช้งานจะย้อนกลับไปใช้เวลาเริ่มเซสชันนั้น ไม่ใช่การเขียนเพื่อทำบัญชี
ในภายหลัง

## การบำรุงรักษาเซสชัน

OpenClaw จำกัดขนาดที่จัดเก็บเซสชันโดยอัตโนมัติตามเวลา ตามค่าเริ่มต้น ระบบจะทำงาน
ในโหมด `warn` (รายงานสิ่งที่จะถูกล้าง) ตั้งค่า `session.maintenance.mode`
เป็น `"enforce"` สำหรับการล้างอัตโนมัติ:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

สำหรับขีดจำกัด `maxEntries` ขนาด production การเขียนของรันไทม์ Gateway จะใช้บัฟเฟอร์ high-water ขนาดเล็กและล้างกลับลงมาถึงเพดานที่กำหนดค่าไว้เป็นชุด วิธีนี้หลีกเลี่ยงการรันการล้างที่จัดเก็บทั้งหมดในทุกเซสชัน Cron ที่แยกไว้ `openclaw sessions cleanup --enforce` จะใช้เพดานทันที

ดูตัวอย่างด้วย `openclaw sessions cleanup --dry-run`

## การตรวจสอบเซสชัน

- `openclaw status` -- พาธที่จัดเก็บเซสชันและกิจกรรมล่าสุด
- `openclaw sessions --json` -- เซสชันทั้งหมด (กรองด้วย `--active <minutes>`)
- `/status` ในแชต -- การใช้บริบท โมเดล และ toggle
- `/context list` -- สิ่งที่อยู่ในพรอมป์ระบบ
