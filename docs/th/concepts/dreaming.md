Dreaming คือระบบรวมความจำในเบื้องหลังของ `memory-core` ระบบนี้ช่วยให้ OpenClaw ย้ายสัญญาณระยะสั้นที่ชัดเจนไปเป็นหน่วยความจำถาวร พร้อมทำให้กระบวนการอธิบายและตรวจสอบได้

<Note>
Dreaming เป็นแบบ **opt-in** และปิดใช้งานโดยค่าเริ่มต้น
</Note>

## สิ่งที่ dreaming เขียน

Dreaming เก็บผลลัพธ์สองชนิด:

- **สถานะเครื่อง** ใน `memory/.dreams/` (ที่เก็บการเรียกคืน, สัญญาณเฟส, เช็กพอยต์การนำเข้า, ล็อก)
- **ผลลัพธ์ที่มนุษย์อ่านได้** ใน `DREAMS.md` (หรือ `dreams.md` ที่มีอยู่) และไฟล์รายงานเฟสที่เลือกใช้ได้ภายใต้ `memory/dreaming/<phase>/YYYY-MM-DD.md`

การเลื่อนระดับระยะยาวยังคงเขียนเฉพาะไปยัง `MEMORY.md`

## โมเดลเฟส

Dreaming ใช้เฟสที่ทำงานร่วมกันสามเฟส:

| เฟส | วัตถุประสงค์                                   | การเขียนแบบถาวร     |
| ----- | ----------------------------------------- | ----------------- |
| Light | จัดเรียงและเตรียมเนื้อหาระยะสั้นล่าสุด | ไม่                |
| Deep  | ให้คะแนนและเลื่อนระดับตัวเลือกที่คงทน      | ใช่ (`MEMORY.md`) |
| REM   | ทบทวนธีมและแนวคิดที่เกิดซ้ำ     | ไม่                |

เฟสเหล่านี้เป็นรายละเอียดการทำงานภายใน ไม่ใช่ "โหมด" แยกต่างหากที่ผู้ใช้กำหนดค่าได้

<AccordionGroup>
  <Accordion title="เฟส Light">
    เฟส Light นำเข้าสัญญาณหน่วยความจำรายวันล่าสุดและร่องรอยการเรียกคืน ลบรายการซ้ำ และเตรียมบรรทัดตัวเลือก

    - อ่านจากสถานะการเรียกคืนระยะสั้น, ไฟล์หน่วยความจำรายวันล่าสุด, และทรานสคริปต์เซสชันที่ปกปิดข้อมูลแล้วเมื่อมี
    - เขียนบล็อก `## Light Sleep` ที่จัดการโดยระบบเมื่อที่เก็บข้อมูลมีเอาต์พุตแบบอินไลน์
    - บันทึกสัญญาณเสริมแรงสำหรับการจัดอันดับแบบ deep ในภายหลัง
    - ไม่เขียนไปยัง `MEMORY.md`

  </Accordion>
  <Accordion title="เฟส Deep">
    เฟส Deep ตัดสินใจว่าอะไรจะกลายเป็นหน่วยความจำระยะยาว

    - จัดอันดับตัวเลือกโดยใช้การให้คะแนนแบบถ่วงน้ำหนักและเกณฑ์ผ่าน
    - ต้องผ่าน `minScore`, `minRecallCount`, และ `minUniqueQueries`
    - เติม snippets กลับจากไฟล์รายวันสดก่อนเขียน เพื่อข้าม snippets ที่ล้าสมัยหรือถูกลบไปแล้ว
    - เพิ่มรายการที่เลื่อนระดับแล้วไปยัง `MEMORY.md`
    - เขียนสรุป `## Deep Sleep` ลงใน `DREAMS.md` และเลือกเขียน `memory/dreaming/deep/YYYY-MM-DD.md` ได้

  </Accordion>
  <Accordion title="เฟส REM">
    เฟส REM ดึงรูปแบบและสัญญาณเชิงทบทวนออกมา

    - สร้างสรุปธีมและการทบทวนจากร่องรอยระยะสั้นล่าสุด
    - เขียนบล็อก `## REM Sleep` ที่จัดการโดยระบบเมื่อที่เก็บข้อมูลมีเอาต์พุตแบบอินไลน์
    - บันทึกสัญญาณเสริมแรงของ REM ที่ใช้โดยการจัดอันดับแบบ deep
    - ไม่เขียนไปยัง `MEMORY.md`

  </Accordion>
</AccordionGroup>

## การนำเข้าทรานสคริปต์เซสชัน

Dreaming สามารถนำเข้าทรานสคริปต์เซสชันที่ปกปิดข้อมูลแล้วเข้าไปในคลังข้อมูล dreaming ได้ เมื่อมีทรานสคริปต์ ข้อมูลเหล่านี้จะถูกป้อนเข้าเฟส Light ควบคู่กับสัญญาณหน่วยความจำรายวันและร่องรอยการเรียกคืน เนื้อหาส่วนบุคคลและละเอียดอ่อนจะถูกปกปิดก่อนนำเข้า

## Dream Diary

Dreaming ยังเก็บ **Dream Diary** แบบเล่าเรื่องไว้ใน `DREAMS.md` ด้วย หลังจากแต่ละเฟสมีเนื้อหาเพียงพอ `memory-core` จะเรียกเทิร์น subagent เบื้องหลังแบบพยายามอย่างดีที่สุด แล้วเพิ่มบันทึกไดอารีสั้นๆ ต่อท้าย โดยใช้โมเดลรันไทม์เริ่มต้น เว้นแต่จะกำหนดค่า `dreaming.model` หากโมเดลที่กำหนดค่าไว้ไม่พร้อมใช้งาน Dream Diary จะลองใหม่หนึ่งครั้งด้วยโมเดลเริ่มต้นของเซสชัน

<Note>
ไดอารีนี้มีไว้ให้มนุษย์อ่านใน UI Dreams ไม่ใช่แหล่งเลื่อนระดับ อาร์ติแฟกต์ไดอารี/รายงานที่สร้างโดย Dreaming จะถูกแยกออกจากการเลื่อนระดับระยะสั้น เฉพาะ snippets หน่วยความจำที่มีหลักฐานรองรับเท่านั้นที่มีสิทธิ์เลื่อนระดับเข้า `MEMORY.md`
</Note>

ยังมีเลน backfill ประวัติที่มีหลักฐานรองรับสำหรับงานตรวจสอบและกู้คืนด้วย:

<AccordionGroup>
  <Accordion title="คำสั่ง Backfill">
    - `memory rem-harness --path ... --grounded` แสดงตัวอย่างเอาต์พุตไดอารีที่มีหลักฐานรองรับจากโน้ต `YYYY-MM-DD.md` ในอดีต
    - `memory rem-backfill --path ...` เขียนรายการไดอารีที่มีหลักฐานรองรับและย้อนกลับได้ลงใน `DREAMS.md`
    - `memory rem-backfill --path ... --stage-short-term` เตรียมตัวเลือกระยะยาวที่มีหลักฐานรองรับเข้าในที่เก็บหลักฐานระยะสั้นเดียวกับที่เฟส Deep ปกติใช้อยู่แล้ว
    - `memory rem-backfill --rollback` และ `--rollback-short-term` ลบอาร์ติแฟกต์ backfill ที่เตรียมไว้เหล่านั้นโดยไม่แตะต้องรายการไดอารีปกติหรือการเรียกคืนระยะสั้นสด

  </Accordion>
</AccordionGroup>

Control UI เปิดเผย flow backfill/reset ไดอารีเดียวกัน เพื่อให้คุณตรวจสอบผลลัพธ์ในฉาก Dreams ก่อนตัดสินใจว่าตัวเลือกที่มีหลักฐานรองรับสมควรได้รับการเลื่อนระดับหรือไม่ Scene ยังแสดงเลนที่มีหลักฐานรองรับแยกต่างหาก เพื่อให้คุณเห็นว่ารายการระยะสั้นที่เตรียมไว้ใดมาจากการ replay ประวัติ รายการที่เลื่อนระดับใดนำโดยหลักฐานรองรับ และล้างเฉพาะรายการที่เตรียมไว้แบบมีหลักฐานรองรับเท่านั้นโดยไม่แตะต้องสถานะระยะสั้นสดปกติ

## สัญญาณการจัดอันดับแบบ Deep

การจัดอันดับแบบ Deep ใช้สัญญาณฐานแบบถ่วงน้ำหนักหกตัวและการเสริมแรงจากเฟส:

| สัญญาณ              | น้ำหนัก | คำอธิบาย                                       |
| ------------------- | ------ | ------------------------------------------------- |
| ความถี่           | 0.24   | จำนวนสัญญาณระยะสั้นที่รายการสะสมไว้ |
| ความเกี่ยวข้อง           | 0.30   | คุณภาพการดึงข้อมูลเฉลี่ยของรายการ           |
| ความหลากหลายของคำค้น     | 0.15   | บริบทคำค้น/วันแยกกันที่ทำให้รายการปรากฏ      |
| ความใหม่             | 0.15   | คะแนนความสดใหม่ที่ลดค่าตามเวลา                      |
| การรวมความจำ       | 0.10   | ความแข็งแรงของการเกิดซ้ำหลายวัน                     |
| ความมั่งคั่งเชิงแนวคิด | 0.06   | ความหนาแน่นของแท็กแนวคิดจาก snippet/path             |

การพบในเฟส Light และ REM เพิ่มบูสต์เล็กน้อยที่ลดค่าตามความใหม่จาก `memory/.dreams/phase-signals.json`

## การตั้งเวลา

เมื่อเปิดใช้งาน `memory-core` จะจัดการ cron job หนึ่งรายการโดยอัตโนมัติสำหรับการ sweep Dreaming แบบเต็ม แต่ละ sweep จะรันเฟสตามลำดับ: light → REM → deep

พฤติกรรม cadence ค่าเริ่มต้น:

| การตั้งค่า              | ค่าเริ่มต้น       |
| -------------------- | ------------- |
| `dreaming.frequency` | `0 3 * * *`   |
| `dreaming.model`     | โมเดลเริ่มต้น |

## ค่าเริ่มต้นหลัก

การตั้งค่าทั้งหมดอยู่ภายใต้ `plugins.entries.memory-core.config.dreaming`

<ParamField path="enabled" type="boolean" default="false">
  เปิดหรือปิดใช้งาน dreaming sweep
</ParamField>
<ParamField path="frequency" type="string" default="0 3 * * *">
  cadence ของ Cron สำหรับ dreaming sweep แบบเต็ม
</ParamField>
<ParamField path="model" type="string">
  การแทนที่โมเดล subagent ของ Dream Diary แบบเลือกได้ ใช้ค่า `provider/model` แบบ canonical เมื่อกำหนด allowlist ของ subagent `allowedModels` ด้วย
</ParamField>

<Warning>
`dreaming.model` ต้องใช้ `plugins.entries.memory-core.subagent.allowModelOverride: true` หากต้องการจำกัด ให้ตั้งค่า `plugins.entries.memory-core.subagent.allowedModels` ด้วย ความล้มเหลวด้านความเชื่อถือหรือ allowlist จะยังแสดงให้เห็นแทนที่จะ fallback แบบเงียบๆ การลองใหม่ครอบคลุมเฉพาะข้อผิดพลาดโมเดลไม่พร้อมใช้งานเท่านั้น
</Warning>

<Note>
นโยบายเฟส, เกณฑ์, และพฤติกรรมที่เก็บข้อมูลเป็นรายละเอียดการทำงานภายใน (ไม่ใช่ config ที่แสดงต่อผู้ใช้) ดู [ข้อมูลอ้างอิงการกำหนดค่าหน่วยความจำ](/th/reference/memory-config#dreaming) สำหรับรายการคีย์ทั้งหมด
</Note>

## UI Dreams

เมื่อเปิดใช้งาน แท็บ **Dreams** ของ Gateway จะแสดง:

- สถานะการเปิดใช้งาน dreaming ปัจจุบัน
- สถานะระดับเฟสและการมีอยู่ของ managed-sweep
- จำนวนรายการระยะสั้น, ที่มีหลักฐานรองรับ, สัญญาณ, และที่เลื่อนระดับวันนี้
- เวลารันตามกำหนดการครั้งถัดไป
- เลน Scene ที่มีหลักฐานรองรับแยกต่างหากสำหรับรายการ replay ประวัติที่เตรียมไว้
- ตัวอ่าน Dream Diary แบบขยายได้ซึ่งรองรับโดย `doctor.memory.dreamDiary`
