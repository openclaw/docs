Dreaming คือระบบรวมหน่วยความจำเบื้องหลังใน `memory-core` ระบบนี้ช่วยให้ OpenClaw ย้ายสัญญาณระยะสั้นที่มีน้ำหนักมากไปยังหน่วยความจำถาวร พร้อมทำให้กระบวนการอธิบายและตรวจทานได้

<Note>
Dreaming เป็นแบบ **เลือกเปิดใช้** และปิดอยู่โดยค่าเริ่มต้น
</Note>

## สิ่งที่ dreaming เขียน

Dreaming เก็บผลลัพธ์สองชนิด:

- **สถานะของเครื่อง** ใน `memory/.dreams/` (ที่เก็บการเรียกคืน, สัญญาณเฟส, จุดตรวจการนำเข้า, ล็อก)
- **ผลลัพธ์ที่มนุษย์อ่านได้** ใน `DREAMS.md` (หรือ `dreams.md` ที่มีอยู่) และไฟล์รายงานเฟสแบบไม่บังคับภายใต้ `memory/dreaming/<phase>/YYYY-MM-DD.md`

การโปรโมตระยะยาวยังคงเขียนเฉพาะไปยัง `MEMORY.md`

## โมเดลเฟส

Dreaming ใช้เฟสที่ทำงานร่วมกันสามเฟส:

| เฟส | วัตถุประสงค์                                   | การเขียนถาวร     |
| ----- | ----------------------------------------- | ----------------- |
| Light | จัดเรียงและเตรียมเนื้อหาระยะสั้นล่าสุด | ไม่                |
| Deep  | ให้คะแนนและโปรโมตตัวเลือกที่คงทน      | ใช่ (`MEMORY.md`) |
| REM   | ทบทวนธีมและแนวคิดที่เกิดซ้ำ     | ไม่                |

เฟสเหล่านี้เป็นรายละเอียดการใช้งานภายใน ไม่ใช่ "โหมด" แยกต่างหากที่ผู้ใช้กำหนดค่า

<AccordionGroup>
  <Accordion title="เฟส Light">
    เฟส Light นำเข้าสัญญาณหน่วยความจำรายวันล่าสุดและร่องรอยการเรียกคืน ลบรายการซ้ำ และเตรียมบรรทัดตัวเลือก

    - อ่านจากสถานะการเรียกคืนระยะสั้น ไฟล์หน่วยความจำรายวันล่าสุด และบันทึกการถอดความเซสชันที่ลบข้อมูลอ่อนไหวแล้วเมื่อมีให้ใช้
    - เขียนบล็อก `## Light Sleep` ที่จัดการให้ เมื่อพื้นที่เก็บข้อมูลรวมผลลัพธ์แบบอินไลน์
    - บันทึกสัญญาณเสริมแรงสำหรับการจัดอันดับ deep ในภายหลัง
    - ไม่เขียนไปยัง `MEMORY.md`

  </Accordion>
  <Accordion title="เฟส Deep">
    เฟส Deep ตัดสินใจว่าอะไรจะกลายเป็นหน่วยความจำระยะยาว

    - จัดอันดับตัวเลือกด้วยการให้คะแนนแบบถ่วงน้ำหนักและเกตเกณฑ์ขั้นต่ำ
    - ต้องผ่าน `minScore`, `minRecallCount` และ `minUniqueQueries`
    - เติมข้อมูล snippet จากไฟล์รายวันสดใหม่ก่อนเขียน ดังนั้น snippet ที่เก่าหรือถูกลบจะถูกข้าม
    - ต่อท้ายรายการที่โปรโมตแล้วลงใน `MEMORY.md`
    - เขียนสรุป `## Deep Sleep` ลงใน `DREAMS.md` และอาจเขียน `memory/dreaming/deep/YYYY-MM-DD.md`

  </Accordion>
  <Accordion title="เฟส REM">
    เฟส REM ดึงรูปแบบและสัญญาณเชิงสะท้อนออกมา

    - สร้างสรุปธีมและการสะท้อนจากร่องรอยระยะสั้นล่าสุด
    - เขียนบล็อก `## REM Sleep` ที่จัดการให้ เมื่อพื้นที่เก็บข้อมูลรวมผลลัพธ์แบบอินไลน์
    - บันทึกสัญญาณเสริมแรง REM ที่ใช้โดยการจัดอันดับ deep
    - ไม่เขียนไปยัง `MEMORY.md`

  </Accordion>
</AccordionGroup>

## การนำเข้าบันทึกการถอดความเซสชัน

Dreaming สามารถนำเข้าบันทึกการถอดความเซสชันที่ลบข้อมูลอ่อนไหวแล้วเข้าสู่คลังข้อมูล dreaming ได้ เมื่อมีบันทึกการถอดความ ระบบจะป้อนข้อมูลเหล่านั้นเข้าเฟส light ควบคู่กับสัญญาณหน่วยความจำรายวันและร่องรอยการเรียกคืน เนื้อหาส่วนบุคคลและเนื้อหาอ่อนไหวจะถูกลบข้อมูลก่อนนำเข้า

## Dream Diary

Dreaming ยังเก็บ **Dream Diary** แบบเล่าเรื่องไว้ใน `DREAMS.md` ด้วย หลังจากแต่ละเฟสมีเนื้อหาเพียงพอ `memory-core` จะรันเทิร์น subagent เบื้องหลังแบบพยายามเต็มที่และต่อท้ายรายการไดอารีสั้น ๆ โดยใช้โมเดล runtime ค่าเริ่มต้น เว้นแต่จะกำหนดค่า `dreaming.model` หากโมเดลที่กำหนดค่าไว้ไม่พร้อมใช้งาน Dream Diary จะลองอีกครั้งหนึ่งด้วยโมเดลค่าเริ่มต้นของเซสชัน

<Note>
ไดอารีนี้มีไว้ให้มนุษย์อ่านใน Dreams UI ไม่ใช่แหล่งสำหรับการโปรโมต อาร์ติแฟกต์ไดอารี/รายงานที่ Dreaming สร้างขึ้นจะถูกยกเว้นจากการโปรโมตระยะสั้น เฉพาะ snippet หน่วยความจำที่มีหลักฐานรองรับเท่านั้นที่มีสิทธิ์โปรโมตเข้า `MEMORY.md`
</Note>

ยังมีเลนเติมข้อมูลย้อนหลังที่มีหลักฐานรองรับสำหรับงานตรวจทานและกู้คืน:

<AccordionGroup>
  <Accordion title="คำสั่งเติมข้อมูลย้อนหลัง">
    - `memory rem-harness --path ... --grounded` แสดงตัวอย่างผลลัพธ์ไดอารีที่มีหลักฐานรองรับจากโน้ต `YYYY-MM-DD.md` ในอดีต
    - `memory rem-backfill --path ...` เขียนรายการไดอารีที่มีหลักฐานรองรับและย้อนกลับได้ลงใน `DREAMS.md`
    - `memory rem-backfill --path ... --stage-short-term` เตรียมตัวเลือกถาวรที่มีหลักฐานรองรับเข้าสู่ที่เก็บหลักฐานระยะสั้นเดียวกับที่เฟส deep ปกติใช้อยู่แล้ว
    - `memory rem-backfill --rollback` และ `--rollback-short-term` ลบอาร์ติแฟกต์การเติมข้อมูลย้อนหลังที่เตรียมไว้เหล่านั้น โดยไม่แตะรายการไดอารีปกติหรือการเรียกคืนระยะสั้นแบบสด

  </Accordion>
</AccordionGroup>

Control UI เปิดเผยโฟลว์เติมข้อมูลย้อนหลัง/รีเซ็ตไดอารีเดียวกัน เพื่อให้คุณตรวจสอบผลลัพธ์ในฉาก Dreams ก่อนตัดสินใจว่าตัวเลือกที่มีหลักฐานรองรับควรได้รับการโปรโมตหรือไม่ Scene ยังแสดงเลนที่มีหลักฐานรองรับแยกต่างหาก เพื่อให้คุณเห็นว่ารายการระยะสั้นที่เตรียมไว้รายการใดมาจากการเล่นซ้ำข้อมูลในอดีต รายการที่โปรโมตแล้วรายการใดนำโดยหลักฐานรองรับ และล้างเฉพาะรายการที่เตรียมไว้แบบมีหลักฐานรองรับเท่านั้นโดยไม่แตะสถานะระยะสั้นสดปกติ

## สัญญาณการจัดอันดับ Deep

การจัดอันดับ Deep ใช้สัญญาณฐานแบบถ่วงน้ำหนักหกอย่าง รวมถึงการเสริมแรงจากเฟส:

| สัญญาณ              | น้ำหนัก | คำอธิบาย                                       |
| ------------------- | ------ | ------------------------------------------------- |
| ความถี่           | 0.24   | จำนวนสัญญาณระยะสั้นที่รายการสะสมไว้ |
| ความเกี่ยวข้อง           | 0.30   | คุณภาพการดึงข้อมูลเฉลี่ยของรายการ           |
| ความหลากหลายของคำค้น     | 0.15   | บริบทคำค้น/วัน distinct ที่ทำให้รายการปรากฏ      |
| ความใหม่             | 0.15   | คะแนนความสดใหม่ที่ลดลงตามเวลา                      |
| การรวมความจำ       | 0.10   | ความแรงของการเกิดซ้ำหลายวัน                     |
| ความสมบูรณ์เชิงแนวคิด | 0.06   | ความหนาแน่นของแท็กแนวคิดจาก snippet/path             |

การพบในเฟส Light และ REM จะเพิ่มบูสต์ขนาดเล็กที่ลดลงตามความใหม่จาก `memory/.dreams/phase-signals.json`

## การจัดตารางเวลา

เมื่อเปิดใช้ `memory-core` จะจัดการงาน cron หนึ่งงานโดยอัตโนมัติสำหรับการ sweep dreaming แบบเต็ม แต่ละ sweep จะรันเฟสตามลำดับ: light → REM → deep

sweep นี้รวม workspace runtime หลักและ workspace ของ agent ที่กำหนดค่าไว้ โดยลบซ้ำตาม path เพื่อให้การ fan-out ของ workspace subagent ไม่ทำให้ `DREAMS.md` และสถานะหน่วยความจำของ agent หลักถูกตัดออก

พฤติกรรม cadence ค่าเริ่มต้น:

| การตั้งค่า              | ค่าเริ่มต้น       |
| -------------------- | ------------- |
| `dreaming.frequency` | `0 3 * * *`   |
| `dreaming.model`     | โมเดลค่าเริ่มต้น |

## ค่าเริ่มต้นสำคัญ

การตั้งค่าทั้งหมดอยู่ภายใต้ `plugins.entries.memory-core.config.dreaming`

<ParamField path="enabled" type="boolean" default="false">
  เปิดหรือปิดการ sweep dreaming
</ParamField>
<ParamField path="frequency" type="string" default="0 3 * * *">
  Cron cadence สำหรับการ sweep dreaming แบบเต็ม
</ParamField>
<ParamField path="model" type="string">
  การ override โมเดล subagent ของ Dream Diary แบบไม่บังคับ ใช้ค่า `provider/model` ที่เป็น canonical เมื่อกำหนด allowlist `allowedModels` ของ subagent ด้วย
</ParamField>

<Warning>
`dreaming.model` ต้องใช้ `plugins.entries.memory-core.subagent.allowModelOverride: true` หากต้องการจำกัด ให้ตั้งค่า `plugins.entries.memory-core.subagent.allowedModels` ด้วย ความล้มเหลวด้านความน่าเชื่อถือหรือ allowlist จะยังมองเห็นได้แทนที่จะ fallback อย่างเงียบ ๆ การลองซ้ำครอบคลุมเฉพาะข้อผิดพลาดโมเดลไม่พร้อมใช้งานเท่านั้น
</Warning>

<Note>
นโยบายเฟส เกณฑ์ขั้นต่ำ และพฤติกรรมพื้นที่เก็บข้อมูลเป็นรายละเอียดการใช้งานภายใน (ไม่ใช่ config สำหรับผู้ใช้) ดู [ข้อมูลอ้างอิงการกำหนดค่าหน่วยความจำ](/th/reference/memory-config#dreaming) สำหรับรายการคีย์ทั้งหมด
</Note>

## Dreams UI

เมื่อเปิดใช้ แท็บ **Dreams** ของ Gateway จะแสดง:

- สถานะเปิดใช้ dreaming ปัจจุบัน
- สถานะระดับเฟสและการมีอยู่ของ managed-sweep
- จำนวนระยะสั้น มีหลักฐานรองรับ สัญญาณ และที่โปรโมตวันนี้
- เวลารันตามกำหนดครั้งถัดไป
- เลน Scene ที่มีหลักฐานรองรับแยกต่างหากสำหรับรายการ replay จากประวัติที่เตรียมไว้
- ตัวอ่าน Dream Diary แบบขยายได้ที่รองรับโดย `doctor.memory.dreamDiary`
