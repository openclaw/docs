OpenClaw มอบเครื่องมือให้เอเจนต์ทำงานข้ามเซสชัน ตรวจสอบสถานะ และควบคุมประสานงานเอเจนต์ย่อย

## เครื่องมือที่พร้อมใช้งาน

| เครื่องมือ               | หน้าที่                                                                |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | แสดงรายการเซสชันพร้อมตัวกรองเสริม (ชนิด ป้ายกำกับ เอเจนต์ คลังเก็บ ตัวอย่าง)  |
| `sessions_history` | อ่านทรานสคริปต์ของเซสชันที่ระบุ                                   |
| `sessions_send`    | ส่งข้อความไปยังเซสชันอื่นและเลือกที่จะรอได้                       |
| `sessions_spawn`   | สร้างเซสชันเอเจนต์ย่อยแบบแยกสำหรับงานเบื้องหลัง                     |
| `sessions_yield`   | สิ้นสุดรอบปัจจุบันและรอผลลัพธ์ติดตามจากเอเจนต์ย่อย               |
| `subagents`        | แสดงรายการสถานะของเอเจนต์ย่อยที่สร้างขึ้นสำหรับเซสชันนี้                              |
| `session_status`   | แสดงการ์ดรูปแบบ `/status` และเลือกตั้งค่าการแทนที่โมเดลรายเซสชัน |

เครื่องมือเหล่านี้ยังคงอยู่ภายใต้โปรไฟล์เครื่องมือที่ใช้งานและนโยบายอนุญาต/ปฏิเสธ `tools.profile: "coding"` มีชุดการควบคุมประสานงานเซสชันทั้งหมด รวมถึง `sessions_spawn`, `sessions_yield` และ `subagents` ส่วน `tools.profile: "messaging"` มีเครื่องมือส่งข้อความข้ามเซสชัน (`sessions_list`, `sessions_history`, `sessions_send`, `session_status`) แต่ไม่มีการสร้างเอเจนต์ย่อย หากต้องการคงโปรไฟล์การส่งข้อความไว้และยังอนุญาตการมอบหมายงานแบบเนทีฟ ให้เพิ่ม:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

นโยบายระดับกลุ่ม ผู้ให้บริการ แซนด์บ็อกซ์ และรายเอเจนต์ยังสามารถนำเครื่องมือเหล่านั้นออกหลังขั้นตอนโปรไฟล์ได้ ใช้ `/tools` จากเซสชันที่ได้รับผลกระทบเพื่อตรวจสอบรายการเครื่องมือที่มีผลจริง

## การแสดงรายการและอ่านเซสชัน

`sessions_list` ส่งคืนเซสชันพร้อมคีย์ agentId ชนิด ช่องทาง โมเดล จำนวนโทเค็น และการประทับเวลา กรองตาม `kinds` (อาร์เรย์; ค่าที่รับได้: `main`, `group`, `cron`, `hook`, `node`, `other`) ค่า `label` ที่ตรงกันทุกประการ ค่า `agentId` ที่ตรงกันทุกประการ ข้อความ `search` หรือความใหม่ล่าสุด (`activeMinutes`) โดยค่าเริ่มต้นจะส่งคืนเซสชันที่ใช้งานอยู่ ส่ง `archived: true` เพื่อตรวจสอบเซสชันที่เก็บถาวรแทน แถวต่าง ๆ มีสถานะ `pinned` และ `archived` ตั้งค่า `includeDerivedTitles`, `includeLastMessage` หรือ `messageLimit` (จำกัดสูงสุดที่ 20) เมื่อต้องการคัดแยกแบบกล่องจดหมาย: ชื่อที่คำนวณตามขอบเขตการมองเห็น ตัวอย่างข้อความล่าสุด หรือข้อความล่าสุดจำนวนจำกัดในแต่ละแถว ชื่อและตัวอย่างที่คำนวณจะสร้างขึ้นเฉพาะสำหรับเซสชันที่ผู้เรียกมองเห็นได้อยู่แล้วภายใต้นโยบายการมองเห็นของเครื่องมือเซสชันที่กำหนดค่าไว้ ดังนั้นเซสชันที่ไม่เกี่ยวข้องจะยังคงถูกซ่อน เมื่อจำกัดการมองเห็น `sessions_list` จะส่งคืนข้อมูลเมตา `visibility` ซึ่งเป็นทางเลือก เพื่อแสดงโหมดที่มีผลจริงและคำเตือนว่าผลลัพธ์อาจถูกจำกัดขอบเขต

`sessions_history` ดึงทรานสคริปต์การสนทนาของเซสชันที่ระบุ โดยค่าเริ่มต้นจะไม่รวมผลลัพธ์ของเครื่องมือ ส่ง `includeTools: true` เพื่อดูผลลัพธ์เหล่านั้น ใช้ `limit` สำหรับส่วนท้ายล่าสุดที่จำกัดขนาด ส่ง `offset: 0` เมื่อต้องการข้อมูลเมตาการแบ่งหน้า แล้วส่งค่า `nextOffset` ที่ได้รับกลับมาเพื่อเลื่อนย้อนหลังผ่านหน้าต่างทรานสคริปต์ OpenClaw ที่เก่ากว่าโดยไม่ต้องอ่านไฟล์ทรานสคริปต์ดิบ หน้าที่ระบุออฟเซ็ตอย่างชัดเจนจะไม่ผสานการนำเข้าสำรองจาก CLI ภายนอก ให้ใช้มุมมองส่วนท้ายล่าสุดตามค่าเริ่มต้น (ไม่มี `offset`) เมื่อต้องการประวัติการแสดงผลที่ผสานแล้ว

มุมมองที่ส่งคืนจะถูกจำกัดขนาดและกรองเพื่อความปลอดภัยโดยตั้งใจ:

- ข้อความของผู้ช่วยจะถูกปรับรูปแบบให้เป็นมาตรฐานก่อนเรียกคืน:
  - แท็กการคิดจะถูกตัดออก
  - บล็อกโครงร่าง `<relevant-memories>` / `<relevant_memories>` จะถูกตัดออก
  - บล็อกเพย์โหลด XML การเรียกเครื่องมือแบบข้อความธรรมดา เช่น `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` และ `<function_calls>...</function_calls>` จะถูกตัดออก รวมถึงเพย์โหลดที่ถูกตัดทอนและไม่เคยปิดอย่างสมบูรณ์
  - โครงร่างการเรียกเครื่องมือ/ผลลัพธ์ที่ถูกลดรูป เช่น `[Tool Call: ...]`, `[Tool Result ...]` และ `[Historical context ...]` จะถูกตัดออก
  - โทเค็นควบคุมโมเดลที่รั่วไหล เช่น `<|assistant|>`, โทเค็น ASCII `<|...|>` อื่น ๆ และรูปแบบตัวเต็ม `<｜...｜>` จะถูกตัดออก
  - XML การเรียกเครื่องมือ MiniMax ที่มีรูปแบบผิด เช่น `<invoke ...>` / `</minimax:tool_call>` จะถูกตัดออก
- ข้อความที่มีลักษณะเป็นข้อมูลประจำตัว/โทเค็นจะถูกปกปิดก่อนส่งคืน
- บล็อกข้อความยาวจะถูกตัดทอน
- ประวัติที่มีขนาดใหญ่มากอาจละทิ้งแถวเก่าหรือแทนที่แถวที่ใหญ่เกินไปด้วย `[sessions_history omitted: message too large]`
- เครื่องมือจะรายงานแฟล็กสรุป เช่น `truncated`, `droppedMessages`, `contentTruncated`, `contentRedacted`, `bytes` และข้อมูลเมตาการแบ่งหน้า

เครื่องมือทั้งสองยอมรับได้ทั้ง **คีย์เซสชัน** (เช่น `"main"`) หรือ **รหัสเซสชัน** จากการเรียกแสดงรายการก่อนหน้า

หากต้องการทรานสคริปต์ดิบที่ตรงทั้งหมด ให้ตรวจสอบแถวทรานสคริปต์ SQLite ที่อยู่ในขอบเขต แทนการถือว่า `sessions_history` เป็นดัมป์ที่ไม่ได้กรอง

## การส่งข้อความข้ามเซสชัน

`sessions_send` ส่งข้อความไปยังเซสชันอื่นและเลือกที่จะรอการตอบกลับได้:

- **ส่งแล้วไม่รอ:** ตั้งค่า `timeoutSeconds: 0` เพื่อเข้าคิวและส่งคืนทันที
- **รอการตอบกลับ:** ตั้งค่าการหมดเวลาและรับการตอบกลับแบบอินไลน์

เซสชันแชตที่จำกัดขอบเขตเฉพาะเธรด เช่น คีย์ที่ลงท้ายด้วย `:thread:<id>` ไม่ใช่เป้าหมาย `sessions_send` ที่ถูกต้อง ให้ใช้คีย์เซสชันช่องทางหลักสำหรับการประสานงานระหว่างเอเจนต์ เพื่อไม่ให้ข้อความที่กำหนดเส้นทางผ่านเครื่องมือปรากฏในเธรดที่กำลังโต้ตอบกับมนุษย์

ข้อความและการตอบกลับติดตามผลแบบ A2A จะถูกระบุเป็นข้อมูลระหว่างเซสชันในพรอมต์ของผู้รับ (`[Inter-session message ... isUser=false]`) และในแหล่งที่มาของทรานสคริปต์ เอเจนต์ผู้รับควรถือว่าสิ่งเหล่านี้เป็นข้อมูลที่กำหนดเส้นทางผ่านเครื่องมือ ไม่ใช่คำสั่งที่ผู้ใช้ปลายทางเขียนโดยตรง

หลังจากเป้าหมายตอบกลับ OpenClaw สามารถเรียกใช้ **ลูปตอบกลับ** ซึ่งเอเจนต์จะสลับกันส่งข้อความ (สูงสุด `session.agentToAgent.maxPingPongTurns`, ช่วง 0-20, ค่าเริ่มต้น 5) เอเจนต์เป้าหมายสามารถตอบ `REPLY_SKIP` เพื่อหยุดก่อนกำหนด

ส่ง `watch: true` เพื่อลงทะเบียนผู้ส่งเป็นผู้เฝ้าดูการเปลี่ยนแปลงสถานะของเป้าหมายด้วย: เมื่อผู้ดำเนินการรายอื่นส่งข้อความจากมนุษย์โดยตรงไปยังเป้าหมายหรือเปลี่ยนเป้าหมายของเป้าหมายในภายหลัง ผู้ส่งจะได้รับประกาศจากระบบที่ชี้ไปยัง `session_status` `changesSince` การลงทะเบียนจะเกิดขึ้นหลังจากการจัดส่งสำเร็จ โดยกำหนดเป้าหมายเป็นเซสชันที่ได้รับข้อความจริง และเริ่มต้นที่เวอร์ชันสถานะปัจจุบันของเซสชันนั้น ดังนั้นเฉพาะการเปลี่ยนแปลงในภายหลังเท่านั้นที่จะสร้างประกาศ ผลลัพธ์จะรายงาน `watched: true` เมื่อลงทะเบียนสำเร็จ ดู [การรับรู้สถานะเซสชัน](/concepts/session-state)

## ตัวช่วยด้านสถานะและการควบคุมประสานงาน

`session_status` คือเครื่องมือแบบเบาที่เทียบเท่า `/status` สำหรับเซสชันปัจจุบันหรือเซสชันอื่นที่มองเห็นได้ โดยรายงานการใช้งาน เวลา สถานะโมเดล/รันไทม์ และบริบทงานเบื้องหลังที่เชื่อมโยงเมื่อมี เช่นเดียวกับ `/status` เครื่องมือนี้สามารถเติมตัวนับโทเค็น/แคชที่มีข้อมูลไม่ครบจากรายการการใช้งานล่าสุดในทรานสคริปต์ และ `model=default` จะล้างการแทนที่รายเซสชัน ใช้ `sessionKey="current"` สำหรับเซสชันปัจจุบันของผู้เรียก ป้ายกำกับไคลเอนต์ที่มองเห็นได้ เช่น `openclaw-tui` ไม่ใช่คีย์เซสชัน

เมื่อมีข้อมูลเมตาของเส้นทาง `session_status` จะรวมบล็อก JSON `Route context` ที่มองเห็นได้และฟิลด์ `details` แบบมีโครงสร้างที่ตรงกันด้วย ฟิลด์เหล่านี้ช่วยแยกความแตกต่างระหว่างคีย์เซสชันกับเส้นทางที่กำลังจัดการการเรียกใช้แบบสดในปัจจุบัน:

- `origin` คือสถานที่ที่สร้างเซสชัน หรือผู้ให้บริการที่อนุมานจากคำนำหน้าคีย์เซสชันที่ส่งมอบได้ เมื่อสถานะเก่าไม่มีข้อมูลเมตาต้นทางที่จัดเก็บไว้
- `active` คือเส้นทางการเรียกใช้แบบสดในปัจจุบัน จะรายงานเฉพาะเซสชันแบบสดหรือเซสชันปัจจุบันที่กำลังได้รับการจัดการอยู่ในขณะนี้
- `deliveryContext` คือเส้นทางการจัดส่งถาวรที่จัดเก็บไว้ในเซสชัน ซึ่ง OpenClaw สามารถนำกลับมาใช้สำหรับการจัดส่งภายหลัง แม้พื้นผิวที่ใช้งานอยู่จะแตกต่างกัน

## การเปลี่ยนแปลงสถานะเซสชัน

OpenClaw เก็บบันทึกสัญญาณแบบถาวรของการเปลี่ยนแปลงสถานะเซสชันที่สำคัญ (ข้อความจากมนุษย์โดยตรงไปยังเซสชันที่เฝ้าดู ผลลัพธ์การเรียกใช้ลูก การเปลี่ยนแปลงเป้าหมาย Compaction) แถว `sessions_list` และ `session_status` เปิดเผย `stateVersion` ของเซสชัน และ `session_status` ยอมรับ `changesSince: <version>` เพื่อส่งคืนเหตุการณ์แบบมีชนิดหลังจากเวอร์ชันนั้น พร้อมการส่งสัญญาณ `historyGap` ที่ตรงทั้งหมดเมื่อเวอร์ชันที่ร้องขอเก่ากว่าประวัติที่เก็บรักษาไว้ ผู้เฝ้าดู — พาเรนต์ที่สร้างจะเป็นโดยอัตโนมัติ ส่วน `sessions_send watch: true` เป็นแบบชัดแจ้ง — จะได้รับประกาศสถานะล้าสมัยที่รวมเป็นหนึ่งรายการ เมื่อผู้ดำเนินการรายอื่นเปลี่ยนแปลงเซสชันที่เฝ้าดู

ดู [การรับรู้สถานะเซสชัน](/concepts/session-state) สำหรับโมเดลฉบับเต็ม: ชนิดเหตุการณ์ การลงทะเบียนผู้เฝ้าดู โปรโตคอลประกาศป้องกันสแปม ขั้นตอนการปรับยอด และข้อจำกัดปัจจุบัน

`sessions_yield` จงใจสิ้นสุดรอบปัจจุบัน เพื่อให้ข้อความถัดไปเป็นเหตุการณ์ติดตามผลที่กำลังรอ ใช้หลังจากสร้างเอเจนต์ย่อยเมื่อต้องการให้ผลลัพธ์การเสร็จสิ้นมาถึงเป็นข้อความถัดไป แทนการสร้างลูปสำรวจสถานะ

`subagents` คือตัวช่วยด้านการมองเห็นสำหรับเอเจนต์ย่อย OpenClaw ที่สร้างไว้แล้ว รองรับ `action: "list"` เพื่อตรวจสอบการเรียกใช้ที่ใช้งานอยู่/ล่าสุด

## การสร้างเอเจนต์ย่อย

`sessions_spawn` สร้างเซสชันแบบแยกสำหรับงานเบื้องหลังโดยค่าเริ่มต้น โดยไม่บล็อกเสมอ และส่งคืนทันทีพร้อม `runId` และ `childSessionKey` การเรียกใช้เอเจนต์ย่อยแบบเนทีฟจะได้รับงานที่มอบหมายในข้อความ `[Subagent Task]` แรกที่มองเห็นได้ของเซสชันลูก ขณะที่พรอมต์ระบบจะมีเฉพาะกฎรันไทม์ของเอเจนต์ย่อยและบริบทการกำหนดเส้นทาง

ตัวเลือกสำคัญ:

- `runtime: "subagent"` (ค่าเริ่มต้น) หรือ `"acp"` สำหรับเอเจนต์จากชุดควบคุมภายนอก
- การแทนที่ `model` และ `thinking` สำหรับเซสชันลูก
- `thread: true` เพื่อผูกการสร้างเข้ากับเธรดแชต (Discord, Slack ฯลฯ)
- `sandbox: "require"` เพื่อบังคับใช้แซนด์บ็อกซ์กับเซสชันลูก
- `context: "fork"` สำหรับเอเจนต์ย่อยแบบเนทีฟ เมื่อเซสชันลูกต้องการทรานสคริปต์ของผู้ร้องขอปัจจุบัน ละเว้นหรือใช้ `context: "isolated"` เพื่อให้เซสชันลูกเริ่มต้นใหม่ `context: "fork"` ใช้ได้เฉพาะกับ `runtime: "subagent"` เท่านั้น เอเจนต์ย่อยแบบเนทีฟที่ผูกกับเธรดมีค่าเริ่มต้นเป็น `context: "fork"` เว้นแต่ `threadBindings.defaultSpawnContext` จะระบุเป็นอย่างอื่น

เอเจนต์ย่อยปลายทางตามค่าเริ่มต้นจะไม่ได้รับเครื่องมือเซสชัน เมื่อ `maxSpawnDepth >= 2` เอเจนต์ย่อยผู้ควบคุมประสานงานระดับความลึก 1 จะได้รับ `sessions_spawn`, `subagents`, `sessions_list` และ `sessions_history` เพิ่มเติม เพื่อให้สามารถจัดการเซสชันลูกของตนเองได้ การเรียกใช้ปลายทางยังคงไม่ได้รับเครื่องมือควบคุมประสานงานแบบเรียกซ้ำ

หลังจากเสร็จสิ้น ขั้นตอนประกาศจะโพสต์ผลลัพธ์ไปยังช่องทางของผู้ร้องขอ การจัดส่งผลลัพธ์เมื่อเสร็จสิ้นจะรักษาการกำหนดเส้นทางเธรด/หัวข้อที่ผูกไว้เมื่อมี และหากต้นทางของผลลัพธ์เมื่อเสร็จสิ้นระบุเพียงช่องทาง OpenClaw ยังคงสามารถนำเส้นทางที่จัดเก็บไว้ของเซสชันผู้ร้องขอ (`lastChannel` / `lastTo`) กลับมาใช้สำหรับการจัดส่งโดยตรงได้

สำหรับพฤติกรรมเฉพาะ ACP โปรดดู [เอเจนต์ ACP](/th/tools/acp-agents)

## การมองเห็น

เครื่องมือเซสชันถูกจำกัดขอบเขตเพื่อจำกัดสิ่งที่เอเจนต์มองเห็นได้:

| ระดับ   | ขอบเขต                                    |
| ------- | ---------------------------------------- |
| `self`  | เฉพาะเซสชันปัจจุบัน                 |
| `tree`  | เซสชันปัจจุบัน + เอเจนต์ย่อยที่สร้างขึ้น     |
| `agent` | เซสชันทั้งหมดสำหรับเอเจนต์นี้              |
| `all`   | เซสชันทั้งหมด (ข้ามเอเจนต์หากกำหนดค่าไว้) |

ค่าเริ่มต้นคือ `tree` เซสชันในแซนด์บ็อกซ์จะถูกจำกัดไว้ที่ `tree` โดยไม่ขึ้นกับการกำหนดค่า
