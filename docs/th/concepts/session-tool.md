OpenClaw มอบเครื่องมือให้เอเจนต์ทำงานข้ามเซสชัน ตรวจสอบสถานะ และ
จัดการเอเจนต์ย่อย

## เครื่องมือที่มีให้ใช้

| เครื่องมือ          | สิ่งที่ทำ                                                                   |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | แสดงรายการเซสชันพร้อมตัวกรองเสริม (ชนิด, ป้ายกำกับ, เอเจนต์, ความล่าสุด, ตัวอย่าง) |
| `sessions_history` | อ่านบันทึกบทสนทนาของเซสชันที่ระบุ                                          |
| `sessions_send`    | ส่งข้อความไปยังเซสชันอื่นและเลือกได้ว่าจะรอหรือไม่                         |
| `sessions_spawn`   | สร้างเซสชันเอเจนต์ย่อยแบบแยกสำหรับงานเบื้องหลัง                           |
| `sessions_yield`   | จบเทิร์นปัจจุบันและรอผลลัพธ์ติดตามจากเอเจนต์ย่อย                           |
| `subagents`        | แสดงรายการสถานะของเอเจนต์ย่อยที่สร้างขึ้นสำหรับเซสชันนี้                   |
| `session_status`   | แสดงการ์ดแบบ `/status` และเลือกตั้งค่าการแทนที่โมเดลต่อเซสชันได้          |

เครื่องมือเหล่านี้ยังอยู่ภายใต้โปรไฟล์เครื่องมือที่ใช้งานอยู่และนโยบาย
อนุญาต/ปฏิเสธ `tools.profile: "coding"` รวมชุดการจัดการเซสชันเต็มรูปแบบ
รวมถึง `sessions_spawn`, `sessions_yield` และ `subagents`
`tools.profile: "messaging"` รวมเครื่องมือส่งข้อความข้ามเซสชัน
(`sessions_list`, `sessions_history`, `sessions_send`, `session_status`) แต่
ไม่รวมการสร้างเอเจนต์ย่อย หากต้องการคงโปรไฟล์การส่งข้อความไว้และยัง
อนุญาตการมอบหมายงานแบบเนทีฟ ให้เพิ่ม:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

นโยบายระดับกลุ่ม ผู้ให้บริการ sandbox และต่อเอเจนต์ยังสามารถลบเครื่องมือเหล่านั้น
หลังขั้นตอนโปรไฟล์ได้ ใช้ `/tools` จากเซสชันที่ได้รับผลกระทบเพื่อตรวจสอบ
รายการเครื่องมือที่มีผลจริง

## การแสดงรายการและอ่านเซสชัน

`sessions_list` ส่งคืนเซสชันพร้อมคีย์, agentId, ชนิด, ช่องทาง, โมเดล,
จำนวนโทเค็น และเวลาประทับ กรองตามชนิด (`main`, `group`, `cron`, `hook`,
`node`), `label` แบบตรงกัน, `agentId` แบบตรงกัน, ข้อความค้นหา หรือความล่าสุด
(`activeMinutes`) ได้ เมื่อต้องคัดแยกแบบกล่องจดหมาย เครื่องมือนี้ยังสามารถขอ
ชื่อเรื่องที่สร้างขึ้นตามขอบเขตการมองเห็น ตัวอย่างข้อความล่าสุด หรือข้อความล่าสุด
แบบมีขอบเขตในแต่ละแถวได้ ชื่อเรื่องและตัวอย่างที่สร้างขึ้นจะผลิตเฉพาะสำหรับเซสชัน
ที่ผู้เรียกมองเห็นได้อยู่แล้วภายใต้นโยบายการมองเห็นเครื่องมือเซสชันที่กำหนดไว้
ดังนั้นเซสชันที่ไม่เกี่ยวข้องจะยังคงถูกซ่อน เมื่อการมองเห็นถูกจำกัด
`sessions_list` จะส่งคืนเมตาดาต้า `visibility` แบบไม่บังคับที่แสดงโหมดที่มีผล
และคำเตือนว่าผลลัพธ์อาจถูกจำกัดตามขอบเขต

`sessions_history` ดึงบันทึกบทสนทนาสำหรับเซสชันที่ระบุ โดยค่าเริ่มต้น
ผลลัพธ์ของเครื่องมือจะถูกตัดออก ส่ง `includeTools: true` เพื่อดูผลลัพธ์เหล่านั้น
มุมมองที่ส่งคืนถูกจำกัดขอบเขตและกรองเพื่อความปลอดภัยโดยตั้งใจ:

- ข้อความของผู้ช่วยจะถูกปรับให้เป็นมาตรฐานก่อนเรียกคืน:
  - แท็ก thinking จะถูกลบออก
  - บล็อกโครง `<relevant-memories>` / `<relevant_memories>` จะถูกลบออก
  - บล็อกเพย์โหลด XML การเรียกเครื่องมือแบบข้อความล้วน เช่น `<tool_call>...</tool_call>`,
    `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` และ
    `<function_calls>...</function_calls>` จะถูกลบออก รวมถึงเพย์โหลดที่ถูกตัดทอน
    ซึ่งไม่เคยปิดอย่างเรียบร้อย
  - โครงการเรียกเครื่องมือ/ผลลัพธ์ที่ถูกลดระดับ เช่น `[Tool Call: ...]`,
    `[Tool Result ...]` และ `[Historical context ...]` จะถูกลบออก
  - โทเค็นควบคุมโมเดลที่รั่วไหล เช่น `<|assistant|>`, โทเค็น ASCII อื่น ๆ
    แบบ `<|...|>` และตัวแปรความกว้างเต็มแบบ `<｜...｜>` จะถูกลบออก
  - XML การเรียกเครื่องมือ MiniMax ที่ผิดรูปแบบ เช่น `<invoke ...>` /
    `</minimax:tool_call>` จะถูกลบออก
- ข้อความที่ดูเหมือนข้อมูลประจำตัว/โทเค็นจะถูกปกปิดก่อนส่งคืน
- บล็อกข้อความยาวจะถูกตัดทอน
- ประวัติขนาดใหญ่มากอาจทิ้งแถวเก่ากว่า หรือแทนที่แถวที่มีขนาดใหญ่เกินด้วย
  `[sessions_history omitted: message too large]`
- เครื่องมือรายงานแฟล็กสรุป เช่น `truncated`, `droppedMessages`,
  `contentTruncated`, `contentRedacted` และ `bytes`

เครื่องมือทั้งสองรับได้ทั้ง **คีย์เซสชัน** (เช่น `"main"`) หรือ **รหัสเซสชัน**
จากการเรียกรายการก่อนหน้า

หากต้องการบันทึกบทสนทนาแบบตรงทุกไบต์ ให้ตรวจสอบไฟล์บันทึกบทสนทนาบนดิสก์
แทนการถือว่า `sessions_history` เป็นดัมพ์ดิบ

## การส่งข้อความข้ามเซสชัน

`sessions_send` ส่งข้อความไปยังเซสชันอื่นและเลือกได้ว่าจะรอคำตอบหรือไม่:

- **ส่งแล้วไม่รอ:** ตั้ง `timeoutSeconds: 0` เพื่อเข้าคิวและส่งคืนทันที
- **รอคำตอบ:** ตั้งเวลาหมดเวลาและรับคำตอบแบบอินไลน์

เซสชันแชทที่มีขอบเขตเป็นเธรด เช่น คีย์ Slack หรือ Discord ที่ลงท้ายด้วย
`:thread:<id>` ไม่ใช่เป้าหมาย `sessions_send` ที่ถูกต้อง ใช้คีย์เซสชันของช่องทางหลัก
สำหรับการประสานงานระหว่างเอเจนต์ เพื่อไม่ให้ข้อความที่กำหนดเส้นทางผ่านเครื่องมือ
ปรากฏในเธรดที่กำลังหันหน้าเข้าหามนุษย์อยู่

ข้อความและการตอบกลับติดตามผล A2A จะถูกทำเครื่องหมายเป็นข้อมูลระหว่างเซสชันใน
พรอมป์ฝั่งรับ (`[Inter-session message ... isUser=false]`) และในแหล่งที่มาของ
บันทึกบทสนทนา เอเจนต์ฝั่งรับควรถือว่าสิ่งเหล่านี้เป็นข้อมูลที่กำหนดเส้นทางผ่านเครื่องมือ
ไม่ใช่คำสั่งที่ผู้ใช้ปลายทางเขียนโดยตรง

หลังจากเป้าหมายตอบกลับ OpenClaw สามารถเรียกใช้ **ลูปตอบกลับกลับไปมา** ซึ่ง
เอเจนต์จะสลับกันส่งข้อความ (สูงสุด `session.agentToAgent.maxPingPongTurns`, ช่วง
0-20, ค่าเริ่มต้น 5) เอเจนต์เป้าหมายสามารถตอบ
`REPLY_SKIP` เพื่อหยุดก่อนกำหนดได้

## ตัวช่วยสถานะและการจัดการ

`session_status` เป็นเครื่องมือเทียบเท่า `/status` แบบเบาสำหรับเซสชันปัจจุบัน
หรือเซสชันอื่นที่มองเห็นได้ โดยรายงานการใช้งาน เวลา สถานะโมเดล/รันไทม์ และ
บริบทงานเบื้องหลังที่เชื่อมโยงเมื่อมีอยู่ เช่นเดียวกับ `/status` เครื่องมือนี้สามารถเติม
ตัวนับโทเค็น/แคชที่มีข้อมูลเบาบางจากรายการการใช้งานล่าสุดในบันทึกบทสนทนาได้ และ
`model=default` จะล้างการแทนที่ต่อเซสชัน ใช้ `sessionKey="current"` สำหรับ
เซสชันปัจจุบันของผู้เรียก ป้ายกำกับไคลเอนต์ที่มองเห็นได้ เช่น `openclaw-tui`
ไม่ใช่คีย์เซสชัน

เมื่อมีเมตาดาต้าเส้นทาง `session_status` จะรวมบล็อก JSON `Route context`
ที่มองเห็นได้ และฟิลด์ `details` แบบมีโครงสร้างที่สอดคล้องกันด้วย ฟิลด์เหล่านี้
ช่วยแยกความแตกต่างระหว่างคีย์เซสชันกับเส้นทางที่กำลังจัดการการรันสดอยู่ในปัจจุบัน:

- `origin` คือจุดที่เซสชันถูกสร้างขึ้น หรือผู้ให้บริการที่อนุมานจากคำนำหน้าคีย์เซสชัน
  ที่ส่งมอบได้ เมื่อสถานะเก่าขาดเมตาดาต้า origin ที่จัดเก็บไว้
- `active` คือเส้นทางการรันสดปัจจุบัน รายงานเฉพาะสำหรับเซสชันสดหรือเซสชันปัจจุบัน
  ที่กำลังถูกจัดการอยู่ตอนนี้
- `deliveryContext` คือเส้นทางการส่งมอบที่คงอยู่ซึ่งจัดเก็บไว้บนเซสชัน
  ซึ่ง OpenClaw สามารถใช้ซ้ำสำหรับการส่งมอบภายหลังได้ แม้เมื่อพื้นผิวที่ใช้งานอยู่
  แตกต่างออกไป

`sessions_yield` ตั้งใจจบเทิร์นปัจจุบันเพื่อให้ข้อความถัดไปเป็นเหตุการณ์ติดตามผล
ที่คุณกำลังรอ ใช้หลังจากสร้างเอเจนต์ย่อยเมื่อคุณต้องการให้ผลลัพธ์การเสร็จสิ้นมาถึง
เป็นข้อความถัดไป แทนการสร้างลูปโพล

`subagents` เป็นตัวช่วยการมองเห็นสำหรับเอเจนต์ย่อย OpenClaw ที่สร้างไว้แล้ว
รองรับ `action: "list"` เพื่อตรวจสอบการรันที่กำลังใช้งาน/ล่าสุด

## การสร้างเอเจนต์ย่อย

`sessions_spawn` สร้างเซสชันแบบแยกสำหรับงานเบื้องหลังโดยค่าเริ่มต้น
เครื่องมือนี้ไม่บล็อกเสมอ โดยส่งคืนทันทีพร้อม `runId` และ
`childSessionKey` การรันเอเจนต์ย่อยแบบเนทีฟจะได้รับงานที่มอบหมายในข้อความ
`[Subagent Task]` แรกที่มองเห็นได้ของเซสชันลูก ขณะที่พรอมป์ระบบจะมีเฉพาะ
กฎรันไทม์ของเอเจนต์ย่อยและบริบทการกำหนดเส้นทาง

ตัวเลือกสำคัญ:

- `runtime: "subagent"` (ค่าเริ่มต้น) หรือ `"acp"` สำหรับเอเจนต์ฮาร์เนสภายนอก
- การแทนที่ `model` และ `thinking` สำหรับเซสชันลูก
- `thread: true` เพื่อผูกการสร้างกับเธรดแชท (Discord, Slack ฯลฯ)
- `sandbox: "require"` เพื่อบังคับใช้ sandbox กับลูก
- `context: "fork"` สำหรับเอเจนต์ย่อยแบบเนทีฟเมื่อลูกต้องการบันทึกบทสนทนา
  ของผู้ร้องขอปัจจุบัน ละไว้หรือใช้ `context: "isolated"` สำหรับลูกที่สะอาด
  เอเจนต์ย่อยแบบเนทีฟที่ผูกกับเธรดจะใช้ค่าเริ่มต้นเป็น `context: "fork"` เว้นแต่
  `threadBindings.defaultSpawnContext` จะระบุไว้เป็นอย่างอื่น

เอเจนต์ย่อยปลายทางค่าเริ่มต้นจะไม่ได้รับเครื่องมือเซสชัน เมื่อ
`maxSpawnDepth >= 2` เอเจนต์ย่อยผู้จัดการที่ความลึก 1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ `sessions_history` เพิ่มเติม
เพื่อให้จัดการลูกของตนเองได้ การรันปลายทางยังคงไม่ได้รับเครื่องมือจัดการแบบเรียกซ้ำ

หลังเสร็จสิ้น ขั้นตอนประกาศจะโพสต์ผลลัพธ์ไปยังช่องทางของผู้ร้องขอ
การส่งมอบผลลัพธ์การเสร็จสิ้นจะคงการกำหนดเส้นทางของเธรด/หัวข้อที่ผูกไว้เมื่อมีอยู่
และหาก origin ของการเสร็จสิ้นระบุได้เพียงช่องทาง OpenClaw ยังสามารถใช้เส้นทาง
ที่จัดเก็บไว้ของเซสชันผู้ร้องขอ (`lastChannel` / `lastTo`) ซ้ำสำหรับการส่งมอบโดยตรงได้

สำหรับพฤติกรรมเฉพาะ ACP ดู [เอเจนต์ ACP](/th/tools/acp-agents)

## การมองเห็น

เครื่องมือเซสชันถูกกำหนดขอบเขตเพื่อจำกัดสิ่งที่เอเจนต์มองเห็นได้:

| ระดับ   | ขอบเขต                                  |
| ------- | ---------------------------------------- |
| `self`  | เฉพาะเซสชันปัจจุบัน                     |
| `tree`  | เซสชันปัจจุบัน + เอเจนต์ย่อยที่สร้างขึ้น |
| `agent` | เซสชันทั้งหมดสำหรับเอเจนต์นี้            |
| `all`   | เซสชันทั้งหมด (ข้ามเอเจนต์หากกำหนดค่าไว้) |
