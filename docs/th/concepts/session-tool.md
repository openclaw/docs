OpenClaw มอบเครื่องมือให้เอเจนต์ทำงานข้ามเซสชัน ตรวจสอบสถานะ และ
ประสานงานเอเจนต์ย่อย

## เครื่องมือที่มีให้ใช้

| เครื่องมือ          | หน้าที่                                                                      |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | แสดงรายการเซสชันพร้อมตัวกรองเสริม (ชนิด ป้ายกำกับ เอเจนต์ ความล่าสุด ตัวอย่าง) |
| `sessions_history` | อ่านทรานสคริปต์ของเซสชันที่ระบุ                                             |
| `sessions_send`    | ส่งข้อความไปยังอีกเซสชันและเลือกได้ว่าจะรอ                                  |
| `sessions_spawn`   | สร้างเซสชันเอเจนต์ย่อยแบบแยกสำหรับงานเบื้องหลัง                              |
| `sessions_yield`   | จบเทิร์นปัจจุบันและรอผลลัพธ์ติดตามผลจากเอเจนต์ย่อย                            |
| `subagents`        | แสดงรายการ ชี้นำ หรือหยุดเอเจนต์ย่อยที่สร้างขึ้นสำหรับเซสชันนี้                 |
| `session_status`   | แสดงการ์ดสไตล์ `/status` และเลือกตั้งค่าการแทนที่โมเดลต่อเซสชันได้             |

เครื่องมือเหล่านี้ยังคงอยู่ภายใต้โปรไฟล์เครื่องมือที่ใช้งานอยู่และนโยบายอนุญาต/ปฏิเสธ
`tools.profile: "coding"` รวมชุดการประสานงานเซสชันทั้งหมด
รวมถึง `sessions_spawn`, `sessions_yield` และ `subagents`
`tools.profile: "messaging"` รวมเครื่องมือส่งข้อความข้ามเซสชัน
(`sessions_list`, `sessions_history`, `sessions_send`, `session_status`) แต่
ไม่รวมการสร้างเอเจนต์ย่อย หากต้องการคงโปรไฟล์การส่งข้อความไว้และยังคง
อนุญาตการมอบหมายงานแบบเนทีฟ ให้เพิ่ม:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

นโยบายระดับกลุ่ม ผู้ให้บริการ sandbox และต่อเอเจนต์ยังสามารถลบเครื่องมือเหล่านั้น
หลังขั้นตอนโปรไฟล์ได้ ใช้ `/tools` จากเซสชันที่ได้รับผลกระทบเพื่อตรวจสอบ
รายการเครื่องมือที่มีผลจริง

## การแสดงรายการและการอ่านเซสชัน

`sessions_list` ส่งคืนเซสชันพร้อม key, agentId, ชนิด, แชนเนล, โมเดล,
จำนวนโทเค็น และเวลา กรองตามชนิด (`main`, `group`, `cron`, `hook`,
`node`), `label` ที่ตรงกัน, `agentId` ที่ตรงกัน, ข้อความค้นหา หรือความล่าสุด
(`activeMinutes`) ได้ เมื่อต้องการคัดแยกแบบกล่องจดหมาย เครื่องมือนี้ยังสามารถขอ
ชื่อเรื่องที่สร้างขึ้นตามขอบเขตการมองเห็น ตัวอย่างข้อความล่าสุด หรือข้อความล่าสุด
แบบจำกัดจำนวนในแต่ละแถวได้ ชื่อเรื่องและตัวอย่างที่สร้างขึ้นจะถูกสร้างเฉพาะสำหรับ
เซสชันที่ผู้เรียกดูได้อยู่แล้วภายใต้นโยบายการมองเห็นเครื่องมือเซสชันที่กำหนดไว้
ดังนั้นเซสชันที่ไม่เกี่ยวข้องจะยังคงถูกซ่อนไว้

`sessions_history` ดึงทรานสคริปต์การสนทนาสำหรับเซสชันที่ระบุ
โดยค่าเริ่มต้น ผลลัพธ์ของเครื่องมือจะไม่ถูกรวมไว้ -- ส่ง `includeTools: true` เพื่อดูผลลัพธ์เหล่านั้น
มุมมองที่ส่งคืนถูกจำกัดขอบเขตและกรองเพื่อความปลอดภัยโดยตั้งใจ:

- ข้อความของผู้ช่วยถูกทำให้เป็นรูปแบบปกติก่อนเรียกคืน:
  - แท็กการคิดถูกตัดออก
  - บล็อกโครงร่าง `<relevant-memories>` / `<relevant_memories>` ถูกตัดออก
  - บล็อก payload XML ของการเรียกเครื่องมือแบบข้อความล้วน เช่น `<tool_call>...</tool_call>`,
    `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>` และ
    `<function_calls>...</function_calls>` ถูกตัดออก รวมถึง payload ที่ถูกตัดทอน
    ซึ่งไม่เคยปิดอย่างถูกต้อง
  - โครงร่างการเรียกเครื่องมือ/ผลลัพธ์ที่ถูกลดระดับ เช่น `[Tool Call: ...]`,
    `[Tool Result ...]` และ `[Historical context ...]` ถูกตัดออก
  - โทเค็นควบคุมโมเดลที่รั่วไหล เช่น `<|assistant|>`, โทเค็น ASCII
    `<|...|>` อื่น ๆ และตัวแปรแบบเต็มความกว้าง `<｜...｜>` ถูกตัดออก
  - XML การเรียกเครื่องมือ MiniMax ที่ผิดรูป เช่น `<invoke ...>` /
    `</minimax:tool_call>` ถูกตัดออก
- ข้อความที่คล้ายข้อมูลรับรอง/โทเค็นจะถูกปกปิดก่อนส่งคืน
- บล็อกข้อความยาวจะถูกตัดทอน
- ประวัติขนาดใหญ่มากอาจทิ้งแถวเก่ากว่า หรือแทนที่แถวที่ใหญ่เกินด้วย
  `[sessions_history omitted: message too large]`
- เครื่องมือรายงานแฟล็กสรุป เช่น `truncated`, `droppedMessages`,
  `contentTruncated`, `contentRedacted` และ `bytes`

ทั้งสองเครื่องมือรับได้ทั้ง **session key** (เช่น `"main"`) หรือ **session ID**
จากการเรียกแสดงรายการก่อนหน้า

หากคุณต้องการทรานสคริปต์แบบตรงทุกไบต์ ให้ตรวจสอบไฟล์ทรานสคริปต์บนดิสก์
แทนการถือว่า `sessions_history` เป็นการดัมป์ดิบ

## การส่งข้อความข้ามเซสชัน

`sessions_send` ส่งข้อความไปยังอีกเซสชันและเลือกได้ว่าจะรอคำตอบหรือไม่:

- **ส่งแล้วไม่รอ:** ตั้ง `timeoutSeconds: 0` เพื่อเข้าคิวและส่งคืนทันที
- **รอคำตอบ:** ตั้งค่า timeout และรับคำตอบแบบ inline

ข้อความและคำตอบติดตามผล A2A จะถูกทำเครื่องหมายเป็นข้อมูลระหว่างเซสชันใน
พรอมป์ฝั่งรับ (`[Inter-session message ... isUser=false]`) และในที่มาของทรานสคริปต์
เอเจนต์ฝั่งรับควรถือว่าสิ่งเหล่านี้เป็นข้อมูลที่ถูกส่งผ่านเครื่องมือ ไม่ใช่
คำสั่งที่ผู้ใช้ปลายทางเขียนโดยตรง

หลังจากเป้าหมายตอบกลับ OpenClaw สามารถรัน **ลูปตอบกลับกลับมา** ซึ่ง
เอเจนต์จะผลัดกันส่งข้อความ (สูงสุด 5 เทิร์น) เอเจนต์เป้าหมายสามารถตอบ
`REPLY_SKIP` เพื่อหยุดก่อนกำหนดได้

## ตัวช่วยสถานะและการประสานงาน

`session_status` เป็นเครื่องมือเทียบเท่า `/status` แบบเบาสำหรับเซสชันปัจจุบัน
หรือเซสชันอื่นที่มองเห็นได้ เครื่องมือนี้รายงานการใช้งาน เวลา สถานะโมเดล/รันไทม์ และ
บริบทงานเบื้องหลังที่เชื่อมโยงเมื่อมี เช่นเดียวกับ `/status` เครื่องมือนี้สามารถเติมกลับ
ตัวนับโทเค็น/cache ที่กระจัดกระจายจากรายการการใช้งานล่าสุดในทรานสคริปต์ได้ และ
`model=default` จะล้างการแทนที่ต่อเซสชัน ใช้ `sessionKey="current"` สำหรับ
เซสชันปัจจุบันของผู้เรียก ป้ายกำกับไคลเอนต์ที่มองเห็นได้ เช่น `openclaw-tui`
ไม่ใช่ session key

`sessions_yield` จงใจจบเทิร์นปัจจุบันเพื่อให้ข้อความถัดไปสามารถเป็น
อีเวนต์ติดตามผลที่คุณรออยู่ ใช้หลังจากสร้างเอเจนต์ย่อยเมื่อต้องการให้
ผลลัพธ์การเสร็จงานมาถึงเป็นข้อความถัดไป แทนการสร้างลูป polling

`subagents` เป็นตัวช่วย control-plane สำหรับเอเจนต์ย่อย OpenClaw
ที่ถูกสร้างไว้แล้ว รองรับ:

- `action: "list"` เพื่อตรวจสอบการรันที่ใช้งานอยู่/ล่าสุด
- `action: "steer"` เพื่อส่งคำแนะนำติดตามผลไปยัง child ที่กำลังรัน
- `action: "kill"` เพื่อหยุด child หนึ่งรายการหรือ `all`

## การสร้างเอเจนต์ย่อย

โดยค่าเริ่มต้น `sessions_spawn` จะสร้างเซสชันแยกสำหรับงานเบื้องหลัง
เครื่องมือนี้ไม่บล็อกเสมอ -- จะส่งคืนทันทีพร้อม `runId` และ
`childSessionKey`

ตัวเลือกหลัก:

- `runtime: "subagent"` (ค่าเริ่มต้น) หรือ `"acp"` สำหรับเอเจนต์ harness ภายนอก
- การแทนที่ `model` และ `thinking` สำหรับเซสชัน child
- `thread: true` เพื่อผูกการสร้างกับเธรดแชต (Discord, Slack ฯลฯ)
- `sandbox: "require"` เพื่อบังคับใช้ sandboxing กับ child
- `context: "fork"` สำหรับเอเจนต์ย่อยแบบเนทีฟเมื่อ child ต้องการทรานสคริปต์ผู้ขอปัจจุบัน
  หากต้องการ child ที่สะอาด ให้ละไว้หรือใช้ `context: "isolated"`

เอเจนต์ย่อยปลายทางเริ่มต้นจะไม่ได้รับเครื่องมือเซสชัน เมื่อ
`maxSpawnDepth >= 2` เอเจนต์ย่อย orchestrator ระดับ depth-1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ `sessions_history` เพิ่มเติม
เพื่อให้จัดการ child ของตนเองได้ การรันปลายทางยังคงไม่ได้รับ
เครื่องมือประสานงานแบบเรียกซ้ำ

หลังเสร็จงาน ขั้นตอนประกาศจะโพสต์ผลลัพธ์ไปยังแชนเนลของผู้ขอ
การส่งมอบเมื่อเสร็จงานจะรักษาเส้นทางเธรด/หัวข้อที่ผูกไว้เมื่อมี และหาก
ต้นทางของการเสร็จงานระบุเฉพาะแชนเนล OpenClaw ยังสามารถใช้เส้นทางที่เก็บไว้ของ
เซสชันผู้ขอ (`lastChannel` / `lastTo`) ซ้ำสำหรับการส่งมอบโดยตรงได้

สำหรับพฤติกรรมเฉพาะ ACP โปรดดู [เอเจนต์ ACP](/th/tools/acp-agents)

## การมองเห็น

เครื่องมือเซสชันถูกกำหนดขอบเขตเพื่อจำกัดสิ่งที่เอเจนต์มองเห็นได้:

| ระดับ   | ขอบเขต                                  |
| ------- | ---------------------------------------- |
| `self`  | เฉพาะเซสชันปัจจุบัน                     |
| `tree`  | เซสชันปัจจุบัน + เอเจนต์ย่อยที่สร้างขึ้น |
| `agent` | ทุกเซสชันสำหรับเอเจนต์นี้                |
| `all`   | ทุกเซสชัน (ข้ามเอเจนต์หากกำหนดค่าไว้)  |

ค่าเริ่มต้นคือ `tree` เซสชันที่ใช้ sandbox จะถูกจำกัดไว้ที่ `tree` ไม่ว่าการกำหนดค่าจะเป็นอย่างไร
