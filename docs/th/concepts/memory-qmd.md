[QMD](https://github.com/tobi/qmd) เป็น sidecar การค้นหาแบบให้ความสำคัญกับการทำงานภายในเครื่องก่อนที่ทำงาน
ควบคู่กับ OpenClaw โดยรวม BM25, การค้นหาแบบเวกเตอร์ และการจัดอันดับใหม่ไว้ใน
ไบนารีเดียว และสามารถทำดัชนีเนื้อหานอกเหนือจากไฟล์หน่วยความจำในเวิร์กสเปซของคุณได้

## สิ่งที่เพิ่มจากระบบในตัว

- **การจัดอันดับใหม่และการขยายคำค้น** เพื่อการเรียกคืนผลลัพธ์ที่ดีขึ้น
- **ทำดัชนีไดเรกทอรีเพิ่มเติม** -- เอกสารโปรเจกต์ โน้ตทีม หรืออะไรก็ได้บนดิสก์
- **ทำดัชนีทรานสคริปต์เซสชัน** -- เรียกคืนบทสนทนาก่อนหน้า
- **ภายในเครื่องทั้งหมด** -- ทำงานร่วมกับ Plugin ผู้ให้บริการ llama.cpp อย่างเป็นทางการและ
  ดาวน์โหลดโมเดล GGUF อัตโนมัติ
- **ทางเลือกสำรองอัตโนมัติ** -- หาก QMD ไม่พร้อมใช้งาน OpenClaw จะย้อนกลับไปใช้
  เอนจินในตัวได้อย่างราบรื่น

## เริ่มต้นใช้งาน

### ข้อกำหนดเบื้องต้น

- ติดตั้ง QMD: `npm install -g @tobilu/qmd` หรือ `bun install -g @tobilu/qmd`
- บิลด์ SQLite ที่อนุญาตส่วนขยาย (`brew install sqlite` บน macOS)
- QMD ต้องอยู่ใน `PATH` ของ Gateway
- macOS และ Linux ใช้งานได้ทันที Windows รองรับได้ดีที่สุดผ่าน WSL2

### เปิดใช้งาน

```json5
{
  memory: {
    backend: "qmd",
  },
}
```

OpenClaw จะสร้างโฮม QMD แบบครบในตัวภายใต้
`~/.openclaw/agents/<agentId>/qmd/` และจัดการวงจรชีวิตของ sidecar
โดยอัตโนมัติ -- คอลเลกชัน การอัปเดต และการรัน embedding จะถูกจัดการให้คุณ
ระบบจะเลือกใช้รูปแบบคอลเลกชัน QMD และรูปแบบคำค้น MCP ปัจจุบันก่อน แต่ยังคงย้อนกลับไปใช้
แฟล็กรูปแบบคอลเลกชันทางเลือกและชื่อเครื่องมือ MCP รุ่นเก่าเมื่อจำเป็น
การปรับให้สอดคล้องตอนบูตยังสร้างคอลเลกชันที่จัดการอยู่ซึ่งล้าสมัยกลับไปเป็น
รูปแบบมาตรฐานอีกครั้ง เมื่อยังมีคอลเลกชัน QMD รุ่นเก่าที่มีชื่อเดียวกันอยู่

## วิธีทำงานของ sidecar

- OpenClaw สร้างคอลเลกชันจากไฟล์หน่วยความจำในเวิร์กสเปซของคุณและ
  `memory.qmd.paths` ใดๆ ที่กำหนดค่าไว้ จากนั้นรัน `qmd update` เมื่อเปิดตัวจัดการ QMD
  และเป็นระยะหลังจากนั้น (ค่าเริ่มต้นทุก 5 นาที) การรีเฟรชเหล่านี้
  ทำงานผ่านซับโปรเซสของ QMD ไม่ใช่การไล่สำรวจระบบไฟล์ในโปรเซสเดียวกัน โหมดเชิงความหมาย
  จะรัน `qmd embed` ด้วย
- คอลเลกชันเวิร์กสเปซเริ่มต้นติดตาม `MEMORY.md` รวมถึงโครงสร้าง `memory/`
  `memory.md` ตัวพิมพ์เล็กจะไม่ถูกทำดัชนีเป็นไฟล์หน่วยความจำราก
- ตัวสแกนของ QMD เองจะละเว้นพาธที่ซ่อนอยู่และไดเรกทอรี dependency/build
  ที่พบบ่อย เช่น `.git`, `.cache`, `node_modules`, `vendor`, `dist` และ
  `build` การเริ่มต้น Gateway จะไม่เริ่ม QMD ตามค่าเริ่มต้น ดังนั้นการบูตแบบ cold boot
  จะหลีกเลี่ยงการนำเข้ารันไทม์หน่วยความจำหรือการสร้าง watcher ที่มีอายุยาวก่อน
  มีการใช้หน่วยความจำครั้งแรก
- หากคุณต้องการให้เริ่ม QMD ตอนเริ่ม Gateway อยู่ดี ให้ตั้งค่า
  `memory.qmd.update.startup` เป็น `idle` หรือ `immediate` เมื่อใช้
  `memory.qmd.update.onBoot: true` การเริ่มต้นจะรันการรีเฟรชครั้งแรก เมื่อใช้
  `onBoot: false` การเริ่มต้นจะข้ามการรีเฟรชทันทีนั้น แต่ยังคงเปิด
  ตัวจัดการที่มีอายุยาวเมื่อมีการกำหนดค่าช่วงเวลา update หรือ embed เพื่อให้ QMD
  เป็นเจ้าของ watcher และตัวจับเวลาปกติของตัวเองได้
- การค้นหาใช้ `searchMode` ที่กำหนดค่าไว้ (ค่าเริ่มต้น: `search`; รองรับ
  `vsearch` และ `query` ด้วย) `search` เป็น BM25 เท่านั้น ดังนั้น OpenClaw จะข้าม
  probe ความพร้อมของเวกเตอร์เชิงความหมายและการดูแล embedding ในโหมดนั้น หากโหมดหนึ่ง
  ล้มเหลว OpenClaw จะลองใหม่ด้วย `qmd query`
- เมื่อ `searchMode` เป็น `query` ให้ตั้งค่า `memory.qmd.rerank` เป็น `false` เพื่อใช้
  เส้นทางคำค้นแบบไฮบริดของ QMD โดยไม่ใช้ reranker OpenClaw ส่ง `--no-rerank` ไปยัง
  เส้นทาง QMD CLI โดยตรง และส่ง `rerank: false` ไปยังเครื่องมือคำค้น MCP ของ QMD ตัวเลือกนี้
  ต้องใช้ QMD 2.1 หรือใหม่กว่า
- สำหรับรุ่น QMD ที่ประกาศตัวกรองหลายคอลเลกชัน OpenClaw จะจัดกลุ่ม
  คอลเลกชันที่มาจากแหล่งเดียวกันให้เป็นการเรียกค้นหา QMD ครั้งเดียว รุ่น QMD ที่เก่ากว่า
  จะยังคงใช้ทางเลือกสำรองแบบต่อคอลเลกชันที่เข้ากันได้
- หาก QMD ล้มเหลวทั้งหมด OpenClaw จะย้อนกลับไปใช้เอนจิน SQLite ในตัว
  ความพยายามซ้ำระหว่างรอบแชตจะหน่วงถอยหลังสั้นๆ หลังการเปิดล้มเหลว เพื่อให้
  ไบนารีที่ขาดหายหรือ dependency ของ sidecar ที่เสียไม่สร้างการลองซ้ำจำนวนมาก;
  `openclaw memory status` และ probe CLI แบบครั้งเดียวจะยังตรวจสอบ QMD โดยตรงอีกครั้ง

<Info>
การค้นหาครั้งแรกอาจช้า -- QMD ดาวน์โหลดโมเดล GGUF อัตโนมัติ (~2 GB) สำหรับ
การจัดอันดับใหม่และการขยายคำค้นเมื่อรัน `qmd query` ครั้งแรก
</Info>

## ประสิทธิภาพการค้นหาและความเข้ากันได้

OpenClaw รักษาเส้นทางการค้นหา QMD ให้เข้ากันได้กับทั้งการติดตั้ง QMD ปัจจุบันและรุ่นเก่า

เมื่อเริ่มต้น OpenClaw จะตรวจข้อความช่วยเหลือของ QMD ที่ติดตั้งไว้หนึ่งครั้งต่อหนึ่งตัวจัดการ หาก
ไบนารีประกาศว่ารองรับตัวกรองหลายคอลเลกชัน OpenClaw จะค้นหาคอลเลกชันทั้งหมด
ที่มาจากแหล่งเดียวกันด้วยคำสั่งเดียว:

```bash
qmd search "router notes" --json -n 10 -c memory-root-main -c memory-dir-main
```

วิธีนี้หลีกเลี่ยงการเริ่มซับโปรเซส QMD หนึ่งตัวสำหรับทุกคอลเลกชันหน่วยความจำถาวร
คอลเลกชันทรานสคริปต์เซสชันจะอยู่ในกลุ่มแหล่งที่มาของตัวเอง ดังนั้นการค้นหาแบบผสม
`memory` + `sessions` ยังคงให้ข้อมูลอินพุตจากทั้งสองแหล่งแก่ตัวกระจายผลลัพธ์

บิลด์ QMD รุ่นเก่ารับตัวกรองคอลเลกชันได้เพียงหนึ่งรายการ เมื่อ OpenClaw ตรวจพบ
บิลด์เหล่านั้น จะคงเส้นทางความเข้ากันได้ไว้และค้นหาแต่ละคอลเลกชัน
แยกกันก่อนรวมและลบผลลัพธ์ซ้ำ

หากต้องการตรวจสัญญาที่ติดตั้งด้วยตนเอง ให้รัน:

```bash
qmd --help | grep -i collection
```

ข้อความช่วยเหลือ QMD ปัจจุบันระบุว่าตัวกรองคอลเลกชันสามารถกำหนดเป้าหมายหนึ่งคอลเลกชันหรือมากกว่า
ข้อความช่วยเหลือรุ่นเก่ามักอธิบายคอลเลกชันเดียว

## การแทนที่โมเดล

ตัวแปรสภาพแวดล้อมของโมเดล QMD จะส่งผ่านจากโปรเซส Gateway โดยไม่เปลี่ยนแปลง
ดังนั้นคุณจึงปรับแต่ง QMD แบบทั่วทั้งระบบได้โดยไม่ต้องเพิ่มการกำหนดค่า OpenClaw ใหม่:

```bash
export QMD_EMBED_MODEL="hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"
export QMD_RERANK_MODEL="/absolute/path/to/reranker.gguf"
export QMD_GENERATE_MODEL="/absolute/path/to/generator.gguf"
```

หลังจากเปลี่ยนโมเดล embedding ให้รัน embeddings อีกครั้งเพื่อให้ดัชนีตรงกับ
ปริภูมิเวกเตอร์ใหม่

## การทำดัชนีพาธเพิ่มเติม

ชี้ QMD ไปยังไดเรกทอรีเพิ่มเติมเพื่อให้ค้นหาได้:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

Snippet จากพาธเพิ่มเติมจะแสดงเป็น `qmd/<collection>/<relative-path>` ใน
ผลการค้นหา `memory_get` เข้าใจคำนำหน้านี้และอ่านจากรากคอลเลกชันที่ถูกต้อง

## การทำดัชนีทรานสคริปต์เซสชัน

เปิดใช้งานการทำดัชนีเซสชันเพื่อเรียกคืนบทสนทนาก่อนหน้า QMD ต้องใช้ทั้งแหล่งเซสชันทั่วไป
`memorySearch` และตัวส่งออกทรานสคริปต์ของ QMD:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        experimental: { sessionMemory: true },
        sources: ["memory", "sessions"],
      },
    },
  },
  memory: {
    backend: "qmd",
    qmd: {
      sessions: { enabled: true },
    },
  },
}
```

ทรานสคริปต์จะถูกส่งออกเป็นเทิร์น User/Assistant ที่ผ่านการล้างข้อมูลแล้วไปยังคอลเลกชัน QMD
เฉพาะภายใต้ `~/.openclaw/agents/<id>/qmd/sessions/` การตั้งค่าเฉพาะ
`memorySearch.experimental.sessionMemory` จะไม่ส่งออกทรานสคริปต์เข้า QMD

ผลลัพธ์จากเซสชันยังคงถูกกรองด้วย
[`tools.sessions.visibility`](/th/gateway/config-tools#toolssessions) ค่าเริ่มต้น
การมองเห็น `tree` จะไม่เปิดเผยเซสชันของ agent เดียวกันที่ไม่เกี่ยวข้อง หาก
เซสชันที่ส่งผ่าน Gateway ควรถูกเรียกคืนได้จากเซสชัน DM แยกต่างหาก ให้ตั้งค่า
`tools.sessions.visibility: "agent"` อย่างตั้งใจ

## ขอบเขตการค้นหา

ตามค่าเริ่มต้น ผลการค้นหา QMD จะแสดงในเซสชันโดยตรงและเซสชันช่องทาง
(ไม่ใช่กลุ่ม) กำหนดค่า `memory.qmd.scope` เพื่อเปลี่ยนสิ่งนี้:

```json5
{
  memory: {
    qmd: {
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
    },
  },
}
```

เมื่อขอบเขตปฏิเสธการค้นหา OpenClaw จะบันทึกคำเตือนพร้อมช่องทางและ
ประเภทแชตที่อนุมานได้ เพื่อให้ดีบักผลลัพธ์ว่างได้ง่ายขึ้น

## การอ้างอิง

เมื่อ `memory.citations` เป็น `auto` หรือ `on` snippet การค้นหาจะรวม
ส่วนท้าย `Source: <path#line>` ตั้งค่า `memory.citations = "off"` เพื่อไม่แสดงส่วนท้าย
แต่ยังส่งพาธให้ agent ภายใน

## ควรใช้เมื่อใด

เลือก QMD เมื่อคุณต้องการ:

- การจัดอันดับใหม่เพื่อผลลัพธ์คุณภาพสูงขึ้น
- ค้นหาเอกสารโปรเจกต์หรือโน้ตนอกเวิร์กสเปซ
- เรียกคืนบทสนทนาเซสชันที่ผ่านมา
- การค้นหาภายในเครื่องทั้งหมดโดยไม่ต้องใช้ API keys

สำหรับการตั้งค่าที่เรียบง่ายกว่า [เอนจินในตัว](/th/concepts/memory-builtin) ใช้งานได้ดี
โดยไม่มี dependency เพิ่มเติม

## การแก้ปัญหา

**ไม่พบ QMD?** ตรวจสอบให้แน่ใจว่าไบนารีอยู่ใน `PATH` ของ Gateway หาก OpenClaw
ทำงานเป็นบริการ ให้สร้าง symlink:
`sudo ln -s ~/.bun/bin/qmd /usr/local/bin/qmd`.

หาก `qmd --version` ใช้งานได้ใน shell ของคุณ แต่ OpenClaw ยังคงรายงาน
`spawn qmd ENOENT` โปรเซส Gateway น่าจะมี `PATH` ต่างจาก
shell แบบโต้ตอบของคุณ ปักหมุดไบนารีอย่างชัดเจน:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      command: "/absolute/path/to/qmd",
    },
  },
}
```

ใช้ `command -v qmd` ในสภาพแวดล้อมที่ติดตั้ง QMD จากนั้นตรวจซ้ำ
ด้วย `openclaw memory status --deep`

**การค้นหาครั้งแรกช้ามาก?** QMD ดาวน์โหลดโมเดล GGUF เมื่อใช้ครั้งแรก อุ่นเครื่องล่วงหน้า
ด้วย `qmd query "test"` โดยใช้ไดเรกทอรี XDG เดียวกับที่ OpenClaw ใช้

**มีซับโปรเซส QMD จำนวนมากระหว่างค้นหา?** อัปเดต QMD หากทำได้ OpenClaw ใช้
โปรเซสเดียวสำหรับการค้นหาหลายคอลเลกชันจากแหล่งเดียวกันเฉพาะเมื่อ QMD ที่ติดตั้งไว้
ประกาศว่ารองรับตัวกรอง `-c` หลายรายการ; มิฉะนั้นจะคงทางเลือกสำรองรุ่นเก่า
แบบต่อคอลเลกชันไว้เพื่อความถูกต้อง

**QMD แบบ BM25 เท่านั้นยังพยายาม build llama.cpp?** ตั้งค่า
`memory.qmd.searchMode = "search"` OpenClaw ถือว่าโหมดนั้นเป็นแบบ lexical-only,
ไม่รัน probe สถานะเวกเตอร์ QMD หรือการดูแล embedding และปล่อยให้
การตรวจความพร้อมเชิงความหมายเป็นหน้าที่ของการตั้งค่า `vsearch` หรือ `query`

**การค้นหาหมดเวลา?** เพิ่ม `memory.qmd.limits.timeoutMs` (ค่าเริ่มต้น: 4000ms)
ตั้งเป็น `120000` สำหรับฮาร์ดแวร์ที่ช้ากว่า

**ผลลัพธ์ว่างในแชตกลุ่ม?** ตรวจ `memory.qmd.scope` -- ค่าเริ่มต้นอนุญาตเฉพาะ
เซสชันโดยตรงและเซสชันช่องทาง

**การค้นหาหน่วยความจำรากกว้างเกินไปอย่างกะทันหัน?** รีสตาร์ต Gateway หรือรอ
การปรับให้สอดคล้องตอนเริ่มต้นครั้งถัดไป OpenClaw สร้างคอลเลกชันที่จัดการอยู่ซึ่งล้าสมัย
กลับไปเป็นรูปแบบมาตรฐาน `MEMORY.md` และ `memory/` เมื่อตรวจพบความขัดแย้ง
ของชื่อเดียวกัน

**รีโปชั่วคราวที่มองเห็นได้จากเวิร์กสเปซทำให้เกิด `ENAMETOOLONG` หรือการทำดัชนีเสีย?**
การไล่สำรวจของ QMD ในปัจจุบันทำตามพฤติกรรมตัวสแกน QMD พื้นฐาน แทนที่จะเป็น
กฎ symlink ในตัวของ OpenClaw เก็บ checkout monorepo ชั่วคราวไว้ภายใต้
ไดเรกทอรีที่ซ่อน เช่น `.tmp/` หรือนอกราก QMD ที่ทำดัชนีไว้ จนกว่า QMD จะเปิดเผย
การไล่สำรวจที่ปลอดภัยต่อ cycle หรือการควบคุมการยกเว้นอย่างชัดเจน
