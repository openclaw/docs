[QMD](https://github.com/tobi/qmd) คือโปรแกรมเสริมการค้นหาแบบเน้นการทำงานในเครื่อง ซึ่งทำงาน
ควบคู่กับ OpenClaw โดยรวม BM25, การค้นหาแบบเวกเตอร์ และการจัดอันดับซ้ำไว้ใน
ไบนารีเดียว และสามารถทำดัชนีเนื้อหาที่อยู่นอกเหนือไฟล์หน่วยความจำในพื้นที่ทำงานได้

## สิ่งที่เพิ่มจากระบบในตัว

- **การจัดอันดับซ้ำและการขยายคำค้น** เพื่อเพิ่มความครอบคลุมในการค้นคืน
- **ทำดัชนีไดเรกทอรีเพิ่มเติม** - เอกสารโครงการ บันทึกของทีม หรือทุกอย่างบนดิสก์
- **ทำดัชนีบันทึกการถอดความเซสชัน** - เรียกคืนบทสนทนาก่อนหน้า
- **ทำงานในเครื่องทั้งหมด** - ทำงานร่วมกับ Plugin ผู้ให้บริการ llama.cpp อย่างเป็นทางการ และ
  ดาวน์โหลดโมเดล GGUF โดยอัตโนมัติ
- **การถอยกลับอัตโนมัติ** - หาก QMD ใช้งานไม่ได้ OpenClaw จะถอยกลับไปใช้
  เอนจินในตัวอย่างราบรื่น

## เริ่มต้นใช้งาน

### ข้อกำหนดเบื้องต้น

- ติดตั้ง QMD: `npm install -g @tobilu/qmd` หรือ `bun install -g @tobilu/qmd`
- บิลด์ SQLite ที่อนุญาตส่วนขยาย (`brew install sqlite` บน macOS)
- QMD ต้องอยู่ใน `PATH` ของ Gateway
- macOS และ Linux ใช้งานได้ทันที ส่วน Windows รองรับได้ดีที่สุดผ่าน WSL2

### เปิดใช้งาน

```json5
{
  memory: {
    backend: "qmd",
  },
}
```

OpenClaw สร้างโฮม QMD แบบครบถ้วนในตัวเองภายใต้
`~/.openclaw/agents/<agentId>/qmd/` และจัดการวงจรชีวิตของโปรแกรมเสริม
โดยอัตโนมัติ ทั้งคอลเลกชัน การอัปเดต และการฝังข้อมูลจะได้รับการจัดการให้
ระบบจะเลือกใช้รูปแบบคอลเลกชันและคำค้น MCP ของ QMD ปัจจุบัน แต่จะถอยกลับไปใช้
แฟล็กรูปแบบคอลเลกชันทางเลือกและชื่อเครื่องมือ MCP รุ่นเก่าเมื่อจำเป็น
การปรับให้สอดคล้องเมื่อเริ่มทำงานยังสร้างคอลเลกชันที่จัดการอยู่แต่ล้าสมัยขึ้นใหม่ให้กลับไปใช้
รูปแบบมาตรฐาน เมื่อยังมีคอลเลกชัน QMD รุ่นเก่าที่ใช้ชื่อเดียวกันอยู่

## วิธีทำงานของโปรแกรมเสริม

- OpenClaw สร้างคอลเลกชันจากไฟล์หน่วยความจำในพื้นที่ทำงานและ
  `memory.qmd.paths` ที่กำหนดค่าไว้ จากนั้นเรียกใช้ `qmd update` เมื่อโปรแกรมจัดการ QMD
  เปิดขึ้นและทำซ้ำเป็นระยะหลังจากนั้น (`memory.qmd.update.interval` ค่าเริ่มต้นคือ
  `5m`) การรีเฟรชทำงานผ่านโพรเซสย่อยของ QMD ไม่ใช่การไล่สำรวจระบบไฟล์
  ภายในโพรเซส โหมดการค้นหาเชิงความหมายยังเรียกใช้ `qmd embed`
  (`memory.qmd.update.embedInterval` ค่าเริ่มต้นคือ `60m`)
- คอลเลกชันพื้นที่ทำงานเริ่มต้นติดตาม `MEMORY.md` รวมถึงโครงสร้าง
  `memory/` ส่วน `memory.md` ที่เป็นตัวพิมพ์เล็กจะไม่ถูกทำดัชนีเป็นไฟล์หน่วยความจำราก
- ตัวสแกนของ QMD จะละเว้นพาธที่ซ่อนอยู่และไดเรกทอรีการขึ้นต่อกัน/การบิลด์
  ที่พบบ่อย เช่น `.git`, `.cache`, `node_modules`, `vendor`, `dist` และ
  `build` โดยค่าเริ่มต้น การเริ่มทำงานของ Gateway จะไม่เริ่มต้น QMD
  (`memory.qmd.update.startup` มีค่าเริ่มต้นเป็น `off`) ดังนั้นการบูตแบบเย็นจึงหลีกเลี่ยง
  การนำเข้ารันไทม์หน่วยความจำหรือการสร้างตัวเฝ้าดูที่ทำงานระยะยาว ก่อนมีการใช้
  หน่วยความจำครั้งแรก
- ตั้งค่า `memory.qmd.update.startup` เป็น `idle` หรือ `immediate` เพื่อเริ่มต้น QMD
  เมื่อ Gateway เริ่มทำงานอยู่ดี `memory.qmd.update.onBoot` มีค่าเริ่มต้นเป็น `true` และ
  เรียกใช้การรีเฟรชครั้งแรกเมื่อเริ่มทำงาน ให้ตั้งเป็น `false` เพื่อข้าม
  การรีเฟรชทันทีดังกล่าว (โปรแกรมจัดการที่ทำงานระยะยาวยังคงเปิดขึ้นเมื่อมีการกำหนด
  ช่วงเวลาการอัปเดตหรือการฝังข้อมูล ดังนั้น QMD จึงยังคงเป็นผู้ควบคุมตัวเฝ้าดู/ตัวจับเวลาตามปกติ)
- การค้นหาใช้ `searchMode` ที่กำหนดค่าไว้ (ค่าเริ่มต้น: `search`; รองรับ
  `vsearch` และ `query` ด้วย) `search` ใช้เฉพาะ BM25 ดังนั้น OpenClaw จึงข้าม
  การตรวจสอบความพร้อมของเวกเตอร์เชิงความหมายและการบำรุงรักษาการฝังข้อมูลในโหมดนั้น หากโหมดหนึ่ง
  ล้มเหลว OpenClaw จะลองใหม่ด้วย `qmd query`
- เมื่อ `searchMode` เป็น `query` ให้ตั้งค่า `memory.qmd.rerank` เป็น `false` เพื่อใช้
  เส้นทางคำค้นแบบไฮบริดของ QMD โดยไม่มีตัวจัดอันดับซ้ำ (ต้องใช้ QMD 2.1 หรือใหม่กว่า)
  OpenClaw ส่ง `--no-rerank` ไปยังเส้นทาง QMD CLI โดยตรง และ
  `rerank: false` ไปยังเครื่องมือคำค้น MCP ของ QMD
- สำหรับ QMD รุ่นที่ประกาศว่ารองรับตัวกรองหลายคอลเลกชัน OpenClaw จะจัดกลุ่ม
  คอลเลกชันจากแหล่งเดียวกันไว้ในการเรียกค้น QMD ครั้งเดียว ส่วน QMD รุ่นเก่า
  จะยังคงใช้การถอยกลับแบบแยกคอลเลกชันที่เข้ากันได้
- หาก QMD ล้มเหลวทั้งหมด OpenClaw จะถอยกลับไปใช้เอนจิน SQLite ในตัว
  ความพยายามซ้ำในแต่ละรอบแชตจะหน่วงระยะสั้นหลังการเปิดล้มเหลว เพื่อไม่ให้
  ไบนารีที่หายไปหรือการขึ้นต่อกันของโปรแกรมเสริมที่เสียหายก่อให้เกิดการลองซ้ำอย่างถี่รัว
  ส่วน `openclaw memory status` และการตรวจสอบ CLI แบบครั้งเดียวจะยังคงตรวจสอบ QMD
  โดยตรงอีกครั้ง

<Info>
การค้นหาครั้งแรกอาจช้า เนื่องจาก QMD จะดาวน์โหลดโมเดล GGUF (~2 GB) โดยอัตโนมัติสำหรับ
การจัดอันดับซ้ำและการขยายคำค้นในการเรียกใช้ `qmd query` ครั้งแรก
</Info>

## ประสิทธิภาพและความเข้ากันได้ของการค้นหา

OpenClaw รักษาเส้นทางการค้นหา QMD ให้เข้ากันได้กับทั้งการติดตั้ง QMD
รุ่นปัจจุบันและรุ่นเก่า

เมื่อเริ่มทำงาน OpenClaw จะตรวจสอบข้อความวิธีใช้ของ QMD ที่ติดตั้งหนึ่งครั้งต่อโปรแกรมจัดการ หาก
ไบนารีประกาศว่ารองรับตัวกรองหลายคอลเลกชัน OpenClaw จะ
ค้นหาคอลเลกชันทั้งหมดจากแหล่งเดียวกันด้วยคำสั่งเดียว:

```bash
qmd search "router notes" --json -n 10 -c memory-root-main -c memory-dir-main
```

วิธีนี้ช่วยหลีกเลี่ยงการเริ่มโพรเซสย่อย QMD หนึ่งรายการต่อคอลเลกชันหน่วยความจำถาวร
คอลเลกชันบันทึกการถอดความเซสชันจะอยู่ในกลุ่มแหล่งที่มาของตนเอง ดังนั้นการค้นหาแบบผสม
`memory` + `sessions` จึงยังส่งข้อมูลจากทั้งสองแหล่งให้
ตัวกระจายผลลัพธ์

บิลด์ QMD รุ่นเก่ารองรับตัวกรองคอลเลกชันได้เพียงหนึ่งรายการ เมื่อ OpenClaw ตรวจพบ
บิลด์ดังกล่าว ระบบจะคงเส้นทางความเข้ากันได้ไว้และค้นหาแต่ละคอลเลกชัน
แยกกัน ก่อนรวมและขจัดผลลัพธ์ที่ซ้ำกัน

หากต้องการตรวจสอบสัญญาของรุ่นที่ติดตั้งด้วยตนเอง ให้เรียกใช้:

```bash
qmd --help | grep -i collection
```

ข้อความวิธีใช้ของ QMD ปัจจุบันระบุถึงการกำหนดเป้าหมายคอลเลกชันหนึ่งรายการหรือมากกว่า ส่วนข้อความวิธีใช้รุ่นเก่า
มักอธิบายถึงคอลเลกชันเดียว

## การแทนที่โมเดล

ตัวแปรสภาพแวดล้อมของโมเดล QMD จะถูกส่งผ่านจากโพรเซส Gateway โดยไม่เปลี่ยนแปลง
จึงสามารถปรับแต่ง QMD โดยรวมได้โดยไม่ต้องเพิ่มการกำหนดค่า OpenClaw ใหม่:

```bash
export QMD_EMBED_MODEL="hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"
export QMD_RERANK_MODEL="/absolute/path/to/reranker.gguf"
export QMD_GENERATE_MODEL="/absolute/path/to/generator.gguf"
```

หลังจากเปลี่ยนโมเดลการฝังข้อมูล ให้เรียกใช้การฝังข้อมูลอีกครั้งเพื่อให้ดัชนีตรงกับ
ปริภูมิเวกเตอร์ใหม่

## การทำดัชนีพาธเพิ่มเติม

กำหนดให้ QMD ชี้ไปยังไดเรกทอรีเพิ่มเติมเพื่อให้สามารถค้นหาได้:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

ส่วนย่อจากพาธเพิ่มเติมจะปรากฏเป็น `qmd/<collection>/<relative-path>` ใน
ผลการค้นหา `memory_get` เข้าใจคำนำหน้านี้และอ่านจาก
รากของคอลเลกชันที่ถูกต้อง

## การทำดัชนีบันทึกการถอดความเซสชัน

เปิดใช้การทำดัชนีเซสชันเพื่อเรียกคืนบทสนทนาก่อนหน้า QMD ต้องใช้ทั้ง
แหล่งเซสชันทั่วไป `memorySearch` และตัวส่งออกบันทึกการถอดความ QMD:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        experimental: { sessionMemory: true },
        sources: ["memory", "sessions"],
      },
    },
  },
  memory: {
    backend: "qmd",
    qmd: {
      sessions: { enabled: true },
    },
  },
}
```

บันทึกการถอดความจะถูกส่งออกเป็นรอบสนทนา User/Assistant ที่ผ่านการล้างข้อมูลแล้ว ไปยังคอลเลกชัน QMD
เฉพาะภายใต้ `~/.openclaw/agents/<id>/qmd/sessions/` การตั้งค่าเฉพาะ
`memorySearch.experimental.sessionMemory` จะไม่ส่งออกบันทึกการถอดความไปยัง
QMD

ผลลัพธ์เซสชันยังคงถูกกรองด้วย
[`tools.sessions.visibility`](/th/gateway/config-tools#toolssessions) การมองเห็น
`tree` เริ่มต้นจะไม่เปิดเผยเซสชันอื่นที่ไม่เกี่ยวข้องของเอเจนต์เดียวกัน หากต้องการให้
เซสชันที่ Gateway จัดส่งสามารถเรียกคืนได้จากเซสชัน DM แยกต่างหาก
ให้ตั้งค่า `tools.sessions.visibility: "agent"` โดยเจตนา

## ขอบเขตการค้นหา

โดยค่าเริ่มต้น ผลการค้นหา QMD จะแสดงเฉพาะในเซสชันโดยตรงเท่านั้น (ไม่ใช่
แชตกลุ่มหรือแชตช่องทาง) กำหนดค่า `memory.qmd.scope` เพื่อเปลี่ยนแปลงสิ่งนี้:

```json5
{
  memory: {
    qmd: {
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
    },
  },
}
```

ส่วนย่อข้างต้นคือกฎเริ่มต้นจริง เมื่อขอบเขตปฏิเสธการค้นหา
OpenClaw จะบันทึกคำเตือนพร้อมช่องทางและประเภทแชตที่อนุมานได้ เพื่อให้
แก้จุดบกพร่องของผลลัพธ์ว่างได้ง่ายขึ้น

## การอ้างอิง

เมื่อ `memory.citations` เป็น `auto` หรือ `on` ส่วนย่อการค้นหาจะได้รับการต่อท้ายด้วย
ส่วนท้าย `Source: <path>#L<line>` (หรือ `#L<start>-L<end>`) ในโหมด `auto`
ส่วนท้ายจะถูกเพิ่มเฉพาะสำหรับเซสชันแชตโดยตรง ตั้งค่า
`memory.citations = "off"` เพื่อละเว้นส่วนท้าย แต่ยังคงส่งพาธให้
เอเจนต์ภายใน

## ควรใช้เมื่อใด

เลือก QMD เมื่อต้องการ:

- การจัดอันดับซ้ำเพื่อผลลัพธ์ที่มีคุณภาพสูงขึ้น
- ค้นหาเอกสารโครงการหรือบันทึกนอกพื้นที่ทำงาน
- เรียกคืนบทสนทนาจากเซสชันที่ผ่านมา
- การค้นหาในเครื่องทั้งหมดโดยไม่ใช้คีย์ API

สำหรับการตั้งค่าที่เรียบง่ายกว่า [เอนจินในตัว](/th/concepts/memory-builtin) ทำงานได้ดี
โดยไม่ต้องมีการขึ้นต่อกันเพิ่มเติม

## การแก้ไขปัญหา

**ไม่พบ QMD?** ตรวจสอบว่าไบนารีอยู่ใน `PATH` ของ Gateway หาก OpenClaw
ทำงานเป็นบริการ ให้สร้างลิงก์เชิงสัญลักษณ์:
`sudo ln -s ~/.bun/bin/qmd /usr/local/bin/qmd`

หาก `qmd --version` ทำงานในเชลล์ แต่ OpenClaw ยังคงรายงาน
`spawn qmd ENOENT` โพรเซส Gateway น่าจะมี `PATH` ต่างจาก
เชลล์แบบโต้ตอบ ให้ระบุไบนารีอย่างชัดเจน:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      command: "/absolute/path/to/qmd",
    },
  },
}
```

ใช้ `command -v qmd` ในสภาพแวดล้อมที่ติดตั้ง QMD แล้วตรวจสอบอีกครั้ง
ด้วย `openclaw memory status --deep`

**การค้นหาครั้งแรกช้ามาก?** QMD ดาวน์โหลดโมเดล GGUF เมื่อใช้งานครั้งแรก เตรียมระบบล่วงหน้า
ด้วย `qmd query "test"` โดยใช้ไดเรกทอรี XDG เดียวกับที่ OpenClaw ใช้

**มีโพรเซสย่อย QMD จำนวนมากระหว่างการค้นหา?** อัปเดต QMD หากทำได้ OpenClaw
ใช้โพรเซสเดียวสำหรับการค้นหาหลายคอลเลกชันจากแหล่งเดียวกัน เฉพาะเมื่อ
QMD ที่ติดตั้งประกาศว่ารองรับตัวกรอง `-c` หลายรายการ มิฉะนั้น
ระบบจะคงการถอยกลับแบบแยกคอลเลกชันรุ่นเก่าไว้เพื่อความถูกต้อง

**QMD แบบ BM25 เท่านั้นยังคงพยายามบิลด์ llama.cpp?** ตั้งค่า
`memory.qmd.searchMode = "search"` OpenClaw ถือว่าโหมดดังกล่าวเป็น
แบบคำศัพท์เท่านั้น ข้ามการตรวจสอบสถานะเวกเตอร์ QMD และการบำรุงรักษาการฝังข้อมูล และ
ปล่อยให้การตรวจสอบความพร้อมเชิงความหมายเป็นหน้าที่ของการตั้งค่า `vsearch` หรือ `query`

**การค้นหาหมดเวลา?** เพิ่ม `memory.qmd.limits.timeoutMs` (ค่าเริ่มต้น: 4000ms)
ตั้งค่าให้สูงขึ้น เช่น `120000` สำหรับฮาร์ดแวร์ที่ช้ากว่า ขีดจำกัดนี้ใช้กับ
คำสั่งค้นหาของ QMD เองระหว่างการเรียก `memory_search` ของเอเจนต์ ส่วนการตั้งค่า การซิงค์
การถอยกลับไปใช้ระบบในตัว และงานคลังข้อมูลเสริมจะยังคงใช้เส้นตายที่สั้นกว่าของตนเอง

**ผลลัพธ์ว่างในแชตกลุ่มหรือแชตช่องทาง?** นี่เป็นพฤติกรรมที่คาดไว้เมื่อใช้
`memory.qmd.scope` เริ่มต้น ซึ่งอนุญาตเฉพาะเซสชันโดยตรง เพิ่มกฎ
`allow` สำหรับประเภทแชต `group` หรือ `channel` หากต้องการผลลัพธ์ QMD
ในบริบทดังกล่าว

**การค้นหาหน่วยความจำรากกว้างเกินไปอย่างกะทันหัน?** เริ่ม Gateway ใหม่หรือรอ
การปรับให้สอดคล้องในการเริ่มทำงานครั้งถัดไป OpenClaw จะสร้างคอลเลกชันที่จัดการอยู่แต่ล้าสมัย
ขึ้นใหม่ให้กลับไปใช้รูปแบบมาตรฐาน `MEMORY.md` และ `memory/` เมื่อตรวจพบ
ความขัดแย้งของชื่อเดียวกัน

**ที่เก็บชั่วคราวที่มองเห็นได้ในพื้นที่ทำงานทำให้เกิด `ENAMETOOLONG` หรือทำดัชนีเสียหาย?**
การไล่สำรวจของ QMD ทำตามตัวสแกน QMD ที่อยู่เบื้องหลัง แทนที่จะใช้
กฎลิงก์เชิงสัญลักษณ์ในตัวของ OpenClaw เก็บสำเนาการเช็กเอาต์ monorepo ชั่วคราวไว้ใน
ไดเรกทอรีที่ซ่อนอยู่ เช่น `.tmp/` หรือนอกราก QMD ที่ทำดัชนี จนกว่า QMD จะมี
การไล่สำรวจที่ปลอดภัยจากวงจรหรือการควบคุมการยกเว้นอย่างชัดเจน

## การกำหนดค่า

กฎขอบเขต และตัวเลือกอื่นทั้งหมด โปรดดู
[ข้อมูลอ้างอิงการกำหนดค่าหน่วยความจำ](/th/reference/memory-config)
