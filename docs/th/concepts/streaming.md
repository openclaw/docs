OpenClaw มีเลเยอร์การสตรีมแยกกันสองชั้น:

- **การสตรีมแบบบล็อก (ช่องทาง):** ส่ง **บล็อก** ที่เสร็จสมบูรณ์ขณะที่ผู้ช่วยเขียนอยู่ ข้อความเหล่านี้เป็นข้อความช่องทางปกติ (ไม่ใช่เดลตาโทเคน)
- **การสตรีมพรีวิว (Telegram/Discord/Slack):** อัปเดต **ข้อความพรีวิว** ชั่วคราวระหว่างการสร้างคำตอบ

ปัจจุบันยังไม่มี **การสตรีมเดลตาโทเคนจริง** ไปยังข้อความช่องทาง การสตรีมพรีวิวอิงตามข้อความ (ส่ง + แก้ไข/ต่อท้าย)

## การสตรีมแบบบล็อก (ข้อความช่องทาง)

การสตรีมแบบบล็อกส่งเอาต์พุตของผู้ช่วยเป็นส่วนขนาดหยาบเมื่อพร้อมใช้งาน

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

คำอธิบายสัญลักษณ์:

- `text_delta/events`: เหตุการณ์สตรีมของโมเดล (อาจมีน้อยสำหรับโมเดลที่ไม่สตรีม)
- `chunker`: `EmbeddedBlockChunker` ที่ใช้ขอบเขตต่ำสุด/สูงสุด + ค่ากำหนดจุดตัด
- `channel send`: ข้อความขาออกจริง (การตอบกลับแบบบล็อก)

**การควบคุม:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (ค่าเริ่มต้นปิด)
- การแทนที่ของช่องทาง: `*.blockStreaming` (และตัวแปรรายบัญชี) เพื่อบังคับ `"on"`/`"off"` ต่อช่องทาง
- `agents.defaults.blockStreamingBreak`: `"text_end"` หรือ `"message_end"`
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (รวมบล็อกที่สตรีมมาก่อนส่ง)
- เพดานบังคับของช่องทาง: `*.textChunkLimit` (เช่น `channels.whatsapp.textChunkLimit`)
- โหมดการแบ่งชิ้นของช่องทาง: `*.chunkMode` (ค่าเริ่มต้น `length`, `newline` แบ่งตามบรรทัดว่าง (ขอบเขตย่อหน้า) ก่อนแบ่งตามความยาว)
- เพดานแบบอ่อนของ Discord: `channels.discord.maxLinesPerMessage` (ค่าเริ่มต้น 17) แบ่งคำตอบที่สูงมากเพื่อหลีกเลี่ยงการถูกตัดใน UI

**ความหมายของขอบเขต:**

- `text_end`: สตรีมบล็อกทันทีที่ตัวแบ่งชิ้นส่งออกมา; flush ทุกครั้งที่ `text_end`
- `message_end`: รอจนข้อความของผู้ช่วยเสร็จ แล้วจึง flush เอาต์พุตที่บัฟเฟอร์ไว้

`message_end` ยังใช้ตัวแบ่งชิ้นอยู่ หากข้อความที่บัฟเฟอร์ไว้เกิน `maxChars` ดังนั้นจึงสามารถส่งออกหลายชิ้นเมื่อสิ้นสุดได้

### การส่งสื่อด้วยการสตรีมแบบบล็อก

คำสั่ง `MEDIA:` เป็นเมทาดาทาการส่งตามปกติ เมื่อการสตรีมแบบบล็อกส่ง
บล็อกสื่อออกไปก่อน OpenClaw จะจดจำการส่งนั้นสำหรับรอบการสนทนา หากเพย์โหลดสุดท้าย
ของผู้ช่วยมี URL สื่อเดียวกันซ้ำ การส่งสุดท้ายจะตัดสื่อที่ซ้ำออก
แทนที่จะส่งไฟล์แนบอีกครั้ง

เพย์โหลดสุดท้ายที่ซ้ำกันทุกประการจะถูกระงับ หากเพย์โหลดสุดท้ายเพิ่ม
ข้อความที่แตกต่างรอบสื่อซึ่งสตรีมไปแล้ว OpenClaw จะยังส่ง
ข้อความใหม่โดยคงให้สื่อนั้นถูกส่งเพียงครั้งเดียว วิธีนี้ป้องกันบันทึกเสียง
หรือไฟล์ซ้ำในช่องทางอย่าง Telegram เมื่อเอเจนต์ส่ง `MEDIA:` ระหว่าง
การสตรีม และผู้ให้บริการรวมสิ่งนั้นไว้ในคำตอบที่เสร็จสมบูรณ์ด้วย

## อัลกอริทึมการแบ่งชิ้น (ขอบเขตต่ำ/สูง)

การแบ่งชิ้นแบบบล็อกถูกใช้งานโดย `EmbeddedBlockChunker`:

- **ขอบเขตต่ำ:** อย่าส่งออกจนกว่าบัฟเฟอร์ >= `minChars` (เว้นแต่ถูกบังคับ)
- **ขอบเขตสูง:** ต้องการแบ่งก่อน `maxChars`; หากถูกบังคับ ให้แบ่งที่ `maxChars`
- **ค่ากำหนดจุดตัด:** `paragraph` → `newline` → `sentence` → `whitespace` → จุดตัดแบบบังคับ
- **โค้ดเฟนซ์:** ไม่แบ่งภายในเฟนซ์; เมื่อถูกบังคับที่ `maxChars` ให้ปิด + เปิดเฟนซ์ใหม่เพื่อให้ Markdown ยังคงถูกต้อง

`maxChars` จะถูกจำกัดให้อยู่ที่ `textChunkLimit` ของช่องทาง ดังนั้นคุณจึงไม่สามารถเกินเพดานต่อช่องทางได้

## การรวมชิ้น (รวมบล็อกที่สตรีม)

เมื่อเปิดใช้งานการสตรีมแบบบล็อก OpenClaw สามารถ **รวมชิ้นบล็อกที่ต่อเนื่องกัน**
ก่อนส่งออกไปได้ วิธีนี้ลด “สแปมบรรทัดเดียว” แต่ยังคงให้
เอาต์พุตแบบค่อยเป็นค่อยไป

- การรวมชิ้นรอ **ช่วงว่างที่ไม่มีการทำงาน** (`idleMs`) ก่อน flush
- บัฟเฟอร์ถูกจำกัดด้วย `maxChars` และจะ flush หากเกินค่านั้น
- `minChars` ป้องกันไม่ให้ส่งส่วนย่อยที่เล็กมากจนกว่าจะมีข้อความสะสมเพียงพอ
  (การ flush สุดท้ายจะส่งข้อความที่เหลือเสมอ)
- ตัวเชื่อมได้มาจาก `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`, `newline` → `\n`, `sentence` → เว้นวรรค)
- การแทนที่ของช่องทางมีให้ผ่าน `*.blockStreamingCoalesce` (รวมถึงการกำหนดค่ารายบัญชี)
- ค่าเริ่มต้นของการรวมชิ้น `minChars` ถูกเพิ่มเป็น 1500 สำหรับ Signal/Slack/Discord เว้นแต่จะถูกแทนที่

## จังหวะเหมือนมนุษย์ระหว่างบล็อก

เมื่อเปิดใช้งานการสตรีมแบบบล็อก คุณสามารถเพิ่ม **การหยุดแบบสุ่ม** ระหว่าง
การตอบกลับแบบบล็อก (หลังบล็อกแรก) ทำให้คำตอบหลายบับเบิลรู้สึก
เป็นธรรมชาติมากขึ้น

- การกำหนดค่า: `agents.defaults.humanDelay` (แทนที่ต่อเอเจนต์ผ่าน `agents.list[].humanDelay`)
- โหมด: `off` (ค่าเริ่มต้น), `natural` (800–2500ms), `custom` (`minMs`/`maxMs`)
- ใช้กับ **การตอบกลับแบบบล็อก** เท่านั้น ไม่ใช่คำตอบสุดท้ายหรือสรุปเครื่องมือ

## "สตรีมเป็นชิ้นหรือทั้งหมด"

สิ่งนี้แมปเป็น:

- **สตรีมเป็นชิ้น:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (ส่งออกระหว่างทาง) ช่องทางที่ไม่ใช่ Telegram ต้องมี `*.blockStreaming: true` ด้วย
- **สตรีมทั้งหมดเมื่อสิ้นสุด:** `blockStreamingBreak: "message_end"` (flush ครั้งเดียว อาจเป็นหลายชิ้นหากยาวมาก)
- **ไม่มีการสตรีมแบบบล็อก:** `blockStreamingDefault: "off"` (เฉพาะคำตอบสุดท้าย)

**หมายเหตุช่องทาง:** การสตรีมแบบบล็อกจะ **ปิด เว้นแต่**
`*.blockStreaming` จะถูกตั้งค่าเป็น `true` อย่างชัดเจน ช่องทางสามารถสตรีมพรีวิวสด
(`channels.<channel>.streaming`) ได้โดยไม่มีการตอบกลับแบบบล็อก

เตือนตำแหน่งการกำหนดค่า: ค่าเริ่มต้น `blockStreaming*` อยู่ภายใต้
`agents.defaults` ไม่ใช่การกำหนดค่ารูท

## โหมดการสตรีมพรีวิว

คีย์มาตรฐาน: `channels.<channel>.streaming`

โหมด:

- `off`: ปิดใช้งานการสตรีมพรีวิว
- `partial`: พรีวิวเดียวที่จะถูกแทนที่ด้วยข้อความล่าสุด
- `block`: พรีวิวอัปเดตเป็นขั้นตอนแบบแบ่งชิ้น/ต่อท้าย
- `progress`: พรีวิวความคืบหน้า/สถานะระหว่างการสร้างคำตอบ คำตอบสุดท้ายเมื่อเสร็จสิ้น

### การแมปช่องทาง

| ช่องทาง    | `off` | `partial` | `block` | `progress`        |
| ---------- | ----- | --------- | ------- | ----------------- |
| Telegram   | ✅    | ✅        | ✅      | แมปเป็น `partial` |
| Discord    | ✅    | ✅        | ✅      | แมปเป็น `partial` |
| Slack      | ✅    | ✅        | ✅      | ✅                |
| Mattermost | ✅    | ✅        | ✅      | ✅                |

เฉพาะ Slack:

- `channels.slack.streaming.nativeTransport` สลับการเรียก Slack native streaming API เมื่อ `channels.slack.streaming.mode="partial"` (ค่าเริ่มต้น: `true`)
- Slack native streaming และสถานะเธรดผู้ช่วยของ Slack ต้องมีเป้าหมายเธรดตอบกลับ; DM ระดับบนสุดจะไม่แสดงพรีวิวแบบเธรดนั้น

การย้ายคีย์เดิม:

- Telegram: ค่าเดิม `streamMode` และค่า `streaming` แบบสเกลาร์/บูลีนจะถูกตรวจพบและย้ายโดยเส้นทางความเข้ากันได้ของ doctor/config ไปยัง `streaming.mode`
- Discord: `streamMode` + `streaming` แบบบูลีนจะย้ายอัตโนมัติไปยัง enum `streaming`
- Slack: `streamMode` จะย้ายอัตโนมัติไปยัง `streaming.mode`; `streaming` แบบบูลีนจะย้ายอัตโนมัติไปยัง `streaming.mode` พร้อม `streaming.nativeTransport`; `nativeStreaming` เดิมจะย้ายอัตโนมัติไปยัง `streaming.nativeTransport`

### พฤติกรรมขณะรัน

Telegram:

- ใช้ `sendMessage` + `editMessageText` สำหรับอัปเดตพรีวิวทั้งใน DM และกลุ่ม/หัวข้อ
- ส่งข้อความสุดท้ายใหม่แทนการแก้ไขในที่เดิมเมื่อพรีวิวแสดงอยู่ประมาณหนึ่งนาที แล้วล้างพรีวิวเพื่อให้เวลาประทับของ Telegram สะท้อนการตอบกลับที่เสร็จสมบูรณ์
- ข้ามการสตรีมพรีวิวเมื่อเปิดใช้งานการสตรีมแบบบล็อกของ Telegram อย่างชัดเจน (เพื่อหลีกเลี่ยงการสตรีมซ้ำสองทาง)
- `/reasoning stream` สามารถเขียนเหตุผลลงในพรีวิวได้

Discord:

- ใช้การส่ง + แก้ไขข้อความพรีวิว
- โหมด `block` ใช้การแบ่งชิ้นแบบร่าง (`draftChunk`)
- ข้ามการสตรีมพรีวิวเมื่อเปิดใช้งานการสตรีมแบบบล็อกของ Discord อย่างชัดเจน
- เพย์โหลดสื่อสุดท้าย ข้อผิดพลาด และการตอบกลับแบบชัดเจนจะยกเลิกพรีวิวที่ค้างอยู่โดยไม่ flush ร่างใหม่ แล้วใช้การส่งตามปกติ

Slack:

- `partial` สามารถใช้ Slack native streaming (`chat.startStream`/`append`/`stop`) เมื่อพร้อมใช้งาน
- `block` ใช้พรีวิวร่างแบบต่อท้าย
- `progress` ใช้ข้อความพรีวิวสถานะ แล้วตามด้วยคำตอบสุดท้าย
- การสตรีมพรีวิวแบบ native และแบบร่างจะระงับการตอบกลับแบบบล็อกสำหรับรอบนั้น เพื่อให้คำตอบของ Slack ถูกสตรีมผ่านเส้นทางการส่งเพียงเส้นทางเดียว
- เพย์โหลดสื่อ/ข้อผิดพลาดสุดท้ายและผลลัพธ์สุดท้ายของความคืบหน้าจะไม่สร้างข้อความร่างชั่วคราว; เฉพาะผลลัพธ์สุดท้ายแบบข้อความ/บล็อกที่แก้ไขพรีวิวได้เท่านั้นที่จะ flush ข้อความร่างที่ค้างอยู่

Mattermost:

- สตรีมการคิด กิจกรรมเครื่องมือ และข้อความตอบกลับบางส่วนเข้าไปในโพสต์พรีวิวร่างเดียว ซึ่งจะ finalize ในที่เดิมเมื่อคำตอบสุดท้ายปลอดภัยพอที่จะส่ง
- ถอยกลับไปส่งโพสต์สุดท้ายใหม่ หากโพสต์พรีวิวถูกลบหรือไม่พร้อมใช้งานในเวลาที่ finalize
- เพย์โหลดสื่อ/ข้อผิดพลาดสุดท้ายจะยกเลิกการอัปเดตพรีวิวที่ค้างอยู่ก่อนการส่งตามปกติ แทนที่จะ flush โพสต์พรีวิวชั่วคราว

Matrix:

- พรีวิวร่างจะ finalize ในที่เดิมเมื่อข้อความสุดท้ายสามารถใช้เหตุการณ์พรีวิวซ้ำได้
- ผลลัพธ์สุดท้ายแบบเฉพาะสื่อ ข้อผิดพลาด และการตอบกลับที่เป้าหมายไม่ตรงกันจะยกเลิกการอัปเดตพรีวิวที่ค้างอยู่ก่อนการส่งตามปกติ; พรีวิวเก่าที่มองเห็นอยู่แล้วจะถูกลบข้อความ

### การอัปเดตพรีวิวความคืบหน้าของเครื่องมือ

การสตรีมพรีวิวยังสามารถรวมการอัปเดต **ความคืบหน้าของเครื่องมือ** ได้ด้วย — บรรทัดสถานะสั้น ๆ เช่น "กำลังค้นหาเว็บ", "กำลังอ่านไฟล์" หรือ "กำลังเรียกเครื่องมือ" — ซึ่งปรากฏในข้อความพรีวิวเดียวกันระหว่างที่เครื่องมือกำลังรัน ก่อนคำตอบสุดท้าย วิธีนี้ทำให้รอบการใช้เครื่องมือหลายขั้นตอนยังดูมีความเคลื่อนไหว แทนที่จะเงียบระหว่างพรีวิวการคิดแรกกับคำตอบสุดท้าย

พื้นผิวที่รองรับ:

- **Discord**, **Slack**, **Telegram** และ **Matrix** สตรีมความคืบหน้าของเครื่องมือเข้าในการแก้ไขพรีวิวสดตามค่าเริ่มต้นเมื่อการสตรีมพรีวิวทำงานอยู่
- Telegram จัดส่งพร้อมการอัปเดตพรีวิวความคืบหน้าของเครื่องมือที่เปิดใช้งานมาตั้งแต่ `v2026.4.22`; การคงให้เปิดไว้จะรักษาพฤติกรรมที่เผยแพร่ออกไปแล้ว
- **Mattermost** รวมกิจกรรมเครื่องมือไว้ในโพสต์พรีวิวร่างเดียวอยู่แล้ว (ดูด้านบน)
- การแก้ไขความคืบหน้าของเครื่องมือทำตามโหมดการสตรีมพรีวิวที่ใช้งานอยู่; จะถูกข้ามเมื่อการสตรีมพรีวิวเป็น `off` หรือเมื่อการสตรีมแบบบล็อกเข้าควบคุมข้อความแล้ว บน Telegram, `streaming.mode: "off"` เป็นแบบเฉพาะผลลัพธ์สุดท้าย: ข้อความความคืบหน้าทั่วไปก็จะถูกระงับแทนที่จะส่งเป็นข้อความ "Working..." แยกต่างหาก ขณะที่พรอมป์อนุมัติ เพย์โหลดสื่อ และข้อผิดพลาดยังคงถูกส่งตามปกติ
- หากต้องการคงการสตรีมพรีวิวไว้แต่ซ่อนบรรทัดความคืบหน้าของเครื่องมือ ให้ตั้งค่า `streaming.preview.toolProgress` เป็น `false` สำหรับช่องทางนั้น หากต้องการปิดการแก้ไขพรีวิวทั้งหมด ให้ตั้งค่า `streaming.mode` เป็น `off`

ตัวอย่าง:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": false
        }
      }
    }
  }
}
```
