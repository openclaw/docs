OpenClaw มีเลเยอร์การสตรีมแยกกันสองชั้น:

- **การสตรีมแบบบล็อก (ช่องทาง):** ส่ง **บล็อก** ที่เสร็จแล้วขณะที่ผู้ช่วยเขียน ข้อความเหล่านี้เป็นข้อความช่องทางปกติ (ไม่ใช่เดลตาโทเค็น)
- **การสตรีมตัวอย่างก่อนส่ง (Telegram/Discord/Slack):** อัปเดต **ข้อความตัวอย่างก่อนส่ง** ชั่วคราวระหว่างสร้างคำตอบ

ปัจจุบันยัง **ไม่มีการสตรีมเดลตาโทเค็นจริง** ไปยังข้อความช่องทาง การสตรีมตัวอย่างก่อนส่งอิงตามข้อความ (ส่ง + แก้ไข/ต่อท้าย)

## การสตรีมแบบบล็อก (ข้อความช่องทาง)

การสตรีมแบบบล็อกส่งเอาต์พุตของผู้ช่วยเป็นส่วนหยาบ ๆ ทันทีที่พร้อมใช้งาน

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

คำอธิบาย:

- `text_delta/events`: เหตุการณ์สตรีมของโมเดล (อาจมีน้อยสำหรับโมเดลที่ไม่สตรีม)
- `chunker`: `EmbeddedBlockChunker` ที่ใช้ขอบเขตต่ำสุด/สูงสุด + ลำดับความสำคัญของจุดตัด
- `channel send`: ข้อความขาออกจริง (การตอบกลับแบบบล็อก)

**การควบคุม:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (ค่าเริ่มต้นคือปิด)
- การแทนที่ต่อช่องทาง: `*.blockStreaming` (และตัวแปรต่อบัญชี) เพื่อบังคับ `"on"`/`"off"` ต่อช่องทาง
- `agents.defaults.blockStreamingBreak`: `"text_end"` หรือ `"message_end"`
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (รวมบล็อกที่สตรีมก่อนส่ง)
- เพดานบังคับของช่องทาง: `*.textChunkLimit` (เช่น `channels.whatsapp.textChunkLimit`)
- โหมดแบ่งส่วนของช่องทาง: `*.chunkMode` (`length` เป็นค่าเริ่มต้น, `newline` แบ่งตามบรรทัดว่าง (ขอบเขตย่อหน้า) ก่อนแบ่งตามความยาว)
- เพดานแบบยืดหยุ่นของ Discord: `channels.discord.maxLinesPerMessage` (ค่าเริ่มต้น 17) แบ่งคำตอบที่สูงเกินไปเพื่อหลีกเลี่ยงการตัดใน UI

**ความหมายของขอบเขต:**

- `text_end`: สตรีมบล็อกทันทีที่ตัวแบ่งส่วนส่งออกมา; ปล่อยบัฟเฟอร์ทุกครั้งที่ `text_end`
- `message_end`: รอจนข้อความผู้ช่วยเสร็จ แล้วจึงปล่อยเอาต์พุตที่บัฟเฟอร์ไว้

`message_end` ยังคงใช้ตัวแบ่งส่วนหากข้อความที่บัฟเฟอร์ไว้เกิน `maxChars` ดังนั้นจึงอาจส่งออกหลายส่วนในตอนท้ายได้

### การส่งสื่อด้วยการสตรีมแบบบล็อก

คำสั่ง `MEDIA:` เป็นเมทาดาทาการส่งปกติ เมื่อการสตรีมแบบบล็อกส่งบล็อกสื่อออกไปก่อน
OpenClaw จะจดจำการส่งนั้นไว้สำหรับรอบสนทนานั้น หากเพย์โหลดสุดท้ายของผู้ช่วยทำซ้ำ URL สื่อเดียวกัน การส่งสุดท้ายจะลบสื่อที่ซ้ำออกแทนการส่งไฟล์แนบอีกครั้ง

เพย์โหลดสุดท้ายที่ซ้ำกันทุกประการจะถูกระงับ หากเพย์โหลดสุดท้ายเพิ่มข้อความที่แตกต่างรอบสื่อที่สตรีมไปแล้ว OpenClaw ยังคงส่งข้อความใหม่พร้อมคงสื่อให้ถูกส่งเพียงครั้งเดียว วิธีนี้ป้องกันโน้ตเสียงหรือไฟล์ซ้ำในช่องทางอย่าง Telegram เมื่อเอเจนต์ส่ง `MEDIA:` ระหว่างการสตรีมและผู้ให้บริการรวมรายการเดียวกันไว้ในคำตอบที่เสร็จแล้วด้วย

## อัลกอริทึมการแบ่งส่วน (ขอบเขตต่ำ/สูง)

การแบ่งบล็อกเป็นส่วนทำโดย `EmbeddedBlockChunker`:

- **ขอบเขตต่ำ:** อย่าส่งออกจนกว่าบัฟเฟอร์ >= `minChars` (เว้นแต่ถูกบังคับ)
- **ขอบเขตสูง:** เลือกจุดแบ่งก่อน `maxChars`; หากถูกบังคับ ให้แบ่งที่ `maxChars`
- **ลำดับความสำคัญของจุดตัด:** `paragraph` → `newline` → `sentence` → `whitespace` → ตัดแบบบังคับ
- **รั้วโค้ด:** ไม่แบ่งภายในรั้ว; เมื่อถูกบังคับที่ `maxChars` ให้ปิด + เปิดรั้วใหม่เพื่อให้ Markdown ถูกต้อง

`maxChars` จะถูกจำกัดตาม `textChunkLimit` ของช่องทาง ดังนั้นคุณไม่สามารถเกินเพดานต่อช่องทางได้

## การรวม (รวมบล็อกที่สตรีม)

เมื่อเปิดใช้งานการสตรีมแบบบล็อก OpenClaw สามารถ **รวมส่วนบล็อกต่อเนื่อง**
ก่อนส่งออกได้ วิธีนี้ลด "สแปมบรรทัดเดียว" ขณะยังคงให้เอาต์พุตแบบค่อยเป็นค่อยไป

- การรวมจะรอ **ช่วงว่างที่ไม่มีการทำงาน** (`idleMs`) ก่อนปล่อยบัฟเฟอร์
- บัฟเฟอร์ถูกจำกัดด้วย `maxChars` และจะถูกปล่อยหากเกินค่านั้น
- `minChars` ป้องกันไม่ให้ส่งส่วนย่อยเล็ก ๆ จนกว่าจะสะสมข้อความพอ
  (การปล่อยครั้งสุดท้ายจะส่งข้อความที่เหลือเสมอ)
- ตัวเชื่อมได้มาจาก `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`, `newline` → `\n`, `sentence` → เว้นวรรค)
- มีการแทนที่ต่อช่องทางผ่าน `*.blockStreamingCoalesce` (รวมถึงการตั้งค่าต่อบัญชี)
- ค่าเริ่มต้นของการรวม `minChars` ถูกเพิ่มเป็น 1500 สำหรับ Signal/Slack/Discord เว้นแต่จะถูกแทนที่

## จังหวะระหว่างบล็อกให้คล้ายมนุษย์

เมื่อเปิดใช้งานการสตรีมแบบบล็อก คุณสามารถเพิ่ม **การหยุดแบบสุ่ม** ระหว่าง
การตอบกลับแบบบล็อก (หลังบล็อกแรก) วิธีนี้ทำให้คำตอบหลายบับเบิลให้ความรู้สึกเป็นธรรมชาติมากขึ้น

- การตั้งค่า: `agents.defaults.humanDelay` (แทนที่ต่อเอเจนต์ผ่าน `agents.list[].humanDelay`)
- โหมด: `off` (ค่าเริ่มต้น), `natural` (800-2500ms), `custom` (`minMs`/`maxMs`)
- ใช้กับ **การตอบกลับแบบบล็อก** เท่านั้น ไม่ใช่คำตอบสุดท้ายหรือสรุปเครื่องมือ

## "สตรีมเป็นส่วน ๆ หรือทั้งหมด"

สิ่งนี้แมปเป็น:

- **สตรีมเป็นส่วน ๆ:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (ส่งออกระหว่างทาง) ช่องทางที่ไม่ใช่ Telegram ยังต้องมี `*.blockStreaming: true`
- **สตรีมทั้งหมดตอนท้าย:** `blockStreamingBreak: "message_end"` (ปล่อยครั้งเดียว อาจเป็นหลายส่วนหากยาวมาก)
- **ไม่มีการสตรีมแบบบล็อก:** `blockStreamingDefault: "off"` (ตอบกลับสุดท้ายเท่านั้น)

**หมายเหตุช่องทาง:** การสตรีมแบบบล็อกจะ **ปิดอยู่ เว้นแต่**
`*.blockStreaming` ถูกตั้งค่าเป็น `true` อย่างชัดเจน ช่องทางสามารถสตรีมตัวอย่างสด
(`channels.<channel>.streaming`) ได้โดยไม่มีการตอบกลับแบบบล็อก

การเตือนตำแหน่งการตั้งค่า: ค่าเริ่มต้น `blockStreaming*` อยู่ใต้
`agents.defaults` ไม่ใช่การตั้งค่าราก

## โหมดการสตรีมตัวอย่างก่อนส่ง

คีย์มาตรฐาน: `channels.<channel>.streaming`

โหมด:

- `off`: ปิดใช้งานการสตรีมตัวอย่างก่อนส่ง
- `partial`: ตัวอย่างก่อนส่งรายการเดียวที่ถูกแทนที่ด้วยข้อความล่าสุด
- `block`: ตัวอย่างก่อนส่งอัปเดตเป็นขั้นตอนแบบแบ่งส่วน/ต่อท้าย
- `progress`: ตัวอย่างความคืบหน้า/สถานะระหว่างสร้างคำตอบ และคำตอบสุดท้ายเมื่อเสร็จ

`streaming.mode: "block"` เป็นโหมดการสตรีมตัวอย่างก่อนส่งสำหรับช่องทางที่แก้ไขได้
เช่น Discord และ Telegram โหมดนี้ไม่ได้เปิดใช้การส่งบล็อกผ่านช่องทางนั้น
ใช้ `streaming.block.enabled` หรือคีย์ช่องทางเดิม `blockStreaming` เมื่อ
คุณต้องการการตอบกลับแบบบล็อกปกติ Microsoft Teams เป็นข้อยกเว้น: ไม่มี
การส่งบล็อกแบบร่างตัวอย่างก่อนส่ง ดังนั้น `streaming.mode: "block"` จึงแมปไปยังการส่งบล็อกของ Teams แทนการสตรีมบางส่วน/ความคืบหน้าแบบเนทีฟ

### การแมปช่องทาง

| ช่องทาง    | `off` | `partial` | `block` | `progress`              |
| ---------- | ----- | --------- | ------- | ----------------------- |
| Telegram   | ✅    | ✅        | ✅      | ร่างความคืบหน้าที่แก้ไขได้ |
| Discord    | ✅    | ✅        | ✅      | ร่างความคืบหน้าที่แก้ไขได้ |
| Slack      | ✅    | ✅        | ✅      | ✅                      |
| Mattermost | ✅    | ✅        | ✅      | ✅                      |
| MS Teams   | ✅    | ✅        | ✅      | สตรีมความคืบหน้าแบบเนทีฟ  |

เฉพาะ Slack:

- `channels.slack.streaming.nativeTransport` เปิด/ปิดการเรียก Slack native streaming API เมื่อ `channels.slack.streaming.mode="partial"` (ค่าเริ่มต้น: `true`)
- Slack native streaming และสถานะเธรดผู้ช่วยของ Slack ต้องมีเป้าหมายเธรดตอบกลับ DM ระดับบนสุดจะไม่แสดงตัวอย่างก่อนส่งแบบเธรดนั้น แต่ยังสามารถใช้โพสต์ร่างตัวอย่างก่อนส่งและการแก้ไขของ Slack ได้

การย้ายคีย์เดิม:

- Telegram: ค่าเดิม `streamMode` และค่า `streaming` แบบสเกลาร์/บูลีนจะถูกตรวจพบและย้ายโดยเส้นทางความเข้ากันได้ของ doctor/config ไปยัง `streaming.mode`
- Discord: `streamMode` + `streaming` แบบบูลีนจะย้ายอัตโนมัติไปยัง enum `streaming`
- Slack: `streamMode` ย้ายอัตโนมัติไปยัง `streaming.mode`; `streaming` แบบบูลีนย้ายอัตโนมัติไปยัง `streaming.mode` พร้อม `streaming.nativeTransport`; `nativeStreaming` เดิมย้ายอัตโนมัติไปยัง `streaming.nativeTransport`

### พฤติกรรมขณะรันไทม์

Telegram:

- ใช้การอัปเดตตัวอย่างก่อนส่งด้วย `sendMessage` + `editMessageText` ทั้งใน DM และกลุ่ม/หัวข้อ
- ข้อความสุดท้ายจะแก้ไขตัวอย่างก่อนส่งที่ใช้งานอยู่แทนที่; ข้อความสุดท้ายที่ยาวจะนำข้อความนั้นกลับมาใช้สำหรับส่วนแรกและส่งเฉพาะส่วนที่เหลือ
- โหมด `progress` จะเก็บความคืบหน้าของเครื่องมือไว้ในร่างสถานะที่แก้ไขได้ ล้างร่างนั้นเมื่อเสร็จ และส่งคำตอบสุดท้ายผ่านการส่งปกติ
- หากการแก้ไขสุดท้ายล้มเหลวก่อนยืนยันข้อความที่เสร็จแล้ว OpenClaw จะใช้การส่งสุดท้ายปกติและล้างตัวอย่างก่อนส่งที่ค้างอยู่
- การสตรีมตัวอย่างก่อนส่งจะถูกข้ามเมื่อเปิดใช้งานการสตรีมแบบบล็อกของ Telegram อย่างชัดเจน (เพื่อหลีกเลี่ยงการสตรีมซ้ำสองทาง)
- `/reasoning stream` สามารถเขียนเหตุผลไปยังตัวอย่างก่อนส่งชั่วคราวที่ถูกลบหลังการส่งสุดท้าย

Discord:

- ใช้การส่ง + แก้ไขข้อความตัวอย่างก่อนส่ง
- โหมด `block` ใช้การแบ่งร่างเป็นส่วน (`draftChunk`)
- การสตรีมตัวอย่างก่อนส่งจะถูกข้ามเมื่อเปิดใช้งานการสตรีมแบบบล็อกของ Discord อย่างชัดเจน
- สื่อสุดท้าย ข้อผิดพลาด และเพย์โหลดตอบกลับแบบชัดเจนจะยกเลิกตัวอย่างก่อนส่งที่รอดำเนินการโดยไม่ปล่อยร่างใหม่ แล้วใช้การส่งปกติ

Slack:

- `partial` สามารถใช้ Slack native streaming (`chat.startStream`/`append`/`stop`) เมื่อพร้อมใช้งาน
- `block` ใช้ตัวอย่างก่อนส่งแบบร่างสไตล์ต่อท้าย
- `progress` ใช้ข้อความตัวอย่างสถานะ แล้วตามด้วยคำตอบสุดท้าย
- DM ระดับบนสุดที่ไม่มีเธรดตอบกลับจะใช้โพสต์ร่างตัวอย่างก่อนส่งและการแก้ไขแทน Slack native streaming
- Native และการสตรีมตัวอย่างก่อนส่งแบบร่างจะระงับการตอบกลับแบบบล็อกสำหรับรอบนั้น ดังนั้นคำตอบ Slack จะถูกสตรีมด้วยเส้นทางการส่งเดียวเท่านั้น
- เพย์โหลดสื่อ/ข้อผิดพลาดสุดท้ายและผลลัพธ์สุดท้ายแบบความคืบหน้าจะไม่สร้างข้อความร่างชั่วคราว; เฉพาะผลลัพธ์สุดท้ายแบบข้อความ/บล็อกที่แก้ไขตัวอย่างก่อนส่งได้เท่านั้นที่จะปล่อยข้อความร่างที่รอดำเนินการ

Mattermost:

- สตรีมการคิด กิจกรรมเครื่องมือ และข้อความตอบกลับบางส่วนเข้าไปในโพสต์ร่างตัวอย่างก่อนส่งเดียว ซึ่งจะสรุปให้เสร็จแทนที่เดิมเมื่อส่งคำตอบสุดท้ายได้อย่างปลอดภัย
- ถอยกลับไปส่งโพสต์สุดท้ายใหม่หากโพสต์ตัวอย่างก่อนส่งถูกลบหรือไม่พร้อมใช้งานในเวลาสรุปให้เสร็จ
- เพย์โหลดสื่อ/ข้อผิดพลาดสุดท้ายจะยกเลิกการอัปเดตตัวอย่างก่อนส่งที่รอดำเนินการก่อนการส่งปกติ แทนการปล่อยโพสต์ตัวอย่างก่อนส่งชั่วคราว

Matrix:

- ร่างตัวอย่างก่อนส่งจะสรุปให้เสร็จแทนที่เดิมเมื่อข้อความสุดท้ายสามารถนำเหตุการณ์ตัวอย่างก่อนส่งกลับมาใช้ได้
- ผลลัพธ์สุดท้ายที่เป็นสื่อเท่านั้น ข้อผิดพลาด และเป้าหมายตอบกลับไม่ตรงกันจะยกเลิกการอัปเดตตัวอย่างก่อนส่งที่รอดำเนินการก่อนการส่งปกติ; ตัวอย่างก่อนส่งเก่าที่มองเห็นแล้วจะถูกปกปิด

### การอัปเดตตัวอย่างความคืบหน้าของเครื่องมือ

การสตรีมตัวอย่างก่อนส่งยังสามารถรวมการอัปเดต **ความคืบหน้าของเครื่องมือ** ได้ด้วย - บรรทัดสถานะสั้น ๆ เช่น "กำลังค้นหาเว็บ", "กำลังอ่านไฟล์" หรือ "กำลังเรียกใช้เครื่องมือ" - ซึ่งปรากฏในข้อความตัวอย่างก่อนส่งเดียวกันระหว่างที่เครื่องมือกำลังทำงาน ก่อนคำตอบสุดท้าย วิธีนี้ทำให้รอบการใช้เครื่องมือหลายขั้นตอนยังคงมีความเคลื่อนไหวทางสายตา แทนที่จะเงียบไประหว่างตัวอย่างการคิดแรกกับคำตอบสุดท้าย

พื้นผิวที่รองรับ:

- **Discord**, **Slack**, **Telegram** และ **Matrix** สตรีมความคืบหน้าของเครื่องมือเข้าสู่การแก้ไขตัวอย่างสดโดยค่าเริ่มต้นเมื่อการสตรีมตัวอย่างเปิดใช้งานอยู่ Microsoft Teams ใช้สตรีมความคืบหน้าแบบเนทีฟในแชทส่วนตัว
- Telegram ได้จัดส่งพร้อมการอัปเดตตัวอย่างความคืบหน้าของเครื่องมือที่เปิดใช้งานมาตั้งแต่ `v2026.4.22`; การเปิดใช้งานต่อไปจะรักษาพฤติกรรมที่เผยแพร่แล้วนั้นไว้
- **Mattermost** รวมกิจกรรมของเครื่องมือไว้ในโพสต์ตัวอย่างฉบับร่างรายการเดียวอยู่แล้ว (ดูด้านบน)
- การแก้ไขความคืบหน้าของเครื่องมือจะเป็นไปตามโหมดการสตรีมตัวอย่างที่เปิดใช้งานอยู่ โดยจะข้ามเมื่อการสตรีมตัวอย่างเป็น `off` หรือเมื่อการสตรีมบล็อกเข้าควบคุมข้อความแล้ว บน Telegram, `streaming.mode: "off"` เป็นแบบผลลัพธ์สุดท้ายเท่านั้น: ข้อความความคืบหน้าทั่วไปจะถูกระงับเช่นกัน แทนที่จะส่งเป็นข้อความสถานะแยกต่างหาก ขณะที่พรอมป์อนุมัติ เพย์โหลดสื่อ และข้อผิดพลาดยังคงถูกกำหนดเส้นทางตามปกติ
- หากต้องการคงการสตรีมตัวอย่างไว้แต่ซ่อนบรรทัดความคืบหน้าของเครื่องมือ ให้ตั้งค่า `streaming.preview.toolProgress` เป็น `false` สำหรับช่องทางนั้น หากต้องการให้บรรทัดความคืบหน้าของเครื่องมือยังมองเห็นได้ขณะที่ซ่อนข้อความคำสั่ง/การดำเนินการ ให้ตั้งค่า `streaming.preview.commandText` เป็น `"status"` หรือ `streaming.progress.commandText` เป็น `"status"`; ค่าเริ่มต้นคือ `"raw"` เพื่อรักษาพฤติกรรมที่เผยแพร่แล้ว นโยบายนี้ใช้ร่วมกันโดยช่องทางฉบับร่าง/ความคืบหน้าที่ใช้ตัวเรนเดอร์ความคืบหน้าแบบกะทัดรัดของ OpenClaw รวมถึง Discord, Matrix, Microsoft Teams, Mattermost, ตัวอย่างฉบับร่างของ Slack และ Telegram หากต้องการปิดใช้งานการแก้ไขตัวอย่างทั้งหมด ให้ตั้งค่า `streaming.mode` เป็น `off`
- การตอบกลับใบเสนอราคาที่เลือกใน Telegram เป็นข้อยกเว้น: เมื่อ `replyToMode` ไม่ใช่ `"off"` และมีข้อความใบเสนอราคาที่เลือกอยู่ OpenClaw จะข้ามสตรีมตัวอย่างคำตอบสำหรับรอบนั้น ดังนั้นบรรทัดตัวอย่างความคืบหน้าของเครื่องมือจึงไม่สามารถเรนเดอร์ได้ การตอบกลับข้อความปัจจุบันที่ไม่มีข้อความใบเสนอราคาที่เลือกยังคงเก็บการสตรีมตัวอย่างไว้ ดูรายละเอียดใน [เอกสารช่องทาง Telegram](/th/channels/telegram)

คงบรรทัดความคืบหน้าให้มองเห็นได้ แต่ซ่อนข้อความคำสั่ง/การดำเนินการแบบดิบ:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```

ใช้รูปแบบเดียวกันภายใต้คีย์ช่องทางความคืบหน้าแบบกะทัดรัดอื่น เช่น `channels.discord`, `channels.matrix`, `channels.msteams`, `channels.mattermost` หรือตัวอย่างฉบับร่างของ Slack สำหรับโหมดฉบับร่างความคืบหน้า ให้วางนโยบายเดียวกันไว้ใต้ `streaming.progress`:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "progress",
        "progress": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```
