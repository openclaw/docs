OpenClaw มีเลเยอร์การสตรีมแยกกันสองชั้น:

- **การสตรีมแบบบล็อก (ช่องทาง):** ส่ง **บล็อก** ที่เสร็จแล้วขณะที่ผู้ช่วยเขียน ข้อความเหล่านี้เป็นข้อความช่องทางปกติ (ไม่ใช่เดลตาของโทเค็น)
- **การสตรีมตัวอย่างแสดงผล (Telegram/Discord/Slack):** อัปเดต **ข้อความตัวอย่างแสดงผล** ชั่วคราวระหว่างการสร้างคำตอบ

ปัจจุบันยัง **ไม่มีการสตรีม token-delta จริง** ไปยังข้อความช่องทาง การสตรีมตัวอย่างแสดงผลอิงตามข้อความ (ส่ง + แก้ไข/ผนวก)

## การสตรีมแบบบล็อก (ข้อความช่องทาง)

การสตรีมแบบบล็อกส่งเอาต์พุตของผู้ช่วยเป็นชิ้นขนาดหยาบเมื่อพร้อมใช้งาน

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

คำอธิบายสัญลักษณ์:

- `text_delta/events`: เหตุการณ์สตรีมจากโมเดล (อาจมีไม่ต่อเนื่องสำหรับโมเดลที่ไม่สตรีม)
- `chunker`: `EmbeddedBlockChunker` ที่ใช้ขอบเขตต่ำสุด/สูงสุด + ค่ากำหนดจุดตัด
- `channel send`: ข้อความขาออกจริง (การตอบกลับแบบบล็อก)

**ตัวควบคุม:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (ค่าเริ่มต้นปิด)
- การแทนที่ระดับช่องทาง: `*.blockStreaming` (และตัวแปรรายบัญชี) เพื่อบังคับ `"on"`/`"off"` ต่อช่องทาง
- `agents.defaults.blockStreamingBreak`: `"text_end"` หรือ `"message_end"`
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (รวมบล็อกที่สตรีมก่อนส่ง)
- เพดานบังคับของช่องทาง: `*.textChunkLimit` (เช่น `channels.whatsapp.textChunkLimit`)
- โหมดแบ่งชิ้นของช่องทาง: `*.chunkMode` (ค่าเริ่มต้น `length`, `newline` แบ่งที่บรรทัดว่าง (ขอบเขตย่อหน้า) ก่อนแบ่งตามความยาว)
- เพดานแบบอ่อนของ Discord: `channels.discord.maxLinesPerMessage` (ค่าเริ่มต้น 17) แบ่งคำตอบที่สูงมากเพื่อหลีกเลี่ยงการถูกตัดใน UI

**ความหมายของขอบเขต:**

- `text_end`: สตรีมบล็อกทันทีที่ chunker ส่งออก; flush ทุกครั้งที่มี `text_end`
- `message_end`: รอจนข้อความของผู้ช่วยเสร็จสิ้น แล้ว flush เอาต์พุตที่บัฟเฟอร์ไว้

`message_end` ยังคงใช้ chunker หากข้อความที่บัฟเฟอร์ไว้เกิน `maxChars` ดังนั้นจึงสามารถส่งออกหลายชิ้นในตอนท้ายได้

### การส่งสื่อพร้อมการสตรีมแบบบล็อก

คำสั่ง `MEDIA:` เป็นเมทาดาทาการส่งแบบปกติ เมื่อการสตรีมแบบบล็อกส่งบล็อกสื่อก่อนเวลา OpenClaw จะจดจำการส่งนั้นสำหรับเทิร์นนั้น หากเพย์โหลดสุดท้ายของผู้ช่วยซ้ำ URL สื่อเดียวกัน การส่งสุดท้ายจะตัดสื่อที่ซ้ำออกแทนการส่งไฟล์แนบอีกครั้ง

เพย์โหลดสุดท้ายที่ซ้ำกันทุกประการจะถูกระงับ หากเพย์โหลดสุดท้ายเพิ่มข้อความที่แตกต่างรอบสื่อที่สตรีมไปแล้ว OpenClaw จะยังคงส่งข้อความใหม่พร้อมคงการส่งสื่อไว้เพียงครั้งเดียว วิธีนี้ป้องกันบันทึกเสียงหรือไฟล์ซ้ำบนช่องทางอย่าง Telegram เมื่อเอเจนต์ส่ง `MEDIA:` ระหว่างการสตรีมและผู้ให้บริการรวมสิ่งเดียวกันไว้ในคำตอบที่เสร็จแล้วด้วย

## อัลกอริทึมการแบ่งชิ้น (ขอบเขตต่ำ/สูง)

การแบ่งชิ้นแบบบล็อกถูกนำไปใช้โดย `EmbeddedBlockChunker`:

- **ขอบเขตต่ำ:** อย่าส่งออกจนกว่าบัฟเฟอร์ >= `minChars` (เว้นแต่ถูกบังคับ)
- **ขอบเขตสูง:** ควรแบ่งก่อน `maxChars`; หากถูกบังคับ ให้แบ่งที่ `maxChars`
- **ค่ากำหนดจุดตัด:** `paragraph` → `newline` → `sentence` → `whitespace` → ตัดแบบแข็ง
- **Code fences:** ห้ามแบ่งภายใน fences; เมื่อถูกบังคับที่ `maxChars` ให้ปิด + เปิด fence ใหม่เพื่อให้ Markdown ยังคงถูกต้อง

`maxChars` จะถูกจำกัดไม่ให้เกิน `textChunkLimit` ของช่องทาง ดังนั้นคุณจึงไม่สามารถเกินเพดานรายช่องทางได้

## การรวมชิ้น (รวมบล็อกที่สตรีม)

เมื่อเปิดใช้การสตรีมแบบบล็อก OpenClaw สามารถ **รวมชิ้นบล็อกต่อเนื่อง**
ก่อนส่งออกได้ วิธีนี้ลด “สแปมบรรทัดเดียว” แต่ยังคงให้เอาต์พุตแบบค่อยเป็นค่อยไป

- การรวมชิ้นจะรอ **ช่วงว่างที่ไม่มีการทำงาน** (`idleMs`) ก่อน flush
- บัฟเฟอร์ถูกจำกัดด้วย `maxChars` และจะ flush หากเกินค่านั้น
- `minChars` ป้องกันไม่ให้ส่งเศษข้อความเล็กมากจนกว่าจะมีข้อความสะสมเพียงพอ
  (flush สุดท้ายจะส่งข้อความที่เหลือเสมอ)
- ตัวเชื่อมได้มาจาก `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`, `newline` → `\n`, `sentence` → เว้นวรรค)
- มีการแทนที่ระดับช่องทางผ่าน `*.blockStreamingCoalesce` (รวมถึงการกำหนดค่ารายบัญชี)
- ค่าเริ่มต้นของการรวมชิ้น `minChars` ถูกเพิ่มเป็น 1500 สำหรับ Signal/Slack/Discord เว้นแต่จะถูกแทนที่

## จังหวะระหว่างบล็อกที่เหมือนมนุษย์

เมื่อเปิดใช้การสตรีมแบบบล็อก คุณสามารถเพิ่ม **การหยุดแบบสุ่ม** ระหว่าง
การตอบกลับแบบบล็อก (หลังบล็อกแรก) วิธีนี้ทำให้คำตอบหลายบับเบิลรู้สึกเป็นธรรมชาติมากขึ้น

- การกำหนดค่า: `agents.defaults.humanDelay` (แทนที่ต่อเอเจนต์ผ่าน `agents.list[].humanDelay`)
- โหมด: `off` (ค่าเริ่มต้น), `natural` (800–2500ms), `custom` (`minMs`/`maxMs`)
- ใช้เฉพาะกับ **การตอบกลับแบบบล็อก** ไม่ใช่คำตอบสุดท้ายหรือสรุปเครื่องมือ

## "สตรีมเป็นชิ้นหรือทั้งหมด"

สิ่งนี้แมปเป็น:

- **สตรีมเป็นชิ้น:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (ส่งออกไปเรื่อย ๆ) ช่องทางที่ไม่ใช่ Telegram ยังต้องมี `*.blockStreaming: true`
- **สตรีมทั้งหมดตอนท้าย:** `blockStreamingBreak: "message_end"` (flush ครั้งเดียว อาจเป็นหลายชิ้นหากยาวมาก)
- **ไม่มีการสตรีมแบบบล็อก:** `blockStreamingDefault: "off"` (เฉพาะคำตอบสุดท้าย)

**หมายเหตุช่องทาง:** การสตรีมแบบบล็อกจะ **ปิด เว้นแต่**
จะตั้งค่า `*.blockStreaming` เป็น `true` อย่างชัดเจน ช่องทางสามารถสตรีมตัวอย่างแสดงผลสด
(`channels.<channel>.streaming`) ได้โดยไม่มีการตอบกลับแบบบล็อก

การเตือนตำแหน่งการกำหนดค่า: ค่าเริ่มต้น `blockStreaming*` อยู่ภายใต้
`agents.defaults` ไม่ใช่การกำหนดค่าราก

## โหมดการสตรีมตัวอย่างแสดงผล

คีย์มาตรฐาน: `channels.<channel>.streaming`

โหมด:

- `off`: ปิดใช้งานการสตรีมตัวอย่างแสดงผล
- `partial`: ตัวอย่างแสดงผลเดียวที่ถูกแทนที่ด้วยข้อความล่าสุด
- `block`: อัปเดตตัวอย่างแสดงผลเป็นขั้นตอนแบบแบ่งชิ้น/ผนวก
- `progress`: ตัวอย่างแสดงผลความคืบหน้า/สถานะระหว่างการสร้าง คำตอบสุดท้ายเมื่อเสร็จสมบูรณ์

### การแมปช่องทาง

| ช่องทาง    | `off` | `partial` | `block` | `progress`        |
| ---------- | ----- | --------- | ------- | ----------------- |
| Telegram   | ✅    | ✅        | ✅      | แมปเป็น `partial` |
| Discord    | ✅    | ✅        | ✅      | แมปเป็น `partial` |
| Slack      | ✅    | ✅        | ✅      | ✅                |
| Mattermost | ✅    | ✅        | ✅      | ✅                |

เฉพาะ Slack:

- `channels.slack.streaming.nativeTransport` สลับการเรียก API การสตรีมแบบเนทีฟของ Slack เมื่อ `channels.slack.streaming.mode="partial"` (ค่าเริ่มต้น: `true`)
- การสตรีมแบบเนทีฟของ Slack และสถานะเธรดผู้ช่วยของ Slack ต้องมีเป้าหมายเธรดตอบกลับ DM ระดับบนสุดไม่แสดงตัวอย่างแสดงผลแบบเธรดนั้น แต่ยังสามารถใช้โพสต์และการแก้ไขตัวอย่างแสดงผลฉบับร่างของ Slack ได้

การย้ายคีย์เดิม:

- Telegram: ค่าเดิม `streamMode` และค่า `streaming` แบบสเกลาร์/บูลีนจะถูกตรวจพบและย้ายโดยเส้นทางความเข้ากันได้ของ doctor/config ไปยัง `streaming.mode`
- Discord: `streamMode` + `streaming` แบบบูลีนจะย้ายอัตโนมัติเป็น enum `streaming`
- Slack: `streamMode` จะย้ายอัตโนมัติไปยัง `streaming.mode`; `streaming` แบบบูลีนจะย้ายอัตโนมัติไปยัง `streaming.mode` พร้อม `streaming.nativeTransport`; `nativeStreaming` เดิมจะย้ายอัตโนมัติไปยัง `streaming.nativeTransport`

### พฤติกรรมขณะรัน

Telegram:

- ใช้ `sendMessage` + `editMessageText` สำหรับการอัปเดตตัวอย่างแสดงผลทั้งใน DM และกลุ่ม/หัวข้อ
- ส่งข้อความสุดท้ายใหม่แทนการแก้ไขแทนที่เดิมเมื่อตัวอย่างแสดงผลมองเห็นมาแล้วประมาณหนึ่งนาที จากนั้นล้างตัวอย่างแสดงผลเพื่อให้ timestamp ของ Telegram สะท้อนเวลาที่ตอบเสร็จ
- ข้ามการสตรีมตัวอย่างแสดงผลเมื่อเปิดใช้การสตรีมแบบบล็อกของ Telegram อย่างชัดเจน (เพื่อหลีกเลี่ยงการสตรีมซ้ำสองทาง)
- `/reasoning stream` สามารถเขียน reasoning ไปยังตัวอย่างแสดงผลได้

Discord:

- ใช้การส่ง + แก้ไขข้อความตัวอย่างแสดงผล
- โหมด `block` ใช้การแบ่งชิ้นฉบับร่าง (`draftChunk`)
- ข้ามการสตรีมตัวอย่างแสดงผลเมื่อเปิดใช้การสตรีมแบบบล็อกของ Discord อย่างชัดเจน
- เพย์โหลดสื่อสุดท้าย ข้อผิดพลาด และเพย์โหลดตอบกลับแบบชัดเจนจะยกเลิกตัวอย่างแสดงผลที่ค้างอยู่โดยไม่ flush ฉบับร่างใหม่ จากนั้นใช้การส่งตามปกติ

Slack:

- `partial` สามารถใช้การสตรีมแบบเนทีฟของ Slack (`chat.startStream`/`append`/`stop`) เมื่อพร้อมใช้งาน
- `block` ใช้ตัวอย่างแสดงผลฉบับร่างแบบผนวก
- `progress` ใช้ข้อความตัวอย่างแสดงผลสถานะ จากนั้นเป็นคำตอบสุดท้าย
- DM ระดับบนสุดที่ไม่มีเธรดตอบกลับจะใช้โพสต์และการแก้ไขตัวอย่างแสดงผลฉบับร่างแทนการสตรีมแบบเนทีฟของ Slack
- การสตรีมตัวอย่างแสดงผลแบบเนทีฟและแบบฉบับร่างจะระงับการตอบกลับแบบบล็อกสำหรับเทิร์นนั้น ดังนั้นคำตอบของ Slack จะถูกสตรีมผ่านเส้นทางการส่งเพียงเส้นทางเดียว
- เพย์โหลดสื่อ/ข้อผิดพลาดสุดท้ายและคำตอบสุดท้ายแบบความคืบหน้าจะไม่สร้างข้อความฉบับร่างทิ้งขว้าง; เฉพาะคำตอบสุดท้ายแบบข้อความ/บล็อกที่สามารถแก้ไขตัวอย่างแสดงผลได้เท่านั้นที่จะ flush ข้อความฉบับร่างที่ค้างอยู่

Mattermost:

- สตรีมการคิด กิจกรรมเครื่องมือ และข้อความตอบกลับบางส่วนไปยังโพสต์ตัวอย่างแสดงผลฉบับร่างเดียวที่ finalize แทนที่เดิมเมื่อคำตอบสุดท้ายปลอดภัยที่จะส่ง
- ถอยกลับไปส่งโพสต์สุดท้ายใหม่หากโพสต์ตัวอย่างแสดงผลถูกลบหรือไม่พร้อมใช้งานในเวลาที่ finalize
- เพย์โหลดสื่อ/ข้อผิดพลาดสุดท้ายจะยกเลิกการอัปเดตตัวอย่างแสดงผลที่ค้างอยู่ก่อนการส่งตามปกติ แทนการ flush โพสต์ตัวอย่างแสดงผลชั่วคราว

Matrix:

- ตัวอย่างแสดงผลฉบับร่างจะ finalize แทนที่เดิมเมื่อข้อความสุดท้ายสามารถใช้เหตุการณ์ตัวอย่างแสดงผลซ้ำได้
- คำตอบสุดท้ายแบบสื่อเท่านั้น ข้อผิดพลาด และเป้าหมายตอบกลับไม่ตรงกันจะยกเลิกการอัปเดตตัวอย่างแสดงผลที่ค้างอยู่ก่อนการส่งตามปกติ; ตัวอย่างแสดงผลเก่าที่มองเห็นอยู่แล้วจะถูก redact

### การอัปเดตตัวอย่างแสดงผลความคืบหน้าของเครื่องมือ

การสตรีมตัวอย่างแสดงผลยังสามารถรวมการอัปเดต **ความคืบหน้าของเครื่องมือ** ได้ด้วย — บรรทัดสถานะสั้น ๆ เช่น "กำลังค้นหาเว็บ", "กำลังอ่านไฟล์" หรือ "กำลังเรียกเครื่องมือ" — ซึ่งปรากฏในข้อความตัวอย่างแสดงผลเดียวกันระหว่างที่เครื่องมือทำงาน ก่อนคำตอบสุดท้าย วิธีนี้ทำให้เทิร์นของเครื่องมือหลายขั้นตอนดูมีความเคลื่อนไหว แทนที่จะเงียบระหว่างตัวอย่างแสดงผลการคิดแรกกับคำตอบสุดท้าย

พื้นผิวที่รองรับ:

- **Discord**, **Slack**, **Telegram** และ **Matrix** สตรีมความคืบหน้าของเครื่องมือเข้าสู่การแก้ไขตัวอย่างแสดงผลสดตามค่าเริ่มต้นเมื่อการสตรีมตัวอย่างแสดงผลทำงานอยู่
- Telegram ส่งมาพร้อมการอัปเดตตัวอย่างแสดงผลความคืบหน้าของเครื่องมือที่เปิดใช้ตั้งแต่ `v2026.4.22`; การคงให้เปิดอยู่จะรักษาพฤติกรรมที่เผยแพร่แล้วนั้นไว้
- **Mattermost** รวมกิจกรรมเครื่องมือไว้ในโพสต์ตัวอย่างแสดงผลฉบับร่างเดียวอยู่แล้ว (ดูด้านบน)
- การแก้ไขความคืบหน้าของเครื่องมือจะตามโหมดการสตรีมตัวอย่างแสดงผลที่ใช้งานอยู่; จะถูกข้ามเมื่อการสตรีมตัวอย่างแสดงผลเป็น `off` หรือเมื่อการสตรีมแบบบล็อกเข้าควบคุมข้อความแล้ว บน Telegram, `streaming.mode: "off"` เป็นแบบคำตอบสุดท้ายเท่านั้น: ข้อความความคืบหน้าทั่วไปจะถูกระงับเช่นกัน แทนการส่งเป็นข้อความ "Working..." แบบเดี่ยว ๆ ขณะที่พรอมป์การอนุมัติ เพย์โหลดสื่อ และข้อผิดพลาดยังคงถูกส่งตามเส้นทางปกติ
- หากต้องการคงการสตรีมตัวอย่างแสดงผลไว้แต่ซ่อนบรรทัดความคืบหน้าของเครื่องมือ ให้ตั้ง `streaming.preview.toolProgress` เป็น `false` สำหรับช่องทางนั้น หากต้องการปิดการแก้ไขตัวอย่างแสดงผลทั้งหมด ให้ตั้ง `streaming.mode` เป็น `off`

ตัวอย่าง:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": false
        }
      }
    }
  }
}
```
