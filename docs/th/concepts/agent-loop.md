ลูปแบบเอเจนต์คือการรัน “จริง” แบบเต็มของเอเจนต์: การรับเข้า → การประกอบบริบท → การอนุมานของโมเดล →
การเรียกใช้เครื่องมือ → การสตรีมคำตอบ → การคงข้อมูล เป็นเส้นทางอ้างอิงที่แปลงข้อความ
ให้เป็นการกระทำและคำตอบสุดท้าย พร้อมรักษาสถานะเซสชันให้สอดคล้องกัน

ใน OpenClaw ลูปคือการรันแบบทำทีละรายการหนึ่งครั้งต่อเซสชัน ซึ่งปล่อยเหตุการณ์วงจรชีวิตและสตรีม
ขณะที่โมเดลคิด เรียกเครื่องมือ และสตรีมเอาต์พุต เอกสารนี้อธิบายว่าลูปของจริงนั้น
ถูกเชื่อมต่อครบตั้งแต่ต้นจนจบอย่างไร

## จุดเข้าใช้งาน

- Gateway RPC: `agent` และ `agent.wait`
- CLI: คำสั่ง `agent`

## วิธีทำงาน (ภาพรวมระดับสูง)

1. `agent` RPC ตรวจสอบพารามิเตอร์ แก้ค่าเซสชัน (sessionKey/sessionId) คงข้อมูลเมทาดาทาของเซสชัน และคืนค่า `{ runId, acceptedAt }` ทันที
2. `agentCommand` รันเอเจนต์:
   - แก้ค่าโมเดล + ค่าเริ่มต้นของ thinking/verbose/trace
   - โหลดสแนปช็อต Skills
   - เรียก `runEmbeddedPiAgent` (รันไทม์ pi-agent-core)
   - ปล่อย **จบ/ข้อผิดพลาดของวงจรชีวิต** หากลูปแบบฝังไม่ได้ปล่อยเอง
3. `runEmbeddedPiAgent`:
   - จัดการรันให้เป็นลำดับผ่านคิวต่อเซสชัน + คิวส่วนกลาง
   - แก้ค่าโมเดล + โปรไฟล์การยืนยันตัวตน และสร้างเซสชัน Pi
   - สมัครรับเหตุการณ์ Pi และสตรีมเดลตาของผู้ช่วย/เครื่องมือ
   - บังคับใช้หมดเวลา -> ยกเลิกรันหากเกินเวลา
   - สำหรับเทิร์นของเซิร์ฟเวอร์แอป Codex ให้ยกเลิกเทิร์นที่รับแล้วซึ่งหยุดสร้างความคืบหน้าของเซิร์ฟเวอร์แอปก่อนถึงเหตุการณ์ปลายทาง
   - คืนค่าเพย์โหลด + เมทาดาทาการใช้งาน
4. `subscribeEmbeddedPiSession` เชื่อมเหตุการณ์ pi-agent-core ไปยังสตรีม `agent` ของ OpenClaw:
   - เหตุการณ์เครื่องมือ => `stream: "tool"`
   - เดลตาของผู้ช่วย => `stream: "assistant"`
   - เหตุการณ์วงจรชีวิต => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` ใช้ `waitForAgentRun`:
   - รอ **จบ/ข้อผิดพลาดของวงจรชีวิต** สำหรับ `runId`
   - คืนค่า `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## การจัดคิว + ภาวะพร้อมกัน

- การรันจะถูกทำให้เป็นลำดับต่อคีย์เซสชัน (เลนเซสชัน) และอาจผ่านเลนส่วนกลางด้วย
- วิธีนี้ป้องกันการแข่งขันของเครื่องมือ/เซสชัน และรักษาประวัติเซสชันให้สอดคล้อง
- ช่องทางข้อความสามารถเลือกโหมดคิว (รวบรวม/ชี้นำ/ติดตามผล) ที่ป้อนเข้าสู่ระบบเลนนี้ได้
  ดู [คิวคำสั่ง](/th/concepts/queue)
- การเขียนทรานสคริปต์ยังถูกป้องกันด้วยล็อกการเขียนเซสชันบนไฟล์เซสชัน ล็อกนี้
  รับรู้กระบวนการและอิงไฟล์ จึงจับตัวเขียนที่ข้ามคิวในกระบวนการหรือมาจาก
  กระบวนการอื่นได้ ตัวเขียนทรานสคริปต์ของเซสชันจะรอได้สูงสุด `session.writeLock.acquireTimeoutMs`
  ก่อนรายงานว่าเซสชันกำลังยุ่งอยู่ ค่าเริ่มต้นคือ `60000` มิลลิวินาที
- ล็อกการเขียนเซสชันโดยค่าเริ่มต้นจะไม่ให้เข้าใช้ซ้ำ หากตัวช่วยตั้งใจซ้อนการขอ
  ล็อกเดียวกันโดยยังรักษาตัวเขียนเชิงตรรกะไว้หนึ่งตัว ต้องเลือกใช้โดยชัดเจนด้วย
  `allowReentrant: true`

## การเตรียมเซสชัน + พื้นที่ทำงาน

- พื้นที่ทำงานถูกแก้ค่าและสร้างขึ้น การรันแบบ sandbox อาจเปลี่ยนเส้นทางไปยังรากพื้นที่ทำงานของ sandbox
- Skills ถูกโหลด (หรือใช้ซ้ำจากสแนปช็อต) และถูกฉีดเข้า env และพรอมป์ต์
- ไฟล์บูตสแตรป/บริบทถูกแก้ค่าและฉีดเข้าในรายงานพรอมป์ต์ระบบ
- มีการขอล็อกการเขียนเซสชัน `SessionManager` ถูกเปิดและเตรียมก่อนเริ่มสตรีม เส้นทางใดๆ
  ภายหลังที่เขียนทรานสคริปต์ใหม่ ทำ Compaction หรือตัดทอน ต้องใช้ล็อกเดียวกันก่อนเปิดหรือ
  แก้ไขไฟล์ทรานสคริปต์

## การประกอบพรอมป์ต์ + พรอมป์ต์ระบบ

- พรอมป์ต์ระบบถูกสร้างจากพรอมป์ต์พื้นฐานของ OpenClaw, พรอมป์ต์ Skills, บริบทบูตสแตรป และการแทนที่รายรัน
- มีการบังคับใช้ขีดจำกัดเฉพาะโมเดลและโทเคนสำรองสำหรับ Compaction
- ดู [พรอมป์ต์ระบบ](/th/concepts/system-prompt) สำหรับสิ่งที่โมเดลเห็น

## จุด Hook (ตำแหน่งที่คุณดักแทรกได้)

OpenClaw มีระบบ Hook สองแบบ:

- **Hook ภายใน** (Hook ของ Gateway): สคริปต์ขับเคลื่อนด้วยเหตุการณ์สำหรับคำสั่งและเหตุการณ์วงจรชีวิต
- **Hook ของ Plugin**: จุดขยายภายในวงจรชีวิตเอเจนต์/เครื่องมือและไปป์ไลน์ Gateway

### Hook ภายใน (Hook ของ Gateway)

- **`agent:bootstrap`**: รันขณะสร้างไฟล์บูตสแตรปก่อนสรุปพรอมป์ต์ระบบ
  ใช้สิ่งนี้เพื่อเพิ่ม/ลบไฟล์บริบทบูตสแตรป
- **Hook คำสั่ง**: `/new`, `/reset`, `/stop` และเหตุการณ์คำสั่งอื่นๆ (ดูเอกสาร Hook)

ดู [Hook](/th/automation/hooks) สำหรับการตั้งค่าและตัวอย่าง

### Hook ของ Plugin (วงจรชีวิตเอเจนต์ + Gateway)

สิ่งเหล่านี้รันภายในลูปเอเจนต์หรือไปป์ไลน์ Gateway:

- **`before_model_resolve`**: รันก่อนเซสชัน (ไม่มี `messages`) เพื่อแทนที่ผู้ให้บริการ/โมเดลอย่างกำหนดแน่นอนก่อนแก้ค่าโมเดล
- **`before_prompt_build`**: รันหลังโหลดเซสชัน (พร้อม `messages`) เพื่อฉีด `prependContext`, `systemPrompt`, `prependSystemContext` หรือ `appendSystemContext` ก่อนส่งพรอมป์ต์ ใช้ `prependContext` สำหรับข้อความไดนามิกต่อเทิร์น และใช้ฟิลด์บริบทระบบสำหรับคำแนะนำที่เสถียรซึ่งควรอยู่ในพื้นที่พรอมป์ต์ระบบ
- **`before_agent_start`**: Hook ความเข้ากันได้แบบเดิมที่อาจรันในเฟสใดเฟสหนึ่ง ควรใช้ Hook ที่ระบุชัดด้านบน
- **`before_agent_reply`**: รันหลังการกระทำแบบอินไลน์และก่อนเรียก LLM ทำให้ Plugin สามารถรับเทิร์นและคืนคำตอบสังเคราะห์ หรือทำให้เทิร์นนั้นเงียบทั้งหมด
- **`agent_end`**: ตรวจสอบรายการข้อความสุดท้ายและเมทาดาทาการรันหลังเสร็จสิ้น
- **`before_compaction` / `after_compaction`**: สังเกตหรือใส่คำอธิบายประกอบให้รอบ Compaction
- **`before_tool_call` / `after_tool_call`**: ดักพารามิเตอร์/ผลลัพธ์ของเครื่องมือ
- **`before_install`**: ตรวจสอบผลการสแกนในตัวและเลือกบล็อกการติดตั้ง skill หรือ Plugin ได้
- **`tool_result_persist`**: แปลงผลลัพธ์ของเครื่องมือแบบซิงโครนัสก่อนเขียนลงทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ
- **`message_received` / `message_sending` / `message_sent`**: Hook ข้อความขาเข้า + ขาออก
- **`session_start` / `session_end`**: ขอบเขตวงจรชีวิตเซสชัน
- **`gateway_start` / `gateway_stop`**: เหตุการณ์วงจรชีวิต Gateway

กฎการตัดสินใจของ Hook สำหรับตัวป้องกันขาออก/เครื่องมือ:

- `before_tool_call`: `{ block: true }` เป็นจุดสิ้นสุดและหยุดตัวจัดการลำดับความสำคัญต่ำกว่า
- `before_tool_call`: `{ block: false }` เป็นการไม่ทำอะไรและไม่ล้างการบล็อกก่อนหน้า
- `before_install`: `{ block: true }` เป็นจุดสิ้นสุดและหยุดตัวจัดการลำดับความสำคัญต่ำกว่า
- `before_install`: `{ block: false }` เป็นการไม่ทำอะไรและไม่ล้างการบล็อกก่อนหน้า
- `message_sending`: `{ cancel: true }` เป็นจุดสิ้นสุดและหยุดตัวจัดการลำดับความสำคัญต่ำกว่า
- `message_sending`: `{ cancel: false }` เป็นการไม่ทำอะไรและไม่ล้างการยกเลิกก่อนหน้า

ดู [Hook ของ Plugin](/th/plugins/hooks) สำหรับ API ของ Hook และรายละเอียดการลงทะเบียน

ชุดทดสอบอาจปรับใช้ Hook เหล่านี้ต่างกัน ชุดทดสอบเซิร์ฟเวอร์แอป Codex รักษา
Hook ของ Plugin ใน OpenClaw ให้เป็นสัญญาความเข้ากันได้สำหรับพื้นผิวที่สะท้อนตามเอกสาร
ขณะที่ Hook แบบเนทีฟของ Codex ยังคงเป็นกลไก Codex ระดับต่ำแยกต่างหาก

## การสตรีม + คำตอบบางส่วน

- เดลตาของผู้ช่วยถูกสตรีมจาก pi-agent-core และปล่อยเป็นเหตุการณ์ `assistant`
- การสตรีมแบบบล็อกสามารถปล่อยคำตอบบางส่วนได้ทั้งบน `text_end` หรือ `message_end`
- การสตรีมเหตุผลสามารถถูกปล่อยเป็นสตรีมแยก หรือเป็นคำตอบแบบบล็อก
- ดู [การสตรีม](/th/concepts/streaming) สำหรับพฤติกรรมการแบ่งชิ้นและคำตอบแบบบล็อก

## การเรียกใช้เครื่องมือ + เครื่องมือส่งข้อความ

- เหตุการณ์เริ่ม/อัปเดต/จบของเครื่องมือถูกปล่อยบนสตรีม `tool`
- ผลลัพธ์เครื่องมือถูกทำความสะอาดเรื่องขนาดและเพย์โหลดรูปภาพก่อนบันทึก/ปล่อย
- การส่งของเครื่องมือส่งข้อความถูกติดตามเพื่อระงับการยืนยันซ้ำจากผู้ช่วย

## การจัดรูปคำตอบ + การระงับ

- เพย์โหลดสุดท้ายถูกประกอบจาก:
  - ข้อความของผู้ช่วย (และเหตุผลที่เป็นตัวเลือก)
  - สรุปเครื่องมือแบบอินไลน์ (เมื่อ verbose + อนุญาต)
  - ข้อความข้อผิดพลาดของผู้ช่วยเมื่อโมเดลเกิดข้อผิดพลาด
- โทเคนเงียบที่ตรงตัว `NO_REPLY` / `no_reply` ถูกกรองออกจาก
  เพย์โหลดขาออก
- รายการซ้ำของเครื่องมือส่งข้อความถูกลบออกจากรายการเพย์โหลดสุดท้าย
- หากไม่มีเพย์โหลดที่เรนเดอร์ได้เหลืออยู่และเครื่องมือเกิดข้อผิดพลาด จะปล่อยคำตอบข้อผิดพลาดเครื่องมือสำรอง
  (ยกเว้นเครื่องมือส่งข้อความได้ส่งคำตอบที่ผู้ใช้เห็นได้ไปแล้ว)

## Compaction + การลองใหม่

- Auto-compaction ปล่อยเหตุการณ์สตรีม `compaction` และสามารถกระตุ้นการลองใหม่
- เมื่อลองใหม่ บัฟเฟอร์ในหน่วยความจำและสรุปเครื่องมือจะถูกรีเซ็ตเพื่อหลีกเลี่ยงเอาต์พุตซ้ำ
- ดู [Compaction](/th/concepts/compaction) สำหรับไปป์ไลน์ Compaction

## สตรีมเหตุการณ์ (ปัจจุบัน)

- `lifecycle`: ปล่อยโดย `subscribeEmbeddedPiSession` (และโดย `agentCommand` เป็นกลไกสำรอง)
- `assistant`: เดลตาที่สตรีมจาก pi-agent-core
- `tool`: เหตุการณ์เครื่องมือที่สตรีมจาก pi-agent-core

## การจัดการช่องทางแชท

- เดลตาของผู้ช่วยถูกบัฟเฟอร์เป็นข้อความ `delta` ของแชท
- `final` ของแชทถูกปล่อยเมื่อ **จบ/ข้อผิดพลาดของวงจรชีวิต**

## การหมดเวลา

- ค่าเริ่มต้นของ `agent.wait`: 30 วินาที (เฉพาะการรอ) พารามิเตอร์ `timeoutMs` แทนที่ได้
- รันไทม์เอเจนต์: ค่าเริ่มต้นของ `agents.defaults.timeoutSeconds` คือ 172800 วินาที (48 ชั่วโมง); บังคับใช้ในตัวจับเวลายกเลิกของ `runEmbeddedPiAgent`
- รันไทม์ Cron: `timeoutSeconds` ของเทิร์นเอเจนต์แบบแยกเป็นของ Cron ตัวกำหนดเวลาเริ่มตัวจับเวลานี้เมื่อการดำเนินการเริ่ม ยกเลิกรันพื้นฐานเมื่อถึงเส้นตายที่กำหนดค่าไว้ จากนั้นทำการล้างข้อมูลแบบมีขอบเขตก่อนบันทึกการหมดเวลา เพื่อไม่ให้เซสชันลูกที่ค้างอยู่ทำให้เลนติดค้าง
- การวินิจฉัยความมีชีวิตของเซสชัน: เมื่อเปิดใช้การวินิจฉัย `diagnostics.stuckSessionWarnMs` จะจัดประเภทเซสชัน `processing` ที่กินเวลานานและไม่มีการตอบกลับ เครื่องมือ สถานะ บล็อก หรือความคืบหน้า ACP ที่สังเกตได้ การรันแบบฝัง การเรียกโมเดล และการเรียกเครื่องมือที่ยังทำงานอยู่จะรายงานเป็น `session.long_running`; งานที่ยังทำงานอยู่แต่ไม่มีความคืบหน้าล่าสุดจะรายงานเป็น `session.stalled`; `session.stuck` สงวนไว้สำหรับการทำบัญชีเซสชันที่ค้างโดยไม่มีงานที่ยังทำงานอยู่ การทำบัญชีเซสชันที่ค้างจะปล่อยเลนเซสชันที่ได้รับผลกระทบทันที; การรันแบบฝังที่หยุดค้างจะถูกยกเลิกและระบายออกหลังหน้าต่างไม่มีความคืบหน้าแบบขยายเท่านั้น (อย่างน้อย 10 นาที และ 5 เท่าของเกณฑ์เตือน) เพื่อให้งานที่เข้าคิวกลับมาทำต่อได้โดยไม่ตัดการรันที่เพียงแค่ช้า การวินิจฉัย `session.stuck` ที่ซ้ำกันจะถอยระยะขณะเซสชันยังไม่เปลี่ยน
- หมดเวลาเมื่อโมเดลว่าง: OpenClaw ยกเลิกคำขอโมเดลเมื่อไม่มีชิ้นส่วนคำตอบมาถึงก่อนหน้าต่างเวลาว่าง `models.providers.<id>.timeoutSeconds` ขยายตัวเฝ้าดูเวลาว่างนี้สำหรับผู้ให้บริการ local/self-hosted ที่ช้า มิฉะนั้น OpenClaw ใช้ `agents.defaults.timeoutSeconds` เมื่อกำหนดค่าไว้ โดยปกติถูกจำกัดไว้ที่ 120 วินาที การรันที่ Cron กระตุ้นโดยไม่มีเวลาหมดของโมเดลหรือเอเจนต์อย่างชัดเจนจะปิดใช้ตัวเฝ้าดูเวลาว่างและพึ่งพาเวลาหมดชั้นนอกของ Cron
- หมดเวลาคำขอ HTTP ของผู้ให้บริการ: `models.providers.<id>.timeoutSeconds` ใช้กับการดึงข้อมูล HTTP ของโมเดลสำหรับผู้ให้บริการนั้น รวมถึงการเชื่อมต่อ ส่วนหัว เนื้อหา หมดเวลาคำขอของ SDK การจัดการยกเลิก guarded-fetch ทั้งหมด และตัวเฝ้าดูเวลาว่างของสตรีมโมเดล ใช้สิ่งนี้สำหรับผู้ให้บริการ local/self-hosted ที่ช้า เช่น Ollama ก่อนเพิ่มเวลาหมดของรันไทม์เอเจนต์ทั้งหมด

## จุดที่สิ่งต่างๆ อาจจบก่อนกำหนด

- เอเจนต์หมดเวลา (ยกเลิก)
- AbortSignal (ยกเลิก)
- Gateway ตัดการเชื่อมต่อหรือ RPC หมดเวลา
- `agent.wait` หมดเวลา (รออย่างเดียว ไม่หยุดเอเจนต์)
