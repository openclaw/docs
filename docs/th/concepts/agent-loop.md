ลูปแบบเอเจนต์คือการรัน “จริง” แบบครบถ้วนของเอเจนต์: รับเข้า → ประกอบบริบท → อนุมานด้วยโมเดล →
ดำเนินการเครื่องมือ → สตรีมคำตอบ → คงข้อมูลไว้ เป็นเส้นทางที่เชื่อถือได้ซึ่งเปลี่ยนข้อความ
ให้เป็นการกระทำและคำตอบสุดท้าย พร้อมรักษาสถานะเซสชันให้สอดคล้องกัน

ใน OpenClaw ลูปคือการรันที่ถูกจัดลำดับครั้งเดียวต่อเซสชัน ซึ่งปล่อยเหตุการณ์วงจรชีวิตและสตรีม
ขณะที่โมเดลคิด เรียกเครื่องมือ และสตรีมเอาต์พุต เอกสารนี้อธิบายวิธีเชื่อมต่อลูปจริงนั้น
ตั้งแต่ต้นจนจบ

## จุดเข้าใช้งาน

- RPC ของ Gateway: `agent` และ `agent.wait`
- CLI: คำสั่ง `agent`

## วิธีทำงาน (ภาพรวมระดับสูง)

1. RPC `agent` ตรวจสอบพารามิเตอร์ แก้ไขเซสชัน (sessionKey/sessionId) คงข้อมูลเมตาของเซสชัน แล้วส่งคืน `{ runId, acceptedAt }` ทันที
2. `agentCommand` รันเอเจนต์:
   - แก้ไขค่าเริ่มต้นของโมเดล + thinking/verbose/trace
   - โหลดสแนปช็อต Skills
   - เรียก `runEmbeddedPiAgent` (รันไทม์ pi-agent-core)
   - ปล่อย **end/error ของวงจรชีวิต** หากลูปแบบฝังไม่ได้ปล่อยเอง
3. `runEmbeddedPiAgent`:
   - จัดลำดับการรันผ่านคิวต่อเซสชัน + คิวส่วนกลาง
   - แก้ไขโมเดล + โปรไฟล์ยืนยันตัวตน และสร้างเซสชัน pi
   - สมัครรับเหตุการณ์ pi และสตรีมเดลตาของผู้ช่วย/เครื่องมือ
   - บังคับใช้เวลาหมด -> ยกเลิกการรันหากเกินกำหนด
   - สำหรับเทิร์นของแอปเซิร์ฟเวอร์ Codex ยกเลิกเทิร์นที่รับแล้วซึ่งหยุดสร้างความคืบหน้าของแอปเซิร์ฟเวอร์ก่อนเหตุการณ์สิ้นสุด
   - ส่งคืน payload + เมตาดาต้าการใช้งาน
4. `subscribeEmbeddedPiSession` เชื่อมเหตุการณ์ pi-agent-core ไปยังสตรีม `agent` ของ OpenClaw:
   - เหตุการณ์เครื่องมือ => `stream: "tool"`
   - เดลตาของผู้ช่วย => `stream: "assistant"`
   - เหตุการณ์วงจรชีวิต => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` ใช้ `waitForAgentRun`:
   - รอ **end/error ของวงจรชีวิต** สำหรับ `runId`
   - ส่งคืน `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## การเข้าคิว + การทำงานพร้อมกัน

- การรันถูกจัดลำดับต่อคีย์เซสชัน (เลนเซสชัน) และอาจผ่านเลนส่วนกลางด้วย
- สิ่งนี้ป้องกันการแข่งขันของเครื่องมือ/เซสชัน และทำให้ประวัติเซสชันสอดคล้องกัน
- ช่องทางรับส่งข้อความสามารถเลือกโหมดคิว (collect/steer/followup) ที่ป้อนเข้าสู่ระบบเลนนี้ได้
  ดู [คิวคำสั่ง](/th/concepts/queue)
- การเขียนทรานสคริปต์ยังได้รับการป้องกันด้วยล็อกเขียนเซสชันบนไฟล์เซสชันด้วย ล็อกนี้
  รับรู้กระบวนการและอิงไฟล์ จึงตรวจจับตัวเขียนที่เลี่ยงคิวในกระบวนการหรือมาจาก
  อีกกระบวนการได้ ตัวเขียนทรานสคริปต์เซสชันจะรอสูงสุด `session.writeLock.acquireTimeoutMs`
  ก่อนรายงานว่าเซสชันไม่ว่าง ค่าเริ่มต้นคือ `60000` มิลลิวินาที
- โดยค่าเริ่มต้น ล็อกเขียนเซสชันไม่ใช่แบบเข้าใหม่ได้ หากตัวช่วยตั้งใจซ้อนการขอล็อกเดียวกัน
  ในขณะที่ยังคงตัวเขียนเชิงตรรกะเพียงตัวเดียว จะต้องเลือกใช้โดยชัดเจนด้วย
  `allowReentrant: true`

## การเตรียมเซสชัน + เวิร์กสเปซ

- เวิร์กสเปซถูกแก้ไขและสร้างขึ้น การรันแบบ sandbox อาจเปลี่ยนเส้นทางไปยังรากเวิร์กสเปซ sandbox
- Skills ถูกโหลด (หรือนำกลับมาใช้จากสแนปช็อต) และฉีดเข้า env และพรอมป์
- ไฟล์บูตสแตรป/บริบทถูกแก้ไขและฉีดเข้าไปในรายงานพรอมป์ระบบ
- มีการขอล็อกเขียนเซสชัน จากนั้นเปิดและเตรียม `SessionManager` ก่อนสตรีม เส้นทางใดก็ตามในภายหลัง
  ที่เขียนทรานสคริปต์ใหม่ ทำ Compaction หรือตัดทอน ต้องใช้ล็อกเดียวกันก่อนเปิดหรือ
  แก้ไขไฟล์ทรานสคริปต์

## การประกอบพรอมป์ + พรอมป์ระบบ

- พรอมป์ระบบถูกสร้างจากพรอมป์พื้นฐานของ OpenClaw, พรอมป์ Skills, บริบทบูตสแตรป และการแทนที่เฉพาะการรัน
- มีการบังคับใช้ขีดจำกัดเฉพาะโมเดลและโทเคนสำรองสำหรับ Compaction
- ดู [พรอมป์ระบบ](/th/concepts/system-prompt) สำหรับสิ่งที่โมเดลเห็น

## จุด Hook (ที่ที่คุณสามารถดักแทรกได้)

OpenClaw มีระบบ Hook สองระบบ:

- **Hook ภายใน** (Hook ของ Gateway): สคริปต์ขับเคลื่อนด้วยเหตุการณ์สำหรับคำสั่งและเหตุการณ์วงจรชีวิต
- **Hook ของ Plugin**: จุดขยายภายในวงจรชีวิตเอเจนต์/เครื่องมือและไปป์ไลน์ Gateway

### Hook ภายใน (Hook ของ Gateway)

- **`agent:bootstrap`**: รันขณะสร้างไฟล์บูตสแตรปก่อนสรุปพรอมป์ระบบ
  ใช้สิ่งนี้เพื่อเพิ่ม/ลบไฟล์บริบทบูตสแตรป
- **Hook คำสั่ง**: `/new`, `/reset`, `/stop` และเหตุการณ์คำสั่งอื่น ๆ (ดูเอกสาร Hook)

ดู [Hook](/th/automation/hooks) สำหรับการตั้งค่าและตัวอย่าง

### Hook ของ Plugin (วงจรชีวิตเอเจนต์ + Gateway)

สิ่งเหล่านี้รันภายในลูปเอเจนต์หรือไปป์ไลน์ Gateway:

- **`before_model_resolve`**: รันก่อนเซสชัน (ไม่มี `messages`) เพื่อแทนที่ผู้ให้บริการ/โมเดลแบบกำหนดแน่นอนก่อนการแก้ไขโมเดล
- **`before_prompt_build`**: รันหลังโหลดเซสชัน (พร้อม `messages`) เพื่อฉีด `prependContext`, `systemPrompt`, `prependSystemContext` หรือ `appendSystemContext` ก่อนส่งพรอมป์ ใช้ `prependContext` สำหรับข้อความไดนามิกต่อเทิร์น และใช้ฟิลด์บริบทระบบสำหรับคำแนะนำที่เสถียรซึ่งควรอยู่ในพื้นที่พรอมป์ระบบ
- **`before_agent_start`**: Hook ความเข้ากันได้แบบเดิมที่อาจรันได้ในทั้งสองเฟส ควรใช้ Hook ที่ชัดเจนด้านบน
- **`before_agent_reply`**: รันหลังการกระทำแบบ inline และก่อนการเรียก LLM ทำให้ Plugin รับช่วงเทิร์นและส่งคืนคำตอบสังเคราะห์หรือปิดเสียงเทิร์นทั้งหมดได้
- **`agent_end`**: ตรวจสอบรายการข้อความสุดท้ายและเมตาดาต้าการรันหลังเสร็จสิ้น
- **`before_compaction` / `after_compaction`**: สังเกตหรือใส่คำอธิบายประกอบรอบ Compaction
- **`before_tool_call` / `after_tool_call`**: ดักพารามิเตอร์/ผลลัพธ์ของเครื่องมือ
- **`before_install`**: ตรวจสอบผลการสแกนในตัว และเลือกบล็อกการติดตั้ง Skill หรือ Plugin ได้
- **`tool_result_persist`**: แปลงผลลัพธ์เครื่องมือแบบซิงโครนัสก่อนเขียนลงทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ
- **`message_received` / `message_sending` / `message_sent`**: Hook ข้อความขาเข้า + ขาออก
- **`session_start` / `session_end`**: ขอบเขตวงจรชีวิตเซสชัน
- **`gateway_start` / `gateway_stop`**: เหตุการณ์วงจรชีวิต Gateway

กฎการตัดสินใจของ Hook สำหรับตัวคุมขาออก/เครื่องมือ:

- `before_tool_call`: `{ block: true }` เป็นสถานะสิ้นสุดและหยุดตัวจัดการที่มีลำดับความสำคัญต่ำกว่า
- `before_tool_call`: `{ block: false }` เป็น no-op และไม่ล้างการบล็อกก่อนหน้า
- `before_install`: `{ block: true }` เป็นสถานะสิ้นสุดและหยุดตัวจัดการที่มีลำดับความสำคัญต่ำกว่า
- `before_install`: `{ block: false }` เป็น no-op และไม่ล้างการบล็อกก่อนหน้า
- `message_sending`: `{ cancel: true }` เป็นสถานะสิ้นสุดและหยุดตัวจัดการที่มีลำดับความสำคัญต่ำกว่า
- `message_sending`: `{ cancel: false }` เป็น no-op และไม่ล้างการยกเลิกก่อนหน้า

ดู [Hook ของ Plugin](/th/plugins/hooks) สำหรับ API ของ Hook และรายละเอียดการลงทะเบียน

ชุดทดสอบอาจปรับใช้ Hook เหล่านี้แตกต่างกัน ชุดทดสอบแอปเซิร์ฟเวอร์ Codex คง
Hook ของ Plugin ของ OpenClaw ไว้เป็นสัญญาความเข้ากันได้สำหรับพื้นผิวที่มิเรอร์และมีเอกสาร
ส่วน Hook ดั้งเดิมของ Codex ยังคงเป็นกลไก Codex ระดับต่ำที่แยกต่างหาก

## การสตรีม + คำตอบบางส่วน

- เดลตาของผู้ช่วยถูกสตรีมจาก pi-agent-core และปล่อยเป็นเหตุการณ์ `assistant`
- การสตรีมแบบบล็อกสามารถปล่อยคำตอบบางส่วนได้ทั้งบน `text_end` หรือ `message_end`
- การสตรีมเหตุผลสามารถถูกปล่อยเป็นสตรีมแยกต่างหากหรือเป็นคำตอบแบบบล็อก
- ดู [การสตรีม](/th/concepts/streaming) สำหรับพฤติกรรมการแบ่งชิ้นและคำตอบแบบบล็อก

## การดำเนินการเครื่องมือ + เครื่องมือรับส่งข้อความ

- เหตุการณ์เริ่ม/อัปเดต/สิ้นสุดของเครื่องมือถูกปล่อยบนสตรีม `tool`
- ผลลัพธ์เครื่องมือถูกทำให้ปลอดภัยด้านขนาดและ payload รูปภาพก่อนบันทึก/ปล่อยเหตุการณ์
- การส่งของเครื่องมือรับส่งข้อความถูกติดตามเพื่อระงับการยืนยันซ้ำจากผู้ช่วย

## การจัดรูปคำตอบ + การระงับ

- payload สุดท้ายถูกประกอบจาก:
  - ข้อความผู้ช่วย (และเหตุผลที่เป็นทางเลือก)
  - สรุปเครื่องมือแบบ inline (เมื่อ verbose + อนุญาต)
  - ข้อความข้อผิดพลาดของผู้ช่วยเมื่อโมเดลผิดพลาด
- โทเคนเงียบที่ตรงเป๊ะ `NO_REPLY` / `no_reply` ถูกกรองออกจาก
  payload ขาออก
- รายการซ้ำของเครื่องมือรับส่งข้อความถูกลบออกจากรายการ payload สุดท้าย
- หากไม่มี payload ที่เรนเดอร์ได้เหลืออยู่และเครื่องมือเกิดข้อผิดพลาด จะปล่อยคำตอบข้อผิดพลาดเครื่องมือสำรอง
  (เว้นแต่เครื่องมือรับส่งข้อความส่งคำตอบที่ผู้ใช้เห็นได้ไปแล้ว)

## Compaction + การลองใหม่

- Auto-compaction ปล่อยเหตุการณ์สตรีม `compaction` และสามารถทริกเกอร์การลองใหม่ได้
- เมื่อลองใหม่ บัฟเฟอร์ในหน่วยความจำและสรุปเครื่องมือจะถูกรีเซ็ตเพื่อหลีกเลี่ยงเอาต์พุตซ้ำ
- ดู [Compaction](/th/concepts/compaction) สำหรับไปป์ไลน์ Compaction

## สตรีมเหตุการณ์ (ปัจจุบัน)

- `lifecycle`: ปล่อยโดย `subscribeEmbeddedPiSession` (และเป็นตัวสำรองโดย `agentCommand`)
- `assistant`: เดลตาที่สตรีมจาก pi-agent-core
- `tool`: เหตุการณ์เครื่องมือที่สตรีมจาก pi-agent-core

## การจัดการช่องทางแชต

- เดลตาของผู้ช่วยถูกบัฟเฟอร์เป็นข้อความ `delta` ของแชต
- `final` ของแชตถูกปล่อยเมื่อเกิด **end/error ของวงจรชีวิต**

## เวลาหมด

- ค่าเริ่มต้นของ `agent.wait`: 30 วินาที (เฉพาะการรอ) พารามิเตอร์ `timeoutMs` ใช้แทนได้
- รันไทม์เอเจนต์: ค่าเริ่มต้นของ `agents.defaults.timeoutSeconds` คือ 172800 วินาที (48 ชั่วโมง); บังคับใช้ในตัวจับเวลายกเลิกของ `runEmbeddedPiAgent`
- รันไทม์ Cron: `timeoutSeconds` ของเทิร์นเอเจนต์แบบแยกเป็นของ cron ตัวกำหนดเวลาจะเริ่มตัวจับเวลานั้นเมื่อเริ่มดำเนินการ ยกเลิกการรันเบื้องหลังเมื่อถึงกำหนดเวลาที่ตั้งไว้ จากนั้นรันการล้างข้อมูลแบบมีขอบเขตก่อนบันทึกเวลาหมด เพื่อให้เซสชันลูกที่ค้างไม่ทำให้เลนติดอยู่
- การวินิจฉัยความมีชีวิตของเซสชัน: เมื่อเปิดใช้ diagnostics, `diagnostics.stuckSessionWarnMs` จัดประเภทเซสชัน `processing` ที่ยาวนานซึ่งไม่มีคำตอบ เครื่องมือ สถานะ บล็อก หรือความคืบหน้า ACP ที่สังเกตได้ การรันแบบฝัง การเรียกโมเดล และการเรียกเครื่องมือที่ยังทำงานอยู่จะรายงานเป็น `session.long_running`; งานที่ยังทำงานอยู่แต่ไม่มีความคืบหน้าล่าสุดจะรายงานเป็น `session.stalled`; `session.stuck` สงวนไว้สำหรับการทำบัญชีเซสชันที่ค้างโดยไม่มีงานที่ทำงานอยู่ และมีเพียงเส้นทางนั้นเท่านั้นที่ปล่อยเลนเซสชันที่ได้รับผลกระทบเพื่อให้งานเริ่มต้นที่อยู่ในคิวระบายออกได้ diagnostics `session.stuck` ที่เกิดซ้ำจะถอยระยะขณะเซสชันยังไม่เปลี่ยนแปลง
- เวลาหมดเมื่อโมเดลไม่มีการตอบสนอง: OpenClaw ยกเลิกคำขอโมเดลเมื่อไม่มีชิ้นส่วนคำตอบมาถึงก่อนหน้าต่างเวลาว่าง `models.providers.<id>.timeoutSeconds` ขยาย watchdog เวลาว่างนี้สำหรับผู้ให้บริการ local/self-hosted ที่ช้า มิฉะนั้น OpenClaw ใช้ `agents.defaults.timeoutSeconds` เมื่อกำหนดค่าไว้ โดยค่าเริ่มต้นจำกัดสูงสุดที่ 120 วินาที การรันที่ทริกเกอร์โดย Cron ซึ่งไม่มีเวลาหมดของโมเดลหรือเอเจนต์ที่ชัดเจนจะปิดใช้ watchdog เวลาว่างและพึ่งพาเวลาหมดภายนอกของ cron
- เวลาหมดของคำขอ HTTP ของผู้ให้บริการ: `models.providers.<id>.timeoutSeconds` ใช้กับการ fetch HTTP ของโมเดลของผู้ให้บริการนั้น รวมถึงการเชื่อมต่อ เฮดเดอร์ เนื้อหา เวลาหมดของคำขอ SDK การจัดการยกเลิก guarded-fetch ทั้งหมด และ watchdog เวลาว่างของสตรีมโมเดล ใช้ค่านี้สำหรับผู้ให้บริการ local/self-hosted ที่ช้า เช่น Ollama ก่อนเพิ่มเวลาหมดของรันไทม์เอเจนต์ทั้งหมด

## จุดที่สิ่งต่าง ๆ สามารถจบก่อนกำหนด

- เวลาหมดของเอเจนต์ (ยกเลิก)
- AbortSignal (ยกเลิก)
- Gateway ตัดการเชื่อมต่อหรือ RPC หมดเวลา
- `agent.wait` หมดเวลา (เฉพาะการรอ ไม่หยุดเอเจนต์)
