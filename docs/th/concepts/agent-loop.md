ลูปเอเจนต์คือการรันแบบเรียงลำดับต่อเซสชัน ซึ่งเปลี่ยนข้อความให้เป็น
การดำเนินการและคำตอบ ได้แก่ การรับข้อมูล การประกอบบริบท การอนุมานของโมเดล การเรียกใช้เครื่องมือ
การสตรีม และการคงข้อมูล

## จุดเข้าใช้งาน

- RPC ของ Gateway: `agent` และ `agent.wait`
- CLI: `openclaw agent`

## ลำดับการรัน

1. `agent` RPC ตรวจสอบพารามิเตอร์ แก้ไขเซสชัน (`sessionKey`/`sessionId`) คงข้อมูลเมตาของเซสชัน และส่งคืน `{ runId, acceptedAt }` ทันที
2. `agentCommand` รันรอบการทำงาน โดยแก้ไขค่าเริ่มต้นของโมเดล + การคิด/ความละเอียด/การติดตาม โหลดสแนปช็อต Skills เรียก `runEmbeddedAgent` และปล่อย **สิ้นสุดวงจรชีวิต/ข้อผิดพลาด** สำรอง หากลูปที่ฝังอยู่ยังไม่ได้ปล่อยเหตุการณ์ดังกล่าว
3. `runEmbeddedAgent`: จัดลำดับการรันผ่านคิวต่อเซสชันและคิวส่วนกลาง แก้ไขโมเดล + โปรไฟล์การยืนยันตัวตน สร้างเซสชัน OpenClaw สมัครรับเหตุการณ์รันไทม์ สตรีมเดลตาของผู้ช่วย/เครื่องมือ บังคับใช้ระยะหมดเวลาของการรัน (ยกเลิกเมื่อหมดเวลา) และส่งคืนเพย์โหลดพร้อมข้อมูลเมตาการใช้งาน สำหรับรอบการทำงานของ app-server ของ Codex ระบบจะยกเลิกรอบการทำงานที่ยอมรับแล้วด้วย หากหยุดสร้างความคืบหน้าของ app-server ก่อนเกิดเหตุการณ์สิ้นสุด
4. `subscribeEmbeddedAgentSession` เชื่อมเหตุการณ์รันไทม์เข้ากับสตรีม `agent`: เหตุการณ์เครื่องมือไปยัง `stream: "tool"` เดลตาของผู้ช่วยไปยัง `stream: "assistant"` และเหตุการณ์วงจรชีวิตไปยัง `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` (`waitForAgentRun`) รอ **สิ้นสุดวงจรชีวิต/ข้อผิดพลาด** บน `runId` และส่งคืน `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## การจัดคิวและการทำงานพร้อมกัน

การรันจะถูกจัดลำดับตามคีย์เซสชัน (เลนเซสชัน) และอาจผ่านเลนส่วนกลางด้วย เพื่อป้องกันภาวะแข่งขันของเครื่องมือ/เซสชัน ช่องทางการรับส่งข้อความจะเลือกโหมดคิว (steer/followup/collect/interrupt) ที่ป้อนเข้าสู่ระบบเลนนี้ โปรดดู [คิวคำสั่ง](/th/concepts/queue)

การเขียนทรานสคริปต์ยังได้รับการป้องกันเพิ่มเติมด้วยล็อกการเขียนเซสชันบนไฟล์เซสชัน ล็อกนี้รับรู้กระบวนการและทำงานบนไฟล์ จึงตรวจจับตัวเขียนที่ข้ามคิวภายในกระบวนการหรือมาจากกระบวนการอื่นได้ ตัวเขียนจะรอสูงสุด `session.writeLock.acquireTimeoutMs` (ค่าเริ่มต้น `60000` ms; กำหนดทับผ่านสภาพแวดล้อมด้วย `OPENCLAW_SESSION_WRITE_LOCK_ACQUIRE_TIMEOUT_MS`) ก่อนรายงานว่าเซสชันไม่ว่าง

โดยค่าเริ่มต้น ล็อกการเขียนเซสชันไม่รองรับการกลับเข้าใช้ซ้ำ ตัวช่วยที่ตั้งใจซ้อนการขอล็อกเดียวกัน โดยยังคงมีตัวเขียนเชิงตรรกะเพียงหนึ่งตัว ต้องเลือกใช้ `allowReentrant: true`

## การเตรียมเซสชันและพื้นที่ทำงาน

- แก้ไขและสร้างพื้นที่ทำงาน โดยการรันในแซนด์บ็อกซ์อาจเปลี่ยนเส้นทางไปยังรากพื้นที่ทำงานของแซนด์บ็อกซ์
- โหลด Skills (หรือนำกลับมาใช้จากสแนปช็อต) และแทรกลงในสภาพแวดล้อมและพรอมต์
- แก้ไขไฟล์บูตสแตรป/บริบท และแทรกลงในพรอมต์ระบบ
- ขอล็อกการเขียนเซสชันและเตรียมเป้าหมายทรานสคริปต์ของเซสชันก่อนเริ่มการสตรีม เส้นทางการเขียนทรานสคริปต์ใหม่ การทำ Compaction หรือการตัดทอนในภายหลังทั้งหมด ต้องขอล็อกเดียวกันก่อนแก้ไขแถวทรานสคริปต์ใน SQLite

## การประกอบพรอมต์

พรอมต์ระบบสร้างขึ้นจากพรอมต์พื้นฐานของ OpenClaw พรอมต์ Skills บริบทบูตสแตรป และการกำหนดทับต่อการรัน ระบบบังคับใช้ขีดจำกัดเฉพาะโมเดลและโทเค็นสำรองสำหรับ Compaction โปรดดู [พรอมต์ระบบ](/th/concepts/system-prompt) สำหรับสิ่งที่โมเดลมองเห็น

## ฮุก

OpenClaw มีระบบฮุกสองระบบ:

- **ฮุกภายใน** (ฮุก Gateway): สคริปต์ที่ขับเคลื่อนด้วยเหตุการณ์สำหรับคำสั่งและเหตุการณ์วงจรชีวิต
- **ฮุก Plugin**: จุดขยายภายในวงจรชีวิตของเอเจนต์/เครื่องมือและไปป์ไลน์ Gateway

### ฮุกภายใน (ฮุก Gateway)

- **`agent:bootstrap`**: ทำงานขณะสร้างไฟล์บูตสแตรปก่อนพรอมต์ระบบจะเสร็จสมบูรณ์ ใช้เพื่อเพิ่มหรือลบไฟล์บริบทบูตสแตรป
- **ฮุกคำสั่ง**: `/new`, `/reset`, `/stop` และเหตุการณ์คำสั่งอื่น ๆ (โปรดดูเอกสารฮุก)

โปรดดู [ฮุก](/th/automation/hooks) สำหรับการตั้งค่าและตัวอย่าง

### ฮุก Plugin

ฮุกเหล่านี้ทำงานภายในลูปเอเจนต์หรือไปป์ไลน์ Gateway:

| ฮุก                                                    | การทำงาน                                                                                                                                                                                                                                                                                        |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `before_model_resolve`                                  | ก่อนเซสชัน (ไม่มี `messages`) เพื่อกำหนดทับผู้ให้บริการ/โมเดลแบบกำหนดผลได้ก่อนการแก้ไข                                                                                                                                                                                                |
| `before_prompt_build`                                   | หลังโหลดเซสชัน (พร้อม `messages`) เพื่อแทรก `prependContext`, `systemPrompt`, `prependSystemContext` หรือ `appendSystemContext` ก่อนส่ง ใช้ `prependContext` สำหรับข้อความแบบไดนามิกต่อรอบ และใช้ฟิลด์บริบทระบบสำหรับคำแนะนำที่คงที่ซึ่งควรอยู่ในพื้นที่พรอมต์ระบบ |
| `before_agent_start`                                    | ฮุกความเข้ากันได้แบบเดิมที่อาจทำงานในเฟสใดเฟสหนึ่ง ควรใช้ฮุกแบบชัดเจนด้านบน                                                                                                                                                                                                    |
| `before_agent_reply`                                    | หลังการดำเนินการแบบอินไลน์ ก่อนเรียก LLM ช่วยให้ Plugin รับช่วงรอบการทำงานและส่งคืนคำตอบสังเคราะห์หรือปิดเสียงทั้งหมดได้                                                                                                                                                                |
| `agent_end`                                             | หลังเสร็จสิ้น พร้อมรายการข้อความสุดท้ายและข้อมูลเมตาการรัน                                                                                                                                                                                                                             |
| `before_compaction` / `after_compaction`                | สังเกตหรือใส่คำอธิบายประกอบวงจร Compaction                                                                                                                                                                                                                                                      |
| `before_tool_call` / `after_tool_call`                  | ดักจับพารามิเตอร์/ผลลัพธ์ของเครื่องมือ                                                                                                                                                                                                                                                              |
| `before_install`                                        | หลังนโยบายการติดตั้งของผู้ดำเนินการทำงาน บนเนื้อหาการติดตั้ง Skills/Plugin ที่จัดเตรียมไว้ เมื่อโหลดฮุก Plugin ในกระบวนการปัจจุบันแล้ว                                                                                                                                                           |
| `tool_result_persist`                                   | แปลงผลลัพธ์ของเครื่องมือแบบซิงโครนัสก่อนเขียนลงในทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ                                                                                                                                                                                      |
| `message_received` / `message_sending` / `message_sent` | ฮุกข้อความขาเข้าและขาออก                                                                                                                                                                                                                                                         |
| `session_start` / `session_end`                         | ขอบเขตวงจรชีวิตของเซสชัน                                                                                                                                                                                                                                                               |
| `gateway_start` / `gateway_stop`                        | เหตุการณ์วงจรชีวิตของ Gateway                                                                                                                                                                                                                                                                   |

กฎการตัดสินใจของฮุกสำหรับตัวป้องกันขาออก/เครื่องมือ:

- `before_tool_call`: `{ block: true }` เป็นสถานะสิ้นสุดและหยุดตัวจัดการที่มีลำดับความสำคัญต่ำกว่า `{ block: false }` ไม่ดำเนินการใด ๆ และไม่ล้างการบล็อกก่อนหน้า
- `before_install`: ใช้ความหมายของสถานะสิ้นสุด/ไม่ดำเนินการเช่นเดียวกับด้านบน ใช้ `security.installPolicy` ไม่ใช่ `before_install` สำหรับการตัดสินใจอนุญาต/บล็อกการติดตั้งที่ผู้ดำเนินการเป็นเจ้าของ ซึ่งต้องครอบคลุมเส้นทางการติดตั้งและอัปเดตผ่าน CLI
- `message_sending`: `{ cancel: true }` เป็นสถานะสิ้นสุดและหยุดตัวจัดการที่มีลำดับความสำคัญต่ำกว่า `{ cancel: false }` ไม่ดำเนินการใด ๆ และไม่ล้างการยกเลิกก่อนหน้า

โปรดดู [ฮุก Plugin](/th/plugins/hooks) สำหรับ API ของฮุกและรายละเอียดการลงทะเบียน

ชุดทดสอบสามารถปรับใช้ฮุกเหล่านี้ได้ ชุดทดสอบ app-server ของ Codex ยังคงใช้ฮุก Plugin ของ OpenClaw เป็นสัญญาความเข้ากันได้สำหรับพื้นผิวจำลองที่มีการจัดทำเอกสาร ส่วนฮุกดั้งเดิมของ Codex เป็นกลไก Codex ระดับต่ำกว่าและแยกต่างหาก

## การสตรีม

- เดลตาของผู้ช่วยสตรีมจากรันไทม์เอเจนต์เป็นเหตุการณ์ `assistant`
- การสตรีมแบบบล็อกสามารถปล่อยคำตอบบางส่วนบน `text_end` หรือ `message_end`
- การสตรีมการให้เหตุผลอาจเป็นสตรีมแยกต่างหากหรือเป็นคำตอบแบบบล็อก
- โปรดดู [การสตรีม](/th/concepts/streaming) สำหรับพฤติกรรมการแบ่งส่วนและคำตอบแบบบล็อก

## การเรียกใช้เครื่องมือ

- เหตุการณ์เริ่ม/อัปเดต/สิ้นสุดของเครื่องมือจะปล่อยบนสตรีม `tool`
- ผลลัพธ์ของเครื่องมือจะถูกทำให้ปลอดภัยด้านขนาดและเพย์โหลดรูปภาพก่อนบันทึก/ปล่อย
- ระบบติดตามการส่งของเครื่องมือรับส่งข้อความเพื่อระงับการยืนยันซ้ำจากผู้ช่วย

## การจัดรูปแบบคำตอบ

เพย์โหลดสุดท้ายประกอบขึ้นจากข้อความของผู้ช่วย (รวมการให้เหตุผลที่เลือกเปิดใช้ได้) สรุปเครื่องมือแบบอินไลน์ (เมื่อเปิดโหมดละเอียดและได้รับอนุญาต) และข้อความข้อผิดพลาดของผู้ช่วยเมื่อโมเดลเกิดข้อผิดพลาด

- โทเค็นเงียบที่ตรงกันทุกประการ `NO_REPLY` จะถูกกรองออกจากเพย์โหลดขาออก
- รายการซ้ำจากเครื่องมือรับส่งข้อความจะถูกลบออกจากรายการเพย์โหลดสุดท้าย
- หากไม่เหลือเพย์โหลดที่แสดงผลได้และเครื่องมือเกิดข้อผิดพลาด ระบบจะปล่อยคำตอบข้อผิดพลาดของเครื่องมือสำรอง เว้นแต่เครื่องมือรับส่งข้อความได้ส่งคำตอบที่ผู้ใช้มองเห็นแล้ว

## Compaction และการลองใหม่

Compaction อัตโนมัติจะปล่อยเหตุการณ์สตรีม `compaction` และอาจเรียกให้ลองใหม่ เมื่อทดลองใหม่ บัฟเฟอร์ในหน่วยความจำและสรุปเครื่องมือจะถูกรีเซ็ตเพื่อหลีกเลี่ยงผลลัพธ์ซ้ำ โปรดดู [Compaction](/th/concepts/compaction)

## สตรีมเหตุการณ์

- `lifecycle`: ปล่อยโดย `subscribeEmbeddedAgentSession` (และปล่อยเป็นตัวสำรองโดย `agentCommand`)
- `assistant`: เดลตาที่สตรีมจากรันไทม์เอเจนต์
- `tool`: เหตุการณ์เครื่องมือที่สตรีมจากรันไทม์เอเจนต์

Gateway ฉายเหตุการณ์วงจรชีวิตและเหตุการณ์เริ่มต้น/สิ้นสุดของเครื่องมือไปยัง
[บัญชีแยกประเภทการตรวจสอบ](/th/cli/audit) ที่มีขอบเขตและเก็บเฉพาะข้อมูลเมตา การฉายนี้บันทึกแหล่งที่มาและ
รหัสผลลัพธ์ โดยไม่คัดลอกพรอมต์ ข้อความ อาร์กิวเมนต์ของเครื่องมือ ผลลัพธ์ของเครื่องมือ
หรือข้อผิดพลาดดิบออกจากเส้นทางทรานสคริปต์/รันไทม์

## การจัดการช่องทางแชต

เดลตาของผู้ช่วยจะสะสมในข้อความแชต `delta` ระบบจะปล่อย `final` ของแชตเมื่อเกิด **สิ้นสุดวงจรชีวิต/ข้อผิดพลาด**

## ระยะหมดเวลา

### การวินิจฉัยเซสชันที่ติดค้าง

เมื่อเปิดใช้การวินิจฉัย `diagnostics.stuckSessionWarnMs` (ค่าเริ่มต้น `120000` ms) จะจำแนกเซสชัน `processing` ที่ทำงานเป็นเวลานานโดยไม่พบการตอบกลับ เครื่องมือ สถานะ บล็อก หรือความคืบหน้าของ ACP ดังนี้:

- การทำงานแบบฝัง การเรียกโมเดล และการเรียกเครื่องมือที่กำลังดำเนินอยู่จะรายงานเป็น `session.long_running` การเรียกโมเดลแบบไม่มีเอาต์พุตที่มีเจ้าของจะคงเป็น `session.long_running` จนถึง `diagnostics.stuckSessionAbortMs` เพื่อไม่ให้ผู้ให้บริการที่ทำงานช้าหรือไม่ใช้สตรีมถูกระบุว่าหยุดค้างเร็วเกินไป
- งานที่กำลังดำเนินอยู่แต่ไม่มีความคืบหน้าล่าสุดจะรายงานเป็น `session.stalled` การเรียกโมเดลที่มีเจ้าของจะเปลี่ยนเป็น `session.stalled` เมื่อถึงหรือเลยเกณฑ์การยกเลิก ส่วนกิจกรรมของโมเดลหรือเครื่องมือที่ค้างและไม่มีเจ้าของจะไม่ถูกซ่อนว่าเป็นงานที่ใช้เวลานาน
- `session.stuck` สงวนไว้สำหรับข้อมูลการจัดการเซสชันที่ค้างซึ่งกู้คืนได้ รวมถึงเซสชันในคิวที่ไม่มีการใช้งานและมีกิจกรรมของโมเดลหรือเครื่องมือที่ค้างโดยไม่มีเจ้าของ

`diagnostics.stuckSessionAbortMs` มีค่าเริ่มต้นอย่างน้อย 5 นาทีและเป็น 3 เท่าของเกณฑ์การเตือน การจัดการข้อมูลเซสชันที่ค้างจะปล่อยเลนของเซสชันที่ได้รับผลกระทบทันทีหลังจากผ่านเงื่อนไขการกู้คืน ส่วนการทำงานแบบฝังที่หยุดค้างจะถูกยกเลิกและรอให้ระบายงานออกหลังจากถึงเกณฑ์การยกเลิกเท่านั้น เพื่อให้งานในคิวกลับมาทำงานต่อได้โดยไม่ตัดการทำงานที่เพียงแค่ช้า การกู้คืนจะส่งผลลัพธ์ที่มีโครงสร้างสำหรับคำขอและการดำเนินการเสร็จสิ้น โดยสถานะการวินิจฉัยจะถูกระบุว่าไม่มีการใช้งานต่อเมื่อเจเนอเรชันการประมวลผลเดียวกันยังเป็นค่าปัจจุบัน และการวินิจฉัย `session.stuck` ซ้ำจะเพิ่มช่วงหน่วงเมื่อเซสชันยังไม่มีการเปลี่ยนแปลง
