วงจรแบบเอเจนต์คือการรัน “จริง” แบบเต็มของเอเจนต์: การรับเข้า → การประกอบบริบท → การอนุมานของโมเดล →
การเรียกใช้เครื่องมือ → การสตรีมคำตอบ → การคงข้อมูลไว้ วงจรนี้เป็นเส้นทางหลักที่เชื่อถือได้ซึ่งเปลี่ยนข้อความ
ให้เป็นการกระทำและคำตอบสุดท้าย พร้อมรักษาสถานะเซสชันให้สอดคล้องกัน

ใน OpenClaw วงจรคือการรันหนึ่งครั้งที่จัดลำดับแบบอนุกรมต่อเซสชัน ซึ่งส่งเหตุการณ์วงจรชีวิตและสตรีม
ขณะที่โมเดลคิด เรียกเครื่องมือ และสตรีมผลลัพธ์ เอกสารนี้อธิบายว่าวงจรจริงดังกล่าว
ถูกเชื่อมต่อแบบครบตั้งแต่ต้นจนจบอย่างไร

## จุดเข้าใช้งาน

- Gateway RPC: `agent` และ `agent.wait`
- CLI: คำสั่ง `agent`

## วิธีทำงาน (ภาพรวมระดับสูง)

1. `agent` RPC ตรวจสอบความถูกต้องของพารามิเตอร์ ระบุเซสชัน (sessionKey/sessionId) บันทึกข้อมูลเมตาของเซสชัน และส่งคืน `{ runId, acceptedAt }` ทันที
2. `agentCommand` รันเอเจนต์:
   - ระบุค่าเริ่มต้นของโมเดล + thinking/verbose/trace
   - โหลดสแนปช็อต Skills
   - เรียก `runEmbeddedPiAgent` (รันไทม์ pi-agent-core)
   - ส่ง **สิ้นสุด/ข้อผิดพลาดของวงจรชีวิต** หากวงจรแบบฝังไม่ได้ส่งเหตุการณ์ดังกล่าว
3. `runEmbeddedPiAgent`:
   - จัดลำดับการรันผ่านคิวต่อเซสชัน + คิวส่วนกลาง
   - ระบุโมเดล + โปรไฟล์การยืนยันตัวตน และสร้างเซสชัน Pi
   - สมัครรับเหตุการณ์ Pi และสตรีมเดลตาของผู้ช่วย/เครื่องมือ
   - บังคับใช้เวลาเกินกำหนด -> ยกเลิกการรันหากเกินกำหนด
   - ส่งคืนเพย์โหลด + ข้อมูลเมตาการใช้งาน
4. `subscribeEmbeddedPiSession` เชื่อมเหตุการณ์ pi-agent-core ไปยังสตรีม `agent` ของ OpenClaw:
   - เหตุการณ์เครื่องมือ => `stream: "tool"`
   - เดลตาของผู้ช่วย => `stream: "assistant"`
   - เหตุการณ์วงจรชีวิต => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` ใช้ `waitForAgentRun`:
   - รอ **สิ้นสุด/ข้อผิดพลาดของวงจรชีวิต** สำหรับ `runId`
   - ส่งคืน `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## การเข้าคิว + ภาวะพร้อมกัน

- การรันจะถูกจัดลำดับแบบอนุกรมต่อคีย์เซสชัน (เลนเซสชัน) และอาจผ่านเลนส่วนกลางด้วย
- วิธีนี้ป้องกัน race ของเครื่องมือ/เซสชัน และรักษาประวัติเซสชันให้สอดคล้องกัน
- ช่องทางข้อความสามารถเลือกโหมดคิว (collect/steer/followup) ที่ป้อนเข้าสู่ระบบเลนนี้ได้
  ดู [คิวคำสั่ง](/th/concepts/queue)
- การเขียนทรานสคริปต์ยังได้รับการป้องกันด้วยล็อกการเขียนเซสชันบนไฟล์เซสชันด้วย ล็อกนี้
  รับรู้กระบวนการและอิงไฟล์ จึงจับตัวเขียนที่ข้ามคิวในกระบวนการหรือมาจาก
  กระบวนการอื่นได้
- ล็อกการเขียนเซสชันไม่อนุญาตให้เข้าได้ซ้ำโดยค่าเริ่มต้น หากตัวช่วยตั้งใจซ้อนการถือครอง
  ล็อกเดียวกันขณะยังคงรักษาตัวเขียนเชิงตรรกะหนึ่งตัวไว้ จะต้องเลือกใช้โดยชัดเจนด้วย
  `allowReentrant: true`

## การเตรียมเซสชัน + เวิร์กสเปซ

- เวิร์กสเปซถูกระบุและสร้างขึ้น การรันแบบ sandbox อาจเปลี่ยนเส้นทางไปยังรากเวิร์กสเปซ sandbox
- Skills ถูกโหลด (หรือนำกลับมาใช้จากสแนปช็อต) และฉีดเข้า env และพรอมป์
- ไฟล์ bootstrap/context ถูกระบุและฉีดเข้าไปในรายงานพรอมป์ระบบ
- มีการถือครองล็อกการเขียนเซสชัน `SessionManager` ถูกเปิดและเตรียมไว้ก่อนการสตรีม เส้นทาง
  การเขียนทรานสคริปต์ใหม่ Compaction หรือการตัดทอนในภายหลัง ต้องถือครองล็อกเดียวกันก่อนเปิดหรือ
  แก้ไขไฟล์ทรานสคริปต์

## การประกอบพรอมป์ + พรอมป์ระบบ

- พรอมป์ระบบสร้างจากพรอมป์พื้นฐานของ OpenClaw, พรอมป์ Skills, บริบท bootstrap และการแทนที่ต่อการรัน
- บังคับใช้ขีดจำกัดเฉพาะโมเดลและโทเค็นสำรองสำหรับ Compaction
- ดู [พรอมป์ระบบ](/th/concepts/system-prompt) สำหรับสิ่งที่โมเดลเห็น

## จุด hook (ตำแหน่งที่คุณสามารถแทรกการทำงานได้)

OpenClaw มีระบบ hook สองระบบ:

- **hook ภายใน** (hook ของ Gateway): สคริปต์ขับเคลื่อนด้วยเหตุการณ์สำหรับคำสั่งและเหตุการณ์วงจรชีวิต
- **hook ของ Plugin**: จุดขยายภายในวงจรชีวิตเอเจนต์/เครื่องมือและไปป์ไลน์ Gateway

### hook ภายใน (hook ของ Gateway)

- **`agent:bootstrap`**: รันขณะสร้างไฟล์ bootstrap ก่อนสรุปพรอมป์ระบบ
  ใช้สิ่งนี้เพื่อเพิ่ม/ลบไฟล์บริบท bootstrap
- **hook คำสั่ง**: `/new`, `/reset`, `/stop` และเหตุการณ์คำสั่งอื่นๆ (ดูเอกสาร Hooks)

ดู [Hooks](/th/automation/hooks) สำหรับการตั้งค่าและตัวอย่าง

### hook ของ Plugin (วงจรชีวิตเอเจนต์ + Gateway)

รายการเหล่านี้รันภายในวงจรเอเจนต์หรือไปป์ไลน์ Gateway:

- **`before_model_resolve`**: รันก่อนเซสชัน (ไม่มี `messages`) เพื่อแทนที่ provider/model แบบกำหนดแน่นอนก่อนการระบุโมเดล
- **`before_prompt_build`**: รันหลังโหลดเซสชัน (พร้อม `messages`) เพื่อฉีด `prependContext`, `systemPrompt`, `prependSystemContext` หรือ `appendSystemContext` ก่อนส่งพรอมป์ ใช้ `prependContext` สำหรับข้อความไดนามิกต่อเทิร์น และใช้ฟิลด์บริบทระบบสำหรับคำแนะนำที่เสถียรซึ่งควรอยู่ในพื้นที่พรอมป์ระบบ
- **`before_agent_start`**: hook ความเข้ากันได้แบบเดิมที่อาจรันในเฟสใดเฟสหนึ่ง ให้ใช้ hook แบบชัดเจนด้านบนเป็นหลัก
- **`before_agent_reply`**: รันหลังการกระทำแบบ inline และก่อนเรียก LLM ทำให้ Plugin สามารถรับช่วงเทิร์นและส่งคืนคำตอบสังเคราะห์ หรือทำให้เทิร์นนั้นเงียบทั้งหมด
- **`agent_end`**: ตรวจสอบรายการข้อความสุดท้ายและข้อมูลเมตาการรันหลังเสร็จสิ้น
- **`before_compaction` / `after_compaction`**: สังเกตหรือใส่คำอธิบายประกอบให้รอบ Compaction
- **`before_tool_call` / `after_tool_call`**: แทรกการทำงานกับพารามิเตอร์/ผลลัพธ์ของเครื่องมือ
- **`before_install`**: ตรวจสอบผลการสแกนในตัว และเลือกบล็อกการติดตั้ง Skill หรือ Plugin ได้
- **`tool_result_persist`**: แปลงผลลัพธ์เครื่องมือแบบซิงโครนัสก่อนเขียนลงในทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ
- **`message_received` / `message_sending` / `message_sent`**: hook ข้อความขาเข้า + ขาออก
- **`session_start` / `session_end`**: ขอบเขตวงจรชีวิตเซสชัน
- **`gateway_start` / `gateway_stop`**: เหตุการณ์วงจรชีวิต Gateway

กฎการตัดสินใจของ hook สำหรับตัวป้องกันขาออก/เครื่องมือ:

- `before_tool_call`: `{ block: true }` เป็นผลลัพธ์สุดท้ายและหยุด handler ที่มีลำดับความสำคัญต่ำกว่า
- `before_tool_call`: `{ block: false }` ไม่ทำอะไรและไม่ล้างการบล็อกก่อนหน้า
- `before_install`: `{ block: true }` เป็นผลลัพธ์สุดท้ายและหยุด handler ที่มีลำดับความสำคัญต่ำกว่า
- `before_install`: `{ block: false }` ไม่ทำอะไรและไม่ล้างการบล็อกก่อนหน้า
- `message_sending`: `{ cancel: true }` เป็นผลลัพธ์สุดท้ายและหยุด handler ที่มีลำดับความสำคัญต่ำกว่า
- `message_sending`: `{ cancel: false }` ไม่ทำอะไรและไม่ล้างการยกเลิกก่อนหน้า

ดู [hook ของ Plugin](/th/plugins/hooks) สำหรับ API ของ hook และรายละเอียดการลงทะเบียน

harness อาจปรับใช้ hook เหล่านี้ต่างกัน harness ของแอปเซิร์ฟเวอร์ Codex จะคง
hook ของ Plugin ของ OpenClaw ไว้เป็นสัญญาความเข้ากันได้สำหรับพื้นผิวมิเรอร์ที่มีเอกสาร
ในขณะที่ hook ดั้งเดิมของ Codex ยังเป็นกลไกระดับล่างของ Codex ที่แยกต่างหาก

## การสตรีม + คำตอบบางส่วน

- เดลตาของผู้ช่วยถูกสตรีมจาก pi-agent-core และส่งออกเป็นเหตุการณ์ `assistant`
- การสตรีมบล็อกสามารถส่งคำตอบบางส่วนได้ทั้งบน `text_end` หรือ `message_end`
- การสตรีม reasoning สามารถส่งออกเป็นสตรีมแยก หรือเป็นคำตอบแบบบล็อกได้
- ดู [การสตรีม](/th/concepts/streaming) สำหรับพฤติกรรมการแบ่งชิ้นและคำตอบแบบบล็อก

## การเรียกใช้เครื่องมือ + เครื่องมือส่งข้อความ

- เหตุการณ์เริ่ม/อัปเดต/จบของเครื่องมือถูกส่งออกบนสตรีม `tool`
- ผลลัพธ์เครื่องมือถูกทำให้ปลอดภัยด้านขนาดและเพย์โหลดรูปภาพก่อนบันทึก/ส่งออก
- การส่งของเครื่องมือส่งข้อความถูกติดตามเพื่อระงับการยืนยันซ้ำจากผู้ช่วย

## การจัดรูปคำตอบ + การระงับ

- เพย์โหลดสุดท้ายประกอบจาก:
  - ข้อความผู้ช่วย (และ reasoning ที่เลือกได้)
  - สรุปเครื่องมือแบบ inline (เมื่อ verbose + อนุญาต)
  - ข้อความข้อผิดพลาดของผู้ช่วยเมื่อโมเดลเกิดข้อผิดพลาด
- โทเค็นเงียบที่ตรงเป๊ะ `NO_REPLY` / `no_reply` ถูกกรองออกจากเพย์โหลด
  ขาออก
- รายการซ้ำจากเครื่องมือส่งข้อความถูกลบออกจากรายการเพย์โหลดสุดท้าย
- หากไม่มีเพย์โหลดที่แสดงผลได้เหลืออยู่และเครื่องมือเกิดข้อผิดพลาด จะส่งคำตอบข้อผิดพลาดของเครื่องมือสำรอง
  (เว้นแต่เครื่องมือส่งข้อความได้ส่งคำตอบที่ผู้ใช้มองเห็นแล้ว)

## Compaction + การลองใหม่

- Auto-compaction ส่งเหตุการณ์สตรีม `compaction` และสามารถทริกเกอร์การลองใหม่ได้
- เมื่อมีการลองใหม่ บัฟเฟอร์ในหน่วยความจำและสรุปเครื่องมือจะถูกรีเซ็ตเพื่อหลีกเลี่ยงผลลัพธ์ซ้ำ
- ดู [Compaction](/th/concepts/compaction) สำหรับไปป์ไลน์ Compaction

## สตรีมเหตุการณ์ (ปัจจุบัน)

- `lifecycle`: ส่งโดย `subscribeEmbeddedPiSession` (และส่งเป็น fallback โดย `agentCommand`)
- `assistant`: เดลตาที่สตรีมจาก pi-agent-core
- `tool`: เหตุการณ์เครื่องมือที่สตรีมจาก pi-agent-core

## การจัดการช่องทางแชต

- เดลตาของผู้ช่วยถูกบัฟเฟอร์เป็นข้อความ `delta` ของแชต
- `final` ของแชตถูกส่งเมื่อ **สิ้นสุด/ข้อผิดพลาดของวงจรชีวิต**

## เวลาเกินกำหนด

- ค่าเริ่มต้นของ `agent.wait`: 30 วินาที (เฉพาะการรอ) พารามิเตอร์ `timeoutMs` จะแทนที่ค่านี้
- รันไทม์เอเจนต์: ค่าเริ่มต้นของ `agents.defaults.timeoutSeconds` คือ 172800 วินาที (48 ชั่วโมง); บังคับใช้ในตัวจับเวลายกเลิกของ `runEmbeddedPiAgent`
- รันไทม์ Cron: `timeoutSeconds` ของเทิร์นเอเจนต์แบบแยกเป็นของ cron ตัวจัดตารางจะเริ่มตัวจับเวลานั้นเมื่อการดำเนินการเริ่มขึ้น ยกเลิกการรันที่อยู่เบื้องล่างเมื่อถึงกำหนดเวลาที่กำหนดค่าไว้ จากนั้นรันการล้างข้อมูลแบบมีขอบเขตก่อนบันทึกเวลาเกินกำหนด เพื่อไม่ให้เซสชันลูกที่ค้างอยู่ทำให้เลนติดค้าง
- การกู้คืนเซสชันที่ติดค้าง: เมื่อเปิดใช้ diagnostics แล้ว `diagnostics.stuckSessionWarnMs` จะตรวจจับเซสชัน `processing` ที่ใช้เวลานาน การรันแบบฝังที่ยังทำงานอยู่ การดำเนินการตอบกลับที่ยังทำงานอยู่ และงานเลนเซสชันที่ยังทำงานอยู่จะยังเป็นเพียงคำเตือนโดยค่าเริ่มต้น หาก diagnostics แสดงว่าไม่มีงานที่ยังทำงานอยู่สำหรับเซสชัน watchdog จะปล่อยเลนเซสชันที่ได้รับผลกระทบเพื่อให้งานเริ่มต้นที่เข้าคิวไว้ระบายออกได้
- เวลาเกินกำหนดเมื่อโมเดลไม่ส่งข้อมูล: OpenClaw ยกเลิกคำขอโมเดลเมื่อไม่มีชิ้นส่วนคำตอบมาถึงก่อนหมดช่วง idle `models.providers.<id>.timeoutSeconds` ขยายตัวเฝ้าระวัง idle นี้สำหรับ provider แบบ local/โฮสต์เองที่ช้า มิฉะนั้น OpenClaw จะใช้ `agents.defaults.timeoutSeconds` เมื่อกำหนดค่าไว้ โดยถูกจำกัดไว้ที่ 120 วินาทีตามค่าเริ่มต้น การรันที่ทริกเกอร์โดย Cron ซึ่งไม่มีเวลาเกินกำหนดของโมเดลหรือเอเจนต์ที่ระบุไว้ชัดเจนจะปิดใช้ตัวเฝ้าระวัง idle และพึ่งพาเวลาเกินกำหนดชั้นนอกของ cron
- เวลาเกินกำหนดของคำขอ HTTP ของ provider: `models.providers.<id>.timeoutSeconds` มีผลกับการ fetch HTTP ของโมเดลสำหรับ provider นั้น รวมถึงการเชื่อมต่อ, headers, body, เวลาเกินกำหนดของคำขอ SDK, การจัดการยกเลิก guarded-fetch ทั้งหมด และตัวเฝ้าระวัง idle ของสตรีมโมเดล ใช้ค่านี้สำหรับ provider แบบ local/โฮสต์เองที่ช้า เช่น Ollama ก่อนเพิ่มเวลาเกินกำหนดของรันไทม์เอเจนต์ทั้งหมด

## ตำแหน่งที่สิ่งต่างๆ อาจจบก่อนกำหนด

- เวลาเกินกำหนดของเอเจนต์ (ยกเลิก)
- AbortSignal (ยกเลิก)
- Gateway ตัดการเชื่อมต่อ หรือเวลาเกินกำหนดของ RPC
- เวลาเกินกำหนดของ `agent.wait` (เฉพาะการรอ ไม่หยุดเอเจนต์)
