สถานะ: UI แชต SwiftUI บน macOS/iOS สื่อสารกับ Gateway WebSocket โดยตรง ไม่มีเบราว์เซอร์แบบฝังตัวและไม่มีเซิร์ฟเวอร์ไฟล์สแตติกภายในเครื่อง

## คืออะไร

- UI แชตแบบเนทีฟสำหรับ Gateway
- ใช้เซสชันและกฎการกำหนดเส้นทางเดียวกับช่องทางอื่น
- การกำหนดเส้นทางแบบกำหนดผลลัพธ์แน่นอน: การตอบกลับจะส่งกลับไปยัง WebChat เสมอ
- ประวัติจะดึงจาก Gateway เสมอ (ไม่มีการเฝ้าดูไฟล์ภายในเครื่อง) หากไม่สามารถเข้าถึง Gateway ได้ WebChat จะเป็นแบบอ่านอย่างเดียว

## เริ่มต้นอย่างรวดเร็ว

1. เริ่ม Gateway
2. เปิด UI ของ WebChat (แอป macOS/iOS) หรือแท็บแชตของ Control UI
3. ตรวจสอบว่าได้กำหนดค่าเส้นทางการยืนยันตัวตนของ Gateway ที่ถูกต้องแล้ว (ค่าเริ่มต้นใช้ข้อมูลลับร่วมกัน แม้บนลูปแบ็ก)

## วิธีการทำงาน

- UI เชื่อมต่อกับ Gateway WebSocket และใช้เมธอด RPC ได้แก่ `chat.history`, `chat.send`, `chat.inject` และ `chat.message.get`
- `chat.history` มีขอบเขตจำกัดเพื่อความเสถียร: Gateway อาจตัดทอนฟิลด์ข้อความยาว ละเว้นเมทาดาทาที่มีขนาดใหญ่ และแทนที่รายการที่ใหญ่เกินไปด้วย `[chat.history omitted: message too large]` ไคลเอนต์ API สามารถส่ง `maxChars` ต่อคำขอเพื่อแทนที่ขีดจำกัดเริ่มต้นสำหรับการเรียกหนึ่งครั้ง
- เมื่อข้อความผู้ช่วยที่มองเห็นได้ถูกตัดทอนใน `chat.history` Control UI สามารถเปิดตัวอ่านด้านข้างและดึงรายการฉบับเต็มที่ปรับให้เป็นมาตรฐานสำหรับการแสดงผลตามคำขอผ่าน `chat.message.get` โดยไม่เพิ่มเพย์โหลดประวัติเริ่มต้น `chat.message.get` ใช้สาขาทรานสคริปต์และกฎการแสดงผลเดียวกับ `chat.history` แต่ระบุเป้าหมายเป็นรายการเดียวด้วย `messageId` และส่งคืนเหตุผลที่ตรงตามจริงว่าไม่พร้อมใช้งานเมื่อไม่สามารถส่งคืนเนื้อหาฉบับเต็มได้อีกต่อไป
- `chat.history` ติดตามสาขาทรานสคริปต์ที่ใช้งานอยู่สำหรับไฟล์เซสชันแบบเพิ่มต่อท้ายเท่านั้น ดังนั้นสาขาการเขียนใหม่ที่ถูกละทิ้งและสำเนาพรอมต์ที่ถูกแทนที่จะไม่แสดงผลใน WebChat
- รายการ Compaction แสดงผลเป็นตัวคั่น "ประวัติที่ถูกย่อรวม" พร้อมคำอธิบายว่าทรานสคริปต์ที่ถูกย่อรวมได้รับการเก็บรักษาไว้เป็นจุดตรวจ และมีการดำเนินการเพื่อเปิดจุดตรวจของเซสชัน (สร้างสาขาหรือคืนค่า เมื่อสิทธิ์อนุญาต)
- Control UI จดจำ Gateway `sessionId` เบื้องหลังที่ `chat.history` ส่งคืน และใส่ค่านี้ในการเรียก `chat.send` ครั้งต่อไป เพื่อให้การเชื่อมต่อใหม่และการรีเฟรชหน้ายังคงสนทนาที่จัดเก็บไว้รายการเดิม เว้นแต่ผู้ใช้จะเริ่มหรือรีเซ็ตเซสชัน
- `chat.send` รับคีย์ไอดempotency (Control UI ใช้ ID การรัน) Gateway จะขจัดคำขอซ้ำที่ใช้คีย์เดียวกัน ดังนั้นการส่งซ้ำหรือการส่งที่กำลังดำเนินการอยู่ซ้ำกันสำหรับเซสชัน/ข้อความ/ไฟล์แนบเดียวกันจะไม่สร้างการรันครั้งที่สอง
- ไฟล์เริ่มต้นของเวิร์กสเปซและคำสั่ง `BOOTSTRAP.md` ที่รอดำเนินการจะส่งผ่านส่วน `# Project Context` ของพรอมต์ระบบเอเจนต์ ไม่ได้คัดลอกลงในข้อความผู้ใช้ของ WebChat หากเนื้อหาบูตสแตรปถูกตัดทอน พรอมต์ระบบจะได้รับ "ประกาศบริบทบูตสแตรป" แบบสั้นแทน ส่วนจำนวนโดยละเอียดและตัวเลือกการกำหนดค่าจะยังคงอยู่บนพื้นผิวการวินิจฉัย
- การปรับให้เป็นมาตรฐานสำหรับการแสดงผลบน `chat.history` จะตัดสิ่งต่อไปนี้ออก: บริบท OpenClaw สำหรับรันไทม์เท่านั้น ตัวห่อหุ้มเอนเวโลปขาเข้า แท็กคำสั่งการนำส่งแบบอินไลน์ เช่น `[[reply_to_current]]`, `[[reply_to:<id>]]` และ `[[audio_as_voice]]`, เพย์โหลด XML ของการเรียกเครื่องมือแบบข้อความธรรมดา (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>` รวมถึงบล็อกที่ถูกตัดทอน) และโทเค็นควบคุมโมเดล ASCII/แบบเต็มความกว้างที่รั่วไหล รายการผู้ช่วยที่ข้อความทั้งหมดซึ่งมองเห็นได้มีเพียงโทเค็นเงียบ `NO_REPLY` (ไม่คำนึงถึงตัวพิมพ์เล็กและใหญ่) จะถูกละเว้น
- เพย์โหลดการตอบกลับที่มีแฟล็กการให้เหตุผล (`isReasoning: true`) จะถูกตัดออกจากเนื้อหาผู้ช่วยใน WebChat ข้อความเล่นซ้ำของทรานสคริปต์ และบล็อกเนื้อหาเสียง ดังนั้นเพย์โหลดที่มีเฉพาะกระบวนการคิดจะไม่ปรากฏเป็นข้อความผู้ช่วยที่มองเห็นได้หรือเสียงที่เล่นได้
- `chat.inject` เพิ่มบันทึกจากผู้ช่วยต่อท้ายทรานสคริปต์โดยตรงและกระจายไปยัง UI (ไม่มีการรันเอเจนต์)
- การรันที่ถูกยกเลิกสามารถคงเอาต์พุตบางส่วนของผู้ช่วยให้มองเห็นใน UI ได้ Gateway จะบันทึกข้อความบางส่วนนั้นลงในประวัติทรานสคริปต์เมื่อมีเอาต์พุตในบัฟเฟอร์ และทำเครื่องหมายรายการด้วยเมทาดาทาการยกเลิก

### โมเดลทรานสคริปต์และการนำส่ง

WebChat มีเส้นทางข้อมูลแยกกันสองเส้นทาง:

- แถวทรานสคริปต์ SQLite คือทรานสคริปต์ถาวรของโมเดล/รันไทม์ สำหรับการรันเอเจนต์ตามปกติ รันไทม์ OpenClaw แบบฝังตัวจะบันทึกข้อความ `user`, `assistant` และ `toolResult` ที่โมเดลมองเห็นผ่านตัวเข้าถึงเซสชัน WebChat จะไม่เขียนข้อความการนำส่ง สถานะ หรือข้อความช่วยเหลือใด ๆ ลงในทรานสคริปต์นั้นโดยพลการ
- เหตุการณ์ `ReplyPayload` ของ Gateway คือภาพฉายการนำส่งแบบสด: ปรับให้เป็นมาตรฐานสำหรับการแสดงผลใน WebChat/ช่องทาง การสตรีมแบบบล็อก แท็กคำสั่ง การฝังสื่อ แฟล็ก TTS/เสียง และลักษณะการทำงานสำรองของ UI เหตุการณ์เหล่านี้ไม่ใช่บันทึกเซสชันหลัก
- ฮาร์เนสที่ต้องการการตอบกลับที่มองเห็นได้ผ่าน `tools.message` ยังคงใช้ WebChat เป็นปลายทางรับการตอบกลับภายในของแหล่งที่มาสำหรับการรันปัจจุบัน `message.send` ที่ไม่มีเป้าหมายจากการรัน WebChat ที่ใช้งานอยู่นั้นจะถูกฉายลงในแชตเดียวกันและทำสำเนาไปยังทรานสคริปต์ของเซสชัน WebChat จะไม่กลายเป็นช่องทางขาออกที่นำกลับมาใช้ใหม่ได้ และจะไม่สืบทอด `lastChannel`
- WebChat จะแทรกรายการผู้ช่วยลงในทรานสคริปต์เฉพาะเมื่อ Gateway เป็นเจ้าของข้อความที่แสดงอยู่นอกเทิร์นเอเจนต์แบบฝังตัวตามปกติ ได้แก่ `chat.inject` การตอบกลับคำสั่งที่ไม่ใช่เอเจนต์ เอาต์พุตบางส่วนที่ถูกยกเลิก และส่วนเสริมทรานสคริปต์สื่อที่ WebChat จัดการ
- หากข้อความผู้ช่วยแบบสดปรากฏระหว่างการรันแต่หายไปหลังโหลดประวัติใหม่ ให้ตรวจสอบตามลำดับดังนี้: ทรานสคริปต์ SQLite มีข้อความผู้ช่วยหรือไม่ ภาพฉายการแสดงผล `chat.history` ตัดข้อความนั้นออกหรือไม่ จากนั้นตรวจสอบว่าการผสานส่วนท้ายแบบมองโลกในแง่ดีของ Control UI แทนที่สถานะการนำส่งภายในเครื่องด้วยสแนปช็อตที่บันทึกไว้หรือไม่

คำตอบสุดท้ายจากการรันเอเจนต์ตามปกติควรคงอยู่ถาวร เนื่องจากรันไทม์แบบฝังตัวเขียน `message_end` ของผู้ช่วย ระบบสำรองใด ๆ ที่ทำสำเนาเพย์โหลดสุดท้ายที่นำส่งแล้วลงในทรานสคริปต์ต้องหลีกเลี่ยงการสร้างเทิร์นผู้ช่วยซ้ำกับที่รันไทม์แบบฝังตัวเขียนไว้แล้วก่อน

## แผงเครื่องมือเอเจนต์ของ Control UI

- แผงเครื่องมือ `/agents` ของ Control UI มีมุมมอง "พร้อมใช้งานขณะนี้" ซึ่งรองรับโดย `tools.effective(sessionKey=...)`: ภาพฉายแบบอ่านอย่างเดียวที่เซิร์ฟเวอร์สร้างจากคลังเครื่องมือของเซสชันปัจจุบัน รวมถึงเครื่องมือหลัก เครื่องมือจาก Plugin เครื่องมือที่ช่องทางเป็นเจ้าของ และเครื่องมือเซิร์ฟเวอร์ MCP ที่ค้นพบแล้ว
- มุมมองแก้ไขการกำหนดค่าแยกต่างหาก (รองรับโดย `tools.catalog`) ครอบคลุมโปรไฟล์ การแทนที่ต่อเอเจนต์ และความหมายของแค็ตตาล็อก
- ความพร้อมใช้งานของรันไทม์มีขอบเขตตามเซสชัน การสลับเซสชันบนเอเจนต์เดียวกันอาจเปลี่ยนรายการ "พร้อมใช้งานขณะนี้" หากเซิร์ฟเวอร์ MCP ที่กำหนดค่าไว้ยังไม่ได้เชื่อมต่อหรือมีการเปลี่ยนแปลงนับตั้งแต่การค้นพบครั้งล่าสุด แผงจะแสดงประกาศแทนการเริ่มทรานสปอร์ต MCP จากเส้นทางการอ่านโดยไม่มีการแจ้ง
- ตัวแก้ไขการกำหนดค่าไม่ได้หมายความว่าจะพร้อมใช้งานในรันไทม์ การเข้าถึงที่มีผลยังคงเป็นไปตามลำดับความสำคัญของนโยบาย (`allow`/`deny` การแทนที่ต่อเอเจนต์และผู้ให้บริการ/ช่องทาง)

## การใช้งานระยะไกล

- โหมดระยะไกลทำทันเนล Gateway WebSocket ผ่าน SSH/Tailscale
- ไม่จำเป็นต้องเรียกใช้เซิร์ฟเวอร์ WebChat แยกต่างหาก

## ข้อมูลอ้างอิงการกำหนดค่า (WebChat)

การกำหนดค่าฉบับเต็ม: [การกำหนดค่า](/th/gateway/configuration)

WebChat ไม่มีส่วนการกำหนดค่าที่บันทึกถาวร Gateway ใช้ขีดจำกัดการแสดงผล `chat.history` ในตัว ไคลเอนต์ API สามารถส่ง `maxChars` ต่อคำขอเพื่อแทนที่ขีดจำกัดสำหรับการเรียกหนึ่งครั้ง การกำหนดค่า `channels.webchat` และ `gateway.webchat` แบบเดิมเลิกใช้แล้ว ให้เรียกใช้ `openclaw doctor --fix` เพื่อลบออก

ตัวเลือกส่วนกลางที่เกี่ยวข้อง:

- `gateway.port`, `gateway.bind`: โฮสต์/พอร์ต WebSocket
- `gateway.auth.mode`, `gateway.auth.token`, `gateway.auth.password`:
  การยืนยันตัวตน WebSocket ด้วยข้อมูลลับร่วมกัน
- `gateway.auth.allowTailscale`: แท็บแชต Control UI ในเบราว์เซอร์สามารถใช้ส่วนหัวข้อมูลประจำตัว
  Tailscale Serve เมื่อเปิดใช้งาน
- `gateway.auth.mode: "trusted-proxy"`: การยืนยันตัวตนผ่านพร็อกซีย้อนกลับสำหรับไคลเอนต์เบราว์เซอร์ที่อยู่หลังแหล่งพร็อกซี **ที่ไม่ใช่ลูปแบ็ก** ซึ่งรับรู้ข้อมูลประจำตัว (ดู [การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth))
- `gateway.remote.url`, `gateway.remote.token`, `gateway.remote.password`: เป้าหมาย Gateway ระยะไกล
- `session.*`: ที่เก็บข้อมูลเซสชันและค่าเริ่มต้นของคีย์หลัก
