Control UI เป็นแอปหน้าเดียวขนาดเล็กแบบ **Vite + Lit** ที่ให้บริการโดย Gateway:

- ค่าเริ่มต้น: `http://<host>:18789/`
- คำนำหน้าเสริม: ตั้งค่า `gateway.controlUi.basePath` (เช่น `/openclaw`)

แอปนี้สื่อสาร **โดยตรงกับ Gateway WebSocket** บนพอร์ตเดียวกัน

## เปิดอย่างรวดเร็ว (ในเครื่อง)

หาก Gateway ทำงานอยู่บนคอมพิวเตอร์เครื่องเดียวกัน ให้เปิด:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (หรือ [http://localhost:18789/](http://localhost:18789/))

หากหน้าโหลดไม่สำเร็จ ให้เริ่ม Gateway ก่อน: `openclaw gateway`

<Note>
บนการ bind LAN ของ Windows แบบเนทีฟ Windows Firewall หรือ Group Policy ที่จัดการโดยองค์กรยังอาจบล็อก LAN URL ที่ประกาศไว้ได้ แม้ว่า `127.0.0.1` จะใช้งานได้บนโฮสต์ Gateway ก็ตาม ให้รัน `openclaw gateway status --deep` บนโฮสต์ Windows; คำสั่งนี้จะรายงานพอร์ตที่น่าจะถูกบล็อก โปรไฟล์ที่ไม่ตรงกัน และกฎไฟร์วอลล์ในเครื่องที่นโยบายอาจละเว้น
</Note>

Auth จะถูกส่งระหว่างการจับมือ WebSocket ผ่าน:

- `connect.params.auth.token`
- `connect.params.auth.password`
- ส่วนหัวตัวตนของ Tailscale Serve เมื่อ `gateway.auth.allowTailscale: true`
- ส่วนหัวตัวตนของ trusted-proxy เมื่อ `gateway.auth.mode: "trusted-proxy"`

แผงการตั้งค่าแดชบอร์ดจะเก็บโทเค็นไว้สำหรับเซสชันแท็บเบราว์เซอร์ปัจจุบันและ URL ของ gateway ที่เลือก; รหัสผ่านจะไม่ถูกคงไว้ Onboarding มักสร้างโทเค็น gateway สำหรับ auth แบบ shared-secret เมื่อเชื่อมต่อครั้งแรก แต่ password auth ก็ใช้งานได้เช่นกันเมื่อ `gateway.auth.mode` เป็น `"password"`

## การจับคู่อุปกรณ์ (การเชื่อมต่อครั้งแรก)

เมื่อคุณเชื่อมต่อกับ Control UI จากเบราว์เซอร์หรืออุปกรณ์ใหม่ Gateway มักต้องการ **การอนุมัติการจับคู่แบบครั้งเดียว** นี่เป็นมาตรการรักษาความปลอดภัยเพื่อป้องกันการเข้าถึงที่ไม่ได้รับอนุญาต

**สิ่งที่คุณจะเห็น:** "disconnected (1008): pairing required"

<Steps>
  <Step title="แสดงรายการคำขอที่รอดำเนินการ">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="อนุมัติด้วย ID คำขอ">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

หากเบราว์เซอร์ลองจับคู่อีกครั้งโดยมีรายละเอียด auth ที่เปลี่ยนไป (role/scopes/public key) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่ และจะสร้าง `requestId` ใหม่ ให้รัน `openclaw devices list` อีกครั้งก่อนอนุมัติ

หากเบราว์เซอร์จับคู่แล้ว และคุณเปลี่ยนจากสิทธิ์อ่านเป็นสิทธิ์เขียน/admin การดำเนินการนี้จะถือเป็นการอัปเกรดการอนุมัติ ไม่ใช่การเชื่อมต่อใหม่แบบเงียบ OpenClaw จะคงการอนุมัติเก่าไว้ บล็อกการเชื่อมต่อใหม่ที่มีขอบเขตกว้างขึ้น และขอให้คุณอนุมัติชุด scope ใหม่อย่างชัดเจน

เมื่ออนุมัติแล้ว อุปกรณ์จะถูกจดจำและจะไม่ต้องอนุมัติซ้ำ เว้นแต่คุณจะเพิกถอนด้วย `openclaw devices revoke --device <id> --role <role>` ดู [Devices CLI](/th/cli/devices) สำหรับการหมุนเวียนและการเพิกถอนโทเค็น

เอเจนต์ Paperclip ที่เชื่อมต่อผ่านอะแดปเตอร์ `openclaw_gateway` ใช้โฟลว์การอนุมัติครั้งแรกเดียวกัน หลังจากพยายามเชื่อมต่อครั้งแรก ให้รัน `openclaw devices approve --latest` เพื่อดูตัวอย่างคำขอที่รอดำเนินการ จากนั้นรันคำสั่ง `openclaw devices approve <requestId>` ที่พิมพ์ออกมาอีกครั้งเพื่ออนุมัติ ส่งค่า `--url` และ `--token` อย่างชัดเจนสำหรับ gateway ระยะไกล เพื่อให้การอนุมัติมีความเสถียรข้ามการรีสตาร์ต ให้กำหนดค่า `adapterConfig.devicePrivateKeyPem` แบบถาวรใน Paperclip แทนการปล่อยให้สร้างตัวตนอุปกรณ์ชั่วคราวใหม่ในแต่ละรัน

<Note>
- การเชื่อมต่อเบราว์เซอร์ผ่าน local loopback โดยตรง (`127.0.0.1` / `localhost`) จะได้รับการอนุมัติโดยอัตโนมัติ
- Tailscale Serve สามารถข้ามรอบการจับคู่สำหรับเซสชันผู้ปฏิบัติการ Control UI ได้เมื่อ `gateway.auth.allowTailscale: true`, ตัวตน Tailscale ตรวจสอบผ่าน และเบราว์เซอร์แสดงตัวตนอุปกรณ์ของตน
- การ bind Tailnet โดยตรง การเชื่อมต่อเบราว์เซอร์ผ่าน LAN และโปรไฟล์เบราว์เซอร์ที่ไม่มีตัวตนอุปกรณ์ยังต้องได้รับการอนุมัติอย่างชัดเจน
- แต่ละโปรไฟล์เบราว์เซอร์จะสร้าง ID อุปกรณ์ที่ไม่ซ้ำกัน ดังนั้นการสลับเบราว์เซอร์หรือการล้างข้อมูลเบราว์เซอร์จะต้องจับคู่ใหม่

</Note>

## ตัวตนส่วนบุคคล (เฉพาะเบราว์เซอร์)

Control UI รองรับตัวตนส่วนบุคคลต่อเบราว์เซอร์ (ชื่อที่แสดงและอวตาร) ซึ่งแนบกับข้อความขาออกเพื่อระบุที่มาในเซสชันที่ใช้ร่วมกัน ข้อมูลนี้อยู่ในที่เก็บข้อมูลของเบราว์เซอร์ ถูกจำกัดขอบเขตตามโปรไฟล์เบราว์เซอร์ปัจจุบัน และไม่ซิงก์ไปยังอุปกรณ์อื่นหรือคงไว้ฝั่งเซิร์ฟเวอร์ นอกเหนือจากเมทาดาทาผู้เขียนทรานสคริปต์ตามปกติบนข้อความที่คุณส่งจริง การล้างข้อมูลไซต์หรือการสลับเบราว์เซอร์จะรีเซ็ตข้อมูลนี้ให้ว่าง

รูปแบบเฉพาะเบราว์เซอร์เดียวกันใช้กับการแทนที่อวตารของผู้ช่วยด้วย อวตารผู้ช่วยที่อัปโหลดจะซ้อนทับตัวตนที่ gateway resolve แล้วเฉพาะในเบราว์เซอร์ภายในเครื่อง และจะไม่เดินทางไปกลับผ่าน `config.patch` ฟิลด์คอนฟิก `ui.assistant.avatar` ที่ใช้ร่วมกันยังคงพร้อมใช้งานสำหรับไคลเอนต์ที่ไม่ใช่ UI ซึ่งเขียนฟิลด์นี้โดยตรง (เช่น gateway แบบสคริปต์หรือแดชบอร์ดแบบกำหนดเอง)

## เอนด์พอยต์คอนฟิกรันไทม์

Control UI ดึงการตั้งค่ารันไทม์จาก `/control-ui-config.json` โดย resolve เทียบกับ base path ของ Control UI ของ gateway (เช่น `/__openclaw__/control-ui-config.json` เมื่อ UI ให้บริการภายใต้ `/__openclaw__/`) เอนด์พอยต์นั้นถูกคุมด้วย gateway auth เดียวกับพื้นผิว HTTP ส่วนที่เหลือ: เบราว์เซอร์ที่ยังไม่ผ่าน auth จะดึงข้อมูลไม่ได้ และการดึงข้อมูลที่สำเร็จต้องมีโทเค็น/รหัสผ่าน gateway ที่ถูกต้องอยู่แล้ว ตัวตน Tailscale Serve หรือ identity ของ trusted-proxy

## การรองรับภาษา

Control UI สามารถแปลตัวเองเป็นภาษาท้องถิ่นเมื่อโหลดครั้งแรกตาม locale ของเบราว์เซอร์คุณ หากต้องการแทนที่ในภายหลัง ให้เปิด **Overview -> Gateway Access -> Language** ตัวเลือก locale อยู่ในการ์ด Gateway Access ไม่ได้อยู่ใต้ Appearance

- locale ที่รองรับ: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- คำแปลที่ไม่ใช่ภาษาอังกฤษจะถูก lazy-loaded ในเบราว์เซอร์
- locale ที่เลือกจะถูกบันทึกในที่เก็บข้อมูลของเบราว์เซอร์และนำมาใช้ซ้ำในการเข้าชมครั้งต่อไป
- คีย์คำแปลที่ขาดหายจะ fallback เป็นภาษาอังกฤษ

การแปลเอกสารถูกสร้างสำหรับชุด locale ที่ไม่ใช่ภาษาอังกฤษเดียวกัน แต่ตัวเลือกภาษา Mintlify ในตัวของไซต์เอกสารถูกจำกัดไว้เฉพาะรหัส locale ที่ Mintlify ยอมรับ เอกสารภาษาไทย (`th`) และเปอร์เซีย (`fa`) ยังคงถูกสร้างใน repo สำหรับเผยแพร่; อาจไม่ปรากฏในตัวเลือกนั้นจนกว่า Mintlify จะรองรับรหัสเหล่านั้น

## ธีมรูปลักษณ์

แผง Appearance เก็บธีม Claw, Knot และ Dash ในตัว รวมถึงช่องนำเข้า tweakcn แบบเฉพาะเบราว์เซอร์หนึ่งช่อง หากต้องการนำเข้าธีม ให้เปิด [ตัวแก้ไข tweakcn](https://tweakcn.com/editor/theme) เลือกหรือสร้างธีม คลิก **Share** แล้ววางลิงก์ธีมที่คัดลอกไว้ลงใน Appearance ตัวนำเข้ายังรองรับ URL registry แบบ `https://tweakcn.com/r/themes/<id>`, URL editor เช่น `https://tweakcn.com/editor/theme?theme=amethyst-haze`, path แบบสัมพัทธ์ `/themes/<id>`, ID ธีมดิบ และชื่อธีมเริ่มต้น เช่น `amethyst-haze`

Appearance ยังมีการตั้งค่าขนาดข้อความแบบเฉพาะเบราว์เซอร์ การตั้งค่านี้ถูกเก็บรวมกับค่ากำหนด Control UI อื่นๆ ใช้กับข้อความแชต ข้อความ composer การ์ดเครื่องมือ และแถบด้านข้างแชต และคงช่องป้อนข้อความไว้ที่อย่างน้อย 16px เพื่อไม่ให้ Safari บนมือถือซูมอัตโนมัติเมื่อ focus

ธีมที่นำเข้าจะถูกเก็บไว้เฉพาะในโปรไฟล์เบราว์เซอร์ปัจจุบันเท่านั้น ธีมเหล่านี้จะไม่ถูกเขียนลงคอนฟิก gateway และไม่ซิงก์ข้ามอุปกรณ์ การแทนที่ธีมที่นำเข้าจะอัปเดตช่องภายในเครื่องหนึ่งช่อง; การล้างช่องนี้จะเปลี่ยนธีมที่ใช้งานกลับเป็น Claw หากธีมที่นำเข้าเคยถูกเลือกไว้

## หน้า MCP

หน้า MCP เฉพาะเป็นมุมมองสำหรับผู้ปฏิบัติงานสำหรับเซิร์ฟเวอร์ MCP ที่ OpenClaw จัดการภายใต้ `mcp.servers` หน้านี้ไม่เริ่มทรานสปอร์ต MCP ด้วยตัวเอง ให้ใช้เพื่อตรวจสอบและแก้ไขค่ากำหนดที่บันทึกไว้ จากนั้นใช้ `openclaw mcp doctor --probe` เมื่อคุณต้องการหลักฐานเซิร์ฟเวอร์แบบสด

เวิร์กโฟลว์ทั่วไป:

1. เปิด **MCP** จากแถบด้านข้าง
2. ตรวจสอบการ์ดสรุปสำหรับจำนวนเซิร์ฟเวอร์ทั้งหมด เปิดใช้งานแล้ว, OAuth และเซิร์ฟเวอร์ที่ถูกกรอง
3. ตรวจสอบแต่ละแถวของเซิร์ฟเวอร์สำหรับทรานสปอร์ต การเปิดใช้งาน การยืนยันตัวตน ตัวกรอง หมดเวลา และคำใบ้คำสั่ง
4. สลับการเปิดใช้งานเมื่อเซิร์ฟเวอร์ควรยังคงถูกกำหนดค่าไว้แต่ไม่เข้าร่วมการค้นพบของรันไทม์
5. แก้ไขส่วนค่ากำหนด `mcp` ที่มีขอบเขตสำหรับนิยามเซิร์ฟเวอร์ ส่วนหัว เส้นทาง TLS/mTLS เมทาดาทา OAuth ตัวกรองเครื่องมือ และเมทาดาทาการฉายภาพของ Codex
6. ใช้ **บันทึก** สำหรับการเขียนค่ากำหนด หรือ **บันทึกและเผยแพร่** เมื่อ Gateway ที่กำลังทำงานควรนำค่ากำหนดที่เปลี่ยนไปใช้
7. รัน `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` หรือ `openclaw mcp reload` จากเทอร์มินัลเมื่อกระบวนการที่แก้ไขต้องการการวินิจฉัยแบบคงที่ หลักฐานแบบสด หรือการทิ้งรันไทม์ที่แคชไว้

หน้าจะปกปิดค่าคล้าย URL ที่มีข้อมูลประจำตัวก่อนแสดงผล และใส่เครื่องหมายคำพูดรอบชื่อเซิร์ฟเวอร์ในตัวอย่างคำสั่ง เพื่อให้คำสั่งที่คัดลอกยังทำงานกับช่องว่างหรืออักขระพิเศษของเชลล์ได้ ข้อมูลอ้างอิง CLI และค่ากำหนดฉบับเต็มอยู่ใน [MCP](/th/cli/mcp)

## แท็บกิจกรรม

แท็บกิจกรรมเป็นตัวสังเกตการณ์เฉพาะเบราว์เซอร์แบบชั่วคราวสำหรับกิจกรรมเครื่องมือแบบสด แท็บนี้ได้มาจากสตรีมเหตุการณ์ Gateway `session.tool` / เครื่องมือเดียวกันที่ขับเคลื่อนการ์ดเครื่องมือใน Chat โดยไม่เพิ่มตระกูลเหตุการณ์ Gateway, endpoint, ที่เก็บกิจกรรมถาวร, ฟีดเมตริก หรือสตรีมตัวสังเกตการณ์ภายนอกอีกชุดหนึ่ง

รายการกิจกรรมเก็บเฉพาะสรุปที่ผ่านการล้างข้อมูลและตัวอย่างเอาต์พุตที่ปกปิดและตัดให้สั้นแล้ว ค่าของอาร์กิวเมนต์เครื่องมือจะไม่ถูกเก็บในสถานะกิจกรรม UI แสดงว่าอาร์กิวเมนต์ถูกซ่อนและบันทึกเฉพาะจำนวนฟิลด์อาร์กิวเมนต์ รายการในหน่วยความจำจะตามแท็บเบราว์เซอร์ปัจจุบัน อยู่รอดเมื่อไปยังหน้าต่าง ๆ ภายใน Control UI และรีเซ็ตเมื่อโหลดหน้าใหม่ สลับเซสชัน หรือกด **ล้าง**

## พฤติกรรม Chat

<AccordionGroup>
  <Accordion title="ความหมายของการส่งและประวัติ">
    - `chat.send` เป็นแบบ **ไม่บล็อก**: ตอบรับทันทีด้วย `{ runId, status: "started" }` และสตรีมคำตอบผ่านเหตุการณ์ `chat` ไคลเอนต์ Control UI ที่เชื่อถือได้อาจได้รับเมทาดาทาเวลา ACK เพิ่มเติมสำหรับการวินิจฉัยในเครื่องด้วย
    - การอัปโหลด Chat รองรับรูปภาพและไฟล์ที่ไม่ใช่วิดีโอ รูปภาพจะคงเส้นทางรูปภาพดั้งเดิมไว้ ส่วนไฟล์อื่นจะถูกจัดเก็บเป็นสื่อที่จัดการและแสดงในประวัติเป็นลิงก์ไฟล์แนบ
    - การส่งซ้ำด้วย `idempotencyKey` เดิมจะคืนค่า `{ status: "in_flight" }` ขณะกำลังรัน และ `{ status: "ok" }` หลังเสร็จสิ้น
    - คำตอบ `chat.history` ถูกจำกัดขนาดเพื่อความปลอดภัยของ UI เมื่อรายการทรานสคริปต์ใหญ่เกินไป Gateway อาจตัดฟิลด์ข้อความยาว ๆ ละเว้นบล็อกเมทาดาทาหนัก ๆ และแทนที่ข้อความที่ใหญ่เกินด้วยตัวแทน (`[chat.history omitted: message too large]`)
    - เมื่อข้อความผู้ช่วยที่มองเห็นได้ถูกตัดใน `chat.history` ตัวอ่านด้านข้างสามารถดึงรายการทรานสคริปต์ฉบับเต็มที่ปรับให้เป็นมาตรฐานสำหรับการแสดงผลตามต้องการผ่าน `chat.message.get` โดยใช้ `sessionKey`, `agentId` ที่ใช้งานอยู่เมื่อจำเป็น และ `messageId` ของทรานสคริปต์ หาก Gateway ยังไม่สามารถคืนข้อมูลเพิ่มเติมได้ ตัวอ่านจะแสดงสถานะไม่พร้อมใช้งานอย่างชัดเจนแทนที่จะทำซ้ำตัวอย่างที่ถูกตัดอย่างเงียบ ๆ
    - รูปภาพที่ผู้ช่วย/ระบบสร้างขึ้นจะถูกบันทึกถาวรเป็นการอ้างอิงสื่อที่จัดการ และให้บริการกลับผ่าน URL สื่อ Gateway ที่ผ่านการยืนยันตัวตน ดังนั้นการโหลดใหม่จึงไม่ขึ้นกับ payload รูปภาพ base64 ดิบที่ยังอยู่ในคำตอบประวัติ Chat
    - เมื่อเรนเดอร์ `chat.history` Control UI จะลบแท็กคำสั่งอินไลน์ที่ใช้เพื่อการแสดงผลเท่านั้นออกจากข้อความผู้ช่วยที่มองเห็นได้ (เช่น `[[reply_to_*]]` และ `[[audio_as_voice]]`), payload XML ของ tool-call แบบข้อความธรรมดา (รวมถึง `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อก tool-call ที่ถูกตัด), และโทเคนควบคุมโมเดล ASCII/แบบเต็มความกว้างที่รั่วไหล และละเว้นรายการผู้ช่วยที่ข้อความมองเห็นได้ทั้งหมดมีเพียงโทเคนเงียบที่ตรงกันทุกประการ `NO_REPLY` / `no_reply` หรือโทเคนตอบรับ Heartbeat `HEARTBEAT_OK`
    - ระหว่างการส่งที่กำลังทำงานและการรีเฟรชประวัติสุดท้าย มุมมอง Chat จะคงข้อความผู้ใช้/ผู้ช่วยแบบ optimistic ในเครื่องให้มองเห็นได้หาก `chat.history` คืน snapshot ที่เก่ากว่าชั่วคราว ทรานสคริปต์ตามบัญญัติจะมาแทนที่ข้อความในเครื่องเหล่านั้นเมื่อประวัติ Gateway ตามทัน
    - เหตุการณ์ `chat` แบบสดเป็นสถานะการส่งมอบ ขณะที่ `chat.history` สร้างใหม่จากทรานสคริปต์เซสชันถาวร หลังเหตุการณ์ tool-final Control UI จะโหลดประวัติใหม่และรวมเฉพาะส่วนท้าย optimistic ขนาดเล็ก ขอบเขตทรานสคริปต์มีบันทึกไว้ใน [WebChat](/th/web/webchat)
    - `chat.inject` เพิ่มโน้ตของผู้ช่วยต่อท้ายทรานสคริปต์เซสชันและกระจายเหตุการณ์ `chat` สำหรับการอัปเดตเฉพาะ UI (ไม่มีการรันเอเจนต์ ไม่มีการส่งมอบช่องทาง)
    - แถบด้านข้างแสดงรายการเซสชันล่าสุดพร้อมการกระทำ New Session, ลิงก์ All Sessions และปุ่มค้นหาเซสชันที่เปิดตัวเลือกเซสชันแบบเต็ม (จำกัดขอบเขตตามเอเจนต์ที่เลือก พร้อมการค้นหาและการแบ่งหน้า) การสลับเอเจนต์จะแสดงเฉพาะเซสชันที่ผูกกับเอเจนต์นั้น และย้อนกลับไปยังเซสชันหลักของเอเจนต์นั้นเมื่อยังไม่มีเซสชันแดชบอร์ดที่บันทึกไว้
    - บนความกว้างเดสก์ท็อป ตัวควบคุม Chat จะอยู่ในแถวกะทัดรัดแถวเดียวและยุบเมื่อเลื่อนลงในทรานสคริปต์ การเลื่อนขึ้น กลับไปด้านบน หรือถึงด้านล่างจะกู้คืนตัวควบคุม
    - ข้อความซ้ำติดต่อกันที่เป็นข้อความล้วนจะแสดงเป็นฟองเดียวพร้อมป้ายจำนวน ข้อความที่มีรูปภาพ ไฟล์แนบ เอาต์พุตเครื่องมือ หรือตัวอย่างแคนวาสจะไม่ถูกยุบ
    - ตัวเลือกโมเดลและการคิดในส่วนหัว Chat จะแพตช์เซสชันที่ใช้งานอยู่ทันทีผ่าน `sessions.patch` เป็นการแทนที่เซสชันแบบถาวร ไม่ใช่ตัวเลือกการส่งสำหรับรอบเดียวเท่านั้น
    - หากคุณส่งข้อความขณะที่การเปลี่ยนตัวเลือกโมเดลสำหรับเซสชันเดียวกันยังบันทึกอยู่ ตัวเขียนจะรอแพตช์เซสชันนั้นก่อนเรียก `chat.send` เพื่อให้การส่งใช้โมเดลที่เลือกไว้
    - การพิมพ์ `/new` ใน Control UI จะสร้างและสลับไปยังเซสชันแดชบอร์ดใหม่แบบเดียวกับ New Chat ยกเว้นเมื่อกำหนดค่า `session.dmScope: "main"` และ parent ปัจจุบันเป็นเซสชันหลักของเอเจนต์ ในกรณีนั้นจะรีเซ็ตเซสชันหลักในที่เดิม การพิมพ์ `/reset` จะคงการรีเซ็ตในที่เดิมอย่างชัดเจนของ Gateway สำหรับเซสชันปัจจุบัน
    - ตัวเลือกโมเดล Chat ขอรับมุมมองโมเดลที่ Gateway กำหนดค่าไว้ หากมี `agents.defaults.models` allowlist นั้นจะขับเคลื่อนตัวเลือก รวมถึงรายการ `provider/*` ที่ทำให้แค็ตตาล็อกตามขอบเขตผู้ให้บริการยังเป็นแบบไดนามิก มิฉะนั้นตัวเลือกจะแสดงรายการ `models.providers.*.models` ที่ระบุชัดเจน บวกกับผู้ให้บริการที่มีการยืนยันตัวตนที่ใช้ได้ แค็ตตาล็อกฉบับเต็มยังคงเข้าถึงได้ผ่าน RPC ดีบัก `models.list` พร้อม `view: "all"`
    - เมื่อรายงานการใช้งานเซสชัน Gateway ล่าสุดมีโทเคนบริบทปัจจุบัน แถบเครื่องมือของตัวเขียน Chat จะแสดงวงแหวนการใช้งานบริบทขนาดเล็กพร้อมเปอร์เซ็นต์ที่ใช้ไป รายละเอียดโทเคนฉบับเต็มอยู่ในทูลทิป วงแหวนจะสลับเป็นสไตล์คำเตือนเมื่อแรงกดดันบริบทสูง และที่ระดับ Compaction ที่แนะนำจะแสดงปุ่มกะทัดรัดที่รันเส้นทาง Compaction ของเซสชันตามปกติ snapshot โทเคนที่ล้าสมัยจะถูกซ่อนไว้จนกว่า Gateway จะรายงานการใช้งานล่าสุดอีกครั้ง

  </Accordion>
  <Accordion title="โหมดพูดคุย (เรียลไทม์ในเบราว์เซอร์)">
    โหมดพูดคุยใช้ผู้ให้บริการเสียงเรียลไทม์ที่ลงทะเบียนไว้ กำหนดค่า OpenAI ด้วย `talk.realtime.provider: "openai"` พร้อมโปรไฟล์การยืนยันตัวตนแบบคีย์ API ของ `openai`, `talk.realtime.providers.openai.apiKey` หรือ `OPENAI_API_KEY`; โปรไฟล์ OAuth ของ OpenAI ไม่ได้กำหนดค่าเสียง Realtime กำหนดค่า Google ด้วย `talk.realtime.provider: "google"` พร้อม `talk.realtime.providers.google.apiKey` เบราว์เซอร์จะไม่ได้รับคีย์ API ผู้ให้บริการมาตรฐาน OpenAI ได้รับ client secret ของ Realtime แบบชั่วคราวสำหรับ WebRTC Google Live ได้รับโทเคนการยืนยันตัวตน Live API แบบใช้ครั้งเดียวที่ถูกจำกัดสำหรับเซสชัน WebSocket ของเบราว์เซอร์ โดยมีคำสั่งและการประกาศเครื่องมือถูกล็อกไว้ในโทเคนโดย Gateway ผู้ให้บริการที่เปิดเผยเฉพาะ backend realtime bridge จะรันผ่านทรานสปอร์ต Gateway relay ดังนั้นข้อมูลประจำตัวและซ็อกเก็ตของผู้ให้บริการจะอยู่ฝั่งเซิร์ฟเวอร์ ขณะที่เสียงของเบราว์เซอร์เคลื่อนผ่าน RPC ของ Gateway ที่ผ่านการยืนยันตัวตน พรอมป์เซสชัน Realtime ประกอบโดย Gateway; `talk.client.create` ไม่ยอมรับการแทนที่คำสั่งที่ผู้เรียกจัดเตรียม

    ตัวเขียน Chat มีปุ่มตัวเลือก Talk อยู่ถัดจากปุ่มเริ่ม/หยุด Talk ตัวเลือกจะใช้กับเซสชัน Talk ถัดไปและสามารถแทนที่ผู้ให้บริการ ทรานสปอร์ต โมเดล เสียง ระดับความพยายามในการให้เหตุผล เกณฑ์ VAD ระยะเวลาเงียบ และ padding คำนำหน้า เมื่อปล่อยตัวเลือกว่าง Gateway จะใช้ค่าเริ่มต้นที่กำหนดค่าไว้เมื่อมี หรือค่าเริ่มต้นของผู้ให้บริการ การเลือก Gateway relay จะบังคับเส้นทาง backend relay ส่วนการเลือก WebRTC จะคงให้เซสชันเป็นของไคลเอนต์และล้มเหลวแทนที่จะย้อนกลับไป relay อย่างเงียบ ๆ หากผู้ให้บริการไม่สามารถสร้างเซสชันเบราว์เซอร์ได้

    ในตัวเขียน Chat ตัวควบคุม Talk คือปุ่มรูปคลื่นถัดจากปุ่มการเขียนตามคำบอกด้วยไมโครโฟน เมื่อ Talk เริ่ม แถวสถานะของตัวเขียนจะแสดง `Connecting Talk...` จากนั้น `Talk live` ขณะเชื่อมต่อเสียงแล้ว หรือ `Asking OpenClaw...` ขณะ tool call แบบเรียลไทม์กำลังปรึกษาโมเดลขนาดใหญ่กว่าที่กำหนดค่าไว้ผ่าน `talk.client.toolCall`

    การทดสอบสดสำหรับผู้ดูแล: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` ตรวจสอบ backend WebSocket bridge ของ OpenAI, การแลกเปลี่ยน SDP ของ WebRTC ในเบราว์เซอร์ OpenAI, การตั้งค่า WebSocket ของเบราว์เซอร์ด้วยโทเคนจำกัดของ Google Live และอะแดปเตอร์เบราว์เซอร์ Gateway relay พร้อมสื่อไมโครโฟนปลอม คำสั่งนี้พิมพ์เฉพาะสถานะผู้ให้บริการและไม่บันทึก secrets

  </Accordion>
  <Accordion title="หยุดและยกเลิก">
    - คลิก **หยุด** (เรียก `chat.abort`)
    - ขณะรันกำลังทำงาน การติดตามผลปกติจะเข้าคิว คลิก **Steer** บนข้อความที่เข้าคิวเพื่อฉีดการติดตามผลนั้นเข้าไปในรอบที่กำลังรัน
    - พิมพ์ `/stop` (หรือวลี abort แบบ standalone เช่น `stop`, `stop action`, `stop run`, `stop openclaw`, `please stop`) เพื่อยกเลิกนอกแบนด์
    - `chat.abort` รองรับ `{ sessionKey }` (ไม่มี `runId`) เพื่อยกเลิกรันที่ใช้งานอยู่ทั้งหมดสำหรับเซสชันนั้น

  </Accordion>

## การติดตั้ง PWA และ web push

Control UI มาพร้อม `manifest.webmanifest` และ service worker ดังนั้นเบราว์เซอร์สมัยใหม่จึงสามารถติดตั้งเป็น PWA แบบ standalone ได้ Web Push ช่วยให้ Gateway ปลุก PWA ที่ติดตั้งแล้วด้วยการแจ้งเตือนแม้แท็บหรือหน้าต่างเบราว์เซอร์จะไม่ได้เปิดอยู่ก็ตาม

ถ้าหน้าแสดง **โปรโตคอลไม่ตรงกัน** ทันทีหลังอัปเดต OpenClaw ให้เปิดแดชบอร์ดใหม่ด้วย `openclaw dashboard` แล้วรีเฟรชหน้าแบบข้ามแคชก่อน ถ้ายังล้มเหลว ให้ล้างข้อมูลไซต์ของต้นทางแดชบอร์ดหรือทดสอบในหน้าต่างเบราว์เซอร์ส่วนตัว แท็บเก่าหรือแคช service worker ของเบราว์เซอร์อาจยังรันบันเดิล Control UI ก่อนอัปเดตกับ Gateway รุ่นใหม่กว่าอยู่

| พื้นผิว                                               | สิ่งที่ทำ                                                       |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | แมนิเฟสต์ PWA เบราว์เซอร์จะเสนอ "ติดตั้งแอป" เมื่อเข้าถึงได้   |
| `ui/public/sw.js`                                     | service worker ที่จัดการเหตุการณ์ `push` และการคลิกการแจ้งเตือน |
| `push/vapid-keys.json` (ใต้ไดเรกทอรีสถานะของ OpenClaw) | คู่คีย์ VAPID ที่สร้างอัตโนมัติ ใช้ลงนามเพย์โหลด Web Push       |
| `push/web-push-subscriptions.json`                    | endpoint การสมัครรับข้อมูลของเบราว์เซอร์ที่บันทึกถาวรไว้                          |

แทนที่คู่คีย์ VAPID ผ่าน env vars บนโปรเซส Gateway เมื่อคุณต้องการตรึงคีย์ (สำหรับการปรับใช้หลายโฮสต์ การหมุนเวียน secret หรือการทดสอบ):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (ค่าเริ่มต้นคือ `https://openclaw.ai`)

Control UI ใช้เมธอด Gateway ที่จำกัดด้วย scope เหล่านี้เพื่อขึ้นทะเบียนและทดสอบการสมัครรับข้อมูลของเบราว์เซอร์:

- `push.web.vapidPublicKey` — ดึงคีย์สาธารณะ VAPID ที่ใช้งานอยู่
- `push.web.subscribe` — ขึ้นทะเบียน `endpoint` พร้อม `keys.p256dh`/`keys.auth`
- `push.web.unsubscribe` — ลบ endpoint ที่ขึ้นทะเบียนไว้
- `push.web.test` — ส่งการแจ้งเตือนทดสอบไปยังการสมัครรับข้อมูลของผู้เรียก

<Note>
Web Push แยกจากเส้นทางรีเลย์ APNS ของ iOS (ดู [การกำหนดค่า](/th/gateway/configuration) สำหรับ push ที่มีรีเลย์รองรับ) และเมธอด `push.test` ที่มีอยู่ ซึ่งมุ่งเป้าไปที่การจับคู่มือถือแบบ native
</Note>

## การฝังแบบโฮสต์

ข้อความของผู้ช่วยสามารถเรนเดอร์เนื้อหาเว็บที่โฮสต์ไว้แบบ inline ด้วย shortcode `[embed ...]` นโยบาย sandbox ของ iframe ถูกควบคุมโดย `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    ปิดการประมวลผลสคริปต์ภายใน embed ที่โฮสต์ไว้
  </Tab>
  <Tab title="scripts (ค่าเริ่มต้น)">
    อนุญาต embed แบบโต้ตอบได้พร้อมคงการแยก origin ไว้ นี่คือค่าเริ่มต้นและมักเพียงพอสำหรับเกม/วิดเจ็ตเบราว์เซอร์แบบ self-contained
  </Tab>
  <Tab title="trusted">
    เพิ่ม `allow-same-origin` เหนือ `allow-scripts` สำหรับเอกสารไซต์เดียวกันที่ตั้งใจต้องการสิทธิ์ที่สูงกว่า
  </Tab>
</Tabs>

ตัวอย่าง:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
ใช้ `trusted` เฉพาะเมื่อเอกสารที่ฝังต้องการพฤติกรรม same-origin จริง ๆ สำหรับเกมและ canvas แบบโต้ตอบที่สร้างโดยเอเจนต์ส่วนใหญ่ `scripts` เป็นตัวเลือกที่ปลอดภัยกว่า
</Warning>

URL embed ภายนอกแบบสัมบูรณ์ `http(s)` จะยังถูกบล็อกตามค่าเริ่มต้น ถ้าคุณตั้งใจให้ `[embed url="https://..."]` โหลดหน้าของบุคคลที่สาม ให้ตั้งค่า `gateway.controlUi.allowExternalEmbedUrls: true`

## ความกว้างของข้อความแชต

ข้อความแชตที่จัดกลุ่มใช้ max-width ค่าเริ่มต้นที่อ่านง่าย การปรับใช้บนจอกว้างสามารถแทนที่ได้โดยไม่ต้องแพตช์ CSS ที่บันเดิลมา ด้วยการตั้งค่า `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

ค่าจะถูกตรวจสอบก่อนถึงเบราว์เซอร์ ค่าที่รองรับรวมถึงความยาวและเปอร์เซ็นต์แบบธรรมดา เช่น `960px` หรือ `82%` รวมถึงนิพจน์ความกว้างแบบจำกัด `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` และ `fit-content(...)`

    ตามค่าเริ่มต้น คำขอ Control UI/WebSocket Serve สามารถยืนยันตัวตนผ่านส่วนหัวตัวตนของ Tailscale (`tailscale-user-login`) เมื่อ `gateway.auth.allowTailscale` เป็น `true` OpenClaw ตรวจสอบตัวตนโดยแก้ที่อยู่ `x-forwarded-for` ด้วย `tailscale whois` และจับคู่กับส่วนหัว และยอมรับเฉพาะเมื่อคำขอเข้าถึง loopback พร้อมส่วนหัว `x-forwarded-*` ของ Tailscale สำหรับเซสชันผู้ปฏิบัติการ Control UI ที่มีตัวตนอุปกรณ์ของเบราว์เซอร์ เส้นทาง Serve ที่ตรวจสอบแล้วนี้จะข้ามรอบการจับคู่อุปกรณ์ด้วย เบราว์เซอร์ที่ไม่มีอุปกรณ์และการเชื่อมต่อบทบาท node ยังคงตามการตรวจสอบอุปกรณ์ปกติ ตั้งค่า `gateway.auth.allowTailscale: false` ถ้าคุณต้องการบังคับใช้ข้อมูลรับรอง shared-secret แบบชัดเจนแม้กับทราฟฟิก Serve จากนั้นใช้ `gateway.auth.mode: "token"` หรือ `"password"`

    สำหรับเส้นทางตัวตน Serve แบบ async นั้น ความพยายามยืนยันตัวตนที่ล้มเหลวสำหรับ IP ไคลเอนต์และ scope การยืนยันตัวตนเดียวกันจะถูกจัดลำดับก่อนเขียน rate-limit ดังนั้นการลองซ้ำที่ผิดพร้อมกันจากเบราว์เซอร์เดียวกันอาจแสดง `retry later` ในคำขอที่สองแทนที่จะเป็น mismatch ธรรมดาสองรายการที่แข่งกันแบบขนาน

    <Warning>
    การยืนยันตัวตน Serve แบบไม่มี token ถือว่าโฮสต์ gateway เชื่อถือได้ ถ้าโค้ด local ที่ไม่น่าเชื่อถืออาจรันบนโฮสต์นั้น ให้บังคับใช้การยืนยันตัวตนด้วย token/password
    </Warning>

  </Tab>
  <Tab title="ผูกกับ tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    จากนั้นเปิด:

    - `http://<tailscale-ip>:18789/` (หรือ `gateway.controlUi.basePath` ที่คุณกำหนดค่าไว้)

    วาง shared secret ที่ตรงกันลงในการตั้งค่า UI (ส่งเป็น `connect.params.auth.token` หรือ `connect.params.auth.password`)

  </Tab>
</Tabs>

## HTTP ที่ไม่ปลอดภัย

ถ้าคุณเปิดแดชบอร์ดผ่าน HTTP แบบธรรมดา (`http://<lan-ip>` หรือ `http://<tailscale-ip>`) เบราว์เซอร์จะรันใน **บริบทที่ไม่ปลอดภัย** และบล็อก WebCrypto ตามค่าเริ่มต้น OpenClaw **บล็อก** การเชื่อมต่อ Control UI ที่ไม่มีตัวตนอุปกรณ์

ข้อยกเว้นที่บันทึกไว้:

- ความเข้ากันได้ของ HTTP ที่ไม่ปลอดภัยเฉพาะ localhost ด้วย `gateway.controlUi.allowInsecureAuth=true`
- การยืนยันตัวตน Control UI ของผู้ปฏิบัติการสำเร็จผ่าน `gateway.auth.mode: "trusted-proxy"`
- ทางเลือกฉุกเฉิน `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**วิธีแก้ที่แนะนำ:** ใช้ HTTPS (Tailscale Serve) หรือเปิด UI แบบ local:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (บนโฮสต์ gateway)

<AccordionGroup>
  <Accordion title="พฤติกรรม toggle การยืนยันตัวตนที่ไม่ปลอดภัย">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` เป็น toggle ความเข้ากันได้แบบ local เท่านั้น:

    - อนุญาตให้เซสชัน Control UI ของ localhost ดำเนินต่อได้โดยไม่มีตัวตนอุปกรณ์ในบริบท HTTP ที่ไม่ปลอดภัย
    - ไม่ข้ามการตรวจสอบการจับคู่
    - ไม่ผ่อนปรนข้อกำหนดตัวตนอุปกรณ์ระยะไกล (ที่ไม่ใช่ localhost)

  </Accordion>
  <Accordion title="ฉุกเฉินเท่านั้น">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` ปิดการตรวจสอบตัวตนอุปกรณ์ของ Control UI และเป็นการลดระดับความปลอดภัยอย่างรุนแรง ให้ย้อนกลับอย่างรวดเร็วหลังการใช้งานฉุกเฉิน
    </Warning>

  </Accordion>
  <Accordion title="หมายเหตุ trusted-proxy">
    - การยืนยันตัวตน trusted-proxy ที่สำเร็จสามารถรับเซสชัน Control UI ของ **ผู้ปฏิบัติการ** ได้โดยไม่มีตัวตนอุปกรณ์
    - สิ่งนี้ **ไม่** ขยายไปถึงเซสชัน Control UI บทบาท node
    - reverse proxy แบบ loopback บนโฮสต์เดียวกันยังไม่เป็นไปตามการยืนยันตัวตน trusted-proxy ดู [การยืนยันตัวตน trusted proxy](/th/gateway/trusted-proxy-auth)

  </Accordion>
</AccordionGroup>

ดู [Tailscale](/th/gateway/tailscale) สำหรับคำแนะนำการตั้งค่า HTTPS

## นโยบายความปลอดภัยของเนื้อหา

Control UI มาพร้อมนโยบาย `img-src` ที่เข้มงวด: อนุญาตเฉพาะ asset **same-origin**, URL `data:` และ URL `blob:` ที่สร้างในเครื่องเท่านั้น URL รูปภาพระยะไกล `http(s)` และแบบ protocol-relative จะถูกเบราว์เซอร์ปฏิเสธและจะไม่ออกคำขอเครือข่าย

ในทางปฏิบัติหมายความว่า:

- อวาตาร์และรูปภาพที่เสิร์ฟใต้พาธสัมพัทธ์ (เช่น `/avatars/<id>`) ยังเรนเดอร์ได้ รวมถึง route อวาตาร์ที่ต้องยืนยันตัวตนซึ่ง UI ดึงมาและแปลงเป็น URL `blob:` แบบ local
- URL inline `data:image/...` ยังเรนเดอร์ได้ (มีประโยชน์สำหรับเพย์โหลดในโปรโตคอล)
- URL `blob:` แบบ local ที่ Control UI สร้างยังเรนเดอร์ได้
- URL อวาตาร์ระยะไกลที่ metadata ของช่องทางปล่อยออกมาจะถูก helper อวาตาร์ของ Control UI ตัดออกและแทนที่ด้วยโลโก้/ป้ายที่มีอยู่ในตัว ดังนั้นช่องทางที่ถูกเจาะหรือเป็นอันตรายจึงไม่สามารถบังคับการดึงรูปภาพระยะไกลจากเบราว์เซอร์ของผู้ปฏิบัติการได้ตามอำเภอใจ

คุณไม่จำเป็นต้องเปลี่ยนอะไรเพื่อให้ได้พฤติกรรมนี้ — เปิดใช้งานเสมอและกำหนดค่าไม่ได้

## การยืนยันตัวตนของ route อวาตาร์

เมื่อกำหนดค่าการยืนยันตัวตน gateway แล้ว endpoint อวาตาร์ของ Control UI ต้องใช้ gateway token เดียวกับส่วนที่เหลือของ API:

- `GET /avatar/<agentId>` ส่งคืนรูปภาพอวาตาร์ให้เฉพาะผู้เรียกที่ยืนยันตัวตนแล้วเท่านั้น `GET /avatar/<agentId>?meta=1` ส่งคืน metadata อวาตาร์ภายใต้กฎเดียวกัน
- คำขอที่ไม่ยืนยันตัวตนไปยัง route ใด route หนึ่งจะถูกปฏิเสธ (ตรงกับ route assistant-media ที่เป็นพี่น้องกัน) สิ่งนี้ป้องกันไม่ให้ route อวาตาร์รั่วตัวตนของเอเจนต์บนโฮสต์ที่ได้รับการปกป้องอยู่แล้ว
- Control UI เองจะส่งต่อ gateway token เป็นส่วนหัว bearer เมื่อดึงอวาตาร์ และใช้ URL blob ที่ยืนยันตัวตนแล้วเพื่อให้รูปภาพยังเรนเดอร์ในแดชบอร์ดได้

ถ้าคุณปิดการยืนยันตัวตน gateway (ไม่แนะนำบนโฮสต์ที่ใช้ร่วมกัน) route อวาตาร์ก็จะไม่ต้องยืนยันตัวตนด้วย สอดคล้องกับส่วนที่เหลือของ gateway

## การยืนยันตัวตนของ route สื่อผู้ช่วย

เมื่อกำหนดค่าการยืนยันตัวตน gateway แล้ว ตัวอย่างสื่อ local ของผู้ช่วยใช้ route สองขั้นตอน:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` ต้องใช้การยืนยันตัวตนผู้ปฏิบัติการ Control UI ปกติ เบราว์เซอร์ส่ง gateway token เป็นส่วนหัว bearer เมื่อตรวจสอบความพร้อมใช้งาน
- การตอบกลับ metadata ที่สำเร็จจะมี `mediaTicket` อายุสั้นที่จำกัดกับพาธต้นทางนั้นโดยตรง
- URL รูปภาพ เสียง วิดีโอ และเอกสารที่เบราว์เซอร์เรนเดอร์ใช้ `mediaTicket=<ticket>` แทน gateway token หรือรหัสผ่านที่ใช้งานอยู่ ticket จะหมดอายุอย่างรวดเร็วและไม่สามารถอนุญาตต้นทางอื่นได้

สิ่งนี้ทำให้การเรนเดอร์สื่อปกติเข้ากันได้กับองค์ประกอบสื่อ native ของเบราว์เซอร์ โดยไม่วางข้อมูลรับรอง gateway ที่นำกลับมาใช้ซ้ำได้ไว้ใน URL สื่อที่มองเห็นได้

## การ build UI

Gateway เสิร์ฟไฟล์ static จาก `dist/control-ui` Build ด้วย:

```bash
pnpm ui:build
```

base แบบสัมบูรณ์ที่เป็นตัวเลือก (เมื่อคุณต้องการ URL asset แบบคงที่):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

สำหรับการพัฒนา local (dev server แยกต่างหาก):

```bash
pnpm ui:dev
```

จากนั้นชี้ UI ไปที่ URL Gateway WS ของคุณ (เช่น `ws://127.0.0.1:18789`)

## หน้า Control UI ว่างเปล่า

ถ้าเบราว์เซอร์โหลดแดชบอร์ดว่างเปล่าและ DevTools ไม่แสดงข้อผิดพลาดที่เป็นประโยชน์ extension หรือ content script ระยะแรกอาจป้องกันไม่ให้แอปโมดูล JavaScript ประเมินผล หน้า static มีแผงกู้คืน HTML แบบธรรมดาที่จะปรากฏเมื่อ `<openclaw-app>` ไม่ถูกขึ้นทะเบียนหลังเริ่มต้น

ใช้การทำงาน **ลองอีกครั้ง** ของแผงหลังเปลี่ยนสภาพแวดล้อมเบราว์เซอร์ หรือโหลดซ้ำด้วยตนเองหลังการตรวจสอบเหล่านี้:

- ปิด extension ที่ inject เข้าไปในทุกหน้า โดยเฉพาะ extension ที่มี content script `<all_urls>`
- ลองหน้าต่างส่วนตัว โปรไฟล์เบราว์เซอร์สะอาด หรือเบราว์เซอร์อื่น
- ให้ Gateway รันอยู่และตรวจสอบ URL แดชบอร์ดเดียวกันหลังเปลี่ยนเบราว์เซอร์

## การดีบัก/ทดสอบ: dev server + Gateway ระยะไกล

Control UI เป็นไฟล์ static เป้าหมาย WebSocket สามารถกำหนดค่าได้และอาจต่างจาก HTTP origin สิ่งนี้สะดวกเมื่อคุณต้องการ Vite dev server แบบ local แต่ Gateway รันที่อื่น

<Steps>
  <Step title="เริ่ม UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="เปิดด้วย gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    การยืนยันตัวตนครั้งเดียวที่เป็นตัวเลือก (ถ้าจำเป็น):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Notes">
    - `gatewayUrl` จะถูกเก็บไว้ใน localStorage หลังจากโหลด แล้วนำออกจาก URL
    - หากคุณส่ง endpoint `ws://` หรือ `wss://` แบบเต็มผ่าน `gatewayUrl` ให้ URL-encode ค่า `gatewayUrl` เพื่อให้เบราว์เซอร์แยกวิเคราะห์ query string ได้อย่างถูกต้อง
    - ควรส่ง `token` ผ่าน URL fragment (`#token=...`) เมื่อเป็นไปได้ Fragment จะไม่ถูกส่งไปยังเซิร์ฟเวอร์ ซึ่งช่วยหลีกเลี่ยงการรั่วไหลใน request-log และ Referer พารามิเตอร์ query แบบเดิม `?token=` ยังคงถูกนำเข้าได้หนึ่งครั้งเพื่อความเข้ากันได้ แต่ใช้เป็น fallback เท่านั้น และจะถูกลบทันทีหลัง bootstrap
    - `password` จะถูกเก็บไว้ในหน่วยความจำเท่านั้น
    - เมื่อตั้งค่า `gatewayUrl` แล้ว UI จะไม่ fallback ไปใช้ข้อมูลรับรองจาก config หรือ environment ให้ระบุ `token` (หรือ `password`) อย่างชัดเจน การไม่มีข้อมูลรับรองที่ระบุชัดเจนถือเป็นข้อผิดพลาด
    - ใช้ `wss://` เมื่อ Gateway อยู่หลัง TLS (Tailscale Serve, HTTPS proxy ฯลฯ)
    - `gatewayUrl` จะยอมรับเฉพาะในหน้าต่างระดับบนสุดเท่านั้น (ไม่ใช่แบบฝัง) เพื่อป้องกัน clickjacking
    - การปรับใช้ Control UI แบบสาธารณะที่ไม่ใช่ loopback ต้องตั้งค่า `gateway.controlUi.allowedOrigins` อย่างชัดเจน (origin แบบเต็ม) การโหลดผ่าน LAN/Tailnet ส่วนตัวแบบ same-origin จาก loopback, RFC1918/link-local, `.local`, `.ts.net` หรือโฮสต์ Tailscale CGNAT จะได้รับการยอมรับโดยไม่ต้องเปิดใช้ Host-header fallback
    - การเริ่มต้น Gateway อาจ seed origin ภายในเครื่อง เช่น `http://localhost:<port>` และ `http://127.0.0.1:<port>` จาก bind และ port ของ runtime ที่มีผล แต่ origin ของเบราว์เซอร์ระยะไกลยังคงต้องมีรายการที่ระบุชัดเจน
    - อย่าใช้ `gateway.controlUi.allowedOrigins: ["*"]` ยกเว้นสำหรับการทดสอบภายในเครื่องที่ควบคุมอย่างเข้มงวด หมายถึงอนุญาต browser origin ใดก็ได้ ไม่ใช่ “จับคู่กับ host ใดก็ตามที่ฉันใช้อยู่”
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` เปิดใช้โหมด Host-header origin fallback แต่เป็นโหมดความปลอดภัยที่อันตราย

  </Accordion>
</AccordionGroup>

ตัวอย่าง:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
