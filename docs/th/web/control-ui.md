UI ควบคุมเป็นแอปหน้าเดียว **Vite + Lit** ขนาดเล็กที่ให้บริการโดย Gateway:

- ค่าเริ่มต้น: `http://<host>:18789/`
- คำนำหน้าที่เลือกได้: ตั้งค่า `gateway.controlUi.basePath` (เช่น `/openclaw`)

แอปนี้สื่อสาร **โดยตรงกับ Gateway WebSocket** บนพอร์ตเดียวกัน

## เปิดอย่างรวดเร็ว (ภายในเครื่อง)

หาก Gateway กำลังทำงานบนคอมพิวเตอร์เครื่องเดียวกัน ให้เปิด:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (หรือ [http://localhost:18789/](http://localhost:18789/))

หากหน้าโหลดไม่สำเร็จ ให้เริ่ม Gateway ก่อน: `openclaw gateway`

ระบบส่งข้อมูลยืนยันตัวตนระหว่างการจับมือ WebSocket ผ่าน:

- `connect.params.auth.token`
- `connect.params.auth.password`
- ส่วนหัวข้อมูลระบุตัวตนของ Tailscale Serve เมื่อ `gateway.auth.allowTailscale: true`
- ส่วนหัวข้อมูลระบุตัวตนของพร็อกซีที่เชื่อถือได้เมื่อ `gateway.auth.mode: "trusted-proxy"`

แผงการตั้งค่าแดชบอร์ดจะเก็บโทเค็นสำหรับเซสชันแท็บเบราว์เซอร์ปัจจุบันและ URL ของ Gateway ที่เลือกไว้ โดยจะไม่เก็บรหัสผ่านถาวร โดยทั่วไปการเริ่มต้นใช้งานจะสร้างโทเค็น Gateway สำหรับการยืนยันตัวตนแบบความลับร่วมเมื่อเชื่อมต่อครั้งแรก แต่การยืนยันตัวตนด้วยรหัสผ่านก็ใช้งานได้เช่นกันเมื่อ `gateway.auth.mode` เป็น `"password"`

## การจับคู่อุปกรณ์ (การเชื่อมต่อครั้งแรก)

เมื่อคุณเชื่อมต่อกับ UI ควบคุมจากเบราว์เซอร์หรืออุปกรณ์ใหม่ Gateway มักจะต้องมี **การอนุมัติการจับคู่ครั้งเดียว** นี่เป็นมาตรการรักษาความปลอดภัยเพื่อป้องกันการเข้าถึงโดยไม่ได้รับอนุญาต

**สิ่งที่คุณจะเห็น:** "disconnected (1008): pairing required"

<Steps>
  <Step title="แสดงรายการคำขอที่รอดำเนินการ">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="อนุมัติตาม ID คำขอ">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

หากเบราว์เซอร์ลองจับคู่อีกครั้งด้วยรายละเอียดการยืนยันตัวตนที่เปลี่ยนไป (บทบาท/ขอบเขต/คีย์สาธารณะ) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่และจะสร้าง `requestId` ใหม่ เรียกใช้ `openclaw devices list` อีกครั้งก่อนอนุมัติ

หากเบราว์เซอร์จับคู่ไว้แล้วและคุณเปลี่ยนจากสิทธิ์อ่านเป็นสิทธิ์เขียน/ผู้ดูแลระบบ ระบบจะถือว่านี่เป็นการอัปเกรดการอนุมัติ ไม่ใช่การเชื่อมต่อใหม่แบบเงียบ OpenClaw จะคงการอนุมัติเดิมไว้ บล็อกการเชื่อมต่อใหม่ที่มีสิทธิ์กว้างขึ้น และขอให้คุณอนุมัติชุดขอบเขตใหม่อย่างชัดเจน

เมื่ออนุมัติแล้ว อุปกรณ์จะถูกจดจำและไม่ต้องอนุมัติซ้ำ เว้นแต่คุณจะเพิกถอนด้วย `openclaw devices revoke --device <id> --role <role>` ดู [CLI อุปกรณ์](/th/cli/devices) สำหรับการหมุนเวียนและการเพิกถอนโทเค็น

<Note>
- การเชื่อมต่อเบราว์เซอร์ local loopback โดยตรง (`127.0.0.1` / `localhost`) จะได้รับการอนุมัติอัตโนมัติ
- Tailscale Serve สามารถข้ามรอบการจับคู่สำหรับเซสชันผู้ปฏิบัติงาน UI ควบคุมได้เมื่อ `gateway.auth.allowTailscale: true`, ตรวจสอบข้อมูลระบุตัวตน Tailscale สำเร็จ และเบราว์เซอร์แสดงข้อมูลระบุตัวตนอุปกรณ์ของตน
- การผูก Tailnet โดยตรง การเชื่อมต่อเบราว์เซอร์ผ่าน LAN และโปรไฟล์เบราว์เซอร์ที่ไม่มีข้อมูลระบุตัวตนอุปกรณ์ยังคงต้องอนุมัติอย่างชัดเจน
- โปรไฟล์เบราว์เซอร์แต่ละรายการจะสร้าง ID อุปกรณ์ที่ไม่ซ้ำกัน ดังนั้นการเปลี่ยนเบราว์เซอร์หรือล้างข้อมูลเบราว์เซอร์จะต้องจับคู่ใหม่

</Note>

## ตัวตนส่วนบุคคล (เฉพาะเบราว์เซอร์)

UI ควบคุมรองรับตัวตนส่วนบุคคลต่อเบราว์เซอร์ (ชื่อที่แสดงและอวาตาร์) ที่แนบไปกับข้อความขาออกเพื่อระบุแหล่งที่มาในเซสชันที่แชร์ ข้อมูลนี้อยู่ในพื้นที่จัดเก็บของเบราว์เซอร์ จำกัดขอบเขตไว้ที่โปรไฟล์เบราว์เซอร์ปัจจุบัน และจะไม่ซิงก์ไปยังอุปกรณ์อื่นหรือคงอยู่ฝั่งเซิร์ฟเวอร์ นอกเหนือจากเมทาดาทาการเป็นผู้เขียนบทสนทนาตามปกติในข้อความที่คุณส่งจริง การล้างข้อมูลไซต์หรือเปลี่ยนเบราว์เซอร์จะรีเซ็ตให้ว่างเปล่า

รูปแบบเฉพาะเบราว์เซอร์เดียวกันนี้ใช้กับการแทนที่อวาตาร์ของผู้ช่วย อวาตาร์ผู้ช่วยที่อัปโหลดจะวางทับข้อมูลระบุตัวตนที่ Gateway แก้ไขแล้วเฉพาะบนเบราว์เซอร์ภายในเครื่องเท่านั้น และจะไม่เดินทางไปกลับผ่าน `config.patch` ฟิลด์คอนฟิกที่แชร์ `ui.assistant.avatar` ยังคงพร้อมใช้งานสำหรับไคลเอนต์ที่ไม่ใช่ UI ซึ่งเขียนฟิลด์นี้โดยตรง (เช่น Gateway แบบสคริปต์หรือแดชบอร์ดที่กำหนดเอง)

## เอนด์พอยต์คอนฟิกรันไทม์

UI ควบคุมดึงการตั้งค่ารันไทม์จาก `/__openclaw/control-ui-config.json` เอนด์พอยต์นั้นถูกควบคุมด้วยการยืนยันตัวตน Gateway แบบเดียวกับพื้นผิว HTTP ส่วนที่เหลือ: เบราว์เซอร์ที่ไม่ได้ยืนยันตัวตนจะดึงข้อมูลไม่ได้ และการดึงข้อมูลที่สำเร็จต้องมีโทเค็น/รหัสผ่าน Gateway ที่ถูกต้องอยู่แล้ว ข้อมูลระบุตัวตน Tailscale Serve หรือข้อมูลระบุตัวตนจากพร็อกซีที่เชื่อถือได้

## การรองรับภาษา

UI ควบคุมสามารถแปลตัวเองเมื่อโหลดครั้งแรกตามภาษาเบราว์เซอร์ของคุณ หากต้องการแทนที่ในภายหลัง ให้เปิด **ภาพรวม -> การเข้าถึง Gateway -> ภาษา** ตัวเลือกภาษาอยู่ในการ์ดการเข้าถึง Gateway ไม่ได้อยู่ใต้รูปลักษณ์

- ภาษาที่รองรับ: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- คำแปลที่ไม่ใช่ภาษาอังกฤษจะถูกโหลดแบบ lazy-loaded ในเบราว์เซอร์
- ภาษาที่เลือกจะถูกบันทึกไว้ในพื้นที่จัดเก็บของเบราว์เซอร์และนำกลับมาใช้ในการเข้าชมครั้งต่อไป
- คีย์คำแปลที่หายไปจะย้อนกลับไปใช้ภาษาอังกฤษ

คำแปลเอกสารถูกสร้างสำหรับชุดภาษาที่ไม่ใช่ภาษาอังกฤษเดียวกัน แต่ตัวเลือกภาษาที่มีในตัวของไซต์เอกสาร Mintlify ถูกจำกัดตามรหัสภาษาที่ Mintlify ยอมรับ เอกสารภาษาไทย (`th`) และเปอร์เซีย (`fa`) ยังคงถูกสร้างในรีโพเผยแพร่ แต่อาจไม่ปรากฏในตัวเลือกนั้นจนกว่า Mintlify จะรองรับรหัสเหล่านั้น

## ธีมรูปลักษณ์

แผงรูปลักษณ์ยังคงมีธีม Claw, Knot และ Dash ที่มีในตัว รวมถึงสล็อตนำเข้า tweakcn เฉพาะเบราว์เซอร์หนึ่งสล็อต หากต้องการนำเข้าธีม ให้เปิด [ตัวแก้ไข tweakcn](https://tweakcn.com/editor/theme) เลือกหรือสร้างธีม คลิก **แชร์** แล้ววางลิงก์ธีมที่คัดลอกไว้ในรูปลักษณ์ ตัวนำเข้ายังรับ URL รีจิสทรี `https://tweakcn.com/r/themes/<id>`, URL ตัวแก้ไขเช่น `https://tweakcn.com/editor/theme?theme=amethyst-haze`, พาธสัมพัทธ์ `/themes/<id>`, ID ธีมดิบ และชื่อธีมเริ่มต้น เช่น `amethyst-haze`

ธีมที่นำเข้าจะถูกเก็บไว้เฉพาะในโปรไฟล์เบราว์เซอร์ปัจจุบันเท่านั้น จะไม่ถูกเขียนลงในคอนฟิก Gateway และไม่ซิงก์ข้ามอุปกรณ์ การแทนที่ธีมที่นำเข้าจะอัปเดตสล็อตภายในเครื่องเพียงสล็อตเดียว การล้างธีมจะเปลี่ยนธีมที่ใช้งานกลับเป็น Claw หากธีมที่นำเข้าถูกเลือกอยู่

    - การเรนเดอร์สคีมา + ฟอร์ม (`config.schema` / `config.schema.lookup` รวมถึงฟิลด์ `title` / `description`, คำแนะนำ UI ที่ตรงกัน, สรุปลูกโดยตรง, เมทาดาทาเอกสารบนโหนดอ็อบเจกต์ซ้อน/ไวลด์การ์ด/อาร์เรย์/คอมโพสชัน รวมถึงสคีมา Plugin + ช่องทางเมื่อพร้อมใช้งาน); ตัวแก้ไข JSON ดิบพร้อมใช้งานเฉพาะเมื่อสแนปชอตมีการไปกลับแบบดิบที่ปลอดภัย
    - หากสแนปชอตไม่สามารถไปกลับข้อความดิบได้อย่างปลอดภัย UI ควบคุมจะบังคับใช้โหมดฟอร์มและปิดใช้งานโหมดดิบสำหรับสแนปชอตนั้น
    - ตัวแก้ไข JSON ดิบ "รีเซ็ตเป็นค่าที่บันทึกไว้" จะรักษารูปทรงที่เขียนแบบดิบไว้ (การจัดรูปแบบ ความคิดเห็น เลย์เอาต์ `$include`) แทนการเรนเดอร์สแนปชอตแบบแบนใหม่ ดังนั้นการแก้ไขภายนอกจะคงอยู่หลังรีเซ็ตเมื่อสแนปชอตสามารถไปกลับได้อย่างปลอดภัย
    - ค่าอ็อบเจกต์ SecretRef แบบมีโครงสร้างจะถูกเรนเดอร์เป็นอ่านอย่างเดียวในช่องป้อนข้อความของฟอร์มเพื่อป้องกันความเสียหายจากการแปลงอ็อบเจกต์เป็นสตริงโดยไม่ตั้งใจ

  </Accordion>
  <Accordion title="ดีบัก บันทึก อัปเดต">
    - ดีบัก: สแนปชอตสถานะ/สุขภาพ/โมเดล + บันทึกเหตุการณ์ + การเรียก RPC ด้วยตนเอง (`status`, `health`, `models.list`)
    - บันทึกเหตุการณ์มีเวลารีเฟรช/RPC ของ UI ควบคุม รวมถึงรายการการตอบสนองของเบราว์เซอร์สำหรับเฟรมแอนิเมชันยาวหรืองานยาวเมื่อเบราว์เซอร์เปิดเผยประเภทเอนทรี PerformanceObserver เหล่านั้น
    - บันทึก: tail สดของบันทึกไฟล์ Gateway พร้อมตัวกรอง/ส่งออก (`logs.tail`)
    - อัปเดต: เรียกใช้การอัปเดตแพ็กเกจ/git + รีสตาร์ท (`update.run`) พร้อมรายงานการรีสตาร์ท จากนั้น poll `update.status` หลังเชื่อมต่อใหม่เพื่อตรวจสอบเวอร์ชัน Gateway ที่กำลังทำงานอยู่

  </Accordion>
  <Accordion title="หมายเหตุแผงงาน Cron">
    - สำหรับงานที่แยกออกมา ค่าเริ่มต้นของการส่งมอบคือประกาศสรุป คุณสามารถเปลี่ยนเป็นไม่มีได้หากต้องการให้เรียกใช้เฉพาะภายใน
    - ฟิลด์ช่องทาง/เป้าหมายจะปรากฏเมื่อเลือกประกาศ
    - โหมด Webhook ใช้ `delivery.mode = "webhook"` โดยตั้งค่า `delivery.to` เป็น URL Webhook HTTP(S) ที่ถูกต้อง
    - สำหรับงานเซสชันหลัก โหมดการส่งมอบแบบ Webhook และไม่มีพร้อมใช้งาน
    - ตัวควบคุมการแก้ไขขั้นสูงประกอบด้วยลบหลังเรียกใช้ ล้างการแทนที่เอเจนต์ ตัวเลือก Cron แบบตรงเวลา/เหลื่อมเวลา การแทนที่โมเดล/การคิดของเอเจนต์ และสวิตช์การส่งมอบแบบพยายามให้ดีที่สุด
    - การตรวจสอบฟอร์มอยู่แบบอินไลน์พร้อมข้อผิดพลาดระดับฟิลด์; ค่าที่ไม่ถูกต้องจะปิดใช้งานปุ่มบันทึกจนกว่าจะแก้ไข
    - ตั้งค่า `cron.webhookToken` เพื่อส่งโทเค็น bearer เฉพาะ หากละไว้ Webhook จะถูกส่งโดยไม่มีส่วนหัว auth
    - ทางเลือกสำรองที่เลิกใช้แล้ว: งานเดิมที่จัดเก็บไว้พร้อม `notify: true` ยังสามารถใช้ `cron.webhook` ได้จนกว่าจะย้ายข้อมูล

  </Accordion>
</AccordionGroup>

## พฤติกรรมการแชต

<AccordionGroup>
  <Accordion title="ความหมายเชิงส่งและประวัติ">
    - `chat.send` เป็นแบบ **ไม่บล็อก**: ตอบรับทันทีด้วย `{ runId, status: "started" }` และสตรีมการตอบกลับผ่านเหตุการณ์ `chat`
    - การอัปโหลดในแชตรับรูปภาพรวมถึงไฟล์ที่ไม่ใช่วิดีโอ รูปภาพจะเก็บพาธรูปภาพดั้งเดิมไว้ ส่วนไฟล์อื่นจะถูกจัดเก็บเป็นสื่อที่มีการจัดการและแสดงในประวัติเป็นลิงก์ไฟล์แนบ
    - การส่งซ้ำด้วย `idempotencyKey` เดิมจะคืน `{ status: "in_flight" }` ระหว่างทำงาน และ `{ status: "ok" }` หลังเสร็จสิ้น
    - การตอบกลับของ `chat.history` ถูกจำกัดขนาดเพื่อความปลอดภัยของ UI เมื่อรายการทรานสคริปต์มีขนาดใหญ่เกินไป Gateway อาจตัดทอนช่องข้อความยาว ละเว้นบล็อกเมทาดาทาที่หนัก และแทนที่ข้อความขนาดใหญ่เกินด้วยตัวยึดตำแหน่ง (`[chat.history omitted: message too large]`)
    - รูปภาพจากผู้ช่วย/ที่สร้างขึ้นจะถูกคงอยู่เป็นการอ้างอิงสื่อที่มีการจัดการ และส่งกลับผ่าน URL สื่อของ Gateway ที่มีการยืนยันตัวตน ดังนั้นการโหลดซ้ำจึงไม่ขึ้นกับเพย์โหลดรูปภาพ base64 ดิบที่ยังคงอยู่ในการตอบกลับประวัติแชต
    - เมื่อเรนเดอร์ `chat.history` Control UI จะตัดแท็กคำสั่งแบบอินไลน์ที่ใช้เพื่อการแสดงผลเท่านั้นออกจากข้อความผู้ช่วยที่มองเห็นได้ (เช่น `[[reply_to_*]]` และ `[[audio_as_voice]]`), เพย์โหลด XML ของการเรียกเครื่องมือแบบข้อความล้วน (รวมถึง `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อกการเรียกเครื่องมือที่ถูกตัดทอน), รวมถึงโทเค็นควบคุมโมเดล ASCII/แบบเต็มความกว้างที่หลุดออกมา และละเว้นรายการผู้ช่วยที่ข้อความที่มองเห็นได้ทั้งหมดเป็นเพียงโทเค็นเงียบตรงตัว `NO_REPLY` / `no_reply` หรือโทเค็นตอบรับ Heartbeat `HEARTBEAT_OK`
    - ระหว่างการส่งที่กำลังทำงานและการรีเฟรชประวัติครั้งสุดท้าย มุมมองแชตจะคงข้อความผู้ใช้/ผู้ช่วยแบบมองโลกในแง่ดีในเครื่องให้มองเห็นอยู่ หาก `chat.history` คืนสแนปช็อตเก่ากว่าชั่วครู่ ทรานสคริปต์มาตรฐานจะแทนที่ข้อความในเครื่องเหล่านั้นเมื่อประวัติ Gateway ตามทัน
    - เหตุการณ์ `chat` แบบสดคือสถานะการส่งมอบ ขณะที่ `chat.history` ถูกสร้างใหม่จากทรานสคริปต์เซสชันที่คงทน หลังเหตุการณ์สุดท้ายของเครื่องมือ Control UI จะโหลดประวัติใหม่และผสานเฉพาะส่วนท้ายแบบมองโลกในแง่ดีขนาดเล็ก ขอบเขตทรานสคริปต์มีเอกสารใน [WebChat](/th/web/webchat)
    - `chat.inject` ต่อท้ายบันทึกของผู้ช่วยเข้ากับทรานสคริปต์เซสชันและกระจายเหตุการณ์ `chat` สำหรับการอัปเดตเฉพาะ UI (ไม่มีการรันเอเจนต์ ไม่มีการส่งมอบผ่านช่องทาง)
    - ส่วนหัวแชตแสดงตัวกรองเอเจนต์ก่อนตัวเลือกเซสชัน และตัวเลือกเซสชันถูกจำกัดขอบเขตตามเอเจนต์ที่เลือก การสลับเอเจนต์จะแสดงเฉพาะเซสชันที่ผูกกับเอเจนต์นั้น และถอยกลับไปยังเซสชันหลักของเอเจนต์นั้นเมื่อยังไม่มีเซสชันแดชบอร์ดที่บันทึกไว้
    - บนความกว้างเดสก์ท็อป ตัวควบคุมแชตจะอยู่ในแถวกะทัดรัดเดียวและยุบลงเมื่อเลื่อนลงในทรานสคริปต์ การเลื่อนขึ้น กลับไปด้านบน หรือไปถึงด้านล่างจะคืนค่าตัวควบคุม
    - ข้อความแบบข้อความล้วนที่ซ้ำกันต่อเนื่องจะแสดงเป็นบับเบิลเดียวพร้อมป้ายจำนวน ข้อความที่มีรูปภาพ ไฟล์แนบ เอาต์พุตเครื่องมือ หรือพรีวิวแคนวาสจะไม่ถูกยุบ
    - ตัวเลือกโมเดลและการคิดในส่วนหัวแชตจะแพตช์เซสชันที่ใช้งานอยู่ทันทีผ่าน `sessions.patch`; สิ่งเหล่านี้เป็นการแทนที่เซสชันแบบถาวร ไม่ใช่ตัวเลือกส่งเฉพาะเทิร์นเดียว
    - การพิมพ์ `/new` ใน Control UI จะสร้างและสลับไปยังเซสชันแดชบอร์ดใหม่เดียวกันกับ New Chat การพิมพ์ `/reset` จะคงการรีเซ็ตในที่ที่ชัดเจนของ Gateway สำหรับเซสชันปัจจุบันไว้
    - ตัวเลือกโมเดลแชตร้องขอมุมมองโมเดลที่กำหนดค่าของ Gateway หากมี `agents.defaults.models` รายการอนุญาตนั้นจะขับเคลื่อนตัวเลือก มิฉะนั้นตัวเลือกจะแสดงรายการ `models.providers.*.models` ที่ระบุชัดเจน รวมถึงผู้ให้บริการที่มีการยืนยันตัวตนที่ใช้ได้ แค็ตตาล็อกเต็มยังคงพร้อมใช้งานผ่าน RPC ดีบัก `models.list` พร้อม `view: "all"`
    - เมื่อรายงานการใช้งานเซสชัน Gateway สดแสดงแรงกดดันบริบทสูง พื้นที่ตัวเขียนแชตจะแสดงประกาศบริบท และในระดับ Compaction ที่แนะนำ จะแสดงปุ่มกะทัดรัดที่รันเส้นทาง Compaction เซสชันปกติ สแนปช็อตโทเค็นที่เก่าจะถูกซ่อนจนกว่า Gateway จะรายงานการใช้งานสดอีกครั้ง

  </Accordion>
  <Accordion title="โหมดพูด (เรียลไทม์บนเบราว์เซอร์)">
    โหมดพูดใช้ผู้ให้บริการเสียงเรียลไทม์ที่ลงทะเบียนไว้ กำหนดค่า OpenAI ด้วย `talk.provider: "openai"` พร้อม `talk.providers.openai.apiKey` หรือกำหนดค่า Google ด้วย `talk.provider: "google"` พร้อม `talk.providers.google.apiKey`; การกำหนดค่าผู้ให้บริการเรียลไทม์ Voice Call ยังคงนำกลับมาใช้เป็นตัวสำรองได้ เบราว์เซอร์จะไม่ได้รับคีย์ API ของผู้ให้บริการมาตรฐาน OpenAI ได้รับไคลเอนต์ซีเคร็ต Realtime ชั่วคราวสำหรับ WebRTC Google Live ได้รับโทเค็นยืนยันตัวตน Live API แบบจำกัดที่ใช้ครั้งเดียวสำหรับเซสชัน WebSocket ของเบราว์เซอร์ โดยมีคำสั่งและการประกาศเครื่องมือถูกล็อกไว้ในโทเค็นโดย Gateway ผู้ให้บริการที่เปิดเผยเฉพาะบริดจ์เรียลไทม์ฝั่งแบ็กเอนด์จะทำงานผ่านทรานสปอร์ตรีเลย์ของ Gateway ดังนั้นข้อมูลประจำตัวและซ็อกเก็ตของผู้ขายจะอยู่ฝั่งเซิร์ฟเวอร์ ขณะที่เสียงจากเบราว์เซอร์เคลื่อนผ่าน RPC ของ Gateway ที่ยืนยันตัวตนแล้ว พรอมป์เซสชัน Realtime ถูกประกอบโดย Gateway; `talk.realtime.session` ไม่รับการแทนที่คำสั่งที่ผู้เรียกส่งมา

    ในตัวเขียน Chat ตัวควบคุม Talk คือปุ่มคลื่นถัดจากปุ่มถอดเสียงจากไมโครโฟน เมื่อ Talk เริ่ม แถวสถานะตัวเขียนจะแสดง `Connecting Talk...` จากนั้นแสดง `Talk live` ขณะเสียงเชื่อมต่ออยู่ หรือ `Asking OpenClaw...` ขณะการเรียกเครื่องมือแบบเรียลไทม์กำลังปรึกษาโมเดลที่ใหญ่กว่าที่กำหนดค่าไว้ผ่าน `chat.send`

    การ smoke แบบสดสำหรับผู้ดูแล: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` ตรวจสอบการแลกเปลี่ยน SDP ของ OpenAI browser WebRTC, การตั้งค่า WebSocket เบราว์เซอร์แบบโทเค็นจำกัดของ Google Live และอะแดปเตอร์เบราว์เซอร์รีเลย์ของ Gateway พร้อมสื่อไมโครโฟนปลอม คำสั่งนี้พิมพ์เฉพาะสถานะผู้ให้บริการและไม่บันทึกซีเคร็ต

  </Accordion>
  <Accordion title="หยุดและยกเลิก">
    - คลิก **Stop** (เรียก `chat.abort`)
    - ขณะรันกำลังทำงาน การติดตามผลปกติจะเข้าคิว คลิก **Steer** บนข้อความที่เข้าคิวเพื่อแทรกการติดตามผลนั้นเข้าไปในเทิร์นที่กำลังทำงาน
    - พิมพ์ `/stop` (หรือวลี abort แบบแยกเดี่ยว เช่น `stop`, `stop action`, `stop run`, `stop openclaw`, `please stop`) เพื่อ abort นอกแบนด์
    - `chat.abort` รองรับ `{ sessionKey }` (ไม่มี `runId`) เพื่อ abort รันที่ใช้งานอยู่ทั้งหมดสำหรับเซสชันนั้น

  </Accordion>
  <Accordion title="การเก็บบางส่วนเมื่อ abort">
    - เมื่อรันถูก abort ข้อความผู้ช่วยบางส่วนยังสามารถแสดงใน UI ได้
    - Gateway คงข้อความผู้ช่วยบางส่วนที่ถูก abort ไว้ในประวัติทรานสคริปต์เมื่อมีเอาต์พุตที่บัฟเฟอร์ไว้
    - รายการที่คงอยู่มีเมทาดาทา abort เพื่อให้ผู้บริโภคทรานสคริปต์แยกบางส่วนจาก abort ออกจากเอาต์พุตการเสร็จสิ้นปกติได้

  </Accordion>
</AccordionGroup>

## การติดตั้ง PWA และ web push

Control UI มาพร้อม `manifest.webmanifest` และ service worker ดังนั้นเบราว์เซอร์สมัยใหม่จึงติดตั้งเป็น PWA แบบสแตนด์อโลนได้ Web Push ให้ Gateway ปลุก PWA ที่ติดตั้งแล้วด้วยการแจ้งเตือนได้ แม้แท็บหรือหน้าต่างเบราว์เซอร์จะไม่ได้เปิดอยู่

| พื้นผิว                                               | สิ่งที่ทำ                                                       |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | manifest ของ PWA เบราว์เซอร์จะเสนอ "Install app" เมื่อเข้าถึงได้   |
| `ui/public/sw.js`                                     | service worker ที่จัดการเหตุการณ์ `push` และการคลิกการแจ้งเตือน |
| `push/vapid-keys.json` (ใต้ไดเรกทอรีสถานะ OpenClaw) | คู่คีย์ VAPID ที่สร้างอัตโนมัติ ใช้ลงนามเพย์โหลด Web Push       |
| `push/web-push-subscriptions.json`                    | endpoint การสมัครสมาชิกเบราว์เซอร์ที่คงอยู่                          |

แทนที่คู่คีย์ VAPID ผ่านตัวแปร env บนโปรเซส Gateway เมื่อคุณต้องการตรึงคีย์ (สำหรับการปรับใช้หลายโฮสต์ การหมุนเวียนซีเคร็ต หรือการทดสอบ):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (ค่าเริ่มต้นเป็น `mailto:openclaw@localhost`)

Control UI ใช้เมธอด Gateway ที่ถูกจำกัดด้วย scope เหล่านี้เพื่อลงทะเบียนและทดสอบการสมัครสมาชิกเบราว์เซอร์:

- `push.web.vapidPublicKey` — ดึงคีย์สาธารณะ VAPID ที่ใช้งานอยู่
- `push.web.subscribe` — ลงทะเบียน `endpoint` พร้อม `keys.p256dh`/`keys.auth`
- `push.web.unsubscribe` — ลบ endpoint ที่ลงทะเบียนไว้
- `push.web.test` — ส่งการแจ้งเตือนทดสอบไปยังการสมัครสมาชิกของผู้เรียก

<Note>
Web Push เป็นอิสระจากเส้นทางรีเลย์ APNS ของ iOS (ดู [การกำหนดค่า](/th/gateway/configuration) สำหรับ push ที่หนุนด้วยรีเลย์) และเมธอด `push.test` ที่มีอยู่ ซึ่งมีเป้าหมายเป็นการจับคู่อุปกรณ์มือถือแบบเนทีฟ
</Note>

## เอ็มเบดที่โฮสต์ไว้

ข้อความผู้ช่วยสามารถเรนเดอร์เนื้อหาเว็บที่โฮสต์ไว้แบบอินไลน์ด้วย shortcode `[embed ...]` นโยบาย sandbox ของ iframe ถูกควบคุมโดย `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    ปิดการทำงานของสคริปต์ภายในเอ็มเบดที่โฮสต์ไว้
  </Tab>
  <Tab title="scripts (default)">
    อนุญาตเอ็มเบดแบบโต้ตอบได้โดยยังคงแยก origin ไว้ นี่คือค่าเริ่มต้นและมักเพียงพอสำหรับเกม/วิดเจ็ตเบราว์เซอร์แบบในตัวเอง
  </Tab>
  <Tab title="trusted">
    เพิ่ม `allow-same-origin` บน `allow-scripts` สำหรับเอกสารไซต์เดียวกันที่ตั้งใจต้องการสิทธิ์ที่สูงกว่า
  </Tab>
</Tabs>

ตัวอย่าง:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
ใช้ `trusted` เฉพาะเมื่อเอกสารที่เอ็มเบดต้องการพฤติกรรม same-origin จริง ๆ สำหรับเกมและแคนวาสแบบโต้ตอบที่เอเจนต์สร้างขึ้นส่วนใหญ่ `scripts` เป็นตัวเลือกที่ปลอดภัยกว่า
</Warning>

URL เอ็มเบด `http(s)` ภายนอกแบบสัมบูรณ์ยังคงถูกบล็อกตามค่าเริ่มต้น หากคุณตั้งใจให้ `[embed url="https://..."]` โหลดหน้าบุคคลที่สาม ให้ตั้งค่า `gateway.controlUi.allowExternalEmbedUrls: true`

## ความกว้างข้อความแชต

ข้อความแชตที่จัดกลุ่มใช้ค่าเริ่มต้น max-width ที่อ่านง่าย การปรับใช้บนจอกว้างสามารถแทนที่ได้โดยไม่ต้องแพตช์ CSS ที่บันเดิลมา ด้วยการตั้งค่า `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

ค่าจะถูกตรวจสอบก่อนถึงเบราว์เซอร์ ค่าที่รองรับรวมถึงความยาวและเปอร์เซ็นต์แบบธรรมดา เช่น `960px` หรือ `82%` รวมถึงนิพจน์ความกว้าง `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` และ `fit-content(...)` ที่มีข้อจำกัด

## การเข้าถึง Tailnet (แนะนำ)

<Tabs>
  <Tab title="Integrated Tailscale Serve (preferred)">
    คง Gateway ไว้บน local loopback แล้วให้ Tailscale Serve พร็อกซีด้วย HTTPS:

    ```bash
    openclaw gateway --tailscale serve
    ```

    เปิด:

    - `https://<magicdns>/` (หรือ `gateway.controlUi.basePath` ที่คุณกำหนดค่าไว้)

    ตามค่าเริ่มต้น คำขอ Control UI/WebSocket Serve สามารถยืนยันตัวตนผ่านเฮดเดอร์ตัวตน Tailscale (`tailscale-user-login`) เมื่อ `gateway.auth.allowTailscale` เป็น `true` OpenClaw ตรวจสอบตัวตนโดย resolve ที่อยู่ `x-forwarded-for` ด้วย `tailscale whois` และจับคู่กับเฮดเดอร์ และยอมรับเฉพาะเมื่อคำขอเข้ามาที่ local loopback พร้อมเฮดเดอร์ `x-forwarded-*` ของ Tailscale สำหรับเซสชันผู้ปฏิบัติการ Control UI ที่มีตัวตนอุปกรณ์เบราว์เซอร์ เส้นทาง Serve ที่ตรวจสอบแล้วนี้จะข้ามรอบการจับคู่อุปกรณ์ด้วย เบราว์เซอร์ที่ไม่มีอุปกรณ์และการเชื่อมต่อบทบาทโหนดยังคงทำตามการตรวจสอบอุปกรณ์ปกติ ตั้งค่า `gateway.auth.allowTailscale: false` หากคุณต้องการบังคับใช้ข้อมูลประจำตัว shared-secret อย่างชัดเจนแม้สำหรับทราฟฟิก Serve จากนั้นใช้ `gateway.auth.mode: "token"` หรือ `"password"`

    สำหรับเส้นทางตัวตน Serve แบบ async นั้น ความพยายามยืนยันตัวตนที่ล้มเหลวสำหรับ IP ไคลเอนต์และ scope การยืนยันตัวตนเดียวกันจะถูก serialize ก่อนเขียน rate-limit ดังนั้นการลองซ้ำที่ผิดพร้อมกันจากเบราว์เซอร์เดียวกันอาจแสดง `retry later` ในคำขอที่สอง แทนที่จะเกิด mismatch ธรรมดาสองรายการที่แข่งกันแบบขนาน

    <Warning>
    การยืนยันตัวตน Serve แบบไม่มีโทเค็นถือว่าโฮสต์ gateway เชื่อถือได้ หากโค้ดเครื่อง local ที่ไม่น่าเชื่อถืออาจรันบนโฮสต์นั้น ให้บังคับใช้การยืนยันตัวตนแบบ token/password
    </Warning>

  </Tab>
  <Tab title="Bind to tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    จากนั้นเปิด:

    - `http://<tailscale-ip>:18789/` (หรือ `gateway.controlUi.basePath` ที่คุณกำหนดค่าไว้)

    วาง shared secret ที่ตรงกันลงในการตั้งค่า UI (ส่งเป็น `connect.params.auth.token` หรือ `connect.params.auth.password`)

  </Tab>

## HTTP ที่ไม่ปลอดภัย

หากคุณเปิดแดชบอร์ดผ่าน HTTP แบบไม่เข้ารหัส (`http://<lan-ip>` หรือ `http://<tailscale-ip>`) เบราว์เซอร์จะทำงานใน **บริบทที่ไม่ปลอดภัย** และบล็อก WebCrypto ตามค่าเริ่มต้น OpenClaw จะ **บล็อก** การเชื่อมต่อ UI ควบคุมที่ไม่มีข้อมูลประจำตัวของอุปกรณ์

ข้อยกเว้นที่บันทึกไว้:

- ความเข้ากันได้กับ HTTP ที่ไม่ปลอดภัยเฉพาะ localhost ด้วย `gateway.controlUi.allowInsecureAuth=true`
- การยืนยันตัวตน UI ควบคุมของผู้ปฏิบัติงานที่สำเร็จผ่าน `gateway.auth.mode: "trusted-proxy"`
- กรณีฉุกเฉิน `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**วิธีแก้ไขที่แนะนำ:** ใช้ HTTPS (Tailscale Serve) หรือเปิด UI ในเครื่อง:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (บนโฮสต์ gateway)

<AccordionGroup>
  <Accordion title="พฤติกรรมของสวิตช์การยืนยันตัวตนที่ไม่ปลอดภัย">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` เป็นสวิตช์ความเข้ากันได้ภายในเครื่องเท่านั้น:

    - อนุญาตให้เซสชัน UI ควบคุมของ localhost ดำเนินต่อได้โดยไม่มีข้อมูลประจำตัวของอุปกรณ์ในบริบท HTTP ที่ไม่ปลอดภัย
    - ไม่ข้ามการตรวจสอบการจับคู่
    - ไม่ผ่อนปรนข้อกำหนดข้อมูลประจำตัวของอุปกรณ์ระยะไกล (ที่ไม่ใช่ localhost)

  </Accordion>
  <Accordion title="สำหรับกรณีฉุกเฉินเท่านั้น">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` ปิดการตรวจสอบข้อมูลประจำตัวของอุปกรณ์สำหรับ UI ควบคุม และเป็นการลดระดับความปลอดภัยอย่างรุนแรง ให้ย้อนกลับโดยเร็วหลังใช้งานฉุกเฉิน
    </Warning>

  </Accordion>
  <Accordion title="หมายเหตุเกี่ยวกับ trusted-proxy">
    - การยืนยันตัวตน trusted-proxy ที่สำเร็จสามารถอนุญาตเซสชัน UI ควบคุมของ **ผู้ปฏิบัติงาน** โดยไม่มีข้อมูลประจำตัวของอุปกรณ์ได้
    - สิ่งนี้ **ไม่** ขยายไปยังเซสชัน UI ควบคุมของบทบาท node
    - reverse proxy แบบ loopback บนโฮสต์เดียวกันยังคงไม่ผ่านการยืนยันตัวตน trusted-proxy; ดู [การยืนยันตัวตนพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth)

  </Accordion>
</AccordionGroup>

## นโยบายความปลอดภัยของเนื้อหา

UI ควบคุมมาพร้อมกับนโยบาย `img-src` ที่เข้มงวด: อนุญาตเฉพาะแอสเซ็ตจาก **ต้นทางเดียวกัน**, URL `data:` และ URL `blob:` ที่สร้างในเครื่องเท่านั้น URL รูปภาพระยะไกลแบบ `http(s)` และแบบอิงโปรโตคอลจะถูกเบราว์เซอร์ปฏิเสธและไม่เกิดการดึงข้อมูลผ่านเครือข่าย

ความหมายในทางปฏิบัติ:

- อวาตาร์และรูปภาพที่ให้บริการภายใต้พาธสัมพัทธ์ (เช่น `/avatars/<id>`) ยังคงแสดงผลได้ รวมถึง route อวาตาร์ที่ต้องยืนยันตัวตนซึ่ง UI ดึงข้อมูลและแปลงเป็น URL `blob:` ในเครื่อง
- URL แบบอินไลน์ `data:image/...` ยังคงแสดงผลได้ (มีประโยชน์สำหรับ payload ภายในโปรโตคอล)
- URL `blob:` ในเครื่องที่สร้างโดย UI ควบคุมยังคงแสดงผลได้
- URL อวาตาร์ระยะไกลที่ส่งออกโดยเมทาดาทาของช่องทางจะถูกตัดออกที่ตัวช่วยอวาตาร์ของ UI ควบคุมและแทนที่ด้วยโลโก้/ป้ายในตัว ดังนั้นช่องทางที่ถูกบุกรุกหรือมุ่งร้ายจึงไม่สามารถบังคับให้เบราว์เซอร์ของผู้ปฏิบัติงานดึงรูปภาพระยะไกลใดๆ ได้

คุณไม่ต้องเปลี่ยนอะไรเพื่อให้ได้พฤติกรรมนี้ โดยจะเปิดใช้อยู่เสมอและกำหนดค่าไม่ได้

## การยืนยันตัวตนของ route อวาตาร์

เมื่อกำหนดค่าการยืนยันตัวตนของ gateway แล้ว endpoint อวาตาร์ของ UI ควบคุมจะต้องใช้ gateway token เดียวกับ API ส่วนที่เหลือ:

- `GET /avatar/<agentId>` ส่งคืนรูปอวาตาร์เฉพาะให้ผู้เรียกที่ยืนยันตัวตนแล้วเท่านั้น `GET /avatar/<agentId>?meta=1` ส่งคืนเมทาดาทาอวาตาร์ภายใต้กฎเดียวกัน
- คำขอที่ไม่ยืนยันตัวตนไปยัง route ใด route หนึ่งจะถูกปฏิเสธ (ตรงกับ route assistant-media ที่อยู่ข้างเคียง) สิ่งนี้ป้องกันไม่ให้ route อวาตาร์รั่วไหลข้อมูลประจำตัวของ agent บนโฮสต์ที่ได้รับการป้องกันอยู่แล้ว
- UI ควบคุมเองจะส่งต่อ gateway token เป็น bearer header เมื่อดึงอวาตาร์ และใช้ URL blob ที่ยืนยันตัวตนแล้วเพื่อให้รูปภาพยังแสดงผลในแดชบอร์ดได้

หากคุณปิดการยืนยันตัวตนของ gateway (ไม่แนะนำบนโฮสต์ที่ใช้ร่วมกัน) route อวาตาร์ก็จะไม่ต้องยืนยันตัวตนเช่นกัน สอดคล้องกับส่วนที่เหลือของ gateway

## การยืนยันตัวตนของ route สื่อผู้ช่วย

เมื่อกำหนดค่าการยืนยันตัวตนของ gateway แล้ว ตัวอย่างสื่อในเครื่องของผู้ช่วยจะใช้ route สองขั้นตอน:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` ต้องใช้การยืนยันตัวตนผู้ปฏิบัติงานของ UI ควบคุมตามปกติ เบราว์เซอร์จะส่ง gateway token เป็น bearer header เมื่อตรวจสอบความพร้อมใช้งาน
- การตอบกลับเมทาดาทาที่สำเร็จจะมี `mediaTicket` อายุสั้นซึ่งจำกัดขอบเขตกับพาธแหล่งที่มานั้นโดยเฉพาะ
- URL รูปภาพ เสียง วิดีโอ และเอกสารที่แสดงผลโดยเบราว์เซอร์จะใช้ `mediaTicket=<ticket>` แทน gateway token หรือรหัสผ่านที่ใช้งานอยู่ ticket จะหมดอายุอย่างรวดเร็วและไม่สามารถอนุญาตแหล่งที่มาอื่นได้

สิ่งนี้ทำให้การแสดงผลสื่อปกติยังเข้ากันได้กับองค์ประกอบสื่อดั้งเดิมของเบราว์เซอร์ โดยไม่ใส่ข้อมูลประจำตัว gateway ที่นำกลับมาใช้ซ้ำได้ไว้ใน URL สื่อที่มองเห็นได้

```bash
pnpm ui:build
```

base แบบ absolute ที่เป็นตัวเลือก (เมื่อคุณต้องการ URL แอสเซ็ตแบบคงที่):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

สำหรับการพัฒนาในเครื่อง (dev server แยกต่างหาก):

```bash
pnpm ui:dev
```

จากนั้นชี้ UI ไปที่ URL Gateway WS ของคุณ (เช่น `ws://127.0.0.1:18789`)

## การดีบัก/ทดสอบ: dev server + Gateway ระยะไกล

UI ควบคุมเป็นไฟล์สแตติก เป้าหมาย WebSocket กำหนดค่าได้และอาจต่างจากต้นทาง HTTP ได้ ซึ่งสะดวกเมื่อคุณต้องการใช้ Vite dev server ในเครื่องแต่ให้ Gateway ทำงานที่อื่น

<Steps>
  <Step title="เริ่ม UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="เปิดด้วย gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    การยืนยันตัวตนครั้งเดียวที่เป็นตัวเลือก (หากจำเป็น):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="หมายเหตุ">
    - `gatewayUrl` จะถูกเก็บไว้ใน localStorage หลังโหลดและถูกนำออกจาก URL
    - หากคุณส่ง endpoint `ws://` หรือ `wss://` แบบเต็มผ่าน `gatewayUrl` ให้เข้ารหัสค่า `gatewayUrl` เป็น URL เพื่อให้เบราว์เซอร์แยกวิเคราะห์ query string ได้ถูกต้อง
    - ควรส่ง `token` ผ่าน fragment ของ URL (`#token=...`) เมื่อเป็นไปได้ Fragment จะไม่ถูกส่งไปยังเซิร์ฟเวอร์ จึงหลีกเลี่ยงการรั่วไหลใน request log และ Referer ได้ พารามิเตอร์ query แบบเดิม `?token=` ยังคงถูกนำเข้าได้หนึ่งครั้งเพื่อความเข้ากันได้ แต่ใช้เป็น fallback เท่านั้น และจะถูกลบออกทันทีหลัง bootstrap
    - `password` จะเก็บไว้ในหน่วยความจำเท่านั้น
    - เมื่อตั้งค่า `gatewayUrl` แล้ว UI จะไม่ fallback ไปใช้ข้อมูลประจำตัวจาก config หรือ environment ให้ระบุ `token` (หรือ `password`) อย่างชัดเจน การขาดข้อมูลประจำตัวที่ระบุชัดเจนถือเป็นข้อผิดพลาด
    - ใช้ `wss://` เมื่อ Gateway อยู่หลัง TLS (Tailscale Serve, พร็อกซี HTTPS ฯลฯ)
    - `gatewayUrl` จะยอมรับเฉพาะในหน้าต่างระดับบนสุดเท่านั้น (ไม่ใช่แบบฝัง) เพื่อป้องกัน clickjacking
    - การปรับใช้ UI ควบคุมที่ไม่ใช่ loopback ต้องตั้งค่า `gateway.controlUi.allowedOrigins` อย่างชัดเจน (origin แบบเต็ม) ซึ่งรวมถึงการตั้งค่า dev ระยะไกลด้วย
    - การเริ่มต้น Gateway อาจ seed origin ในเครื่อง เช่น `http://localhost:<port>` และ `http://127.0.0.1:<port>` จาก bind และ port ของ runtime ที่มีผล แต่ origin ของเบราว์เซอร์ระยะไกลยังต้องมีรายการที่ระบุชัดเจน
    - อย่าใช้ `gateway.controlUi.allowedOrigins: ["*"]` ยกเว้นสำหรับการทดสอบในเครื่องที่ควบคุมอย่างเข้มงวด หมายถึงอนุญาต origin ของเบราว์เซอร์ใดๆ ไม่ใช่ "จับคู่กับโฮสต์ใดก็ตามที่ฉันกำลังใช้"
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` เปิดใช้งานโหมด fallback ของ origin จาก Host header แต่เป็นโหมดความปลอดภัยที่อันตราย

  </Accordion>
</AccordionGroup>

ตัวอย่าง:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
