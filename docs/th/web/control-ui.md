Control UI เป็นแอปหน้าเดียวขนาดเล็กที่สร้างด้วย **Vite + Lit** และให้บริการโดย Gateway:

- ค่าเริ่มต้น: `http://<host>:18789/`
- คำนำหน้าแบบไม่บังคับ: ตั้งค่า `gateway.controlUi.basePath` (เช่น `/openclaw`)

แอปสื่อสาร **โดยตรงกับ Gateway WebSocket** บนพอร์ตเดียวกัน

## เปิดอย่างรวดเร็ว (ภายในเครื่อง)

หาก Gateway ทำงานอยู่บนคอมพิวเตอร์เครื่องเดียวกัน ให้เปิด [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (หรือ [http://localhost:18789/](http://localhost:18789/))

หากโหลดหน้าไม่ได้ ให้เริ่ม Gateway ก่อน: `openclaw gateway`

<Note>
เมื่อผูกกับ LAN บน Windows โดยตรง Windows Firewall หรือ Group Policy ที่องค์กรจัดการอาจยังบล็อก URL ของ LAN ที่ประกาศไว้ แม้ว่า `127.0.0.1` จะทำงานบนโฮสต์ Gateway ก็ตาม ให้เรียกใช้ `openclaw gateway status --deep` บนโฮสต์ Windows ซึ่งจะรายงานพอร์ตที่อาจถูกบล็อก โปรไฟล์ที่ไม่ตรงกัน และกฎไฟร์วอลล์ภายในเครื่องที่นโยบายอาจละเว้น
</Note>

ระบบส่งข้อมูลการยืนยันตัวตนระหว่างการจับมือ WebSocket ผ่าน:

- `connect.params.auth.token`
- `connect.params.auth.password`
- ส่วนหัวข้อมูลประจำตัว Tailscale Serve เมื่อ `gateway.auth.allowTailscale: true`
- ส่วนหัวข้อมูลประจำตัวพร็อกซีที่เชื่อถือได้เมื่อ `gateway.auth.mode: "trusted-proxy"`

แผงการตั้งค่าของแดชบอร์ดจะเก็บโทเค็นไว้สำหรับเซสชันของแท็บเบราว์เซอร์ปัจจุบันและ URL ของ Gateway ที่เลือกไว้ แต่จะไม่บันทึกรหัสผ่าน โดยทั่วไป การเริ่มต้นใช้งานจะสร้างโทเค็น Gateway สำหรับการยืนยันตัวตนด้วยข้อมูลลับร่วมเมื่อเชื่อมต่อครั้งแรก แต่การยืนยันตัวตนด้วยรหัสผ่านก็ใช้งานได้เช่นกันเมื่อ `gateway.auth.mode` เป็น `"password"`

## การจับคู่อุปกรณ์ (การเชื่อมต่อครั้งแรก)

โดยทั่วไป การเชื่อมต่อจากเบราว์เซอร์หรืออุปกรณ์ใหม่ต้องได้รับ **การอนุมัติการจับคู่เพียงครั้งเดียว** ซึ่งแสดงเป็น `disconnected (1008): pairing required`

<Steps>
  <Step title="แสดงรายการคำขอที่รอดำเนินการ">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="อนุมัติด้วย ID คำขอ">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

หากเบราว์เซอร์ลองจับคู่ใหม่โดยเปลี่ยนรายละเอียดการยืนยันตัวตน (บทบาท/ขอบเขต/กุญแจสาธารณะ) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่และระบบจะสร้าง `requestId` ใหม่ ให้เรียกใช้ `openclaw devices list` อีกครั้งก่อนอนุมัติ

การเปลี่ยนเบราว์เซอร์ที่จับคู่แล้วจากสิทธิ์อ่านเป็นสิทธิ์เขียน/ผู้ดูแลระบบจะถือเป็นการยกระดับการอนุมัติ ไม่ใช่การเชื่อมต่อใหม่โดยอัตโนมัติ: OpenClaw จะคงการอนุมัติเดิมไว้ บล็อกการเชื่อมต่อใหม่ที่มีสิทธิ์กว้างกว่า และขอให้อนุมัติชุดขอบเขตใหม่อย่างชัดเจน

เมื่ออนุมัติแล้ว ระบบจะจดจำอุปกรณ์และไม่ต้องอนุมัติซ้ำ เว้นแต่จะเพิกถอนด้วย `openclaw devices revoke --device <id> --role <role>` โปรดดู [CLI สำหรับอุปกรณ์](/th/cli/devices) สำหรับการหมุนเวียนโทเค็น การเพิกถอน และขั้นตอนการอนุมัติเมื่อเรียกใช้ Paperclip / `openclaw_gateway` ครั้งแรก

<Note>
- การเชื่อมต่อเบราว์เซอร์ผ่านลูปแบ็กภายในเครื่องโดยตรง (`127.0.0.1` / `localhost`) จะได้รับการอนุมัติโดยอัตโนมัติ
- Tailscale Serve สามารถข้ามขั้นตอนการจับคู่ไปกลับสำหรับเซสชันผู้ควบคุม Control UI ได้เมื่อ `gateway.auth.allowTailscale: true` ข้อมูลประจำตัว Tailscale ผ่านการตรวจสอบ และเบราว์เซอร์แสดงข้อมูลประจำตัวอุปกรณ์ของตน เบราว์เซอร์ที่ไม่มีข้อมูลอุปกรณ์และการเชื่อมต่อที่มีบทบาทเป็น Node ยังคงผ่านการตรวจสอบอุปกรณ์ตามปกติ
- การผูกกับ Tailnet โดยตรง การเชื่อมต่อเบราว์เซอร์ผ่าน LAN และโปรไฟล์เบราว์เซอร์ที่ไม่มีข้อมูลประจำตัวอุปกรณ์ยังคงต้องได้รับการอนุมัติอย่างชัดเจน
- โปรไฟล์เบราว์เซอร์แต่ละโปรไฟล์จะสร้าง ID อุปกรณ์ที่ไม่ซ้ำกัน ดังนั้นการเปลี่ยนเบราว์เซอร์หรือล้างข้อมูลเบราว์เซอร์จึงต้องจับคู่ใหม่

</Note>

## จับคู่อุปกรณ์เคลื่อนที่

ผู้ดูแลระบบที่จับคู่แล้วสามารถสร้าง QR สำหรับเชื่อมต่อ iOS/Android ได้โดยไม่ต้องเปิดเทอร์มินัล:

<Steps>
  <Step title="เปิดการจับคู่อุปกรณ์เคลื่อนที่">
    เลือก **อุปกรณ์** แล้วคลิก **จับคู่อุปกรณ์เคลื่อนที่** ในการ์ด **อุปกรณ์**
  </Step>
  <Step title="เชื่อมต่อโทรศัพท์">
    ในแอป OpenClaw บนอุปกรณ์เคลื่อนที่ ให้เปิด **การตั้งค่า** → **Gateway** แล้วสแกนรหัส QR หรือจะคัดลอกและวางรหัสตั้งค่าแทนก็ได้
  </Step>
  <Step title="ยืนยันการเชื่อมต่อ">
    แอปอย่างเป็นทางการสำหรับ iOS/Android จะเชื่อมต่อโดยอัตโนมัติ หาก **รอการอนุมัติ** แสดงคำขอ ให้ตรวจสอบบทบาทและขอบเขตก่อนอนุมัติ
  </Step>
</Steps>

การสร้างรหัสตั้งค่าต้องใช้ `operator.admin` ปุ่มนี้จะถูกปิดใช้งานสำหรับเซสชันที่ไม่มีสิทธิ์ดังกล่าว รหัสตั้งค่าประกอบด้วยข้อมูลรับรองเริ่มต้นที่มีอายุสั้น ดังนั้นให้ปฏิบัติต่อ QR และรหัสที่คัดลอกเสมือนเป็นรหัสผ่านตราบใดที่ยังใช้งานได้ สำหรับการจับคู่ระยะไกล Gateway ต้องแก้ชื่อไปยัง `wss://` ได้ (เช่น ผ่าน Tailscale Serve/Funnel) ส่วน `ws://` แบบธรรมดาจะจำกัดเฉพาะที่อยู่ลูปแบ็กและ LAN ส่วนตัว โปรดดูรายละเอียดทั้งหมดเกี่ยวกับความปลอดภัยและการสำรองที่ [การจับคู่](/th/channels/pairing#pair-from-the-control-ui-recommended)

## ข้อมูลประจำตัวส่วนบุคคล (เฉพาะเบราว์เซอร์)

Control UI รองรับข้อมูลประจำตัวส่วนบุคคลแยกตามเบราว์เซอร์ (ชื่อที่แสดงและอวาตาร์) ซึ่งแนบไปกับข้อความขาออกเพื่อระบุแหล่งที่มาในเซสชันที่ใช้ร่วมกัน ข้อมูลนี้อยู่ในพื้นที่จัดเก็บของเบราว์เซอร์ จำกัดขอบเขตไว้ที่โปรไฟล์เบราว์เซอร์ปัจจุบัน และจะไม่ซิงค์ไปยังอุปกรณ์อื่นหรือบันทึกไว้ฝั่งเซิร์ฟเวอร์ นอกเหนือจากเมทาดาทาการเป็นผู้เขียนในบันทึกการสนทนาตามปกติของข้อความที่ส่ง การล้างข้อมูลเว็บไซต์หรือเปลี่ยนเบราว์เซอร์จะรีเซ็ตข้อมูลนี้เป็นค่าว่าง

การแทนที่อวาตาร์ของผู้ช่วยใช้รูปแบบเฉพาะเบราว์เซอร์เดียวกัน: การแทนที่ที่อัปโหลดจะซ้อนทับข้อมูลประจำตัวที่ Gateway แก้ไขแล้วเฉพาะภายในเครื่อง และจะไม่เดินทางไปกลับผ่าน `config.patch` ฟิลด์การกำหนดค่าที่ใช้ร่วมกัน `ui.assistant.avatar` ยังคงพร้อมใช้งานสำหรับไคลเอนต์ที่ไม่ใช่ UI ซึ่งเขียนฟิลด์นี้โดยตรง

## จุดปลายการกำหนดค่ารันไทม์

Control UI จะดึงการตั้งค่ารันไทม์จาก `/control-ui-config.json` ซึ่งแก้ไขโดยอิงกับพาธฐานของ Control UI ของ Gateway (เช่น `/__openclaw__/control-ui-config.json` ภายใต้พาธฐาน `/__openclaw__/`) จุดปลายนี้ถูกควบคุมด้วยการยืนยันตัวตนของ Gateway แบบเดียวกับพื้นผิว HTTP ส่วนอื่น: เบราว์เซอร์ที่ไม่ได้ยืนยันตัวตนจะดึงข้อมูลไม่ได้ และการดึงข้อมูลสำเร็จต้องมีโทเค็น/รหัสผ่าน Gateway ที่ถูกต้อง ข้อมูลประจำตัว Tailscale Serve หรือข้อมูลประจำตัวพร็อกซีที่เชื่อถือได้

## สถานะโฮสต์ Gateway

เปิด **Settings → General** เพื่อดูการ์ด **Gateway Host** ซึ่งแสดงเครื่อง Gateway ที่อยู่ LAN ระบบปฏิบัติการ รันไทม์ ระยะเวลาทำงาน โหลด CPU หน่วยความจำ และพื้นที่ดิสก์ของโวลุ่มสถานะ การ์ดจะรีเฟรชทุก 10 วินาทีขณะที่มองเห็นผ่าน RPC ของ Gateway ที่ชื่อ `system.info` ซึ่งต้องใช้ขอบเขต `operator.read` Gateway รุ่นเก่าและการเชื่อมต่อที่ไม่มีขอบเขตดังกล่าวจะไม่แสดงการ์ดนี้

## การรองรับภาษา

Control UI จะปรับภาษาเองในการโหลดครั้งแรกตามภาษาของเบราว์เซอร์ หากต้องการเปลี่ยนภายหลัง ให้เปิด **Settings -> General -> Language** (ตัวเลือกภาษาอยู่ในหน้า General ไม่ได้อยู่ใต้ Appearance)

- ภาษาที่รองรับ: `en`, `ar`, `de`, `es`, `fa`, `fr`, `hi`, `id`, `it`, `ja-JP`, `ko`, `nl`, `pl`, `pt-BR`, `ru`, `th`, `tr`, `uk`, `vi`, `zh-CN`, `zh-TW`
- คำแปลที่ไม่ใช่ภาษาอังกฤษจะโหลดแบบเลซีในเบราว์เซอร์
- ภาษาที่เลือกจะถูกบันทึกไว้ในพื้นที่จัดเก็บของเบราว์เซอร์และนำกลับมาใช้ในการเข้าชมครั้งถัดไป
- คีย์คำแปลที่ขาดหายจะใช้ภาษาอังกฤษแทน

ระบบสร้างคำแปลเอกสารสำหรับชุดภาษาที่ไม่ใช่ภาษาอังกฤษชุดเดียวกัน แต่ตัวเลือกภาษา Mintlify ในตัวของเว็บไซต์เอกสารจะแสดงเฉพาะรหัสภาษาที่ Mintlify ยอมรับ เอกสารภาษาไทย (`th`) และภาษาเปอร์เซีย (`fa`) ยังคงถูกสร้างในที่เก็บสำหรับเผยแพร่ แต่อาจไม่ปรากฏในตัวเลือกดังกล่าวจนกว่า Mintlify จะรองรับรหัสเหล่านี้

## ธีมรูปลักษณ์

แผงรูปลักษณ์มีธีม Claw, Knot และ Dash ในตัว (ค่าเริ่มต้นคือ Claw) พร้อมช่องนำเข้า tweakcn เฉพาะเบราว์เซอร์หนึ่งช่อง หากต้องการนำเข้าธีม ให้เปิด [ตัวแก้ไข tweakcn](https://tweakcn.com/editor/theme) เลือกหรือสร้างธีม คลิก **Share** แล้ววางลิงก์ที่คัดลอกไว้ในรูปลักษณ์ ตัวนำเข้ายังรองรับ URL รีจิสทรี `https://tweakcn.com/r/themes/<id>`, URL ตัวแก้ไข เช่น `https://tweakcn.com/editor/theme?theme=amethyst-haze`, พาธ `/themes/<id>` แบบสัมพัทธ์, ID ธีมแบบดิบ และชื่อธีมเริ่มต้น เช่น `amethyst-haze`

ธีมที่นำเข้าจะถูกเก็บไว้เฉพาะในโปรไฟล์เบราว์เซอร์ปัจจุบัน โดยจะไม่เขียนลงในการกำหนดค่า Gateway และไม่ซิงค์ข้ามอุปกรณ์ การแทนที่ธีมที่นำเข้าจะอัปเดตช่องภายในเครื่องช่องเดียว การล้างธีมจะสลับกลับเป็น Claw หากธีมที่นำเข้าใช้งานอยู่

รูปลักษณ์ยังมีการตั้งค่าขนาดข้อความ ซึ่งใช้กับข้อความแชต ข้อความในช่องเขียน การ์ดเครื่องมือ และแถบด้านข้างของแชต พร้อมทั้งรักษาขนาดช่องป้อนข้อความไว้อย่างน้อย 16px เพื่อไม่ให้ Safari บนอุปกรณ์เคลื่อนที่ซูมโดยอัตโนมัติเมื่อโฟกัส

ธีม โหมดธีม ขนาดข้อความ ภาษา และค่ากำหนดการแสดงผลแชตจะซิงค์ผ่านการกำหนดค่า Gateway (`ui.prefs`) จึงติดตามไปยังอุปกรณ์ต่าง ๆ และเอเจนต์สามารถเปลี่ยนค่าเหล่านี้ผ่านด่านอนุมัติได้ โดยไคลเอนต์ที่เชื่อมต่อจะใช้การเปลี่ยนแปลงแบบสดผ่านประกาศ `config.changed` ของ Gateway แต่ละเบราว์เซอร์จะเก็บสำเนาภายในเครื่องเพื่อเริ่มทำงานได้ทันที ไคลเอนต์ที่เขียนการกำหนดค่าไม่ได้ (ขอบเขตผู้ดู ออฟไลน์) จะเก็บการเปลี่ยนแปลงไว้เฉพาะอุปกรณ์ โปรดดู [ข้อมูลอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference#ui)

## การดูแลระบบ OpenClaw

เปิด **OpenClaw** ในแถบด้านข้างเพื่อพูดคุยกับเอเจนต์ตั้งค่าและซ่อมแซมระบบ นอกขั้นตอนเริ่มต้นใช้งาน หน้านี้สามารถแสดงชิปเหตุการณ์ที่ปิดได้สูงสุดหนึ่งรายการต่อการเข้าชม โดยจะไม่แสดงสิ่งใดสำหรับทราฟฟิก Gateway ตามปกติ และตอบสนองเฉพาะสแนปช็อตสถานะที่รายงานว่าตัวโหลดการกำหนดค่าใหม่ถูกปิดใช้งาน ช่องทางที่กำหนดค่าไว้ขาดการเชื่อมต่อ/ประสิทธิภาพลดลง การตรวจสอบช่องทางล้มเหลว หรือข้อมูลรับรองช่องทางไม่พร้อมใช้งาน เหตุการณ์ใหม่กว่าจะแทนที่ชิปที่รอดำเนินการเฉพาะเมื่อมีความรุนแรงมากกว่า การปิดหรือใช้ชิปจะระงับข้อความแจ้งเหตุการณ์ตลอดการเข้าชมนั้น การคลิกชิปจะส่งคำถามวินิจฉัยเป็นข้อความ `openclaw.chat` จริง ดังนั้นบันทึกการสนทนาจะบันทึกคำขอไว้และ OpenClaw จะดำเนินการวินิจฉัย ขั้นตอนเริ่มต้นใช้งานจะไม่แสดงชิปเหตุการณ์เหล่านี้

## จัดการ Plugin

เปิด **Plugin** ในแถบด้านข้าง หรือใช้ `/settings/plugins` โดยอิงกับ
พาธฐาน Control UI ที่กำหนดค่าไว้ เพื่อเรียกดูและจัดการ Plugin โดยไม่ต้องออกจาก
Control UI ตัวอย่างเช่น พาธฐาน `/openclaw` จะใช้
`/openclaw/settings/plugins` หน้านี้พร้อมใช้งานเสมอ แม้ว่า
Plugin เสริมทั้งหมดจะถูกปิดใช้งาน

Plugin เป็นศูนย์รวมที่มีสี่แท็บ: **ติดตั้งแล้ว** และ **ค้นพบ** ใช้จัดการโค้ด
Plugin ที่ `/settings/plugins`, **Skills** มีตัวจัดการ Skills แยกตามเอเจนต์ที่
`/skills` และ **เวิร์กช็อป** มีการรีวิวข้อเสนอ Skill Workshop ที่
`/skills/workshop` แต่ละแท็บจะเก็บ URL ของตนเอง และแถบด้านข้างจะแสดง
รายการ Plugin เพียงรายการเดียวสำหรับทั้งหมด

แท็บ **ติดตั้งแล้ว** แสดงรายการภายในเครื่องทั้งหมดโดยจัดกลุ่มตามหมวดหมู่ พร้อม
จำนวนภาพรวม แต่ละแถวจะเปิดมุมมองรายละเอียด เมนูรายการเพิ่มเติม (`…`) ของแถว
ใช้เปิดหรือปิด Plugin และมี **นำออก** สำหรับ Plugin ที่ติดตั้งจากภายนอก
นอกจากนี้ยังแสดงรายการ [เซิร์ฟเวอร์ MCP](/th/cli/mcp) ที่กำหนดค่าไว้ และรองรับการเพิ่ม การปิดใช้งาน
และการนำออกภายในหน้าเดียวกัน ส่วนควบคุมเซิร์ฟเวอร์ชุดเดียวกันอยู่ที่ **Settings → MCP**
แท็บ **ค้นพบ** คือร้านค้า ซึ่งมี Plugin แนะนำที่มาพร้อมกับ OpenClaw,
Plugin ภายนอกอย่างเป็นทางการ และตัวเชื่อมต่อ MCP แบบคลิกเดียวสำหรับบริการยอดนิยม
การพิมพ์ในช่องค้นหาจะค้นหา
[ClawHub](https://clawhub.ai/plugins) ภายในหน้า และเพิ่มส่วน **จาก ClawHub**
พร้อมจำนวนการดาวน์โหลดและป้ายยืนยันแหล่งที่มา Deep Link สามารถ
เปิดร้านค้าโดยตรงด้วย `/settings/plugins?tab=discover`

แท็บ **Skills** เก็บรายงานสถานะ Skills สวิตช์เปิด/ปิด การป้อน
คีย์ API และการค้นหา Skills ใน ClawHub ภายในหน้า โดยจำกัดขอบเขตตามเอเจนต์ที่เลือก แท็บ
**เวิร์กช็อป** เก็บบอร์ด Skill Workshop และขั้นตอนรีวิวประจำวันนี้สำหรับ
[ข้อเสนอ Skills](/th/tools/skill-workshop) **ค้นหาแนวคิด Skills** จะรีวิวชุด
เซสชันสำคัญที่มีขอบเขตจำกัดจากใหม่ที่สุดไปเก่าที่สุด และเก็บผลลัพธ์ไว้เป็น
ข้อเสนอที่รอดำเนินการ แผงจะแสดงความครอบคลุมสะสม **สแกนงานก่อนหน้า**
จะดำเนินการต่อจากเคอร์เซอร์ที่บันทึกไว้ แล้วเปลี่ยนเป็น **สแกนงานใหม่** หลังจากประวัติ
เก่าหมดแล้ว การรีวิวประวัติด้วยตนเองทำงานได้ขณะที่การเรียนรู้ด้วยตนเองแบบอัตโนมัติ
ถูกปิดใช้งาน และใช้โมเดลที่กำหนดค่าไว้ของเอเจนต์ที่เลือก

Plugin ที่รวมมาให้มีอยู่ใน Gateway แล้ว และจะแสดง **เปิดใช้งาน** หรือ
**ปิดใช้งาน** แทน **ติดตั้ง** ตัวอย่างเช่น Workboard รวมมากับ
OpenClaw แต่ถูกปิดใช้งานโดยค่าเริ่มต้น ดังนั้นการดำเนินการคือ **เปิดใช้งาน** Plugin ที่รวมมา
ไม่สามารถนำออกได้ ทำได้เพียงปิดใช้งานเท่านั้น

การอ่านแค็ตตาล็อกและการค้นหา ClawHub ต้องใช้ `operator.read` ส่วนการติดตั้ง
เปิดใช้งาน ปิดใช้งาน หรือลบ Plugin และการเปลี่ยนเซิร์ฟเวอร์ MCP ต้องใช้
`operator.admin`; การดำเนินการเหล่านั้นจะยังคงถูกปิดใช้งานสำหรับผู้ปฏิบัติงานที่มีสิทธิ์อ่านอย่างเดียว

การติดตั้งจาก ClawHub ดำเนินการผ่าน Gateway และใช้การตรวจสอบนโยบายด้านความน่าเชื่อถือ
ความสมบูรณ์ และการติดตั้ง Plugin แบบเดียวกับการติดตั้งอื่นที่ดำเนินการผ่าน Gateway การติดตั้ง
หรือลบโค้ด Plugin ต้องรีสตาร์ต Gateway การเปิดหรือปิดใช้งาน
Plugin ที่ติดตั้งแล้วสามารถมีผลได้โดยไม่ต้องรีสตาร์ตเมื่อ Plugin และรันไทม์
Gateway ปัจจุบันรองรับ มิฉะนั้น UI จะแจ้งว่าต้อง
รีสตาร์ต ตัวเชื่อมต่อ MCP ที่ใช้ OAuth ต้องดำเนินการ
`openclaw mcp login <name>` หนึ่งครั้งจาก CLI หลังจากเพิ่มแล้ว

หน้านี้มุ่งเน้นที่รายการ การค้นหา การติดตั้ง การเปิดใช้งาน
และการลบโดยเจตนา ใช้ [`openclaw plugins`](/th/cli/plugins) สำหรับแหล่งที่มาจาก npm, git หรือ
พาธภายในเครื่องแบบกำหนดเอง การอัปเดต และการกำหนดค่า Plugin ขั้นสูง

## การนำทางในแถบด้านข้าง

แถบด้านข้างจัดระเบียบทุกอย่างโดยมีเอเจนต์เป็นศูนย์กลาง แถวข้อมูลประจำตัวด้านบนคือเอเจนต์ที่ใช้งานอยู่ ส่วน **Pages** ด้านล่างเริ่มต้นด้วย **Home** ซึ่งเป็นเซสชันหลักแบบต่อเนื่องของเอเจนต์ พร้อมป้ายแสดงสถานะยังไม่ได้อ่านหรือกำลังทำงาน ตามด้วยปลายทางที่ปักหมุดไว้ (ค่าเริ่มต้นคือ **Usage**, **Automations** และ **Plugins**) ตัวควบคุมปรับแต่งบนส่วนหัว Pages จะเปิดเมนูที่มีปลายทางอื่นทั้งหมด รวมถึงแท็บที่ Plugin จัดเตรียม พร้อม **Edit pinned items**; การคลิกขวาบริเวณนำทางจะเปิดตัวแก้ไขการปักหมุดโดยตรง รายการเซสชันด้านล่างแบ่งเป็นโซน ได้แก่ **Threads** สำหรับเซสชันแชตของเอเจนต์ (เซสชันหลักอยู่ภายใต้ Home; เซสชันที่เซสชันหลักสร้างขึ้นจะแสดงที่นี่เป็นเธรดระดับบนสุด และเธรดที่ตั้งชื่อแล้วจะแสดงโดยไม่มีคำนำหน้าประเภท), **Groups** สำหรับการสนทนาแบบกลุ่มและในห้อง และ **Coding** สำหรับเซสชันที่ผูกกับเวิร์กทรีที่มีการจัดการหรือโหนด exec (แต่ละแถวแสดงบรรทัด `repo ⎇ branch` พร้อมโฮสต์ของโหนด), เซสชันฮาร์เนสที่ใช้ ACP และแค็ตตาล็อก Codex/Claude CLI Coding จะยุบอยู่เมื่อเรียกใช้ครั้งแรกและจดจำตัวเลือกของคุณ ส่วนหัวขณะยุบจะแสดงจำนวนจริงและตัวบ่งชี้การทำงานขณะที่เซสชันภายในกำลังทำงาน กลุ่มกำหนดเอง (`category` ของเซสชัน) และแถว **Pinned** อยู่เหนือ Threads และการกำหนดเซสชันให้กลุ่มกำหนดเองจะมีผลเหนือการจำแนกโซนอัตโนมัติเสมอ ส่วนหัว Threads มีตัวควบคุมการเรียงลำดับ (Created หรือ Last updated พร้อมสวิตช์ Group by) และปุ่ม **+** ที่เปิดหน้าสร้างเซสชันใหม่ การเปิดเซสชันจะย้ายไฮไลต์การเลือกโดยไม่จัดลำดับแถวใหม่ เซสชันหลักที่มีการเรียกใช้ย่อยล่าสุดจะแสดงตัวเปิดเผยและจำนวนรายการย่อย ให้ขยายเพื่อตรวจสอบเซสชันย่อยที่ซ้อนกัน สถานะแบบสดหรือสิ้นสุด และรันไทม์โดยไม่ต้องออกจากแถบด้านข้าง การเลือกเซสชันย่อยจะเปิดแชตและแสดงเส้นทางบรรพบุรุษโดยอัตโนมัติ แถวรายการย่อยจะไม่รวมอยู่ในการจัดกลุ่มระดับราก การปักหมุด การลาก การเลือกหลายรายการ และการแบ่งหน้า ส่วนโซนที่ยุบจะไม่ใช้โควตาหน้าจอที่มองเห็นได้ เซสชันที่มีกิจกรรมใหม่นับจากครั้งล่าสุดที่อ่านจะแสดงจุดยังไม่ได้อ่าน และการเปิดเซสชันจะทำเครื่องหมายว่าอ่านแล้ว สถานะวงจรชีวิตของเวิร์กเกอร์บนคลาวด์ใช้ป้ายรูปโลก ส่วนเซสชันภายในเครื่องและเซสชันที่เรียกคืนจะไม่แสดงป้ายตำแหน่ง เนื่องจากการดำเนินการภายในเครื่องเป็นค่าเริ่มต้น แต่ละแถวเซสชันระดับรากมีเมนูบริบท (ปุ่มจุดสามจุดหรือคลิกขวา) ซึ่งประกอบด้วย Pin/Unpin, Mark as unread/read, Rename, Fork, Move to group (รวมถึง New group และ Remove from group), Archive และ Delete; เลย์เอาต์แบบสัมผัสจะแสดงตัวควบคุมการปักหมุดและเมนูโดยตรงไว้เสมอ การ Cmd/Ctrl-click จะสลับแถวระดับรากเข้าสู่การเลือกหลายรายการ และ Shift-click จะขยายการเลือกตามลำดับที่มองเห็นได้ จากนั้นการเปิดเมนูบนแถวที่เลือกจะแสดงการดำเนินการแบบกลุ่ม (Mark N as unread/read, Move N to group, Archive N, Delete N) ซึ่งใช้กับทุกเซสชันที่เลือก โดยยืนยันเพียงครั้งเดียวสำหรับการลบแบบกลุ่ม ลากเซสชันระดับรากไปยัง **Pinned** เพื่อปักหมุด หรือลากไปยังกลุ่มกำหนดเองเพื่อย้าย ส่วนหัวกลุ่มกำหนดเองสามารถยุบ ขยาย หรือลากเพื่อจัดลำดับใหม่ได้ ชื่อกลุ่มและลำดับจะอยู่ใน gateway (`sessions.groups.*`) จึงติดตามคุณข้ามเบราว์เซอร์ได้ ส่วนสถานะการยุบจะอยู่ในโปรไฟล์เบราว์เซอร์ ส่วนหัวกลุ่มยังมีเมนู (ปุ่มจุดสามจุดหรือคลิกขวา) ซึ่งประกอบด้วย Rename group, New group และ Delete group; การเปลี่ยนชื่อหรือลบกลุ่มจะอัปเดตเซสชันสมาชิกทุกเซสชันที่ฝั่งเซิร์ฟเวอร์ รวมถึงเซสชันที่เก็บถาวร และการลบกลุ่มจะเก็บเซสชันไว้พร้อมย้ายกลับไปยัง Threads

## หน้าสร้างเซสชันใหม่

ปุ่ม **+** ในส่วนหัวรายการเซสชันของแถบด้านข้างจะเปิดฉบับร่างแบบเต็มหน้าที่ `/new`: จะไม่มีสิ่งใดถูกสร้างจนกว่าจะส่งข้อความแรก แถวเป้าหมายเหนือกล่องข้อความใช้เลือกตำแหน่งที่เซสชันทำงาน ได้แก่ เอเจนต์ (การตั้งค่าแบบหลายเอเจนต์), ตำแหน่งที่ exec ทำงาน (**Gateway · local** หรือโหนดที่จับคู่ซึ่งเปิดเผย `system.run`; ต้องใช้ `operator.admin`), โฟลเดอร์ (ค่าเริ่มต้นคือเวิร์กสเปซของเอเจนต์; พาธ Gateway แบบสัมบูรณ์อื่นต้องใช้ `operator.admin` และเวิร์กทรี) และสวิตช์ **Worktree** แบบไม่บังคับพร้อมตัวเลือกแบรนช์ฐาน (รองรับโดย `worktrees.branches` จึงไม่มีการ fetch) และชื่อเวิร์กทรีแบบไม่บังคับ (แบรนช์จะกลายเป็น `openclaw/<name>`) ส่วนท้ายของตัวเขียนข้อความใช้เลือกโมเดลและระดับการให้เหตุผลของเซสชันใหม่ และการเริ่มทำงานบนคลาวด์จะบันทึกตัวเลือกทั้งสองก่อนส่งเซสชันไปยังเวิร์กเกอร์ ปุ่มเรียกดูของชิปโฟลเดอร์จะเปิดตัวเลือกไดเรกทอรีแบบอินไลน์ซึ่งรองรับโดยเมธอด `fs.listDir` ที่ใช้ได้เฉพาะผู้ดูแลระบบ ระดับบนสุดจะแสดง Gateway และทุกโหนดที่ระบบรู้จัก โหนดออฟไลน์และโหนดที่ไม่รองรับการเรียกดูไดเรกทอรียังคงมองเห็นได้แต่ถูกปิดใช้งาน การเลือก Gateway จะเริ่มจากโฟลเดอร์ปัจจุบันหรือโฮมของ Gateway การเลือกโหนดที่รองรับจะเรียกดูระบบไฟล์ของโฮสต์โหนดนั้น ผูก exec เข้ากับโหนด และใช้พาธโหนดแบบสัมบูรณ์ที่เลือกโดยตรง (เวิร์กทรีที่มีการจัดการยังคงใช้ได้เฉพาะ Gateway) การส่งจะเรียก `sessions.create` พร้อมข้อความแรก เพื่อให้การทำงานเริ่มต้นภายในการรับส่งรอบเดียวกันและ UI เปลี่ยนไปยังแชตของเซสชันใหม่ หาก Gateway สร้างเซสชันแล้วแต่ปฏิเสธการส่งครั้งแรก แชตจะเก็บพรอมต์และข้อผิดพลาดไว้แม้โหลดใหม่; **Retry** จะส่งผ่านเซสชันที่สร้างไว้แล้วแทนการสร้างเซสชันอีกเซสชัน

ภายใน **Settings** แถบด้านข้างเฉพาะจะเริ่มด้วยช่อง **Search settings** สำหรับค้นหาส่วนการตั้งค่าอย่างรวดเร็ว

ช่อง **Search** ที่ด้านบนของแถบด้านข้างจะเปิดพาเล็ตคำสั่ง (⌘K) การคลิกแถวข้อมูลประจำตัวเอเจนต์ที่ด้านบนของแถบด้านข้างจะเปิดเมนูเอเจนต์; **Home** จะเปิดเซสชันหลัก เมื่อมีสิ่งที่ต้องดำเนินการ เช่น งาน Cron ที่ล้มเหลวหรือเกินกำหนด หรือการตรวจสอบสิทธิ์ของโมเดลที่กำลังจะหมดอายุหรือหมดอายุแล้ว ชิปแจ้งเตือนขนาดกะทัดรัดจะปรากฏเหนือส่วนท้ายของแถบด้านข้างและคลิกไปยังหน้าที่รับผิดชอบได้ แถวข้อมูลประจำตัวจะแสดงอวตารของเอเจนต์ (รูปภาพข้อมูลประจำตัวหรืออีโมจิ), ชื่อ, จุดสถานะการเชื่อมต่อ และคำบรรยายรองแบบสด การคลิกจะเปิดเมนูเอเจนต์ ซึ่งประกอบด้วยตัวสลับเอเจนต์ (การตั้งค่าแบบหลายเอเจนต์), "What can this agent do?", **Agent settings**, **Settings**, การจับคู่กับอุปกรณ์เคลื่อนที่, **Docs**, ชิปบิลด์ และสวิตช์โหมดสี รายชื่อที่มีเอเจนต์มากกว่าสิบรายการจะมีช่องตัวกรองและแสดงเอเจนต์ที่ปักหมุดไว้ก่อน ปักหมุดหรือเลิกปักหมุดเอเจนต์จากหน้าการตั้งค่า Agents โดยชุดที่ปักหมุดจะจัดเก็บในโปรไฟล์เบราว์เซอร์ การเลือกเอเจนต์จะกำหนดขอบเขต Chat รวมถึง Usage, Automations, Tasks, Workboard และ Sessions ให้กับเอเจนต์นั้น แต่ละหน้าที่มีขอบเขตจะแสดงตัวควบคุม **Agent** พร้อม **All agents** เป็นทางออก ซึ่งจะขยายขอบเขตหน้าที่ใช้ร่วมกันโดยไม่เปลี่ยนเอเจนต์แชตที่ระบุไว้ ขณะที่ลิงก์เซสชันโดยตรงยังคงเปิดเป้าหมายของตน หน้าการตั้งค่า Agents จะเก็บการเลือก `?agent=` ของตนเองและไม่ติดตามขอบเขตหน้าที่ใช้ร่วมกัน แถบส่วนท้ายประกอบด้วยโลโก้ผลิตภัณฑ์ ชิปบิลด์ จุดสถานะการเชื่อมต่อ gateway และทางลัด Settings เมื่อ gateway ทำงานจากซอร์สเช็กเอาต์บนแบรนช์อื่นที่ไม่ใช่ `main` ส่วนท้ายจะแสดงชื่อแบรนช์นั้นเป็นสีแดงด้วย เพื่อให้สังเกตได้ทันทีว่าเป็น gateway ที่ไม่ใช่รุ่นเผยแพร่ (การติดตั้งรุ่นเผยแพร่จะไม่แสดงชื่อนี้) Shift-Command-Comma จะเปิด **Settings** โดยไม่แทนที่ทางลัด Command-Comma ของเบราว์เซอร์ ส่วนหัวของแถบด้านข้างยังมีสวิตช์ยุบ (⌘B); การยุบจะซ่อนแถบด้านข้างทั้งหมดเพื่อให้ได้เวิร์กสเปซเต็มความกว้าง และตัวควบคุมขยายแบบลอย (หรือ ⌘B) จะนำกลับมา ส่วนแอป macOS จะโฮสต์สวิตช์นั้นแบบเนทีฟในแถบชื่อเรื่องแทน แถบด้านข้างเป็นองค์ประกอบนำทางเพียงอย่างเดียวบนเดสก์ท็อป โดยไม่มีแถบด้านบน วิวพอร์ตแคบจะเปลี่ยนแถบด้านข้างเป็นลิ้นชักแบบเลื่อนทับหลังแถวส่วนหัวขนาดกะทัดรัดที่มีสวิตช์ลิ้นชัก แบรนด์ และการค้นหาพาเล็ตคำสั่ง บนโทรศัพท์ Chat จะรวมแถวนำทางนั้นไว้ในแถบชื่อเรื่อง โดยมีเมนูและตัวควบคุมการค้นหาอยู่ข้างชื่อเซสชัน ในแอป macOS แถวส่วนหัวแยกจะรวมพื้นที่เผื่อของแถบชื่อเรื่องเป็นแถบขนาดกะทัดรัดเพียงแถบเดียวข้างตัวควบคุมหน้าต่าง การนำทางใช้ประวัติเบราว์เซอร์ตามปกติ ดังนั้นปุ่มย้อนกลับ/ไปข้างหน้าของเบราว์เซอร์จึงเลื่อนผ่านประวัติได้ แอป macOS เพิ่มสวิตช์แถบด้านข้างแบบเนทีฟข้างตัวควบคุมหน้าต่าง พร้อมท่าทางปัดบนแทร็กแพด โดยมีปุ่มย้อนกลับ/ไปข้างหน้าที่ขอบขวาของแถบด้านข้างเมื่อขยาย และมีปุ่มค้นหาแบบเนทีฟ (พาเล็ตคำสั่ง) กับปุ่มสร้างเซสชันใหม่เมื่อยุบ

การอนุมัติที่รอดำเนินการจะแสดงชิปแจ้งเตือนเหนือส่วนท้ายของแถบด้านข้างด้วย
เลือกชิปเพื่อเปิดหน้า Approvals ที่รับผิดชอบ

    - การวางเมาส์เหนือหรือใช้แป้นพิมพ์โฟกัสลิงก์ issue หรือ pull request สาธารณะของ GitHub จะแสดงสถานะ ชื่อเรื่อง ผู้เขียน กิจกรรมล่าสุด ความคิดเห็น และสถิติการเปลี่ยนแปลง Gateway ที่เชื่อมต่อจะดึงและแคชเมทาดาทาสาธารณะโดยไม่เปลี่ยนปลายทางของลิงก์ รวมถึงเมื่อ UI ใช้ Gateway ระยะไกล Gateway จะใช้ `GH_TOKEN` หรือ `GITHUB_TOKEN` หากมี หลังจากยืนยันว่ารีพอซิทอรีเป็นสาธารณะ มิฉะนั้นจะใช้ API แบบไม่ระบุตัวตนของ GitHub พร้อมระยะเวลาแคชที่นานกว่า
    - สนทนาผ่านเซสชันเรียลไทม์ของเบราว์เซอร์ OpenAI ใช้ WebRTC โดยตรง, Google Live ใช้โทเค็นเบราว์เซอร์แบบใช้ครั้งเดียวที่มีข้อจำกัดผ่าน WebSocket และ Plugin เสียงเรียลไทม์ที่ทำงานเฉพาะแบ็กเอนด์ใช้การรับส่งผ่านรีเลย์ของ Gateway เซสชันเบราว์เซอร์ที่รองรับวิดีโอสามารถเลือกกล้องภายในอุปกรณ์ได้ใน Settings หรือสลับกล้องจากตัวอย่างภาพสด โดยเบราว์เซอร์จะจับภาพเฟรม JPEG สำหรับผู้ให้บริการเรียลไทม์โดยไม่สตรีมวิดีโอจากกล้องผ่าน Gateway เซสชันผู้ให้บริการที่ไคลเอนต์เป็นเจ้าของเริ่มต้นด้วย `talk.client.create`; เซสชันรีเลย์ของ Gateway เริ่มต้นด้วย `talk.session.create` รีเลย์จะเก็บข้อมูลประจำตัวของผู้ให้บริการไว้บน Gateway ขณะที่เบราว์เซอร์สตรีม PCM จากไมโครโฟนผ่าน `talk.session.appendAudio`, ส่งต่อการเรียกใช้เครื่องมือของผู้ให้บริการ `openclaw_agent_consult` ผ่าน `talk.client.toolCall` เพื่อใช้นโยบายของ Gateway และโมเดล OpenClaw ขนาดใหญ่กว่าที่กำหนดค่าไว้ และส่งการควบคุมการทำงานที่กำลังดำเนินอยู่ด้วยเสียงผ่าน `talk.client.steer` หรือ `talk.session.steer`
    - สตรีมการเรียกใช้เครื่องมือและการ์ดเอาต์พุตสดของเครื่องมือในแชต (เหตุการณ์ของเอเจนต์) กิจกรรมเครื่องมือจะแสดงเป็นแถวตามประเภท: คำสั่งเชลล์แสดงคำสั่งพร้อมไฮไลต์ไวยากรณ์และเอาต์พุตแบบเทอร์มินัล; การเรียกแก้ไขและเขียนที่รองรับจะแสดง diff แบบอินไลน์ที่มีขอบเขตจำกัด หมายเลขบรรทัดเมื่อมี และสถิติ `+added -removed`; และการเรียกต่อเนื่องจะยุบเป็นสรุป เช่น "เรียกใช้ 13 คำสั่ง อ่าน 6 ไฟล์ แก้ไข 9 ไฟล์" ระหว่างที่การทำงานยังดำเนินอยู่ ชื่อการเรียกล่าสุดที่กำลังทำงานจะปรากฏเป็นส่วนหัวของกลุ่ม ขยายแถวเพื่อตรวจสอบอาร์กิวเมนต์ที่เหลือและเอาต์พุตดิบ
    - ชื่อวัตถุประสงค์จาก AI สำหรับการเรียกใช้เครื่องมือที่ซับซ้อน (คำสั่งเชลล์ยาว เครื่องมือ Plugin ที่มีอาร์กิวเมนต์จำนวนมาก) ซึ่งเลือกเปิดใช้ได้ด้วย `gateway.controlUi.toolTitles: true` (ปิดโดยค่าเริ่มต้น) ชื่อมาจากเมธอด `chat.toolTitles` แบบแบตช์ผ่านการกำหนดเส้นทางโมเดลยูทิลิตีมาตรฐาน โดยใช้ `utilityModel` ที่ระบุอย่างชัดเจน (ผู้ให้บริการที่ผู้ดำเนินการเลือก เช่นเดียวกับงานยูทิลิตีอื่น) หรือใช้ค่าเริ่มต้นของโมเดลขนาดเล็กที่ประกาศโดยผู้ให้บริการของเซสชัน และแคชบนฝั่ง Gateway แยกตามเอเจนต์ เมื่อไม่ได้เลือกเปิดใช้หรือไม่มีโมเดลต้นทุนต่ำที่ใช้งานได้ แถวจะคงป้ายกำกับที่กำหนดแน่นอนไว้และจะไม่มีการเรียกโมเดล
    - เริ่มหรือยกเลิกงานติดตามผลชั่วคราวที่โมเดลแนะนำ โดยคำแนะนำที่ยอมรับจะเปิดเซสชันเวิร์กทรีที่มีการจัดการใหม่พร้อมพรอมต์ที่เสนอ
    - แท็บกิจกรรมที่มีสรุปกิจกรรมเครื่องมือสดจากการส่ง `session.tool` / เหตุการณ์เครื่องมือที่มีอยู่ โดยจัดเก็บภายในเบราว์เซอร์และปกปิดข้อมูลก่อน

  </Accordion>
  <Accordion title="ช่องทาง เซสชัน และหน่วยความจำ">
    - ช่องทาง: สถานะของช่องทางในตัวและช่องทางจาก Plugin ที่รวมมาด้วย/ภายนอก การเข้าสู่ระบบด้วย QR และการกำหนดค่ารายช่องทาง (`channels.status`, `web.login.*`, `config.patch`)
    - การรีเฟรชการตรวจสอบช่องทางจะคงสแนปช็อตก่อนหน้าให้มองเห็นได้ขณะที่รอการตรวจสอบผู้ให้บริการที่ใช้เวลานานเสร็จสิ้น และติดป้ายกำกับสแนปช็อตบางส่วนเมื่อการตรวจสอบหรือการตรวจประเมินใช้เวลาเกินงบเวลาของ UI
    - เธรด (หน้าเวิร์กสเปซที่ `/sessions` โดยมีแท็บ **เวิร์กทรี** อยู่ข้างกัน): แสดงรายการเซสชันของเอเจนต์ที่กำหนดค่าไว้โดยค่าเริ่มต้น ปักหมุดเซสชันที่ใช้บ่อย เปลี่ยนชื่อ เก็บถาวรหรือกู้คืนเซสชันที่ไม่ได้ใช้งาน ย้อนกลับจากคีย์เซสชันของเอเจนต์ที่ไม่ได้กำหนดค่าซึ่งล้าสมัย และใช้การตั้งค่าแทนที่โมเดล/การคิด/ความเร็วสูง/รายละเอียด/การติดตาม/การใช้เหตุผลแยกตามเซสชัน (`sessions.list`, `sessions.patch`) เซสชันที่ปักหมุดจะเรียงอยู่เหนือเซสชันล่าสุดที่ไม่ได้ปักหมุด; เซสชันที่เก็บถาวรอยู่ในมุมมองรายการเก็บถาวรของหน้าเธรดและยังคงเก็บบทสนทนาไว้ แถวจะแสดงจุดที่ยังไม่ได้อ่านสำหรับเซสชันที่มีกิจกรรมนับตั้งแต่อ่านครั้งล่าสุด พร้อมการดำเนินการทำเครื่องหมายว่ายังไม่ได้อ่าน/อ่านแล้ว (`sessions.patch { unread }`) และการดำเนินการแยกแขนงที่แตกบทสนทนาเป็นเซสชันใหม่ (`sessions.create { parentSessionKey, fork: true }`) ไทล์ภาพรวมเหนือตารางจะสรุปรายการที่โหลดแล้ว (จำนวนเซสชัน การทำงานสด เซสชันที่ยังไม่ได้อ่าน โทเค็นทั้งหมด) แต่ละแถวมีสัญลักษณ์ประเภทพร้อมจุดแสดงการทำงานสด สถานะแสดงเป็นจุดธรรมดาพร้อมป้ายกำกับ และคอลัมน์โทเค็นจะแสดงมาตรวัดการใช้หน้าต่างบริบทเมื่อเซสชันรายงานขนาดโทเค็นและบริบท การดำเนินการจัดการแถวอยู่ในเมนูประจำแถว (ปุ่มจุดสามจุดแนวตั้งหรือคลิกขวา) ซึ่งสอดคล้องกับเมนูเซสชันของแถบด้านข้าง และลิ้นชักแถวจะแสดงรันไทม์ของเอเจนต์และระยะเวลาการทำงานควบคู่กับรายละเอียดเซสชันอื่น
    - แค็ตตาล็อกแถบด้านข้างแบบเนทีฟของ Claude และ Codex จะสตรีมทีละโฮสต์ จากนั้นปรับข้อมูลให้สอดคล้องหลังการเชื่อมต่อ Node เปลี่ยนแปลง เมื่อหน้าได้รับโฟกัส และไม่เกินทุก 30 วินาทีขณะที่มองเห็นอยู่ การเปลี่ยนแปลงแค็ตตาล็อกจะเรียกการตรวจสอบติดตามผลที่เร็วขึ้น เพื่อให้เซสชันที่สร้างในเครื่องมือแบบเนทีฟปรากฏขึ้นโดยไม่ต้องโหลด Control UI ใหม่
    - การจัดกลุ่มเซสชัน: ตัวควบคุม Group by จัดระเบียบตารางเซสชันเป็นส่วนต่าง ๆ ตามกลุ่มที่กำหนดเอง ช่องทาง ประเภท เอเจนต์ หรือวันที่ กลุ่มที่กำหนดเองจะคงอยู่แยกตามเซสชันผ่าน `sessions.patch` (`category`) จึงสามารถจัดหมวดหมู่เซสชันที่เริ่มจากช่องทางข้อความ (Discord, Telegram, WhatsApp, ...) ได้ด้วย; กำหนดกลุ่มโดยลากแถวไปยังส่วน หรือใช้ตัวเลือกกลุ่มประจำแถว และสร้างกลุ่มด้วยการดำเนินการ New group
    - หน่วยความจำ (แท็บบนหน้าเอเจนต์ ซึ่งจำกัดขอบเขตเฉพาะเอเจนต์ที่เลือก): สถานะ Dreaming ตัวสลับเปิด/ปิด และตัวอ่านบันทึกความฝัน (`doctor.memory.status`, `doctor.memory.dreamDiary`, `config.patch`)
    - นำเข้าหน่วยความจำ (`/memory-import` เข้าถึงจากแท็บหน่วยความจำของหน้าเอเจนต์): แสดงตัวอย่างและคัดลอกหน่วยความจำอัตโนมัติภายในเครื่องของ Claude Code หน่วยความจำรวมของ Codex หรือไฟล์หน่วยความจำ Hermes ไปยังเวิร์กสเปซของเอเจนต์ที่เลือก (`migrations.memory.plan`, `migrations.memory.apply`)
    - ข้อเสนอหน่วยความจำระหว่างการเริ่มต้นใช้งาน: เมื่อ Control UI เปิดในโหมดการเริ่มต้นใช้งาน (`?onboarding=1` ซึ่งแอปคู่หู Linux ใช้หลังการติดตั้งครั้งแรก) กล่องโต้ตอบหน้าเดียวจะเสนอให้นำเข้าหน่วยความจำที่ตรวจพบด้วยขั้นตอนวางแผน/นำไปใช้แบบเดียวกัน; การข้ามจะคงหน้าการตั้งค่าไว้เป็นจุดเข้าสำหรับภายหลัง

  <Accordion title="Cron งาน Plugin Skills อุปกรณ์ และการอนุมัติการดำเนินการ">
    - ระบบอัตโนมัติ (งาน cron): การ์ดสถิติ (จำนวนระบบอัตโนมัติ จำนวนที่ล้มเหลว สถานะตัวจัดกำหนดการ การปลุกครั้งถัดไป) เหนือตัวสลับแท็บ Automations/Run history; แท็บ Automations แสดงรายการงานในตารางที่กรองได้ (All/Active/Paused, การค้นหา ตัวกรองกำหนดการและการทำงานล่าสุด เมนูการดำเนินการประจำแถว) พร้อมคำแนะนำเริ่มต้นด้านล่าง และแท็บ Run history แสดงการทำงานล่าสุดของระบบอัตโนมัติทั้งหมด (`cron.*`)
    - งาน: บัญชีแยกประเภทแบบสดของงานเบื้องหลังที่กำลังทำงานและล่าสุด พร้อมเซสชันที่เชื่อมโยงและการยกเลิก (`tasks.*`) รางงานเบื้องหลังของแชตจะจัดกลุ่มงานที่กำลังทำงานและเสร็จสิ้น; เลือกแถวเพื่อตรวจสอบพรอมต์ที่มีขอบเขตจำกัดและสรุปเอาต์พุตหรือข้อผิดพลาด
    - Plugin: เรียกดูรายการที่ติดตั้งและร้านค้าที่คัดสรร ค้นหา ClawHub ติดตั้งและนำโค้ด Plugin ออก และเปิดหรือปิดใช้ Plugin ที่ติดตั้ง (`plugins.*`); แถวเซิร์ฟเวอร์ MCP จะแก้ไข `mcp.servers` ผ่านเมธอดการกำหนดค่า
    - Skills: สถานะ เปิด/ปิดใช้ ติดตั้ง อัปเดตคีย์ API (`skills.*`)
    - อุปกรณ์: รายการเดียวรวมระเบียนอุปกรณ์ที่จับคู่ แค็ตตาล็อก Node และสถานะออนไลน์แบบสด (`device.pair.list`, `node.list`, `system-presence`) โฮสต์ Gateway จะถูกปักหมุดเป็นอันดับแรก; ไคลเอนต์ที่จับคู่จะแสดงสถานะการเชื่อมต่อ บทบาท โทเค็น ความสามารถ และคำสั่ง การจับคู่ที่ซ้ำกันจะยุบเป็นกลุ่มที่ขยายได้ และ **ล้างข้อมูลที่ล้าสมัย N รายการ** จะนำรายการซ้ำที่ออฟไลน์ออกเป็นกลุ่มหลังผู้ดูแลระบบยืนยัน ซึ่งเป็นรายการที่ได้รับอนุมัติอัตโนมัติ (ภายในเครื่องแบบเงียบ CIDR ที่เชื่อถือได้ หรือยืนยันผ่าน SSH) หรือมีอยู่ก่อนเริ่มบันทึกที่มาของการอนุมัติ สามารถนำรายการออกได้ (`node.pair.remove`, `device.pair.remove`) จัดการการจับคู่อุปกรณ์และการอนุมัติ Node ซ้ำแบบอินไลน์ (`device.pair.*`, `node.pair.approve`/`reject`) และสร้างรหัสตั้งค่ามือถือจากการ์ดเดียวกัน
    - การอนุมัติ Exec: แก้ไขรายการอนุญาตของ Gateway หรือ Node และนโยบายการถามสำหรับ `exec host=gateway/node` (`exec.approvals.*`)

  </Accordion>

    - การนำทางใน Settings จัดกลุ่มหน้าตามลำดับความสำคัญที่ต้องให้ความสนใจ ได้แก่ General, Appearance และ Notifications ที่ด้านบน; Connections (Connection, Channels, Communications, Devices); Agents & Tools (Agents, AI & Agents, Model Providers, MCP, Automation, Labs); Privacy & Security (Security, Approvals); และ System (Infrastructure, Advanced, Debug, Logs, About) โดย General เป็นศูนย์รวมแบบกระชับสำหรับค่าเริ่มต้นของโมเดล ภาษา และสถิติโฮสต์ Gateway ส่วนการตั้งค่าอื่นทุกอย่างจะอยู่ในหน้าใดหน้าหนึ่งเพียงหน้าเดียว
    - Privacy & Security: แถวที่คัดสรรไว้สำหรับการยืนยันตัวตนของ Gateway, นโยบายการดำเนินการ, การเปิดใช้งานเบราว์เซอร์, โปรไฟล์เครื่องมือ, การยืนยันตัวตนอุปกรณ์ และการจับคู่กับอุปกรณ์เคลื่อนที่ ซึ่งอยู่เหนือส่วน `security`/`approvals` ที่อิงตามสคีมา
    - Approvals มีประวัติคำขอที่ได้รับการแก้ไขแล้วสำหรับการดำเนินการ, Plugin และเอเจนต์ระบบเป็นเวลา 30 วัน โดยเรียงรายการใหม่ที่สุดก่อน กรองตามประเภทหรือเลื่อนไปยังแถวเก่าเพื่อทบทวนการตัดสินใจ เหตุผล เซสชันต้นทาง และข้อมูลผู้แก้ไขที่ Gateway บันทึกไว้
    - Labs แสดงสวิตช์ฟีเจอร์ทดลองที่จัดส่งแล้ว โดย Code Mode เป็นรายการปัจจุบันและบันทึก `tools.codeMode.enabled` ทันที ส่วนการทดลองที่ยังไม่ได้จัดส่งจะไม่ปรากฏและไม่เขียนคีย์การกำหนดค่าเชิงคาดการณ์
    - Notifications: สถานะเว็บพุชของเบราว์เซอร์ การสมัคร/ยกเลิกการสมัคร และการส่งทดสอบ
    - Advanced: ส่วนการกำหนดค่าทุกส่วนที่ไม่มีหน้าหลักเฉพาะ รวมถึงตัวแก้ไข JSON5 แบบดิบ (เดิมคือโหมด Advanced ของหน้า General)
    - Model Setup (`/settings/model-setup`) เป็นหน้าย่อยของ Model Providers ซึ่งเปิดจากส่วนหัวของหน้านั้น
    - Agents: หน้าการตั้งค่า (**Settings → Agents**, `/settings/agents`) ที่มีแท็บต่อเอเจนต์ (Overview, Files, Tools, Skills, Channels, Automations, Memory) แท็บ Overview ใช้แก้ไขอัตลักษณ์ของเอเจนต์ ได้แก่ ชื่อที่แสดง อีโมจิ และรูปอวาตาร์ที่เบราว์เซอร์จะลดขนาดและจำกัดขนาดก่อน `agents.update` การบันทึกจะเก็บฟิลด์อัตลักษณ์ที่กำหนดค่าไว้และสะท้อนไปยัง `IDENTITY.md` ของพื้นที่ทำงาน โดยค่าที่กำหนดไว้จะมีลำดับความสำคัญเหนือการแก้ไขฟิลด์เดียวกันในไฟล์ด้วยตนเอง
    - Profile: หน้าการตั้งค่าที่แสดงอัตลักษณ์ของเอเจนต์เริ่มต้นพร้อมสถิติการใช้งานตลอดเวลา ได้แก่ จำนวนโทเค็นตลอดอายุการใช้งาน วันที่ใช้งานสูงสุด เซสชันที่ยาวนานที่สุด ช่วงกิจกรรมต่อเนื่อง ฮีตแมปโทเค็นตลอดหนึ่งปี เครื่องมือที่ใช้มากที่สุด และไฮไลต์ของช่องทาง (`usage.cost`, `sessions.usage`)
    - MCP มีหน้าการตั้งค่าเฉพาะพร้อมแถวเซิร์ฟเวอร์ (ข้อมูลสรุปการรับส่ง การเปิดใช้งาน OAuth/ตัวกรอง/การทำงานแบบขนาน) ตัวควบคุมสำหรับเพิ่ม/เปิดใช้/ปิดใช้/นำออกโดยตรง คำสั่งทั่วไปสำหรับผู้ดูแล และตัวแก้ไขการกำหนดค่า `mcp` ที่จำกัดขอบเขต ส่วนหน้า Plugins ยังคงเป็นหน้าหลักสำหรับตัวเชื่อมต่อแบบคลิกเดียวและการค้นหา
    - Model Providers: หน้าการตั้งค่าที่แสดงผู้ให้บริการโมเดลทุกรายที่กำหนดค่าไว้ พร้อมไอคอนแบรนด์ สถานะการยืนยันตัวตน (`models.authStatus`) ความพร้อมใช้งานของโมเดล (`models.list`) ข้อมูลแพ็กเกจ/โควตา/การเรียกเก็บเงินแบบสดในกรณีที่ผู้ให้บริการรายงานข้อมูลดังกล่าว (`usage.status`) และค่าใช้จ่ายของเซสชันภายในเครื่องในช่วง 30 วันที่ผ่านมา (`sessions.usage`) การดำเนินการ Refresh จะอ่านสถานะข้อมูลประจำตัวและการใช้งานของผู้ให้บริการใหม่
    - Connection: หน้าการตั้งค่า (ภายใต้ **Connections**) ที่จัดการลิงก์ Gateway ของแดชบอร์ดเอง ได้แก่ URL ของ WebSocket, โทเค็น Gateway, รหัสผ่าน และคีย์เซสชันเริ่มต้น พร้อมสแนปช็อตแฮนด์เชกล่าสุด (สถานะ เวลาทำงาน ช่วงเวลาทิก และเวลาที่รีเฟรชช่องทางล่าสุด) ประตูเข้าสู่ระบบแบบออฟไลน์จะจัดการกรณีที่ไม่ได้เชื่อมต่อ ส่วนหน้านี้ใช้แก้ไขการเชื่อมต่อขณะที่เชื่อมต่ออยู่
    - นำไปใช้และรีสตาร์ตพร้อมการตรวจสอบความถูกต้อง (`config.apply`) จากนั้นปลุกเซสชันที่ใช้งานล่าสุด
    - การเขียนมีตัวป้องกันด้วยแฮชฐานเพื่อไม่ให้เขียนทับการแก้ไขที่เกิดขึ้นพร้อมกัน
    - การเขียน (`config.set`/`config.apply`/`config.patch`) จะตรวจสอบล่วงหน้าว่าสามารถแก้ไข SecretRef ที่ใช้งานอยู่สำหรับการอ้างอิงในเพย์โหลดการกำหนดค่าที่ส่งมาได้หรือไม่ โดยการอ้างอิงที่ส่งมาซึ่งใช้งานอยู่แต่แก้ไขไม่ได้จะถูกปฏิเสธก่อนเขียน
    - การบันทึกฟอร์มจะละทิ้งตัวยึดตำแหน่งที่ปกปิดข้อมูลไว้ซึ่งล้าสมัยและไม่สามารถคืนค่าจากการกำหนดค่าที่บันทึกไว้ได้ ขณะเดียวกันจะรักษาค่าที่ปกปิดข้อมูลไว้ซึ่งยังคงเชื่อมโยงกับข้อมูลลับที่บันทึกไว้
    - สคีมาและการแสดงผลฟอร์มมาจาก `config.schema` / `config.schema.lookup` ซึ่งรวมถึงฟิลด์ `title`/`description` คำแนะนำ UI ที่ตรงกัน ข้อมูลสรุปขององค์ประกอบลูกโดยตรง เมทาดาทาเอกสารบนโหนดออบเจ็กต์ซ้อน/ไวลด์การ์ด/อาร์เรย์/องค์ประกอบ รวมถึงสคีมาของ Plugin และช่องทางเมื่อมี ตัวแก้ไข JSON แบบดิบจะพร้อมใช้งานเฉพาะเมื่อสแนปช็อตรองรับการไปกลับของข้อมูลดิบอย่างปลอดภัย มิฉะนั้น Control UI จะบังคับใช้โหมด Form
    - คำสั่ง "Reset to saved" ในตัวแก้ไข JSON แบบดิบจะรักษารูปแบบที่เขียนแบบดิบไว้ (การจัดรูปแบบ ความคิดเห็น และโครงร่าง `$include`) แทนการแสดงผลสแนปช็อตแบบแบนใหม่ เพื่อให้การแก้ไขจากภายนอกยังคงอยู่หลังรีเซ็ตเมื่อสแนปช็อตสามารถไปกลับได้อย่างปลอดภัย
    - ค่าออบเจ็กต์ SecretRef แบบมีโครงสร้างจะแสดงเป็นแบบอ่านอย่างเดียวในช่องป้อนข้อความของฟอร์ม เพื่อป้องกันไม่ให้ออบเจ็กต์เสียหายจากการแปลงเป็นสตริงโดยไม่ตั้งใจ

## นำเข้าหน่วยความจำของผู้ช่วย

เปิด **Settings** → **Import Memory** เพื่อนำหน่วยความจำ Codex หรือ Claude Code ภายในเครื่อง
เข้าสู่เอเจนต์ OpenClaw โดย Gateway จะค้นหาหน่วยความจำภายในเครื่องที่รองรับบน
โฮสต์ของตนเอง ดังนั้น Control UI ระยะไกลจึงนำเข้าจากคอมพิวเตอร์ของ Gateway แทน
คอมพิวเตอร์ที่ใช้เบราว์เซอร์

1. เลือกเอเจนต์ปลายทาง
2. ตรวจสอบคอลเลกชันต้นทางและชื่อไฟล์ Markdown ที่ตรวจพบ เนื้อหาไฟล์
   จะไม่ถูกส่งในการตอบกลับแผนหรือแสดงในหน้า
3. เลือกคอลเลกชันที่จะนำเข้าและยืนยัน การนำไปใช้จะสร้างแผนใหม่ก่อน
   เขียน เพื่อให้การเลือกที่ล้าสมัยล้มเหลวอย่างปลอดภัย
4. หากมีไฟล์อยู่แล้ว ให้เปิดใช้ **Replace existing imports** รีเฟรช
   ตัวอย่าง แล้วจึงยืนยันการแทนที่

Codex นำเข้าเฉพาะ `MEMORY.md` และ `memory_summary.md` ที่รวมไว้แล้ว ส่วน Claude
Code นำเข้า Markdown จากไดเรกทอรีหน่วยความจำอัตโนมัติของโปรเจกต์และ
`autoMemoryDirectory` ที่กำหนดค่าไว้ โดยจะไม่นำเข้าเซสชัน การตั้งค่า คำสั่ง หรือ
ข้อมูลประจำตัวผ่านหน้านี้ ไฟล์จะถูกคัดลอกไว้ใต้ `memory/imports/` ใน
พื้นที่ทำงานที่เลือก ซึ่ง Plugin หน่วยความจำที่ใช้งานอยู่สามารถจัดทำดัชนีได้ แหล่งข้อมูล
จะไม่ถูกเปลี่ยนแปลง

การวางแผนและการนำไปใช้ต้องใช้ `operator.admin` การนำไปใช้แต่ละครั้งจะสร้างข้อมูลสำรอง
OpenClaw ที่ตรวจสอบแล้วเมื่อมีสถานะอยู่ เขียนรายงานการย้ายที่ปกปิดข้อมูล และเก็บ
ข้อมูลสำรองระดับรายการก่อนแทนที่ไฟล์ปลายทางที่มีอยู่ ดูเส้นทางและ
พฤติกรรมการเรียกคืนได้ที่ [ภาพรวมหน่วยความจำ](/th/concepts/memory#import-from-coding-assistants)

## หน้า MCP

หน้า MCP เฉพาะเป็นมุมมองสำหรับผู้ดูแลเซิร์ฟเวอร์ MCP ที่ OpenClaw จัดการภายใต้ `mcp.servers` หน้านี้ไม่ได้เริ่มการรับส่ง MCP ด้วยตนเอง ให้ใช้เพื่อตรวจสอบและแก้ไขการกำหนดค่าที่บันทึกไว้ จากนั้นใช้ `openclaw mcp doctor --probe` เมื่อต้องการหลักฐานการทำงานของเซิร์ฟเวอร์แบบสด

ขั้นตอนการทำงานทั่วไป:

1. เปิด **MCP** จากแถบด้านข้าง
2. ตรวจสอบการ์ดข้อมูลสรุปเพื่อดูจำนวนเซิร์ฟเวอร์ทั้งหมด ที่เปิดใช้งาน ใช้ OAuth และมีการกรอง
3. ตรวจสอบแต่ละแถวของเซิร์ฟเวอร์เพื่อดูการรับส่ง การเปิดใช้งาน การยืนยันตัวตน ตัวกรอง ระยะหมดเวลา และคำแนะนำคำสั่ง
4. เพิ่ม เปิดใช้ ปิดใช้ หรือนำเซิร์ฟเวอร์ออกได้โดยตรงในหน้า MCP ใช้หน้า **Plugins** สำหรับตัวเชื่อมต่อแบบคลิกเดียวและการค้นหา
5. แก้ไขส่วนการกำหนดค่า `mcp` ที่จำกัดขอบเขตสำหรับคำจำกัดความเซิร์ฟเวอร์ ส่วนหัว เส้นทาง TLS/mTLS เมทาดาทา OAuth ตัวกรองเครื่องมือ และเมทาดาทาการฉายภาพ Codex
6. ใช้ **Save** เพื่อเขียนการกำหนดค่า หรือ **Save & Publish** เมื่อต้องการให้ Gateway ที่กำลังทำงานนำการกำหนดค่าที่เปลี่ยนแปลงไปใช้
7. เรียกใช้ `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` หรือ `openclaw mcp reload` จากเทอร์มินัลสำหรับการวินิจฉัยแบบคงที่ การพิสูจน์แบบสด หรือการทิ้งรันไทม์ที่แคชไว้

หน้าดังกล่าวจะปกปิดค่าที่คล้าย URL ซึ่งมีข้อมูลประจำตัวก่อนแสดงผล และใส่ชื่อเซิร์ฟเวอร์ไว้ในเครื่องหมายอัญประกาศภายในส่วนย่อยคำสั่ง เพื่อให้คำสั่งที่คัดลอกยังคงทำงานได้เมื่อชื่อมีช่องว่างหรืออักขระพิเศษของเชลล์ ข้อมูลอ้างอิง CLI และการกำหนดค่าฉบับเต็ม: [MCP](/th/cli/mcp)

## แท็บ Activity

แท็บ Activity อยู่ใน **Settings › System** ถัดจาก Logs และ Debug โดยเป็นตัวสังเกตการณ์ชั่วคราวภายในเบราว์เซอร์สำหรับกิจกรรมเครื่องมือแบบสด ซึ่งได้มาจากสตรีมเหตุการณ์ `session.tool` / เครื่องมือของ Gateway เดียวกับที่ขับเคลื่อนการ์ดเครื่องมือใน Chat โดยจะไม่เพิ่มตระกูลเหตุการณ์ของ Gateway, จุดเชื่อมต่อ, ที่เก็บกิจกรรมถาวร, ฟีดเมตริก หรือสตรีมผู้สังเกตการณ์ภายนอกอีกชุดหนึ่ง

รายการ Activity จะเก็บเฉพาะข้อมูลสรุปที่ผ่านการปรับให้ปลอดภัยและตัวอย่างผลลัพธ์ที่ปกปิดข้อมูลและตัดทอนแล้ว ค่าของอาร์กิวเมนต์เครื่องมือจะไม่ถูกจัดเก็บในสถานะ Activity โดย UI จะแสดงว่าอาร์กิวเมนต์ถูกซ่อนและบันทึกเฉพาะจำนวนฟิลด์อาร์กิวเมนต์ รายการในหน่วยความจำจะผูกกับแท็บเบราว์เซอร์ปัจจุบัน ยังคงอยู่ระหว่างการนำทางภายใน Control UI และรีเซ็ตเมื่อโหลดหน้าใหม่ สลับเซสชัน หรือกด **Clear**

## เทอร์มินัลสำหรับผู้ดูแล

เทอร์มินัลสำหรับผู้ดูแลแบบวางเทียบได้ถูกปิดใช้งานตามค่าเริ่มต้น หากต้องการเปิดใช้ ให้ตั้งค่า `gateway.terminal.enabled: true` และรีสตาร์ต Gateway เทอร์มินัลต้องใช้การเชื่อมต่อ `operator.admin` และจะเปิด PTY ของโฮสต์ในพื้นที่ทำงานของเอเจนต์ที่ใช้งานอยู่ แท็บใหม่จะใช้เอเจนต์แชตที่เลือกอยู่ในขณะนั้น

<Warning>
เทอร์มินัลเป็นเชลล์ของโฮสต์ที่ไม่มีการจำกัดขอบเขตและสืบทอดสภาพแวดล้อมของกระบวนการ Gateway เปิดใช้งานเฉพาะในการติดตั้งใช้งานโดยผู้ดำเนินการที่เชื่อถือได้เท่านั้น OpenClaw ปฏิเสธเซสชันเทอร์มินัลสำหรับเอเจนต์ที่มี `sandbox.mode: "all"`; การเปลี่ยนเอเจนต์ที่กำลังทำงานไปเป็นโหมดดังกล่าวจะปิดเซสชันเทอร์มินัลที่มีอยู่และที่กำลังดำเนินการทั้งหมด
</Warning>

ใช้ **Ctrl + backtick** เพื่อสลับการแสดงด็อก เลย์เอาต์รองรับการด็อกด้านล่างและด้านขวา ปรับขนาดตามวิวพอร์ตของเบราว์เซอร์ และเก็บแท็บเชลล์ได้หลายแท็บ ดู `gateway.terminal.enabled` และการแทนที่ `gateway.terminal.shell` ซึ่งเป็นทางเลือกได้ที่ [การกำหนดค่า Gateway](/th/gateway/configuration-reference#gateway)

เอเจนต์ที่เจ้าของอนุญาตและไม่ได้อยู่ในแซนด์บ็อกซ์สามารถใช้เครื่องมือ `terminal` สำหรับงานที่ใช้เวลานานหรืองานแบบโต้ตอบซึ่งผู้ดำเนินการควรเฝ้าดู การเรียกใช้เครื่องมือแต่ละครั้งสามารถเปิด อ่าน เขียน ปรับขนาด ปิด หรือแสดงรายการ PTY ของ Gateway ที่เป็นของเอเจนต์เองได้ โดยค่าเริ่มต้น เซสชันใหม่จะเปิดแท็บ Control UI ที่เชื่อมต่อร่วมกัน เพื่อให้เอเจนต์และผู้ดำเนินการใช้เอาต์พุตร่วมกัน และทั้งสองฝ่ายสามารถพิมพ์หรือปรับขนาดได้ การเข้าถึงของเอเจนต์จำกัดเฉพาะเซสชันอย่างเคร่งครัด: เอเจนต์ไม่สามารถอ่านหรือควบคุมเทอร์มินัลที่ผู้ดำเนินการสร้างหรือเทอร์มินัลที่เปิดโดยเซสชันเอเจนต์อื่น

ลากไฟล์อย่างน้อยหนึ่งไฟล์ไปยังเทอร์มินัลที่ใช้งานอยู่ หรือใช้ปุ่มคลิปหนีบกระดาษเพื่อเลือกไฟล์ OpenClaw จะจัดเตรียมแต่ละไฟล์ไว้บนเครื่องที่เป็นเจ้าของ PTY และวางพาธสัมบูรณ์ที่ผ่านการใส่เครื่องหมายอัญประกาศสำหรับเชลล์ไว้ที่เคอร์เซอร์ โดยจะไม่กด Enter หรือดำเนินการอินพุต ตัวบ่งชี้ชุดงานแบบกะทัดรัดจะแสดงไฟล์ปัจจุบันและจำนวนที่เสร็จสิ้น การยกเลิกจะหยุดส่วนที่เหลือของชุดงานโดยไม่วางพาธ การถ่ายโอนที่ล้มเหลวจะยังคงแสดงอยู่ เพื่อให้ลองใหม่จากไฟล์นั้นได้โดยไม่ต้องอัปโหลดไฟล์ที่เสร็จแล้วซ้ำ รองรับรูปภาพ PDF ไฟล์เก็บถาวร และไฟล์ประเภทอื่น โดยมีขนาดสูงสุด 16 MiB ต่อไฟล์ ไฟล์ที่จัดเตรียมไว้จะใช้ไดเรกทอรีชั่วคราวของระบบแบบส่วนตัวบนโฮสต์ POSIX (โหมดไดเรกทอรี `0700`, โหมดไฟล์ `0600`) หรือไดเรกทอรีภายในขอบเขต ACL ของโปรไฟล์ผู้ใช้บน Windows พร้อมตัวตั้งเวลาล้างข้อมูล 24 ชั่วโมง ดังนั้นให้ย้ายหรือคัดลอกสิ่งที่ต้องการเก็บไว้

การแทรกพาธรองรับ PowerShell, `cmd.exe` และเชลล์ POSIX ที่ระบบรู้จัก (`sh`, Bash, Dash, Ash, Ksh, Zsh และ Fish) รวมถึง Git Bash บน Windows การแทนที่ด้วยเชลล์อื่นจะถูกปฏิเสธ เนื่องจากไม่สามารถอนุมานกฎการใส่เครื่องหมายอัญประกาศได้อย่างปลอดภัย ให้เรียกใช้ Gateway ภายใน WSL เพื่อใช้เทอร์มินัล WSL แบบเนทีฟและพาธอัปโหลดของ Linux พาธ `cmd.exe` ที่มี `%` หรือ `!` จะถูกปฏิเสธเช่นกัน เนื่องจากเชลล์ดังกล่าวจะขยายอักขระเหล่านั้นแม้อยู่ภายในเครื่องหมายอัญประกาศคู่

เซสชัน Codex และ Claude Code ที่ค้นพบในแถบด้านข้างของเซสชันสามารถเปิดใน CLI แบบเนทีฟภายในแผงเทอร์มินัลเดียวกันได้ ใน **Settings › Chat** ให้ตั้งค่า **Open Codex/Claude threads in** เป็น **Terminal** เพื่อให้การคลิกแถวตามปกติเปิด `codex resume` หรือ `claude --resume`; ค่าเริ่มต้นยังคงเป็นตัวแสดง OpenClaw แบบอ่านอย่างเดียว เมนูคลิกขวาหรือเมนูจุดสามจุดของแถวจะมีตัวเลือกทั้งสองเสมอ และส่วนหัวของตัวแสดงจะมี **Open in terminal** เมื่อเซสชันนั้นมีสิทธิ์

สิทธิ์จะกำหนดแยกตามแต่ละเซสชันและแต่ละโฮสต์ เซสชันที่อยู่ในเครื่องเดียวกับ Gateway จะเริ่มคำสั่งดำเนินการต่อที่ผู้ให้บริการเป็นเจ้าของบนโฮสต์ Gateway เซสชันของ Node ที่จับคู่แล้วจะเริ่มคำสั่งของผู้ให้บริการที่อยู่ในรายการอนุญาตบน Node เจ้าของ และส่งต่อเฉพาะเอาต์พุต อินพุต และเหตุการณ์การปรับขนาดของ PTY นั้นเท่านั้น การทำเช่นนี้จะไม่เปิดเผยเชลล์ทั่วไปของ Node หรือรับคำสั่งที่เบราว์เซอร์ส่งมา การอัปโหลดไฟล์ใช้คำสั่ง Node `terminal.upload` ที่แยกต่างหากและมีการจำกัดขนาด และยังคงผูกกับเซสชันเทอร์มินัลที่เปิดอยู่แล้ว อนุมัติการอัปเกรดการจับคู่ Node เมื่อคำสั่งดังกล่าวปรากฏเป็นครั้งแรก Node ที่ไม่ได้ประกาศคำสั่งดำเนินการต่อของเทอร์มินัลที่ตรงกัน รวมถึงบริดจ์เวิร์กเกอร์แบบฝังที่ไม่มีการสตรีมสองทิศทาง จะยังคงใช้ตัวแสดงได้และแสดงว่าการเปิดเทอร์มินัลไม่พร้อมใช้งาน ส่วน Node รุ่นเก่ายังคงเรียกใช้เทอร์มินัลได้ แต่ไม่สามารถรับไฟล์ที่ลากมาวางได้

เซสชันที่การเชื่อมต่อเป็นเจ้าของจะยังคงอยู่หลังการตัดการเชื่อมต่อ: การโหลดหน้าใหม่ แล็ปท็อปเข้าสู่โหมดพัก หรือปัญหาเครือข่ายชั่วคราวจะทำให้เซสชันแยกออกจาก Gateway แทนที่จะถูกปิด และแท็บเบราว์เซอร์เดิมจะเชื่อมต่อกลับเมื่อการเชื่อมต่อกลับมา พร้อมเล่นเอาต์พุตล่าสุดซ้ำ เซสชันที่การเชื่อมต่อเป็นเจ้าของและถูกแยกออกจะถูกปิดหลังจาก `gateway.terminal.detachedSessionTimeoutSeconds` (ค่าเริ่มต้น 300 วินาที; `0` จะคืนค่าการปิดเมื่อตัดการเชื่อมต่อ) การเชื่อมต่อเข้ากับเซสชันเหล่านี้ยังคงเป็นการเข้าควบคุมแบบ tmux

เซสชันที่เอเจนต์เป็นเจ้าของไม่ได้ผูกกับการเชื่อมต่อของเบราว์เซอร์ `terminal.attach` จะเพิ่มเบราว์เซอร์แต่ละตัวเป็นผู้ดูโดยไม่รับความเป็นเจ้าของ และการปิดแท็บผู้ดูจะแยกเฉพาะเบราว์เซอร์นั้น PTY จะยังคงอยู่จนกว่าเอเจนต์เจ้าของจะปิด กระบวนการของ PTY สิ้นสุด นโยบายปิดใช้งาน หรือ Gateway ปิดตัวลง `terminal.list` จะระบุแต่ละรายการว่าเป็นของการเชื่อมต่อหรือเอเจนต์ และ `terminal.text` ช่วยให้การเชื่อมต่อของผู้ดูแลระบบอ่านเอาต์พุตข้อความธรรมดาล่าสุดได้โดยไม่ต้องเชื่อมต่อเข้ากับเซสชัน

เทอร์มินัลยังพร้อมใช้งานเป็นเอกสารแบบเต็มหน้าจอที่มีเฉพาะเทอร์มินัลที่ `/?view=terminal` แอป iOS และ Android จะฝังหน้านี้ในหน้าจอ Terminal โดยใช้ข้อมูลประจำตัวของ Gateway ที่จัดเก็บไว้ ความพร้อมใช้งานเป็นไปตามเงื่อนไข `gateway.terminal.enabled` และ `operator.admin` เดียวกัน และหน้าจะแสดงประกาศเมื่อ Gateway ที่เชื่อมต่อไม่ให้บริการเทอร์มินัล

## แผงเบราว์เซอร์

Control UI มาพร้อมแผงเบราว์เซอร์ที่ด็อกได้ ซึ่งแสดงผลเบราว์เซอร์ที่ควบคุมโดย Gateway (ตัวเดียวกับที่เอเจนต์ควบคุมผ่าน [เครื่องมือเบราว์เซอร์](/th/tools/browser-control)) ในเว็บเบราว์เซอร์ทั่วไปได้โดยไม่ต้องใช้ webview แบบเนทีฟ แผงนี้จะปรากฏเมื่อ Gateway ที่เชื่อมต่อประกาศ `browser.request` ไปยังการเชื่อมต่อ `operator.admin`; ปุ่มรูปลูกโลกในรางพื้นที่ทำงานของเธรดใช้สลับการแสดงแผง แผงจะแสดงภาพสแนปช็อตสดของหน้าพร้อมแท็บ แถบ URL ที่แก้ไขได้ ปุ่มย้อนกลับ/ไปข้างหน้า/โหลดใหม่ และตัวเลือกเปิดในเบราว์เซอร์ของผู้ใช้ สามารถด็อกด้านขวาหรือด้านล่าง และส่งต่อการคลิก การเลื่อนด้วยล้อเมาส์ และการพิมพ์พื้นฐานไปยังหน้าระยะไกล

โหมดการจับภาพสองโหมดจะรวบรวมบริบทของหน้าให้เอเจนต์:

- **ใส่คำอธิบายประกอบ (ดินสอ)**: วาดเครื่องหมายแบบอิสระทับบนหน้า **ส่งไปยังแชต** จะรวมเส้นที่วาดลงในภาพหน้าจอ แนบรูปภาพเข้ากับช่องเขียนแชตที่ใช้งานอยู่ และกรอกพรอมต์ล่วงหน้าที่อธิบาย URL ของหน้า ชื่อเรื่อง และแต่ละบริเวณที่ทำเครื่องหมายไว้ เพื่อให้เอเจนต์ทราบอย่างแม่นยำว่ามีการวงส่วนใดไว้
- **ตรวจสอบ (ตัวชี้)**: วางเมาส์เหนือเพื่อดูองค์ประกอบใต้เคอร์เซอร์ (ตัวเลือก ชื่อที่เข้าถึงได้ บทบาท ขนาด) คลิกเพื่อส่งรายละเอียดขององค์ประกอบนั้นพร้อมภาพหน้าจอที่ไฮไลต์ผ่านขั้นตอนช่องเขียนเดียวกัน การตรวจสอบ การเลื่อนด้วยล้อเมาส์ และการย้อนกลับ/ไปข้างหน้าต้องใช้ `browser.evaluateEnabled` (เปิดโดยค่าเริ่มต้น)

แอป macOS ยังคงใช้แถบด้านข้างของเบราว์เซอร์ลิงก์แบบเนทีฟสำหรับลิงก์ที่คลิกในแดชบอร์ด แผงเบราว์เซอร์ทำงานในแอปนั้นด้วย และเป็นวิธีใส่คำอธิบายประกอบหน้าเว็บบนแพลตฟอร์มอื่นทั้งหมด

## ลักษณะการทำงานของแชต

<AccordionGroup>
  <Accordion title="ความหมายของการส่งและประวัติ">
    - `chat.send` เป็นแบบ **ไม่บล็อก**: ระบบจะตอบรับทันทีด้วย `{ runId, status: "started" }` และสตรีมการตอบกลับผ่านเหตุการณ์ `chat` ไคลเอนต์ Control UI ที่เชื่อถือได้อาจได้รับข้อมูลเมตาเกี่ยวกับเวลาของ ACK เพิ่มเติมสำหรับการวินิจฉัยภายในเครื่องด้วย
    - การอัปโหลดในแชตรองรับรูปภาพและไฟล์ที่ไม่ใช่วิดีโอ รูปภาพจะใช้เส้นทางรูปภาพดั้งเดิมต่อไป ส่วนไฟล์อื่นจะถูกจัดเก็บเป็นสื่อที่มีการจัดการและแสดงในประวัติเป็นลิงก์ไฟล์แนบ
    - การส่งซ้ำด้วย `idempotencyKey` เดิมจะคืนค่า `{ status: "in_flight" }` ขณะกำลังทำงาน และคืนค่า `{ status: "ok" }` หลังเสร็จสิ้น
    - การตอบกลับของ `chat.history` ถูกจำกัดขนาดเพื่อความปลอดภัยของ UI เมื่อรายการทรานสคริปต์มีขนาดใหญ่เกินไป Gateway อาจตัดทอนฟิลด์ข้อความที่ยาว ละเว้นบล็อกข้อมูลเมตาขนาดใหญ่ และแทนที่ข้อความที่มีขนาดเกินกำหนดด้วยตัวยึดตำแหน่ง (`[chat.history omitted: message too large]`)
    - เมื่อข้อความผู้ช่วยที่มองเห็นถูกตัดทอนใน `chat.history` ตัวอ่านด้านข้างสามารถดึงรายการทรานสคริปต์ฉบับเต็มที่ปรับรูปแบบเพื่อการแสดงผลแล้วได้ตามต้องการผ่าน `chat.message.get` โดยใช้ `sessionKey`, `agentId` ที่ใช้งานอยู่เมื่อจำเป็น และ `messageId` ของทรานสคริปต์ หาก Gateway ยังคงไม่สามารถส่งข้อมูลเพิ่มเติมได้ ตัวอ่านจะแสดงสถานะว่าไม่พร้อมใช้งานอย่างชัดเจน แทนที่จะแสดงตัวอย่างที่ถูกตัดทอนซ้ำโดยไม่แจ้งให้ทราบ
    - รูปภาพจากผู้ช่วยหรือที่สร้างขึ้นจะถูกเก็บถาวรเป็นการอ้างอิงสื่อที่มีการจัดการ และส่งกลับผ่าน URL สื่อของ Gateway ที่มีการยืนยันตัวตน เพื่อให้การโหลดซ้ำไม่ต้องพึ่งพาเพย์โหลดรูปภาพ base64 แบบดิบที่คงอยู่ในการตอบกลับประวัติแชต
    - เมื่อเรนเดอร์ `chat.history` Control UI จะลบแท็กคำสั่งแบบอินไลน์ที่มีไว้สำหรับการแสดงผลเท่านั้นออกจากข้อความผู้ช่วยที่มองเห็น (เช่น `[[reply_to_*]]` และ `[[audio_as_voice]]`) รวมถึงเพย์โหลด XML ของการเรียกเครื่องมือที่เป็นข้อความล้วน (ได้แก่ `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อกการเรียกเครื่องมือที่ถูกตัดทอน) และโทเค็นควบคุมโมเดลแบบ ASCII/ความกว้างเต็มที่รั่วไหลออกมา ระบบจะละเว้นรายการของผู้ช่วยที่ข้อความที่มองเห็นทั้งหมดมีเพียงโทเค็นเงียบที่ตรงกันทุกประการ `NO_REPLY` / `no_reply` หรือโทเค็นตอบรับ Heartbeat `HEARTBEAT_OK`
    - ระหว่างการส่งที่กำลังดำเนินอยู่และการรีเฟรชประวัติครั้งสุดท้าย มุมมองแชตจะคงแสดงข้อความผู้ใช้/ผู้ช่วยเชิงคาดการณ์ในเครื่องไว้ หาก `chat.history` คืนค่าสแนปช็อตที่เก่ากว่าเป็นเวลาสั้น ๆ ทรานสคริปต์มาตรฐานจะแทนที่ข้อความในเครื่องเหล่านั้นเมื่อประวัติของ Gateway ตามทัน
    - เหตุการณ์ `chat` แบบสดเป็นสถานะการส่งมอบ ส่วน `chat.history` จะถูกสร้างใหม่จากทรานสคริปต์เซสชันที่คงทน หลังเหตุการณ์สิ้นสุดของเครื่องมือ Control UI จะโหลดประวัติใหม่และผสานเฉพาะส่วนท้ายเชิงคาดการณ์ขนาดเล็กเท่านั้น ขอบเขตของทรานสคริปต์มีคำอธิบายไว้ใน [WebChat](/th/web/webchat)
    - `chat.inject` จะเพิ่มบันทึกของผู้ช่วยต่อท้ายทรานสคริปต์เซสชันและกระจายเหตุการณ์ `chat` สำหรับการอัปเดตเฉพาะ UI (ไม่มีการรันเอเจนต์และไม่มีการส่งผ่านช่องทาง)
    - แถบด้านข้างแสดงรายการเซสชันที่ใช้งานอยู่และโหลดแล้วทั้งหมด โดยแบ่งตามส่วนของเอเจนต์และกลุ่มปักหมุด/ช่องทาง/งาน/กำหนดเอง/แชต พร้อมการดำเนินการ New Session เพียงรายการเดียวซึ่งเปิดกล่องโต้ตอบฉบับร่าง การเปิดแถวที่มองเห็นจะย้ายเฉพาะแถบไฮไลต์ สามารถลากเซสชันไปวางที่ Pinned เพื่อปักหมุด หรือลากไปยังกลุ่มกำหนดเองหรือ Chats เพื่อย้ายได้ กลุ่มกำหนดเองสามารถยุบและจัดลำดับใหม่ด้วยการลาก ชื่อและลำดับกลุ่มจะซิงค์ผ่าน Gateway และสถานะยุบจะคงอยู่ในเบราว์เซอร์ เซสชันแดชบอร์ดใหม่จะได้รับชื่อเรื่องสั้น ๆ ที่สร้างแบบอะซิงโครนัสจากข้อความแรกที่ไม่ใช่คำสั่ง ชื่อที่กำหนดไว้อย่างชัดเจนจะไม่ถูกแทนที่ ตั้งค่า `agents.defaults.utilityModel` (หรือ `agents.list[].utilityModel`) เพื่อกำหนดเส้นทางการเรียกโมเดลแยกนี้ไปยังโมเดลที่มีต้นทุนต่ำกว่า การขยายส่วนของเอเจนต์อื่นช่วยให้เรียกดูเซสชันของเอเจนต์นั้นได้โดยไม่ต้องออกจากแชตที่เปิดอยู่
    - การค้นหาเธรดอยู่ในแผงคำสั่ง (⌘K หรือช่อง Search ที่ด้านบนของแถบด้านข้าง): เมื่อพิมพ์คำค้น ระบบจะค้นหาหน้าผลลัพธ์ที่ตรงกันในจำนวนที่จำกัดทั่วทุกเอเจนต์ กรองแถวลูก/แถว Cron ภายใน และแสดงผลลัพธ์ที่มองเห็นถัดจากคำสั่งนำทาง หน้า Threads ยังคงแสดงรายการทั้งหมดที่ค้นหาได้พร้อมตัวกรอง
    - แต่ละแถวในแถบด้านข้างมีการเข้าถึงการปักหมุดโดยตรง รวมถึงเมนูบริบทแบบเต็มสำหรับสถานะยังไม่ได้อ่าน การเปลี่ยนชื่อ การแยกสาขา การจัดกลุ่ม การเก็บถาวร และการลบ แถวที่เลือกหลายรายการ (Cmd/Ctrl-click และ Shift-click สำหรับเลือกช่วง) จะมีเมนูแบบกลุ่มที่ครอบคลุมสถานะยังไม่ได้อ่าน การจัดกลุ่ม การเก็บถาวร และการลบ การเก็บถาวร/ลบแบบกลุ่มจะยังคงถูกปิดใช้งาน เว้นแต่ทุกเซสชันที่เลือกจะสามารถเก็บถาวรได้ ไม่สามารถเก็บถาวรการรันที่กำลังทำงานและเซสชันหลักของเอเจนต์ได้ การเก็บถาวรหรือลบเซสชันที่เลือกอยู่ในปัจจุบันจะสลับ Chat กลับไปยังเซสชันหลักของเอเจนต์นั้น
    - ในแอป macOS เครื่องหมาย OpenClaw จะใช้แถบว่างของแถบชื่อเรื่องแบบเนทีฟข้างตัวควบคุมหน้าต่าง แทนที่จะใช้พื้นที่หนึ่งแถวในแถบด้านข้าง
    - บนเดสก์ท็อป ตัวควบคุมแชตจะอยู่ในแถวกะทัดรัดเพียงแถวเดียวและยุบลงขณะเลื่อนทรานสคริปต์ลง การเลื่อนขึ้น กลับไปด้านบน หรือเลื่อนไปถึงด้านล่างจะคืนค่าตัวควบคุม
    - ข้อความที่มีเฉพาะข้อความและซ้ำกันอย่างต่อเนื่องจะแสดงเป็นบับเบิลเดียวพร้อมป้ายจำนวน ข้อความที่มีรูปภาพ ไฟล์แนบ เอาต์พุตเครื่องมือ หรือตัวอย่าง Canvas จะไม่ถูกยุบรวม
    - บับเบิลข้อความผู้ใช้มีการดำเนินการกับทรานสคริปต์ ได้แก่ ปุ่มย้อนกลับเมื่อวางเมาส์เหนือ (ป๊อปโอเวอร์ยืนยันพร้อมตัวเลือก "Don't ask again") รวมถึง **ย้อนกลับมาที่นี่** และ **แยกสาขาจากที่นี่** ในเมนูคลิกขวา การย้อนกลับจะชี้เซสชันไปยังสถานะก่อนข้อความนั้น และนำข้อความกลับไปยังตัวเขียนเพื่อแก้ไขและส่งใหม่ (`sessions.rewind`, `operator.admin`) ส่วนการแยกสาขาจะสร้างเซสชันใหม่จากคำนำหน้าของเส้นทางที่ใช้งานอยู่ก่อนข้อความ เปิดเซสชันนั้น และใส่ข้อความเดียวกันไว้ในตัวเขียน (`sessions.fork`, `operator.write`) การดำเนินการทั้งสองจะถูกปิดใช้งานพร้อมคำอธิบายในทูลทิปขณะที่เอเจนต์กำลังทำงาน ใช้ได้เฉพาะกับข้อความผู้ใช้ที่เก็บถาวรแล้ว และจะถูกปฏิเสธสำหรับเซสชันที่บทสนทนาอยู่ภายใต้การดูแลของชุดเครื่องมือเอเจนต์ภายนอก การย้อนกลับจะย้ายเฉพาะบริบทแชต โดยไฟล์และผลข้างเคียงอื่นจากเครื่องมือจะไม่ถูกย้อนคืน และทรานสคริปต์ก่อนการย้อนกลับจะยังคงถูกเก็บรักษาในที่เก็บเซสชันแบบเพิ่มต่อท้ายเท่านั้น เมื่อที่เก็บดังกล่าวมีทรานสคริปต์หลายสาขา แถบชื่อเรื่องแชตจะแสดงเมนูสาขาพร้อมข้อความล่าสุด จำนวนข้อความ และความใหม่ล่าสุดของแต่ละสาขา การเลือกสาขาที่ไม่ได้ใช้งานจะสลับเซสชันปัจจุบันกลับไปยังเส้นทางที่เก็บรักษาไว้นั้น (`sessions.branches.list`, `operator.read`; `sessions.branches.switch`, `operator.admin`) การสลับสาขาจะไม่พร้อมใช้งานขณะที่เอเจนต์กำลังทำงานเช่นกัน และการเลือกสาขาที่ใช้งานอยู่แล้วจะเป็นข้อผิดพลาดแบบ no-op ที่ระบุชนิดไว้ ณ ขอบเขต RPC การดำเนินการซ่อนแยกต่างหากบนบับเบิลผู้ใช้จะซ่อนข้อความเฉพาะในเบราว์เซอร์ปัจจุบันเท่านั้น ข้อความยังคงอยู่ในทรานสคริปต์และเอเจนต์ยังคงมองเห็น
    - เมื่อเช็กเอาต์ของเซสชันอยู่บนสาขาที่ไม่ใช่ค่าเริ่มต้นของรีโพซิทอรี GitHub มุมมองแชตจะปักหมุดชิป pull request เหนือตัวเขียน ได้แก่ หมายเลข PR, รีโพซิทอรี, สาขา, จำนวนส่วนต่าง, ป้าย CI และสถานะฉบับร่าง/ผสานแล้ว/ปิด โดยแต่ละรายการเชื่อมโยงไปยัง PR แถวจะแสดงชิปได้สูงสุดสองรายการ โดยแสดง PR แบบสด (เปิด/ฉบับร่าง) ก่อน และปุ่ม "Show more" จะแสดงประวัติที่ผสานแล้ว/ปิดซึ่งถูกยุบไว้ ป้าย CI จะเปิดป๊อปโอเวอร์ตรวจสอบ CI ขนาดเล็กซึ่งแสดงจำนวนการตรวจสอบที่ผ่าน/ล้มเหลว/กำลังทำงาน/ข้าม และลิงก์ไปยังหน้าการตรวจสอบของ PR การตรวจจับทำงานฝั่งเซิร์ฟเวอร์ผ่าน `controlUi.sessionPullRequests` ซึ่งนำ `GH_TOKEN`/`GITHUB_TOKEN` ของ Gateway มาใช้ซ้ำเมื่อตั้งค่าไว้ เมื่อถึงขีดจำกัดอัตราของ GitHub API ชิปจะคงสถานะล่าสุดที่ทราบไว้และแสดงคำเตือนว่าสถานะอาจล้าสมัย การปิดชิปจะซ่อนชิปนั้นสำหรับเซสชันดังกล่าวในโปรไฟล์เบราว์เซอร์ปัจจุบัน ก่อนที่จะมี PR แถวจะแสดงตัวสาขาเอง ได้แก่ รีโพซิทอรี ชื่อสาขา และขนาด +/− ของส่วนต่างเทียบกับฐานการผสานของสาขาเริ่มต้น (รวมงานที่คอมมิตและยังไม่ได้คอมมิต) เมื่อสาขาที่พุชมีคอมมิตให้เปรียบเทียบแล้ว แถวจะเพิ่มปุ่ม Create PR ซึ่งเปิดหน้าสร้าง pull request ใหม่ของ GitHub ก่อนหน้านั้น เซสชันที่มีไฟล์เปลี่ยนแปลง (คอมมิตแล้ว ยังไม่ได้คอมมิต หรือยังไม่ได้ติดตาม) จะยังคงมีแถวนี้แต่ไม่มีปุ่ม แถวจะซ่อนตัวเองเมื่อมี PR ที่เปิดอยู่หรือเป็นฉบับร่าง แถวสาขามาจาก git ในเครื่องเท่านั้น จึงยังคงพร้อมใช้งานขณะที่ GitHub ถูกจำกัดอัตรา และแสดงคำเตือนสถานะล้าสมัยเดียวกัน เนื่องจากยังเชื่อถือผล "ไม่พบ PR" ไม่ได้จนกว่าขีดจำกัดจะถูกรีเซ็ต
    - แผงส่วนต่างของเซสชันแสดงสิ่งที่เช็กเอาต์ของเซสชันเปลี่ยนแปลงจริง: ปุ่มสาขาในแถบพื้นที่ทำงานหรือแถบชื่อเรื่องแชตจะเปิดแผงรายละเอียดที่แสดงส่วนต่างแยกตามไฟล์ของงานบนสาขา งานที่ยังไม่ได้คอมมิต และงานที่ยังไม่ได้ติดตาม เทียบกับฐานการผสานของสาขาเริ่มต้นของเช็กเอาต์ โดยมีจุดสถานะ ลูกศรเปลี่ยนชื่อ จำนวน +/− ต่อไฟล์ ไฟล์ที่ยุบได้ และเครื่องหมาย "N บรรทัดที่ไม่เปลี่ยนแปลง" ระหว่างฮังก์ ส่วนต่างคำนวณฝั่งเซิร์ฟเวอร์ผ่านเมธอด Gateway `sessions.diff` (ขอบเขต `operator.read`) ไฟล์ไบนารีและไฟล์ที่มีขนาดเกินกำหนดจะลดรูปเหลือรายการสถิติเท่านั้น และปุ่มจะปรากฏเฉพาะเมื่อ Gateway ที่เชื่อมต่อประกาศ `sessions.diff`
    - ทุกบานหน้าต่าง Chat มีแถบชื่อเรื่อง คลิกชื่อเซสชันเพื่อเปลี่ยนชื่อ ชิปพื้นที่ทำงานจะคัดลอกเส้นทางหรือสาขาของเช็กเอาต์ และสามารถแสดงพื้นที่ทำงานของ Gateway ภายในเครื่องในตัวจัดการไฟล์ของโฮสต์ได้ เซสชันระยะไกลและ exec-node ยังคงมีการดำเนินการคัดลอก แต่ซ่อนการแสดงในตัวจัดการไฟล์
    - แถบพื้นที่ทำงานของเธรดในแต่ละบานหน้าต่าง Chat แสดงรายการไฟล์เธรด ไฟล์โปรเจกต์ และอาร์ติแฟกต์ โดยค่าเริ่มต้นจะเทียบชิดขอบขวาของบานหน้าต่าง ลากส่วนหัวของแถบ (หรือใช้ปุ่มเทียบชิด) เพื่อย้ายไปด้านล่าง และตัวเลือกนี้จะถูกจัดเก็บในโปรไฟล์เบราว์เซอร์ปัจจุบัน แถบที่ยุบจะไม่กินพื้นที่เลย โดยเปิดใหม่ได้ด้วย ⇧⌘B หรือปุ่มสลับไฟล์ในแถบชื่อเรื่อง ซึ่งมีป้ายจำนวนไฟล์ที่เปลี่ยนแปลง แผงรายละเอียดไฟล์ เครื่องมือ และ Canvas ที่แยกต่างหากจะไม่ได้รับผลกระทบ
    - การคลิกการอ้างอิงไฟล์ในแชต เส้นทางไฟล์ในการ์ดเครื่องมืออ่าน/แก้ไข/เขียนที่ขยายอยู่ หรือแถวไฟล์ในแถบพื้นที่ทำงาน จะเปิดแผงรายละเอียดไฟล์: มุมมองโค้ดที่ใช้ CodeMirror พร้อมการไฮไลต์ไวยากรณ์ หมายเลขบรรทัด การข้ามไปยังบรรทัด การค้นหาภายในไฟล์ การดำเนินการคัดลอก และเมนูเปิดในโปรแกรมแก้ไขภายนอก เมื่อ Gateway ประกาศ `sessions.files.set` ให้กับการเชื่อมต่อ `operator.admin` แผงจะเพิ่มโหมด Edit พร้อมการติดตามสถานะที่แก้ไขและการบันทึกด้วย Cmd/Ctrl-S ฉบับร่างที่ยังไม่ได้บันทึกจะยังคงอยู่เมื่อมีการนำทางระหว่างไฟล์ แผง และเซสชันภายในแท็บเบราว์เซอร์ปัจจุบัน จนกว่าจะบันทึกหรือยกเลิกอย่างชัดเจน การบันทึกใช้การเปรียบเทียบและสลับตามแฮชเนื้อหาที่ `sessions.files.get` คืนค่า: หากไฟล์บนดิสก์เปลี่ยนไปหลังจากโหลดแล้ว (เช่น เพราะเอเจนต์ยังคงทำงานต่อ) แผงจะแสดงการแจ้งเตือนข้อขัดแย้งพร้อมการดำเนินการ Reload (ใช้เนื้อหาล่าสุด) และ Overwrite (คงการแก้ไขในเครื่องไว้) การเขียนใช้ตัวป้องกันพื้นที่ทำงานที่ปลอดภัยต่อระบบไฟล์ชุดเดียวกับการอ่าน ได้แก่ การจำกัดให้อยู่ภายในเส้นทาง การปฏิเสธ symlink/hardlink และขีดจำกัด UTF-8 ที่ 256 KB และจะเขียนทับเฉพาะไฟล์ที่มีอยู่เท่านั้น โปรแกรมแก้ไขจะไม่สร้างหรือลบไฟล์
    - แถบงานเบื้องหลังในแต่ละบานหน้าต่าง Chat แสดงรายการงานเบื้องหลังและเอเจนต์ย่อยของเอเจนต์ปัจจุบัน (`tasks.list` ที่กำหนดขอบเขตตามเอเจนต์และอัปเดตแบบสดด้วยเหตุการณ์ `task`): งานที่กำลังทำจะแสดงตัวจับเวลาที่ผ่านไปแบบสด จำนวนการใช้เครื่องมือ เครื่องมือที่กำลังใช้อยู่ และตัวควบคุมหยุด ส่วนที่เสร็จสิ้นซึ่งยุบได้จะเพิ่มระยะเวลาการรัน และลิงก์ View transcript จะเปิดเซสชันลูกของงานในบานหน้าต่าง เปิดด้วยปุ่มสลับกิจกรรมในแถบชื่อเรื่อง สแนปช็อตงานจะโหลดล่วงหน้าทันที จึงแสดงป้ายจำนวนงานที่กำลังทำโดยไม่ต้องเปิดแถบก่อน หน้า Tasks ยังคงเป็นบัญชีรายการแบบเต็มที่ครอบคลุมทุกเอเจนต์

    - เอเจนต์ที่มีเครื่องมือ `screen` สามารถขอเปลี่ยนแปลงบานหน้าต่าง แถบด้านข้าง เทอร์มินัล เบราว์เซอร์ โฟกัส และการนำทางแบบเดียวกันได้ ขณะที่เชื่อมต่อ Control UI ที่รองรับอยู่ โปรโตคอล v1 จะใช้คำสั่งกับ Control UI ที่เชื่อมต่อและรองรับทุกตัว โปรดดู [หน้าจอ](/th/tools/screen)
    - ลากเซสชันจากแถบด้านข้างมายังแชตเพื่อเปิดในบานหน้าต่าง ตัวอย่างตำแหน่งวางแบบเคลื่อนไหวจะเลื่อนไประหว่างโซนต่าง ๆ และระบุผลลัพธ์ โดยแสดง "แยก" เหนือครึ่งพื้นที่ที่บานหน้าต่างใหม่จะใช้พอดี และ "เปิดที่นี่" เหนือบานหน้าต่างทั้งบาน ทั้งยังวางได้จากโหมดบานหน้าต่างเดียว
    - บานหน้าต่างแยกที่ใช้งานอยู่จะกำหนดรายการที่เลือกในแถบด้านข้างและ URL แถบชื่อของบานหน้าต่างมีตัวควบคุมสำหรับแยกและปิด ตัวแบ่งใช้ปรับขนาดคอลัมน์และบานหน้าต่างที่เรียงซ้อนกันได้ และเบราว์เซอร์จะจัดเก็บเค้าโครงไว้ภายในเครื่องข้ามการโหลดหน้าใหม่
    - บนหน้าจอแคบ มุมมองแบบแยกจะคงเค้าโครงไว้แต่แสดงผลเฉพาะบานหน้าต่างที่ใช้งานอยู่ รวมถึงส่วนหัวที่มีตัวควบคุมปิด
    - หากส่งข้อความขณะที่การเปลี่ยนตัวเลือกโมเดลของเซสชันเดียวกันยังอยู่ระหว่างการบันทึก ช่องเขียนข้อความจะรอให้การแพตช์เซสชันนั้นเสร็จก่อนเรียก `chat.send` เพื่อให้การส่งใช้โมเดลที่เลือก
    - การพิมพ์ `/new` จะสร้างและสลับไปยังเซสชันแดชบอร์ดใหม่แบบเดียวกับแชตใหม่ ยกเว้นเมื่อกำหนดค่า `session.dmScope: "main"` ไว้และพาเรนต์ปัจจุบันเป็นเซสชันหลักของเอเจนต์ ซึ่งในกรณีนั้นระบบจะรีเซ็ตเซสชันหลักโดยไม่สร้างใหม่ การพิมพ์ `/reset` จะคงการรีเซ็ตเซสชันปัจจุบันโดยไม่สร้างใหม่ตามที่ Gateway ระบุไว้อย่างชัดเจน
    - ตัวเลือกโมเดลของแชตจะขอมุมมองโมเดลที่กำหนดค่าไว้ของ Gateway หาก `agents.defaults.modelPolicy.allow` ไม่ว่าง นโยบายนั้นจะกำหนดตัวเลือก รวมถึงรายการ `provider/*` ที่ทำให้แค็ตตาล็อกตามขอบเขตผู้ให้บริการยังเป็นแบบไดนามิก มิฉะนั้น ตัวเลือกจะแสดงรายการที่กำหนดค่าไว้ร่วมกับผู้ให้บริการที่มีการรับรองความถูกต้องที่ใช้งานได้ นามแฝงและการตั้งค่าภายใต้ `agents.defaults.models` จะไม่จำกัดตัวเลือกดังกล่าว แค็ตตาล็อกฉบับเต็มยังคงเข้าถึงได้ผ่าน RPC สำหรับดีบัก `models.list` ด้วย `view: "all"`
    - เมื่อรายงานการใช้งานเซสชันล่าสุดจาก Gateway มีจำนวนโทเค็นบริบทปัจจุบัน แถบเครื่องมือของช่องเขียนข้อความแชตจะแสดงวงแหวนการใช้งานบริบทขนาดเล็กพร้อมเปอร์เซ็นต์ที่ใช้ เปิดวงแหวนเพื่อดูหน้าต่างบริบทปัจจุบัน จำนวนโทเค็นของการทำงานล่าสุดและค่าใช้จ่ายรวมโดยประมาณ ข้อมูลระบุผู้ให้บริการ/โมเดล และรายละเอียดค่าใช้จ่ายด้านอินพุต/เอาต์พุต/แคชของการตอบกลับล่าสุดจากผู้ให้บริการเมื่อมีการรายงาน วงแหวนจะเปลี่ยนเป็นรูปแบบคำเตือนเมื่อบริบทใกล้เต็มในระดับสูง และเมื่อถึงระดับที่แนะนำให้ทำ Compaction จะแสดงปุ่มขนาดกะทัดรัดสำหรับเรียกใช้เส้นทาง Compaction ตามปกติของเซสชัน ภาพรวมโทเค็นที่ล้าสมัยจะถูกซ่อนไว้จนกว่า Gateway จะรายงานการใช้งานล่าสุดอีกครั้ง

  </Accordion>
  <Accordion title="โหมดสนทนา (เรียลไทม์ผ่านเบราว์เซอร์)">
    โหมดสนทนาใช้ผู้ให้บริการเสียงแบบเรียลไทม์ที่ลงทะเบียนไว้ กำหนดค่า OpenAI ด้วย `talk.realtime.provider: "openai"` พร้อมโปรไฟล์คีย์ API `openai`, `talk.realtime.providers.openai.apiKey` หรือ `OPENAI_API_KEY` OpenAI Realtime ใช้ Platform API สาธารณะและต้องใช้คีย์ Platform API โดยการเข้าสู่ระบบด้วย Codex OAuth ไม่สามารถใช้กับส่วนนี้ได้ กำหนดค่า Google ด้วย `talk.realtime.provider: "google"` พร้อม `talk.realtime.providers.google.apiKey` เบราว์เซอร์จะไม่ได้รับคีย์ API มาตรฐานของผู้ให้บริการ: OpenAI จะได้รับข้อมูลลับไคลเอนต์ Realtime แบบชั่วคราวสำหรับ WebRTC และ Google Live จะได้รับโทเค็นการตรวจสอบสิทธิ์ Live API แบบจำกัดที่ใช้ได้ครั้งเดียวสำหรับเซสชัน WebSocket ของเบราว์เซอร์ โดย Gateway จะล็อกคำสั่งและการประกาศเครื่องมือไว้ในโทเค็น ผู้ให้บริการที่เปิดให้ใช้เฉพาะบริดจ์เรียลไทม์ฝั่งแบ็กเอนด์จะทำงานผ่านการส่งต่อของ Gateway ดังนั้นข้อมูลประจำตัวและซ็อกเก็ตของผู้ให้บริการจึงอยู่ฝั่งเซิร์ฟเวอร์ ขณะที่เสียงจากเบราว์เซอร์ส่งผ่าน RPC ของ Gateway ที่ผ่านการตรวจสอบสิทธิ์ พรอมต์เซสชัน Realtime จะประกอบขึ้นโดย Gateway และ `talk.client.create` ไม่ยอมรับการแทนที่คำสั่งที่ผู้เรียกกำหนดเอง

    ค่าเริ่มต้นถาวรสำหรับผู้ให้บริการ โมเดล เสียง การรับส่งข้อมูล ระดับการใช้เหตุผล เกณฑ์ VAD ที่แน่นอน ระยะเวลาความเงียบ และระยะเติมนำหน้า อยู่ใน **Settings → Communications → Talk** การเปลี่ยนค่าเหล่านี้ต้องมีสิทธิ์เข้าถึง `operator.admin` การกำหนดค่าการส่งต่อของ Gateway จะบังคับใช้เส้นทางส่งต่อฝั่งแบ็กเอนด์ ส่วนการกำหนดค่า WebRTC จะคงให้ไคลเอนต์เป็นเจ้าของเซสชัน และจะล้มเหลวแทนการถอยกลับไปใช้การส่งต่อโดยไม่แจ้ง หากผู้ให้บริการไม่สามารถสร้างเซสชันเบราว์เซอร์ได้

    ตัวควบคุมการสนทนาคือปุ่มไมโครโฟนในแถบเครื่องมือของช่องเขียน เครื่องหมายแคเร็ตจะแสดง **System default** และไมโครโฟนทุกตัวที่เบราว์เซอร์ตรวจพบ รวมถึงอินพุต USB, Bluetooth และอินพุตเสมือน รหัสอุปกรณ์ที่เลือกจะเก็บไว้เฉพาะในเบราว์เซอร์และไม่ถูกส่งไปยัง Gateway หากอุปกรณ์นั้นหายไป โหมดสนทนาจะขอให้เลือกอินพุตอื่น แทนที่จะบันทึกเสียงจากไมโครโฟนอื่นโดยไม่แจ้ง ขณะที่โหมดสนทนาทำงาน ปุ่มไมโครโฟนจะเปลี่ยนเป็นปุ่มทรงแคปซูลที่แสดงมาตรวัดระดับอินพุตสด การคลิกจะหยุดอินพุตเสียง และการวางเมาส์เหนือปุ่มจะแสดงสัญลักษณ์หยุด โปรแกรมอ่านหน้าจอจะประกาศ `Connecting voice input...`, `Listening...` หรือ `Asking OpenClaw...` ขณะที่การเรียกเครื่องมือแบบเรียลไทม์กำลังปรึกษาโมเดลขนาดใหญ่กว่าที่กำหนดค่าผ่าน `talk.client.toolCall` การหยุดการตอบกลับของเอเจนต์ที่กำลังทำงานยังคงใช้ตัวควบคุมสี่เหลี่ยม **Stop** แยกต่างหากถัดจากปุ่มทรงแคปซูล

    **การสนทนาด้วยวิดีโอ** ใช้ได้กับเซสชัน OpenAI Realtime WebRTC และ Google Live ผ่านเบราว์เซอร์ คลิกปุ่มกล้อง อนุญาตการเข้าถึงกล้องและไมโครโฟน แล้วตรวจสอบภาพตัวอย่างในเครื่อง OpenAI จะส่งเฟรม JPEG ที่จำกัดขนาดหนึ่งเฟรมผ่านช่องข้อมูลของเบราว์เซอร์เมื่อ `describe_view` ร้องขอบริบทภาพ Google Live จะส่งเฟรม JPEG ที่จำกัดขนาดจากเบราว์เซอร์ไปยังผู้ให้บริการโดยตรง ด้วยอัตราสูงสุดที่รองรับคือหนึ่งเฟรมต่อวินาที และตอบการเรียกฟังก์ชัน `describe_view` ด้วยสถานะสตรีมกล้อง เฟรมจากกล้องจะไม่ผ่าน Gateway การหยุดโหมดสนทนาจะปิดภาพตัวอย่างและปล่อยแทร็กสื่อทั้งสอง ดูสัญญาการรับส่งข้อมูลของผู้ให้บริการได้ที่ [ความสามารถของ Live API](https://ai.google.dev/gemini-api/docs/live-api/capabilities#video) และ [คู่มือการเรียกฟังก์ชัน](https://ai.google.dev/gemini-api/docs/live-api/tools) ของ Google

    การทดสอบควันแบบสดสำหรับผู้ดูแล: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` ตรวจสอบบริดจ์ WebSocket ฝั่งแบ็กเอนด์ของ OpenAI, การแลกเปลี่ยน SDP ของ OpenAI WebRTC ผ่านเบราว์เซอร์, การตั้งค่าเบราว์เซอร์ด้วยโทเค็นแบบจำกัดของ Google Live พร้อมเฟรม JPEG และการรับส่งฟังก์ชัน `describe_view` แบบไปกลับ รวมถึงอะแดปเตอร์เบราว์เซอร์สำหรับการส่งต่อผ่าน Gateway ด้วยสื่อไมโครโฟนจำลอง คำสั่งนี้แสดงเฉพาะสถานะผู้ให้บริการและไม่บันทึกข้อมูลลับ

## การเชื่อมต่อขาดหายและการเชื่อมต่อใหม่

เมื่อสร้างเซสชันแล้ว การเชื่อมต่อ Gateway ที่ขาดหายจะไม่ทำให้คุณออกจากระบบ แดชบอร์ดยังคงแสดงอยู่ โดยมีปุ่มทรงแคปซูลสีเหลืองอำพันลอยอยู่ใต้แถบด้านบนพร้อมข้อความ "การเชื่อมต่อ Gateway ขาดหาย — กำลังเชื่อมต่อใหม่…" ขณะที่ไคลเอนต์ลองใหม่โดยอัตโนมัติด้วยการหน่วงเวลาแบบเพิ่มขึ้น (800 ms สูงสุด 15 s) การอัปเดตสดและการดำเนินการแบบเรียลไทม์/เซสชันจะหยุดชั่วคราวจนกว่าการเชื่อมต่อจะกลับมา **Retry now** ในปุ่มทรงแคปซูลจะบังคับให้ลองเชื่อมต่อทันที แชตยังคงแก้ไขได้: การส่งข้อความธรรมดาและไฟล์แนบจะถูกเก็บไว้ในพื้นที่จัดเก็บของเบราว์เซอร์ในแท็บปัจจุบันที่กำหนดขอบเขตตาม Gateway/เซสชัน แสดงสถานะว่ากำลังรอเชื่อมต่อใหม่ และส่งโดยอัตโนมัติเมื่อ Gateway กลับมา ตัวควบคุมแบบสดและคำสั่งแบบสแลชจะยังใช้ไม่ได้ขณะออฟไลน์

เมื่อเบราว์เซอร์นี้มีข้อมูลประจำตัวอยู่แล้ว (โทเค็น/รหัสผ่านที่กำหนดค่าไว้ หรือโทเค็นอุปกรณ์ที่ได้รับอนุมัติ) การเปิดครั้งแรกและการโหลดซ้ำจะแสดงเครื่องหมาย OpenClaw แบบเคลื่อนไหวขนาดเล็กขณะกำลังเชื่อมต่อ แทนการกะพริบแสดงหน้าล็อกอิน หน้าล็อกอินจะปรากฏเฉพาะเมื่อยังไม่มีข้อมูลประจำตัวจัดเก็บไว้ หรือเมื่อ Gateway ปฏิเสธข้อมูลดังกล่าวโดยตรง (โทเค็น/รหัสผ่านไม่ถูกต้อง หรือการจับคู่ถูกเพิกถอน) ซึ่งเป็นสถานะที่ต้องการข้อมูลจากคุณแทนการรอ

## การติดตั้ง PWA และ Web Push

Control UI มาพร้อม `manifest.webmanifest` และ service worker ดังนั้นเบราว์เซอร์สมัยใหม่จึงติดตั้งเป็น PWA แบบสแตนด์อโลนได้ Web Push ช่วยให้ Gateway ปลุก PWA ที่ติดตั้งไว้ด้วยการแจ้งเตือนได้ แม้ไม่ได้เปิดแท็บหรือหน้าต่างเบราว์เซอร์อยู่

ภายในแอป macOS หน้าการตั้งค่าการแจ้งเตือนจะแสดงสิทธิ์การแจ้งเตือนแบบเนทีฟของแอปแทนการพุชผ่านเบราว์เซอร์ เนื่องจากแอปส่งการแจ้งเตือนแบบเนทีฟ

หากหน้าแสดง **Protocol mismatch** ทันทีหลังอัปเดต OpenClaw ให้เปิดแดชบอร์ดอีกครั้งด้วย `openclaw dashboard` และบังคับรีเฟรชก่อน หากยังล้มเหลว ให้ล้างข้อมูลไซต์สำหรับต้นทางของแดชบอร์ด หรือทดสอบในหน้าต่างเบราว์เซอร์แบบส่วนตัว แท็บเก่าหรือแคช service worker ของเบราว์เซอร์อาจยังคงเรียกใช้บันเดิล Control UI ก่อนการอัปเดตร่วมกับ Gateway เวอร์ชันใหม่กว่า

| ส่วนติดต่อ                                         | หน้าที่                                                                       |
| -------------------------------------------------- | ---------------------------------------------------------------------------- |
| `ui/public/manifest.webmanifest`                   | แมนิเฟสต์ PWA เบราว์เซอร์จะแสดงตัวเลือก "Install app" เมื่อเข้าถึงได้แล้ว             |
| `ui/public/sw.js`                                  | Service worker ที่จัดการเหตุการณ์ `push` และการคลิกการแจ้งเตือน           |
| `state/openclaw.sqlite` → `web_push_vapid_keys`    | คู่คีย์ VAPID ที่สร้างอัตโนมัติสำหรับลงนามเพย์โหลด Web Push                 |
| `state/openclaw.sqlite` → `web_push_subscriptions` | ปลายทางการสมัครรับข้อมูลของเบราว์เซอร์ คีย์ และเวลาประทับการลงทะเบียนที่บันทึกไว้อย่างถาวร |

การอัปเกรดจากที่จัดเก็บ `push/vapid-keys.json` และ `push/web-push-subscriptions.json` ที่เลิกใช้แล้วจะถูกนำเข้าโดย `openclaw doctor --fix` ให้หยุด Gateway ก่อนเรียกใช้การซ่อมแซม เพื่อไม่ให้กระบวนการเก่าสร้างสถานะที่เลิกใช้แล้วขึ้นใหม่ระหว่างการนำเข้า เรียกใช้การซ่อมแซมก่อนใช้ Web Push หลังอัปเกรด การลงทะเบียน การส่ง การลบ และการแก้ไขคีย์จะไม่ดำเนินการตราบใดที่ยังมีแหล่งข้อมูลที่เลิกใช้แล้วหรือการอ้างสิทธิ์ของ Doctor ที่ถูกขัดจังหวะอยู่ รันไทม์ Gateway อ่านและเขียนเฉพาะ SQLite

แทนที่คู่คีย์ VAPID ผ่านตัวแปรสภาพแวดล้อมในกระบวนการ Gateway เมื่อต้องการตรึงคีย์ (การปรับใช้หลายโฮสต์ การหมุนเวียนข้อมูลลับ หรือการทดสอบ):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (ค่าเริ่มต้นคือ `https://openclaw.ai`)

Control UI ใช้เมธอด Gateway ที่จำกัดตามขอบเขตต่อไปนี้ เพื่อลงทะเบียนและทดสอบการสมัครรับข้อมูลของเบราว์เซอร์:

- `push.web.vapidPublicKey` ดึงคีย์สาธารณะ VAPID ที่ใช้งานอยู่
- `push.web.subscribe` ลงทะเบียน `endpoint` พร้อม `keys.p256dh`/`keys.auth`
- `push.web.unsubscribe` ลบปลายทางที่ลงทะเบียนไว้
- `push.web.test` ส่งการแจ้งเตือนทดสอบไปยังการสมัครรับข้อมูลของผู้เรียก

<Note>
Web Push เป็นอิสระจากเส้นทางส่งต่อ APNS ของ iOS (ดูการพุชที่รองรับด้วยการส่งต่อได้ที่ [การกำหนดค่า](/th/gateway/configuration)) และเมธอด `push.test` ซึ่งกำหนดเป้าหมายไปยังการจับคู่อุปกรณ์เคลื่อนที่แบบเนทีฟ
</Note>

## การฝังเนื้อหาแบบโฮสต์

ข้อความของผู้ช่วยสามารถแสดงเนื้อหาเว็บแบบโฮสต์ในบรรทัดด้วยชอร์ตโค้ด `[embed ...]` นโยบายแซนด์บ็อกซ์ของ iframe ควบคุมโดย `gateway.controlUi.embedSandbox`:

เครื่องมือหลัก [`show_widget`](/th/tools/show-widget) แสดงผล SVG หรือ HTML ที่สมบูรณ์ในตัวโดยตรงจากการเรียกเครื่องมือ เบราว์เซอร์และไคลเอนต์แชตแบบเนทีฟที่รองรับจะประกาศความสามารถ Gateway `inline-widgets` และเอกสาร Canvas ที่ได้จะยังคงพร้อมใช้งานเมื่อโหลดประวัติแชตใหม่ Discord Activities มีชื่อเครื่องมือเดียวกันบน Discord ส่วนการทำงานที่เริ่มจากช่องทางอื่นจะไม่ได้รับเครื่องมือนี้

<Tabs>
  <Tab title="เข้มงวด">
    ปิดการเรียกใช้สคริปต์ภายในเนื้อหาแบบฝังที่โฮสต์ไว้
  </Tab>
  <Tab title="สคริปต์ (ค่าเริ่มต้น)">
    อนุญาตเนื้อหาแบบฝังเชิงโต้ตอบโดยยังคงแยกต้นทาง โดยทั่วไปเพียงพอสำหรับเกม/วิดเจ็ตบนเบราว์เซอร์ที่สมบูรณ์ในตัว
  </Tab>
  <Tab title="เชื่อถือได้">
    เพิ่ม `allow-same-origin` เหนือ `allow-scripts` สำหรับเอกสารในไซต์เดียวกันที่ต้องการสิทธิ์ระดับสูงกว่าโดยตั้งใจ
  </Tab>
</Tabs>

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
ใช้ `trusted` เฉพาะเมื่อเอกสารที่ฝังต้องการพฤติกรรมแบบต้นทางเดียวกันอย่างแท้จริง สำหรับเกมและแคนวาสเชิงโต้ตอบส่วนใหญ่ที่เอเจนต์สร้างขึ้น `scripts` เป็นตัวเลือกที่ปลอดภัยกว่า
</Warning>

URL ฝัง `http(s)` ภายนอกแบบสัมบูรณ์จะยังถูกบล็อกตามค่าเริ่มต้น หากต้องการให้ `[embed url="https://..."]` โหลดหน้าของบุคคลที่สาม ให้ตั้งค่า `gateway.controlUi.allowExternalEmbedUrls: true`

## ความกว้างของข้อความแชต

ทรานสคริปต์แชตใช้กรอบที่อ่านง่ายซึ่งจัดกึ่งกลางและวางแนวตรงกับช่องเขียน เอาต์พุตจากผู้ช่วยและเครื่องมือจะชิดซ้าย ส่วนกรอบข้อความของผู้ใช้จะชิดขวาภายในกรอบนั้น การปรับใช้บนจอภาพกว้างสามารถแทนที่ความกว้างของทรานสคริปต์โดยไม่ต้องแก้ไข CSS ที่รวมมา ด้วยการตั้งค่า `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

ค่าจะถูกตรวจสอบก่อนส่งถึงเบราว์เซอร์ รูปแบบที่รองรับ ได้แก่ ความยาวทั่วไปและเปอร์เซ็นต์ เช่น `960px` หรือ `82%` รวมถึงนิพจน์ความกว้าง `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` และ `fit-content(...)` แบบมีข้อจำกัด

## การเข้าถึง Tailnet (แนะนำ)

<Tabs>
  <Tab title="Tailscale Serve แบบผสานรวม (แนะนำ)">
    ให้ Gateway ทำงานบนลูปแบ็ก และให้ Tailscale Serve ทำหน้าที่เป็นพร็อกซีผ่าน HTTPS:

    ```bash
    openclaw gateway --tailscale serve
    ```

    เปิด `https://<magicdns>/` (หรือ `gateway.controlUi.basePath` ที่กำหนดค่าไว้)

    โดยค่าเริ่มต้น คำขอ Control UI/WebSocket Serve สามารถยืนยันตัวตนผ่านส่วนหัวข้อมูลประจำตัว Tailscale (`tailscale-user-login`) เมื่อ `gateway.auth.allowTailscale` เป็น `true` OpenClaw ตรวจสอบข้อมูลประจำตัวโดยแก้ที่อยู่ `x-forwarded-for` ด้วย `tailscale whois` และจับคู่กับส่วนหัว โดยจะยอมรับเฉพาะเมื่อคำขอเข้าถึง loopback พร้อมส่วนหัว `x-forwarded-*` ของ Tailscale เท่านั้น สำหรับเซสชันผู้ดำเนินการ Control UI ที่มีข้อมูลประจำตัวอุปกรณ์ของเบราว์เซอร์ เส้นทาง Serve ที่ผ่านการตรวจสอบนี้จะข้ามขั้นตอนการจับคู่อุปกรณ์ไปกลับด้วย ส่วนเบราว์เซอร์ที่ไม่มีอุปกรณ์และการเชื่อมต่อบทบาท Node ยังคงดำเนินการตรวจสอบอุปกรณ์ตามปกติ ตั้งค่า `gateway.auth.allowTailscale: false` หากต้องการบังคับให้ใช้ข้อมูลประจำตัวแบบความลับร่วมอย่างชัดเจนแม้กับทราฟฟิก Serve จากนั้นใช้ `gateway.auth.mode: "token"` หรือ `"password"`

    สำหรับเส้นทางข้อมูลประจำตัว Serve แบบอะซิงโครนัสดังกล่าว ความพยายามยืนยันตัวตนที่ล้มเหลวสำหรับ IP ไคลเอนต์และขอบเขตการยืนยันตัวตนเดียวกันจะถูกจัดลำดับก่อนเขียนข้อมูลการจำกัดอัตรา ดังนั้นการลองซ้ำที่ไม่ถูกต้องพร้อมกันจากเบราว์เซอร์เดียวกันอาจแสดง `retry later` ในคำขอที่สอง แทนที่จะเกิดข้อผิดพลาดไม่ตรงกันธรรมดาสองรายการที่แข่งขันกันแบบขนาน

    <Warning>
    การยืนยันตัวตน Serve โดยไม่ใช้โทเค็นถือว่าโฮสต์ Gateway เชื่อถือได้ หากอาจมีโค้ดภายในเครื่องที่ไม่น่าเชื่อถือทำงานบนโฮสต์นั้น ให้บังคับใช้การยืนยันตัวตนด้วยโทเค็น/รหัสผ่าน
    </Warning>

  </Tab>
  <Tab title="ผูกกับ tailnet + โทเค็น">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    เปิด `http://<tailscale-ip>:18789/` (หรือ `gateway.controlUi.basePath` ที่กำหนดค่าไว้)

    วางความลับร่วมที่ตรงกันในการตั้งค่า UI (ส่งเป็น `connect.params.auth.token` หรือ `connect.params.auth.password`)

  </Tab>

## HTTP ที่ไม่ปลอดภัย

หากเปิดแดชบอร์ดผ่าน HTTP ธรรมดา (`http://<lan-ip>` หรือ `http://<tailscale-ip>`) เบราว์เซอร์จะทำงานใน **บริบทที่ไม่ปลอดภัย** และบล็อก WebCrypto โดยค่าเริ่มต้น OpenClaw จะ **บล็อก** การเชื่อมต่อ Control UI ที่ไม่มีข้อมูลประจำตัวอุปกรณ์

ข้อยกเว้นที่มีการบันทึกไว้:

- ความเข้ากันได้กับ HTTP ที่ไม่ปลอดภัยเฉพาะ localhost ด้วย `gateway.controlUi.allowInsecureAuth=true`
- การยืนยันตัวตน Control UI ของผู้ดำเนินการที่สำเร็จผ่าน `gateway.auth.mode: "trusted-proxy"`
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` สำหรับกรณีฉุกเฉิน

**วิธีแก้ไขที่แนะนำ:** ใช้ HTTPS (Tailscale Serve) หรือเปิด UI ภายในเครื่องที่ `https://<magicdns>/` (Serve) หรือ `http://127.0.0.1:18789/` (บนโฮสต์ Gateway)

<AccordionGroup>
  <Accordion title="ลักษณะการทำงานของตัวสลับการยืนยันตัวตนที่ไม่ปลอดภัย">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` เป็นเพียงตัวสลับความเข้ากันได้ภายในเครื่อง:

    - ช่วยให้เซสชัน Control UI บน localhost ดำเนินการต่อได้โดยไม่มีข้อมูลประจำตัวอุปกรณ์ในบริบท HTTP ที่ไม่ปลอดภัย
    - ไม่ได้ข้ามการตรวจสอบการจับคู่
    - ไม่ได้ผ่อนปรนข้อกำหนดข้อมูลประจำตัวอุปกรณ์ระยะไกล (ที่ไม่ใช่ localhost)

  </Accordion>
  <Accordion title="สำหรับกรณีฉุกเฉินเท่านั้น">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` ปิดใช้งานการตรวจสอบข้อมูลประจำตัวอุปกรณ์ของ Control UI และลดระดับความปลอดภัยอย่างร้ายแรง ให้เปลี่ยนกลับโดยเร็วหลังใช้งานในกรณีฉุกเฉิน
    </Warning>

  </Accordion>
  <Accordion title="หมายเหตุเกี่ยวกับพร็อกซีที่เชื่อถือได้">
    - การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้ซึ่งสำเร็จสามารถอนุญาตเซสชัน Control UI ของ **ผู้ดำเนินการ** โดยไม่มีข้อมูลประจำตัวอุปกรณ์
    - การอนุญาตนี้ **ไม่** ครอบคลุมถึงเซสชัน Control UI ที่มีบทบาท Node
    - พร็อกซีย้อนกลับแบบ loopback บนโฮสต์เดียวกันยังคงไม่ผ่านการยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้ โปรดดู [การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth)

  </Accordion>
</AccordionGroup>

โปรดดูคำแนะนำการตั้งค่า HTTPS ที่ [Tailscale](/th/gateway/tailscale)

## นโยบายความปลอดภัยของเนื้อหา

Control UI มาพร้อมนโยบาย `img-src` ที่เข้มงวด โดยอนุญาตเฉพาะแอสเซ็ตจาก **ต้นทางเดียวกัน**, URL แบบ `data:` และ URL แบบ `blob:` ที่สร้างภายในเครื่อง เบราว์เซอร์จะปฏิเสธ `http(s)` ระยะไกลและ URL รูปภาพแบบอ้างอิงโปรโตคอลโดยสัมพัทธ์ และจะไม่ส่งคำขอดึงข้อมูลผ่านเครือข่าย

ในทางปฏิบัติ:

- อวาตาร์และรูปภาพที่ให้บริการภายใต้พาธสัมพัทธ์ (เช่น `/avatars/<id>`) ยังคงแสดงผลได้ รวมถึงเส้นทางอวาตาร์ที่ต้องยืนยันตัวตนซึ่ง UI ดึงข้อมูลและแปลงเป็น URL แบบ `blob:` ภายในเครื่อง
- URL แบบ `data:image/...` ที่ฝังในบรรทัดยังคงแสดงผลได้
- URL แบบ `blob:` ภายในเครื่องที่ Control UI สร้างยังคงแสดงผลได้
- Gateway จะดึงอวาตาร์ตัวอย่างลิงก์ GitHub จากโฮสต์อวาตาร์แบบคงที่ของ GitHub และส่งกลับเป็น URL แบบ `data:` ที่มีขอบเขตจำกัด เบราว์เซอร์ของผู้ดำเนินการจะไม่ติดต่อโฮสต์อวาตาร์ระยะไกล
- URL อวาตาร์ระยะไกลที่มาจากข้อมูลเมตาของช่องทางจะถูกนำออกโดยตัวช่วยอวาตาร์ของ Control UI และแทนที่ด้วยโลโก้/ป้ายในตัว เพื่อไม่ให้ช่องทางที่ถูกเจาะระบบหรือเป็นอันตรายบังคับให้เบราว์เซอร์ของผู้ดำเนินการดึงรูปภาพระยะไกลโดยพลการ

คุณลักษณะนี้เปิดใช้งานตลอดเวลาและกำหนดค่าไม่ได้

## การยืนยันตัวตนของเส้นทางอวาตาร์

เมื่อกำหนดค่าการยืนยันตัวตนของ Gateway แล้ว จุดปลายอวาตาร์ของ Control UI ต้องใช้โทเค็น Gateway เดียวกับส่วนอื่นของ API:

- `GET /avatar/<agentId>` ส่งคืนรูปภาพอวาตาร์ให้เฉพาะผู้เรียกที่ผ่านการยืนยันตัวตนเท่านั้น ส่วน `GET /avatar/<agentId>?meta=1` ส่งคืนข้อมูลเมตาของอวาตาร์ภายใต้กฎเดียวกัน
- คำขอที่ไม่ผ่านการยืนยันตัวตนไปยังเส้นทางใดเส้นทางหนึ่งจะถูกปฏิเสธ (สอดคล้องกับเส้นทางสื่อของผู้ช่วยที่อยู่ข้างเคียง) ดังนั้นเส้นทางอวาตาร์จึงไม่สามารถทำให้ข้อมูลประจำตัวของเอเจนต์รั่วไหลบนโฮสต์ที่ได้รับการป้องกันในส่วนอื่น
- Control UI ส่งต่อโทเค็น Gateway เป็นส่วนหัว bearer เมื่อดึงอวาตาร์ และใช้ URL แบบ blob ที่ผ่านการยืนยันตัวตน เพื่อให้รูปภาพยังคงแสดงผลในแดชบอร์ด

หากปิดใช้งานการยืนยันตัวตนของ Gateway (ไม่แนะนำบนโฮสต์ที่ใช้ร่วมกัน) เส้นทางอวาตาร์จะไม่ต้องผ่านการยืนยันตัวตนเช่นกัน ซึ่งสอดคล้องกับส่วนอื่นของ Gateway

## การยืนยันตัวตนของเส้นทางสื่อของผู้ช่วย

เมื่อกำหนดค่าการยืนยันตัวตนของ Gateway แล้ว ตัวอย่างสื่อภายในเครื่องของผู้ช่วยจะใช้เส้นทางสองขั้นตอน:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` ต้องใช้การยืนยันตัวตนผู้ดำเนินการ Control UI ตามปกติ เบราว์เซอร์จะส่งโทเค็น Gateway เป็นส่วนหัว bearer เมื่อตรวจสอบความพร้อมใช้งาน
- การตอบกลับข้อมูลเมตาที่สำเร็จจะมี `mediaTicket` อายุสั้นซึ่งจำกัดขอบเขตไว้เฉพาะพาธต้นทางนั้น
- URL รูปภาพ เสียง วิดีโอ และเอกสารที่เบราว์เซอร์แสดงผลจะใช้ `mediaTicket=<ticket>` แทนโทเค็นหรือรหัสผ่าน Gateway ที่ใช้งานอยู่ ตั๋วจะหมดอายุอย่างรวดเร็วและไม่สามารถให้สิทธิ์แก่ต้นทางอื่นได้

วิธีนี้ทำให้การแสดงผลสื่อยังคงเข้ากันได้กับองค์ประกอบสื่อแบบเนทีฟของเบราว์เซอร์ โดยไม่ใส่ข้อมูลประจำตัว Gateway ที่นำกลับมาใช้ซ้ำได้ใน URL สื่อที่มองเห็นได้

## ลิงก์การอนุมัติ

การแจ้งเตือนการอนุมัติสำหรับผู้ดำเนินการสามารถเชื่อมโยงโดยตรงไปยังเอกสารการอนุมัติแบบสแตนด์อโลนที่ให้บริการภายใต้เนมสเปซ `${controlUiBasePath}/approve/{approvalId}` ที่สงวนไว้ (เช่น `/approve/<approvalId>` หรือ `/openclaw/approve/<approvalId>` เมื่อกำหนดค่าพาธฐาน) URL จะคงที่ตลอดอายุของการอนุมัติและส่งต่อระหว่างอุปกรณ์ของคุณเองได้อย่างปลอดภัย โดยระบุเฉพาะการอนุมัติและไม่ให้สิทธิ์อนุมัติ

- Gateway สงวนเนมสเปซ `/approve/<approvalId>` แบบหนึ่งเซกเมนต์ไว้ก่อนเส้นทาง HTTP ของ Plugin สำหรับเมธอด HTTP **ทั้งหมด** ดังนั้นเส้นทางของ Plugin จึงไม่สามารถบดบังหรือสกัดกั้นเอกสารการอนุมัติได้
- การเปิดเอกสารการอนุมัติต้องใช้การยืนยันตัวตน Gateway แบบเดียวกับส่วนอื่นของ Control UI (โทเค็น/รหัสผ่าน ข้อมูลประจำตัว Tailscale Serve หรือข้อมูลประจำตัวจากพร็อกซีที่เชื่อถือได้) ข้อมูลประจำตัวจะไม่เป็นส่วนหนึ่งของ URL การอนุมัติ
- เมื่อปิดใช้งานการให้บริการ Control UI คำขอไปยังเนมสเปซจะส่งคืน `404` แทนการส่งต่อไปยังตัวจัดการ Plugin
- การลงชื่อเข้าใช้ในเอกสารการอนุมัติมีผลชั่วคราวเฉพาะหน้านั้น โดยจะไม่เขียนทับการเลือก Gateway หรือการตั้งค่าที่ Control UI แบบเต็มบันทึกไว้ในเบราว์เซอร์เดียวกัน

Gateway ให้บริการไฟล์แบบคงที่จาก `dist/control-ui`:

```bash
pnpm ui:build
```

พาธฐานแบบสัมบูรณ์ที่เป็นตัวเลือก (URL แอสเซ็ตแบบคงที่):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

การพัฒนาภายในเครื่อง (เซิร์ฟเวอร์สำหรับพัฒนาแยกต่างหาก):

```bash
pnpm ui:dev
```

จากนั้นกำหนดให้ UI ชี้ไปยัง URL WS ของ Gateway (เช่น `ws://127.0.0.1:18789`)

## หน้า Control UI ว่างเปล่า

หากเบราว์เซอร์โหลดแดชบอร์ดว่างเปล่าและ DevTools ไม่แสดงข้อผิดพลาดที่เป็นประโยชน์ ส่วนขยายหรือสคริปต์เนื้อหาที่ทำงานตั้งแต่ต้นอาจขัดขวางไม่ให้แอปโมดูล JavaScript ประมวลผล หน้าคงที่มีแผงกู้คืน HTML ธรรมดาที่จะปรากฏเมื่อ `<openclaw-app>` ไม่ได้ลงทะเบียนหลังเริ่มต้นระบบ

ใช้การดำเนินการ **Try again** ของแผงหลังเปลี่ยนสภาพแวดล้อมเบราว์เซอร์ หรือโหลดใหม่ด้วยตนเองหลังตรวจสอบรายการต่อไปนี้:

- ปิดใช้งานส่วนขยายที่แทรกเนื้อหาลงในทุกหน้า โดยเฉพาะส่วนขยายที่มีสคริปต์เนื้อหา `<all_urls>`
- ลองใช้หน้าต่างส่วนตัว โปรไฟล์เบราว์เซอร์ใหม่ หรือเบราว์เซอร์อื่น
- ให้ Gateway ทำงานต่อไปและตรวจสอบ URL แดชบอร์ดเดิมหลังเปลี่ยนเบราว์เซอร์

## การดีบัก/ทดสอบ: เซิร์ฟเวอร์สำหรับพัฒนา + Gateway ระยะไกล

Control UI เป็นไฟล์แบบคงที่ ส่วนเป้าหมาย WebSocket สามารถกำหนดค่าและแตกต่างจากต้นทาง HTTP ได้ ซึ่งสะดวกเมื่อต้องการใช้เซิร์ฟเวอร์สำหรับพัฒนา Vite ภายในเครื่อง แต่ให้ Gateway ทำงานที่อื่น

<Steps>
  <Step title="เริ่มเซิร์ฟเวอร์สำหรับพัฒนา UI">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="เปิดด้วย gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    การยืนยันตัวตนครั้งเดียวที่เป็นตัวเลือก (หากจำเป็น):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="หมายเหตุ">
    - `gatewayUrl` จะถูกจัดเก็บใน localStorage หลังโหลดและนำออกจาก URL
    - หากส่งจุดปลาย `ws://` หรือ `wss://` แบบเต็มผ่าน `gatewayUrl` ให้เข้ารหัสค่าเป็น URL เพื่อให้เบราว์เซอร์แยกวิเคราะห์สตริงการสืบค้นได้อย่างถูกต้อง
    - ควรส่ง `token` ผ่านส่วน fragment ของ URL (`#token=...`) เมื่อเป็นไปได้ ส่วน fragment จะไม่ถูกส่งไปยังเซิร์ฟเวอร์ จึงหลีกเลี่ยงการรั่วไหลผ่านบันทึกคำขอและ Referer พารามิเตอร์การสืบค้น `?token=` แบบเดิมยังคงถูกนำเข้าหนึ่งครั้งเพื่อความเข้ากันได้ แต่ใช้เป็นทางเลือกสำรองเท่านั้น และจะถูกนำออกทันทีหลัง bootstrap
    - `password` จะถูกเก็บไว้ในหน่วยความจำเท่านั้น
    - เมื่อตั้งค่า `gatewayUrl` แล้ว UI จะไม่ย้อนกลับไปใช้ข้อมูลประจำตัวจากการกำหนดค่าหรือสภาพแวดล้อม ต้องระบุ `token` (หรือ `password`) อย่างชัดเจน การขาดข้อมูลประจำตัวที่ระบุไว้อย่างชัดเจนถือเป็นข้อผิดพลาด
    - ใช้ `wss://` เมื่อ Gateway อยู่หลัง TLS (Tailscale Serve, พร็อกซี HTTPS เป็นต้น)
    - `gatewayUrl` จะได้รับการยอมรับเฉพาะในหน้าต่างระดับบนสุด (ไม่ใช่แบบฝัง) เพื่อป้องกัน clickjacking
    - การติดตั้งใช้งาน Control UI สาธารณะที่ไม่ใช่ loopback ต้องตั้งค่า `gateway.controlUi.allowedOrigins` อย่างชัดเจน (ต้นทางแบบเต็ม) การโหลดจาก LAN/Tailnet ส่วนตัวที่มีต้นทางเดียวกันผ่าน loopback, RFC1918/link-local, `.local`, `.ts.net` หรือโฮสต์ Tailscale CGNAT จะได้รับการยอมรับโดยไม่ต้องเปิดใช้ทางเลือกสำรองส่วนหัว Host
    - การเริ่มต้น Gateway อาจเติมต้นทางภายในเครื่อง เช่น `http://localhost:<port>` และ `http://127.0.0.1:<port>` จากการผูกและพอร์ตของรันไทม์ที่มีผล แต่ต้นทางของเบราว์เซอร์ระยะไกลยังคงต้องมีรายการที่ระบุไว้อย่างชัดเจน
    - ห้ามใช้ `gateway.controlUi.allowedOrigins: ["*"]` ยกเว้นสำหรับการทดสอบภายในเครื่องที่ควบคุมอย่างเข้มงวด เนื่องจากหมายถึงอนุญาตต้นทางของเบราว์เซอร์ใดก็ได้ ไม่ใช่ "จับคู่กับโฮสต์ใดก็ตามที่กำลังใช้งาน"
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` เปิดใช้งานโหมดทางเลือกสำรองต้นทางจากส่วนหัว Host แต่เป็นโหมดความปลอดภัยที่อันตราย

  </Accordion>
</AccordionGroup>

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
