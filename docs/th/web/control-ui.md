Control UI เป็นแอปหน้าเดียวขนาดเล็กแบบ **Vite + Lit** ที่ให้บริการโดย Gateway:

- ค่าเริ่มต้น: `http://<host>:18789/`
- คำนำหน้าแบบเลือกได้: ตั้งค่า `gateway.controlUi.basePath` (เช่น `/openclaw`)

แอปนี้สื่อสาร **โดยตรงกับ Gateway WebSocket** บนพอร์ตเดียวกัน

## เปิดอย่างรวดเร็ว (ภายในเครื่อง)

หาก Gateway กำลังทำงานบนคอมพิวเตอร์เครื่องเดียวกัน ให้เปิด:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (หรือ [http://localhost:18789/](http://localhost:18789/))

หากหน้าโหลดไม่สำเร็จ ให้เริ่ม Gateway ก่อน: `openclaw gateway`

ระบบส่ง auth ระหว่าง WebSocket handshake ผ่าน:

- `connect.params.auth.token`
- `connect.params.auth.password`
- ส่วนหัวตัวตนของ Tailscale Serve เมื่อ `gateway.auth.allowTailscale: true`
- ส่วนหัวตัวตนของพร็อกซีที่เชื่อถือได้เมื่อ `gateway.auth.mode: "trusted-proxy"`

แผงการตั้งค่าแดชบอร์ดจะเก็บโทเค็นไว้สำหรับเซสชันแท็บเบราว์เซอร์ปัจจุบันและ URL ของ gateway ที่เลือก; รหัสผ่านจะไม่ถูกจัดเก็บไว้ Onboarding มักจะสร้างโทเค็น gateway สำหรับ auth แบบ shared-secret เมื่อเชื่อมต่อครั้งแรก แต่ auth ด้วยรหัสผ่านก็ใช้ได้เช่นกันเมื่อ `gateway.auth.mode` เป็น `"password"`

## การจับคู่อุปกรณ์ (การเชื่อมต่อครั้งแรก)

เมื่อคุณเชื่อมต่อกับ Control UI จากเบราว์เซอร์หรืออุปกรณ์ใหม่ Gateway มักจะต้องการ **การอนุมัติการจับคู่แบบครั้งเดียว** นี่เป็นมาตรการรักษาความปลอดภัยเพื่อป้องกันการเข้าถึงโดยไม่ได้รับอนุญาต

**สิ่งที่คุณจะเห็น:** "disconnected (1008): pairing required"

<Steps>
  <Step title="แสดงรายการคำขอที่รออนุมัติ">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="อนุมัติด้วย ID คำขอ">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

หากเบราว์เซอร์ลองจับคู่อีกครั้งด้วยรายละเอียด auth ที่เปลี่ยนไป (role/scopes/public key) คำขอที่รออนุมัติก่อนหน้าจะถูกแทนที่และสร้าง `requestId` ใหม่ ให้รัน `openclaw devices list` อีกครั้งก่อนอนุมัติ

หากเบราว์เซอร์จับคู่แล้วและคุณเปลี่ยนจากสิทธิ์อ่านเป็นสิทธิ์เขียน/ผู้ดูแล ระบบจะถือว่าเป็นการอัปเกรดการอนุมัติ ไม่ใช่การเชื่อมต่อใหม่แบบเงียบ OpenClaw จะคงการอนุมัติเก่าไว้ บล็อกการเชื่อมต่อใหม่ที่มีสิทธิ์กว้างกว่า และขอให้คุณอนุมัติชุด scope ใหม่อย่างชัดเจน

เมื่ออนุมัติแล้ว ระบบจะจดจำอุปกรณ์และจะไม่ต้องอนุมัติซ้ำ เว้นแต่คุณจะเพิกถอนด้วย `openclaw devices revoke --device <id> --role <role>` ดู [CLI อุปกรณ์](/th/cli/devices) สำหรับการหมุนเวียนและการเพิกถอนโทเค็น

<Note>
- การเชื่อมต่อเบราว์เซอร์ผ่าน local loopback โดยตรง (`127.0.0.1` / `localhost`) จะได้รับการอนุมัติโดยอัตโนมัติ
- Tailscale Serve สามารถข้ามรอบการจับคู่สำหรับเซสชันผู้ปฏิบัติงานของ Control UI ได้เมื่อ `gateway.auth.allowTailscale: true`, ตัวตน Tailscale ยืนยันสำเร็จ และเบราว์เซอร์แสดงตัวตนอุปกรณ์ของตน
- การผูก Tailnet โดยตรง, การเชื่อมต่อเบราว์เซอร์บน LAN, และโปรไฟล์เบราว์เซอร์ที่ไม่มีตัวตนอุปกรณ์ยังต้องการการอนุมัติอย่างชัดเจน
- แต่ละโปรไฟล์เบราว์เซอร์จะสร้าง ID อุปกรณ์ที่ไม่ซ้ำกัน ดังนั้นการสลับเบราว์เซอร์หรือล้างข้อมูลเบราว์เซอร์จะต้องจับคู่ใหม่

</Note>

## ตัวตนส่วนบุคคล (เฉพาะเบราว์เซอร์)

Control UI รองรับตัวตนส่วนบุคคลต่อเบราว์เซอร์ (ชื่อที่แสดงและอวาตาร์) ที่แนบกับข้อความขาออกเพื่อระบุแหล่งที่มาในเซสชันที่ใช้ร่วมกัน ตัวตนนี้อยู่ในพื้นที่จัดเก็บของเบราว์เซอร์ จำกัดขอบเขตอยู่ที่โปรไฟล์เบราว์เซอร์ปัจจุบัน และจะไม่ซิงค์ไปยังอุปกรณ์อื่นหรือคงอยู่ฝั่งเซิร์ฟเวอร์เกินกว่าเมทาดาทาการเป็นผู้เขียนทรานสคริปต์ตามปกติบนข้อความที่คุณส่งจริง การล้างข้อมูลไซต์หรือสลับเบราว์เซอร์จะรีเซ็ตให้ว่างเปล่า

รูปแบบเฉพาะเบราว์เซอร์แบบเดียวกันนี้ใช้กับการแทนที่อวาตาร์ของผู้ช่วย อวาตาร์ผู้ช่วยที่อัปโหลดจะซ้อนทับตัวตนที่ gateway แก้ไขได้เฉพาะบนเบราว์เซอร์ภายในเครื่องเท่านั้น และจะไม่ส่งไปกลับผ่าน `config.patch` ฟิลด์ config ที่ใช้ร่วมกัน `ui.assistant.avatar` ยังคงพร้อมใช้งานสำหรับไคลเอนต์ที่ไม่ใช่ UI ที่เขียนฟิลด์โดยตรง (เช่น gateway แบบสคริปต์หรือแดชบอร์ดแบบกำหนดเอง)

## จุดปลาย config ขณะรัน

Control UI ดึงการตั้งค่าขณะรันจาก `/__openclaw/control-ui-config.json` จุดปลายนี้ถูกควบคุมด้วย auth ของ gateway เดียวกับพื้นผิว HTTP ที่เหลือ: เบราว์เซอร์ที่ยังไม่ได้ auth จะดึงข้อมูลไม่ได้ และการดึงข้อมูลที่สำเร็จต้องมีโทเค็น/รหัสผ่าน gateway ที่ถูกต้องอยู่แล้ว, ตัวตน Tailscale Serve, หรือตัวตนของพร็อกซีที่เชื่อถือได้

## การรองรับภาษา

Control UI สามารถแปลตัวเองตามภาษาเบราว์เซอร์ของคุณเมื่อโหลดครั้งแรก หากต้องการแทนที่ภายหลัง ให้เปิด **Overview -> Gateway Access -> Language** ตัวเลือกภาษาอยู่ในการ์ด Gateway Access ไม่ใช่ใต้ Appearance

- ภาษาที่รองรับ: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- คำแปลที่ไม่ใช่ภาษาอังกฤษจะถูกโหลดแบบ lazy-load ในเบราว์เซอร์
- ภาษาที่เลือกจะถูกบันทึกในพื้นที่จัดเก็บของเบราว์เซอร์และนำกลับมาใช้ในการเข้าชมครั้งต่อไป
- คีย์คำแปลที่ขาดหายจะย้อนกลับไปใช้ภาษาอังกฤษ

คำแปลของเอกสารถูกสร้างสำหรับชุดภาษาที่ไม่ใช่ภาษาอังกฤษเดียวกัน แต่ตัวเลือกภาษา Mintlify ในตัวของไซต์เอกสารถูกจำกัดไว้เฉพาะรหัสภาษาที่ Mintlify ยอมรับ เอกสารภาษาไทย (`th`) และเปอร์เซีย (`fa`) ยังคงถูกสร้างใน repo สำหรับเผยแพร่; เอกสารเหล่านี้อาจไม่ปรากฏในตัวเลือกนั้นจนกว่า Mintlify จะรองรับรหัสเหล่านี้

## ธีมลักษณะที่ปรากฏ

แผง Appearance เก็บธีม Claw, Knot และ Dash ในตัวไว้ รวมถึงช่องนำเข้า tweakcn หนึ่งช่องแบบเฉพาะเบราว์เซอร์ หากต้องการนำเข้าธีม ให้เปิด [ธีม tweakcn](https://tweakcn.com/themes), เลือกหรือสร้างธีม, คลิก **Share**, แล้ววางลิงก์ธีมที่คัดลอกไว้ลงใน Appearance ตัวนำเข้ายอมรับ URL registry แบบ `https://tweakcn.com/r/themes/<id>`, URL ตัวแก้ไขอย่าง `https://tweakcn.com/editor/theme?theme=amethyst-haze`, path แบบสัมพัทธ์ `/themes/<id>`, ID ธีมดิบ และชื่อธีมเริ่มต้น เช่น `amethyst-haze` ด้วย

ธีมที่นำเข้าจะถูกจัดเก็บเฉพาะในโปรไฟล์เบราว์เซอร์ปัจจุบันเท่านั้น ธีมเหล่านี้จะไม่ถูกเขียนลง config ของ gateway และไม่ซิงค์ข้ามอุปกรณ์ การแทนที่ธีมที่นำเข้าจะอัปเดตช่องภายในเครื่องช่องเดียว; การล้างธีมจะสลับธีมที่ใช้งานอยู่กลับไปเป็น Claw หากธีมที่นำเข้าถูกเลือกอยู่

    - พูดคุยผ่านเซสชันเรียลไทม์ของเบราว์เซอร์ OpenAI ใช้ WebRTC โดยตรง, Google Live ใช้โทเค็นเบราว์เซอร์แบบใช้ครั้งเดียวที่ถูกจำกัดผ่าน WebSocket และ Plugin เสียงเรียลไทม์ที่ทำงานเฉพาะแบ็กเอนด์ใช้การขนส่งรีเลย์ของ Gateway รีเลย์จะเก็บข้อมูลประจำตัวของผู้ให้บริการไว้บน Gateway ขณะที่เบราว์เซอร์สตรีม PCM จากไมโครโฟนผ่าน RPC `talk.realtime.relay*` และส่งการเรียกเครื่องมือ `openclaw_agent_consult` กลับผ่าน `chat.send` สำหรับโมเดล OpenClaw ขนาดใหญ่กว่าที่กำหนดค่าไว้

  <Accordion title="Config">
    - ดู/แก้ไข `~/.openclaw/openclaw.json` (`config.get`, `config.set`)
    - นำไปใช้ + รีสตาร์ทพร้อมการตรวจสอบความถูกต้อง (`config.apply`) และปลุกเซสชันล่าสุดที่ใช้งานอยู่
    - การเขียนมีตัวป้องกัน base-hash เพื่อป้องกันการเขียนทับการแก้ไขที่เกิดขึ้นพร้อมกัน
    - การเขียน (`config.set`/`config.apply`/`config.patch`) จะตรวจสอบล่วงหน้าการแก้ไข SecretRef ที่ใช้งานอยู่สำหรับ ref ใน payload config ที่ส่งมา; ref ที่ส่งมาและยังใช้งานอยู่แต่แก้ไขไม่ได้จะถูกปฏิเสธก่อนเขียน
    - schema + การเรนเดอร์ฟอร์ม (`config.schema` / `config.schema.lookup`, รวมถึงฟิลด์ `title` / `description`, คำแนะนำ UI ที่ตรงกัน, สรุปลูกโดยตรง, เมทาดาทาเอกสารบนโหนดออบเจ็กต์ซ้อน/wildcard/array/composition, รวมถึง schema ของ Plugin + ช่องทางเมื่อมี); ตัวแก้ไข Raw JSON จะพร้อมใช้งานเฉพาะเมื่อ snapshot มีการ round-trip แบบ raw ที่ปลอดภัย
    - หาก snapshot ไม่สามารถ round-trip ข้อความ raw ได้อย่างปลอดภัย Control UI จะบังคับใช้โหมด Form และปิดใช้งานโหมด Raw สำหรับ snapshot นั้น
    - ตัวแก้ไข Raw JSON "Reset to saved" จะรักษารูปร่างที่เขียนแบบ raw ไว้ (การจัดรูปแบบ, ความคิดเห็น, เลย์เอาต์ `$include`) แทนการเรนเดอร์ snapshot ที่ทำให้แบนใหม่ ดังนั้นการแก้ไขภายนอกจะยังคงอยู่หลังรีเซ็ตเมื่อ snapshot สามารถ round-trip ได้อย่างปลอดภัย
    - ค่าออบเจ็กต์ SecretRef แบบมีโครงสร้างจะถูกเรนเดอร์เป็นอ่านอย่างเดียวในช่องกรอกข้อความของฟอร์มเพื่อป้องกันความเสียหายจากการแปลงออบเจ็กต์เป็นสตริงโดยไม่ตั้งใจ

  <Accordion title="หมายเหตุแผงงาน Cron">
    - สำหรับงานที่แยกเดี่ยว การส่งจะมีค่าเริ่มต้นเป็นการประกาศสรุป คุณสามารถสลับเป็นไม่มีได้หากต้องการการรันภายในเท่านั้น
    - ฟิลด์ช่องทาง/เป้าหมายจะปรากฏเมื่อเลือกประกาศ
    - โหมด Webhook ใช้ `delivery.mode = "webhook"` โดยตั้งค่า `delivery.to` เป็น URL Webhook HTTP(S) ที่ถูกต้อง
    - สำหรับงานเซสชันหลัก มีโหมดการส่ง webhook และไม่มีให้ใช้
    - ตัวควบคุมการแก้ไขขั้นสูงมี delete-after-run, ล้างการแทนที่เอเจนต์, ตัวเลือก cron exact/stagger, การแทนที่โมเดล/thinking ของเอเจนต์ และสวิตช์การส่งแบบ best-effort
    - การตรวจสอบความถูกต้องของฟอร์มเป็นแบบ inline พร้อมข้อผิดพลาดระดับฟิลด์; ค่าที่ไม่ถูกต้องจะปิดใช้งานปุ่มบันทึกจนกว่าจะแก้ไข
    - ตั้งค่า `cron.webhookToken` เพื่อส่ง bearer token เฉพาะ หากละไว้ Webhook จะถูกส่งโดยไม่มีส่วนหัว auth
    - fallback ที่เลิกใช้แล้ว: งาน legacy ที่จัดเก็บพร้อม `notify: true` ยังคงใช้ `cron.webhook` ได้จนกว่าจะถูกย้ายข้อมูล

  </Accordion>
</AccordionGroup>

## พฤติกรรม Chat

<AccordionGroup>
  <Accordion title="ความหมายของการส่งและประวัติ">
    - `chat.send` เป็นแบบ **ไม่บล็อก**: ตอบรับทันทีด้วย `{ runId, status: "started" }` และการตอบกลับจะสตรีมผ่านเหตุการณ์ `chat`
    - การอัปโหลดแชตรองรับรูปภาพพร้อมไฟล์ที่ไม่ใช่วิดีโอ รูปภาพจะเก็บพาธรูปภาพดั้งเดิมไว้ ส่วนไฟล์อื่นจะถูกจัดเก็บเป็นสื่อที่จัดการโดยระบบและแสดงในประวัติเป็นลิงก์ไฟล์แนบ
    - การส่งซ้ำด้วย `idempotencyKey` เดิมจะคืนค่า `{ status: "in_flight" }` ขณะกำลังทำงาน และ `{ status: "ok" }` หลังเสร็จสิ้น
    - คำตอบของ `chat.history` ถูกจำกัดขนาดเพื่อความปลอดภัยของ UI เมื่อรายการบทสนทนามีขนาดใหญ่เกินไป Gateway อาจตัดทอนฟิลด์ข้อความยาว ละเว้นบล็อกเมทาดาทาขนาดใหญ่ และแทนที่ข้อความที่ใหญ่เกินด้วยตัวแทน (`[chat.history omitted: message too large]`)
    - รูปภาพจาก Assistant/ที่สร้างขึ้นจะถูกบันทึกเป็นการอ้างอิงสื่อที่จัดการโดยระบบและส่งกลับผ่าน URL สื่อของ Gateway ที่ผ่านการยืนยันตัวตนแล้ว ดังนั้นการโหลดใหม่จึงไม่ขึ้นกับเพย์โหลดรูปภาพ base64 ดิบที่ยังคงอยู่ในคำตอบประวัติแชต
    - `chat.history` ยังลบแท็กคำสั่งแบบอินไลน์ที่ใช้เพื่อการแสดงผลเท่านั้นออกจากข้อความ Assistant ที่มองเห็นได้ (เช่น `[[reply_to_*]]` และ `[[audio_as_voice]]`), เพย์โหลด XML ของการเรียกเครื่องมือแบบข้อความล้วน (รวมถึง `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อกการเรียกเครื่องมือที่ถูกตัดทอน), และโทเค็นควบคุมโมเดล ASCII/แบบเต็มความกว้างที่รั่วออกมา และละเว้นรายการ Assistant ที่ข้อความที่มองเห็นได้ทั้งหมดเป็นเพียงโทเค็นเงียบแบบตรงตัว `NO_REPLY` / `no_reply`
    - ระหว่างการส่งที่กำลังทำงานและการรีเฟรชประวัติครั้งสุดท้าย มุมมองแชตจะยังแสดงข้อความผู้ใช้/Assistant แบบ optimistic ในเครื่องไว้ หาก `chat.history` ส่งคืนสแนปช็อตเก่าชั่วคราว บทสนทนามาตรฐานจะแทนที่ข้อความในเครื่องเหล่านั้นเมื่อประวัติ Gateway ตามทัน
    - `chat.inject` ผนวกบันทึกของ Assistant ลงในบทสนทนาของเซสชันและกระจายเหตุการณ์ `chat` สำหรับการอัปเดตเฉพาะ UI (ไม่มีการรัน agent ไม่มีการส่งผ่านช่องทาง)
    - ตัวเลือกโมเดลและการคิดในส่วนหัวแชตจะแพตช์เซสชันที่ใช้งานอยู่ทันทีผ่าน `sessions.patch`; สิ่งเหล่านี้เป็นการ override เซสชันแบบถาวร ไม่ใช่ตัวเลือกการส่งที่ใช้เพียงหนึ่งเทิร์น
    - ตัวเลือกโมเดลแชตจะขอมุมมองโมเดลที่กำหนดค่าไว้ของ Gateway หากมี `agents.defaults.models` รายการอนุญาตนั้นจะขับเคลื่อนตัวเลือก มิฉะนั้นตัวเลือกจะแสดงรายการ `models.providers.*.models` ที่ระบุชัดเจนพร้อมผู้ให้บริการที่มีการยืนยันตัวตนที่ใช้งานได้ แคตตาล็อกเต็มยังคงพร้อมใช้งานผ่าน RPC ดีบัก `models.list` ด้วย `view: "all"`
    - เมื่อรายงานการใช้งานเซสชัน Gateway ล่าสุดแสดงว่ามีแรงกดดันบริบทสูง พื้นที่เขียนข้อความแชตจะแสดงประกาศบริบท และในระดับ Compaction ที่แนะนำ จะแสดงปุ่มย่อที่รันเส้นทาง Compaction ของเซสชันตามปกติ สแนปช็อตโทเค็นที่ค้างจะถูกซ่อนไว้จนกว่า Gateway จะรายงานการใช้งานใหม่อีกครั้ง

## การติดตั้ง PWA และ Web Push

Control UI มาพร้อม `manifest.webmanifest` และ service worker ดังนั้นเบราว์เซอร์สมัยใหม่จึงสามารถติดตั้งเป็น PWA แบบสแตนด์อโลนได้ Web Push ช่วยให้ Gateway ปลุก PWA ที่ติดตั้งแล้วด้วยการแจ้งเตือนได้แม้เมื่อแท็บหรือหน้าต่างเบราว์เซอร์ไม่ได้เปิดอยู่

| พื้นผิว                                               | ทำอะไร                                                       |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | manifest ของ PWA เบราว์เซอร์จะแสดงตัวเลือก "Install app" เมื่อเข้าถึงได้   |
| `ui/public/sw.js`                                     | service worker ที่จัดการเหตุการณ์ `push` และการคลิกการแจ้งเตือน |
| `push/vapid-keys.json` (ใต้ไดเรกทอรีสถานะของ OpenClaw) | คู่คีย์ VAPID ที่สร้างอัตโนมัติ ใช้ลงนามเพย์โหลด Web Push       |
| `push/web-push-subscriptions.json`                    | endpoint การสมัครรับข้อมูลของเบราว์เซอร์ที่บันทึกไว้                          |

Override คู่คีย์ VAPID ผ่านตัวแปร env บนโปรเซส Gateway เมื่อต้องการตรึงคีย์ (สำหรับการปรับใช้หลายโฮสต์ การหมุนเวียน secret หรือการทดสอบ):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (ค่าเริ่มต้นเป็น `mailto:openclaw@localhost`)

Control UI ใช้เมธอด Gateway ที่จำกัดด้วย scope เหล่านี้เพื่อลงทะเบียนและทดสอบการสมัครรับข้อมูลของเบราว์เซอร์:

- `push.web.vapidPublicKey` — ดึงคีย์สาธารณะ VAPID ที่ใช้งานอยู่
- `push.web.subscribe` — ลงทะเบียน `endpoint` พร้อม `keys.p256dh`/`keys.auth`
- `push.web.unsubscribe` — ลบ endpoint ที่ลงทะเบียนไว้
- `push.web.test` — ส่งการแจ้งเตือนทดสอบไปยังการสมัครรับข้อมูลของผู้เรียก

<Note>
Web Push แยกจากเส้นทางรีเลย์ APNS ของ iOS (ดู [การกำหนดค่า](/th/gateway/configuration) สำหรับ push ที่มีรีเลย์รองรับ) และเมธอด `push.test` เดิม ซึ่งมุ่งเป้าไปที่การจับคู่มือถือแบบเนทีฟ
</Note>

## embed ที่โฮสต์

ข้อความ Assistant สามารถเรนเดอร์เนื้อหาเว็บที่โฮสต์แบบอินไลน์ด้วย shortcode `[embed ...]` นโยบาย sandbox ของ iframe ถูกควบคุมโดย `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    ปิดใช้งานการรันสคริปต์ภายใน embed ที่โฮสต์
  </Tab>
  <Tab title="scripts (default)">
    อนุญาต embed แบบโต้ตอบได้พร้อมคงการแยก origin; นี่เป็นค่าเริ่มต้นและมักเพียงพอสำหรับเกม/วิดเจ็ตเบราว์เซอร์แบบจบในตัว
  </Tab>
  <Tab title="trusted">
    เพิ่ม `allow-same-origin` บน `allow-scripts` สำหรับเอกสารในไซต์เดียวกันที่ตั้งใจต้องการสิทธิ์ที่สูงกว่า
  </Tab>
</Tabs>

ตัวอย่าง:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
ใช้ `trusted` เฉพาะเมื่อเอกสารที่ฝังต้องการพฤติกรรม same-origin จริง ๆ เท่านั้น สำหรับเกมและ canvas แบบโต้ตอบส่วนใหญ่ที่ agent สร้าง `scripts` เป็นตัวเลือกที่ปลอดภัยกว่า
</Warning>

URL embed ภายนอกแบบสัมบูรณ์ `http(s)` ยังคงถูกบล็อกตามค่าเริ่มต้น หากคุณตั้งใจให้ `[embed url="https://..."]` โหลดหน้าจากบุคคลที่สาม ให้ตั้งค่า `gateway.controlUi.allowExternalEmbedUrls: true`

    เปิด:

    - `https://<magicdns>/` (หรือ `gateway.controlUi.basePath` ที่คุณกำหนดค่าไว้)

    ตามค่าเริ่มต้น คำขอ Control UI/WebSocket Serve สามารถยืนยันตัวตนผ่าน header ตัวตนของ Tailscale (`tailscale-user-login`) เมื่อ `gateway.auth.allowTailscale` เป็น `true` OpenClaw ตรวจสอบตัวตนโดย resolve ที่อยู่ `x-forwarded-for` ด้วย `tailscale whois` แล้วจับคู่กับ header และยอมรับเฉพาะเมื่อคำขอเข้ามาที่ loopback พร้อม header `x-forwarded-*` ของ Tailscale เท่านั้น สำหรับเซสชันผู้ปฏิบัติการ Control UI ที่มีตัวตนอุปกรณ์ของเบราว์เซอร์ เส้นทาง Serve ที่ตรวจสอบแล้วนี้ยังข้ามรอบการจับคู่อุปกรณ์ด้วย เบราว์เซอร์ที่ไม่มีอุปกรณ์และการเชื่อมต่อบทบาท node ยังคงตามการตรวจสอบอุปกรณ์ตามปกติ ตั้งค่า `gateway.auth.allowTailscale: false` หากคุณต้องการบังคับให้ใช้ข้อมูลรับรอง shared-secret แบบชัดเจนแม้สำหรับทราฟฟิก Serve จากนั้นใช้ `gateway.auth.mode: "token"` หรือ `"password"`

    สำหรับเส้นทางตัวตน Serve แบบ async นั้น ความพยายามยืนยันตัวตนที่ล้มเหลวสำหรับ IP ไคลเอนต์และ scope การยืนยันตัวตนเดียวกันจะถูก serialize ก่อนเขียน rate-limit ดังนั้นการลองผิดซ้ำพร้อมกันจากเบราว์เซอร์เดียวกันจึงอาจแสดง `retry later` ในคำขอที่สอง แทนที่จะเป็น mismatch ธรรมดาสองรายการที่แข่งกันแบบขนาน

    <Warning>
    การยืนยันตัวตน Serve แบบไม่มีโทเค็นถือว่าโฮสต์ gateway เชื่อถือได้ หากโค้ดในเครื่องที่ไม่น่าเชื่อถืออาจรันบนโฮสต์นั้น ให้บังคับใช้การยืนยันตัวตนด้วย token/password
    </Warning>

  </Tab>
  <Tab title="Bind กับ tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    จากนั้นเปิด:

    - `http://<tailscale-ip>:18789/` (หรือ `gateway.controlUi.basePath` ที่คุณกำหนดค่าไว้)

    วาง shared secret ที่ตรงกันลงในการตั้งค่า UI (ส่งเป็น `connect.params.auth.token` หรือ `connect.params.auth.password`)

  </Tab>
</Tabs>

## HTTP ที่ไม่ปลอดภัย

หากคุณเปิดแดชบอร์ดผ่าน HTTP ธรรมดา (`http://<lan-ip>` หรือ `http://<tailscale-ip>`) เบราว์เซอร์จะทำงานใน **บริบทที่ไม่ปลอดภัย** และบล็อก WebCrypto ตามค่าเริ่มต้น OpenClaw **บล็อก** การเชื่อมต่อ Control UI ที่ไม่มีตัวตนอุปกรณ์

ข้อยกเว้นที่บันทึกไว้:

- ความเข้ากันได้กับ HTTP ที่ไม่ปลอดภัยเฉพาะ localhost ด้วย `gateway.controlUi.allowInsecureAuth=true`
- การยืนยันตัวตน Control UI ของผู้ปฏิบัติการที่สำเร็จผ่าน `gateway.auth.mode: "trusted-proxy"`
- break-glass `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**วิธีแก้ที่แนะนำ:** ใช้ HTTPS (Tailscale Serve) หรือเปิด UI ในเครื่อง:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (บนโฮสต์ gateway)

<AccordionGroup>
  <Accordion title="พฤติกรรม toggle การยืนยันตัวตนที่ไม่ปลอดภัย">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` เป็น toggle ความเข้ากันได้ในเครื่องเท่านั้น:

    - อนุญาตให้เซสชัน Control UI ของ localhost ดำเนินต่อโดยไม่มีตัวตนอุปกรณ์ในบริบท HTTP ที่ไม่ปลอดภัย
    - ไม่ข้ามการตรวจสอบการจับคู่
    - ไม่ผ่อนปรนข้อกำหนดตัวตนอุปกรณ์ระยะไกล (ไม่ใช่ localhost)

  </Accordion>
  <Accordion title="Break-glass เท่านั้น">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` จะปิดการตรวจสอบตัวตนอุปกรณ์ของ Control UI และเป็นการลดระดับความปลอดภัยอย่างร้ายแรง ให้ย้อนกลับโดยเร็วหลังจากใช้งานในกรณีฉุกเฉิน
    </Warning>

  </Accordion>
  <Accordion title="หมายเหตุ trusted-proxy">
    - การยืนยันตัวตน trusted-proxy ที่สำเร็จสามารถอนุญาตเซสชัน Control UI แบบ **operator** ได้โดยไม่มีตัวตนอุปกรณ์
    - สิ่งนี้**ไม่**ขยายไปถึงเซสชัน Control UI บทบาท node
    - รีเวิร์สพร็อกซี loopback บนโฮสต์เดียวกันยังคงไม่ผ่านการยืนยันตัวตน trusted-proxy; ดู [การยืนยันตัวตนด้วยพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth)

  </Accordion>
</AccordionGroup>

ดูคำแนะนำการตั้งค่า HTTPS ได้ที่ [Tailscale](/th/gateway/tailscale)

## นโยบายความปลอดภัยของเนื้อหา

Control UI มาพร้อมกับนโยบาย `img-src` ที่เข้มงวด: อนุญาตเฉพาะแอสเซ็ตจาก **same-origin**, URL `data:` และ URL `blob:` ที่สร้างขึ้นในเครื่องเท่านั้น URL รูปภาพแบบ `http(s)` ระยะไกลและแบบอ้างอิงโปรโตคอลจะถูกเบราว์เซอร์ปฏิเสธและจะไม่ส่งคำขอดึงข้อมูลผ่านเครือข่าย

ความหมายในทางปฏิบัติ:

- อวาตาร์และรูปภาพที่ให้บริการภายใต้พาธแบบสัมพัทธ์ (เช่น `/avatars/<id>`) ยังคงแสดงผลได้ รวมถึงเส้นทางอวาตาร์ที่ต้องยืนยันตัวตนซึ่ง UI ดึงข้อมูลและแปลงเป็น URL `blob:` ในเครื่อง
- URL แบบอินไลน์ `data:image/...` ยังคงแสดงผลได้ (มีประโยชน์สำหรับ payload ในโปรโตคอล)
- URL `blob:` ในเครื่องที่สร้างโดย Control UI ยังคงแสดงผลได้
- URL อวาตาร์ระยะไกลที่ปล่อยออกมาจาก metadata ของช่องทางจะถูกตัดออกที่ตัวช่วยอวาตาร์ของ Control UI และแทนที่ด้วยโลโก้/ป้ายในตัว ดังนั้นช่องทางที่ถูกบุกรุกหรือมีเจตนาร้ายจึงไม่สามารถบังคับให้เบราว์เซอร์ของ operator ดึงรูปภาพระยะไกลใดๆ ได้

คุณไม่จำเป็นต้องเปลี่ยนสิ่งใดเพื่อให้ได้พฤติกรรมนี้ — ระบบเปิดใช้งานเสมอและกำหนดค่าไม่ได้

## การยืนยันตัวตนของเส้นทางอวาตาร์

เมื่อกำหนดค่าการยืนยันตัวตนของ Gateway แล้ว endpoint อวาตาร์ของ Control UI ต้องใช้โทเค็น Gateway เดียวกับ API ส่วนที่เหลือ:

- `GET /avatar/<agentId>` จะคืนรูปภาพอวาตาร์ให้เฉพาะผู้เรียกที่ยืนยันตัวตนแล้วเท่านั้น `GET /avatar/<agentId>?meta=1` จะคืน metadata ของอวาตาร์ภายใต้กฎเดียวกัน
- คำขอที่ไม่ได้ยืนยันตัวตนไปยังเส้นทางใดเส้นทางหนึ่งจะถูกปฏิเสธ (ตรงกับเส้นทาง assistant-media ข้างเคียง) สิ่งนี้ป้องกันไม่ให้เส้นทางอวาตาร์รั่วไหลตัวตนของ agent บนโฮสต์ที่ได้รับการป้องกันอยู่แล้ว
- Control UI จะส่งต่อโทเค็น Gateway เป็น bearer header เมื่อดึงอวาตาร์ และใช้ URL blob ที่ยืนยันตัวตนแล้วเพื่อให้รูปภาพยังคงแสดงผลในแดชบอร์ด

หากคุณปิดการยืนยันตัวตนของ Gateway (ไม่แนะนำบนโฮสต์ที่ใช้ร่วมกัน) เส้นทางอวาตาร์ก็จะไม่ต้องยืนยันตัวตนเช่นกัน สอดคล้องกับส่วนที่เหลือของ Gateway

## การสร้าง UI

Gateway ให้บริการไฟล์สแตติกจาก `dist/control-ui` สร้างไฟล์เหล่านี้ด้วย:

```bash
pnpm ui:build
```

base แบบสัมบูรณ์ที่เลือกใช้ได้ (เมื่อคุณต้องการ URL แอสเซ็ตแบบคงที่):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

สำหรับการพัฒนาในเครื่อง (dev server แยกต่างหาก):

```bash
pnpm ui:dev
```

จากนั้นชี้ UI ไปยัง URL Gateway WS ของคุณ (เช่น `ws://127.0.0.1:18789`)

## การดีบัก/ทดสอบ: dev server + Gateway ระยะไกล

Control UI เป็นไฟล์สแตติก; เป้าหมาย WebSocket สามารถกำหนดค่าได้และอาจแตกต่างจาก HTTP origin วิธีนี้สะดวกเมื่อคุณต้องการใช้ Vite dev server ในเครื่อง แต่ให้ Gateway ทำงานอยู่ที่อื่น

<Steps>
  <Step title="เริ่ม UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="เปิดด้วย gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    การยืนยันตัวตนครั้งเดียวที่เลือกใช้ได้ (หากจำเป็น):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="หมายเหตุ">
    - `gatewayUrl` จะถูกเก็บไว้ใน localStorage หลังจากโหลด และถูกลบออกจาก URL
    - หากคุณส่ง endpoint `ws://` หรือ `wss://` แบบเต็มผ่าน `gatewayUrl` ให้เข้ารหัสค่า `gatewayUrl` แบบ URL-encode เพื่อให้เบราว์เซอร์แยกวิเคราะห์ query string ได้อย่างถูกต้อง
    - ควรส่ง `token` ผ่าน URL fragment (`#token=...`) เมื่อเป็นไปได้ Fragment จะไม่ถูกส่งไปยังเซิร์ฟเวอร์ จึงหลีกเลี่ยงการรั่วไหลผ่าน request-log และ Referer พารามิเตอร์ query แบบเก่า `?token=` ยังคงถูกนำเข้าเพียงครั้งเดียวเพื่อความเข้ากันได้ แต่ใช้เป็น fallback เท่านั้น และจะถูกลบทันทีหลัง bootstrap
    - `password` จะถูกเก็บไว้ในหน่วยความจำเท่านั้น
    - เมื่อตั้งค่า `gatewayUrl` แล้ว UI จะไม่ fallback ไปใช้ข้อมูลประจำตัวจาก config หรือ environment ให้ระบุ `token` (หรือ `password`) อย่างชัดเจน การไม่มีข้อมูลประจำตัวที่ระบุชัดเจนถือเป็นข้อผิดพลาด
    - ใช้ `wss://` เมื่อ Gateway อยู่หลัง TLS (Tailscale Serve, พร็อกซี HTTPS ฯลฯ)
    - `gatewayUrl` รับค่าได้เฉพาะในหน้าต่างระดับบนสุด (ไม่ใช่แบบฝัง) เพื่อป้องกัน clickjacking
    - การปรับใช้ Control UI ที่ไม่ใช่ loopback ต้องตั้งค่า `gateway.controlUi.allowedOrigins` อย่างชัดเจน (origin แบบเต็ม) ซึ่งรวมถึงการตั้งค่า dev ระยะไกลด้วย
    - การเริ่มต้น Gateway อาจใส่ origin ในเครื่อง เช่น `http://localhost:<port>` และ `http://127.0.0.1:<port>` จาก bind และ port ของ runtime ที่มีผล แต่ origin ของเบราว์เซอร์ระยะไกลยังคงต้องมีรายการระบุไว้อย่างชัดเจน
    - อย่าใช้ `gateway.controlUi.allowedOrigins: ["*"]` ยกเว้นการทดสอบในเครื่องที่ควบคุมอย่างเข้มงวด หมายถึงอนุญาต origin ของเบราว์เซอร์ใดก็ได้ ไม่ใช่ "จับคู่กับโฮสต์ใดก็ตามที่ฉันใช้อยู่"
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` เปิดใช้โหมด fallback origin จาก Host-header แต่เป็นโหมดความปลอดภัยที่อันตราย

  </Accordion>
</AccordionGroup>

ตัวอย่าง:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
