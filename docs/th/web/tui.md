## เริ่มต้นอย่างรวดเร็ว

### โหมด Gateway

1. เริ่ม Gateway

```bash
openclaw gateway
```

2. เปิด TUI

```bash
openclaw tui
```

3. พิมพ์ข้อความแล้วกด Enter

Gateway ระยะไกล:

```bash
openclaw tui --url ws://<host>:<port> --token <gateway-token>
```

ใช้ `--password` หาก Gateway ใช้การยืนยันตัวตนด้วยรหัสผ่าน

### โหมดภายในเครื่อง

เรียกใช้ TUI โดยไม่มี Gateway:

```bash
openclaw chat
# หรือ
openclaw tui --local
```

- `openclaw chat` และ `openclaw terminal` เป็นนามแฝงของ `openclaw tui --local`
- `--local` ไม่สามารถใช้ร่วมกับ `--url`, `--token` หรือ `--password`
- โหมดภายในเครื่องใช้รันไทม์เอเจนต์แบบฝังโดยตรง เครื่องมือภายในเครื่องส่วนใหญ่ใช้งานได้ แต่ฟีเจอร์ที่ใช้ได้เฉพาะกับ Gateway จะไม่พร้อมใช้งาน
- เมื่อใช้เพียง `openclaw` (ไม่มีคำสั่งย่อย) ระบบจะเลือกเป้าหมายโดยอัตโนมัติ: การติดตั้งที่ยังไม่ได้กำหนดค่าจะเริ่มกระบวนการเริ่มต้นใช้งานการอนุมาน การกำหนดค่าที่ไม่ถูกต้องจะเปิดคำแนะนำ Doctor แบบคลาสสิก Gateway ที่กำหนดค่าไว้และเข้าถึงได้จะเปิดเชลล์ TUI นี้ในโหมด Gateway มิฉะนั้นโมเดลภายในเครื่องที่กำหนดค่าไว้จะเปิดเชลล์นี้ในโหมดภายในเครื่อง

## สิ่งที่เห็น

- ส่วนหัว: URL การเชื่อมต่อ เอเจนต์ปัจจุบัน เซสชันปัจจุบัน
- บันทึกแชต: ข้อความของผู้ใช้ คำตอบของผู้ช่วย การแจ้งเตือนระบบ การ์ดเครื่องมือ
- บรรทัดสถานะ: สถานะการเชื่อมต่อ/การทำงาน (กำลังเชื่อมต่อ กำลังทำงาน กำลังสตรีม ไม่ทำงาน ข้อผิดพลาด)
- ส่วนท้าย: เอเจนต์ + เซสชัน + โมเดล + สถานะเป้าหมาย + think/fast/verbose/trace/reasoning + จำนวนโทเค็น + การส่งมอบ เมื่อเปิดใช้ `tui.footer.showRemoteHost` การเชื่อมต่อ Gateway ระยะไกลจะแสดงโฮสต์การเชื่อมต่อด้วย
- ช่องป้อนข้อมูล: โปรแกรมแก้ไขข้อความพร้อมการเติมข้อความอัตโนมัติ

## แบบจำลองทางความคิด: เอเจนต์ + เซสชัน

- เอเจนต์คือ slug ที่ไม่ซ้ำกัน (เช่น `main`, `research`) Gateway จะแสดงรายการดังกล่าว
- เซสชันเป็นของเอเจนต์ปัจจุบัน
- คีย์เซสชันจัดเก็บในรูปแบบ `agent:<agentId>:<sessionKey>`
  - หากพิมพ์ `/session main` TUI จะขยายเป็น `agent:<currentAgent>:main`
  - หากพิมพ์ `/session agent:other:main` ระบบจะสลับไปยังเซสชันของเอเจนต์นั้นโดยระบุอย่างชัดเจน
- ขอบเขตเซสชัน:
  - `per-sender` (ค่าเริ่มต้น): แต่ละเอเจนต์มีหลายเซสชัน
  - `global`: TUI จะใช้เซสชัน `global` เสมอ (ตัวเลือกอาจว่างเปล่า)
- เอเจนต์และเซสชันปัจจุบันจะแสดงอยู่ในส่วนท้ายเสมอ
- หากต้องการแสดงโฮสต์ Gateway สำหรับการเชื่อมต่อที่อิง URL และไม่ใช่ภายในเครื่อง ให้เลือกเปิดใช้ด้วย:

  ```bash
  openclaw config set tui.footer.showRemoteHost true
  ```

  ค่าเริ่มต้นคือ `false` การเชื่อมต่อแบบลูปแบ็กและแบบฝังภายในเครื่องจะไม่แสดงป้ายกำกับโฮสต์

- หากเซสชันมี [เป้าหมาย](/th/tools/goal) ส่วนท้ายจะแสดงสถานะแบบย่อ:
  `Pursuing goal`, `Goal paused (/goal resume)`, `Goal blocked (/goal resume)` หรือ `Goal achieved`
- เมื่อเริ่มโดยไม่มี `--session` TUI ในโหมด Gateway จะกลับมาใช้เซสชันที่เลือกล่าสุดสำหรับ Gateway เอเจนต์ และขอบเขตเซสชันเดียวกัน หากเซสชันนั้นยังมีอยู่ การส่ง `--session`, `/session`, `/new` หรือ `/reset` ยังคงเป็นการระบุอย่างชัดเจน

## การส่ง + การส่งมอบ

- ข้อความจะส่งไปยัง Gateway เสมอ (หรือรันไทม์แบบฝังในโหมดภายในเครื่อง) การส่งคำตอบของผู้ช่วยกลับออกไปยังผู้ให้บริการแชตเป็นขั้นตอนแยกต่างหากซึ่งปิดไว้โดยค่าเริ่มต้น
- TUI เป็นพื้นผิวต้นทางภายในเช่นเดียวกับ WebChat ไม่ใช่ช่องทางขาออกทั่วไป ชุดทดสอบที่ต้องใช้ `tools.message` เพื่อให้เห็นคำตอบสามารถตอบสนองเทิร์น TUI ที่ทำงานอยู่ด้วย `message.send` ที่ไม่มีเป้าหมายได้ ส่วนการส่งมอบไปยังผู้ให้บริการที่ระบุอย่างชัดเจนยังคงใช้ช่องทางที่กำหนดค่าตามปกติและจะไม่ย้อนกลับไปใช้ `lastChannel`
- การส่งมอบถูกกำหนดตายตัวสำหรับเซสชัน TUI ทั้งหมดตั้งแต่เปิดใช้งาน: เริ่มด้วย `openclaw tui --deliver` เพื่อเปิดใช้ ไม่มีคำสั่งสแลช `/deliver` หรือสวิตช์ Settings สำหรับเปลี่ยนระหว่างเซสชัน ให้เริ่ม TUI ใหม่เพื่อเปลี่ยนค่า

## ตัวเลือก + โอเวอร์เลย์

- ตัวเลือกโมเดล: แสดงรายการโมเดลที่พร้อมใช้งานและตั้งค่าการแทนที่ของเซสชัน
- ตัวเลือกเอเจนต์: เลือกเอเจนต์อื่น
- ตัวเลือกเซสชัน: แสดงเซสชันสูงสุด 50 รายการของเอเจนต์ปัจจุบันที่ได้รับการอัปเดตภายใน 7 วันที่ผ่านมา ใช้ `/session <key>` เพื่อไปยังเซสชันเก่าที่ทราบ
- Settings (`/settings`): สลับการขยายเอาต์พุตเครื่องมือและการแสดงกระบวนการคิด แผงนี้ไม่ได้ควบคุมการส่งมอบ

## แป้นพิมพ์ลัด

- Enter: ส่งข้อความ
- Esc: ยกเลิกการทำงานที่กำลังดำเนินอยู่
- Ctrl+C: ล้างช่องป้อนข้อมูล (กดสองครั้งเพื่อออก)
- Ctrl+D: ออก
- Ctrl+L: ตัวเลือกโมเดล
- Ctrl+G: ตัวเลือกเอเจนต์
- Ctrl+P: ตัวเลือกเซสชัน
- Ctrl+O: สลับการขยายเอาต์พุตเครื่องมือ
- Ctrl+T: สลับการแสดงกระบวนการคิด (โหลดประวัติใหม่)

## คำสั่งเชลล์ภายในเครื่อง

- เติม `!` หน้าบรรทัดเพื่อเรียกใช้คำสั่งเชลล์ภายในเครื่องบนโฮสต์ TUI
- TUI จะถามหนึ่งครั้งต่อเซสชันเพื่ออนุญาตการดำเนินการภายในเครื่อง หากปฏิเสธ `!` จะยังคงปิดใช้งานตลอดเซสชัน
- คำสั่งจะทำงานในเชลล์ใหม่แบบไม่โต้ตอบในไดเรกทอรีทำงานของ TUI (ไม่มี `cd`/env ที่คงอยู่)
- คำสั่งเชลล์ภายในเครื่องจะได้รับ `OPENCLAW_SHELL=tui-local` ในสภาพแวดล้อม
- หากมีเพียง `!` จะถูกส่งเป็นข้อความปกติ การเว้นวรรคนำหน้าจะไม่เรียกใช้การดำเนินการภายในเครื่อง

## ตัวช่วยตั้งค่าและซ่อมแซม OpenClaw

OpenClaw เป็นผู้ช่วยตั้งค่า/ซ่อมแซมระดับ ring-zero ซึ่งเปิดให้ใช้เป็น `openclaw setup` หลังจากโมเดลเริ่มต้นที่กำหนดค่าผ่านการตรวจสอบการอนุมานจริง หากการอนุมานไม่พร้อมใช้งาน การเรียกใช้แบบโต้ตอบจะกลับไปยังกระบวนการเริ่มต้นใช้งานการอนุมาน และระบบอัตโนมัติจะล้มเหลวพร้อมคำแนะนำการซ่อมแซม ผู้ช่วยทำงานภายในเชลล์ TUI ภายในเครื่องเดียวกับ `openclaw tui --local` โดยมีเอเจนต์ AI ที่ถูกจำกัดให้ใช้เฉพาะการดำเนินการแบบมีชนิดของ OpenClaw ซึ่งต้องผ่านการอนุมัติ:

```bash
openclaw setup                       # เริ่มแบบโต้ตอบ
openclaw setup -m "status"           # เรียกใช้หนึ่งคำขอแล้วออก
openclaw setup -m "set default model openai/gpt-5.2" --yes   # ใช้การเขียนค่ากำหนด
```

- การเขียนค่ากำหนดแบบถาวรต้องได้รับการอนุมัติ: ยืนยันแบบโต้ตอบหรือส่ง `--yes`
- `--json` จะแสดงภาพรวมการเริ่มต้นเป็น JSON แทนการเริ่มแชต
- จากภายใน OpenClaw คำขอ `open-tui` (ตัวอย่างเช่น ขอคุยกับเอเจนต์ปกติ) จะออกจาก OpenClaw และเปิด TUI เอเจนต์ปกติ ใช้ `/openclaw` ที่นั่นเพื่อกลับมา

ใช้โหมดภายในเครื่องเมื่อการกำหนดค่าปัจจุบันผ่านการตรวจสอบแล้ว และต้องการให้เอเจนต์แบบฝังตรวจสอบการกำหนดค่าบนเครื่องเดียวกัน เปรียบเทียบกับเอกสาร และช่วยซ่อมแซมความคลาดเคลื่อนโดยไม่ต้องพึ่ง Gateway ที่กำลังทำงาน

หาก `openclaw config validate` ล้มเหลวอยู่แล้ว ให้เริ่มด้วย `openclaw configure` หรือ `openclaw doctor --fix` ก่อน เนื่องจาก `openclaw chat` ยังต้องใช้การกำหนดค่าที่โหลดได้เพื่อเริ่มทำงาน

วงจรการใช้งานทั่วไป:

1. เริ่มโหมดภายในเครื่อง:

```bash
openclaw chat
```

2. บอกเอเจนต์ว่าต้องการตรวจสอบอะไร ตัวอย่างเช่น:

```text
เปรียบเทียบการกำหนดค่าการยืนยันตัวตนของ Gateway ของฉันกับเอกสาร และแนะนำการแก้ไขที่เล็กที่สุด
```

3. ใช้คำสั่งเชลล์ภายในเครื่องเพื่อรับหลักฐานและตรวจสอบอย่างแม่นยำ:

```text
!openclaw config file
!openclaw docs gateway auth token secretref
!openclaw config validate
!openclaw doctor
```

4. ใช้การเปลี่ยนแปลงแบบเฉพาะเจาะจงด้วย `openclaw config set` หรือ `openclaw configure` แล้วเรียกใช้ `!openclaw config validate` อีกครั้ง
5. หาก Doctor แนะนำการย้ายข้อมูลหรือซ่อมแซมอัตโนมัติ ให้ตรวจสอบแล้วเรียกใช้ `!openclaw doctor --fix`

เคล็ดลับ:

- ควรใช้ `openclaw config set` หรือ `openclaw configure` แทนการแก้ไข `openclaw.json` ด้วยตนเอง
- `openclaw docs "<query>"` ค้นหาดัชนีเอกสารปัจจุบันจากเครื่องเดียวกัน
- `openclaw config validate --json` มีประโยชน์เมื่อต้องการสคีมาแบบมีโครงสร้าง รวมถึงข้อผิดพลาด SecretRef/ความสามารถในการแก้ไขค่า

## เอาต์พุตเครื่องมือ

- การเรียกใช้เครื่องมือจะแสดงเป็นการ์ดพร้อมอาร์กิวเมนต์ + ผลลัพธ์
- Ctrl+O สลับระหว่างมุมมองแบบยุบ/ขยาย
- ขณะที่เครื่องมือทำงาน การอัปเดตบางส่วนจะสตรีมเข้ามาในการ์ดเดียวกัน

## สีเทอร์มินัล

- TUI ใช้สีข้อความเนื้อหาของผู้ช่วยตามสีพื้นหน้าค่าเริ่มต้นของเทอร์มินัล เพื่อให้ทั้งเทอร์มินัลพื้นมืดและพื้นสว่างอ่านได้
- หากเทอร์มินัลใช้พื้นหลังสีสว่างและการตรวจจับอัตโนมัติไม่ถูกต้อง ให้ตั้งค่า `OPENCLAW_THEME=light` ก่อนเปิด `openclaw tui`
- หากต้องการบังคับใช้ชุดสีเข้มแบบเดิม ให้ตั้งค่า `OPENCLAW_THEME=dark`

## ประวัติ + การสตรีม

- เมื่อเชื่อมต่อ TUI จะโหลดประวัติล่าสุด (ค่าเริ่มต้น 200 ข้อความ)
- คำตอบแบบสตรีมจะอัปเดตในตำแหน่งเดิมจนเสร็จสมบูรณ์
- TUI ยังรับฟังเหตุการณ์เครื่องมือของเอเจนต์เพื่อแสดงการ์ดเครื่องมือที่มีรายละเอียดมากขึ้น

## รายละเอียดการเชื่อมต่อ

- TUI เชื่อมต่อด้วยรหัสไคลเอนต์ `openclaw-tui` ภายใต้โหมดไคลเอนต์ระดับกว้าง `ui` (เป็นโหมดเดียวกับที่ Control UI และ WebChat ใช้สำหรับนโยบาย Gateway)
- เมื่อเชื่อมต่อใหม่ ระบบจะแสดงข้อความระบบ ส่วนช่วงเหตุการณ์ที่ขาดหายจะแสดงในบันทึก

## ตัวเลือก

- `--local`: เรียกใช้กับรันไทม์เอเจนต์แบบฝังภายในเครื่อง
- `--url <url>`: URL WebSocket ของ Gateway (ค่าเริ่มต้นคือ `gateway.remote.url` จากการกำหนดค่า หรือ `ws://127.0.0.1:<port>` บนลูปแบ็ก)
- `--token <token>`: โทเค็น Gateway (หากจำเป็น)
- `--password <password>`: รหัสผ่าน Gateway (หากจำเป็น)
- `--tls-fingerprint <sha256>`: ลายนิ้วมือใบรับรอง TLS ที่คาดไว้สำหรับ Gateway `wss://` ที่ปักหมุดไว้
- `--session <key>`: คีย์เซสชัน (ค่าเริ่มต้น: `main` หรือ `global` เมื่อขอบเขตเป็นแบบส่วนกลาง)
- `--deliver`: ส่งคำตอบของผู้ช่วยไปยังผู้ให้บริการ (ค่าเริ่มต้นปิดอยู่)
- `--thinking <level>`: แทนที่ระดับการคิดสำหรับการส่ง
- `--message <text>`: ส่งข้อความเริ่มต้นหลังจากเชื่อมต่อ
- `--timeout-ms <ms>`: ระยะหมดเวลาของเอเจนต์เป็นมิลลิวินาที (ค่าเริ่มต้นคือ `agents.defaults.timeoutSeconds`)
- `--history-limit <n>`: จำนวนรายการประวัติที่จะโหลด (ค่าเริ่มต้น `200`)

<Warning>
เมื่อตั้งค่า `--url` แล้ว TUI จะไม่ย้อนกลับไปใช้ข้อมูลประจำตัวจากการกำหนดค่าหรือตัวแปรสภาพแวดล้อม ให้ส่ง `--token` หรือ `--password` อย่างชัดเจน รวมถึง `--tls-fingerprint` เมื่อเป้าหมายใช้ใบรับรองที่ปักหมุดไว้ การไม่ระบุข้อมูลประจำตัวอย่างชัดเจนถือเป็นข้อผิดพลาด ในโหมดภายในเครื่อง อย่าส่ง `--url`, `--token`, `--password` หรือ `--tls-fingerprint`
</Warning>

## การแก้ไขปัญหา

ไม่มีเอาต์พุตหลังจากส่งข้อความ:

- เรียกใช้ `/status` ใน TUI เพื่อยืนยันว่า Gateway เชื่อมต่อแล้วและอยู่ในสถานะว่าง/ไม่ว่าง
- ตรวจสอบบันทึกของ Gateway: `openclaw logs --follow`
- ยืนยันว่าเอเจนต์สามารถทำงานได้: `openclaw status` และ `openclaw models status`
- หากคาดว่าจะได้รับข้อความในช่องทางแชต ให้ยืนยันว่า TUI เริ่มต้นด้วย `--deliver` (ไม่สามารถเปิดใช้ภายหลังได้โดยไม่เริ่มการทำงานใหม่)

## การแก้ไขปัญหาการเชื่อมต่อ

- `disconnected`: ตรวจสอบว่า Gateway กำลังทำงานและ `--url/--token/--password` ของคุณถูกต้อง
- ไม่มีเอเจนต์ในตัวเลือก: ตรวจสอบ `openclaw agents list` และการกำหนดค่าการกำหนดเส้นทางของคุณ
- ตัวเลือกเซสชันว่างเปล่า: คุณอาจอยู่ในขอบเขตส่วนกลางหรือยังไม่มีเซสชัน
