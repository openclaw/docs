# การค้นหาและการขนส่ง

OpenClaw มีปัญหาสองอย่างที่ดูคล้ายกันเมื่อมองผิวเผิน:

1. **การควบคุมระยะไกลของผู้ปฏิบัติงาน**: แอปแถบเมนู macOS ที่ควบคุม Gateway ซึ่งทำงานอยู่ที่อื่น
2. **การจับคู่ Node**: iOS/Android (และ Node ในอนาคต) ค้นหา Gateway และจับคู่อย่างปลอดภัย

เป้าหมายของการออกแบบคือเก็บการค้นหา/การประกาศผ่านเครือข่ายทั้งหมดไว้ใน **Node Gateway** (`openclaw gateway`) และให้ไคลเอนต์ (แอป mac, iOS) เป็นผู้บริโภค

## คำศัพท์

- **Gateway**: กระบวนการ Gateway ที่ทำงานระยะยาวเพียงตัวเดียว ซึ่งเป็นเจ้าของสถานะ (เซสชัน การจับคู่ รีจิสทรี Node) และเรียกใช้ช่องทาง การตั้งค่าส่วนใหญ่ใช้หนึ่งตัวต่อโฮสต์ แต่การตั้งค่าแบบหลาย Gateway ที่แยกกันก็ทำได้
- **Gateway WS (ระนาบควบคุม)**: เอนด์พอยต์ WebSocket บน `127.0.0.1:18789` โดยค่าเริ่มต้น สามารถผูกกับ LAN/tailnet ได้ผ่าน `gateway.bind`
- **การขนส่ง Direct WS**: เอนด์พอยต์ Gateway WS ที่เปิดรับฝั่ง LAN/tailnet (ไม่มี SSH)
- **การขนส่ง SSH (ทางเลือกสำรอง)**: การควบคุมระยะไกลโดยส่งต่อ `127.0.0.1:18789` ผ่าน SSH
- **บริดจ์ TCP ดั้งเดิม (ถูกนำออกแล้ว)**: การขนส่ง Node แบบเก่า (ดู
  [โปรโตคอลบริดจ์](/th/gateway/bridge-protocol)); ไม่ได้ถูกประกาศสำหรับ
  การค้นหาอีกต่อไป และไม่เป็นส่วนหนึ่งของบิลด์ปัจจุบันแล้ว

รายละเอียดโปรโตคอล:

- [โปรโตคอล Gateway](/th/gateway/protocol)
- [โปรโตคอลบริดจ์ (ดั้งเดิม)](/th/gateway/bridge-protocol)

## เหตุผลที่เราคงไว้ทั้ง "direct" และ SSH

- **Direct WS** ให้ UX ที่ดีที่สุดบนเครือข่ายเดียวกันและภายใน tailnet:
  - ค้นหาอัตโนมัติบน LAN ผ่าน Bonjour
  - โทเค็นการจับคู่ + ACL ที่ Gateway เป็นเจ้าของ
  - ไม่ต้องใช้สิทธิ์เข้าถึงเชลล์ พื้นผิวโปรโตคอลจึงคงให้แคบและตรวจสอบได้
- **SSH** ยังคงเป็นทางเลือกสำรองสากล:
  - ใช้ได้ทุกที่ที่คุณมีสิทธิ์เข้าถึง SSH (แม้ข้ามเครือข่ายที่ไม่เกี่ยวข้องกัน)
  - ทนต่อปัญหา multicast/mDNS
  - ไม่ต้องมีพอร์ตขาเข้าใหม่นอกจาก SSH

## อินพุตการค้นหา (ไคลเอนต์เรียนรู้ตำแหน่ง Gateway ได้อย่างไร)

### 1) การค้นหา Bonjour / DNS-SD

Bonjour แบบ multicast เป็นแบบพยายามอย่างดีที่สุดและไม่ข้ามเครือข่าย OpenClaw ยังสามารถเรียกดู
บีคอน Gateway เดียวกันผ่านโดเมน DNS-SD แบบ wide-area ที่กำหนดค่าไว้ได้ ดังนั้นการค้นหาจึงครอบคลุมได้ดังนี้:

- `local.` บน LAN เดียวกัน
- โดเมน DNS-SD แบบ unicast ที่กำหนดค่าไว้สำหรับการค้นหาข้ามเครือข่าย

ทิศทางเป้าหมาย:

- **Gateway** ประกาศเอนด์พอยต์ WS ของตนผ่าน Bonjour
- ไคลเอนต์เรียกดูและแสดงรายการ “เลือก Gateway” จากนั้นจัดเก็บเอนด์พอยต์ที่เลือก

รายละเอียดการแก้ไขปัญหาและบีคอน: [Bonjour](/th/gateway/bonjour)

#### รายละเอียดบีคอนบริการ

- ประเภทบริการ:
  - `_openclaw-gw._tcp` (บีคอนการขนส่ง Gateway)
- คีย์ TXT (ไม่ใช่ความลับ):
  - `role=gateway`
  - `transport=gateway`
  - `displayName=<friendly name>` (ชื่อที่แสดงซึ่งผู้ปฏิบัติงานกำหนดค่า)
  - `lanHost=<hostname>.local`
  - `gatewayPort=18789` (Gateway WS + HTTP)
  - `gatewayTls=1` (เฉพาะเมื่อเปิดใช้ TLS)
  - `gatewayTlsSha256=<sha256>` (เฉพาะเมื่อเปิดใช้ TLS และมีลายนิ้วมือพร้อมใช้งาน)
  - `canvasPort=<port>` (พอร์ตโฮสต์ canvas; ขณะนี้เหมือนกับ `gatewayPort` เมื่อเปิดใช้โฮสต์ canvas)
  - `tailnetDns=<magicdns>` (คำใบ้เสริม; ตรวจพบอัตโนมัติเมื่อ Tailscale พร้อมใช้งาน)
  - `sshPort=<port>` (โหมด mDNS full เท่านั้น; DNS-SD แบบ wide-area อาจละเว้น ซึ่งในกรณีนั้นค่าเริ่มต้น SSH จะยังคงอยู่ที่ `22`)
  - `cliPath=<path>` (โหมด mDNS full เท่านั้น; DNS-SD แบบ wide-area ยังเขียนค่านี้เป็นคำใบ้การติดตั้งระยะไกล)

หมายเหตุด้านความปลอดภัย:

- ระเบียน TXT ของ Bonjour/mDNS **ไม่ได้รับการยืนยันตัวตน** ไคลเอนต์ต้องถือค่าของ TXT เป็นเพียงคำใบ้ด้าน UX เท่านั้น
- การกำหนดเส้นทาง (โฮสต์/พอร์ต) ควรเลือกใช้ **เอนด์พอยต์บริการที่ resolve แล้ว** (SRV + A/AAAA) แทน `lanHost`, `tailnetDns` หรือ `gatewayPort` ที่มาจาก TXT
- การ pin TLS ต้องไม่อนุญาตให้ `gatewayTlsSha256` ที่ถูกประกาศมาแทนที่ pin ที่เคยจัดเก็บไว้ก่อนหน้า
- Node iOS/Android ควรกำหนดให้มีการยืนยันอย่างชัดเจนว่า “เชื่อถือลายนิ้วมือนี้” ก่อนจัดเก็บ pin ครั้งแรก (การตรวจสอบนอกช่องทาง) ทุกครั้งที่เส้นทางที่เลือกเป็นแบบปลอดภัย/อิง TLS

ปิดใช้/แทนที่:

- `OPENCLAW_DISABLE_BONJOUR=1` ปิดการประกาศ
- เมื่อไม่ได้ตั้งค่า `OPENCLAW_DISABLE_BONJOUR` Bonjour จะประกาศบนโฮสต์ปกติ
  และปิดตัวเองอัตโนมัติภายในคอนเทนเนอร์ที่ตรวจพบ ใช้ `0` เฉพาะบนโฮสต์, macvlan,
  หรือเครือข่ายอื่นที่รองรับ mDNS; ใช้ `1` เพื่อบังคับปิด
- `gateway.bind` ใน `~/.openclaw/openclaw.json` ควบคุมโหมดการ bind ของ Gateway
- `OPENCLAW_SSH_PORT` แทนที่พอร์ต SSH ที่ประกาศเมื่อมีการปล่อย `sshPort`
- `OPENCLAW_TAILNET_DNS` เผยแพร่คำใบ้ `tailnetDns` (MagicDNS)
- `OPENCLAW_CLI_PATH` แทนที่พาธ CLI ที่ประกาศ

### 2) Tailnet (ข้ามเครือข่าย)

สำหรับการตั้งค่าแบบ London/Vienna, Bonjour จะไม่ช่วย เป้าหมาย “direct” ที่แนะนำคือ:

- ชื่อ Tailscale MagicDNS (แนะนำ) หรือ IP tailnet ที่เสถียร

หาก Gateway ตรวจพบว่ากำลังทำงานภายใต้ Tailscale จะเผยแพร่ `tailnetDns` เป็นคำใบ้เสริมสำหรับไคลเอนต์ (รวมถึงบีคอนแบบ wide-area)

ขณะนี้แอป macOS เลือกใช้ชื่อ MagicDNS มากกว่า IP Tailscale แบบดิบสำหรับการค้นหา Gateway ซึ่งช่วยเพิ่มความน่าเชื่อถือเมื่อ IP tailnet เปลี่ยน (เช่น หลังจากรีสตาร์ต Node หรือมีการกำหนด CGNAT ใหม่) เพราะชื่อ MagicDNS จะ resolve ไปยัง IP ปัจจุบันโดยอัตโนมัติ

สำหรับการจับคู่ Node มือถือ คำใบ้การค้นหาจะไม่ผ่อนคลายความปลอดภัยของการขนส่งบนเส้นทาง tailnet/public:

- iOS/Android ยังคงต้องใช้เส้นทางการเชื่อมต่อ tailnet/public ครั้งแรกที่ปลอดภัย (`wss://` หรือ Tailscale Serve/Funnel)
- IP tailnet แบบดิบที่ค้นพบเป็นคำใบ้การกำหนดเส้นทาง ไม่ใช่สิทธิ์ในการใช้ `ws://` ระยะไกลแบบข้อความธรรมดา
- ยังรองรับการเชื่อมต่อโดยตรงผ่าน LAN ส่วนตัวด้วย `ws://`
- หากคุณต้องการเส้นทาง Tailscale ที่ง่ายที่สุดสำหรับ Node มือถือ ให้ใช้ Tailscale Serve เพื่อให้ทั้งการค้นหาและโค้ดตั้งค่า resolve ไปยังเอนด์พอยต์ MagicDNS ที่ปลอดภัยเดียวกัน

### 3) เป้าหมายแบบ Manual / SSH

เมื่อไม่มีเส้นทาง direct (หรือ direct ถูกปิดใช้) ไคลเอนต์สามารถเชื่อมต่อผ่าน SSH ได้เสมอโดยส่งต่อพอร์ต Gateway แบบ loopback

ดู [การเข้าถึงระยะไกล](/th/gateway/remote)

## การเลือกการขนส่ง (นโยบายไคลเอนต์)

พฤติกรรมไคลเอนต์ที่แนะนำ:

1. หากมีการกำหนดค่าเอนด์พอยต์ direct ที่จับคู่ไว้และเข้าถึงได้ ให้ใช้เอนด์พอยต์นั้น
2. มิฉะนั้น หากการค้นหาพบ Gateway บน `local.` หรือโดเมน wide-area ที่กำหนดค่าไว้ ให้เสนอทางเลือกแบบแตะครั้งเดียว “ใช้ Gateway นี้” และบันทึกเป็นเอนด์พอยต์ direct
3. มิฉะนั้น หากมีการกำหนดค่า DNS/IP ของ tailnet ให้ลอง direct
   สำหรับ Node มือถือบนเส้นทาง tailnet/public, direct หมายถึงเอนด์พอยต์ที่ปลอดภัย ไม่ใช่ `ws://` ระยะไกลแบบข้อความธรรมดา
4. มิฉะนั้น ให้ถอยกลับไปใช้ SSH

## การจับคู่ + การยืนยันตัวตน (การขนส่ง direct)

Gateway คือแหล่งข้อมูลจริงสำหรับการรับ Node/ไคลเอนต์เข้าระบบ

- คำขอจับคู่ถูกสร้าง/อนุมัติ/ปฏิเสธใน Gateway (ดู [การจับคู่ Gateway](/th/gateway/pairing))
- Gateway บังคับใช้:
  - การยืนยันตัวตน (โทเค็น / คู่กุญแจ)
  - ขอบเขต/ACL (Gateway ไม่ใช่ proxy ดิบไปยังทุกเมธอด)
  - การจำกัดอัตรา

## ความรับผิดชอบตามคอมโพเนนต์

- **Gateway**: ประกาศบีคอนการค้นหา เป็นเจ้าของการตัดสินใจจับคู่ และโฮสต์เอนด์พอยต์ WS
- **แอป macOS**: ช่วยคุณเลือก Gateway แสดงพรอมต์การจับคู่ และใช้ SSH เป็นทางเลือกสำรองเท่านั้น
- **Node iOS/Android**: เรียกดู Bonjour เพื่อความสะดวก และเชื่อมต่อกับ Gateway WS ที่จับคู่แล้ว
