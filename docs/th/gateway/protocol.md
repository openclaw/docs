Gateway WS protocol เป็นระนาบควบคุมเดียวและการขนส่ง Node สำหรับ
OpenClaw ไคลเอนต์ของผู้ควบคุมและ Node (CLI, เว็บ UI, แอป macOS, Node iOS/Android,
Node แบบไม่มีส่วนติดต่อผู้ใช้) เชื่อมต่อผ่าน WebSocket และประกาศ **บทบาท** กับ **ขอบเขต** ใน
ระหว่างการจับมือเชื่อมต่อ

## การขนส่งและการจัดเฟรม

- WebSocket, เฟรมข้อความ, เพย์โหลด JSON
- เฟรมแรก **ต้อง** เป็นคำขอ `connect`
- เฟรมก่อนเชื่อมต่อถูกจำกัดไว้ที่ 64 KiB (`MAX_PREAUTH_PAYLOAD_BYTES`) หลัง
  การจับมือเชื่อมต่อ ให้ปฏิบัติตาม `hello-ok.policy.maxPayload` และ
  `hello-ok.policy.maxBufferedBytes` เมื่อเปิดใช้การวินิจฉัย เฟรมขาเข้าที่มีขนาดเกินกำหนด
  และบัฟเฟอร์ขาออกที่ทำงานช้าจะส่งเหตุการณ์ `payload.large` ก่อนที่
  Gateway จะปิดการเชื่อมต่อหรือทิ้งเฟรม เหตุการณ์เหล่านี้มี `surface`, ขนาดเป็นไบต์,
  ขีดจำกัด และรหัสเหตุผลที่ปลอดภัย โดยจะไม่มีเนื้อหาข้อความ เนื้อหาไฟล์แนบ
  ไบต์ดิบของเฟรม โทเค็น คุกกี้ หรือข้อมูลลับ

รูปแบบเฟรม:

- คำขอ: `{type:"req", id, method, params}`
- การตอบกลับ: `{type:"res", id, ok, payload|error}`
- เหตุการณ์: `{type:"event", event, payload, seq?, stateVersion?}`

เมธอดที่ก่อให้เกิดผลข้างเคียงต้องใช้คีย์ idempotency (ดูสคีมา)

## การจับมือเชื่อมต่อ

Gateway ส่งคำท้าทายก่อนเชื่อมต่อ:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

ไคลเอนต์ตอบกลับด้วย `connect`:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway ตอบกลับด้วย `hello-ok`:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`, `features`, `snapshot`, `policy` และ `auth` ล้วนเป็นค่าที่
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) กำหนดให้ต้องมี `auth`
รายงานบทบาท/ขอบเขตที่เจรจาตกลงกัน แม้จะไม่มีการออกโทเค็นอุปกรณ์ (รูปแบบ
ข้างต้น) `pluginSurfaceUrls` เป็นตัวเลือกและแมปชื่อพื้นผิว Plugin (เช่น
`canvas`) ไปยัง URL แบบโฮสต์ที่มีการกำหนดขอบเขต ซึ่งอาจหมดอายุได้ ดังนั้น Node จึงเรียก
`node.pluginSurface.refresh` พร้อม `{ "surface": "canvas" }` เพื่อรับรายการใหม่
เส้นทาง `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
ที่เลิกใช้แล้วไม่รองรับ ให้ใช้พื้นผิว Plugin
`appliedConfigHash` ซึ่งเป็นตัวเลือกของสแนปช็อต คือรีวิชันการกำหนดค่าต้นทางที่แก้ไขแล้ว
ซึ่งรันไทม์ Gateway ที่ใช้งานอยู่ยอมรับ ไคลเอนต์สามารถเปรียบเทียบค่านี้กับ
`config.get.configRevisionHash` เพื่อพิจารณาว่าการกำหนดค่าที่บันทึกใหม่กว่ายังคง
ต้องรีสตาร์ตหรือไม่ `config.get.hash` ยังคงเป็นรีวิชันไฟล์รากดิบที่ใช้โดย
ตัวป้องกันความขัดแย้งในการเขียนการกำหนดค่า

ระหว่างที่ Gateway ยังคงเริ่มต้นไซด์คาร์ให้เสร็จสิ้น `connect` อาจส่งคืน
ข้อผิดพลาด `UNAVAILABLE` ที่ลองใหม่ได้ พร้อม `details.reason: "startup-sidecars"` และ
`retryAfterMs` ให้ลองใหม่ภายในงบประมาณเวลาการเชื่อมต่อ แทนที่จะถือว่าเป็น
ความล้มเหลวขั้นสุดท้ายของการจับมือเชื่อมต่อ

เมื่อมีการออกโทเค็นอุปกรณ์ `hello-ok.auth` จะเพิ่มโทเค็นดังกล่าว:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

บูตสแตรปด้วย QR/รหัสตั้งค่าที่มีมาให้ในตัวเป็นเส้นทางส่งต่อสำหรับอุปกรณ์เคลื่อนที่ การเชื่อมต่อ
ด้วยรหัสตั้งค่าพื้นฐานที่สำเร็จจะส่งคืนโทเค็น Node หลักพร้อมโทเค็น
ผู้ควบคุมที่มีขอบเขตจำกัดหนึ่งรายการ:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

การส่งต่อให้ผู้ควบคุมนี้มีขอบเขตจำกัดโดยตั้งใจ: เพียงพอสำหรับเริ่มลูปผู้ควบคุม
บนอุปกรณ์เคลื่อนที่และการตั้งค่าแบบเนทีฟ รวมถึง `operator.talk.secrets` สำหรับการอ่าน
การกำหนดค่า Talk แต่ไม่มีขอบเขตสำหรับแก้ไขการจับคู่และไม่มี `operator.admin` การเข้าถึง
การจับคู่/ผู้ดูแลระบบที่กว้างกว่านี้ต้องผ่านขั้นตอนการจับคู่หรือโทเค็นที่ได้รับอนุมัติแยกต่างหาก ให้คงข้อมูล
`hello-ok.auth.deviceTokens` ไว้เฉพาะเมื่อการยืนยันตัวตนแบบบูตสแตรปทำงานผ่านการขนส่งที่เชื่อถือได้
(`wss://` หรือการจับคู่ผ่านลูปแบ็ก/ภายในเครื่อง)

ไคลเอนต์แบ็กเอนด์ในโปรเซสเดียวกันที่เชื่อถือได้ (`client.id: "gateway-client"`,
`client.mode: "backend"`) อาจไม่ระบุ `device` ในการเชื่อมต่อลูปแบ็กโดยตรงเมื่อ
ยืนยันตัวตนด้วยโทเค็น/รหัสผ่าน Gateway ที่ใช้ร่วมกัน เส้นทางนี้สงวนไว้
สำหรับ RPC ของระนาบควบคุมภายใน (เช่น การอัปเดตเซสชันของเอเจนต์ย่อย) และหลีกเลี่ยง
ไม่ให้ค่าพื้นฐานการจับคู่ CLI/อุปกรณ์ที่ล้าสมัยขัดขวางงานแบ็กเอนด์ภายในเครื่อง ไคลเอนต์จากระยะไกล
จากเบราว์เซอร์, Node และไคลเอนต์ที่ระบุโทเค็นอุปกรณ์/ข้อมูลประจำตัวอุปกรณ์อย่างชัดเจนยังคง
ผ่านการตรวจสอบการจับคู่และการยกระดับขอบเขตตามปกติ

### บทบาทเวิร์กเกอร์และโปรโตคอลปิด

เวิร์กเกอร์ระบบคลาวด์ใช้ทางเข้าลูปแบ็กเฉพาะผ่านอุโมงค์ SSH ที่ Gateway เป็นเจ้าของ
และตรึงคีย์โฮสต์ไว้ โดยยอมรับเฉพาะข้อมูลประจำตัวของเวิร์กเกอร์ และจะไม่ส่งต่อ
การยืนยันตัวตนทั่วไป เหตุการณ์ Node, RPC ของผู้ควบคุม หรือเมธอด Plugin โดยเด็ดขาด `connect`
แบบเข้มงวดจะตรวจสอบข้อมูลรับรองอายุสั้นที่เก็บเป็นแฮช ซึ่งผูกกับสภาพแวดล้อม แฮชบันเดิล
ยุคของเจ้าของ เวอร์ชันชุด RPC เวลาหมดอายุ และเซสชันที่เป็นค่าว่างได้หนึ่งรายการ อีกทั้ง
ตรวจสอบเวอร์ชันและชุดฟีเจอร์ปัจจุบันแยกต่างหาก เมื่อสำเร็จจะส่งคืน
`worker-hello-ok` ขั้นต่ำ การเจรจาฟีเจอร์เป็นอิสระจากเวอร์ชันโปรโตคอลทั่วไป
เฟรมยังคงมีขนาดต่ำกว่า 64 KiB ยกเว้นเฟรม `worker.inference.start`
ที่ผ่านการเจรจาแล้วอาจมีขนาดสูงสุด 25 MiB รายการอนุญาตแบบปิดประกอบด้วย `worker.heartbeat`,
`worker.transcript.commit`, `worker.live-event`, `worker.inference.start` และ
`worker.inference.cancel`

การคอมมิตทรานสคริปต์ใช้การกั้นด้วยยุคของเจ้าของ การผูกเซสชันที่ Gateway เป็นเจ้าของ
การเปรียบเทียบและสลับลีฟฐาน และการเล่นซ้ำลำดับแบบคงทน โดย Gateway จะสร้าง
รายการทรานสคริปต์และ ID ของพาเรนต์ผ่านตัวเขียนเซสชันตามปกติ มีการตรวจสอบความเป็นเจ้าของและ
เวลาหมดอายุอีกครั้งในทุก RPC

### ความสามารถของไคลเอนต์

ไคลเอนต์ผู้ควบคุมอาจประกาศความสามารถเพิ่มเติมใน `connect.params.caps`:

- `tool-events`: รองรับเหตุการณ์วงจรชีวิตเครื่องมือแบบมีโครงสร้าง
- `inline-widgets`: สามารถเรนเดอร์ผลลัพธ์เครื่องมือวิดเจ็ตอินไลน์แบบโฮสต์

ความสามารถของไคลเอนต์อธิบายไคลเอนต์ที่เชื่อมต่ออยู่ ไม่ใช่การอนุญาต เครื่องมือของเอเจนต์อาจประกาศความสามารถที่จำเป็น โดย Gateway จะละเว้นเครื่องมือเหล่านั้น เว้นแต่ข้อกำหนดทั้งหมดจะปรากฏใน `caps` ของไคลเอนต์ต้นทาง การทำงานที่เริ่มจากช่องทางไม่มีความสามารถของไคลเอนต์ Gateway ดังนั้นเครื่องมือที่ถูกจำกัดด้วยความสามารถจะใช้งานไม่ได้ แม้นโยบายเครื่องมือจะอนุญาตอย่างชัดเจนก็ตาม

### ตัวอย่างการเชื่อมต่อ Node

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "ios-node",
      "version": "1.2.3",
      "platform": "ios",
      "mode": "node"
    },
    "role": "node",
    "scopes": [],
    "caps": ["camera", "canvas", "screen", "location", "voice"],
    "commands": ["camera.snap", "canvas.navigate", "screen.record", "location.get"],
    "permissions": { "camera.capture": true, "screen.record": false },
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-ios/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Node ประกาศการอ้างสิทธิ์ความสามารถในขณะเชื่อมต่อ:

- `caps`: หมวดหมู่ระดับสูง เช่น `camera`, `canvas`, `screen`,
  `location`, `voice`, `talk`
- `commands`: รายการอนุญาตคำสั่งสำหรับการเรียกใช้
- `permissions`: ตัวสลับแบบละเอียด (เช่น `screen.record`, `camera.capture`)

Gateway ถือว่าค่าเหล่านี้เป็นการอ้างสิทธิ์และบังคับใช้รายการอนุญาตฝั่งเซิร์ฟเวอร์

## บทบาทและขอบเขต

สำหรับโมเดลขอบเขตของผู้ควบคุมแบบเต็ม การตรวจสอบ ณ เวลาอนุมัติ และความหมายของข้อมูลลับที่ใช้ร่วมกัน
โปรดดู [ขอบเขตของผู้ควบคุม](/th/gateway/operator-scopes)

บทบาท:

- `operator`: ไคลเอนต์ระนาบควบคุม (CLI/UI/ระบบอัตโนมัติ)
- `node`: โฮสต์ความสามารถ (camera/screen/canvas/system.run)
- `worker`: โฮสต์การดำเนินการบนระบบคลาวด์ผ่านโปรโตคอลเวิร์กเกอร์เฉพาะแบบปิด

ขอบเขตของผู้ควบคุม (`src/gateway/operator-scopes.ts`) ซึ่งเป็นชุดปิดทั้งหมด:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`talk.config` ที่มี `includeSecrets: true` ต้องใช้ `operator.talk.secrets` (หรือ
`operator.admin`) เมื่อรวมข้อมูลลับ ให้อ่านข้อมูลรับรองของผู้ให้บริการ Talk ที่ใช้งานอยู่
จาก `talk.resolved.config.apiKey`; `talk.providers.<id>.apiKey`
ยังคงมีรูปแบบตามต้นทาง และอาจเป็นอ็อบเจ็กต์ SecretRef หรือสตริงที่ถูกปกปิด

เมธอด RPC ของ Gateway ที่ลงทะเบียนโดย Plugin อาจร้องขอขอบเขตของผู้ควบคุมของตนเอง
แต่คำนำหน้าหลักที่สงวนไว้เหล่านี้จะถูกแก้ไขเป็น `operator.admin`
(`src/shared/gateway-method-policy.ts`) เสมอ: `config.*`, `exec.approvals.*`,
`wizard.*`, `update.*`

ขอบเขตของเมธอดเป็นเพียงด่านแรก คำสั่งแบบสแลชบางรายการที่เข้าถึงผ่าน
`chat.send` จะใช้การตรวจสอบระดับคำสั่งที่เข้มงวดกว่า: การเขียน `/config set` และ
`/config unset` แบบถาวรต้องใช้ `operator.admin` แม้สำหรับไคลเอนต์ Gateway ที่
มีขอบเขตของผู้ควบคุมระดับต่ำกว่าอยู่แล้ว

`node.pair.approve` มีการตรวจสอบขอบเขตเพิ่มเติม ณ เวลาอนุมัติ นอกเหนือจากขอบเขต
เมธอดพื้นฐาน (`operator.pairing`) โดยอิงตาม `commands`
(`src/infra/node-pairing-authz.ts`) ที่ประกาศไว้ในคำขอที่รอดำเนินการ:

| คำสั่งที่ประกาศ                                                                                                             | ขอบเขตที่จำเป็น                       |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| ไม่มี                                                                                                                          | `operator.pairing`                    |
| คำสั่งทั่วไป                                                                                                             | `operator.pairing` + `operator.write` |
| รวม `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir` หรือ `system.execApprovals.get/set` | `operator.pairing` + `operator.admin` |

### Caps/คำสั่ง/สิทธิ์ (Node)

Node ประกาศการอ้างสิทธิ์ความสามารถในขณะเชื่อมต่อ:

- `caps`: หมวดหมู่ความสามารถระดับสูง เช่น `camera`, `canvas`, `screen`,
  `location`, `voice` และ `talk`
- `commands`: รายการอนุญาตคำสั่งสำหรับการเรียกใช้
- `permissions`: ตัวสลับแบบละเอียด (เช่น `screen.record`, `camera.capture`)

Gateway จะถือว่าสิ่งเหล่านี้เป็น **คำกล่าวอ้าง** และบังคับใช้รายการอนุญาตที่ฝั่งเซิร์ฟเวอร์
Node ที่เชื่อมต่อสามารถเผยแพร่ตัวอธิบาย Plugin หรือเครื่องมือ MCP ที่ Agent มองเห็นได้ซึ่งเป็นทางเลือก
ด้วย `node.pluginTools.update` หลังจากเชื่อมต่อหรือ
เชื่อมต่อใหม่สำเร็จ โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้จะเริ่มทำงานใหม่เพื่อใช้การเปลี่ยนแปลงคลัง MCP
แบบประกาศ วิธีอัปเดตนี้เป็นเส้นทางเดียวสำหรับการเผยแพร่ และระบบไม่ยอมรับตัวอธิบายเครื่องมือ Plugin ใน
พารามิเตอร์ `connect` ตัวอธิบายแต่ละรายการต้องใช้ `name` ของเครื่องมือที่ปลอดภัยสำหรับผู้ให้บริการ และระบุ
`command` ที่อยู่ในรายการอนุญาตคำสั่งปัจจุบันของ Node Gateway เชื่อถือข้อมูลเมตาของตัวอธิบาย
จาก Node ที่จับคู่ไว้ กรองตัวอธิบายที่อยู่นอกขอบเขตคำสั่งที่ได้รับอนุมัติ
นำออกเมื่อ Node ตัดการเชื่อมต่อ และปฏิเสธความพยายามของผู้ดำเนินการ
ในการแก้ไขแค็ตตาล็อกของ Node อื่น ตั้งค่า `gateway.nodes.pluginTools.enabled: false`
เพื่อละเว้นตัวอธิบายที่ Node เผยแพร่

โฮสต์ Node ที่เชื่อมต่อจะเผยแพร่แค็ตตาล็อกทดแทน Skills ฉบับสมบูรณ์ของตนด้วย
`node.skills.update` เมธอดสำหรับบทบาท Node นี้เป็นเส้นทางเดียวในการเผยแพร่ Skills ของ Node
และระบบไม่ยอมรับ Skills ในพารามิเตอร์ `connect` ตัวอธิบายแต่ละรายการประกอบด้วย
ชื่อที่ปลอดภัย คำอธิบาย และเนื้อหา `SKILL.md` ที่มีขอบเขตจำกัด Gateway จะแยกวิเคราะห์
เนื้อหานั้นด้วยตัวโหลด Skills ตามปกติ รวมไว้ในสแนปช็อต Skills ของ Agent
ขณะที่ Node เชื่อมต่ออยู่ และนำออกเมื่อตัดการเชื่อมต่อ ตั้งค่า
`gateway.nodes.skills.enabled: false` เพื่อละเว้น Skills ที่ Node เผยแพร่

## สถานะการปรากฏตัว

- `system-presence` ส่งคืนรายการที่ใช้ข้อมูลประจำตัวของอุปกรณ์เป็นคีย์ รวมถึง
  `deviceId`, `roles` และ `scopes` เพื่อให้ UI แสดงหนึ่งแถวต่ออุปกรณ์ได้ แม้
  อุปกรณ์นั้นจะเชื่อมต่อทั้งในฐานะผู้ดำเนินการและ Node
- `node.list` มี `lastSeenAtMs` และ `lastSeenReason` ที่เป็นทางเลือก Node ที่เชื่อมต่อ
  จะรายงานเวลาเชื่อมต่อปัจจุบันด้วยเหตุผล `connect` และ Node ที่จับคู่ไว้ยังสามารถ
  รายงานสถานะการทำงานเบื้องหลังแบบคงอยู่ผ่านเหตุการณ์ Node ที่เชื่อถือได้

Node แบบเนทีฟบน macOS ยังสามารถส่งเหตุการณ์ `node.presence.activity` ที่ผ่านการตรวจสอบสิทธิ์
พร้อมระยะเวลาที่ไม่มีการป้อนข้อมูลซึ่งมีขอบเขตจำกัด Gateway จะคำนวณการประทับเวลากิจกรรมด้วย
นาฬิกาของตนเอง เปิดเผย Mac ที่เชื่อมต่อและมีข้อมูลล่าสุดผ่าน `node.list` และ
`node.describe` และกระจายการอัปเดต `node.presence` ไปยังไคลเอ็นต์ที่มีขอบเขตสิทธิ์อ่าน
ดูพฤติกรรมการเลือก ความเป็นส่วนตัว บริบทของโมเดล
และการกำหนดเส้นทางการแจ้งเตือนได้ที่ [สถานะการปรากฏตัวของคอมพิวเตอร์ที่ใช้งานอยู่](/nodes/presence)

### เหตุการณ์ที่ Node ยังทำงานอยู่เบื้องหลัง

Node เรียก `node.event` พร้อม `event: "node.presence.alive"` เพื่อบันทึกว่า
Node ที่จับคู่ไว้ยังทำงานอยู่ระหว่างการปลุกเบื้องหลัง โดยไม่ทำเครื่องหมายว่าเชื่อมต่ออยู่:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` เป็น enum แบบปิด: `background`, `silent_push`, `bg_app_refresh`,
`significant_location`, `manual`, `connect` ค่าที่ไม่รู้จักจะถูกปรับให้อยู่ในรูปมาตรฐานเป็น
`background` (`src/shared/node-presence.ts`) เหตุการณ์จะถูกเก็บถาวรเฉพาะสำหรับ
เซสชันอุปกรณ์ Node ที่ผ่านการตรวจสอบสิทธิ์เท่านั้น เซสชันที่ไม่มีอุปกรณ์หรือไม่ได้จับคู่จะส่งคืน
`handled: false`

Gateway ที่ดำเนินการสำเร็จจะส่งคืนผลลัพธ์แบบมีโครงสร้าง:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

Gateway รุ่นเก่าอาจส่งคืนเพียง `{ "ok": true }` สำหรับ `node.event` ให้ถือว่านั่น
เป็น RPC ที่ได้รับการตอบรับแล้ว ไม่ใช่การเก็บสถานะการปรากฏตัวแบบคงอยู่

## การกำหนดขอบเขตเหตุการณ์แบบกระจาย

เหตุการณ์แบบกระจายที่เซิร์ฟเวอร์พุชจะถูกควบคุมตามขอบเขต เพื่อให้เซสชันที่มีขอบเขตเฉพาะการจับคู่หรือเฉพาะ Node
ไม่ได้รับเนื้อหาเซสชันโดยอัตโนมัติ
(`src/gateway/server-broadcast.ts`):

- เฟรมแชต Agent และผลลัพธ์เครื่องมือ (เหตุการณ์ `agent` แบบสตรีม เหตุการณ์ผลลัพธ์เครื่องมือ)
  ต้องมีอย่างน้อย `operator.read` เซสชันที่ไม่มีสิทธิ์นี้จะข้ามเฟรมเหล่านี้
  ทั้งหมด
- การกระจาย `plugin.*` ที่ Plugin กำหนดจะถูกควบคุมเป็น `operator.write` หรือ
  `operator.admin` โดยค่าเริ่มต้น ส่วนรายการที่ระบุชัดเจน เช่น
  `plugin.approval.requested` / `plugin.approval.resolved` จะใช้
  `operator.approvals` แทน
- เหตุการณ์สถานะ/การขนส่ง (`heartbeat`, `presence`, `tick`, วงจรชีวิตการเชื่อมต่อ/ตัดการเชื่อมต่อ)
  ยังคงไม่ถูกจำกัด เพื่อให้ทุกเซสชันที่ผ่านการตรวจสอบสิทธิ์
  สังเกตสถานะการทำงานของการขนส่งได้
- กลุ่มเหตุการณ์แบบกระจายที่ไม่รู้จักจะถูกควบคุมตามขอบเขตโดยค่าเริ่มต้น (ปิดเมื่อเกิดข้อผิดพลาด)
  เว้นแต่ตัวจัดการที่ลงทะเบียนไว้จะผ่อนคลายข้อจำกัดอย่างชัดเจน

การเชื่อมต่อของไคลเอ็นต์แต่ละรายการจะเก็บหมายเลขลำดับเฉพาะของไคลเอ็นต์นั้นเอง ทำให้เหตุการณ์แบบกระจาย
ยังคงเรียงลำดับเพิ่มขึ้นอย่างต่อเนื่องบนซ็อกเก็ตนั้น แม้ไคลเอ็นต์แต่ละรายจะเห็น
ชุดย่อยของสตรีมเหตุการณ์ที่ผ่านการกรองตามขอบเขตต่างกัน

    - `terminal.list` ส่งคืนเซสชันที่เชื่อมต่อได้; `terminal.attach` ผูกเซสชันที่ยังทำงานอยู่หรือถูกถอดการเชื่อมโยงเข้ากับการเชื่อมต่อที่เรียกใช้ใหม่ และส่งคืนบัฟเฟอร์สำหรับเล่นซ้ำ (การเข้าควบคุมแบบ tmux — เจ้าของเดิมที่ยังเชื่อมต่ออยู่จะได้รับ `terminal.exit` พร้อมเหตุผล `detached`); `terminal.text` อ่านบัฟเฟอร์เป็นข้อความล้วนโดยไม่เชื่อมต่อ
    - เมธอดเทอร์มินัลทุกเมธอดต้องมี `operator.admin`; `gateway.terminal.enabled` ต้องเป็น true อย่างชัดเจน ระบบจะปฏิเสธเอเจนต์ที่อยู่ในแซนด์บ็อกซ์อย่างสมบูรณ์ และการเปลี่ยนนโยบายเอเจนต์จะปิด PTY ที่มีอยู่และที่กำลังดำเนินการทั้งหมด รวมถึง PTY ที่ถูกถอดการเชื่อมโยง

  <Accordion title="การสนทนาและ TTS">
    - `talk.catalog` ส่งคืนแค็ตตาล็อกผู้ให้บริการ Talk แบบอ่านอย่างเดียวสำหรับเสียงพูด การถอดเสียงแบบสตรีม และเสียงแบบเรียลไทม์ ได้แก่ ID ผู้ให้บริการมาตรฐาน นามแฝงในรีจิสทรี ป้ายกำกับ สถานะที่กำหนดค่า ผลลัพธ์ `ready` ระดับกลุ่มซึ่งมีหรือไม่มีก็ได้ ID โมเดล/เสียงที่เปิดเผย โหมดมาตรฐาน การส่งผ่าน กลยุทธ์สมอง และแฟล็กเสียง/ความสามารถแบบเรียลไทม์ โดยไม่ส่งคืนข้อมูลลับของผู้ให้บริการหรือแก้ไขการกำหนดค่าส่วนกลาง Gateway ปัจจุบันจะตั้งค่า `ready` หลังใช้การเลือกผู้ให้บริการขณะรันไทม์; หากไม่มีค่านี้ใน Gateway รุ่นเก่า ให้ถือว่ายังไม่ได้รับการยืนยัน
    - `talk.config` ส่งคืนเพย์โหลดการกำหนดค่า Talk ที่มีผล; `includeSecrets` ต้องมี `operator.talk.secrets` (หรือ `operator.admin`)

    - `talk.session.appendAudio` เพิ่มเสียงอินพุต PCM แบบ base64 ต่อท้ายเซสชันรีเลย์แบบเรียลไทม์และการถอดเสียงที่ Gateway เป็นเจ้าของ

    - `talk.session.submitToolResult` ทำให้การเรียกใช้เครื่องมือของผู้ให้บริการที่ส่งจากเซสชันรีเลย์แบบเรียลไทม์ซึ่ง Gateway เป็นเจ้าของเสร็จสมบูรณ์ คำขอจะรอสัญญาณการเสร็จสิ้นแบบอะซิงโครนัสใด ๆ ที่บริดจ์ผู้ให้บริการเปิดเผย; การส่งที่ล้มเหลวจะคงการรันที่เชื่อมโยงไว้และไม่ส่งเหตุการณ์ผลลัพธ์เครื่องมือที่สำเร็จ ส่ง `options: { willContinue: true }` สำหรับเอาต์พุตเครื่องมือระหว่างดำเนินการ หรือ `options: { suppressResponse: true }` เมื่อบริดจ์ผู้ให้บริการประกาศว่ารองรับการระงับและผลลัพธ์ไม่ควรเริ่มการตอบกลับอีกครั้ง

    - `talk.mode` ตั้งค่า/กระจายสถานะโหมด Talk ปัจจุบันให้ไคลเอนต์ WebChat/Control UI
    - `talk.client.create` สร้างเซสชันผู้ให้บริการแบบเรียลไทม์ที่ไคลเอนต์เป็นเจ้าของโดยใช้ `webrtc` หรือ `provider-websocket` ขณะที่ Gateway เป็นเจ้าของการกำหนดค่า ข้อมูลประจำตัว คำสั่ง และนโยบายเครื่องมือ

    - `talk.client.steer` ส่งการควบคุมด้วยเสียงของการรันที่ใช้งานอยู่สำหรับการส่งผ่านแบบเรียลไทม์ที่ไคลเอนต์เป็นเจ้าของ Gateway จะแก้ไขการรันแบบฝังที่ใช้งานอยู่จาก `sessionKey` และส่งคืนผลลัพธ์การยอมรับ/ปฏิเสธแบบมีโครงสร้าง แทนการละทิ้งการกำกับโดยไม่แจ้ง

    - `talk.speak` สังเคราะห์เสียงพูดผ่านผู้ให้บริการเสียงพูดของ Talk ที่ใช้งานอยู่
    - `tts.status` ส่งคืนสถานะการเปิดใช้ TTS ผู้ให้บริการที่ใช้งานอยู่ ผู้ให้บริการสำรอง และสถานะการกำหนดค่าผู้ให้บริการ

    - `tts.setProvider` อัปเดตผู้ให้บริการ TTS ที่ต้องการ
    - `tts.convert` เรียกใช้การแปลงข้อความเป็นเสียงพูดแบบครั้งเดียว

    - `config.get` ส่งคืนสแนปช็อตการกำหนดค่าปัจจุบันบนดิสก์ `hash` ของไฟล์รากแบบดิบ `configRevisionHash` ที่แก้ไขแล้ว และ `appliedConfigHash` ซึ่งมีหรือไม่มีก็ได้สำหรับรีวิชันที่แก้ไขแล้วซึ่งรันไทม์ Gateway ที่ใช้งานอยู่ยอมรับ
    - `config.set` เขียนเพย์โหลดการกำหนดค่าที่ผ่านการตรวจสอบ
    - `config.patch` ผสานการอัปเดตการกำหนดค่าบางส่วน การแทนที่อาร์เรย์แบบทำลายข้อมูลต้องระบุพาธที่ได้รับผลกระทบใน `replacePaths`; อาร์เรย์ซ้อนภายใต้รายการอาร์เรย์ใช้พาธ `[]` เช่น `agents.list[].skills`
    - `config.apply` ตรวจสอบและแทนที่เพย์โหลดการกำหนดค่าทั้งหมด
    - `config.schema` ส่งคืนเพย์โหลดสคีมาการกำหนดค่าแบบสดที่ Control UI และเครื่องมือ CLI ใช้ ได้แก่ สคีมา `uiHints` เวอร์ชัน ข้อมูลเมตาการสร้าง และข้อมูลเมตาสคีมาของ Plugin กับช่องทางเมื่อโหลดได้ โดยมีข้อมูลเมตา `title` / `description` จากป้ายกำกับ/ข้อความช่วยเหลือชุดเดียวกับ UI รวมถึงแขนงการประกอบของออบเจ็กต์ซ้อน ไวลด์การ์ด รายการอาร์เรย์ และ `anyOf` / `oneOf` / `allOf` เมื่อมีเอกสารฟิลด์ที่ตรงกัน
    - `config.schema.lookup` ส่งคืนเพย์โหลดการค้นหาที่จำกัดขอบเขตตามพาธสำหรับพาธการกำหนดค่าหนึ่งรายการ ได้แก่ พาธที่ปรับเป็นมาตรฐาน โหนดสคีมาแบบตื้น คำแนะนำที่ตรงกันพร้อม `hintPath`, `reloadKind` ซึ่งมีหรือไม่มีก็ได้ และสรุปองค์ประกอบลูกระดับถัดไปสำหรับการเจาะลึกใน UI/CLI โดย `reloadKind` เป็นหนึ่งใน `restart`, `hot` หรือ `none` (`src/config/schema.ts`) และสะท้อนตัววางแผนโหลดการกำหนดค่า Gateway ใหม่สำหรับพาธที่ขอ โหนดสคีมาการค้นหาจะเก็บเอกสารที่แสดงต่อผู้ใช้และฟิลด์การตรวจสอบทั่วไป (`title`, `description`, `type`, `enum`, `const`, `format`, `pattern`, ขอบเขตตัวเลข/สตริง/อาร์เรย์/ออบเจ็กต์, `additionalProperties`, `deprecated`, `readOnly`, `writeOnly`) สรุปองค์ประกอบลูกจะแสดง `key`, `path` ที่ปรับเป็นมาตรฐาน, `type`, `required`, `hasChildren`, `reloadKind` ซึ่งมีหรือไม่มีก็ได้ พร้อม `hint` / `hintPath` ที่ตรงกัน
    - `update.run` เรียกใช้ขั้นตอนการอัปเดต Gateway และกำหนดเวลาการเริ่มใหม่เฉพาะเมื่อการอัปเดตสำเร็จ; ผู้เรียกที่มีเซสชันสามารถระบุ `continuationMessage` เพื่อให้การเริ่มต้นดำเนินเทิร์นติดตามผลของเอเจนต์หนึ่งเทิร์นต่อผ่านคิวดำเนินการต่อหลังการเริ่มใหม่ การอัปเดตด้วยตัวจัดการแพ็กเกจและการอัปเดตเช็กเอาต์ git ที่มีการควบคุมดูแลจากระนาบควบคุมจะใช้การส่งมอบให้บริการที่มีการจัดการซึ่งแยกตัวออกมา แทนการแทนที่โครงสร้างแพ็กเกจหรือแก้ไขเอาต์พุตของเช็กเอาต์/บิลด์ภายใน Gateway ที่กำลังทำงาน การส่งมอบที่เริ่มแล้วจะส่งคืน `ok: true` พร้อม `result.reason: "managed-service-handoff-started"` และ `handoff.status: "started"`; การส่งมอบที่ใช้ไม่ได้หรือล้มเหลวจะส่งคืน `ok: false` พร้อม `managed-service-handoff-unavailable` หรือ `managed-service-handoff-failed` และ `handoff.command` เมื่อจำเป็นต้องอัปเดตผ่านเชลล์ด้วยตนเอง ใช้ไม่ได้หมายถึง OpenClaw ไม่มีขอบเขตซูเปอร์ไวเซอร์ที่ปลอดภัยหรือข้อมูลระบุตัวตนของบริการที่คงทน เช่น `OPENCLAW_SYSTEMD_UNIT` สำหรับ systemd ระหว่างการส่งมอบที่เริ่มแล้ว ตัวบ่งชี้การเริ่มใหม่อาจรายงาน `stats.reason: "restart-health-pending"` ชั่วครู่; การดำเนินการต่อจะถูกหน่วงจนกว่า CLI จะตรวจสอบ Gateway ที่เริ่มใหม่และเขียนตัวบ่งชี้ `ok` ขั้นสุดท้าย
    - `update.status` รีเฟรชและส่งคืนตัวบ่งชี้การเริ่มใหม่จากการอัปเดตล่าสุด รวมถึงเวอร์ชันที่กำลังทำงานหลังการเริ่มใหม่เมื่อมี
    - `wizard.start`, `wizard.next`, `wizard.status` และ `wizard.cancel` เปิดให้ใช้งานวิซาร์ดเริ่มต้นใช้งานผ่าน WS RPC

  </Accordion>

  <Accordion title="ตัวช่วยสำหรับเอเจนต์และเวิร์กสเปซ">
    - `agents.list` ส่งคืนรายการเอเจนต์ที่กำหนดค่าไว้ รวมถึงข้อมูลเมตาของโมเดลและรันไทม์ที่มีผลใช้งาน
    - `agents.create`, `agents.update` และ `agents.delete` จัดการระเบียนเอเจนต์และการเชื่อมโยงเวิร์กสเปซ
    - `agents.files.list`, `agents.files.get` และ `agents.files.set` จัดการไฟล์เวิร์กสเปซเริ่มต้นที่เปิดให้เอเจนต์เข้าถึง
    - `audit.activity.list` ส่งคืนบัญชีกิจกรรมที่มีการกำหนดเวอร์ชันและมีเฉพาะข้อมูลเมตา ส่วน `audit.list` ยังคงเป็น RPC สำหรับการรัน/เครื่องมือที่รักษาความเข้ากันได้อย่างปลอดภัย
    - `agents.workspace.list` และ `agents.workspace.get` (`operator.read`) เปิดให้ไคลเอนต์ในโดเมนผู้ดำเนินการที่เชื่อถือได้ตามที่อธิบายไว้ใน [ขอบเขตของผู้ดำเนินการ](/th/gateway/operator-scopes) เรียกดูไดเรกทอรีเวิร์กสเปซของเอเจนต์แบบอ่านอย่างเดียวและแบ่งหน้าได้ คำขอรับเฉพาะพาธที่สัมพันธ์กับเวิร์กสเปซ การอ่านถูกจำกัดให้อยู่ภายในรากเวิร์กสเปซที่ผ่านการแปลงเป็นพาธจริง (ปฏิเสธการหลบออกผ่าน symlink และ hardlink) มีการจำกัดขนาด และรองรับเฉพาะข้อความ UTF-8 รวมถึงชนิดรูปภาพทั่วไป (base64) การตอบกลับจะไม่เปิดเผยพาธเวิร์กสเปซบนโฮสต์ เนมสเปซนี้ไม่มีการดำเนินการเขียน
    - `tasks.list`, `tasks.get` และ `tasks.cancel` เปิดให้ SDK และไคลเอนต์ผู้ดำเนินการเข้าถึงบัญชีงานของ Gateway ดู [RPC ของบัญชีงาน](#task-ledger-rpcs) ด้านล่าง
    - `artifacts.list`, `artifacts.get` และ `artifacts.download` เปิดให้เข้าถึงสรุปและดาวน์โหลดอาร์ติแฟกต์ที่ได้จากทรานสคริปต์สำหรับขอบเขต `sessionKey`, `runId` หรือ `taskId` ที่ระบุอย่างชัดเจน การสอบถามการรันและงานจะระบุเซสชันเจ้าของที่ฝั่งเซิร์ฟเวอร์ และส่งคืนเฉพาะสื่อทรานสคริปต์ที่มีแหล่งที่มาตรงกัน แหล่ง URL ที่ไม่ปลอดภัยหรือเป็น URL ภายในเครื่องจะส่งคืนการดาวน์โหลดที่ไม่รองรับแทนการดึงข้อมูลที่ฝั่งเซิร์ฟเวอร์
    - `environments.list` และ `environments.status` คงการค้นหาสภาพแวดล้อมภายใน Gateway และ Node ไว้ เวิร์กเกอร์ระบบคลาวด์ที่กำหนดค่าไว้และระเบียนถาวรที่โปรไฟล์ก่อนหน้าทิ้งไว้จะเพิ่มข้อมูลเมตา `worker` ซึ่งมี `providerId`, `leaseId` ที่ไม่บังคับ, `state`, `ageMs`, `idleMs` ที่ไม่บังคับ และ `attachedSessionIds` สถานะวงจรชีวิตของเวิร์กเกอร์ ได้แก่ `requested`, `provisioning`, `bootstrapping`, `ready`, `attached`, `idle`, `draining`, `destroying`, `destroyed`, `failed` และ `orphaned`
    - `environments.create` (`{ profileId, idempotencyKey }`) จัดเตรียมเวิร์กเกอร์จากโปรไฟล์ผู้ให้บริการ Plugin ที่กำหนดค่าไว้ การลองใหม่ด้วยคีย์เดิมจะใช้การดำเนินการถาวรเดิมซ้ำ `environments.destroy` (`{ environmentId }`) ขอให้รื้อถอนสภาพแวดล้อมเวิร์กเกอร์ถาวรแบบทำซ้ำได้โดยให้ผลเหมือนเดิม ทั้งสองรายการต้องใช้ `operator.admin` เป็นการเขียนในระนาบควบคุม และส่งคืนสรุปสภาพแวดล้อมในรูปแบบเดียวกับที่ใช้ในการตอบกลับสถานะ
    - `agent.identity.get` ส่งคืนอัตลักษณ์ผู้ช่วยที่มีผลใช้งานสำหรับเอเจนต์หรือเซสชัน
    - `agent.wait` รอให้การรันเสร็จสิ้นและส่งคืนสแนปช็อตปลายทางเมื่อพร้อมใช้งาน

  </Accordion>

  <Accordion title="การควบคุมเซสชัน">
    - `sessions.list` ส่งคืนดัชนีเซสชันปัจจุบัน รวมถึงข้อมูลเมตา `agentRuntime` ต่อแถวเมื่อมีการกำหนดค่าแบ็กเอนด์รันไทม์ของเอเจนต์ เมื่อเปิดใช้การจัดวางเวิร์กเกอร์ระบบคลาวด์หรือมีสถานะการกู้คืนถาวร แถวเซสชันจะรวมสถานะ `placement` แบบปิด (`local`, `requested`, `provisioning`, `syncing`, `starting`, `active`, `draining`, `reconciling`, `reclaimed` หรือ `failed`) พร้อมฟิลด์สภาพแวดล้อม ยุคเจ้าของ เวิร์กสเปซ บันเดิล เคอร์เซอร์ ACK หรือการกู้คืนที่เฉพาะเจาะจงกับสถานะ
    - `sessions.subscribe` และ `sessions.unsubscribe` เปิดหรือปิดการสมัครรับเหตุการณ์การเปลี่ยนแปลงเซสชันสำหรับไคลเอนต์ WS ปัจจุบัน
    - `sessions.messages.subscribe` และ `sessions.messages.unsubscribe` เปิดหรือปิดการสมัครรับเหตุการณ์ทรานสคริปต์/ข้อความสำหรับหนึ่งเซสชัน ส่ง `includeApprovals: true` เพื่อรับเหตุการณ์วงจรชีวิต `session.approval` ที่ผ่านการล้างข้อมูลแล้วเพิ่มเติม สำหรับการอนุมัติที่กลุ่มผู้รับซึ่งบันทึกไว้นั้นมีเซสชันดังกล่าวตรงกันทุกประการ และการผูกผู้ตรวจสอบอนุญาตไคลเอนต์ที่สมัครรับ การตอบกลับการสมัครรับจะรวม `approvalReplay` ที่รอดำเนินการและมีขอบเขตจำกัด โดยถือเป็นข้อมูลที่มีอำนาจตัดสินเมื่อ `truncated` เป็น false การเลือกรับนี้มีผลต่อการเรียกสมัครรับแต่ละครั้งและไม่คงอยู่ การสมัครรับเซสชันเดิมอีกครั้งโดยไม่มี `includeApprovals: true` จะนำการสมัครรับการอนุมัติที่มีอยู่ออก นอกจากสิทธิ์อ่านเซสชันตามปกติแล้ว การเลือกรับนี้ต้องใช้ `operator.admin` หรือ `operator.approvals` บนอุปกรณ์ที่จับคู่แล้ว
    - `sessions.preview` ส่งคืนตัวอย่างทรานสคริปต์ที่มีขอบเขตจำกัดสำหรับคีย์เซสชันที่ระบุ
    - `sessions.describe` ส่งคืนแถวเซสชัน Gateway หนึ่งแถวสำหรับคีย์เซสชันที่ตรงกันทุกประการ
    - `sessions.resolve` ระบุหรือปรับเป้าหมายเซสชันให้อยู่ในรูปแบบมาตรฐาน
    - `sessions.create` สร้างรายการเซสชันใหม่ ค่า `model` และ `thinkingLevel` ที่ไม่บังคับจะบันทึกการแทนที่โมเดลและการให้เหตุผลเริ่มต้นแบบอะตอมมิก `worktree: true` จัดเตรียม worktree ที่มีการจัดการ โดย `worktreeBaseRef`/`worktreeName` ที่ไม่บังคับใช้เลือก ref ฐานและชื่อสาขา และ `execNode` (`operator.admin`) ผูกการดำเนินการของเซสชันเข้ากับโฮสต์ Node worktree ที่สร้างขึ้นจะถูกส่งกลับในผลลัพธ์และบันทึกไว้ในแถวเซสชัน (`worktree: { id, branch, repoRoot }`) เมื่อสร้างรายการสำเร็จแต่ `chat.send` เริ่มต้นที่ซ้อนอยู่ถูกปฏิเสธ ผลลัพธ์ที่สำเร็จจะรวม `runStarted: false` และ `runError` ไคลเอนต์สามารถเก็บพรอมต์ไว้และลองอีกครั้งโดยใช้คีย์เซสชันที่ส่งคืน
    - `sessions.dispatch` (`operator.admin`) ย้ายเซสชัน OpenClaw ภายในเครื่องที่มี worktree ซึ่งเซสชันเป็นเจ้าของและมีการจัดการ ไปยังโปรไฟล์เวิร์กเกอร์ระบบคลาวด์ที่กำหนดค่าไว้ ส่ง `{ key, profileId, agentId? }` เมธอดนี้จะไม่มีอยู่เมื่อไม่ได้กำหนดค่าโปรไฟล์เวิร์กเกอร์ โดยจะปิดการรับเทิร์นภายในเครื่องก่อนรอให้งานที่ทำอยู่เสร็จ และส่งคืนเมื่อการจัดวางเข้าสู่การเป็นเจ้าของเวิร์กเกอร์ `active` แล้วเท่านั้น การส่งต่อเป็นแบบทางเดียว RPC นี้ไม่รองรับการดึงกลับจากเวิร์กเกอร์มายังเครื่องภายใน
    - `sessions.groups.list`, `sessions.groups.put`, `sessions.groups.rename` และ `sessions.groups.delete` จัดการแค็ตตาล็อกกลุ่มเซสชันแบบกำหนดเองที่ Gateway เป็นเจ้าของ (ชื่อ + ลำดับการแสดงผล) การเป็นสมาชิกยังคงเก็บไว้ในฟิลด์ `category` ของแต่ละเซสชัน การเปลี่ยนชื่อและลบจะอัปเดตเซสชันสมาชิกที่ฝั่งเซิร์ฟเวอร์
    - `sessions.send` ส่งข้อความเข้าสู่เซสชันที่มีอยู่
    - `sessions.steer` เป็นรูปแบบขัดจังหวะและชี้นำสำหรับเซสชันที่กำลังทำงาน
    - `sessions.abort` ยกเลิกงานที่กำลังทำอยู่สำหรับเซสชัน ส่ง `key` พร้อม `runId` ที่ไม่บังคับ หรือส่งเฉพาะ `runId` สำหรับการรันที่กำลังทำงานซึ่ง Gateway สามารถระบุไปยังเซสชันได้
    - `sessions.patch` อัปเดตข้อมูลเมตา/การแทนที่ของเซสชัน และรายงานโมเดลมาตรฐานที่ระบุได้พร้อม `agentRuntime` ที่มีผลใช้งาน
    - `sessions.reset`, `sessions.delete` และ `sessions.compact` ดำเนินการบำรุงรักษาเซสชัน
    - `sessions.get` ส่งคืนแถวเซสชันที่จัดเก็บไว้ทั้งหมด
    - การดำเนินการแชตยังคงใช้ `chat.history`, `chat.send`, `chat.abort` และ `chat.inject` ส่วน `chat.history` ถูกปรับรูปแบบการแสดงผลให้เป็นมาตรฐานสำหรับไคลเอนต์ UI โดยจะตัดแท็กคำสั่งแบบอินไลน์ออกจากข้อความที่มองเห็น ตัดเพย์โหลด XML ของการเรียกเครื่องมือในรูปข้อความธรรมดา (`<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อกการเรียกเครื่องมือที่ถูกตัดทอน) รวมถึงโทเค็นควบคุมโมเดลแบบ ASCII/ความกว้างเต็มที่รั่วไหลออกมา ละเว้นแถวผู้ช่วยที่มีเฉพาะโทเค็นเงียบ (ตรงกับ `NO_REPLY` / `no_reply` ทุกประการ) และอาจแทนที่แถวที่มีขนาดใหญ่เกินไปด้วยตัวยึดตำแหน่ง
    - `chat.message.get` เป็นตัวอ่านข้อความฉบับเต็มแบบเพิ่มเติมที่มีขอบเขตจำกัดสำหรับรายการทรานสคริปต์ที่มองเห็นได้หนึ่งรายการ ส่ง `sessionKey`, `agentId` ที่ไม่บังคับเมื่อการเลือกเซสชันมีขอบเขตตามเอเจนต์ และ `messageId` ของทรานสคริปต์ที่เคยแสดงผ่าน `chat.history` มาก่อน Gateway จะส่งคืนโปรเจกชันที่ปรับรูปแบบการแสดงผลให้เป็นมาตรฐานแบบเดียวกัน โดยไม่มีขีดจำกัดการตัดทอนของประวัติแบบน้ำหนักเบา หากรายการที่จัดเก็บยังคงพร้อมใช้งานและมีขนาดไม่ใหญ่เกินไป
    - `chat.toolTitles` ส่งคืนชื่อวัตถุประสงค์แบบสั้นสำหรับการเรียกเครื่องมือที่แสดงใน Control UI (ประมวลผลเป็นชุด สูงสุด 24 รายการ โดยมีอินพุตที่จำกัดขอบเขต) ฟีเจอร์นี้ต้องเลือกรับผ่าน `gateway.controlUi.toolTitles` (ค่าเริ่มต้นปิด) Gateway ที่ปิดใช้จะตอบ `{ titles: {}, disabled: true }` โดยไม่เรียกโมเดลเพื่อให้ไคลเอนต์หยุดส่งคำขอ เมื่อเปิดใช้ ชื่อจะใช้การกำหนดเส้นทางโมเดลยูทิลิตีมาตรฐาน ได้แก่ `utilityModel` ที่กำหนดค่าไว้อย่างชัดเจน (การตัดสินใจของผู้ดำเนินการซึ่งอาจส่งเนื้อหางานที่มีขอบเขตจำกัดไปยังผู้ให้บริการที่เลือก เช่นเดียวกับงานยูทิลิตีทั้งหมด) หรือค่าเริ่มต้นของโมเดลขนาดเล็กที่ผู้ให้บริการของเซสชันประกาศ เพื่อไม่ให้มีปลายทางส่งออกใหม่เกิดขึ้นโดยปริยาย ส่วน `utilityModel` ที่ว่างเปล่าจะปิดใช้ทั้งหมด ชื่อจะไม่ย้อนกลับไปใช้โมเดลหลัก ผลลัพธ์จะถูกแคชในฐานข้อมูลสถานะต่อเอเจนต์ โดยใช้ชื่อเครื่องมือ + อินพุตเป็นคีย์ ดังนั้นการดูซ้ำจะไม่เรียกเก็บเงินสำหรับการเรียกเดิมอีก
    - `chat.send` ยอมรับ `fastMode: "auto"` แบบหนึ่งเทิร์นเพื่อใช้โหมดเร็วสำหรับการเรียกโมเดลที่เริ่มก่อนจุดตัดอัตโนมัติ จากนั้นจึงเริ่มการลองใหม่ การใช้ตัวสำรอง การส่งผลลัพธ์เครื่องมือ หรือการเรียกต่อเนื่องในภายหลังโดยไม่ใช้โหมดเร็ว จุดตัดมีค่าเริ่มต้น 60 วินาที (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) และสามารถกำหนดค่าต่อโมเดลด้วย `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` ผู้เรียก `chat.send` สามารถส่ง `fastAutoOnSeconds` แบบหนึ่งเทิร์นเพื่อแทนที่จุดตัดสำหรับคำขอนั้น ส่ง `queueMode` (`steer`, `followup`, `collect` หรือ `interrupt`) เพื่อแทนที่โหมดคิวที่จัดเก็บไว้เฉพาะสำหรับคำขอนี้ การดำเนินการชี้นำอย่างชัดเจนใน Control UI ใช้ `queueMode: "steer"`

### กลุ่มเหตุการณ์ทั่วไป

- `chat`: การอัปเดตแชตใน UI เช่น `chat.inject` และเหตุการณ์แชตอื่น
  ที่มีเฉพาะในบันทึกบทสนทนา ในโปรโตคอล v4 เพย์โหลดส่วนต่างมี `deltaText`; ส่วน `message` ยังคงเป็น
  สแนปช็อตสะสมของผู้ช่วย การแทนที่ที่ไม่ใช่คำนำหน้าจะตั้งค่า
  `replace=true` และใช้ `deltaText` เป็นข้อความแทนที่
- `session.message`, `session.operation`, `session.tool`: การอัปเดตบันทึกบทสนทนา การดำเนินการของเซสชัน
  ที่กำลังดำเนินอยู่ และสตรีมเหตุการณ์สำหรับเซสชันที่สมัครรับข้อมูล
- `session.approval`: ข้อมูลจริงของการอนุมัติที่รอดำเนินการและสิ้นสุดซึ่งผ่านการกรองข้อมูลแล้ว สำหรับผู้สมัครรับข้อมูล
  ของเซสชันที่ตรงกันทุกประการซึ่งเลือกเข้าร่วมอย่างชัดเจน การอนุมัติของรายการลูกใช้กลุ่มผู้รับของรายการบรรพบุรุษ
  ที่คงอยู่ เหตุการณ์จะไม่แก้ไขบันทึกบทสนทนาหรือปลุกเอเจนต์
- `sessions.changed`: ดัชนีหรือข้อมูลเมตาของเซสชันเปลี่ยนแปลง
- `presence`: การอัปเดตสแนปช็อตสถานะการปรากฏตัวของระบบ
- `tick`: เหตุการณ์ตรวจสอบการเชื่อมต่อ/ความพร้อมใช้งานเป็นระยะ
- `health`: การอัปเดตสแนปช็อตสุขภาพของ Gateway
- `heartbeat`: การอัปเดตสตรีมเหตุการณ์ Heartbeat
- `cron`: เหตุการณ์การเปลี่ยนแปลงการเรียกใช้/งาน Cron
- `shutdown`: การแจ้งเตือนการปิดระบบ Gateway
- `node.pair.requested` / `node.pair.resolved`: วงจรชีวิตการจับคู่ Node
- `node.invoke.request`: การกระจายคำขอเรียกใช้ Node
- `device.pair.requested` / `device.pair.resolved`: วงจรชีวิตอุปกรณ์ที่จับคู่แล้ว
- `voicewake.changed`: การกำหนดค่าทริกเกอร์คำปลุกเปลี่ยนแปลง
- `exec.approval.requested` / `exec.approval.resolved`: วงจรชีวิต
  การอนุมัติการดำเนินการ
- `plugin.approval.requested` / `plugin.approval.resolved`: วงจรชีวิต
  การอนุมัติ Plugin

### เมธอดตัวช่วยของ Node

Node สามารถเรียก `skills.bins` เพื่อดึงรายการไฟล์ปฏิบัติการของ Skills ปัจจุบัน
สำหรับการตรวจสอบการอนุญาตอัตโนมัติ

## RPC บัญชีแยกประเภทการตรวจสอบ

`audit.activity.list` ช่วยให้ไคลเอนต์ของผู้ปฏิบัติงานมีมุมมองที่เสถียรโดยเรียงจากใหม่ที่สุดของข้อมูลเมตา
วงจรชีวิตการเรียกใช้เอเจนต์ การดำเนินการของเครื่องมือ และข้อความที่เลือกเข้าร่วม โดยต้องมี
`operator.read` คิวรีจะไม่รวมระเบียนที่เก่ากว่า 30 วัน และบัญชีแยกประเภท
SQLite ที่ใช้ร่วมกันจำกัดไว้ที่ 100,000 ระเบียน แถวที่หมดอายุจะถูกลบระหว่าง
การเริ่มต้น Gateway การบำรุงรักษารายชั่วโมง และการเขียนในภายหลัง ดู
[ประวัติการตรวจสอบ](/gateway/audit) สำหรับโมเดลข้อมูลและความหมายด้านความเป็นส่วนตัว

- พารามิเตอร์: `agentId`, `sessionKey` หรือ `runId` ที่ตรงกันทุกประการซึ่งเป็นตัวเลือก; `kind` ที่เป็นตัวเลือก
  (`"agent_run"`, `"tool_action"` หรือ `"message"`); `status` ที่เป็นตัวเลือก
  (`"started"`, `"succeeded"`, `"failed"`, `"cancelled"`, `"timed_out"`,
  `"blocked"` หรือ `"unknown"`); ข้อความ `direction` ที่เป็นตัวเลือก (`"inbound"` หรือ
  `"outbound"`) และ `channel` ที่ตรงกันทุกประการ; ขอบเขต Unix มิลลิวินาทีแบบรวมค่า `after` / `before`
  ที่เป็นตัวเลือก; `limit` ที่เป็นตัวเลือกตั้งแต่ `1` ถึง `500`; และสตริง
  `cursor` ที่เป็นตัวเลือกจากหน้าก่อนหน้า
- ผลลัพธ์: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`

ยูเนียนผลลัพธ์ V1 แบบมีชื่อมีสคีมาแยกกันสำหรับการเรียกใช้เอเจนต์ การดำเนินการของเครื่องมือ ข้อความขาเข้า
และข้อความขาออก ตัวจำแนก `eventType` มีค่าเป็น
`agent_run`, `tool_action`, `inbound_message` หรือ `outbound_message` ตามลำดับ ส่วน `kind` และ
ข้อความ `direction` ยังคงใช้สำหรับการกรองและการแสดงผลได้ ทุกเหตุการณ์มี
`schemaVersion: 1` ซึ่งเป็นจำนวนเต็ม การอ้างอิงข้อมูลประจำตัวของข้อความใช้รูปแบบ
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` ที่ตรงกันทุกประการ โดย ID ของผู้กระทำที่เป็นผู้ส่งผ่านช่องทาง
ใช้รูปแบบเดียวกัน

ตัวแปรทั้งหมดต้องมี `eventType`, `schemaVersion`, `eventId`, `sequence`,
`sourceSequence`, `occurredAt`, `kind`, `action`, `status`, `actor` และ
`redaction` ฟิลด์ของตัวแปรมีดังนี้:

| `eventType`        | ฟิลด์ที่จำเป็น                                                   | ฟิลด์ที่เป็นตัวเลือก                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`, `runId`; `kind: "agent_run"`                           | `sessionKey`, `sessionId`, `errorCode`                                                                                          |
| `tool_action`      | `agentId`, `runId`; `kind: "tool_action"`                         | `sessionKey`, `sessionId`, `toolCallId`, `toolName`, `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`, `channel`, `conversationKind`, `outcome`  | `agentId`, `runId`, `durationMs`, `resultCount`, การอ้างอิงข้อมูลประจำตัว, `reasonCode`, `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`, `channel`, `conversationKind`, `outcome` | `agentId`, `runId`, `durationMs`, `resultCount`, การอ้างอิงข้อมูลประจำตัว, `reasonCode`, `deliveryKind`, `failureStage`, `errorCode` |

อีนัมข้อความแบบปิดมีดังนี้:

- `conversationKind`: `direct`, `group`, `channel` หรือ `unknown`
- `outcome` ขาเข้า: `completed`, `skipped` หรือ `failed`; `reasonCode`
  ที่เป็นตัวเลือก: `duplicate`, `reply_operation_active`,
  `reply_operation_aborted`, `fast_abort`, `plugin_bound_handled`,
  `plugin_bound_unavailable`, `plugin_bound_declined`, `plugin_bound_error`,
  `before_dispatch_handled`, `acp_dispatch_completed`, `acp_dispatch_failed`,
  `acp_dispatch_empty` หรือ `acp_dispatch_aborted`
- `outcome` ขาออก: `sent`, `suppressed`, `failed` หรือ `unknown`; `reasonCode`
  ที่เป็นตัวเลือก: `cancelled_by_message_sending_hook`,
  `cancelled_by_reply_payload_sending_hook`,
  `empty_after_message_sending_hook`, `empty_after_reply_payload_sending_hook`
  หรือ `no_visible_payload` อะแดปเตอร์ที่ไม่ส่งคืนข้อมูลประจำตัวของแพลตฟอร์มคือ
  `unknown` เนื่องจากไม่สามารถพิสูจน์ได้ว่าไม่มีผลข้างเคียงภายนอกเกิดขึ้น
- `deliveryKind`: `text`, `media` หรือ `other`; `failureStage`:
  `platform_send`, `queue` หรือ `unknown`

ฟิลด์สิ้นสุดมีความสัมพันธ์กัน ไม่ได้เป็นตัวเลือกโดยอิสระ:

| ตัวแปร          | การแมปสถานะสิ้นสุด                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| การเรียกใช้เอเจนต์        | `started` ไม่มี `errorCode`; สถานะเสร็จสิ้นที่ไม่สำเร็จแต่ละสถานะต้องมีรหัส `run_*` ที่ตรงกัน                                                                 |
| การดำเนินการของเครื่องมือ      | `started` และสถานะสำเร็จไม่มี `errorCode`; สถานะเสร็จสิ้นอื่นแต่ละสถานะต้องมีรหัส `tool_*` ที่ตรงกัน                                                       |
| ข้อความขาเข้า  | สำเร็จ = `completed`; ถูกบล็อก = `skipped`; ล้มเหลว = `failed` พร้อม `message_processing_failed` เมื่อมี `reasonCode` ค่านั้นต้องอยู่ในกลุ่มสถานะสิ้นสุดดังกล่าว |
| ข้อความขาออก | สำเร็จ = `sent`; ถูกบล็อก = `suppressed` พร้อม `reasonCode`; ล้มเหลว = `failed` พร้อม `errorCode` และ `failureStage`; ไม่ทราบ = `unknown` พร้อม `failureStage`      |

แต่ละเหตุการณ์กิจกรรมประกอบด้วยรหัสเหตุการณ์ที่คงที่ ลำดับบัญชีที่เพิ่มขึ้นทางเดียว
ลำดับเหตุการณ์ต้นทาง การประทับเวลา ผู้กระทำ การดำเนินการ สถานะ จำนวนเต็ม
`schemaVersion: 1` และ `redaction: "metadata_only"` ระเบียนการทำงานและเครื่องมือ
ต้องมีข้อมูลที่มาของเอเจนต์และการทำงาน และอาจมีข้อมูลที่มาของเซสชัน ระเบียนข้อความ
อาจมีรหัสเอเจนต์และรหัสการทำงาน แต่จงใจไม่รวม
`sessionKey` หรือ `sessionId` โดยเด็ดขาด ดังนั้นตัวกรองคำค้น `sessionKey` จึงใช้กับ
แถวการทำงานและเครื่องมือเท่านั้น เหตุการณ์เครื่องมืออาจมีรหัสการเรียกเครื่องมือและชื่อเครื่องมือ

ระเบียนข้อความใช้ `message.inbound.processed` หรือ
`message.outbound.finished` และเพิ่มทิศทาง ช่องทาง ประเภทการสนทนา
ผลลัพธ์ที่ปรับเป็นมาตรฐาน และข้อมูลเสริม ได้แก่ ประเภทการส่ง ขั้นตอนที่ล้มเหลว ระยะเวลา
จำนวนผลลัพธ์ รหัสเหตุผล และนามแฝงของบัญชี/การสนทนา/ข้อความ/เป้าหมาย
ที่ใช้คีย์เฉพาะของการติดตั้ง นามแฝงเหล่านี้ช่วย
เชื่อมโยงข้อมูล แต่ไม่ใช่การทำให้เป็นนิรนาม เนื่องจากฐานข้อมูลสถานะมีคีย์ของนามแฝง
ขณะที่การส่งออกผ่าน RPC และ CLI ไม่มี บัญชีแยกประเภทไม่จัดเก็บพรอมต์ เนื้อหา
ข้อความ อาร์กิวเมนต์เครื่องมือ ผลลัพธ์เครื่องมือ เอาต์พุตคำสั่ง หรือข้อความข้อผิดพลาดดิบ
ค่า `sessionKey` ของการทำงาน/เครื่องมือยังคงเป็นเมทาดาทาดิบสำหรับการเชื่อมโยง และอาจฝัง
รหัสบัญชีแพลตฟอร์มหรือรหัสคู่สื่อสาร ส่วนระเบียนข้อความจะไม่รวมคีย์เซสชัน

สำหรับแถวขาเข้า `durationMs` วัดการส่งต่อภายในแกนหลักจนถึงสถานะสิ้นสุด และ
`resultCount` นับเพย์โหลดเครื่องมือ บล็อก และการตอบกลับที่เข้าคิวและได้รับการสรุปผลแล้ว สำหรับ
แถวขาออก `durationMs` ครอบคลุมตั้งแต่การรับผิดชอบการส่งไปจนถึงการตอบรับ
จดหมายตาย หรือการกระทบยอด (รวมเวลารอในคิว) และ `resultCount`
นับการส่งจริงบนแพลตฟอร์มที่ระบุได้ เมื่อมี `deliveryKind`
ค่านี้จะอธิบายเพย์โหลดที่มีผลหลังผ่านฮุกและการเรนเดอร์ ส่วนแถวที่ถูกระงับหรือ
กำกวมเนื่องจากการขัดข้องจะไม่มีค่านี้

ความครอบคลุมของข้อความในปัจจุบันรวมข้อความขาเข้าที่ได้รับการยอมรับและไปถึง
การส่งต่อภายในแกนหลัก รวมถึงผลลัพธ์ซ้ำ/สิ้นสุดของแกนหลัก ความครอบคลุมขาออกจะเขียน
แถวสิ้นสุดหนึ่งแถวต่อเพย์โหลดการตอบกลับเชิงตรรกะดั้งเดิมแต่ละรายการที่ไปถึงการส่งแบบคงทนร่วมกัน
โดยรวมการแบ่งส่วนและการกระจายผ่านอะแดปเตอร์ไว้ใน `resultCount` การส่งที่เข้าคิว
ซึ่งลองใหม่ได้หรือมีความกำกวม จะถูกบันทึกหลังการตอบรับ จดหมายตาย
หรือการกระทบยอดเท่านั้น เส้นทางภายใน Plugin และเส้นทางการส่งโดยตรงที่ข้าม
ขอบเขตร่วมเหล่านั้นยังไม่ครอบคลุม คิวผู้ปฏิบัติงานแบบมีขอบเขตทำงานตามความสามารถ
และอาจทิ้งระเบียนเมื่อเกิดความล้มเหลวหรือคิวเต็ม ดังนั้นพื้นผิวนี้จึงไม่ใช่
คลังการปฏิบัติตามข้อกำหนดแบบไม่สูญเสียข้อมูล

การบันทึกเปิดใช้งานโดยค่าเริ่มต้นและควบคุมด้วย
[`audit.enabled`](/th/gateway/configuration-reference#audit) การบันทึกข้อความ
ควบคุมแยกต่างหากด้วย `audit.messages` และมีค่าเริ่มต้นเป็น `"off"` เมื่อ
ปิดการบันทึก `audit.activity.list` จะยังคงให้บริการระเบียนที่เขียนไว้
ก่อนหน้านี้จนกว่าจะหมดอายุ

สคีมาคำขอ ผลลัพธ์ และ `AuditEvent` ของ `audit.list` ที่จัดส่งแล้ว
ยังคงไม่เปลี่ยนแปลง และส่งคืนเฉพาะระเบียนการทำงานของเอเจนต์และการดำเนินการของเครื่องมือ ไคลเอนต์
ผู้ดำเนินการรุ่นใหม่ควรเรียก `audit.activity.list` เมื่อ Gateway ประกาศว่ารองรับ Gateway
รุ่นเก่าอาจรายงาน `unknown method: audit.activity.list` หรือ `missing scope:
operator.admin` ให้กับคำขอที่มีขอบเขตการอ่าน
เนื่องจากในเวอร์ชันที่จัดส่งแล้ว การตรวจสอบสิทธิ์เกิดก่อนการค้นหาเมธอด ให้ถือว่ากรณีหลังหมายถึง
ไม่มีเมธอดเฉพาะเมื่อไม่มีการประกาศเมธอดนั้น จากนั้นไคลเอนต์อาจลอง `audit.list` ใหม่
ได้เฉพาะเมื่อไม่ต้องใช้การรองรับตัวกรองประเภทข้อความ ทิศทาง หรือช่องทาง

ใช้ [`openclaw audit`](/th/cli/audit) สำหรับคำค้นแบบข้อความและการส่งออก JSON ที่มีขอบเขต

## RPC ของบัญชีแยกประเภทงาน

ไคลเอนต์ผู้ดำเนินการตรวจสอบและยกเลิกระเบียนงานเบื้องหลังของ Gateway ผ่าน
RPC ของบัญชีแยกประเภทงาน (`packages/gateway-protocol/src/schema/tasks.ts`) ซึ่งจะ
ส่งคืนสรุปงานที่ผ่านการทำให้ปลอดภัยแล้ว ไม่ใช่สถานะรันไทม์ดิบ

- `tasks.list` ต้องใช้ `operator.read`
  - พารามิเตอร์: `status` ที่ระบุหรือไม่ก็ได้ (`"queued"`, `"running"`, `"completed"`,
    `"failed"`, `"cancelled"` หรือ `"timed_out"`) หรืออาร์เรย์ของสถานะเหล่านั้น
    `agentId` ที่ระบุหรือไม่ก็ได้, `sessionKey` ที่ระบุหรือไม่ก็ได้, `limit` ที่ระบุหรือไม่ก็ได้ตั้งแต่ `1` ถึง
    `500` และสตริง `cursor` ที่ระบุหรือไม่ก็ได้
  - ผลลัพธ์: `{ "tasks": TaskSummary[], "nextCursor"?: string }`
- `tasks.get` ต้องใช้ `operator.read`
  - พารามิเตอร์: `{ "taskId": string }`
  - ผลลัพธ์: `{ "task": TaskSummary }`
  - รหัสงานที่ไม่มีอยู่จะส่งคืนรูปแบบข้อผิดพลาดไม่พบของ Gateway
- `tasks.cancel` ต้องใช้ `operator.write`
  - พารามิเตอร์: `{ "taskId": string, "reason"?: string }`
  - ผลลัพธ์: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`
  - `found` รายงานว่าบัญชีแยกประเภทมีงานที่ตรงกันหรือไม่ `cancelled`
    รายงานว่ารันไทม์ยอมรับหรือบันทึกการยกเลิกหรือไม่

`TaskSummary` ประกอบด้วย `id`, `status` และเมทาดาทาที่ระบุหรือไม่ก็ได้ ได้แก่ `kind`,
`runtime`, `title`, `agentId`, `sessionKey`, `childSessionKey`, `ownerKey`,
`runId`, `taskId`, `flowId`, `parentTaskId`, `sourceId`, การประทับเวลา ความคืบหน้า
สรุปสถานะสิ้นสุด และข้อความข้อผิดพลาดที่ผ่านการทำให้ปลอดภัยแล้ว `agentId` ระบุเอเจนต์
ที่ดำเนินงาน ส่วน `sessionKey` และ `ownerKey` เก็บรักษาบริบทของผู้ร้องขอและการควบคุม

## เมธอดตัวช่วยสำหรับผู้ดำเนินการ

- `commands.list` (`operator.read`) ดึงรายการคำสั่งรันไทม์สำหรับ
  เอเจนต์
  - `agentId` ระบุหรือไม่ก็ได้ หากละไว้จะอ่านพื้นที่ทำงานของเอเจนต์เริ่มต้น
  - `scope` ควบคุมว่าค่า `name` หลักจะอ้างถึงพื้นผิวใด: `text` ส่งคืน
    โทเค็นคำสั่งข้อความหลักโดยไม่มี `/` นำหน้า ส่วน `native` และ
    เส้นทางเริ่มต้น `both` จะส่งคืนชื่อเนทีฟที่รับรู้ผู้ให้บริการเมื่อมี
  - `textAliases` มีนามแฝงแบบเครื่องหมายทับที่ตรงตัว เช่น `/model` และ `/m`
  - `nativeName` มีชื่อคำสั่งเนทีฟที่รับรู้ผู้ให้บริการเมื่อมี
  - `provider` ระบุหรือไม่ก็ได้ และมีผลเฉพาะต่อการตั้งชื่อแบบเนทีฟและความพร้อมใช้งานของคำสั่ง
    Plugin แบบเนทีฟ
  - `includeArgs=false` จะไม่รวมเมทาดาทาอาร์กิวเมนต์แบบซีเรียลไลซ์ไว้ในผลตอบกลับ
- `tools.catalog` (`operator.read`) ดึงแค็ตตาล็อกเครื่องมือรันไทม์สำหรับ
  เอเจนต์ ผลตอบกลับประกอบด้วยเครื่องมือที่จัดกลุ่มและเมทาดาทาที่มา:
  - `source`: `core` หรือ `plugin`
  - `pluginId`: เจ้าของ Plugin เมื่อเป็น `source="plugin"`
  - `optional`: เครื่องมือ Plugin เป็นตัวเลือกหรือไม่
- `tools.effective` (`operator.read`) ดึงรายการเครื่องมือที่มีผลจริงในรันไทม์
  สำหรับเซสชัน
  - ต้องระบุ `sessionKey`
  - Gateway จะอนุมานบริบทรันไทม์ที่เชื่อถือได้จากเซสชันทางฝั่งเซิร์ฟเวอร์
    แทนการยอมรับบริบทการยืนยันตัวตนหรือการส่งที่ผู้เรียกเป็นผู้ระบุ
  - ผลตอบกลับเป็นภาพฉายของรายการที่ใช้งานอยู่ซึ่งอนุมานโดยเซิร์ฟเวอร์และมีขอบเขตเฉพาะเซสชัน
    โดยรวมเครื่องมือของแกนหลัก Plugin ช่องทาง และเซิร์ฟเวอร์ MCP
    ที่ค้นพบแล้ว
  - `tools.effective` เป็นแบบอ่านอย่างเดียวสำหรับ MCP โดยอาจฉายแค็ตตาล็อก MCP
    ของเซสชันที่อุ่นแล้วผ่านนโยบายเครื่องมือขั้นสุดท้าย แต่จะไม่สร้างรันไทม์ MCP
    เชื่อมต่อการรับส่ง หรือส่ง `tools/list` หากไม่มีแค็ตตาล็อกที่อุ่นแล้วและตรงกัน
    ผลตอบกลับอาจมีประกาศ เช่น `mcp-not-yet-connected`,
    `mcp-not-yet-listed` หรือ `mcp-stale-catalog`
  - รายการเครื่องมือที่มีผลใช้ `source="core"`, `source="plugin"`,
    `source="channel"` หรือ `source="mcp"`
- `tools.invoke` (`operator.write`) เรียกใช้เครื่องมือที่พร้อมใช้งานหนึ่งรายการผ่าน
  เส้นทางนโยบาย Gateway เดียวกับ `/tools/invoke`
  - ต้องระบุ `name` ส่วน `args`, `sessionKey`, `agentId`, `confirm` และ
    `idempotencyKey` ระบุหรือไม่ก็ได้
  - หากมีทั้ง `sessionKey` และ `agentId` เอเจนต์ของเซสชันที่แก้ไขได้
    ต้องตรงกับ `agentId`
  - ตัวหุ้มแกนหลักสำหรับเจ้าของเท่านั้น เช่น `cron`, `gateway` และ `nodes` ต้องใช้
    ตัวตนเจ้าของ/ผู้ดูแลระบบ (`operator.admin`) แม้ว่า `tools.invoke` เอง
    จะเป็น `operator.write`
  - ผลตอบกลับเป็นซองข้อมูลสำหรับ SDK ที่มี `ok`, `toolName`, `output`
    ที่ระบุหรือไม่ก็ได้ และฟิลด์ `error` แบบมีชนิด การปฏิเสธเนื่องจากการอนุมัติหรือนโยบายจะส่งคืน
    `ok:false` ในเพย์โหลด แทนการข้ามไปป์ไลน์นโยบายเครื่องมือของ Gateway
- `skills.status` (`operator.read`) ดึงรายการ Skills ที่เอเจนต์
  มองเห็นได้
  - `agentId` ระบุหรือไม่ก็ได้ หากละไว้จะอ่านพื้นที่ทำงานของเอเจนต์เริ่มต้น
  - ผลตอบกลับประกอบด้วยสถานะการมีสิทธิ์ ข้อกำหนดที่ขาดหาย การตรวจสอบการกำหนดค่า
    และตัวเลือกการติดตั้งที่ผ่านการทำให้ปลอดภัยแล้ว โดยไม่เปิดเผยค่าความลับดิบ
- `skills.search` และ `skills.detail` (`operator.read`) ส่งคืนเมทาดาทา
  การค้นพบของ ClawHub
- `skills.upload.begin`, `skills.upload.chunk` และ `skills.upload.commit`
  (`operator.admin`) จัดเตรียมไฟล์เก็บถาวรของ Skills แบบส่วนตัวก่อนติดตั้ง เส้นทางนี้
  เป็นเส้นทางอัปโหลดสำหรับผู้ดูแลระบบที่แยกต่างหากสำหรับไคลเอนต์ที่เชื่อถือได้ ไม่ใช่โฟลว์ติดตั้ง Skills
  ของ ClawHub ตามปกติ และปิดใช้งานโดยค่าเริ่มต้น เว้นแต่จะเปิดใช้
  `skills.install.allowUploadedArchives`
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })`
    สร้างการอัปโหลดที่ผูกกับ slug และค่าบังคับดังกล่าว
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })` ต่อท้ายไบต์ที่
    ออฟเซ็ตที่ถอดรหัสแล้วอย่างตรงตัว
  - `skills.upload.commit({ uploadId, sha256? })` ตรวจสอบขนาดสุดท้ายและ
    SHA-256 การคอมมิตเพียงทำให้การอัปโหลดเสร็จสมบูรณ์ แต่ไม่ได้ติดตั้ง Skills
  - ไฟล์เก็บถาวรของ Skills ที่อัปโหลดเป็นไฟล์ zip ที่มีรูท `SKILL.md` ชื่อ
    ไดเรกทอรีภายในไฟล์เก็บถาวรจะไม่ใช้เลือกเป้าหมายการติดตั้ง
- `skills.install` (`operator.admin`) มีสามโหมด:
  - โหมด ClawHub: `{ source: "clawhub", slug, version?, force? }` ติดตั้ง
    โฟลเดอร์ Skills ลงในไดเรกทอรี `skills/` ของพื้นที่ทำงานเอเจนต์เริ่มต้น
  - โหมดอัปโหลด: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }`
    ติดตั้งการอัปโหลดที่คอมมิตแล้วลงในไดเรกทอรี `skills/<slug>`
    ของพื้นที่ทำงานเอเจนต์เริ่มต้น slug และค่าบังคับต้องตรงกับ
    คำขอ `skills.upload.begin` เดิม ระบบจะปฏิเสธ เว้นแต่
    จะเปิดใช้ `skills.install.allowUploadedArchives` การตั้งค่านี้ไม่มีผลต่อ
    การติดตั้งผ่าน ClawHub
  - โหมดตัวติดตั้งของ Gateway: `{ name, installId, timeoutMs? }` เรียกใช้การดำเนินการ
    `metadata.openclaw.install` ที่ประกาศไว้บนโฮสต์ Gateway ไคลเอนต์รุ่นเก่าอาจ
    ยังคงส่ง `dangerouslyForceUnsafeInstall` ฟิลด์นี้เลิกใช้แล้ว
    ยอมรับเพื่อความเข้ากันได้ของโปรโตคอลเท่านั้น และจะถูกละเว้น ใช้
    `security.installPolicy` สำหรับการตัดสินใจติดตั้งที่ผู้ดำเนินการเป็นเจ้าของ
- `skills.update` (`operator.admin`) มีสองโหมด:
  - โหมด ClawHub อัปเดต slug ที่ติดตามหนึ่งรายการหรือการติดตั้ง ClawHub ที่ติดตามทั้งหมดใน
    พื้นที่ทำงานเอเจนต์เริ่มต้น
  - โหมดการกำหนดค่าจะแพตช์ค่า `skills.entries.<skillKey>` เช่น `enabled`,
    `apiKey` และ `env`

### มุมมอง `models.list`

`models.list` ยอมรับพารามิเตอร์ `view` ที่ระบุหรือไม่ก็ได้
(`src/agents/model-catalog-visibility.ts`):

- ละไว้หรือ `"default"`: หากกำหนดค่า `agents.defaults.models` ไว้ การตอบกลับจะเป็นแค็ตตาล็อกที่อนุญาต รวมถึงโมเดลที่ค้นพบแบบไดนามิกสำหรับรายการ `provider/*` มิฉะนั้น การตอบกลับจะเป็นแค็ตตาล็อก Gateway ทั้งหมด
- `"configured"`: ลักษณะการทำงานสำหรับตัวเลือก หากกำหนดค่า `agents.defaults.models` ไว้ ค่านี้ยังคงมีผลเหนือกว่า รวมถึงการค้นพบภายในขอบเขตผู้ให้บริการสำหรับรายการ `provider/*` หากไม่มีรายการอนุญาต การตอบกลับจะใช้รายการ `models.providers.<provider>.models` ที่ระบุไว้อย่างชัดเจน และจะใช้แค็ตตาล็อกทั้งหมดเป็นทางเลือกสำรองเฉพาะเมื่อไม่มีแถวโมเดลที่กำหนดค่าไว้
- `"provider-config"`: รายการคลัง `models.providers.*.models` ที่กำหนดโดยแหล่งที่มา โดยไม่ขึ้นกับรายการอนุญาตของตัวเลือก แถวต่างๆ มีความสามารถสาธารณะของโมเดลและสถานะความพร้อมใช้งานตามเส้นทาง แต่ไม่รวมปลายทางของผู้ให้บริการ ข้อมูลการยืนยันตัวตน และการกำหนดค่าคำขอรันไทม์
- `"all"`: แค็ตตาล็อก Gateway ทั้งหมดโดยข้าม `agents.defaults.models` ใช้สำหรับ UI การวินิจฉัย/การค้นหา ไม่ใช่ตัวเลือกโมเดลทั่วไป

## การอนุมัติการดำเนินการ

- เมื่อคำขอดำเนินการต้องได้รับการอนุมัติ Gateway จะกระจาย `exec.approval.requested`
- ไคลเอ็นต์ของผู้ดำเนินการแก้ไขคำขอโดยเรียก `exec.approval.resolve` (ต้องมี `operator.approvals`)
- สำหรับ `host=node` ค่า `exec.approval.request` ต้องมี `systemRunPlan` (ข้อมูลเมตา `argv`/`cwd`/`rawCommand`/เซสชันตามรูปแบบมาตรฐาน) คำขอที่ไม่มี `systemRunPlan` จะถูกปฏิเสธ
- หลังได้รับการอนุมัติ การเรียก `node.invoke system.run` ที่ส่งต่อจะใช้ `systemRunPlan` ตามรูปแบบมาตรฐานนั้นซ้ำเป็นบริบทคำสั่ง/cwd/เซสชันที่มีอำนาจตัดสิน
- หากผู้เรียกเปลี่ยนแปลง `command`, `rawCommand`, `cwd`, `agentId` หรือ `sessionKey` ระหว่างการเตรียมและการส่งต่อ `system.run` ที่ได้รับอนุมัติขั้นสุดท้าย Gateway จะปฏิเสธการรันแทนที่จะเชื่อถือเพย์โหลดที่ถูกเปลี่ยนแปลง

## ทางเลือกสำรองในการส่งมอบของเอเจนต์

- คำขอ `agent` สามารถมี `deliver=true` เพื่อร้องขอการส่งมอบขาออก
- `bestEffortDeliver=false` (ค่าเริ่มต้น) จะคงลักษณะการทำงานที่เข้มงวดไว้: เป้าหมายการส่งมอบที่ระบุไม่ได้หรือใช้ภายในเท่านั้นจะส่งคืน `INVALID_REQUEST`
- `bestEffortDeliver=true` อนุญาตให้ใช้การดำเนินการภายในเซสชันเท่านั้นเป็นทางเลือกสำรอง เมื่อไม่สามารถระบุเส้นทางที่ส่งมอบไปยังภายนอกได้ (เช่น เซสชันภายใน/เว็บแชต หรือการกำหนดค่าแบบหลายช่องทางที่กำกวม)
- ผลลัพธ์ `agent` ขั้นสุดท้ายอาจมี `result.deliveryStatus` เมื่อมีการร้องขอการส่งมอบ โดยใช้สถานะ `sent`, `suppressed`, `partial_failed` และ `failed` เดียวกับที่ระบุไว้สำหรับ [`openclaw agent --json --deliver`](/th/cli/agent#json-delivery-status)

## การกำหนดเวอร์ชัน

- `PROTOCOL_VERSION`, `MIN_CLIENT_PROTOCOL_VERSION`, `MIN_NODE_PROTOCOL_VERSION` และ `MIN_PROBE_PROTOCOL_VERSION` อยู่ใน `packages/gateway-protocol/src/version.ts`
- ไคลเอ็นต์ส่ง `minProtocol` + `maxProtocol` ไคลเอ็นต์ของผู้ดำเนินการและ UI ต้องรวมโปรโตคอลปัจจุบันไว้ในช่วงดังกล่าว ปัจจุบันไคลเอ็นต์และเซิร์ฟเวอร์ใช้โปรโตคอล v4
- ไคลเอ็นต์ที่ยืนยันตัวตนแล้วและมีทั้ง `role: "node"` และ `client.mode: "node"` สามารถใช้โปรโตคอล Node รุ่น N-1 (ปัจจุบันคือ v3) โพรบตรวจสอบการรีสตาร์ตแบบน้ำหนักเบาใช้ช่วง N-1 เดียวกัน การยืนยันตัวตนอุปกรณ์ การจับคู่ ขอบเขต นโยบายคำสั่ง และการอนุมัติการดำเนินการจะไม่เปลี่ยนแปลงจากช่วงความเข้ากันได้นี้ ความสามารถและคำสั่งของ Node ที่ Plugin เป็นเจ้าของจะถูกระงับไว้จนกว่า Node จะอัปเกรดเป็นโปรโตคอลปัจจุบัน เนื่องจากพื้นผิวที่โฮสต์โดยส่วนเหล่านั้นไม่ได้เป็นส่วนหนึ่งของสัญญา N-1
- สคีมาและโมเดลสร้างขึ้นจากคำจำกัดความ TypeBox:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

| ระยะหน่วงก่อนลองเชื่อมต่อใหม่ครั้งแรก                 | `1_000` ms                                            | `packages/gateway-client/src/client.ts` (`GATEWAY_RECONNECT_POLICY`)                                                      |
| ระยะหน่วงสูงสุดก่อนลองเชื่อมต่อใหม่                     | `30_000` ms                                           | `packages/gateway-client/src/client.ts` (`GATEWAY_RECONNECT_POLICY`)                                                      |
| ขีดจำกัดการลองใหม่อย่างรวดเร็วหลังปิดเนื่องจากโทเค็นอุปกรณ์ | `250` ms                                              | `packages/gateway-client/src/client.ts`                                                                                   |
| ช่วงผ่อนผันก่อนบังคับหยุด `terminate()`     | `250` ms                                              | `FORCE_STOP_TERMINATE_GRACE_MS`                                                                                           |
| ระยะหมดเวลาเริ่มต้นของ `stopAndWait()`           | `1_000` ms                                            | `STOP_AND_WAIT_TIMEOUT_MS`                                                                                                |
| ช่วงห่าง Tick เริ่มต้น (ก่อน `hello-ok`)    | `30_000` ms                                           | `packages/gateway-client/src/client.ts`                                                                                   |
| การปิดเมื่อ Tick หมดเวลา                        | รหัส `4000` เมื่อไม่มีการสื่อสารนานเกิน `tickIntervalMs * 2` | `packages/gateway-client/src/client.ts`                                                                                   |
| `MAX_PAYLOAD_BYTES`                       | `25 * 1024 * 1024` (25 MB)                            | `src/gateway/server-constants.ts`                                                                                         |

เซิร์ฟเวอร์ประกาศค่า `policy.tickIntervalMs`, `policy.maxPayload` และ `policy.maxBufferedBytes` ที่มีผลจริงใน `hello-ok` ไคลเอ็นต์ควรยึดค่าดังกล่าวแทนค่าเริ่มต้นก่อนการจับมือ

ไคลเอ็นต์อ้างอิงอนุญาตให้คำขอที่มีระยะเวลาจำกัดควบคุมเส้นตายที่กำหนดค่าไว้ เมื่อคำขอที่รอดำเนินการทุกรายการมีเส้นตาย หากคำขอ `expectFinal` ไม่มี `timeoutMs` ที่มีค่าจำกัด คำขอใดๆ ที่มี `timeoutMs: null` หรือมีทั้งคำขอแบบจำกัดเวลาและไม่จำกัดเวลา ระบบจะคงตัวเฝ้าระวัง Tick ไว้ หากเหตุการณ์ขาเข้าและการตอบกลับยังคงเงียบเกินเกณฑ์ระยะหมดเวลาของ Tick ไคลเอ็นต์จะปิดซ็อกเก็ตด้วยรหัส `4000` ปฏิเสธคำขอที่รอดำเนินการทั้งหมด และเชื่อมต่อใหม่ โดยจะไม่ส่งคำขอที่ถูกปฏิเสธซ้ำหลังเชื่อมต่อใหม่

## การยืนยันตัวตน

- การตรวจสอบสิทธิ์ Gateway ด้วยข้อมูลลับที่ใช้ร่วมกันใช้ `connect.params.auth.token` หรือ
  `connect.params.auth.password` โดยขึ้นอยู่กับ
  `gateway.auth.mode` ที่กำหนดค่าไว้ (`"none" | "token" | "password" | "trusted-proxy"`)
- โหมดที่มีข้อมูลระบุตัวตน เช่น Tailscale Serve (`gateway.auth.allowTailscale: true`)
  หรือ `gateway.auth.mode: "trusted-proxy"` ที่ไม่ใช่ลูปแบ็ก จะผ่านการตรวจสอบสิทธิ์
  การเชื่อมต่อจากส่วนหัวคำขอแทน `connect.params.auth.*`
- `gateway.auth.mode: "none"` สำหรับทางเข้าระบบส่วนตัวจะข้ามการตรวจสอบสิทธิ์การเชื่อมต่อด้วยข้อมูลลับที่ใช้ร่วมกัน
  ทั้งหมด อย่าเปิดเผยโหมดดังกล่าวบนทางเข้าระบบสาธารณะหรือที่ไม่น่าเชื่อถือ
- หลังจากจับคู่แล้ว Gateway จะออกโทเค็นอุปกรณ์ที่จำกัดขอบเขตตามบทบาท
  และขอบเขตของการเชื่อมต่อ โดยส่งคืนใน `hello-ok.auth.deviceToken` ไคลเอ็นต์ควร
  จัดเก็บโทเค็นนี้หลังจากเชื่อมต่อสำเร็จทุกครั้ง
- เมื่อเชื่อมต่อใหม่ด้วยโทเค็นอุปกรณ์ที่จัดเก็บไว้ ควรนำชุดขอบเขต
  ที่ได้รับอนุมัติและจัดเก็บไว้สำหรับโทเค็นนั้นกลับมาใช้ด้วย การดำเนินการนี้จะคงสิทธิ์เข้าถึง
  สำหรับการอ่าน การตรวจสอบ และสถานะที่ให้ไว้แล้ว และหลีกเลี่ยงการทำให้การเชื่อมต่อใหม่
  ถูกลดเหลือขอบเขตโดยนัยที่แคบกว่าและจำกัดเฉพาะผู้ดูแลระบบโดยไม่มีการแจ้งเตือน
- การประกอบข้อมูลตรวจสอบสิทธิ์การเชื่อมต่อฝั่งไคลเอ็นต์ (`selectConnectAuth` ใน
  `packages/gateway-client/src/client.ts`):
  - `auth.password` เป็นอิสระจากส่วนอื่นและจะถูกส่งต่อเสมอเมื่อกำหนดค่าไว้
  - `auth.token` จะถูกเติมตามลำดับความสำคัญ ได้แก่ โทเค็นที่ใช้ร่วมกันซึ่งระบุอย่างชัดเจนก่อน
    จากนั้นเป็น `deviceToken` ที่ระบุอย่างชัดเจน แล้วจึงเป็นโทเค็นที่จัดเก็บไว้รายอุปกรณ์ (ใช้
    `deviceId` + `role` เป็นคีย์)
  - `auth.bootstrapToken` จะถูกส่งเฉพาะเมื่อไม่มีรายการข้างต้นใดสามารถกำหนด
    `auth.token` ได้ โทเค็นที่ใช้ร่วมกันหรือโทเค็นอุปกรณ์ใดๆ ที่กำหนดได้จะระงับการส่งค่านี้
  - การเลื่อนระดับโทเค็นอุปกรณ์ที่จัดเก็บไว้โดยอัตโนมัติในการลองใหม่ครั้งเดียวของ
    `AUTH_TOKEN_MISMATCH` จำกัดไว้เฉพาะปลายทางที่เชื่อถือได้ ได้แก่ ลูปแบ็ก
    หรือ `wss://` ที่มี `tlsFingerprint` ปักหมุดไว้ `wss://` สาธารณะที่ไม่มีการปักหมุด
    ไม่เข้าเกณฑ์
- การเริ่มต้นระบบด้วยรหัสตั้งค่าที่มีมาให้จะส่งคืน
  `hello-ok.auth.deviceToken` ของ Node หลัก พร้อมโทเค็นผู้ปฏิบัติงานแบบจำกัดใน
  `hello-ok.auth.deviceTokens` สำหรับการส่งต่อไปยังอุปกรณ์เคลื่อนที่ที่เชื่อถือได้ โทเค็นผู้ปฏิบัติงาน
  มี `operator.talk.secrets` สำหรับการอ่านการกำหนดค่า Talk แบบเนทีฟ แต่
  ไม่รวมขอบเขตการเปลี่ยนแปลงการจับคู่และ `operator.admin`
- ขณะที่การเริ่มต้นระบบด้วยรหัสตั้งค่าที่ไม่ใช่ค่าพื้นฐานกำลังรอการอนุมัติ
  รายละเอียด `PAIRING_REQUIRED` จะมี `recommendedNextStep: "wait_then_retry"`,
  `retryable: true` และ `pauseReconnect: false` ให้เชื่อมต่อใหม่ต่อไปด้วย
  โทเค็นเริ่มต้นระบบเดิมจนกว่าคำขอจะได้รับอนุมัติหรือโทเค็นใช้ไม่ได้
- จัดเก็บ `hello-ok.auth.deviceTokens` เฉพาะเมื่อการเชื่อมต่อใช้การตรวจสอบสิทธิ์
  แบบเริ่มต้นระบบผ่านการรับส่งข้อมูลที่เชื่อถือได้ เช่น `wss://` หรือการจับคู่ผ่านลูปแบ็ก/ภายในเครื่อง
- หากไคลเอ็นต์ระบุ `deviceToken` หรือ `scopes` อย่างชัดเจน
  ชุดขอบเขตที่ผู้เรียกขอจะยังคงมีอำนาจสูงสุด ขอบเขตที่แคชไว้จะถูกนำกลับมาใช้
  เฉพาะเมื่อไคลเอ็นต์นำโทเค็นที่จัดเก็บไว้รายอุปกรณ์กลับมาใช้
- โทเค็นอุปกรณ์สามารถหมุนเวียนหรือเพิกถอนได้ผ่าน `device.token.rotate` และ
  `device.token.revoke` (ต้องมี `operator.pairing`) การหมุนเวียนหรือเพิกถอน
  บทบาท Node หรือบทบาทอื่นที่ไม่ใช่ผู้ปฏิบัติงานต้องมี `operator.admin` ด้วย
- `device.token.rotate` ส่งคืนข้อมูลเมตาการหมุนเวียน โดยจะส่งโทเค็นผู้ถือฉบับทดแทน
  กลับมาเฉพาะการเรียกจากอุปกรณ์เดียวกันซึ่งตรวจสอบสิทธิ์ด้วยโทเค็นอุปกรณ์นั้นแล้ว
  เพื่อให้ไคลเอ็นต์ที่ใช้เฉพาะโทเค็นจัดเก็บโทเค็นทดแทนก่อนเชื่อมต่อใหม่ได้
  การหมุนเวียนโดยใช้ข้อมูลลับที่ใช้ร่วมกันหรือสิทธิ์ผู้ดูแลระบบจะไม่ส่งโทเค็นผู้ถือกลับมา
- การออก การหมุนเวียน และการเพิกถอนโทเค็นยังคงจำกัดอยู่ในชุดบทบาท
  ที่ได้รับอนุมัติและบันทึกไว้ในรายการจับคู่ของอุปกรณ์นั้น การเปลี่ยนแปลงโทเค็นไม่สามารถขยาย
  หรือกำหนดเป้าหมายไปยังบทบาทอุปกรณ์ที่การอนุมัติการจับคู่ไม่เคยให้ไว้
- สำหรับเซสชันโทเค็นของอุปกรณ์ที่จับคู่แล้ว การจัดการอุปกรณ์จะจำกัดอยู่ที่ตนเอง เว้นแต่
  ผู้เรียกจะมี `operator.admin` ด้วย ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบสามารถจัดการได้เฉพาะ
  โทเค็นผู้ปฏิบัติงานสำหรับรายการอุปกรณ์ของตนเอง การจัดการโทเค็นของ Node และ
  บทบาทอื่นที่ไม่ใช่ผู้ปฏิบัติงานจำกัดเฉพาะผู้ดูแลระบบ แม้เป็นอุปกรณ์ของผู้เรียกเอง
- `device.token.rotate` และ `device.token.revoke` ยังตรวจสอบชุดขอบเขต
  ของโทเค็นผู้ปฏิบัติงานเป้าหมายกับขอบเขตเซสชันปัจจุบันของผู้เรียก
  ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบไม่สามารถหมุนเวียนหรือเพิกถอนโทเค็นผู้ปฏิบัติงานที่มีขอบเขตกว้างกว่า
  ขอบเขตที่ตนมีอยู่แล้ว
- ความล้มเหลวในการตรวจสอบสิทธิ์มี `error.details.code` พร้อมคำแนะนำในการกู้คืน:
  - `error.details.canRetryWithDeviceToken` (บูลีน)
  - `error.details.recommendedNextStep`: ค่าใดค่าหนึ่งจาก `retry_with_device_token`,
    `update_auth_configuration`, `update_auth_credentials`,
    `wait_then_retry`, `review_auth_configuration`
    (`packages/gateway-protocol/src/connect-error-details.ts`)
- พฤติกรรมของไคลเอ็นต์สำหรับ `AUTH_TOKEN_MISMATCH`:
  - ไคลเอ็นต์ที่เชื่อถือได้อาจลองใหม่หนึ่งครั้งแบบจำกัดด้วยโทเค็นรายอุปกรณ์
    ที่แคชไว้
  - หากการลองใหม่นั้นล้มเหลว ให้หยุดวงจรเชื่อมต่อใหม่อัตโนมัติและแสดงคำแนะนำ
    เกี่ยวกับการดำเนินการที่ผู้ปฏิบัติงานต้องทำ
- `AUTH_SCOPE_MISMATCH` หมายความว่าระบบรู้จักโทเค็นอุปกรณ์ แต่โทเค็น
  ไม่ครอบคลุมบทบาทหรือขอบเขตที่ขอ อย่าแสดงว่าเป็นโทเค็นไม่ถูกต้อง ให้แจ้ง
  ผู้ปฏิบัติงานให้จับคู่ใหม่หรืออนุมัติสัญญาขอบเขตที่แคบลงหรือกว้างขึ้น

## ข้อมูลระบุตัวตนและการจับคู่อุปกรณ์

- Node ควรมีข้อมูลระบุตัวตนอุปกรณ์ที่คงที่ (`device.id`) ซึ่งได้มาจาก
  ลายนิ้วมือของคู่กุญแจ
- Gateway ออกโทเค็นแยกตามอุปกรณ์ + บทบาท
- รหัสอุปกรณ์ใหม่ต้องได้รับการอนุมัติการจับคู่ เว้นแต่เปิดใช้
  การอนุมัติอัตโนมัติภายในเครื่อง
- การอนุมัติการจับคู่อัตโนมัติเน้นการเชื่อมต่อลูปแบ็กภายในเครื่องโดยตรง
- OpenClaw ยังมีเส้นทางเชื่อมต่อตนเองภายในแบ็กเอนด์/คอนเทนเนอร์แบบจำกัด
  สำหรับโฟลว์ตัวช่วยข้อมูลลับที่ใช้ร่วมกันซึ่งเชื่อถือได้
- การเชื่อมต่อผ่าน tailnet หรือ LAN บนโฮสต์เดียวกันยังถือเป็นการเชื่อมต่อระยะไกล
  สำหรับการจับคู่และต้องได้รับอนุมัติ
- โดยปกติไคลเอ็นต์ WS จะมีข้อมูลระบุตัวตน `device` ระหว่าง `connect` (ผู้ปฏิบัติงาน +
  Node) ข้อยกเว้นสำหรับผู้ปฏิบัติงานที่ไม่มีอุปกรณ์มีเฉพาะเส้นทางความเชื่อถือที่ระบุอย่างชัดเจน:
  - `gateway.controlUi.allowInsecureAuth=true` สำหรับความเข้ากันได้กับ HTTP ที่ไม่ปลอดภัย
    และจำกัดเฉพาะ localhost
  - การตรวจสอบสิทธิ์ Control UI ของผู้ปฏิบัติงานผ่าน `gateway.auth.mode: "trusted-proxy"` สำเร็จ
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (ทางเลือกฉุกเฉิน
    ที่ลดระดับความปลอดภัยอย่างรุนแรง)
  - RPC แบ็กเอนด์ `gateway-client` ผ่านลูปแบ็กโดยตรงบนเส้นทางตัวช่วยภายใน
    ที่สงวนไว้
- การละเว้นข้อมูลระบุตัวตนอุปกรณ์มีผลต่อขอบเขต เมื่ออนุญาตการเชื่อมต่อของ
  ผู้ปฏิบัติงานที่ไม่มีอุปกรณ์ผ่านเส้นทางความเชื่อถือที่ระบุอย่างชัดเจน OpenClaw
  จะยังคงล้างขอบเขตที่ประกาศเองให้เป็นชุดว่าง เว้นแต่เส้นทางนั้นมี
  ข้อยกเว้นการคงขอบเขตที่มีชื่อกำกับ จากนั้นเมธอดที่มีการควบคุมด้วยขอบเขตจะล้มเหลวด้วย
  `missing scope`
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` เป็นเส้นทางฉุกเฉินสำหรับ Control UI
  ที่คงขอบเขตไว้ โดยไม่ได้ให้ขอบเขตแก่ไคลเอ็นต์ WebSocket ของแบ็กเอนด์แบบกำหนดเอง
  หรือไคลเอ็นต์ที่มีรูปแบบ CLI โดยพลการ
- เส้นทางตัวช่วยแบ็กเอนด์ `gateway-client` ผ่านลูปแบ็กโดยตรงที่สงวนไว้จะคง
  ขอบเขตเฉพาะ RPC ของระนาบควบคุมภายในเครื่องเท่านั้น รหัสแบ็กเอนด์แบบกำหนดเอง
  จะไม่ได้รับข้อยกเว้นนี้
- การเชื่อมต่อทั้งหมดต้องลงนาม nonce `connect.challenge` ที่เซิร์ฟเวอร์ให้มา

### การวินิจฉัยการย้ายระบบตรวจสอบสิทธิ์อุปกรณ์

สำหรับไคลเอ็นต์แบบเดิมที่ยังคงใช้พฤติกรรมการลงนามก่อนการท้าทาย `connect`
จะส่งคืนรหัสรายละเอียด `DEVICE_AUTH_*` ภายใต้ `error.details.code` พร้อม
`error.details.reason` ที่คงที่

ความล้มเหลวในการย้ายระบบที่พบบ่อย:

| ข้อความ                     | details.code                     | details.reason           | ความหมาย                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | ไคลเอ็นต์ละเว้น `device.nonce` (หรือส่งค่าว่าง)     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | ไคลเอ็นต์ลงนามด้วย nonce ที่เก่าหรือไม่ถูกต้อง            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | เพย์โหลดลายเซ็นไม่ตรงกับเพย์โหลด v2       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | การประทับเวลาที่ลงนามอยู่นอกช่วงความคลาดเคลื่อนที่อนุญาต          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` ไม่ตรงกับลายนิ้วมือกุญแจสาธารณะ |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | รูปแบบกุญแจสาธารณะหรือการปรับให้อยู่ในรูปแบบมาตรฐานล้มเหลว         |

เป้าหมายการย้ายระบบ:

- รอ `connect.challenge` เสมอ
- ลงนามเพย์โหลด v2 ที่มี nonce ของเซิร์ฟเวอร์
- ส่ง nonce เดียวกันใน `connect.params.device.nonce`
- เพย์โหลดลายเซ็นที่แนะนำคือ `v3`
  (`buildDeviceAuthPayloadV3` ใน `packages/gateway-client/src/device-auth.ts`)
  ซึ่งผูก `platform` และ `deviceFamily` เพิ่มเติมจาก
  ฟิลด์อุปกรณ์/ไคลเอ็นต์/บทบาท/ขอบเขต/โทเค็น/nonce
- ยังคงยอมรับลายเซ็น `v2` แบบเดิมเพื่อความเข้ากันได้ แต่การปักหมุด
  ข้อมูลเมตาของอุปกรณ์ที่จับคู่แล้วยังคงควบคุมนโยบายคำสั่งเมื่อเชื่อมต่อใหม่

## TLS และการปักหมุด

- รองรับ TLS สำหรับการเชื่อมต่อ WS (การกำหนดค่า `gateway.tls`)
- ไคลเอ็นต์สามารถเลือกปักหมุดลายนิ้วมือใบรับรองของ Gateway ผ่าน
  `gateway.remote.tlsFingerprint` หรือ CLI `--tls-fingerprint`

## ขอบเขต

โปรโตคอลนี้เปิดเผย API ทั้งหมดของ Gateway ได้แก่ สถานะ ช่องทาง โมเดล แชต
เอเจนต์ เซสชัน Node การอนุมัติ และอื่นๆ พื้นผิวที่แน่นอนกำหนดโดย
สคีมา TypeBox ที่ส่งออกซ้ำจาก `packages/gateway-protocol/src/schema.ts`
