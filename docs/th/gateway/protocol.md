โปรโตคอล WS ของ Gateway เป็นระนาบควบคุมและการขนส่ง Node แบบรวมศูนย์สำหรับ
OpenClaw ไคลเอ็นต์ผู้ปฏิบัติงานและ Node (CLI, UI เว็บ, แอป macOS, Node iOS/Android,
Node แบบไม่มีส่วนติดต่อผู้ใช้) เชื่อมต่อผ่าน WebSocket และประกาศ **บทบาท** และ **ขอบเขต** ณ
เวลาจับมือเชื่อมต่อ

## การขนส่งและการจัดเฟรม

- WebSocket, เฟรมข้อความ, เพย์โหลด JSON
- เฟรมแรก **ต้อง** เป็นคำขอ `connect`
- เฟรมก่อนเชื่อมต่อจำกัดไว้ที่ 64 KiB (`MAX_PREAUTH_PAYLOAD_BYTES`) หลังจาก
  จับมือเชื่อมต่อแล้ว ให้เป็นไปตาม `hello-ok.policy.maxPayload` และ
  `hello-ok.policy.maxBufferedBytes` เมื่อเปิดใช้การวินิจฉัย เฟรมขาเข้าที่มีขนาดเกินกำหนด
  และบัฟเฟอร์ขาออกที่ทำงานช้าจะส่งเหตุการณ์ `payload.large` ก่อนที่
  Gateway จะปิดการเชื่อมต่อหรือทิ้งเฟรม เหตุการณ์เหล่านี้มี `surface`, ขนาดเป็นไบต์,
  ขีดจำกัด และรหัสเหตุผลที่ปลอดภัย โดยจะไม่มีเนื้อความของข้อความ เนื้อหาไฟล์แนบ
  ไบต์เฟรมดิบ โทเค็น คุกกี้ หรือข้อมูลลับ

รูปแบบเฟรม:

- คำขอ: `{type:"req", id, method, params}`
- การตอบกลับ: `{type:"res", id, ok, payload|error}`
- เหตุการณ์: `{type:"event", event, payload, seq?, stateVersion?}`

ข้อผิดพลาดในการตอบกลับใช้ `{ code, message, details?, retryable?, retryAfterMs? }`
ไคลเอ็นต์ควรแยกเส้นทางตาม `code` และ `details.code`; `message` ยังคงอยู่ในรูปแบบที่มนุษย์อ่านได้
และอาจเปลี่ยนแปลงได้ เว้นแต่หมายเหตุด้านความเข้ากันได้จะระบุไว้เป็นอย่างอื่น ความล้มเหลวในการอนุญาต
ระดับเมธอดใช้ `code: "FORBIDDEN"` ระดับบนสุด พร้อมรายละเอียดขอบเขตที่ขาดหายไป
แบบมีโครงสร้าง:

- ขอบเขตที่ขาดหายไป: `{ code: "MISSING_SCOPE", missingScope, requiredScopes }`
  `requiredScopes` คือชุดขอบเขตที่รู้จักทั้งหมดสำหรับการดำเนินการที่ร้องขอ
  ข้อความเดิม `missing scope: <scope>` ยังคงเก็บไว้สำหรับไคลเอ็นต์รุ่นเก่า

ไคลเอ็นต์ควรอ่าน `details` ก่อน และใช้ข้อความเดิมเป็นเพียงทางเลือกสำรอง
เพื่อความเข้ากันได้ `readMissingScopeError` และ `readMissingScopeErrorDetails` ส่งออกจาก
`@openclaw/gateway-protocol/gateway-error-details`; ไคลเอ็นต์ Gateway ที่ปลอดภัยสำหรับเบราว์เซอร์
ส่งออกทั้งสองรายการอีกครั้งจาก `@openclaw/gateway-client/browser`

สคีมาส่งออกในชื่อ `GatewayErrorDetailsSchema`,
`MissingScopeErrorDetailsSchema` จาก `@openclaw/gateway-protocol/schema`
ความล้มเหลวของขอบเขต HTTP สะท้อนออบเจ็กต์ `MISSING_SCOPE` ภายใต้ `error.details` และ
ใช้สถานะ HTTP `403`

เมธอดที่ก่อให้เกิดผลข้างเคียงต้องใช้คีย์ภาวะทำซ้ำได้อย่างปลอดภัย (ดูสคีมา)

## การจับมือเชื่อมต่อ

Gateway ส่งคำท้าทายก่อนเชื่อมต่อ:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

ไคลเอ็นต์ตอบกลับด้วย `connect`:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway ตอบกลับด้วย `hello-ok`:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`, `features`, `snapshot`, `policy` และ `auth` ล้วนเป็นสิ่งที่
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) กำหนดให้ต้องมี `auth`
รายงานบทบาท/ขอบเขตที่เจรจาตกลงกัน แม้จะไม่มีการออกโทเค็นอุปกรณ์ (รูปแบบ
ดังข้างต้น) `pluginSurfaceUrls` เป็นตัวเลือก และแมปชื่อพื้นผิว Plugin (เช่น
`canvas`) ไปยัง URL ที่โฮสต์และกำหนดขอบเขตไว้ ซึ่งอาจหมดอายุได้ ดังนั้น Node จึงเรียก
`node.pluginSurface.refresh` พร้อม `{ "surface": "canvas" }` เพื่อรับรายการใหม่
เส้นทาง `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
ที่เลิกใช้แล้วไม่ได้รับการรองรับ ให้ใช้พื้นผิว Plugin
`appliedConfigHash` ซึ่งเป็นตัวเลือกในสแนปช็อต คือรุ่นแก้ไขของการกำหนดค่าต้นทางที่ผ่านการแก้ค่าแล้ว
ซึ่งรันไทม์ Gateway ที่ทำงานอยู่ยอมรับ ไคลเอ็นต์สามารถเปรียบเทียบกับ
`config.get.configRevisionHash` เพื่อพิจารณาว่าการกำหนดค่าที่บันทึกไว้ซึ่งใหม่กว่ายังคง
ต้องเริ่มระบบใหม่หรือไม่ `config.get.hash` ยังคงเป็นรุ่นแก้ไขดิบของไฟล์รากที่ใช้โดย
กลไกป้องกันข้อขัดแย้งในการเขียนการกำหนดค่า

ขณะที่ Gateway ยังเริ่มต้นไซด์คาร์ไม่เสร็จ `connect` อาจส่งคืน
ข้อผิดพลาด `UNAVAILABLE` ที่ลองใหม่ได้ พร้อม `details.reason: "startup-sidecars"` และ
`retryAfterMs` ให้ลองใหม่ภายในงบประมาณการเชื่อมต่อ แทนที่จะถือว่าเป็น
ความล้มเหลวขั้นสุดท้ายของการจับมือเชื่อมต่อ

เมื่อมีการออกโทเค็นอุปกรณ์ `hello-ok.auth` จะเพิ่มโทเค็นดังกล่าว:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

การเริ่มต้นระบบด้วย QR/รหัสตั้งค่าในตัวเป็นเส้นทางส่งต่อไปยังอุปกรณ์เคลื่อนที่ การเชื่อมต่อ
ด้วยรหัสตั้งค่าพื้นฐานที่สำเร็จจะส่งคืนโทเค็น Node หลัก พร้อมโทเค็น
ผู้ปฏิบัติงานที่มีขอบเขตจำกัดหนึ่งรายการ:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

การส่งต่อให้ผู้ปฏิบัติงานนี้ถูกจำกัดโดยเจตนา: เพียงพอสำหรับเริ่มลูป
ผู้ปฏิบัติงานบนอุปกรณ์เคลื่อนที่และการตั้งค่าแบบเนทีฟ รวมถึง `operator.talk.secrets` สำหรับการอ่าน
การกำหนดค่า Talk แต่ไม่มีขอบเขตสำหรับเปลี่ยนแปลงการจับคู่ และไม่มี `operator.admin` การเข้าถึง
การจับคู่/ผู้ดูแลระบบที่กว้างกว่านี้ต้องใช้การจับคู่หรือขั้นตอนโทเค็นที่ได้รับอนุมัติแยกต่างหาก ให้จัดเก็บ
`hello-ok.auth.deviceTokens` เฉพาะเมื่อการตรวจสอบสิทธิ์สำหรับเริ่มต้นระบบทำงานผ่านการขนส่ง
ที่เชื่อถือได้ (`wss://` หรือการจับคู่ผ่านลูปแบ็ก/ภายในเครื่อง)

ไคลเอ็นต์แบ็กเอนด์ในกระบวนการเดียวกันที่เชื่อถือได้ (`client.id: "gateway-client"`,
`client.mode: "backend"`) อาจละ `device` ในการเชื่อมต่อลูปแบ็กโดยตรง เมื่อ
ตรวจสอบสิทธิ์ด้วยโทเค็น/รหัสผ่าน Gateway ที่ใช้ร่วมกัน เส้นทางนี้สงวนไว้
สำหรับ RPC ของระนาบควบคุมภายใน (เช่น การอัปเดตเซสชันของเอเจนต์ย่อย) และหลีกเลี่ยงไม่ให้
ข้อมูลพื้นฐานการจับคู่ CLI/อุปกรณ์ที่ล้าสมัยขัดขวางงานแบ็กเอนด์ภายในเครื่อง ไคลเอ็นต์ระยะไกล
ที่มาจากเบราว์เซอร์, Node และไคลเอ็นต์ที่ระบุโทเค็นอุปกรณ์/ข้อมูลประจำตัวอุปกรณ์อย่างชัดเจน ยังคง
ผ่านการตรวจสอบการจับคู่และการยกระดับขอบเขตตามปกติ

### บทบาทของเวิร์กเกอร์และโปรโตคอลแบบปิด

เวิร์กเกอร์บนคลาวด์ใช้ช่องทางเข้าลูปแบ็กเฉพาะผ่านอุโมงค์ SSH ที่ Gateway เป็นเจ้าของ
และตรึงคีย์โฮสต์ไว้ ช่องทางนี้ยอมรับเฉพาะข้อมูลประจำตัวของเวิร์กเกอร์ และจะไม่ส่งต่อ
การตรวจสอบสิทธิ์ทั่วไป เหตุการณ์ Node, RPC ของผู้ปฏิบัติงาน หรือเมธอด Plugin โดยเด็ดขาด `connect`
ที่เข้มงวดจะตรวจสอบข้อมูลประจำตัวอายุสั้นซึ่งจัดเก็บเป็นแฮชและผูกกับสภาพแวดล้อม แฮชของบันเดิล
ยุคของเจ้าของ เวอร์ชันชุด RPC เวลาหมดอายุ และเซสชันที่เป็นค่าว่างได้หนึ่งรายการ พร้อมทั้ง
ตรวจสอบเวอร์ชันและชุดคุณลักษณะปัจจุบันแยกต่างหาก เมื่อสำเร็จจะส่งคืน
`worker-hello-ok` ขั้นต่ำ การเจรจาคุณลักษณะแยกจากเวอร์ชันโปรโตคอลทั่วไป
เฟรมยังคงมีขนาดต่ำกว่า 64 KiB ยกเว้นเฟรม `worker.inference.start`
ที่เจรจาไว้ ซึ่งอาจมีขนาดสูงสุด 25 MiB รายการอนุญาตแบบปิดประกอบด้วย `worker.heartbeat`,
`worker.transcript.commit`, `worker.live-event`, `worker.inference.start` และ
`worker.inference.cancel`

การคอมมิตทรานสคริปต์ใช้การกั้นด้วยยุคของเจ้าของ การผูกเซสชันที่ Gateway เป็นเจ้าของ
การเปรียบเทียบและสลับค่าของลีฟฐาน และการเล่นซ้ำตามลำดับอย่างถาวร โดย Gateway จะสร้าง
รหัสรายการทรานสคริปต์และรหัสพาเรนต์ผ่านตัวเขียนเซสชันตามปกติ ระบบจะตรวจสอบความเป็นเจ้าของและ
เวลาหมดอายุซ้ำในแต่ละ RPC

### ความสามารถของไคลเอ็นต์

ไคลเอ็นต์ผู้ปฏิบัติงานอาจประกาศความสามารถที่เป็นตัวเลือกใน `connect.params.caps`:

- `tool-events`: ยอมรับเหตุการณ์วงจรชีวิตของเครื่องมือแบบมีโครงสร้าง
- `inline-widgets`: สามารถเรนเดอร์ผลลัพธ์เครื่องมือวิดเจ็ตแบบอินไลน์ที่โฮสต์ไว้

ความสามารถของไคลเอ็นต์อธิบายไคลเอ็นต์ที่เชื่อมต่อ ไม่ใช่การอนุญาต เครื่องมือเอเจนต์อาจประกาศความสามารถที่จำเป็น โดย Gateway จะละเครื่องมือเหล่านั้น เว้นแต่ข้อกำหนดทุกรายการจะปรากฏใน `caps` ของไคลเอ็นต์ต้นทาง การทำงานที่มีต้นทางจากช่องทางไม่มีความสามารถของไคลเอ็นต์ Gateway ดังนั้นเครื่องมือที่ถูกจำกัดด้วยความสามารถจะใช้งานไม่ได้ แม้นโยบายเครื่องมือจะอนุญาตอย่างชัดเจนก็ตาม

### ตัวอย่างการเชื่อมต่อของ Node

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "ios-node",
      "version": "1.2.3",
      "platform": "ios",
      "mode": "node"
    },
    "role": "node",
    "scopes": [],
    "caps": ["camera", "canvas", "screen", "location", "voice"],
    "commands": ["camera.snap", "canvas.navigate", "screen.record", "location.get"],
    "permissions": { "camera.capture": true, "screen.record": false },
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-ios/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Node ประกาศการอ้างความสามารถ ณ เวลาเชื่อมต่อ:

- `caps`: หมวดหมู่ระดับสูง เช่น `camera`, `canvas`, `screen`,
  `location`, `voice`, `talk`
- `commands`: รายการอนุญาตคำสั่งสำหรับการเรียกใช้
- `permissions`: ตัวสลับแบบละเอียด (เช่น `screen.record`, `camera.capture`)

Gateway ถือว่าสิ่งเหล่านี้เป็นการอ้างความสามารถ และบังคับใช้รายการอนุญาตฝั่งเซิร์ฟเวอร์

## บทบาทและขอบเขต

สำหรับโมเดลขอบเขตของผู้ปฏิบัติงานทั้งหมด การตรวจสอบ ณ เวลาอนุมัติ และความหมายของข้อมูลลับ
ที่ใช้ร่วมกัน โปรดดู [ขอบเขตของผู้ปฏิบัติงาน](/th/gateway/operator-scopes)

บทบาท:

- `operator`: ไคลเอ็นต์ระนาบควบคุม (CLI/UI/ระบบอัตโนมัติ)
- `node`: โฮสต์ความสามารถ (กล้อง/หน้าจอ/แคนวาส/system.run)
- `worker`: โฮสต์การดำเนินการบนคลาวด์บนโปรโตคอลเวิร์กเกอร์เฉพาะแบบปิด

ขอบเขตของผู้ปฏิบัติงาน (`src/gateway/operator-scopes.ts`) ซึ่งเป็นชุดปิดทั้งหมด:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`talk.config` ที่มี `includeSecrets: true` ต้องใช้ `operator.talk.secrets` (หรือ
`operator.admin`) เมื่อรวมข้อมูลลับ ให้อ่านข้อมูลประจำตัวของผู้ให้บริการ Talk ที่ทำงานอยู่
จาก `talk.resolved.config.apiKey`; `talk.providers.<id>.apiKey`
ยังคงมีรูปแบบตามต้นทาง และอาจเป็นออบเจ็กต์ SecretRef หรือสตริงที่ปกปิดข้อมูลแล้ว

เมธอด RPC ของ Gateway ที่ลงทะเบียนโดย Plugin อาจร้องขอขอบเขตผู้ปฏิบัติงานของตนเอง
แต่คำนำหน้าหลักที่สงวนไว้เหล่านี้จะถูกแก้ค่าเป็น `operator.admin`
(`src/shared/gateway-method-policy.ts`) เสมอ: `config.*`, `exec.approvals.*`,
`wizard.*`, `update.*`

ขอบเขตของเมธอดเป็นเพียงด่านแรก คำสั่งแบบทับบางรายการที่เข้าถึงผ่าน
`chat.send` ใช้การตรวจสอบระดับคำสั่งที่เข้มงวดกว่า: การเขียน `/config set` และ
`/config unset` แบบถาวรต้องใช้ `operator.admin` แม้สำหรับไคลเอ็นต์ Gateway ที่
มีขอบเขตผู้ปฏิบัติงานระดับต่ำกว่าอยู่แล้ว

`node.pair.approve` มีการตรวจสอบขอบเขต ณ เวลาอนุมัติเพิ่มเติม นอกเหนือจาก
ขอบเขตพื้นฐานของเมธอด (`operator.pairing`) โดยอิงตาม `commands`
(`src/infra/node-pairing-authz.ts`) ที่คำขอซึ่งรอดำเนินการประกาศไว้:

| คำสั่งที่ประกาศ                                                                                                             | ขอบเขตที่ต้องใช้                       |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| ไม่มี                                                                                                                          | `operator.pairing`                    |
| คำสั่งทั่วไป                                                                                                             | `operator.pairing` + `operator.write` |
| มี `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir` หรือ `system.execApprovals.get/set` | `operator.pairing` + `operator.admin` |

### ความสามารถ/คำสั่ง/สิทธิ์ (Node)

Node ประกาศการอ้างความสามารถ ณ เวลาเชื่อมต่อ:

- `caps`: หมวดหมู่ความสามารถระดับสูง เช่น `camera`, `canvas`, `screen`,
  `location`, `voice` และ `talk`
- `commands`: รายการคำสั่งที่อนุญาตสำหรับการเรียกใช้
- `permissions`: ตัวสลับแบบละเอียด (เช่น `screen.record`, `camera.capture`)

Gateway ถือว่าสิ่งเหล่านี้เป็น **คำกล่าวอ้าง** และบังคับใช้รายการที่อนุญาตในฝั่งเซิร์ฟเวอร์
Node ที่เชื่อมต่อสามารถเผยแพร่ตัวอธิบาย Plugin หรือเครื่องมือ MCP ที่เอเจนต์มองเห็นได้โดยไม่บังคับ
ผ่าน `node.pluginTools.update` หลังจากเชื่อมต่อหรือ
เชื่อมต่อใหม่สำเร็จ โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้จะรีสตาร์ตเพื่อใช้การเปลี่ยนแปลง
รายการ MCP แบบประกาศ วิธีอัปเดตนี้เป็นช่องทางเผยแพร่เพียงช่องทางเดียว โดยไม่ยอมรับตัวอธิบายเครื่องมือ Plugin ใน
พารามิเตอร์ `connect` ตัวอธิบายแต่ละรายการต้องใช้ `name` ของเครื่องมือที่ปลอดภัยสำหรับผู้ให้บริการ และระบุ
`command` ซึ่งอยู่ในรายการคำสั่งที่อนุญาตปัจจุบันของ Node Gateway เชื่อถือข้อมูลเมตาของตัวอธิบาย
จาก Node ที่จับคู่ กรองตัวอธิบายที่อยู่นอกขอบเขตคำสั่งที่ได้รับอนุมัติ
นำออกเมื่อ Node ตัดการเชื่อมต่อ และปฏิเสธความพยายามของผู้ควบคุมระบบ
ในการแก้ไขแค็ตตาล็อกของ Node อื่น ตั้งค่า `gateway.nodes.pluginTools.enabled: false`
เพื่อไม่ใช้ตัวอธิบายที่ Node เผยแพร่

โฮสต์ Node ที่เชื่อมต่อเผยแพร่แค็ตตาล็อก Skills ทดแทนทั้งหมดผ่าน
`node.skills.update` เมธอดสำหรับบทบาท Node นี้เป็นช่องทางเผยแพร่ Skills ของ Node
เพียงช่องทางเดียว โดยไม่ยอมรับ Skills ในพารามิเตอร์ `connect` ตัวอธิบายแต่ละรายการประกอบด้วย
ชื่อที่ปลอดภัย คำอธิบาย และเนื้อหา `SKILL.md` ที่มีขอบเขตจำกัด Gateway แยกวิเคราะห์
เนื้อหาดังกล่าวด้วยตัวโหลด Skills ตามปกติ รวมไว้ในสแนปช็อต Skills ของเอเจนต์
ขณะที่ Node เชื่อมต่ออยู่ และนำออกเมื่อตัดการเชื่อมต่อ ตั้งค่า
`gateway.nodes.skills.enabled: false` เพื่อไม่ใช้ Skills ที่ Node เผยแพร่

## สถานะการปรากฏตัว

- `system-presence` ส่งคืนรายการที่ใช้ข้อมูลประจำตัวของอุปกรณ์เป็นคีย์ รวมถึง
  `deviceId`, `roles` และ `scopes` เพื่อให้ UI แสดงหนึ่งแถวต่ออุปกรณ์ได้ แม้ว่า
  อุปกรณ์นั้นจะเชื่อมต่อทั้งในฐานะผู้ควบคุมระบบและ Node
- `node.list` มี `lastSeenAtMs` และ `lastSeenReason` ซึ่งเป็นตัวเลือกเพิ่มเติม Node ที่เชื่อมต่อ
  รายงานเวลาเชื่อมต่อปัจจุบันพร้อมเหตุผล `connect` นอกจากนี้ Node ที่จับคู่แล้ว
  ยังสามารถรายงานสถานะการปรากฏตัวในเบื้องหลังแบบคงอยู่ผ่านเหตุการณ์ Node ที่เชื่อถือได้

Node บน macOS แบบเนทีฟยังสามารถส่งเหตุการณ์ `node.presence.activity` ที่ผ่านการตรวจสอบสิทธิ์
พร้อมระยะเวลาที่ไม่มีการป้อนข้อมูลซึ่งมีขอบเขตจำกัด Gateway คำนวณการประทับเวลากิจกรรมด้วย
นาฬิกาของตนเอง แสดง Mac ที่เชื่อมต่อล่าสุดผ่าน `node.list` และ
`node.describe` และกระจายการอัปเดต `node.presence` ไปยังไคลเอนต์ที่มีขอบเขตสิทธิ์อ่าน
ดูพฤติกรรมการเลือก ความเป็นส่วนตัว บริบทโมเดล
และการกำหนดเส้นทางการแจ้งเตือนได้ที่ [สถานะการปรากฏตัวของคอมพิวเตอร์ที่ใช้งานอยู่](/th/nodes/presence)

### เหตุการณ์ Node ยังทำงานอยู่ในเบื้องหลัง

Node เรียก `node.event` พร้อม `event: "node.presence.alive"` เพื่อบันทึกว่า
Node ที่จับคู่แล้วยังทำงานอยู่ระหว่างการปลุกในเบื้องหลัง โดยไม่ทำเครื่องหมายว่าเชื่อมต่ออยู่:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` เป็น enum แบบปิด: `background`, `silent_push`, `bg_app_refresh`,
`significant_location`, `manual`, `connect` ค่าที่ไม่รู้จักจะถูกปรับให้เป็น
`background` (`src/shared/node-presence.ts`) เหตุการณ์จะคงอยู่เฉพาะสำหรับ
เซสชันอุปกรณ์ Node ที่ผ่านการตรวจสอบสิทธิ์เท่านั้น เซสชันที่ไม่มีอุปกรณ์หรือไม่ได้จับคู่จะส่งคืน
`handled: false`

Gateway ที่ดำเนินการสำเร็จจะส่งคืนผลลัพธ์แบบมีโครงสร้าง:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

Gateway รุ่นเก่าอาจส่งคืนเพียง `{ "ok": true }` สำหรับ `node.event` ให้ถือว่า
เป็น RPC ที่ได้รับการตอบรับแล้ว ไม่ใช่การคงสถานะการปรากฏตัวแบบถาวร

## การกำหนดขอบเขตเหตุการณ์แบบกระจาย

เหตุการณ์แบบกระจายที่เซิร์ฟเวอร์พุชจะถูกควบคุมด้วยขอบเขตสิทธิ์ เพื่อให้เซสชัน
ที่มีขอบเขตเฉพาะการจับคู่หรือเฉพาะ Node ไม่ได้รับเนื้อหาเซสชันโดยอัตโนมัติ
(`src/gateway/server-broadcast.ts`):

- เฟรมแชต เอเจนต์ และผลลัพธ์เครื่องมือ (เหตุการณ์ `agent` แบบสตรีม เหตุการณ์ผลลัพธ์
  เครื่องมือ) ต้องมีอย่างน้อย `operator.read` เซสชันที่ไม่มีสิทธิ์ดังกล่าวจะข้าม
  เฟรมเหล่านี้ทั้งหมด
- การกระจาย `plugin.*` ที่กำหนดโดย Plugin จะถูกจำกัดไว้ที่ `operator.write` หรือ
  `operator.admin` โดยค่าเริ่มต้น ส่วนรายการที่ระบุอย่างชัดเจน เช่น
  `plugin.approval.requested` / `plugin.approval.resolved` จะใช้
  `operator.approvals` แทน
- เหตุการณ์สถานะ/การขนส่ง (`heartbeat`, `presence`, `tick` และวงจรการเชื่อมต่อ/
  ตัดการเชื่อมต่อ) ยังคงไม่ถูกจำกัด เพื่อให้ทุกเซสชันที่ผ่านการตรวจสอบสิทธิ์
  สังเกตสถานะความพร้อมของการขนส่งได้
- กลุ่มเหตุการณ์แบบกระจายที่ไม่รู้จักจะถูกควบคุมด้วยขอบเขตสิทธิ์โดยค่าเริ่มต้น (ปิดเมื่อเกิดข้อผิดพลาด)
  เว้นแต่ตัวจัดการที่ลงทะเบียนไว้จะผ่อนปรนข้อจำกัดอย่างชัดเจน

การเชื่อมต่อของไคลเอนต์แต่ละรายการจะเก็บหมายเลขลำดับแยกสำหรับไคลเอนต์นั้น เพื่อให้การกระจาย
ยังคงเรียงลำดับเพิ่มขึ้นอย่างสม่ำเสมอบนซ็อกเก็ตนั้น แม้ว่าไคลเอนต์แต่ละรายจะเห็น
ชุดย่อยของสตรีมเหตุการณ์ที่ผ่านการกรองตามขอบเขตต่างกัน

## กลุ่มเมธอด RPC

`hello-ok.features.methods` เป็นรายการการค้นพบแบบระมัดระวังที่สร้างจาก
`src/gateway/server-methods-list.ts` รวมกับการส่งออกเมธอดของ Plugin/ช่องทาง
ที่โหลดแล้ว ไม่ใช่รายการเมธอดทั้งหมดที่สร้างขึ้นโดยอัตโนมัติ และบางเมธอด (เช่น
`push.test`, `web.login.start`, `web.login.wait`, `sessions.usage`)
ถูกแยกออกจากการค้นพบโดยเจตนา แม้ว่าจะเป็นเมธอดจริงที่เรียกใช้ได้
ให้ถือว่านี่เป็นการค้นพบคุณสมบัติ ไม่ใช่รายการทั้งหมดของ
`src/gateway/server-methods/*.ts`

  <Accordion title="การจัดการ Plugin">
    - `plugins.list` (`operator.read`) ส่งคืนรายการ Plugin ที่ติดตั้ง พร้อมรายการแนะนำอย่างเป็นทางการที่คัดสรรไว้ในเครื่อง ข้อมูลวินิจฉัย และข้อมูลว่าโหมดการติดตั้งปัจจุบันอนุญาตให้แก้ไขหรือไม่

    - `plugins.install` (`operator.admin`) ติดตั้งรายการแค็ตตาล็อกอย่างเป็นทางการด้วย `{ source: "official", pluginId }` หรือแพ็กเกจ ClawHub ด้วย `{ source: "clawhub", packageName, version?, acknowledgeClawHubRisk? }` การติดตั้งจาก ClawHub จะรักษาการตรวจสอบความเชื่อถือ ความสมบูรณ์ และนโยบายการติดตั้งของ Gateway การติดตั้งที่สำเร็จต้องรีสตาร์ต Gateway
    - `plugins.setEnabled` (`operator.admin`) เปลี่ยนนโยบายการเปิดใช้งานของ Plugin ที่ติดตั้งหนึ่งรายการด้วย `{ pluginId, enabled }` การตอบกลับประกอบด้วยรายการแค็ตตาล็อกที่อัปเดต ข้อมูลเมตาการรีสตาร์ต และคำเตือนเกี่ยวกับการเลือกสล็อต
    - `plugins.uninstall` (`operator.admin`) ลบ Plugin ที่ติดตั้งจากภายนอกหนึ่งรายการด้วย `{ pluginId }` ได้แก่ การอ้างอิงการกำหนดค่า ระเบียนการติดตั้ง และไฟล์ที่จัดการ ไม่สามารถถอนการติดตั้ง Plugin ที่รวมมาให้ได้ แต่ปิดใช้งานได้เท่านั้น การตอบกลับจะแสดงรายการการดำเนินการลบ และต้องรีสตาร์ต Gateway เสมอ

  </Accordion>

    - `terminal.input`, `terminal.resize` และ `terminal.close` ทำงานเฉพาะกับเซสชันที่การเชื่อมต่อซึ่งเรียกใช้เป็นเจ้าของเท่านั้น
    - `terminal.upload` รับไฟล์ base64 หนึ่งไฟล์ขนาดสูงสุด 16 MiB จัดเตรียมไฟล์ไว้ในไดเรกทอรีชั่วคราวส่วนตัวอายุ 24 ชั่วโมงบน Gateway ของเซสชันหรือโฮสต์โหนดที่จับคู่ และส่งคืนพาธแบบสัมบูรณ์ ผู้เรียกยังคงต้องวางหรือใช้พาธนั้นด้วยวิธีอื่น RPC จะไม่เขียนอินพุตไปยังเทอร์มินัลหรือเรียกใช้คำสั่ง
    - เหตุการณ์ `terminal.data` และ `terminal.exit` จะสตรีมไปยังการเชื่อมต่อที่เป็นเจ้าของเซสชันเท่านั้น

    - `terminal.list` ส่งคืนเซสชันที่เชื่อมต่อได้; `terminal.attach` ผูกเซสชันที่ทำงานอยู่หรือถูกแยกออกเข้ากับการเชื่อมต่อที่เรียกใช้ใหม่ และส่งคืนบัฟเฟอร์สำหรับเล่นซ้ำ (การเข้าควบคุมแบบ tmux — เจ้าของเดิมที่ยังทำงานอยู่จะได้รับ `terminal.exit` พร้อมเหตุผล `detached`); `terminal.text` อ่านบัฟเฟอร์เป็นข้อความธรรมดาโดยไม่เชื่อมต่อ
    - เมธอดเทอร์มินัลทุกเมธอดต้องใช้ `operator.admin`; ต้องกำหนด `gateway.terminal.enabled` เป็น true อย่างชัดเจน ระบบจะปฏิเสธเอเจนต์ที่อยู่ในแซนด์บ็อกซ์เต็มรูปแบบ และการเปลี่ยนนโยบายเอเจนต์จะปิด PTY ที่มีอยู่และกำลังดำเนินการทั้งหมด รวมถึง PTY ที่ถูกแยกออก

  <Accordion title="การสนทนาและ TTS">

    - `talk.client.transcript` เพิ่มรายการ `{ role, text }` ที่เสร็จสมบูรณ์หนึ่งรายการไปยังเซสชันเอเจนต์ปกติ `entryId` ที่จำเป็นมีคุณสมบัติทำซ้ำได้โดยไม่เกิดผลซ้ำภายใน `voiceSessionId`; การลองใหม่จะไม่ทำให้ข้อความบันทึกการสนทนาซ้ำ
    - `talk.client.close` ปิดเซสชันเสียงเชิงตรรกะหลังการเขียนบันทึกการสนทนาที่ค้างอยู่ การปิดมีคุณสมบัติทำซ้ำได้โดยไม่เกิดผลซ้ำ และอาจส่งสรุปการเรียกที่มีเฉพาะการเปลี่ยนแปลงไปยังช่องทางล่าสุดของเซสชันที่ไม่ใช่ WebChat
    - `talk.client.toolCall` ช่วยให้การขนส่งแบบเรียลไทม์ที่ไคลเอนต์เป็นเจ้าของส่งต่อการเรียกเครื่องมือของผู้ให้บริการไปยังนโยบายของ Gateway เครื่องมือแรกที่รองรับคือ `openclaw_agent_consult`; ไคลเอนต์จะได้รับ ID รันและรอเหตุการณ์วงจรชีวิตแชตตามปกติก่อนส่งผลลัพธ์เครื่องมือเฉพาะของผู้ให้บริการ การดำเนินการที่มีผลกระทบสูงและผูกกับเสียงจะส่งคืน `VOICE_CONFIRMATION_REQUIRED:<id>` จนกว่าคำพูดของผู้ใช้ที่เสร็จสมบูรณ์ในภายหลังจะยืนยันการดำเนินการนั้นอย่างชัดเจน และการปรึกษาครั้งถัดไปจะระบุ `confirmationId`
    - `talk.client.steer` ส่งการควบคุมด้วยเสียงของรันที่กำลังทำงานสำหรับการขนส่งแบบเรียลไทม์ที่ไคลเอนต์เป็นเจ้าของ Gateway จะแก้หารันแบบฝังที่กำลังทำงานจาก `sessionKey` และส่งคืนผลลัพธ์การยอมรับ/ปฏิเสธแบบมีโครงสร้างแทนการละทิ้งคำสั่งควบคุมอย่างเงียบ ๆ
    - `talk.event` เป็นช่องทางเหตุการณ์ Talk เดียวสำหรับอะแดปเตอร์แบบเรียลไทม์ การถอดเสียง STT/TTS ห้องที่มีการจัดการ โทรศัพท์ และการประชุม
    - `talk.speak` สังเคราะห์เสียงพูดผ่านผู้ให้บริการเสียงพูด Talk ที่กำลังใช้งาน
    - `tts.status` ส่งคืนสถานะเปิดใช้งาน TTS ผู้ให้บริการที่กำลังใช้งาน ผู้ให้บริการสำรอง และสถานะการกำหนดค่าผู้ให้บริการ
    - `tts.providers` ส่งคืนรายการผู้ให้บริการ TTS ที่มองเห็นได้
    - `tts.enable` และ `tts.disable` สลับสถานะค่ากำหนด TTS
    - `tts.setProvider` อัปเดตผู้ให้บริการ TTS ที่ต้องการ
    - `tts.convert` เรียกใช้การแปลงข้อความเป็นเสียงพูดแบบครั้งเดียว
    - `tts.speak` (`operator.write`) เรนเดอร์ `text` ที่ไม่ว่างด้วยสายโซ่ผู้ให้บริการ TTS ทั่วไปที่กำหนดค่าไว้ และส่งคืนคลิปทั้งคลิปแบบอินไลน์เป็น `audioBase64` พร้อม `provider` และข้อมูลเมตา `outputFormat`, `mimeType` และ `fileExtension` ที่เป็นทางเลือก ต่างจาก `tts.convert` เมธอดนี้ไม่ส่งคืนพาธภายใน Gateway; ต่างจาก `talk.speak` เมธอดนี้ไม่ต้องใช้ผู้ให้บริการ Talk ข้อความที่เกิน `messages.tts.maxTextLength` จะส่งคืน `INVALID_REQUEST`; ความล้มเหลวในการสังเคราะห์จะส่งคืน `UNAVAILABLE`

  <Accordion title="ข้อมูลลับ การกำหนดค่า การอัปเดต และวิซาร์ด">
    - `secrets.reload` แก้ค่า SecretRefs ที่กำลังใช้งานอีกครั้งและเผยแพร่สถานะรันไทม์ที่ตระหนักถึงเจ้าของแบบอะตอมมิก ความล้มเหลวของเจ้าของที่มีสิทธิ์สามารถเผยแพร่เป็นการลดระดับแบบเย็นหรือล้าสมัยพร้อม `warningCount`; ความล้มเหลวแบบเข้มงวดหรือที่ไม่ได้แมปจะปฏิเสธการโหลดใหม่และรักษาสแนปช็อตที่กำลังใช้งานไว้
    - `secrets.resolve` แก้ค่าการกำหนดข้อมูลลับของเป้าหมายคำสั่งสำหรับชุดคำสั่ง/เป้าหมายที่ระบุ
    - `config.get` ส่งคืนสแนปช็อตการกำหนดค่าบนดิสก์ปัจจุบัน `hash` ของไฟล์รากแบบดิบ `configRevisionHash` ที่แก้ค่าแล้ว และ `appliedConfigHash` ที่เป็นทางเลือกสำหรับรีวิชันที่แก้ค่าแล้วซึ่งรันไทม์ Gateway ที่กำลังใช้งานยอมรับ
    - `config.set` เขียนเพย์โหลดการกำหนดค่าที่ผ่านการตรวจสอบ
    - `config.patch` ผสานการอัปเดตการกำหนดค่าบางส่วน การแทนที่อาร์เรย์แบบทำลายข้อมูลต้องระบุพาธที่ได้รับผลกระทบใน `replacePaths`; อาร์เรย์ที่ซ้อนอยู่ใต้รายการอาร์เรย์ใช้พาธ `[]` เช่น `agents.list[].skills`
    - `config.apply` ตรวจสอบและแทนที่เพย์โหลดการกำหนดค่าทั้งหมด
    - `config.schema` ส่งคืนเพย์โหลดสคีมาการกำหนดค่าแบบสดที่เครื่องมือ Control UI และ CLI ใช้ ซึ่งประกอบด้วยสคีมา `uiHints` เวอร์ชัน ข้อมูลเมตาการสร้าง และข้อมูลเมตาสคีมาของ Plugin กับช่องทางเมื่อโหลดได้ โดยรวมข้อมูลเมตา `title` / `description` จากข้อความป้ายกำกับ/วิธีใช้ชุดเดียวกับ UI รวมถึงออบเจ็กต์ซ้อน ไวลด์การ์ด รายการอาร์เรย์ และแขนงการประกอบ `anyOf` / `oneOf` / `allOf` เมื่อมีเอกสารประกอบฟิลด์ที่ตรงกัน
    - `config.schema.lookup` ส่งคืนเพย์โหลดการค้นหาที่จำกัดขอบเขตตามพาธสำหรับพาธการกำหนดค่าหนึ่งพาธ ซึ่งประกอบด้วยพาธที่ปรับให้เป็นมาตรฐาน โหนดสคีมาแบบตื้น คำแนะนำที่ตรงกันพร้อม `hintPath`, `reloadKind` ที่เป็นทางเลือก และสรุปลูกระดับถัดไปสำหรับการเจาะลึกผ่าน UI/CLI `reloadKind` เป็นหนึ่งใน `restart`, `hot` หรือ `none` (`src/config/schema.ts`) และสะท้อนตัววางแผนโหลดการกำหนดค่า Gateway ใหม่สำหรับพาธที่ขอ โหนดสคีมาการค้นหาจะคงเอกสารที่ผู้ใช้มองเห็นและฟิลด์การตรวจสอบทั่วไป (`title`, `description`, `type`, `enum`, `const`, `format`, `pattern`, ขอบเขตตัวเลข/สตริง/อาร์เรย์/ออบเจ็กต์, `additionalProperties`, `deprecated`, `readOnly`, `writeOnly`) สรุปลูกจะเปิดเผย `key`, `path` ที่ปรับให้เป็นมาตรฐาน, `type`, `required`, `hasChildren`, `reloadKind` ที่เป็นทางเลือก พร้อม `hint` / `hintPath` ที่ตรงกัน
    - `update.run` เรียกใช้ขั้นตอนการอัปเดต Gateway และกำหนดเวลารีสตาร์ตเฉพาะเมื่ออัปเดตสำเร็จเท่านั้น ผู้เรียกที่มีเซสชันสามารถรวม `continuationMessage` เพื่อให้การเริ่มทำงานดำเนินรอบเอเจนต์ติดตามผลหนึ่งรอบต่อผ่านคิวดำเนินการต่อหลังรีสตาร์ต การอัปเดตด้วยตัวจัดการแพ็กเกจและการอัปเดตเช็กเอาต์ git ที่มีการควบคุมจากระนาบควบคุมจะใช้การส่งมอบให้บริการที่มีการจัดการแบบแยกออก แทนการแทนที่โครงสร้างแพ็กเกจหรือแก้ไขเอาต์พุตเช็กเอาต์/บิลด์ภายใน Gateway ที่กำลังทำงาน การส่งมอบที่เริ่มแล้วจะส่งคืน `ok: true` พร้อม `result.reason: "managed-service-handoff-started"` และ `handoff.status: "started"` `update.run` รายการที่สองซึ่งทำงานพร้อมกันและจัดการโดยกระบวนการ Gateway เดียวกันจะส่งคืน `ok: false` พร้อม `result.reason: "managed-service-handoff-already-running"` และ `handoff.status: "already-running"`; ระบบจะไม่ยอมรับการดำเนินการต่อของรายการนั้น ดังนั้นผู้เรียกจึงลองใหม่ได้หลังการอัปเดตที่กำลังทำงานเสร็จสิ้น ตัวอัปเดต CLI แบบสแตนด์อโลนและกระบวนการ Gateway ที่ใช้แทนอยู่นอกการป้องกันภายในกระบวนการนี้ การส่งมอบที่ใช้ไม่ได้หรือล้มเหลวจะส่งคืน `ok: false` พร้อม `managed-service-handoff-unavailable` หรือ `managed-service-handoff-failed` รวมถึง `handoff.command` เมื่อจำเป็นต้องอัปเดตด้วยเชลล์ด้วยตนเอง ใช้ไม่ได้หมายถึง OpenClaw ไม่มีขอบเขตซูเปอร์ไวเซอร์ที่ปลอดภัยหรือข้อมูลประจำตัวบริการที่คงทน เช่น `OPENCLAW_SYSTEMD_UNIT` สำหรับ systemd ระหว่างการส่งมอบที่เริ่มแล้ว ตัวบ่งชี้รีสตาร์ตอาจรายงาน `stats.reason: "restart-health-pending"` ชั่วครู่; การดำเนินการต่อจะล่าช้าจนกว่า CLI จะตรวจสอบ Gateway ที่รีสตาร์ตแล้วและเขียนตัวบ่งชี้ `ok` ขั้นสุดท้าย
    - `update.status` รีเฟรชและส่งคืนตัวบ่งชี้รีสตาร์ตจากการอัปเดตล่าสุด รวมถึงเวอร์ชันที่กำลังทำงานหลังรีสตาร์ตเมื่อมี
    - `wizard.start`, `wizard.next`, `wizard.status` และ `wizard.cancel` เปิดให้ใช้งานวิซาร์ดการเริ่มต้นใช้งานผ่าน WS RPC

    - `agents.list` ส่งคืนรายการเอเจนต์ที่กำหนดค่าไว้ รวมถึงข้อมูลเมตาของโมเดลและรันไทม์ที่มีผลใช้งาน

    - `agents.files.list`, `agents.files.get` และ `agents.files.set` จัดการไฟล์เวิร์กสเปซสำหรับเริ่มต้นระบบที่เปิดให้เอเจนต์ใช้งาน

    - `agents.workspace.list` และ `agents.workspace.get` (`operator.read`) เปิดให้ไคลเอนต์ในโดเมนผู้ปฏิบัติงานที่เชื่อถือได้ซึ่งอธิบายไว้ใน [ขอบเขตผู้ปฏิบัติงาน](/th/gateway/operator-scopes) เรียกดูไดเรกทอรีเวิร์กสเปซของเอเจนต์แบบอ่านอย่างเดียวและแบ่งหน้า คำขอรับเฉพาะพาธที่สัมพันธ์กับเวิร์กสเปซ การอ่านจะถูกจำกัดให้อยู่ภายในรากเวิร์กสเปซที่ผ่านการหาพาธจริง (ปฏิเสธการหลบออกผ่าน symlink และ hardlink) มีการจำกัดขนาด และรองรับเฉพาะข้อความ UTF-8 กับประเภทรูปภาพทั่วไป (base64) การตอบกลับไม่เปิดเผยพาธเวิร์กสเปซบนโฮสต์ เนมสเปซนี้ไม่มีการดำเนินการเขียน
    - `tasks.list`, `tasks.get` และ `tasks.cancel` เปิดให้ไคลเอนต์ SDK และผู้ปฏิบัติงานเข้าถึงบัญชีงานของ Gateway ดู [RPC ของบัญชีงาน](#task-ledger-rpcs) ด้านล่าง
    - `artifacts.list`, `artifacts.get` และ `artifacts.download` เปิดให้เข้าถึงสรุปอาร์ติแฟกต์ที่ได้จากทรานสคริปต์และดาวน์โหลดสำหรับขอบเขต `sessionKey`, `runId` หรือ `taskId` ที่ระบุอย่างชัดเจน คำค้นหาการรันและงานจะระบุเซสชันเจ้าของที่ฝั่งเซิร์ฟเวอร์ และส่งคืนเฉพาะสื่อจากทรานสคริปต์ที่มีแหล่งที่มาตรงกัน แหล่ง URL ที่ไม่ปลอดภัยหรือเป็นโลคัลจะส่งคืนการดาวน์โหลดที่ไม่รองรับแทนการดึงข้อมูลที่ฝั่งเซิร์ฟเวอร์
    - `environments.list` และ `environments.status` รักษาการค้นหาสภาพแวดล้อมภายใน Gateway และ Node เวิร์กเกอร์คลาวด์ที่กำหนดค่าไว้และระเบียนถาวรที่โปรไฟล์ก่อนหน้าทิ้งไว้จะเพิ่มข้อมูลเมตา `worker` ซึ่งประกอบด้วย `providerId`, `leaseId` ที่ไม่บังคับ, `state`, `ageMs`, `idleMs` ที่ไม่บังคับ และ `attachedSessionIds` สถานะวงจรชีวิตของเวิร์กเกอร์ได้แก่ `requested`, `provisioning`, `bootstrapping`, `ready`, `attached`, `idle`, `draining`, `destroying`, `destroyed`, `failed` และ `orphaned`
    - `environments.create` (`{ profileId, idempotencyKey }`) จัดเตรียมเวิร์กเกอร์จากโปรไฟล์ผู้ให้บริการ Plugin ที่กำหนดค่าไว้ การลองใหม่ด้วยคีย์เดิมจะใช้การดำเนินการถาวรซ้ำ `environments.destroy` (`{ environmentId }`) ร้องขอการรื้อถอนสภาพแวดล้อมเวิร์กเกอร์ถาวรแบบทำซ้ำได้อย่างปลอดภัย ทั้งสองต้องใช้ `operator.admin` เป็นการเขียนในระนาบควบคุม และส่งคืนสรุปสภาพแวดล้อมในรูปแบบเดียวกับที่การตอบกลับสถานะใช้
    - `agent.identity.get` ส่งคืนตัวตนผู้ช่วยที่มีผลใช้งานสำหรับเอเจนต์หรือเซสชัน
    - `agent.wait` รอให้การรันเสร็จสิ้นและส่งคืนสแนปช็อตปลายทางเมื่อมี

    - `sessions.list` ส่งคืนดัชนีเซสชันปัจจุบัน รวมถึงข้อมูลเมตา `agentRuntime` ต่อแถวเมื่อมีการกำหนดค่าแบ็กเอนด์รันไทม์ของเอเจนต์ เมื่อเปิดใช้การจัดตำแหน่งบนเวิร์กเกอร์คลาวด์หรือมีสถานะการกู้คืนถาวร แถวเซสชันจะมีสถานะ `placement` แบบปิดเพิ่มเติม (`local`, `requested`, `provisioning`, `syncing`, `starting`, `active`, `draining`, `reconciling`, `reclaimed` หรือ `failed`) พร้อมฟิลด์สภาพแวดล้อม ยุคของเจ้าของ เวิร์กสเปซ บันเดิล เคอร์เซอร์ ACK หรือการกู้คืนที่เฉพาะเจาะจงตามสถานะ
    - `sessions.subscribe` และ `sessions.unsubscribe` เปิดหรือปิดการสมัครรับเหตุการณ์การเปลี่ยนแปลงเซสชันสำหรับไคลเอนต์ WS ปัจจุบัน
    - `sessions.messages.subscribe` และ `sessions.messages.unsubscribe` เปิดหรือปิดการสมัครรับเหตุการณ์ทรานสคริปต์/ข้อความสำหรับหนึ่งเซสชัน ส่ง `includeApprovals: true` เพื่อรับเหตุการณ์วงจรชีวิต `session.approval` ที่ผ่านการล้างข้อมูลแล้วเพิ่มเติม สำหรับการอนุมัติที่กลุ่มผู้รับซึ่งบันทึกไว้นั้นรวมเซสชันดังกล่าวโดยตรง และการผูกผู้ตรวจสอบอนุญาตไคลเอนต์ที่สมัครรับ การตอบกลับการสมัครรับจะมี `approvalReplay` ที่รอดำเนินการแบบมีขอบเขต ซึ่งถือเป็นข้อมูลอ้างอิงหลักเมื่อ `truncated` เป็น false การเลือกเข้าร่วมมีผลต่อการเรียกสมัครรับแต่ละครั้ง ไม่ได้คงอยู่ถาวร การสมัครรับเซสชันเดิมซ้ำโดยไม่มี `includeApprovals: true` จะลบการสมัครรับการอนุมัติที่มีอยู่ นอกจากสิทธิ์อ่านเซสชันตามปกติแล้ว การเลือกเข้าร่วมนี้ต้องใช้ `operator.admin` หรือ `operator.approvals` บนอุปกรณ์ที่จับคู่แล้ว
    - `sessions.preview` ส่งคืนตัวอย่างทรานสคริปต์แบบมีขอบเขตสำหรับคีย์เซสชันที่ระบุ
    - `sessions.describe` ส่งคืนแถวเซสชัน Gateway หนึ่งแถวสำหรับคีย์เซสชันที่ตรงกันทุกประการ
    - `sessions.resolve` ระบุหรือทำเป้าหมายเซสชันให้อยู่ในรูปแบบมาตรฐาน
    - `sessions.create` สร้างรายการเซสชันใหม่ ค่า `model` และ `thinkingLevel` ที่ไม่บังคับจะบันทึกการแทนที่โมเดลและการให้เหตุผลเริ่มต้นแบบอะตอมมิก `worktree: true` จัดเตรียม worktree ที่มีการจัดการ ค่า `worktreeBaseRef`/`worktreeName` ที่ไม่บังคับใช้เลือก ref ฐานและชื่อสาขา และ `execNode` (`operator.admin`) ผูกการดำเนินการของเซสชันเข้ากับโฮสต์ Node worktree ที่สร้างขึ้นจะถูกส่งกลับในผลลัพธ์และบันทึกไว้ในแถวเซสชัน (`worktree: { id, branch, repoRoot }`) เมื่อสร้างรายการสำเร็จแต่ `chat.send` เริ่มต้นที่ซ้อนอยู่ถูกปฏิเสธ ผลลัพธ์ที่สำเร็จจะมี `runStarted: false` และ `runError` ไคลเอนต์สามารถเก็บพรอมต์ไว้และลองใหม่โดยใช้คีย์เซสชันที่ส่งคืน ผู้เรียกที่ส่ง `parentSessionKey` พร้อม `emitCommandHooks: true` ควรประกาศการจัดการวงจรชีวิตของเซสชันลูกที่แยกต่างหากด้วย โดย `succeedsParent: true` จะสิ้นสุดเซสชันแม่ด้วย `session_end` ส่วน `false` จะคงเซสชันแม่ให้ทำงานต่อและปล่อยเฉพาะ `session_start` ของเซสชันลูก การละ `succeedsParent` จะคงพฤติกรรมการเปลี่ยนผ่านเซสชันแม่แบบเดิมสำหรับไคลเอนต์ที่มีอยู่ การจัดการนี้ต้องมีทั้งการเชื่อมโยงเซสชันแม่และฮุกคำสั่ง การ fork ไม่สามารถทำให้เซสชันแม่สำเร็จได้ พฤติกรรมรีเซ็ตเซสชันหลักในตำแหน่งเดิมไม่เปลี่ยนแปลง เพราะไม่มีการสร้างเซสชันลูกที่แยกต่างหาก
    - `sessions.dispatch` (`operator.admin`) ย้ายเซสชัน OpenClaw ในเครื่องที่มี worktree แบบมีการจัดการซึ่งเซสชันเป็นเจ้าของ ไปยังโปรไฟล์เวิร์กเกอร์คลาวด์ที่กำหนดค่าไว้ ส่ง `{ key, profileId, agentId? }` เมธอดนี้จะไม่มีอยู่เมื่อไม่ได้กำหนดค่าโปรไฟล์เวิร์กเกอร์ จะปิดการรับเทิร์นในเครื่องก่อนรอให้งานที่กำลังทำอยู่เสร็จ และจะส่งคืนเมื่อการจัดตำแหน่งถึงการเป็นเจ้าของโดยเวิร์กเกอร์ `active` เท่านั้น การส่งต่อเป็นทางเดียว RPC นี้ไม่รวมการดึงกลับจากเวิร์กเกอร์มายังเครื่อง
    - `sessions.groups.list`, `sessions.groups.put`, `sessions.groups.rename` และ `sessions.groups.delete` จัดการแค็ตตาล็อกกลุ่มเซสชันแบบกำหนดเองที่ Gateway เป็นเจ้าของ (ชื่อ + ลำดับการแสดงผล) การเป็นสมาชิกยังคงอยู่ในฟิลด์ `category` ของแต่ละเซสชัน การเปลี่ยนชื่อและการลบจะอัปเดตเซสชันสมาชิกที่ฝั่งเซิร์ฟเวอร์
    - `sessions.send` ส่งข้อความเข้าสู่เซสชันที่มีอยู่
    - `sessions.steer` เป็นรูปแบบขัดจังหวะและชี้นำสำหรับเซสชันที่กำลังทำงาน
    - `sessions.abort` ยกเลิกงานที่กำลังทำอยู่สำหรับเซสชัน ส่ง `key` พร้อม `runId` ที่ไม่บังคับ หรือส่ง `runId` เพียงอย่างเดียวสำหรับการรันที่กำลังทำงานซึ่ง Gateway สามารถระบุเซสชันได้
    - `sessions.patch` อัปเดตข้อมูลเมตา/การแทนที่ของเซสชัน และรายงานโมเดลมาตรฐานที่ระบุได้พร้อม `agentRuntime` ที่มีผลใช้งาน
    - `sessions.reset`, `sessions.delete` และ `sessions.compact` ดำเนินการบำรุงรักษาเซสชัน
    - `sessions.get` ส่งคืนแถวเซสชันที่จัดเก็บไว้ทั้งหมด
    - การดำเนินการแชตยังคงใช้ `chat.history`, `chat.send`, `chat.abort` และ `chat.inject` โดย `chat.history` จะถูกปรับรูปแบบการแสดงผลให้เป็นมาตรฐานสำหรับไคลเอนต์ UI: แท็กคำสั่งแบบอินไลน์จะถูกตัดออกจากข้อความที่มองเห็นได้, เพย์โหลด XML ของการเรียกเครื่องมือแบบข้อความล้วน (`<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อกการเรียกเครื่องมือที่ถูกตัดทอน) รวมถึงโทเค็นควบคุมโมเดลแบบ ASCII/เต็มความกว้างที่รั่วออกมาจะถูกตัดออก, แถวผู้ช่วยที่มีเฉพาะโทเค็นเงียบ (ตรงกับ `NO_REPLY` / `no_reply` ทุกประการ) จะถูกละเว้น และแถวที่มีขนาดใหญ่เกินไปอาจถูกแทนที่ด้วยตัวยึดตำแหน่ง
    - `chat.message.get` เป็นตัวอ่านข้อความฉบับเต็มแบบมีขอบเขตที่เพิ่มเข้ามาสำหรับรายการทรานสคริปต์ที่มองเห็นได้หนึ่งรายการ ส่ง `sessionKey`, `agentId` ที่ไม่บังคับเมื่อการเลือกเซสชันจำกัดขอบเขตตามเอเจนต์ และ `messageId` ของทรานสคริปต์ที่เคยเปิดเผยผ่าน `chat.history` โดย Gateway จะส่งคืนการฉายภาพที่ปรับรูปแบบการแสดงผลเป็นมาตรฐานแบบเดียวกัน โดยไม่มีขีดจำกัดการตัดทอนประวัติแบบน้ำหนักเบา เมื่อรายการที่จัดเก็บไว้ยังคงมีอยู่และมีขนาดไม่ใหญ่เกินไป
    - `chat.toolTitles` ส่งคืนชื่อวัตถุประสงค์แบบสั้นสำหรับการเรียกเครื่องมือที่แสดงใน Control UI (ประมวลผลเป็นชุด สูงสุด 24 รายการพร้อมอินพุตแบบมีขอบเขต) ฟีเจอร์นี้ต้องเลือกเปิดใช้ผ่าน `gateway.controlUi.toolTitles` (ค่าเริ่มต้นปิด) Gateway ที่ปิดใช้จะตอบ `{ titles: {}, disabled: true }` โดยไม่เรียกโมเดลเพื่อให้ไคลเอนต์หยุดร้องขอ เมื่อเปิดใช้ ชื่อจะใช้การกำหนดเส้นทางโมเดลยูทิลิตีมาตรฐาน ได้แก่ `utilityModel` ที่กำหนดค่าไว้อย่างชัดเจน (การตัดสินใจของผู้ปฏิบัติงานซึ่งอาจส่งเนื้อหางานแบบมีขอบเขตไปยังผู้ให้บริการที่เลือก เช่นเดียวกับงานยูทิลิตีทั้งหมด) หรือค่าเริ่มต้นของโมเดลขนาดเล็กที่ผู้ให้บริการของเซสชันประกาศไว้ เพื่อไม่ให้เกิดปลายทางขาออกใหม่โดยปริยาย ส่วน `utilityModel` ที่ว่างจะปิดใช้ทั้งหมด ชื่อจะไม่ย้อนกลับไปใช้โมเดลหลัก ผลลัพธ์จะถูกแคชในฐานข้อมูลสถานะต่อเอเจนต์โดยใช้ชื่อเครื่องมือ + อินพุตเป็นคีย์ ดังนั้นการดูซ้ำจะไม่เรียกเก็บเงินซ้ำสำหรับการเรียกเดิม
    - `chat.send` รับ `fastMode: "auto"` แบบหนึ่งเทิร์นเพื่อใช้โหมดเร็วกับการเรียกโมเดลที่เริ่มก่อนจุดตัดอัตโนมัติ จากนั้นเริ่มการลองใหม่ การย้อนกลับ ผลลัพธ์เครื่องมือ หรือการเรียกต่อเนื่องในภายหลังโดยไม่ใช้โหมดเร็ว จุดตัดมีค่าเริ่มต้นเป็น 60 วินาที (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) และสามารถกำหนดค่าต่อโมเดลด้วย `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` ผู้เรียก `chat.send` สามารถส่ง `fastAutoOnSeconds` แบบหนึ่งเทิร์นเพื่อแทนที่จุดตัดสำหรับคำขอนั้น ส่ง `queueMode` (`steer`, `followup`, `collect` หรือ `interrupt`) เพื่อแทนที่โหมดคิวที่จัดเก็บไว้เฉพาะสำหรับคำขอนี้ การชี้นำอย่างชัดเจนใน Control UI ใช้ `queueMode: "steer"`

  </Accordion>

    - `node.pending.pull` และ `node.pending.ack` คือ API คิวของ Node ที่เชื่อมต่ออยู่
    - `node.pending.enqueue` และ `node.pending.drain` จัดการงานที่รอดำเนินการแบบคงทนสำหรับ Node ที่ออฟไลน์/ยกเลิกการเชื่อมต่อ

  </Accordion>

  <Accordion title="กลุ่มการอนุมัติ">

    - `exec.approval.waitDecision` รอการอนุมัติ exec ที่รอดำเนินการหนึ่งรายการและส่งคืนคำตัดสินสุดท้าย (หรือ `null` เมื่อหมดเวลา)

    - `exec.approvals.node.get` และ `exec.approvals.node.set` จัดการนโยบายการอนุมัติ exec ภายใน Node ผ่านคำสั่งรีเลย์ของ Node
    - `plugin.approval.request`, `plugin.approval.list`, `plugin.approval.waitDecision` และ `plugin.approval.resolve` ครอบคลุมขั้นตอนการอนุมัติที่ Plugin กำหนด

  ในโปรโตคอล v4 เพย์โหลดส่วนต่างจะบรรจุ `deltaText` ส่วน `message` ยังคงเป็น
  สแนปช็อตสะสมของผู้ช่วย การแทนที่ที่ไม่ใช่คำนำหน้าจะตั้งค่า

## RPC บัญชีแยกประเภทการตรวจสอบ

`audit.activity.list` มอบมุมมองแบบคงที่ซึ่งเรียงใหม่สุดก่อนแก่ไคลเอนต์ผู้ปฏิบัติงาน สำหรับเมตาดาต้าวงจร
การเรียกใช้เอเจนต์ การดำเนินการของเครื่องมือ และข้อความแบบเลือกรับ โดยต้องใช้
`operator.read` คิวรีจะไม่รวมระเบียนที่เก่ากว่า 30 วัน และบัญชีแยกประเภท SQLite ที่ใช้ร่วมกัน
จำกัดไว้ที่ 100,000 ระเบียน แถวที่หมดอายุจะถูกลบระหว่าง
การเริ่มต้น Gateway การบำรุงรักษารายชั่วโมง และการเขียนในภายหลัง ดู
[ประวัติการตรวจสอบ](/th/gateway/audit)สำหรับโมเดลข้อมูลและความหมายด้านความเป็นส่วนตัว

- พารามิเตอร์: `agentId`, `sessionKey` หรือ `runId` แบบตรงกันทุกประการซึ่งเป็นทางเลือก; `kind`
  (`"agent_run"`, `"tool_action"` หรือ `"message"`) ซึ่งเป็นทางเลือก; `status`
  (`"started"`, `"succeeded"`, `"failed"`, `"cancelled"`, `"timed_out"`,
  `"blocked"` หรือ `"unknown"`) ซึ่งเป็นทางเลือก; ข้อความ `direction` (`"inbound"` หรือ
  `"outbound"`) และ `channel` แบบตรงกันทุกประการซึ่งเป็นทางเลือก; ขอบเขตมิลลิวินาที Unix แบบรวมค่า `after` / `before`
  ซึ่งเป็นทางเลือก; `limit` ตั้งแต่ `1` ถึง `500` ซึ่งเป็นทางเลือก; และสตริง
  `cursor` จากหน้าก่อนหน้าซึ่งเป็นทางเลือก
- ผลลัพธ์: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`

ยูเนียนผลลัพธ์ V1 ที่มีชื่อมีสคีมาแยกกันสำหรับการเรียกใช้เอเจนต์ การดำเนินการของเครื่องมือ ข้อความขาเข้า
และข้อความขาออก ตัวจำแนก `eventType` คือ
`agent_run`, `tool_action`, `inbound_message` หรือ `outbound_message` ตามลำดับ ส่วน `kind` และ
ข้อความ `direction` ยังคงใช้สำหรับการกรองและแสดงผลได้ ทุกเหตุการณ์มี
`schemaVersion: 1` เป็นจำนวนเต็ม การอ้างอิงตัวตนของข้อความใช้รูปแบบ
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` ที่ตรงกันทุกประการ ส่วน ID ผู้ดำเนินการที่เป็นผู้ส่งของช่องทาง
ใช้รูปแบบเดียวกัน

ตัวแปรย่อยทั้งหมดกำหนดให้มี `eventType`, `schemaVersion`, `eventId`, `sequence`,
`sourceSequence`, `occurredAt`, `kind`, `action`, `status`, `actor` และ
`redaction` ฟิลด์ของตัวแปรย่อยมีดังนี้:

| `eventType`        | ฟิลด์ที่จำเป็น                                                   | ฟิลด์ที่เป็นทางเลือก                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`, `runId`; `kind: "agent_run"`                           | `sessionKey`, `sessionId`, `errorCode`                                                                                          |
| `tool_action`      | `agentId`, `runId`; `kind: "tool_action"`                         | `sessionKey`, `sessionId`, `toolCallId`, `toolName`, `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`, `channel`, `conversationKind`, `outcome`  | `agentId`, `runId`, `durationMs`, `resultCount`, การอ้างอิงตัวตน, `reasonCode`, `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`, `channel`, `conversationKind`, `outcome` | `agentId`, `runId`, `durationMs`, `resultCount`, การอ้างอิงตัวตน, `reasonCode`, `deliveryKind`, `failureStage`, `errorCode` |

เอนัมข้อความแบบปิดมีดังนี้:

- `conversationKind`: `direct`, `group`, `channel` หรือ `unknown`
- `outcome` ขาเข้า: `completed`, `skipped` หรือ `failed`; `reasonCode` ซึ่งเป็นทางเลือก:
  `duplicate`, `reply_operation_active`,
  `reply_operation_aborted`, `fast_abort`, `plugin_bound_handled`,
  `plugin_bound_unavailable`, `plugin_bound_declined`, `plugin_bound_error`,
  `before_dispatch_handled`, `acp_dispatch_completed`, `acp_dispatch_failed`,
  `acp_dispatch_empty` หรือ `acp_dispatch_aborted`
- `outcome` ขาออก: `sent`, `suppressed`, `failed` หรือ `unknown`; `reasonCode` ซึ่งเป็นทางเลือก:
  `cancelled_by_message_sending_hook`,
  `cancelled_by_reply_payload_sending_hook`,
  `empty_after_message_sending_hook`, `empty_after_reply_payload_sending_hook`
  หรือ `no_visible_payload` อะแดปเตอร์ที่ไม่ส่งคืนตัวตนของแพลตฟอร์มคือ
  `unknown` เนื่องจากไม่สามารถพิสูจน์ได้ว่าไม่มีผลข้างเคียงภายนอกเกิดขึ้น
- `deliveryKind`: `text`, `media` หรือ `other`; `failureStage`:
  `platform_send`, `queue` หรือ `unknown`

ฟิลด์ขั้นสุดท้ายมีความสัมพันธ์กัน ไม่ได้เป็นทางเลือกโดยอิสระ:

| รูปแบบ           | การแมปสถานะสิ้นสุด                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| การรันเอเจนต์        | `started` ไม่มี `errorCode`; ทุกสถานะสิ้นสุดที่ไม่สำเร็จต้องมีรหัส `run_*` ที่ตรงกัน                                                                 |
| การดำเนินการของเครื่องมือ      | `started` และสถานะสำเร็จไม่มี `errorCode`; สถานะสิ้นสุดอื่นแต่ละสถานะต้องมีรหัส `tool_*` ที่ตรงกัน                                                       |
| ข้อความขาเข้า  | สำเร็จ = `completed`; ถูกบล็อก = `skipped`; ล้มเหลว = `failed` พร้อม `message_processing_failed` เมื่อมี `reasonCode` ค่านั้นต้องอยู่ในตระกูลสถานะสิ้นสุดดังกล่าว |
| ข้อความขาออก | สำเร็จ = `sent`; ถูกบล็อก = `suppressed` พร้อม `reasonCode`; ล้มเหลว = `failed` พร้อม `errorCode` และ `failureStage`; ไม่ทราบ = `unknown` พร้อม `failureStage`      |

แต่ละเหตุการณ์กิจกรรมประกอบด้วยรหัสเหตุการณ์ที่คงที่ ลำดับบัญชีแยกประเภทที่เพิ่มขึ้นทางเดียว
ลำดับเหตุการณ์ต้นทาง การประทับเวลา ผู้กระทำ การดำเนินการ สถานะ ค่า整数
`schemaVersion: 1` และ `redaction: "metadata_only"` ระเบียนการรันและเครื่องมือ
ต้องมีแหล่งที่มาของเอเจนต์และการรัน และอาจมีแหล่งที่มาของเซสชัน ระเบียนข้อความ
อาจมีรหัสเอเจนต์และการรัน แต่ตั้งใจไม่ให้มี
`sessionKey` หรือ `sessionId`; ดังนั้นตัวกรองการค้นหา `sessionKey` จึงใช้กับ
แถวการรันและเครื่องมือเท่านั้น เหตุการณ์เครื่องมืออาจมีรหัสการเรียกเครื่องมือและชื่อเครื่องมือ

ระเบียนข้อความใช้ `message.inbound.processed` หรือ
`message.outbound.finished` และเพิ่มทิศทาง ช่องทาง ประเภทการสนทนา
ผลลัพธ์ที่ปรับเป็นมาตรฐาน และข้อมูลเสริม ได้แก่ ประเภทการส่ง ขั้นตอนที่ล้มเหลว ระยะเวลา
จำนวนผลลัพธ์ รหัสเหตุผล และนามแฝงแบบใช้คีย์เฉพาะการติดตั้งสำหรับ
บัญชี/การสนทนา/ข้อความ/เป้าหมาย นามแฝงเหล่านี้ช่วยในการ
เชื่อมโยง แต่ไม่ใช่การทำให้ไม่ระบุตัวตน: ฐานข้อมูลสถานะมีคีย์ของนามแฝง
แต่การส่งออกผ่าน RPC และ CLI ไม่มี บัญชีแยกประเภทไม่จัดเก็บพรอมต์ เนื้อหา
ข้อความ อาร์กิวเมนต์เครื่องมือ ผลลัพธ์เครื่องมือ เอาต์พุตคำสั่ง หรือข้อความข้อผิดพลาดดิบ
ค่า `sessionKey` ของการรัน/เครื่องมือยังคงเป็นข้อมูลเมตาการเชื่อมโยงแบบดิบและอาจฝัง
รหัสบัญชีแพลตฟอร์มหรือรหัสคู่สื่อสารไว้ ระเบียนข้อความจะไม่รวมคีย์เซสชัน

สำหรับแถวขาเข้า `durationMs` วัดระยะตั้งแต่การส่งต่อของแกนกลางจนถึงสถานะสิ้นสุด และ
`resultCount` นับเพย์โหลดเครื่องมือ การบล็อก และการตอบกลับที่เข้าคิวและเสร็จสมบูรณ์ สำหรับ
แถวขาออก `durationMs` ครอบคลุมตั้งแต่การรับผิดชอบการส่งจนถึงการยืนยันรับ
จดหมายตาย หรือการกระทบยอด (รวมเวลารอในคิว) และ `resultCount`
นับการส่งทางกายภาพบนแพลตฟอร์มที่ระบุได้ เมื่อมี `deliveryKind`
ค่านี้จะอธิบายเพย์โหลดที่มีผลหลังผ่านฮุกและการเรนเดอร์ แถวที่ถูกระงับหรือ
มีความกำกวมจากการขัดข้องจะไม่มีค่านี้

ขอบเขตข้อความปัจจุบันครอบคลุมข้อความขาเข้าที่ได้รับการยอมรับและไปถึงการส่งต่อของแกนกลาง
รวมถึงผลลัพธ์รายการซ้ำ/สถานะสิ้นสุดของแกนกลาง ขอบเขตขาออกเขียน
หนึ่งแถวสถานะสิ้นสุดต่อเพย์โหลดการตอบกลับเชิงตรรกะต้นฉบับที่ไปถึงการส่งแบบคงทนร่วม
โดยรวมการแบ่งส่วนและการกระจายออกของอะแดปเตอร์ไว้ใน `resultCount` การส่งที่เข้าคิว
และสามารถลองใหม่ได้หรือมีความกำกวมจะถูกบันทึกหลังการยืนยันรับ จดหมายตาย
หรือการกระทบยอดเท่านั้น เส้นทางภายใน Plugin และเส้นทางส่งโดยตรงที่ข้าม
ขอบเขตร่วมเหล่านั้นยังไม่ครอบคลุม คิวเวิร์กเกอร์แบบมีขอบเขตทำงานตามความพยายามสูงสุด
และอาจทำระเบียนสูญหายเมื่อเกิดความล้มเหลวหรืออิ่มตัว ดังนั้นส่วนนี้จึงไม่ใช่
คลังข้อมูลการปฏิบัติตามข้อกำหนดที่รับประกันว่าไม่มีข้อมูลสูญหาย

การบันทึกเปิดอยู่ตามค่าเริ่มต้นและควบคุมด้วย
[`audit.enabled`](/th/gateway/configuration-reference#audit) การบันทึกข้อความ
ควบคุมแยกต่างหากด้วย `audit.messages` และมีค่าเริ่มต้นเป็น `"off"` เมื่อ
ปิดใช้งานการบันทึก `audit.activity.list` จะยังให้บริการระเบียนที่เขียนไว้
ก่อนหน้านี้จนกว่าจะหมดอายุ

สคีมาคำขอ ผลลัพธ์ และ `AuditEvent` ของ `audit.list` ที่เผยแพร่แล้วยังคง
ไม่เปลี่ยนแปลงและส่งคืนเฉพาะระเบียนการรันเอเจนต์และการดำเนินการของเครื่องมือ ไคลเอนต์
ผู้ปฏิบัติงานรุ่นใหม่ควรเรียก `audit.activity.list` เมื่อ Gateway ประกาศว่ารองรับ Gateway
รุ่นเก่าอาจรายงาน `unknown method: audit.activity.list` หรือเนื่องจาก
การให้สิทธิ์เกิดก่อนการค้นหาเมธอดในเวอร์ชันที่เผยแพร่แล้ว จึงอาจรายงาน `missing scope:
operator.admin` ต่อคำขอที่มีขอบเขตการอ่าน ให้ถือว่ากรณีหลังหมายถึงไม่มีเมธอด
เฉพาะเมื่อไม่ได้ประกาศเมธอดนั้น จากนั้นไคลเอนต์อาจลอง `audit.list` อีกครั้ง
ได้เฉพาะเมื่อตัวกรองไม่ต้องรองรับประเภทข้อความ ทิศทาง หรือช่องทาง

ใช้ [`openclaw audit`](/th/cli/audit) สำหรับการค้นหาแบบข้อความและการส่งออก JSON แบบมีขอบเขต

## RPC ของบัญชีแยกประเภทงาน

ไคลเอนต์ผู้ปฏิบัติงานตรวจสอบและยกเลิกระเบียนงานเบื้องหลังของ Gateway ผ่าน
RPC ของบัญชีแยกประเภทงาน (`packages/gateway-protocol/src/schema/tasks.ts`) RPC เหล่านี้
ส่งคืนสรุปงานที่ผ่านการกรองข้อมูลละเอียดอ่อน ไม่ใช่สถานะรันไทม์ดิบ

- `tasks.list` ต้องใช้ `operator.read`
  - พารามิเตอร์: `status` ซึ่งระบุหรือไม่ก็ได้ (`"queued"`, `"running"`, `"completed"`,
    `"failed"`, `"cancelled"` หรือ `"timed_out"`) หรืออาร์เรย์ของสถานะเหล่านั้น
    `agentId` ซึ่งระบุหรือไม่ก็ได้, `sessionKey` ซึ่งระบุหรือไม่ก็ได้, `limit` ซึ่งระบุหรือไม่ก็ได้ตั้งแต่ `1` ถึง
    `500` และสตริง `cursor` ซึ่งระบุหรือไม่ก็ได้
  - ผลลัพธ์: `{ "tasks": TaskSummary[], "nextCursor"?: string }`
- `tasks.get` ต้องใช้ `operator.read`
  - พารามิเตอร์: `{ "taskId": string }`
  - ผลลัพธ์: `{ "task": TaskSummary }`
  - รหัสงานที่ไม่มีอยู่จะส่งคืนข้อผิดพลาดรูปแบบไม่พบของ Gateway
- `tasks.cancel` ต้องใช้ `operator.write`
  - พารามิเตอร์: `{ "taskId": string, "reason"?: string }`
  - ผลลัพธ์: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`
  - `found` ระบุว่าบัญชีแยกประเภทมีงานที่ตรงกันหรือไม่ `cancelled`
    ระบุว่ารันไทม์ยอมรับหรือบันทึกการยกเลิกหรือไม่

`TaskSummary` ประกอบด้วย `id`, `status` และข้อมูลเมตาที่ระบุหรือไม่ก็ได้: `kind`,
`runtime`, `title`, `agentId`, `sessionKey`, `childSessionKey`, `ownerKey`,
`runId`, `taskId`, `flowId`, `parentTaskId`, `sourceId`, การประทับเวลา ความคืบหน้า
สรุปสถานะสิ้นสุด และข้อความข้อผิดพลาดที่ผ่านการกรองข้อมูลละเอียดอ่อน `agentId` ระบุเอเจนต์
ที่ดำเนินงาน ส่วน `sessionKey` และ `ownerKey` จะคงบริบทของผู้ร้องขอและการควบคุมไว้

## เมธอดตัวช่วยสำหรับผู้ปฏิบัติงาน

- `commands.list` (`operator.read`) ดึงรายการคำสั่งรันไทม์สำหรับ
  เอเจนต์
  - `agentId` เป็นตัวเลือกเสริม หากละไว้ ระบบจะอ่านพื้นที่ทำงานเริ่มต้นของเอเจนต์
  - `scope` ควบคุมว่า `name` หลักจะกำหนดเป้าหมายพื้นผิวใด: `text` ส่งคืน
    โทเค็นคำสั่งข้อความหลักโดยไม่มี `/` นำหน้า ส่วน `native` และเส้นทาง
    `both` เริ่มต้นจะส่งคืนชื่อเนทีฟที่คำนึงถึงผู้ให้บริการเมื่อมี
  - `textAliases` มีนามแฝงแบบสแลชที่ตรงตัว เช่น `/model` และ `/m`
  - `nativeName` มีชื่อคำสั่งเนทีฟที่คำนึงถึงผู้ให้บริการเมื่อมี
  - `provider` เป็นตัวเลือกเสริมและมีผลเฉพาะต่อการตั้งชื่อแบบเนทีฟ รวมถึงความพร้อมใช้งานของคำสั่ง Plugin
    แบบเนทีฟ
  - `includeArgs=false` จะตัดข้อมูลเมตาของอาร์กิวเมนต์ที่ซีเรียลไลซ์แล้วออกจากการตอบกลับ
- `tools.catalog` (`operator.read`) ดึงแค็ตตาล็อกเครื่องมือรันไทม์สำหรับ
  เอเจนต์ การตอบกลับประกอบด้วยเครื่องมือที่จัดกลุ่มและข้อมูลเมตาแหล่งที่มา:
  - `source`: `core` หรือ `plugin`
  - `pluginId`: เจ้าของ Plugin เมื่อเป็น `source="plugin"`
  - `optional`: เครื่องมือ Plugin เป็นตัวเลือกเสริมหรือไม่
- `tools.effective` (`operator.read`) ดึงรายการเครื่องมือที่มีผลจริงในรันไทม์
  สำหรับเซสชัน
  - `sessionKey` จำเป็นต้องระบุ
  - Gateway จะอนุมานบริบทรันไทม์ที่เชื่อถือได้จากเซสชันฝั่งเซิร์ฟเวอร์
    แทนการยอมรับบริบทการยืนยันตัวตนหรือการส่งมอบที่ผู้เรียกส่งมา
  - การตอบกลับเป็นภาพฉายขอบเขตเซสชันที่เซิร์ฟเวอร์อนุมานจากรายการที่ใช้งานอยู่
    ซึ่งรวมเครื่องมือหลัก เครื่องมือ Plugin เครื่องมือช่องทาง และเครื่องมือเซิร์ฟเวอร์ MCP
    ที่ค้นพบแล้ว
  - `tools.effective` เป็นแบบอ่านอย่างเดียวสำหรับ MCP: อาจฉายแค็ตตาล็อก MCP
    ของเซสชันที่อุ่นอยู่ผ่านนโยบายเครื่องมือขั้นสุดท้าย แต่จะไม่สร้างรันไทม์ MCP
    เชื่อมต่อทรานสปอร์ต หรือออก `tools/list` หากไม่มีแค็ตตาล็อกที่อุ่นอยู่และตรงกัน
    การตอบกลับอาจมีประกาศ เช่น `mcp-not-yet-connected`,
    `mcp-not-yet-listed` หรือ `mcp-stale-catalog`
  - รายการเครื่องมือที่มีผลจริงใช้ `source="core"`, `source="plugin"`,
    `source="channel"` หรือ `source="mcp"`
- `tools.invoke` (`operator.write`) เรียกใช้เครื่องมือที่พร้อมใช้งานหนึ่งรายการผ่าน
  เส้นทางนโยบาย Gateway เดียวกับ `/tools/invoke`
  - `name` จำเป็นต้องระบุ ส่วน `args`, `sessionKey`, `agentId`, `confirm` และ
    `idempotencyKey` เป็นตัวเลือกเสริม
  - หากมีทั้ง `sessionKey` และ `agentId` เอเจนต์ของเซสชันที่แก้ไขแล้ว
    ต้องตรงกับ `agentId`
  - ตัวห่อหุ้มแกนหลักเฉพาะเจ้าของ เช่น `cron`, `gateway` และ `nodes` ต้องใช้
    ตัวตนเจ้าของ/ผู้ดูแลระบบ (`operator.admin`) แม้ว่า `tools.invoke` เอง
    จะเป็น `operator.write`
  - การตอบกลับเป็นเอนเวโลปสำหรับ SDK ซึ่งมี `ok`, `toolName`,
    `output` ซึ่งเป็นตัวเลือกเสริม และฟิลด์ `error` แบบมีชนิด การปฏิเสธเนื่องจากการอนุมัติหรือนโยบายจะส่งคืน
    `ok:false` ในเพย์โหลด แทนการข้ามไปป์ไลน์นโยบายเครื่องมือของ Gateway
- `skills.status` (`operator.read`) ดึงรายการ Skills ที่มองเห็นได้สำหรับ
  เอเจนต์
  - `agentId` เป็นตัวเลือกเสริม หากละไว้ ระบบจะอ่านพื้นที่ทำงานเริ่มต้นของเอเจนต์
  - การตอบกลับประกอบด้วยสิทธิ์ใช้งาน ข้อกำหนดที่ขาดหาย การตรวจสอบการกำหนดค่า
    และตัวเลือกการติดตั้งที่ผ่านการล้างข้อมูลแล้ว โดยไม่เปิดเผยค่าความลับดิบ
- `skills.search` และ `skills.detail` (`operator.read`) ส่งคืนข้อมูลเมตา
  การค้นพบของ ClawHub
- `skills.upload.begin`, `skills.upload.chunk` และ `skills.upload.commit`
  (`operator.admin`) จัดเตรียมไฟล์เก็บถาวร Skills ส่วนตัวก่อนติดตั้ง นี่คือ
  เส้นทางอัปโหลดของผู้ดูแลระบบแยกต่างหากสำหรับไคลเอนต์ที่เชื่อถือได้ ไม่ใช่ขั้นตอนการติดตั้ง
  Skills ปกติของ ClawHub และจะปิดใช้งานโดยค่าเริ่มต้น เว้นแต่เปิดใช้
  `skills.install.allowUploadedArchives`
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })`
    สร้างการอัปโหลดที่ผูกกับ slug และค่า force นั้น
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })` ต่อท้ายไบต์ที่
    ออฟเซ็ตที่ถอดรหัสแล้วอย่างตรงตัว
  - `skills.upload.commit({ uploadId, sha256? })` ตรวจสอบขนาดสุดท้ายและ
    SHA-256 การคอมมิตเพียงทำให้การอัปโหลดเสร็จสิ้นเท่านั้น โดยไม่ได้ติดตั้ง Skills
  - ไฟล์เก็บถาวร Skills ที่อัปโหลดเป็นไฟล์ zip ซึ่งมีราก `SKILL.md` ชื่อไดเรกทอรี
    ภายในไฟล์เก็บถาวรจะไม่เป็นตัวกำหนดเป้าหมายการติดตั้ง
- `skills.install` (`operator.admin`) มีสามโหมด:
  - โหมด ClawHub: `{ source: "clawhub", slug, version?, force? }` ติดตั้ง
    โฟลเดอร์ Skills ลงในไดเรกทอรี `skills/` ของพื้นที่ทำงานเริ่มต้นของเอเจนต์
  - โหมดอัปโหลด: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }`
    ติดตั้งการอัปโหลดที่คอมมิตแล้วลงในไดเรกทอรี `skills/<slug>`
    ของพื้นที่ทำงานเริ่มต้นของเอเจนต์ slug และค่า force ต้องตรงกับคำขอ
    `skills.upload.begin` เดิม ระบบจะปฏิเสธเว้นแต่เปิดใช้
    `skills.install.allowUploadedArchives` การตั้งค่านี้ไม่มีผลต่อการติดตั้งผ่าน ClawHub
  - โหมดตัวติดตั้ง Gateway: `{ name, installId, timeoutMs? }` เรียกใช้การดำเนินการ
    `metadata.openclaw.install` ที่ประกาศไว้บนโฮสต์ Gateway ไคลเอนต์รุ่นเก่าอาจยังคง
    ส่ง `dangerouslyForceUnsafeInstall` ฟิลด์นี้เลิกใช้แล้ว
    ยอมรับไว้เพื่อความเข้ากันได้ของโปรโตคอลเท่านั้น และจะถูกละเว้น ใช้
    `security.installPolicy` สำหรับการตัดสินใจติดตั้งที่ผู้ดำเนินการเป็นเจ้าของ
- `skills.update` (`operator.admin`) มีสองโหมด:
  - โหมด ClawHub อัปเดต slug ที่ติดตามหนึ่งรายการ หรือการติดตั้ง ClawHub ที่ติดตามทั้งหมดใน
    พื้นที่ทำงานเริ่มต้นของเอเจนต์
  - โหมดการกำหนดค่าจะแพตช์ค่า `skills.entries.<skillKey>` เช่น `enabled`,
    `apiKey` และ `env`

### มุมมอง `models.list`

`models.list` ยอมรับพารามิเตอร์ `view` ซึ่งเป็นตัวเลือกเสริม
(`src/agents/model-catalog-visibility.ts`):

- ละไว้หรือ `"default"`: หากกำหนดค่า `agents.defaults.modelPolicy.allow` แล้ว
  การตอบกลับจะเป็นแค็ตตาล็อกที่อนุญาต รวมถึงโมเดลที่ค้นพบแบบไดนามิก
  สำหรับรายการ `provider/*` มิฉะนั้นการตอบกลับจะเป็นแค็ตตาล็อก Gateway
  แบบเต็ม
- `"configured"`: ลักษณะการทำงานที่ปรับขนาดสำหรับตัวเลือก หากกำหนดค่า `agents.defaults.modelPolicy.allow`
  แล้ว ค่านี้ยังคงมีผลเหนือกว่า รวมถึงการค้นพบในขอบเขตผู้ให้บริการสำหรับ
  รายการ `provider/*` หากไม่มีรายการอนุญาต การตอบกลับจะใช้รายการ
  `models.providers.<provider>.models` ที่ระบุไว้อย่างชัดเจน และจะย้อนกลับไปใช้แค็ตตาล็อก
  แบบเต็มเฉพาะเมื่อไม่มีแถวโมเดลที่กำหนดค่าไว้
- `"provider-config"`: รายการ `models.providers.*.models` ที่กำหนดโดยแหล่งที่มา
  ซึ่งไม่ขึ้นกับรายการอนุญาตของตัวเลือก แถวต่าง ๆ ประกอบด้วยความสามารถสาธารณะของโมเดลและ
  ความพร้อมใช้งานที่คำนึงถึงเส้นทาง แต่ไม่รวมเอ็นด์พอยต์ของผู้ให้บริการ ข้อมูลการยืนยันตัวตน และ
  การกำหนดค่าคำขอรันไทม์
- `"all"`: แค็ตตาล็อก Gateway แบบเต็ม โดยข้าม `agents.defaults.modelPolicy.allow` ใช้สำหรับ
  UI การวินิจฉัย/การค้นพบ ไม่ใช่ตัวเลือกโมเดลทั่วไป

## การอนุมัติ Exec

- เมื่อคำขอ Exec ต้องได้รับการอนุมัติ Gateway จะกระจาย
  `exec.approval.requested`
- ไคลเอนต์ผู้ดำเนินการดำเนินการให้เสร็จโดยเรียก `exec.approval.resolve` (ต้องใช้
  `operator.approvals`)
- สำหรับ `host=node` นั้น `exec.approval.request` ต้องมี `systemRunPlan`
  (ข้อมูลเมตา `argv`/`cwd`/`rawCommand`/เซสชันตามแบบมาตรฐาน) คำขอที่ไม่มี
  `systemRunPlan` จะถูกปฏิเสธ
- หลังจากอนุมัติแล้ว การเรียก `node.invoke system.run` ที่ส่งต่อจะใช้
  `systemRunPlan` ตามแบบมาตรฐานนั้นซ้ำเป็นบริบทคำสั่ง/cwd/เซสชันที่เชื่อถือได้
- หากผู้เรียกเปลี่ยนแปลง `command`, `rawCommand`, `cwd`, `agentId` หรือ
  `sessionKey` ระหว่างการเตรียมและการส่งต่อ `system.run` ขั้นสุดท้ายที่ได้รับอนุมัติ
  Gateway จะปฏิเสธการเรียกใช้แทนการเชื่อถือเพย์โหลดที่ถูกเปลี่ยนแปลง

## การย้อนกลับสำหรับการส่งมอบของเอเจนต์

- คำขอ `agent` สามารถมี `deliver=true` เพื่อร้องขอการส่งมอบขาออก
- `bestEffortDeliver=false` (ค่าเริ่มต้น) คงลักษณะการทำงานแบบเคร่งครัด: เป้าหมายการส่งมอบที่แก้ไขไม่ได้หรือ
  ใช้ภายในเท่านั้นจะส่งคืน `INVALID_REQUEST`
- `bestEffortDeliver=true` อนุญาตให้ย้อนกลับไปใช้การดำเนินการเฉพาะเซสชันเมื่อไม่สามารถ
  แก้ไขเส้นทางที่ส่งมอบออกภายนอกได้ (เช่น เซสชันภายใน/เว็บแชต
  หรือการกำหนดค่าหลายช่องทางที่กำกวม)
- ผลลัพธ์ `agent` ขั้นสุดท้ายอาจมี `result.deliveryStatus` เมื่อมีการร้องขอ
  การส่งมอบ โดยใช้สถานะ `sent`, `suppressed`, `partial_failed` และ
  `failed` เดียวกับที่ระบุไว้สำหรับ
  [`openclaw agent --json --deliver`](/th/cli/agent#json-delivery-status)

## การกำหนดเวอร์ชัน

- `PROTOCOL_VERSION`, `MIN_CLIENT_PROTOCOL_VERSION`,
  `MIN_NODE_PROTOCOL_VERSION` และ `MIN_PROBE_PROTOCOL_VERSION` อยู่ใน
  `packages/gateway-protocol/src/version.ts`
- ไคลเอนต์ส่ง `minProtocol` + `maxProtocol` ไคลเอนต์ผู้ดำเนินการและ UI ต้อง
  มีโปรโตคอลปัจจุบันอยู่ในช่วงดังกล่าว ไคลเอนต์และเซิร์ฟเวอร์ปัจจุบันใช้
  โปรโตคอล v4
- ไคลเอนต์ที่ยืนยันตัวตนแล้วและมีทั้ง `role: "node"` และ `client.mode: "node"`
  อาจใช้โปรโตคอล Node แบบ N-1 (ปัจจุบันคือ v3) โพรบรีสตาร์ตแบบน้ำหนักเบาใช้
  ช่วง N-1 เดียวกัน การยืนยันตัวตนอุปกรณ์ การจับคู่ ขอบเขต นโยบายคำสั่ง และการอนุมัติ
  Exec จะไม่เปลี่ยนแปลงจากช่วงความเข้ากันได้นี้ ความสามารถและคำสั่งของ Node
  ที่ Plugin เป็นเจ้าของจะถูกระงับไว้จนกว่า Node จะอัปเกรดเป็นโปรโตคอลปัจจุบัน
  เนื่องจากพื้นผิวที่โฮสต์โดยสิ่งเหล่านี้ไม่ได้เป็นส่วนหนึ่งของสัญญา N-1
- สคีมาและโมเดลสร้างขึ้นจากคำจำกัดความ TypeBox:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

## การตรวจสอบสิทธิ์

- การตรวจสอบสิทธิ์ Gateway ด้วยข้อมูลลับที่ใช้ร่วมกันใช้ `connect.params.auth.token` หรือ
  `connect.params.auth.password` โดยขึ้นอยู่กับ
  `gateway.auth.mode` ที่กำหนดค่าไว้ (`"none" | "token" | "password" | "trusted-proxy"`)
- โหมดที่มีข้อมูลประจำตัว เช่น Tailscale Serve (`gateway.auth.allowTailscale: true`)
  หรือ `gateway.auth.mode: "trusted-proxy"` ที่ไม่ใช่ลูปแบ็ก จะผ่านการตรวจสอบสิทธิ์
  การเชื่อมต่อจากส่วนหัวของคำขอแทน `connect.params.auth.*`
- `gateway.auth.mode: "none"` สำหรับทางเข้าระบบส่วนตัวจะข้ามการตรวจสอบสิทธิ์การเชื่อมต่อด้วยข้อมูลลับที่ใช้ร่วมกัน
  ทั้งหมด ห้ามเปิดเผยโหมดดังกล่าวบนทางเข้าระบบสาธารณะ/ที่ไม่น่าเชื่อถือ
- หลังจากจับคู่ Gateway จะออกโทเค็นอุปกรณ์ที่จำกัดขอบเขตตามบทบาทและขอบเขต
  ของการเชื่อมต่อ โดยส่งคืนใน `hello-ok.auth.deviceToken` ไคลเอนต์ควร
  จัดเก็บโทเค็นหลังจากเชื่อมต่อสำเร็จทุกครั้ง
- เมื่อเชื่อมต่อใหม่ด้วยโทเค็นอุปกรณ์ที่จัดเก็บไว้ ควรนำชุดขอบเขตที่อนุมัติและจัดเก็บไว้
  สำหรับโทเค็นนั้นกลับมาใช้ด้วย วิธีนี้จะรักษาสิทธิ์การอ่าน/ตรวจสอบ/ดูสถานะ
  ที่ได้รับแล้ว และหลีกเลี่ยงไม่ให้การเชื่อมต่อใหม่ถูกลดเหลือขอบเขตโดยนัย
  ที่แคบกว่าและจำกัดเฉพาะผู้ดูแลระบบโดยไม่มีการแจ้งเตือน
- การประกอบข้อมูลตรวจสอบสิทธิ์การเชื่อมต่อฝั่งไคลเอนต์ (`selectConnectAuth` ใน
  `packages/gateway-client/src/client.ts`):
  - `auth.password` เป็นอิสระจากส่วนอื่นและจะถูกส่งต่อเสมอเมื่อตั้งค่าไว้
  - `auth.token` จะถูกเติมตามลำดับความสำคัญดังนี้: โทเค็นที่ใช้ร่วมกันซึ่งระบุอย่างชัดเจนก่อน
    จากนั้น `deviceToken` ที่ระบุอย่างชัดเจน แล้วจึงโทเค็นที่จัดเก็บไว้ต่ออุปกรณ์ (อ้างอิงด้วย
    `deviceId` + `role`)
  - `auth.bootstrapToken` จะถูกส่งเฉพาะเมื่อไม่มีรายการข้างต้นใดระบุ
    `auth.token` ได้ โทเค็นที่ใช้ร่วมกันหรือโทเค็นอุปกรณ์ใดก็ตามที่ระบุได้จะระงับการส่งค่านี้
  - การเลื่อนระดับโทเค็นอุปกรณ์ที่จัดเก็บไว้โดยอัตโนมัติในการลองใหม่ครั้งเดียว
    ด้วย `AUTH_TOKEN_MISMATCH` จำกัดไว้เฉพาะปลายทางที่เชื่อถือได้เท่านั้น ได้แก่ ลูปแบ็ก
    หรือ `wss://` ที่มี `tlsFingerprint` ซึ่งปักหมุดไว้ `wss://` สาธารณะที่ไม่มีการปักหมุด
    ไม่เข้าเกณฑ์
- การเริ่มต้นระบบด้วยรหัสตั้งค่าในตัวจะส่งคืน
  `hello-ok.auth.deviceToken` ของ Node หลัก พร้อมโทเค็นผู้ปฏิบัติงานที่มีขอบเขตจำกัดใน
  `hello-ok.auth.deviceTokens` สำหรับส่งต่อไปยังอุปกรณ์เคลื่อนที่ที่เชื่อถือได้ โทเค็นผู้ปฏิบัติงาน
  มี `operator.talk.secrets` สำหรับอ่านการกำหนดค่า Talk แบบเนทีฟ แต่
  ไม่รวมขอบเขตการแก้ไขการจับคู่และ `operator.admin`
- ระหว่างที่การเริ่มต้นระบบด้วยรหัสตั้งค่าที่ไม่ใช่ค่าพื้นฐานรอการอนุมัติ
  รายละเอียด `PAIRING_REQUIRED` จะมี `recommendedNextStep: "wait_then_retry"`,
  `retryable: true` และ `pauseReconnect: false` ให้เชื่อมต่อใหม่ต่อไปด้วย
  โทเค็นเริ่มต้นระบบเดิมจนกว่าคำขอจะได้รับอนุมัติหรือโทเค็น
  ใช้ไม่ได้
- จัดเก็บ `hello-ok.auth.deviceTokens` เฉพาะเมื่อการเชื่อมต่อใช้การตรวจสอบสิทธิ์แบบเริ่มต้นระบบ
  ผ่านการรับส่งข้อมูลที่เชื่อถือได้ เช่น `wss://` หรือการจับคู่ผ่านลูปแบ็ก/ภายในเครื่อง
- หากไคลเอนต์ระบุ `deviceToken` หรือ `scopes` อย่างชัดเจน ชุดขอบเขต
  ที่ผู้เรียกขอยังคงเป็นค่าหลัก ส่วนขอบเขตที่แคชไว้จะถูกนำกลับมาใช้เฉพาะ
  เมื่อไคลเอนต์นำโทเค็นที่จัดเก็บไว้ต่ออุปกรณ์กลับมาใช้
- สามารถหมุนเวียน/เพิกถอนโทเค็นอุปกรณ์ได้ผ่าน `device.token.rotate` และ
  `device.token.revoke` (ต้องใช้ `operator.pairing`) การหมุนเวียนหรือเพิกถอน
  โทเค็นของ Node หรือบทบาทอื่นที่ไม่ใช่ผู้ปฏิบัติงานต้องใช้ `operator.admin` ด้วย
- `device.token.rotate` ส่งคืนข้อมูลเมตาการหมุนเวียน โดยจะแสดงโทเค็นผู้ถือทดแทน
  เฉพาะสำหรับการเรียกจากอุปกรณ์เดียวกันที่ผ่านการตรวจสอบสิทธิ์ด้วยโทเค็นอุปกรณ์นั้นแล้ว
  เพื่อให้ไคลเอนต์ที่ใช้เฉพาะโทเค็นสามารถจัดเก็บโทเค็นทดแทนก่อน
  เชื่อมต่อใหม่ การหมุนเวียนด้วยข้อมูลลับที่ใช้ร่วมกัน/ผู้ดูแลระบบจะไม่แสดงโทเค็นผู้ถือ
- การออก การหมุนเวียน และการเพิกถอนโทเค็นจะยังคงจำกัดอยู่ภายในชุดบทบาทที่อนุมัติ
  ซึ่งบันทึกไว้ในรายการจับคู่ของอุปกรณ์นั้น การแก้ไขโทเค็นไม่สามารถขยายหรือ
  กำหนดเป้าหมายบทบาทอุปกรณ์ที่ไม่เคยได้รับอนุญาตจากการอนุมัติการจับคู่
- สำหรับเซสชันโทเค็นของอุปกรณ์ที่จับคู่แล้ว การจัดการอุปกรณ์จะจำกัดเฉพาะตนเอง เว้นแต่
  ผู้เรียกจะมี `operator.admin` ด้วย: ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบสามารถจัดการได้เฉพาะ
  โทเค็นผู้ปฏิบัติงานของรายการอุปกรณ์ตนเอง การจัดการโทเค็นของ Node และบทบาทอื่น
  ที่ไม่ใช่ผู้ปฏิบัติงานสงวนไว้เฉพาะผู้ดูแลระบบ แม้จะเป็นอุปกรณ์ของผู้เรียกเองก็ตาม
- `device.token.rotate` และ `device.token.revoke` ยังตรวจสอบชุดขอบเขตของ
  โทเค็นผู้ปฏิบัติงานเป้าหมายเทียบกับขอบเขตเซสชันปัจจุบันของผู้เรียก
  ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบไม่สามารถหมุนเวียนหรือเพิกถอนโทเค็นผู้ปฏิบัติงานที่มีขอบเขตกว้างกว่า
  ขอบเขตที่ตนมีอยู่แล้ว
- ความล้มเหลวในการตรวจสอบสิทธิ์มี `error.details.code` พร้อมคำแนะนำการกู้คืน:
  - `error.details.canRetryWithDeviceToken` (บูลีน)
  - `error.details.recommendedNextStep`: ค่าใดค่าหนึ่งจาก `retry_with_device_token`,
    `update_auth_configuration`, `update_auth_credentials`,
    `wait_then_retry`, `review_auth_configuration`
    (`packages/gateway-protocol/src/connect-error-details.ts`)
- พฤติกรรมไคลเอนต์สำหรับ `AUTH_TOKEN_MISMATCH`:
  - ไคลเอนต์ที่เชื่อถือได้อาจลองใหม่หนึ่งครั้งภายในขอบเขตที่จำกัดด้วยโทเค็น
    ต่ออุปกรณ์ที่แคชไว้
  - หากการลองใหม่นั้นล้มเหลว ให้หยุดวงรอบการเชื่อมต่อใหม่อัตโนมัติและแสดงคำแนะนำ
    การดำเนินการแก่ผู้ปฏิบัติงาน
- `AUTH_SCOPE_MISMATCH` หมายความว่าระบบรู้จักโทเค็นอุปกรณ์ แต่โทเค็นนั้นไม่ครอบคลุม
  บทบาท/ขอบเขตที่ขอ อย่าแสดงว่าเป็นโทเค็นที่ไม่ถูกต้อง ให้แจ้ง
  ผู้ปฏิบัติงานให้จับคู่ใหม่หรืออนุมัติสัญญาขอบเขตที่แคบกว่า/กว้างกว่า

## ข้อมูลประจำตัวและการจับคู่อุปกรณ์

- Node ควรระบุข้อมูลประจำตัวอุปกรณ์ที่คงที่ (`device.id`) ซึ่งได้มาจาก
  ลายนิ้วมือของคู่กุญแจ
- Gateway ออกโทเค็นต่ออุปกรณ์ + บทบาท
- ต้องได้รับการอนุมัติการจับคู่สำหรับ ID อุปกรณ์ใหม่ เว้นแต่จะเปิดใช้
  การอนุมัติอัตโนมัติภายในเครื่อง
- การอนุมัติการจับคู่อัตโนมัติมุ่งเน้นการเชื่อมต่อผ่านลูปแบ็กภายในเครื่องโดยตรง
- OpenClaw ยังมีเส้นทางเชื่อมต่อตนเองภายในแบ็กเอนด์/คอนเทนเนอร์แบบจำกัด
  สำหรับโฟลว์ตัวช่วยที่เชื่อถือได้ซึ่งใช้ข้อมูลลับร่วมกัน
- การเชื่อมต่อผ่าน tailnet หรือ LAN บนโฮสต์เดียวกันยังคงถือเป็นการเชื่อมต่อระยะไกลสำหรับการจับคู่
  และต้องได้รับการอนุมัติ
- โดยปกติไคลเอนต์ WS จะระบุข้อมูลประจำตัว `device` ระหว่าง `connect` (ผู้ปฏิบัติงาน +
  Node) ข้อยกเว้นสำหรับผู้ปฏิบัติงานที่ไม่มีอุปกรณ์มีเพียงเส้นทางความเชื่อถือที่ระบุไว้อย่างชัดเจน:
  - `gateway.controlUi.allowInsecureAuth=true` สำหรับความเข้ากันได้กับ
    HTTP ที่ไม่ปลอดภัยและจำกัดเฉพาะ localhost
  - การตรวจสอบสิทธิ์ Control UI ของผู้ปฏิบัติงานด้วย `gateway.auth.mode: "trusted-proxy"` ที่สำเร็จ
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (ใช้ในกรณีฉุกเฉิน เป็นการ
    ลดระดับความปลอดภัยอย่างร้ายแรง)
  - RPC แบ็กเอนด์ `gateway-client` ผ่านลูปแบ็กโดยตรงบนเส้นทางตัวช่วยภายใน
    ที่สงวนไว้
- การละเว้นข้อมูลประจำตัวอุปกรณ์ส่งผลต่อขอบเขต เมื่อการเชื่อมต่อของ
  ผู้ปฏิบัติงานที่ไม่มีอุปกรณ์ได้รับอนุญาตผ่านเส้นทางความเชื่อถือที่ระบุไว้อย่างชัดเจน OpenClaw
  จะยังคงล้างขอบเขตที่ประกาศเองให้เป็นชุดว่าง เว้นแต่เส้นทางนั้นจะมี
  ข้อยกเว้นการรักษาขอบเขตที่มีชื่อกำกับ จากนั้นเมธอดที่ควบคุมด้วยขอบเขตจะล้มเหลวด้วย
  `missing scope`
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` เป็นเส้นทางรักษาขอบเขตของ Control UI
  สำหรับกรณีฉุกเฉิน เส้นทางนี้ไม่ได้ให้ขอบเขตแก่ไคลเอนต์ WebSocket
  แบ็กเอนด์แบบกำหนดเองหรือที่มีรูปแบบเหมือน CLI โดยพลการ
- เส้นทางตัวช่วยแบ็กเอนด์ `gateway-client` ผ่านลูปแบ็กโดยตรงที่สงวนไว้จะรักษา
  ขอบเขตเฉพาะสำหรับ RPC ของระนาบควบคุมภายในเครื่องเท่านั้น ID แบ็กเอนด์แบบกำหนดเอง
  จะไม่ได้รับข้อยกเว้นนี้
- การเชื่อมต่อทั้งหมดต้องลงนาม nonce `connect.challenge` ที่เซิร์ฟเวอร์ให้มา

### การวินิจฉัยการย้ายระบบตรวจสอบสิทธิ์อุปกรณ์

สำหรับไคลเอนต์รุ่นเก่าที่ยังคงใช้พฤติกรรมการลงนามก่อนการท้าทาย `connect`
จะส่งคืนรหัสรายละเอียด `DEVICE_AUTH_*` ภายใต้ `error.details.code` พร้อม
`error.details.reason` ที่คงที่

ความล้มเหลวในการย้ายระบบที่พบบ่อย:

| ข้อความ                     | details.code                     | details.reason           | ความหมาย                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | ไคลเอ็นต์ไม่ได้ระบุ `device.nonce` (หรือส่งค่าว่าง)     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | ไคลเอ็นต์ลงนามด้วย nonce ที่เก่าหรือไม่ถูกต้อง            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | เพย์โหลดลายเซ็นไม่ตรงกับเพย์โหลด v2       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | การประทับเวลาที่ลงนามอยู่นอกช่วงความคลาดเคลื่อนที่อนุญาต          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` ไม่ตรงกับลายนิ้วมือของกุญแจสาธารณะ |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | การจัดรูปแบบหรือการแปลงกุญแจสาธารณะให้อยู่ในรูปแบบมาตรฐานล้มเหลว         |

เป้าหมายการย้ายระบบ:

- รอ `connect.challenge` เสมอ
- ลงนามเพย์โหลด v2 ที่มี nonce ของเซิร์ฟเวอร์
- ส่ง nonce เดียวกันใน `connect.params.device.nonce`
- เพย์โหลดลายเซ็นที่แนะนำคือ `v3`
  (`buildDeviceAuthPayloadV3` ใน `packages/gateway-client/src/device-auth.ts`),
  ซึ่งผูก `platform` และ `deviceFamily` เพิ่มเติมจาก
  ฟิลด์อุปกรณ์/ไคลเอ็นต์/บทบาท/ขอบเขต/โทเค็น/nonce
- ลายเซ็น `v2` แบบเดิมยังคงได้รับการยอมรับเพื่อความเข้ากันได้ แต่การตรึง
  เมทาดาทาของอุปกรณ์ที่จับคู่ยังคงควบคุมนโยบายคำสั่งเมื่อเชื่อมต่ออีกครั้ง

## TLS และการตรึง

- รองรับ TLS สำหรับการเชื่อมต่อ WS (การกำหนดค่า `gateway.tls`)
- ไคลเอ็นต์สามารถเลือกตรึงลายนิ้วมือใบรับรองของ Gateway ผ่าน
  `gateway.remote.tlsFingerprint` หรือ CLI `--tls-fingerprint`

## ขอบเขต

โปรโตคอลนี้เปิดเผย API ของ Gateway ทั้งหมด ได้แก่ สถานะ ช่องทาง โมเดล แชต
เอเจนต์ เซสชัน โหนด การอนุมัติ และอื่นๆ พื้นผิวที่แน่นอนกำหนดโดย
สคีมา TypeBox ที่ส่งออกซ้ำจาก `packages/gateway-protocol/src/schema.ts`
