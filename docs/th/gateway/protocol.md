โปรโตคอล Gateway WS เป็นระนาบควบคุมและการขนส่ง Node เพียงหนึ่งเดียวสำหรับ
OpenClaw ไคลเอนต์ของผู้ดำเนินการและ Node (CLI, UI เว็บ, แอป macOS, Node iOS/Android,
Node แบบไม่มีส่วนติดต่อผู้ใช้) เชื่อมต่อผ่าน WebSocket และประกาศ **บทบาท** และ **ขอบเขต** ณ
เวลาจับมือเชื่อมต่อ

## การขนส่งและการจัดเฟรม

- WebSocket, เฟรมข้อความ, เพย์โหลด JSON
- เฟรมแรก **ต้อง** เป็นคำขอ `connect`
- เฟรมก่อนเชื่อมต่อจำกัดไว้ที่ 64 KiB (`MAX_PREAUTH_PAYLOAD_BYTES`) หลังจาก
  จับมือเชื่อมต่อ ให้ปฏิบัติตาม `hello-ok.policy.maxPayload` และ
  `hello-ok.policy.maxBufferedBytes` เมื่อเปิดใช้การวินิจฉัย เฟรมขาเข้าที่มีขนาดเกินกำหนด
  และบัฟเฟอร์ขาออกที่ทำงานช้าจะปล่อยเหตุการณ์ `payload.large` ก่อนที่
  Gateway จะปิดหรือทิ้งเฟรม เหตุการณ์เหล่านี้มี `surface` ขนาดเป็นไบต์
  ขีดจำกัด และรหัสเหตุผลที่ปลอดภัย โดยจะไม่มีเนื้อหาข้อความ เนื้อหาไฟล์แนบ
  ไบต์ดิบของเฟรม โทเค็น คุกกี้ หรือข้อมูลลับ

รูปแบบเฟรม:

- คำขอ: `{type:"req", id, method, params}`
- การตอบกลับ: `{type:"res", id, ok, payload|error}`
- เหตุการณ์: `{type:"event", event, payload, seq?, stateVersion?}`

ข้อผิดพลาดในการตอบกลับใช้ `{ code, message, details?, retryable?, retryAfterMs? }`
ไคลเอนต์ควรแยกแขนงตาม `code` และ `details.code`; `message` ยังคงอยู่ในรูปแบบที่มนุษย์อ่านได้
และอาจเปลี่ยนแปลงได้ เว้นแต่หมายเหตุด้านความเข้ากันได้จะระบุไว้เป็นอย่างอื่น ความล้มเหลวด้าน
การอนุญาตระดับเมธอดใช้ `code: "FORBIDDEN"` ระดับบนสุดพร้อมรายละเอียด
ขอบเขตที่ขาดหายไปแบบมีโครงสร้าง:

- ขอบเขตที่ขาดหายไป: `{ code: "MISSING_SCOPE", missingScope, requiredScopes }`
  `requiredScopes` คือชุดขอบเขตที่รู้จักทั้งหมดสำหรับการดำเนินการที่ร้องขอ
  ข้อความรุ่นเก่า `missing scope: <scope>` ยังคงเก็บไว้สำหรับไคลเอนต์รุ่นเก่า

ไคลเอนต์ควรอ่าน `details` ก่อน และใช้ข้อความรุ่นเก่าเป็นเพียงทางเลือกสำรอง
เพื่อความเข้ากันได้ `readMissingScopeError` และ `readMissingScopeErrorDetails` ส่งออกจาก
`@openclaw/gateway-protocol/gateway-error-details`; ไคลเอนต์ Gateway ที่ปลอดภัยสำหรับเบราว์เซอร์
ส่งออกค่าเหล่านี้ซ้ำจาก `@openclaw/gateway-client/browser`

สคีมาส่งออกเป็น `GatewayErrorDetailsSchema`,
`MissingScopeErrorDetailsSchema` จาก `@openclaw/gateway-protocol/schema`
ความล้มเหลวของขอบเขต HTTP สะท้อนออบเจ็กต์ `MISSING_SCOPE` ภายใต้ `error.details` และ
ใช้สถานะ HTTP `403`

เมธอดที่ก่อให้เกิดผลข้างเคียงต้องใช้คีย์ idempotency (ดูสคีมา)

## การจับมือเชื่อมต่อ

Gateway ส่งคำท้าทายก่อนเชื่อมต่อ:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

ไคลเอนต์ตอบกลับด้วย `connect`:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway ตอบกลับด้วย `hello-ok`:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`, `features`, `snapshot`, `policy` และ `auth` ล้วนเป็นค่าที่
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) กำหนดให้ต้องมี `auth`
รายงานบทบาท/ขอบเขตที่เจรจาตกลงกัน แม้ไม่มีการออกโทเค็นอุปกรณ์ (รูปแบบ
ด้านบน) `pluginSurfaceUrls` เป็นค่าที่ไม่บังคับและจับคู่ชื่อพื้นผิว Plugin (เช่น
`canvas`) กับ URL ที่โฮสต์และกำหนดขอบเขตไว้ ค่านี้อาจหมดอายุ ดังนั้น Node จึงเรียก
`node.pluginSurface.refresh` พร้อม `{ "surface": "canvas" }` เพื่อรับรายการใหม่
เส้นทาง `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
ที่เลิกใช้แล้วไม่ได้รับการรองรับ ให้ใช้พื้นผิว Plugin
ค่า `appliedConfigHash` ที่ไม่บังคับของสแนปช็อตคือรุ่นแก้ไขของการกำหนดค่าต้นทางที่ผ่านการแก้ค่าแล้ว
ซึ่งรันไทม์ Gateway ที่ทำงานอยู่ยอมรับ ไคลเอนต์สามารถเปรียบเทียบค่านี้กับ
`config.get.configRevisionHash` เพื่อพิจารณาว่าการกำหนดค่าที่บันทึกไว้ซึ่งใหม่กว่ายังคง
ต้องรีสตาร์ตหรือไม่ `config.get.hash` ยังคงเป็นรุ่นแก้ไขดิบของไฟล์รากที่ใช้โดย
กลไกป้องกันความขัดแย้งในการเขียนการกำหนดค่า

ระหว่างที่ Gateway ยังคงเริ่มต้น sidecar ให้เสร็จสิ้น `connect` อาจส่งคืน
ข้อผิดพลาด `UNAVAILABLE` ที่ลองใหม่ได้ พร้อม `details.reason: "startup-sidecars"` และ
`retryAfterMs` ให้ลองใหม่ภายในงบประมาณการเชื่อมต่อ แทนที่จะถือว่าเป็น
ความล้มเหลวขั้นสุดท้ายของการจับมือเชื่อมต่อ

เมื่อมีการออกโทเค็นอุปกรณ์ `hello-ok.auth` จะเพิ่มโทเค็นดังกล่าว:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

การบูตสแตรปด้วย QR/รหัสตั้งค่าที่มีมาให้ในตัวเป็นเส้นทางส่งต่อไปยังอุปกรณ์เคลื่อนที่
การเชื่อมต่อด้วยรหัสตั้งค่าพื้นฐานที่สำเร็จจะส่งคืนโทเค็น Node หลักและโทเค็น
ผู้ดำเนินการแบบจำกัดหนึ่งรายการ:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

การส่งต่อให้ผู้ดำเนินการนี้ถูกจำกัดไว้โดยเจตนา: เพียงพอสำหรับเริ่มลูปผู้ดำเนินการ
บนอุปกรณ์เคลื่อนที่และการตั้งค่าแบบเนทีฟ รวมถึง `operator.talk.secrets` สำหรับการอ่าน
การกำหนดค่า Talk แต่ไม่มีขอบเขตสำหรับเปลี่ยนแปลงการจับคู่และไม่มี `operator.admin` การเข้าถึง
การจับคู่/การดูแลระบบที่กว้างขึ้นต้องผ่านขั้นตอนการจับคู่หรือโทเค็นที่ได้รับอนุมัติแยกต่างหาก ให้เก็บ
`hello-ok.auth.deviceTokens` ไว้เฉพาะเมื่อการยืนยันตัวตนสำหรับบูตสแตรปทำงานผ่าน
การขนส่งที่เชื่อถือได้ (`wss://` หรือการจับคู่ผ่าน loopback/ภายในเครื่อง)

ไคลเอนต์แบ็กเอนด์ในโพรเซสเดียวกันที่เชื่อถือได้ (`client.id: "gateway-client"`,
`client.mode: "backend"`) อาจละ `device` ในการเชื่อมต่อ loopback โดยตรงเมื่อ
ยืนยันตัวตนด้วยโทเค็น/รหัสผ่าน Gateway ที่ใช้ร่วมกัน เส้นทางนี้สงวนไว้
สำหรับ RPC ของระนาบควบคุมภายใน (เช่น การอัปเดตเซสชันของเอเจนต์ย่อย) และหลีกเลี่ยง
ไม่ให้ค่าพื้นฐานการจับคู่ CLI/อุปกรณ์ที่ล้าสมัยขัดขวางงานแบ็กเอนด์ภายในเครื่อง ไคลเอนต์ระยะไกล
ไคลเอนต์จากเบราว์เซอร์ Node และไคลเอนต์ที่ใช้โทเค็นอุปกรณ์/ข้อมูลประจำตัวอุปกรณ์อย่างชัดเจน
ยังคงผ่านการตรวจสอบการจับคู่และการยกระดับขอบเขตตามปกติ

### บทบาท Worker และโปรโตคอลแบบปิด

Worker บนคลาวด์ใช้ช่องทางเข้าผ่าน loopback โดยเฉพาะ ผ่านทันเนล SSH ที่ Gateway เป็นเจ้าของ
และตรึงคีย์โฮสต์ไว้ ช่องทางนี้ยอมรับเฉพาะข้อมูลประจำตัวของ Worker และไม่ส่งต่อ
การยืนยันตัวตนทั่วไป เหตุการณ์ Node, RPC ของผู้ดำเนินการ หรือเมธอด Plugin โดยเด็ดขาด `connect` ที่เข้มงวด
ตรวจสอบข้อมูลประจำตัวอายุสั้นซึ่งจัดเก็บเป็นแฮชและผูกกับสภาพแวดล้อม แฮชบันเดิล
ยุคของเจ้าของ เวอร์ชันชุด RPC เวลาหมดอายุ และเซสชันที่เป็น null ได้หนึ่งรายการ โดยจะ
ตรวจสอบเวอร์ชันปัจจุบันและชุดฟีเจอร์แยกต่างหาก เมื่อสำเร็จจะส่งคืน
`worker-hello-ok` ขั้นต่ำ การเจรจาฟีเจอร์เป็นอิสระจากเวอร์ชันโปรโตคอลทั่วไป
เฟรมมีขนาดต่ำกว่า 64 KiB ยกเว้นเฟรม `worker.inference.start` ที่เจรจาตกลงแล้ว
ซึ่งอาจมีขนาดได้ถึง 25 MiB รายการอนุญาตแบบปิดประกอบด้วย `worker.heartbeat`,
`worker.transcript.commit`, `worker.live-event`, `worker.inference.start` และ
`worker.inference.cancel`

การคอมมิตทรานสคริปต์ใช้การกั้นด้วยยุคของเจ้าของ การผูกเซสชันที่ Gateway เป็นเจ้าของ
การเปรียบเทียบและสลับค่าใบฐาน และการเล่นซ้ำลำดับแบบคงทน Gateway จะสร้าง
รหัสรายการทรานสคริปต์และรหัสรายการแม่ผ่านตัวเขียนเซสชันตามปกติ ระบบจะตรวจสอบความเป็นเจ้าของและ
เวลาหมดอายุซ้ำในทุก RPC

### ความสามารถของไคลเอนต์

ไคลเอนต์ผู้ดำเนินการอาจประกาศความสามารถที่ไม่บังคับใน `connect.params.caps`:

- `tool-events`: ยอมรับเหตุการณ์วงจรชีวิตของเครื่องมือแบบมีโครงสร้าง
- `inline-widgets`: สามารถแสดงผลลัพธ์ของเครื่องมือวิดเจ็ตแบบอินไลน์ที่โฮสต์ไว้

ความสามารถของไคลเอนต์อธิบายไคลเอนต์ที่เชื่อมต่อ ไม่ใช่การอนุญาต เครื่องมือของเอเจนต์อาจประกาศความสามารถที่จำเป็น Gateway จะละเครื่องมือเหล่านั้น เว้นแต่ข้อกำหนดทั้งหมดจะปรากฏใน `caps` ของไคลเอนต์ต้นทาง การรันที่เริ่มจากช่องทางไม่มีความสามารถของไคลเอนต์ Gateway ดังนั้นเครื่องมือที่จำกัดด้วยความสามารถจึงใช้ไม่ได้ แม้นโยบายเครื่องมือจะอนุญาตอย่างชัดเจนก็ตาม

```json
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Node ประกาศการอ้างสิทธิ์ความสามารถ ณ เวลาเชื่อมต่อ:

- `caps`: หมวดหมู่ระดับสูง เช่น `camera`, `canvas`, `screen`,
  `location`, `voice`, `talk`
- `commands`: รายการอนุญาตคำสั่งสำหรับการเรียกใช้
- `permissions`: ตัวสลับแบบละเอียด (เช่น `screen.record`, `camera.capture`)

Gateway ถือว่าค่าเหล่านี้เป็นการอ้างสิทธิ์และบังคับใช้รายการอนุญาตฝั่งเซิร์ฟเวอร์

## บทบาทและขอบเขต

สำหรับโมเดลขอบเขตของผู้ดำเนินการฉบับเต็ม การตรวจสอบ ณ เวลาอนุมัติ และความหมายของข้อมูลลับ
ที่ใช้ร่วมกัน โปรดดู [ขอบเขตของผู้ดำเนินการ](/th/gateway/operator-scopes)

บทบาท:

- `operator`: ไคลเอนต์ระนาบควบคุม (CLI/UI/ระบบอัตโนมัติ)
- `node`: โฮสต์ความสามารถ (กล้อง/หน้าจอ/แคนวาส/system.run)
- `worker`: โฮสต์การดำเนินการบนคลาวด์บนโปรโตคอล Worker แบบปิดโดยเฉพาะ

ขอบเขตของผู้ดำเนินการ (`src/gateway/operator-scopes.ts`) ซึ่งเป็นชุดปิดทั้งหมด:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`talk.config` พร้อม `includeSecrets: true` ต้องใช้ `operator.talk.secrets` (หรือ
`operator.admin`) เมื่อมีข้อมูลลับรวมอยู่ด้วย ให้อ่านข้อมูลประจำตัวของผู้ให้บริการ Talk ที่ทำงานอยู่
จาก `talk.resolved.config.apiKey`; `talk.providers.<id>.apiKey`
ยังคงมีรูปแบบตามต้นทางและอาจเป็นออบเจ็กต์ SecretRef หรือสตริงที่ปกปิดข้อมูลแล้ว

เมธอด RPC ของ Gateway ที่ลงทะเบียนโดย Plugin อาจร้องขอขอบเขตผู้ดำเนินการของตนเอง
แต่คำนำหน้าหลักที่สงวนไว้เหล่านี้จะถูกแก้ค่าเป็น `operator.admin`
(`src/shared/gateway-method-policy.ts`) เสมอ: `config.*`, `exec.approvals.*`,
`wizard.*`, `update.*`

ขอบเขตของเมธอดเป็นเพียงด่านแรก คำสั่งแบบทับบางคำสั่งที่เข้าถึงผ่าน
`chat.send` ใช้การตรวจสอบระดับคำสั่งที่เข้มงวดยิ่งขึ้น: การเขียน `/config set` และ
`/config unset` แบบถาวรต้องใช้ `operator.admin` แม้สำหรับไคลเอนต์ Gateway ที่
มีขอบเขตผู้ดำเนินการระดับต่ำกว่าอยู่แล้ว

`node.pair.approve` มีการตรวจสอบขอบเขตเพิ่มเติม ณ เวลาอนุมัติ นอกเหนือจาก
ขอบเขตพื้นฐานของเมธอด (`operator.pairing`) โดยอิงตาม `commands`
(`src/infra/node-pairing-authz.ts`) ที่คำขอซึ่งรอดำเนินการประกาศไว้:

| คำสั่งที่ประกาศ                                                                                                             | ขอบเขตที่จำเป็น                       |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| ไม่มี                                                                                                                          | `operator.pairing`                    |
| คำสั่งทั่วไป                                                                                                             | `operator.pairing` + `operator.write` |
| มี `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir` หรือ `system.execApprovals.get/set` | `operator.pairing` + `operator.admin` |

### ความสามารถ/คำสั่ง/สิทธิ์ (Node)

Node ประกาศการอ้างสิทธิ์ความสามารถเมื่อเชื่อมต่อ:

- `caps`: หมวดหมู่ความสามารถระดับสูง เช่น `camera`, `canvas`, `screen`,
  `location`, `voice` และ `talk`
- `commands`: รายการคำสั่งที่อนุญาตสำหรับการเรียกใช้
- `permissions`: ตัวสลับแบบละเอียด (เช่น `screen.record`, `camera.capture`)

Gateway ถือว่าสิ่งเหล่านี้เป็น **การอ้างสิทธิ์** และบังคับใช้รายการที่อนุญาตฝั่งเซิร์ฟเวอร์
Node ที่เชื่อมต่อสามารถเผยแพร่ตัวอธิบาย Plugin หรือเครื่องมือ MCP ที่เป็นตัวเลือกและมองเห็นได้โดยเอเจนต์
ด้วย `node.pluginTools.update` หลังจากเชื่อมต่อหรือ
เชื่อมต่อใหม่สำเร็จ โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้จะรีสตาร์ตเพื่อใช้การเปลี่ยนแปลงรายการ MCP
ที่ประกาศไว้ วิธีอัปเดตนี้เป็นช่องทางเผยแพร่เพียงช่องทางเดียว โดยไม่รับตัวอธิบายเครื่องมือ Plugin ใน
พารามิเตอร์ `connect` ตัวอธิบายแต่ละรายการต้องใช้ `name` ของเครื่องมือที่ปลอดภัยสำหรับผู้ให้บริการ และระบุ
`command` ที่อยู่ในรายการคำสั่งที่อนุญาตปัจจุบันของ Node Gateway เชื่อถือเมตาดาต้า
ของตัวอธิบายจาก Node ที่จับคู่ไว้ กรองตัวอธิบายที่อยู่นอกขอบเขตคำสั่ง
ที่ได้รับอนุมัติ นำออกเมื่อ Node ตัดการเชื่อมต่อ และปฏิเสธความพยายามของผู้ปฏิบัติงาน
ในการแก้ไขแคตตาล็อกของ Node อื่น ตั้งค่า `gateway.nodes.pluginTools.enabled: false`
เพื่อละเว้นตัวอธิบายที่ Node เผยแพร่

โฮสต์ Node ที่เชื่อมต่อเผยแพร่แคตตาล็อก Skills ทดแทนฉบับสมบูรณ์ด้วย
`node.skills.update` เมธอดสำหรับบทบาท Node นี้เป็นช่องทางเผยแพร่ Skills ของ Node
เพียงช่องทางเดียว โดยไม่รับ Skills ในพารามิเตอร์ `connect` ตัวอธิบายแต่ละรายการประกอบด้วย
ชื่อที่ปลอดภัย คำอธิบาย และเนื้อหา `SKILL.md` ที่มีขอบเขตจำกัด Gateway จะแยกวิเคราะห์
เนื้อหานั้นด้วยตัวโหลด Skills ตามปกติ รวมไว้ในสแนปช็อต Skills ของเอเจนต์
ขณะที่ Node เชื่อมต่อ และนำออกเมื่อตัดการเชื่อมต่อ ตั้งค่า
`gateway.nodes.skills.enabled: false` เพื่อละเว้น Skills ที่ Node เผยแพร่

## สถานะการปรากฏ

- `system-presence` ส่งคืนรายการที่ใช้ข้อมูลประจำตัวอุปกรณ์เป็นคีย์ รวมถึง
  `deviceId`, `roles` และ `scopes` เพื่อให้ UI แสดงหนึ่งแถวต่ออุปกรณ์ได้ แม้
  อุปกรณ์นั้นจะเชื่อมต่อทั้งในฐานะผู้ปฏิบัติงานและ Node
- `node.list` มี `lastSeenAtMs` และ `lastSeenReason` ซึ่งเป็นตัวเลือก Node ที่เชื่อมต่อ
  จะรายงานเวลาเชื่อมต่อปัจจุบันพร้อมเหตุผล `connect` และ Node ที่จับคู่แล้วสามารถ
  รายงานสถานะการปรากฏเบื้องหลังแบบถาวรผ่านเหตุการณ์ Node ที่เชื่อถือได้ด้วย

Node macOS แบบเนทีฟยังสามารถส่งเหตุการณ์ `node.presence.activity` ที่ผ่านการรับรองความถูกต้อง
พร้อมเวลาที่ไม่มีการป้อนข้อมูลซึ่งมีขอบเขตจำกัด Gateway จะคำนวณเวลาประทับของกิจกรรมด้วย
นาฬิกาของตนเอง เปิดเผย Mac ที่เชื่อมต่อล่าสุดผ่าน `node.list` และ
`node.describe` และกระจายการอัปเดต `node.presence` ไปยังไคลเอนต์ที่มีขอบเขตสิทธิ์อ่าน
ดู [สถานะการปรากฏของคอมพิวเตอร์ที่ใช้งานอยู่](/th/nodes/presence) สำหรับพฤติกรรมการเลือก ความเป็นส่วนตัว บริบทของโมเดล
และการกำหนดเส้นทางการแจ้งเตือน

### เหตุการณ์ Node ยังทำงานอยู่เบื้องหลัง

Node เรียก `node.event` พร้อม `event: "node.presence.alive"` เพื่อบันทึกว่า
Node ที่จับคู่แล้วยังทำงานอยู่ระหว่างการปลุกเบื้องหลัง โดยไม่ทำเครื่องหมายว่าเชื่อมต่ออยู่:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` เป็น enum แบบปิด: `background`, `silent_push`, `bg_app_refresh`,
`significant_location`, `manual`, `connect` ค่าที่ไม่รู้จักจะถูกปรับให้เป็น
`background` (`src/shared/node-presence.ts`) เหตุการณ์จะคงอยู่เฉพาะสำหรับ
เซสชันอุปกรณ์ Node ที่ผ่านการรับรองความถูกต้องเท่านั้น เซสชันที่ไม่มีอุปกรณ์หรือไม่ได้จับคู่จะส่งคืน
`handled: false`

Gateway ที่ดำเนินการสำเร็จจะส่งคืนผลลัพธ์แบบมีโครงสร้าง:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

Gateway รุ่นเก่าอาจส่งคืนเพียง `{ "ok": true }` สำหรับ `node.event` ให้ถือว่า
เป็น RPC ที่ได้รับการตอบรับ ไม่ใช่การบันทึกสถานะการปรากฏแบบถาวร

## การกำหนดขอบเขตเหตุการณ์กระจาย

เหตุการณ์กระจายที่เซิร์ฟเวอร์พุชจะถูกจำกัดตามขอบเขต เพื่อให้เซสชันที่มีขอบเขตเฉพาะการจับคู่หรือเฉพาะ Node
ไม่ได้รับเนื้อหาเซสชันโดยอัตโนมัติ
(`src/gateway/server-broadcast.ts`):

- เฟรมแชต เอเจนต์ และผลลัพธ์เครื่องมือ (เหตุการณ์ `agent` แบบสตรีม เหตุการณ์ผลลัพธ์
  เครื่องมือ) ต้องมีอย่างน้อย `operator.read` เซสชันที่ไม่มีสิทธิ์นี้จะข้าม
  เฟรมเหล่านี้ทั้งหมด
- การกระจาย `plugin.*` ที่ Plugin กำหนดจะถูกจำกัดไว้ที่ `operator.write` หรือ
  `operator.admin` โดยค่าเริ่มต้น ส่วนรายการที่ระบุอย่างชัดเจน เช่น
  `plugin.approval.requested` / `plugin.approval.resolved` จะใช้
  `operator.approvals` แทน
- เหตุการณ์สถานะ/การขนส่ง (`heartbeat`, `presence`, `tick`, วงจรชีวิต
  การเชื่อมต่อ/ตัดการเชื่อมต่อ) ยังคงไม่ถูกจำกัด เพื่อให้ทุกเซสชัน
  ที่ผ่านการรับรองความถูกต้องสังเกตสถานะการทำงานของการขนส่งได้
- กลุ่มเหตุการณ์กระจายที่ไม่รู้จักจะถูกจำกัดตามขอบเขตโดยค่าเริ่มต้น (ปฏิเสธเมื่อไม่แน่ใจ)
  เว้นแต่ตัวจัดการที่ลงทะเบียนจะผ่อนคลายข้อจำกัดอย่างชัดเจน

การเชื่อมต่อของไคลเอนต์แต่ละรายการจะเก็บหมายเลขลำดับเฉพาะไคลเอนต์ของตนเอง ทำให้เหตุการณ์กระจาย
ยังคงเรียงตามลำดับเพิ่มขึ้นบนซ็อกเก็ตนั้น แม้ไคลเอนต์แต่ละรายจะเห็น
ชุดย่อยของสตรีมเหตุการณ์ที่ผ่านการกรองตามขอบเขตแตกต่างกัน

    - `talk.client.transcript` เพิ่มรายการ `{ role, text }` ที่เสร็จสมบูรณ์แล้วหนึ่งรายการต่อท้ายเซสชันเอเจนต์ปกติ `entryId` ที่จำเป็นมีคุณสมบัติ idempotent ภายใน `voiceSessionId`; การลองใหม่จะไม่สร้างข้อความบทสนทนาซ้ำ
    - `talk.client.close` ปิดเซสชันเสียงเชิงตรรกะหลังจากการเขียนบทสนทนาที่ค้างอยู่ การปิดมีคุณสมบัติ idempotent และอาจส่งข้อมูลสรุปการเรียกที่มีเฉพาะการแก้ไขไปยังช่องทางล่าสุดของเซสชันที่ไม่ใช่ WebChat
    - `talk.client.toolCall` ช่วยให้การรับส่งข้อมูลแบบเรียลไทม์ที่ไคลเอนต์เป็นเจ้าของส่งต่อการเรียกเครื่องมือของผู้ให้บริการไปยังนโยบายของ Gateway เครื่องมือแรกที่รองรับคือ `openclaw_agent_consult`; ไคลเอนต์จะได้รับ ID การรันและรอเหตุการณ์วงจรชีวิตแชตตามปกติก่อนส่งผลลัพธ์เครื่องมือเฉพาะผู้ให้บริการ การดำเนินการที่มีผลกระทบสูงซึ่งผูกกับเสียงจะส่งคืน `VOICE_CONFIRMATION_REQUIRED:<id>` จนกว่าคำพูดของผู้ใช้ที่เสร็จสมบูรณ์ในภายหลังจะยืนยันการดำเนินการนั้นอย่างชัดเจน และการปรึกษาครั้งถัดไปจะระบุ `confirmationId`
    - `talk.client.steer` ส่งการควบคุมเสียงของการรันที่ทำงานอยู่สำหรับการรับส่งข้อมูลแบบเรียลไทม์ที่ไคลเอนต์เป็นเจ้าของ Gateway จะระบุการรันแบบฝังที่ทำงานอยู่จาก `sessionKey` และส่งคืนผลลัพธ์การยอมรับ/ปฏิเสธแบบมีโครงสร้าง แทนที่จะละทิ้งการควบคุมโดยไม่แจ้ง
    - `talk.event` คือช่องทางเหตุการณ์ Talk เดียวสำหรับอะแดปเตอร์แบบเรียลไทม์ การถอดเสียง STT/TTS ห้องที่จัดการ โทรศัพท์ และการประชุม
    - `talk.speak` สังเคราะห์เสียงพูดผ่านผู้ให้บริการเสียงพูดของ Talk ที่ทำงานอยู่
    - `tts.status` ส่งคืนสถานะการเปิดใช้ TTS ผู้ให้บริการที่ทำงานอยู่ ผู้ให้บริการสำรอง และสถานะการกำหนดค่าผู้ให้บริการ
    - `tts.providers` ส่งคืนรายการผู้ให้บริการ TTS ที่มองเห็นได้
    - `tts.enable` และ `tts.disable` สลับสถานะการกำหนดลักษณะ TTS
    - `tts.setProvider` อัปเดตผู้ให้บริการ TTS ที่ต้องการ
    - `tts.convert` เรียกใช้การแปลงข้อความเป็นเสียงพูดแบบครั้งเดียว
    - `tts.speak` (`operator.write`) เรนเดอร์ `text` ที่ไม่ว่างด้วยลำดับผู้ให้บริการ TTS ทั่วไปที่กำหนดค่าไว้ และส่งคืนคลิปทั้งคลิปแบบอินไลน์เป็น `audioBase64` พร้อม `provider` และเมตาดาต้า `outputFormat`, `mimeType` และ `fileExtension` ซึ่งอาจมีหรือไม่มีก็ได้ วิธีนี้ไม่ส่งคืนพาธภายใน Gateway ซึ่งต่างจาก `tts.convert` และไม่ต้องใช้ผู้ให้บริการ Talk ซึ่งต่างจาก `talk.speak` ข้อความที่เกิน `messages.tts.maxTextLength` จะส่งคืน `INVALID_REQUEST`; ความล้มเหลวในการสังเคราะห์จะส่งคืน `UNAVAILABLE`

  <Accordion title="ข้อมูลลับ การกำหนดค่า การอัปเดต และวิซาร์ด">
    - `secrets.reload` ระบุ SecretRef ที่ทำงานอยู่อีกครั้งและเผยแพร่สถานะรันไทม์ที่รับรู้เจ้าของแบบอะตอมิก ความล้มเหลวของเจ้าของที่เข้าเกณฑ์สามารถเผยแพร่เป็นการลดระดับแบบเย็นหรือล้าสมัยพร้อม `warningCount`; ความล้มเหลวแบบเข้มงวดหรือที่ไม่มีการแมปจะปฏิเสธการโหลดใหม่และรักษาสแนปช็อตที่ทำงานอยู่
    - `secrets.resolve` ระบุการกำหนดข้อมูลลับเป้าหมายคำสั่งสำหรับชุดคำสั่ง/เป้าหมายที่เจาะจง
    - `config.get` ส่งคืนสแนปช็อตการกำหนดค่าปัจจุบันบนดิสก์ `hash` ของไฟล์รากแบบดิบ `configRevisionHash` ที่ระบุแล้ว และ `appliedConfigHash` ซึ่งอาจมีหรือไม่มีก็ได้ สำหรับรีวิชันที่ระบุแล้วซึ่งรันไทม์ Gateway ที่ทำงานอยู่ยอมรับ
    - `config.set` เขียนเพย์โหลดการกำหนดค่าที่ผ่านการตรวจสอบ
    - `config.patch` ผสานการอัปเดตการกำหนดค่าบางส่วน การแทนที่อาร์เรย์แบบทำลายข้อมูลต้องระบุพาธที่ได้รับผลกระทบใน `replacePaths`; อาร์เรย์ที่ซ้อนอยู่ภายใต้รายการอาร์เรย์ใช้พาธ `[]` เช่น `agents.list[].skills`
    - `config.apply` ตรวจสอบและแทนที่เพย์โหลดการกำหนดค่าทั้งหมด
    - `config.schema` ส่งคืนเพย์โหลดสคีมาการกำหนดค่าแบบสดที่เครื่องมือ Control UI และ CLI ใช้ ได้แก่ สคีมา `uiHints` เวอร์ชัน เมตาดาต้าการสร้าง และเมตาดาต้าสคีมาของ Plugin และช่องทางเมื่อโหลดได้ โดยรวมเมตาดาต้า `title` / `description` จากป้ายกำกับ/ข้อความช่วยเหลือชุดเดียวกับ UI รวมถึงสาขาการประกอบของออบเจ็กต์ที่ซ้อนอยู่ ไวลด์การ์ด รายการอาร์เรย์ และ `anyOf` / `oneOf` / `allOf` เมื่อมีเอกสารฟิลด์ที่ตรงกัน
    - `config.schema.lookup` ส่งคืนเพย์โหลดการค้นหาที่กำหนดขอบเขตตามพาธสำหรับพาธการกำหนดค่าหนึ่งรายการ ได้แก่ พาธที่ปรับให้เป็นมาตรฐาน โหนดสคีมาแบบตื้น คำแนะนำที่ตรงกันพร้อม `hintPath`, `reloadKind` ซึ่งอาจมีหรือไม่มีก็ได้ และข้อมูลสรุปของรายการลูกโดยตรงสำหรับการเจาะลึกใน UI/CLI `reloadKind` เป็นหนึ่งใน `restart`, `hot` หรือ `none` (`src/config/schema.ts`) และสะท้อนตัววางแผนการโหลดการกำหนดค่า Gateway ใหม่สำหรับพาธที่ร้องขอ โหนดสคีมาของการค้นหาจะเก็บเอกสารสำหรับผู้ใช้และฟิลด์การตรวจสอบทั่วไปไว้ (`title`, `description`, `type`, `enum`, `const`, `format`, `pattern`, ขอบเขตตัวเลข/สตริง/อาร์เรย์/ออบเจ็กต์, `additionalProperties`, `deprecated`, `readOnly`, `writeOnly`) ข้อมูลสรุปของรายการลูกเปิดเผย `key`, `path` ที่ปรับให้เป็นมาตรฐาน, `type`, `required`, `hasChildren`, `reloadKind` ซึ่งอาจมีหรือไม่มีก็ได้ พร้อม `hint` / `hintPath` ที่ตรงกัน
    - `update.run` เรียกใช้ขั้นตอนการอัปเดต Gateway และกำหนดเวลาการรีสตาร์ตเฉพาะเมื่ออัปเดตสำเร็จเท่านั้น ผู้เรียกที่มีเซสชันสามารถระบุ `continuationMessage` เพื่อให้ขั้นตอนเริ่มต้นกลับมาดำเนินรอบเอเจนต์ติดตามผลหนึ่งรอบผ่านคิวดำเนินการต่อหลังรีสตาร์ต การอัปเดตจากตัวจัดการแพ็กเกจและการอัปเดต checkout ของ Git ที่อยู่ภายใต้การควบคุมจากระนาบควบคุมจะใช้การส่งต่องานไปยังบริการที่จัดการแบบแยกออก แทนการแทนที่โครงสร้างแพ็กเกจหรือแก้ไขเอาต์พุตของ checkout/บิลด์ภายใน Gateway ที่กำลังทำงาน การส่งต่องานที่เริ่มแล้วจะส่งคืน `ok: true` พร้อม `result.reason: "managed-service-handoff-started"` และ `handoff.status: "started"` การเรียก `update.run` พร้อมกันครั้งที่สองซึ่งประมวลผลโดยกระบวนการ Gateway เดียวกันจะส่งคืน `ok: false` พร้อม `result.reason: "managed-service-handoff-already-running"` และ `handoff.status: "already-running"`; ระบบจะไม่รับการดำเนินการต่อของการเรียกนั้น เพื่อให้ผู้เรียกลองใหม่ได้หลังการอัปเดตที่ทำงานอยู่เสร็จสมบูรณ์ ตัวอัปเดต CLI แบบสแตนด์อโลนและกระบวนการ Gateway ที่เข้ามาแทนที่อยู่นอกการป้องกันเฉพาะกระบวนการนี้ การส่งต่องานที่ไม่พร้อมใช้งานหรือล้มเหลวจะส่งคืน `ok: false` พร้อม `managed-service-handoff-unavailable` หรือ `managed-service-handoff-failed` และ `handoff.command` เมื่อจำเป็นต้องอัปเดตด้วยเชลล์ด้วยตนเอง ไม่พร้อมใช้งานหมายถึง OpenClaw ไม่มีขอบเขตผู้ควบคุมที่ปลอดภัยหรือข้อมูลประจำตัวบริการแบบคงทน เช่น `OPENCLAW_SYSTEMD_UNIT` สำหรับ systemd ระหว่างการส่งต่องานที่เริ่มแล้ว ตัวบ่งชี้การรีสตาร์ตอาจรายงาน `stats.reason: "restart-health-pending"` ชั่วครู่; การดำเนินการต่อจะถูกหน่วงไว้จนกว่า CLI จะตรวจสอบ Gateway ที่รีสตาร์ตแล้วและเขียนตัวบ่งชี้ `ok` ขั้นสุดท้าย
    - `update.status` รีเฟรชและส่งคืนตัวบ่งชี้การรีสตาร์ตจากการอัปเดตล่าสุด รวมถึงเวอร์ชันที่ทำงานหลังรีสตาร์ตเมื่อมี
    - `wizard.start`, `wizard.next`, `wizard.status` และ `wizard.cancel` เปิดเผยวิซาร์ดการเริ่มต้นใช้งานผ่าน WS RPC

  </Accordion>

  <Accordion title="ตัวช่วยสำหรับเอเจนต์และเวิร์กสเปซ">
    - `agents.list` ส่งคืนรายการเอเจนต์ที่กำหนดค่าไว้ รวมถึงข้อมูลเมตาของโมเดลและรันไทม์ที่มีผล

    - `audit.activity.list` ส่งคืนบัญชีกิจกรรมแบบมีเวอร์ชันซึ่งมีเฉพาะข้อมูลเมตา ส่วน `audit.list` ยังคงเป็น RPC สำหรับการรัน/เครื่องมือที่ปลอดภัยด้านความเข้ากันได้

    - `artifacts.list`, `artifacts.get` และ `artifacts.download` เปิดให้เข้าถึงข้อมูลสรุปและการดาวน์โหลดอาร์ติแฟกต์ที่ได้จากทรานสคริปต์สำหรับขอบเขต `sessionKey`, `runId` หรือ `taskId` ที่ระบุอย่างชัดเจน คิวรีการรันและงานจะระบุเซสชันเจ้าของที่ฝั่งเซิร์ฟเวอร์ และส่งคืนเฉพาะสื่อในทรานสคริปต์ที่มีแหล่งที่มาตรงกัน ส่วนแหล่ง URL ที่ไม่ปลอดภัยหรือเป็น URL ภายในเครื่องจะส่งคืนว่าไม่รองรับการดาวน์โหลด แทนที่จะดึงข้อมูลจากฝั่งเซิร์ฟเวอร์
    - `environments.list` และ `environments.status` รักษาการค้นหาสภาพแวดล้อมภายใน Gateway และ Node ไว้ เวิร์กเกอร์คลาวด์ที่กำหนดค่าไว้และระเบียนถาวรที่โปรไฟล์ก่อนหน้าทิ้งไว้จะเพิ่มข้อมูลเมตา `worker` ซึ่งประกอบด้วย `providerId`, `leaseId` ซึ่งเป็นทางเลือก, `state`, `ageMs`, `idleMs` ซึ่งเป็นทางเลือก และ `attachedSessionIds` สถานะวงจรชีวิตของเวิร์กเกอร์ได้แก่ `requested`, `provisioning`, `bootstrapping`, `ready`, `attached`, `idle`, `draining`, `destroying`, `destroyed`, `failed` และ `orphaned`
    - `environments.create` (`{ profileId, idempotencyKey }`) จัดเตรียมเวิร์กเกอร์จากโปรไฟล์ผู้ให้บริการ Plugin ที่กำหนดค่าไว้ การลองใหม่ด้วยคีย์เดิมจะใช้การดำเนินการถาวรเดิมซ้ำ `environments.destroy` (`{ environmentId }`) ร้องขอให้รื้อถอนสภาพแวดล้อมเวิร์กเกอร์ถาวรแบบทำซ้ำได้อย่างปลอดภัย ทั้งสองรายการต้องใช้ `operator.admin` เป็นการเขียนในระนาบควบคุม และส่งคืนข้อมูลสรุปสภาพแวดล้อมในรูปแบบเดียวกับที่ใช้ในการตอบกลับสถานะ
    - `agent.identity.get` ส่งคืนข้อมูลประจำตัวผู้ช่วยที่มีผลสำหรับเอเจนต์หรือเซสชัน
    - `agent.wait` รอให้การรันเสร็จสิ้นและส่งคืนสแนปช็อตปลายทางเมื่อพร้อมใช้งาน

    - `sessions.list` ส่งคืนดัชนีเซสชันปัจจุบัน รวมถึงข้อมูลเมตา `agentRuntime` ของแต่ละแถวเมื่อกำหนดค่าแบ็กเอนด์รันไทม์ของเอเจนต์ไว้ เมื่อเปิดใช้การจัดวางบนเวิร์กเกอร์คลาวด์หรือมีสถานะการกู้คืนถาวร แถวเซสชันจะรวมสถานะปิด `placement` (`local`, `requested`, `provisioning`, `syncing`, `starting`, `active`, `draining`, `reconciling`, `reclaimed` หรือ `failed`) พร้อมฟิลด์สภาพแวดล้อม, owner-epoch, เวิร์กสเปซ, บันเดิล, เคอร์เซอร์ ACK หรือการกู้คืนตามแต่ละสถานะ
    - `sessions.subscribe` และ `sessions.unsubscribe` เปิดหรือปิดการสมัครรับเหตุการณ์การเปลี่ยนแปลงเซสชันสำหรับไคลเอนต์ WS ปัจจุบัน
    - `sessions.messages.subscribe` และ `sessions.messages.unsubscribe` เปิดหรือปิดการสมัครรับเหตุการณ์ทรานสคริปต์/ข้อความสำหรับหนึ่งเซสชัน ส่ง `includeApprovals: true` เพื่อรับเหตุการณ์วงจรชีวิต `session.approval` ที่ผ่านการล้างข้อมูลแล้วสำหรับการอนุมัติที่กลุ่มเป้าหมายซึ่งบันทึกถาวรไว้มีเซสชันนั้นตรงกันทุกประการ และการผูกผู้ตรวจสอบอนุญาตไคลเอนต์ที่สมัครรับอยู่ จากนั้นการตอบกลับการสมัครรับจะรวม `approvalReplay` ที่รอดำเนินการแบบมีขอบเขตจำกัด ซึ่งถือเป็นข้อมูลที่เชื่อถือได้เมื่อ `truncated` เป็น false การเลือกเข้าร่วมมีผลต่อการเรียกสมัครรับแต่ละครั้งและไม่คงอยู่ การสมัครรับเซสชันเดิมอีกครั้งโดยไม่มี `includeApprovals: true` จะลบการสมัครรับการอนุมัติที่มีอยู่ นอกเหนือจากสิทธิ์อ่านเซสชันตามปกติ การเลือกเข้าร่วมนี้ต้องใช้ `operator.admin` หรือ `operator.approvals` บนอุปกรณ์ที่จับคู่แล้ว
    - `sessions.preview` ส่งคืนตัวอย่างทรานสคริปต์แบบมีขอบเขตจำกัดสำหรับคีย์เซสชันที่ระบุ
    - `sessions.describe` ส่งคืนหนึ่งแถวเซสชันของ Gateway สำหรับคีย์เซสชันที่ตรงกันทุกประการ
    - `sessions.resolve` ระบุหรือทำเป้าหมายเซสชันให้อยู่ในรูปแบบมาตรฐาน
    - `sessions.create` สร้างรายการเซสชันใหม่ ค่า `model` และ `thinkingLevel` ซึ่งเป็นทางเลือกจะบันทึกค่าลบล้างโมเดลและการให้เหตุผลเริ่มต้นแบบอะตอมมิก `worktree: true` จัดเตรียม worktree ที่มีการจัดการ โดย `worktreeBaseRef`/`worktreeName` ซึ่งเป็นทางเลือกใช้เลือก ref ฐานและชื่อแบรนช์ และ `execNode` (`operator.admin`) ผูกการดำเนินการของเซสชันเข้ากับโฮสต์ Node worktree ที่สร้างขึ้นจะถูกส่งกลับในผลลัพธ์และบันทึกไว้ในแถวเซสชัน (`worktree: { id, branch, repoRoot }`) เมื่อสร้างรายการสำเร็จแต่ `chat.send` เริ่มต้นที่ซ้อนอยู่ถูกปฏิเสธ ผลลัพธ์ที่สำเร็จจะรวม `runStarted: false` และ `runError` ไคลเอนต์สามารถเก็บพรอมต์ไว้และลองใหม่กับคีย์เซสชันที่ส่งคืน ผู้เรียกที่ส่ง `parentSessionKey` พร้อม `emitCommandHooks: true` ควรประกาศการจัดการวงจรชีวิตของรายการลูกที่แยกต่างหากด้วย โดย `succeedsParent: true` สิ้นสุดรายการแม่ด้วย `session_end` ส่วน `false` ทำให้รายการแม่ยังคงทำงานและปล่อยเฉพาะ `session_start` ของรายการลูก การละ `succeedsParent` จะรักษาพฤติกรรมการเปลี่ยนผ่านรายการแม่แบบเดิมสำหรับไคลเอนต์ที่มีอยู่ การจัดการนี้ต้องมีทั้งการเชื่อมโยงรายการแม่และฮุกคำสั่ง fork ไม่สามารถทำให้รายการแม่สำเร็จได้ พฤติกรรมรีเซ็ตเซสชันหลักในตำแหน่งเดิมไม่เปลี่ยนแปลง เนื่องจากไม่มีการสร้างรายการลูกที่แยกต่างหาก
    - `sessions.dispatch` (`operator.admin`) ย้ายเซสชัน OpenClaw ภายในเครื่องที่มี worktree แบบจัดการซึ่งเซสชันเป็นเจ้าของ ไปยังโปรไฟล์เวิร์กเกอร์คลาวด์ที่กำหนดค่าไว้ ส่ง `{ key, profileId, agentId? }` เมธอดนี้จะไม่มีอยู่เมื่อไม่ได้กำหนดค่าโปรไฟล์เวิร์กเกอร์ โดยจะปิดการรับเทิร์นภายในเครื่องก่อนรอให้งานที่กำลังทำอยู่เสร็จ และจะส่งคืนเมื่อการจัดวางไปถึงความเป็นเจ้าของเวิร์กเกอร์ `active` แล้วเท่านั้น การส่งต่อเป็นทางเดียว RPC นี้ไม่รองรับการดึงกลับจากเวิร์กเกอร์มายังเครื่องภายใน
    - `sessions.groups.list`, `sessions.groups.put`, `sessions.groups.rename` และ `sessions.groups.delete` จัดการแค็ตตาล็อกกลุ่มเซสชันแบบกำหนดเองที่ Gateway เป็นเจ้าของ (ชื่อ + ลำดับการแสดงผล) สมาชิกภาพยังคงอยู่ในฟิลด์ `category` ของแต่ละเซสชัน การเปลี่ยนชื่อและการลบจะอัปเดตเซสชันสมาชิกที่ฝั่งเซิร์ฟเวอร์
    - `sessions.send` ส่งข้อความเข้าสู่เซสชันที่มีอยู่
    - `sessions.steer` เป็นรูปแบบขัดจังหวะและชี้นำสำหรับเซสชันที่กำลังทำงาน
    - `sessions.abort` ยกเลิกงานที่กำลังทำอยู่ของเซสชัน ส่ง `key` พร้อม `runId` ซึ่งเป็นทางเลือก หรือส่งเฉพาะ `runId` สำหรับการรันที่กำลังทำงานซึ่ง Gateway สามารถระบุเซสชันได้
    - `sessions.patch` อัปเดตข้อมูลเมตา/ค่าลบล้างของเซสชัน และรายงานโมเดลมาตรฐานที่ระบุได้พร้อม `agentRuntime` ที่มีผล
    - `sessions.reset`, `sessions.delete` และ `sessions.compact` ดำเนินการบำรุงรักษาเซสชัน
    - `sessions.get` ส่งคืนแถวเซสชันที่จัดเก็บไว้ทั้งหมด
    - การดำเนินการแชตยังคงใช้ `chat.history`, `chat.send`, `chat.abort` และ `chat.inject` ส่วน `chat.history` จะถูกปรับรูปแบบการแสดงผลให้เป็นมาตรฐานสำหรับไคลเอนต์ UI โดยลบแท็กไดเรกทีฟแบบอินไลน์ออกจากข้อความที่มองเห็น ลบเพย์โหลด XML ของการเรียกเครื่องมือแบบข้อความล้วน (`<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` และบล็อกการเรียกเครื่องมือที่ถูกตัดทอน) รวมถึงโทเค็นควบคุมโมเดล ASCII/แบบเต็มความกว้างที่รั่วออกมา ละแถวผู้ช่วยที่มีเฉพาะโทเค็นเงียบ (ตรงกับ `NO_REPLY` / `no_reply` ทุกประการ) และอาจแทนที่แถวที่มีขนาดใหญ่เกินไปด้วยตัวยึดตำแหน่ง
    - `chat.message.get` เป็นตัวอ่านข้อความฉบับเต็มแบบมีขอบเขตจำกัดที่เพิ่มเข้ามาสำหรับรายการทรานสคริปต์ที่มองเห็นได้หนึ่งรายการ ส่ง `sessionKey`, `agentId` ซึ่งเป็นทางเลือกเมื่อการเลือกเซสชันมีขอบเขตตามเอเจนต์ และ `messageId` ของทรานสคริปต์ที่เคยแสดงผ่าน `chat.history` มาก่อน Gateway จะส่งคืนการฉายภาพที่ปรับรูปแบบการแสดงผลเป็นมาตรฐานแบบเดียวกันโดยไม่มีขีดจำกัดการตัดทอนประวัติแบบน้ำหนักเบา หากรายการที่จัดเก็บยังพร้อมใช้งานและมีขนาดไม่ใหญ่เกินไป
    - `chat.toolTitles` ส่งคืนชื่อวัตถุประสงค์สั้น ๆ สำหรับการเรียกเครื่องมือที่แสดงใน Control UI (ประมวลผลเป็นชุด สูงสุด 24 รายการพร้อมอินพุตที่มีขอบเขตจำกัด) ต้องเลือกเปิดใช้คุณสมบัตินี้ผ่าน `gateway.controlUi.toolTitles` (ค่าเริ่มต้นปิด) Gateway ที่ปิดใช้งานจะตอบ `{ titles: {}, disabled: true }` โดยไม่เรียกโมเดล เพื่อให้ไคลเอนต์หยุดส่งคำขอ เมื่อเปิดใช้งาน ชื่อจะใช้การกำหนดเส้นทางโมเดลอรรถประโยชน์มาตรฐาน ได้แก่ `utilityModel` ที่กำหนดค่าไว้อย่างชัดเจน (เป็นการตัดสินใจของผู้ดำเนินการ ซึ่งเช่นเดียวกับงานอรรถประโยชน์ทั้งหมด อาจส่งเนื้อหางานที่มีขอบเขตจำกัดไปยังผู้ให้บริการที่เลือก) มิฉะนั้นจะใช้ค่าเริ่มต้นโมเดลขนาดเล็กที่ผู้ให้บริการของเซสชันประกาศไว้ เพื่อไม่ให้เกิดปลายทางข้อมูลขาออกใหม่โดยปริยาย ส่วน `utilityModel` ที่ว่างเปล่าจะปิดใช้งานทั้งหมด ชื่อจะไม่ย้อนกลับไปใช้โมเดลหลัก ผลลัพธ์จะถูกแคชในฐานข้อมูลสถานะของแต่ละเอเจนต์โดยใช้ชื่อเครื่องมือ + อินพุตเป็นคีย์ ดังนั้นการดูซ้ำจะไม่คิดค่าบริการการเรียกเดิมอีก
    - `chat.send` ยอมรับ `fastMode: "auto"` แบบหนึ่งเทิร์นเพื่อใช้โหมดเร็วกับการเรียกโมเดลที่เริ่มก่อนจุดตัดอัตโนมัติ จากนั้นเริ่มการลองใหม่ การใช้ทางเลือกสำรอง การส่งผลลัพธ์เครื่องมือ หรือการเรียกต่อเนื่องในภายหลังโดยไม่ใช้โหมดเร็ว จุดตัดมีค่าเริ่มต้น 60 วินาที (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) และสามารถกำหนดค่าต่อโมเดลด้วย `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` ผู้เรียก `chat.send` สามารถส่ง `fastAutoOnSeconds` แบบหนึ่งเทิร์นเพื่อแทนที่จุดตัดสำหรับคำขอนั้น ส่ง `queueMode` (`steer`, `followup`, `collect` หรือ `interrupt`) เพื่อแทนที่โหมดคิวที่จัดเก็บไว้สำหรับคำขอนี้เท่านั้น การดำเนินการชี้นำอย่างชัดเจนใน Control UI ใช้ `queueMode: "steer"`

## RPC บัญชีแยกประเภทการตรวจสอบ

`audit.activity.list` มอบมุมมองที่คงที่และเรียงจากใหม่ที่สุดให้แก่ไคลเอนต์ผู้ดำเนินการสำหรับข้อมูลเมตา
วงจรชีวิตของการเรียกใช้เอเจนต์ การดำเนินการของเครื่องมือ และข้อความที่เลือกเข้าร่วม โดยต้องใช้
`operator.read` การสืบค้นจะไม่รวมระเบียนที่เก่ากว่า 30 วัน และบัญชีแยกประเภท SQLite ที่ใช้ร่วมกัน
จำกัดไว้ที่ 100,000 ระเบียน แถวที่หมดอายุจะถูกลบระหว่างการเริ่มต้น Gateway
การบำรุงรักษารายชั่วโมง และการเขียนครั้งต่อ ๆ ไป ดู
[ประวัติการตรวจสอบ](/th/gateway/audit)สำหรับโมเดลข้อมูลและความหมายด้านความเป็นส่วนตัว

- พารามิเตอร์: `agentId`, `sessionKey` หรือ `runId` ที่ตรงกันทุกประการและไม่บังคับ; `kind` ที่ไม่บังคับ
  (`"agent_run"`, `"tool_action"` หรือ `"message"`); `status` ที่ไม่บังคับ
  (`"started"`, `"succeeded"`, `"failed"`, `"cancelled"`, `"timed_out"`,
  `"blocked"` หรือ `"unknown"`); ข้อความ `direction` ที่ไม่บังคับ (`"inbound"` หรือ
  `"outbound"`) และ `channel` ที่ตรงกันทุกประการ; ขอบเขตเวลาแบบรวมค่าปลาย `after` / `before`
  หน่วยมิลลิวินาที Unix ที่ไม่บังคับ; `limit` ที่ไม่บังคับตั้งแต่ `1` ถึง `500`; และ
  สตริง `cursor` ที่ไม่บังคับจากหน้าก่อนหน้า
- ผลลัพธ์: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`

ยูเนียนผลลัพธ์ V1 ที่มีชื่อประกอบด้วยสคีมาแยกกันสำหรับการเรียกใช้เอเจนต์ การดำเนินการของเครื่องมือ ข้อความขาเข้า
และข้อความขาออก ตัวจำแนก `eventType` คือ
`agent_run`, `tool_action`, `inbound_message` หรือ `outbound_message` ตามลำดับ ส่วน `kind` และ
ข้อความ `direction` ยังคงใช้สำหรับการกรองและแสดงผลได้ ทุกเหตุการณ์มี
`schemaVersion: 1` ที่เป็นจำนวนเต็ม การอ้างอิงข้อมูลประจำตัวของข้อความใช้รูปแบบ
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` ที่ตรงกันทุกประการ ID ของผู้กระทำที่เป็นผู้ส่งจากช่องทาง
ใช้รูปแบบเดียวกัน

ตัวแปรทั้งหมดกำหนดให้มี `eventType`, `schemaVersion`, `eventId`, `sequence`,
`sourceSequence`, `occurredAt`, `kind`, `action`, `status`, `actor` และ
`redaction` ฟิลด์ของแต่ละตัวแปรมีดังนี้:

| `eventType`        | ฟิลด์ที่จำเป็น                                                   | ฟิลด์ที่ไม่บังคับ                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`, `runId`; `kind: "agent_run"`                           | `sessionKey`, `sessionId`, `errorCode`                                                                                          |
| `tool_action`      | `agentId`, `runId`; `kind: "tool_action"`                         | `sessionKey`, `sessionId`, `toolCallId`, `toolName`, `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`, `channel`, `conversationKind`, `outcome`  | `agentId`, `runId`, `durationMs`, `resultCount`, การอ้างอิงข้อมูลประจำตัว, `reasonCode`, `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`, `channel`, `conversationKind`, `outcome` | `agentId`, `runId`, `durationMs`, `resultCount`, การอ้างอิงข้อมูลประจำตัว, `reasonCode`, `deliveryKind`, `failureStage`, `errorCode` |

enum แบบปิดของข้อความมีดังนี้:

- `conversationKind`: `direct`, `group`, `channel` หรือ `unknown`
- `outcome` ขาเข้า: `completed`, `skipped` หรือ `failed`; `reasonCode` ที่ไม่บังคับ:
  `duplicate`, `reply_operation_active`,
  `reply_operation_aborted`, `fast_abort`, `plugin_bound_handled`,
  `plugin_bound_unavailable`, `plugin_bound_declined`, `plugin_bound_error`,
  `before_dispatch_handled`, `acp_dispatch_completed`, `acp_dispatch_failed`,
  `acp_dispatch_empty` หรือ `acp_dispatch_aborted`
- `outcome` ขาออก: `sent`, `suppressed`, `failed` หรือ `unknown`; `reasonCode` ที่ไม่บังคับ:
  `cancelled_by_message_sending_hook`,
  `cancelled_by_reply_payload_sending_hook`,
  `empty_after_message_sending_hook`, `empty_after_reply_payload_sending_hook`
  หรือ `no_visible_payload` อะแดปเตอร์ที่ไม่ส่งคืนข้อมูลประจำตัวของแพลตฟอร์มจะเป็น
  `unknown` เนื่องจากไม่สามารถพิสูจน์ได้ว่าไม่มีผลข้างเคียงภายนอกเกิดขึ้น
- `deliveryKind`: `text`, `media` หรือ `other`; `failureStage`:
  `platform_send`, `queue` หรือ `unknown`

ฟิลด์สถานะสิ้นสุดมีความสัมพันธ์กัน ไม่ใช่ฟิลด์ที่ไม่บังคับอย่างเป็นอิสระ:

| รูปแบบ           | การแมปสถานะสิ้นสุด                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| การรันเอเจนต์        | `started` ไม่มี `errorCode`; ทุกสถานะสิ้นสุดที่ไม่สำเร็จต้องมีรหัส `run_*` ที่ตรงกัน                                                                 |
| การดำเนินการของเครื่องมือ      | `started` และสถานะสำเร็จไม่มี `errorCode`; สถานะสิ้นสุดอื่นทุกสถานะต้องมีรหัส `tool_*` ที่ตรงกัน                                                       |
| ข้อความขาเข้า  | สำเร็จ = `completed`; ถูกบล็อก = `skipped`; ล้มเหลว = `failed` พร้อม `message_processing_failed` เมื่อมี `reasonCode` ค่านี้ต้องอยู่ในกลุ่มสถานะสิ้นสุดนั้น |
| ข้อความขาออก | สำเร็จ = `sent`; ถูกบล็อก = `suppressed` พร้อม `reasonCode`; ล้มเหลว = `failed` พร้อม `errorCode` และ `failureStage`; ไม่ทราบ = `unknown` พร้อม `failureStage`      |

แต่ละเหตุการณ์กิจกรรมประกอบด้วยรหัสเหตุการณ์ที่คงที่ ลำดับบัญชีแยกประเภทที่เพิ่มขึ้นอย่างต่อเนื่อง
ลำดับเหตุการณ์ต้นทาง การประทับเวลา ผู้กระทำ การดำเนินการ สถานะ จำนวนเต็ม
`schemaVersion: 1` และ `redaction: "metadata_only"` ระเบียนการรันและเครื่องมือ
ต้องมีข้อมูลที่มาของเอเจนต์และการรัน และอาจมีข้อมูลที่มาของเซสชัน ระเบียนข้อความ
อาจมีรหัสเอเจนต์และรหัสการรัน แต่ตั้งใจไม่รวม
`sessionKey` หรือ `sessionId` ดังนั้นตัวกรองคำค้น `sessionKey` จึงใช้กับ
แถวการรันและเครื่องมือเท่านั้น เหตุการณ์เครื่องมืออาจมีรหัสการเรียกเครื่องมือและชื่อเครื่องมือ

ระเบียนข้อความใช้ `message.inbound.processed` หรือ
`message.outbound.finished` และเพิ่มทิศทาง ช่องทาง ประเภทการสนทนา
ผลลัพธ์ที่ปรับเป็นมาตรฐาน ตลอดจนประเภทการส่ง ขั้นตอนที่ล้มเหลว ระยะเวลา
จำนวนผลลัพธ์ รหัสเหตุผล และนามแฝงบัญชี/การสนทนา/ข้อความ/เป้าหมาย
แบบใช้คีย์เฉพาะการติดตั้งซึ่งเป็นข้อมูลเสริม นามแฝงเหล่านี้ช่วย
เชื่อมโยงข้อมูลแต่ไม่ใช่การทำให้เป็นนิรนาม เนื่องจากฐานข้อมูลสถานะมีคีย์ของนามแฝงเหล่านี้
แต่การส่งออกผ่าน RPC และ CLI ไม่มี บัญชีแยกประเภทไม่จัดเก็บพรอมต์ เนื้อหา
ข้อความ อาร์กิวเมนต์เครื่องมือ ผลลัพธ์เครื่องมือ เอาต์พุตคำสั่ง หรือข้อความข้อผิดพลาดดิบ
ค่า `sessionKey` ของการรัน/เครื่องมือยังคงเป็นข้อมูลเมตาการเชื่อมโยงแบบดิบและอาจฝัง
รหัสบัญชีหรือรหัสเพียร์ของแพลตฟอร์ม ระเบียนข้อความจะไม่รวมคีย์เซสชัน

สำหรับแถวขาเข้า `durationMs` วัดการส่งต่อของแกนหลักจนถึงสถานะสิ้นสุด และ
`resultCount` นับเพย์โหลดเครื่องมือ การบล็อก และการตอบกลับในคิวที่ได้รับการสรุปผลแล้ว สำหรับ
แถวขาออก `durationMs` ครอบคลุมตั้งแต่การรับผิดชอบการส่งจนถึงการตอบรับ
จดหมายที่ส่งไม่สำเร็จ หรือการกระทบยอด (รวมเวลารอในคิว) และ `resultCount`
นับการส่งจริงที่ระบุได้บนแพลตฟอร์ม เมื่อมี `deliveryKind`
ค่านี้จะอธิบายเพย์โหลดที่มีผลหลังผ่านฮุกและการเรนเดอร์ แถวที่ถูกระงับหรือ
มีความกำกวมจากการขัดข้องจะไม่รวมค่านี้

ความครอบคลุมของข้อความในปัจจุบันรวมข้อความขาเข้าที่ได้รับการยอมรับและไปถึงการส่งต่อ
ของแกนหลัก รวมถึงผลลัพธ์การทำซ้ำ/สถานะสิ้นสุดของแกนหลัก ความครอบคลุมขาออกจะเขียน
หนึ่งแถวสถานะสิ้นสุดต่อเพย์โหลดการตอบกลับเชิงตรรกะต้นฉบับที่ไปถึงการส่งแบบคงทน
ที่ใช้ร่วมกัน การแบ่งส่วนและการกระจายต่อของอะแดปเตอร์จะถูกรวมไว้ใน `resultCount` การส่ง
ที่อยู่ในคิวและลองใหม่ได้หรือมีความกำกวมจะถูกบันทึกหลังการตอบรับ จดหมายที่ส่งไม่สำเร็จ
หรือการกระทบยอดเท่านั้น เส้นทางเฉพาะ Plugin และเส้นทางส่งตรงที่ข้ามขอบเขต
ที่ใช้ร่วมกันเหล่านั้นยังไม่ครอบคลุม คิวเวิร์กเกอร์ที่มีขอบเขตจำกัดทำงานแบบพยายามให้ดีที่สุด
และอาจทิ้งระเบียนเมื่อล้มเหลวหรืออิ่มตัว ดังนั้นส่วนนี้จึงไม่ใช่
คลังข้อมูลการปฏิบัติตามข้อกำหนดที่ไม่สูญเสียข้อมูล

การบันทึกเปิดอยู่โดยค่าเริ่มต้นและควบคุมด้วย
[`audit.enabled`](/th/gateway/configuration-reference#audit) การบันทึกข้อความ
ควบคุมแยกต่างหากด้วย `audit.messages` และมีค่าเริ่มต้นเป็น `"off"` เมื่อ
ปิดใช้งานการบันทึก `audit.activity.list` จะยังให้บริการระเบียนที่เขียนไว้
ก่อนหน้านี้จนกว่าจะหมดอายุ

สคีมาคำขอ ผลลัพธ์ และ `AuditEvent` ของ `audit.list` ที่เผยแพร่แล้วยังคง
ไม่เปลี่ยนแปลงและส่งคืนเฉพาะระเบียนการรันเอเจนต์และการดำเนินการของเครื่องมือ ไคลเอนต์
ผู้ดำเนินการใหม่ควรเรียก `audit.activity.list` เมื่อ Gateway ประกาศว่ารองรับ Gateway
รุ่นเก่าอาจรายงาน `unknown method: audit.activity.list` หรือเนื่องจาก
การตรวจสอบสิทธิ์เกิดก่อนการค้นหาเมธอดในรุ่นที่เผยแพร่แล้ว จึงอาจรายงาน `missing scope:
operator.admin` ต่อคำขอที่มีขอบเขตการอ่าน ให้ถือว่ากรณีหลังหมายถึงไม่มีเมธอด
เฉพาะเมื่อไม่มีการประกาศเมธอดนั้น จากนั้นไคลเอนต์อาจลอง `audit.list` ใหม่
เฉพาะเมื่อตัวกรองไม่ต้องการรองรับประเภทข้อความ ทิศทาง หรือช่องทาง

ใช้ [`openclaw audit`](/th/cli/audit) สำหรับคำค้นแบบข้อความและการส่งออก JSON ที่มีขอบเขตจำกัด

## RPC ของบัญชีแยกประเภทงาน

ไคลเอนต์ผู้ดำเนินการตรวจสอบและยกเลิกระเบียนงานเบื้องหลังของ Gateway ผ่าน
RPC ของบัญชีแยกประเภทงาน (`packages/gateway-protocol/src/schema/tasks.ts`) RPC เหล่านี้
ส่งคืนสรุปงานที่ผ่านการล้างข้อมูลแล้ว ไม่ใช่สถานะรันไทม์ดิบ

- `tasks.list` ต้องใช้ `operator.read`
  - พารามิเตอร์: `status` ซึ่งเป็นค่าที่ไม่บังคับ (`"queued"`, `"running"`, `"completed"`,
    `"failed"`, `"cancelled"` หรือ `"timed_out"`) หรืออาร์เรย์ของสถานะเหล่านั้น
    `agentId` ซึ่งเป็นค่าที่ไม่บังคับ, `sessionKey` ซึ่งเป็นค่าที่ไม่บังคับ, `limit` ซึ่งเป็นค่าที่ไม่บังคับและมีค่าตั้งแต่ `1` ถึง
    `500` และสตริง `cursor` ซึ่งเป็นค่าที่ไม่บังคับ
  - ผลลัพธ์: `{ "tasks": TaskSummary[], "nextCursor"?: string }`
- `tasks.get` ต้องใช้ `operator.read`
  - พารามิเตอร์: `{ "taskId": string }`
  - ผลลัพธ์: `{ "task": TaskSummary }`
  - รหัสงานที่ไม่มีอยู่จะส่งคืนรูปแบบข้อผิดพลาดไม่พบของ Gateway
- `tasks.cancel` ต้องใช้ `operator.write`
  - พารามิเตอร์: `{ "taskId": string, "reason"?: string }`
  - ผลลัพธ์: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`
  - `found` รายงานว่าบัญชีแยกประเภทมีงานที่ตรงกันหรือไม่ `cancelled`
    รายงานว่ารันไทม์ยอมรับหรือบันทึกการยกเลิกหรือไม่

`TaskSummary` ประกอบด้วย `id`, `status` และข้อมูลเมตาที่ไม่บังคับ ได้แก่ `kind`,
`runtime`, `title`, `agentId`, `sessionKey`, `childSessionKey`, `ownerKey`,
`runId`, `taskId`, `flowId`, `parentTaskId`, `sourceId`, การประทับเวลา ความคืบหน้า
สรุปสถานะสิ้นสุด และข้อความข้อผิดพลาดที่ผ่านการล้างข้อมูลแล้ว `agentId` ระบุเอเจนต์
ที่ดำเนินงาน ส่วน `sessionKey` และ `ownerKey` จะเก็บรักษาบริบทของผู้ร้องขอและการควบคุม

## เมธอดตัวช่วยสำหรับผู้ดำเนินการ

- `commands.list` (`operator.read`) ดึงรายการคำสั่งรันไทม์สำหรับ
  เอเจนต์
  - `agentId` เป็นตัวเลือกเสริม หากละไว้ ระบบจะอ่านพื้นที่ทำงานของเอเจนต์เริ่มต้น
  - `scope` ควบคุมว่า `name` หลักจะกำหนดเป้าหมายไปยังพื้นผิวใด: `text` จะส่งคืน
    โทเค็นคำสั่งข้อความหลักโดยไม่มี `/` นำหน้า ส่วน `native` และเส้นทาง
    `both` เริ่มต้นจะส่งคืนชื่อเนทีฟที่รับรู้ผู้ให้บริการเมื่อมี
  - `textAliases` มีนามแฝงแบบสแลชที่ตรงกันทุกประการ เช่น `/model` และ `/m`
  - `nativeName` มีชื่อคำสั่งเนทีฟที่รับรู้ผู้ให้บริการเมื่อ
    มีชื่อดังกล่าว
  - `provider` เป็นตัวเลือกเสริมและมีผลเฉพาะต่อการตั้งชื่อเนทีฟ รวมถึงความพร้อมใช้งานของคำสั่ง Plugin
    แบบเนทีฟ
  - `includeArgs=false` จะละข้อมูลเมตาของอาร์กิวเมนต์ที่ผ่านการซีเรียลไลซ์ออกจากการตอบกลับ
- `tools.catalog` (`operator.read`) ดึงแค็ตตาล็อกเครื่องมือรันไทม์สำหรับ
  เอเจนต์ การตอบกลับประกอบด้วยเครื่องมือที่จัดกลุ่มและข้อมูลเมตาแหล่งที่มา:
  - `source`: `core` หรือ `plugin`
  - `pluginId`: เจ้าของ Plugin เมื่อเป็น `source="plugin"`
  - `optional`: ระบุว่าเครื่องมือของ Plugin เป็นตัวเลือกเสริมหรือไม่
- `tools.effective` (`operator.read`) ดึงรายการเครื่องมือที่มีผลจริงในรันไทม์
  สำหรับเซสชัน
  - `sessionKey` จำเป็นต้องระบุ
  - Gateway จะอนุมานบริบทรันไทม์ที่เชื่อถือได้จากเซสชันทางฝั่งเซิร์ฟเวอร์
    แทนการยอมรับบริบทการยืนยันตัวตนหรือการส่งมอบที่ผู้เรียกส่งมา
  - การตอบกลับเป็นมุมมองของรายการที่ใช้งานอยู่ซึ่งจำกัดขอบเขตตามเซสชันและได้มาจากเซิร์ฟเวอร์
    รวมถึงเครื่องมือหลัก เครื่องมือจาก Plugin เครื่องมือของช่องทาง และเครื่องมือจากเซิร์ฟเวอร์ MCP
    ที่ค้นพบแล้ว
  - `tools.effective` เป็นแบบอ่านอย่างเดียวสำหรับ MCP: สามารถฉายแค็ตตาล็อก MCP
    ของเซสชันที่วอร์มแล้วผ่านนโยบายเครื่องมือขั้นสุดท้าย แต่จะไม่สร้างรันไทม์ MCP
    เชื่อมต่อการขนส่ง หรือออก `tools/list` หากไม่มีแค็ตตาล็อกที่วอร์มแล้วซึ่งตรงกัน
    การตอบกลับอาจมีข้อความแจ้ง เช่น `mcp-not-yet-connected`,
    `mcp-not-yet-listed` หรือ `mcp-stale-catalog`
  - รายการเครื่องมือที่มีผลจริงใช้ `source="core"`, `source="plugin"`,
    `source="channel"` หรือ `source="mcp"`
- `tools.invoke` (`operator.write`) เรียกใช้เครื่องมือที่พร้อมใช้งานหนึ่งรายการผ่าน
  เส้นทางนโยบาย Gateway เดียวกับ `/tools/invoke`
  - `name` จำเป็นต้องระบุ ส่วน `args`, `sessionKey`, `agentId`, `confirm` และ
    `idempotencyKey` เป็นตัวเลือกเสริม
  - หากมีทั้ง `sessionKey` และ `agentId` เอเจนต์ของเซสชันที่แก้ไขได้
    ต้องตรงกับ `agentId`
  - ตัวห่อหุ้มหลักสำหรับเจ้าของเท่านั้น เช่น `cron`, `gateway` และ `nodes` ต้องใช้
    ข้อมูลประจำตัวของเจ้าของ/ผู้ดูแลระบบ (`operator.admin`) แม้ว่า `tools.invoke` เอง
    จะเป็น `operator.write`
  - การตอบกลับเป็นเอนเวโลปสำหรับ SDK ที่มี `ok`, `toolName`, `output`
    ซึ่งเป็นตัวเลือกเสริม และฟิลด์ `error` ที่มีชนิด การปฏิเสธจากการอนุมัติหรือนโยบายจะส่งคืน
    `ok:false` ในเพย์โหลด แทนการข้ามไปป์ไลน์นโยบายเครื่องมือของ Gateway
- `skills.status` (`operator.read`) ดึงรายการ Skills ที่มองเห็นได้สำหรับ
  เอเจนต์
  - `agentId` เป็นตัวเลือกเสริม หากละไว้ ระบบจะอ่านพื้นที่ทำงานของเอเจนต์เริ่มต้น
  - การตอบกลับประกอบด้วยคุณสมบัติการมีสิทธิ์ ข้อกำหนดที่ขาดหาย การตรวจสอบการกำหนดค่า
    และตัวเลือกการติดตั้งที่ผ่านการกรอง โดยไม่เปิดเผยค่าความลับดิบ
- `skills.search` และ `skills.detail` (`operator.read`) ส่งคืนข้อมูลเมตา
  การค้นพบของ ClawHub
- `skills.upload.begin`, `skills.upload.chunk` และ `skills.upload.commit`
  (`operator.admin`) จัดเตรียมไฟล์เก็บถาวร Skills ส่วนตัวก่อนติดตั้ง ขั้นตอนนี้
  เป็นเส้นทางอัปโหลดแยกต่างหากสำหรับผู้ดูแลระบบจากไคลเอนต์ที่เชื่อถือได้ ไม่ใช่ขั้นตอนติดตั้ง Skills
  ของ ClawHub ตามปกติ และจะปิดใช้งานโดยค่าเริ่มต้น เว้นแต่จะเปิดใช้
  `skills.install.allowUploadedArchives`
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })`
    สร้างการอัปโหลดที่ผูกกับ slug และค่าบังคับดังกล่าว
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })` ต่อท้ายไบต์ที่
    ออฟเซ็ตที่ถอดรหัสแล้วอย่างตรงกันทุกประการ
  - `skills.upload.commit({ uploadId, sha256? })` ตรวจสอบขนาดสุดท้ายและ
    SHA-256 การคอมมิตเพียงทำให้การอัปโหลดเสร็จสมบูรณ์เท่านั้น โดยไม่ได้ติดตั้ง Skills
  - ไฟล์เก็บถาวร Skills ที่อัปโหลดเป็นไฟล์ zip ซึ่งมีราก `SKILL.md` ชื่อไดเรกทอรี
    ภายในไฟล์เก็บถาวรจะไม่ใช้เลือกเป้าหมายการติดตั้ง
- `skills.install` (`operator.admin`) มีสามโหมด:
  - โหมด ClawHub: `{ source: "clawhub", slug, version?, force? }` ติดตั้ง
    โฟลเดอร์ Skills ลงในไดเรกทอรี `skills/` ของพื้นที่ทำงานเอเจนต์เริ่มต้น
  - โหมดอัปโหลด: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }`
    ติดตั้งการอัปโหลดที่คอมมิตแล้วลงในไดเรกทอรี `skills/<slug>`
    ของพื้นที่ทำงานเอเจนต์เริ่มต้น slug และค่าบังคับต้องตรงกับคำขอ
    `skills.upload.begin` เดิม ระบบจะปฏิเสธเว้นแต่จะเปิดใช้
    `skills.install.allowUploadedArchives` การตั้งค่านี้ไม่มีผลต่อการติดตั้งจาก ClawHub
  - โหมดตัวติดตั้ง Gateway: `{ name, installId, timeoutMs? }` เรียกใช้การดำเนินการ
    `metadata.openclaw.install` ที่ประกาศไว้บนโฮสต์ Gateway ไคลเอนต์รุ่นเก่าอาจ
    ยังคงส่ง `dangerouslyForceUnsafeInstall` ฟิลด์นี้เลิกใช้งานแล้ว
    ยอมรับไว้เพื่อความเข้ากันได้ของโปรโตคอลเท่านั้น และจะถูกเพิกเฉย ใช้
    `security.installPolicy` สำหรับการตัดสินใจติดตั้งที่ผู้ปฏิบัติงานเป็นเจ้าของ
- `skills.update` (`operator.admin`) มีสองโหมด:
  - โหมด ClawHub อัปเดต slug ที่ติดตามหนึ่งรายการหรือการติดตั้งจาก ClawHub
    ที่ติดตามทั้งหมดในพื้นที่ทำงานเอเจนต์เริ่มต้น
  - โหมดการกำหนดค่าแก้ไขค่า `skills.entries.<skillKey>` เช่น `enabled`,
    `apiKey` และ `env`

### มุมมอง `models.list`

`models.list` ยอมรับพารามิเตอร์ `view` ซึ่งเป็นตัวเลือกเสริม
(`src/agents/model-catalog-visibility.ts`):

- ละไว้หรือเป็น `"default"`: หากกำหนดค่า `agents.defaults.modelPolicy.allow` ไว้
  การตอบกลับจะเป็นแค็ตตาล็อกที่อนุญาต รวมถึงโมเดลที่ค้นพบแบบไดนามิก
  สำหรับรายการ `provider/*` มิฉะนั้น การตอบกลับจะเป็นแค็ตตาล็อก Gateway
  แบบเต็ม
- `"configured"`: ลักษณะการทำงานที่มีขนาดเหมาะกับตัวเลือก หากกำหนดค่า `agents.defaults.modelPolicy.allow`
  ไว้ ค่านี้ยังคงมีผลเหนือกว่า รวมถึงการค้นพบที่จำกัดขอบเขตตามผู้ให้บริการสำหรับ
  รายการ `provider/*` หากไม่มีรายการอนุญาต การตอบกลับจะใช้รายการ
  `models.providers.<provider>.models` ที่ระบุอย่างชัดเจน และจะย้อนกลับไปใช้แค็ตตาล็อกแบบเต็ม
  เฉพาะเมื่อไม่มีแถวโมเดลที่กำหนดค่าไว้
- `"provider-config"`: รายการ `models.providers.*.models` ที่กำหนดโดยแหล่งที่มา
  ซึ่งไม่ขึ้นกับรายการอนุญาตของตัวเลือก แถวต่าง ๆ มีความสามารถสาธารณะของโมเดลและ
  ความพร้อมใช้งานที่รับรู้เส้นทาง แต่จะละเอ็นด์พอยต์ของผู้ให้บริการ ข้อมูลการยืนยันตัวตน และ
  การกำหนดค่าคำขอรันไทม์
- `"all"`: แค็ตตาล็อก Gateway แบบเต็ม โดยข้าม `agents.defaults.modelPolicy.allow` ใช้สำหรับ
  UI การวินิจฉัย/การค้นพบ ไม่ใช่ตัวเลือกโมเดลทั่วไป

## การอนุมัติการดำเนินการ

- เมื่อคำขอดำเนินการต้องได้รับการอนุมัติ Gateway จะกระจาย
  `exec.approval.requested`
- ไคลเอนต์ของผู้ปฏิบัติงานดำเนินการตัดสินโดยเรียก `exec.approval.resolve` (ต้องมี
  `operator.approvals`)
- สำหรับ `host=node` นั้น `exec.approval.request` ต้องมี `systemRunPlan`
  (ข้อมูลเมตา `argv`/`cwd`/`rawCommand`/เซสชันแบบบัญญัติ) คำขอที่ไม่มี
  `systemRunPlan` จะถูกปฏิเสธ
- หลังการอนุมัติ การเรียก `node.invoke system.run` ที่ส่งต่อจะนำ
  `systemRunPlan` แบบบัญญัตินั้นกลับมาใช้เป็นบริบทคำสั่ง/cwd/เซสชันที่มีอำนาจตัดสิน
- หากผู้เรียกเปลี่ยนแปลง `command`, `rawCommand`, `cwd`, `agentId` หรือ
  `sessionKey` ระหว่างการเตรียมและการส่งต่อ `system.run` ที่ได้รับอนุมัติขั้นสุดท้าย
  Gateway จะปฏิเสธการเรียกใช้แทนการเชื่อถือเพย์โหลดที่ถูกเปลี่ยนแปลง

## การย้อนกลับของการส่งมอบจากเอเจนต์

- คำขอ `agent` สามารถมี `deliver=true` เพื่อร้องขอการส่งมอบขาออก
- `bestEffortDeliver=false` (ค่าเริ่มต้น) คงลักษณะการทำงานที่เข้มงวด: เป้าหมายการส่งมอบที่แก้ไขไม่ได้หรือ
  ใช้ภายในเท่านั้นจะส่งคืน `INVALID_REQUEST`
- `bestEffortDeliver=true` อนุญาตให้ย้อนกลับไปดำเนินการเฉพาะในเซสชัน เมื่อไม่สามารถ
  ระบุเส้นทางที่ส่งมอบภายนอกได้ (เช่น เซสชันภายใน/เว็บแชต
  หรือการกำหนดค่าหลายช่องทางที่กำกวม)
- ผลลัพธ์ `agent` ขั้นสุดท้ายอาจมี `result.deliveryStatus` เมื่อมีการร้องขอ
  การส่งมอบ โดยใช้สถานะ `sent`, `suppressed`, `partial_failed` และ
  `failed` เดียวกับที่บันทึกไว้สำหรับ
  [`openclaw agent --json --deliver`](/th/cli/agent#json-delivery-status)

## การกำหนดเวอร์ชัน

- `PROTOCOL_VERSION`, `MIN_CLIENT_PROTOCOL_VERSION`,
  `MIN_NODE_PROTOCOL_VERSION` และ `MIN_PROBE_PROTOCOL_VERSION` อยู่ใน
  `packages/gateway-protocol/src/version.ts`
- ไคลเอนต์ส่ง `minProtocol` + `maxProtocol` ไคลเอนต์ของผู้ปฏิบัติงานและ UI ต้อง
  รวมโปรโตคอลปัจจุบันไว้ในช่วงดังกล่าว ไคลเอนต์และเซิร์ฟเวอร์ปัจจุบันใช้
  โปรโตคอล v4
- ไคลเอนต์ที่ยืนยันตัวตนแล้วและมีทั้ง `role: "node"` และ `client.mode: "node"`
  สามารถใช้โปรโตคอล Node รุ่น N-1 (ปัจจุบันคือ v3) โพรบรีสตาร์ตแบบเบาใช้
  ช่วง N-1 เดียวกัน การยืนยันตัวตนอุปกรณ์ การจับคู่ ขอบเขต นโยบายคำสั่ง และการอนุมัติ
  การดำเนินการจะไม่เปลี่ยนแปลงจากช่วงความเข้ากันได้นี้ ความสามารถและคำสั่งของ Node
  ที่ Plugin เป็นเจ้าของจะถูกระงับไว้จนกว่า Node จะอัปเกรดเป็นโปรโตคอลปัจจุบัน
  เนื่องจากพื้นผิวที่โฮสต์โดยรายการเหล่านั้นไม่ได้เป็นส่วนหนึ่งของสัญญา N-1
- สคีมาและโมเดลสร้างขึ้นจากคำนิยาม TypeBox:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

## การยืนยันตัวตน

- การยืนยันตัวตน Gateway ด้วยความลับร่วมใช้ `connect.params.auth.token` หรือ
  `connect.params.auth.password` ขึ้นอยู่กับ
  `gateway.auth.mode` ที่กำหนดค่าไว้ (`"none" | "token" | "password" | "trusted-proxy"`)
- โหมดที่มีข้อมูลประจำตัว เช่น Tailscale Serve (`gateway.auth.allowTailscale: true`)
  หรือ `gateway.auth.mode: "trusted-proxy"` ที่ไม่ใช่ลูปแบ็ก จะผ่านการตรวจสอบ
  การยืนยันตัวตนขณะเชื่อมต่อจากส่วนหัวคำขอแทน `connect.params.auth.*`
- `gateway.auth.mode: "none"` แบบทางเข้าที่เป็นส่วนตัวจะข้ามการยืนยันตัวตน
  ขณะเชื่อมต่อด้วยความลับร่วมทั้งหมด ห้ามเปิดเผยโหมดดังกล่าวบนทางเข้าสาธารณะ/ที่ไม่น่าเชื่อถือ
- หลังจากจับคู่ Gateway จะออกโทเค็นอุปกรณ์ที่จำกัดขอบเขตตามบทบาท +
  ขอบเขตของการเชื่อมต่อ ซึ่งส่งคืนใน `hello-ok.auth.deviceToken` ไคลเอนต์ควร
  เก็บโทเค็นนี้ไว้หลังการเชื่อมต่อสำเร็จทุกครั้ง
- การเชื่อมต่อใหม่ด้วยโทเค็นอุปกรณ์ที่จัดเก็บไว้นั้นควรใช้ชุด
  ขอบเขตที่อนุมัติและจัดเก็บไว้สำหรับโทเค็นดังกล่าวซ้ำด้วย วิธีนี้จะรักษาสิทธิ์การอ่าน/ตรวจสอบ/สถานะ
  ที่ให้ไว้แล้ว และหลีกเลี่ยงไม่ให้การเชื่อมต่อใหม่ถูกลดเหลือขอบเขตเฉพาะผู้ดูแลระบบโดยปริยาย
  ที่แคบกว่าโดยไม่มีการแจ้ง
- การประกอบข้อมูลยืนยันตัวตนสำหรับการเชื่อมต่อฝั่งไคลเอนต์ (`selectConnectAuth` ใน
  `packages/gateway-client/src/client.ts`):
  - `auth.password` เป็นอิสระจากส่วนอื่นและจะถูกส่งต่อเสมอเมื่อมีการตั้งค่า
  - `auth.token` จะถูกเติมตามลำดับความสำคัญ: โทเค็นร่วมที่ระบุชัดเจนก่อน
    จากนั้น `deviceToken` ที่ระบุชัดเจน แล้วจึงเป็นโทเค็นต่ออุปกรณ์ที่จัดเก็บไว้ (กำหนดคีย์โดย
    `deviceId` + `role`)
  - `auth.bootstrapToken` จะถูกส่งเฉพาะเมื่อไม่มีรายการใดข้างต้นกำหนด
    `auth.token` ได้ โทเค็นร่วมหรือโทเค็นอุปกรณ์ใด ๆ ที่กำหนดได้จะระงับการส่งรายการนี้
  - การเลื่อนระดับโทเค็นอุปกรณ์ที่จัดเก็บไว้โดยอัตโนมัติในการลองใหม่แบบครั้งเดียว
    `AUTH_TOKEN_MISMATCH` จะอนุญาตเฉพาะปลายทางที่เชื่อถือได้เท่านั้น ได้แก่ ลูปแบ็ก
    หรือ `wss://` ที่ตรึง `tlsFingerprint` ไว้ ส่วน `wss://` สาธารณะที่ไม่มีการตรึง
    จะไม่เข้าเกณฑ์
- การบูตสแตรปด้วยรหัสตั้งค่าในตัวจะส่งคืน
  `hello-ok.auth.deviceToken` ของ Node หลัก พร้อมโทเค็นผู้ปฏิบัติงานแบบมีขอบเขตใน
  `hello-ok.auth.deviceTokens` สำหรับการส่งต่อไปยังอุปกรณ์เคลื่อนที่ที่เชื่อถือได้ โทเค็นผู้ปฏิบัติงาน
  มี `operator.talk.secrets` สำหรับอ่านการกำหนดค่า Talk แบบเนทีฟ แต่
  ไม่รวมขอบเขตการแก้ไขการจับคู่และ `operator.admin`
- ขณะที่การบูตสแตรปด้วยรหัสตั้งค่าที่ไม่ใช่ค่าพื้นฐานกำลังรอการอนุมัติ
  รายละเอียด `PAIRING_REQUIRED` จะมี `recommendedNextStep: "wait_then_retry"`,
  `retryable: true` และ `pauseReconnect: false` ให้เชื่อมต่อใหม่ต่อไปด้วย
  โทเค็นบูตสแตรปเดิมจนกว่าคำขอจะได้รับการอนุมัติหรือโทเค็นใช้การไม่ได้
- เก็บ `hello-ok.auth.deviceTokens` ไว้เฉพาะเมื่อการเชื่อมต่อใช้การยืนยันตัวตน
  แบบบูตสแตรปบนการรับส่งข้อมูลที่เชื่อถือได้ เช่น `wss://` หรือการจับคู่แบบลูปแบ็ก/ภายในเครื่อง
- หากไคลเอนต์ระบุ `deviceToken` หรือ `scopes` อย่างชัดเจน ชุด
  ขอบเขตที่ผู้เรียกร้องขอนั้นยังคงเป็นข้อกำหนดหลัก ขอบเขตที่แคชไว้จะถูก
  ใช้ซ้ำเฉพาะเมื่อไคลเอนต์ใช้โทเค็นต่ออุปกรณ์ที่จัดเก็บไว้ซ้ำ
- โทเค็นอุปกรณ์สามารถหมุนเวียน/เพิกถอนได้ผ่าน `device.token.rotate` และ
  `device.token.revoke` (ต้องมี `operator.pairing`) การหมุนเวียนหรือเพิกถอน
  โทเค็นของ Node หรือบทบาทอื่นที่ไม่ใช่ผู้ปฏิบัติงานต้องมี `operator.admin` ด้วย
- `device.token.rotate` ส่งคืนข้อมูลเมตาของการหมุนเวียน โดยจะส่งโทเค็น
  bearer ทดแทนกลับมาเฉพาะการเรียกจากอุปกรณ์เดียวกันที่ยืนยันตัวตนด้วยโทเค็นอุปกรณ์นั้นแล้ว
  เพื่อให้ไคลเอนต์ที่ใช้เฉพาะโทเค็นสามารถเก็บโทเค็นทดแทนไว้ก่อน
  เชื่อมต่อใหม่ การหมุนเวียนด้วยโทเค็นร่วม/ผู้ดูแลระบบจะไม่ส่งโทเค็น bearer กลับมา
- การออก การหมุนเวียน และการเพิกถอนโทเค็นยังคงจำกัดอยู่ภายในชุดบทบาทที่อนุมัติ
  และบันทึกไว้ในรายการจับคู่ของอุปกรณ์นั้น การแก้ไขโทเค็นไม่สามารถขยายหรือ
  กำหนดเป้าหมายบทบาทอุปกรณ์ที่การอนุมัติการจับคู่ไม่เคยให้สิทธิ์ไว้
- สำหรับเซสชันโทเค็นของอุปกรณ์ที่จับคู่แล้ว การจัดการอุปกรณ์จะจำกัดอยู่ในขอบเขต
  ของตนเอง เว้นแต่ผู้เรียกจะมี `operator.admin` ด้วย ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบสามารถจัดการได้เฉพาะ
  โทเค็นผู้ปฏิบัติงานสำหรับรายการอุปกรณ์ของตนเองเท่านั้น การจัดการโทเค็นของ Node และบทบาทอื่น
  ที่ไม่ใช่ผู้ปฏิบัติงานทำได้เฉพาะผู้ดูแลระบบ แม้จะเป็นอุปกรณ์ของผู้เรียกเองก็ตาม
- `device.token.rotate` และ `device.token.revoke` ยังตรวจสอบชุดขอบเขต
  ของโทเค็นผู้ปฏิบัติงานเป้าหมายกับขอบเขตเซสชันปัจจุบันของผู้เรียก
  ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบไม่สามารถหมุนเวียนหรือเพิกถอนโทเค็นผู้ปฏิบัติงานที่มีขอบเขตกว้างกว่า
  ที่ตนมีอยู่แล้ว
- ความล้มเหลวในการยืนยันตัวตนประกอบด้วย `error.details.code` พร้อมคำแนะนำการกู้คืน:
  - `error.details.canRetryWithDeviceToken` (บูลีน)
  - `error.details.recommendedNextStep`: ค่าใดค่าหนึ่งจาก `retry_with_device_token`,
    `update_auth_configuration`, `update_auth_credentials`,
    `wait_then_retry`, `review_auth_configuration`
    (`packages/gateway-protocol/src/connect-error-details.ts`)
- พฤติกรรมของไคลเอนต์สำหรับ `AUTH_TOKEN_MISMATCH`:
  - ไคลเอนต์ที่เชื่อถือได้อาจลองใหม่หนึ่งครั้งแบบมีขอบเขตโดยใช้โทเค็นต่ออุปกรณ์
    ที่แคชไว้
  - หากการลองใหม่นั้นล้มเหลว ให้หยุดวงรอบการเชื่อมต่อใหม่อัตโนมัติและแสดงคำแนะนำ
    เกี่ยวกับการดำเนินการของผู้ปฏิบัติงาน
- `AUTH_SCOPE_MISMATCH` หมายความว่าระบบรู้จักโทเค็นอุปกรณ์ แต่โทเค็นนั้นไม่
  ครอบคลุมบทบาท/ขอบเขตที่ร้องขอ อย่าแสดงกรณีนี้ว่าเป็นโทเค็นที่ไม่ถูกต้อง ให้แจ้ง
  ผู้ปฏิบัติงานให้จับคู่ใหม่หรืออนุมัติสัญญาขอบเขตที่แคบกว่า/กว้างกว่า

## ข้อมูลประจำตัวและการจับคู่อุปกรณ์

- Node ควรมีข้อมูลประจำตัวอุปกรณ์ที่คงที่ (`device.id`) ซึ่งได้มาจาก
  ลายนิ้วมือของคู่กุญแจ
- Gateway ออกโทเค็นแยกตามอุปกรณ์ + บทบาท
- รหัสอุปกรณ์ใหม่ต้องได้รับการอนุมัติการจับคู่ เว้นแต่เปิดใช้
  การอนุมัติอัตโนมัติภายในเครื่อง
- การอนุมัติการจับคู่โดยอัตโนมัติมุ่งเน้นการเชื่อมต่อโดยตรงผ่านลูปแบ็กภายในเครื่อง
- OpenClaw ยังมีเส้นทางการเชื่อมต่อตนเองแบบแคบภายในแบ็กเอนด์/คอนเทนเนอร์สำหรับ
  โฟลว์ตัวช่วยที่เชื่อถือได้ซึ่งใช้ความลับร่วม
- การเชื่อมต่อผ่าน tailnet หรือ LAN บนโฮสต์เดียวกันยังคงถือว่าเป็นการเชื่อมต่อระยะไกลสำหรับการจับคู่
  และต้องได้รับการอนุมัติ
- โดยปกติไคลเอนต์ WS จะใส่ข้อมูลประจำตัว `device` ระหว่าง `connect` (ผู้ปฏิบัติงาน +
  Node) ข้อยกเว้นเดียวสำหรับผู้ปฏิบัติงานที่ไม่มีอุปกรณ์คือเส้นทางความเชื่อถือที่ระบุชัดเจน:
  - `gateway.controlUi.allowInsecureAuth=true` สำหรับความเข้ากันได้กับ
    HTTP ที่ไม่ปลอดภัยเฉพาะ localhost
  - การยืนยันตัวตน Control UI ของผู้ปฏิบัติงานด้วย `gateway.auth.mode: "trusted-proxy"` สำเร็จ
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (ใช้ในกรณีฉุกเฉิน เป็นการลดระดับ
    ความปลอดภัยอย่างรุนแรง)
  - RPC แบ็กเอนด์ `gateway-client` แบบลูปแบ็กโดยตรงบนเส้นทางตัวช่วยภายใน
    ที่สงวนไว้
- การไม่ระบุข้อมูลประจำตัวอุปกรณ์ส่งผลต่อขอบเขต เมื่ออนุญาตการเชื่อมต่อ
  ของผู้ปฏิบัติงานที่ไม่มีอุปกรณ์ผ่านเส้นทางความเชื่อถือที่ระบุชัดเจน OpenClaw
  ยังคงล้างขอบเขตที่ประกาศเองให้เป็นชุดว่าง เว้นแต่เส้นทางนั้นจะมี
  ข้อยกเว้นการคงขอบเขตที่มีชื่อกำกับ จากนั้นเมธอดที่ควบคุมด้วยขอบเขตจะล้มเหลวด้วย
  `missing scope`
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` เป็นเส้นทาง
  การคงขอบเขตในกรณีฉุกเฉินของ Control UI โดยไม่ได้ให้ขอบเขตแก่
  ไคลเอนต์ WebSocket แบ็กเอนด์แบบกำหนดเองหรือรูปแบบ CLI โดยพลการ
- เส้นทางตัวช่วยแบ็กเอนด์ `gateway-client` แบบลูปแบ็กโดยตรงที่สงวนไว้จะคง
  ขอบเขตเฉพาะสำหรับ RPC ของระนาบควบคุมภายในเครื่องเท่านั้น รหัสแบ็กเอนด์แบบกำหนดเองจะ
  ไม่ได้รับข้อยกเว้นนี้
- การเชื่อมต่อทั้งหมดต้องลงนาม nonce `connect.challenge` ที่เซิร์ฟเวอร์ให้มา

### การวินิจฉัยการย้ายระบบยืนยันตัวตนอุปกรณ์

สำหรับไคลเอนต์รุ่นเก่าที่ยังคงใช้พฤติกรรมการลงนามก่อนการท้าทาย `connect`
จะส่งคืนรหัสรายละเอียด `DEVICE_AUTH_*` ภายใต้ `error.details.code` พร้อม
`error.details.reason` ที่คงที่

ความล้มเหลวทั่วไปในการย้ายระบบ:

| ข้อความ                     | details.code                     | details.reason           | ความหมาย                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | ไคลเอนต์ไม่ได้ระบุ `device.nonce` (หรือส่งค่าว่าง)     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | ไคลเอนต์ลงนามด้วย nonce ที่ล้าสมัย/ไม่ถูกต้อง            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | เพย์โหลดลายเซ็นไม่ตรงกับเพย์โหลด v2       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | การประทับเวลาที่ลงนามอยู่นอกช่วงความคลาดเคลื่อนที่อนุญาต          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` ไม่ตรงกับลายนิ้วมือของกุญแจสาธารณะ |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | รูปแบบ/การปรับให้อยู่ในรูปแบบมาตรฐานของกุญแจสาธารณะล้มเหลว         |

เป้าหมายการย้ายระบบ:

- รอ `connect.challenge` เสมอ
- ลงนามเพย์โหลด v2 ที่มี nonce ของเซิร์ฟเวอร์
- ส่ง nonce เดียวกันใน `connect.params.device.nonce`
- เพย์โหลดลายเซ็นที่แนะนำคือ `v3`
  (`buildDeviceAuthPayloadV3` ใน `packages/gateway-client/src/device-auth.ts`),
  ซึ่งผูก `platform` และ `deviceFamily` เพิ่มเติมจาก
  ฟิลด์อุปกรณ์/ไคลเอนต์/บทบาท/ขอบเขตสิทธิ์/โทเค็น/nonce
- ยังคงยอมรับลายเซ็น `v2` แบบเดิมเพื่อความเข้ากันได้ แต่การตรึง
  เมทาดาทาของอุปกรณ์ที่จับคู่แล้วยังคงควบคุมนโยบายคำสั่งเมื่อเชื่อมต่อใหม่

## TLS และการตรึง

- รองรับ TLS สำหรับการเชื่อมต่อ WS (การกำหนดค่า `gateway.tls`)
- ไคลเอนต์สามารถเลือกตรึงลายนิ้วมือใบรับรองของ Gateway ผ่าน
  `gateway.remote.tlsFingerprint` หรือ CLI `--tls-fingerprint`
