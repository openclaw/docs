OpenClaw สามารถสร้างไฟล์ zip วินิจฉัยภายในเครื่องสำหรับรายงานข้อบกพร่องได้ โดยรวม
สถานะ Gateway ที่ผ่านการทำให้ปลอดภัยแล้ว สุขภาพ บันทึก รูปทรงการตั้งค่า และเหตุการณ์เสถียรภาพล่าสุดที่ไม่มีเพย์โหลด

ปฏิบัติต่อชุดวินิจฉัยเหมือนความลับจนกว่าคุณจะตรวจสอบแล้ว ชุดเหล่านี้
ออกแบบมาให้ละเว้นหรือปกปิดเพย์โหลดและข้อมูลรับรอง แต่ยังคงสรุป
บันทึก Gateway ภายในเครื่องและสถานะรันไทม์ระดับโฮสต์

## เริ่มต้นอย่างรวดเร็ว

```bash
openclaw gateway diagnostics export
```

คำสั่งจะพิมพ์พาธไฟล์ zip ที่เขียนไว้ หากต้องการเลือกพาธ:

```bash
openclaw gateway diagnostics export --output openclaw-diagnostics.zip
```

สำหรับระบบอัตโนมัติ:

```bash
openclaw gateway diagnostics export --json
```

## คำสั่งแชต

เจ้าของสามารถใช้ `/diagnostics [note]` ในแชตเพื่อขอให้ส่งออก Gateway ภายในเครื่อง
ใช้สิ่งนี้เมื่อข้อบกพร่องเกิดขึ้นในการสนทนาจริง และคุณต้องการรายงานหนึ่งฉบับ
ที่คัดลอกและวางได้สำหรับฝ่ายสนับสนุน:

1. ส่ง `/diagnostics` ในการสนทนาที่คุณสังเกตเห็นปัญหา เพิ่ม
   หมายเหตุสั้นๆ หากช่วยได้ เช่น `/diagnostics bad tool choice`
2. OpenClaw จะส่งคำนำการวินิจฉัยและขอการอนุมัติ exec แบบชัดเจนหนึ่งครั้ง
   การอนุมัติจะรัน `openclaw gateway diagnostics export --json`
   อย่าอนุมัติการวินิจฉัยผ่านกฎอนุญาตทั้งหมด
3. หลังอนุมัติ OpenClaw จะตอบกลับด้วยรายงานที่วางได้ ซึ่งมีพาธชุดข้อมูลภายในเครื่อง,
   สรุป manifest, หมายเหตุความเป็นส่วนตัว และรหัสเซสชันที่เกี่ยวข้อง

ในแชตกลุ่ม เจ้าของยังคงรัน `/diagnostics` ได้ แต่ OpenClaw จะไม่
โพสต์รายละเอียดการวินิจฉัยกลับเข้าไปในแชตที่แชร์อยู่ ระบบจะส่งคำนำ,
พรอมป์อนุมัติ, ผลลัพธ์การส่งออก Gateway และรายละเอียดแยกย่อยของเซสชัน/เธรด Codex
ให้เจ้าของผ่านเส้นทางอนุมัติส่วนตัว กลุ่มจะได้รับเพียงประกาศสั้นๆ
ว่าโฟลว์การวินิจฉัยถูกส่งแบบส่วนตัวแล้ว หาก OpenClaw หาเส้นทางเจ้าของแบบส่วนตัว
ไม่พบ คำสั่งจะล้มเหลวแบบปิดและขอให้เจ้าของรันจาก DM

เมื่อเซสชัน OpenClaw ที่ใช้งานอยู่ใช้ฮาร์เนส OpenAI Codex แบบเนทีฟ
การอนุมัติ exec เดียวกันจะครอบคลุมการอัปโหลดฟีดแบ็ก OpenAI สำหรับเธรดรันไทม์
Codex ที่ OpenClaw รู้จักด้วย การอัปโหลดนั้นแยกจากไฟล์ zip Gateway ภายในเครื่อง
และปรากฏเฉพาะสำหรับเซสชันฮาร์เนส Codex ก่อนอนุมัติ พรอมป์จะอธิบายว่า
การอนุมัติการวินิจฉัยจะส่งฟีดแบ็ก Codex ด้วย แต่จะไม่แสดงรายการรหัสเซสชัน
หรือเธรด Codex หลังอนุมัติ การตอบกลับในแชตจะแสดงรายการช่องทาง, รหัสเซสชัน OpenClaw,
รหัสเธรด Codex และคำสั่ง resume ภายในเครื่องสำหรับเธรดที่ถูกส่งไปยังเซิร์ฟเวอร์ OpenAI
หากคุณปฏิเสธหรือเพิกเฉยต่อการอนุมัติ OpenClaw จะไม่รันการส่งออก ไม่ส่งฟีดแบ็ก Codex
และไม่พิมพ์รหัส Codex

สิ่งนี้ทำให้ลูปดีบัก Codex ทั่วไปสั้นลง: สังเกตพฤติกรรมที่ไม่ถูกต้องใน
Telegram, Discord หรือช่องทางอื่น รัน `/diagnostics` อนุมัติหนึ่งครั้ง แชร์
รายงานกับฝ่ายสนับสนุน แล้วรันคำสั่ง `codex resume <thread-id>` ที่พิมพ์ไว้
ภายในเครื่อง หากคุณต้องการตรวจสอบเธรด Codex แบบเนทีฟด้วยตนเอง ดู
[ฮาร์เนส Codex](/th/plugins/codex-harness#inspect-a-codex-thread-from-the-cli) สำหรับ
เวิร์กโฟลว์การตรวจสอบนั้น

## สิ่งที่การส่งออกมี

ไฟล์ zip ประกอบด้วย:

- `summary.md`: ภาพรวมที่มนุษย์อ่านได้สำหรับฝ่ายสนับสนุน
- `diagnostics.json`: สรุปที่เครื่องอ่านได้ของการตั้งค่า บันทึก สถานะ สุขภาพ
  และข้อมูลเสถียรภาพ
- `manifest.json`: เมทาดาทาการส่งออกและรายการไฟล์
- รูปทรงการตั้งค่าที่ผ่านการทำให้ปลอดภัยแล้วและรายละเอียดการตั้งค่าที่ไม่ใช่ความลับ
- สรุปบันทึกที่ผ่านการทำให้ปลอดภัยแล้วและบรรทัดบันทึกล่าสุดที่ถูกปกปิด
- สแนปช็อตสถานะและสุขภาพ Gateway แบบพยายามดีที่สุด
- `stability/latest.json`: ชุดข้อมูลเสถียรภาพที่คงไว้ล่าสุด เมื่อมีให้ใช้

การส่งออกมีประโยชน์แม้เมื่อ Gateway ไม่สมบูรณ์ หาก Gateway ไม่สามารถ
ตอบคำขอสถานะหรือสุขภาพได้ บันทึกภายในเครื่อง รูปทรงการตั้งค่า และ
ชุดข้อมูลเสถียรภาพล่าสุดจะยังถูกรวบรวมเมื่อมีให้ใช้

## โมเดลความเป็นส่วนตัว

การวินิจฉัยออกแบบมาให้แชร์ได้ การส่งออกจะเก็บข้อมูลการปฏิบัติงาน
ที่ช่วยในการดีบัก เช่น:

- ชื่อระบบย่อย, รหัส plugin, รหัสผู้ให้บริการ, รหัสช่องทาง และโหมดที่ตั้งค่าไว้
- รหัสสถานะ, ระยะเวลา, จำนวนไบต์, สถานะคิว และค่าการอ่านหน่วยความจำ
- เมทาดาทาบันทึกที่ผ่านการทำให้ปลอดภัยแล้วและข้อความการปฏิบัติงานที่ถูกปกปิด
- รูปทรงการตั้งค่าและการตั้งค่าฟีเจอร์ที่ไม่ใช่ความลับ

การส่งออกจะละเว้นหรือปกปิด:

- ข้อความแชต, พรอมป์, คำสั่ง, เนื้อหา webhook และผลลัพธ์เครื่องมือ
- ข้อมูลรับรอง, คีย์ API, โทเค็น, คุกกี้ และค่าลับ
- เนื้อหาคำขอหรือคำตอบแบบดิบ
- รหัสบัญชี, รหัสข้อความ, รหัสเซสชันแบบดิบ, ชื่อโฮสต์ และชื่อผู้ใช้ภายในเครื่อง

เมื่อข้อความบันทึกดูเหมือนข้อความเพย์โหลดของผู้ใช้ แชต พรอมป์ หรือเครื่องมือ
การส่งออกจะเก็บไว้เพียงว่ามีข้อความถูกละเว้นและจำนวนไบต์

## ตัวบันทึกเสถียรภาพ

Gateway จะบันทึกสตรีมเสถียรภาพแบบมีขอบเขตและไม่มีเพย์โหลดโดยค่าเริ่มต้นเมื่อ
เปิดใช้การวินิจฉัย สิ่งนี้มีไว้สำหรับข้อเท็จจริงด้านการปฏิบัติงาน ไม่ใช่เนื้อหา

Heartbeat การวินิจฉัยเดียวกันจะบันทึกตัวอย่างความมีชีวิตเมื่อ Gateway ยัง
ทำงานอยู่แต่ event loop ของ Node.js หรือ CPU ดูอิ่มตัว เหตุการณ์
`diagnostic.liveness.warning` เหล่านี้รวมถึงความล่าช้า event-loop, การใช้งาน
event-loop, อัตราส่วนคอร์ CPU และจำนวนเซสชันที่ active/waiting/queued ตัวอย่างที่ไม่ได้ใช้งาน
จะอยู่ใน telemetry ที่ระดับ `info` ตัวอย่างความมีชีวิตจะกลายเป็นคำเตือน Gateway
เฉพาะเมื่องานกำลังรอหรืออยู่ในคิว หรือเมื่องานที่ใช้งานอยู่ทับซ้อนกับ
ความล่าช้า event-loop ที่ต่อเนื่อง การพุ่งขึ้นของ max-delay ชั่วคราวระหว่างงานเบื้องหลัง
ที่ยังสุขภาพดีจะอยู่ในบันทึก debug เหตุการณ์เหล่านี้จะไม่รีสตาร์ต Gateway ด้วยตัวเอง

ตรวจสอบตัวบันทึกสด:

```bash
openclaw gateway stability
openclaw gateway stability --type payload.large
openclaw gateway stability --json
```

ตรวจสอบชุดข้อมูลเสถียรภาพที่คงไว้ล่าสุดหลังการออกแบบร้ายแรง, timeout ตอนปิดระบบ
หรือความล้มเหลวในการเริ่มต้นหลังรีสตาร์ต:

```bash
openclaw gateway stability --bundle latest
```

สร้างไฟล์ zip วินิจฉัยจากชุดข้อมูลที่คงไว้ล่าสุด:

```bash
openclaw gateway stability --bundle latest --export
```

ชุดข้อมูลที่คงไว้อยู่ใต้ `~/.openclaw/logs/stability/` เมื่อมีเหตุการณ์

## ตัวเลือกที่เป็นประโยชน์

```bash
openclaw gateway diagnostics export \
  --output openclaw-diagnostics.zip \
  --log-lines 5000 \
  --log-bytes 1000000
```

- `--output <path>`: เขียนไปยังพาธ zip ที่ระบุ
- `--log-lines <count>`: จำนวนบรรทัดบันทึกที่ผ่านการทำให้ปลอดภัยแล้วสูงสุดที่จะรวม
- `--log-bytes <bytes>`: จำนวนไบต์บันทึกสูงสุดที่จะตรวจสอบ
- `--url <url>`: URL WebSocket ของ Gateway สำหรับสแนปช็อตสถานะและสุขภาพ
- `--token <token>`: โทเค็น Gateway สำหรับสแนปช็อตสถานะและสุขภาพ
- `--password <password>`: รหัสผ่าน Gateway สำหรับสแนปช็อตสถานะและสุขภาพ
- `--timeout <ms>`: timeout ของสแนปช็อตสถานะและสุขภาพ
- `--no-stability-bundle`: ข้ามการค้นหาชุดข้อมูลเสถียรภาพที่คงไว้
- `--json`: พิมพ์เมทาดาทาการส่งออกที่เครื่องอ่านได้

## ปิดใช้การวินิจฉัย

การวินิจฉัยเปิดใช้งานโดยค่าเริ่มต้น หากต้องการปิดใช้ตัวบันทึกเสถียรภาพและ
การรวบรวมเหตุการณ์วินิจฉัย:

```json5
{
  diagnostics: {
    enabled: false,
  },
}
```

การปิดใช้การวินิจฉัยจะลดรายละเอียดรายงานข้อบกพร่อง แต่ไม่กระทบต่อ
การบันทึก Gateway ตามปกติ
