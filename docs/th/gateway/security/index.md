  **โมเดลความไว้วางใจของผู้ช่วยส่วนตัว** คำแนะนำนี้ถือว่ามีขอบเขต
  ผู้ปฏิบัติการที่เชื่อถือได้หนึ่งขอบเขตต่อ gateway (โมเดลผู้ช่วยส่วนตัวแบบผู้ใช้คนเดียว)
  OpenClaw **ไม่ใช่** ขอบเขตความปลอดภัยแบบหลายผู้เช่าที่เป็นศัตรู สำหรับผู้ใช้หลายคน
  ที่มีเจตนาโจมตีและใช้เอเจนต์หรือ gateway เดียวกันร่วมกัน หากคุณต้องการการทำงานแบบความไว้วางใจผสมหรือ
  ผู้ใช้ที่มีเจตนาโจมตี ให้แยกขอบเขตความไว้วางใจ (gateway +
  ข้อมูลประจำตัวแยกกัน และควรแยกผู้ใช้ OS หรือโฮสต์ด้วย)
</Warning>

## ขอบเขตก่อน: โมเดลความปลอดภัยของผู้ช่วยส่วนตัว

คำแนะนำด้านความปลอดภัยของ OpenClaw ถือว่าเป็นการปรับใช้แบบ **ผู้ช่วยส่วนตัว**: ขอบเขตผู้ปฏิบัติการที่เชื่อถือได้หนึ่งขอบเขต และอาจมีเอเจนต์หลายตัว

- สถานะความปลอดภัยที่รองรับ: ผู้ใช้/ขอบเขตความไว้วางใจหนึ่งรายต่อ gateway (ควรใช้ผู้ใช้ OS/โฮสต์/VPS หนึ่งรายต่อขอบเขต)
- ไม่ใช่ขอบเขตความปลอดภัยที่รองรับ: gateway/เอเจนต์ที่ใช้ร่วมกันหนึ่งตัวซึ่งถูกใช้โดยผู้ใช้ที่ไม่ไว้วางใจกันหรือมีเจตนาโจมตี
- หากต้องมีการแยกผู้ใช้ที่มีเจตนาโจมตี ให้แยกตามขอบเขตความไว้วางใจ (gateway + ข้อมูลประจำตัวแยกกัน และควรแยกผู้ใช้ OS/โฮสต์ด้วย)
- หากผู้ใช้ที่ไม่เชื่อถือหลายคนสามารถส่งข้อความถึงเอเจนต์หนึ่งตัวที่เปิดใช้เครื่องมือได้ ให้ถือว่าทุกคนใช้สิทธิ์เครื่องมือที่มอบหมายชุดเดียวกันสำหรับเอเจนต์นั้นร่วมกัน

หน้านี้อธิบายการเสริมความแข็งแกร่ง **ภายในโมเดลนั้น** ไม่ได้อ้างว่ามีการแยกแบบหลายผู้เช่าที่เป็นศัตรูบน gateway ที่ใช้ร่วมกันหนึ่งตัว

## ตรวจสอบอย่างรวดเร็ว: `openclaw security audit`

ดูเพิ่มเติม: [การตรวจสอบอย่างเป็นทางการ (โมเดลความปลอดภัย)](/th/security/formal-verification)

เรียกใช้สิ่งนี้เป็นประจำ (โดยเฉพาะหลังจากเปลี่ยน config หรือเปิดพื้นผิวเครือข่าย):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

`security audit --fix` ถูกจำกัดขอบเขตอย่างตั้งใจ: จะเปลี่ยนนโยบายกลุ่มแบบเปิดทั่วไป
เป็น allowlists, คืนค่า `logging.redactSensitive: "tools"`, เพิ่มความเข้มงวดของ
สิทธิ์ state/config/include-file และใช้การรีเซ็ต ACL ของ Windows แทน
POSIX `chmod` เมื่อทำงานบน Windows

จะแจ้งเตือน footguns ที่พบบ่อย (การเปิดเผยการยืนยันตัวตนของ Gateway, การเปิดเผยการควบคุมเบราว์เซอร์, allowlists ที่ยกระดับ, สิทธิ์ระบบไฟล์, การอนุมัติ exec ที่ผ่อนปรน และการเปิดเผยเครื่องมือในช่องทางแบบเปิด)

OpenClaw เป็นทั้งผลิตภัณฑ์และการทดลอง: คุณกำลังเชื่อมพฤติกรรมของโมเดลแนวหน้าเข้ากับพื้นผิวการส่งข้อความจริงและเครื่องมือจริง **ไม่มีการตั้งค่าใดที่ “ปลอดภัยสมบูรณ์แบบ”** เป้าหมายคือการตั้งใจอย่างชัดเจนเกี่ยวกับ:

- ใครสามารถคุยกับบอตของคุณได้
- บอตได้รับอนุญาตให้ดำเนินการที่ไหน
- บอตแตะต้องอะไรได้บ้าง

เริ่มด้วยสิทธิ์เข้าถึงที่น้อยที่สุดแต่ยังทำงานได้ จากนั้นจึงขยายเมื่อคุณมั่นใจมากขึ้น

### การปรับใช้และความไว้วางใจโฮสต์

OpenClaw ถือว่าขอบเขตโฮสต์และ config นั้นเชื่อถือได้:

- หากมีใครแก้ไขสถานะ/config ของโฮสต์ Gateway (`~/.openclaw` รวมถึง `openclaw.json`) ได้ ให้ถือว่าบุคคลนั้นเป็นผู้ปฏิบัติการที่เชื่อถือได้
- การรัน Gateway หนึ่งตัวสำหรับผู้ปฏิบัติการหลายคนที่ไม่ไว้วางใจกัน/มีเจตนาโจมตี **ไม่ใช่การตั้งค่าที่แนะนำ**
- สำหรับทีมที่มีความไว้วางใจผสม ให้แยกขอบเขตความไว้วางใจด้วย gateway แยกกัน (หรืออย่างน้อยที่สุดคือผู้ใช้ OS/โฮสต์แยกกัน)
- ค่าเริ่มต้นที่แนะนำ: ผู้ใช้หนึ่งคนต่อเครื่อง/โฮสต์ (หรือ VPS), gateway หนึ่งตัวสำหรับผู้ใช้นั้น และเอเจนต์หนึ่งตัวหรือมากกว่าใน gateway นั้น
- ภายในอินสแตนซ์ Gateway หนึ่งตัว การเข้าถึงของผู้ปฏิบัติการที่ยืนยันตัวตนแล้วคือบทบาท control-plane ที่เชื่อถือได้ ไม่ใช่บทบาทผู้เช่าต่อผู้ใช้
- ตัวระบุเซสชัน (`sessionKey`, ID เซสชัน, ป้ายกำกับ) เป็นตัวเลือกการกำหนดเส้นทาง ไม่ใช่โทเค็นการอนุญาต
- หากหลายคนสามารถส่งข้อความถึงเอเจนต์หนึ่งตัวที่เปิดใช้เครื่องมือได้ แต่ละคนจะสามารถควบคุมชุดสิทธิ์เดียวกันนั้นได้ การแยกเซสชัน/หน่วยความจำต่อผู้ใช้ช่วยเรื่องความเป็นส่วนตัว แต่ไม่ได้แปลงเอเจนต์ที่ใช้ร่วมกันให้เป็นการอนุญาตโฮสต์ต่อผู้ใช้

### พื้นที่ทำงาน Slack ที่ใช้ร่วมกัน: ความเสี่ยงจริง

หาก "ทุกคนใน Slack สามารถส่งข้อความถึงบอตได้" ความเสี่ยงหลักคือสิทธิ์เครื่องมือที่มอบหมาย:

- ผู้ส่งที่ได้รับอนุญาตคนใดก็ได้สามารถชักนำให้เกิดการเรียกเครื่องมือ (`exec`, เบราว์เซอร์, เครื่องมือเครือข่าย/ไฟล์) ภายในนโยบายของเอเจนต์;
- การฉีด prompt/content จากผู้ส่งหนึ่งคนอาจทำให้เกิดการดำเนินการที่กระทบต่อสถานะ อุปกรณ์ หรือผลลัพธ์ที่ใช้ร่วมกัน;
- หากเอเจนต์ที่ใช้ร่วมกันหนึ่งตัวมีข้อมูลประจำตัว/ไฟล์ที่ละเอียดอ่อน ผู้ส่งที่ได้รับอนุญาตคนใดก็อาจขับเคลื่อนการรั่วไหลผ่านการใช้งานเครื่องมือได้

ใช้เอเจนต์/gateway แยกกันพร้อมเครื่องมือขั้นต่ำสำหรับเวิร์กโฟลว์ของทีม; เก็บเอเจนต์ข้อมูลส่วนตัวไว้เป็นส่วนตัว

### เอเจนต์ที่ใช้ร่วมกันในบริษัท: รูปแบบที่ยอมรับได้

สิ่งนี้ยอมรับได้เมื่อทุกคนที่ใช้เอเจนต์นั้นอยู่ในขอบเขตความไว้วางใจเดียวกัน (เช่น ทีมบริษัทเดียวกัน) และเอเจนต์ถูกจำกัดขอบเขตเฉพาะงานธุรกิจอย่างเข้มงวด

- รันบนเครื่อง/VM/คอนเทนเนอร์เฉพาะ;
- ใช้ผู้ใช้ OS เฉพาะ + เบราว์เซอร์/โปรไฟล์/บัญชีเฉพาะสำหรับ runtime นั้น;
- อย่าลงชื่อเข้าใช้บัญชี Apple/Google ส่วนตัว หรือโปรไฟล์ตัวจัดการรหัสผ่าน/เบราว์เซอร์ส่วนตัวใน runtime นั้น

หากคุณผสมตัวตนส่วนตัวและบริษัทใน runtime เดียวกัน คุณจะทำให้การแยกนั้นยุบลงและเพิ่มความเสี่ยงในการเปิดเผยข้อมูลส่วนตัว

## แนวคิดความไว้วางใจของ Gateway และ node

ให้ถือว่า Gateway และ node เป็นโดเมนความไว้วางใจของผู้ปฏิบัติการเดียวกัน แต่มีบทบาทต่างกัน:

- **Gateway** คือ control plane และพื้นผิวนโยบาย (`gateway.auth`, นโยบายเครื่องมือ, การกำหนดเส้นทาง)
- **Node** คือพื้นผิวการดำเนินการระยะไกลที่จับคู่กับ Gateway นั้น (คำสั่ง, การดำเนินการบนอุปกรณ์, ความสามารถเฉพาะโฮสต์)
- ผู้เรียกที่ยืนยันตัวตนกับ Gateway แล้วถือว่าเชื่อถือได้ในขอบเขต Gateway หลังจากจับคู่แล้ว การดำเนินการของ node ถือเป็นการดำเนินการของผู้ปฏิบัติการที่เชื่อถือได้บน node นั้น
- ระดับขอบเขตของผู้ปฏิบัติการและการตรวจสอบขณะอนุมัติสรุปไว้ใน
  [ขอบเขตผู้ปฏิบัติการ](/th/gateway/operator-scopes)
- ไคลเอนต์ backend แบบ loopback โดยตรงที่ยืนยันตัวตนด้วยโทเค็น/รหัสผ่าน gateway
  ที่ใช้ร่วมกันสามารถเรียก RPC ภายในของ control-plane ได้โดยไม่ต้องแสดงตัวตน
  อุปกรณ์ของผู้ใช้ นี่ไม่ใช่การข้ามการจับคู่ระยะไกลหรือเบราว์เซอร์: ไคลเอนต์เครือข่าย,
  ไคลเอนต์ node, ไคลเอนต์ device-token และตัวตนอุปกรณ์ที่ระบุชัดเจน
  ยังคงผ่านการจับคู่และการบังคับใช้การยกระดับขอบเขต
- `sessionKey` คือการเลือกการกำหนดเส้นทาง/บริบท ไม่ใช่การยืนยันตัวตนต่อผู้ใช้
- การอนุมัติ Exec (allowlist + ask) เป็น guardrails สำหรับเจตนาของผู้ปฏิบัติการ ไม่ใช่การแยกแบบหลายผู้เช่าที่เป็นศัตรู
- ค่าเริ่มต้นของผลิตภัณฑ์ OpenClaw สำหรับการตั้งค่าแบบผู้ปฏิบัติการเดียวที่เชื่อถือได้คือ อนุญาต host exec บน `gateway`/`node` โดยไม่ต้องมี prompt อนุมัติ (`security="full"`, `ask="off"` เว้นแต่คุณจะเพิ่มความเข้มงวด) ค่าเริ่มต้นนั้นเป็น UX ที่ตั้งใจ ไม่ใช่ช่องโหว่โดยตัวมันเอง
- การอนุมัติ Exec ผูกกับบริบทคำขอที่แน่นอนและ operand ไฟล์ local โดยตรงแบบ best-effort; ไม่ได้จำลองเชิงความหมายทุกเส้นทาง loader ของ runtime/interpreter ใช้ sandboxing และการแยกโฮสต์เพื่อขอบเขตที่แข็งแรง

หากคุณต้องการการแยกผู้ใช้ที่เป็นศัตรู ให้แยกขอบเขตความไว้วางใจตามผู้ใช้ OS/โฮสต์ และรัน gateway แยกกัน

## เมทริกซ์ขอบเขตความไว้วางใจ

ใช้สิ่งนี้เป็นโมเดลอย่างรวดเร็วเมื่อคัดแยกความเสี่ยง:

| ขอบเขตหรือการควบคุม                                      | ความหมาย                                           | การอ่านผิดที่พบบ่อย                                                           |
| --------------------------------------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------- |
| `gateway.auth` (token/password/trusted-proxy/device auth) | ยืนยันตัวตนผู้เรียกไปยัง API ของ gateway          | "ต้องมีลายเซ็นต่อข้อความในทุกเฟรมจึงจะปลอดภัย"                              |
| `sessionKey`                                              | คีย์กำหนดเส้นทางสำหรับการเลือกบริบท/เซสชัน        | "session key คือขอบเขตการยืนยันตัวตนของผู้ใช้"                               |
| Prompt/content guardrails                                 | ลดความเสี่ยงการใช้โมเดลในทางที่ผิด                | "prompt injection เพียงอย่างเดียวพิสูจน์การข้าม auth"                        |
| `canvas.eval` / browser evaluate                          | ความสามารถของผู้ปฏิบัติการโดยตั้งใจเมื่อเปิดใช้   | "primitive ของ JS eval ใดๆ เป็น vuln โดยอัตโนมัติในโมเดลความไว้วางใจนี้"      |
| เชลล์ `!` ของ TUI local                                   | การดำเนินการ local ที่ผู้ปฏิบัติการเรียกโดยชัดเจน | "คำสั่งอำนวยความสะดวกของเชลล์ local คือ remote injection"                    |
| การจับคู่ Node และคำสั่ง node                             | การดำเนินการระยะไกลระดับผู้ปฏิบัติการบนอุปกรณ์ที่จับคู่แล้ว | "การควบคุมอุปกรณ์ระยะไกลควรถูกมองเป็นการเข้าถึงของผู้ใช้ที่ไม่เชื่อถือโดยค่าเริ่มต้น" |
| `gateway.nodes.pairing.autoApproveCidrs`                  | นโยบายการลงทะเบียน node บนเครือข่ายที่เชื่อถือแบบ opt-in | "allowlist ที่ปิดโดยค่าเริ่มต้นคือช่องโหว่การจับคู่อัตโนมัติ"              |

จะแสดงให้เห็นการข้ามขอบเขตจริงได้:

- เชนที่เป็น prompt-injection-only โดยไม่มีการข้ามนโยบาย, auth หรือ sandbox

  `sessions.list` / `sessions.preview` / `chat.history`) เป็น IDOR ใน
  การตั้งค่า shared-gateway

  ใน repo นี้

- ข้อค้นพบ "ขาดการอนุญาตต่อผู้ใช้" ที่ถือว่า `sessionKey` เป็น

## baseline ที่เสริมความแข็งแกร่งใน 60 วินาที

ใช้ baseline นี้ก่อน แล้วจึงเลือกเปิดใช้เครื่องมืออีกครั้งต่อเอเจนต์ที่เชื่อถือได้:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

สิ่งนี้ทำให้ Gateway เป็น local-only, แยก DM ออกจากกัน และปิดใช้เครื่องมือ control-plane/runtime โดยค่าเริ่มต้น

## กฎด่วนสำหรับกล่องข้อความที่ใช้ร่วมกัน

หากมีมากกว่าหนึ่งคนสามารถ DM บอตของคุณได้:

- ตั้งค่า `session.dmScope: "per-channel-peer"` (หรือ `"per-account-channel-peer"` สำหรับช่องทางหลายบัญชี)
- คง `dmPolicy: "pairing"` หรือ allowlists ที่เข้มงวดไว้
- อย่ารวม DM ที่ใช้ร่วมกันกับสิทธิ์เข้าถึงเครื่องมือแบบกว้าง
- สิ่งนี้ช่วยเสริมความแข็งแกร่งให้กล่องข้อความแบบร่วมมือ/ใช้ร่วมกัน แต่ไม่ได้ออกแบบมาเป็นการแยกผู้เช่าร่วมที่เป็นศัตรูเมื่อผู้ใช้มีสิทธิ์เขียนโฮสต์/config ร่วมกัน

## โมเดลการมองเห็นบริบท

OpenClaw แยกสองแนวคิดออกจากกัน:

- **การอนุญาตทริกเกอร์**: ใครสามารถทริกเกอร์เอเจนต์ได้ (`dmPolicy`, `groupPolicy`, allowlists, ประตู mention)
- **การมองเห็นบริบท**: บริบทเสริมใดถูกฉีดเข้าไปใน input ของโมเดล (เนื้อหาตอบกลับ, ข้อความที่อ้างอิง, ประวัติ thread, metadata ที่ส่งต่อ)

Allowlists ควบคุมทริกเกอร์และการอนุญาตคำสั่ง การตั้งค่า `contextVisibility` ควบคุมวิธีกรองบริบทเสริม (การตอบกลับที่อ้างอิง, root ของ thread, ประวัติที่ดึงมา):

- `contextVisibility: "all"` (ค่าเริ่มต้น) คงบริบทเสริมไว้ตามที่ได้รับ
- `contextVisibility: "allowlist"` กรองบริบทเสริมให้เหลือผู้ส่งที่ได้รับอนุญาตจากการตรวจสอบ allowlist ที่ใช้งานอยู่
- `contextVisibility: "allowlist_quote"` ทำงานเหมือน `allowlist` แต่ยังคงการตอบกลับที่อ้างอิงอย่างชัดเจนหนึ่งรายการไว้

ตั้งค่า `contextVisibility` ต่อช่องทางหรือต่อห้อง/การสนทนา ดู [แชตกลุ่ม](/th/channels/groups#context-visibility-and-allowlists) สำหรับรายละเอียดการตั้งค่า

คำแนะนำการคัดแยกเชิงแนะนำ:

- ข้อกล่าวอ้างที่แสดงเพียงว่า "โมเดลสามารถเห็นข้อความที่อ้างอิงหรือข้อความประวัติจากผู้ส่งที่ไม่ได้อยู่ใน allowlist" เป็นข้อค้นพบด้านการเสริมความแข็งแกร่งที่จัดการได้ด้วย `contextVisibility` ไม่ใช่การข้ามขอบเขต auth หรือ sandbox โดยตัวมันเอง.
- หากจะมีผลกระทบด้านความปลอดภัย รายงานยังต้องแสดงการข้ามขอบเขตความไว้วางใจที่พิสูจน์ได้ (auth, policy, sandbox, approval หรือขอบเขตอื่นที่มีเอกสารกำกับ).

## สิ่งที่การตรวจสอบประเมิน (ภาพรวม)

- **การเข้าถึงขาเข้า** (นโยบาย DM, นโยบายกลุ่ม, allowlists): คนแปลกหน้าสามารถเรียกบอตได้หรือไม่?
- **ขอบเขตผลกระทบของเครื่องมือ** (เครื่องมือที่มีสิทธิ์สูง + ห้องแบบเปิด): prompt injection อาจกลายเป็นการกระทำกับ shell/ไฟล์/เครือข่ายได้หรือไม่?
- **การเลื่อนหลุดของการอนุมัติ exec** (`security=full`, `autoAllowSkills`, allowlists ของ interpreter ที่ไม่มี `strictInlineEval`): guardrail ของ host-exec ยังทำสิ่งที่คุณคิดว่าทำอยู่หรือไม่?
  - `security="full"` เป็นคำเตือนเชิงท่าทีแบบกว้าง ไม่ใช่หลักฐานของบั๊ก เป็นค่าเริ่มต้นที่เลือกไว้สำหรับการตั้งค่าผู้ช่วยส่วนตัวที่เชื่อถือได้; ทำให้เข้มงวดขึ้นเฉพาะเมื่อ threat model ของคุณต้องการ guardrail แบบ approval หรือ allowlist.
- **การเปิดเผยเครือข่าย** (การ bind/auth ของ Gateway, Tailscale Serve/Funnel, โทเค็น auth ที่อ่อนแอ/สั้น).
- **การเปิดเผยการควบคุมเบราว์เซอร์** (Node ระยะไกล, พอร์ต relay, endpoint CDP ระยะไกล).
- **สุขอนามัยของดิสก์ในเครื่อง** (สิทธิ์, symlinks, config includes, เส้นทาง "โฟลเดอร์ที่ซิงค์").
- **Plugin** (Plugin โหลดโดยไม่มี allowlist ที่ระบุชัดเจน).
- **การเลื่อนหลุด/การตั้งค่าผิดของนโยบาย** (ตั้งค่า sandbox docker ไว้แต่โหมด sandbox ปิดอยู่; รูปแบบ `gateway.nodes.denyCommands` ไม่ได้ผลเพราะการจับคู่เป็นชื่อคำสั่งแบบตรงตัวเท่านั้น (เช่น `system.run`) และไม่ตรวจข้อความ shell; รายการ `gateway.nodes.allowCommands` ที่อันตราย; `tools.profile="minimal"` ระดับ global ถูกแทนที่ด้วยโปรไฟล์ราย agent; เครื่องมือที่ Plugin เป็นเจ้าของเข้าถึงได้ภายใต้นโยบายเครื่องมือที่ผ่อนปรน).
- **การเลื่อนหลุดของความคาดหวังขณะรันไทม์** (เช่น สมมติว่า implicit exec ยังหมายถึง `sandbox` ในขณะที่ `tools.exec.host` ตอนนี้มีค่าเริ่มต้นเป็น `auto`, หรือตั้งค่า `tools.exec.host="sandbox"` อย่างชัดเจนทั้งที่โหมด sandbox ปิดอยู่).
- **สุขอนามัยของโมเดล** (เตือนเมื่อโมเดลที่ตั้งค่าดูล้าสมัย; ไม่ใช่ hard block).

หากคุณรัน `--deep`, OpenClaw จะพยายาม probe Gateway แบบ live ด้วย best-effort ด้วย.

## แผนผังที่เก็บข้อมูลรับรอง

ใช้ส่วนนี้เมื่อ audit การเข้าถึงหรือตัดสินใจว่าจะสำรองข้อมูลอะไร:

- **WhatsApp**: `~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
- **โทเค็นบอต Telegram**: config/env หรือ `channels.telegram.tokenFile` (ไฟล์ปกติเท่านั้น; symlinks จะถูกปฏิเสธ)
- **โทเค็นบอต Discord**: config/env หรือ SecretRef (provider แบบ env/file/exec)
- **โทเค็น Slack**: config/env (`channels.slack.*`)
- **Pairing allowlists**:
  - `~/.openclaw/credentials/<channel>-allowFrom.json` (บัญชีเริ่มต้น)
  - `~/.openclaw/credentials/<channel>-<accountId>-allowFrom.json` (บัญชีที่ไม่ใช่ค่าเริ่มต้น)
- **โปรไฟล์ auth ของโมเดล**: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- **สถานะรันไทม์ Codex**: `~/.openclaw/agents/<agentId>/agent/codex-home/`
- **payload secrets ที่อิงไฟล์ (ไม่บังคับ)**: `~/.openclaw/secrets.json`
- **การนำเข้า OAuth แบบเดิม**: `~/.openclaw/credentials/oauth.json`

## เช็กลิสต์การตรวจสอบความปลอดภัย

เมื่อการตรวจสอบพิมพ์ข้อค้นพบ ให้ถือว่านี่คือลำดับความสำคัญ:

1. **สิ่งใดก็ตามที่ “เปิด” + เปิดใช้เครื่องมือ**: ล็อก DM/กลุ่มก่อน (pairing/allowlists) จากนั้นทำให้นโยบายเครื่องมือ/sandboxing เข้มงวดขึ้น.
2. **การเปิดเผยเครือข่ายสาธารณะ** (LAN bind, Funnel, ไม่มี auth): แก้ไขทันที.
3. **การเปิดเผยการควบคุมเบราว์เซอร์ระยะไกล**: ปฏิบัติกับสิ่งนี้เหมือนการเข้าถึงของ operator (เฉพาะ tailnet, pair Node อย่างตั้งใจ, หลีกเลี่ยงการเปิดเผยต่อสาธารณะ).
4. **สิทธิ์**: ตรวจให้แน่ใจว่า state/config/credentials/auth ไม่สามารถอ่านได้โดย group/world.
5. **Plugin**: โหลดเฉพาะสิ่งที่คุณเชื่อถืออย่างชัดเจนเท่านั้น.
6. **การเลือกโมเดล**: เลือกโมเดลสมัยใหม่ที่เสริมความแข็งแกร่งด้าน instruction สำหรับบอตใดๆ ที่มีเครื่องมือ.

## อภิธานศัพท์การตรวจสอบความปลอดภัย

ข้อค้นพบแต่ละรายการของการตรวจสอบถูกกำกับด้วย `checkId` แบบมีโครงสร้าง (เช่น
`gateway.bind_no_auth` หรือ `tools.exec.security_full_configured`). คลาสความรุนแรงระดับ
critical ที่พบบ่อย:

- `fs.*` — สิทธิ์ระบบไฟล์ของ state, config, credentials, โปรไฟล์ auth.
- `gateway.*` — โหมด bind, auth, Tailscale, Control UI, การตั้งค่า trusted-proxy.
- `hooks.*`, `browser.*`, `sandbox.*`, `tools.exec.*` — การเสริมความแข็งแกร่งรายพื้นผิว.
- `plugins.*`, `skills.*` — supply chain ของ Plugin/skill และข้อค้นพบจากการสแกน.
- `security.exposure.*` — การตรวจสอบแบบตัดข้ามส่วนที่นโยบายการเข้าถึงมาบรรจบกับขอบเขตผลกระทบของเครื่องมือ.

ดูแค็ตตาล็อกฉบับเต็มพร้อมระดับความรุนแรง, fix keys และการรองรับ auto-fix ได้ที่
[การตรวจสอบ security audit](/th/gateway/security/audit-checks).

## Control UI ผ่าน HTTP

Control UI ต้องใช้ **secure context** (HTTPS หรือ localhost) เพื่อสร้าง identity
ของอุปกรณ์. `gateway.controlUi.allowInsecureAuth` เป็น toggle ความเข้ากันได้ในเครื่อง:

- บน localhost จะอนุญาตให้ Control UI auth โดยไม่มี identity ของอุปกรณ์เมื่อหน้า
  โหลดผ่าน HTTP ที่ไม่ปลอดภัย.
- ไม่ได้ข้ามการตรวจ pairing.
- ไม่ได้ผ่อนปรนข้อกำหนด identity ของอุปกรณ์ระยะไกล (ที่ไม่ใช่ localhost).

ควรใช้ HTTPS (Tailscale Serve) หรือเปิด UI บน `127.0.0.1`.

สำหรับสถานการณ์ break-glass เท่านั้น, `gateway.controlUi.dangerouslyDisableDeviceAuth`
จะปิดการตรวจ identity ของอุปกรณ์ทั้งหมด นี่เป็นการลดระดับความปลอดภัยอย่างรุนแรง;
ปิดไว้เสมอเว้นแต่ว่าคุณกำลังดีบักอยู่จริงและสามารถย้อนกลับได้อย่างรวดเร็ว.

แยกจาก flag อันตรายเหล่านั้น, `gateway.auth.mode: "trusted-proxy"` ที่สำเร็จ
สามารถรับ session Control UI ของ **operator** โดยไม่มี identity ของอุปกรณ์ได้ นั่นเป็น
พฤติกรรมของโหมด auth โดยตั้งใจ ไม่ใช่ทางลัดของ `allowInsecureAuth` และยังคง
ไม่ขยายไปยัง session Control UI บทบาท Node.

`openclaw security audit` จะเตือนเมื่อเปิดใช้การตั้งค่านี้.

## การกำหนดค่า reverse proxy

หากคุณรัน Gateway หลัง reverse proxy (nginx, Caddy, Traefik ฯลฯ) ให้กำหนดค่า
`gateway.trustedProxies` เพื่อให้จัดการ IP ไคลเอนต์ที่ถูก forward อย่างถูกต้อง.

เมื่อ Gateway ตรวจพบ header ของ proxy จาก address ที่ **ไม่** อยู่ใน `trustedProxies` มันจะ **ไม่** ถือว่า connection เป็นไคลเอนต์ในเครื่อง หากปิดใช้ gateway auth ไว้ connection เหล่านั้นจะถูกปฏิเสธ สิ่งนี้ป้องกันการข้าม authentication ที่ connection ผ่าน proxy มิฉะนั้นจะดูเหมือนมาจาก localhost และได้รับความไว้วางใจอัตโนมัติ.

`gateway.trustedProxies` ยังป้อนให้ `gateway.auth.mode: "trusted-proxy"` ด้วย แต่โหมด auth นั้นเข้มงวดกว่า:

- trusted-proxy auth **ล้มเหลวแบบปิดกับ proxy ที่มีแหล่งที่มาเป็น loopback ตามค่าเริ่มต้น**
- reverse proxy แบบ loopback บน host เดียวกันสามารถใช้ `gateway.trustedProxies` สำหรับการตรวจหา local-client และการจัดการ forwarded IP
- reverse proxy แบบ loopback บน host เดียวกันสามารถผ่าน `gateway.auth.mode: "trusted-proxy"` ได้เฉพาะเมื่อ `gateway.auth.trustedProxy.allowLoopback = true`; มิฉะนั้นให้ใช้ token/password auth

```yaml
gateway:
  trustedProxies:
    - "10.0.0.1" # reverse proxy IP
  # Optional. Default false.
  # Only enable if your proxy cannot provide X-Forwarded-For.
  allowRealIpFallback: false
  auth:
    mode: password
    password: ${OPENCLAW_GATEWAY_PASSWORD}
```

เมื่อกำหนดค่า `trustedProxies` แล้ว Gateway จะใช้ `X-Forwarded-For` เพื่อระบุ IP ของไคลเอนต์ `X-Real-IP` จะถูกละเว้นตามค่าเริ่มต้น เว้นแต่จะตั้งค่า `gateway.allowRealIpFallback: true` อย่างชัดเจน.

header ของ trusted proxy ไม่ได้ทำให้การ pair อุปกรณ์ Node น่าเชื่อถือโดยอัตโนมัติ.
`gateway.nodes.pairing.autoApproveCidrs` เป็นนโยบาย operator ที่แยกต่างหากและปิดไว้ตามค่าเริ่มต้น.
แม้จะเปิดใช้แล้ว เส้นทาง header ของ trusted-proxy ที่มีแหล่งที่มาเป็น loopback
จะถูกแยกออกจากการ auto-approval ของ Node เพราะผู้เรียกในเครื่องสามารถปลอม header เหล่านั้นได้
รวมถึงเมื่อเปิดใช้ loopback trusted-proxy auth อย่างชัดเจนแล้วก็ตาม.

พฤติกรรม reverse proxy ที่ดี (เขียนทับ forwarding header ขาเข้า):

```nginx
proxy_set_header X-Forwarded-For $remote_addr;
proxy_set_header X-Real-IP $remote_addr;
```

พฤติกรรม reverse proxy ที่ไม่ดี (ต่อท้าย/คง forwarding header ที่ไม่น่าเชื่อถือไว้):

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```

## หมายเหตุเกี่ยวกับ HSTS และ origin

- Gateway ของ OpenClaw เน้น local/loopback ก่อน หากคุณ terminate TLS ที่ reverse proxy ให้ตั้ง HSTS บนโดเมน HTTPS ฝั่ง proxy ที่นั่น.
- หาก gateway terminate HTTPS เอง คุณสามารถตั้ง `gateway.http.securityHeaders.strictTransportSecurity` เพื่อส่ง header HSTS จาก response ของ OpenClaw.
- คำแนะนำการ deploy แบบละเอียดอยู่ใน [Trusted Proxy Auth](/th/gateway/trusted-proxy-auth#tls-termination-and-hsts).
- สำหรับ deployment ของ Control UI ที่ไม่ใช่ loopback, ต้องมี `gateway.controlUi.allowedOrigins` ตามค่าเริ่มต้น.
- `gateway.controlUi.allowedOrigins: ["*"]` เป็นนโยบายอนุญาต browser-origin ทั้งหมดแบบชัดเจน ไม่ใช่ค่าเริ่มต้นที่เสริมความแข็งแกร่งแล้ว หลีกเลี่ยงนอกการทดสอบในเครื่องที่ควบคุมอย่างเข้มงวด.
- ความล้มเหลวของ browser-origin auth บน loopback ยังคงถูก rate-limit แม้เมื่อเปิดใช้
  การยกเว้น loopback ทั่วไป แต่ key ของ lockout จะ scoped ต่อค่า `Origin`
  ที่ normalize แล้ว แทนที่จะเป็น bucket localhost ร่วมกันหนึ่งชุด.
- `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` เปิดใช้โหมด fallback ของ origin จาก Host-header; ให้ถือว่านี่เป็นนโยบายอันตรายที่ operator เลือก.
- ปฏิบัติกับ DNS rebinding และพฤติกรรม proxy-host header เป็นข้อกังวลด้านการเสริมความแข็งแกร่งของ deployment; ตั้ง `trustedProxies` ให้แคบและหลีกเลี่ยงการเปิด gateway ให้สาธารณะอินเทอร์เน็ตโดยตรง.

## บันทึก session ในเครื่องอยู่บนดิสก์

OpenClaw เก็บ transcript ของ session บนดิสก์ใต้ `~/.openclaw/agents/<agentId>/sessions/*.jsonl`.
สิ่งนี้จำเป็นสำหรับความต่อเนื่องของ session และ (ไม่บังคับ) การทำดัชนี memory ของ session แต่ยังหมายความว่า
**process/user ใดๆ ที่เข้าถึงระบบไฟล์ได้สามารถอ่านบันทึกเหล่านั้นได้**. ให้ถือว่าการเข้าถึงดิสก์เป็น
ขอบเขตความไว้วางใจ และล็อกสิทธิ์บน `~/.openclaw` (ดูส่วนการตรวจสอบด้านล่าง). หากคุณต้องการ
การแยกที่แข็งแรงกว่าระหว่าง agent ให้รันภายใต้ผู้ใช้ OS แยกกันหรือ host แยกกัน.

## การเรียกใช้ Node (system.run)

หากมีการ pair Node macOS แล้ว Gateway สามารถเรียก `system.run` บน Node นั้นได้ นี่คือ **การเรียกใช้โค้ดระยะไกล** บน Mac:

- ต้องใช้การจับคู่ Node (การอนุมัติ + โทเค็น)
- การจับคู่ Node ของ Gateway ไม่ใช่พื้นผิวการอนุมัติรายคำสั่ง แต่เป็นการสร้างตัวตน/ความเชื่อถือของ Node และการออกโทเค็น
- Gateway ใช้นโยบายคำสั่ง Node ส่วนกลางแบบหยาบผ่าน `gateway.nodes.allowCommands` / `denyCommands`
- ควบคุมบน Mac ผ่าน **Settings → Exec approvals** (ความปลอดภัย + การถาม + รายการอนุญาต)
- นโยบาย `system.run` ราย Node คือไฟล์การอนุมัติการรันคำสั่งของ Node เอง (`exec.approvals.node.*`) ซึ่งอาจเข้มงวดกว่าหรือผ่อนคลายกว่านโยบายรหัสคำสั่งส่วนกลางของ Gateway
- Node ที่รันด้วย `security="full"` และ `ask="off"` กำลังทำตามโมเดลผู้ปฏิบัติการที่เชื่อถือได้ตามค่าเริ่มต้น ให้ถือว่านี่เป็นพฤติกรรมที่คาดไว้ เว้นแต่การปรับใช้ของคุณต้องการจุดยืนด้านการอนุมัติหรือรายการอนุญาตที่เข้มงวดกว่านี้อย่างชัดเจน
- โหมดการอนุมัติผูกกับบริบทคำขอที่ตรงกันทุกประการ และเมื่อเป็นไปได้ จะผูกกับโอเปอแรนด์สคริปต์/ไฟล์ในเครื่องที่เป็นรูปธรรมหนึ่งรายการ หาก OpenClaw ไม่สามารถระบุไฟล์ในเครื่องโดยตรงได้อย่างแน่ชัดเพียงหนึ่งไฟล์สำหรับคำสั่งอินเทอร์พรีเตอร์/รันไทม์ การรันที่อ้างอิงการอนุมัติจะถูกปฏิเสธ แทนที่จะสัญญาว่าครอบคลุมความหมายทั้งหมด
- สำหรับ `host=node` การรันที่อ้างอิงการอนุมัติจะจัดเก็บ
  `systemRunPlan` ที่เตรียมไว้ในรูปแบบมาตรฐานด้วย; การส่งต่อที่ได้รับอนุมัติภายหลังจะใช้แผนนั้นซ้ำ และการตรวจสอบของ Gateway
  จะปฏิเสธการแก้ไขบริบท command/cwd/session โดยผู้เรียกหลังจากสร้าง
  คำขออนุมัติแล้ว
- หากคุณไม่ต้องการการรันจากระยะไกล ให้ตั้งค่าความปลอดภัยเป็น **deny** และลบการจับคู่ Node สำหรับ Mac เครื่องนั้น

ความแตกต่างนี้สำคัญต่อการคัดแยก:

- Node ที่จับคู่ไว้ซึ่งเชื่อมต่อใหม่และประกาศรายการคำสั่งที่ต่างออกไป ไม่ถือเป็นช่องโหว่ในตัวเอง หากนโยบายส่วนกลางของ Gateway และการอนุมัติการรันคำสั่งในเครื่องของ Node ยังคงบังคับใช้ขอบเขตการรันจริง
- รายงานที่มองเมตาดาตาการจับคู่ Node เป็นชั้นการอนุมัติรายคำสั่งที่ซ่อนอยู่อีกชั้นหนึ่ง โดยทั่วไปคือความสับสนด้านนโยบาย/ประสบการณ์ผู้ใช้ ไม่ใช่การข้ามขอบเขตความปลอดภัย

## แนวคิดหลัก: ควบคุมการเข้าถึงก่อนความฉลาด

ความล้มเหลวส่วนใหญ่ที่นี่ไม่ใช่ช่องโหว่ซับซ้อน แต่คือ “มีคนส่งข้อความหาบอต แล้วบอตทำตามที่เขาขอ”

จุดยืนของ OpenClaw:

- **ตัวตนก่อน:** ตัดสินใจว่าใครคุยกับบอตได้ (การจับคู่ DM / รายการอนุญาต / “เปิด” อย่างชัดเจน)
- **ขอบเขตถัดไป:** ตัดสินใจว่าบอตได้รับอนุญาตให้ดำเนินการที่ใด (รายการอนุญาตของกลุ่ม + การกำหนดให้กล่าวถึง, เครื่องมือ, แซนด์บ็อกซ์, สิทธิ์อุปกรณ์)
- **โมเดลสุดท้าย:** สมมติว่าโมเดลถูกชักจูงได้; ออกแบบให้การชักจูงมีรัศมีผลกระทบจำกัด

และ [คำสั่ง Slash](/th/tools/slash-commands)) หากรายการอนุญาตของช่องทางว่างเปล่าหรือมี `"*"`,
คำสั่งจะเปิดใช้งานได้จริงสำหรับช่องทางนั้น

`/exec` เป็นเพียงความสะดวกเฉพาะเซสชันสำหรับผู้ปฏิบัติการที่ได้รับอนุญาต มัน **ไม่** เขียนค่ากำหนดหรือ
เปลี่ยนเซสชันอื่น

## ความเสี่ยงของเครื่องมือระนาบควบคุม

เครื่องมือในตัวสองรายการสามารถทำการเปลี่ยนแปลงระนาบควบคุมแบบถาวรได้:

- `gateway` สามารถตรวจสอบค่ากำหนดด้วย `config.schema.lookup` / `config.get` และสามารถทำการเปลี่ยนแปลงถาวรด้วย `config.apply`, `config.patch`, และ `update.run`
- `cron` สามารถสร้างงานตามกำหนดเวลาที่ยังคงรันต่อหลังจากแชต/งานเดิมสิ้นสุดแล้ว

เครื่องมือรันไทม์ `gateway` สำหรับเจ้าของเท่านั้นยังคงปฏิเสธการเขียนทับ
`tools.exec.ask` หรือ `tools.exec.security`; นามแฝงเดิม `tools.bash.*` จะถูก
ปรับให้เป็นเส้นทาง exec ที่ได้รับการป้องกันเดียวกันก่อนเขียน
การแก้ไข `gateway config.apply` และ `gateway config.patch` ที่ขับเคลื่อนโดยเอเจนต์จะ
ล้มเหลวแบบปิดตามค่าเริ่มต้น: มีเพียงชุดเส้นทาง prompt, model, และ mention-gating
ที่แคบเท่านั้นที่เอเจนต์ปรับได้ ดังนั้นทรีค่ากำหนดละเอียดอ่อนใหม่จึงได้รับการป้องกัน
เว้นแต่จะถูกเพิ่มเข้าไปในรายการอนุญาตโดยตั้งใจ

สำหรับเอเจนต์/พื้นผิวใดก็ตามที่จัดการเนื้อหาที่ไม่เชื่อถือ ให้ปฏิเสธรายการเหล่านี้ตามค่าเริ่มต้น:

```json5
{
  tools: {
    deny: ["gateway", "cron", "sessions_spawn", "sessions_send"],
  },
}
```

`commands.restart=false` บล็อกเฉพาะการกระทำรีสตาร์ตเท่านั้น ไม่ได้ปิดใช้งานการกระทำค่ากำหนด/อัปเดตของ `gateway`

## Plugin

Plugin รัน **ในโปรเซสเดียวกัน** กับ Gateway ให้ถือว่าเป็นโค้ดที่เชื่อถือได้:

- ติดตั้งเฉพาะ Plugin จากแหล่งที่คุณเชื่อถือ
- ควรใช้รายการอนุญาต `plugins.allow` ที่ชัดเจน
- ตรวจทานค่ากำหนด Plugin ก่อนเปิดใช้งาน
- รีสตาร์ต Gateway หลังการเปลี่ยนแปลง Plugin
- หากคุณติดตั้งหรืออัปเดต Plugin (`openclaw plugins install <package>`, `openclaw plugins update <id>`) ให้ถือว่าเหมือนการรันโค้ดที่ไม่เชื่อถือ:
  - เส้นทางติดตั้งคือไดเรกทอรีราย Plugin ภายใต้รูทติดตั้ง Plugin ที่ใช้งานอยู่
  - OpenClaw รันการสแกนโค้ดอันตรายในตัวก่อนติดตั้ง/อัปเดต รายการตรวจพบระดับ `critical` จะบล็อกตามค่าเริ่มต้น
  - การติดตั้ง Plugin จาก npm และ git จะรันการปรับ dependency ผ่านตัวจัดการแพ็กเกจให้สอดคล้องกันเฉพาะระหว่างโฟลว์ติดตั้ง/อัปเดตที่ชัดเจนเท่านั้น เส้นทางในเครื่องและไฟล์เก็บถาวรถือเป็นแพ็กเกจ Plugin ที่สมบูรณ์ในตัว; OpenClaw จะคัดลอก/อ้างอิงโดยไม่รัน `npm install`
  - ควรปักหมุดเวอร์ชันแบบเจาะจงตรงตัว (`@scope/pkg@1.2.3`) และตรวจสอบโค้ดที่แตกไฟล์ไว้บนดิสก์ก่อนเปิดใช้งาน
  - `--dangerously-force-unsafe-install` เป็นทางเลือกฉุกเฉินเฉพาะสำหรับผลบวกลวงจากการสแกนในตัวในโฟลว์ติดตั้ง/อัปเดต Plugin เท่านั้น มันไม่ข้ามการบล็อกนโยบายฮุก `before_install` ของ Plugin และไม่ข้ามความล้มเหลวจากการสแกน
  - การติดตั้ง dependency ของ Skills ที่อิง Gateway ใช้การแบ่งอันตราย/น่าสงสัยแบบเดียวกัน: รายการตรวจพบ `critical` ในตัวจะบล็อก เว้นแต่ผู้เรียกจะตั้งค่า `dangerouslyForceUnsafeInstall` อย่างชัดเจน ส่วนรายการตรวจพบน่าสงสัยยังคงเตือนเท่านั้น `openclaw skills install` ยังคงเป็นโฟลว์ดาวน์โหลด/ติดตั้ง Skills จาก ClawHub แยกต่างหาก

รายละเอียด: [Plugin](/th/tools/plugin)

## โมเดลการเข้าถึง DM: การจับคู่, รายการอนุญาต, เปิด, ปิดใช้งาน

ช่องทางปัจจุบันทั้งหมดที่รองรับ DM รองรับนโยบาย DM (`dmPolicy` หรือ `*.dm.policy`) ที่คัดกรอง DM ขาเข้า **ก่อน** ประมวลผลข้อความ:

- `pairing` (ค่าเริ่มต้น): ผู้ส่งที่ไม่รู้จักจะได้รับรหัสจับคู่สั้น ๆ และบอตจะเพิกเฉยต่อข้อความของพวกเขาจนกว่าจะได้รับอนุมัติ รหัสหมดอายุหลัง 1 ชั่วโมง; DM ซ้ำจะไม่ส่งรหัสใหม่จนกว่าจะสร้างคำขอใหม่ คำขอที่รอดำเนินการจำกัดไว้ที่ **3 ต่อช่องทาง** ตามค่าเริ่มต้น
- `allowlist`: ผู้ส่งที่ไม่รู้จักจะถูกบล็อก (ไม่มีการจับคู่แบบจับมือ)
- `open`: อนุญาตให้ทุกคน DM ได้ (สาธารณะ) **ต้องมี** รายการอนุญาตของช่องทางที่รวม `"*"` (เลือกเข้าร่วมอย่างชัดเจน)
- `disabled`: เพิกเฉยต่อ DM ขาเข้าทั้งหมด

อนุมัติผ่าน CLI:

```bash
openclaw pairing list <channel>
openclaw pairing approve <channel> <code>
```

รายละเอียด + ไฟล์บนดิสก์: [การจับคู่](/th/channels/pairing)

## การแยกเซสชัน DM (โหมดผู้ใช้หลายคน)

ตามค่าเริ่มต้น OpenClaw จะส่ง **DM ทั้งหมดเข้าสู่เซสชันหลัก** เพื่อให้ผู้ช่วยของคุณมีความต่อเนื่องข้ามอุปกรณ์และช่องทาง หาก **หลายคน** สามารถ DM หาบอตได้ (DM แบบเปิดหรือรายการอนุญาตหลายคน) ให้พิจารณาแยกเซสชัน DM:

```json5
{
  session: { dmScope: "per-channel-peer" },
}
```

วิธีนี้ป้องกันการรั่วไหลของบริบทข้ามผู้ใช้ ขณะยังคงแยกแชตกลุ่มออกจากกัน

นี่คือขอบเขตบริบทการส่งข้อความ ไม่ใช่ขอบเขตผู้ดูแลโฮสต์ หากผู้ใช้เป็นฝ่ายตรงข้ามกันและใช้โฮสต์/ค่ากำหนด Gateway เดียวกัน ให้รัน Gateway แยกตามขอบเขตความเชื่อถือแทน

### โหมด DM ปลอดภัย (แนะนำ)

ให้ถือว่าสนิปเพ็ตด้านบนเป็น **โหมด DM ปลอดภัย**:

- ค่าเริ่มต้น: `session.dmScope: "main"` (DM ทั้งหมดใช้เซสชันเดียวร่วมกันเพื่อความต่อเนื่อง)
- ค่าเริ่มต้นของการเริ่มต้นใช้งาน CLI ในเครื่อง: เขียน `session.dmScope: "per-channel-peer"` เมื่อยังไม่ได้ตั้งค่า (คงค่าที่ตั้งไว้อย่างชัดเจนเดิมไว้)
- โหมด DM ปลอดภัย: `session.dmScope: "per-channel-peer"` (คู่ช่องทาง+ผู้ส่งแต่ละคู่ได้บริบท DM ที่แยกกัน)
- การแยก peer ข้ามช่องทาง: `session.dmScope: "per-peer"` (ผู้ส่งแต่ละคนได้หนึ่งเซสชันข้ามทุกช่องทางชนิดเดียวกัน)

หากคุณรันหลายบัญชีบนช่องทางเดียวกัน ให้ใช้ `per-account-channel-peer` แทน หากคนเดียวกันติดต่อคุณผ่านหลายช่องทาง ให้ใช้ `session.identityLinks` เพื่อรวมเซสชัน DM เหล่านั้นเป็นตัวตนมาตรฐานเดียว ดู [การจัดการเซสชัน](/th/concepts/session) และ [การกำหนดค่า](/th/gateway/configuration)

## รายการอนุญาตสำหรับ DM และกลุ่ม

OpenClaw มีชั้น “ใครทริกเกอร์ฉันได้?” แยกกันสองชั้น:

- **รายการอนุญาต DM** (`allowFrom` / `channels.discord.allowFrom` / `channels.slack.allowFrom`; เดิม: `channels.discord.dm.allowFrom`, `channels.slack.dm.allowFrom`): ใครได้รับอนุญาตให้คุยกับบอตในข้อความส่วนตัว
  - เมื่อ `dmPolicy="pairing"` การอนุมัติจะถูกเขียนไปยังที่เก็บรายการอนุญาตการจับคู่ตามขอบเขตบัญชีภายใต้ `~/.openclaw/credentials/` (`<channel>-allowFrom.json` สำหรับบัญชีค่าเริ่มต้น, `<channel>-<accountId>-allowFrom.json` สำหรับบัญชีที่ไม่ใช่ค่าเริ่มต้น) แล้วรวมกับรายการอนุญาตในค่ากำหนด
- **รายการอนุญาตกลุ่ม** (เฉพาะช่องทาง): กลุ่ม/ช่องทาง/กิลด์ใดที่บอตจะรับข้อความเลย
  - รูปแบบทั่วไป:
    - `channels.whatsapp.groups`, `channels.telegram.groups`, `channels.imessage.groups`: ค่าเริ่มต้นรายกลุ่ม เช่น `requireMention`; เมื่อตั้งค่าแล้ว จะทำหน้าที่เป็นรายการอนุญาตกลุ่มด้วย (รวม `"*"` เพื่อคงพฤติกรรมอนุญาตทั้งหมด)
    - `groupPolicy="allowlist"` + `groupAllowFrom`: จำกัดว่าใครสามารถทริกเกอร์บอต _ภายใน_ เซสชันกลุ่ม (WhatsApp/Telegram/Signal/iMessage/Microsoft Teams)
    - `channels.discord.guilds` / `channels.slack.channels`: รายการอนุญาตรายพื้นผิว + ค่าเริ่มต้นการกล่าวถึง
  - การตรวจกลุ่มรันตามลำดับนี้: `groupPolicy`/รายการอนุญาตกลุ่มก่อน, การเปิดใช้งานด้วยการกล่าวถึง/ตอบกลับเป็นลำดับที่สอง
  - การตอบกลับข้อความของบอต (การกล่าวถึงโดยนัย) **ไม่** ข้ามรายการอนุญาตผู้ส่ง เช่น `groupAllowFrom`
  - **หมายเหตุด้านความปลอดภัย:** ให้ถือว่า `dmPolicy="open"` และ `groupPolicy="open"` เป็นการตั้งค่าทางเลือกสุดท้าย ควรใช้น้อยมาก; ควรใช้การจับคู่ + รายการอนุญาต เว้นแต่คุณเชื่อถือสมาชิกทุกคนในห้องอย่างเต็มที่

รายละเอียด: [การกำหนดค่า](/th/gateway/configuration) และ [กลุ่ม](/th/channels/groups)

## Prompt injection (คืออะไร และทำไมจึงสำคัญ)

Prompt injection คือเมื่อผู้โจมตีสร้างข้อความที่ชักจูงโมเดลให้ทำสิ่งที่ไม่ปลอดภัย (“เพิกเฉยต่อคำสั่งของคุณ”, “ดัมป์ระบบไฟล์ของคุณ”, “ตามลิงก์นี้แล้วรันคำสั่ง” ฯลฯ)

แม้จะมี system prompt ที่เข้มแข็ง **prompt injection ก็ยังไม่ได้รับการแก้ไขอย่างสมบูรณ์** แนวป้องกันใน system prompt เป็นเพียงคำแนะนำแบบนุ่มนวลเท่านั้น; การบังคับใช้อย่างแข็งมาจากนโยบายเครื่องมือ, การอนุมัติ exec, แซนด์บ็อกซ์, และรายการอนุญาตของช่องทาง (และผู้ปฏิบัติการสามารถปิดใช้งานสิ่งเหล่านี้ได้ตามการออกแบบ) สิ่งที่ช่วยได้ในทางปฏิบัติ:

- ล็อก DM ขาเข้าให้เข้มงวด (การจับคู่/allowlists)
- แนะนำให้ใช้การกั้นด้วยการกล่าวถึงในกลุ่ม หลีกเลี่ยงบอตแบบ “เปิดตลอดเวลา” ในห้องสาธารณะ
- ถือว่าลิงก์ ไฟล์แนบ และคำสั่งที่วางเข้ามาเป็นสิ่งที่เป็นอันตรายโดยค่าเริ่มต้น
- เรียกใช้การประมวลผลเครื่องมือที่ละเอียดอ่อนใน sandbox เก็บความลับออกจาก filesystem ที่ agent เข้าถึงได้
- หมายเหตุ: sandboxing เป็นแบบ opt-in หากปิดโหมด sandbox อยู่ `host=auto` โดยนัยจะ resolve ไปยังโฮสต์ Gateway ส่วน `host=sandbox` ที่ระบุชัดเจนจะยังล้มเหลวแบบปิดกั้น เพราะไม่มี runtime ของ sandbox ให้ใช้งาน ตั้งค่า `host=gateway` หากคุณต้องการให้พฤติกรรมนั้นชัดเจนใน config
- จำกัดเครื่องมือที่มีความเสี่ยงสูง (`exec`, `browser`, `web_fetch`, `web_search`) ให้ใช้ได้เฉพาะ agent ที่เชื่อถือได้หรือ allowlists ที่ระบุชัดเจน
- หากคุณ allowlist interpreter (`python`, `node`, `ruby`, `perl`, `php`, `lua`, `osascript`) ให้เปิดใช้ `tools.exec.strictInlineEval` เพื่อให้รูปแบบ inline eval ยังต้องได้รับการอนุมัติอย่างชัดเจน
- การวิเคราะห์การอนุมัติ Shell ยังปฏิเสธรูปแบบ POSIX parameter-expansion (`$VAR`, `$?`, `$$`, `$1`, `$@`, `${…}`) ภายใน **heredocs ที่ไม่ถูก quote** ดังนั้นเนื้อหา heredoc ที่อยู่ใน allowlist จึงไม่สามารถลอบนำ shell expansion ผ่านการตรวจ allowlist ในฐานะข้อความธรรมดาได้ ให้ quote ตัวปิด heredoc (เช่น `<<'EOF'`) เพื่อเลือกใช้ semantic ของเนื้อหาแบบ literal; heredocs ที่ไม่ถูก quote ซึ่งจะขยายตัวแปรจะถูกปฏิเสธ
- **การเลือกโมเดลสำคัญ:** โมเดลที่เก่ากว่า/เล็กกว่า/legacy มีความทนทานต่อ prompt injection และการใช้เครื่องมือผิดวัตถุประสงค์ต่ำกว่ามาก สำหรับ agent ที่เปิดใช้เครื่องมือ ให้ใช้โมเดลรุ่นล่าสุดที่แข็งแกร่งที่สุดและถูก harden ตามคำสั่งมากที่สุดเท่าที่มีให้ใช้

สัญญาณเตือนที่ควรถือว่าไม่น่าเชื่อถือ:

- “อ่านไฟล์/URL นี้แล้วทำตามที่ระบุทุกอย่าง”
- “ละเว้น system prompt หรือกฎความปลอดภัยของคุณ”
- “เปิดเผยคำสั่งที่ซ่อนอยู่หรือเอาต์พุตเครื่องมือของคุณ”
- “วางเนื้อหาทั้งหมดของ ~/.openclaw หรือ logs ของคุณ”

## การทำความสะอาด special-token ของเนื้อหาภายนอก

OpenClaw จะลบ literal ของ special-token สำหรับ chat-template ของ LLM แบบ self-hosted ที่พบบ่อยออกจากเนื้อหาและ metadata ภายนอกที่ถูกห่อหุ้ม ก่อนที่สิ่งเหล่านั้นจะไปถึงโมเดล กลุ่ม marker ที่ครอบคลุมรวมถึง role/turn tokens ของ Qwen/ChatML, Llama, Gemma, Mistral, Phi และ GPT-OSS

เหตุผล:

- backend ที่เข้ากันได้กับ OpenAI ซึ่งอยู่หน้าโมเดลแบบ self-hosted บางครั้งจะคง special token ที่ปรากฏในข้อความผู้ใช้ไว้ แทนที่จะ mask ออก ผู้โจมตีที่สามารถเขียนลงในเนื้อหาภายนอกขาเข้า (หน้าที่ fetch มา เนื้อหาอีเมล เอาต์พุตเครื่องมือที่เป็นเนื้อหาไฟล์) อาจ inject ขอบเขต role `assistant` หรือ `system` สังเคราะห์ แล้วหลุดออกจาก guardrails ของเนื้อหาที่ถูกห่อหุ้มได้
- การทำความสะอาดเกิดที่ชั้นห่อหุ้มเนื้อหาภายนอก จึงใช้สม่ำเสมอกับเครื่องมือ fetch/read และเนื้อหาช่องทางขาเข้า แทนที่จะทำแยกตาม provider
- การตอบกลับของโมเดลขาออกมี sanitizer แยกต่างหากอยู่แล้ว ซึ่งลบ `<tool_call>`, `<function_calls>`, `<system-reminder>`, `<previous_response>` และ scaffolding runtime ภายในที่คล้ายกันซึ่งรั่วไหล ออกจากคำตอบที่ผู้ใช้มองเห็น ณ ขอบเขตการส่งมอบช่องทางสุดท้าย sanitizer ของเนื้อหาภายนอกคือคู่ขาเข้าของสิ่งนั้น

สิ่งนี้ไม่ได้แทนที่การ hardening อื่นบนหน้านี้ — `dmPolicy`, allowlists, การอนุมัติ exec, sandboxing และ `contextVisibility` ยังคงทำงานหลักอยู่ สิ่งนี้ปิด bypass เฉพาะจุดหนึ่งในชั้น tokenizer ต่อ stack แบบ self-hosted ที่ส่งต่อข้อความผู้ใช้พร้อม special tokens โดยไม่แก้ไข

## flag bypass เนื้อหาภายนอกที่ไม่ปลอดภัย

OpenClaw มี flag bypass ที่ระบุชัดเจนซึ่งปิดการห่อหุ้มความปลอดภัยของเนื้อหาภายนอก:

- `hooks.mappings[].allowUnsafeExternalContent`
- `hooks.gmail.allowUnsafeExternalContent`
- ฟิลด์ payload ของ Cron `allowUnsafeExternalContent`

คำแนะนำ:

- ปล่อยให้ unset/false ใน production
- เปิดใช้ชั่วคราวเฉพาะสำหรับการ debug ที่มีขอบเขตแคบเท่านั้น
- หากเปิดใช้ ให้แยก agent นั้นออก (sandbox + เครื่องมือน้อยที่สุด + namespace ของ session เฉพาะ)

หมายเหตุความเสี่ยงของ hooks:

- payload ของ Hook เป็นเนื้อหาที่ไม่น่าเชื่อถือ แม้การส่งมอบจะมาจากระบบที่คุณควบคุม (เนื้อหา mail/docs/web สามารถพก prompt injection ได้)
- tier ของโมเดลที่อ่อนจะเพิ่มความเสี่ยงนี้ สำหรับ automation ที่ขับเคลื่อนด้วย hook แนะนำให้ใช้ tier ของโมเดลสมัยใหม่ที่แข็งแกร่งและกำหนดนโยบายเครื่องมือให้เข้มงวด (`tools.profile: "messaging"` หรือเข้มงวดกว่า) รวมถึง sandboxing เมื่อทำได้

### Prompt injection ไม่จำเป็นต้องมี DM สาธารณะ

แม้ว่า **มีเพียงคุณเท่านั้น** ที่ส่งข้อความถึงบอตได้ prompt injection ก็ยังเกิดขึ้นได้ผ่าน
**เนื้อหาที่ไม่น่าเชื่อถือ** ใดๆ ที่บอตอ่าน (ผลลัพธ์ web search/fetch, หน้า browser,
อีเมล, docs, ไฟล์แนบ, logs/code ที่วางเข้ามา) กล่าวอีกอย่าง: ผู้ส่งไม่ใช่
พื้นที่โจมตีเดียว; **ตัวเนื้อหาเอง** สามารถพกคำสั่งที่เป็นปฏิปักษ์มาได้

เมื่อเปิดใช้เครื่องมือ ความเสี่ยงทั่วไปคือการ exfiltrate context หรือ trigger
การเรียกเครื่องมือ ลด blast radius โดย:

- ใช้ **reader agent** แบบอ่านอย่างเดียวหรือปิดเครื่องมือเพื่อสรุปเนื้อหาที่ไม่น่าเชื่อถือ
  แล้วส่งสรุปนั้นต่อไปยัง agent หลักของคุณ
- ปิด `web_search` / `web_fetch` / `browser` สำหรับ agent ที่เปิดใช้เครื่องมือ เว้นแต่จำเป็น
- สำหรับอินพุต URL ของ OpenResponses (`input_file` / `input_image`) ให้ตั้งค่า
  `gateway.http.endpoints.responses.files.urlAllowlist` และ
  `gateway.http.endpoints.responses.images.urlAllowlist` ให้เข้มงวด และตั้ง `maxUrlParts` ให้ต่ำ
  allowlists ที่ว่างเปล่าจะถือว่าไม่ได้ตั้งค่า; ใช้ `files.allowUrl: false` / `images.allowUrl: false`
  หากคุณต้องการปิดการ fetch URL ทั้งหมด
- สำหรับอินพุตไฟล์ของ OpenResponses ข้อความ `input_file` ที่ decode แล้วจะยังถูก inject ในฐานะ
  **เนื้อหาภายนอกที่ไม่น่าเชื่อถือ** อย่าพึ่งพาว่าข้อความในไฟล์น่าเชื่อถือเพียงเพราะ
  Gateway decode ภายในเครื่องแล้ว block ที่ inject เข้ามายังคงมี marker ขอบเขต
  `<<<EXTERNAL_UNTRUSTED_CONTENT ...>>>` อย่างชัดเจน พร้อม metadata `Source: External`
  แม้ path นี้จะละเว้น banner `SECURITY NOTICE:` ที่ยาวกว่า
- การห่อหุ้มด้วย marker แบบเดียวกันจะถูกใช้เมื่อ media-understanding ดึงข้อความ
  จากเอกสารที่แนบมาก่อนผนวกข้อความนั้นเข้ากับ media prompt
- เปิดใช้ sandboxing และ allowlists เครื่องมือที่เข้มงวดสำหรับ agent ใดๆ ที่แตะอินพุตที่ไม่น่าเชื่อถือ
- เก็บความลับออกจาก prompts; ส่งผ่าน env/config บนโฮสต์ Gateway แทน

### backend LLM แบบ self-hosted

backend แบบ self-hosted ที่เข้ากันได้กับ OpenAI เช่น vLLM, SGLang, TGI, LM Studio,
หรือ stack tokenizer ของ Hugging Face แบบกำหนดเอง อาจแตกต่างจาก provider แบบ hosted ในวิธี
จัดการ special token ของ chat-template หาก backend tokenize สตริง literal
เช่น `<|im_start|>`, `<|start_header_id|>` หรือ `<start_of_turn>` เป็น
token เชิงโครงสร้างของ chat-template ภายในเนื้อหาผู้ใช้ ข้อความที่ไม่น่าเชื่อถืออาจพยายาม
ปลอมขอบเขต role ที่ชั้น tokenizer ได้

OpenClaw จะลบ literal ของ special-token ที่พบบ่อยตามตระกูลโมเดลออกจาก
เนื้อหาภายนอกที่ถูกห่อหุ้มก่อน dispatch ไปยังโมเดล เปิดการห่อหุ้มเนื้อหาภายนอกไว้
และแนะนำให้ใช้การตั้งค่า backend ที่แยกหรือ escape special token
ในเนื้อหาที่ผู้ใช้ให้มาเมื่อมีให้ใช้ provider แบบ hosted เช่น OpenAI
และ Anthropic ใช้การทำความสะอาดฝั่ง request ของตนเองอยู่แล้ว

### ความแข็งแกร่งของโมเดล (หมายเหตุด้านความปลอดภัย)

ความต้านทานต่อ prompt injection **ไม่** สม่ำเสมอในทุก tier ของโมเดล โดยทั่วไปโมเดลที่เล็กกว่า/ถูกกว่าไวต่อการใช้เครื่องมือผิดวัตถุประสงค์และการ hijack คำสั่งมากกว่า โดยเฉพาะภายใต้ prompt ที่เป็นปฏิปักษ์

<Warning>
สำหรับ agent ที่เปิดใช้เครื่องมือหรือ agent ที่อ่านเนื้อหาที่ไม่น่าเชื่อถือ ความเสี่ยง prompt-injection เมื่อใช้โมเดลที่เก่ากว่า/เล็กกว่ามักสูงเกินไป อย่าเรียกใช้ workload เหล่านั้นบน tier โมเดลที่อ่อน
</Warning>

คำแนะนำ:

- **ใช้โมเดลรุ่นล่าสุดและ tier ดีที่สุด** สำหรับบอตใดๆ ที่สามารถเรียกใช้เครื่องมือหรือแตะไฟล์/เครือข่ายได้
- **อย่าใช้ tier ที่เก่ากว่า/อ่อนกว่า/เล็กกว่า** สำหรับ agent ที่เปิดใช้เครื่องมือหรือ inbox ที่ไม่น่าเชื่อถือ; ความเสี่ยง prompt-injection สูงเกินไป
- หากจำเป็นต้องใช้โมเดลที่เล็กกว่า ให้ **ลด blast radius** (เครื่องมืออ่านอย่างเดียว, sandboxing ที่แข็งแกร่ง, การเข้าถึง filesystem น้อยที่สุด, allowlists ที่เข้มงวด)
- เมื่อเรียกใช้โมเดลขนาดเล็ก ให้ **เปิดใช้ sandboxing สำหรับทุก session** และ **ปิด web_search/web_fetch/browser** เว้นแต่อินพุตจะถูกควบคุมอย่างเข้มงวด
- สำหรับผู้ช่วยส่วนตัวแบบ chat-only ที่มีอินพุตที่เชื่อถือได้และไม่มีเครื่องมือ โมเดลที่เล็กกว่ามักใช้ได้

## Reasoning และเอาต์พุต verbose ในกลุ่ม

`/reasoning`, `/verbose` และ `/trace` สามารถเปิดเผย reasoning ภายใน เอาต์พุตเครื่องมือ
หรือ diagnostics ของ Plugin ที่
ไม่ได้ตั้งใจให้เผยแพร่ในช่องสาธารณะ ในบริบทกลุ่ม ให้ถือว่าสิ่งเหล่านี้เป็น **debug
เท่านั้น** และปิดไว้ เว้นแต่คุณต้องการใช้อย่างชัดเจน

คำแนะนำ:

- ปิด `/reasoning`, `/verbose` และ `/trace` ไว้ในห้องสาธารณะ
- หากเปิดใช้ ให้ทำเฉพาะใน DM ที่เชื่อถือได้หรือห้องที่ควบคุมอย่างเข้มงวด
- จำไว้ว่า: เอาต์พุต verbose และ trace อาจรวม args ของเครื่องมือ, URLs, diagnostics ของ Plugin และข้อมูลที่โมเดลเห็น

## ตัวอย่างการ hardening configuration

### สิทธิ์ไฟล์

เก็บ config + state ให้เป็นส่วนตัวบนโฮสต์ Gateway:

- `~/.openclaw/openclaw.json`: `600` (ผู้ใช้อ่าน/เขียนเท่านั้น)
- `~/.openclaw`: `700` (ผู้ใช้เท่านั้น)

`openclaw doctor` สามารถเตือนและเสนอให้ปรับสิทธิ์เหล่านี้ให้เข้มงวดขึ้นได้

### การเปิดเผยเครือข่าย (bind, port, firewall)

Gateway multiplex **WebSocket + HTTP** บน port เดียว:

- ค่าเริ่มต้น: `18789`
- Config/flags/env: `gateway.port`, `--port`, `OPENCLAW_GATEWAY_PORT`

พื้นผิว HTTP นี้รวมถึง Control UI และ canvas host:

- Control UI (SPA assets) (base path เริ่มต้น `/`)
- Canvas host: `/__openclaw__/canvas/` และ `/__openclaw__/a2ui/` (HTML/JS ใดๆ; ให้ถือว่าเป็นเนื้อหาที่ไม่น่าเชื่อถือ)

หากคุณโหลดเนื้อหา canvas ใน browser ปกติ ให้ปฏิบัติต่อมันเหมือนหน้าเว็บที่ไม่น่าเชื่อถืออื่นๆ:

- อย่าเปิดเผย canvas host ต่อเครือข่าย/ผู้ใช้ที่ไม่น่าเชื่อถือ
- อย่าทำให้เนื้อหา canvas ใช้ origin เดียวกับพื้นผิวเว็บที่มีสิทธิ์สูง เว้นแต่คุณเข้าใจผลกระทบอย่างถ่องแท้

โหมด bind ควบคุมว่า Gateway รับฟังที่ใด:

- `gateway.bind: "loopback"` (ค่าเริ่มต้น): เฉพาะ client ภายในเครื่องเท่านั้นที่เชื่อมต่อได้
- bind ที่ไม่ใช่ loopback (`"lan"`, `"tailnet"`, `"custom"`) จะขยายพื้นที่โจมตี ใช้เฉพาะพร้อม auth ของ gateway (shared token/password หรือ trusted proxy ที่กำหนดค่าอย่างถูกต้อง) และ firewall จริงเท่านั้น

แนวทางทั่วไป:

- แนะนำให้ใช้ Tailscale Serve แทน LAN binds (Serve ทำให้ Gateway อยู่บน loopback และ Tailscale จัดการการเข้าถึง)
- หากจำเป็นต้อง bind ไปยัง LAN ให้ตั้ง firewall ให้ port อนุญาตเฉพาะ allowlist ของ source IPs ที่แคบ; อย่า port-forward แบบกว้าง
- อย่าเปิดเผย Gateway แบบไม่ผ่าน authentication บน `0.0.0.0`

### การ publish port ของ Docker ด้วย UFW

หากคุณเรียกใช้ OpenClaw ด้วย Docker บน VPS โปรดจำว่า container ports ที่ publish
(`-p HOST:CONTAINER` หรือ Compose `ports:`) จะถูก route ผ่าน forwarding
chains ของ Docker ไม่ใช่เฉพาะกฎ `INPUT` ของโฮสต์เท่านั้น

เพื่อให้ traffic ของ Docker สอดคล้องกับนโยบาย firewall ของคุณ ให้บังคับใช้กฎใน
`DOCKER-USER` (chain นี้ถูกประเมินก่อนกฎ accept ของ Docker เอง)
บน distro สมัยใหม่หลายตัว `iptables`/`ip6tables` ใช้ frontend `iptables-nft`
และยังคงนำกฎเหล่านี้ไปใช้กับ backend nftables

ตัวอย่าง allowlist ขั้นต่ำ (IPv4):

```bash
# /etc/ufw/after.rules (append as its own *filter section)
*filter
:DOCKER-USER - [0:0]
-A DOCKER-USER -m conntrack --ctstate ESTABLISHED,RELATED -j RETURN
-A DOCKER-USER -s 127.0.0.0/8 -j RETURN
-A DOCKER-USER -s 10.0.0.0/8 -j RETURN
-A DOCKER-USER -s 172.16.0.0/12 -j RETURN
-A DOCKER-USER -s 192.168.0.0/16 -j RETURN
-A DOCKER-USER -s 100.64.0.0/10 -j RETURN
-A DOCKER-USER -p tcp --dport 80 -j RETURN
-A DOCKER-USER -p tcp --dport 443 -j RETURN
-A DOCKER-USER -m conntrack --ctstate NEW -j DROP
-A DOCKER-USER -j RETURN
COMMIT
```

IPv6 มีตารางแยกต่างหาก เพิ่มนโยบายที่ตรงกันใน `/etc/ufw/after6.rules` หาก
เปิดใช้ Docker IPv6

หลีกเลี่ยงการ hardcode ชื่อ interface เช่น `eth0` ใน snippet ของ docs ชื่อ interface
แตกต่างกันไปตาม image ของ VPS (`ens3`, `enp*` เป็นต้น) และการไม่ตรงกันอาจทำให้
ข้ามกฎ deny ของคุณโดยไม่ตั้งใจ

การตรวจสอบอย่างรวดเร็วหลัง reload:

```bash
ufw reload
iptables -S DOCKER-USER
ip6tables -S DOCKER-USER
nmap -sT -p 1-65535 <public-ip> --open
```

port ภายนอกที่คาดไว้ควรมีเฉพาะสิ่งที่คุณตั้งใจเปิดเผยเท่านั้น (สำหรับ setup ส่วนใหญ่:
SSH + port ของ reverse proxy ของคุณ)

### การค้นพบ mDNS/Bonjour

Gateway จะ broadcast การมีอยู่ของตนผ่าน mDNS (`_openclaw-gw._tcp` บน port 5353) สำหรับการค้นพบอุปกรณ์ภายในเครื่อง ในโหมดเต็ม สิ่งนี้รวม TXT records ที่อาจเปิดเผยรายละเอียดการปฏิบัติงาน:

- `cliPath`: พาธระบบไฟล์แบบเต็มไปยังไบนารี CLI (เปิดเผยชื่อผู้ใช้และตำแหน่งที่ติดตั้ง)
- `sshPort`: ประกาศความพร้อมใช้งาน SSH บนโฮสต์
- `displayName`, `lanHost`: ข้อมูลชื่อโฮสต์

**ข้อพิจารณาด้านความปลอดภัยในการปฏิบัติการ:** การเผยแพร่รายละเอียดโครงสร้างพื้นฐานทำให้ทุกคนบนเครือข่ายภายในสอดแนมได้ง่ายขึ้น แม้แต่ข้อมูลที่ดู "ไม่เป็นอันตราย" เช่น พาธระบบไฟล์และความพร้อมใช้งาน SSH ก็ช่วยให้ผู้โจมตีทำแผนผังสภาพแวดล้อมของคุณได้

**คำแนะนำ:**

1. **โหมดขั้นต่ำ** (ค่าเริ่มต้น แนะนำสำหรับ Gateway ที่เปิดเผย): ละเว้นฟิลด์ละเอียดอ่อนจากการเผยแพร่ mDNS:

   ```json5
   {
     discovery: {
       mdns: { mode: "minimal" },
     },
   }
   ```

2. **ปิดทั้งหมด** หากคุณไม่ต้องการการค้นหาอุปกรณ์ภายในเครื่อง:

   ```json5
   {
     discovery: {
       mdns: { mode: "off" },
     },
   }
   ```

3. **โหมดเต็ม** (เลือกเปิดใช้): รวม `cliPath` + `sshPort` ในระเบียน TXT:

   ```json5
   {
     discovery: {
       mdns: { mode: "full" },
     },
   }
   ```

4. **ตัวแปรสภาพแวดล้อม** (ทางเลือก): ตั้งค่า `OPENCLAW_DISABLE_BONJOUR=1` เพื่อปิด mDNS โดยไม่ต้องเปลี่ยนการกำหนดค่า

ในโหมดขั้นต่ำ Gateway จะยังเผยแพร่ข้อมูลเพียงพอสำหรับการค้นหาอุปกรณ์ (`role`, `gatewayPort`, `transport`) แต่ละเว้น `cliPath` และ `sshPort` แอปที่ต้องการข้อมูลพาธ CLI สามารถดึงข้อมูลผ่านการเชื่อมต่อ WebSocket ที่ยืนยันตัวตนแล้วแทนได้

### จำกัดการเข้าถึง Gateway WebSocket (การยืนยันตัวตนภายในเครื่อง)

การยืนยันตัวตนของ Gateway **จำเป็นตามค่าเริ่มต้น** หากไม่ได้กำหนดค่าพาธการยืนยันตัวตน Gateway ที่ถูกต้อง
Gateway จะปฏิเสธการเชื่อมต่อ WebSocket (ปิดเมื่อผิดพลาด)

การเริ่มต้นใช้งานจะสร้างโทเค็นตามค่าเริ่มต้น (แม้สำหรับ loopback) ดังนั้น
ไคลเอนต์ภายในเครื่องต้องยืนยันตัวตน

ตั้งค่าโทเค็นเพื่อให้ไคลเอนต์ WS **ทั้งหมด** ต้องยืนยันตัวตน:

```json5
{
  gateway: {
    auth: { mode: "token", token: "your-token" },
  },
}
```

Doctor สามารถสร้างให้คุณได้: `openclaw doctor --generate-gateway-token`.

<Note>
`gateway.remote.token` และ `gateway.remote.password` เป็นแหล่งข้อมูลประจำตัวของไคลเอนต์ ข้อมูลเหล่านี้ **ไม่ได้** ปกป้องการเข้าถึง WS ภายในเครื่องด้วยตัวเอง พาธการเรียกภายในเครื่องสามารถใช้ `gateway.remote.*` เป็นค่าทดแทนได้เฉพาะเมื่อไม่ได้ตั้งค่า `gateway.auth.*` หาก `gateway.auth.token` หรือ `gateway.auth.password` ถูกกำหนดอย่างชัดเจนผ่าน SecretRef และแก้ค่าไม่ได้ การแก้ค่าจะล้มเหลวแบบปิด (ไม่มี remote fallback มาปิดบัง)
</Note>
ทางเลือก: ปักหมุด TLS ระยะไกลด้วย `gateway.remote.tlsFingerprint` เมื่อใช้ `wss://`.
Plaintext `ws://` เป็น loopback-only ตามค่าเริ่มต้น สำหรับพาธเครือข่ายส่วนตัวที่เชื่อถือได้
ให้ตั้งค่า `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` บนกระบวนการไคลเอนต์เป็น
break-glass สิ่งนี้ตั้งใจให้เป็นเฉพาะสภาพแวดล้อมของกระบวนการ ไม่ใช่คีย์การกำหนดค่า
`openclaw.json`
การจับคู่อุปกรณ์มือถือและเส้นทาง Gateway แบบกำหนดเองหรือสแกนบน Android เข้มงวดกว่า:
ยอมรับ cleartext สำหรับ loopback แต่ private-LAN, link-local, `.local` และ
ชื่อโฮสต์แบบไม่มีจุดต้องใช้ TLS เว้นแต่คุณจะเลือกใช้พาธ cleartext บนเครือข่ายส่วนตัวที่เชื่อถือได้อย่างชัดเจน

การจับคู่อุปกรณ์ภายในเครื่อง:

- การจับคู่อุปกรณ์จะได้รับการอนุมัติโดยอัตโนมัติสำหรับการเชื่อมต่อ local loopback โดยตรง เพื่อให้
  ไคลเอนต์บนโฮสต์เดียวกันลื่นไหล
- OpenClaw ยังมีพาธ self-connect เฉพาะ backend/container-local แบบแคบสำหรับ
  โฟลว์ตัวช่วยที่ใช้ shared-secret ที่เชื่อถือได้
- การเชื่อมต่อ Tailnet และ LAN รวมถึง tailnet bind บนโฮสต์เดียวกัน จะถูกถือว่าเป็น
  ระยะไกลสำหรับการจับคู่และยังต้องได้รับอนุมัติ
- หลักฐาน forwarded-header บนคำขอ loopback จะทำให้คุณสมบัติความเป็น loopback
  locality ไม่ผ่าน การอนุมัติอัตโนมัติแบบ metadata-upgrade ถูกจำกัดขอบเขตอย่างแคบ ดู
  [การจับคู่ Gateway](/th/gateway/pairing) สำหรับกฎทั้งสองข้อ

โหมดการยืนยันตัวตน:

- `gateway.auth.mode: "token"`: โทเค็น bearer ที่ใช้ร่วมกัน (แนะนำสำหรับการตั้งค่าส่วนใหญ่)
- `gateway.auth.mode: "password"`: การยืนยันตัวตนด้วยรหัสผ่าน (ควรตั้งค่าผ่าน env: `OPENCLAW_GATEWAY_PASSWORD`)
- `gateway.auth.mode: "trusted-proxy"`: เชื่อถือ reverse proxy ที่รับรู้ตัวตนให้ยืนยันตัวตนผู้ใช้และส่งตัวตนผ่านส่วนหัว (ดู [การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth))

รายการตรวจสอบการหมุนเวียน (โทเค็น/รหัสผ่าน):

1. สร้าง/ตั้งค่าความลับใหม่ (`gateway.auth.token` หรือ `OPENCLAW_GATEWAY_PASSWORD`)
2. รีสตาร์ท Gateway (หรือรีสตาร์ทแอป macOS หากแอปกำกับดูแล Gateway)
3. อัปเดตไคลเอนต์ระยะไกลใด ๆ (`gateway.remote.token` / `.password` บนเครื่องที่เรียกเข้า Gateway)
4. ตรวจสอบว่าคุณไม่สามารถเชื่อมต่อด้วยข้อมูลประจำตัวเดิมได้อีก

### ส่วนหัวตัวตนของ Tailscale Serve

เมื่อ `gateway.auth.allowTailscale` เป็น `true` (ค่าเริ่มต้นสำหรับ Serve) OpenClaw
จะยอมรับส่วนหัวตัวตนของ Tailscale Serve (`tailscale-user-login`) สำหรับการยืนยันตัวตน Control
UI/WebSocket OpenClaw ตรวจสอบตัวตนโดยแก้ที่อยู่
`x-forwarded-for` ผ่านเดมอน Tailscale ภายในเครื่อง (`tailscale whois`)
และจับคู่กับส่วนหัว สิ่งนี้จะทำงานเฉพาะกับคำขอที่เข้าถึง loopback
และมี `x-forwarded-for`, `x-forwarded-proto` และ `x-forwarded-host` ตามที่
Tailscale แทรกเข้ามา
สำหรับพาธการตรวจสอบตัวตนแบบอะซิงโครนัสนี้ ความพยายามที่ล้มเหลวสำหรับ `{scope, ip}` เดียวกัน
จะถูกจัดลำดับก่อนที่ตัวจำกัดจะบันทึกความล้มเหลว ดังนั้นการลองซ้ำที่ผิดพลาดพร้อมกัน
จากไคลเอนต์ Serve เดียวกันอาจล็อกความพยายามครั้งที่สองทันที
แทนที่จะแข่งผ่านไปเป็นความไม่ตรงกันธรรมดาสองครั้ง
ปลายทาง HTTP API (เช่น `/v1/*`, `/tools/invoke` และ `/api/channels/*`)
**ไม่** ใช้การยืนยันตัวตนด้วยส่วนหัวตัวตนของ Tailscale แต่ยังคงทำตาม
โหมดการยืนยันตัวตน HTTP ที่กำหนดค่าของ Gateway

หมายเหตุขอบเขตสำคัญ:

- การยืนยันตัวตน HTTP bearer ของ Gateway มีผลเท่ากับการเข้าถึงระดับผู้ปฏิบัติการแบบทั้งหมดหรือไม่เลย
- ถือว่าข้อมูลประจำตัวที่เรียก `/v1/chat/completions`, `/v1/responses` หรือ `/api/channels/*` ได้ เป็นความลับของผู้ปฏิบัติการที่มีสิทธิ์เต็มสำหรับ Gateway นั้น
- บนพื้นผิว HTTP ที่เข้ากันได้กับ OpenAI การยืนยันตัวตน bearer แบบ shared-secret จะกู้คืนขอบเขตผู้ปฏิบัติการเริ่มต้นเต็มรูปแบบ (`operator.admin`, `operator.approvals`, `operator.pairing`, `operator.read`, `operator.talk.secrets`, `operator.write`) และความหมายของเจ้าของสำหรับ agent turns; ค่า `x-openclaw-scopes` ที่แคบกว่าไม่ลดทอนพาธ shared-secret นั้น
- ความหมายของขอบเขตแบบต่อคำขอบน HTTP จะใช้เฉพาะเมื่อคำขอมาจากโหมดที่มีตัวตน เช่น การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้ หรือ `gateway.auth.mode="none"` บนทางเข้าที่เป็นส่วนตัว
- ในโหมดที่มีตัวตนเหล่านั้น การละเว้น `x-openclaw-scopes` จะย้อนกลับไปใช้ชุดขอบเขตเริ่มต้นของผู้ปฏิบัติการตามปกติ ส่งส่วนหัวนี้อย่างชัดเจนเมื่อคุณต้องการชุดขอบเขตที่แคบกว่า
- `/tools/invoke` ทำตามกฎ shared-secret เดียวกัน: การยืนยันตัวตน bearer ด้วยโทเค็น/รหัสผ่านถูกถือว่าเป็นการเข้าถึงผู้ปฏิบัติการเต็มรูปแบบที่นั่นด้วย ขณะที่โหมดที่มีตัวตนยังคงเคารพขอบเขตที่ประกาศ
- อย่าแชร์ข้อมูลประจำตัวเหล่านี้กับผู้เรียกที่ไม่น่าเชื่อถือ ควรใช้ Gateway แยกตามขอบเขตความเชื่อถือ

**สมมติฐานความเชื่อถือ:** การยืนยันตัวตน Serve แบบไม่มีโทเค็นถือว่าโฮสต์ Gateway เชื่อถือได้
อย่าถือว่าสิ่งนี้เป็นการป้องกันกระบวนการบนโฮสต์เดียวกันที่เป็นอันตราย หากโค้ดภายในเครื่องที่ไม่น่าเชื่อถือ
อาจทำงานบนโฮสต์ Gateway ให้ปิด `gateway.auth.allowTailscale`
และบังคับใช้การยืนยันตัวตน shared-secret อย่างชัดเจนด้วย `gateway.auth.mode: "token"` หรือ
`"password"`

**กฎความปลอดภัย:** อย่าส่งต่อส่วนหัวเหล่านี้จาก reverse proxy ของคุณเอง หาก
คุณยุติ TLS หรือใช้ proxy หน้า Gateway ให้ปิด
`gateway.auth.allowTailscale` และใช้การยืนยันตัวตน shared-secret (`gateway.auth.mode:
"token"` หรือ `"password"`) หรือ [การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth)
แทน

พร็อกซีที่เชื่อถือได้:

- หากคุณยุติ TLS หน้า Gateway ให้ตั้งค่า `gateway.trustedProxies` เป็น IP ของพร็อกซีของคุณ
- OpenClaw จะเชื่อถือ `x-forwarded-for` (หรือ `x-real-ip`) จาก IP เหล่านั้นเพื่อกำหนด IP ไคลเอนต์สำหรับการตรวจสอบการจับคู่ภายในเครื่องและการตรวจสอบ HTTP auth/local
- ตรวจสอบให้แน่ใจว่าพร็อกซีของคุณ **เขียนทับ** `x-forwarded-for` และบล็อกการเข้าถึงพอร์ต Gateway โดยตรง

ดู [Tailscale](/th/gateway/tailscale) และ [ภาพรวมเว็บ](/th/web)

### การควบคุมเบราว์เซอร์ผ่าน node host (แนะนำ)

หาก Gateway ของคุณอยู่ระยะไกลแต่เบราว์เซอร์ทำงานบนเครื่องอื่น ให้รัน **node host**
บนเครื่องเบราว์เซอร์และให้ Gateway proxy การกระทำของเบราว์เซอร์ (ดู [เครื่องมือเบราว์เซอร์](/th/tools/browser))
ถือว่าการจับคู่ node เป็นเหมือนการเข้าถึงระดับผู้ดูแลระบบ

รูปแบบที่แนะนำ:

- ให้ Gateway และ node host อยู่บน tailnet เดียวกัน (Tailscale)
- จับคู่ node อย่างตั้งใจ ปิดการกำหนดเส้นทาง proxy ของเบราว์เซอร์หากคุณไม่ต้องการ

หลีกเลี่ยง:

- การเปิดเผยพอร์ต relay/control ผ่าน LAN หรืออินเทอร์เน็ตสาธารณะ
- Tailscale Funnel สำหรับปลายทางควบคุมเบราว์เซอร์ (การเปิดเผยต่อสาธารณะ)

### ความลับบนดิสก์

ให้ถือว่าทุกอย่างภายใต้ `~/.openclaw/` (หรือ `$OPENCLAW_STATE_DIR/`) อาจมีความลับหรือข้อมูลส่วนตัว:

- `openclaw.json`: การกำหนดค่าอาจมีโทเค็น (Gateway, Gateway ระยะไกล), การตั้งค่า provider และ allowlist
- `credentials/**`: ข้อมูลประจำตัวของช่องทาง (ตัวอย่าง: ข้อมูลประจำตัว WhatsApp), allowlist การจับคู่, การนำเข้า OAuth แบบเดิม
- `agents/<agentId>/agent/auth-profiles.json`: คีย์ API, โปรไฟล์โทเค็น, โทเค็น OAuth และ `keyRef`/`tokenRef` แบบไม่บังคับ
- `agents/<agentId>/agent/codex-home/**`: บัญชี app-server ของ Codex ต่อ agent, การกำหนดค่า, Skills, plugins, สถานะเธรด native และการวินิจฉัย
- `secrets.json` (ไม่บังคับ): payload ความลับที่สำรองด้วยไฟล์ ซึ่งใช้โดย provider SecretRef แบบ `file` (`secrets.providers`)
- `agents/<agentId>/agent/auth.json`: ไฟล์ความเข้ากันได้แบบเดิม รายการ `api_key` แบบคงที่จะถูกล้างเมื่อค้นพบ
- `agents/<agentId>/sessions/**`: ทรานสคริปต์เซสชัน (`*.jsonl`) + เมตาดาตาการกำหนดเส้นทาง (`sessions.json`) ที่อาจมีข้อความส่วนตัวและผลลัพธ์เครื่องมือ
- แพ็กเกจ Plugin ที่บันเดิลมา: Plugin ที่ติดตั้งแล้ว (รวมถึง `node_modules/` ของ Plugin เหล่านั้น)
- `sandboxes/**`: เวิร์กสเปซ sandbox ของเครื่องมือ อาจสะสมสำเนาของไฟล์ที่คุณอ่าน/เขียนภายใน sandbox

เคล็ดลับการเสริมความปลอดภัย:

- รักษาสิทธิ์ให้เข้มงวด (`700` บนไดเรกทอรี, `600` บนไฟล์)
- ใช้การเข้ารหัสดิสก์ทั้งลูกบนโฮสต์ Gateway
- ควรใช้บัญชีผู้ใช้ OS เฉพาะสำหรับ Gateway หากโฮสต์ถูกใช้งานร่วมกัน

### ไฟล์ `.env` ของเวิร์กสเปซ

OpenClaw โหลดไฟล์ `.env` ภายในเวิร์กสเปซสำหรับ agents และเครื่องมือ แต่จะไม่ยอมให้ไฟล์เหล่านั้นเขียนทับการควบคุมรันไทม์ของ Gateway อย่างเงียบ ๆ

- คีย์ใด ๆ ที่ขึ้นต้นด้วย `OPENCLAW_*` จะถูกบล็อกจากไฟล์ `.env` ของเวิร์กสเปซที่ไม่น่าเชื่อถือ
- การตั้งค่าปลายทางช่องทางสำหรับ Matrix, Mattermost, IRC และ Synology Chat ก็ถูกบล็อกจากการเขียนทับ `.env` ของเวิร์กสเปซเช่นกัน ดังนั้นเวิร์กสเปซที่โคลนมาจึงไม่สามารถเปลี่ยนเส้นทางการรับส่งข้อมูลของ bundled connector ผ่านการกำหนดค่าปลายทางภายในเครื่องได้ คีย์ env ของปลายทาง (เช่น `MATRIX_HOMESERVER`, `MATTERMOST_URL`, `IRC_HOST`, `SYNOLOGY_CHAT_INCOMING_URL`) ต้องมาจากสภาพแวดล้อมกระบวนการ Gateway หรือ `env.shellEnv` ไม่ใช่จาก `.env` ที่โหลดจากเวิร์กสเปซ
- การบล็อกเป็นแบบ fail-closed: ตัวแปรควบคุมรันไทม์ใหม่ที่เพิ่มในรุ่นอนาคตไม่สามารถสืบทอดจาก `.env` ที่เช็กอินหรือผู้โจมตีจัดหาให้ได้ คีย์จะถูกละเว้นและ Gateway จะคงค่าของตัวเองไว้
- ตัวแปรสภาพแวดล้อมของกระบวนการ/OS ที่เชื่อถือได้ (shell ของ Gateway เอง, หน่วย launchd/systemd, app bundle) ยังคงมีผล — สิ่งนี้จำกัดเฉพาะการโหลดไฟล์ `.env` เท่านั้น

เหตุผล: ไฟล์ `.env` ของเวิร์กสเปซมักอยู่ข้างโค้ด agent, ถูก commit โดยไม่ตั้งใจ หรือถูกเครื่องมือเขียน การบล็อกคำนำหน้า `OPENCLAW_*` ทั้งหมดหมายความว่าการเพิ่มแฟล็ก `OPENCLAW_*` ใหม่ในภายหลังจะไม่มีทางถดถอยไปเป็นการสืบทอดจากสถานะเวิร์กสเปซอย่างเงียบ ๆ

### บันทึกและทรานสคริปต์ (การปกปิดและการเก็บรักษา)

บันทึกและทรานสคริปต์อาจรั่วไหลข้อมูลละเอียดอ่อนได้ แม้เมื่อการควบคุมการเข้าถึงถูกต้อง:

- บันทึก Gateway อาจมีสรุปเครื่องมือ ข้อผิดพลาด และ URL
- ทรานสคริปต์เซสชันอาจมีความลับที่วางไว้ เนื้อหาไฟล์ ผลลัพธ์คำสั่ง และลิงก์

คำแนะนำ:

- เปิดการปกปิดบันทึกและทรานสคริปต์ไว้ (`logging.redactSensitive: "tools"`; ค่าเริ่มต้น)
- เพิ่มรูปแบบที่กำหนดเองสำหรับสภาพแวดล้อมของคุณผ่าน `logging.redactPatterns` (โทเค็น, ชื่อโฮสต์, URL ภายใน)
- เมื่อแชร์การวินิจฉัย ควรใช้ `openclaw status --all` (วางได้, ปกปิดความลับแล้ว) แทนบันทึกดิบ
- ลบทิ้งทรานสคริปต์เซสชันและไฟล์บันทึกเก่าหากคุณไม่ต้องการเก็บไว้นาน

รายละเอียด: [การบันทึก](/th/gateway/logging)

### หมายเลขแยกต่างหาก (WhatsApp, Signal, Telegram)

สำหรับช่องทางที่ใช้หมายเลขโทรศัพท์ ให้พิจารณารัน AI ของคุณบนหมายเลขโทรศัพท์แยกจากหมายเลขส่วนตัว:

- หมายเลขส่วนตัว: บทสนทนาของคุณยังคงเป็นส่วนตัว
- หมายเลขบอต: AI จัดการรายการเหล่านี้ พร้อมขอบเขตที่เหมาะสม

### โหมดอ่านอย่างเดียว (ผ่าน sandbox และเครื่องมือ)

คุณสามารถสร้างโปรไฟล์อ่านอย่างเดียวได้โดยผสาน:

- `agents.defaults.sandbox.workspaceAccess: "ro"` (หรือ `"none"` เพื่อไม่ให้เข้าถึง workspace)
- รายการอนุญาต/ปฏิเสธเครื่องมือที่บล็อก `write`, `edit`, `apply_patch`, `exec`, `process` ฯลฯ

ตัวเลือกเพิ่มความแข็งแรงเพิ่มเติม:

- `tools.exec.applyPatch.workspaceOnly: true` (ค่าเริ่มต้น): ทำให้แน่ใจว่า `apply_patch` ไม่สามารถเขียน/ลบนอกไดเรกทอรี workspace ได้แม้เมื่อปิด sandboxing ตั้งเป็น `false` เฉพาะเมื่อคุณตั้งใจให้ `apply_patch` แตะไฟล์นอก workspace
- `tools.fs.workspaceOnly: true` (ไม่บังคับ): จำกัดพาธ `read`/`write`/`edit`/`apply_patch` และพาธโหลดภาพ prompt แบบ native อัตโนมัติให้อยู่ในไดเรกทอรี workspace (มีประโยชน์หากวันนี้คุณอนุญาตพาธแบบสัมบูรณ์และต้องการ guardrail เดียว)
- จำกัดรากของระบบไฟล์ให้แคบ: หลีกเลี่ยงรากที่กว้าง เช่น ไดเรกทอรี home ของคุณ สำหรับ agent workspaces/sandbox workspaces รากที่กว้างอาจเปิดเผยไฟล์ภายในเครื่องที่อ่อนไหว (เช่น state/config ใต้ `~/.openclaw`) ให้เครื่องมือระบบไฟล์

### baseline ที่ปลอดภัย (คัดลอก/วาง)

คอนฟิก “ค่าเริ่มต้นที่ปลอดภัย” หนึ่งชุดที่ทำให้ Gateway เป็นส่วนตัว ต้องมีการจับคู่ DM และหลีกเลี่ยงบอตกลุ่มที่เปิดทำงานตลอดเวลา:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    port: 18789,
    auth: { mode: "token", token: "your-long-random-token" },
  },
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } },
    },
  },
}
```

หากคุณต้องการให้การเรียกใช้เครื่องมือ “ปลอดภัยขึ้นตามค่าเริ่มต้น” ด้วย ให้เพิ่ม sandbox + ปฏิเสธเครื่องมืออันตรายสำหรับ agent ที่ไม่ใช่เจ้าของ (ตัวอย่างอยู่ด้านล่างใน “โปรไฟล์การเข้าถึงต่อ agent”)

baseline ในตัวสำหรับรอบ agent ที่ขับเคลื่อนด้วยแชต: ผู้ส่งที่ไม่ใช่เจ้าของไม่สามารถใช้เครื่องมือ `cron` หรือ `gateway` ได้

## Sandboxing (แนะนำ)

เอกสารเฉพาะ: [Sandboxing](/th/gateway/sandboxing)

สองแนวทางที่เสริมกัน:

- **รัน Gateway ทั้งหมดใน Docker** (ขอบเขตคอนเทนเนอร์): [Docker](/th/install/docker)
- **Tool sandbox** (`agents.defaults.sandbox`, host gateway + เครื่องมือที่แยกด้วย sandbox; Docker เป็น backend เริ่มต้น): [Sandboxing](/th/gateway/sandboxing)

<Note>
เพื่อป้องกันการเข้าถึงข้าม agent ให้คง `agents.defaults.sandbox.scope` ไว้ที่ `"agent"` (ค่าเริ่มต้น) หรือ `"session"` เพื่อการแยกต่อ session ที่เข้มงวดยิ่งขึ้น `scope: "shared"` ใช้คอนเทนเนอร์หรือ workspace เดียว
</Note>

พิจารณาการเข้าถึง agent workspace ภายใน sandbox ด้วย:

- `agents.defaults.sandbox.workspaceAccess: "none"` (ค่าเริ่มต้น) ทำให้ agent workspace เป็นพื้นที่ห้ามเข้าถึง; เครื่องมือรันกับ sandbox workspace ใต้ `~/.openclaw/sandboxes`
- `agents.defaults.sandbox.workspaceAccess: "ro"` เมานต์ agent workspace แบบอ่านอย่างเดียวที่ `/agent` (ปิดใช้งาน `write`/`edit`/`apply_patch`)
- `agents.defaults.sandbox.workspaceAccess: "rw"` เมานต์ agent workspace แบบอ่าน/เขียนที่ `/workspace`
- `sandbox.docker.binds` เพิ่มเติมจะถูกตรวจสอบกับพาธต้นทางที่ normalize และ canonicalize แล้ว เทคนิค parent-symlink และ alias home แบบ canonical ยัง fail closed หาก resolve เข้าไปในรากที่ถูกบล็อก เช่น `/etc`, `/var/run` หรือไดเรกทอรีข้อมูลรับรองใต้ home ของ OS

<Warning>
`tools.elevated` คือช่องทางหลบเลี่ยง baseline ระดับ global ที่รัน exec นอก sandbox โฮสต์ที่มีผลคือ `gateway` ตามค่าเริ่มต้น หรือ `node` เมื่อกำหนดเป้าหมาย exec เป็น `node` จำกัด `tools.elevated.allowFrom` ให้เข้มงวด และอย่าเปิดใช้ให้คนแปลกหน้า คุณสามารถจำกัด elevated ต่อ agent เพิ่มเติมได้ผ่าน `agents.list[].tools.elevated` ดู [โหมด Elevated](/th/tools/elevated)
</Warning>

### guardrail สำหรับการมอบหมาย sub-agent

หากคุณอนุญาตเครื่องมือ session ให้ถือว่าการรัน sub-agent ที่ถูกมอบหมายเป็นการตัดสินใจอีกขอบเขตหนึ่ง:

- ปฏิเสธ `sessions_spawn` เว้นแต่ agent จำเป็นต้องมอบหมายงานจริงๆ
- จำกัด `agents.defaults.subagents.allowAgents` และ override `agents.list[].subagents.allowAgents` ต่อ agent ให้เฉพาะ target agents ที่ทราบว่าปลอดภัย
- สำหรับ workflow ใดๆ ที่ต้องยังอยู่ใน sandbox ให้เรียก `sessions_spawn` ด้วย `sandbox: "require"` (ค่าเริ่มต้นคือ `inherit`)
- `sandbox: "require"` จะล้มเหลวอย่างรวดเร็วเมื่อ runtime ลูกเป้าหมายไม่ได้อยู่ใน sandbox

## ความเสี่ยงของการควบคุมเบราว์เซอร์

การเปิดใช้งานการควบคุมเบราว์เซอร์ทำให้โมเดลสามารถขับเบราว์เซอร์จริงได้
หากโปรไฟล์เบราว์เซอร์นั้นมี session ที่ล็อกอินอยู่แล้ว โมเดลสามารถ
เข้าถึงบัญชีและข้อมูลเหล่านั้นได้ ให้ถือว่าโปรไฟล์เบราว์เซอร์เป็น **สถานะที่อ่อนไหว**:

- ควรใช้โปรไฟล์เฉพาะสำหรับ agent (โปรไฟล์ `openclaw` เริ่มต้น)
- หลีกเลี่ยงการชี้ agent ไปยังโปรไฟล์ส่วนตัวที่คุณใช้ประจำวัน
- ปิดการควบคุมเบราว์เซอร์ของโฮสต์สำหรับ agent ที่อยู่ใน sandbox เว้นแต่คุณไว้ใจ agent เหล่านั้น
- API ควบคุมเบราว์เซอร์แบบ standalone loopback รองรับเฉพาะ shared-secret auth
  (gateway token bearer auth หรือรหัสผ่าน gateway) ไม่ใช้
  trusted-proxy หรือ header ระบุตัวตนของ Tailscale Serve
- ถือว่าดาวน์โหลดจากเบราว์เซอร์เป็น input ที่ไม่น่าเชื่อถือ; ควรใช้ไดเรกทอรีดาวน์โหลดที่แยกไว้
- ปิด browser sync/password managers ในโปรไฟล์ agent หากเป็นไปได้ (ลด blast radius)
- สำหรับ remote gateways ให้ถือว่า “การควบคุมเบราว์เซอร์” เทียบเท่ากับ “การเข้าถึงแบบ operator” ต่อทุกอย่างที่โปรไฟล์นั้นเข้าถึงได้
- ให้ Gateway และโฮสต์ node อยู่ใน tailnet เท่านั้น; หลีกเลี่ยงการเปิดเผยพอร์ตควบคุมเบราว์เซอร์ต่อ LAN หรืออินเทอร์เน็ตสาธารณะ
- ปิดการ routing proxy ของเบราว์เซอร์เมื่อคุณไม่ต้องใช้ (`gateway.nodes.browser.mode="off"`)
- โหมด existing-session ของ Chrome MCP **ไม่ได้** “ปลอดภัยกว่า”; มันสามารถทำหน้าที่เป็นคุณในสิ่งใดก็ตามที่โปรไฟล์ Chrome บนโฮสต์นั้นเข้าถึงได้

### นโยบาย Browser SSRF (เข้มงวดโดยค่าเริ่มต้น)

นโยบายการนำทางเบราว์เซอร์ของ OpenClaw เข้มงวดโดยค่าเริ่มต้น: ปลายทาง private/internal ยังคงถูกบล็อก เว้นแต่คุณ opt in อย่างชัดเจน

- ค่าเริ่มต้น: ไม่ได้ตั้ง `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` ดังนั้นการนำทางเบราว์เซอร์จะยังบล็อกปลายทาง private/internal/special-use
- alias แบบเก่า: `browser.ssrfPolicy.allowPrivateNetwork` ยังยอมรับเพื่อความเข้ากันได้
- โหมด opt-in: ตั้ง `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` เพื่ออนุญาตปลายทาง private/internal/special-use
- ในโหมดเข้มงวด ให้ใช้ `hostnameAllowlist` (patterns เช่น `*.example.com`) และ `allowedHostnames` (ข้อยกเว้น host แบบ exact รวมถึงชื่อที่ถูกบล็อก เช่น `localhost`) สำหรับข้อยกเว้นที่ชัดเจน
- การนำทางจะถูกตรวจสอบก่อน request และพยายามตรวจสอบซ้ำแบบ best-effort บน URL `http(s)` สุดท้ายหลังการนำทาง เพื่อลด pivot ที่อิงกับ redirect

ตัวอย่างนโยบายเข้มงวด:

```json5
{
  browser: {
    ssrfPolicy: {
      dangerouslyAllowPrivateNetwork: false,
      hostnameAllowlist: ["*.example.com", "example.com"],
      allowedHostnames: ["localhost"],
    },
  },
}
```

## โปรไฟล์การเข้าถึงต่อ agent (multi-agent)

ด้วยการ routing แบบ multi-agent แต่ละ agent สามารถมี sandbox + นโยบายเครื่องมือของตัวเอง:
ใช้สิ่งนี้เพื่อให้ **เข้าถึงเต็มรูปแบบ**, **อ่านอย่างเดียว** หรือ **ไม่ให้เข้าถึง** ต่อ agent
ดู [Multi-Agent Sandbox & Tools](/th/tools/multi-agent-sandbox-tools) สำหรับรายละเอียดทั้งหมด
และกฎ precedence

กรณีใช้งานทั่วไป:

- agent ส่วนตัว: เข้าถึงเต็มรูปแบบ, ไม่มี sandbox
- agent สำหรับครอบครัว/งาน: อยู่ใน sandbox + เครื่องมืออ่านอย่างเดียว
- agent สาธารณะ: อยู่ใน sandbox + ไม่มีเครื่องมือระบบไฟล์/shell

### ตัวอย่าง: เข้าถึงเต็มรูปแบบ (ไม่มี sandbox)

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        workspace: "~/.openclaw/workspace-personal",
        sandbox: { mode: "off" },
      },
    ],
  },
}
```

### ตัวอย่าง: เครื่องมืออ่านอย่างเดียว + workspace อ่านอย่างเดียว

```json5
{
  agents: {
    list: [
      {
        id: "family",
        workspace: "~/.openclaw/workspace-family",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "ro",
        },
        tools: {
          allow: ["read"],
          deny: ["write", "edit", "apply_patch", "exec", "process", "browser"],
        },
      },
    ],
  },
}
```

### ตัวอย่าง: ไม่มีการเข้าถึงระบบไฟล์/shell (อนุญาตการส่งข้อความผ่าน provider)

```json5
{
  agents: {
    list: [
      {
        id: "public",
        workspace: "~/.openclaw/workspace-public",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "none",
        },
        // Session tools can reveal sensitive data from transcripts. By default OpenClaw limits these tools
        // to the current session + spawned subagent sessions, but you can clamp further if needed.
        // See `tools.sessions.visibility` in the configuration reference.
        tools: {
          sessions: { visibility: "tree" }, // self | tree | agent | all
          allow: [
            "sessions_list",
            "sessions_history",
            "sessions_send",
            "sessions_spawn",
            "session_status",
            "whatsapp",
            "telegram",
            "slack",
            "discord",
          ],
          deny: [
            "read",
            "write",
            "edit",
            "apply_patch",
            "exec",
            "process",
            "browser",
            "canvas",
            "nodes",
            "cron",
            "gateway",
            "image",
          ],
        },
      },
    ],
  },
}
```

## การตอบสนองต่อเหตุการณ์

หาก AI ของคุณทำสิ่งที่ไม่ดี:

### ควบคุม

1. **หยุดมัน:** หยุดแอป macOS (หากแอปดูแล Gateway) หรือยุติ process `openclaw gateway` ของคุณ
2. **ปิดช่องทางเปิดเผย:** ตั้ง `gateway.bind: "loopback"` (หรือปิดใช้งาน Tailscale Funnel/Serve) จนกว่าคุณจะเข้าใจสิ่งที่เกิดขึ้น
3. **หยุดการเข้าถึงชั่วคราว:** เปลี่ยน DM/กลุ่มที่เสี่ยงเป็น `dmPolicy: "disabled"` / ต้องมีการ mention และลบ entry อนุญาตทั้งหมด `"*"` หากคุณมีอยู่

### หมุนเวียน (ถือว่าถูก compromise หาก secrets รั่วไหล)

1. หมุนเวียน Gateway auth (`gateway.auth.token` / `OPENCLAW_GATEWAY_PASSWORD`) และ restart
2. หมุนเวียน remote client secrets (`gateway.remote.token` / `.password`) บนเครื่องใดๆ ที่สามารถเรียก Gateway ได้
3. หมุนเวียนข้อมูลรับรอง provider/API (ข้อมูลรับรอง WhatsApp, tokens ของ Slack/Discord, keys ของ model/API ใน `auth-profiles.json` และค่า payload ของ encrypted secrets เมื่อใช้งาน)

### Audit

1. ตรวจสอบ Gateway logs: `/tmp/openclaw/openclaw-YYYY-MM-DD.log` (หรือ `logging.file`)
2. ตรวจทาน transcript ที่เกี่ยวข้อง: `~/.openclaw/agents/<agentId>/sessions/*.jsonl`
3. ตรวจทานการเปลี่ยนแปลง config ล่าสุด (อะไรก็ตามที่อาจขยายการเข้าถึง: `gateway.bind`, `gateway.auth`, นโยบาย dm/group, `tools.elevated`, การเปลี่ยนแปลง plugin)
4. รัน `openclaw security audit --deep` อีกครั้ง และยืนยันว่า findings ระดับ critical ได้รับการแก้ไขแล้ว

### รวบรวมเพื่อรายงาน

- Timestamp, OS ของโฮสต์ gateway + เวอร์ชัน OpenClaw
- transcript ของ session + log tail สั้นๆ (หลัง redacting)
- สิ่งที่ผู้โจมตีส่ง + สิ่งที่ agent ทำ
- Gateway ถูกเปิดเผยนอกเหนือจาก loopback หรือไม่ (LAN/Tailscale Funnel/Serve)

## การสแกน Secret

CI รัน hook pre-commit `detect-private-key` เหนือ repository หาก
ล้มเหลว ให้ลบหรือหมุนเวียน key material ที่ commit ไปแล้ว จากนั้น reproduce ในเครื่อง:

```bash
pre-commit run --all-files detect-private-key
```
