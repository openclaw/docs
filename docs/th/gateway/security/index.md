  **โมเดลความไว้วางใจของผู้ช่วยส่วนตัว** คำแนะนำนี้ถือว่ามีขอบเขต
  โอเปอเรเตอร์ที่เชื่อถือได้หนึ่งขอบเขตต่อ Gateway (โมเดลผู้ใช้คนเดียว,
  ผู้ช่วยส่วนตัว) OpenClaw **ไม่ใช่** ขอบเขตความปลอดภัยแบบหลายผู้เช่าที่เป็นปรปักษ์
  สำหรับผู้ใช้ที่เป็นคู่กรณีหลายคนซึ่งใช้เอเจนต์หรือ Gateway เดียวกันร่วมกัน
  หากคุณต้องการการทำงานแบบความไว้วางใจผสมหรือผู้ใช้ที่เป็นปรปักษ์ ให้แยกขอบเขต
  ความไว้วางใจ (Gateway + ข้อมูลประจำตัวแยกกัน และในอุดมคติคือผู้ใช้ระบบปฏิบัติการหรือโฮสต์แยกกัน)
</Warning>

## กำหนดขอบเขตก่อน: โมเดลความปลอดภัยของผู้ช่วยส่วนตัว

คำแนะนำด้านความปลอดภัยของ OpenClaw ถือว่าเป็นการปรับใช้แบบ **ผู้ช่วยส่วนตัว**: ขอบเขตโอเปอเรเตอร์ที่เชื่อถือได้หนึ่งขอบเขต และอาจมีเอเจนต์จำนวนมาก

- สถานะความปลอดภัยที่รองรับ: ผู้ใช้/ขอบเขตความไว้วางใจหนึ่งรายการต่อ Gateway (แนะนำให้ใช้ผู้ใช้ระบบปฏิบัติการ/โฮสต์/VPS หนึ่งรายการต่อขอบเขต)
- ไม่ใช่ขอบเขตความปลอดภัยที่รองรับ: Gateway/เอเจนต์ร่วมหนึ่งรายการที่ใช้โดยผู้ใช้ที่ไม่ไว้วางใจกันหรือเป็นปรปักษ์ต่อกัน
- หากจำเป็นต้องแยกผู้ใช้ที่เป็นปรปักษ์ ให้แยกตามขอบเขตความไว้วางใจ (Gateway + ข้อมูลประจำตัวแยกกัน และในอุดมคติคือผู้ใช้/โฮสต์ระบบปฏิบัติการแยกกัน)
- หากผู้ใช้ที่ไม่ไว้วางใจกันหลายคนสามารถส่งข้อความถึงเอเจนต์ที่เปิดใช้เครื่องมือไว้ได้ ให้ถือว่าพวกเขาใช้สิทธิ์เครื่องมือที่มอบหมายชุดเดียวกันสำหรับเอเจนต์นั้นร่วมกัน

หน้านี้อธิบายการเพิ่มความแข็งแกร่ง **ภายในโมเดลนั้น** ไม่ได้อ้างว่ามีการแยกแบบหลายผู้เช่าที่เป็นปรปักษ์บน Gateway ร่วมหนึ่งรายการ

## ตรวจสอบอย่างรวดเร็ว: `openclaw security audit`

ดูเพิ่มเติม: [การยืนยันอย่างเป็นทางการ (โมเดลความปลอดภัย)](/th/security/formal-verification)

เรียกใช้คำสั่งนี้เป็นประจำ (โดยเฉพาะหลังจากเปลี่ยนการกำหนดค่าหรือเปิดเผยพื้นผิวเครือข่าย):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

`security audit --fix` ถูกจำกัดขอบเขตอย่างตั้งใจ: มันเปลี่ยนนโยบายกลุ่มแบบเปิดที่พบบ่อยให้เป็นรายการอนุญาต กู้คืน `logging.redactSensitive: "tools"` ทำให้สิทธิ์ของสถานะ/การกำหนดค่า/ไฟล์รวมเข้มงวดขึ้น และใช้การรีเซ็ต ACL ของ Windows แทน POSIX `chmod` เมื่อทำงานบน Windows

มันแจ้งเตือนจุดพลาดที่พบบ่อย (การเปิดเผยการยืนยันตัวตนของ Gateway, การเปิดเผยการควบคุมเบราว์เซอร์, รายการอนุญาตที่มีสิทธิ์สูง, สิทธิ์ของระบบไฟล์, การอนุมัติ exec ที่ผ่อนปรน และการเปิดเผยเครื่องมือในช่องทางแบบเปิด)

OpenClaw เป็นทั้งผลิตภัณฑ์และการทดลอง: คุณกำลังเชื่อมพฤติกรรมของโมเดลแนวหน้ากับพื้นผิวการส่งข้อความจริงและเครื่องมือจริง **ไม่มีการตั้งค่าใดที่ "ปลอดภัยสมบูรณ์แบบ"** เป้าหมายคือการตัดสินใจอย่างรอบคอบเกี่ยวกับ:

- ใครสามารถคุยกับบอตของคุณได้
- บอตได้รับอนุญาตให้ทำงานที่ใด
- บอตแตะต้องอะไรได้บ้าง

เริ่มด้วยสิทธิ์เข้าถึงที่เล็กที่สุดซึ่งยังใช้งานได้ แล้วค่อยขยายเมื่อคุณมั่นใจมากขึ้น

### การปรับใช้และความไว้วางใจในโฮสต์

OpenClaw ถือว่าโฮสต์และขอบเขตการกำหนดค่าเชื่อถือได้:

- หากบางคนสามารถแก้ไขสถานะ/การกำหนดค่าของโฮสต์ Gateway (`~/.openclaw` รวมถึง `openclaw.json`) ให้ถือว่าคนนั้นเป็นโอเปอเรเตอร์ที่เชื่อถือได้
- การเรียกใช้ Gateway หนึ่งรายการสำหรับโอเปอเรเตอร์หลายคนที่ไม่ไว้วางใจกันหรือเป็นปรปักษ์ต่อกัน **ไม่ใช่การตั้งค่าที่แนะนำ**
- สำหรับทีมที่มีความไว้วางใจผสม ให้แยกขอบเขตความไว้วางใจด้วย Gateway แยกกัน (หรืออย่างน้อยที่สุดคือผู้ใช้/โฮสต์ระบบปฏิบัติการแยกกัน)
- ค่าเริ่มต้นที่แนะนำ: ผู้ใช้หนึ่งคนต่อเครื่อง/โฮสต์ (หรือ VPS), Gateway หนึ่งรายการสำหรับผู้ใช้นั้น และเอเจนต์หนึ่งรายการขึ้นไปใน Gateway นั้น
- ภายในอินสแตนซ์ Gateway หนึ่งรายการ การเข้าถึงของโอเปอเรเตอร์ที่ยืนยันตัวตนแล้วคือบทบาทระดับ control plane ที่เชื่อถือได้ ไม่ใช่บทบาทผู้เช่ารายผู้ใช้
- ตัวระบุเซสชัน (`sessionKey`, ID เซสชัน, ป้ายกำกับ) เป็นตัวเลือกเส้นทาง ไม่ใช่โทเค็นการอนุญาต
- หากหลายคนสามารถส่งข้อความถึงเอเจนต์ที่เปิดใช้เครื่องมือไว้ได้ แต่ละคนสามารถควบคุมชุดสิทธิ์เดียวกันนั้นได้ การแยกเซสชัน/หน่วยความจำรายผู้ใช้ช่วยเรื่องความเป็นส่วนตัว แต่ไม่ได้เปลี่ยนเอเจนต์ร่วมให้เป็นการอนุญาตโฮสต์รายผู้ใช้

### การดำเนินการกับไฟล์อย่างปลอดภัย

OpenClaw ใช้ `@openclaw/fs-safe` สำหรับการเข้าถึงไฟล์ที่จำกัดตาม root, การเขียนแบบอะตอมิก, การแตกไฟล์เก็บถาวร, พื้นที่ทำงานชั่วคราว และตัวช่วยไฟล์ลับ ค่าเริ่มต้นของ OpenClaw ปิดตัวช่วย POSIX Python แบบเลือกใช้ของ fs-safe ไว้; ตั้ง `OPENCLAW_FS_SAFE_PYTHON_MODE=auto` หรือ `require` เฉพาะเมื่อคุณต้องการการเพิ่มความแข็งแกร่งในการกลายพันธุ์แบบ fd-relative เพิ่มเติม และสามารถรองรับรันไทม์ Python ได้

รายละเอียด: [การดำเนินการกับไฟล์อย่างปลอดภัย](/th/gateway/security/secure-file-operations)

### พื้นที่ทำงาน Slack ที่ใช้ร่วมกัน: ความเสี่ยงจริง

หาก "ทุกคนใน Slack สามารถส่งข้อความถึงบอตได้" ความเสี่ยงหลักคือสิทธิ์เครื่องมือที่ถูกมอบหมาย:

- ผู้ส่งที่ได้รับอนุญาตทุกคนสามารถกระตุ้นการเรียกเครื่องมือ (`exec`, เบราว์เซอร์, เครื่องมือเครือข่าย/ไฟล์) ภายในนโยบายของเอเจนต์ได้;
- การแทรก prompt/เนื้อหาจากผู้ส่งหนึ่งคนสามารถทำให้เกิดการกระทำที่ส่งผลต่อสถานะ อุปกรณ์ หรือเอาต์พุตที่ใช้ร่วมกันได้;
- หากเอเจนต์ร่วมหนึ่งรายการมีข้อมูลประจำตัว/ไฟล์ที่ละเอียดอ่อน ผู้ส่งที่ได้รับอนุญาตทุกคนอาจสามารถขับเคลื่อนการดึงข้อมูลออกผ่านการใช้เครื่องมือได้

ใช้เอเจนต์/Gateway แยกกันพร้อมเครื่องมือขั้นต่ำสำหรับเวิร์กโฟลว์ของทีม; เก็บเอเจนต์ที่มีข้อมูลส่วนตัวไว้เป็นส่วนตัว

### เอเจนต์ที่บริษัทใช้ร่วมกัน: รูปแบบที่ยอมรับได้

รูปแบบนี้ยอมรับได้เมื่อทุกคนที่ใช้เอเจนต์นั้นอยู่ในขอบเขตความไว้วางใจเดียวกัน (เช่น ทีมบริษัทหนึ่งทีม) และเอเจนต์ถูกจำกัดขอบเขตไว้เฉพาะงานธุรกิจอย่างเคร่งครัด

- เรียกใช้บนเครื่อง/VM/คอนเทนเนอร์เฉพาะ;
- ใช้ผู้ใช้ระบบปฏิบัติการเฉพาะ + เบราว์เซอร์/โปรไฟล์/บัญชีเฉพาะสำหรับรันไทม์นั้น;
- อย่าลงชื่อเข้ารันไทม์นั้นด้วยบัญชี Apple/Google ส่วนตัว หรือโปรไฟล์ตัวจัดการรหัสผ่าน/เบราว์เซอร์ส่วนตัว

หากคุณผสมตัวตนส่วนตัวกับตัวตนบริษัทบนรันไทม์เดียวกัน คุณจะทำให้การแยกนั้นหายไปและเพิ่มความเสี่ยงต่อการเปิดเผยข้อมูลส่วนตัว

## แนวคิดความไว้วางใจของ Gateway และ Node

ให้ถือว่า Gateway และ Node เป็นโดเมนความไว้วางใจของโอเปอเรเตอร์เดียวกัน โดยมีบทบาทต่างกัน:

- **Gateway** คือ control plane และพื้นผิวนโยบาย (`gateway.auth`, นโยบายเครื่องมือ, การกำหนดเส้นทาง)
- **Node** คือพื้นผิวการดำเนินการระยะไกลที่จับคู่กับ Gateway นั้น (คำสั่ง, การกระทำกับอุปกรณ์, ความสามารถเฉพาะโฮสต์)
- ผู้เรียกที่ยืนยันตัวตนกับ Gateway แล้วได้รับความไว้วางใจในขอบเขต Gateway หลังจากจับคู่แล้ว การกระทำของ Node จะเป็นการกระทำของโอเปอเรเตอร์ที่เชื่อถือได้บน Node นั้น
- ระดับขอบเขตของโอเปอเรเตอร์และการตรวจสอบขณะอนุมัติสรุปไว้ใน
  [ขอบเขตของโอเปอเรเตอร์](/th/gateway/operator-scopes)
- ไคลเอนต์ backend แบบ loopback โดยตรงที่ยืนยันตัวตนด้วยโทเค็น/รหัสผ่าน Gateway ที่ใช้ร่วมกันสามารถทำ RPC ภายใน control plane ได้โดยไม่ต้องแสดงตัวตนอุปกรณ์ของผู้ใช้ นี่ไม่ใช่การเลี่ยงการจับคู่ระยะไกลหรือเบราว์เซอร์: ไคลเอนต์เครือข่าย, ไคลเอนต์ Node, ไคลเอนต์โทเค็นอุปกรณ์ และตัวตนอุปกรณ์ที่ระบุอย่างชัดเจนยังคงผ่านการจับคู่และการบังคับใช้การยกระดับขอบเขต
- `sessionKey` คือการเลือกเส้นทาง/บริบท ไม่ใช่การยืนยันตัวตนรายผู้ใช้
- การอนุมัติ exec (รายการอนุญาต + การถาม) เป็นราวกั้นสำหรับเจตนาของโอเปอเรเตอร์ ไม่ใช่การแยกแบบหลายผู้เช่าที่เป็นปรปักษ์
- ค่าเริ่มต้นของผลิตภัณฑ์ OpenClaw สำหรับการตั้งค่าแบบโอเปอเรเตอร์เดี่ยวที่เชื่อถือได้คืออนุญาตให้ host exec บน `gateway`/`node` ทำงานได้โดยไม่มี prompt ขออนุมัติ (`security="full"`, `ask="off"` เว้นแต่คุณจะทำให้เข้มงวดขึ้น) ค่าเริ่มต้นนั้นเป็น UX ที่ตั้งใจไว้ ไม่ใช่ช่องโหว่โดยตัวมันเอง
- การอนุมัติ exec ผูกกับบริบทคำขอที่แน่นอนและ operand ไฟล์ภายในเครื่องโดยตรงแบบ best-effort; มันไม่ได้สร้างโมเดลเชิงความหมายให้ทุกเส้นทาง loader ของรันไทม์/interpreter ใช้ sandboxing และการแยกโฮสต์สำหรับขอบเขตที่แข็งแรง

หากคุณต้องการแยกผู้ใช้ที่เป็นปรปักษ์ ให้แยกขอบเขตความไว้วางใจตามผู้ใช้/โฮสต์ระบบปฏิบัติการ และเรียกใช้ Gateway แยกกัน

## เมทริกซ์ขอบเขตความไว้วางใจ

ใช้สิ่งนี้เป็นโมเดลด่วนเมื่อคัดแยกความเสี่ยง:

| ขอบเขตหรือการควบคุม                                      | ความหมาย                                          | การเข้าใจผิดที่พบบ่อย                                                          |
| --------------------------------------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------- |
| `gateway.auth` (token/password/trusted-proxy/device auth) | ยืนยันตัวตนผู้เรียกไปยัง API ของ Gateway         | "ต้องมีลายเซ็นรายข้อความในทุกเฟรมจึงจะปลอดภัย"                               |
| `sessionKey`                                              | คีย์กำหนดเส้นทางสำหรับการเลือกบริบท/เซสชัน       | "คีย์เซสชันคือขอบเขตการยืนยันตัวตนของผู้ใช้"                                  |
| ราวกั้น prompt/เนื้อหา                                    | ลดความเสี่ยงจากการใช้โมเดลในทางที่ผิด            | "การแทรก prompt เพียงอย่างเดียวพิสูจน์การเลี่ยง auth"                         |
| `canvas.eval` / browser evaluate                          | ความสามารถของโอเปอเรเตอร์โดยเจตนาเมื่อเปิดใช้    | "primitive ของ JS eval ใด ๆ เป็นช่องโหว่โดยอัตโนมัติในโมเดลความไว้วางใจนี้"   |
| เชลล์ `!` ของ TUI ภายในเครื่อง                            | การดำเนินการภายในเครื่องที่โอเปอเรเตอร์กระตุ้นอย่างชัดเจน | "คำสั่งอำนวยความสะดวกของเชลล์ภายในเครื่องคือการแทรกระยะไกล"                  |
| การจับคู่ Node และคำสั่ง Node                             | การดำเนินการระยะไกลระดับโอเปอเรเตอร์บนอุปกรณ์ที่จับคู่แล้ว | "การควบคุมอุปกรณ์ระยะไกลควรถูกถือเป็นการเข้าถึงของผู้ใช้ที่ไม่ไว้วางใจโดยค่าเริ่มต้น" |
| `gateway.nodes.pairing.autoApproveCidrs`                  | นโยบายการลงทะเบียน Node บนเครือข่ายที่เชื่อถือได้แบบเลือกใช้ | "รายการอนุญาตที่ปิดไว้โดยค่าเริ่มต้นคือช่องโหว่การจับคู่อัตโนมัติ"            |

## ไม่ใช่ช่องโหว่โดยการออกแบบ

<Accordion title="Common findings that are out of scope">

รูปแบบเหล่านี้ถูกรายงานบ่อย และมักถูกปิดโดยไม่ดำเนินการ เว้นแต่จะมีการสาธิตการเลี่ยงขอบเขตจริง:

- เชนที่เป็นเพียงการแทรก prompt โดยไม่มีการเลี่ยงนโยบาย, auth หรือ sandbox
- ข้อกล่าวอ้างที่สมมติการทำงานแบบหลายผู้เช่าที่เป็นปรปักษ์บนโฮสต์หรือการกำหนดค่าร่วมหนึ่งรายการ
- ข้อกล่าวอ้างที่จัดประเภทการเข้าถึงเส้นทางอ่านปกติของโอเปอเรเตอร์ (เช่น `sessions.list` / `sessions.preview` / `chat.history`) เป็น IDOR ในการตั้งค่า Gateway ร่วม
- ข้อค้นพบในการปรับใช้แบบเฉพาะ localhost (เช่น HSTS บน Gateway แบบเฉพาะ loopback)
- ข้อค้นพบลายเซ็น Webhook ขาเข้าของ Discord สำหรับเส้นทางขาเข้าที่ไม่มีอยู่ใน repo นี้
- รายงานที่ถือว่าเมทาดาทาการจับคู่ Node เป็นชั้นอนุมัติรายคำสั่งชั้นที่สองที่ซ่อนอยู่สำหรับ `system.run` ในขณะที่ขอบเขตการดำเนินการจริงยังคงเป็นนโยบายคำสั่ง Node แบบ global ของ Gateway บวกกับการอนุมัติ exec ของ Node เอง
- รายงานที่ถือว่า `gateway.nodes.pairing.autoApproveCidrs` ที่กำหนดค่าไว้เป็นช่องโหว่โดยตัวมันเอง การตั้งค่านี้ปิดอยู่โดยค่าเริ่มต้น ต้องมีรายการ CIDR/IP อย่างชัดเจน ใช้กับการจับคู่ครั้งแรกของ `role: node` ที่ไม่มีขอบเขตที่ร้องขอเท่านั้น และไม่อนุมัติโอเปอเรเตอร์/เบราว์เซอร์/Control UI, WebChat, การยกระดับบทบาท, การยกระดับขอบเขต, การเปลี่ยนเมทาดาทา, การเปลี่ยน public-key หรือเส้นทาง header trusted-proxy ของ loopback บนโฮสต์เดียวกันโดยอัตโนมัติ เว้นแต่จะเปิดใช้ auth แบบ trusted-proxy ของ loopback ไว้อย่างชัดเจน
- ข้อค้นพบ "ไม่มีการอนุญาตรายผู้ใช้" ที่ถือว่า `sessionKey` เป็นโทเค็น auth

## ค่าพื้นฐานที่เพิ่มความแข็งแกร่งใน 60 วินาที

ใช้ค่าพื้นฐานนี้ก่อน แล้วค่อยเปิดใช้เครื่องมืออีกครั้งแบบเลือกเฉพาะต่อเอเจนต์ที่เชื่อถือได้:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

การตั้งค่านี้ทำให้ Gateway เป็นแบบเฉพาะภายในเครื่อง แยก DM และปิดใช้งานเครื่องมือ control-plane/รันไทม์โดยค่าเริ่มต้น

## กฎด่วนสำหรับกล่องขาเข้าที่ใช้ร่วมกัน

หากมีมากกว่าหนึ่งคนสามารถ DM บอตของคุณได้:

- ตั้ง `session.dmScope: "per-channel-peer"` (หรือ `"per-account-channel-peer"` สำหรับช่องทางหลายบัญชี)
- ใช้ `dmPolicy: "pairing"` หรือรายการอนุญาตที่เข้มงวดต่อไป
- อย่าผสม DM ที่ใช้ร่วมกันกับสิทธิ์เข้าถึงเครื่องมือแบบกว้าง
- สิ่งนี้เพิ่มความแข็งแกร่งให้กล่องขาเข้าแบบร่วมมือ/ใช้ร่วมกัน แต่ไม่ได้ออกแบบมาเพื่อเป็นการแยกผู้เช่าร่วมที่เป็นปรปักษ์เมื่อผู้ใช้ใช้สิทธิ์เขียนโฮสต์/การกำหนดค่าร่วมกัน

## โมเดลการมองเห็นบริบท

OpenClaw แยกแนวคิดสองอย่าง:

- **การอนุญาตการกระตุ้น**: ใครสามารถกระตุ้นเอเจนต์ได้ (`dmPolicy`, `groupPolicy`, รายการอนุญาต, ประตูการกล่าวถึง)
- **การมองเห็นบริบท**: บริบทเสริมใดถูกฉีดเข้าไปในอินพุตของโมเดล (เนื้อหาการตอบกลับ, ข้อความที่อ้างอิง, ประวัติ thread, เมทาดาทาที่ส่งต่อ)

รายการอนุญาตควบคุมการกระตุ้นและการอนุญาตคำสั่ง การตั้งค่า `contextVisibility` ควบคุมวิธีกรองบริบทเสริม (การตอบกลับที่อ้างอิง, รากของ thread, ประวัติที่ดึงมา):

- `contextVisibility: "all"` (ค่าเริ่มต้น) เก็บบริบทเสริมไว้ตามที่ได้รับ
- `contextVisibility: "allowlist"` กรองบริบทเสริมเพื่อส่งเฉพาะผู้ส่งที่ได้รับอนุญาตโดยการตรวจสอบ allowlist ที่ใช้งานอยู่
- `contextVisibility: "allowlist_quote"` ทำงานเหมือน `allowlist` แต่ยังเก็บการตอบกลับที่ถูกอ้างอิงไว้อย่างชัดเจนหนึ่งรายการ

ตั้งค่า `contextVisibility` ต่อช่องทางหรือต่อห้อง/การสนทนา ดูรายละเอียดการตั้งค่าได้ที่ [แชทกลุ่ม](/th/channels/groups#context-visibility-and-allowlists)

คำแนะนำการคัดแยกรายงานด้านความปลอดภัย:

- ข้อกล่าวอ้างที่แสดงเพียงว่า "โมเดลสามารถเห็นข้อความที่ถูกอ้างอิงหรือข้อความย้อนหลังจากผู้ส่งที่ไม่ได้อยู่ใน allowlist" เป็นข้อค้นพบด้านการเพิ่มความแข็งแรงที่แก้ไขได้ด้วย `contextVisibility` ไม่ใช่การข้ามขอบเขต auth หรือ sandbox โดยตัวมันเอง
- หากจะถือว่ามีผลกระทบด้านความปลอดภัย รายงานยังต้องแสดงให้เห็นการข้าม trust boundary อย่างชัดเจน (auth, นโยบาย, sandbox, การอนุมัติ หรือขอบเขตอื่นที่มีเอกสารกำกับ)

## สิ่งที่ audit ตรวจสอบ (ภาพรวมระดับสูง)

- **การเข้าถึงขาเข้า** (นโยบาย DM, นโยบายกลุ่ม, allowlist): คนแปลกหน้าสามารถเรียกใช้บอตได้หรือไม่
- **รัศมีผลกระทบของเครื่องมือ** (เครื่องมือสิทธิ์ยกระดับ + ห้องที่เปิดอยู่): prompt injection อาจกลายเป็นการกระทำกับ shell/ไฟล์/เครือข่ายได้หรือไม่
- **ความคลาดเคลื่อนของการอนุมัติ Exec** (`security=full`, `autoAllowSkills`, interpreter allowlist ที่ไม่มี `strictInlineEval`): guardrail สำหรับ host-exec ยังทำงานตามที่คุณคิดอยู่หรือไม่
  - `security="full"` เป็นคำเตือน posture แบบกว้าง ไม่ใช่หลักฐานของบั๊ก เป็นค่าเริ่มต้นที่เลือกไว้สำหรับการตั้งค่าผู้ช่วยส่วนตัวที่ไว้วางใจได้; จำกัดให้เข้มขึ้นเฉพาะเมื่อ threat model ของคุณต้องใช้ guardrail แบบการอนุมัติหรือ allowlist
- **การเปิดเผยทางเครือข่าย** (การ bind/auth ของ Gateway, Tailscale Serve/Funnel, token auth ที่อ่อนหรือสั้น)
- **การเปิดเผยการควบคุมเบราว์เซอร์** (remote nodes, relay ports, remote CDP endpoints)
- **สุขอนามัยของดิสก์ภายในเครื่อง** (permissions, symlinks, config includes, พาธ "synced folder")
- **Plugin** (Plugin โหลดโดยไม่มี allowlist แบบชัดเจน)
- **ความคลาดเคลื่อน/การตั้งค่าผิดของนโยบาย** (ตั้งค่า sandbox docker ไว้แต่ปิดโหมด sandbox; รูปแบบ `gateway.nodes.denyCommands` ไม่มีผลเพราะการจับคู่เป็นชื่อคำสั่งแบบ exact เท่านั้น (เช่น `system.run`) และไม่ตรวจสอบข้อความ shell; รายการ `gateway.nodes.allowCommands` ที่อันตราย; `tools.profile="minimal"` ระดับ global ถูกแทนที่ด้วยโปรไฟล์ราย agent; เครื่องมือที่ Plugin เป็นเจ้าของเข้าถึงได้ภายใต้นโยบายเครื่องมือที่ผ่อนปรน)
- **ความคลาดเคลื่อนของความคาดหวังขณะ runtime** (เช่น สมมติว่า implicit exec ยังหมายถึง `sandbox` เมื่อ `tools.exec.host` ตอนนี้มีค่าเริ่มต้นเป็น `auto` หรือกำหนด `tools.exec.host="sandbox"` ไว้อย่างชัดเจนขณะที่ปิดโหมด sandbox)
- **สุขอนามัยของโมเดล** (เตือนเมื่อโมเดลที่ตั้งค่าดูเป็น legacy; ไม่ใช่ hard block)

หากคุณเรียกใช้ `--deep` OpenClaw จะพยายาม probe Gateway สดแบบ best-effort ด้วย

## แผนที่การจัดเก็บข้อมูลประจำตัว

ใช้ส่วนนี้เมื่อตรวจสอบการเข้าถึงหรือตัดสินใจว่าจะสำรองข้อมูลอะไร:

- **WhatsApp**: `~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
- **Telegram bot token**: config/env หรือ `channels.telegram.tokenFile` (ไฟล์ปกติเท่านั้น; ปฏิเสธ symlink)
- **Discord bot token**: config/env หรือ SecretRef (ผู้ให้บริการ env/file/exec)
- **Slack tokens**: config/env (`channels.slack.*`)
- **Pairing allowlists**:
  - `~/.openclaw/credentials/<channel>-allowFrom.json` (บัญชีเริ่มต้น)
  - `~/.openclaw/credentials/<channel>-<accountId>-allowFrom.json` (บัญชีที่ไม่ใช่ค่าเริ่มต้น)
- **โปรไฟล์ auth ของโมเดล**: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- **สถานะ runtime ของ Codex**: `~/.openclaw/agents/<agentId>/agent/codex-home/`
- **payload secrets ที่มีไฟล์รองรับ (ไม่บังคับ)**: `~/.openclaw/secrets.json`
- **การนำเข้า OAuth legacy**: `~/.openclaw/credentials/oauth.json`

## เช็กลิสต์ audit ด้านความปลอดภัย

เมื่อ audit แสดงข้อค้นพบ ให้ปฏิบัติตามลำดับความสำคัญนี้:

1. **ทุกอย่างที่ "เปิด" + เปิดใช้เครื่องมือ**: ล็อก DM/กลุ่มก่อน (pairing/allowlist) แล้วจึงจำกัดนโยบายเครื่องมือ/sandboxing ให้เข้มขึ้น
2. **การเปิดเผยเครือข่ายสาธารณะ** (LAN bind, Funnel, ไม่มี auth): แก้ไขทันที
3. **การเปิดเผยการควบคุมเบราว์เซอร์ระยะไกล**: ปฏิบัติเหมือนการเข้าถึงระดับ operator (tailnet-only, pair nodes อย่างตั้งใจ, หลีกเลี่ยงการเปิดเผยต่อสาธารณะ)
4. **Permissions**: ตรวจสอบให้แน่ใจว่า state/config/credentials/auth ไม่ได้อ่านได้โดย group/world
5. **Plugin**: โหลดเฉพาะสิ่งที่คุณไว้วางใจอย่างชัดเจน
6. **การเลือกโมเดล**: เลือกโมเดลรุ่นใหม่ที่เสริมความแข็งแรงด้าน instruction สำหรับบอตใดก็ตามที่มีเครื่องมือ

## อภิธานศัพท์ audit ด้านความปลอดภัย

ข้อค้นพบแต่ละรายการของ audit จะผูกกับ `checkId` แบบมีโครงสร้าง (เช่น
`gateway.bind_no_auth` หรือ `tools.exec.security_full_configured`) กลุ่ม
ความรุนแรง critical ที่พบบ่อย:

- `fs.*` - permissions ของ filesystem สำหรับ state, config, credentials, โปรไฟล์ auth
- `gateway.*` - โหมด bind, auth, Tailscale, Control UI, การตั้งค่า trusted-proxy
- `hooks.*`, `browser.*`, `sandbox.*`, `tools.exec.*` - การเพิ่มความแข็งแรงรายพื้นผิว
- `plugins.*`, `skills.*` - supply chain ของ Plugin/Skills และข้อค้นพบจากการสแกน
- `security.exposure.*` - การตรวจสอบข้ามส่วนที่นโยบายการเข้าถึงมาบรรจบกับรัศมีผลกระทบของเครื่องมือ

ดูแคตตาล็อกฉบับเต็มพร้อมระดับความรุนแรง, fix keys และการรองรับ auto-fix ได้ที่
[การตรวจสอบ audit ด้านความปลอดภัย](/th/gateway/security/audit-checks)

## Control UI ผ่าน HTTP

Control UI ต้องใช้ **secure context** (HTTPS หรือ localhost) เพื่อสร้าง device
identity `gateway.controlUi.allowInsecureAuth` เป็นสวิตช์ความเข้ากันได้ภายในเครื่อง:

- บน localhost จะอนุญาตให้ใช้ auth ของ Control UI โดยไม่มี device identity เมื่อโหลดหน้าเว็บผ่าน HTTP ที่ไม่ปลอดภัย
- ไม่ได้ข้ามการตรวจสอบ pairing
- ไม่ได้ผ่อนคลายข้อกำหนด device identity สำหรับ remote (ไม่ใช่ localhost)

ควรใช้ HTTPS (Tailscale Serve) หรือเปิด UI บน `127.0.0.1`

สำหรับสถานการณ์ break-glass เท่านั้น `gateway.controlUi.dangerouslyDisableDeviceAuth`
จะปิดการตรวจสอบ device identity ทั้งหมด นี่เป็นการลดระดับความปลอดภัยอย่างรุนแรง;
ให้ปิดไว้ เว้นแต่คุณกำลัง debug อยู่จริงและสามารถย้อนกลับได้อย่างรวดเร็ว

แยกจาก flag อันตรายเหล่านั้น การใช้ `gateway.auth.mode: "trusted-proxy"` สำเร็จ
สามารถรับ session ของ Control UI ระดับ **operator** โดยไม่มี device identity ได้ นี่เป็น
พฤติกรรมของโหมด auth ที่ตั้งใจไว้ ไม่ใช่ทางลัดของ `allowInsecureAuth` และยังคง
ไม่ขยายไปถึง session ของ Control UI บทบาท node

## การตั้งค่า reverse proxy

หากคุณเรียกใช้ Gateway หลัง reverse proxy (nginx, Caddy, Traefik ฯลฯ) ให้ตั้งค่า
`gateway.trustedProxies` เพื่อจัดการ IP ของ forwarded-client ให้ถูกต้อง

เมื่อ Gateway ตรวจพบ proxy headers จากที่อยู่ที่ **ไม่ได้** อยู่ใน `trustedProxies` จะ **ไม่** ถือว่าการเชื่อมต่อเป็น client ภายในเครื่อง หากปิด auth ของ gateway การเชื่อมต่อเหล่านั้นจะถูกปฏิเสธ สิ่งนี้ป้องกันการข้าม authentication ที่การเชื่อมต่อผ่าน proxy ไม่เช่นนั้นจะดูเหมือนมาจาก localhost และได้รับความไว้วางใจโดยอัตโนมัติ

`gateway.trustedProxies` ยังป้อนค่าให้ `gateway.auth.mode: "trusted-proxy"` ด้วย แต่โหมด auth นั้นเข้มงวดกว่า:

- auth แบบ trusted-proxy **จะ fail closed กับ proxy ที่มี source เป็น loopback ตามค่าเริ่มต้น**
- same-host loopback reverse proxy สามารถใช้ `gateway.trustedProxies` สำหรับการตรวจจับ local-client และการจัดการ forwarded IP
- same-host loopback reverse proxy สามารถผ่าน `gateway.auth.mode: "trusted-proxy"` ได้เฉพาะเมื่อ `gateway.auth.trustedProxy.allowLoopback = true`; มิฉะนั้นให้ใช้ token/password auth

```yaml
gateway:
  trustedProxies:
    - "10.0.0.1" # reverse proxy IP
  # Optional. Default false.
  # Only enable if your proxy cannot provide X-Forwarded-For.
  allowRealIpFallback: false
  auth:
    mode: password
    password: ${OPENCLAW_GATEWAY_PASSWORD}
```

เมื่อตั้งค่า `trustedProxies` แล้ว Gateway จะใช้ `X-Forwarded-For` เพื่อกำหนด IP ของ client `X-Real-IP` จะถูกละเว้นตามค่าเริ่มต้น เว้นแต่จะตั้งค่า `gateway.allowRealIpFallback: true` อย่างชัดเจน

Trusted proxy headers ไม่ได้ทำให้การ pair อุปกรณ์ node ได้รับความไว้วางใจโดยอัตโนมัติ
`gateway.nodes.pairing.autoApproveCidrs` เป็นนโยบาย operator แยกต่างหากที่ปิดไว้ตามค่าเริ่มต้น
แม้จะเปิดใช้แล้ว เส้นทาง trusted-proxy header ที่มี source เป็น loopback
จะถูกแยกออกจากการอนุมัติ node อัตโนมัติ เพราะผู้เรียกภายในเครื่องสามารถปลอมแปลง
headers เหล่านั้นได้ รวมถึงเมื่อเปิดใช้ auth แบบ loopback trusted-proxy อย่างชัดเจน

พฤติกรรม reverse proxy ที่ดี (เขียนทับ forwarding headers ขาเข้า):

```nginx
proxy_set_header X-Forwarded-For $remote_addr;
proxy_set_header X-Real-IP $remote_addr;
```

พฤติกรรม reverse proxy ที่ไม่ดี (ต่อท้าย/คง forwarding headers ที่ไม่น่าเชื่อถือไว้):

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```

## หมายเหตุเกี่ยวกับ HSTS และ origin

- OpenClaw gateway ให้ความสำคัญกับ local/local loopback ก่อน หากคุณ terminate TLS ที่ reverse proxy ให้ตั้งค่า HSTS บนโดเมน HTTPS ที่หันเข้าหา proxy ที่นั่น
- หาก gateway เองเป็นผู้ terminate HTTPS คุณสามารถตั้งค่า `gateway.http.securityHeaders.strictTransportSecurity` เพื่อส่ง header HSTS จาก response ของ OpenClaw
- คำแนะนำการ deploy แบบละเอียดอยู่ใน [Trusted Proxy Auth](/th/gateway/trusted-proxy-auth#tls-termination-and-hsts)
- สำหรับการ deploy Control UI ที่ไม่ใช่ loopback ต้องมี `gateway.controlUi.allowedOrigins` ตามค่าเริ่มต้น
- `gateway.controlUi.allowedOrigins: ["*"]` เป็นนโยบาย browser-origin แบบอนุญาตทั้งหมดอย่างชัดเจน ไม่ใช่ค่าเริ่มต้นที่เพิ่มความแข็งแรงแล้ว หลีกเลี่ยงนอกการทดสอบภายในเครื่องที่ควบคุมอย่างเข้มงวด
- ความล้มเหลวของ auth แบบ browser-origin บน loopback ยังคงถูก rate-limit แม้เมื่อเปิดใช้
  การยกเว้น loopback ทั่วไป แต่ lockout key จะ scope ต่อค่า
  `Origin` ที่ normalize แล้ว แทนที่จะใช้ bucket localhost ร่วมกันหนึ่งรายการ
- `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` เปิดใช้โหมด Host-header origin fallback; ให้ถือว่าเป็นนโยบายอันตรายที่ operator เลือกเอง
- ให้ถือว่า DNS rebinding และพฤติกรรม proxy-host header เป็นประเด็นการเพิ่มความแข็งแรงของ deployment; จำกัด `trustedProxies` ให้แน่นและหลีกเลี่ยงการเปิด gateway โดยตรงต่ออินเทอร์เน็ตสาธารณะ

## บันทึก session ภายในเครื่องอยู่บนดิสก์

OpenClaw เก็บทรานสคริปต์ของเซสชันไว้บนดิสก์ภายใต้ `~/.openclaw/agents/<agentId>/sessions/*.jsonl`
สิ่งนี้จำเป็นสำหรับความต่อเนื่องของเซสชัน และ (หากเลือกใช้) การจัดทำดัชนีหน่วยความจำของเซสชัน แต่ก็หมายความว่า
**กระบวนการ/ผู้ใช้ใด ๆ ที่มีสิทธิ์เข้าถึงระบบไฟล์สามารถอ่านบันทึกเหล่านั้นได้** ให้ถือว่าสิทธิ์เข้าถึงดิสก์เป็นขอบเขตความไว้วางใจ
และล็อกสิทธิ์บน `~/.openclaw` ให้รัดกุม (ดูส่วนการตรวจสอบด้านล่าง) หากคุณต้องการ
การแยกที่เข้มงวดยิ่งขึ้นระหว่างเอเจนต์ ให้รันเอเจนต์เหล่านั้นภายใต้ผู้ใช้ OS คนละคนหรือโฮสต์คนละเครื่อง

## การดำเนินการของ Node (system.run)

หากจับคู่ Node ของ macOS แล้ว Gateway สามารถเรียกใช้ `system.run` บน Node นั้นได้ นี่คือ **การดำเนินการโค้ดจากระยะไกล** บน Mac:

- ต้องมีการจับคู่ Node (การอนุมัติ + โทเค็น)
- การจับคู่ Node ของ Gateway ไม่ใช่พื้นผิวการอนุมัติรายคำสั่ง แต่เป็นการสร้างอัตลักษณ์/ความไว้วางใจของ Node และการออกโทเค็น
- Gateway ใช้นโยบายคำสั่ง Node แบบรวมทั่วระบบในระดับหยาบผ่าน `gateway.nodes.allowCommands` / `denyCommands`
- ควบคุมบน Mac ผ่าน **Settings → Exec approvals** (ความปลอดภัย + การถามยืนยัน + รายการอนุญาต)
- นโยบาย `system.run` ราย Node คือไฟล์การอนุมัติการดำเนินการของ Node เอง (`exec.approvals.node.*`) ซึ่งอาจเข้มงวดกว่าหรือผ่อนปรนกว่านโยบาย ID คำสั่งรวมทั่วระบบของ Gateway
- Node ที่รันด้วย `security="full"` และ `ask="off"` กำลังทำตามโมเดลผู้ปฏิบัติงานที่เชื่อถือได้ตามค่าเริ่มต้น ให้ถือว่านี่เป็นพฤติกรรมที่คาดไว้ เว้นแต่การปรับใช้ของคุณกำหนดท่าทีการอนุมัติหรือรายการอนุญาตที่เข้มงวดยิ่งขึ้นอย่างชัดเจน
- โหมดการอนุมัติจะผูกกับบริบทคำขอที่ตรงกันทุกประการ และเมื่อเป็นไปได้ จะผูกกับออปเปอแรนด์สคริปต์/ไฟล์โลคัลที่เป็นรูปธรรมหนึ่งรายการ หาก OpenClaw ไม่สามารถระบุไฟล์โลคัลโดยตรงสำหรับคำสั่งอินเทอร์พรีเตอร์/รันไทม์ได้อย่างแน่นอนหนึ่งไฟล์ การดำเนินการที่อิงการอนุมัติจะถูกปฏิเสธแทนที่จะให้คำมั่นว่าจะครอบคลุมความหมายทั้งหมด
- สำหรับ `host=node` การรันที่อิงการอนุมัติจะเก็บ `systemRunPlan` ที่เตรียมไว้แบบมาตรฐานด้วย
  การส่งต่อที่ได้รับอนุมัติในภายหลังจะนำแผนที่เก็บไว้นั้นกลับมาใช้ และการตรวจสอบของ Gateway
  จะปฏิเสธการแก้ไขคำสั่ง/cwd/บริบทเซสชันจากผู้เรียกหลังจากสร้าง
  คำขออนุมัติแล้ว
- หากคุณไม่ต้องการการดำเนินการจากระยะไกล ให้ตั้งค่าความปลอดภัยเป็น **deny** และนำการจับคู่ Node สำหรับ Mac เครื่องนั้นออก

ความแตกต่างนี้สำคัญต่อการคัดแยก:

- Node ที่จับคู่แล้วซึ่งเชื่อมต่อกลับมาใหม่และประกาศรายการคำสั่งที่ต่างออกไป โดยตัวมันเองไม่ใช่ช่องโหว่ หากนโยบายรวมทั่วระบบของ Gateway และการอนุมัติการดำเนินการภายในของ Node ยังบังคับใช้ขอบเขตการดำเนินการจริงอยู่
- รายงานที่ถือว่าเมตาดาต้าการจับคู่ Node เป็นชั้นการอนุมัติรายคำสั่งที่ซ่อนอยู่อีกชั้น มักเป็นความสับสนด้านนโยบาย/UX ไม่ใช่การเลี่ยงขอบเขตความปลอดภัย

## Skills แบบไดนามิก (ตัวเฝ้าดู / Node ระยะไกล)

OpenClaw สามารถรีเฟรชรายการ Skills ระหว่างเซสชันได้:

- **ตัวเฝ้าดู Skills**: การเปลี่ยนแปลงใน `SKILL.md` สามารถอัปเดตสแนปชอตของ Skills ในเทิร์นถัดไปของเอเจนต์ได้
- **Node ระยะไกล**: การเชื่อมต่อ Node ของ macOS สามารถทำให้ Skills เฉพาะ macOS มีสิทธิ์ใช้งานได้ (อิงจากการตรวจหา bin)

ให้ถือว่าโฟลเดอร์ Skills เป็น **โค้ดที่เชื่อถือได้** และจำกัดว่าใครสามารถแก้ไขได้

## โมเดลภัยคุกคาม

ผู้ช่วย AI ของคุณสามารถ:

- ดำเนินการคำสั่งเชลล์ใดก็ได้
- อ่าน/เขียนไฟล์
- เข้าถึงบริการเครือข่าย
- ส่งข้อความถึงใครก็ได้ (หากคุณให้สิทธิ์เข้าถึง WhatsApp)

ผู้ที่ส่งข้อความถึงคุณสามารถ:

- ใช้วิศวกรรมสังคมเพื่อเข้าถึงข้อมูลของคุณ
- ตรวจสอบรายละเอียดโครงสร้างพื้นฐาน

## แนวคิดหลัก: การควบคุมการเข้าถึงก่อนความฉลาด

ความล้มเหลวส่วนใหญ่ในที่นี้ไม่ใช่การเจาะระบบที่ซับซ้อน แต่เป็น “มีคนส่งข้อความถึงบอต แล้วบอตก็ทำตามที่เขาขอ”

ท่าทีของ OpenClaw:

- **อัตลักษณ์ก่อน:** ตัดสินใจว่าใครคุยกับบอตได้ (การจับคู่ DM / รายการอนุญาต / การ “เปิด” อย่างชัดเจน)
- **ขอบเขตถัดมา:** ตัดสินใจว่าบอตได้รับอนุญาตให้ทำงานที่ไหน (รายการอนุญาตกลุ่ม + การเปิดใช้งานด้วยการกล่าวถึง, เครื่องมือ, แซนด์บ็อกซ์, สิทธิ์อุปกรณ์)
- **โมเดลสุดท้าย:** สมมติว่าโมเดลอาจถูกชักจูงได้ ออกแบบให้การชักจูงมีรัศมีผลกระทบจำกัด

## โมเดลการอนุญาตคำสั่ง

คำสั่งสแลชและไดเรกทีฟจะได้รับการยอมรับเฉพาะสำหรับ **ผู้ส่งที่ได้รับอนุญาต** เท่านั้น การอนุญาตมาจาก
รายการอนุญาต/การจับคู่ของช่องทาง รวมกับ `commands.useAccessGroups` (ดู [การกำหนดค่า](/th/gateway/configuration)

คำสั่งจะเปิดใช้งานได้อย่างมีผลสำหรับช่องทางนั้น

`/exec` เป็นเพียงความสะดวกเฉพาะเซสชันสำหรับผู้ปฏิบัติงานที่ได้รับอนุญาต ไม่ได้เขียนการกำหนดค่าหรือ
เปลี่ยนเซสชันอื่น

## ความเสี่ยงของเครื่องมือ Control plane

เครื่องมือในตัวสองตัวสามารถเปลี่ยนแปลง Control plane แบบถาวรได้:

- `gateway` สามารถตรวจสอบการกำหนดค่าด้วย `config.schema.lookup` / `config.get` และสามารถเปลี่ยนแปลงแบบถาวรด้วย `config.apply`, `config.patch` และ `update.run`
- `cron` สามารถสร้างงานตามกำหนดเวลาที่รันต่อไปหลังจากแชต/งานต้นทางจบลงแล้ว

เครื่องมือรันไทม์ `gateway` แบบเจ้าของเท่านั้นยังคงปฏิเสธการเขียนทับ
`tools.exec.ask` หรือ `tools.exec.security`; นามแฝงเดิม `tools.bash.*` จะถูก
ทำให้เป็นมาตรฐานไปยังพาธ exec ที่ได้รับการป้องกันชุดเดียวกันก่อนการเขียน
การแก้ไข `gateway config.apply` และ `gateway config.patch` ที่ขับเคลื่อนโดยเอเจนต์จะ
ปิดแบบปลอดภัยตามค่าเริ่มต้น: มีเพียงชุดพาธ prompt, model และ mention-gating
ที่จำกัดเท่านั้นที่เอเจนต์ปรับแต่งได้ ดังนั้นต้นไม้การกำหนดค่าที่ละเอียดอ่อนใหม่จะได้รับการป้องกัน
เว้นแต่จะถูกเพิ่มเข้าในรายการอนุญาตโดยเจตนา

สำหรับเอเจนต์/พื้นผิวใด ๆ ที่จัดการเนื้อหาที่ไม่น่าเชื่อถือ ให้ปฏิเสธสิ่งเหล่านี้ตามค่าเริ่มต้น:

```json5
{
  tools: {
    deny: ["gateway", "cron", "sessions_spawn", "sessions_send"],
  },
}
```

`commands.restart=false` บล็อกเฉพาะการดำเนินการรีสตาร์ตเท่านั้น ไม่ได้ปิดใช้งานการดำเนินการ config/update ของ `gateway`

## Plugin

Plugin รัน **ในกระบวนการเดียวกัน** กับ Gateway ให้ถือว่าเป็นโค้ดที่เชื่อถือได้:

- ติดตั้ง Plugin จากแหล่งที่คุณเชื่อถือเท่านั้น
- ควรใช้รายการอนุญาต `plugins.allow` ที่ชัดเจน
- ตรวจสอบการกำหนดค่า Plugin ก่อนเปิดใช้งาน
- รีสตาร์ต Gateway หลังจากเปลี่ยนแปลง Plugin
- หากคุณติดตั้งหรืออัปเดต Plugin (`openclaw plugins install <package>`, `openclaw plugins update <id>`) ให้ถือว่าเหมือนการรันโค้ดที่ไม่น่าเชื่อถือ:
  - พาธติดตั้งคือไดเรกทอรีราย Plugin ภายใต้รากการติดตั้ง Plugin ที่ใช้งานอยู่
  - OpenClaw รันการสแกนโค้ดอันตรายในตัวก่อนติดตั้ง/อัปเดต ผลการตรวจพบระดับ `critical` จะบล็อกตามค่าเริ่มต้น
  - การติดตั้ง Plugin ผ่าน npm และ git จะรันการทำให้ดีเพนเดนซีของตัวจัดการแพ็กเกจบรรจบกันเฉพาะระหว่างโฟลว์ติดตั้ง/อัปเดตที่ชัดเจนเท่านั้น พาธโลคัลและไฟล์เก็บถาวรถูกถือเป็นแพ็กเกจ Plugin ที่มีทุกอย่างในตัว OpenClaw จะคัดลอก/อ้างอิงโดยไม่รัน `npm install`
  - ควรปักหมุดเวอร์ชันที่แน่นอน (`@scope/pkg@1.2.3`) และตรวจสอบโค้ดที่แตกไฟล์แล้วบนดิสก์ก่อนเปิดใช้งาน
  - `--dangerously-force-unsafe-install` ใช้เป็นทางออกฉุกเฉินเฉพาะสำหรับผลบวกลวงจากการสแกนในตัวบนโฟลว์ติดตั้ง/อัปเดต Plugin เท่านั้น ไม่ได้เลี่ยงบล็อกนโยบายฮุก `before_install` ของ Plugin และไม่ได้เลี่ยงความล้มเหลวจากการสแกน
  - การติดตั้งดีเพนเดนซีของ Skills ที่อิง Gateway ใช้การแยก dangerous/suspicious แบบเดียวกัน: ผลการตรวจพบระดับ `critical` ในตัวจะบล็อก เว้นแต่ผู้เรียกจะตั้งค่า `dangerouslyForceUnsafeInstall` อย่างชัดเจน ขณะที่ผลการตรวจพบที่น่าสงสัยยังคงเตือนเท่านั้น `openclaw skills install` ยังคงเป็นโฟลว์ดาวน์โหลด/ติดตั้ง Skills จาก ClawHub ที่แยกต่างหาก

รายละเอียด: [Plugin](/th/tools/plugin)

## โมเดลการเข้าถึง DM: การจับคู่, รายการอนุญาต, เปิด, ปิดใช้งาน

ช่องทางปัจจุบันทั้งหมดที่รองรับ DM รองรับนโยบาย DM (`dmPolicy` หรือ `*.dm.policy`) ที่กั้น DM ขาเข้า **ก่อน** ประมวลผลข้อความ:

- `pairing` (ค่าเริ่มต้น): ผู้ส่งที่ไม่รู้จักจะได้รับรหัสจับคู่สั้น ๆ และบอตจะเพิกเฉยต่อข้อความจนกว่าจะได้รับอนุมัติ รหัสหมดอายุหลัง 1 ชั่วโมง; การส่ง DM ซ้ำจะไม่ส่งรหัสซ้ำจนกว่าจะสร้างคำขอใหม่ คำขอที่รอดำเนินการถูกจำกัดไว้ที่ **3 ต่อช่องทาง** ตามค่าเริ่มต้น
- `allowlist`: ผู้ส่งที่ไม่รู้จักจะถูกบล็อก (ไม่มีการจับคู่แบบ handshake)
- `open`: อนุญาตให้ใครก็ได้ส่ง DM (สาธารณะ) **ต้อง** ให้รายการอนุญาตของช่องทางมี `"*"` (เลือกใช้อย่างชัดเจน)
- `disabled`: เพิกเฉยต่อ DM ขาเข้าทั้งหมด

อนุมัติผ่าน CLI:

```bash
openclaw pairing list <channel>
openclaw pairing approve <channel> <code>
```

รายละเอียด + ไฟล์บนดิสก์: [การจับคู่](/th/channels/pairing)

## การแยกเซสชัน DM (โหมดผู้ใช้หลายคน)

ตามค่าเริ่มต้น OpenClaw จะกำหนดเส้นทาง **DM ทั้งหมดเข้าสู่เซสชันหลัก** เพื่อให้ผู้ช่วยของคุณมีความต่อเนื่องข้ามอุปกรณ์และช่องทาง หาก **หลายคน** สามารถส่ง DM ถึงบอตได้ (DM แบบเปิดหรือรายการอนุญาตหลายคน) ให้พิจารณาแยกเซสชัน DM:

```json5
{
  session: { dmScope: "per-channel-peer" },
}
```

สิ่งนี้ป้องกันการรั่วไหลของบริบทข้ามผู้ใช้ ขณะยังคงแยกแชตกลุ่มไว้

นี่เป็นขอบเขตบริบทการส่งข้อความ ไม่ใช่ขอบเขตผู้ดูแลโฮสต์ หากผู้ใช้มีความเป็นปฏิปักษ์กันและใช้โฮสต์/การกำหนดค่า Gateway เดียวกัน ให้รัน Gateway แยกกันตามขอบเขตความไว้วางใจแทน

### โหมด DM ที่ปลอดภัย (แนะนำ)

ให้ถือว่าสนิปเป็ตด้านบนเป็น **โหมด DM ที่ปลอดภัย**:

- ค่าเริ่มต้น: `session.dmScope: "main"` (DM ทั้งหมดใช้เซสชันเดียวร่วมกันเพื่อความต่อเนื่อง)
- ค่าเริ่มต้นของการเริ่มต้นใช้งาน CLI โลคัล: เขียน `session.dmScope: "per-channel-peer"` เมื่อยังไม่ได้ตั้งค่า (คงค่าที่ตั้งไว้อย่างชัดเจนเดิมไว้)
- โหมด DM ที่ปลอดภัย: `session.dmScope: "per-channel-peer"` (แต่ละคู่ช่องทาง+ผู้ส่งได้บริบท DM ที่แยกกัน)
- การแยก peer ข้ามช่องทาง: `session.dmScope: "per-peer"` (ผู้ส่งแต่ละคนได้หนึ่งเซสชันข้ามทุกช่องทางชนิดเดียวกัน)

หากคุณรันหลายบัญชีบนช่องทางเดียวกัน ให้ใช้ `per-account-channel-peer` แทน หากคนเดียวกันติดต่อคุณผ่านหลายช่องทาง ให้ใช้ `session.identityLinks` เพื่อรวมเซสชัน DM เหล่านั้นให้เป็นอัตลักษณ์มาตรฐานเดียว ดู [การจัดการเซสชัน](/th/concepts/session) และ [การกำหนดค่า](/th/gateway/configuration)

## รายการอนุญาตสำหรับ DM และกลุ่ม

OpenClaw มีชั้น “ใครสามารถกระตุ้นฉันได้?” แยกกันสองชั้น:

- **รายการอนุญาต DM** (`allowFrom` / `channels.discord.allowFrom` / `channels.slack.allowFrom`; แบบเดิม: `channels.discord.dm.allowFrom`, `channels.slack.dm.allowFrom`): ใครได้รับอนุญาตให้คุยกับบอตในข้อความโดยตรง
  - เมื่อ `dmPolicy="pairing"` การอนุมัติจะถูกเขียนไปยังที่เก็บรายการอนุญาตการจับคู่ตามขอบเขตบัญชีภายใต้ `~/.openclaw/credentials/` (`<channel>-allowFrom.json` สำหรับบัญชีเริ่มต้น, `<channel>-<accountId>-allowFrom.json` สำหรับบัญชีที่ไม่ใช่ค่าเริ่มต้น) และรวมกับรายการอนุญาตในการกำหนดค่า
- **รายการอนุญาตกลุ่ม** (เฉพาะช่องทาง): กลุ่ม/ช่องทาง/กิลด์ใดที่บอตจะยอมรับข้อความ
  - รูปแบบที่พบบ่อย:
    - `channels.whatsapp.groups`, `channels.telegram.groups`, `channels.imessage.groups`: ค่าเริ่มต้นรายกลุ่ม เช่น `requireMention`; เมื่อตั้งค่าแล้ว จะทำหน้าที่เป็นรายการอนุญาตกลุ่มด้วย (ใส่ `"*"` เพื่อคงพฤติกรรมอนุญาตทั้งหมด)
    - `groupPolicy="allowlist"` + `groupAllowFrom`: จำกัดว่าใครสามารถกระตุ้นบอต _ภายใน_ เซสชันกลุ่ม (WhatsApp/Telegram/Signal/iMessage/Microsoft Teams)
    - `channels.discord.guilds` / `channels.slack.channels`: รายการอนุญาตรายพื้นผิว + ค่าเริ่มต้นการกล่าวถึง
  - การตรวจสอบกลุ่มรันตามลำดับนี้: `groupPolicy`/รายการอนุญาตกลุ่มก่อน, การเปิดใช้งานด้วยการกล่าวถึง/ตอบกลับตามมา
  - การตอบกลับข้อความของบอต (การกล่าวถึงโดยนัย) จะ **ไม่** เลี่ยงรายการอนุญาตผู้ส่งอย่าง `groupAllowFrom`
  - **หมายเหตุด้านความปลอดภัย:** ให้ถือว่า `dmPolicy="open"` และ `groupPolicy="open"` เป็นการตั้งค่าสำหรับทางเลือกสุดท้าย ควรใช้น้อยมาก; ควรใช้การจับคู่ + รายการอนุญาต เว้นแต่คุณจะเชื่อถือสมาชิกทุกคนในห้องอย่างเต็มที่

รายละเอียด: [การกำหนดค่า](/th/gateway/configuration) และ [กลุ่ม](/th/channels/groups)

## Prompt injection (คืออะไร และทำไมจึงสำคัญ)

Prompt injection คือการที่ผู้โจมตีสร้างข้อความที่ชักจูงโมเดลให้ทำสิ่งที่ไม่ปลอดภัย (“ละเว้นคำสั่งของคุณ”, “ดัมป์ระบบไฟล์ของคุณ”, “ตามลิงก์นี้แล้วรันคำสั่ง” ฯลฯ)

แม้มี system prompt ที่เข้มงวด **prompt injection ก็ยังไม่ได้ถูกแก้ให้หมดไป** รั้วป้องกันของ system prompt เป็นเพียงคำแนะนำแบบอ่อนเท่านั้น; การบังคับใช้อย่างแข็งแรงมาจากนโยบายเครื่องมือ, การอนุมัติ exec, แซนด์บ็อกซ์ และรายการอนุญาตช่องทาง (และผู้ปฏิบัติงานสามารถปิดสิ่งเหล่านี้ได้โดยการออกแบบ) สิ่งที่ช่วยได้ในทางปฏิบัติ:

- ล็อกดาวน์ DM ขาเข้าไว้ (การจับคู่/allowlist)
- ในกลุ่มควรใช้ mention gating; หลีกเลี่ยงบอทแบบ "always-on" ในห้องสาธารณะ
- ถือว่าลิงก์ ไฟล์แนบ และคำสั่งที่วางเข้ามาเป็นอันตรายโดยค่าเริ่มต้น
- รันการเรียกใช้เครื่องมือที่อ่อนไหวใน sandbox; เก็บความลับไว้นอกระบบไฟล์ที่ agent เข้าถึงได้
- หมายเหตุ: sandboxing เป็นแบบเลือกเปิดใช้ หากโหมด sandbox ปิดอยู่ `host=auto` โดยนัยจะ resolve ไปยังโฮสต์ Gateway ส่วน `host=sandbox` ที่ระบุชัดเจนยังคงล้มเหลวแบบปิด เพราะไม่มี runtime ของ sandbox ให้ใช้งาน ตั้งค่า `host=gateway` หากคุณต้องการให้พฤติกรรมนี้ระบุไว้อย่างชัดเจนใน config
- จำกัดเครื่องมือความเสี่ยงสูง (`exec`, `browser`, `web_fetch`, `web_search`) ให้ใช้ได้เฉพาะกับ agent ที่เชื่อถือได้หรือ allowlist ที่ระบุชัดเจน
- หากคุณ allowlist interpreter (`python`, `node`, `ruby`, `perl`, `php`, `lua`, `osascript`) ให้เปิดใช้ `tools.exec.strictInlineEval` เพื่อให้รูปแบบ inline eval ยังคงต้องได้รับอนุมัติอย่างชัดเจน
- การวิเคราะห์การอนุมัติ shell ยังปฏิเสธรูปแบบ POSIX parameter-expansion (`$VAR`, `$?`, `$$`, `$1`, `$@`, `${…}`) ภายใน **heredoc ที่ไม่ได้ quote** ดังนั้น body ของ heredoc ที่อยู่ใน allowlist จะไม่สามารถลอบให้ shell expansion ผ่านการตรวจ allowlist ในฐานะข้อความธรรมดาได้ ให้ quote ตัว terminator ของ heredoc (เช่น `<<'EOF'`) เพื่อเลือกใช้ semantics ของ body แบบ literal; heredoc ที่ไม่ได้ quote ซึ่งจะขยายตัวแปรจะถูกปฏิเสธ
- **การเลือกรุ่นโมเดลมีความสำคัญ:** โมเดลรุ่นเก่า/เล็กกว่า/legacy ทนทานต่อ prompt injection และการใช้เครื่องมือผิดวัตถุประสงค์ได้น้อยกว่ามาก สำหรับ agent ที่เปิดใช้เครื่องมือ ให้ใช้โมเดลเจเนอเรชันล่าสุดที่แข็งแกร่งที่สุดและเสริมความทนทานต่อคำสั่งแล้วเท่าที่มีให้ใช้

สัญญาณอันตรายที่ควรถือว่าไม่น่าเชื่อถือ:

- "อ่านไฟล์/URL นี้ แล้วทำตามที่บอกทุกอย่าง"
- "ละเว้น system prompt หรือกฎความปลอดภัยของคุณ"
- "เปิดเผยคำสั่งที่ซ่อนอยู่หรือ output ของเครื่องมือ"
- "วางเนื้อหาทั้งหมดของ ~/.openclaw หรือ log ของคุณ"

## การ sanitize special-token ของเนื้อหาภายนอก

OpenClaw จะตัด literal ของ special-token ใน chat-template ของ LLM ที่ self-hosted ทั่วไปออกจากเนื้อหาภายนอกและ metadata ที่ถูกห่อไว้ ก่อนที่ข้อมูลเหล่านั้นจะไปถึงโมเดล กลุ่ม marker ที่ครอบคลุมรวมถึง Qwen/ChatML, Llama, Gemma, Mistral, Phi และ token บทบาท/เทิร์นของ GPT-OSS

เหตุผล:

- backend ที่เข้ากันได้กับ OpenAI ซึ่งอยู่หน้าโมเดล self-hosted บางครั้งจะคง special token ที่ปรากฏในข้อความผู้ใช้ไว้ แทนที่จะ mask token เหล่านั้น ผู้โจมตีที่สามารถเขียนเข้าไปในเนื้อหาภายนอกขาเข้า (หน้าเว็บที่ fetch มา, body ของอีเมล, output จากเครื่องมืออ่านเนื้อหาไฟล์) อาจ inject ขอบเขตบทบาท `assistant` หรือ `system` สังเคราะห์ และหลุดออกจาก guardrail ของ wrapped-content ได้
- การ sanitize เกิดขึ้นที่ชั้น wrapping ของ external-content จึงมีผลอย่างสม่ำเสมอทั้งกับเครื่องมือ fetch/read และเนื้อหาช่องทางขาเข้า แทนที่จะทำแยกตาม provider
- การตอบกลับขาออกของโมเดลมี sanitizer แยกต่างหากอยู่แล้ว ซึ่งตัด `<tool_call>`, `<function_calls>`, `<system-reminder>`, `<previous_response>` และโครงนั่งร้าน runtime ภายในที่คล้ายกันซึ่งรั่วไหลออก จากคำตอบที่ผู้ใช้มองเห็นได้ ณ ขอบเขตการส่งมอบช่องทางสุดท้าย sanitizer ของ external-content คือส่วนคู่กันสำหรับขาเข้า

สิ่งนี้ไม่ได้แทนที่การ hardening อื่นในหน้านี้ - `dmPolicy`, allowlist, การอนุมัติ exec, sandboxing และ `contextVisibility` ยังคงเป็นกลไกหลัก มันปิด bypass เฉพาะจุดหนึ่งในชั้น tokenizer ต่อสแต็ก self-hosted ที่ส่งต่อข้อความผู้ใช้พร้อม special token โดยไม่เปลี่ยนแปลง

## flag bypass เนื้อหาภายนอกที่ไม่ปลอดภัย

OpenClaw มี flag bypass ที่ระบุชัดเจน ซึ่งปิดการ wrapping เพื่อความปลอดภัยของ external-content:

- `hooks.mappings[].allowUnsafeExternalContent`
- `hooks.gmail.allowUnsafeExternalContent`
- ฟิลด์ payload ของ Cron `allowUnsafeExternalContent`

แนวทาง:

- ปล่อยค่าเหล่านี้ให้ unset/false ใน production
- เปิดใช้ชั่วคราวเท่านั้นสำหรับการ debug ที่จำกัดขอบเขตอย่างเข้มงวด
- หากเปิดใช้ ให้ isolate agent นั้น (sandbox + เครื่องมือน้อยที่สุด + namespace ของ session เฉพาะ)

หมายเหตุความเสี่ยงของ hook:

- payload ของ hook เป็นเนื้อหาที่ไม่น่าเชื่อถือ แม้การส่งมอบจะมาจากระบบที่คุณควบคุม (mail/docs/web content สามารถพา prompt injection มาได้)
- tier ของโมเดลที่อ่อนเพิ่มความเสี่ยงนี้ สำหรับ automation ที่ขับเคลื่อนด้วย hook ให้เลือก tier โมเดลสมัยใหม่ที่แข็งแกร่ง และควบคุมนโยบายเครื่องมือให้รัดกุม (`tools.profile: "messaging"` หรือเข้มงวดกว่า) พร้อม sandboxing หากทำได้

### Prompt injection ไม่จำเป็นต้องมาจาก DM สาธารณะ

แม้ว่า **มีเพียงคุณ** ที่ส่งข้อความถึงบอทได้ prompt injection ก็ยังเกิดขึ้นได้ผ่าน
**เนื้อหาที่ไม่น่าเชื่อถือ** ใดๆ ที่บอทอ่าน (ผลลัพธ์ web search/fetch, หน้า browser,
อีเมล, docs, ไฟล์แนบ, log/code ที่วางเข้ามา) กล่าวอีกอย่างคือ ผู้ส่งไม่ใช่
พื้นผิวภัยคุกคามเพียงอย่างเดียว; **ตัวเนื้อหาเอง** สามารถพาคำสั่งที่เป็นปฏิปักษ์มาได้

เมื่อเปิดใช้เครื่องมือ ความเสี่ยงทั่วไปคือการ exfiltrate context หรือกระตุ้น
tool call ลด blast radius โดย:

- ใช้ **reader agent** แบบ read-only หรือปิดเครื่องมือ เพื่อสรุปเนื้อหาที่ไม่น่าเชื่อถือ
  จากนั้นส่งสรุปให้ agent หลักของคุณ
- ปิด `web_search` / `web_fetch` / `browser` สำหรับ agent ที่เปิดใช้เครื่องมือ เว้นแต่จำเป็น
- สำหรับ input URL ของ OpenResponses (`input_file` / `input_image`) ให้ตั้งค่า
  `gateway.http.endpoints.responses.files.urlAllowlist` และ
  `gateway.http.endpoints.responses.images.urlAllowlist` ให้เข้มงวด และคง `maxUrlParts` ไว้ต่ำ
  allowlist ว่างจะถือว่าไม่ได้ตั้งค่า; ใช้ `files.allowUrl: false` / `images.allowUrl: false`
  หากคุณต้องการปิดการ fetch URL ทั้งหมด
- สำหรับ input ไฟล์ของ OpenResponses ข้อความ `input_file` ที่ decode แล้วจะยังถูก inject เป็น
  **เนื้อหาภายนอกที่ไม่น่าเชื่อถือ** อย่าพึ่งพาว่าข้อความในไฟล์เชื่อถือได้เพียงเพราะ
  Gateway decode มันในเครื่อง block ที่ inject แล้วยังคงมี boundary marker
  `<<<EXTERNAL_UNTRUSTED_CONTENT ...>>>` อย่างชัดเจน พร้อม metadata `Source: External`
  แม้ว่าเส้นทางนี้จะละ banner `SECURITY NOTICE:` ที่ยาวกว่า
- การ wrapping แบบใช้ marker เดียวกันจะถูกนำไปใช้เมื่อ media-understanding ดึงข้อความ
  จากเอกสารที่แนบมาก่อน append ข้อความนั้นเข้าไปใน media prompt
- เปิดใช้ sandboxing และ allowlist เครื่องมือที่เข้มงวดสำหรับ agent ใดๆ ที่แตะ input ที่ไม่น่าเชื่อถือ
- เก็บความลับไว้นอก prompt; ส่งผ่าน env/config บนโฮสต์ Gateway แทน

### backend LLM แบบ self-hosted

backend แบบ self-hosted ที่เข้ากันได้กับ OpenAI เช่น vLLM, SGLang, TGI, LM Studio,
หรือสแต็ก tokenizer ของ Hugging Face แบบ custom อาจแตกต่างจาก provider แบบ hosted ในวิธีจัดการ
special token ของ chat-template หาก backend tokenize string literal
เช่น `<|im_start|>`, `<|start_header_id|>`, หรือ `<start_of_turn>` เป็น
token โครงสร้างของ chat-template ภายในเนื้อหาผู้ใช้ ข้อความที่ไม่น่าเชื่อถืออาจพยายาม
ปลอมขอบเขตบทบาทที่ชั้น tokenizer ได้

OpenClaw จะตัด literal ของ special-token จากตระกูลโมเดลที่พบบ่อยออกจาก
เนื้อหาภายนอกที่ wrapped ก่อน dispatch ไปยังโมเดล เปิดใช้
external-content wrapping ไว้ และเลือกการตั้งค่า backend ที่แยกหรือ escape special
token ในเนื้อหาที่ผู้ใช้ให้มาเมื่อมีให้ใช้ provider แบบ hosted เช่น OpenAI
และ Anthropic มีการ sanitize ฝั่ง request ของตนเองอยู่แล้ว

### ความแข็งแกร่งของโมเดล (หมายเหตุด้านความปลอดภัย)

ความต้านทานต่อ prompt injection **ไม่** เท่ากันในทุก tier ของโมเดล โมเดลขนาดเล็กกว่า/ราคาถูกกว่ามักไวต่อการใช้เครื่องมือผิดวัตถุประสงค์และการ hijack คำสั่งมากกว่า โดยเฉพาะภายใต้ prompt ที่เป็นปฏิปักษ์

<Warning>
สำหรับ agent ที่เปิดใช้เครื่องมือหรือ agent ที่อ่านเนื้อหาที่ไม่น่าเชื่อถือ ความเสี่ยง prompt-injection กับโมเดลรุ่นเก่า/เล็กกว่ามักสูงเกินไป อย่ารัน workload เหล่านั้นบน tier โมเดลที่อ่อน
</Warning>

คำแนะนำ:

- **ใช้โมเดลเจเนอเรชันล่าสุด tier ดีที่สุด** สำหรับบอทใดๆ ที่สามารถรันเครื่องมือหรือแตะไฟล์/เครือข่ายได้
- **อย่าใช้ tier ที่เก่ากว่า/อ่อนกว่า/เล็กกว่า** สำหรับ agent ที่เปิดใช้เครื่องมือหรือ inbox ที่ไม่น่าเชื่อถือ; ความเสี่ยง prompt-injection สูงเกินไป
- หากคุณจำเป็นต้องใช้โมเดลขนาดเล็กกว่า ให้ **ลด blast radius** (เครื่องมือ read-only, sandboxing ที่แข็งแกร่ง, การเข้าถึงระบบไฟล์น้อยที่สุด, allowlist ที่เข้มงวด)
- เมื่อรันโมเดลขนาดเล็ก ให้ **เปิดใช้ sandboxing สำหรับทุก session** และ **ปิด web_search/web_fetch/browser** เว้นแต่ input จะถูกควบคุมอย่างเข้มงวด
- สำหรับผู้ช่วยส่วนตัวแบบ chat-only ที่มี input เชื่อถือได้และไม่มีเครื่องมือ โมเดลขนาดเล็กกว่ามักใช้ได้

## Reasoning และ output แบบ verbose ในกลุ่ม

`/reasoning`, `/verbose` และ `/trace` อาจเปิดเผย reasoning ภายใน, output ของเครื่องมือ
หรือข้อมูลวินิจฉัยของ Plugin ที่
ไม่ได้ตั้งใจให้ปรากฏในช่องทางสาธารณะ ในบริบทของกลุ่ม ให้ถือว่าสิ่งเหล่านี้เป็น **debug
เท่านั้น** และปิดไว้ เว้นแต่คุณต้องใช้อย่างชัดเจน

แนวทาง:

- ปิด `/reasoning`, `/verbose` และ `/trace` ไว้ในห้องสาธารณะ
- หากคุณเปิดใช้ ให้ทำเฉพาะใน DM ที่เชื่อถือได้หรือห้องที่ควบคุมอย่างเข้มงวด
- จำไว้ว่า output แบบ verbose และ trace อาจรวม args ของเครื่องมือ, URL, ข้อมูลวินิจฉัยของ Plugin และข้อมูลที่โมเดลเห็น

## ตัวอย่างการ hardening configuration

### สิทธิ์ของไฟล์

เก็บ config + state ให้เป็นส่วนตัวบนโฮสต์ Gateway:

- `~/.openclaw/openclaw.json`: `600` (ผู้ใช้อ่าน/เขียนเท่านั้น)
- `~/.openclaw`: `700` (ผู้ใช้เท่านั้น)

`openclaw doctor` สามารถเตือนและเสนอให้ปรับสิทธิ์เหล่านี้ให้เข้มงวดขึ้นได้

### การเปิดเผยเครือข่าย (bind, port, firewall)

Gateway multiplex **WebSocket + HTTP** บน port เดียว:

- ค่าเริ่มต้น: `18789`
- Config/flags/env: `gateway.port`, `--port`, `OPENCLAW_GATEWAY_PORT`

พื้นผิว HTTP นี้รวม Control UI และ canvas host:

- Control UI (SPA assets) (base path ค่าเริ่มต้น `/`)
- Canvas host: `/__openclaw__/canvas/` และ `/__openclaw__/a2ui/` (HTML/JS ตามอำเภอใจ; ถือว่าเป็นเนื้อหาที่ไม่น่าเชื่อถือ)

หากคุณโหลด canvas content ใน browser ปกติ ให้ปฏิบัติกับมันเหมือนหน้าเว็บที่ไม่น่าเชื่อถืออื่นๆ:

- อย่าเปิดเผย canvas host ต่อเครือข่าย/ผู้ใช้ที่ไม่น่าเชื่อถือ
- อย่าให้ canvas content ใช้ origin เดียวกับพื้นผิวเว็บที่มีสิทธิ์สูง เว้นแต่คุณเข้าใจผลกระทบอย่างถ่องแท้

โหมด bind ควบคุมว่า Gateway จะ listen ที่ใด:

- `gateway.bind: "loopback"` (ค่าเริ่มต้น): เฉพาะ client ในเครื่องเท่านั้นที่เชื่อมต่อได้
- bind ที่ไม่ใช่ loopback (`"lan"`, `"tailnet"`, `"custom"`) ขยายพื้นผิวการโจมตี ใช้เฉพาะร่วมกับ auth ของ Gateway (token/password ที่ใช้ร่วมกันหรือ trusted proxy ที่กำหนดค่าอย่างถูกต้อง) และ firewall จริง

หลักทั่วไป:

- เลือกใช้ Tailscale Serve แทน bind บน LAN (Serve ทำให้ Gateway อยู่บน loopback และ Tailscale จัดการการเข้าถึง)
- หากจำเป็นต้อง bind กับ LAN ให้ firewall port ไปยัง allowlist ของ source IP ที่จำกัดอย่างเข้มงวด; อย่า port-forward แบบกว้าง
- อย่าเปิดเผย Gateway ที่ไม่มี authentication บน `0.0.0.0` เด็ดขาด

### การ publish port ของ Docker กับ UFW

หากคุณรัน OpenClaw ด้วย Docker บน VPS โปรดจำว่า port ของ container ที่ publish
(`-p HOST:CONTAINER` หรือ Compose `ports:`) จะถูก route ผ่าน forwarding
chain ของ Docker ไม่ใช่เฉพาะกฎ `INPUT` ของ host เท่านั้น

เพื่อให้ traffic ของ Docker สอดคล้องกับนโยบาย firewall ของคุณ ให้ enforce กฎใน
`DOCKER-USER` (chain นี้ถูกประเมินก่อนกฎ accept ของ Docker เอง)
บน distro สมัยใหม่จำนวนมาก `iptables`/`ip6tables` ใช้ frontend `iptables-nft`
และยังนำกฎเหล่านี้ไปใช้กับ backend nftables

ตัวอย่าง allowlist ขั้นต่ำ (IPv4):

```bash
# /etc/ufw/after.rules (append as its own *filter section)
*filter
:DOCKER-USER - [0:0]
-A DOCKER-USER -m conntrack --ctstate ESTABLISHED,RELATED -j RETURN
-A DOCKER-USER -s 127.0.0.0/8 -j RETURN
-A DOCKER-USER -s 10.0.0.0/8 -j RETURN
-A DOCKER-USER -s 172.16.0.0/12 -j RETURN
-A DOCKER-USER -s 192.168.0.0/16 -j RETURN
-A DOCKER-USER -s 100.64.0.0/10 -j RETURN
-A DOCKER-USER -p tcp --dport 80 -j RETURN
-A DOCKER-USER -p tcp --dport 443 -j RETURN
-A DOCKER-USER -m conntrack --ctstate NEW -j DROP
-A DOCKER-USER -j RETURN
COMMIT
```

IPv6 มี table แยกต่างหาก เพิ่ม policy ที่ตรงกันใน `/etc/ufw/after6.rules` หาก
เปิดใช้ Docker IPv6

หลีกเลี่ยงการ hardcode ชื่อ interface เช่น `eth0` ใน snippet ของ docs ชื่อ interface
แตกต่างกันไปตาม image ของ VPS (`ens3`, `enp*` ฯลฯ) และการไม่ตรงกันอาจทำให้
ข้ามกฎ deny ของคุณโดยไม่ตั้งใจ

การตรวจสอบอย่างรวดเร็วหลัง reload:

```bash
ufw reload
iptables -S DOCKER-USER
ip6tables -S DOCKER-USER
nmap -sT -p 1-65535 <public-ip> --open
```

port ภายนอกที่คาดไว้ควรมีเฉพาะสิ่งที่คุณตั้งใจเปิดเผย (สำหรับ setup ส่วนใหญ่:
SSH + port ของ reverse proxy ของคุณ)

### การค้นพบ mDNS/Bonjour

เมื่อเปิดใช้ Plugin `bonjour` ที่ bundled มา Gateway จะ broadcast การมีอยู่ของมันผ่าน mDNS (`_openclaw-gw._tcp` บน port 5353) สำหรับการค้นหาอุปกรณ์ในเครื่อง ในโหมดเต็ม สิ่งนี้รวมถึงระเบียน TXT ที่อาจเปิดเผยรายละเอียดการปฏิบัติการ:

- `cliPath`: เส้นทางระบบไฟล์แบบเต็มไปยังไบนารี CLI (เปิดเผยชื่อผู้ใช้และตำแหน่งติดตั้ง)
- `sshPort`: ประกาศความพร้อมใช้งาน SSH บนโฮสต์
- `displayName`, `lanHost`: ข้อมูลชื่อโฮสต์

**ข้อควรพิจารณาด้านความปลอดภัยในการปฏิบัติงาน:** การบรอดแคสต์รายละเอียดโครงสร้างพื้นฐานทำให้ผู้ที่อยู่ในเครือข่ายภายในสืบค้นข้อมูลได้ง่ายขึ้น แม้แต่ข้อมูลที่ดู "ไม่เป็นอันตราย" เช่น เส้นทางระบบไฟล์และความพร้อมใช้งาน SSH ก็ช่วยให้ผู้โจมตีทำแผนที่สภาพแวดล้อมของคุณได้

**คำแนะนำ:**

1. **ปิด Bonjour ไว้ เว้นแต่จำเป็นต้องใช้การค้นพบผ่าน LAN** Bonjour เริ่มทำงานอัตโนมัติบนโฮสต์ macOS และเป็นแบบเลือกเปิดบนที่อื่น URL ของ Gateway โดยตรง, Tailnet, SSH หรือ DNS-SD แบบพื้นที่กว้างช่วยหลีกเลี่ยงมัลติแคสต์ภายในเครื่องได้

2. **โหมดขั้นต่ำ** (ค่าเริ่มต้นเมื่อเปิดใช้ Bonjour และแนะนำสำหรับ Gateway ที่เปิดเผยต่อภายนอก): ละเว้นฟิลด์ละเอียดอ่อนจากการบรอดแคสต์ mDNS:

   ```json5
   {
     discovery: {
       mdns: { mode: "minimal" },
     },
   }
   ```

3. **ปิดโหมด mDNS** หากคุณต้องการเปิด Plugin ไว้แต่ระงับการค้นพบอุปกรณ์ภายในเครื่อง:

   ```json5
   {
     discovery: {
       mdns: { mode: "off" },
     },
   }
   ```

4. **โหมดเต็ม** (เลือกเปิด): รวม `cliPath` + `sshPort` ในระเบียน TXT:

   ```json5
   {
     discovery: {
       mdns: { mode: "full" },
     },
   }
   ```

5. **ตัวแปรสภาพแวดล้อม** (ทางเลือก): ตั้งค่า `OPENCLAW_DISABLE_BONJOUR=1` เพื่อปิดใช้ mDNS โดยไม่ต้องเปลี่ยน config

เมื่อเปิดใช้ Bonjour ในโหมดขั้นต่ำ Gateway จะบรอดแคสต์ข้อมูลที่เพียงพอสำหรับการค้นพบอุปกรณ์ (`role`, `gatewayPort`, `transport`) แต่ละเว้น `cliPath` และ `sshPort` แอปที่ต้องใช้ข้อมูลเส้นทาง CLI สามารถดึงข้อมูลผ่านการเชื่อมต่อ WebSocket ที่ผ่านการยืนยันตัวตนแล้วแทน

### ล็อกดาวน์ WebSocket ของ Gateway (การยืนยันตัวตนภายในเครื่อง)

ต้องใช้การยืนยันตัวตน Gateway **โดยค่าเริ่มต้น** หากไม่ได้กำหนดเส้นทางการยืนยันตัวตนของ gateway ที่ถูกต้องไว้
Gateway จะปฏิเสธการเชื่อมต่อ WebSocket (ปิดเมื่อผิดพลาด)

Onboarding จะสร้างโทเค็นตามค่าเริ่มต้น (แม้แต่สำหรับ loopback) ดังนั้น
ไคลเอนต์ภายในเครื่องต้องยืนยันตัวตน

ตั้งค่าโทเค็นเพื่อให้ไคลเอนต์ WS **ทั้งหมด** ต้องยืนยันตัวตน:

```json5
{
  gateway: {
    auth: { mode: "token", token: "your-token" },
  },
}
```

Doctor สามารถสร้างให้คุณได้: `openclaw doctor --generate-gateway-token`

<Note>
`gateway.remote.token` และ `gateway.remote.password` เป็นแหล่งข้อมูลรับรองของไคลเอนต์ สิ่งเหล่านี้ **ไม่ได้** ปกป้องการเข้าถึง WS ภายในเครื่องด้วยตัวเอง เส้นทางการเรียกภายในเครื่องสามารถใช้ `gateway.remote.*` เป็นทางสำรองได้เฉพาะเมื่อไม่ได้ตั้งค่า `gateway.auth.*` หาก `gateway.auth.token` หรือ `gateway.auth.password` ถูกกำหนดค่าอย่างชัดเจนผ่าน SecretRef และแก้ค่าไม่ได้ การแก้ค่าจะปิดเมื่อผิดพลาด (ไม่มีการใช้ทางสำรองระยะไกลมาปกปิด)
</Note>
ทางเลือก: ปักหมุด TLS ระยะไกลด้วย `gateway.remote.tlsFingerprint` เมื่อใช้ `wss://`
Plaintext `ws://` จำกัดเฉพาะ loopback ตามค่าเริ่มต้น สำหรับเส้นทางเครือข่ายส่วนตัวที่เชื่อถือ
ให้ตั้งค่า `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` บนโปรเซสไคลเอนต์เป็น
ทางออกฉุกเฉิน สิ่งนี้ตั้งใจให้เป็นสภาพแวดล้อมของโปรเซสเท่านั้น ไม่ใช่คีย์ config
ใน `openclaw.json`
การจับคู่อุปกรณ์มือถือและเส้นทาง Gateway แบบกำหนดเองหรือสแกนบน Android เข้มงวดกว่า:
ยอมรับ cleartext สำหรับ loopback แต่ชื่อโฮสต์ private-LAN, link-local, `.local` และ
ชื่อโฮสต์ที่ไม่มีจุดต้องใช้ TLS เว้นแต่คุณจะเลือกใช้เส้นทาง cleartext ของเครือข่ายส่วนตัว
ที่เชื่อถืออย่างชัดเจน

การจับคู่อุปกรณ์ภายในเครื่อง:

- การจับคู่อุปกรณ์ได้รับการอนุมัติอัตโนมัติสำหรับการเชื่อมต่อ local loopback โดยตรง เพื่อให้
  ไคลเอนต์บนโฮสต์เดียวกันทำงานได้ราบรื่น
- OpenClaw ยังมีเส้นทาง self-connect ของ backend/container-local แบบแคบสำหรับ
  โฟลว์ตัวช่วย shared-secret ที่เชื่อถือ
- การเชื่อมต่อ Tailnet และ LAN รวมถึงการ bind tailnet บนโฮสต์เดียวกัน จะถูกมองว่าเป็น
  ระยะไกลสำหรับการจับคู่และยังต้องได้รับการอนุมัติ
- หลักฐาน forwarded-header บนคำขอ loopback ทำให้ไม่ผ่านคุณสมบัติความเป็น loopback
  locality การอนุมัติอัตโนมัติสำหรับ metadata-upgrade ถูกจำกัดขอบเขตอย่างแคบ ดู
  [การจับคู่ Gateway](/th/gateway/pairing) สำหรับกฎทั้งสองข้อ

โหมดการยืนยันตัวตน:

- `gateway.auth.mode: "token"`: โทเค็น bearer แบบใช้ร่วมกัน (แนะนำสำหรับการตั้งค่าส่วนใหญ่)
- `gateway.auth.mode: "password"`: การยืนยันตัวตนด้วยรหัสผ่าน (ควรตั้งค่าผ่าน env: `OPENCLAW_GATEWAY_PASSWORD`)
- `gateway.auth.mode: "trusted-proxy"`: เชื่อถือ reverse proxy ที่รู้จักตัวตนเพื่อยืนยันตัวตนผู้ใช้และส่งผ่านตัวตนผ่าน headers (ดู [การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือ](/th/gateway/trusted-proxy-auth))

เช็กลิสต์การหมุนเวียน (โทเค็น/รหัสผ่าน):

1. สร้าง/ตั้งค่า secret ใหม่ (`gateway.auth.token` หรือ `OPENCLAW_GATEWAY_PASSWORD`)
2. รีสตาร์ต Gateway (หรือรีสตาร์ตแอป macOS หากแอปนั้นกำกับดูแล Gateway)
3. อัปเดตไคลเอนต์ระยะไกลใดๆ (`gateway.remote.token` / `.password` บนเครื่องที่เรียกเข้า Gateway)
4. ตรวจสอบว่าคุณไม่สามารถเชื่อมต่อด้วยข้อมูลรับรองเก่าได้อีก

### Headers ตัวตนของ Tailscale Serve

เมื่อ `gateway.auth.allowTailscale` เป็น `true` (ค่าเริ่มต้นสำหรับ Serve) OpenClaw
จะยอมรับ headers ตัวตนของ Tailscale Serve (`tailscale-user-login`) สำหรับการยืนยันตัวตน
Control UI/WebSocket OpenClaw ตรวจสอบตัวตนโดยแก้ที่อยู่
`x-forwarded-for` ผ่าน daemon ของ Tailscale ภายในเครื่อง (`tailscale whois`)
และจับคู่กับ header สิ่งนี้จะทำงานเฉพาะกับคำขอที่เข้ามาที่ loopback
และมี `x-forwarded-for`, `x-forwarded-proto` และ `x-forwarded-host` ตามที่
Tailscale แทรกเข้ามา
สำหรับเส้นทางตรวจสอบตัวตนแบบ async นี้ ความพยายามที่ล้มเหลวสำหรับ `{scope, ip}` เดียวกัน
จะถูกทำเป็นลำดับก่อนที่ limiter จะบันทึกความล้มเหลว ดังนั้นการลองซ้ำที่ไม่ถูกต้องพร้อมกัน
จากไคลเอนต์ Serve หนึ่งรายอาจทำให้ความพยายามครั้งที่สองถูกล็อกทันที
แทนที่จะแข่งผ่านไปเป็นการไม่ตรงกันธรรมดาสองครั้ง
ปลายทาง HTTP API (เช่น `/v1/*`, `/tools/invoke` และ `/api/channels/*`)
**ไม่** ใช้การยืนยันตัวตนผ่าน header ตัวตนของ Tailscale ปลายทางเหล่านี้ยังคงทำตาม
โหมดการยืนยันตัวตน HTTP ที่กำหนดค่าไว้ของ gateway

หมายเหตุขอบเขตสำคัญ:

- การยืนยันตัวตน HTTP bearer ของ Gateway มีผลเทียบเท่าการเข้าถึงของ operator แบบทั้งหมดหรือไม่มีเลย
- ให้ถือว่าข้อมูลรับรองที่เรียก `/v1/chat/completions`, `/v1/responses` หรือ `/api/channels/*` ได้ เป็น secret ของ operator ที่มีสิทธิ์เข้าถึงเต็มสำหรับ gateway นั้น
- บนพื้นผิว HTTP ที่เข้ากันได้กับ OpenAI การยืนยันตัวตน bearer แบบ shared-secret จะคืนค่าขอบเขต operator เริ่มต้นทั้งหมด (`operator.admin`, `operator.approvals`, `operator.pairing`, `operator.read`, `operator.talk.secrets`, `operator.write`) และความหมายของ owner สำหรับเทิร์นของ agent; ค่า `x-openclaw-scopes` ที่แคบกว่าไม่ได้ลดขอบเขตของเส้นทาง shared-secret นั้น
- ความหมายของ scope ต่อคำขอบน HTTP จะใช้เฉพาะเมื่อคำขอมาจากโหมดที่มีตัวตน เช่น การยืนยันตัวตนผ่าน trusted proxy หรือ `gateway.auth.mode="none"` บน ingress ส่วนตัว
- ในโหมดที่มีตัวตนเหล่านั้น หากละเว้น `x-openclaw-scopes` จะ fallback ไปยังชุด scope ค่าเริ่มต้นของ operator ตามปกติ; ส่ง header อย่างชัดเจนเมื่อคุณต้องการชุด scope ที่แคบกว่า
- `/tools/invoke` ทำตามกฎ shared-secret เดียวกัน: การยืนยันตัวตน bearer ด้วย token/password ถูกมองว่าเป็นการเข้าถึง operator เต็มรูปแบบที่นั่นเช่นกัน ขณะที่โหมดที่มีตัวตนยังคงเคารพ scope ที่ประกาศไว้
- อย่าแชร์ข้อมูลรับรองเหล่านี้กับผู้เรียกที่ไม่น่าเชื่อถือ; ควรใช้ Gateway แยกกันตามขอบเขตความเชื่อถือ

**สมมติฐานความเชื่อถือ:** การยืนยันตัวตน Serve แบบไม่ใช้โทเค็นถือว่าโฮสต์ gateway เป็นที่เชื่อถือ
อย่ามองว่าสิ่งนี้เป็นการป้องกันโปรเซสบนโฮสต์เดียวกันที่ไม่ประสงค์ดี หากโค้ดภายในเครื่อง
ที่ไม่น่าเชื่อถืออาจรันบนโฮสต์ gateway ให้ปิดใช้ `gateway.auth.allowTailscale`
และบังคับใช้การยืนยันตัวตน shared-secret อย่างชัดเจนด้วย `gateway.auth.mode: "token"` หรือ
`"password"`

**กฎความปลอดภัย:** อย่าส่งต่อ headers เหล่านี้จาก reverse proxy ของคุณเอง หาก
คุณ terminate TLS หรือ proxy ด้านหน้า gateway ให้ปิดใช้
`gateway.auth.allowTailscale` และใช้การยืนยันตัวตน shared-secret (`gateway.auth.mode:
"token"` หรือ `"password"`) หรือ [การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือ](/th/gateway/trusted-proxy-auth)
แทน

พร็อกซีที่เชื่อถือ:

- หากคุณ terminate TLS ด้านหน้า Gateway ให้ตั้งค่า `gateway.trustedProxies` เป็น IP ของพร็อกซีของคุณ
- OpenClaw จะเชื่อถือ `x-forwarded-for` (หรือ `x-real-ip`) จาก IP เหล่านั้นเพื่อกำหนด IP ของไคลเอนต์สำหรับการตรวจสอบการจับคู่ภายในเครื่องและการตรวจสอบ HTTP auth/local
- ตรวจสอบให้แน่ใจว่าพร็อกซีของคุณ **เขียนทับ** `x-forwarded-for` และบล็อกการเข้าถึงพอร์ต Gateway โดยตรง

ดู [Tailscale](/th/gateway/tailscale) และ [ภาพรวมเว็บ](/th/web)

### การควบคุมเบราว์เซอร์ผ่านโฮสต์ node (แนะนำ)

หาก Gateway ของคุณอยู่ระยะไกล แต่เบราว์เซอร์รันบนอีกเครื่องหนึ่ง ให้รัน **โฮสต์ node**
บนเครื่องที่มีเบราว์เซอร์ และให้ Gateway พร็อกซีการกระทำของเบราว์เซอร์ (ดู [เครื่องมือเบราว์เซอร์](/th/tools/browser))
ให้ถือว่าการจับคู่ node เหมือนการเข้าถึง admin

รูปแบบที่แนะนำ:

- ให้ Gateway และโฮสต์ node อยู่บน tailnet เดียวกัน (Tailscale)
- จับคู่ node อย่างตั้งใจ; ปิดใช้การกำหนดเส้นทางพร็อกซีของเบราว์เซอร์หากคุณไม่ต้องการ

หลีกเลี่ยง:

- การเปิดเผยพอร์ต relay/control ผ่าน LAN หรืออินเทอร์เน็ตสาธารณะ
- Tailscale Funnel สำหรับปลายทางควบคุมเบราว์เซอร์ (การเปิดเผยต่อสาธารณะ)

### Secrets บนดิสก์

ให้ถือว่าทุกอย่างภายใต้ `~/.openclaw/` (หรือ `$OPENCLAW_STATE_DIR/`) อาจมี secrets หรือข้อมูลส่วนตัว:

- `openclaw.json`: config อาจมีโทเค็น (gateway, gateway ระยะไกล), การตั้งค่า provider และ allowlists
- `credentials/**`: ข้อมูลรับรอง channel (ตัวอย่าง: ข้อมูลรับรอง WhatsApp), allowlists การจับคู่, การนำเข้า OAuth แบบเดิม
- `agents/<agentId>/agent/auth-profiles.json`: API keys, โปรไฟล์โทเค็น, โทเค็น OAuth และ `keyRef`/`tokenRef` แบบเลือกได้
- `agents/<agentId>/agent/codex-home/**`: บัญชี app-server ของ Codex ต่อ agent, config, Skills, plugins, สถานะ thread แบบ native และ diagnostics
- `secrets.json` (เลือกได้): payload secret ที่อิงไฟล์ซึ่งใช้โดย provider SecretRef แบบ `file` (`secrets.providers`)
- `agents/<agentId>/agent/auth.json`: ไฟล์ความเข้ากันได้แบบเดิม รายการ `api_key` แบบ static จะถูกขัดออกเมื่อพบ
- `agents/<agentId>/sessions/**`: transcript ของเซสชัน (`*.jsonl`) + metadata การกำหนดเส้นทาง (`sessions.json`) ที่อาจมีข้อความส่วนตัวและผลลัพธ์ของเครื่องมือ
- แพ็กเกจ Plugin ที่ bundled: plugins ที่ติดตั้งแล้ว (รวมถึง `node_modules/` ของพวกมัน)
- `sandboxes/**`: workspace sandbox ของเครื่องมือ; อาจสะสมสำเนาไฟล์ที่คุณอ่าน/เขียนภายใน sandbox

เคล็ดลับการเสริมความแข็งแกร่ง:

- รักษาสิทธิ์ให้เข้มงวด (`700` บน dirs, `600` บน files)
- ใช้การเข้ารหัสดิสก์ทั้งลูกบนโฮสต์ gateway
- ควรใช้บัญชีผู้ใช้ OS เฉพาะสำหรับ Gateway หากโฮสต์ถูกแชร์

### ไฟล์ `.env` ของ Workspace

OpenClaw โหลดไฟล์ `.env` ภายใน workspace สำหรับ agents และเครื่องมือ แต่ไม่เคยปล่อยให้ไฟล์เหล่านั้น override การควบคุม runtime ของ gateway อย่างเงียบๆ

- คีย์ใดๆ ที่ขึ้นต้นด้วย `OPENCLAW_*` จะถูกบล็อกจากไฟล์ `.env` ของ workspace ที่ไม่น่าเชื่อถือ
- การตั้งค่าปลายทาง channel สำหรับ Matrix, Mattermost, IRC และ Synology Chat ก็ถูกบล็อกจากการ override ผ่าน `.env` ของ workspace เช่นกัน ดังนั้น workspace ที่ clone มาจะไม่สามารถเปลี่ยนเส้นทางทราฟฟิก connector ที่ bundled ผ่าน config ปลายทางภายในเครื่องได้ คีย์ env ของปลายทาง (เช่น `MATRIX_HOMESERVER`, `MATTERMOST_URL`, `IRC_HOST`, `SYNOLOGY_CHAT_INCOMING_URL`) ต้องมาจากสภาพแวดล้อมของโปรเซส gateway หรือ `env.shellEnv` ไม่ใช่จาก `.env` ที่โหลดจาก workspace
- การบล็อกนี้เป็นแบบปิดเมื่อผิดพลาด: ตัวแปรควบคุม runtime ใหม่ที่เพิ่มใน release ในอนาคตไม่สามารถสืบทอดจาก `.env` ที่ commit ไว้หรือที่ผู้โจมตีจัดหาให้; คีย์จะถูกละเว้นและ gateway จะคงค่าของตัวเองไว้
- ตัวแปรสภาพแวดล้อมของ process/OS ที่เชื่อถือ (shell ของ gateway เอง, หน่วย launchd/systemd, app bundle) ยังคงมีผล - สิ่งนี้จำกัดเฉพาะการโหลดไฟล์ `.env`

เหตุผล: ไฟล์ `.env` ของ workspace มักอยู่ถัดจากโค้ด agent, ถูก commit โดยไม่ตั้งใจ หรือถูกเขียนโดยเครื่องมือ การบล็อก prefix `OPENCLAW_*` ทั้งหมดหมายความว่าการเพิ่ม flag `OPENCLAW_*` ใหม่ในภายหลังจะไม่มีทางถดถอยเป็นการสืบทอดอย่างเงียบๆ จากสถานะ workspace

### Logs และ transcripts (การ redaction และ retention)

Logs และ transcripts อาจรั่วไหลข้อมูลละเอียดอ่อนได้แม้เมื่อการควบคุมการเข้าถึงถูกต้อง:

- Logs ของ Gateway อาจมีสรุปเครื่องมือ, ข้อผิดพลาด และ URL
- Transcripts ของเซสชันอาจมี secrets ที่วางไว้, เนื้อหาไฟล์, ผลลัพธ์คำสั่ง และลิงก์

คำแนะนำ:

- เปิดการ redaction ของ log และ transcript ไว้ (`logging.redactSensitive: "tools"`; ค่าเริ่มต้น)
- เพิ่มรูปแบบกำหนดเองสำหรับสภาพแวดล้อมของคุณผ่าน `logging.redactPatterns` (โทเค็น, ชื่อโฮสต์, URL ภายใน)
- เมื่อแชร์ diagnostics ควรใช้ `openclaw status --all` (วางได้, secrets ถูก redacted) แทน logs ดิบ
- ล้าง transcripts ของเซสชันและไฟล์ log เก่าหากคุณไม่ต้องการเก็บไว้นาน

รายละเอียด: [Logging](/th/gateway/logging)

### แยกหมายเลขต่างหาก (WhatsApp, Signal, Telegram)

สำหรับช่องทางที่ใช้หมายเลขโทรศัพท์ ให้พิจารณาเรียกใช้ AI ของคุณบนหมายเลขโทรศัพท์ที่แยกจากหมายเลขส่วนตัว:

- หมายเลขส่วนตัว: บทสนทนาของคุณยังคงเป็นส่วนตัว
- หมายเลขบอต: AI จัดการรายการเหล่านี้ภายใต้ขอบเขตที่เหมาะสม

### โหมดอ่านอย่างเดียว (ผ่าน sandbox และเครื่องมือ)

คุณสามารถสร้างโปรไฟล์แบบอ่านอย่างเดียวได้โดยผสาน:

- `agents.defaults.sandbox.workspaceAccess: "ro"` (หรือ `"none"` หากไม่ให้เข้าถึง workspace)
- รายการอนุญาต/ปฏิเสธเครื่องมือที่บล็อก `write`, `edit`, `apply_patch`, `exec`, `process` เป็นต้น

ตัวเลือกเพิ่มความแข็งแรงเพิ่มเติม:

- `tools.exec.applyPatch.workspaceOnly: true` (ค่าเริ่มต้น): รับประกันว่า `apply_patch` ไม่สามารถเขียน/ลบนอกไดเรกทอรี workspace ได้ แม้เมื่อปิด sandbox อยู่ ตั้งเป็น `false` เฉพาะเมื่อคุณตั้งใจให้ `apply_patch` แตะไฟล์นอก workspace เท่านั้น
- `tools.fs.workspaceOnly: true` (ไม่บังคับ): จำกัดเส้นทาง `read`/`write`/`edit`/`apply_patch` และเส้นทางโหลดรูปภาพ prompt อัตโนมัติแบบ native ให้อยู่ในไดเรกทอรี workspace (มีประโยชน์หากวันนี้คุณอนุญาตเส้นทาง absolute และต้องการ guardrail เดียว)
- จำกัดรากของระบบไฟล์ให้แคบ: หลีกเลี่ยงรากที่กว้าง เช่น ไดเรกทอรี home ของคุณ สำหรับ workspace/sandbox workspace ของ agent รากที่กว้างอาจเปิดเผยไฟล์ในเครื่องที่ละเอียดอ่อน (เช่น state/config ใต้ `~/.openclaw`) ให้เครื่องมือระบบไฟล์เห็นได้

### ค่าพื้นฐานที่ปลอดภัย (คัดลอก/วาง)

ตัวอย่าง config แบบ "ค่าเริ่มต้นที่ปลอดภัย" ที่ทำให้ Gateway เป็นส่วนตัว ต้องจับคู่ DM และหลีกเลี่ยงบอตกลุ่มที่เปิดทำงานตลอดเวลา:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    port: 18789,
    auth: { mode: "token", token: "your-long-random-token" },
  },
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } },
    },
  },
}
```

หากคุณต้องการให้การเรียกใช้เครื่องมือ "ปลอดภัยกว่าโดยค่าเริ่มต้น" ด้วย ให้เพิ่ม sandbox + ปฏิเสธเครื่องมืออันตรายสำหรับ agent ที่ไม่ใช่เจ้าของ (ตัวอย่างอยู่ด้านล่างใน "โปรไฟล์การเข้าถึงต่อ agent")

ค่าพื้นฐานในตัวสำหรับรอบการทำงานของ agent ที่ขับเคลื่อนด้วยแชท: ผู้ส่งที่ไม่ใช่เจ้าของไม่สามารถใช้เครื่องมือ `cron` หรือ `gateway` ได้

## การแซนด์บ็อกซ์ (แนะนำ)

เอกสารเฉพาะ: [การแซนด์บ็อกซ์](/th/gateway/sandboxing)

สองแนวทางที่เสริมกัน:

- **เรียกใช้ Gateway ทั้งหมดใน Docker** (ขอบเขตคอนเทนเนอร์): [Docker](/th/install/docker)
- **Tool sandbox** (`agents.defaults.sandbox`, gateway บน host + เครื่องมือที่ถูกแยกด้วย sandbox; Docker เป็น backend เริ่มต้น): [การแซนด์บ็อกซ์](/th/gateway/sandboxing)

<Note>
เพื่อป้องกันการเข้าถึงข้าม agent ให้คง `agents.defaults.sandbox.scope` เป็น `"agent"` (ค่าเริ่มต้น) หรือ `"session"` สำหรับการแยกต่อ session ที่เข้มงวดยิ่งขึ้น `scope: "shared"` ใช้คอนเทนเนอร์หรือ workspace เดียวร่วมกัน
</Note>

พิจารณาการเข้าถึง workspace ของ agent ภายใน sandbox ด้วย:

- `agents.defaults.sandbox.workspaceAccess: "none"` (ค่าเริ่มต้น) ทำให้ workspace ของ agent เป็นพื้นที่ห้ามเข้าถึง; เครื่องมือทำงานกับ sandbox workspace ใต้ `~/.openclaw/sandboxes`
- `agents.defaults.sandbox.workspaceAccess: "ro"` mount workspace ของ agent แบบอ่านอย่างเดียวที่ `/agent` (ปิดใช้งาน `write`/`edit`/`apply_patch`)
- `agents.defaults.sandbox.workspaceAccess: "rw"` mount workspace ของ agent แบบอ่าน/เขียนที่ `/workspace`
- `sandbox.docker.binds` เพิ่มเติมจะถูกตรวจสอบกับเส้นทาง source ที่ normalize และ canonicalize แล้ว เทคนิค parent-symlink และ alias home แบบ canonical ยังล้มเหลวแบบปิดหาก resolve ไปยังรากที่ถูกบล็อก เช่น `/etc`, `/var/run` หรือไดเรกทอรี credential ใต้ home ของ OS

<Warning>
`tools.elevated` คือช่องทางหลบหนีค่าพื้นฐานระดับ global ที่เรียกใช้ exec นอก sandbox host ที่มีผลคือ `gateway` โดยค่าเริ่มต้น หรือ `node` เมื่อเป้าหมาย exec ถูกกำหนดค่าเป็น `node` ควรจำกัด `tools.elevated.allowFrom` ให้แน่น และอย่าเปิดใช้กับคนแปลกหน้า คุณสามารถจำกัด elevated เพิ่มเติมต่อ agent ผ่าน `agents.list[].tools.elevated` ได้ ดู [โหมด Elevated](/th/tools/elevated)
</Warning>

### Guardrail สำหรับการมอบหมาย sub-agent

หากคุณอนุญาตเครื่องมือ session ให้ถือว่าการเรียกใช้ sub-agent ที่มอบหมายเป็นการตัดสินใจเรื่องขอบเขตอีกแบบหนึ่ง:

- ปฏิเสธ `sessions_spawn` เว้นแต่ agent จำเป็นต้องมอบหมายงานจริง ๆ
- จำกัด `agents.defaults.subagents.allowAgents` และ override ต่อ agent ใด ๆ ใน `agents.list[].subagents.allowAgents` ให้เฉพาะ target agent ที่ทราบว่าปลอดภัย
- สำหรับ workflow ใด ๆ ที่ต้องยังคงอยู่ใน sandbox ให้เรียก `sessions_spawn` พร้อม `sandbox: "require"` (ค่าเริ่มต้นคือ `inherit`)
- `sandbox: "require"` จะล้มเหลวทันทีเมื่อ runtime ลูกเป้าหมายไม่ได้อยู่ใน sandbox

## ความเสี่ยงของการควบคุมเบราว์เซอร์

การเปิดใช้การควบคุมเบราว์เซอร์ทำให้โมเดลสามารถขับเคลื่อนเบราว์เซอร์จริงได้
หากโปรไฟล์เบราว์เซอร์นั้นมี session ที่เข้าสู่ระบบไว้อยู่แล้ว โมเดลสามารถ
เข้าถึงบัญชีและข้อมูลเหล่านั้นได้ ให้ถือว่าโปรไฟล์เบราว์เซอร์เป็น **สถานะที่ละเอียดอ่อน**:

- ควรใช้โปรไฟล์เฉพาะสำหรับ agent (โปรไฟล์เริ่มต้น `openclaw`)
- หลีกเลี่ยงการชี้ agent ไปยังโปรไฟล์ส่วนตัวที่คุณใช้ประจำทุกวัน
- ปิดการควบคุมเบราว์เซอร์บน host สำหรับ agent ที่อยู่ใน sandbox เว้นแต่คุณจะไว้ใจ agent เหล่านั้น
- API ควบคุมเบราว์เซอร์แบบ standalone loopback ยอมรับเฉพาะ auth แบบ shared-secret
  (gateway token bearer auth หรือรหัสผ่าน gateway) และไม่ใช้ header ตัวตน
  trusted-proxy หรือ Tailscale Serve
- ถือว่าไฟล์ดาวน์โหลดจากเบราว์เซอร์เป็น input ที่ไม่น่าเชื่อถือ; ควรใช้ไดเรกทอรี downloads ที่แยกไว้
- ปิด browser sync/password managers ในโปรไฟล์ agent หากเป็นไปได้ (ลดรัศมีผลกระทบ)
- สำหรับ remote gateways ให้ถือว่า "การควบคุมเบราว์เซอร์" เทียบเท่า "การเข้าถึงของ operator" ต่อทุกอย่างที่โปรไฟล์นั้นเข้าถึงได้
- คงให้ host ของ Gateway และ node อยู่เฉพาะใน tailnet; หลีกเลี่ยงการเปิดเผยพอร์ตควบคุมเบราว์เซอร์ไปยัง LAN หรืออินเทอร์เน็ตสาธารณะ
- ปิดการ route ผ่าน browser proxy เมื่อคุณไม่ต้องใช้ (`gateway.nodes.browser.mode="off"`)
- Chrome MCP existing-session mode **ไม่ได้** "ปลอดภัยกว่า"; มันสามารถกระทำแทนคุณในทุกสิ่งที่โปรไฟล์ Chrome บน host นั้นเข้าถึงได้

### นโยบาย Browser SSRF (เข้มงวดโดยค่าเริ่มต้น)

นโยบายการนำทางเบราว์เซอร์ของ OpenClaw เข้มงวดโดยค่าเริ่มต้น: ปลายทาง private/internal จะยังคงถูกบล็อก เว้นแต่คุณ opt in อย่างชัดเจน

- ค่าเริ่มต้น: ไม่ได้ตั้งค่า `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` ดังนั้นการนำทางเบราว์เซอร์จะยังคงบล็อกปลายทาง private/internal/special-use
- Alias เดิม: `browser.ssrfPolicy.allowPrivateNetwork` ยังยอมรับเพื่อความเข้ากันได้
- โหมด opt-in: ตั้ง `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` เพื่ออนุญาตปลายทาง private/internal/special-use
- ในโหมดเข้มงวด ให้ใช้ `hostnameAllowlist` (pattern เช่น `*.example.com`) และ `allowedHostnames` (ข้อยกเว้น host แบบตรงตัว รวมถึงชื่อที่ถูกบล็อก เช่น `localhost`) สำหรับข้อยกเว้นที่ชัดเจน
- การนำทางจะถูกตรวจสอบก่อน request และพยายามตรวจซ้ำบน URL `http(s)` สุดท้ายหลังการนำทาง เพื่อลด pivot ที่อาศัย redirect

ตัวอย่างนโยบายเข้มงวด:

```json5
{
  browser: {
    ssrfPolicy: {
      dangerouslyAllowPrivateNetwork: false,
      hostnameAllowlist: ["*.example.com", "example.com"],
      allowedHostnames: ["localhost"],
    },
  },
}
```

## โปรไฟล์การเข้าถึงต่อ agent (หลาย agent)

ด้วยการ route แบบหลาย agent แต่ละ agent สามารถมี sandbox + นโยบายเครื่องมือของตัวเอง:
ใช้สิ่งนี้เพื่อให้ **การเข้าถึงเต็มรูปแบบ**, **อ่านอย่างเดียว** หรือ **ไม่ให้เข้าถึง** ต่อ agent
ดูรายละเอียดทั้งหมดและกฎ precedence ได้ที่ [Multi-Agent Sandbox & Tools](/th/tools/multi-agent-sandbox-tools)

กรณีใช้งานทั่วไป:

- agent ส่วนตัว: เข้าถึงเต็มรูปแบบ, ไม่มี sandbox
- agent สำหรับครอบครัว/งาน: อยู่ใน sandbox + เครื่องมืออ่านอย่างเดียว
- agent สาธารณะ: อยู่ใน sandbox + ไม่มีเครื่องมือระบบไฟล์/shell

### ตัวอย่าง: เข้าถึงเต็มรูปแบบ (ไม่มี sandbox)

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        workspace: "~/.openclaw/workspace-personal",
        sandbox: { mode: "off" },
      },
    ],
  },
}
```

### ตัวอย่าง: เครื่องมืออ่านอย่างเดียว + workspace อ่านอย่างเดียว

```json5
{
  agents: {
    list: [
      {
        id: "family",
        workspace: "~/.openclaw/workspace-family",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "ro",
        },
        tools: {
          allow: ["read"],
          deny: ["write", "edit", "apply_patch", "exec", "process", "browser"],
        },
      },
    ],
  },
}
```

## การตอบสนองต่อเหตุการณ์

หาก AI ของคุณทำสิ่งที่ไม่ดี:

### จำกัดความเสียหาย

1. **หยุดมัน:** หยุดแอป macOS (หากแอปนั้น supervise Gateway) หรือ terminate process `openclaw gateway` ของคุณ
2. **ปิดการเปิดเผย:** ตั้ง `gateway.bind: "loopback"` (หรือปิดใช้ Tailscale Funnel/Serve) จนกว่าคุณจะเข้าใจว่าเกิดอะไรขึ้น
3. **หยุดการเข้าถึงชั่วคราว:** สลับ DM/groups ที่เสี่ยงไปเป็น `dmPolicy: "disabled"` / require mentions และลบรายการ allow-all `"*"` หากคุณมีรายการเหล่านั้น

### หมุนเวียนค่าใหม่ (ถือว่าถูก compromise หาก secret รั่วไหล)

1. หมุนเวียน auth ของ Gateway (`gateway.auth.token` / `OPENCLAW_GATEWAY_PASSWORD`) แล้ว restart
2. หมุนเวียน secret ของ remote client (`gateway.remote.token` / `.password`) บนเครื่องใด ๆ ที่สามารถเรียก Gateway ได้
3. หมุนเวียน credential ของ provider/API (WhatsApp creds, Slack/Discord tokens, model/API keys ใน `auth-profiles.json` และค่า payload secret ที่เข้ารหัสเมื่อใช้งาน)

### ตรวจสอบ

1. ตรวจ log ของ Gateway: `/tmp/openclaw/openclaw-YYYY-MM-DD.log` (หรือ `logging.file`)
2. ตรวจ transcript ที่เกี่ยวข้อง: `~/.openclaw/agents/<agentId>/sessions/*.jsonl`
3. ตรวจการเปลี่ยนแปลง config ล่าสุด (สิ่งใด ๆ ที่อาจขยายการเข้าถึง: `gateway.bind`, `gateway.auth`, นโยบาย dm/group, `tools.elevated`, การเปลี่ยนแปลง Plugin)
4. เรียก `openclaw security audit --deep` อีกครั้ง และยืนยันว่า critical findings ได้รับการแก้ไขแล้ว

### รวบรวมสำหรับรายงาน

- Timestamp, OS ของ gateway host + เวอร์ชัน OpenClaw
- transcript ของ session + tail log สั้น ๆ (หลัง redacting)
- สิ่งที่ผู้โจมตีส่ง + สิ่งที่ agent ทำ
- Gateway ถูกเปิดเผยเกิน loopback หรือไม่ (LAN/Tailscale Funnel/Serve)

## การสแกน secret

CI เรียกใช้ hook pre-commit `detect-private-key` ทั่ว repository หากล้มเหลว
ให้ลบหรือหมุนเวียน key material ที่ commit แล้ว จากนั้น reproduce ในเครื่อง:
