คู่มือฉบับย่อสำหรับตรวจสอบการเชื่อมต่อของช่องทางโดยไม่ต้องคาดเดา

## การตรวจสอบด่วน

- `openclaw status` - สรุปภายในเครื่อง: การเข้าถึง/โหมดของ Gateway คำแนะนำการอัปเดต อายุการยืนยันตัวตนของช่องทางที่เชื่อมโยง เซสชันและกิจกรรมล่าสุด
- `openclaw status --all` - การวินิจฉัยภายในเครื่องแบบครบถ้วน (อ่านอย่างเดียว มีสี และปลอดภัยสำหรับวางเพื่อดีบัก)
- `openclaw status --deep` - ขอให้ Gateway ที่กำลังทำงานทำการตรวจสอบแบบสด (`health` พร้อม `probe:true`) รวมถึงการตรวจสอบช่องทางแยกตามบัญชีเมื่อรองรับ
- `openclaw status --usage` - แสดงสแนปช็อตการใช้งาน/โควตาของผู้ให้บริการโมเดล
- `openclaw health` - ขอให้ Gateway ที่กำลังทำงานส่งสแนปช็อตสถานะสุขภาพ (ผ่าน WS เท่านั้น CLI จะไม่เชื่อมต่อซ็อกเก็ตของช่องทางโดยตรง)
- `openclaw health --verbose` (นามแฝง `--debug`) - บังคับตรวจสอบสถานะสุขภาพแบบสดและแสดงรายละเอียดการเชื่อมต่อ Gateway
- `openclaw health --json` - เอาต์พุตสแนปช็อตสถานะสุขภาพที่เครื่องอ่านได้
- ส่ง `/status` เป็นคำสั่งแชตเดี่ยวในช่องทางใดก็ได้ เพื่อรับการตอบกลับสถานะโดยไม่เรียกใช้เอเจนต์
- บันทึก: ติดตาม `/tmp/openclaw/openclaw-*.log` และกรองหา `web-heartbeat`, `web-reconnect`, `web-auto-reply`, `web-inbound`

สำหรับ Discord และผู้ให้บริการแชตอื่น ๆ แถวเซสชันไม่ได้แสดงว่าซ็อกเก็ตยังทำงานอยู่
`openclaw sessions`, Gateway `sessions.list` และเครื่องมือ `sessions_list` ของเอเจนต์
อ่านสถานะการสนทนาที่จัดเก็บไว้ ผู้ให้บริการอาจเชื่อมต่อใหม่และแสดงสถานะช่องทางว่าปกติ
ก่อนที่จะสร้างแถวเซสชันใหม่ ใช้คำสั่งสถานะช่องทางและ
สถานะสุขภาพข้างต้นเพื่อตรวจสอบการเชื่อมต่อแบบสด

## การวินิจฉัยเชิงลึก

- ข้อมูลรับรองบนดิสก์: `ls -l ~/.openclaw/credentials/whatsapp/<accountId>/creds.json` (mtime ควรเป็นเวลาล่าสุด)
- ที่เก็บเซสชัน: `ls -l ~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite` จำนวนและผู้รับล่าสุดจะแสดงผ่าน `status`
- ขั้นตอนเชื่อมโยงใหม่: `openclaw channels logout && openclaw channels login --verbose` เมื่อรหัสสถานะเป็น 409-515 หรือพบ `loggedOut` ในบันทึก ขั้นตอนเข้าสู่ระบบด้วย QR จะเริ่มใหม่โดยอัตโนมัติหนึ่งครั้งสำหรับสถานะ 515 หลังจับคู่
- การวินิจฉัยเปิดใช้งานเป็นค่าเริ่มต้น (`diagnostics.enabled: false` ใช้ปิดการทำงาน) เหตุการณ์หน่วยความจำจะบันทึกจำนวนไบต์ RSS/ฮีป รวมถึงแรงกดดันจากเกณฑ์/การเติบโต แรงกดดันหน่วยความจำระดับวิกฤตจะถูกบันทึกผ่านตัวบันทึกของ Gateway และเมื่อกำหนด `diagnostics.memoryPressureSnapshot: true` ไว้ ระบบจะเขียนชุดข้อมูลความเสถียรก่อน OOM เพิ่มเติมด้วย (สถิติฮีป V8, ตัวนับ cgroup ของ Linux เมื่อมี, จำนวนทรัพยากรที่ใช้งานอยู่, ไฟล์เซสชัน/ทรานสคริปต์ที่ใหญ่ที่สุดตามพาธสัมพัทธ์ที่ปกปิดข้อมูลแล้ว) คำเตือนความพร้อมใช้งานจะบันทึกความล่าช้า/การใช้งานของลูปเหตุการณ์ อัตราส่วนคอร์ CPU และจำนวนเซสชันที่ทำงาน/รอ/อยู่ในคิว เมื่อกระบวนการยังทำงานอยู่แต่เต็มขีดความสามารถ เหตุการณ์เพย์โหลดขนาดใหญ่เกินกำหนดจะบันทึกสิ่งที่ถูกปฏิเสธ/ตัดทอน/แบ่งส่วน พร้อมขนาดและขีดจำกัด แต่จะไม่บันทึกข้อความ เนื้อหาไฟล์แนบ เนื้อหา Webhook เนื้อหาคำขอ/การตอบกลับดิบ โทเค็น คุกกี้ หรือค่าลับ
- Heartbeat เดียวกันจะขับเคลื่อนตัวบันทึกความเสถียรแบบจำกัดขนาด: `openclaw gateway stability` (หรือ Gateway RPC `diagnostics.stability`) การออกจาก Gateway แบบร้ายแรง การหมดเวลาขณะปิดระบบ ความล้มเหลวในการเริ่มทำงานหลังรีสตาร์ต และแรงกดดันหน่วยความจำระดับวิกฤต (เมื่อ `diagnostics.memoryPressureSnapshot: true`) จะบันทึกสแนปช็อตล่าสุดไว้ภายใต้ `~/.openclaw/logs/stability/` ตรวจสอบชุดข้อมูลล่าสุดด้วย `openclaw gateway stability --bundle latest`
- สำหรับรายงานข้อบกพร่อง ให้เรียกใช้ `openclaw gateway diagnostics export` และแนบไฟล์ zip ที่สร้างขึ้น ซึ่งประกอบด้วยสรุป Markdown ชุดข้อมูลความเสถียรล่าสุด เมทาดาทาบันทึกที่ผ่านการล้างข้อมูล สแนปช็อตสถานะ/สุขภาพของ Gateway ที่ผ่านการล้างข้อมูล และโครงสร้างการกำหนดค่า ข้อความแชต เนื้อหา Webhook เอาต์พุตเครื่องมือ ข้อมูลรับรอง คุกกี้ ตัวระบุบัญชี/ข้อความ และค่าลับจะถูกละเว้นหรือปกปิด ดู [การส่งออกข้อมูลวินิจฉัย](/th/gateway/diagnostics)

## การกำหนดค่าตัวตรวจสอบสถานะสุขภาพ

- `gateway.channelHealthCheckMinutes`: ความถี่ที่ Gateway ตรวจสอบสถานะสุขภาพของช่องทาง ค่าเริ่มต้น: `5` กำหนด `0` เพื่อปิดการรีสตาร์ตโดยตัวตรวจสอบสถานะสุขภาพทั่วทั้งระบบ
- `gateway.channelStaleEventThresholdMinutes`: ระยะเวลาที่ช่องทางซึ่งเชื่อมต่อแล้วสามารถไม่มีการใช้งาน ก่อนที่ตัวตรวจสอบสถานะสุขภาพจะถือว่าล้าสมัยและรีสตาร์ต ค่าเริ่มต้น: `30` ค่านี้ต้องมากกว่าหรือเท่ากับ `gateway.channelHealthCheckMinutes`
- `gateway.channelMaxRestartsPerHour`: ขีดจำกัดแบบต่อเนื่องหนึ่งชั่วโมงสำหรับจำนวนการรีสตาร์ตโดยตัวตรวจสอบสถานะสุขภาพต่อช่องทาง/บัญชี ค่าเริ่มต้น: `10`
- `channels.<provider>.healthMonitor.enabled`: ปิดการรีสตาร์ตโดยตัวตรวจสอบสถานะสุขภาพสำหรับช่องทางที่ระบุ โดยยังคงเปิดการตรวจสอบทั่วทั้งระบบ
- `channels.<provider>.accounts.<accountId>.healthMonitor.enabled`: ค่าลบล้างสำหรับหลายบัญชีซึ่งมีลำดับความสำคัญเหนือการตั้งค่าระดับช่องทาง
- ค่าลบล้างรายช่องทางเหล่านี้ใช้กับช่องทางในตัวที่เปิดให้กำหนดค่าได้ในปัจจุบัน ได้แก่ Discord, Google Chat, iMessage, IRC, Microsoft Teams, Signal, Slack, Telegram และ WhatsApp

## การตรวจสอบเวลาทำงาน

บริการตรวจสอบเวลาทำงานภายนอกควรใช้เอนด์พอยต์ `/health` โดยเฉพาะ ไม่ใช่ `/v1/chat/completions`

- **ควรใช้:** `GET /health` - ตอบกลับทันที ไม่สร้างเซสชัน ไม่เรียก LLM และส่งคืน `{"ok":true,"status":"live"}`
- **อย่าใช้:** `/v1/chat/completions` สำหรับการตรวจสอบสถานะสุขภาพ - ทุกคำขอจะสร้างเซสชันเอเจนต์แบบเต็ม พร้อมสแนปช็อต Skills การประกอบบริบท และการเรียก LLM

เมื่อไม่มีส่วนหัว `x-openclaw-session-key` หรือฟิลด์ `user` ระบบ `/v1/chat/completions` จะสร้างเซสชันแบบสุ่มใหม่สำหรับทุกคำขอ บริการตรวจสอบที่ส่งคำขอทุก 15 นาทีจะสร้างประมาณ 96 เซสชัน/วัน โดยแต่ละเซสชันใช้พื้นที่ 4-22KB เมื่อเวลาผ่านไปจะทำให้ที่เก็บเซสชันมีขนาดใหญ่เกินจำเป็น และอาจทำให้หน้าต่างบริบทล้น

### ตัวอย่างการตั้งค่าบริการตรวจสอบ

- **BetterStack:** กำหนด URL ตรวจสอบสถานะสุขภาพเป็น `https://<your-gateway-host>:<port>/health`
- **UptimeRobot:** เพิ่มตัวตรวจสอบ HTTP ใหม่โดยใช้ URL `https://<your-gateway-host>:<port>/health`
- **ทั่วไป:** คำขอ HTTP GET ใด ๆ ไปยัง `/health` จะส่งคืน 200 พร้อม `{"ok":true}` เมื่อ Gateway มีสถานะปกติ

## เมื่อเกิดความล้มเหลว

- `logged out` หรือสถานะ 409-515 -> เชื่อมโยงใหม่ด้วย `openclaw channels logout` แล้วตามด้วย `openclaw channels login`
- เข้าถึง Gateway ไม่ได้ -> เริ่มการทำงานด้วย: `openclaw gateway --port 18789` (ใช้ `--force` หากพอร์ตไม่ว่าง)
- ไม่มีข้อความขาเข้า -> ยืนยันว่าโทรศัพท์ที่เชื่อมโยงออนไลน์อยู่และอนุญาตผู้ส่งแล้ว (`channels.whatsapp.allowFrom`) สำหรับแชตกลุ่ม ตรวจสอบว่ารายการอนุญาตและกฎการกล่าวถึงตรงกัน (`channels.whatsapp.groups`, `agents.list[].groupChat.mentionPatterns`)

## คำสั่ง "health" โดยเฉพาะ

`openclaw health` ขอให้ Gateway ที่กำลังทำงานส่งสแนปช็อตสถานะสุขภาพ (ไม่มีการเชื่อมต่อซ็อกเก็ต
ของช่องทางโดยตรงจาก CLI) โดยค่าเริ่มต้น คำสั่งจะส่งคืนสแนปช็อต Gateway ที่แคชไว้และยังใหม่
จากนั้น Gateway จะรีเฟรชแคชดังกล่าวในเบื้องหลัง ส่วน `--verbose` จะบังคับตรวจสอบแบบสดแทน
คำสั่งนี้รายงานข้อมูลรับรองที่เชื่อมโยง/อายุการยืนยันตัวตนเมื่อมี สรุปการตรวจสอบรายช่องทาง
สรุปที่เก็บเซสชัน และระยะเวลาการตรวจสอบ คำสั่งจะออกด้วยรหัสที่ไม่ใช่ศูนย์หาก
เข้าถึง Gateway ไม่ได้ หรือการตรวจสอบล้มเหลว/หมดเวลา

ตัวเลือก:

- `--json`: เอาต์พุต JSON ที่เครื่องอ่านได้
- `--timeout <ms>`: ลบล้างระยะหมดเวลาการตรวจสอบเริ่มต้น 10s
- `--verbose`: บังคับตรวจสอบแบบสดและแสดงรายละเอียดการเชื่อมต่อ Gateway
- `--debug`: นามแฝงสำหรับ `--verbose`

สแนปช็อตสถานะสุขภาพประกอบด้วย: `ok` (ค่าบูลีน), `ts` (การประทับเวลา), `durationMs` (เวลาที่ใช้ตรวจสอบ), สถานะรายช่องทาง ความพร้อมใช้งานของเอเจนต์ และสรุปที่เก็บเซสชัน
