# การบันทึก

สำหรับภาพรวมที่ผู้ใช้เห็น (CLI + Control UI + การกำหนดค่า) ดู [/logging](/th/logging)

OpenClaw มี “พื้นผิว” การบันทึกสองแบบ:

- **เอาต์พุตคอนโซล** (สิ่งที่คุณเห็นในเทอร์มินัล / Debug UI)
- **บันทึกไฟล์** (บรรทัด JSON) ที่เขียนโดยตัวบันทึกของ Gateway

## ตัวบันทึกแบบไฟล์

- ไฟล์บันทึกแบบหมุนเวียนเริ่มต้นอยู่ใต้ `/tmp/openclaw/` (หนึ่งไฟล์ต่อวัน): `openclaw-YYYY-MM-DD.log`
  - วันที่ใช้เขตเวลาท้องถิ่นของโฮสต์ Gateway
- ไฟล์บันทึกที่ใช้งานอยู่จะหมุนเวียนเมื่อถึง `logging.maxFileBytes` (ค่าเริ่มต้น: 100 MB) โดยเก็บ
  ไฟล์เก็บถาวรแบบมีหมายเลขไว้ได้สูงสุดห้าไฟล์ และเขียนไฟล์ที่ใช้งานอยู่ไฟล์ใหม่ต่อไป
- สามารถกำหนดค่าพาธและระดับของไฟล์บันทึกผ่าน `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

รูปแบบไฟล์คือออบเจ็กต์ JSON หนึ่งรายการต่อบรรทัด

แท็บ Logs ของ Control UI จะติดตามไฟล์นี้ผ่าน Gateway (`logs.tail`)
CLI ทำแบบเดียวกันได้:

```bash
openclaw logs --follow
```

**โหมดรายละเอียดเทียบกับระดับบันทึก**

- **บันทึกไฟล์** ถูกควบคุมโดย `logging.level` เท่านั้น
- `--verbose` มีผลเฉพาะกับ **ระดับรายละเอียดของคอนโซล** (และรูปแบบบันทึก WS); มัน **ไม่**
  เพิ่มระดับบันทึกของไฟล์
- หากต้องการจับรายละเอียดที่มีเฉพาะในโหมดรายละเอียดลงในบันทึกไฟล์ ให้ตั้ง `logging.level` เป็น `debug` หรือ
  `trace`

## การจับเอาต์พุตคอนโซล

CLI จับ `console.log/info/warn/error/debug/trace` และเขียนลงในบันทึกไฟล์
ในขณะที่ยังคงพิมพ์ไปยัง stdout/stderr

คุณสามารถปรับระดับรายละเอียดของคอนโซลได้อย่างอิสระผ่าน:

- `logging.consoleLevel` (ค่าเริ่มต้น `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## การปกปิดข้อมูล

OpenClaw สามารถปกปิดโทเค็นที่ละเอียดอ่อนก่อนที่เอาต์พุตบันทึกหรือทรานสคริปต์จะออกจาก
โปรเซส นโยบายการปกปิดข้อมูลของการบันทึกนี้จะถูกใช้กับซิงก์ข้อความคอนโซล, บันทึกไฟล์, ระเบียนบันทึก OTLP
และทรานสคริปต์เซสชัน ดังนั้นค่าลับที่ตรงกันจะถูก
ปกปิดก่อนที่บรรทัด JSONL หรือข้อความจะถูกเขียนลงดิสก์

- `logging.redactSensitive`: `off` | `tools` (ค่าเริ่มต้น: `tools`)
- `logging.redactPatterns`: อาร์เรย์ของสตริง regex (แทนที่ค่าเริ่มต้น)
  - ใช้สตริง regex ดิบ (เพิ่ม `gi` อัตโนมัติ) หรือ `/pattern/flags` หากคุณต้องการแฟล็กแบบกำหนดเอง
  - รายการที่ตรงกันจะถูกปกปิดโดยเก็บอักขระ 6 ตัวแรก + 4 ตัวสุดท้ายไว้ (ความยาว >= 18) มิฉะนั้นใช้ `***`
  - ค่าเริ่มต้นครอบคลุมการกำหนดคีย์ทั่วไป, แฟล็ก CLI, ฟิลด์ JSON, ส่วนหัว bearer, บล็อก PEM และคำนำหน้าโทเค็นยอดนิยม

ขอบเขตความปลอดภัยบางส่วนจะปกปิดข้อมูลเสมอไม่ว่า `logging.redactSensitive` จะเป็นค่าใด
ซึ่งรวมถึงเหตุการณ์การเรียกเครื่องมือของ Control UI, เอาต์พุตเครื่องมือ `sessions_history`,
การส่งออกการสนับสนุนด้านการวินิจฉัย, การสังเกตข้อผิดพลาดของผู้ให้บริการ, การแสดงคำสั่งอนุมัติ exec
และบันทึกโปรโตคอล WebSocket ของ Gateway พื้นผิวเหล่านี้อาจยังใช้
`logging.redactPatterns` เป็นรูปแบบเพิ่มเติมได้ แต่ `redactSensitive: "off"`
จะไม่ทำให้มันปล่อยค่าลับดิบออกมา

## บันทึก WebSocket ของ Gateway

Gateway พิมพ์บันทึกโปรโตคอล WebSocket ในสองโหมด:

- **โหมดปกติ (ไม่มี `--verbose`)**: พิมพ์เฉพาะผลลัพธ์ RPC ที่ “น่าสนใจ”:
  - ข้อผิดพลาด (`ok=false`)
  - การเรียกที่ช้า (ค่าเกณฑ์เริ่มต้น: `>= 50ms`)
  - ข้อผิดพลาดในการแยกวิเคราะห์
- **โหมดรายละเอียด (`--verbose`)**: พิมพ์ทราฟฟิกคำขอ/การตอบกลับ WS ทั้งหมด

### รูปแบบบันทึก WS

`openclaw gateway` รองรับสวิตช์รูปแบบต่อ Gateway:

- `--ws-log auto` (ค่าเริ่มต้น): โหมดปกติถูกปรับให้เหมาะสม; โหมดรายละเอียดใช้เอาต์พุตแบบกะทัดรัด
- `--ws-log compact`: เอาต์พุตแบบกะทัดรัด (คำขอ/การตอบกลับเป็นคู่) เมื่ออยู่ในโหมดรายละเอียด
- `--ws-log full`: เอาต์พุตเต็มรายเฟรมเมื่ออยู่ในโหมดรายละเอียด
- `--compact`: ชื่อแฝงของ `--ws-log compact`

ตัวอย่าง:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## การจัดรูปแบบคอนโซล (การบันทึกของระบบย่อย)

ตัวจัดรูปแบบคอนโซล **รับรู้ TTY** และพิมพ์บรรทัดที่สม่ำเสมอพร้อมคำนำหน้า
ตัวบันทึกของระบบย่อยช่วยจัดกลุ่มเอาต์พุตและทำให้อ่านกวาดได้ง่าย

พฤติกรรม:

- **คำนำหน้าระบบย่อย** ในทุกบรรทัด (เช่น `[gateway]`, `[canvas]`, `[tailscale]`)
- **สีของระบบย่อย** (คงที่ต่อระบบย่อย) รวมกับการใส่สีตามระดับ
- **มีสีเมื่อเอาต์พุตเป็น TTY หรือสภาพแวดล้อมดูเหมือนเทอร์มินัลแบบสมบูรณ์** (`TERM`/`COLORTERM`/`TERM_PROGRAM`) และเคารพ `NO_COLOR`
- **คำนำหน้าระบบย่อยแบบย่อ**: ตัด `gateway/` + `channels/` ด้านหน้าออก และเก็บ 2 ส่วนสุดท้ายไว้ (เช่น `whatsapp/outbound`)
- **ตัวบันทึกย่อยตามระบบย่อย** (คำนำหน้าอัตโนมัติ + ฟิลด์แบบมีโครงสร้าง `{ subsystem }`)
- **`logRaw()`** สำหรับเอาต์พุต QR/UX (ไม่มีคำนำหน้า ไม่มีการจัดรูปแบบ)
- **รูปแบบคอนโซล** (เช่น `pretty | compact | json`)
- **ระดับบันทึกคอนโซล** แยกจากระดับบันทึกไฟล์ (ไฟล์ยังเก็บรายละเอียดเต็มเมื่อ `logging.level` ถูกตั้งเป็น `debug`/`trace`)
- **เนื้อหาข้อความ WhatsApp** ถูกบันทึกที่ `debug` (ใช้ `--verbose` เพื่อดู)

สิ่งนี้ช่วยให้บันทึกไฟล์เดิมยังคงเสถียร ในขณะที่ทำให้เอาต์พุตแบบโต้ตอบอ่านกวาดได้ง่าย
