# การบันทึกล็อก

สำหรับภาพรวมที่ผู้ใช้เห็น (CLI + Control UI + การกำหนดค่า) ดูที่ [/logging](/th/logging)

OpenClaw มี “พื้นผิว” ของล็อกสองแบบ:

- **เอาต์พุตคอนโซล** (สิ่งที่คุณเห็นในเทอร์มินัล / Debug UI)
- **ล็อกไฟล์** (บรรทัด JSON) ที่เขียนโดยตัวบันทึกล็อกของ Gateway

เมื่อเริ่มต้น Gateway จะบันทึกโมเดลเริ่มต้นของเอเจนต์ที่ resolve แล้ว พร้อมกับ
ค่าเริ่มต้นของโหมดที่มีผลต่อเซสชันใหม่ เช่น:

```text
agent model: openai-codex/gpt-5.5 (thinking=medium, fast=on)
```

`thinking` มาจากเอเจนต์เริ่มต้น, พารามิเตอร์โมเดล, หรือค่าเริ่มต้นส่วนกลางของเอเจนต์;
เมื่อไม่ได้ตั้งค่าไว้ สรุปตอนเริ่มต้นจะแสดง `medium` `fast` มาจาก
เอเจนต์เริ่มต้นหรือพารามิเตอร์ `fastMode` ของโมเดล

## ตัวบันทึกล็อกแบบไฟล์

- ไฟล์ล็อกแบบ rolling เริ่มต้นอยู่ใต้ `/tmp/openclaw/` (หนึ่งไฟล์ต่อวัน): `openclaw-YYYY-MM-DD.log`
  - วันที่ใช้เขตเวลาท้องถิ่นของโฮสต์ Gateway
- ไฟล์ล็อกที่ใช้งานอยู่จะหมุนเวียนเมื่อถึง `logging.maxFileBytes` (ค่าเริ่มต้น: 100 MB) โดยเก็บ
  ไฟล์เก็บถาวรแบบมีหมายเลขไว้สูงสุดห้าไฟล์ และเขียนต่อไปยังไฟล์ที่ใช้งานอยู่ไฟล์ใหม่
- เส้นทางและระดับของไฟล์ล็อกสามารถกำหนดค่าได้ผ่าน `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

รูปแบบไฟล์คือหนึ่งออบเจ็กต์ JSON ต่อหนึ่งบรรทัด

แท็บ Logs ใน Control UI tail ไฟล์นี้ผ่าน Gateway (`logs.tail`)
CLI ก็ทำแบบเดียวกันได้:

```bash
openclaw logs --follow
```

**Verbose เทียบกับระดับล็อก**

- **ล็อกไฟล์** ถูกควบคุมโดย `logging.level` เท่านั้น
- `--verbose` มีผลเฉพาะกับ **ความละเอียดของคอนโซล** (และรูปแบบล็อก WS) โดย **ไม่**
  เพิ่มระดับล็อกของไฟล์
- หากต้องการเก็บรายละเอียดที่มีเฉพาะใน verbose ลงในล็อกไฟล์ ให้ตั้ง `logging.level` เป็น `debug` หรือ
  `trace`
- การบันทึกล็อกระดับ trace ยังรวมสรุปเวลาเชิงวินิจฉัยสำหรับ hot paths บางรายการ
  เช่น การเตรียม factory ของเครื่องมือ Plugin ดู
  [/tools/plugin#slow-plugin-tool-setup](/th/tools/plugin#slow-plugin-tool-setup)

## การจับคอนโซล

CLI จับ `console.log/info/warn/error/debug/trace` และเขียนลงในล็อกไฟล์
ขณะที่ยังพิมพ์ไปยัง stdout/stderr ตามปกติ

คุณสามารถปรับความละเอียดของคอนโซลได้อย่างอิสระผ่าน:

- `logging.consoleLevel` (ค่าเริ่มต้น `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## การปกปิดข้อมูล

OpenClaw สามารถปกปิดโทเค็นที่อ่อนไหวก่อนที่เอาต์พุตล็อกหรือทรานสคริปต์จะออกจาก
โปรเซส นโยบายการปกปิดข้อมูลสำหรับล็อกนี้ถูกใช้กับ sink ของข้อความสำหรับคอนโซล, ไฟล์ล็อก, ระเบียนล็อก OTLP,
และทรานสคริปต์เซสชัน ดังนั้นค่าลับที่ตรงกันจะถูก
ปกปิดก่อนที่บรรทัด JSONL หรือข้อความจะถูกเขียนลงดิสก์

- `logging.redactSensitive`: `off` | `tools` (ค่าเริ่มต้น: `tools`)
- `logging.redactPatterns`: อาร์เรย์ของสตริง regex (แทนที่ค่าเริ่มต้น)
  - ใช้สตริง regex ดิบ (เพิ่ม `gi` อัตโนมัติ) หรือ `/pattern/flags` หากต้องการ flags แบบกำหนดเอง
  - รายการที่ตรงกันจะถูกปกปิดโดยเก็บอักขระ 6 ตัวแรก + 4 ตัวสุดท้าย (ความยาว >= 18) มิฉะนั้นใช้ `***`
  - ค่าเริ่มต้นครอบคลุมการกำหนด key ทั่วไป, CLI flags, ฟิลด์ JSON, bearer headers, บล็อก PEM, prefix โทเค็นยอดนิยม และชื่อฟิลด์ข้อมูลรับรองการชำระเงิน เช่น หมายเลขบัตร, CVC/CVV, โทเค็นการชำระเงินที่ใช้ร่วมกัน และข้อมูลรับรองการชำระเงิน

ขอบเขตความปลอดภัยบางส่วนจะปกปิดข้อมูลเสมอ ไม่ว่า `logging.redactSensitive` จะเป็นค่าใดก็ตาม
ซึ่งรวมถึงเหตุการณ์การเรียกเครื่องมือใน Control UI, เอาต์พุตเครื่องมือ `sessions_history`,
การส่งออกข้อมูลสนับสนุนการวินิจฉัย, การสังเกตข้อผิดพลาดจากผู้ให้บริการ, การแสดงคำสั่งอนุมัติ exec
และล็อกโปรโตคอล WebSocket ของ Gateway พื้นผิวเหล่านี้ยังอาจใช้
`logging.redactPatterns` เป็นรูปแบบเพิ่มเติมได้ แต่ `redactSensitive: "off"`
จะไม่ทำให้พื้นผิวเหล่านี้ปล่อยความลับแบบดิบออกมา

## ล็อก WebSocket ของ Gateway

Gateway พิมพ์ล็อกโปรโตคอล WebSocket ในสองโหมด:

- **โหมดปกติ (ไม่มี `--verbose`)**: พิมพ์เฉพาะผลลัพธ์ RPC ที่ “น่าสนใจ”:
  - ข้อผิดพลาด (`ok=false`)
  - การเรียกที่ช้า (เกณฑ์เริ่มต้น: `>= 50ms`)
  - ข้อผิดพลาดการ parse
- **โหมด Verbose (`--verbose`)**: พิมพ์ทราฟฟิกคำขอ/คำตอบ WS ทั้งหมด

### รูปแบบล็อก WS

`openclaw gateway` รองรับสวิตช์รูปแบบต่อ Gateway:

- `--ws-log auto` (ค่าเริ่มต้น): โหมดปกติถูกปรับให้เหมาะสม; โหมด verbose ใช้เอาต์พุตแบบ compact
- `--ws-log compact`: เอาต์พุตแบบ compact (จับคู่คำขอ/คำตอบ) เมื่อ verbose
- `--ws-log full`: เอาต์พุตต่อเฟรมแบบเต็มเมื่อ verbose
- `--compact`: alias สำหรับ `--ws-log compact`

ตัวอย่าง:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## การจัดรูปแบบคอนโซล (การบันทึกล็อกระบบย่อย)

ตัวจัดรูปแบบคอนโซล **รับรู้ TTY** และพิมพ์บรรทัดที่สอดคล้องกันพร้อม prefix
ตัวบันทึกล็อกของระบบย่อยทำให้เอาต์พุตถูกจัดกลุ่มและสแกนอ่านได้ง่าย

พฤติกรรม:

- **Prefix ของระบบย่อย** ในทุกบรรทัด (เช่น `[gateway]`, `[canvas]`, `[tailscale]`)
- **สีของระบบย่อย** (คงที่ต่อระบบย่อย) พร้อมการใส่สีตามระดับ
- **สีเมื่อเอาต์พุตเป็น TTY หรือสภาพแวดล้อมดูเหมือนเทอร์มินัลแบบสมบูรณ์** (`TERM`/`COLORTERM`/`TERM_PROGRAM`) และเคารพ `NO_COLOR`
- **Prefix ของระบบย่อยแบบย่อ**: ตัด `gateway/` + `channels/` ด้านหน้าออก, เก็บ 2 segment สุดท้ายไว้ (เช่น `whatsapp/outbound`)
- **ตัวบันทึกล็อกย่อยตามระบบย่อย** (prefix อัตโนมัติ + ฟิลด์มีโครงสร้าง `{ subsystem }`)
- **`logRaw()`** สำหรับเอาต์พุต QR/UX (ไม่มี prefix, ไม่มีการจัดรูปแบบ)
- **รูปแบบคอนโซล** (เช่น `pretty | compact | json`)
- **ระดับล็อกคอนโซล** แยกจากระดับล็อกไฟล์ (ไฟล์ยังเก็บรายละเอียดเต็มเมื่อ `logging.level` ถูกตั้งเป็น `debug`/`trace`)
- **เนื้อหาข้อความ WhatsApp** ถูกบันทึกที่ `debug` (ใช้ `--verbose` เพื่อดู)

สิ่งนี้ทำให้ล็อกไฟล์เดิมยังคงเสถียร ขณะทำให้เอาต์พุตเชิงโต้ตอบสแกนอ่านได้ง่าย
