# การบันทึกล็อก

สำหรับภาพรวมที่ผู้ใช้เห็น (CLI + Control UI + การกำหนดค่า) ดู [/logging](/th/logging)

OpenClaw มี “พื้นผิว” การบันทึกล็อกสองแบบ:

- **เอาต์พุตคอนโซล** (สิ่งที่คุณเห็นในเทอร์มินัล / Debug UI)
- **ไฟล์ล็อก** (บรรทัด JSON) ที่เขียนโดยตัวบันทึกล็อกของ Gateway

## ตัวบันทึกล็อกแบบไฟล์

- ไฟล์ล็อกแบบหมุนเวียนเริ่มต้นอยู่ภายใต้ `/tmp/openclaw/` (หนึ่งไฟล์ต่อวัน): `openclaw-YYYY-MM-DD.log`
  - วันที่ใช้เขตเวลาท้องถิ่นของโฮสต์ Gateway
- ไฟล์ล็อกที่ใช้งานอยู่จะหมุนเวียนเมื่อถึง `logging.maxFileBytes` (ค่าเริ่มต้น: 100 MB) โดยเก็บ
  ไฟล์เก็บถาวรแบบมีหมายเลขไว้สูงสุดห้าไฟล์และเขียนไฟล์ที่ใช้งานอยู่ไฟล์ใหม่ต่อไป
- เส้นทางไฟล์ล็อกและระดับสามารถกำหนดค่าได้ผ่าน `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

รูปแบบไฟล์คืออ็อบเจ็กต์ JSON หนึ่งรายการต่อหนึ่งบรรทัด

แท็บ Logs ใน Control UI ติดตามไฟล์นี้ผ่าน Gateway (`logs.tail`)
CLI สามารถทำแบบเดียวกันได้:

```bash
openclaw logs --follow
```

**Verbose เทียบกับระดับล็อก**

- **ไฟล์ล็อก** ถูกควบคุมโดย `logging.level` เท่านั้น
- `--verbose` มีผลเฉพาะกับ **ความละเอียดของคอนโซล** (และรูปแบบล็อก WS) เท่านั้น โดย **ไม่**
  เพิ่มระดับไฟล์ล็อก
- หากต้องการบันทึกรายละเอียดที่มีเฉพาะในโหมด verbose ลงในไฟล์ล็อก ให้ตั้ง `logging.level` เป็น `debug` หรือ
  `trace`

## การจับคอนโซล

CLI จับ `console.log/info/warn/error/debug/trace` และเขียนลงในไฟล์ล็อก
ขณะเดียวกันยังพิมพ์ไปยัง stdout/stderr

คุณสามารถปรับความละเอียดของคอนโซลแยกต่างหากได้ผ่าน:

- `logging.consoleLevel` (ค่าเริ่มต้น `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## การปกปิด

OpenClaw สามารถปกปิดโทเค็นที่ละเอียดอ่อนได้ก่อนที่เอาต์พุตล็อกหรือทรานสคริปต์จะออกจาก
โปรเซส นโยบายการปกปิดการบันทึกล็อกนี้ถูกใช้กับปลายทางข้อความของคอนโซล ไฟล์ล็อก ระเบียนล็อก OTLP
และทรานสคริปต์เซสชัน ดังนั้นค่าความลับที่ตรงกันจะถูก
ปกปิดก่อนที่บรรทัด JSONL หรือข้อความจะถูกเขียนลงดิสก์

- `logging.redactSensitive`: `off` | `tools` (ค่าเริ่มต้น: `tools`)
- `logging.redactPatterns`: อาร์เรย์ของสตริง regex (แทนที่ค่าเริ่มต้น)
  - ใช้สตริง regex ดิบ (เพิ่ม `gi` อัตโนมัติ) หรือ `/pattern/flags` หากคุณต้องการแฟล็กแบบกำหนดเอง
  - รายการที่ตรงกันจะถูกปกปิดโดยคง 6 อักขระแรก + 4 อักขระสุดท้ายไว้ (ความยาว >= 18) มิฉะนั้นใช้ `***`
  - ค่าเริ่มต้นครอบคลุมการกำหนดค่าคีย์ทั่วไป แฟล็ก CLI ฟิลด์ JSON ส่วนหัว bearer บล็อก PEM พรีฟิกซ์โทเค็นยอดนิยม และชื่อฟิลด์ข้อมูลประจำตัวการชำระเงิน เช่น หมายเลขบัตร CVC/CVV โทเค็นการชำระเงินที่ใช้ร่วมกัน และข้อมูลประจำตัวการชำระเงิน

ขอบเขตความปลอดภัยบางส่วนจะปกปิดเสมอไม่ว่า `logging.redactSensitive` จะเป็นค่าใด
ซึ่งรวมถึงเหตุการณ์การเรียกเครื่องมือของ Control UI, เอาต์พุตเครื่องมือ `sessions_history`,
การส่งออกการสนับสนุนด้านวินิจฉัย, การสังเกตข้อผิดพลาดของผู้ให้บริการ, การแสดงคำสั่งอนุมัติ exec
และล็อกโปรโตคอล WebSocket ของ Gateway พื้นผิวเหล่านี้ยังอาจใช้
`logging.redactPatterns` เป็นแพตเทิร์นเพิ่มเติมได้ แต่ `redactSensitive: "off"`
จะไม่ทำให้ส่งออกความลับแบบดิบ

## ล็อก WebSocket ของ Gateway

Gateway พิมพ์ล็อกโปรโตคอล WebSocket ในสองโหมด:

- **โหมดปกติ (ไม่มี `--verbose`)**: พิมพ์เฉพาะผลลัพธ์ RPC ที่ “น่าสนใจ”:
  - ข้อผิดพลาด (`ok=false`)
  - การเรียกที่ช้า (เกณฑ์เริ่มต้น: `>= 50ms`)
  - ข้อผิดพลาดในการแยกวิเคราะห์
- **โหมด Verbose (`--verbose`)**: พิมพ์ทราฟฟิกคำขอ/คำตอบ WS ทั้งหมด

### รูปแบบล็อก WS

`openclaw gateway` รองรับสวิตช์รูปแบบต่อ Gateway:

- `--ws-log auto` (ค่าเริ่มต้น): โหมดปกติได้รับการปรับให้เหมาะสม โหมด verbose ใช้เอาต์พุตแบบ compact
- `--ws-log compact`: เอาต์พุตแบบ compact (จับคู่คำขอ/คำตอบ) เมื่อ verbose
- `--ws-log full`: เอาต์พุตเต็มต่อเฟรมเมื่อ verbose
- `--compact`: นามแฝงของ `--ws-log compact`

ตัวอย่าง:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## การจัดรูปแบบคอนโซล (การบันทึกล็อกของระบบย่อย)

ตัวจัดรูปแบบคอนโซล **รับรู้ TTY** และพิมพ์บรรทัดที่สอดคล้องกันพร้อมพรีฟิกซ์
ตัวบันทึกล็อกของระบบย่อยช่วยให้เอาต์พุตถูกจัดกลุ่มและสแกนอ่านได้ง่าย

พฤติกรรม:

- **พรีฟิกซ์ระบบย่อย** ในทุกบรรทัด (เช่น `[gateway]`, `[canvas]`, `[tailscale]`)
- **สีของระบบย่อย** (คงที่ต่อระบบย่อย) รวมกับการลงสีตามระดับ
- **ใช้สีเมื่อเอาต์พุตเป็น TTY หรือสภาพแวดล้อมดูเหมือนเทอร์มินัลที่มีความสามารถสูง** (`TERM`/`COLORTERM`/`TERM_PROGRAM`) และเคารพ `NO_COLOR`
- **พรีฟิกซ์ระบบย่อแบบสั้น**: ตัด `gateway/` + `channels/` ด้านหน้าออก เก็บ 2 ส่วนสุดท้ายไว้ (เช่น `whatsapp/outbound`)
- **ตัวบันทึกล็อกย่อยตามระบบย่อย** (พรีฟิกซ์อัตโนมัติ + ฟิลด์แบบมีโครงสร้าง `{ subsystem }`)
- **`logRaw()`** สำหรับเอาต์พุต QR/UX (ไม่มีพรีฟิกซ์ ไม่มีการจัดรูปแบบ)
- **รูปแบบคอนโซล** (เช่น `pretty | compact | json`)
- **ระดับล็อกคอนโซล** แยกจากระดับไฟล์ล็อก (ไฟล์เก็บรายละเอียดทั้งหมดเมื่อ `logging.level` ถูกตั้งเป็น `debug`/`trace`)
- **เนื้อหาข้อความ WhatsApp** ถูกบันทึกที่ `debug` (ใช้ `--verbose` เพื่อดู)

สิ่งนี้ทำให้ไฟล์ล็อกที่มีอยู่ยังคงเสถียร พร้อมทำให้เอาต์พุตแบบโต้ตอบสแกนอ่านได้ง่าย
