OpenClaw รองรับ SecretRef แบบเพิ่มเติม เพื่อให้ข้อมูลประจำตัวที่รองรับไม่จำเป็นต้องจัดเก็บเป็นข้อความธรรมดาในการกำหนดค่า

<Note>
ยังคงใช้ข้อความธรรมดาได้ SecretRef เป็นตัวเลือกที่ต้องเปิดใช้แยกสำหรับข้อมูลประจำตัวแต่ละรายการ
</Note>

<Warning>
ข้อมูลประจำตัวแบบข้อความธรรมดายังคงอ่านได้โดยเอเจนต์ หากอยู่ในไฟล์ที่เอเจนต์ตรวจสอบได้ รวมถึง `openclaw.json`, `auth-profiles.json`, `.env` หรือไฟล์ `agents/*/agent/models.json` ที่สร้างขึ้น SecretRef จะลดขอบเขตผลกระทบเฉพาะที่นี้ได้ต่อเมื่อย้ายข้อมูลประจำตัวที่รองรับทั้งหมดแล้ว และ `openclaw secrets audit --check` รายงานว่าไม่มีข้อความธรรมดาหลงเหลืออยู่
</Warning>

## โมเดลรันไทม์

- ข้อมูลลับจะถูกแก้ค่าเป็นสแนปช็อตรันไทม์ในหน่วยความจำล่วงหน้าระหว่างการเปิดใช้งาน ไม่ใช่แบบหน่วงเวลาบนเส้นทางคำขอ
- การเริ่มต้นจะล้มเหลวทันทีเมื่อไม่สามารถแก้ค่า SecretRef ที่มีผลใช้งานจริงได้
- การโหลดซ้ำเป็นการสลับแบบอะตอม: ต้องสำเร็จทั้งหมด มิฉะนั้นจะคงสแนปช็อตล่าสุดที่ทราบว่าใช้งานได้ไว้
- การละเมิดนโยบาย (เช่น โปรไฟล์การยืนยันตัวตนโหมด OAuth ที่ใช้ร่วมกับอินพุต SecretRef) จะทำให้การเปิดใช้งานล้มเหลวก่อนสลับรันไทม์
- คำขอรันไทม์จะอ่านเฉพาะสแนปช็อตในหน่วยความจำที่ใช้งานอยู่ ข้อมูลประจำตัว SecretRef ของผู้ให้บริการโมเดลจะส่งผ่านพื้นที่จัดเก็บการยืนยันตัวตนและตัวเลือกสตรีมในรูปค่าตัวแทนเฉพาะโปรเซสจนถึงจุดส่งออก เส้นทางการส่งขาออก (การส่งการตอบกลับ/เธรดของ Discord และการส่งการดำเนินการของ Telegram) จะอ่านสแนปช็อตนั้นเช่นกัน และไม่แก้ค่าอ้างอิงใหม่ทุกครั้งที่ส่ง

วิธีนี้ช่วยไม่ให้การขัดข้องของผู้ให้บริการข้อมูลลับกระทบเส้นทางคำขอที่ใช้งานหนัก

## การแทรกค่า ณ เวลาส่งออก (ค่าตัวแทน)

สำหรับข้อมูลประจำตัวของผู้ให้บริการโมเดลที่อ้างอิงด้วย SecretRef นั้น OpenClaw จะสร้างค่าตัวแทนทึบแสงเฉพาะโปรเซสระหว่างการแก้ค่าการยืนยันตัวตนของโมเดล ดังนั้นพื้นที่จัดเก็บการยืนยันตัวตน ตัวเลือกสตรีม การกำหนดค่า SDK บันทึก ออบเจ็กต์ข้อผิดพลาด และการตรวจสอบรันไทม์ส่วนใหญ่จะเห็นค่าอย่างเช่น `oc-sent-v1-...` แทนข้อมูลประจำตัวของผู้ให้บริการ การดึงข้อมูลโมเดลที่มีการป้องกันและโพรบตรวจสอบสถานะของผู้ให้บริการภายในที่ได้รับการจัดการ จะแทนที่ค่าตัวแทนที่รู้จักในค่า URL และส่วนหัวทันทีก่อนที่แต่ละคำขอจะออกจากโปรเซส

ค่าที่มีรูปแบบเหมือนค่าตัวแทนแต่ไม่เป็นที่รู้จักจะล้มเหลวแบบปิดก่อนมีกิจกรรมเครือข่าย OpenClaw จะปฏิเสธการส่งคำขอแทนที่จะส่งต่อค่าตัวแทนที่ยังไม่ได้แก้ค่าไปยังผู้ให้บริการ ค่าข้อมูลลับที่แก้แล้วจะถูกลงทะเบียนเพื่อปกปิดค่าที่ตรงกันทุกประการในบันทึกด้วย เพื่อเป็นมาตรการป้องกันเชิงลึก

อะแดปเตอร์ของผู้ให้บริการใช้จุดแทรกค่าที่ท้ายสุดซึ่ง SDK รองรับ:

- SDK ที่มีตัวเลือก fetch แบบกำหนดเองจะได้รับ fetch ที่มีการป้องกันของ OpenClaw เพื่อให้ SDK คงค่าตัวแทนไว้
- SDK ที่ไม่มีตัวเลือก fetch แบบกำหนดเองจะแกะค่าตัวแทนทันทีก่อนสร้างไคลเอนต์ สตรีมของผู้ให้บริการที่ Plugin เป็นเจ้าของและชุดควบคุมเอเจนต์จะแกะค่าที่จุดส่งมอบสุดท้ายซึ่งคอร์เป็นเจ้าของ เนื่องจากการรับส่งข้อมูลเหล่านั้นไม่ได้ใช้ fetch ที่มีการป้องกันของ OpenClaw ร่วมกัน

ค่าตัวแทนช่วยลดการเปิดเผยข้อความธรรมดาตลอดสายโซ่การเรียกโมเดล แต่ไม่ใช่การแยกโปรเซส ค่าจริงยังคงอยู่ในหน่วยความจำของโปรเซสเดียวกัน และปรากฏที่ขอบเขตอะแดปเตอร์สุดท้าย ข้อมูลประจำตัวจากสภาพแวดล้อมแบบข้อความธรรมดาที่ไม่ได้กำหนดค่าผ่าน SecretRef จะยังคงเป็นข้อความธรรมดาและอยู่นอกกลไกนี้

ตั้งค่า `OPENCLAW_SECRET_SENTINELS=off` (ยอมรับ `0` หรือ `false` ด้วย โดยไม่คำนึงถึงตัวพิมพ์เล็ก-ใหญ่) เพื่อปิดการสร้างค่าตัวแทนระหว่างการตอบสนองต่อเหตุการณ์หรือการแก้ปัญหาความเข้ากันได้ สวิตช์ฉุกเฉินนี้จะไม่ปิดการลงทะเบียนการปกปิดค่าที่ตรงกันทุกประการ

## ขอบเขตการเข้าถึงของเอเจนต์

SecretRef ป้องกันไม่ให้ข้อมูลประจำตัวถูกจัดเก็บถาวรในการกำหนดค่าและไฟล์โมเดลที่สร้างขึ้น แต่ไม่ใช่ขอบเขตการแยกโปรเซส ข้อมูลประจำตัวแบบข้อความธรรมดาที่เหลืออยู่บนดิสก์ในเส้นทางที่เอเจนต์อ่านได้ ยังคงอ่านได้ผ่านเครื่องมือไฟล์หรือเชลล์ ซึ่งข้ามการปกปิดระดับ API

สำหรับการติดตั้งใช้งานจริงที่รวมไฟล์ซึ่งเอเจนต์เข้าถึงได้ไว้ในขอบเขต ให้ถือว่าการย้ายเสร็จสมบูรณ์ก็ต่อเมื่อเป็นจริงทุกข้อต่อไปนี้:

- ข้อมูลประจำตัวที่รองรับใช้ SecretRef แทนค่าแบบข้อความธรรมดา
- กำจัดข้อความธรรมดาเดิมที่หลงเหลืออยู่ออกจาก `openclaw.json`, `auth-profiles.json`, `.env` และไฟล์ `models.json` ที่สร้างขึ้นแล้ว
- `openclaw secrets audit --check` ไม่มีรายการตกค้างหลังการย้าย
- ข้อมูลประจำตัวที่ยังไม่รองรับหรือมีการหมุนเวียนรายการอื่น ๆ ได้รับการป้องกันด้วยการแยกระดับระบบปฏิบัติการ การแยกคอนเทนเนอร์ หรือพร็อกซีข้อมูลประจำตัวภายนอก

นี่คือเหตุผลที่เวิร์กโฟลว์ตรวจสอบ/กำหนดค่า/นำไปใช้เป็นด่านการย้ายด้านความปลอดภัย ไม่ใช่เพียงตัวช่วยอำนวยความสะดวก

<Warning>
SecretRef ไม่ได้ทำให้ไฟล์ใด ๆ ที่อ่านได้มีความปลอดภัย ข้อมูลสำรอง สำเนาการกำหนดค่า แค็ตตาล็อกโมเดลเก่าที่สร้างขึ้น และประเภทข้อมูลประจำตัวที่ยังไม่รองรับ ยังคงเป็นข้อมูลลับของระบบจริงจนกว่าจะถูกลบ ย้ายออกนอกขอบเขตความไว้วางใจของเอเจนต์ หรือแยกออกต่างหาก
</Warning>

## การกรองพื้นผิวที่ใช้งานอยู่

SecretRef จะได้รับการตรวจสอบเฉพาะบนพื้นผิวที่มีผลใช้งานจริง:

- **พื้นผิวที่เปิดใช้งาน**: ค่าอ้างอิงที่แก้ไม่ได้จะขัดขวางการเริ่มต้น/โหลดซ้ำ
- **พื้นผิวที่ไม่ใช้งาน**: ค่าอ้างอิงที่แก้ไม่ได้จะไม่ขัดขวางการเริ่มต้น/โหลดซ้ำ แต่จะแสดงการวินิจฉัย `SECRETS_REF_IGNORED_INACTIVE_SURFACE` ที่ไม่ทำให้ระบบล้มเหลว

<Accordion title="ตัวอย่างพื้นผิวที่ไม่ใช้งาน">
- รายการช่องทาง/บัญชีที่ปิดใช้งาน
- ข้อมูลประจำตัวช่องทางระดับบนสุดที่ไม่มีบัญชีที่เปิดใช้งานสืบทอดไปใช้
- พื้นผิวเครื่องมือ/ฟีเจอร์ที่ปิดใช้งาน
- คีย์เฉพาะผู้ให้บริการค้นหาเว็บที่ไม่ได้เลือกโดย `tools.web.search.provider` ในโหมดอัตโนมัติ (ไม่ได้ตั้งค่าผู้ให้บริการ) ระบบจะพิจารณาคีย์ตามลำดับความสำคัญเพื่อตรวจหาอัตโนมัติจนกว่าจะมีคีย์หนึ่งที่แก้ค่าได้ หลังจากเลือกแล้ว คีย์ของผู้ให้บริการที่ไม่ได้เลือกจะไม่ทำงาน
- ข้อมูลการยืนยันตัวตน SSH ของแซนด์บ็อกซ์ (`agents.defaults.sandbox.ssh.identityData`, `certificateData`, `knownHostsData` รวมถึงค่าที่แทนที่แยกตามเอเจนต์) จะทำงานเฉพาะเมื่อแบ็กเอนด์แซนด์บ็อกซ์ที่มีผลเป็น `ssh` และโหมดแซนด์บ็อกซ์ไม่ใช่ `off` สำหรับเอเจนต์เริ่มต้นหรือเอเจนต์ที่เปิดใช้งาน
- SecretRef ของ `gateway.remote.token` / `gateway.remote.password` จะทำงานหากข้อใดข้อหนึ่งต่อไปนี้เป็นจริง:
  - `gateway.mode=remote`
  - มีการกำหนดค่า `gateway.remote.url`
  - `gateway.tailscale.mode` เป็น `serve` หรือ `funnel`
  - ในโหมดภายในที่ไม่มีพื้นผิวระยะไกลเหล่านั้น: `gateway.remote.token` จะทำงานเมื่อการยืนยันตัวตนด้วยโทเค็นมีสิทธิ์เหนือกว่าและไม่ได้กำหนดค่าโทเค็นจากสภาพแวดล้อม/การยืนยันตัวตน ส่วน `gateway.remote.password` จะทำงานเฉพาะเมื่อการยืนยันตัวตนด้วยรหัสผ่านมีสิทธิ์เหนือกว่าและไม่ได้กำหนดค่ารหัสผ่านจากสภาพแวดล้อม/การยืนยันตัวตน
- SecretRef ของ `gateway.auth.token` จะไม่ทำงานสำหรับการแก้ค่าการยืนยันตัวตนเมื่อเริ่มต้น หากตั้งค่า `OPENCLAW_GATEWAY_TOKEN` เนื่องจากอินพุตโทเค็นจากสภาพแวดล้อมมีสิทธิ์เหนือกว่าสำหรับรันไทม์นั้น

</Accordion>

## การวินิจฉัยพื้นผิวการยืนยันตัวตนของ Gateway

เมื่อตั้งค่า SecretRef บน `gateway.auth.token`, `gateway.auth.password`, `gateway.remote.token` หรือ `gateway.remote.password` การเริ่มต้น/โหลดซ้ำของ Gateway จะบันทึกสถานะพื้นผิวภายใต้รหัส `SECRETS_GATEWAY_AUTH_SURFACE`:

- `active`: SecretRef เป็นส่วนหนึ่งของพื้นผิวการยืนยันตัวตนที่มีผล และต้องแก้ค่าได้
- `inactive`: พื้นผิวการยืนยันตัวตนอื่นมีสิทธิ์เหนือกว่า หรือการยืนยันตัวตนระยะไกลถูกปิดใช้งาน/ไม่ได้ทำงาน

รายการบันทึกจะมีเหตุผลที่นโยบายพื้นผิวที่ใช้งานเลือกใช้

## การตรวจสอบล่วงหน้าสำหรับข้อมูลอ้างอิงระหว่างการเริ่มต้นใช้งาน

ในการเริ่มต้นใช้งานแบบโต้ตอบ การเลือกจัดเก็บด้วย SecretRef จะเรียกใช้การตรวจสอบล่วงหน้าก่อนบันทึก:

- ค่าอ้างอิง Env: ตรวจสอบชื่อของตัวแปรสภาพแวดล้อมและยืนยันว่ามองเห็นค่าที่ไม่ว่างระหว่างการตั้งค่า
- ค่าอ้างอิงผู้ให้บริการ (`file` หรือ `exec`): ตรวจสอบการเลือกผู้ให้บริการ แก้ค่า `id` และตรวจสอบชนิดของค่าที่แก้แล้ว
- โฟลว์เริ่มต้นอย่างรวดเร็ว: เมื่อ `gateway.auth.token` เป็น SecretRef อยู่แล้ว การเริ่มต้นใช้งานจะแก้ค่าก่อนเริ่มต้นโพรบ/แดชบอร์ด (สำหรับค่าอ้างอิง `env`, `file` และ `exec`) โดยใช้ด่านล้มเหลวทันทีเดียวกัน

หากการตรวจสอบล้มเหลว ระบบจะแสดงข้อผิดพลาดและให้ลองใหม่ได้

## สัญญา SecretRef

ใช้ออบเจ็กต์รูปแบบเดียวกันทุกแห่ง:

```json5
{ source: "env" | "file" | "exec", provider: "default", id: "..." }
```

<Tabs>
  <Tab title="env">
    ```json5
    { source: "env", provider: "default", id: "OPENAI_API_KEY" }
    ```

    ฟิลด์ SecretInput ยอมรับสตริงแบบย่อด้วย:

    ```json5
    "${OPENAI_API_KEY}"
    "$OPENAI_API_KEY"
    ```

    การตรวจสอบ:

    - `provider` ต้องตรงกับ `^[a-z][a-z0-9_-]{0,63}$`
    - `id` ต้องตรงกับ `^[A-Z][A-Z0-9_]{0,127}$`

  </Tab>
  <Tab title="file">
    ```json5
    { source: "file", provider: "filemain", id: "/providers/openai/apiKey" }
    ```

    การตรวจสอบ:

    - `provider` ต้องตรงกับ `^[a-z][a-z0-9_-]{0,63}$`
    - `id` ต้องเป็นตัวชี้ JSON แบบสัมบูรณ์ (`/...`) หรือเป็นค่าตรงตัว `value` สำหรับผู้ให้บริการ `singleValue`
    - การหลีกอักขระตาม RFC 6901 ในแต่ละส่วน: `~` จะกลายเป็น `~0` และ `/` จะกลายเป็น `~1`

  </Tab>
  <Tab title="exec">
    ```json5
    { source: "exec", provider: "vault", id: "providers/openai/apiKey#value" }
    ```

    การตรวจสอบ:

    - `provider` ต้องตรงกับ `^[a-z][a-z0-9_-]{0,63}$`
    - `id` ต้องตรงกับ `^[A-Za-z0-9][A-Za-z0-9._:/#-]{0,255}$` (รองรับตัวเลือก เช่น `secret#json_key`)
    - `id` ต้องไม่มี `.` หรือ `..` เป็นส่วนของเส้นทางที่คั่นด้วยเครื่องหมายทับ (เช่น `a/../b` จะถูกปฏิเสธ)

  </Tab>
</Tabs>

## การกำหนดค่าผู้ให้บริการ

กำหนดผู้ให้บริการภายใต้ `secrets.providers`:

```json5
{
  secrets: {
    providers: {
      default: { source: "env" },
      filemain: {
        source: "file",
        path: "~/.openclaw/secrets.json",
        mode: "json", // or "singleValue"
      },
      vault: {
        source: "exec",
        command: "/usr/local/bin/openclaw-vault-resolver",
        args: ["--profile", "prod"],
        passEnv: ["PATH", "VAULT_ADDR"],
        jsonOnly: true,
      },
      "team-secrets": {
        source: "exec",
        pluginIntegration: {
          pluginId: "acme-secrets",
          integrationId: "secret-store",
        },
      },
    },
    defaults: {
      env: "default",
      file: "filemain",
      exec: "vault",
    },
    resolution: {
      maxProviderConcurrency: 4,
      maxRefsPerProvider: 512,
      maxBatchBytes: 262144,
    },
  },
}
```

<Accordion title="ผู้ให้บริการ Env">
- รายการอนุญาตแบบระบุชื่อตรงกันทุกประการซึ่งเลือกใช้ได้ผ่าน `allowlist`
- ค่า Env ที่ไม่มีหรือว่างจะทำให้การแก้ค่าล้มเหลว

</Accordion>

<Accordion title="ผู้ให้บริการไฟล์">
- อ่านไฟล์ภายในที่ `path`
- `mode: "json"` (ค่าเริ่มต้น) ต้องการเพย์โหลดออบเจ็กต์ JSON และแก้ค่า `id` เป็นตัวชี้ JSON
- `mode: "singleValue"` ต้องการรหัสอ้างอิง `"value"` และส่งคืนเนื้อหาไฟล์ดิบ (ตัดอักขระขึ้นบรรทัดใหม่ที่ท้ายออก)
- เส้นทางต้องผ่านการตรวจสอบความเป็นเจ้าของ/สิทธิ์ โดย `timeoutMs` (ค่าเริ่มต้น 5000) และ `maxBytes` (ค่าเริ่มต้น 1 MiB) จำกัดการอ่าน
- Windows ใช้การล้มเหลวแบบปิด: หากตรวจสอบ ACL ของเส้นทางไม่ได้ การแก้ค่าจะล้มเหลว สำหรับเส้นทางที่เชื่อถือได้เท่านั้น ให้ตั้งค่า `allowInsecurePath: true` บนผู้ให้บริการนั้นเพื่อข้ามการตรวจสอบ

</Accordion>

<Accordion title="ผู้ให้บริการ Exec">
- เรียกใช้พาธสัมบูรณ์ของไบนารีที่กำหนดค่าไว้โดยตรงโดยไม่ผ่านเชลล์
- โดยค่าเริ่มต้น `command` ต้องเป็นไฟล์ปกติ ไม่ใช่ลิงก์สัญลักษณ์ ตั้งค่า `allowSymlinkCommand: true` เพื่ออนุญาตพาธคำสั่งที่เป็นลิงก์สัญลักษณ์ (เช่น ชิมของ Homebrew) และใช้ร่วมกับ `trustedDirs` (เช่น `["/opt/homebrew"]`) เพื่อให้เฉพาะพาธของตัวจัดการแพ็กเกจเท่านั้นที่ผ่านเกณฑ์
- รองรับ `timeoutMs` (ค่าเริ่มต้น 5000), `noOutputTimeoutMs` (ค่าเริ่มต้นเท่ากับ `timeoutMs`), `maxOutputBytes` (ค่าเริ่มต้น 1 MiB), รายการอนุญาต `env`/`passEnv` และ `trustedDirs`
- `jsonOnly` มีค่าเริ่มต้นเป็น `true` เมื่อใช้ `jsonOnly: false` และร้องขอเพียง id เดียว ระบบจะยอมรับ stdout แบบข้อความธรรมดาที่ไม่ใช่ JSON เป็นค่าของ id นั้น
- Windows ใช้หลักปิดเมื่อยืนยันไม่ได้: หากไม่สามารถตรวจสอบ ACL ของพาธคำสั่งได้ การแก้ค่าจะล้มเหลว สำหรับพาธที่เชื่อถือได้เท่านั้น ให้ตั้งค่า `allowInsecurePath: true` ในผู้ให้บริการนั้นเพื่อข้ามการตรวจสอบ
- ผู้ให้บริการ exec ที่จัดการโดย Plugin สามารถใช้ `pluginIntegration` แทน `command`/`args` ที่คัดลอกมา OpenClaw จะแก้รายละเอียดคำสั่งปัจจุบันจากไฟล์ประกาศของ Plugin ที่ติดตั้งไว้ระหว่างการเริ่มต้น/โหลดใหม่ หาก Plugin ถูกปิดใช้งาน ถูกลบ ไม่น่าเชื่อถือ หรือไม่ได้ประกาศการผสานรวมอีกต่อไป SecretRef ที่ใช้งานอยู่ในผู้ให้บริการนั้นจะล้มเหลวแบบปิด

เพย์โหลดคำขอ (stdin):

```json
{ "protocolVersion": 1, "provider": "vault", "ids": ["providers/openai/apiKey"] }
```

เพย์โหลดการตอบกลับ (stdout):

```jsonc
{ "protocolVersion": 1, "values": { "providers/openai/apiKey": "<openai-api-key>" } } // pragma: allowlist secret
```

ข้อผิดพลาดต่อ id ซึ่งไม่บังคับ:

```json
{
  "protocolVersion": 1,
  "values": {},
  "errors": { "providers/openai/apiKey": { "code": "NOT_FOUND" } }
}
```

`code` เป็นข้อมูลวินิจฉัยที่เครื่องอ่านได้ซึ่งไม่บังคับ OpenClaw จะแสดง
รหัสที่รู้จัก `NOT_FOUND` และ `AMBIGUOUS_DUPLICATE_KEY` พร้อมผู้ให้บริการและ ref id ส่วนรหัสอื่น
และฟิลด์รูปแบบอิสระ เช่น `message` จะได้รับการยอมรับเพื่อความเข้ากันได้กับโปรโตคอล v1
แต่จะไม่แสดง เนื่องจากผลลัพธ์ของตัวแก้ค่าอาจมีข้อมูลข้อมูลประจำตัว

</Accordion>

## คีย์ API ที่เก็บในไฟล์

อย่าใส่สตริง `file:...` ในบล็อก `env` ของการกำหนดค่า บล็อกนั้นเป็นค่าตามตัวอักษรและไม่แทนที่ค่า ดังนั้นระบบจึงไม่แก้ค่า `file:...` ในตำแหน่งนั้น

ให้ใช้ SecretRef แบบไฟล์ในฟิลด์ข้อมูลประจำตัวที่รองรับแทน:

```json5
{
  secrets: {
    providers: {
      xai_key_file: {
        source: "file",
        path: "~/.openclaw/secrets/xai-api-key.txt",
        mode: "singleValue",
      },
    },
  },
  models: {
    providers: {
      xai: {
        apiKey: { source: "file", provider: "xai_key_file", id: "value" },
      },
    },
  },
}
```

สำหรับ `mode: "singleValue"` ค่า `id` ของ SecretRef คือ `"value"` สำหรับ `mode: "json"` ให้ใช้ตัวชี้ JSON แบบสัมบูรณ์ เช่น `"/providers/xai/apiKey"`

ดูฟิลด์ที่ยอมรับ SecretRef ได้ที่ [พื้นผิวข้อมูลประจำตัว SecretRef](/th/reference/secretref-credential-surface)

    ```js
    #!/usr/bin/env node
    const { spawnSync } = require("node:child_process");

    let stdin = "";
    process.stdin.setEncoding("utf8");
    process.stdin.on("data", (chunk) => {
      stdin += chunk;
    });
    process.stdin.on("error", (err) => {
      process.stderr.write(`${err.message}\n`);
      process.exit(1);
    });
    process.stdin.on("end", () => {
      let request;
      try {
        request = JSON.parse(stdin || "{}");
      } catch (err) {
        process.stderr.write(`แยกวิเคราะห์คำขอไม่สำเร็จ: ${err.message}\n`);
        process.exit(1);
      }

      const passBin = process.env.PASS_BIN || "pass";
      const values = {};
      const errors = {};

      for (const id of request.ids ?? []) {
        const result = spawnSync(passBin, ["show", id], { encoding: "utf8" });
        if (result.status === 0) {
          values[id] = result.stdout.split(/\r?\n/, 1)[0] ?? "";
        } else {
          errors[id] = { message: (result.stderr || `pass จบการทำงานด้วย ${result.status}`).trim() };
        }
      }

      process.stdout.write(JSON.stringify({ protocolVersion: 1, values, errors }));
    });
    ```

## ตัวแปรสภาพแวดล้อมของเซิร์ฟเวอร์ MCP

```json5
{
  plugins: {
    entries: {
      acpx: {
        enabled: true,
        config: {
          mcpServers: {
            github: {
              command: "npx",
              args: ["-y", "@modelcontextprotocol/server-github"],
              env: {
                GITHUB_PERSONAL_ACCESS_TOKEN: {
                  source: "env",
                  provider: "default",
                  id: "MCP_GITHUB_PAT",
                },
              },
            },
          },
        },
      },
    },
  },
}
```

ค่าสตริงข้อความธรรมดายังคงใช้งานได้ ระบบจะแก้ค่า ref เทมเพลตสภาพแวดล้อม เช่น `${MCP_SERVER_API_KEY}` และออบเจ็กต์ SecretRef ระหว่างการเปิดใช้งาน Gateway ก่อนสร้างกระบวนการเซิร์ฟเวอร์ MCP เช่นเดียวกับพื้นผิว SecretRef อื่น ref ที่แก้ค่าไม่ได้จะขัดขวางการเปิดใช้งานเฉพาะเมื่อ Plugin `acpx` ทำงานอยู่จริงเท่านั้น

## ข้อมูลการยืนยันตัวตน SSH ของ Sandbox

แบ็กเอนด์ Sandbox หลัก `ssh` ยังรองรับ SecretRef สำหรับข้อมูลการยืนยันตัวตน SSH ด้วย:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "ssh",
        ssh: {
          target: "user@gateway-host:22",
          identityData: { source: "env", provider: "default", id: "SSH_IDENTITY" },
          certificateData: { source: "env", provider: "default", id: "SSH_CERTIFICATE" },
          knownHostsData: { source: "env", provider: "default", id: "SSH_KNOWN_HOSTS" },
        },
      },
    },
  },
}
```

ลักษณะการทำงานขณะรัน:

- OpenClaw จะแก้ไขการอ้างอิงเหล่านี้ระหว่างการเปิดใช้งานแซนด์บ็อกซ์ ไม่ใช่แบบล่าช้าในการเรียก SSH แต่ละครั้ง
- ค่าที่แก้ไขแล้วจะถูกเขียนลงในไดเรกทอรีชั่วคราวโดยมีสิทธิ์อนุญาตไฟล์แบบจำกัด (`0o600`) และใช้ในไฟล์กำหนดค่า SSH ที่สร้างขึ้น
- หากแบ็กเอนด์แซนด์บ็อกซ์ที่มีผลไม่ใช่ `ssh` (หรือโหมดแซนด์บ็อกซ์เป็น `off`) การอ้างอิงเหล่านี้จะยังคงไม่ทำงานและไม่ขัดขวางการเริ่มต้นระบบ

## ขอบเขตข้อมูลประจำตัวที่รองรับ

ข้อมูลประจำตัวที่รองรับและไม่รองรับตามมาตรฐานแสดงไว้ใน [ขอบเขตข้อมูลประจำตัว SecretRef](/th/reference/secretref-credential-surface)

<Note>
ข้อมูลประจำตัวที่สร้างขึ้นขณะรันหรือมีการหมุนเวียน และข้อมูลสำหรับรีเฟรช OAuth ถูกยกเว้นจากการแก้ไข SecretRef แบบอ่านอย่างเดียวโดยตั้งใจ
</Note>

## ลักษณะการทำงานและลำดับความสำคัญที่กำหนด

- ฟิลด์ที่ไม่มีการอ้างอิง: ไม่เปลี่ยนแปลง
- ฟิลด์ที่มีการอ้างอิง: จำเป็นสำหรับพื้นผิวที่ใช้งานอยู่ระหว่างการเปิดใช้งาน
- หากมีทั้งข้อความธรรมดาและการอ้างอิง การอ้างอิงจะมีลำดับความสำคัญเหนือกว่าบนเส้นทางลำดับความสำคัญที่รองรับ
- ค่าเซนทิเนลสำหรับการปกปิดข้อมูล `__OPENCLAW_REDACTED__` สงวนไว้สำหรับการปกปิด/กู้คืนการกำหนดค่าภายใน และจะถูกปฏิเสธเมื่อส่งเป็นข้อมูลการกำหนดค่าแบบค่าตรงตัว

สัญญาณเตือนและการตรวจสอบ:

- `SECRETS_REF_OVERRIDES_PLAINTEXT` (คำเตือนขณะรัน)
- `REF_SHADOWED` (ผลการตรวจสอบเมื่อข้อมูลประจำตัว `auth-profiles.json` มีลำดับความสำคัญเหนือกว่าการอ้างอิง `openclaw.json`)

ความเข้ากันได้กับ Google Chat: `serviceAccountRef` มีลำดับความสำคัญเหนือกว่าข้อความธรรมดา `serviceAccount`; ค่าข้อความธรรมดาจะถูกละเว้นเมื่อกำหนดการอ้างอิงที่เป็นฟิลด์พี่น้องแล้ว

## ตัวกระตุ้นการเปิดใช้งาน

การเปิดใช้งานข้อมูลลับจะทำงานเมื่อ:

- เริ่มต้นระบบ (การตรวจสอบก่อนดำเนินการและการเปิดใช้งานขั้นสุดท้าย)
- เส้นทางใช้การเปลี่ยนแปลงทันทีเมื่อโหลดการกำหนดค่าใหม่
- เส้นทางตรวจสอบการเริ่มระบบใหม่เมื่อโหลดการกำหนดค่าใหม่
- โหลดใหม่ด้วยตนเองผ่าน `secrets.reload`
- การตรวจสอบก่อนดำเนินการของ RPC สำหรับเขียนการกำหนดค่า Gateway (`config.set` / `config.apply` / `config.patch`) โดยตรวจสอบว่าสามารถแก้ไข SecretRef บนพื้นผิวที่ใช้งานอยู่ภายในเพย์โหลดการกำหนดค่าที่ส่งมาได้ ก่อนบันทึกการแก้ไขอย่างถาวร

สัญญาการเปิดใช้งาน:

- เมื่อสำเร็จ ระบบจะสลับสแนปช็อตแบบอะตอมมิก
- ความล้มเหลวระหว่างเริ่มต้นจะยกเลิกการเริ่มต้น Gateway
- ความล้มเหลวในการโหลดใหม่ขณะรันจะคงสแนปช็อตล่าสุดที่ทราบว่าใช้งานได้ไว้
- ความล้มเหลวในการตรวจสอบก่อนดำเนินการของ RPC สำหรับเขียนจะปฏิเสธการกำหนดค่าที่ส่งมา โดยทั้งการกำหนดค่าบนดิสก์และสแนปช็อตขณะรันที่ใช้งานอยู่จะไม่เปลี่ยนแปลง
- การระบุโทเค็นช่องทางสำหรับแต่ละการเรียกอย่างชัดเจนให้แก่การเรียกตัวช่วย/เครื่องมือขาออกจะไม่กระตุ้นการเปิดใช้งาน SecretRef โดยจุดเปิดใช้งานยังคงเป็นการเริ่มต้นระบบ การโหลดใหม่ และ `secrets.reload` ที่เรียกอย่างชัดเจน

## สัญญาณสถานะเสื่อมประสิทธิภาพและฟื้นคืน

เมื่อการเปิดใช้งานระหว่างโหลดใหม่ล้มเหลวหลังจากอยู่ในสถานะปกติ OpenClaw จะเข้าสู่สถานะข้อมูลลับที่เสื่อมประสิทธิภาพ พร้อมส่งเหตุการณ์ระบบแบบครั้งเดียวและรหัสบันทึก:

- `SECRETS_RELOADER_DEGRADED`
- `SECRETS_RELOADER_RECOVERED`

ลักษณะการทำงาน:

- เสื่อมประสิทธิภาพ: ขณะรันจะคงสแนปช็อตล่าสุดที่ทราบว่าใช้งานได้ไว้
- ฟื้นคืน: ส่งหนึ่งครั้งหลังจากการเปิดใช้งานครั้งถัดไปสำเร็จ
- ความล้มเหลวซ้ำขณะที่อยู่ในสถานะเสื่อมประสิทธิภาพแล้วจะบันทึกคำเตือน แต่ไม่ส่งเหตุการณ์ซ้ำ
- การหยุดทันทีเมื่อเริ่มต้นล้มเหลวจะไม่ส่งเหตุการณ์สถานะเสื่อมประสิทธิภาพ เนื่องจากขณะรันไม่เคยเข้าสู่สถานะใช้งาน

## การแก้ไขเส้นทางคำสั่ง

เส้นทางคำสั่งสามารถเลือกใช้การแก้ไข SecretRef ที่รองรับผ่าน RPC สแนปช็อตของ Gateway ได้ โดยมีลักษณะการทำงานหลักสองแบบ:

<Tabs>
  <Tab title="เส้นทางคำสั่งแบบเคร่งครัด">
    ตัวอย่างเช่น เส้นทางหน่วยความจำระยะไกล `openclaw memory` และ `openclaw qr --remote` เมื่อต้องใช้การอ้างอิงข้อมูลลับที่ใช้ร่วมกันจากระยะไกล เส้นทางเหล่านี้อ่านจากสแนปช็อตที่ใช้งานอยู่และหยุดทันทีเมื่อ SecretRef ที่จำเป็นไม่พร้อมใช้งาน
  </Tab>
  <Tab title="เส้นทางคำสั่งแบบอ่านอย่างเดียว">
    ตัวอย่างเช่น `openclaw status`, `openclaw status --all`, `openclaw channels status`, `openclaw channels resolve`, `openclaw security audit` และขั้นตอนการซ่อมแซม doctor/การกำหนดค่าแบบอ่านอย่างเดียว เส้นทางเหล่านี้จะเลือกใช้สแนปช็อตที่ใช้งานอยู่ก่อนเช่นกัน แต่จะลดระดับการทำงานแทนการยกเลิกเมื่อ SecretRef เป้าหมายไม่พร้อมใช้งาน

    ลักษณะการทำงานแบบอ่านอย่างเดียว:

    - เมื่อ Gateway กำลังทำงาน คำสั่งเหล่านี้จะอ่านจากสแนปช็อตที่ใช้งานอยู่ก่อน
    - หากการแก้ไขผ่าน Gateway ไม่สมบูรณ์หรือ Gateway ไม่พร้อมใช้งาน ระบบจะลองใช้ทางเลือกสำรองภายในเครื่องที่เจาะจงสำหรับพื้นผิวของคำสั่งนั้น
    - หาก SecretRef เป้าหมายยังคงไม่พร้อมใช้งาน คำสั่งจะทำงานต่อด้วยผลลัพธ์แบบอ่านอย่างเดียวที่ลดระดับลง พร้อมข้อความวินิจฉัยอย่างชัดเจนว่ามีการกำหนดค่าการอ้างอิงไว้ แต่ไม่พร้อมใช้งานในเส้นทางคำสั่งนี้
    - ลักษณะการทำงานที่ลดระดับนี้มีผลเฉพาะภายในคำสั่งเท่านั้น และไม่ทำให้ข้อกำหนดของการเริ่มต้นขณะรัน การโหลดใหม่ หรือเส้นทางส่ง/ตรวจสอบสิทธิ์อ่อนลง

  </Tab>
</Tabs>

หมายเหตุอื่นๆ:

- การรีเฟรชสแนปช็อตหลังจากหมุนเวียนข้อมูลลับของแบ็กเอนด์ดำเนินการโดย `openclaw secrets reload`
- เมธอด RPC ของ Gateway ที่เส้นทางคำสั่งเหล่านี้ใช้: `secrets.resolve`

## เวิร์กโฟลว์การตรวจสอบและกำหนดค่า

ขั้นตอนเริ่มต้นสำหรับผู้ดำเนินการ:

<Steps>
  <Step title="ตรวจสอบสถานะปัจจุบัน">
    ```bash
    openclaw secrets audit --check
    ```
  </Step>
  <Step title="กำหนดค่าและใช้ SecretRef">
    ```bash
    openclaw secrets configure --apply
    ```
  </Step>
  <Step title="ตรวจสอบซ้ำ">
    ```bash
    openclaw secrets audit --check
    ```
  </Step>
</Steps>

อย่าถือว่าการย้ายข้อมูลเสร็จสมบูรณ์จนกว่าการตรวจสอบซ้ำจะไม่พบปัญหา หากการตรวจสอบยังคงรายงานค่าข้อความธรรมดาที่จัดเก็บอยู่ ความเสี่ยงที่เอเจนต์จะเข้าถึงได้ยังคงอยู่ แม้ว่า API ขณะรันจะส่งคืนค่าที่ปกปิดแล้วก็ตาม

หากบันทึกแผนแทนการนำไปใช้ระหว่าง `configure` ให้นำแผนที่บันทึกไว้นั้นไปใช้ด้วย `openclaw secrets apply --from <plan-path>` ก่อนการตรวจสอบซ้ำ

<AccordionGroup>
  <Accordion title="secrets audit">
    ผลการตรวจสอบประกอบด้วย:

    - ค่าข้อความธรรมดาที่จัดเก็บอยู่ (`openclaw.json`, `auth-profiles.json`, `.env` และ `agents/*/agent/models.json` ที่สร้างขึ้น)
    - ข้อมูลส่วนหัวที่ละเอียดอ่อนของผู้ให้บริการซึ่งตกค้างอยู่ในรายการ `models.json` ที่สร้างขึ้น
    - การอ้างอิงที่แก้ไขไม่ได้
    - การบดบังตามลำดับความสำคัญ (`auth-profiles.json` มีลำดับความสำคัญเหนือกว่าการอ้างอิง `openclaw.json`)
    - ข้อมูลตกค้างแบบเดิม (`auth.json`, การแจ้งเตือนเกี่ยวกับ OAuth)

    หมายเหตุเกี่ยวกับ exec: โดยค่าเริ่มต้น การตรวจสอบจะข้ามการตรวจสอบความสามารถในการแก้ไข SecretRef แบบ exec เพื่อหลีกเลี่ยงผลข้างเคียงของคำสั่ง ใช้ `openclaw secrets audit --allow-exec` เพื่อเรียกใช้ผู้ให้บริการ exec ระหว่างการตรวจสอบ

    หมายเหตุเกี่ยวกับข้อมูลส่วนหัวตกค้าง: การตรวจหาส่วนหัวที่ละเอียดอ่อนของผู้ให้บริการใช้การวิเคราะห์ชื่อแบบฮิวริสติก (ชื่อและส่วนประกอบของชื่อส่วนหัวที่เกี่ยวกับการตรวจสอบสิทธิ์/ข้อมูลประจำตัวซึ่งพบได้ทั่วไป เช่น `authorization`, `x-api-key`, `token`, `secret`, `password` และ `credential`)

  </Accordion>
  <Accordion title="secrets configure">
    ตัวช่วยแบบโต้ตอบที่:

    - กำหนดค่า `secrets.providers` ก่อน (`env`/`file`/`exec`, เพิ่ม/แก้ไข/ลบ)
    - ให้เลือกฟิลด์ที่รองรับซึ่งมีข้อมูลลับใน `openclaw.json` รวมถึง `auth-profiles.json` สำหรับขอบเขตของเอเจนต์หนึ่งรายการ
    - สามารถสร้างการแมป `auth-profiles.json` ใหม่ได้โดยตรงในตัวเลือกเป้าหมาย
    - บันทึกรายละเอียด SecretRef (`source`, `provider`, `id`)
    - เรียกใช้การแก้ไขเพื่อตรวจสอบก่อนดำเนินการและสามารถนำไปใช้ได้ทันที

    หมายเหตุเกี่ยวกับ exec: การตรวจสอบก่อนดำเนินการจะข้ามการตรวจสอบ SecretRef แบบ exec เว้นแต่จะกำหนด `--allow-exec` หากนำไปใช้โดยตรงจาก `configure --apply` และแผนมีการอ้างอิง/ผู้ให้บริการแบบ exec ให้คงการตั้งค่า `--allow-exec` ไว้สำหรับขั้นตอนการนำไปใช้ด้วย

    โหมดที่เป็นประโยชน์:

    - `openclaw secrets configure --providers-only`
    - `openclaw secrets configure --skip-provider-setup`
    - `openclaw secrets configure --agent <id>`

    ค่าเริ่มต้นในการนำ `configure` ไปใช้:

    - ล้างข้อมูลประจำตัวแบบคงที่ที่ตรงกันออกจาก `auth-profiles.json` สำหรับผู้ให้บริการเป้าหมาย
    - ล้างรายการ `api_key` แบบคงที่รุ่นเดิมออกจาก `auth.json`
    - ล้างบรรทัดข้อมูลลับที่ทราบและตรงกันออกจาก `<config-dir>/.env`

  </Accordion>
  <Accordion title="secrets apply">

    ```bash
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --allow-exec
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run --allow-exec
    ```

    หมายเหตุเกี่ยวกับ exec: การทดลองทำจะข้ามการตรวจสอบ exec เว้นแต่จะกำหนด `--allow-exec`; โหมดเขียนจะปฏิเสธแผนที่มี SecretRef/ผู้ให้บริการแบบ exec เว้นแต่จะกำหนด `--allow-exec`

    สำหรับรายละเอียดสัญญาเป้าหมาย/เส้นทางแบบเคร่งครัดและกฎการปฏิเสธที่แน่นอน โปรดดู [สัญญาแผนการนำข้อมูลลับไปใช้](/th/gateway/secrets-plan-contract)

  </Accordion>
</AccordionGroup>

## นโยบายความปลอดภัยแบบทางเดียว

<Warning>
OpenClaw ตั้งใจไม่เขียนข้อมูลสำรองสำหรับย้อนกลับที่มีค่าข้อมูลลับแบบข้อความธรรมดาในอดีต
</Warning>

รูปแบบความปลอดภัย:

- การตรวจสอบก่อนดำเนินการต้องสำเร็จก่อนเข้าสู่โหมดเขียน
- ต้องตรวจสอบความถูกต้องของการเปิดใช้งานขณะรันก่อนคอมมิต
- การนำไปใช้จะอัปเดตไฟล์ด้วยการแทนที่ไฟล์แบบอะตอมมิกและพยายามกู้คืนอย่างเต็มความสามารถเมื่อล้มเหลว

## หมายเหตุความเข้ากันได้กับการตรวจสอบสิทธิ์รุ่นเดิม

สำหรับข้อมูลประจำตัวแบบคงที่ ขณะรันจะไม่พึ่งพาพื้นที่จัดเก็บการตรวจสอบสิทธิ์แบบข้อความธรรมดารุ่นเดิมอีกต่อไป

- แหล่งข้อมูลประจำตัวขณะรันคือสแนปช็อตในหน่วยความจำที่แก้ไขแล้ว
- รายการ `api_key` แบบคงที่รุ่นเดิมจะถูกล้างเมื่อตรวจพบ
- ลักษณะการทำงานเพื่อความเข้ากันได้ที่เกี่ยวข้องกับ OAuth ยังคงแยกต่างหาก

## หมายเหตุเกี่ยวกับ Web UI

ยูเนียน SecretInput บางรายการกำหนดค่าในโหมดตัวแก้ไขข้อมูลดิบได้ง่ายกว่าในโหมดฟอร์ม
