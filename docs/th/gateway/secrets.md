OpenClaw รองรับ SecretRefs แบบเพิ่มต่อยอดได้ เพื่อให้ข้อมูลรับรองที่รองรับไม่จำเป็นต้องจัดเก็บเป็นข้อความธรรมดาในการกำหนดค่า

<Note>
ยังคงใช้ข้อความธรรมดาได้ โดยต้องเลือกใช้ SecretRefs แยกสำหรับข้อมูลรับรองแต่ละรายการ
</Note>

<Warning>
ข้อมูลรับรองแบบข้อความธรรมดายังคงอ่านได้โดยเอเจนต์ หากอยู่ในไฟล์ที่เอเจนต์ตรวจสอบได้ รวมถึง `openclaw.json`, `auth-profiles.json`, `.env` หรือไฟล์ `agents/*/agent/models.json` ที่สร้างขึ้น SecretRefs จะลดขอบเขตผลกระทบภายในเครื่องได้ก็ต่อเมื่อย้ายข้อมูลรับรองที่รองรับทั้งหมดแล้ว และ `openclaw secrets audit --check` รายงานว่าไม่มีข้อความธรรมดาตกค้าง
</Warning>

## โมเดลรันไทม์

- ข้อมูลลับจะถูกแก้ค่าเป็นสแนปช็อตรันไทม์ในหน่วยความจำล่วงหน้าระหว่างการเปิดใช้งาน ไม่ใช่แก้ค่าแบบล่าช้าบนเส้นทางคำขอ
- การเริ่มต้น Gateway แบบ cold จะแยกความล้มเหลวของ SecretRef ที่ลองใหม่ได้ไปยังเจ้าของที่ทราบแน่ชัดซึ่งไม่ใช่ Gateway เมื่อเจ้าของนั้นรองรับการแยก เจ้าของประเภทที่แมปไว้ประกอบด้วยผู้ให้บริการโมเดลและ Skills, ผู้ให้บริการสื่อ/TTS/cron, โปรไฟล์การยืนยันตัวตนที่เข้าเกณฑ์, หน่วยความจำแยกตามเอเจนต์, SSH ของแซนด์บ็อกซ์, บัญชีช่องทาง และเส้นทาง Plugin ที่ประกาศในแมนิเฟสต์ Gateway จะเริ่มทำงาน บันทึกเจ้าของว่าได้รับการกำหนดค่าแต่ไม่พร้อมใช้งาน และส่งคำเตือนการลดระดับการทำงานที่ปกปิดข้อมูลแล้ว การยืนยันตัวตนขาเข้าของ Gateway, การอ้างอิงหรือค่าที่แก้แล้วซึ่งมีโครงสร้างไม่ถูกต้อง, เจ้าของที่ปิดล้มเหลว และการอ้างอิงที่ยังไม่ได้แมปเจ้าของรันไทม์ จะยังคงทำให้การเริ่มต้นล้มเหลว
- การโหลดใหม่จะตรวจสอบเจ้าของแต่ละรายที่แมปไว้อย่างอิสระ จากนั้นเผยแพร่สแนปช็อตเดียวแบบอะตอม เจ้าของที่ปกติจะได้รับการรีเฟรช เจ้าของที่ล้มเหลวและเข้าเกณฑ์จะคงค่าล่าสุดที่ทราบว่าใช้งานได้ และจะกลายเป็น stale เฉพาะเมื่ออัตลักษณ์การอ้างอิง คำจำกัดความผู้ให้บริการ และสัญญาของเจ้าของที่ไม่เป็นข้อมูลลับทั้งหมดไม่เปลี่ยนแปลง ส่วนเจ้าของที่เปลี่ยนแปลงหรือเพิ่มใหม่และล้มเหลวจะกลายเป็น cold ความล้มเหลวแบบเข้มงวดจะปฏิเสธการโหลดใหม่และเก็บรักษาสแนปช็อตที่ใช้งานอยู่
- การละเมิดนโยบาย (ตัวอย่างเช่น โปรไฟล์การยืนยันตัวตนโหมด OAuth ที่ใช้ร่วมกับอินพุต SecretRef) จะทำให้การเปิดใช้งานล้มเหลวก่อนสลับรันไทม์
- คำขอรันไทม์จะอ่านเฉพาะสแนปช็อตในหน่วยความจำที่ใช้งานอยู่ ข้อมูลรับรอง SecretRef ของผู้ให้บริการโมเดลจะผ่านพื้นที่จัดเก็บการยืนยันตัวตนและตัวเลือกสตรีมในรูปเซนทิเนลเฉพาะภายในกระบวนการจนถึงขาออก เส้นทางการส่งออก (การส่งคำตอบ/เธรดของ Discord และการส่งการดำเนินการของ Telegram) จะอ่านสแนปช็อตนั้นเช่นกัน และจะไม่แก้ค่าการอ้างอิงใหม่ทุกครั้งที่ส่ง

วิธีนี้ทำให้เหตุขัดข้องของผู้ให้บริการข้อมูลลับไม่กระทบเส้นทางคำขอที่ใช้งานหนัก

การป้องกันขาเข้าของ Gateway, การกำหนดค่าหรือค่าที่แก้แล้วซึ่งมีโครงสร้างไม่ถูกต้อง, การละเมิดนโยบาย และความเป็นเจ้าของที่ไม่ทราบแน่ชัด จะยังคงปิดล้มเหลว เจ้าของที่ถูกแยกจะไม่ย้อนกลับไปใช้แหล่งข้อมูลรับรองที่มีลำดับความสำคัญต่ำกว่า

## การแทรกขณะส่งออก (เซนทิเนล)

สำหรับข้อมูลรับรองของผู้ให้บริการโมเดลที่ใช้ SecretRefs เป็นแหล่งข้อมูล OpenClaw จะสร้างเซนทิเนลทึบแสงเฉพาะภายในกระบวนการระหว่างการแก้ค่าการยืนยันตัวตนของโมเดล ดังนั้นพื้นที่จัดเก็บการยืนยันตัวตน ตัวเลือกสตรีม การกำหนดค่า SDK บันทึก ออบเจ็กต์ข้อผิดพลาด และการตรวจสอบภายในรันไทม์ส่วนใหญ่จะเห็นค่าเช่น `oc-sent-v1-...` แทนข้อมูลรับรองของผู้ให้บริการ การดึงข้อมูลโมเดลที่มีการป้องกันและโพรบตรวจสอบสถานะผู้ให้บริการภายในเครื่องที่มีการจัดการ จะแทนที่เซนทิเนลที่รู้จักในค่า URL และส่วนหัวทันทีก่อนที่แต่ละคำขอจะออกจากกระบวนการ

ค่าที่มีรูปแบบคล้ายเซนทิเนลแต่ไม่รู้จักจะถูกปิดล้มเหลวก่อนมีกิจกรรมเครือข่าย OpenClaw จะปฏิเสธการส่งคำขอแทนที่จะส่งต่อเซนทิเนลที่ยังไม่ได้แก้ค่าไปยังผู้ให้บริการ ค่าข้อมูลลับที่แก้แล้วจะได้รับการลงทะเบียนเพื่อปกปิดค่าแบบตรงกันทุกประการในบันทึกด้วย เพื่อเป็นมาตรการป้องกันหลายชั้น

อะแดปเตอร์ผู้ให้บริการใช้จุดแทรกล่าสุดที่ SDK ของตนรองรับ:

- SDK ที่มีตัวเลือก fetch แบบกำหนดเองจะได้รับ fetch ที่มีการป้องกันของ OpenClaw เพื่อให้ SDK ยังคงเก็บเซนทิเนลไว้
- SDK ที่ไม่มีตัวเลือก fetch แบบกำหนดเองจะแกะเซนทิเนลทันทีก่อนสร้างไคลเอนต์ สตรีมของผู้ให้บริการที่ Plugin เป็นเจ้าของและชุดควบคุมเอเจนต์จะแกะค่าที่จุดส่งมอบสุดท้ายซึ่ง core เป็นเจ้าของ เนื่องจากการขนส่งเหล่านั้นไม่ได้ใช้ fetch ที่มีการป้องกันของ OpenClaw ร่วมกัน

เซนทิเนลช่วยลดการเปิดเผยข้อความธรรมดาตลอดสายการเรียกโมเดล แต่ไม่ใช่การแยกกระบวนการ ค่าจริงยังคงอยู่ในหน่วยความจำของกระบวนการเดียวกันและปรากฏที่ขอบเขตอะแดปเตอร์สุดท้าย ข้อมูลรับรองสภาพแวดล้อมแบบข้อความธรรมดาที่ไม่ได้กำหนดค่าผ่าน SecretRefs จะยังคงเป็นข้อความธรรมดาและอยู่นอกกลไกนี้

ตั้งค่า `OPENCLAW_SECRET_SENTINELS=off` (รองรับ `0` หรือ `false` ด้วย โดยไม่คำนึงถึงตัวพิมพ์เล็ก-ใหญ่) เพื่อปิดการสร้างเซนทิเนลระหว่างการรับมือเหตุการณ์หรือการแก้ปัญหาความเข้ากันได้ สวิตช์หยุดฉุกเฉินนี้จะไม่ปิดการลงทะเบียนการปกปิดค่าแบบตรงกันทุกประการ

## ขอบเขตการเข้าถึงของเอเจนต์

SecretRefs ป้องกันไม่ให้ข้อมูลรับรองถูกจัดเก็บถาวรในการกำหนดค่าและไฟล์โมเดลที่สร้างขึ้น แต่ไม่ใช่ขอบเขตการแยกกระบวนการ ข้อมูลรับรองแบบข้อความธรรมดาที่หลงเหลืออยู่บนดิสก์ในเส้นทางที่เอเจนต์อ่านได้ ยังคงอ่านได้ผ่านเครื่องมือไฟล์หรือเชลล์ ซึ่งข้ามการปกปิดระดับ API

สำหรับการนำไปใช้งานจริงที่ครอบคลุมไฟล์ซึ่งเอเจนต์เข้าถึงได้ ให้ถือว่าการย้ายเสร็จสมบูรณ์ต่อเมื่อเงื่อนไขทั้งหมดต่อไปนี้เป็นจริง:

- ข้อมูลรับรองที่รองรับใช้ SecretRefs แทนค่าแบบข้อความธรรมดา
- ล้างข้อความธรรมดาเดิมที่ตกค้างออกจาก `openclaw.json`, `auth-profiles.json`, `.env` และไฟล์ `models.json` ที่สร้างขึ้นแล้ว
- `openclaw secrets audit --check` ไม่มีสิ่งตกค้างหลังการย้าย
- ข้อมูลรับรองที่ยังไม่รองรับหรือมีการหมุนเวียนซึ่งเหลืออยู่ ได้รับการป้องกันด้วยการแยกระดับ OS, การแยกคอนเทนเนอร์ หรือพร็อกซีข้อมูลรับรองภายนอก

นี่คือเหตุผลที่เวิร์กโฟลว์ตรวจสอบ/กำหนดค่า/นำไปใช้เป็นเกตการย้ายด้านความปลอดภัย ไม่ใช่เพียงตัวช่วยอำนวยความสะดวก

<Warning>
SecretRefs ไม่ได้ทำให้ไฟล์ใดๆ ที่อ่านได้ปลอดภัย ข้อมูลสำรอง การกำหนดค่าที่คัดลอก แค็ตตาล็อกโมเดลเก่าที่สร้างขึ้น และข้อมูลรับรองประเภทที่ไม่รองรับ ยังคงเป็นข้อมูลลับสำหรับการใช้งานจริงจนกว่าจะถูกลบ ย้ายออกนอกขอบเขตความเชื่อถือของเอเจนต์ หรือแยกต่างหาก
</Warning>

## การกรองพื้นผิวที่ใช้งานอยู่

SecretRefs จะได้รับการตรวจสอบเฉพาะบนพื้นผิวที่มีผลว่าใช้งานอยู่:

- **พื้นผิวที่เปิดใช้งาน**: ความล้มเหลวที่ลองใหม่ได้สำหรับเจ้าของที่แมปและแยกได้จะเข้าสู่การลดระดับแบบ cold หรือ stale ความล้มเหลวแบบเข้มงวด ปิดล้มเหลว จำเป็นต่อ Gateway หรือยังไม่ได้แมป จะขัดขวางการเริ่มต้น/โหลดใหม่
- **พื้นผิวที่ไม่ได้ใช้งาน**: การอ้างอิงที่ยังไม่ได้แก้ค่าจะไม่ขัดขวางการเริ่มต้น/โหลดใหม่ แต่จะส่งการวินิจฉัย `SECRETS_REF_IGNORED_INACTIVE_SURFACE` ที่ไม่ร้ายแรง

<Accordion title="ตัวอย่างพื้นผิวที่ไม่ได้ใช้งาน">
- รายการช่องทาง/บัญชีที่ปิดใช้งาน
- ข้อมูลรับรองช่องทางระดับบนสุดที่ไม่มีบัญชีซึ่งเปิดใช้งานสืบทอดไปใช้
- พื้นผิวเครื่องมือ/ฟีเจอร์ที่ปิดใช้งาน
- คีย์เฉพาะผู้ให้บริการค้นหาเว็บที่ไม่ได้เลือกโดย `tools.web.search.provider` ในโหมดอัตโนมัติ (ไม่ได้ตั้งค่าผู้ให้บริการ) ระบบจะตรวจสอบคีย์ตามลำดับความสำคัญเพื่อการตรวจหาอัตโนมัติจนกว่าจะมีคีย์หนึ่งแก้ค่าได้ หลังการเลือก คีย์ของผู้ให้บริการที่ไม่ได้เลือกจะไม่ได้ใช้งาน
- ข้อมูลการยืนยันตัวตน SSH ของแซนด์บ็อกซ์ (`agents.defaults.sandbox.ssh.identityData`, `certificateData`, `knownHostsData` รวมถึงค่าที่แทนที่แยกตามเอเจนต์) จะใช้งานเฉพาะเมื่อแบ็กเอนด์แซนด์บ็อกซ์ที่มีผลเป็น `ssh` และโหมดแซนด์บ็อกซ์ไม่ใช่ `off` สำหรับเอเจนต์เริ่มต้นหรือเอเจนต์ที่เปิดใช้งาน
- SecretRefs ของ `gateway.remote.token` / `gateway.remote.password` จะใช้งานหากเงื่อนไขใดเงื่อนไขหนึ่งต่อไปนี้เป็นจริง:
  - `gateway.mode=remote`
  - มีการกำหนดค่า `gateway.remote.url`
  - `gateway.tailscale.mode` เป็น `serve` หรือ `funnel`
  - ในโหมดภายในเครื่องที่ไม่มีพื้นผิวระยะไกลเหล่านั้น: `gateway.remote.token` จะใช้งานเมื่อการยืนยันตัวตนด้วยโทเค็นสามารถชนะได้และไม่ได้กำหนดค่าโทเค็นสภาพแวดล้อม/การยืนยันตัวตน ส่วน `gateway.remote.password` จะใช้งานเฉพาะเมื่อการยืนยันตัวตนด้วยรหัสผ่านสามารถชนะได้และไม่ได้กำหนดค่ารหัสผ่านสภาพแวดล้อม/การยืนยันตัวตน
- SecretRef ของ `gateway.auth.token` จะไม่ใช้งานสำหรับการแก้ค่าการยืนยันตัวตนขณะเริ่มต้นเมื่อกำหนด `OPENCLAW_GATEWAY_TOKEN` ไว้ เนื่องจากอินพุตโทเค็นจากสภาพแวดล้อมมีลำดับความสำคัญสูงกว่าสำหรับรันไทม์นั้น

</Accordion>

## การวินิจฉัยพื้นผิวการยืนยันตัวตนของ Gateway

เมื่อกำหนด SecretRef บน `gateway.auth.token`, `gateway.auth.password`, `gateway.remote.token` หรือ `gateway.remote.password` การเริ่มต้น/โหลดใหม่ของ Gateway จะบันทึกสถานะพื้นผิวภายใต้รหัส `SECRETS_GATEWAY_AUTH_SURFACE`:

- `active`: SecretRef เป็นส่วนหนึ่งของพื้นผิวการยืนยันตัวตนที่มีผลและต้องแก้ค่าได้
- `inactive`: พื้นผิวการยืนยันตัวตนอื่นมีลำดับความสำคัญสูงกว่า หรือการยืนยันตัวตนระยะไกลถูกปิดใช้งาน/ไม่ได้ใช้งาน

รายการบันทึกจะระบุเหตุผลที่นโยบายพื้นผิวที่ใช้งานเลือกใช้

## การตรวจสอบล่วงหน้าของการอ้างอิงระหว่างการเริ่มต้นใช้งาน

ในการเริ่มต้นใช้งานแบบโต้ตอบ การเลือกจัดเก็บด้วย SecretRef จะเรียกใช้การตรวจสอบล่วงหน้าก่อนบันทึก:

- การอ้างอิง Env: ตรวจสอบชื่อของตัวแปรสภาพแวดล้อมและยืนยันว่ามองเห็นค่าที่ไม่ว่างเปล่าระหว่างการตั้งค่า
- การอ้างอิงผู้ให้บริการ (`file` หรือ `exec`): ตรวจสอบการเลือกผู้ให้บริการ แก้ค่า `id` และตรวจสอบชนิดของค่าที่แก้แล้ว
- ขั้นตอนเริ่มต้นด่วน: เมื่อ `gateway.auth.token` เป็น SecretRef อยู่แล้ว การเริ่มต้นใช้งานจะแก้ค่าก่อนเริ่มต้นโพรบ/แดชบอร์ด (สำหรับการอ้างอิง `env`, `file` และ `exec`) โดยใช้เกตที่ล้มเหลวทันทีแบบเดียวกัน

เมื่อการตรวจสอบล้มเหลว ระบบจะแสดงข้อผิดพลาดและให้ลองใหม่ได้

## สัญญา SecretRef

ใช้รูปแบบออบเจ็กต์เดียวกันทุกที่:

```json5
{ source: "env" | "file" | "exec", provider: "default", id: "..." }
```

<Tabs>
  <Tab title="env">
    ```json5
    { source: "env", provider: "default", id: "OPENAI_API_KEY" }
    ```

    ช่อง SecretInput รองรับสตริงแบบย่อด้วย:

    ```json5
    "${OPENAI_API_KEY}"
    "$OPENAI_API_KEY"
    ```

    การตรวจสอบ:

    - `provider` ต้องตรงกับ `^[a-z][a-z0-9_-]{0,63}$`
    - `id` ต้องตรงกับ `^[A-Z][A-Z0-9_]{0,127}$`

  </Tab>
  <Tab title="file">
    ```json5
    { source: "file", provider: "filemain", id: "/providers/openai/apiKey" }
    ```

    การตรวจสอบ:

    - `provider` ต้องตรงกับ `^[a-z][a-z0-9_-]{0,63}$`
    - `id` ต้องเป็นตัวชี้ JSON แบบสัมบูรณ์ (`/...`) หรือเป็นค่าตรงตัว `value` สำหรับผู้ให้บริการ `singleValue`
    - การหลีกอักขระตาม RFC 6901 ในแต่ละเซกเมนต์: `~` จะกลายเป็น `~0`, `/` จะกลายเป็น `~1`

  </Tab>
  <Tab title="exec">
    ```json5
    { source: "exec", provider: "vault", id: "providers/openai/apiKey#value" }
    ```

    การตรวจสอบ:

    - `provider` ต้องตรงกับ `^[a-z][a-z0-9_-]{0,63}$`
    - `id` ต้องตรงกับ `^[A-Za-z0-9][A-Za-z0-9._:/#-]{0,255}$` (รองรับตัวเลือก เช่น `secret#json_key`)
    - `id` ต้องไม่มี `.` หรือ `..` เป็นเซกเมนต์เส้นทางที่คั่นด้วยเครื่องหมายทับ (ตัวอย่างเช่น `a/../b` จะถูกปฏิเสธ)

  </Tab>
</Tabs>

## การกำหนดค่าผู้ให้บริการ

กำหนดผู้ให้บริการภายใต้ `secrets.providers`:

```json5
{
  secrets: {
    providers: {
      default: { source: "env" },
      filemain: {
        source: "file",
        path: "~/.openclaw/secrets.json",
        mode: "json", // or "singleValue"
      },
      vault: {
        source: "exec",
        command: "/usr/local/bin/openclaw-vault-resolver",
        args: ["--profile", "prod"],
        passEnv: ["PATH", "VAULT_ADDR"],
        jsonOnly: true,
      },
      "team-secrets": {
        source: "exec",
        pluginIntegration: {
          pluginId: "acme-secrets",
          integrationId: "secret-store",
        },
      },
    },
    defaults: {
      env: "default",
      file: "filemain",
      exec: "vault",
    },
    resolution: {
      maxProviderConcurrency: 4,
      maxRefsPerProvider: 512,
      maxBatchBytes: 262144,
    },
  },
}
```

<Accordion title="ผู้ให้บริการ Env">
- รายการอนุญาตชื่อแบบตรงกันทุกประการที่เลือกใช้ได้ผ่าน `allowlist`
- ค่าสภาพแวดล้อมที่หายไปหรือว่างเปล่าจะทำให้การแก้ค่าล้มเหลว

</Accordion>

<Accordion title="ผู้ให้บริการไฟล์">
- อ่านไฟล์ภายในเครื่องที่ `path`
- `mode: "json"` (ค่าเริ่มต้น) คาดว่าเพย์โหลดเป็นออบเจ็กต์ JSON และตีความ `id` เป็นตัวชี้ JSON
- `mode: "singleValue"` คาดว่าจะได้รับรหัสอ้างอิง `"value"` และส่งคืนเนื้อหาไฟล์ดิบ (ตัดอักขระขึ้นบรรทัดใหม่ที่ท้ายออก)
- พาธต้องผ่านการตรวจสอบความเป็นเจ้าของ/สิทธิ์ โดย `timeoutMs` (ค่าเริ่มต้น 5000) และ `maxBytes` (ค่าเริ่มต้น 1 MiB) จำกัดการอ่าน
- Windows ปฏิเสธโดยค่าเริ่มต้น: หากตรวจสอบ ACL สำหรับพาธไม่ได้ การแก้ค่าจะล้มเหลว สำหรับพาธที่เชื่อถือได้เท่านั้น ให้ตั้งค่า `allowInsecurePath: true` บนผู้ให้บริการนั้นเพื่อข้ามการตรวจสอบ

</Accordion>

<Accordion title="ผู้ให้บริการ Exec">
- เรียกใช้พาธไบนารีแบบสัมบูรณ์ที่กำหนดค่าไว้โดยตรงโดยไม่ใช้เชลล์
- โดยค่าเริ่มต้น `command` ต้องเป็นไฟล์ปกติ ไม่ใช่ลิงก์สัญลักษณ์ ตั้งค่า `allowSymlinkCommand: true` เพื่ออนุญาตพาธคำสั่งที่เป็นลิงก์สัญลักษณ์ (เช่น ชิมของ Homebrew) และใช้ร่วมกับ `trustedDirs` (เช่น `["/opt/homebrew"]`) เพื่อให้เฉพาะพาธของตัวจัดการแพ็กเกจเท่านั้นที่ผ่านเกณฑ์
- รองรับ `timeoutMs` (ค่าเริ่มต้น 5000), `noOutputTimeoutMs` (ค่าเริ่มต้นเท่ากับ `timeoutMs`), `maxOutputBytes` (ค่าเริ่มต้น 1 MiB), รายการอนุญาต `env`/`passEnv` และ `trustedDirs`
- `jsonOnly` มีค่าเริ่มต้นเป็น `true` เมื่อใช้ `jsonOnly: false` และร้องขอเพียงรหัสเดียว ระบบจะยอมรับ stdout แบบข้อความธรรมดาที่ไม่ใช่ JSON เป็นค่าของรหัสนั้น
- Windows ปฏิเสธโดยค่าเริ่มต้น: หากตรวจสอบ ACL สำหรับพาธคำสั่งไม่ได้ การแก้ค่าจะล้มเหลว สำหรับพาธที่เชื่อถือได้เท่านั้น ให้ตั้งค่า `allowInsecurePath: true` บนผู้ให้บริการนั้นเพื่อข้ามการตรวจสอบ
- ผู้ให้บริการ exec ที่จัดการโดย Plugin สามารถใช้ `pluginIntegration` แทน `command`/`args` ที่คัดลอกมาได้ OpenClaw จะแก้รายละเอียดคำสั่งปัจจุบันจากไฟล์ manifest ของ Plugin ที่ติดตั้งระหว่างการเริ่มต้น/โหลดซ้ำ หาก Plugin ถูกปิดใช้งาน ถูกลบ ไม่น่าเชื่อถือ หรือไม่ได้ประกาศการผสานรวมอีกต่อไป SecretRef ที่ใช้งานอยู่บนผู้ให้บริการนั้นจะปฏิเสธโดยค่าเริ่มต้น

เพย์โหลดคำขอ (stdin):

```json
{ "protocolVersion": 1, "provider": "vault", "ids": ["providers/openai/apiKey"] }
```

เพย์โหลดการตอบกลับ (stdout):

```jsonc
{ "protocolVersion": 1, "values": { "providers/openai/apiKey": "<openai-api-key>" } } // pragma: allowlist secret
```

ข้อผิดพลาดแยกตามรหัสซึ่งระบุหรือไม่ก็ได้:

```json
{
  "protocolVersion": 1,
  "values": {},
  "errors": { "providers/openai/apiKey": { "code": "NOT_FOUND" } }
}
```

`code` คือข้อมูลวินิจฉัยที่เครื่องอ่านได้ซึ่งระบุหรือไม่ก็ได้ OpenClaw แสดง
รหัสที่รู้จัก ได้แก่ `NOT_FOUND` และ `AMBIGUOUS_DUPLICATE_KEY` พร้อมผู้ให้บริการและรหัสอ้างอิง ส่วนรหัสอื่น
และฟิลด์รูปแบบอิสระ เช่น `message` จะได้รับการยอมรับเพื่อความเข้ากันได้กับ protocol-v1
แต่จะไม่แสดง เนื่องจากเอาต์พุตของตัวแก้ค่าอาจมีข้อมูลข้อมูลประจำตัว

</Accordion>

## คีย์ API ที่เก็บในไฟล์

อย่าใส่สตริง `file:...` ในบล็อก `env` ของการกำหนดค่า บล็อกดังกล่าวเป็นค่าตามตัวอักษรและไม่เขียนทับค่า ดังนั้นจึงไม่มีการแก้ค่า `file:...` ในบล็อกนั้น

ให้ใช้ SecretRef แบบไฟล์กับฟิลด์ข้อมูลประจำตัวที่รองรับแทน:

```json5
{
  secrets: {
    providers: {
      xai_key_file: {
        source: "file",
        path: "~/.openclaw/secrets/xai-api-key.txt",
        mode: "singleValue",
      },
    },
  },
  models: {
    providers: {
      xai: {
        apiKey: { source: "file", provider: "xai_key_file", id: "value" },
      },
    },
  },
}
```

สำหรับ `mode: "singleValue"` ค่า `id` ของ SecretRef คือ `"value"` สำหรับ `mode: "json"` ให้ใช้ตัวชี้ JSON แบบสัมบูรณ์ เช่น `"/providers/xai/apiKey"`

ดูฟิลด์ที่ยอมรับ SecretRef ได้ที่ [พื้นผิวข้อมูลประจำตัว SecretRef](/th/reference/secretref-credential-surface)

    ใช้ตัวห่อหุ้มตัวแก้ค่าขนาดเล็กเพื่อแมปรหัส SecretRef กับรายการ `pass` โดยตรง บันทึกไฟล์นี้เป็นไฟล์ปฏิบัติการในพาธสัมบูรณ์ที่ผ่านการตรวจสอบพาธของผู้ให้บริการ exec เช่น `/usr/local/bin/openclaw-pass-resolver` บรรทัด shebang `#!/usr/bin/env node` จะแก้ค่า `node` จาก `PATH` ของกระบวนการตัวแก้ค่า ดังนั้นให้รวม `PATH` ไว้ใน `passEnv` หาก `pass` ไม่อยู่ใน `PATH` ดังกล่าว ให้ตั้งค่า `PASS_BIN` ในสภาพแวดล้อมแม่และรวมไว้ใน `passEnv` ด้วย:

    ```js
    #!/usr/bin/env node
    const { spawnSync } = require("node:child_process");

    let stdin = "";
    process.stdin.setEncoding("utf8");
    process.stdin.on("data", (chunk) => {
      stdin += chunk;
    });
    process.stdin.on("error", (err) => {
      process.stderr.write(`${err.message}\n`);
      process.exit(1);
    });
    process.stdin.on("end", () => {
      let request;
      try {
        request = JSON.parse(stdin || "{}");
      } catch (err) {
        process.stderr.write(`แยกวิเคราะห์คำขอไม่สำเร็จ: ${err.message}\n`);
        process.exit(1);
      }

      const passBin = process.env.PASS_BIN || "pass";
      const values = {};
      const errors = {};

      for (const id of request.ids ?? []) {
        const result = spawnSync(passBin, ["show", id], { encoding: "utf8" });
        if (result.status === 0) {
          values[id] = result.stdout.split(/\r?\n/, 1)[0] ?? "";
        } else {
          errors[id] = { message: (result.stderr || `pass จบการทำงานด้วย ${result.status}`).trim() };
        }
      }

      process.stdout.write(JSON.stringify({ protocolVersion: 1, values, errors }));
    });
    ```

    จากนั้นกำหนดค่าผู้ให้บริการ exec และชี้ `apiKey` ไปยังพาธรายการ `pass`:

    ```json5
    {
      secrets: {
        providers: {
          pass_store: {
            source: "exec",
            command: "/usr/local/bin/openclaw-pass-resolver",
            passEnv: ["PATH", "HOME", "GNUPGHOME", "GPG_TTY", "PASSWORD_STORE_DIR", "PASS_BIN"],
            jsonOnly: true,
          },
        },
      },
      models: {
        providers: {
          openai: {
            baseUrl: "https://api.openai.com/v1",
            models: [{ id: "gpt-5", name: "gpt-5" }],
            apiKey: {
              source: "exec",
              provider: "pass_store",
              id: "openclaw/providers/openai/apiKey",
            },
          },
        },
      },
    }
    ```

    เก็บข้อมูลลับไว้ในบรรทัดแรกของรายการ `pass` หรือปรับแต่งตัวห่อหุ้มให้ส่งคืนเอาต์พุต `pass show` ทั้งหมดแทน หลังอัปเดตการกำหนดค่า ให้ตรวจสอบทั้งการตรวจสอบแบบสถิตและพาธตัวแก้ค่า exec:

    ```bash
    openclaw secrets audit --check
    openclaw secrets audit --allow-exec
    ```

## ตัวแปรสภาพแวดล้อมของเซิร์ฟเวอร์ MCP

ตัวแปรสภาพแวดล้อมของเซิร์ฟเวอร์ MCP ที่กำหนดค่าผ่าน `plugins.entries.acpx.config.mcpServers` ยอมรับ SecretInput ซึ่งช่วยไม่ให้คีย์ API และโทเค็นอยู่ในการกำหนดค่าแบบข้อความธรรมดา:

```json5
{
  plugins: {
    entries: {
      acpx: {
        enabled: true,
        config: {
          mcpServers: {
            github: {
              command: "npx",
              args: ["-y", "@modelcontextprotocol/server-github"],
              env: {
                GITHUB_PERSONAL_ACCESS_TOKEN: {
                  source: "env",
                  provider: "default",
                  id: "MCP_GITHUB_PAT",
                },
              },
            },
          },
        },
      },
    },
  },
}
```

ค่าสตริงข้อความธรรมดายังคงใช้งานได้ การอ้างอิงเทมเพลตสภาพแวดล้อม เช่น `${MCP_SERVER_API_KEY}` และออบเจ็กต์ SecretRef จะได้รับการแก้ไขระหว่างการเปิดใช้งาน Gateway ก่อนที่กระบวนการเซิร์ฟเวอร์ MCP จะเริ่มทำงาน เช่นเดียวกับพื้นผิว SecretRef อื่น ๆ การอ้างอิงที่แก้ไขไม่ได้จะขัดขวางการเปิดใช้งานเฉพาะเมื่อ Plugin `acpx` ทำงานอยู่จริงเท่านั้น

## ข้อมูลการยืนยันตัวตน SSH สำหรับ sandbox

แบ็กเอนด์ sandbox หลัก `ssh` ยังรองรับ SecretRef สำหรับข้อมูลการยืนยันตัวตน SSH ด้วย:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "ssh",
        ssh: {
          target: "user@gateway-host:22",
          identityData: { source: "env", provider: "default", id: "SSH_IDENTITY" },
          certificateData: { source: "env", provider: "default", id: "SSH_CERTIFICATE" },
          knownHostsData: { source: "env", provider: "default", id: "SSH_KNOWN_HOSTS" },
        },
      },
    },
  },
}
```

พฤติกรรมขณะรันไทม์:

- OpenClaw จะแก้ไขการอ้างอิงเหล่านี้ระหว่างการเปิดใช้งาน sandbox ไม่ใช่แบบหน่วงเวลาในทุกการเรียก SSH
- ค่าที่แก้ไขแล้วจะถูกเขียนลงในไดเรกทอรีชั่วคราวโดยกำหนดสิทธิ์ไฟล์แบบเข้มงวด (`0o600`) และนำไปใช้ในคอนฟิก SSH ที่สร้างขึ้น
- หากแบ็กเอนด์ sandbox ที่มีผลไม่ใช่ `ssh` (หรือโหมด sandbox เป็น `off`) การอ้างอิงเหล่านี้จะยังไม่ทำงานและไม่ขัดขวางการเริ่มต้นระบบ

## พื้นผิวข้อมูลประจำตัวที่รองรับ

รายการข้อมูลประจำตัวที่รองรับและไม่รองรับอย่างเป็นทางการอยู่ใน [พื้นผิวข้อมูลประจำตัว SecretRef](/th/reference/secretref-credential-surface)

ข้อมูลประจำตัวที่สร้างขึ้นขณะรันไทม์หรือมีการหมุนเวียน และข้อมูลสำหรับรีเฟรช OAuth ถูกจงใจยกเว้นจากการแก้ไข SecretRef แบบอ่านอย่างเดียว
</Note>

## พฤติกรรมและลำดับความสำคัญที่กำหนด

- ฟิลด์ที่ไม่มีการอ้างอิง: ไม่เปลี่ยนแปลง
- ฟิลด์ที่มีการอ้างอิง: จำเป็นสำหรับพื้นผิวที่ทำงานอยู่ระหว่างการเปิดใช้งาน
- หากมีทั้งข้อความธรรมดาและการอ้างอิง การอ้างอิงจะมีลำดับความสำคัญเหนือกว่าในเส้นทางลำดับความสำคัญที่รองรับ
- ค่าเซนทิเนลสำหรับการปกปิดข้อมูล `__OPENCLAW_REDACTED__` สงวนไว้สำหรับการปกปิด/คืนค่าคอนฟิกภายใน และจะถูกปฏิเสธหากส่งเป็นข้อมูลคอนฟิกแบบค่าตรงตัว

สัญญาณเตือนและการตรวจสอบ:

- `SECRETS_REF_OVERRIDES_PLAINTEXT` (คำเตือนขณะรันไทม์)
- `REF_SHADOWED` (ผลการตรวจสอบเมื่อข้อมูลประจำตัว `auth-profiles.json` มีลำดับความสำคัญเหนือกว่าการอ้างอิง `openclaw.json`)

ความเข้ากันได้กับ Google Chat: `serviceAccountRef` มีลำดับความสำคัญเหนือกว่า `serviceAccount` แบบข้อความธรรมดา โดยค่าข้อความธรรมดาจะถูกละเว้นเมื่อกำหนดการอ้างอิงที่อยู่คู่กันแล้ว

## ทริกเกอร์การเปิดใช้งาน

การเปิดใช้งานข้อมูลลับจะทำงานเมื่อ:

- เริ่มต้นระบบ (การตรวจสอบล่วงหน้าและการเปิดใช้งานขั้นสุดท้าย)
- เส้นทางใช้คอนฟิกใหม่แบบทันทีเมื่อโหลดคอนฟิกซ้ำ
- เส้นทางตรวจสอบการเริ่มระบบใหม่เมื่อโหลดคอนฟิกซ้ำ
- โหลดซ้ำด้วยตนเองผ่าน `secrets.reload`
- การตรวจสอบล่วงหน้าของ RPC สำหรับเขียนคอนฟิก Gateway (`config.set` / `config.apply` / `config.patch`) ซึ่งตรวจสอบ SecretRef ของพื้นผิวที่ทำงานอยู่ภายในเพย์โหลดคอนฟิกที่ส่งมา ก่อนบันทึกการแก้ไขอย่างถาวร

สัญญาการเปิดใช้งาน:

- เมื่อสำเร็จ จะสลับสแนปช็อตแบบอะตอมมิก
- ความล้มเหลวแบบเข้มงวดขณะเริ่มต้นระบบจะยกเลิกการเริ่มต้น Gateway
- ระหว่างการเริ่มต้นแบบเย็น หากการแก้ไขล้มเหลวแบบลองใหม่ได้สำหรับเจ้าของที่ไม่ใช่ Gateway ซึ่งมีการแมปและแยกออกได้ ระบบอาจเผยแพร่สแนปช็อตโดยกำหนดให้เจ้าของรายนั้นไม่พร้อมใช้งานตามคอนฟิก คำขอสำหรับเจ้าของจะล้มเหลวด้วย `SECRET_SURFACE_UNAVAILABLE`; เจ้าของผู้ให้บริการโมเดลจะไม่ย้อนกลับไปใช้ข้อมูลประจำตัวจากสภาพแวดล้อมหรือโปรไฟล์การยืนยันตัวตน หลังจากการอ้างอิงที่ระบุไว้อย่างชัดเจนล้มเหลว
- การโหลดซ้ำและการตรวจสอบการเริ่มระบบใหม่จะแยกเจ้าของที่มีสิทธิ์และมีการแมปออกจากกัน เอกลักษณ์ของการอ้างอิงที่ไม่เปลี่ยนแปลง พร้อมคำจำกัดความผู้ให้บริการที่ไม่เปลี่ยนแปลงและสัญญาเจ้าของที่สมบูรณ์ซึ่งไม่เป็นข้อมูลลับที่ไม่เปลี่ยนแปลง จะคงค่าล่าสุดที่ทราบว่าใช้งานได้เดิมไว้อย่างถูกต้องในสถานะเก่า; การอ้างอิงที่เปลี่ยนแปลงหรือกำหนดค่าใหม่แต่แก้ไขไม่ได้ จะเผยแพร่สถานะเย็นเฉพาะสำหรับเจ้าของรายนั้น ความล้มเหลวแบบเข้มงวดในการโหลดซ้ำจะรักษาสแนปช็อตที่ทำงานอยู่ก่อนหน้าไว้
- `config.set`, `config.apply` และ `config.patch` ยอมรับการอ้างอิงที่แก้ไขไม่ได้แต่ถูกต้องตามไวยากรณ์สำหรับเจ้าของที่แยกออกได้ และส่งคืนรายงาน `degradedSecretOwners` ที่ปกปิดข้อมูลแล้ว การยืนยันตัวตนขาเข้าของ Gateway, คอนฟิกหรือค่าที่แก้ไขแล้วซึ่งไม่ถูกต้องตามโครงสร้าง, การละเมิดนโยบาย และเจ้าของที่ไม่รู้จัก จะยังคงถูกปฏิเสธก่อนแก้ไขข้อมูลบนดิสก์
- เจ้าของอื่นที่อยู่ในสถานะปกติจะได้รับการแก้ไขและเผยแพร่ตามปกติ แม้เจ้าของรายอื่นจะอยู่ในสถานะเย็นหรือเก่า
- การระบุโทเค็นช่องทางอย่างชัดเจนต่อการเรียกหนึ่งครั้งให้กับการเรียกตัวช่วย/เครื่องมือขาออก จะไม่ทริกเกอร์การเปิดใช้งาน SecretRef; จุดเปิดใช้งานยังคงเป็นการเริ่มต้นระบบ การโหลดซ้ำ และ `secrets.reload` ที่ระบุอย่างชัดเจน

## สัญญาณสถานะเสื่อมและกู้คืนแล้ว

เมื่อการเปิดใช้งานขณะโหลดซ้ำล้มเหลวหลังจากอยู่ในสถานะปกติ OpenClaw จะเข้าสู่สถานะข้อมูลลับเสื่อม พร้อมส่งเหตุการณ์ระบบครั้งเดียวและรหัสบันทึก:

- `SECRETS_RELOADER_DEGRADED`
- `SECRETS_RELOADER_RECOVERED`

พฤติกรรม:

- สถานะเสื่อม: เจ้าของที่ปกติจะรีเฟรช เจ้าของที่เก่าจะเก็บค่าล่าสุดที่ทราบว่าใช้งานได้ และเจ้าของที่เย็นจะยังคงไม่พร้อมใช้งาน
- กู้คืนแล้ว: ส่งหนึ่งครั้งหลังจากการเปิดใช้งานครั้งถัดไปสำเร็จ
- ความล้มเหลวซ้ำขณะที่อยู่ในสถานะเสื่อมแล้วจะบันทึกคำเตือน แต่ไม่ส่งเหตุการณ์ซ้ำ
- ความล้มเหลวแบบเข้มงวดขณะเริ่มต้นระบบจะไม่ส่งเหตุการณ์สถานะเสื่อม เนื่องจากรันไทม์ไม่เคยทำงาน การเริ่มต้นระบบที่สำเร็จโดยมีเจ้าของสถานะเย็นจะบันทึกการเสื่อมของเจ้าของ แต่ไม่ส่งเหตุการณ์ของตัวโหลดซ้ำ
- ความล้มเหลวขณะเริ่มต้นระบบและโหลดซ้ำที่จำกัดขอบเขตตามการอ้างอิง จะส่งคำเตือน `SECRETS_DEGRADED` แบบมีโครงสร้างสำหรับเจ้าของแต่ละรายที่ได้รับผลกระทบ การหยุดให้บริการที่จำกัดขอบเขตตามผู้ให้บริการจะส่งคำเตือน `SECRETS_PROVIDER_DEGRADED` หนึ่งรายการ พร้อมผู้ให้บริการและรายชื่อเจ้าของที่ได้รับผลกระทบทั้งหมด แทนการแจ้งความล้มเหลวของผู้ให้บริการซ้ำสำหรับเจ้าของแต่ละราย คำเตือนประกอบด้วยเหตุผลที่ปกปิดข้อมูลแล้ว สถานะเจ้าของ `cold` หรือ `stale` และคำแนะนำให้ลองใหม่ `openclaw secrets reload` โดยจะไม่มีค่าที่แก้ไขแล้วหรือรหัส SecretRef
- `openclaw doctor` แสดงรายการเจ้าของสถานะเย็นและเก่า พร้อมเส้นทางคอนฟิกที่ได้รับผลกระทบ เหตุผลที่ปกปิดข้อมูลแล้ว และคำแนะนำในการลองใหม่

  <Tab title="เส้นทางคำสั่งแบบเข้มงวด">

  </Tab>
  <Tab title="เส้นทางคำสั่งแบบอ่านอย่างเดียว">
    ตัวอย่างเช่น `openclaw status`, `openclaw status --all`, `openclaw channels status`, `openclaw channels resolve`, `openclaw security audit` และโฟลว์ doctor/ซ่อมแซมคอนฟิกแบบอ่านอย่างเดียว เส้นทางเหล่านี้จะเลือกใช้สแนปช็อตที่ทำงานอยู่ก่อนเช่นกัน แต่จะลดระดับการทำงานแทนการยกเลิกเมื่อ SecretRef เป้าหมายไม่พร้อมใช้งาน

    พฤติกรรมแบบอ่านอย่างเดียว:

    - เมื่อ Gateway กำลังทำงาน คำสั่งเหล่านี้จะอ่านจากสแนปช็อตที่ทำงานอยู่ก่อน
    - หากการแก้ไขผ่าน Gateway ไม่สมบูรณ์หรือ Gateway ไม่พร้อมใช้งาน ระบบจะพยายามใช้ทางเลือกสำรองเฉพาะเป้าหมายในเครื่องสำหรับพื้นผิวคำสั่งนั้น
    - หาก SecretRef เป้าหมายยังคงไม่พร้อมใช้งาน คำสั่งจะทำงานต่อด้วยเอาต์พุตแบบอ่านอย่างเดียวที่ลดระดับ และการวินิจฉัยที่ระบุชัดว่ามีการกำหนดค่าการอ้างอิงไว้ แต่ไม่พร้อมใช้งานในเส้นทางคำสั่งนี้
    - พฤติกรรมที่ลดระดับนี้จำกัดอยู่เฉพาะคำสั่งเท่านั้น และไม่ลดความเข้มงวดของการเริ่มต้นรันไทม์ การโหลดซ้ำ หรือเส้นทางการส่ง/การยืนยันตัวตน

  </Tab>
</Tabs>

หมายเหตุอื่น ๆ:

- การรีเฟรชสแนปช็อตหลังการหมุนเวียนข้อมูลลับของแบ็กเอนด์ดำเนินการโดย `openclaw secrets reload`
- เมธอด RPC ของ Gateway ที่เส้นทางคำสั่งเหล่านี้ใช้: `secrets.resolve`

## เวิร์กโฟลว์การตรวจสอบและกำหนดค่า

โฟลว์เริ่มต้นสำหรับผู้ดำเนินการ:

<Steps>
  <Step title="ตรวจสอบสถานะปัจจุบัน">
    ```bash
    openclaw secrets audit --check
    ```
  </Step>
  <Step title="กำหนดค่าและใช้ SecretRef">
    ```bash
    openclaw secrets configure --apply
    ```
  </Step>
  <Step title="ตรวจสอบซ้ำ">
    ```bash
    openclaw secrets audit --check
    ```
  </Step>
</Steps>

อย่าถือว่าการย้ายข้อมูลเสร็จสมบูรณ์จนกว่าการตรวจสอบซ้ำจะไม่มีปัญหา หากการตรวจสอบยังคงรายงานค่าข้อความธรรมดาที่จัดเก็บอยู่ ความเสี่ยงที่เอเจนต์จะเข้าถึงข้อมูลยังคงมีอยู่ แม้ API ขณะรันไทม์จะส่งคืนค่าที่ปกปิดข้อมูลแล้วก็ตาม

หากบันทึกแผนแทนการนำไปใช้ระหว่าง `configure` ให้นำแผนที่บันทึกไว้นั้นไปใช้ด้วย `openclaw secrets apply --from <plan-path>` ก่อนการตรวจสอบซ้ำ

<AccordionGroup>
  <Accordion title="การตรวจสอบข้อมูลลับ">
    ผลการตรวจสอบประกอบด้วย:

    - ค่าข้อความธรรมดาที่จัดเก็บอยู่ (`openclaw.json`, `auth-profiles.json`, `.env` และ `agents/*/agent/models.json` ที่สร้างขึ้น)
    - ข้อมูลตกค้างของส่วนหัวผู้ให้บริการที่ละเอียดอ่อนแบบข้อความธรรมดาในรายการ `models.json` ที่สร้างขึ้น
    - การอ้างอิงที่แก้ไขไม่ได้
    - การบดบังตามลำดับความสำคัญ (`auth-profiles.json` มีลำดับความสำคัญเหนือกว่าการอ้างอิง `openclaw.json`)
    - ข้อมูลตกค้างจากระบบเดิม (`auth.json`, การแจ้งเตือน OAuth)

    หมายเหตุเกี่ยวกับ exec: โดยค่าเริ่มต้น การตรวจสอบจะข้ามการตรวจสอบว่า SecretRef แบบ exec แก้ไขได้หรือไม่ เพื่อหลีกเลี่ยงผลข้างเคียงจากคำสั่ง ใช้ `openclaw secrets audit --allow-exec` เพื่อเรียกใช้ผู้ให้บริการ exec ระหว่างการตรวจสอบ

    หมายเหตุเกี่ยวกับข้อมูลตกค้างในส่วนหัว: การตรวจหาส่วนหัวผู้ให้บริการที่ละเอียดอ่อนอาศัยฮิวริสติกจากชื่อ (ชื่อส่วนหัวและส่วนย่อยที่ใช้ทั่วไปสำหรับการยืนยันตัวตน/ข้อมูลประจำตัว เช่น `authorization`, `x-api-key`, `token`, `secret`, `password` และ `credential`)

  </Accordion>
  <Accordion title="กำหนดค่าข้อมูลลับ">
    ตัวช่วยแบบโต้ตอบที่:

    - กำหนดค่า `secrets.providers` ก่อน (`env`/`file`/`exec`, เพิ่ม/แก้ไข/ลบ)
    - ช่วยให้เลือกฟิลด์ที่รองรับและมีข้อมูลลับใน `openclaw.json` รวมถึง `auth-profiles.json` สำหรับขอบเขตเอเจนต์หนึ่งรายการ
    - สามารถสร้างการแมป `auth-profiles.json` ใหม่โดยตรงในตัวเลือกเป้าหมาย
    - เก็บรายละเอียด SecretRef (`source`, `provider`, `id`)
    - เรียกใช้การแก้ไขล่วงหน้าและสามารถนำไปใช้ได้ทันที

    หมายเหตุเกี่ยวกับ exec: การตรวจสอบล่วงหน้าจะข้ามการตรวจ SecretRef แบบ exec เว้นแต่จะกำหนด `--allow-exec` หากนำไปใช้โดยตรงจาก `configure --apply` และแผนมีการอ้างอิง/ผู้ให้บริการแบบ exec ให้คงการกำหนด `--allow-exec` ไว้สำหรับขั้นตอนนำไปใช้ด้วย

    โหมดที่เป็นประโยชน์:

    - `openclaw secrets configure --providers-only`
    - `openclaw secrets configure --skip-provider-setup`
    - `openclaw secrets configure --agent <id>`

    ค่าเริ่มต้นในการนำ `configure` ไปใช้:

    - ล้างข้อมูลประจำตัวแบบคงที่ที่ตรงกันออกจาก `auth-profiles.json` สำหรับผู้ให้บริการเป้าหมาย
    - ล้างรายการ `api_key` แบบคงที่จากระบบเดิมออกจาก `auth.json`
    - ล้างบรรทัดข้อมูลลับที่รู้จักและตรงกันออกจากไฟล์ `.env` ของสถานะที่มีผลและคอนฟิกที่ทำงานอยู่ (กำจัดรายการซ้ำเมื่อทั้งสองเส้นทางตรงกัน)

  </Accordion>
  <Accordion title="นำข้อมูลลับไปใช้">
    นำแผนที่บันทึกไว้ไปใช้:

    ```bash
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --allow-exec
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run
    openclaw secrets apply --from /tmp/openclaw-secrets-plan.json --dry-run --allow-exec
    ```

    หมายเหตุเกี่ยวกับ exec: การทดลองทำงานจะข้ามการตรวจ exec เว้นแต่จะกำหนด `--allow-exec`; โหมดเขียนจะปฏิเสธแผนที่มี SecretRef/ผู้ให้บริการแบบ exec เว้นแต่จะกำหนด `--allow-exec`

    สำหรับรายละเอียดสัญญาเป้าหมาย/เส้นทางแบบเข้มงวดและกฎการปฏิเสธที่แน่นอน โปรดดู [สัญญาแผนการนำข้อมูลลับไปใช้](/th/gateway/secrets-plan-contract)

  </Accordion>
</AccordionGroup>

## นโยบายความปลอดภัยแบบทางเดียว

<Warning>
OpenClaw จงใจไม่เขียนข้อมูลสำรองสำหรับย้อนกลับที่มีค่าข้อมูลลับแบบข้อความธรรมดาในอดีต
</Warning>

โมเดลความปลอดภัย:

- การตรวจสอบล่วงหน้าต้องสำเร็จก่อนเข้าสู่โหมดเขียน
- ต้องตรวจสอบความถูกต้องของการเปิดใช้งานรันไทม์ก่อนคอมมิต
- การนำไปใช้จะอัปเดตไฟล์ด้วยการแทนที่ไฟล์แบบอะตอมมิก และพยายามคืนค่าอย่างเต็มความสามารถเมื่อเกิดความล้มเหลว

## หมายเหตุความเข้ากันได้กับการยืนยันตัวตนแบบเดิม

สำหรับข้อมูลประจำตัวแบบคงที่ รันไทม์ไม่ต้องพึ่งพาที่จัดเก็บการยืนยันตัวตนแบบข้อความธรรมดาจากระบบเดิมอีกต่อไป

- แหล่งที่มาของข้อมูลรับรองขณะรันไทม์คือสแนปชอตในหน่วยความจำที่ผ่านการแก้ไขแล้ว
- รายการ `api_key` แบบคงที่รุ่นเก่าจะถูกลบข้อมูลเมื่อมีการตรวจพบ
- ลักษณะการทำงานเพื่อความเข้ากันได้ที่เกี่ยวข้องกับ OAuth ยังคงแยกจากกัน

## หมายเหตุเกี่ยวกับ Web UI
