ในการจับคู่ที่ Gateway เป็นเจ้าของ **Gateway** คือแหล่งข้อมูลจริงสำหรับกำหนดว่า Node ใด
ได้รับอนุญาตให้เข้าร่วม UI (แอป macOS, ไคลเอนต์ในอนาคต) เป็นเพียงส่วนหน้า
สำหรับอนุมัติหรือปฏิเสธคำขอที่รอดำเนินการ

**สำคัญ:** Node ของ WS ใช้ **การจับคู่อุปกรณ์** (บทบาท `node`) ระหว่าง `connect`
`node.pair.*` เป็นพื้นที่จัดเก็บการจับคู่แยกต่างหาก และ **ไม่ได้** ควบคุมการแฮนด์เชก WS
เฉพาะไคลเอนต์ที่เรียก `node.pair.*` โดยตรงเท่านั้นที่ใช้โฟลว์นี้

## แนวคิด

- **คำขอที่รอดำเนินการ**: Node ขอเข้าร่วม ต้องได้รับการอนุมัติ
- **Node ที่จับคู่แล้ว**: Node ที่ได้รับอนุมัติพร้อมโทเค็นรับรองความถูกต้องที่ออกให้แล้ว
- **ทรานสปอร์ต**: จุดปลาย WS ของ Gateway ส่งต่อคำขอ แต่ไม่ได้ตัดสิน
  สมาชิกภาพ (การรองรับบริดจ์ TCP แบบเก่าถูกลบออกแล้ว)

## การจับคู่ทำงานอย่างไร

1. Node เชื่อมต่อกับ WS ของ Gateway และขอจับคู่
2. Gateway เก็บ **คำขอที่รอดำเนินการ** และส่งอีเวนต์ `node.pair.requested`
3. คุณอนุมัติหรือปฏิเสธคำขอ (CLI หรือ UI)
4. เมื่ออนุมัติ Gateway จะออก **โทเค็นใหม่** (โทเค็นจะถูกหมุนเวียนเมื่อจับคู่ใหม่)
5. Node เชื่อมต่อใหม่โดยใช้โทเค็น และขณะนี้ “จับคู่แล้ว”

คำขอที่รอดำเนินการจะหมดอายุโดยอัตโนมัติหลังจาก **5 นาที**

## เวิร์กโฟลว์ CLI (เหมาะกับระบบไร้หน้าจอ)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` แสดง Node ที่จับคู่/เชื่อมต่อแล้วและความสามารถของแต่ละ Node

## พื้นผิว API (โปรโตคอล Gateway)

อีเวนต์:

- `node.pair.requested` — ส่งเมื่อมีการสร้างคำขอที่รอดำเนินการใหม่
- `node.pair.resolved` — ส่งเมื่อคำขอได้รับการอนุมัติ/ปฏิเสธ/หมดอายุ

เมธอด:

- `node.pair.request` — สร้างหรือนำคำขอที่รอดำเนินการกลับมาใช้ซ้ำ
- `node.pair.list` — แสดงรายการ Node ที่รอดำเนินการ + จับคู่แล้ว (`operator.pairing`)
- `node.pair.approve` — อนุมัติคำขอที่รอดำเนินการ (ออกโทเค็น)
- `node.pair.reject` — ปฏิเสธคำขอที่รอดำเนินการ
- `node.pair.remove` — ลบรายการ Node ที่จับคู่แล้วซึ่งค้างอยู่
- `node.pair.verify` — ตรวจสอบ `{ nodeId, token }`

หมายเหตุ:

- `node.pair.request` เป็น idempotent ต่อ Node: การเรียกซ้ำจะคืนค่า
  คำขอที่รอดำเนินการเดียวกัน
- คำขอซ้ำสำหรับ Node ที่รอดำเนินการเดียวกันยังรีเฟรชเมทาดาทาของ Node ที่จัดเก็บไว้
  และสแนปชอตคำสั่งที่ประกาศล่าสุดซึ่งอยู่ในรายการอนุญาต เพื่อให้โอเปอเรเตอร์มองเห็น
- การอนุมัติจะสร้างโทเค็นใหม่ **เสมอ** ไม่มีการคืนค่าโทเค็นจาก
  `node.pair.request` เลย
- คำขออาจมี `silent: true` เป็นคำใบ้สำหรับโฟลว์อนุมัติอัตโนมัติ
- `node.pair.approve` ใช้คำสั่งที่ประกาศในคำขอที่รอดำเนินการเพื่อบังคับใช้
  ขอบเขตการอนุมัติเพิ่มเติม:
  - คำขอที่ไม่มีคำสั่ง: `operator.pairing`
  - คำขอคำสั่งที่ไม่ใช่ exec: `operator.pairing` + `operator.write`
  - คำขอ `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
การจับคู่ Node เป็นโฟลว์ความเชื่อถือและตัวตน พร้อมการออกโทเค็น แต่ **ไม่ได้** ตรึงพื้นผิวคำสั่งสดของ Node ต่อ Node

- คำสั่งสดของ Node มาจากสิ่งที่ Node ประกาศเมื่อเชื่อมต่อ หลังจากมีการใช้นโยบายคำสั่ง Node ส่วนกลางของ Gateway (`gateway.nodes.allowCommands` และ `denyCommands`)
- นโยบายอนุญาตและถามสำหรับ `system.run` ต่อ Node อยู่บน Node ใน `exec.approvals.node.*` ไม่ได้อยู่ในบันทึกการจับคู่

</Warning>

## การควบคุมคำสั่ง Node (2026.3.31+)

<Warning>
**การเปลี่ยนแปลงที่ทำให้เข้ากันไม่ได้:** ตั้งแต่ `2026.3.31` เป็นต้นไป คำสั่ง Node จะถูกปิดใช้งานจนกว่าการจับคู่ Node จะได้รับการอนุมัติ การจับคู่อุปกรณ์เพียงอย่างเดียวไม่เพียงพออีกต่อไปสำหรับเปิดเผยคำสั่ง Node ที่ประกาศไว้
</Warning>

เมื่อ Node เชื่อมต่อเป็นครั้งแรก ระบบจะขอการจับคู่โดยอัตโนมัติ จนกว่าคำขอจับคู่จะได้รับการอนุมัติ คำสั่ง Node ที่รอดำเนินการทั้งหมดจาก Node นั้นจะถูกกรองและจะไม่ถูกดำเนินการ เมื่อสร้างความเชื่อถือผ่านการอนุมัติการจับคู่แล้ว คำสั่งที่ Node ประกาศไว้จะพร้อมใช้งานตามนโยบายคำสั่งปกติ

ซึ่งหมายความว่า:

- Node ที่ก่อนหน้านี้พึ่งพาเฉพาะการจับคู่อุปกรณ์เพื่อเปิดเผยคำสั่ง ต้องทำการจับคู่ Node ให้เสร็จสมบูรณ์ในตอนนี้
- คำสั่งที่เข้าคิวไว้ก่อนการอนุมัติการจับคู่จะถูกทิ้ง ไม่ได้ถูกเลื่อนเวลาไว้

## ขอบเขตความเชื่อถือของอีเวนต์ Node (2026.3.31+)

<Warning>
**การเปลี่ยนแปลงที่ทำให้เข้ากันไม่ได้:** การรันที่มีต้นทางจาก Node จะอยู่บนพื้นผิวที่เชื่อถือได้แบบลดขอบเขตแล้ว
</Warning>

สรุปที่มีต้นทางจาก Node และอีเวนต์เซสชันที่เกี่ยวข้องถูกจำกัดไว้ที่พื้นผิวที่เชื่อถือได้ตามเจตนา โฟลว์ที่ขับเคลื่อนด้วยการแจ้งเตือนหรือถูกทริกเกอร์โดย Node ซึ่งก่อนหน้านี้พึ่งพาการเข้าถึงเครื่องโฮสต์หรือเครื่องมือเซสชันที่กว้างกว่า อาจต้องปรับเปลี่ยน การเสริมความแข็งแรงนี้ช่วยให้แน่ใจว่าอีเวนต์ Node ไม่สามารถยกระดับไปสู่การเข้าถึงเครื่องมือระดับโฮสต์เกินกว่าที่ขอบเขตความเชื่อถือของ Node อนุญาต

การอัปเดตสถานะการปรากฏของ Node แบบคงทนใช้ขอบเขตตัวตนเดียวกัน อีเวนต์ `node.presence.alive`
จะถูกยอมรับเฉพาะจากเซสชันอุปกรณ์ Node ที่ผ่านการรับรองความถูกต้อง และจะอัปเดตเมทาดาทาการจับคู่เฉพาะเมื่อ
ตัวตนอุปกรณ์/Node จับคู่แล้วเท่านั้น ค่า `client.id` ที่ประกาศเองไม่เพียงพอสำหรับเขียน
สถานะที่พบล่าสุด

## การอนุมัติอัตโนมัติ (แอป macOS)

แอป macOS สามารถพยายามทำ **การอนุมัติแบบเงียบ** ได้ตามตัวเลือก เมื่อ:

- คำขอถูกทำเครื่องหมายเป็น `silent` และ
- แอปสามารถตรวจสอบการเชื่อมต่อ SSH ไปยังโฮสต์ Gateway โดยใช้ผู้ใช้เดียวกันได้

หากการอนุมัติแบบเงียบล้มเหลว ระบบจะย้อนกลับไปใช้พรอมป์ “อนุมัติ/ปฏิเสธ” ตามปกติ

## การอนุมัติอุปกรณ์อัตโนมัติด้วย CIDR ที่เชื่อถือได้

การจับคู่อุปกรณ์ WS สำหรับ `role: node` ยังคงเป็นแบบดำเนินการด้วยตนเองตามค่าเริ่มต้น สำหรับเครือข่าย
Node ส่วนตัวที่ Gateway เชื่อถือเส้นทางเครือข่ายอยู่แล้ว โอเปอเรเตอร์สามารถ
เลือกใช้ด้วย CIDR ที่ระบุชัดเจนหรือ IP ตรงได้:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ขอบเขตความปลอดภัย:

- ปิดใช้งานเมื่อไม่ได้ตั้งค่า `gateway.nodes.pairing.autoApproveCidrs`
- ไม่มีโหมดอนุมัติอัตโนมัติแบบครอบคลุม LAN หรือเครือข่ายส่วนตัวทั้งหมด
- เฉพาะการจับคู่อุปกรณ์ `role: node` ใหม่ที่ไม่มีขอบเขตที่ร้องขอเท่านั้นที่เข้าเกณฑ์
- ไคลเอนต์โอเปอเรเตอร์ เบราว์เซอร์ Control UI และ WebChat ยังคงเป็นแบบดำเนินการด้วยตนเอง
- การอัปเกรดบทบาท ขอบเขต เมทาดาทา และกุญแจสาธารณะยังคงเป็นแบบดำเนินการด้วยตนเอง
- เส้นทางส่วนหัวพร็อกซีที่เชื่อถือได้แบบ local loopback บนโฮสต์เดียวกันไม่เข้าเกณฑ์ เพราะเส้นทางนั้น
  อาจถูกปลอมแปลงโดยผู้เรียกในเครื่องได้

## การอนุมัติอัตโนมัติสำหรับการอัปเกรดเมทาดาทา

เมื่ออุปกรณ์ที่จับคู่แล้วเชื่อมต่อใหม่โดยมีเฉพาะการเปลี่ยนแปลงเมทาดาทาที่ไม่ละเอียดอ่อน
(เช่น ชื่อที่แสดงหรือคำใบ้แพลตฟอร์มของไคลเอนต์) OpenClaw จะถือว่า
เป็น `metadata-upgrade` การอนุมัติอัตโนมัติแบบเงียบมีขอบเขตแคบ: ใช้เฉพาะ
กับการเชื่อมต่อใหม่ในเครื่องที่เชื่อถือได้และไม่ใช่เบราว์เซอร์ ซึ่งพิสูจน์การครอบครองข้อมูลรับรองในเครื่อง
หรือข้อมูลรับรองที่ใช้ร่วมกันแล้ว รวมถึงการเชื่อมต่อใหม่จากแอปเนทีฟบนโฮสต์เดียวกันหลังจากมีการเปลี่ยนแปลง
เมทาดาทาเวอร์ชัน OS ไคลเอนต์เบราว์เซอร์/Control UI และไคลเอนต์ระยะไกลยังคง
ใช้โฟลว์อนุมัติใหม่แบบชัดเจน การอัปเกรดขอบเขต (อ่านเป็นเขียน/admin) และ
การเปลี่ยนกุญแจสาธารณะ **ไม่** เข้าเกณฑ์สำหรับการอนุมัติอัตโนมัติแบบอัปเกรดเมทาดาทา —
รายการเหล่านี้ยังคงเป็นคำขออนุมัติใหม่แบบชัดเจน

## ตัวช่วยจับคู่ QR

`/pair qr` แสดงเพย์โหลดการจับคู่เป็นสื่อแบบมีโครงสร้าง เพื่อให้ไคลเอนต์มือถือและ
เบราว์เซอร์สามารถสแกนได้โดยตรง

การลบอุปกรณ์ยังล้างคำขอจับคู่ที่รอดำเนินการซึ่งค้างอยู่สำหรับ
รหัสอุปกรณ์นั้น ดังนั้น `nodes pending` จะไม่แสดงแถวที่หลงเหลือหลังจากเพิกถอน

## ความเป็นท้องถิ่นและส่วนหัวที่ส่งต่อ

การจับคู่ของ Gateway จะถือว่าการเชื่อมต่อเป็น loopback เฉพาะเมื่อทั้งซ็อกเก็ตดิบ
และหลักฐานพร็อกซีต้นทางเห็นตรงกัน หากคำขอมาถึงบน loopback แต่
มีส่วนหัว `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`
ที่ชี้ไปยังต้นทางที่ไม่ใช่ในเครื่อง หลักฐานจากส่วนหัวที่ส่งต่อนั้นจะตัดสิทธิ์
การอ้างความเป็นท้องถิ่นแบบ loopback จากนั้นเส้นทางการจับคู่จะต้องได้รับการอนุมัติอย่างชัดเจน
แทนที่จะถือว่าคำขอเป็นการเชื่อมต่อจากโฮสต์เดียวกันแบบเงียบ ดู
[การรับรองความถูกต้องของพร็อกซีที่เชื่อถือได้](/th/gateway/trusted-proxy-auth) สำหรับกฎเทียบเท่าบน
การรับรองความถูกต้องของโอเปอเรเตอร์

## พื้นที่จัดเก็บ (ในเครื่อง, ส่วนตัว)

สถานะการจับคู่ถูกเก็บไว้ใต้ไดเรกทอรีสถานะ Gateway (ค่าเริ่มต้น `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

หากคุณแทนที่ `OPENCLAW_STATE_DIR` โฟลเดอร์ `nodes/` จะย้ายไปพร้อมกัน

หมายเหตุด้านความปลอดภัย:

- โทเค็นเป็นความลับ ให้ถือว่า `paired.json` เป็นข้อมูลละเอียดอ่อน
- การหมุนเวียนโทเค็นต้องมีการอนุมัติใหม่ (หรือลบรายการ Node)

## พฤติกรรมทรานสปอร์ต

- ทรานสปอร์ตเป็นแบบ **ไร้สถานะ** ไม่ได้จัดเก็บสมาชิกภาพ
- หาก Gateway ออฟไลน์หรือการจับคู่ถูกปิดใช้งาน Node จะจับคู่ไม่ได้
- หาก Gateway อยู่ในโหมดระยะไกล การจับคู่ยังคงเกิดขึ้นกับพื้นที่จัดเก็บของ Gateway ระยะไกล
