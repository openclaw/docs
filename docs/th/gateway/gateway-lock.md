## เหตุผล

- ตรวจสอบให้แน่ใจว่ามี Gateway เพียงอินสแตนซ์เดียวที่ทำงานต่อพอร์ตฐานหนึ่งพอร์ตบนโฮสต์เดียวกัน; Gateway เพิ่มเติมต้องใช้โปรไฟล์ที่แยกกันและพอร์ตที่ไม่ซ้ำกัน
- ทนต่อการแครช/SIGKILL โดยไม่ทิ้งไฟล์ล็อกค้างไว้
- ล้มเหลวอย่างรวดเร็วพร้อมข้อผิดพลาดที่ชัดเจนเมื่อพอร์ตควบคุมถูกใช้งานอยู่แล้ว

## กลไก

- Gateway จะรับไฟล์ล็อกต่อคอนฟิกหนึ่งไฟล์ภายใต้ไดเรกทอรีล็อกสถานะก่อน แล้วตรวจสอบพอร์ตที่กำหนดค่าไว้เพื่อหาตัวรับฟังที่มีอยู่
- หากเจ้าของล็อกที่บันทึกไว้หายไป, พอร์ตว่าง, หรือล็อกเก่าเกินไป การเริ่มต้นจะยึดล็อกคืนและดำเนินการต่อ
- จากนั้น Gateway จะผูกตัวรับฟัง HTTP/WebSocket (ค่าเริ่มต้น `ws://127.0.0.1:18789`) โดยใช้ตัวรับฟัง TCP แบบเอกสิทธิ์
- หากการผูกล้มเหลวด้วย `EADDRINUSE` การเริ่มต้นจะโยน `GatewayLockError("another gateway instance is already listening on ws://127.0.0.1:<port>")`
- เมื่อปิดการทำงาน Gateway จะปิดเซิร์ฟเวอร์ HTTP/WebSocket และลบไฟล์ล็อก

## พื้นผิวข้อผิดพลาด

- หากกระบวนการอื่นถือพอร์ตไว้ การเริ่มต้นจะโยน `GatewayLockError("another gateway instance is already listening on ws://127.0.0.1:<port>")`
- ความล้มเหลวอื่นๆ ในการผูกจะแสดงเป็น `GatewayLockError("failed to bind gateway socket on ws://127.0.0.1:<port>: …")`

## หมายเหตุการปฏิบัติงาน

- หากพอร์ตถูกใช้งานโดยกระบวนการ_อื่น_ ข้อผิดพลาดจะเหมือนกัน; ให้ปล่อยพอร์ตหรือเลือกพอร์ตอื่นด้วย `openclaw gateway --port <port>`
- ภายใต้ตัวควบคุมบริการ กระบวนการ Gateway ใหม่ที่พบตัวตอบกลับ `/healthz` ที่สมบูรณ์อยู่แล้วจะออกจากการทำงานสำเร็จและปล่อยให้กระบวนการนั้นควบคุมต่อไป หากกระบวนการที่มีอยู่ไม่เคยอยู่ในสถานะสมบูรณ์ การลองใหม่จะถูกจำกัดจำนวน และการเริ่มต้นจะล้มเหลวด้วยข้อผิดพลาดล็อกที่ชัดเจนแทนที่จะวนซ้ำไม่รู้จบ
- แอป macOS ยังคงรักษาตัวป้องกัน PID น้ำหนักเบาของตัวเองก่อนสปอว์น Gateway; ล็อกรันไทม์ถูกบังคับใช้โดยไฟล์ล็อกพร้อมกับการผูก HTTP/WebSocket
