ขอบเขตของผู้ปฏิบัติการควบคุมสิ่งที่ไคลเอนต์ Gateway สามารถทำได้หลังจากยืนยันตัวตนแล้ว
ขอบเขตเหล่านี้เป็นมาตรการป้องกันของระนาบควบคุมภายในโดเมนผู้ปฏิบัติการ Gateway ที่เชื่อถือได้หนึ่งโดเมน
ไม่ใช่การแยกผู้เช่าหลายรายที่ไม่ไว้วางใจกัน หากต้องการการแยกที่เข้มงวดระหว่างบุคคล
ทีม หรือเครื่อง ให้เรียกใช้ Gateway แยกกันภายใต้ผู้ใช้ระบบปฏิบัติการหรือโฮสต์ที่แยกจากกัน

เนื้อหาที่เกี่ยวข้อง: [ความปลอดภัย](/th/gateway/security), [โปรโตคอล Gateway](/th/gateway/protocol),
[การจับคู่ Gateway](/th/gateway/pairing), [CLI สำหรับอุปกรณ์](/th/cli/devices)

## บทบาท

ไคลเอนต์ WebSocket ของ Gateway ทุกตัวเชื่อมต่อด้วยบทบาทหนึ่งบทบาท:

- `operator`: ไคลเอนต์ระนาบควบคุม เช่น CLI, UI ควบคุม, ระบบอัตโนมัติ และ
  กระบวนการตัวช่วยที่เชื่อถือได้
- `node`: โฮสต์ความสามารถ (macOS, iOS, Android, แบบไม่มีส่วนติดต่อผู้ใช้) ที่เปิดให้ใช้
  คำสั่งผ่าน `node.invoke`

เมธอด RPC ของผู้ปฏิบัติการต้องใช้บทบาท `operator`; เมธอดที่เริ่มต้นจากโหนด
ต้องใช้บทบาท `node`

## ระดับขอบเขต

| ขอบเขต                   | ความหมาย                                                                                                                                                       |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `operator.read`         | สถานะ รายการ แค็ตตาล็อก บันทึก การอ่านเซสชัน และการเรียกอื่นๆ ที่ไม่เปลี่ยนแปลงข้อมูล ซึ่งเป็นแบบอ่านอย่างเดียว                                                                          |
| `operator.write`        | การดำเนินการของผู้ปฏิบัติการที่เปลี่ยนแปลงข้อมูล: การส่งข้อความ การเรียกใช้เครื่องมือ การอัปเดตการตั้งค่าการพูด/เสียง และการส่งต่อคำสั่งของโหนด รวมถึงเป็นไปตาม `operator.read` ด้วย                |
| `operator.admin`        | การเข้าถึงระดับผู้ดูแลระบบ เป็นไปตามขอบเขต `operator.*` ทุกขอบเขต จำเป็นสำหรับการเปลี่ยนแปลงการกำหนดค่า การอัปเดต ฮุกแบบเนทีฟ เนมสเปซที่สงวนไว้ และการอนุมัติที่มีความเสี่ยงสูง |
| `operator.pairing`      | การจัดการการจับคู่อุปกรณ์และโหนด: แสดงรายการ อนุมัติ ปฏิเสธ ลบ หมุนเวียน และเพิกถอน                                                                            |
| `operator.approvals`    | API สำหรับการอนุมัติการดำเนินการและ Plugin                                                                                                                                |
| `operator.talk.secrets` | การอ่านการกำหนดค่า Talk โดยรวมข้อมูลลับไว้ด้วย                                                                                                             |

ขอบเขต `operator.*` ที่ไม่รู้จักซึ่งอาจเพิ่มในอนาคตต้องตรงกันทุกประการ เว้นแต่ผู้เรียก
จะมี `operator.admin` อยู่แล้ว

## ขอบเขตของเมธอดเป็นเพียงด่านแรก

RPC ของ Gateway แต่ละรายการมีขอบเขตเมธอดตามหลักสิทธิ์ขั้นต่ำ ซึ่งกำหนดว่า
คำขอจะไปถึงตัวจัดการหรือไม่ จากนั้นตัวจัดการบางตัวจะใช้การตรวจสอบที่เข้มงวดยิ่งขึ้นโดยอิงกับ
สิ่งที่กำลังได้รับอนุมัติหรือถูกเปลี่ยนแปลงจริง:

- `device.pair.approve` สามารถเข้าถึงได้ด้วย `operator.pairing` แต่การอนุมัติ
  อุปกรณ์ของผู้ปฏิบัติการจะออกหรือคงไว้ได้เฉพาะขอบเขตที่ผู้เรียกมีอยู่แล้วเท่านั้น
- `node.pair.approve` สามารถเข้าถึงได้ด้วย `operator.pairing` จากนั้นจะหา
  ขอบเขตการอนุมัติเพิ่มเติมจากรายการคำสั่งที่โหนดซึ่งรอการอนุมัติประกาศไว้
- `chat.send` เป็นเมธอดที่มีขอบเขตการเขียน แต่คำสั่งแชต `/config set` และ
  `/config unset` ต้องใช้ `operator.admin` เพิ่มเติมจากนั้น
  โดยไม่คำนึงถึงขอบเขตการส่งแชตของผู้เรียก

การทำเช่นนี้ช่วยให้ผู้ปฏิบัติการที่มีขอบเขตต่ำกว่าสามารถดำเนินการจับคู่ที่มีความเสี่ยงต่ำได้
โดยไม่ทำให้การอนุมัติการจับคู่ทั้งหมดจำกัดเฉพาะผู้ดูแลระบบ

## การอนุมัติการจับคู่อุปกรณ์

ระเบียนการจับคู่อุปกรณ์เป็นแหล่งข้อมูลถาวรของบทบาทและขอบเขตที่ได้รับอนุมัติ
อุปกรณ์ที่จับคู่แล้วจะไม่ได้รับสิทธิ์การเข้าถึงที่กว้างขึ้นโดยอัตโนมัติ: การเชื่อมต่อใหม่
ที่ขอบทบาทหรือขอบเขตที่กว้างขึ้นจะสร้างคำขออัปเกรดใหม่ที่รอการอนุมัติ

การอนุมัติคำขออุปกรณ์:

- คำขอที่ไม่มีบทบาทผู้ปฏิบัติการไม่จำเป็นต้องได้รับการอนุมัติขอบเขตของผู้ปฏิบัติการ
- คำขอสำหรับบทบาทอุปกรณ์ที่ไม่ใช่ผู้ปฏิบัติการ (ตัวอย่างเช่น `node`) ต้องใช้
  `operator.admin` แม้ว่า `device.pair.approve` เองจะต้องใช้เพียง
  `operator.pairing`
- คำขอสำหรับ `operator.read`, `operator.write`, `operator.approvals`,
  `operator.pairing` หรือ `operator.talk.secrets` กำหนดให้ผู้เรียกต้อง
  มีขอบเขตนั้นหรือ `operator.admin` อยู่แล้ว
- คำขอสำหรับ `operator.admin` ต้องใช้ `operator.admin`
- คำขอซ่อมแซมที่ไม่ระบุขอบเขตอย่างชัดเจนสามารถสืบทอดขอบเขตของโทเค็น
  ผู้ปฏิบัติการที่มีอยู่ได้ หากโทเค็นนั้นมีขอบเขตผู้ดูแลระบบ การอนุมัติยังคงต้องใช้
  `operator.admin`

เซสชันที่ใช้ข้อมูลลับร่วมกันและพร็อกซีที่เชื่อถือได้ซึ่งไม่ใช่ผู้ดูแลระบบ สามารถอนุมัติ
คำขออุปกรณ์ผู้ปฏิบัติการได้เฉพาะภายในขอบเขตผู้ปฏิบัติการที่ตนประกาศไว้เท่านั้น การอนุมัติ
บทบาทที่ไม่ใช่ผู้ปฏิบัติการจำกัดเฉพาะผู้ดูแลระบบ แม้ว่าเซสชันเหล่านั้นจะสามารถใช้
`operator.pairing` ได้ในกรณีอื่นก็ตาม

สำหรับเซสชันโทเค็นของอุปกรณ์ที่จับคู่แล้ว การจัดการจะจำกัดเฉพาะตนเอง เว้นแต่ผู้เรียก
จะมี `operator.admin`: ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบจะเห็นเฉพาะรายการการจับคู่ของตนเอง และ
สามารถอนุมัติ ปฏิเสธ หมุนเวียน เพิกถอน หรือลบได้เฉพาะรายการอุปกรณ์ของตนเองเท่านั้น

## การอนุมัติการจับคู่โหนด

เมธอด `node.pair.*` แบบเดิมใช้ที่เก็บข้อมูลการจับคู่โหนดแยกต่างหากซึ่ง Gateway เป็นเจ้าของ
โหนด WS ใช้การจับคู่อุปกรณ์ (`role: node`) แทน แต่ยังใช้
คำศัพท์เกี่ยวกับการอนุมัติแบบเดียวกัน ดูความสัมพันธ์ระหว่างที่เก็บข้อมูลทั้งสองได้ที่
[การจับคู่ Gateway](/th/gateway/pairing)

`node.pair.approve` หาขอบเขตเพิ่มเติมที่จำเป็นจากรายการ
คำสั่งของคำขอที่รอการอนุมัติ:

| คำสั่งที่ประกาศ                                                                                                    | ขอบเขตที่จำเป็น                       |
| -------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| ไม่มี                                                                                                                 | `operator.pairing`                    |
| คำสั่งโหนดทั่วไป                                                                                               | `operator.pairing` + `operator.write` |
| `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir` หรือ `system.execApprovals.get/set` | `operator.pairing` + `operator.admin` |

การอนุมัติการประกาศของโหนดไม่ได้เปิดใช้งานคำสั่งที่มีด่านรายการอนุญาต
ขณะรันแยกต่างหาก ตัวอย่างเช่น การอนุมัติโหนดที่ประกาศ
`computer.act` ต้องใช้การจับคู่ร่วมกับขอบเขตการเขียน แต่เป็นเพียงการบันทึกพื้นผิวเท่านั้น
ผู้ดูแลระบบหรือเจ้าของยังคงต้องเปิดใช้งาน `computer.act` ขณะที่ยังคง
เปิดใช้งานอยู่ การเรียกผ่านเมธอด `node.invoke` ที่มีขอบเขตการเขียนไม่จำเป็นต้องใช้
ขอบเขตผู้ดูแลระบบสำหรับแต่ละการดำเนินการ

การจับคู่โหนดกำหนดข้อมูลประจำตัวและความไว้วางใจ แต่ไม่ได้แทนที่นโยบาย
การอนุมัติการดำเนินการ `system.run` ของโหนดเอง

## การยืนยันตัวตนด้วยข้อมูลลับร่วมกัน

การยืนยันตัวตนด้วยโทเค็น/รหัสผ่าน Gateway ที่ใช้ร่วมกันถือเป็นการเข้าถึงของผู้ปฏิบัติการ
ที่เชื่อถือได้สำหรับ Gateway นั้น พื้นผิว HTTP ที่เข้ากันได้กับ OpenAI, `/tools/invoke` และปลายทาง HTTP
สำหรับประวัติเซสชันจะคืนค่าชุดขอบเขตเริ่มต้นทั้งหมดของผู้ปฏิบัติการสำหรับ
การยืนยันตัวตนแบบ Bearer ด้วยข้อมูลลับร่วมกัน แม้ว่าผู้เรียกจะส่งขอบเขตที่ประกาศไว้แคบกว่าก็ตาม

โหมดที่มีข้อมูลประจำตัว เช่น การยืนยันตัวตนผ่านพร็อกซีที่เชื่อถือได้หรือ `none` สำหรับทางเข้าระบบแบบส่วนตัว
ยังคงสามารถเคารพขอบเขตที่ประกาศไว้อย่างชัดเจนได้ ใช้ Gateway แยกกันเพื่อแยก
ขอบเขตความไว้วางใจอย่างแท้จริง
