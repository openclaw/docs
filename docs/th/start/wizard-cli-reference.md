หน้านี้ครอบคลุมลักษณะการทำงาน ผลลัพธ์ และการทำงานภายในของขั้นตอนเริ่มต้นใช้งานแบบทีละขั้นตอน
สำหรับคำแนะนำแบบลงมือทำ โปรดดู [การเริ่มต้นใช้งาน (CLI)](/th/start/wizard) สำหรับเอกสารอ้างอิงแฟล็ก CLI ฉบับเต็ม
(ทุก `--flag`, ตัวอย่างแบบไม่โต้ตอบ และคำสั่งเฉพาะผู้ให้บริการ)
โปรดดู [`openclaw onboard`](/th/cli/onboard)

## วิซาร์ดทำอะไรบ้าง

โหมดภายในเครื่อง (ค่าเริ่มต้น) จะแนะนำคุณตลอดขั้นตอนต่อไปนี้:

- การตั้งค่าโมเดลและการยืนยันตัวตน (Anthropic, OAuth สำหรับการสมัครใช้งาน OpenAI Code, xAI, OpenCode, ปลายทางแบบกำหนดเอง และขั้นตอนการยืนยันตัวตนอื่นๆ ที่ผู้ให้บริการเป็นเจ้าของ)
- ตำแหน่งพื้นที่ทำงานและไฟล์บูตสแตรป
- การตั้งค่า Gateway (พอร์ต, การผูก, การยืนยันตัวตน, Tailscale)
- ช่องทางและผู้ให้บริการ (Discord, Feishu, Google Chat, iMessage, Mattermost, Microsoft Teams, QQ Bot, Signal, Slack, Telegram, WhatsApp และช่องทางอื่นๆ ที่รวมมาด้วยหรือเป็น Plugin)
- ผู้ให้บริการค้นหาเว็บ (ไม่บังคับ)
- การติดตั้งดีมอน (LaunchAgent, ยูนิตผู้ใช้ systemd หรือ Windows Scheduled Task แบบเนทีฟพร้อมใช้โฟลเดอร์ Startup เป็นทางเลือกสำรอง)
- การตรวจสอบสถานะระบบ
- การตั้งค่า Skills

โหมดระยะไกลกำหนดค่าเครื่องนี้ให้เชื่อมต่อกับ Gateway ที่อยู่ที่อื่น โดยจะ
ไม่ติดตั้งหรือแก้ไขสิ่งใดบนโฮสต์ระยะไกล

## รายละเอียดขั้นตอนภายในเครื่อง

<Steps>
  <Step title="การตรวจหาการกำหนดค่าที่มีอยู่">
    - หากมี `~/.openclaw/openclaw.json` ให้เลือก **คงค่าปัจจุบันไว้**, **ตรวจสอบและอัปเดต** หรือ **รีเซ็ตก่อนตั้งค่า**
    - การเรียกใช้วิซาร์ดอีกครั้งจะไม่ล้างสิ่งใด เว้นแต่คุณจะเลือก Reset อย่างชัดเจน (หรือส่ง `--reset`)
    - ค่าเริ่มต้นของ CLI `--reset` คือ `config+creds+sessions`; ใช้ `--reset-scope full` เพื่อลบพื้นที่ทำงานด้วย
    - หากการกำหนดค่าไม่ถูกต้องหรือมีคีย์รุ่นเก่า วิซาร์ดจะหยุดและขอให้คุณเรียกใช้ `openclaw doctor` ก่อนดำเนินการต่อ
    - การรีเซ็ตจะย้ายสถานะไปยัง Trash (ไม่ลบโดยตรงเด็ดขาด) และมีขอบเขตให้เลือกดังนี้:
      - เฉพาะการกำหนดค่า
      - การกำหนดค่า + ข้อมูลประจำตัว + เซสชัน
      - รีเซ็ตทั้งหมด (ลบพื้นที่ทำงานด้วย)

  </Step>
  <Step title="โมเดลและการยืนยันตัวตน">
    - เมทริกซ์ตัวเลือกทั้งหมดอยู่ใน [ตัวเลือกการยืนยันตัวตนและโมเดล](#auth-and-model-options)

  </Step>
  <Step title="พื้นที่ทำงาน">
    - ค่าเริ่มต้นคือ `~/.openclaw/workspace` (กำหนดค่าได้)
    - สร้างไฟล์พื้นที่ทำงานเริ่มต้นที่จำเป็นสำหรับการบูตสแตรปเมื่อเรียกใช้ครั้งแรก
    - โครงสร้างพื้นที่ทำงาน: [พื้นที่ทำงานของเอเจนต์](/th/concepts/agent-workspace)

  </Step>
  <Step title="Gateway">
    - แจ้งให้ระบุพอร์ต การผูก โหมดการยืนยันตัวตน และการเปิดให้เข้าถึงผ่าน Tailscale
    - คำแนะนำ: เปิดใช้การยืนยันตัวตนด้วยโทเค็นไว้แม้ใช้ลูปแบ็ก เพื่อให้ไคลเอ็นต์ WS ภายในเครื่องต้องยืนยันตัวตน
    - ในโหมดโทเค็น การตั้งค่าแบบโต้ตอบมีตัวเลือกดังนี้:
      - **สร้าง/จัดเก็บโทเค็นเป็นข้อความธรรมดา** (ค่าเริ่มต้น)
      - **ใช้ SecretRef** (เลือกใช้เอง)
    - ในโหมดรหัสผ่าน การตั้งค่าแบบโต้ตอบรองรับการจัดเก็บเป็นข้อความธรรมดาหรือ SecretRef เช่นกัน
    - พาธ SecretRef ของโทเค็นแบบไม่โต้ตอบ: `--gateway-token-ref-env <ENV_VAR>`
      - ต้องมีตัวแปรสภาพแวดล้อมที่ไม่ว่างในสภาพแวดล้อมของกระบวนการเริ่มต้นใช้งาน
      - ใช้ร่วมกับ `--gateway-token` ไม่ได้
    - ปิดใช้การยืนยันตัวตนเฉพาะเมื่อคุณเชื่อถือทุกกระบวนการภายในเครื่องอย่างเต็มที่เท่านั้น
    - การผูกที่ไม่ใช่ลูปแบ็กยังคงต้องมีการยืนยันตัวตน

  </Step>
  <Step title="ช่องทาง">
    - [WhatsApp](/th/channels/whatsapp): การเข้าสู่ระบบด้วย QR ที่ไม่บังคับ
    - [Telegram](/th/channels/telegram): โทเค็นบอต
    - [Discord](/th/channels/discord): โทเค็นบอต
    - [Google Chat](/th/channels/googlechat): JSON ของบัญชีบริการ + กลุ่มเป้าหมาย Webhook
    - [Mattermost](/th/channels/mattermost): โทเค็นบอต + URL ฐาน
    - [Signal](/th/channels/signal): การติดตั้ง `signal-cli` ที่ไม่บังคับ + การกำหนดค่าบัญชี
    - [iMessage](/th/channels/imessage): พาธ CLI ของ `imsg` + สิทธิ์เข้าถึงฐานข้อมูล Messages; ใช้ตัวห่อหุ้ม SSH เมื่อ Gateway ทำงานนอก Mac
    - ความปลอดภัยของ DM: ค่าเริ่มต้นคือการจับคู่ DM แรกจะส่งรหัส; อนุมัติผ่าน
      `openclaw pairing approve <channel> <code>` หรือใช้รายการอนุญาต
  </Step>
  <Step title="การค้นหาเว็บ">
    - เลือกผู้ให้บริการ (Brave, DuckDuckGo, Exa, Firecrawl, Gemini, Grok, Kimi, MiniMax Search, Ollama Web Search, Perplexity, SearXNG, Tavily) หรือข้าม
    - ข้ามขั้นตอนนี้ด้วย `--skip-search`; กำหนดค่าใหม่ภายหลังด้วย `openclaw configure --section web`

  </Step>
  <Step title="การติดตั้งดีมอน">
    - macOS: LaunchAgent
      - ต้องมีเซสชันผู้ใช้ที่เข้าสู่ระบบอยู่; สำหรับระบบแบบไม่มีหน้าจอ ให้ใช้ LaunchDaemon แบบกำหนดเอง (ไม่มีมาให้)
    - Linux และ Windows ผ่าน WSL2: ยูนิตผู้ใช้ systemd
      - วิซาร์ดจะพยายามใช้ `loginctl enable-linger <user>` เพื่อให้ Gateway ทำงานต่อหลังออกจากระบบ
      - อาจขอ sudo (เขียน `/var/lib/systemd/linger`); โดยจะลองโดยไม่ใช้ sudo ก่อน
    - Windows แบบเนทีฟ: ใช้ Scheduled Task ก่อน
      - หากระบบปฏิเสธการสร้างงาน OpenClaw จะเปลี่ยนไปใช้รายการเข้าสู่ระบบในโฟลเดอร์ Startup สำหรับผู้ใช้แต่ละราย และเริ่ม Gateway ทันที
      - ยังคงแนะนำให้ใช้ Scheduled Tasks เพราะแสดงสถานะตัวควบคุมดูแลได้ดีกว่า
    - การเลือกรันไทม์: ต้องใช้ Node เนื่องจากที่เก็บสถานะรันไทม์มาตรฐานของ OpenClaw ใช้ `node:sqlite`

  </Step>
  <Step title="การตรวจสอบสถานะระบบ">
    - เริ่ม Gateway (หากจำเป็น) และเรียกใช้ `openclaw health`
    - `openclaw status --deep` เพิ่มการตรวจสอบสถานะ Gateway ที่กำลังทำงานลงในผลลัพธ์สถานะ รวมถึงการตรวจสอบช่องทางเมื่อรองรับ

  </Step>
  <Step title="Skills">
    - อ่าน Skills ที่พร้อมใช้งานและตรวจสอบข้อกำหนด
    - ให้คุณเลือกตัวจัดการ Node: npm, pnpm หรือ bun
    - ติดตั้งการขึ้นต่อกันที่ไม่บังคับสำหรับ Skills ที่เชื่อถือได้ซึ่งรวมมาด้วย เมื่อมี
      โปรแกรมติดตั้งที่จำเป็น
    - ข้ามโปรแกรมติดตั้ง Homebrew, uv และ Go ที่ไม่พร้อมใช้งาน จากนั้นจัดกลุ่ม
      Skills ที่ได้รับผลกระทบพร้อมคำแนะนำในการตั้งค่าด้วยตนเอง เรียกใช้ `openclaw doctor` หลังจากติดตั้ง
      ข้อกำหนดเบื้องต้นที่ขาดหายไป

  </Step>
  <Step title="เสร็จสิ้น">
    - สรุปและขั้นตอนถัดไป รวมถึงตัวเลือกแอป iOS, Android และ macOS

  </Step>
</Steps>

<Note>
หากตรวจไม่พบ GUI วิซาร์ดจะแสดงคำแนะนำการส่งต่อพอร์ต SSH สำหรับ Control UI แทนการเปิดเบราว์เซอร์
หากไม่มีแอสเซ็ตของ Control UI วิซาร์ดจะพยายามสร้างแอสเซ็ตดังกล่าว; ทางเลือกสำรองคือ `pnpm ui:build` (ติดตั้งการขึ้นต่อกันของ UI โดยอัตโนมัติ)
</Note>

## รายละเอียดโหมดระยะไกล

โหมดระยะไกลกำหนดค่าเครื่องนี้ให้เชื่อมต่อกับ Gateway ที่อยู่ที่อื่น โดยจะ
ไม่ติดตั้งหรือแก้ไขสิ่งใดบนโฮสต์ระยะไกล

สิ่งที่คุณตั้งค่า:

- URL ของ Gateway ระยะไกล (`ws://...` หรือ `wss://...`)
- โทเค็น รหัสผ่าน หรือไม่ใช้การยืนยันตัวตน ให้ตรงกับการกำหนดค่าของ Gateway ระยะไกล

<Steps>
  <Step title="การค้นหา (ไม่บังคับ)">
    หากมี `dns-sd` (macOS) หรือ `avahi-browse` (Linux) ขั้นตอนเริ่มต้นใช้งาน
    จะเสนอให้ค้นหาบีคอน Gateway ของ Bonjour/mDNS ก่อนเปลี่ยนไปใช้
    การป้อน URL ด้วยตนเอง นอกจากนี้ยังพยายามค้นหาผ่าน DNS-SD แบบเครือข่ายบริเวณกว้างเมื่อ
    มีการกำหนดค่าไว้ เอกสาร: [การค้นหา Gateway](/th/gateway/discovery), [Bonjour](/th/gateway/bonjour)
  </Step>
  <Step title="วิธีการเชื่อมต่อ">
    เมื่อเลือกบีคอนแล้ว ให้เลือก WebSocket โดยตรงหรืออุโมงค์ SSH:
    - **โดยตรง**: เชื่อมต่อผ่าน `wss://` และแจ้งให้เชื่อถือลายนิ้วมือ
      TLS ที่ค้นพบ (การตรึงแบบเชื่อถือเมื่อใช้ครั้งแรก; ตรึงเฉพาะเมื่อคุณยอมรับ)
    - **อุโมงค์ SSH**: แสดงคำสั่ง `ssh -N -L 18789:127.0.0.1:18789 <user>@<host>`
      ที่ต้องเรียกใช้ก่อน จากนั้นเชื่อมต่อกับปลายทางอุโมงค์ภายในเครื่อง
  </Step>
  <Step title="การยืนยันตัวตน">
    เลือกโทเค็น (แนะนำ), รหัสผ่าน หรือไม่ใช้การยืนยันตัวตน จากนั้นเลือกได้ว่าจะจัดเก็บเป็น
    SecretRef แทนข้อความธรรมดาหรือไม่
  </Step>
</Steps>

<Note>
หาก Gateway ใช้เฉพาะลูปแบ็กและค้นหาไม่พบ ให้ใช้อุโมงค์ SSH หรือ tailnet ด้วยตนเอง
ระบบยอมรับ `ws://` แบบข้อความธรรมดาสำหรับลูปแบ็ก ลิเทอรัล IP ส่วนตัว `.local` และ URL ของ Tailnet `*.ts.net`; ชื่อ DNS ส่วนตัวอื่นๆ ต้องใช้ `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`
</Note>

## ตัวเลือกการยืนยันตัวตนและโมเดล

หากขั้นตอนตั้งค่าผู้ให้บริการล้มเหลวระหว่างการเริ่มต้นใช้งานแบบโต้ตอบ (เช่น ตัวเลือกการใช้ CLI ซ้ำ
โดยไม่มีการเข้าสู่ระบบภายในเครื่อง) วิซาร์ดจะแสดงข้อผิดพลาดและกลับไปยังตัวเลือกผู้ให้บริการ
แทนการออกจากโปรแกรม การเรียกใช้ `--auth-choice` อย่างชัดเจนยังคงล้มเหลวทันทีสำหรับระบบอัตโนมัติ

  </Accordion>

- เมื่อการเริ่มต้นใช้งานเริ่มจากตัวเลือกการยืนยันตัวตนของผู้ให้บริการ ตัวเลือกโมเดลจะให้ความสำคัญกับ
  ผู้ให้บริการนั้นโดยอัตโนมัติ สำหรับ Volcengine และ BytePlus การตั้งค่านี้
  ยังตรงกับรูปแบบแพ็กเกจการเขียนโค้ดของผู้ให้บริการเหล่านั้นด้วย (`volcengine-plan/*`,
  `byteplus-plan/*`)
- หากตัวกรองผู้ให้บริการที่ต้องการนั้นไม่พบรายการใด ตัวเลือกจะกลับไปใช้
  แค็ตตาล็อกทั้งหมดแทนการไม่แสดงโมเดลใดเลย
- ตัวช่วยสร้างจะตรวจสอบโมเดลและเตือนหากไม่รู้จักโมเดลที่กำหนดค่าไว้หรือไม่มีการยืนยันตัวตน

พาธข้อมูลประจำตัวและโปรไฟล์:

- โปรไฟล์การยืนยันตัวตน (คีย์ API + OAuth): `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- การนำเข้า OAuth แบบเดิม: `~/.openclaw/credentials/oauth.json`

โหมดการจัดเก็บข้อมูลประจำตัว:

- โดยค่าเริ่มต้น การเริ่มต้นใช้งานจะจัดเก็บคีย์ API เป็นค่าข้อความธรรมดาในโปรไฟล์การยืนยันตัวตน
- `--secret-input-mode ref` เปิดใช้โหมดการอ้างอิงแทนการจัดเก็บคีย์เป็นข้อความธรรมดา
  ในการตั้งค่าแบบโต้ตอบ สามารถเลือกได้ดังนี้:
  - การอ้างอิงตัวแปรสภาพแวดล้อม (ตัวอย่างเช่น `keyRef: { source: "env", provider: "default", id: "OPENAI_API_KEY" }`)
  - การอ้างอิงผู้ให้บริการที่กำหนดค่าไว้ (`file` หรือ `exec`) พร้อมนามแฝงผู้ให้บริการ + ID
- โหมดการอ้างอิงแบบโต้ตอบจะตรวจสอบล่วงหน้าอย่างรวดเร็วก่อนบันทึก
  - การอ้างอิงตัวแปรสภาพแวดล้อม: ตรวจสอบชื่อตัวแปรและค่าที่ไม่ว่างในสภาพแวดล้อมการเริ่มต้นใช้งานปัจจุบัน
  - การอ้างอิงผู้ให้บริการ: ตรวจสอบการกำหนดค่าผู้ให้บริการและแก้ไข ID ที่ร้องขอ
  - หากการตรวจสอบล่วงหน้าล้มเหลว การเริ่มต้นใช้งานจะแสดงข้อผิดพลาดและให้ลองอีกครั้ง
- ในโหมดไม่โต้ตอบ `--secret-input-mode ref` รองรับเฉพาะตัวแปรสภาพแวดล้อม
  - ตั้งค่าตัวแปรสภาพแวดล้อมของผู้ให้บริการในสภาพแวดล้อมของกระบวนการเริ่มต้นใช้งาน
  - แฟล็กคีย์แบบอินไลน์ (ตัวอย่างเช่น `--openai-api-key`) กำหนดให้ต้องตั้งค่าตัวแปรสภาพแวดล้อมนั้น มิฉะนั้นการเริ่มต้นใช้งานจะล้มเหลวทันที
  - สำหรับผู้ให้บริการแบบกำหนดเอง โหมดไม่โต้ตอบ `ref` จะจัดเก็บ `models.providers.<id>.apiKey` เป็น `{ source: "env", provider: "default", id: "CUSTOM_API_KEY" }`
  - ในกรณีผู้ให้บริการแบบกำหนดเองดังกล่าว `--custom-api-key` กำหนดให้ต้องตั้งค่า `CUSTOM_API_KEY` มิฉะนั้นการเริ่มต้นใช้งานจะล้มเหลวทันที
- ข้อมูลประจำตัวสำหรับการยืนยันตัวตนของ Gateway รองรับตัวเลือกข้อความธรรมดาและ SecretRef ในการตั้งค่าแบบโต้ตอบ:
  - โหมดโทเค็น: **สร้าง/จัดเก็บโทเค็นเป็นข้อความธรรมดา** (ค่าเริ่มต้น) หรือ **ใช้ SecretRef**
  - โหมดรหัสผ่าน: ข้อความธรรมดาหรือ SecretRef
- พาธ SecretRef ของโทเค็นในโหมดไม่โต้ตอบ: `--gateway-token-ref-env <ENV_VAR>`
- การตั้งค่าแบบข้อความธรรมดาที่มีอยู่จะยังคงทำงานโดยไม่มีการเปลี่ยนแปลง

<Note>
เคล็ดลับสำหรับระบบไร้หน้าจอและเซิร์ฟเวอร์: ดำเนินการ OAuth ให้เสร็จบนเครื่องที่มีเบราว์เซอร์ จากนั้นคัดลอก

</Note>

### คำแนะนำแอปที่ติดตั้ง

หลังจากการตรวจสอบการเข้าถึงโมเดลสำเร็จ การเริ่มต้นใช้งานแบบโต้ตอบดั้งเดิมบน macOS จะสแกนชื่อแอปพลิเคชันและ Bundle ID โดยไม่ขอสิทธิ์ความเป็นส่วนตัวของ macOS โดยค้นหาในแค็ตตาล็อก Plugin อย่างเป็นทางการและ ClawHub จากนั้นขอให้โมเดลที่กำหนดค่าไว้คัดชื่อที่ตรงกันผิดออก และแนะนำ Plugin หรือ Skills ที่เกี่ยวข้อง รายการที่แนะนำจะถูกเลือกโดยค่าเริ่มต้น ส่วนรายการเสริมต้องเลือกอย่างชัดเจน

หน้าจอผลลัพธ์จะแสดงรายการแอปพลิเคชันที่ตรวจพบและข้อความ: "จับคู่ชื่อแอปโดยใช้โมเดลที่กำหนดค่าไว้และการค้นหาของ ClawHub" ตั้งค่า `wizard.appRecommendations` เป็น `false` เพื่อปิดทั้งขั้นตอนการเริ่มต้นใช้งานนี้และการเข้าถึงรายการแอปของ Node จาก Gateway การสแกนนี้จะไม่ใช้ในการเริ่มต้นอย่างรวดเร็วหรือการเริ่มต้นใช้งานบนระบบที่ไม่ใช่ macOS

## การตั้งค่าแบบไม่โต้ตอบ

`--non-interactive` ต้องใช้ `--accept-risk` (ยอมรับว่าเอเจนต์มีความสามารถสูง
และการเข้าถึงระบบทั้งหมดมีความเสี่ยง):

```bash
openclaw onboard --non-interactive --accept-risk \
  --auth-choice apiKey \
  --anthropic-api-key "$ANTHROPIC_API_KEY"
```

ข้อมูลอ้างอิงแฟล็กฉบับเต็มและตัวอย่างเฉพาะผู้ให้บริการ: [`openclaw onboard`](/th/cli/onboard), [การทำงานอัตโนมัติด้วย CLI](/th/start/wizard-cli-automation)

## RPC ของตัวช่วยสร้าง Gateway

- `wizard.start`
- `wizard.next`
- `wizard.cancel`
- `wizard.status`

ไคลเอนต์ (แอป macOS และ UI ควบคุม) สามารถแสดงขั้นตอนโดยไม่ต้องนำตรรกะการเริ่มต้นใช้งานไปสร้างใหม่

## ลักษณะการทำงานของการตั้งค่า Signal

- ดาวน์โหลดอาร์ติแฟกต์รีลีสที่เหมาะสมจากรีลีส GitHub อย่างเป็นทางการของ `signal-cli` (บิลด์เนทีฟ รองรับเฉพาะ Linux x86-64)
- บนแพลตฟอร์มอื่น (macOS, Linux ที่ไม่ใช่ x64) จะติดตั้งผ่าน Homebrew แทน
- จัดเก็บการติดตั้งจากอาร์ติแฟกต์รีลีสไว้ภายใต้ `~/.openclaw/tools/signal-cli/<version>/`
- เขียน `channels.signal.cliPath` ลงในการกำหนดค่า
- ยังไม่รองรับ Windows แบบเนทีฟ ให้เรียกใช้การเริ่มต้นใช้งานภายใน WSL2 เพื่อรับพาธการติดตั้งของ Linux
