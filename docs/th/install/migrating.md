OpenClaw รองรับเส้นทางการย้ายข้อมูลสามรูปแบบ ได้แก่ การนำเข้าจากระบบเอเจนต์อื่น การย้ายการติดตั้งที่มีอยู่ไปยังเครื่องใหม่ และการอัปเกรด Plugin ในตำแหน่งเดิม

## นำเข้าจากระบบเอเจนต์อื่น

ผู้ให้บริการย้ายข้อมูลที่มาพร้อมระบบจะนำคำสั่ง เซิร์ฟเวอร์ MCP, Skills, การกำหนดค่าโมเดล และคีย์ API (เมื่อเลือกใช้) เข้าสู่ OpenClaw โดยจะแสดงตัวอย่างแผนก่อนทำการเปลี่ยนแปลง ปกปิดข้อมูลลับในรายงาน และสำรองข้อมูลพร้อมตรวจสอบความถูกต้องก่อนนำแผนไปใช้

<CardGroup cols={2}>
  <Card title="ย้ายข้อมูลจาก Claude" href="/th/install/migrating-claude" icon="brain">
    นำเข้าสถานะของ Claude Code และ Claude Desktop รวมถึง `CLAUDE.md` เซิร์ฟเวอร์ MCP, Skills และคำสั่งของโปรเจกต์
  </Card>
  <Card title="ย้ายข้อมูลจาก Hermes" href="/th/install/migrating-hermes" icon="feather">
    นำเข้าการกำหนดค่า ผู้ให้บริการ เซิร์ฟเวอร์ MCP หน่วยความจำ Skills และคีย์ `.env` ที่รองรับจาก Hermes
  </Card>
</CardGroup>

จุดเริ่มต้นของ CLI คือ [`openclaw migrate`](/th/cli/migrate) การเริ่มต้นใช้งานยังสามารถเสนอให้ย้ายข้อมูลเมื่อตรวจพบแหล่งที่มาที่รู้จัก (`openclaw onboard --flow import`)

## ย้าย OpenClaw ไปยังเครื่องใหม่

คัดลอก **ไดเรกทอรีสถานะ** (ค่าเริ่มต้นคือ `~/.openclaw/`) และ **พื้นที่ทำงาน** ของคุณเพื่อรักษารายการต่อไปนี้ไว้:

- **การกำหนดค่า** — `openclaw.json` และการตั้งค่า Gateway ทั้งหมด
- **การยืนยันตัวตน** — `auth-profiles.json` ของแต่ละเอเจนต์ (คีย์ API และ OAuth) รวมถึงสถานะของช่องทางหรือผู้ให้บริการภายใต้ `credentials/`
- **เซสชัน** — ประวัติการสนทนาและสถานะของเอเจนต์
- **สถานะช่องทาง** — การเข้าสู่ระบบ WhatsApp, เซสชัน Telegram และรายการอื่นที่คล้ายกัน
- **ไฟล์พื้นที่ทำงาน** — `MEMORY.md`, `USER.md`, Skills และพรอมต์

<Tip>
เรียกใช้ `openclaw status` บนเครื่องเก่าเพื่อยืนยันพาธไดเรกทอรีสถานะของคุณ โปรไฟล์ที่กำหนดเองจะใช้ `~/.openclaw-<profile>/` หรือพาธที่ตั้งค่าผ่าน `OPENCLAW_STATE_DIR`
</Tip>

### ขั้นตอนการย้ายข้อมูล

<Steps>
  <Step title="หยุด Gateway และสำรองข้อมูล">
    บนเครื่อง **เก่า** ให้หยุด Gateway เพื่อป้องกันไม่ให้ไฟล์เปลี่ยนแปลงระหว่างการคัดลอก จากนั้นสร้างไฟล์เก็บถาวร:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    หากคุณใช้หลายโปรไฟล์ (เช่น `~/.openclaw-work`) ให้สร้างไฟล์เก็บถาวรแยกสำหรับแต่ละโปรไฟล์

  </Step>

  <Step title="ติดตั้ง OpenClaw บนเครื่องใหม่">
    [ติดตั้ง](/th/install) CLI (และ Node หากจำเป็น) บนเครื่องใหม่ หากการเริ่มต้นใช้งานสร้าง `~/.openclaw/` ใหม่ก็ไม่เป็นปัญหา เพราะคุณจะเขียนทับในขั้นตอนถัดไป
  </Step>

  <Step title="คัดลอกไดเรกทอรีสถานะและพื้นที่ทำงาน">
    ถ่ายโอนไฟล์เก็บถาวรผ่าน `scp`, `rsync -a` หรือไดรฟ์ภายนอก จากนั้นแตกไฟล์:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    ยืนยันว่ามีการรวมไดเรกทอรีที่ซ่อนไว้ และความเป็นเจ้าของไฟล์ตรงกับผู้ใช้ที่จะเรียกใช้ Gateway

  </Step>

  <Step title="เรียกใช้ Doctor และตรวจสอบ">
    บนเครื่องใหม่ ให้เรียกใช้ [Doctor](/th/gateway/doctor) เพื่อใช้การย้ายรูปแบบการกำหนดค่าและซ่อมแซมบริการ:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

หาก Telegram หรือ Discord ใช้ตัวแปรสภาพแวดล้อมสำรองตามค่าเริ่มต้น (`TELEGRAM_BOT_TOKEN` หรือ `DISCORD_BOT_TOKEN`) ให้ตรวจสอบว่า `.env` ในไดเรกทอรีสถานะที่ย้ายมามีคีย์เหล่านั้น โดยไม่แสดงค่าข้อมูลลับ:

```bash
awk -F= '/^(TELEGRAM_BOT_TOKEN|DISCORD_BOT_TOKEN)=/ { print $1 "=present" }' ~/.openclaw/.env
```

`openclaw doctor` ยังจะแจ้งเตือนเมื่อบัญชี Telegram หรือ Discord เริ่มต้นที่เปิดใช้งานไม่มีโทเค็นที่กำหนดค่าไว้ และตัวแปรสภาพแวดล้อมที่ตรงกันไม่พร้อมใช้งานสำหรับกระบวนการ Doctor

### ข้อผิดพลาดที่พบบ่อย

<AccordionGroup>
  <Accordion title="โปรไฟล์หรือไดเรกทอรีสถานะไม่ตรงกัน">
    หาก Gateway เดิมใช้ `--profile` หรือ `OPENCLAW_STATE_DIR` แต่ Gateway ใหม่ไม่ได้ใช้ ช่องทางต่าง ๆ จะแสดงว่าออกจากระบบแล้วและเซสชันจะว่างเปล่า ให้เปิดใช้ Gateway ด้วยโปรไฟล์หรือไดเรกทอรีสถานะ **เดียวกัน** กับที่คุณย้ายมา จากนั้นเรียกใช้ `openclaw doctor` อีกครั้ง
  </Accordion>

  <Accordion title="คัดลอกเฉพาะ openclaw.json">
    ไฟล์การกำหนดค่าเพียงอย่างเดียวไม่เพียงพอ โปรไฟล์การยืนยันตัวตนของโมเดลอยู่ภายใต้ `agents/<agentId>/agent/auth-profiles.json` ส่วนสถานะของช่องทางและผู้ให้บริการอยู่ภายใต้ `credentials/` ให้ย้ายไดเรกทอรีสถานะ **ทั้งหมด** เสมอ
  </Accordion>

  <Accordion title="สิทธิ์และความเป็นเจ้าของ">
    หากคุณคัดลอกในฐานะ root หรือเปลี่ยนผู้ใช้ Gateway อาจไม่สามารถอ่านข้อมูลประจำตัวได้ ตรวจสอบให้แน่ใจว่าไดเรกทอรีสถานะและพื้นที่ทำงานเป็นของผู้ใช้ที่เรียกใช้ Gateway
  </Accordion>

  <Accordion title="โหมดระยะไกล">
    หาก UI ของคุณเชื่อมต่อกับ Gateway **ระยะไกล** โฮสต์ระยะไกลจะเป็นเจ้าของเซสชันและพื้นที่ทำงาน ให้ย้ายโฮสต์ Gateway ไม่ใช่แล็ปท็อปภายในเครื่องของคุณ ดู [คำถามที่พบบ่อย](/th/help/faq#where-things-live-on-disk)
  </Accordion>

  <Accordion title="ข้อมูลลับในข้อมูลสำรอง">
    ไดเรกทอรีสถานะประกอบด้วยโปรไฟล์การยืนยันตัวตน ข้อมูลประจำตัวของช่องทาง และสถานะอื่น ๆ ของผู้ให้บริการ ให้จัดเก็บข้อมูลสำรองในรูปแบบเข้ารหัส หลีกเลี่ยงช่องทางการถ่ายโอนที่ไม่ปลอดภัย และหมุนเวียนคีย์หากสงสัยว่ามีการเปิดเผย
  </Accordion>
</AccordionGroup>

### รายการตรวจสอบการยืนยัน

บนเครื่องใหม่ ให้ยืนยันว่า:

- [ ] `openclaw status` แสดงว่า Gateway กำลังทำงาน
- [ ] ช่องทางต่าง ๆ ยังคงเชื่อมต่ออยู่ (ไม่ต้องจับคู่ใหม่)
- [ ] แดชบอร์ดเปิดได้และแสดงเซสชันที่มีอยู่
- [ ] มีไฟล์พื้นที่ทำงาน (หน่วยความจำและการกำหนดค่า) ครบถ้วน

## อัปเกรด Plugin ในตำแหน่งเดิม

การอัปเกรด Plugin ในตำแหน่งเดิมจะคงรหัส Plugin และคีย์การกำหนดค่าเดิมไว้ แต่อาจย้ายสถานะบนดิสก์ไปยังโครงสร้างปัจจุบัน คู่มือการอัปเกรดเฉพาะ Plugin อยู่ร่วมกับช่องทางที่เกี่ยวข้อง:

- [การย้ายข้อมูล Matrix](/th/channels/matrix-migration): ขีดจำกัดในการกู้คืนสถานะที่เข้ารหัส ลักษณะการทำงานของสแนปช็อตอัตโนมัติ และคำสั่งกู้คืนด้วยตนเอง

## เนื้อหาที่เกี่ยวข้อง

- [`openclaw migrate`](/th/cli/migrate): เอกสารอ้างอิง CLI สำหรับการนำเข้าข้ามระบบ

- [Doctor](/th/gateway/doctor): การตรวจสอบความสมบูรณ์หลังการย้ายข้อมูล
- [ถอนการติดตั้ง](/th/install/uninstall): การนำ OpenClaw ออกอย่างสมบูรณ์
