# การแก้ไขการพึ่งพาของ Plugin

OpenClaw จัดการงานการพึ่งพาของ Plugin ในช่วงติดตั้ง/อัปเดต การโหลดขณะรัน
จะไม่เรียกใช้ตัวจัดการแพ็กเกจ ซ่อมแซมแผนผังการพึ่งพา หรือแก้ไขไดเรกทอรีแพ็กเกจของ OpenClaw

## การแบ่งความรับผิดชอบ

แพ็กเกจ Plugin เป็นเจ้าของกราฟการพึ่งพาของตัวเอง:

- การพึ่งพาขณะรันอยู่ใน `dependencies` หรือ `optionalDependencies` ของแพ็กเกจ Plugin
- การนำเข้า SDK/แกนหลักเป็นแบบ peer หรือเป็นการนำเข้าที่ OpenClaw จัดให้
- Plugin สำหรับการพัฒนาในเครื่องนำการพึ่งพาที่ติดตั้งไว้แล้วของตัวเองมาด้วย
- Plugin จาก npm และ git ถูกติดตั้งลงในรากแพ็กเกจที่ OpenClaw เป็นเจ้าของ

OpenClaw เป็นเจ้าของเฉพาะวงจรชีวิตของ Plugin:

- ค้นหาแหล่งที่มาของ Plugin
- ติดตั้งหรืออัปเดตแพ็กเกจเมื่อมีการขออย่างชัดเจน
- บันทึกเมทาดาทาการติดตั้ง
- โหลดจุดเข้าใช้งานของ Plugin
- ล้มเหลวพร้อมข้อผิดพลาดที่ดำเนินการต่อได้เมื่อการพึ่งพาหายไป

## รากการติดตั้ง

OpenClaw ใช้รากที่เสถียรแยกตามแหล่งที่มา:

- แพ็กเกจ npm ติดตั้งภายใต้ `~/.openclaw/npm`
- แพ็กเกจ git โคลนภายใต้ `~/.openclaw/git`
- การติดตั้งแบบ local/path/archive จะถูกคัดลอกหรืออ้างอิงโดยไม่มีการซ่อมแซมการพึ่งพา

การติดตั้ง npm ทำงานในราก npm ด้วย:

```bash
npm install --prefix ~/.openclaw/npm <spec> --omit=dev --ignore-scripts --no-audit --no-fund
```

npm อาจยกการพึ่งพาทางอ้อมไปไว้ที่ `~/.openclaw/npm/node_modules` ข้าง
แพ็กเกจ Plugin ได้ OpenClaw จะสแกนราก npm ที่จัดการไว้ก่อนเชื่อถือ
การติดตั้ง และใช้ npm เพื่อลบแพ็กเกจที่ npm จัดการระหว่างการถอนการติดตั้ง ดังนั้น
การพึ่งพาขณะรันที่ถูกยกขึ้นจะยังอยู่ภายในขอบเขตการล้างข้อมูลที่จัดการไว้

การติดตั้ง git จะโคลนหรือรีเฟรชที่เก็บ แล้วเรียกใช้:

```bash
npm install --omit=dev --ignore-scripts --no-audit --no-fund
```

จากนั้น Plugin ที่ติดตั้งแล้วจะโหลดจากไดเรกทอรีแพ็กเกจนั้น ดังนั้นการแก้ไข
การอ้างอิง `node_modules` เฉพาะแพ็กเกจและของพาเรนต์จะทำงานเหมือนกับที่ทำ
ในแพ็กเกจ Node ปกติ

## Plugin ในเครื่อง

Plugin ในเครื่องถือเป็นไดเรกทอรีที่นักพัฒนาควบคุม OpenClaw จะไม่
เรียกใช้ `npm install`, `pnpm install` หรือซ่อมแซมการพึ่งพาสำหรับสิ่งเหล่านี้ หาก Plugin
ในเครื่องมีการพึ่งพา ให้ติดตั้งการพึ่งพาเหล่านั้นใน Plugin นั้นก่อนโหลด

Plugin TypeScript ในเครื่องของบุคคลที่สามสามารถใช้เส้นทาง Jiti ฉุกเฉินได้ Plugin
JavaScript แบบแพ็กเกจและ Plugin ภายในที่บันเดิลมาจะโหลดผ่าน
import/require แบบเนทีฟแทน Jiti

## การเริ่มต้นและการโหลดซ้ำ

การเริ่มต้น Gateway และการโหลดคอนฟิกซ้ำจะไม่ติดตั้งการพึ่งพาของ Plugin สิ่งเหล่านี้จะอ่าน
ระเบียนการติดตั้ง Plugin คำนวณจุดเข้าใช้งาน และโหลดจุดนั้น

หากการพึ่งพาหายไปขณะรัน Plugin จะโหลดไม่สำเร็จ และข้อผิดพลาด
ควรชี้ผู้ปฏิบัติงานไปยังวิธีแก้ที่ชัดเจน:

```bash
openclaw plugins update <id>
openclaw plugins install <source>
openclaw doctor --fix
```

`doctor --fix` สามารถล้างสถานะการพึ่งพารุ่นเก่าที่ OpenClaw สร้างขึ้น และติดตั้ง
Plugin ที่ดาวน์โหลดได้ซึ่งกำหนดค่าไว้แต่หายไปจากระเบียนการติดตั้งในเครื่องได้
คำสั่งนี้จะไม่ซ่อมแซมการพึ่งพาสำหรับ Plugin ในเครื่องที่ติดตั้งไว้แล้ว

## Plugin ที่บันเดิลมา

Plugin ที่บันเดิลมาซึ่งมีน้ำหนักเบาและสำคัญต่อแกนหลักจะถูกส่งมาพร้อมกับ OpenClaw
Plugin เหล่านี้ควรไม่มีแผนผังการพึ่งพาขณะรันที่หนัก หรือควรถูกย้ายออกไปเป็น
แพ็กเกจที่ดาวน์โหลดได้บน ClawHub/npm

สำหรับรายการที่สร้างขึ้นล่าสุดของ Plugin ที่ส่งมาในแพ็กเกจแกนหลัก ติดตั้ง
ภายนอก หรือคงไว้เฉพาะในซอร์ส โปรดดู [คลังรายการ Plugin](/th/plugins/plugin-inventory)

รายการกำกับของ Plugin ที่บันเดิลมาต้องไม่ขอการจัดเตรียมการพึ่งพา ฟังก์ชัน
Plugin ขนาดใหญ่หรือเป็นตัวเลือกควรถูกแพ็กเกจเป็น Plugin ปกติและติดตั้งผ่าน
เส้นทาง npm/git/ClawHub เดียวกับ Plugin ของบุคคลที่สาม

ในเช็กเอาต์ซอร์ส OpenClaw ถือว่าที่เก็บเป็น pnpm monorepo หลังจาก
`pnpm install` แล้ว Plugin ที่บันเดิลมาจะโหลดจาก `extensions/<id>` เพื่อให้การพึ่งพา
เวิร์กสเปซเฉพาะแพ็กเกจพร้อมใช้งานและรับการแก้ไขโดยตรง การพัฒนาจาก
เช็กเอาต์ซอร์สรองรับเฉพาะ pnpm เท่านั้น; การใช้ `npm install` ธรรมดาที่ราก
ที่เก็บไม่ใช่วิธีที่รองรับสำหรับเตรียมการพึ่งพาของ Plugin ที่บันเดิลมา

| รูปแบบการติดตั้ง                    | ตำแหน่ง Plugin ที่บันเดิลมา               | เจ้าของการพึ่งพา                                                     |
| -------------------------------- | ------------------------------------- | -------------------------------------------------------------------- |
| `npm install -g openclaw`        | แผนผังรันไทม์ที่สร้างไว้ภายในแพ็กเกจ | แพ็กเกจ OpenClaw และโฟลว์ติดตั้ง/อัปเดต/doctor ของ Plugin ที่ชัดเจน     |
| เช็กเอาต์ Git พร้อม `pnpm install` | แพ็กเกจเวิร์กสเปซ `extensions/<id>`  | เวิร์กสเปซ pnpm รวมถึงการพึ่งพาของแต่ละแพ็กเกจ Plugin เอง |
| `openclaw plugins install ...`   | ราก Plugin npm/git/ClawHub ที่จัดการไว้   | โฟลว์ติดตั้ง/อัปเดต Plugin                                       |

## การล้างข้อมูลรุ่นเก่า

OpenClaw รุ่นเก่ากว่าสร้างรากการพึ่งพาของ Plugin ที่บันเดิลมาเมื่อเริ่มต้น หรือ
ระหว่างการซ่อมแซมด้วย doctor การล้างข้อมูลด้วย doctor ปัจจุบันจะลบไดเรกทอรีและ
ซิมลิงก์เก่าเหล่านั้นเมื่อใช้ `--fix` รวมถึงราก `plugin-runtime-deps` เดิม ซิมลิงก์
แพ็กเกจคำนำหน้า Node ระดับโกลบอลที่ชี้ไปยังเป้าหมาย `plugin-runtime-deps` ที่ถูกตัดทิ้ง
รายการกำกับ `.openclaw-runtime-deps*`, `node_modules` ของ Plugin ที่สร้างขึ้น
ไดเรกทอรีระยะการติดตั้ง และสโตร์ pnpm เฉพาะแพ็กเกจ postinstall แบบแพ็กเกจยัง
ลบซิมลิงก์โกลบอลเหล่านั้นก่อนตัดรากเป้าหมายรุ่นเก่า เพื่อไม่ให้การอัปเกรด
ทิ้งการนำเข้าแพ็กเกจ ESM ที่ห้อยค้างไว้

เส้นทางเหล่านี้เป็นเพียงเศษตกค้างจากรุ่นเก่า การติดตั้งใหม่ไม่ควรสร้างเส้นทางเหล่านี้
