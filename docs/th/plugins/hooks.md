Hook ของ Plugin คือจุดส่วนขยายภายในโปรเซสสำหรับ Plugin ของ OpenClaw ซึ่งใช้ตรวจสอบหรือ
เปลี่ยนแปลงการรันของเอเจนต์ การเรียกใช้เครื่องมือ โฟลว์ข้อความ วงจรชีวิตของเซสชัน การกำหนดเส้นทาง
เอเจนต์ย่อย การติดตั้ง หรือการเริ่มต้น Gateway

ให้ใช้ [Hook ภายใน](/th/automation/hooks) แทน สำหรับสคริปต์ `HOOK.md` ขนาดเล็กที่ผู้ดำเนินการติดตั้ง
เพื่อให้ตอบสนองต่อเหตุการณ์ของคำสั่งและ Gateway เช่น `/new`,
`/reset`, `/stop`, `agent:bootstrap` หรือ `gateway:startup`

## เริ่มต้นอย่างรวดเร็ว

ลงทะเบียน Hook แบบมีชนิดข้อมูลด้วย `api.on(...)` จากจุดเข้าของ Plugin:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

export default definePluginEntry({
  id: "tool-preflight",
  name: "Tool Preflight",
  register(api) {
    api.on(
      "before_tool_call",
      async (event) => {
        if (event.toolName !== "web_search") {
          return;
        }

        return {
          requireApproval: {
            title: "Run web search",
            description: `Allow search query: ${String(event.params.query ?? "")}`,
            severity: "info",
            timeoutMs: 60_000,
          },
        };
      },
      { priority: 50 },
    );
  },
});
```

ตัวจัดการที่สามารถส่งคืนการตัดสินใจหรือการแก้ไขจะทำงานตามลำดับ
`priority` จากมากไปน้อย ส่วนตัวจัดการที่มีลำดับความสำคัญเท่ากันจะคงลำดับการลงทะเบียนไว้
ตัวจัดการที่ใช้สังเกตการณ์เท่านั้นจะทำงานแบบขนาน และการส่งการสังเกตการณ์แบบไม่รอผล
อาจทับซ้อนกับเหตุการณ์ในภายหลัง อย่าใช้ลำดับความสำคัญเพื่อจัดลำดับ
ผลข้างเคียงจากการสังเกตการณ์

`api.on(name, handler, opts?)` รองรับ:

| ตัวเลือก      | ผล                                                                                                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `priority`  | ลำดับการทำงาน ค่าที่สูงกว่าจะทำงานก่อน                                                                                                                                                                      |
| `timeoutMs` | งบเวลารอต่อ Hook เมื่อหมดเวลา OpenClaw จะหยุดรอตัวจัดการนั้นและดำเนินการต่อ โดยจะไม่ยกเลิกตัวจัดการหรือผลข้างเคียงของตัวจัดการ หากละไว้จะใช้ระยะหมดเวลาต่อ Hook เริ่มต้นของตัวรัน |

ผู้ดำเนินการสามารถกำหนดงบเวลาของ Hook ได้โดยไม่ต้องแก้ไขโค้ด Plugin:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "hooks": {
          "timeoutMs": 30000,
          "timeouts": {
            "before_prompt_build": 90000,
            "agent_end": 60000
          }
        }
      }
    }
  }
}
```

`hooks.timeouts.<hookName>` มีผลเหนือ `hooks.timeoutMs` ซึ่งมีผลเหนือค่า
`api.on(..., { timeoutMs })` ที่กำหนดโดยผู้สร้าง Plugin แต่ละค่าต้องเป็น
จำนวนเต็มบวกไม่เกิน 600000 ms ควรใช้การกำหนดค่าเฉพาะ Hook สำหรับ
Hook ที่ทราบว่าทำงานช้า เพื่อไม่ให้ Plugin หนึ่งได้รับงบเวลาที่ยาวขึ้นในทุกจุด

Promise ของตัวจัดการที่หมดเวลาจะยังคงทำงานต่อ เนื่องจาก Callback ของ Hook
ไม่ได้รับสัญญาณยกเลิก การส่ง Hook สามารถคืนโควตาการรับงานของ Gateway
ขณะที่งานของ Plugin นั้นยังคงดำเนินอยู่ Plugin ที่เป็นเจ้าของ
งานที่ใช้เวลานานต้องจัดเตรียมวงจรชีวิตสำหรับการยกเลิกและการปิดระบบด้วยตนเอง

Hook สำหรับแก้ไขข้อมูลขาออก `message_sending` และ `reply_payload_sending` ใช้ค่าเริ่มต้น
15 วินาทีต่อตัวจัดการ หากตัวใดหมดเวลา OpenClaw จะบันทึกข้อผิดพลาดของ Plugin
และดำเนินการต่อด้วยเพย์โหลดล่าสุด เพื่อให้เลนการส่งที่ทำงานแบบอนุกรม
สิ้นสุดได้ กำหนดงบเวลาต่อ Hook ให้มากขึ้นสำหรับ Plugin ที่ตั้งใจทำงานช้ากว่า
ก่อนส่งมอบ

Plugin ของช่องทางที่ใช้ `createReplyDispatcher` สามารถประกาศ
งบเวลาต่อขั้นที่เป็นจำนวนบวกและมากขึ้นด้วย `beforeDeliverOptions: { timeoutMs }` ได้เช่นกัน หรือเมื่อ
เพิ่มงานต่อท้ายด้วย `dispatcher.appendBeforeDeliver(handler, { timeoutMs })`
หากเจ้าของไม่ได้ประกาศงบเวลา Callback เหล่านั้นจะใช้ค่าเริ่มต้น
15 วินาทีเช่นเดียวกัน เพื่อไม่ให้ Callback ที่ค้างยึดเลนการส่งแบบอนุกรมไว้

Hook แต่ละรายการจะได้รับ `event.context.pluginConfig` ซึ่งเป็นการกำหนดค่าที่ผ่านการแก้ไขแล้วสำหรับ
Plugin ที่ลงทะเบียนตัวจัดการนั้น OpenClaw จะแทรกค่านี้แยกตามตัวจัดการโดยไม่
เปลี่ยนแปลงออบเจ็กต์เหตุการณ์ที่ใช้ร่วมกันซึ่ง Plugin อื่นมองเห็น

## แค็ตตาล็อก Hook

Hook ถูกจัดกลุ่มตามพื้นผิวที่ Hook ขยาย ชื่อที่เป็น **ตัวหนา** รองรับผลลัพธ์
การตัดสินใจ (บล็อก ยกเลิก เขียนทับ หรือขออนุมัติ) ส่วนที่เหลือ
ใช้สำหรับการสังเกตการณ์เท่านั้น

**เทิร์นของเอเจนต์**

| Hook                            | วัตถุประสงค์                                                                                  |
| ------------------------------- | ---------------------------------------------------------------------------------------- |
| `before_model_resolve`          | เขียนทับผู้ให้บริการหรือโมเดลก่อนโหลดข้อความของเซสชัน                                  |
| `agent_turn_prepare`            | ใช้การแทรกข้อมูลเทิร์นจาก Plugin ที่อยู่ในคิว และเพิ่มบริบทสำหรับเทิร์นเดียวกันก่อน Prompt Hook      |
| `before_prompt_build`           | เพิ่มบริบทแบบไดนามิกหรือข้อความ Prompt ระบบก่อนเรียกโมเดล                          |
| `before_agent_start`            | ระยะรวมสำหรับความเข้ากันได้เท่านั้น ควรใช้ Hook สองรายการข้างต้น                            |
| **`before_agent_run`**          | ตรวจสอบ Prompt สุดท้ายและข้อความของเซสชันก่อนส่งไปยังโมเดล และสามารถบล็อกการรันได้ |
| **`before_agent_reply`**        | ลัดวงจรเทิร์นของโมเดลด้วยการตอบกลับสังเคราะห์หรือไม่ตอบกลับ                           |
| **`before_agent_finalize`**     | ตรวจสอบคำตอบสุดท้ายตามธรรมชาติและขอให้โมเดลประมวลผลอีกหนึ่งรอบ                         |
| `agent_end`                     | สังเกตข้อความสุดท้าย สถานะความสำเร็จ และระยะเวลาการรัน                                  |
| `heartbeat_prompt_contribution` | เพิ่มบริบทเฉพาะ Heartbeat สำหรับ Plugin ตรวจสอบเบื้องหลังและวงจรชีวิต                  |

**การสังเกตการณ์การสนทนา**

| Hook                                      | วัตถุประสงค์                                                                                                            |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `model_call_started` / `model_call_ended` | เมตาดาต้าการเรียกผู้ให้บริการ/โมเดลที่ผ่านการล้างข้อมูลแล้ว ได้แก่ เวลา ผลลัพธ์ และแฮช ID คำขอแบบจำกัด ไม่มีเนื้อหา Prompt หรือการตอบกลับ |
| `llm_input`                               | อินพุตของผู้ให้บริการ: Prompt ระบบ, Prompt, ประวัติ                                                                     |
| `llm_output`                              | เอาต์พุตของผู้ให้บริการ การใช้งาน และ `contextTokenBudget` ที่ผ่านการแก้ไขแล้วเมื่อพร้อมใช้งาน                                       |

**เครื่องมือ**

| Hook                       | วัตถุประสงค์                                                   |
| -------------------------- | --------------------------------------------------------- |
| **`before_tool_call`**     | เขียนพารามิเตอร์เครื่องมือใหม่ บล็อกการดำเนินการ หรือขออนุมัติ |
| `after_tool_call`          | สังเกตผลลัพธ์ของเครื่องมือ ข้อผิดพลาด และระยะเวลา                |
| `resolve_exec_env`         | เพิ่มตัวแปรสภาพแวดล้อมที่ Plugin เป็นเจ้าของให้กับ `exec`   |
| **`tool_result_persist`**  | เขียนข้อความผู้ช่วยที่สร้างจากผลลัพธ์ของเครื่องมือใหม่ |
| **`before_message_write`** | ตรวจสอบหรือบล็อกการเขียนข้อความที่กำลังดำเนินอยู่ (พบได้น้อย)      |

**ข้อความและการส่งมอบ**

| Hook                            | วัตถุประสงค์                                                           |
| ------------------------------- | ----------------------------------------------------------------- |
| **`inbound_claim`**             | รับสิทธิ์จัดการข้อความขาเข้าก่อนกำหนดเส้นทางไปยังเอเจนต์ (การตอบกลับสังเคราะห์) |
| **`channel_pairing_requested`** | สังเกตคำขอจับคู่ DM ที่สร้างขึ้นใหม่                         |
| `message_received`              | สังเกตเนื้อหาขาเข้า ผู้ส่ง เธรด และเมตาดาต้า             |
| **`message_sending`**           | เขียนเนื้อหาขาออกใหม่หรือยกเลิกการส่งมอบ                       |
| **`reply_payload_sending`**     | เปลี่ยนแปลงหรือยกเลิกเพย์โหลดการตอบกลับที่ปรับให้อยู่ในรูปมาตรฐานแล้วก่อนส่งมอบ        |
| `message_sent`                  | สังเกตความสำเร็จหรือความล้มเหลวของการส่งมอบขาออก                      |
| **`before_dispatch`**           | ตรวจสอบหรือเขียนการส่งต่อขาออกใหม่ก่อนส่งมอบให้ช่องทาง    |
| **`reply_dispatch`**            | เข้าร่วมไปป์ไลน์การส่งการตอบกลับขั้นสุดท้าย                  |

**เซสชันและ Compaction**

| Hook                                     | วัตถุประสงค์                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `session_start` / `session_end`          | ติดตามขอบเขตวงจรชีวิตของเซสชัน `reason` เป็นหนึ่งใน `new`, `reset`, `idle`, `daily`, `compaction`, `deleted`, `shutdown`, `restart` หรือ `unknown` โดย `shutdown`/`restart` จะทำงานจากตัวปิดขั้นสุดท้ายของการปิด Gateway เมื่อโปรเซสหยุดหรือเริ่มใหม่ขณะที่ยังมีเซสชันทำงานอยู่ เพื่อให้ Plugin (หน่วยความจำและที่เก็บทรานสคริปต์) ปิดแถวที่หลงเหลือให้สมบูรณ์แทนที่จะปล่อยให้เปิดค้างข้ามการเริ่มใหม่ ตัวปิดขั้นสุดท้ายมีการจำกัดเวลาเพื่อไม่ให้ Plugin ที่ทำงานช้าบล็อก SIGTERM/SIGINT |
| `before_compaction` / `after_compaction` | สังเกตหรือใส่คำอธิบายประกอบรอบ Compaction                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `before_reset`                           | สังเกตเหตุการณ์รีเซ็ตเซสชัน (`/reset`, การรีเซ็ตโดยโปรแกรม)                                                                                                                                                                                                                                                                                                                                                                                                     |

**เอเจนต์ย่อย**

- `subagent_spawned` / `subagent_ended` - สังเกตการเริ่มต้นและการทำงานเสร็จสิ้นของเอเจนต์ย่อย
- `subagent_delivery_target` - ฮุกความเข้ากันได้สำหรับส่งมอบผลการทำงานเสร็จสิ้น เมื่อไม่มีการผูกเซสชันของแกนหลักที่สามารถกำหนดเส้นทางได้
- `subagent_spawning` - ฮุกความเข้ากันได้ที่เลิกใช้แล้ว ขณะนี้แกนหลักเตรียมการผูกเอเจนต์ย่อย `thread: true` ผ่านอะแดปเตอร์การผูกเซสชันของช่องทางก่อนที่ `subagent_spawned` จะทำงาน
- `subagent_spawned` จะมี `resolvedModel` และ `resolvedProvider` เมื่อ OpenClaw ระบุโมเดลเนทีฟของเซสชันลูกได้แล้วก่อนเริ่มต้น
- `subagent_ended` มี `targetSessionKey` (ข้อมูลประจำตัว - ตรงกับ `subagent_spawned.childSessionKey`), `targetKind` (`"subagent"` หรือ `"acp"`), `reason`, `outcome` ซึ่งไม่บังคับ (`"ok"`, `"error"`, `"timeout"`, `"killed"`, `"reset"` หรือ `"deleted"`), `error` ซึ่งไม่บังคับ, `runId`, `endedAt`, `accountId` และ `sendFarewell` โดย **ไม่มี** `agentId` หรือ `childSessionKey`; ใช้ `targetSessionKey` เพื่อเชื่อมโยงกับเหตุการณ์ `subagent_spawned` ที่ตรงกัน

**วงจรชีวิต**

| ฮุก                              | วัตถุประสงค์                                                                                              |
| -------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `gateway_start` / `gateway_stop` | เริ่มหรือหยุดบริการที่ Plugin เป็นเจ้าของพร้อมกับ Gateway                                                 |
| `deactivate`                     | นามแฝงเพื่อความเข้ากันได้ที่เลิกใช้แล้วสำหรับ `gateway_stop`; ใช้ `gateway_stop` ใน Plugin ใหม่                 |
| `cron_reconciled`                | ปรับสถานะให้สอดคล้องกับสถานะ Cron ทั้งหมดของ Gateway หลังเริ่มต้นหรือโหลดใหม่                            |
| `cron_changed`                   | สังเกตการเปลี่ยนแปลงวงจรชีวิต Cron ที่ Gateway เป็นเจ้าของ (เพิ่ม อัปเดต ลบ เริ่ม เสร็จสิ้น กำหนดเวลา) |
| **`before_install`**             | ตรวจสอบข้อมูลติดตั้ง Skills หรือ Plugin ที่จัดเตรียมไว้จากรันไทม์ Plugin ที่โหลดแล้ว                         |

### คำขอจับคู่ช่องทาง

ใช้ `channel_pairing_requested` เมื่อ Plugin ต้องแจ้งผู้ดูแลระบบหรือ
เขียนบันทึกการตรวจสอบหลังจากผู้ส่ง DM ที่ยังไม่ได้จับคู่สร้างคำขอจับคู่
ที่รอดำเนินการ ฮุกจะถูกเรียกเมื่อสร้างคำขอ โดยการส่ง
ข้อความตอบกลับการจับคู่ผ่านช่องทางจะไม่ล่าช้าเพราะตัวจัดการฮุกทำงานช้าหรือล้มเหลว

```typescript
api.on("channel_pairing_requested", async (event) => {
  await notifyOperator({
    text: `คำขอจับคู่ ${event.channel} ใหม่จาก ${event.senderId}: ${event.code}`,
  });
});
```

ฮุกนี้ใช้เพื่อสังเกตการณ์เท่านั้น โดยจะไม่อนุมัติ ปฏิเสธ ระงับ หรือเขียน
ข้อความตอบกลับการจับคู่ใหม่ เพย์โหลดประกอบด้วยช่องทาง, `accountId` ซึ่งไม่บังคับ,
`senderId` ที่จำกัดขอบเขตตามช่องทาง, `code` สำหรับการจับคู่ และข้อมูลเมตาของช่องทาง ให้ถือว่า
รหัสจับคู่เป็นข้อมูลรับรองสำหรับอนุมัติที่ใช้งานได้จริงและใช้ได้ครั้งเดียว และส่งรหัสดังกล่าวไปยัง
ปลายทางของผู้ดูแลระบบที่เชื่อถือได้เท่านั้น ให้ถือว่า `metadata` เป็นข้อความข้อมูลประจำตัว
ที่ผู้ส่งซึ่งไม่น่าเชื่อถือเป็นผู้ระบุ ฮุกนี้ไม่มีเนื้อหาหรือสื่อของข้อความขาเข้า

## ฮุกรันไทม์สำหรับดีบัก

ใช้ `before_model_resolve` เพื่อสลับผู้ให้บริการหรือโมเดลสำหรับหนึ่งรอบการทำงานของเอเจนต์ โดยฮุกนี้
ทำงานก่อนการระบุโมเดล ส่วน `llm_output` จะทำงานหลังจากความพยายามเรียกใช้โมเดล
สร้างเอาต์พุตจากผู้ช่วยแล้วเท่านั้น

หากต้องการหลักฐานของโมเดลเซสชันที่มีผลจริง ให้ตรวจสอบการลงทะเบียนรันไทม์ จากนั้น
ใช้ `openclaw sessions` หรือพื้นผิวเซสชัน/สถานะของ Gateway หากต้องการดีบัก
เพย์โหลดของผู้ให้บริการ ให้เริ่ม Gateway ด้วย `--raw-stream` และ
`--raw-stream-path <path>` เพื่อเขียนเหตุการณ์สตรีมโมเดลดิบลงในไฟล์ jsonl

## นโยบายการเรียกใช้เครื่องมือ

`before_tool_call` ได้รับ:

- `event.toolName`
- `event.params`
- `event.toolKind` และ `event.toolInputKind` ซึ่งไม่บังคับ เป็นตัวจำแนกที่โฮสต์มีอำนาจกำหนด
  สำหรับเครื่องมือที่ตั้งใจใช้ชื่อร่วมกัน ตัวอย่างเช่น การเรียก `exec` ใน
  โหมดโค้ดชั้นนอกจะใช้ `toolKind: "code_mode_exec"` และมี
  `toolInputKind: "javascript" | "typescript"` เมื่อทราบภาษาของอินพุต
- `event.derivedPaths` ซึ่งไม่บังคับ เป็นคำใบ้เส้นทางเป้าหมายที่โฮสต์อนุมานอย่างสุดความสามารถ
  สำหรับเอนเวโลปเครื่องมือที่รู้จักกันดี เช่น `apply_patch`; เส้นทางเหล่านี้อาจ
  ไม่ครบถ้วนหรือครอบคลุมเกินกว่าสิ่งที่เครื่องมือจะแตะต้องจริง (ตัวอย่างเช่น
  เมื่ออินพุตผิดรูปแบบหรือไม่สมบูรณ์)
- `event.runId` ซึ่งไม่บังคับ
- `event.toolCallId` ซึ่งไม่บังคับ
- ฟิลด์บริบท เช่น `ctx.agentId`, `ctx.sessionKey`, `ctx.sessionId`,
  `ctx.runId`, `ctx.toolKind`, `ctx.toolInputKind` และ `ctx.trace` สำหรับการวินิจฉัย

สามารถส่งคืน:

```typescript
type BeforeToolCallResult = {
  params?: Record<string, unknown>;
  block?: boolean;
  blockReason?: string;
  requireApproval?: {
    title: string;
    description: string;
    severity?: "info" | "warning" | "critical";
    timeoutMs?: number;
    /** @deprecated การอนุมัติที่ไม่ได้ข้อยุติจะถูกปฏิเสธเสมอ */
    timeoutBehavior?: "allow" | "deny";
    allowedDecisions?: Array<"allow-once" | "allow-always" | "deny">;
    pluginId?: string;
    onResolution?: (
      decision: "allow-once" | "allow-always" | "deny" | "timeout" | "cancelled",
    ) => Promise<void> | void;
  };
};
```

พฤติกรรมตัวป้องกันสำหรับฮุกวงจรชีวิตแบบมีชนิด:

- `block: true` เป็นผลลัพธ์สุดท้ายและจะข้ามตัวจัดการที่มีลำดับความสำคัญต่ำกว่า
- `block: false` จะถือว่าไม่มีการตัดสินใจ
- `params` จะเขียนพารามิเตอร์เครื่องมือใหม่สำหรับการดำเนินการ
- `requireApproval` จะหยุดการทำงานของเอเจนต์ชั่วคราวและสอบถามผู้ใช้ผ่านการอนุมัติของ Plugin
  `/approve` สามารถอนุมัติได้ทั้งการอนุมัติ exec และ Plugin สำหรับรีเลย์ `PreToolUse` แบบเนทีฟ
  ในโหมดรายงานของแอปเซิร์ฟเวอร์ Codex ระบบจะส่งต่อให้คำขออนุมัติของแอปเซิร์ฟเวอร์
  ที่ตรงกัน โปรดดู [รันไทม์ชุดควบคุม Codex](/th/plugins/codex-harness-runtime#hook-boundaries)
- `block: true` ที่มีลำดับความสำคัญต่ำกว่ายังคงบล็อกได้ หลังจากฮุกที่มีลำดับความสำคัญสูงกว่า
  ร้องขอการอนุมัติแล้ว
- `onResolution` จะได้รับผลการตัดสินใจสุดท้าย ได้แก่ `allow-once`, `allow-always`,
  `deny`, `timeout` หรือ `cancelled`

โปรดดู [คำขอสิทธิ์ของ Plugin](/th/plugins/plugin-permission-requests) สำหรับ
การกำหนดเส้นทางการอนุมัติ พฤติกรรมการตัดสินใจ และกรณีที่ควรใช้ `requireApproval` แทน
เครื่องมือที่ไม่บังคับหรือการอนุมัติ exec

Plugin ที่ต้องใช้นโยบายระดับโฮสต์สามารถลงทะเบียนนโยบายเครื่องมือที่เชื่อถือได้ด้วย
`api.registerTrustedToolPolicy(...)` นโยบายเหล่านี้ทำงานก่อนฮุก
`before_tool_call` ทั่วไปและก่อนการตัดสินใจของฮุกตามปกติ นโยบายที่เชื่อถือได้ของ
Plugin ที่รวมมาในชุดจะทำงานก่อน จากนั้นจึงเป็นนโยบายที่เชื่อถือได้ของ Plugin ที่ติดตั้ง
ตามลำดับการโหลด Plugin และฮุก `before_tool_call` ทั่วไปจะทำงานหลังจากนั้น Plugin ที่รวมมาในชุด
ยังคงใช้เส้นทางนโยบายที่เชื่อถือได้เดิม ส่วน Plugin ที่ติดตั้งต้องได้รับการเปิดใช้อย่างชัดแจ้ง
และประกาศรหัสนโยบายทุกรหัสใน `contracts.trustedToolPolicies`; รหัสที่ไม่ได้ประกาศ
จะถูกปฏิเสธก่อนลงทะเบียน รหัสนโยบายมีขอบเขตเฉพาะ Plugin ที่ลงทะเบียน
ดังนั้น Plugin ต่างกันจึงสามารถใช้รหัสภายในเดียวกันซ้ำได้ ใช้ระดับนี้เฉพาะ
สำหรับด่านที่โฮสต์เชื่อถือ เช่น นโยบายพื้นที่ทำงาน การบังคับใช้งบประมาณ หรือ
ความปลอดภัยของเวิร์กโฟลว์ที่สงวนไว้

### ฮุกสภาพแวดล้อม Exec

`resolve_exec_env` ช่วยให้ Plugin เพิ่มตัวแปรสภาพแวดล้อมให้กับการเรียกใช้เครื่องมือ `exec`
ก่อนรันคำสั่ง โดยได้รับ:

- `event.sessionKey`
- `event.toolName` ซึ่งขณะนี้เป็น `"exec"` เสมอ
- `event.host` ซึ่งเป็นหนึ่งใน `"gateway"`, `"sandbox"` หรือ `"node"`
- ฟิลด์บริบท เช่น `ctx.agentId`, `ctx.sessionKey`,
  `ctx.messageProvider` และ `ctx.channelId`

ส่งคืน `Record<string, string>` เพื่อผสานเข้ากับสภาพแวดล้อม exec ตัวจัดการ
ทำงานตามลำดับความสำคัญ โดยผลลัพธ์ที่มาทีหลังจะแทนที่ผลลัพธ์ก่อนหน้าสำหรับคีย์เดียวกัน

เอาต์พุตของฮุกจะถูกกรองผ่านนโยบายคีย์สภาพแวดล้อม exec ของโฮสต์ก่อน
ผสาน `PATH` จะถูกตัดทิ้งเสมอ (การระบุคำสั่งและการตรวจสอบไบนารีที่ปลอดภัย
ขึ้นอยู่กับค่านี้) คีย์ที่ไม่ถูกต้องและคีย์อันตรายที่ใช้แทนค่าของโฮสต์ เช่น `LD_*`,
`DYLD_*`, `NODE_OPTIONS`, ตัวแปรพร็อกซี (`HTTP_PROXY`, `HTTPS_PROXY`,
`ALL_PROXY`, `NO_PROXY`) และตัวแปรแทนค่า TLS (`NODE_TLS_REJECT_UNAUTHORIZED`,
`SSL_CERT_FILE` และค่าที่คล้ายกัน) จะถูกตัดทิ้ง สภาพแวดล้อมของ Plugin ที่ผ่านการกรองจะรวมอยู่
ในข้อมูลเมตาการอนุมัติ/การตรวจสอบของ Gateway และส่งต่อไปยังคำขอดำเนินการของโฮสต์ Node

### การคงผลลัพธ์เครื่องมือไว้

ผลลัพธ์เครื่องมือสามารถมี `details` แบบมีโครงสร้างสำหรับการเรนเดอร์ UI การวินิจฉัย
การกำหนดเส้นทางสื่อ หรือข้อมูลเมตาที่ Plugin เป็นเจ้าของ ให้ถือว่า `details` เป็นข้อมูลเมตารันไทม์
ไม่ใช่เนื้อหาพรอมต์:

- OpenClaw จะตัด `toolResult.details` ออกก่อนส่งซ้ำให้ผู้ให้บริการและก่อนเป็นอินพุตของ Compaction
  เพื่อไม่ให้ข้อมูลเมตากลายเป็นบริบทของโมเดล
- รายการเซสชันที่คงไว้จะเก็บเฉพาะ `details` ที่จำกัดขนาด รายละเอียดที่มีขนาดใหญ่เกินไปจะ
  ถูกแทนที่ด้วยสรุปแบบย่อและ `persistedDetailsTruncated: true`
- `tool_result_persist` และ `before_message_write` จะทำงานก่อน
  ขีดจำกัดสุดท้ายของการคงข้อมูลไว้ รักษา `details` ที่ส่งคืนให้มีขนาดเล็ก และหลีกเลี่ยงการใส่
  ข้อความที่เกี่ยวข้องกับพรอมต์ไว้เฉพาะใน `details`; ให้ใส่เอาต์พุตเครื่องมือที่โมเดลมองเห็นได้ใน
  `content`

## ฮุกพรอมต์และโมเดล

ใช้ฮุกเฉพาะเฟสสำหรับ Plugin ใหม่:

- `before_model_resolve`: รับเฉพาะพรอมต์ปัจจุบันและข้อมูลเมตาของไฟล์แนบ
  ส่งคืน `providerOverride` หรือ `modelOverride`
- `agent_turn_prepare`: รับพรอมต์ปัจจุบัน ข้อความเซสชันที่เตรียมไว้
  และการแทรกแบบครั้งเดียวแน่นอนที่อยู่ในคิวและถูกนำออกมาสำหรับเซสชันนี้
  ส่งคืน `prependContext` หรือ `appendContext`
- `before_prompt_build`: รับพรอมต์ปัจจุบันและข้อความเซสชัน
  ส่งคืน `prependContext`, `appendContext`, `systemPrompt`,
  `prependSystemContext` หรือ `appendSystemContext`
- `heartbeat_prompt_contribution`: ทำงานเฉพาะรอบ Heartbeat และส่งคืน
  `prependContext` หรือ `appendContext` มีไว้สำหรับตัวตรวจสอบเบื้องหลังที่
  ต้องสรุปสถานะปัจจุบันโดยไม่เปลี่ยนแปลงรอบที่ผู้ใช้เป็นผู้เริ่ม

`before_agent_start` ยังคงมีไว้เพื่อความเข้ากันได้ ควรเลือกใช้ฮุกที่ชัดเจน
ข้างต้น เพื่อให้ Plugin ไม่ต้องพึ่งพาเฟสรวมแบบเดิม

`before_agent_run` ทำงานหลังการสร้างพรอมต์และก่อนอินพุตใดๆ ของโมเดล
รวมถึงการโหลดรูปภาพเฉพาะพรอมต์และการสังเกตการณ์ `llm_input` โดยรับ
อินพุตผู้ใช้ปัจจุบันเป็น `prompt` รวมถึงประวัติเซสชันที่โหลดไว้ใน `messages`
และพรอมต์ระบบที่ใช้งานอยู่ ส่งคืน `{ outcome: "block", reason, message? }`
เพื่อหยุดการทำงานก่อนโมเดลอ่านพรอมต์ `reason` เป็นค่าภายใน;
`message` เป็นค่าทดแทนที่แสดงต่อผู้ใช้ รองรับเฉพาะผลลัพธ์ `pass` และ `block`;
รูปแบบการตัดสินใจที่ไม่รองรับจะปฏิเสธโดยปริยาย

เมื่อการทำงานถูกบล็อก OpenClaw จะจัดเก็บเฉพาะข้อความทดแทนใน
`message.content` พร้อมข้อมูลเมตาการบล็อกที่ไม่ละเอียดอ่อน เช่น รหัส
Plugin ที่บล็อกและการประทับเวลา ข้อความต้นฉบับของผู้ใช้จะไม่ถูกเก็บไว้ในทรานสคริปต์
หรือบริบทในอนาคต เหตุผลการบล็อกภายในถือเป็นข้อมูลละเอียดอ่อนและ
จะไม่รวมอยู่ในเพย์โหลดของทรานสคริปต์ ประวัติ การกระจายข้อความ บันทึก และการวินิจฉัย
การสังเกตการณ์ควรใช้ฟิลด์ที่ผ่านการล้างข้อมูลแล้ว เช่น รหัสผู้บล็อก ผลลัพธ์
การประทับเวลา หรือหมวดหมู่ที่ปลอดภัย

`before_agent_start` และ `agent_end` จะมี `event.runId` เมื่อ OpenClaw สามารถ
ระบุการทำงานที่ใช้งานอยู่ได้ โดยค่าเดียวกันนี้ยังอยู่ใน `ctx.runId` การทำงานที่ขับเคลื่อนด้วย
Cron จะแสดง `ctx.jobId` (รหัสงาน Cron ต้นทาง) ในบริบทรอบการทำงานของเอเจนต์ด้วย
เพื่อให้ฮุกสามารถจำกัดขอบเขตเมตริก ผลข้างเคียง หรือสถานะไว้กับงานที่กำหนดเวลาไว้
งานใดงานหนึ่งโดยเฉพาะ `ctx.jobId` ไม่ได้เป็นส่วนหนึ่งของบริบทเครื่องมือ `before_tool_call`

สำหรับการเรียกใช้ที่มาจากช่องทาง `ctx.channel` และ `ctx.messageProvider` ระบุ
พื้นผิวของผู้ให้บริการ เช่น `discord` หรือ `telegram` ขณะที่ `ctx.channelId` คือ
ตัวระบุเป้าหมายการสนทนาเมื่อ OpenClaw สามารถอนุมานได้จาก
คีย์เซสชันหรือเมทาดาทาการส่ง

เมื่อมีข้อมูลอัตลักษณ์ของผู้ส่ง บริบทของฮุกเอเจนต์จะประกอบด้วย:

- `ctx.senderId` - ID ผู้ส่งที่มีขอบเขตเฉพาะช่องทาง (เช่น Feishu `open_id`, ID ผู้ใช้
  Discord) จะมีค่าเมื่อการเรียกใช้มีต้นทางจากข้อความของผู้ใช้ที่มี
  เมทาดาทาผู้ส่งที่ทราบ
- `ctx.chatId` - ตัวระบุการสนทนาแบบเนทีฟของการขนส่ง (เช่น Feishu
  `chat_id`, Telegram `chat_id`) จะมีค่าเมื่อช่องทางต้นทาง
  ให้ ID การสนทนาแบบเนทีฟ
- `ctx.channelContext.sender.id` - ID ผู้ส่งเดียวกับ `ctx.senderId` ซึ่งอยู่ภายใต้
  ออบเจ็กต์ที่ช่องทางเป็นเจ้าของและ Plugin สามารถขยายด้วยฟิลด์เฉพาะช่องทางได้
- `ctx.channelContext.chat.id` - ID การสนทนาเดียวกับ `ctx.chatId`
  ซึ่งอยู่ภายใต้ออบเจ็กต์ที่ช่องทางเป็นเจ้าของและ Plugin สามารถขยายด้วย
  ฟิลด์เฉพาะช่องทางได้

แกนหลักกำหนดเฉพาะฟิลด์ `id` ที่ซ้อนอยู่เท่านั้น Plugin ช่องทางที่ส่ง
เมทาดาทาผู้ส่งหรือแชตที่ละเอียดกว่าผ่านตัวช่วยขาเข้าสามารถเพิ่มฟิลด์ให้
`PluginHookChannelSenderContext` หรือ `PluginHookChannelChatContext` จาก
`openclaw/plugin-sdk/channel-inbound` ได้:

```ts
declare module "openclaw/plugin-sdk/channel-inbound" {
  interface PluginHookChannelSenderContext {
    unionId?: string;
    userId?: string;
  }
}
```

Plugin ช่องทางส่งฟิลด์เหล่านั้นผ่านตัวช่วย SDK ขาเข้า:

```ts
buildChannelInboundEventContext({
  // ...
  channelContext: {
    sender: { id: senderOpenId, unionId, userId },
    chat: { id: chatId },
  },
});
```

ฟิลด์เหล่านี้เป็นทางเลือกและจะไม่มีอยู่สำหรับการเรียกใช้ที่มีต้นทางจากระบบ (Heartbeat,
Cron, เหตุการณ์ exec)

`ctx.senderExternalId` ยังคงอยู่ในฐานะฟิลด์ความเข้ากันได้ระดับซอร์สที่เลิกใช้แล้วสำหรับ
Plugin รุ่นเก่า แกนหลักจะไม่ใส่ค่าให้ฟิลด์นี้ อัตลักษณ์ผู้ส่งเฉพาะช่องทางแบบใหม่
ควรอยู่ภายใต้ `ctx.channelContext.sender` ผ่านการเพิ่มส่วนขยายให้โมดูล

`agent_end` เป็นฮุกสำหรับการสังเกต เส้นทาง Gateway และฮาร์เนสแบบถาวรจะเรียกใช้
แบบไม่รอผลหลังจบรอบ ขณะที่เส้นทาง CLI แบบครั้งเดียวที่มีอายุสั้นจะรอ
พรอมิสของฮุกก่อนล้างกระบวนการ เพื่อให้ Plugin ที่เชื่อถือได้สามารถเขียน
ข้อมูลการสังเกตการณ์ปลายทางหรือบันทึกสถานะให้เสร็จ ตัวเรียกใช้ฮุกกำหนดเวลาหมด 30 วินาที
เพื่อไม่ให้ Plugin หรือปลายทางการฝังที่ค้างทำให้พรอมิสของฮุก
รออยู่ตลอดไป ระบบจะบันทึกเวลาหมดและ OpenClaw จะทำงานต่อ โดยจะไม่
ยกเลิกงานเครือข่ายที่ Plugin เป็นเจ้าของ เว้นแต่ Plugin จะใช้สัญญาณยกเลิก
ของตนเองด้วย

ใช้ `model_call_started` และ `model_call_ended` สำหรับเทเลเมทรีการเรียกผู้ให้บริการ
ที่ไม่ควรได้รับพรอมต์ดิบ ประวัติ การตอบกลับ ส่วนหัว เนื้อหาคำขอ
หรือ ID คำขอของผู้ให้บริการ ฮุกเหล่านี้มีเมทาดาทาที่คงที่ เช่น
`runId`, `callId`, `provider`, `model`, `api`/`transport` ที่เป็นทางเลือก, ค่าปลายทาง
`durationMs`/`outcome` และ `upstreamRequestIdHash` เมื่อ OpenClaw สามารถอนุมาน
แฮช ID คำขอของผู้ให้บริการที่มีขอบเขตจำกัดได้ เมื่อรันไทม์แก้ไข
เมทาดาทาหน้าต่างบริบทแล้ว เหตุการณ์และบริบทของฮุกจะมี
`contextTokenBudget` ซึ่งเป็นงบโทเค็นที่มีผลหลังใช้ขีดจำกัดของโมเดล/การกำหนดค่า/เอเจนต์
รวมถึง `contextWindowSource` และ `contextWindowReferenceTokens` เมื่อมีการใช้
ขีดจำกัดที่ต่ำกว่า

`before_agent_finalize` จะทำงานเฉพาะเมื่อฮาร์เนสกำลังจะยอมรับคำตอบสุดท้ายตามธรรมชาติ
จากผู้ช่วย ไม่ใช่เส้นทางการยกเลิก `/stop` และจะไม่
ทำงานเมื่อผู้ใช้ยกเลิกรอบ ส่งคืน `{ action: "revise", reason }` เพื่อขอให้
ฮาร์เนสเรียกโมเดลอีกหนึ่งรอบก่อนสรุปผล, `{ action:
"finalize", reason? }` เพื่อบังคับสรุปผล หรือไม่ส่งคืนผลลัพธ์เพื่อดำเนินการต่อ
ตัวจัดการมีงบเวลาเริ่มต้น 15 วินาที เมื่อหมดเวลา OpenClaw จะบันทึกความล้มเหลวและ
ดำเนินการต่อด้วยคำตอบสุดท้ายเดิม
ฮุกเนทีฟ `Stop` ของ Codex จะถูกส่งต่อมายังฮุกนี้ในรูปแบบการตัดสินใจ
`before_agent_finalize` ของ OpenClaw

เมื่อส่งคืน `action: "revise"` Plugin สามารถใส่เมทาดาทา `retry` เพื่อ
ทำให้การเรียกโมเดลรอบเพิ่มเติมมีขอบเขตจำกัดและปลอดภัยต่อการเล่นซ้ำ:

```typescript
type BeforeAgentFinalizeRetry = {
  instruction: string;
  idempotencyKey?: string;
  maxAttempts?: number;
};
```

`instruction` จะถูกต่อท้ายเหตุผลการแก้ไขที่ส่งไปยังฮาร์เนส
`idempotencyKey` ช่วยให้โฮสต์นับจำนวนการลองใหม่สำหรับคำขอ Plugin เดียวกัน
ในการตัดสินใจสรุปผลที่เทียบเท่ากัน และ `maxAttempts` จำกัดจำนวนรอบเพิ่มเติม
ที่โฮสต์จะอนุญาตก่อนดำเนินการต่อด้วยคำตอบสุดท้ายตามธรรมชาติ

Plugin ที่ไม่ได้รวมมากับระบบและต้องการฮุกการสนทนาดิบ (`before_model_resolve`,
`before_agent_reply`, `llm_input`, `llm_output`, `before_agent_finalize`,
`agent_end` หรือ `before_agent_run`) ต้องตั้งค่า:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "hooks": {
          "allowConversationAccess": true
        }
      }
    }
  }
}
```

สามารถปิดใช้งานฮุกที่แก้ไขพรอมต์และการแทรกข้อมูลถาวรสำหรับรอบถัดไปเป็นราย
Plugin ได้ด้วย `plugins.entries.<id>.hooks.allowPromptInjection=false`

### ส่วนขยายเซสชันและการแทรกข้อมูลสำหรับรอบถัดไป

Plugin เวิร์กโฟลว์สามารถเก็บสถานะเซสชันขนาดเล็กที่เข้ากันได้กับ JSON อย่างถาวรด้วย
`api.session.state.registerSessionExtension(...)` และอัปเดตผ่านเมธอด
`sessions.pluginPatch` ของ Gateway แถวเซสชันจะแสดงสถานะส่วนขยายที่ลงทะเบียนไว้
ผ่าน `pluginExtensions` ทำให้ Control UI และไคลเอนต์อื่น
แสดงสถานะที่ Plugin เป็นเจ้าของได้โดยไม่ต้องทราบรายละเอียดภายในของ Plugin
`api.registerSessionExtension(...)` ยังคงใช้งานได้แต่เลิกใช้แล้ว โดยควรใช้
เนมสเปซ `api.session.state` แทน

ใช้ `api.session.workflow.enqueueNextTurnInjection(...)` เมื่อ Plugin ต้องการ
ส่งบริบทแบบถาวรไปยังรอบโมเดลถัดไปเพียงครั้งเดียวพอดี (`api.enqueueNextTurnInjection(...)` ระดับบนสุด
เป็นนามแฝงที่เลิกใช้แล้วและมีลักษณะการทำงานเดียวกัน)
OpenClaw จะระบายรายการแทรกที่อยู่ในคิวก่อนฮุกพรอมต์ ทิ้งรายการแทรก
ที่หมดอายุ และขจัดรายการซ้ำตาม `idempotencyKey` ต่อ Plugin นี่คือ
จุดเชื่อมต่อที่เหมาะสมสำหรับการดำเนินการต่อหลังการอนุมัติ สรุปนโยบาย เดลตาจากตัวตรวจสอบ
เบื้องหลัง และการดำเนินคำสั่งต่อที่ควรให้โมเดลเห็นใน
รอบถัดไป แต่ไม่ควรกลายเป็นข้อความพรอมต์ระบบถาวร

ความหมายของการล้างข้อมูลเป็นส่วนหนึ่งของสัญญา การล้างส่วนขยายเซสชันและ
คอลแบ็กการล้างวงจรชีวิตรันไทม์จะได้รับ `reset`, `delete`, `disable` หรือ
`restart` โฮสต์จะลบสถานะส่วนขยายเซสชันถาวรของ Plugin เจ้าของ
และรายการแทรกสำหรับรอบถัดไปที่ค้างอยู่เมื่อรีเซ็ต/ลบ/ปิดใช้งาน ส่วนการรีสตาร์ต
จะเก็บสถานะเซสชันถาวรไว้ ขณะที่คอลแบ็กการล้างข้อมูลช่วยให้ Plugin ปล่อย
งานของตัวจัดกำหนดการ บริบทการเรียกใช้ และทรัพยากรนอกแบนด์อื่นๆ ของรุ่น
รันไทม์เดิม

## ฮุกข้อความ

ใช้ฮุกข้อความสำหรับการกำหนดเส้นทางและนโยบายการส่งในระดับช่องทาง:

- `message_received`: สังเกตเนื้อหาขาเข้า ผู้ส่ง `threadId`,
  `messageId`, `senderId`, ความสัมพันธ์กับการเรียกใช้/เซสชันที่เป็นทางเลือก และเมทาดาทา
- `message_sending`: เขียน `content` ใหม่หรือส่งคืน `{ cancel: true }`
- `reply_payload_sending`: เขียนออบเจ็กต์ `ReplyPayload` ที่ปรับให้เป็นมาตรฐานใหม่
  (รวมถึง `presentation`, `delivery`, การอ้างอิงสื่อ และข้อความ) หรือส่งคืน
  `{ cancel: true }`
- `message_sent`: สังเกตความสำเร็จหรือความล้มเหลวขั้นสุดท้าย

สำหรับการตอบกลับ TTS ที่มีเฉพาะเสียง `content` อาจมีบทถอดเสียงพูด
ที่ซ่อนอยู่ แม้เพย์โหลดของช่องทางจะไม่มีข้อความ/คำบรรยายที่มองเห็นได้
การเขียน `content` นั้นใหม่จะอัปเดตเฉพาะบทถอดเสียงที่ฮุกมองเห็น โดยจะไม่
แสดงผลเป็นคำบรรยายสื่อ

เหตุการณ์ `reply_payload_sending` อาจมี `usageState` ซึ่งเป็นสแนปช็อตแบบสด
ของโมเดล/การใช้งาน/บริบทต่อรอบตามความพยายามที่ดีที่สุด การส่งแบบถาวร การเล่นซ้ำที่กู้คืน
และการตอบกลับที่ไม่มีความสัมพันธ์กับการเรียกใช้ที่แน่นอนจะไม่มีค่านี้

บริบทฮุกข้อความเปิดเผยฟิลด์ความสัมพันธ์ที่คงที่เมื่อมี:
`ctx.sessionKey`, `ctx.runId`, `ctx.messageId`, `ctx.senderId`, `ctx.trace`,
`ctx.traceId`, `ctx.spanId`, `ctx.parentSpanId` และ `ctx.callDepth` บริบทขาเข้า
และ `before_dispatch` ยังเปิดเผยเมทาดาทาการตอบกลับเมื่อช่องทาง
มีข้อมูลข้อความที่ยกมาซึ่งกรองตามการมองเห็นแล้ว ได้แก่ `replyToId`, `replyToIdFull`,
`replyToBody`, `replyToSender` และ `replyToIsQuote` ควรใช้
ฟิลด์ระดับเฟิร์สต์คลาสเหล่านี้ก่อนอ่านเมทาดาทาแบบเดิม

ควรใช้ฟิลด์ `threadId` และ `replyToId` ที่มีชนิดข้อมูลก่อนใช้เมทาดาทา
เฉพาะช่องทาง

กฎการตัดสินใจ:

- `message_sending` ที่มี `cancel: true` ถือเป็นขั้นสุดท้าย
- `message_sending` ที่มี `cancel: false` ถือว่าไม่มีการตัดสินใจ
- `content` ที่เขียนใหม่จะดำเนินต่อไปยังฮุกที่มีลำดับความสำคัญต่ำกว่า เว้นแต่ฮุกภายหลัง
  จะยกเลิกการส่ง
- `reply_payload_sending` ทำงานหลังการปรับเพย์โหลดให้เป็นมาตรฐานและก่อนการส่งผ่านช่องทาง
  รวมถึงการตอบกลับที่กำหนดเส้นทางกลับไปยังช่องทางต้นทาง
  ตัวจัดการทำงานตามลำดับ และแต่ละตัวจะเห็นเพย์โหลดล่าสุดที่สร้างโดย
  ตัวจัดการที่มีลำดับความสำคัญสูงกว่า
- เพย์โหลด `reply_payload_sending` จะไม่เปิดเผยเครื่องหมายความเชื่อถือของรันไทม์ เช่น
  `trustedLocalMedia` โดย Plugin สามารถแก้ไขรูปร่างเพย์โหลดได้ แต่ไม่สามารถมอบความเชื่อถือ
  ให้สื่อภายในเครื่องได้
- `message_sending` สามารถส่งคืน `cancelReason` และ `metadata` ที่มีขอบเขตจำกัดพร้อม
  การยกเลิก API วงจรชีวิตข้อความแบบใหม่จะแสดงสิ่งนี้เป็นผลลัพธ์
  การระงับการส่งพร้อมเหตุผล `cancelled_by_message_sending_hook` ส่วนการส่งโดยตรง
  แบบเดิมยังคงส่งคืนอาร์เรย์ผลลัพธ์ว่างเพื่อความเข้ากันได้
- `message_sent` ใช้เพื่อการสังเกตเท่านั้น ความล้มเหลวของตัวจัดการจะถูกบันทึกและไม่
  เปลี่ยนผลลัพธ์การส่ง

## ฮุกการติดตั้ง

ใช้ `security.installPolicy` สำหรับการตัดสินใจอนุญาต/บล็อกที่ผู้ปฏิบัติงานเป็นเจ้าของ
นโยบายนั้นทำงานจากการกำหนดค่า OpenClaw ครอบคลุมเส้นทางการติดตั้งและอัปเดต CLI
และจะปฏิเสธโดยค่าเริ่มต้นเมื่อเปิดใช้งานแต่ไม่พร้อมใช้งาน

`before_install` เป็นฮุกวงจรชีวิตรันไทม์ของ Plugin โดยจะทำงานหลัง
`security.installPolicy` เฉพาะในกระบวนการ OpenClaw ที่โหลดฮุกของ Plugin
ไว้แล้ว เช่น โฟลว์การติดตั้งที่มี Gateway รองรับ ฮุกนี้มีประโยชน์สำหรับ
การสังเกต คำเตือน และการตรวจสอบความเข้ากันได้ที่ Plugin เป็นเจ้าของ แต่ไม่ใช่
ขอบเขตความปลอดภัยหลักขององค์กรหรือโฮสต์สำหรับการติดตั้ง ฟิลด์
`builtinScan` ยังคงอยู่ในเพย์โหลดเหตุการณ์เพื่อความเข้ากันได้ แต่
OpenClaw จะไม่เรียกใช้การบล็อกโค้ดอันตรายขณะติดตั้งที่มีมาให้ในตัวอีกต่อไป ดังนั้นจึงเป็นผลลัพธ์
`ok` ว่าง ส่งคืนข้อค้นพบเพิ่มเติมหรือ
`{ block: true, blockReason }` เพื่อหยุดการติดตั้งในกระบวนการนั้น

`block: true` ถือเป็นขั้นสุดท้าย `block: false` ถือว่าไม่มีการตัดสินใจ ความล้มเหลวของตัวจัดการ
จะบล็อกการติดตั้งโดยปฏิเสธตามค่าเริ่มต้น

## วงจรชีวิต Gateway

ใช้ `gateway_start` เพื่อเริ่มบริการทั่วไปของ Plugin และ `gateway_stop` เพื่อ
ล้างทรัพยากรที่ทำงานระยะยาว ตัวจัดกำหนดการ Cron อาจยังโหลดอยู่เมื่อ
`gateway_start` ทำงาน ดังนั้นอย่าใช้ค่านี้เป็นสัญญาณพื้นฐานสำหรับการฉายภาพ
Cron ภายนอก

อย่าพึ่งพาฮุกภายใน `gateway:startup` สำหรับบริการรันไทม์
ที่ Plugin เป็นเจ้าของ

`cron_reconciled` จะทำงานหลังตัวจัดกำหนดการ Cron ของ Gateway และตัวเฝ้าดูเมื่อออก
ได้กระทบยอดสถานะถาวรแล้ว โดยจะทำงานทั้งตอนเริ่มต้นครั้งแรก
และเมื่อแทนที่ตัวจัดกำหนดการระหว่างการโหลดการกำหนดค่าใหม่ เหตุการณ์จะรายงาน
`reason` (`startup` หรือ `reload`) และสถานะ `enabled` ที่มีผล แม้ปิดใช้งาน
Cron ก็ยังส่งเหตุการณ์พร้อม `enabled: false` ทำให้การฉายภาพภายนอกสามารถ
ล้างการปลุกที่ค้างอยู่ได้ ใช้ `ctx.getCron?.()` สำหรับอินสแตนซ์ตัวจัดกำหนดการที่แน่นอนซึ่ง
กระทบยอดเสร็จแล้ว การโหลดใหม่ภายหลังจะไม่กำหนดเป้าหมายคอลแบ็กนั้นใหม่
`ctx.abortSignal` เป็นเจ้าของสแนปช็อตตัวจัดกำหนดการเดียวกัน Gateway จะยกเลิกสัญญาณนี้
ทันทีที่ตัวจัดกำหนดการที่ใหม่กว่าพร้อมทำงานหรือเริ่มปิดระบบ ส่งต่อสัญญาณนี้ไปยังทุก
ผลข้างเคียงแบบถาวร และอย่ายอมรับสแนปช็อตหลังจากสัญญาณถูกยกเลิก
นี่คือสัญญาณวงจรชีวิตตัวจัดกำหนดการ ไม่ใช่สัญญาณการเปิดใช้งาน Plugin:
การโหลดใหม่แบบร้อนเฉพาะ Plugin จะไม่เล่นสัญญาณนี้ซ้ำ ผู้บริโภคที่เพิ่งเปิดใช้งานจะได้รับ
ข้อมูลพื้นฐานครั้งแรกเมื่อมีการแทนที่ตัวจัดกำหนดการครั้งถัดไปหรือเมื่อ Gateway เริ่มทำงาน

เช่นเดียวกับฮุกสำหรับการสังเกตอื่นๆ คอลแบ็ก `gateway_start` และ `cron_reconciled`
สามารถทำงานซ้อนกันได้ หากตัวจัดการทั้งสองใช้การเริ่มต้น Plugin ร่วมกัน ให้ประสานงาน
ด้วยพรอมิสความพร้อมภายใน Plugin แทนการพึ่งพาลำดับคอลแบ็ก

`cron_changed` จะทำงานสำหรับเหตุการณ์วงจรชีวิต Cron ที่ Gateway เป็นเจ้าของ โดยมีเพย์โหลดเหตุการณ์แบบระบุชนิดซึ่งครอบคลุมเหตุผล `added`, `updated`, `removed`, `started`, `finished`
และ `scheduled` เหตุการณ์จะมีสแนปช็อต `PluginHookGatewayCronJob`
(รวมถึง `state.nextRunAtMs`, `state.lastRunStatus` และ
`state.lastError` เมื่อมี) พร้อม `PluginHookGatewayCronDeliveryStatus`
ของ `not-requested` | `delivered` | `not-delivered` | `unknown` เหตุการณ์ที่ถูกลบ
เป็นแบบหลังคอมมิต กล่าวคือจะทำงานหลังจากลบข้อมูลแบบคงทนสำเร็จแล้วเท่านั้น และยังคงมี
สแนปช็อตของงานที่ถูกลบ เพื่อให้ตัวจัดกำหนดการภายนอกกระทบยอดสถานะได้

เหตุการณ์ `scheduled` เป็นแบบหลังคอมมิต กล่าวคือจะทำงานเฉพาะหลังจากการเขียนแบบคงทน
ที่สำเร็จเปลี่ยนค่า `nextRunAtMs` ที่มีผลของงานที่มีอยู่ โดยไม่รวมเหตุการณ์วงจรชีวิต
`added`, `updated` หรือ `removed` ที่ระบุชัดเจนของงานนั้น ค่า
`event.nextRunAtMs` ระดับบนสุดคือเวลาปลุกครั้งถัดไปที่คอมมิตแล้ว เมื่อไม่มีค่านี้ งานจะ
ไม่มีเวลาปลุกครั้งถัดไป ให้ถือว่าเหตุการณ์เหล่านี้เป็นคำใบ้สำหรับการกระทบยอด ไม่ใช่บันทึกเดลตา
ที่เรียงลำดับ ใช้เป็นคำใบ้ที่รวมกันได้เพื่ออ่านตัวจัดกำหนดการที่
`cron_reconciled` จับไว้ล่าสุดอีกครั้ง อย่านำตัวจัดกำหนดการจากบริบท `cron_changed` มาใช้
ให้ OpenClaw เป็นแหล่งข้อมูลจริงสำหรับการตรวจสอบว่าถึงกำหนดและการดำเนินการ

### การฉายภาพ Cron ภายนอกอย่างปลอดภัย

ฉายภาพสแนปช็อตเวลาปลุกที่สมบูรณ์ แทนการส่งต่อเดลตาของเหตุการณ์ Cron การดำเนินการ
`replaceAll` ของอะแดปเตอร์ภายนอกต้องเป็นแบบอะตอมมิกและทำซ้ำได้โดยให้ผลเหมือนเดิม และ
ต้องเสร็จสิ้นหลังจากโฮสต์ยอมรับสแนปช็อตแบบคงทนแล้วเท่านั้น นอกจากนี้ยังต้อง
ปฏิบัติตามสัญญาณยกเลิกที่ส่งมา หากสัญญาณยกเลิกก่อนการยอมรับแบบคงทน
อะแดปเตอร์ต้องไม่ยอมรับสแนปช็อตนั้น

รูปแบบนี้ทำให้มีเวิร์กเกอร์สถานะล่าสุดที่กำลังทำงานอยู่เพียงหนึ่งตัว เฉพาะ `cron_reconciled`
เท่านั้นที่นำอินสแตนซ์ตัวจัดกำหนดการมาใช้ ส่วน `cron_changed` เพียงขอให้เวิร์กเกอร์นั้นอ่าน
อินสแตนซ์ที่เป็นแหล่งข้อมูลจริงอีกครั้ง ดังนั้นคำใบ้ที่มาช้าจึงไม่สามารถคืนค่าตัวจัดกำหนดการรุ่นเก่าได้
รีวิชันที่ใหม่กว่าจะยกเลิกความพยายามของโฮสต์ที่กำลังทำงาน ก่อนที่จะยอมรับ
สแนปช็อตที่ล้าสมัยได้

```typescript
import { setTimeout as sleep } from "node:timers/promises";
import type { OpenClawPluginApi } from "openclaw/plugin-sdk/plugin-entry";

type ExternalWake = { jobId: string; runAtMs: number };

type ExternalWakeHost = {
  replaceAll(wakes: readonly ExternalWake[], options: { signal: AbortSignal }): Promise<void>;
  close(): Promise<void>;
};

type CronReader = {
  list(options: { includeDisabled: true }): Promise<
    Array<{
      id: string;
      enabled?: boolean;
      state?: { nextRunAtMs?: number };
    }>
  >;
};

export function registerCronProjection(api: OpenClawPluginApi, host: ExternalWakeHost) {
  const lifecycle = new AbortController();
  let cron: CronReader | undefined;
  let enabled = false;
  let hasBaseline = false;
  let reconciliationSignal: AbortSignal | undefined;
  let requestedRevision = 0;
  let appliedRevision = 0;
  let worker = Promise.resolve();
  let activeAttempt: AbortController | undefined;

  const projectLatest = async () => {
    let retryMs = 1_000;

    while (!lifecycle.signal.aborted && appliedRevision < requestedRevision) {
      const ownerSignal = reconciliationSignal;
      if (!ownerSignal || ownerSignal.aborted) {
        return;
      }
      const targetRevision = requestedRevision;
      const attempt = new AbortController();
      const signal = AbortSignal.any([lifecycle.signal, ownerSignal, attempt.signal]);
      activeAttempt = attempt;

      try {
        const jobs = enabled && cron ? await cron.list({ includeDisabled: true }) : [];
        if (signal.aborted || targetRevision !== requestedRevision) {
          continue;
        }
        const wakes = jobs
          .flatMap((job): ExternalWake[] => {
            const runAtMs = job.enabled === false ? undefined : job.state?.nextRunAtMs;
            return runAtMs === undefined ? [] : [{ jobId: job.id, runAtMs }];
          })
          .sort((a, b) => a.runAtMs - b.runAtMs || a.jobId.localeCompare(b.jobId));

        await host.replaceAll(wakes, { signal });
        if (signal.aborted || targetRevision !== requestedRevision) {
          continue;
        }
        appliedRevision = targetRevision;
        retryMs = 1_000;
      } catch {
        if (lifecycle.signal.aborted || ownerSignal.aborted) {
          return;
        }
        if (attempt.signal.aborted) {
          continue;
        }
        api.logger.warn(`การฉายภาพ Cron ภายนอกล้มเหลว จะลองอีกครั้งใน ${retryMs}ms`);
        try {
          await sleep(retryMs, undefined, { signal });
        } catch {
          if (lifecycle.signal.aborted) {
            return;
          }
          if (attempt.signal.aborted) {
            continue;
          }
        }
        retryMs = Math.min(retryMs * 2, 30_000);
      } finally {
        if (activeAttempt === attempt) {
          activeAttempt = undefined;
        }
      }
    }
  };

  const requestProjection = () => {
    const targetRevision = ++requestedRevision;
    activeAttempt?.abort();
    worker = worker.then(async () => {
      if (!lifecycle.signal.aborted && appliedRevision < targetRevision) {
        await projectLatest();
      }
    });
    return worker;
  };

  api.on("cron_reconciled", (event, ctx) => {
    const reconciledCron = ctx.getCron?.();
    if (event.enabled && !reconciledCron) {
      api.logger.warn("การกระทบยอด Cron ไม่ได้เปิดเผยตัวจัดกำหนดการ");
      return;
    }
    cron = reconciledCron;
    enabled = event.enabled;
    hasBaseline = true;
    reconciliationSignal = ctx.abortSignal;
    return requestProjection();
  });

  api.on("cron_changed", () => {
    if (hasBaseline) {
      return requestProjection();
    }
  });

  api.on("gateway_stop", async () => {
    lifecycle.abort();
    await worker;
    await host.close();
  });
}
```

เมื่อ `cron_reconciled` รายงาน `enabled: false` พาธเดียวกันจะเรียก
`replaceAll([])` และล้างเวลาปลุกภายนอกที่ล้าสมัย การลองใหม่/การหน่วงเวลาแบบเพิ่มขึ้นในตัวอย่างนี้
เป็นแบบเฉพาะโปรเซส และถือว่าความล้มเหลวของอะแดปเตอร์ขณะรันเป็นเหตุขัดข้องชั่วคราว ให้ตรวจสอบ
การกำหนดค่าที่ไม่ควรลองใหม่ก่อนลงทะเบียน OpenClaw ไม่มีเอาต์บ็อกซ์
สำหรับผลที่เกิดจากฮุกของ Plugin หากโปรเซสออกก่อนการยอมรับแบบคงทน
การเริ่ม Gateway ครั้งถัดไปจะส่งสแนปช็อต `cron_reconciled` ที่เป็นแหล่งข้อมูลจริงใหม่
`gateway_stop` จะยกเลิกงานโฮสต์ที่กำลังดำเนินอยู่ รอให้เวิร์กเกอร์ทำงานจนเสร็จ แล้วจึง
ปิดอะแดปเตอร์

## การเลิกใช้งานที่กำลังจะมาถึง

พื้นผิวบางส่วนที่อยู่ใกล้เคียงกับฮุกถูกเลิกใช้แล้วแต่ยังคงรองรับอยู่ โปรดย้ายข้อมูล
ก่อนรุ่นหลักถัดไป:

- **เอนเวโลปช่องทางแบบข้อความธรรมดา** ในตัวจัดการ `inbound_claim` และ `message_received`
  ให้อ่าน `BodyForAgent` และบล็อกบริบทผู้ใช้แบบมีโครงสร้าง
  แทนการแยกวิเคราะห์ข้อความเอนเวโลปแบบแบน ดู
  [เอนเวโลปช่องทางแบบข้อความธรรมดา → BodyForAgent](/th/plugins/sdk-migration#active-deprecations)
- **`before_agent_start`** ยังคงมีไว้เพื่อความเข้ากันได้ Plugin ใหม่ควรใช้
  `before_model_resolve` และ `before_prompt_build` แทน
  เฟสที่รวมกัน
- **`subagent_spawning`** ยังคงมีไว้เพื่อความเข้ากันได้กับ Plugin รุ่นเก่า แต่
  Plugin ใหม่ไม่ควรส่งคืนการกำหนดเส้นทางเธรดจากค่านี้ Core จะเตรียม
  การผูกซับเอเจนต์ `thread: true` ผ่านอะแดปเตอร์การผูกเซสชันของช่องทาง
  ก่อนที่ `subagent_spawned` จะทำงาน
- **`deactivate`** ยังคงเป็นนามแฝงความเข้ากันได้สำหรับการล้างข้อมูลที่เลิกใช้แล้ว จนถึง
  หลัง 2026-08-16 Plugin ใหม่ควรใช้ `gateway_stop`
- **`onResolution` ใน `before_tool_call`** ขณะนี้ใช้ยูเนียนแบบระบุชนิด
  `PluginApprovalResolution` (`allow-once` / `allow-always` / `deny` /
  `timeout` / `cancelled`) แทน `string` แบบรูปแบบอิสระ
- **`api.registerSessionExtension` / `api.enqueueNextTurnInjection`** ยังคง
  เป็นนามแฝงความเข้ากันได้ระดับบนสุด Plugin ใหม่ควรใช้
  `api.session.state.registerSessionExtension(...)` และ
  `api.session.workflow.enqueueNextTurnInjection(...)`

สำหรับรายการทั้งหมด ได้แก่ การลงทะเบียนความสามารถของหน่วยความจำ โปรไฟล์การคิดของผู้ให้บริการ
ผู้ให้บริการการยืนยันตัวตนภายนอก ชนิดการค้นหาผู้ให้บริการ ตัวเข้าถึงรันไทม์ของงาน
และการเปลี่ยนชื่อ `command-auth` → `command-status` โปรดดู
[การย้ายข้อมูล Plugin SDK → การเลิกใช้งานที่ยังมีผล](/th/plugins/sdk-migration#active-deprecations)
