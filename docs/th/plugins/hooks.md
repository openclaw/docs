จุดเชื่อมต่อของ Plugin เป็นจุดขยายภายในโปรเซสสำหรับ Plugin ของ OpenClaw ซึ่งใช้ตรวจสอบหรือ
เปลี่ยนแปลงการรันของเอเจนต์ การเรียกใช้เครื่องมือ โฟลว์ข้อความ วงจรชีวิตของเซสชัน การกำหนดเส้นทางซับเอเจนต์
การติดตั้ง หรือการเริ่มต้น Gateway

ให้ใช้ [จุดเชื่อมต่อภายใน](/th/automation/hooks) แทน สำหรับสคริปต์ขนาดเล็กที่ผู้ปฏิบัติงานติดตั้ง
`HOOK.md` ซึ่งตอบสนองต่อเหตุการณ์คำสั่งและ Gateway เช่น `/new`,
`/reset`, `/stop`, `agent:bootstrap` หรือ `gateway:startup`

## เริ่มต้นอย่างรวดเร็ว

ลงทะเบียนจุดเชื่อมต่อแบบมีชนิดด้วย `api.on(...)` จากจุดเข้าใช้งานของ Plugin:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

export default definePluginEntry({
  id: "tool-preflight",
  name: "Tool Preflight",
  register(api) {
    api.on(
      "before_tool_call",
      async (event) => {
        if (event.toolName !== "web_search") {
          return;
        }

        return {
          requireApproval: {
            title: "เรียกใช้การค้นหาเว็บ",
            description: `อนุญาตคำค้นหา: ${String(event.params.query ?? "")}`,
            severity: "info",
            timeoutMs: 60_000,
          },
        };
      },
      { priority: 50 },
    );
  },
});
```

ตัวจัดการที่สามารถส่งคืนการตัดสินใจหรือการแก้ไขจะทำงานตามลำดับ
`priority` จากมากไปน้อย โดยตัวจัดการที่มีลำดับความสำคัญเท่ากันจะคงลำดับการลงทะเบียนไว้
ตัวจัดการที่มีไว้สำหรับสังเกตการณ์เท่านั้นจะทำงานแบบขนาน และการส่งงานสังเกตการณ์
แบบไม่รอผลสามารถทำงานซ้อนกับเหตุการณ์ถัดไปได้ อย่าใช้ลำดับความสำคัญเพื่อจัดลำดับ
ผลข้างเคียงจากการสังเกตการณ์

`api.on(name, handler, opts?)` ยอมรับ:

| ตัวเลือก      | ผล                                                                                                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `priority`  | การจัดลำดับ โดยค่าที่สูงกว่าจะทำงานก่อน                                                                                                                                                                      |
| `timeoutMs` | งบเวลารอสำหรับแต่ละจุดเชื่อมต่อ เมื่อหมดเวลา OpenClaw จะหยุดรอตัวจัดการนั้นและดำเนินการต่อ โดยจะไม่ยกเลิกตัวจัดการหรือผลข้างเคียงของตัวจัดการ ละเว้นเพื่อใช้ระยะหมดเวลาเริ่มต้นต่อจุดเชื่อมต่อของตัวรัน |

ผู้ปฏิบัติงานสามารถตั้งงบเวลาของจุดเชื่อมต่อได้โดยไม่ต้องแก้ไขโค้ด Plugin:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "hooks": {
          "timeoutMs": 30000,
          "timeouts": {
            "before_prompt_build": 90000,
            "agent_end": 60000
          }
        }
      }
    }
  }
}
```

`hooks.timeouts.<hookName>` จะแทนที่ `hooks.timeoutMs` ซึ่งจะแทนที่ค่า
`api.on(..., { timeoutMs })` ที่ผู้เขียน Plugin กำหนด แต่ละค่าต้องเป็น
จำนวนเต็มบวกไม่เกิน 600000 ms ควรใช้การกำหนดค่าเฉพาะจุดเชื่อมต่อสำหรับ
จุดเชื่อมต่อที่ทราบว่าทำงานช้า เพื่อไม่ให้ Plugin หนึ่งได้รับงบเวลาที่ยาวขึ้นในทุกตำแหน่ง

พรอมิสของตัวจัดการที่หมดเวลาจะยังคงทำงานต่อ เนื่องจากคอลแบ็กของจุดเชื่อมต่อไม่ได้รับ
สัญญาณยกเลิก การส่งงานของจุดเชื่อมต่อสามารถคืนสิทธิ์การรับงานของ Gateway
ขณะที่งานของ Plugin นั้นยังดำเนินอยู่ Plugin ที่เป็นเจ้าของ
งานซึ่งใช้เวลานานต้องจัดเตรียมวงจรชีวิตการยกเลิกและปิดระบบของตนเอง

จุดเชื่อมต่อแก้ไขขาออก `message_sending` และ `reply_payload_sending` ใช้ค่าเริ่มต้น
15 วินาทีต่อตัวจัดการ หากตัวใดหมดเวลา OpenClaw จะบันทึกข้อผิดพลาดของ Plugin
และดำเนินการต่อด้วยเพย์โหลดล่าสุด เพื่อให้เลนการนำส่งแบบเรียงลำดับ
สิ้นสุดลงได้ ตั้งงบเวลาต่อจุดเชื่อมต่อให้มากขึ้นสำหรับ Plugin ที่ตั้งใจทำงานช้ากว่า
ก่อนการนำส่ง

Plugin ช่องทางที่ใช้ `createReplyDispatcher` สามารถประกาศ
งบเวลาบวกต่อขั้นตอนที่มากขึ้นด้วย `beforeDeliverOptions: { timeoutMs }` หรือเมื่อ
เพิ่มงานต่อท้ายด้วย `dispatcher.appendBeforeDeliver(handler, { timeoutMs })` ได้เช่นกัน
หากไม่มีงบเวลาที่เจ้าของประกาศ คอลแบ็กเหล่านั้นจะใช้ค่าเริ่มต้น
15 วินาทีเดียวกัน เพื่อไม่ให้คอลแบ็กที่ค้างยึดเลนการนำส่งแบบเรียงลำดับไว้

แต่ละจุดเชื่อมต่อจะได้รับ `event.context.pluginConfig` ซึ่งเป็นการกำหนดค่าที่ผ่านการแก้ไขแล้วสำหรับ
Plugin ที่ลงทะเบียนตัวจัดการนั้น OpenClaw จะแทรกค่านี้แยกตามตัวจัดการโดยไม่
เปลี่ยนแปลงออบเจ็กต์เหตุการณ์ที่ใช้ร่วมกันซึ่ง Plugin อื่นมองเห็น

## แค็ตตาล็อกจุดเชื่อมต่อ

จุดเชื่อมต่อจัดกลุ่มตามพื้นผิวที่ขยาย ชื่อที่เป็น **ตัวหนา** ยอมรับผลลัพธ์
การตัดสินใจ (บล็อก ยกเลิก แทนที่ หรือกำหนดให้ต้องอนุมัติ) ส่วนที่เหลือ
ใช้สำหรับสังเกตการณ์เท่านั้น

**รอบการทำงานของเอเจนต์**

| จุดเชื่อมต่อ                            | วัตถุประสงค์                                                                                  |
| ------------------------------- | ---------------------------------------------------------------------------------------- |
| `before_model_resolve`          | แทนที่ผู้ให้บริการหรือโมเดลก่อนโหลดข้อความของเซสชัน                                  |
| `agent_turn_prepare`            | ใช้ข้อมูลแทรกรอบการทำงานของ Plugin ที่อยู่ในคิว และเพิ่มบริบทของรอบเดียวกันก่อนจุดเชื่อมต่อพรอมต์      |
| `before_prompt_build`           | เพิ่มบริบทแบบไดนามิกหรือข้อความพรอมต์ระบบก่อนเรียกใช้โมเดล                          |
| `before_agent_start`            | เฟสรวมสำหรับความเข้ากันได้เท่านั้น ควรใช้จุดเชื่อมต่อสองรายการด้านบน                            |
| **`before_agent_run`**          | ตรวจสอบพรอมต์สุดท้ายและข้อความเซสชันก่อนส่งให้โมเดล และสามารถบล็อกการรันได้ |
| **`before_agent_reply`**        | ลัดวงจรรอบการทำงานของโมเดลด้วยการตอบกลับสังเคราะห์หรือไม่ตอบกลับ                           |
| **`before_agent_finalize`**     | ตรวจสอบคำตอบสุดท้ายตามธรรมชาติและร้องขอให้โมเดลประมวลผลอีกหนึ่งรอบ                         |
| `agent_end`                     | สังเกตข้อความสุดท้าย สถานะความสำเร็จ และระยะเวลาการรัน                                  |
| `heartbeat_prompt_contribution` | เพิ่มบริบทสำหรับ Heartbeat เท่านั้นให้กับ Plugin ตรวจสอบเบื้องหลังและวงจรชีวิต                  |

**การสังเกตการณ์การสนทนา**

| จุดเชื่อมต่อ                                      | วัตถุประสงค์                                                                                                            |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `model_call_started` / `model_call_ended` | ข้อมูลเมตาการเรียกผู้ให้บริการ/โมเดลที่ผ่านการล้างข้อมูลแล้ว ได้แก่ เวลา ผลลัพธ์ และแฮชรหัสคำขอแบบจำกัด ไม่มีเนื้อหาพรอมต์หรือการตอบกลับ |
| `llm_input`                               | อินพุตของผู้ให้บริการ ได้แก่ พรอมต์ระบบ พรอมต์ และประวัติ                                                                     |
| `llm_output`                              | เอาต์พุตของผู้ให้บริการ การใช้งาน และ `contextTokenBudget` ที่ผ่านการแก้ไขแล้วเมื่อมี                                       |

**เครื่องมือ**

| จุดเชื่อมต่อ                       | วัตถุประสงค์                                                   |
| -------------------------- | --------------------------------------------------------- |
| **`before_tool_call`**     | เขียนพารามิเตอร์เครื่องมือใหม่ บล็อกการทำงาน หรือกำหนดให้ต้องอนุมัติ |
| `after_tool_call`          | สังเกตผลลัพธ์ของเครื่องมือ ข้อผิดพลาด และระยะเวลา                |
| `resolve_exec_env`         | เพิ่มตัวแปรสภาพแวดล้อมที่ Plugin เป็นเจ้าของให้กับ `exec`   |
| **`tool_result_persist`**  | เขียนข้อความผู้ช่วยที่สร้างจากผลลัพธ์ของเครื่องมือใหม่ |
| **`before_message_write`** | ตรวจสอบหรือบล็อกการเขียนข้อความที่กำลังดำเนินอยู่ (พบได้น้อย)      |

**ข้อความและการนำส่ง**

| จุดเชื่อมต่อ                            | วัตถุประสงค์                                                           |
| ------------------------------- | ----------------------------------------------------------------- |
| **`inbound_claim`**             | รับสิทธิ์จัดการข้อความขาเข้าก่อนกำหนดเส้นทางไปยังเอเจนต์ (การตอบกลับสังเคราะห์) |
| **`channel_pairing_requested`** | สังเกตคำขอจับคู่ DM ที่สร้างขึ้นใหม่                         |
| `message_received`              | สังเกตเนื้อหาขาเข้า ผู้ส่ง เธรด และข้อมูลเมตา             |
| **`message_sending`**           | เขียนเนื้อหาขาออกใหม่หรือยกเลิกการนำส่ง                       |
| **`reply_payload_sending`**     | เปลี่ยนแปลงหรือยกเลิกเพย์โหลดการตอบกลับที่ปรับให้เป็นมาตรฐานก่อนการนำส่ง        |
| `message_sent`                  | สังเกตความสำเร็จหรือความล้มเหลวของการนำส่งขาออก                      |
| **`before_dispatch`**           | ตรวจสอบหรือเขียนการส่งงานขาออกใหม่ก่อนส่งต่อให้ช่องทาง    |
| **`reply_dispatch`**            | มีส่วนร่วมในไปป์ไลน์การส่งการตอบกลับขั้นสุดท้าย                  |

**เซสชันและ Compaction**

| จุดเชื่อมต่อ                                     | วัตถุประสงค์                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `session_start` / `session_end`          | ติดตามขอบเขตวงจรชีวิตของเซสชัน `reason` เป็นหนึ่งใน `new`, `reset`, `idle`, `daily`, `compaction`, `deleted`, `shutdown`, `restart` หรือ `unknown` จุดเชื่อมต่อ `shutdown`/`restart` จะทำงานจากตัวจบการปิดระบบของ Gateway เมื่อโปรเซสหยุดหรือเริ่มใหม่โดยยังมีเซสชันทำงานอยู่ เพื่อให้ Plugin (หน่วยความจำ ที่เก็บทรานสคริปต์) ปิดแถวค้างให้สมบูรณ์แทนที่จะปล่อยให้เปิดค้างข้ามการเริ่มใหม่ ตัวจบการทำงานมีขอบเขตเวลา เพื่อไม่ให้ Plugin ที่ช้าบล็อก SIGTERM/SIGINT |
| `before_compaction` / `after_compaction` | สังเกตหรือใส่คำอธิบายประกอบรอบ Compaction                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `before_reset`                           | สังเกตเหตุการณ์รีเซ็ตเซสชัน (`/reset`, การรีเซ็ตผ่านโปรแกรม)                                                                                                                                                                                                                                                                                                                                                                                                     |

สำหรับการเรียก `sessions.create` ที่มี `parentSessionKey` และ `emitCommandHooks: true` เซสชันลูกที่แยกต่างหากจะได้รับ `session_start` เสมอ ผู้เรียกประกาศว่าเซสชันแม่จะได้รับ `session_end` ปลายทางด้วยหรือไม่ผ่าน `succeedsParent`: `true` หมายถึงตัวสืบทอด ส่วน `false` หมายถึงเซสชันลูกแบบขนาน การละเว้นจะคงพฤติกรรมการเปลี่ยนผ่านเซสชันแม่แบบเดิมไว้ จุดเชื่อมต่อ `command:new` และ `before_reset` ยังคงอธิบายการดำเนินการ `/new` ที่ร้องขอในทั้งสองกรณี

**ซับเอเจนต์**

- `subagent_spawned` / `subagent_ended` - สังเกตการเริ่มทำงานและการเสร็จสิ้นของเอเจนต์ย่อย
- `subagent_delivery_target` - ฮุกความเข้ากันได้สำหรับส่งมอบการเสร็จสิ้นเมื่อไม่มีการเชื่อมโยงเซสชันแกนหลักที่สามารถฉายเส้นทางได้
- `subagent_spawning` - ฮุกความเข้ากันได้ที่เลิกใช้แล้ว ขณะนี้แกนหลักเตรียมการเชื่อมโยงเอเจนต์ย่อย `thread: true` ผ่านอะแดปเตอร์การเชื่อมโยงเซสชันของช่องก่อนที่ `subagent_spawned` จะทำงาน
- `subagent_spawned` มี `resolvedModel` และ `resolvedProvider` เมื่อ OpenClaw ระบุโมเดลเนทีฟของเซสชันลูกได้แล้วก่อนเริ่มทำงาน
- `subagent_ended` ส่งต่อ `targetSessionKey` (ข้อมูลระบุตัวตน - ตรงกับ `subagent_spawned.childSessionKey`), `targetKind` (`"subagent"` หรือ `"acp"`), `reason`, `outcome` ซึ่งไม่บังคับ (`"ok"`, `"error"`, `"timeout"`, `"killed"`, `"reset"` หรือ `"deleted"`), `error` ซึ่งไม่บังคับ, `runId`, `endedAt`, `accountId` และ `sendFarewell` แต่ **ไม่มี** `agentId` หรือ `childSessionKey`; ให้ใช้ `targetSessionKey` เพื่อเชื่อมโยงกับเหตุการณ์ `subagent_spawned` ที่ตรงกัน

| `gateway_start` / `gateway_stop` | เริ่มหรือหยุดบริการที่ Plugin เป็นเจ้าของพร้อมกับ Gateway                                                 |
| `deactivate`                     | นามแฝงความเข้ากันได้ที่เลิกใช้แล้วสำหรับ `gateway_stop`; ใช้ `gateway_stop` ใน Plugin ใหม่                 |
| `cron_reconciled`                | ปรับสถานะให้สอดคล้องกับสถานะ Cron ทั้งหมดของ Gateway หลังเริ่มต้นหรือโหลดซ้ำ                            |
| `cron_changed`                   | สังเกตการเปลี่ยนแปลงวงจรชีวิต Cron ที่ Gateway เป็นเจ้าของ (เพิ่ม อัปเดต ลบ เริ่ม เสร็จสิ้น กำหนดเวลา) |
| **`before_install`**             | ตรวจสอบวัสดุการติดตั้ง Skills หรือ Plugin ที่จัดเตรียมไว้จากรันไทม์ Plugin ที่โหลดแล้ว                         |

### คำขอจับคู่ช่อง

ใช้ `channel_pairing_requested` เมื่อ Plugin ต้องแจ้งผู้ปฏิบัติงานหรือ
เขียนบันทึกการตรวจสอบหลังจากผู้ส่ง DM ที่ยังไม่ได้จับคู่สร้างคำขอจับคู่
ที่รอดำเนินการ ฮุกจะถูกส่งเมื่อสร้างคำขอแล้ว การส่งคำตอบการจับคู่
ผ่านช่องจะไม่ล่าช้าเพราะตัวจัดการฮุกที่ช้าหรือล้มเหลว

```typescript
api.on("channel_pairing_requested", async (event) => {
  await notifyOperator({
    text: `คำขอจับคู่ ${event.channel} ใหม่จาก ${event.senderId}: ${event.code}`,
  });
});
```

ฮุกนี้ใช้สำหรับการสังเกตเท่านั้น ไม่ได้อนุมัติ ปฏิเสธ ระงับ หรือเขียน
คำตอบการจับคู่ใหม่ เพย์โหลดประกอบด้วยช่อง, `accountId` ซึ่งไม่บังคับ,
`senderId` ที่มีขอบเขตตามช่อง, `code` สำหรับการจับคู่ และข้อมูลเมตาของช่อง ให้ถือว่า
รหัสจับคู่เป็นข้อมูลรับรองการอนุมัติแบบใช้ครั้งเดียวที่ยังใช้งานอยู่ และส่งรหัสนี้ไปยัง
ปลายทางของผู้ปฏิบัติงานที่เชื่อถือได้เท่านั้น ให้ถือว่า `metadata` เป็นข้อความข้อมูลระบุตัวตน
ที่ผู้ส่งซึ่งไม่น่าเชื่อถือเป็นผู้ระบุ ฮุกนี้ไม่มีเนื้อหาหรือสื่อของข้อความขาเข้า

## ฮุกรันไทม์สำหรับการดีบัก

ใช้ `before_model_resolve` เพื่อสลับผู้ให้บริการหรือโมเดลสำหรับรอบการทำงานของเอเจนต์ โดยฮุกนี้
ทำงานก่อนการระบุโมเดล ส่วน `llm_output` จะทำงานหลังจากความพยายามใช้โมเดล
สร้างเอาต์พุตของผู้ช่วยแล้วเท่านั้น

หากต้องการหลักฐานเกี่ยวกับโมเดลที่มีผลจริงในเซสชัน ให้ตรวจสอบการลงทะเบียนรันไทม์ แล้ว
ใช้ `openclaw sessions` หรือพื้นผิวเซสชัน/สถานะของ Gateway หากต้องการดีบัก
เพย์โหลดของผู้ให้บริการ ให้เริ่ม Gateway ด้วย `--raw-stream` และ
`--raw-stream-path <path>` เพื่อเขียนเหตุการณ์สตรีมดิบของโมเดลลงในไฟล์ jsonl

## นโยบายการเรียกใช้เครื่องมือ

`before_tool_call` รับข้อมูลต่อไปนี้:

- `event.toolName`
- `event.params`
- `event.toolKind` และ `event.toolInputKind` ซึ่งไม่บังคับ เป็นตัวจำแนกที่โฮสต์เป็นผู้กำหนด
  สำหรับเครื่องมือที่ตั้งใจใช้ชื่อร่วมกัน ตัวอย่างเช่น การเรียก `exec`
  ในโหมดโค้ดชั้นนอกใช้ `toolKind: "code_mode_exec"` และมี
  `toolInputKind: "javascript" | "typescript"` เมื่อทราบภาษาอินพุต
- `event.derivedPaths` ซึ่งไม่บังคับ เป็นคำใบ้พาธเป้าหมายที่โฮสต์อนุมานอย่างเต็มความสามารถ
  สำหรับเอนเวโลปเครื่องมือที่รู้จักกันดี เช่น `apply_patch`; พาธเหล่านี้อาจ
  ไม่ครบถ้วนหรือครอบคลุมเกินกว่าสิ่งที่เครื่องมือจะดำเนินการจริง (เช่น
  เมื่ออินพุตมีรูปแบบไม่ถูกต้องหรือไม่สมบูรณ์)
- `event.runId` ซึ่งไม่บังคับ
- `event.toolCallId` ซึ่งไม่บังคับ
- ฟิลด์บริบท เช่น `ctx.agentId`, `ctx.sessionKey`, `ctx.sessionId`,
  `ctx.runId`, `ctx.toolKind`, `ctx.toolInputKind` และ `ctx.trace` สำหรับการวินิจฉัย

สามารถส่งคืน:

```typescript
type BeforeToolCallResult = {
  params?: Record<string, unknown>;
  block?: boolean;
  blockReason?: string;
  requireApproval?: {
    title: string;
    description: string;
    severity?: "info" | "warning" | "critical";
    timeoutMs?: number;
    /** @deprecated การอนุมัติที่ยังไม่ได้ข้อยุติจะถูกปฏิเสธเสมอ */
    timeoutBehavior?: "allow" | "deny";
    allowedDecisions?: Array<"allow-once" | "allow-always" | "deny">;
    pluginId?: string;
    onResolution?: (
      decision: "allow-once" | "allow-always" | "deny" | "timeout" | "cancelled",
    ) => Promise<void> | void;
  };
};
```

พฤติกรรมการป้องกันสำหรับฮุกวงจรชีวิตที่ระบุชนิด:

- `block: true` เป็นผลลัพธ์สุดท้ายและจะข้ามตัวจัดการที่มีลำดับความสำคัญต่ำกว่า
- `block: false` จะถือว่าไม่มีการตัดสินใจ
- `params` เขียนพารามิเตอร์เครื่องมือใหม่สำหรับการดำเนินการ
- `requireApproval` หยุดการทำงานของเอเจนต์ชั่วคราวและถามผู้ใช้ผ่านการอนุมัติของ Plugin
  `/approve` สามารถอนุมัติได้ทั้งการดำเนินคำสั่งและการอนุมัติของ Plugin ในรีเลย์ `PreToolUse`
  แบบเนทีฟของ app-server ในโหมดรายงานของ Codex การดำเนินการนี้จะส่งต่อให้คำขออนุมัติ
  ของ app-server ที่ตรงกัน โปรดดู
  [รันไทม์ฮาร์เนส Codex](/th/plugins/codex-harness-runtime#hook-boundaries)
- `block: true` ที่มีลำดับความสำคัญต่ำกว่ายังคงบล็อกได้ หลังจากฮุกที่มีลำดับความสำคัญสูงกว่าร้องขอการอนุมัติแล้ว
- `onResolution` รับผลการตัดสินใจที่ได้ข้อยุติแล้ว ได้แก่ `allow-once`, `allow-always`,
  `deny`, `timeout` หรือ `cancelled`

โปรดดู [คำขอสิทธิ์ของ Plugin](/th/plugins/plugin-permission-requests) สำหรับ
การกำหนดเส้นทางการอนุมัติ พฤติกรรมการตัดสินใจ และกรณีที่ควรใช้ `requireApproval` แทน
เครื่องมือที่ไม่บังคับหรือการอนุมัติการดำเนินคำสั่ง

Plugin ที่ต้องใช้นโยบายระดับโฮสต์สามารถลงทะเบียนนโยบายเครื่องมือที่เชื่อถือได้ด้วย
`api.registerTrustedToolPolicy(...)` นโยบายเหล่านี้ทำงานก่อนฮุก
`before_tool_call` ทั่วไปและก่อนการตัดสินใจตามฮุกตามปกติ นโยบายที่เชื่อถือได้ของ Plugin ที่รวมมาพร้อมระบบ
จะทำงานก่อน จากนั้นนโยบายที่เชื่อถือได้ของ Plugin ที่ติดตั้งจะทำงานตามลำดับ
การโหลด Plugin และฮุก `before_tool_call` ทั่วไปจะทำงานภายหลัง Plugin ที่รวมมาพร้อมระบบยังคงใช้
เส้นทางนโยบายที่เชื่อถือได้เดิม Plugin ที่ติดตั้งต้องได้รับการเปิดใช้อย่างชัดเจน
และประกาศรหัสนโยบายทุกตัวใน `contracts.trustedToolPolicies`; รหัสที่ไม่ได้ประกาศ
จะถูกปฏิเสธก่อนลงทะเบียน รหัสนโยบายมีขอบเขตเฉพาะ Plugin ที่ลงทะเบียน
ดังนั้น Plugin ต่างกันจึงสามารถใช้รหัสภายในเดียวกันซ้ำได้ ใช้ระดับนี้เฉพาะ
สำหรับเกตที่โฮสต์เชื่อถือ เช่น นโยบายเวิร์กสเปซ การบังคับใช้งบประมาณ หรือ
ความปลอดภัยของเวิร์กโฟลว์ที่สงวนไว้

### ฮุกสภาพแวดล้อมการดำเนินคำสั่ง

`resolve_exec_env` ช่วยให้ Plugin เพิ่มตัวแปรสภาพแวดล้อมให้กับการเรียกใช้เครื่องมือ
`exec` ก่อนเรียกใช้คำสั่ง โดยรับข้อมูลต่อไปนี้:

- `event.sessionKey`
- `event.toolName` ซึ่งปัจจุบันเป็น `"exec"` เสมอ
- `event.host` ซึ่งเป็นหนึ่งใน `"gateway"`, `"sandbox"` หรือ `"node"`
- ฟิลด์บริบท เช่น `ctx.agentId`, `ctx.sessionKey`,
  `ctx.messageProvider` และ `ctx.channelId`

ส่งคืน `Record<string, string>` เพื่อผสานเข้ากับสภาพแวดล้อมการดำเนินคำสั่ง ตัวจัดการ
ทำงานตามลำดับความสำคัญ ผลลัพธ์ภายหลังจะเขียนทับผลลัพธ์ก่อนหน้าสำหรับคีย์เดียวกัน

เอาต์พุตของฮุกจะถูกกรองผ่านนโยบายคีย์สภาพแวดล้อมการดำเนินคำสั่งของโฮสต์ก่อน
ผสาน `PATH` จะถูกตัดออกเสมอ (การระบุคำสั่งและการตรวจสอบไบนารีที่ปลอดภัย
ขึ้นอยู่กับค่านี้) คีย์ที่ไม่ถูกต้องและคีย์อันตรายที่เขียนทับค่าของโฮสต์ เช่น `LD_*`,
`DYLD_*`, `NODE_OPTIONS`, ตัวแปรพร็อกซี (`HTTP_PROXY`, `HTTPS_PROXY`,
`ALL_PROXY`, `NO_PROXY`) และตัวแปรที่เขียนทับ TLS (`NODE_TLS_REJECT_UNAUTHORIZED`,
`SSL_CERT_FILE` และตัวแปรที่คล้ายกัน) จะถูกตัดออก สภาพแวดล้อมของ Plugin ที่กรองแล้วจะรวมอยู่
ในข้อมูลเมตาการอนุมัติ/การตรวจสอบของ Gateway และส่งต่อไปยังคำขอดำเนินการ
ของโฮสต์ Node

### การคงอยู่ของผลลัพธ์เครื่องมือ

ผลลัพธ์เครื่องมือสามารถมี `details` แบบมีโครงสร้างสำหรับการแสดงผลใน UI การวินิจฉัย
การกำหนดเส้นทางสื่อ หรือข้อมูลเมตาที่ Plugin เป็นเจ้าของ ให้ถือว่า `details` เป็นข้อมูลเมตารันไทม์
ไม่ใช่เนื้อหาพรอมป์:

- OpenClaw ตัด `toolResult.details` ออกก่อนการเล่นซ้ำของผู้ให้บริการและอินพุต
  Compaction เพื่อไม่ให้ข้อมูลเมตากลายเป็นบริบทของโมเดล
- รายการเซสชันที่คงไว้จะเก็บเฉพาะ `details` ที่มีขนาดจำกัด รายละเอียดที่มีขนาดเกิน
  จะถูกแทนที่ด้วยสรุปแบบย่อและ `persistedDetailsTruncated: true`
- `tool_result_persist` และ `before_message_write` ทำงานก่อนขีดจำกัดสุดท้าย
  ของการคงอยู่ ให้ `details` ที่ส่งคืนมีขนาดเล็ก และหลีกเลี่ยงการใส่
  ข้อความที่เกี่ยวข้องกับพรอมป์ไว้เฉพาะใน `details`; ให้ใส่เอาต์พุตเครื่องมือที่โมเดลมองเห็นได้ใน
  `content`

## ฮุกพรอมป์และโมเดล

ใช้ฮุกเฉพาะระยะสำหรับ Plugin ใหม่:

- `before_model_resolve`: รับเฉพาะพรอมป์ปัจจุบันและข้อมูลเมตา
  ของไฟล์แนบ ส่งคืน `providerOverride` หรือ `modelOverride`
- `agent_turn_prepare`: รับพรอมป์ปัจจุบัน ข้อความเซสชัน
  ที่เตรียมไว้ และการแทรกคิวแบบครั้งเดียวพอดีที่ระบายออกสำหรับเซสชันนี้
  ส่งคืน `prependContext` หรือ `appendContext`
- `before_prompt_build`: รับพรอมป์ปัจจุบันและข้อความเซสชัน
  ส่งคืน `prependContext`, `appendContext`, `systemPrompt`,
  `prependSystemContext` หรือ `appendSystemContext`
- `heartbeat_prompt_contribution`: ทำงานเฉพาะรอบ Heartbeat และส่งคืน
  `prependContext` หรือ `appendContext` มีไว้สำหรับตัวตรวจสอบเบื้องหลังที่
  ต้องสรุปสถานะปัจจุบันโดยไม่เปลี่ยนรอบที่ผู้ใช้เป็นผู้เริ่ม

`before_agent_start` ยังคงอยู่เพื่อความเข้ากันได้ ควรใช้ฮุกที่ระบุชัดเจน
ข้างต้น เพื่อไม่ให้ Plugin ขึ้นอยู่กับระยะแบบรวมดั้งเดิม

`before_agent_run` ทำงานหลังสร้างพรอมป์และก่อนอินพุตโมเดลใด ๆ
รวมถึงการโหลดรูปภาพเฉพาะพรอมป์และการสังเกต `llm_input` โดยรับ
อินพุตผู้ใช้ปัจจุบันเป็น `prompt` พร้อมกับประวัติเซสชันที่โหลดแล้วใน `messages`
และพรอมป์ระบบที่ใช้งานอยู่ ส่งคืน `{ outcome: "block", reason, message? }`
เพื่อหยุดการทำงานก่อนที่โมเดลจะอ่านพรอมป์ `reason` ใช้ภายใน
ส่วน `message` เป็นตัวแทนที่ผู้ใช้มองเห็นได้ รองรับเฉพาะผลลัพธ์ `pass` และ `block`
เท่านั้น รูปแบบการตัดสินใจที่ไม่รองรับจะถูกปฏิเสธโดยปริยาย

เมื่อการทำงานถูกบล็อก OpenClaw จะจัดเก็บเฉพาะข้อความแทนที่ใน
`message.content` พร้อมข้อมูลเมตาการบล็อกที่ไม่ละเอียดอ่อน เช่น รหัสของ Plugin
ที่บล็อกและการประทับเวลา ข้อความเดิมของผู้ใช้จะไม่ถูกเก็บไว้ในทรานสคริปต์
หรือบริบทในอนาคต เหตุผลการบล็อกภายในถือเป็นข้อมูลละเอียดอ่อนและ
ไม่รวมอยู่ในเพย์โหลดทรานสคริปต์ ประวัติ การเผยแพร่ บันทึก และการวินิจฉัย
การสังเกตการณ์ควรใช้ฟิลด์ที่ผ่านการล้างข้อมูล เช่น รหัสผู้บล็อก ผลลัพธ์
การประทับเวลา หรือหมวดหมู่ที่ปลอดภัย

`before_agent_start` และ `agent_end` มี `event.runId` เมื่อ OpenClaw สามารถ
ระบุการทำงานที่ใช้งานอยู่ได้ ค่าเดียวกันนี้ยังอยู่ใน `ctx.runId` การทำงานที่ขับเคลื่อนด้วย
Cron ยังเปิดเผย `ctx.jobId` (รหัสงาน Cron ต้นทาง) ในบริบทรอบการทำงาน
ของเอเจนต์ เพื่อให้ฮุกสามารถจำกัดขอบเขตเมตริก ผลข้างเคียง หรือสถานะไว้ที่งาน
ตามกำหนดการที่เฉพาะเจาะจง `ctx.jobId` ไม่ได้เป็นส่วนหนึ่งของบริบทเครื่องมือ `before_tool_call`

สำหรับการรันที่เริ่มต้นจากช่องทาง `ctx.channel` และ `ctx.messageProvider` ระบุ
พื้นผิวของผู้ให้บริการ เช่น `discord` หรือ `telegram` ส่วน `ctx.channelId` คือ
ตัวระบุเป้าหมายการสนทนาเมื่อ OpenClaw สามารถอนุมานได้จาก
คีย์เซสชันหรือเมทาดาทาการส่ง

เมื่อมีข้อมูลอัตลักษณ์ของผู้ส่ง บริบทฮุกของเอเจนต์จะมีรายการต่อไปนี้ด้วย:

- `ctx.senderId` - ID ผู้ส่งที่มีขอบเขตเฉพาะช่องทาง (เช่น Feishu `open_id`, ID ผู้ใช้
  Discord) กำหนดค่าเมื่อการรันเริ่มต้นจากข้อความของผู้ใช้ที่มี
  เมทาดาทาผู้ส่งที่ทราบ
- `ctx.chatId` - ตัวระบุการสนทนาแบบเนทีฟของการขนส่ง (เช่น Feishu
  `chat_id`, Telegram `chat_id`) กำหนดค่าเมื่อช่องทางต้นทาง
  มี ID การสนทนาแบบเนทีฟ
- `ctx.channelContext.sender.id` - ID ผู้ส่งเดียวกับ `ctx.senderId` ภายใต้
  ออบเจ็กต์ที่ช่องทางเป็นเจ้าของ ซึ่ง Plugin สามารถขยายด้วยฟิลด์เฉพาะช่องทาง
- `ctx.channelContext.chat.id` - ID การสนทนาเดียวกับ `ctx.chatId`
  ภายใต้ออบเจ็กต์ที่ช่องทางเป็นเจ้าของ ซึ่ง Plugin สามารถขยายด้วยฟิลด์
  เฉพาะช่องทาง

คอร์กำหนดเฉพาะฟิลด์ `id` ที่ซ้อนอยู่เท่านั้น Plugin ของช่องทางที่ส่ง
เมทาดาทาผู้ส่งหรือแชตที่สมบูรณ์ยิ่งขึ้นผ่านตัวช่วยขาเข้า สามารถเสริม
`PluginHookChannelSenderContext` หรือ `PluginHookChannelChatContext` จาก
`openclaw/plugin-sdk/channel-inbound` ได้:

```ts
declare module "openclaw/plugin-sdk/channel-inbound" {
  interface PluginHookChannelSenderContext {
    unionId?: string;
    userId?: string;
  }
}
```

Plugin ของช่องทางส่งฟิลด์เหล่านั้นผ่านตัวช่วย SDK ขาเข้า:

```ts
buildChannelInboundEventContext({
  // ...
  channelContext: {
    sender: { id: senderOpenId, unionId, userId },
    chat: { id: chatId },
  },
});
```

ฟิลด์เหล่านี้เป็นตัวเลือกและจะไม่มีในการรันที่เริ่มต้นจากระบบ (heartbeat,
cron, exec-event)

`ctx.senderExternalId` ยังคงเป็นฟิลด์ความเข้ากันได้กับซอร์สที่เลิกใช้แล้วสำหรับ
Plugin รุ่นเก่า คอร์ไม่กำหนดค่าฟิลด์นี้ อัตลักษณ์ผู้ส่งเฉพาะช่องทางใหม่
ควรอยู่ภายใต้ `ctx.channelContext.sender` ผ่านการเสริม
โมดูล

`agent_end` เป็นฮุกสำหรับสังเกตการณ์ เส้นทาง Gateway และฮาร์เนสถาวรจะเรียกใช้
แบบไม่รอผลหลังจบรอบ ขณะที่เส้นทาง CLI แบบครั้งเดียวอายุสั้นจะรอ
พรอมิสของฮุกก่อนล้างกระบวนการ เพื่อให้ Plugin ที่เชื่อถือได้สามารถฟลัช
ข้อมูลสังเกตการณ์ของเทอร์มินัลหรือบันทึกสถานะได้ ตัวเรียกใช้ฮุกกำหนดเวลาหมด 30 วินาที
เพื่อไม่ให้ Plugin หรือปลายทางการฝังที่ค้างทำให้พรอมิสของฮุก
รอดำเนินการตลอดไป ระบบจะบันทึกเวลาหมดและ OpenClaw จะทำงานต่อ โดยจะไม่
ยกเลิกงานเครือข่ายที่ Plugin เป็นเจ้าของ เว้นแต่ Plugin จะใช้สัญญาณยกเลิก
ของตนเองด้วย

ใช้ `model_call_started` และ `model_call_ended` สำหรับเทเลเมทรีการเรียกผู้ให้บริการ
ที่ไม่ควรได้รับพรอมต์ดิบ ประวัติ การตอบกลับ ส่วนหัว เนื้อหาคำขอ
หรือ ID คำขอของผู้ให้บริการ ฮุกเหล่านี้มีเมทาดาทาที่คงที่ เช่น
`runId`, `callId`, `provider`, `model`, `api`/`transport` ที่เป็นตัวเลือก, ค่าเทอร์มินัล
`durationMs`/`outcome` และ `upstreamRequestIdHash` เมื่อ OpenClaw สามารถอนุมาน
แฮช ID คำขอของผู้ให้บริการที่มีขอบเขตจำกัดได้ เมื่อรันไทม์แก้ไข
เมทาดาทาหน้าต่างบริบทแล้ว เหตุการณ์และบริบทของฮุกจะมี
`contextTokenBudget` ซึ่งเป็นงบประมาณโทเค็นที่มีผลหลังใช้ขีดจำกัดของโมเดล/การกำหนดค่า/เอเจนต์
รวมถึง `contextWindowSource` และ `contextWindowReferenceTokens` เมื่อมีการใช้
ขีดจำกัดที่ต่ำกว่า

`before_agent_finalize` จะทำงานเฉพาะเมื่อฮาร์เนสกำลังจะยอมรับคำตอบสุดท้าย
ตามธรรมชาติของผู้ช่วย ไม่ใช่เส้นทางยกเลิก `/stop` และจะไม่
ทำงานเมื่อผู้ใช้ยกเลิกรอบ ส่งคืน `{ action: "revise", reason }` เพื่อขอ
ให้ฮาร์เนสเรียกโมเดลอีกหนึ่งรอบก่อนจบกระบวนการ ส่งคืน `{ action:
"finalize", reason? }` เพื่อบังคับให้จบกระบวนการ หรือไม่ต้องส่งคืนผลลัพธ์เพื่อดำเนินการต่อ
ตัวจัดการมีงบประมาณเริ่มต้น 15 วินาที เมื่อหมดเวลา OpenClaw จะบันทึกความล้มเหลวและ
ดำเนินการต่อด้วยคำตอบสุดท้ายเดิม
ฮุกเนทีฟ `Stop` ของ Codex จะถูกถ่ายทอดมายังฮุกนี้เป็นการตัดสินใจ
`before_agent_finalize` ของ OpenClaw

เมื่อส่งคืน `action: "revise"` Plugin สามารถใส่เมทาดาทา `retry` เพื่อ
กำหนดขอบเขตการเรียกโมเดลเพิ่มเติมและทำให้เล่นซ้ำได้อย่างปลอดภัย:

```typescript
type BeforeAgentFinalizeRetry = {
  instruction: string;
  idempotencyKey?: string;
  maxAttempts?: number;
};
```

`instruction` จะถูกต่อท้ายเหตุผลการแก้ไขที่ส่งไปยังฮาร์เนส
`idempotencyKey` ช่วยให้โฮสต์นับจำนวนการลองใหม่สำหรับคำขอ Plugin เดียวกัน
ในชุดการตัดสินใจจบกระบวนการที่เทียบเท่ากัน และ `maxAttempts` จำกัดจำนวนรอบ
เพิ่มเติมสูงสุดที่โฮสต์จะอนุญาต ก่อนดำเนินการต่อด้วยคำตอบสุดท้ายตามธรรมชาติ

Plugin ที่ไม่ได้รวมมาในชุดและต้องการฮุกการสนทนาดิบ (`before_model_resolve`,
`before_agent_reply`, `llm_input`, `llm_output`, `before_agent_finalize`,
`agent_end` หรือ `before_agent_run`) ต้องตั้งค่า:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "hooks": {
          "allowConversationAccess": true
        }
      }
    }
  }
}
```

สามารถปิดใช้งานฮุกที่แก้ไขพรอมต์และการแทรกถาวรสำหรับรอบถัดไปเป็นราย
Plugin ได้ด้วย `plugins.entries.<id>.hooks.allowPromptInjection=false`

### ส่วนขยายเซสชันและการแทรกสำหรับรอบถัดไป

Plugin เวิร์กโฟลว์สามารถเก็บสถานะเซสชันขนาดเล็กที่เข้ากันได้กับ JSON อย่างถาวรด้วย
`api.session.state.registerSessionExtension(...)` และอัปเดตผ่านเมธอด
`sessions.pluginPatch` ของ Gateway แถวเซสชันจะแสดงสถานะส่วนขยายที่ลงทะเบียนไว้
ผ่าน `pluginExtensions` ทำให้ Control UI และไคลเอนต์อื่น
แสดงสถานะที่ Plugin เป็นเจ้าของได้โดยไม่ต้องรู้รายละเอียดภายในของ Plugin
`api.registerSessionExtension(...)` ยังคงใช้งานได้แต่เลิกใช้แล้ว โดยแนะนำให้ใช้
เนมสเปซ `api.session.state` แทน

ใช้ `api.session.workflow.enqueueNextTurnInjection(...)` เมื่อ Plugin ต้องการ
บริบทถาวรที่จะส่งไปถึงรอบโมเดลถัดไปเพียงครั้งเดียว (ฟิลด์ระดับบนสุด
`api.enqueueNextTurnInjection(...)` เป็นนามแฝงที่เลิกใช้แล้วและมี
ลักษณะการทำงานเดียวกัน) OpenClaw จะนำการแทรกที่เข้าคิวออกมาใช้ก่อนฮุกพรอมต์ ทิ้ง
การแทรกที่หมดอายุ และขจัดรายการซ้ำตาม `idempotencyKey` สำหรับแต่ละ Plugin นี่คือ
จุดเชื่อมต่อที่เหมาะสมสำหรับการทำงานต่อหลังอนุมัติ สรุปนโยบาย ค่าความเปลี่ยนแปลงจากตัวตรวจสอบเบื้องหลัง
และการทำงานต่อของคำสั่ง ซึ่งควรปรากฏต่อโมเดลใน
รอบถัดไปแต่ไม่ควรกลายเป็นข้อความพรอมต์ระบบถาวร

ความหมายของการล้างข้อมูลเป็นส่วนหนึ่งของสัญญา การล้างส่วนขยายเซสชันและ
คอลแบ็กการล้างวงจรชีวิตรันไทม์จะได้รับ `reset`, `delete`, `disable` หรือ
`restart` โฮสต์จะลบสถานะส่วนขยายเซสชันถาวร
และการแทรกสำหรับรอบถัดไปที่รอดำเนินการของ Plugin เจ้าของ เมื่อรีเซ็ต/ลบ/ปิดใช้งาน ส่วนการเริ่มใหม่
จะคงสถานะเซสชันถาวรไว้ ขณะที่คอลแบ็กการล้างช่วยให้ Plugin ปล่อย
งานตัวจัดกำหนดการ บริบทการรัน และทรัพยากรนอกแบนด์อื่นของ
รันไทม์รุ่นเก่า

## ฮุกข้อความ

ใช้ฮุกข้อความสำหรับการกำหนดเส้นทางและนโยบายการส่งในระดับช่องทาง:

- `message_received`: สังเกตเนื้อหาขาเข้า ผู้ส่ง `threadId`,
  `messageId`, `senderId`, ความสัมพันธ์กับการรัน/เซสชันที่เป็นตัวเลือก และเมทาดาทา
- `message_sending`: เขียน `content` ใหม่หรือส่งคืน `{ cancel: true }`
- `reply_payload_sending`: เขียนออบเจ็กต์ `ReplyPayload` ที่ปรับเป็นมาตรฐานแล้วใหม่
  (รวมถึง `presentation`, `delivery`, การอ้างอิงสื่อ และข้อความ) หรือส่งคืน
  `{ cancel: true }`
- `message_sent`: สังเกตผลสำเร็จหรือความล้มเหลวสุดท้าย

สำหรับการตอบกลับ TTS ที่มีเฉพาะเสียง `content` อาจมีบทถอดเสียงพูด
ที่ซ่อนอยู่ แม้เพย์โหลดช่องทางจะไม่มีข้อความ/คำบรรยายที่มองเห็นได้
การเขียน `content` นั้นใหม่จะอัปเดตเฉพาะบทถอดเสียงที่ฮุกมองเห็น และจะไม่
แสดงผลเป็นคำบรรยายสื่อ

เหตุการณ์ `reply_payload_sending` อาจมี `usageState` ซึ่งเป็นสแนปช็อตสดแบบ
พยายามอย่างดีที่สุดของโมเดล/การใช้งาน/บริบทต่อรอบ การส่งถาวร การเล่นซ้ำที่กู้คืน และ
การตอบกลับที่ไม่มีความสัมพันธ์กับการรันที่แน่นอนจะไม่มีฟิลด์นี้

บริบทฮุกข้อความจะแสดงฟิลด์ความสัมพันธ์ที่คงที่เมื่อมี:
`ctx.sessionKey`, `ctx.runId`, `ctx.messageId`, `ctx.senderId`, `ctx.trace`,
`ctx.traceId`, `ctx.spanId`, `ctx.parentSpanId` และ `ctx.callDepth` บริบทขาเข้า
และ `before_dispatch` จะแสดงเมทาดาทาการตอบกลับด้วย เมื่อช่องทาง
มีข้อมูลข้อความที่ยกมาซึ่งกรองตามการมองเห็นแล้ว: `replyToId`, `replyToIdFull`,
`replyToBody`, `replyToSender` และ `replyToIsQuote` ควรใช้
ฟิลด์ชั้นหนึ่งเหล่านี้ก่อนอ่านเมทาดาทาแบบเดิม

ควรใช้ฟิลด์ `threadId` และ `replyToId` ที่มีชนิดข้อมูลก่อนใช้เมทาดาทา
เฉพาะช่องทาง

กฎการตัดสินใจ:

- `message_sending` ที่มี `cancel: true` เป็นสถานะสิ้นสุด
- `message_sending` ที่มี `cancel: false` จะถือว่าไม่มีการตัดสินใจ
- `content` ที่เขียนใหม่จะดำเนินการต่อไปยังฮุกที่มีลำดับความสำคัญต่ำกว่า เว้นแต่ฮุกภายหลัง
  จะยกเลิกการส่ง
- `reply_payload_sending` ทำงานหลังการปรับเพย์โหลดเป็นมาตรฐานและก่อนการส่งผ่านช่องทาง
  รวมถึงการตอบกลับที่กำหนดเส้นทางกลับไปยังช่องทางต้นทาง
  ตัวจัดการจะทำงานตามลำดับ และแต่ละตัวจัดการจะเห็นเพย์โหลดล่าสุดที่
  ตัวจัดการลำดับความสำคัญสูงกว่าสร้างขึ้น
- เพย์โหลด `reply_payload_sending` จะไม่แสดงเครื่องหมายความเชื่อถือของรันไทม์ เช่น
  `trustedLocalMedia`; Plugin สามารถแก้ไขรูปทรงเพย์โหลดได้ แต่ไม่สามารถให้สิทธิ์ความเชื่อถือ
  แก่สื่อภายในเครื่อง
- `message_sending` สามารถส่งคืน `cancelReason` และ `metadata` ที่มีขอบเขตจำกัดพร้อม
  การยกเลิก API วงจรชีวิตข้อความใหม่จะแสดงกรณีนี้เป็นผลลัพธ์
  การส่งที่ถูกระงับด้วยเหตุผล `cancelled_by_message_sending_hook` ส่วนการส่งโดยตรง
  แบบเดิมจะยังคงส่งคืนอาร์เรย์ผลลัพธ์ว่างเพื่อความเข้ากันได้
- `message_sent` ใช้สำหรับสังเกตการณ์เท่านั้น ความล้มเหลวของตัวจัดการจะถูกบันทึกและไม่
  เปลี่ยนผลลัพธ์การส่ง

## ฮุกการติดตั้ง

ใช้ `security.installPolicy` สำหรับการตัดสินใจอนุญาต/บล็อกที่ผู้ดำเนินการเป็นเจ้าของ
นโยบายดังกล่าวทำงานจากการกำหนดค่า OpenClaw ครอบคลุมเส้นทางติดตั้งและอัปเดตของ CLI และ
จะปฏิเสธโดยปริยายเมื่อเปิดใช้งานแต่ไม่พร้อมใช้งาน

`before_install` เป็นฮุกวงจรชีวิตรันไทม์ของ Plugin โดยจะทำงานหลัง
`security.installPolicy` เฉพาะในกระบวนการ OpenClaw ที่โหลดฮุกของ Plugin
ไว้แล้ว เช่น ขั้นตอนการติดตั้งที่มี Gateway รองรับ ฮุกนี้มีประโยชน์สำหรับ
การสังเกตการณ์ คำเตือน และการตรวจสอบความเข้ากันได้ที่ Plugin เป็นเจ้าของ แต่ไม่ใช่
ขอบเขตความปลอดภัยหลักขององค์กรหรือโฮสต์สำหรับการติดตั้ง ฟิลด์
`builtinScan` ยังคงอยู่ในเพย์โหลดเหตุการณ์เพื่อความเข้ากันได้ แต่
OpenClaw ไม่เรียกใช้การบล็อกโค้ดอันตรายในตัวระหว่างการติดตั้งอีกต่อไป ดังนั้น
จึงเป็นผลลัพธ์ `ok` ว่าง ส่งคืนข้อค้นพบเพิ่มเติมหรือ
`{ block: true, blockReason }` เพื่อหยุดการติดตั้งในกระบวนการนั้น

`block: true` เป็นสถานะสิ้นสุด `block: false` จะถือว่าไม่มีการตัดสินใจ ความล้มเหลวของตัวจัดการ
จะบล็อกการติดตั้งแบบปฏิเสธโดยปริยาย

## วงจรชีวิต Gateway

ใช้ `gateway_start` เพื่อเริ่มบริการ Plugin ทั่วไป และใช้ `gateway_stop` เพื่อ
ล้างทรัพยากรที่ทำงานระยะยาว ตัวจัดกำหนดการ cron อาจยังโหลดอยู่เมื่อ
`gateway_start` ทำงาน ดังนั้นอย่าใช้ฟิลด์นี้เป็นสัญญาณฐานสำหรับการฉาย
cron ภายนอก

อย่าพึ่งพาฮุกภายใน `gateway:startup` สำหรับบริการรันไทม์
ที่ Plugin เป็นเจ้าของ

`cron_reconciled` จะทำงานหลังจากตัวจัดกำหนดการ cron ของ Gateway และตัวเฝ้าดูเมื่อออก
ปรับสถานะถาวรของตนให้สอดคล้องกันแล้ว โดยจะทำงานทั้งในการ
เริ่มต้นครั้งแรกและการแทนที่ตัวจัดกำหนดการระหว่างโหลดการกำหนดค่าใหม่ เหตุการณ์จะรายงาน
`reason` (`startup` หรือ `reload`) และสถานะ `enabled` ที่มีผล แม้ปิดใช้งาน
cron ก็ยังส่งเหตุการณ์พร้อม `enabled: false` ทำให้การฉายภายนอกสามารถ
ล้างการปลุกที่ค้างได้ ใช้ `ctx.getCron?.()` สำหรับอินสแตนซ์ตัวจัดกำหนดการที่แน่นอนซึ่ง
ปรับสถานะสอดคล้องเสร็จแล้ว การโหลดใหม่ในภายหลังจะไม่เปลี่ยนเป้าหมายของคอลแบ็กนั้น
`ctx.abortSignal` เป็นเจ้าของสแนปช็อตตัวจัดกำหนดการเดียวกัน Gateway จะยกเลิกสแนปช็อตนี้
ทันทีที่ตัวจัดกำหนดการใหม่พร้อมทำงานหรือเริ่มปิดระบบ ส่งต่อสแนปช็อตนี้ผ่าน
ผลข้างเคียงถาวรทุกรายการ และอย่ายอมรับสแนปช็อตหลังจากถูกยกเลิก
นี่เป็นสัญญาณวงจรชีวิตตัวจัดกำหนดการ ไม่ใช่สัญญาณเปิดใช้งาน Plugin:
การโหลดใหม่แบบร้อนเฉพาะ Plugin จะไม่เล่นเหตุการณ์นี้ซ้ำ ผู้ใช้ใหม่ที่เปิดใช้งาน
จะได้รับฐานแรกเมื่อมีการแทนที่ตัวจัดกำหนดการครั้งถัดไปหรือเมื่อ Gateway เริ่มทำงาน

เช่นเดียวกับฮุกสังเกตการณ์อื่น คอลแบ็ก `gateway_start` และ `cron_reconciled`
สามารถทำงานทับซ้อนกันได้ หากตัวจัดการทั้งสองใช้การเริ่มต้น Plugin ร่วมกัน ให้ประสานงาน
ด้วยพรอมิสความพร้อมภายใน Plugin แทนการพึ่งพาลำดับคอลแบ็ก

`cron_changed` จะทำงานเมื่อเกิดเหตุการณ์วงจรชีวิต Cron ที่ Gateway เป็นเจ้าของ โดยมีเพย์โหลดเหตุการณ์แบบมีชนิดซึ่งครอบคลุมเหตุผล `added`, `updated`, `removed`, `started`, `finished`
และ `scheduled` เหตุการณ์นี้มีสแนปช็อต `PluginHookGatewayCronJob`
(รวมถึง `state.nextRunAtMs`, `state.lastRunStatus` และ
`state.lastError` หากมี) พร้อม `PluginHookGatewayCronDeliveryStatus`
ของ `not-requested` | `delivered` | `not-delivered` | `unknown` เหตุการณ์การลบ
เกิดขึ้นหลังการคอมมิต โดยจะทำงานต่อเมื่อการลบแบบถาวรสำเร็จแล้วเท่านั้น และยังคงมี
สแนปช็อตของงานที่ถูกลบเพื่อให้ตัวจัดกำหนดการภายนอกสามารถปรับสถานะให้สอดคล้องกันได้

เหตุการณ์ `scheduled` เกิดขึ้นหลังการคอมมิต โดยจะทำงานต่อเมื่อการเขียนแบบถาวร
สำเร็จและเปลี่ยนค่า `nextRunAtMs` ที่มีผลของงานที่มีอยู่เท่านั้น โดยไม่นับรวมเหตุการณ์วงจรชีวิต
`added`, `updated` หรือ `removed` ที่ระบุอย่างชัดเจนของงานนั้น ค่า
`event.nextRunAtMs` ระดับบนสุดคือเวลาปลุกครั้งถัดไปที่คอมมิตแล้ว หากไม่มีค่านี้ แสดงว่างาน
ไม่มีเวลาปลุกครั้งถัดไป ให้ถือว่าเหตุการณ์เหล่านี้เป็นคำใบ้สำหรับการปรับสถานะให้สอดคล้องกัน ไม่ใช่บันทึกเดลตา
ที่เรียงตามลำดับ ใช้เหตุการณ์เหล่านี้เป็นคำใบ้ที่รวมกันได้เพื่ออ่านตัวจัดกำหนดการซ้ำ โดยใช้ตัวจัดกำหนดการล่าสุดที่บันทึกไว้โดย
`cron_reconciled` ห้ามนำตัวจัดกำหนดการจากบริบท `cron_changed` มาใช้
ให้ OpenClaw เป็นแหล่งข้อมูลจริงสำหรับการตรวจสอบกำหนดเวลาและการดำเนินงาน

### การฉายภาพ Cron ภายนอกอย่างปลอดภัย

ฉายภาพสแนปช็อตเวลาปลุกแบบสมบูรณ์แทนการส่งต่อเดลตาของเหตุการณ์ Cron การดำเนินการ
`replaceAll` ของอะแดปเตอร์ภายนอกต้องเป็นแบบอะตอมมิกและทำซ้ำได้โดยไม่เกิดผลข้างเคียงเพิ่มเติม และต้อง
เสร็จสิ้นต่อเมื่อโฮสต์ยอมรับสแนปช็อตแบบถาวรแล้วเท่านั้น นอกจากนี้ยังต้อง
ปฏิบัติตามสัญญาณยกเลิกที่ให้มา หากสัญญาณถูกยกเลิกก่อนการยอมรับ
แบบถาวร อะแดปเตอร์ต้องไม่ยอมรับสแนปช็อตนั้น

รูปแบบนี้ทำให้มีเวิร์กเกอร์สำหรับสถานะล่าสุดทำงานอยู่เพียงหนึ่งตัว เฉพาะ `cron_reconciled`
เท่านั้นที่นำอินสแตนซ์ของตัวจัดกำหนดการมาใช้ ส่วน `cron_changed` เพียงขอให้เวิร์กเกอร์นั้นอ่าน
อินสแตนซ์ที่เป็นแหล่งข้อมูลจริงซ้ำ ดังนั้นคำใบ้ที่มาถึงล่าช้าจึงไม่สามารถนำตัวจัดกำหนดการเก่ากว่ากลับมาใช้ได้
รีวิชันที่ใหม่กว่าจะยกเลิกความพยายามของโฮสต์ที่กำลังทำงานอยู่ ก่อนที่ความพยายามนั้นจะยอมรับ
สแนปช็อตที่ล้าสมัยได้

```typescript
import { setTimeout as sleep } from "node:timers/promises";
import type { OpenClawPluginApi } from "openclaw/plugin-sdk/plugin-entry";

type ExternalWake = { jobId: string; runAtMs: number };

type ExternalWakeHost = {
  replaceAll(wakes: readonly ExternalWake[], options: { signal: AbortSignal }): Promise<void>;
  close(): Promise<void>;
};

type CronReader = {
  list(options: { includeDisabled: true }): Promise<
    Array<{
      id: string;
      enabled?: boolean;
      state?: { nextRunAtMs?: number };
    }>
  >;
};

export function registerCronProjection(api: OpenClawPluginApi, host: ExternalWakeHost) {
  const lifecycle = new AbortController();
  let cron: CronReader | undefined;
  let enabled = false;
  let hasBaseline = false;
  let reconciliationSignal: AbortSignal | undefined;
  let requestedRevision = 0;
  let appliedRevision = 0;
  let worker = Promise.resolve();
  let activeAttempt: AbortController | undefined;

  const projectLatest = async () => {
    let retryMs = 1_000;

    while (!lifecycle.signal.aborted && appliedRevision < requestedRevision) {
      const ownerSignal = reconciliationSignal;
      if (!ownerSignal || ownerSignal.aborted) {
        return;
      }
      const targetRevision = requestedRevision;
      const attempt = new AbortController();
      const signal = AbortSignal.any([lifecycle.signal, ownerSignal, attempt.signal]);
      activeAttempt = attempt;

      try {
        const jobs = enabled && cron ? await cron.list({ includeDisabled: true }) : [];
        if (signal.aborted || targetRevision !== requestedRevision) {
          continue;
        }
        const wakes = jobs
          .flatMap((job): ExternalWake[] => {
            const runAtMs = job.enabled === false ? undefined : job.state?.nextRunAtMs;
            return runAtMs === undefined ? [] : [{ jobId: job.id, runAtMs }];
          })
          .sort((a, b) => a.runAtMs - b.runAtMs || a.jobId.localeCompare(b.jobId));

        await host.replaceAll(wakes, { signal });
        if (signal.aborted || targetRevision !== requestedRevision) {
          continue;
        }
        appliedRevision = targetRevision;
        retryMs = 1_000;
      } catch {
        if (lifecycle.signal.aborted || ownerSignal.aborted) {
          return;
        }
        if (attempt.signal.aborted) {
          continue;
        }
        api.logger.warn(`external cron projection failed; retrying in ${retryMs}ms`);
        try {
          await sleep(retryMs, undefined, { signal });
        } catch {
          if (lifecycle.signal.aborted) {
            return;
          }
          if (attempt.signal.aborted) {
            continue;
          }
        }
        retryMs = Math.min(retryMs * 2, 30_000);
      } finally {
        if (activeAttempt === attempt) {
          activeAttempt = undefined;
        }
      }
    }
  };

  const requestProjection = () => {
    const targetRevision = ++requestedRevision;
    activeAttempt?.abort();
    worker = worker.then(async () => {
      if (!lifecycle.signal.aborted && appliedRevision < targetRevision) {
        await projectLatest();
      }
    });
    return worker;
  };

  api.on("cron_reconciled", (event, ctx) => {
    const reconciledCron = ctx.getCron?.();
    if (event.enabled && !reconciledCron) {
      api.logger.warn("cron reconciliation did not expose a scheduler");
      return;
    }
    cron = reconciledCron;
    enabled = event.enabled;
    hasBaseline = true;
    reconciliationSignal = ctx.abortSignal;
    return requestProjection();
  });

  api.on("cron_changed", () => {
    if (hasBaseline) {
      return requestProjection();
    }
  });

  api.on("gateway_stop", async () => {
    lifecycle.abort();
    await worker;
    await host.close();
  });
}
```

เมื่อ `cron_reconciled` รายงาน `enabled: false` เส้นทางเดียวกันจะเรียก
`replaceAll([])` และล้างเวลาปลุกภายนอกที่ล้าสมัย การลองใหม่/การหน่วงเวลาแบบทวีคูณในตัวอย่างนี้
มีผลเฉพาะภายในโปรเซส และถือว่าความล้มเหลวของอะแดปเตอร์รันไทม์เป็นปัญหาชั่วคราว โปรดตรวจสอบ
การกำหนดค่าที่ไม่ควรลองใหม่ก่อนลงทะเบียน OpenClaw ไม่มี
เอาต์บ็อกซ์สำหรับผลกระทบจากฮุกของ Plugin หากโปรเซสสิ้นสุดก่อนการยอมรับแบบถาวร
การเริ่มต้น Gateway ครั้งถัดไปจะส่งสแนปช็อต `cron_reconciled` ใหม่ที่เป็นแหล่งข้อมูลจริง
`gateway_stop` จะยกเลิกงานของโฮสต์ที่กำลังดำเนินอยู่ รอให้เวิร์กเกอร์หยุดนิ่ง แล้วจึง
ปิดอะแดปเตอร์

## การเลิกใช้งานที่กำลังจะมาถึง

พื้นผิวบางส่วนที่เกี่ยวข้องกับฮุกถูกเลิกใช้แล้วแต่ยังคงรองรับอยู่ โปรดย้าย
ก่อนรีลีสหลักครั้งถัดไป:

- **เอนเวโลปช่องทางแบบข้อความธรรมดา** ในตัวจัดการ `inbound_claim` และ `message_received`
  ให้อ่าน `BodyForAgent` และบล็อกบริบทผู้ใช้แบบมีโครงสร้าง
  แทนการแยกวิเคราะห์ข้อความเอนเวโลปแบบแบน ดู
  [เอนเวโลปช่องทางแบบข้อความธรรมดา → BodyForAgent](/th/plugins/sdk-migration#active-deprecations)
- **`before_agent_start`** ยังคงมีไว้เพื่อความเข้ากันได้ Plugin ใหม่ควรใช้
  `before_model_resolve` และ `before_prompt_build` แทนเฟส
  แบบรวม
- **`subagent_spawning`** ยังคงมีไว้เพื่อความเข้ากันได้กับ Plugin รุ่นเก่า แต่
  Plugin ใหม่ไม่ควรส่งคืนการกำหนดเส้นทางเธรดจากส่วนนี้ Core จะเตรียม
  การเชื่อมโยงเอเจนต์ย่อย `thread: true` ผ่านอะแดปเตอร์การเชื่อมโยงเซสชันของช่องทาง
  ก่อนที่ `subagent_spawned` จะทำงาน
- **`deactivate`** ยังคงเป็นนามแฝงความเข้ากันได้สำหรับการล้างข้อมูลที่เลิกใช้แล้ว จนถึง
  หลังวันที่ 2026-08-16 Plugin ใหม่ควรใช้ `gateway_stop`
- **`onResolution` ใน `before_tool_call`** ขณะนี้ใช้ยูเนียนแบบมีชนิด
  `PluginApprovalResolution` (`allow-once` / `allow-always` / `deny` /
  `timeout` / `cancelled`) แทน `string` แบบอิสระ
- **`api.registerSessionExtension` / `api.enqueueNextTurnInjection`** ยังคงอยู่
  ในฐานะนามแฝงความเข้ากันได้ระดับบนสุด Plugin ใหม่ควรใช้
  `api.session.state.registerSessionExtension(...)` และ
  `api.session.workflow.enqueueNextTurnInjection(...)`

สำหรับรายการทั้งหมด ได้แก่ การลงทะเบียนความสามารถของหน่วยความจำ โปรไฟล์การคิดของผู้ให้บริการ
ผู้ให้บริการการยืนยันตัวตนภายนอก ชนิดการค้นหาผู้ให้บริการ ตัวเข้าถึงรันไทม์ของงาน
และการเปลี่ยนชื่อ `command-auth` → `command-status` โปรดดู
[การย้าย Plugin SDK → การเลิกใช้งานที่ยังมีผลอยู่](/th/plugins/sdk-migration#active-deprecations)
