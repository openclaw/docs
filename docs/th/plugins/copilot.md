Plugin ภายนอก `@openclaw/copilot` เรียกใช้รอบการทำงานของเอเจนต์ Copilot แบบการสมัครใช้งานที่ฝังตัวผ่าน GitHub Copilot CLI (`@github/copilot-sdk`) แทน
ชุดควบคุมในตัวของ OpenClaw เซสชัน Copilot CLI เป็นเจ้าของลูปเอเจนต์ระดับล่าง:
การเรียกใช้เครื่องมือแบบเนทีฟ, Compaction แบบเนทีฟ (`infiniteSessions`) และ
สถานะเธรดที่จัดการโดย CLI ภายใต้ `copilotHome` ส่วน OpenClaw ยังคงเป็นเจ้าของช่องทางแชต
ไฟล์เซสชัน การเลือกโมเดล เครื่องมือแบบไดนามิก (เชื่อมผ่านบริดจ์) การอนุมัติ
การส่งสื่อ สำเนาทรานสคริปต์ที่มองเห็นได้ คำถามแทรก `/btw` (ดู
[คำถามแทรก (`/btw`)](#side-questions-btw)) และ `openclaw doctor`

สำหรับภาพรวมที่กว้างขึ้นของการแบ่งโมเดล/ผู้ให้บริการ/รันไทม์ ให้เริ่มที่
[รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes)

## ข้อกำหนด

- OpenClaw ที่ติดตั้ง Plugin `@openclaw/copilot` แล้ว
- หากการกำหนดค่าของคุณใช้ `plugins.allow` ให้รวม `copilot` (รหัสแมนิเฟสต์ที่
  Plugin ประกาศ) รายการอนุญาตสำหรับชื่อแพ็กเกจ npm
  `@openclaw/copilot` จะไม่ตรงกันและทำให้ Plugin ยังคงถูกบล็อก แม้จะตั้งค่า
  `agentRuntime.id: "copilot"` แล้วก็ตาม
- การสมัครใช้งาน GitHub Copilot ที่สามารถควบคุม Copilot CLI หรือ
  ตัวแปรสภาพแวดล้อม `gitHubToken` / รายการโปรไฟล์การยืนยันตัวตนสำหรับการทำงานแบบไม่มีส่วนติดต่อผู้ใช้หรือ Cron
- ไดเรกทอรี `copilotHome` ที่เขียนได้ ค่าเริ่มต้นคือ `<agentDir>/copilot` เมื่อ
  OpenClaw ระบุไดเรกทอรีเอเจนต์ มิฉะนั้นจะเป็น
  `~/.openclaw/agents/<agentId>/copilot`

`openclaw doctor` เรียกใช้[สัญญา doctor](#doctor) ของ Plugin สำหรับ
ความเป็นเจ้าของสถานะเซสชันและการย้ายการกำหนดค่าในอนาคต โดยไม่ได้ตรวจสอบ
สภาพแวดล้อม Copilot CLI

## การติดตั้ง

รันไทม์ Copilot จัดส่งเป็น Plugin ภายนอก เพื่อให้แพ็กเกจหลัก `openclaw`
ไม่ต้องรวม `@github/copilot-sdk` หรือไบนารี CLI
`@github/copilot-<platform>-<arch>` ที่เฉพาะเจาะจงตามแพลตฟอร์ม (รวมกันประมาณ 260 MB)
ติดตั้งเฉพาะสำหรับเอเจนต์ที่เลือกใช้รันไทม์นี้:

```bash
openclaw plugins install @openclaw/copilot
```

วิซาร์ดการตั้งค่าจะติดตั้ง Plugin โดยอัตโนมัติเมื่อคุณเลือกโมเดล
`github-copilot/*` เป็นครั้งแรก **และ** การกำหนดค่าของคุณกำหนดเส้นทางโมเดลนั้น (หรือ
ผู้ให้บริการของโมเดล) ไปยังรันไทม์ Copilot ผ่าน `agentRuntime: { id: "copilot" }`; ดู
[เริ่มต้นอย่างรวดเร็ว](#quickstart) หากไม่มีการเลือกใช้นี้ OpenClaw จะใช้
ผู้ให้บริการ GitHub Copilot ในตัวและจะไม่ติดตั้ง Plugin นี้

รันไทม์ค้นหา SDK ตามลำดับนี้:

1. `import("@github/copilot-sdk")` จากแพ็กเกจ `@openclaw/copilot`
   ที่ติดตั้งแล้ว
2. ไดเรกทอรีสำรอง `~/.openclaw/npm-runtime/copilot/` (เป้าหมายการติดตั้งแบบ
   ตามต้องการรุ่นเก่า)

หาก SDK หายไป ระบบจะแสดงข้อผิดพลาดหนึ่งรายการพร้อมรหัส `COPILOT_SDK_MISSING` และ
คำสั่งติดตั้งใหม่ข้างต้น

## เริ่มต้นอย่างรวดเร็ว

ตรึงโมเดลหนึ่งรายการ (หรือผู้ให้บริการหนึ่งราย) ไว้กับชุดควบคุม:

```json5
{
  agents: {
    defaults: {
      model: "github-copilot/auto",
      models: {
        "github-copilot/auto": {
          agentRuntime: { id: "copilot" },
        },
      },
    },
  },
}
```

ตั้งค่า `agentRuntime.id` ในรายการโมเดลเดียวเพื่อกำหนดเส้นทางเฉพาะโมเดลนั้นผ่าน
ชุดควบคุม หรือตั้งค่าในผู้ให้บริการเพื่อกำหนดเส้นทางทุกโมเดลภายใต้ผู้ให้บริการนั้น

`github-copilot/auto` เป็นจุดเริ่มต้นที่ใช้งานได้ทั่วไป โมเดล Copilot ที่ระบุชื่อขึ้นอยู่กับ
นโยบายบัญชีและองค์กร โปรดยืนยันว่า Copilot CLI ที่ยืนยันตัวตนแล้ว
แสดงโมเดลดังกล่าวจริงก่อนตรึงโมเดลนั้น

## ผู้ให้บริการที่รองรับ

ชุดควบคุมรองรับผู้ให้บริการมาตรฐาน `github-copilot` (เป็นเจ้าของโดย
`extensions/github-copilot`) รวมถึงรายการ `models.providers` แบบกำหนดเอง เมื่อ
โมเดลมี `baseUrl` ที่ไม่ว่างและมีรูปแบบ `api` รูปแบบใดรูปแบบหนึ่งต่อไปนี้:

- `anthropic-messages`
- `azure-openai-responses`
- `ollama` (การเติมข้อความที่เข้ากันได้กับ OpenAI)
- `openai-completions`
- `openai-responses`

รหัสผู้ให้บริการแบบเนทีฟ (`openai`, `anthropic`, `google`, `ollama`) ยังคงอยู่ภายใต้การดูแลของ
รันไทม์แบบเนทีฟของแต่ละราย ให้ใช้รหัสผู้ให้บริการแบบกำหนดเองที่แตกต่างกันเพื่อกำหนดเส้นทางเอนด์พอยต์
ผ่าน Copilot BYOK แทน

เอนด์พอยต์ Copilot BYOK ต้องเป็น URL HTTPS สาธารณะ ชุดควบคุมจะมอบ
พร็อกซีลูปแบ็กแยกสำหรับแต่ละครั้งที่ลองให้แก่ Copilot SDK จากนั้นส่งต่อทราฟฟิกของผู้ให้บริการ
ผ่านเส้นทางการดึงข้อมูลที่มีการป้องกันของ OpenClaw เพื่อให้การตรึง DNS และนโยบาย SSRF ยังคง
อยู่ภายใต้การดูแลของ OpenClaw ให้ใช้รันไทม์ OpenClaw แบบเนทีฟสำหรับ Ollama หรือ LM
Studio ภายในเครื่อง หรือเซิร์ฟเวอร์โมเดลบน LAN

## BYOK

Copilot BYOK ใช้สัญญาผู้ให้บริการแบบกำหนดเองระดับเซสชันของ SDK โดย OpenClaw
ส่งเอนด์พอยต์โมเดลที่ค้นหาได้ คีย์ API โหมด bearer token ส่วนหัว รหัสโมเดล
และขีดจำกัดบริบท/เอาต์พุต ส่วนตรรกะการขนส่งของผู้ให้บริการยังคงอยู่ใน SDK ไม่ใช่
แกนหลัก

```json5
{
  agents: {
    defaults: {
      model: "custom-proxy/llama-3.1-8b",
      models: {
        "custom-proxy/llama-3.1-8b": {
          agentRuntime: { id: "copilot" },
        },
      },
    },
  },
  models: {
    mode: "merge",
    providers: {
      "custom-proxy": {
        baseUrl: "https://api.example.com/v1",
        apiKey: "${CUSTOM_PROXY_API_KEY}",
        api: "openai-responses",
        authHeader: true,
        models: [{ id: "llama-3.1-8b", name: "Llama 3.1 8B" }],
      },
    },
  },
}
```

เซสชัน BYOK ใช้คีย์แยกจากเซสชันการสมัครใช้งาน และแยกจาก
เอนด์พอยต์หรือข้อมูลประจำตัว BYOK อื่น การหมุนเวียนคีย์ ส่วนหัว โมเดล หรือเอนด์พอยต์
จะเริ่มเซสชัน Copilot SDK ใหม่ แทนการดำเนินสถานะที่เข้ากันไม่ได้ต่อ

## การยืนยันตัวตน

ลำดับความสำคัญที่ใช้ต่อเอเจนต์ระหว่าง `runCopilotAttempt`:

1. **`useLoggedInUser: true` ที่ระบุชัดเจน** ในอินพุตของการลอง — ใช้
   ผู้ใช้ที่เข้าสู่ระบบ Copilot CLI ภายใต้ `copilotHome` ของเอเจนต์
2. **`gitHubToken` ที่ระบุชัดเจน** ในอินพุตของการลอง (ต้องมี `profileId` +
   `profileVersion`) สำหรับการเรียกใช้ CLI โดยตรงและการทดสอบที่ต้อง
   ข้ามการค้นหาโปรไฟล์การยืนยันตัวตน
3. **`resolvedApiKey` + `authProfileId` ที่ค้นหาตามสัญญา** — เส้นทางหลัก
   สำหรับการใช้งานจริง แกนหลักจะค้นหาโปรไฟล์การยืนยันตัวตน `github-copilot`
   ที่กำหนดค่าไว้ของเอเจนต์ (`src/infra/provider-usage.auth.ts:resolveProviderAuths`) ก่อน
   เรียกใช้ชุดควบคุม ดังนั้นโปรไฟล์การยืนยันตัวตน `github-copilot:<profile>` จึงทำงานได้
   ตั้งแต่ต้นจนจบสำหรับการตั้งค่าแบบไม่มีส่วนติดต่อผู้ใช้, Cron หรือหลายโปรไฟล์โดยไม่ต้องใช้ตัวแปรสภาพแวดล้อม
4. **การสำรองด้วยตัวแปรสภาพแวดล้อม** ตรวจสอบตามลำดับนี้ (ค่าที่ไม่ว่างค่าแรกมีผล
   สตริงว่างถือว่าไม่มีค่า และสะท้อนลำดับความสำคัญของผู้ให้บริการ `github-copilot`
   ที่จัดส่งแล้วใน `extensions/github-copilot/auth.ts`):
   1. `OPENCLAW_GITHUB_TOKEN` — ค่าทับเฉพาะชุดควบคุม ช่วยให้คุณตรึง
      โทเค็นสำหรับชุดควบคุม OpenClaw ได้โดยไม่รบกวน `gh` ทั่วทั้งระบบ /
      การกำหนดค่า Copilot CLI
   2. `COPILOT_GITHUB_TOKEN` — ตัวแปรสภาพแวดล้อมมาตรฐานของ Copilot SDK / CLI
   3. `GH_TOKEN` — ตัวแปรสภาพแวดล้อมมาตรฐานของ CLI `gh`
   4. `GITHUB_TOKEN` — โทเค็นสำรองทั่วไปของ GitHub

   รหัสโปรไฟล์พูลที่สังเคราะห์ขึ้นคือ `env:<NAME>`; เวอร์ชันโปรไฟล์เป็น
   ลายนิ้วมือ sha256 ของโทเค็นที่ย้อนกลับไม่ได้ ดังนั้นการหมุนเวียนค่าตัวแปรสภาพแวดล้อม
   จะล้างพูลไคลเอนต์อย่างหมดจด

5. **ค่าเริ่มต้น `useLoggedInUser`** เมื่อไม่มีสัญญาณโทเค็น

แต่ละเอเจนต์จะได้รับ `copilotHome` ของตนเอง เพื่อไม่ให้โทเค็น เซสชัน และ
การกำหนดค่า Copilot CLI รั่วไหลระหว่างเอเจนต์บนเครื่องเดียวกัน ค่าเริ่มต้น:
`<agentDir>/copilot` (เก็บสถานะ SDK ไว้นอกไดเรกทอรีเดียวกับ
`models.json` / `auth-profiles.json` ของ OpenClaw) หรือ
`~/.openclaw/agents/<agentId>/copilot` เมื่อไม่ได้ระบุไดเรกทอรีเอเจนต์
ใช้ `copilotHome: <path>` ในอินพุตของการลองเพื่อกำหนดตำแหน่งแบบกำหนดเอง
(ตัวอย่างเช่น เมานต์ที่ใช้ร่วมกันสำหรับการย้ายข้อมูล)

การทดสอบชุดควบคุมแบบสดใช้ `OPENCLAW_COPILOT_AGENT_LIVE_TOKEN` สำหรับโทเค็น
โดยตรง การตั้งค่าการทดสอบแบบสดที่ใช้ร่วมกันจะล้าง `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`
และ `GITHUB_TOKEN` หลังจากเตรียมโปรไฟล์การยืนยันตัวตนจริงไว้ในโฮมสำหรับการทดสอบที่แยกไว้
ดังนั้นค่า `gh auth token` ที่ส่งผ่านตัวแปรเฉพาะจึงช่วยหลีกเลี่ยง
การข้ามที่ผิดพลาดโดยไม่รั่วไหลไปยังชุดทดสอบที่ไม่เกี่ยวข้อง

## พื้นผิวการกำหนดค่า

ชุดควบคุมอ่านการกำหนดค่าจากอินพุตต่อการลอง (`runCopilotAttempt({...})`)
รวมกับค่าเริ่มต้นจากตัวแปรสภาพแวดล้อมชุดเล็กภายใน `extensions/copilot/src/`:

| ฟิลด์                    | วัตถุประสงค์                                                                                                                                                                                                                                                                                         |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `copilotHome`            | ไดเรกทอรีสถานะ CLI ต่อเอเจนต์ (ค่าเริ่มต้นอยู่ด้านบน)                                                                                                                                                                                                                                                 |
| `model`                  | สตริงหรือ `{ provider, id, api?, baseUrl?, headers?, authHeader? }` ละเว้นเพื่อใช้การเลือกโมเดลตามปกติของเอเจนต์ ชุดควบคุมจะตรวจสอบว่ารองรับผู้ให้บริการที่ค้นหาได้                                                                                                                   |
| `reasoningEffort`        | `"low" \| "medium" \| "high" \| "xhigh"` แมปจากการค้นหา `ThinkLevel` / `ReasoningLevel` ของ OpenClaw ใน `auto-reply/thinking.ts`                                                                                                                                                          |
| `infiniteSessionConfig`  | ค่าทับเพิ่มเติมสำหรับบล็อก `infiniteSessions` ของ SDK ที่ขับเคลื่อนโดย `harness.compact` ปล่อยไว้ตามเดิมได้อย่างปลอดภัย                                                                                                                                                                                        |
| `hooksConfig`            | การกำหนดค่า `SessionHooks` แบบเนทีฟของ Copilot SDK ที่เลือกใช้ได้ สำหรับคอลแบ็กเครื่องมือ/MCP พรอมต์ผู้ใช้ เซสชัน และข้อผิดพลาด แยกจากฮุกวงจรชีวิตแบบพกพาของ OpenClaw                                                                                                                                   |
| `permissionPolicy`       | ค่าทับเพิ่มเติมสำหรับตัวจัดการ `onPermissionRequest` ของ SDK สำหรับชนิดเครื่องมือในตัวของ SDK (`shell`, `write`, `read`, `url`, `mcp`, `memory`, `hook`) ค่าเริ่มต้นคือ `rejectAllPolicy` เพื่อเป็นกลไกนิรภัย ดูเหตุผลที่ระบบจะไม่เรียกใช้จริงใน[สิทธิ์และ ask_user](#permissions-and-ask_user) |
| `enableSessionTelemetry` | แฟล็กเทเลเมทรีของเซสชัน SDK ที่เลือกใช้ได้                                                                                                                                                                                                                                                            |

ฮุก Plugin ของ OpenClaw ไม่ต้องใช้การกำหนดค่าการลองที่เฉพาะเจาะจงกับ Copilot
ชุดควบคุมเรียกใช้ `before_prompt_build` (และฮุกความเข้ากันได้รุ่นเก่า `before_agent_start`)
`llm_input`, `llm_output` และ `agent_end` ผ่าน
ตัวช่วยมาตรฐานของชุดควบคุม Compaction ของ SDK ที่สำเร็จยังเรียกใช้
`before_compaction` และ `after_compaction` ด้วย เครื่องมือ OpenClaw ที่เชื่อมผ่านบริดจ์จะเรียกใช้
`before_tool_call` และรายงาน `after_tool_call`; `hooksConfig` ยังคงใช้สำหรับ
คอลแบ็กเฉพาะ SDK แบบเนทีฟที่ไม่มีรายการเทียบเท่าแบบพกพา

ส่วนอื่นใน OpenClaw ไม่จำเป็นต้องทราบเกี่ยวกับฟิลด์เหล่านี้ Plugin อื่น
ช่องทาง และโค้ดแกนหลักจะเห็นเฉพาะรูปแบบมาตรฐาน `AgentHarnessAttemptParams` /
`AgentHarnessAttemptResult`

## Compaction

เมื่อ `harness.compact` ทำงาน ชุดควบคุม Copilot SDK จะ:

1. ดำเนินเซสชัน SDK ที่ติดตามไว้ต่อโดยไม่ดำเนินงานที่ค้างอยู่ต่อ
2. เรียก RPC การ Compaction ประวัติระดับเซสชันของ SDK
3. ส่งคืนผลลัพธ์การ Compaction ของ SDK โดยไม่เขียนไฟล์เครื่องหมาย
   ความเข้ากันได้ภายใต้เวิร์กสเปซ

สำเนาทรานสคริปต์ฝั่ง OpenClaw (ด้านล่าง) จะยังคงรับข้อความหลังการ Compaction
เพื่อให้ประวัติแชตที่ผู้ใช้เห็นยังคงสอดคล้องกัน

## การทำสำเนาทรานสคริปต์

`runCopilotAttempt` เขียนข้อความที่สามารถทำสำเนาได้ของแต่ละรอบแบบคู่ขนานลงใน
ทรานสคริปต์การตรวจสอบของ OpenClaw ผ่าน
`extensions/copilot/src/dual-write-transcripts.ts` สำเนานี้มีขอบเขตแยกตาม
เซสชัน (`copilot:${sessionId}`) และใช้คีย์แยกตามข้อความ
(`${role}:${sha256_16(role,content)}`) ดังนั้นรายการจากรอบก่อนหน้าที่ถูกส่งออกมาอีกครั้ง
จะชนกับคีย์ที่มีอยู่บนดิสก์แทนที่จะสร้างรายการซ้ำ

การจำกัดขอบเขตความล้มเหลวสองชั้นครอบการทำสำเนาไว้ เพื่อให้ความล้มเหลวในการเขียน
ทรานสคริปต์ไม่ทำให้ความพยายามล้มเหลว ได้แก่ ตัวห่อภายในแบบพยายามให้ดีที่สุด และ
`.catch(...)` แบบการป้องกันเชิงลึกที่ระดับความพยายาม ระบบจะบันทึก
ความล้มเหลวลงในล็อกโดยไม่แสดงออกมา

ทำให้พฤติกรรมของ `/btw` เหมือนกับรันไทม์อื่นที่ไม่ใช่ Codex

## ข้อจำกัด

- ฮาร์เนสประกาศรองรับ `github-copilot` รวมถึง ID ผู้ให้บริการ BYOK แบบกำหนดเองที่ไม่มีเจ้าของ
  ID ผู้ให้บริการเนทีฟที่มีแมนิเฟสต์เป็นเจ้าของจะยังคงอยู่บนรันไทม์ของเจ้าของ แม้ว่า
  `agentRuntime.id` จะถูกบังคับเป็น `copilot`
- ไม่มีพื้นผิว TUI โดย TUI ของ PI ยังคงเป็นทางเลือกสำรองสำหรับรันไทม์ที่ไม่มี
  พื้นผิวเทียบเท่า
- สถานะเซสชัน PI จะไม่ถูกย้ายเมื่อเอเจนต์เปลี่ยนไปใช้ `copilot`
  การเลือกเกิดขึ้นแยกตามแต่ละความพยายาม โดยเซสชัน PI ที่มีอยู่ยังคงใช้งานได้
- `ask_user` ใช้รันไทม์คำถามของ Gateway ที่เป็นกลางต่อผู้ให้บริการ Control
  UI แสดงการ์ดคำถามแบบเดียวกับคำถามอื่นๆ ของ OpenClaw ช่องทางที่รองรับ
  จะแสดงปุ่มตัวเลือก และข้อความข้อความธรรมดารายการถัดไปในคิว
  จะตอบระเบียน Gateway นั้นให้เสร็จก่อนที่คำขอ SDK จะส่งผลลัพธ์กลับ

## สิทธิ์และ ask_user

การบังคับใช้นโยบายสิทธิ์สำหรับเครื่องมือ OpenClaw ที่เชื่อมผ่านบริดจ์เกิดขึ้น **ภายในตัวห่อเครื่องมือ**
ไม่ใช่ผ่านคอลแบ็ก `onPermissionRequest` ของ SDK โดยใช้
`wrapToolWithBeforeToolCallHook` เดียวกับที่ PI ใช้
(`src/agents/agent-tools.before-tool-call.ts`) ซึ่ง
`createOpenClawCodingTools` นำไปใช้กับเครื่องมือเขียนโค้ดทุกตัว ได้แก่ การตรวจจับลูป นโยบาย
Plugin ที่เชื่อถือได้ ฮุกก่อนเรียกเครื่องมือ และการอนุมัติ Plugin สองระยะผ่าน
Gateway (`plugin.approval.request`) ทั้งหมดทำงานผ่านเส้นทางโค้ดเดียวกันทุกประการ
กับความพยายามของ PI แบบเนทีฟ

เครื่องมือ SDK แต่ละตัวที่บริดจ์เครื่องมือ Copilot ส่งกลับจะมีเครื่องหมายดังนี้:

- `overridesBuiltInTool: true` — แทนที่เครื่องมือในตัวของ Copilot CLI ที่มี
  ชื่อเดียวกัน (edit, read, write, bash, ...) เพื่อให้ทุกการเรียกเครื่องมือถูกส่งกลับ
  ไปยัง OpenClaw
- `skipPermission: true` — สั่ง SDK ไม่ให้เรียก
  `onPermissionRequest({kind: "custom-tool"})` ก่อนเรียกใช้เครื่องมือ
  `execute()` ที่ถูกห่อได้ดำเนินการตรวจสอบนโยบาย OpenClaw ที่ครอบคลุมกว่าอยู่แล้ว
  พรอมต์ระดับ SDK จะไม่ก็ข้ามการบังคับใช้นโยบายของ OpenClaw
  (อนุญาตทั้งหมด) ก็ปิดกั้นทุกการเรียกเครื่องมือ (ปฏิเสธทั้งหมด) ซึ่งทั้งสองแบบ
  ไม่ตรงกับความเท่าเทียมกับ PI

ฮาร์เนส Codex ภายในทรีใช้การแบ่งแบบเดียวกัน โดยเครื่องมือ OpenClaw ที่เชื่อมผ่านบริดจ์
จะถูกห่อ (`extensions/codex/src/app-server/dynamic-tools.ts`) และชนิดการอนุมัติเนทีฟของ
codex-app-server เอง
(`item/commandExecution/requestApproval`, `item/fileChange/requestApproval`,
`item/permissions/requestApproval`) จะถูกส่งผ่าน `plugin.approval.request`
(`extensions/codex/src/app-server/approval-bridge.ts`) กลไกเทียบเท่าของ Copilot SDK
ซึ่งคือ `rejectAllPolicy` แบบปิดเมื่อเกิดความล้มเหลวสำหรับชนิดใดๆ ที่ไม่ใช่ `custom-tool`
ซึ่งมาถึง `onPermissionRequest` เป็นกลไกนิรภัยแบบเดียวกัน และในทางปฏิบัติ
จะไม่ทำงานเลยเพราะ `overridesBuiltInTool: true` เข้าแทนที่เครื่องมือ
ในตัวทั้งหมด

เพื่อให้ชั้นเครื่องมือที่ถูกห่อสามารถตัดสินใจด้านนโยบายได้เทียบเท่ากับ PI
ฮาร์เนสจะส่งต่อบริบทเครื่องมือของความพยายาม PI ทั้งหมดไปยัง
`createOpenClawCodingTools`: ข้อมูลระบุตัวตน (`senderIsOwner`, `memberRoleIds`,
`ownerOnlyToolAllowlist`, ...), ช่องทาง/การกำหนดเส้นทาง (`groupId`,
`currentChannelId`, `replyToMode`, ตัวสลับเครื่องมือข้อความ), การยืนยันตัวตน
(`authProfileStore`), ข้อมูลระบุการรัน (`sessionKey` / `runSessionKey` ที่ได้มา
จาก `sandboxSessionKey`, `runId`), บริบทโมเดล (`modelApi`,
`modelContextWindowTokens`, `modelCompat`, `modelHasVision`) และฮุกการรัน
(`onToolOutcome`, `onYield`) หากไม่มีฟิลด์เหล่านี้ รายการอนุญาตเฉพาะเจ้าของ
จะปฏิเสธโดยค่าเริ่มต้นโดยไม่แจ้ง นโยบายความเชื่อถือของ Plugin จะไม่สามารถระบุ
ขอบเขตที่ถูกต้องได้ และ `session_status: "current"` จะให้ผลเป็นคีย์แซนด์บ็อกซ์ที่ล้าสมัย
ตัวสร้างบริดจ์คือ `extensions/copilot/src/tool-bridge.ts` ซึ่งสะท้อนการเรียก PI
ที่เป็นแหล่งข้อมูลหลัก ณ `src/agents/embedded-agent-runner/run/attempt.ts:1262`
`runAttempt` ระบุบริบทแซนด์บ็อกซ์ผ่านจุดเชื่อมร่วม
`resolveSandboxContext` ส่งไดเรกทอรีทำงานที่มีผลจริงให้ SDK
และส่งต่อ `sandbox` รวมถึงเวิร์กสเปซสำหรับสร้างเอเจนต์ย่อยไปยังบริดจ์เครื่องมือ
บริดจ์ยังส่งต่อส่วนควบคุมการสร้างเครื่องมือแบบจำกัดขอบเขตที่สามารถบังคับใช้ได้
ณ ขอบเขต SDK ได้แก่ `includeCoreTools`, รายการอนุญาตเครื่องมือของรันไทม์
และ `toolConstructionPlan`

บริดจ์ยังใช้ตัวช่วยพื้นผิวเครื่องมือของฮาร์เนสร่วมจาก
`openclaw/plugin-sdk/agent-harness-tool-runtime` เพื่อให้เทียบเท่ากับ PI เมื่อ
เปิดใช้งานการค้นหาเครื่องมือ SDK จะเห็นเครื่องมือควบคุมแบบกะทัดรัดพร้อมตัวดำเนินการ
แค็ตตาล็อกที่ซ่อนอยู่ แทนที่จะเห็นสคีมาเครื่องมือ OpenClaw ทุกตัว เมื่อเปิดใช้งาน
โหมดโค้ด ตัวช่วยจะสร้างพื้นผิวควบคุมโหมดโค้ดและวงจรชีวิตแค็ตตาล็อกแบบเดียวกับ
ที่ฮาร์เนสเอเจนต์อื่นใช้ ค่าเริ่มต้นแบบกระชับสำหรับโมเดลภายใน การกรองสคีมาที่เข้ากันได้
กับรันไทม์ การเติมข้อมูลไดเรกทอรี และการล้างแค็ตตาล็อกจะยังคงอยู่ในตัวช่วยร่วม
เพื่อไม่ให้ฮาร์เนส Copilot และฮาร์เนสที่เกี่ยวข้องกับ Codex มีพฤติกรรมคลาดเคลื่อนกัน

### โทเค็น GitHub ระดับเซสชัน

สัญญา Copilot SDK แยกโทเค็น GitHub **ระดับไคลเอนต์**
(`CopilotClientOptions.gitHubToken` ซึ่งใช้ยืนยันตัวตนของกระบวนการ CLI เอง)
ออกจากโทเค็น **ระดับเซสชัน** (`SessionConfig.gitHubToken` ซึ่งกำหนด
การยกเว้นเนื้อหา การกำหนดเส้นทางโมเดล และโควตาสำหรับเซสชันนั้น โดยมีผลกับ
ทั้ง `createSession` และ `resumeSession`) ฮาร์เนสระบุการยืนยันตัวตนหนึ่งครั้งผ่าน
`resolveCopilotAuth` และตั้งค่าทั้งสองฟิลด์เมื่อโหมดการยืนยันตัวตนเป็น `gitHubToken`
(ค่า `auth.gitHubToken` ที่ระบุอย่างชัดเจน หรือ `resolvedApiKey` ที่ระบุตามสัญญาจาก
โปรไฟล์การยืนยันตัวตน `github-copilot` ที่กำหนดค่าไว้) เมื่อโหมดที่ระบุได้เป็น
`useLoggedInUser` ระบบจะละฟิลด์ระดับเซสชันไว้ เพื่อให้ SDK ยังคง
อนุมานข้อมูลระบุตัวตนจากข้อมูลระบุตัวตนที่เข้าสู่ระบบอยู่

`ask_user` ใช้ `SessionConfig.onUserInputRequest` บริดจ์จะลงทะเบียน
ตัวเลือก SDK หรือพรอมต์ข้อความอิสระที่ไม่มีตัวเลือกเป็นคำถามของ Gateway โดยยอมรับ
ดัชนีหรือลาเบลตัวเลือกสำหรับคำขอแบบมีตัวเลือกตายตัว และยอมรับคำตอบแบบอิสระ
เมื่อคำขอ SDK อนุญาต การยกเลิกความพยายามของ OpenClaw จะยกเลิก
ระเบียน Gateway และส่งคำตอบ SDK ว่างกลับ
