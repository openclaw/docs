Plugin ภายนอก `@openclaw/copilot` เรียกใช้รอบการทำงานของเอเจนต์ Copilot แบบการสมัครใช้งานที่ฝังอยู่ผ่าน GitHub Copilot CLI (`@github/copilot-sdk`) แทน
ฮาร์เนสในตัวของ OpenClaw เซสชัน Copilot CLI เป็นเจ้าของลูปเอเจนต์ระดับล่าง:
การเรียกใช้เครื่องมือแบบเนทีฟ, Compaction แบบเนทีฟ (`infiniteSessions`) และ
สถานะเธรดที่ CLI จัดการภายใต้ `copilotHome` ส่วน OpenClaw ยังคงเป็นเจ้าของช่องทางแชต
ไฟล์เซสชัน การเลือกโมเดล เครื่องมือแบบไดนามิก (ผ่านบริดจ์) การอนุมัติ
การส่งสื่อ มิเรอร์ทรานสคริปต์ที่มองเห็นได้ คำถามแทรก `/btw` (ดู
[คำถามแทรก (`/btw`)](#side-questions-btw)) และ `openclaw doctor`

สำหรับภาพรวมการแบ่งความรับผิดชอบระหว่างโมเดล/ผู้ให้บริการ/รันไทม์ ให้เริ่มที่
[รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes)

## ข้อกำหนด

- OpenClaw ที่ติดตั้ง Plugin `@openclaw/copilot` แล้ว
- หากการกำหนดค่าของคุณใช้ `plugins.allow` ให้รวม `copilot` (รหัสแมนิเฟสต์ที่
  Plugin ประกาศ) รายการอนุญาตที่ใช้ชื่อแพ็กเกจ npm
  `@openclaw/copilot` จะไม่ตรงกันและทำให้ Plugin ยังคงถูกบล็อก แม้จะตั้งค่า
  `agentRuntime.id: "copilot"` แล้วก็ตาม
- การสมัครใช้งาน GitHub Copilot ที่สามารถควบคุม Copilot CLI หรือ
  ตัวแปรสภาพแวดล้อม `gitHubToken` / รายการโปรไฟล์การยืนยันตัวตนสำหรับการทำงานแบบไม่มีส่วนติดต่อผู้ใช้หรือ Cron
- ไดเรกทอรี `copilotHome` ที่เขียนได้ ค่าเริ่มต้นคือ `<agentDir>/copilot` เมื่อ
  OpenClaw ระบุไดเรกทอรีเอเจนต์ มิฉะนั้นคือ
  `~/.openclaw/agents/<agentId>/copilot`

`openclaw doctor` เรียกใช้[สัญญา doctor](#doctor) ของ Plugin สำหรับ
ความเป็นเจ้าของสถานะเซสชันและการย้ายการกำหนดค่าในอนาคต โดยจะไม่ตรวจสอบ
สภาพแวดล้อม Copilot CLI

## การติดตั้ง

รันไทม์ Copilot จัดส่งเป็น Plugin ภายนอก เพื่อให้แพ็กเกจหลัก `openclaw`
ไม่ต้องรวม `@github/copilot-sdk` หรือไบนารี CLI `@github/copilot-<platform>-<arch>`
เฉพาะแพลตฟอร์ม (รวมกันประมาณ 260 MB)
ติดตั้งเฉพาะสำหรับเอเจนต์ที่เลือกใช้รันไทม์นี้:

```bash
openclaw plugins install @openclaw/copilot
```

วิซาร์ดตั้งค่าจะติดตั้ง Plugin โดยอัตโนมัติในครั้งแรกที่คุณเลือก
โมเดล `github-copilot/*` **และ** การกำหนดค่าของคุณกำหนดเส้นทางโมเดลนั้น (หรือ
ผู้ให้บริการของโมเดล) ไปยังรันไทม์ Copilot ผ่าน `agentRuntime: { id: "copilot" }`; ดู
[เริ่มต้นใช้งานด่วน](#quickstart) หากไม่มีการเลือกใช้นี้ OpenClaw จะใช้
ผู้ให้บริการ GitHub Copilot ในตัวและจะไม่ติดตั้ง Plugin นี้

รันไทม์จะค้นหา SDK ตามลำดับนี้:

1. `import("@github/copilot-sdk")` จากแพ็กเกจ `@openclaw/copilot`
   ที่ติดตั้งไว้
2. ไดเรกทอรีสำรอง `~/.openclaw/npm-runtime/copilot/` (เป้าหมายการติดตั้ง
   แบบตามความต้องการรุ่นเก่า)

หากไม่มี SDK ระบบจะแสดงข้อผิดพลาดหนึ่งรายการพร้อมรหัส `COPILOT_SDK_MISSING` และ
คำสั่งติดตั้งใหม่ด้านบน

## เริ่มต้นใช้งานด่วน

ปักหมุดโมเดลหนึ่งรายการ (หรือผู้ให้บริการหนึ่งราย) เข้ากับฮาร์เนส:

```json5
{
  agents: {
    defaults: {
      model: "github-copilot/auto",
      models: {
        "github-copilot/auto": {
          agentRuntime: { id: "copilot" },
        },
      },
    },
  },
}
```

ตั้งค่า `agentRuntime.id` ในรายการโมเดลเดียวเพื่อกำหนดเส้นทางเฉพาะโมเดลนั้นผ่าน
ฮาร์เนส หรือตั้งค่าในผู้ให้บริการเพื่อกำหนดเส้นทางทุกโมเดลภายใต้ผู้ให้บริการนั้น

`github-copilot/auto` เป็นจุดเริ่มต้นที่ใช้ได้ทั่วไป โมเดล Copilot ที่ระบุชื่อ
ขึ้นอยู่กับบัญชีและนโยบายขององค์กร โปรดยืนยันว่า Copilot CLI ที่ยืนยันตัวตนแล้ว
แสดงโมเดลนั้นจริงก่อนปักหมุด

## ผู้ให้บริการที่รองรับ

ฮาร์เนสรองรับผู้ให้บริการมาตรฐาน `github-copilot` (เป็นของ
`extensions/github-copilot`) รวมถึงรายการ `models.providers` แบบกำหนดเองเมื่อ
โมเดลมี `baseUrl` ที่ไม่ว่างและใช้รูปแบบ `api` รูปแบบใดรูปแบบหนึ่งต่อไปนี้:

- `anthropic-messages`
- `azure-openai-responses`
- `ollama` (completions ที่เข้ากันได้กับ OpenAI)
- `openai-completions`
- `openai-responses`

รหัสผู้ให้บริการแบบเนทีฟ (`openai`, `anthropic`, `google`, `ollama`) ยังคงเป็นของ
รันไทม์แบบเนทีฟที่เกี่ยวข้อง หากต้องการกำหนดเส้นทางเอนด์พอยต์ผ่าน Copilot BYOK
ให้ใช้รหัสผู้ให้บริการแบบกำหนดเองที่แตกต่างกันแทน

เอนด์พอยต์ Copilot BYOK ต้องเป็น URL HTTPS สาธารณะ ฮาร์เนสจะมอบ
พร็อกซีลูปแบ็กต่อความพยายามหนึ่งครั้งให้ Copilot SDK แล้วส่งต่อทราฟฟิกของผู้ให้บริการ
ผ่านเส้นทาง fetch ที่มีการป้องกันของ OpenClaw เพื่อให้การตรึง DNS และนโยบาย SSRF
ยังคงเป็นความรับผิดชอบของ OpenClaw ใช้รันไทม์ OpenClaw แบบเนทีฟสำหรับ Ollama,
LM Studio หรือเซิร์ฟเวอร์โมเดลใน LAN

## BYOK

Copilot BYOK ใช้สัญญาผู้ให้บริการแบบกำหนดเองระดับเซสชันของ SDK โดย OpenClaw
จะส่งเอนด์พอยต์โมเดลที่แก้ไขแล้ว คีย์ API โหมดโทเค็น bearer ส่วนหัว รหัสโมเดล
และขีดจำกัดบริบท/เอาต์พุต ส่วนตรรกะการส่งข้อมูลของผู้ให้บริการยังคงอยู่ใน SDK ไม่ใช่
แกนหลัก

```json5
{
  agents: {
    defaults: {
      model: "custom-proxy/llama-3.1-8b",
      models: {
        "custom-proxy/llama-3.1-8b": {
          agentRuntime: { id: "copilot" },
        },
      },
    },
  },
  models: {
    mode: "merge",
    providers: {
      "custom-proxy": {
        baseUrl: "https://api.example.com/v1",
        apiKey: "${CUSTOM_PROXY_API_KEY}",
        api: "openai-responses",
        authHeader: true,
        models: [{ id: "llama-3.1-8b", name: "Llama 3.1 8B" }],
      },
    },
  },
}
```

เซสชัน BYOK ใช้คีย์แยกจากเซสชันการสมัครใช้งาน รวมถึงจากเอนด์พอยต์หรือข้อมูลประจำตัว
BYOK อื่น การหมุนเวียนคีย์ ส่วนหัว โมเดล หรือเอนด์พอยต์จะเริ่มเซสชัน Copilot SDK
ใหม่ แทนที่จะกลับมาทำงานต่อจากสถานะที่เข้ากันไม่ได้

## การยืนยันตัวตน

ลำดับความสำคัญที่ใช้กับแต่ละเอเจนต์ระหว่าง `runCopilotAttempt`:

1. **`useLoggedInUser: true` ที่ระบุอย่างชัดเจน** ในอินพุตของความพยายาม — ใช้
   ผู้ใช้ที่เข้าสู่ระบบ Copilot CLI ภายใต้ `copilotHome` ของเอเจนต์
2. **`gitHubToken` ที่ระบุอย่างชัดเจน** ในอินพุตของความพยายาม (ต้องใช้ `profileId` +
   `profileVersion`) สำหรับการเรียกใช้ CLI โดยตรงและการทดสอบที่ต้อง
   ข้ามการแก้ไขโปรไฟล์การยืนยันตัวตน
3. **`resolvedApiKey` + `authProfileId` ที่แก้ไขตามสัญญา** — เส้นทางหลัก
   ในระบบใช้งานจริง แกนหลักจะแก้ไขโปรไฟล์การยืนยันตัวตน `github-copilot` ที่กำหนดค่าไว้
   ของเอเจนต์ (`src/infra/provider-usage.auth.ts:resolveProviderAuths`) ก่อน
   เรียกใช้ฮาร์เนส ดังนั้นโปรไฟล์การยืนยันตัวตน `github-copilot:<profile>` จึงทำงานได้
   ครบวงจรสำหรับการตั้งค่าแบบไม่มีส่วนติดต่อผู้ใช้, Cron หรือหลายโปรไฟล์โดยไม่ต้องใช้ตัวแปรสภาพแวดล้อม
4. **การใช้ตัวแปรสภาพแวดล้อมเป็นทางเลือกสำรอง** ตรวจสอบตามลำดับนี้ (ค่าที่ไม่ว่างค่าแรกจะถูกใช้
   สตริงว่างถือว่าไม่มีค่า โดยสะท้อนลำดับความสำคัญของผู้ให้บริการ `github-copilot`
   ที่จัดส่งใน `extensions/github-copilot/auth.ts`):
   1. `OPENCLAW_GITHUB_TOKEN` — การแทนที่เฉพาะฮาร์เนส ช่วยให้คุณปักหมุด
      โทเค็นสำหรับฮาร์เนส OpenClaw โดยไม่รบกวน `gh` ทั่วทั้งระบบ /
      การกำหนดค่า Copilot CLI
   2. `COPILOT_GITHUB_TOKEN` — ตัวแปรสภาพแวดล้อมมาตรฐานของ Copilot SDK / CLI
   3. `GH_TOKEN` — ตัวแปรสภาพแวดล้อมมาตรฐานของ CLI `gh`
   4. `GITHUB_TOKEN` — ทางเลือกสำรองสำหรับโทเค็น GitHub แบบทั่วไป

   รหัสโปรไฟล์พูลที่สังเคราะห์ขึ้นคือ `env:<NAME>`; เวอร์ชันโปรไฟล์เป็น
   ลายนิ้วมือ sha256 ของโทเค็นที่ย้อนกลับไม่ได้ ดังนั้นการหมุนเวียนค่าในตัวแปรสภาพแวดล้อม
   จะล้างพูลไคลเอนต์อย่างสมบูรณ์

5. **`useLoggedInUser` เริ่มต้น** เมื่อไม่มีสัญญาณโทเค็น

แต่ละเอเจนต์จะได้รับ `copilotHome` ของตนเอง เพื่อให้โทเค็น เซสชัน และ
การกำหนดค่า Copilot CLI ไม่รั่วไหลระหว่างเอเจนต์บนเครื่องเดียวกัน ค่าเริ่มต้น:
`<agentDir>/copilot` (เก็บสถานะ SDK ไว้นอกไดเรกทอรีเดียวกับ
`models.json` / `auth-profiles.json` ของ OpenClaw) หรือ
`~/.openclaw/agents/<agentId>/copilot` เมื่อไม่ได้ระบุไดเรกทอรีเอเจนต์
แทนที่ด้วย `copilotHome: <path>` ในอินพุตของความพยายามเพื่อใช้
ตำแหน่งที่กำหนดเอง (เช่น เมานต์ที่ใช้ร่วมกันสำหรับการย้ายข้อมูล)

การทดสอบฮาร์เนสแบบสดใช้ `OPENCLAW_COPILOT_AGENT_LIVE_TOKEN` สำหรับโทเค็น
โดยตรง การตั้งค่าการทดสอบแบบสดที่ใช้ร่วมกันจะล้าง `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`
และ `GITHUB_TOKEN` หลังจากจัดเตรียมโปรไฟล์การยืนยันตัวตนจริงในโฮมสำหรับการทดสอบ
ที่แยกออกมา ดังนั้นค่า `gh auth token` ที่ส่งผ่านตัวแปรเฉพาะจะช่วยหลีกเลี่ยง
การข้ามการทดสอบที่ผิดพลาดโดยไม่รั่วไหลไปยังชุดทดสอบอื่น

## พื้นผิวการกำหนดค่า

ฮาร์เนสอ่านการกำหนดค่าจากอินพุตต่อความพยายาม (`runCopilotAttempt({...})`)
ร่วมกับค่าเริ่มต้นจากตัวแปรสภาพแวดล้อมจำนวนเล็กน้อยภายใน `extensions/copilot/src/`:

| ฟิลด์                    | วัตถุประสงค์                                                                                                                                                                                                                                                                                         |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `copilotHome`            | ไดเรกทอรีสถานะ CLI ต่อเอเจนต์ (ค่าเริ่มต้นอยู่ด้านบน)                                                                                                                                                                                                                                                 |
| `model`                  | สตริงหรือ `{ provider, id, api?, baseUrl?, headers?, authHeader? }` ละเว้นเพื่อใช้การเลือกโมเดลปกติของเอเจนต์ ฮาร์เนสจะตรวจสอบว่ารองรับผู้ให้บริการที่แก้ไขแล้ว                                                                                                                   |
| `reasoningEffort`        | `"low" \| "medium" \| "high" \| "xhigh"` แมปจากการแก้ไข `ThinkLevel` / `ReasoningLevel` ของ OpenClaw ใน `auto-reply/thinking.ts`                                                                                                                                                          |
| `infiniteSessionConfig`  | การแทนที่ที่เป็นทางเลือกสำหรับบล็อก `infiniteSessions` ของ SDK ซึ่งขับเคลื่อนโดย `harness.compact` ปล่อยไว้ตามเดิมได้อย่างปลอดภัย                                                                                                                                                                                        |
| `hooksConfig`            | การกำหนดค่า `SessionHooks` แบบเนทีฟของ Copilot SDK ที่เป็นทางเลือกสำหรับคอลแบ็กของเครื่องมือ/MCP พรอมต์ผู้ใช้ เซสชัน และข้อผิดพลาด โดยแยกจากฮุกวงจรชีวิตแบบพกพาของ OpenClaw                                                                                                                                   |
| `permissionPolicy`       | การแทนที่ที่เป็นทางเลือกสำหรับตัวจัดการ `onPermissionRequest` ของ SDK สำหรับชนิดเครื่องมือ SDK ในตัว (`shell`, `write`, `read`, `url`, `mcp`, `memory`, `hook`) ค่าเริ่มต้นคือ `rejectAllPolicy` เพื่อเป็นกลไกนิรภัย ดูเหตุผลที่กลไกนี้ไม่เคยทำงานจริงได้ที่ [สิทธิ์และ ask_user](#permissions-and-ask_user) |
| `enableSessionTelemetry` | แฟล็กเทเลเมทรีเซสชัน SDK ที่เป็นทางเลือก                                                                                                                                                                                                                                                            |

ฮุก Plugin ของ OpenClaw ไม่ต้องใช้การกำหนดค่าความพยายามเฉพาะ Copilot
ฮาร์เนสเรียกใช้ `before_prompt_build` (และฮุกความเข้ากันได้รุ่นเก่า `before_agent_start`),
`llm_input`, `llm_output` และ `agent_end` ผ่าน
ตัวช่วยฮาร์เนสมาตรฐาน การ Compaction ของ SDK ที่สำเร็จจะเรียกใช้
`before_compaction` และ `after_compaction` ด้วย เครื่องมือ OpenClaw ที่ผ่านบริดจ์จะเรียกใช้
`before_tool_call` และรายงาน `after_tool_call`; `hooksConfig` ยังคงใช้สำหรับ
คอลแบ็กเฉพาะ SDK แบบเนทีฟที่ไม่มีสิ่งเทียบเท่าแบบพกพา

ส่วนอื่นของ OpenClaw ไม่จำเป็นต้องทราบเกี่ยวกับฟิลด์เหล่านี้ Plugin อื่น
ช่องทาง และโค้ดแกนหลักจะเห็นเฉพาะรูปแบบมาตรฐาน `AgentHarnessAttemptParams` /
`AgentHarnessAttemptResult`

## Compaction

เมื่อ `harness.compact` ทำงาน ฮาร์เนส Copilot SDK จะ:

1. กลับมาทำงานต่อจากเซสชัน SDK ที่ติดตามไว้โดยไม่ดำเนินงานที่ค้างอยู่ต่อ
2. เรียก RPC การ Compaction ประวัติระดับเซสชันของ SDK
3. ส่งคืนผลลัพธ์การ Compaction ของ SDK โดยไม่เขียนไฟล์มาร์กเกอร์
   ความเข้ากันได้ภายใต้เวิร์กสเปซ

มิเรอร์ทรานสคริปต์ฝั่ง OpenClaw (ด้านล่าง) ยังคงรับข้อความหลังการ Compaction
เพื่อให้ประวัติแชตที่ผู้ใช้มองเห็นยังคงสอดคล้องกัน

## การมิเรอร์ทรานสคริปต์

`runCopilotAttempt` เขียนข้อความที่ทำสำเนาได้ของแต่ละรอบพร้อมกันลงใน
ทรานสคริปต์การตรวจสอบของ OpenClaw ผ่าน
`extensions/copilot/src/dual-write-transcripts.ts` สำเนานี้จำกัดขอบเขตแยกตาม
เซสชัน (`copilot:${sessionId}`) และกำหนดคีย์แยกตามข้อความ
(`${role}:${sha256_16(role,content)}`) ดังนั้นรายการจากรอบก่อนหน้าที่ถูกส่งออกมาอีกครั้ง
จะชนกับคีย์บนดิสก์ที่มีอยู่แทนที่จะเกิดรายการซ้ำ

การควบคุมความล้มเหลวสองชั้นครอบการทำสำเนานี้ไว้ เพื่อให้ความล้มเหลวในการเขียน
ทรานสคริปต์ไม่ทำให้ความพยายามล้มเหลว ได้แก่ ตัวห่อหุ้มแบบพยายามให้ดีที่สุดภายใน และ
`.catch(...)` แบบการป้องกันเชิงลึกที่ระดับความพยายาม ระบบจะบันทึก
ความล้มเหลวลงในล็อกโดยไม่แสดงออกมา

## คำถามเสริม (`/btw`)

`/btw` **ไม่ใช่** ความสามารถเนทีฟบนฮาร์เนสนี้ `createCopilotAgentHarness()`
จงใจปล่อยให้ `harness.runSideQuestion` ไม่มีการกำหนดค่า
(ยืนยันใน `extensions/copilot/harness.test.ts`, `describe("runSideQuestion")`)
ดังนั้นตัวจัดส่ง `/btw` ของ OpenClaw (`src/agents/btw.ts`) จึงไหลต่อไปยัง
เส้นทางเดียวกับที่ใช้กับรันไทม์ที่ไม่ใช่ Codex ทุกตัว กล่าวคือเรียกผู้ให้บริการโมเดล
ที่กำหนดค่าไว้โดยตรงด้วยพรอมต์คำถามเสริมแบบสั้น และสตรีมผลลัพธ์กลับผ่าน
`streamSimple` (ไม่มีเซสชัน CLI และไม่ใช้สล็อตพูลเพิ่มเติม)

การทำเช่นนี้สงวนเซสชัน Copilot CLI ไว้สำหรับลูปรอบหลักของเอเจนต์ และ
ทำให้ลักษณะการทำงานของ `/btw` เหมือนกับรันไทม์ที่ไม่ใช่ Codex อื่น ๆ

## ข้อจำกัด

- ฮาร์เนสประกาศรองรับ `github-copilot` รวมถึง ID ผู้ให้บริการ BYOK แบบกำหนดเองที่ไม่มีเจ้าของ
  ID ผู้ให้บริการเนทีฟที่มีเจ้าของตามแมนิเฟสต์จะยังคงอยู่บนรันไทม์ของเจ้าของ แม้ว่า
  `agentRuntime.id` จะถูกบังคับเป็น `copilot`
- ไม่มีพื้นผิว TUI โดย TUI ของ PI ยังคงเป็นทางเลือกสำรองสำหรับรันไทม์ที่ไม่มีพื้นผิว
  คู่กัน
- สถานะเซสชัน PI จะไม่ย้ายเมื่อเอเจนต์เปลี่ยนไปใช้ `copilot`
  การเลือกเกิดขึ้นแยกตามความพยายาม ส่วนเซสชัน PI ที่มีอยู่ยังคงใช้งานได้
- `ask_user` ใช้เส้นทางพรอมต์และการตอบกลับของ OpenClaw แบบเดียวกับฮาร์เนส Codex:
  เมื่อ Copilot SDK ขอข้อมูลจากผู้ใช้ OpenClaw จะโพสต์
  พรอมต์แบบบล็อกไปยังช่องทาง/TUI ที่ใช้งานอยู่ และข้อความผู้ใช้ลำดับถัดไป
  ในคิวจะดำเนินการตามคำขอของ SDK ให้เสร็จสิ้น

## สิทธิ์และ ask_user

การบังคับใช้สิทธิ์สำหรับเครื่องมือ OpenClaw ที่เชื่อมผ่านบริดจ์เกิดขึ้น **ภายในตัวห่อหุ้มเครื่องมือ**
ไม่ใช่ผ่านคอลแบ็ก `onPermissionRequest` ของ SDK โดย
`wrapToolWithBeforeToolCallHook` เดียวกับที่ PI ใช้
(`src/agents/agent-tools.before-tool-call.ts`) จะถูกนำไปใช้โดย
`createOpenClawCodingTools` กับเครื่องมือเขียนโค้ดทุกตัว ได้แก่ การตรวจจับลูป นโยบาย
Plugin ที่เชื่อถือได้ ฮุกก่อนเรียกเครื่องมือ และการอนุมัติ Plugin แบบสองระยะผ่าน
Gateway (`plugin.approval.request`) ทั้งหมดทำงานผ่านเส้นทางโค้ดเดียวกันทุกประการ
กับความพยายามของ PI แบบเนทีฟ

เครื่องมือ SDK แต่ละรายการที่บริดจ์เครื่องมือ Copilot ส่งคืนจะถูกทำเครื่องหมายด้วย:

- `overridesBuiltInTool: true` — แทนที่เครื่องมือในตัวของ Copilot CLI
  ที่มีชื่อเดียวกัน (แก้ไข อ่าน เขียน bash, ...) เพื่อให้ทุกการเรียกเครื่องมือกำหนดเส้นทางกลับ
  ไปยัง OpenClaw
- `skipPermission: true` — บอก SDK ไม่ให้เรียก
  `onPermissionRequest({kind: "custom-tool"})` ก่อนเรียกใช้เครื่องมือ
  `execute()` ที่ห่อหุ้มไว้ดำเนินการตรวจสอบนโยบาย OpenClaw ที่ละเอียดกว่าอยู่แล้ว
  พรอมต์ระดับ SDK จะข้ามการบังคับใช้ของ OpenClaw
  (อนุญาตทั้งหมด) หรือบล็อกทุกการเรียกเครื่องมือ (ปฏิเสธทั้งหมด) ซึ่งไม่มีแบบใด
  เทียบเท่ากับ PI

ฮาร์เนส Codex ภายในทรีใช้การแบ่งแบบเดียวกัน: เครื่องมือ OpenClaw ที่เชื่อมผ่านบริดจ์จะถูก
ห่อหุ้ม (`extensions/codex/src/app-server/dynamic-tools.ts`) และประเภทการอนุมัติเนทีฟของ
codex-app-server เอง
(`item/commandExecution/requestApproval`, `item/fileChange/requestApproval`,
`item/permissions/requestApproval`) จะกำหนดเส้นทางผ่าน `plugin.approval.request`
(`extensions/codex/src/app-server/approval-bridge.ts`) ส่วนสิ่งที่เทียบเท่ากันใน Copilot SDK
คือ `rejectAllPolicy` แบบปิดเมื่อล้มเหลวสำหรับประเภทที่ไม่ใช่ `custom-tool`
ใด ๆ ที่เข้าถึง `onPermissionRequest` ซึ่งเป็นกลไกนิรภัยแบบเดียวกัน และ
ในทางปฏิบัติจะไม่ทำงานเลย เนื่องจาก `overridesBuiltInTool: true` แทนที่
เครื่องมือในตัวทุกตัว

เพื่อให้ชั้นเครื่องมือที่ห่อหุ้มไว้ตัดสินใจด้านนโยบายได้เทียบเท่ากับ PI
ฮาร์เนสจะส่งต่อบริบทเครื่องมือของความพยายาม PI ทั้งหมดไปยัง
`createOpenClawCodingTools`: อัตลักษณ์ (`senderIsOwner`, `memberRoleIds`,
`ownerOnlyToolAllowlist`, ...), ช่องทาง/การกำหนดเส้นทาง (`groupId`,
`currentChannelId`, `replyToMode`, ตัวสลับเครื่องมือข้อความ), การตรวจสอบสิทธิ์
(`authProfileStore`), อัตลักษณ์การรัน (`sessionKey` / `runSessionKey` ที่ได้มา
จาก `sandboxSessionKey`, `runId`), บริบทโมเดล (`modelApi`,
`modelContextWindowTokens`, `modelCompat`, `modelHasVision`) และฮุกการรัน
(`onToolOutcome`, `onYield`) หากไม่มีฟิลด์เหล่านี้ รายการอนุญาตเฉพาะเจ้าของ
จะปฏิเสธโดยปริยายอย่างเงียบ ๆ นโยบายความเชื่อถือของ Plugin จะไม่สามารถระบุขอบเขตที่ถูกต้อง
และ `session_status: "current"` จะระบุเป็นคีย์แซนด์บ็อกซ์ที่ล้าสมัย ตัวสร้าง
บริดจ์คือ `extensions/copilot/src/tool-bridge.ts` ซึ่งสะท้อนการเรียกที่เป็นแหล่งอ้างอิงหลักของ PI
ที่ `src/agents/embedded-agent-runner/run/attempt.ts:1262`
`runAttempt` ระบุบริบทแซนด์บ็อกซ์ผ่านจุดเชื่อมร่วม
`resolveSandboxContext` ส่งไดเรกทอรีทำงานที่มีผลให้ SDK
และส่งต่อ `sandbox` รวมถึงพื้นที่ทำงานสำหรับการสร้างเอเจนต์ย่อยไปยังบริดจ์เครื่องมือ
บริดจ์ยังส่งต่อตัวควบคุมการสร้างเครื่องมือแบบมีขอบเขตที่สามารถบังคับใช้ได้
ที่ขอบเขต SDK ได้แก่ `includeCoreTools`, รายการอนุญาตเครื่องมือของรันไทม์
และ `toolConstructionPlan`

บริดจ์ยังใช้ตัวช่วยพื้นผิวเครื่องมือของฮาร์เนสร่วมจาก
`openclaw/plugin-sdk/agent-harness-tool-runtime` เพื่อให้เทียบเท่ากับ PI เมื่อ
เปิดใช้งานการค้นหาเครื่องมือ SDK จะเห็นเครื่องมือควบคุมแบบกระชับพร้อมตัวดำเนินการแค็ตตาล็อก
ที่ซ่อนอยู่ แทนที่จะเห็นสคีมาเครื่องมือ OpenClaw ทุกตัว เมื่อเปิดใช้งานโหมดโค้ด
ตัวช่วยจะสร้างพื้นผิวควบคุมโหมดโค้ดและวงจรชีวิตแค็ตตาล็อกแบบเดียวกับที่ใช้ใน
ฮาร์เนสเอเจนต์อื่น ๆ ค่าเริ่มต้นแบบกระชับสำหรับโมเดลภายใน การกรองสคีมา
ที่เข้ากันได้กับรันไทม์ การเติมข้อมูลไดเรกทอรี และการล้างแค็ตตาล็อก
ทั้งหมดอยู่ในตัวช่วยร่วม เพื่อไม่ให้ฮาร์เนส Copilot และฮาร์เนสที่อยู่ใกล้เคียงกับ Codex
เกิดความคลาดเคลื่อน

### โทเค็น GitHub ระดับเซสชัน

สัญญาของ Copilot SDK แยกโทเค็น GitHub **ระดับไคลเอนต์**
(`CopilotClientOptions.gitHubToken` ซึ่งใช้ตรวจสอบสิทธิ์ของกระบวนการ CLI เอง)
ออกจากโทเค็น **ระดับเซสชัน** (`SessionConfig.gitHubToken` ซึ่งกำหนด
การยกเว้นเนื้อหา การกำหนดเส้นทางโมเดล และโควตาสำหรับเซสชันนั้น โดยมีผลกับทั้ง
`createSession` และ `resumeSession`) ฮาร์เนสระบุการตรวจสอบสิทธิ์หนึ่งครั้งผ่าน
`resolveCopilotAuth` และตั้งค่าทั้งสองฟิลด์เมื่อโหมดการตรวจสอบสิทธิ์เป็น `gitHubToken`
(`auth.gitHubToken` ที่ระบุอย่างชัดเจน หรือ `resolvedApiKey` ที่ระบุตามสัญญาจาก
โปรไฟล์การตรวจสอบสิทธิ์ `github-copilot` ที่กำหนดค่าไว้) เมื่อโหมดที่ระบุได้เป็น
`useLoggedInUser` ระบบจะละเว้นฟิลด์ระดับเซสชัน เพื่อให้ SDK ยังคง
หาอัตลักษณ์จากอัตลักษณ์ที่เข้าสู่ระบบอยู่

`ask_user` ใช้ `SessionConfig.onUserInputRequest` บริดจ์ยอมรับดัชนี
หรือลาเบลตัวเลือกสำหรับคำขอแบบมีตัวเลือกตายตัว ยอมรับคำตอบแบบข้อความอิสระเมื่อ
คำขอของ SDK อนุญาต และยกเลิกคำขอที่รอดำเนินการเมื่อความพยายามของ OpenClaw
