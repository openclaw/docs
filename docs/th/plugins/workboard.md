Plugin Workboard เพิ่มบอร์ดสไตล์ Kanban ซึ่งเปิดใช้ได้ตามต้องการใน
[UI ควบคุม](/th/web/control-ui): การ์ดงานขนาดเหมาะสำหรับเอเจนต์ การมอบหมายให้เอเจนต์
และลิงก์ย้อนกลับไปยังงาน การรัน และเซสชันแดชบอร์ดของการ์ด

Workboard ตั้งใจออกแบบให้มีขนาดเล็ก โดยติดตามงานปฏิบัติการภายในเครื่องสำหรับ
OpenClaw Gateway หนึ่งรายการ ไม่ได้ใช้แทน GitHub Issues, Linear, Jira หรือ
ระบบจัดการโครงการของทีมอื่นๆ

## เปิดใช้งาน

Workboard รวมมาให้แล้วแต่ปิดใช้งานโดยค่าเริ่มต้น:

1. เปิด **Plugins** ใน UI ควบคุม หรือใช้ `/settings/plugins` ซึ่งสัมพันธ์กับ
   พาธฐานของ UI ควบคุมที่กำหนดค่าไว้ ตัวอย่างเช่น พาธฐาน `/openclaw`
   จะใช้ `/openclaw/settings/plugins`
2. ค้นหา **Workboard** แล้วเลือก **Enable** เนื่องจาก Workboard รวมอยู่ใน
   OpenClaw แล้ว จึงไม่ต้องดำเนินการ **Install**
3. หาก UI รายงานว่าต้องรีสตาร์ต ให้รีสตาร์ต Gateway

แท็บ Workboard จะปรากฏในการนำทางของแดชบอร์ดหลังจากรันไทม์ของ Plugin โหลดแล้ว
ขณะที่ปิดใช้งาน แท็บนี้จะถูกซ่อนจากการนำทาง การเปิดเส้นทาง
`/workboard` โดยตรงขณะที่ Plugin ปิดใช้งานหรือถูกบล็อกโดย
`plugins.allow`/`plugins.deny` จะแสดงสถานะ Plugin ไม่พร้อมใช้งานแทนข้อมูล
การ์ด

เวิร์กโฟลว์ CLI ที่เทียบเท่ากันคือ:

```bash
openclaw plugins enable workboard
openclaw gateway restart
openclaw dashboard
```

## การกำหนดค่า

Workboard ไม่มีการกำหนดค่าเฉพาะ Plugin เปิดหรือปิดใช้งานด้วยรายการ
Plugin มาตรฐาน:

```json5
{
  plugins: {
    entries: {
      workboard: {
        enabled: true,
        config: {},
      },
    },
  },
}
```

```bash
openclaw plugins disable workboard
openclaw gateway restart
```

## ฟิลด์ของการ์ด

| ฟิลด์       | ค่า                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------- |
| `status`    | `triage`, `backlog`, `todo`, `scheduled`, `ready`, `running`, `review`, `blocked`, `done`                     |
| `priority`  | `low`, `normal`, `high`, `urgent`                                                                             |
| `labels`    | สตริงรูปแบบอิสระ                                                                                             |
| `agentId`   | เอเจนต์ที่มอบหมาย ซึ่งระบุหรือไม่ก็ได้                                                                                       |
| การอ้างอิงที่ลิงก์ไว้ | งาน การรัน เซสชัน หรือ URL ต้นทาง ซึ่งระบุหรือไม่ก็ได้                                                                    |
| `execution` | ข้อมูลเมตาที่ระบุหรือไม่ก็ได้สำหรับการรัน Codex/Claude ที่เริ่มจากการ์ด (เอนจิน โหมด โมเดล เซสชัน รหัสการรัน สถานะ) |

การ์ดยังมีข้อมูลเมตาแบบกะทัดรัดสำหรับความพยายาม ความคิดเห็น ลิงก์ หลักฐาน
อาร์ติแฟกต์ การตั้งค่าระบบอัตโนมัติ ไฟล์แนบ บันทึกของเวิร์กเกอร์ สถานะโปรโตคอล
ของเวิร์กเกอร์ การอ้างสิทธิ์ การวินิจฉัย การแจ้งเตือน รหัสเทมเพลต สถานะการเก็บถาวร และ
การตรวจหาเซสชันที่ล้าสมัย รวมถึงรายการเหตุการณ์ล่าสุด (`created`, `edited`,
`moved`, `linked`, `specified`, `decomposed`, `claimed`, `heartbeat`,
`execution_updated`, `attempt_started`, `attempt_updated`, `comment_added`,
`link_added`, `proof_added`, `artifact_added`, `attachment_added`,
`diagnostic`, `notification`, `dispatch`, `orchestration`,
`protocol_violation`, `archived`, `unarchived`, `stale`) ข้อมูลเมตานี้ช่วยให้
ผู้ปฏิบัติงานเห็นว่าการ์ดเคลื่อนผ่านบอร์ดอย่างไรโดยไม่ต้องเปิดเซสชันที่ลิงก์ไว้
ข้อมูลนี้เป็นบริบทการปฏิบัติงานภายในเครื่อง ไม่ใช่สิ่งทดแทนทรานสคริปต์ของเซสชัน
หรือประวัติปัญหาของ GitHub

Plugin และ UI ควบคุมใช้สัญญาการ์ด Workboard เดียวกัน ดังนั้นการรีเฟรชแดชบอร์ด
จึงรักษาที่มาและสิทธิ์ควบคุมของเวิร์กสเปซ สถานะการอ้างสิทธิ์ การดำเนินการ
วินิจฉัย และหมายเลขลำดับการแจ้งเตือน แทนที่จะฉายสำเนาการ์ดขนาดเล็กกว่าซึ่งใช้
เฉพาะใน UI ชนิดการวินิจฉัย ระดับความรุนแรงของการวินิจฉัย และชนิดการแจ้งเตือน
ที่ไม่รู้จักจะถูกละเว้นจนกว่าทั้งสองพื้นผิวจะรองรับ โดยจะไม่ถูกเขียนใหม่เป็น
สถานะอื่นที่ถูกต้อง

แดชบอร์ดที่เปิดอยู่จะอัปเดตจากการทำให้ `plugin.workboard.changed` ใช้ไม่ได้ แต่ละ
เหตุการณ์มีเพียงเอพ็อกและรีวิชันของที่เก็บ จากนั้น UI จะอ่านการ์ดมาตรฐานซ้ำ
ผ่าน RPC `operator.read` ตามปกติ หลายรีวิชันจะถูกรวมเป็นการอ่านติดตามผล
ครั้งเดียว Workboard จะเลื่อนการอ่านนั้นออกไประหว่างที่กำลังลาก แก้ไข หรือเขียน
การ์ด แล้วดำเนินการต่อหลังจากการโต้ตอบภายในเครื่องเสร็จสิ้น การเชื่อมต่อใหม่
จะโหลดข้อมูลมาตรฐานอีกครั้งเสมอ ไม่มีการโพลการ์ดทั้งใบเป็นประจำ และ **Refresh**
ยังคงพร้อมใช้สำหรับการกู้คืนด้วยตนเอง

เมื่อมีมากกว่าหนึ่งบอร์ด แถบเครื่องมือจะมีตัวกรอง **Board** ซึ่งใช้ข้อมูลเมตา
ของบอร์ดที่คงอยู่ แทนที่จะอ้างอิงเพียงการ์ดที่มองเห็นอยู่ในปัจจุบัน ดังนั้น
บอร์ดว่างและบอร์ดที่เก็บถาวรแล้วจึงยังคงเลือกได้ การ์ดที่ไม่มีรหัสบอร์ดอย่าง
ชัดเจนจะอยู่ในบอร์ดมาตรฐาน `default` บอร์ดที่เลือกจะถูกจัดเก็บ
ในพารามิเตอร์คำค้น `?board=` จึงสามารถบุ๊กมาร์กหรือแชร์ URL ของ
Workboard ที่กรองแล้วได้ การเลือก **All boards** จะนำพารามิเตอร์ออก

การ์ดจะถูกจัดเก็บในสถานะ Gateway ของ Plugin เอง และย้ายไปพร้อมกับสถานะ
OpenClaw ที่เหลือของ Gateway นั้น (ดู [พื้นที่จัดเก็บ](#storage))

## การเริ่มงานจากการ์ด

การ์ดที่ยังไม่ได้ลิงก์สามารถเริ่มงานได้โดยตรง:

- **Run Codex** / **Run Claude** เริ่มการรันเอเจนต์ที่ติดตามด้วยงานโดยใช้
  เอนจินที่ระบุ ส่งพรอมต์ของการ์ด และทำเครื่องหมายการ์ดเป็น `running` การรัน Codex
  ใช้ `openai/gpt-5.6-sol`; การรัน Claude ใช้ `anthropic/claude-sonnet-4-6`
- **Open Codex** / **Open Claude** สร้างเซสชันแดชบอร์ดที่ลิงก์ไว้โดยไม่
  ส่งพรอมต์ของการ์ดหรือย้ายการ์ด สำหรับงานที่ทำด้วยตนเองซึ่งยังคงเชื่อมโยง
  กับบอร์ด

การเริ่มโดยอัตโนมัติใช้เส้นทางการรันเอเจนต์ที่ติดตามด้วยงานของ Gateway
(ใช้เอเจนต์และโมเดลเริ่มต้น เว้นแต่จะเลือก Codex/Claude อย่างชัดเจน) จากนั้น
Workboard จะลิงก์งาน รหัสการรัน และคีย์เซสชันที่ได้กลับไปยังการ์ด การดำเนินการ
ที่ลิงก์แต่ละครั้งยังบันทึกสรุปความพยายาม (เอนจิน โหมด โมเดล รหัสการรัน
การประทับเวลา สถานะ จำนวนความล้มเหลวสะสม) เพื่อให้ความล้มเหลวซ้ำๆ ยังคง
มองเห็นได้

แดชบอร์ดจะรีเฟรชสถานะงานจากบัญชีแยกประเภทงานของ Gateway โดยจับคู่งานกับ
การ์ดตามรหัสงาน รหัสการรัน หรือคีย์เซสชันที่ลิงก์ไว้ งานที่อยู่ในคิวหรือกำลัง
ทำงานจะทำให้วงจรชีวิตของการ์ดยังคงทำงานอยู่ ส่วนงานที่เสร็จสิ้น ล้มเหลว
หมดเวลา หรือถูกยกเลิก จะย้ายการ์ดไปยัง `review` หรือ `blocked`
โดยใช้กฎการซิงค์เดียวกับเซสชันที่ลิงก์ไว้ (ดู [การซิงค์วงจรชีวิตของเซสชัน](#session-lifecycle-sync))

| `workboard_list`                                                                                                                                 | แสดงรายการการ์ดแบบย่อพร้อมสถานะการอ้างสิทธิ์/การวินิจฉัย และกรองตามบอร์ดได้                                                                                                                    |

| `workboard_promote` / `workboard_reassign` / `workboard_reclaim`                                                                                 | กู้คืนหรือส่งต่องานที่ติดค้าง                                                                                                                                                           |
| `workboard_comment` / `workboard_proof`                                                                                                          | เพิ่มบันทึกการส่งต่องานหรือแนบข้อมูลอ้างอิงหลักฐาน/อาร์ติแฟกต์                                                                                                                                    |
| `workboard_unblock`                                                                                                                              | ย้ายงานที่ถูกบล็อกกลับไปยัง `todo`                                                                                                                                                         |
| `workboard_move`                                                                                                                                 | ย้ายการ์ดไปยังสถานะอื่น โดยการ์ดที่ถูกอ้างสิทธิ์ต้องใช้ขอบเขตการอ้างสิทธิ์ของเอเจนต์ผู้เรียก                                                                                                      |
| `workboard_dispatch`                                                                                                                             | กระตุ้นการเลื่อนสถานะตามการขึ้นต่อกันหรือการล้างการอ้างสิทธิ์ที่ค้างโดยไม่เริ่มผู้ปฏิบัติงาน การเริ่มผู้ปฏิบัติงานใช้ Gateway หรือการจัดส่งผ่านคำสั่งสแลช                                                        |

การ์ดที่ถูกอ้างสิทธิ์จะปฏิเสธการเปลี่ยนแปลงผ่านเครื่องมือเอเจนต์จากเอเจนต์อื่น เว้นแต่ผู้เรียก
จะมีโทเค็นการอ้างสิทธิ์ที่ส่งคืนโดย `workboard_claim` การ์ดทุกใบที่ส่งคืนโดย
เครื่องมือเอเจนต์หรือการเรียก Gateway RPC จะปกปิด `metadata.claim.token` เป็น `[redacted]`
(ตัวโทเค็นจะถูกส่งคืนเพียงครั้งเดียวที่ระดับบนสุด และส่งคืนจาก `workboard_claim` เท่านั้น)
เพื่อให้ผู้ดูแลแดชบอร์ดและเอเจนต์อื่นตรวจสอบสถานะการอ้างสิทธิ์ได้โดยไม่
เห็นโทเค็นที่ใช้งานได้ การกู้คืนดำเนินการผ่าน
`workboard_promote`/`workboard_reassign`/`workboard_reclaim` ซึ่งไม่
ต้องใช้โทเค็น

## การจัดส่ง

การจัดส่งทำงานภายใน Gateway โดยไม่สร้างกระบวนการ OS ใด ๆ ตามอำเภอใจ เซสชันซับเอเจนต์ตามปกติของ
OpenClaw ยังคงเป็นเจ้าของการดำเนินงาน การจัดส่งหนึ่งรอบจะ:

1. เลื่อนสถานะการ์ดที่การขึ้นต่อกันพร้อมแล้ว
2. บันทึกเมทาดาทาการจัดส่งในการ์ดที่พร้อม
3. บล็อกการอ้างสิทธิ์ที่หมดอายุหรือการรันที่หมดเวลา
4. ทำเครื่องหมายการ์ดคัดแยกที่กำหนดค่าโดยบอร์ดให้เป็นรายการที่อาจใช้สำหรับการประสานงาน
5. อ้างสิทธิ์การ์ดที่พร้อมเป็นชุดขนาดเล็กและเริ่มการรันของผู้ปฏิบัติงานผ่าน
   รันไทม์ซับเอเจนต์ของ Gateway

ผู้ปฏิบัติงานจะได้รับบริบทของการ์ดที่จำกัดขอบเขต พร้อมโทเค็นการอ้างสิทธิ์ที่จำเป็นสำหรับส่ง Heartbeat
ทำให้การ์ดเสร็จสมบูรณ์ หรือบล็อกการ์ดผ่านเครื่องมือ Workboard

พาธพื้นที่ทำงานเป็นไปตามสิทธิ์ระบบไฟล์ที่มีอยู่ของผู้เรียก ไคลเอนต์ Gateway
ที่มี `operator.write` สามารถใช้พื้นที่ทำงานของเอเจนต์ที่กำหนดค่าไว้
ไคลเอนต์ `operator.admin` สามารถใช้เช็กเอาต์อื่นบนโฮสต์ได้ เครื่องมือเอเจนต์ในแซนด์บ็อกซ์ใช้
สิทธิ์เข้าถึงพื้นที่ทำงานของแซนด์บ็อกซ์ ส่วนเครื่องมือที่ไม่อยู่ในแซนด์บ็อกซ์และจำกัดเฉพาะพื้นที่ทำงานจะใช้
รากพื้นที่ทำงานที่กำหนดค่าไว้ Workboard จะบันทึกสิทธิ์ดังกล่าวเมื่อมีการ
กำหนดพื้นที่ทำงาน และหาจุดร่วมกับสิทธิ์ปัจจุบันของผู้เรียกอีกครั้งขณะจัดส่ง
เพื่อให้การ์ดที่บันทึกถาวรไม่สามารถขยายสิทธิ์เข้าถึงของผู้เรียกในภายหลังได้ การ์ดรุ่นเก่าที่มี
พื้นที่ทำงานบนโฮสต์ระบุไว้อย่างชัดเจนแต่ไม่มีสิทธิ์ที่บันทึกไว้ ต้องบันทึกพื้นที่ทำงานนั้น
อีกครั้งก่อนการจัดส่งแบบเข้าถึงโฮสต์เต็มรูปแบบ ส่วนการ์ดที่ไม่มีพาธบนโฮสต์จะรับ
สิทธิ์ของผู้เรียกปัจจุบันเมื่อนำไปจัดส่งครั้งแรก

การจัดส่งที่ผูกกับพื้นที่ทำงานจะยอมรับไดเรกทอรีหรือเช็กเอาต์ Git เฉพาะเมื่อ
รากของรีโพซิทอรีตรงกับพื้นที่ทำงานของเอเจนต์เป้าหมายทุกประการ คำขอ worktree
จะถูกจำกัดให้เหลือไดเรกทอรีนั้นและบันทึกถาวรเป็นพื้นที่ทำงานแบบไดเรกทอรี เพื่อให้
โฮสต์ไม่ต้องสร้างเช็กเอาต์หรือเรียกใช้โค้ดตั้งค่ารีโพซิทอรี ผู้ปฏิบัติงานเป้าหมาย
ต้องใช้แซนด์บ็อกซ์ Docker ที่เขียนได้และไม่ใช้ร่วมกันสำหรับพื้นที่ทำงานนั้นโดยเฉพาะ
โดยไม่มีการดำเนินการแบบยกระดับ การตั้งค่าแทนที่การเรียกใช้บนโฮสต์/Node ที่บันทึกถาวร หรือ
เครื่องมือ Plugin และ MCP ที่ยังไม่ได้จัดประเภท Workboard จะแจกแจงเครื่องมือที่ลงทะเบียนไว้
แทนที่จะเชื่อถือคำนำหน้า `workboard_*` และการจัดส่งจะปฏิเสธคอนเทนเนอร์ Docker
ที่กำลังทำงานซึ่งแฮชของเมานต์/การกำหนดค่าปัจจุบันล้าสมัย การจัดส่งจะรายงาน
นโยบายเป้าหมายที่เข้ากันไม่ได้ แทนที่จะเริ่มผู้ปฏิบัติงานที่มีการจำกัดขอบเขตน้อยกว่า
การจัดส่งแบบเข้าถึงโฮสต์เต็มรูปแบบสามารถกำหนดเป้าหมายเช็กเอาต์ภายในเครื่องอื่น และคงการตั้งค่า
worktree ที่มีการจัดการตามปกติไว้

สิทธิ์พื้นที่ทำงานไม่ได้สร้างโมเดลสิทธิ์วงจรชีวิตการ์ดชุดที่สอง
ผู้เรียกที่มีสิทธิ์เปลี่ยนแปลงการ์ด Workboard สามารถย้ายการ์ดด้วยตนเองผ่าน
สถานะเดียวกันบนทุกพื้นผิว ส่วนสิทธิ์พื้นที่ทำงานแบบอ่านอย่างเดียวจะป้องกันเฉพาะ
การจัดส่งผู้ปฏิบัติงานที่ต้องใช้สิทธิ์เขียน

### การเลือกผู้ปฏิบัติงาน

แต่ละรอบจะเริ่ม **ผู้ปฏิบัติงานได้สูงสุด 3 รายโดยค่าเริ่มต้น** การ์ดที่พร้อมจะเรียงตาม
ลำดับความสำคัญ จากนั้นตามตำแหน่ง แล้วตามเวลาที่สร้าง แต่ละรอบจะเริ่มเพียงหนึ่งการ์ดต่อ
เจ้าของ/เอเจนต์ และข้ามเจ้าของที่มีงานกำลังทำหรืออยู่ระหว่างตรวจสอบบน
บอร์ดอยู่แล้ว การ์ดที่เก็บถาวร การ์ดที่มีการอ้างสิทธิ์ที่ใช้งานอยู่ และการ์ดที่ไม่ได้อยู่ในสถานะ `ready`
จะไม่ถูกเลือกให้เริ่มผู้ปฏิบัติงานเด็ดขาด (แต่ยังอาจได้รับผลจาก
ด้านข้อมูลของการจัดส่ง ได้แก่ การล้างการอ้างสิทธิ์ที่ค้าง การเลื่อนสถานะตามการขึ้นต่อกัน และการล้างรายการที่
หมดเวลา)

คีย์เซสชันเป็นแบบกำหนดได้ตายตัวต่อบอร์ด/การ์ด ดังนั้นการจัดส่งซ้ำจะส่ง
กลับไปยังเลนผู้ปฏิบัติงานเดิม แทนที่จะสร้างเซสชันที่ไม่เกี่ยวข้อง:

- การ์ดที่ได้รับมอบหมาย: `agent:<agentId>:subagent:workboard-<boardId>-<cardId>`
- การ์ดที่ยังไม่ได้รับมอบหมาย: `subagent:workboard-<boardId>-<cardId>` (Gateway จะกำหนด
  เอเจนต์เริ่มต้นที่กำหนดค่าไว้)

หากไม่สามารถเริ่มผู้ปฏิบัติงานได้หลังจากการ์ดถูกอ้างสิทธิ์ Workboard จะบล็อก
การ์ด ล้างการอ้างสิทธิ์ บันทึกความล้มเหลวในการเริ่มรัน และเพิ่มบรรทัด
ล็อกของผู้ปฏิบัติงาน ซึ่งมองเห็นได้ในแดชบอร์ด, CLI JSON, เครื่องมือเอเจนต์ และ
การวินิจฉัยการ์ด

### จุดเริ่มต้น

- การดำเนินการส่งงานจากแดชบอร์ด
- `openclaw workboard dispatch`
- `/workboard dispatch` บนช่องทางที่รองรับคำสั่ง

ทั้งสามวิธีใช้รันไทม์เอเจนต์ย่อยของ Gateway เมื่อ Gateway พร้อมใช้งาน
CLI มีทางเลือกสำรองสำหรับผู้ปฏิบัติงานหนึ่งทาง: หากการเรียก Gateway ล้มเหลวด้วย
ข้อผิดพลาดการเชื่อมต่อ/ไม่พร้อมใช้งาน (หรือข้อผิดพลาด `unknown method` สำหรับ
Gateway รุ่นเก่า) และไม่มีเป้าหมาย `--url`/`--token` ที่ระบุไว้อย่างชัดเจน รวมทั้งไม่มี Gateway
ระยะไกลที่กำหนดค่าไว้ (`OPENCLAW_GATEWAY_URL` หรือ `gateway.mode: remote`) ที่ใช้ได้ CLI จะเรียกใช้
การส่งงานแบบข้อมูลเท่านั้นกับสถานะ SQLite ภายในเครื่อง โดยสามารถเลื่อนระดับการขึ้นต่อกัน
ล้างการอ้างสิทธิ์ที่ล้าสมัย และบล็อกการทำงานที่หมดเวลาได้ แต่ไม่สามารถเริ่มเวิร์กเกอร์ได้ ความล้มเหลวด้านการยืนยันตัวตน
สิทธิ์ และการตรวจสอบความถูกต้องจาก Gateway ที่เข้าถึงได้จะไม่ถือว่า
ไม่พร้อมใช้งาน แต่จะแสดงเป็นข้อผิดพลาดของคำสั่ง เช่นเดียวกับความล้มเหลวใดๆ ของ Gateway
เมื่อมีการระบุเป้าหมาย `--url`/`--token` ไว้อย่างชัดเจน

ข้อมูลเมตาของบอร์ดสามารถตั้งค่า `autoDecompose`, `autoDecomposePerDispatch`,
`defaultAssignee` และ `orchestratorProfile` ได้ OpenClaw จะบันทึกเจตนานี้และ
แสดงในบริบทของเวิร์กเกอร์ ส่วนการจัดทำข้อกำหนด/การแยกงานจริงยังคงทำงาน
ผ่านเครื่องมือ Workboard ตามปกติ

## CLI และคำสั่งแบบสแลช

```bash
openclaw workboard list [--board <id>] [--status <status>] [--include-archived] [--json]
openclaw workboard create "แก้ไขวงจรชีวิตของการ์ดที่ล้าสมัย" --priority high --labels bug,workboard
openclaw workboard show <card-id> [--json]
openclaw workboard move <card-id> --status <status> [--json]
openclaw workboard dispatch [--board <id>] [--json]
```

เอาต์พุตข้อความ `list` จะซ่อนการ์ดที่เก็บถาวรไว้โดยค่าเริ่มต้น (`--include-archived`
ใช้ลบล้างได้) ส่วน `--json` จะรวมการ์ดที่เก็บถาวรไว้เสมอ ซึ่งตรงกับสัญญาการ์ดฉบับเต็ม
ที่สคริปต์เดิมใช้งาน `show` และ `move` ยอมรับคำนำหน้ารหัส
ที่ระบุได้อย่างชัดเจน `list`, `create`, `show` และ `move` จะอ่าน/เขียนสถานะ Plugin
ภายในเครื่องโดยตรงเสมอ มีเพียง `dispatch` ที่เรียก Gateway ซึ่งกำลังทำงาน โดยใช้ทางเลือกสำรอง
ที่อธิบายไว้ข้างต้น

ดูแฟล็กทั้งหมด เอาต์พุต JSON พฤติกรรมทางเลือกสำรองของ Gateway
การจัดการคำนำหน้ารหัส กฎการเลือกการส่งงาน และ
การแก้ไขปัญหาได้ที่ [CLI ของ Workboard](/th/cli/workboard)

`/workboard list`, `/workboard show <card-id>`, `/workboard create <title>`,
`/workboard move <card-id> --status <status>` และ `/workboard dispatch` ทำงานเหมือนกับ
CLI การแสดงรายการและการแสดงรายละเอียดเป็นการดำเนินการอ่านสำหรับผู้ส่งคำสั่งที่ได้รับอนุญาตทุกคน
การสร้าง ย้าย และส่งงานต้องมีสถานะเจ้าของบนพื้นผิวแชต หรือเป็นไคลเอนต์ Gateway
ที่มี `operator.write`/`operator.admin` การย้ายด้วยตนเองโดยผู้ปฏิบัติงานใช้
พฤติกรรมลบล้างการอ้างสิทธิ์แบบเดียวกับการลากแล้ววางบนแดชบอร์ด การเข้าถึงเวิร์กทรี
ยังคงเป็นไปตามขอบเขตพื้นที่ทำงานเดียวกันที่อธิบายไว้ข้างต้น

## การซิงค์วงจรชีวิตของเซสชัน

การ์ดสามารถเชื่อมโยงกับเซสชันแดชบอร์ดที่มีอยู่ หรือเซสชันที่สร้างขึ้นเมื่อ
เริ่มงานจากการ์ด การ์ดที่เชื่อมโยงจะแสดงวงจรชีวิตของเซสชันในบรรทัด:
กำลังทำงาน ล้าสมัย เชื่อมโยงแต่ไม่ได้ใช้งาน เสร็จสิ้น ล้มเหลว หรือหายไป นอกจากนี้ยังสามารถบันทึก
เซสชันที่มีอยู่จากแท็บ Sessions ด้วย **Add to Workboard** ได้ โดยการ์ด
จะเชื่อมโยงกับเซสชันนั้น ใช้ป้ายกำกับเซสชันหรือพรอมต์ล่าสุดของผู้ใช้เป็นชื่อเรื่อง
และเติมหมายเหตุเริ่มต้นจากพรอมต์ล่าสุดของผู้ใช้ร่วมกับคำตอบล่าสุดของผู้ช่วย
เมื่อมีข้อมูลดังกล่าว

หากเซสชันที่เชื่อมโยงหายไป การ์ดจะยังคงเชื่อมโยงไว้เพื่อรักษาบริบท และ
ยังมีตัวควบคุมเริ่มงานเพื่อเริ่มใหม่ในเซสชันใหม่ หากเซสชันที่เชื่อมโยงซึ่งทำงานอยู่
หยุดรายงานกิจกรรมล่าสุด Workboard จะทำเครื่องหมายการ์ดเป็น
`stale` และจัดเก็บสถานะนั้นเป็นข้อมูลเมตาจนกว่าวงจรชีวิตจะล้างออก

ขณะที่การ์ดอยู่ในสถานะงานที่ทำงานอยู่ Workboard จะติดตามเซสชันที่เชื่อมโยง:

| สถานะเซสชันที่เชื่อมโยง                  | สถานะการ์ด |
| ------------------------------------- | ----------- |
| ทำงานอยู่                                | `running`   |
| เสร็จสมบูรณ์                             | `review`    |
| ล้มเหลว ถูกยุติ หมดเวลา หรือถูกยกเลิก | `blocked`   |

**สถานะการตรวจสอบด้วยตนเองมีสิทธิ์เหนือกว่า** การย้ายการ์ดไปยัง `review`, `blocked` หรือ `done`
จะหยุดการซิงค์อัตโนมัติสำหรับการ์ดนั้นจนกว่าจะย้ายกลับไปยัง `todo` หรือ `running`

การเริ่มการ์ดใช้เซสชัน Gateway ตามปกติ Workboard จัดเก็บเพียงข้อมูลเมตา
และลิงก์ของการ์ดเท่านั้น บันทึกบทสนทนา การเลือกโมเดล และวงจรชีวิต
ของการทำงานยังคงอยู่ภายใต้ระบบเซสชันปกติ ใช้ **Stop** บนการ์ดที่เชื่อมโยงและกำลังทำงาน
เพื่อยกเลิกการทำงานที่กำลังใช้งานอยู่ โดย Workboard จะทำเครื่องหมายการ์ดนั้นเป็น `blocked` เพื่อให้
ยังคงมองเห็นสำหรับการติดตามผล

การ์ดใหม่สามารถเริ่มจากเทมเพลต Workboard (`bugfix`, `docs`, `release`,
`pr_review`, `plugin`) เทมเพลตจะกรอกชื่อเรื่อง หมายเหตุ ป้ายกำกับ และลำดับความสำคัญไว้ล่วงหน้า
รหัสเทมเพลตจะถูกจัดเก็บเป็นข้อมูลเมตาของการ์ด

## เวิร์กโฟลว์ของแดชบอร์ด

1. เปิดแท็บ Workboard ใน Control UI
2. สร้างการ์ดพร้อมชื่อเรื่อง หมายเหตุ ลำดับความสำคัญ ป้ายกำกับ เอเจนต์ที่ระบุหรือไม่ก็ได้ และ
   เซสชันที่เชื่อมโยงซึ่งระบุหรือไม่ก็ได้ หรือเปิด Sessions แล้วเลือก **Add to Workboard**
   สำหรับเซสชันที่มีอยู่
3. ลากการ์ดระหว่างคอลัมน์ หรือโฟกัสตัวควบคุมสถานะแบบกะทัดรัดแล้วใช้
   เมนูหรือ ArrowLeft/ArrowRight ระหว่างการลาก การ์ดต้นทางจะจางลงและ
   คอลัมน์ที่วางได้จะแสดงเส้นขอบ
4. เริ่มงานจากการ์ดเพื่อสร้างหรือใช้เซสชันแดชบอร์ดซ้ำ
5. เปิดเซสชันที่เชื่อมโยงจากการ์ดขณะที่เอเจนต์กำลังทำงาน
6. ปล่อยให้การซิงค์วงจรชีวิตย้ายงานที่กำลังทำงานไปยัง `review`/`blocked` แล้วจึงย้าย
   การ์ดไปยัง `done` ด้วยตนเองเมื่อยอมรับแล้ว

## การวินิจฉัย

การวินิจฉัยคำนวณจากข้อมูลเมตาของการ์ดภายในเครื่อง การตรวจสอบในตัวจะแจ้งเตือน:

| ชนิด                        | เงื่อนไข                                                                      |
| --------------------------- | ------------------------------------------------------------------------------ |
| `stranded_ready`            | การ์ด `todo`/`backlog`/`ready` ที่มอบหมายแล้วไม่มีการอัปเดตนานกว่า 1 ชั่วโมง             |
| `running_without_heartbeat` | การ์ด `running` ที่ไม่มี Heartbeat ของการอ้างสิทธิ์หรือการอัปเดตการดำเนินการนานกว่า 20 นาที |
| `blocked_too_long`          | การ์ด `blocked` ไม่มีการอัปเดตนานกว่า 24 ชั่วโมง                                   |
| `repeated_failures`         | จำนวนความล้มเหลวที่ติดตามของการ์ดมีค่า 2 ครั้งขึ้นไป                                |
| `missing_proof`             | การ์ด `done` ที่ไม่มีหลักฐาน อาร์ติแฟกต์ หรือไฟล์แนบ                          |
| `orphaned_session`          | การ์ด `running` ที่มี `sessionKey` แต่ไม่มีข้อมูลเมตา `execution`                |

## สิทธิ์

เมธอด RPC ของ Gateway อยู่ภายใต้ `workboard.*`:

| ขอบเขต            | เมธอด                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `operator.read`  | `cards.list`, `cards.export`, `cards.diagnostics`, แสดงรายการ/รับไฟล์แนบ, อ่านเหตุการณ์การแจ้งเตือน, `boards.list`, `cards.stats`, `cards.runs`                                                                                                                                                                                                                                       |
| `operator.write` | `cards.diagnostics.refresh`, สร้าง/อัปเดต/ย้าย/ลบ/แสดงความคิดเห็น/เชื่อมโยง/linkDependency/หลักฐาน/อาร์ติแฟกต์, เพิ่ม/ลบไฟล์แนบ, บันทึกเวิร์กเกอร์, การละเมิดโปรโตคอล, อ้างสิทธิ์/Heartbeat/ปล่อย/เลื่อนระดับ/มอบหมายใหม่/เรียกคืน/เสร็จสิ้น/บล็อก/เลิกบล็อก, `cards.dispatch`, `cards.bulk`, เก็บถาวร, `boards.upsert`/`archive`/`delete`, `cards.specify`/`decompose`, สมัครรับ/ลบ/เลื่อนการแจ้งเตือน |

ไม่มีเมธอด RPC ใดต้องใช้ `operator.admin` เบราว์เซอร์ที่เชื่อมต่อด้วยสิทธิ์
ผู้ปฏิบัติงานแบบอ่านอย่างเดียวสามารถตรวจสอบบอร์ดได้ แต่ไม่สามารถแก้ไขการ์ด ขอบเขตผู้ดูแลระบบ
จะขยายพาธโฮสต์ Workboard ที่ยอมรับได้ แต่ไม่เปลี่ยนเมธอดที่พร้อมใช้งาน

## พื้นที่จัดเก็บ

Workboard จัดเก็บข้อมูลถาวรในฐานข้อมูล SQLite เชิงสัมพันธ์ที่ Plugin เป็นเจ้าของ
ภายใต้ไดเรกทอรีสถานะของ OpenClaw โดยบอร์ด การ์ด ป้ายกำกับ เหตุการณ์วงจรชีวิต
ความพยายามเรียกใช้ ความคิดเห็น ลิงก์การขึ้นต่อกัน หลักฐาน การอ้างอิงอาร์ติแฟกต์
ข้อมูลเมตาและบล็อบของไฟล์แนบ การวินิจฉัย การแจ้งเตือน บันทึกเวิร์กเกอร์
สถานะโปรโตคอล และการสมัครรับข้อมูลทั้งหมดจะอยู่ในตาราง Workboard (ไม่ใช่
รายการคีย์-ค่าของ Plugin) การส่งออกการ์ดจะรักษาลำดับเรื่องราวของบอร์ด
โดยไม่ฝังเนื้อหาบล็อบของไฟล์แนบไว้ภายใน

การติดตั้งที่ใช้ Workboard ในรุ่น `.28` สามารถเรียกใช้
`openclaw doctor --fix` เพื่อย้ายเนมสเปซสถานะ Plugin แบบเดิมที่เผยแพร่แล้ว
(`workboard.cards`, `workboard.boards`, `workboard.notify` และหากมี
`workboard.attachments`) ไปยังฐานข้อมูลเชิงสัมพันธ์

## การแก้ไขปัญหา

**แท็บแจ้งว่า Workboard ไม่พร้อมใช้งาน**

```bash
openclaw plugins inspect workboard --runtime --json
```

หากกำหนดค่า `plugins.allow` ไว้ ให้เพิ่ม `workboard` ลงไป หาก `plugins.deny`
มี `workboard` ให้ลบออกก่อนเปิดใช้งาน Plugin

**การ์ดไม่บันทึก**

ยืนยันว่าการเชื่อมต่อของเบราว์เซอร์มีสิทธิ์เข้าถึง `operator.write` เซสชันผู้ปฏิบัติงาน
แบบอ่านอย่างเดียวสามารถแสดงรายการการ์ดได้ แต่ไม่สามารถสร้าง แก้ไข ย้าย หรือลบได้

**การเริ่มการ์ดไม่เปิดเซสชันที่คาดไว้**

ตรวจสอบรหัสเอเจนต์และเซสชันที่เชื่อมโยงของการ์ด จากนั้นเปิด Sessions หรือ Chat เพื่อ
ตรวจสอบสถานะการทำงานจริง

**การส่งงานไม่เริ่มเวิร์กเกอร์**

ยืนยันว่ามีการ์ด `ready` อย่างน้อยหนึ่งใบที่ไม่มีการอ้างสิทธิ์ซึ่งทำงานอยู่:

```bash
openclaw workboard list --status ready
```

หาก CLI รายงานการส่งงานแบบข้อมูลเท่านั้น ให้เริ่มหรือรีสตาร์ต Gateway แล้ว
ลองใหม่ การส่งงานแบบข้อมูลเท่านั้นจะอัปเดตสถานะบอร์ดภายในเครื่อง แต่ไม่สามารถเริ่ม
การทำงานของเวิร์กเกอร์เอเจนต์ย่อยได้ นอกจากนี้ การ์ดอาจถูกข้ามเมื่อการ์ดอีกใบของ
เจ้าของหรือเอเจนต์เดียวกันกำลังทำงานหรือรอการตรวจสอบอยู่ ให้ดำเนินงานที่ทำงานอยู่นั้น
ให้เสร็จ บล็อก หรือปล่อยก่อนส่งงานเพิ่มเติมสำหรับเจ้าของ
รายเดียวกัน
