## โมเดลความสามารถสาธารณะ

ความสามารถคือโมเดล **Plugin แบบเนทีฟ** สาธารณะภายใน OpenClaw Plugin แบบเนทีฟของ OpenClaw ทุกตัวจะลงทะเบียนกับประเภทความสามารถอย่างน้อยหนึ่งประเภท:

| ความสามารถ             | เมธอดการลงทะเบียน                              | ตัวอย่าง Plugin                |
| ---------------------- | ------------------------------------------------ | ------------------------------ |
| การอนุมานข้อความ         | `api.registerProvider(...)`                      | `anthropic`, `openai`          |
| แบ็กเอนด์การอนุมานของ CLI | `api.registerCliBackend(...)`                    | `anthropic`, `openai`          |
| เอ็มเบดดิง              | `api.registerEmbeddingProvider(...)`             | Plugin เวกเตอร์ที่ผู้ให้บริการเป็นเจ้าของ  |
| เสียงพูด                 | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`      |
| การถอดเสียงแบบเรียลไทม์ | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                       |
| เสียงแบบเรียลไทม์        | `api.registerRealtimeVoiceProvider(...)`         | `google`, `openai`             |
| การทำความเข้าใจสื่อ      | `api.registerMediaUnderstandingProvider(...)`    | `google`, `openai`             |
| แหล่งที่มาของบทถอดเสียง | `api.registerTranscriptSourceProvider(...)`      | `discord`                      |
| การสร้างรูปภาพ           | `api.registerImageGenerationProvider(...)`       | `fal`, `google`, `openai`      |
| การสร้างเพลง             | `api.registerMusicGenerationProvider(...)`       | `fal`, `google`, `minimax`     |
| การสร้างวิดีโอ           | `api.registerVideoGenerationProvider(...)`       | `fal`, `google`, `qwen`        |
| การดึงข้อมูลเว็บ          | `api.registerWebFetchProvider(...)`              | `firecrawl`                    |
| การค้นหาเว็บ             | `api.registerWebSearchProvider(...)`             | `brave`, `firecrawl`, `google` |
| ช่องทาง / การรับส่งข้อความ | `api.registerChannel(...)`                       | `matrix`, `msteams`            |
| การค้นหา Gateway         | `api.registerGatewayDiscoveryService(...)`       | `bonjour`                      |

<Note>
Plugin ที่ลงทะเบียนความสามารถเป็นศูนย์ แต่มีฮุก เครื่องมือ บริการค้นหา หรือบริการเบื้องหลัง คือ Plugin **แบบเดิมที่มีเฉพาะฮุก** รูปแบบนี้ยังคงได้รับการรองรับอย่างสมบูรณ์
</Note>

### จุดยืนด้านความเข้ากันได้ภายนอก

โมเดลความสามารถได้รวมอยู่ในแกนหลักแล้วและปัจจุบันใช้โดย Plugin แบบรวมชุด/เนทีฟ แต่ความเข้ากันได้ของ Plugin ภายนอกยังต้องใช้เกณฑ์ที่เข้มงวดกว่า "ส่งออกแล้วจึงถือว่าคงที่"

| สถานการณ์ของ Plugin                                  | แนวทาง                                                                                         |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Plugin ภายนอกที่มีอยู่เดิม                         | รักษาการทำงานของการผสานรวมที่ใช้ฮุก นี่คือเกณฑ์พื้นฐานด้านความเข้ากันได้                        |
| Plugin แบบรวมชุด/เนทีฟใหม่                        | ควรใช้การลงทะเบียนความสามารถอย่างชัดเจนแทนการเข้าถึงภายในเฉพาะผู้จำหน่ายหรือการออกแบบใหม่ที่มีเฉพาะฮุก |
| Plugin ภายนอกที่นำการลงทะเบียนความสามารถมาใช้ | อนุญาต แต่ให้ถือว่าพื้นผิวตัวช่วยเฉพาะความสามารถยังคงเปลี่ยนแปลงได้ เว้นแต่เอกสารระบุว่าเสถียร |

การลงทะเบียนความสามารถคือทิศทางที่มุ่งหมายไว้ ฮุกแบบเดิมยังคงเป็นเส้นทางที่ปลอดภัยที่สุดในการหลีกเลี่ยงความเสียหายสำหรับ Plugin ภายนอกระหว่างช่วงเปลี่ยนผ่าน พาธย่อยของตัวช่วยที่ส่งออกไม่ได้มีสถานะเท่าเทียมกันทั้งหมด — ควรเลือกใช้สัญญาที่แคบและมีเอกสารกำกับแทนการส่งออกตัวช่วยที่เกิดขึ้นโดยบังเอิญ

### รูปแบบ Plugin

OpenClaw จำแนก Plugin ที่โหลดทุกตัวเป็นรูปแบบตามพฤติกรรมการลงทะเบียนจริง (ไม่ใช่เพียงเมทาดาทาแบบคงที่):

<AccordionGroup>
  <Accordion title="plain-capability">
    ลงทะเบียนประเภทความสามารถเพียงหนึ่งประเภท (ตัวอย่างเช่น Plugin ที่มีเฉพาะผู้ให้บริการอย่าง `arcee` หรือ `chutes`)
  </Accordion>
  <Accordion title="hybrid-capability">
    ลงทะเบียนความสามารถหลายประเภท (ตัวอย่างเช่น `openai` เป็นเจ้าของการอนุมานข้อความ เสียงพูด การทำความเข้าใจสื่อ และการสร้างรูปภาพ)
  </Accordion>
  <Accordion title="hook-only">
    ลงทะเบียนเฉพาะฮุก (แบบมีชนิดหรือกำหนดเอง) โดยไม่มีความสามารถ เครื่องมือ คำสั่ง หรือบริการ
  </Accordion>
  <Accordion title="non-capability">
    ลงทะเบียนเครื่องมือ คำสั่ง บริการ หรือเส้นทาง แต่ไม่มีความสามารถ
  </Accordion>
</AccordionGroup>

ใช้ `openclaw plugins inspect <id>` เพื่อดูรูปแบบและรายละเอียดความสามารถของ Plugin ดูรายละเอียดที่[เอกสารอ้างอิง CLI](/th/cli/plugins#inspect)

### ฮุกแบบเดิม

ฮุก `before_agent_start` ยังคงได้รับการรองรับในฐานะเส้นทางความเข้ากันได้สำหรับ Plugin ที่มีเฉพาะฮุก Plugin แบบเดิมที่ใช้งานจริงยังคงพึ่งพาฮุกนี้

ทิศทาง:

- รักษาให้ใช้งานได้
- ระบุในเอกสารว่าเป็นแบบเดิม
- ควรใช้ `before_model_resolve` สำหรับงานแทนที่โมเดล/ผู้ให้บริการ
- ควรใช้ `before_prompt_build` สำหรับงานแก้ไขพรอมต์
- นำออกต่อเมื่อการใช้งานจริงลดลงและความครอบคลุมของฟิกซ์เจอร์พิสูจน์ว่าการย้ายมีความปลอดภัยแล้วเท่านั้น

### สัญญาณความเข้ากันได้

`openclaw doctor`, `openclaw plugins inspect <id>`, `openclaw status --all` และ `openclaw plugins doctor` แสดงประกาศด้านความเข้ากันได้เหล่านี้:

| สัญญาณ                                     | ความหมาย                                                                                                       |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------- |
| **การกำหนดค่าถูกต้อง**                           | แยกวิเคราะห์การกำหนดค่าได้ตามปกติและสามารถแก้ไขตำแหน่ง Plugin ได้                                                                        |
| **เฉพาะฮุก** (ข้อมูล)                       | Plugin ลงทะเบียนเฉพาะฮุก เป็นเส้นทางที่รองรับ แต่ยังไม่ได้ย้ายไปใช้การลงทะเบียนความสามารถ                |
| **`before_agent_start` แบบเดิม** (คำเตือน)     | Plugin ใช้ฮุก `before_agent_start` ที่เลิกใช้แล้วแทน `before_model_resolve`/`before_prompt_build`  |
| **API เอ็มเบดดิงหน่วยความจำที่เลิกใช้แล้ว** (คำเตือน) | Plugin ที่ไม่ได้รวมชุดใช้ API ผู้ให้บริการเอ็มเบดดิงเฉพาะหน่วยความจำแบบเก่าแทน `registerEmbeddingProvider` |
| **ข้อผิดพลาดร้ายแรง**                             | การกำหนดค่าไม่ถูกต้องหรือโหลด Plugin ไม่สำเร็จ                                                                    |

ปัจจุบันสัญญาณแนะนำ/คำเตือนเหล่านี้ไม่ทำให้ Plugin ของคุณเสียหาย สัญญาณเหล่านี้ยังปรากฏใน `openclaw status --all` และ `openclaw plugins doctor` ด้วย

## ภาพรวมสถาปัตยกรรม

ระบบ Plugin ของ OpenClaw มีสี่ชั้น:

<Steps>
  <Step title="Manifest + การค้นหา">
    OpenClaw ค้นหา Plugin ที่อาจใช้งานได้จากพาธที่กำหนดค่าไว้ รากเวิร์กสเปซ ราก Plugin ส่วนกลาง และ Plugin แบบรวมชุด การค้นหาจะอ่าน manifest แบบเนทีฟ `openclaw.plugin.json` และ manifest ของชุดรวมที่รองรับก่อน
  </Step>
  <Step title="การเปิดใช้งาน + การตรวจสอบความถูกต้อง">
    แกนหลักตัดสินว่า Plugin ที่ค้นพบควรเปิดใช้งาน ปิดใช้งาน บล็อก หรือเลือกสำหรับสล็อตเฉพาะ เช่น หน่วยความจำ
  </Step>
  <Step title="การโหลดขณะทำงาน">
    Plugin แบบเนทีฟของ OpenClaw ถูกโหลดภายในโพรเซสและลงทะเบียนความสามารถลงในรีจิสทรีส่วนกลาง JavaScript ที่จัดแพ็กเกจจะโหลดผ่าน `require` แบบเนทีฟ ส่วน TypeScript ซอร์สภายในเครื่องจากบุคคลที่สามใช้ Jiti เป็นทางเลือกฉุกเฉิน ชุดรวมที่เข้ากันได้จะถูกปรับให้อยู่ในรูปแบบเรกคอร์ดรีจิสทรีโดยไม่นำเข้าโค้ดขณะทำงาน
  </Step>
  <Step title="การใช้งานพื้นผิว">
    ส่วนอื่นของ OpenClaw อ่านรีจิสทรีเพื่อเปิดให้ใช้งานเครื่องมือ ช่องทาง การตั้งค่าผู้ให้บริการ ฮุก เส้นทาง HTTP คำสั่ง CLI และบริการ
  </Step>
</Steps>

สำหรับ CLI ของ Plugin โดยเฉพาะ การค้นหาคำสั่งรากแบ่งออกเป็นสองระยะ:

- เมทาดาทาขณะแยกวิเคราะห์มาจาก `registerCli(..., { descriptors: [...] })`
- โมดูล CLI จริงของ Plugin สามารถคงการโหลดแบบหน่วงเวลาและลงทะเบียนเมื่อถูกเรียกใช้ครั้งแรก

วิธีนี้ทำให้โค้ด CLI ที่ Plugin เป็นเจ้าของยังคงอยู่ภายใน Plugin พร้อมกับช่วยให้ OpenClaw จองชื่อคำสั่งรากก่อนการแยกวิเคราะห์ได้

ขอบเขตการออกแบบที่สำคัญ:

- การตรวจสอบ manifest/การกำหนดค่าควรทำงานได้จาก **เมทาดาทา manifest/สคีมา** โดยไม่เรียกใช้โค้ด Plugin
- การค้นหาความสามารถแบบเนทีฟอาจโหลดโค้ดจุดเข้าของ Plugin ที่เชื่อถือได้เพื่อสร้างสแนปช็อตรีจิสทรีที่ไม่เปิดใช้งาน
- พฤติกรรมขณะทำงานแบบเนทีฟมาจากพาธ `register(api)` ของโมดูล Plugin โดยมี `api.registrationMode === "full"`

การแบ่งส่วนนี้ช่วยให้ OpenClaw ตรวจสอบการกำหนดค่า อธิบาย Plugin ที่หายไป/ถูกปิดใช้งาน และสร้างคำแนะนำ UI/สคีมาก่อนที่รันไทม์แบบเต็มจะทำงาน

### สแนปช็อตเมทาดาทา Plugin และตารางค้นหา

เมื่อ Gateway เริ่มทำงาน ระบบจะสร้าง `PluginMetadataSnapshot` หนึ่งรายการสำหรับสแนปช็อตการกำหนดค่าปัจจุบัน สแนปช็อตนี้มีเฉพาะเมทาดาทา โดยจัดเก็บดัชนี Plugin ที่ติดตั้ง รีจิสทรี manifest การวินิจฉัย manifest แผนผังเจ้าของ ตัวปรับ Plugin ID ให้เป็นมาตรฐาน และเรกคอร์ด manifest สแนปช็อตนี้ไม่มีโมดูล Plugin ที่โหลดแล้ว SDK ของผู้ให้บริการ เนื้อหาแพ็กเกจ หรือการส่งออกขณะทำงาน

การตรวจสอบการกำหนดค่าที่รับรู้ Plugin การเปิดใช้งานอัตโนมัติเมื่อเริ่มต้น และการบูตสแตรป Plugin ของ Gateway จะใช้สแนปช็อตนี้แทนการสร้างเมทาดาทา manifest/ดัชนีใหม่แยกจากกัน `PluginLookUpTable` สร้างมาจากสแนปช็อตเดียวกันและเพิ่มแผน Plugin เมื่อเริ่มต้นสำหรับการกำหนดค่ารันไทม์ปัจจุบัน

หลังจากเริ่มต้น Gateway จะเก็บสแนปช็อตเมทาดาทาปัจจุบันไว้เป็นผลิตผลรันไทม์ที่แทนที่ได้ การค้นหาผู้ให้บริการขณะทำงานซ้ำ ๆ สามารถยืมสแนปช็อตนี้แทนการสร้างดัชนีที่ติดตั้งและรีจิสทรี manifest ใหม่สำหรับการตรวจแค็ตตาล็อกผู้ให้บริการแต่ละครั้ง สแนปช็อตจะถูกล้างหรือแทนที่เมื่อ Gateway ปิดตัว เมื่อการกำหนดค่า/รายการ Plugin เปลี่ยนแปลง และเมื่อมีการเขียนดัชนีที่ติดตั้ง ผู้เรียกจะย้อนกลับไปใช้พาธ manifest/ดัชนีแบบเย็นเมื่อไม่มีสแนปช็อตปัจจุบันที่เข้ากันได้ การตรวจสอบความเข้ากันได้ต้องรวมรากการค้นหา Plugin เช่น `plugins.load.paths` และเวิร์กสเปซเริ่มต้นของเอเจนต์ เพราะ Plugin ในเวิร์กสเปซเป็นส่วนหนึ่งของขอบเขตเมทาดาทา

สแนปช็อตและตารางค้นหาช่วยให้การตัดสินใจซ้ำ ๆ เมื่อเริ่มต้นอยู่บนพาธที่รวดเร็ว:

- ความเป็นเจ้าของช่องทาง
- การเริ่มต้นช่องทางแบบเลื่อนเวลา
- ID ของ Plugin เมื่อเริ่มต้น
- ความเป็นเจ้าของผู้ให้บริการและแบ็กเอนด์ CLI
- ความเป็นเจ้าของผู้ให้บริการตั้งค่า นามแฝงคำสั่ง ผู้ให้บริการแค็ตตาล็อกโมเดล และสัญญา manifest
- การตรวจสอบสคีมาการกำหนดค่า Plugin และสคีมาการกำหนดค่าช่องทาง
- การตัดสินใจเปิดใช้งานอัตโนมัติเมื่อเริ่มต้น

ขอบเขตความปลอดภัยคือการแทนที่สแนปช็อต ไม่ใช่การแก้ไขสแนปช็อต ให้สร้างสแนปช็อตใหม่เมื่อการกำหนดค่า รายการ Plugin เรกคอร์ดการติดตั้ง หรือนโยบายดัชนีที่คงอยู่เปลี่ยนแปลง อย่าถือว่าสแนปช็อตเป็นรีจิสทรีส่วนกลางแบบแก้ไขได้ในวงกว้าง และอย่าเก็บสแนปช็อตย้อนหลังอย่างไม่จำกัด การโหลด Plugin ขณะทำงานยังคงแยกจากสแนปช็อตเมทาดาทา เพื่อไม่ให้สถานะรันไทม์ที่ล้าสมัยถูกซ่อนไว้หลังแคชเมทาดาทา

กฎของแคชมีระบุไว้ใน[รายละเอียดภายในสถาปัตยกรรม Plugin](/th/plugins/architecture-internals#plugin-cache-boundary): เมทาดาทา manifest และการค้นหาจะเป็นข้อมูลใหม่เสมอ เว้นแต่ผู้เรียกจะถือสแนปช็อต ตารางค้นหา หรือรีจิสทรี manifest ที่ชัดเจนสำหรับโฟลว์ปัจจุบัน แคชเมทาดาทาที่ซ่อนอยู่และ TTL ตามเวลานาฬิกาไม่ใช่ส่วนหนึ่งของการโหลด Plugin เฉพาะแคชตัวโหลดรันไทม์ โมดูล และอาร์ติแฟกต์การขึ้นต่อกันเท่านั้นที่อาจคงอยู่หลังจากโหลดโค้ดหรืออาร์ติแฟกต์ที่ติดตั้งจริงแล้ว

ผู้เรียกในพาธแบบเย็นบางรายยังคงสร้างรีจิสทรี manifest ใหม่โดยตรงจากดัชนี Plugin ที่ติดตั้งซึ่งคงอยู่ แทนที่จะรับ `PluginLookUpTable` ของ Gateway ปัจจุบันพาธนี้สร้างรีจิสทรีใหม่ตามต้องการ เมื่อผู้เรียกมีตารางค้นหาปัจจุบันหรือรีจิสทรี manifest ที่ชัดเจนอยู่แล้ว ควรส่งต่อสิ่งนั้นผ่านโฟลว์รันไทม์

### การวางแผนการเปิดใช้งาน

การวางแผนการเปิดใช้งานเป็นส่วนหนึ่งของระนาบควบคุม ผู้เรียกสามารถสอบถามได้ว่า Plugin ใดเกี่ยวข้องกับคำสั่ง ผู้ให้บริการ ช่องทาง เส้นทาง ชุดควบคุมเอเจนต์ หรือความสามารถที่ระบุ ก่อนโหลดรีจิสทรีรันไทม์ที่กว้างขึ้น

ตัววางแผนยังคงเข้ากันได้กับพฤติกรรมปัจจุบันของแมนิเฟสต์:

- ฟิลด์ `activation.*` เป็นคำแนะนำที่ระบุอย่างชัดเจนสำหรับตัววางแผน
- `providers`, `channels`, `commandAliases`, `setup.providers`, `contracts.tools` และฮุกต่าง ๆ ยังคงเป็นกลไกสำรองตามความเป็นเจ้าของในแมนิเฟสต์
- API ของตัววางแผนที่ส่งคืนเฉพาะรหัสยังคงพร้อมใช้งานสำหรับผู้เรียกเดิม
- API แผนจะรายงานป้ายกำกับเหตุผล เพื่อให้การวินิจฉัยแยกแยะคำแนะนำที่ระบุอย่างชัดเจนออกจากกลไกสำรองตามความเป็นเจ้าของได้

<Warning>
อย่าถือว่า `activation` เป็นฮุกวงจรชีวิตหรือสิ่งทดแทน `register(...)` เพราะนี่คือเมทาดาทาที่ใช้จำกัดขอบเขตการโหลด ให้ใช้ฟิลด์ความเป็นเจ้าของเป็นหลักเมื่อฟิลด์เหล่านั้นอธิบายความสัมพันธ์อยู่แล้ว และใช้ `activation` เฉพาะสำหรับคำแนะนำเพิ่มเติมแก่ตัววางแผน
</Warning>

### Plugin ช่องทางและเครื่องมือข้อความที่ใช้ร่วมกัน

Plugin ช่องทางไม่จำเป็นต้องลงทะเบียนเครื่องมือส่ง/แก้ไข/แสดงปฏิกิริยาแยกต่างหากสำหรับการดำเนินการแชตทั่วไป OpenClaw เก็บเครื่องมือ `message` ที่ใช้ร่วมกันไว้หนึ่งรายการในแกนหลัก และ Plugin ช่องทางเป็นเจ้าของการค้นหาและการดำเนินการเฉพาะช่องทางที่อยู่เบื้องหลังเครื่องมือนี้

ขอบเขตปัจจุบันคือ:

- แกนหลักเป็นเจ้าของโฮสต์เครื่องมือ `message` ที่ใช้ร่วมกัน การเชื่อมต่อพรอมต์ การจัดการเซสชัน/เธรด และการส่งต่อเพื่อดำเนินการ
- Plugin ช่องทางเป็นเจ้าของการค้นหาการดำเนินการตามขอบเขต การค้นหาความสามารถ และส่วนย่อยของสคีมาที่เฉพาะเจาะจงกับช่องทาง
- Plugin ช่องทางเป็นเจ้าของไวยากรณ์การสนทนาในเซสชันที่เฉพาะเจาะจงกับผู้ให้บริการ เช่น วิธีที่รหัสการสนทนาเข้ารหัสรหัสเธรดหรือสืบทอดจากการสนทนาหลัก
- Plugin ช่องทางดำเนินการขั้นสุดท้ายผ่านอะแดปเตอร์การดำเนินการของตน

สำหรับ Plugin ช่องทาง พื้นผิว SDK คือ `ChannelMessageActionAdapter.describeMessageTool(...)` การเรียกเพื่อค้นหาแบบรวมนี้ช่วยให้ Plugin ส่งคืนการดำเนินการที่มองเห็นได้ ความสามารถ และส่วนเพิ่มเติมของสคีมาพร้อมกัน เพื่อไม่ให้องค์ประกอบเหล่านี้คลาดเคลื่อนจากกัน

เมื่อพารามิเตอร์เครื่องมือข้อความที่เฉพาะเจาะจงกับช่องทางมีแหล่งสื่อ เช่น พาธภายในเครื่องหรือ URL สื่อระยะไกล Plugin ควรส่งคืน `mediaSourceParams` จาก `describeMessageTool(...)` ด้วย แกนหลักใช้รายการที่ระบุอย่างชัดเจนนี้เพื่อปรับพาธแซนด์บ็อกซ์ให้เป็นมาตรฐานและใช้คำแนะนำการเข้าถึงสื่อขาออก โดยไม่ฮาร์ดโค้ดชื่อพารามิเตอร์ที่ Plugin เป็นเจ้าของ ควรใช้แมปตามขอบเขตการดำเนินการในตำแหน่งนี้ ไม่ใช่รายการแบบแบนรายการเดียวสำหรับทั้งช่องทาง เพื่อไม่ให้พารามิเตอร์สื่อที่ใช้เฉพาะโปรไฟล์ถูกปรับให้เป็นมาตรฐานกับการดำเนินการที่ไม่เกี่ยวข้อง เช่น `send`

แกนหลักส่งขอบเขตรันไทม์เข้าสู่ขั้นตอนการค้นหานั้น ฟิลด์สำคัญประกอบด้วย:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` ขาเข้าที่เชื่อถือได้

สิ่งนี้สำคัญต่อ Plugin ที่ขึ้นอยู่กับบริบท ช่องทางสามารถซ่อนหรือแสดงการดำเนินการข้อความตามบัญชีที่ใช้งานอยู่ ห้อง/เธรด/ข้อความปัจจุบัน หรือตัวตนของผู้ร้องขอที่เชื่อถือได้ โดยไม่ต้องฮาร์ดโค้ดแขนงเฉพาะช่องทางไว้ในเครื่องมือ `message` ของแกนหลัก

นี่คือเหตุผลที่การเปลี่ยนแปลงการกำหนดเส้นทางของรันเนอร์แบบฝังตัวยังคงเป็นงานของ Plugin: รันเนอร์มีหน้าที่ส่งต่อตัวตนของแชต/เซสชันปัจจุบันไปยังขอบเขตการค้นหาของ Plugin เพื่อให้เครื่องมือ `message` ที่ใช้ร่วมกันแสดงพื้นผิวที่ช่องทางเป็นเจ้าของได้อย่างถูกต้องสำหรับรอบปัจจุบัน

สำหรับตัวช่วยดำเนินการที่ช่องทางเป็นเจ้าของ Plugin ที่รวมมาในชุดควรเก็บรันไทม์การดำเนินการไว้ภายในโมดูล Plugin ของตนเอง แกนหลักไม่ได้เป็นเจ้าของรันไทม์การดำเนินการข้อความของ Discord, Slack, Telegram หรือ WhatsApp ภายใต้ `src/agents/tools` อีกต่อไป เราไม่เผยแพร่พาธย่อย `plugin-sdk/*-action-runtime` แยกต่างหาก และ Plugin ที่รวมมาในชุดควรนำเข้าโค้ดรันไทม์ภายในของตนโดยตรงจากโมดูลที่ Plugin ของตนเป็นเจ้าของ

ขอบเขตเดียวกันนี้ใช้กับจุดเชื่อมต่อ SDK ที่ตั้งชื่อตามผู้ให้บริการโดยทั่วไป กล่าวคือ แกนหลักไม่ควรนำเข้าบาร์เรลอำนวยความสะดวกเฉพาะช่องทางสำหรับ Discord, Signal, Slack, WhatsApp หรือ Plugin ที่คล้ายกัน หากแกนหลักต้องการพฤติกรรมใด ให้ใช้บาร์เรล `api.ts` / `runtime-api.ts` ของ Plugin ที่รวมมาในชุดนั้นเอง หรือยกระดับความต้องการดังกล่าวให้เป็นความสามารถทั่วไปที่มีขอบเขตแคบใน SDK ที่ใช้ร่วมกัน

Plugin ที่รวมมาในชุดใช้กฎเดียวกัน `runtime-api.ts` ของ Plugin ที่รวมมาในชุดไม่ควรส่งออกซ้ำฟาซาด `openclaw/plugin-sdk/<plugin-id>` ที่มีแบรนด์ของตนเอง ฟาซาดที่มีแบรนด์เหล่านั้นยังคงเป็นชิมความเข้ากันได้สำหรับ Plugin ภายนอกและผู้ใช้รุ่นเก่า แต่ Plugin ที่รวมมาในชุดควรใช้การส่งออกภายในร่วมกับพาธย่อย SDK ทั่วไปที่มีขอบเขตแคบ เช่น `openclaw/plugin-sdk/channel-policy`, `openclaw/plugin-sdk/runtime-store` หรือ `openclaw/plugin-sdk/webhook-ingress` โค้ดใหม่ไม่ควรเพิ่มฟาซาด SDK ที่เฉพาะเจาะจงกับรหัส Plugin เว้นแต่ขอบเขตความเข้ากันได้ของระบบนิเวศภายนอกที่มีอยู่จำเป็นต้องใช้

สำหรับโพลโดยเฉพาะ มีเส้นทางดำเนินการสองเส้นทาง:

- `outbound.sendPoll` เป็นค่าพื้นฐานที่ใช้ร่วมกันสำหรับช่องทางที่สอดคล้องกับโมเดลโพลทั่วไป
- `actions.handleAction("poll")` เป็นเส้นทางที่แนะนำสำหรับความหมายของโพลที่เฉพาะเจาะจงกับช่องทางหรือพารามิเตอร์โพลเพิ่มเติม

ขณะนี้แกนหลักจะเลื่อนการแยกวิเคราะห์โพลที่ใช้ร่วมกันออกไปจนกว่าการส่งต่อโพลไปยัง Plugin จะปฏิเสธการดำเนินการ เพื่อให้ตัวจัดการโพลที่ Plugin เป็นเจ้าของสามารถรับฟิลด์โพลเฉพาะช่องทางได้โดยไม่ถูกตัวแยกวิเคราะห์โพลทั่วไปขัดขวางก่อน

ดูลำดับการเริ่มต้นทั้งหมดได้ที่ [รายละเอียดภายในสถาปัตยกรรม Plugin](/th/plugins/architecture-internals)

## โมเดลความเป็นเจ้าของความสามารถ

OpenClaw ถือว่า Plugin แบบเนทีฟเป็นขอบเขตความเป็นเจ้าของของ **บริษัท** หรือ **ฟีเจอร์** ไม่ใช่ที่รวมการผสานการทำงานที่ไม่เกี่ยวข้องกัน

ซึ่งหมายความว่า:

- โดยทั่วไป Plugin ของบริษัทควรเป็นเจ้าของพื้นผิวทั้งหมดของบริษัทนั้นที่เชื่อมกับ OpenClaw
- โดยทั่วไป Plugin ของฟีเจอร์ควรเป็นเจ้าของพื้นผิวทั้งหมดของฟีเจอร์ที่ตนนำเข้ามา
- ช่องทางควรใช้ความสามารถของแกนหลักที่ใช้ร่วมกัน แทนการนำพฤติกรรมของผู้ให้บริการมาสร้างใหม่แบบเฉพาะกิจ

    `google` เป็นเจ้าของการอนุมานข้อความ แบ็กเอนด์ CLI เอ็มเบดดิง เสียง เสียงแบบเรียลไทม์ การทำความเข้าใจสื่อ การสร้างรูปภาพ/เพลง/วิดีโอ และการค้นหาเว็บ `openai` เป็นเจ้าของการอนุมานข้อความ เอ็มเบดดิง เสียง การถอดเสียงแบบเรียลไทม์ เสียงแบบเรียลไทม์ การทำความเข้าใจสื่อ และการสร้างรูปภาพ/วิดีโอ `minimax` เป็นเจ้าของการอนุมานข้อความ รวมถึงการทำความเข้าใจสื่อ เสียง การสร้างรูปภาพ/เพลง/วิดีโอ และการค้นหาเว็บ
  </Accordion>

  </Accordion>
  <Accordion title="Plugin ฟีเจอร์">
    `voice-call` เป็นเจ้าของการขนส่งสาย เครื่องมือ CLI เส้นทาง และการเชื่อมต่อสตรีมสื่อของ Twilio แต่ใช้ความสามารถด้านเสียง การถอดเสียงแบบเรียลไทม์ และเสียงแบบเรียลไทม์ที่ใช้ร่วมกัน แทนการนำเข้า Plugin ของผู้ให้บริการโดยตรง
  </Accordion>
</AccordionGroup>

สถานะปลายทางที่ต้องการคือ:

- พื้นผิวของผู้ให้บริการที่เชื่อมกับ OpenClaw อยู่ใน Plugin เดียว แม้ว่าจะครอบคลุมโมเดลข้อความ เสียง รูปภาพ และวิดีโอ
- ผู้ให้บริการรายอื่นสามารถทำแบบเดียวกันกับพื้นที่พื้นผิวของตน
- ช่องทางไม่สนใจว่า Plugin ของผู้ให้บริการใดเป็นเจ้าของผู้ให้บริการนั้น แต่จะใช้สัญญาความสามารถที่ใช้ร่วมกันซึ่งแกนหลักเปิดเผย

นี่คือความแตกต่างสำคัญ:

- **Plugin** = ขอบเขตความเป็นเจ้าของ
- **ความสามารถ** = สัญญาของแกนหลักที่ Plugin หลายรายการสามารถนำไปใช้หรือใช้งานได้

ดังนั้น หาก OpenClaw เพิ่มโดเมนใหม่ เช่น วิดีโอ คำถามแรกไม่ใช่ "ผู้ให้บริการรายใดควรฮาร์ดโค้ดการจัดการวิดีโอ" แต่คือ "สัญญาความสามารถด้านวิดีโอของแกนหลักคืออะไร" เมื่อมีสัญญานั้นแล้ว Plugin ของผู้ให้บริการจึงลงทะเบียนกับสัญญาได้ และ Plugin ช่องทาง/ฟีเจอร์ก็สามารถใช้งานสัญญานั้นได้

หากยังไม่มีความสามารถดังกล่าว โดยทั่วไปแนวทางที่ถูกต้องคือ:

<Steps>
  <Step title="กำหนดความสามารถ">
    กำหนดความสามารถที่ขาดหายไปในแกนหลัก
  </Step>
  <Step title="เปิดเผยผ่าน SDK">
    เปิดเผยความสามารถนั้นผ่าน API/รันไทม์ของ Plugin แบบมีชนิดข้อมูล
  </Step>
  <Step title="เชื่อมต่อผู้ใช้งาน">
    เชื่อมต่อช่องทาง/ฟีเจอร์เข้ากับความสามารถนั้น
  </Step>
  <Step title="การนำไปใช้โดยผู้ให้บริการ">
    ให้ Plugin ของผู้ให้บริการลงทะเบียนการนำไปใช้
  </Step>
</Steps>

วิธีนี้ทำให้ความเป็นเจ้าของชัดเจน พร้อมหลีกเลี่ยงพฤติกรรมในแกนหลักที่ขึ้นอยู่กับผู้ให้บริการรายเดียวหรือพาธโค้ดเฉพาะ Plugin แบบใช้ครั้งเดียว

### การแบ่งชั้นความสามารถ

ใช้โมเดลแนวคิดนี้เมื่อตัดสินใจว่าโค้ดควรอยู่ที่ใด:

<Tabs>
  <Tab title="ชั้นความสามารถของแกนหลัก">
    การประสานงาน นโยบาย กลไกสำรอง กฎการผสานการกำหนดค่า ความหมายของการส่งมอบ และสัญญาแบบมีชนิดข้อมูลที่ใช้ร่วมกัน
  </Tab>
  <Tab title="ชั้น Plugin ของผู้ให้บริการ">
    API เฉพาะผู้ให้บริการ การยืนยันตัวตน แค็ตตาล็อกโมเดล การสังเคราะห์เสียง การสร้างรูปภาพ แบ็กเอนด์วิดีโอ และเอนด์พอยต์การใช้งาน
  </Tab>
  <Tab title="ชั้น Plugin ช่องทาง/ฟีเจอร์">
    การผสานการทำงานกับ Discord/Slack/voice-call/อื่น ๆ ซึ่งใช้ความสามารถของแกนหลักและนำเสนอผ่านพื้นผิว
  </Tab>
</Tabs>

ตัวอย่างเช่น TTS ใช้โครงสร้างนี้:

- แกนหลักเป็นเจ้าของนโยบาย TTS ขณะตอบกลับ ลำดับกลไกสำรอง ค่ากำหนด และการส่งมอบผ่านช่องทาง
- `elevenlabs`, `google`, `microsoft` และ `openai` เป็นเจ้าของการนำการสังเคราะห์ไปใช้
- `voice-call` ใช้ตัวช่วยรันไทม์ TTS สำหรับระบบโทรศัพท์

ควรใช้รูปแบบเดียวกันนี้เป็นหลักสำหรับความสามารถในอนาคต

### ตัวอย่าง Plugin บริษัทที่มีหลายความสามารถ

Plugin ของบริษัทควรให้ความรู้สึกเป็นหนึ่งเดียวจากภายนอก หาก OpenClaw มีสัญญาที่ใช้ร่วมกันสำหรับโมเดล เสียง การถอดเสียงแบบเรียลไทม์ เสียงแบบเรียลไทม์ การทำความเข้าใจสื่อ การสร้างรูปภาพ การสร้างวิดีโอ การดึงข้อมูลจากเว็บ และการค้นหาเว็บ ผู้ให้บริการสามารถเป็นเจ้าของพื้นผิวทั้งหมดของตนไว้ในที่เดียว:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";
import { createPluginBackedWebSearchProvider } from "openclaw/plugin-sdk/provider-web-search";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          ...req,
          provider: "exampleai",
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          ...req,
          provider: "exampleai",
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

สิ่งสำคัญไม่ใช่ชื่อของตัวช่วยที่ตรงกันทุกประการ แต่คือโครงสร้าง:

- Plugin เดียวเป็นเจ้าของพื้นผิวของผู้ให้บริการ
- แกนหลักยังคงเป็นเจ้าของสัญญาความสามารถ
- ช่องทางและ Plugin ฟีเจอร์ใช้ตัวช่วย `api.runtime.*` ไม่ใช่โค้ดของผู้ให้บริการ
- การทดสอบสัญญาสามารถตรวจสอบได้ว่า Plugin ได้ลงทะเบียนความสามารถที่อ้างว่าเป็นเจ้าของแล้ว

### ตัวอย่างความสามารถ: การทำความเข้าใจวิดีโอ

OpenClaw ถือว่าการทำความเข้าใจรูปภาพ/เสียง/วิดีโอเป็นความสามารถที่ใช้ร่วมกันหนึ่งรายการอยู่แล้ว โดยใช้โมเดลความเป็นเจ้าของเดียวกัน:

<Steps>
  <Step title="แกนหลักกำหนดสัญญา">
    แกนหลักกำหนดสัญญาการทำความเข้าใจสื่อ
  </Step>
  <Step title="Plugin ของผู้ให้บริการลงทะเบียน">
    Plugin ของผู้ให้บริการลงทะเบียน `describeImage`, `transcribeAudio` และ `describeVideo` ตามความเหมาะสม
  </Step>
  <Step title="ผู้ใช้งานใช้พฤติกรรมที่ใช้ร่วมกัน">
    ช่องทางและ Plugin ฟีเจอร์ใช้พฤติกรรมของแกนหลักที่ใช้ร่วมกัน แทนการเชื่อมต่อโดยตรงกับโค้ดของผู้ให้บริการ
  </Step>
</Steps>

วิธีนี้หลีกเลี่ยงการฝังสมมติฐานด้านวิดีโอของผู้ให้บริการรายหนึ่งไว้ในแกนหลัก Plugin เป็นเจ้าของพื้นผิวของผู้ให้บริการ ส่วนแกนหลักเป็นเจ้าของสัญญาความสามารถและพฤติกรรมกลไกสำรอง

การสร้างวิดีโอใช้ลำดับเดียวกันนี้อยู่แล้ว: แกนหลักเป็นเจ้าของสัญญาความสามารถแบบมีชนิดข้อมูลและตัวช่วยรันไทม์ ส่วน Plugin ของผู้ให้บริการลงทะเบียนการนำ `api.registerVideoGenerationProvider(...)` ไปใช้กับสัญญานั้น

ต้องการรายการตรวจสอบการทยอยใช้งานที่เป็นรูปธรรมหรือไม่ ดู [ตำราการใช้ความสามารถ](/th/plugins/adding-capabilities)

## สัญญาและการบังคับใช้

พื้นผิว API ของ Plugin ได้รับการกำหนดชนิดและรวมศูนย์ไว้ใน `OpenClawPluginApi` โดยตั้งใจ สัญญานี้กำหนดจุดลงทะเบียนที่รองรับและตัวช่วยรันไทม์ที่ Plugin สามารถใช้งานได้

เหตุผลที่เรื่องนี้สำคัญ:

- ผู้เขียน Plugin มีมาตรฐานภายในที่เสถียรเพียงหนึ่งเดียว
- แกนหลักสามารถปฏิเสธการเป็นเจ้าของที่ซ้ำกัน เช่น Plugin สองตัวลงทะเบียนรหัสผู้ให้บริการเดียวกัน
- การเริ่มต้นระบบสามารถแสดงข้อมูลวินิจฉัยที่นำไปแก้ไขได้สำหรับการลงทะเบียนที่มีรูปแบบไม่ถูกต้อง
- การทดสอบสัญญาสามารถบังคับใช้ความเป็นเจ้าของของ Plugin ที่รวมมาให้และป้องกันการคลาดเคลื่อนโดยไม่มีการแจ้งเตือน

การบังคับใช้มีสองชั้น:

<AccordionGroup>
  <Accordion title="การบังคับใช้การลงทะเบียนรันไทม์">
    รีจิสทรี Plugin จะตรวจสอบการลงทะเบียนขณะโหลด Plugin ตัวอย่างเช่น รหัสผู้ให้บริการซ้ำ รหัสผู้ให้บริการเสียงพูดซ้ำ และการลงทะเบียนที่มีรูปแบบไม่ถูกต้อง จะสร้างข้อมูลวินิจฉัย Plugin แทนพฤติกรรมที่ไม่กำหนด
  </Accordion>
  <Accordion title="การทดสอบสัญญา">
    Plugin ที่รวมมาให้จะถูกบันทึกในรีจิสทรีสัญญาระหว่างการทดสอบ เพื่อให้ OpenClaw สามารถยืนยันความเป็นเจ้าของได้อย่างชัดเจน ปัจจุบันใช้กับผู้ให้บริการโมเดล ผู้ให้บริการเสียงพูด ผู้ให้บริการค้นหาเว็บ และความเป็นเจ้าของการลงทะเบียนที่รวมมาให้
  </Accordion>
</AccordionGroup>

ผลในทางปฏิบัติคือ OpenClaw ทราบล่วงหน้าว่า Plugin ใดเป็นเจ้าของพื้นผิวใด ซึ่งช่วยให้แกนหลักและช่องทางต่าง ๆ ทำงานร่วมกันได้อย่างราบรื่น เพราะความเป็นเจ้าของได้รับการประกาศ กำหนดชนิด และทดสอบได้ แทนที่จะเป็นสิ่งโดยนัย

## รูปแบบการดำเนินการ

Plugin แบบเนทีฟของ OpenClaw ทำงาน **ภายในโปรเซส** ร่วมกับ Gateway โดยไม่ได้อยู่ในแซนด์บ็อกซ์ Plugin แบบเนทีฟที่โหลดแล้วมีขอบเขตความเชื่อถือระดับโปรเซสเดียวกับโค้ดแกนหลัก

<Warning>
ผลกระทบของ Plugin แบบเนทีฟ: Plugin สามารถลงทะเบียนเครื่องมือ ตัวจัดการเครือข่าย ฮุก และบริการได้ ข้อบกพร่องของ Plugin อาจทำให้ Gateway ขัดข้องหรือไม่เสถียร และ Plugin แบบเนทีฟที่เป็นอันตรายเทียบเท่ากับการเรียกใช้โค้ดใด ๆ ภายในโปรเซส OpenClaw
</Warning>

บันเดิลที่เข้ากันได้มีความปลอดภัยกว่าโดยค่าเริ่มต้น เนื่องจากปัจจุบัน OpenClaw ถือว่าบันเดิลเหล่านี้เป็นแพ็กข้อมูลเมตา/เนื้อหา ในรุ่นปัจจุบัน ส่วนใหญ่หมายถึง Skills ที่รวมมาให้

ใช้รายการอนุญาตและเส้นทางติดตั้ง/โหลดที่ระบุชัดเจนสำหรับ Plugin ที่ไม่ได้รวมมาให้ ให้ถือว่า Plugin ในเวิร์กสเปซเป็นโค้ดสำหรับช่วงการพัฒนา ไม่ใช่ค่าเริ่มต้นสำหรับการใช้งานจริง

สำหรับชื่อแพ็กเกจในเวิร์กสเปซที่รวมมาให้ ให้ยึดรหัส Plugin ไว้กับชื่อ npm: ใช้ `@openclaw/<id>` เป็นค่าเริ่มต้น หรือใช้ส่วนต่อท้ายที่กำหนดชนิดและได้รับอนุมัติ เช่น `-provider`, `-plugin`, `-speech`, `-sandbox` หรือ `-media-understanding` เมื่อแพ็กเกจตั้งใจเปิดเผยบทบาท Plugin ที่จำกัดกว่า

<Note>
**หมายเหตุด้านความเชื่อถือ:** `plugins.allow` เชื่อถือ **รหัส Plugin** ไม่ใช่แหล่งที่มาของซอร์ส Plugin ในเวิร์กสเปซที่มีรหัสเดียวกับ Plugin ที่รวมมาให้จะแทนที่สำเนาที่รวมมาให้โดยตั้งใจ เมื่อเปิดใช้งาน/เพิ่ม Plugin ในเวิร์กสเปซนั้นลงในรายการอนุญาต ลักษณะนี้เป็นเรื่องปกติและมีประโยชน์สำหรับการพัฒนาในเครื่อง การทดสอบแพตช์ และการแก้ไขเร่งด่วน ความเชื่อถือของ Plugin ที่รวมมาให้จะพิจารณาจากสแนปช็อตของซอร์ส ซึ่งได้แก่ไฟล์แมนิเฟสต์และโค้ดบนดิสก์ ณ เวลาโหลด ไม่ใช่จากข้อมูลเมตาการติดตั้ง ระเบียนการติดตั้งที่เสียหายหรือถูกแทนที่จะไม่สามารถขยายพื้นผิวความเชื่อถือของ Plugin ที่รวมมาให้โดยไม่มีการแจ้งเตือน เกินกว่าสิ่งที่ซอร์สจริงระบุไว้
</Note>

## ขอบเขตการส่งออก

OpenClaw ส่งออกความสามารถ ไม่ใช่ความสะดวกของการใช้งานจริง

คงการลงทะเบียนความสามารถให้เป็นสาธารณะ ตัดการส่งออกตัวช่วยที่ไม่ได้อยู่ในสัญญา:

- พาธย่อยของตัวช่วยเฉพาะ Plugin ที่รวมมาให้
- พาธย่อยของกลไกรันไทม์ที่ไม่ได้มีไว้เป็น API สาธารณะ
- ตัวช่วยอำนวยความสะดวกเฉพาะผู้จำหน่าย
- ตัวช่วยการตั้งค่า/การเริ่มต้นใช้งานที่เป็นรายละเอียดการใช้งานจริง

พาธย่อยของตัวช่วยเฉพาะ Plugin ที่รวมมาให้ซึ่งสงวนไว้ ถูกยกเลิกจากแมปการส่งออก SDK ที่สร้างขึ้นแล้ว ให้เก็บตัวช่วยเฉพาะเจ้าของไว้ภายในแพ็กเกจ Plugin ของเจ้าของนั้น และยกระดับเฉพาะพฤติกรรมโฮสต์ที่นำกลับมาใช้ใหม่ได้เป็นสัญญา SDK ทั่วไป เช่น `plugin-sdk/gateway-runtime`, `plugin-sdk/security-runtime` และ `plugin-sdk/plugin-config-runtime`

## รายละเอียดภายในและข้อมูลอ้างอิง

สำหรับไปป์ไลน์การโหลด โมเดลรีจิสทรี ฮุกรันไทม์ของผู้ให้บริการ เส้นทาง HTTP ของ Gateway สคีมาเครื่องมือข้อความ การแปลงเป้าหมายช่องทาง แค็ตตาล็อกผู้ให้บริการ Plugin ของกลไกบริบท และคู่มือการเพิ่มความสามารถใหม่ โปรดดู [รายละเอียดภายในของสถาปัตยกรรม Plugin](/th/plugins/architecture-internals)
