Plugin SDK คือสัญญาแบบมีชนิดข้อมูลระหว่าง Plugin กับแกนหลัก หน้านี้เป็น
เอกสารอ้างอิงสำหรับ **สิ่งที่ต้องนำเข้า** และ **สิ่งที่สามารถลงทะเบียนได้**

<Note>
  หน้านี้มีไว้สำหรับผู้เขียน Plugin ที่ใช้ `openclaw/plugin-sdk/*` ภายใน
  OpenClaw สำหรับแอปภายนอก สคริปต์ แดชบอร์ด งาน CI และส่วนขยาย IDE
  ที่ต้องการเรียกใช้เอเจนต์ผ่าน Gateway ให้ใช้
  [การผสานรวม Gateway สำหรับแอปภายนอก](/th/gateway/external-apps) แทน
</Note>

<Tip>
หากกำลังมองหาคู่มือวิธีใช้งาน ให้เริ่มจาก [การสร้าง Plugin](/th/plugins/building-plugins) ใช้ [Plugin ช่องทาง](/th/plugins/sdk-channel-plugins) สำหรับช่องทาง, [Plugin ผู้ให้บริการ](/th/plugins/sdk-provider-plugins) สำหรับผู้ให้บริการโมเดล, [Plugin แบ็กเอนด์ CLI](/th/plugins/cli-backend-plugins) สำหรับแบ็กเอนด์ AI CLI ภายในเครื่อง, [Plugin ชุดควบคุมเอเจนต์](/th/plugins/sdk-agent-harness) สำหรับตัวดำเนินการเอเจนต์แบบเนทีฟ และ [ฮุกของ Plugin](/th/plugins/hooks) สำหรับฮุกเครื่องมือหรือวงจรชีวิต
</Tip>

## รูปแบบการนำเข้า

ให้นำเข้าจากพาธย่อยที่เฉพาะเจาะจงเสมอ:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { defineChannelPluginEntry } from "openclaw/plugin-sdk/channel-core";
```

แต่ละพาธย่อยเป็นโมดูลขนาดเล็กที่ทำงานได้อย่างเป็นอิสระ ซึ่งช่วยให้เริ่มต้นระบบได้รวดเร็วและ
ป้องกันปัญหาการพึ่งพาแบบวนซ้ำ สำหรับตัวช่วยจุดเข้าใช้งาน/การบิลด์เฉพาะช่องทาง
ให้เลือกใช้ `openclaw/plugin-sdk/channel-core`; คง `openclaw/plugin-sdk/core` ไว้สำหรับ
พื้นผิวครอบคลุมที่กว้างกว่าและตัวช่วยที่ใช้ร่วมกัน เช่น
`buildChannelConfigSchema`

สำหรับการกำหนดค่าช่องทาง ให้เผยแพร่ JSON Schema ที่ช่องทางเป็นเจ้าของผ่าน
`openclaw.plugin.json#channelConfigs` พาธย่อย `plugin-sdk/channel-config-schema`
มีไว้สำหรับองค์ประกอบพื้นฐานของสคีมาที่ใช้ร่วมกันและตัวสร้างทั่วไป Plugin
ที่มาพร้อม OpenClaw ใช้ `plugin-sdk/bundled-channel-config-schema` สำหรับสคีมาช่องทางที่มาพร้อมระบบ
ซึ่งยังคงเก็บไว้ การส่งออกเพื่อความเข้ากันได้ที่เลิกใช้แล้วจะยังอยู่ใน
`plugin-sdk/channel-config-schema-legacy`; พาธย่อยของสคีมาที่มาพร้อมระบบทั้งสองแบบ
ไม่ใช่รูปแบบสำหรับ Plugin ใหม่

<Warning>
  อย่านำเข้าจุดเชื่อมต่อเพื่อความสะดวกที่มีชื่อแบรนด์ของผู้ให้บริการหรือช่องทาง (ตัวอย่างเช่น
  `openclaw/plugin-sdk/slack`, `.../discord`, `.../signal`, `.../whatsapp`)
  Plugin ที่มาพร้อมระบบจะประกอบพาธย่อย SDK ทั่วไปภายในแบร์เรล `api.ts` /
  `runtime-api.ts` ของตนเอง ผู้ใช้แกนหลักควรใช้แบร์เรลภายใน Plugin เหล่านั้น
  หรือเพิ่มสัญญา SDK ทั่วไปที่มีขอบเขตแคบเมื่อความต้องการนั้นใช้ข้ามช่องทาง
  อย่างแท้จริง

จุดเชื่อมต่อตัวช่วยของ Plugin ที่มาพร้อมระบบชุดเล็กบางส่วนยังคงปรากฏในแมปการส่งออก
ที่สร้างขึ้นเมื่อมีการติดตามการใช้งานโดยเจ้าของ จุดเชื่อมต่อเหล่านี้มีไว้สำหรับ
การบำรุงรักษา Plugin ที่มาพร้อมระบบเท่านั้น และไม่ใช่พาธนำเข้าที่แนะนำสำหรับ
Plugin บุคคลที่สามรายใหม่

`openclaw/plugin-sdk/discord` และ `openclaw/plugin-sdk/telegram-account`
ยังคงเก็บไว้เป็นส่วนหน้าความเข้ากันได้ที่เลิกใช้แล้วสำหรับการใช้งานของเจ้าของที่มีการติดตาม
อย่าคัดลอกพาธนำเข้าเหล่านั้นไปยัง Plugin ใหม่ ให้ใช้ตัวช่วยรันไทม์ที่ฉีดเข้ามาและ
พาธย่อย SDK ช่องทางทั่วไปแทน
</Warning>

## เอกสารอ้างอิงพาธย่อย

Plugin SDK เปิดให้ใช้งานเป็นชุดพาธย่อยที่มีขอบเขตแคบ โดยจัดกลุ่มตามพื้นที่ (จุดเข้าใช้งาน
Plugin, ช่องทาง, ผู้ให้บริการ, การยืนยันตัวตน, รันไทม์, ความสามารถ, หน่วยความจำ และตัวช่วย
Plugin ที่มาพร้อมระบบซึ่งสงวนไว้) สำหรับแค็ตตาล็อกฉบับเต็มที่จัดกลุ่มและเชื่อมโยงแล้ว โปรดดู
[พาธย่อยของ Plugin SDK](/th/plugins/sdk-subpaths)

รายการจุดเข้าใช้งานของคอมไพเลอร์อยู่ใน
`scripts/lib/plugin-sdk-entrypoints.json`; การส่งออกแพ็กเกจสร้างจาก
ชุดย่อยสาธารณะหลังหักพาธย่อยสำหรับการทดสอบ/ภายในเฉพาะรีโพที่ระบุไว้ใน
`scripts/lib/plugin-sdk-private-local-only-subpaths.json` เรียกใช้
`pnpm plugin-sdk:surface` เพื่อตรวจสอบจำนวนการส่งออกสาธารณะ พาธย่อยสาธารณะ
ที่เลิกใช้แล้วซึ่งเก่าพอและไม่ได้ใช้โดยโค้ดโปรดักชันของส่วนขยายที่มาพร้อมระบบ
จะถูกติดตามใน `scripts/lib/plugin-sdk-deprecated-public-subpaths.json`; แบร์เรลส่งออกซ้ำแบบกว้าง
ที่เลิกใช้แล้วจะถูกติดตามใน
`scripts/lib/plugin-sdk-deprecated-barrel-subpaths.json`

## API การลงทะเบียน

คอลแบ็ก `register(api)` รับอ็อบเจ็กต์ `OpenClawPluginApi` ที่มี
เมธอดต่อไปนี้:

### การลงทะเบียนความสามารถ

| เมธอด                                           | สิ่งที่ลงทะเบียน                                                                 |
| ------------------------------------------------ | --------------------------------------------------------------------------------- |
| `api.registerProvider(...)`                      | การอนุมานข้อความ (LLM)                                                              |
| `api.registerWorkerProvider(...)`                | ลีสวงจรชีวิตของเวิร์กเกอร์บนคลาวด์                                                     |
| `api.registerModelCatalogProvider(...)`          | แถวแค็ตตาล็อกโมเดลสำหรับการสร้างข้อความและสื่อ                                  |
| `api.registerAgentHarness(...)`                  | ตัวดำเนินการเอเจนต์แบบเนทีฟ [ทดลอง](/th/plugins/sdk-agent-harness) (Codex, Copilot) |
| `api.registerCliBackend(...)`                    | แบ็กเอนด์การอนุมาน CLI ภายในเครื่อง                                                       |
| `api.registerChannel(...)`                       | ช่องทางรับส่งข้อความ                                                                 |
| `api.registerEmbeddingProvider(...)`             | ผู้ให้บริการเวกเตอร์เอ็มเบดดิงที่ใช้ซ้ำได้                                                |
| `api.registerSpeechProvider(...)`                | การสังเคราะห์ข้อความเป็นเสียงพูด / STT                                                    |
| `api.registerRealtimeTranscriptionProvider(...)` | การถอดเสียงแบบเรียลไทม์ชนิดสตรีม                                                  |
| `api.registerRealtimeVoiceProvider(...)`         | เซสชันเสียงเรียลไทม์แบบสองทิศทาง                                                    |
| `api.registerMediaUnderstandingProvider(...)`    | การวิเคราะห์รูปภาพ/เสียง/วิดีโอ                                                        |
| `api.registerTranscriptSourceProvider(...)`      | แหล่งข้อมูลบทถอดเสียงการประชุมแบบสดหรือนำเข้า                                        |
| `api.registerImageGenerationProvider(...)`       | การสร้างรูปภาพ                                                                  |
| `api.registerMusicGenerationProvider(...)`       | การสร้างเพลง                                                                  |
| `api.registerVideoGenerationProvider(...)`       | การสร้างวิดีโอ                                                                  |
| `api.registerWebFetchProvider(...)`              | ผู้ให้บริการดึงข้อมูลเว็บ / เก็บข้อมูลจากเว็บ                                                       |
| `api.registerWebSearchProvider(...)`             | การค้นหาเว็บ                                                                        |
| `api.registerCompactionProvider(...)`            | แบ็กเอนด์ Compaction บทถอดเสียงแบบถอดเปลี่ยนได้                                           |

ผู้ให้บริการเวิร์กเกอร์ต้องประกาศ ID ของตนใน `contracts.workerProviders` ด้วย
แกนหลักจะคงเจตนาที่ถาวรก่อน `provision(profile, operationId)` ผู้ให้บริการจะตรวจสอบการตั้งค่าก่อนจัดสรรทรัพยากรภายนอก และส่งข้อผิดพลาด `WorkerProviderError` เมื่อปฏิเสธโปรไฟล์อย่างถาวร `provision` ต้องรับช่วงลีสเดิมเมื่อมี ID การดำเนินการซ้ำ
แกนหลักจะคงการตั้งค่าโปรไฟล์ที่ตรวจสอบแล้วไว้กับลีส และส่งสแนปช็อตนั้นให้ `destroy({ leaseId, profile })` ซึ่งต้องทำซ้ำได้โดยให้ผลเดิม และ `inspect({ leaseId, profile })` ซึ่งส่งคืน `active`, `destroyed` หรือ `unknown` วิธีนี้ช่วยให้ผู้ให้บริการกำหนดเส้นทางการเรียกวงจรชีวิตหลัง Gateway เริ่มทำงานใหม่หรือหลังลบโปรไฟล์ที่มีชื่อ ปลายทาง SSH ใช้ `SecretRef` สำหรับ `keyRef` และห้ามใส่เนื้อหาคีย์แบบอินไลน์โดยเด็ดขาด รวมทั้งต้องมี `hostKey` จากผลลัพธ์การจัดเตรียมที่เชื่อถือได้เป็น `algorithm base64` ทุกประการ โดยไม่มีชื่อโฮสต์หรือความคิดเห็น แกนหลักจะตรึง `hostKey` และไม่เชื่อถือคีย์จากการเชื่อมต่อครั้งแรก ผู้ให้บริการที่สร้าง `keyRef` แบบไดนามิกสามารถใช้งาน `resolveSshIdentity({ leaseId, profile, keyRef })`; เมื่อมีตัวแก้ไขนี้ ตัวแก้ไขดังกล่าวจะเป็นแหล่งข้อมูลที่มีอำนาจ ส่วนผู้ให้บริการที่ไม่มีจะใช้ตัวแก้ไขข้อมูลลับทั่วไปที่กำหนดค่าไว้
ผู้ให้บริการที่มีลีสซึ่งต่ออายุได้สามารถใช้งาน `renew(leaseId)` ได้ด้วย
`inspect` ต้องส่งข้อผิดพลาดเมื่อเกิดความล้มเหลวชั่วคราวหรือไม่สามารถระบุได้; ให้ส่งคืน `unknown` เฉพาะเมื่อยืนยันได้อย่างเด็ดขาดว่าไม่มีอยู่ แกนหลักจะทำเครื่องหมายระเบียนภายในเครื่องที่ทำงานอยู่ว่าไม่มีเจ้าของ หรือถือว่าการไม่มีอยู่นั้นเป็นการรื้อถอนเสร็จสมบูรณ์หลังคำขอทำลายที่คงไว้แล้ว

ผู้ให้บริการเอ็มเบดดิงที่ลงทะเบียนด้วย `api.registerEmbeddingProvider(...)` ต้อง
อยู่ในรายการ `contracts.embeddingProviders` ในไฟล์กำกับ Plugin ด้วย
นี่คือพื้นผิวเอ็มเบดดิงทั่วไปสำหรับการสร้างเวกเตอร์ที่ใช้ซ้ำได้ การค้นหาหน่วยความจำ
สามารถใช้งานพื้นผิวผู้ให้บริการทั่วไปนี้ได้ จุดเชื่อมต่อรุ่นเก่า
`api.registerMemoryEmbeddingProvider(...)` และ
`contracts.memoryEmbeddingProviders` เป็นความเข้ากันได้ที่เลิกใช้แล้วระหว่างที่
ผู้ให้บริการเฉพาะหน่วยความจำเดิมกำลังย้ายระบบ

ผู้ให้บริการเฉพาะหน่วยความจำที่ยังคงเปิดเผย `batchEmbed(...)` ของรันไทม์ จะยังใช้
สัญญาการจัดชุดต่อไฟล์ที่มีอยู่ เว้นแต่รันไทม์จะตั้งค่า
`sourceWideBatchEmbed: true` อย่างชัดเจน การเลือกใช้นี้ช่วยให้โฮสต์หน่วยความจำส่งชิ้นข้อมูลจาก
ไฟล์หน่วยความจำที่มีการเปลี่ยนแปลงหลายไฟล์และแหล่งข้อมูลที่เปิดใช้งานในการเรียก `batchEmbed(...)` ครั้งเดียว
ได้ไม่เกินขีดจำกัดชุดของโฮสต์ อะแดปเตอร์ชุดที่อัปโหลดไฟล์คำขอ JSONL ต้อง
แบ่งงานของผู้ให้บริการก่อนถึงขีดจำกัดขนาดการอัปโหลด รวมถึงขีดจำกัด
จำนวนคำขอ ผู้ให้บริการต้องส่งคืนเอ็มเบดดิงหนึ่งรายการต่อชิ้นข้อมูลนำเข้าในลำดับเดียวกับ
`batch.chunks`; ไม่ต้องใส่แฟล็กนี้เมื่อผู้ให้บริการคาดหวังชุดข้อมูลเฉพาะไฟล์ หรือ
ไม่สามารถรักษาลำดับข้อมูลนำเข้าตลอดงานระดับแหล่งข้อมูลที่ใหญ่กว่าได้

### เครื่องมือและคำสั่ง

ใช้ [`defineToolPlugin`](/th/plugins/tool-plugins) สำหรับ Plugin ที่มีเฉพาะเครื่องมืออย่างง่าย
ซึ่งมีชื่อเครื่องมือคงที่ ใช้ `api.registerTool(...)` โดยตรงสำหรับ Plugin แบบผสม
หรือการลงทะเบียนเครื่องมือแบบไดนามิกเต็มรูปแบบ

| เมธอด                                 | สิ่งที่ลงทะเบียน                                                                                                                        |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `api.registerTool(tool, opts?)`        | เครื่องมือเอเจนต์ (จำเป็นหรือ `{ optional: true }`)                                                                                            |
| `api.registerCommand(def)`             | คำสั่งกำหนดเอง (ข้าม LLM)                                                                                                        |
| `api.registerNodeHostCommand(command)` | คำสั่งที่จัดการโดย `openclaw node run`; ข้อมูลเมตา `agentTool` ที่เป็นทางเลือกสามารถเปิดเผยคำสั่งนี้เป็นเครื่องมือที่เอเจนต์มองเห็นได้ขณะที่ Node เชื่อมต่ออยู่ |

คำสั่ง Plugin สามารถตั้งค่า `agentPromptGuidance` เมื่อเอเจนต์ต้องการคำใบ้
การกำหนดเส้นทางแบบสั้นที่คำสั่งเป็นเจ้าของ ให้ข้อความนั้นกล่าวถึงตัวคำสั่งเองเท่านั้น อย่าเพิ่ม
นโยบายเฉพาะผู้ให้บริการหรือ Plugin ไปยังตัวสร้างพรอมต์ของแกนหลัก

รายการคำแนะนำอาจเป็นสตริงแบบเดิม ซึ่งใช้กับพื้นผิวพรอมต์ทั้งหมด หรือ
รายการแบบมีโครงสร้าง:

```ts
agentPromptGuidance: [
  "คำใบ้คำสั่งส่วนกลาง",
  { text: "แสดงข้อความนี้เฉพาะในพรอมต์หลักของ OpenClaw", surfaces: ["openclaw_main"] },
];
```

`surfaces` แบบมีโครงสร้างอาจมี `openclaw_main`, `codex_app_server`,
`cli_backend`, `acp_backend` หรือ `subagent` `pi_main` ยังคงเป็นนามแฝง
ที่เลิกใช้แล้วของ `openclaw_main` ไม่ต้องระบุ `surfaces` สำหรับคำแนะนำที่ตั้งใจให้ใช้กับทุกพื้นผิว อย่า
ส่งอาร์เรย์ `surfaces` ว่าง ระบบจะปฏิเสธเพื่อป้องกันไม่ให้การสูญเสียขอบเขตโดยไม่ตั้งใจ
กลายเป็นข้อความพรอมต์ส่วนกลาง

คำสั่งสำหรับนักพัฒนาของเซิร์ฟเวอร์แอป Codex แบบเนทีฟเข้มงวดกว่าพื้นผิวพรอมต์
อื่น ๆ: เฉพาะคำแนะนำที่กำหนดขอบเขตเป็น `codex_app_server` อย่างชัดเจนเท่านั้นที่จะถูกยกระดับไปยัง
ช่องทางที่มีลำดับความสำคัญสูงกว่านั้น คำแนะนำแบบสตริงเดิมและคำแนะนำแบบมีโครงสร้าง
ที่ไม่ได้กำหนดขอบเขตจะยังคงใช้ได้กับพื้นผิวพรอมต์ที่ไม่ใช่ Codex เพื่อความเข้ากันได้

คำสั่งโฮสต์ Node ทำงานบนโฮสต์ Node ที่เชื่อมต่อ ไม่ใช่ภายในโปรเซส Gateway
หากมี `agentTool` Node จะเผยแพร่ตัวบรรยายหลังเชื่อมต่อ Gateway
สำเร็จ; Gateway จะเปิดเผยตัวบรรยายนั้นแก่การเรียกใช้เอเจนต์เฉพาะขณะที่ Node นั้น
เชื่อมต่ออยู่ และเฉพาะเมื่อ `command` ของตัวบรรยายอยู่ในพื้นผิวคำสั่ง
ที่ Node อนุมัติ ตั้งค่า `agentTool.defaultPlatforms` เพื่อเลือกเพิ่มคำสั่ง
ที่ไม่เป็นอันตรายลงในรายการอนุญาตคำสั่งเริ่มต้นของ Node; มิฉะนั้นต้องระบุ
`gateway.nodes.allowCommands` อย่างชัดเจนหรือนโยบายการเรียกใช้ Node `agentTool.name`
ต้องปลอดภัยสำหรับผู้ให้บริการ: ขึ้นต้นด้วยตัวอักษร ใช้ได้เฉพาะตัวอักษร ตัวเลข
ขีดล่าง หรือขีดกลาง และมีความยาวไม่เกิน 64 อักขระ เครื่องมือ Node
ที่มี MCP เป็นแบ็กเอนด์สามารถตั้งค่าข้อมูลเมตา `agentTool.mcp` เพื่อให้พื้นผิว
แค็ตตาล็อกและการค้นหาเครื่องมือแสดงข้อมูลประจำตัวของเซิร์ฟเวอร์/เครื่องมือ MCP ระยะไกลได้
แต่การดำเนินการยังคงผ่านคำสั่ง Node ที่ประกาศไว้

#### งาน Webhook หลังการตอบรับ

เส้นทาง Webhook ที่ตอบรับคำขอก่อนการประมวลผลเสร็จสิ้นต้องย้าย
งานที่แยกออกมานั้นไปยังรากการรับงานที่ติดตามแยกต่างหาก:

```typescript
import { runDetachedWebhookWork } from "openclaw/plugin-sdk/webhook-request-guards";

void runDetachedWebhookWork(() => processWebhookEvent(event)).catch((error) => {
  runtime.error?.(`การส่งต่อ webhook ล้มเหลว: ${String(error)}`);
});
```

เรียก `runDetachedWebhookWork(...)` แบบซิงโครนัสขณะที่คำขอ HTTP ยัง
ได้รับอนุญาตให้ดำเนินการ ตัวช่วยจะสำรองรากอิสระทันที จากนั้นเริ่ม
คอลแบ็กในไมโครทาสก์ถัดไป เพื่อให้ตัวจัดการคำขอเขียน
การตอบรับก่อน Promise ที่ส่งคืนจะรับผลลัพธ์ของคอลแบ็ก โดยผู้เรียก
ยังคงต้องจัดการการปฏิเสธเอง วิธีนี้ช่วยให้งานในคิวหลังการตอบรับได้รับการยอมรับ และทำให้
การระบายงานเมื่อรีสตาร์ตหรือระงับรอให้งานดังกล่าวเสร็จสิ้น ตัวจัดการที่รอการประมวลผลทั้งหมด
ก่อนส่งคืนไม่จำเป็นต้องใช้ตัวช่วยนี้

#### การเชื่อมต่อ MCP ที่จำกัดขอบเขตตามผู้ร้องขอ

กำหนด **ข้อมูลประจำตัว** ของเซิร์ฟเวอร์ MCP (ชื่อ ตัวกรองเครื่องมือ) ให้คงที่ใน `mcp.servers` หรือ
ไฟล์กำกับบันเดิล สามารถลงทะเบียนตัวแก้ไขการเชื่อมต่อเพิ่มเติมเพื่อให้ผู้ร้องขอข้อความ
ที่เชื่อถือได้แต่ละรายได้รับการรับส่งของตนเอง:

```ts
api.registerMcpServerConnectionResolver({
  serverName: "user-email",
  resolve: async (ctx) => {
    // ctx.requesterSenderId ได้รับความเชื่อถือจากโฮสต์ ห้ามสร้างข้อมูลประจำตัวผู้ส่งขึ้นเองที่นี่
    const token = await lookupUserToken(ctx.requesterSenderId);
    if (!token) {
      return null; // ละเว้นเซิร์ฟเวอร์นี้สำหรับการทำงานปัจจุบัน
    }
    return {
      url: "https://mcp.example.com/email",
      headers: { Authorization: `Bearer ${token}` },
    };
  },
});
```

หมายเหตุเกี่ยวกับสัญญา:

- บริบทของตัวแก้ไขมีเฉพาะข้อมูลประจำตัวที่โฮสต์เชื่อถือ (`requesterSenderId`,
  และ `agentAccountId` / `messageChannel` ซึ่งเป็นทางเลือก) ในอนาคตสามารถเพิ่มฟิลด์ที่เชื่อถือได้
  (เช่น บริบทผู้ใช้ของ cron/เอเจนต์ย่อย) แบบเพิ่มเติมได้
- หนึ่ง Plugin เป็นเจ้าของชื่อเซิร์ฟเวอร์หนึ่งชื่อ: หากมี
  `registerMcpServerConnectionResolver` ซ้ำสำหรับ `serverName` เดียวกันจาก
  Plugin อื่น ระบบจะปฏิเสธพร้อมข้อมูลวินิจฉัยข้อผิดพลาด (การลงทะเบียนแรกมีผล) ดังนั้น
  ความเป็นเจ้าของการเชื่อมต่อจึงไม่ขึ้นกับลำดับการโหลด Plugin
- ชื่อเครื่องมือได้มาจากชุดเซิร์ฟเวอร์ที่ประกาศทั้งหมด เพื่อให้การแก้ไขเพียงบางส่วน
  ไม่เปลี่ยนชื่อเซิร์ฟเวอร์ที่ปลอดภัยระหว่างผู้ร้องขอหรือรอบการทำงาน Core ไม่
  ตรวจสอบว่าปลายทางของผู้ร้องขอแต่ละรายให้สคีมาเครื่องมือเหมือนกันหรือไม่ ตัวแก้ไข
  ต้องชี้ผู้ร้องขอทุกรายไปยังบริการเชิงตรรกะเดียวกัน มิฉะนั้นสคีมาเครื่องมือ
  (และความเสถียรของแคชพรอมต์) จะแตกต่างกันตามผู้ร้องขอ
- การทำงานที่ไม่มี `requesterSenderId` ที่เชื่อถือได้ (cron, เอเจนต์ย่อย, Heartbeat, Gateway
  สาธารณะ) จะไม่สร้างเซิร์ฟเวอร์ที่จำกัดขอบเขตตามผู้ร้องขอขึ้นมา ไม่มี
  การเชื่อมต่อสำรองที่ใช้ร่วมกัน
- `resolve` ถูกจำกัดไว้ที่ 10 วินาทีต่อเซิร์ฟเวอร์ หากหมดเวลาหรือเกิดข้อยกเว้น ระบบจะละเว้น
  เซิร์ฟเวอร์นั้นสำหรับการทำงาน โดยไม่ทำให้ MCP แบบคงที่ล้มเหลว
- การเชื่อมต่อที่แก้ไขแล้วจะได้รับการตรวจสอบซ้ำไม่เกินทุก 5 นาทีต่อผู้ร้องขอ:
  การหมุนเวียนจะสร้างการรับส่งใหม่ด้วยข้อมูลรับรองล่าสุด และผลลัพธ์ `null`
  จะเพิกถอนการเชื่อมต่อ (รันไทม์ที่แคชไว้จะถูกกำจัดแม้อยู่ระหว่างเซสชัน) ดังนั้นข้อมูลรับรองที่ถูกเพิกถอนหรือ
  หมุนเวียนแล้วอาจยังถูกใช้งานต่อได้นานถึง 5 นาที
- `headers` ที่แก้ไขแล้วจะไม่ถูกบันทึกในล็อกหรือคงอยู่ถาวร Core เก็บไว้เพียงไดเจสต์แบบใช้คีย์
  ในหน่วยความจำชั่วคราว (HMAC เฉพาะกระบวนการ) เพื่อตรวจจับการหมุนเวียนข้อมูลรับรอง และ
  ลงทะเบียนค่าข้อมูลรับรองในส่วนหัว/URL ที่แก้ไขแล้วกับรีจิสทรีการปกปิดข้อมูล
  สำหรับล็อก/การจับข้อมูลดีบัก
- เซิร์ฟเวอร์ที่จำกัดขอบเขตตามผู้ร้องขอจะไม่สร้างมุมมอง MCP App เนื่องจากมุมมองมีอายุยาวกว่า
  การทำงานที่ยืนยันตัวตนผู้ร้องขอ และขอบเขตมุมมองของ Gateway ไม่มีข้อมูลประจำตัว
  ผู้ร้องขอ ดังนั้นตัวอย่างแอปจะยังคงปิดกั้นโดยค่าเริ่มต้นสำหรับเซิร์ฟเวอร์เหล่านี้ ผลลัพธ์เครื่องมือ
  ไม่ได้รับผลกระทบ
- เซิร์ฟเวอร์แบบคงที่ที่ไม่มีตัวแก้ไขจะคงวงจรชีวิตที่จำกัดขอบเขตตามเซสชันแบบเดิม
- **กฎการส่งมอบของฮาร์เนส:** เซิร์ฟเวอร์ที่จำกัดขอบเขตตามผู้ร้องขอจะไม่เข้าสู่
  การกำหนดค่าไคลเอนต์ MCP แบบเนทีฟของฮาร์เนส (เธรด Codex `mcp_servers`, CLI `-c mcp_servers=…` หรือ
  การฉาย MCP อื่นใดที่ใช้ร่วมกันในเซสชัน) แต่ฮาร์เนสจะส่งมอบเซิร์ฟเวอร์เหล่านี้เป็นเครื่องมือที่จำกัดขอบเขต
  ตามการทำงานแทน:
  - ตัวรันแบบฝัง: รันไทม์ MCP ของเซสชัน + เครื่องมือบันเดิล (แบบคงที่ + แบบจำกัดขอบเขต)
  - เซิร์ฟเวอร์แอป Codex: เครื่องมือแบบไดนามิกผ่าน
    `materializeRequesterScopedMcpToolsForHarnessRun` (เฉพาะแบบจำกัดขอบเขต ส่วนเซิร์ฟเวอร์แบบคงที่
    ยังคงใช้ไคลเอนต์ MCP แบบเนทีฟของ Codex)
- **ข้อกำหนด** ของเครื่องมือแบบจำกัดขอบเขตจะคงที่ตลอดเซสชันหลังการแก้ไขสำเร็จครั้งแรก
  ในเซสชันนั้น เพื่อให้ฮาร์เนสที่ใช้เธรดร่วมกัน (Codex) ไม่หมุนเวียนเธรดเมื่อ
  ผู้ส่งเปลี่ยนไป ก่อนที่ผู้ร้องขอรายใดจะแก้ไขสำเร็จ จะไม่มีการประกาศข้อกำหนดแบบจำกัดขอบเขต
- ผู้ร้องขอที่ไม่ได้ยืนยันตัวตนบนฮาร์เนสที่ใช้เธรดร่วมกันยังคงเห็นเครื่องมือแบบจำกัดขอบเขต
  ที่ประกาศไว้ การเรียกใช้เครื่องมือดังกล่าวจะส่งคืนข้อผิดพลาดเครื่องมือว่าไม่ได้เชื่อมต่ออย่างชัดเจนสำหรับ
  ผู้ร้องขอรายนั้น OpenClaw จะไม่ย้อนกลับไปใช้ข้อมูลรับรองของผู้ร้องขอรายอื่นเด็ดขาด

ตัวสร้างส่วนเสริมพรอมต์หน่วยความจำจะได้รับบริบท `agentId`,
`agentSessionKey` และ `sandboxed` ซึ่งเป็นทางเลือก การเรียกส่วนเสริมคลังข้อมูลหน่วยความจำ `search`
และ `get` จะได้รับบริบท `agentId` และ `sandboxed` ซึ่งเป็นทางเลือก Plugin ที่มี
พื้นที่จัดเก็บซึ่งเอเจนต์เป็นเจ้าของควรแก้ไขพื้นที่จัดเก็บนั้นสำหรับแต่ละการเรียก แทนการ
จับพาธส่วนกลางเพียงพาธเดียวระหว่างการลงทะเบียน หากการดำเนินการแบบหลายเอเจนต์ต้องใช้รหัสเอเจนต์แต่
ไม่มีรหัสดังกล่าว ให้ปิดกั้นโดยค่าเริ่มต้นแทนการเลือกเอเจนต์

ใช้ `registerMemoryPromptPreparation(...)` เมื่อข้อความพรอมต์ขึ้นอยู่กับสถานะ
อะซิงโครนัสของ Plugin คอลแบ็กจะทำงานหนึ่งครั้งก่อนพรอมต์เอเจนต์ฉบับเต็มแต่ละครั้ง และได้รับ
บริบทเครื่องมือ เอเจนต์ เซสชัน และแซนด์บ็อกซ์เดียวกับตัวสร้างพรอมต์หน่วยความจำ
แบบซิงโครนัส ตรวจสอบอินสแตนซ์เจ้าของพื้นที่จัดเก็บปัจจุบันก่อนโหลดสถานะที่คงอยู่
จากนั้นส่งคืนเฉพาะบรรทัดสำหรับการทำงานนั้น OpenClaw จะตรึงบรรทัดเหล่านั้นและ
ส่งผลลัพธ์ที่เปลี่ยนแปลงไม่ได้ให้กับการประกอบพรอมต์แบบซิงโครนัส เก็บการคงอยู่
การแทนที่แบบอะตอมมิก และการลบเมื่อถอดเจ้าของไว้ภายใน Plugin ที่เป็นเจ้าของ ห้าม
สำรวจหรืออ่านไฟล์จากตัวสร้างพรอมต์

ตัวจัดการแบบโต้ตอบของ Telegram สามารถส่งคืน `{ submitText }` เพื่อกำหนดเส้นทางข้อความผ่าน
เส้นทางเอเจนต์ขาเข้าปกติของ Telegram หลังจากตัวจัดการทำงานสำเร็จ OpenClaw จะเก็บ
ปุ่มคอลแบ็กไว้เมื่อนโยบายขาเข้าข้ามข้อความหรือการประมวลผลล้มเหลว เพื่อให้
ผู้ใช้ลองอีกครั้งได้หลังจากเงื่อนไขที่ขัดขวางเปลี่ยนไป ฟิลด์ผลลัพธ์นี้
เป็นฟิลด์เฉพาะของ Telegram ส่วนช่องทางอื่นจะใช้สัญญาผลลัพธ์แบบโต้ตอบของตนเอง

### ฮุกของโฮสต์สำหรับ Plugin เวิร์กโฟลว์

ฮุกของโฮสต์คือจุดเชื่อมต่อ SDK สำหรับ Plugin ที่ต้องมีส่วนร่วมในวงจรชีวิตของโฮสต์
แทนที่จะเพิ่มเพียงผู้ให้บริการ ช่องทาง หรือเครื่องมือ ฮุกเหล่านี้เป็น
สัญญาทั่วไป Plan Mode สามารถใช้ได้ เช่นเดียวกับเวิร์กโฟลว์การอนุมัติ
จุดควบคุมนโยบายพื้นที่ทำงาน ตัวเฝ้าติดตามเบื้องหลัง วิซาร์ดการตั้งค่า และ Plugin คู่หู
สำหรับ UI

| เมธอด                                                                               | สัญญาที่เมธอดเป็นเจ้าของ                                                                                                                                           |
| ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `api.session.state.registerSessionExtension(...)`                                    | สถานะเซสชันที่ Plugin เป็นเจ้าของและเข้ากันได้กับ JSON ซึ่งฉายผ่านเซสชันของ Gateway                                                                             |
| `api.session.workflow.enqueueNextTurnInjection(...)`                                 | บริบทแบบ exactly-once ที่คงทน ซึ่งแทรกเข้าสู่รอบถัดไปของเอเจนต์สำหรับหนึ่งเซสชัน                                                                             |
| `api.registerTrustedToolPolicy(...)`                                                 | นโยบายเครื่องมือก่อน Plugin ที่เชื่อถือได้และควบคุมด้วย manifest ซึ่งสามารถบล็อกหรือเขียนพารามิเตอร์เครื่องมือใหม่ได้                                                                        |
| `api.registerToolMetadata(...)`                                                      | เมทาดาทาการแสดงผลแค็ตตาล็อกเครื่องมือโดยไม่เปลี่ยนการทำงานของเครื่องมือ                                                                                     |
| `api.registerCommand(...)`                                                           | คำสั่ง Plugin ที่จำกัดขอบเขต; ผลลัพธ์ของคำสั่งสามารถตั้งค่า `continueAgent: true` หรือ `suppressReply: true`; คำสั่งเนทีฟของ Discord รองรับ `descriptionLocalizations` |
| `api.session.controls.registerControlUiDescriptor(...)`                              | ตัวอธิบายส่วนเสริมของ Control UI สำหรับพื้นผิวเซสชัน เครื่องมือ การรัน การตั้งค่า หรือแท็บ                                                                      |
| `api.lifecycle.registerRuntimeLifecycle(...)`                                        | คอลแบ็กการล้างข้อมูลสำหรับทรัพยากรรันไทม์ที่ Plugin เป็นเจ้าของในเส้นทางรีเซ็ต/ลบ/โหลดใหม่                                                                          |
| `api.agent.events.registerAgentEventSubscription(...)`                               | การสมัครรับเหตุการณ์ที่ผ่านการทำให้ปลอดภัยสำหรับสถานะเวิร์กโฟลว์และตัวเฝ้าติดตาม                                                                                              |
| `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`  | สถานะชั่วคราวของ Plugin ต่อการรัน ซึ่งถูกล้างเมื่อวงจรชีวิตการรันสิ้นสุด                                                                                             |
| `api.session.workflow.registerSessionSchedulerJob(...)`                              | เมทาดาทาการล้างข้อมูลสำหรับงานตัวจัดกำหนดการที่ Plugin เป็นเจ้าของ; ไม่ได้จัดกำหนดการทำงานหรือสร้างระเบียนงาน                                                            |
| `api.session.workflow.sendSessionAttachment(...)`                                    | การส่งไฟล์แนบที่สื่อกลางโดยโฮสต์และใช้ได้เฉพาะ Plugin ที่รวมมากับระบบ ไปยังเส้นทางเซสชันขาออกโดยตรงที่ใช้งานอยู่                                                            |
| `api.session.workflow.scheduleSessionTurn(...)` / `unscheduleSessionTurnsByTag(...)` | รอบเซสชันตามกำหนดการที่ใช้ Cron และใช้ได้เฉพาะ Plugin ที่รวมมากับระบบ พร้อมการล้างข้อมูลตามแท็ก                                                                                    |

ตัวอธิบาย `surface: "tab"` จะเพิ่มแท็บแถบด้านข้างไปยัง Control UI ตัวอธิบายแท็บ
ของ Plugin ที่ใช้งานอยู่จะถูกประกาศแก่ไคลเอนต์แดชบอร์ดในข้อความ hello ของ Gateway
(`controlUiTabs`) ดังนั้นแท็บจึงปรากฏเฉพาะขณะที่เปิดใช้งาน Plugin
Plugin ที่รวมมากับระบบอาจมีมุมมองแดชบอร์ดระดับเฟิร์สต์คลาสสำหรับแท็บของตน; Plugin อื่น
สามารถตั้งค่า `path` ให้เป็นเส้นทาง HTTP ของ Plugin (ดู
`api.registerHttpRoute(...)`) ซึ่งแดชบอร์ดจะแสดงผลในเฟรมแบบแซนด์บ็อกซ์
`icon` เป็นคำใบ้ชื่อไอคอนแดชบอร์ด, `group` เลือกส่วนของแถบด้านข้าง
(`control` หรือ `agent`), `order` ใช้เรียงลำดับระหว่างแท็บของ Plugin และ `requiredScopes`
ซ่อนแท็บจากการเชื่อมต่อที่ไม่มีขอบเขตผู้ดำเนินการเหล่านั้น:

สำหรับแท็บภายนอกที่ Gateway ปกป้อง ให้ลงทะเบียนตัวอธิบาย `path` ภายใต้
เส้นทาง HTTP `auth: "gateway"` ของ Plugin เดียวกัน หลังจากบูตสแตรปที่ผ่านการยืนยันตัวตนแล้ว เบราว์เซอร์จะได้รับ
สิทธิ์อนุญาต HttpOnly อายุสั้นที่จำกัดขอบเขตไว้เฉพาะ Plugin และรูทของเส้นทางนั้น เพื่อให้
เฟรมแบบแซนด์บ็อกซ์โหลดได้โดยไม่ต้องคัดลอก bearer token ของ Gateway ลงใน URL
หรือ JavaScript พาเรนต์ที่ผ่านการยืนยันตัวตนจะต่ออายุสิทธิ์อนุญาตขณะที่แท็บภายนอก
ทำงานอยู่และก่อนเมานต์แท็บหลังการนำทางหรือการกลับมาทำงานของเบราว์เซอร์ นอกจากนี้ยัง
ตรวจสอบสิทธิ์อนุญาตจากแซนด์บ็อกซ์แบบ opaque เดียวกันก่อนเมานต์ เพื่อให้โหมดความเป็นส่วนตัว
ของเบราว์เซอร์ที่บล็อกคุกกี้ปฏิเสธอย่างปลอดภัยด้วยแผงที่ไม่พร้อมใช้งาน
สิทธิ์อนุญาตของเฟรมยอมรับเฉพาะ `GET` และ `HEAD` และมี
`operator.read` เสมอ; `requiredScopes` ควบคุมการมองเห็นแท็บแต่ไม่เคยขยาย
สิทธิ์อนุญาตของคุกกี้ การเปลี่ยนแปลงข้อมูลยังคงอยู่บนพื้นผิวพาเรนต์หรือ bearer ที่ยืนยันตัวตนกับ Gateway อย่างชัดเจน
แท็บภายนอกต้องใช้ HTTPS/Tailscale Serve หรือต้นทาง loopback
ที่เบราว์เซอร์เชื่อถือ; HTTP ธรรมดาบนโฮสต์ LAN จะแสดงข้อผิดพลาด secure-context
แทนการเมานต์แผงที่ไม่สามารถยืนยันตัวตนได้
การบล็อกคุกกี้บุคคลที่สามทั้งหมดจะทำให้แท็บที่ Gateway ปกป้องไม่พร้อมใช้งานเช่นกัน
เช่นเดียวกับพื้นผิว Plugin แบบเนทีฟทั้งหมด เฟรมยังคงอยู่ภายในขอบเขตความเชื่อถือของ
Plugin ที่ติดตั้ง; OpenClaw ไม่ถือว่า Plugin ที่ติดตั้งเป็นตัวการด้านความปลอดภัยของ
เบราว์เซอร์ที่แยกออกจากกัน
สิทธิ์อนุญาตคุกกี้ใช้ขอบเขตชื่อโฮสต์ของเบราว์เซอร์ ไม่ใช่ขอบเขตพอร์ต
อย่าโฮสต์ร่วมกับบริการที่ไม่ไว้วางใจซึ่งกันและกันบนชื่อโฮสต์ของ Gateway แม้จะอยู่คนละ
พอร์ตก็ตาม
แท็บที่ใช้การยืนยันตัวตนที่ Plugin จัดการจะคงพฤติกรรม iframe โดยตรงไว้ และไม่
ร้องขอหรือต้องใช้สิทธิ์อนุญาตของ Gateway นี้

```typescript
api.session.controls.registerControlUiDescriptor({
  surface: "tab",
  id: "logbook",
  label: "บันทึกประจำวัน",
  description: "วันของคุณในรูปแบบไทม์ไลน์ ซึ่งสร้างจากภาพบันทึกหน้าจอ",
  icon: "sun",
  group: "control",
  requiredScopes: ["operator.write"],
});
```

สัญญาแยกอำนาจโดยเจตนา:

- Plugin ภายนอกสามารถเป็นเจ้าของส่วนขยายเซสชัน ตัวอธิบาย UI คำสั่ง เมทาดาทา
  เครื่องมือ การแทรกในรอบถัดไป และฮุกทั่วไป
- นโยบายเครื่องมือที่เชื่อถือได้ทำงานก่อนฮุก `before_tool_call` ทั่วไป และได้รับ
  ความไว้วางใจจากโฮสต์ นโยบายที่รวมมากับระบบทำงานก่อน; นโยบายของ Plugin ที่ติดตั้งต้องมี
  การเปิดใช้งานอย่างชัดเจนพร้อมรหัสภายในของนโยบายใน
  `contracts.trustedToolPolicies` และทำงานถัดไปตามลำดับการโหลด Plugin รหัสนโยบาย
  จำกัดขอบเขตไว้ที่ Plugin ที่ลงทะเบียน
- การเป็นเจ้าของคำสั่งที่สงวนไว้มีไว้เฉพาะ Plugin ที่รวมมากับระบบ Plugin ภายนอกควรใช้
  ชื่อคำสั่งหรือชื่อแฝงของตนเอง
- `allowPromptInjection=false` ปิดใช้งานฮุกที่เปลี่ยนพรอมต์ รวมถึง
  `agent_turn_prepare`, `before_prompt_build`, `heartbeat_prompt_contribution`,
  ฟิลด์พรอมต์จาก `before_agent_start` แบบเดิม และ
  `enqueueNextTurnInjection`

### การลงทะเบียนการค้นพบ Gateway

`api.registerGatewayDiscoveryService(...)` ช่วยให้ Plugin ประกาศ Gateway ที่ใช้งานอยู่
บนการขนส่งเพื่อการค้นหาในเครื่อง เช่น mDNS/Bonjour OpenClaw เรียกใช้
บริการระหว่างการเริ่มต้น Gateway เมื่อเปิดใช้งานการค้นหาในเครื่อง ส่งต่อ
พอร์ต Gateway ปัจจุบันและข้อมูลคำใบ้ TXT ที่ไม่ใช่ความลับ และเรียกตัวจัดการ
`stop` ที่ส่งกลับมาระหว่างการปิด Gateway

```typescript
api.registerGatewayDiscoveryService({
  id: "my-discovery",
  async advertise(ctx) {
    const handle = await startMyAdvertiser({
      gatewayPort: ctx.gatewayPort,
      tls: ctx.gatewayTlsEnabled,
      displayName: ctx.machineDisplayName,
    });
    return { stop: () => handle.stop() };
  },
});
```

Plugin การค้นพบ Gateway ต้องไม่ถือว่าค่า TXT ที่ประกาศเป็นความลับหรือ
การยืนยันตัวตน การค้นพบเป็นเพียงคำใบ้การกำหนดเส้นทาง; การยืนยันตัวตนของ Gateway และการตรึง TLS ยังคง
เป็นเจ้าของความเชื่อถือ

### เมทาดาทาการลงทะเบียน CLI

`api.registerCli(registrar, opts?)` ยอมรับเมทาดาทาคำสั่งสองประเภท:

- `commands`: ชื่อคำสั่งที่ชัดเจนซึ่งผู้ลงทะเบียนเป็นเจ้าของ
- `descriptors`: ตัวอธิบายคำสั่งขณะแยกวิเคราะห์ที่ใช้สำหรับความช่วยเหลือ CLI
  การกำหนดเส้นทาง และการลงทะเบียน CLI ของ Plugin แบบโหลดเมื่อต้องใช้
- `parentPath`: เส้นทางคำสั่งแม่แบบไม่บังคับสำหรับกลุ่มคำสั่งซ้อน เช่น
  `["nodes"]`

สำหรับฟีเจอร์ Node ที่จับคู่แล้ว ควรใช้
`api.registerNodeCliFeature(registrar, opts?)` ซึ่งเป็นตัวห่อขนาดเล็กรอบ
`api.registerCli(..., { parentPath: ["nodes"] })` และทำให้คำสั่ง เช่น
`openclaw nodes canvas` เป็นฟีเจอร์ Node ที่ Plugin เป็นเจ้าของอย่างชัดเจน

หากต้องการให้คำสั่ง Plugin ยังคงโหลดเมื่อต้องใช้ในเส้นทาง CLI รูทปกติ
ให้ระบุ `descriptors` ที่ครอบคลุมรูทคำสั่งระดับบนสุดทุกคำสั่งที่
ผู้ลงทะเบียนนั้นเปิดเผย

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerMatrixCli } = await import("./src/cli.js");
    registerMatrixCli({ program });
  },
  {
    descriptors: [
      {
        name: "matrix",
        description: "จัดการบัญชี Matrix การยืนยัน อุปกรณ์ และสถานะโปรไฟล์",
        hasSubcommands: true,
      },
    ],
  },
);
```

คำสั่งที่ซ้อนกันจะได้รับคำสั่งแม่ที่แก้ไขแล้วเป็น `program`:

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerNodesCanvasCommands } = await import("./src/cli.js");
    registerNodesCanvasCommands(program);
  },
  {
    parentPath: ["nodes"],
    descriptors: [
      {
        name: "canvas",
        description: "จับภาพหรือเรนเดอร์เนื้อหาแคนวาสจาก Node ที่จับคู่ไว้",
        hasSubcommands: true,
      },
    ],
  },
);
```

ใช้ `commands` เพียงอย่างเดียวเฉพาะเมื่อไม่ต้องการการลงทะเบียน CLI รากแบบ lazy
เส้นทางความเข้ากันได้แบบ eager นี้ยังคงรองรับอยู่ แต่จะไม่ติดตั้ง
ตัวยึดตำแหน่งที่มี descriptor รองรับสำหรับการโหลดแบบ lazy ขณะพาร์ส

### การลงทะเบียนแบ็กเอนด์ CLI

`api.registerCliBackend(...)` ช่วยให้ Plugin เป็นเจ้าของการกำหนดค่าเริ่มต้นสำหรับแบ็กเอนด์
AI CLI ภายในเครื่อง เช่น `claude-cli` หรือ `my-cli`

- `id` ของแบ็กเอนด์จะกลายเป็นคำนำหน้าผู้ให้บริการในการอ้างอิงโมเดล เช่น `my-cli/gpt-5`
- `config` ของแบ็กเอนด์ใช้โครงสร้างเดียวกับ `agents.defaults.cliBackends.<id>`
- การกำหนดค่าของผู้ใช้ยังคงมีลำดับความสำคัญสูงกว่า OpenClaw จะผสาน `agents.defaults.cliBackends.<id>` ทับ
  ค่าเริ่มต้นของ Plugin ก่อนเรียกใช้ CLI
- ใช้ `normalizeConfig` เมื่อแบ็กเอนด์ต้องเขียนค่าความเข้ากันได้ใหม่หลังการผสาน
  (เช่น ปรับโครงสร้างแฟล็กแบบเก่าให้เป็นมาตรฐาน)
- ใช้ `resolveExecutionArgs` สำหรับการเขียน argv ใหม่ภายในขอบเขตคำขอ ซึ่งเป็นส่วนหนึ่งของ
  รูปแบบภาษาของ CLI เช่น การแมประดับการคิดของ OpenClaw ไปยังแฟล็กระดับความพยายาม
  แบบเนทีฟ Hook จะได้รับ `ctx.executionMode`; ใช้ `"side-question"` เพื่อเพิ่ม
  แฟล็กการแยกแบบเนทีฟของแบ็กเอนด์สำหรับการเรียก `/btw` ชั่วคราว หากแฟล็กเหล่านั้น
  ปิดใช้งานเครื่องมือเนทีฟได้อย่างเชื่อถือได้สำหรับ CLI ที่มิฉะนั้นจะเปิดใช้งานตลอดเวลา ให้ประกาศ
  `sideQuestionToolMode: "disabled"` ด้วย
- ใช้ `prepareExecution` สำหรับสภาพแวดล้อมการเปิดใช้งานที่แบ็กเอนด์เป็นเจ้าของ หรือบริดจ์
  การยืนยันตัวตน/การกำหนดค่าชั่วคราว `ctx.contextTokenBudget` ของรายการนี้คือขีดจำกัดโทเค็นที่มีผล
  ซึ่งเลือกไว้สำหรับการทำงาน เพื่อให้แบ็กเอนด์ที่ทำ Compaction แบบเนทีฟปรับเกณฑ์
  ของตนเองให้สอดคล้องกันได้โดยไม่ต้องมีสาขาแกนหลักเฉพาะผู้ให้บริการ
- แบ็กเอนด์ที่ปิดใช้งานเครื่องมือเนทีฟทั้งหมดสำหรับการทำงานหนึ่ง ๆ ได้ อาจประกาศ
  `nativeToolMode: "selectable"` การเรียกแบบจำกัดจะส่งทูเพิล
  `ctx.toolAvailability.native` ที่ว่างพร้อมรายการอนุญาต MCP ที่แยกจากโฮสต์อย่างเจาะจง
  `resolveExecutionArgs` ต้องบังคับใช้ทั้งสองอย่างกับ argv สุดท้ายของการทำงานใหม่หรือการทำงานต่อ
  OpenClaw จะปฏิเสธการทำงานเพื่อความปลอดภัยหากแบ็กเอนด์ไม่สามารถทำเช่นนั้นได้

สำหรับคู่มือการเขียนแบบครบวงจร โปรดดู
[Plugin แบ็กเอนด์ CLI](/th/plugins/cli-backend-plugins)

### อะแดปเตอร์ฝังข้อมูลหน่วยความจำที่เลิกใช้แล้ว

| เมธอด                                         | สิ่งที่ลงทะเบียน                              |
| ---------------------------------------------- | ---------------------------------------------- |
| `api.registerMemoryEmbeddingProvider(adapter)` | อะแดปเตอร์ฝังข้อมูลหน่วยความจำสำหรับ Plugin ที่ใช้งานอยู่ |

- `registerMemoryCapability` เป็น API Plugin หน่วยความจำแบบเฉพาะที่แนะนำ
- `registerMemoryCapability` อาจเปิดเผย `publicArtifacts.listArtifacts(...)` ด้วย
  เพื่อให้ Plugin ที่ทำงานร่วมกันใช้สิ่งประดิษฐ์หน่วยความจำที่ส่งออกผ่าน
  `openclaw/plugin-sdk/memory-host-core` แทนการเข้าถึงโครงสร้างส่วนตัวของ
  Plugin หน่วยความจำเฉพาะ
- `registerMemoryPromptSection`, `registerMemoryFlushPlan` และ
  `registerMemoryRuntime` เป็น API Plugin หน่วยความจำแบบเฉพาะที่เข้ากันได้กับระบบเดิม
- `MemoryFlushPlan.model` สามารถตรึงรอบการล้างไว้กับการอ้างอิง `provider/model`
  ที่เจาะจง เช่น `ollama/qwen3:8b` โดยไม่สืบทอดสายการสำรองที่ใช้งานอยู่
- `registerMemoryEmbeddingProvider` เลิกใช้แล้ว ผู้ให้บริการการฝังข้อมูลรายใหม่
  ควรใช้ `api.registerEmbeddingProvider(...)` และ
  `contracts.embeddingProviders`
- ผู้ให้บริการเฉพาะหน่วยความจำที่มีอยู่จะยังคงทำงานต่อไปในช่วงเวลา
  การย้ายข้อมูล แต่การตรวจสอบ Plugin จะรายงานรายการนี้เป็นหนี้ด้านความเข้ากันได้สำหรับ
  Plugin ที่ไม่ได้รวมมาในชุด

### ความหมายของการตัดสินใจของ Hook

`before_install` เป็น Hook วงจรชีวิตของรันไทม์ Plugin ไม่ใช่พื้นผิวนโยบายการติดตั้ง
ของผู้ดำเนินการ ใช้ `security.installPolicy` เมื่อการตัดสินใจอนุญาต/บล็อกต้อง
ครอบคลุมเส้นทางการติดตั้งหรืออัปเดตที่รองรับโดย CLI และ Gateway

- `before_tool_call`: การคืนค่า `{ block: true }` ถือเป็นที่สิ้นสุด เมื่อ Handler ใดตั้งค่าแล้ว Handler ที่มีลำดับความสำคัญต่ำกว่าจะถูกข้าม
- `before_tool_call`: การคืนค่า `{ block: false }` ถือว่าไม่มีการตัดสินใจ (เช่นเดียวกับการละ `block`) ไม่ใช่การแทนที่
- `before_install`: การคืนค่า `{ block: true }` ถือเป็นที่สิ้นสุด เมื่อ Handler ใดตั้งค่าแล้ว Handler ที่มีลำดับความสำคัญต่ำกว่าจะถูกข้าม
- `before_install`: การคืนค่า `{ block: false }` ถือว่าไม่มีการตัดสินใจ (เช่นเดียวกับการละ `block`) ไม่ใช่การแทนที่
- `reply_dispatch`: การคืนค่า `{ handled: true, ... }` ถือเป็นที่สิ้นสุด เมื่อ Handler ใดรับการจัดส่งแล้ว Handler ที่มีลำดับความสำคัญต่ำกว่าและเส้นทางการจัดส่งโมเดลเริ่มต้นจะถูกข้าม
- `message_sending`: การคืนค่า `{ cancel: true }` ถือเป็นที่สิ้นสุด เมื่อ Handler ใดตั้งค่าแล้ว Handler ที่มีลำดับความสำคัญต่ำกว่าจะถูกข้าม
- `message_sending`: การคืนค่า `{ cancel: false }` ถือว่าไม่มีการตัดสินใจ (เช่นเดียวกับการละ `cancel`) ไม่ใช่การแทนที่
- `message_received`: ใช้ฟิลด์ `threadId` แบบมีชนิดเมื่อต้องการกำหนดเส้นทางเธรด/หัวข้อขาเข้า เก็บ `metadata` ไว้สำหรับข้อมูลเพิ่มเติมเฉพาะช่องทาง
- `message_sending`: ใช้ฟิลด์กำหนดเส้นทาง `replyToId` / `threadId` แบบมีชนิด ก่อนย้อนกลับไปใช้ `metadata` เฉพาะช่องทาง
- `gateway_start`: ใช้ `ctx.config`, `ctx.workspaceDir` และ `ctx.getCron?.()` สำหรับสถานะเริ่มต้นที่ Gateway เป็นเจ้าของ แทนการพึ่งพา Hook `gateway:startup` ภายใน ณ จุดนี้ Cron อาจยังโหลดอยู่
- `cron_reconciled`: สร้างการฉายภาพ Cron ภายนอกแบบเต็มใหม่หลังเริ่มต้นหรือโหลดตัวกำหนดตารางเวลาใหม่ โดยรวม `reason` และสถานะ `enabled` ที่มีผล รวมถึง `enabled: false` ขณะที่ `ctx.getCron?.()` คืนค่าตัวกำหนดตารางเวลาที่ปรับให้ตรงกันอย่างเจาะจง ส่ง `ctx.abortSignal` เข้าไปในงานฉายภาพแบบคงทน ซึ่งจะยกเลิกเมื่อตัวอย่างตัวกำหนดตารางเวลานั้นถูกแทนที่หรือ Gateway ปิดลง
- `cron_changed`: สังเกตการเปลี่ยนแปลงวงจรชีวิต Cron ที่ Gateway เป็นเจ้าของ เหตุการณ์ `scheduled` และ `removed` เป็นคำแนะนำการปรับให้ตรงกันหลังคอมมิต ไม่ใช่บันทึกเดลตาที่เรียงลำดับ `event.nextRunAtMs` ของเหตุการณ์ตามกำหนดการจะไม่มีอยู่เมื่องานไม่มีเวลาปลุกครั้งถัดไป ส่วนเหตุการณ์ที่ถูกนำออกยังคงมีสแนปช็อตของงานที่ถูกลบ

ตัวกำหนดเวลาปลุกภายนอกควรหน่วงหรืองบรวมเหตุการณ์ `cron_changed`
จากนั้นอ่านมุมมองคงทนแบบเต็มอีกครั้งจากตัวกำหนดตารางเวลาที่บันทึกล่าสุดโดย
`cron_reconciled` อย่านำตัวกำหนดตารางเวลาจากบริบท `cron_changed` มาใช้:
คำแนะนำที่แยกออกจากตัวกำหนดตารางเวลารุ่นเก่าอาจทับซ้อนกับการโหลดใหม่ในภายหลัง

ใช้ `cron_reconciled` เป็นตัวกระตุ้นสแนปช็อตแบบเต็มสำหรับสถานะคงทนที่โหลดเมื่อ
Gateway เริ่มต้นหรือเมื่อแทนที่ตัวกำหนดตารางเวลา ระบบจะไม่เล่นซ้ำสำหรับการโหลดใหม่แบบ hot reload
เฉพาะ Plugin Handler สำหรับการสังเกตจะทำงานแบบขนาน และการจัดส่งแบบไม่รอผล
อาจทับซ้อนกัน ดังนั้นผู้ใช้ข้อมูลต้องไม่พึ่งพาลำดับการเสร็จสิ้นของเหตุการณ์
ให้ OpenClaw เป็นแหล่งข้อมูลจริงสำหรับการตรวจสอบกำหนดเวลาและการดำเนินการ

สำหรับอะแดปเตอร์แบบ single-flight ที่มีการแทนที่อย่างคงทน การลองใหม่/ถอยระยะ และการ
ปิดระบบอย่างเรียบร้อย โปรดดู [การฉายภาพ Cron ภายนอกอย่างปลอดภัย](/th/plugins/hooks#safe-external-cron-projection)

## ข้อตกลงของโมดูลภายใน

ภายใน Plugin ให้ใช้ไฟล์ barrel ภายในเครื่องสำหรับการนำเข้าภายใน:

```text
my-plugin/
  api.ts            # การส่งออกสาธารณะสำหรับผู้ใช้ภายนอก
  runtime-api.ts    # การส่งออกสำหรับรันไทม์ภายในเท่านั้น
  index.ts          # จุดเริ่มต้นของ Plugin
  setup-entry.ts    # จุดเริ่มต้นเฉพาะการตั้งค่าที่มีขนาดเล็ก (ไม่บังคับ)
```

<Warning>
  ห้ามนำเข้า Plugin ของตนเองผ่าน `openclaw/plugin-sdk/<your-plugin>`
  จากโค้ดที่ใช้จริง กำหนดเส้นทางการนำเข้าภายในผ่าน `./api.ts` หรือ
  `./runtime-api.ts` พาธ SDK เป็นสัญญาสำหรับภายนอกเท่านั้น
</Warning>

พื้นผิวสาธารณะของ Plugin แบบรวมที่โหลดผ่าน facade (`api.ts`, `runtime-api.ts`,
`index.ts`, `setup-entry.ts` และไฟล์จุดเข้าใช้งานสาธารณะที่คล้ายกัน) จะเลือกใช้
สแนปช็อตการกำหนดค่ารันไทม์ที่ใช้งานอยู่เมื่อ OpenClaw กำลังทำงานอยู่ หากยังไม่มี
สแนปช็อตรันไทม์ ระบบจะย้อนกลับไปใช้ไฟล์การกำหนดค่าที่ผ่านการแก้ไขพาธแล้วบนดิสก์
ควรโหลด facade ของ Plugin แบบรวมในแพ็กเกจผ่านตัวโหลด facade ของ Plugin
ของ OpenClaw การนำเข้าโดยตรงจาก `dist/extensions/...` จะข้ามการตรวจสอบไฟล์ manifest
และ sidecar ของรันไทม์ ซึ่งการติดตั้งแบบแพ็กเกจใช้กับโค้ดที่ Plugin เป็นเจ้าของ

Plugin ผู้ให้บริการสามารถเปิดเผย barrel ของสัญญาเฉพาะภายใน Plugin ที่มีขอบเขตจำกัดได้ เมื่อ
ตัวช่วยนั้นมีไว้สำหรับผู้ให้บริการโดยเฉพาะอย่างตั้งใจและยังไม่ควรอยู่ในพาธย่อย SDK
แบบทั่วไป ตัวอย่างแบบรวม:

- **Anthropic**: ส่วนเชื่อมต่อสาธารณะ `api.ts` / `contract-api.ts` สำหรับตัวช่วย
  beta-header และสตรีม `service_tier` ของ Claude
- **`@openclaw/openai-provider`**: `api.ts` ส่งออกตัวสร้างผู้ให้บริการ
  ตัวช่วยโมเดลเริ่มต้น และตัวสร้างผู้ให้บริการแบบเรียลไทม์
- **`@openclaw/openrouter-provider`**: `api.ts` ส่งออกตัวสร้างผู้ให้บริการ
  พร้อมตัวช่วยการเริ่มต้นใช้งาน/การกำหนดค่า

<Warning>
  โค้ดส่วนขยายที่ใช้จริงควรหลีกเลี่ยงการนำเข้า `openclaw/plugin-sdk/<other-plugin>`
  ด้วย หากตัวช่วยใช้ร่วมกันจริง ให้ย้ายไปยังพาธย่อย SDK ที่เป็นกลาง
  เช่น `openclaw/plugin-sdk/speech`, `.../provider-model-shared` หรือพื้นผิวอื่นที่
  มุ่งเน้นความสามารถ แทนการเชื่อมโยง Plugin สองรายการเข้าด้วยกัน
</Warning>
