SDK ของ Plugin คือสัญญาที่มีการกำหนดชนิดข้อมูลระหว่าง Plugin และแกนหลัก หน้านี้เป็น
เอกสารอ้างอิงสำหรับ **สิ่งที่ต้องนำเข้า** และ **สิ่งที่คุณสามารถลงทะเบียนได้**

<Note>
  หน้านี้มีไว้สำหรับผู้เขียน Plugin ที่ใช้ `openclaw/plugin-sdk/*` ภายใน
  OpenClaw สำหรับแอปภายนอก สคริปต์ แดชบอร์ด งาน CI และส่วนขยาย IDE
  ที่ต้องการเรียกใช้เอเจนต์ผ่าน Gateway ให้ใช้
  [การผสานรวม Gateway สำหรับแอปภายนอก](/th/gateway/external-apps) แทน
</Note>

<Tip>
กำลังมองหาคู่มือวิธีใช้อยู่ใช่ไหม เริ่มต้นด้วย [การสร้าง Plugin](/th/plugins/building-plugins) ใช้ [Plugin ช่องทาง](/th/plugins/sdk-channel-plugins) สำหรับช่องทาง, [Plugin ผู้ให้บริการ](/th/plugins/sdk-provider-plugins) สำหรับผู้ให้บริการโมเดล, [Plugin แบ็กเอนด์ CLI](/th/plugins/cli-backend-plugins) สำหรับแบ็กเอนด์ AI CLI ภายในเครื่อง, [Plugin ชุดควบคุมเอเจนต์](/th/plugins/sdk-agent-harness) สำหรับตัวดำเนินการเอเจนต์แบบเนทีฟ และ [ฮุกของ Plugin](/th/plugins/hooks) สำหรับฮุกของเครื่องมือหรือวงจรชีวิต
</Tip>

## รูปแบบการนำเข้า

ให้นำเข้าจากพาธย่อยที่เฉพาะเจาะจงเสมอ:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { defineChannelPluginEntry } from "openclaw/plugin-sdk/channel-core";
```

แต่ละพาธย่อยเป็นโมดูลขนาดเล็กที่ทำงานได้ด้วยตัวเอง วิธีนี้ช่วยให้เริ่มต้นระบบได้รวดเร็วและ
ป้องกันปัญหาการพึ่งพาแบบวนซ้ำ สำหรับตัวช่วยจุดเข้าใช้งาน/การบิลด์ที่เฉพาะเจาะจงกับช่องทาง
ควรใช้ `openclaw/plugin-sdk/channel-core`; เก็บ `openclaw/plugin-sdk/core` ไว้สำหรับ
พื้นผิวครอบคลุมที่กว้างกว่าและตัวช่วยที่ใช้ร่วมกัน เช่น
`buildChannelConfigSchema`

สำหรับการกำหนดค่าช่องทาง ให้เผยแพร่ JSON Schema ที่ช่องทางเป็นเจ้าของผ่าน
`openclaw.plugin.json#channelConfigs` พาธย่อย `plugin-sdk/channel-config-schema`
ใช้สำหรับองค์ประกอบพื้นฐานของสคีมาที่ใช้ร่วมกันและตัวสร้างทั่วไป Plugin ที่รวมมากับ OpenClaw
ใช้ `plugin-sdk/bundled-channel-config-schema` สำหรับสคีมาของช่องทางที่รวมมา
และยังคงเก็บไว้ พาธย่อยสคีมาที่รวมมาดังกล่าวไม่ใช่รูปแบบสำหรับ
Plugin ใหม่

<Warning>
  อย่านำเข้าจุดเชื่อมต่ออำนวยความสะดวกที่มีแบรนด์ของผู้ให้บริการหรือช่องทาง (ตัวอย่างเช่น
  `openclaw/plugin-sdk/slack`, `.../discord`, `.../signal`, `.../whatsapp`)
  Plugin ที่รวมมาจะประกอบพาธย่อยทั่วไปของ SDK ภายใน barrel `api.ts` /
  `runtime-api.ts` ของตนเอง ผู้ใช้แกนหลักควรใช้ barrel ภายใน
  Plugin เหล่านั้น หรือเพิ่มสัญญา SDK ทั่วไปที่มีขอบเขตแคบเมื่อความต้องการนั้น
  ใช้ร่วมกันข้ามช่องทางอย่างแท้จริง

จุดเชื่อมต่อตัวช่วยของ Plugin ที่รวมมาจำนวนเล็กน้อยยังคงปรากฏในแมปการส่งออกที่สร้างขึ้น
เมื่อมีการติดตามการใช้งานโดยเจ้าของ จุดเชื่อมต่อเหล่านี้มีไว้สำหรับการบำรุงรักษา
Plugin ที่รวมมาเท่านั้น และไม่แนะนำให้เป็นพาธนำเข้าสำหรับ Plugin
ของบุคคลที่สามรายใหม่

`openclaw/plugin-sdk/discord` และ `openclaw/plugin-sdk/telegram-account`
ยังคงเก็บไว้เป็น facade ความเข้ากันได้ที่เลิกใช้แล้วสำหรับการใช้งานที่ติดตามโดยเจ้าของ อย่า
คัดลอกพาธนำเข้าเหล่านั้นไปยัง Plugin ใหม่ ให้ใช้ตัวช่วยรันไทม์ที่ฉีดเข้ามาและ
พาธย่อย SDK ของช่องทางแบบทั่วไปแทน
</Warning>

## เอกสารอ้างอิงพาธย่อย

SDK ของ Plugin เปิดให้ใช้งานเป็นชุดพาธย่อยที่มีขอบเขตแคบ โดยจัดกลุ่มตามพื้นที่ (จุดเข้าใช้งาน
Plugin, ช่องทาง, ผู้ให้บริการ, การยืนยันตัวตน, รันไทม์, ความสามารถ, หน่วยความจำ และตัวช่วย
Plugin ที่รวมมาซึ่งสงวนไว้) สำหรับรายการทั้งหมดที่จัดกลุ่มและเชื่อมโยงไว้แล้ว โปรดดู
[พาธย่อยของ SDK Plugin](/th/plugins/sdk-subpaths)

รายการจุดเข้าใช้งานของคอมไพเลอร์อยู่ใน
`scripts/lib/plugin-sdk-entrypoints.json`; การส่งออกแพ็กเกจสร้างขึ้นจาก
ชุดย่อยสาธารณะหลังจากหักพาธย่อยสำหรับการทดสอบ/ภายในเฉพาะรีโปที่ระบุไว้ใน
`scripts/lib/plugin-sdk-private-local-only-subpaths.json` เรียกใช้
`pnpm plugin-sdk:surface` เพื่อตรวจสอบจำนวนการส่งออกสาธารณะ พาธย่อยสาธารณะที่เลิกใช้แล้ว
ซึ่งเก่าพอและไม่ได้ใช้โดยโค้ดโปรดักชันของส่วนขยายที่รวมมา จะติดตามไว้ใน
`scripts/lib/plugin-sdk-deprecated-public-subpaths.json`; barrel การส่งออกซ้ำที่เลิกใช้แล้วและมีขอบเขตกว้างจะติดตามไว้ใน
`scripts/lib/plugin-sdk-deprecated-barrel-subpaths.json`

## API การลงทะเบียน

คอลแบ็ก `register(api)` จะได้รับออบเจ็กต์ `OpenClawPluginApi` ที่มีเมธอด
ต่อไปนี้:

Plugin ที่มีพื้นผิวแชตของทีมภายนอกสำหรับเซสชันสามารถลงทะเบียน
ผู้ให้บริการเดียวที่ใช้ร่วมกันทั้งกระบวนการ ซึ่งส่งออกโดย
`openclaw/plugin-sdk/session-discussion` เมธอด `info({ sessionKey })` ของผู้ให้บริการ
รายงานว่าการสนทนาไม่พร้อมใช้งาน พร้อมให้เปิด หรือเปิดอยู่แล้ว
`open({ sessionKey })` จะสร้างหรือแก้ไขการสนทนา และส่งคืน URL สำหรับฝัง
และ URL ภายนอก การลงทะเบียนผู้ให้บริการรายอื่นจะแทนที่ผู้ให้บริการปัจจุบัน

### การลงทะเบียนความสามารถ

| เมธอด                                           | สิ่งที่ลงทะเบียน                                                                 |
| ------------------------------------------------ | --------------------------------------------------------------------------------- |
| `api.registerProvider(...)`                      | การอนุมานข้อความ (LLM)                                                              |
| `api.registerWorkerProvider(...)`                | สัญญาเช่าวงจรชีวิตของผู้ปฏิบัติงานบนคลาวด์                                                     |
| `api.registerModelCatalogProvider(...)`          | แถวแค็ตตาล็อกโมเดลสำหรับการสร้างข้อความและสื่อ                                  |
| `api.registerAgentHarness(...)`                  | ตัวดำเนินการเอเจนต์แบบเนทีฟ [ทดลอง](/th/plugins/sdk-agent-harness) (Codex, Copilot) |
| `api.registerCliBackend(...)`                    | แบ็กเอนด์การอนุมาน CLI ภายในเครื่อง                                                       |
| `api.registerChannel(...)`                       | ช่องทางรับส่งข้อความ                                                                 |
| `api.registerEmbeddingProvider(...)`             | ผู้ให้บริการเวกเตอร์เอ็มเบดดิงที่นำกลับมาใช้ใหม่ได้                                                |
| `api.registerSpeechProvider(...)`                | การสังเคราะห์ข้อความเป็นเสียงพูด / STT                                                    |
| `api.registerRealtimeTranscriptionProvider(...)` | การถอดเสียงแบบเรียลไทม์ชนิดสตรีม                                                  |
| `api.registerRealtimeVoiceProvider(...)`         | เซสชันเสียงแบบเรียลไทม์สองทิศทาง                                                    |
| `api.registerMediaUnderstandingProvider(...)`    | การวิเคราะห์รูปภาพ/เสียง/วิดีโอ                                                        |
| `api.registerTranscriptSourceProvider(...)`      | แหล่งที่มาของบทถอดเสียงการประชุมสดหรือนำเข้า                                        |
| `api.registerImageGenerationProvider(...)`       | การสร้างรูปภาพ                                                                  |
| `api.registerMusicGenerationProvider(...)`       | การสร้างเพลง                                                                  |
| `api.registerVideoGenerationProvider(...)`       | การสร้างวิดีโอ                                                                  |
| `api.registerWebFetchProvider(...)`              | ผู้ให้บริการดึงข้อมูล / เก็บข้อมูลจากเว็บ                                                       |
| `api.registerWebSearchProvider(...)`             | การค้นหาเว็บ                                                                        |
| `api.registerCompactionProvider(...)`            | แบ็กเอนด์ Compaction บทถอดเสียงแบบเสียบเปลี่ยนได้                                           |

ผู้ให้บริการผู้ปฏิบัติงานต้องประกาศ ID ของตนใน `contracts.workerProviders` ด้วย
แกนหลักจะคงเจตนาที่ถาวรไว้ก่อน `provision(profile, operationId)` ผู้ให้บริการจะตรวจสอบการตั้งค่าก่อนการจัดสรรภายนอก และโยน `WorkerProviderError` เมื่อโปรไฟล์ถูกปฏิเสธอย่างถาวร `provision` ต้องรับสัญญาเช่าเดิมมาใช้เมื่อรหัสการดำเนินการซ้ำ
แกนหลักจะคงการตั้งค่าโปรไฟล์ที่ผ่านการตรวจสอบไว้พร้อมกับสัญญาเช่า และส่งสแนปช็อตดังกล่าวให้ `destroy({ leaseId, profile })` ซึ่งต้องทำซ้ำได้โดยให้ผลเหมือนเดิม และ `inspect({ leaseId, profile })` ซึ่งส่งคืน `active`, `destroyed` หรือ `unknown` วิธีนี้ช่วยให้ผู้ให้บริการกำหนดเส้นทางการเรียกวงจรชีวิตหลังจาก Gateway รีสตาร์ตหรือโปรไฟล์ที่มีชื่อถูกนำออก ปลายทาง SSH ใช้ `SecretRef` สำหรับ `keyRef` ห้ามใส่เนื้อหาคีย์ไว้ในบรรทัดโดยตรง และต้องมี `hostKey` จากเอาต์พุตการจัดสรรที่เชื่อถือได้ในรูปแบบ `algorithm base64` ทุกประการ โดยไม่มีชื่อโฮสต์หรือความคิดเห็น แกนหลักจะตรึง `hostKey` และจะไม่เชื่อถือคีย์จากการเชื่อมต่อครั้งแรก ผู้ให้บริการที่สร้าง `keyRef` แบบไดนามิกสามารถใช้งาน `resolveSshIdentity({ leaseId, profile, keyRef })` ได้ เมื่อมีตัวแก้ไขนี้ ตัวแก้ไขดังกล่าวจะเป็นแหล่งอ้างอิงหลัก ส่วนผู้ให้บริการที่ไม่มีตัวแก้ไขนี้จะใช้ตัวแก้ไขข้อมูลลับทั่วไปที่กำหนดค่าไว้
ผู้ให้บริการที่มีสัญญาเช่าต่ออายุได้สามารถใช้งาน `renew(leaseId)` ได้ด้วย
`inspect` ต้องโยนข้อผิดพลาดเมื่อเกิดความล้มเหลวชั่วคราวหรือไม่สามารถระบุได้ ให้ส่งคืน `unknown` เฉพาะเมื่อยืนยันได้อย่างแน่ชัดว่าไม่มีอยู่ แกนหลักจะทำเครื่องหมายระเบียนภายในเครื่องที่ใช้งานอยู่ว่าไม่มีเจ้าของ หรือถือว่าการไม่มีอยู่หมายถึงการรื้อถอนเสร็จสิ้นแล้วหลังจากคำขอทำลายถูกคงไว้

ผู้ให้บริการเอ็มเบดดิงที่ลงทะเบียนด้วย `api.registerEmbeddingProvider(...)` ต้อง
มีรายชื่ออยู่ใน `contracts.embeddingProviders` ในแมนิเฟสต์ของ Plugin ด้วย นี่คือ
พื้นผิวเอ็มเบดดิงทั่วไปสำหรับการสร้างเวกเตอร์ที่นำกลับมาใช้ใหม่ได้ การค้นหาหน่วยความจำ
สามารถใช้พื้นผิวผู้ให้บริการทั่วไปนี้ได้ จุดเชื่อมต่อ
`api.registerMemoryEmbeddingProvider(...)` และ
`contracts.memoryEmbeddingProviders` รุ่นเก่าเป็นความเข้ากันได้ที่เลิกใช้แล้ว ระหว่างที่
ผู้ให้บริการเฉพาะหน่วยความจำที่มีอยู่ย้ายระบบ

ผู้ให้บริการเฉพาะหน่วยความจำที่ยังเปิดเผย `batchEmbed(...)` ของรันไทม์จะคงใช้
สัญญาการแบ่งชุดต่อไฟล์ที่มีอยู่ เว้นแต่รันไทม์จะตั้งค่า
`sourceWideBatchEmbed: true` อย่างชัดเจน การเลือกใช้นี้ช่วยให้โฮสต์หน่วยความจำส่งชิ้นข้อมูลจาก
ไฟล์หน่วยความจำที่มีการแก้ไขหลายไฟล์และแหล่งที่มาที่เปิดใช้งานในการเรียก `batchEmbed(...)` ครั้งเดียว
ภายในขีดจำกัดชุดของโฮสต์ อะแดปเตอร์ชุดที่อัปโหลดไฟล์คำขอ JSONL ต้อง
แยกงานของผู้ให้บริการก่อนถึงขีดจำกัดขนาดการอัปโหลดและขีดจำกัดจำนวน
คำขอ ผู้ให้บริการต้องส่งคืนเอ็มเบดดิงหนึ่งรายการต่อชิ้นข้อมูลอินพุตในลำดับเดียวกับ
`batch.chunks` ให้ละเว้นแฟล็กนี้เมื่อผู้ให้บริการคาดหวังชุดข้อมูลเฉพาะไฟล์ หรือ
ไม่สามารถรักษาลำดับอินพุตในงานขนาดใหญ่ที่ครอบคลุมทั้งแหล่งที่มาได้

### เครื่องมือและคำสั่ง

ใช้ [`defineToolPlugin`](/th/plugins/tool-plugins) สำหรับ Plugin ที่มีเฉพาะเครื่องมืออย่างง่าย
และมีชื่อเครื่องมือคงที่ ใช้ `api.registerTool(...)` โดยตรงสำหรับ Plugin แบบผสม
หรือการลงทะเบียนเครื่องมือแบบไดนามิกเต็มรูปแบบ

| เมธอด                                 | สิ่งที่ลงทะเบียน                                                                                                                        |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `api.registerTool(tool, opts?)`        | เครื่องมือเอเจนต์ (จำเป็นหรือ `{ optional: true }`)                                                                                            |
| `api.registerCommand(def)`             | คำสั่งแบบกำหนดเอง (ข้าม LLM)                                                                                                        |
| `api.registerNodeHostCommand(command)` | คำสั่งที่จัดการโดย `openclaw node run`; เมทาดาทา `agentTool` ซึ่งเป็นทางเลือกสามารถเปิดเผยคำสั่งนี้เป็นเครื่องมือที่เอเจนต์มองเห็นได้ขณะที่ Node เชื่อมต่ออยู่ |

คำสั่งของ Plugin สามารถตั้งค่า `agentPromptGuidance` เมื่อเอเจนต์ต้องการคำใบ้สั้น ๆ
สำหรับการกำหนดเส้นทางซึ่งคำสั่งเป็นเจ้าของ ให้ข้อความนั้นกล่าวถึงตัวคำสั่งเอง อย่าเพิ่ม
นโยบายเฉพาะผู้ให้บริการหรือ Plugin ลงในตัวสร้างพรอมป์ของแกนหลัก

รายการคำแนะนำอาจเป็นสตริงแบบเดิม ซึ่งใช้กับทุกพื้นผิวพรอมป์ หรือ
รายการแบบมีโครงสร้าง:

```ts
agentPromptGuidance: [
  "คำใบ้คำสั่งส่วนกลาง",
  { text: "แสดงข้อความนี้เฉพาะในพรอมป์หลักของ OpenClaw", surfaces: ["openclaw_main"] },
];
```

`surfaces` แบบมีโครงสร้างอาจมี `openclaw_main`, `codex_app_server`,
`cli_backend`, `acp_backend` หรือ `subagent` โดย `pi_main` ยังคงเป็นนามแฝง
ที่เลิกใช้แล้วของ `openclaw_main` ให้ละเว้น `surfaces` สำหรับคำแนะนำที่ตั้งใจให้ใช้กับทุกพื้นผิว อย่า
ส่งอาร์เรย์ `surfaces` ที่ว่างเปล่า ระบบจะปฏิเสธเพื่อไม่ให้การสูญเสียขอบเขตโดยไม่ได้ตั้งใจ
กลายเป็นข้อความพรอมป์ส่วนกลาง

คำสั่งสำหรับนักพัฒนาของแอปเซิร์ฟเวอร์ Codex แบบเนทีฟเข้มงวดกว่าพื้นผิวพรอมป์
อื่น ๆ: เฉพาะคำแนะนำที่กำหนดขอบเขตเป็น `codex_app_server` อย่างชัดเจนเท่านั้นที่จะถูกเลื่อนไปยัง
ช่องทางที่มีลำดับความสำคัญสูงกว่านั้น คำแนะนำแบบสตริงเดิมและคำแนะนำแบบมีโครงสร้าง
ที่ไม่ได้กำหนดขอบเขตยังคงพร้อมใช้งานสำหรับพื้นผิวพรอมป์ที่ไม่ใช่ Codex เพื่อความเข้ากันได้

คำสั่งโฮสต์ Node ทำงานบนโฮสต์ Node ที่เชื่อมต่ออยู่ ไม่ใช่ภายในโปรเซส
Gateway หากมี `agentTool` Node จะเผยแพร่ตัวอธิบายหลังจากเชื่อมต่อ
Gateway สำเร็จ โดย Gateway จะเปิดให้การรันเอเจนต์เข้าถึงตัวอธิบายนี้เฉพาะขณะที่
Node นั้นเชื่อมต่ออยู่ และเฉพาะเมื่อ `command` ของตัวอธิบายอยู่ในพื้นผิวคำสั่ง
ที่ได้รับอนุมัติของ Node ตั้งค่า `agentTool.defaultPlatforms` เพื่อเพิ่มคำสั่ง
ที่ไม่เป็นอันตรายลงในรายการอนุญาตคำสั่ง Node เริ่มต้น มิฉะนั้นต้องใช้
`gateway.nodes.allowCommands` อย่างชัดเจนหรือนโยบายการเรียกใช้ Node ส่วน `agentTool.name`
ต้องปลอดภัยสำหรับผู้ให้บริการ: ขึ้นต้นด้วยตัวอักษร ใช้เฉพาะตัวอักษร ตัวเลข
ขีดล่าง หรือขีดกลาง และมีความยาวไม่เกิน 64 อักขระ เครื่องมือ Node ที่รองรับด้วย MCP
สามารถตั้งค่าข้อมูลเมตา `agentTool.mcp` เพื่อให้พื้นผิวแค็ตตาล็อกและการค้นหาเครื่องมือแสดง
ข้อมูลประจำตัวของเซิร์ฟเวอร์/เครื่องมือ MCP ระยะไกลได้ แต่การดำเนินการยังคงผ่าน
คำสั่ง Node ที่ประกาศไว้

#### งาน Webhook หลังการตอบรับ

เส้นทาง Webhook ที่ตอบรับคำขอก่อนการประมวลผลเสร็จสิ้นต้องย้าย
งานที่แยกออกมานั้นไปยังรูทการรับเข้าที่ติดตามแยกต่างหาก:

```typescript
import { runDetachedWebhookWork } from "openclaw/plugin-sdk/webhook-request-guards";

void runDetachedWebhookWork(() => processWebhookEvent(event)).catch((error) => {
  runtime.error?.(`webhook dispatch failed: ${String(error)}`);
});
```

เรียก `runDetachedWebhookWork(...)` แบบซิงโครนัสขณะที่คำขอ HTTP ยัง
ได้รับอนุญาตอยู่ ตัวช่วยจะสำรองรูทอิสระทันที จากนั้นเริ่ม
คอลแบ็กในไมโครทาสก์ถัดไป เพื่อให้ตัวจัดการคำขอสามารถเขียน
การตอบรับก่อนได้ Promise ที่ส่งคืนจะรับผลลัพธ์ของคอลแบ็ก ผู้เรียก
ยังคงต้องรับผิดชอบการจัดการการปฏิเสธ วิธีนี้ทำให้งานในคิวหลังการตอบรับได้รับการยอมรับ และทำให้
กระบวนการระบายงานเมื่อรีสตาร์ตหรือพักการทำงานรอให้งานดังกล่าวเสร็จ ตัวจัดการที่รอการประมวลผลทั้งหมด
ก่อนส่งคืนไม่จำเป็นต้องใช้ตัวช่วยนี้

#### การเชื่อมต่อ MCP ที่กำหนดขอบเขตตามผู้ร้องขอ

เก็บ **ข้อมูลประจำตัว** ของเซิร์ฟเวอร์ MCP (ชื่อ ตัวกรองเครื่องมือ) ให้คงที่ใน `mcp.servers` หรือ
ไฟล์แมนิเฟสต์ของบันเดิล สามารถลงทะเบียนตัวแก้ไขการเชื่อมต่อเพิ่มเติมเพื่อให้ผู้ร้องขอข้อความ
ที่เชื่อถือได้แต่ละรายได้รับทรานสปอร์ตของตนเอง:

```ts
api.registerMcpServerConnectionResolver({
  serverName: "user-email",
  resolve: async (ctx) => {
    // ctx.requesterSenderId is host-trusted; never invent sender identity here.
    const token = await lookupUserToken(ctx.requesterSenderId);
    if (!token) {
      return null; // omit this server for the current run
    }
    return {
      url: "https://mcp.example.com/email",
      headers: { Authorization: `Bearer ${token}` },
    };
  },
});
```

หมายเหตุเกี่ยวกับสัญญา:

- บริบทของตัวแก้ไขมีเฉพาะข้อมูลประจำตัวจากโฮสต์ที่เชื่อถือได้เท่านั้น (`requesterSenderId`,
  `agentAccountId` / `messageChannel` ที่เป็นตัวเลือก) สามารถเพิ่มฟิลด์ที่เชื่อถือได้ในอนาคต (เช่น
  บริบทผู้ใช้ของ Cron/เอเจนต์ย่อย) แบบเพิ่มเติมได้
- Plugin หนึ่งรายการเป็นเจ้าของชื่อเซิร์ฟเวอร์หนึ่งชื่อ: หากมี
  `registerMcpServerConnectionResolver` ซ้ำสำหรับ `serverName` เดียวกันจาก
  Plugin อื่น ระบบจะปฏิเสธพร้อมข้อความวินิจฉัยข้อผิดพลาด (การลงทะเบียนครั้งแรกมีผล) เพื่อให้
  ความเป็นเจ้าของการเชื่อมต่อไม่ขึ้นกับลำดับการโหลด Plugin
- ชื่อเครื่องมือได้มาจากชุดเซิร์ฟเวอร์ที่ประกาศทั้งหมด ดังนั้นการแก้ไขเพียงบางส่วน
  จะไม่เปลี่ยนชื่อเซิร์ฟเวอร์ที่ปลอดภัยระหว่างผู้ร้องขอหรือรอบการทำงาน Core ไม่
  ตรวจสอบว่าปลายทางของผู้ร้องขอแต่ละรายให้บริการสคีมาเครื่องมือเหมือนกันหรือไม่
  ตัวแก้ไขจึงต้องชี้ผู้ร้องขอทุกรายไปยังบริการเชิงตรรกะเดียวกัน มิฉะนั้นสคีมา
  เครื่องมือ (และความเสถียรของแคชพรอมต์) จะแตกต่างกันตามผู้ร้องขอ
- การรันที่ไม่มี `requesterSenderId` ที่เชื่อถือได้ (Cron, เอเจนต์ย่อย, Heartbeat, Gateway
  สาธารณะ) จะไม่สร้างเซิร์ฟเวอร์ที่กำหนดขอบเขตตามผู้ร้องขอ และไม่มี
  การเชื่อมต่อสำรองแบบใช้ร่วมกัน
- `resolve` ถูกจำกัดไว้ที่ 10 วินาทีต่อเซิร์ฟเวอร์ หากหมดเวลาหรือเกิดการ throw ระบบจะละเว้น
  เซิร์ฟเวอร์นั้นสำหรับการรัน โดยไม่ทำให้ MCP แบบคงที่ล้มเหลว
- การเชื่อมต่อที่แก้ไขแล้วจะได้รับการตรวจสอบซ้ำไม่เกินหนึ่งครั้งทุก 5 นาทีต่อผู้ร้องขอ:
  การหมุนเวียนจะสร้างทรานสปอร์ตใหม่ด้วยข้อมูลรับรองชุดใหม่ และผลลัพธ์ `null`
  จะเพิกถอนการเชื่อมต่อ (รันไทม์ที่แคชไว้จะถูกกำจัดแม้กำลังอยู่ระหว่างเซสชัน) ดังนั้นข้อมูลรับรอง
  ที่ถูกเพิกถอนหรือหมุนเวียนอาจยังคงถูกใช้งานได้นานถึง 5 นาที
- `headers` ที่แก้ไขแล้วจะไม่ถูกบันทึกในล็อกหรือจัดเก็บถาวร Core จะเก็บเพียงไดเจสต์
  แบบคีย์ในหน่วยความจำชั่วคราว (HMAC ภายในโปรเซส) เพื่อตรวจจับการหมุนเวียนข้อมูลรับรอง และ
  ลงทะเบียนค่าข้อมูลรับรองในส่วนหัว/URL ที่แก้ไขแล้วกับรีจิสทรีการปกปิดข้อมูล
  สำหรับล็อก/การจับข้อมูลดีบัก
- เซิร์ฟเวอร์ที่กำหนดขอบเขตตามผู้ร้องขอจะไม่สร้างมุมมอง MCP App: มุมมองมีอายุยาวกว่า
  การรันที่ยืนยันตัวตนผู้ร้องขอ และขอบเขตมุมมองของ Gateway ไม่มีข้อมูลประจำตัวผู้ร้องขอ
  ดังนั้นตัวอย่างแอปจึงยังคงปิดโดยค่าเริ่มต้นสำหรับเซิร์ฟเวอร์เหล่านี้ ผลลัพธ์เครื่องมือ
  ไม่ได้รับผลกระทบ
- เซิร์ฟเวอร์แบบคงที่ที่ไม่มีตัวแก้ไขจะยังคงใช้วงจรชีวิตที่กำหนดขอบเขตตามเซสชันเดิม
- **กฎการส่งมอบของ Harness:** เซิร์ฟเวอร์ที่กำหนดขอบเขตตามผู้ร้องขอจะไม่เข้าสู่
  การกำหนดค่าไคลเอนต์ MCP ดั้งเดิมของ Harness (เธรด Codex `mcp_servers`, CLI `-c mcp_servers=…` หรือ
  การฉายภาพ MCP ที่ใช้ร่วมกันในเซสชันแบบอื่น) แต่ Harness จะส่งมอบเซิร์ฟเวอร์เหล่านี้เป็น
  เครื่องมือที่กำหนดขอบเขตตามการรันแทน:
  - ตัวรันแบบฝัง: รันไทม์ MCP ของเซสชัน + เครื่องมือบันเดิล (แบบคงที่ + แบบกำหนดขอบเขต)
  - เซิร์ฟเวอร์แอป Codex: เครื่องมือแบบไดนามิกผ่าน
    `materializeRequesterScopedMcpToolsForHarnessRun` (เฉพาะแบบกำหนดขอบเขต โดยเซิร์ฟเวอร์แบบคงที่
    ยังคงอยู่บนไคลเอนต์ MCP ดั้งเดิมของ Codex)
- **ข้อกำหนด** ของเครื่องมือแบบกำหนดขอบเขตจะคงที่ตลอดเซสชันหลังจากแก้ไขสำเร็จครั้งแรกใน
  เซสชันนั้น เพื่อให้ Harness ที่ใช้เธรดร่วมกัน (Codex) ไม่หมุนเวียนเธรดเมื่อ
  ผู้ส่งเปลี่ยน ก่อนที่ผู้ร้องขอรายใดจะแก้ไขสำเร็จ จะไม่มีการประกาศข้อกำหนดแบบกำหนดขอบเขต
- ผู้ร้องขอที่ไม่ได้ยืนยันตัวตนบน Harness ที่ใช้เธรดร่วมกันยังคงเห็นเครื่องมือแบบกำหนดขอบเขต
  ที่ประกาศไว้ การเรียกเครื่องมือดังกล่าวจะส่งคืนข้อผิดพลาดเครื่องมือว่าไม่ได้เชื่อมต่ออย่างชัดเจนสำหรับ
  ผู้ร้องขอรายนั้น OpenClaw จะไม่ย้อนกลับไปใช้ข้อมูลรับรองของผู้ร้องขอรายอื่น

ตัวสร้างส่วนเสริมพรอมต์หน่วยความจำจะได้รับบริบท `agentId`,
`agentSessionKey` และ `sandboxed` ที่เป็นตัวเลือก การเรียกใช้ `search`
และ `get` ของส่วนเสริมคอร์ปัสหน่วยความจำจะได้รับบริบท `agentId` และ `sandboxed` ที่เป็นตัวเลือก Plugin ที่มี
พื้นที่จัดเก็บซึ่งเอเจนต์เป็นเจ้าของควรแก้ไขพื้นที่จัดเก็บนั้นสำหรับแต่ละการเรียก แทนการ
จับพาธส่วนกลางเพียงพาธเดียวระหว่างการลงทะเบียน หากจำเป็นต้องมีรหัสเอเจนต์แต่ไม่มี
ในการดำเนินการแบบหลายเอเจนต์ ให้ปิดโดยค่าเริ่มต้นแทนการเลือกเอเจนต์
ใดเอเจนต์หนึ่งโดยพลการ

ใช้ `registerMemoryPromptPreparation(...)` เมื่อข้อความพรอมต์ขึ้นกับสถานะ Plugin
แบบอะซิงโครนัส คอลแบ็กจะทำงานหนึ่งครั้งก่อนพรอมต์เอเจนต์ฉบับเต็มแต่ละครั้ง และได้รับ
บริบทเครื่องมือ เอเจนต์ เซสชัน และแซนด์บ็อกซ์เดียวกับตัวสร้างพรอมต์หน่วยความจำ
แบบซิงโครนัส ตรวจสอบอินสแตนซ์เจ้าของพื้นที่จัดเก็บปัจจุบันก่อนโหลดสถานะที่จัดเก็บถาวร
จากนั้นส่งคืนเฉพาะบรรทัดสำหรับการรันนั้น OpenClaw จะตรึงบรรทัดเหล่านั้นและ
ส่งผลลัพธ์ที่เปลี่ยนแปลงไม่ได้ไปยังการประกอบพรอมต์แบบซิงโครนัส เก็บการจัดเก็บถาวร
การแทนที่แบบอะตอมมิก และการลบเมื่อถอดเจ้าของไว้ภายใน Plugin เจ้าของ อย่า
สำรวจหรืออ่านไฟล์จากตัวสร้างพรอมต์

ตัวจัดการแบบโต้ตอบของ Telegram สามารถส่งคืน `{ submitText }` เพื่อส่งข้อความผ่าน
เส้นทางเอเจนต์ขาเข้าปกติของ Telegram หลังจากตัวจัดการทำงานสำเร็จ OpenClaw จะคง
ปุ่มคอลแบ็กไว้เมื่อนโยบายขาเข้าข้ามข้อความหรือการประมวลผลล้มเหลว เพื่อให้
ผู้ใช้ลองใหม่ได้หลังจากเงื่อนไขที่ขัดขวางเปลี่ยนแปลง ฟิลด์ผลลัพธ์นี้
เฉพาะสำหรับ Telegram ส่วนช่องทางอื่นจะคงสัญญาผลลัพธ์แบบโต้ตอบของตนเองไว้

### ฮุกโฮสต์สำหรับ Plugin เวิร์กโฟลว์

ฮุกโฮสต์คือรอยต่อ SDK สำหรับ Plugin ที่ต้องมีส่วนร่วมในวงจรชีวิตของโฮสต์
แทนการเพิ่มเฉพาะผู้ให้บริการ ช่องทาง หรือเครื่องมือ ฮุกเหล่านี้เป็น
สัญญาทั่วไป Plan Mode สามารถใช้ได้ เช่นเดียวกับเวิร์กโฟลว์การอนุมัติ
เกตนโยบายเวิร์กสเปซ ตัวตรวจสอบเบื้องหลัง วิซาร์ดการตั้งค่า และ Plugin คู่หู
สำหรับ UI

| เมธอด                                                                               | สัญญาที่เมธอดเป็นเจ้าของ                                                                                                                                           |
| ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `api.session.state.registerSessionExtension(...)`                                    | สถานะเซสชันที่ Plugin เป็นเจ้าของและเข้ากันได้กับ JSON ซึ่งฉายผ่านเซสชันของ Gateway                                                                             |
| `api.session.workflow.enqueueNextTurnInjection(...)`                                 | บริบทแบบคงทนที่รับประกันการทำงานครั้งเดียวพอดี ซึ่งแทรกในการทำงานรอบถัดไปของเอเจนต์สำหรับหนึ่งเซสชัน                                                                             |
| `api.registerTrustedToolPolicy(...)`                                                 | นโยบายเครื่องมือก่อน Plugin ที่เชื่อถือได้และควบคุมด้วย manifest ซึ่งสามารถบล็อกหรือเขียนพารามิเตอร์เครื่องมือใหม่                                                                        |
| `api.registerToolMetadata(...)`                                                      | เมทาดาทาการแสดงผลแค็ตตาล็อกเครื่องมือโดยไม่เปลี่ยนการติดตั้งใช้งานเครื่องมือ                                                                                     |
| `api.registerCommand(...)`                                                           | คำสั่ง Plugin แบบจำกัดขอบเขต ผลลัพธ์คำสั่งสามารถตั้งค่า `continueAgent: true` หรือ `suppressReply: true` ได้ คำสั่งเนทีฟของ Discord รองรับ `descriptionLocalizations` |
| `api.session.controls.registerControlUiDescriptor(...)`                              | ตัวอธิบายส่วนประกอบ Control UI สำหรับพื้นผิวเซสชัน เครื่องมือ การทำงาน การตั้งค่า หรือแท็บ                                                                      |
| `api.lifecycle.registerRuntimeLifecycle(...)`                                        | คอลแบ็กการล้างข้อมูลสำหรับทรัพยากรรันไทม์ที่ Plugin เป็นเจ้าของในเส้นทางรีเซ็ต/ลบ/โหลดใหม่                                                                          |
| `api.agent.events.registerAgentEventSubscription(...)`                               | การสมัครรับเหตุการณ์ที่ผ่านการกรองสำหรับสถานะเวิร์กโฟลว์และตัวตรวจสอบ                                                                                              |
| `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`  | สถานะชั่วคราวของ Plugin ต่อการทำงาน ซึ่งล้างเมื่อวงจรชีวิตการทำงานสิ้นสุดลง                                                                                             |
| `api.session.workflow.registerSessionSchedulerJob(...)`                              | เมทาดาทาการล้างข้อมูลสำหรับงานตัวกำหนดเวลาที่ Plugin เป็นเจ้าของ ไม่ได้กำหนดเวลางานหรือสร้างระเบียนงาน                                                            |
| `api.session.workflow.sendSessionAttachment(...)`                                    | การส่งไฟล์แนบผ่านโฮสต์สำหรับ Plugin แบบรวมมาให้เท่านั้น ไปยังเส้นทางเซสชันส่งออกโดยตรงที่ใช้งานอยู่                                                            |
| `api.session.workflow.scheduleSessionTurn(...)` / `unscheduleSessionTurnsByTag(...)` | การทำงานรอบเซสชันตามกำหนดเวลาที่ใช้ Cron พร้อมการล้างข้อมูลตามแท็ก สำหรับ Plugin แบบรวมมาให้เท่านั้น                                                                                    |
| `api.session.controls.registerSessionAction(...)`                                    | การดำเนินการเซสชันแบบมีชนิดข้อมูลที่ไคลเอนต์ส่งผ่าน Gateway ได้                                                                                             |

ตัวอธิบาย `surface: "tab"` จะเพิ่มแท็บแถบด้านข้างลงใน Control UI ตัวอธิบายแท็บของ
Plugin ที่ใช้งานอยู่จะประกาศต่อไคลเอนต์แดชบอร์ดในข้อความ hello ของ Gateway
(`controlUiTabs`) ดังนั้นแท็บจึงปรากฏเฉพาะเมื่อเปิดใช้งาน Plugin เท่านั้น
Plugin แบบรวมมาให้อาจจัดส่งมุมมองแดชบอร์ดชั้นหนึ่งสำหรับแท็บของตน ส่วน Plugin อื่น
สามารถตั้งค่า `path` เป็นเส้นทาง HTTP ของ Plugin (ดู
`api.registerHttpRoute(...)`) ซึ่งแดชบอร์ดจะแสดงผลในเฟรมแบบแซนด์บ็อกซ์
`icon` เป็นคำแนะนำชื่อไอคอนแดชบอร์ด `group` ใช้เลือกส่วนของแถบด้านข้าง
(`control` หรือ `agent`) `order` ใช้จัดลำดับระหว่างแท็บ Plugin และ `requiredScopes`
ซ่อนแท็บจากการเชื่อมต่อที่ไม่มีขอบเขตผู้ควบคุมเหล่านั้น:

สำหรับแท็บภายนอกที่ Gateway ปกป้อง ให้ลงทะเบียนตัวอธิบาย `path` ภายใต้
เส้นทาง HTTP `auth: "gateway"` ของ Plugin เดียวกัน หลังจากบูตสแตรปที่ผ่านการยืนยันตัวตน เบราว์เซอร์จะได้รับ
สิทธิ์อนุญาตแบบ HttpOnly อายุสั้นซึ่งจำกัดขอบเขตไว้ที่ Plugin และรากของเส้นทางนั้น เพื่อให้
เฟรมแบบแซนด์บ็อกซ์โหลดได้โดยไม่ต้องคัดลอก bearer token ของ Gateway ลงใน URL
หรือ JavaScript พาเรนต์ที่ผ่านการยืนยันตัวตนจะต่ออายุสิทธิ์อนุญาตขณะที่แท็บภายนอก
ทำงานอยู่ และก่อนเมานต์แท็บหลังการนำทางหรือการกลับมาทำงานของเบราว์เซอร์ นอกจากนี้ยัง
ตรวจสอบสิทธิ์อนุญาตจากแซนด์บ็อกซ์ทึบเดียวกันก่อนเมานต์ เพื่อให้โหมดความเป็นส่วนตัวของ
เบราว์เซอร์ที่บล็อกคุกกี้ปฏิเสธอย่างปลอดภัยด้วยแผงที่ไม่พร้อมใช้งาน
สิทธิ์อนุญาตของเฟรมยอมรับเฉพาะ `GET` และ `HEAD` และมี
`operator.read` เสมอ `requiredScopes` ควบคุมการมองเห็นแท็บแต่ไม่ขยาย
สิทธิ์อนุญาตของคุกกี้ การเปลี่ยนแปลงข้อมูลยังคงดำเนินการบนพื้นผิวพาเรนต์ที่ยืนยันตัวตนกับ Gateway อย่างชัดเจนหรือ
พื้นผิว bearer แท็บภายนอกต้องใช้ HTTPS/Tailscale Serve หรือ
ต้นทาง loopback ที่เบราว์เซอร์เชื่อถือ HTTP ธรรมดาบนโฮสต์ LAN จะแสดง
ข้อผิดพลาดบริบทที่ปลอดภัยแทนการเมานต์แผงที่ไม่สามารถยืนยันตัวตนได้
การบล็อกคุกกี้ของบุคคลที่สามอย่างสมบูรณ์ยังทำให้แท็บที่ Gateway ปกป้องไม่พร้อมใช้งานด้วย
เช่นเดียวกับพื้นผิว Plugin เนทีฟทั้งหมด เฟรมยังคงอยู่ภายในขอบเขตความเชื่อถือของ
Plugin ที่ติดตั้ง OpenClaw ไม่ถือว่า Plugin ที่ติดตั้งเป็นหลักความปลอดภัยของ
เบราว์เซอร์ที่แยกออกจากกัน
สิทธิ์อนุญาตคุกกี้ใช้ขอบเขตชื่อโฮสต์ของเบราว์เซอร์ ไม่ใช่ขอบเขตพอร์ต
อย่าโฮสต์บริการที่ไม่เชื่อถือซึ่งกันและกันร่วมกันบนชื่อโฮสต์ของ Gateway แม้จะใช้
พอร์ตอื่นก็ตาม
แท็บที่ใช้การยืนยันตัวตนซึ่ง Plugin จัดการจะคงพฤติกรรม iframe โดยตรง และไม่
ร้องขอหรือต้องการสิทธิ์อนุญาตจาก Gateway นี้

```typescript
api.session.controls.registerControlUiDescriptor({
  surface: "tab",
  id: "logbook",
  label: "บันทึกประจำวัน",
  description: "วันของคุณในรูปแบบไทม์ไลน์ ซึ่งสร้างจากภาพบันทึกหน้าจอ",
  icon: "sun",
  group: "control",
  requiredScopes: ["operator.write"],
});
```

`api.registerSessionExtension`, `api.enqueueNextTurnInjection`,
`api.registerControlUiDescriptor`, `api.registerRuntimeLifecycle`,
`api.registerAgentEventSubscription`, `api.emitAgentEvent`,
`api.setRunContext`, `api.getRunContext`, `api.clearRunContext`,
`api.registerSessionSchedulerJob`, `api.registerSessionAction`,

`scheduleSessionTurn(...)` เป็นส่วนอำนวยความสะดวกที่จำกัดขอบเขตตามเซสชันเหนือ
ตัวกำหนดเวลา Cron ของ Gateway โดย Cron เป็นเจ้าของการกำหนดเวลาและสร้างระเบียนงานเบื้องหลังเมื่อ
รอบการทำงานเริ่มขึ้น ส่วน Plugin SDK จำกัดเฉพาะเซสชันเป้าหมาย การตั้งชื่อที่
Plugin เป็นเจ้าของ และการล้างข้อมูล ใช้ `api.runtime.tasks.managedFlows` ภายใน
รอบการทำงานตามกำหนดเวลา เมื่อตัวงานเองต้องใช้สถานะ Task Flow หลายขั้นตอนแบบคงทน

สัญญาแยกอำนาจโดยเจตนา:

- Plugin ภายนอกสามารถเป็นเจ้าของส่วนขยายเซสชัน ตัวอธิบาย UI คำสั่ง เมทาดาทา
  เครื่องมือ การแทรกในรอบถัดไป และฮุกปกติ
- นโยบายเครื่องมือที่เชื่อถือได้ทำงานก่อนฮุก `before_tool_call` ทั่วไปและ
  ได้รับความเชื่อถือจากโฮสต์ นโยบายแบบรวมมาให้จะทำงานก่อน ส่วนนโยบายของ Plugin ที่ติดตั้งต้องมี
  การเปิดใช้งานอย่างชัดเจนพร้อมรหัสภายในของนโยบายใน
  `contracts.trustedToolPolicies` และทำงานถัดไปตามลำดับการโหลด Plugin รหัสนโยบาย
  มีขอบเขตเฉพาะ Plugin ที่ลงทะเบียน
- การเป็นเจ้าของคำสั่งที่สงวนไว้มีเฉพาะแบบรวมมาให้เท่านั้น Plugin ภายนอกควรใช้
  ชื่อคำสั่งหรือนามแฝงของตนเอง
- `allowPromptInjection=false` ปิดใช้งานฮุกที่เปลี่ยนพรอมต์ รวมถึง
  `agent_turn_prepare`, `before_prompt_build`, `heartbeat_prompt_contribution`
  และ `enqueueNextTurnInjection`

### การลงทะเบียนการค้นหา Gateway

`api.registerGatewayDiscoveryService(...)` ช่วยให้ Plugin ประกาศ Gateway ที่ใช้งานอยู่
ผ่านทรานสปอร์ตการค้นหาในเครื่อง เช่น mDNS/Bonjour OpenClaw เรียก
บริการระหว่างการเริ่มต้น Gateway เมื่อเปิดใช้งานการค้นหาในเครื่อง ส่ง
พอร์ต Gateway ปัจจุบันและข้อมูลคำแนะนำ TXT ที่ไม่เป็นความลับ และเรียกตัวจัดการ
`stop` ที่ส่งคืนมาระหว่างการปิด Gateway

```typescript
api.registerGatewayDiscoveryService({
  id: "my-discovery",
  async advertise(ctx) {
    const handle = await startMyAdvertiser({
      gatewayPort: ctx.gatewayPort,
      tls: ctx.gatewayTlsEnabled,
      displayName: ctx.machineDisplayName,
    });
    return { stop: () => handle.stop() };
  },
});
```

Plugin การค้นหา Gateway ต้องไม่ถือว่าค่า TXT ที่ประกาศเป็นความลับหรือ
การยืนยันตัวตน การค้นหาเป็นเพียงคำแนะนำในการกำหนดเส้นทาง การยืนยันตัวตนของ Gateway และการปักหมุด TLS ยังคง
เป็นเจ้าของความเชื่อถือ

### เมทาดาทาการลงทะเบียน CLI

`api.registerCli(registrar, opts?)` ยอมรับเมทาดาทาคำสั่งสองประเภท:

- `commands`: ชื่อคำสั่งที่ชัดเจนซึ่งผู้ลงทะเบียนเป็นเจ้าของ
- `descriptors`: ตัวอธิบายคำสั่งในเวลาวิเคราะห์ ใช้สำหรับความช่วยเหลือ CLI
  การกำหนดเส้นทาง และการลงทะเบียน CLI ของ Plugin แบบโหลดเมื่อใช้งาน
- `parentPath`: เส้นทางคำสั่งพาเรนต์แบบไม่บังคับสำหรับกลุ่มคำสั่งซ้อน เช่น
  `["nodes"]`

สำหรับฟีเจอร์ Node ที่จับคู่กัน ควรใช้
`api.registerNodeCliFeature(registrar, opts?)` ซึ่งเป็นตัวห่อขนาดเล็กรอบ
`api.registerCli(..., { parentPath: ["nodes"] })` และทำให้คำสั่ง เช่น
`openclaw nodes canvas` เป็นฟีเจอร์ Node ที่ Plugin เป็นเจ้าของอย่างชัดเจน

หากต้องการให้คำสั่ง Plugin ยังคงโหลดเมื่อใช้งานในเส้นทาง CLI รากตามปกติ
ให้ระบุ `descriptors` ที่ครอบคลุมรากคำสั่งระดับบนสุดทุกรายการซึ่ง
ผู้ลงทะเบียนนั้นเปิดเผย

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerMatrixCli } = await import("./src/cli.js");
    registerMatrixCli({ program });
  },
  {
    descriptors: [
      {
        name: "matrix",
        description: "จัดการบัญชี Matrix การยืนยัน อุปกรณ์ และสถานะโปรไฟล์",
        hasSubcommands: true,
      },
    ],
  },
);
```

คำสั่งที่ซ้อนกันจะได้รับคำสั่งแม่ที่ผ่านการแก้ไขแล้วเป็น `program`:

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerNodesCanvasCommands } = await import("./src/cli.js");
    registerNodesCanvasCommands(program);
  },
  {
    parentPath: ["nodes"],
    descriptors: [
      {
        name: "canvas",
        description: "จับภาพหรือเรนเดอร์เนื้อหาแคนวาสจาก Node ที่จับคู่ไว้",
        hasSubcommands: true,
      },
    ],
  },
);
```

ใช้ `commands` เพียงอย่างเดียวเฉพาะเมื่อไม่จำเป็นต้องลงทะเบียน CLI รากแบบโหลดภายหลัง
เส้นทางความเข้ากันได้แบบโหลดทันทีนั้นยังคงรองรับอยู่ แต่จะไม่ติดตั้ง
ตัวยึดตำแหน่งที่อิงตัวอธิบายสำหรับการโหลดภายหลังในเวลาพาร์ส

### การลงทะเบียนแบ็กเอนด์ CLI

`api.registerCliBackend(...)` ช่วยให้ Plugin เป็นเจ้าของการกำหนดค่าเริ่มต้นสำหรับแบ็กเอนด์
AI CLI ภายในเครื่อง เช่น `claude-cli` หรือ `my-cli`

- `id` ของแบ็กเอนด์จะกลายเป็นคำนำหน้าผู้ให้บริการในการอ้างอิงโมเดล เช่น `my-cli/gpt-5`
- `config` ของแบ็กเอนด์ใช้โครงสร้างเดียวกับ `agents.defaults.cliBackends.<id>`
- การกำหนดค่าของผู้ใช้ยังคงมีลำดับความสำคัญสูงกว่า OpenClaw จะผสาน `agents.defaults.cliBackends.<id>` ทับ
  ค่าเริ่มต้นของ Plugin ก่อนเรียกใช้ CLI
- ใช้ `normalizeConfig` เมื่อแบ็กเอนด์ต้องเขียนค่าใหม่เพื่อความเข้ากันได้หลังการผสาน
  (เช่น การปรับรูปแบบแฟล็กเก่าให้เป็นมาตรฐาน)
- ใช้ `resolveExecutionArgs` สำหรับการเขียน argv ใหม่ในขอบเขตคำขอซึ่งเป็นส่วนหนึ่งของ
  รูปแบบเฉพาะของ CLI เช่น การแมประดับการคิดของ OpenClaw ไปยังแฟล็กระดับความพยายาม
  แบบเนทีฟ ฮุกจะได้รับ `ctx.executionMode`; ใช้ `"side-question"` เพื่อเพิ่ม
  แฟล็กการแยกสภาพแวดล้อมแบบเนทีฟของแบ็กเอนด์สำหรับการเรียก `/btw` แบบชั่วคราว หากแฟล็กเหล่านั้น
  ปิดใช้งานเครื่องมือเนทีฟได้อย่างเชื่อถือได้สำหรับ CLI ที่เปิดใช้งานอยู่เสมอในกรณีอื่น ให้ประกาศ
  `sideQuestionToolMode: "disabled"` ด้วย
- ใช้ `prepareExecution` สำหรับสภาพแวดล้อมการเปิดใช้งานที่แบ็กเอนด์เป็นเจ้าของ หรือบริดจ์
  การยืนยันตัวตน/การกำหนดค่าแบบชั่วคราว `ctx.contextTokenBudget` ของมันคือขีดจำกัดโทเค็นที่มีผล
  ซึ่งเลือกไว้สำหรับการเรียกใช้ เพื่อให้แบ็กเอนด์ที่ทำ Compaction แบบเนทีฟสามารถปรับเกณฑ์
  ของตนเองให้สอดคล้องกันได้โดยไม่ต้องมีสาขาคอร์เฉพาะผู้ให้บริการ
- แบ็กเอนด์ที่สามารถปิดใช้งานเครื่องมือเนทีฟทั้งหมดสำหรับการเรียกใช้หนึ่ง ๆ อาจประกาศ
  `nativeToolMode: "selectable"` การเรียกแบบจำกัดจะส่งทูเพิล
  `ctx.toolAvailability.native` ว่างพร้อมรายการอนุญาต MCP ที่แยกจากโฮสต์อย่างแม่นยำ;
  `resolveExecutionArgs` ต้องบังคับใช้ทั้งสองอย่างกับ argv สุดท้ายสำหรับการเริ่มใหม่หรือการทำต่อ
  OpenClaw จะปิดกั้นโดยค่าเริ่มต้นหากแบ็กเอนด์ไม่สามารถทำเช่นนั้นได้

สำหรับคู่มือการเขียนแบบครบวงจร โปรดดู
[Plugin แบ็กเอนด์ CLI](/th/plugins/cli-backend-plugins)

### อะแดปเตอร์การฝังหน่วยความจำที่เลิกใช้แล้ว

| เมธอด                                         | สิ่งที่ลงทะเบียน                              |
| ---------------------------------------------- | ---------------------------------------------- |
| `api.registerMemoryEmbeddingProvider(adapter)` | อะแดปเตอร์การฝังหน่วยความจำสำหรับ Plugin ที่ใช้งานอยู่ |

- `registerMemoryCapability` คือ API เฉพาะสำหรับ Plugin หน่วยความจำ
- `registerMemoryCapability` อาจเปิดเผย `publicArtifacts.listArtifacts(...)`
  สำหรับการส่งออกที่โฮสต์จัดการด้วย Plugin คู่ที่แจกแจงอาร์ติแฟกต์ที่ประกาศไว้เหล่านั้น
  ยังคงใช้ `listActiveMemoryPublicArtifacts(...)` จากฟาซาด
  `openclaw/plugin-sdk/memory-host-core` ที่คงไว้จนกว่าจะมี API ผู้ใช้สาธารณะที่มุ่งเน้นเฉพาะด้าน;
  โดยต้องไม่เข้าถึงโครงสร้างส่วนตัวของ Plugin อื่น
- `MemoryFlushPlan.model` สามารถตรึงรอบการฟลัชไว้กับการอ้างอิง `provider/model`
  ที่แน่นอน เช่น `ollama/qwen3:8b` โดยไม่สืบทอดสายโซ่สำรองที่ใช้งานอยู่
- `registerMemoryEmbeddingProvider` เลิกใช้แล้ว ผู้ให้บริการการฝังรายใหม่
  ควรใช้ `api.registerEmbeddingProvider(...)` และ
  `contracts.embeddingProviders`
- ผู้ให้บริการเฉพาะหน่วยความจำที่มีอยู่จะยังคงทำงานต่อไปในช่วงเวลา
  การย้าย แต่การตรวจสอบ Plugin จะรายงานสิ่งนี้เป็นภาระด้านความเข้ากันได้สำหรับ
  Plugin ที่ไม่ได้รวมมาในตัว

### ความหมายของการตัดสินใจจากฮุก

`before_install` เป็นฮุกวงจรชีวิตของรันไทม์ Plugin ไม่ใช่พื้นผิวนโยบาย
การติดตั้งของผู้ดำเนินการ ใช้ `security.installPolicy` เมื่อการตัดสินใจอนุญาต/บล็อกต้อง
ครอบคลุมเส้นทางการติดตั้งหรืออัปเดตที่รองรับโดย CLI และ Gateway

- `before_tool_call`: การคืนค่า `{ block: true }` ถือเป็นจุดสิ้นสุด เมื่อแฮนด์เลอร์ใดตั้งค่าแล้ว แฮนด์เลอร์ที่มีลำดับความสำคัญต่ำกว่าจะถูกข้าม
- `before_tool_call`: การคืนค่า `{ block: false }` ถือว่าไม่มีการตัดสินใจ (เหมือนกับการละ `block`) ไม่ใช่การเขียนทับ
- `before_install`: การคืนค่า `{ block: true }` ถือเป็นจุดสิ้นสุด เมื่อแฮนด์เลอร์ใดตั้งค่าแล้ว แฮนด์เลอร์ที่มีลำดับความสำคัญต่ำกว่าจะถูกข้าม
- `before_install`: การคืนค่า `{ block: false }` ถือว่าไม่มีการตัดสินใจ (เหมือนกับการละ `block`) ไม่ใช่การเขียนทับ
- `reply_dispatch`: การคืนค่า `{ handled: true, ... }` ถือเป็นจุดสิ้นสุด เมื่อแฮนด์เลอร์ใดรับผิดชอบการส่งต่อแล้ว แฮนด์เลอร์ที่มีลำดับความสำคัญต่ำกว่าและเส้นทางการส่งต่อไปยังโมเดลเริ่มต้นจะถูกข้าม
- `message_sending`: การคืนค่า `{ cancel: true }` ถือเป็นจุดสิ้นสุด เมื่อแฮนด์เลอร์ใดตั้งค่าแล้ว แฮนด์เลอร์ที่มีลำดับความสำคัญต่ำกว่าจะถูกข้าม
- `message_sending`: การคืนค่า `{ cancel: false }` ถือว่าไม่มีการตัดสินใจ (เหมือนกับการละ `cancel`) ไม่ใช่การเขียนทับ
- `message_received`: ใช้ฟิลด์แบบระบุชนิด `threadId` เมื่อต้องกำหนดเส้นทางเธรด/หัวข้อขาเข้า เก็บ `metadata` ไว้สำหรับข้อมูลเพิ่มเติมเฉพาะช่องทาง
- `message_sending`: ใช้ฟิลด์การกำหนดเส้นทางแบบระบุชนิด `replyToId` / `threadId` ก่อนย้อนกลับไปใช้ `metadata` เฉพาะช่องทาง
- `gateway_start`: ใช้ `ctx.config`, `ctx.workspaceDir` และ `ctx.getCron?.()` สำหรับสถานะการเริ่มต้นที่ Gateway เป็นเจ้าของ แทนการพึ่งพาฮุกภายใน `gateway:startup` Cron อาจยังคงกำลังโหลดอยู่ ณ จุดนี้
- `cron_reconciled`: สร้างโปรเจกชัน Cron ภายนอกฉบับเต็มใหม่หลังการเริ่มต้นหรือการโหลดตัวจัดกำหนดการใหม่ โดยรวม `reason` และสถานะ `enabled` ที่มีผล รวมถึง `enabled: false` ขณะที่ `ctx.getCron?.()` จะคืนค่าตัวจัดกำหนดการที่กระทบยอดแล้วอย่างแม่นยำ ส่ง `ctx.abortSignal` ให้กับงานโปรเจกชันแบบคงทน; งานจะยุติเมื่อตัวจัดกำหนดการสแนปช็อตนั้นถูกแทนที่หรือ Gateway ปิดลง
- `cron_changed`: สังเกตการเปลี่ยนแปลงวงจรชีวิต Cron ที่ Gateway เป็นเจ้าของ เหตุการณ์ `scheduled` และ `removed` เป็นคำแนะนำการกระทบยอดหลังคอมมิต ไม่ใช่บันทึกเดลตาตามลำดับ `event.nextRunAtMs` ของเหตุการณ์ที่จัดกำหนดการจะไม่มีค่าเมื่องานไม่มีเวลาปลุกครั้งถัดไป; เหตุการณ์ที่ถูกนำออกยังคงมีสแนปช็อตของงานที่ลบแล้ว

ตัวจัดกำหนดการปลุกภายนอกควรหน่วงหรือรวมเหตุการณ์ `cron_changed`
จากนั้นอ่านมุมมองแบบคงทนฉบับเต็มอีกครั้งจากตัวจัดกำหนดการล่าสุดที่บันทึกไว้โดย
`cron_reconciled` ห้ามนำตัวจัดกำหนดการจากบริบท `cron_changed` มาใช้:
คำแนะนำที่แยกออกมาจากตัวจัดกำหนดการเก่าอาจทับซ้อนกับการโหลดใหม่ในภายหลัง

ใช้ `cron_reconciled` เป็นทริกเกอร์สแนปช็อตฉบับเต็มสำหรับสถานะแบบคงทนที่โหลดเมื่อ
Gateway เริ่มต้นหรือมีการแทนที่ตัวจัดกำหนดการ โดยจะไม่เล่นซ้ำสำหรับการโหลดใหม่แบบทันที
เฉพาะ Plugin แฮนด์เลอร์การสังเกตทำงานแบบขนาน และการส่งแบบไม่รอผล
อาจทับซ้อนกันได้ ดังนั้นผู้ใช้ข้อมูลต้องไม่พึ่งพาลำดับการเสร็จสิ้นของเหตุการณ์
ให้ OpenClaw เป็นแหล่งข้อมูลจริงสำหรับการตรวจสอบกำหนดเวลาและการดำเนินการ

สำหรับอะแดปเตอร์ที่ทำงานครั้งละหนึ่งรายการพร้อมการแทนที่แบบคงทน การลองใหม่/หน่วงเวลาถอยกลับ และการ
ปิดระบบอย่างเรียบร้อย โปรดดู [โปรเจกชัน Cron ภายนอกที่ปลอดภัย](/th/plugins/hooks#safe-external-cron-projection)

## แบบแผนโมดูลภายใน

ภายใน Plugin ให้ใช้ไฟล์ barrel ภายในเครื่องสำหรับการนำเข้าภายใน:

```text
my-plugin/
  api.ts            # การส่งออกสาธารณะสำหรับผู้ใช้ภายนอก
  runtime-api.ts    # การส่งออกรันไทม์สำหรับใช้ภายในเท่านั้น
  index.ts          # จุดเริ่มต้นของ Plugin
  setup-entry.ts    # จุดเริ่มต้นแบบเบาสำหรับการตั้งค่าเท่านั้น (ไม่บังคับ)
```

<Warning>
  ห้ามนำเข้า Plugin ของตนเองผ่าน `openclaw/plugin-sdk/<your-plugin>`
  จากโค้ดที่ใช้งานจริง ให้กำหนดเส้นทางการนำเข้าภายในผ่าน `./api.ts` หรือ
  `./runtime-api.ts` พาธ SDK เป็นสัญญาภายนอกเท่านั้น
</Warning>

พื้นผิวสาธารณะของ Plugin แบบบันเดิลที่โหลดผ่าน facade (`api.ts`, `runtime-api.ts`,
`index.ts`, `setup-entry.ts` และไฟล์จุดเข้าถึงสาธารณะในลักษณะเดียวกัน) จะเลือกใช้
สแนปช็อตการกำหนดค่ารันไทม์ที่ใช้งานอยู่เมื่อ OpenClaw กำลังทำงานอยู่ หากยังไม่มี
สแนปช็อตรันไทม์ ระบบจะย้อนกลับไปใช้ไฟล์การกำหนดค่าที่ผ่านการแก้ไขค่าแล้วบนดิสก์
facade ของ Plugin แบบบันเดิลที่แพ็กเกจแล้วควรโหลดผ่านตัวโหลด facade ของ Plugin
ใน OpenClaw การนำเข้าโดยตรงจาก `dist/extensions/...` จะข้ามการตรวจสอบ manifest
และ runtime sidecar ที่การติดตั้งแบบแพ็กเกจใช้กับโค้ดที่ Plugin เป็นเจ้าของ

Plugin ของผู้ให้บริการสามารถเปิดเผย barrel สัญญาเฉพาะภายใน Plugin ที่มีขอบเขตจำกัดได้ เมื่อ
ตัวช่วยนั้นเจาะจงสำหรับผู้ให้บริการโดยตั้งใจ และยังไม่ควรอยู่ในพาธย่อย SDK แบบทั่วไป
ตัวอย่างแบบบันเดิล:

- **Anthropic**: จุดเชื่อมต่อสาธารณะ `api.ts` / `contract-api.ts` สำหรับตัวช่วย
  beta-header ของ Claude และสตรีม `service_tier`
- **`@openclaw/openai-provider`**: `api.ts` ส่งออกตัวสร้างผู้ให้บริการ
  ตัวช่วยโมเดลเริ่มต้น และตัวสร้างผู้ให้บริการแบบเรียลไทม์
- **`@openclaw/openrouter-provider`**: `api.ts` ส่งออกตัวสร้างผู้ให้บริการ
  รวมถึงตัวช่วยสำหรับการเริ่มต้นใช้งาน/การกำหนดค่า

<Warning>
  โค้ดส่วนขยายสำหรับการใช้งานจริงควรหลีกเลี่ยงการนำเข้า `openclaw/plugin-sdk/<other-plugin>`
  เช่นกัน หากตัวช่วยหนึ่งใช้ร่วมกันอย่างแท้จริง ให้ยกระดับไปยังพาธย่อย SDK ที่เป็นกลาง
  เช่น `openclaw/plugin-sdk/speech`, `.../provider-model-shared` หรือพื้นผิวอื่น
  ที่จัดตามความสามารถ แทนการผูก Plugin สองตัวเข้าด้วยกัน
</Warning>
