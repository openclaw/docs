สำหรับโมเดลความสามารถสาธารณะ รูปแบบ Plugin และสัญญาการเป็นเจ้าของ/การดำเนินการ
โปรดดู [สถาปัตยกรรม Plugin](/th/plugins/architecture) หน้านี้อธิบายกลไกภายใน ได้แก่ ไปป์ไลน์การโหลด รีจิสทรี ฮุกของรันไทม์ เส้นทาง HTTP ของ Gateway
พาธการนำเข้า และตารางสคีมา

## ไปป์ไลน์การโหลด

เมื่อเริ่มต้น OpenClaw จะดำเนินการโดยคร่าว ๆ ดังนี้:

1. ค้นหารูทของ Plugin ที่อาจใช้งานได้
2. อ่านไฟล์แมนิเฟสต์ของบันเดิลแบบเนทีฟหรือที่เข้ากันได้ รวมถึงข้อมูลเมตาของแพ็กเกจ
3. ปฏิเสธรายการที่ไม่ปลอดภัย
4. ปรับการกำหนดค่า Plugin ให้เป็นรูปแบบมาตรฐาน (`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. ตัดสินใจว่าจะเปิดใช้งานแต่ละรายการหรือไม่
6. โหลดโมดูลเนทีฟที่เปิดใช้งาน: โมดูลแบบบันเดิลที่สร้างแล้วใช้ตัวโหลดเนทีฟ;
   ซอร์ส TypeScript ภายในเครื่องจากบุคคลที่สามใช้ Jiti สำรองสำหรับกรณีฉุกเฉิน
7. เรียกฮุก `register(api)` แบบเนทีฟ และรวบรวมการลงทะเบียนเข้าสู่รีจิสทรีของ Plugin
8. เปิดให้คำสั่งและส่วนต่าง ๆ ของรันไทม์เข้าถึงรีจิสทรี

<Note>
`activate` เป็นชื่อแทนแบบเดิมของ `register` — ตัวโหลดจะเลือกตัวที่มีอยู่ (`def.register ?? def.activate`) และเรียกใช้ ณ จุดเดียวกัน Plugin แบบบันเดิลทั้งหมดใช้ `register`; สำหรับ Plugin ใหม่ควรใช้ `register`
</Note>

ด่านตรวจสอบความปลอดภัยทำงาน **ก่อน** การดำเนินการของรันไทม์ การค้นหาจะบล็อกรายการ
เมื่อ:

- ตำแหน่งเริ่มต้นที่ได้รับการแก้พาธแล้วออกนอกกรอบรูทของ Plugin
- พาธ (หรือไดเรกทอรีรูท) เปิดให้ผู้ใช้ทุกคนเขียนได้
- สำหรับ Plugin ที่ไม่ใช่แบบบันเดิล เจ้าของพาธไม่ตรงกับ uid ปัจจุบัน (หรือ root)

ระบบจะพยายามซ่อมแซมไดเรกทอรีแบบบันเดิลที่ผู้ใช้ทุกคนเขียนได้ด้วย `chmod` ณ ตำแหน่งเดิม
ก่อน (การติดตั้งผ่าน npm/แบบส่วนกลางอาจให้ไดเรกทอรีแพ็กเกจมีค่าเป็น `0777`) แล้วจึงตรวจสอบ
ด่านอีกครั้ง ส่วนการตรวจสอบเจ้าของจะถูกข้ามทั้งหมดสำหรับแหล่งที่มาแบบบันเดิล

รายการที่ถูกบล็อกยังคงมีรหัส Plugin อยู่ในการวินิจฉัยที่ส่งออก หากทราบรหัสดังกล่าว
(รวมถึงรหัสที่ได้จากแมนิเฟสต์ภายในไดเรกทอรีที่ถูกปฏิเสธ
ด้วยเหตุผลอื่น) ดังนั้นการกำหนดค่าที่อ้างอิงรหัสดังกล่าวจะพบ Plugin ที่ถูกบล็อก
พร้อมคำเตือนด้านความปลอดภัยของพาธ แทนข้อผิดพลาด "Plugin ที่ไม่รู้จัก"
ซึ่งไม่เกี่ยวข้องกัน

### การทำงานที่ใช้แมนิเฟสต์ก่อน

แมนิเฟสต์คือแหล่งข้อมูลจริงของระนาบควบคุม OpenClaw ใช้แมนิเฟสต์เพื่อ:

- ระบุ Plugin
- ค้นหาช่องทาง/Skills/สคีมาการกำหนดค่าหรือความสามารถของบันเดิลที่ประกาศไว้
- ตรวจสอบความถูกต้องของ `plugins.entries.<id>.config`
- เพิ่มป้ายกำกับ/ข้อความตัวอย่างในช่องกรอกของ Control UI
- แสดงข้อมูลเมตาการติดตั้ง/แค็ตตาล็อก
- เก็บคำอธิบายการเปิดใช้งานและการตั้งค่าที่ประมวลผลได้รวดเร็วไว้โดยไม่โหลดรันไทม์ของ Plugin

สำหรับ Plugin แบบเนทีฟ โมดูลรันไทม์คือส่วนของระนาบข้อมูล โมดูลนี้ลงทะเบียน
พฤติกรรมจริง เช่น ฮุก เครื่องมือ คำสั่ง หรือโฟลว์ของผู้ให้บริการ

บล็อก `activation` และ `setup` ในแมนิเฟสต์ซึ่งเป็นทางเลือกจะคงอยู่บนระนาบควบคุม
บล็อกเหล่านี้เป็นเพียงคำอธิบายข้อมูลเมตาสำหรับการวางแผนเปิดใช้งานและการค้นหาการตั้งค่า;
ไม่ได้ใช้แทนการลงทะเบียนรันไทม์, `register(...)` หรือ `setupEntry`
ส่วนที่ใช้การเปิดใช้งานแบบสดจะใช้คำใบ้เกี่ยวกับคำสั่ง ช่องทาง และผู้ให้บริการในแมนิเฟสต์
เพื่อจำกัดขอบเขตการโหลด Plugin ก่อนสร้างรีจิสทรีในวงกว้าง:

- การโหลด CLI จำกัดเฉพาะ Plugin ที่เป็นเจ้าของคำสั่งหลักที่ร้องขอ
- การตั้งค่าช่องทาง/การแก้ไข Plugin จำกัดเฉพาะ Plugin ที่เป็นเจ้าของ
  รหัสช่องทางที่ร้องขอ
- การตั้งค่า/การแก้ไขรันไทม์ของผู้ให้บริการอย่างชัดเจนจำกัดเฉพาะ Plugin ที่เป็นเจ้าของ
  รหัสผู้ให้บริการที่ร้องขอ
- การวางแผนเริ่มต้น Gateway ใช้ `activation.onStartup` สำหรับการนำเข้าเมื่อเริ่มต้น
  ที่ระบุไว้อย่างชัดเจน; Plugin ที่ไม่มีข้อมูลเมตาการเริ่มต้นจะโหลดผ่านทริกเกอร์
  การเปิดใช้งานที่มีขอบเขตแคบกว่าเท่านั้น

ตัววางแผนการเปิดใช้งานมีทั้ง API ที่คืนเฉพาะรหัสสำหรับผู้เรียกเดิม และ
API แผนสำหรับการวินิจฉัย รายการในแผนรายงานเหตุผลที่เลือก Plugin
โดยแยกคำใบ้ `activation.*` ที่ระบุชัดเจนออกจากการสำรองด้วยความเป็นเจ้าของในแมนิเฟสต์:

| เหตุผล (จากคำใบ้ `activation.*`)   | เหตุผล (จากความเป็นเจ้าของในแมนิเฟสต์)                                                             |
| ------------------------------------ | -------------------------------------------------------------------------------------------- |
| `activation-agent-harness-hint`      | —                                                                                            |
| `activation-capability-hint`         | —                                                                                            |
| `activation-channel-hint`            | `manifest-channel-owner` (`channels`)                                                        |
| `activation-command-hint`            | `manifest-command-alias` (`commandAliases`)                                                  |
| `activation-provider-hint`           | `manifest-provider-owner` (`providers`), `manifest-setup-provider-owner` (`setup.providers`) |
| `activation-route-hint`              | —                                                                                            |
| — (ทริกเกอร์ฮุกไม่มีรูปแบบคำใบ้) | `manifest-hook-owner` (`hooks`), `manifest-tool-contract` (`contracts.tools`)                |

การแยกเหตุผลดังกล่าวคือขอบเขตความเข้ากันได้: ข้อมูลเมตาของ Plugin ที่มีอยู่
ยังคงทำงานต่อไป ขณะที่โค้ดใหม่สามารถตรวจจับคำใบ้แบบกว้างหรือพฤติกรรมสำรอง
ได้โดยไม่เปลี่ยนความหมายของการโหลดรันไทม์

การโหลดรันไทม์ล่วงหน้าขณะประมวลผลคำขอที่ขอขอบเขต `all` แบบกว้าง ยังคงคำนวณ
ชุดรหัส Plugin ที่มีผลอย่างชัดเจนจากการกำหนดค่า การวางแผนเริ่มต้น ช่องทางที่กำหนดค่าไว้
สล็อต และกฎการเปิดใช้งานอัตโนมัติ
(`resolveEffectivePluginIds` ใน `src/plugins/effective-plugin-ids.ts`) หากชุดที่คำนวณได้
ว่างเปล่า OpenClaw จะคงขอบเขตนั้นให้ว่าง แทนที่จะขยายไปยัง
Plugin ทุกตัวที่ค้นพบได้

การค้นหาการตั้งค่าจะเลือกใช้รหัสที่คำอธิบายเป็นเจ้าของ เช่น `setup.providers` และ
`setup.cliBackends` เพื่อจำกัด Plugin ที่อาจใช้งานได้ ก่อนใช้
`setup-api` เป็นทางสำรองสำหรับ Plugin ที่ยังต้องใช้ฮุกรันไทม์ขณะตั้งค่า รายการตั้งค่า
ผู้ให้บริการใช้ `providerAuthChoices` จากแมนิเฟสต์ ตัวเลือกการตั้งค่าที่ได้จากคำอธิบาย
และข้อมูลเมตาของแค็ตตาล็อกการติดตั้งโดยไม่โหลดรันไทม์ของผู้ให้บริการ ค่า
`setup.requiresRuntime: false` ที่ระบุชัดเจนเป็นจุดตัดที่ใช้เฉพาะคำอธิบาย; การละเว้น
`requiresRuntime` จะคงทางสำรอง setup-api แบบเดิมไว้เพื่อความเข้ากันได้ หาก
Plugin ที่ค้นพบมากกว่าหนึ่งตัวอ้างสิทธิ์ในผู้ให้บริการตั้งค่าหรือรหัสแบ็กเอนด์ CLI
ที่ปรับเป็นรูปแบบมาตรฐานเดียวกัน การค้นหาการตั้งค่าจะปฏิเสธเจ้าของที่กำกวม แทนการพึ่งพา
ลำดับการค้นหา เมื่อรันไทม์การตั้งค่าทำงานจริง การวินิจฉัยของรีจิสทรีจะรายงาน
ความคลาดเคลื่อนระหว่าง `setup.providers` / `setup.cliBackends` กับผู้ให้บริการหรือแบ็กเอนด์ CLI
ที่ setup-api ลงทะเบียนจริง โดยไม่บล็อก Plugin แบบเดิม

### ขอบเขตแคชของ Plugin

OpenClaw ไม่แคชผลการค้นหา Plugin หรือข้อมูลรีจิสทรีแมนิเฟสต์โดยตรง
ด้วยกรอบเวลาตามนาฬิกา การติดตั้ง การแก้ไขแมนิเฟสต์ และการเปลี่ยนแปลงพาธการโหลด
ต้องปรากฏในการอ่านข้อมูลเมตาอย่างชัดเจนหรือการสร้างสแนปช็อตใหม่ครั้งถัดไป
ตัวแยกวิเคราะห์ไฟล์แมนิเฟสต์เก็บแคชลายเซ็นไฟล์แบบจำกัดขนาด โดยใช้
พาธแมนิเฟสต์ที่เปิดร่วมกับอุปกรณ์/inode ขนาด และ mtime/ctime เป็นคีย์; แคชดังกล่าวมีไว้
เพื่อหลีกเลี่ยงการแยกวิเคราะห์ไบต์ที่ไม่เปลี่ยนแปลงซ้ำเท่านั้น และต้องไม่แคชคำตอบเกี่ยวกับการค้นหา รีจิสทรี
เจ้าของ หรือนโยบาย

เส้นทางลัดของข้อมูลเมตาที่ปลอดภัยคือการส่งต่อความเป็นเจ้าของของออบเจ็กต์อย่างชัดเจน ไม่ใช่แคชที่ซ่อนอยู่
เส้นทางด่วนขณะเริ่มต้น Gateway ควรส่ง `PluginMetadataSnapshot` ปัจจุบัน
`PluginLookUpTable` ที่คำนวณได้ หรือรีจิสทรีแมนิเฟสต์ที่ระบุชัดเจนผ่านสายการเรียกใช้
การตรวจสอบความถูกต้องของการกำหนดค่า การเปิดใช้งานอัตโนมัติเมื่อเริ่มต้น การบูตสแตรป Plugin และการเลือก
ผู้ให้บริการสามารถนำออบเจ็กต์เหล่านี้กลับมาใช้ได้ ตราบใดที่ยังเป็นตัวแทนของการกำหนดค่าและ
รายการ Plugin ปัจจุบัน การค้นหาการตั้งค่ายังคงสร้างข้อมูลเมตาของแมนิเฟสต์ใหม่เมื่อต้องการ
เว้นแต่เส้นทางการตั้งค่าเฉพาะนั้นจะได้รับรีจิสทรีแมนิเฟสต์อย่างชัดเจน; ให้คงพฤติกรรมนี้
เป็นทางสำรองของเส้นทางที่ไม่ได้ใช้บ่อย แทนการเพิ่มแคชการค้นหาที่ซ่อนอยู่ เมื่อข้อมูลนำเข้า
เปลี่ยนแปลง ให้สร้างและแทนที่สแนปช็อตใหม่ แทนการแก้ไขสแนปช็อตเดิมหรือ
เก็บสำเนาย้อนหลัง มุมมองของรีจิสทรี Plugin ที่ใช้งานอยู่และตัวช่วยบูตสแตรป
ช่องทางแบบบันเดิลควรถูกคำนวณใหม่จากรีจิสทรี/รูทปัจจุบัน
สามารถใช้แมปอายุสั้นภายในการเรียกหนึ่งครั้งเพื่อขจัดงานซ้ำหรือป้องกันการกลับเข้ามาซ้ำได้;
แต่ต้องไม่กลายเป็นแคชข้อมูลเมตาระดับโพรเซส

สำหรับการโหลด Plugin ชั้นแคชถาวรคือการโหลดรันไทม์ ซึ่งสามารถนำ
สถานะของตัวโหลดกลับมาใช้ได้เมื่อมีการโหลดโค้ดหรืออาร์ติแฟกต์ที่ติดตั้งไว้จริง เช่น:

- `PluginLoaderCacheState` และรีจิสทรีรันไทม์ที่ใช้งานอยู่ซึ่งเข้ากันได้
- แคช jiti/โมดูล และแคชตัวโหลดของพื้นผิวสาธารณะที่ใช้เพื่อหลีกเลี่ยงการนำเข้า
  พื้นผิวรันไทม์เดียวกันซ้ำ
- แคชระบบไฟล์สำหรับอาร์ติแฟกต์ของ Plugin ที่ติดตั้งแล้ว
- แมปอายุสั้นต่อการเรียกสำหรับการปรับพาธเป็นรูปแบบมาตรฐานหรือการแก้ไขรายการซ้ำ

แคชเหล่านี้เป็นรายละเอียดการใช้งานของระนาบข้อมูล และต้องไม่ตอบ
คำถามของระนาบควบคุม เช่น "Plugin ใดเป็นเจ้าของผู้ให้บริการนี้" เว้นแต่
ผู้เรียกจะขอให้โหลดรันไทม์โดยเจตนา

อย่าเพิ่มแคชถาวรหรือแคชตามกรอบเวลาของนาฬิกาสำหรับ:

- ผลการค้นหา
- รีจิสทรีแมนิเฟสต์โดยตรง
- รีจิสทรีแมนิเฟสต์ที่สร้างใหม่จากดัชนี Plugin ที่ติดตั้ง
- การค้นหาเจ้าของผู้ให้บริการ การระงับโมเดล นโยบายผู้ให้บริการ หรือข้อมูลเมตา
  ของอาร์ติแฟกต์สาธารณะ
- คำตอบอื่นใดที่ได้จากแมนิเฟสต์ ซึ่งแมนิเฟสต์ ดัชนีที่ติดตั้ง
  หรือพาธการโหลดที่เปลี่ยนแปลงควรปรากฏในการอ่านข้อมูลเมตาครั้งถัดไป

ผู้เรียกที่สร้างข้อมูลเมตาแมนิเฟสต์ใหม่จากดัชนี Plugin ที่ติดตั้งและคงอยู่ถาวร
จะสร้างรีจิสทรีดังกล่าวใหม่เมื่อต้องการ ดัชนีที่ติดตั้งเป็นสถานะถาวร
ของระนาบแหล่งข้อมูล; ไม่ใช่แคชข้อมูลเมตาภายในโพรเซสที่ซ่อนอยู่

## โมเดลรีจิสทรี

Plugin ที่โหลดแล้วจะไม่แก้ไขตัวแปรส่วนกลางแบบสุ่มของแกนระบบโดยตรง แต่จะลงทะเบียนเข้าสู่
รีจิสทรี Plugin ส่วนกลาง (`PluginRegistry` ใน `src/plugins/registry-types.ts`)
ซึ่งติดตามเรกคอร์ดของ Plugin (ข้อมูลประจำตัว แหล่งที่มา ต้นทาง สถานะ การวินิจฉัย)
รวมถึงอาร์เรย์สำหรับทุกความสามารถ ได้แก่ เครื่องมือ ฮุกแบบเดิมและฮุกที่มีชนิดข้อมูล
ช่องทาง ผู้ให้บริการ ตัวจัดการ RPC ของ Gateway เส้นทาง HTTP ตัวลงทะเบียน CLI
บริการเบื้องหลัง คำสั่งที่ Plugin เป็นเจ้าของ และกลุ่มผู้ให้บริการแบบมีชนิดข้อมูลอื่น ๆ
อีกหลายสิบกลุ่ม (เสียงพูด เอ็มเบดดิง การสร้างรูปภาพ/วิดีโอ/เพลง การดึงข้อมูล/ค้นหา
บนเว็บ ชุดเครื่องมือเอเจนต์ การดำเนินการกับเซสชัน และอื่น ๆ)

จากนั้นฟีเจอร์ของแกนระบบจะอ่านจากรีจิสทรีดังกล่าว แทนการติดต่อกับโมดูล
Plugin โดยตรง ทำให้การโหลดเป็นไปในทิศทางเดียว:

- โมดูล Plugin -> การลงทะเบียนรีจิสทรี
- รันไทม์แกนระบบ -> การใช้รีจิสทรี

การแยกส่วนนี้สำคัญต่อความสามารถในการบำรุงรักษา หมายความว่าพื้นผิวส่วนใหญ่ของแกนระบบ
ต้องมีจุดเชื่อมต่อเพียงจุดเดียวคือ "อ่านรีจิสทรี" ไม่ใช่ "จัดการแต่ละโมดูล
Plugin เป็นกรณีพิเศษ"

## คอลแบ็กการผูกบทสนทนา

Plugin ที่ผูกบทสนทนาสามารถตอบสนองเมื่อการอนุมัติได้รับการตัดสินแล้ว

ใช้ `api.onConversationBindingResolved(...)` เพื่อรับคอลแบ็กหลังคำขอผูก
ได้รับการอนุมัติหรือปฏิเสธ:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // ขณะนี้มีการผูกสำหรับ Plugin และบทสนทนานี้แล้ว
        console.log(event.binding?.conversationId);
        return;
      }

      // คำขอถูกปฏิเสธ ให้ล้างสถานะรอดำเนินการภายในเครื่อง
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

ฟิลด์ของเพย์โหลดคอลแบ็ก:

- `status`: `"approved"` หรือ `"denied"`
- `decision`: `"allow-once"`, `"allow-always"` หรือ `"deny"`
- `binding`: การผูกที่ได้รับการตัดสินแล้วสำหรับคำขอที่อนุมัติ
- `request`: สรุปคำขอเดิม คำใบ้การยกเลิกการเชื่อมต่อ รหัสผู้ส่ง และ
  ข้อมูลเมตาของบทสนทนา

คอลแบ็กนี้มีไว้สำหรับการแจ้งเตือนเท่านั้น ไม่ได้เปลี่ยนว่าใครได้รับอนุญาตให้ผูก
บทสนทนา และจะทำงานหลังจากแกนระบบจัดการการอนุมัติเสร็จสิ้น

## ฮุกรันไทม์ของผู้ให้บริการ

Plugin ผู้ให้บริการมีสามชั้น:

- **เมทาดาทาของ Manifest** สำหรับการค้นหาราคาประหยัดก่อนรันไทม์:
  `setup.providers[].envVars`, ความเข้ากันได้ที่เลิกแนะนำแล้ว `providerAuthEnvVars`,
  `providerAuthAliases`, `providerAuthChoices` และ `channelEnvVars`
- **ฮุกขณะกำหนดค่า**: `catalog` (แบบเดิม `discovery`) รวมถึง
  `applyConfigDefaults`
- **ฮุกรันไทม์**: ฮุกเสริมมากกว่า 40 รายการ ซึ่งครอบคลุมการยืนยันตัวตน การแก้ไขโมเดล
  การห่อหุ้มสตรีม ระดับการคิด นโยบายการเล่นซ้ำ และเอ็นด์พอยต์การใช้งาน ดู
  [ลำดับและการใช้งานฮุก](#hook-order-and-usage)

OpenClaw ยังคงเป็นเจ้าของลูปเอเจนต์ทั่วไป การสลับไปใช้ระบบสำรอง การจัดการทรานสคริปต์ และ
นโยบายเครื่องมือ ฮุกเหล่านี้เป็นพื้นผิวส่วนขยายสำหรับพฤติกรรมเฉพาะของผู้ให้บริการ
โดยไม่จำเป็นต้องมีทรานสปอร์ตการอนุมานแบบกำหนดเองทั้งชุด

ใช้ Manifest `setup.providers[].envVars` เมื่อผู้ให้บริการมี
ข้อมูลประจำตัวที่อิงตามตัวแปรสภาพแวดล้อม ซึ่งเส้นทางทั่วไปสำหรับการยืนยันตัวตน/สถานะ/ตัวเลือกโมเดลควรมองเห็นได้โดยไม่ต้อง
โหลดรันไทม์ของ Plugin ระบบยังคงอ่าน `providerAuthEnvVars` ที่เลิกแนะนำแล้วผ่าน
อะแดปเตอร์ความเข้ากันได้ในช่วงเวลาการเลิกใช้งาน และ Plugin ที่ไม่ได้รวมมาในชุด
ซึ่งใช้รายการนี้จะได้รับการวินิจฉัย Manifest ใช้ Manifest `providerAuthAliases`
เมื่อ ID ผู้ให้บริการหนึ่งควรใช้ตัวแปรสภาพแวดล้อม โปรไฟล์การยืนยันตัวตน
การยืนยันตัวตนที่อิงตามการกำหนดค่า และตัวเลือกการเริ่มต้นใช้งานด้วยคีย์ API ของ ID ผู้ให้บริการอื่นซ้ำ ใช้ Manifest
`providerAuthChoices` เมื่อพื้นผิว CLI สำหรับการเริ่มต้นใช้งาน/การเลือกการยืนยันตัวตนควรทราบ
ID ตัวเลือก ป้ายกำกับกลุ่ม และการเชื่อมต่อการยืนยันตัวตนแบบง่ายด้วยแฟล็กเดียวของผู้ให้บริการ โดยไม่ต้อง
โหลดรันไทม์ของผู้ให้บริการ เก็บ `envVars` ของรันไทม์ผู้ให้บริการ
ไว้สำหรับคำแนะนำที่แสดงต่อผู้ปฏิบัติงาน เช่น ป้ายกำกับการเริ่มต้นใช้งานหรือตัวแปรการตั้งค่า
ID ไคลเอ็นต์/ข้อมูลลับไคลเอ็นต์ OAuth

ใช้ Manifest `channelEnvVars` เมื่อช่องทางมีการยืนยันตัวตนหรือการตั้งค่าที่ขับเคลื่อนด้วยตัวแปรสภาพแวดล้อม ซึ่ง
กลไกสำรองตัวแปรสภาพแวดล้อมของเชลล์ทั่วไป การตรวจสอบการกำหนดค่า/สถานะ หรือพรอมต์การตั้งค่าควรมองเห็นได้
โดยไม่ต้องโหลดรันไทม์ของช่องทาง

### ลำดับและการใช้งานฮุก

สำหรับ Plugin โมเดล/ผู้ให้บริการ OpenClaw จะเรียกฮุกตามลำดับคร่าว ๆ ดังนี้
คอลัมน์ "ควรใช้เมื่อใด" เป็นคู่มือช่วยตัดสินใจอย่างรวดเร็ว
ฟิลด์ผู้ให้บริการที่มีไว้เพื่อความเข้ากันได้เท่านั้นและ OpenClaw ไม่เรียกใช้อีกต่อไป เช่น
`ProviderPlugin.capabilities` และ `suppressBuiltInModel` จะไม่แสดงไว้

หากผู้ให้บริการต้องใช้โพรโทคอลการสื่อสารแบบกำหนดเองทั้งหมดหรือตัวดำเนินการคำขอแบบกำหนดเอง นั่นถือเป็นส่วนขยายอีกประเภทหนึ่ง ฮุกเหล่านี้มีไว้สำหรับพฤติกรรมของผู้ให้บริการที่ยังคงทำงานบนลูปการอนุมานปกติของ OpenClaw

`resolveUsageAuth` จะตัดสินใจว่า OpenClaw ควรเรียก `fetchUsageSnapshot` หรือย้อนกลับไปใช้การแก้ไขข้อมูลประจำตัวแบบทั่วไปสำหรับพื้นผิวการใช้งาน/สถานะ ให้คืนค่า `{ token, accountId?, subscriptionType?, rateLimitTier? }` เมื่อผู้ให้บริการมีข้อมูลประจำตัวสำหรับการใช้งาน (ข้อมูลเมตาของแผนซึ่งเป็นทางเลือกจะส่งต่อไปยัง `fetchUsageSnapshot`) ให้คืนค่า `{ handled: true }` เมื่อการยืนยันตัวตนสำหรับการใช้งานที่ผู้ให้บริการเป็นเจ้าของได้จัดการคำขอแล้ว และต้องระงับการย้อนกลับไปใช้คีย์ API/OAuth แบบทั่วไป และให้คืนค่า `null` หรือ `undefined` เมื่อผู้ให้บริการไม่ได้จัดการการยืนยันตัวตนสำหรับการใช้งาน

ประกาศข้อมูลประจำตัวขององค์กรหรือการเรียกเก็บเงินในรายการกำกับ
`providerUsageAuthEnvVars` วิธีนี้ช่วยให้พื้นผิวการค้นหาและการล้างข้อมูลลับแบบทั่วไปจดจำข้อมูลเหล่านั้นได้ โดยไม่ทำให้ข้อมูลดังกล่าวเป็นตัวเลือกสำหรับการยืนยันตัวตนในการอนุมาน

### ตัวอย่างผู้ให้บริการ

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

## ตัวช่วยรันไทม์

Plugin สามารถเข้าถึงตัวช่วยหลักที่เลือกไว้ผ่าน `api.runtime` สำหรับ TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

หมายเหตุ:

- `textToSpeech` คืนเพย์โหลดเอาต์พุต TTS หลักตามปกติสำหรับพื้นผิวไฟล์/บันทึกเสียง
- ใช้การกำหนดค่า `messages.tts` หลักและการเลือกผู้ให้บริการ
- คืนบัฟเฟอร์เสียง PCM + อัตราการสุ่มตัวอย่าง Plugin ต้องสุ่มตัวอย่างใหม่/เข้ารหัสสำหรับผู้ให้บริการ
- `listVoices` เป็นทางเลือกสำหรับผู้ให้บริการแต่ละราย ใช้สำหรับตัวเลือกเสียงหรือขั้นตอนการตั้งค่าที่ผู้จำหน่ายเป็นเจ้าของ
- ส่วนหลักจะส่งกำหนดเวลาสิ้นสุดคำขอที่แก้ไขแล้วไปยังฮุก `listVoices` ของผู้ให้บริการ โดยการตั้งค่าการหมดเวลาเฉพาะผู้ให้บริการอาจแทนที่ค่านี้
- รายการเสียงสามารถมีข้อมูลเมตาที่ละเอียดขึ้น เช่น ภาษาและภูมิภาค เพศ และแท็กบุคลิกภาพ สำหรับตัวเลือกที่รับรู้ผู้ให้บริการ
- ปัจจุบัน OpenAI และ ElevenLabs รองรับระบบโทรศัพท์ ส่วน Microsoft ไม่รองรับ

Plugin ยังสามารถลงทะเบียนผู้ให้บริการเสียงพูดผ่าน `api.registerSpeechProvider(...)`

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

หมายเหตุ:

- เก็บนโยบาย TTS การย้อนกลับ และการส่งคำตอบไว้ในส่วนหลัก
- ใช้ผู้ให้บริการเสียงพูดสำหรับพฤติกรรมการสังเคราะห์ที่ผู้จำหน่ายเป็นเจ้าของ
- อินพุต `edge` แบบเดิมของ Microsoft จะถูกปรับให้เป็นมาตรฐานเป็นรหัสผู้ให้บริการ `microsoft`
- รูปแบบความเป็นเจ้าของที่แนะนำคือมุ่งเน้นตามบริษัท: Plugin ของผู้จำหน่ายหนึ่งตัวสามารถเป็นเจ้าของผู้ให้บริการข้อความ เสียงพูด รูปภาพ และสื่อในอนาคตได้ ขณะที่ OpenClaw เพิ่มสัญญาความสามารถเหล่านั้น

สำหรับการทำความเข้าใจรูปภาพ/เสียง/วิดีโอ Plugin จะลงทะเบียนผู้ให้บริการทำความเข้าใจสื่อแบบมีชนิดหนึ่งราย แทนถุงคีย์/ค่าแบบทั่วไป:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

หมายเหตุ:

- เก็บการประสานงาน การย้อนกลับ การกำหนดค่า และการเชื่อมต่อช่องทางไว้ในส่วนหลัก
- เก็บพฤติกรรมของผู้จำหน่ายไว้ใน Plugin ของผู้ให้บริการ
- การขยายแบบเพิ่มเติมควรยังคงมีชนิด: เมธอดทางเลือกใหม่ ฟิลด์ผลลัพธ์ทางเลือกใหม่ ความสามารถทางเลือกใหม่
- การสร้างวิดีโอใช้รูปแบบเดียวกันอยู่แล้ว:
  - ส่วนหลักเป็นเจ้าของสัญญาความสามารถและตัวช่วยรันไทม์
  - Plugin ของผู้จำหน่ายลงทะเบียน `api.registerVideoGenerationProvider(...)`
  - Plugin ของฟีเจอร์/ช่องทางใช้ `api.runtime.videoGeneration.*`

สำหรับตัวช่วยรันไทม์การทำความเข้าใจสื่อ Plugin สามารถเรียก:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});

const extraction = await api.runtime.mediaUnderstanding.extractStructuredWithModel({
  provider: "codex",
  model: "gpt-5.6-sol",
  input: [
    {
      type: "image",
      buffer: receiptImageBuffer,
      fileName: "receipt.png",
      mime: "image/png",
    },
    { type: "text", text: "Use the printed fields as the source of truth." },
  ],
  instructions: "Return entities and searchable tags.",
  schemaName: "example.evidence",
  jsonSchema: {
    type: "object",
    properties: {
      entities: { type: "array", items: { type: "string" } },
      tags: { type: "array", items: { type: "string" } },
    },
  },
  cfg: api.config,
});
```

สำหรับการถอดเสียง Plugin สามารถใช้รันไทม์การทำความเข้าใจสื่อหรือนามแฝง STT รุ่นเก่าก็ได้:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

หมายเหตุ:

- `api.runtime.mediaUnderstanding.*` เป็นพื้นผิวร่วมที่แนะนำสำหรับการทำความเข้าใจรูปภาพ/เสียง/วิดีโอ
- `extractStructuredWithModel(...)` เป็นซีมสำหรับ Plugin เพื่อการแยกข้อมูลที่ให้ความสำคัญกับรูปภาพ มีขอบเขตจำกัด และผู้ให้บริการเป็นเจ้าของ ต้องมีอินพุตรูปภาพอย่างน้อยหนึ่งรายการ ส่วนอินพุตข้อความเป็นบริบทเสริม Plugin ของผลิตภัณฑ์เป็นเจ้าของเส้นทางและสคีมาของตน ขณะที่ OpenClaw เป็นเจ้าของขอบเขตผู้ให้บริการ/รันไทม์
- ใช้การกำหนดค่าเสียงสำหรับการทำความเข้าใจสื่อหลัก (`tools.media.audio`) และลำดับการย้อนกลับของผู้ให้บริการ
- คืนค่า `{ text: undefined }` เมื่อไม่มีเอาต์พุตการถอดเสียงเกิดขึ้น (เช่น อินพุตถูกข้าม/ไม่รองรับ)
- `api.runtime.stt.transcribeAudioFile(...)` ยังคงเป็นนามแฝงเพื่อความเข้ากันได้

Plugin ยังสามารถเริ่มการทำงานของเอเจนต์ย่อยในเบื้องหลังผ่าน `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  toolsAlsoAllow: ["my_plugin_progress"],
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

หมายเหตุ:

- `provider` และ `model` เป็นการแทนที่ค่าระดับการทำงานซึ่งเป็นทางเลือก ไม่ใช่การเปลี่ยนแปลงเซสชันแบบถาวร
- `toolsAlsoAllow` ยอมรับชื่อเครื่องมือแบบตรงทั้งหมดซึ่ง Plugin ที่เรียกเป็นเจ้าของแต่เพียงผู้เดียวและได้ลงทะเบียนไว้ ชื่อของส่วนหลักและชื่อที่กำกวมจะถูกปฏิเสธ ค่านี้เป็นส่วนเพิ่มเติมจากโปรไฟล์ปกติ แต่รายการอนุญาตและรายการปฏิเสธของผู้ปฏิบัติงานยังคงมีอำนาจสูงสุด
- OpenClaw จะยอมรับฟิลด์แทนที่เหล่านั้นสำหรับผู้เรียกที่เชื่อถือได้เท่านั้น
- สำหรับการทำงานแบบย้อนกลับที่ Plugin เป็นเจ้าของ ผู้ปฏิบัติงานต้องเลือกเข้าร่วมด้วย `plugins.entries.<id>.subagent.allowModelOverride: true`
- ใช้ `plugins.entries.<id>.subagent.allowedModels` เพื่อจำกัด Plugin ที่เชื่อถือได้ไว้ที่เป้าหมาย `provider/model` มาตรฐานที่ระบุ หรือใช้ `"*"` เพื่ออนุญาตเป้าหมายใด ๆ อย่างชัดแจ้ง
- การทำงานของเอเจนต์ย่อยจาก Plugin ที่ไม่น่าเชื่อถือยังคงทำงานได้ แต่คำขอแทนที่ค่าจะถูกปฏิเสธ แทนที่จะย้อนกลับอย่างเงียบ ๆ
- เซสชันเอเจนต์ย่อยที่ Plugin สร้างจะถูกติดแท็กด้วยรหัส Plugin ที่สร้าง การย้อนกลับ `api.runtime.subagent.deleteSession(...)` อาจลบเฉพาะเซสชันที่เป็นเจ้าของเหล่านั้นเท่านั้น ส่วนการลบเซสชันใด ๆ โดยพลการยังคงต้องใช้คำขอ Gateway ที่มีขอบเขตผู้ดูแลระบบ

สำหรับการค้นหาเว็บ Plugin สามารถใช้ตัวช่วยรันไทม์ร่วมแทนการเข้าถึงการเชื่อมต่อเครื่องมือของเอเจนต์โดยตรง:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

Plugin ยังสามารถลงทะเบียนผู้ให้บริการค้นหาเว็บผ่าน
`api.registerWebSearchProvider(...)`

หมายเหตุ:

- เก็บการเลือกผู้ให้บริการ การแก้ไขข้อมูลประจำตัว และความหมายร่วมของคำขอไว้ในแกนหลัก
- ใช้ผู้ให้บริการค้นหาเว็บสำหรับทรานสปอร์ตการค้นหาเฉพาะผู้จำหน่าย
- `api.runtime.webSearch.*` เป็นพื้นผิวร่วมที่แนะนำสำหรับ Plugin ฟีเจอร์/ช่องทางที่ต้องการพฤติกรรมการค้นหาโดยไม่ขึ้นต่อแรปเปอร์เครื่องมือเอเจนต์

## เส้นทาง HTTP ของ Gateway

Plugin สามารถเปิดเผยปลายทาง HTTP ด้วย `api.registerHttpRoute(...)`

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

ฟิลด์เส้นทาง:

- `path`: พาธของเส้นทางภายใต้เซิร์ฟเวอร์ HTTP ของ Gateway
- `auth`: จำเป็นต้องระบุเป็น `"gateway"` หรือ `"plugin"` ใช้ `"gateway"` เพื่อกำหนดให้ใช้การยืนยันตัวตนตามปกติของ Gateway หรือ `"plugin"` สำหรับการยืนยันตัวตน/การตรวจสอบ Webhook ที่ Plugin จัดการ
- `match`: ไม่บังคับ เป็น `"exact"` (ค่าเริ่มต้น) หรือ `"prefix"`
- `handleUpgrade`: ตัวจัดการที่ไม่บังคับสำหรับคำขออัปเกรด WebSocket บนเส้นทางเดียวกัน
- `replaceExisting`: ไม่บังคับ อนุญาตให้ Plugin เดิมแทนที่การลงทะเบียนเส้นทางที่มีอยู่ของตนเอง
- `handler`: คืนค่า `true` เมื่อเส้นทางจัดการคำขอแล้ว

หมายเหตุ:

- `api.registerHttpHandler(...)` ถูกนำออกแล้วและจะทำให้เกิดข้อผิดพลาดขณะโหลด Plugin ให้ใช้ `api.registerHttpRoute(...)` แทน
- เส้นทางของ Plugin ต้องประกาศ `auth` อย่างชัดเจน
- ข้อขัดแย้งของ `path + match` ที่ตรงกันทุกประการจะถูกปฏิเสธ เว้นแต่ใช้ `replaceExisting: true` และ Plugin หนึ่งไม่สามารถแทนที่เส้นทางของ Plugin อื่นได้
- เส้นทางที่ทับซ้อนกันและมีระดับ `auth` ต่างกันจะถูกปฏิเสธ ให้เก็บสายโซ่การส่งต่อ `exact`/`prefix` ไว้ในระดับการยืนยันตัวตนเดียวกันเท่านั้น
- เส้นทาง `auth: "plugin"` จะ**ไม่ได้**รับขอบเขตรันไทม์ของผู้ปฏิบัติการโดยอัตโนมัติ เส้นทางเหล่านี้มีไว้สำหรับ Webhook/การตรวจสอบลายเซ็นที่ Plugin จัดการ ไม่ใช่การเรียกตัวช่วย Gateway ที่มีสิทธิ์สูง
- เส้นทาง `auth: "gateway"` ทำงานภายในขอบเขตรันไทม์ของคำขอ Gateway พื้นผิวเริ่มต้น (`gatewayRuntimeScopeSurface: "write-default"`) ถูกออกแบบให้มีข้อจำกัดอย่างรอบคอบ:
  - การยืนยันตัวตนแบบ bearer ด้วยข้อมูลลับร่วม (`gateway.auth.mode = "token"` / `"password"`) และวิธีการยืนยันตัวตนใด ๆ ที่ไม่ใช่พร็อกซีที่เชื่อถือได้ จะได้รับขอบเขต `operator.write` เพียงขอบเขตเดียว แม้ผู้เรียกจะส่ง `x-openclaw-scopes`
  - ผู้เรียก `trusted-proxy` ที่ไม่มีส่วนหัว `x-openclaw-scopes` อย่างชัดเจนจะยังคงใช้พื้นผิวแบบเดิมที่มีเฉพาะ `operator.write`
  - ผู้เรียก `trusted-proxy` ที่ส่ง `x-openclaw-scopes` จะได้รับขอบเขตที่ประกาศไว้แทน
  - เส้นทางสามารถเลือกใช้ `gatewayRuntimeScopeSurface: "trusted-operator"` เพื่อให้เคารพ `x-openclaw-scopes` เสมอสำหรับโหมดการยืนยันตัวตนที่มีข้อมูลระบุตัวตน (โดยย้อนกลับไปใช้ชุดขอบเขตเริ่มต้นทั้งหมดของ CLI เมื่อไม่มีส่วนหัว)
- แท็บ Control UI ภายนอกที่อยู่ในแซนด์บ็อกซ์และรองรับโดยเส้นทาง `auth: "gateway"` ใช้สิทธิ์อนุญาตจากคุกกี้แบบลงนามที่มีอายุสั้น ซึ่งสร้างขึ้นโดยบูตสแตรปที่ยืนยันตัวตนแล้วเท่านั้น ส่วนแท็บที่ใช้การยืนยันตัวตนของ Plugin จะยังคงใช้พาธ iframe โดยตรง ก่อนเมานต์ พาเรนต์จะเรียกโพรบที่เส้นทางเป็นเจ้าของภายในแซนด์บ็อกซ์ทึบเดียวกัน และปิดกั้นโดยค่าเริ่มต้นเมื่อนโยบายความเป็นส่วนตัวของเบราว์เซอร์บล็อกคุกกี้ สิทธิ์อนุญาตจะผูกกับ Plugin เจ้าของ รากเส้นทางที่ตรงกัน และรุ่นการยืนยันตัวตนปัจจุบัน ชื่อคุกกี้ที่สุ่มต่อโปรเซสช่วยป้องกันไม่ให้ Gateway บนโฮสต์เดียวกันที่เชื่อถือได้เขียนทับกัน แต่คุกกี้ไม่เคยแยกพอร์ต TCP ดังนั้นชื่อโฮสต์ของ Gateway จึงเป็นขอบเขตข้อมูลประจำตัวหนึ่งขอบเขต: อย่าโฮสต์บริการที่ไม่ไว้วางใจกันร่วมกันบนชื่อโฮสต์นั้น รวมถึงบนพอร์ตอื่น การจัดส่งเส้นทางจะปฏิเสธการใช้ซ้ำกับเส้นทางซ้อนที่ Plugin อื่นเป็นเจ้าของ เนื่องจากลูกหลานของแซนด์บ็อกซ์ถือเป็นข้ามไซต์สำหรับวัตถุประสงค์ของคุกกี้ สิทธิ์อนุญาตจึงยอมรับเฉพาะ `GET` และ `HEAD` ที่มี `operator.read` ส่วนการแก้ไขข้อมูลและการอัปเกรด WebSocket จะยังคงอยู่บนพื้นผิวที่ยืนยันตัวตนกับ Gateway อย่างชัดเจน คุกกี้นี้ตั้งใจไม่ใช้ CHIPS: เบราว์เซอร์ปัจจุบันรวมบิตบรรพบุรุษข้ามไซต์ไว้ในคีย์พาร์ทิชัน ดังนั้นเฟรมแซนด์บ็อกซ์ทึบที่ซ้อนกันจะสูญเสียการเข้าถึงแอสเซ็ตของเส้นทางเดียวกัน คุกกี้ต้องใช้บริบทที่ปลอดภัยและสิทธิ์ของเบราว์เซอร์สำหรับคุกกี้ข้ามไซต์ ดังนั้นแท็บภายนอกที่ใช้การยืนยันตัวตนของ Gateway จึงไม่พร้อมใช้งานบนต้นทาง LAN แบบ HTTP ธรรมดาหรือเมื่อมีการบล็อกคุกกี้บุคคลที่สามทั้งหมด ให้ใช้ HTTPS/Tailscale Serve หรือลูปแบ็กที่เบราว์เซอร์เชื่อถือพร้อมนโยบายคุกกี้ที่เข้ากันได้
- สิทธิ์อนุญาตนี้ป้องกันการเปิดเผย bearer token ของ Gateway และการใช้เส้นทาง/ขอบเขตซ้ำโดยไม่ตั้งใจ แต่ไม่ได้สร้างขอบเขตความปลอดภัยระหว่าง Plugin แบบเนทีฟ โค้ด Plugin แบบเนทีฟและเนื้อหา UI ที่โค้ดนั้นให้บริการยังคงเป็นส่วนหนึ่งของขอบเขต Plugin ภายในโปรเซสเดียวกันที่เชื่อถือได้
- กฎเชิงปฏิบัติ: อย่าถือว่าเส้นทาง Plugin ที่ใช้การยืนยันตัวตนของ Gateway เป็นพื้นผิวผู้ดูแลระบบโดยปริยาย หากเส้นทางต้องการพฤติกรรมเฉพาะผู้ดูแลระบบ ให้เลือกใช้พื้นผิวขอบเขต `trusted-operator` กำหนดให้ใช้โหมดการยืนยันตัวตนที่มีข้อมูลระบุตัวตน และจัดทำเอกสารสัญญาส่วนหัว `x-openclaw-scopes` อย่างชัดเจน
- หลังจากจับคู่เส้นทางและยืนยันตัวตนแล้ว ตัวจัดการทั่วไปจะเข้าร่วมการรับงานระดับรากของ Gateway Gateway ที่เตรียมพร้อมหรือกำลังเริ่มใหม่จะคืนค่า `503` ก่อนเรียกตัวจัดการ ข้อยกเว้นแบบจำกัดคือเส้นทาง `auth: "gateway"` ที่ได้รับสิทธิ์จากไฟล์ manifest และเลือกใช้พื้นผิว `trusted-operator` เฉพาะเส้นทางด้วย เส้นทางนี้ยังคงเข้าถึงได้เพื่อไม่ให้การจัดส่งการควบคุมการระงับติดค้าง ขณะที่เส้นทางพี่น้องทั่วไปจาก Plugin เดียวกันยังคงอยู่หลังขอบเขตการรับงาน การเป็นเจ้าของ `handleUpgrade` ของ WebSocket ใช้ขอบเขตการรับงานแบบอะตอมเดียวกัน เมื่อให้ตัวจัดการยอมรับซ็อกเก็ตแล้ว อายุการใช้งานในภายหลังของซ็อกเก็ตจะอยู่ภายใต้การเป็นเจ้าของของ Plugin และขอบเขตนี้จะไม่ติดตาม

## พาธนำเข้า Plugin SDK

ใช้พาธย่อย SDK แบบเฉพาะเจาะจงแทนแบร์เรลราก `openclaw/plugin-sdk` แบบรวมศูนย์
เมื่อสร้าง Plugin ใหม่ พาธย่อยหลัก:

| พาธย่อย                             | วัตถุประสงค์                                            |
| ----------------------------------- | -------------------------------------------------- |
| `openclaw/plugin-sdk/plugin-entry`  | พรีมิทีฟสำหรับลงทะเบียน Plugin                     |
| `openclaw/plugin-sdk/channel-core`  | ตัวช่วยสำหรับจุดเข้าใช้งาน/การบิลด์ช่องทาง                        |
| `openclaw/plugin-sdk/core`          | ตัวช่วยร่วมทั่วไปและสัญญาครอบคลุม       |
| `openclaw/plugin-sdk/config-schema` | สคีมา Zod `openclaw.json` ระดับราก (`OpenClawSchema`) |

Plugin ช่องทางเลือกใช้จากชุดรอยต่อแบบเฉพาะเจาะจง ได้แก่ `channel-setup`,
`setup-runtime`, `setup-tools`, `channel-pairing`,
`channel-contract`, `channel-feedback`, `channel-inbound`, `channel-outbound`,
`command-auth`, `secret-input`, `webhook-ingress`,
`channel-targets` และ `channel-actions` พฤติกรรมการอนุมัติควรรวมศูนย์
ไว้ในสัญญา `approvalCapability` เดียว แทนการผสมข้ามฟิลด์
Plugin ที่ไม่เกี่ยวข้อง ดู [Plugin ช่องทาง](/th/plugins/sdk-channel-plugins)

ตัวช่วยรันไทม์และการกำหนดค่าอยู่ภายใต้พาธย่อย `*-runtime` แบบเฉพาะเจาะจงที่ตรงกัน
(`approval-runtime`, `agent-runtime`, `lazy-runtime`, `directory-runtime`,
`text-runtime`, `runtime-store`, `system-event-runtime`, `heartbeat-runtime`,
`channel-activity-runtime` เป็นต้น) ให้เลือกใช้ `config-contracts`,
`plugin-config-runtime`, `runtime-config-snapshot` และ `config-mutation`
แทนแบร์เรลความเข้ากันได้ `config-runtime` แบบกว้าง

<Info>
`openclaw/plugin-sdk/channel-runtime`, `openclaw/plugin-sdk/channel-lifecycle`,
ฟาซาดตัวช่วยช่องทางขนาดเล็ก, `openclaw/plugin-sdk/outbound-runtime`,
`openclaw/plugin-sdk/outbound-send-deps`, `openclaw/plugin-sdk/config-runtime`
และ `openclaw/plugin-sdk/infra-runtime` เป็นชิมความเข้ากันได้ที่เลิกใช้แล้วสำหรับ
Plugin รุ่นเก่า โค้ดใหม่ควรนำเข้าพรีมิทีฟทั่วไปที่เฉพาะเจาะจงกว่าแทน
</Info>

จุดเข้าใช้งานภายในรีโพ (ต่อรากแพ็กเกจ Plugin ที่บันเดิลมา):

- `index.js` — จุดเข้าใช้งาน Plugin ที่บันเดิลมา
- `api.js` — แบร์เรลตัวช่วย/ชนิดข้อมูล
- `runtime-api.js` — แบร์เรลเฉพาะรันไทม์
- `setup-entry.js` — จุดเข้าใช้งาน Plugin สำหรับการตั้งค่า

Plugin ภายนอกควรนำเข้าเฉพาะพาธย่อย `openclaw/plugin-sdk/*` เท่านั้น ห้าม
นำเข้า `src/*` ของแพ็กเกจ Plugin อื่นจากแกนหลักหรือจาก Plugin อื่น
จุดเข้าใช้งานที่โหลดผ่านฟาซาดจะเลือกใช้สแนปช็อตการกำหนดค่ารันไทม์ที่ใช้งานอยู่เมื่อมี
จากนั้นจึงย้อนกลับไปใช้ไฟล์การกำหนดค่าที่แก้ไขตำแหน่งแล้วบนดิสก์

พาธย่อยเฉพาะความสามารถ เช่น `image-generation`, `media-understanding`
และ `speech` มีอยู่เนื่องจาก Plugin ที่บันเดิลมาใช้งานในปัจจุบัน แต่ไม่ได้
เป็นสัญญาภายนอกระยะยาวที่ถูกตรึงโดยอัตโนมัติ — ให้ตรวจสอบหน้าอ้างอิง SDK
ที่เกี่ยวข้องเมื่อต้องพึ่งพาพาธเหล่านี้

## สคีมาเครื่องมือข้อความ

Plugin ควรเป็นเจ้าของส่วนสนับสนุนสคีมา `describeMessageTool(...)`
เฉพาะช่องทางสำหรับพรีมิทีฟที่ไม่ใช่ข้อความ เช่น ปฏิกิริยา การอ่าน และแบบสำรวจ
การนำเสนอการส่งแบบร่วมควรใช้สัญญา `MessagePresentation` ทั่วไป
แทนฟิลด์ปุ่ม คอมโพเนนต์ บล็อก หรือการ์ดแบบเนทีฟของผู้ให้บริการ
ดูสัญญา กฎการย้อนกลับ การแมปผู้ให้บริการ และรายการตรวจสอบผู้สร้าง Plugin
ได้ที่ [การนำเสนอข้อความ](/th/plugins/message-presentation)

Plugin ที่ส่งได้จะประกาศสิ่งที่สามารถเรนเดอร์ผ่านความสามารถของข้อความ:

- `presentation` สำหรับบล็อกการนำเสนอเชิงความหมาย (`text`, `context`,
  `divider`, `chart`, `table`, `buttons`, `select`)
- `delivery-pin` สำหรับคำขอจัดส่งแบบปักหมุด

แกนหลักจะตัดสินใจว่าจะเรนเดอร์การนำเสนอแบบเนทีฟหรือลดระดับเป็นข้อความ
อย่าเปิดเผยช่องทางเลี่ยง UI แบบเนทีฟของผู้ให้บริการจากเครื่องมือข้อความทั่วไป
ตัวช่วย SDK ที่เลิกใช้แล้วสำหรับสคีมาเนทีฟแบบเดิมยังคงส่งออกไว้สำหรับ
Plugin บุคคลที่สามที่มีอยู่ แต่ Plugin ใหม่ไม่ควรใช้

## การแก้ไขเป้าหมายช่องทาง

Plugin ช่องทางควรเป็นเจ้าของความหมายของเป้าหมายเฉพาะช่องทาง ให้โฮสต์
ขาออกร่วมคงความเป็นทั่วไป และใช้พื้นผิวอะแดปเตอร์การรับส่งข้อความสำหรับกฎของผู้ให้บริการ:

- `messaging.inferTargetChatType({ to })` ตัดสินใจว่าควรถือว่าเป้าหมายที่ปรับเป็นมาตรฐานแล้วเป็น
  `direct`, `group` หรือ `channel` ก่อนค้นหาไดเรกทอรี
- `messaging.targetResolver.looksLikeId(raw, normalized)` แจ้งแกนหลักว่า
  อินพุตควรข้ามไปยังการแก้ไขแบบคล้ายรหัสโดยตรงแทนการค้นหาไดเรกทอรีหรือไม่
- `messaging.targetResolver.reservedLiterals` แสดงรายการคำเดี่ยวที่เป็น
  การอ้างอิงช่องทาง/เซสชันสำหรับผู้ให้บริการนั้น การแก้ไขจะคงรายการไดเรกทอรีที่กำหนดค่าไว้
  ก่อนปฏิเสธลิเทอรัลที่สงวนไว้ จากนั้นจะปิดกั้นโดยค่าเริ่มต้นเมื่อค้นหาในไดเรกทอรีไม่พบ
- `messaging.targetResolver.resolveTarget(...)` เป็นทางเลือกย้อนกลับของ Plugin เมื่อ
  แกนหลักต้องการการแก้ไขขั้นสุดท้ายที่ผู้ให้บริการเป็นเจ้าของ หลังการปรับเป็นมาตรฐานหรือ
  หลังค้นหาในไดเรกทอรีไม่พบ
- `messaging.resolveOutboundSessionRoute(...)` เป็นเจ้าของการสร้างเส้นทางเซสชัน
  เฉพาะผู้ให้บริการเมื่อแก้ไขเป้าหมายแล้ว

การแบ่งที่แนะนำ:

- ใช้ `inferTargetChatType` สำหรับการตัดสินใจด้านหมวดหมู่ที่ควรเกิดขึ้นก่อน
  ค้นหาเพียร์/กลุ่ม
- ใช้ `looksLikeId` สำหรับการตรวจสอบว่า “ให้ถือสิ่งนี้เป็นรหัสเป้าหมายแบบชัดแจ้ง/เนทีฟ”
- ใช้ `resolveTarget` สำหรับทางเลือกย้อนกลับของการปรับเป็นมาตรฐานเฉพาะผู้ให้บริการ ไม่ใช่สำหรับ
  การค้นหาไดเรกทอรีแบบกว้าง
- เก็บรหัสแบบเนทีฟของผู้ให้บริการ เช่น รหัสแชต รหัสเธรด JID แฮนเดิล และรหัสห้อง
  ไว้ภายในค่า `target` หรือพารามิเตอร์เฉพาะผู้ให้บริการ ไม่ใช่ในฟิลด์ SDK
  ทั่วไป

## ไดเรกทอรีที่รองรับด้วยการกำหนดค่า

Plugin ที่สร้างรายการไดเรกทอรีจากการกำหนดค่าควรเก็บตรรกะนั้นไว้ใน
Plugin และใช้ตัวช่วยร่วมจาก
`openclaw/plugin-sdk/directory-runtime` ซ้ำ

ใช้สิ่งนี้เมื่อช่องทางต้องการเพียร์/กลุ่มที่รองรับด้วยการกำหนดค่า เช่น:

- คู่สนทนา DM ที่ควบคุมด้วยรายการอนุญาต
- การแมปช่อง/กลุ่มที่กำหนดค่าไว้
- ทางเลือกสำรองของไดเรกทอรีแบบคงที่ที่จำกัดขอบเขตตามบัญชี

ตัวช่วยที่ใช้ร่วมกันใน `directory-runtime` จัดการเฉพาะการดำเนินการทั่วไป:

- การกรองคิวรี
- การใช้ขีดจำกัด
- ตัวช่วยขจัดรายการซ้ำ/ปรับให้อยู่ในรูปแบบมาตรฐาน
- การสร้าง `ChannelDirectoryEntry[]`

การตรวจสอบบัญชีและการปรับ ID ให้อยู่ในรูปแบบมาตรฐานที่เฉพาะเจาะจงกับช่องควรอยู่ในการติดตั้งใช้งาน
Plugin

## แค็ตตาล็อกผู้ให้บริการ

Plugin ผู้ให้บริการสามารถกำหนดแค็ตตาล็อกโมเดลสำหรับการอนุมานด้วย
`registerProvider({ catalog: { run(...) { ... } } })`

`catalog.run(...)` คืนค่าในรูปแบบเดียวกับที่ OpenClaw เขียนลงใน
`models.providers`:

- `{ provider }` สำหรับรายการผู้ให้บริการหนึ่งรายการ
- `{ providers }` สำหรับรายการผู้ให้บริการหลายรายการ

ใช้ `catalog` เมื่อ Plugin เป็นเจ้าของ ID โมเดล, URL ฐาน
เริ่มต้น หรือเมทาดาทาของโมเดลที่ควบคุมด้วยการตรวจสอบสิทธิ์ซึ่งเฉพาะเจาะจงกับผู้ให้บริการ

`catalog.order` ควบคุมว่าแค็ตตาล็อกของ Plugin จะผสานเมื่อใดเมื่อเทียบกับผู้ให้บริการโดยนัย
ที่มีมาให้ใน OpenClaw:

- `simple`: ผู้ให้บริการที่ใช้คีย์ API โดยตรงหรือขับเคลื่อนด้วยสภาพแวดล้อม
- `profile`: ผู้ให้บริการที่ปรากฏเมื่อมีโปรไฟล์การตรวจสอบสิทธิ์
- `paired`: ผู้ให้บริการที่สังเคราะห์รายการผู้ให้บริการที่เกี่ยวข้องกันหลายรายการ
- `late`: รอบสุดท้าย หลังจากผู้ให้บริการโดยนัยอื่นๆ

ผู้ให้บริการที่มาภายหลังจะมีผลเหนือกว่าเมื่อคีย์ชนกัน ดังนั้น Plugin จึงสามารถแทนที่
รายการผู้ให้บริการที่มีมาให้ซึ่งใช้ ID ผู้ให้บริการเดียวกันโดยเจตนาได้

Plugin ยังสามารถเผยแพร่แถวโมเดลแบบอ่านอย่างเดียวผ่าน
`api.registerModelCatalogProvider({ provider, kinds, staticCatalog, liveCatalog
})` ได้ด้วย นี่คือแนวทางในอนาคตสำหรับพื้นผิวรายการ/ความช่วยเหลือ/ตัวเลือก และรองรับแถว
`text`, `voice`, `image_generation`, `video_generation` และ `music_generation`
Plugin ผู้ให้บริการยังคงเป็นเจ้าของการเรียกปลายทางจริง การแลกเปลี่ยนโทเค็น และ
การแมปการตอบกลับของผู้จำหน่าย ส่วนคอร์เป็นเจ้าของรูปแบบแถวร่วม ป้ายกำกับแหล่งที่มา และ
การจัดรูปแบบความช่วยเหลือสำหรับเครื่องมือสื่อ การลงทะเบียนผู้ให้บริการสร้างสื่อจะสังเคราะห์
แถวแค็ตตาล็อกแบบคงที่โดยอัตโนมัติจาก `defaultModel`, `models` และ
`capabilities`

ความเข้ากันได้:

- `discovery` ยังคงใช้งานได้ในฐานะนามแฝงแบบเดิม แต่จะแสดงคำเตือนการเลิกใช้
- หากลงทะเบียนทั้ง `catalog` และ `discovery` OpenClaw จะใช้ `catalog`
  และแสดงคำเตือน
- `augmentModelCatalog` เลิกใช้แล้ว ผู้ให้บริการที่รวมมาให้ควรเผยแพร่
  แถวเสริมผ่าน `registerModelCatalogProvider`

## การตรวจสอบช่องแบบอ่านอย่างเดียว

หาก Plugin ลงทะเบียนช่อง ควรติดตั้งใช้งาน
`plugin.config.inspectAccount(cfg, accountId)` ควบคู่กับ `resolveAccount(...)`

เหตุผล:

- `resolveAccount(...)` คือเส้นทางรันไทม์ เส้นทางนี้สามารถถือว่าข้อมูลรับรอง
  ถูกจัดเตรียมไว้อย่างสมบูรณ์แล้ว และสามารถล้มเหลวทันทีเมื่อไม่มีข้อมูลลับที่จำเป็น
- เส้นทางคำสั่งแบบอ่านอย่างเดียว เช่น `openclaw status`, `openclaw status --all`,
  `openclaw channels status`, `openclaw channels resolve` และโฟลว์ซ่อมแซม doctor/config
  ไม่ควรต้องจัดเตรียมข้อมูลรับรองสำหรับรันไทม์เพียงเพื่อ
  อธิบายการกำหนดค่า

พฤติกรรม `inspectAccount(...)` ที่แนะนำ:

- คืนค่าเฉพาะสถานะบัญชีเชิงอธิบาย
- คง `enabled` และ `configured` ไว้
- รวมฟิลด์แหล่งที่มา/สถานะของข้อมูลรับรองเมื่อเกี่ยวข้อง เช่น:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- ไม่จำเป็นต้องคืนค่าโทเค็นดิบเพียงเพื่อรายงานความพร้อมใช้งาน
  แบบอ่านอย่างเดียว การคืนค่า `tokenStatus: "available"` (และฟิลด์แหล่งที่มา
  ที่ตรงกัน) ก็เพียงพอสำหรับคำสั่งลักษณะรายงานสถานะ
- ใช้ `configured_unavailable` เมื่อกำหนดค่าข้อมูลรับรองผ่าน SecretRef แต่
  ไม่พร้อมใช้งานในเส้นทางคำสั่งปัจจุบัน

วิธีนี้ช่วยให้คำสั่งแบบอ่านอย่างเดียวรายงานว่า "กำหนดค่าแล้วแต่ไม่พร้อมใช้งานในเส้นทางคำสั่งนี้"
แทนที่จะขัดข้องหรือรายงานผิดว่าบัญชียังไม่ได้กำหนดค่า

## แพ็กแพ็กเกจ

ไดเรกทอรี Plugin อาจมี `package.json` ที่ประกอบด้วย `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

แต่ละรายการจะกลายเป็น Plugin หากแพ็กระบุส่วนขยายหลายรายการ ID ของ Plugin
จะเป็น `<manifestOrPackageName>/<fileBase>` (ID จากแมนิเฟสต์มีผลเหนือกว่าเมื่อ
มีอยู่ มิฉะนั้นจะใช้ชื่อ `package.json` ที่ไม่มีขอบเขต)

หาก Plugin นำเข้าดีเพนเดนซี npm ให้ติดตั้งในไดเรกทอรีนั้นเพื่อให้
`node_modules` พร้อมใช้งาน (`npm install` / `pnpm install`)

ข้อจำกัดด้านความปลอดภัย: ทุกรายการ `openclaw.extensions` ต้องอยู่ภายในไดเรกทอรี Plugin
หลังจากแก้ไข symlink แล้ว รายการที่ออกนอกไดเรกทอรีแพ็กเกจจะ
ถูกปฏิเสธ

หมายเหตุด้านความปลอดภัย: `openclaw plugins install` ติดตั้งดีเพนเดนซีของ Plugin ด้วย
`npm install --omit=dev --ignore-scripts` ภายในโปรเจกต์ (ไม่มีสคริปต์วงจรชีวิต
และไม่มีดีเพนเดนซีสำหรับการพัฒนาในรันไทม์) โดยไม่สนใจการตั้งค่าการติดตั้ง npm ส่วนกลางที่สืบทอดมา
รักษาโครงสร้างดีเพนเดนซีของ Plugin ให้เป็น "JS/TS ล้วน" และหลีกเลี่ยงแพ็กเกจที่ต้องใช้
บิลด์ `postinstall`

ไม่บังคับ: `openclaw.setupEntry` สามารถชี้ไปยังโมดูลขนาดเล็กที่ใช้สำหรับการตั้งค่าเท่านั้น
เมื่อ OpenClaw ต้องใช้พื้นผิวการตั้งค่าสำหรับ Plugin ช่องที่ปิดใช้งาน หรือ
เมื่อเปิดใช้ Plugin ช่องแล้วแต่ยังไม่ได้กำหนดค่า ระบบจะโหลด `setupEntry`
แทนรายการหลักของ Plugin ซึ่งช่วยให้การเริ่มต้นและการตั้งค่าเบาลง
เมื่อรายการหลักของ Plugin เชื่อมต่อเครื่องมือ ฮุก หรือโค้ดอื่นๆ
ที่ใช้เฉพาะรันไทม์ด้วย

ไม่บังคับ: `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
สามารถเลือกให้ Plugin ช่องเข้าสู่เส้นทาง `setupEntry` เดียวกันในช่วงเริ่มต้นก่อนรับฟังของ Gateway
แม้ว่าช่องจะได้รับการกำหนดค่าแล้วก็ตาม

ใช้ตัวเลือกนี้เฉพาะเมื่อ `setupEntry` ครอบคลุมพื้นผิวการเริ่มต้นทั้งหมดที่ต้องมี
ก่อน Gateway เริ่มรับฟัง ในทางปฏิบัติ หมายความว่ารายการตั้งค่า
ต้องลงทะเบียนทุกความสามารถที่ช่องเป็นเจ้าของซึ่งการเริ่มต้นต้องพึ่งพา เช่น:

- การลงทะเบียนช่องเอง
- เส้นทาง HTTP ใดๆ ที่ต้องพร้อมใช้งานก่อน Gateway เริ่มรับฟัง
- เมธอด Gateway เครื่องมือ หรือบริการใดๆ ที่ต้องมีอยู่ในช่วงเวลาเดียวกัน

หากรายการเต็มของคุณยังคงเป็นเจ้าของความสามารถที่จำเป็นต่อการเริ่มต้น อย่าเปิดใช้
แฟล็กนี้ ให้ Plugin ใช้พฤติกรรมเริ่มต้นต่อไปและปล่อยให้ OpenClaw โหลด
รายการเต็มระหว่างการเริ่มต้น

ช่องที่รวมมาให้ยังสามารถเผยแพร่ตัวช่วยพื้นผิวสัญญาที่ใช้เฉพาะการตั้งค่า ซึ่งคอร์
สามารถตรวจสอบก่อนโหลดรันไทม์ช่องแบบเต็มได้ พื้นผิวการเลื่อนระดับการตั้งค่าปัจจุบันคือ:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

คอร์ใช้พื้นผิวดังกล่าวเมื่อต้องเลื่อนระดับการกำหนดค่าช่องแบบบัญชีเดียวเดิม
เป็น `channels.<id>.accounts.*` โดยไม่โหลดรายการเต็มของ Plugin
Matrix คือตัวอย่างที่รวมมาให้ในปัจจุบัน โดยจะย้ายเฉพาะคีย์การตรวจสอบสิทธิ์/บูตสแตรปไปยัง
บัญชีที่มีชื่อซึ่งเลื่อนระดับแล้วเมื่อมีบัญชีที่มีชื่ออยู่ และสามารถคง
คีย์บัญชีเริ่มต้นที่ไม่ใช่แบบมาตรฐานซึ่งกำหนดค่าไว้ แทนที่จะสร้าง
`accounts.default` เสมอ

อะแดปเตอร์แพตช์การตั้งค่าเหล่านี้ทำให้การค้นหาพื้นผิวสัญญาที่รวมมาให้ยังคงโหลดแบบล่าช้า
เวลาในการนำเข้ายังคงเบา พื้นผิวการเลื่อนระดับจะถูกโหลดเฉพาะเมื่อใช้งานครั้งแรก แทนที่จะ
กลับเข้าสู่การเริ่มต้นช่องที่รวมมาให้อีกครั้งระหว่างการนำเข้าโมดูล

เมื่อพื้นผิวการเริ่มต้นเหล่านั้นมีเมธอด RPC ของ Gateway ให้เก็บเมธอดเหล่านั้นไว้ภายใต้
คำนำหน้าเฉพาะ Plugin เนมสเปซการดูแลระบบของคอร์ (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) ยังคงถูกสงวนไว้และจะแก้ไขเป็น
`operator.admin` เสมอ แม้ว่า Plugin จะร้องขอขอบเขตที่แคบกว่าก็ตาม

ตัวอย่าง:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### เมทาดาทาแค็ตตาล็อกช่อง

Plugin ช่องสามารถประกาศเมทาดาทาการตั้งค่า/การค้นพบผ่าน `openclaw.channel` และ
คำแนะนำการติดตั้งผ่าน `openclaw.install` ซึ่งช่วยให้แค็ตตาล็อกของคอร์ไม่ต้องมีข้อมูล

ตัวอย่าง:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (โฮสต์ด้วยตนเอง)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "แชตที่โฮสต์ด้วยตนเองผ่านบอต Webhook ของ Nextcloud Talk",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

ฟิลด์ `openclaw.channel` ที่มีประโยชน์นอกเหนือจากตัวอย่างขั้นต่ำ:

- `detailLabel`: ป้ายกำกับรองสำหรับพื้นผิวแค็ตตาล็อก/สถานะที่มีรายละเอียดมากขึ้น
- `docsLabel`: แทนที่ข้อความลิงก์สำหรับลิงก์เอกสาร
- `preferOver`: ID Plugin/ช่องที่มีลำดับความสำคัญต่ำกว่าซึ่งรายการแค็ตตาล็อกนี้ควรมีลำดับเหนือกว่า
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: การควบคุมข้อความบนพื้นผิวการเลือก
- `markdownCapable`: ทำเครื่องหมายว่าช่องรองรับ Markdown สำหรับการตัดสินใจจัดรูปแบบขาออก
- `exposure.configured`: ซ่อนช่องจากพื้นผิวรายการช่องที่กำหนดค่าแล้วเมื่อตั้งค่าเป็น `false`
- `exposure.setup`: ซ่อนช่องจากตัวเลือกการตั้งค่า/กำหนดค่าแบบโต้ตอบเมื่อตั้งค่าเป็น `false`
- `exposure.docs`: ทำเครื่องหมายว่าช่องเป็นช่องภายใน/ส่วนตัวสำหรับพื้นผิวการนำทางเอกสาร
- `showConfigured` / `showInSetup`: ยังคงยอมรับนามแฝงแบบเดิมเพื่อความเข้ากันได้ ควรใช้ `exposure`
- `quickstartAllowFrom`: เลือกให้ช่องเข้าสู่โฟลว์เริ่มต้นด่วน `allowFrom` มาตรฐาน
- `forceAccountBinding`: กำหนดให้ผูกบัญชีอย่างชัดเจน แม้ว่าจะมีเพียงบัญชีเดียว
- `preferSessionLookupForAnnounceTarget`: เลือกใช้การค้นหาเซสชันก่อนเมื่อแก้ไขเป้าหมายการประกาศ

OpenClaw ยังสามารถผสาน **แค็ตตาล็อกช่องภายนอก** (ตัวอย่างเช่น การส่งออกรีจิสทรี
MPM) วางไฟล์ JSON ไว้ที่ตำแหน่งใดตำแหน่งหนึ่งต่อไปนี้:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

หรือกำหนดให้ `OPENCLAW_PLUGIN_CATALOG_PATHS` (หรือ `OPENCLAW_MPM_CATALOG_PATHS`) ชี้ไปยัง
ไฟล์ JSON หนึ่งไฟล์ขึ้นไป (คั่นด้วยจุลภาค/อัฒภาค/`PATH`) แต่ละไฟล์ควร
มี `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }` ตัวแยกวิเคราะห์ยังยอมรับ `"packages"` หรือ `"plugins"` เป็นนามแฝงแบบเดิมสำหรับคีย์ `"entries"`

รายการแค็ตตาล็อกช่องและรายการแค็ตตาล็อกการติดตั้งผู้ให้บริการที่สร้างขึ้นจะแสดง
ข้อเท็จจริงของแหล่งติดตั้งที่ปรับให้อยู่ในรูปแบบมาตรฐานถัดจากบล็อก `openclaw.install` ดิบ
ข้อเท็จจริงที่ปรับให้อยู่ในรูปแบบมาตรฐานระบุว่าสเปก npm เป็นเวอร์ชันแบบระบุชัดเจนหรือตัวเลือก
แบบลอย มีเมทาดาทาความสมบูรณ์ที่คาดไว้หรือไม่ และมีเส้นทาง
แหล่งที่มาในเครื่องด้วยหรือไม่ เมื่อทราบอัตลักษณ์ของแค็ตตาล็อก/แพ็กเกจ
ข้อเท็จจริงที่ปรับให้อยู่ในรูปแบบมาตรฐานจะเตือนหากชื่อแพ็กเกจ npm ที่แยกวิเคราะห์ได้เบี่ยงเบนจากอัตลักษณ์ดังกล่าว
และยังเตือนเมื่อ `defaultChoice` ไม่ถูกต้องหรือชี้ไปยังแหล่งที่มา
ที่ไม่พร้อมใช้งาน และเมื่อมีเมทาดาทาความสมบูรณ์ของ npm โดยไม่มีแหล่งที่มา npm
ที่ถูกต้อง ผู้ใช้ข้อมูลควรถือว่า `installSource` เป็นฟิลด์เสริมแบบเพิ่มเข้าไป เพื่อให้
รายการที่สร้างด้วยตนเองและชิมแค็ตตาล็อกไม่ต้องสังเคราะห์ฟิลด์นี้
วิธีนี้ช่วยให้การเริ่มต้นใช้งานและการวินิจฉัยอธิบายสถานะของระนาบแหล่งที่มาได้โดยไม่ต้อง
นำเข้ารันไทม์ของ Plugin

รายการ npm ภายนอกอย่างเป็นทางการควรใช้ `npmSpec` แบบระบุชัดเจนร่วมกับ
`expectedIntegrity` เป็นหลัก ชื่อแพ็กเกจเปล่าและ dist-tag ยังคงใช้งานได้เพื่อ
ความเข้ากันได้ แต่จะแสดงคำเตือนในระนาบแหล่งที่มา เพื่อให้แค็ตตาล็อกค่อย ๆ เปลี่ยนไปใช้
การติดตั้งที่ตรึงเวอร์ชันและตรวจสอบความสมบูรณ์ โดยไม่ทำให้ plugins ที่มีอยู่เสียหาย
เมื่อการเริ่มต้นใช้งานติดตั้งจากพาธแค็ตตาล็อกในเครื่อง ระบบจะบันทึกรายการดัชนี
plugin ที่มีการจัดการ โดยใช้ `source: "path"` และ
`sourcePath` ที่สัมพันธ์กับเวิร์กสเปซเมื่อเป็นไปได้ พาธโหลดเชิงปฏิบัติการแบบสัมบูรณ์จะยังคงอยู่ใน
`plugins.load.paths`; ระเบียนการติดตั้งจะหลีกเลี่ยงการทำสำเนาพาธของเวิร์กสเตชันในเครื่อง
ลงในการกำหนดค่าที่มีอายุยาวนาน วิธีนี้ทำให้การติดตั้งเพื่อการพัฒนาในเครื่องยังคงปรากฏต่อ
การวินิจฉัยระนาบแหล่งที่มา โดยไม่เพิ่มพื้นผิวการเปิดเผยพาธระบบไฟล์ดิบอีกชุดหนึ่ง
ตาราง SQLite `installed_plugin_index` ที่บันทึกถาวรเป็นแหล่งข้อมูลจริง
ของการติดตั้ง และสามารถรีเฟรชได้โดยไม่ต้องโหลดโมดูลรันไทม์ของ plugin
แมป `installRecords` ของตารางนี้คงอยู่ถาวร แม้ว่าแมนิเฟสต์ของ plugin จะหายไปหรือ
ไม่ถูกต้อง ส่วนเพย์โหลด `plugins` เป็นมุมมองแมนิเฟสต์ที่สร้างใหม่ได้

## Plugins ของกลไกบริบท

Plugins ของกลไกบริบทเป็นเจ้าของการประสานบริบทเซสชันสำหรับการรับเข้า การประกอบ
และ Compaction ลงทะเบียนจาก plugin ของคุณด้วย
`api.registerContextEngine(id, factory)` จากนั้นเลือกกลไกที่ใช้งานอยู่ด้วย
`plugins.slots.contextEngine`

ใช้วิธีนี้เมื่อ plugin ของคุณต้องแทนที่หรือขยายไปป์ไลน์บริบทเริ่มต้น
แทนที่จะเพียงเพิ่มการค้นหาหน่วยความจำหรือฮุก

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";
import { resolveSessionAgentId } from "openclaw/plugin-sdk/memory-host-core";

export default function (api) {
  api.registerContextEngine("lossless-claw", (ctx) => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, sessionKey, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentId: resolveSessionAgentId({ config: ctx.config, sessionKey }),
          agentSessionKey: sessionKey,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

แฟกทอรี `ctx` เปิดเผยค่า `config`, `agentDir` และ `workspaceDir`
แบบไม่บังคับสำหรับการกำหนดค่าเริ่มต้น ณ เวลาสร้าง

โฮสต์จะเตรียมพรอมต์หน่วยความจำแบบอะซิงโครนัสที่ลงทะเบียนไว้ให้เสร็จก่อนเรียก
`assemble()` ของกลไกที่ไม่ใช่แบบเดิม `buildMemorySystemPromptAddition(...)` ยังคง
ทำงานแบบซิงโครนัสและอ่านสแนปช็อตการรันที่เปลี่ยนแปลงไม่ได้ดังกล่าว ขณะที่ `assemble()` ทำงานอยู่
ส่งต่อบริบทเครื่องมือและการอ้างอิงที่ให้มาโดยไม่เปลี่ยนแปลง เพื่อไม่ให้สแนปช็อต
ข้ามขอบเขตการรัน

`assemble()` อาจคืนค่า `contextProjection` เมื่อฮาร์เนสที่ใช้งานอยู่มี
เธรดแบ็กเอนด์ถาวร ให้ละเว้นค่านี้สำหรับการฉายภาพต่อรอบแบบเดิม คืนค่า
`{ mode: "thread_bootstrap", epoch }` เมื่อควรแทรกบริบทที่ประกอบแล้ว
ลงในเธรดแบ็กเอนด์ครั้งเดียว และนำกลับมาใช้ซ้ำจนกว่า epoch จะเปลี่ยนแปลง ให้เปลี่ยน
epoch หลังจากบริบทเชิงความหมายของกลไกเปลี่ยนแปลง เช่น หลังจากรอบ
Compaction ที่กลไกเป็นเจ้าของ โฮสต์อาจคงข้อมูลเมตาของการเรียกเครื่องมือ รูปแบบอินพุต
และผลลัพธ์เครื่องมือที่ปกปิดข้อมูลไว้ในการฉายภาพสำหรับบูตสแตรปเธรด เพื่อให้
เธรดแบ็กเอนด์ใหม่ยังคงความต่อเนื่องของเครื่องมือได้โดยไม่คัดลอกเพย์โหลดดิบ
ที่มีข้อมูลลับ

หากกลไกของคุณ **ไม่ได้** เป็นเจ้าของอัลกอริทึม Compaction ให้คงการใช้งาน `compact()`
ไว้และมอบหมายอย่างชัดเจน:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";
import { resolveSessionAgentId } from "openclaw/plugin-sdk/memory-host-core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", (ctx) => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, sessionKey, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentId: resolveSessionAgentId({ config: ctx.config, sessionKey }),
          agentSessionKey: sessionKey,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## การเพิ่มความสามารถใหม่

เมื่อ plugin ต้องการพฤติกรรมที่ API ปัจจุบันไม่รองรับ อย่าข้าม
ระบบ plugin ด้วยการเข้าถึงภายในแบบส่วนตัว ให้เพิ่มความสามารถที่ขาดหายไป

ลำดับที่แนะนำ:

1. **กำหนดสัญญาของคอร์** ตัดสินใจว่าคอร์ควรเป็นเจ้าของพฤติกรรมร่วมใดบ้าง:
   นโยบาย การย้อนกลับ การผสานการกำหนดค่า วงจรชีวิต ความหมายที่ช่องทางใช้
   และรูปแบบตัวช่วยรันไทม์
2. **เพิ่มพื้นผิวการลงทะเบียน/รันไทม์ของ plugin ที่มีชนิดข้อมูลกำกับ** ขยาย
   `OpenClawPluginApi` และ/หรือ `api.runtime` ด้วยพื้นผิวความสามารถที่มีชนิดข้อมูลกำกับ
   ซึ่งเล็กที่สุดแต่ยังใช้งานได้
3. **เชื่อมต่อคอร์กับผู้ใช้ช่องทาง/ฟีเจอร์** ช่องทางและ plugins ฟีเจอร์
   ควรใช้ความสามารถใหม่ผ่านคอร์ ไม่ใช่นำเข้าอิมพลีเมนเทชันของผู้ให้บริการโดยตรง
4. **ลงทะเบียนอิมพลีเมนเทชันของผู้ให้บริการ** จากนั้น plugins ของผู้ให้บริการจะลงทะเบียน
   แบ็กเอนด์ของตนกับความสามารถดังกล่าว
5. **เพิ่มการครอบคลุมสัญญา** เพิ่มการทดสอบเพื่อให้ความเป็นเจ้าของและรูปแบบการลงทะเบียน
   ยังคงชัดเจนเมื่อเวลาผ่านไป

นี่คือวิธีที่ OpenClaw ยังคงมีจุดยืนชัดเจนโดยไม่ถูกฮาร์ดโค้ดให้ยึดติดกับ
มุมมองของผู้ให้บริการรายเดียว ดูรายการตรวจสอบไฟล์ที่เป็นรูปธรรมและตัวอย่างที่ทำไว้แล้วได้ที่ [คู่มือสูตรความสามารถ](/th/plugins/adding-capabilities)

### รายการตรวจสอบความสามารถ

เมื่อเพิ่มความสามารถใหม่ โดยทั่วไปอิมพลีเมนเทชันควรครอบคลุมพื้นผิวเหล่านี้
ร่วมกัน:

- ชนิดข้อมูลสัญญาของคอร์ใน `src/<capability>/types.ts`
- ตัวรัน/ตัวช่วยรันไทม์ของคอร์ใน `src/<capability>/runtime.ts`
- พื้นผิวการลงทะเบียน API ของ plugin ใน `src/plugins/types.ts`
- การเชื่อมต่อรีจิสทรีของ plugin ใน `src/plugins/registry.ts`
- การเปิดเผยรันไทม์ของ plugin ใน `src/plugins/runtime/*` เมื่อ plugins ของฟีเจอร์/ช่องทาง
  จำเป็นต้องใช้งาน
- ตัวช่วยจับข้อมูล/ทดสอบใน `src/test-utils/plugin-registration.ts`
- การยืนยันความเป็นเจ้าของ/สัญญาใน `src/plugins/contracts/registry.ts`
- เอกสารสำหรับผู้ดูแลระบบ/plugin ใน `docs/`

หากพื้นผิวใดพื้นผิวหนึ่งขาดหายไป โดยทั่วไปเป็นสัญญาณว่าความสามารถดังกล่าว
ยังไม่ได้ผสานรวมอย่างสมบูรณ์

### เทมเพลตความสามารถ

รูปแบบขั้นต่ำ:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

รูปแบบการทดสอบสัญญา (`src/plugins/contracts/registry.ts` เปิดเผยการค้นหา
ความเป็นเจ้าของ เช่น `providerContractPluginIds`; การทดสอบยืนยันว่ารายการ
`contracts.videoGenerationProviders` ของ plugin ตรงกับสิ่งที่ลงทะเบียนจริง):

```ts
expect(pluginManifest.contracts?.videoGenerationProviders).toEqual(["openai"]);
```

วิธีนี้ทำให้กฎเรียบง่าย:

- คอร์เป็นเจ้าของสัญญาความสามารถและการประสานงาน
- plugins ของผู้ให้บริการเป็นเจ้าของอิมพลีเมนเทชันของผู้ให้บริการ
- plugins ของฟีเจอร์/ช่องทางใช้ตัวช่วยรันไทม์
- การทดสอบสัญญาทำให้ความเป็นเจ้าของชัดเจน
