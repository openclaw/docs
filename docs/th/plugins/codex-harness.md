Plugin `codex` ที่รวมมาให้ทำให้ OpenClaw รันรอบ agent แบบฝังตัวผ่าน
เซิร์ฟเวอร์แอป Codex แทนฮาร์เนส PI ในตัว

ใช้สิ่งนี้เมื่อคุณต้องการให้ Codex เป็นเจ้าของเซสชัน agent ระดับล่าง ได้แก่ การค้นหา
โมเดล การกลับมาทำงานต่อของเธรดแบบเนทีฟ Compaction แบบเนทีฟ และการทำงานผ่านเซิร์ฟเวอร์แอป
OpenClaw ยังคงเป็นเจ้าของช่องทางแชต ไฟล์เซสชัน การเลือกโมเดล เครื่องมือ
การอนุมัติ การส่งสื่อ และสำเนาทรานสคริปต์ที่มองเห็นได้

เมื่อรอบแชตต้นทางรันผ่านฮาร์เนส Codex การตอบกลับที่มองเห็นได้จะตั้งค่าเริ่มต้น
เป็นเครื่องมือ `message` ของ OpenClaw หากดีพลอยเมนต์ไม่ได้กำหนดค่า
`messages.visibleReplies` ไว้อย่างชัดเจน agent ยังสามารถจบรอบ Codex ของตนแบบส่วนตัวได้
และจะโพสต์ไปยังช่องทางเฉพาะเมื่อเรียก `message(action="send")` เท่านั้น ตั้งค่า
`messages.visibleReplies: "automatic"` เพื่อคงการตอบกลับสุดท้ายของแชตโดยตรงไว้บน
เส้นทางการส่งแบบอัตโนมัติดั้งเดิม

รอบ Heartbeat ของ Codex จะได้รับเครื่องมือ `heartbeat_respond` เป็นค่าเริ่มต้นด้วย เพื่อให้
agent บันทึกได้ว่าการปลุกควรเงียบต่อไปหรือแจ้งเตือน โดยไม่ต้องเข้ารหัส
โฟลว์ควบคุมนั้นไว้ในข้อความสุดท้าย

คำแนะนำเชิงรุกเฉพาะ Heartbeat จะถูกส่งเป็นคำสั่งสำหรับนักพัฒนาในโหมดการทำงานร่วมกันของ Codex
บนรอบ Heartbeat เอง รอบแชตทั่วไปจะกู้คืนโหมดค่าเริ่มต้นของ Codex
แทนที่จะพกปรัชญา Heartbeat ไว้ในพรอมป์ตรันไทม์ปกติ

หากคุณกำลังพยายามทำความเข้าใจภาพรวม ให้เริ่มที่
[รันไทม์ของ agent](/th/concepts/agent-runtimes) สรุปสั้น ๆ คือ:
`openai/gpt-5.5` คือการอ้างอิงโมเดล, `codex` คือรันไทม์ และ Telegram,
Discord, Slack หรือช่องทางอื่นยังคงเป็นพื้นผิวการสื่อสาร

## การกำหนดค่าด่วน

ผู้ใช้ส่วนใหญ่ที่ต้องการ "Codex ใน OpenClaw" ต้องการเส้นทางนี้: ลงชื่อเข้าใช้ด้วย
การสมัครใช้งาน ChatGPT/Codex จากนั้นรันรอบ agent แบบฝังตัวผ่านรันไทม์
เซิร์ฟเวอร์แอป Codex แบบเนทีฟ การอ้างอิงโมเดลยังคงเป็นรูปแบบมาตรฐานเป็น
`openai/gpt-*`; การยืนยันตัวตนจากการสมัครใช้งานมาจากบัญชี/โปรไฟล์ Codex ไม่ได้มาจาก
คำนำหน้าโมเดล `openai-codex/*`

ก่อนอื่นให้ลงชื่อเข้าใช้ด้วย Codex OAuth หากคุณยังไม่ได้ทำ:

```bash
openclaw models auth login --provider openai-codex
```

จากนั้นเปิดใช้ Plugin `codex` ที่รวมมาให้ และบังคับใช้รันไทม์ Codex:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

หากการกำหนดค่าของคุณใช้ `plugins.allow` ให้ใส่ `codex` ไว้ที่นั่นด้วย:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

อย่าใช้ `openai-codex/gpt-*` เมื่อคุณหมายถึงรันไทม์ Codex แบบเนทีฟ คำนำหน้านั้น
คือเส้นทาง "Codex OAuth ผ่าน PI" แบบชัดเจน การเปลี่ยนแปลงการกำหนดค่าจะมีผลกับ
เซสชันใหม่หรือเซสชันที่รีเซ็ตแล้ว เซสชันที่มีอยู่จะเก็บรันไทม์ที่บันทึกไว้

## Plugin นี้เปลี่ยนอะไร

Plugin `codex` ที่รวมมาให้เพิ่มความสามารถแยกกันหลายอย่าง:

| ความสามารถ                        | วิธีใช้งาน                                      | สิ่งที่ทำ                                                                  |
| --------------------------------- | --------------------------------------------------- | ----------------------------------------------------------------------------- |
| รันไทม์ฝังตัวแบบเนทีฟ           | `agentRuntime.id: "codex"`                          | รันรอบ agent แบบฝังตัวของ OpenClaw ผ่านเซิร์ฟเวอร์แอป Codex                  |
| คำสั่งควบคุมแชตแบบเนทีฟ      | `/codex bind`, `/codex resume`, `/codex steer`, ... | ผูกและควบคุมเธรดเซิร์ฟเวอร์แอป Codex จากการสนทนาผ่านข้อความ    |
| ผู้ให้บริการ/แค็ตตาล็อกเซิร์ฟเวอร์แอป Codex | ส่วนภายในของ `codex` ซึ่งแสดงผ่านฮาร์เนส     | ทำให้รันไทม์ค้นหาและตรวจสอบโมเดลเซิร์ฟเวอร์แอปได้                     |
| เส้นทางทำความเข้าใจสื่อของ Codex    | เส้นทางความเข้ากันได้ของโมเดลภาพ `codex/*`           | รันรอบเซิร์ฟเวอร์แอป Codex แบบจำกัดขอบเขตสำหรับโมเดลทำความเข้าใจภาพที่รองรับ |
| รีเลย์ฮุกแบบเนทีฟ                 | ฮุกของ Plugin รอบเหตุการณ์เนทีฟของ Codex             | ทำให้ OpenClaw สังเกต/บล็อกเหตุการณ์เครื่องมือ/การจบงานแบบเนทีฟของ Codex ที่รองรับได้  |

การเปิดใช้ Plugin ทำให้ความสามารถเหล่านั้นพร้อมใช้งาน สิ่งนี้ **ไม่ได้**:

- เริ่มใช้ Codex สำหรับโมเดล OpenAI ทุกตัว
- แปลงการอ้างอิงโมเดล `openai-codex/*` เป็นรันไทม์แบบเนทีฟ
- ทำให้ ACP/acpx เป็นเส้นทาง Codex เริ่มต้น
- สลับเซสชันที่มีอยู่ซึ่งบันทึกรันไทม์ PI ไว้แล้วแบบทันที
- แทนที่การส่งช่องทางของ OpenClaw, ไฟล์เซสชัน, ที่เก็บโปรไฟล์การยืนยันตัวตน หรือ
  การกำหนดเส้นทางข้อความ

Plugin เดียวกันนี้ยังเป็นเจ้าของพื้นผิวคำสั่งควบคุมแชต `/codex` แบบเนทีฟด้วย หาก
เปิดใช้ Plugin แล้วผู้ใช้ขอให้ผูก กลับมาทำงานต่อ ชี้นำ หยุด หรือตรวจสอบ
เธรด Codex จากแชต agent ควรเลือกใช้ `/codex ...` แทน ACP ACP ยังคงเป็น
ทางเลือกสำรองแบบชัดเจนเมื่อผู้ใช้ขอ ACP/acpx หรือกำลังทดสอบอะแดปเตอร์
Codex ของ ACP

รอบ Codex แบบเนทีฟจะคงฮุก Plugin ของ OpenClaw ไว้เป็นเลเยอร์ความเข้ากันได้สาธารณะ
สิ่งเหล่านี้เป็นฮุก OpenClaw ภายในกระบวนการ ไม่ใช่ฮุกคำสั่ง `hooks.json` ของ Codex:

- `before_prompt_build`
- `before_compaction`, `after_compaction`
- `llm_input`, `llm_output`
- `before_tool_call`, `after_tool_call`
- `before_message_write` สำหรับเรกคอร์ดทรานสคริปต์ที่ทำสำเนา
- `before_agent_finalize` ผ่านรีเลย์ `Stop` ของ Codex
- `agent_end`

Plugin ยังสามารถลงทะเบียนมิดเดิลแวร์ผลลัพธ์เครื่องมือที่เป็นกลางต่อรันไทม์ เพื่อเขียน
ผลลัพธ์เครื่องมือแบบไดนามิกของ OpenClaw ใหม่หลังจาก OpenClaw ดำเนินการเครื่องมือและก่อนที่
ผลลัพธ์จะถูกส่งกลับไปยัง Codex สิ่งนี้แยกจากฮุก Plugin สาธารณะ
`tool_result_persist` ซึ่งแปลงการเขียนผลลัพธ์เครื่องมือในทรานสคริปต์ที่ OpenClaw เป็นเจ้าของ

สำหรับความหมายของฮุก Plugin เอง โปรดดู [ฮุก Plugin](/th/plugins/hooks)
และ [พฤติกรรมการ์ดของ Plugin](/th/tools/plugin)

ฮาร์เนสปิดอยู่ตามค่าเริ่มต้น การกำหนดค่าใหม่ควรคงการอ้างอิงโมเดล OpenAI
ในรูปแบบมาตรฐานเป็น `openai/gpt-*` และบังคับใช้
`agentRuntime.id: "codex"` หรือ `OPENCLAW_AGENT_RUNTIME=codex` อย่างชัดเจนเมื่อ
ต้องการการทำงานผ่านเซิร์ฟเวอร์แอปแบบเนทีฟ การอ้างอิงโมเดล `codex/*` แบบเก่ายังคงเลือก
ฮาร์เนสโดยอัตโนมัติเพื่อความเข้ากันได้ แต่คำนำหน้าผู้ให้บริการแบบเก่าที่มีรันไทม์รองรับ
จะไม่ถูกแสดงเป็นตัวเลือกโมเดล/ผู้ให้บริการปกติ

หากเปิดใช้ Plugin `codex` แต่โมเดลหลักยังเป็น
`openai-codex/*` อยู่ `openclaw doctor` จะเตือนแทนที่จะเปลี่ยนเส้นทาง สิ่งนั้น
เป็นเจตนา: `openai-codex/*` ยังคงเป็นเส้นทาง Codex OAuth/การสมัครใช้งานผ่าน PI และ
การทำงานผ่านเซิร์ฟเวอร์แอปแบบเนทีฟยังคงเป็นตัวเลือกรันไทม์ที่ต้องระบุอย่างชัดเจน

## แผนผังเส้นทาง

ใช้ตารางนี้ก่อนเปลี่ยนการกำหนดค่า:

| พฤติกรรมที่ต้องการ                                     | การอ้างอิงโมเดล                  | การกำหนดค่ารันไทม์                         | เส้นทางการยืนยันตัวตน/โปรไฟล์           | ป้ายสถานะที่คาดหวัง          |
| ---------------------------------------------------- | -------------------------- | -------------------------------------- | ---------------------------- | ------------------------------ |
| การสมัครใช้งาน ChatGPT/Codex พร้อมรันไทม์ Codex แบบเนทีฟ | `openai/gpt-*`             | `agentRuntime.id: "codex"`             | Codex OAuth หรือบัญชี Codex | `Runtime: OpenAI Codex`        |
| OpenAI API ผ่านตัวรัน OpenClaw ปกติ            | `openai/gpt-*`             | ละไว้ หรือ `runtime: "pi"`             | คีย์ OpenAI API               | `Runtime: OpenClaw Pi Default` |
| การสมัครใช้งาน ChatGPT/Codex ผ่าน PI                | `openai-codex/gpt-*`       | ละไว้ หรือ `runtime: "pi"`             | ผู้ให้บริการ OpenAI Codex OAuth  | `Runtime: OpenClaw Pi Default` |
| ผู้ให้บริการผสมพร้อมโหมดอัตโนมัติแบบระมัดระวัง          | การอ้างอิงเฉพาะผู้ให้บริการ     | `agentRuntime.id: "auto"`              | ตามผู้ให้บริการที่เลือก        | ขึ้นอยู่กับรันไทม์ที่เลือก    |
| เซสชันอะแดปเตอร์ Codex ACP แบบชัดเจน                   | ขึ้นอยู่กับพรอมป์ต์/โมเดลของ ACP | `sessions_spawn` พร้อม `runtime: "acp"` | การยืนยันตัวตนแบ็กเอนด์ ACP             | สถานะงาน/เซสชัน ACP        |

จุดแบ่งสำคัญคือผู้ให้บริการเทียบกับรันไทม์:

- `openai-codex/*` ตอบคำถามว่า "PI ควรใช้เส้นทางผู้ให้บริการ/การยืนยันตัวตนใด?"
- `agentRuntime.id: "codex"` ตอบคำถามว่า "ลูปใดควรดำเนินการ
  รอบแบบฝังตัวนี้?"
- `/codex ...` ตอบคำถามว่า "การสนทนา Codex แบบเนทีฟใดที่แชตนี้ควรผูก
  หรือควบคุม?"
- ACP ตอบคำถามว่า "กระบวนการฮาร์เนสภายนอกใดที่ acpx ควรเปิด?"

## เลือกคำนำหน้าโมเดลที่ถูกต้อง

เส้นทางตระกูล OpenAI เป็นแบบเฉพาะคำนำหน้า สำหรับการตั้งค่าทั่วไปที่เป็นการสมัครใช้งานพร้อม
รันไทม์ Codex แบบเนทีฟ ให้ใช้ `openai/*` พร้อม `agentRuntime.id: "codex"`
ใช้ `openai-codex/*` เฉพาะเมื่อคุณตั้งใจใช้ Codex OAuth ผ่าน PI:

| การอ้างอิงโมเดล                                     | เส้นทางรันไทม์                                 | ใช้เมื่อ                                                                  |
| --------------------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------- |
| `openai/gpt-5.4`                              | ผู้ให้บริการ OpenAI ผ่านระบบท่อ OpenClaw/PI | คุณต้องการเข้าถึง OpenAI Platform API โดยตรงในปัจจุบันด้วย `OPENAI_API_KEY` |
| `openai-codex/gpt-5.5`                        | OpenAI Codex OAuth ผ่าน OpenClaw/PI       | คุณต้องการการยืนยันตัวตนจากการสมัครใช้งาน ChatGPT/Codex ด้วยตัวรัน PI เริ่มต้น      |
| `openai/gpt-5.5` + `agentRuntime.id: "codex"` | ฮาร์เนสเซิร์ฟเวอร์แอป Codex                     | คุณต้องการการยืนยันตัวตนจากการสมัครใช้งาน ChatGPT/Codex พร้อมการทำงาน Codex แบบเนทีฟ     |

GPT-5.5 สามารถปรากฏได้ทั้งบนเส้นทางคีย์ API ของ OpenAI โดยตรงและเส้นทางการสมัครใช้งาน Codex
เมื่อบัญชีของคุณเปิดให้ใช้ ใช้ `openai/gpt-5.5` พร้อมฮาร์เนสเซิร์ฟเวอร์แอป Codex
สำหรับรันไทม์ Codex แบบเนทีฟ, `openai-codex/gpt-5.5` สำหรับ PI OAuth หรือ
`openai/gpt-5.5` โดยไม่มีการแทนที่รันไทม์ Codex สำหรับทราฟฟิกด้วยคีย์ API โดยตรง

การอ้างอิง `codex/gpt-*` แบบเก่ายังคงยอมรับเป็นนามแฝงเพื่อความเข้ากันได้ Doctor
สำหรับการย้ายความเข้ากันได้จะเขียนการอ้างอิงรันไทม์หลักแบบเก่าใหม่เป็นการอ้างอิงโมเดล
มาตรฐาน และบันทึกนโยบายรันไทม์แยกต่างหาก ขณะที่การอ้างอิงแบบเก่าที่เป็นตัวสำรองเท่านั้น
จะไม่ถูกเปลี่ยนแปลง เพราะรันไทม์ถูกกำหนดค่าสำหรับคอนเทนเนอร์ agent ทั้งหมด
การกำหนดค่า PI Codex OAuth ใหม่ควรใช้ `openai-codex/gpt-*`; การกำหนดค่าฮาร์เนส
เซิร์ฟเวอร์แอปแบบเนทีฟใหม่ควรใช้ `openai/gpt-*` ร่วมกับ
`agentRuntime.id: "codex"`

`agents.defaults.imageModel` ใช้การแบ่งคำนำหน้าแบบเดียวกัน ใช้
`openai-codex/gpt-*` เมื่อการทำความเข้าใจภาพควรรันผ่านเส้นทางผู้ให้บริการ OpenAI
Codex OAuth ใช้ `codex/gpt-*` เมื่อการทำความเข้าใจภาพควรรัน
ผ่านรอบเซิร์ฟเวอร์แอป Codex แบบจำกัดขอบเขต โมเดลเซิร์ฟเวอร์แอป Codex ต้อง
ประกาศการรองรับอินพุตภาพ โมเดล Codex ที่รองรับเฉพาะข้อความจะล้มเหลวก่อนที่รอบสื่อ
จะเริ่ม

ใช้ `/status` เพื่อยืนยันฮาร์เนสที่มีผลสำหรับเซสชันปัจจุบัน หากการเลือก
น่าประหลาดใจ ให้เปิดใช้การบันทึกดีบักสำหรับระบบย่อย `agents/harness`
และตรวจสอบเรกคอร์ดแบบมีโครงสร้าง `agent harness selected` ของ Gateway ซึ่ง
รวมถึงรหัสฮาร์เนสที่เลือก เหตุผลในการเลือก นโยบายรันไทม์/ตัวสำรอง และ
ในโหมด `auto` ผลการรองรับของผู้สมัคร Plugin แต่ละรายการ

### คำเตือนของ doctor หมายถึงอะไร

`openclaw doctor` จะเตือนเมื่อทุกข้อต่อไปนี้เป็นจริง:

- Plugin `codex` ที่รวมมาให้เปิดใช้งานหรืออนุญาตอยู่
- โมเดลหลักของ agent เป็น `openai-codex/*`
- รันไทม์ที่มีผลของ agent นั้นไม่ใช่ `codex`

คำเตือนนั้นมีอยู่เพราะผู้ใช้มักคาดว่า "เปิดใช้ Plugin Codex" จะหมายถึง
"รันไทม์เซิร์ฟเวอร์แอป Codex แบบเนทีฟ" OpenClaw ไม่ได้สรุปเช่นนั้น คำเตือน
หมายถึง:

- **ไม่จำเป็นต้องเปลี่ยนแปลงอะไร** หากคุณตั้งใจใช้ ChatGPT/Codex OAuth ผ่าน PI
- เปลี่ยนโมเดลเป็น `openai/<model>` และตั้งค่า
  `agentRuntime.id: "codex"` หากคุณตั้งใจใช้การทำงานผ่านเซิร์ฟเวอร์แอป
  แบบเนทีฟ
- เซสชันที่มีอยู่ยังต้องใช้ `/new` หรือ `/reset` หลังจากเปลี่ยนรันไทม์
  เพราะพินรันไทม์ของเซสชันเป็นแบบติดแน่น

การเลือกฮาร์เนสไม่ใช่ตัวควบคุมเซสชันแบบสด เมื่อรอบแบบฝังตัวรัน
OpenClaw จะบันทึกรหัสฮาร์เนสที่เลือกไว้บนเซสชันนั้น และใช้ต่อไปสำหรับ
รอบหลัง ๆ ในรหัสเซสชันเดียวกัน เปลี่ยนการกำหนดค่า `agentRuntime` หรือ
`OPENCLAW_AGENT_RUNTIME` เมื่อคุณต้องการให้เซสชันในอนาคตใช้ฮาร์เนสอื่น;
ใช้ `/new` หรือ `/reset` เพื่อเริ่มเซสชันใหม่ก่อนสลับการสนทนาที่มีอยู่
ระหว่าง PI และ Codex วิธีนี้หลีกเลี่ยงการเล่นทรานสคริปต์เดียวซ้ำผ่าน
ระบบเซสชันเนทีฟสองระบบที่เข้ากันไม่ได้

เซสชันแบบเดิมที่สร้างก่อนการปักหมุดฮาร์เนสจะถือว่าปักหมุดกับ PI เมื่อมีประวัติทรานสคริปต์แล้ว ใช้ `/new` หรือ `/reset` เพื่อเลือกให้บทสนทนานั้นเข้าสู่ Codex หลังจากเปลี่ยนการกำหนดค่า

`/status` แสดงรันไทม์โมเดลที่มีผลอยู่ ฮาร์เนส PI เริ่มต้นจะแสดงเป็น `Runtime: OpenClaw Pi Default` และฮาร์เนสเซิร์ฟเวอร์แอป Codex จะแสดงเป็น `Runtime: OpenAI Codex`

## ข้อกำหนด

- OpenClaw ที่มี Plugin `codex` แบบรวมมาให้พร้อมใช้งาน
- เซิร์ฟเวอร์แอป Codex `0.125.0` หรือใหม่กว่า Plugin ที่รวมมาให้จะจัดการไบนารีเซิร์ฟเวอร์แอป Codex ที่เข้ากันได้โดยค่าเริ่มต้น ดังนั้นคำสั่ง `codex` ภายในเครื่องบน `PATH` จะไม่ส่งผลต่อการเริ่มต้นฮาร์เนสตามปกติ
- การรับรองความถูกต้องของ Codex พร้อมใช้งานสำหรับโปรเซสเซิร์ฟเวอร์แอปหรือสะพานการรับรองความถูกต้อง Codex ของ OpenClaw การเปิดเซิร์ฟเวอร์แอปภายในเครื่องใช้โฮม Codex ที่ OpenClaw จัดการสำหรับแต่ละเอเจนต์และ `HOME` ของโปรเซสลูกที่แยกออกมา ดังนั้นโดยค่าเริ่มต้นจึงไม่อ่านบัญชี `~/.codex` ส่วนตัว, Skills, Plugin, การกำหนดค่า, สถานะเธรด หรือ `$HOME/.agents/skills` แบบเนทีฟของคุณ

Plugin จะบล็อกการจับมือกับเซิร์ฟเวอร์แอปที่เก่ากว่าหรือไม่มีเวอร์ชัน วิธีนี้ทำให้ OpenClaw อยู่บนพื้นผิวโปรโตคอลที่ผ่านการทดสอบแล้ว

สำหรับการทดสอบควันแบบสดและ Docker โดยปกติการรับรองความถูกต้องมาจากบัญชี CLI ของ Codex หรือโปรไฟล์การรับรองความถูกต้อง `openai-codex` ของ OpenClaw การเปิดเซิร์ฟเวอร์แอป stdio ภายในเครื่องยังสามารถถอยกลับไปใช้ `CODEX_API_KEY` / `OPENAI_API_KEY` ได้เมื่อไม่มีบัญชีอยู่

## ไฟล์บูตสแตรปของพื้นที่ทำงาน

Codex จัดการ `AGENTS.md` เองผ่านการค้นพบเอกสารโปรเจกต์แบบเนทีฟ OpenClaw ไม่เขียนไฟล์เอกสารโปรเจกต์ Codex สังเคราะห์หรือพึ่งพาชื่อไฟล์สำรองของ Codex สำหรับไฟล์บุคลิกภาพ เพราะชื่อสำรองของ Codex จะใช้เฉพาะเมื่อไม่มี `AGENTS.md`

เพื่อให้พื้นที่ทำงาน OpenClaw เทียบเท่ากัน ฮาร์เนส Codex จะ resolve ไฟล์บูตสแตรปอื่นๆ (`SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md` และ `MEMORY.md` เมื่อมี) และส่งต่อผ่านคำสั่งการกำหนดค่า Codex บน `thread/start` และ `thread/resume` วิธีนี้ทำให้ `SOUL.md` และบริบทบุคลิกภาพ/โปรไฟล์ของพื้นที่ทำงานที่เกี่ยวข้องยังมองเห็นได้โดยไม่ต้องทำซ้ำ `AGENTS.md`

## เพิ่ม Codex ควบคู่กับโมเดลอื่น

อย่าตั้งค่า `agentRuntime.id: "codex"` แบบทั่วทั้งระบบหากเอเจนต์เดียวกันควรสลับได้อย่างอิสระระหว่าง Codex กับโมเดลผู้ให้บริการที่ไม่ใช่ Codex รันไทม์ที่บังคับใช้จะใช้กับทุกเทิร์นที่ฝังไว้สำหรับเอเจนต์หรือเซสชันนั้น หากคุณเลือกโมเดล Anthropic ขณะที่รันไทม์นั้นถูกบังคับ OpenClaw จะยังลองใช้ฮาร์เนส Codex และล้มเหลวแบบปิด แทนที่จะกำหนดเส้นทางเทิร์นนั้นผ่าน PI อย่างเงียบๆ

ใช้รูปแบบใดรูปแบบหนึ่งต่อไปนี้แทน:

- วาง Codex ไว้บนเอเจนต์เฉพาะด้วย `agentRuntime.id: "codex"`
- ให้เอเจนต์เริ่มต้นอยู่บน `agentRuntime.id: "auto"` และใช้ PI fallback สำหรับการใช้งานผู้ให้บริการแบบผสมตามปกติ
- ใช้ refs แบบเดิม `codex/*` เพื่อความเข้ากันได้เท่านั้น การกำหนดค่าใหม่ควรใช้ `openai/*` พร้อมนโยบายรันไทม์ Codex ที่ชัดเจน

ตัวอย่างเช่น สิ่งนี้ทำให้เอเจนต์เริ่มต้นอยู่บนการเลือกอัตโนมัติปกติและเพิ่มเอเจนต์ Codex แยกต่างหาก:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      agentRuntime: {
        id: "auto",
      },
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
        agentRuntime: {
          id: "codex",
        },
      },
    ],
  },
}
```

ด้วยรูปแบบนี้:

- เอเจนต์เริ่มต้น `main` ใช้เส้นทางผู้ให้บริการปกติและ PI compatibility fallback
- เอเจนต์ `codex` ใช้ฮาร์เนสเซิร์ฟเวอร์แอป Codex
- หาก Codex ขาดหายไปหรือไม่รองรับสำหรับเอเจนต์ `codex` เทิร์นจะล้มเหลวแทนที่จะใช้ PI อย่างเงียบๆ

## การกำหนดเส้นทางคำสั่งเอเจนต์

เอเจนต์ควรกำหนดเส้นทางคำขอของผู้ใช้ตามเจตนา ไม่ใช่จากคำว่า "Codex" เพียงอย่างเดียว:

| ผู้ใช้ขอให้...                                       | เอเจนต์ควรใช้...                              |
| ------------------------------------------------------ | ------------------------------------------------ |
| "ผูกแชทนี้กับ Codex"                              | `/codex bind`                                    |
| "ดำเนินเธรด Codex `<id>` ต่อที่นี่"                      | `/codex resume <id>`                             |
| "แสดงเธรด Codex"                                   | `/codex threads`                                 |
| "ส่งรายงานสนับสนุนสำหรับการรัน Codex ที่ผิดพลาด"            | `/diagnostics [note]`                            |
| "ส่งข้อเสนอแนะ Codex เฉพาะสำหรับเธรดที่แนบมานี้เท่านั้น"    | `/codex diagnostics [note]`                      |
| "ใช้การสมัครสมาชิก ChatGPT/Codex ของฉันกับรันไทม์ Codex" | `openai/*` บวก `agentRuntime.id: "codex"`       |
| "ใช้การสมัครสมาชิก ChatGPT/Codex ของฉันผ่าน PI"         | refs โมเดล `openai-codex/*`                      |
| "รัน Codex ผ่าน ACP/acpx"                           | ACP `sessions_spawn({ runtime: "acp", ... })`    |
| "เริ่ม Claude Code/Gemini/OpenCode/Cursor ในเธรด" | ACP/acpx ไม่ใช่ `/codex` และไม่ใช่เอเจนต์ย่อยแบบเนทีฟ |

OpenClaw จะแสดงคำแนะนำการ spawn ของ ACP ให้เอเจนต์เห็นเฉพาะเมื่อ ACP เปิดใช้งาน ส่งต่อได้ และมีแบ็กเอนด์รันไทม์ที่โหลดแล้วรองรับ หาก ACP ไม่พร้อมใช้งาน system prompt และ Plugin Skills ไม่ควรสอนเอเจนต์เรื่องการกำหนดเส้นทาง ACP

## การปรับใช้ที่ใช้ Codex เท่านั้น

บังคับใช้ฮาร์เนส Codex เมื่อคุณต้องพิสูจน์ว่าทุกเทิร์นของเอเจนต์ที่ฝังไว้ใช้ Codex รันไทม์ Plugin ที่ระบุชัดเจนจะล้มเหลวแบบปิดและจะไม่ลองใหม่ผ่าน PI อย่างเงียบๆ:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

การ override ด้วยสภาพแวดล้อม:

```bash
OPENCLAW_AGENT_RUNTIME=codex openclaw gateway run
```

เมื่อบังคับใช้ Codex แล้ว OpenClaw จะล้มเหลวตั้งแต่เนิ่นๆ หาก Plugin Codex ถูกปิดใช้งาน, เซิร์ฟเวอร์แอปเก่าเกินไป หรือเซิร์ฟเวอร์แอปเริ่มไม่ได้

## Codex รายเอเจนต์

คุณสามารถทำให้เอเจนต์หนึ่งใช้ Codex เท่านั้น ขณะที่เอเจนต์เริ่มต้นยังคงใช้การเลือกอัตโนมัติปกติ:

```json5
{
  agents: {
    defaults: {
      agentRuntime: {
        id: "auto",
      },
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
        agentRuntime: {
          id: "codex",
        },
      },
    ],
  },
}
```

ใช้คำสั่งเซสชันปกติเพื่อสลับเอเจนต์และโมเดล `/new` สร้างเซสชัน OpenClaw ใหม่ และฮาร์เนส Codex จะสร้างหรือดำเนินเธรดเซิร์ฟเวอร์แอป sidecar ต่อเมื่อจำเป็น `/reset` ล้างการผูกเซสชัน OpenClaw สำหรับเธรดนั้น และให้เทิร์นถัดไป resolve ฮาร์เนสจากการกำหนดค่าปัจจุบันอีกครั้ง

## การค้นพบโมเดล

โดยค่าเริ่มต้น Plugin Codex จะถามเซิร์ฟเวอร์แอปถึงโมเดลที่พร้อมใช้งาน หากการค้นพบล้มเหลวหรือหมดเวลา จะใช้แค็ตตาล็อก fallback ที่รวมมาให้สำหรับ:

- GPT-5.5
- GPT-5.4 mini
- GPT-5.2

คุณสามารถปรับการค้นพบภายใต้ `plugins.entries.codex.config.discovery`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          discovery: {
            enabled: true,
            timeoutMs: 2500,
          },
        },
      },
    },
  },
}
```

ปิดใช้งานการค้นพบเมื่อคุณต้องการให้การเริ่มต้นหลีกเลี่ยงการ probe Codex และยึดตามแค็ตตาล็อก fallback:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          discovery: {
            enabled: false,
          },
        },
      },
    },
  },
}
```

## การเชื่อมต่อและนโยบายของเซิร์ฟเวอร์แอป

โดยค่าเริ่มต้น Plugin จะเริ่มไบนารี Codex ที่ OpenClaw จัดการภายในเครื่องด้วย:

```bash
codex app-server --listen stdio://
```

ไบนารีที่จัดการจะจัดส่งมากับแพ็กเกจ Plugin `codex` วิธีนี้ทำให้เวอร์ชันเซิร์ฟเวอร์แอปผูกกับ Plugin ที่รวมมาให้ แทนที่จะเป็น Codex CLI แยกต่างหากตัวใดก็ตามที่บังเอิญติดตั้งอยู่ภายในเครื่อง ตั้งค่า `appServer.command` เฉพาะเมื่อคุณตั้งใจต้องการรัน executable อื่น

โดยค่าเริ่มต้น OpenClaw เริ่มเซสชันฮาร์เนส Codex ภายในเครื่องในโหมด YOLO: `approvalPolicy: "never"`, `approvalsReviewer: "user"` และ `sandbox: "danger-full-access"` นี่คือท่าทีของผู้ปฏิบัติงานภายในเครื่องที่เชื่อถือได้ซึ่งใช้สำหรับ Heartbeat อัตโนมัติ: Codex สามารถใช้เครื่องมือเชลล์และเครือข่ายได้โดยไม่หยุดรอพรอมป์อนุมัติแบบเนทีฟที่ไม่มีใครอยู่ตอบ

หากต้องการเลือกใช้การอนุมัติที่ตรวจทานโดย guardian ของ Codex ให้ตั้งค่า `appServer.mode: "guardian"`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            mode: "guardian",
            serviceTier: "fast",
          },
        },
      },
    },
  },
}
```

โหมด Guardian ใช้เส้นทางการอนุมัติ auto-review แบบเนทีฟของ Codex เมื่อ Codex ขอออกจาก sandbox, เขียนนอกพื้นที่ทำงาน หรือเพิ่มสิทธิ์ เช่น การเข้าถึงเครือข่าย Codex จะกำหนดเส้นทางคำขออนุมัตินั้นไปยังผู้ตรวจทานแบบเนทีฟแทนพรอมป์มนุษย์ ผู้ตรวจทานจะใช้กรอบความเสี่ยงของ Codex และอนุมัติหรือปฏิเสธคำขอเฉพาะนั้น ใช้ Guardian เมื่อคุณต้องการรั้วป้องกันมากกว่าโหมด YOLO แต่ยังต้องการให้เอเจนต์ที่ไม่มีผู้ดูแลทำงานต่อไปได้

พรีเซ็ต `guardian` ขยายเป็น `approvalPolicy: "on-request"`, `approvalsReviewer: "auto_review"` และ `sandbox: "workspace-write"` ฟิลด์นโยบายแต่ละรายการยังคง override `mode` ได้ ดังนั้นการปรับใช้ขั้นสูงสามารถผสมพรีเซ็ตกับตัวเลือกที่ระบุชัดเจนได้ ค่า reviewer แบบเก่า `guardian_subagent` ยังยอมรับเป็น alias เพื่อความเข้ากันได้ แต่การกำหนดค่าใหม่ควรใช้ `auto_review`

สำหรับเซิร์ฟเวอร์แอปที่รันอยู่แล้ว ให้ใช้การขนส่ง WebSocket:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://127.0.0.1:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
            requestTimeoutMs: 60000,
          },
        },
      },
    },
  },
}
```

การเปิดเซิร์ฟเวอร์แอป Stdio จะสืบทอดสภาพแวดล้อมของโปรเซส OpenClaw โดยค่าเริ่มต้น แต่ OpenClaw เป็นเจ้าของสะพานบัญชีเซิร์ฟเวอร์แอป Codex และตั้งค่าทั้ง `CODEX_HOME` และ `HOME` เป็นไดเรกทอรีรายเอเจนต์ภายใต้สถานะ OpenClaw ของเอเจนต์นั้น ตัวโหลด skill ของ Codex เองอ่าน `$CODEX_HOME/skills` และ `$HOME/.agents/skills` ดังนั้นค่าทั้งสองจึงถูกแยกสำหรับการเปิดเซิร์ฟเวอร์แอปภายในเครื่อง วิธีนี้ทำให้ Skills, Plugin, การกำหนดค่า, บัญชี และสถานะเธรดแบบเนทีฟของ Codex ถูกจำกัดขอบเขตไว้ที่เอเจนต์ OpenClaw แทนที่จะรั่วเข้ามาจากโฮม Codex CLI ส่วนตัวของผู้ปฏิบัติงาน

Plugin ของ OpenClaw และสแนปชอต skill ของ OpenClaw ยังไหลผ่านรีจิสทรี Plugin และตัวโหลด skill ของ OpenClaw เอง สินทรัพย์ Codex CLI ส่วนตัวจะไม่ไหลผ่าน หากคุณมี Skills หรือ Plugin ของ Codex CLI ที่เป็นประโยชน์และควรกลายเป็นส่วนหนึ่งของเอเจนต์ OpenClaw ให้ทำ inventory อย่างชัดเจน:

```bash
openclaw migrate codex --dry-run
openclaw migrate apply codex --yes
```

ผู้ให้บริการการย้าย Codex จะคัดลอก Skills ไปยังพื้นที่ทำงานเอเจนต์ OpenClaw ปัจจุบัน Plugin, hook และไฟล์กำหนดค่าแบบเนทีฟของ Codex จะถูกรายงานหรือเก็บถาวรเพื่อให้ตรวจทานด้วยตนเองแทนที่จะเปิดใช้งานโดยอัตโนมัติ เพราะสิ่งเหล่านี้สามารถรันคำสั่ง เปิดเผยเซิร์ฟเวอร์ MCP หรือพกข้อมูลประจำตัวได้

การรับรองความถูกต้องจะถูกเลือกตามลำดับนี้:

1. โปรไฟล์การรับรองความถูกต้อง OpenClaw Codex ที่ระบุชัดเจนสำหรับเอเจนต์
2. บัญชีที่มีอยู่ของเซิร์ฟเวอร์แอปในโฮม Codex ของเอเจนต์นั้น
3. สำหรับการเปิดเซิร์ฟเวอร์แอป stdio ภายในเครื่องเท่านั้น ใช้ `CODEX_API_KEY` แล้วตามด้วย `OPENAI_API_KEY` เมื่อไม่มีบัญชีเซิร์ฟเวอร์แอปอยู่และยังต้องใช้การรับรองความถูกต้อง OpenAI

เมื่อ OpenClaw เห็นโปรไฟล์การรับรองความถูกต้อง Codex แบบการสมัครสมาชิก ChatGPT จะลบ `CODEX_API_KEY` และ `OPENAI_API_KEY` ออกจากโปรเซสลูก Codex ที่ spawn วิธีนี้ทำให้คีย์ API ระดับ Gateway ยังพร้อมใช้สำหรับ embeddings หรือโมเดล OpenAI โดยตรง โดยไม่ทำให้เทิร์นเซิร์ฟเวอร์แอป Codex แบบเนทีฟถูกคิดค่าใช้จ่ายผ่าน API โดยไม่ตั้งใจ โปรไฟล์คีย์ API Codex ที่ระบุชัดเจนและ fallback คีย์สภาพแวดล้อม stdio ภายในเครื่องใช้การเข้าสู่ระบบเซิร์ฟเวอร์แอปแทน env ของโปรเซสลูกที่สืบทอดมา การเชื่อมต่อเซิร์ฟเวอร์แอป WebSocket จะไม่ได้รับ fallback คีย์ API env ของ Gateway ใช้โปรไฟล์การรับรองความถูกต้องที่ระบุชัดเจนหรือบัญชีของเซิร์ฟเวอร์แอประยะไกลเอง

หากการปรับใช้ต้องการการแยกสภาพแวดล้อมเพิ่มเติม ให้เพิ่มตัวแปรเหล่านั้นใน `appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` มีผลเฉพาะกับกระบวนการลูกของ app-server ของ Codex ที่ถูกสร้างขึ้นเท่านั้น

เครื่องมือแบบไดนามิกของ Codex ใช้โปรไฟล์ `native-first` เป็นค่าเริ่มต้น ในโหมดนั้น
OpenClaw จะไม่เปิดเผยเครื่องมือแบบไดนามิกที่ซ้ำกับการดำเนินการในเวิร์กสเปซแบบเนทีฟของ Codex:
`read`, `write`, `edit`, `apply_patch`, `exec`, `process` และ
`update_plan` เครื่องมือผสานรวมของ OpenClaw เช่น การส่งข้อความ เซสชัน สื่อ
Cron เบราว์เซอร์ Node Gateway `heartbeat_respond` และ `web_search` ยังคง
พร้อมใช้งาน

ฟิลด์ระดับบนสุดของ Plugin Codex ที่รองรับ:

| ฟิลด์                      | ค่าเริ่มต้น          | ความหมาย                                                                                   |
| -------------------------- | ---------------- | ----------------------------------------------------------------------------------------- |
| `codexDynamicToolsProfile` | `"native-first"` | ใช้ `"openclaw-compat"` เพื่อเปิดเผยชุดเครื่องมือแบบไดนามิกทั้งหมดของ OpenClaw ให้กับ app-server ของ Codex |
| `codexDynamicToolsExclude` | `[]`             | ชื่อเครื่องมือแบบไดนามิกของ OpenClaw เพิ่มเติมที่จะละเว้นจากเทิร์นของ app-server ของ Codex               |

ฟิลด์ `appServer` ที่รองรับ:

| ฟิลด์               | ค่าเริ่มต้น                                  | ความหมาย                                                                                                                                                                                                                              |
| ------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `transport`         | `"stdio"`                                | `"stdio"` จะสร้าง Codex; `"websocket"` จะเชื่อมต่อกับ `url`                                                                                                                                                                             |
| `command`           | ไบนารี Codex ที่มีการจัดการ                     | ไฟล์ปฏิบัติการสำหรับทรานสปอร์ต stdio เว้นว่างไว้เพื่อใช้ไบนารีที่มีการจัดการ; ตั้งค่าเฉพาะเมื่อต้องการเขียนทับอย่างชัดเจน                                                                                                                         |
| `args`              | `["app-server", "--listen", "stdio://"]` | อาร์กิวเมนต์สำหรับทรานสปอร์ต stdio                                                                                                                                                                                                       |
| `url`               | ไม่ได้ตั้งค่า                                    | URL ของ app-server แบบ WebSocket                                                                                                                                                                                                            |
| `authToken`         | ไม่ได้ตั้งค่า                                    | โทเคน Bearer สำหรับทรานสปอร์ต WebSocket                                                                                                                                                                                                |
| `headers`           | `{}`                                     | เฮดเดอร์ WebSocket เพิ่มเติม                                                                                                                                                                                                             |
| `clearEnv`          | `[]`                                     | ชื่อตัวแปรสภาพแวดล้อมเพิ่มเติมที่ถูกลบออกจากกระบวนการ stdio app-server ที่ถูกสร้างขึ้น หลังจาก OpenClaw สร้างสภาพแวดล้อมที่สืบทอดแล้ว `CODEX_HOME` และ `HOME` ถูกสงวนไว้สำหรับการแยก Codex รายเอเจนต์ของ OpenClaw ในการเปิดใช้งานภายในเครื่อง |
| `requestTimeoutMs`  | `60000`                                  | หมดเวลาสำหรับการเรียก control-plane ของ app-server                                                                                                                                                                                          |
| `mode`              | `"yolo"`                                 | พรีเซ็ตสำหรับการดำเนินการแบบ YOLO หรือแบบที่ guardian ตรวจทาน                                                                                                                                                                                      |
| `approvalPolicy`    | `"never"`                                | นโยบายการอนุมัติแบบเนทีฟของ Codex ที่ส่งไปยังการเริ่ม/ดำเนินการต่อ/เทิร์นของเธรด                                                                                                                                                                       |
| `sandbox`           | `"danger-full-access"`                   | โหมดแซนด์บ็อกซ์แบบเนทีฟของ Codex ที่ส่งไปยังการเริ่ม/ดำเนินการต่อของเธรด                                                                                                                                                                               |
| `approvalsReviewer` | `"user"`                                 | ใช้ `"auto_review"` เพื่อให้ Codex ตรวจทานพรอมป์การอนุมัติแบบเนทีฟ `guardian_subagent` ยังคงเป็นชื่อแฝงแบบเดิม                                                                                                                         |
| `serviceTier`       | ไม่ได้ตั้งค่า                                    | ระดับบริการ app-server ของ Codex แบบเลือกได้: `"fast"`, `"flex"` หรือ `null` ค่าเดิมที่ไม่ถูกต้องจะถูกละเว้น                                                                                                                            |

การเรียกเครื่องมือแบบไดนามิกที่ OpenClaw เป็นเจ้าของมีขอบเขตแยกจาก
`appServer.requestTimeoutMs`: คำขอ `item/tool/call` ของ Codex แต่ละครั้งต้องได้รับ
การตอบกลับจาก OpenClaw ภายใน 30 วินาที เมื่อหมดเวลา OpenClaw จะยกเลิกสัญญาณเครื่องมือ
ในกรณีที่รองรับ และส่งคืนการตอบกลับเครื่องมือแบบไดนามิกที่ล้มเหลวไปยัง Codex เพื่อให้
เทิร์นดำเนินต่อได้แทนที่จะปล่อยให้เซสชันอยู่ในสถานะ `processing`

หลังจาก OpenClaw ตอบกลับคำขอ app-server ของ Codex ที่มีขอบเขตตามเทิร์นแล้ว harness
ยังคาดว่า Codex จะจบเทิร์นเนทีฟด้วย `turn/completed` ด้วย หาก
app-server เงียบไป 60 วินาทีหลังจากการตอบกลับนั้น OpenClaw จะพยายามอย่างดีที่สุด
ในการขัดจังหวะเทิร์นของ Codex บันทึกการหมดเวลาเพื่อวินิจฉัย และปล่อย lane ของเซสชัน
OpenClaw เพื่อไม่ให้ข้อความแชทติดตามผลถูกเข้าคิวอยู่หลังเทิร์นเนทีฟที่ค้าง

การเขียนทับสภาพแวดล้อมยังคงพร้อมใช้งานสำหรับการทดสอบภายในเครื่อง:

- `OPENCLAW_CODEX_APP_SERVER_BIN`
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

`OPENCLAW_CODEX_APP_SERVER_BIN` จะข้ามไบนารีที่มีการจัดการเมื่อ
`appServer.command` ไม่ได้ตั้งค่า

`OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1` ถูกนำออกแล้ว ใช้
`plugins.entries.codex.config.appServer.mode: "guardian"` แทน หรือ
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` สำหรับการทดสอบภายในเครื่องแบบครั้งเดียว ควรใช้การกำหนดค่า
สำหรับการปรับใช้ที่ทำซ้ำได้ เพราะจะเก็บพฤติกรรมของ Plugin ไว้ในไฟล์ที่ตรวจทานเดียวกัน
กับการตั้งค่า harness ของ Codex ที่เหลือ

## การใช้คอมพิวเตอร์

การใช้คอมพิวเตอร์ครอบคลุมอยู่ในคู่มือตั้งค่าเฉพาะของมัน:
[การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use)

สรุปสั้น ๆ: OpenClaw ไม่ได้นำแอปควบคุมเดสก์ท็อปมาเป็นส่วนหนึ่งของแพ็กเกจหรือดำเนินการ
บนเดสก์ท็อปด้วยตัวเอง แต่จะเตรียม app-server ของ Codex ตรวจสอบว่าเซิร์ฟเวอร์ MCP
`computer-use` พร้อมใช้งาน แล้วปล่อยให้ Codex จัดการการเรียกเครื่องมือ MCP แบบเนทีฟ
ระหว่างเทิร์นในโหมด Codex

สำหรับการเข้าถึงไดรเวอร์ TryCua โดยตรงนอกโฟลว์ marketplace ของ Codex ให้ลงทะเบียน
`cua-driver mcp` ด้วย `openclaw mcp set cua-driver '{"command":"cua-driver","args":["mcp"]}'`
ดู [การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use) สำหรับความแตกต่าง
ระหว่างการใช้คอมพิวเตอร์ที่ Codex เป็นเจ้าของกับการลงทะเบียน MCP โดยตรง

การกำหนดค่าขั้นต่ำ:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          computerUse: {
            autoInstall: true,
          },
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

สามารถตรวจสอบหรือติดตั้งการตั้งค่าได้จากพื้นผิวคำสั่ง:

- `/codex computer-use status`
- `/codex computer-use install`
- `/codex computer-use install --source <marketplace-source>`
- `/codex computer-use install --marketplace-path <path>`

การใช้คอมพิวเตอร์เฉพาะสำหรับ macOS และอาจต้องมีสิทธิ์ของระบบปฏิบัติการภายในเครื่องก่อนที่
เซิร์ฟเวอร์ MCP ของ Codex จะควบคุมแอปได้ หาก `computerUse.enabled` เป็น true และเซิร์ฟเวอร์ MCP
ไม่พร้อมใช้งาน เทิร์นในโหมด Codex จะล้มเหลวก่อนเริ่มเธรด แทนที่จะทำงานเงียบ ๆ
โดยไม่มีเครื่องมือการใช้คอมพิวเตอร์แบบเนทีฟ ดู
[การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use) สำหรับตัวเลือก marketplace,
ขีดจำกัดของแค็ตตาล็อกระยะไกล เหตุผลสถานะ และการแก้ไขปัญหา

เมื่อ `computerUse.autoInstall` เป็น true OpenClaw สามารถลงทะเบียน marketplace
Codex Desktop มาตรฐานที่มาพร้อมกันจาก
`/Applications/Codex.app/Contents/Resources/plugins/openai-bundled` ได้ หาก Codex
ยังไม่พบ marketplace ภายในเครื่อง ใช้ `/new` หรือ `/reset` หลังจากเปลี่ยน
การกำหนดค่า runtime หรือการใช้คอมพิวเตอร์ เพื่อไม่ให้เซสชันที่มีอยู่คงการผูกกับ PI หรือเธรด Codex เก่าไว้

## สูตรทั่วไป

Codex ภายในเครื่องพร้อมทรานสปอร์ต stdio ค่าเริ่มต้น:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

การตรวจสอบความถูกต้องของ harness เฉพาะ Codex:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

การอนุมัติของ Codex ที่ guardian ตรวจทาน:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            mode: "guardian",
            approvalPolicy: "on-request",
            approvalsReviewer: "auto_review",
            sandbox: "workspace-write",
          },
        },
      },
    },
  },
}
```

app-server ระยะไกลพร้อมเฮดเดอร์ที่ระบุอย่างชัดเจน:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            headers: {
              "X-OpenClaw-Agent": "main",
            },
          },
        },
      },
    },
  },
}
```

การสลับโมเดลยังคงถูกควบคุมโดย OpenClaw เมื่อเซสชัน OpenClaw ถูกแนบกับเธรด Codex
ที่มีอยู่ เทิร์นถัดไปจะส่งโมเดล OpenAI ที่เลือกอยู่ปัจจุบัน ผู้ให้บริการ นโยบายการอนุมัติ
แซนด์บ็อกซ์ และระดับบริการไปยัง app-server อีกครั้ง การสลับจาก `openai/gpt-5.5`
ไปเป็น `openai/gpt-5.2` จะคงการผูกเธรดไว้ แต่ขอให้ Codex ดำเนินการต่อด้วยโมเดลที่เลือกใหม่

## คำสั่ง Codex

Plugin ที่มาพร้อมกันลงทะเบียน `/codex` เป็นคำสั่ง slash ที่ได้รับอนุญาต คำสั่งนี้
เป็นแบบทั่วไปและใช้ได้กับทุกช่องทางที่รองรับคำสั่งข้อความของ OpenClaw

รูปแบบทั่วไป:

- `/codex status` แสดงการเชื่อมต่อ app-server แบบสด, โมเดล, บัญชี, rate limit, MCP servers และ skills.
- `/codex models` แสดงรายการโมเดล Codex app-server แบบสด.
- `/codex threads [filter]` แสดงรายการเธรด Codex ล่าสุด.
- `/codex resume <thread-id>` แนบเซสชัน OpenClaw ปัจจุบันเข้ากับเธรด Codex ที่มีอยู่.
- `/codex compact` ขอให้ Codex app-server compact เธรดที่แนบอยู่.
- `/codex review` เริ่มการ review แบบ native ของ Codex สำหรับเธรดที่แนบอยู่.
- `/codex diagnostics [note]` ถามก่อนส่ง feedback การวินิจฉัย Codex สำหรับเธรดที่แนบอยู่.
- `/codex computer-use status` ตรวจสอบ Plugin Computer Use และ MCP server ที่กำหนดค่าไว้.
- `/codex computer-use install` ติดตั้ง Plugin Computer Use ที่กำหนดค่าไว้และโหลด MCP servers ใหม่.
- `/codex account` แสดงสถานะบัญชีและ rate-limit.
- `/codex mcp` แสดงรายการสถานะ MCP server ของ Codex app-server.
- `/codex skills` แสดงรายการ skills ของ Codex app-server.

### เวิร์กโฟลว์การดีบักทั่วไป

เมื่อ agent ที่มี Codex เป็นแบ็กเอนด์ทำสิ่งที่ไม่คาดคิดใน Telegram, Discord, Slack,
หรือช่องทางอื่น ให้เริ่มจากบทสนทนาที่เกิดปัญหา:

1. เรียกใช้ `/diagnostics bad tool choice after image upload` หรือหมายเหตุสั้น ๆ อื่น
   ที่อธิบายสิ่งที่คุณเห็น.
2. อนุมัติคำขอวินิจฉัยหนึ่งครั้ง การอนุมัติจะสร้างไฟล์ zip การวินิจฉัย Gateway
   ในเครื่อง และเนื่องจากเซสชันใช้ harness ของ Codex จึงส่งชุด feedback Codex
   ที่เกี่ยวข้องไปยังเซิร์ฟเวอร์ OpenAI ด้วย.
3. คัดลอกคำตอบการวินิจฉัยที่เสร็จแล้วไปยังรายงานบั๊กหรือเธรดสนับสนุน.
   คำตอบนั้นมีพาธชุดข้อมูลในเครื่อง, สรุปความเป็นส่วนตัว, id เซสชัน OpenClaw,
   id เธรด Codex และบรรทัด `Inspect locally` สำหรับแต่ละเธรด Codex.
4. หากคุณต้องการดีบักการรันด้วยตนเอง ให้รันคำสั่ง `Inspect locally`
   ที่พิมพ์ไว้ในเทอร์มินัล คำสั่งจะมีรูปแบบเช่น `codex resume <thread-id>` และเปิด
   เธรด Codex แบบ native เพื่อให้คุณตรวจสอบบทสนทนา, ทำต่อในเครื่อง,
   หรือถาม Codex ว่าทำไมจึงเลือกเครื่องมือหรือแผนหนึ่ง ๆ.

ใช้ `/codex diagnostics [note]` เฉพาะเมื่อคุณต้องการอัปโหลด feedback Codex
สำหรับเธรดที่แนบอยู่ในปัจจุบันโดยไม่รวมชุดการวินิจฉัย Gateway ของ OpenClaw
แบบเต็ม สำหรับรายงานสนับสนุนส่วนใหญ่ `/diagnostics [note]` เป็นจุดเริ่มต้น
ที่ดีกว่า เพราะผูกสถานะ Gateway ในเครื่องและ id เธรด Codex เข้าด้วยกันในคำตอบเดียว ดู [การส่งออกการวินิจฉัย](/th/gateway/diagnostics)
สำหรับโมเดลความเป็นส่วนตัวและพฤติกรรมในแชตกลุ่มทั้งหมด.

แกนหลักของ OpenClaw ยังเปิดเผย `/diagnostics [note]` เฉพาะเจ้าของในฐานะคำสั่งการวินิจฉัย
Gateway ทั่วไปด้วย prompt การอนุมัติของคำสั่งนี้แสดงคำนำเกี่ยวกับข้อมูลอ่อนไหว,
ลิงก์ไปยัง [Diagnostics Export](/th/gateway/diagnostics), และร้องขอ
`openclaw gateway diagnostics export --json` ผ่านการอนุมัติ exec อย่างชัดเจน
ทุกครั้ง อย่าอนุมัติการวินิจฉัยด้วยกฎ allow-all หลังอนุมัติแล้ว
OpenClaw จะส่งรายงานที่วางต่อได้พร้อมพาธชุดข้อมูลในเครื่องและสรุป manifest
เมื่อเซสชัน OpenClaw ที่ใช้งานอยู่ใช้ harness ของ Codex การอนุมัติเดียวกันนั้น
จะอนุญาตให้ส่งชุด feedback Codex ที่เกี่ยวข้องไปยังเซิร์ฟเวอร์ OpenAI ด้วย
prompt การอนุมัติระบุว่าจะส่ง feedback Codex แต่จะไม่แสดง id เซสชันหรือเธรด
Codex ก่อนการอนุมัติ.

หาก `/diagnostics` ถูกเรียกโดยเจ้าของในแชตกลุ่ม OpenClaw จะรักษาช่องทางร่วม
ให้สะอาด: กลุ่มจะได้รับเพียงประกาศสั้น ๆ ขณะที่คำนำการวินิจฉัย,
prompt การอนุมัติ และ id เซสชัน/เธรด Codex จะถูกส่งไปยังเจ้าของผ่านเส้นทาง
การอนุมัติแบบส่วนตัว หากไม่มีเส้นทางเจ้าของแบบส่วนตัว OpenClaw จะปฏิเสธคำขอกลุ่ม
และขอให้เจ้าของรันจาก DM.

การอัปโหลด Codex ที่อนุมัติแล้วจะเรียก Codex app-server `feedback/upload` และขอให้
app-server รวม log สำหรับแต่ละเธรดที่ระบุและ subthread Codex ที่ถูกสร้างขึ้น
เมื่อพร้อมใช้งาน การอัปโหลดผ่านเส้นทาง feedback ปกติของ Codex ไปยังเซิร์ฟเวอร์
OpenAI; หาก feedback Codex ถูกปิดใช้งานใน app-server นั้น คำสั่งจะส่งคืน
ข้อผิดพลาดของ app-server คำตอบการวินิจฉัยที่เสร็จแล้วจะแสดงรายการช่องทาง,
id เซสชัน OpenClaw, id เธรด Codex และคำสั่ง `codex resume <thread-id>`
ในเครื่องสำหรับเธรดที่ถูกส่ง หากคุณปฏิเสธหรือเพิกเฉยต่อการอนุมัติ
OpenClaw จะไม่พิมพ์ id Codex เหล่านั้น การอัปโหลดนี้ไม่แทนที่การส่งออก
การวินิจฉัย Gateway ในเครื่อง.

`/codex resume` เขียนไฟล์ binding sidecar เดียวกับที่ harness ใช้สำหรับ turn ปกติ
ในข้อความถัดไป OpenClaw จะ resume เธรด Codex นั้น ส่งโมเดล OpenClaw
ที่เลือกในปัจจุบันเข้าไปยัง app-server และคงการเปิดใช้ประวัติแบบขยายไว้.

### ตรวจสอบเธรด Codex จาก CLI

วิธีที่เร็วที่สุดในการทำความเข้าใจการรัน Codex ที่ผิดพลาดมักเป็นการเปิดเธรด Codex
แบบ native โดยตรง:

```sh
codex resume <thread-id>
```

ใช้สิ่งนี้เมื่อคุณสังเกตเห็นบั๊กในบทสนทนาของช่องทางและต้องการตรวจสอบ
เซสชัน Codex ที่เป็นปัญหา, ทำต่อในเครื่อง, หรือถาม Codex ว่าทำไมจึงเลือก
เครื่องมือหรือการให้เหตุผลแบบหนึ่ง เส้นทางที่ง่ายที่สุดมักเป็นการรัน
`/diagnostics [note]` ก่อน: หลังคุณอนุมัติแล้ว รายงานที่เสร็จแล้วจะแสดงรายการ
แต่ละเธรด Codex และพิมพ์คำสั่ง `Inspect locally` เช่น
`codex resume <thread-id>` คุณสามารถคัดลอกคำสั่งนั้นลงในเทอร์มินัลได้โดยตรง.

คุณยังสามารถรับ id เธรดจาก `/codex binding` สำหรับแชตปัจจุบันหรือ
`/codex threads [filter]` สำหรับเธรด Codex app-server ล่าสุด จากนั้นรันคำสั่ง
`codex resume` เดียวกันใน shell ของคุณ.

พื้นผิวคำสั่งต้องใช้ Codex app-server `0.125.0` หรือใหม่กว่า วิธีควบคุมแต่ละรายการ
จะถูกรายงานเป็น `unsupported by this Codex app-server` หาก app-server
ในอนาคตหรือแบบกำหนดเองไม่เปิดเผยวิธี JSON-RPC นั้น.

## ขอบเขต hook

harness ของ Codex มี hook สามชั้น:

| ชั้น                                  | เจ้าของ                  | วัตถุประสงค์                                                         |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| hook ของ Plugin OpenClaw              | OpenClaw                 | ความเข้ากันได้ของผลิตภัณฑ์/Plugin ระหว่าง harness ของ PI และ Codex. |
| middleware ส่วนขยายของ Codex app-server | Plugin ที่ bundled มากับ OpenClaw | พฤติกรรม adapter ต่อ turn รอบเครื่องมือ dynamic ของ OpenClaw.       |
| hook แบบ native ของ Codex             | Codex                    | lifecycle ระดับต่ำของ Codex และนโยบายเครื่องมือ native จาก config ของ Codex. |

OpenClaw ไม่ใช้ไฟล์ `hooks.json` ของโปรเจกต์หรือแบบ global ของ Codex เพื่อ route
พฤติกรรม Plugin ของ OpenClaw สำหรับ bridge เครื่องมือ native และสิทธิ์ที่รองรับ
OpenClaw inject config Codex ต่อเธรดสำหรับ `PreToolUse`, `PostToolUse`,
`PermissionRequest`, และ `Stop` hook อื่นของ Codex เช่น `SessionStart` และ
`UserPromptSubmit` ยังคงเป็นการควบคุมระดับ Codex; สิ่งเหล่านี้ไม่ถูกเปิดเผยเป็น
hook ของ Plugin OpenClaw ในสัญญา v1.

สำหรับเครื่องมือ dynamic ของ OpenClaw, OpenClaw จะ execute เครื่องมือหลังจาก Codex ขอ
call ดังนั้น OpenClaw จึงยิงพฤติกรรม Plugin และ middleware ที่ตนเป็นเจ้าของใน
adapter ของ harness สำหรับเครื่องมือแบบ native ของ Codex, Codex เป็นเจ้าของ record
เครื่องมือ canonical OpenClaw สามารถ mirror event ที่เลือกได้ แต่ไม่สามารถ rewrite
เธรด Codex แบบ native เว้นแต่ Codex จะเปิดเผยการดำเนินการนั้นผ่าน app-server หรือ
callback ของ hook แบบ native.

projection ของ Compaction และ lifecycle ของ LLM มาจาก notification ของ Codex app-server
และสถานะ adapter ของ OpenClaw ไม่ใช่คำสั่ง hook แบบ native ของ Codex
event `before_compaction`, `after_compaction`, `llm_input`, และ
`llm_output` ของ OpenClaw เป็น observation ระดับ adapter ไม่ใช่ capture
payload คำขอภายในหรือ compaction ของ Codex แบบ byte-for-byte.

notification `hook/started` และ `hook/completed` ของ Codex native ใน app-server
ถูก project เป็น event agent `codex_app_server.hook` สำหรับ trajectory และการดีบัก
สิ่งเหล่านี้ไม่ invoke hook ของ Plugin OpenClaw.

## สัญญาการรองรับ V1

โหมด Codex ไม่ใช่ PI ที่มีการเรียกโมเดลคนละแบบอยู่ข้างใต้ Codex เป็นเจ้าของ
model loop แบบ native มากกว่า และ OpenClaw ปรับพื้นผิว Plugin และเซสชันของตน
รอบขอบเขตนั้น.

รองรับใน runtime Codex v1:

| พื้นผิว                                       | การรองรับ                                | เหตุผล                                                                                                                                                                                                 |
| --------------------------------------------- | --------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| OpenAI model loop ผ่าน Codex                  | รองรับ                                  | Codex app-server เป็นเจ้าของ turn ของ OpenAI, การ resume เธรด native และการ continuation เครื่องมือ native.                                                                                          |
| การ route และส่งมอบช่องทางของ OpenClaw        | รองรับ                                  | Telegram, Discord, Slack, WhatsApp, iMessage และช่องทางอื่นยังคงอยู่นอก runtime ของโมเดล.                                                                                                           |
| เครื่องมือ dynamic ของ OpenClaw               | รองรับ                                  | Codex ขอให้ OpenClaw execute เครื่องมือเหล่านี้ ดังนั้น OpenClaw จึงยังอยู่ในเส้นทางการ execute.                                                                                                    |
| Plugin prompt และ context                     | รองรับ                                  | OpenClaw สร้าง prompt overlay และ project context เข้าไปใน turn ของ Codex ก่อนเริ่มหรือ resume เธรด.                                                                                                |
| lifecycle ของ context engine                  | รองรับ                                  | การ assemble, ingest หรือ maintenance หลัง turn และการประสานงาน compaction ของ context-engine จะทำงานสำหรับ turn ของ Codex.                                                                         |

## เครื่องมือ สื่อ และ Compaction

Codex harness เปลี่ยนเฉพาะ agent executor แบบฝังระดับต่ำเท่านั้น

OpenClaw ยังคงสร้างรายการเครื่องมือและรับผลลัพธ์เครื่องมือแบบไดนามิกจาก
harness ข้อความ รูปภาพ วิดีโอ เพลง TTS การอนุมัติ และเอาต์พุตของเครื่องมือส่งข้อความ
ยังคงผ่านเส้นทางการส่งมอบปกติของ OpenClaw

native hook relay ตั้งใจให้เป็นแบบทั่วไป แต่สัญญาการรองรับ v1
จำกัดอยู่ที่เส้นทางเครื่องมือและสิทธิ์แบบ Codex-native ที่ OpenClaw ทดสอบ ใน
runtime ของ Codex ซึ่งรวมถึง payload ของ shell, patch และ MCP `PreToolUse`,
`PostToolUse` และ `PermissionRequest` อย่าสันนิษฐานว่าเหตุการณ์ hook ของ Codex
ในอนาคตทุกอย่างเป็นพื้นผิว Plugin ของ OpenClaw จนกว่า runtime contract จะระบุชื่อ
สิ่งนั้น

สำหรับ `PermissionRequest` OpenClaw จะส่งคืนการตัดสินใจ allow หรือ deny อย่างชัดเจน
เฉพาะเมื่อ policy ตัดสินใจ ผลลัพธ์แบบไม่มีการตัดสินใจไม่ใช่การ allow Codex ถือว่าเป็นการไม่มี
การตัดสินใจจาก hook และส่งต่อไปยังเส้นทาง guardian หรือการอนุมัติจากผู้ใช้ของตัวเอง

การขออนุมัติเครื่องมือ Codex MCP จะถูกส่งผ่าน flow การอนุมัติ Plugin ของ OpenClaw
เมื่อ Codex ทำเครื่องหมาย `_meta.codex_approval_kind` เป็น
`"mcp_tool_call"` prompt ของ Codex `request_user_input` จะถูกส่งกลับไปยัง
แชตต้นทาง และข้อความติดตามผลถัดไปในคิวจะตอบคำขอ server แบบเนทีฟนั้น
แทนที่จะถูกบังคับทิศทางเป็น context เพิ่มเติม คำขอ elicitation อื่นของ MCP
ยังคงปิดล้มเหลว

การบังคับทิศทางคิวของ active-run map เข้ากับ Codex app-server `turn/steer` ด้วย
ค่าเริ่มต้น `messages.queue.mode: "steer"` OpenClaw จะ batch ข้อความแชตในคิว
ตามช่วง quiet window ที่กำหนดค่าไว้ แล้วส่งเป็นคำขอ `turn/steer` เดียวตาม
ลำดับที่มาถึง โหมด legacy `queue` จะส่งคำขอ `turn/steer` แยกกัน turn สำหรับ review และ
Compaction แบบ manual ของ Codex สามารถปฏิเสธ same-turn steering ได้ ซึ่งในกรณีนั้น
OpenClaw จะใช้ followup queue เมื่อโหมดที่เลือกอนุญาต fallback ดู
[คิว Steering](/th/concepts/queue-steering)

เมื่อ model ที่เลือกใช้ Codex harness, Compaction ของ thread เนทีฟจะถูก
มอบหมายให้ Codex app-server OpenClaw เก็บ transcript mirror สำหรับประวัติ channel,
การค้นหา, `/new`, `/reset` และการสลับ model หรือ harness ในอนาคต
mirror รวมถึง prompt ของผู้ใช้ ข้อความ assistant สุดท้าย และระเบียน reasoning หรือ plan
แบบเบาของ Codex เมื่อ app-server ปล่อยออกมา วันนี้ OpenClaw บันทึกเฉพาะ
สัญญาณเริ่มต้นและเสร็จสิ้น Compaction เนทีฟเท่านั้น ยังไม่ได้เปิดเผย
สรุป Compaction ที่มนุษย์อ่านได้ หรือรายการที่ audit ได้ว่า Codex
เก็บรายการใดไว้หลัง Compaction

เนื่องจาก Codex เป็นเจ้าของ thread เนทีฟ canonical, `tool_result_persist` จึงยังไม่
เขียนระเบียนผลลัพธ์เครื่องมือแบบ Codex-native ใหม่ในขณะนี้ ใช้เฉพาะเมื่อ
OpenClaw กำลังเขียนผลลัพธ์เครื่องมือ transcript ของ session ที่ OpenClaw เป็นเจ้าของ

การสร้างสื่อไม่ต้องใช้ PI รูปภาพ วิดีโอ เพลง PDF TTS และความเข้าใจสื่อ
ยังคงใช้การตั้งค่า provider/model ที่ตรงกัน เช่น
`agents.defaults.imageGenerationModel`, `videoGenerationModel`, `pdfModel` และ
`messages.tts`

## การแก้ไขปัญหา

**Codex ไม่ปรากฏเป็น provider `/model` ปกติ:** นี่เป็นสิ่งที่คาดไว้สำหรับ
config ใหม่ เลือก model `openai/gpt-*` พร้อม
`agentRuntime.id: "codex"` (หรือ ref legacy `codex/*`), เปิดใช้
`plugins.entries.codex.enabled` และตรวจสอบว่า `plugins.allow` ไม่ได้ยกเว้น
`codex`

**OpenClaw ใช้ PI แทน Codex:** `agentRuntime.id: "auto"` ยังสามารถใช้ PI เป็น
backend ความเข้ากันได้เมื่อไม่มี Codex harness ใดรับ run ตั้งค่า
`agentRuntime.id: "codex"` เพื่อบังคับการเลือก Codex ระหว่างทดสอบ runtime ของ Codex
ที่ถูกบังคับจะล้มเหลวแทนที่จะ fallback ไปยัง PI เมื่อเลือก Codex app-server แล้ว
ความล้มเหลวของมันจะแสดงโดยตรง

**app-server ถูกปฏิเสธ:** อัปเกรด Codex เพื่อให้ handshake ของ app-server
รายงานเวอร์ชัน `0.125.0` หรือใหม่กว่า prerelease เวอร์ชันเดียวกันหรือเวอร์ชันที่มี suffix build
เช่น `0.125.0-alpha.2` หรือ `0.125.0+custom` จะถูกปฏิเสธ เพราะ
stable `0.125.0` protocol floor คือสิ่งที่ OpenClaw ทดสอบ

**การค้นพบ model ช้า:** ลด `plugins.entries.codex.config.discovery.timeoutMs`
หรือปิดการค้นพบ

**WebSocket transport ล้มเหลวทันที:** ตรวจสอบ `appServer.url`, `authToken`,
และตรวจสอบว่า app-server ระยะไกลพูด protocol version เดียวกันของ Codex app-server

**model ที่ไม่ใช่ Codex ใช้ PI:** นี่เป็นสิ่งที่คาดไว้ เว้นแต่คุณจะบังคับ
`agentRuntime.id: "codex"` สำหรับ agent นั้น หรือเลือก ref legacy
`codex/*` ref แบบธรรมดา `openai/gpt-*` และ provider อื่นจะยังอยู่บนเส้นทาง
provider ปกติในโหมด `auto` หากคุณบังคับ `agentRuntime.id: "codex"` ทุก turn แบบฝัง
สำหรับ agent นั้นต้องเป็น model OpenAI ที่ Codex รองรับ

**Computer Use ติดตั้งแล้วแต่เครื่องมือไม่ทำงาน:** ตรวจสอบ
`/codex computer-use status` จาก session ใหม่ หากเครื่องมือรายงาน
`Native hook relay unavailable` ให้ใช้ `/new` หรือ `/reset`; หากยังคงอยู่ ให้ restart
gateway เพื่อล้างการลงทะเบียน native hook ที่ค้าง หาก `computer-use.list_apps`
timeout ให้ restart Codex Computer Use หรือ Codex Desktop แล้วลองใหม่
