ปลั๊กอิน `codex` อย่างเป็นทางการเรียกใช้รอบการทำงานของเอเจนต์ OpenAI แบบฝังผ่าน Codex
app-server แทนชุดควบคุม OpenClaw ในตัว โดย Codex เป็นผู้จัดการ
เซสชันเอเจนต์ระดับล่าง ได้แก่ การดำเนินเธรดต่อแบบเนทีฟ การดำเนินเครื่องมือต่อแบบเนทีฟ
Compaction แบบเนทีฟ และการประมวลผลผ่าน app-server ส่วน OpenClaw ยังคงจัดการช่องทาง
แชต ไฟล์เซสชัน การเลือกโมเดล เครื่องมือแบบไดนามิกของ OpenClaw การอนุมัติ
การส่งสื่อ และสำเนาทรานสคริปต์ที่มองเห็นได้

ใช้การอ้างอิงโมเดล OpenAI มาตรฐาน เช่น `openai/gpt-5.6-sol` อย่ากำหนดค่า
การอ้างอิง Codex GPT แบบเดิม ให้ระบุลำดับการยืนยันตัวตนของเอเจนต์ OpenAI ไว้ภายใต้ `auth.order.openai`
รหัสโปรไฟล์การยืนยันตัวตน Codex แบบเดิมและรายการลำดับการยืนยันตัวตน Codex แบบเดิมจะได้รับ
การซ่อมแซมโดย `openclaw doctor --fix`

เมื่อไม่ได้ตั้งค่านโยบายรันไทม์ของผู้ให้บริการ/โมเดลหรือกำหนดเป็น `auto` คำนำหน้า `openai/*` เพียงอย่างเดียว
จะไม่เลือกชุดควบคุมนี้ OpenAI อาจเลือก Codex โดยนัยได้เฉพาะเมื่อเป็น
เส้นทาง Platform Responses หรือ ChatGPT Responses ผ่าน HTTPS อย่างเป็นทางการที่ตรงกันทุกประการ โดยไม่มี
การเขียนทับคำขอที่ผู้ใช้กำหนดไว้ ดู
[รันไทม์เอเจนต์โดยนัยของ OpenAI](/th/providers/openai#implicit-agent-runtime)
หาก Codex เป็นผู้จัดการการยืนยันตัวตนก่อนจะทราบการกำหนดเส้นทางระหว่าง Platform กับ ChatGPT ทาง OpenClaw
ยังคงกำหนดให้ทุกเส้นทางที่เป็นตัวเลือกต้องประกาศความเข้ากันได้กับ Codex
การเป็นเจ้าของการยืนยันตัวตนแบบเนทีฟเพียงอย่างเดียวจะไม่ข้ามการตรวจสอบเส้นทางนั้น

เมื่อไม่มีแซนด์บ็อกซ์ OpenClaw ทำงานอยู่ OpenClaw จะเริ่มเธรด Codex app-server
โดยเปิดใช้โหมดโค้ดแบบเนทีฟของ Codex (ค่าเริ่มต้นยังคงปิดโหมด code-mode-only) เพื่อให้
ความสามารถด้านพื้นที่ทำงาน/โค้ดแบบเนทีฟยังคงใช้งานได้ควบคู่กับเครื่องมือแบบไดนามิกของ OpenClaw
ที่กำหนดเส้นทางผ่านบริดจ์ `item/tool/call` ของ app-server แซนด์บ็อกซ์ OpenClaw ที่ทำงานอยู่
หรือนโยบายเครื่องมือแบบจำกัดจะปิดใช้งานโหมดโค้ดแบบเนทีฟทั้งหมด เว้นแต่จะเลือกใช้
เส้นทาง exec-server ของแซนด์บ็อกซ์แบบทดลอง

เมื่อใช้ค่าเริ่มต้น `tools.exec.host: "auto"` และไม่มีแซนด์บ็อกซ์ OpenClaw ทำงานอยู่
Codex จะได้รับเครื่องมือ `node_exec` และ `node_process` สำหรับคำสั่งบน Node
ที่จับคู่ไว้ด้วย เชลล์แบบเนทีฟยังคงทำงานบนโฮสต์และพื้นที่ทำงานของ Codex app-server
(อยู่ในเครื่อง Gateway สำหรับการติดตั้งใช้งาน stdio เริ่มต้น) ส่วน `node_exec` จะเลือก Node ด้วย
ชื่อหรือรหัส และยังคงบังคับใช้นโยบายการอนุมัติ Node ของ OpenClaw หากรายการอนุญาต
ของรันไทม์แบบจำกัดปิดใช้งาน Code Mode แบบเนทีฟและทำให้รอบการทำงานไม่มี
สภาพแวดล้อมการประมวลผล OpenClaw จะยังคงมีเครื่องมือ `exec` และ `process`
ที่กรองตามนโยบายให้ใช้งานแทน สำหรับการประมวลผลโดยตรงแบบไม่ใช้แซนด์บ็อกซ์

คุณลักษณะแบบเนทีฟของ Codex นี้แยกจาก
[โหมดโค้ดของ OpenClaw](/th/reference/code-mode) ซึ่งเป็นรันไทม์ QuickJS-WASI แบบเลือกใช้
สำหรับการทำงานทั่วไปของ OpenClaw ที่มีรูปแบบอินพุต `exec` ต่างกัน สำหรับภาพรวม
การแบ่งแยกโมเดล/ผู้ให้บริการ/รันไทม์ โปรดเริ่มจาก
[รันไทม์เอเจนต์](/th/concepts/agent-runtimes): `openai/gpt-5.6-sol` คือการอ้างอิง
โมเดล `codex` คือรันไทม์ และ Telegram, Discord, Slack หรือช่องทางอื่น
คือพื้นผิวการสื่อสาร

## ข้อกำหนด

- ติดตั้งปลั๊กอิน `@openclaw/codex` อย่างเป็นทางการแล้ว หากการกำหนดค่าใช้รายการอนุญาต ให้รวม `codex` ไว้ใน
  `plugins.allow`
- Codex app-server `0.143.0` หรือใหม่กว่า ปลั๊กอินจะจัดการไบนารีที่เข้ากันได้
  โดยค่าเริ่มต้น ดังนั้นคำสั่ง `codex` บน `PATH` จึงไม่ส่งผลต่อการเริ่มทำงาน
  ตามปกติ
- การยืนยันตัวตน Codex ผ่าน `openclaw models auth login --provider openai`
  บัญชี app-server ที่มีอยู่แล้วใน Codex home ของเอเจนต์ หรือ
  โปรไฟล์การยืนยันตัวตนด้วยคีย์ API ของ Codex ที่ระบุอย่างชัดเจน

สำหรับลำดับความสำคัญของการยืนยันตัวตน การแยกสภาพแวดล้อม คำสั่ง app-server แบบกำหนดเอง
การค้นหาโมเดล และรายการฟิลด์การกำหนดค่าทั้งหมด โปรดดู
[ข้อมูลอ้างอิงชุดควบคุม Codex](/th/plugins/codex-harness-reference)

## เริ่มต้นอย่างรวดเร็ว

ติดตั้งปลั๊กอินอย่างเป็นทางการ แล้วลงชื่อเข้าใช้ด้วย Codex OAuth:

```bash
openclaw plugins install @openclaw/codex
openclaw models auth login --provider openai
```

เปิดใช้ปลั๊กอิน `codex` และเลือกโมเดลเอเจนต์ OpenAI:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

หากการกำหนดค่าใช้ `plugins.allow` ให้เพิ่ม `codex` ไว้ที่นั่นด้วย:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

รีสตาร์ต Gateway หลังจากเปลี่ยนการกำหนดค่าปลั๊กอิน หากแชตมี
เซสชันอยู่แล้ว ให้เรียกใช้ `/new` หรือ `/reset` ก่อน เพื่อให้รอบการทำงานถัดไปแก้ไขชุดควบคุม
จากการกำหนดค่าปัจจุบัน

## แชร์เธรดกับ Codex Desktop และ CLI

ค่าเริ่มต้น `appServer.homeScope: "agent"` จะแยกเอเจนต์ OpenClaw แต่ละตัวออกจาก
สถานะ Codex แบบเนทีฟของผู้ดำเนินการ หากต้องการให้เจ้าของตรวจสอบและจัดการ
เธรดแบบเนทีฟเดียวกับที่แสดงโดย Codex Desktop และ Codex CLI ให้เลือกใช้
Codex home ของผู้ใช้:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            homeScope: "user",
          },
        },
      },
    },
  },
}
```

โหมด home ของผู้ใช้รองรับโพรเซส stdio ที่จัดการภายในเครื่องหรือทรานสปอร์ต Unix-socket
ที่ใช้ร่วมกัน โดยใช้ `$CODEX_HOME` เมื่อตั้งค่าไว้ และใช้ `~/.codex` ในกรณีอื่น รวมถึง
การยืนยันตัวตน การกำหนดค่า ปลั๊กอิน และที่เก็บเธรดแบบเนทีฟของ Codex ใน home นั้น OpenClaw
จะไม่แทรกโปรไฟล์การยืนยันตัวตน OpenClaw เข้าไปใน app-server นี้

รอบการทำงานของเจ้าของจะได้รับเครื่องมือ `codex_threads`: แสดงรายการ ค้นหา อ่าน ฟอร์ก เปลี่ยนชื่อ
เก็บถาวร และกู้คืนเธรดแบบเนทีฟ ฟอร์กเธรดเพื่อดำเนินการต่อใน
OpenClaw โดยฟอร์กจะเชื่อมกับเซสชัน OpenClaw ปัจจุบันและยังคง
มองเห็นได้สำหรับไคลเอนต์ Codex แบบเนทีฟอื่น การเก็บถาวรต้องมีการยืนยันอย่างชัดเจน
ว่าเธรดถูกปิดในที่อื่นแล้ว เมื่อเปิดใช้การกำกับดูแลด้วย ฟิลด์ทรานสคริปต์
และการเปลี่ยนแปลงต้องเปิดใช้ `supervision.allowRawTranscripts` หรือ `supervision.allowWriteControls` ที่ตรงกัน

อย่าดำเนินการต่อหรือเขียนเธรดเดียวกันพร้อมกันผ่าน App Server แบบ stdio
ที่จัดการโดยโพรเซสอิสระ Codex จะประสานผู้เขียนที่ทำงานอยู่ภายใน App Server เดียวกัน ไม่ใช่
ข้ามโพรเซสที่แยกจากกัน การฟอร์กเป็นเส้นทางการทำงานร่วมกันอย่างปลอดภัยสำหรับเซสชัน stdio
ใน home ของผู้ใช้โดยทั่วไป

`appServer.homeScope: "user"` เพียงอย่างเดียวไม่ได้ควบคุมแค็ตตาล็อกของระบบทั้งหมด การค้นหา
เซสชันแบบเนทีฟจะเปิดใช้งานขณะที่ปลั๊กอินทำงานอยู่ ให้ตั้งค่า
`sessionCatalog.enabled: false` เพื่อนำออกจากแถบด้านข้างของ OpenClaw โดยไม่
ปิดใช้งาน Codex แค็ตตาล็อกใช้การเชื่อมต่อสำหรับการกำกับดูแลที่แยกต่างหาก หากไม่มี
การตั้งค่าการเชื่อมต่อ `appServer` อย่างชัดเจน การเชื่อมต่อนั้นจะใช้ค่าเริ่มต้นเป็น stdio
ใน home ของผู้ใช้ที่มีการจัดการ ขณะที่ชุดควบคุมทั่วไปยังคงมีขอบเขตเฉพาะเอเจนต์ การตั้งค่า
`appServer` อย่างชัดเจนจะมีผลกับทั้งสองเส้นทาง ให้ตั้งค่า `homeScope: "user"`
อย่างชัดเจนดังตัวอย่างข้างต้น เมื่อต้องการให้ชุดควบคุมทั่วไปแชร์สถานะแบบเนทีฟด้วย

## กำกับดูแลเซสชัน Codex

ปลั๊กอิน `codex` เดียวกันสามารถแสดงรายการเซสชัน Codex ที่ไม่ได้เก็บถาวรจากคอมพิวเตอร์
Gateway และ Node ที่จับคู่ซึ่งเลือกเข้าร่วม เซสชันที่จัดเก็บไว้หรือไม่ได้ใช้งานซึ่งอยู่ในเครื่อง Gateway สามารถ
สร้างแชตที่ล็อกโมเดล ซึ่งสะท้อนประวัติผู้ใช้และผู้ช่วยที่จัดเก็บถาวรแบบจำกัดขอบเขต
การผูกแบบส่วนตัวของแชตจะใช้การเชื่อมต่อสำหรับการกำกับดูแลเพื่อรับสแนปช็อตแบบเนทีฟ
บรานช์มาตรฐาน และรอบการทำงานภายหลัง ขณะที่เซสชัน Codex ทั่วไปยังคงมีขอบเขตเฉพาะ
เอเจนต์ การเริ่มบรานช์มาตรฐานครั้งแรกจะใช้โมเดลและผู้ให้บริการที่
Codex ส่งคืนสำหรับฟอร์กของสแนปช็อตทุกประการ การดำเนินการต่อในภายหลังจะปล่อยให้การเลือกเป็นไปตาม
การกำหนดค่าแบบเนทีฟของ Codex โดยโมเดล OpenClaw ภายนอกและสายโซ่สำรองจะไม่เข้ามาแทนที่
แถวที่จัดเก็บไว้และไม่ได้ใช้งานสามารถเก็บถาวรได้หลังจากยืนยันอย่างชัดเจนว่าไม่มีตัวดำเนินการอื่น
แหล่งที่ทำงานอยู่ไม่สามารถสร้างบรานช์หรือเก็บถาวรได้ แต่ยังคงเปิด
แชตที่กำกับดูแลซึ่งมีอยู่แล้วได้ เซสชันบน Node ที่จับคู่จะยังคงเป็นเฉพาะข้อมูลเมตา

ดู [กำกับดูแลเซสชัน Codex](/plugins/codex-supervision) สำหรับการตั้งค่า กฎการสร้างบรานช์
ขีดจำกัดของ Node ที่จับคู่ การเปิดเผยข้อมูลเมตา และการแก้ไขปัญหา

| ลงชื่อเข้าใช้ด้วย ChatGPT/Codex OAuth               | `openclaw models auth login --provider openai`                                                   | โปรไฟล์การยืนยันตัวตน CLI          |
| เพิ่มคีย์ API สำรองสำหรับการทำงานของ Codex          | โปรไฟล์คีย์ API `openai:*` ที่แสดงไว้หลังการยืนยันตัวตนแบบสมัครสมาชิกใน `auth.order.openai`                 | โปรไฟล์การยืนยันตัวตน CLI + การกำหนดค่า OpenClaw |
| ปิดแบบปลอดภัยเมื่อ Codex ไม่พร้อมใช้งาน             | ผู้ให้บริการหรือโมเดล `agentRuntime.id: "codex"`                                                     | การกำหนดค่าโมเดล/ผู้ให้บริการ OpenClaw |
| ใช้การรับส่งข้อมูล OpenAI API โดยตรง                | ผู้ให้บริการหรือโมเดล `agentRuntime.id: "openclaw"` พร้อมการยืนยันตัวตน OpenAI ตามปกติ                          | การกำหนดค่าโมเดล/ผู้ให้บริการ OpenClaw |
| ปรับแต่งพฤติกรรม app-server                         | `plugins.entries.codex.config.appServer.*`                                                       | การกำหนดค่าปลั๊กอิน Codex          |
| เปิดใช้แอปปลั๊กอิน Codex แบบเนทีฟ                   | `plugins.entries.codex.config.codexPlugins.*`                                                    | การกำหนดค่าปลั๊กอิน Codex          |
| เปิดใช้ Codex Computer Use                          | `plugins.entries.codex.config.computerUse.*`                                                     | การกำหนดค่าปลั๊กอิน Codex          |

ควรใช้ `auth.order.openai` สำหรับการจัดลำดับแบบใช้การสมัครสมาชิกก่อน/ใช้คีย์ API สำรอง
รหัสโปรไฟล์การยืนยันตัวตน Codex แบบเดิมและลำดับการยืนยันตัวตน Codex แบบเดิมที่มีอยู่
เป็นสถานะแบบเดิมสำหรับ doctor เท่านั้น อย่าเขียนการอ้างอิง Codex GPT แบบเดิมรายการใหม่

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

สำหรับเส้นทางที่มีผลและเข้ากันได้กับ Codex โปรไฟล์ทั้งสองข้างต้นยังคงเป็นตัวเลือก
สำหรับการทำงาน Codex เดียวกัน ลำดับโปรไฟล์ใช้เลือกข้อมูลประจำตัว ไม่ใช่รันไทม์
การเปลี่ยนลำดับการยืนยันตัวตนไม่ได้ทำให้เส้นทางแบบกำหนดเอง, Completions, HTTP หรือ
เส้นทางที่มีการเขียนทับคำขอเข้ากันได้กับ Codex

### Compaction

อย่าตั้งค่า `compaction.model` หรือ `compaction.provider` บนเอเจนต์
ที่ทำงานผ่าน Codex เนื่องจาก Codex ทำ Compaction ผ่านสถานะเธรด app-server แบบเนทีฟ
OpenClaw จึงละเว้นการเขียนทับตัวสรุปภายในเครื่องเหล่านั้นระหว่างรันไทม์ และ
`openclaw doctor --fix` จะนำออกเมื่อเอเจนต์ใช้ Codex

ยังคงรองรับ Lossless ในฐานะกลไกบริบทสำหรับการประกอบ การนำเข้า และ
การบำรุงรักษารอบการทำงาน Codex โดยกำหนดค่าผ่าน
`plugins.slots.contextEngine: "lossless-claw"` และ
`plugins.entries.lossless-claw.config.summaryModel` ไม่ใช่ผ่าน
`agents.defaults.compaction.provider` โดย `openclaw doctor --fix` จะย้าย
รูปแบบเก่า `compaction.provider: "lossless-claw"` ไปยังช่องกลไกบริบท
Lossless เมื่อ Codex เป็นรันไทม์ที่ทำงานอยู่ แต่ Codex แบบเนทีฟยังคง
เป็นผู้จัดการ Compaction ชุดควบคุม app-server แบบเนทีฟรองรับกลไกบริบท
ที่ต้องประกอบข้อมูลก่อนพรอมต์ ส่วนแบ็กเอนด์ CLI ทั่วไป รวมถึง `codex-cli`
ไม่มีความสามารถของโฮสต์ดังกล่าว

สำหรับเอเจนต์ที่ทำงานผ่าน Codex คำสั่ง `/compact` จะเริ่ม Compaction ของ
Codex app-server แบบเนทีฟบนเธรดที่ผูกไว้ OpenClaw จะไม่รอให้ดำเนินการเสร็จ
ไม่กำหนดเวลาหมดอายุของ OpenClaw ไม่รีสตาร์ต app-server ที่ใช้ร่วมกัน และไม่สลับไปใช้
กลไกบริบทหรือตัวสรุป OpenAI สาธารณะเป็นทางเลือกสำรอง หากการผูกเธรด Codex แบบเนทีฟ
หายไปหรือล้าสมัย คำสั่งจะล้มเหลวแบบปิดอย่างปลอดภัย แทนที่จะสลับ
แบ็กเอนด์ Compaction โดยไม่แจ้งให้ทราบ

ส่วนที่เหลือของหน้านี้ครอบคลุมรูปแบบการติดตั้งใช้งาน การกำหนดเส้นทางแบบปิดเมื่อเกิดข้อผิดพลาด นโยบายการอนุมัติ
ของ guardian ปลั๊กอิน Codex แบบเนทีฟ และ Computer Use สำหรับรายการตัวเลือก
ค่าเริ่มต้น enum การค้นหา การแยกสภาพแวดล้อม เวลาหมดอายุ และ
ฟิลด์ทรานสปอร์ต app-server ทั้งหมด โปรดดู
[ข้อมูลอ้างอิงชุดควบคุม Codex](/th/plugins/codex-harness-reference)

## ตรวจสอบรันไทม์ Codex

ใช้ `/status` ในแชตที่คาดว่าจะใช้ Codex เทิร์นของเอเจนต์ OpenAI
ที่ทำงานผ่าน Codex จะแสดง:

```text
รันไทม์: OpenAI Codex
```

จากนั้นตรวจสอบสถานะ app-server ของ Codex:

```text
/codex status
/codex models
```

`/codex status` รายงานการเชื่อมต่อ app-server, บัญชี, ขีดจำกัดอัตราการใช้งาน, เซิร์ฟเวอร์ MCP
และ Skills ส่วน `/codex models` แสดงแค็ตตาล็อก app-server ของ Codex ที่ใช้งานอยู่
สำหรับชุดควบคุมและบัญชี หาก `/status` ให้ผลลัพธ์ที่ไม่คาดคิด โปรดดู
[การแก้ไขปัญหา](#troubleshooting)

## การกำหนดเส้นทางและการเลือกโมเดล

แยกการอ้างอิงผู้ให้บริการออกจากนโยบายรันไทม์:

- ใช้ `openai/gpt-*` สำหรับการเลือกโมเดล OpenAI มาตรฐาน คำนำหน้าเพียงอย่างเดียว
  จะไม่เลือก Codex
- เมื่อไม่ได้ตั้งค่ารันไทม์หรือใช้ `auto` เฉพาะเส้นทาง HTTPS Platform Responses
  หรือ ChatGPT Responses อย่างเป็นทางการที่ตรงกันทุกประการและไม่มีการเขียนการแทนที่คำขอไว้เท่านั้น
  ที่อาจเลือก Codex โดยปริยาย
- อย่าใช้การอ้างอิง Codex GPT แบบเดิมในการกำหนดค่า ให้เรียกใช้ `openclaw doctor --fix` เพื่อ
  ซ่อมแซมการอ้างอิงแบบเดิมและพินเส้นทางเซสชันที่ล้าสมัย
- `agentRuntime.id: "codex"` กำหนดให้ Codex เป็นข้อบังคับแบบปิดเมื่อล้มเหลวสำหรับ
  เส้นทางที่เข้ากันได้ แต่ไม่ได้ทำให้เส้นทางที่มีผลและไม่เข้ากันกลายเป็นเส้นทางที่เข้ากันได้
- `agentRuntime.id: "openclaw"` เลือกให้ผู้ให้บริการหรือโมเดลใช้รันไทม์
  OpenClaw แบบฝังตัวเมื่อมีเจตนาเช่นนั้น
- `/codex ...` ควบคุมการสนทนา app-server แบบเนทีฟของ Codex จากแชต
- ACP/acpx เป็นเส้นทางชุดควบคุมภายนอกที่แยกต่างหาก ใช้เฉพาะเมื่อผู้ใช้

## รูปแบบการติดตั้งใช้งาน

### การติดตั้งใช้งาน Codex ขั้นพื้นฐาน

ใช้การกำหนดค่าเริ่มต้นอย่างรวดเร็วสำหรับโมเดล OpenAI ซึ่งมีเส้นทาง HTTPS อย่างเป็นทางการที่มีผล
และมีสิทธิ์เลือก Codex โดยปริยาย:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

### การติดตั้งใช้งานผู้ให้บริการแบบผสม

คง Claude เป็นเอเจนต์เริ่มต้นและเพิ่มเอเจนต์ Codex ที่มีชื่อ:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-6",
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.6-sol",
      },
    ],
  },
}
```

เอเจนต์ `main` ใช้เส้นทางผู้ให้บริการตามปกติ เอเจนต์ `codex` ใช้
app-server ของ Codex เมื่อเส้นทาง OpenAI ที่มีผลยังคงเข้ากันได้ ให้เพิ่ม
`agentRuntime.id: "codex"` ที่กำหนดขอบเขตระดับโมเดลอย่างชัดเจนเมื่อควรกำหนดเป็นข้อบังคับ
แบบปิดเมื่อล้มเหลว

### การติดตั้งใช้งาน Codex แบบปิดเมื่อล้มเหลว

เส้นทาง OpenAI แบบ HTTPS อย่างเป็นทางการที่ตรงกันทุกประการและมีสิทธิ์สามารถแก้ค่าเป็น Codex ได้
เมื่อ Plugin ที่รวมมาให้พร้อมใช้งาน เพิ่มนโยบายรันไทม์อย่างชัดเจนสำหรับกฎ
แบบปิดเมื่อล้มเหลวที่กำหนดไว้:

```json5
{
  models: {
    providers: {
      openai: {
        agentRuntime: {
          id: "codex",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

เมื่อบังคับใช้ Codex OpenClaw จะล้มเหลวตั้งแต่เนิ่นๆ หากไม่ได้ประกาศว่าเส้นทางที่มีผล
เข้ากันได้กับ Codex, Plugin ถูกปิดใช้งาน, app-server เก่าเกินไป หรือ
ไม่สามารถเริ่ม app-server ได้

## นโยบาย app-server

ตามค่าเริ่มต้น Plugin จะเริ่มไบนารี Codex ที่ OpenClaw จัดการไว้ในเครื่องโดยใช้
การรับส่งผ่าน stdio ตั้งค่า `appServer.command` เฉพาะเมื่อตั้งใจเรียกใช้
ไฟล์ปฏิบัติการอื่น ใช้การรับส่งผ่าน WebSocket เฉพาะเมื่อมี app-server
ทำงานอยู่ที่อื่นแล้ว:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
          },
        },
      },
    },
  },
}
```

เซสชัน app-server แบบ stdio ในเครื่องใช้ท่าทีของผู้ดำเนินการภายในเครื่องที่เชื่อถือได้
เป็นค่าเริ่มต้น: `approvalPolicy: "never"`, `approvalsReviewer: "user"` และ
`sandbox: "danger-full-access"` หากข้อกำหนด Codex ภายในเครื่องไม่อนุญาต
ท่าที YOLO โดยปริยายดังกล่าว OpenClaw จะเลือกสิทธิ์ของผู้พิทักษ์ที่อนุญาต
แทน เมื่อแซนด์บ็อกซ์ OpenClaw ทำงานอยู่สำหรับเซสชัน OpenClaw
จะปิดใช้งาน Code Mode แบบเนทีฟของ Codex, เซิร์ฟเวอร์ MCP ของผู้ใช้ และการทำงาน
ของ Plugin ที่มีแอปสนับสนุนสำหรับเทิร์นนั้น แทนการพึ่งพาแซนด์บ็อกซ์ฝั่งโฮสต์ของ Codex
การเข้าถึงเชลล์จะดำเนินการผ่านเครื่องมือแบบไดนามิกที่ใช้แซนด์บ็อกซ์ของ OpenClaw
เช่น `sandbox_exec` และ `sandbox_process` เมื่อเครื่องมือ exec/process ตามปกติ
พร้อมใช้งาน

ใช้โหมด exec มาตรฐานของ OpenClaw สำหรับการตรวจสอบอัตโนมัติแบบเนทีฟของ Codex ก่อน
การหลบออกจากแซนด์บ็อกซ์หรือการให้สิทธิ์เพิ่มเติม:

```json5
{
  tools: {
    exec: {
      mode: "auto",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

สำหรับเซสชัน app-server ของ Codex `tools.exec.mode: "auto"` จะจับคู่กับการอนุมัติ
ที่ผ่านการตรวจสอบโดย Codex Guardian: โดยปกติคือ `approvalPolicy: "on-request"`,
`approvalsReviewer: "auto_review"` และ `sandbox: "workspace-write"` เมื่อ
ข้อกำหนดภายในเครื่องอนุญาตค่าเหล่านั้น ใน `tools.exec.mode: "auto"`
OpenClaw จะไม่คงการแทนที่ Codex แบบเดิมที่ไม่ปลอดภัยอย่าง `approvalPolicy: "never"` หรือ
`sandbox: "danger-full-access"` ให้ใช้ `tools.exec.mode: "full"` สำหรับ
ท่าที Codex ที่ตั้งใจไม่ใช้การอนุมัติ ค่าที่ตั้งไว้ล่วงหน้าแบบเดิม
`plugins.entries.codex.config.appServer.mode: "guardian"` ยังคงใช้งานได้
แต่ `tools.exec.mode: "auto"` คือพื้นผิวมาตรฐานของ OpenClaw

สำหรับการเปรียบเทียบระดับโหมดกับการอนุมัติ exec ของโฮสต์และสิทธิ์ ACPX
โปรดดู [โหมดสิทธิ์](/th/tools/permission-modes) สำหรับฟิลด์ app-server ทุกฟิลด์
ลำดับการยืนยันตัวตน การแยกสภาพแวดล้อม และลักษณะการทำงานเมื่อหมดเวลา
โปรดดู [ข้อมูลอ้างอิงชุดควบคุม Codex](/th/plugins/codex-harness-reference)

## คำสั่งและการวินิจฉัย

Plugin `codex` จะลงทะเบียน `/codex` เป็นคำสั่งแบบสแลชในทุกช่องทางที่
รองรับคำสั่งข้อความของ OpenClaw

การดำเนินการและการควบคุมแบบเนทีฟต้องใช้เจ้าของหรือไคลเอนต์ Gateway
`operator.admin`: การผูกหรือดำเนินเธรดต่อ, การส่งหรือหยุดเทิร์น,
การเปลี่ยนโมเดล โหมดเร็ว หรือสถานะสิทธิ์, การกระชับหรือการตรวจสอบ และ
การยกเลิกการผูก ผู้ส่งอื่นที่ได้รับอนุญาตจะยังคงใช้คำสั่งแบบอ่านอย่างเดียวสำหรับสถานะ
ความช่วยเหลือ บัญชี โมเดล เธรด เซิร์ฟเวอร์ MCP, Skills และการตรวจสอบการผูก

รูปแบบทั่วไป:

- `/codex status` ตรวจสอบการเชื่อมต่อ app-server, โมเดล, บัญชี, ขีดจำกัด
  อัตราการใช้งาน, เซิร์ฟเวอร์ MCP และ Skills
- `/codex models` แสดงรายการโมเดล app-server ของ Codex ที่ใช้งานอยู่
- `/codex threads [filter]` แสดงรายการเธรด app-server ของ Codex ล่าสุด
- `/codex resume <thread-id>` แนบเซสชัน OpenClaw ปัจจุบันเข้ากับ
  เธรด Codex ที่มีอยู่
- `/codex bind [thread-id] [--cwd <path>] [--model <model>] [--provider <provider>]`
  แนบแชตปัจจุบัน
- `/codex detach` (หรือ `/codex unbind`) ยกเลิกการผูกปัจจุบัน
- `/codex binding` อธิบายการผูกปัจจุบัน
- `/codex stop` หยุดเทิร์นที่ทำงานอยู่ ส่วน `/codex steer <text>` ใช้กำกับเทิร์นนั้น
- `/codex model <model>`, `/codex fast [on|off|status]` และ
  `/codex permissions [default|yolo|status]` เปลี่ยนสถานะรายบทสนทนา
- `/codex compact` ขอให้ app-server ของ Codex กระชับเธรดที่แนบอยู่
- `/codex review` เริ่มการตรวจสอบแบบเนทีฟของ Codex สำหรับเธรดที่แนบอยู่
- `/codex diagnostics [note]` ขอการยืนยันก่อนส่งข้อเสนอแนะเกี่ยวกับ Codex สำหรับ
  เธรดที่แนบอยู่
- `/codex account` แสดงสถานะบัญชีและขีดจำกัดอัตราการใช้งาน
- `/codex mcp` แสดงรายการสถานะเซิร์ฟเวอร์ MCP ของ app-server Codex
- `/codex skills` แสดงรายการ Skills ของ app-server Codex
- `/codex plugins list`, `/codex plugins enable <name>` และ
  `/codex plugins disable <name>` จัดการ Plugin แบบเนทีฟของ Codex ที่กำหนดค่าไว้
- `/codex computer-use [status|install]` จัดการ Codex Computer Use
- `/codex help` แสดงโครงสร้างคำสั่งทั้งหมด

สำหรับรายงานปัญหาด้านการสนับสนุนส่วนใหญ่ ให้เริ่มด้วย `/diagnostics [note]` ใน
บทสนทนาที่เกิดข้อบกพร่อง คำสั่งนี้จะสร้างรายงานการวินิจฉัย Gateway หนึ่งรายการ
และสำหรับเซสชัน Codex harness จะขอการอนุมัติเพื่อส่ง
ชุดข้อมูลคำติชม Codex ที่เกี่ยวข้อง โปรดดู
[การส่งออกข้อมูลวินิจฉัย](/th/gateway/diagnostics) สำหรับรูปแบบความเป็นส่วนตัวและลักษณะการทำงาน
ของแชตกลุ่ม ใช้ `/codex diagnostics [note]` เฉพาะเมื่อคุณต้องการ
อัปโหลดคำติชม Codex สำหรับเธรดที่แนบอยู่ในปัจจุบันโดยเฉพาะ โดยไม่รวม
ชุดข้อมูลวินิจฉัย Gateway ฉบับเต็ม

### ตรวจสอบเธรด Codex ในเครื่อง

วิธีที่เร็วที่สุดในการตรวจสอบการทำงานของ Codex ที่ผิดพลาดมักเป็นการเปิด
เธรด Codex แบบเนทีฟโดยตรง:

```bash
codex resume <thread-id>
```

รับรหัสเธรดจากการตอบกลับ `/diagnostics` ที่เสร็จสมบูรณ์, `/codex binding`,
หรือ `/codex threads [filter]`

สำหรับกลไกการอัปโหลดและขอบเขตการวินิจฉัยระดับรันไทม์ โปรดดู
[รันไทม์ Codex harness](/th/plugins/codex-harness-runtime#codex-feedback-upload)

### ลำดับการยืนยันตัวตน

ในโฮมเริ่มต้นของแต่ละเอเจนต์ ระบบจะเลือกการยืนยันตัวตนตามลำดับนี้:

1. โปรไฟล์การยืนยันตัวตน OpenAI ที่เรียงลำดับไว้สำหรับเอเจนต์ โดยควรอยู่ภายใต้
   `auth.order.openai` เรียกใช้ `openclaw doctor --fix` เพื่อย้ายรหัสโปรไฟล์
   การยืนยันตัวตน Codex แบบเก่าและลำดับการยืนยันตัวตน Codex แบบเก่า
2. บัญชีที่มีอยู่ของ app-server ในโฮม Codex ของเอเจนต์นั้น
3. เฉพาะการเปิด app-server ผ่าน stdio ในเครื่องเท่านั้น ให้ใช้ `CODEX_API_KEY` แล้วจึง
   `OPENAI_API_KEY` เมื่อไม่มีบัญชี app-server และยังคงจำเป็นต้องใช้การยืนยันตัวตน
   OpenAI

เมื่อ OpenClaw พบโปรไฟล์การยืนยันตัวตน Codex แบบการสมัครสมาชิก ChatGPT ระบบจะ
ลบ `CODEX_API_KEY` และ `OPENAI_API_KEY` ออกจากโพรเซสลูก Codex
ที่สร้างขึ้น วิธีนี้ทำให้คีย์ API ระดับ Gateway ยังคงใช้ได้สำหรับ embeddings หรือ
โมเดล OpenAI โดยตรง โดยไม่ทำให้เทิร์นของ Codex app-server แบบเนทีฟ
ถูกเรียกเก็บเงินผ่าน API โดยไม่ได้ตั้งใจ โปรไฟล์คีย์ API ของ Codex ที่ระบุอย่างชัดเจนและ
การสำรองไปใช้คีย์จากสภาพแวดล้อมของ stdio ในเครื่องจะใช้การเข้าสู่ระบบ app-server แทน
สภาพแวดล้อมของโพรเซสลูกที่สืบทอดมา การเชื่อมต่อ app-server ผ่าน WebSocket
จะไม่ได้รับคีย์ API สำรองจากสภาพแวดล้อมของ Gateway ให้ใช้โปรไฟล์การยืนยันตัวตน
ที่ระบุอย่างชัดเจนหรือบัญชีของ app-server ระยะไกลเอง

หากโปรไฟล์การสมัครสมาชิกถึงขีดจำกัดการใช้งาน Codex OpenClaw จะบันทึก
เวลารีเซ็ตเมื่อ Codex รายงาน และลองใช้โปรไฟล์การยืนยันตัวตนถัดไปตามลำดับ
สำหรับการทำงาน Codex ครั้งเดิม เมื่อพ้นเวลารีเซ็ต โปรไฟล์การสมัครสมาชิก
จะมีสิทธิ์ใช้งานอีกครั้งโดยไม่เปลี่ยนโมเดล `openai/gpt-*`
หรือรันไทม์ Codex ที่เลือกไว้

เมื่อกำหนดค่า Plugin Codex แบบเนทีฟไว้ OpenClaw จะติดตั้งหรือรีเฟรช
Plugin เหล่านั้นผ่าน app-server ที่เชื่อมต่ออยู่ก่อนเปิดให้เธรด Codex
เข้าถึงแอปที่ Plugin เป็นเจ้าของ `app/list` ยังคงเป็นแหล่งข้อมูลจริงสำหรับรหัส
แอป การเข้าถึง และข้อมูลเมตา แต่ OpenClaw เป็นผู้ควบคุมการตัดสินใจ
เปิดใช้งานในแต่ละเธรด: หากนโยบายอนุญาตแอปที่เข้าถึงได้ในรายการ OpenClaw
จะส่ง `thread/start.config.apps[appId].enabled = true` แม้ว่า `app/list`
จะรายงานว่าแอปนั้นปิดใช้งานอยู่ในขณะนี้ เส้นทางนี้จะไม่สร้างการติดตั้งแอป
สำหรับรหัสที่ไม่รู้จัก OpenClaw จะเปิดใช้งานเฉพาะ Plugin จากมาร์เก็ตเพลส
ด้วย `plugin/install` แล้วจึงรีเฟรชรายการทรัพยากร

### การแยกสภาพแวดล้อม

สำหรับการเปิด app-server ผ่าน stdio ในเครื่อง OpenClaw จะตั้งค่า `CODEX_HOME` เป็น
ไดเรกทอรีเฉพาะเอเจนต์ เพื่อให้การกำหนดค่า Codex, ไฟล์การยืนยันตัวตน/บัญชี, แคช/ข้อมูล
ของ Plugin และสถานะเธรดแบบเนทีฟไม่อ่านหรือเขียนโฮม Codex ส่วนตัว
`~/.codex` ของผู้ดำเนินการโดยค่าเริ่มต้น OpenClaw จะคงค่า `HOME`
ของโพรเซสตามปกติไว้ โพรเซสย่อยที่ทำงานผ่าน Codex จึงยังค้นหาการกำหนดค่าและโทเค็น
ในโฮมของผู้ใช้ได้ และ Codex อาจค้นพบรายการ `$HOME/.agents/skills` และ
`$HOME/.agents/plugins/marketplace.json` ที่ใช้ร่วมกัน เมื่อใช้
`appServer.homeScope: "user"` OpenClaw จะใช้โฮม Codex แบบเนทีฟของผู้ใช้
และบัญชีที่มีอยู่แทน โดยไม่แทรกโปรไฟล์การยืนยันตัวตนของ OpenClaw

หากการติดตั้งใช้งานต้องแยกสภาพแวดล้อมเพิ่มเติม ให้เพิ่มตัวแปรเหล่านั้น
ใน `appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` มีผลเฉพาะกับโพรเซสลูก Codex app-server
ที่สร้างขึ้น OpenClaw จะลบ `CODEX_HOME` และ `HOME` ออกจากรายการนี้ระหว่าง
การปรับรูปแบบการเปิดในเครื่อง: `CODEX_HOME` จะยังชี้ไปยังขอบเขต
เอเจนต์หรือผู้ใช้ที่เลือก และ `HOME` จะยังคงสืบทอดค่าเพื่อให้โพรเซสย่อยใช้
สถานะในโฮมของผู้ใช้ตามปกติได้

### เครื่องมือแบบไดนามิกและการค้นหาเว็บ

เครื่องมือแบบไดนามิกของ Codex ใช้การโหลดแบบ `searchable` โดยค่าเริ่มต้น ตามปกติ OpenClaw
จะไม่เปิดเผยเครื่องมือแบบไดนามิกที่ซ้ำกับการดำเนินการพื้นที่ทำงานแบบเนทีฟของ Codex:
`read`, `write`, `edit`, `apply_patch`, `exec`, `process`, `update_plan`,
`tool_call`, `tool_describe`, `tool_search` และ `tool_search_code` เครื่องมือ
การผสานรวม OpenClaw ส่วนใหญ่ที่เหลือ เช่น การส่งข้อความ สื่อ Cron
เบราว์เซอร์ Node, Gateway และ `heartbeat_respond` สามารถใช้ผ่าน
การค้นหาเครื่องมือของ Codex ภายใต้เนมสเปซ `openclaw` ซึ่งช่วยให้บริบทเริ่มต้น
ของโมเดลมีขนาดเล็กลง การสำรองไปใช้เชลล์สำหรับเทิร์นที่ถูกจำกัดเป็นข้อยกเว้นสำหรับ
`exec` และ `process` เมื่อรายการอนุญาตแบบจำกัดปิดใช้งาน Code Mode แบบเนทีฟ
รายการอนุญาตของรันไทม์และ `codexDynamicToolsExclude` ยังคงมีผล

เครื่องมือที่ทำเครื่องหมาย `catalogMode: "direct-only"` รวมถึงเครื่องมือ `computer`
ของ OpenClaw จะใช้เนมสเปซ `openclaw_direct` แทน Codex ถือว่าเนมสเปซนั้น
เป็น `DirectModelOnly` ดังนั้นเครื่องมือเหล่านั้นจึงยังปรากฏต่อโมเดลโดยตรงในเธรด
แบบปกติและแบบ Code Mode เท่านั้น แทนที่จะผ่านการเรียก Code Mode `tools.*`
ที่ซ้อนกัน

การค้นหาเว็บใช้เครื่องมือ `web_search` แบบโฮสต์ของ Codex โดยค่าเริ่มต้น เมื่อเปิดใช้การค้นหา
และไม่ได้เลือกผู้ให้บริการที่มีการจัดการ การค้นหาแบบโฮสต์เนทีฟและ
เครื่องมือแบบไดนามิก `web_search` ที่ OpenClaw จัดการจะทำงานร่วมกันไม่ได้ เพื่อไม่ให้
การค้นหาที่มีการจัดการหลีกเลี่ยงข้อจำกัดโดเมนแบบเนทีฟ OpenClaw จะใช้
เครื่องมือที่มีการจัดการเมื่อการค้นหาแบบโฮสต์ใช้งานไม่ได้ ถูกปิดใช้งานอย่างชัดเจน หรือ
ถูกแทนที่ด้วยผู้ให้บริการที่มีการจัดการซึ่งเลือกไว้ OpenClaw จะปิดใช้งานส่วนขยาย
`web.run` แบบแยกเดี่ยวของ Codex ต่อไป เนื่องจากทราฟฟิก app-server ในระบบจริงปฏิเสธ
เนมสเปซ `web` ที่ผู้ใช้กำหนด `tools.web.search.enabled: false`
จะปิดใช้งานทั้งสองเส้นทาง เช่นเดียวกับการทำงานแบบ LLM เท่านั้นที่ปิดเครื่องมือ Codex ถือว่า
`"cached"` เป็นค่ากำหนด และแปลงเป็นการเข้าถึงภายนอกแบบสดสำหรับ
เทิร์น app-server ที่ไม่ถูกจำกัด การสำรองอัตโนมัติไปยังเครื่องมือที่มีการจัดการจะปฏิเสธการทำงาน
เมื่อมีการตั้งค่า `allowedDomains` แบบเนทีฟ เพื่อไม่ให้หลีกเลี่ยงรายการอนุญาตได้
การเปลี่ยนนโยบายการค้นหาที่มีผลแบบถาวรจะหมุนเวียนเธรด Codex ที่ผูกไว้
ก่อนเทิร์นถัดไป ส่วนข้อจำกัดชั่วคราวรายเทิร์นจะใช้เธรดที่จำกัดชั่วคราว
และคงการผูกเดิมไว้สำหรับการทำงานต่อในภายหลัง

การตอบกลับจากแหล่งข้อมูลแบบ `sessions_yield` และแบบใช้เฉพาะเครื่องมือข้อความจะยังคงส่งตรง
เนื่องจากเป็นสัญญาควบคุมเทิร์น `sessions_spawn` จะยังค้นหาได้ เพื่อให้
`spawn_agent` แบบเนทีฟของ Codex ยังคงเป็นพื้นผิวเอเจนต์ย่อยหลักของ Codex
ขณะที่การมอบหมายให้ OpenClaw หรือ ACP โดยชัดแจ้งยังคงใช้ได้ผ่าน
เนมสเปซเครื่องมือแบบไดนามิก `openclaw` คำสั่งการทำงานร่วมกันของ Heartbeat
จะแจ้งให้ Codex ค้นหา `heartbeat_respond` ก่อนสิ้นสุดเทิร์น Heartbeat
เมื่อยังไม่ได้โหลดเครื่องมือ

ตั้งค่า `codexDynamicToolsLoading: "direct"` เฉพาะเมื่อเชื่อมต่อกับ

| `mode`                                        | `"yolo"` เว้นแต่ข้อกำหนด Codex ภายในเครื่องไม่อนุญาต YOLO | ค่าที่ตั้งไว้ล่วงหน้าสำหรับการดำเนินการแบบ YOLO หรือที่ผ่านการตรวจสอบโดยผู้พิทักษ์ ข้อกำหนด stdio ภายในเครื่องที่ไม่มี `danger-full-access`, การอนุมัติ `never` หรือผู้ตรวจสอบ `user` จะทำให้ค่าเริ่มต้นโดยนัยเป็นผู้พิทักษ์                                                                                                                                                                                                           |
| `approvalPolicy`                              | `"never"` หรือนโยบายการอนุมัติของผู้พิทักษ์ที่อนุญาต       | นโยบายการอนุมัติแบบเนทีฟของ Codex ที่ส่งเมื่อเริ่มเธรด ดำเนินเธรดต่อ หรือเริ่มเทิร์น ค่าเริ่มต้นของผู้พิทักษ์จะเลือก `"on-request"` เมื่อได้รับอนุญาต                                                                                                                                                                                                                                                                            |
| `sandbox`                                     | `"danger-full-access"` หรือแซนด์บ็อกซ์ของผู้พิทักษ์ที่อนุญาต  | โหมดแซนด์บ็อกซ์แบบเนทีฟของ Codex ที่ส่งเมื่อเริ่มหรือดำเนินเธรดต่อ ค่าเริ่มต้นของผู้พิทักษ์จะเลือก `"workspace-write"` เมื่อได้รับอนุญาต มิฉะนั้นจะเลือก `"read-only"` เมื่อแซนด์บ็อกซ์ OpenClaw ทำงานอยู่ เทิร์น `danger-full-access` จะใช้ `workspace-write` ของ Codex โดยให้สิทธิ์เข้าถึงเครือข่ายตามการตั้งค่าการส่งข้อมูลออกของแซนด์บ็อกซ์ OpenClaw                                                                                     |
| `approvalsReviewer`                           | `"user"` หรือผู้ตรวจสอบที่เป็นผู้พิทักษ์ที่อนุญาต               | ใช้ `"auto_review"` เพื่อให้ Codex ตรวจสอบข้อความแจ้งขออนุมัติแบบเนทีฟเมื่อได้รับอนุญาต มิฉะนั้นใช้ `guardian_subagent` หรือ `user` โดย `guardian_subagent` ยังคงเป็นนามแฝงแบบเดิม                                                                                                                                                                                                                              |

| `networkProxy`                                | ปิดใช้งาน                                               | เลือกใช้เครือข่ายตามโปรไฟล์สิทธิ์ของ Codex สำหรับคำสั่ง app-server โดย OpenClaw จะกำหนดการกำหนดค่า `permissions.<profile>.network` ที่เลือกและเลือกใช้ด้วย `default_permissions` แทนการส่ง `sandbox`                                                                                                                                                                             |
| `experimental.sandboxExecServer`              | `false`                                                | การเลือกเข้าร่วมฟีเจอร์ตัวอย่าง ซึ่งลงทะเบียนสภาพแวดล้อม Codex ที่ใช้แซนด์บ็อกซ์ OpenClaw กับ app-server ของ Codex ที่รองรับ เพื่อให้การดำเนินการแบบเนทีฟของ Codex ทำงานภายในแซนด์บ็อกซ์ OpenClaw ที่ใช้งานอยู่ได้                                                                                                                                                                                                            |

`appServer.networkProxy` เป็นการตั้งค่าอย่างชัดเจน เนื่องจากเปลี่ยนสัญญาแซนด์บ็อกซ์ของ Codex
เมื่อเปิดใช้ OpenClaw จะตั้งค่า `features.network_proxy.enabled`
และ `default_permissions` ในการกำหนดค่าเธรด Codex ด้วย เพื่อให้โปรไฟล์
สิทธิ์ที่สร้างขึ้นสามารถเริ่มเครือข่ายที่ Codex จัดการได้ ตามค่าเริ่มต้น OpenClaw
จะสร้างชื่อโปรไฟล์ `openclaw-network-<fingerprint>` ที่ทนต่อการชนกัน
จากเนื้อหาของโปรไฟล์; ใช้ `profileName` เฉพาะเมื่อต้องการชื่อภายในเครื่อง
ที่คงที่

```json5
{
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            sandbox: "workspace-write",
            networkProxy: {
              enabled: true,
              domains: {
                "api.openai.com": "allow",
                "blocked.example.com": "deny",
              },
              unixSockets: {
                "/tmp/proxy.sock": "allow",
                "/tmp/blocked.sock": "none",
              },
              allowUpstreamProxy: true,
              proxyUrl: "http://127.0.0.1:3128",
            },
          },
        },
      },
    },
  },
}
```

หากรันไทม์ app-server ปกติจะเป็น `danger-full-access` การเปิดใช้
`networkProxy` จะใช้การเข้าถึงระบบไฟล์แบบ workspace สำหรับโปรไฟล์
สิทธิ์ที่สร้างขึ้น: การบังคับใช้นโยบายเครือข่ายที่ Codex จัดการเป็นเครือข่าย
ที่อยู่ใน sandbox ดังนั้นโปรไฟล์แบบเข้าถึงได้เต็มรูปแบบจะไม่ป้องกันทราฟฟิกขาออก
รายการโดเมนใช้ `allow` หรือ `deny`; รายการซ็อกเก็ต Unix ใช้ค่า
`allow` หรือ `none` ของ Codex

### การหมดเวลาในการเรียกใช้เครื่องมือแบบไดนามิก

การเรียกใช้เครื่องมือแบบไดนามิกที่ OpenClaw เป็นเจ้าของถูกจำกัดโดยแยกเป็นอิสระจาก
`appServer.requestTimeoutMs`: คำขอ `item/tool/call` ของ Codex ใช้ตัวเฝ้าระวังของ OpenClaw
เป็นเวลา 90 วินาทีโดยค่าเริ่มต้น อาร์กิวเมนต์ `timeoutMs` ต่อการเรียก
ที่เป็นค่าบวกจะเพิ่มหรือลดงบเวลาของเครื่องมือนั้นโดยเฉพาะ โดยจำกัดสูงสุดที่ 600000 ms
เครื่องมือ `image_generate` ใช้ `agents.defaults.imageGenerationModel.timeoutMs`
เมื่อการเรียกใช้เครื่องมือไม่ได้กำหนดเวลาหมดเวลาของตนเอง มิฉะนั้นจะใช้ค่าเริ่มต้น
สำหรับการสร้างภาพที่ 120 วินาที เครื่องมือ `image` สำหรับการทำความเข้าใจสื่อ
ใช้ `tools.media.image.timeoutSeconds` หรือค่าเริ่มต้นสำหรับสื่อที่ 60 วินาที; สำหรับ
การทำความเข้าใจภาพ เวลาหมดเวลานั้นใช้กับคำขอเองและไม่ถูกลดลง
จากงานเตรียมการก่อนหน้า เมื่อหมดเวลา OpenClaw จะยกเลิกสัญญาณของเครื่องมือ
ในกรณีที่รองรับ และส่งการตอบกลับของเครื่องมือแบบไดนามิกที่ล้มเหลวกลับไปยัง Codex
เพื่อให้เทิร์นดำเนินต่อได้ แทนที่จะปล่อยให้เซสชันค้างอยู่ใน `processing`
ตัวเฝ้าระวังนี้เป็นงบ `item/tool/call` แบบไดนามิกชั้นนอก; เวลาหมดเวลาของคำขอ
ที่เฉพาะเจาะจงกับผู้ให้บริการจะทำงานภายในการเรียกนั้นและคงความหมายของเวลาหมดเวลา
ของตนเองไว้

หลังจาก Codex ยอมรับเทิร์น และหลังจาก OpenClaw ตอบกลับคำขอ app-server
ที่มีขอบเขตเฉพาะเทิร์น ชุดควบคุมคาดว่า Codex จะดำเนินความคืบหน้าของเทิร์นปัจจุบัน
และสิ้นสุดเทิร์นเนทีฟด้วย `turn/completed` ในที่สุด หาก
app-server ไม่มีความเคลื่อนไหวเป็นเวลา `appServer.turnCompletionIdleTimeoutMs` OpenClaw
จะพยายามขัดจังหวะเทิร์น Codex แบบดีที่สุดเท่าที่ทำได้ บันทึกการหมดเวลาเพื่อการวินิจฉัย และ
ปล่อยเลนเซสชัน OpenClaw เพื่อไม่ให้ข้อความแชตที่ตามมา
เข้าคิวรออยู่หลังเทิร์นเนทีฟที่ค้าง การแจ้งเตือนส่วนใหญ่ที่ไม่ใช่สถานะสิ้นสุดสำหรับ
เทิร์นเดียวกันจะยกเลิกตัวเฝ้าระวังระยะสั้นนั้น เนื่องจาก Codex ได้พิสูจน์แล้วว่าเทิร์น
ยังคงทำงานอยู่

การส่งต่อเครื่องมือใช้งบเวลาว่างหลังเครื่องมือที่ยาวกว่า: หลังจาก OpenClaw ส่งคืน
การตอบกลับ `item/tool/call` หลังจากรายการเครื่องมือเนทีฟ เช่น
`commandExecution` เสร็จสิ้น หลังจากการทำงาน `custom_tool_call_output` แบบดิบ
เสร็จสมบูรณ์ และหลังจากความคืบหน้าของผู้ช่วยแบบดิบหลังใช้เครื่องมือ การทำงาน
ด้านการให้เหตุผลแบบดิบเสร็จสมบูรณ์ หรือความคืบหน้าด้านการให้เหตุผล ตัวป้องกันใช้
`appServer.postToolRawAssistantCompletionIdleTimeoutMs` เมื่อกำหนดค่าไว้ และ
ใช้ค่าเริ่มต้นเป็นห้านาทีในกรณีอื่น; งบเดียวกันนั้นยังขยาย
ตัวเฝ้าระวังความคืบหน้าสำหรับช่วงการสังเคราะห์ที่ไม่มีการส่งข้อมูล ก่อนที่ Codex จะส่ง
เหตุการณ์ถัดไปของเทิร์นปัจจุบัน การแจ้งเตือน app-server ส่วนกลาง เช่น
การอัปเดตขีดจำกัดอัตรา จะไม่รีเซ็ตความคืบหน้าเวลาว่างของเทิร์น การทำงานด้านการให้เหตุผล
เสร็จสมบูรณ์ การทำงาน `agentMessage` แบบ commentary เสร็จสมบูรณ์ และความคืบหน้า
ด้านการให้เหตุผลหรือผู้ช่วยแบบดิบก่อนใช้เครื่องมืออาจตามด้วยคำตอบสุดท้ายอัตโนมัติ
ดังนั้นจึงใช้ตัวป้องกันการตอบกลับหลังความคืบหน้า แทนที่จะปล่อยเลนเซสชัน
ทันที

เฉพาะรายการ `agentMessage` ที่เสร็จสมบูรณ์แบบสุดท้าย/ไม่ใช่ commentary และการทำงาน
ของผู้ช่วยแบบดิบก่อนใช้เครื่องมือเสร็จสมบูรณ์เท่านั้นที่เปิดใช้งานการปล่อยเอาต์พุตผู้ช่วย:
หากหลังจากนั้น Codex ไม่มีความเคลื่อนไหวโดยไม่มี `turn/completed` OpenClaw จะพยายาม
ขัดจังหวะเทิร์นเนทีฟและปล่อยเลนเซสชันแบบดีที่สุดเท่าที่ทำได้ หากตัวเฝ้าดูเทิร์นอื่น
ชนะการแข่งขันในการปล่อยนั้น OpenClaw ยังคงยอมรับรายการผู้ช่วยสุดท้ายที่เสร็จสมบูรณ์
เมื่อไม่มีคำขอเนทีฟ รายการ หรือการทำงานของเครื่องมือแบบไดนามิกที่เสร็จสมบูรณ์
หลงเหลืออยู่ และการปล่อยเอาต์พุตผู้ช่วยยังคงเป็นของรายการล่าสุดที่เสร็จสมบูรณ์
โดยไม่มีรายการที่เสร็จสมบูรณ์ภายหลัง วิธีนี้สามารถรักษาคำตอบสุดท้ายหลังจาก
งานเครื่องมือเสร็จสมบูรณ์โดยไม่เล่นเทิร์นซ้ำ เดลตาของผู้ช่วยที่ยังไม่สมบูรณ์
คำตอบก่อนหน้าที่ล้าสมัย และการทำงานภายหลังที่ว่างเปล่าไม่เข้าเกณฑ์

ความล้มเหลวของ stdio app-server ที่สามารถเล่นซ้ำได้อย่างปลอดภัย รวมถึงการหมดเวลา
จากการไม่มีความเคลื่อนไหวระหว่างการเสร็จสิ้นเทิร์น โดยไม่มีหลักฐานจากผู้ช่วย เครื่องมือ
รายการที่ทำงานอยู่ หรือผลข้างเคียง จะถูกลองใหม่หนึ่งครั้งด้วยการพยายาม app-server ใหม่
การหมดเวลาที่ไม่ปลอดภัยจะยังคงปลดระวางไคลเอนต์ app-server ที่ค้างและปล่อยเลนเซสชัน
OpenClaw; นอกจากนี้ยังล้างการผูกเธรดเนทีฟที่ล้าสมัย แทนที่จะเล่นซ้ำ
โดยอัตโนมัติ การหมดเวลาของการเฝ้าดูการเสร็จสมบูรณ์จะแสดงข้อความการหมดเวลา
เฉพาะ Codex: กรณีที่เล่นซ้ำได้อย่างปลอดภัยจะระบุว่าการตอบกลับอาจไม่สมบูรณ์ ส่วนกรณี
ที่ไม่ปลอดภัยจะแจ้งให้ผู้ใช้ตรวจสอบสถานะปัจจุบันก่อนลองใหม่ การวินิจฉัยการหมดเวลา
สาธารณะประกอบด้วยฟิลด์เชิงโครงสร้าง เช่น เมธอดการแจ้งเตือน app-server ล่าสุด
รหัส/ประเภท/บทบาทของรายการตอบกลับผู้ช่วยแบบดิบ จำนวนคำขอ/รายการที่ทำงานอยู่
และสถานะตัวเฝ้าดูที่เปิดใช้งาน; เมื่อการแจ้งเตือนล่าสุดเป็นรายการตอบกลับผู้ช่วยแบบดิบ
การวินิจฉัยจะรวมตัวอย่างข้อความผู้ช่วยที่จำกัดขนาดไว้ด้วย แต่จะไม่รวมพรอมต์ดิบ
หรือเนื้อหาเครื่องมือ

### การเขียนทับค่าตัวแปรสภาพแวดล้อมสำหรับการทดสอบในเครื่อง

- `OPENCLAW_CODEX_APP_SERVER_BIN` ข้ามไบนารีที่มีการจัดการเมื่อ
  ไม่ได้ตั้งค่า `appServer.command`
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

`OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1` ถูกนำออกแล้ว ให้ใช้
`plugins.entries.codex.config.appServer.mode: "guardian"` แทน หรือ
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` สำหรับการทดสอบในเครื่องแบบครั้งเดียว แนะนำให้ใช้การกำหนดค่า
สำหรับการติดตั้งใช้งานที่ทำซ้ำได้ เนื่องจากจะเก็บพฤติกรรมของ Plugin
ไว้ในไฟล์ที่ผ่านการตรวจสอบเดียวกับการตั้งค่าชุดควบคุม Codex ส่วนที่เหลือ

## Plugin เนทีฟของ Codex

การรองรับ Plugin เนทีฟของ Codex ใช้ความสามารถด้านแอปและ Plugin
ของ app-server ของ Codex เองในเธรด Codex เดียวกับเทิร์นของชุดควบคุม OpenClaw
OpenClaw ไม่ได้แปลง Plugin ของ Codex เป็นเครื่องมือแบบไดนามิก
`codex_plugin_*` ของ OpenClaw ที่สร้างขึ้นจำลอง

`codexPlugins` มีผลเฉพาะกับเซสชันที่เลือกชุดควบคุม Codex เนทีฟ
ไม่มีผลต่อการรันชุดควบคุมในตัว การรันผู้ให้บริการ OpenAI ตามปกติ การผูกบทสนทนา
ACP หรือชุดควบคุมอื่น

การกำหนดค่าขั้นต่ำที่ย้ายแล้ว:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

การกำหนดค่าแอปของเธรดจะคำนวณเมื่อ OpenClaw สร้างเซสชันชุดควบคุม Codex
หรือแทนที่การผูกเธรด Codex ที่ล้าสมัย; ระบบจะไม่คำนวณใหม่ในทุกเทิร์น
หลังจากเปลี่ยน `codexPlugins` ให้ใช้ `/new`, `/reset` หรือรีสตาร์ต
Gateway เพื่อให้เซสชันชุดควบคุม Codex ในอนาคตเริ่มต้นด้วยชุดแอป
ที่อัปเดตแล้ว

สำหรับคุณสมบัติในการย้าย รายการแอป นโยบายการดำเนินการที่ทำลายข้อมูล
การร้องขอข้อมูล และการวินิจฉัย Plugin เนทีฟ โปรดดู
[Plugin เนทีฟของ Codex](/th/plugins/codex-native-plugins)

การเข้าถึงแอปและ Plugin ฝั่ง OpenAI ถูกควบคุมโดยบัญชี Codex
ที่ลงชื่อเข้าใช้ และสำหรับ workspace แบบ Business และ Enterprise/Edu จะถูกควบคุม
โดยการควบคุมแอปของ workspace โปรดดู
[การใช้ Codex กับแผน ChatGPT ของคุณ](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan)
สำหรับภาพรวมการควบคุมบัญชีและ workspace ของ OpenAI

## การใช้คอมพิวเตอร์

การใช้คอมพิวเตอร์มีคู่มือการตั้งค่าแยกต่างหาก:
[การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use)

ฉบับย่อ: OpenClaw ไม่ได้รวมแอปควบคุมเดสก์ท็อปไว้ในแพ็กเกจหรือดำเนินการ
บนเดสก์ท็อปด้วยตนเอง แต่จะเตรียม app-server ของ Codex ตรวจสอบว่าเซิร์ฟเวอร์ MCP
`computer-use` พร้อมใช้งาน แล้วปล่อยให้ Codex เป็นเจ้าของการเรียกใช้เครื่องมือ MCP
เนทีฟระหว่างเทิร์นในโหมด Codex

## ขอบเขตรันไทม์

ชุดควบคุม Codex เปลี่ยนเฉพาะตัวดำเนินการเอเจนต์แบบฝังระดับล่าง

- รองรับเครื่องมือแบบไดนามิกของ OpenClaw โดย Codex ขอให้ OpenClaw ดำเนินการ
  เครื่องมือเหล่านั้น ดังนั้น OpenClaw จึงยังคงอยู่ในเส้นทางการดำเนินการ
- เชลล์ แพตช์ MCP และเครื่องมือแอปเนทีฟของ Codex เป็นของ Codex
  OpenClaw สามารถสังเกตหรือบล็อกเหตุการณ์เนทีฟบางรายการผ่านรีเลย์
  ที่รองรับ แต่จะไม่เขียนอาร์กิวเมนต์ของเครื่องมือเนทีฟใหม่
- Codex เป็นเจ้าของ Compaction เนทีฟ OpenClaw เก็บสำเนาทรานสคริปต์สำหรับ
  ประวัติช่องทาง การค้นหา `/new`, `/reset` และการสลับโมเดลหรือชุดควบคุม
  ในอนาคต แต่จะไม่แทนที่ Compaction ของ Codex ด้วยตัวสรุปของ OpenClaw
  หรือกลไกบริบท
- การสร้างสื่อ การทำความเข้าใจสื่อ TTS การอนุมัติ และเอาต์พุตของเครื่องมือ
  ส่งข้อความยังคงผ่านการตั้งค่าผู้ให้บริการ/โมเดล OpenClaw ที่ตรงกัน
- `tool_result_persist` ใช้กับผลลัพธ์เครื่องมือทรานสคริปต์ที่ OpenClaw เป็นเจ้าของ
  ไม่ใช่ระเบียนผลลัพธ์เครื่องมือเนทีฟของ Codex

สำหรับชั้นฮุก พื้นผิว V1 ที่รองรับ การจัดการสิทธิ์เนทีฟ การกำหนดทิศทางคิว
กลไกการอัปโหลดข้อเสนอแนะของ Codex และรายละเอียด Compaction โปรดดู
[รันไทม์ชุดควบคุม Codex](/th/plugins/codex-harness-runtime)

`plugins.entries.codex.enabled` และตรวจสอบว่า `plugins.allow` ยกเว้น
`codex` หรือไม่

**OpenClaw ใช้ชุดควบคุมในตัวแทน Codex:** ยืนยันว่าเส้นทางที่มีผล
เป็นเส้นทาง HTTPS Platform Responses หรือ ChatGPT Responses อย่างเป็นทางการที่ตรงกันทุกประการ
ไม่มีการเขียนทับคำขอที่กำหนดขึ้นเอง และ Plugin Codex ได้รับการติดตั้งและ
เปิดใช้งานแล้ว คำนำหน้า `openai/gpt-*` เพียงอย่างเดียวไม่เพียงพอ สำหรับหลักฐานที่เข้มงวด
ขณะทดสอบ ให้ตั้งค่า `agentRuntime.id: "codex"` ของผู้ให้บริการหรือโมเดล; การบังคับใช้ Codex
จะล้มเหลวแทนการย้อนกลับเมื่อเส้นทางหรือชุดควบคุมเข้ากันไม่ได้

**รันไทม์ OpenAI Codex ย้อนกลับไปใช้เส้นทาง API key:** รวบรวมข้อความบางส่วน
ของ Gateway ที่ปกปิดข้อมูลลับแล้ว ซึ่งแสดงโมเดล รันไทม์ ผู้ให้บริการที่เลือก และ
ความล้มเหลว ขอให้ผู้ร่วมงานที่ได้รับผลกระทบรันคำสั่งแบบอ่านอย่างเดียวนี้บนโฮสต์
OpenClaw ของตน:

```bash
(
  pattern='openai/gpt-5\.[45]|openai[-]codex|agentRuntime(\.id)?|harnessRuntime|Runtime: OpenAI Codex|legacy OpenAI Codex prefix|resolveSelectedOpenAIRuntimeProvider|candidateProvider[": ]+openai|status[": ]+401|Incorrect API key|No API key|api-key path|API-key path|OAuth'

  if ls /tmp/openclaw/openclaw-*.log >/dev/null 2>&1; then
    grep -E -i -n "$pattern" /tmp/openclaw/openclaw-*.log 2>/dev/null || true
  else
    journalctl --user -u openclaw-gateway --since today --no-pager 2>/dev/null \
      | grep -E -i "$pattern" || true
  fi
) | sed -E \
    -e 's/(Authorization: Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(api[_ -]?key[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/(OPENAI_API_KEY[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/sk-[A-Za-z0-9_-]{12,}/sk-[REDACTED]/g' \
    -e 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL-REDACTED]/g' \
  | tail -200
```

ข้อความบางส่วนที่มีประโยชน์มักประกอบด้วย `openai/gpt-5.6-sol` หรือ `openai/gpt-5.6-luna`,
`Runtime: OpenAI Codex`, `agentRuntime.id` หรือ `harnessRuntime`,
`candidateProvider: "openai"` และผลลัพธ์ `401`, `Incorrect API key` หรือ
`No API key` การรันที่แก้ไขแล้วควรแสดงเส้นทาง OAuth ของ OpenAI
แทนความล้มเหลว API key ของ OpenAI แบบธรรมดา

**ยังคงมีการกำหนดค่าอ้างอิงโมเดล Codex แบบเดิม:** รัน `openclaw doctor --fix`
Doctor จะเขียนการอ้างอิงโมเดลแบบเดิมใหม่เป็น `openai/*` ลบพินรันไทม์
ของเซสชันและทั้งเอเจนต์ที่ล้าสมัย และรักษาการเขียนทับโปรไฟล์การยืนยันตัวตนที่มีอยู่

**app-server ถูกปฏิเสธ:** ใช้ app-server ของ Codex `0.143.0` หรือใหม่กว่า
เวอร์ชันก่อนเผยแพร่ที่มีหมายเลขเวอร์ชันเดียวกันหรือเวอร์ชันที่มีส่วนต่อท้ายของบิลด์ เช่น
`0.143.0-alpha.2` หรือ `0.143.0+custom` จะถูกปฏิเสธ เนื่องจาก OpenClaw ทดสอบ
เกณฑ์ขั้นต่ำของโปรโตคอล `0.143.0` เวอร์ชันเสถียร

**`/codex status` ไม่สามารถเชื่อมต่อได้:** ตรวจสอบว่าเปิดใช้งาน Plugin `codex` แล้ว, `plugins.allow` รวม Plugin ดังกล่าวไว้เมื่อมีการกำหนดรายการอนุญาต และ `appServer.command`, `url`, `authToken` แบบกำหนดเอง หรือ
ส่วนหัวต่าง ๆ ถูกต้อง

**การค้นหาโมเดลทำงานช้า:** ลดค่า
`plugins.entries.codex.config.discovery.timeoutMs` หรือปิดใช้งานการค้นหา
ดู[ข้อมูลอ้างอิงของ Codex harness](/th/plugins/codex-harness-reference#model-discovery)

**การรับส่งข้อมูลผ่าน WebSocket ล้มเหลวทันที:** ตรวจสอบ `appServer.url`,
`authToken`, ส่วนหัวต่าง ๆ และตรวจสอบว่า app-server ระยะไกลใช้โปรโตคอล Codex
app-server เวอร์ชันเดียวกัน

**เครื่องมือเชลล์แบบเนทีฟหรือเครื่องมือแพตช์ถูกบล็อกด้วย `Native hook relay
unavailable`:** เธรด Codex ยังคงพยายามใช้รหัสรีเลย์ฮุกแบบเนทีฟ
ที่ไม่ได้ลงทะเบียนไว้กับ OpenClaw อีกต่อไป นี่เป็นปัญหาเกี่ยวกับการรับส่งข้อมูลของฮุก Codex แบบเนทีฟ
ไม่ใช่ความล้มเหลวของแบ็กเอนด์ ACP, ผู้ให้บริการ, GitHub หรือคำสั่งเชลล์
เริ่มเซสชันใหม่ในแชตที่ได้รับผลกระทบด้วย `/new` หรือ `/reset`
แล้วลองใช้คำสั่งที่ไม่ก่อให้เกิดอันตรายอีกครั้ง หากใช้ได้ครั้งหนึ่ง แต่การเรียกเครื่องมือแบบเนทีฟครั้งถัดไป
ล้มเหลวอีก ให้ถือว่า `/new` เป็นเพียงวิธีแก้ปัญหาชั่วคราว: คัดลอก
พรอมต์ไปยังเซสชันใหม่หลังจากรีสตาร์ต Codex app-server หรือ
OpenClaw Gateway เพื่อให้เธรดเก่าถูกยกเลิกและสร้างการลงทะเบียนฮุกแบบเนทีฟขึ้นใหม่

**การเรียกเครื่องมือ Codex สร้างกระบวนการฮุกอายุสั้นมากเกินไป:** ตั้งค่า
`plugins.entries.codex.config.appServer.loopDetectionPreToolUseRelay: false`
แล้วรีสตาร์ต Gateway การดำเนินการนี้จะปิดใช้งานเฉพาะกระบวนการย่อย `PreToolUse` ของ Codex
ที่ใช้สำหรับการตรวจหาลูปของ OpenClaw และเครื่องหมายไม่มีนโยบายของกระบวนการดังกล่าว รีเลย์นโยบาย
`before_tool_call` ที่จำเป็นและรีเลย์นโยบายเครื่องมือที่เชื่อถือได้ยังคงเปิดใช้งานอยู่

**โมเดลที่ไม่ใช่ Codex ใช้ harness ในตัว:** เป็นพฤติกรรมที่คาดไว้ เว้นแต่นโยบายรันไทม์ของผู้ให้บริการ
หรือโมเดลจะกำหนดเส้นทางไปยัง harness อื่น การอ้างอิงผู้ให้บริการทั่วไปที่ไม่ใช่ OpenAI
ยังคงใช้เส้นทางผู้ให้บริการตามปกติในโหมด `auto`

**ติดตั้ง Computer Use แล้ว แต่เครื่องมือไม่ทำงาน:** ตรวจสอบ
`/codex computer-use status` จากเซสชันใหม่ หากเครื่องมือรายงาน
`Native hook relay unavailable` ให้ใช้วิธีกู้คืนรีเลย์ฮุกแบบเนทีฟข้างต้น
ดู[Codex Computer Use](/th/plugins/codex-computer-use#troubleshooting)
