Plugin `codex` ที่บันเดิลมาด้วยช่วยให้ OpenClaw รันรอบเอเจนต์แบบฝังผ่าน
แอปเซิร์ฟเวอร์ Codex แทนฮาร์เนส PI ในตัว

ใช้สิ่งนี้เมื่อคุณต้องการให้ Codex เป็นเจ้าของเซสชันเอเจนต์ระดับล่าง:
การค้นหาโมเดล, การกลับมาใช้งานเธรดแบบเนทีฟ, Compaction แบบเนทีฟ และการดำเนินการผ่านแอปเซิร์ฟเวอร์
OpenClaw ยังคงเป็นเจ้าของช่องทางแชท, ไฟล์เซสชัน, การเลือกโมเดล, เครื่องมือ,
การอนุมัติ, การส่งมอบสื่อ และมิเรอร์ทรานสคริปต์ที่มองเห็นได้

เมื่อรอบแชทต้นทางรันผ่านฮาร์เนส Codex การตอบกลับที่มองเห็นได้จะใช้ค่าเริ่มต้น
เป็นเครื่องมือ `message` ของ OpenClaw หากการดีพลอยไม่ได้กำหนดค่า
`messages.visibleReplies` ไว้อย่างชัดเจน เอเจนต์ยังสามารถจบรอบ Codex ของตนแบบส่วนตัวได้;
ระบบจะโพสต์ไปยังช่องทางเฉพาะเมื่อเรียก `message(action="send")` ตั้งค่า
`messages.visibleReplies: "automatic"` เพื่อคงการตอบกลับสุดท้ายของแชทโดยตรงไว้บน
เส้นทางการส่งมอบอัตโนมัติแบบเดิม

รอบ Heartbeat ของ Codex จะได้รับเครื่องมือ `heartbeat_respond` ตามค่าเริ่มต้นเช่นกัน เพื่อให้
เอเจนต์สามารถบันทึกได้ว่าการปลุกควรเงียบต่อไปหรือแจ้งเตือน โดยไม่ต้องเข้ารหัส
โฟลว์ควบคุมนั้นไว้ในข้อความสุดท้าย

หากคุณกำลังพยายามทำความเข้าใจภาพรวม ให้เริ่มที่
[รันไทม์เอเจนต์](/th/concepts/agent-runtimes) สรุปสั้น ๆ คือ:
`openai/gpt-5.5` คือการอ้างอิงโมเดล, `codex` คือรันไทม์ และ Telegram,
Discord, Slack หรือช่องทางอื่นยังคงเป็นพื้นผิวการสื่อสาร

## การตั้งค่าแบบเร็ว

ผู้ใช้ส่วนใหญ่ที่ต้องการ "Codex ใน OpenClaw" ต้องการเส้นทางนี้: ลงชื่อเข้าใช้ด้วย
การสมัครสมาชิก ChatGPT/Codex แล้วรันรอบเอเจนต์แบบฝังผ่านรันไทม์
แอปเซิร์ฟเวอร์ Codex แบบเนทีฟ การอ้างอิงโมเดลยังคงเป็นมาตรฐานในรูป
`openai/gpt-*`; การยืนยันตัวตนแบบสมัครสมาชิกมาจากบัญชี/โปรไฟล์ Codex ไม่ใช่
จากคำนำหน้าโมเดล `openai-codex/*`

ลงชื่อเข้าใช้ด้วย Codex OAuth ก่อน หากคุณยังไม่ได้ทำ:

```bash
openclaw models auth login --provider openai-codex
```

จากนั้นเปิดใช้ Plugin `codex` ที่บันเดิลมา และบังคับใช้รันไทม์ Codex:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
        fallback: "none",
      },
    },
  },
}
```

หากการตั้งค่าของคุณใช้ `plugins.allow` ให้ใส่ `codex` ไว้ที่นั่นด้วย:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

อย่าใช้ `openai-codex/gpt-*` เมื่อคุณหมายถึงรันไทม์ Codex แบบเนทีฟ คำนำหน้านั้น
คือเส้นทาง "Codex OAuth ผ่าน PI" อย่างชัดเจน การเปลี่ยนแปลงการตั้งค่าจะมีผลกับเซสชันใหม่หรือ
เซสชันที่รีเซ็ตแล้ว; เซสชันที่มีอยู่จะคงรันไทม์ที่บันทึกไว้

## สิ่งที่ Plugin นี้เปลี่ยน

Plugin `codex` ที่บันเดิลมามอบความสามารถแยกกันหลายอย่าง:

| ความสามารถ                        | วิธีใช้                                      | สิ่งที่ทำ                                                                  |
| --------------------------------- | --------------------------------------------------- | ----------------------------------------------------------------------------- |
| รันไทม์แบบฝังเนทีฟ           | `agentRuntime.id: "codex"`                          | รันรอบเอเจนต์แบบฝังของ OpenClaw ผ่านแอปเซิร์ฟเวอร์ Codex                  |
| คำสั่งควบคุมแชทเนทีฟ      | `/codex bind`, `/codex resume`, `/codex steer`, ... | ผูกและควบคุมเธรดแอปเซิร์ฟเวอร์ Codex จากการสนทนาในระบบข้อความ    |
| ผู้ให้บริการ/แคตตาล็อกแอปเซิร์ฟเวอร์ Codex | อินเทอร์นัลของ `codex` ที่แสดงผ่านฮาร์เนส     | ให้รันไทม์ค้นหาและตรวจสอบโมเดลแอปเซิร์ฟเวอร์                     |
| เส้นทางทำความเข้าใจสื่อของ Codex    | เส้นทางความเข้ากันได้ของโมเดลภาพ `codex/*`           | รันรอบแอปเซิร์ฟเวอร์ Codex แบบจำกัดขอบเขตสำหรับโมเดลทำความเข้าใจภาพที่รองรับ |
| รีเลย์ฮุกเนทีฟ                 | ฮุก Plugin รอบเหตุการณ์เนทีฟของ Codex             | ให้ OpenClaw สังเกต/บล็อกเหตุการณ์เครื่องมือ/การสรุปผลเนทีฟของ Codex ที่รองรับ  |

การเปิดใช้ Plugin ทำให้ความสามารถเหล่านั้นพร้อมใช้งาน สิ่งนี้ **ไม่**:

- เริ่มใช้ Codex สำหรับโมเดล OpenAI ทุกตัว
- แปลงการอ้างอิงโมเดล `openai-codex/*` ให้เป็นรันไทม์เนทีฟ
- ทำให้ ACP/acpx เป็นเส้นทาง Codex เริ่มต้น
- สลับแบบทันทีให้เซสชันที่มีอยู่ซึ่งบันทึกรันไทม์ PI ไว้แล้ว
- แทนที่การส่งมอบช่องทางของ OpenClaw, ไฟล์เซสชัน, พื้นที่เก็บโปรไฟล์การยืนยันตัวตน หรือ
  การกำหนดเส้นทางข้อความ

Plugin เดียวกันนี้ยังเป็นเจ้าของพื้นผิวคำสั่งควบคุมแชทเนทีฟ `/codex` ด้วย หาก
เปิดใช้ Plugin แล้วและผู้ใช้ขอให้ผูก, กลับมาใช้งานต่อ, ชี้นำ, หยุด หรือตรวจสอบ
เธรด Codex จากแชท เอเจนต์ควรเลือกใช้ `/codex ...` แทน ACP ACP ยังคงเป็น
ทางสำรองที่ชัดเจนเมื่อผู้ใช้ขอ ACP/acpx หรือกำลังทดสอบอะแดปเตอร์
ACP Codex

รอบ Codex แบบเนทีฟยังคงใช้ฮุก Plugin ของ OpenClaw เป็นเลเยอร์ความเข้ากันได้สาธารณะ
สิ่งเหล่านี้คือฮุก OpenClaw ในโปรเซส ไม่ใช่ฮุกคำสั่ง `hooks.json` ของ Codex:

- `before_prompt_build`
- `before_compaction`, `after_compaction`
- `llm_input`, `llm_output`
- `before_tool_call`, `after_tool_call`
- `before_message_write` สำหรับเรคคอร์ดทรานสคริปต์ที่มิเรอร์
- `before_agent_finalize` ผ่านรีเลย์ `Stop` ของ Codex
- `agent_end`

Plugin ยังสามารถลงทะเบียนมิดเดิลแวร์ผลลัพธ์เครื่องมือที่ไม่ผูกกับรันไทม์ เพื่อเขียนใหม่
ผลลัพธ์เครื่องมือแบบไดนามิกของ OpenClaw หลังจาก OpenClaw ดำเนินการเครื่องมือแล้วและก่อน
ส่งผลลัพธ์กลับไปยัง Codex สิ่งนี้แยกจากฮุก Plugin สาธารณะ
`tool_result_persist` ซึ่งแปลงการเขียนผลลัพธ์เครื่องมือในทรานสคริปต์ที่ OpenClaw เป็นเจ้าของ

สำหรับความหมายของฮุก Plugin เอง ดู [ฮุก Plugin](/th/plugins/hooks)
และ [พฤติกรรมการ์ดของ Plugin](/th/tools/plugin)

ฮาร์เนสปิดอยู่ตามค่าเริ่มต้น การตั้งค่าใหม่ควรคงการอ้างอิงโมเดล OpenAI
ให้อยู่ในรูปมาตรฐาน `openai/gpt-*` และบังคับใช้อย่างชัดเจนด้วย
`agentRuntime.id: "codex"` หรือ `OPENCLAW_AGENT_RUNTIME=codex` เมื่อต้องการ
การดำเนินการผ่านแอปเซิร์ฟเวอร์แบบเนทีฟ การอ้างอิงโมเดล `codex/*` แบบเดิมยังคงเลือก
ฮาร์เนสอัตโนมัติเพื่อความเข้ากันได้ แต่คำนำหน้าผู้ให้บริการเดิมที่มีรันไทม์รองรับจะ
ไม่แสดงเป็นตัวเลือกโมเดล/ผู้ให้บริการปกติ

หากเปิดใช้ Plugin `codex` แล้ว แต่โมเดลหลักยังคงเป็น
`openai-codex/*`, `openclaw doctor` จะเตือนแทนการเปลี่ยนเส้นทาง นี่เป็น
พฤติกรรมโดยเจตนา: `openai-codex/*` ยังคงเป็นเส้นทาง Codex OAuth/การสมัครสมาชิกผ่าน PI และ
การดำเนินการผ่านแอปเซิร์ฟเวอร์แบบเนทีฟยังคงเป็นตัวเลือกรันไทม์ที่ต้องระบุอย่างชัดเจน

## แผนผังเส้นทาง

ใช้ตารางนี้ก่อนเปลี่ยนการตั้งค่า:

| พฤติกรรมที่ต้องการ                                     | การอ้างอิงโมเดล                  | การตั้งค่ารันไทม์                         | เส้นทางการยืนยันตัวตน/โปรไฟล์           | ป้ายสถานะที่คาดหวัง          |
| ---------------------------------------------------- | -------------------------- | -------------------------------------- | ---------------------------- | ------------------------------ |
| การสมัครสมาชิก ChatGPT/Codex พร้อมรันไทม์ Codex แบบเนทีฟ | `openai/gpt-*`             | `agentRuntime.id: "codex"`             | Codex OAuth หรือบัญชี Codex | `Runtime: OpenAI Codex`        |
| OpenAI API ผ่านรันเนอร์ OpenClaw ปกติ            | `openai/gpt-*`             | ละไว้ หรือ `runtime: "pi"`             | คีย์ OpenAI API               | `Runtime: OpenClaw Pi Default` |
| การสมัครสมาชิก ChatGPT/Codex ผ่าน PI                | `openai-codex/gpt-*`       | ละไว้ หรือ `runtime: "pi"`             | ผู้ให้บริการ OpenAI Codex OAuth  | `Runtime: OpenClaw Pi Default` |
| ผู้ให้บริการแบบผสมพร้อมโหมดอัตโนมัติแบบอนุรักษ์นิยม          | การอ้างอิงเฉพาะผู้ให้บริการ     | `agentRuntime.id: "auto"`              | ตามผู้ให้บริการที่เลือก        | ขึ้นอยู่กับรันไทม์ที่เลือก    |
| เซสชันอะแดปเตอร์ Codex ACP แบบชัดเจน                   | ขึ้นอยู่กับพรอมป์/โมเดล ACP | `sessions_spawn` พร้อม `runtime: "acp"` | การยืนยันตัวตนแบ็กเอนด์ ACP             | สถานะงาน/เซสชัน ACP        |

จุดแบ่งที่สำคัญคือผู้ให้บริการกับรันไทม์:

- `openai-codex/*` ตอบคำถามว่า "PI ควรใช้เส้นทางผู้ให้บริการ/การยืนยันตัวตนใด?"
- `agentRuntime.id: "codex"` ตอบคำถามว่า "ลูปใดควรดำเนินการ
  รอบแบบฝังนี้?"
- `/codex ...` ตอบคำถามว่า "การสนทนา Codex แบบเนทีฟใดที่แชทนี้ควรผูก
  หรือควบคุม?"
- ACP ตอบคำถามว่า "กระบวนการฮาร์เนสภายนอกใดที่ acpx ควรเปิด?"

## เลือกคำนำหน้าโมเดลให้ถูกต้อง

เส้นทางตระกูล OpenAI ขึ้นกับคำนำหน้า สำหรับการตั้งค่าทั่วไปแบบสมัครสมาชิกพร้อม
รันไทม์ Codex แบบเนทีฟ ให้ใช้ `openai/*` พร้อม `agentRuntime.id: "codex"`
ใช้ `openai-codex/*` เฉพาะเมื่อคุณตั้งใจต้องการ Codex OAuth ผ่าน PI:

| การอ้างอิงโมเดล                                     | เส้นทางรันไทม์                                 | ใช้เมื่อ                                                                  |
| --------------------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------- |
| `openai/gpt-5.4`                              | ผู้ให้บริการ OpenAI ผ่านระบบเชื่อม OpenClaw/PI | คุณต้องการเข้าถึง OpenAI Platform API โดยตรงในปัจจุบันด้วย `OPENAI_API_KEY` |
| `openai-codex/gpt-5.5`                        | OpenAI Codex OAuth ผ่าน OpenClaw/PI       | คุณต้องการการยืนยันตัวตนแบบสมัครสมาชิก ChatGPT/Codex กับรันเนอร์ PI เริ่มต้น      |
| `openai/gpt-5.5` + `agentRuntime.id: "codex"` | ฮาร์เนสแอปเซิร์ฟเวอร์ Codex                     | คุณต้องการการยืนยันตัวตนแบบสมัครสมาชิก ChatGPT/Codex พร้อมการดำเนินการ Codex แบบเนทีฟ     |

GPT-5.5 อาจปรากฏได้ทั้งบนเส้นทางคีย์ OpenAI API โดยตรงและเส้นทางการสมัครสมาชิก Codex
เมื่อบัญชีของคุณเปิดเผยให้ใช้ ใช้ `openai/gpt-5.5` พร้อมฮาร์เนสแอปเซิร์ฟเวอร์ Codex
สำหรับรันไทม์ Codex แบบเนทีฟ, ใช้ `openai-codex/gpt-5.5` สำหรับ PI OAuth หรือ
`openai/gpt-5.5` โดยไม่มีการแทนที่รันไทม์ Codex สำหรับทราฟฟิกผ่านคีย์ API โดยตรง

การอ้างอิง `codex/gpt-*` แบบเดิมยังคงรับได้ในฐานะนามแฝงเพื่อความเข้ากันได้ Doctor
การย้ายข้อมูลเพื่อความเข้ากันได้จะเขียนการอ้างอิงรันไทม์หลักแบบเดิมใหม่เป็นการอ้างอิงโมเดล
มาตรฐาน และบันทึกนโยบายรันไทม์แยกต่างหาก ขณะที่การอ้างอิงแบบเดิมที่เป็นเฉพาะ fallback
จะถูกปล่อยไว้ไม่เปลี่ยนแปลง เพราะรันไทม์ถูกกำหนดค่าสำหรับคอนเทนเนอร์เอเจนต์ทั้งชุด
การตั้งค่า PI Codex OAuth ใหม่ควรใช้ `openai-codex/gpt-*`; การตั้งค่าฮาร์เนส
แอปเซิร์ฟเวอร์เนทีฟใหม่ควรใช้ `openai/gpt-*` พร้อม
`agentRuntime.id: "codex"`

`agents.defaults.imageModel` ใช้การแยกคำนำหน้าแบบเดียวกัน ใช้
`openai-codex/gpt-*` เมื่อการทำความเข้าใจภาพควรรันผ่านเส้นทางผู้ให้บริการ OpenAI
Codex OAuth ใช้ `codex/gpt-*` เมื่อการทำความเข้าใจภาพควรรัน
ผ่านรอบแอปเซิร์ฟเวอร์ Codex แบบจำกัดขอบเขต โมเดลแอปเซิร์ฟเวอร์ Codex ต้อง
ประกาศการรองรับอินพุตภาพ; โมเดล Codex แบบข้อความเท่านั้นจะล้มเหลวก่อนที่รอบสื่อ
จะเริ่ม

ใช้ `/status` เพื่อยืนยันฮาร์เนสที่มีผลสำหรับเซสชันปัจจุบัน หาก
การเลือกดูผิดคาด ให้เปิดใช้การบันทึกดีบักสำหรับระบบย่อย `agents/harness`
และตรวจดูเรคคอร์ดแบบมีโครงสร้าง `agent harness selected` ของ Gateway
โดยมีรหัสฮาร์เนสที่เลือก, เหตุผลการเลือก, นโยบายรันไทม์/fallback และ
ในโหมด `auto` ผลการรองรับของผู้สมัคร Plugin แต่ละตัว

### คำเตือนจาก doctor หมายความว่าอย่างไร

`openclaw doctor` เตือนเมื่อทุกข้อเหล่านี้เป็นจริง:

- เปิดใช้หรืออนุญาต Plugin `codex` ที่บันเดิลมา
- โมเดลหลักของเอเจนต์เป็น `openai-codex/*`
- รันไทม์ที่มีผลของเอเจนต์นั้นไม่ใช่ `codex`

คำเตือนนั้นมีอยู่เพราะผู้ใช้มักคาดว่า "เปิดใช้ Plugin Codex" จะหมายถึง
"รันไทม์แอปเซิร์ฟเวอร์ Codex แบบเนทีฟ" OpenClaw ไม่สรุปเช่นนั้น คำเตือน
หมายความว่า:

- **ไม่จำเป็นต้องเปลี่ยนแปลง** หากคุณตั้งใจใช้ ChatGPT/Codex OAuth ผ่าน PI
- เปลี่ยนโมเดลเป็น `openai/<model>` และตั้งค่า
  `agentRuntime.id: "codex"` หากคุณตั้งใจใช้การดำเนินการผ่านแอปเซิร์ฟเวอร์
  แบบเนทีฟ
- เซสชันที่มีอยู่ยังต้องใช้ `/new` หรือ `/reset` หลังเปลี่ยนรันไทม์
  เพราะพินรันไทม์ของเซสชันเป็นแบบคงอยู่

การเลือกฮาร์เนสไม่ใช่ตัวควบคุมเซสชันแบบสด เมื่อรอบแบบฝังรัน
OpenClaw จะบันทึกรหัสฮาร์เนสที่เลือกไว้บนเซสชันนั้นและใช้ต่อไปสำหรับ
รอบภายหลังในรหัสเซสชันเดียวกัน เปลี่ยนการตั้งค่า `agentRuntime` หรือ
`OPENCLAW_AGENT_RUNTIME` เมื่อคุณต้องการให้เซสชันในอนาคตใช้ฮาร์เนสอื่น;
ใช้ `/new` หรือ `/reset` เพื่อเริ่มเซสชันใหม่ก่อนสลับการสนทนาที่มีอยู่
ระหว่าง PI และ Codex วิธีนี้หลีกเลี่ยงการเล่นทรานสคริปต์เดียวซ้ำผ่าน
ระบบเซสชันเนทีฟสองระบบที่เข้ากันไม่ได้

เซสชันเดิมที่สร้างก่อนมีพินฮาร์เนสจะถือว่าเป็นแบบพินกับ PI เมื่อ
มีประวัติทรานสคริปต์แล้ว ใช้ `/new` หรือ `/reset` เพื่อเลือกให้การสนทนานั้น
เข้าสู่ Codex หลังเปลี่ยนการตั้งค่า

`/status` แสดงรันไทม์ของโมเดลที่มีผลใช้งาน ฮาร์เนส PI เริ่มต้นจะแสดงเป็น
`Runtime: OpenClaw Pi Default` และฮาร์เนสแอปเซิร์ฟเวอร์ Codex จะแสดงเป็น
`Runtime: OpenAI Codex`.

## ข้อกำหนด

- OpenClaw พร้อม Plugin `codex` ที่รวมมาให้ใช้งานได้
- แอปเซิร์ฟเวอร์ Codex `0.125.0` หรือใหม่กว่า Plugin ที่รวมมาจะจัดการไบนารีแอปเซิร์ฟเวอร์ Codex ที่เข้ากันได้ตามค่าเริ่มต้น ดังนั้นคำสั่ง `codex` ภายในเครื่องบน `PATH` จะไม่ส่งผลต่อการเริ่มฮาร์เนสตามปกติ
- มีการยืนยันตัวตน Codex ให้พร้อมใช้งานสำหรับโปรเซสแอปเซิร์ฟเวอร์หรือบริดจ์การยืนยันตัวตน Codex ของ OpenClaw การเปิดแอปเซิร์ฟเวอร์ภายในเครื่องจะใช้โฮม Codex ที่ OpenClaw จัดการให้สำหรับแต่ละเอเจนต์ และใช้ `HOME` ของโปรเซสลูกที่แยกออกมา ดังนั้นตามค่าเริ่มต้น จึงจะไม่อ่านบัญชี `~/.codex` ส่วนตัว Skills plugins การกำหนดค่า สถานะเธรด หรือ `$HOME/.agents/skills` ดั้งเดิมของคุณ

Plugin จะบล็อกแฮนด์เชคของแอปเซิร์ฟเวอร์ที่เก่ากว่าหรือไม่มีเวอร์ชัน วิธีนี้ทำให้
OpenClaw อยู่บนพื้นผิวโปรโตคอลที่ผ่านการทดสอบแล้ว

สำหรับการทดสอบควันแบบสดและ Docker การยืนยันตัวตนมักมาจากบัญชี Codex CLI
หรือโปรไฟล์การยืนยันตัวตน `openai-codex` ของ OpenClaw การเปิดแอปเซิร์ฟเวอร์ stdio ภายในเครื่องยังสามารถถอยกลับไปใช้ `CODEX_API_KEY` / `OPENAI_API_KEY` ได้เมื่อไม่มีบัญชีอยู่

## เพิ่ม Codex ควบคู่กับโมเดลอื่น

อย่าตั้ง `agentRuntime.id: "codex"` แบบส่วนกลางหากเอเจนต์เดียวกันควรสลับได้อย่างอิสระระหว่าง Codex และโมเดลผู้ให้บริการที่ไม่ใช่ Codex รันไทม์ที่บังคับใช้จะมีผลกับทุกเทิร์นที่ฝังอยู่สำหรับเอเจนต์หรือเซสชันนั้น หากคุณเลือกโมเดล Anthropic ในขณะที่บังคับใช้รันไทม์นั้น OpenClaw ยังคงพยายามใช้ฮาร์เนส Codex และปิดแบบล้มเหลวแทนที่จะส่งเทิร์นนั้นผ่าน PI อย่างเงียบๆ

ให้ใช้รูปแบบใดรูปแบบหนึ่งเหล่านี้แทน:

- วาง Codex ไว้บนเอเจนต์เฉพาะด้วย `agentRuntime.id: "codex"`
- ให้เอเจนต์เริ่มต้นใช้ `agentRuntime.id: "auto"` และการถอยกลับไปใช้ PI สำหรับการใช้งานผู้ให้บริการแบบผสมตามปกติ
- ใช้การอ้างอิง `codex/*` แบบเดิมเพื่อความเข้ากันได้เท่านั้น การกำหนดค่าใหม่ควรเลือกใช้ `openai/*` พร้อมนโยบายรันไทม์ Codex ที่ระบุชัดเจน

ตัวอย่างเช่น รูปแบบนี้คงเอเจนต์เริ่มต้นไว้กับการเลือกอัตโนมัติตามปกติ และเพิ่มเอเจนต์ Codex แยกต่างหาก:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      agentRuntime: {
        id: "auto",
        fallback: "pi",
      },
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
        agentRuntime: {
          id: "codex",
        },
      },
    ],
  },
}
```

ด้วยรูปแบบนี้:

- เอเจนต์ `main` เริ่มต้นใช้เส้นทางผู้ให้บริการปกติและการถอยกลับเพื่อความเข้ากันได้กับ PI
- เอเจนต์ `codex` ใช้ฮาร์เนสแอปเซิร์ฟเวอร์ Codex
- หาก Codex หายไปหรือไม่รองรับสำหรับเอเจนต์ `codex` เทิร์นจะล้มเหลวแทนที่จะใช้ PI อย่างเงียบๆ

## การจัดเส้นทางคำสั่งของเอเจนต์

เอเจนต์ควรจัดเส้นทางคำขอของผู้ใช้ตามเจตนา ไม่ใช่จากคำว่า "Codex" เพียงอย่างเดียว:

| ผู้ใช้ขอให้...                                       | เอเจนต์ควรใช้...                              |
| ------------------------------------------------------ | ------------------------------------------------ |
| "ผูกแชตนี้กับ Codex"                              | `/codex bind`                                    |
| "ดำเนินเธรด Codex `<id>` ต่อที่นี่"                      | `/codex resume <id>`                             |
| "แสดงเธรด Codex"                                   | `/codex threads`                                 |
| "ส่งรายงานสนับสนุนสำหรับการรัน Codex ที่มีปัญหา"            | `/diagnostics [note]`                            |
| "ส่งคำติชม Codex สำหรับเธรดที่แนบมานี้เท่านั้น"    | `/codex diagnostics [note]`                      |
| "ใช้การสมัครสมาชิก ChatGPT/Codex ของฉันกับรันไทม์ Codex" | `openai/*` บวก `agentRuntime.id: "codex"`       |
| "ใช้การสมัครสมาชิก ChatGPT/Codex ของฉันผ่าน PI"         | การอ้างอิงโมเดล `openai-codex/*`                      |
| "รัน Codex ผ่าน ACP/acpx"                           | ACP `sessions_spawn({ runtime: "acp", ... })`    |
| "เริ่ม Claude Code/Gemini/OpenCode/Cursor ในเธรด" | ACP/acpx ไม่ใช่ `/codex` และไม่ใช่เอเจนต์ย่อยดั้งเดิม |

OpenClaw จะประกาศคำแนะนำการสปอว์น ACP ให้เอเจนต์เฉพาะเมื่อ ACP เปิดใช้งาน ส่งต่อได้ และมีแบ็กเอนด์รันไทม์ที่โหลดอยู่รองรับ หาก ACP ไม่พร้อมใช้งาน พรอมป์ระบบและ Plugin Skills ไม่ควรสอนเอเจนต์เกี่ยวกับการจัดเส้นทาง ACP

## การปรับใช้เฉพาะ Codex

บังคับใช้ฮาร์เนส Codex เมื่อคุณต้องพิสูจน์ว่าทุกเทิร์นของเอเจนต์ที่ฝังอยู่ใช้ Codex รันไทม์ Plugin ที่ระบุชัดเจนจะมีค่าเริ่มต้นเป็นไม่มีการถอยกลับไปใช้ PI ดังนั้น `fallback: "none"` จึงเป็นทางเลือก แต่ก็มักมีประโยชน์ในฐานะเอกสารประกอบ:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
        fallback: "none",
      },
    },
  },
}
```

การแทนที่ด้วยสภาพแวดล้อม:

```bash
OPENCLAW_AGENT_RUNTIME=codex openclaw gateway run
```

เมื่อบังคับใช้ Codex แล้ว OpenClaw จะล้มเหลวตั้งแต่ต้นหาก Plugin Codex ถูกปิดใช้งาน แอปเซิร์ฟเวอร์เก่าเกินไป หรือแอปเซิร์ฟเวอร์เริ่มไม่ได้ ตั้ง `OPENCLAW_AGENT_HARNESS_FALLBACK=pi` เฉพาะเมื่อคุณตั้งใจให้ PI จัดการกรณีที่เลือกฮาร์เนสไม่ได้

## Codex ต่อเอเจนต์

คุณสามารถทำให้เอเจนต์หนึ่งเป็นเฉพาะ Codex ในขณะที่เอเจนต์เริ่มต้นยังคงการเลือกอัตโนมัติตามปกติ:

```json5
{
  agents: {
    defaults: {
      agentRuntime: {
        id: "auto",
        fallback: "pi",
      },
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
        agentRuntime: {
          id: "codex",
          fallback: "none",
        },
      },
    ],
  },
}
```

ใช้คำสั่งเซสชันปกติเพื่อสลับเอเจนต์และโมเดล `/new` สร้างเซสชัน OpenClaw ใหม่ และฮาร์เนส Codex จะสร้างหรือดำเนินเธรดแอปเซิร์ฟเวอร์ sidecar ของตนต่อตามจำเป็น `/reset` ล้างการผูกเซสชัน OpenClaw สำหรับเธรดนั้น และให้เทิร์นถัดไปแก้รันไทม์จากการกำหนดค่าปัจจุบันอีกครั้ง

## การค้นหาโมเดล

ตามค่าเริ่มต้น Plugin Codex จะถามแอปเซิร์ฟเวอร์ถึงโมเดลที่พร้อมใช้งาน หากการค้นหาล้มเหลวหรือหมดเวลา จะใช้แค็ตตาล็อกสำรองที่รวมมาให้สำหรับ:

- GPT-5.5
- GPT-5.4 mini
- GPT-5.2

คุณสามารถปรับแต่งการค้นหาได้ภายใต้ `plugins.entries.codex.config.discovery`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          discovery: {
            enabled: true,
            timeoutMs: 2500,
          },
        },
      },
    },
  },
}
```

ปิดใช้งานการค้นหาเมื่อคุณต้องการให้การเริ่มต้นหลีกเลี่ยงการโพรบ Codex และยึดตามแค็ตตาล็อกสำรอง:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          discovery: {
            enabled: false,
          },
        },
      },
    },
  },
}
```

## การเชื่อมต่อและนโยบายแอปเซิร์ฟเวอร์

ตามค่าเริ่มต้น Plugin จะเริ่มไบนารี Codex ที่ OpenClaw จัดการภายในเครื่องด้วย:

```bash
codex app-server --listen stdio://
```

ไบนารีที่จัดการจะถูกจัดส่งมาพร้อมกับแพ็กเกจ Plugin `codex` วิธีนี้ทำให้เวอร์ชันแอปเซิร์ฟเวอร์ผูกกับ Plugin ที่รวมมา แทนที่จะขึ้นกับ Codex CLI แยกต่างหากใดๆ ที่บังเอิญติดตั้งไว้ภายในเครื่อง ตั้ง `appServer.command` เฉพาะเมื่อคุณตั้งใจจะรันไฟล์ปฏิบัติการอื่น

ตามค่าเริ่มต้น OpenClaw จะเริ่มเซสชันฮาร์เนส Codex ภายในเครื่องในโหมด YOLO:
`approvalPolicy: "never"`, `approvalsReviewer: "user"` และ
`sandbox: "danger-full-access"` นี่คือท่าทีของผู้ปฏิบัติการภายในเครื่องที่เชื่อถือแล้วซึ่งใช้สำหรับ Heartbeat อัตโนมัติ: Codex สามารถใช้เครื่องมือเชลล์และเครือข่ายได้โดยไม่หยุดรอพรอมป์อนุมัติดั้งเดิมที่ไม่มีใครอยู่ตอบ

หากต้องการเลือกใช้การอนุมัติที่ผู้พิทักษ์ Codex ตรวจทาน ให้ตั้ง `appServer.mode:
"guardian"`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            mode: "guardian",
            serviceTier: "fast",
          },
        },
      },
    },
  },
}
```

โหมดผู้พิทักษ์ใช้เส้นทางการอนุมัติแบบตรวจทานอัตโนมัติดั้งเดิมของ Codex เมื่อ Codex ขอออกจากแซนด์บ็อกซ์ เขียนนอกพื้นที่ทำงาน หรือเพิ่มสิทธิ์ เช่น การเข้าถึงเครือข่าย Codex จะจัดเส้นทางคำขออนุมัตินั้นไปยังผู้ตรวจทานดั้งเดิมแทนพรอมป์มนุษย์ ผู้ตรวจทานจะใช้กรอบความเสี่ยงของ Codex และอนุมัติหรือปฏิเสธคำขอเฉพาะนั้น ใช้ผู้พิทักษ์เมื่อคุณต้องการกรอบควบคุมมากกว่าโหมด YOLO แต่ยังต้องให้เอเจนต์ที่ไม่มีคนเฝ้าดำเนินงานต่อได้

พรีเซ็ต `guardian` ขยายเป็น `approvalPolicy: "on-request"`,
`approvalsReviewer: "auto_review"` และ `sandbox: "workspace-write"`
ฟิลด์นโยบายแต่ละรายการยังคงแทนที่ `mode` ได้ ดังนั้นการปรับใช้ขั้นสูงจึงผสมพรีเซ็ตกับตัวเลือกที่ระบุชัดเจนได้ ค่ารีวิวเวอร์ `guardian_subagent` แบบเก่ายังคงยอมรับเป็นนามแฝงเพื่อความเข้ากันได้ แต่การกำหนดค่าใหม่ควรใช้ `auto_review`

สำหรับแอปเซิร์ฟเวอร์ที่กำลังรันอยู่แล้ว ให้ใช้การรับส่ง WebSocket:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://127.0.0.1:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
            requestTimeoutMs: 60000,
          },
        },
      },
    },
  },
}
```

การเปิดแอปเซิร์ฟเวอร์ stdio จะสืบทอดสภาพแวดล้อมโปรเซสของ OpenClaw ตามค่าเริ่มต้น แต่ OpenClaw เป็นเจ้าของบริดจ์บัญชีแอปเซิร์ฟเวอร์ Codex และตั้งทั้ง `CODEX_HOME` และ `HOME` เป็นไดเรกทอรีต่อเอเจนต์ภายใต้สถานะ OpenClaw ของเอเจนต์นั้น ตัวโหลด Skill ของ Codex เองอ่าน `$CODEX_HOME/skills` และ `$HOME/.agents/skills` ดังนั้นค่าทั้งสองจึงถูกแยกสำหรับการเปิดแอปเซิร์ฟเวอร์ภายในเครื่อง วิธีนี้ทำให้ Skills ดั้งเดิมของ Codex, plugins, การกำหนดค่า, บัญชี และสถานะเธรด ถูกจำกัดขอบเขตไว้กับเอเจนต์ OpenClaw แทนที่จะรั่วเข้ามาจากโฮม Codex CLI ส่วนตัวของผู้ปฏิบัติการ

OpenClaw plugins และสแนปชอต OpenClaw Skills ยังคงไหลผ่านรีจิสทรี Plugin และตัวโหลด Skill ของ OpenClaw เอง ทรัพยากร Codex CLI ส่วนตัวจะไม่ไหลผ่าน หากคุณมี Codex CLI Skills หรือ plugins ที่มีประโยชน์และควรกลายเป็นส่วนหนึ่งของเอเจนต์ OpenClaw ให้ทำบัญชีรายการอย่างชัดเจน:

```bash
openclaw migrate codex --dry-run
openclaw migrate apply codex --yes
```

ผู้ให้บริการการย้ายข้อมูล Codex จะคัดลอก Skills ไปยังพื้นที่ทำงานของเอเจนต์ OpenClaw ปัจจุบัน Plugin ดั้งเดิมของ Codex, hooks และไฟล์กำหนดค่าจะถูกรายงานหรือเก็บถาวรเพื่อให้ตรวจทานด้วยตนเองแทนการเปิดใช้งานอัตโนมัติ เพราะสิ่งเหล่านี้สามารถเรียกใช้คำสั่ง เปิดเผยเซิร์ฟเวอร์ MCP หรือพกข้อมูลรับรองได้

การยืนยันตัวตนจะถูกเลือกตามลำดับนี้:

1. โปรไฟล์การยืนยันตัวตน OpenClaw Codex ที่ระบุชัดเจนสำหรับเอเจนต์
2. บัญชีที่มีอยู่ของแอปเซิร์ฟเวอร์ในโฮม Codex ของเอเจนต์นั้น
3. สำหรับการเปิดแอปเซิร์ฟเวอร์ stdio ภายในเครื่องเท่านั้น ใช้ `CODEX_API_KEY` แล้วจึงใช้
   `OPENAI_API_KEY` เมื่อไม่มีบัญชีแอปเซิร์ฟเวอร์อยู่และยังต้องใช้การยืนยันตัวตน OpenAI

เมื่อ OpenClaw เห็นโปรไฟล์การยืนยันตัวตน Codex แบบการสมัครสมาชิก ChatGPT จะนำ `CODEX_API_KEY` และ `OPENAI_API_KEY` ออกจากโปรเซสลูก Codex ที่สปอว์นขึ้น วิธีนี้ทำให้คีย์ API ระดับ Gateway ยังคงพร้อมใช้งานสำหรับการฝังหรือโมเดล OpenAI โดยตรง โดยไม่ทำให้เทิร์นแอปเซิร์ฟเวอร์ Codex ดั้งเดิมถูกคิดค่าบริการผ่าน API โดยไม่ตั้งใจ โปรไฟล์คีย์ API ของ Codex ที่ระบุชัดเจนและการถอยกลับไปใช้คีย์สภาพแวดล้อม stdio ภายในเครื่องจะใช้การเข้าสู่ระบบแอปเซิร์ฟเวอร์แทนการสืบทอดสภาพแวดล้อมโปรเซสลูก การเชื่อมต่อแอปเซิร์ฟเวอร์ WebSocket จะไม่ได้รับการถอยกลับไปใช้คีย์ API จากสภาพแวดล้อม Gateway ให้ใช้โปรไฟล์การยืนยันตัวตนที่ระบุชัดเจนหรือบัญชีของแอปเซิร์ฟเวอร์ระยะไกลเอง

หากการปรับใช้ต้องการการแยกสภาพแวดล้อมเพิ่มเติม ให้เพิ่มตัวแปรเหล่านั้นใน
`appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` มีผลเฉพาะกับโปรเซสลูกแอปเซิร์ฟเวอร์ Codex ที่สปอว์นขึ้น

เครื่องมือแบบไดนามิกของ Codex มีค่าเริ่มต้นเป็นโปรไฟล์ `native-first` ในโหมดนั้น OpenClaw จะไม่เปิดเผยเครื่องมือแบบไดนามิกที่ซ้ำกับการดำเนินงานพื้นที่ทำงานดั้งเดิมของ Codex: `read`, `write`, `edit`, `apply_patch`, `exec`, `process` และ
`update_plan` เครื่องมือบูรณาการ OpenClaw เช่น การส่งข้อความ เซสชัน สื่อ cron เบราว์เซอร์ nodes, gateway, `heartbeat_respond` และ `web_search` ยังคงพร้อมใช้งาน

ฟิลด์ระดับบนสุดของ Plugin Codex ที่รองรับ:

| Field                      | Default          | Meaning                                                                                   |
| -------------------------- | ---------------- | ----------------------------------------------------------------------------------------- |
| `codexDynamicToolsProfile` | `"native-first"` | ใช้ `"openclaw-compat"` เพื่อเปิดเผยชุดเครื่องมือแบบไดนามิกทั้งหมดของ OpenClaw ให้กับ Codex app-server |
| `codexDynamicToolsExclude` | `[]`             | ชื่อเครื่องมือแบบไดนามิกเพิ่มเติมของ OpenClaw ที่ต้องละเว้นจากเทิร์นของ Codex app-server               |

ฟิลด์ `appServer` ที่รองรับ:

| Field               | Default                                  | Meaning                                                                                                                                                                                                                              |
| ------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `transport`         | `"stdio"`                                | `"stdio"` สร้าง Codex; `"websocket"` เชื่อมต่อกับ `url`                                                                                                                                                                             |
| `command`           | managed Codex binary                     | ไฟล์ปฏิบัติการสำหรับ stdio transport เว้นว่างไว้เพื่อใช้ไบนารีที่จัดการให้; ตั้งค่าเฉพาะเมื่อต้องการ override อย่างชัดเจน                                                                                                                         |
| `args`              | `["app-server", "--listen", "stdio://"]` | อาร์กิวเมนต์สำหรับ stdio transport                                                                                                                                                                                                       |
| `url`               | unset                                    | URL ของ WebSocket app-server                                                                                                                                                                                                            |
| `authToken`         | unset                                    | Bearer token สำหรับ WebSocket transport                                                                                                                                                                                                |
| `headers`           | `{}`                                     | เฮดเดอร์ WebSocket เพิ่มเติม                                                                                                                                                                                                             |
| `clearEnv`          | `[]`                                     | ชื่อตัวแปรสภาพแวดล้อมเพิ่มเติมที่ถูกนำออกจากโปรเซส stdio app-server ที่สร้างขึ้น หลังจาก OpenClaw สร้างสภาพแวดล้อมที่สืบทอดมาแล้ว `CODEX_HOME` และ `HOME` ถูกสงวนไว้สำหรับการแยก Codex ต่อเอเจนต์ของ OpenClaw ในการเรียกใช้งานแบบโลคัล |
| `requestTimeoutMs`  | `60000`                                  | เวลาหมดอายุสำหรับการเรียก control-plane ของ app-server                                                                                                                                                                                          |
| `mode`              | `"yolo"`                                 | พรีเซ็ตสำหรับการดำเนินการแบบ YOLO หรือแบบผ่านการตรวจสอบโดย guardian                                                                                                                                                                                      |
| `approvalPolicy`    | `"never"`                                | นโยบายการอนุมัติ native ของ Codex ที่ส่งไปยังการเริ่ม/ดำเนินการต่อ/เทิร์นของเธรด                                                                                                                                                                       |
| `sandbox`           | `"danger-full-access"`                   | โหมด sandbox native ของ Codex ที่ส่งไปยังการเริ่ม/ดำเนินการต่อของเธรด                                                                                                                                                                               |
| `approvalsReviewer` | `"user"`                                 | ใช้ `"auto_review"` เพื่อให้ Codex ตรวจสอบพรอมต์การอนุมัติ native `guardian_subagent` ยังคงเป็น alias เดิม                                                                                                                         |
| `serviceTier`       | unset                                    | service tier ของ Codex app-server แบบไม่บังคับ: `"fast"`, `"flex"` หรือ `null` ค่า legacy ที่ไม่ถูกต้องจะถูกละเว้น                                                                                                                            |

การเรียกเครื่องมือแบบไดนามิกที่ OpenClaw เป็นเจ้าของมีขอบเขตแยกจาก
`appServer.requestTimeoutMs`: คำขอ Codex `item/tool/call` แต่ละรายการต้องได้รับ
การตอบกลับจาก OpenClaw ภายใน 30 วินาที เมื่อหมดเวลา OpenClaw จะยกเลิกสัญญาณ
ของเครื่องมือในจุดที่รองรับ และส่งการตอบกลับเครื่องมือแบบไดนามิกที่ล้มเหลวกลับไปยัง Codex เพื่อให้
เทิร์นดำเนินต่อได้ แทนที่จะปล่อยให้เซสชันค้างอยู่ใน `processing`

หลังจาก OpenClaw ตอบกลับคำขอ app-server ที่อยู่ในขอบเขตของเทิร์น Codex แล้ว harness
ยังคาดหวังให้ Codex จบเทิร์น native ด้วย `turn/completed` ด้วย หาก
app-server เงียบไป 60 วินาทีหลังจากการตอบกลับนั้น OpenClaw จะพยายาม
ขัดจังหวะเทิร์นของ Codex อย่างดีที่สุด บันทึกการหมดเวลาเพื่อการวินิจฉัย และปล่อย
lane ของเซสชัน OpenClaw เพื่อไม่ให้ข้อความแชตถัดไปถูกจัดคิวอยู่หลังเทิร์น native
ที่ค้างอยู่

การ override ด้วยสภาพแวดล้อมยังคงพร้อมใช้งานสำหรับการทดสอบแบบโลคัล:

- `OPENCLAW_CODEX_APP_SERVER_BIN`
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

`OPENCLAW_CODEX_APP_SERVER_BIN` ข้ามไบนารีที่จัดการให้เมื่อ
`appServer.command` ไม่ได้ตั้งค่าไว้

`OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1` ถูกนำออกแล้ว ให้ใช้
`plugins.entries.codex.config.appServer.mode: "guardian"` แทน หรือ
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` สำหรับการทดสอบแบบโลคัลครั้งเดียว แนะนำให้ใช้ config
สำหรับการติดตั้งใช้งานที่ทำซ้ำได้ เพราะทำให้พฤติกรรมของ Plugin อยู่ในไฟล์ที่ผ่านการตรวจสอบเดียวกัน
กับการตั้งค่า Codex harness ส่วนที่เหลือ

## การใช้คอมพิวเตอร์

การใช้คอมพิวเตอร์มีคู่มือการตั้งค่าแยกของตัวเอง:
[การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use)

ฉบับย่อ: OpenClaw ไม่ได้รวมแอปควบคุมเดสก์ท็อปไว้ภายในหรือดำเนินการ
กับเดสก์ท็อปเอง OpenClaw เตรียม Codex app-server ตรวจสอบว่า
เซิร์ฟเวอร์ MCP `computer-use` พร้อมใช้งาน แล้วปล่อยให้ Codex จัดการการเรียก
เครื่องมือ MCP native ระหว่างเทิร์นโหมด Codex

สำหรับการเข้าถึงไดรเวอร์ TryCua โดยตรงนอกโฟลว์ marketplace ของ Codex ให้ลงทะเบียน
`cua-driver mcp` ด้วย `openclaw mcp set cua-driver '{"command":"cua-driver","args":["mcp"]}'`
ดู [การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use) สำหรับความแตกต่าง
ระหว่างการใช้คอมพิวเตอร์ที่ Codex เป็นเจ้าของกับการลงทะเบียน MCP โดยตรง

config ขั้นต่ำ:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          computerUse: {
            autoInstall: true,
          },
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
        fallback: "none",
      },
    },
  },
}
```

สามารถตรวจสอบหรือติดตั้งการตั้งค่าได้จากพื้นผิวคำสั่ง:

- `/codex computer-use status`
- `/codex computer-use install`
- `/codex computer-use install --source <marketplace-source>`
- `/codex computer-use install --marketplace-path <path>`

การใช้คอมพิวเตอร์ใช้ได้เฉพาะ macOS และอาจต้องการสิทธิ์ OS แบบโลคัลก่อนที่
เซิร์ฟเวอร์ MCP ของ Codex จะควบคุมแอปได้ หาก `computerUse.enabled` เป็น true และเซิร์ฟเวอร์ MCP
ไม่พร้อมใช้งาน เทิร์นโหมด Codex จะล้มเหลวก่อนเริ่มเธรด แทนที่จะ
ทำงานแบบเงียบๆ โดยไม่มีเครื่องมือ native Computer Use ดู
[การใช้คอมพิวเตอร์ของ Codex](/th/plugins/codex-computer-use) สำหรับตัวเลือก marketplace,
ข้อจำกัดของแค็ตตาล็อกจากระยะไกล เหตุผลของสถานะ และการแก้ไขปัญหา

เมื่อ `computerUse.autoInstall` เป็น true OpenClaw สามารถลงทะเบียน
marketplace มาตรฐาน Codex Desktop ที่ bundled มาจาก
`/Applications/Codex.app/Contents/Resources/plugins/openai-bundled` หาก Codex
ยังไม่พบ marketplace แบบโลคัล ใช้ `/new` หรือ `/reset` หลังจาก
เปลี่ยน runtime หรือ config การใช้คอมพิวเตอร์ เพื่อไม่ให้เซสชันที่มีอยู่ยังคง
ผูกกับ PI หรือเธรด Codex เก่า

## สูตรที่ใช้บ่อย

Codex แบบโลคัลพร้อม stdio transport เริ่มต้น:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

การตรวจสอบ Codex-only harness:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

การอนุมัติ Codex ที่ผ่านการตรวจสอบโดย guardian:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            mode: "guardian",
            approvalPolicy: "on-request",
            approvalsReviewer: "auto_review",
            sandbox: "workspace-write",
          },
        },
      },
    },
  },
}
```

app-server จากระยะไกลพร้อมเฮดเดอร์ที่ระบุชัดเจน:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            headers: {
              "X-OpenClaw-Agent": "main",
            },
          },
        },
      },
    },
  },
}
```

การสลับโมเดลยังคงควบคุมโดย OpenClaw เมื่อเซสชัน OpenClaw แนบอยู่
กับเธรด Codex ที่มีอยู่ เทิร์นถัดไปจะส่งโมเดล OpenAI
ผู้ให้บริการ นโยบายการอนุมัติ sandbox และ service tier ที่เลือกอยู่ในปัจจุบันไปยัง
app-server อีกครั้ง การสลับจาก `openai/gpt-5.5` เป็น `openai/gpt-5.2` จะคง
การผูกเธรดไว้ แต่ขอให้ Codex ดำเนินการต่อด้วยโมเดลที่เลือกใหม่

## คำสั่ง Codex

Plugin ที่ bundled มาลงทะเบียน `/codex` เป็นคำสั่ง slash ที่ได้รับอนุญาต คำสั่งนี้เป็น

รูปแบบที่ใช้บ่อย:

- `/codex threads [filter]` แสดงรายการเธรด Codex ล่าสุด

- `/codex compact` ขอให้ Codex app-server compact เธรดที่แนบอยู่

### เวิร์กโฟลว์การดีบักที่ใช้บ่อย

เมื่อเอเจนต์ที่รองรับด้วย Codex ทำสิ่งที่น่าประหลาดใจใน Telegram, Discord, Slack,
หรือช่องทางอื่น ให้เริ่มจากการสนทนาที่เกิดปัญหา:

1. เรียกใช้ `/diagnostics bad tool choice after image upload` หรือโน้ตสั้นอื่น
   ที่อธิบายสิ่งที่คุณเห็น
2. อนุมัติคำขอวินิจฉัยหนึ่งครั้ง การอนุมัติจะสร้างไฟล์ zip การวินิจฉัยของ Gateway

   ส่งบันเดิลข้อเสนอแนะ Codex ที่เกี่ยวข้องไปยังเซิร์ฟเวอร์ OpenAI ด้วย
3. คัดลอกคำตอบการวินิจฉัยที่เสร็จแล้วลงในรายงานบั๊กหรือเธรดสนับสนุน
   คำตอบนั้นมีพาธบันเดิลในเครื่อง สรุปความเป็นส่วนตัว รหัสเซสชัน OpenClaw
   รหัสเธรด Codex และบรรทัด `Inspect locally` สำหรับแต่ละเธรด Codex
4. หากคุณต้องการดีบักการรันด้วยตัวเอง ให้รันคำสั่ง `Inspect locally`
   ที่พิมพ์ออกมาในเทอร์มินัล คำสั่งจะมีลักษณะเหมือน `codex resume <thread-id>` และเปิด
   เธรด Codex ดั้งเดิม เพื่อให้คุณตรวจสอบการสนทนา ทำต่อในเครื่อง
   หรือถาม Codex ว่าเหตุใดจึงเลือกเครื่องมือหรือแผนนั้น

ใช้ `/codex diagnostics [note]` เฉพาะเมื่อคุณต้องการอัปโหลดข้อเสนอแนะ Codex
สำหรับเธรดที่แนบอยู่ในปัจจุบันโดยเฉพาะ โดยไม่มีบันเดิลการวินิจฉัย OpenClaw
Gateway แบบเต็ม สำหรับรายงานสนับสนุนส่วนใหญ่ `/diagnostics [note]` เป็น
จุดเริ่มต้นที่ดีกว่า เพราะผูกสถานะ Gateway ในเครื่องกับรหัสเธรด Codex
ไว้ด้วยกันในคำตอบเดียว ดู [การส่งออกการวินิจฉัย](/th/gateway/diagnostics)
สำหรับโมเดลความเป็นส่วนตัวและพฤติกรรมในแชตกลุ่มแบบเต็ม

แกนหลักของ OpenClaw ยังเปิดให้เจ้าของเท่านั้นใช้ `/diagnostics [note]` เป็นคำสั่ง
การวินิจฉัย Gateway ทั่วไป พรอมป์อนุมัติของคำสั่งจะแสดงคำนำเรื่องข้อมูลละเอียดอ่อน
ลิงก์ไปยัง [การส่งออกการวินิจฉัย](/th/gateway/diagnostics) และขอ
`openclaw gateway diagnostics export --json` ผ่านการอนุมัติ exec อย่างชัดเจน
ทุกครั้ง อย่าอนุมัติการวินิจฉัยด้วยกฎ allow-all หลังอนุมัติแล้ว
OpenClaw จะส่งรายงานที่วางต่อได้พร้อมพาธบันเดิลในเครื่องและสรุป manifest
เมื่อเซสชัน OpenClaw ที่ใช้งานอยู่กำลังใช้ harness ของ Codex การอนุมัติ
เดียวกันนั้นยังอนุญาตให้ส่งบันเดิลข้อเสนอแนะ Codex ที่เกี่ยวข้องไปยัง
เซิร์ฟเวอร์ OpenAI ด้วย พรอมป์อนุมัติจะบอกว่าข้อเสนอแนะ Codex จะถูกส่ง แต่
จะไม่แสดงรหัสเซสชันหรือรหัสเธรด Codex ก่อนการอนุมัติ

หากเจ้าของเรียกใช้ `/diagnostics` ในแชตกลุ่ม OpenClaw จะรักษาช่องทางร่วม
ให้สะอาด: กลุ่มจะได้รับเพียงประกาศสั้นๆ ขณะที่คำนำการวินิจฉัย พรอมป์อนุมัติ
และรหัสเซสชัน/เธรด Codex จะถูกส่งถึงเจ้าของผ่านเส้นทางอนุมัติส่วนตัว
หากไม่มีเส้นทางเจ้าของแบบส่วนตัว OpenClaw จะปฏิเสธคำขอจากกลุ่มและขอให้เจ้าของ
รันจาก DM

การอัปโหลด Codex ที่ได้รับอนุมัติจะเรียก Codex app-server `feedback/upload` และขอให้
app-server รวมล็อกสำหรับแต่ละเธรดที่ระบุและเธรดย่อย Codex ที่ถูกสร้างขึ้น
เมื่อมีให้ใช้งาน การอัปโหลดจะผ่านเส้นทางข้อเสนอแนะปกติของ Codex ไปยัง
เซิร์ฟเวอร์ OpenAI หากข้อเสนอแนะ Codex ถูกปิดใช้งานใน app-server นั้น คำสั่งจะส่งคืน
ข้อผิดพลาดของ app-server คำตอบการวินิจฉัยที่เสร็จแล้วจะแสดงรายการช่องทาง
รหัสเซสชัน OpenClaw รหัสเธรด Codex และคำสั่ง `codex resume <thread-id>`
ในเครื่องสำหรับเธรดที่ถูกส่ง หากคุณปฏิเสธหรือเพิกเฉยการอนุมัติ
OpenClaw จะไม่พิมพ์รหัส Codex เหล่านั้น การอัปโหลดนี้ไม่ได้แทนที่การส่งออก
การวินิจฉัย Gateway ในเครื่อง

`/codex resume` เขียนไฟล์ binding ข้างเคียงไฟล์เดียวกับที่ harness ใช้สำหรับ
turn ปกติ ในข้อความถัดไป OpenClaw จะ resume เธรด Codex นั้น ส่งโมเดล OpenClaw
ที่เลือกอยู่ในปัจจุบันเข้า app-server และเปิดใช้ประวัติแบบขยายต่อไป

### ตรวจสอบเธรด Codex จาก CLI

วิธีที่เร็วที่สุดในการทำความเข้าใจการรัน Codex ที่ผิดพลาดมักเป็นการเปิดเธรด Codex
ดั้งเดิมโดยตรง:

```sh
codex resume <thread-id>
```

ใช้สิ่งนี้เมื่อคุณสังเกตเห็นบั๊กในการสนทนาช่องทางและต้องการตรวจสอบ
เซสชัน Codex ที่มีปัญหา ทำต่อในเครื่อง หรือถาม Codex ว่าเหตุใดจึงเลือก
เครื่องมือหรือแนวทางการให้เหตุผลนั้น เส้นทางที่ง่ายที่สุดมักเป็นการรัน
`/diagnostics [note]` ก่อน: หลังจากคุณอนุมัติ รายงานที่เสร็จแล้วจะแสดงรายการ
แต่ละเธรด Codex และพิมพ์คำสั่ง `Inspect locally` เช่น
`codex resume <thread-id>` คุณสามารถคัดลอกคำสั่งนั้นลงในเทอร์มินัลได้โดยตรง

คุณยังสามารถรับรหัสเธรดจาก `/codex binding` สำหรับแชตปัจจุบัน หรือ
`/codex threads [filter]` สำหรับเธรด Codex app-server ล่าสุด แล้วรันคำสั่ง
`codex resume` เดียวกันในเชลล์ของคุณ

พื้นผิวคำสั่งต้องใช้ Codex app-server `0.125.0` หรือใหม่กว่า เมธอดควบคุม
แต่ละรายการจะถูกรายงานเป็น `unsupported by this Codex app-server` หาก
app-server ในอนาคตหรือแบบกำหนดเองไม่ได้เปิดเผยเมธอด JSON-RPC นั้น

## ขอบเขตฮุก

harness ของ Codex มีเลเยอร์ฮุกสามชั้น:

| เลเยอร์                                 | เจ้าของ                    | วัตถุประสงค์                                                             |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| ฮุก Plugin ของ OpenClaw                 | OpenClaw                 | ความเข้ากันได้ของผลิตภัณฑ์/Plugin ระหว่าง harness ของ PI และ Codex         |
| มิดเดิลแวร์ส่วนขยาย Codex app-server | Plugin ที่ bundled มากับ OpenClaw | พฤติกรรมอะแดปเตอร์ต่อ turn รอบเครื่องมือไดนามิกของ OpenClaw            |
| ฮุกดั้งเดิมของ Codex                    | Codex                    | ไลฟ์ไซเคิล Codex ระดับต่ำและนโยบายเครื่องมือดั้งเดิมจากการกำหนดค่า Codex |

OpenClaw ไม่ใช้ไฟล์ `hooks.json` ของโปรเจกต์หรือระดับ global ของ Codex เพื่อกำหนดเส้นทาง
พฤติกรรม Plugin ของ OpenClaw สำหรับบริดจ์เครื่องมือดั้งเดิมและสิทธิ์ที่รองรับ
OpenClaw ฉีดการกำหนดค่า Codex ต่อเธรดสำหรับ `PreToolUse`, `PostToolUse`,
`PermissionRequest` และ `Stop` ฮุก Codex อื่นๆ เช่น `SessionStart` และ
`UserPromptSubmit` ยังคงเป็นการควบคุมระดับ Codex; ฮุกเหล่านี้ไม่ได้ถูกเปิดเผยเป็น
ฮุก Plugin ของ OpenClaw ในสัญญา v1

สำหรับเครื่องมือไดนามิกของ OpenClaw, OpenClaw จะดำเนินการเครื่องมือหลังจาก Codex ขอ
การเรียก ดังนั้น OpenClaw จึงเรียกใช้พฤติกรรม Plugin และมิดเดิลแวร์ที่ตนเป็นเจ้าของใน
อะแดปเตอร์ harness สำหรับเครื่องมือดั้งเดิมของ Codex, Codex เป็นเจ้าของบันทึกเครื่องมือ
แบบ canonical OpenClaw สามารถ mirror เหตุการณ์ที่เลือกได้ แต่ไม่สามารถเขียนเธรด Codex
ดั้งเดิมใหม่ เว้นแต่ Codex จะเปิดเผยการดำเนินการนั้นผ่าน app-server หรือ callback
ของฮุกดั้งเดิม

การฉายภาพ Compaction และไลฟ์ไซเคิล LLM มาจากการแจ้งเตือนของ Codex app-server
และสถานะอะแดปเตอร์ OpenClaw ไม่ใช่คำสั่งฮุกดั้งเดิมของ Codex เหตุการณ์
`before_compaction`, `after_compaction`, `llm_input` และ
`llm_output` ของ OpenClaw เป็นการสังเกตการณ์ระดับอะแดปเตอร์ ไม่ใช่การจับ payload
คำขอภายในหรือ Compaction ของ Codex แบบตรงทุกไบต์

การแจ้งเตือน app-server ของฮุกดั้งเดิม Codex `hook/started` และ `hook/completed`
ถูกฉายเป็นเหตุการณ์ agent `codex_app_server.hook` สำหรับ trajectory และการดีบัก
การแจ้งเตือนเหล่านี้ไม่เรียกฮุก Plugin ของ OpenClaw

## เครื่องมือ สื่อ และ Compaction

ฮาร์เนส Codex เปลี่ยนเฉพาะตัวดำเนินการเอเจนต์แบบฝังระดับล่างเท่านั้น

OpenClaw ยังคงสร้างรายการเครื่องมือและรับผลลัพธ์เครื่องมือแบบไดนามิกจาก
ฮาร์เนส ข้อความ รูปภาพ วิดีโอ เพลง TTS การอนุมัติ และเอาต์พุตเครื่องมือส่งข้อความ
ยังคงผ่านเส้นทางการส่งมอบ OpenClaw ตามปกติ

รีเลย์ฮุคเนทีฟตั้งใจให้เป็นแบบทั่วไป แต่สัญญาการรองรับ v1
จำกัดอยู่ที่เส้นทางเครื่องมือและสิทธิ์เนทีฟของ Codex ที่ OpenClaw ทดสอบ ใน
รันไทม์ Codex นั่นรวมถึงเพย์โหลด shell, patch และ MCP `PreToolUse`,
`PostToolUse` และ `PermissionRequest` อย่าสันนิษฐานว่าเหตุการณ์ฮุคของ
Codex ในอนาคตทุกอย่างเป็นพื้นผิว Plugin ของ OpenClaw จนกว่าสัญญารันไทม์จะระบุ
ชื่อนั้น

สำหรับ `PermissionRequest` OpenClaw จะส่งคืนการตัดสินใจอนุญาตหรือปฏิเสธอย่างชัดเจน
เฉพาะเมื่อนโยบายตัดสินแล้ว ผลลัพธ์แบบไม่มีการตัดสินใจไม่ใช่การอนุญาต Codex ถือว่านั่นเป็น
ไม่มีการตัดสินใจจากฮุค และปล่อยผ่านไปยัง guardian ของตนเองหรือเส้นทางการอนุมัติจากผู้ใช้

การขออนุมัติเครื่องมือ MCP ของ Codex จะถูกส่งผ่านโฟลว์การอนุมัติ Plugin ของ OpenClaw
เมื่อ Codex ทำเครื่องหมาย `_meta.codex_approval_kind` เป็น
`"mcp_tool_call"` พรอมป์ `request_user_input` ของ Codex จะถูกส่งกลับไปยัง
แชทต้นทาง และข้อความติดตามผลถัดไปที่อยู่ในคิวจะตอบคำขอเซิร์ฟเวอร์เนทีฟนั้น
แทนที่จะถูกนำทางเป็นบริบทเพิ่มเติม คำขอเรียกข้อมูล MCP อื่นยังคงล้มเหลวแบบปิด

การนำทางคิวขณะรันอยู่แมปกับ `turn/steer` ของเซิร์ฟเวอร์แอป Codex ด้วย
ค่าเริ่มต้น `messages.queue.mode: "steer"` OpenClaw จะรวมข้อความแชทที่อยู่ในคิว
ตามหน้าต่างเวลาสงบที่กำหนดค่าไว้ และส่งเป็นคำขอ `turn/steer` เดียว
ตามลำดับที่มาถึง โหมด `queue` เดิมจะส่งคำขอ `turn/steer` แยกกัน Codex
อาจปฏิเสธการนำทางในเทิร์นเดียวกันสำหรับเทิร์นรีวิวและ Compaction แบบแมนนวล ซึ่งในกรณีนั้น
OpenClaw จะใช้คิวติดตามผลเมื่อโหมดที่เลือกอนุญาตให้ fallback ดู
[คิวการนำทาง](/th/concepts/queue-steering)

เมื่อโมเดลที่เลือกใช้ฮาร์เนส Codex, Compaction ของเธรดเนทีฟจะถูกมอบหมายให้
เซิร์ฟเวอร์แอป Codex OpenClaw เก็บมิเรอร์ทรานสคริปต์ไว้สำหรับประวัติช่องทาง,
การค้นหา, `/new`, `/reset` และการสลับโมเดลหรือฮาร์เนสในอนาคต
มิเรอร์รวมพรอมป์ผู้ใช้ ข้อความผู้ช่วยสุดท้าย และเรคคอร์ดการให้เหตุผลหรือแผนแบบเบาของ Codex
เมื่อเซิร์ฟเวอร์แอปปล่อยออกมา วันนี้ OpenClaw บันทึกเฉพาะสัญญาณเริ่มต้นและเสร็จสิ้น
ของ Compaction เนทีฟเท่านั้น ยังไม่ได้เปิดเผยสรุป Compaction ที่มนุษย์อ่านได้
หรือรายการที่ตรวจสอบย้อนหลังได้ว่า Codex เก็บรายการใดไว้หลัง Compaction

เนื่องจาก Codex เป็นเจ้าของเธรดเนทีฟที่เป็นแหล่งจริง `tool_result_persist` จึงไม่ได้
เขียนเรคคอร์ดผลลัพธ์เครื่องมือเนทีฟของ Codex ใหม่ในปัจจุบัน มันมีผลเฉพาะเมื่อ
OpenClaw กำลังเขียนผลลัพธ์เครื่องมือทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ

การสร้างสื่อไม่ต้องใช้ PI รูปภาพ วิดีโอ เพลง PDF, TTS และการทำความเข้าใจสื่อ
ยังคงใช้การตั้งค่าผู้ให้บริการ/โมเดลที่ตรงกัน เช่น
`agents.defaults.imageGenerationModel`, `videoGenerationModel`, `pdfModel` และ
`messages.tts`

## การแก้ไขปัญหา

**Codex ไม่ปรากฏเป็นผู้ให้บริการ `/model` ปกติ:** นั่นเป็นสิ่งที่คาดไว้สำหรับ
การกำหนดค่าใหม่ เลือกโมเดล `openai/gpt-*` พร้อม
`agentRuntime.id: "codex"` (หรือ ref `codex/*` เดิม), เปิดใช้
`plugins.entries.codex.enabled` และตรวจสอบว่า `plugins.allow` กีดกัน
`codex` หรือไม่

**OpenClaw ใช้ PI แทน Codex:** `agentRuntime.id: "auto"` ยังสามารถใช้ PI เป็น
แบ็กเอนด์ความเข้ากันได้เมื่อไม่มีฮาร์เนส Codex รับรัน ตั้งค่า
`agentRuntime.id: "codex"` เพื่อบังคับเลือก Codex ขณะทดสอบ
รันไทม์ Codex ที่ถูกบังคับตอนนี้จะล้มเหลวแทนที่จะ fallback ไปยัง PI เว้นแต่คุณจะ
ตั้งค่า `agentRuntime.fallback: "pi"` อย่างชัดเจน เมื่อเลือกเซิร์ฟเวอร์แอป Codex แล้ว
ความล้มเหลวของมันจะแสดงโดยตรงโดยไม่มีการกำหนดค่า fallback เพิ่มเติม

**เซิร์ฟเวอร์แอปถูกปฏิเสธ:** อัปเกรด Codex เพื่อให้แฮนด์เชกเซิร์ฟเวอร์แอป
รายงานเวอร์ชัน `0.125.0` หรือใหม่กว่า พรีรีลีสเวอร์ชันเดียวกันหรือเวอร์ชันที่มีส่วนต่อท้ายบิลด์
เช่น `0.125.0-alpha.2` หรือ `0.125.0+custom` จะถูกปฏิเสธ เพราะ
โปรโตคอลขั้นต่ำของเวอร์ชันเสถียร `0.125.0` คือสิ่งที่ OpenClaw ทดสอบ

**การค้นพบโมเดลช้า:** ลดค่า `plugins.entries.codex.config.discovery.timeoutMs`
หรือปิดใช้การค้นพบ

**ทรานสปอร์ต WebSocket ล้มเหลวทันที:** ตรวจสอบ `appServer.url`, `authToken`,
และตรวจสอบว่าเซิร์ฟเวอร์แอประยะไกลพูดเวอร์ชันโปรโตคอลเซิร์ฟเวอร์แอป Codex เดียวกัน

**โมเดลที่ไม่ใช่ Codex ใช้ PI:** นั่นเป็นสิ่งที่คาดไว้ เว้นแต่คุณจะบังคับ
`agentRuntime.id: "codex"` สำหรับเอเจนต์นั้น หรือเลือก ref `codex/*`
เดิม ref `openai/gpt-*` ธรรมดาและ ref ผู้ให้บริการอื่นยังคงอยู่บน
เส้นทางผู้ให้บริการปกติในโหมด `auto` หากคุณบังคับ `agentRuntime.id: "codex"` ทุกเทิร์นแบบฝัง
สำหรับเอเจนต์นั้นต้องเป็นโมเดล OpenAI ที่ Codex รองรับ

**ติดตั้ง Computer Use แล้ว แต่เครื่องมือไม่ทำงาน:** ตรวจสอบ
`/codex computer-use status` จากเซสชันใหม่ หากเครื่องมือรายงานว่า
`Native hook relay unavailable` ให้ใช้ `/new` หรือ `/reset`; หากยังคงอยู่ ให้รีสตาร์ท
gateway เพื่อล้างการลงทะเบียนฮุคเนทีฟที่ค้างอยู่ หาก `computer-use.list_apps`
หมดเวลา ให้รีสตาร์ท Codex Computer Use หรือ Codex Desktop แล้วลองอีกครั้ง
