**เอเจนต์ฮาร์เนส** คือตัวดำเนินการระดับล่างสำหรับหนึ่งรอบการทำงานของเอเจนต์ OpenClaw ที่เตรียมไว้แล้ว ไม่ใช่ผู้ให้บริการโมเดล ไม่ใช่ช่องทาง และไม่ใช่รีจิสทรีเครื่องมือ สำหรับแนวคิดเชิงภาพรวมที่นำเสนอต่อผู้ใช้ โปรดดู [รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes)

ใช้พื้นผิวนี้เฉพาะกับ Plugin แบบเนทีฟที่มาพร้อมระบบหรือเชื่อถือได้เท่านั้น สัญญานี้ยังอยู่ในขั้นทดลอง เนื่องจากชนิดพารามิเตอร์ตั้งใจให้สอดคล้องกับตัวรันแบบฝังตัวในปัจจุบัน

## เมื่อใดควรใช้ฮาร์เนส

ลงทะเบียนเอเจนต์ฮาร์เนสเมื่อกลุ่มโมเดลมีรันไทม์เซสชันแบบเนทีฟของตนเอง และการขนส่งผ่านผู้ให้บริการตามปกติของ OpenClaw ไม่ใช่นามธรรมที่เหมาะสม:

- เซิร์ฟเวอร์เอเจนต์เขียนโค้ดแบบเนทีฟที่จัดการเธรดและ Compaction
- CLI หรือดีมอนภายในเครื่องที่ต้องสตรีมเหตุการณ์แผน/การให้เหตุผล/เครื่องมือแบบเนทีฟ
- รันไทม์โมเดลที่ต้องใช้รหัสสำหรับทำงานต่อของตนเองเพิ่มเติมจากทรานสคริปต์เซสชันของ OpenClaw

**อย่า** ลงทะเบียนฮาร์เนสเพียงเพื่อเพิ่ม API ของ LLM ใหม่ สำหรับ API โมเดลแบบ HTTP หรือ WebSocket ทั่วไป ให้สร้าง [Plugin ผู้ให้บริการ](/th/plugins/sdk-provider-plugins)

## สิ่งที่คอร์ยังคงเป็นเจ้าของ

ก่อนเลือกฮาร์เนส OpenClaw ได้กำหนดสิ่งต่อไปนี้แล้ว:

- ผู้ให้บริการและโมเดล
- สถานะการยืนยันตัวตนของรันไทม์ เว้นแต่ฮาร์เนสจะประกาศว่าเป็นเจ้าของการเริ่มต้นการยืนยันตัวตน
- ระดับการคิดและงบประมาณบริบท
- ไฟล์ทรานสคริปต์/เซสชันของ OpenClaw
- พื้นที่ทำงาน แซนด์บ็อกซ์ และนโยบายเครื่องมือ
- คอลแบ็กการตอบกลับของช่องทางและคอลแบ็กการสตรีม
- นโยบายการใช้โมเดลสำรองและการสลับโมเดลแบบสด

ฮาร์เนสเรียกใช้ความพยายามที่เตรียมไว้แล้ว โดยจะไม่เลือกผู้ให้บริการ แทนที่การส่งผ่านช่องทาง หรือสลับโมเดลโดยไม่แจ้งให้ทราบ

### การเริ่มต้นการยืนยันตัวตนที่ฮาร์เนสเป็นเจ้าของ

โดยค่าเริ่มต้น คอร์จะกำหนดข้อมูลประจำตัวของผู้ให้บริการก่อนเรียกฮาร์เนส ฮาร์เนสที่เชื่อถือได้และสามารถยืนยันตัวตนผ่านรันไทม์แบบเนทีฟของตนเองอาจตั้งค่า
`authBootstrap: "harness"` ในการลงทะเบียน `AgentHarness` แบบคงที่ จากนั้นคอร์จะข้ามการเริ่มต้นข้อมูลประจำตัวทั่วไปของผู้ให้บริการและความล้มเหลวจากการไม่มีข้อมูลประจำตัว สำหรับทุกความพยายามที่ฮาร์เนสนั้นรับผิดชอบ

คอร์ยังคงส่งต่อโปรไฟล์การยืนยันตัวตน OpenClaw ที่เข้ากันได้และถูกเลือกหรือจัดลำดับไว้อย่างชัดเจน รวมถึงพื้นที่จัดเก็บที่จำกัดขอบเขตของโปรไฟล์นั้น เมื่อมีอยู่ ฮาร์เนสต้องกำหนดโปรไฟล์นั้นหรือข้อมูลประจำตัวแบบเนทีฟก่อนส่งคำขอโมเดล จำกัดขอบเขตความลับไว้เฉพาะความพยายามนั้น และแสดงข้อผิดพลาดการยืนยันตัวตนที่นำไปแก้ไขได้ อย่าตั้งค่าความสามารถนี้ให้ฮาร์เนสที่เป็นเจ้าของการยืนยันตัวตนเพียงบางครั้งเท่านั้น

### อาร์ติแฟกต์รันไทม์สำหรับการตั้งค่าที่ผ่านการตรวจสอบ

ฮาร์เนสภายในเครื่องที่สามารถจัดเตรียมการอนุมานสำหรับการตั้งค่าครั้งแรกต้องรับรองอิมพลีเมนเทชันที่ทำโพรบเสร็จสมบูรณ์ เมื่อ
`params.captureRuntimeArtifact` เป็น true ให้ส่งคืน
`result.runtimeArtifact` แบบทึบที่มีรหัสเสถียรและลายนิ้วมือของเนื้อหา ลงทะเบียนความสามารถ `runtimeArtifact.validate(...)` ที่ตรงกัน ซึ่งตรวจสอบการผูกนั้นอีกครั้งโดยไม่โหลดฮาร์เนสอื่นหรือสแกน Plugin ที่ไม่เกี่ยวข้อง

การทำงานต่อของ OpenClaw ที่ผ่านการตรวจสอบจะส่ง `params.expectedRuntimeArtifact` ด้วย
ฮาร์เนสต้องเปรียบเทียบค่านี้กับโปรเซสแบบเนทีฟที่ได้รับมาจริง และต้องล้มเหลวก่อนเริ่มหรือทำงานต่อในเธรดแบบเนทีฟหากค่าไม่ตรงกัน รอบการทำงานทั่วไปของเอเจนต์จะละเว้นทั้งสองฟิลด์ เพื่อไม่ให้การแฮชเนื้อหาอยู่ในเส้นทางด่วนของคำขอปกติ ฮาร์เนสระยะไกล/WebSocket ต้องมีสัญญาการรับรองจากเซิร์ฟเวอร์ก่อนจึงจะเข้าร่วมได้ สตริงเวอร์ชันเพียงอย่างเดียวไม่ใช่อัตลักษณ์ของอาร์ติแฟกต์

ความพยายามที่เตรียมไว้ยังมี `params.runtimePlan` ซึ่งเป็นชุดนโยบายที่ OpenClaw เป็นเจ้าของสำหรับการตัดสินใจของรันไทม์ที่ต้องใช้ร่วมกันระหว่าง OpenClaw และฮาร์เนสแบบเนทีฟ:

- `runtimePlan.tools.normalize(...)` และ `runtimePlan.tools.logDiagnostics(...)`
  สำหรับนโยบายสคีมาเครื่องมือที่รับรู้ผู้ให้บริการ
- `runtimePlan.transcript.resolvePolicy(...)` สำหรับการทำให้ทรานสคริปต์ปลอดภัยและ
  นโยบายซ่อมแซมการเรียกเครื่องมือ
- `runtimePlan.delivery.isSilentPayload(...)` สำหรับ `NO_REPLY` ที่ใช้ร่วมกันและการระงับ
  การส่งสื่อ
- `runtimePlan.outcome.classifyRunResult(...)` สำหรับการจำแนกประเภท
  การใช้โมเดลสำรอง
- `runtimePlan.observability` สำหรับเมทาดาทาผู้ให้บริการ/โมเดล/ฮาร์เนสที่กำหนดแล้ว

ฮาร์เนสอาจใช้แผนนี้สำหรับการตัดสินใจที่ต้องสอดคล้องกับพฤติกรรมของ OpenClaw แต่ให้ถือว่าเป็นสถานะความพยายามที่โฮสต์เป็นเจ้าของ อย่าแก้ไขหรือใช้แผนนี้เพื่อสลับผู้ให้บริการ/โมเดลภายในรอบการทำงาน

### สัญญาการขนส่งคำขอ

`supports(ctx)` รับการขนส่งโมเดลที่กำหนดแล้วใน `ctx.modelProvider`
ข้อเท็จจริงที่ไม่มีความลับและผู้ให้บริการเป็นเจ้าของสองรายการใช้อธิบายเส้นทางที่เลือก:

- `runtimePolicy.compatibleIds` แสดงรหัสรันไทม์ที่ผู้ให้บริการประกาศว่า
  เข้ากันได้กับเส้นทางเฉพาะนั้น การไม่มีนโยบายหมายความว่าผู้ให้บริการไม่ได้
  ประกาศความเข้ากันได้ระดับเส้นทาง ไม่ใช่การอนุญาตให้สันนิษฐานว่ารองรับ
- `requestTransportOverrides: "none"` หมายความว่าไม่ต้องสร้างการแทนที่คำขอ
  ผู้ให้บริการ/โมเดลที่กำหนดโดยผู้เขียนขึ้นใหม่ `"present"` หมายความว่ามีส่วนหัวที่กำหนดโดยผู้เขียน การขนส่ง
  การยืนยันตัวตน พร็อกซี TLS บริการภายในเครื่อง พฤติกรรมเครือข่ายส่วนตัว หรือพารามิเตอร์
  คำขอ ข้อเท็จจริงนี้ไม่เปิดเผยค่าเหล่านั้น

ส่งคืน `{ supported: false, reason }` เมื่อฮาร์เนสไม่สามารถสร้างการขนส่งที่เตรียมไว้ขึ้นใหม่
อย่าอนุมานการรองรับด้วยการอ่านการกำหนดค่าดิบหลังการเลือก เมื่อการเตรียมการยืนยันตัวตนให้เส้นทางสำหรับลองใหม่หลายเส้นทาง ฮาร์เนสหนึ่งรายการต้องรองรับทั้งหมดก่อนส่งงาน การเลือกโดยปริยายจะใช้ OpenClaw หากไม่มี Plugin ใดเป็นเจ้าของชุดทั้งหมดได้ ส่วนการเลือก Plugin อย่างชัดเจนหรือที่บันทึกไว้จะล้มเหลวแบบปิด

## ลงทะเบียนฮาร์เนส

**นำเข้า:** `openclaw/plugin-sdk/agent-harness`

```typescript
import type { AgentHarness } from "openclaw/plugin-sdk/agent-harness";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const myHarness: AgentHarness = {
  id: "my-harness",
  label: "ฮาร์เนสเอเจนต์แบบเนทีฟของฉัน",

  supports(ctx) {
    const routeSupportsHarness =
      ctx.modelProvider?.runtimePolicy?.compatibleIds.includes("my-harness") === true;
    const canReproduceRequest = ctx.modelProvider?.requestTransportOverrides !== "present";
    return ctx.provider === "my-provider" && routeSupportsHarness && canReproduceRequest
      ? { supported: true, priority: 100 }
      : { supported: false, reason: "เส้นทางที่มีผลไม่เข้ากันกับฮาร์เนส" };
  },

  async runAttempt(params) {
    // เริ่มหรือทำงานต่อในเธรดแบบเนทีฟของคุณ
    // ใช้ params.prompt, params.tools, params.images, params.onPartialReply,
    // params.onAgentEvent และฟิลด์อื่นของความพยายามที่เตรียมไว้
    return await runMyNativeTurn(params);
  },
};

export default definePluginEntry({
  id: "my-native-agent",
  name: "เอเจนต์แบบเนทีฟของฉัน",
  description: "เรียกใช้โมเดลที่เลือกผ่านดีมอนเอเจนต์แบบเนทีฟ",
  register(api) {
    api.registerAgentHarness(myHarness);
  },
});
```

ตัวอย่างทั่วไปนี้ตั้งใจละเว้น `authBootstrap` เพิ่ม
`authBootstrap: "harness"` เฉพาะเมื่อฮาร์เนสเป็นไปตามสัญญาข้างต้นเท่านั้น

### การดำเนินการที่มอบหมาย

เจ้าของฮาร์เนสอาจตั้งค่า `delegatedExecutionPluginIds` เป็นรหัสของ Plugin ที่เชื่อถือได้
ซึ่งต้องดำเนินการเซสชันที่มีอยู่และถูกล็อกกับโมเดล เช่น การขนส่งเสียงที่ดำเนินบทสนทนาซึ่งใช้ Codex ต่อไป นี่คือความยินยอมแบบคงที่จากเจ้าของ ไม่ใช่รายการอนุญาตของคอร์ โปรดจำกัดขอบเขตให้แคบ

ผู้รับมอบหมายจะได้รับเฉพาะการรับงานและการดำเนินการแบบฝังตัว OpenClaw กำหนดให้ใช้
คีย์เซสชัน เส้นทางพื้นที่จัดเก็บ และรหัสเซสชันที่บันทึกไว้ตรงกันทุกประการ; `modelSelectionLocked:
true`; และค่า `agentHarnessId` กับ `agentHarnessRuntimeOverride` ที่ตรงกัน
จากนั้นการเรียกใช้จะถูกจำกัดขอบเขตผ่านเจ้าของฮาร์เนส การสร้าง แพตช์
รีเซ็ต ลบ และเก็บถาวรเซสชัน รวมถึงการเปลี่ยนแปลง Gateway ยังคงทำได้เฉพาะเจ้าของ

## นโยบายการเลือก

OpenClaw เลือกฮาร์เนสหลังจากกำหนดผู้ให้บริการ/โมเดลแล้ว:

1. นโยบายรันไทม์ที่จำกัดขอบเขตตามโมเดลมีลำดับความสำคัญสูงสุด
2. นโยบายรันไทม์ที่จำกัดขอบเขตตามผู้ให้บริการมีลำดับถัดมา
3. `auto` สอบถามฮาร์เนสที่ลงทะเบียนไว้ว่ารองรับเส้นทางที่มีผลซึ่งกำหนดแล้วหรือไม่
   คำนำหน้าผู้ให้บริการ/โมเดลเพียงอย่างเดียวจะไม่เลือกฮาร์เนส
4. หากไม่มีฮาร์เนสที่ลงทะเบียนไว้ตรงกัน OpenClaw จะใช้รันไทม์แบบฝังตัว

ความล้มเหลวของฮาร์เนส Plugin จะแสดงเป็นความล้มเหลวของการเรียกใช้ ในโหมด `auto`
การใช้รันไทม์แบบฝังตัวสำรองจะใช้เฉพาะเมื่อไม่มีฮาร์เนส Plugin ที่ลงทะเบียนไว้รองรับผู้ให้บริการ/โมเดลที่กำหนดแล้ว เมื่อฮาร์เนส Plugin รับผิดชอบการเรียกใช้แล้ว OpenClaw จะไม่เล่นรอบการทำงานเดียวกันซ้ำผ่านรันไทม์อื่น เพราะอาจเปลี่ยนความหมายของการยืนยันตัวตน/รันไทม์หรือทำให้เกิดผลข้างเคียงซ้ำ

นโยบายรันไทม์ที่กำหนดค่าไว้ยังคงเป็นแหล่งข้อมูลหลักของรันไทม์ที่ต้องการ
`agentHarnessId` ของเซสชันที่บันทึกไว้จะรักษาความเป็นเจ้าของทรานสคริปต์แบบเนทีฟ
ขณะที่การเตรียมเส้นทาง/การยืนยันตัวตนยังอยู่ระหว่างดำเนินการ ทั้งสองอย่างไม่ได้ทำให้เส้นทางที่ไม่เข้ากันกลายเป็นเข้ากันได้ เมื่อมีข้อเท็จจริงที่เตรียมไว้แล้ว ฮาร์เนสที่เลือกหรือตรึงไว้ต้องรองรับข้อเท็จจริงเหล่านั้น มิฉะนั้นการเรียกใช้จะล้มเหลวแบบปิด `/status` แสดงรันไทม์ที่มีผล
ซึ่งเลือกจากนโยบาย ความเป็นเจ้าของที่บันทึกไว้ และการรองรับเส้นทาง
สถานะการเตรียมจะระบุไว้อย่างชัดเจน: หากไม่มี `runtimePolicy` จะยังคงถือว่าไม่ได้ประกาศ
แทนที่จะอนุมานจากฟิลด์การขนส่งใดก็ตามที่ปรากฏอยู่
เมื่อการยืนยันตัวตนที่ฮาร์เนสเป็นเจ้าของยังไม่สามารถตัดสินระหว่างเส้นทางกายภาพหลายเส้นทางได้
ข้อเท็จจริงการรองรับที่เตรียมไว้จะเป็นส่วนร่วมของรหัสรันไทม์ที่เข้ากันได้ และ
รายงานการแทนที่คำขอหากตัวเลือกใดมีการแทนที่ ดังนั้นตัวเลือกที่ไม่ได้ประกาศเพียงหนึ่งรายการ
จะทำให้ความเข้ากันได้แบบเนทีฟว่างเปล่า; `preparedAuth.source: "harness"`
คือเจ้าของการยืนยันตัวตน ไม่ใช่การอนุญาตให้อนุมานการรองรับเส้นทาง

หากฮาร์เนสที่เลือกไม่เป็นไปตามคาด ให้เปิดใช้การบันทึกดีบัก `agents/harness`
และตรวจสอบระเบียน `agent harness selected` แบบมีโครงสร้างของ Gateway ซึ่ง
ประกอบด้วยรหัสฮาร์เนสที่เลือก เหตุผลในการเลือก นโยบายรันไทม์/การใช้สำรอง
และในโหมด `auto` จะมีผลการรองรับของตัวเลือก Plugin แต่ละรายการ

Plugin Codex ที่มาพร้อมระบบลงทะเบียน `codex` เป็นรหัสฮาร์เนส คอร์ถือว่าค่านี้
เป็นรหัสฮาร์เนส Plugin ทั่วไป นามแฝงเฉพาะ Codex ต้องอยู่ใน Plugin
หรือการกำหนดค่าของผู้ปฏิบัติงาน ไม่ใช่ในตัวเลือกรันไทม์ที่ใช้ร่วมกัน

## การจับคู่ผู้ให้บริการกับฮาร์เนส

ฮาร์เนสส่วนใหญ่ควรลงทะเบียนผู้ให้บริการด้วย ผู้ให้บริการทำให้การอ้างอิงโมเดล
สถานะการยืนยันตัวตน เมทาดาทาโมเดล และการเลือก `/model` ปรากฏต่อส่วนอื่นของ
OpenClaw จากนั้นฮาร์เนสจะรับผิดชอบผู้ให้บริการนั้นใน `supports(...)`

Plugin Codex ที่มาพร้อมระบบใช้รูปแบบนี้:

- การอ้างอิงโมเดลที่แนะนำสำหรับผู้ใช้: `openai/gpt-5.6-sol`
- การอ้างอิงเพื่อความเข้ากันได้: ยังคงยอมรับการอ้างอิง `codex/gpt-*` แบบเดิม แต่การกำหนดค่าใหม่
  ไม่ควรใช้เป็นการอ้างอิงผู้ให้บริการ/โมเดลตามปกติ
- รหัสฮาร์เนส: `codex`
- การยืนยันตัวตน: ความพร้อมใช้งานของผู้ให้บริการแบบสังเคราะห์ เนื่องจากฮาร์เนส Codex เป็นเจ้าของ
  การเข้าสู่ระบบ/เซสชัน Codex แบบเนทีฟ
- คำขอ app-server: OpenClaw ส่งรหัสโมเดลเปล่าไปยัง Codex และปล่อยให้
  ฮาร์เนสสื่อสารกับโปรโตคอล app-server แบบเนทีฟ

Plugin Codex เป็นส่วนเพิ่มเติม เมื่อนโยบายรันไทม์ไม่ได้ตั้งค่าหรือเป็น `auto` OpenAI อาจ
เลือก Codex ได้เฉพาะเมื่อสัญญาเส้นทางที่ผู้ให้บริการเป็นเจ้าของประกาศว่าเข้ากันได้กับ `codex`:
เส้นทาง Platform Responses หรือ ChatGPT Responses อย่างเป็นทางการผ่าน HTTPS ที่ตรงกันทุกประการ
และไม่มีการแทนที่คำขอที่กำหนดโดยผู้เขียน คำนำหน้า `openai/*` เพียงอย่างเดียวจะไม่
เลือก Codex ปลายทางที่กำหนดเอง อะแดปเตอร์ Completions และพฤติกรรมคำขอที่กำหนดโดยผู้เขียน
จะยังคงใช้ OpenClaw ปลายทาง HTTP แบบข้อความธรรมดาที่เป็นทางการจะถูกปฏิเสธ การอ้างอิง `codex/gpt-*` รุ่นเก่า
ยังคงเป็นอินพุตเพื่อความเข้ากันได้ โปรดดู
[รันไทม์เอเจนต์โดยปริยายของ OpenAI](/th/providers/openai#implicit-agent-runtime)

สำหรับการตั้งค่าโดยผู้ปฏิบัติงาน ตัวอย่างคำนำหน้าโมเดล และการกำหนดค่าเฉพาะ Codex โปรดดู
[ฮาร์เนส Codex](/th/plugins/codex-harness)

Plugin Codex บังคับใช้เวอร์ชันขั้นต่ำของ app-server ที่ระบุไว้ใน
[ฮาร์เนส Codex](/th/plugins/codex-harness) โดยตรวจสอบแฮนด์เชกการเริ่มต้นและ
บล็อกเซิร์ฟเวอร์รุ่นเก่าหรือที่ไม่มีเวอร์ชัน เพื่อให้ OpenClaw ทำงานเฉพาะกับพื้นผิวโปรโตคอล
ที่ผ่านการทดสอบแล้ว

### มิดเดิลแวร์ผลลัพธ์เครื่องมือ

Plugin ที่มาพร้อมระบบและ Plugin ที่ติดตั้งซึ่งเปิดใช้อย่างชัดเจนและมีสัญญา
แมนิเฟสต์ตรงกัน สามารถแนบมิดเดิลแวร์ผลลัพธ์เครื่องมือที่เป็นกลางต่อรันไทม์ผ่าน
`api.registerAgentToolResultMiddleware(...)` เมื่อแมนิเฟสต์ประกาศ
รหัสรันไทม์เป้าหมายใน `contracts.agentToolResultMiddleware` จุดเชื่อมต่อที่เชื่อถือได้นี้
มีไว้สำหรับการแปลงผลลัพธ์เครื่องมือแบบอะซิงโครนัสที่ต้องทำงานก่อน OpenClaw หรือ
Codex ส่งเอาต์พุตเครื่องมือกลับเข้าสู่โมเดล

Plugin แบบรวมชุดรุ่นเดิมยังคงสามารถใช้
`api.registerCodexAppServerExtensionFactory(...)` สำหรับมิดเดิลแวร์เฉพาะ app-server ของ Codex
ได้ แต่การแปลงผลลัพธ์ใหม่ควรใช้ API ที่ไม่ขึ้นกับรันไทม์ ฮุก
`api.registerEmbeddedExtensionFactory(...)` ที่ใช้ได้เฉพาะ embedded runner
ถูกนำออกแล้ว การแปลงผลลัพธ์เครื่องมือแบบฝังตัวต้องใช้มิดเดิลแวร์ที่ไม่ขึ้นกับรันไทม์

### การจำแนกผลลัพธ์ปลายทาง

ฮาร์เนสแบบเนทีฟที่จัดการการฉายโปรโตคอลของตนเองสามารถใช้
`classifyAgentHarnessTerminalOutcome(...)` จาก
`openclaw/plugin-sdk/agent-harness-runtime` เมื่อเทิร์นที่เสร็จสมบูรณ์ไม่สร้าง
ข้อความผู้ช่วยที่มองเห็นได้ ตัวช่วยจะคืนค่า `empty`, `reasoning-only` หรือ
`planning-only` เพื่อให้นโยบายทางเลือกสำรองของ OpenClaw ตัดสินใจว่าจะลองใหม่ด้วย
โมเดลอื่นหรือไม่ `planning-only` ต้องใช้ฟิลด์ `planText` ที่ฮาร์เนสระบุ
อย่างชัดเจน OpenClaw จะไม่อนุมานค่านี้จากข้อความร้อยแก้วของผู้ช่วย ตัวช่วย
จงใจไม่จำแนกข้อผิดพลาดของพรอมต์ เทิร์นที่กำลังดำเนินอยู่ และ
การตอบกลับแบบเงียบโดยเจตนา เช่น `NO_REPLY`

### ผลข้างเคียงเมื่อเอเจนต์สิ้นสุด

ฮาร์เนสแบบเนทีฟต้องเรียก `runAgentEndSideEffects(...)` จาก
`openclaw/plugin-sdk/agent-harness-runtime` หลังจากสรุปผลความพยายามแล้ว โดยจะ
เรียกใช้ฮุก `agent_end` แบบพกพาและการบันทึกงานวิจัยของ OpenClaw
โดยไม่ทำให้การตอบกลับแบบโต้ตอบล่าช้า ใช้ `awaitAgentEndSideEffects(...)` สำหรับ
การรันภายในแบบไม่โต้ตอบ ซึ่งความพยายามต้องยังไม่ถือว่าเสร็จจนกว่า
ผลข้างเคียงเหล่านั้นจะเสร็จสิ้น ตัวช่วยทั้งสองรับเพย์โหลด `{ event, ctx }` เดียวกันกับ
`runAgentHarnessAgentEndHook(...)` ความล้มเหลวของตัวช่วยจะไม่เปลี่ยนแปลง
ผลลัพธ์ของความพยายามที่เสร็จสมบูรณ์แล้ว

### อินพุตของผู้ใช้และพื้นผิวเครื่องมือ

ฮาร์เนสแบบเนทีฟที่เปิดเผยคำขออินพุตของผู้ใช้ระดับรันไทม์ควรใช้
ตัวช่วยอินพุตของผู้ใช้จาก `openclaw/plugin-sdk/agent-harness-runtime` เพื่อจัดรูปแบบ
พรอมต์ ส่งผ่านเส้นทางการตอบกลับแบบบล็อกของ OpenClaw และปรับคำตอบ
แบบตัวเลือก/ข้อความอิสระกลับเป็นรูปแบบการตอบกลับเนทีฟของรันไทม์
ตัวช่วยช่วยให้การนำเสนอผ่านช่องทาง/TUI สอดคล้องกัน ขณะที่แต่ละฮาร์เนสยังคงจัดการ
การแยกวิเคราะห์โปรโตคอลและวงจรชีวิตของคำขอที่รอดำเนินการของตนเอง

ฮาร์เนสแบบเนทีฟที่ต้องการการกำหนดเส้นทางเครื่องมือแบบกระชับคล้าย PI ควรใช้
`createAgentHarnessToolSurfaceRuntime(...)` จาก
`openclaw/plugin-sdk/agent-harness-tool-runtime` ซึ่งจัดการ
การเลือกตัวควบคุมการค้นหาเครื่องมือ/โหมดโค้ด ค่าเริ่มต้นแบบกระชับสำหรับโมเดลภายใน
การกรองสคีมาที่เข้ากันได้กับรันไทม์ การเรียกใช้แค็ตตาล็อกที่ซ่อนอยู่ การเติมข้อมูล
ไดเรกทอรี และการล้างแค็ตตาล็อก ฮาร์เนสยังคงจัดการการแปลงเครื่องมือ
เฉพาะ SDK และคอลแบ็กการเรียกใช้แบบเนทีฟของตนเอง

### โหมดฮาร์เนส Codex แบบเนทีฟ

ฮาร์เนส `codex` แบบรวมชุดคือโหมด Codex แบบเนทีฟสำหรับเทิร์นของเอเจนต์ OpenClaw
แบบฝังตัว เปิดใช้ Plugin `codex` แบบรวมชุดก่อน และเพิ่ม `codex` ใน
`plugins.allow` หากการกำหนดค่าของคุณใช้รายการอนุญาตแบบจำกัด การกำหนดค่า app-server
แบบเนทีฟควรใช้ `openai/gpt-*` เทิร์นของเอเจนต์ OpenAI จะเลือกฮาร์เนส Codex
เฉพาะเมื่อเส้นทางที่มีผลประกาศความเข้ากันได้กับ Codex เท่านั้น การอ้างอิงโมเดล Codex
รุ่นเดิมควรซ่อมแซมด้วย `openclaw doctor --fix` และการอ้างอิงโมเดล `codex/*`
รุ่นเดิมยังคงเป็นนามแฝงเพื่อความเข้ากันได้สำหรับฮาร์เนสแบบเนทีฟ

เมื่อโหมดนี้ทำงาน Codex จะเป็นผู้จัดการ ID เธรดแบบเนทีฟ พฤติกรรมการดำเนินการต่อ
Compaction และการเรียกใช้ app-server ส่วน OpenClaw ยังคงจัดการช่องทางแชต
สำเนาทรานสคริปต์ที่มองเห็นได้ นโยบายเครื่องมือ การอนุมัติ การส่งสื่อ และการเลือก
เซสชัน ใช้ผู้ให้บริการ/โมเดล `agentRuntime.id: "codex"` เมื่อต้องการ
พิสูจน์ว่ามีเพียงเส้นทาง app-server ของ Codex เท่านั้นที่รับช่วงการรันได้ รันไทม์
Plugin ที่ระบุอย่างชัดเจนจะปฏิเสธแบบปิด ความล้มเหลวในการเลือก app-server ของ Codex
และความล้มเหลวของรันไทม์จะไม่ถูกลองใหม่ผ่านรันไทม์อื่น

## ความเข้มงวดของรันไทม์

โดยค่าเริ่มต้น OpenClaw ใช้นโยบายรันไทม์ผู้ให้บริการ/โมเดล `auto`:
ฮาร์เนส Plugin ที่ลงทะเบียนแล้วสามารถรับช่วงเส้นทางที่มีผลและเข้ากันได้ และรันไทม์
แบบฝังตัวจะจัดการเทิร์นเมื่อไม่มีฮาร์เนสใดตรงกัน คำนำหน้าผู้ให้บริการ/โมเดลเพียงอย่างเดียว
จะไม่เลือกฮาร์เนส ใช้รันไทม์ Plugin ของผู้ให้บริการ/โมเดลที่ระบุอย่างชัดเจน เช่น
`agentRuntime.id: "codex"` เมื่อต้องการให้การเลือกฮาร์เนสที่ไม่พบล้มเหลว
แทนที่จะกำหนดเส้นทางผ่านรันไทม์แบบฝังตัว การเลือกอย่างชัดเจนไม่ได้ทำให้
เส้นทางที่เข้ากันไม่ได้กลายเป็นเข้ากันได้ ความล้มเหลวของฮาร์เนส Plugin ที่เลือกจะ
ล้มเหลวอย่างเด็ดขาดเสมอ การดำเนินการนี้ไม่ขัดขวาง
`agentRuntime.id: "openclaw"` ของผู้ให้บริการ/โมเดลที่ระบุอย่างชัดเจน

สำหรับการรันแบบฝังตัวที่ใช้ Codex เท่านั้น:

```json
{
  "models": {
    "providers": {
      "openai": {
        "agentRuntime": {
          "id": "codex"
        }
      }
    }
  },
  "agents": {
    "defaults": {
      "model": "openai/gpt-5.6-sol"
    }
  }
}
```

หากต้องการแบ็กเอนด์ CLI สำหรับโมเดลมาตรฐานเพียงโมเดลเดียว ให้กำหนดรันไทม์ไว้ใน
รายการของโมเดลนั้น:

```json
{
  "agents": {
    "defaults": {
      "model": "anthropic/claude-opus-4-8",
      "models": {
        "anthropic/claude-opus-4-8": {
          "agentRuntime": {
            "id": "claude-cli"
          }
        }
      }
    }
  }
}
```

การแทนที่สำหรับแต่ละเอเจนต์ใช้รูปแบบที่กำหนดขอบเขตตามโมเดลเดียวกัน:

```json
{
  "agents": {
    "list": [
      {
        "id": "codex-only",
        "model": "openai/gpt-5.6-sol",
        "models": {
          "openai/gpt-5.6-sol": {
            "agentRuntime": { "id": "codex" }
          }
        }
      }
    ]
  }
}
```

ตัวอย่างรันไทม์ระดับเอเจนต์ทั้งหมดแบบเดิม เช่นตัวอย่างนี้ จะถูกละเว้น:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "codex"
      }
    }
  }
}
```

เมื่อใช้รันไทม์ Plugin ที่ระบุอย่างชัดเจน เซสชันจะล้มเหลวตั้งแต่เนิ่น ๆ เมื่อ
ฮาร์เนสที่ร้องขอไม่ได้ลงทะเบียน ไม่รองรับผู้ให้บริการ/โมเดลที่แก้ไขแล้ว หรือ
ล้มเหลวก่อนสร้างผลข้างเคียงของเทิร์น นี่เป็นพฤติกรรมโดยเจตนาสำหรับการติดตั้งใช้งาน
ที่ใช้ Codex เท่านั้น และสำหรับการทดสอบจริงที่ต้องพิสูจน์ว่าเส้นทาง app-server
ของ Codex กำลังถูกใช้งานจริง

การตั้งค่านี้ควบคุมเฉพาะฮาร์เนสเอเจนต์แบบฝังตัวเท่านั้น ไม่ได้ปิดใช้งาน
การกำหนดเส้นทางโมเดลเฉพาะผู้ให้บริการสำหรับรูปภาพ วิดีโอ เพลง TTS, PDF หรืออื่น ๆ

## เซสชันแบบเนทีฟและสำเนาทรานสคริปต์

ฮาร์เนสอาจเก็บ ID เซสชันแบบเนทีฟ ID เธรด หรือโทเค็นดำเนินการต่อ
ฝั่งดีมอน ให้เชื่อมโยงการผูกนี้กับเซสชัน OpenClaw อย่างชัดเจน และ
ทำสำเนาเอาต์พุตของผู้ช่วย/เครื่องมือที่ผู้ใช้มองเห็นลงในทรานสคริปต์ OpenClaw
อย่างต่อเนื่อง

ทรานสคริปต์ OpenClaw ยังคงเป็นชั้นความเข้ากันได้สำหรับ:

- ประวัติเซสชันที่มองเห็นได้ในช่องทาง
- การค้นหาและการทำดัชนีทรานสคริปต์
- การสลับกลับไปใช้ฮาร์เนส OpenClaw ในตัวในเทิร์นภายหลัง
- พฤติกรรมทั่วไปของ `/new`, `/reset` และการลบเซสชัน

หากฮาร์เนสเก็บการผูกไว้ในไฟล์ประกอบ ให้ติดตั้ง `reset(...)` เพื่อให้ OpenClaw
สามารถล้างข้อมูลนั้นได้เมื่อเซสชัน OpenClaw เจ้าของถูกรีเซ็ต

## ผลลัพธ์ของเครื่องมือและสื่อ

แกนหลักสร้างรายการเครื่องมือ OpenClaw และส่งไปยังความพยายาม
ที่เตรียมไว้ เมื่อฮาร์เนสเรียกใช้เครื่องมือแบบไดนามิก ให้คืนผลลัพธ์ของเครื่องมือ
ผ่านรูปแบบผลลัพธ์ของฮาร์เนส แทนที่จะส่งสื่อผ่านช่องทางด้วยตนเอง

วิธีนี้ทำให้เอาต์พุตข้อความ รูปภาพ วิดีโอ เพลง TTS การอนุมัติ และเครื่องมือส่งข้อความ
อยู่บนเส้นทางการส่งเดียวกับการรันที่ใช้ OpenClaw

### ผลลัพธ์ปลายทางของเครื่องมือ

`AgentHarnessAttemptParams.observeToolTerminal` คือตัวสะสมผลลัพธ์
ปลายทางที่โฮสต์เป็นผู้จัดการ ฮาร์เนสที่เรียกใช้เครื่องมือแบบไดนามิกของ OpenClaw
หรือเครื่องมือแบบเนทีฟต้องเรียกใช้ตัวสะสมนี้เมื่อเครื่องมือแต่ละรายการถึงผลลัพธ์ปลายทางหนึ่งรายการ
ก่อนสรุปผลลัพธ์ของความพยายาม ฮาร์เนสที่ไม่ได้เรียกใช้เครื่องมือไม่จำเป็นต้อง
เรียกใช้ตัวสะสมนี้

รายงานข้อเท็จจริงจากขอบเขตการเรียกใช้:

- ส่ง ID การเรียกโปรโตคอลเมื่อมี ชื่อเครื่องมือมาตรฐาน และ
  อาร์กิวเมนต์ที่ส่งถึงเครื่องมือจริงหลังการเตรียมหรือการเขียนใหม่โดยฮุก
- ตั้งค่า `executionStarted: false` เมื่อการตรวจสอบ การอนุมัติ หรือกลไกป้องกันอื่น
  หยุดการเรียกก่อนเริ่มการทำงานของเครื่องมือ เมื่ออาจมีการส่งต่อไปแล้ว ให้รายงาน
  `true` อย่างระมัดระวัง
- รายงาน `outcome: "success"` หรือ `outcome: "failure"` รวมฟิลด์ความล้มเหลว
  แบบมีโครงสร้างที่รันไทม์มีให้ แทนการอนุมานความล้มเหลวจากข้อความแสดงผล
- ใช้ `nativeMutation` เฉพาะกับเครื่องมือแบบเนทีฟที่ไม่ได้ใช้คำจำกัดความ
  เครื่องมือของ OpenClaw ระบุข้อเท็จจริงด้านการเปลี่ยนแปลงและการเล่นซ้ำที่โปรโตคอล
  เป็นผู้จัดการในจุดนั้น ห้ามคัดลอกตัวจำแนกการเปลี่ยนแปลงของ OpenClaw เข้าไปในฮาร์เนส

คอลแบ็กจะคืนค่าการแก้ไขมาตรฐานสำหรับการเรียกนั้น ส่งต่อ
`lastToolError` ไปยัง `AgentHarnessAttemptResult` และใช้ข้อเท็จจริงด้านการเรียกใช้
อาร์กิวเมนต์ และผลข้างเคียงของค่านี้ในการฉายของฮาร์เนส แทนการสร้างสถานะ
คู่ขนาน โฮสต์จะคงความล้มเหลวจากการเปลี่ยนแปลงที่ยังไม่ได้แก้ไขไว้แม้มีเครื่องมืออื่น
ที่ไม่เกี่ยวข้องทำงานสำเร็จ และจะล้างความล้มเหลวนั้นเฉพาะเมื่อการดำเนินการที่ตรงกันสำเร็จ

คอลแบ็กยังคงเป็นทางเลือกเพื่อความเข้ากันได้ของซอร์สกับฮาร์เนสทดลอง
รุ่นเก่า การเป็นทางเลือกไม่ได้หมายความว่าฮาร์เนสที่เรียกใช้เครื่องมือจะละเลยได้:
หากไม่มีรายงานผลลัพธ์ปลายทาง OpenClaw จะไม่สามารถรักษาข้อเท็จจริงของความล้มเหลว
จากเครื่องมือที่ทำให้เกิดการเปลี่ยนแปลงไว้ข้ามการเรียกเครื่องมือภายหลัง รวมถึงการเสร็จสิ้น
Heartbeat แบบเงียบ

## ข้อจำกัดปัจจุบัน

- พาธการนำเข้าสาธารณะเป็นแบบทั่วไป แต่นามแฝงของชนิดความพยายาม/ผลลัพธ์
  บางรายการยังคงใช้ชื่อรุ่นเดิมเพื่อความเข้ากันได้
- การติดตั้งฮาร์เนสของบุคคลที่สามยังอยู่ในขั้นทดลอง ควรเลือกใช้ Plugin
  ของผู้ให้บริการจนกว่าจะต้องใช้รันไทม์เซสชันแบบเนทีฟ
- รองรับการสลับฮาร์เนสระหว่างเทิร์น ห้ามสลับฮาร์เนสระหว่าง
  กลางเทิร์น หลังจากเครื่องมือแบบเนทีฟ การอนุมัติ ข้อความผู้ช่วย หรือการส่งข้อความ
  เริ่มทำงานแล้ว
