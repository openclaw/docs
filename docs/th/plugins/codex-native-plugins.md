การรองรับ Plugin ของ Codex แบบเนทีฟช่วยให้เอเจนต์ OpenClaw ในโหมด Codex ใช้ความสามารถของแอปและ Plugin จาก app-server ของ Codex ภายในเธรด Codex เดียวกันกับที่จัดการรอบการทำงานของ OpenClaw การเรียก Plugin จะยังคงอยู่ในทรานสคริปต์ Codex แบบเนทีฟ โดย app-server ของ Codex เป็นผู้ดำเนินการ MCP ที่มีแอปรองรับ OpenClaw จะไม่แปลง Plugin ของ Codex เป็นเครื่องมือไดนามิก `codex_plugin_*` สังเคราะห์ของ OpenClaw

ใช้หน้านี้หลังจากที่ [ชุดควบคุม Codex](/th/plugins/codex-harness) พื้นฐานทำงานแล้ว

## ข้อกำหนด

- รันไทม์ของเอเจนต์ต้องเป็นชุดควบคุม Codex แบบเนทีฟ
- `plugins.entries.codex.enabled` ต้องเป็น `true`
- `plugins.entries.codex.config.codexPlugins.enabled` ต้องเป็น `true`
- app-server ของ Codex เป้าหมายต้องมองเห็นรายการมาร์เก็ตเพลส, Plugin และแอปที่คาดไว้
- การย้ายข้อมูลรองรับเฉพาะ Plugin `openai-curated` ที่ตรวจพบว่าติดตั้งจากซอร์สใน Codex home ต้นทาง
- Plugin `workspace-directory` ที่กำหนดค่าด้วยตนเองต้องใช้ app-server ของ Codex ซึ่ง `plugin/list` ยอมรับ `marketplaceKinds` และข้อมูลสรุปเวิร์กสเปซที่ไม่มีพาธมี `remotePluginId` Plugin ต้องติดตั้งและเปิดใช้งานไว้แล้ว และแอปที่ Plugin เป็นเจ้าของต้องเข้าถึงได้ใน `app/list`

`codexPlugins` ไม่มีผลต่อการทำงานผ่านผู้ให้บริการ OpenClaw, การเชื่อมโยงบทสนทนา ACP หรือชุดควบคุมอื่น เนื่องจากเส้นทางเหล่านั้นไม่เคยสร้างเธรด app-server ของ Codex ที่มีการกำหนดค่า `apps` แบบเนทีฟ

บัญชี Codex, ความพร้อมใช้งานของแอป และการควบคุมแอป/Plugin ของเวิร์กสเปซในฝั่ง OpenAI มาจากบัญชี Codex ที่ลงชื่อเข้าใช้ โปรดดู
[การใช้ Codex กับแพ็กเกจ ChatGPT ของคุณ](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan)
สำหรับรูปแบบบัญชีและการดูแลระบบของ OpenAI

## เริ่มต้นอย่างรวดเร็ว

แสดงตัวอย่างการย้ายข้อมูลจาก Codex home ต้นทาง:

```bash
openclaw migrate codex --dry-run
```

เพิ่ม `--verify-plugin-apps` เพื่อให้การย้ายข้อมูลเรียก `app/list` ของต้นทาง และกำหนดให้แอปทุกแอปที่ Plugin เป็นเจ้าของต้องมีอยู่ เปิดใช้งาน และเข้าถึงได้ก่อนวางแผนเปิดใช้งานแบบเนทีฟ:

```bash
openclaw migrate codex --dry-run --verify-plugin-apps
```

นำการย้ายข้อมูลไปใช้เมื่อแผนถูกต้อง:

```bash
openclaw migrate apply codex --yes
```

การย้ายข้อมูลจะเขียนรายการ `codexPlugins` อย่างชัดเจนสำหรับ Plugin ที่มีสิทธิ์ และเรียก `plugin/install` ของ app-server ของ Codex สำหรับ Plugin ที่เลือก การกำหนดค่าหลังย้ายข้อมูลจะมีลักษณะดังนี้:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

การย้ายข้อมูลยังคงจำกัดเฉพาะ `openai-curated` หากต้องการใช้ Plugin `workspace-directory` ที่มีอยู่ ให้เพิ่มด้วยตนเองโดยใช้ `summary.id` แบบระบุมาร์เก็ตเพลสอย่างถูกต้องตรงตามที่ `plugin/list` ส่งคืน ตัวอย่างเช่น หาก Codex ส่งคืน `example-plugin@workspace-directory` ให้กำหนดค่าด้วยค่าที่ครบถ้วนนั้นแทนชื่อที่แสดง:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            plugins: {
              "example-plugin": {
                enabled: true,
                marketplaceName: "workspace-directory",
                pluginName: "example-plugin@workspace-directory",
              },
            },
          },
        },
      },
    },
  },
}
```

OpenClaw จะไม่เรียก `plugin/install` หรือเริ่มการยืนยันตัวตนสำหรับ Plugin `workspace-directory` ให้ติดตั้ง เปิดใช้งาน และยืนยันตัวตนใน Codex ก่อนเพิ่มหรือเปิดใช้งานนโยบาย OpenClaw โดย OpenClaw จะซ่อนแอปไว้เมื่อการตอบกลับไม่มีหลักฐานที่ตรงกันทุกประการเกี่ยวกับมาร์เก็ตเพลส, รหัส Plugin, รหัสรายละเอียด หรือความพร้อมของแอป หาก Codex ปฏิเสธคำขอ `plugin/list` ของเวิร์กสเปซแบบระบุชัดเจน OpenClaw จะรายงาน `marketplace_missing` สำหรับ Plugin ของเวิร์กสเปซที่เปิดใช้งานแต่ละรายการ และยังคงให้ Plugin แบบคัดสรรที่ค้นพบแยกต่างหากใช้งานได้

หลังจากเปลี่ยน `codexPlugins` บทสนทนา Codex ใหม่จะรับชุดแอปที่อัปเดตโดยอัตโนมัติ เรียกใช้ `/new` หรือ `/reset` เพื่อรีเฟรชบทสนทนาปัจจุบัน การเปลี่ยนการเปิด/ปิดใช้งาน Plugin ไม่จำเป็นต้องรีสตาร์ต Gateway

## จัดการ Plugin จากแชต

`/codex plugins` ใช้ตรวจสอบหรือเปลี่ยน Plugin ของ Codex แบบเนทีฟที่กำหนดค่าไว้จากแชตเดียวกับที่คุณใช้ควบคุมชุดควบคุม Codex:

```text
/codex plugins
/codex plugins list
/codex plugins disable google-calendar
/codex plugins enable google-calendar
```

`/codex plugins` เป็นนามแฝงของ `/codex plugins list` รายการจะแสดงคีย์ สถานะเปิด/ปิด ชื่อ Plugin ของ Codex และมาร์เก็ตเพลสของ Plugin ที่กำหนดค่าแต่ละรายการจาก `plugins.entries.codex.config.codexPlugins.plugins`

`enable`/`disable` เขียนเฉพาะใน `~/.openclaw/openclaw.json` เท่านั้น โดยจะไม่แก้ไข `~/.codex/config.toml` หรือติดตั้ง Plugin ของ Codex ใหม่ มีเพียงเจ้าของหรือไคลเอนต์ Gateway ที่มีขอบเขต `operator.admin` เท่านั้นที่เรียกใช้ได้

การเปิดใช้งาน Plugin ที่กำหนดค่าไว้จะเปิดสวิตช์ส่วนกลาง `codexPlugins.enabled` ด้วย หาก Plugin แบบคัดสรรถูกเขียนไว้ในสถานะปิดใช้งานเพราะการย้ายข้อมูลส่งคืน `auth_required` ให้อนุญาตแอปอีกครั้งใน Codex ก่อนเปิดใช้งานใน OpenClaw สำหรับรายการ `workspace-directory` การเปิดใช้งานที่นี่จะเปลี่ยนเฉพาะนโยบาย OpenClaw เท่านั้น โดย Plugin และแอปต้องทำงานอยู่ใน Codex แล้ว

## วิธีการตั้งค่า Plugin แบบเนทีฟ

การผสานรวมติดตามสถานะสามสถานะ:

| สถานะ      | ความหมาย                                                                                                                            |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| ติดตั้งแล้ว  | Codex มีชุด Plugin อยู่ในรันไทม์ app-server เป้าหมาย                                                                      |
| เปิดใช้งานแล้ว    | Codex รายงานว่า Plugin เปิดใช้งานอยู่ และการกำหนดค่า OpenClaw อนุญาตให้ใช้ในรอบการทำงานของชุดควบคุม Codex                                           |
| เข้าถึงได้ | app-server ของ Codex ยืนยันว่ารายการแอปของ Plugin พร้อมใช้งานสำหรับบัญชีที่ใช้งานอยู่และเชื่อมโยงกับข้อมูลประจำตัว Plugin ที่กำหนดค่าไว้ |

สำหรับ Plugin `openai-curated` การย้ายข้อมูลคือขั้นตอนการติดตั้ง/ตรวจสอบสิทธิ์ที่คงทน:

- ระหว่างการวางแผน OpenClaw จะอ่านรายละเอียด `plugin/read` ของ Codex ต้นทาง และตรวจสอบว่าบัญชี app-server ของ Codex ต้นทางเป็นบัญชีที่สมัครใช้งาน ChatGPT การตอบกลับที่ระบุว่าไม่ใช่บัญชี ChatGPT หรือไม่มีบัญชี จะข้าม Plugin ที่มีแอปรองรับด้วย `codex_subscription_required`
- โดยค่าเริ่มต้น การย้ายข้อมูลจะข้ามการเรียก `app/list` ของต้นทาง: Plugin ต้นทางที่มีแอปรองรับและผ่านเกณฑ์บัญชีจะถูกวางแผนโดยไม่ตรวจสอบการเข้าถึงแอปต้นทาง และความล้มเหลวในการรับข้อมูลบัญชีผ่านการรับส่งข้อมูลจะถูกข้ามด้วย `codex_account_unavailable`
- เมื่อใช้ `--verify-plugin-apps` การย้ายข้อมูลจะบันทึกสแนปช็อต `app/list` ของต้นทางใหม่ และกำหนดให้แอปทุกแอปที่ Plugin เป็นเจ้าของต้องมีอยู่ เปิดใช้งาน และเข้าถึงได้ก่อนวางแผนเปิดใช้งานแบบเนทีฟ จากนั้นความล้มเหลวในการรับข้อมูลบัญชีผ่านการรับส่งข้อมูลจะไปตรวจสอบกับเกณฑ์รายการแอปต้นทางแทนที่จะข้ามทันที

สำหรับ Plugin `workspace-directory` การตั้งค่าจะเกิดขึ้นภายนอก OpenClaw โดย OpenClaw จะสอบถามมาร์เก็ตเพลสนั้นเฉพาะเมื่อมีการกำหนดค่ารายการเวิร์กสเปซที่เปิดใช้งานอย่างน้อยหนึ่งรายการ จากนั้นจะค้นหา Plugin แต่ละรายการด้วย `summary.id` ที่ตรงกันทุกประการ และนำการตรวจสอบความเป็นเจ้าของผ่าน `plugin/read` กับความพร้อมผ่าน `app/list` ที่มีอยู่กลับมาใช้ Plugin ที่ยังไม่ได้ติดตั้ง ปิดใช้งาน เข้าถึงไม่ได้ หรือยังไม่ได้ยืนยันตัวตนจะไม่เปิดเผยแอปใด ๆ และ OpenClaw จะไม่พยายามติดตั้งหรือยืนยันตัวตน

รายการแอปของรันไทม์เป็นการตรวจสอบการเข้าถึงของเซสชันเป้าหมายสำหรับทั้ง Plugin แบบคัดสรรที่ย้ายข้อมูลแล้วและ Plugin ของเวิร์กสเปซที่กำหนดค่าด้วยตนเอง การตั้งค่าเซสชันของชุดควบคุม Codex จะคำนวณการกำหนดค่าแอปแบบจำกัดสำหรับเธรดจากแอปของ Plugin ที่เปิดใช้งานและเข้าถึงได้ โดยจะไม่คำนวณใหม่ในทุกรอบการทำงาน ดังนั้น `/codex plugins enable`/`disable` จะมีผลเฉพาะกับบทสนทนา Codex ใหม่ ใช้ `/new` หรือ `/reset` เพื่อรับการเปลี่ยนแปลงในบทสนทนาปัจจุบัน

## ขอบเขตการรองรับ V1

- เฉพาะ Plugin `openai-curated` ที่ติดตั้งอยู่แล้วในรายการ app-server ของ Codex ต้นทางเท่านั้นที่มีสิทธิ์ย้ายข้อมูล
- รันไทม์ยังรองรับรายการ `workspace-directory` ที่ระบุชัดเจนบน app-server รุ่นที่ `plugin/list` รองรับ `marketplaceKinds` และส่งคืน `remotePluginId` สำหรับข้อมูลสรุปเวิร์กสเปซที่ไม่มีพาธ รายการเหล่านี้ต้องใช้ `summary.id` แบบระบุมาร์เก็ตเพลสที่ตรงกันทุกประการ และต้องติดตั้ง เปิดใช้งาน และเข้าถึงแอปได้อยู่แล้ว คำขอรายการเวิร์กสเปซที่ถูกปฏิเสธจะสร้างการวินิจฉัย `marketplace_missing` ต่อ Plugin ตามที่มีอยู่ การไม่มีหลักฐานเกี่ยวกับมาร์เก็ตเพลส, Plugin, รายละเอียด หรือแอป จะไม่เปิดเผยแอปของเวิร์กสเปซ ส่วนรายการแบบคัดสรรจากคำขอรายการเริ่มต้นจะยังคงใช้งานได้
- Plugin ต้นทางที่มีแอปรองรับต้องผ่านเกณฑ์การสมัครใช้งานระหว่างการย้ายข้อมูล `--verify-plugin-apps` จะเพิ่มเกณฑ์รายการแอปต้นทาง บัญชีที่ไม่ผ่านเกณฑ์การสมัครใช้งาน รวมถึงแอปต้นทางที่เข้าถึงไม่ได้/ปิดใช้งาน/หายไป หรือความล้มเหลวในการรีเฟรชรายการแอปในโหมดตรวจสอบ จะถูกรายงานเป็นรายการที่ข้ามและต้องจัดการด้วยตนเอง แทนที่จะเป็นรายการกำหนดค่าที่เปิดใช้งาน รายละเอียด Plugin ที่อ่านไม่ได้จะถูกข้ามก่อนเกณฑ์รายการแอป
- การย้ายข้อมูลจะเขียนข้อมูลประจำตัว Plugin อย่างชัดเจน (`marketplaceName` และ `pluginName`) โดยจะไม่เขียนพาธแคช `marketplacePath` ภายในเครื่อง
- `codexPlugins.enabled` เป็นสวิตช์เปิดใช้งานส่วนกลางเพียงรายการเดียว ไม่มีไวลด์การ์ด `plugins["*"]` หรือคีย์การกำหนดค่าที่มอบอำนาจติดตั้งโดยพลการ
- มาร์เก็ตเพลสที่ไม่ใช่แบบคัดสรร ชุด Plugin ที่แคชไว้ ฮุก และไฟล์การกำหนดค่า Codex จะถูกเก็บไว้ในรายงานการย้ายข้อมูลเพื่อให้ตรวจสอบด้วยตนเอง และไม่เปิดใช้งานโดยอัตโนมัติ รันไทม์ยอมรับรายการ `workspace-directory` ที่กำหนดค่าด้วยตนเอง แต่มาร์เก็ตเพลสอื่นยังไม่รองรับ

## รายการแอปและความเป็นเจ้าของ

OpenClaw อ่านรายการแอปของ Codex ผ่าน `app/list` ของ app-server แคชไว้ในหน่วยความจำเป็นเวลาหนึ่งชั่วโมง และรีเฟรชรายการที่ล้าสมัยหรือขาดหายแบบอะซิงโครนัส แคชนี้อยู่เฉพาะในกระบวนการ การรีสตาร์ต CLI หรือ Gateway จะล้างแคช และ OpenClaw จะสร้างใหม่จากการอ่าน `app/list` ครั้งถัดไป

การย้ายข้อมูลและรันไทม์ใช้คีย์แคชแยกกัน:

- การตรวจสอบการย้ายข้อมูลต้นทางใช้ Codex home และตัวเลือกเริ่มต้นของต้นทาง โดยทำงานเฉพาะเมื่อใช้ `--verify-plugin-apps` และบังคับให้ไล่อ่าน `app/list` ของต้นทางใหม่สำหรับการวางแผนรอบนั้น
- การตั้งค่ารันไทม์เป้าหมายใช้ข้อมูลประจำตัว app-server ของ Codex ของเอเจนต์เป้าหมายเมื่อสร้างการกำหนดค่าแอปของเธรด การเปิดใช้งาน Plugin แบบคัดสรรจะทำให้คีย์แคชเป้าหมายนั้นใช้ไม่ได้ แล้วบังคับรีเฟรชหลังจาก `plugin/install` การตั้งค่า `workspace-directory` จะไม่ผ่านเส้นทางการเปิดใช้งานนี้

แอปของ Plugin จะถูกเปิดเผยเฉพาะเมื่อ OpenClaw สามารถเชื่อมโยงกลับไปยัง Plugin ที่กำหนดค่าไว้ผ่านความเป็นเจ้าของที่เสถียร ได้แก่ รหัสแอปที่ตรงกันทุกประการจากรายละเอียด Plugin, ชื่อเซิร์ฟเวอร์ MCP ที่รู้จัก หรือเมทาดาทาที่เสถียรและไม่ซ้ำ ความเป็นเจ้าของที่อิงเฉพาะชื่อที่แสดงหรือมีความกำกวมจะถูกตัดออก จนกว่าการรีเฟรชรายการครั้งถัดไปจะพิสูจน์ความเป็นเจ้าของได้

## แอปของบัญชีที่เชื่อมต่อ

เอเจนต์ที่เจ้าของเป็นผู้ดำเนินการสามารถเลือกใช้แอปทั้งหมดที่เชื่อมต่อกับบัญชี Codex อยู่แล้ว โดยไม่ต้องมีแพ็กเกจ Plugin ที่ตรงกัน:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_all_plugins: true,
            allow_destructive_actions: "auto",
          },
        },
      },
    },
  },
}
```

`allow_all_plugins: true` จะบันทึกสแนปช็อต `app/list` แบบครบถ้วนเมื่อสร้างเธรด Codex แบบเนทีฟใหม่ และอนุญาตเฉพาะแอปที่ทำเครื่องหมายว่าเข้าถึงได้สำหรับบัญชีนั้น โดยจะไม่ติดตั้ง ยืนยันตัวตน หรือเปิดใช้งานแอปแบบส่วนกลาง เธรดที่มีอยู่จะเก็บชุดแอปที่บันทึกไว้ ใช้ `/new`, `/reset` หรือรีสตาร์ต Gateway เพื่อรับแอปที่เพิ่งเชื่อมต่อหรือถูกเพิกถอน

แอปของบัญชีจะรับค่า `codexPlugins.allow_destructive_actions` ส่วนกลาง ซึ่งยอมรับ `true`, `false`, `"auto"` หรือ `"ask"` นโยบายต่อ Plugin ที่ระบุชัดเจนจะแทนที่นโยบายส่วนกลางสำหรับรหัสแอปที่ทับซ้อนกัน ความล้มเหลวของรายการจะปฏิเสธการเข้าถึงโดยค่าเริ่มต้น แทนที่จะย้อนกลับไปใช้ค่าเริ่มต้นที่ไม่จำกัด

## การกำหนดค่าแอปของเธรด

OpenClaw แทรกแพตช์ `config.apps` ที่มีข้อจำกัดเข้มงวดสำหรับเธรด Codex:
ปิดใช้งาน `_default` และเปิดใช้งานเฉพาะแอปที่เป็นของ Plugin ที่กำหนดค่าและเปิดใช้งานไว้ หรือแอปของบัญชีที่เข้าถึงได้และได้รับอนุญาตจาก `allow_all_plugins`

ค่า `destructive_enabled` ของแต่ละแอปมาจากนโยบาย `allow_destructive_actions` ส่วนกลางหรือระดับ Plugin ที่มีผลบังคับใช้ โดย `true`, `"auto"` และ `"ask"` จะตั้งค่า `destructive_enabled: true` ส่วน `false` จะตั้งค่าเป็น `false` Codex ยังคงบังคับใช้ข้อมูลเมตาของเครื่องมือที่มีผลทำลายจากคำอธิบายประกอบเครื่องมือของแอปโดยตรง
ระบบปิดใช้งาน `_default` ด้วย `open_world_enabled: false` ส่วนแอปของ Plugin ที่เปิดใช้งานจะได้รับ `open_world_enabled: true` OpenClaw ไม่มีตัวเลือกนโยบายแบบเปิดกว้างแยกต่างหากในระดับ Plugin และไม่เก็บรายการปฏิเสธชื่อเครื่องมือที่มีผลทำลายแยกตาม Plugin

โหมดอนุมัติเครื่องมือมีค่าเริ่มต้นเป็นอัตโนมัติสำหรับแอปที่ได้รับอนุญาต ดังนั้นเครื่องมืออ่านที่ไม่มีผลทำลายจะทำงานโดยไม่แสดงข้อความขออนุมัติในเธรดเดียวกัน เครื่องมือที่มีผลทำลายยังคงอยู่ภายใต้การควบคุมของนโยบาย `destructive_enabled` ของแต่ละแอป

## นโยบายการดำเนินการที่มีผลทำลาย

โดยค่าเริ่มต้น ระบบอนุญาตคำขอข้อมูลเพิ่มเติมที่มีผลทำลายจาก Plugin สำหรับ Plugin ของ Codex ที่กำหนดค่าไว้ แต่จะปฏิเสธโดยอัตโนมัติเมื่อสคีมาไม่ปลอดภัยหรือไม่สามารถระบุความเป็นเจ้าของได้อย่างชัดเจน:

- `allow_destructive_actions` ส่วนกลางมีค่าเริ่มต้นเป็น `true`
- `allow_destructive_actions` ระดับ Plugin จะลบล้างนโยบายส่วนกลางสำหรับ
  Plugin นั้น
- `false`: OpenClaw ส่งคืนการปฏิเสธที่ให้ผลลัพธ์แน่นอน
- `true`: OpenClaw ยอมรับโดยอัตโนมัติเฉพาะสคีมาที่ปลอดภัยและสามารถแมปเป็นการตอบกลับการอนุมัติได้
  เช่น ฟิลด์อนุมัติแบบบูลีน
- `"auto"`: OpenClaw เปิดเผยการดำเนินการที่มีผลทำลายของ Plugin ให้ Codex จากนั้น
  แปลงคำขออนุมัติของ MCP ที่พิสูจน์ความเป็นเจ้าของได้เป็นการอนุมัติ Plugin ของ OpenClaw
  ก่อนส่งคืนการตอบกลับการอนุมัติของ Codex
- `"ask"`: OpenClaw ใช้การควบคุมการเขียนและการดำเนินการที่มีผลทำลายของ Codex แบบเดียวกับ
  `"auto"` ล้างค่าลบล้างการอนุมัติรายเครื่องมือแบบถาวรของ Codex สำหรับแอป
  ก่อนเริ่มเธรด และเสนอเฉพาะการอนุมัติหรือปฏิเสธแบบครั้งเดียว เพื่อไม่ให้
  การอนุมัติแบบถาวรระงับข้อความขออนุมัติการเขียนในภายหลัง สำหรับแต่ละ
  แอปที่ได้รับอนุญาตซึ่งใช้ `"ask"` OpenClaw จะเลือกผู้ตรวจสอบการอนุมัติโดยมนุษย์ของ Codex
  สำหรับแอปนั้น เพื่อให้ Codex ส่งคำขออนุมัติมายัง
  OpenClaw ส่วนแอปอื่นและการอนุมัติในเธรดที่ไม่เกี่ยวกับแอปจะยังคงใช้
  ผู้ตรวจสอบและนโยบายที่กำหนดค่าไว้
- หากไม่มีข้อมูลระบุตัวตนของ Plugin ไม่สามารถระบุความเป็นเจ้าของได้อย่างชัดเจน ไม่มีรหัสรอบการทำงานหรือรหัสไม่ตรงกัน
  หรือสคีมาคำขอข้อมูลเพิ่มเติมไม่ปลอดภัย ระบบจะปฏิเสธแทนการแสดงข้อความขออนุมัติ

**ติดตั้ง Plugin พื้นที่ทำงานแล้วแต่ไม่ปรากฏ:** ยืนยันว่าผลลัพธ์ `plugin/list`
ของพื้นที่ทำงานรายงานรหัสที่กำหนดค่าไว้อย่างตรงกันทุกประการว่าติดตั้งและเปิดใช้งานแล้ว
จากนั้นยืนยันว่า `app/list` รายงานว่าแอปทั้งหมดที่เป็นเจ้าของสามารถเข้าถึงได้ด้วยบัญชี Codex
เดียวกัน OpenClaw สามารถเปิดใช้งานแอปที่เข้าถึงได้สำหรับเธรด แม้ว่ารายการของ
บัญชีจะรายงานว่าแอปนั้นถูกปิดใช้งานอยู่ก็ตาม หากคุณเปลี่ยนสถานะดังกล่าวหลังจาก Gateway แคชรายการ
แอปแล้ว ให้รอการรีเฟรชแคชหนึ่งชั่วโมงหรือเริ่ม Gateway ใหม่ จากนั้นใช้
`/new` หรือ `/reset` OpenClaw จะไม่ซ่อมแซมหรือยืนยันตัวตนให้ Plugin พื้นที่ทำงาน
หากคำขอรายการพื้นที่ทำงานที่ระบุอย่างชัดเจนถูกปฏิเสธ รายการพื้นที่ทำงานที่เปิดใช้งานแต่ละรายการ
จะรายงาน `marketplace_missing` ส่วนรายการที่คัดสรรไว้อื่นซึ่งไม่เกี่ยวข้องจะดำเนินการต่อ
จากการตอบกลับรายการเริ่มต้น

สำหรับ `plugin_detail_unavailable` สรุปพื้นที่ทำงานที่ไม่มีพาธต้องมี
`remotePluginId` โดย OpenClaw จะซ่อนแอปที่เป็นเจ้าของไว้เมื่อไม่มีตัวเลือกดังกล่าวหรือ
ไม่มีผลลัพธ์ `plugin/read` ที่ตามมา สำหรับ
`plugin_activation_failed` Plugin ที่คัดสรรไว้อาจรายงานความล้มเหลวของมาร์เก็ตเพลส การยืนยันตัวตน หรือ
การรีเฟรชหลังการติดตั้ง Plugin พื้นที่ทำงานจะรายงานรหัสนี้เมื่อยังไม่ได้
เปิดใช้งาน ให้ติดตั้ง เปิดใช้งาน และยืนยันตัวตนนอก OpenClaw

**เปลี่ยนการกำหนดค่าแล้วแต่เอเจนต์มองไม่เห็น Plugin:** เรียกใช้ `/codex plugins
list` เพื่อยืนยันสถานะที่กำหนดค่าไว้ จากนั้นใช้ `/new` หรือ `/reset` การผูกเธรด
Codex ที่มีอยู่จะคงการกำหนดค่าแอปที่ใช้เมื่อเริ่มต้นไว้ จนกว่า OpenClaw
จะสร้างเซสชันชุดควบคุมใหม่หรือแทนที่การผูกที่ล้าสมัย

**การดำเนินการที่มีผลทำลายถูกปฏิเสธ:** ตรวจสอบค่า `allow_destructive_actions`
ส่วนกลางและระดับ Plugin แม้ใช้ `true`, `"auto"` หรือ `"ask"`
สคีมาคำขอข้อมูลเพิ่มเติมที่ไม่ปลอดภัยและข้อมูลระบุตัวตนของ Plugin ที่กำกวมจะยังคงถูกปฏิเสธโดยอัตโนมัติ
