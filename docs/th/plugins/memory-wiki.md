`memory-wiki` เป็น Plugin ที่รวมมาให้ ซึ่งคอมไพล์ความรู้ที่คงทนเป็น
วิกิที่นำทางได้: หน้าที่กำหนดผลลัพธ์ได้แน่นอน ข้อกล่าวอ้างแบบมีโครงสร้างพร้อมหลักฐาน
ที่มา แดชบอร์ด และบทสรุปที่เครื่องอ่านได้

Plugin นี้ไม่ได้มาแทนที่ Plugin Active Memory การเรียกคืน การเลื่อนระดับ การทำดัชนี และ
Dreaming ยังคงเป็นความรับผิดชอบของแบ็กเอนด์หน่วยความจำที่กำหนดค่าไว้
(`memory-core`, QMD, Honcho เป็นต้น) `memory-wiki` ทำงานควบคู่กันและคอมไพล์
ความรู้เป็นชั้นวิกิที่ได้รับการดูแลรักษา

| ชั้น                 | รับผิดชอบ                                                                          |
| -------------------- | --------------------------------------------------------------------------------- |
| Plugin Active Memory | การเรียกคืน การค้นหาเชิงความหมาย การเลื่อนระดับ Dreaming รันไทม์หน่วยความจำ       |
| `memory-wiki`        | หน้าวิกิที่คอมไพล์แล้ว การสังเคราะห์ที่มีข้อมูลที่มาครบถ้วน แดชบอร์ด การค้นหา/รับ/นำวิกิไปใช้ |

กฎเชิงปฏิบัติ:

- `memory_search` สำหรับการเรียกคืนแบบกว้างหนึ่งรอบจากคลังข้อมูลใดก็ตามที่กำหนดค่าไว้
- `wiki_search` / `wiki_get` เมื่อต้องการการจัดอันดับ ข้อมูลที่มา หรือโครงสร้างความเชื่อระดับหน้าที่เฉพาะเจาะจงกับวิกิ
- `memory_search corpus=all` เพื่อครอบคลุมทั้งสองชั้นในการเรียกครั้งเดียว เมื่อ Plugin Active Memory รองรับการเลือกคลังข้อมูล

การตั้งค่าแบบเน้นภายในเครื่องที่ใช้กันทั่วไป: ใช้ QMD เป็นแบ็กเอนด์ Active Memory สำหรับการเรียกคืน และ
ใช้ `memory-wiki` ในโหมด `bridge` สำหรับหน้าสังเคราะห์ที่คงทน ดู
ตัวอย่าง QMD + โหมดบริดจ์ในหัวข้อ [การกำหนดค่า](#configuration)

หากโหมดบริดจ์รายงานอาร์ติแฟกต์ที่ส่งออกเป็นศูนย์ แสดงว่า Plugin Active Memory
ยังไม่ได้เปิดเผยอินพุตบริดจ์สาธารณะในขณะนี้ ให้เรียกใช้ `openclaw wiki doctor` ก่อน
จากนั้นยืนยันว่า Plugin Active Memory รองรับอาร์ติแฟกต์สาธารณะ

## โหมดคลัง

- `isolated` (ค่าเริ่มต้น): มีคลังและแหล่งข้อมูลของตัวเอง โดยไม่ขึ้นกับ Plugin Active Memory ใช้สำหรับคลังความรู้ที่คัดสรรและทำงานได้ด้วยตัวเอง
- `bridge`: อ่านอาร์ติแฟกต์หน่วยความจำสาธารณะและบันทึกเหตุการณ์จาก Plugin Active Memory ผ่านช่องทาง SDK ของ Plugin แบบสาธารณะ ใช้เพื่อคอมไพล์อาร์ติแฟกต์ที่ Plugin หน่วยความจำส่งออกโดยไม่เข้าถึงส่วนภายในแบบส่วนตัวของ Plugin
- `unsafe-local`: ช่องทางหลีกเลี่ยงข้อจำกัดแบบชัดเจนสำหรับพาธส่วนตัวภายในเครื่องเดียวกัน เป็นฟีเจอร์ทดลองและย้ายข้ามระบบไม่ได้โดยเจตนา ใช้เฉพาะเมื่อเข้าใจขอบเขตความเชื่อถือและต้องการการเข้าถึงระบบไฟล์ภายในเครื่องที่โหมดบริดจ์ไม่สามารถให้ได้โดยเฉพาะ

โหมดคลังและขอบเขตคลังเป็นตัวเลือกที่แยกจากกัน:

- `vaultMode` เลือกแหล่งที่มาของอินพุตวิกิ
- `vault.scope` เลือกว่าจะให้เอเจนต์ทั้งหมดใช้คลังเดียวกัน หรือให้แต่ละเอเจนต์มีคลังย่อยของตนเอง

`vault.scope: "global"` เป็นค่าเริ่มต้นและคงพฤติกรรมคลังเดียวที่มีอยู่ไว้
ใช้ `vault.scope: "agent"` กับโหมด `isolated` หรือ `bridge` เมื่อ
เอเจนต์ต้องไม่ใช้หน้าวิกิ บทสรุปที่คอมไพล์แล้ว ผลการค้นหา หรือการเขียนร่วมกัน
ขอบเขตเอเจนต์ใช้ร่วมกับโหมด `unsafe-local` ไม่ได้ เนื่องจากพาธส่วนตัวที่กำหนดค่าไว้เหล่านั้น
ไม่ใช่อินพุตที่เอเจนต์เป็นเจ้าของ การตรวจสอบความถูกต้องของการกำหนดค่าปฏิเสธ
ชุดค่าผสมนี้

โหมดบริดจ์สามารถทำดัชนีรายการต่อไปนี้ตามสวิตช์การกำหนดค่า `bridge.*`:

- อาร์ติแฟกต์หน่วยความจำที่ส่งออก (`indexMemoryRoot`)
- บันทึกรายวัน (`indexDailyNotes`)
- รายงาน Dreaming (`indexDreamReports`)
- บันทึกเหตุการณ์หน่วยความจำ (`followMemoryEvents`)

เมื่อโหมดบริดจ์ทำงานอยู่และเปิดใช้ `bridge.readMemoryArtifacts`
`openclaw wiki status`, `openclaw wiki doctor` และ `openclaw wiki bridge
import` จะส่งต่อผ่าน Gateway ที่กำลังทำงาน เพื่อให้เห็นบริบท Plugin Active Memory เดียวกัน
กับหน่วยความจำของเอเจนต์/รันไทม์ หากปิดใช้บริดจ์หรือปิดการอ่านอาร์ติแฟกต์
คำสั่งเหล่านั้นจะยังคงทำงานแบบภายในเครื่อง/ออฟไลน์

## โครงสร้างคลัง

```text
<vault>/
  AGENTS.md
  WIKI.md
  index.md
  inbox.md
  entities/
  concepts/
  syntheses/
  sources/
  reports/
  _attachments/
  _views/
  .openclaw-wiki/
```

เนื้อหาที่จัดการจะอยู่ภายในบล็อกที่สร้างขึ้น ส่วนบล็อกบันทึกที่มนุษย์เขียน
จะได้รับการคงไว้เมื่อสร้างใหม่

- `sources/`: วัตถุดิบดิบที่นำเข้าและหน้าที่อิงกับบริดจ์/ภายในเครื่องที่ไม่ปลอดภัย
- `entities/`: สิ่ง บุคคล ระบบ โครงการ และวัตถุที่คงทน
- `concepts/`: แนวคิด นามธรรม รูปแบบ นโยบาย (รวมถึงเป็นตำแหน่งปลายทางสำหรับการนำเข้า OKF)
- `syntheses/`: สรุปที่คอมไพล์แล้วและผลรวมที่ได้รับการดูแลรักษา
- `reports/`: แดชบอร์ดที่สร้างขึ้น

## การนำเข้า Open Knowledge Format

```bash
openclaw wiki okf import ./bundles/ga4
```

นำเข้าบันเดิล Open Knowledge Format ที่แตกไฟล์แล้วไปยังหน้าแนวคิดของวิกิ เหมาะอย่างยิ่ง
เมื่อแค็ตตาล็อกข้อมูล โปรแกรมรวบรวมข้อมูลเอกสาร หรือเอเจนต์เพิ่มคุณค่า
สร้าง OKF อยู่แล้ว: ให้ OKF เป็นอาร์ติแฟกต์แลกเปลี่ยนแบบพกพา และให้ `memory-wiki`
แปลงเป็นหน้าแนวคิดและบทสรุปที่คอมไพล์แล้วในรูปแบบเนทีฟของ OpenClaw

- ไฟล์ `.md` ที่ไม่สงวนไว้เป็นเอกสารแนวคิด
- แนวคิดที่นำเข้าแต่ละรายการต้องมีฟิลด์ frontmatter `type` ที่ไม่ว่าง หากไม่มี `type` ระบบจะสร้างคำเตือน `missing-type` และข้ามไฟล์นั้น
- ยอมรับค่า `type` ที่ไม่รู้จักเป็นแนวคิดทั่วไป
- `index.md` และ `log.md` ถูกสงวนไว้และจะไม่ถูกนำเข้าเป็นแนวคิด
- ลิงก์ Markdown ที่เสียหรือชี้ไปภายนอกจะไม่ถูกแก้ไข

หน้าที่นำเข้าจะถูกปรับให้อยู่ภายใต้ `concepts/` เพื่อให้ขั้นตอนการคอมไพล์ การค้นหา การรับ และ
แดชบอร์ดที่มีอยู่มองเห็นหน้าเหล่านี้ได้โดยไม่ต้องมีโครงสร้างวิกิที่สอง แต่ละหน้าจะเก็บ
รหัสแนวคิด OKF เดิม พาธแหล่งที่มา `type`, `resource`, `tags`, การประทับเวลา
และ frontmatter ทั้งหมดจากผู้สร้าง ลิงก์ OKF ภายในจะถูกเขียนใหม่ให้ชี้ไปยัง
หน้าแนวคิดวิกิที่สร้างขึ้น และยังสร้างรายการ `relationships` แบบมีโครงสร้างพร้อม
`kind: okf-link`

## ข้อกล่าวอ้างและหลักฐานแบบมีโครงสร้าง

หน้าต่าง ๆ มี frontmatter `claims` แบบมีโครงสร้าง ไม่ใช่เพียงข้อความอิสระ แต่ละ
ข้อกล่าวอ้างสามารถมี `id`, `text`, `status`, `confidence`, `evidence[]` และ
`updatedAt` รายการหลักฐานแต่ละรายการสามารถมี `kind`, `sourceId`, `path`,
`lines`, `weight`, `confidence`, `privacyTier`, `note` และ `updatedAt`

สิ่งนี้ทำให้วิกิทำหน้าที่เป็นชั้นความเชื่อ ไม่ใช่ที่ทิ้งบันทึกแบบเฉื่อย
ข้อกล่าวอ้างสามารถได้รับการติดตาม ให้คะแนน โต้แย้ง และหาข้อยุติโดยย้อนกลับไปยังแหล่งข้อมูล

## เมตาดาต้าเอนทิตีสำหรับเอเจนต์

หน้าเอนทิตีมีเมตาดาต้าการกำหนดเส้นทางแบบทั่วไป ซึ่งใช้ได้กับบุคคล ทีม
ระบบ โครงการ หรือเอนทิตีประเภทอื่น:

- `entityType`: ตัวอย่างเช่น `person`, `team`, `system`, `project`
- `canonicalId`: คีย์ข้อมูลประจำตัวที่คงที่ข้ามนามแฝงและการนำเข้า
- `aliases`: ชื่อ แฮนเดิล หรือป้ายกำกับที่ชี้ไปยังหน้าเดียวกัน
- `privacyTier`: สตริงรูปแบบอิสระ โดย `public` ถือว่าไม่ต้องตรวจทาน ส่วนค่าอื่นใด (ตัวอย่างเช่น `local-private`, `sensitive`, `confirm-before-use`) จะถูกทำเครื่องหมายใน `reports/privacy-review.md`
- `bestUsedFor` / `notEnoughFor`: คำใบ้การกำหนดเส้นทางแบบกระชับ
- `lastRefreshedAt`: การประทับเวลาการรีเฟรชแหล่งข้อมูล ซึ่งแยกจากเวลาแก้ไขหน้า
- `personCard`: การ์ดกำหนดเส้นทางเฉพาะบุคคลแบบไม่บังคับ (แฮนเดิล โซเชียล อีเมล เขตเวลา เลน เรื่องที่ควรถาม เรื่องที่ควรหลีกเลี่ยงการถาม ความเชื่อมั่น ระดับความเป็นส่วนตัว)
- `relationships`: ขอบแบบมีชนิดไปยังหน้าที่เกี่ยวข้อง (เป้าหมาย ชนิด น้ำหนัก ความเชื่อมั่น ชนิดหลักฐาน ระดับความเป็นส่วนตัว บันทึก)

สำหรับวิกิบุคคล ให้เริ่มด้วย `reports/person-agent-directory.md` จากนั้นเปิด
หน้าบุคคลด้วย `wiki_get` ก่อนใช้รายละเอียดการติดต่อหรือข้อเท็จจริง
ที่อนุมานขึ้น

## ไปป์ไลน์การคอมไพล์

การคอมไพล์จะอ่านหน้าวิกิ ปรับสรุปให้เป็นรูปแบบมาตรฐาน และจัดเก็บสแนปช็อตสำหรับเครื่อง
ไว้ในสถานะ Plugin SQLite ที่ใช้ร่วมกันของ OpenClaw โค้ดรันไทม์ใช้
สแนปช็อตของเจ้าของที่วงจรชีวิตเป็นผู้ดูแลเพื่อโหลด SQLite ระหว่างการเตรียมพรอมต์แบบอะซิงโครนัส
การประกอบพรอมต์แบบซิงโครนัสจะไม่ดึงข้อมูลจาก Markdown หรืออ่านไฟล์แคช
ผลลัพธ์ที่คอมไพล์แล้วยังขับเคลื่อนการทำดัชนีวิกิรอบแรกสำหรับการค้นหา/รับ การค้นหา
รหัสข้อกล่าวอ้างย้อนกลับไปยังหน้าที่เป็นเจ้าของ ส่วนเสริมพรอมต์แบบกระชับ และการสร้าง
รายงาน

การแก้ไขแหล่งข้อมูลและการคืนค่าคลังจะพร้อมให้เครื่องใช้หลังจากการคอมไพล์ครั้งถัดไป
เท่านั้น การรีสตาร์ตหรือรีเฟรชวงจรชีวิตของ Plugin จะเปรียบเทียบสิ่งเผยแพร่จากการคอมไพล์
ของคลังซึ่งเชื่อมโยงกันตามเหตุและผลกับ SQLite และปฏิเสธสแนปช็อตจากสถานะที่ใหม่กว่า
ซึ่งถูกย้อนกลับแล้ว คอมไพเลอร์ที่เริ่มทำงานก่อนการย้อนกลับจะไม่สามารถเผยแพร่ทับ
สถานะก่อนหน้าที่ได้รับการคืนค่า การเตรียมพรอมต์จะไม่ตรวจสอบคลังเป็นระยะหรือ
ติดตั้งตัวเฝ้าดูไฟล์
หลังการกักกันจากการย้อนกลับ การคอมไพล์ในโปรเซสที่กำลังทำงานจะล้างเจ้าของ
ทันที ส่วนโปรเซสคอมไพเลอร์แยกต่างหากต้องรีเฟรชวงจรชีวิตของ Plugin เพื่อให้
ดีมอนยืนยันสิ่งเผยแพร่ใหม่ที่คงทนได้
แคชที่คอมไพล์แล้วสามารถสร้างใหม่ได้: แถวแคชจากก่อนยุคการเผยแพร่จะถูก
ถือว่าไม่พบและแทนที่ด้วยการคอมไพล์ครั้งถัดไป โดยจะไม่ถูกย้ายข้อมูล

## การค้นหาและการเรียกคืน

แบ็กเอนด์การค้นหาสองแบบ:

- `shared`: ใช้ขั้นตอนการค้นหาหน่วยความจำที่ใช้ร่วมกันเมื่อพร้อมใช้งาน
- `local`: ค้นหาวิกิภายในเครื่อง

คลังข้อมูลสามชุด: `wiki`, `memory`, `all`

- `wiki_search` / `wiki_get` ใช้บทสรุปที่คอมไพล์แล้วเป็นรอบแรกเมื่อทำได้
- รหัสข้อกล่าวอ้างจะถูกชี้กลับไปยังหน้าที่เป็นเจ้าของ
- ข้อกล่าวอ้างที่ถูกโต้แย้ง/ล้าสมัย/สดใหม่มีผลต่อการจัดอันดับ
- ป้ายกำกับข้อมูลที่มายังคงอยู่ในผลลัพธ์

โหมดการค้นหา (พารามิเตอร์ `--mode` / เครื่องมือ `mode`):

| โหมด              | เพิ่มประสิทธิภาพ                                                         |
| ----------------- | -------------------------------------------------------------- |
| `auto`            | ค่าเริ่มต้นที่สมดุล                                               |
| `find-person`     | เอนทิตีที่คล้ายบุคคล นามแฝง ชื่อเรียก บัญชีโซเชียล และ ID มาตรฐาน |
| `route-question`  | การ์ดเอเจนต์ คำแนะนำว่าควรถามเรื่องใด/เหมาะใช้กับเรื่องใดที่สุด และบริบทความสัมพันธ์ |
| `source-evidence` | หน้าต้นทางและข้อมูลเมตาหลักฐานแบบมีโครงสร้าง                  |
| `raw-claim`       | การอ้างสิทธิ์แบบมีโครงสร้างที่ตรงกัน โดยส่งคืนข้อมูลเมตาของการอ้างสิทธิ์/หลักฐาน    |

เมื่อผลลัพธ์ตรงกับการอ้างสิทธิ์แบบมีโครงสร้าง `wiki_search` จะส่งคืน
`matchedClaimId`, `matchedClaimStatus`, `matchedClaimConfidence`,
`evidenceKinds` และ `evidenceSourceIds` ในเพย์โหลดรายละเอียด เอาต์พุตข้อความ
จะมีบรรทัด `Claim:` และ `Evidence:` แบบกระชับเมื่อพร้อมใช้งาน

## เครื่องมือเอเจนต์

| เครื่องมือ          | วัตถุประสงค์                                                                                                                                                       |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `wiki_status` | โหมดและขอบเขตของคลังปัจจุบัน เอเจนต์ที่ได้รับการแก้ไขแล้ว สถานะความพร้อมใช้งาน และความพร้อมใช้งานของ Obsidian CLI                                                                               |
| `wiki_search` | ค้นหาหน้าวิกิและคลังหน่วยความจำที่ใช้ร่วมกันเมื่อกำหนดค่าไว้ รองรับ `mode` สำหรับค้นหาบุคคล กำหนดเส้นทางคำถาม ตรวจสอบหลักฐานต้นทาง หรือเจาะลึกการอ้างสิทธิ์ดิบ |
| `wiki_get`    | อ่านหน้าวิกิตาม ID/พาธ โดยย้อนกลับไปใช้คลังหน่วยความจำที่ใช้ร่วมกันเมื่อเปิดใช้การค้นหาร่วมกันและไม่พบรายการที่ค้นหา                                     |
| `wiki_apply`  | การสังเคราะห์/แก้ไขข้อมูลเมตาแบบจำกัดขอบเขต โดยไม่ผ่าตัดหน้าแบบอิสระ                                                                                             |
| `wiki_lint`   | การตรวจสอบโครงสร้าง ช่องว่างของที่มา ความขัดแย้ง และคำถามที่ยังเปิดอยู่                                                                                            |

Plugin ยังลงทะเบียนส่วนเสริมคลังหน่วยความจำแบบไม่ผูกขาด เพื่อให้
`memory_search` และ `memory_get` ที่ใช้ร่วมกันเข้าถึงวิกิได้ เมื่อ Plugin หน่วยความจำที่ใช้งานอยู่
รองรับการเลือกคลังข้อมูล

## ลักษณะการทำงานของพรอมต์และบริบท

เมื่อเปิดใช้ `context.includeCompiledDigestPrompt` ส่วนหน่วยความจำของพรอมต์
จะต่อท้ายสแนปช็อตที่คอมไพล์แบบกระชับจากสถานะ Plugin ได้แก่ เฉพาะหน้าสำคัญ
เฉพาะการอ้างสิทธิ์สำคัญ จำนวนความขัดแย้ง จำนวนคำถาม และตัวระบุคุณสมบัติด้านความเชื่อมั่น/ความสดใหม่
คุณลักษณะนี้เป็นแบบเลือกใช้ เนื่องจากเปลี่ยนรูปแบบพรอมต์ โดยมีความสำคัญหลัก
ต่อเอนจินบริบทหรือการประกอบพรอมต์ที่ใช้ส่วนเสริมหน่วยความจำอย่างชัดเจน

## การกำหนดค่า

ใส่การกำหนดค่าไว้ใต้ `plugins.entries.memory-wiki.config`:

```json5
{
  plugins: {
    entries: {
      "memory-wiki": {
        enabled: true,
        config: {
          vaultMode: "isolated",
          vault: {
            scope: "global",
            path: "~/.openclaw/wiki/main",
            renderMode: "obsidian",
          },
          obsidian: {
            enabled: true,
            useOfficialCli: true,
            vaultName: "OpenClaw Wiki",
            openAfterWrites: false,
          },
          bridge: {
            enabled: false,
            readMemoryArtifacts: true,
            indexDreamReports: true,
            indexDailyNotes: true,
            indexMemoryRoot: true,
            followMemoryEvents: true,
          },
          unsafeLocal: {
            allowPrivateMemoryCoreAccess: false,
            paths: [],
          },
          ingest: {
            autoCompile: true,
            maxConcurrentJobs: 1,
            allowUrlIngest: true,
          },
          search: {
            backend: "shared",
            corpus: "wiki",
          },
          context: {
            includeCompiledDigestPrompt: false,
          },
          render: {
            preserveHumanBlocks: true,
            createBacklinks: true,
            createDashboards: true,
          },
        },
      },
    },
  },
}
```

ตัวเลือกสำคัญ:

| คีย์                                        | ค่า / ค่าเริ่มต้น                               | หมายเหตุ                                                                         |
| ------------------------------------------ | ---------------------------------------------- | ----------------------------------------------------------------------------- |
| `vaultMode`                                | `isolated` (ค่าเริ่มต้น), `bridge`, `unsafe-local` | เลือกลักษณะการทำงานของอินพุตและการผสานรวม                                        |
| `vault.scope`                              | `global` (ค่าเริ่มต้น), `agent`                    | คลังที่ใช้ร่วมกันหนึ่งแห่งหรือคลังลูกหนึ่งแห่งต่อเอเจนต์                                 |
| `vault.path`                               | ค่าเริ่มต้นส่วนกลาง `~/.openclaw/wiki/main`         | คลังที่แน่นอนสำหรับส่วนกลาง โดยไดเรกทอรีแม่ในขอบเขตเอเจนต์มีค่าเริ่มต้นเป็น `~/.openclaw/wiki`       |
| `vault.renderMode`                         | `native` (ค่าเริ่มต้น), `obsidian`                 |                                                                               |
| `bridge.readMemoryArtifacts`               | ค่าเริ่มต้น `true`                                 | นำเข้าอาร์ติแฟกต์สาธารณะของ Plugin หน่วยความจำที่ใช้งานอยู่                                  |
| `bridge.followMemoryEvents`                | ค่าเริ่มต้น `true`                                 | รวมบันทึกเหตุการณ์ในโหมดบริดจ์                                             |
| `unsafeLocal.allowPrivateMemoryCoreAccess` | ค่าเริ่มต้น `false`                                | จำเป็นสำหรับการเรียกใช้การนำเข้า `unsafe-local`                                        |
| `unsafeLocal.paths`                        | ค่าเริ่มต้น `[]`                                   | พาธภายในเครื่องที่ระบุอย่างชัดเจนเพื่อนำเข้าในโหมด `unsafe-local`                         |
| `search.backend`                           | `shared` (ค่าเริ่มต้น), `local`                    |                                                                               |
| `search.corpus`                            | `wiki` (ค่าเริ่มต้น), `memory`, `all`              |                                                                               |
| `context.includeCompiledDigestPrompt`      | ค่าเริ่มต้น `false`                                | ต่อท้ายสแนปช็อตบทสรุปแบบกระชับของเอเจนต์ที่เลือกในส่วนหน่วยความจำของพรอมต์ |
| `render.createBacklinks`                   | ค่าเริ่มต้น `true`                                 | สร้างบล็อกที่เกี่ยวข้องแบบกำหนดผลลัพธ์ตายตัว                                         |
| `render.createDashboards`                  | ค่าเริ่มต้น `true`                                 | สร้างหน้าแดชบอร์ด                                                      |

### คลังแยกตามเอเจนต์

ตั้งค่า `vault.scope` เป็น `agent` เพื่อให้เอเจนต์ที่กำหนดค่าแต่ละรายมีวิกิแยกกัน
ในขอบเขตนี้ `vault.path` คือไดเรกทอรีแม่ และ OpenClaw จะต่อท้าย
ID เอเจนต์ที่ปรับให้อยู่ในรูปแบบมาตรฐาน:

```json5
{
  agents: {
    list: [{ id: "support" }, { id: "marketing" }],
  },
  plugins: {
    entries: {
      "memory-wiki": {
        enabled: true,
        config: {
          vaultMode: "bridge",
          vault: {
            scope: "agent",
            path: "~/.openclaw/wiki",
          },
          bridge: {
            enabled: true,
            readMemoryArtifacts: true,
          },
        },
      },
    },
  },
}
```

ค่านี้จะได้รับการแก้ไขเป็น `~/.openclaw/wiki/support` และ
`~/.openclaw/wiki/marketing` หากละเว้น `vault.path` ในขอบเขตเอเจนต์
ไดเรกทอรีแม่จะมีค่าเริ่มต้นเป็น `~/.openclaw/wiki` ดังนั้นเอเจนต์ `main` เริ่มต้นจึงยังคงใช้
พาธ `~/.openclaw/wiki/main` ที่มีอยู่

เครื่องมือเอเจนต์ บทสรุปพรอมต์ที่คอมไพล์ และส่วนเสริมวิกิที่เปิดให้ใช้งานผ่าน
`memory_search` / `memory_get` จะแก้ไขคลังจากบริบทเอเจนต์ที่ใช้งานอยู่
สำหรับการเรียก CLI และ Gateway ในการตั้งค่าที่มีเอเจนต์ที่กำหนดค่าไว้หลายราย ให้ระบุ
เอเจนต์อย่างชัดเจนด้วย `openclaw wiki --agent <agentId> ...` หรือ `agentId` ของคำขอ
Gateway เอเจนต์ที่กำหนดค่าไว้เพียงรายเดียวยังคงเป็นค่าเริ่มต้นเมื่อไม่ได้ระบุ ID

ในโหมดบริดจ์ การนำเข้าในขอบเขตเอเจนต์จะยอมรับอาร์ติแฟกต์หน่วยความจำสาธารณะเฉพาะเมื่อ
`agentIds` ของอาร์ติแฟกต์นั้นมีเอเจนต์ที่เลือกอยู่ อาร์ติแฟกต์ที่เอเจนต์อื่นเป็นเจ้าของ
ไม่มีข้อมูลเมตาความเป็นเจ้าของ หรือมีเจ้าของที่ไม่รู้จักจะถูกข้าม ขอบเขตส่วนกลาง
จะคงลักษณะการทำงานเดิมสำหรับอาร์ติแฟกต์ที่ใช้ร่วมกัน

<Warning>
การเปลี่ยน `vault.scope` จะไม่คัดลอกหรือแบ่งคลังที่มีอยู่ ในขอบเขตเอเจนต์
`vault.path` ที่กำหนดค่าไว้อย่างชัดเจนจะกลายเป็นไดเรกทอรีแม่ ดังนั้นควรย้ายหรือ
นำเข้าหน้าที่มีอยู่อย่างรอบคอบก่อนสลับเอเจนต์ที่ใช้งานจริง สำรองข้อมูล
คลังก่อน

คลังแยกตามเอเจนต์เป็นขอบเขตความรู้ภายในโพรเซสเดียวกัน ไม่ใช่ขอบเขตความปลอดภัย
ของระบบปฏิบัติการ Plugin และเครื่องมือที่ไม่ได้อยู่ในแซนด์บ็อกซ์ซึ่งเข้าถึงระบบไฟล์ของโฮสต์ได้
ยังคงอ่านไดเรกทอรีของเอเจนต์อื่นได้ ใช้[การแซนด์บ็อกซ์](/th/gateway/sandboxing) หรือ
[โปรไฟล์ Gateway แยกกัน](/th/gateway/multiple-gateways) เมื่อเอเจนต์ไม่ไว้วางใจ
ซึ่งกันและกัน
</Warning>

### ตัวอย่าง: QMD + โหมดบริดจ์

ใช้รูปแบบนี้เมื่อต้องการใช้ QMD สำหรับการเรียกคืน และ `memory-wiki` สำหรับเลเยอร์
ความรู้ที่มีการบำรุงรักษา แต่ละเลเยอร์จะยังคงมุ่งเน้นหน้าที่ของตนเอง: QMD ทำให้โน้ตดิบ การส่งออก
เซสชัน และคอลเลกชันเพิ่มเติมยังคงค้นหาได้ ขณะที่ `memory-wiki` คอมไพล์
เอนทิตี การอ้างสิทธิ์ แดชบอร์ด และหน้าต้นทางที่เสถียร

```json5
{
  memory: {
    backend: "qmd",
  },
  plugins: {
    entries: {
      "memory-wiki": {
        enabled: true,
        config: {
          vaultMode: "bridge",
          bridge: {
            enabled: true,
            readMemoryArtifacts: true,
            indexDreamReports: true,
            indexDailyNotes: true,
            indexMemoryRoot: true,
            followMemoryEvents: true,
          },
          search: {
            backend: "shared",
            corpus: "all",
          },
          context: {
            includeCompiledDigestPrompt: false,
          },
        },
      },
    },
  },
}
```

รูปแบบนี้ทำให้ QMD ยังคงควบคุมการเรียกคืน Active Memory, `memory-wiki` มุ่งเน้น
หน้าที่คอมไพล์และแดชบอร์ด และรูปแบบพรอมต์ไม่เปลี่ยนแปลงจนกว่าจะ
เปิดใช้พรอมต์บทสรุปที่คอมไพล์โดยเจตนา

`wiki okf import`, `wiki apply metadata`, `wiki unsafe-local import`,
`wiki chatgpt import` / `wiki chatgpt rollback` และชุดคำสั่งย่อย `wiki obsidian`
ฉบับเต็ม

## การรองรับ Obsidian

เมื่อ `vault.renderMode` เป็น `obsidian` Plugin จะเขียน
Markdown ที่เป็นมิตรกับ Obsidian และสามารถเลือกใช้ CLI ทางการ `obsidian` สำหรับตรวจสอบ
สถานะ ค้นหาคลัง เปิดหน้า เรียกใช้คำสั่ง และไปยัง
โน้ตรายวันได้ คุณลักษณะนี้เป็นทางเลือก วิกิยังคงทำงานในโหมดเนทีฟได้โดยไม่มี
Obsidian

คลังในขอบเขตเอเจนต์ยังคงใช้ Markdown ที่เป็นมิตรกับ Obsidian ได้ แต่การตรวจสอบ
การกำหนดค่าจะปฏิเสธ `obsidian.useOfficialCli: true` เมื่อใช้ร่วมกับ `vault.scope: "agent"`
การตั้งค่า `obsidian.vaultName` ปัจจุบันเป็นแบบส่วนกลางและไม่สามารถเลือก
คลัง Obsidian แยกกันสำหรับแต่ละเอเจนต์ได้ ให้ใช้เครื่องมือวิกิและการดำเนินการ CLI แทน
หรือคงวิกิที่ดำเนินการโดย Obsidian ไว้ในขอบเขตส่วนกลาง

## เวิร์กโฟลว์ที่แนะนำ

<Steps>
<Step title="คง Plugin หน่วยความจำที่ใช้งานอยู่ไว้สำหรับการเรียกคืน">
การเรียกคืน การเลื่อนระดับ และ Dreaming ยังคงอยู่ภายใต้การดูแลของแบ็กเอนด์หน่วยความจำที่กำหนดค่าไว้
</Step>
<Step title="เปิดใช้ memory-wiki">
เริ่มต้นด้วยโหมด `isolated` เว้นแต่ต้องการใช้โหมดบริดจ์อย่างชัดเจน
</Step>
<Step title="ใช้ wiki_search / wiki_get เมื่อแหล่งที่มามีความสำคัญ">
เลือกใช้เครื่องมือเหล่านี้แทน `memory_search` เมื่อต้องการการจัดอันดับเฉพาะสำหรับวิกิหรือโครงสร้างความเชื่อระดับหน้า
</Step>
<Step title="ใช้ wiki_apply สำหรับการสังเคราะห์แบบเฉพาะเจาะจงหรือการอัปเดตเมทาดาทา">
หลีกเลี่ยงการแก้ไขบล็อกที่ระบบจัดการและสร้างขึ้นด้วยตนเอง
</Step>
<Step title="เรียกใช้ wiki_lint หลังจากมีการเปลี่ยนแปลงที่สำคัญ">
ตรวจจับข้อขัดแย้ง คำถามที่ยังไม่มีคำตอบ และช่องว่างของแหล่งที่มา
</Step>
<Step title="เปิดใช้แดชบอร์ดเพื่อให้มองเห็นข้อมูลที่ล้าสมัยหรือข้อขัดแย้ง">
ตั้งค่า `render.createDashboards: true` (ค่าเริ่มต้น)
</Step>
</Steps>
