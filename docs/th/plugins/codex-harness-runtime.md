หน้านี้จัดทำเอกสารสัญญารันไทม์สำหรับเทิร์นของฮาร์เนส Codex สำหรับการตั้งค่าและ
การกำหนดเส้นทาง ให้เริ่มที่ [ฮาร์เนส Codex](/th/plugins/codex-harness) สำหรับฟิลด์การกำหนดค่า
ดู [ข้อมูลอ้างอิงฮาร์เนส Codex](/th/plugins/codex-harness-reference)

## ภาพรวม

โหมด Codex ไม่ใช่ PI ที่มีการเรียกโมเดลคนละแบบอยู่ข้างใต้ Codex เป็นเจ้าของส่วนที่มากกว่า
ของลูปโมเดลดั้งเดิม และ OpenClaw ปรับพื้นผิว plugin, เครื่องมือ, เซสชัน และ
การวินิจฉัยของตนรอบขอบเขตนั้น

OpenClaw ยังคงเป็นเจ้าของการกำหนดเส้นทางช่องทาง, ไฟล์เซสชัน, การส่งข้อความที่มองเห็นได้,
เครื่องมือแบบไดนามิกของ OpenClaw, การอนุมัติ, การส่งสื่อ และสำเนาทรานสคริปต์
Codex เป็นเจ้าของเธรดดั้งเดิมที่เป็นแหล่งจริง, ลูปโมเดลดั้งเดิม, การดำเนินต่อของเครื่องมือดั้งเดิม
และ Compaction ดั้งเดิม

## การผูกเธรดและการเปลี่ยนโมเดล

เมื่อเซสชัน OpenClaw ถูกแนบกับเธรด Codex ที่มีอยู่ เทิร์นถัดไปจะส่งโมเดล OpenAI
ที่เลือกอยู่ในปัจจุบัน, นโยบายการอนุมัติ, แซนด์บ็อกซ์ และระดับบริการ
ไปยัง app-server อีกครั้ง การสลับจาก `openai/gpt-5.5` เป็น
`openai/gpt-5.2` จะคงการผูกเธรดไว้ แต่ขอให้ Codex ดำเนินต่อด้วย
โมเดลที่เลือกใหม่

## คำตอบที่มองเห็นได้และ Heartbeat

เมื่อเทิร์นจากแชตต้นทางทำงานผ่านฮาร์เนส Codex คำตอบที่มองเห็นได้จะใช้
เครื่องมือ `message` ของ OpenClaw เป็นค่าเริ่มต้น หากการติดตั้งไม่ได้กำหนดค่า
`messages.visibleReplies` ไว้อย่างชัดเจน เอเจนต์ยังสามารถจบเทิร์น Codex ของตนแบบส่วนตัวได้
โดยจะโพสต์ไปยังช่องทางก็ต่อเมื่อเรียก `message(action="send")` เท่านั้น ตั้งค่า
`messages.visibleReplies: "automatic"` เพื่อคงคำตอบสุดท้ายของแชตโดยตรงไว้บน
เส้นทางการส่งอัตโนมัติแบบเดิม

เทิร์น Heartbeat ของ Codex ยังได้รับ `heartbeat_respond` ในแค็ตตาล็อกเครื่องมือ OpenClaw
ที่ค้นหาได้เป็นค่าเริ่มต้น เพื่อให้เอเจนต์บันทึกได้ว่าการปลุกควรเงียบต่อไป
หรือควรแจ้งเตือน โดยไม่ต้องเข้ารหัสโฟลว์ควบคุมนั้นในข้อความสุดท้าย

คำแนะนำด้านความริเริ่มเฉพาะ Heartbeat จะถูกส่งเป็นคำสั่งสำหรับนักพัฒนา
ในโหมดการทำงานร่วมกันของ Codex บนเทิร์น Heartbeat นั้นเอง เทิร์นแชตปกติจะคืนค่า
โหมด Codex Default แทนที่จะพกปรัชญา Heartbeat ไว้ในพรอมป์รันไทม์ปกติ

## ขอบเขตของฮุก

ฮาร์เนส Codex มีฮุกสามชั้น:

| ชั้น                                  | เจ้าของ                    | วัตถุประสงค์                                                          |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| ฮุก plugin ของ OpenClaw               | OpenClaw                 | ความเข้ากันได้ของผลิตภัณฑ์/plugin ระหว่างฮาร์เนส PI และ Codex       |
| มิดเดิลแวร์ส่วนขยาย app-server ของ Codex | plugin ที่มากับ OpenClaw | พฤติกรรมอะแดปเตอร์รายเทิร์นรอบเครื่องมือแบบไดนามิกของ OpenClaw     |
| ฮุกดั้งเดิมของ Codex                  | Codex                    | วงจรชีวิตระดับต่ำของ Codex และนโยบายเครื่องมือดั้งเดิมจากการกำหนดค่า Codex |

OpenClaw ไม่ใช้ไฟล์ `hooks.json` ระดับโปรเจ็กต์หรือระดับโกลบอลของ Codex เพื่อกำหนดเส้นทาง
พฤติกรรม plugin ของ OpenClaw สำหรับสะพานเครื่องมือดั้งเดิมและสิทธิ์ที่รองรับ
OpenClaw จะฉีดการกำหนดค่า Codex รายเธรดสำหรับ `PreToolUse`, `PostToolUse`,
`PermissionRequest` และ `Stop`

เมื่อเปิดใช้การอนุมัติของ app-server ของ Codex ซึ่งหมายความว่า `approvalPolicy` ไม่ใช่
`"never"` การกำหนดค่าฮุกดั้งเดิมที่ฉีดเป็นค่าเริ่มต้นจะละเว้น `PermissionRequest` เพื่อให้
ผู้ตรวจสอบ app-server ของ Codex และสะพานการอนุมัติของ OpenClaw จัดการการยกระดับจริง
หลังการตรวจสอบ ผู้ปฏิบัติการสามารถเพิ่ม `permission_request` ลงใน
`nativeHookRelay.events` อย่างชัดเจนได้เมื่อจำเป็นต้องใช้รีเลย์ความเข้ากันได้

ฮุก Codex อื่น ๆ เช่น `SessionStart` และ `UserPromptSubmit` ยังคงเป็น
การควบคุมระดับ Codex ฮุกเหล่านี้ไม่ได้ถูกเปิดเผยเป็นฮุก plugin ของ OpenClaw ในสัญญา v1

สำหรับเครื่องมือแบบไดนามิกของ OpenClaw นั้น OpenClaw จะเรียกใช้เครื่องมือหลังจาก Codex ขอ
การเรียก ดังนั้น OpenClaw จึงยิงพฤติกรรม plugin และมิดเดิลแวร์ที่ตนเป็นเจ้าของใน
อะแดปเตอร์ฮาร์เนส สำหรับเครื่องมือดั้งเดิมของ Codex นั้น Codex เป็นเจ้าของบันทึกเครื่องมือที่เป็นแหล่งจริง
OpenClaw สามารถทำสำเนาเหตุการณ์ที่เลือกได้ แต่ไม่สามารถเขียนเธรด Codex ดั้งเดิมใหม่
เว้นแต่ Codex จะเปิดเผยการดำเนินการนั้นผ่าน app-server หรือคอลแบ็กฮุกดั้งเดิม

การฉายภาพ Compaction และวงจรชีวิต LLM มาจากการแจ้งเตือนของ app-server ของ Codex
และสถานะอะแดปเตอร์ของ OpenClaw ไม่ใช่คำสั่งฮุกดั้งเดิมของ Codex
เหตุการณ์ `before_compaction`, `after_compaction`, `llm_input` และ
`llm_output` ของ OpenClaw เป็นการสังเกตระดับอะแดปเตอร์ ไม่ใช่การจับภาพแบบไบต์ต่อไบต์
ของคำขอภายในหรือเพย์โหลด Compaction ของ Codex

การแจ้งเตือน app-server `hook/started` และ `hook/completed` ดั้งเดิมของ Codex
จะถูกฉายเป็นเหตุการณ์เอเจนต์ `codex_app_server.hook` สำหรับเส้นทางการทำงานและการดีบัก
เหตุการณ์เหล่านี้ไม่เรียกฮุก plugin ของ OpenClaw

## สิทธิ์ดั้งเดิมและ MCP elicitations

สำหรับ `PermissionRequest` นั้น OpenClaw จะคืนเฉพาะการตัดสินใจอนุญาตหรือปฏิเสธอย่างชัดเจน
เมื่อมีการตัดสินโดยนโยบาย ผลลัพธ์ที่ไม่มีการตัดสินใจไม่ใช่การอนุญาต Codex ถือว่ามันเป็น
ไม่มีการตัดสินใจจากฮุกและปล่อยผ่านไปยังเส้นทาง guardian หรือการอนุมัติของผู้ใช้ของตนเอง

โหมดการอนุมัติของ app-server ของ Codex จะละเว้นฮุกดั้งเดิมนี้เป็นค่าเริ่มต้น พฤติกรรมนี้
มีผลเมื่อ `permission_request` ถูกระบุไว้อย่างชัดเจนใน
`nativeHookRelay.events` หรือเมื่อรันไทม์ความเข้ากันได้ติดตั้งไว้

เมื่อผู้ปฏิบัติงานเลือก `allow-always` สำหรับคำขอสิทธิ์แบบเนทีฟของ Codex
OpenClaw จะจดจำลายนิ้วมือที่ตรงกันของผู้ให้บริการ/เซสชัน/อินพุตเครื่องมือ/cwd นั้น
ไว้ภายในหน้าต่างเซสชันที่มีขอบเขต การตัดสินใจที่จดจำไว้นี้ตั้งใจให้ใช้เฉพาะแบบตรงกันทุกประการเท่านั้น:
คำสั่ง อาร์กิวเมนต์ เพย์โหลดเครื่องมือ หรือ cwd ที่เปลี่ยนไปจะสร้างการอนุมัติใหม่

การร้องขออนุมัติเครื่องมือ Codex MCP จะถูกส่งผ่านโฟลว์การอนุมัติ Plugin ของ
OpenClaw เมื่อ Codex ทำเครื่องหมาย `_meta.codex_approval_kind` เป็น
`"mcp_tool_call"` พรอมป์ `request_user_input` ของ Codex จะถูกส่งกลับไปยังแชตต้นทาง
และข้อความติดตามผลลำดับถัดไปในคิวจะตอบคำขอเซิร์ฟเวอร์เนทีฟนั้น
แทนที่จะถูกนำทางเป็นบริบทเพิ่มเติม คำขอร้องขออื่น ๆ ของ MCP จะล้มเหลวแบบปิด

## การนำทางคิว

การนำทางคิวขณะรันอยู่แมปกับ `turn/steer` ของเซิร์ฟเวอร์แอป Codex ด้วยค่าเริ่มต้น
`messages.queue.mode: "steer"` OpenClaw จะรวมข้อความแชตในคิวสำหรับช่วงเวลานิ่งที่กำหนดค่าไว้
แล้วส่งเป็นคำขอ `turn/steer` เดียวตามลำดับที่มาถึง โหมด `queue` แบบเดิมจะส่งคำขอ
`turn/steer` แยกกัน

เทิร์นรีวิว Codex และเทิร์น Compaction แบบแมนนวลอาจปฏิเสธการนำทางในเทิร์นเดียวกันได้
ในกรณีนั้น OpenClaw จะใช้คิวติดตามผลเมื่อโหมดที่เลือกอนุญาตให้ย้อนกลับไปใช้ได้
ดู [คิวการนำทาง](/th/concepts/queue-steering)

## การอัปโหลดข้อเสนอแนะ Codex

เมื่อ `/diagnostics [note]` ได้รับอนุมัติสำหรับเซสชันที่ใช้ฮาร์เนส Codex แบบเนทีฟ
OpenClaw จะเรียก `feedback/upload` ของเซิร์ฟเวอร์แอป Codex สำหรับเธรด Codex ที่เกี่ยวข้องด้วย
การอัปโหลดจะขอให้เซิร์ฟเวอร์แอปรวมบันทึกสำหรับแต่ละเธรดที่ระบุ
และเธรดย่อย Codex ที่ถูกสร้างขึ้นเมื่อมี

การอัปโหลดจะผ่านเส้นทางข้อเสนอแนะปกติของ Codex ไปยังเซิร์ฟเวอร์ OpenAI หากข้อเสนอแนะ Codex
ถูกปิดใช้งานในเซิร์ฟเวอร์แอปนั้น คำสั่งจะคืนข้อผิดพลาดของเซิร์ฟเวอร์แอป
การตอบกลับการวินิจฉัยที่เสร็จสมบูรณ์จะแสดงรายการช่องทาง รหัสเซสชัน OpenClaw
รหัสเธรด Codex และคำสั่ง `codex resume <thread-id>` ในเครื่องสำหรับเธรด
ที่ถูกส่งไป

หากคุณปฏิเสธหรือเพิกเฉยต่อการอนุมัติ OpenClaw จะไม่พิมพ์รหัส Codex เหล่านั้น
และจะไม่ส่งข้อเสนอแนะ Codex การอัปโหลดไม่ได้แทนที่การส่งออกการวินิจฉัย Gateway
ในเครื่อง ดู [การส่งออกการวินิจฉัย](/th/gateway/diagnostics) สำหรับพฤติกรรมด้านการอนุมัติ
ความเป็นส่วนตัว บันเดิลในเครื่อง และแชตกลุ่ม

ใช้ `/codex diagnostics [note]` เฉพาะเมื่อคุณต้องการอัปโหลดข้อเสนอแนะ Codex
สำหรับเธรดที่แนบอยู่ในปัจจุบันโดยเฉพาะ โดยไม่มีบันเดิลการวินิจฉัย Gateway แบบเต็ม

## Compaction และมิเรอร์ทรานสคริปต์

เมื่อโมเดลที่เลือกใช้ฮาร์เนส Codex การทำ Compaction เธรดแบบเนทีฟจะถูกมอบหมายให้เซิร์ฟเวอร์แอป Codex
OpenClaw จะเก็บมิเรอร์ทรานสคริปต์ไว้สำหรับประวัติช่องทาง การค้นหา `/new`, `/reset`
และการสลับโมเดลหรือฮาร์เนสในอนาคต

มิเรอร์ประกอบด้วยพรอมป์ของผู้ใช้ ข้อความผู้ช่วยสุดท้าย และบันทึกการให้เหตุผลหรือแผน Codex
แบบเบาเมื่อเซิร์ฟเวอร์แอปส่งออกมา ปัจจุบัน OpenClaw บันทึกเฉพาะสัญญาณเริ่มต้นและเสร็จสิ้น
ของ Compaction แบบเนทีฟเท่านั้น ยังไม่เปิดเผยสรุป Compaction ที่มนุษย์อ่านได้
หรือรายการที่ตรวจสอบได้ว่า Codex เก็บรายการใดไว้หลัง Compaction

เนื่องจาก Codex เป็นเจ้าของเธรดเนทีฟตามบัญญัติ `tool_result_persist` จึงยังไม่เขียนบันทึก
ผลลัพธ์เครื่องมือแบบเนทีฟของ Codex ใหม่ในปัจจุบัน โดยจะใช้เฉพาะเมื่อ OpenClaw กำลังเขียน
ผลลัพธ์เครื่องมือทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ

## สื่อและการส่งมอบ

OpenClaw ยังคงเป็นเจ้าของการส่งมอบสื่อและการเลือกผู้ให้บริการสื่อ รูปภาพ
วิดีโอ เพลง PDF, TTS และการทำความเข้าใจสื่อใช้การตั้งค่าผู้ให้บริการ/โมเดลที่ตรงกัน
เช่น `agents.defaults.imageGenerationModel`, `videoGenerationModel`,
`pdfModel` และ `messages.tts`

ข้อความ รูปภาพ วิดีโอ เพลง TTS การอนุมัติ และเอาต์พุตเครื่องมือรับส่งข้อความยังคงผ่าน
เส้นทางการส่งมอบปกติของ OpenClaw การสร้างสื่อไม่ต้องใช้ PI
เมื่อ Codex ส่งออกรายการสร้างรูปภาพแบบเนทีฟที่มี `savedPath` OpenClaw จะส่งต่อไฟล์นั้นทุกประการ
ผ่านเส้นทางสื่อตอบกลับปกติ แม้ว่าเทิร์น Codex จะไม่มีข้อความผู้ช่วยก็ตาม
