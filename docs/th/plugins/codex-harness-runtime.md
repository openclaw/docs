หน้านี้บันทึกสัญญารันไทม์สำหรับเทิร์นของ Codex harness สำหรับการตั้งค่าและ
การกำหนดเส้นทาง ให้เริ่มที่ [Codex harness](/th/plugins/codex-harness) สำหรับฟิลด์คอนฟิก
ดู [ข้อมูลอ้างอิง Codex harness](/th/plugins/codex-harness-reference)

## ภาพรวม

โหมด Codex ไม่ใช่ OpenClaw ที่มีการเรียกโมเดลอีกแบบหนึ่งอยู่ข้างใต้ Codex เป็นเจ้าของส่วนที่มากกว่า
ของลูปโมเดลแบบเนทีฟ และ OpenClaw ปรับพื้นผิว Plugin, เครื่องมือ, เซสชัน และ
การวินิจฉัยของตัวเองรอบขอบเขตนั้น

OpenClaw ยังคงเป็นเจ้าของการกำหนดเส้นทางช่องทาง, ไฟล์เซสชัน, การส่งข้อความที่มองเห็นได้,
เครื่องมือไดนามิกของ OpenClaw, การอนุมัติ, การส่งสื่อ และมิเรอร์ทรานสคริปต์
Codex เป็นเจ้าของเธรดเนทีฟที่เป็นมาตรฐาน, ลูปโมเดลเนทีฟ, การดำเนินต่อของเครื่องมือเนทีฟ
และ Compaction แบบเนทีฟ

การกำหนดเส้นทางพรอมป์จะตามรันไทม์ที่เลือก ไม่ใช่แค่สตริงผู้ให้บริการ เทิร์น
Codex แบบเนทีฟจะได้รับคำสั่งสำหรับนักพัฒนาของแอปเซิร์ฟเวอร์ Codex ในขณะที่
เส้นทางความเข้ากันได้ของ OpenClaw ที่ระบุชัดเจนจะยังคงใช้พรอมป์ระบบ OpenClaw ปกติ แม้
จะใช้การยืนยันตัวตนหรือการขนส่ง OpenAI แบบ Codex ก็ตาม

Codex แบบเนทีฟจะเก็บคำสั่งพื้นฐาน/โมเดลและพฤติกรรมเอกสารโปรเจกต์ที่ Codex เป็นเจ้าของ
ตามคอนฟิกเธรด Codex ที่ใช้งานอยู่ OpenClaw เริ่มและดำเนินเธรด
Codex แบบเนทีฟต่อโดยปิดบุคลิกภาพในตัวของ Codex เพื่อให้ไฟล์บุคลิกภาพ
ของเวิร์กสเปซและตัวตนเอเจนต์ OpenClaw ยังคงเป็นแหล่งอ้างอิงหลัก การรัน OpenClaw
แบบเบายังคงรักษาการระงับเอกสารโปรเจกต์ที่มีอยู่เดิม คำสั่งสำหรับนักพัฒนา
ของ OpenClaw ครอบคลุมข้อกังวลด้านรันไทม์ของ OpenClaw เช่น การส่งผ่านช่องทางต้นทาง,
เครื่องมือไดนามิกของ OpenClaw, การมอบหมาย ACP, บริบทอะแดปเตอร์ และไฟล์โปรไฟล์
เวิร์กสเปซเอเจนต์ที่ใช้งานอยู่ แคตตาล็อก Skills ของ OpenClaw และตัวชี้
`MEMORY.md` ที่กำหนดเส้นทางผ่านเครื่องมือจะถูกฉายเป็นคำสั่งสำหรับนักพัฒนา
ด้านการทำงานร่วมกันที่จำกัดตามเทิร์นสำหรับ Codex แบบเนทีฟ เนื้อหา `BOOTSTRAP.md`
ที่ใช้งานอยู่และการฉีด fallback ของ `MEMORY.md` แบบเต็มยังคงใช้บริบทอ้างอิงอินพุตของเทิร์น

## การผูกเธรดและการเปลี่ยนโมเดล

เมื่อเซสชัน OpenClaw ถูกแนบกับเธรด Codex ที่มีอยู่ เทิร์นถัดไป
จะส่งโมเดล OpenAI, นโยบายการอนุมัติ, แซนด์บ็อกซ์ และระดับบริการที่เลือกอยู่ในปัจจุบัน
ไปยังแอปเซิร์ฟเวอร์อีกครั้ง การสลับจาก `openai/gpt-5.5` เป็น
`openai/gpt-5.2` จะคงการผูกเธรดไว้ แต่ขอให้ Codex ดำเนินต่อด้วย
โมเดลที่เลือกใหม่

## การตอบกลับที่มองเห็นได้และ Heartbeat

เมื่อเทิร์นแชทโดยตรง/จากต้นทางรันผ่าน Codex harness การตอบกลับที่มองเห็นได้
จะมีค่าเริ่มต้นเป็นการส่งผู้ช่วยขั้นสุดท้ายอัตโนมัติสำหรับพื้นผิว WebChat ภายใน
สิ่งนี้ทำให้ Codex สอดคล้องกับสัญญาพรอมป์ของ Pi harness: เอเจนต์ตอบกลับ
ตามปกติ และ OpenClaw โพสต์ข้อความสุดท้ายไปยังการสนทนาต้นทาง ตั้งค่า
`messages.visibleReplies: "message_tool"` เมื่อแชทโดยตรง/จากต้นทางควร
ตั้งใจเก็บข้อความผู้ช่วยขั้นสุดท้ายเป็นส่วนตัว เว้นแต่เอเจนต์จะเรียก
`message(action="send")`

เทิร์น Heartbeat ของ Codex ยังได้รับ `heartbeat_respond` ในแคตตาล็อกเครื่องมือ
OpenClaw ที่ค้นหาได้โดยค่าเริ่มต้น เพื่อให้เอเจนต์บันทึกได้ว่าการปลุกควร
เงียบต่อไปหรือแจ้งเตือน โดยไม่ต้องเข้ารหัสโฟลว์ควบคุมนั้นในข้อความสุดท้าย

คำแนะนำด้าน initiative เฉพาะ Heartbeat จะถูกส่งเป็นคำสั่งสำหรับนักพัฒนา
โหมดการทำงานร่วมกันของ Codex ในเทิร์น Heartbeat นั้นเอง เทิร์นแชทปกติจะคืนค่า
โหมด Codex Default แทนการพกปรัชญา Heartbeat ไว้ในพรอมป์รันไทม์ปกติ
เมื่อมี `HEARTBEAT.md` ที่ไม่ว่าง คำสั่งโหมดการทำงานร่วมกันของ Heartbeat
จะชี้ Codex ไปที่ไฟล์แทนการฝังเนื้อหาไว้โดยตรง

## ขอบเขตของ hook

Codex harness มีชั้น hook สามชั้น:

| ชั้น                                  | เจ้าของ                   | วัตถุประสงค์                                                        |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| hook ของ Plugin OpenClaw              | OpenClaw                 | ความเข้ากันได้ของผลิตภัณฑ์/Plugin ระหว่าง OpenClaw และ Codex harness |
| มิดเดิลแวร์ส่วนขยายแอปเซิร์ฟเวอร์ Codex | Plugin ที่บันเดิลกับ OpenClaw | พฤติกรรมอะแดปเตอร์ต่อเทิร์นรอบเครื่องมือไดนามิกของ OpenClaw        |
| hook เนทีฟของ Codex                   | Codex                    | วงจรชีวิต Codex ระดับต่ำและนโยบายเครื่องมือเนทีฟจากคอนฟิก Codex     |

OpenClaw ไม่ใช้ไฟล์ `hooks.json` ของโปรเจกต์หรือของ Codex ระดับโกลบอลเพื่อกำหนดเส้นทาง
พฤติกรรม Plugin ของ OpenClaw สำหรับเครื่องมือเนทีฟและบริดจ์สิทธิ์ที่รองรับ
OpenClaw จะฉีดคอนฟิก Codex ต่อเธรดสำหรับ `PreToolUse`, `PostToolUse`,
`PermissionRequest` และ `Stop`

เมื่อเปิดใช้การอนุมัติของแอปเซิร์ฟเวอร์ Codex ซึ่งหมายถึง `approvalPolicy` ไม่ใช่
`"never"` คอนฟิก hook เนทีฟที่ฉีดโดยค่าเริ่มต้นจะละ `PermissionRequest` เพื่อให้
ผู้ตรวจสอบของแอปเซิร์ฟเวอร์ Codex และบริดจ์การอนุมัติของ OpenClaw จัดการ
การยกระดับจริงหลังการตรวจสอบ ผู้ปฏิบัติงานสามารถเพิ่ม `permission_request` ไปยัง
`nativeHookRelay.events` อย่างชัดเจนเมื่อจำเป็นต้องใช้รีเลย์ความเข้ากันได้

hook อื่นของ Codex เช่น `SessionStart` และ `UserPromptSubmit` ยังคงเป็น
การควบคุมระดับ Codex สิ่งเหล่านี้ไม่ถูกเปิดเผยเป็น hook ของ Plugin OpenClaw ในสัญญา v1

สำหรับเครื่องมือไดนามิกของ OpenClaw นั้น OpenClaw จะดำเนินเครื่องมือหลังจาก Codex ขอ
การเรียก ดังนั้น OpenClaw จึงยิงพฤติกรรม Plugin และมิดเดิลแวร์ที่ตัวเองเป็นเจ้าของใน
อะแดปเตอร์ harness สำหรับเครื่องมือเนทีฟของ Codex นั้น Codex เป็นเจ้าของระเบียนเครื่องมือมาตรฐาน
OpenClaw สามารถมิเรอร์เหตุการณ์ที่เลือกได้ แต่ไม่สามารถเขียนเธรด Codex
แบบเนทีฟใหม่ได้ เว้นแต่ Codex จะเปิดเผยการดำเนินการนั้นผ่านแอปเซิร์ฟเวอร์หรือคอลแบ็ก
hook เนทีฟ

เหตุการณ์ `PreToolUse` โหมดรายงานของแอปเซิร์ฟเวอร์ Codex จะเลื่อนคำขออนุมัติของ Plugin
ไปยังการอนุมัติของแอปเซิร์ฟเวอร์ที่ตรงกัน หาก hook `before_tool_call` ของ OpenClaw
ส่งคืน `requireApproval` ขณะที่เพย์โหลดเนทีฟตั้งค่าโหมดการอนุมัติแบบรายงาน
(`openclaw_approval_mode` เป็น `"report"`) รีเลย์ hook เนทีฟจะบันทึก
ข้อกำหนดการอนุมัติของ Plugin และไม่ส่งคืนการตัดสินใจเนทีฟ เมื่อ Codex ส่ง
คำขออนุมัติของแอปเซิร์ฟเวอร์สำหรับการใช้เครื่องมือเดียวกัน OpenClaw จะเปิดพรอมป์
การอนุมัติของ Plugin และแมปการตัดสินใจกลับไปยัง Codex เหตุการณ์ `PermissionRequest`
ของ Codex เป็นเส้นทางการอนุมัติแยกต่างหาก และยังสามารถกำหนดเส้นทางผ่านการอนุมัติ
ของ OpenClaw ได้เมื่อรันไทม์ถูกตั้งค่าสำหรับบริดจ์นั้น

การแจ้งเตือนรายการของแอปเซิร์ฟเวอร์ Codex ยังให้ observation ของ `after_tool_call`
แบบอะซิงก์สำหรับการเสร็จสิ้นของเครื่องมือเนทีฟที่ยังไม่ได้ถูกครอบคลุมโดยรีเลย์
`PostToolUse` เนทีฟ observation เหล่านี้มีไว้สำหรับ telemetry และความเข้ากันได้
ของ Plugin เท่านั้น ไม่สามารถบล็อก หน่วงเวลา หรือเปลี่ยนแปลงการเรียกเครื่องมือเนทีฟได้

การฉาย Compaction และวงจรชีวิต LLM มาจากการแจ้งเตือนของแอปเซิร์ฟเวอร์ Codex
และสถานะอะแดปเตอร์ OpenClaw ไม่ใช่คำสั่ง hook เนทีฟของ Codex
เหตุการณ์ `before_compaction`, `after_compaction`, `llm_input` และ
`llm_output` ของ OpenClaw เป็น observation ระดับอะแดปเตอร์ ไม่ใช่การจับ
คำขอภายในหรือเพย์โหลด Compaction ของ Codex แบบไบต์ต่อไบต์

การแจ้งเตือนแอปเซิร์ฟเวอร์ `hook/started` และ `hook/completed` แบบเนทีฟของ Codex
จะถูกฉายเป็นเหตุการณ์เอเจนต์ `codex_app_server.hook` สำหรับ trajectory และการดีบัก
สิ่งเหล่านี้ไม่เรียก hook ของ Plugin OpenClaw

## สิทธิ์แบบเนทีฟและการขอข้อมูล MCP

สำหรับ `PermissionRequest` OpenClaw จะส่งคืนเฉพาะการตัดสินใจอนุญาตหรือปฏิเสธอย่างชัดเจน
เมื่อมีการตัดสินโดยนโยบาย ผลลัพธ์ที่ไม่มีการตัดสินใจไม่ใช่การอนุญาต Codex ถือว่าเป็นการไม่มี
การตัดสินใจจากฮุก และปล่อยให้ตกต่อไปยังเส้นทางตัวพิทักษ์หรือการอนุมัติจากผู้ใช้ของตนเอง

โหมดการอนุมัติของ Codex app-server จะละเว้นฮุกแบบเนทีฟนี้โดยค่าเริ่มต้น พฤติกรรมนี้
มีผลเมื่อ `permission_request` ถูกระบุไว้อย่างชัดเจนใน
`nativeHookRelay.events` หรือเมื่อรันไทม์เพื่อความเข้ากันได้ติดตั้งฮุกนี้

เมื่อผู้ปฏิบัติการเลือก `allow-always` สำหรับคำขอสิทธิ์แบบเนทีฟของ Codex
OpenClaw จะจดจำลายนิ้วมือผู้ให้บริการ/เซสชัน/อินพุตเครื่องมือ/cwd ที่ตรงกันนั้นสำหรับ
หน้าต่างเซสชันที่มีขอบเขต การตัดสินใจที่จดจำไว้นั้นตั้งใจให้ตรงกันแบบเป๊ะเท่านั้น:
คำสั่ง อาร์กิวเมนต์ เพย์โหลดเครื่องมือ หรือ cwd ที่เปลี่ยนไปจะสร้าง
การอนุมัติใหม่

การขออนุมัติเครื่องมือ Codex MCP ถูกกำหนดเส้นทางผ่านโฟลว์การอนุมัติ
Plugin ของ OpenClaw เมื่อ Codex ระบุ `_meta.codex_approval_kind` เป็น
`"mcp_tool_call"` พรอมป์ Codex `request_user_input` จะถูกส่งกลับไปยังแชต
ต้นทาง และข้อความติดตามผลถัดไปในคิวจะตอบคำขอเซิร์ฟเวอร์แบบเนทีฟนั้น
แทนที่จะถูกนำทางเป็นบริบทเพิ่มเติม คำขอข้อมูล MCP อื่น ๆ
จะล้มเหลวแบบปิด

สำหรับโฟลว์การอนุมัติ Plugin ทั่วไปที่พาพรอมป์เหล่านี้ โปรดดู
[คำขอสิทธิ์ Plugin](/th/plugins/plugin-permission-requests)

## การนำทางคิว

การนำทางคิวของรันที่กำลังทำงานแมปเข้ากับ Codex app-server `turn/steer` ด้วย
ค่าเริ่มต้น `messages.queue.mode: "steer"` OpenClaw จะรวมกลุ่มข้อความแชต
โหมด steer สำหรับหน้าต่างเงียบที่กำหนดค่าไว้ และส่งเป็นคำขอ `turn/steer`
เดียวตามลำดับการมาถึง

การรีวิวของ Codex และเทิร์น Compaction แบบแมนนวลอาจปฏิเสธการควบคุมทิศทางในเทิร์นเดียวกันได้ ใน
กรณีนั้น OpenClaw จะรอให้การรันที่ใช้งานอยู่เสร็จสิ้นก่อนเริ่มพรอมต์
ใช้ `/queue followup` หรือ `/queue collect` เมื่อข้อความควรเข้าคิวเป็นค่าเริ่มต้น
แทนการควบคุมทิศทาง ดู [คิวการควบคุมทิศทาง](/th/concepts/queue-steering)

## การอัปโหลดข้อเสนอแนะของ Codex

เมื่อ `/diagnostics [note]` ได้รับการอนุมัติสำหรับเซสชันที่ใช้ harness ของ Codex
แบบเนทีฟ OpenClaw จะเรียก Codex app-server `feedback/upload` สำหรับเธรด Codex
ที่เกี่ยวข้องด้วย การอัปโหลดจะขอให้ app-server รวมบันทึกสำหรับแต่ละเธรดที่ระบุ
และเธรดย่อย Codex ที่ถูกสร้างขึ้น เมื่อมีให้ใช้งาน

การอัปโหลดจะผ่านเส้นทางข้อเสนอแนะปกติของ Codex ไปยังเซิร์ฟเวอร์ OpenAI หากปิดใช้
ข้อเสนอแนะของ Codex ใน app-server นั้น คำสั่งจะส่งคืนข้อผิดพลาดของ app-server
คำตอบ diagnostics ที่เสร็จสมบูรณ์จะแสดงรายการช่องทาง, id เซสชัน OpenClaw,
id เธรด Codex และคำสั่ง `codex resume <thread-id>` ในเครื่องสำหรับเธรด
ที่ถูกส่งไป

หากคุณปฏิเสธหรือเพิกเฉยต่อการอนุมัติ OpenClaw จะไม่พิมพ์ id Codex เหล่านั้น
และจะไม่ส่งข้อเสนอแนะของ Codex การอัปโหลดไม่ได้แทนที่การส่งออก diagnostics
ของ Gateway ในเครื่อง ดู [การส่งออก diagnostics](/th/gateway/diagnostics) สำหรับ
การอนุมัติ ความเป็นส่วนตัว บันเดิลในเครื่อง และพฤติกรรมของแชทกลุ่ม

ใช้ `/codex diagnostics [note]` เฉพาะเมื่อคุณต้องการอัปโหลดข้อเสนอแนะของ Codex
สำหรับเธรดที่แนบอยู่ในปัจจุบันโดยเฉพาะ โดยไม่มีบันเดิล diagnostics ของ Gateway
แบบเต็ม

## Compaction และมิเรอร์ทรานสคริปต์

เมื่อโมเดลที่เลือกใช้ harness ของ Codex การทำ Compaction เธรดแบบเนทีฟเป็นของ
Codex app-server OpenClaw จะไม่รัน Compaction ก่อนการตรวจสอบสำหรับเทิร์น Codex,
จะไม่แทนที่ Compaction ของ Codex ด้วย Compaction ของ context-engine และจะไม่
ถอยกลับไปใช้การสรุปของ OpenClaw หรือ OpenAI สาธารณะเมื่อไม่สามารถเริ่ม
Compaction ของ Codex แบบเนทีฟได้ OpenClaw เก็บมิเรอร์ทรานสคริปต์ไว้สำหรับ
ประวัติช่องทาง การค้นหา `/new`, `/reset` และการสลับโมเดลหรือ harness ในอนาคต

คำขอ Compaction แบบชัดเจน เช่น `/compact` หรือการดำเนินการ compact แบบแมนนวล
ที่ Plugin ร้องขอ จะเริ่ม Compaction ของ Codex แบบเนทีฟด้วย `thread/compact/start`
OpenClaw จะเปิดคำขอและลีสของไคลเอนต์ที่ใช้ร่วมกันไว้จนกว่า Codex จะปล่อยรายการ
เสร็จสิ้น `contextCompaction` ที่ตรงกัน แล้วจึงรายงานเทิร์น Compaction ว่าเสร็จสมบูรณ์
หากเทิร์นปลายทางนั้นเกินเวลาหมดเวลาของ Compaction ที่กำหนดไว้ OpenClaw จะขอ
ขัดจังหวะเทิร์นแบบเนทีฟ ลีสและรั้ว Compaction ต่อเธรดจะยังคงถูกถือไว้จนกว่า
Codex จะรายงานสถานะปลายทางหรือยืนยัน interrupt RPC หาก Codex ไม่ยืนยันภายใน
ช่วงผ่อนผันการขัดจังหวะ OpenClaw จะเลิกใช้การเชื่อมต่อก่อนปล่อยรั้ว การเชื่อมต่อ
ระยะไกลจะถอดการผูกเธรดที่ตรงกันออกด้วย เพื่อให้งานภายหลังไม่สามารถทับซ้อนกับ
เทิร์นระยะไกลที่ยังไม่ได้ยืนยันได้ เทิร์นอื่นบนการเชื่อมต่อที่ถูกเลิกใช้จะล้มเหลว
และสามารถลองใหม่บนไคลเอนต์ใหม่ได้ การปิดไคลเอนต์ การยกเลิกคำขอ หรือเทิร์น
Compaction ที่ล้มเหลวจะส่งคืนการดำเนินการที่ล้มเหลว

เมื่อ context engine ร้องขอการฉายภาพเพื่อบูตสแตรปเธรด Codex OpenClaw จะฉาย
ชื่อและ id ของการเรียกเครื่องมือ รูปแบบอินพุต และเนื้อหาผลลัพธ์เครื่องมือที่ถูกปกปิด
เข้าไปในเธรด Codex ใหม่ โดยจะไม่คัดลอกค่าดิบของอาร์กิวเมนต์การเรียกเครื่องมือ
เข้าไปในการฉายภาพนั้น

มิเรอร์มีพรอมต์ของผู้ใช้ ข้อความผู้ช่วยขั้นสุดท้าย และเรคคอร์ดการให้เหตุผลหรือแผน
ของ Codex แบบเบาเมื่อ app-server ปล่อยออกมา OpenClaw บันทึกการเริ่ม Compaction
แบบเนทีฟและสถานะปลายทาง แต่จะไม่เปิดเผยสรุป Compaction ที่มนุษย์อ่านได้ หรือ
รายการที่ตรวจสอบย้อนหลังได้ว่า Codex เก็บรายการใดไว้หลัง Compaction

เนื่องจาก Codex เป็นเจ้าของเธรดเนทีฟตามบัญญัติ `tool_result_persist` จึงยังไม่
เขียนเรคคอร์ดผลลัพธ์เครื่องมือแบบ Codex-native ใหม่ในปัจจุบัน โดยจะมีผลเฉพาะเมื่อ
OpenClaw กำลังเขียนผลลัพธ์เครื่องมือของทรานสคริปต์เซสชันที่ OpenClaw เป็นเจ้าของ

## สื่อและการส่งมอบ

OpenClaw ยังคงเป็นเจ้าของการส่งมอบสื่อและการเลือกผู้ให้บริการสื่อ รูปภาพ,
วิดีโอ, เพลง, PDF, TTS และการทำความเข้าใจสื่อใช้การตั้งค่าผู้ให้บริการ/โมเดล
ที่ตรงกัน เช่น `agents.defaults.imageGenerationModel`, `videoGenerationModel`,
`pdfModel` และ `messages.tts`

ข้อความ รูปภาพ วิดีโอ เพลง TTS การอนุมัติ และเอาต์พุตของเครื่องมือส่งข้อความ
ยังคงผ่านเส้นทางการส่งมอบปกติของ OpenClaw การสร้างสื่อไม่ต้องใช้ runtime แบบเดิม
เมื่อ Codex ปล่อยรายการสร้างรูปภาพแบบเนทีฟที่มี `savedPath` OpenClaw จะส่งต่อ
ไฟล์นั้นทุกประการผ่านเส้นทาง reply-media ปกติ แม้ว่าเทิร์น Codex จะไม่มีข้อความ
ผู้ช่วยก็ตาม
