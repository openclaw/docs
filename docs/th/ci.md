CI ของ OpenClaw ทำงานเมื่อมีการพุชไปยัง `main` (เส้นทาง Markdown และ `docs/**` จะถูกละเว้น
ที่ทริกเกอร์) ในทุก pull request ที่ไม่ใช่ฉบับร่าง และเมื่อสั่งทำงานด้วยตนเอง
การพุช `main` มาตรฐานจะทำงานได้ครั้งละหนึ่งรายการ: กลุ่มการทำงานพร้อมกัน `CI` อนุญาตให้
รอบการผสานรวมหนึ่งรอบทำงานจนเสร็จ ขณะที่ GitHub เก็บไว้เฉพาะการพุชที่รอดำเนินการล่าสุด
การผสานใหม่จะแทนที่งานที่รอดำเนินการนั้น แทนที่จะยกเลิกงานที่
ลงทะเบียนเมทริกซ์ Blacksmith แล้ว ส่วน pull request ยังคงยกเลิก head ที่ถูกแทนที่
และการสั่งทำงานด้วยตนเองใช้กลุ่มแยกกัน `preflight` จำแนก diff และ
ปิดเลนที่ใช้ทรัพยากรมากเมื่อมีเพียงส่วนที่ไม่เกี่ยวข้องเท่านั้นที่เปลี่ยนแปลง การทำงาน
`workflow_dispatch` แบบสั่งด้วยตนเองจะข้ามการกำหนดขอบเขตอัจฉริยะโดยตั้งใจ และกระจาย
กราฟทั้งหมดสำหรับ release candidate และการตรวจสอบในวงกว้าง เลน Android ยังคง
เป็นแบบเลือกใช้ผ่าน `include_android` (หรืออินพุต `release_gate`) ความครอบคลุมของ
Plugin เฉพาะรุ่นเผยแพร่อยู่ในเวิร์กโฟลว์
[`Plugin Prerelease`](#plugin-prerelease) แยกต่างหาก และทำงานเฉพาะจาก
[`Full Release Validation`](#full-release-validation) หรือการสั่งทำงานด้วยตนเอง
อย่างชัดเจนเท่านั้น

## ลำดับหยุดทันทีเมื่อพบความล้มเหลว

1. `preflight` กำหนดว่าเลนใดมีอยู่บ้าง ตรรกะ `docs-scope` และ `changed-scope` เป็นขั้นตอนภายในงานนี้ ไม่ใช่งานแบบสแตนด์อโลน `main` มาตรฐานเริ่มทำงานทันที แต่กลุ่มการทำงานพร้อมกันอนุญาตให้มีการทำงานจนเสร็จได้เพียงหนึ่งรายการ และรวมการพุชภายหลังให้เป็นการทำงานที่รอดำเนินการล่าสุดเพียงรายการเดียว การพุชไปยัง main ที่เกี่ยวข้องกับ Node ยังจัดลำดับตัวเขียนดิสก์การขึ้นต่อกันเพียงรายเดียวและการดูแลขนาดไว้ที่นี่ ก่อนที่งานปลายทางจะเมานต์คีย์ได้ Blacksmith อาจเปิดเผยคอมมิตใหม่ให้เห็นได้เฉพาะในการทำงานของเวิร์กโฟลว์ครั้งถัดไป ดังนั้นผู้ใช้ภายในรอบเดียวกันจึงยังคงใช้ fallback ภายในเครื่องที่ตรวจสอบด้วยมาร์กเกอร์
2. `security-fast`, `check-*`, `check-additional-*`, `check-docs` และ `skills-python` ล้มเหลวอย่างรวดเร็วโดยไม่รอเมทริกซ์อาร์ติแฟกต์และแพลตฟอร์มที่หนักกว่า
3. `build-artifacts` และการตรวจสอบ `control-ui-i18n` แบบคำแนะนำทำงานซ้อนทับกับเลน Linux แบบเร็ว PR ต้นทางไม่รวมสแนปช็อตโลเคลที่สร้างขึ้น เวิร์กโฟลว์รีเฟรชแบบสแตนด์อโลนจะซ่อมแซมและผสาน PR ที่สร้างขึ้นแบบแยกโดยอัตโนมัติในเบื้องหลัง แบรนช์ `release/YYYY.M.PATCH` มาตรฐานอาจรวมการซ่อมแซมโลเคลสำหรับเตรียมรุ่นเผยแพร่ไว้กับผลลัพธ์รุ่นเผยแพร่อื่นที่สร้างขึ้น
4. จากนั้นเลนแพลตฟอร์มและรันไทม์ที่หนักกว่าจะกระจายงาน ได้แก่ `checks-fast-core`, `checks-fast-contracts-plugins-*`, `checks-fast-contracts-channels-*`, `checks-node-*`, `checks-windows`, `macos-node`, `macos-swift`, `ios-build` และ `android`
5. `openclaw/ci-gate` รอทุกเลนที่เลือก การตรวจสอบเบื้องต้นและความปลอดภัยต้องสำเร็จ งานปลายทางข้ามได้เฉพาะเมื่อรายการไม่ได้เลือกงานเหล่านั้น เลนที่เลือกซึ่งล้มเหลวหรือถูกยกเลิกจะทำให้ผลรวมล้มเหลว

ตัวประสานงานการผสานอาจนำ `openclaw/ci-gate` ที่ยืนยันตัวตนแล้วและสำเร็จ
สำหรับ head ของ pull request เดียวกันกลับมาใช้ได้นานสูงสุด 24 ชั่วโมง ซึ่งช่วยหลีกเลี่ยงการเขียน
แบรนช์ของผู้มีส่วนร่วมใหม่หลังจากการเปลี่ยนแปลง `main` ที่ไม่เกี่ยวข้อง ผลลัพธ์ที่นำกลับมาใช้ได้ไม่
ทดแทนการตรวจสอบ test-merge แบบเข้มงวดแยกต่างหากซึ่ง App เป็นเจ้าของกับ `main` ปัจจุบัน
การทำงานซ้ำที่รอดำเนินการหรือล้มเหลวภายหลังจะไม่ลบผลลัพธ์ที่สำเร็จก่อนหน้า
สำหรับ head ที่ไม่เปลี่ยนแปลงนั้นภายในช่วงเวลาความสดใหม่

ชุดกฎของสาขาเริ่มต้นกำหนดให้ใช้การตรวจสอบ `openclaw/ci-gate` ที่ GitHub Actions เป็นเจ้าของ ผู้ดูแลและผู้ดูแลระบบของที่เก็บมีการข้ามแบบฉุกเฉินที่มีการตรวจสอบย้อนหลัง ซึ่งมีไว้เฉพาะสำหรับการนำการเปลี่ยนแปลงแบบ fast-forward โดยตรงที่ลงนามแล้วเท่านั้น ส่วนชุดกฎขององค์กรยังคงบล็อกการลบและการอัปเดตแบบ non-fast-forward การผสาน pull request ตามปกติควรใช้เกตต่อไปแทนการข้าม CI ที่ล้มเหลว การตรวจสอบ test-merge แบบเข้มงวดที่ App เป็นเจ้าของซึ่งแยกต่างหากยังคงผูก head ไว้กับ `main` ปัจจุบัน

GitHub อาจทำเครื่องหมายงาน pull request ที่ถูกแทนที่เป็น `cancelled` เมื่อมี head ใหม่กว่าเข้ามา ให้ถือว่าเป็นสัญญาณรบกวนจาก CI เว้นแต่การรันล่าสุดของ PR เดียวกันจะล้มเหลวด้วย การรัน `main` ที่เป็นมาตรฐานจะไม่ถูกยกเลิกหลังจากรับเข้าแล้ว เมื่อมีทราฟฟิกการผสานเข้ามา GitHub จะแทนที่เฉพาะการรันเก่าที่ยังค้างอยู่ด้วยปลายล่าสุด งานเมทริกซ์ใช้ `fail-fast: false` และ `build-artifacts` รายงานความล้มเหลวของช่องทางแบบฝัง ขอบเขตการรองรับของแกนหลัก และการเฝ้าดู Gateway โดยตรง แทนการจัดคิวงานตรวจสอบขนาดเล็ก คีย์การทำงานพร้อมกันอัตโนมัติของ CI มีการกำหนดเวอร์ชัน (`CI-v7-*`) เพื่อไม่ให้ซอมบี้ฝั่ง GitHub ในกลุ่มคิวเก่าบล็อกการรัน main ที่ใหม่กว่าอย่างไม่มีกำหนด การรันชุดทดสอบเต็มรูปแบบด้วยตนเองใช้ `CI-manual-v1-*` และไม่ยกเลิกการรันที่กำลังดำเนินอยู่ ตัวป้องกันหน่วยความจำเมื่อเริ่มต้นสำหรับรายการ Plugin กำหนดเพดานไว้ที่ 350 MiB บน Blacksmith Linux แบบ self-hosted และอนุญาต 425 MiB บน Linux ที่โฮสต์โดย GitHub ซึ่งมีค่า RSS พื้นฐานสูงกว่าสำหรับ CLI ที่สร้างแล้วชุดเดียวกัน

ใช้ `pnpm ci:timings`, `pnpm ci:timings:recent` หรือ `node scripts/ci-run-timings.mjs <run-id>` เพื่อสรุปเวลาที่ใช้จริง เวลารอคิว งานที่ช้าที่สุด ความล้มเหลว และแนวกั้น fanout `pnpm-store-warmup` จาก GitHub Actions งาน `ci-timings-summary` ภายในเวิร์กโฟลว์มีอยู่ใน `ci.yml` แต่ปัจจุบันถูกปิดใช้งาน (`if: false`) ให้เรียกใช้ตัวช่วยจับเวลาภายในเครื่องแทน สำหรับเวลาการสร้าง ให้ตรวจสอบขั้นตอน `Build dist` ของงาน `build-artifacts`: `pnpm build:ci-artifacts` จะแสดง `[build-all] phase timings:` และรวม `ui:build` นอกจากนี้งานยังอัปโหลดอาร์ติแฟกต์ `startup-memory`

## บริบทและหลักฐานของ PR

PR ของผู้ร่วมพัฒนาภายนอกจะเรียกใช้เกตตรวจสอบบริบทและหลักฐานของ PR จาก
`.github/workflows/real-behavior-proof.yml` เวิร์กโฟลว์จะเช็กเอาต์
รุ่นแก้ไขของเวิร์กโฟลว์ที่เชื่อถือได้ (`github.workflow_sha`) และประเมินเฉพาะเนื้อหา PR
โดยไม่เรียกใช้โค้ดจากสาขาของผู้ร่วมพัฒนา

เกตนี้ใช้กับผู้เขียน PR ที่ไม่ใช่เจ้าของที่เก็บ สมาชิก
ผู้ร่วมงาน หรือบอต เกตจะผ่านเมื่อเนื้อหา PR มีส่วน
`What Problem This Solves` และ `Evidence` ที่ผู้เขียนจัดทำไว้ หลักฐานอาจเป็นการทดสอบเฉพาะจุด
ผลลัพธ์ CI ภาพหน้าจอ การบันทึก เอาต์พุตเทอร์มินัล การสังเกตการณ์จริง
บันทึกที่ปกปิดข้อมูลแล้ว หรือลิงก์อาร์ติแฟกต์ เนื้อหาจะระบุเจตนาและการตรวจสอบความถูกต้องที่เป็นประโยชน์
ผู้รีวิวจะตรวจสอบโค้ด การทดสอบ และ CI เพื่อประเมินความถูกต้อง

เมื่อการตรวจสอบล้มเหลว ให้อัปเดตเนื้อหา PR แทนการพุชคอมมิตโค้ดเพิ่ม

## ขอบเขตและการกำหนดเส้นทาง

ตรรกะของขอบเขตอยู่ใน `scripts/ci-changed-scope.mjs` และครอบคลุมด้วยการทดสอบหน่วยใน `src/scripts/ci-changed-scope.test.ts` การสั่งรันด้วยตนเองจะข้ามการตรวจหาขอบเขตที่เปลี่ยนแปลง และทำให้แมนิเฟสต์ preflight ทำงานเสมือนว่าทุกพื้นที่ในขอบเขตมีการเปลี่ยนแปลง

เวิร์กโฟลว์ Periphery แยกกันสำหรับ iOS และ macOS บังคับใช้นโยบายโค้ดที่ไม่ถูกใช้งานซึ่งต้องไม่มีข้อค้นพบ แต่ละเวิร์กโฟลว์จะทำงานเฉพาะเมื่อ pull request ที่ไม่ใช่ฉบับร่างแตะขอบเขตการสแกนแบบเนทีฟของตน หรือเมื่อมีการสั่งรันด้วยตนเอง

- **การแก้ไขเวิร์กโฟลว์ CI** ตรวจสอบกราฟ CI ของ Node การตรวจ lint ของเวิร์กโฟลว์ และเลน Windows (`ci.yml` เป็นตัวเรียกใช้) แต่การแก้ไขเหล่านี้ไม่ได้บังคับให้สร้างแบบเนทีฟสำหรับ iOS, Android หรือ macOS ด้วยตัวเอง เลนของแพลตฟอร์มเหล่านั้นยังคงจำกัดขอบเขตไว้เฉพาะการเปลี่ยนแปลงซอร์สของแพลตฟอร์ม
- **การตรวจสอบความสมเหตุสมผลของเวิร์กโฟลว์** เรียกใช้ `actionlint`, `zizmor` กับไฟล์ YAML ของเวิร์กโฟลว์ทั้งหมด ตัวป้องกันการแทรกค่าของ composite action และตัวป้องกันเครื่องหมายข้อขัดแย้ง งาน `security-fast` ที่จำกัดขอบเขตตาม PR ยังเรียกใช้ `zizmor` กับไฟล์เวิร์กโฟลว์ที่เปลี่ยนแปลง เพื่อให้ข้อค้นพบด้านความปลอดภัยของเวิร์กโฟลว์ทำให้ล้มเหลวได้ตั้งแต่เนิ่นๆ ในกราฟ CI หลัก
- **เอกสารในการพุช `main`** จะถูกตรวจสอบโดยเวิร์กโฟลว์ `Docs` แบบสแตนด์อโลนด้วยมิเรอร์เอกสาร ClawHub เดียวกับที่ CI ใช้ เพื่อให้การพุชแบบผสมระหว่างโค้ดกับเอกสารไม่จัดคิวชาร์ด `check-docs` ของ CI เพิ่มเติม Pull request และ CI ที่สั่งรันด้วยตนเองยังคงเรียกใช้ `check-docs` จาก CI เมื่อเอกสารมีการเปลี่ยนแปลง
- **TUI PTY** ทำงานในชาร์ด Linux Node `checks-node-core-runtime-tui-pty` สำหรับการเปลี่ยนแปลง TUI ชาร์ดนี้เรียกใช้ `test/vitest/vitest.tui-pty.config.ts` ด้วย `OPENCLAW_TUI_PTY_INCLUDE_LOCAL=1` จึงครอบคลุมทั้งเลนฟิกซ์เจอร์ `TuiBackend` แบบกำหนดแน่นอน และการทดสอบ smoke `tui --local` ที่ช้ากว่าและจำลองเฉพาะปลายทางโมเดลภายนอก
- **การแก้ไขเฉพาะการกำหนดเส้นทาง CI ชุดฟิกซ์เจอร์การทดสอบแกนหลักขนาดเล็กที่งานแบบเร็วเรียกใช้โดยตรง และการแก้ไขตัวช่วยสัญญา Plugin แบบจำกัดขอบเขต** ใช้เส้นทางแมนิเฟสต์แบบเร็วที่ใช้เฉพาะ Node: `preflight`, `security-fast` และเฉพาะเลนแบบเร็วที่การเปลี่ยนแปลงแตะต้อง ได้แก่ งานกำหนดเส้นทาง CI `checks-fast-core` หนึ่งงาน ชาร์ดสัญญา Plugin สองชาร์ด หรือทั้งสองอย่าง เส้นทางนี้จะข้ามอาร์ติแฟกต์การสร้าง ความเข้ากันได้กับ Node 22 สัญญาช่องทาง ชาร์ดแกนหลักแบบเต็ม ชาร์ด Plugin ที่รวมมาให้ และเมทริกซ์ตัวป้องกันเพิ่มเติม
- **การตรวจสอบ Node บน Windows** จำกัดขอบเขตไว้เฉพาะตัวห่อหุ้มกระบวนการ/พาธสำหรับ Windows ตัวช่วยรัน npm/pnpm/UI การกำหนดค่าตัวจัดการแพ็กเกจ และพื้นผิวเวิร์กโฟลว์ CI ที่เรียกใช้เลนนั้น การเปลี่ยนแปลงซอร์ส Plugin การทดสอบ smoke สำหรับการติดตั้ง และการเปลี่ยนแปลงเฉพาะการทดสอบที่ไม่เกี่ยวข้องจะยังคงทำงานบนเลน Linux Node

กลุ่มการทดสอบ Node ที่ช้าที่สุดจะถูกแยกหรือถ่วงสมดุล เพื่อให้งานแต่ละงานมีขนาดเล็กโดยไม่จองรันเนอร์เกินความจำเป็น:

- สัญญา Plugin และสัญญาช่องทางแต่ละรายการทำงานเป็นชาร์ดถ่วงน้ำหนักที่รองรับโดย Blacksmith จำนวนสองชาร์ด พร้อมใช้รันเนอร์มาตรฐานของ GitHub เป็นทางเลือกสำรอง
- เลน unit แบบเร็ว/สนับสนุนของคอร์ทำงานแยกกัน ส่วนโครงสร้างพื้นฐานรันไทม์ของคอร์แบ่งเป็นชาร์ดโดเมน process, shared, hooks, secrets และ cron สามชาร์ด
- การตอบกลับอัตโนมัติทำงานเป็นเวิร์กเกอร์ที่ปรับสมดุล โดยแบ่งทรีย่อยการตอบกลับเป็นชาร์ด agent-runner, commands, dispatch, session และ state-routing
- การกำหนดค่า Gateway/เซิร์ฟเวอร์แบบเอเจนต์ (ระนาบควบคุม) แบ่งเป็นเลน chat, auth, model, HTTP/plugin, runtime และ startup แทนการรออาร์ติแฟกต์ที่สร้างเสร็จแล้ว
- CI ปกติรวมเฉพาะชาร์ดรูปแบบ include ของโครงสร้างพื้นฐานที่แยกจากกันเป็นบันเดิลแบบกำหนดผลลัพธ์ได้ ซึ่งมีไฟล์ทดสอบไม่เกิน 64 ไฟล์ ช่วยลดเมทริกซ์ Node โดยไม่รวมชุดคำสั่ง/cron ที่ไม่ได้แยกจากกัน, agents-core ที่มีสถานะ หรือชุด Gateway/เซิร์ฟเวอร์เข้าด้วยกัน ชุดหนักแบบคงที่ยังคงใช้ 8 vCPU ส่วนเลนแบบบันเดิลและเลนที่มีน้ำหนักต่ำกว่าใช้ 4 vCPU
- Pull request ในรีโพซิทอรีหลักนำตัวแก้ไขการทดสอบที่เปลี่ยนแปลงแล้วมาใช้ซ้ำกับ diff ของทรีที่ผสานแบบสังเคราะห์ การเปลี่ยนแปลงที่ระบุได้อย่างแม่นยำจะรันงาน Node แบบเจาะจงหนึ่งงาน โดยไฟล์ทดสอบแต่ละไฟล์ที่เลือกจะมีกระบวนการของตนเอง เพื่อคงการแยกชุดที่มีสถานะไว้อย่างครบถ้วน ตัววางแผนรวมการทดสอบข้างเคียงกับรายการที่ขึ้นต่อกันตามกราฟการนำเข้า และย้อนกลับไปใช้แผนชุดทดสอบเต็มแบบกะทัดรัดจำนวน 14 งานที่มีอยู่ สำหรับการเปลี่ยนแปลงแพ็กเกจเวิร์กสเปซ, แพ็กเกจ/lockfile, ฮาร์เนสที่ใช้ร่วมกัน, การกำหนดค่าแบบแยก, การเปลี่ยนชื่อหรือการลบ, การเปลี่ยนแปลงสัญญาส่วนขยายสาธารณะ, การทดสอบที่มีการตั้งค่าชาร์ดพิเศษ, เป้าหมายที่แก้ไขได้เพียงบางส่วนหรือว่างเปล่า, แผนพาธหรือเป้าหมายที่มีขนาดใหญ่เกินไป และข้อผิดพลาดของตัววางแผน แผนแบบเจาะจงจะคงเกตขอบเขตอาร์ติแฟกต์ที่สร้างแล้วแบบเต็มไว้เสมอ เพราะตัวสแกนรีโพซิทอรีของเกตนี้ไม่สามารถอนุมานจากการนำเข้าได้ การพุช `main` จะรันชุดเต็มแบบกะทัดรัดเดียวกัน: เหตุการณ์พุชระหว่างทางที่รอดำเนินการอาจถูกรวมเข้าด้วยกัน ดังนั้นรอบล่าสุดที่ยังคงอยู่ต้องตรวจสอบทรีการผสานรวมทั้งหมด ไม่ใช่เฉพาะ diff ของการพุชครั้งสุดท้ายเพียงครั้งเดียว การสั่งทำงานด้วยตนเองและเกตการเผยแพร่ยังคงใช้เมทริกซ์แยกตามชาร์ดที่มีชื่อแบบเต็ม
- เมทริกซ์ Node แบบเต็มรับเครื่องมือแบบอนุกรมที่ช้าอย่างสม่ำเสมอ, ชาร์ดคำสั่งตอบกลับอัตโนมัติ และตัวเขียนแคช core-fast แบบกว้างก่อน วิธีนี้คงขีดจำกัดที่ 28 งานไว้ พร้อมป้องกันไม่ให้งานบนเส้นทางวิกฤตและเมล็ด transform ของรอบถัดไปเลื่อนไปอยู่ในระลอกหลัง
- การทดสอบเบราว์เซอร์, QA, สื่อ และ Plugin เบ็ดเตล็ดแบบกว้างใช้การกำหนดค่า Vitest เฉพาะของตน แทนตัวรวบรวม Plugin แบบใช้ร่วมกัน ชาร์ดรูปแบบ include บันทึกรายการเวลาโดยใช้ชื่อชาร์ด CI เพื่อให้ `.artifacts/vitest-shard-timings.json` แยกความแตกต่างระหว่างการกำหนดค่าทั้งชุดกับชาร์ดที่ถูกกรองได้
- งานชาร์ด Linux Node เก็บรักษาแคชโมดูลบนระบบไฟล์แบบทดลองของ Vitest ผ่าน Actions cache API ต้นทาง ซึ่ง Blacksmith เร่งความเร็วบนรันเนอร์ของตนอย่างโปร่งใส ชาร์ด CI ทุกชาร์ดเป็นแบบกู้คืนเท่านั้น และแตกเมล็ดที่ได้รับการป้องกันไว้ในรูทภายในรันเนอร์ของตน จากนั้นตัวห่อชาร์ดจะกำหนดไดเรกทอรีย่อยที่ใช้งานอยู่แยกกันให้กระบวนการ Vitest ที่ทำงานพร้อมกัน เฉพาะตัวอุ่นแคชรายวันที่ไม่ถูกยกเลิกหรือที่สั่งทำงานอย่างชัดเจนเท่านั้นที่บันทึกอาร์ไคฟ์แบบเปลี่ยนแปลงไม่ได้ชุดใหม่ ดังนั้น Pull request จึงไม่สามารถเผยแพร่ transform หรือสร้างตระกูลแคชแยกตาม PR ได้ ลายนิ้วมือของอินพุต transform จะล้างรุ่นของ lockfile, แพ็กเกจ, tsconfig และการกำหนดค่า Vitest ที่เข้ากันไม่ได้ ตัวเขียนที่ได้รับการป้องกันจะสแกนและตัดแคชที่กู้คืนแล้วให้เหลือ 75% หลังจากแคชมีขนาดเกิน 2 GiB Vitest แฮชรหัสโมดูล, เนื้อหาซอร์ส, สภาพแวดล้อม และการกำหนดค่า transform ที่แก้ไขแล้ว ดังนั้นการเปลี่ยนแปลงซอร์สบางส่วนตามปกติจะทำให้รายการที่ไม่เปลี่ยนแปลงยังคงอุ่นอยู่ ขณะที่โมดูลที่เปลี่ยนแปลงจะพลาดแคชอย่างปลอดภัย คำนำหน้าการกู้คืนแบบหยาบช่วยเชื่อมระหว่างรอบเวิร์กโฟลว์ ส่วน LRU และการขับออกเมื่อไม่มีการใช้งานตามปกติของ Actions cache จะจำกัดอาร์ไคฟ์แบบเปลี่ยนแปลงไม่ได้รุ่นเก่า
- งาน Linux Node ที่เชื่อถือได้ยังผูก pnpm store และ `node_modules` จากดิสก์การขึ้นต่อกันที่ได้รับการป้องกันหนึ่งชุดต่อสาย Node ที่รองรับ ไฟล์รายการแพ็กเกจ, การตั้งค่าการติดตั้ง, แพลตฟอร์มรันเนอร์ และแพตช์ Node ที่แน่นอนจะไม่รวมอยู่ในคีย์ดิสก์ โดยลายนิ้วมือรันไทม์และอินพุตการติดตั้งที่ตรงกันจะตัดสินว่างานนำทรีกลับมาใช้ซ้ำ หรือติดตั้งใหม่และรีเฟรชดิสก์เดิม ไฟล์รายการจะถูกทำให้เป็นรูปแบบมาตรฐานก่อนแฮช ฮุกระดับรูทโดยตรงที่ผ่านการตรวจสอบจะคงไว้เฉพาะสคริปต์วงจรชีวิตการติดตั้งของ pnpm ดังนั้นการแก้ไขการจัดรูปแบบและสคริปต์ทดสอบ/บิลด์ทั่วไปจะยังคงใช้ทรีการขึ้นต่อกันที่อุ่นอยู่ได้ การเปลี่ยนแปลงฮุกวงจรชีวิตที่ยังไม่ผ่านการตรวจสอบจะล้มเหลวแบบปิด จนกว่าอินพุตซอร์สของฮุกจะเข้าร่วมสัญญาลายนิ้วมือ การเปลี่ยนแปลงการขึ้นต่อกัน, ตัวจัดการแพ็กเกจ, ซอร์สของฮุก และ lockfile จะทำให้สแนปช็อตใช้ไม่ได้เสมอ Pull request ที่มีลายนิ้วมือของสแนปช็อตแบบอ่านอย่างเดียวแตกต่างออกไป จะยกเลิกการผูกเวิร์กสเปซและติดตั้งลงในพื้นที่จัดเก็บภายในรันเนอร์ เพื่อหลีกเลี่ยงการเขียนที่ช้าลงในโคลนที่ไม่สามารถเผยแพร่ได้ การติดตั้งแบบ cold บน sticky disk จะปิดใช้การลองดึงข้อมูลซ้ำภายในของ pnpm และพยายามติดตั้งแบบเต็มโดยมีขอบเขตสูงสุดสามครั้งจาก store ที่อุ่นขึ้นเรื่อย ๆ แต่การหมดเวลายังคงถือเป็นความล้มเหลว หลังจากกู้คืนได้ตรงกันหรือติดตั้งด้วย frozen-lockfile แล้ว ขั้นตอนตั้งค่าจะปิดการตรวจสอบการขึ้นต่อกันก่อนรันที่ซ้ำซ้อนของ pnpm: รีโพซิทอรีจงใจตัด `node_modules` ภายใน Plugin ออก ซึ่งมิฉะนั้น pnpm จะถือว่าล้าสมัยและซ่อมแซมผ่านการติดตั้งโดยนัยพร้อมกันที่ไม่ปลอดภัยระหว่างการกระจายชาร์ด ขั้นตอน preflight ของ main หลักเป็นผู้เขียนเพียงรายเดียว และวัดขนาด store ทุกครั้งที่รีเฟรช โดยรัน `pnpm store prune` หลังจากเวอร์ชันแพ็กเกจที่เลิกใช้แล้วผลักขนาดให้เกิน 8 GiB เท่านั้น การเผยแพร่สแนปช็อตของ Blacksmith เป็นแบบอะซิงโครนัส แม้งานผู้เขียนจะเสร็จสิ้นแล้ว ดังนั้นรอบแรกหลังมีคีย์หรือลายนิ้วมือใหม่อาจยังคงเป็น cold ได้ การกู้คืนด้วยตัวทำเครื่องหมายที่ตรงกันในภายหลังคือหลักฐานการทยอยใช้งาน งาน CI ที่จำเป็นและ Pull request จะได้รับโคลนแบบใช้แล้วทิ้ง ดังนั้นการเปลี่ยนแปลงการขึ้นต่อกันจึงไม่สร้างดิสก์ใหม่, สแนปช็อตที่แข่งขันกัน หรือล็อกแคชที่อาจยกเลิกบิลด์
- งานชาร์ด Node และงานอาร์ติแฟกต์บิลด์ยังคืนค่าแคชคอมไพล์บนดิสก์แบบพกพาของ Node ผ่าน Actions cache แบบเปลี่ยนแปลงไม่ได้ เนมสเปซ `test` และ `build` ที่แยกจากกันป้องกันไม่ให้ตัวเขียนของแต่ละฝ่ายแทนที่อาร์ไคฟ์ของอีกฝ่าย: ตัวอุ่นแคชการทดสอบตามกำหนดเวลาเป็นเจ้าของเมล็ดการทดสอบที่ได้รับการป้องกัน ส่วน `build-artifacts` อาจเผยแพร่อาร์ไคฟ์บิลด์ที่ได้รับการป้องกันได้สูงสุดหนึ่งชุดต่อวัน UTC จากการพุช `main` ที่เชื่อถือได้ งาน PR และงานทดสอบทั่วไปจะอ่านเฉพาะสแนปช็อตที่ได้รับการป้องกัน ดังนั้น bytecode ของฟีเจอร์บรานช์จะไม่เข้าสู่เมล็ดที่ใช้ร่วมกัน และทราฟฟิก PR จะไม่สร้างอาร์ไคฟ์แคช วิธีนี้นำ V8 bytecode กลับมาใช้ซ้ำสำหรับการประสานงานที่โหลดโดย Node, เครื่องมือบิลด์ และการขึ้นต่อกันภายนอกระหว่างพาธ checkout ที่แตกต่างกัน รวมถึงเมื่อมีการเปลี่ยนแปลงเพียงบางส่วนของกราฟซอร์ส กระบวนการลูกของ Vitest ปิดใช้แคชคอมไพล์ที่สืบทอดมา เพราะอาจเปิดใช้ coverage ภายในการกำหนดค่าแบบไดนามิก และ V8 coverage อาจสูญเสียความแม่นยำของตำแหน่งซอร์สเมื่อดีซีเรียลไลซ์สคริปต์จาก bytecode
- งานอาร์ติแฟกต์บิลด์ยังเก็บรักษาเอาต์พุตขั้นตอน `build-all` ที่มีลายนิ้วมือตามเนื้อหา การประกาศ Plugin SDK ที่ CI สร้างขึ้นเองจะแฮชกราฟซอร์ส TypeScript/JSON ทั้งหมดที่รีโพซิทอรีเป็นเจ้าของ ยกเว้นไดเรกทอรีที่ติดตั้งและสร้างขึ้น และกู้คืนทั้งการประกาศแบบแบนและบริดจ์แพ็กเกจหลังจาก `tsdown` ล้าง `dist` การเปลี่ยนแปลงเอกสาร, เวิร์กโฟลว์, Plugin และส่วนอื่นนอกกราฟนั้นสามารถนำสแนปช็อตการประกาศกลับมาใช้ซ้ำได้ ส่วนการเปลี่ยนแปลงซอร์สจะสร้างใหม่ก่อนที่เกตการส่งออกจะทำงาน
- บิลด์การประกาศแบบเต็มแบ่ง `tsdown` ออกเป็นกลุ่ม AI, workspace-package และ unified แต่ละกลุ่มแคชเฉพาะการประกาศ จากนั้นยังคงสร้าง JavaScript รันไทม์ใหม่ก่อนกู้คืนการประกาศเหล่านั้น ดังนั้นการเปลี่ยนแปลงคอร์หรือ Plugin จะทำให้ใช้ไม่ได้เฉพาะกราฟ unified ขนาดใหญ่ ส่วนการเปลี่ยนแปลงแพ็กเกจเวิร์กสเปซจะทำให้ทุกกลุ่มการประกาศที่ขึ้นต่อกันใช้ไม่ได้อย่างเคร่งครัด โดยทั่วไปบิลด์สาธารณะแบบเต็มใช้ Actions cache แบบเปลี่ยนแปลงไม่ได้ คีย์กู้คืนแบบหยาบจะเป็นเมล็ดให้การเปลี่ยนแปลงบางส่วน ลายนิ้วมือตามเนื้อหาของแต่ละกลุ่มจะปฏิเสธข้อมูลล้าสมัย และโควตาแคชของ GitHub จะขับรุ่นเก่าออก ส่วนเลน Node 22 รายสัปดาห์จะเผยแพร่อาร์ติแฟกต์อายุ 14 วันหลังจากรัน `main` สำเร็จ และกู้คืนเฉพาะอาร์ติแฟกต์ที่สามารถแก้ไขอัตลักษณ์ผู้สร้างแบบเปลี่ยนแปลงไม่ได้ไปยังเวิร์กโฟลว์นั้นบน `main` เพื่อหลีกเลี่ยงการหมุนเวียนโควตาโดยไม่อนุญาตให้โค้ด PR เขียนแคชที่ใช้ร่วมกัน การประกาศ Private-QA จะไม่ถูกเก็บรักษาใน Actions cache เพราะเนมสเปซแคชไม่ใช่ขอบเขตการรักษาความลับ
- `check-additional-*` แบ่งรายการตัวตรวจสอบขอบเขตเสริม (`scripts/run-additional-boundary-checks.mjs`) เป็นหนึ่งชาร์ดที่ใช้พรอมต์หนัก (`check-additional-boundaries-a` ซึ่งรวมการตรวจสอบความคลาดเคลื่อนของสแนปช็อตพรอมต์ Codex) และหนึ่งชาร์ดรวมสำหรับแถบที่เหลือ (`check-additional-boundaries-bcd`) โดยแต่ละชาร์ดรันตัวตรวจสอบอิสระพร้อมกันและพิมพ์เวลาของแต่ละการตรวจสอบ งานคอมไพล์/คานารีของขอบเขตแพ็กเกจยังคงอยู่ด้วยกัน และสถาปัตยกรรมโทโพโลยีรันไทม์ทำงานแยกจาก coverage การเฝ้าดู Gateway ที่ฝังอยู่ใน `build-artifacts`

ระยะหมดเวลาแบตช์ 120 นาที ส่วนกลุ่มรูปแบบ include ใช้งบประมาณงานที่มีขอบเขตเดียวกันร่วมกัน

Android CI รันทั้ง `testPlayDebugUnitTest` และ `testThirdPartyDebugUnitTest` จากนั้นสร้าง Play debug APK flavor ของบุคคลที่สามไม่มีชุดซอร์สหรือ manifest แยกต่างหาก เลน unit-test ของ flavor นี้ยังคงคอมไพล์ flavor พร้อมแฟล็ก BuildConfig สำหรับ SMS/call-log ขณะเดียวกันก็หลีกเลี่ยงงานแพ็กเกจ debug APK ที่ซ้ำกันในทุกการพุชที่เกี่ยวข้องกับ Android งาน Gradle ปัจจุบันแต่ละงานมี sticky disk ที่ได้รับการป้องกันหนึ่งชุด งาน PR ใช้โคลนแบบใช้แล้วทิ้ง ส่วนรอบที่ได้รับการป้องกันจะรีเฟรชรายการ Gradle ที่ระบุที่อยู่ด้วยเนื้อหาในตำแหน่งเดิม

คีย์ sticky-disk ของ Blacksmith ถูกจำกัดโดยเจตนาตามมิติรันไทม์หรืองานที่รองรับเท่านั้น โดยไม่ใช้หมายเลข PR, คอมมิต, รอบ, บรานช์ หรือแฮชการขึ้นต่อกัน แคช transform และคอมไพล์ของรันไทม์ใช้ Actions cache แทน sticky disk เพราะอาร์ไคฟ์แบบเปลี่ยนแปลงไม่ได้ให้ผลลัพธ์การกู้คืน/บันทึกที่ตรวจสอบได้ และหลีกเลี่ยงความล้มเหลวในการเลื่อนระดับสแนปช็อตแบบเปลี่ยนแปลงได้ หลังจากย้ายเวอร์ชันคีย์ sticky แล้ว ให้เพิ่มเฉพาะคีย์, สถาปัตยกรรม และอัตลักษณ์ภูมิภาคเก่าที่แน่นอนลงใน `.github/retired-sticky-disks.json` สั่งทำงาน `Sticky Disk Cleanup` จาก `main` ด้วยมิติและการยืนยันเดียวกัน ตรวจสอบการลบ แล้วจึงนำรายการเหล่านั้นออก เวิร์กโฟลว์กำหนดเส้นทางอัตลักษณ์ ARM ไปยังรันเนอร์ ARM, ปฏิเสธภูมิภาครันเนอร์ที่ไม่ตรงกัน, ใช้แอ็กชันลบตามคีย์ที่ตรงกันของ Blacksmith และไม่ลบแคชตัวสร้าง Docker หรือคำนำหน้าแบบ wildcard อาร์ไคฟ์ Actions cache ใช้ LRU และการขับออกเมื่อไม่มีการใช้งานตามปกติ

ชาร์ด `check-dependencies` รันการตรวจสอบการขึ้นต่อกัน, ไฟล์ที่ไม่ได้ใช้ และ export ที่ไม่ได้ใช้ในโค้ดโปรดักชันด้วย Knip ตัวตรวจสอบไฟล์ที่ไม่ได้ใช้จะล้มเหลวเมื่อ PR เพิ่มไฟล์ที่ไม่ได้ใช้ใหม่ซึ่งยังไม่ได้รับการรีวิว หรือทิ้งรายการ allowlist ที่ล้าสมัยไว้ ขณะเดียวกันยังคงรักษาพื้นผิว Plugin แบบไดนามิก, พื้นผิวที่สร้างขึ้น, บิลด์, การทดสอบสด และบริดจ์แพ็กเกจที่ตั้งใจไว้ ซึ่ง Knip ไม่สามารถแก้ไขแบบสแตติกได้ ตัวตรวจสอบ export ที่ไม่ได้ใช้จะไม่รวมไฟล์สนับสนุนการทดสอบ และล้มเหลวเมื่อพบ export ในโค้ดโปรดักชันที่ไม่ได้ใช้ทุกรายการ ผู้ใช้แบบไดนามิกที่ตั้งใจไว้ต้องถูกจำลองใน `config/knip.config.ts` เป้าหมายในอดีตจะรันตัวตรวจสอบ export เมื่อมีให้ใช้งาน และยังคงใช้ทางเลือกสำรองสำหรับโค้ดที่ไม่ทำงานแบบเดิมในกรณีอื่น

## การส่งต่อกิจกรรม ClawSweeper

`.github/workflows/clawsweeper-dispatch.yml` เป็นบริดจ์ฝั่งเป้าหมายจากกิจกรรมในรีโพซิทอรี OpenClaw ไปยัง ClawSweeper โดยจะไม่ checkout หรือเรียกใช้โค้ด Pull request ที่ไม่น่าเชื่อถือ เวิร์กโฟลว์สร้างโทเค็น GitHub App จาก `CLAWSWEEPER_APP_PRIVATE_KEY` แล้วส่งเพย์โหลด `repository_dispatch` แบบกะทัดรัดไปยัง `openclaw/clawsweeper`

เวิร์กโฟลว์มีสี่เลน:

- `clawsweeper_item` สำหรับคำขอรีวิว issue และ pull request ที่ระบุชัดเจน
- `clawsweeper_comment` สำหรับคำสั่ง ClawSweeper ที่ระบุชัดเจนในความคิดเห็นของ issue
- `clawsweeper_commit_review` สำหรับคำขอรีวิวระดับ commit เมื่อ push ไปยัง `main`
- `github_activity` สำหรับกิจกรรมทั่วไปบน GitHub ที่เอเจนต์ ClawSweeper อาจตรวจสอบ

เลน `github_activity` ส่งต่อเฉพาะข้อมูลเมตาที่ปรับให้อยู่ในรูปแบบมาตรฐานแล้ว ได้แก่ ประเภทเหตุการณ์ การดำเนินการ ผู้ดำเนินการ repository หมายเลขรายการ URL ชื่อเรื่อง สถานะ และข้อความตัดตอนสั้นๆ จากความคิดเห็นหรือรีวิวเมื่อมี โดยตั้งใจหลีกเลี่ยงการส่งต่อเนื้อหา Webhook ทั้งหมด เวิร์กโฟลว์ที่รับข้อมูลใน `openclaw/clawsweeper` คือ `.github/workflows/github-activity.yml` ซึ่งโพสต์เหตุการณ์ที่ปรับให้อยู่ในรูปแบบมาตรฐานแล้วไปยังฮุกของ OpenClaw Gateway สำหรับเอเจนต์ ClawSweeper

กิจกรรมทั่วไปมีไว้เพื่อสังเกตการณ์ ไม่ใช่ส่งต่อโดยค่าเริ่มต้น เอเจนต์ ClawSweeper จะได้รับเป้าหมาย Discord ในพรอมต์ และควรโพสต์ไปยัง `#clawsweeper` เฉพาะเมื่อเหตุการณ์นั้นไม่คาดคิด ต้องดำเนินการ มีความเสี่ยง หรือเป็นประโยชน์ต่อการปฏิบัติงาน การเปิดรายการตามปกติ การแก้ไข ความเคลื่อนไหวจากบอต สัญญาณรบกวนจาก Webhook ที่ซ้ำกัน และทราฟฟิกรีวิวตามปกติ ควรส่งผลเป็น `NO_REPLY`

ให้ถือว่าชื่อเรื่อง ความคิดเห็น เนื้อหา ข้อความรีวิว ชื่อ branch และข้อความ commit ของ GitHub เป็นข้อมูลที่ไม่น่าเชื่อถือตลอดทั้งเส้นทางนี้ ข้อมูลเหล่านี้เป็นอินพุตสำหรับการสรุปและคัดแยก ไม่ใช่คำสั่งสำหรับเวิร์กโฟลว์หรือรันไทม์ของเอเจนต์

## การสั่งทำงานด้วยตนเอง

การสั่งทำงาน CI ด้วยตนเองจะเรียกใช้กราฟงานเดียวกับ CI ปกติ แต่บังคับเปิดทุกเลนที่กำหนดขอบเขตและไม่ใช่ Android ได้แก่ ชาร์ด Linux Node, ชาร์ด Plugin ที่รวมมาในแพ็กเกจ, ชาร์ดสัญญาของ Plugin และช่องทาง, ความเข้ากันได้กับ Node 22, `check-*`, `check-additional-*`, การตรวจสอบ smoke ของอาร์ติแฟกต์ที่สร้างแล้ว, การตรวจสอบเอกสาร, Skills ของ Python, Windows, macOS, บิลด์ iOS และ i18n ของ Control UI ความสอดคล้องของโลแคล Control UI เป็นเพียงคำแนะนำในการรัน PR อัตโนมัติและ `main` เนื่องจากเวิร์กโฟลว์รีเฟรชแบบแยกต่างหากจะแก้ไขความคลาดเคลื่อนของข้อมูลที่สร้างขึ้นในเบื้องหลัง แต่จะเป็นเงื่อนไขบังคับใน CI ที่สั่งด้วยตนเอง และจึงเป็นเงื่อนไขบังคับในการตรวจสอบความถูกต้องของรุ่นฉบับเต็ม การเตรียมรีลีสจะเรียกใช้การซิงค์โลแคลแบบเดียวกันก่อนตรึง Code SHA แล้วจึงตรวจสอบสถานะที่ไม่มี fallback อย่างเคร่งครัด การสั่ง CI แบบแยกต่างหากด้วยตนเองจะเรียกใช้ Android เฉพาะเมื่อใช้ `include_android=true` (อินพุต `release_gate` จะบังคับ Android เช่นกัน) ส่วนเวิร์กโฟลว์ครอบคลุมรีลีสฉบับเต็มจะเปิดใช้ Android โดยส่ง `include_android=true` การตรวจสอบแบบสแตติกก่อนรีลีสของ Plugin, ชาร์ด `agentic-plugins` ที่ใช้เฉพาะรีลีส, การกวาดตรวจส่วนขยายทั้งชุด และเลน Docker ก่อนรีลีสของ Plugin จะไม่รวมอยู่ใน CI ชุด Docker ก่อนรีลีสจะทำงานเฉพาะเมื่อ `Full Release Validation` สั่งเวิร์กโฟลว์ `Plugin Prerelease` แยกต่างหากโดยเปิดใช้เกตตรวจสอบรีลีส

การตรวจสอบจำนวนบรรทัดสูงสุดของ PR จะหา baseline จากต้นไม้ merge สังเคราะห์ที่ checkout ไว้ และตรวจสอบ parent ของ head เทียบกับ head ของเหตุการณ์ การรันด้วยตนเองใช้กลุ่ม concurrency ที่ไม่ซ้ำกัน เพื่อไม่ให้ชุดทดสอบเต็มของ release candidate ถูกยกเลิกโดยการ push หรือการรัน PR อื่นบน ref เดียวกัน อินพุต `target_ref` ที่เป็นทางเลือกช่วยให้ผู้เรียกที่เชื่อถือได้เรียกใช้กราฟนั้นกับ branch, tag หรือ commit SHA แบบเต็ม โดยใช้ไฟล์เวิร์กโฟลว์จาก ref ที่เลือกสำหรับการสั่งทำงาน ส่วน baseline ของจำนวนบรรทัดสูงสุดจะเปรียบเทียบกับ merge base ของเป้าหมายกับ head ของ branch เริ่มต้นที่ได้รับการแก้ค่าแล้วสำหรับการรันนั้น อินพุต `release_gate` เป็นทางเลือกสำรองแบบ exact-SHA สำหรับผู้ดูแล เมื่อ CI ของ PR ติดขัดด้านความจุ โดยกำหนดให้ `target_ref` เป็น commit SHA แบบเต็มที่ตรงกับ head ของ branch ที่สั่งทำงาน และกำหนดให้ `pull_request_number` ระบุ PR ที่เปิดอยู่ซึ่งต้นไม้ merge จะได้รับการตรวจสอบ

```bash
gh workflow run ci.yml --ref release/YYYY.M.PATCH
gh workflow run ci.yml --ref main -f target_ref=<branch-or-sha> -f include_android=true
gh workflow run full-release-validation.yml --ref main -f ref=<branch-or-sha>
```

เส้นทาง extended-stable รายเดือนที่เผยแพร่เฉพาะ npm เป็นข้อยกเว้น: ให้สั่งทั้ง preflight `OpenClaw NPM
Release` และ `Full Release Validation` จาก branch
`extended-stable/YYYY.M.33` ที่ตรงกันทุกประการ เก็บรักษา ID การรันของทั้งสองรายการ และส่ง ID ทั้งสองไปยัง
การรันเผยแพร่ npm โดยตรง ดูคำสั่ง ข้อกำหนดด้านอัตลักษณ์ที่ตรงกันทุกประการ การอ่านค่ากลับจากรีจิสทรี และขั้นตอน
การซ่อมแซมตัวเลือกได้ที่ [การเผยแพร่ extended-stable รายเดือนเฉพาะ npm](/th/reference/RELEASING#monthly-npm-only-extended-stable-publication)
เส้นทางนี้จะไม่สั่งการเผยแพร่ Plugin, macOS, Windows, GitHub
Release, dist-tag ส่วนตัว หรือแพลตฟอร์มอื่น

## งบประมาณการลงทะเบียนรันเนอร์

บักเก็ตการลงทะเบียนรันเนอร์ GitHub ปัจจุบันของ OpenClaw รายงานการลงทะเบียนรันเนอร์แบบ self-hosted 10,000
ครั้งต่อ 5 นาทีใน `ghx api rate_limit` ให้ตรวจสอบ
`actions_runner_registration` อีกครั้งก่อนการปรับแต่งแต่ละรอบ เนื่องจาก GitHub อาจเปลี่ยน
บักเก็ตนี้ ขีดจำกัดนี้ใช้ร่วมกันระหว่างการลงทะเบียนรันเนอร์ Blacksmith ทั้งหมดใน
องค์กร `openclaw` ดังนั้นการเพิ่มการติดตั้ง Blacksmith อีกหนึ่งรายการจะไม่เพิ่ม
บักเก็ตใหม่

ให้ถือว่าป้ายกำกับ Blacksmith เป็นทรัพยากรที่ขาดแคลนสำหรับควบคุมการทำงานแบบกระชาก งานที่
มีเพียงการกำหนดเส้นทาง แจ้งเตือน สรุป เลือกชาร์ด หรือเรียกใช้การสแกน CodeQL ระยะสั้น ควร
อยู่บนรันเนอร์ที่โฮสต์โดย GitHub เว้นแต่จะมีความต้องการเฉพาะของ Blacksmith ที่วัดผลแล้ว
เมทริกซ์ Blacksmith ใหม่, `max-parallel` ที่ใหญ่ขึ้น หรือเวิร์กโฟลว์
ความถี่สูงใดๆ ต้องแสดงจำนวนการลงทะเบียนในกรณีเลวร้ายที่สุด และรักษาเป้าหมายระดับองค์กร
ให้ต่ำกว่าประมาณ 60% ของบักเก็ตปัจจุบัน เมื่อบักเก็ตปัจจุบันรองรับการลงทะเบียน 10,000
ครั้ง นั่นหมายถึงเป้าหมายการดำเนินงานที่ 6,000 ครั้ง โดยเว้นพื้นที่รองรับ repository
ที่ทำงานพร้อมกัน การลองใหม่ และการทับซ้อนของการทำงานแบบกระชาก

แผน PR ที่อิงตามเป้าหมายที่เปลี่ยนแปลงจะลดการทำงานแบบกระชากของการทดสอบ Node ทั่วไปจากการลงทะเบียน Blacksmith 14 ครั้งเหลือหนึ่งครั้ง PR ที่มีความเสี่ยงในวงกว้างจะยังคงใช้ทางเลือกสำรองแบบกระชับที่มีการลงทะเบียน 14 ครั้ง ดังนั้นกรณีเลวร้ายที่สุดจึงไม่เพิ่มขึ้น

CI ของ repository canonical ยังคงใช้ Blacksmith เป็นเส้นทางรันเนอร์เริ่มต้นสำหรับการรัน push และ pull request ตามปกติ `workflow_dispatch` และการรัน repository ที่ไม่ใช่ canonical จะใช้รันเนอร์ที่โฮสต์โดย GitHub แต่ปัจจุบันการรัน canonical ตามปกติจะไม่ตรวจสอบสถานะคิวของ Blacksmith หรือใช้ป้ายกำกับที่โฮสต์โดย GitHub เป็นทางเลือกสำรองโดยอัตโนมัติเมื่อ Blacksmith ใช้งานไม่ได้

## ประสิทธิภาพของ OpenClaw

`OpenClaw Performance` คือเวิร์กโฟลว์ประสิทธิภาพของผลิตภัณฑ์/รันไทม์ โดยทำงานทุกวันบน `main` และสามารถสั่งทำงานด้วยตนเองได้:

```bash
gh workflow run openclaw-performance.yml --ref main -f profile=diagnostic -f repeat=3
gh workflow run openclaw-performance.yml --ref main -f profile=smoke -f repeat=1 -f deep_profile=true -f live_openai_candidate=true
gh workflow run openclaw-performance.yml --ref main -f target_ref=v2026.5.2 -f profile=diagnostic -f repeat=3
```

โดยปกติการสั่งทำงานด้วยตนเองจะวัดประสิทธิภาพของ ref ของเวิร์กโฟลว์ ตั้งค่า `target_ref` เพื่อวัดประสิทธิภาพของ tag รีลีสหรือ branch อื่นด้วยการนำเวิร์กโฟลว์ปัจจุบันไปใช้ เส้นทางรายงานที่เผยแพร่และตัวชี้ล่าสุดจะกำหนดคีย์ตาม ref ที่ทดสอบ และ `index.md` แต่ละรายการจะบันทึก ref/SHA ที่ทดสอบ, ref/SHA ของเวิร์กโฟลว์, ref ของ Kova, โปรไฟล์, โหมดการยืนยันตัวตนของเลน, โมเดล, จำนวนครั้งที่ทำซ้ำ และตัวกรองสถานการณ์

เวิร์กโฟลว์จะติดตั้ง OCM จากรีลีสที่ตรึงไว้ และติดตั้ง Kova จาก `openclaw/Kova` ตามอินพุต `kova_ref` ที่ตรึงไว้ จากนั้นเรียกใช้สามเลน:

- `mock-provider`: สถานการณ์การวินิจฉัย Kova กับรันไทม์ที่สร้างในเครื่อง โดยใช้การตรวจสอบสิทธิ์แบบจำลองที่เข้ากันได้กับ OpenAI ซึ่งให้ผลลัพธ์คงที่
- `mock-deep-profile`: การทำโปรไฟล์ CPU/ฮีป/เทรซสำหรับจุดที่ใช้ทรัพยากรสูงในการเริ่มต้น Gateway และรอบการทำงานของเอเจนต์ ทำงานตามกำหนดเวลา หรือเมื่อสั่งรันด้วย `deep_profile=true`
- `live-openai-candidate`: รอบการทำงานจริงของเอเจนต์ OpenAI `openai/gpt-5.6-luna` ซึ่งจะข้ามเมื่อ `OPENAI_API_KEY` ไม่พร้อมใช้งาน ทำงานตามกำหนดเวลา หรือเมื่อสั่งรันด้วย `live_openai_candidate=true`

เลนผู้ให้บริการจำลองยังเรียกใช้โพรบซอร์สที่เป็นเนทีฟของ OpenClaw หลังจาก Kova ผ่านแล้ว ได้แก่ เวลาและหน่วยความจำในการบูต Gateway สำหรับกรณีเริ่มต้นแบบค่าเริ่มต้น ข้ามช่องทาง ฮุกภายใน และ Plugin ห้าสิบรายการ; RSS ของการนำเข้า Plugin ที่รวมมาให้ ลูปทักทาย `channel-chat-baseline` กับ OpenAI จำลองซ้ำ ๆ คำสั่งเริ่มต้น CLI กับ Gateway ที่บูตแล้ว และโพรบประสิทธิภาพ smoke ของสถานะ SQLite เมื่อมีรายงานซอร์สผู้ให้บริการจำลองที่เผยแพร่ก่อนหน้าสำหรับ ref ที่ทดสอบ สรุปซอร์สจะเปรียบเทียบค่า RSS และฮีปปัจจุบันกับค่าพื้นฐานนั้น และทำเครื่องหมายการเพิ่มขึ้นอย่างมากของ RSS เป็น `watch` สรุปโพรบซอร์สในรูปแบบ Markdown อยู่ที่ `source/index.md` ภายในชุดรายงาน โดยมี JSON ดิบอยู่ข้างกัน

ทุกเลนอัปโหลดอาร์ติแฟกต์ GitHub ฉบับสมบูรณ์ ซึ่งรวมชุดการวินิจฉัย CPU ฮีป เทรซ และชุดที่บีบอัดแล้ว งานผู้เผยแพร่แยกต่างหากจะดาวน์โหลดและตรวจสอบอาร์ติแฟกต์เหล่านั้น จากนั้นสร้างโทเค็น GitHub App ของ ClawSweeper ที่มีอายุสั้นและจำกัดขอบเขตไว้เฉพาะเนื้อหา `openclaw/clawgrit-reports` แล้วส่งโทเค็นนี้ให้เฉพาะขั้นตอน Git push เท่านั้น งานนี้คอมมิต `report.json`, `report.md`, `index.md`, อาร์ติแฟกต์โพรบซอร์ส และข้อมูลเมตา/ผลรวมตรวจสอบของชุดไว้ภายใต้ `openclaw-performance/<tested-ref>/<run-id>-<attempt>/<lane>/`; ส่วนคลังการวินิจฉัยฉบับเต็มจะคงอยู่ในอาร์ติแฟกต์ Actions ที่ลิงก์ไว้ ผู้เผยแพร่จะปฏิเสธไฟล์รายงานใด ๆ ที่มีขนาดเกิน 50 MB ก่อนพยายามพุช ตัวชี้ ref ที่ทดสอบในปัจจุบันคือ `openclaw-performance/<tested-ref>/latest-<lane>.json` การรันตามกำหนดเวลาและการสั่งรัน `profile=release` จะล้มเหลวหากสร้างโทเค็นแอปหรือเผยแพร่รายงานไม่สำเร็จ การสั่งรันด้วยตนเองที่ไม่ใช่รุ่นเผยแพร่จะถือการเผยแพร่เป็นเพียงคำแนะนำ และเก็บอาร์ติแฟกต์ GitHub ไว้เมื่อการตรวจสอบสิทธิ์หรือการเผยแพร่ล้มเหลว ค่าพื้นฐานซอร์สก่อนหน้าจะถูกดึงแบบไม่ระบุตัวตนจากที่เก็บรายงานสาธารณะ ดังนั้นการดึงค่าพื้นฐานสำเร็จไม่ได้พิสูจน์ว่าการตรวจสอบสิทธิ์ของผู้เผยแพร่สำเร็จ

## การตรวจสอบรุ่นเผยแพร่แบบเต็ม

`Full Release Validation` คือเวิร์กโฟลว์ครอบคลุมแบบสั่งด้วยตนเองสำหรับ "เรียกใช้ทุกอย่างก่อนเผยแพร่รุ่น" โดยรับสาขา แท็ก หรือ SHA ของคอมมิตแบบเต็ม สั่งรันเวิร์กโฟลว์ `CI` แบบสั่งด้วยตนเองด้วยเป้าหมายนั้น (รวม Android) สั่งรัน `Plugin Prerelease` เพื่อพิสูจน์ Plugin/แพ็กเกจ/สแตติก/Docker ที่ใช้เฉพาะรุ่นเผยแพร่ สั่งรัน `OpenClaw Performance` กับ SHA เป้าหมาย และสั่งรัน `OpenClaw Release Checks` สำหรับ smoke การติดตั้ง การยอมรับแพ็กเกจ การตรวจสอบแพ็กเกจข้ามระบบปฏิบัติการ ความสอดคล้องของ QA Lab, Matrix, Telegram และเลน Discord, WhatsApp และ Slack ที่มีเกต (การเรนเดอร์สกอร์การ์ดระดับความพร้อมที่เป็นเพียงคำแนะนำสามารถเลือกเปิดได้ผ่าน `run_maturity_scorecard`) โปรไฟล์เสถียรและแบบเต็มจะรวมความครอบคลุม live/E2E อย่างละเอียดและการทดสอบ soak ของเส้นทางรุ่นเผยแพร่ Docker เสมอ ส่วนโปรไฟล์เบต้าสามารถเลือกเปิดได้ด้วย `run_release_soak=true` E2E ของ Telegram สำหรับแพ็กเกจมาตรฐานทำงานภายใน Package Acceptance ดังนั้นตัวเลือกรุ่นเต็มจะไม่เริ่มตัวโพล live ซ้ำ หลังเผยแพร่ ให้ส่ง `release_package_spec` เพื่อใช้แพ็กเกจ npm ที่เผยแพร่แล้วซ้ำในการตรวจสอบรุ่นเผยแพร่ Package Acceptance, Docker, ข้ามระบบปฏิบัติการ และ Telegram โดยไม่สร้างใหม่ ใช้ `npm_telegram_package_spec` เฉพาะสำหรับการรัน Telegram ซ้ำแบบเจาะจงกับแพ็กเกจที่เผยแพร่แล้ว เลนแพ็กเกจ live ของ Plugin Codex ใช้สถานะที่เลือกเดียวกันโดยค่าเริ่มต้น: `release_package_spec=openclaw@<tag>` ที่เผยแพร่แล้วจะกำหนด `codex_plugin_spec=npm:@openclaw/codex@<tag>` ส่วนการรันจาก SHA/อาร์ติแฟกต์จะแพ็ก `extensions/codex` จาก ref ที่เลือก ตั้งค่า `codex_plugin_spec` อย่างชัดเจนสำหรับซอร์ส Plugin แบบกำหนดเอง เช่นข้อกำหนด `npm:`, `npm-pack:` หรือ `git:` การพิสูจน์เอเจนต์ live ของเลนนี้จะส่งความคืบหน้าที่มองเห็นได้ ดำเนินการต่อผ่านการอ่านเวิร์กสเปซแบบสุ่มและการเขียนอาร์ติแฟกต์ที่ตรงตามกำหนดทุกประการ แล้วจึงส่งสถานะเสร็จสิ้น

ดู [การตรวจสอบรุ่นเผยแพร่แบบเต็ม](/th/reference/full-release-validation) สำหรับ
เมทริกซ์ขั้นตอน ชื่องานเวิร์กโฟลว์ที่แน่นอน ความแตกต่างของโปรไฟล์ อาร์ติแฟกต์ และ
ตัวจัดการการรันซ้ำแบบเจาะจง

`OpenClaw Release Publish` คือเวิร์กโฟลว์เผยแพร่แบบเปลี่ยนแปลงสถานะที่สั่งด้วยตนเอง ให้สั่งเผยแพร่
เบต้าและรุ่นเสถียรตามปกติจาก `main` ที่เชื่อถือได้ หลังจากมีแท็กรุ่นเผยแพร่แล้ว
และหลังจากการตรวจสอบเบื้องต้นของ npm สำหรับ OpenClaw สำเร็จ (การตรวจสอบเบื้องต้นจะเรียกใช้
`pnpm plugins:sync:check` เป็นหนึ่งในการตรวจสอบ) แท็กยังคงเลือก
คอมมิตของรุ่นเผยแพร่ที่แน่นอน รวมถึงคอมมิตบน `release/YYYY.M.PATCH`; การเผยแพร่แอลฟาของ Tideclaw
ยังคงใช้สาขาแอลฟาที่ตรงกัน เวิร์กโฟลว์นี้ต้องมี
`preflight_run_id` ที่บันทึกไว้ รวมถึง
`full_release_validation_run_id` ที่สำเร็จและ
`full_release_validation_run_attempt` ที่ตรงกันทุกประการ จากนั้นสั่งรัน `Plugin NPM Release` สำหรับ
แพ็กเกจ Plugin ทั้งหมดที่เผยแพร่ได้ สั่งรัน `Plugin ClawHub Release` สำหรับ
SHA รุ่นเผยแพร่เดียวกัน และหลังจากนั้นเท่านั้นจึงสั่งรัน `OpenClaw NPM Release` การเผยแพร่รุ่นเสถียรยัง
ต้องมี `windows_node_tag` ที่ตรงกันทุกประการ; เวิร์กโฟลว์จะตรวจสอบรุ่นเผยแพร่ซอร์สของ Windows
และเปรียบเทียบตัวติดตั้ง x64/ARM64 กับอินพุต `windows_node_installer_digests`
ที่ตัวเลือกรุ่นอนุมัติ ก่อนเรียกเวิร์กโฟลว์ย่อยสำหรับเผยแพร่ใด ๆ จากนั้นจึงโปรโมต
และตรวจสอบไดเจสต์ตัวติดตั้งที่ตรึงไว้ชุดเดิม รวมทั้งสัญญาของอาร์ติแฟกต์ประกอบ
และผลรวมตรวจสอบที่ตรงกันทุกประการ ก่อนเผยแพร่ฉบับร่างรุ่นเผยแพร่บน GitHub
การซ่อมแซมเฉพาะ Plugin แบบเจาะจงใช้ `plugin_publish_scope=selected` พร้อม
รายการแพ็กเกจที่ไม่ว่าง การรัน `all-publishable` เฉพาะ Plugin ต้องใช้หลักฐาน
การตรวจสอบเบื้องต้น npm แบบเปลี่ยนแปลงไม่ได้และการตรวจสอบรุ่นเผยแพร่แบบเต็มชุดเดียวกับการเผยแพร่คอร์

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH-beta.N \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

สำหรับการพิสูจน์คอมมิตที่ตรึงไว้บนสาขาที่เปลี่ยนแปลงรวดเร็ว ให้ใช้ตัวช่วยแทน
`gh workflow run ... --ref main -f ref=<sha>`:

```bash
pnpm ci:full-release --sha <full-sha>
```

ref สำหรับสั่งรันเวิร์กโฟลว์ GitHub ต้องเป็นสาขาหรือแท็ก ไม่ใช่ SHA ของคอมมิตดิบ
ตัวช่วยจะพุชสาขา `release-ci/<sha>-...` ชั่วคราวที่ SHA ของเวิร์กโฟลว์
`main` ที่เชื่อถือได้ ส่ง SHA เป้าหมายที่ร้องขอผ่านอินพุต `ref`
ของเวิร์กโฟลว์ ใช้หลักฐานเป้าหมายตรงกันแบบเข้มงวดซ้ำเมื่อมี ตรวจสอบว่า
`headSha` ของเวิร์กโฟลว์ย่อยทุกตัวตรงกับ SHA ของเวิร์กโฟลว์ที่เชื่อถือได้ และลบสาขาชั่วคราว
เมื่อการรันเสร็จสิ้น ส่ง `-f reuse_evidence=false` เพื่อบังคับให้
ตรวจสอบใหม่ เวิร์กโฟลว์ตรวจสอบแบบครอบคลุมจะล้มเหลวด้วย หากเวิร์กโฟลว์ย่อยใดทำงานด้วย
SHA ของเวิร์กโฟลว์อื่น

`release_profile` ควบคุมขอบเขต live/ผู้ให้บริการที่ส่งไปยังการตรวจสอบรุ่นเผยแพร่
เวิร์กโฟลว์รุ่นเผยแพร่แบบสั่งด้วยตนเองมีค่าเริ่มต้นเป็น `stable`; ใช้ `full` เฉพาะเมื่อ
ต้องการเมทริกซ์ผู้ให้บริการ/สื่อแบบกว้างที่เป็นเพียงคำแนะนำโดยตั้งใจ การตรวจสอบรุ่นเสถียรและรุ่นเต็ม
จะเรียกใช้การทดสอบ live/E2E อย่างละเอียดและ soak ของเส้นทางรุ่นเผยแพร่ Docker เสมอ;
โปรไฟล์เบต้าสามารถเลือกเปิดได้ด้วย `run_release_soak=true`

- `beta` คงเลน OpenAI/คอร์ที่เร็วที่สุดและสำคัญต่อรุ่นเผยแพร่ไว้
- `stable` เพิ่มชุดผู้ให้บริการ/แบ็กเอนด์ที่เสถียร
- `full` เรียกใช้เมทริกซ์ผู้ให้บริการ/สื่อแบบกว้างที่เป็นเพียงคำแนะนำ

เวิร์กโฟลว์ครอบคลุมจะบันทึก ID การรันของเวิร์กโฟลว์ย่อยที่สั่ง และงานสุดท้าย `Verify full validation` จะตรวจสอบผลสรุปปัจจุบันของการรันย่อยอีกครั้ง พร้อมต่อท้ายตารางงานที่ใช้เวลานานที่สุดสำหรับการรันย่อยแต่ละรายการ หากมีการรันเวิร์กโฟลว์ย่อยซ้ำและเปลี่ยนเป็นผ่าน ให้รันซ้ำเฉพาะงานตรวจสอบของเวิร์กโฟลว์แม่ เพื่อรีเฟรชผลลัพธ์ครอบคลุมและสรุปเวลา

สำหรับการกู้คืน ทั้ง `Full Release Validation` และ `OpenClaw Release Checks` รองรับ `rerun_group` ใช้ `all` สำหรับตัวเลือกรุ่นเผยแพร่ ใช้ `ci` สำหรับเวิร์กโฟลว์ย่อย CI แบบเต็มตามปกติเท่านั้น ใช้ `plugin-prerelease` สำหรับเวิร์กโฟลว์ย่อยก่อนเผยแพร่ Plugin เท่านั้น ใช้ `performance` สำหรับเวิร์กโฟลว์ย่อย OpenClaw Performance เท่านั้น ใช้ `release-checks` สำหรับเวิร์กโฟลว์ย่อยรุ่นเผยแพร่ทั้งหมด หรือใช้กลุ่มที่แคบกว่า ได้แก่ `install-smoke`, `cross-os`, `live-e2e`, `package`, `qa`, `qa-parity`, `qa-live` หรือ `npm-telegram` บนเวิร์กโฟลว์ครอบคลุม วิธีนี้ช่วยจำกัดขอบเขตการรันกล่องรุ่นเผยแพร่ที่ล้มเหลวซ้ำหลังแก้ไขแบบเจาะจง สำหรับเลนข้ามระบบปฏิบัติการที่ล้มเหลวหนึ่งเลน ให้ใช้ `rerun_group=cross-os` ร่วมกับ `cross_os_suite_filter` เช่น `windows/packaged-upgrade`; คำสั่งข้ามระบบปฏิบัติการที่ใช้เวลานานจะส่งบรรทัด Heartbeat และสรุปการอัปเกรดแพ็กเกจจะรวมเวลาของแต่ละเฟส เลน QA ของ Matrix และ Telegram ที่เลือกจะบล็อกการตรวจสอบรุ่นเผยแพร่ตามปกติ เช่นเดียวกับเกตมาตรฐานสำหรับความครอบคลุมของเครื่องมือรันไทม์ ความสอดคล้องของ QA ความสอดคล้องของรันไทม์ และเลน live ของ Discord, WhatsApp และ Slack ที่มีเกตเป็นเพียงคำแนะนำ

`OpenClaw Release Checks` ใช้ ref ของเวิร์กโฟลว์ที่เชื่อถือได้เพื่อแปลง ref ที่เลือกเพียงครั้งเดียวเป็น tarball `release-package-under-test` จากนั้นส่งอาร์ติแฟกต์นั้นไปยังการตรวจสอบข้ามระบบปฏิบัติการและ Package Acceptance รวมถึงเวิร์กโฟลว์ Docker ของเส้นทางรุ่นเผยแพร่ live/E2E เมื่อมีการทดสอบ soak วิธีนี้ทำให้ไบต์ของแพ็กเกจสอดคล้องกันในทุกกล่องรุ่นเผยแพร่ และหลีกเลี่ยงการแพ็กตัวเลือกรุ่นเดียวกันซ้ำในงานย่อยหลายงาน สำหรับเลน live ของ Plugin npm Codex การตรวจสอบรุ่นเผยแพร่จะส่งข้อกำหนด Plugin ที่เผยแพร่แล้วและตรงกันซึ่งได้มาจาก `release_package_spec` ส่ง `codex_plugin_spec` ที่ผู้ปฏิบัติงานกำหนด หรือปล่อยอินพุตว่างเพื่อให้สคริปต์ Docker แพ็ก Plugin Codex จากเช็กเอาต์ที่เลือก

การรัน `Full Release Validation` ซ้ำสำหรับ `ref=main` และ `rerun_group=all`
จะแทนที่เวิร์กโฟลว์ครอบคลุมที่เก่ากว่า ตัวตรวจสอบของเวิร์กโฟลว์แม่จะยกเลิกเวิร์กโฟลว์ย่อยใด ๆ ที่
สั่งรันไปแล้วเมื่อเวิร์กโฟลว์แม่ถูกยกเลิก เพื่อให้การตรวจสอบ main ที่ใหม่กว่า
ไม่ต้องรอหลังการรันตรวจสอบรุ่นเผยแพร่เก่าที่ใช้เวลาสองชั่วโมง การตรวจสอบสาขา/แท็กรุ่นเผยแพร่
และกลุ่มการรันซ้ำแบบเจาะจงจะคง `cancel-in-progress: false` ไว้

## ชาร์ด live และ E2E

เวิร์กโฟลว์ย่อย live/E2E ของรุ่นเผยแพร่ยังคงความครอบคลุม `pnpm test:live` แบบเนทีฟที่กว้าง แต่เรียกใช้เป็นชาร์ดที่มีชื่อผ่าน `scripts/test-live-shard.mjs` แทนงานแบบอนุกรมงานเดียว:

- `native-live-src-agents` และ `native-live-src-agents-zai-coding`
- `native-live-src-gateway-core`
- งาน `native-live-src-gateway-profiles` ที่กรองตามผู้ให้บริการ
- `native-live-src-gateway-backends`
- `native-live-src-infra`
- `native-live-test`
- `native-live-extensions-a-k`
- `native-live-extensions-l-n`
- `native-live-extensions-moonshot`
- `native-live-extensions-openai`
- `native-live-extensions-o-z-other`
- `native-live-extensions-xai`
- ชาร์ดเสียง/วิดีโอของสื่อที่แยกกัน และชาร์ดเพลงที่กรองตามผู้ให้บริการ

วิธีนี้คงความครอบคลุมไฟล์ชุดเดิมไว้ พร้อมช่วยให้รันซ้ำและวินิจฉัยความล้มเหลวของผู้ให้บริการ live ที่ทำงานช้าได้ง่ายขึ้น ชื่อชาร์ดรวม `native-live-src-gateway`, `native-live-extensions-o-z`, `native-live-extensions-media` และ `native-live-extensions-media-music` ยังคงใช้ได้สำหรับการรันซ้ำแบบครั้งเดียวด้วยตนเอง

ชาร์ดสื่อ live แบบเนทีฟทำงานใน `ghcr.io/openclaw/openclaw-live-media-runner:ubuntu-24.04` ซึ่งสร้างโดยเวิร์กโฟลว์ `Live Media Runner Image` อิมเมจดังกล่าวติดตั้ง `ffmpeg` และ `ffprobe` ไว้ล่วงหน้า; งานสื่อจะเพียงตรวจสอบไบนารีก่อนตั้งค่า ให้ชุดทดสอบ live ที่อาศัย Docker ทำงานบนรันเนอร์ Blacksmith ปกติ — งานคอนเทนเนอร์ไม่เหมาะสำหรับเรียกใช้การทดสอบ Docker ซ้อน

ชาร์ดโมเดล/แบ็กเอนด์ live ที่อาศัย Docker ใช้อิมเมจ `ghcr.io/openclaw/openclaw-live-test:<sha>-<extensions>` ร่วมกันแยกต่างหากสำหรับแต่ละคอมมิตที่เลือก เวิร์กโฟลว์รุ่นเผยแพร่ live จะสร้างและพุชอิมเมจนั้นครั้งเดียว จากนั้นชาร์ดโมเดล Docker live, Gateway ที่แบ่งชาร์ดตามผู้ให้บริการ, แบ็กเอนด์ CLI, การผูก ACP และชุดทดสอบ Codex จะทำงานด้วย `OPENCLAW_SKIP_DOCKER_BUILD=1` ชาร์ด Gateway Docker มีขีดจำกัด `timeout` ระดับสคริปต์ที่กำหนดไว้อย่างชัดเจนและต่ำกว่าเวลาหมดของงานเวิร์กโฟลว์ เพื่อให้คอนเทนเนอร์หรือเส้นทางล้างข้อมูลที่ค้างล้มเหลวอย่างรวดเร็วแทนที่จะใช้โควตาการตรวจสอบรุ่นเผยแพร่ทั้งหมด หากชาร์ดเหล่านั้นสร้างเป้าหมาย Docker ของซอร์สฉบับเต็มใหม่แยกกัน การรันรุ่นเผยแพร่มีการกำหนดค่าผิดและจะเสียเวลาไปกับการสร้างอิมเมจซ้ำ

## การยอมรับแพ็กเกจ

ใช้ `Package Acceptance` เมื่อคำถามคือ "แพ็กเกจ OpenClaw ที่ติดตั้งได้นี้ทำงานเป็นผลิตภัณฑ์หรือไม่" ซึ่งแตกต่างจาก CI ปกติ: CI ปกติตรวจสอบแผนผังซอร์ส ส่วนการยอมรับแพ็กเกจจะตรวจสอบ tarball เดียวผ่านชุดทดสอบ Docker E2E เดียวกับที่ผู้ใช้เรียกใช้หลังติดตั้งหรืออัปเดต

### งาน

1. `resolve_package` เช็กเอาต์ `workflow_ref` แก้ไขตัวเลือกแพ็กเกจหนึ่งรายการ เขียน `.artifacts/docker-e2e-package/openclaw-current.tgz` เขียน `.artifacts/docker-e2e-package/package-candidate.json` อัปโหลดทั้งสองรายการเป็นอาร์ติแฟกต์ `package-under-test` และพิมพ์แหล่งที่มา การอ้างอิงเวิร์กโฟลว์ การอ้างอิงแพ็กเกจ เวอร์ชัน SHA-256 และโปรไฟล์ในสรุปขั้นตอนของ GitHub
2. `package_integrity` ดาวน์โหลดอาร์ติแฟกต์ `package-under-test` และบังคับใช้สัญญาทาร์บอลแพ็กเกจสาธารณะด้วย `scripts/check-openclaw-package-tarball.mjs`
3. `docker_acceptance` เรียก `openclaw-live-and-e2e-checks-reusable.yml` ด้วย SHA ของแหล่งที่มาแพ็กเกจที่แก้ไขแล้ว (โดยใช้ `workflow_ref` เป็นทางเลือกสำรอง) และ `package_artifact_name=package-under-test` เวิร์กโฟลว์ที่นำกลับมาใช้ใหม่ได้จะดาวน์โหลดอาร์ติแฟกต์นั้น ตรวจสอบรายการภายในทาร์บอล เตรียมอิมเมจ Docker ที่อิงไดเจสต์แพ็กเกจเมื่อจำเป็น และเรียกใช้เลน Docker ที่เลือกกับแพ็กเกจนั้นแทนการแพ็กเช็กเอาต์ของเวิร์กโฟลว์ เมื่อโปรไฟล์เลือก `docker_lanes` เป้าหมายหลายรายการ เวิร์กโฟลว์ที่นำกลับมาใช้ใหม่ได้จะเตรียมแพ็กเกจและอิมเมจที่ใช้ร่วมกันเพียงครั้งเดียว จากนั้นกระจายเลนเหล่านั้นเป็นงาน Docker เป้าหมายที่ทำงานขนานกันพร้อมอาร์ติแฟกต์ที่ไม่ซ้ำกัน
4. `package_telegram` เรียก `NPM Telegram Beta E2E` หรือไม่ก็ได้ โดยจะทำงานเมื่อ `telegram_mode` ไม่ใช่ `none` และติดตั้งอาร์ติแฟกต์ `package-under-test` เดียวกันเมื่อ Package Acceptance แก้ไขได้หนึ่งรายการ ส่วนการสั่งงาน Telegram แบบสแตนด์อโลนยังคงติดตั้งสเปก npm ที่เผยแพร่แล้วได้
5. `summary` ทำให้เวิร์กโฟลว์ล้มเหลว หากการแก้ไขแพ็กเกจ ความสมบูรณ์ การยอมรับของ Docker หรือเลน Telegram ที่เป็นตัวเลือกล้มเหลว อินพุต `advisory` จะลดระดับความล้มเหลวในการยอมรับให้เป็นคำเตือนสำหรับผู้เรียกใช้งานเชิงให้คำแนะนำ

### แหล่งที่มาของตัวเลือก

- `source=npm` ยอมรับเฉพาะ `openclaw@extended-stable`, `openclaw@beta`, `openclaw@latest` หรือเวอร์ชันรีลีส OpenClaw แบบตรงกันทุกประการ เช่น `openclaw@2026.4.27-beta.2` ใช้ตัวเลือกนี้สำหรับการยอมรับ extended-stable, prerelease หรือ stable ที่เผยแพร่แล้ว
- `source=ref` แพ็ก branch, tag หรือ SHA ของ commit แบบเต็มของ `package_ref` ที่เชื่อถือได้ ตัวแก้ไขจะดึง branch/tag ของ OpenClaw ตรวจสอบว่า commit ที่เลือกเข้าถึงได้จากประวัติ branch ของที่เก็บหรือแท็กรีลีส ติดตั้งการขึ้นต่อกันใน worktree ที่แยกออกมา และแพ็กด้วย `scripts/package-openclaw-for-docker.mjs`
- `source=url` ดาวน์โหลด `.tgz` ผ่าน HTTPS สาธารณะ โดยต้องระบุ `package_sha256` พาธนี้จะปฏิเสธข้อมูลประจำตัวใน URL, พอร์ต HTTPS ที่ไม่ใช่ค่าเริ่มต้น, ชื่อโฮสต์หรือ IP ที่แก้ไขได้ซึ่งเป็นแบบส่วนตัว/ภายใน/ใช้งานเฉพาะ และการเปลี่ยนเส้นทางที่อยู่นอกนโยบายความปลอดภัยสาธารณะเดียวกัน
- `source=trusted-url` ดาวน์โหลด `.tgz` ผ่าน HTTPS จากนโยบายแหล่งที่มาที่เชื่อถือได้ซึ่งตั้งชื่อไว้ใน `.github/package-trusted-sources.json` โดยต้องระบุ `package_sha256` และ `trusted_source_id` ใช้ตัวเลือกนี้เฉพาะกับมิเรอร์ระดับองค์กรหรือที่เก็บแพ็กเกจส่วนตัวที่ผู้ดูแลเป็นเจ้าของ ซึ่งต้องใช้โฮสต์ พอร์ต คำนำหน้าพาธ โฮสต์เปลี่ยนเส้นทาง หรือการแก้ไขเครือข่ายส่วนตัวที่กำหนดค่าไว้ หากนโยบายประกาศการตรวจสอบสิทธิ์แบบ bearer เวิร์กโฟลว์จะใช้ secret `OPENCLAW_TRUSTED_PACKAGE_TOKEN` แบบตายตัว ส่วนข้อมูลประจำตัวที่ฝังใน URL ยังคงถูกปฏิเสธ
- `source=artifact` ดาวน์โหลด `.tgz` หนึ่งรายการจาก `artifact_run_id` และ `artifact_name` โดย `package_sha256` เป็นตัวเลือก แต่ควรระบุสำหรับอาร์ติแฟกต์ที่แชร์ภายนอก

แยก `workflow_ref` และ `package_ref` ออกจากกัน `workflow_ref` คือโค้ดเวิร์กโฟลว์/ชุดทดสอบที่เชื่อถือได้ซึ่งเรียกใช้การทดสอบ ส่วน `package_ref` คือ commit ของแหล่งที่มาที่จะถูกแพ็กเมื่อ `source=ref` วิธีนี้ทำให้ชุดทดสอบปัจจุบันสามารถตรวจสอบ commit ของแหล่งที่มาที่เชื่อถือได้รุ่นเก่า โดยไม่ต้องเรียกใช้ตรรกะเวิร์กโฟลว์เก่า

### โปรไฟล์ชุดทดสอบ

- `smoke` — `npm-onboard-channel-agent`, `gateway-network`, `config-reload`
- `package` — `npm-onboard-channel-agent`, `doctor-switch`, `update-channel-switch`, `skill-install`, `update-corrupt-plugin`, `upgrade-survivor`, `published-upgrade-survivor`, `root-managed-vps-upgrade`, `update-restart-auth`, `plugins-offline`, `plugin-update`
- `product` — ชุด `package` ที่ใช้ความครอบคลุม `plugins` แบบสดแทน `plugins-offline` รวมถึง `mcp-channels`, `cron-mcp-cleanup`, `openai-web-search-minimal`, `openwebui`
- `full` — ส่วนย่อยของพาธรีลีส Docker แบบเต็มพร้อม OpenWebUI
- `custom` — `docker_lanes` แบบตรงกันทุกประการ ซึ่งจำเป็นเมื่อ `suite_profile=custom`

โปรไฟล์ `package` ใช้ความครอบคลุม Plugin แบบออฟไลน์ เพื่อไม่ให้การตรวจสอบแพ็กเกจที่เผยแพร่แล้วขึ้นอยู่กับความพร้อมใช้งานของ ClawHub แบบสด เลน Telegram ที่เป็นตัวเลือกจะนำอาร์ติแฟกต์ `package-under-test` กลับมาใช้ใน `NPM Telegram Beta E2E` โดยยังคงพาธสเปก npm ที่เผยแพร่แล้วไว้สำหรับการสั่งงานแบบสแตนด์อโลน

สำหรับนโยบายเฉพาะด้านการอัปเดตและการทดสอบ Plugin รวมถึงคำสั่งภายในเครื่อง
เลน Docker, อินพุต Package Acceptance, ค่าเริ่มต้นของรีลีส และการคัดแยกความล้มเหลว
โปรดดู [การทดสอบการอัปเดตและ Plugin](/th/help/testing-updates-plugins)

การตรวจสอบรีลีสเรียก Package Acceptance ด้วย `source=artifact`, อาร์ติแฟกต์แพ็กเกจรีลีสที่เตรียมไว้, `suite_profile=custom`, `docker_lanes='doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor root-managed-vps-upgrade update-restart-auth plugins-offline plugin-update plugin-binding-command-escape'` และ `telegram_mode=mock-openai` วิธีนี้ทำให้การย้ายแพ็กเกจ การอัปเดต การติดตั้ง skill ของ ClawHub แบบสด การล้างการขึ้นต่อกันของ Plugin ที่ล้าสมัย การซ่อมแซมการติดตั้ง Plugin ที่กำหนดค่าไว้ Plugin แบบออฟไลน์ การอัปเดต Plugin และหลักฐาน Telegram ใช้ทาร์บอลแพ็กเกจที่แก้ไขแล้วรายการเดียวกัน ตั้งค่า `release_package_spec` ใน Full Release Validation หรือ OpenClaw Release Checks หลังเผยแพร่เบตา เพื่อเรียกใช้เมทริกซ์เดียวกันกับแพ็กเกจ npm ที่จัดส่งแล้วโดยไม่สร้างใหม่ ตั้งค่า `package_acceptance_package_spec` เฉพาะเมื่อ Package Acceptance ต้องใช้แพ็กเกจที่ต่างจากส่วนอื่นของการตรวจสอบรีลีส การตรวจสอบรีลีสข้ามระบบปฏิบัติการยังคงครอบคลุมการเริ่มต้นใช้งาน ตัวติดตั้ง และพฤติกรรมแพลตฟอร์มเฉพาะระบบปฏิบัติการ ส่วนการตรวจสอบผลิตภัณฑ์ด้านแพ็กเกจ/การอัปเดตควรเริ่มด้วย Package Acceptance

เลน Docker `published-upgrade-survivor` ตรวจสอบค่าฐานของแพ็กเกจที่เผยแพร่แล้วหนึ่งรายการต่อการรันในพาธรีลีสแบบบล็อก ใน Package Acceptance ทาร์บอล `package-under-test` ที่แก้ไขแล้วจะเป็นตัวเลือกเสมอ และ `published_upgrade_survivor_baseline` จะเลือกค่าฐานที่เผยแพร่แล้วสำหรับใช้เป็นทางเลือกสำรอง โดยมีค่าเริ่มต้นเป็น `openclaw@latest` คำสั่งเรียกใช้เลนที่ล้มเหลวซ้ำจะคงค่าฐานนั้นไว้ Full Release Validation ที่ใช้ `run_release_soak=true` หรือ `release_profile=full` จะตั้งค่า `published_upgrade_survivor_baselines='last-stable-4 2026.4.23 2026.5.2 2026.4.15'` และ `published_upgrade_survivor_scenarios=reported-issues` เพื่อขยายให้ครอบคลุมรีลีส npm แบบ stable ล่าสุดสี่รายการ รวมถึงรีลีสขอบเขตความเข้ากันได้ของ Plugin ที่ปักหมุดไว้ และฟิกซ์เจอร์ตามรูปแบบปัญหาสำหรับการกำหนดค่า Feishu, ไฟล์ bootstrap/persona ที่เก็บรักษาไว้, การติดตั้ง Plugin ของ OpenClaw ที่กำหนดค่าแล้ว, พาธบันทึกที่ใช้เครื่องหมายทิลด์ และรากการขึ้นต่อกันของ Plugin รุ่นเก่าที่ล้าสมัย การเลือกผู้รอดจากการอัปเกรดที่เผยแพร่แล้วซึ่งมีหลายค่าฐานจะถูกแบ่งชาร์ดตามค่าฐานเป็นงานตัวรัน Docker เป้าหมายแยกกัน เวิร์กโฟลว์ `Update Migration` ที่แยกต่างหากใช้เลน Docker `update-migration` พร้อมค่าฐาน `all-since-2026.4.23` และสถานการณ์ `plugin-deps-cleanup` เมื่อโจทย์คือการล้างการอัปเดตที่เผยแพร่แล้วอย่างครอบคลุม ไม่ใช่ขอบเขตปกติของ Full Release CI การรันแบบรวมภายในเครื่องสามารถส่งสเปกแพ็กเกจที่ตรงกันทุกประการด้วย `OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPECS` คงไว้เพียงเลนเดียวด้วย `OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPEC` เช่น `openclaw@2026.4.15` หรือตั้งค่า `OPENCLAW_UPGRADE_SURVIVOR_SCENARIOS` สำหรับเมทริกซ์สถานการณ์ เลนที่เผยแพร่แล้วจะกำหนดค่าฐานด้วยสูตรคำสั่ง `openclaw config set` ที่ฝังไว้ บันทึกขั้นตอนของสูตรใน `summary.json` และตรวจสอบ `/healthz`, `/readyz` รวมถึงสถานะ RPC หลังเริ่ม Gateway เลนแพ็กเกจ Windows และเลนตัวติดตั้งใหม่ยังตรวจสอบด้วยว่าแพ็กเกจที่ติดตั้งแล้วสามารถนำเข้าโอเวอร์ไรด์การควบคุมเบราว์เซอร์จากพาธ Windows แบบสัมบูรณ์ดิบได้ OpenAI agent-turn smoke ข้ามระบบปฏิบัติการจะใช้ค่าเริ่มต้นเป็น `OPENCLAW_CROSS_OS_OPENAI_MODEL` เมื่อตั้งค่าไว้ มิฉะนั้นจะใช้ `openai/gpt-5.6-luna` เพื่อให้หลักฐานการติดตั้งและ Gateway ใช้ระดับการทดสอบ GPT-5.6 ที่มีต้นทุนต่ำกว่า

### ช่วงความเข้ากันได้กับระบบเดิม

Package Acceptance มีช่วงความเข้ากันได้กับระบบเดิมแบบจำกัดสำหรับแพ็กเกจที่เผยแพร่ไปแล้ว แพ็กเกจจนถึง `2026.4.25` รวมถึง `2026.4.25-beta.*` อาจใช้พาธความเข้ากันได้ดังนี้:

- รายการ QA ส่วนตัวที่ทราบใน `dist/postinstall-inventory.json` อาจชี้ไปยังไฟล์ที่ไม่รวมอยู่ในทาร์บอล
- `doctor-switch` อาจข้ามกรณีย่อยด้านการคงอยู่ของ `gateway install --wrapper` เมื่อแพ็กเกจไม่ได้เปิดเผยแฟล็กนั้น
- `update-channel-switch` อาจตัด `patchedDependencies` ของ pnpm ที่หายไปออกจากฟิกซ์เจอร์ git จำลองที่สร้างจากทาร์บอล และอาจบันทึก `update.channel` ที่คงอยู่แต่หายไป
- การทดสอบ smoke ของ Plugin อาจอ่านตำแหน่งบันทึกการติดตั้งรุ่นเก่า หรือยอมรับการไม่มีการคงอยู่ของบันทึกการติดตั้งจาก marketplace
- `plugin-update` อาจอนุญาตให้ย้ายข้อมูลเมตาของการกำหนดค่า ขณะที่ยังคงกำหนดให้บันทึกการติดตั้งและพฤติกรรมที่ไม่ติดตั้งซ้ำต้องไม่เปลี่ยนแปลง

แพ็กเกจ `2026.4.26` ที่เผยแพร่แล้วอาจเตือนเกี่ยวกับไฟล์ประทับข้อมูลเมตาการสร้างภายในเครื่องที่จัดส่งไปแล้ว และแพ็กเกจจนถึง `2026.5.20` อาจเตือนแทนที่จะล้มเหลวเมื่อไม่มี `npm-shrinkwrap.json` แพ็กเกจรุ่นหลังต้องเป็นไปตามสัญญาสมัยใหม่ โดยเงื่อนไขเดียวกันจะทำให้ล้มเหลวแทนการเตือนหรือข้าม

### ตัวอย่าง

```bash
# ตรวจสอบแพ็กเกจเบตาปัจจุบันด้วยความครอบคลุมระดับผลิตภัณฑ์
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f telegram_mode=mock-openai

# ตรวจสอบแพ็กเกจ extended-stable ที่เผยแพร่แล้วด้วยความครอบคลุมของแพ็กเกจ
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@extended-stable \
  -f suite_profile=package \
  -f telegram_mode=mock-openai

# แพ็กและตรวจสอบ branch รีลีสด้วยชุดทดสอบปัจจุบัน
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=ref \
  -f package_ref=release/YYYY.M.PATCH \
  -f suite_profile=package \
  -f telegram_mode=mock-openai

# ตรวจสอบ URL ของทาร์บอล ต้องระบุ SHA-256 สำหรับ source=url
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=url \
  -f package_url=https://example.com/openclaw-current.tgz \
  -f package_sha256=<64-char-sha256> \
  -f suite_profile=smoke

# ตรวจสอบทาร์บอลจากนโยบายมิเรอร์ส่วนตัวที่เชื่อถือได้ซึ่งตั้งชื่อไว้
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=trusted-url \
  -f trusted_source_id=enterprise-artifactory \
  -f package_url=https://packages.example.internal:8443/artifactory/openclaw/openclaw-current.tgz \
  -f package_sha256=<64-char-sha256> \
  -f suite_profile=smoke

# นำทาร์บอลที่อัปโหลดโดยการรัน Actions อื่นกลับมาใช้
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=artifact \
  -f artifact_run_id=<run-id> \
  -f artifact_name=package-under-test \
  -f suite_profile=custom \
  -f docker_lanes='install-e2e plugin-update'
```

เมื่อแก้ไขข้อบกพร่องของการรัน Package Acceptance ที่ล้มเหลว ให้เริ่มจากสรุป `resolve_package` เพื่อยืนยันแหล่งที่มาแพ็กเกจ เวอร์ชัน และ SHA-256 จากนั้นตรวจสอบการรันลูก `docker_acceptance` และอาร์ติแฟกต์ Docker ได้แก่ `.artifacts/docker-tests/**/summary.json`, `failures.json`, บันทึกของเลน ระยะเวลาของแต่ละเฟส และคำสั่งเรียกใช้ซ้ำ ควรเลือกเรียกใช้โปรไฟล์แพ็กเกจที่ล้มเหลวหรือเลน Docker ที่ตรงกันทุกประการซ้ำ แทนการเรียกใช้การตรวจสอบรีลีสแบบเต็มซ้ำ

## การทดสอบ smoke ของการติดตั้ง

เวิร์กโฟลว์ `Install Smoke` จะไม่ทำงานบน pull request หรือการพุช `main` อีกต่อไป ทั้ง wrapper แบบรายคืน/ด้วยตนเองและการตรวจสอบรีลีสจะเรียกแกน `install-smoke-reusable.yml` แบบอ่านอย่างเดียว และทุกการรันจะใช้พาธการทดสอบ smoke ของการติดตั้งแบบเต็มบนตัวรันที่โฮสต์โดย GitHub:

- อิมเมจ smoke ของ Dockerfile ที่รากจะถูกสร้างหนึ่งครั้งต่อ SHA เป้าหมาย ผูกกับรุ่นแก้ไขของเวิร์กโฟลว์และความพยายามของผู้สร้างในอาร์ติแฟกต์ที่เปลี่ยนแปลงไม่ได้ จากนั้นโหลดโดยการทดสอบ smoke ของ CLI, การทดสอบ smoke ของ CLI ที่เอเจนต์ลบพื้นที่ทำงานที่ใช้ร่วมกัน, E2E เครือข่าย Gateway ของคอนเทนเนอร์ และการทดสอบ smoke ของอาร์กิวเมนต์การสร้าง Plugin `matrix` ที่รวมมา การทดสอบ smoke ของ Plugin จะตรวจสอบการสะท้อนการติดตั้งการขึ้นต่อกันขณะรัน และตรวจสอบว่า Plugin โหลดได้โดยไม่มีการวินิจฉัยการหลุดออกจาก entry
- การติดตั้งแพ็กเกจ QR และการทดสอบ smoke ของ Docker สำหรับตัวติดตั้ง/การอัปเดต (รวมถึงเลนตัวติดตั้ง Rocky Linux และเลนการอัปเดตเทียบกับค่าฐาน npm `update_baseline_version` ที่กำหนดค่าได้) จะทำงานเป็นงานแยกกัน เพื่อให้งานตัวติดตั้งไม่ต้องรอต่อจากการทดสอบ smoke ของอิมเมจราก

การทดสอบ smoke ของผู้ให้บริการอิมเมจสำหรับการติดตั้ง Bun แบบ global ที่ใช้เวลานานถูกควบคุมด้วยเกต `run_bun_global_install_smoke` แยกต่างหาก โดยทำงานตามกำหนดการรายคืน เปิดใช้งานเป็นค่าเริ่มต้นสำหรับการเรียกเวิร์กโฟลว์จากการตรวจสอบรีลีส และการ dispatch `Install Smoke` แบบแมนนวลสามารถเลือกเข้าร่วมได้ ส่วน CI ของ PR ปกติยังคงรันเลนทดสอบการถดถอยของตัวเรียกใช้ Bun แบบรวดเร็วสำหรับการเปลี่ยนแปลงที่เกี่ยวข้องกับ Node การทดสอบ Docker สำหรับ QR และตัวติดตั้งยังคงใช้ Dockerfile ที่มุ่งเน้นการติดตั้งของตนเอง

## Docker E2E ภายในเครื่อง

`pnpm test:docker:all` สร้างอิมเมจทดสอบแบบ live ที่ใช้ร่วมกันหนึ่งรายการไว้ล่วงหน้า แพ็ก OpenClaw หนึ่งครั้งเป็น npm tarball และสร้างอิมเมจ `scripts/e2e/Dockerfile` ที่ใช้ร่วมกันสองรายการ:

- รันเนอร์ Node/Git เปล่าสำหรับเลนตัวติดตั้ง/อัปเดต/การขึ้นต่อกันของ Plugin;
- อิมเมจที่ใช้งานได้ซึ่งติดตั้ง tarball เดียวกันลงใน `/app` สำหรับเลนการทำงานปกติ

ข้อกำหนดเลน Docker อยู่ใน `scripts/lib/docker-e2e-scenarios.mjs` ตรรกะตัววางแผนอยู่ใน `scripts/lib/docker-e2e-plan.mjs` และรันเนอร์จะดำเนินการเฉพาะแผนที่เลือก ตัวจัดกำหนดการเลือกอิมเมจต่อเลนด้วย `OPENCLAW_DOCKER_E2E_BARE_IMAGE` และ `OPENCLAW_DOCKER_E2E_FUNCTIONAL_IMAGE` จากนั้นรันเลนด้วย `OPENCLAW_SKIP_DOCKER_BUILD=1`

### เวิร์กโฟลว์ live/E2E ที่ใช้ซ้ำได้

เวิร์กโฟลว์ live/E2E ที่ใช้ซ้ำได้จะสอบถาม `scripts/test-docker-all.mjs --plan-json` ว่าต้องใช้แพ็กเกจ ชนิดอิมเมจ อิมเมจ live เลน และความครอบคลุมของข้อมูลประจำตัวใด จากนั้น `scripts/docker-e2e.mjs` จะแปลงแผนนั้นเป็นเอาต์พุตและสรุปของ GitHub โดยจะแพ็ก OpenClaw ผ่าน `scripts/package-openclaw-for-docker.mjs` ดาวน์โหลดอาร์ติแฟกต์แพ็กเกจของการรันปัจจุบัน หรือดาวน์โหลดอาร์ติแฟกต์แพ็กเกจจาก `package_artifact_run_id` แล้วตรวจสอบรายการภายใน tarball พาธเริ่มต้น `no-push-artifact` จะสร้างอิมเมจเปล่า/ใช้งานได้ที่ติดแท็กด้วยไดเจสต์แพ็กเกจผ่านแคชเลเยอร์ Docker ของ Blacksmith แพ็กไบต์ของอิมเมจที่ตรงกันเป็นอาร์ติแฟกต์เวิร์กโฟลว์ที่เปลี่ยนแปลงไม่ได้ และให้ผู้ใช้แต่ละรายตรวจสอบและโหลดอาร์ติแฟกต์นั้น ส่วน `existing-only` กำหนดให้ระบุรีเฟอเรนซ์ GHCR `docker_e2e_bare_image`/`docker_e2e_functional_image` อย่างชัดเจน และไม่สร้างหรือพุชเลย การดึงจากรีจิสทรีเหล่านั้นใช้ระยะหมดเวลาต่อครั้งที่จำกัดไว้ที่ 180 วินาที เพื่อให้สตรีมที่ค้างลองใหม่อย่างรวดเร็วแทนที่จะใช้เวลาส่วนใหญ่ของเส้นทางวิกฤตของ CI หลังจากการตรวจสอบตามกำหนดการสำเร็จ `openclaw-scheduled-live-checks.yml` จะส่งมานิเฟสต์อิมเมจที่ผ่านการทดสอบและเปลี่ยนแปลงไม่ได้ไปยังตัวเผยแพร่ที่เขียนแพ็กเกจแยกต่างหาก ส่วนผู้เรียกสำหรับรีลีสและพรีรีลีสแบบอ่านอย่างเดียวจะไม่ผ่านตัวเขียนนั้น

### ส่วนย่อยของพาธรีลีส

ความครอบคลุม Docker สำหรับรีลีสจะรันงานแบบแบ่งส่วนย่อยที่เล็กลงด้วย `OPENCLAW_SKIP_DOCKER_BUILD=1` เพื่อให้แต่ละส่วนย่อยตรวจสอบและโหลดเฉพาะชนิดอิมเมจที่มีอาร์ติแฟกต์รองรับซึ่งจำเป็น (หรือดึงภายใต้การใช้ซ้ำ `existing-only` ที่ระบุอย่างชัดเจน) และดำเนินการหลายเลนผ่านตัวจัดกำหนดการแบบถ่วงน้ำหนักเดียวกัน:

- `OPENCLAW_DOCKER_ALL_PROFILE=release-path`
- `OPENCLAW_DOCKER_ALL_CHUNK=core | package-update-openai | package-update-anthropic | package-update-core | plugins-runtime-plugins | plugins-runtime-services | plugins-runtime-install-a..h | openwebui`

ส่วนย่อย Docker สำหรับรีลีสปัจจุบันคือ `core`, `package-update-openai`, `package-update-anthropic`, `package-update-core`, `plugins-runtime-plugins`, `plugins-runtime-services`, `plugins-runtime-install-a` ถึง `plugins-runtime-install-h` และ `openwebui` โดย `package-update-openai` มีเลนแพ็กเกจ Plugin Codex แบบ live ซึ่งติดตั้งแพ็กเกจ OpenClaw รุ่นผู้สมัคร ติดตั้ง Plugin Codex จาก `codex_plugin_spec` หรือ tarball ของรีเฟอเรนซ์เดียวกันโดยมีการอนุมัติการติดตั้ง Codex CLI อย่างชัดเจน รันการตรวจสอบล่วงหน้าของ Codex CLI และรอบการทำงานของเอเจนต์ในเซสชันเดียวกัน จากนั้นรันรอบที่ใช้ระดับการคิดปานกลางโดยไม่ลองใหม่ ซึ่งส่งความคืบหน้า อ่านอินพุตพื้นที่ทำงานที่สุ่มไว้ เขียนอาร์ติแฟกต์ที่ตรงกันทุกประการ และส่งการเสร็จสิ้น `plugins-runtime-core`, `plugins-runtime` และ `plugins-integrations` ยังคงเป็นนามแฝงรวมของ Plugin/รันไทม์ นามแฝงเลน `install-e2e` ยังคงเป็นนามแฝงรวมสำหรับการรันใหม่แบบแมนนวลของทั้งสองเลนตัวติดตั้งผู้ให้บริการ

OpenWebUI ทำงานเป็นส่วนย่อย `openwebui` แบบสแตนด์อโลนบนรันเนอร์ Blacksmith ที่มีดิสก์ขนาดใหญ่โดยเฉพาะ เมื่อใดก็ตามที่ความครอบคลุมพาธรีลีสแบบเสถียรหรือแบบเต็มร้องขอ แม้ว่าเวิร์กโฟลว์ที่ใช้ซ้ำได้จะกำหนดเส้นทางงานที่รองรับไปยังรันเนอร์ที่โฮสต์โดย GitHub ก็ตาม การแยกการดึงอิมเมจภายนอกออกช่วยป้องกันไม่ให้อิมเมจขนาดใหญ่แย่งทรัพยากรกับอิมเมจแพ็กเกจและ Plugin ที่ใช้ร่วมกันใน `plugins-runtime-services`; ส่วนย่อยรวม Plugin/รันไทม์แบบเดิมยังคงมี OpenWebUI สำหรับการรันใหม่แบบแมนนวลที่เข้ากันได้ เลนอัปเดตช่องทางที่รวมมาให้จะลองใหม่หนึ่งครั้งสำหรับความล้มเหลวชั่วคราวของเครือข่าย npm

แต่ละส่วนย่อยอัปโหลด `.artifacts/docker-tests/` พร้อมบันทึกเลน เวลา `summary.json` `failures.json` เวลาของแต่ละเฟส JSON แผนตัวจัดกำหนดการ ตารางเลนช้า และคำสั่งรันใหม่รายเลน อินพุต `docker_lanes` ของเวิร์กโฟลว์จะรันเลนที่เลือกกับอิมเมจที่เตรียมไว้สำหรับการรันนั้นแทนงานแบบแบ่งส่วนย่อย ซึ่งจำกัดการดีบักเลนที่ล้มเหลวไว้ที่งาน Docker เป้าหมายเดียว หากเลนที่เลือกเป็นเลน Docker แบบ live งานเป้าหมายจะสร้างอิมเมจทดสอบแบบ live ภายในงานสำหรับการรันใหม่นั้น ตัวช่วยรันใหม่จะตรวจสอบ SHA เป้าหมายที่เลือกแบบตรงทั้งหมดของอาร์ติแฟกต์ความล้มเหลว และการ dispatch แบบแมนนวลจะแพ็กรีเฟอเรนซ์นั้นใหม่ เนื่องจากทูเพิลแพ็กเกจของเวิร์กโฟลว์ที่ใช้ซ้ำได้ภายในไม่ได้เป็นส่วนหนึ่งของสคีมา `workflow_dispatch` คำสั่งที่สร้างขึ้นจะรวมอินพุตอิมเมจที่เตรียมไว้และ `shared_image_policy=existing-only` เฉพาะเมื่ออินพุตเหล่านั้นมี GHCR รองรับ โดยละเว้นแท็กอาร์ติแฟกต์ภายในรันเนอร์เพื่อให้รันเนอร์ใหม่สร้างขึ้นใหม่ การแทนที่เป้าหมายอย่างชัดเจนจะตัดรีเฟอเรนซ์อิมเมจ GHCR ที่กู้คืนมาออก เว้นแต่อาร์ติแฟกต์จะพิสูจน์ว่าตรงกับค่าที่แทนที่ นอกจากนี้ยังละเว้นรีเฟอเรนซ์ข้อกำหนดเวิร์กโฟลว์ที่สร้างจากอาร์ติแฟกต์ เนื่องจากแบรนช์ชั่วคราวสำหรับรีลีสแบบเต็มจะถูกลบ การ dispatch จะใช้แบรนช์เริ่มต้นของรีโพซิทอรี เว้นแต่ผู้ดำเนินการจะแทนที่อย่างชัดเจน

```bash
pnpm test:docker:rerun <run-id>      # ดาวน์โหลดอาร์ติแฟกต์ Docker และพิมพ์คำสั่งรันใหม่แบบกำหนดเป้าหมายทั้งแบบรวม/รายเลน
pnpm test:docker:timings <summary>   # สรุปเลนช้าและเส้นทางวิกฤตของแต่ละเฟส
```

เวิร์กโฟลว์ live/E2E ตามกำหนดการจะรันชุด Docker ของพาธรีลีสแบบเต็มทุกวัน และหลังจากสำเร็จแล้ว จะเรียกใช้ตัวเผยแพร่ที่ระบุอย่างชัดเจนสำหรับอาร์ติแฟกต์อิมเมจที่ผ่านการทดสอบตรงกันทุกประการ

## พรีรีลีสของ Plugin

`Plugin Prerelease` เป็นความครอบคลุมผลิตภัณฑ์/แพ็กเกจที่มีต้นทุนสูงกว่า จึงเป็นเวิร์กโฟลว์แยกที่ dispatch โดย `Full Release Validation` หรือโดยผู้ดำเนินการอย่างชัดเจน pull request ปกติ การพุช `main` และการ dispatch CI แบบแมนนวลเดี่ยวจะปิดชุดดังกล่าวไว้ โดยกระจายการทดสอบ Plugin ที่รวมมาให้ระหว่างเวิร์กเกอร์ส่วนขยายแปดตัว งานชาร์ดส่วนขยายเหล่านั้นรันกลุ่มการกำหนดค่า Plugin พร้อมกันได้สูงสุดสองกลุ่ม โดยมีเวิร์กเกอร์ Vitest หนึ่งตัวต่อกลุ่มและฮีป Node ที่ใหญ่ขึ้น เพื่อไม่ให้ชุด Plugin ที่นำเข้าโมดูลจำนวนมากสร้างงาน CI เพิ่มเติม พาธพรีรีลีส Docker สำหรับรีลีสเท่านั้น (เปิดใช้งานโดยอินพุต `full_release_validation`) จะจัดกลุ่มเลน Docker เป้าหมายครั้งละสี่เลน เพื่อหลีกเลี่ยงการจองรันเนอร์หลายสิบตัวสำหรับงานที่ใช้เวลาหนึ่งถึงสามนาที นอกจากนี้เวิร์กโฟลว์ยังอัปโหลดอาร์ติแฟกต์ `plugin-inspector-advisory` เพื่อให้ข้อมูลจาก `@openclaw/plugin-inspector`; ข้อค้นพบของตัวตรวจสอบเป็นข้อมูลสำหรับการคัดแยกและไม่เปลี่ยนเกตพรีรีลีสของ Plugin ที่บล็อกการดำเนินการ

## QA Lab

QA Lab มีเลน CI โดยเฉพาะนอกเวิร์กโฟลว์หลักที่กำหนดขอบเขตอย่างชาญฉลาด ความเท่าเทียมเชิงเอเจนต์ซ้อนอยู่ภายใต้ชุดทดสอบ QA และรีลีสแบบกว้าง ไม่ใช่เวิร์กโฟลว์ PR แบบสแตนด์อโลน ใช้ `Full Release Validation` ร่วมกับ `rerun_group=qa-parity` เมื่อต้องการให้การตรวจสอบความเท่าเทียมทำงานไปกับการตรวจสอบแบบกว้าง

- เวิร์กโฟลว์ `QA-Lab - All Lanes` ทำงานทุกคืนบน `main` และเมื่อ dispatch แบบแมนนวล โดยแยกกระจายงานความเท่าเทียมแบบจำลอง รวมถึงงาน Matrix, Telegram, Discord, WhatsApp และ Slack แบบ live งานแบบ live ใช้สภาพแวดล้อม `qa-live-shared`; Telegram, Discord, WhatsApp และ Slack ใช้สัญญาเช่า Convex ส่วน Matrix จัดเตรียมข้อมูลประจำตัวภายในเครื่องแบบใช้แล้วทิ้ง

การตรวจสอบรีลีสจะรันเลนการขนส่ง Matrix และ Telegram แบบ live ด้วยผู้ให้บริการแบบจำลองที่กำหนดผลได้แน่นอนและโมเดลที่ระบุคุณสมบัติ mock (`mock-openai/gpt-5.6-luna` และ `mock-openai/gpt-5.6-luna-alt`) เพื่อแยกสัญญาช่องทางออกจากเวลาแฝงของโมเดลแบบ live และการเริ่มต้น Plugin ผู้ให้บริการตามปกติ Gateway การขนส่งแบบ live จะปิดใช้งานการค้นหาหน่วยความจำ เนื่องจากความเท่าเทียมของ QA ครอบคลุมพฤติกรรมหน่วยความจำแยกต่างหาก ส่วนการเชื่อมต่อผู้ให้บริการครอบคลุมโดยชุดทดสอบโมเดลแบบ live ผู้ให้บริการแบบเนทีฟ และผู้ให้บริการ Docker ที่แยกกัน

เกต Matrix ตามกำหนดการและสำหรับรีลีสใช้โฮสต์ชุด QA Lab ที่ใช้ร่วมกันและอะแดปเตอร์แบบ live กับสถานการณ์รีลีส ค่าเริ่มต้นของ CLI และอินพุตเวิร์กโฟลว์แบบแมนนวลยังคงเป็น `all`; การ dispatch `all` แบบแมนนวลจะแยกกระจายโปรไฟล์ `transport`, `media`, `e2ee-smoke`, `e2ee-deep` และ `e2ee-cli` เพื่อให้หลักฐาน 93 สถานการณ์อยู่ภายในระยะหมดเวลาต่องาน การ dispatch แบบแมนนวลที่เจาะจงจะเลือก `fast`, `release` หรือ `transport` ในงานเดียว

`OpenClaw Release Checks` ยังรันเลน QA Lab ที่สำคัญต่อรีลีสก่อนอนุมัติรีลีส โดยเกตความเท่าเทียมของ QA จะรันแพ็กผู้สมัครและแพ็กฐานเป็นงานเลนคู่ขนาน จากนั้นดาวน์โหลดอาร์ติแฟกต์ทั้งสองไปยังงานรายงานขนาดเล็กเพื่อเปรียบเทียบความเท่าเทียมขั้นสุดท้าย

สำหรับ PR ปกติ ให้ยึดตามหลักฐาน CI/การตรวจสอบที่กำหนดขอบเขต แทนการถือว่าความเท่าเทียมเป็นสถานะที่จำเป็น

## CodeQL

เวิร์กโฟลว์ `CodeQL` ตั้งใจให้เป็นเครื่องสแกนความปลอดภัยรอบแรกที่มีขอบเขตแคบ ไม่ใช่การกวาดตรวจทั้งรีโพซิทอรี การรันรายวัน แบบแมนนวล การพุช `main` และเกต pull request ที่ไม่ใช่ฉบับร่างจะสแกนโค้ดเวิร์กโฟลว์ Actions รวมถึงพื้นผิว JavaScript/TypeScript ที่มีความเสี่ยงสูงสุด ด้วยคิวรีความปลอดภัยที่มีความเชื่อมั่นสูงซึ่งกรองเฉพาะ `security-severity` ระดับสูง/วิกฤต

เกต pull request ยังคงมีน้ำหนักเบา โดยจะเริ่มเฉพาะการเปลี่ยนแปลงภายใต้ `.github/actions`, `.github/codeql`, `.github/workflows`, `packages`, `scripts`, `src` หรือพาธรันไทม์ของ Plugin ที่รวมมาให้และเป็นเจ้าของกระบวนการ และจะรันเมทริกซ์ความปลอดภัยที่มีความเชื่อมั่นสูงชุดเดียวกับเวิร์กโฟลว์ตามกำหนดการ CodeQL สำหรับ Android และ macOS ยังคงไม่รวมอยู่ในค่าเริ่มต้นของ PR

### หมวดหมู่ความปลอดภัย

| หมวดหมู่                                          | พื้นผิว                                                                                                                             |
| ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `/codeql-security-high/core-auth-secrets`         | เกณฑ์พื้นฐานของการยืนยันตัวตน ข้อมูลลับ แซนด์บ็อกซ์ Cron และ Gateway                                                                                  |
| `/codeql-security-high/channel-runtime-boundary`  | สัญญาการใช้งานหลักของช่องทาง รวมถึงรันไทม์ Plugin ของช่องทาง, Gateway, Plugin SDK, ข้อมูลลับ และจุดเชื่อมต่อสำหรับการตรวจสอบ              |
| `/codeql-security-high/network-ssrf-boundary`     | พื้นผิวนโยบาย SSRF ของ SSRF หลัก การแยกวิเคราะห์ IP ตัวป้องกันเครือข่าย การดึงข้อมูลจากเว็บ และ Plugin SDK                                                |
| `/codeql-security-high/mcp-process-tool-boundary` | เซิร์ฟเวอร์ MCP ตัวช่วยดำเนินการโปรเซส การส่งออก และด่านควบคุมการเรียกใช้เครื่องมือของเอเจนต์                                           |
| `/codeql-security-high/process-exec-boundary`     | เชลล์ภายในเครื่อง ตัวช่วยสร้างโปรเซส รันไทม์ Plugin แบบรวมที่เป็นเจ้าของโปรเซสย่อย และส่วนเชื่อมสคริปต์เวิร์กโฟลว์                             |
| `/codeql-security-high/plugin-trust-boundary`     | พื้นผิวความเชื่อถือของการติดตั้ง Plugin ตัวโหลด แมนิเฟสต์ รีจิสทรี การติดตั้งด้วยตัวจัดการแพ็กเกจ การโหลดซอร์ส และสัญญาแพ็กเกจ Plugin SDK |

### ชาร์ดความปลอดภัยเฉพาะแพลตฟอร์ม

- `CodeQL Android Critical Security` — ชาร์ดความปลอดภัย Android ตามกำหนดเวลา สร้างแอป Android ด้วยตนเองสำหรับ CodeQL บนรันเนอร์ Blacksmith Linux ขนาดเล็กที่สุดที่การตรวจสอบความสมเหตุสมผลของเวิร์กโฟลว์ยอมรับ อัปโหลดภายใต้ `/codeql-critical-security/android`
- `CodeQL macOS Critical Security` — ชาร์ดความปลอดภัย macOS รายสัปดาห์/แบบสั่งเอง สร้างแอป macOS ด้วยตนเองสำหรับ CodeQL บน Blacksmith macOS กรองผลลัพธ์การสร้างของส่วนพึ่งพาออกจาก SARIF ที่อัปโหลด และอัปโหลดภายใต้ `/codeql-critical-security/macos` แยกออกจากค่าเริ่มต้นรายวันเนื่องจากการสร้าง macOS ใช้เวลารันส่วนใหญ่แม้ไม่มีปัญหา

### หมวดหมู่คุณภาพที่สำคัญ

`CodeQL Critical Quality` คือชาร์ดที่ตรงกันสำหรับส่วนที่ไม่ใช่ความปลอดภัย โดยเรียกใช้เฉพาะคิวรีคุณภาพ JavaScript/TypeScript ที่ไม่เกี่ยวกับความปลอดภัยและมีระดับความรุนแรงเป็นข้อผิดพลาด บนพื้นผิวที่มีมูลค่าสูงและจำกัดขอบเขต โดยใช้รันเนอร์ Linux ที่โฮสต์โดย GitHub เพื่อไม่ให้การสแกนคุณภาพใช้งบประมาณการลงทะเบียนรันเนอร์ Blacksmith ด่านควบคุม pull request ของชาร์ดนี้ตั้งใจให้มีขนาดเล็กกว่าโปรไฟล์ตามกำหนดเวลา: PR ที่ไม่ใช่ฉบับร่างจะเรียกใช้เฉพาะชาร์ดที่ตรงกับพื้นผิวซึ่งได้รับผลกระทบ จากชาร์ดที่กำหนดเส้นทางสำหรับ PR ได้ทั้งหมดสิบสามรายการ ได้แก่ `agent-runtime-boundary`, `channel-runtime-boundary`, `config-boundary`, `core-auth-secrets`, `gateway-runtime-boundary`, `mcp-process-runtime-boundary`, `memory-runtime-boundary`, `network-runtime-boundary`, `plugin-boundary`, `plugin-sdk-package-contract`, `plugin-sdk-reply-runtime`, `provider-runtime-boundary` และ `session-diagnostics-boundary` ส่วน `ui-control-plane` และ `web-media-runtime-boundary` จะไม่รวมอยู่ในการรันของ PR การเปลี่ยนแปลงการกำหนดค่า CodeQL และเวิร์กโฟลว์คุณภาพจะเรียกใช้ชุดชาร์ด PR ทั้งหมด (ชาร์ดรันไทม์เครือข่ายจะทริกเกอร์จากไฟล์การกำหนดค่า CodeQL ของตนเองและพาธซอร์สที่เป็นเจ้าของเครือข่าย)

คุณภาพจะแยกจากความปลอดภัยเพื่อให้สามารถกำหนดเวลา วัดผล ปิดใช้งาน หรือขยายผลการตรวจพบด้านคุณภาพได้โดยไม่บดบังสัญญาณความปลอดภัย ควรเพิ่มการขยาย CodeQL สำหรับ Swift, Python และ Plugin แบบรวมกลับมาเป็นงานติดตามผลที่กำหนดขอบเขตหรือแบ่งเป็นชาร์ดเท่านั้น หลังจากโปรไฟล์ขอบเขตแคบมีเวลารันและสัญญาณที่เสถียรแล้ว

## เวิร์กโฟลว์การบำรุงรักษา

### เอเจนต์เอกสาร

เวิร์กโฟลว์ `Docs Agent` เป็นเลนการบำรุงรักษา Codex ที่ขับเคลื่อนด้วยเหตุการณ์สำหรับทำให้เอกสารที่มีอยู่สอดคล้องกับการเปลี่ยนแปลงที่รวมเข้ามาเมื่อไม่นานนี้ ไม่มีการกำหนดเวลาโดยตรง: การรัน CI จากการพุชที่สำเร็จและไม่ได้มาจากบอตบน `main` สามารถทริกเกอร์ได้ และการสั่งทำงานด้วยตนเองสามารถเรียกใช้ได้โดยตรง การเรียกใช้จาก workflow-run จะข้ามเมื่อ `main` เคลื่อนไปแล้ว หรือเมื่อมีการสร้างการรันเอเจนต์เอกสารอื่นที่ไม่ได้ถูกข้ามภายในชั่วโมงที่ผ่านมา เมื่อทำงาน ระบบจะตรวจสอบช่วงคอมมิตตั้งแต่ SHA ซอร์สของเอเจนต์เอกสารครั้งก่อนที่ไม่ได้ถูกข้ามจนถึง `main` ปัจจุบัน ดังนั้นการรันหนึ่งครั้งต่อชั่วโมงจึงครอบคลุมการเปลี่ยนแปลงทั้งหมดบน main ที่สะสมมาตั้งแต่การตรวจเอกสารครั้งล่าสุดได้

### เอเจนต์ประสิทธิภาพการทดสอบ

เวิร์กโฟลว์ `Test Performance Agent` เป็นเลนการบำรุงรักษา Codex ที่ขับเคลื่อนด้วยเหตุการณ์สำหรับการทดสอบที่ช้า ไม่มีการกำหนดเวลาโดยตรง: การรัน CI จากการพุชที่สำเร็จและไม่ได้มาจากบอตบน `main` สามารถทริกเกอร์ได้ แต่จะข้ามหากมีการเรียกใช้จาก workflow-run อื่นที่ทำงานไปแล้วหรือกำลังทำงานในวัน UTC เดียวกัน การสั่งทำงานด้วยตนเองจะข้ามด่านกิจกรรมรายวันนี้ เลนนี้สร้างรายงานประสิทธิภาพ Vitest แบบจัดกลุ่มสำหรับชุดทดสอบทั้งหมด อนุญาตให้ Codex แก้ไขประสิทธิภาพการทดสอบเพียงเล็กน้อยโดยคงความครอบคลุมไว้แทนการปรับโครงสร้างครั้งใหญ่ จากนั้นเรียกใช้รายงานชุดทดสอบทั้งหมดอีกครั้งและปฏิเสธการเปลี่ยนแปลงที่ลดจำนวนการทดสอบผ่านตามเกณฑ์พื้นฐาน รายงานแบบจัดกลุ่มบันทึกเวลา wall time และ RSS สูงสุดต่อการกำหนดค่าบน Linux และ macOS เพื่อให้การเปรียบเทียบก่อน/หลังแสดงส่วนต่างหน่วยความจำของการทดสอบควบคู่กับส่วนต่างระยะเวลา หากเกณฑ์พื้นฐานมีการทดสอบที่ล้มเหลว Codex จะแก้ไขได้เฉพาะความล้มเหลวที่ชัดเจน และรายงานชุดทดสอบทั้งหมดหลังการทำงานของเอเจนต์ต้องผ่านก่อนจึงจะคอมมิตสิ่งใดได้ เมื่อ `main` เดินหน้าต่อก่อนที่การพุชของบอตจะรวมเข้า เลนนี้จะรีเบสแพตช์ที่ผ่านการตรวจสอบ เรียกใช้ `pnpm check:changed` อีกครั้ง และลองพุชใหม่ โดยจะข้ามแพตช์เก่าที่ขัดแย้งกัน เลนนี้ใช้ Ubuntu ที่โฮสต์โดย GitHub เพื่อให้แอ็กชัน Codex รักษาแนวทางความปลอดภัยแบบตัดสิทธิ์ sudo เช่นเดียวกับเอเจนต์เอกสาร

### PR ซ้ำหลังการรวม

เวิร์กโฟลว์ `Duplicate PRs After Merge` เป็นเวิร์กโฟลว์แบบสั่งเองสำหรับผู้ดูแล เพื่อทำความสะอาดรายการซ้ำหลังการรวม โดยมีค่าเริ่มต้นเป็นการทดลองรัน และจะปิดเฉพาะ PR ที่ระบุไว้อย่างชัดเจนเมื่อ `apply=true` ก่อนแก้ไขข้อมูลบน GitHub ระบบจะตรวจสอบว่า PR ที่รวมเข้าได้รับการรวมแล้ว และรายการซ้ำแต่ละรายการมีอ้างอิงถึง issue เดียวกันหรือมีส่วนโค้ดที่เปลี่ยนแปลงทับซ้อนกัน

```bash
gh workflow run duplicate-after-merge.yml \
  -f landed_pr=70532 \
  -f duplicate_prs='70530,70592' \
  -f apply=true
```

## ด่านตรวจสอบภายในเครื่องและการกำหนดเส้นทางตามการเปลี่ยนแปลง

ตรรกะเลนที่เปลี่ยนแปลงภายในเครื่องอยู่ใน `scripts/changed-lanes.mjs` และเรียกใช้โดย `scripts/check-changed.mjs` ด่านตรวจสอบภายในเครื่องนี้เข้มงวดกับขอบเขตสถาปัตยกรรมมากกว่าขอบเขตแพลตฟอร์ม CI แบบกว้าง:

- การเปลี่ยนแปลงโค้ดโปรดักชันหลักจะเรียกใช้การตรวจสอบชนิดของโปรดักชันหลักและการทดสอบหลัก รวมถึง lint/ด่านควบคุมหลัก;
- การเปลี่ยนแปลงเฉพาะการทดสอบหลักจะเรียกใช้เฉพาะการตรวจสอบชนิดของการทดสอบหลักและ lint หลัก;
- การเปลี่ยนแปลงโค้ดโปรดักชันของส่วนขยายจะเรียกใช้การตรวจสอบชนิดของโปรดักชันส่วนขยายและการทดสอบส่วนขยาย รวมถึง lint ส่วนขยาย;
- การเปลี่ยนแปลงเฉพาะการทดสอบส่วนขยายจะเรียกใช้การตรวจสอบชนิดของการทดสอบส่วนขยายและ lint ส่วนขยาย;
- การเปลี่ยนแปลง Plugin SDK สาธารณะหรือสัญญา Plugin จะขยายไปถึงการตรวจสอบชนิดของส่วนขยาย เนื่องจากส่วนขยายพึ่งพาสัญญาหลักเหล่านั้น (การกวาดทดสอบส่วนขยายด้วย Vitest ยังคงเป็นงานทดสอบที่ต้องระบุอย่างชัดเจน);
- การเพิ่มเวอร์ชันเฉพาะเมทาดาทาของรีลีสจะเรียกใช้การตรวจสอบเวอร์ชัน/การกำหนดค่า/ส่วนพึ่งพารากแบบเจาะจง;
- การเปลี่ยนแปลงราก/การกำหนดค่าที่ไม่รู้จักจะทำงานแบบปลอดภัยโดยเรียกใช้เลนตรวจสอบทั้งหมด

การกำหนดเส้นทางการทดสอบที่เปลี่ยนแปลงภายในเครื่องอยู่ใน `scripts/test-projects.test-support.mjs` และตั้งใจให้ประหยัดกว่า `check:changed`: การแก้ไขการทดสอบโดยตรงจะเรียกใช้ตัวเอง การแก้ไขซอร์สจะเลือกการแมปที่ระบุชัดเจนก่อน ตามด้วยการทดสอบข้างเคียงและส่วนที่ขึ้นต่อกันในกราฟการนำเข้า การกำหนดค่าการส่งในห้องกลุ่มที่ใช้ร่วมกันเป็นหนึ่งในการแมปที่ระบุชัดเจน: การเปลี่ยนแปลงการกำหนดค่าการตอบกลับที่มองเห็นได้ของกลุ่ม โหมดการส่งการตอบกลับจากซอร์ส หรือพรอมต์ระบบของเครื่องมือข้อความ จะกำหนดเส้นทางผ่านการทดสอบการตอบกลับหลัก รวมถึงการทดสอบการถดถอยของการส่งบน Discord และ Slack เพื่อให้การเปลี่ยนแปลงค่าเริ่มต้นร่วมล้มเหลวก่อนการพุช PR ครั้งแรก ใช้ `OPENCLAW_TEST_CHANGED_BROAD=1 pnpm test:changed` เฉพาะเมื่อการเปลี่ยนแปลงครอบคลุมชุดทดสอบกว้างจนชุดการแมปราคาประหยัดไม่ใช่ตัวแทนที่เชื่อถือได้

## การตรวจสอบ Testbox

Crabbox เป็น wrapper ของ remote-box ที่รีโปเป็นเจ้าของสำหรับการพิสูจน์บน Linux ของผู้ดูแล เซสชันของเอเจนต์
จะเก็บการทดสอบแบบเจาะจงหนึ่งหรือไม่กี่รายการและการตรวจสอบแบบสแตติกที่ใช้ทรัพยากรน้อยไว้ในเครื่องเฉพาะเมื่อ
ซอร์สเชื่อถือได้และการติดตั้ง dependency ที่มีอยู่พร้อมใช้งาน โดยใช้ Crabbox สำหรับชุดทดสอบที่ใหญ่กว่าและ
งานที่ใช้การประมวลผลสูง รวมถึงการบิลด์ การตรวจสอบชนิด การกระจายงาน lint
Docker เลนแพ็กเกจ E2E การพิสูจน์แบบสด และความสอดคล้องกับ CI การพิสูจน์หนักสำหรับผู้ดูแลที่เชื่อถือได้
ใช้ `blacksmith-testbox` เป็นค่าเริ่มต้น และขณะนี้ `.crabbox.yaml` ก็ใช้ค่านี้เป็นค่าเริ่มต้น เวิร์กโฟลว์ที่กำหนดค่าไว้
จะเติมข้อมูลรับรองของผู้ให้บริการและเอเจนต์ ดังนั้นโค้ดจากผู้มีส่วนร่วมที่ไม่น่าเชื่อถือหรือ
fork ต้องใช้ CI ของ fork ที่ไม่มี secret หรือ AWS Crabbox โดยตรงที่ผ่านการทำให้ปลอดภัยแทน
การรัน AWS ที่ผ่านการทำให้ปลอดภัยจะตั้งค่า `CRABBOX_ENV_ALLOW=CI` ส่ง
`--no-hydrate` และใช้ `HOME` ระยะไกลชั่วคราวใหม่ ซึ่งป้องกันไม่ให้รายการอนุญาต
`OPENCLAW_*` ของรีโปและโปรไฟล์การยืนยันตัวตนที่มีอยู่เข้าถึงโค้ดที่ไม่น่าเชื่อถือ
การรันเหล่านี้ใช้ลีสที่วอร์มใหม่ซึ่งใช้เฉพาะกับซอร์สที่ไม่น่าเชื่อถือนั้นเท่านั้น ห้ามใช้
ลีสที่เชื่อถือได้หรือเคยผ่านการเติมข้อมูลรับรองแล้ว เรียกใช้ไบนารี Crabbox
ที่ติดตั้งจากแหล่งที่เชื่อถือได้บน checkout `main` ที่สะอาดและเชื่อถือได้ และดึงเฉพาะ PR ระยะไกลด้วย
`--fresh-pr` ห้ามเรียกใช้ wrapper หรือการกำหนดค่าของ checkout ที่ไม่น่าเชื่อถือในเครื่อง
ยกเลิกการตั้งค่า `CRABBOX_AWS_INSTANCE_PROFILE` และปฏิเสธการทำงานโดยค่าเริ่มต้น เว้นแต่
`aws.instanceProfile` ที่แก้ค่าแล้วจะว่างเปล่า ก่อนติดตั้ง/ทดสอบใดๆ ให้ใช้
เครื่องมือพาธสัมบูรณ์ที่เชื่อถือได้เพื่อบังคับใช้โทเค็น IMDSv2 พิสูจน์ว่า endpoint ข้อมูลรับรอง IAM
ส่งคืน 404 และเปรียบเทียบ `git rev-parse HEAD` ระยะไกลกับ SHA แบบเต็ม
ของ head PR ที่รีวิวแล้ว ผูกลีสกับ SHA นั้น และหยุด/วอร์มใหม่เมื่อ head เปลี่ยน
อัปโหลด `scripts/crabbox-untrusted-bootstrap.sh` ที่เชื่อถือได้จาก `main` ที่สะอาด
พร้อมกับ `--fresh-pr` ซึ่งจะติดตั้ง Node/pnpm เวอร์ชันที่ปักหมุดไว้ ตรวจสอบ SHA และ
การปักหมุดตัวจัดการแพ็กเกจ แยก `HOME` ติดตั้ง dependency แล้วจึงเรียกใช้
การทดสอบที่ร้องขอ
ยกเลิกการตั้งค่า override `CRABBOX_TAILSCALE*` ทั้งหมด บังคับใช้ `--network public
--tailscale=false` ล้างแฟล็ก exit-node/LAN และกำหนดให้ `crabbox inspect`
รายงานเครือข่ายสาธารณะโดยไม่มีสถานะ Tailscale ก่อนอัปโหลดสคริปต์ใดๆ
ความจุ AWS/Hetzner ที่เป็นเจ้าของยังคงเป็นทางเลือกสำรองเมื่อ Blacksmith ขัดข้อง
มีปัญหาโควตา หรือมีการทดสอบความจุที่เป็นเจ้าของอย่างชัดเจน

เอเจนต์จะไม่วอร์มล่วงหน้าสำหรับงานที่คาดว่าจะทำ ให้รับ Testbox แบบตามความจำเป็นเมื่อ
คำสั่งหนักคำสั่งแรกพร้อมใช้งาน ใช้ id `tbx_...` ที่ส่งคืนซ้ำสำหรับคำสั่งหนัก
รายการถัดไป ซิงค์ checkout ปัจจุบันในการรันทุกครั้ง และหยุดก่อนส่งมอบงาน

การรัน Blacksmith ที่ใช้ Crabbox จะวอร์ม รับสิทธิ์ ซิงค์ รัน รายงาน และล้าง
Testbox แบบใช้ครั้งเดียว การตรวจสอบความถูกต้องของการซิงค์ในตัวจะล้มเหลวทันทีเมื่อ
`git status --short` บนกล่องที่ซิงค์แสดงการลบไฟล์ที่ติดตามอย่างน้อย 200 รายการ
ซึ่งตรวจจับไฟล์รากที่หายไป เช่น `pnpm-lock.yaml` สำหรับ PR ที่ตั้งใจ
ลบไฟล์จำนวนมาก ให้ตั้งค่า `CRABBOX_ALLOW_MASS_DELETIONS=1` สำหรับคำสั่งระยะไกล

Crabbox ยังยุติการเรียกใช้ Blacksmith CLI ในเครื่องที่ค้างอยู่ใน
ขั้นตอนการซิงค์นานกว่า 5 นาทีโดยไม่มีเอาต์พุตหลังการซิงค์ ตั้งค่า
`CRABBOX_BLACKSMITH_SYNC_TIMEOUT_MS=0` เพื่อปิดใช้งานการป้องกันนี้ หรือใช้ค่า
มิลลิวินาทีที่มากขึ้นสำหรับ diff ในเครื่องที่มีขนาดใหญ่ผิดปกติ

ก่อนรันครั้งแรก ให้ตรวจสอบ wrapper จากรากของรีโป:

```bash
pnpm crabbox:run -- --help | sed -n '1,120p'
```

wrapper ของรีโปจะปฏิเสธไบนารี Crabbox ที่ล้าสมัยซึ่งไม่ได้ประกาศผู้ให้บริการที่เลือกไว้ และการรันที่ใช้ Blacksmith ต้องใช้ Crabbox 0.22.0 หรือใหม่กว่า เพื่อให้ wrapper ได้รับพฤติกรรมการซิงค์ คิว และการล้าง Testbox ปัจจุบัน ใน worktree ของ Codex หรือ checkout แบบเชื่อมโยง/กระจัดกระจาย ให้หลีกเลี่ยงสคริปต์ `pnpm crabbox:run` ในเครื่อง เพราะ pnpm อาจปรับ dependency ให้ตรงกันก่อน Crabbox เริ่มทำงาน ให้เรียกใช้ node wrapper โดยตรงแทน:

```bash
node scripts/crabbox-wrapper.mjs run --provider blacksmith-testbox --timing-json --shell -- "pnpm test <path-or-filter>"
```

เมื่อใช้ checkout ที่อยู่ข้างเคียง ให้บิลด์ไบนารีในเครื่องที่ถูกละเว้นใหม่ก่อนงานจับเวลาหรือพิสูจน์:

```bash
version="$(git -C ../crabbox describe --tags --always --dirty | sed 's/^v//')" \
  && go build -C ../crabbox -trimpath -ldflags "-s -w -X github.com/openclaw/crabbox/internal/cli.version=${version}" -o bin/crabbox ./cmd/crabbox
```

บล็อก `blacksmith:` ใน `.crabbox.yaml` ปักหมุดค่าเริ่มต้นขององค์กร เวิร์กโฟลว์ งาน และ ref ไว้แล้ว ดังนั้นแฟล็กที่ระบุชัดเจนด้านล่างจึงไม่บังคับใช้ เกตการเปลี่ยนแปลง:

```bash
pnpm crabbox:run -- --provider blacksmith-testbox \
  --blacksmith-org openclaw \
  --blacksmith-workflow .github/workflows/ci-check-testbox.yml \
  --blacksmith-job check \
  --blacksmith-ref main \
  --idle-timeout 90m \
  --ttl 240m \
  --timing-json \
  --shell -- \
  "corepack pnpm check:changed"
```

รันการทดสอบแบบเจาะจงซ้ำบน Testbox เมื่อ dependency ในเครื่องไม่พร้อมใช้งานหรือ
เป้าหมายกระจายงานออกไป:

```bash
pnpm crabbox:run -- --provider blacksmith-testbox \
  --idle-timeout 90m \
  --ttl 240m \
  --timing-json \
  --shell -- \
  "corepack pnpm test <path-or-filter>"
```

ชุดทดสอบทั้งหมด:

```bash
pnpm crabbox:run -- --provider blacksmith-testbox \
  --idle-timeout 90m \
  --ttl 240m \
  --timing-json \
  --shell -- \
  "corepack pnpm test"
```

อ่านสรุป JSON สุดท้าย ฟิลด์ที่มีประโยชน์คือ `provider`, `leaseId`,
`syncDelegated`, `exitCode`, `commandMs` และ `totalMs` สำหรับการรัน
Blacksmith Testbox ที่มอบหมายให้ทำ รหัสออกของ Crabbox wrapper และสรุป JSON คือ
ผลลัพธ์ของคำสั่ง การรัน GitHub Actions ที่เชื่อมโยงอยู่เป็นเจ้าของการเติมข้อมูลและ keepalive ซึ่ง
อาจจบด้วย `cancelled` เมื่อ Testbox ถูกหยุดจากภายนอกหลังคำสั่ง SSH
ส่งคืนแล้ว ให้ถือว่านั่นเป็นอาร์ติแฟกต์การล้าง/สถานะ เว้นแต่
`exitCode` ของ wrapper จะไม่เป็นศูนย์ หรือเอาต์พุตคำสั่งแสดงว่าการทดสอบล้มเหลว
การรัน Crabbox แบบใช้ครั้งเดียวที่ใช้ Blacksmith ควรหยุด Testbox โดยอัตโนมัติ
หากการรันถูกขัดจังหวะหรือสถานะการล้างไม่ชัดเจน ให้ตรวจสอบกล่องที่ทำงานอยู่และหยุดเฉพาะ
กล่องที่สร้างขึ้นเอง:

```bash
blacksmith testbox list --all
blacksmith testbox status --id <tbx_id>
blacksmith testbox stop --id <tbx_id>
```

ใช้การนำกลับมาใช้ใหม่เฉพาะเมื่อตั้งใจต้องใช้หลายคำสั่งบนกล่องที่เติมข้อมูลแล้วกล่องเดียวกัน:

```bash
node scripts/crabbox-wrapper.mjs run --provider blacksmith-testbox --id <tbx_id> --timing-json --shell -- "corepack pnpm test <path-or-filter>"
pnpm crabbox:stop -- <tbx_id>
```

นำลีสกลับมาใช้ใหม่ ไม่ใช่ซอร์สที่ล้าสมัย ละเว้น `--no-sync` เพื่อให้การรันแต่ละครั้งอัปโหลด
checkout ปัจจุบัน ใช้เฉพาะเมื่อตั้งใจรันโครงสร้างที่ไม่เปลี่ยนแปลงและซิงค์แล้วซ้ำ
โค้ดจากผู้มีส่วนร่วมหรือ fork ที่ไม่น่าเชื่อถือต้องใช้
`CRABBOX_ENV_ALLOW=CI`, `--provider aws --no-hydrate` และ `HOME`
ระยะไกลชั่วคราวใหม่สำหรับทุกคำสั่ง โดยติดตั้ง dependency ภายในคำสั่งที่ผ่านการทำให้ปลอดภัยนั้น
ก่อนทดสอบ นำกลับมาใช้ใหม่ได้เฉพาะลีสที่วอร์มใหม่ซึ่งใช้เฉพาะกับซอร์สที่ไม่น่าเชื่อถือ
รายการเดียวกันเท่านั้น ห้ามใช้ลีสที่เชื่อถือได้หรือเคยผ่านการเติมข้อมูลรับรองแล้ว ห้าม
เรียกใช้ wrapper หรือการกำหนดค่าของ checkout ที่ไม่น่าเชื่อถือในเครื่อง ให้เรียกใช้ไบนารี Crabbox
ที่ติดตั้งจากแหล่งที่เชื่อถือได้บน `main` ที่สะอาดและเชื่อถือได้ และส่ง `--fresh-pr` ในทุก
การรัน อย่าตั้งค่า `CRABBOX_AWS_INSTANCE_PROFILE` ปฏิเสธโปรไฟล์อินสแตนซ์
ที่แก้ค่าแล้วและไม่ว่างเปล่า บังคับใช้การพิสูจน์ IMDS ระยะไกลที่เชื่อถือได้ว่าไม่มี role และตรวจสอบ
SHA ของ head ที่รีวิวแล้วก่อนติดตั้ง/ทดสอบ ผูกลีสกับ SHA นั้น แล้วหยุดและ
วอร์มใหม่หลังจาก head เปลี่ยน หากไม่มี PR ระยะไกล ให้ใช้ CI ของ fork ที่ไม่มี secret
ห้ามเลือก `hydrate-github` หรือเวิร์กโฟลว์ Blacksmith ที่เติมข้อมูลรับรอง
สำหรับซอร์สที่ไม่น่าเชื่อถือ

หาก Crabbox เป็นเลเยอร์ที่เสีย แต่ Blacksmith เองยังทำงาน ให้ใช้
Blacksmith โดยตรงเฉพาะสำหรับการวินิจฉัย เช่น `list`, `status` และการล้าง แก้ไข
พาธ Crabbox ก่อนถือว่าการรัน Blacksmith โดยตรงเป็นการพิสูจน์ของผู้ดูแล

หาก `blacksmith testbox list --all` และ `blacksmith testbox status` ทำงาน แต่การ
วอร์มใหม่ค้างอยู่ที่ `queued` โดยไม่มี IP หรือ URL การรัน Actions หลังผ่านไปสองสามนาที
ให้ถือว่าเป็นแรงกดดันจากผู้ให้บริการ Blacksmith คิว การเรียกเก็บเงิน หรือขีดจำกัดองค์กร หยุด
id ที่อยู่ในคิวซึ่งสร้างขึ้น หลีกเลี่ยงการเริ่ม Testbox เพิ่ม และย้ายการพิสูจน์ไปยัง
เส้นทางความจุ Crabbox ที่เป็นเจ้าของด้านล่าง ขณะที่มีผู้ตรวจสอบแดชบอร์ด Blacksmith
การเรียกเก็บเงิน และขีดจำกัดองค์กร

ยกระดับไปใช้ความจุ Crabbox ที่เป็นเจ้าของเฉพาะเมื่อ Blacksmith หยุดทำงาน ถูกจำกัดโควตา ไม่มีสภาพแวดล้อมที่ต้องการ หรือมีเป้าหมายชัดเจนว่าจะใช้ความจุที่เป็นเจ้าของ:

```bash
CRABBOX_CAPACITY_REGIONS=eu-west-1,eu-west-2,eu-central-1,us-east-1,us-west-2 \
  pnpm crabbox:warmup -- --provider aws --class standard --market on-demand --idle-timeout 90m
pnpm crabbox:hydrate -- --provider aws --id <cbx_id-or-slug>
pnpm crabbox:run -- --provider aws --id <cbx_id-or-slug> --timing-json --shell -- "pnpm check:changed"
pnpm crabbox:stop -- --provider aws <cbx_id-or-slug>
```

ภายใต้แรงกดดันของ AWS ให้หลีกเลี่ยง `class=beast` เว้นแต่งานต้องใช้ CPU ระดับ 48xlarge จริงๆ คำขอ `beast` เริ่มต้นที่ 192 vCPUs และเป็นวิธีที่ทำให้ชนโควตา EC2 Spot หรือ On-Demand Standard ระดับภูมิภาคได้ง่ายที่สุด `.crabbox.yaml` ที่รีโปเป็นเจ้าของใช้ `class: standard` ตลาด on-demand และ `capacity.hints: true` เป็นค่าเริ่มต้น เพื่อให้ลีส AWS ผ่านโบรกเกอร์พิมพ์ภูมิภาค/ตลาดที่เลือก แรงกดดันด้านโควตา การสำรองไปใช้ Spot และคำเตือนคลาสที่มีแรงกดดันสูง ใช้ `fast` สำหรับการตรวจสอบกว้างที่หนักขึ้น ใช้ `large` เฉพาะหลังจาก standard/fast ไม่เพียงพอ และใช้ `beast` เฉพาะสำหรับเลนที่ใช้ CPU สูงเป็นกรณีพิเศษ เช่น ชุดทดสอบทั้งหมดหรือเมทริกซ์ Docker ของ Plugin ทั้งหมด การตรวจสอบรีลีส/ตัวบล็อกอย่างชัดเจน หรือการทำโปรไฟล์ประสิทธิภาพแบบคอร์สูง ห้ามใช้ `beast` สำหรับ `pnpm check:changed` การทดสอบแบบเจาะจง งานเฉพาะเอกสาร lint/การตรวจสอบชนิดทั่วไป การทำซ้ำ E2E ขนาดเล็ก หรือการวิเคราะห์ Blacksmith ขัดข้อง ใช้ `--market on-demand` สำหรับการวินิจฉัยความจุ เพื่อไม่ให้ความผันผวนของตลาด Spot ปะปนกับสัญญาณ

`.crabbox.yaml` เป็นเจ้าของค่าเริ่มต้นของผู้ให้บริการ การซิงค์ และการเติมข้อมูล GitHub Actions การซิงค์ของ Crabbox จะไม่ถ่ายโอน `.git` ดังนั้น checkout ของ Actions ที่เติมข้อมูลแล้วจะเก็บเมทาดาทา Git ระยะไกลของตัวเองไว้ แทนที่จะซิงค์ remote และ object store ในเครื่องของผู้ดูแล และการกำหนดค่าของรีโปยังยกเว้นอาร์ติแฟกต์รันไทม์/บิลด์ในเครื่องเพิ่มเติม (เช่น `.artifacts` และรายงานการทดสอบ) ที่ไม่ควรถูกถ่ายโอนโดยเด็ดขาด `.github/workflows/crabbox-hydrate.yml` เป็นเจ้าของ checkout การตั้งค่า Node/pnpm การดึง `origin/main` และการส่งต่อสภาพแวดล้อมที่ไม่ใช่ secret สำหรับคำสั่ง `crabbox run --id <cbx_id>` บนคลาวด์ที่เป็นเจ้าของ
