# บันทึกประจำรุ่น OpenClaw v2026.6.11 (2026-06-30)

เราได้รับฟังความคิดเห็นของคุณแล้ว v2026.6.11 มุ่งเน้นแก้ไขจุดที่ยังไม่ราบรื่นซึ่งทำให้ OpenClaw ดูไม่น่าเชื่อถือเท่าที่ควร โดยแก้ปัญหาการตอบกลับผิดตำแหน่ง การส่งที่ค้าง การเชื่อมต่อใหม่ ความล้มเหลวในการตั้งค่าโมเดล และค่าเริ่มต้นสำหรับผู้ดูแลระบบที่ปลอดภัยยิ่งขึ้น

## ไฮไลต์

### ความน่าเชื่อถือในการส่งผ่านช่องทาง

ใน [Telegram](/th/channels/telegram), [WhatsApp](/th/channels/whatsapp), [Matrix](/th/channels/matrix), [Google Chat](/th/channels/googlechat), [iMessage](/th/channels/imessage), [Feishu](/th/channels/feishu) และ [Mattermost](/th/channels/mattermost) การตอบกลับ คำสั่ง ข้อความในคิว และไฟล์แนบมีโอกาสน้อยลงที่จะสูญหาย ซ้ำกัน ถูกส่งผิดเส้นทาง หรือแนบไปยังบทสนทนาที่ไม่ถูกต้อง

### การกู้คืนผู้ให้บริการและโมเดล

[การเลือกโมเดล](/th/providers/models) และการตั้งค่าสามารถกู้คืนได้ชัดเจนยิ่งขึ้นเมื่อแค็ตตาล็อก ข้อมูลประจำตัว สตรีม การหมดเวลา Compaction หรือกลไกสำรองเกิดข้อผิดพลาด ขณะนี้การกำหนดค่า [OpenAI](/th/providers/openai), [OpenRouter](/th/providers/openrouter) และ [OpenCode Go](/th/providers/opencode-go) ที่ได้รับผลกระทบจะรายงานหรือกู้คืนจากความล้มเหลวได้สม่ำเสมอยิ่งขึ้น แทนที่จะปล่อยให้ผู้ใช้ติดอยู่กับตัวเลือกที่ล้าสมัยหรือคำขอที่หยุดชะงัก

การแก้ไขเพิ่มเติมสำหรับ [โหมดเร็ว](/th/tools/thinking#fast-mode-fast) ช่วยปรับปรุงการทำงานในเส้นทางของผู้ให้บริการและกลไกสำรองที่ได้รับผลกระทบ ทั้งนี้ โหมดเร็วอัตโนมัติไม่ใช่คุณสมบัติใหม่ในรุ่นนี้

### ความต่อเนื่องของเซสชัน หน่วยความจำ และความเชื่อถือ

[เซสชัน](/th/concepts/session), [Compaction](/th/concepts/compaction), [หน่วยความจำ](/th/concepts/memory) และ[หน่วยความจำที่ทำงานด้วย QMD](/th/concepts/memory-qmd) ช่วยรักษาบทสนทนาที่ต้องการและบริบทที่เป็นประโยชน์ให้สม่ำเสมอยิ่งขึ้นตลอดการทำงานระยะยาว การเชื่อมต่อใหม่ การอัปเกรด และการซ่อมแซมบันทึกบทสนทนา นอกจากนี้ [การค้นหาเครื่องมือ](/th/tools/tool-search) ยังทำงานได้อย่างน่าเชื่อถือมากขึ้นเมื่อตัวแทนต้องกู้คืนบริบทหรือความสามารถที่ถูกต้อง

การกู้คืน [Matrix](/th/channels/matrix) ที่เข้ารหัสจะหยุดอย่างปลอดภัยเมื่อไม่สามารถตรวจสอบสถานะคีย์ที่จำเป็นได้ [นโยบายเครื่องมือ](/th/gateway/sandbox-vs-tool-policy-vs-elevated), [การอนุมัติ](/th/cli/approvals) และ[การจัดการข้อมูลลับ](/th/gateway/secrets) จะยังคงผูกกับสถานะรันไทม์ที่ตั้งใจไว้ โดยการดำเนินการที่มีความเสี่ยงสูงกว่าจะยังคงถูกปิดใช้งาน เว้นแต่จะเปิดใช้งานอย่างชัดเจน

### การติดตั้งและซ่อมแซม Plugin อย่างเป็นทางการ

ขณะนี้[การจัดการ Plugin](/th/plugins/manage-plugins) รองรับการผสานรวมอย่างเป็นทางการเพิ่มขึ้นผ่านโฟลว์การติดตั้งและซ่อมแซมแพ็กเกจภายนอกตามปกติ [รายการ Plugin](/th/plugins/plugin-inventory) และการตรวจสอบการตั้งค่าที่เกี่ยวข้องจะแสดงผลลัพธ์ที่ชัดเจนขึ้นเมื่อแพ็กเกจขาดหาย ไม่เข้ากัน หรือต้องติดตั้งใหม่

<Accordion title="แหล่งที่มาและผู้มีส่วนร่วม">

- เมื่อ `plugins.allow` ใช้ชื่อช่องทางหรือชื่อแพ็กเกจแทนรหัส Plugin จริง ขณะนี้คำแนะนำเมื่อเริ่มต้นระบบจะระบุรายการที่ไม่ตรงกันและแสดงรหัส Plugin ที่ค้นพบซึ่งจำเป็นสำหรับแก้ไขการกำหนดค่า [#68389](https://github.com/openclaw/openclaw/pull/68389) เกี่ยวข้องกับ [#68352](https://github.com/openclaw/openclaw/issues/68352) ขอขอบคุณ @aym9999, @jirboy, @lyfuci, @pahuchi-joe, @zmxccxy
- ขณะนี้คำเตือนเรื่องความน่าเชื่อถือของ Plugin สำหรับการติดตั้งครั้งแรกหรือการติดตั้งใหม่มีตัวอย่าง `plugins.allow` ที่พร้อมคัดลอกและคำสั่งสำหรับแสดงรายการหรือตรวจสอบรหัส Plugin เพื่อให้ผู้ใช้แก้ไขคำเตือนได้ก่อนเชื่อถือหรือติดตั้งโค้ด Plugin ใหม่ [#78105](https://github.com/openclaw/openclaw/pull/78105) เกี่ยวข้องกับ [#68780](https://github.com/openclaw/openclaw/issues/68780) ขอขอบคุณ @jirboy, @pahuchi-joe
- ขณะนี้การย้ายข้อมูลของ Codex ใช้งานได้กับการติดตั้ง Plugin ส่วนกลางแบบมาตรฐาน เนื่องจาก `openclaw migrate` สามารถค้นหาผู้ให้บริการที่ติดตั้งไว้แทนที่จะล้มเหลวด้วย `Unknown migration provider` [#89612](https://github.com/openclaw/openclaw/pull/89612) เกี่ยวข้องกับ [#89609](https://github.com/openclaw/openclaw/issues/89609) ขอขอบคุณ @mugabuga, @zerone0x
- การติดตั้งและอัปเดต Plugin สามารถกู้คืนจากการตรึงเวอร์ชันการขึ้นต่อกันที่ล้าสมัยซึ่ง OpenClaw จัดการ แทนที่จะล้มเหลวด้วย `npm EOVERRIDE` โดยการซิงโครไนซ์ในภายหลังจะไม่ดาวน์เกรดหรือลบแพ็กเกจที่ผู้ใช้ติดตั้งไว้อย่างชัดเจน [#91786](https://github.com/openclaw/openclaw/pull/91786) เกี่ยวข้องกับ [#91772](https://github.com/openclaw/openclaw/issues/91772) ขอขอบคุณ @amknight, @mkdelta221
- ขณะนี้นักพัฒนา Plugin ช่องทางสามารถส่งตัวระบุผู้ส่งและการสนทนาแบบเนทีฟผ่านฮุกและเวิร์กโฟลว์การเรียกใช้ที่เลือกได้ ทำให้การผสานรวมกำหนดเส้นทางได้แม่นยำขึ้นโดยไม่ทำให้ฟิลด์ผู้ส่งและแชตที่มีอยู่เสียหาย [#91903](https://github.com/openclaw/openclaw/pull/91903) ขอขอบคุณ @lanzhi-lee, @vincentkoc

- ห่วงโซ่การตอบกลับของ Telegram ยังคงแนบการตอบกลับที่แคชไว้หลังจากบริบทเปลี่ยนแปลง แทนที่จะล้มเหลวเมื่อมีการนำการตอบกลับที่แคชไว้เหล่านั้นกลับมาใช้ [#82909](https://github.com/openclaw/openclaw/pull/82909) ขอขอบคุณ @lidge-jun

## Gateway ความปลอดภัย และความน่าเชื่อถือ

### การกู้คืนจากการรีสตาร์ตและความพร้อมใช้งาน

ขณะนี้สัญญาณ[สถานะการทำงานของ Gateway](/th/gateway/health) และ[การแก้ไขปัญหา](/th/gateway/troubleshooting) สอดคล้องกับสถานะจริงของ OpenClaw มากขึ้นว่าพร้อมใช้งาน กำลังรีสตาร์ต หรือไม่สามารถทำงานต่อได้ การเรียกใช้เอเจนต์ที่เริ่มผ่าน[CLI](/th/cli/agent) กู้คืนจากการตัดการเชื่อมต่อ การปิดระบบ การเปลี่ยนแปลงการกำหนดเส้นทาง และเงื่อนไขการเริ่มต้นระบบที่ล้มเหลวได้อย่างราบรื่นยิ่งขึ้น

<Accordion title="แหล่งที่มาและผู้มีส่วนร่วม">

- เมื่อ Gateway ระยะไกลที่กำหนดค่าไว้หรือระบุอย่างชัดเจนทำงานช้าแต่ยังเข้าถึงได้ ตอนนี้ `openclaw gateway probe --timeout ...` จะรอตามระยะหมดเวลาที่ร้องขอ แทนที่จะรายงานว่าเข้าถึงไม่ได้หลังจากถึงขีดจำกัดภายในที่สั้นกว่า [#89859](https://github.com/openclaw/openclaw/pull/89859) ที่เกี่ยวข้อง [#65355](https://github.com/openclaw/openclaw/issues/65355) ขอบคุณ @hellocli, @mushuiyu886
- ตอนนี้การทำงานของเอเจนต์ย่อยภายในที่ใช้เวลานานหรือทำงานแบบขนานจะหลีกเลี่ยงการประมวลผลตัวอย่างแบบสดที่ไม่จำเป็น ขณะที่เซสชันเอเจนต์ย่อยที่มองเห็นได้ยังคงแสดงการอัปเดตแบบสดและคำตอบสุดท้าย [#91906](https://github.com/openclaw/openclaw/pull/91906) ขอบคุณ @lanzhi-lee, @vincentkoc
- ตอนนี้การสนทนา ACP โดยเฉพาะเธรดที่ใช้ Kiro เป็นระบบเบื้องหลัง สามารถดำเนินต่อหลังจากการตอบครั้งแรกได้ โดยเริ่มเซสชันใหม่เมื่อระบบเบื้องหลังไม่สามารถดำเนินเซสชันเดิมต่อได้อีก [#93547](https://github.com/openclaw/openclaw/pull/93547) ที่เกี่ยวข้อง [#87830](https://github.com/openclaw/openclaw/issues/87830) ขอบคุณ @amersheeny, @chouzz
- เมื่อภาวะหน่วยความจำตึงตัวของ Linux ยุติคำสั่งหรือเซสชันลูก ตอนนี้ Gateway ของ OpenClaw ที่จัดการโดย systemd จะยังคงทำงานและรักษาการเชื่อมต่อช่องทางไว้ พร้อมรายงานความล้มเหลวของโปรเซสลูก [#93585](https://github.com/openclaw/openclaw/pull/93585) ขอบคุณ @snowzlm
- ตอนนี้การยกเลิกการทำงานของ OpenClaw ระหว่างที่เครื่องมือกำลังทำงานจะยุติการทำงานทันที แทนที่จะเริ่มรอบการทำงานของโมเดลอีกครั้งหรือปล่อยให้เซสชันถูกล็อก [#94412](https://github.com/openclaw/openclaw/pull/94412) ขอบคุณ @szsip239, @vincentkoc

## ไคลเอนต์และอินเทอร์เฟซ

### การส่งของไคลเอนต์และการเชื่อมต่อใหม่

[WebChat](/th/web/webchat) และ [Control UI](/th/web/control-ui) แสดงการสนทนาที่กำลังดำเนินอยู่และสถานะความล้มเหลวได้อย่างสม่ำเสมอยิ่งขึ้นหลังจากเชื่อมต่อใหม่ ขณะที่ไคลเอนต์อุปกรณ์เคลื่อนที่และ [อินเทอร์เฟซเทอร์มินัล](/th/cli/tui) สามารถกู้คืนการส่งที่เสร็จสิ้น ถูกปฏิเสธ หรือถูกขัดจังหวะ โดยไม่ทำให้ดูเหมือนยังค้างอยู่

<Accordion title="แหล่งที่มาและผู้มีส่วนร่วม">

### การแก้ไขอินเทอร์เฟซ การตั้งค่า และการเริ่มต้นใช้งาน

## เอกสารและเครื่องมือผู้ดูแลระบบ

### ความน่าเชื่อถือของการตั้งค่าและคำสั่ง
