# บันทึกประจำรุ่น OpenClaw v2026.6.11 (2026-06-30)

เราได้รับฟังความคิดเห็นแล้ว v2026.6.11 มุ่งเน้นจุดติดขัดที่ทำให้ OpenClaw รู้สึกพึ่งพาได้น้อยลง พร้อมการแก้ไขสำหรับการตอบกลับผิดที่ การส่งที่ค้าง การเชื่อมต่อใหม่ ความล้มเหลวในการตั้งค่าโมเดล และค่าเริ่มต้นสำหรับผู้ดูแลระบบที่ปลอดภัยยิ่งขึ้น

## ไฮไลต์

### ความน่าเชื่อถือในการส่งผ่านช่องทาง

ใน [Telegram](/th/channels/telegram), [WhatsApp](/th/channels/whatsapp), [Matrix](/th/channels/matrix), [Google Chat](/th/channels/googlechat), [iMessage](/th/channels/imessage), [Feishu](/th/channels/feishu), และ [Mattermost](/th/channels/mattermost) การตอบกลับ คำสั่ง ข้อความที่อยู่ในคิว และไฟล์แนบมีโอกาสน้อยลงที่จะถูกทิ้ง ส่งซ้ำ ส่งผิดเส้นทาง หรือแนบกับการสนทนาผิดรายการ

### การกู้คืนผู้ให้บริการและโมเดล

[การเลือกโมเดล](/th/providers/models) และการตั้งค่ากู้คืนได้ชัดเจนขึ้นเมื่อแค็ตตาล็อก ข้อมูลประจำตัว สตรีม การหมดเวลา Compaction หรือเส้นทางสำรองเกิดปัญหา การกำหนดค่า [OpenAI](/th/providers/openai), [OpenRouter](/th/providers/openrouter) และ [OpenCode Go](/th/providers/opencode-go) ที่ได้รับผลกระทบตอนนี้รายงานหรือกู้คืนจากความล้มเหลวได้สม่ำเสมอยิ่งขึ้น แทนที่จะปล่อยให้ผู้ใช้เจอกับตัวเลือกที่ล้าสมัยหรือคำขอที่หยุดค้าง

การแก้ไขต่อเนื่องสำหรับ[โหมดเร็ว](/th/tools/thinking#fast-mode-fast) ปรับปรุงลักษณะการทำงานในเส้นทางผู้ให้บริการและเส้นทางสำรองที่ได้รับผลกระทบ ส่วนโหมดเร็วอัตโนมัติเองไม่ใช่ของใหม่ในรุ่นนี้

### เซสชัน หน่วยความจำ และความต่อเนื่องของความไว้วางใจ

[เซสชัน](/th/concepts/session), [Compaction](/th/concepts/compaction), [หน่วยความจำ](/th/concepts/memory) และ[หน่วยความจำที่รองรับด้วย QMD](/th/concepts/memory-qmd) ช่วยรักษาบทสนทนาที่ตั้งใจไว้และบริบทที่มีประโยชน์ได้อย่างสม่ำเสมอยิ่งขึ้นตลอดงานที่ทำงานยาวนาน การเชื่อมต่อใหม่ การอัปเกรด และการซ่อมแซมทรานสคริปต์ [การค้นหาเครื่องมือ](/th/tools/tool-search) ยังทำงานได้เชื่อถือได้มากขึ้นเมื่อเอเจนต์ต้องกู้คืนบริบทหรือความสามารถที่ถูกต้อง

การกู้คืน [Matrix](/th/channels/matrix) แบบเข้ารหัสจะหยุดอย่างปลอดภัยเมื่อไม่สามารถตรวจสอบสถานะคีย์ที่จำเป็นได้ [นโยบายเครื่องมือ](/th/gateway/sandbox-vs-tool-policy-vs-elevated), [การอนุมัติ](/th/cli/approvals) และ[การจัดการความลับ](/th/gateway/secrets) ยังคงผูกกับสถานะรันไทม์ที่ตั้งใจไว้ โดยการกระทำที่มีความเสี่ยงสูงกว่าจะยังคงถูกปิดใช้งาน เว้นแต่จะเปิดใช้งานอย่างชัดเจน

### การติดตั้งและซ่อมแซม Plugin ทางการ

[การจัดการ Plugin](/th/plugins/manage-plugins) ตอนนี้รองรับการผสานรวมทางการเพิ่มเติมผ่านโฟลว์การติดตั้งและซ่อมแซมแพ็กเกจภายนอกตามปกติ [คลังรายการ Plugin](/th/plugins/plugin-inventory) และการตรวจสอบการตั้งค่าที่เกี่ยวข้องให้ผลลัพธ์ที่ชัดเจนขึ้นแก่ผู้ใช้เมื่อแพ็กเกจขาดหาย ไม่เข้ากัน หรือต้องติดตั้งใหม่

<Accordion title="Sources and contributors">

- เมื่อ `plugins.allow` ใช้ชื่อช่องทางหรือชื่อแพ็กเกจแทน id Plugin จริง คำแนะนำตอนเริ่มทำงานจะระบุรายการที่ไม่ตรงกันและแสดง id Plugin ที่ค้นพบซึ่งจำเป็นต่อการแก้ไขการกำหนดค่า [#68389](https://github.com/openclaw/openclaw/pull/68389) เกี่ยวข้องกับ [#68352](https://github.com/openclaw/openclaw/issues/68352). ขอบคุณ @aym9999, @jirboy, @lyfuci, @pahuchi-joe, @zmxccxy.
- คำเตือนความน่าเชื่อถือของ Plugin สำหรับการติดตั้งครั้งแรกหรือการติดตั้งใหม่ตอนนี้มีตัวอย่าง `plugins.allow` ที่พร้อมคัดลอก และคำสั่งสำหรับแสดงรายการหรือตรวจสอบ id Plugin เพื่อให้ผู้ใช้แก้ไขคำเตือนได้ก่อนเชื่อถือหรือติดตั้งโค้ด Plugin ใหม่ [#78105](https://github.com/openclaw/openclaw/pull/78105) เกี่ยวข้องกับ [#68780](https://github.com/openclaw/openclaw/issues/68780). ขอบคุณ @jirboy, @pahuchi-joe.
- การย้ายข้อมูล Codex ตอนนี้ทำงานกับการติดตั้ง Plugin ส่วนกลางมาตรฐานได้แล้ว เพราะ `openclaw migrate` สามารถค้นหาผู้ให้บริการที่ติดตั้งไว้ได้แทนที่จะล้มเหลวด้วย `Unknown migration provider` [#89612](https://github.com/openclaw/openclaw/pull/89612) เกี่ยวข้องกับ [#89609](https://github.com/openclaw/openclaw/issues/89609). ขอบคุณ @mugabuga, @zerone0x.
- การติดตั้งและอัปเดต Plugin สามารถกู้คืนจากพิน dependency ที่จัดการโดย OpenClaw ซึ่งล้าสมัย แทนที่จะล้มเหลวด้วย `npm EOVERRIDE` โดยไม่ทำให้การซิงโครไนซ์ภายหลังลดเวอร์ชันหรือลบแพ็กเกจที่ผู้ใช้ติดตั้งไว้อย่างชัดเจน [#91786](https://github.com/openclaw/openclaw/pull/91786) เกี่ยวข้องกับ [#91772](https://github.com/openclaw/openclaw/issues/91772). ขอบคุณ @amknight, @mkdelta221.
- นักพัฒนา Plugin ช่องทางสามารถส่งตัวระบุผู้ส่งและการสนทนาแบบเนทีฟผ่าน hook และเวิร์กโฟลว์ exec ที่เลือกได้แล้ว ทำให้การผสานรวมกำหนดเส้นทางได้แม่นยำขึ้นโดยไม่ทำให้ช่องผู้ส่งและแชตเดิมเสียหาย [#91903](https://github.com/openclaw/openclaw/pull/91903) ขอบคุณ @lanzhi-lee, @vincentkoc.
- การค้นพบ Plugin ตอนนี้ทำการตรวจสอบระบบไฟล์ที่บล็อกระหว่างเริ่มทำงานน้อยลง ลดงาน cold-start ที่หลีกเลี่ยงได้สำหรับแผนผัง Plugin ที่รวมมา โดยเฉพาะบนระบบไฟล์ Windows ที่ช้ากว่า โดยไม่เปลี่ยนพฤติกรรมการค้นพบบันเดิล [#93919](https://github.com/openclaw/openclaw/pull/93919) เกี่ยวข้องกับ [#76209](https://github.com/openclaw/openclaw/issues/76209). ขอบคุณ @ml12580, @shenhonglong456-ai.
- เมธอด Plugin Gateway ตอนนี้ทำงานผ่าน `openclaw gateway call` หลังลงทะเบียนแล้ว เพื่อให้ผู้เขียน Plugin ใช้งานจากสคริปต์และงาน Cron ได้แทนที่จะเจอข้อผิดพลาด `unknown method` [#94154](https://github.com/openclaw/openclaw/pull/94154) เกี่ยวข้องกับ [#94127](https://github.com/openclaw/openclaw/issues/94127). ขอบคุณ @brycemurray, @pick-cat, @vincentkoc.
- การค้นพบและการตรวจสอบการติดตั้ง Skills ของ ClawHub มีโอกาสค้างหรือทำให้ OpenClaw ล่มน้อยลง เพราะการตอบกลับ marketplace ที่ใหญ่เกินไปหรือค้างอยู่จะถูกหยุดก่อนที่จะใช้หน่วยความจำจนหมด [#95226](https://github.com/openclaw/openclaw/pull/95226) ขอบคุณ @alix-007.

## Gateway ความปลอดภัย และความน่าเชื่อถือ

<Accordion title="แหล่งที่มาและผู้มีส่วนร่วม">

- ตอนนี้ Control UI สามารถสร้างงาน Early Morning ด้วยพรีเซ็ต Silent ในเซสชันหลักและไม่ต้องมีการแจ้งเตือน แทนที่จะปล่อยกล่องโต้ตอบค้างไว้โดยไม่มีผลลัพธ์ที่มองเห็นได้ [#95459](https://github.com/openclaw/openclaw/pull/95459) เกี่ยวข้องกับ [#95073](https://github.com/openclaw/openclaw/issues/95073) ขอบคุณ @vincentkoc, @vporton, @zoowh

## เอกสารและเครื่องมือผู้ดูแลระบบ

### ความน่าเชื่อถือของการตั้งค่าและคำสั่ง

ตอนนี้ [คำสั่ง CLI](/th/cli/index) ทั่วไปจัดการการกำหนดค่า พาธ เอาต์พุต และกรณีล้มเหลวได้สม่ำเสมอยิ่งขึ้น [Shell completion](/th/cli/completion), [doctor](/th/cli/doctor), [คำสั่ง config](/th/cli/config) และ [การกำหนดค่า Gateway](/th/gateway/configuration) ให้คำแนะนำที่ชัดเจนขึ้นและการกู้คืนที่ปลอดภัยขึ้นเมื่อการติดตั้งหรือการตั้งค่าต้องได้รับการดูแล

<Accordion title="แหล่งที่มาและผู้มีส่วนร่วม">

- ตอนนี้เมนูเติมแท็บของ zsh สำหรับ OpenClaw แสดงคำอธิบายตัวเลือกที่มีตัวแปร `$` หรือตัวอย่างที่ครอบด้วย backtick แบบตรงตามตัวอักษร แทนที่จะประเมินเป็นอินพุตของ shell แล้วทำให้เมนูเสียหาย [#64490](https://github.com/openclaw/openclaw/pull/64490) ขอบคุณ @edenkangdw.
- หลังอัปเกรดจากพื้นที่จัดเก็บ sandbox รุ่นเก่า ตอนนี้ผู้ดูแลระบบจะได้รับคำเตือน `openclaw doctor` ที่ชัดเจนเกี่ยวกับไฟล์ registry ที่เหลืออยู่ และสามารถใช้ `openclaw doctor --fix` เพื่อย้ายหรือทำความสะอาดไฟล์เหล่านั้นได้ [#84326](https://github.com/openclaw/openclaw/pull/84326) ขอบคุณ @giodl73-repo.
- ตอนนี้ผู้ดูแลระบบสามารถใช้ `doctor --lint` เพื่อตรวจหาบริการ Gateway แบบ legacy ที่ค้างอยู่และดูตัวอย่างการทำความสะอาดได้ ขณะที่บริการเพิ่มเติมที่ตั้งใจมีไว้จะยังคงเป็นข้อมูลประกอบและไม่ทำให้การตรวจสอบค่าเริ่มต้นล้มเหลว [#84340](https://github.com/openclaw/openclaw/pull/84340) ขอบคุณ @giodl73-repo.
- ตอนนี้ผู้ดูแล Gateway บน macOS จะได้รับคำเตือนก่อนที่การติดตั้งใหม่ ซ่อมแซม หรือรีสตาร์ตจะเขียนทับพฤติกรรม wrapper ของ LaunchAgent ที่ปรับแต่งไว้ ขณะที่ `openclaw status` แยกการตรวจ secret ที่ขาดหายเฉพาะ CLI ออกจากบริการที่ติดตั้งแล้ว [#90537](https://github.com/openclaw/openclaw/pull/90537) เกี่ยวข้องกับ [#90518](https://github.com/openclaw/openclaw/issues/90518). ขอบคุณ @turbotheturtle, @vincentkoc.
- ตอนนี้สามารถส่งพรอมป์ที่ยาว มีหลายบรรทัด หรือมีโค้ดจำนวนมากไปยัง `openclaw agent` ด้วย `--message-file` ได้ ช่วยหลีกเลี่ยงการ quote ใน shell ที่เปราะบาง และรายงานไฟล์ที่ไม่ถูกต้องก่อน dispatch [#93351](https://github.com/openclaw/openclaw/pull/93351) ขอบคุณ @ooiuuii.
- แก้ไขการซ่อมแซม `doctor --fix --non-interactive` แบบกำหนดเวลาไม่ให้รีสตาร์ต Gateway ที่กำลังทำงานอยู่หลังการตรวจสุขภาพล้มเหลวชั่วคราว ทำให้การบำรุงรักษาแบบไม่ต้องเฝ้าดูไม่รบกวนบริการที่กำลังทำงานอีกต่อไป [#94148](https://github.com/openclaw/openclaw/pull/94148) เกี่ยวข้องกับ [#78217](https://github.com/openclaw/openclaw/issues/78217). ขอบคุณ @esqandil, @zhangguiping-xydt.
- ตอนนี้ `openclaw configure` และ `openclaw config` แบบไม่มีอาร์กิวเมนต์จะหยุดพร้อมคำแนะนำ subcommand ที่ชัดเจนเมื่อรันจากสคริปต์หรือ pipe แทนที่จะเปิด wizard แบบโต้ตอบที่ไม่สมบูรณ์แล้วออกโดยไม่ชัดเจน [#94238](https://github.com/openclaw/openclaw/pull/94238) เกี่ยวข้องกับ [#93953](https://github.com/openclaw/openclaw/issues/93953). ขอบคุณ @nianjiuzst, @ruomuxydt.
