# การเผยแพร่

การเผยแพร่บน ClawHub มีขอบเขตตามเจ้าของ: ทุกการเผยแพร่จะกำหนดเป้าหมายเป็นผู้เผยแพร่หนึ่งราย และ
เซิร์ฟเวอร์จะตัดสินว่าผู้ใช้ที่ลงชื่อเข้าใช้นั้นได้รับอนุญาตให้เผยแพร่ที่นั่นหรือไม่

## เจ้าของ

เจ้าของคือแฮนเดิลผู้เผยแพร่ของ ClawHub เช่น `@alice` หรือ `@openclaw`
เจ้าของส่วนบุคคลจะถูกสร้างสำหรับผู้ใช้ ส่วนเจ้าของแบบองค์กรสามารถมีสมาชิกหลายคนได้

เมื่อคุณเผยแพร่ คุณจะใช้เจ้าของส่วนบุคคลของคุณ หรือเลือกเจ้าของแบบองค์กร
ที่คุณมีสิทธิ์เข้าถึงในฐานะผู้เผยแพร่

## Skills

Skills ถูกเผยแพร่จากโฟลเดอร์ Skills หน้าสาธารณะคือ:

```text
https://clawhub.ai/<owner>/<slug>
```

ตัวอย่าง:

```text
https://clawhub.ai/alice/review-helper
```

คำขอเผยแพร่ประกอบด้วยเจ้าของที่เลือก, slug, เวอร์ชัน, บันทึกการเปลี่ยนแปลง และ
ไฟล์ เซิร์ฟเวอร์จะตรวจสอบว่าผู้ดำเนินการสามารถเผยแพร่ในนามเจ้าของรายนั้นได้ก่อน
สร้างรีลีส

หากต้องการย้าย Skills ที่มีอยู่ไปยังเจ้าของรายอื่นในขณะเผยแพร่เวอร์ชันใหม่ ให้เลือก
เจ้าของรายใหม่และยืนยันการย้ายความเป็นเจ้าของอย่างชัดเจน ใน CLI/API ให้ส่ง
เจ้าของเป้าหมายพร้อมการเลือกเข้าร่วมการย้าย:

```sh
clawhub skill publish ./review-helper --owner openclaw --migrate-owner --version 1.2.0
```

การย้ายเจ้าของ Skills ต้องมีสิทธิ์ผู้ดูแลระบบหรือเจ้าของทั้งในเจ้าของปัจจุบัน
และเจ้าของปลายทาง โดยจะคง Skills, ประวัติเวอร์ชัน, สถิติ,
ความคิดเห็น, fork, alias และบันทึกการตรวจสอบไว้ URL ของเจ้าของเดิมจะยังใช้งานได้ผ่าน
เส้นทาง alias/redirect

## Plugin

Plugin ใช้ชื่อแพ็กเกจแบบ npm ชื่อแพ็กเกจที่มี scope จะรวมเจ้าของไว้ใน
ส่วนแรกของชื่อ:

```text
@owner/package-name
```

scope ต้องตรงกับเจ้าของการเผยแพร่ที่เลือก หากแพ็กเกจของคุณชื่อ
`@openclaw/dronzer` จะเผยแพร่ได้เฉพาะในนาม `@openclaw` เท่านั้น หากคุณเผยแพร่ในนาม
`@vintageayu` ให้เปลี่ยนชื่อแพ็กเกจเป็น `@vintageayu/dronzer`

สิ่งนี้ป้องกันไม่ให้แพ็กเกจอ้างสิทธิ์ namespace ขององค์กรที่ผู้เผยแพร่
ไม่ได้ควบคุม

## ขั้นตอนการรีลีส

1. UI, CLI หรือเวิร์กโฟลว์ GitHub รวบรวมเมทาดาทาและไฟล์ของแพ็กเกจ
2. คำขอเผยแพร่ถูกส่งไปยัง ClawHub พร้อมเจ้าของที่เลือก
3. เซิร์ฟเวอร์ตรวจสอบสิทธิ์ของเจ้าของ, scope ของแพ็กเกจ, ชื่อแพ็กเกจ, เวอร์ชัน,
   ขีดจำกัดไฟล์ และเมทาดาทาแหล่งที่มา
4. ClawHub จัดเก็บรีลีสและเริ่มการตรวจสอบความปลอดภัยอัตโนมัติ
5. รีลีสใหม่จะถูกซ่อนจากพื้นที่ติดตั้ง/ดาวน์โหลดตามปกติจนกว่าการตรวจสอบ
   และการยืนยันจะเสร็จสิ้น

หากการตรวจสอบไม่ผ่าน รีลีสจะไม่ถูกสร้าง

## FAQ

### scope ของแพ็กเกจต้องตรงกับเจ้าของที่เลือก

หาก scope ของแพ็กเกจและเจ้าของที่เลือกไม่ตรงกัน ClawHub จะปฏิเสธ
การเผยแพร่:

```text
Package scope "@openclaw" must match selected owner "@vintageayu".
Publish as "@openclaw" or rename this package to "@vintageayu/dronzer".
```

หากต้องการแก้ไข ให้เลือกเจ้าของที่ระบุโดย scope ของแพ็กเกจ หรือเปลี่ยนชื่อ
แพ็กเกจเพื่อให้ scope ตรงกับเจ้าของที่คุณสามารถเผยแพร่ในนามได้

หากชื่อแพ็กเกจมี scope ที่ถูกต้องอยู่แล้ว แต่แพ็กเกจเป็นของผู้เผยแพร่
ผิดราย ให้โอนความเป็นเจ้าของแทน:

```sh
clawhub package transfer @opik/opik-openclaw --to opik
```

ใช้การโอนแพ็กเกจเฉพาะเมื่อคุณมีสิทธิ์ผู้ดูแลระบบทั้งกับเจ้าของแพ็กเกจปัจจุบัน
และผู้เผยแพร่ปลายทาง การโอนนี้ไม่ได้ทำให้คุณเผยแพร่เข้าไปใน scope ที่คุณ
จัดการไม่ได้

สิ่งนี้ปกป้อง namespace ขององค์กร แพ็กเกจชื่อ `@openclaw/dronzer` อ้างสิทธิ์
namespace `@openclaw` ดังนั้นเฉพาะผู้เผยแพร่ที่มีสิทธิ์เข้าถึงเจ้าของ `@openclaw`
เท่านั้นที่สามารถเผยแพร่ได้
