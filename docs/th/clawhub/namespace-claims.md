# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลเจ้าของ แฮนเดิลองค์กร slug ของสกิล ชื่อแพ็กเกจ Plugin และ
ขอบเขตแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูก
อ้างสิทธิ์แล้ว สงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub ให้ขอให้เจ้าหน้าที่ตรวจสอบ
ผ่าน
[แบบฟอร์มปัญหาการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะและไม่อ่อนไหว อย่าใช้รายงาน
ในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## เมื่อใดควรเปิดการอ้างสิทธิ์

เปิดการอ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
เนมสเปซควรถูกสงวน โอน เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias
หรือเปลี่ยนแปลงด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างเช่น:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub โปรเจกต์ บริษัท หรือชุมชนของคุณ
- ขอบเขตแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ภายใต้เจ้าของ
  ClawHub ที่ตรงกันเท่านั้น
- slug ของสกิลหรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่มีสิทธิ์

หากรายการไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทเรื่องความเป็นเจ้าของ
ให้ทำตามคำแนะนำด้านการดูแลเนื้อหาหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มการอ้างสิทธิ์
เนมสเปซมีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อที่มีขอบเขต เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้เนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณจัดการเจ้าของปัจจุบันไม่ได้ หรือเมื่อเจ้าหน้าที่จำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรใส่

ใช้หลักฐานสาธารณะที่ไม่อ่อนไหว หลักฐานที่เป็นประโยชน์รวมถึง:

- ประวัติองค์กร GitHub, repo, release หรือผู้ดูแล
- เอกสารทางการของโปรเจกต์ที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลทางการ
- การควบคุมขอบเขตของ npm, PyPI, crates.io หรือรีจิสทรีแพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโปรเจกต์ที่สามารถพูดคุย
  ต่อสาธารณะได้อย่างปลอดภัย
- ประวัติ repository ต้นทาง ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ สกิล Plugin แพ็กเกจ หรือ issue ของ ClawHub ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือข้อมูลลับ

## สิ่งที่ไม่ควรใส่

อย่าใส่ข้อมูลลับหรือหลักฐานส่วนตัวใน issue สาธารณะบน GitHub อย่าใส่:

- โทเค็น API, signing key หรือข้อมูลรับรอง
- โทเค็น DNS challenge
- ไฟล์กฎหมายหรือสัญญาส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มการอ้างสิทธิ์จะถามว่าหลักฐานที่อ่อนไหวจำเป็นต้องใช้ช่องทางส่วนตัวกับเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์เนื้อหาที่อ่อนไหวต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่ม alias หรือ redirect ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอน
เจ้าหน้าที่จะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
