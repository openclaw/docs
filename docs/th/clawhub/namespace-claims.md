# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลเจ้าของ แฮนเดิลองค์กร slug ของ skill ชื่อแพ็กเกจ plugin และ
scope ของแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูก
อ้างสิทธิ์ จองไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทอยู่แล้วบน ClawHub ให้ขอให้เจ้าหน้าที่ตรวจสอบ
ด้วย
[แบบฟอร์มประเด็นการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะและไม่ละเอียดอ่อน ห้ามใช้รายงาน
ในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## เมื่อใดควรเปิดการอ้างสิทธิ์

เปิดการอ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
ควรจอง โอน เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias หรือ
เปลี่ยนแปลงเนมสเปซด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่าง ได้แก่:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub, โปรเจกต์, บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ได้เฉพาะภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- slug ของ skill หรือชื่อแพ็กเกจ plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ได้ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซที่มีสิทธิ์
  อย่างถูกต้อง

หากรายการไม่ปลอดภัย มีเจตนาร้าย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทด้านความเป็นเจ้าของ
ให้ปฏิบัติตามแนวทางการดูแลหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มการอ้างสิทธิ์เนมสเปซ
มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ plugin ชื่อแบบมี scope เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อเจ้าหน้าที่จำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่ละเอียดอ่อน หลักฐานที่เป็นประโยชน์ ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือ maintainer
- เอกสารทางการของโปรเจกต์ที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลทางการ
- การควบคุม scope บน npm, PyPI, crates.io หรือ registry แพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโปรเจกต์ที่ปลอดภัยสำหรับการพูดคุย
  ต่อสาธารณะ
- ประวัติ repository ต้นทาง ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อแบบสาธารณะ
- ลิงก์ไปยังเจ้าของ ClawHub, skill, plugin, แพ็กเกจ หรือ issue ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรรวมไว้

ห้ามใส่ความลับหรือหลักฐานส่วนตัวใน issue สาธารณะบน GitHub ห้ามรวม:

- token ของ API, คีย์ลงนาม หรือข้อมูลรับรอง
- token สำหรับ DNS challenge
- ไฟล์กฎหมายหรือสัญญาส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มการอ้างสิทธิ์จะถามว่าหลักฐานละเอียดอ่อนจำเป็นต้องใช้ช่องทางส่วนตัวกับเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์ข้อมูลละเอียดอ่อนต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

เจ้าหน้าที่ ClawHub อาจจองเนมสเปซ โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร
ซ่อนหรือกักกันรายการที่มีอยู่ เพิ่ม alias หรือ redirect ขอหลักฐานเพิ่มเติม
หรือปฏิเสธคำขอ ทั้งนี้ขึ้นอยู่กับหลักฐานและความเสี่ยง

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าทุกชื่อที่ตรงกันจะถูกโอน
เจ้าหน้าที่จะชั่งน้ำหนักหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
