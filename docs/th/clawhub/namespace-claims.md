# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลเจ้าของ แฮนเดิลองค์กร สลักของสกิล ชื่อแพ็กเกจ Plugin และ
สโคปแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โครงการ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรที่มีอยู่จริง แต่มีผู้
อ้างสิทธิ์ ถูกสงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub อยู่แล้ว ให้ขอให้เจ้าหน้าที่ตรวจสอบ
ผ่าน
[แบบฟอร์มปัญหาการอ้างสิทธิ์องค์กร/เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบสิทธิ์ความเป็นเจ้าของที่เป็นสาธารณะและไม่มีข้อมูลละเอียดอ่อน อย่าใช้รายงาน
ภายในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## เมื่อใดควรเปิดคำขออ้างสิทธิ์

เปิดคำขออ้างสิทธิ์เนมสเปซเมื่อเห็นว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
ควรสงวน โอน เปลี่ยนชื่อ ซ่อน กักกัน กำหนดนามแฝง
หรือเปลี่ยนแปลงเนมสเปซด้วยวิธีอื่นเนื่องจากสิทธิ์ความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่าง ได้แก่:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub โครงการ บริษัท หรือชุมชนของคุณ
- สโคปแพ็กเกจ เช่น `@example-org/*` ซึ่งควรเผยแพร่ได้เฉพาะภายใต้
  เจ้าของ ClawHub ที่ตรงกัน
- สลักของสกิลหรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโครงการ
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโครงการ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่มีการใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่มีสิทธิ์โดยชอบ

หากรายการนั้นไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดนอกเหนือจากข้อพิพาทด้านสิทธิ์ความเป็นเจ้าของ
ให้ปฏิบัติตามคำแนะนำด้านการดูแลเนื้อหาหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มอ้างสิทธิ์เนมสเปซ
มีไว้สำหรับการตรวจสอบสิทธิ์ความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนยื่นคำขอ

ก่อนอื่น ให้ยืนยันว่ากำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อแบบมีสโคป เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้คำขออ้างสิทธิ์
เมื่อไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อจำเป็นต้องให้เจ้าหน้าที่แก้ไข
ข้อพิพาท

## หลักฐานที่ควรแนบ

ใช้หลักฐานที่เป็นสาธารณะและไม่มีข้อมูลละเอียดอ่อน หลักฐานที่เป็นประโยชน์ ได้แก่:

- ประวัติองค์กร รีโพซิทอรี รุ่นเผยแพร่ หรือผู้ดูแลบน GitHub
- เอกสารอย่างเป็นทางการของโครงการที่ระบุชื่อเนมสเปซ
- หลักฐานจากโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุมสโคปบน npm, PyPI, crates.io หรือทะเบียนแพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโครงการที่สามารถอภิปราย
  ต่อสาธารณะได้อย่างปลอดภัย
- ประวัติรีโพซิทอรีต้นทาง ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อที่เป็นสาธารณะ
- ลิงก์ไปยังเจ้าของ สกิล Plugin แพ็กเกจ หรือปัญหาบน ClawHub ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรสามารถเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองหรือข้อมูลลับส่วนตัว

## สิ่งที่ไม่ควรแนบ

อย่าใส่ข้อมูลลับหรือหลักฐานส่วนตัวในปัญหา GitHub สาธารณะ อย่าแนบ:

- โทเค็น API คีย์ลงนาม หรือข้อมูลรับรอง
- โทเค็นการยืนยัน DNS
- เอกสารทางกฎหมายหรือสัญญาส่วนตัว
- เอกสารยืนยันตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มอ้างสิทธิ์จะสอบถามว่าหลักฐานละเอียดอ่อนจำเป็นต้องส่งผ่านช่องทางส่วนตัวของเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์ข้อมูลละเอียดอ่อนต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ
โอนสิทธิ์ความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่มนามแฝงหรือการเปลี่ยนเส้นทาง ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ ทั้งนี้ขึ้นอยู่กับหลักฐานและความเสี่ยง

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าจะมีการโอนชื่อที่ตรงกันทุกรายการ
เจ้าหน้าที่จะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
