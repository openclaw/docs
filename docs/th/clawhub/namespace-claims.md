# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลเจ้าของ แฮนเดิลองค์กร slug ของ Skills ชื่อแพ็กเกจ Plugin และ
scope ของแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูก
อ้างสิทธิ์ สงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub แล้ว ให้ขอให้ทีมงานตรวจสอบ
ด้วย
[แบบฟอร์ม issue สำหรับการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะที่ไม่ใช่ข้อมูลอ่อนไหว อย่าใช้รายงาน
ในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## ควรเปิดการอ้างสิทธิ์เมื่อใด

เปิดการอ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าทีมงาน ClawHub ควรตรวจสอบว่า
เนมสเปซควรถูกสงวนไว้ โอน เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias
หรือเปลี่ยนแปลงด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub โปรเจกต์ บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ได้เฉพาะภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- slug ของ Skills หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ได้ใช้งาน หรือติดต่อไม่ได้ ซึ่งขวางเจ้าของเนมสเปซ
  ที่ชอบธรรม

หากรายการไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดนอกเหนือจากข้อพิพาทเรื่องความเป็นเจ้าของ
ให้ทำตามแนวทางการดูแลหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มการอ้างสิทธิ์เนมสเปซ
มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ขั้นแรกให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อแบบมี scope เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณจัดการเจ้าของปัจจุบันไม่ได้ หรือเมื่อทีมงานจำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่ใช่ข้อมูลอ่อนไหว หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือ maintainer
- เอกสารโปรเจกต์อย่างเป็นทางการที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุม scope บน npm, PyPI, crates.io หรือ registry แพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโปรเจกต์ที่ปลอดภัยต่อการพูดคุย
  แบบสาธารณะ
- ประวัติ source repository ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อแบบสาธารณะ
- ลิงก์ไปยังเจ้าของ ClawHub, Skills, Plugin, แพ็กเกจ หรือ issue ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร ทีมงานควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรรวมไว้

อย่าใส่ความลับหรือหลักฐานส่วนตัวใน issue สาธารณะบน GitHub อย่ารวม:

- โทเค็น API, signing keys หรือข้อมูลรับรอง
- โทเค็น DNS challenge
- ไฟล์กฎหมายหรือสัญญาส่วนตัว
- เอกสารยืนยันตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มการอ้างสิทธิ์จะถามว่าหลักฐานอ่อนไหวจำเป็นต้องใช้ช่องทางส่วนตัวกับทีมงานหรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์ข้อมูลอ่อนไหวแบบสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ทีมงาน ClawHub อาจสงวนเนมสเปซ โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร
ซ่อนหรือกักกันรายการที่มีอยู่ เพิ่ม alias หรือ redirect ขอหลักฐานเพิ่มเติม
หรือปฏิเสธคำขอ ทั้งนี้ขึ้นอยู่กับหลักฐานและความเสี่ยง

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอน
ทีมงานจะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
