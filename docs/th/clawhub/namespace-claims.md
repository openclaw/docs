# การอ้างสิทธิ์ Org และ Namespace

ClawHub ใช้ owner handles, org handles, skill slugs, ชื่อแพ็กเกจ Plugin และ
ขอบเขตแพ็กเกจเป็น namespace สาธารณะ หาก namespace ดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูกอ้างสิทธิ์แล้ว
ถูกสงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub ให้ขอให้ทีมงานตรวจสอบด้วย
[แบบฟอร์มปัญหาการอ้างสิทธิ์ Org / Namespace](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้เส้นทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะที่ไม่อ่อนไหว อย่าใช้รายงานในผลิตภัณฑ์
หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์ namespace

## ควรเปิดการอ้างสิทธิ์เมื่อใด

เปิดการอ้างสิทธิ์ namespace เมื่อคุณเชื่อว่าทีมงาน ClawHub ควรตรวจสอบว่า
namespace ควรถูกสงวนไว้ โอน เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias
หรือเปลี่ยนแปลงในรูปแบบอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างรวมถึง:

- org handle ที่ตรงกับ GitHub org, โปรเจกต์ บริษัท หรือชุมชนของคุณ
- ขอบเขตแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- skill slug หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนปลอมแปลงเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของ namespace
  ที่มีสิทธิ์โดยชอบธรรม

หากรายการไม่ปลอดภัย มีเจตนาร้าย หรือทำให้เข้าใจผิดนอกเหนือจากข้อพิพาทความเป็นเจ้าของ
ให้ปฏิบัติตามคำแนะนำด้านการดูแลหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มอ้างสิทธิ์ namespace
มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับ namespace
สำหรับแพ็กเกจ Plugin ชื่อแบบมีขอบเขต เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณสามารถจัดการเจ้าของปัจจุบันได้ ให้แก้ไข namespace โดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อทีมงานจำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่อ่อนไหว หลักฐานที่เป็นประโยชน์รวมถึง:

- ประวัติ GitHub org, repo, release หรือ maintainer
- เอกสารโปรเจกต์อย่างเป็นทางการที่ระบุ namespace
- หลักฐานโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุมขอบเขต npm, PyPI, crates.io หรือ package-registry อื่นๆ
- หลักฐานเครื่องหมายการค้า แบรนด์ หรือความเป็นเจ้าของโปรเจกต์ที่สามารถพูดคุย
  ต่อสาธารณะได้อย่างปลอดภัย
- ประวัติ repository ต้นทาง ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ Skills, Plugin, แพ็กเกจ หรือปัญหา ClawHub ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร ทีมงานควรสามารถเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรรวมไว้

อย่าใส่ความลับหรือหลักฐานส่วนตัวใน GitHub issue สาธารณะ อย่ารวม:

- API tokens, signing keys หรือ credentials
- DNS challenge tokens
- ไฟล์กฎหมายหรือสัญญาส่วนตัว
- เอกสารยืนยันตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มอ้างสิทธิ์จะถามว่าหลักฐานที่อ่อนไหวต้องใช้ช่องทางส่วนตัวกับทีมงานหรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์เนื้อหาที่อ่อนไหวต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง ทีมงาน ClawHub อาจสงวน namespace
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่ม alias หรือการเปลี่ยนเส้นทาง ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบ namespace ไม่รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอน
ทีมงานจะชั่งน้ำหนักหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
