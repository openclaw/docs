# การอ้างสิทธิ์องค์กรและ Namespace

ClawHub ใช้แฮนเดิลเจ้าของ แฮนเดิลองค์กร slug ของทักษะ ชื่อแพ็กเกจ Plugin และ
scope ของแพ็กเกจเป็น namespace สาธารณะ หาก namespace ดูเหมือนเป็นของ
โปรเจกต์จริง แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรจริง แต่ถูกอ้างสิทธิ์แล้ว
ถูกสงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub ให้ขอให้ทีมงานตรวจสอบด้วย
[แบบฟอร์มปัญหาการอ้างสิทธิ์องค์กร / Namespace](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้เส้นทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะและไม่ละเอียดอ่อน อย่าใช้รายงานในผลิตภัณฑ์
หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์ namespace

## เมื่อใดควรเปิดคำร้องอ้างสิทธิ์

เปิดคำร้องอ้างสิทธิ์ namespace เมื่อคุณเชื่อว่าทีมงาน ClawHub ควรตรวจสอบว่า
namespace ควรถูกสงวนไว้ โอนสิทธิ์ เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias
หรือเปลี่ยนแปลงด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub, โปรเจกต์, บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ได้เฉพาะภายใต้เจ้าของ
  ClawHub ที่ตรงกันเท่านั้น
- slug ของทักษะหรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ได้ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของ namespace
  ที่มีสิทธิ์

หากรายการไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทเรื่องความเป็นเจ้าของ
ให้ปฏิบัติตามคำแนะนำด้านการกลั่นกรองหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มอ้างสิทธิ์
namespace มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับ namespace
สำหรับแพ็กเกจ Plugin ชื่อที่มี scope เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้ไข namespace โดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอนสิทธิ์ ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้คำร้องอ้างสิทธิ์
เมื่อคุณจัดการเจ้าของปัจจุบันไม่ได้ หรือเมื่อทีมงานจำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่ละเอียดอ่อน หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือ maintainer
- เอกสารทางการของโปรเจกต์ที่ระบุชื่อ namespace
- หลักฐานโดเมนหรือโดเมนอีเมลทางการ
- การควบคุม scope ใน npm, PyPI, crates.io หรือ registry แพ็กเกจอื่น
- หลักฐานเครื่องหมายการค้า แบรนด์ หรือความเป็นเจ้าของโปรเจกต์ที่ปลอดภัยต่อการพูดคุย
  สาธารณะ
- ประวัติ repository ต้นทาง ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ ทักษะ Plugin แพ็กเกจ หรือ issue บน ClawHub ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร ทีมงานควรเข้าใจความสัมพันธ์ได้
โดยไม่ต้องใช้ข้อมูลรับรองหรือความลับส่วนตัว

## สิ่งที่ไม่ควรรวมไว้

อย่าใส่ความลับหรือหลักฐานส่วนตัวใน issue GitHub สาธารณะ อย่ารวม:

- API tokens, signing keys หรือข้อมูลรับรอง
- DNS challenge tokens
- ไฟล์กฎหมายหรือสัญญาส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มอ้างสิทธิ์จะถามว่าหลักฐานละเอียดอ่อนต้องใช้ช่องทางส่วนตัวกับทีมงานหรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์ข้อมูลละเอียดอ่อนต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง ทีมงาน ClawHub อาจสงวน namespace
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่ม alias หรือ redirect ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบ namespace ไม่รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอนสิทธิ์
ทีมงานจะชั่งน้ำหนักหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
