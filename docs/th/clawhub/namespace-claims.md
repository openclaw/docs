# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้ชื่ออ้างอิงเจ้าของ ชื่ออ้างอิงองค์กร slug ของ skill ชื่อแพ็กเกจ Plugin และ
scope ของแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โครงการ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรที่มีอยู่จริง แต่ถูก
อ้างสิทธิ์ สงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub อยู่แล้ว โปรดขอให้เจ้าหน้าที่ตรวจสอบ
โดยใช้
[แบบฟอร์มแจ้งปัญหาการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบสิทธิ์ความเป็นเจ้าของที่เป็นข้อมูลสาธารณะและไม่ละเอียดอ่อน อย่าใช้รายงาน
ภายในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## เมื่อใดควรยื่นคำขออ้างสิทธิ์

ยื่นคำขออ้างสิทธิ์เนมสเปซเมื่อคุณเห็นว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
ควรสงวน โอน เปลี่ยนชื่อ ซ่อน กักกัน เพิ่มนามแฝง
หรือเปลี่ยนแปลงเนมสเปซด้วยวิธีอื่นเนื่องจากสิทธิ์ความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- ชื่ออ้างอิงองค์กรที่ตรงกับองค์กร GitHub โครงการ บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ซึ่งควรเผยแพร่ภายใต้เจ้าของ
  `example-org` ที่ตรงกันบน ClawHub เท่านั้น
- slug ของ skill หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโครงการ
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโครงการ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่มีความเคลื่อนไหว หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่มีสิทธิ์โดยชอบธรรม

หากรายการไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดนอกเหนือจากข้อพิพาทเรื่องความเป็นเจ้าของ
ให้ปฏิบัติตามคำแนะนำด้านการควบคุมเนื้อหาหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มการอ้างสิทธิ์
เนมสเปซมีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนยื่นคำขอ

ก่อนอื่น ให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อที่มี scope เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณสามารถจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้คำขออ้างสิทธิ์
เมื่อคุณไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อเจ้าหน้าที่ต้องเข้ามาระงับ
ข้อพิพาท

## หลักฐานที่ควรแนบ

ใช้หลักฐานที่เป็นข้อมูลสาธารณะและไม่ละเอียดอ่อน หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติองค์กร GitHub ที่เก็บโค้ด รุ่นเผยแพร่ หรือผู้ดูแล
- เอกสารอย่างเป็นทางการของโครงการที่ระบุชื่อเนมสเปซ
- หลักฐานจากโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุม scope บน npm, PyPI, crates.io หรือทะเบียนแพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโครงการที่สามารถอภิปราย
  ต่อสาธารณะได้อย่างปลอดภัย
- ประวัติที่เก็บซอร์สโค้ด ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ skill, Plugin, แพ็กเกจ หรือปัญหาบน ClawHub ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรสามารถเข้าใจ
ความสัมพันธ์ดังกล่าวได้โดยไม่ต้องใช้ข้อมูลประจำตัวหรือข้อมูลลับส่วนตัว

## สิ่งที่ไม่ควรแนบ

อย่าใส่ข้อมูลลับหรือหลักฐานส่วนตัวในปัญหาสาธารณะบน GitHub อย่าแนบ:

- โทเค็น API คีย์สำหรับลงนาม หรือข้อมูลประจำตัว
- โทเค็นสำหรับการท้าทาย DNS
- ไฟล์หรือสัญญาทางกฎหมายส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มการอ้างสิทธิ์จะถามว่าหลักฐานละเอียดอ่อนจำเป็นต้องส่งผ่านช่องทางส่วนตัวของเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์เนื้อหาละเอียดอ่อนต่อสาธารณะ

## ผลลัพธ์ที่อาจเกิดขึ้น

เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่มนามแฝงหรือการเปลี่ยนเส้นทาง ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ ทั้งนี้ขึ้นอยู่กับหลักฐานและความเสี่ยง

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าจะมีการโอนชื่อที่ตรงกันทุกรายการ
เจ้าหน้าที่จะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
