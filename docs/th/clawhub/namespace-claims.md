# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลของเจ้าของ แฮนเดิลขององค์กร slug ของ Skills ชื่อแพ็กเกจ Plugin และ
scope ของแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูก
อ้างสิทธิ์แล้ว สงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub ให้ขอให้เจ้าหน้าที่ตรวจสอบ
ด้วย
[แบบฟอร์ม issue การอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้เส้นทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะที่ไม่ละเอียดอ่อน ห้ามใช้รายงาน
ในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## ควรเปิดการอ้างสิทธิ์เมื่อใด

เปิดการอ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
ควรสงวน โอน เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias
หรือเปลี่ยนแปลงเนมสเปซด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub โปรเจกต์ บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- slug ของ Skills หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ได้ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่มีสิทธิ์

หากรายการไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทด้านความเป็นเจ้าของ
ให้ทำตามคำแนะนำด้านการดูแลเนื้อหาหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มอ้างสิทธิ์เนมสเปซ
มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อแบบมี scope เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณจัดการเจ้าของปัจจุบันไม่ได้ หรือเมื่อเจ้าหน้าที่จำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่ละเอียดอ่อน หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือผู้ดูแล
- เอกสารโปรเจกต์อย่างเป็นทางการที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุม scope ใน npm, PyPI, crates.io หรือ registry แพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโปรเจกต์ที่ปลอดภัยต่อการพูดคุย
  แบบสาธารณะ
- ประวัติ repository ต้นทาง ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อแบบสาธารณะ
- ลิงก์ไปยังเจ้าของ ClawHub, Skills, Plugin, แพ็กเกจ หรือ issue ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรรวมไว้

ห้ามใส่ความลับหรือหลักฐานส่วนตัวใน issue GitHub สาธารณะ ห้ามรวม:

- โทเค็น API, signing keys หรือข้อมูลรับรอง
- โทเค็น DNS challenge
- ไฟล์หรือสัญญาทางกฎหมายส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มอ้างสิทธิ์จะถามว่าหลักฐานที่ละเอียดอ่อนจำเป็นต้องใช้ช่องทางเจ้าหน้าที่ส่วนตัวหรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์เนื้อหาที่ละเอียดอ่อนแบบสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่ม alias หรือ redirect ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอน
เจ้าหน้าที่จะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
