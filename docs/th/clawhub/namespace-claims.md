# การอ้างสิทธิ์องค์กรและ Namespace

ClawHub ใช้ handle ของเจ้าของ, handle ขององค์กร, slug ของ skill, ชื่อแพ็กเกจ Plugin และ
scope ของแพ็กเกจเป็น namespace สาธารณะ หาก namespace ดูเหมือนเป็นของ
โครงการ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูก
อ้างสิทธิ์ สงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub อยู่แล้ว ให้ขอให้ทีมงานตรวจสอบ
ผ่าน
[ฟอร์ม issue สำหรับการอ้างสิทธิ์องค์กร / Namespace](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะและไม่ละเอียดอ่อน ห้ามใช้รายงาน
ในผลิตภัณฑ์หรือฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์ namespace

## ควรเปิดการอ้างสิทธิ์เมื่อใด

เปิดการอ้างสิทธิ์ namespace เมื่อคุณเชื่อว่าทีมงาน ClawHub ควรตรวจสอบว่า
namespace ควรถูกสงวนไว้ โอน เปลี่ยนชื่อ ซ่อน กักกัน สร้าง alias
หรือเปลี่ยนแปลงด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- handle ขององค์กรที่ตรงกับองค์กร GitHub, โครงการ, บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ได้เฉพาะภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- slug ของ skill หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโครงการ
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโครงการ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ใช้งาน หรือติดต่อไม่ได้ ซึ่งขวางเจ้าของ namespace
  ที่ชอบธรรม

หากรายการนั้นไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทเรื่องความเป็นเจ้าของ
ให้ปฏิบัติตามแนวทางการกลั่นกรองหรือความปลอดภัยที่เกี่ยวข้องด้วย ฟอร์มการอ้างสิทธิ์ namespace
มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับ namespace
สำหรับแพ็กเกจ Plugin ชื่อแบบมี scope เช่น `@example-org/example-plugin` ต้องถูก
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้ namespace โดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณจัดการเจ้าของปัจจุบันไม่ได้ หรือเมื่อทีมงานต้องเข้ามาแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่ละเอียดอ่อน หลักฐานที่มีประโยชน์ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือ maintainer
- เอกสารทางการของโครงการที่ระบุชื่อ namespace
- หลักฐานโดเมนหรือโดเมนอีเมลทางการ
- การควบคุม scope บน npm, PyPI, crates.io หรือ registry แพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโครงการที่ปลอดภัยต่อการพูดคุย
  ในที่สาธารณะ
- ประวัติ source repository, ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อแบบสาธารณะ
- ลิงก์ไปยังเจ้าของ ClawHub, skill, Plugin, แพ็กเกจ หรือ issue ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร ทีมงานควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรรวมไว้

ห้ามใส่ความลับหรือหลักฐานส่วนตัวใน issue สาธารณะบน GitHub ห้ามรวม:

- โทเค็น API, signing key หรือข้อมูลรับรอง
- โทเค็น DNS challenge
- ไฟล์กฎหมายหรือสัญญาส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

ฟอร์มการอ้างสิทธิ์จะถามว่าหลักฐานละเอียดอ่อนจำเป็นต้องใช้ช่องทางส่วนตัวกับทีมงานหรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์ข้อมูลละเอียดอ่อนต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง ทีมงาน ClawHub อาจสงวน namespace,
โอนความเป็นเจ้าของ, เปลี่ยนชื่อทรัพยากร, ซ่อนหรือกักกันรายการที่มีอยู่,
เพิ่ม alias หรือ redirect, ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบ namespace ไม่รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอน
ทีมงานจะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
