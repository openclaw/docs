# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลเจ้าของ, แฮนเดิลองค์กร, slug ของ Skills, ชื่อแพ็กเกจ Plugin และ
ขอบเขตแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์จริง, แบรนด์, ระบบนิเวศแพ็กเกจ หรือองค์กร แต่ถูก
อ้างสิทธิ์ไว้แล้ว, สงวนไว้, ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub ให้ขอให้เจ้าหน้าที่ตรวจสอบ
ด้วย
[แบบฟอร์ม issue สำหรับการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้เส้นทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะและไม่ละเอียดอ่อน อย่าใช้รายงาน
ในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## ควรเปิดการอ้างสิทธิ์เมื่อใด

เปิดการอ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
เนมสเปซควรถูกสงวนไว้, โอนย้าย, เปลี่ยนชื่อ, ซ่อน, กักกัน, ทำเป็นนามแฝง
หรือเปลี่ยนแปลงในลักษณะอื่นเพราะความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- แฮนเดิลองค์กรที่ตรงกับ GitHub org, โปรเจกต์, บริษัท หรือชุมชนของคุณ
- ขอบเขตแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ภายใต้เจ้าของ
  ClawHub ที่ตรงกันเท่านั้น
- slug ของ Skills หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์, เครื่องหมายการค้า, การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ, ไม่ใช้งาน หรือไม่สามารถติดต่อได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่ชอบธรรม

หากรายการนั้นไม่ปลอดภัย, เป็นอันตราย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทด้านความเป็นเจ้าของ
ให้ปฏิบัติตามคำแนะนำด้านการดูแลเนื้อหาหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มการอ้างสิทธิ์
เนมสเปซมีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อที่มีขอบเขต เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณสามารถจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่,
เปลี่ยนชื่อ, โอนย้าย, ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้การอ้างสิทธิ์
เมื่อคุณไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อเจ้าหน้าที่ต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรรวมไว้

ใช้หลักฐานสาธารณะที่ไม่ละเอียดอ่อน หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติ GitHub org, repo, release หรือผู้ดูแล
- เอกสารโปรเจกต์อย่างเป็นทางการที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุมขอบเขตบน npm, PyPI, crates.io หรือรีจิสทรีแพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า, แบรนด์ หรือโปรเจกต์ที่ปลอดภัยต่อการพูดคุย
  ต่อสาธารณะ
- ประวัติที่เก็บซอร์ส, ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ, Skills, Plugin, แพ็กเกจ หรือ issue ของ ClawHub ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลประจำตัวส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรรวมไว้

อย่าใส่ความลับหรือหลักฐานส่วนตัวใน GitHub issue สาธารณะ อย่ารวม:

- โทเค็น API, กุญแจลงนาม หรือข้อมูลประจำตัว
- โทเค็น DNS challenge
- ไฟล์หรือสัญญาทางกฎหมายส่วนตัว
- เอกสารยืนยันตัวตนส่วนบุคคล
- อีเมลส่วนตัว, รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มการอ้างสิทธิ์จะถามว่าหลักฐานละเอียดอ่อนจำเป็นต้องใช้ช่องทางส่วนตัวกับเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์ข้อมูลละเอียดอ่อนต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ,
โอนความเป็นเจ้าของ, เปลี่ยนชื่อทรัพยากร, ซ่อนหรือกักกันรายการที่มีอยู่,
เพิ่มนามแฝงหรือการเปลี่ยนเส้นทาง, ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอนย้าย
เจ้าหน้าที่จะพิจารณาหลักฐานสาธารณะ, การใช้งานที่มีอยู่, ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
