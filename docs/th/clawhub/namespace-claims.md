# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้ handle ของเจ้าของ, handle ขององค์กร, slug ของ Skills, ชื่อแพ็กเกจ Plugin และ
scope ของแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูก
อ้างสิทธิ์ สงวนไว้ ทำให้เข้าใจผิด หรือมีข้อพิพาทบน ClawHub แล้ว ให้ขอให้เจ้าหน้าที่ตรวจสอบ
โดยใช้
[แบบฟอร์มปัญหาการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะที่ไม่ละเอียดอ่อน ห้ามใช้รายงาน
ภายในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## ควรเปิดคำร้องเมื่อใด

เปิดคำร้องอ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
เนมสเปซควรถูกสงวน โอน เปลี่ยนชื่อ ซ่อน กักกัน ทำ alias
หรือเปลี่ยนแปลงในรูปแบบอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- handle ขององค์กรที่ตรงกับองค์กร GitHub, โปรเจกต์, บริษัท หรือชุมชนของคุณ
- scope ของแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ได้เฉพาะภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- slug ของ Skills หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อพิพาทเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่มีสิทธิ์โดยชอบ

หากรายการนั้นไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดเกินกว่าข้อพิพาทด้านความเป็นเจ้าของ
ให้ทำตามคำแนะนำด้านการควบคุมดูแลหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มอ้างสิทธิ์เนมสเปซ
มีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่นคำร้อง

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อที่มี scope เช่น `@example-org/example-plugin` ต้อง
เผยแพร่เป็นเจ้าของ `example-org` ที่ตรงกัน

หากคุณจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้คำร้องอ้างสิทธิ์
เมื่อคุณไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อเจ้าหน้าที่จำเป็นต้องแก้ไข
ข้อพิพาท

## หลักฐานที่ควรใส่

ใช้หลักฐานสาธารณะที่ไม่ละเอียดอ่อน หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือ maintainer
- เอกสารโปรเจกต์อย่างเป็นทางการที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลอย่างเป็นทางการ
- การควบคุม scope บน npm, PyPI, crates.io หรือ registry แพ็กเกจอื่น
- หลักฐานเครื่องหมายการค้า แบรนด์ หรือความเป็นเจ้าของโปรเจกต์ที่ปลอดภัยต่อการอภิปราย
  แบบสาธารณะ
- ประวัติ repository ต้นทาง, ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ ClawHub, Skills, Plugin, แพ็กเกจ หรือ issue ที่มีข้อพิพาท

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลรับรองส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรใส่

อย่าใส่ความลับหรือหลักฐานส่วนตัวใน issue สาธารณะบน GitHub ห้ามใส่:

- โทเค็น API, คีย์สำหรับลงนาม หรือข้อมูลรับรอง
- โทเค็น DNS challenge
- ไฟล์หรือสัญญาทางกฎหมายส่วนตัว
- เอกสารระบุตัวตนส่วนบุคคล
- อีเมลส่วนตัว, รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มคำร้องจะถามว่าหลักฐานละเอียดอ่อนจำเป็นต้องใช้ช่องทางส่วนตัวกับเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์เนื้อหาละเอียดอ่อนแบบสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่ม alias หรือ redirect ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าทุกชื่อที่ตรงกันจะถูกโอน
เจ้าหน้าที่จะพิจารณาหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
