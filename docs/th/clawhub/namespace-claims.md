# การอ้างสิทธิ์องค์กรและเนมสเปซ

ClawHub ใช้แฮนเดิลเจ้าของ แฮนเดิลองค์กร slug ของ skill ชื่อแพ็กเกจ Plugin และ
ขอบเขตแพ็กเกจเป็นเนมสเปซสาธารณะ หากเนมสเปซดูเหมือนเป็นของ
โปรเจกต์ แบรนด์ ระบบนิเวศแพ็กเกจ หรือองค์กรในโลกจริง แต่ถูกอ้างสิทธิ์ไว้แล้ว
ถูกสงวนไว้ ทำให้เข้าใจผิด หรือมีข้อโต้แย้งบน ClawHub ให้ขอให้เจ้าหน้าที่ตรวจสอบ
ด้วย
[แบบฟอร์มปัญหาการอ้างสิทธิ์องค์กร / เนมสเปซ](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)

ใช้ช่องทางนี้สำหรับการตรวจสอบความเป็นเจ้าของแบบสาธารณะและไม่อ่อนไหว อย่าใช้รายงาน
ในผลิตภัณฑ์หรือแบบฟอร์มอุทธรณ์บัญชีสำหรับการอ้างสิทธิ์เนมสเปซ

## เมื่อใดควรเปิดคำขออ้างสิทธิ์

เปิดคำขออ้างสิทธิ์เนมสเปซเมื่อคุณเชื่อว่าเจ้าหน้าที่ ClawHub ควรตรวจสอบว่า
เนมสเปซควรถูกสงวนไว้ โอน เปลี่ยนชื่อ ซ่อน กักกัน ตั้งนามแฝง
หรือเปลี่ยนแปลงด้วยวิธีอื่นเนื่องจากความเป็นเจ้าของในโลกจริงหรือไม่

ตัวอย่างได้แก่:

- แฮนเดิลองค์กรที่ตรงกับองค์กร GitHub, โปรเจกต์, บริษัท หรือชุมชนของคุณ
- ขอบเขตแพ็กเกจ เช่น `@example-org/*` ที่ควรเผยแพร่ได้เฉพาะภายใต้
  เจ้าของ ClawHub ที่ตรงกันเท่านั้น
- slug ของ skill หรือชื่อแพ็กเกจ Plugin ที่ดูเหมือนแอบอ้างเป็นโปรเจกต์
- ข้อโต้แย้งเกี่ยวกับแบรนด์ เครื่องหมายการค้า การเปลี่ยนชื่อโปรเจกต์ หรือประวัติแพ็กเกจ
- เจ้าของที่ถูกลบ ไม่ได้ใช้งาน หรือติดต่อไม่ได้ ซึ่งขัดขวางเจ้าของเนมสเปซ
  ที่ชอบธรรม

หากรายการไม่ปลอดภัย เป็นอันตราย หรือทำให้เข้าใจผิดมากกว่าข้อโต้แย้งเรื่องความเป็นเจ้าของ
ให้ปฏิบัติตามคำแนะนำด้านการดูแลเนื้อหาหรือความปลอดภัยที่เกี่ยวข้องด้วย แบบฟอร์มอ้างสิทธิ์
เนมสเปซมีไว้สำหรับการตรวจสอบความเป็นเจ้าของ ไม่ใช่การเปิดเผยช่องโหว่ฉุกเฉิน

## ก่อนที่คุณจะยื่น

ก่อนอื่นให้ยืนยันว่าคุณกำลังเผยแพร่ด้วยเจ้าของที่ตรงกับเนมสเปซ
สำหรับแพ็กเกจ Plugin ชื่อแบบมีขอบเขต เช่น `@example-org/example-plugin` ต้อง
เผยแพร่ในฐานะเจ้าของ `example-org` ที่ตรงกัน

หากคุณสามารถจัดการเจ้าของปัจจุบันได้ ให้แก้ไขเนมสเปซโดยตรงด้วยการเผยแพร่
เปลี่ยนชื่อ โอน ซ่อน หรือลบทรัพยากรที่ได้รับผลกระทบ ใช้คำขออ้างสิทธิ์
เมื่อคุณไม่สามารถจัดการเจ้าของปัจจุบันได้ หรือเมื่อเจ้าหน้าที่ต้องแก้ไข
ข้อโต้แย้ง

## หลักฐานที่ควรระบุ

ใช้หลักฐานสาธารณะที่ไม่อ่อนไหว หลักฐานที่เป็นประโยชน์ได้แก่:

- ประวัติองค์กร GitHub, repo, release หรือ maintainer
- เอกสารทางการของโปรเจกต์ที่ระบุชื่อเนมสเปซ
- หลักฐานโดเมนหรือโดเมนอีเมลทางการ
- การควบคุมขอบเขต npm, PyPI, crates.io หรือรีจิสทรีแพ็กเกจอื่น
- หลักฐานความเป็นเจ้าของเครื่องหมายการค้า แบรนด์ หรือโปรเจกต์ที่สามารถพูดคุย
  ต่อสาธารณะได้อย่างปลอดภัย
- ประวัติซอร์ส repository, ประวัติแพ็กเกจ หรือประกาศเปลี่ยนชื่อสาธารณะ
- ลิงก์ไปยังเจ้าของ ClawHub, skill, Plugin, แพ็กเกจ หรือ issue ที่มีข้อโต้แย้ง

อธิบายว่าแต่ละลิงก์พิสูจน์อะไร เจ้าหน้าที่ควรเข้าใจ
ความสัมพันธ์ได้โดยไม่ต้องใช้ข้อมูลประจำตัวส่วนตัวหรือความลับ

## สิ่งที่ไม่ควรระบุ

อย่าใส่ความลับหรือหลักฐานส่วนตัวใน issue สาธารณะของ GitHub อย่ารวม:

- โทเค็น API, signing keys หรือข้อมูลประจำตัว
- โทเค็น DNS challenge
- ไฟล์หรือสัญญาทางกฎหมายส่วนตัว
- เอกสารยืนยันตัวตนส่วนบุคคล
- อีเมลส่วนตัว รายงานความปลอดภัยส่วนตัว หรือข้อมูลลูกค้าที่เป็นความลับ

แบบฟอร์มอ้างสิทธิ์จะถามว่าหลักฐานอ่อนไหวต้องใช้ช่องทางส่วนตัวกับเจ้าหน้าที่หรือไม่
ให้ใช้ตัวเลือกนั้นแทนการโพสต์เนื้อหาอ่อนไหวต่อสาธารณะ

## ผลลัพธ์ที่เป็นไปได้

ขึ้นอยู่กับหลักฐานและความเสี่ยง เจ้าหน้าที่ ClawHub อาจสงวนเนมสเปซ
โอนความเป็นเจ้าของ เปลี่ยนชื่อทรัพยากร ซ่อนหรือกักกันรายการที่มีอยู่
เพิ่มนามแฝงหรือการเปลี่ยนเส้นทาง ขอหลักฐานเพิ่มเติม หรือปฏิเสธคำขอ

การตรวจสอบเนมสเปซไม่ได้รับประกันว่าชื่อที่ตรงกันทุกชื่อจะถูกโอน
เจ้าหน้าที่จะชั่งน้ำหนักหลักฐานสาธารณะ การใช้งานที่มีอยู่ ความเสี่ยงด้านความปลอดภัย และผลกระทบต่อผู้ใช้
