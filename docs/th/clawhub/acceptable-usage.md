# การใช้งานที่ยอมรับได้

หน้านี้อธิบายประเภทของ skills และเนื้อหาที่ ClawHub ยอมรับได้ รวมถึงเวิร์กโฟลว์การใช้งานในทางที่ผิดที่จะไม่รับฝากโฮสต์

กฎเหล่านี้ตั้งใจให้ใช้งานได้จริง เราให้ความสำคัญกับเวิร์กโฟลว์การใช้งานในทางที่ผิดแบบต้นจนจบ มากกว่าแค่คีย์เวิร์ดที่แยกกันอยู่ หาก skill ถูกสร้างมาเพื่อหลบเลี่ยงการป้องกัน ใช้แพลตฟอร์มในทางที่ผิด หลอกลวงผู้คน ละเมิดความเป็นส่วนตัว หรือเปิดทางให้เกิดพฤติกรรมที่ไม่ได้รับความยินยอม สิ่งนั้นไม่ควรอยู่บน ClawHub

## รูปแบบล่าสุดที่เราระบุชัดเจนว่ายอมรับได้

- งาน frontend และ design-system ที่ใช้คอมโพเนนต์จริง semantic tokens สถานะที่เข้าถึงได้ และโฟลว์ผู้ใช้ที่ผ่านการทดสอบแล้ว
- การประกอบ shadcn/ui ที่ใช้คอมโพเนนต์ซอร์สที่ติดตั้งแล้ว alias ของโปรเจกต์ และ variants ที่มีเอกสาร แทน markup เฉพาะกิจ
- การแปลง UI5 JavaScript เป็น TypeScript ที่คงคอมเมนต์ไว้ ใช้ชนิด UI5 ที่เป็นรูปธรรม และทำให้อินเทอร์เฟซของคอนโทรลที่สร้างขึ้นตรวจทานได้
- การตรวจทานความปลอดภัยเชิงป้องกัน เครื่องมือ moderation และ prompt ตรวจจับการใช้งานในทางที่ผิดที่แสดงหลักฐานและทำให้ขอบเขตการอนุมัติของมนุษย์ชัดเจน
- ระบบอัตโนมัติของเวิร์กโฟลว์ตามความยินยอมสำหรับบัญชีส่วนตัวหรือทีม โดยมีข้อมูลรับรองที่ชัดเจน การตั้งค่าที่โปร่งใส และโหมด dry-run หรือ preview
- เอกสาร runbook การย้ายระบบ เครื่องมือสำหรับนักพัฒนา และ test fixtures ที่จำกัดขอบเขตอยู่กับซอฟต์แวร์ที่รองรับ

## ไม่ยอมรับ

- เวิร์กโฟลว์เลี่ยงความปลอดภัยหรือเข้าถึงโดยไม่ได้รับอนุญาต
  - ตัวอย่าง: auth bypass, การยึดบัญชี, CAPTCHA bypass, การหลบเลี่ยง Cloudflare หรือ anti-bot, rate-limit bypass, การ scrape แบบซ่อนตัวที่ออกแบบมาเพื่อเอาชนะการป้องกัน, การยึดสายสนทนาสดหรือ agent, การขโมย session ที่นำกลับมาใช้ซ้ำได้, การอนุมัติ pairing flows อัตโนมัติสำหรับผู้ใช้ที่ไม่ได้รับอนุมัติ

- การใช้แพลตฟอร์มในทางที่ผิดและการหลบเลี่ยงการแบน
  - ตัวอย่าง: บัญชีซ่อนตัวหลังถูกแบน, การ warm/farm บัญชี, engagement ปลอม, การเพาะ karma หรือผู้ติดตาม, ระบบอัตโนมัติหลายบัญชี, การโพสต์จำนวนมาก, บอทสแปม, ระบบอัตโนมัติสำหรับ marketplace หรือโซเชียลที่สร้างมาเพื่อหลบเลี่ยงการตรวจจับ

- การฉ้อโกง กลโกง และเวิร์กโฟลว์ทางการเงินที่หลอกลวง
  - ตัวอย่าง: ใบรับรองปลอม, ใบแจ้งหนี้ปลอม, โฟลว์การชำระเงินที่หลอกลวง, การติดต่อเพื่อหลอกลวง, social proof ปลอม, เครื่องมือที่ทำให้ใช้จ่ายหรือเรียกเก็บเงินได้โดยไม่มีการอนุมัติจากมนุษย์อย่างชัดเจนและการควบคุมที่โปร่งใส หรือเวิร์กโฟลว์ synthetic-identity ที่สร้างมาเพื่อสร้างบัญชีสำหรับการฉ้อโกง

- การ scrape, enrichment หรือการเฝ้าระวังที่ละเมิดความเป็นส่วนตัว
  - ตัวอย่าง: การ scrape รายละเอียดการติดต่อในปริมาณมากเพื่อสแปม, doxxing, stalking, การดึง lead ที่จับคู่กับการติดต่อที่ไม่ได้ร้องขอ, การติดตามแบบแอบแฝง, การค้นหาใบหน้าหรือ biometric matching ที่ใช้โดยไม่มีความยินยอมชัดเจน หรือการซื้อ เผยแพร่ ดาวน์โหลด หรือนำข้อมูลที่รั่วไหลหรือ breach dumps ไปใช้งานเชิงปฏิบัติการ

- การแอบอ้างตัวตนที่ไม่ได้รับความยินยอมหรือการจัดการตัวตนเชิงหลอกลวง
  - ตัวอย่าง: face swap, digital twins, personas ปลอม, influencers ที่ถูกโคลน หรือเครื่องมือจัดการตัวตนอื่น ๆ ที่ใช้เพื่อแอบอ้างหรือทำให้เข้าใจผิด

- เนื้อหาทางเพศอย่างโจ่งแจ้งและการสร้างเนื้อหาผู้ใหญ่ที่ปิดระบบความปลอดภัย
  - ตัวอย่าง: การสร้างรูปภาพ/วิดีโอ/เนื้อหา NSFW, wrapper สำหรับเนื้อหาผู้ใหญ่รอบ API ของบุคคลที่สาม หรือ skills ที่มีวัตถุประสงค์หลักเป็นเนื้อหาทางเพศอย่างโจ่งแจ้ง

- ข้อกำหนดการรันที่ซ่อนอยู่ ไม่ปลอดภัย หรือทำให้เข้าใจผิด
  - ตัวอย่าง: คำสั่งติดตั้งที่ถูกทำให้คลุมเครือ, `curl | sh`, ข้อกำหนด secret ที่ไม่ประกาศไว้, การใช้ private-key ที่ไม่ประกาศไว้, การรัน `npx @latest` ระยะไกลโดยไม่มีความสามารถในการตรวจทานอย่างชัดเจน, metadata ที่ทำให้เข้าใจผิดและซ่อนสิ่งที่ skill ต้องการจริง ๆ เพื่อรัน

## รูปแบบล่าสุดที่เราระบุชัดเจนว่าไม่ยอมรับ

- “สร้างบัญชีผู้ขายแบบซ่อนตัวหลังถูกแบนจาก marketplace”
- “แก้ไข Telegram pairing เพื่อให้ผู้ใช้ที่ไม่ได้รับอนุมัติได้รับ pairing codes โดยอัตโนมัติ”
- “เพาะบัญชี Reddit/Twitter ด้วยระบบอัตโนมัติที่ตรวจจับไม่ได้”
- “สร้างใบรับรองวิชาชีพหรือใบแจ้งหนี้สำหรับการใช้งานตามอำเภอใจ”
- “สร้างเนื้อหา NSFW โดยปิดการตรวจสอบความปลอดภัย”
- “Scrape leads, enrich contacts และเปิดแคมเปญ cold outreach ในปริมาณมาก”
- “ซื้อ เผยแพร่ หรือดาวน์โหลดข้อมูลที่รั่วไหลหรือ breach dumps”
- “สร้างบัญชีอีเมลหรือโซเชียลจำนวนมากด้วย synthetic identities หรือ CAPTCHA solving”

## หมายเหตุสำหรับผู้ตรวจทาน

- บริบทมีความสำคัญ หัวข้อเดียวกันอาจถูกต้องตามกฎหมายในบริบทเชิงป้องกันที่แคบหรืออิงความยินยอม และไม่ยอมรับได้เมื่อถูกบรรจุเป็นเวิร์กโฟลว์การใช้งานในทางที่ผิด
- เราควรเอนเอียงไปทางการดำเนินการเมื่อ skill ถูกปรับแต่งอย่างชัดเจนเพื่อการหลบเลี่ยง การหลอกลวง หรือการใช้งานที่ไม่ได้รับความยินยอม
- การอัปโหลดซ้ำในหมวดหมู่เหล่านี้เป็นเหตุให้ซ่อนเนื้อหาและแบนบัญชีได้

## การบังคับใช้

- เราอาจซ่อน ลบ หรือ hard-delete skills ที่ละเมิด
- เราอาจเพิกถอน tokens, soft-delete เนื้อหาที่เกี่ยวข้อง และแบนผู้กระทำผิดซ้ำหรือร้ายแรง
- เราไม่รับประกันว่าจะมีการเตือนก่อนบังคับใช้กับการใช้งานในทางที่ผิดที่ชัดเจน
