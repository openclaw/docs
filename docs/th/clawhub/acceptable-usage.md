# การใช้งานที่ยอมรับได้

หน้านี้อธิบายประเภทของทักษะและเนื้อหาที่ ClawHub ยอมรับได้ และเวิร์กโฟลว์การละเมิดที่ ClawHub จะไม่โฮสต์

กฎเหล่านี้ตั้งใจให้ใช้งานได้จริง เราให้ความสำคัญมากที่สุดกับเวิร์กโฟลว์การละเมิดแบบครบวงจร ไม่ใช่แค่คำหลักที่แยกเป็นส่วนๆ หากทักษะถูกสร้างขึ้นเพื่อหลบเลี่ยงการป้องกัน ละเมิดแพลตฟอร์ม หลอกลวงผู้คน ละเมิดความเป็นส่วนตัว หรือเปิดทางให้เกิดพฤติกรรมที่ไม่ได้รับความยินยอม ทักษะนั้นไม่ควรอยู่บน ClawHub

## รูปแบบล่าสุดที่เราระบุชัดเจนว่ายอมรับได้

- งานฟรอนต์เอนด์และระบบดีไซน์ที่ใช้คอมโพเนนต์จริง โทเคนเชิงความหมาย สถานะที่เข้าถึงได้ และโฟลว์ผู้ใช้ที่ผ่านการทดสอบ
- การประกอบ shadcn/ui ที่ใช้คอมโพเนนต์ต้นทางที่ติดตั้งแล้ว alias ของโปรเจกต์ และ variant ที่มีเอกสารกำกับ แทนมาร์กอัปเฉพาะกิจ
- การแปลง UI5 JavaScript เป็น TypeScript ที่คงความคิดเห็นไว้ ใช้ชนิด UI5 ที่เป็นรูปธรรม และทำให้อินเทอร์เฟซคอนโทรลที่สร้างขึ้นตรวจทานได้
- การทบทวนความปลอดภัยเชิงป้องกัน เครื่องมือมอดเดอเรชัน และพรอมป์ตรวจจับการละเมิดที่แสดงหลักฐานและรักษาขอบเขตการอนุมัติของมนุษย์ให้ชัดเจน
- การทำเวิร์กโฟลว์อัตโนมัติบนพื้นฐานความยินยอมสำหรับบัญชีส่วนบุคคลหรือทีม โดยมีข้อมูลประจำตัวที่ชัดเจน การตั้งค่าที่โปร่งใส และโหมด dry-run หรือ preview
- เอกสาร คู่มือการย้ายระบบสำหรับปฏิบัติงานจริง ยูทิลิตีสำหรับนักพัฒนา และ fixture สำหรับทดสอบที่จำกัดขอบเขตตามซอฟต์แวร์ที่รองรับ

## ไม่ยอมรับ

- เวิร์กโฟลว์ที่บายพาสความปลอดภัยหรือเข้าถึงโดยไม่ได้รับอนุญาต
  - ตัวอย่าง: การบายพาสการยืนยันตัวตน การยึดบัญชี การบายพาส CAPTCHA การหลบเลี่ยง Cloudflare หรือระบบป้องกันบอต การบายพาสขีดจำกัดอัตรา การขูดข้อมูลแบบล่องหนที่ออกแบบมาเพื่อเอาชนะการป้องกัน การยึดสายสนทนาสดหรือ agent การขโมยเซสชันที่นำกลับมาใช้ซ้ำได้ การอนุมัติโฟลว์การจับคู่อัตโนมัติสำหรับผู้ใช้ที่ไม่ได้รับอนุมัติ

- การละเมิดแพลตฟอร์มและการหลบเลี่ยงการแบน
  - ตัวอย่าง: บัญชีล่องหนหลังถูกแบน การวอร์มหรือฟาร์มบัญชี การสร้างการมีส่วนร่วมปลอม การปั้น karma หรือผู้ติดตาม การทำงานอัตโนมัติหลายบัญชี การโพสต์จำนวนมาก บอตสแปม การทำงานอัตโนมัติบนมาร์เก็ตเพลซหรือโซเชียลที่สร้างขึ้นเพื่อหลบเลี่ยงการตรวจจับ

- การฉ้อโกง การหลอกลวง และเวิร์กโฟลว์ทางการเงินที่หลอกลวง
  - ตัวอย่าง: ใบรับรองปลอม ใบแจ้งหนี้ปลอม โฟลว์การชำระเงินที่หลอกลวง การติดต่อเพื่อหลอกลวง หลักฐานทางสังคมปลอม เครื่องมือที่เปิดทางให้ใช้จ่ายหรือเรียกเก็บเงินโดยไม่มีการอนุมัติจากมนุษย์อย่างชัดเจนและการควบคุมที่โปร่งใส หรือเวิร์กโฟลว์ตัวตนสังเคราะห์ที่สร้างขึ้นเพื่อสร้างบัญชีสำหรับการฉ้อโกง

- การขูดข้อมูล การเพิ่มคุณค่าข้อมูล หรือการเฝ้าระวังที่ละเมิดความเป็นส่วนตัว
  - ตัวอย่าง: การขูดรายละเอียดติดต่อในวงกว้างเพื่อสแปม การ doxxing การสะกดรอย การดึง lead ควบคู่กับการติดต่อที่ไม่ได้รับเชิญ การเฝ้าติดตามแบบปกปิด การค้นหาใบหน้าหรือการจับคู่ชีวมิติที่ใช้โดยไม่มีความยินยอมอย่างชัดเจน หรือการซื้อ เผยแพร่ ดาวน์โหลด หรือทำให้ข้อมูลรั่วไหลหรือชุดข้อมูลจากการละเมิดพร้อมใช้งานในการปฏิบัติการ

- การสวมรอยโดยไม่ได้รับความยินยอมหรือการบิดเบือนตัวตนที่หลอกลวง
  - ตัวอย่าง: face swap, digital twins, persona ปลอม, influencer ที่ถูกโคลน หรือเครื่องมือบิดเบือนตัวตนอื่นๆ ที่ใช้เพื่อสวมรอยหรือทำให้เข้าใจผิด

- เนื้อหาทางเพศอย่างชัดเจนและการสร้างเนื้อหาผู้ใหญ่โดยปิดความปลอดภัย
  - ตัวอย่าง: การสร้างรูปภาพ/วิดีโอ/เนื้อหา NSFW, wrapper เนื้อหาผู้ใหญ่รอบ API ของบุคคลที่สาม หรือทักษะที่มีวัตถุประสงค์หลักเป็นเนื้อหาทางเพศอย่างชัดเจน

- ข้อกำหนดการรันที่ซ่อนอยู่ ไม่ปลอดภัย หรือทำให้เข้าใจผิด
  - ตัวอย่าง: คำสั่งติดตั้งที่ทำให้อ่านยาก, `curl | sh`, ข้อกำหนด secret ที่ไม่ได้ประกาศ, การใช้ private key ที่ไม่ได้ประกาศ, การรัน `npx @latest` ระยะไกลโดยไม่มีความสามารถในการตรวจทานที่ชัดเจน, metadata ที่ทำให้เข้าใจผิดซึ่งซ่อนสิ่งที่ทักษะต้องใช้จริงในการรัน

## รูปแบบล่าสุดที่เราระบุชัดเจนว่าไม่ยอมรับ

- “สร้างบัญชีผู้ขายแบบล่องหนหลังถูกแบนจากมาร์เก็ตเพลซ”
- “แก้ไขการจับคู่ Telegram เพื่อให้ผู้ใช้ที่ไม่ได้รับอนุมัติได้รับรหัสจับคู่อัตโนมัติ”
- “ปั้นบัญชี Reddit/Twitter ด้วยการทำงานอัตโนมัติที่ตรวจจับไม่ได้”
- “สร้างใบรับรองวิชาชีพหรือใบแจ้งหนี้สำหรับการใช้งานตามอำเภอใจ”
- “สร้างเนื้อหา NSFW โดยปิดการตรวจสอบความปลอดภัย”
- “ขูด lead เพิ่มคุณค่าข้อมูลติดต่อ และเริ่มการติดต่อแบบ cold outreach ในวงกว้าง”
- “ซื้อ เผยแพร่ หรือดาวน์โหลดข้อมูลรั่วไหลหรือชุดข้อมูลจากการละเมิด”
- “สร้างบัญชีอีเมลหรือโซเชียลจำนวนมากด้วยตัวตนสังเคราะห์หรือการแก้ CAPTCHA”

## หมายเหตุสำหรับผู้ตรวจทาน

- บริบทมีความสำคัญ หัวข้อเดียวกันอาจชอบธรรมได้ในบริบทเชิงป้องกันที่แคบหรือมีพื้นฐานจากความยินยอม และไม่อาจยอมรับได้เมื่อถูกบรรจุเป็นเวิร์กโฟลว์การละเมิด
- เราควรเอนเอียงไปทางการลงมือเมื่อทักษะถูกปรับให้เหมาะกับการหลบเลี่ยง การหลอกลวง หรือการใช้งานที่ไม่ได้รับความยินยอมอย่างชัดเจน
- การอัปโหลดซ้ำในหมวดหมู่เหล่านี้เป็นเหตุให้ซ่อนเนื้อหาและแบนบัญชีได้

## การบังคับใช้

- เราอาจซ่อน ลบ หรือ hard-delete ทักษะที่ละเมิด
- เราอาจเพิกถอนโทเคน soft-delete เนื้อหาที่เกี่ยวข้อง และแบนผู้กระทำผิดซ้ำหรือร้ายแรง
- เราไม่รับประกันการบังคับใช้แบบเตือนก่อนสำหรับการละเมิดที่ชัดเจน
