# การใช้งานที่ยอมรับได้

หน้านี้อธิบายประเภทของ Skills และเนื้อหาที่ ClawHub ยอมรับได้ รวมถึงเวิร์กโฟลว์การละเมิดที่ ClawHub จะไม่โฮสต์

กฎเหล่านี้ตั้งใจให้ใช้งานได้จริง เราให้ความสำคัญกับเวิร์กโฟลว์การละเมิดแบบครบวงจรมากที่สุด ไม่ใช่แค่คีย์เวิร์ดที่แยกกันอยู่ หาก Skills ถูกสร้างขึ้นเพื่อหลบเลี่ยงการป้องกัน ใช้งานแพลตฟอร์มในทางที่ผิด หลอกลวงผู้คน ละเมิดความเป็นส่วนตัว หรือเอื้อให้เกิดพฤติกรรมที่ไม่ได้รับความยินยอม Skills นั้นไม่เหมาะกับ ClawHub

## รูปแบบล่าสุดที่เรายอมรับอย่างชัดเจน

- งาน frontend และ design-system ที่ใช้คอมโพเนนต์จริง semantic tokens สถานะที่เข้าถึงได้ และโฟลว์ผู้ใช้ที่ผ่านการทดสอบแล้ว
- การประกอบ shadcn/ui ที่ใช้คอมโพเนนต์ซอร์สที่ติดตั้งแล้ว alias ของโปรเจกต์ และ variants ที่มีเอกสาร แทน markup แบบเฉพาะกิจ
- การแปลง UI5 JavaScript-to-TypeScript ที่คงคอมเมนต์ไว้ ใช้ชนิด UI5 ที่เป็นรูปธรรม และทำให้อินเทอร์เฟซของคอนโทรลที่สร้างขึ้นตรวจทานได้
- การตรวจทานความปลอดภัยเชิงป้องกัน เครื่องมือ moderation และพรอมป์ตรวจจับการละเมิดที่แสดงหลักฐานและรักษาขอบเขตการอนุมัติโดยมนุษย์ให้ชัดเจน
- ระบบอัตโนมัติของเวิร์กโฟลว์ที่อิงความยินยอมสำหรับบัญชีส่วนตัวหรือบัญชีทีม พร้อมข้อมูลรับรองที่ชัดเจน การตั้งค่าที่โปร่งใส และโหมด dry-run หรือ preview
- เอกสาร runbook สำหรับการย้ายระบบ ยูทิลิตีสำหรับนักพัฒนา และ test fixtures ที่กำหนดขอบเขตตามซอฟต์แวร์ที่รองรับ

## ไม่ยอมรับ

- เวิร์กโฟลว์การข้ามความปลอดภัยหรือการเข้าถึงโดยไม่ได้รับอนุญาต
  - ตัวอย่าง: auth bypass, account takeover, CAPTCHA bypass, การหลบเลี่ยง Cloudflare หรือระบบป้องกันบอต, rate-limit bypass, การ scraping แบบแอบแฝงที่ออกแบบมาเพื่อเอาชนะการป้องกัน, การยึดสายสดหรือยึด agent, การขโมยเซสชันที่นำกลับมาใช้ซ้ำได้, การอนุมัติโฟลว์ pairing อัตโนมัติสำหรับผู้ใช้ที่ไม่ได้รับอนุมัติ

- การใช้แพลตฟอร์มในทางที่ผิดและการหลบเลี่ยงการแบน
  - ตัวอย่าง: บัญชีล่องหนหลังถูกแบน, การ warming/farming บัญชี, engagement ปลอม, การเพาะ karma หรือผู้ติดตาม, ระบบอัตโนมัติหลายบัญชี, การโพสต์จำนวนมาก, บอตสแปม, ระบบอัตโนมัติสำหรับ marketplace หรือ social ที่สร้างขึ้นเพื่อหลบการตรวจจับ

- การฉ้อโกง กลโกง และเวิร์กโฟลว์การเงินที่หลอกลวง
  - ตัวอย่าง: ใบรับรองปลอม, ใบแจ้งหนี้ปลอม, โฟลว์การชำระเงินที่หลอกลวง, การติดต่อเพื่อหลอกลวง, social proof ปลอม, เครื่องมือที่เอื้อให้เกิดการใช้จ่ายหรือเรียกเก็บเงินโดยไม่มีการอนุมัติจากมนุษย์ที่ชัดเจนและการควบคุมที่โปร่งใส, หรือเวิร์กโฟลว์ synthetic-identity ที่สร้างขึ้นเพื่อสร้างบัญชีสำหรับการฉ้อโกง

- การ scraping, enrichment หรือ surveillance ที่ละเมิดความเป็นส่วนตัว
  - ตัวอย่าง: การ scraping รายละเอียดติดต่อในปริมาณมากเพื่อสแปม, doxxing, stalking, การดึง lead ที่จับคู่กับการติดต่อที่ไม่ได้ร้องขอ, การเฝ้าติดตามแบบแอบแฝง, การค้นหาใบหน้าหรือการจับคู่ชีวมิติที่ใช้โดยไม่มีความยินยอมที่ชัดเจน, หรือการซื้อ เผยแพร่ ดาวน์โหลด หรือนำข้อมูลรั่วไหลหรือ breach dumps ไปใช้ในการปฏิบัติการ

- การปลอมตัวหรือการจัดการอัตลักษณ์แบบหลอกลวงโดยไม่ได้รับความยินยอม
  - ตัวอย่าง: face swap, digital twins, persona ปลอม, influencer ที่ถูกโคลน, หรือเครื่องมือจัดการอัตลักษณ์อื่นๆ ที่ใช้เพื่อปลอมตัวหรือทำให้เข้าใจผิด

- เนื้อหาทางเพศอย่างชัดเจนและการสร้างเนื้อหาผู้ใหญ่ที่ปิดระบบความปลอดภัย
  - ตัวอย่าง: การสร้างรูปภาพ/วิดีโอ/เนื้อหา NSFW, wrapper เนื้อหาผู้ใหญ่รอบ APIs ของบุคคลที่สาม, หรือ Skills ที่มีวัตถุประสงค์หลักเป็นเนื้อหาทางเพศอย่างชัดเจน

- ข้อกำหนดการรันที่ซ่อนอยู่ ไม่ปลอดภัย หรือทำให้เข้าใจผิด
  - ตัวอย่าง: คำสั่งติดตั้งที่ทำให้อ่านยาก, `curl | sh`, ข้อกำหนด secret ที่ไม่ได้ประกาศ, การใช้ private-key ที่ไม่ได้ประกาศ, การรัน `npx @latest` ระยะไกลโดยไม่มีความสามารถในการตรวจทานที่ชัดเจน, metadata ที่ทำให้เข้าใจผิดและซ่อนสิ่งที่ Skills ต้องใช้จริงในการรัน

## รูปแบบล่าสุดที่เราไม่ยอมรับอย่างชัดเจน

- “สร้างบัญชีผู้ขายแบบล่องหนหลังถูกแบนจาก marketplace”
- “แก้ไข Telegram pairing เพื่อให้ผู้ใช้ที่ไม่ได้รับอนุมัติได้รับรหัส pairing โดยอัตโนมัติ”
- “เพาะบัญชี Reddit/Twitter ด้วยระบบอัตโนมัติที่ตรวจจับไม่ได้”
- “สร้างใบรับรองหรือใบแจ้งหนี้ระดับมืออาชีพเพื่อใช้งานตามอำเภอใจ”
- “สร้างเนื้อหา NSFW โดยปิดการตรวจสอบความปลอดภัย”
- “Scrape leads, enrich contacts และเริ่ม cold outreach ในปริมาณมาก”
- “ซื้อ เผยแพร่ หรือดาวน์โหลดข้อมูลรั่วไหลหรือ breach dumps”
- “สร้างบัญชีอีเมลหรือบัญชี social จำนวนมากด้วย synthetic identities หรือการแก้ CAPTCHA”

## หมายเหตุสำหรับผู้ตรวจทาน

- บริบทมีความสำคัญ หัวข้อเดียวกันอาจถูกต้องตามกฎหมายในบริบทเชิงป้องกันหรืออิงความยินยอมที่แคบ และไม่สามารถยอมรับได้เมื่อถูกแพ็กเกจเป็นเวิร์กโฟลว์การละเมิด
- เราควรเอนเอียงไปทางการดำเนินการเมื่อ Skills ถูกปรับให้เหมาะกับการหลบเลี่ยง การหลอกลวง หรือการใช้งานที่ไม่ได้รับความยินยอมอย่างชัดเจน
- การอัปโหลดซ้ำในหมวดหมู่เหล่านี้เป็นเหตุให้ซ่อนเนื้อหาและแบนบัญชีได้

## การบังคับใช้

- เราอาจซ่อน ลบ หรือ hard-delete Skills ที่ละเมิด
- เราอาจเพิกถอนโทเค็น soft-delete เนื้อหาที่เกี่ยวข้อง และแบนผู้กระทำผิดซ้ำหรือร้ายแรง
- เราไม่รับประกันการบังคับใช้แบบเตือนก่อนสำหรับการละเมิดที่ชัดเจน
