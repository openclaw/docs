**Node** คืออุปกรณ์ร่วม (macOS/iOS/watchOS/Android/แบบไม่มีส่วนติดต่อผู้ใช้) ที่เชื่อมต่อกับ Gateway ด้วย `role: "node"` และเปิดเผยพื้นผิวคำสั่ง (เช่น `canvas.*`, `camera.*`, `device.*`, `notifications.*`, `system.*`) ผ่าน `node.invoke` โดย Node ส่วนใหญ่ใช้ Gateway WebSocket บนพอร์ตของผู้ดำเนินการ ส่วน Node ของ Apple Watch แบบเชื่อมต่อโดยตรงซึ่งเป็นตัวเลือก จะใช้การสำรวจผ่าน HTTPS ที่มีลายเซ็นบนพอร์ตเดียวกัน เนื่องจาก watchOS ปิดกั้นเครือข่ายระดับต่ำทั่วไปสำหรับแอปทั่วไป รายละเอียดโปรโตคอล: [โปรโตคอล Gateway](/th/gateway/protocol)

การรับส่งแบบเดิม: [โปรโตคอล Bridge](/th/gateway/bridge-protocol) (TCP JSONL; มีไว้เป็นประวัติเท่านั้นสำหรับ Node ปัจจุบัน)

macOS ยังทำงานใน **โหมด Node** ได้ด้วย: แอปแถบเมนูจะเชื่อมต่อกับเซิร์ฟเวอร์
WS ของ Gateway ในฐานะ Node หนึ่งตัว (ดังนั้น `openclaw nodes …` จึงทำงานกับ Mac เครื่องนี้ได้) แอป
จะเพิ่มคำสั่งแบบเนทีฟสำหรับ Canvas, กล้อง, หน้าจอ, การแจ้งเตือน และการควบคุมคอมพิวเตอร์
ลงในพื้นผิวคำสั่งของโฮสต์ Node เดียวกับที่ `openclaw node run` ใช้ อย่าเริ่ม
Node แบบ CLI ตัวที่สองบน Mac เครื่องนั้น แอปจะเรียกใช้รันไทม์โฮสต์ Node แบบ CLI ที่ตรงกัน
เป็นเวิร์กเกอร์ภายใน และยังคงเป็นการเชื่อมต่อ Gateway และข้อมูลประจำตัว Node เพียงรายการเดียว

Node เป็น **อุปกรณ์ต่อพ่วง** ไม่ใช่ Gateway: Node ไม่ได้เรียกใช้บริการ Gateway และข้อความจากช่องทาง (Telegram, WhatsApp เป็นต้น) จะมาถึง Gateway ไม่ใช่ Node

คู่มือการแก้ไขปัญหา: [/nodes/troubleshooting](/th/nodes/troubleshooting)

## การจับคู่ + สถานะ

Node ใช้ **การจับคู่อุปกรณ์** ระหว่างเชื่อมต่อ Node จะแสดงข้อมูลประจำตัวอุปกรณ์ที่มีลายเซ็น จากนั้น Gateway จะสร้างคำขอจับคู่อุปกรณ์สำหรับ `role: node` อนุมัติผ่าน CLI สำหรับอุปกรณ์ (หรือ UI) การตั้งค่า Apple Watch แบบเชื่อมต่อโดยตรงใช้รหัสตั้งค่าเฉพาะ Node ที่มีอายุสั้นและผู้ดูแลระบบสร้างขึ้น เพื่ออนุมัติพื้นผิวคำสั่งแบบคงที่ซึ่งมีความเสี่ยงต่ำ ส่วนการขยายความสามารถในภายหลังยังคงต้องได้รับการอนุมัติตามปกติ

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
openclaw nodes status
openclaw nodes describe --node <idOrNameOrIp>
```

คำขอจับคู่ที่รอดำเนินการจะหมดอายุ 5 นาทีหลังจากการลองใหม่ครั้งล่าสุดของอุปกรณ์ โดยอุปกรณ์ที่เชื่อมต่อใหม่อย่างต่อเนื่องจะรักษาคำขอที่รอดำเนินการรายการเดิม (และ `requestId`) ให้ยังมีผล แทนที่จะสร้างพรอมต์ใหม่ทุกไม่กี่นาที โปรดดูวงจรชีวิตคำขอ/การอนุมัติทั้งหมดที่ [การจับคู่ Node](/th/gateway/pairing) หาก Node ลองใหม่ด้วยรายละเอียดการยืนยันตัวตนที่เปลี่ยนไป (บทบาท/ขอบเขต/กุญแจสาธารณะ) คำขอที่รอดำเนินการก่อนหน้าจะถูกแทนที่ และจะสร้าง `requestId` ใหม่ โดยไคลเอนต์จะได้รับเหตุการณ์ `device.pair.resolved` สำหรับคำขอที่ถูกแทนที่ และควรเรียกใช้ `openclaw devices list` อีกครั้งก่อนอนุมัติ

- `nodes status` ทำเครื่องหมาย Node ว่า **จับคู่แล้ว** เมื่อบทบาทการจับคู่อุปกรณ์มี `node`
- Mac แบบเนทีฟที่เชื่อมต่ออยู่และมีสิทธิ์การช่วยการเข้าถึง สามารถรายงานกิจกรรม
  อินพุตทางกายภาพที่รวมเข้าด้วยกันได้ Gateway จะทำเครื่องหมาย Mac ที่มีคุณสมบัติเหมาะสมและใหม่ล่าสุดเป็น
  `active` ให้คำใบ้รหัส Node ที่คงที่แก่เอเจนต์ และกำหนดเส้นทางการแจ้งเตือนการเชื่อมต่อ Node
  ไปยังเครื่องนั้นก่อนใช้ทางเลือกสำรองหลังจากหน่วงเวลา โปรดดูการตั้งค่า ความเป็นส่วนตัว เวลา และ
  การแก้ไขปัญหาที่ [สถานะการใช้งานคอมพิวเตอร์](/nodes/presence)
- ระเบียนการจับคู่อุปกรณ์คือสัญญาบทบาทที่ได้รับอนุมัติซึ่งคงอยู่ถาวร การหมุนเวียนโทเค็นยังคงอยู่ภายในสัญญานั้น และไม่สามารถยกระดับ Node ที่จับคู่แล้วไปเป็นบทบาทที่การอนุมัติการจับคู่ไม่เคยให้ไว้
- `node.pair.*` (CLI: `openclaw nodes pending/approve/reject/remove/rename`) คือที่เก็บการจับคู่ Node แยกต่างหากซึ่ง Gateway เป็นเจ้าของ โดยติดตามพื้นผิวคำสั่ง/ความสามารถที่ได้รับอนุมัติของ Node ระหว่างการเชื่อมต่อใหม่ ที่เก็บนี้ **ไม่ได้** ควบคุมการยืนยันตัวตนของการรับส่งข้อมูล เพราะการจับคู่อุปกรณ์เป็นผู้ดำเนินการดังกล่าว
- `openclaw nodes remove --node <id|name|ip>` ลบการจับคู่ Node สำหรับ Node ที่มีอุปกรณ์รองรับ ระบบจะเพิกถอนบทบาท `node` ของอุปกรณ์ในที่เก็บอุปกรณ์ที่จับคู่แล้ว และยุติการเชื่อมต่อเซสชันบทบาท Node ของอุปกรณ์นั้น: อุปกรณ์ที่มีหลายบทบาทจะคงแถวข้อมูลไว้และสูญเสียเฉพาะบทบาท `node` ส่วนแถวของอุปกรณ์ที่มีเฉพาะบทบาท Node จะถูกลบ นอกจากนี้ยังล้างรายการที่ตรงกันออกจากที่เก็บการจับคู่ Node แยกต่างหากด้วย `operator.pairing` อาจลบแถว Node ที่ไม่ใช่ผู้ดำเนินการบนอุปกรณ์อื่น ส่วนผู้เรียกที่ใช้โทเค็นอุปกรณ์และเพิกถอนบทบาท Node ของตนเองบนอุปกรณ์ที่มีหลายบทบาท จะต้องมี `operator.admin` เพิ่มเติม
- ขอบเขตการอนุมัติเป็นไปตามคำสั่งที่ประกาศไว้ในคำขอที่รอดำเนินการ:
  - คำขอที่ไม่มีคำสั่ง: `operator.pairing`
  - คำสั่ง Node ที่ไม่ใช่การดำเนินการ: `operator.pairing` + `operator.write`
  - `system.run` / `system.run.prepare` / `system.which`: `operator.pairing` + `operator.admin`

## ความคลาดเคลื่อนของเวอร์ชันและลำดับการอัปเกรด

Gateway WebSocket ยอมรับไคลเอนต์ Node ที่ยืนยันตัวตนแล้วภายในช่วงโปรโตคอล N-1
ดังนั้น Gateway v4 ปัจจุบันจึงยอมรับ Node v3 เมื่อการเชื่อมต่อประกาศ
ทั้ง `role: "node"` และ `client.mode: "node"` แต่เซสชันของผู้ดำเนินการและ UI
ยังคงต้องใช้โปรโตคอลปัจจุบัน

สำหรับการอัปเกรดกลุ่มอุปกรณ์เป็นระยะ ให้อัปเกรด Gateway ก่อน แล้วจึงอัปเกรด Node แต่ละตัว
Node แบบ N-1 จะยังคงมองเห็นและจัดการได้ในระหว่างการอัปเกรด โดย Gateway
จะบันทึก `legacy node protocol accepted` พร้อมคำแนะนำให้อัปเกรด การจับคู่
การยืนยันตัวตนอุปกรณ์ รายการคำสั่งที่อนุญาต และการอนุมัติการดำเนินการยังคงมีผล
ความสามารถและคำสั่งที่ Plugin เป็นเจ้าของจะยังคงถูกซ่อนจนกว่า Node จะอัปเกรดเป็น
โปรโตคอลปัจจุบัน Node ที่เก่ากว่า N-1 ต้องอัปเกรดผ่านช่องทางอื่นก่อน
เชื่อมต่อใหม่

การรับส่ง HTTPS แบบตรงของ watchOS ต้องใช้โปรโตคอลเวอร์ชันปัจจุบัน ให้อัปเดต
แอปนาฬิกาพร้อมกับ Gateway ก่อนเปิดใช้โหมดเชื่อมต่อโดยตรง

## โฮสต์ Node ระยะไกล (system.run)

ใช้ **โฮสต์ Node** เมื่อ Gateway ทำงานบนเครื่องหนึ่ง แต่ต้องการให้คำสั่งทำงานบนอีกเครื่องหนึ่ง โมเดลยังคงสื่อสารกับ **Gateway** ส่วน Gateway จะส่งต่อการเรียก `exec` ไปยัง **โฮสต์ Node** เมื่อเลือก `host=node`

| บทบาท         | ความรับผิดชอบ                                                   |
| ------------ | ---------------------------------------------------------------- |
| โฮสต์ Gateway | รับข้อความ เรียกใช้โมเดล และกำหนดเส้นทางการเรียกเครื่องมือ            |
| โฮสต์ Node    | ดำเนินการ `system.run`/`system.which` บนเครื่อง Node        |
| การอนุมัติ    | บังคับใช้บนโฮสต์ Node ผ่าน `~/.openclaw/exec-approvals.json` |

หมายเหตุเกี่ยวกับการอนุมัติ:

- การทำงานของ Node ที่อาศัยการอนุมัติจะผูกกับบริบทคำขอที่แน่นอน เส้นทางการดำเนินการจะเตรียม `systemRunPlan` มาตรฐานก่อนการอนุมัติ เมื่อได้รับอนุมัติแล้ว Gateway จะส่งต่อแผนที่จัดเก็บไว้นั้น ไม่ใช่ฟิลด์คำสั่ง/cwd/เซสชันที่ผู้เรียกแก้ไขในภายหลัง และจะตรวจสอบไดเรกทอรีทำงานซ้ำก่อนดำเนินการ
- สำหรับการดำเนินการไฟล์เชลล์/รันไทม์โดยตรง OpenClaw จะพยายามผูกโอเปอแรนด์ไฟล์ภายในเครื่องที่เป็นรูปธรรมหนึ่งรายการด้วย และปฏิเสธการดำเนินการหากไฟล์นั้นเปลี่ยนแปลงก่อนการดำเนินการ
- หาก OpenClaw ไม่สามารถระบุไฟล์ภายในเครื่องที่เป็นรูปธรรมได้หนึ่งรายการพอดีสำหรับคำสั่งอินเทอร์พรีเตอร์/รันไทม์ การดำเนินการที่อาศัยการอนุมัติจะถูกปฏิเสธ แทนที่จะแสร้งว่าครอบคลุมรันไทม์ทั้งหมด สำหรับความหมายของอินเทอร์พรีเตอร์ที่กว้างขึ้น ให้ใช้แซนด์บ็อกซ์ โฮสต์แยกต่างหาก หรือรายการที่อนุญาต/เวิร์กโฟลว์เต็มรูปแบบที่เชื่อถือได้อย่างชัดเจน

### เริ่มโฮสต์ Node (เบื้องหน้า)

บนเครื่อง Node:

```bash
openclaw node run --host <gateway-host> --port 18789 --display-name "Build Node"
```

`node run` ยังรองรับ `--context-path` (เส้นทางบริบท WS ของ Gateway), `--tls`, `--tls-fingerprint <sha256>` และ `--node-id` (เขียนทับรหัสอินสแตนซ์ไคลเอนต์แบบเดิม ซึ่งไม่รีเซ็ตการจับคู่)

### Gateway ระยะไกลผ่านอุโมงค์ SSH (ผูกกับลูปแบ็ก)

หาก Gateway ผูกกับลูปแบ็ก (`gateway.bind=loopback` ซึ่งเป็นค่าเริ่มต้นในโหมดภายในเครื่อง) โฮสต์ Node ระยะไกลจะเชื่อมต่อโดยตรงไม่ได้ ให้สร้างอุโมงค์ SSH และกำหนดให้โฮสต์ Node ชี้ไปยังปลายอุโมงค์ฝั่งภายในเครื่อง

ตัวอย่าง (โฮสต์ Node -> โฮสต์ Gateway):

```bash
# เทอร์มินัล A (ให้ทำงานต่อไป): ส่งต่อพอร์ตภายในเครื่อง 18790 -> Gateway 127.0.0.1:18789
ssh -N -L 18790:127.0.0.1:18789 user@gateway-host

# เทอร์มินัล B: ส่งออกโทเค็น Gateway และเชื่อมต่อผ่านอุโมงค์
export OPENCLAW_GATEWAY_TOKEN="<gateway-token>"
openclaw node run --host 127.0.0.1 --port 18790 --display-name "Build Node"
```

หมายเหตุ:

- `openclaw node run` รองรับการยืนยันตัวตนด้วยโทเค็นหรือรหัสผ่าน
- แนะนำให้ใช้ตัวแปรสภาพแวดล้อม: `OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`
- การกำหนดค่าสำรองคือ `gateway.auth.token` / `gateway.auth.password`
- ในโหมดภายในเครื่อง โฮสต์ Node จะละเว้น `gateway.remote.token` / `gateway.remote.password` โดยเจตนา
- ในโหมดระยะไกล `gateway.remote.token` / `gateway.remote.password` สามารถใช้ได้ตามกฎลำดับความสำคัญของโหมดระยะไกล
- หากกำหนดค่า SecretRefs ของ `gateway.auth.*` ภายในเครื่องที่ใช้งานอยู่แต่ไม่สามารถแก้ไขค่าได้ การยืนยันตัวตนของโฮสต์ Node จะล้มเหลวแบบปิด
- การแก้ไขค่าการยืนยันตัวตนของโฮสต์ Node รองรับเฉพาะตัวแปรสภาพแวดล้อม `OPENCLAW_GATEWAY_*`

### เริ่มโฮสต์ Node (บริการ)

```bash
openclaw node install --host <gateway-host> --port 18789 --display-name "Build Node"
openclaw node start
openclaw node restart
```

`node install` ยังรองรับ `--context-path`, `--tls`, `--tls-fingerprint`, `--node-id` (เฉพาะรหัสอินสแตนซ์ไคลเอนต์แบบเดิม), `--runtime <node>` (ค่าเริ่มต้น: node) และ `--force` สำหรับติดตั้งใหม่ นอกจากนี้ยังมี `node status`, `node stop` และ `node uninstall`

### จับคู่ + ตั้งชื่อ

บนโฮสต์ Gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw nodes status
```

หาก Node ลองใหม่ด้วยรายละเอียดการยืนยันตัวตนที่เปลี่ยนไป ให้เรียกใช้ `openclaw devices list` อีกครั้งและอนุมัติ `requestId` ปัจจุบัน

ตัวเลือกการตั้งชื่อ:

- `--display-name` บน `openclaw node run` / `openclaw node install` (คงอยู่ในแถว SQLite ของ `node_host_config` ที่ใช้ร่วมกัน พร้อมกับรหัสอินสแตนซ์ไคลเอนต์และเมทาดาทาการเชื่อมต่อ Gateway)
- `openclaw nodes rename --node <id|name|ip> --name "Build Node"` (การเขียนทับโดย Gateway)

### เซิร์ฟเวอร์ MCP ที่โฮสต์บน Node

กำหนดค่าเซิร์ฟเวอร์ MCP ใน `openclaw.json` บนเครื่อง Node ไม่ใช่บน
Gateway:

```json5
{
  nodeHost: {
    mcp: {
      servers: {
        localDocs: {
          command: "npx",
          args: ["-y", "@modelcontextprotocol/server-filesystem", "/srv/docs"],
          toolFilter: {
            include: ["read_*", "search"],
          },
        },
        internalApi: {
          url: "https://mcp.internal.example/mcp",
          transport: "streamable-http",
          headers: {
            Authorization: "Bearer ${INTERNAL_MCP_TOKEN}",
          },
        },
      },
    },
  },
}
```

โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้จะเริ่มเซิร์ฟเวอร์เหล่านี้ แสดงรายการเครื่องมือ และเผยแพร่
ตัวบ่งชี้หลังเชื่อมต่อ การเรียกเครื่องมือจะย้อนกลับไปยัง Node นั้นผ่าน
`mcp.tools.call.v1` โดย Gateway ไม่จำเป็นต้องมีการกำหนดค่า MCP หรือ Plugin JS
ที่ตรงกัน เส้นทาง v1 ที่โฮสต์บน Node นี้ไม่รองรับเซิร์ฟเวอร์ OAuth MCP

โฮสต์ Node ปัจจุบันประกาศตระกูลคำสั่ง `mcp.tools.call.v1` ที่มีมาให้ในตัวระหว่าง
การจับคู่ครั้งแรก แม้ไม่ได้กำหนดค่าเซิร์ฟเวอร์ MCP ไว้ก็ตาม Node ที่จับคู่บน
OpenClaw เวอร์ชันเก่าอาจขออัปเกรดพื้นผิวคำสั่งเพียงครั้งเดียวหลังจาก
อัปเดตโฮสต์ Node แล้ว การเพิ่ม ลบ หรือกรองเซิร์ฟเวอร์หลังจากนั้นไม่
จำเป็นต้องจับคู่ใหม่ เนื่องจากตระกูลคำสั่งที่ได้รับอนุมัติไม่เปลี่ยนแปลง ให้รีสตาร์ต
`openclaw node run` หรือ `openclaw node restart` เพื่อใช้การเปลี่ยนแปลงการกำหนดค่า MCP ของ Node
โฮสต์ Node จะไม่เฝ้าดูการกำหนดค่านี้

ผู้ดำเนินการ Gateway สามารถเพิกเฉยต่อเครื่องมือทั้งหมดที่เอเจนต์มองเห็นซึ่งเผยแพร่โดย Node ที่จับคู่แล้ว
รวมถึงเครื่องมือ MCP ที่โฮสต์บน Node ด้วย
`gateway.nodes.pluginTools.enabled: false` การปฏิเสธคำสั่งแบบตรงกันทุกประการ เช่น
`gateway.nodes.denyCommands: ["mcp.tools.call.v1"]` จะปิดกั้นการดำเนินการด้วย

### Skills ที่โฮสต์บน Node

ติดตั้ง Skills ไว้ใต้ไดเรกทอรี Skills ของ OpenClaw ที่ใช้งานอยู่บนเครื่อง Node
ซึ่งค่าเริ่มต้นคือ `~/.openclaw/skills` ส่วน `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR` และ
`OPENCLAW_CONFIG_PATH` จะย้ายโปรไฟล์ที่ใช้งานอยู่นั้น `OPENCLAW_STATE_DIR` มี
ลำดับความสำคัญเหนือกว่าสำหรับ Skills มิฉะนั้น `skills/` จะอยู่ข้างพาธที่แสดงโดย
`openclaw config file` โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้จะเผยแพร่ไฟล์ `SKILL.md` ที่ถูกต้อง
หลังจากเชื่อมต่อ และ Gateway จะเพิ่มไฟล์เหล่านั้นลงในสแนปช็อต Skills ของเอเจนต์เฉพาะขณะที่
Node นั้นยังคงเชื่อมต่ออยู่ ชื่อไดเรกทอรีของ Skills แต่ละรายการต้องตรงกับฟิลด์ frontmatter
`name` เพื่อให้ตัวระบุตำแหน่ง Node แบบนามธรรมแมปไปยังหนึ่งรายการโดยไม่ต้องเพิ่ม
ฟิลด์โปรโตคอลอีกฟิลด์หนึ่ง

การจับคู่บทบาท Node ครั้งแรกเป็นการอนุมัติการเผยแพร่ Skills การเพิ่ม การนำออก หรือ
การเปลี่ยนแปลง Skills ไม่จำเป็นต้องจับคู่อีกครั้งหรือเปลี่ยนการกำหนดค่า
Gateway รีสตาร์ต `openclaw node run` หรือ `openclaw node restart` หลังจากเปลี่ยน
ไฟล์ Skills ของ Node เนื่องจากโฮสต์ Node ไม่ได้เฝ้าดูไดเรกทอรี Skills

รายการ Skills ที่โฮสต์บน Node จะระบุ Node ของตนและตำแหน่งการดำเนินการ
ไฟล์ Skills พาธสัมพัทธ์ที่อ้างอิง และไบนารีจะยังคงอยู่บน Node นั้น
เอเจนต์อ่านตำแหน่ง `node://.../SKILL.md` ที่ประกาศไว้ด้วยเครื่องมือ
`read` ตามปกติ `file_fetch` ยอมรับพาธสัมบูรณ์บน Node ที่ผู้ปฏิบัติงานอนุมัติ
ไม่ใช่ตัวระบุตำแหน่ง Skills ของ Node ส่วนรันไทม์ที่ไม่มีเครื่องมืออ่านตามปกติสามารถเรียกใช้
`cat SKILL.md` ผ่าน `exec host=node node=<node-id>` แทน โดยใช้ไดเรกทอรี
`node://.../skills/<name>` ที่ประกาศไว้เป็น `workdir` ไฟล์และไบนารีที่อ้างอิง
ใช้เป้าหมายการดำเนินการและไดเรกทอรีทำงานเดียวกัน โฮสต์ Node จะแก้ตัวระบุตำแหน่งนั้นโดยอิงกับ
ไดเรกทอรีสถานะ OpenClaw ที่ใช้งานอยู่ ดังนั้นพาธสัมพัทธ์จึงถูกแก้บน Node
ไม่ใช่เครื่อง Gateway Node ที่เผยแพร่ต้องได้รับอนุมัติ `system.run`
และนโยบายการดำเนินการของเอเจนต์ต้องอนุญาต `host=node` มิฉะนั้น Skills จะไม่อยู่
ในสแนปช็อตของเอเจนต์นั้น

ตั้งค่า `nodeHost.skills.enabled: false` บน Node เพื่อหยุดการเผยแพร่ ผู้ปฏิบัติงาน
Gateway สามารถเพิกเฉยต่อ Skills จาก Node ที่จับคู่ทั้งหมดได้ด้วย
`gateway.nodes.skills.enabled: false`

### สถานะข้อมูลประจำตัวแบบไม่มีส่วนติดต่อผู้ใช้

Node แบบไม่มีส่วนติดต่อผู้ใช้เก็บระเบียนสถานะแยกกันสามรายการ:

- `~/.openclaw/state/openclaw.sqlite` (`node_host_config`): ID อินสแตนซ์ไคลเอนต์ ชื่อที่แสดง และข้อมูลเมตาการเชื่อมต่อ Gateway
- `~/.openclaw/identity/device.json`: คู่คีย์อุปกรณ์ที่ลงนามและ ID อุปกรณ์เชิงเข้ารหัสที่ได้มา
- `~/.openclaw/identity/device-auth.json`: โทเค็นการตรวจสอบสิทธิ์ของอุปกรณ์ที่จับคู่ โดยใช้ ID อุปกรณ์เชิงเข้ารหัสและบทบาทเป็นคีย์

สำหรับ Node ที่ลงนาม Gateway จะใช้ ID อุปกรณ์เชิงเข้ารหัสสำหรับการจับคู่และ
การกำหนดเส้นทาง Node ส่วน ID อินสแตนซ์ไคลเอนต์เป็นเพียงข้อมูลเมตาการเชื่อมต่อ ดังนั้นการเปลี่ยน
`--node-id` หรือการย้าย `node.json` ที่เลิกใช้งานแล้วจึงไม่รีเซ็ตการจับคู่ ดู
[สถานะข้อมูลประจำตัวและการจับคู่](/th/cli/node#identity-and-pairing-state) สำหรับขั้นตอน
เพิกถอนและจับคู่ใหม่ที่รองรับ รวมถึงหมายเหตุการอัปเกรด

### เพิ่มคำสั่งลงในรายการที่อนุญาต

การอนุมัติการดำเนินการเป็นแบบ **แยกตามโฮสต์ Node** เพิ่มรายการในรายการที่อนุญาตจาก Gateway:

```bash
openclaw approvals allowlist add --node <id|name|ip> "/usr/bin/uname"
openclaw approvals allowlist add --node <id|name|ip> "/usr/bin/sw_vers"
```

การอนุมัติจะอยู่บนโฮสต์ Node ที่ `~/.openclaw/exec-approvals.json`

### กำหนดให้การดำเนินการชี้ไปยัง Node

กำหนดค่าเริ่มต้น (การกำหนดค่า Gateway):

```bash
openclaw config set tools.exec.host node
openclaw config set tools.exec.security allowlist
openclaw config set tools.exec.node "<id-or-name>"
```

หรือกำหนดแยกตามเซสชัน:

```text
/exec host=node security=allowlist node=<id-or-name>
```

เมื่อตั้งค่าแล้ว การเรียก `exec` ใด ๆ ที่มี `host=node` จะทำงานบนโฮสต์ Node (ภายใต้รายการที่อนุญาต/การอนุมัติของ Node)

`host=auto` จะไม่เลือก Node โดยปริยายด้วยตัวเอง แต่อนุญาตให้ส่งคำขอ `host=node` แบบระบุชัดเจนต่อการเรียกจาก `auto` ได้ หากต้องการให้การดำเนินการบน Node เป็นค่าเริ่มต้นของเซสชัน ให้ตั้งค่า `tools.exec.host=node` หรือ `/exec host=node ...` อย่างชัดเจน

เนื้อหาที่เกี่ยวข้อง:

- [CLI ของโฮสต์ Node](/th/cli/node)
- [เครื่องมือดำเนินการ](/th/tools/exec)
- [การอนุมัติการดำเนินการ](/th/tools/exec-approvals)

### การอนุมานโมเดลภายในเครื่อง

Node บนเดสก์ท็อปหรือเซิร์ฟเวอร์สามารถเปิดให้ใช้โมเดลที่รองรับการแชตจากเซิร์ฟเวอร์ Ollama ที่ทำงานบน Node นั้นได้ เอเจนต์ใช้เครื่องมือ `node_inference` ของ Plugin Ollama เพื่อค้นหาโมเดลที่ติดตั้งและเรียกใช้พรอมต์ที่มีขอบเขตจำกัดจากระยะไกล โดย Gateway ไม่จำเป็นต้องเข้าถึง Ollama ผ่านเครือข่ายโดยตรง ดู [การอนุมาน Ollama ภายใน Node](/th/providers/ollama#node-local-inference) สำหรับการตั้งค่า การกรองโมเดล และคำสั่งตรวจสอบโดยตรง

### เซสชันและทรานสคริปต์ Codex

Plugin `codex` อย่างเป็นทางการสามารถเปิดให้ใช้เซสชัน Codex ที่ไม่ได้เก็บถาวรบน
โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้หรือ Node macOS แบบเนทีฟ การลงทะเบียนแค็ตตาล็อกไม่ขึ้นอยู่กับ
`supervision.enabled` อีกต่อไป ตัวเลือกดังกล่าวควบคุมเครื่องมือกำกับดูแลที่เอเจนต์ใช้
ตั้งค่า `sessionCatalog.enabled: false` ในการกำหนดค่า Plugin Codex เพื่อปิดใช้งาน
คำสั่งแค็ตตาล็อกของผู้ปฏิบัติงานและแค็ตตาล็อก Node ที่จับคู่ โดยไม่ปิดใช้งาน
ผู้ให้บริการหรือชุดควบคุม
Plugin ยังคงต้องทำงานบนคอมพิวเตอร์ทั้งสองเครื่อง และการตั้งค่าบน Node ยังคงเป็น
ความยินยอมภายในเครื่อง การเปิดใช้งานเฉพาะ Gateway ไม่ทำให้สามารถอ่านสถานะ Codex
ของคอมพิวเตอร์อีกเครื่องได้

Node ประกาศคำสั่งแบบอ่านอย่างเดียวที่มีการกำหนดเวอร์ชัน
`codex.appServer.threads.list.v1` และ
`codex.appServer.thread.turns.list.v1` โฮสต์ Node แบบเนทีฟที่มี
Codex CLI พร้อมใช้งานจะประกาศ `codex.terminal.resume.v1` ด้วย อนุมัติการอัปเกรดการจับคู่ Node
เมื่อคำสั่งเหล่านี้ปรากฏครั้งแรก Gateway จะเรียกคำสั่งดังกล่าวผ่าน
นโยบาย Node ของ Plugin ตามปกติและแยกความล้มเหลวตามโฮสต์

แถวของ Node ที่จับคู่จะปรากฏเป็นกลุ่ม **Codex** ในแถบด้านข้างเซสชันตามปกติ
โดยค่าเริ่มต้น การเลือกแถวจะเปิดบานหน้าต่างแชตตามปกติและอ่านทรานสคริปต์ที่คงอยู่
ผ่านการเรียก `thread/turns/list` ที่มีขอบเขตจำกัดและแบ่งหน้าด้วยเคอร์เซอร์
พร้อมการฉายรายการแบบเต็ม ใช้เมนูของแถว ส่วนหัวของตัวแสดง หรือค่ากำหนด **Open Codex/Claude sessions in** เพื่อเริ่ม `codex resume <thread-id>` ในเทอร์มินัลของผู้ปฏิบัติงานบนคอมพิวเตอร์ที่เป็นเจ้าของเซสชัน พาธเทอร์มินัลของ Node ที่จับคู่เป็นรีเลย์ PTY ในรายการที่อนุญาตซึ่ง Plugin Codex เป็นเจ้าของ ไม่ใช่การดำเนินการคำสั่งบน Node โดยพลการ

รีเลย์ไม่ได้ให้สัญญาการดำเนินการต่อของชุดควบคุม OpenClaw และการเป็นเจ้าของคลังเก็บถาวรอย่างครบถ้วน ดังนั้น **Continue** และ **Archive** จึงใช้ไม่ได้กับแถวระยะไกล บนคอมพิวเตอร์ Gateway แถวที่จัดเก็บและไม่ได้ทำงาน
สามารถเริ่มสาขาแชตแยกต่างหากที่ล็อกกับโมเดลได้ ทั้งสองแบบสามารถเก็บถาวรได้เฉพาะ
หลังจากผู้ปฏิบัติงานยืนยันว่าไม่มีไคลเอนต์ Codex อื่นกำลังใช้งานอยู่ กิจกรรมปัจจุบันของแถว
ที่จัดเก็บยังคงไม่ทราบ แถวที่ทำงานอยู่ไม่สามารถแตกสาขาหรือเก็บถาวรได้

ดู [กำกับดูแลเซสชัน Codex](/th/plugins/codex-supervision) สำหรับการตั้งค่า
การแบ่งหน้า การดำเนินการต่อภายในเครื่อง และขอบเขตความปลอดภัยของข้อมูลเมตา

### เซสชันและทรานสคริปต์ Claude

Plugin `anthropic` ที่รวมมาให้จะค้นหาเซสชัน Claude CLI และ Claude
Desktop ที่ไม่ได้เก็บถาวรบน Gateway และ Node ที่จับคู่โดยค่าเริ่มต้น ตั้งค่า
`plugins.entries.anthropic.config.sessionCatalog.enabled: false` เพื่อปิดใช้งาน
คำสั่งแค็ตตาล็อกของผู้ปฏิบัติงานและแค็ตตาล็อก Node ที่จับคู่ โดยไม่ปิดใช้งานโมเดล
Anthropic หรือแบ็กเอนด์ Claude CLI
Node ของแอป macOS ระยะไกลจะประกาศ
`anthropic.claude.sessions.list.v1` และ `anthropic.claude.sessions.read.v1`
เมื่อเปิดใช้งาน Plugin Anthropic และมี `~/.claude/projects/` อยู่ อนุมัติ
การอัปเกรดการจับคู่ Node เมื่อคำสั่งเหล่านี้ปรากฏครั้งแรก

โฮสต์ Node แบบเนทีฟที่มี Claude CLI พร้อมใช้งานจะประกาศ
`anthropic.claude.terminal.resume.v1` ด้วย แถว CLI และ Desktop ที่เข้าเกณฑ์สามารถเปิด
`claude --resume <session-id>` ในเทอร์มินัลของผู้ปฏิบัติงานบนโฮสต์ที่เป็นเจ้าของ
นี่คือการเข้าควบคุมเซสชันแบบเนทีฟ ซึ่งต่างจากการรับช่วงของ OpenClaw ตรงที่ไม่ได้
แยกเซสชัน Claude ออกมาก่อน

แค็ตตาล็อกรวมระเบียนดัชนีโปรเจกต์ Claude CLI ที่ถูกต้องเข้ากับคำนำหน้าข้อมูลเมตา
ที่มีขอบเขตจำกัดจากไฟล์ JSONL `sdk-cli` ปัจจุบัน ข้อมูลเมตาภายในเครื่องของ Claude Desktop
ให้ชื่อ Desktop และสถานะการเก็บถาวร ข้อมูลเมตา Desktop มีลำดับความสำคัญสูงกว่าเมื่อ
ทั้งสองแหล่งอ้างถึง ID เซสชัน Claude Code เดียวกัน ทรานสคริปต์ที่มีเฉพาะ CLI
ยังคงมองเห็นได้เนื่องจาก CLI ไม่มีแฟล็กเก็บถาวร การอ่านทรานสคริปต์ใช้เคอร์เซอร์
ออฟเซ็ตไบต์แบบทึบและการอ่านไฟล์ย้อนหลังที่มีขอบเขตจำกัด ดังนั้นการเลือกเซสชันขนาดใหญ่
หรือโหลดหน้าที่เก่ากว่าจะไม่อ่านประวัติ JSONL ทั้งหมดลงในการตอบกลับ Gateway ครั้งเดียว

คำสั่งแสดงรายการและอ่านเป็นแบบอ่านอย่างเดียว โดยเปิดเผยข้อมูลเมตาแค็ตตาล็อกและเนื้อหา
ทรานสคริปต์ผ่านเมธอดทั่วไป `sessions.catalog.list` และ
`sessions.catalog.read` เท่านั้น ไปยังการเชื่อมต่อของผู้ปฏิบัติงานที่ผ่านการตรวจสอบสิทธิ์และมี
`operator.write` แถว Claude CLI ภายใน Gateway สามารถรับช่วงได้จากตัวเขียนข้อความแชต
ตามปกติ โดย OpenClaw จะนำเข้าประวัติที่มองเห็นได้ซึ่งมีขอบเขตจำกัด ดำเนินการต่อด้วย
`--fork-session` ในเทิร์นแรก และปล่อยให้ทรานสคริปต์ต้นทางไม่ถูกแก้ไข

โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้สามารถเลือกใช้ขั้นตอนการดำเนินการต่อแบบเดียวกันได้:

```json5
{
  nodeHost: {
    agentRuns: {
      claude: { enabled: true },
    },
  },
}
```

Node จะประกาศ `agent.cli.claude.run.v1` เฉพาะเมื่อเปิดใช้งานการตั้งค่าภายใน Node นี้
และสามารถหาไฟล์ปฏิบัติการ `claude` บน Node นั้นได้ Gateway ไม่สามารถ
เปิดใช้งานจากระยะไกลได้ คำสั่งนี้ยังต้องผ่านนโยบายการอนุมัติการดำเนินการที่มีอยู่ของ Node
เมื่อคำสั่ง Claude ทั้งสามได้รับการประกาศและนโยบายคำสั่ง Node ของ Gateway อนุญาต
แถว Claude CLI บน Node นั้นจะดำเนินการต่อได้ โดย OpenClaw จะนำเข้าประวัติที่มีขอบเขตจำกัด ผูก
เซสชันที่รับช่วงเข้ากับ Node และไดเรกทอรีทำงานที่แค็ตตาล็อกรายงาน แล้ว
เรียกใช้แต่ละเทิร์น `claude -p` แบบครั้งเดียวที่นั่น เทิร์นแรกยังคงใช้
`--fork-session` เพื่อรักษาทรานสคริปต์ต้นทางไว้

เทิร์นที่ทำงานบน Node ใช้ค่าเริ่มต้น Claude ของ Node ใน v1 เทิร์นเหล่านี้ไม่ได้รับ
การกำหนดค่า MCP แบบลูปแบ็กของ Gateway หรือ Plugin Skills ของ Gateway ไม่สามารถเริ่มต้นใหม่จาก
ทรานสคริปต์ Gateway และจะปฏิเสธไฟล์แนบและรูปภาพ แถว Claude Desktop และ
Node ที่ไม่ได้ประกาศคำสั่งเรียกใช้จะยังคงดูได้อย่างเดียว ขณะนี้ Node ของแอป macOS
ยังไม่ประกาศคำสั่งนี้ ดังนั้นแถวของมันจึงยังคงดูได้อย่างเดียว

### เซสชัน OpenCode และ Pi

Plugin OpenCode และ ACPX ที่รวมมาให้ยังค้นหาแค็ตตาล็อกเซสชันแบบเนทีฟ
ที่อ่านอย่างเดียวบน Gateway และ Node ที่จับคู่ด้วย Node จะประกาศ
`opencode.sessions.list.v1` / `opencode.sessions.read.v1` เมื่อติดตั้ง CLI `opencode`
และประกาศ `acpx.pi.sessions.list.v1` / `acpx.pi.sessions.read.v1`
เมื่อมีไดเรกทอรีเซสชันของ Pi อนุมัติการอัปเกรดการจับคู่ Node เมื่อ
คำสั่งใหม่ปรากฏครั้งแรก เมื่อ CLI ที่ตรงกันพร้อมใช้งานด้วย Node จะเพิ่ม
`opencode.terminal.resume.v1` หรือ `acpx.pi.terminal.resume.v1` จากนั้นเมนูแถว
และส่วนหัวตัวแสดงที่มีอยู่จะสามารถเปิดเซสชันที่เลือกอีกครั้งในเทอร์มินัลของเจ้าของ
ด้วย `opencode --session <id>` หรือ `pi --session <id>`

OpenCode อ่านผ่านพื้นผิว JSON/การส่งออกของ CLI อย่างเป็นทางการ Pi อ่านจาก
ที่จัดเก็บเซสชัน JSONL ที่มีเอกสารกำกับ รวมถึงไดเรกทอรีเซสชัน `settings.json`
ระดับโปรเจกต์และส่วนกลาง ตลอดจนการแทนที่ `PI_CODING_AGENT_DIR` และ
`PI_CODING_AGENT_SESSION_DIR` แค็ตตาล็อกทั้งสองเปิดใช้งานโดยค่าเริ่มต้น
ปิดได้ใน Web UI ที่ **Config > Plugins**

การดำเนินการต่อในเทอร์มินัลใช้ไดเรกทอรีทำงานของเซสชันที่จัดเก็บไว้และ
รีเลย์ PTY สองทิศทางในรายการที่อนุญาตแบบเดียวกับ Codex และ Claude โดยไม่เปิดให้
ดำเนินการคำสั่งบน Node โดยพลการ

### การอัปโหลดไฟล์ผ่านเทอร์มินัล

Control UI สามารถลากไฟล์ลงในเทอร์มินัลของ Node ที่จับคู่ซึ่งเปิดอยู่ได้ โฮสต์ Node แบบเนทีฟจะประกาศคำสั่ง `terminal.upload` สำหรับผู้ดูแลระบบเท่านั้น ให้อนุมัติการอัปเกรดการจับคู่เมื่อคำสั่งนี้ปรากฏครั้งแรก แต่ละไฟล์จำกัดขนาดที่ 16 MiB โดยจะถูกพักไว้ในไดเรกทอรีชั่วคราวส่วนตัวบน Node นั้น และส่งกลับไปยังเทอร์มินัลเป็นพาธที่ใส่เครื่องหมายอัญประกาศสำหรับเชลล์แล้วโดยไม่มีการเรียกใช้

การแทรกพาธรองรับ PowerShell, `cmd.exe` และเชลล์ POSIX ที่รู้จัก (`sh`, Bash, Dash, Ash, Ksh, Zsh และ Fish) รวมถึง Git Bash บน Windows การแทนที่เชลล์อื่นจะถูกปฏิเสธ เนื่องจากไม่สามารถอนุมานกฎการใส่เครื่องหมายอัญประกาศได้อย่างปลอดภัย ให้เรียกใช้โฮสต์ Node ภายใน WSL เพื่อใช้พาธ WSL แบบเนทีฟ พาธ `cmd.exe` ที่มี `%` หรือ `!` จะถูกปฏิเสธด้วย เนื่องจากเชลล์ดังกล่าวจะขยายอักขระเหล่านั้นแม้อยู่ภายในอัญประกาศคู่

## การเรียกใช้คำสั่ง

ระดับล่าง (RPC แบบดิบ):

```bash
openclaw nodes invoke --node <idOrNameOrIp> --command canvas.eval --params '{"javaScript":"location.href"}'
```

`nodes invoke` บล็อก `system.run` และ `system.run.prepare` คำสั่งเหล่านั้นทำงานผ่านเครื่องมือ `exec` ที่มี `host=node` เท่านั้น (ดูด้านบน) มีตัวช่วยระดับสูงกว่าสำหรับเวิร์กโฟลว์ทั่วไปที่ต้องการ "มอบไฟล์แนบ MEDIA ให้เอเจนต์" (แคนวาส กล้อง หน้าจอ ตำแหน่งที่ตั้ง ด้านล่าง)

คำสั่ง Node แบบสตรีมมิงที่ทำงานเป็นเวลานานใช้เหตุการณ์ `node.invoke.progress`
แบบเพิ่มต่อเนื่อง แต่ละเหตุการณ์มี ID การเรียกใช้ หมายเลขลำดับที่เริ่มจากศูนย์ และส่วนข้อความ
UTF-8 ที่มีขนาดจำกัด โดย Gateway จะเรียงลำดับส่วนข้อความก่อนส่งให้
ผู้เรียกใช้ `node.invoke.result` ที่มีอยู่ยังคงเป็นการตอบกลับสุดท้ายเพียงรายการเดียว
ผู้เรียกใช้แบบสตรีมมิงสามารถกำหนดเส้นตายเมื่อไม่มีกิจกรรม ซึ่งเริ่มนับเมื่อได้รับ
เหตุการณ์ความคืบหน้าครั้งแรก และเริ่มนับใหม่หลังจากได้รับความคืบหน้าครั้งต่อๆ ไป ขณะที่ยังคง
ระยะหมดเวลาแบบตายตัวแยกต่างหากของการเรียกใช้ระหว่างการอนุมัติและการดำเนินการ ผลลัพธ์ การหมดเวลา
แบบตายตัว การหมดเวลาเมื่อไม่มีกิจกรรม และการตัดการเชื่อมต่อของ Node จะลบสถานะสตรีมที่
รอดำเนินการทั้งหมด การยกเลิกโดยผู้เรียกใช้จะส่ง `node.invoke.cancel` จากนั้นโฮสต์ Node
จะยุติแผนผังกระบวนการที่ตรงกัน คำสั่งคำขอ/การตอบกลับที่มีอยู่ไม่มีการเปลี่ยนแปลง

## นโยบายคำสั่ง

คำสั่ง Node ต้องผ่านเกตสองขั้นก่อนจึงจะเรียกใช้ได้:

1. Node ต้องประกาศคำสั่งไว้ในข้อมูลเมตาการเชื่อมต่อที่ผ่านการยืนยันตัวตน (`connect.commands`)
2. รายการอนุญาตของ Gateway ที่ได้จากแพลตฟอร์มและการอนุมัติต้องมีคำสั่งที่ประกาศไว้

รายการอนุญาตเริ่มต้นตามแพลตฟอร์ม (ก่อนค่าเริ่มต้นของ Plugin และการแทนที่ด้วย `allowCommands`/`denyCommands`):

| แพลตฟอร์ม | คำสั่งที่อนุญาตโดยค่าเริ่มต้น                                                                                                                                                                                                                                                                                           |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| iOS      | `camera.list`, `location.get`, `device.info`, `device.status`, `contacts.search`, `calendar.events`, `reminders.list`, `photos.latest`, `motion.activity`, `motion.pedometer`, `system.notify`                                                                                                                        |
| watchOS  | `device.info`, `device.status`, `system.notify`                                                                                                                                                                                                                                                                       |
| Android  | `camera.list`, `location.get`, `notifications.list`, `notifications.actions`, `system.notify`, `device.info`, `device.status`, `device.permissions`, `device.health`, `device.apps`, `contacts.search`, `calendar.events`, `callLog.search`, `reminders.list`, `photos.latest`, `motion.activity`, `motion.pedometer` |
| macOS    | `camera.list`, `location.get`, `device.info`, `device.status`, `contacts.search`, `calendar.events`, `reminders.list`, `photos.latest`, `motion.activity`, `motion.pedometer`, `system.notify`                                                                                                                        |
| Windows  | `camera.list`, `location.get`, `device.info`, `device.status`, `system.notify`                                                                                                                                                                                                                                        |
| Linux    | `system.notify` (คำสั่งโฮสต์ Node เช่น `system.run` ต้องผ่านการอนุมัติ โปรดดูด้านล่าง)                                                                                                                                                                                                                                  |

แถวเหล่านี้อธิบายขีดจำกัดสูงสุดของนโยบาย Gateway ไม่ใช่คำสั่งที่แอป Node ทุกแอปนำไปใช้ คำสั่งจะใช้งานได้ก็ต่อเมื่อ Node ที่เชื่อมต่อประกาศคำสั่งนั้นด้วย โดยเฉพาะอย่างยิ่ง แอป macOS ปัจจุบันไม่ได้ประกาศกลุ่มคำสั่งเกี่ยวกับอุปกรณ์และข้อมูลส่วนบุคคลที่ระบุไว้ในแถวนโยบาย macOS

คำสั่ง `canvas.*` (`canvas.present`, `canvas.hide`, `canvas.navigate`, `canvas.eval`, `canvas.snapshot`, `canvas.a2ui.*`) เป็นค่าเริ่มต้นของ Plugin บน iOS, Android, macOS, Windows, Linux และแพลตฟอร์มที่ไม่ทราบ Linux Node จะประกาศคำสั่งเหล่านี้เฉพาะเมื่อมีซ็อกเก็ต Canvas ภายในเครื่องของแอปเดสก์ท็อปเท่านั้น คำสั่ง Canvas ทั้งหมดบน iOS จำกัดให้ใช้ได้เฉพาะเมื่อแอปอยู่เบื้องหน้า

`talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` และ `talk.ptt.once` จะได้รับอนุญาตโดยค่าเริ่มต้นสำหรับ Node ใดๆ ที่โฆษณาความสามารถ `talk` หรือประกาศคำสั่ง `talk.*` โดยไม่ขึ้นกับป้ายกำกับแพลตฟอร์ม

คำสั่งโฮสต์เดสก์ท็อป (`system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `mcp.tools.call.v1` และ `screen.snapshot` บน macOS/Windows) ไม่ได้เป็นส่วนหนึ่งของตารางค่าเริ่มต้นแบบคงที่ตามแพลตฟอร์มด้านบน คำสั่งเหล่านี้จะใช้งานได้เมื่อผู้ดำเนินการอนุมัติคำขอจับคู่ที่ประกาศคำสั่งดังกล่าว หลังจากนั้นชุดคำสั่งที่ได้รับอนุมัติของ Node จะคงคำสั่งเหล่านี้ไว้เมื่อเชื่อมต่อใหม่

คำสั่งที่เป็นอันตรายหรือเกี่ยวข้องกับความเป็นส่วนตัวสูงยังคงต้องเลือกเข้าร่วมอย่างชัดเจนด้วย `gateway.nodes.allowCommands` แม้ว่า Node จะประกาศคำสั่งเหล่านั้นก็ตาม: `camera.snap`, `camera.clip`, `screen.record`, `computer.act`, `contacts.add`, `calendar.add`, `reminders.add`, `health.summary`, `sms.send`, `sms.search` โดย `gateway.nodes.denyCommands` มีผลเหนือกว่าค่าเริ่มต้นและรายการอนุญาตเพิ่มเติมเสมอ โปรดดู [ข้อมูลสรุป HealthKit](/platforms/ios-healthkit) สำหรับเกตความยินยอมของ iPhone และ [การควบคุมคอมพิวเตอร์](/th/nodes/computer-use) สำหรับเกตเพิ่มเติมเกี่ยวกับ macOS นโยบายเครื่องมือ และการเตรียมพร้อมสำหรับอินพุตเดสก์ท็อป

คำสั่ง Node ที่ Plugin เป็นเจ้าของสามารถเพิ่มนโยบายการเรียกใช้ Node ของ Gateway ได้ นโยบายดังกล่าวจะทำงานหลังจากตรวจสอบรายการอนุญาตและก่อนส่งต่อไปยัง Node ดังนั้น `node.invoke` แบบดิบ ตัวช่วย CLI และเครื่องมือเอเจนต์เฉพาะทางจึงใช้ขอบเขตสิทธิ์ของ Plugin เดียวกัน คำสั่ง Node ของ Plugin ที่เป็นอันตรายยังคงต้องเลือกเข้าร่วมด้วย `gateway.nodes.allowCommands` อย่างชัดเจน

หลังจาก Node เปลี่ยนรายการคำสั่งที่ประกาศไว้ ให้ปฏิเสธการจับคู่อุปกรณ์เดิมและอนุมัติคำขอใหม่ เพื่อให้ Gateway จัดเก็บสแนปช็อตคำสั่งที่อัปเดตแล้ว

## การกำหนดค่า (`openclaw.json`)

การตั้งค่าที่เกี่ยวข้องกับ Node อยู่ภายใต้ `gateway.nodes` และ `tools.exec`:

```json5
{
  gateway: {
    nodes: {
      // อนุมัติการจับคู่ Node ครั้งแรกจากเครือข่ายที่เชื่อถือโดยอัตโนมัติ (รายการ CIDR)
      // ปิดใช้งานเมื่อไม่ได้กำหนด ใช้เฉพาะกับคำขอ role:node ครั้งแรก
      // ที่ไม่ได้ขอขอบเขตสิทธิ์ และไม่อนุมัติการอัปเกรดโดยอัตโนมัติ
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
        // การอนุมัติอัตโนมัติที่ตรวจสอบผ่าน SSH (ค่าเริ่มต้น: เปิดใช้งาน) อนุมัติการจับคู่
        // Node ครั้งแรกเมื่อคีย์อุปกรณ์ตรงกันทุกประการตามที่อ่านกลับผ่าน SSH
        sshVerify: true,
      },
      // เชื่อถือเครื่องมือ Plugin ที่เอเจนต์มองเห็นซึ่งเผยแพร่โดย Node ที่จับคู่แล้ว (ค่าเริ่มต้น: true)
      pluginTools: {
        enabled: true,
      },
      // เลือกเข้าร่วมเพื่อใช้คำสั่ง Node ที่เป็นอันตราย/เกี่ยวข้องกับความเป็นส่วนตัวสูง (camera.snap เป็นต้น)
      allowCommands: ["camera.snap", "screen.record"],
      // บล็อกชื่อคำสั่งที่ตรงกันทุกประการ แม้ว่าค่าเริ่มต้นหรือ allowCommands จะรวมคำสั่งเหล่านั้นไว้
      denyCommands: ["camera.clip"],
    },
  },
  tools: {
    exec: {
      // โฮสต์ exec เริ่มต้น: "node" จะกำหนดเส้นทางการเรียก exec ทั้งหมดไปยัง Node ที่จับคู่แล้ว
      host: "node",
      // โหมดความปลอดภัยสำหรับ exec ของ Node: อนุญาตเฉพาะคำสั่งที่ได้รับอนุมัติ/อยู่ในรายการอนุญาต
      security: "allowlist",
      // ตรึง exec ไว้กับ Node ที่ระบุ (ID หรือชื่อ) ละเว้นเพื่ออนุญาต Node ใดก็ได้
      node: "build-node",
    },
  },
}
```

ใช้ชื่อคำสั่ง Node ที่ตรงกันทุกประการ `denyCommands` จะนำคำสั่งออก แม้ว่าค่าเริ่มต้นของแพลตฟอร์มหรือรายการ `allowCommands` จะอนุญาตคำสั่งนั้นก็ตาม Node ที่จับคู่แล้วอาจเผยแพร่ตัวอธิบายเครื่องมือ Plugin ที่เอเจนต์มองเห็นได้โดยค่าเริ่มต้น แต่คำสั่งของตัวอธิบายแต่ละรายการยังคงต้องอยู่ในพื้นผิวคำสั่งที่ได้รับอนุมัติของ Node ตั้งค่า `gateway.nodes.pluginTools.enabled: false` เพื่อละเว้นตัวอธิบายดังกล่าวทั้งหมด โปรดดู [ข้อมูลอ้างอิงการกำหนดค่า Gateway](/th/gateway/configuration-reference#gateway) สำหรับรายละเอียดฟิลด์การจับคู่ Node และนโยบายคำสั่งของ Gateway

การแทนที่ Node สำหรับ exec แยกตามเอเจนต์:

```json5
{
  agents: {
    list: [
      {
        id: "main",
        tools: { exec: { node: "build-node" } },
      },
    ],
  },
}
```

## ภาพหน้าจอ (สแนปช็อต Canvas)

หาก Node กำลังแสดง Canvas (WebView) `canvas.snapshot` จะส่งคืน `{ format, base64 }`

ตัวช่วย CLI (เขียนไปยังไฟล์ชั่วคราวและพิมพ์พาธที่บันทึกไว้):

```bash
openclaw nodes canvas snapshot --node <idOrNameOrIp> --format png
openclaw nodes canvas snapshot --node <idOrNameOrIp> --format jpg --max-width 1200 --quality 0.9
```

### การควบคุม Canvas

```bash
openclaw nodes canvas present --node <idOrNameOrIp> --target https://example.com
openclaw nodes canvas hide --node <idOrNameOrIp>
openclaw nodes canvas navigate https://example.com --node <idOrNameOrIp>
openclaw nodes canvas eval --node <idOrNameOrIp> --js "document.title"
```

หมายเหตุ:

- `canvas present` ยอมรับ URL หรือพาธไฟล์ภายในเครื่อง (`--target`) บน Node ที่รองรับพาธภายในเครื่อง รวมถึง `--x/--y/--width/--height` ซึ่งเป็นตัวเลือกสำหรับการกำหนดตำแหน่ง Canvas บน Linux ยอมรับ URL แบบ HTTP(S) หรือตัวเรนเดอร์ A2UI ที่รวมมาให้
- `canvas eval` ยอมรับ JS แบบอินไลน์ (`--js`) หรืออาร์กิวเมนต์ตามตำแหน่ง

### A2UI (Canvas)

```bash
openclaw nodes canvas a2ui push --node <idOrNameOrIp> --text "Hello"
openclaw nodes canvas a2ui push --node <idOrNameOrIp> --jsonl ./payload.jsonl
openclaw nodes canvas a2ui reset --node <idOrNameOrIp>
```

หมายเหตุ:

- Node บนมือถือและเดสก์ท็อป Linux ใช้หน้า A2UI ที่รวมมาให้และแอปเป็นเจ้าของสำหรับการเรนเดอร์ที่รองรับการดำเนินการ
- รองรับเฉพาะ JSONL ของ A2UI v0.8 เท่านั้น (ระบบจะปฏิเสธ v0.9/createSurface)
- iOS และ Android เรนเดอร์หน้า Canvas ของ Gateway ระยะไกล แต่จะส่งการดำเนินการจากปุ่ม A2UI เฉพาะจากหน้า A2UI ที่รวมมาให้และแอปเป็นเจ้าของเท่านั้น หน้า A2UI แบบ HTTP/HTTPS ที่โฮสต์โดย Gateway บนไคลเอนต์มือถือเหล่านั้นใช้สำหรับเรนเดอร์เท่านั้น
- macOS สามารถส่งการดำเนินการจากหน้า A2UI ของ Gateway ที่จำกัดตามความสามารถและแอปเลือกไว้อย่างตรงกัน หน้า HTTP/HTTPS อื่นๆ ยังคงใช้สำหรับเรนเดอร์เท่านั้น
- Linux ส่งการดำเนินการเฉพาะจากหน้า A2UI ที่รวมมาให้เท่านั้น หน้า HTTP/HTTPS อื่นๆ ยังคงใช้สำหรับเรนเดอร์เท่านั้น และ Linux Node แบบไม่มีส่วนติดต่อที่ไม่มีแอปเดสก์ท็อปจะไม่โฆษณา Canvas

## รูปภาพและวิดีโอ (กล้องของ Node)

รูปภาพ (`jpg`):

```bash
openclaw nodes camera list --node <idOrNameOrIp>
openclaw nodes camera snap --node <idOrNameOrIp>            # ค่าเริ่มต้น: กล้องทั้งสองด้าน (2 บรรทัด MEDIA)
openclaw nodes camera snap --node <idOrNameOrIp> --facing front
openclaw nodes camera snap --node <idOrNameOrIp> --device-id <id> --max-width 1200 --quality 0.9 --delay-ms 2000
```

คลิปวิดีโอ (`mp4`):

```bash
openclaw nodes camera clip --node <idOrNameOrIp> --duration 10s
openclaw nodes camera clip --node <idOrNameOrIp> --duration 3000 --no-audio
```

หมายเหตุ:

- Node ต้องอยู่ที่ **เบื้องหน้า** สำหรับ `canvas.*` และ `camera.*` (การเรียกจากเบื้องหลังจะส่งคืน `NODE_BACKGROUND_UNAVAILABLE`)
- Node จำกัดระยะเวลาคลิปเพื่อให้ขนาดเพย์โหลด base64 จัดการได้ (โปรดดู [การบันทึกจากกล้อง](/th/nodes/camera) สำหรับขีดจำกัดที่แน่นอนของแต่ละแพลตฟอร์ม) เครื่องมือเอเจนต์ `nodes` ยังจำกัด `durationMs` ที่ร้องขอไว้ที่ 300000 (5 นาที) ก่อนส่งต่อการเรียก โดยตัว Node เองจะบังคับใช้ขีดจำกัดที่เข้มงวดกว่า
- Android จะแจ้งขอสิทธิ์ `CAMERA`/`RECORD_AUDIO` เมื่อทำได้ หากปฏิเสธสิทธิ์ การดำเนินการจะล้มเหลวด้วย `*_PERMISSION_REQUIRED`

## การบันทึกหน้าจอ (Node)

Node ที่รองรับจะเปิดให้ใช้ `screen.record` (mp4) ตัวอย่าง:

```bash
openclaw nodes screen record --node <idOrNameOrIp> --duration 10s --fps 10
openclaw nodes screen record --node <idOrNameOrIp> --duration 10s --fps 10 --no-audio
```

หมายเหตุ:

- ความพร้อมใช้งานของ `screen.record` ขึ้นอยู่กับแพลตฟอร์มของ Node
- เครื่องมือเอเจนต์ `nodes` จำกัด `durationMs` ที่ร้องขอไว้ที่ 300000 (5 นาที) โดย Node อาจบังคับใช้ขีดจำกัดที่เข้มงวดยิ่งขึ้นเพื่อจำกัดขนาดเพย์โหลดที่ส่งคืน
- `--no-audio` ปิดใช้งานการจับเสียงจากไมโครโฟนบนแพลตฟอร์มที่รองรับ
- ใช้ `--screen <index>` เพื่อเลือกจอแสดงผลเมื่อมีหลายหน้าจอ (0 = จอหลัก)

## ตำแหน่งที่ตั้ง (Node)

Node เปิดให้ใช้ `location.get` เมื่อตั้งค่าเปิดใช้งานตำแหน่งที่ตั้ง

ตัวช่วย CLI:

```bash
openclaw nodes location get --node <idOrNameOrIp>
openclaw nodes location get --node <idOrNameOrIp> --accuracy precise --max-age 15000 --location-timeout 10000
```

หมายเหตุ:

- ตำแหน่งที่ตั้ง **ปิดโดยค่าเริ่มต้น**
- "Always" ต้องได้รับสิทธิ์จากระบบ การดึงข้อมูลในเบื้องหลังจะดำเนินการอย่างเต็มความสามารถโดยไม่รับประกันผลสำเร็จ
- การตอบกลับประกอบด้วยละติจูด/ลองจิจูด ความแม่นยำ (เมตร) และการประทับเวลา
- รูปแบบพารามิเตอร์/การตอบกลับทั้งหมดและรหัสข้อผิดพลาด: [คำสั่งตำแหน่งที่ตั้ง](/th/nodes/location-command)

## SMS (Node Android)

Node Android สามารถเปิดให้ใช้ `sms.send` และ `sms.search` เมื่อผู้ใช้ให้สิทธิ์ **SMS** และอุปกรณ์รองรับโทรศัพท์ ทั้งสองคำสั่งเป็นอันตรายโดยค่าเริ่มต้น ผู้ควบคุม Gateway ต้องเพิ่มคำสั่งเหล่านี้ใน `gateway.nodes.allowCommands` ด้วยจึงจะเรียกใช้ได้ (ดู [นโยบายคำสั่ง](#command-policy))

สำหรับการค้นหา SMS แบบอ่านอย่างเดียว ให้เลือกเข้าร่วมอย่างชัดเจนใน `openclaw.json`:

```json5
{
  gateway: {
    nodes: {
      allowCommands: ["sms.search"],
    },
  },
}
```

เพิ่ม `sms.send` แยกต่างหากเฉพาะเมื่อ Node ควรส่งข้อความได้ด้วย สิทธิ์ของ Android และการอนุญาตคำสั่งของ Gateway เป็นอิสระต่อกัน การให้สิทธิ์บนโทรศัพท์จะไม่แก้ไขนโยบายของ Gateway

การเรียกใช้ระดับต่ำ:

```bash
openclaw nodes invoke --node <idOrNameOrIp> --command sms.send --params '{"to":"+15555550123","message":"Hello from OpenClaw"}'
```

หมายเหตุ:

- สามารถประกาศ `sms.search` ก่อนให้สิทธิ์ `READ_SMS` เพื่อให้การเรียกใช้ส่งคืนข้อมูลวินิจฉัยสิทธิ์ได้ แต่การอ่านข้อความยังคงต้องใช้สิทธิ์ Android ดังกล่าว
- อุปกรณ์ที่ใช้เฉพาะ Wi-Fi และไม่มีระบบโทรศัพท์จะไม่ประกาศ `sms.send`
- ข้อผิดพลาด `requires explicit gateway.nodes.allowCommands opt-in` หมายความว่าโทรศัพท์ประกาศคำสั่งแล้ว แต่ผู้ควบคุม Gateway ยังไม่ได้อนุญาต

## คำสั่งเกี่ยวกับอุปกรณ์และข้อมูลส่วนบุคคล

Node iOS และ Android ประกาศคำสั่งข้อมูลแบบอ่านอย่างเดียวหลายรายการโดยค่าเริ่มต้น (ดูตาราง [นโยบายคำสั่ง](#command-policy)) ส่วน Android ยังเปิดให้ใช้ชุดคำสั่งขนาดใหญ่กว่าซึ่งควบคุมด้วยการตั้งค่าภายในแอปของ Android เอง

กลุ่มคำสั่งที่พร้อมใช้งาน:

- `device.status`, `device.info` — iOS, Android, Windows
- `device.permissions`, `device.health`, `device.apps` — เฉพาะ Android โดย `device.apps` ต้องเปิดใช้งานการแชร์ Installed Apps ใน Android Settings และส่งคืนแอปที่มองเห็นได้ในตัวเปิดแอปโดยค่าเริ่มต้น
- `notifications.list`, `notifications.actions` — เฉพาะ Android
- `photos.latest` — iOS, Android
- `contacts.search` — iOS, Android (อ่านอย่างเดียวโดยค่าเริ่มต้น) โดย `contacts.add` เป็นอันตรายและต้องใช้ `gateway.nodes.allowCommands`
- `calendar.events` — iOS, Android (อ่านอย่างเดียวโดยค่าเริ่มต้น) โดย `calendar.add` เป็นอันตรายและต้องใช้ `gateway.nodes.allowCommands`
- `reminders.list` — iOS, Android (อ่านอย่างเดียวโดยค่าเริ่มต้น) โดย `reminders.add` เป็นอันตรายและต้องใช้ `gateway.nodes.allowCommands`
- `callLog.search` — เฉพาะ Android
- `motion.activity`, `motion.pedometer` — iOS, Android โดยถูกจำกัดตามความสามารถของเซ็นเซอร์ที่พร้อมใช้งาน

ตัวอย่างการเรียกใช้:

```bash
openclaw nodes invoke --node <idOrNameOrIp> --command device.status --params '{}'
openclaw nodes invoke --node <idOrNameOrIp> --command device.apps --params '{"limit":10}'
openclaw nodes invoke --node <idOrNameOrIp> --command notifications.list --params '{}'
openclaw nodes invoke --node <idOrNameOrIp> --command photos.latest --params '{"limit":1}'
```

## คำสั่งระบบ (โฮสต์ Node / Node Mac)

Node macOS เปิดให้ใช้ `system.run`, `system.which`, `system.notify` และ `system.execApprovals.get/set` ส่วนโฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้เปิดให้ใช้ `system.run.prepare`, `system.run`, `system.which` และ `system.execApprovals.get/set`

ตัวอย่าง:

```bash
openclaw nodes notify --node <idOrNameOrIp> --title "Ping" --body "Gateway ready"
openclaw nodes invoke --node <idOrNameOrIp> --command system.which --params '{"bins":["git"]}'
```

หมายเหตุ:

- `system.run` ส่งคืน stdout/stderr/รหัสออกในเพย์โหลด
- ขณะนี้การดำเนินการเชลล์จะผ่านเครื่องมือ `exec` พร้อม `host=node` ส่วน `nodes` ยังคงเป็นพื้นผิว RPC โดยตรงสำหรับคำสั่ง Node ที่ระบุอย่างชัดเจน
- `nodes invoke` ไม่เปิดให้ใช้ `system.run` หรือ `system.run.prepare` ซึ่งยังคงอยู่เฉพาะบนเส้นทาง exec
- เส้นทาง exec จะเตรียม `systemRunPlan` มาตรฐานก่อนขออนุมัติ เมื่อได้รับการอนุมัติแล้ว Gateway จะส่งต่อแผนที่จัดเก็บไว้นั้น ไม่ใช่ฟิลด์คำสั่ง/cwd/เซสชันที่ผู้เรียกแก้ไขในภายหลัง
- `system.notify` ปฏิบัติตามสถานะสิทธิ์การแจ้งเตือนในแอป macOS และรองรับ `--priority <passive|active|timeSensitive>` กับ `--delivery <system|overlay|auto>`
- ข้อมูลเมตา `platform` / `deviceFamily` ของ Node ที่ไม่รู้จักจะใช้รายการอนุญาตเริ่มต้นแบบระมัดระวัง ซึ่งไม่รวม `system.run` และ `system.which` หากตั้งใจต้องใช้คำสั่งเหล่านั้นสำหรับแพลตฟอร์มที่ไม่รู้จัก ให้เพิ่มอย่างชัดเจนผ่าน `gateway.nodes.allowCommands`
- `system.run` รองรับ `--cwd`, `--env KEY=VAL`, `--command-timeout` และ `--needs-screen-recording`
- สำหรับตัวห่อเชลล์ (`bash|sh|zsh ... -c/-lc`) ค่า `--env` ที่มีขอบเขตตามคำขอจะถูกจำกัดเหลือรายการอนุญาตที่ระบุชัดเจน (`TERM`, `LANG`, `LC_*`, `COLORTERM`, `NO_COLOR`, `FORCE_COLOR`)
- สำหรับการตัดสินใจอนุญาตตลอดในโหมดรายการอนุญาต ตัวห่อการส่งต่อที่รู้จัก (`env`, `flock`, `nice`, `nohup`, `stdbuf`, `timeout`) จะบันทึกพาธของไฟล์ปฏิบัติการภายในแทนพาธของตัวห่อ หากไม่สามารถแกะตัวห่อได้อย่างปลอดภัย จะไม่มีการบันทึกรายการอนุญาตโดยอัตโนมัติ
- บนโฮสต์ Node Windows ในโหมดรายการอนุญาต การทำงานผ่านตัวห่อเชลล์โดยใช้ `cmd.exe /c` ต้องได้รับการอนุมัติ (มีเพียงรายการในรายการอนุญาตจะไม่อนุญาตรูปแบบตัวห่อโดยอัตโนมัติ)
- โฮสต์ Node จะละเว้นการแทนที่ `PATH` ใน `--env` และนำตัวแปรเริ่มต้นของอินเทอร์พรีเตอร์/เชลล์ชุดใหญ่ที่มีการดูแลออก (ตัวอย่างเช่น `NODE_OPTIONS`, `PYTHONPATH`, `BASH_ENV`, `DYLD_*`, `LD_*`) ก่อนเรียกใช้คำสั่ง หากต้องการรายการ PATH เพิ่มเติม ให้กำหนดค่าสภาพแวดล้อมบริการของโฮสต์ Node (หรือติดตั้งเครื่องมือในตำแหน่งมาตรฐาน) แทนการส่ง `PATH` ผ่าน `--env`
- ในโหมด Node บน macOS การใช้ `system.run` ถูกควบคุมด้วยการอนุมัติ exec ในแอป macOS (Settings → Exec approvals) โดย Ask/allowlist/full ทำงานเหมือนกับโฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้ และพรอมต์ที่ถูกปฏิเสธจะส่งคืน `SYSTEM_RUN_DENIED`
- บนโฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้ การใช้ `system.run` ถูกควบคุมด้วยการอนุมัติ exec (`~/.openclaw/exec-approvals.json`) สำหรับ macOS โดยเฉพาะ โปรดดูตัวแปรสภาพแวดล้อมการกำหนดเส้นทางโฮสต์ exec ในหัวข้อ [โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้](#headless-node-host-cross-platform) ด้านล่าง

## การผูก Node สำหรับ exec

เมื่อมี Node หลายรายการ สามารถผูก exec เข้ากับ Node ที่ระบุได้ การตั้งค่านี้กำหนด Node เริ่มต้นสำหรับ `exec host=node` (และสามารถแทนที่เป็นรายเอเจนต์ได้)

ค่าเริ่มต้นส่วนกลาง:

```bash
openclaw config set tools.exec.node "node-id-or-name"
```

การแทนที่รายเอเจนต์:

```bash
openclaw config get agents.list
openclaw config set 'agents.list[0].tools.exec.node' "node-id-or-name"
```

ยกเลิกการตั้งค่าเพื่ออนุญาต Node ใดก็ได้:

```bash
openclaw config unset tools.exec.node
openclaw config unset 'agents.list[0].tools.exec.node'
```

## แผนผังสิทธิ์

Node อาจมีแผนผัง `permissions` ใน `node.list` / `node.describe` โดยใช้ชื่อสิทธิ์เป็นคีย์ (เช่น `screenRecording`, `accessibility`, `location`) และมีค่าบูลีน (`true` = ได้รับสิทธิ์)

## โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้ (ข้ามแพลตฟอร์ม)

OpenClaw สามารถเรียกใช้ **โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้** (ไม่มี UI) ซึ่งเชื่อมต่อกับ Gateway WebSocket และเปิดให้ใช้ `system.run` / `system.which` ซึ่งมีประโยชน์บน Linux/Windows หรือสำหรับเรียกใช้ Node ขนาดเล็กควบคู่กับเซิร์ฟเวอร์

เริ่มต้น:

```bash
openclaw node run --host <gateway-host> --port 18789
```

หมายเหตุ:

- ยังคงต้องจับคู่ (Gateway จะแสดงพรอมต์จับคู่อุปกรณ์)
- ข้อมูลเมตาของอินสแตนซ์ไคลเอนต์ ข้อมูลประจำตัวอุปกรณ์ที่ลงนามแล้ว และการตรวจสอบสิทธิ์การจับคู่ใช้ไฟล์แยกกัน โปรดดู [สถานะข้อมูลประจำตัวแบบไม่มีส่วนติดต่อผู้ใช้](#headless-identity-state)
- การอนุมัติ exec ถูกบังคับใช้ภายในเครื่องผ่าน `~/.openclaw/exec-approvals.json` (ดู [การอนุมัติ exec](/th/tools/exec-approvals))
- บน macOS โฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้จะเรียกใช้ `system.run` ภายในเครื่องโดยค่าเริ่มต้น ตั้งค่า `OPENCLAW_NODE_EXEC_HOST=app` เพื่อกำหนดเส้นทาง `system.run` ผ่านโฮสต์ exec ของแอปคู่หู และเพิ่ม `OPENCLAW_NODE_EXEC_FALLBACK=0` เพื่อบังคับให้ใช้โฮสต์ของแอปและปฏิเสธโดยอัตโนมัติหากโฮสต์ไม่พร้อมใช้งาน
- เพิ่ม `--tls` / `--tls-fingerprint` เมื่อ Gateway WS ใช้ TLS

## โหมด Node ของ Mac

- แอปแถบเมนู macOS เชื่อมต่อกับเซิร์ฟเวอร์ Gateway WS ในฐานะ Node (ดังนั้น `openclaw nodes …` จึงทำงานกับ Mac เครื่องนี้)
- ในโหมดระยะไกล แอปจะเปิดทันเนล SSH สำหรับพอร์ต Gateway และเชื่อมต่อกับ `localhost`
