# การรองรับรูปภาพและสื่อ (2025-12-05)

ช่องทาง WhatsApp ทำงานผ่าน **Baileys Web** เอกสารนี้สรุปกฎการจัดการสื่อปัจจุบันสำหรับการส่ง, Gateway, และการตอบกลับของเอเจนต์

## เป้าหมาย

- ส่งสื่อพร้อมคำบรรยายเพิ่มเติมได้ผ่าน `openclaw message send --media`
- อนุญาตให้การตอบกลับอัตโนมัติจากกล่องข้อความเว็บมีสื่อควบคู่กับข้อความได้
- รักษาขีดจำกัดแยกตามประเภทให้สมเหตุสมผลและคาดการณ์ได้

## พื้นผิว CLI

- `openclaw message send --media <path-or-url> [--message <caption>]`
  - `--media` เป็นทางเลือก; คำบรรยายสามารถเว้นว่างได้สำหรับการส่งเฉพาะสื่อ
  - `--dry-run` พิมพ์เพย์โหลดที่แก้ไขแล้ว; `--json` ส่งออก `{ channel, to, messageId, mediaUrl, caption }`

## พฤติกรรมของช่องทาง WhatsApp Web

- อินพุต: พาธไฟล์ในเครื่อง **หรือ** URL HTTP(S)
- โฟลว์: โหลดเป็น Buffer, ตรวจจับชนิดสื่อ, แล้วสร้างเพย์โหลดที่ถูกต้อง:
  - **รูปภาพ:** ปรับขนาดและบีบอัดซ้ำเป็น JPEG (ด้านยาวสุด 2048px) โดยตั้งเป้าไว้ที่ `channels.whatsapp.mediaMaxMb` (ค่าเริ่มต้น: 50 MB)
  - **เสียง/เสียงพูด/วิดีโอ:** ส่งผ่านได้สูงสุด 16 MB; เสียงจะถูกส่งเป็นโน้ตเสียง (`ptt: true`)
  - **เอกสาร:** อื่นๆ ทั้งหมด สูงสุด 100 MB พร้อมคงชื่อไฟล์ไว้เมื่อมี
- การเล่นแบบ GIF ของ WhatsApp: ส่ง MP4 พร้อม `gifPlayback: true` (CLI: `--gif-playback`) เพื่อให้ไคลเอนต์มือถือวนเล่นแบบอินไลน์
- การตรวจจับ MIME จะใช้ magic bytes ก่อน จากนั้นใช้ส่วนหัว แล้วจึงใช้นามสกุลไฟล์
- คำบรรยายมาจาก `--message` หรือ `reply.text`; อนุญาตให้คำบรรยายว่างได้
- การบันทึกล็อก: โหมดไม่ละเอียดแสดง `↩️`/`✅`; โหมดละเอียดรวมขนาดและพาธ/URL ต้นทาง

## ไปป์ไลน์การตอบกลับอัตโนมัติ

- `getReplyFromConfig` คืนค่า `{ text?, mediaUrl?, mediaUrls? }`
- เมื่อมีสื่อ ตัวส่งบนเว็บจะแก้ไขพาธในเครื่องหรือ URL โดยใช้ไปป์ไลน์เดียวกับ `openclaw message send`
- หากระบุรายการสื่อหลายรายการ จะส่งตามลำดับทีละรายการ

## สื่อขาเข้าไปยังคำสั่ง (Pi)

- เมื่อข้อความเว็บขาเข้ามีสื่อ OpenClaw จะดาวน์โหลดไปยังไฟล์ชั่วคราวและเปิดเผยตัวแปรเทมเพลต:
  - `{{MediaUrl}}` pseudo-URL สำหรับสื่อขาเข้า
  - `{{MediaPath}}` พาธชั่วคราวในเครื่องที่เขียนไว้ก่อนรันคำสั่ง
- เมื่อเปิดใช้ sandbox Docker ต่อเซสชัน สื่อขาเข้าจะถูกคัดลอกเข้า workspace ของ sandbox และ `MediaPath`/`MediaUrl` จะถูกเขียนใหม่เป็นพาธสัมพัทธ์ เช่น `media/inbound/<filename>`
- การทำความเข้าใจสื่อ (หากกำหนดค่าผ่าน `tools.media.*` หรือ `tools.media.models` ที่ใช้ร่วมกัน) จะทำงานก่อนการเทมเพลตและสามารถแทรกบล็อก `[Image]`, `[Audio]`, และ `[Video]` ลงใน `Body`
  - เสียงจะตั้งค่า `{{Transcript}}` และใช้ทรานสคริปต์สำหรับการแยกวิเคราะห์คำสั่ง เพื่อให้คำสั่งแบบสแลชยังทำงานได้
  - คำอธิบายวิดีโอและรูปภาพจะคงข้อความคำบรรยายใดๆ ไว้สำหรับการแยกวิเคราะห์คำสั่ง
  - หากโมเดลรูปภาพหลักที่ใช้งานอยู่รองรับวิชันโดยตรงอยู่แล้ว OpenClaw จะข้ามบล็อกสรุป `[Image]` และส่งรูปภาพต้นฉบับให้โมเดลแทน
- โดยค่าเริ่มต้น จะประมวลผลเฉพาะไฟล์แนบรูปภาพ/เสียง/วิดีโอรายการแรกที่ตรงกันเท่านั้น; ตั้งค่า `tools.media.<cap>.attachments` เพื่อประมวลผลไฟล์แนบหลายรายการ

## ขีดจำกัดและข้อผิดพลาด

**ขีดจำกัดการส่งขาออก (การส่งผ่าน WhatsApp web)**

- รูปภาพ: สูงสุด `channels.whatsapp.mediaMaxMb` (ค่าเริ่มต้น: 50 MB) หลังบีบอัดซ้ำ
- เสียง/เสียงพูด/วิดีโอ: ขีดจำกัด 16 MB; เอกสาร: ขีดจำกัด 100 MB
- สื่อที่ใหญ่เกินไปหรืออ่านไม่ได้ → ข้อผิดพลาดที่ชัดเจนในล็อก และการตอบกลับจะถูกข้าม

**ขีดจำกัดการทำความเข้าใจสื่อ (การถอดเสียง/คำอธิบาย)**

- ค่าเริ่มต้นของรูปภาพ: 10 MB (`tools.media.image.maxBytes`)
- ค่าเริ่มต้นของเสียง: 20 MB (`tools.media.audio.maxBytes`)
- ค่าเริ่มต้นของวิดีโอ: 50 MB (`tools.media.video.maxBytes`)
- สื่อที่ใหญ่เกินไปจะข้ามการทำความเข้าใจ แต่การตอบกลับยังคงดำเนินต่อพร้อมเนื้อหาต้นฉบับ

## หมายเหตุสำหรับการทดสอบ

- ครอบคลุมโฟลว์การส่งและการตอบกลับสำหรับกรณีรูปภาพ/เสียง/เอกสาร
- ตรวจสอบการบีบอัดซ้ำสำหรับรูปภาพ (ขอบเขตขนาด) และแฟล็กโน้ตเสียงสำหรับเสียง
- ตรวจให้แน่ใจว่าการตอบกลับที่มีสื่อหลายรายการกระจายออกเป็นการส่งตามลำดับ
