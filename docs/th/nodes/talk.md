โหมด Talk ครอบคลุมรูปแบบรันไทม์ 5 รูปแบบ:

- **Talk แบบเนทีฟบน macOS/iOS/Android**: การรู้จำเสียงพูดภายในเครื่อง การแชตผ่าน Gateway และ TTS ของ `talk.speak` โดย Node จะประกาศความสามารถ `talk` และระบุคำสั่ง `talk.*` ที่รองรับ
- **Talk บน iOS (เรียลไทม์)**: WebRTC ที่ไคลเอนต์เป็นผู้ควบคุม สำหรับการกำหนดค่าเรียลไทม์ของ OpenAI ที่เลือกทรานสปอร์ต `webrtc` หรือไม่ได้ระบุทรานสปอร์ต การกำหนดค่าเรียลไทม์ที่ระบุ `gateway-relay`, `provider-websocket` และที่ไม่ใช่ OpenAI อย่างชัดเจนจะยังคงใช้รีเลย์ที่ Gateway เป็นผู้ควบคุม ส่วนการกำหนดค่าที่ไม่ใช่เรียลไทม์จะใช้ลูปเสียงพูดแบบเนทีฟ
- **Talk บนเบราว์เซอร์**: `talk.client.create` สำหรับเซสชัน `webrtc`/`provider-websocket` ที่ไคลเอนต์เป็นผู้ควบคุม หรือ `talk.session.create` สำหรับเซสชัน `gateway-relay` ที่ Gateway เป็นผู้ควบคุม โดยสงวน `managed-room` ไว้สำหรับการส่งมอบให้ Gateway และห้องวอล์กกีทอล์กกี
- **Talk บน Android (เรียลไทม์)**: เลือกใช้ด้วย `talk.realtime.mode: "realtime"` และ `talk.realtime.transport: "gateway-relay"` มิฉะนั้น Android จะยังคงใช้การรู้จำเสียงพูดแบบเนทีฟ การแชตผ่าน Gateway และ `talk.speak`
- **ไคลเอนต์สำหรับถอดเสียงเท่านั้น**: `talk.session.create({ mode: "transcription", transport: "gateway-relay", brain: "none" })` ตามด้วย `talk.session.appendAudio`, `talk.session.cancelTurn` และ `talk.session.close` สำหรับคำบรรยายแทนเสียง/การป้อนตามคำบอกโดยไม่มีเสียงตอบกลับจากผู้ช่วย ส่วนบันทึกเสียงที่อัปโหลดแบบครั้งเดียวยังคงใช้เส้นทางเสียงของ [การทำความเข้าใจสื่อ](/th/nodes/media-understanding)

Talk แบบเนทีฟเป็นลูปต่อเนื่อง: ฟังเสียงพูด ส่งข้อความถอดเสียงไปยังโมเดลผ่านเซสชันที่ใช้งานอยู่ รอการตอบกลับ แล้วจึงอ่านออกเสียงผ่านผู้ให้บริการ Talk ที่กำหนดค่าไว้ (`talk.speak`)

Talk แบบเรียลไทม์ที่ไคลเอนต์เป็นผู้ควบคุมจะส่งต่อการเรียกใช้เครื่องมือของผู้ให้บริการผ่าน `talk.client.toolCall` แทนการเรียก `chat.send` โดยตรง ขณะที่การปรึกษาแบบเรียลไทม์ทำงานอยู่ ไคลเอนต์สามารถเรียก `talk.client.steer` หรือ `talk.session.steer` เพื่อจำแนกอินพุตเสียงพูดเป็น `status`, `steer`, `cancel` หรือ `followup` การชี้นำที่ยอมรับแล้วจะถูกจัดเข้าคิวของการรันแบบฝังที่ใช้งานอยู่ ส่วนการชี้นำที่ถูกปฏิเสธจะส่งคืนเหตุผล เช่น `no_active_run`, `not_streaming` หรือ `compacting`

ถ้อยคำเรียลไทม์ของผู้ใช้และผู้ช่วยที่เสร็จสมบูรณ์แล้วจะถูกผนวกแบบสดเข้ากับเซสชันเอเจนต์ที่ใช้งานอยู่เสมอ เพื่อให้รอบการแชตและเสียงครั้งถัดไปใช้ประวัติเดียวกัน ทรานสปอร์ตที่ไคลเอนต์เป็นผู้ควบคุมจะรายงานข้อความถอดเสียงที่เสร็จสมบูรณ์พร้อมรหัสรายการที่คงที่ ส่วนเซสชันรีเลย์ของ Gateway จะผนวกเหตุการณ์เดียวกันที่ฝั่งเซิร์ฟเวอร์ เซสชันของผู้ให้บริการยังได้รับบริบทโปรไฟล์เรียลไทม์แบบจำกัดขอบเขตที่ Discord Voice ใช้อีกด้วย

การรันการปรึกษาที่เริ่มจากเสียงต้องได้รับการยืนยันด้วยเสียงครั้งใหม่ที่ตรงตามข้อความทุกประการ ก่อนดำเนินการที่มีผลกระทบสูง เช่น การส่งข้อความ การควบคุม Node การดำเนินการผ่านเบราว์เซอร์/คอมพิวเตอร์ การเปลี่ยนแปลงบริการ คำสั่งเชลล์ที่ทำลายข้อมูล หรือการเผยแพร่ การยืนยันนี้ใช้ได้เฉพาะกับอาร์กิวเมนต์ของเครื่องมือที่ถูกบล็อกชุดนั้นเท่านั้นและใช้ได้ครั้งเดียว การรันอื่นที่เกิดขึ้นพร้อมกันและไม่เกี่ยวข้องจะไม่ได้รับผลกระทบ เมื่อการโทรสิ้นสุดลง OpenClaw สามารถส่งสรุปแบบกระชับ **การเปลี่ยนแปลงจากการโทรด้วยเสียง** สำหรับเครื่องมือที่แก้ไขข้อมูล ไปยังเป้าหมายการส่งล่าสุดของเซสชันที่ไม่ใช่ WebChat

Talk สำหรับถอดเสียงเท่านั้นจะปล่อยซองเหตุการณ์ Talk แบบเดียวกับเซสชันเรียลไทม์และ STT/TTS แต่ใช้ `mode: "transcription"` และ `brain: "none"` เซสชัน Talk ทั้งหมดจะกระจายเหตุการณ์บนช่อง `talk.event` โดยไคลเอนต์สมัครรับข้อมูลช่องนี้เพื่อรับการอัปเดตข้อความถอดเสียงบางส่วน/ฉบับสมบูรณ์ (`transcript.delta`/`transcript.done`) และข้อมูลเทเลเมทรีอื่นของเซสชัน

Video Talk บนเบราว์เซอร์พร้อมใช้งานสำหรับ OpenAI Realtime WebRTC และเซสชัน
WebSocket ของผู้ให้บริการ Google Live โดย OpenAI จะได้รับภาพ JPEG แบบจำกัดขอบเขตเพียงภาพเดียวเมื่อ
`describe_view` ขอข้อมูลบริบทภาพ และจะไม่ได้รับแทร็กกล้องแบบต่อเนื่อง
Google Live จะได้รับเฟรม JPEG แบบจำกัดขอบเขตโดยตรงจากเบราว์เซอร์
สูงสุดหนึ่งเฟรมต่อวินาที ขณะที่ `describe_view` รายงานสถานะ

## ลักษณะการทำงาน (macOS)

- โอเวอร์เลย์เปิดตลอดเวลาขณะที่เปิดใช้งานโหมด Talk
- การเปลี่ยนช่วง **กำลังฟัง &rarr; กำลังคิด &rarr; กำลังพูด**
- เมื่อหยุดชั่วครู่ (ช่วงที่ไม่มีเสียง) ระบบจะส่งข้อความถอดเสียงปัจจุบัน
- การตอบกลับจะถูกเขียนลงใน WebChat (เหมือนกับการพิมพ์)
- **ขัดจังหวะเมื่อมีเสียงพูด** (เปิดไว้โดยค่าเริ่มต้น): หากผู้ใช้พูดขณะที่ผู้ช่วยกำลังพูด การเล่นเสียงจะหยุดลง และระบบจะบันทึกเวลาที่เกิดการขัดจังหวะไว้สำหรับพรอมต์ถัดไป

## คำสั่งเสียงในการตอบกลับ

ผู้ช่วยสามารถเติมบรรทัด JSON หนึ่งบรรทัดไว้หน้าการตอบกลับเพื่อควบคุมเสียง:

```json
{ "voice": "<voice-id>", "once": true }
```

กฎ:

- ใช้เฉพาะบรรทัดแรกที่ไม่ว่างเท่านั้น โดยระบบจะนำบรรทัด JSON ออกก่อนเล่นเสียง TTS
- ระบบจะละเว้นคีย์ที่ไม่รู้จัก
- `once: true` มีผลกับการตอบกลับปัจจุบันเท่านั้น หากไม่มีค่านี้ เสียงดังกล่าวจะกลายเป็นค่าเริ่มต้นใหม่ของโหมด Talk

คีย์ที่รองรับ: `voice` / `voice_id` / `voiceId`, `model` / `model_id` / `modelId`, `speed`, `rate` (คำต่อนาที), `stability`, `similarity`, `style`, `speakerBoost`, `seed`, `normalize`, `lang`, `output_format`, `latency_tier`, `once`

## การกำหนดค่า (`~/.openclaw/openclaw.json`)

```json5
{
  talk: {
    provider: "elevenlabs",
    providers: {
      elevenlabs: {
        voiceId: "elevenlabs_voice_id",
        modelId: "eleven_v3",
        outputFormat: "mp3_44100_128",
        apiKey: "elevenlabs_api_key",
      },
      mlx: {
        modelId: "mlx-community/Soprano-80M-bf16",
      },
      system: {},
    },
    speechLocale: "ru-RU",
    silenceTimeoutMs: 1500,
    interruptOnSpeech: true,
    realtime: {
      provider: "openai",
      providers: {
        openai: {
          apiKey: "openai_api_key",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
        },
      },
      instructions: "Speak warmly and keep answers brief.",
      mode: "realtime",
      transport: "webrtc",
      brain: "agent-consult",
    },
  },
}
```

| `realtime.consultRouting`                | -                                          | `provider-direct` รักษาการตอบกลับโดยตรงของผู้ให้บริการไว้เมื่อข้าม `openclaw_agent_consult`; ส่วน `force-agent-consult` กำหนดเส้นทางข้อความถอดเสียงของผู้ใช้ที่เสร็จสมบูรณ์แล้วผ่าน OpenClaw แทน                                                                                          |
| `realtime.instructions`                  | -                                          | เพิ่มคำสั่งระบบสำหรับผู้ให้บริการต่อท้ายพรอมต์เรียลไทม์ในตัวของ OpenClaw (รูปแบบ/น้ำเสียง); คำแนะนำ `openclaw_agent_consult` เริ่มต้นจะยังคงอยู่                                                                                                                |

`talk.catalog` เปิดเผยรหัสผู้ให้บริการมาตรฐานและนามแฝงในรีจิสทรี รวมถึงโหมด/การรับส่ง/กลยุทธ์สมอง/รูปแบบเสียงเรียลไทม์/แฟล็กความสามารถที่ถูกต้องของผู้ให้บริการแต่ละราย และผลความพร้อมที่รันไทม์เลือก ไคลเอนต์ Talk ของบุคคลที่หนึ่งควรอ่านแค็ตตาล็อกดังกล่าวแทนการดูแลนามแฝงผู้ให้บริการไว้ภายในเครื่อง สำหรับ Gateway รุ่นเก่าที่ไม่มีข้อมูลความพร้อมของกลุ่ม ให้ถือว่ายังไม่ได้รับการตรวจสอบแทนที่จะสรุปว่าไม่ได้กำหนดค่าอย่างแน่นอน ระบบจะค้นพบผู้ให้บริการถอดเสียงแบบสตรีมผ่าน `talk.catalog.transcription`; รีเลย์ของ Gateway ปัจจุบันใช้การกำหนดค่าผู้ให้บริการสตรีมของ Voice Call จนกว่าจะมีพื้นผิวการกำหนดค่าการถอดเสียงสำหรับ Talk โดยเฉพาะ

## UI ของ macOS

- สวิตช์แถบเมนู: **Talk**
- แท็บการกำหนดค่า: กลุ่ม **Talk Mode** (รหัสเสียง + สวิตช์การขัดจังหวะ)
- โอเวอร์เลย์: ลูกแก้วแสดงรูปคลื่นการสนทนาสากล (ใช้ร่วมกับ iOS, watchOS และ Android) ขณะฟัง รูปคลื่นจะเคลื่อนไหวตามระดับไมโครโฟนแบบสด ขณะพูด รูปคลื่นจะเคลื่อนไหวตามเอนเวโลปการเล่น TTS จริง และขณะคิด รูปคลื่นจะเต้นเบา ๆ คลิกลูกแก้วเพื่อหยุดชั่วคราว/เล่นต่อ ดับเบิลคลิกเพื่อหยุดพูด และคลิก X เพื่อออกจากโหมด Talk

## UI ของ Android

- การนำทางหลักของ Android คือ **Home**, **Chat** และ **Settings** อินพุตเสียง
  อยู่ในช่องเขียนข้อความของ Chat แทนที่จะเป็นแท็บ Voice แยกต่างหาก
- แตะไมโครโฟนในช่องเขียนข้อความเพื่อป้อนข้อความตามคำบอกบนอุปกรณ์ กดค้างเพื่อบันทึก
  ไฟล์แนบบันทึกเสียง เริ่ม Talk แบบต่อเนื่องจากรูปคลื่น Talk
- การป้อนข้อความตามคำบอก การบันทึกเสียง และ Talk เป็นเส้นทางไมโครโฟนที่ใช้ร่วมกันไม่ได้
  การเริ่มเส้นทางหนึ่งจะหยุดหรือบล็อกเส้นทางอื่น
- Talk แบบเรียลไทม์จะเลือกใช้ไมโครโฟนของชุดหูฟัง Bluetooth Classic หรือ BLE ที่เชื่อมต่ออยู่ก่อน
  หากตัดการเชื่อมต่อ แอปจะขอใช้อินพุตจากชุดหูฟังอื่น หรือ
  ใช้ไมโครโฟนเริ่มต้นเป็นทางเลือกสำรอง และคืนค่าการกำหนดลักษณะเริ่มต้นเมื่อ
  การบันทึกสิ้นสุดลง
- การป้อนข้อความตามคำบอกและการบันทึกเสียงจะหยุดเมื่อแอปออกจากเบื้องหน้า หรือ
  ผู้ใช้ออกจาก Chat
- Talk Mode จะทำงานต่อไปจนกว่าจะปิดสวิตช์หรือ Node ตัดการเชื่อมต่อ โดยใช้ประเภทบริการเบื้องหน้าสำหรับไมโครโฟนของ Android ขณะทำงาน
- Android รองรับรูปแบบเอาต์พุต `pcm_16000`, `pcm_22050`, `pcm_24000` และ `pcm_44100` สำหรับการสตรีม `AudioTrack` ที่มีเวลาแฝงต่ำ

## หมายเหตุ

- ต้องมีสิทธิ์เข้าถึงเสียงพูดและไมโครโฟน
- Talk แบบเนทีฟใช้เซสชัน Gateway ที่ทำงานอยู่ และจะใช้การสำรวจประวัติเป็นทางเลือกสำรองเฉพาะเมื่อเหตุการณ์การตอบกลับไม่พร้อมใช้งาน
- Gateway กำหนดเส้นทางการเล่นของ Talk ผ่าน `talk.speak` โดยใช้ผู้ให้บริการ Talk ที่ทำงานอยู่ Android จะใช้ TTS ของระบบภายในเครื่องเป็นทางเลือกสำรองเฉพาะเมื่อ RPC ดังกล่าวไม่พร้อมใช้งาน
- การเล่น MLX ภายในเครื่องของ macOS ใช้ตัวช่วย `openclaw-mlx-tts` ที่มาพร้อมกันเมื่อมี หรือใช้ไฟล์ปฏิบัติการใน `PATH` ตั้งค่า `OPENCLAW_MLX_TTS_BIN` ให้ชี้ไปยังไบนารีตัวช่วยแบบกำหนดเองระหว่างการพัฒนา
- ช่วงค่าของไดเรกทิฟเสียง (ElevenLabs): `stability`, `similarity` และ `style` ยอมรับ `0..1`; `speed` ยอมรับ `0.5..2`; `latency_tier` ยอมรับ `0..4`
