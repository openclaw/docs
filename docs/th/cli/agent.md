# `openclaw agent`

เรียกใช้หนึ่งรอบการทำงานของเอเจนต์ผ่าน Gateway หากคำขอไปยัง Gateway ล้มเหลว ระบบจะถอยกลับไปใช้เอเจนต์แบบฝังตัว ส่ง `--local` เพื่อบังคับใช้การทำงานแบบฝังตัวตั้งแต่ต้น

ระบุตัวเลือกเซสชันอย่างน้อยหนึ่งรายการ: `--to`, `--session-key`, `--session-id` หรือ `--agent`

ที่เกี่ยวข้อง: [เครื่องมือส่งของเอเจนต์](/th/tools/agent-send)

## ตัวเลือก

- `-m, --message <text>`: เนื้อหาข้อความ
- `--message-file <path>`: อ่านเนื้อหาข้อความจากไฟล์ UTF-8
- `-t, --to <dest>`: ผู้รับที่ใช้สร้างคีย์เซสชัน
- `--session-key <key>`: คีย์เซสชันที่ระบุอย่างชัดเจนเพื่อใช้กำหนดเส้นทาง
- `--session-id <id>`: รหัสเซสชันที่ระบุอย่างชัดเจน
- `--agent <id>`: รหัสเอเจนต์ มีผลแทนการผูกเส้นทาง
- `--model <id>`: แทนที่โมเดลสำหรับการทำงานครั้งนี้ (`provider/model` หรือรหัสโมเดล)
- `--thinking <level>`: ระดับการคิดของเอเจนต์ (`off`, `minimal`, `low`, `medium`, `high` รวมถึงระดับกำหนดเองที่ผู้ให้บริการรองรับ เช่น `xhigh`, `adaptive` หรือ `max`)
- `--verbose <on|off>`: บันทึกระดับความละเอียดสำหรับเซสชัน
- `--channel <channel>`: ช่องทางการส่ง หากไม่ระบุจะใช้ช่องทางของเซสชันหลัก
- `--reply-to <target>`: แทนที่ปลายทางการส่ง
- `--reply-channel <channel>`: แทนที่ช่องทางการส่ง
- `--reply-account <id>`: แทนที่บัญชีที่ใช้ส่ง
- `--local`: เรียกใช้เอเจนต์แบบฝังตัวโดยตรง (หลังจากโหลดรีจิสทรี Plugin ล่วงหน้า)
- `--deliver`: ส่งการตอบกลับไปยังช่องทาง/ปลายทางที่เลือก
- `--timeout <seconds>`: แทนที่กำหนดเวลาสิ้นสุดรอบการทำงานของเอเจนต์สำหรับคำสั่งนี้ (ค่าเริ่มต้น 600 หรือ `agents.defaults.timeoutSeconds`); `0` ปิดใช้งานกำหนดเวลาสิ้นสุดโดยรวม ค่าสำรอง 600 วินาทีเป็นของคำสั่ง CLI นี้ ไม่ใช่รอบการทำงาน Gateway ทั่วไป ซึ่งมีค่าเริ่มต้น 48 ชั่วโมง
- `--json`: แสดงผลเป็น JSON

## ตัวอย่าง

```bash
openclaw agent --to +15555550123 --message "อัปเดตสถานะ" --deliver
openclaw agent --agent ops --message "สรุปบันทึก"
openclaw agent --agent ops --message-file ./task.md
openclaw agent --agent ops --model openai/gpt-5.4 --message "สรุปบันทึก"
openclaw agent --session-key agent:ops:incident-42 --message "สรุปสถานะ"
openclaw agent --agent ops --session-key incident-42 --message "สรุปสถานะ"
openclaw agent --session-id 1234 --message "สรุปกล่องข้อความ" --thinking medium
openclaw agent --to +15555550123 --message "ติดตามบันทึก" --verbose on --json
openclaw agent --agent ops --message "สร้างรายงาน" --deliver --reply-channel slack --reply-to "#reports"
openclaw agent --agent ops --message "เรียกใช้ภายในเครื่อง" --local
```

## หมายเหตุ

- ระบุ `--message` หรือ `--message-file` เพียงรายการเดียวเท่านั้น `--message-file` จะตัด UTF-8 BOM ที่อยู่ด้านหน้าออกและคงเนื้อหาหลายบรรทัดไว้ โดยจะปฏิเสธไฟล์ที่ไม่ใช่ UTF-8 ที่ถูกต้อง ไฟล์ขนาดใหญ่กว่า 4 MiB จะถูกปฏิเสธก่อนส่งต่อ
- คำสั่งแบบทับ (เช่น `/compact`) ไม่สามารถเรียกใช้ผ่าน `--message` ได้ CLI จะปฏิเสธคำสั่งดังกล่าวและแนะนำให้ใช้คำสั่งเฉพาะโดยตรงแทน (`openclaw sessions compact <key>` สำหรับ Compaction)
- `--local` และการทำงานสำรองแบบฝังตัวเป็นการทำงานครั้งเดียว: ทรัพยากรลูปแบ็ก MCP ที่รวมมาให้และเซสชัน stdio ของ Claude แบบพร้อมใช้งานซึ่งเปิดสำหรับการทำงานครั้งนั้นจะถูกยุติหลังจากตอบกลับ เพื่อให้การเรียกใช้ด้วยสคริปต์ไม่ทิ้งโปรเซสลูกภายในเครื่องไว้ ส่วนการทำงานผ่าน Gateway จะเก็บทรัพยากรลูปแบ็ก MCP ที่ Gateway เป็นเจ้าของไว้ภายใต้โปรเซส Gateway ที่กำลังทำงาน
- การทำงานแบบฝังตัวโดยลำพัง (`--local` และการถอยกลับของการขนส่ง) จะไม่ยอมใช้เซสชันหลักที่มีอยู่ซ้ำขณะที่การกู้คืนหลังเริ่มระบบใหม่ยังค้างอยู่ ให้เรียกใช้รอบการทำงานผ่าน Gateway ที่ทำงานเป็นปกติ หรือรีเซ็ตที่ Gateway ด้วย `/new` หรือ `/reset`; โปรเซสแบบฝังตัวที่แยกต่างหากไม่สามารถประสานเจ้าของการกู้คืนนั้นกับตัวสแกน Gateway ได้อย่างปลอดภัย
- เมื่อใช้ `--agent`, `--channel` และ `--to` ร่วมกัน การกำหนดเส้นทางเซสชันจะใช้ผู้รับมาตรฐานของช่องทางและ `session.dmScope` ช่องทางที่มีข้อมูลประจำตัวผู้รับเฉพาะขาออกซึ่งคงที่จะใช้เซสชันที่ผู้ให้บริการเป็นเจ้าของและแยกจากเซสชันหลักของเอเจนต์ `--reply-channel` และ `--reply-account` มีผลต่อการส่งเท่านั้น
- `--session-key` เลือกคีย์เซสชันที่ระบุอย่างชัดเจน คีย์ที่ขึ้นต้นด้วยเอเจนต์ต้องใช้ `agent:<agent-id>:<session-key>` และเมื่อระบุทั้งสองรายการ `--agent` ต้องตรงกับรหัสเอเจนต์ของคีย์ คีย์เปล่าที่ไม่ใช่ค่าพิเศษจะจำกัดขอบเขตไว้ที่ `--agent` หากระบุไว้ มิฉะนั้นจะใช้เอเจนต์เริ่มต้นที่กำหนดค่าไว้ ตัวอย่างเช่น `--agent ops --session-key incident-42` จะกำหนดเส้นทางไปยัง `agent:ops:incident-42` คีย์ค่าตรงตัว `global` และ `unknown` จะไม่มีขอบเขตเฉพาะต่อเมื่อไม่ได้ระบุ `--agent` เท่านั้น
- `--json` สงวน stdout ไว้สำหรับการตอบกลับ JSON ส่วนข้อมูลวินิจฉัยจาก Gateway, Plugin และการถอยกลับแบบฝังตัวจะไปยัง stderr เพื่อให้สคริปต์แยกวิเคราะห์ stdout ได้โดยตรง
- JSON จากการถอยกลับแบบฝังตัวมี `meta.transport: "embedded"` และ `meta.fallbackFrom: "gateway"` เพื่อให้สคริปต์ตรวจพบการทำงานสำรองได้
- หาก Gateway ยอมรับการทำงานแล้ว แต่ CLI หมดเวลาขณะรอการตอบกลับสุดท้าย การถอยกลับแบบฝังตัวจะใช้รหัสเซสชัน/การทำงาน `gateway-fallback-*` ใหม่ และรายงาน `meta.fallbackReason: "gateway_timeout"` พร้อมฟิลด์เซสชันสำรอง แทนที่จะแข่งขันกับทรานสคริปต์ที่ Gateway เป็นเจ้าของหรือแทนที่เซสชันเดิมโดยไม่แจ้ง
- `SIGTERM`/`SIGINT` จะขัดจังหวะคำขอผ่าน Gateway ที่กำลังรอ หาก Gateway ยอมรับการทำงานแล้ว CLI จะส่ง `chat.abort` สำหรับรหัสการทำงานนั้นก่อนออกด้วย `--local` และการทำงานสำรองแบบฝังตัวจะได้รับสัญญาณเดียวกัน แต่จะไม่ส่ง `chat.abort` หากคีย์ภายในสำหรับขจัดการทำงานซ้ำมีการทำงานที่ยังใช้งานอยู่สำหรับเซสชันนี้แล้ว การตอบกลับจะรายงาน `status: "in_flight"` และ CLI ที่ไม่ใช่ JSON จะแสดงข้อมูลวินิจฉัยทาง stderr แทนการตอบกลับว่าง สำหรับตัวห่อ cron/systemd ภายนอก ให้คงกลไกบังคับยุติสำรอง เช่น `timeout -k 60 600 openclaw agent ...` เพื่อให้ตัวควบคุมสามารถเก็บกวาดโปรเซสได้หากการปิดระบบไม่สามารถรอให้งานสิ้นสุด
- เมื่อคำสั่งนี้ทริกเกอร์การสร้าง `models.json` ใหม่ ข้อมูลรับรองของผู้ให้บริการที่จัดการด้วย SecretRef จะถูกบันทึกเป็นเครื่องหมายที่ไม่ใช่ข้อมูลลับ (เช่น ชื่อตัวแปรสภาพแวดล้อม, `secretref-env:ENV_VAR_NAME` หรือ `secretref-managed`) และจะไม่บันทึกข้อความธรรมดาของข้อมูลลับที่ผ่านการแปลงค่าแล้ว การเขียนเครื่องหมายมาจากสแนปช็อตการกำหนดค่าต้นทางที่ใช้งานอยู่ ไม่ใช่ค่าข้อมูลลับของรันไทม์ที่ผ่านการแปลงค่าแล้ว

## สถานะการส่งใน JSON

เมื่อใช้ `--json --deliver` การตอบกลับ JSON ของ CLI จะมี `deliveryStatus` ที่ระดับบนสุด เพื่อให้สคริปต์แยกความแตกต่างระหว่างการส่งสำเร็จ การระงับ การส่งบางส่วน และการส่งล้มเหลวได้:

```json
{
  "payloads": [{ "text": "รายงานพร้อมแล้ว", "mediaUrl": null }],
  "meta": { "durationMs": 1200 },
  "deliveryStatus": {
    "requested": true,
    "attempted": true,
    "status": "sent",
    "succeeded": true,
    "resultCount": 1
  }
}
```

การตอบกลับ CLI ผ่าน Gateway จะคงรูปแบบผลลัพธ์ดิบของ Gateway ไว้ที่ `result.deliveryStatus` ด้วย

`deliveryStatus.status` เป็นหนึ่งในค่าต่อไปนี้:

| สถานะ           | ความหมาย                                                                                                                                    |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `sent`           | การส่งเสร็จสมบูรณ์                                                                                                                        |
| `suppressed`     | มีการจงใจไม่ส่ง (เช่น ฮุกส่งข้อความยกเลิก หรือไม่มีผลลัพธ์ที่มองเห็นได้) เป็นสถานะสิ้นสุดและจะไม่ลองใหม่ |
| `partial_failed` | ส่งเพย์โหลดอย่างน้อยหนึ่งรายการสำเร็จก่อนที่เพย์โหลดรายการถัดมาจะล้มเหลว                                                                                   |
| `failed`         | ไม่มีการส่งแบบคงทนที่เสร็จสมบูรณ์ หรือการตรวจสอบก่อนส่งล้มเหลว                                                                                   |

ฟิลด์ทั่วไป:

- `requested`: เป็น `true` เสมอเมื่อมีออบเจ็กต์นี้
- `attempted`: เป็น `true` เมื่อเส้นทางการส่งแบบคงทนทำงานแล้ว และเป็น `false` เมื่อการตรวจสอบก่อนส่งล้มเหลวหรือไม่มีเพย์โหลดที่มองเห็นได้
- `succeeded`: `true`, `false` หรือ `"partial"`; `"partial"` ใช้คู่กับ `status: "partial_failed"`
- `reason`: เหตุผลแบบ snake-case ตัวพิมพ์เล็กจากการส่งแบบคงทนหรือการตรวจสอบก่อนส่ง ค่าที่รู้จักประกอบด้วย `cancelled_by_message_sending_hook`, `no_visible_payload`, `no_visible_result`, `channel_resolved_to_internal`, `unknown_channel`, `invalid_delivery_target` และ `no_delivery_target`; การส่งแบบคงทนที่ล้มเหลวอาจรายงานขั้นตอนที่ล้มเหลวด้วย ให้ถือว่าค่าที่ไม่รู้จักเป็นข้อมูลทึบ เนื่องจากชุดค่านี้สามารถขยายได้
- `resultCount`: จำนวนผลลัพธ์การส่งของช่องทาง หากมี
- `sentBeforeError`: เป็น `true` เมื่อความล้มเหลวบางส่วนส่งเพย์โหลดอย่างน้อยหนึ่งรายการได้ก่อนเกิดข้อผิดพลาด
- `error`: เป็น `true` สำหรับการส่งที่ล้มเหลวหรือล้มเหลวบางส่วน
- `errorMessage`: มีเฉพาะเมื่อบันทึกข้อความข้อผิดพลาดจากการส่งที่เป็นสาเหตุได้ การตรวจสอบก่อนส่งที่ล้มเหลวจะมี `error`/`reason` แต่ไม่มี `errorMessage`
- `payloadOutcomes`: ผลลัพธ์ต่อเพย์โหลดที่ไม่บังคับ พร้อม `index`, `status`, `reason`, `resultCount`, `error`, `stage`, `sentBeforeError` หรือข้อมูลเมตาของฮุกเมื่อมี
