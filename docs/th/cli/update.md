# `openclaw update`

อัปเดต OpenClaw อย่างปลอดภัยและสลับระหว่างช่อง stable/beta/dev

หากคุณติดตั้งผ่าน **npm/pnpm/bun** (ติดตั้งแบบ global ไม่มีข้อมูลเมตา git)
การอัปเดตจะเกิดขึ้นผ่านขั้นตอนตัวจัดการแพ็กเกจใน [การอัปเดต](/th/install/updating)

## การใช้งาน

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## ตัวเลือก

- `--no-restart`: ข้ามการรีสตาร์ตบริการ Gateway หลังจากอัปเดตสำเร็จ การอัปเดตผ่านตัวจัดการแพ็กเกจที่รีสตาร์ต Gateway จะตรวจสอบว่าบริการที่รีสตาร์ตรายงานเวอร์ชันที่อัปเดตตามที่คาดไว้ก่อนคำสั่งจะสำเร็จ
- `--channel <stable|beta|dev>`: ตั้งค่าช่องอัปเดต (git + npm; เก็บถาวรใน config)
- `--tag <dist-tag|version|spec>`: แทนที่เป้าหมายแพ็กเกจสำหรับการอัปเดตครั้งนี้เท่านั้น สำหรับการติดตั้งแบบแพ็กเกจ `main` จะแมปไปที่ `github:openclaw/openclaw#main`
- `--dry-run`: แสดงตัวอย่างการดำเนินการอัปเดตที่วางแผนไว้ (ช่อง/แท็ก/เป้าหมาย/ขั้นตอนรีสตาร์ต) โดยไม่เขียน config, ติดตั้ง, ซิงก์ plugins หรือรีสตาร์ต
- `--json`: พิมพ์ JSON `UpdateRunResult` ที่เครื่องอ่านได้ รวมถึง
  `postUpdate.plugins.integrityDrifts` เมื่อตรวจพบความคลาดเคลื่อนของอาร์ติแฟกต์ Plugin npm
  ระหว่างการซิงก์ Plugin หลังอัปเดต
- `--timeout <seconds>`: เวลาหมดอายุต่อขั้นตอน (ค่าเริ่มต้นคือ 1800s)
- `--yes`: ข้ามพรอมต์ยืนยัน (เช่น การยืนยันการดาวน์เกรด)

<Warning>
การดาวน์เกรดต้องมีการยืนยัน เพราะเวอร์ชันเก่าอาจทำให้การกำหนดค่าเสียหายได้
</Warning>

## `update status`

แสดงช่องอัปเดตที่ใช้งานอยู่ + แท็ก/สาขา/SHA ของ git (สำหรับเช็กเอาต์จากซอร์ส) รวมถึงความพร้อมใช้งานของอัปเดต

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

ตัวเลือก:

- `--json`: พิมพ์ JSON สถานะที่เครื่องอ่านได้
- `--timeout <seconds>`: เวลาหมดอายุสำหรับการตรวจสอบ (ค่าเริ่มต้นคือ 3s)

## `update wizard`

ขั้นตอนแบบโต้ตอบเพื่อเลือกช่องอัปเดตและยืนยันว่าจะรีสตาร์ต Gateway
หลังจากอัปเดตหรือไม่ (ค่าเริ่มต้นคือรีสตาร์ต) หากคุณเลือก `dev` โดยไม่มีเช็กเอาต์ git ระบบจะ
เสนอให้สร้างเช็กเอาต์หนึ่งรายการ

ตัวเลือก:

- `--timeout <seconds>`: เวลาหมดอายุสำหรับแต่ละขั้นตอนอัปเดต (ค่าเริ่มต้น `1800`)

## สิ่งที่คำสั่งทำ

เมื่อคุณสลับช่องอย่างชัดเจน (`--channel ...`) OpenClaw จะรักษา
วิธีติดตั้งให้สอดคล้องด้วย:

- `dev` → ตรวจให้แน่ใจว่ามีเช็กเอาต์ git (ค่าเริ่มต้น: `~/openclaw`, แทนที่ด้วย `OPENCLAW_GIT_DIR`),
  อัปเดตเช็กเอาต์นั้น และติดตั้ง CLI แบบ global จากเช็กเอาต์นั้น
- `stable` → ติดตั้งจาก npm โดยใช้ `latest`
- `beta` → เลือกใช้ dist-tag `beta` ของ npm ก่อน แต่จะย้อนกลับไปใช้ `latest` เมื่อ beta
  ไม่มีอยู่หรือเก่ากว่ารุ่น stable ปัจจุบัน

ตัวอัปเดตอัตโนมัติของแกน Gateway (เมื่อเปิดใช้ผ่าน config) จะเปิดเส้นทางอัปเดต CLI
นอกตัวจัดการคำขอ Gateway ที่กำลังทำงานอยู่ การอัปเดตผ่านตัวจัดการแพ็กเกจของ control-plane `update.run`
จะบังคับให้รีสตาร์ตการอัปเดตแบบไม่เลื่อนเวลาและไม่มีคูลดาวน์หลังจากสลับแพ็กเกจแล้ว
เพราะกระบวนการ Gateway เก่าอาจยังมีชิ้นส่วนในหน่วยความจำที่ชี้ไปยัง
ไฟล์ที่แพ็กเกจใหม่ลบออกไปแล้ว

สำหรับการติดตั้งผ่านตัวจัดการแพ็กเกจ `openclaw update` จะแก้หาเวอร์ชันแพ็กเกจ
เป้าหมายก่อนเรียกใช้ตัวจัดการแพ็กเกจ การติดตั้ง npm แบบ global ใช้การติดตั้งแบบ staging:
OpenClaw ติดตั้งแพ็กเกจใหม่ลงใน prefix ชั่วคราวของ npm, ตรวจสอบ
รายการสินค้าคงคลัง `dist` ที่บรรจุอยู่ที่นั่น แล้วสลับแผนผังแพ็กเกจที่สะอาดนั้นเข้าไปใน
prefix global จริง หากการตรวจสอบล้มเหลว doctor หลังอัปเดต, การซิงก์ Plugin และ
งานรีสตาร์ตจะไม่ทำงานจากแผนผังที่น่าสงสัยนั้น แม้ว่าเวอร์ชันที่ติดตั้ง
จะตรงกับเป้าหมายอยู่แล้ว คำสั่งจะรีเฟรชการติดตั้งแพ็กเกจ global
จากนั้นเรียกใช้การซิงก์ Plugin, การรีเฟรชการเติมคำสั่งแกนให้สมบูรณ์ และงานรีสตาร์ต สิ่งนี้
ทำให้ sidecar ที่บรรจุมาและระเบียน Plugin ที่ช่องเป็นเจ้าของสอดคล้องกับ
บิลด์ OpenClaw ที่ติดตั้งไว้ ขณะที่ปล่อยให้การสร้างการเติมคำสั่ง Plugin ทั้งหมดใหม่เป็นหน้าที่ของ
การเรียกใช้ `openclaw completion --write-state` อย่างชัดเจน

เมื่อมีการติดตั้งบริการ Gateway ภายในเครื่องแบบจัดการไว้และเปิดใช้การรีสตาร์ต
การอัปเดตผ่านตัวจัดการแพ็กเกจจะหยุดบริการที่กำลังทำงานก่อนแทนที่แผนผังแพ็กเกจ
จากนั้นรีเฟรชข้อมูลเมตาของบริการจากการติดตั้งที่อัปเดตแล้ว รีสตาร์ต
บริการ และตรวจสอบว่า Gateway ที่รีสตาร์ตรายงานเวอร์ชันตามที่คาดไว้ ด้วย
`--no-restart` การแทนที่แพ็กเกจยังคงทำงาน แต่บริการที่จัดการไว้จะไม่ถูก
หยุดหรือรีสตาร์ต ดังนั้น Gateway ที่กำลังทำงานอาจยังใช้โค้ดเก่าจนกว่าคุณจะรีสตาร์ต
ด้วยตนเอง

## ขั้นตอนเช็กเอาต์ git

### การเลือกช่อง

- `stable`: เช็กเอาต์แท็กล่าสุดที่ไม่ใช่ beta จากนั้น build และ doctor
- `beta`: เลือกแท็ก `-beta` ล่าสุดก่อน แต่จะย้อนกลับไปใช้แท็ก stable ล่าสุดเมื่อ beta ไม่มีอยู่หรือเก่ากว่า
- `dev`: เช็กเอาต์ `main` จากนั้น fetch และ rebase

### ขั้นตอนอัปเดต

<Steps>
  <Step title="ตรวจสอบ worktree ที่สะอาด">
    ต้องไม่มีการเปลี่ยนแปลงที่ยังไม่ได้ commit
  </Step>
  <Step title="สลับช่อง">
    สลับไปยังช่องที่เลือก (แท็กหรือสาขา)
  </Step>
  <Step title="ดึงข้อมูลจาก upstream">
    เฉพาะ Dev เท่านั้น
  </Step>
  <Step title="บิลด์ preflight (เฉพาะ dev)">
    เรียกใช้ lint และบิลด์ TypeScript ใน worktree ชั่วคราว หาก tip ล้มเหลว จะย้อนกลับได้สูงสุด 10 commits เพื่อค้นหาบิลด์ที่สะอาดล่าสุด
  </Step>
  <Step title="Rebase">
    Rebase ไปยัง commit ที่เลือก (เฉพาะ dev)
  </Step>
  <Step title="ติดตั้ง dependencies">
    ใช้ตัวจัดการแพ็กเกจของ repo สำหรับเช็กเอาต์ pnpm ตัวอัปเดตจะ bootstrap `pnpm` เมื่อต้องการ (ผ่าน `corepack` ก่อน แล้วจึงใช้ fallback ชั่วคราว `npm install pnpm@10`) แทนการเรียกใช้ `npm run build` ภายใน workspace pnpm
  </Step>
  <Step title="บิลด์ UI ควบคุม">
    บิลด์ gateway และ UI ควบคุม
  </Step>
  <Step title="เรียกใช้ doctor">
    `openclaw doctor` ทำงานเป็นการตรวจสอบ safe-update ขั้นสุดท้าย
  </Step>
  <Step title="ซิงก์ plugins">
    ซิงก์ plugins ไปยังช่องที่ใช้งานอยู่ Dev ใช้ plugins ที่บันเดิลมา ส่วน stable และ beta ใช้ npm อัปเดต plugins ที่ติดตั้งจาก npm
  </Step>
</Steps>

<Warning>
หากการอัปเดต Plugin npm ที่ pin แบบแน่นอนแก้ไปยังอาร์ติแฟกต์ที่มี integrity ต่างจากระเบียนการติดตั้งที่จัดเก็บไว้ `openclaw update` จะยกเลิกการอัปเดตอาร์ติแฟกต์ Plugin นั้นแทนการติดตั้ง ติดตั้งใหม่หรืออัปเดต Plugin อย่างชัดเจนเฉพาะหลังจากตรวจสอบแล้วว่าคุณเชื่อถืออาร์ติแฟกต์ใหม่นั้น
</Warning>

<Note>
ความล้มเหลวในการซิงก์ Plugin หลังอัปเดตจะทำให้ผลการอัปเดตล้มเหลวและหยุดงานติดตามผลการรีสตาร์ต แก้ข้อผิดพลาดการติดตั้งหรืออัปเดต Plugin แล้วเรียกใช้ `openclaw update` อีกครั้ง

เมื่อ Gateway ที่อัปเดตแล้วเริ่มทำงาน การโหลด Plugin จะเป็นแบบตรวจสอบเท่านั้น: startup จะไม่เรียกใช้ตัวจัดการแพ็กเกจหรือเปลี่ยนแปลงแผนผัง dependency การรีสตาร์ต `update.run` ของตัวจัดการแพ็กเกจจะข้ามการเลื่อนเวลาขณะ idle และคูลดาวน์การรีสตาร์ตตามปกติหลังจากสลับแผนผังแพ็กเกจแล้ว เพื่อให้กระบวนการเก่าไม่สามารถ lazy-load ชิ้นส่วนที่ถูกลบออกได้ต่อไป

หาก bootstrap ของ pnpm ยังล้มเหลว ตัวอัปเดตจะหยุดตั้งแต่ต้นพร้อมข้อผิดพลาดเฉพาะของตัวจัดการแพ็กเกจ แทนที่จะลอง `npm run build` ภายในเช็กเอาต์
</Note>

## ชวเลข `--update`

`openclaw --update` จะเขียนใหม่เป็น `openclaw update` (มีประโยชน์สำหรับ shell และสคริปต์ launcher)
