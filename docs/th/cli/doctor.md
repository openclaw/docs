# `openclaw doctor`

การตรวจสอบสถานะและการแก้ไขด่วนสำหรับ Gateway, ช่องทาง, Plugin, Skills, การกำหนดเส้นทางโมเดล, สถานะภายในเครื่อง และการย้ายข้อมูลการกำหนดค่า ใช้เมื่อใดก็ตามที่ระบบทำงานไม่เป็นไปตามที่คาดไว้และต้องการใช้คำสั่งเดียวเพื่ออธิบายสิ่งที่ผิดปกติ

เนื้อหาที่เกี่ยวข้อง:

- การแก้ไขปัญหา: [การแก้ไขปัญหา](/th/gateway/troubleshooting)
- การตรวจสอบความปลอดภัย: [ความปลอดภัย](/th/gateway/security)

## รูปแบบการทำงาน

Doctor มีรูปแบบการทำงาน 5 แบบ:

| รูปแบบการทำงาน                   | คำสั่ง                                   | ลักษณะการทำงาน                                                                        |
| ------------------------- | ----------------------------------------- | ------------------------------------------------------------------------------- |
| ตรวจสอบ                   | `openclaw doctor`                         | การตรวจสอบที่มุ่งให้มนุษย์อ่านและพรอมต์ที่ให้คำแนะนำ                                       |
| ซ่อมแซม                    | `openclaw doctor --fix`                   | ใช้การซ่อมแซมที่รองรับ โดยใช้พรอมต์ เว้นแต่จะซ่อมแซมแบบไม่โต้ตอบได้อย่างปลอดภัย |
| ตรวจวิเคราะห์                      | `openclaw doctor --lint`                  | ผลการตรวจพบแบบมีโครงสร้างและอ่านอย่างเดียวสำหรับ CI การตรวจสอบเบื้องต้น และเกตการตรวจทาน              |
| บำรุงรักษา SQLite ที่ใช้ร่วมกัน | `openclaw doctor --state-sqlite compact`  | สร้างเช็กพอยต์ กระชับ และตรวจสอบฐานข้อมูลสถานะที่ใช้ร่วมกันซึ่งเป็นแหล่งข้อมูลหลักอย่างชัดเจน   |
| ย้ายข้อมูล SQLite ของเซสชัน  | `openclaw doctor --session-sqlite <mode>` | ตรวจสอบ นำเข้า ตรวจสอบความถูกต้อง กระชับ กู้คืน หรือคืนค่าสถานะเซสชัน    |

ควรใช้ `--lint` เมื่อระบบอัตโนมัติต้องการผลลัพธ์ที่เสถียร ควรใช้ `--fix` เมื่อผู้ปฏิบัติงานต้องการให้ doctor แก้ไขการกำหนดค่าหรือสถานะ

## ตัวอย่าง

```bash
openclaw doctor
openclaw doctor --lint
openclaw doctor --lint --json
openclaw doctor --lint --severity-min warning
openclaw doctor --lint --all
openclaw doctor --lint --allow-exec
openclaw doctor --deep
openclaw doctor --fix
openclaw doctor --fix --non-interactive
openclaw doctor --generate-gateway-token
openclaw doctor --post-upgrade
openclaw doctor --post-upgrade --json
openclaw doctor --state-sqlite compact
openclaw doctor --state-sqlite compact --json
openclaw doctor --session-sqlite inspect --session-sqlite-all-agents
openclaw doctor --session-sqlite dry-run --session-sqlite-agent main --json
openclaw doctor --session-sqlite import --session-sqlite-all-agents
openclaw doctor --session-sqlite validate --session-sqlite-all-agents --json
openclaw doctor --session-sqlite compact --session-sqlite-all-agents
openclaw doctor --session-sqlite recover --github-issue
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

สำหรับสิทธิ์เฉพาะช่องทาง ให้ใช้โพรบของช่องทางแทน `doctor`:

```bash
openclaw channels capabilities --channel discord --target channel:<channel-id>
openclaw channels status --probe
```

`channels capabilities` รายงานสิทธิ์ที่มีผลจริงของบอตสำหรับเป้าหมายช่องทางที่ระบุ `channels status --probe` ตรวจสอบช่องทางที่กำหนดค่าไว้ทั้งหมดและเป้าหมายการเข้าร่วมเสียงอัตโนมัติ

## ตัวเลือก

| ตัวเลือก                          | ผล                                                                                                                                                                                  |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--no-workspace-suggestions`    | ปิดใช้คำแนะนำเกี่ยวกับหน่วยความจำ/การค้นหาของพื้นที่ทำงาน                                                                                                                                            |
| `--yes`                         | ยอมรับค่าเริ่มต้นโดยไม่แสดงพรอมต์                                                                                                                                                      |
| `--repair` / `--fix`            | ใช้การซ่อมแซมที่แนะนำซึ่งไม่เกี่ยวกับบริการโดยไม่แสดงพรอมต์ (`--fix` เป็นชื่อเรียกแทน) การติดตั้ง/เขียนบริการ Gateway ใหม่ยังคงต้องมีการยืนยันแบบโต้ตอบหรือคำสั่ง `gateway` ที่ระบุชัดเจน |
| `--force`                       | ใช้การซ่อมแซมเชิงรุก รวมถึงเขียนทับการกำหนดค่าบริการแบบกำหนดเอง                                                                                                                  |
| `--non-interactive`             | ทำงานโดยไม่มีพรอมต์ โดยดำเนินการเฉพาะการย้ายข้อมูลที่ปลอดภัยและการซ่อมแซมที่ไม่เกี่ยวกับบริการ                                                                                                                      |
| `--generate-gateway-token`      | สร้างและกำหนดค่าโทเค็น Gateway                                                                                                                                                 |
| `--allow-exec`                  | อนุญาตให้ doctor เรียกใช้ SecretRefs ของ `exec` ที่กำหนดค่าไว้ขณะตรวจสอบข้อมูลลับ                                                                                                           |
| `--deep`                        | สแกนบริการระบบเพื่อค้นหาการติดตั้ง Gateway เพิ่มเติม และรายงานการส่งต่องานรีสตาร์ตของตัวควบคุม Gateway ล่าสุด                                                                                     |
| `--lint`                        | เรียกใช้การตรวจสอบสถานะที่ปรับให้ทันสมัยในโหมดอ่านอย่างเดียวและแสดงผลการตรวจวินิจฉัย                                                                                                            |
| `--post-upgrade`                | เรียกใช้โพรบความเข้ากันได้ของ Plugin หลังการอัปเกรด ผลการตรวจพบจะส่งไปยัง stdout และใช้รหัสออก 1 หากมีผลการตรวจพบระดับข้อผิดพลาดอย่างน้อยหนึ่งรายการ                                                                 |
| `--state-sqlite <mode>`         | เรียกใช้การบำรุงรักษา SQLite ของสถานะที่ใช้ร่วมกันอย่างชัดเจน โหมดเดียวคือ `compact`                                                                                                               |
| `--session-sqlite <mode>`       | เรียกใช้โหมดการย้ายข้อมูล SQLite ของเซสชันตามเป้าหมาย: `inspect`, `dry-run`, `import`, `validate`, `compact`, `recover` หรือ `restore`                                                         |
| `--session-sqlite-store <path>` | เมื่อใช้ร่วมกับ `--session-sqlite`: เลือกพาธที่เก็บ `sessions.json` แบบเดิมหนึ่งพาธ                                                                                                                  |
| `--session-sqlite-agent <id>`   | เมื่อใช้ร่วมกับ `--session-sqlite`: เลือกเอเจนต์ที่กำหนดค่าไว้หนึ่งรายการ                                                                                                                                   |
| `--session-sqlite-all-agents`   | เมื่อใช้ร่วมกับ `--session-sqlite`: เลือกที่เก็บเอเจนต์ที่กำหนดค่าไว้และที่ค้นพบ                                                                                                                 |
| `--github-issue`                | เมื่อใช้ร่วมกับ `--session-sqlite recover`: จัดเตรียมรายงานปัญหา openclaw/openclaw ที่ลบข้อมูลอ่อนไหวแล้ว โดย doctor จะสร้างรายงานด้วย `gh` หลังจาก `--yes` หรือได้รับการยืนยันแบบโต้ตอบ                             |
| `--json`                        | เมื่อใช้ร่วมกับ `--lint`: ผลการตรวจพบในรูปแบบ JSON เมื่อใช้ร่วมกับ `--post-upgrade`: `{ probesRun, findings }` เมื่อใช้ร่วมกับ `--state-sqlite` หรือ `--session-sqlite`: รายงานการบำรุงรักษาในรูปแบบ JSON                            |
| `--severity-min <level>`        | เมื่อใช้ร่วมกับ `--lint`: ตัดผลการตรวจพบที่ต่ำกว่า `info`, `warning` หรือ `error`                                                                                                                       |
| `--all`                         | เมื่อใช้ร่วมกับ `--lint`: เรียกใช้การตรวจสอบที่ลงทะเบียนไว้ทั้งหมด รวมถึงการตรวจสอบแบบเลือกเข้าร่วมซึ่งไม่รวมอยู่ในชุดเริ่มต้น                                                                                        |
| `--skip <id>`                   | เมื่อใช้ร่วมกับ `--lint`: ข้ามรหัสการตรวจสอบหนึ่งรายการ ระบุซ้ำได้                                                                                                                                             |
| `--only <id>`                   | เมื่อใช้ร่วมกับ `--lint`: เรียกใช้เฉพาะรหัสการตรวจสอบที่กำหนด ระบุซ้ำได้                                                                                                                              |

`--severity-min`, `--all`, `--only` และ `--skip` จะยอมรับเฉพาะเมื่อใช้ร่วมกับ `--lint` เท่านั้น ส่วน `--json` จะยอมรับเมื่อใช้ร่วมกับ `--lint`, `--post-upgrade`, `--state-sqlite` และ `--session-sqlite`

## โหมดตรวจวิเคราะห์

`openclaw doctor --lint` เป็นแบบอ่านอย่างเดียว: ไม่มีพรอมต์ ไม่มีการซ่อมแซม และไม่มีการเขียนการกำหนดค่า/สถานะใหม่

```bash
openclaw doctor --lint
openclaw doctor --lint --severity-min warning
openclaw doctor --lint --json
openclaw doctor --lint --all
openclaw doctor --lint --allow-exec
openclaw doctor --lint --only core/doctor/gateway-config --json
openclaw doctor --lint --only core/doctor/local-audio-acceleration --severity-min info
```

ผลลัพธ์สำหรับมนุษย์มีขนาดกะทัดรัด:

```text
doctor --lint: เรียกใช้การตรวจสอบ 6 รายการ พบผลการตรวจ 1 รายการ
  [warning] core/doctor/gateway-config gateway.mode - ยังไม่ได้ตั้งค่า gateway.mode การเริ่มต้น Gateway จะถูกบล็อก
    วิธีแก้ไข: เรียกใช้ `openclaw configure` และตั้งค่าโหมด Gateway (local/remote) หรือใช้ `openclaw config set gateway.mode local`
```

ผลลัพธ์ JSON เป็นอินเทอร์เฟซสำหรับสคริปต์:

```json
{
  "ok": false,
  "checksRun": 5,
  "checksSkipped": 0,
  "findings": [
    {
      "checkId": "core/doctor/gateway-config",
      "severity": "warning",
      "message": "ยังไม่ได้ตั้งค่า gateway.mode การเริ่มต้น Gateway จะถูกบล็อก",
      "path": "gateway.mode",
      "fixHint": "เรียกใช้ `openclaw configure` และตั้งค่าโหมด Gateway (local/remote) หรือใช้ `openclaw config set gateway.mode local`"
    }
  ]
}
```

รหัสออก:

| รหัส | ความหมาย                                                       |
| ---- | ------------------------------------------------------------- |
| `0`  | ไม่มีผลการตรวจพบที่มีระดับเท่ากับหรือสูงกว่าเกณฑ์ความรุนแรงที่เลือก      |
| `1`  | มีผลการตรวจพบอย่างน้อยหนึ่งรายการที่ตรงตามเกณฑ์ที่เลือก            |
| `2`  | คำสั่ง/รันไทม์ล้มเหลวก่อนที่จะสร้างผลการตรวจวิเคราะห์ได้ |

`--severity-min` ควบคุมทั้งผลการตรวจพบที่จะแสดงและเกณฑ์รหัสออก: `openclaw doctor --lint --severity-min error` อาจไม่แสดงสิ่งใดและออกด้วยรหัส `0` แม้จะมีผลการตรวจพบ `info`/`warning` ที่มีระดับความรุนแรงต่ำกว่า

`--all` ควบคุมว่าจะเลือกการตรวจสอบใดก่อนกรองตามความรุนแรง การตรวจวิเคราะห์เริ่มต้นไม่รวมการตรวจสอบเชิงลึก เชิงประวัติ หรือที่มีแนวโน้มจะพบสิ่งตกค้างจากระบบเดิมที่ซ่อมแซมได้มากกว่า ให้ใช้ `--all` สำหรับรายการทั้งหมด `--only <id>` เป็นตัวเลือกที่แม่นยำที่สุดและสามารถเรียกใช้การตรวจสอบที่ลงทะเบียนไว้รายการใดก็ได้ตามรหัส

`core/doctor/local-audio-acceleration` รายงานคำสั่ง STT ภายในเครื่องที่เลือกโดยอัตโนมัติ หลักฐานแบ็กเอนด์ที่รองรับ/ร้องขอ/ตรวจพบซึ่งแยกจากกัน และลำดับการสำรองโดยไม่โหลดโมเดลเสียง โดยจะสร้างผลการตรวจพบระดับข้อมูล ดังนั้นให้รวม `--severity-min info` เพื่อแสดงผล

## การตรวจสอบสถานะแบบมีโครงสร้าง

การตรวจสอบ doctor แบบใหม่ใช้สัญญาแยกส่วนขนาดเล็ก:

```ts
detect(ctx, scope?) -> HealthFinding[]
repair?(ctx, findings) -> HealthRepairResult
```

`detect()` ขับเคลื่อน `doctor --lint` ส่วน `repair()` เป็นทางเลือกและทำงานเฉพาะภายใต้ `doctor --fix` / `doctor --repair` การตรวจสอบที่ยังไม่ได้ย้ายมาใช้รูปแบบนี้จะยังคงใช้โฟลว์การสนับสนุน doctor แบบเดิม

บริบทการซ่อมแซมสามารถมีคำขอ `dryRun`/`diff` ผลการซ่อมแซมสามารถส่งคืน `diffs` (การแก้ไขการกำหนดค่า/ไฟล์) และ `effects` (บริการ กระบวนการ แพ็กเกจ สถานะ หรือผลข้างเคียงอื่น ๆ) แบบมีโครงสร้าง เพื่อให้การตรวจสอบที่แปลงแล้วสามารถพัฒนาไปสู่ `doctor --fix --dry-run` ได้โดยไม่ย้ายการวางแผนการเปลี่ยนแปลงไปไว้ใน `detect()`

`repair()` รายงาน `status: "repaired" | "skipped" | "failed"` (หากละเว้นสถานะ หมายถึง `repaired`) เมื่อการซ่อมแซมส่งคืน `skipped` หรือ `failed` doctor จะรายงานสาเหตุและข้ามการตรวจสอบความถูกต้องสำหรับการตรวจนั้น หลังจากซ่อมแซมสำเร็จ doctor จะเรียกใช้ `detect()` อีกครั้งโดยจำกัดขอบเขตเฉพาะข้อค้นพบที่ซ่อมแซมแล้ว หากข้อค้นพบยังคงอยู่ doctor จะรายงานคำเตือนการซ่อมแซมแทนการถือว่าการเปลี่ยนแปลงเสร็จสมบูรณ์

ข้อค้นพบประกอบด้วย:

| ฟิลด์             | วัตถุประสงค์                                                |
| ----------------- | ------------------------------------------------------ |
| `checkId`         | ID ที่คงที่สำหรับตัวกรองข้าม/เฉพาะและรายการอนุญาตของ CI     |
| `severity`        | `info`, `warning` หรือ `error`                         |
| `message`         | คำอธิบายปัญหาที่มนุษย์อ่านเข้าใจได้                      |
| `path`            | การกำหนดค่า ไฟล์ หรือพาธเชิงตรรกะเมื่อมีข้อมูล          |
| `line` / `column` | ตำแหน่งต้นทางเมื่อมีข้อมูล                        |
| `ocPath`          | ที่อยู่ `oc://` ที่แม่นยำเมื่อการตรวจสามารถระบุได้ |
| `fixHint`         | การดำเนินการที่แนะนำสำหรับผู้ปฏิบัติงานหรือสรุปการซ่อมแซม           |

การตรวจ doctor หลักที่ปรับให้ทันสมัยยังคงเชื่อมโยงกับส่วนสนับสนุน doctor ตามลำดับที่เป็นเจ้าของพฤติกรรม `doctor` / `doctor --fix` สำหรับมนุษย์ รีจิสทรีสุขภาพแบบมีโครงสร้างที่ใช้ร่วมกันคือจุดขยาย: การตรวจที่มากับระบบและที่รองรับด้วย Plugin จะทำงานหลังการตรวจ doctor หลัก เมื่อแพ็กเกจเจ้าของลงทะเบียนการตรวจเหล่านั้นในพาธคำสั่งที่ใช้งานอยู่ `openclaw/plugin-sdk/health` เปิดเผยสัญญาเดียวกันสำหรับผู้เขียน Plugin

## การเลือกการตรวจ

```bash
openclaw doctor --lint --only core/doctor/gateway-config --json
openclaw doctor --lint --skip core/doctor/skills-readiness
openclaw doctor --lint --all --skip core/doctor/session-locks
```

`--only` และ `--skip` รับ ID การตรวจแบบเต็มและสามารถระบุซ้ำได้ หาก ID `--only` ไม่ได้ลงทะเบียน จะไม่มีการตรวจใดทำงานสำหรับ ID นั้น ใช้ `checksRun`/`checksSkipped` ในเอาต์พุตเพื่อยืนยันว่าเกตแบบเจาะจงเลือกการตรวจตามที่คาดไว้

## โหมดหลังการอัปเกรด

`openclaw doctor --post-upgrade` เรียกใช้โพรบความเข้ากันได้ของ Plugin สำหรับเชื่อมต่อการทำงานหลังการบิลด์หรืออัปเกรด ข้อค้นพบจะส่งไปยัง stdout โดยรหัสออกเป็น 1 หากข้อค้นพบใดมี `level: "error"` เพิ่ม `--json` เพื่อรับซองข้อมูลที่เครื่องอ่านได้ (`{ probesRun, findings }`) ซึ่งเหมาะสำหรับ CI, Skills ชุมชน `fork-upgrade` และเครื่องมือ smoke หลังการอัปเกรดอื่นๆ หากดัชนี Plugin ที่ติดตั้งหายไปหรือมีรูปแบบไม่ถูกต้อง โหมด JSON จะยังคงส่งซองข้อมูลพร้อมข้อค้นพบข้อผิดพลาด `plugin.index_unavailable`

การเริ่มต้นอิมเมจคอนเทนเนอร์เป็นข้อยกเว้นของขั้นตอนปกติ "เรียกใช้ doctor หลัง
การอัปเดต" เมื่อ `openclaw gateway run` เริ่มทำงานบน OpenClaw เวอร์ชันใหม่ ระบบจะ
ดำเนินการซ่อมแซมสถานะและ Plugin ที่ปลอดภัยก่อนรายงานว่าพร้อม หากการซ่อมแซมไม่สามารถ
เสร็จสิ้นได้อย่างปลอดภัย การเริ่มต้นจะออกและแจ้งให้เรียกใช้อิมเมจเดียวกันหนึ่งครั้งด้วย
`openclaw doctor --fix` กับสถานะ/การกำหนดค่าที่เมานต์ชุดเดิม ก่อนเริ่ม
คอนเทนเนอร์ตามปกติอีกครั้ง

## Compaction ของ SQLite สถานะที่ใช้ร่วมกัน

`openclaw doctor --state-sqlite compact` คือการบำรุงรักษาแบบออฟไลน์อย่างชัดเจนสำหรับ
ฐานข้อมูลสถานะที่ใช้ร่วมกันมาตรฐานที่
`<state-dir>/state/openclaw.sqlite` คำสั่งนี้ไม่รับพาธฐานข้อมูล
ตามอำเภอใจ ไม่เคยถูกเรียกใช้โดยการทำงานปกติของ Gateway และไม่ได้เป็นส่วนหนึ่งของ
`openclaw doctor --fix` คำสั่งจะรับล็อกความเป็นเจ้าของสถานะเดียวกับ
การเริ่มต้น Gateway และถือล็อกไว้ตลอดการตรวจสอบความถูกต้อง การทำ checkpoint, `VACUUM` และ
การตรวจสอบความสมบูรณ์ขั้นสุดท้าย คำสั่งจะปฏิเสธการทำงานขณะที่ Gateway หรือคำสั่งบำรุงรักษา
SQLite อื่นเป็นเจ้าของล็อกนั้น ล็อกสถานะยังคงทำงานเมื่อ
`OPENCLAW_ALLOW_MULTI_GATEWAY=1` ข้ามซิงเกิลตัน Gateway ต่อการกำหนดค่า ดังนั้น
เชลล์ของผู้ปฏิบัติงานจึงไม่จำเป็นต้องสืบทอดสภาพแวดล้อมของบริการ Gateway เพื่อให้
การบำรุงรักษาตรวจพบได้

หยุด Gateway และสร้างข้อมูลสำรองที่ตรวจสอบแล้วก่อน:

```bash
openclaw gateway stop
openclaw backup create --verify
openclaw doctor --state-sqlite compact --json
openclaw gateway start
```

คำสั่งนี้:

1. ต้องมีไฟล์ปกติที่พาธสถานะที่ใช้ร่วมกันมาตรฐาน หาก
   ฐานข้อมูลหายไป จะรายงานเป็น `skipped` และออกโดยสำเร็จ
2. ตรวจสอบเวอร์ชันสคีมาที่รองรับในปัจจุบันและ
   `schema_meta.role = "global"` ก่อนทำ checkpoint หรือเปลี่ยนแปลงไฟล์
3. ต้องมี `wal_checkpoint(TRUNCATE)` ที่ไม่อยู่ในสถานะไม่ว่าง หยุดกระบวนการ OpenClaw
   ที่เหลืออยู่ทั้งหมดแล้วลองอีกครั้งหาก checkpoint ไม่ว่าง
4. ตั้งค่า `auto_vacuum` เป็น `INCREMENTAL` เรียกใช้ `VACUUM` แบบเต็ม และทำ checkpoint
   อีกครั้ง
5. เรียกใช้ `quick_check`, `integrity_check` และ `foreign_key_check` จากนั้น
   ใช้สิทธิ์เฉพาะเจ้าของกับฐานข้อมูลและไฟล์ sidecar ของ SQLite อีกครั้ง

เอาต์พุต JSON รายงานขนาดฐานข้อมูลและ WAL, จำนวนหน้า freelist, ขนาดหน้า และ
ค่า `auto_vacuum` ก่อนและหลัง Compaction รวมถึงจำนวนไบต์ที่เรียกคืนและผลลัพธ์
`quick_check` และ `integrity_check` ระบบบังคับใช้ `foreign_key_check`
แบบ fail-closed และไม่มีฟิลด์ความสำเร็จแยกต่างหาก SQLite รายงาน `auto_vacuum` เป็น
`0` สำหรับไม่มี, `1` สำหรับเต็ม และ `2` สำหรับแบบเพิ่มทีละส่วน

Compaction จะล้มเหลวโดยไม่มีการเปลี่ยนแปลงเมื่อสคีมาเก่า ใหม่กว่า
บิลด์ OpenClaw ที่กำลังทำงาน หรือเป็นของฐานข้อมูลเอเจนต์ เรียกใช้
`openclaw doctor --fix` ก่อนสำหรับสคีมาสถานะที่ใช้ร่วมกันที่เก่ากว่า กู้คืน
ข้อมูลสำรองที่เข้ากันได้หรืออัปเกรด OpenClaw สำหรับสคีมาที่ใหม่กว่า

## การย้ายข้อมูล SQLite ของเซสชัน

OpenClaw นำเข้าแถวเซสชันแบบเดิมและประวัติทรานสคริปต์เข้าสู่ฐานข้อมูล
SQLite ของแต่ละเอเจนต์โดยอัตโนมัติระหว่างการเริ่มต้น Gateway และระหว่าง
`openclaw doctor --fix` `openclaw doctor --session-sqlite <mode>` คือ
เครื่องมือตรวจสอบและยืนยันแบบเจาะจงสำหรับการย้ายข้อมูลนั้น แถวเซสชันรันไทม์
ปัจจุบันอยู่ใน
`~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite` ไฟล์
`sessions.json` แบบเดิมเป็นแหล่งที่มาสำหรับการย้ายข้อมูล ไฟล์ JSONL ของทรานสคริปต์ที่ใช้งานอยู่จะ
ถูกนำเข้าและเก็บถาวรออกจากไดเรกทอรีเซสชันที่ใช้งานอยู่หลังจาก
นำเข้าสำเร็จ ไฟล์ JSONL ระดับคลังเก็บยังคงเป็นอาร์ติแฟกต์สำหรับการสนับสนุน ไม่ใช่
ทางเลือกสำรองของรันไทม์

โหมด:

| โหมด       | พฤติกรรม                                                                                                               |
| ---------- | ---------------------------------------------------------------------------------------------------------------------- |
| `inspect`  | อ่านจำนวนรายการแบบเดิมและ SQLite รวมถึงไฟล์ JSONL ที่ไม่มีการอ้างอิง โดยไม่นำเข้า                                       |
| `dry-run`  | แยกวิเคราะห์รายการแบบเดิมและไฟล์ JSONL ของทรานสคริปต์ นับแถวที่นำเข้าได้ และรายงานปัญหาโดยไม่เขียนแถว SQLite |
| `import`   | นำเข้ารายการแบบเดิมและเหตุการณ์ทรานสคริปต์เข้าสู่ SQLite สำหรับเป้าหมายที่เลือก                                      |
| `validate` | เปรียบเทียบแหล่งข้อมูลแบบเดิมที่เลือกกับแถว SQLite และจำนวนเหตุการณ์ทรานสคริปต์                                   |
| `compact`  | ทำ checkpoint และ VACUUM ฐานข้อมูล SQLite ของเอเจนต์ที่เลือกเพื่อเรียกคืนหน้าว่างหลังการลบจำนวนมากหรือการล้างคลังเก็บ    |
| `recover`  | กู้คืนการย้ายข้อมูลที่ล้มเหลวครั้งล่าสุด ตรวจสอบเป้าหมาย และเตรียมรายงานปัญหา GitHub ที่ผ่านการล้างข้อมูล            |
| `restore`  | กู้คืนอาร์ติแฟกต์ทรานสคริปต์ที่เก็บถาวรจากไฟล์ manifest การย้ายข้อมูลที่บันทึกไว้ โดยไม่ลบข้อมูล SQLite                  |

ตัวเลือก:

- ค่าเริ่มต้น: ที่เก็บของเอเจนต์เริ่มต้นที่กำหนดค่าไว้ เมื่อไฟล์ที่เก็บแบบเดิมนั้นมีอยู่
- `--session-sqlite-agent <id>`: เอเจนต์ที่กำหนดค่าไว้หนึ่งตัว
- `--session-sqlite-all-agents`: ที่เก็บของเอเจนต์ที่กำหนดค่าไว้ รวมถึงที่เก็บของเอเจนต์ที่ค้นพบ
- `--session-sqlite-store <path>`: พาธ `sessions.json` แบบเดิมที่ระบุอย่างชัดเจนหนึ่งพาธ

ลำดับการตรวจสอบด้วยตนเอง:

```bash
openclaw doctor --session-sqlite inspect --session-sqlite-all-agents
openclaw doctor --session-sqlite dry-run --session-sqlite-all-agents --json
openclaw doctor --session-sqlite import --session-sqlite-all-agents
openclaw doctor --session-sqlite validate --session-sqlite-all-agents --json
openclaw doctor --session-sqlite compact --session-sqlite-all-agents
openclaw doctor --session-sqlite recover --github-issue
```

สำรองข้อมูลไดเรกทอรีสถานะ OpenClaw ก่อนเรียกใช้ `import` กับการติดตั้งที่มี
ประวัติสำคัญ `validate` จะออกด้วยรหัสที่ไม่ใช่ศูนย์เมื่อรายการแบบเดิมที่เลือก
หายไปจาก SQLite, ID เซสชันแตกต่างกัน หรือจำนวนเหตุการณ์ทรานสคริปต์แตกต่างกัน
เมื่อใช้ `--session-sqlite-store <path>` ให้ตรวจสอบว่ารายงานมี
จำนวนเป้าหมายตามที่คาดไว้ พาธที่เก็บแบบระบุชัดเจนซึ่งไม่มีอยู่จะไม่เลือกเป้าหมายใดเลย

การลบใน SQLite จะเรียกคืนหน้าภายในฐานข้อมูลก่อน แต่ไม่ได้ทำให้
ไฟล์ฐานข้อมูลมีขนาดเล็กลงทันทีเสมอไป หลังจากลบหรือเก็บทรานสคริปต์ขนาดใหญ่
เข้าคลังแล้ว ให้เรียกใช้ `openclaw doctor --session-sqlite compact --session-sqlite-all-agents`
เพื่อทำ checkpoint ไฟล์ WAL, เรียกใช้ `VACUUM` และรายงานขนาดฐานข้อมูลและ WAL
ก่อน/หลัง Compaction ต้องใช้ไฟล์ปกติที่มีสคีมาเอเจนต์ปัจจุบัน
เมทาดาทาเจ้าของแบบคงทนของเอเจนต์ที่เลือก และไม่มีแฮนเดิลเปิดอยู่ในกระบวนการ
doctor โหมดทำลายข้อมูล `import`, `compact`, `recover` และ `restore`
จะถือล็อกความเป็นเจ้าของสถานะเดียวกับการเริ่มต้น Gateway ตลอดการดำเนินงาน
`inspect`, `dry-run` และ `validate` ยังคงเป็นแบบอ่านอย่างเดียวและไม่รับล็อกนั้น ให้หยุด
Gateway ก่อน โหมดทำลายข้อมูลจะล้มเหลวแทนที่จะแข่งขันกับการเขียนสดหรือ
แข่งขันกับคำสั่งบำรุงรักษาอื่น เป้าหมาย `--session-sqlite-store`
แบบทำลายข้อมูลต้องอยู่ภายในไดเรกทอรีสถานะที่ใช้งานอยู่ ตั้งค่า `OPENCLAW_STATE_DIR` เป็น
ไดเรกทอรีสถานะที่เป็นเจ้าของที่เก็บก่อนบำรุงรักษาการติดตั้งอื่น
เป้าหมายที่เป็นฮาร์ดลิงก์อยู่แล้วจะถูกปฏิเสธ เนื่องจากพาธอื่นสามารถใช้ inode ฐานข้อมูล
เดียวกันร่วมกันนอกไดเรกทอรีสถานะที่ล็อกไว้ การตรวจสอบความเป็นเจ้าของเดียวกันนี้
ครอบคลุมไฟล์ sidecar ของ SQLite ได้แก่ WAL, หน่วยความจำที่ใช้ร่วมกัน และ rollback journal

การนำเข้าแต่ละครั้งจะเขียนไฟล์ manifest ไว้ใต้
`~/.openclaw/session-sqlite-migration-runs/` ก่อนย้ายอาร์ติแฟกต์ทรานสคริปต์
เข้าสู่คลังเก็บ หากการเริ่มต้นรายงานว่าการย้ายข้อมูล SQLite ของเซสชันล้มเหลวหลังจาก
ย้ายอาร์ติแฟกต์แล้ว ให้เรียกใช้การกู้คืน:

```bash
openclaw doctor --session-sqlite recover --github-issue
```

การกู้คืนจะเลือกไฟล์ manifest การย้ายข้อมูลที่ล้มเหลวล่าสุด กู้คืนเฉพาะ
อาร์ติแฟกต์ที่เก็บถาวรของไฟล์ manifest ตรวจสอบเป้าหมายที่ได้รับผลกระทบ รีเฟรช
รายงาน `.failure.md` และ `.failure.json` ที่ผ่านการล้างข้อมูล และเตรียมเนื้อหา
ปัญหา GitHub ที่หลีกเลี่ยงเนื้อหาทรานสคริปต์ สภาพแวดล้อมดิบ ข้อมูลลับ และ
การกำหนดค่าที่ไม่มีขอบเขตจำกัด เมื่อไม่มีไฟล์ manifest การย้ายข้อมูลที่ล้มเหลว แต่ฐานข้อมูล
SQLite ของเอเจนต์ที่เลือกเสียหาย ไม่ใช่ฐานข้อมูล หรือมีไฟล์ sidecar ของ journal โดยไม่มี
ฐานข้อมูลหลัก การกู้คืนจะคัดลอกชุดไฟล์ทั้งหมดไปยังไดเรกทอรีตรวจสอบ
ชั่วคราว SQLite สามารถย้อนกลับ hot journal ที่ถูกต้องในสำเนาที่ใช้แล้วทิ้งนั้น
ก่อนเรียกใช้ `quick_check`, `integrity_check` และ `foreign_key_check` ขณะที่
ไฟล์นิติวิทยาศาสตร์ต้นฉบับยังคงไม่ถูกแตะต้อง การตรวจสอบความสมบูรณ์ที่ล้มเหลวหรือ
ไฟล์ sidecar กำพร้าจะรักษาไฟล์ DB, WAL, SHM และ rollback journal ไว้โดยเปลี่ยนชื่อ

### การดาวน์เกรดหลังการย้ายข้อมูล SQLite ของเซสชัน

ก่อนเริ่ม OpenClaw เวอร์ชันเก่าที่ใช้ไฟล์เป็นฐาน ให้กู้คืน
อาร์ติแฟกต์ทรานสคริปต์แบบเดิมที่เก็บถาวรไว้:

```bash
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

เวอร์ชันเก่าจะอ่านรายการ `sessions.json` และพาธ `sessionFile` ที่บันทึกไว้
ในรายการเหล่านั้น หลังการย้ายข้อมูลไปยัง SQLite การนำเข้าสำเร็จจะย้ายทรานสคริปต์ JSONL
ที่ใช้งานอยู่ไปยัง `session-sqlite-import-archive/` ดังนั้นรันไทม์รุ่นเก่าจะไม่สามารถ
เห็นประวัตินั้นได้จนกว่าการกู้คืนจะย้ายอาร์ติแฟกต์ที่บันทึกไว้ในแมนิเฟสต์เหล่านั้นกลับไปยัง
พาธเดิม

การกู้คืนจะไม่ลบข้อมูล SQLite เซสชันที่สร้างหลังจากเปลี่ยนไปใช้ SQLite
มีอยู่เฉพาะใน SQLite และจะไม่ปรากฏในรันไทม์รุ่นเก่า หากภายหลัง
อัปเกรดอีกครั้ง ให้เรียกใช้ลำดับการตรวจสอบความถูกต้องของการย้ายข้อมูลตามปกติด้านบน เพื่อให้ OpenClaw
สามารถเปรียบเทียบอาร์ติแฟกต์แบบเดิมที่กู้คืนแล้วกับแถว SQLite ก่อนนำเข้า

## หมายเหตุ

- ในโหมด Nix (`OPENCLAW_NIX_MODE=1`) การตรวจสอบ doctor แบบอ่านอย่างเดียวยังคงทำงานได้ แต่ `doctor --fix`, `doctor --repair`, `doctor --yes` และ `doctor --generate-gateway-token` ถูกปิดใช้งาน เนื่องจาก `openclaw.json` เปลี่ยนแปลงไม่ได้ ให้แก้ไขซอร์ส Nix สำหรับการติดตั้งนี้แทน สำหรับ nix-openclaw ให้ใช้ [คู่มือเริ่มต้นฉบับย่อ](https://github.com/openclaw/nix-openclaw#quick-start) ที่ให้เอเจนต์ดำเนินการก่อน
- พรอมต์แบบโต้ตอบ (การแก้ไขพวงกุญแจ/OAuth เป็นต้น) จะทำงานเฉพาะเมื่อ stdin เป็น TTY และ **ไม่ได้** ตั้งค่า `--non-interactive` การทำงานแบบไม่มีส่วนติดต่อ (cron, Telegram, ไม่มีเทอร์มินัล) จะข้ามพรอมต์
- การเรียกใช้ `doctor` แบบไม่โต้ตอบจะข้ามการโหลด Plugin ล่วงหน้า เพื่อให้การตรวจสอบสถานะแบบไม่มีส่วนติดต่อทำงานได้รวดเร็ว เซสชันแบบโต้ตอบยังคงโหลดพื้นผิว Plugin ที่จำเป็นสำหรับโฟลว์ตรวจสอบสถานะ/ซ่อมแซมแบบเดิม
- `--lint` เข้มงวดกว่า `--non-interactive`: อ่านอย่างเดียวเสมอ ไม่แสดงพรอมต์ และไม่ใช้การย้ายข้อมูลที่ปลอดภัย ใช้ `doctor --fix` หรือ `doctor --repair` เมื่อต้องการให้ doctor ทำการเปลี่ยนแปลง
- โดยค่าเริ่มต้น doctor จะไม่เรียกใช้ SecretRef `exec` ขณะตรวจสอบข้อมูลลับ ใช้ `--allow-exec` (โดยมีหรือไม่มี `--lint`) เฉพาะเมื่อตั้งใจให้ doctor เรียกใช้ตัวแก้ไขข้อมูลลับที่กำหนดค่าไว้เหล่านั้น
- การเขียนการกำหนดค่าใดๆ (รวมถึงการซ่อมแซม `--fix`) จะหมุนเวียนข้อมูลสำรองไปยัง `~/.openclaw/openclaw.json.bak` (พร้อมวงแหวนแบบมีหมายเลข `.bak.1`..`.bak.4`) นอกจากนี้ `--fix` ยังลบคีย์การกำหนดค่าที่ไม่รู้จักซึ่งรายงานโดยการตรวจสอบสคีมา พร้อมแสดงรายการที่ลบแต่ละรายการ โดยจะข้ามขั้นตอนนี้ขณะที่กำลังอัปเดต เพื่อไม่ให้สถานะการอัปเกรดที่เขียนเพียงบางส่วนถูกตัดออกก่อนการย้ายข้อมูลเสร็จสิ้น
- ตั้งค่า `OPENCLAW_SERVICE_REPAIR_POLICY=external` เมื่อตัวควบคุมอื่นเป็นเจ้าของวงจรชีวิตของ Gateway Doctor ยังคงรายงานสถานะของ Gateway/บริการและใช้การซ่อมแซมที่ไม่เกี่ยวกับบริการ แต่จะข้ามการติดตั้ง/เริ่ม/รีสตาร์ต/บูตสแตรปบริการ และการล้างบริการแบบเดิม
- บน Linux doctor จะละเว้นยูนิต systemd เพิ่มเติมที่คล้าย Gateway ซึ่งไม่ได้ทำงาน และจะไม่เขียนเมทาดาทาคำสั่ง/จุดเริ่มต้นใหม่สำหรับบริการ Gateway ของ systemd ที่กำลังทำงานอยู่ระหว่างการซ่อมแซม ให้หยุดบริการก่อน หรือใช้ `openclaw gateway install --force` เพื่อแทนที่ตัวเรียกใช้งานที่ใช้งานอยู่
- `doctor --fix --non-interactive` รายงานข้อกำหนดบริการ Gateway ที่หายไปหรือล้าสมัย แต่จะไม่ติดตั้งหรือเขียนใหม่หากอยู่นอกโหมดซ่อมแซมการอัปเดต เรียกใช้ `openclaw gateway install` สำหรับบริการที่หายไป หรือ `openclaw gateway install --force` เพื่อแทนที่ตัวเรียกใช้งาน
- การตรวจสอบความสมบูรณ์ของสถานะจะตรวจพบไฟล์ทรานสคริปต์กำพร้าในไดเรกทอรีเซสชัน การเก็บถาวรเป็น `.deleted.<timestamp>` ต้องได้รับการยืนยันแบบโต้ตอบ ส่วน `--fix`, `--yes` และการทำงานแบบไม่มีส่วนติดต่อจะปล่อยไฟล์ไว้ที่เดิม
- Doctor สแกน `~/.openclaw/cron/jobs.json` (หรือ `cron.store`) เพื่อค้นหารูปแบบงาน cron แบบเดิม และเขียนใหม่ก่อนนำเข้าแถวรูปแบบมาตรฐานลงใน SQLite
- Doctor รายงานงาน cron ที่มีการแทนที่ `payload.model` อย่างชัดเจน รวมถึงจำนวนตามเนมสเปซของผู้ให้บริการและความไม่ตรงกันกับ `agents.defaults.model` เพื่อให้มองเห็นงานตามกำหนดเวลาที่ไม่ได้สืบทอดโมเดลเริ่มต้นระหว่างการตรวจสอบปัญหาการยืนยันตัวตนหรือการเรียกเก็บเงิน
- Doctor รายงานงาน cron ที่ยังคงทำเครื่องหมายว่ากำลังดำเนินการอยู่ (`state.runningAtMs`) ซึ่งอาจทำให้ `openclaw cron list` แสดงงานเหล่านั้นเป็น `running` การตรวจสอบนี้เป็นแบบอ่านอย่างเดียว หากไม่มี Gateway กำลังเรียกใช้งานที่ทำเครื่องหมายไว้ การเริ่มบริการ cron ครั้งถัดไปจะบันทึกการทำงานที่ถูกขัดจังหวะและล้างเครื่องหมาย
- บน Linux doctor จะแจ้งเตือนเมื่อ crontab ของผู้ใช้ยังคงเรียกใช้ `~/.openclaw/bin/ensure-whatsapp.sh` แบบเดิมที่ไม่มีการบำรุงรักษา ซึ่งอาจรายงาน `Gateway inactive` ผิดพลาดเมื่อ cron ไม่มีสภาพแวดล้อมบัสผู้ใช้ของ systemd
- เมื่อเปิดใช้งาน WhatsApp doctor จะตรวจสอบลูปเหตุการณ์ของ Gateway ที่ประสิทธิภาพลดลงและยังมีไคลเอนต์ `openclaw-tui` ในเครื่องทำงานอยู่ `doctor --fix` จะหยุดเฉพาะไคลเอนต์ TUI ในเครื่องที่ตรวจสอบแล้ว เพื่อไม่ให้การตอบกลับ WhatsApp รอคิวอยู่หลังลูปรีเฟรช TUI ที่ค้าง
- Doctor เขียนการอ้างอิงโมเดล `codex/*` และ `openai-codex/*` แบบเดิมใหม่เป็นการอ้างอิง `openai/*` รูปแบบมาตรฐานในโมเดลหลัก โมเดลสำรอง รายการโมเดลที่อนุญาต โมเดลสร้างรูปภาพ/วิดีโอ การแทนที่ Heartbeat/เอเจนต์ย่อย/Compaction ฮุก การแทนที่โมเดลของช่องทาง เพย์โหลด cron และพินเส้นทางเซสชัน/ทรานสคริปต์ที่ล้าสมัย นอกจากนี้ `--fix` ยังผสานการกำหนดค่า `models.providers.codex` และ `models.providers.openai-codex` แบบเดิมเมื่อทำได้อย่างปลอดภัย ย้ายโปรไฟล์การยืนยันตัวตน `openai-codex:*` และรายการ `auth.order.openai-codex` แบบเดิมไปยัง `openai:*` ย้ายเจตนาการใช้ Codex ไปยังรายการ `agentRuntime.id: "codex"` ที่กำหนดขอบเขตตามผู้ให้บริการ/โมเดล ลบพินรันไทม์ระดับเอเจนต์ทั้งหมด/เซสชันที่ล้าสมัย และคงการอ้างอิงเอเจนต์ OpenAI ที่ซ่อมแซมแล้วไว้บนการกำหนดเส้นทางการยืนยันตัวตน Codex แทนการยืนยันตัวตนโดยตรงด้วยคีย์ API ของ OpenAI
- Doctor รายงานรายการ `auth.order.<provider>` ที่ไม่ว่าง ซึ่งโปรไฟล์ที่อ้างอิงทั้งหมดหายไปแล้ว ขณะที่ยังมีข้อมูลประจำตัวที่เข้ากันได้จัดเก็บอยู่ `doctor --fix` จะลบเฉพาะการแทนที่ที่ล้าสมัยเหล่านั้น เพื่อคืนค่าการเลือกข้อมูลประจำตัวอัตโนมัติสำหรับแต่ละเอเจนต์ ลำดับที่ว่างอย่างชัดเจน รายการที่บางส่วนยังใช้งานได้ และลำดับที่ไม่มีข้อมูลประจำตัวที่เข้ากันได้จัดเก็บอยู่จะไม่เปลี่ยนแปลง หากที่เก็บการยืนยันตัวตน SQLite ที่ใช้งานอยู่ไม่สามารถอ่านได้หรือมีรูปแบบผิด doctor จะอธิบายเหตุผลที่ข้ามการซ่อมแซมนี้ รีสตาร์ต Gateway ที่กำลังทำงานก่อนตรวจสอบสถานะการยืนยันตัวตนอีกครั้ง หากโหมดโหลดการกำหนดค่าใหม่ไม่ได้ใช้การเขียนโดยอัตโนมัติ
- Doctor ล้างสถานะการจัดเตรียมการขึ้นต่อกันของ Plugin แบบเดิมจาก OpenClaw เวอร์ชันเก่า และเชื่อมโยงแพ็กเกจโฮสต์ `openclaw` ใหม่สำหรับ Plugin npm ที่จัดการโดยระบบซึ่งประกาศแพ็กเกจนี้เป็นการขึ้นต่อกันแบบเพียร์ นอกจากนี้ยังซ่อมแซม Plugin ที่ดาวน์โหลดได้ซึ่งหายไปและถูกอ้างอิงโดยการกำหนดค่า (`plugins.entries`, ช่องทางที่กำหนดค่าไว้ การตั้งค่าผู้ให้บริการ/การค้นหาที่กำหนดค่าไว้ และรันไทม์เอเจนต์ที่กำหนดค่าไว้) ระหว่างการอัปเดตแพ็กเกจ doctor จะข้ามการซ่อมแซม Plugin ผ่านตัวจัดการแพ็กเกจจนกว่าการสลับแพ็กเกจจะเสร็จสิ้น จากนั้นเรียกใช้ `openclaw doctor --fix` อีกครั้ง หาก Plugin ที่กำหนดค่าไว้ยังต้องกู้คืน หากการดาวน์โหลดล้มเหลว doctor จะรายงานข้อผิดพลาดในการติดตั้งและคงรายการ Plugin ที่กำหนดค่าไว้สำหรับการซ่อมแซมครั้งถัดไป
- Doctor ซ่อมแซมการกำหนดค่า Plugin ที่ล้าสมัย โดยลบรหัส Plugin ที่หายไปออกจาก `plugins.allow`/`plugins.deny`/`plugins.entries` รวมถึงการกำหนดค่าช่องทางที่ค้าง เป้าหมาย Heartbeat และการแทนที่โมเดลของช่องทางที่ตรงกัน เมื่อการค้นพบ Plugin ทำงานเป็นปกติ
- Doctor กักกันการกำหนดค่า Plugin ที่ไม่ถูกต้อง โดยปิดใช้งานรายการ `plugins.entries.<id>` ที่ได้รับผลกระทบและลบเพย์โหลด `config` ที่ไม่ถูกต้อง การเริ่มต้น Gateway จะข้ามเฉพาะ Plugin ที่มีปัญหานั้นอยู่แล้ว เพื่อให้ Plugin และช่องทางอื่นทำงานต่อไป
- Doctor ลบ `plugins.entries.codex.config.codexDynamicToolsProfile` ที่เลิกใช้แล้ว เซิร์ฟเวอร์แอป Codex จะคงเครื่องมือพื้นที่ทำงานแบบเนทีฟของ Codex ไว้เป็นเนทีฟเสมอ
- Doctor ย้ายการกำหนดค่า Talk แบบแบนเดิม (`talk.voiceId`, `talk.modelId` และรายการที่เกี่ยวข้อง) ไปยัง `talk.provider` + `talk.providers.<provider>` โดยอัตโนมัติ การเรียกใช้ `doctor --fix` ซ้ำจะไม่รายงาน/ใช้การปรับรูปแบบ Talk อีกต่อไป เมื่อความแตกต่างมีเพียงลำดับคีย์ของออบเจ็กต์
- Doctor มีการตรวจสอบความพร้อมของการค้นหาหน่วยความจำ และสามารถแนะนำ `openclaw configure --section model` เมื่อไม่มีข้อมูลประจำตัวสำหรับการฝัง
- Doctor จะแจ้งเตือนเมื่อไม่ได้กำหนดค่าเจ้าของคำสั่ง เจ้าของคำสั่งคือบัญชีผู้ปฏิบัติงานที่เป็นมนุษย์ซึ่งได้รับอนุญาตให้เรียกใช้คำสั่งเฉพาะเจ้าของและอนุมัติการดำเนินการที่อันตราย การจับคู่ DM เพียงอนุญาตให้บุคคลสนทนากับบอตเท่านั้น หากอนุมัติผู้ส่งก่อนมีการบูตสแตรปเจ้าของคนแรก ให้ตั้งค่า `commands.ownerAllowFrom` อย่างชัดเจน
- Doctor รายงานหมายเหตุข้อมูลเมื่อมีการกำหนดค่าเอเจนต์ในโหมด Codex และมีแอสเซ็ต Codex CLI ส่วนบุคคลอยู่ในโฮม Codex ของผู้ปฏิบัติงาน การเริ่มเซิร์ฟเวอร์แอป Codex ในเครื่องจะใช้โฮมแยกสำหรับแต่ละเอเจนต์ หากจำเป็น ให้ติดตั้ง Plugin Codex ก่อน จากนั้นใช้ `openclaw migrate plan codex` เพื่อจัดทำรายการแอสเซ็ตที่ควรเลื่อนระดับอย่างตั้งใจ
- Doctor จะแจ้งเตือนเมื่อ Skills ที่อนุญาตสำหรับเอเจนต์เริ่มต้นไม่พร้อมใช้งานในสภาพแวดล้อมรันไทม์ปัจจุบัน (ไม่มีไบนารี ตัวแปรสภาพแวดล้อม การกำหนดค่า หรือข้อกำหนดของระบบปฏิบัติการ) `doctor --fix` สามารถปิดใช้งาน Skills ที่ไม่พร้อมเหล่านั้นด้วย `skills.entries.<skill>.enabled=false` แต่หากต้องการให้ Skill ยังคงทำงานอยู่ ให้ติดตั้ง/กำหนดค่าข้อกำหนดที่ขาดหายไปแทน
- หากเปิดใช้งานโหมดแซนด์บ็อกซ์แต่ Docker ไม่พร้อมใช้งาน doctor จะรายงานคำเตือนที่ชัดเจนพร้อมวิธีแก้ไข (`install Docker` หรือ `openclaw config set agents.defaults.sandbox.mode off`)
- หากมีไฟล์รีจิสทรีแซนด์บ็อกซ์แบบเดิมหรือไดเรกทอรีชาร์ด (`~/.openclaw/sandbox/containers.json`, `~/.openclaw/sandbox/browsers.json`, `~/.openclaw/sandbox/containers/` หรือ `~/.openclaw/sandbox/browsers/`) doctor จะรายงานรายการเหล่านั้น ส่วน `--fix` จะย้ายรายการที่ถูกต้องไปยัง SQLite และกักกันไฟล์แบบเดิมที่ไม่ถูกต้อง
- หาก `gateway.auth.token`/`gateway.auth.password` จัดการโดย SecretRef และไม่พร้อมใช้งานในพาธคำสั่งปัจจุบัน doctor จะรายงานคำเตือนแบบอ่านอย่างเดียวและไม่เขียนข้อมูลประจำตัวสำรองเป็นข้อความธรรมดา สำหรับ SecretRef ที่ใช้ exec doctor จะข้ามการเรียกใช้ เว้นแต่มี `--allow-exec`
- หากการตรวจสอบ SecretRef ของช่องทางล้มเหลวในพาธการแก้ไข doctor จะดำเนินการต่อและรายงานคำเตือนแทนการออกก่อนกำหนด
- หลังการย้ายไดเรกทอรีสถานะ doctor จะแจ้งเตือนเมื่อบัญชี Telegram หรือ Discord เริ่มต้นที่เปิดใช้งานต้องพึ่งพาค่าสำรองจากสภาพแวดล้อม และ `TELEGRAM_BOT_TOKEN` หรือ `DISCORD_BOT_TOKEN` ไม่พร้อมใช้งานสำหรับโปรเซส doctor
- การแก้ชื่อผู้ใช้ `allowFrom` ของ Telegram โดยอัตโนมัติ (`doctor --fix`) ต้องใช้โทเค็น Telegram ที่แก้ไขได้ในพาธคำสั่งปัจจุบัน หากตรวจสอบโทเค็นไม่ได้ doctor จะรายงานคำเตือนและข้ามการแก้ชื่อโดยอัตโนมัติสำหรับรอบนั้น

## macOS: การแทนที่ตัวแปรสภาพแวดล้อม `launchctl`

หากก่อนหน้านี้เคยเรียกใช้ `launchctl setenv OPENCLAW_GATEWAY_TOKEN ...` (หรือ `...PASSWORD`) ค่านั้นจะแทนที่ไฟล์การกำหนดค่าและอาจทำให้เกิดข้อผิดพลาด "unauthorized" อย่างต่อเนื่อง

```bash
launchctl getenv OPENCLAW_GATEWAY_TOKEN
launchctl getenv OPENCLAW_GATEWAY_PASSWORD

launchctl unsetenv OPENCLAW_GATEWAY_TOKEN
launchctl unsetenv OPENCLAW_GATEWAY_PASSWORD
```
