# `openclaw devices`

จัดการคำขอจับคู่อุปกรณ์และโทเค็นที่ผูกกับอุปกรณ์

## คำสั่ง

### `openclaw devices list`

แสดงรายการคำขอจับคู่ที่รอดำเนินการและอุปกรณ์ที่จับคู่แล้ว

```
openclaw devices list
openclaw devices list --json
```

เอาต์พุตของคำขอที่รอดำเนินการจะแสดงสิทธิ์เข้าถึงที่ร้องขอถัดจากสิทธิ์เข้าถึงปัจจุบันที่อนุมัติแล้วของอุปกรณ์ เมื่ออุปกรณ์นั้นจับคู่ไว้แล้ว สิ่งนี้ทำให้การอัปเกรดขอบเขต/บทบาทชัดเจน แทนที่จะดูเหมือนว่าการจับคู่หายไป

### `openclaw devices remove <deviceId>`

ลบรายการอุปกรณ์ที่จับคู่แล้วหนึ่งรายการ

เมื่อคุณตรวจสอบสิทธิ์ด้วยโทเค็นอุปกรณ์ที่จับคู่แล้ว ผู้เรียกที่ไม่ใช่ผู้ดูแลระบบจะลบได้เฉพาะรายการอุปกรณ์ของ **ตนเอง** เท่านั้น การลบอุปกรณ์อื่นต้องมี `operator.admin`

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

ล้างอุปกรณ์ที่จับคู่แล้วแบบเป็นกลุ่ม

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

อนุมัติคำขอจับคู่อุปกรณ์ที่รอดำเนินการด้วย `requestId` ที่ตรงกันทุกประการ หากละ `requestId` ไว้หรือส่ง `--latest` OpenClaw จะพิมพ์เฉพาะคำขอที่รอดำเนินการซึ่งเลือกไว้แล้วออกจากโปรแกรม ให้รันการอนุมัติอีกครั้งด้วย ID คำขอที่ตรงกันทุกประการหลังจากตรวจสอบรายละเอียดแล้ว

<Note>
หากอุปกรณ์ลองจับคู่อีกครั้งด้วยรายละเอียดการตรวจสอบสิทธิ์ที่เปลี่ยนไป (บทบาท ขอบเขต หรือกุญแจสาธารณะ) OpenClaw จะแทนที่รายการที่รอดำเนินการก่อนหน้าและออก `requestId` ใหม่ รัน `openclaw devices list` ทันที ก่อนอนุมัติเพื่อใช้ ID ปัจจุบัน
</Note>

หากอุปกรณ์จับคู่ไว้แล้วและขอขอบเขตที่กว้างขึ้นหรือบทบาทที่กว้างขึ้น OpenClaw จะคงการอนุมัติเดิมไว้และสร้างคำขออัปเกรดใหม่ที่รอดำเนินการ ตรวจสอบคอลัมน์ `Requested` เทียบกับ `Approved` ใน `openclaw devices list` หรือใช้ `openclaw devices approve --latest` เพื่อดูตัวอย่างการอัปเกรดที่ตรงกันทุกประการก่อนอนุมัติ

หาก Gateway ถูกกำหนดค่าไว้อย่างชัดเจนด้วย `gateway.nodes.pairing.autoApproveCidrs` คำขอครั้งแรกที่เป็น `role: node` จาก IP ไคลเอ็นต์ที่ตรงกันอาจได้รับการอนุมัติก่อนที่จะปรากฏในรายการนี้ นโยบายนี้ปิดใช้งานโดยค่าเริ่มต้น และไม่ใช้กับไคลเอ็นต์ผู้ปฏิบัติการ/เบราว์เซอร์หรือคำขออัปเกรด

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

ปฏิเสธคำขอจับคู่อุปกรณ์ที่รอดำเนินการ

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

หมุนเวียนโทเค็นอุปกรณ์สำหรับบทบาทที่ระบุ (อัปเดตขอบเขตได้ตามต้องการ)
บทบาทเป้าหมายต้องมีอยู่แล้วในสัญญาการจับคู่ที่อนุมัติแล้วของอุปกรณ์นั้น การหมุนเวียนไม่สามารถออกบทบาทใหม่ที่ยังไม่ได้อนุมัติได้
หากคุณละ `--scope` ไว้ การเชื่อมต่อใหม่ภายหลังด้วยโทเค็นที่หมุนเวียนและจัดเก็บไว้จะใช้ขอบเขตที่อนุมัติแล้วซึ่งแคชไว้ของโทเค็นนั้นซ้ำ หากคุณส่งค่า `--scope` อย่างชัดเจน ค่าเหล่านั้นจะกลายเป็นชุดขอบเขตที่จัดเก็บไว้สำหรับการเชื่อมต่อใหม่ในอนาคตด้วยโทเค็นที่แคชไว้
ผู้เรียกที่เป็นอุปกรณ์จับคู่แล้วและไม่ใช่ผู้ดูแลระบบจะหมุนเวียนได้เฉพาะโทเค็นอุปกรณ์ของ **ตนเอง** เท่านั้น
ชุดขอบเขตของโทเค็นเป้าหมายต้องอยู่ภายในขอบเขตผู้ปฏิบัติการของเซสชันผู้เรียกเอง การหมุนเวียนไม่สามารถออกหรือคงโทเค็นผู้ปฏิบัติการที่กว้างกว่าที่ผู้เรียกมีอยู่แล้วได้

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

ส่งคืนเมตาดาต้าการหมุนเวียนเป็น JSON หากผู้เรียกกำลังหมุนเวียนโทเค็นของตนเองขณะตรวจสอบสิทธิ์ด้วยโทเค็นอุปกรณ์นั้น การตอบกลับจะรวมโทเค็นทดแทนด้วย เพื่อให้ไคลเอ็นต์สามารถบันทึกไว้ก่อนเชื่อมต่อใหม่ การหมุนเวียนแบบแชร์/ผู้ดูแลระบบจะไม่สะท้อน bearer token กลับมา

### `openclaw devices revoke --device <id> --role <role>`

เพิกถอนโทเค็นอุปกรณ์สำหรับบทบาทที่ระบุ

ผู้เรียกที่เป็นอุปกรณ์จับคู่แล้วและไม่ใช่ผู้ดูแลระบบจะเพิกถอนได้เฉพาะโทเค็นอุปกรณ์ของ **ตนเอง** เท่านั้น
การเพิกถอนโทเค็นของอุปกรณ์อื่นต้องมี `operator.admin`
ชุดขอบเขตของโทเค็นเป้าหมายต้องอยู่ภายในขอบเขตผู้ปฏิบัติการของเซสชันผู้เรียกเองด้วย ผู้เรียกที่มีเฉพาะการจับคู่ไม่สามารถเพิกถอนโทเค็นผู้ปฏิบัติการระดับผู้ดูแลระบบ/เขียนได้

```
openclaw devices revoke --device <deviceId> --role node
```

ส่งคืนผลการเพิกถอนเป็น JSON

## ตัวเลือกทั่วไป

- `--url <url>`: URL WebSocket ของ Gateway (ค่าเริ่มต้นคือ `gateway.remote.url` เมื่อกำหนดค่าไว้)
- `--token <token>`: โทเค็น Gateway (หากจำเป็น)
- `--password <password>`: รหัสผ่าน Gateway (การตรวจสอบสิทธิ์ด้วยรหัสผ่าน)
- `--timeout <ms>`: เวลาหมดอายุ RPC
- `--json`: เอาต์พุต JSON (แนะนำสำหรับการเขียนสคริปต์)

<Warning>
เมื่อคุณตั้งค่า `--url` CLI จะไม่ย้อนกลับไปใช้ข้อมูลประจำตัวจากการกำหนดค่าหรือสภาพแวดล้อม ส่ง `--token` หรือ `--password` อย่างชัดเจน การไม่มีข้อมูลประจำตัวที่ระบุอย่างชัดเจนถือเป็นข้อผิดพลาด
</Warning>

## หมายเหตุ

- การหมุนเวียนโทเค็นจะส่งคืนโทเค็นใหม่ (ข้อมูลอ่อนไหว) ให้ปฏิบัติกับมันเหมือนความลับ
- คำสั่งเหล่านี้ต้องมีขอบเขต `operator.pairing` (หรือ `operator.admin`)
- `gateway.nodes.pairing.autoApproveCidrs` เป็นนโยบาย Gateway แบบเลือกใช้สำหรับการจับคู่อุปกรณ์โหนดใหม่เท่านั้น และไม่เปลี่ยนอำนาจการอนุมัติของ CLI
- การหมุนเวียนและการเพิกถอนโทเค็นจะยังอยู่ภายในชุดบทบาทการจับคู่ที่อนุมัติแล้วและเส้นฐานขอบเขตที่อนุมัติแล้วสำหรับอุปกรณ์นั้น รายการโทเค็นที่แคชไว้แบบหลงเหลือจะไม่ให้สิทธิ์เป้าหมายการจัดการโทเค็น
- สำหรับเซสชันโทเค็นอุปกรณ์ที่จับคู่แล้ว การจัดการข้ามอุปกรณ์ทำได้เฉพาะผู้ดูแลระบบเท่านั้น: `remove`, `rotate` และ `revoke` ทำได้เฉพาะของตนเอง เว้นแต่ผู้เรียกจะมี `operator.admin`
- การเปลี่ยนแปลงโทเค็นยังถูกจำกัดตามขอบเขตของผู้เรียกด้วย: เซสชันที่มีเฉพาะการจับคู่ไม่สามารถหมุนเวียนหรือเพิกถอนโทเค็นที่ขณะนี้มี `operator.admin` หรือ `operator.write`
- `devices clear` ตั้งใจให้ถูกควบคุมด้วย `--yes`
- หากขอบเขตการจับคู่ไม่พร้อมใช้งานบน local loopback (และไม่มีการส่ง `--url` อย่างชัดเจน) การแสดงรายการ/อนุมัติสามารถใช้ทางเลือกสำรองการจับคู่ภายในเครื่องได้
- `devices approve` ต้องมี ID คำขอที่ระบุอย่างชัดเจนก่อนออกโทเค็น การละ `requestId` หรือส่ง `--latest` จะเป็นเพียงการแสดงตัวอย่างคำขอที่รอดำเนินการล่าสุดเท่านั้น

## รายการตรวจสอบการกู้คืนเมื่อโทเค็นคลาดเคลื่อน

ใช้รายการนี้เมื่อ Control UI หรือไคลเอ็นต์อื่นยังล้มเหลวด้วย `AUTH_TOKEN_MISMATCH` หรือ `AUTH_DEVICE_TOKEN_MISMATCH`

1. ยืนยันแหล่งที่มาของโทเค็น Gateway ปัจจุบัน:

```bash
openclaw config get gateway.auth.token
```

2. แสดงรายการอุปกรณ์ที่จับคู่แล้วและระบุ ID อุปกรณ์ที่ได้รับผลกระทบ:

```bash
openclaw devices list
```

3. หมุนเวียนโทเค็นผู้ปฏิบัติการสำหรับอุปกรณ์ที่ได้รับผลกระทบ:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. หากการหมุนเวียนยังไม่เพียงพอ ให้ลบการจับคู่ที่ค้างอยู่และอนุมัติอีกครั้ง:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. ลองเชื่อมต่อไคลเอ็นต์อีกครั้งด้วยโทเค็น/รหัสผ่านที่แชร์อยู่ในปัจจุบัน

หมายเหตุ:

- ลำดับความสำคัญของการตรวจสอบสิทธิ์เมื่อเชื่อมต่อใหม่ตามปกติคือ โทเค็น/รหัสผ่านที่แชร์ซึ่งระบุอย่างชัดเจนก่อน จากนั้นจึงเป็น `deviceToken` ที่ระบุอย่างชัดเจน จากนั้นจึงเป็นโทเค็นอุปกรณ์ที่จัดเก็บไว้ และสุดท้ายคือโทเค็นบูตสแตรป
- การกู้คืน `AUTH_TOKEN_MISMATCH` ที่เชื่อถือได้สามารถส่งทั้งโทเค็นที่แชร์และโทเค็นอุปกรณ์ที่จัดเก็บไว้พร้อมกันชั่วคราวสำหรับการลองซ้ำหนึ่งครั้งที่มีขอบเขตจำกัด
