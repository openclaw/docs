# `openclaw channels`

จัดการบัญชีช่องแชตและสถานะรันไทม์ของบัญชีเหล่านั้นบน Gateway

เอกสารที่เกี่ยวข้อง:

- คู่มือช่อง: [ช่อง](/th/channels)
- การกำหนดค่า Gateway: [การกำหนดค่า](/th/gateway/configuration)

## คำสั่งทั่วไป

```bash
openclaw channels list
openclaw channels list --all
openclaw channels status
openclaw channels capabilities
openclaw channels capabilities --channel discord --target channel:123
openclaw channels resolve --channel slack "#general" "@jane"
openclaw channels logs --channel all
openclaw channels dead-letters list --channel telegram --account default
```

`channels list` แสดงเฉพาะช่องแชต โดยค่าเริ่มต้นจะแสดงบัญชีที่กำหนดค่าแล้ว พร้อมแท็กสถานะ `installed`, `configured` และ `enabled` ต่อบัญชี (`--json` สำหรับเอาต์พุตที่เครื่องอ่านได้) ส่ง `--all` เพื่อแสดงช่องที่รวมมาให้แต่ยังไม่มีบัญชีที่กำหนดค่า และช่องจากแค็ตตาล็อกที่ติดตั้งได้แต่ยังไม่มีอยู่บนดิสก์ด้วย การยืนยันตัวตนของผู้ให้บริการและการใช้งานโมเดลอยู่ในส่วนอื่น ได้แก่ `openclaw models auth list` สำหรับโปรไฟล์การยืนยันตัวตนของผู้ให้บริการ และ `openclaw status` หรือ `openclaw models list` สำหรับการใช้งาน/โควตา

## สถานะ / ความสามารถ / การแปลงชื่อ / บันทึก

- `channels status`: `--channel <name>`, `--probe`, `--timeout <ms>` (ค่าเริ่มต้น `10000`), `--json`
- `channels capabilities`: `--channel <name>`, `--account <id>` (ต้องมี `--channel`), `--target <dest>` (ต้องมี `--channel`), `--timeout <ms>` (ค่าเริ่มต้น `10000`, จำกัดสูงสุดที่ `30000`), `--json`
- `channels resolve <entries...>`: `--channel <name>`, `--account <id>`, `--kind <auto|user|group>` (ค่าเริ่มต้น `auto`), `--json`
- `channels logs`: `--channel <name|all>` (ค่าเริ่มต้น `all`), `--lines <n>` (ค่าเริ่มต้น `200`), `--json`

`channels status --probe` เป็นเส้นทางแบบสด เมื่อเข้าถึง Gateway ได้ ระบบจะเรียกใช้การตรวจสอบ
`probeAccount` ต่อบัญชีและการตรวจสอบ `auditAccount` ซึ่งเป็นทางเลือก ดังนั้นเอาต์พุตจึงอาจมีทั้ง
สถานะการขนส่งและผลการตรวจสอบ เช่น `works`, `probe failed`, `audit ok` หรือ `audit failed`
หากเข้าถึง Gateway ไม่ได้ `channels status` จะเปลี่ยนไปใช้ข้อมูลสรุปจากการกำหนดค่าเท่านั้น
แทนเอาต์พุตจากการตรวจสอบแบบสด

## จดหมายที่ส่งเข้าแล้วไม่สำเร็จ

เหตุการณ์ขาเข้าที่ใช้ความพยายามตามนโยบายการลองใหม่จนหมดจะยังคงอยู่ในฐานข้อมูลสถานะที่ใช้ร่วมกันตามระยะเวลาเก็บรักษารายการที่ล้มเหลวซึ่งมีอยู่ของคิว ตรวจสอบบัญชีช่องหนึ่งบัญชีด้วย:

```bash
openclaw channels dead-letters list --channel telegram --account default
openclaw channels dead-letters list --channel telegram --account default --json
```

มุมมองข้อความจะแสดงรหัสเหตุการณ์ สาเหตุที่ล้มเหลว จำนวนครั้งที่พยายาม และระยะเวลาตั้งแต่ล้มเหลว เอาต์พุต JSON ยังมีเพย์โหลดที่เก็บรักษาไว้ เมทาดาทา เลน และการประทับเวลาของแต่ละครั้งที่พยายามเพื่อใช้ในการวินิจฉัย

หลังจากแก้ไขปัญหาต้นเหตุแล้ว ให้นำเหตุการณ์หนึ่งรายการเข้าคิวอีกครั้งด้วยรหัสเหตุการณ์เดิม:

```bash
openclaw channels dead-letters resubmit <event-id> --channel telegram --account default
```

เรียกใช้คำสั่งเหล่านี้บนโฮสต์ Gateway เพื่อให้เข้าถึงฐานข้อมูลสถานะที่ใช้ร่วมกันฐานเดียวกับรันไทม์ของช่อง การส่งซ้ำจะคงเพย์โหลด เมทาดาทา และเลนไว้ แต่รีเซ็ตตัวนับจำนวนครั้งที่พยายามและอายุของคิว ระบบจะแทนที่เครื่องหมายล้มเหลวของเหตุการณ์นั้นแบบอะตอมมิก ดังนั้นหากเรียกคำสั่งซ้ำขณะที่เหตุการณ์กำลังรอดำเนินการหรือถูกอ้างสิทธิ์ ระบบจะปฏิเสธแทนที่จะสร้างการส่งครั้งที่สอง ช่องที่กำลังทำงานจะรับเหตุการณ์ในการระบายข้อมูลขาเข้าครั้งถัดไป เหตุการณ์ที่เสร็จสมบูรณ์แล้วจะยังคงเป็นสถานะสิ้นสุดและไม่สามารถส่งซ้ำได้ แถวที่ล้มเหลวซึ่งสร้างขึ้นก่อนเพิ่มการเก็บรักษาเพย์โหลดยังอาจปรากฏในรายการ แต่การส่งซ้ำจะถูกปฏิเสธเนื่องจากไม่มีเพย์โหลดของแถวนั้น

`openclaw health` รายงานจำนวนจดหมายที่ส่งไม่สำเร็จและระยะเวลาตั้งแต่ความล้มเหลวที่เก่าที่สุดต่อบัญชีช่อง `openclaw doctor` ระบุชื่อบัญชีที่ได้รับผลกระทบและอ้างกลับไปยังคำสั่งตรวจสอบ

อย่าใช้ `openclaw sessions`, `sessions.list` ของ Gateway หรือเครื่องมือ
`sessions_list` ของเอเจนต์เป็นสัญญาณบอกความสมบูรณ์ของซ็อกเก็ตช่อง ส่วนเหล่านั้นรายงาน
แถวการสนทนาที่จัดเก็บไว้ ไม่ใช่สถานะรันไทม์ของผู้ให้บริการ หลังจากรีสตาร์ตผู้ให้บริการ Discord
บัญชีที่เชื่อมต่ออยู่แต่ไม่มีความเคลื่อนไหวอาจทำงานเป็นปกติ แม้จะยังไม่มีแถวเซสชัน Discord
ปรากฏขึ้นจนกว่าจะมีเหตุการณ์การสนทนาขาเข้าหรือขาออกครั้งถัดไป

## เพิ่ม / ลบบัญชี

```bash
openclaw channels add --channel telegram --token <bot-token>
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY"
openclaw channels remove --channel telegram --delete
```

<Tip>
`openclaw channels add --help` แสดงแฟล็กของแต่ละช่อง (โทเค็น คีย์ส่วนตัว โทเค็นแอป พาธ signal-cli เป็นต้น)
</Tip>

`channels remove` ทำงานเฉพาะกับ Plugin ช่องที่ติดตั้ง/กำหนดค่าแล้ว ใช้ `channels add` ก่อนสำหรับช่องจากแค็ตตาล็อกที่ติดตั้งได้ หากไม่มี `--delete` ระบบจะถามว่าจะปิดใช้งานบัญชีหรือไม่และจะเก็บการกำหนดค่าไว้ ส่วน `--delete` จะลบรายการการกำหนดค่าโดยไม่ถาม
สำหรับ Plugin ช่องที่มีรันไทม์รองรับ `channels remove` จะขอให้ Gateway ที่กำลังทำงานหยุดบัญชีที่เลือกก่อนอัปเดตการกำหนดค่าด้วย เพื่อให้การปิดใช้งานหรือลบบัญชีไม่ปล่อยให้ Listener เดิมยังทำงานอยู่จนกว่าจะรีสตาร์ต

แฟล็กการเพิ่มแบบไม่โต้ตอบที่ใช้ร่วมกันระหว่างช่อง ได้แก่ `--account <id>`, `--name <name>`, `--token`, `--token-file`, `--bot-token`, `--app-token`, `--secret`, `--secret-file`, `--password`, `--cli-path`, `--url`, `--base-url`, `--http-url`, `--auth-dir` และ `--use-env` (การยืนยันตัวตนที่อ้างอิงจากตัวแปรสภาพแวดล้อม ใช้เฉพาะบัญชีเริ่มต้นและเฉพาะที่รองรับ) แฟล็กเฉพาะช่องประกอบด้วย:

| ช่อง        | แฟล็ก                                                                                                |
| ----------- | ---------------------------------------------------------------------------------------------------- |
| Google Chat | `--webhook-path`, `--webhook-url`, `--audience-type`, `--audience`                                   |
| iMessage    | `--cli-path`, `--db-path`, `--service`, `--region`                                                   |
| Matrix      | `--homeserver`, `--user-id`, `--access-token`, `--password`, `--device-name`, `--initial-sync-limit` |
| Nostr       | `--private-key`, `--relay-urls`                                                                      |
| Signal      | `--signal-number`, `--cli-path`, `--http-url`, `--http-host`, `--http-port`                          |
| Tlon        | `--ship`, `--url`, `--code`, `--group-channels`, `--dm-allowlist`, `--auto-discover-channels`        |
| WhatsApp    | `--auth-dir`                                                                                         |

หากต้องติดตั้ง Plugin ช่องระหว่างคำสั่งเพิ่มที่ขับเคลื่อนด้วยแฟล็ก OpenClaw จะใช้แหล่งติดตั้งเริ่มต้นของช่องโดยไม่เปิดพรอมต์ติดตั้ง Plugin แบบโต้ตอบ

เมื่อเรียกใช้ `openclaw channels add` โดยไม่มีแฟล็ก วิซาร์ดแบบโต้ตอบสามารถถามข้อมูลต่อไปนี้:

- รหัสบัญชีสำหรับแต่ละช่องที่เลือก
- ชื่อที่แสดงซึ่งเป็นทางเลือกสำหรับบัญชีเหล่านั้น
- `Route these channel accounts to agents now?`

หากยืนยันให้ผูกตอนนี้ วิซาร์ดจะถามว่าเอเจนต์ใดควรเป็นเจ้าของบัญชีช่องที่กำหนดค่าแต่ละบัญชี และเขียนการผูกการกำหนดเส้นทางแบบกำหนดขอบเขตบัญชี

นอกจากนี้ยังสามารถจัดการกฎการกำหนดเส้นทางเดียวกันภายหลังได้ด้วย `openclaw agents bindings`, `openclaw agents bind` และ `openclaw agents unbind` (ดู [เอเจนต์](/th/cli/agents))

เมื่อเพิ่มบัญชีที่ไม่ใช่บัญชีเริ่มต้นลงในช่องที่ยังใช้การตั้งค่าระดับบนสุดแบบบัญชีเดียว OpenClaw จะยกระดับค่าระดับบนสุดเหล่านั้นเข้าไปในแมปบัญชีของช่องก่อนเขียนบัญชีใหม่ การยกระดับจะนำบัญชีที่ตั้งชื่อไว้เดิมกลับมาใช้เมื่อช่องมีบัญชีดังกล่าวเพียงบัญชีเดียว หรือเมื่อ `defaultAccount` ชี้ไปยังบัญชีหนึ่ง มิฉะนั้นค่าจะถูกเก็บไว้ใน `channels.<channel>.accounts.default`

ลักษณะการกำหนดเส้นทางยังคงสอดคล้องกัน:

- การผูกที่มีอยู่ซึ่งระบุเฉพาะช่อง (ไม่มี `accountId`) จะยังคงตรงกับบัญชีเริ่มต้น
- `channels add` จะไม่สร้างหรือเขียนการผูกใหม่โดยอัตโนมัติในโหมดไม่โต้ตอบ
- การตั้งค่าแบบโต้ตอบสามารถเลือกเพิ่มการผูกแบบกำหนดขอบเขตบัญชีได้

หากการกำหนดค่ามีสถานะแบบผสมอยู่แล้ว (มีบัญชีที่ตั้งชื่อไว้และยังตั้งค่าระดับบนสุดแบบบัญชีเดียวอยู่) ให้เรียกใช้ `openclaw doctor --fix` เพื่อย้ายค่าที่กำหนดขอบเขตบัญชีไปยังบัญชีที่ยกระดับซึ่งเลือกไว้สำหรับช่องนั้น

## เข้าสู่ระบบและออกจากระบบ (แบบโต้ตอบ)

```bash
openclaw channels login --channel whatsapp
openclaw channels logout --channel whatsapp
```

- `channels login` รองรับ `--account <id>` และ `--verbose`; `channels logout` รองรับ `--account <id>`
- `channels login` และ `logout` สามารถอนุมานช่องได้เมื่อมีช่องที่กำหนดค่าเพียงช่องเดียวที่รองรับการดำเนินการนั้น หากมีหลายช่อง ให้ส่ง `--channel`
- `channels logout` จะเลือกเส้นทาง Gateway แบบสดก่อนเมื่อเข้าถึงได้ เพื่อให้การออกจากระบบหยุด Listener ที่ทำงานอยู่ก่อนล้างสถานะการยืนยันตัวตนของช่อง หากเข้าถึง Gateway ภายในเครื่องไม่ได้ ระบบจะเปลี่ยนไปล้างการยืนยันตัวตนภายในเครื่อง แต่หากใช้ `gateway.mode: "remote"` ข้อผิดพลาดของ Gateway จะทำให้คำสั่งล้มเหลวแทน
- หลังเข้าสู่ระบบสำเร็จ CLI จะขอให้ Gateway ภายในเครื่องที่เข้าถึงได้เริ่มต้นบัญชี ส่วนในโหมดระยะไกล ระบบจะบันทึกการยืนยันตัวตนไว้ภายในเครื่องและแจ้งว่ารันไทม์ระยะไกลไม่ได้ถูกรีสตาร์ต
- เรียกใช้ `channels login` จากเทอร์มินัลบนโฮสต์ Gateway `exec` ของเอเจนต์จะบล็อกขั้นตอนเข้าสู่ระบบแบบโต้ตอบนี้ ควรใช้เครื่องมือเข้าสู่ระบบของเอเจนต์ที่เป็นของช่องโดยตรง เช่น `whatsapp_login` จากแชตเมื่อมีให้ใช้

## การแก้ไขปัญหา

- เรียกใช้ `openclaw status --deep` เพื่อตรวจสอบในวงกว้าง
- ใช้ `openclaw doctor` เพื่อรับการแนะนำในการแก้ไข
- `openclaw channels status` จะเปลี่ยนไปใช้ข้อมูลสรุปจากการกำหนดค่าเท่านั้นเมื่อเข้าถึง Gateway ไม่ได้ หากข้อมูลประจำตัวของช่องที่รองรับถูกกำหนดค่าผ่าน SecretRef แต่ไม่พร้อมใช้งานในเส้นทางคำสั่งปัจจุบัน ระบบจะรายงานว่าบัญชีนั้นกำหนดค่าแล้วพร้อมหมายเหตุว่าสถานะลดระดับ แทนที่จะแสดงว่ายังไม่ได้กำหนดค่า

## การตรวจสอบความสามารถ

ดึงคำแนะนำเกี่ยวกับความสามารถของผู้ให้บริการ (Intent/ขอบเขต หากมี) พร้อมการรองรับฟีเจอร์แบบคงที่:

```bash
openclaw channels capabilities
openclaw channels capabilities --channel discord --target channel:123
```

หมายเหตุ:

- `--channel` เป็นทางเลือก หากไม่ระบุ ระบบจะแสดงทุกช่อง (รวมถึงช่องที่ Plugin จัดเตรียมให้)
- `--account` ใช้ได้เฉพาะกับ `--channel`
- `--target` ยอมรับ `channel:<id>` หรือรหัสช่องที่เป็นตัวเลขดิบ และใช้ได้กับ Discord เท่านั้น สำหรับช่องเสียง Discord การตรวจสอบสิทธิ์จะระบุเมื่อขาด `ViewChannel`, `Connect`, `Speak`, `SendMessages` และ `ReadMessageHistory`
- การตรวจสอบจะเฉพาะเจาะจงตามผู้ให้บริการ ได้แก่ ข้อมูลประจำตัวบอต Discord + Intent พร้อมสิทธิ์ของช่องซึ่งเป็นทางเลือก; ขอบเขตบอต + ผู้ใช้ของ Slack; แฟล็กบอต Telegram + Webhook; เวอร์ชันดีมอนของ Signal; โทเค็นแอป Microsoft Teams + บทบาท/ขอบเขต Graph (มีคำอธิบายประกอบในส่วนที่ทราบ) ช่องที่ไม่มีการตรวจสอบจะรายงาน `Probe: unavailable`

## แปลงชื่อเป็นรหัส

แปลงชื่อช่อง/ผู้ใช้เป็นรหัสโดยใช้ไดเรกทอรีของผู้ให้บริการ:

```bash
openclaw channels resolve --channel slack "#general" "@jane"
openclaw channels resolve --channel discord "My Server/#support" "@someone"
openclaw channels resolve --channel matrix "Project Room"
```

หมายเหตุ:

- ใช้ `--kind user|group|auto` เพื่อบังคับประเภทเป้าหมาย
- การแปลงชื่อจะเลือกผลลัพธ์ที่ใช้งานอยู่ก่อนเมื่อหลายรายการใช้ชื่อเดียวกัน
- `channels resolve` เป็นแบบอ่านอย่างเดียว หากบัญชีที่เลือกกำหนดค่าผ่าน SecretRef แต่ข้อมูลประจำตัวนั้นไม่พร้อมใช้งานในเส้นทางคำสั่งปัจจุบัน คำสั่งจะคืนผลลัพธ์ที่แปลงไม่ได้พร้อมสถานะลดระดับและหมายเหตุ แทนที่จะยกเลิกการทำงานทั้งหมด
- `channels resolve` จะไม่ติดตั้ง Plugin ช่อง ใช้ `channels add --channel <name>` ก่อนแปลงชื่อสำหรับช่องจากแค็ตตาล็อกที่ติดตั้งได้
