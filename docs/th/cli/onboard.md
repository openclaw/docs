# `openclaw onboard`

การตั้งค่าแบบมีคำแนะนำที่เริ่มจากการทำให้การอนุมานใช้งานได้ก่อน โดยจะตรวจหาการเข้าถึง AI ที่มีอยู่
กำหนดให้ต้องมีการตอบกลับจริง บันทึกเฉพาะเส้นทางที่ใช้งานได้ แล้วจึงเริ่ม
OpenClaw เพื่อกำหนดค่าส่วนที่เหลือ `openclaw setup` จะเข้าสู่ขั้นตอนนี้ในระบบใหม่
หรือเมื่อมีตัวเลือกการเริ่มต้นใช้งาน ส่วนระบบที่กำหนดค่าแล้วจะใช้
`openclaw setup` แบบไม่มีอาร์กิวเมนต์สำหรับแชตกับเอเจนต์ระบบ `openclaw setup --baseline` จะ
เขียนเฉพาะการกำหนดค่าพื้นฐาน/เวิร์กสเปซ

## ตัวอย่าง

```bash
openclaw onboard
openclaw onboard --tui
openclaw onboard --classic
openclaw onboard --modern
openclaw onboard --flow quickstart
openclaw onboard --flow manual
openclaw onboard --flow import
openclaw onboard --import-from hermes --import-source ~/.hermes
openclaw onboard --skip-bootstrap
openclaw onboard recommendations --json
openclaw onboard recommendations acknowledge
openclaw onboard recommendations acknowledge --retry "<failed-id>"
openclaw onboard recommendations refresh
openclaw onboard --mode remote --remote-url wss://gateway-host:18789
```

`openclaw onboard recommendations` จะอ่านรายการที่ตรงกับคำแนะนำแอปซึ่งยังรอดำเนินการ
และบันทึกไว้ระหว่างการเริ่มต้นใช้งาน เพิ่ม `--json` เพื่อรับรายการที่เครื่องอ่านได้ซึ่งใช้โดย
การบูตครั้งแรก คำสั่งนี้จะไม่สแกนแอปที่ติดตั้งใหม่หรือเรียก
โมเดล ผลลัพธ์มีเฉพาะ ID การติดตั้งที่ผ่านการตรวจสอบ แหล่งที่มา และระดับ โดย
ตั้งใจละเว้นข้อความจากมาร์เก็ตเพลสที่ไม่น่าเชื่อถือ เหตุผลจากโมเดล และป้ายชื่อแอปในเครื่อง
หลังจากตอบข้อเสนอคำแนะนำแล้ว คำสั่งจะคืนค่า
รายการว่าง และการเริ่มต้นใช้งานครั้งต่อไปจะข้ามขั้นตอนนี้ทั้งหมด
`openclaw onboard recommendations refresh` จะล้างข้อเสนอที่บันทึกไว้ เพื่อให้การ
เริ่มต้นใช้งานครั้งถัดไปสแกนแอปที่ติดตั้งใหม่และสร้างข้อเสนอใหม่

เวิร์กสเปซใหม่จะเลื่อนการเลือกคำแนะนำไปยังการสนทนาในขั้นตอนบูต
หลังจากการสนทนานั้นจัดการตัวเลือกของผู้ใช้แล้ว
`openclaw onboard recommendations acknowledge` จะทำเครื่องหมายว่าข้อเสนอที่บันทึกไว้ได้รับคำตอบแล้ว
การยืนยันนี้ทำซ้ำได้โดยให้ผลลัพธ์เดิม หากการติดตั้งที่เลือกไม่สำเร็จ ให้ส่ง
ID ทึบแสงแต่ละรายการที่ล้มเหลวด้วย `--retry <id...>`; รายการที่ติดตั้งสำเร็จและรายการที่ปฏิเสธจะถูกนำออก
ส่วนรายการที่ล้มเหลวจะยังคงรอดำเนินการสำหรับการเริ่มต้นใช้งานครั้งถัดไป ID ที่ไม่รู้จัก
จะทำให้ล้มเหลวโดยไม่เปลี่ยนข้อเสนอที่บันทึกไว้ หลังจากการติดตั้ง Skills จาก ClawHub
ถูกขัดจังหวะ เป้าหมายที่มีอยู่จะถือว่าสำเร็จเฉพาะเมื่อ
`openclaw skills verify "@owner/slug"` สำเร็จสำหรับ
ID คำแนะนำเดียวกันที่ระบุผู้เผยแพร่ และผลลัพธ์ JSON รายงาน
`openclaw.resolution.source: "installed"` การตรวจสอบรีจิสทรีเพียงอย่างเดียวไม่ใช่
หลักฐานว่ามีการติดตั้งในเครื่อง มิฉะนั้น ให้คง ID นั้นไว้เป็นรายการรอดำเนินการด้วย `--retry` และ
อย่าเขียนทับ Skills ที่มีอยู่

- `--classic`: เปิดวิซาร์ดแบบทีละขั้นตอนฉบับเต็ม ไม่สามารถใช้ร่วมกับ
  `--non-interactive` ได้; ให้ละเว้น `--classic` สำหรับการตั้งค่าอัตโนมัติ
- `--flow quickstart`: เปิดวิซาร์ดแบบคลาสสิกที่มีพรอมต์ให้น้อยที่สุด และ
  สร้างโทเค็น Gateway โดยอัตโนมัติ
- `--flow manual` (นามแฝง `advanced`): เปิดวิซาร์ดแบบคลาสสิกพร้อมพรอมต์ฉบับเต็ม
  สำหรับพอร์ต การผูก และการยืนยันตัวตน
- `--flow import`: เรียกใช้ผู้ให้บริการย้ายข้อมูลที่ตรวจพบ (เช่น Hermes ผ่าน `--import-from hermes`) แสดงตัวอย่างแผน แล้วจึงนำไปใช้หลังจากยืนยัน การนำเข้าจะทำงานเฉพาะกับการตั้งค่า OpenClaw ใหม่เท่านั้น หากมีการกำหนดค่า ข้อมูลประจำตัว เซสชัน หรือสถานะเวิร์กสเปซอยู่แล้ว ให้รีเซ็ตก่อน ใช้ [`openclaw migrate`](/th/cli/migrate) สำหรับแผนการทดลองใช้งาน โหมดเขียนทับ รายงาน และการจับคู่ที่แน่นอน
- `--remote-url` และ `--remote-token`: กรอกขั้นตอน Gateway ระยะไกลแบบคลาสสิกไว้ล่วงหน้า และแทนที่ค่าระยะไกลที่บันทึกไว้สำหรับการเรียกใช้ครั้งนี้ การเปลี่ยน URL จะไม่นำข้อมูลประจำตัวที่บันทึกไว้กลับมาใช้ เว้นแต่จะส่งโทเค็นด้วย โทเค็นจะยังคงถูกปกปิดในพรอมต์ และเป็นไปตามตัวเลือกการจัดเก็บแบบข้อความธรรมดาหรือ SecretRef ที่มีอยู่ของวิซาร์ด
- `--tailscale-reset-on-exit` และ `--no-tailscale-reset-on-exit`: ควบคุมอย่างชัดเจนว่าจะรีเซ็ตการกำหนดค่า Tailscale Serve หรือ Funnel เมื่อ Gateway ออกจากการทำงานหรือไม่ การละเว้นทั้งสองรายการจะคงการตั้งค่าปัจจุบันไว้ระหว่างการเรียกใช้ซ้ำแบบไม่โต้ตอบ
- `--modern` เป็นนามแฝงเพื่อความเข้ากันได้สำหรับผู้ช่วยตั้งค่า
  OpenClaw แบบสนทนา โดยใช้เกตการอนุมานจริงเดียวกับ `openclaw setup` และ
  ยอมรับเฉพาะ `--workspace`, `--accept-risk`,
  `--non-interactive` และ `--json` แฟล็กการตั้งค่าอื่นจะถูกปฏิเสธแทนที่จะ
  ถูกละเว้นโดยไม่มีการแจ้งเตือน

## ขั้นตอนแบบมีคำแนะนำ

`openclaw onboard` แบบไม่มีอาร์กิวเมนต์จะเริ่มขั้นตอนแบบมีคำแนะนำ โดยจะแสดงประกาศด้านความปลอดภัย
จากนั้นถามคำถามหนึ่งข้อในตอนเริ่มต้น: **การเข้าถึงแบบเต็ม** (แนะนำ — การตั้งค่าจะค้นหา
แอป AI คีย์ และรันไทม์ในเครื่องโดยอัตโนมัติ) หรือ **ถามก่อน** (การตั้งค่าจะถาม
หนึ่งครั้งก่อนตรวจสอบระบบ หรือให้กำหนดค่าด้วยตนเอง) ตัวเลือกนี้
จะถูกบันทึกเป็น `wizard.accessMode` เมื่ออนุญาตให้ค้นหา การเริ่มต้นใช้งาน
จะตรวจหาการเข้าถึง AI ที่มีอยู่แล้วผ่านโมเดลที่กำหนดค่าไว้ ตัวแปรสภาพแวดล้อม
ของคีย์ API และ CLI ในเครื่องที่รองรับ แล้วทดสอบตัวเลือกที่แนะนำ
ด้วยการตอบกลับจริง หากตัวเลือกหนึ่งล้มเหลว การเริ่มต้นใช้งานจะลอง
ตัวเลือกถัดไปที่ใช้งานได้โดยไม่รบกวน และสรุปรายการที่ไม่ตอบสนองไว้ใน
บรรทัดเดียว เส้นทางที่ใช้งานได้จะถูกประกาศพร้อมตัวเลือกกดปุ่มเดียวเพื่อดู
ตัวเลือกอื่นทั้งหมดแทน

หากการตรวจหาอัตโนมัติไม่พบตัวเลือก ตัวเลือกผู้ให้บริการจะแสดง OpenAI,
Anthropic, xAI (Grok), Google และ OpenRouter ก่อน เลือก **More…** สำหรับ
ผู้ให้บริการอื่นทั้งหมดที่รองรับ โดยจัดกลุ่มตามผู้ให้บริการ จากนั้นภูมิภาค แผน และวิธีการยืนยันตัวตน
จะปรากฏในเมนูที่สอง การลงชื่อเข้าใช้ผ่านเบราว์เซอร์หรืออุปกรณ์ที่รองรับ และวิธีใช้
คีย์ API หรือโทเค็นที่ถูกปกปิด จะใช้เส้นทางการตอบกลับจริงเดียวกัน OpenClaw จะบันทึก
เฉพาะเส้นทางโมเดลที่ตรวจสอบแล้วและข้อมูลประจำตัวของเส้นทางนั้นหลังจากการทดสอบสำเร็จ
ตัวเลือกที่ล้มเหลวจะไม่แทนที่โมเดลที่กำหนดค่าไว้หรือบันทึก
ข้อมูลประจำตัวที่ทดลองใช้ เลือก **Skip for now** เพื่อออกโดยไม่เริ่ม OpenClaw และ
เรียกใช้ `openclaw onboard` อีกครั้งเมื่อพร้อม การตั้งค่าเวิร์กสเปซและ Gateway จะยังคง
ไม่เปลี่ยนแปลงจนกว่า OpenClaw จะเริ่มทำงาน

ในโหมดมีคำแนะนำ `--workspace <dir>` จะระบุเวิร์กสเปซที่ OpenClaw เสนอ
และบริบทการอนุมานที่แยกออกจากกัน โดยจะไม่ถูกบันทึกจนกว่าจะอนุมัติ
ข้อเสนอการตั้งค่า OpenClaw การเริ่มต้นใช้งานแบบคลาสสิกและแบบไม่โต้ตอบจะบันทึก
เวิร์กสเปซผ่านขั้นตอนการตั้งค่าปกติของแต่ละแบบ

หลังจากการอนุมานผ่าน การเริ่มต้นใช้งานจะตรวจหาความทรงจำจากเครื่องมือ AI ในเครื่อง
ที่รองรับ ได้แก่ หน่วยความจำอัตโนมัติของ Claude Code, หน่วยความจำแบบรวมของ Codex และไฟล์หน่วยความจำ
ของ Hermes เมื่อพบรายการใด หน้าหนึ่งจะเสนอให้คัดลอกลงในเวิร์กสเปซของเอเจนต์
ภายใต้ `memory/imports/` เพื่อให้เรียกคืนผ่านดัชนีได้ จะไม่มีการนำเข้าสิ่งใดโดยไม่ได้รับ
การยืนยัน ไฟล์ที่นำเข้าแล้วจะถูกข้าม และสามารถนำเข้า
ภายหลังได้เสมอจาก [หน้านำเข้าหน่วยความจำ](/th/web/control-ui) ใน Control UI ซึ่งมี
ขอบเขตเฉพาะหน่วยความจำแบบเดียวกัน (การเรียกใช้ [`openclaw migrate`](/th/cli/migrate) แบบเต็มจะ
ครอบคลุมกว่านั้น โดยยังสามารถนำเข้าการกำหนดค่า Skills และข้อมูลประจำตัวได้ด้วย) วิซาร์ดแบบคลาสสิก
จะแสดงหน้าเดียวกันหลังจากเตรียมเวิร์กสเปซ

หลังจากการอนุมานผ่าน (และหลังข้อเสนอการนำเข้าหน่วยความจำ) การเริ่มต้นใช้งานแบบมีคำแนะนำ
จะใช้การตั้งค่ามาตรฐานโดยอัตโนมัติ ได้แก่ เวิร์กสเปซ Gateway และเซสชัน
ซึ่งเป็นแผนเดียวกับที่แชต `openclaw setup` แบบสนทนาจะนำไปใช้เมื่อได้รับคำตอบว่า "yes" —
จากนั้นจะเสนอคำแนะนำ Plugin และ Skills จากแอปที่ติดตั้ง โดยชื่อแอป
จะถูกจับคู่ผ่านโมเดลที่กำหนดค่าและการค้นหา ClawHub และสามารถ
ปิดใช้งานขั้นตอนนี้ด้วย [`wizard.appRecommendations`](/th/gateway/configuration-reference#wizard)
ในเซสชันเดสก์ท็อป macOS, Linux หรือ Windows จากนั้นระบบจะเปิด
แดชบอร์ด Control UI ที่ผ่านการยืนยันตัวตนและรอสูงสุด 60 วินาทีให้ไคลเอนต์เบราว์เซอร์
เชื่อมต่อ บน Linux แบบไร้หน้าจอหรือผ่าน SSH ระบบจะพิมพ์
URL แดชบอร์ดที่โดดเด่นและคัดลอกวางได้ ซึ่งรวมถึงคำสั่งส่งต่อพอร์ต SSH สำหรับ Gateway
แบบลูปแบ็ก และรอสูงสุดห้านาที การเชื่อมต่อที่สำเร็จจะดำเนินการต่อในเบราว์เซอร์
ส่วน Gateway ที่เข้าถึงไม่ได้หรือหมดเวลาจะย้อนกลับไปยังทางออกผ่านเทอร์มินัลแบบเดิม
ส่ง `--tui` เพื่อข้ามการส่งต่อไปยังเบราว์เซอร์และบังคับใช้ทางออกผ่านเทอร์มินัลนั้น
หากการใช้การตั้งค่าล้มเหลว การเริ่มต้นใช้งานจะย้อนกลับไปยังแชต OpenClaw
แบบสนทนาเพื่อดำเนินการให้เสร็จแบบโต้ตอบ ช่องทาง เอเจนต์
Plugin และคุณสมบัติเสริมอื่น ๆ ยังคงจัดการผ่านแชต OpenClaw โดยให้เรียกใช้
`openclaw` และใช้ `open channel wizard for <channel>` เพื่อส่งต่อการรวบรวม
ข้อมูลประจำตัวของช่องทางไปยังวิซาร์ดเทอร์มินัลที่ปกปิดข้อมูล หากต้องการเปลี่ยน
ผู้ให้บริการโมเดลหรือการยืนยันตัวตน ให้ออกจาก OpenClaw แล้วเรียกใช้ `openclaw onboard`;
OpenClaw จะไม่เปิดขั้นตอนผู้ให้บริการแบบมีคำแนะนำหรือแบบคลาสสิก

ในการติดตั้งที่กำหนดค่าแล้ว การเรียกใช้ `openclaw onboard` อีกครั้งจะตรวจสอบ
โมเดลเริ่มต้นปัจจุบันก่อน ดังนั้นขั้นตอนเดียวกันจึงทำหน้าที่เป็นการตรวจสอบและซ่อมแซม —
โดยจะไม่ใช้การตั้งค่าใหม่ ติดตั้งใหม่ หรือเริ่มบริการ Gateway ใหม่
หากการตรวจสอบล้มเหลว โมเดลที่กำหนดค่าไว้จะไม่ถูกแทนที่โดยอัตโนมัติ —
การเริ่มต้นใช้งานจะหยุดและถามว่าจะดำเนินการต่ออย่างไร การตรวจสอบจะทำงานนอก
เวิร์กสเปซ ดังนั้นโมเดลที่มาจาก Plugin ของเวิร์กสเปซอาจล้มเหลวที่นี่แต่ยังคง
ทำงานในเอเจนต์ได้
ใช้ `openclaw onboard --classic` สำหรับการยืนยันตัวตนเฉพาะผู้ให้บริการ ช่องทาง Skills
การตั้งค่า Gateway ระยะไกล การนำเข้า หรือการควบคุม Gateway แบบเต็ม สำหรับการตั้งค่าและซ่อมแซม
แบบสนทนาที่ไม่เกี่ยวกับการอนุมาน ให้เรียกใช้ `openclaw setup`; `openclaw onboard
--modern` เป็นนามแฝงเพื่อความเข้ากันได้ที่ผ่านเกตการอนุมานเดียวกัน วิซาร์ดแบบคลาสสิก
สามารถเลือกตรวจสอบโมเดลเริ่มต้นด้วยการตอบกลับจริงได้ แต่
OpenClaw จะไม่เริ่มทำงานจนกว่าการตรวจสอบการอนุมานจริงของตัวเองจะผ่าน

ในเทอร์มินัลแบบโต้ตอบ `openclaw` แบบไม่มีอาร์กิวเมนต์ (ไม่มีคำสั่งย่อย) จะกำหนดเส้นทางตามสถานะ
การกำหนดค่า:

- หากไฟล์การกำหนดค่าที่ใช้งานอยู่ไม่มีอยู่หรือไม่มีการตั้งค่าที่ผู้ใช้กำหนด (ว่างเปล่าหรือ
  มีเฉพาะข้อมูลเมตา) ระบบจะเริ่มการเริ่มต้นใช้งานแบบมีคำแนะนำ
- หากไฟล์การกำหนดค่ามีอยู่แต่ไม่ผ่านการตรวจสอบ ระบบจะเริ่มขั้นตอน
  การเริ่มต้นใช้งานแบบคลาสสิกพร้อมคำแนะนำ `openclaw doctor` OpenClaw ต้องมี
  การอนุมานที่ใช้งานได้ และจะไม่ถูกใช้เพื่อซ่อมแซมสถานะก่อนการอนุมานนี้
- หากไฟล์การกำหนดค่าถูกต้อง ระบบจะเปิด TUI ของเอเจนต์ตามปกติ Gateway ที่กำหนดค่าไว้
  ซึ่งเข้าถึงได้และมีเอเจนต์กับโมเดล จะเข้าสู่ UI นั้นโดยตรงโดยไม่ผ่าน
  การเริ่มต้นใช้งานหรือ OpenClaw ในการติดตั้งที่กำหนดค่าแล้ว ให้เข้าถึง OpenClaw ด้วย
  `/openclaw` ภายใน TUI หรือ `openclaw setup`

ระบบยอมรับ `ws://` แบบข้อความธรรมดาสำหรับลูปแบ็ก ลิเทอรัล IP ส่วนตัว, `.local` และ URL Gateway ของ Tailnet `*.ts.net` สำหรับชื่อ DNS ส่วนตัวอื่นที่เชื่อถือได้ ให้ตั้งค่า `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` ในสภาพแวดล้อมของกระบวนการเริ่มต้นใช้งาน

## รีเซ็ต

```bash
openclaw onboard --reset
openclaw onboard --reset --reset-scope full
```

`--reset` จะล้างสถานะก่อนเรียกใช้การตั้งค่า `--reset-scope` ควบคุมขอบเขตการล้าง ได้แก่ `config` (เฉพาะการกำหนดค่า), `config+creds+sessions` (ค่าเริ่มต้นเมื่อส่ง `--reset` โดยไม่มีขอบเขต) หรือ `full` (รีเซ็ตเวิร์กสเปซด้วย) การรีเซ็ตเวิร์กสเปซจะเกิดขึ้นเฉพาะเมื่อใช้ `--reset-scope full`

## ภาษาและภูมิภาค

การเริ่มต้นใช้งานแบบโต้ตอบจะใช้ภาษาของวิซาร์ด CLI สำหรับข้อความการตั้งค่าแบบคงที่ โดยใช้ค่าที่ไม่ว่างค่าแรกตามลำดับต่อไปนี้:

1. `OPENCLAW_LOCALE`
2. `LC_ALL`
3. `LC_MESSAGES`
4. `LANG`
5. ใช้ภาษาอังกฤษเป็นค่าเริ่มต้น

ภาษาของวิซาร์ดที่รองรับคือ `en`, `zh-CN` และ `zh-TW` ค่าภาษาอาจใช้รูปแบบขีดล่างหรือส่วนต่อท้าย POSIX เช่น `zh_CN.UTF-8` ชื่อผลิตภัณฑ์ ชื่อคำสั่ง คีย์การกำหนดค่า URL, ID ผู้ให้บริการ, ID โมเดล และป้ายกำกับ Plugin/ช่องทางจะคงเดิม

```bash
OPENCLAW_LOCALE=zh-CN openclaw onboard
OPENCLAW_LOCALE=en openclaw onboard # กำหนดให้ใช้ภาษาอังกฤษอย่างชัดเจน
```

## การตั้งค่าแบบไม่โต้ตอบ

`--non-interactive` ต้องใช้ `--accept-risk` (เป็นการยอมรับว่าเอเจนต์มีความสามารถสูงและการเข้าถึงระบบทั้งหมดมีความเสี่ยง) `--mode` มีค่าเริ่มต้นเป็น `local`

```bash
openclaw onboard --non-interactive \
  --auth-choice custom-api-key \
  --custom-base-url "https://llm.example.com/v1" \
  --custom-model-id "foo-large" \
  --custom-api-key "$CUSTOM_API_KEY" \
  --secret-input-mode plaintext \
  --custom-compatibility openai \
  --custom-image-input
```

`--custom-api-key` เป็นตัวเลือกเสริม หากละไว้ การเริ่มต้นใช้งานจะตรวจสอบ `CUSTOM_API_KEY` ใน env โดยอัตโนมัติ OpenClaw ทำเครื่องหมาย ID ของโมเดลด้านการมองเห็นที่ใช้กันทั่วไป (GPT-4o/4.1/5.x, Claude 3/4, Gemini, Qwen-VL, LLaVA, Pixtral และโมเดลที่คล้ายกัน) ว่ารองรับรูปภาพโดยอัตโนมัติ ส่ง `--custom-image-input` สำหรับ ID ของโมเดลด้านการมองเห็นแบบกำหนดเองที่ไม่รู้จัก หรือ `--custom-text-input` เพื่อบังคับใช้เมทาดาทาแบบข้อความเท่านั้น ใช้ `--custom-compatibility openai-responses` สำหรับปลายทางที่เข้ากันได้กับ OpenAI ซึ่งรองรับ `/v1/responses` แต่ไม่รองรับ `/v1/chat/completions` โดยค่าที่ใช้ได้คือ `openai` (ค่าเริ่มต้น), `openai-responses`, `anthropic`

LM Studio ยังมีแฟล็กคีย์เฉพาะผู้ให้บริการด้วย:

```bash
openclaw onboard --non-interactive \
  --auth-choice lmstudio \
  --custom-base-url "http://localhost:1234/v1" \
  --custom-model-id "qwen/qwen3.5-9b" \
  --lmstudio-api-key "$LM_API_TOKEN" \
  --accept-risk
```

Ollama แบบไม่โต้ตอบ:

```bash
openclaw onboard --non-interactive \
  --auth-choice ollama \
  --custom-base-url "http://ollama-host:11434" \
  --custom-model-id "qwen3.5:27b" \
  --accept-risk
```

`--custom-base-url` มีค่าเริ่มต้นเป็น `http://127.0.0.1:11434` ส่วน `--custom-model-id` เป็นตัวเลือกเสริม หากละไว้ การเริ่มต้นใช้งานจะใช้ค่าเริ่มต้นที่ Ollama แนะนำ ID ของโมเดลคลาวด์ เช่น `kimi-k2.5:cloud` ก็ใช้ได้ที่นี่เช่นกัน

จัดเก็บคีย์ของผู้ให้บริการเป็นการอ้างอิงแทนข้อความธรรมดา:

```bash
openclaw onboard --non-interactive \
  --auth-choice openai-api-key \
  --secret-input-mode ref \
  --accept-risk
```

เมื่อใช้ `--secret-input-mode ref` การเริ่มต้นใช้งานจะเขียนการอ้างอิงที่อิงกับ env แทนค่าคีย์แบบข้อความธรรมดา สำหรับผู้ให้บริการที่อิงกับโปรไฟล์การยืนยันตัวตน ระบบจะเขียน `keyRef: { source: "env", provider: "default", id: <envVar> }` ส่วนผู้ให้บริการแบบกำหนดเองจะเขียน `models.providers.<id>.apiKey` ด้วยวิธีเดียวกัน (ตัวอย่างเช่น `{ source: "env", provider: "default", id: "CUSTOM_API_KEY" }`) ข้อกำหนด: ตั้งค่าตัวแปร env ของผู้ให้บริการในสภาพแวดล้อมของกระบวนการเริ่มต้นใช้งาน (ตัวอย่างเช่น `OPENAI_API_KEY`) และอย่าส่งแฟล็กคีย์แบบอินไลน์ด้วย เว้นแต่จะตั้งค่าตัวแปร env นั้นแล้ว หากมีค่าแฟล็กโดยไม่มีตัวแปร env ที่ตรงกัน ระบบจะหยุดทันทีพร้อมคำแนะนำ

### การยืนยันตัวตนของ Gateway (แบบไม่โต้ตอบ)

- `--gateway-auth token --gateway-token <token>` จัดเก็บโทเค็นแบบข้อความธรรมดา `token` เป็นโหมดการยืนยันตัวตนเริ่มต้น
- `--gateway-auth token --gateway-token-ref-env <name>` จัดเก็บ `gateway.auth.token` เป็น SecretRef จาก env โดยต้องมีตัวแปร env ชื่อนั้นที่ไม่ว่างในสภาพแวดล้อมของกระบวนการเริ่มต้นใช้งาน
- `--gateway-token` และ `--gateway-token-ref-env` ใช้ร่วมกันไม่ได้
- เมื่อใช้ `--install-daemon`: ระบบจะตรวจสอบ `gateway.auth.token` ที่จัดการด้วย SecretRef แต่จะไม่บันทึกค่าข้อความธรรมดาที่แก้ไขแล้วไว้ในเมทาดาทาสภาพแวดล้อมของบริการผู้ควบคุม หากไม่สามารถแก้ไขการอ้างอิงได้ การติดตั้งจะปฏิเสธการดำเนินการพร้อมคำแนะนำในการแก้ไข หากกำหนดค่าทั้ง `gateway.auth.token` และ `gateway.auth.password` แต่ไม่ได้ตั้งค่า `gateway.auth.mode` การติดตั้งจะถูกระงับจนกว่าจะกำหนดโหมดอย่างชัดเจน
- การเริ่มต้นใช้งานภายในเครื่องจะเขียน `gateway.mode="local"` ลงในการกำหนดค่า หากไฟล์การกำหนดค่าในภายหลังไม่มี `gateway.mode` แสดงว่าการกำหนดค่าเสียหายหรือมีการแก้ไขด้วยตนเองไม่สมบูรณ์ ไม่ใช่ทางลัดของโหมดภายในเครื่องที่ถูกต้อง
- การเริ่มต้นใช้งานภายในเครื่องจะติดตั้ง Plugin ที่ดาวน์โหลดได้ซึ่งเส้นทางการตั้งค่าที่เลือกต้องใช้ (ตัวอย่างเช่น Plugin รันไทม์ Codex หรือ Copilot สำหรับตัวเลือกการยืนยันตัวตนเหล่านั้น) การเริ่มต้นใช้งานระยะไกลจะเขียนเฉพาะข้อมูลการเชื่อมต่อสำหรับ Gateway ระยะไกลเท่านั้น และจะไม่ติดตั้งแพ็กเกจ Plugin ภายในเครื่อง
- `--allow-unconfigured` เป็นช่องทางหลีกเลี่ยง `openclaw gateway run` ที่แยกต่างหาก และไม่ได้อนุญาตให้การเริ่มต้นใช้งานข้าม `gateway.mode`

```bash
export OPENAI_API_KEY="your-provider-key"
export OPENCLAW_GATEWAY_TOKEN="your-token"
openclaw onboard --non-interactive \
  --mode local \
  --auth-choice openai-api-key \
  --secret-input-mode ref \
  --gateway-auth token \
  --gateway-token-ref-env OPENCLAW_GATEWAY_TOKEN \
  --accept-risk
```

### สถานะความพร้อมของ Gateway ภายในเครื่อง

- หากไม่ส่ง `--skip-health` การเริ่มต้นใช้งานจะรอจนสามารถเข้าถึง Gateway ภายในเครื่องได้ก่อนจึงจะออกโดยสำเร็จ
- `--install-daemon` จะเริ่มเส้นทางการติดตั้ง Gateway ที่มีการจัดการก่อน หากไม่มีค่านี้ Gateway ภายในเครื่องต้องทำงานอยู่แล้ว (ตัวอย่างเช่น `openclaw gateway run`)
- `--skip-health` จะข้ามการรอ หากต้องการเพียงเขียนการกำหนดค่า/พื้นที่ทำงาน/ข้อมูลเริ่มต้นในระบบอัตโนมัติ
- `--skip-bootstrap` จะตั้งค่า `agents.defaults.skipBootstrap: true` และข้ามการสร้าง `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` และ `BOOTSTRAP.md`
- บน Windows แบบเนทีฟ `--install-daemon` จะลองใช้ Scheduled Tasks ก่อน และหากการสร้างงานถูกปฏิเสธ จะเปลี่ยนไปใช้รายการเข้าสู่ระบบในโฟลเดอร์ Startup ระดับผู้ใช้

### โหมดการอ้างอิงแบบโต้ตอบ

- เมื่อได้รับข้อความแจ้ง ให้เลือก **ใช้การอ้างอิงข้อมูลลับ** จากนั้นเลือก **ตัวแปรสภาพแวดล้อม** หรือผู้ให้บริการข้อมูลลับที่กำหนดค่าไว้ (`file` หรือ `exec`)
- การเริ่มต้นใช้งานจะตรวจสอบเบื้องต้นอย่างรวดเร็วก่อนบันทึกการอ้างอิง และให้ลองใหม่ได้หากล้มเหลว

# การเลือกปลายทางโดยไม่มีข้อความแจ้ง

```

```

## แฟล็กแบบไม่โต้ตอบเพิ่มเติม

การยืนยันตัวตนของโมเดลด้วยโทเค็น (ใช้ร่วมกับ `--auth-choice token`):

| แฟล็ก                            | คำอธิบาย                                                                                                                 |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `--token-provider <id>`         | ID ผู้ให้บริการโทเค็นที่ออกโทเค็น                                                                                         |
| `--token <token>`               | ค่าโทเค็นสำหรับการยืนยันตัวตนของโมเดล                                                                                        |
| `--token-profile-id <id>`       | ID โปรไฟล์การยืนยันตัวตน (ค่าเริ่มต้น `<provider>:manual`; ขั้นตอนบางรายการที่ผู้ให้บริการเป็นเจ้าของจะใช้ค่าเริ่มต้นของตนเอง เช่น `anthropic:default`) |
| `--token-expires-in <duration>` | ระยะเวลาหมดอายุของโทเค็นที่เป็นตัวเลือกเสริม (เช่น `365d`, `12h`)                                                                         |

Cloudflare AI Gateway: `--cloudflare-ai-gateway-account-id <id>`, `--cloudflare-ai-gateway-gateway-id <id>`

การควบคุมการติดตั้งดีมอน: `--no-install-daemon` / `--skip-daemon` (นามแฝง ข้ามการติดตั้งบริการ Gateway), `--daemon-runtime <node>`

Skills: `--node-manager <npm|pnpm|bun>` (ค่าเริ่มต้น `npm`), `--skip-skills`

การตั้งค่า UI และฮุก: `--skip-ui` (ข้ามข้อความแจ้งของ Control UI/TUI), `--skip-hooks` (ข้ามการตั้งค่า Webhook/ฮุก), `--skip-channels`, `--skip-search`

เอาต์พุต: `--suppress-gateway-token-output` จะระงับเอาต์พุตของ Gateway/UI ที่มีโทเค็น (คำใบ้เกี่ยวกับโทเค็น, URL เข้าสู่ระบบอัตโนมัติที่ฝังโทเค็น และการเปิด Control UI โดยอัตโนมัติ) ซึ่งมีประโยชน์ในเทอร์มินัลที่ใช้ร่วมกันและ CI

<Note>
`--json` ไม่ได้หมายถึงโหมดไม่โต้ตอบในการเริ่มต้นใช้งานแบบมีคำแนะนำหรือแบบคลาสสิก
เมื่อใช้ `--modern` JSON จะแสดงภาพรวม OpenClaw แบบครั้งเดียวและออกหลังจาก
ผลลัพธ์เดียวนั้น ใช้ `--non-interactive` สำหรับสคริปต์อื่น
</Note>

## การกรองผู้ให้บริการล่วงหน้า

เมื่อตัวเลือกการยืนยันตัวตนระบุผู้ให้บริการที่ต้องการ การเริ่มต้นใช้งานจะกรองตัวเลือกโมเดลเริ่มต้นและรายการอนุญาตล่วงหน้าให้เหลือเฉพาะโมเดลของผู้ให้บริการนั้น ตัวกรองยังจับคู่ผู้ให้บริการรายอื่นที่ Plugin เดียวกันเป็นเจ้าของ ซึ่งครอบคลุมตัวแปร Coding Plan เช่น `volcengine`/`volcengine-plan` และ `byteplus`/`byteplus-plan` หากตัวกรองผู้ให้บริการที่ต้องการไม่พบโมเดลที่โหลดไว้ การเริ่มต้นใช้งานจะกลับไปใช้แค็ตตาล็อกที่ไม่ได้กรองแทนการปล่อยให้ตัวเลือกว่างเปล่า

## ข้อความแจ้งติดตามผลสำหรับการค้นหาเว็บ

ผู้ให้บริการค้นหาเว็บบางรายจะเรียกข้อความแจ้งติดตามผลเฉพาะผู้ให้บริการระหว่างการเริ่มต้นใช้งาน:

- **Grok** สามารถเสนอการตั้งค่า `x_search` แบบตัวเลือกเสริมโดยใช้การยืนยันตัวตน xAI เดียวกันและตัวเลือกโมเดล `x_search`
- **Kimi** สามารถถามภูมิภาคของ Moonshot API (`api.moonshot.ai` เทียบกับ `api.moonshot.cn`) และโมเดลค้นหาเว็บ Kimi เริ่มต้น

## ลักษณะการทำงานอื่นๆ

- ลักษณะการทำงานของขอบเขต DM ในการเริ่มต้นใช้งานภายในเครื่อง: [ข้อมูลอ้างอิงการตั้งค่า CLI](/th/start/wizard-cli-reference#outputs-and-internals)
- การเริ่มแชตครั้งแรกที่เร็วที่สุด: `openclaw dashboard` (Control UI โดยไม่ตั้งค่าช่องทาง)
- ผู้ให้บริการแบบกำหนดเอง: เชื่อมต่อปลายทางใดๆ ที่เข้ากันได้กับ OpenAI หรือ Anthropic รวมถึงผู้ให้บริการแบบโฮสต์ที่ไม่ได้ระบุไว้ ใช้ความเข้ากันได้แบบ **ไม่ทราบ** เพื่อตรวจหาโดยอัตโนมัติผ่านการทดสอบจริง
- หากตรวจพบสถานะ Hermes การเริ่มต้นใช้งานจะเสนอขั้นตอนการย้ายข้อมูล (ดู `--flow import` ด้านบน)

## คำสั่งติดตามผลที่ใช้บ่อย

ใช้ `openclaw configure` ภายหลังสำหรับการเปลี่ยนแปลงแบบเจาะจงที่ไม่ใช้การอนุมาน และใช้ `openclaw
channels add` สำหรับการตั้งค่าเฉพาะช่องทาง สำหรับการเปลี่ยนแปลงผู้ให้บริการโมเดลหรือเส้นทางการยืนยันตัวตน
ให้เรียกใช้ `openclaw onboard` แทน

```bash
openclaw channels add
openclaw configure
openclaw agents add <name>
```
