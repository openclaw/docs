# `openclaw status`

การวินิจฉัยสำหรับช่องทาง + เซสชัน

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

หมายเหตุ:

- `--deep` เรียกใช้การตรวจสอบแบบสด (WhatsApp Web + Telegram + Discord + Slack + Signal)
- `openclaw status` แบบธรรมดายังคงอยู่บนเส้นทางอ่านอย่างเดียวที่รวดเร็ว และทำเครื่องหมายหน่วยความจำเป็น `not checked` แทนที่จะเป็นไม่พร้อมใช้งานเมื่อข้ามการตรวจสอบหน่วยความจำ การตรวจสอบความปลอดภัยแบบหนัก การตรวจสอบความเข้ากันได้ของ plugin และการตรวจสอบเวกเตอร์หน่วยความจำจะปล่อยให้ `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` และ `openclaw memory status --deep` จัดการ
- `status --json --all` รายงานรายละเอียดหน่วยความจำจากรันไทม์ของ plugin หน่วยความจำที่ใช้งานอยู่ซึ่งเลือกโดย `plugins.slots.memory` plugin หน่วยความจำแบบกำหนดเองสามารถปล่อยให้ `agents.defaults.memorySearch.enabled` ในตัวปิดใช้งานไว้ และยังคงรายงานไฟล์ ชังก์ เวกเตอร์ และสถานะ FTS ของตนเองได้
- `--usage` พิมพ์หน้าต่างการใช้งานของผู้ให้บริการที่ปรับให้อยู่ในรูปมาตรฐานเป็น `X% left`
- เอาต์พุตสถานะเซสชันแยก `Execution:` ออกจาก `Runtime:` `Execution` คือเส้นทาง sandbox (`direct`, `docker/*`) ขณะที่ `Runtime` บอกคุณว่าเซสชันกำลังใช้ `OpenClaw Pi Default`, `OpenAI Codex`, แบ็กเอนด์ CLI หรือแบ็กเอนด์ ACP เช่น `codex (acp/acpx)` ดู [รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes) สำหรับความแตกต่างระหว่างผู้ให้บริการ/โมเดล/รันไทม์
- ฟิลด์ดิบ `usage_percent` / `usagePercent` ของ MiniMax คือโควตาที่เหลืออยู่ ดังนั้น OpenClaw จะกลับค่าเหล่านั้นก่อนแสดงผล ฟิลด์แบบอิงจำนวนจะมีสิทธิ์เหนือกว่าเมื่อมีอยู่ คำตอบ `model_remains` จะเลือกใช้รายการโมเดลแชท สร้างป้ายหน้าต่างจากเวลาเมื่อจำเป็น และรวมชื่อโมเดลไว้ในป้ายแผน
- เมื่อ snapshot ของเซสชันปัจจุบันมีข้อมูลเบาบาง `/status` สามารถเติมตัวนับโทเค็นและแคชย้อนหลังจากบันทึกการใช้งาน transcript ล่าสุดได้ ค่าสดที่ไม่เป็นศูนย์ที่มีอยู่ยังคงมีสิทธิ์เหนือค่าทางเลือกจาก transcript
- `/status` รวม uptime ของกระบวนการ Gateway และ uptime ของระบบโฮสต์แบบย่อ
- ตัวสำรองจาก transcript ยังสามารถกู้คืนป้ายโมเดลรันไทม์ที่ใช้งานอยู่เมื่อรายการเซสชันสดไม่มีข้อมูลนี้ได้ หากโมเดลจาก transcript นั้นต่างจากโมเดลที่เลือกไว้ status จะคำนวณ context window ตามโมเดลรันไทม์ที่กู้คืนมาแทนโมเดลที่เลือกไว้
- สำหรับการนับขนาดพรอมป์ ตัวสำรองจาก transcript จะเลือกยอดรวมที่มุ่งเน้นพรอมป์ซึ่งมีค่ามากกว่าเมื่อ metadata ของเซสชันหายไปหรือมีค่าน้อยกว่า เพื่อให้เซสชันผู้ให้บริการแบบกำหนดเองไม่ยุบเหลือการแสดงโทเค็นเป็น `0`
- เอาต์พุตรวมที่เก็บเซสชันต่อเอเจนต์เมื่อมีการกำหนดค่าเอเจนต์หลายตัว
- ภาพรวมรวมสถานะการติดตั้ง/รันไทม์ของบริการโฮสต์ Gateway + node เมื่อพร้อมใช้งาน
- ภาพรวมรวมช่องทางอัปเดต + git SHA (สำหรับ source checkouts)
- ข้อมูลอัปเดตแสดงในภาพรวม หากมีอัปเดตพร้อมใช้งาน status จะพิมพ์คำแนะนำให้เรียกใช้ `openclaw update` (ดู [การอัปเดต](/th/install/updating))
- พื้นผิวสถานะแบบอ่านอย่างเดียว (`status`, `status --json`, `status --all`) จะแก้ไข SecretRefs ที่รองรับสำหรับเส้นทาง config เป้าหมายเมื่อทำได้
- หากมีการกำหนดค่า SecretRef ของช่องทางที่รองรับไว้แต่ไม่พร้อมใช้งานในเส้นทางคำสั่งปัจจุบัน status จะยังคงเป็นแบบอ่านอย่างเดียวและรายงานเอาต์พุตที่ลดระดับแทนการแครช เอาต์พุตสำหรับมนุษย์จะแสดงคำเตือน เช่น “configured token unavailable in this command path” และเอาต์พุต JSON จะรวม `secretDiagnostics`
- เมื่อการแก้ไข SecretRef เฉพาะคำสั่งสำเร็จ status จะเลือกใช้ snapshot ที่แก้ไขแล้ว และล้างตัวทำเครื่องหมายชั่วคราวของช่องทางว่า “secret unavailable” ออกจากเอาต์พุตสุดท้าย
- `status --all` รวมแถวภาพรวม Secrets และส่วนวินิจฉัยที่สรุปข้อมูลวินิจฉัย secret (ตัดให้สั้นลงเพื่อให้อ่านง่าย) โดยไม่หยุดการสร้างรายงาน
