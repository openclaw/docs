การวินิจฉัยสำหรับช่องทาง + เซสชัน

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

หมายเหตุ:

- `--deep` เรียกใช้การตรวจสอบแบบสด (WhatsApp Web + Telegram + Discord + Slack + Signal)
- `openclaw status` แบบปกติจะอยู่บนเส้นทางอ่านอย่างเดียวที่รวดเร็ว และทำเครื่องหมายหน่วยความจำเป็น `not checked` แทน unavailable เมื่อข้ามการตรวจสอบหน่วยความจำ การตรวจสอบความปลอดภัยแบบหนัก, ความเข้ากันได้ของ Plugin และการตรวจสอบเวกเตอร์หน่วยความจำจะปล่อยให้ `openclaw status --all`, `openclaw status --deep`, `openclaw security audit` และ `openclaw memory status --deep` จัดการ
- `status --json --all` รายงานรายละเอียดหน่วยความจำจากรันไทม์ Plugin หน่วยความจำที่ใช้งานอยู่ซึ่งเลือกโดย `plugins.slots.memory` Plugin หน่วยความจำแบบกำหนดเองสามารถปล่อยให้ `agents.defaults.memorySearch.enabled` ในตัวถูกปิดใช้งานไว้ และยังคงรายงานไฟล์, ชังก์, เวกเตอร์ และสถานะ FTS ของตัวเองได้
- `--usage` พิมพ์หน้าต่างการใช้งานของผู้ให้บริการที่ทำให้เป็นมาตรฐานแล้วในรูปแบบ `X% left`
- เอาต์พุตสถานะเซสชันแยก `Execution:` ออกจาก `Runtime:` `Execution` คือเส้นทางแซนด์บ็อกซ์ (`direct`, `docker/*`) ส่วน `Runtime` จะบอกว่าเซสชันกำลังใช้ `OpenClaw Pi Default`, `OpenAI Codex`, แบ็กเอนด์ CLI หรือแบ็กเอนด์ ACP เช่น `codex (acp/acpx)` ดู [รันไทม์ของเอเจนต์](/th/concepts/agent-runtimes) สำหรับความแตกต่างระหว่างผู้ให้บริการ/โมเดล/รันไทม์
- ฟิลด์ดิบ `usage_percent` / `usagePercent` ของ MiniMax คือโควต้าที่เหลืออยู่ ดังนั้น OpenClaw จะกลับค่าก่อนแสดงผล ฟิลด์แบบนับจำนวนจะมีลำดับความสำคัญเมื่อมีอยู่ การตอบกลับ `model_remains` จะเลือกเอนทรีโมเดลแชทก่อน, อนุมานป้ายกำกับหน้าต่างจากไทม์สแตมป์เมื่อจำเป็น และรวมชื่อโมเดลไว้ในป้ายกำกับแผน
- เมื่อสแนปช็อตเซสชันปัจจุบันมีข้อมูลน้อย `/status` สามารถเติมตัวนับโทเค็นและแคชย้อนหลังจากบันทึกการใช้งานทรานสคริปต์ล่าสุดได้ ค่าจริงแบบสดที่ไม่เป็นศูนย์ที่มีอยู่ยังคงมีลำดับความสำคัญเหนือค่ fallback จากทรานสคริปต์
- `/status` รวมเวลาทำงานของโปรเซส Gateway และเวลาทำงานของระบบโฮสต์แบบย่อ
- การ fallback จากทรานสคริปต์ยังสามารถกู้คืนป้ายกำกับโมเดลรันไทม์ที่ใช้งานอยู่ได้เมื่อเอนทรีเซสชันแบบสดไม่มีข้อมูลนี้ หากโมเดลในทรานสคริปต์นั้นต่างจากโมเดลที่เลือก สถานะจะ resolve หน้าต่างบริบทเทียบกับโมเดลรันไทม์ที่กู้คืนมาแทนโมเดลที่เลือก
- สำหรับการนับขนาดพรอมป์ การ fallback จากทรานสคริปต์จะเลือกยอดรวมที่เน้นพรอมป์ซึ่งมีค่ามากกว่าเมื่อเมตาดาต้าเซสชันหายไปหรือมีค่าน้อยกว่า เพื่อให้เซสชันผู้ให้บริการแบบกำหนดเองไม่ยุบเหลือการแสดงโทเค็นเป็น `0`
- เอาต์พุตรวมที่เก็บเซสชันต่อเอเจนต์เมื่อมีการกำหนดค่าเอเจนต์หลายตัว
- ภาพรวมรวมสถานะการติดตั้ง/รันไทม์ของบริการโฮสต์ Gateway + node เมื่อมีข้อมูล
- ภาพรวมรวมช่องทางอัปเดต + git SHA (สำหรับเช็กเอาต์จากซอร์ส)
- ข้อมูลอัปเดตจะแสดงในภาพรวม หากมีอัปเดตพร้อมใช้งาน สถานะจะพิมพ์คำแนะนำให้เรียกใช้ `openclaw update` (ดู [การอัปเดต](/th/install/updating))
- ความล้มเหลวในการรีเฟรชราคาของโมเดลจะแสดงเป็นคำเตือนราคาที่เป็นทางเลือก สิ่งเหล่านี้
  ไม่ได้หมายความว่า Gateway หรือช่องทางมีปัญหา
- พื้นผิวสถานะแบบอ่านอย่างเดียว (`status`, `status --json`, `status --all`) จะ resolve SecretRefs ที่รองรับสำหรับเส้นทางคอนฟิกเป้าหมายเมื่อทำได้
- หากมีการกำหนดค่า SecretRef ของช่องทางที่รองรับแต่ไม่พร้อมใช้งานในเส้นทางคำสั่งปัจจุบัน สถานะจะยังคงเป็นอ่านอย่างเดียวและรายงานเอาต์พุตแบบ degraded แทนที่จะขัดข้อง เอาต์พุตสำหรับมนุษย์จะแสดงคำเตือน เช่น "โทเค็นที่กำหนดค่าไว้ไม่พร้อมใช้งานในเส้นทางคำสั่งนี้" และเอาต์พุต JSON จะรวม `secretDiagnostics`
- เมื่อการ resolve SecretRef ภายในคำสั่งสำเร็จ สถานะจะเลือกสแนปช็อตที่ resolve แล้ว และล้างเครื่องหมายช่องทาง "secret unavailable" ชั่วคราวออกจากเอาต์พุตสุดท้าย
- `status --all` รวมแถวภาพรวม Secrets และส่วนการวินิจฉัยที่สรุปการวินิจฉัย secret (ตัดให้สั้นลงเพื่อให้อ่านง่าย) โดยไม่หยุดการสร้างรายงาน
