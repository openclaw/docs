# `openclaw dashboard`

เปิด Control UI โดยใช้การยืนยันตัวตนปัจจุบัน

```bash
openclaw dashboard
openclaw dashboard --no-open
openclaw dashboard --json
openclaw dashboard --yes
```

- `--no-open`: แสดง URL แต่ไม่เปิดเบราว์เซอร์
- `--json`: แสดงออบเจ็กต์การเชื่อมต่อหนึ่งรายการในรูปแบบที่เครื่องอ่านได้ โดยไม่เปิดเบราว์เซอร์ ไม่ใช้คลิปบอร์ด ไม่แสดงพรอมต์ และไม่เริ่ม Gateway
- `--yes`: เริ่ม/ติดตั้ง Gateway โดยไม่แสดงพรอมต์เมื่อจำเป็น

## เอาต์พุตที่เครื่องอ่านได้

ใช้ `--json` สำหรับการผสานรวมกับเดสก์ท็อปและสคริปต์ที่ต้องใช้ URL ของ Control UI ที่ผ่านการแก้ค่าแล้ว:

```bash
openclaw dashboard --json
```

การตอบกลับประกอบด้วย `url`, `httpUrl`, `wsUrl`, `port` และ `tokenIncluded` หาก Gateway ยังไม่พร้อม คำสั่งจะคืนค่า `{"ok":false,"reason":"..."}` และออกด้วยรหัสที่ไม่ใช่ศูนย์ โทเค็นที่จัดการด้วย SecretRef จะไม่รวมอยู่ใน `url`

หมายเหตุ:

- แก้ค่า SecretRef ของ `gateway.auth.token` ที่กำหนดค่าไว้เมื่อทำได้
- เป็นไปตาม `gateway.tls.enabled`: Gateway ที่เปิดใช้ TLS จะแสดง/เปิด URL ของ Control UI แบบ `https://` และเชื่อมต่อผ่าน `wss://`
- สำหรับการผูกกับ `lan` หรือไวลด์การ์ด `custom` การเปิดบนโฮสต์เดียวกันจะใช้ลูปแบ็กเสมอ เนื่องจากไวลด์การ์ดไม่ใช่ปลายทางของเบราว์เซอร์ การผูกแบบข้อความธรรมดาของ `tailnet` และ `custom` จะใช้ `127.0.0.1` เช่นกัน เพื่อให้เบราว์เซอร์มีบริบทที่ปลอดภัย ส่วนโฮสต์เฉพาะที่เปิดใช้ TLS จะคงที่อยู่ที่กำหนดค่าไว้เพื่อให้ชื่อใบรับรองตรงกัน
- ก่อนส่ง URL ลูปแบ็กที่ผ่านการยืนยันตัวตนสำหรับการผูกกับอินเทอร์เฟซเฉพาะ คำสั่งจะตรวจสอบอินเทอร์เฟซที่กำหนดค่าไว้และยืนยันว่าอินเทอร์เฟซนั้นกับ `127.0.0.1` อยู่ภายใต้กระบวนการ Gateway เดียวกัน หากไม่สามารถระบุความเป็นเจ้าของลิสเทนเนอร์ได้อย่างชัดเจน ระบบจะปฏิเสธอย่างปลอดภัยพร้อมคำแนะนำเกี่ยวกับสถานะ
- สำหรับโทเค็นที่จัดการด้วย SecretRef (ไม่ว่าจะแก้ค่าแล้วหรือยังไม่ได้แก้ค่า) URL ที่แสดง คัดลอก หรือเปิดจะไม่มีโทเค็นอยู่ด้วยเสมอ เพื่อไม่ให้ข้อมูลลับภายนอกรั่วไหลไปยังเอาต์พุตของเทอร์มินัล ประวัติคลิปบอร์ด หรืออาร์กิวเมนต์สำหรับเปิดเบราว์เซอร์
- หาก `gateway.auth.token` จัดการด้วย SecretRef แต่ยังไม่ได้แก้ค่า คำสั่งจะแสดง URL ที่ไม่มีโทเค็นและคำแนะนำในการแก้ไข แทนตัวยึดตำแหน่งโทเค็นที่ไม่ถูกต้อง
- หากการส่งผ่านคลิปบอร์ด/เบราว์เซอร์ล้มเหลวสำหรับ URL ที่ยืนยันตัวตนด้วยโทเค็น คำสั่งจะบันทึกคำแนะนำที่ปลอดภัยสำหรับการยืนยันตัวตนด้วยตนเอง โดยระบุชื่อ `OPENCLAW_GATEWAY_TOKEN`, `gateway.auth.token` และคีย์ส่วนย่อย URL `token` โดยไม่แสดงค่าของโทเค็น
