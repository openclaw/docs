# `openclaw dashboard`

เปิด UI ควบคุมโดยใช้ข้อมูลการยืนยันตัวตนปัจจุบันของคุณ

```bash
openclaw dashboard
openclaw dashboard --no-open
openclaw dashboard --yes
```

- `--no-open`: แสดง URL แต่ไม่เปิดเบราว์เซอร์
- `--yes`: เริ่ม/ติดตั้ง Gateway โดยไม่ถามยืนยันเมื่อจำเป็น

หมายเหตุ:

- แก้ไขการอ้างอิงความลับ SecretRef ของ `gateway.auth.token` ที่กำหนดค่าไว้เมื่อทำได้
- ทำงานตาม `gateway.tls.enabled`: Gateway ที่เปิดใช้ TLS จะแสดง/เปิด URL ของ UI ควบคุมที่ใช้ `https://` และเชื่อมต่อผ่าน `wss://`
- สำหรับการผูกกับ `lan` หรือ `custom` แบบไวลด์การ์ด การเปิดบนโฮสต์เดียวกันจะใช้ลูปแบ็กเสมอ เนื่องจากไวลด์การ์ดไม่ใช่ปลายทางสำหรับเบราว์เซอร์ การผูก `tailnet` และ `custom` แบบข้อความธรรมดาจะใช้ `127.0.0.1` เช่นกัน เพื่อให้เบราว์เซอร์มีบริบทที่ปลอดภัย ส่วนโฮสต์ที่ระบุเฉพาะเจาะจงและเปิดใช้ TLS จะใช้ที่อยู่ที่กำหนดค่าไว้ เพื่อให้ชื่อในใบรับรองตรงกัน
- ก่อนส่งมอบ URL ลูปแบ็กที่ผ่านการยืนยันตัวตนสำหรับการผูกกับอินเทอร์เฟซเฉพาะ คำสั่งจะตรวจสอบอินเทอร์เฟซที่กำหนดค่าไว้ และยืนยันว่าทั้งอินเทอร์เฟซนั้นและ `127.0.0.1` เป็นของกระบวนการ Gateway เดียวกัน หากไม่สามารถระบุความเป็นเจ้าของตัวรับฟังได้อย่างชัดเจน ระบบจะปฏิเสธโดยค่าเริ่มต้นพร้อมคำแนะนำเกี่ยวกับสถานะ
- สำหรับโทเค็นที่จัดการด้วย SecretRef (ไม่ว่าจะแก้ไขค่าได้หรือไม่ได้) URL ที่แสดง/คัดลอก/เปิดจะไม่รวมโทเค็น เพื่อไม่ให้ความลับภายนอกรั่วไหลไปยังเอาต์พุตเทอร์มินัล ประวัติคลิปบอร์ด หรืออาร์กิวเมนต์สำหรับเปิดเบราว์เซอร์
- หาก `gateway.auth.token` จัดการด้วย SecretRef แต่ยังแก้ไขค่าไม่ได้ คำสั่งจะแสดง URL ที่ไม่มีโทเค็นและคำแนะนำในการแก้ไข แทนตัวยึดตำแหน่งโทเค็นที่ใช้ไม่ได้
- หากการส่งมอบผ่านคลิปบอร์ด/เบราว์เซอร์ล้มเหลวสำหรับ URL ที่ยืนยันตัวตนด้วยโทเค็น คำสั่งจะบันทึกคำแนะนำที่ปลอดภัยสำหรับการยืนยันตัวตนด้วยตนเอง โดยระบุ `OPENCLAW_GATEWAY_TOKEN`, `gateway.auth.token` และคีย์ส่วนย่อย URL `token` โดยไม่แสดงค่าของโทเค็น
