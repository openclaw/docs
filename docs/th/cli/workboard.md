`openclaw workboard` คืออินเทอร์เฟซเทอร์มินัลสำหรับ [Workboard plugin](/th/plugins/workboard) ที่มาพร้อมระบบ โดยช่วยให้ผู้ปฏิบัติงานแสดงรายการการ์ด สร้างการ์ด ตรวจสอบการ์ดหนึ่งใบ และสั่งให้ Gateway ที่กำลังทำงานส่งงานที่พร้อมไปยังการทำงานของผู้ปฏิบัติงานซึ่งเป็นเอเจนต์ย่อย

เปิดใช้งาน Plugin ก่อนใช้คำสั่ง:

```bash
openclaw plugins enable workboard
openclaw gateway restart
```

## การใช้งาน

```bash
openclaw workboard list [--board <id>] [--status <status>] [--include-archived] [--json]
openclaw workboard create <title...> [--notes <text>] [--status <status>] [--priority <priority>] [--agent <id>] [--board <id>] [--labels <items>] [--json]
openclaw workboard show <id> [--json]
openclaw workboard move <id> --status <status> [--json]
openclaw workboard dispatch [--board <id>] [--max-starts <count>] [--admin] [--url <url>] [--token <token>] [--timeout <ms>] [--json]
```

คำสั่งนี้อ่านและเขียนฐานข้อมูล SQLite ที่ Plugin เป็นเจ้าของชุดเดียวกับที่แดชบอร์ดและเครื่องมือเอเจนต์ Workboard ใช้ รหัสการ์ดเป็น UUID ส่วนคำสั่งที่รับรหัสการ์ดจะรับคำนำหน้ารหัสที่ระบุได้อย่างไม่กำกวมด้วย (เอาต์พุตข้อความแบบกระชับแสดงอักขระ 8 ตัวแรก)

ค่าที่ใช้ได้สำหรับ `status`: `triage`, `backlog`, `todo`, `scheduled`, `ready`, `running`, `review`, `blocked`, `done` ค่าที่ใช้ได้สำหรับ `priority`: `low`, `normal`, `high`, `urgent`

## `list`

```bash
openclaw workboard list
openclaw workboard list --board default --status ready
openclaw workboard list --json
```

เอาต์พุตข้อความมีรูปแบบกระชับ:

```text
7f4a2c10  ready     high    default agent-a  แก้ไข Heartbeat ของผู้ปฏิบัติงานที่ล้าสมัย
```

คอลัมน์ประกอบด้วยคำนำหน้ารหัส สถานะ ลำดับความสำคัญ รหัสบอร์ด รหัสเอเจนต์ซึ่งอาจไม่มี และชื่อเรื่อง

| แฟล็ก                 | วัตถุประสงค์                                       |
| -------------------- | --------------------------------------------- |
| `--board <id>`       | จำกัดผลลัพธ์ไว้ที่เนมสเปซของบอร์ดหนึ่งรายการ          |
| `--status <status>`  | จำกัดผลลัพธ์ไว้ที่สถานะ Workboard หนึ่งสถานะ         |
| `--include-archived` | รวมการ์ดที่เก็บถาวรไว้ในเอาต์พุตข้อความแบบกระชับ |
| `--json`             | พิมพ์รายการการ์ดทั้งหมดเป็น JSON สำหรับเครื่อง      |

โดยค่าเริ่มต้น เอาต์พุตข้อความแบบกระชับจะซ่อนการ์ดที่เก็บถาวรเพื่อให้ CLI สอดคล้องกับ `/workboard list` ส่ง `--include-archived` เพื่อแสดงการ์ดเหล่านั้น เอาต์พุต JSON จะเก็บรายการการ์ดทั้งหมดไว้เสมอ รวมถึงการ์ดที่เก็บถาวร เพื่อรองรับระบบอัตโนมัติที่มีอยู่

## `create`

```bash
openclaw workboard create "แก้ไข Heartbeat ของผู้ปฏิบัติงานที่ล้าสมัย" --priority high --labels bug,workboard
openclaw workboard create "เขียนเอกสาร Workboard" --status ready --agent docs-agent --board docs --notes "ครอบคลุม CLI, คำสั่งแบบสแลช, การส่งงาน และสถานะ SQLite"
```

| แฟล็ก                    | วัตถุประสงค์                                 |
| ----------------------- | --------------------------------------- |
| `--notes <text>`        | บันทึกเริ่มต้นของการ์ด                      |
| `--status <status>`     | สถานะเริ่มต้น ค่าเริ่มต้นคือ `todo`          |
| `--priority <priority>` | ลำดับความสำคัญ ค่าเริ่มต้นคือ `normal`              |
| `--agent <id>`          | กำหนดการ์ดให้เอเจนต์หรือรหัสเจ้าของ |
| `--board <id>`          | จัดเก็บการ์ดในเนมสเปซของบอร์ด     |
| `--labels <items>`      | ป้ายกำกับที่คั่นด้วยจุลภาค                  |
| `--json`                | พิมพ์การ์ดที่สร้างเป็น JSON สำหรับเครื่อง  |

`create` เขียนไปยังสถานะ SQLite ของ Workboard โดยตรง การ์ดจะปรากฏในแท็บ Workboard ของ Control UI และในเครื่องมือ Workboard ทันที

## `show`

```bash
openclaw workboard show 7f4a2c10
openclaw workboard show 7f4a2c10 --json
```

เอาต์พุตข้อความจะพิมพ์บรรทัดการ์ดแบบกระชับและบันทึก เอาต์พุต JSON จะส่งคืนระเบียนการ์ดทั้งหมด รวมถึงข้อมูลเมตาการดำเนินการ จำนวนครั้งที่พยายาม ความคิดเห็น ลิงก์ หลักฐาน อาร์ติแฟกต์ บันทึกของผู้ปฏิบัติงาน สถานะโปรโตคอล ข้อมูลวินิจฉัย และข้อมูลเมตาของระบบอัตโนมัติ

## `move`

```bash
openclaw workboard move 7f4a2c10 --status review
openclaw workboard move 7f4a2c10 --status done --json
```

`move` เปลี่ยนสถานะของการ์ดโดยใช้เส้นทางผู้ปฏิบัติงานแบบกำหนดเองเดียวกับการลากการ์ดในแดชบอร์ด โดยรับรหัสการ์ดแบบเต็มหรือคำนำหน้าที่ระบุได้อย่างไม่กำกวม การระงับเนื่องจากการขึ้นต่อกันและกำหนดเวลาที่ยังทำงานอยู่ยังคงมีผล ผู้ปฏิบัติงานสามารถย้ายการ์ดที่มีการอ้างสิทธิ์ได้โดยไม่ต้องใช้โทเค็นการอ้างสิทธิ์ของเอเจนต์ โทเค็นการอ้างสิทธิ์ยังคงจำกัดขอบเขตไว้สำหรับการแก้ไขผ่านเครื่องมือเอเจนต์และถูกปกปิดจากเอาต์พุต JSON

## `dispatch`

```bash
openclaw workboard dispatch
openclaw workboard dispatch --json
openclaw workboard dispatch --max-starts 10
openclaw workboard dispatch --admin
openclaw workboard dispatch --url http://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

ขั้นแรก `dispatch` จะเรียกเมธอด RPC `workboard.cards.dispatch` ของ Gateway ที่กำลังทำงาน ซึ่งใช้รันไทม์เอเจนต์ย่อยเดียวกับการดำเนินการส่งงานในแดชบอร์ด ดังนั้นการ์ดที่พร้อมจะกลายเป็นการทำงานของผู้ปฏิบัติงานที่ติดตามเป็นงานพร้อมคีย์เซสชันที่เชื่อมโยงกัน `--max-starts` ใช้เมธอดเสริม `workboard.cards.dispatchWithOptions` เพื่อให้ Gateway รุ่นเก่าปฏิเสธตัวเลือกก่อนเริ่มผู้ปฏิบัติงานใด ๆ หลังอัปเกรด ให้รีสตาร์ต Gateway ก่อนใช้แฟล็กนี้ การ์ดที่กำหนดเอเจนต์ไว้จะใช้คีย์เซสชันเอเจนต์ย่อยที่จำกัดขอบเขตตามเอเจนต์ ส่วนการ์ดที่ไม่ได้กำหนดเอเจนต์จะใช้คีย์เอเจนต์ย่อยที่ไม่จำกัดขอบเขตต่อไป เพื่อคงเอเจนต์เริ่มต้นที่กำหนดค่าไว้ของ Gateway

ลูปการส่งงาน:

1. เลื่อนงานลูกที่การขึ้นต่อกันพร้อมแล้วไปเป็น `ready`
2. บล็อกการอ้างสิทธิ์ที่หมดอายุหรือการทำงานของผู้ปฏิบัติงานที่หมดเวลา
3. บันทึกข้อมูลเมตาการส่งงานในการ์ดที่พร้อม
4. เลือกการ์ดที่พร้อมและยังไม่มีการอ้างสิทธิ์เป็นชุดขนาดเล็ก
5. อ้างสิทธิ์การ์ดแต่ละใบที่เลือกให้แก่ตัวส่งงานหรือเอเจนต์ที่กำหนด
6. เริ่มการทำงานของผู้ปฏิบัติงานซึ่งเป็นเอเจนต์ย่อยด้วยบริบทการ์ดที่จำกัดและโทเค็นการอ้างสิทธิ์ของการ์ด
7. จัดเก็บรหัสการทำงานของผู้ปฏิบัติงาน คีย์เซสชัน การเชื่อมโยงงานเมื่อบัญชีแยกประเภทงานของ Gateway รายงานข้อมูลดังกล่าว สถานะการดำเนินการ และบันทึกของผู้ปฏิบัติงานไว้ในการ์ด

การเลือกดำเนินการอย่างระมัดระวัง โดยค่าเริ่มต้น การส่งงานหนึ่งครั้งจะเริ่มผู้ปฏิบัติงานไม่เกินสามราย ข้ามการ์ดที่เก็บถาวรหรือมีการอ้างสิทธิ์แล้ว และเริ่มการ์ดเพียงหนึ่งใบต่อเจ้าของหรือเอเจนต์ในการประมวลผลหนึ่งรอบ การ์ดที่มีงานซึ่งกำลังทำงานหรืออยู่ระหว่างตรวจทานและเจ้าของกำลังดำเนินการอยู่แล้วจะถูกเก็บไว้สำหรับการส่งงานครั้งถัดไป ส่ง `--max-starts <count>` พร้อมจำนวนเต็มบวกเพื่อเปลี่ยนขีดจำกัดต่อรอบ กฎหนึ่งการ์ดต่อเจ้าของยังคงมีผล ดังนั้นจำนวนการเริ่มทำงานจริงอาจต่ำกว่า

หากการเริ่มผู้ปฏิบัติงานล้มเหลวหลังจากมีการอ้างสิทธิ์การ์ดแล้ว Workboard จะบล็อกการ์ดนั้น ล้างการอ้างสิทธิ์ และบันทึกความล้มเหลวไว้ในข้อมูลเมตาการดำเนินการของการ์ดและบันทึกผู้ปฏิบัติงาน เพื่อให้การเริ่มที่ล้มเหลวยังคงมองเห็นได้แทนที่จะส่งการ์ดกลับเข้าคิวโดยไม่แจ้ง

หากไม่ได้ระบุเป้าหมาย Gateway อย่างชัดเจน และ Gateway ภายในเครื่องไม่พร้อมใช้งานหรือยังไม่เปิดเผยเมธอดการส่งงานของ Workboard CLI จะใช้การส่งงานแบบข้อมูลเท่านั้นกับสถานะ Workboard ภายในเครื่องแทน การส่งงานแบบข้อมูลเท่านั้นยังคงสามารถเลื่อนการขึ้นต่อกัน ล้างการอ้างสิทธิ์ที่ล้าสมัย และบล็อกการทำงานที่หมดเวลาได้ แต่จะไม่เริ่มผู้ปฏิบัติงาน ความล้มเหลวด้านการยืนยันตัวตน สิทธิ์ และการตรวจสอบความถูกต้อง รวมถึงความล้มเหลวของเป้าหมาย `--url` หรือ `--token` ที่ระบุอย่างชัดเจน จะถูกรายงานโดยตรงแทนที่จะเรียกใช้การทำงานสำรอง

เอาต์พุตข้อความรายงานการเริ่มผู้ปฏิบัติงาน:

```text
การส่งงานเสร็จสมบูรณ์: เริ่มแล้ว=2 ล้มเหลว=0
```

เอาต์พุตการทำงานสำรองระบุไว้อย่างชัดเจน:

```text
Gateway ไม่พร้อมใช้งาน; ส่งงานแบบข้อมูลเท่านั้น: เลื่อนแล้ว=1 บล็อกแล้ว=0
```

เอาต์พุต JSON มีผลลัพธ์การส่งงาน การส่งงานที่รองรับโดย Gateway อาจมี `started` และ `startFailures` ส่วนการทำงานสำรองแบบข้อมูลเท่านั้นมี `gatewayUnavailable: true` โทเค็นการอ้างสิทธิ์ถูกปกปิดจากเอาต์พุต JSON ของการ์ด

ในแดชบอร์ด ผลลัพธ์การส่งงานเดียวกันจะแสดงเป็นสรุปสั้น ๆ เพื่อให้ผู้ปฏิบัติงานเห็นจำนวนการ์ดที่เริ่ม เลื่อน บล็อก เรียกคืน หรือทำงานล้มเหลวได้โดยไม่ต้องเปิดรายละเอียดการ์ด

## ความสอดคล้องกับคำสั่งแบบสแลช

ช่องทางที่รองรับคำสั่งสามารถใช้คำสั่งแบบสแลชที่ตรงกัน:

```text
/workboard list
/workboard show 7f4a2c10
/workboard create แก้ไข Heartbeat ของผู้ปฏิบัติงานที่ล้าสมัย
/workboard move 7f4a2c10 --status review
/workboard dispatch
```

การส่งงานด้วยคำสั่งแบบสแลชยังใช้รันไทม์เอเจนต์ย่อยของ Gateway ดังนั้นจึงมีพฤติกรรมการอ้างสิทธิ์ การเริ่มผู้ปฏิบัติงาน และความล้มเหลวเช่นเดียวกับเส้นทาง Gateway ของแดชบอร์ดและ CLI

`/workboard list` และ `/workboard show` เป็นคำสั่งอ่านสำหรับผู้ส่งคำสั่งที่ได้รับอนุญาต `/workboard create`, `/workboard move` และ `/workboard dispatch` แก้ไขสถานะบอร์ดและกำหนดให้มีสถานะเจ้าของบนพื้นผิวแชต หรือเป็นไคลเอนต์ Gateway ที่มี `operator.write` หรือ `operator.admin`

## สิทธิ์

โดยปกติ เส้นทางการส่งงานของ CLI จะขอขอบเขต Gateway `operator.write` และ `operator.read` การ์ดที่ผูกกับพื้นที่ทำงานจะทำงานโดยตรงในพื้นที่ทำงานของเอเจนต์ที่กำหนดค่าไว้อย่างตรงกัน คำขอเวิร์กทรีจะถูกจำกัดไว้ที่ไดเรกทอรีนั้น แทนที่จะปล่อยให้โฮสต์นำโค้ดที่ควบคุมโดยที่เก็บมาใช้งาน ผู้ปฏิบัติงานที่เลือกต้องมีสิทธิ์เข้าถึงแซนด์บ็อกซ์ Docker แบบเขียนได้และไม่ใช้ร่วมกันสำหรับพื้นที่ทำงานนั้นโดยเฉพาะ มีแฮชคอนเทนเนอร์ที่ใช้งานอยู่ซึ่งตรงกับเมานต์และนโยบายที่ร้องขอ และไม่มีความสามารถในการหลบหนีไปยังโฮสต์ ส่ง `--admin` เพื่อขอ `operator.admin` อย่างชัดเจน อนุญาตให้ใช้เช็กเอาต์อื่นของโฮสต์ และใช้การตั้งค่าเวิร์กทรีที่มีการจัดการตามปกติ การเชื่อมต่อจะล้มเหลวหากขอบเขตนั้นไม่ได้รับอนุมัติสำหรับไคลเอนต์ โทเค็น Gateway แบบอ่านอย่างเดียวสามารถตรวจสอบข้อมูล Workboard ผ่านเมธอดอ่านได้ แต่ไม่สามารถสร้างการ์ดหรือส่งผู้ปฏิบัติงานได้ มิฉะนั้น ขีดจำกัดพื้นที่ทำงานจะไม่เปลี่ยนแปลงการย้ายการ์ดด้วยตนเองสำหรับผู้เรียกที่มีสิทธิ์แก้ไข Workboard

คำสั่ง `list`, `create`, `show` และ `move` ภายในเครื่องจะทำงานกับไดเรกทอรีสถานะ OpenClaw ภายในเครื่องที่โปรไฟล์ปัจจุบันใช้ ใช้ `--dev` หรือ `--profile <name>` กับคำสั่งระดับบนสุด `openclaw` เมื่อต้องการรูทสถานะอื่น

## การแก้ไขปัญหา

### ไม่มีการ์ดปรากฏ

ยืนยันว่าเปิดใช้งาน Plugin สำหรับโปรไฟล์และรูทสถานะเดียวกัน:

```bash
openclaw plugins inspect workboard --runtime --json
```

หากแดชบอร์ดแสดงการ์ดแต่ CLI ไม่แสดง ให้ตรวจสอบว่าทั้งสองคำสั่งใช้การตั้งค่า `--dev` หรือ `--profile` เดียวกัน

### การส่งงานระบุว่าเป็นแบบข้อมูลเท่านั้น

เริ่มหรือรีสตาร์ต Gateway:

```bash
openclaw gateway restart
openclaw gateway status --deep
```

จากนั้นลอง `openclaw workboard dispatch` อีกครั้ง การทำงานสำรองแบบข้อมูลเท่านั้นมีประโยชน์สำหรับการล้างสถานะภายในเครื่อง แต่การทำงานของผู้ปฏิบัติงานต้องใช้ Gateway ที่กำลังทำงาน

### การส่งงานไม่เริ่มสิ่งใด

ตรวจสอบว่ามีการ์ด `ready` อย่างน้อยหนึ่งใบที่ไม่มีการอ้างสิทธิ์ที่ยังทำงานอยู่:

```bash
openclaw workboard list --status ready
```

การ์ดอาจถูกข้ามเมื่อเจ้าของเดียวกันมีงานที่กำลังทำงานหรืออยู่ระหว่างตรวจทานอยู่แล้ว ย้ายงานที่เสร็จสมบูรณ์ไปยัง `done` ปล่อยการอ้างสิทธิ์ที่ล้าสมัยผ่านเครื่องมือ Workboard หรือเรียกใช้การส่งงานอีกครั้งหลังจากผู้ปฏิบัติงานที่ทำงานอยู่เสร็จสิ้น
