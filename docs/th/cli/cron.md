# `openclaw cron`

จัดการงาน Cron สำหรับตัวกำหนดตารางเวลาของ Gateway

<Tip>
เรียกใช้ `openclaw cron --help` เพื่อดูพื้นผิวคำสั่งทั้งหมด ดู [งาน Cron](/th/automation/cron-jobs) สำหรับคู่มือเชิงแนวคิด
</Tip>

## เซสชัน

`--session` รับค่า `main`, `isolated`, `current` หรือ `session:<id>`

<AccordionGroup>
  <Accordion title="คีย์เซสชัน">
    - `main` ผูกกับเซสชันหลักของเอเจนต์
    - `isolated` สร้างทรานสคริปต์และรหัสเซสชันใหม่สำหรับการรันแต่ละครั้ง
    - `current` ผูกกับเซสชันที่ใช้งานอยู่ ณ เวลาสร้าง
    - `session:<id>` ปักหมุดกับคีย์เซสชันแบบถาวรที่ระบุอย่างชัดเจน

  </Accordion>
  <Accordion title="ความหมายของเซสชันแบบแยก">
    การรันแบบแยกจะรีเซ็ตบริบทการสนทนาแวดล้อม การกำหนดเส้นทางช่องทางและกลุ่ม นโยบายส่ง/เข้าคิว การยกระดับ แหล่งที่มา และการผูก ACP runtime จะถูกรีเซ็ตสำหรับการรันใหม่ การตั้งค่าที่ปลอดภัยและโมเดลที่ผู้ใช้เลือกไว้อย่างชัดเจนหรือการแทนที่การยืนยันตัวตนสามารถส่งต่อข้ามการรันได้
  </Accordion>
</AccordionGroup>

## การนำส่ง

`openclaw cron list` และ `openclaw cron show <job-id>` แสดงตัวอย่างเส้นทางการนำส่งที่ถูกแก้ไขแล้ว สำหรับ `channel: "last"` ตัวอย่างจะแสดงว่าเส้นทางถูกแก้ไขจากเซสชันหลักหรือเซสชันปัจจุบัน หรือจะล้มเหลวแบบปิด

เป้าหมายที่มีคำนำหน้าผู้ให้บริการช่วยแยกแยะช่องทางประกาศที่ยังไม่ถูกแก้ไขได้ ตัวอย่างเช่น `to: "telegram:123"` เลือก Telegram เมื่อ `delivery.channel` ถูกละไว้หรือเป็น `last` เฉพาะคำนำหน้าที่ Plugin ที่โหลดอยู่ประกาศไว้เท่านั้นที่เป็นตัวเลือกผู้ให้บริการ หาก `delivery.channel` ถูกระบุชัดเจน คำนำหน้าต้องตรงกับช่องทางนั้น; `channel: "whatsapp"` พร้อม `to: "telegram:123"` จะถูกปฏิเสธ คำนำหน้าบริการ เช่น `imessage:` และ `sms:` ยังคงเป็นไวยากรณ์เป้าหมายที่ช่องทางเป็นเจ้าของ

<Note>
งาน `cron add` แบบแยกมีค่าเริ่มต้นเป็นการนำส่งแบบ `--announce` ใช้ `--no-deliver` เพื่อเก็บผลลัพธ์ไว้ภายใน `--deliver` ยังคงอยู่เป็นนามแฝงที่เลิกใช้แล้วสำหรับ `--announce`
</Note>

### ความเป็นเจ้าของการนำส่ง

การนำส่งแชท Cron แบบแยกแชร์ร่วมกันระหว่างเอเจนต์และตัวรัน:

- เอเจนต์สามารถส่งโดยตรงด้วยเครื่องมือ `message` เมื่อมีเส้นทางแชทพร้อมใช้งาน
- `announce` นำส่งคำตอบสุดท้ายแบบสำรองเฉพาะเมื่อเอเจนต์ไม่ได้ส่งโดยตรงไปยังเป้าหมายที่แก้ไขแล้ว
- `webhook` โพสต์เพย์โหลดที่เสร็จสมบูรณ์ไปยัง URL
- `none` ปิดใช้งานการนำส่งสำรองของตัวรัน

`--announce` คือการนำส่งสำรองของตัวรันสำหรับคำตอบสุดท้าย `--no-deliver` ปิดใช้งานการสำรองนั้น แต่ไม่ได้ลบเครื่องมือ `message` ของเอเจนต์เมื่อมีเส้นทางแชทพร้อมใช้งาน

รายการเตือนความจำที่สร้างจากแชทที่ใช้งานอยู่จะเก็บเป้าหมายการนำส่งแชทสดไว้สำหรับการนำส่งประกาศสำรอง คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็ก อย่าใช้เป็นแหล่งความจริงสำหรับรหัสผู้ให้บริการที่แยกแยะตัวพิมพ์เล็กใหญ่ เช่น รหัสห้อง Matrix

### การนำส่งเมื่อเกิดความล้มเหลว

การแจ้งเตือนความล้มเหลวจะถูกแก้ไขตามลำดับนี้:

1. `delivery.failureDestination` ในงาน
2. `cron.failureDestination` แบบส่วนกลาง
3. เป้าหมายประกาศหลักของงาน (เมื่อไม่ได้ตั้งค่าปลายทางความล้มเหลวอย่างชัดเจน)

<Note>
งานเซสชันหลักสามารถใช้ `delivery.failureDestination` ได้เฉพาะเมื่อโหมดการนำส่งหลักเป็น `webhook` งานแบบแยกรับค่านี้ได้ในทุกโหมด
</Note>

หมายเหตุ: การรัน Cron แบบแยกถือว่าความล้มเหลวของเอเจนต์ระดับการรันเป็นข้อผิดพลาดของงาน แม้เมื่อไม่มีการสร้างเพย์โหลดคำตอบ ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการยังคงเพิ่มตัวนับข้อผิดพลาดและทริกเกอร์การแจ้งเตือนความล้มเหลว

## การจัดตารางเวลา

### งานแบบครั้งเดียว

`--at <datetime>` จัดตารางการรันแบบครั้งเดียว วันที่เวลาแบบไม่มีออฟเซ็ตจะถือว่าเป็น UTC เว้นแต่คุณจะส่ง `--tz <iana>` ด้วย ซึ่งจะแปลความหมายเวลาตามนาฬิกาในเขตเวลาที่กำหนด

<Note>
งานแบบครั้งเดียวจะลบหลังจากสำเร็จโดยค่าเริ่มต้น ใช้ `--keep-after-run` เพื่อเก็บไว้
</Note>

### งานที่เกิดซ้ำ

งานที่เกิดซ้ำใช้การหน่วงลองซ้ำแบบเอ็กซ์โพเนนเชียลหลังจากเกิดข้อผิดพลาดต่อเนื่อง: 30s, 1m, 5m, 15m, 60m ตารางเวลาจะกลับสู่ปกติหลังจากการรันที่สำเร็จครั้งถัดไป

การรันที่ถูกข้ามจะถูกติดตามแยกจากข้อผิดพลาดในการดำเนินการ การรันเหล่านี้ไม่มีผลต่อการหน่วงลองซ้ำ แต่ `openclaw cron edit <job-id> --failure-alert-include-skipped` สามารถเลือกให้การแจ้งเตือนความล้มเหลวรวมการแจ้งเตือนการรันที่ถูกข้ามซ้ำได้

สำหรับงานแบบแยกที่กำหนดเป้าหมายไปยังผู้ให้บริการโมเดลที่ตั้งค่าไว้ภายในเครื่อง Cron จะรันการตรวจสอบล่วงหน้าผู้ให้บริการแบบเบาก่อนเริ่มเทิร์นเอเจนต์ ผู้ให้บริการ `api: "ollama"` แบบ Loopback, เครือข่ายส่วนตัว และ `.local` จะถูกตรวจที่ `/api/tags`; ผู้ให้บริการภายในเครื่องที่เข้ากันได้กับ OpenAI เช่น vLLM, SGLang และ LM Studio จะถูกตรวจที่ `/models` หากเข้าถึงปลายทางไม่ได้ การรันจะถูกบันทึกเป็น `skipped` และลองใหม่ในตารางเวลาภายหลัง; ปลายทางที่ตายและตรงกันจะถูกแคชไว้ 5 นาทีเพื่อหลีกเลี่ยงไม่ให้งานจำนวนมากกระหน่ำเซิร์ฟเวอร์ภายในเครื่องเดียวกัน

หมายเหตุ: นิยามงาน Cron อยู่ใน `jobs.json` ขณะที่สถานะรันไทม์ที่รอดำเนินการอยู่ใน `jobs-state.json` หาก `jobs.json` ถูกแก้ไขจากภายนอก Gateway จะโหลดตารางเวลาที่เปลี่ยนแปลงใหม่และล้างสล็อตที่รอดำเนินการซึ่งค้างอยู่; การเขียนใหม่ที่เปลี่ยนเฉพาะการจัดรูปแบบจะไม่ล้างสล็อตที่รอดำเนินการ

### การรันด้วยตนเอง

`openclaw cron run` คืนค่าทันทีที่การรันด้วยตนเองถูกเข้าคิว คำตอบที่สำเร็จจะมี `{ ok: true, enqueued: true, runId }` ใช้ `openclaw cron runs --id <job-id>` เพื่อติดตามผลลัพธ์สุดท้าย

<Note>
`openclaw cron run <job-id>` บังคับรันโดยค่าเริ่มต้น ใช้ `--due` เพื่อคงพฤติกรรมเดิมแบบ "รันเฉพาะเมื่อถึงกำหนด"
</Note>

## โมเดล

`cron add|edit --model <ref>` เลือกโมเดลที่อนุญาตสำหรับงาน

<Warning>
หากโมเดลไม่ได้รับอนุญาตหรือไม่สามารถแก้ไขได้ Cron จะทำให้การรันล้มเหลวพร้อมข้อผิดพลาดการตรวจสอบที่ชัดเจน แทนที่จะย้อนกลับไปใช้เอเจนต์ของงานหรือการเลือกโมเดลเริ่มต้น
</Warning>

Cron `--model` เป็น **ค่าหลักของงาน** ไม่ใช่การแทนที่ `/model` ของเซสชันแชท นั่นหมายความว่า:

- การสำรองโมเดลที่ตั้งค่าไว้ยังคงมีผลเมื่อโมเดลงานที่เลือกไว้ล้มเหลว
- `fallbacks` ในเพย์โหลดรายงานจะแทนที่รายการสำรองที่ตั้งค่าไว้เมื่อมีอยู่
- รายการสำรองรายงานที่ว่างเปล่า (`fallbacks: []` ในเพย์โหลด/API ของงาน) ทำให้การรัน Cron เข้มงวด
- เมื่องานมี `--model` แต่ไม่ได้ตั้งค่ารายการสำรอง OpenClaw จะส่งการแทนที่รายการสำรองว่างอย่างชัดเจนเพื่อไม่ให้ค่าหลักของเอเจนต์ถูกผนวกเป็นเป้าหมายลองซ้ำแบบซ่อน

### ลำดับความสำคัญของโมเดล Cron แบบแยก

Cron แบบแยกแก้ไขโมเดลที่ใช้งานอยู่ตามลำดับนี้:

1. การแทนที่ Gmail-hook
2. `--model` รายงาน
3. การแทนที่โมเดลเซสชัน Cron ที่จัดเก็บไว้ (เมื่อผู้ใช้เลือกไว้)
4. การเลือกโมเดลของเอเจนต์หรือค่าเริ่มต้น

### โหมดเร็ว

โหมดเร็วของ Cron แบบแยกจะตามการเลือกโมเดลสดที่แก้ไขแล้ว การตั้งค่าโมเดล `params.fastMode` มีผลโดยค่าเริ่มต้น แต่การแทนที่ `fastMode` ของเซสชันที่จัดเก็บไว้ยังคงชนะค่าคอนฟิก

### การลองซ้ำเมื่อสลับโมเดลสด

หากการรันแบบแยกโยน `LiveSessionModelSwitchError` Cron จะคงผู้ให้บริการและโมเดลที่ถูกสลับไว้ (และการแทนที่โปรไฟล์การยืนยันตัวตนที่ถูกสลับเมื่อมีอยู่) สำหรับการรันที่ใช้งานอยู่ก่อนลองซ้ำ ลูปลองซ้ำชั้นนอกถูกจำกัดไว้ที่การลองซ้ำการสลับสองครั้งหลังจากความพยายามแรก จากนั้นจะยกเลิกแทนที่จะวนตลอดไป

## ผลลัพธ์การรันและการปฏิเสธ

### การระงับการตอบรับที่ค้างเก่า

เทิร์น Cron แบบแยกจะระงับคำตอบที่เป็นเพียงการตอบรับซึ่งค้างเก่า หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะชั่วคราวและไม่มีการรันเอเจนต์ย่อยลูกหลานที่รับผิดชอบคำตอบสุดท้าย Cron จะพร้อมต์ซ้ำหนึ่งครั้งเพื่อขอผลลัพธ์จริงก่อนนำส่ง

### การระงับโทเคนเงียบ

หากการรัน Cron แบบแยกคืนเฉพาะโทเคนเงียบ (`NO_REPLY` หรือ `no_reply`) Cron จะระงับทั้งการนำส่งออกโดยตรงและเส้นทางสรุปที่เข้าคิวแบบสำรอง ดังนั้นจะไม่มีสิ่งใดถูกโพสต์กลับไปยังแชท

### การปฏิเสธแบบมีโครงสร้าง

การรัน Cron แบบแยกจะเลือกใช้เมทาดาทาการปฏิเสธการดำเนินการแบบมีโครงสร้างจากการรันที่ฝังอยู่ก่อน จากนั้นจึงย้อนกลับไปใช้ตัวทำเครื่องหมายการปฏิเสธที่รู้จักในผลลัพธ์สุดท้าย เช่น `SYSTEM_RUN_DENIED`, `INVALID_REQUEST` และวลีปฏิเสธการผูกการอนุมัติ

`cron list` และประวัติการรันแสดงเหตุผลการปฏิเสธ แทนที่จะรายงานคำสั่งที่ถูกบล็อกว่าเป็น `ok`

## การเก็บรักษา

การเก็บรักษาและการตัดทอนถูกควบคุมในคอนฟิก:

- `cron.sessionRetention` (ค่าเริ่มต้น `24h`) ตัดทอนเซสชันการรันแบบแยกที่เสร็จสมบูรณ์แล้ว
- `cron.runLog.maxBytes` และ `cron.runLog.keepLines` ตัดทอน `~/.openclaw/cron/runs/<jobId>.jsonl`

## การย้ายงานเก่า

<Note>
หากคุณมีงาน Cron จากก่อนรูปแบบการนำส่งและที่เก็บปัจจุบัน ให้รัน `openclaw doctor --fix` Doctor ทำให้ฟิลด์ Cron แบบเดิมเป็นมาตรฐาน (`jobId`, `schedule.cron`, ฟิลด์การนำส่งระดับบนรวมถึง `threadId` แบบเดิม, นามแฝงการนำส่ง `provider` ในเพย์โหลด) และย้ายงานสำรอง Webhook แบบ `notify: true` อย่างง่ายไปเป็นการนำส่ง Webhook อย่างชัดเจนเมื่อมีการตั้งค่า `cron.webhook`

Doctor ยังลบ sentinel ของ Cron `payload.model` ที่คงไว้ เช่น `"default"`, `"null"`, สตริงว่าง และ JSON `null` รันไทม์ Cron ยังคงถือว่าสตริง `payload.model` ที่ไม่ว่างใด ๆ เป็นการแทนที่โมเดลอย่างชัดเจนและตรวจสอบกับ `agents.defaults.models`; ให้ละคีย์โมเดลเมื่อควรให้งานใช้การเลือกโมเดลของเอเจนต์/ค่าเริ่มต้น
</Note>

## การแก้ไขทั่วไป

อัปเดตการตั้งค่าการนำส่งโดยไม่เปลี่ยนข้อความ:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

ปิดใช้งานการนำส่งสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --no-deliver
```

เปิดใช้งานบริบทบูตสแตรปแบบเบาสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --light-context
```

ประกาศไปยังช่องทางเฉพาะ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

ประกาศไปยังหัวข้อฟอรัม Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

สร้างงานแบบแยกพร้อมบริบทบูตสแตรปแบบเบา:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` ใช้กับงานเทิร์นเอเจนต์แบบแยกเท่านั้น สำหรับการรัน Cron โหมดเบาจะทำให้บริบทบูตสแตรปว่าง แทนที่จะฉีดชุดบูตสแตรปเวิร์กสเปซทั้งหมด

## คำสั่งผู้ดูแลทั่วไป

การรันด้วยตนเองและการตรวจสอบ:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

`openclaw cron list` แสดงงานที่ตรงกันทั้งหมดโดยค่าเริ่มต้น ส่ง `--agent <id>` เพื่อแสดงเฉพาะงานที่รหัสเอเจนต์ปกติที่มีผลตรงกัน; งานที่ไม่มีรหัสเอเจนต์ที่จัดเก็บไว้จะนับเป็นเอเจนต์เริ่มต้นที่ตั้งค่าไว้

รายการ `cron runs` มีการวินิจฉัยการนำส่งพร้อมเป้าหมาย Cron ที่ตั้งใจไว้ เป้าหมายที่แก้ไขแล้ว การส่งด้วยเครื่องมือข้อความ การใช้สำรอง และสถานะนำส่งแล้ว

การกำหนดเป้าหมายเอเจนต์และเซสชันใหม่:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` เตือนเมื่อ `--agent` ถูกละไว้ในงานเทิร์นเอเจนต์และย้อนกลับไปใช้เอเจนต์เริ่มต้น (`main`) ส่ง `--agent <id>` ตอนสร้างเพื่อปักหมุดเอเจนต์เฉพาะ

การปรับแต่งการนำส่ง:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
