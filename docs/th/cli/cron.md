# `openclaw cron`

จัดการงาน Cron สำหรับตัวจัดกำหนดการของ Gateway

<Tip>
เรียกใช้ `openclaw cron --help` เพื่อดูคำสั่งทั้งหมด โปรดดู [งาน Cron](/th/automation/cron-jobs) สำหรับคู่มือเชิงแนวคิด
</Tip>

<Note>
การแก้ไข Cron ทั้งหมด (`add`/`create`, `update`/`edit`, `remove`, `run`) ต้องใช้ `operator.admin` การเรียกใช้เพย์โหลดคำสั่งจะดำเนินการโดยตรงในโปรเซส Gateway ไม่ใช่ในรูปแบบการเรียกเครื่องมือ `tools.exec` ของเอเจนต์ โดย `tools.exec.*` และการอนุมัติการดำเนินการยังคงควบคุมเครื่องมือดำเนินการที่โมเดลมองเห็น
</Note>

## สร้างงานอย่างรวดเร็ว

`openclaw cron create` เป็นนามแฝงของ `openclaw cron add` สำหรับงานใหม่ ให้ระบุกำหนดการก่อนและพรอมต์เป็นลำดับถัดไป:

```bash
openclaw cron create "0 7 * * *" \
  "สรุปข้อมูลอัปเดตในช่วงข้ามคืน" \
  --name "สรุปยามเช้า" \
  --agent ops
```

ใช้ `--webhook <url>` เมื่อต้องการให้งานส่งเพย์โหลดที่เสร็จแล้วด้วย POST แทนการส่งไปยังเป้าหมายแชต:

```bash
openclaw cron create "0 18 * * 1-5" \
  "สรุปการปรับใช้ของวันนี้เป็น JSON" \
  --name "สรุปการปรับใช้" \
  --webhook "https://example.invalid/openclaw/cron"
```

ใช้ `--command` สำหรับงานรูปแบบเชลล์ที่ให้ผลลัพธ์แน่นอน ซึ่งทำงานภายใน Cron ของ OpenClaw โดยไม่เริ่มการเรียกใช้เอเจนต์/โมเดลแบบแยก:

```bash
openclaw cron create "*/15 * * * *" \
  --name "ตรวจสอบความลึกของคิว" \
  --command "scripts/check-queue.sh" \
  --command-cwd "/srv/app" \
  --announce \
  --channel telegram \
  --to "-1001234567890"
```

`--command <shell>` จัดเก็บ `argv: ["sh", "-lc", <shell>]` ใช้ `--command-argv '["node","scripts/report.mjs"]'` สำหรับการดำเนินการ argv อย่างแม่นยำ งานคำสั่งจะบันทึก stdout/stderr บันทึกประวัติ Cron ตามปกติ และส่งเอาต์พุตผ่านโหมดการนำส่ง `announce`, `webhook` หรือ `none` เช่นเดียวกับงานแบบแยก คำสั่งที่พิมพ์เฉพาะ `NO_REPLY` จะถูกระงับ

## เซสชัน

`--session` ยอมรับ `main`, `isolated`, `current` หรือ `session:<id>`

<AccordionGroup>
  <Accordion title="คีย์เซสชัน">
    - `main` ผูกกับเซสชันหลักของเอเจนต์
    - `isolated` สร้างทรานสคริปต์และรหัสเซสชันใหม่สำหรับการเรียกใช้แต่ละครั้ง
    - `current` ผูกกับเซสชันที่ใช้งานอยู่ ณ เวลาที่สร้าง
    - `session:<id>` ตรึงกับคีย์เซสชันถาวรที่ระบุอย่างชัดเจน

  </Accordion>
  <Accordion title="ความหมายของเซสชันแบบแยก">
    การเรียกใช้แบบแยกจะรีเซ็ตบริบทการสนทนาแวดล้อม การกำหนดเส้นทางช่องทางและกลุ่ม นโยบายการส่ง/การเข้าคิว การยกระดับ สิ่งที่มา และการผูกกับรันไทม์ ACP จะถูกรีเซ็ตสำหรับการเรียกใช้ใหม่ ค่ากำหนดที่ปลอดภัยและการแทนที่โมเดลหรือการตรวจสอบสิทธิ์ที่ผู้ใช้เลือกอย่างชัดเจนสามารถส่งต่อระหว่างการเรียกใช้ได้
  </Accordion>
</AccordionGroup>

## การนำส่ง

`openclaw cron list` และ `openclaw cron show <job-id>` แสดงตัวอย่างเส้นทางการนำส่งที่แก้ไขแล้ว สำหรับ `channel: "last"` ตัวอย่างจะแสดงว่าเส้นทางถูกแก้ไขจากเซสชันหลักหรือเซสชันปัจจุบัน หรือจะล้มเหลวแบบปิดกั้น

เป้าหมายที่มีคำนำหน้าผู้ให้บริการสามารถช่วยแยกความกำกวมของช่องทางประกาศที่ยังแก้ไขไม่ได้ ตัวอย่างเช่น `to: "telegram:123"` จะเลือก Telegram เมื่อไม่ได้ระบุ `delivery.channel` หรือเมื่อเป็น `last` เฉพาะคำนำหน้าที่ Plugin ซึ่งโหลดอยู่ประกาศรองรับเท่านั้นที่เป็นตัวเลือกผู้ให้บริการ หากระบุ `delivery.channel` อย่างชัดเจน คำนำหน้าต้องตรงกับช่องทางนั้น โดย `channel: "whatsapp"` ที่ใช้ร่วมกับ `to: "telegram:123"` จะถูกปฏิเสธ คำนำหน้าบริการ เช่น `imessage:` และ `sms:` ยังคงเป็นไวยากรณ์เป้าหมายที่ช่องทางเป็นเจ้าของ

<Note>
งาน `cron add` แบบแยกจะใช้การนำส่ง `--announce` เป็นค่าเริ่มต้น ใช้ `--no-deliver` เพื่อเก็บเอาต์พุตไว้ภายใน `--deliver` ยังคงเป็นนามแฝงที่เลิกใช้แล้วของ `--announce`
</Note>

### ความเป็นเจ้าของการนำส่ง

การนำส่งแชต Cron แบบแยกเป็นความรับผิดชอบร่วมกันระหว่างเอเจนต์กับตัวเรียกใช้:

- เอเจนต์สามารถส่งโดยตรงด้วยเครื่องมือ `message` เมื่อมีเส้นทางแชต
- `announce` จะนำส่งการตอบกลับสุดท้ายเป็นทางเลือกสำรอง เฉพาะเมื่อเอเจนต์ไม่ได้ส่งโดยตรงไปยังเป้าหมายที่แก้ไขแล้ว
- `webhook` ส่งเพย์โหลดที่เสร็จแล้วไปยัง URL
- `none` ปิดใช้การนำส่งสำรองของตัวเรียกใช้

ใช้ `cron add|create --webhook <url>` หรือ `cron edit <job-id> --webhook <url>` เพื่อตั้งค่าการนำส่ง Webhook ห้ามใช้ `--webhook` ร่วมกับแฟล็กการนำส่งแชต เช่น `--announce`, `--no-deliver`, `--channel`, `--to`, `--thread-id` หรือ `--account`

`cron edit <job-id>` สามารถยกเลิกการตั้งค่าฟิลด์การกำหนดเส้นทางการนำส่งแต่ละรายการด้วย `--clear-channel`, `--clear-to`, `--clear-thread-id` และ `--clear-account` (แต่ละรายการจะถูกปฏิเสธเมื่อใช้ร่วมกับแฟล็กตั้งค่าที่ตรงกัน) ต่างจาก `--no-deliver` ซึ่งเพียงปิดใช้การนำส่งสำรองของตัวเรียกใช้ ตัวเลือกเหล่านี้จะลบฟิลด์ที่จัดเก็บไว้ เพื่อให้งานแก้ไขส่วนนั้นของเส้นทางจากค่าเริ่มต้นอีกครั้ง

`--announce` คือการนำส่งสำรองของตัวเรียกใช้สำหรับการตอบกลับสุดท้าย `--no-deliver` ปิดใช้ทางเลือกสำรองดังกล่าว แต่ไม่ลบเครื่องมือ `message` ของเอเจนต์เมื่อมีเส้นทางแชต

การแจ้งเตือนที่สร้างจากแชตที่ใช้งานอยู่จะรักษาเป้าหมายการนำส่งของแชตปัจจุบันไว้สำหรับการนำส่งประกาศสำรอง คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็ก ห้ามใช้เป็นแหล่งข้อมูลจริงสำหรับรหัสผู้ให้บริการที่คำนึงถึงตัวพิมพ์เล็กและใหญ่ เช่น รหัสห้อง Matrix

### การนำส่งเมื่อเกิดความล้มเหลว

การแจ้งเตือนความล้มเหลวจะได้รับการแก้ไขตามลำดับต่อไปนี้:

1. `delivery.failureDestination` ในงาน
2. `cron.failureDestination` ส่วนกลาง
3. เป้าหมายประกาศหลักของงาน (เมื่อสองรายการข้างต้นไม่ได้แก้ไขเป็นปลายทางที่เจาะจง)

<Note>
งานในเซสชันหลักสามารถใช้ `delivery.failureDestination` ได้เฉพาะเมื่อโหมดการนำส่งหลักเป็น `webhook` งานแบบแยกยอมรับตัวเลือกนี้ในทุกโหมด
</Note>

การเรียกใช้ Cron แบบแยกจะถือว่าความล้มเหลวระดับการเรียกใช้ของเอเจนต์เป็นข้อผิดพลาดของงาน แม้จะไม่มีการสร้างเพย์โหลดตอบกลับ ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการจึงยังคงเพิ่มตัวนับข้อผิดพลาดและเรียกใช้การแจ้งเตือนความล้มเหลว

งานคำสั่ง Cron จะไม่เริ่มเทิร์นของเอเจนต์แบบแยก รหัสออกเป็นศูนย์จะบันทึก `ok`; รหัสออกที่ไม่ใช่ศูนย์ สัญญาณ การหมดเวลา หรือการหมดเวลาเพราะไม่มีเอาต์พุต จะบันทึก `error` และสามารถเรียกใช้เส้นทางการแจ้งเตือนความล้มเหลวเดียวกันได้

หากการเรียกใช้แบบแยกหมดเวลาก่อนคำขอโมเดลครั้งแรก `openclaw cron show` และ `openclaw cron runs` จะมีข้อผิดพลาดเฉพาะระยะ เช่น `setup timed out before runner start` หรือข้อความการหยุดชะงักที่ระบุระยะเริ่มต้นล่าสุดที่ทราบ (ตัวอย่างเช่น `context-engine`) สำหรับผู้ให้บริการที่ทำงานผ่าน CLI ตัวเฝ้าระวังก่อนเรียกโมเดลจะยังทำงานอยู่จนกว่าเทิร์นของ CLI ภายนอกจะเริ่มต้น ดังนั้นการหยุดชะงักในการค้นหาเซสชัน ฮุก การตรวจสอบสิทธิ์ พรอมต์ และการตั้งค่า CLI จะถูกรายงานเป็นความล้มเหลวของ Cron ก่อนเรียกโมเดล

## การจัดกำหนดการ

### งานที่เรียกใช้ครั้งเดียว

`--at <datetime>` จัดกำหนดการเรียกใช้ครั้งเดียว วันเวลาโดยไม่มีออฟเซ็ตจะถือว่าเป็น UTC เว้นแต่จะส่ง `--tz <iana>` ด้วย ซึ่งจะตีความเวลาตามนาฬิกาในเขตเวลาที่กำหนด

<Note>
งานที่เรียกใช้ครั้งเดียวจะถูกลบหลังสำเร็จโดยค่าเริ่มต้น ใช้ `--keep-after-run` เพื่อเก็บงานไว้
</Note>

### งานที่เกิดซ้ำ

งานที่เกิดซ้ำใช้การหน่วงเวลาก่อนลองใหม่แบบเอ็กซ์โพเนนเชียลหลังเกิดข้อผิดพลาดติดต่อกัน: 30s, 1m, 5m, 15m, 60m กำหนดการจะกลับเป็นปกติหลังการเรียกใช้ที่สำเร็จครั้งถัดไป

การเรียกใช้ที่ข้ามไปจะถูกติดตามแยกจากข้อผิดพลาดในการดำเนินการ และไม่ส่งผลต่อการหน่วงเวลาก่อนลองใหม่ แต่ `openclaw cron edit <job-id> --failure-alert-include-skipped` สามารถเลือกให้การแจ้งเตือนความล้มเหลวรวมการแจ้งเตือนซ้ำสำหรับการเรียกใช้ที่ข้ามไปได้

สำหรับงานแบบแยกที่กำหนดเป้าหมายไปยังผู้ให้บริการโมเดลภายในที่กำหนดค่าไว้ (URL ฐานบนลูปแบ็ก เครือข่ายส่วนตัว หรือ `.local`) Cron จะตรวจสอบผู้ให้บริการเบื้องต้นแบบเบาก่อนเริ่มเทิร์นของเอเจนต์ โดยผู้ให้บริการ `api: "ollama"` จะได้รับการตรวจสอบที่ `/api/tags`; ส่วนผู้ให้บริการที่เข้ากันได้กับ OpenAI ภายในรายอื่น (`api: "openai-completions"` เช่น vLLM, SGLang, LM Studio) จะได้รับการตรวจสอบที่ `/models` หากเข้าถึงปลายทางไม่ได้ การเรียกใช้จะถูกบันทึกเป็น `skipped` และลองใหม่ในกำหนดการภายหลัง ผลการตรวจสอบการเข้าถึงจะถูกแคชแยกตามปลายทางเป็นเวลา 5 นาที เพื่อไม่ให้งานจำนวนมากที่ใช้เซิร์ฟเวอร์ภายในเดียวกันส่งคำขอตรวจสอบซ้ำอย่างหนัก

งาน Cron สถานะรันไทม์ที่รอดำเนินการ และประวัติการเรียกใช้จะอยู่ในฐานข้อมูลสถานะ SQLite ที่ใช้ร่วมกัน ไฟล์เดิม `jobs.json`, `<name>-state.json` และ `runs/*.jsonl` จะถูกนำเข้าหนึ่งครั้งและเปลี่ยนชื่อด้วยส่วนต่อท้าย `.migrated` หลังนำเข้าแล้ว ให้แก้ไขกำหนดการด้วย `openclaw cron add|edit|remove` แทนการแก้ไขไฟล์ JSON

### การเรียกใช้ด้วยตนเอง

`openclaw cron run <job-id>` บังคับเรียกใช้โดยค่าเริ่มต้น และส่งคืนทันทีที่การเรียกใช้ด้วยตนเองถูกเพิ่มเข้าคิว การตอบกลับที่สำเร็จจะมี `{ ok: true, enqueued: true, runId }` ใช้ `runId` ที่ส่งคืนมาเพื่อตรวจสอบผลลัพธ์ในภายหลัง:

```bash
openclaw cron run <job-id>
openclaw cron runs --id <job-id> --run-id <run-id>
```

เพิ่ม `--wait` เมื่อสคริปต์ควรรอจนกว่าการเรียกใช้ในคิวรายการนั้นจะบันทึกสถานะสิ้นสุด:

```bash
openclaw cron run <job-id> --wait --wait-timeout 10m --poll-interval 2s
```

เมื่อใช้ `--wait` CLI จะยังคงเรียก `cron.run` ก่อน จากนั้นจึงสำรวจ `cron.runs` สำหรับ `runId` ที่ส่งคืนมา คำสั่งจะออกด้วย `0` เฉพาะเมื่อการเรียกใช้เสร็จสิ้นด้วยสถานะ `ok` โดยจะออกด้วยค่าที่ไม่ใช่ศูนย์เมื่อการเรียกใช้เสร็จสิ้นด้วย `error` หรือ `skipped`, เมื่อการตอบกลับของ Gateway ไม่มี `runId` หรือเมื่อ `--wait-timeout` หมดเวลา (ค่าเริ่มต้นคือ `10m` และสำรวจทุก `2s` ตามค่าเริ่มต้น) `--poll-interval` ต้องมากกว่าศูนย์

<Note>
ใช้ `--due` เมื่อต้องการให้คำสั่งด้วยตนเองทำงานเฉพาะในกรณีที่งานถึงกำหนดในขณะนั้น หาก `--due --wait` ไม่ได้เพิ่มการเรียกใช้เข้าคิว คำสั่งจะส่งคืนการตอบกลับปกติที่ไม่มีการเรียกใช้ แทนการสำรวจ
</Note>

## โมเดล

`cron add|edit --model <ref>` เลือกโมเดลที่อนุญาตสำหรับงาน `cron add|edit --fallbacks <list>` ตั้งค่าโมเดลสำรองแยกตามงาน ตัวอย่างเช่น `--fallbacks openrouter/gpt-4.1-mini,openai/gpt-5`; ส่ง `--fallbacks ""` เพื่อเรียกใช้อย่างเคร่งครัดโดยไม่มีโมเดลสำรอง `cron edit <job-id> --clear-fallbacks` ลบการแทนที่โมเดลสำรองแยกตามงาน `cron edit <job-id> --clear-model` ลบการแทนที่โมเดลแยกตามงาน เพื่อให้งานใช้ลำดับความสำคัญในการเลือกโมเดล Cron ตามปกติ (การแทนที่เซสชัน Cron ที่จัดเก็บไว้ หากมี มิฉะนั้นจะใช้โมเดลของเอเจนต์/ค่าเริ่มต้น) และไม่สามารถใช้ร่วมกับ `--model` ได้ `cron add|edit --thinking <level>` ตั้งค่าการแทนที่ระดับการคิดแยกตามงาน ส่วน `cron edit <job-id> --clear-thinking` จะลบการแทนที่ดังกล่าว เพื่อให้งานใช้ลำดับความสำคัญของการคิด Cron ตามปกติ และไม่สามารถใช้ร่วมกับ `--thinking` ได้

<Warning>
หากโมเดลไม่ได้รับอนุญาตหรือไม่สามารถแก้ไขได้ Cron จะทำให้การเรียกใช้ล้มเหลวด้วยข้อผิดพลาดการตรวจสอบที่ชัดเจน แทนการย้อนกลับไปใช้การเลือกโมเดลของเอเจนต์หรือโมเดลเริ่มต้นของงาน
</Warning>

`--model` ของ Cron คือ **โมเดลหลักของงาน** ไม่ใช่การแทนที่ `/model` ของเซสชันแชต ซึ่งหมายความว่า:

- โมเดลสำรองที่กำหนดค่าไว้ยังคงถูกใช้เมื่อโมเดลของงานที่เลือกไว้ล้มเหลว
- `fallbacks` ในเพย์โหลดแยกตามงานจะแทนที่รายการสำรองที่กำหนดค่าไว้เมื่อมีการระบุ
- รายการสำรองแยกตามงานที่ว่างเปล่า (`--fallbacks ""` หรือ `fallbacks: []` ในเพย์โหลด/API ของงาน) ทำให้การเรียกใช้ Cron เป็นแบบเคร่งครัด
- เมื่องานมี `--model` แต่ไม่มีรายการสำรองที่กำหนดค่าไว้ OpenClaw จะส่งการแทนที่รายการสำรองแบบว่างอย่างชัดเจน เพื่อไม่ให้โมเดลหลักของเอเจนต์ถูกเพิ่มเป็นเป้าหมายการลองใหม่ที่ซ่อนอยู่
- การตรวจสอบล่วงหน้าของผู้ให้บริการภายในจะไล่ตรวจสอบโมเดลสำรองที่กำหนดค่าไว้ ก่อนทำเครื่องหมายการเรียกใช้ Cron เป็น `skipped`

`openclaw doctor` รายงานงานที่ตั้งค่า `payload.model` ไว้แล้ว รวมถึงจำนวนตามเนมสเปซของผู้ให้บริการและความไม่ตรงกันกับ `agents.defaults.model` ใช้การตรวจสอบนี้เมื่อพฤติกรรมด้านการตรวจสอบสิทธิ์ ผู้ให้บริการ หรือการเรียกเก็บเงินแตกต่างกันระหว่างแชตสดกับงานตามกำหนดการ

### ลำดับความสำคัญของโมเดล Cron แบบแยก

Cron แบบแยกจะแก้ไขโมเดลที่ใช้งานอยู่ตามลำดับต่อไปนี้:

1. การแทนที่จากฮุก Gmail
2. `--model` แยกตามงาน
3. การแทนที่โมเดลของเซสชัน Cron ที่จัดเก็บไว้ (เมื่อผู้ใช้เลือกไว้)
4. การเลือกโมเดลของเอเจนต์หรือโมเดลเริ่มต้น

### โหมดเร็ว

โหมดเร็วของ Cron แบบแยกทำงานตามการเลือกโมเดลสดที่ผ่านการแก้ค่าแล้ว การกำหนดค่าโมเดล `params.fastMode` จะมีผลโดยค่าเริ่มต้น แต่การเขียนทับ `fastMode` ของเซสชันที่จัดเก็บไว้ยังคงมีลำดับความสำคัญเหนือกว่าการกำหนดค่า เมื่อโหมดที่ผ่านการแก้ค่าแล้วเป็น `auto` การตัดสิ้นจะใช้ค่า `params.fastAutoOnSeconds` ของโมเดลที่เลือก โดยมีค่าเริ่มต้นเป็น 60 วินาที

### การลองใหม่เมื่อสลับโมเดลสด

หากการทำงานแบบแยกส่งข้อผิดพลาด `LiveSessionModelSwitchError` Cron จะคงผู้ให้บริการและโมเดลที่สลับแล้วไว้ (รวมถึงการเขียนทับโปรไฟล์การยืนยันตัวตนที่สลับแล้ว หากมี) สำหรับการทำงานที่กำลังดำเนินอยู่ก่อนลองใหม่ ลูปการลองใหม่ชั้นนอกจำกัดการลองสลับไว้สองครั้งหลังจากความพยายามครั้งแรก จากนั้นจะยุติแทนที่จะวนซ้ำตลอดไป

## เอาต์พุตการทำงานและการปฏิเสธ

### การระงับการตอบรับที่ล้าสมัย

รอบการทำงาน Cron แบบแยกจะระงับการตอบกลับที่เป็นเพียงการรับทราบซึ่งล้าสมัย หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะระหว่างดำเนินการ และไม่มีการทำงานของเอเจนต์ย่อยที่สืบทอดลงมารับผิดชอบคำตอบสุดท้าย Cron จะส่งพรอมต์ซ้ำหนึ่งครั้งเพื่อขอผลลัพธ์จริงก่อนส่งมอบ

### การระงับโทเค็นเงียบ

หากการทำงาน Cron แบบแยกส่งคืนเฉพาะโทเค็นเงียบ (`NO_REPLY` หรือ `no_reply`) Cron จะระงับทั้งการส่งออกโดยตรงและเส้นทางสรุปสำรองที่เข้าคิว จึงไม่มีสิ่งใดถูกโพสต์กลับไปยังแชต

### การปฏิเสธแบบมีโครงสร้าง

การทำงาน Cron แบบแยกใช้เมทาดาทาการปฏิเสธการดำเนินการแบบมีโครงสร้างจากการทำงานที่ฝังอยู่ (ข้อผิดพลาดร้ายแรงของเครื่องมือดำเนินการที่มีรหัส `SYSTEM_RUN_DENIED` หรือ `INVALID_REQUEST`) เป็นสัญญาณการปฏิเสธที่เชื่อถือได้ นอกจากนี้ยังรองรับ Wrapper `UNAVAILABLE` ของโฮสต์ Node ที่ห่อหุ้มข้อผิดพลาดแบบมีโครงสร้างซ้อนอยู่ซึ่งมีหนึ่งในรหัสดังกล่าว

Cron จะไม่จัดประเภทข้อความร้อยแก้วในเอาต์พุตสุดท้ายหรือวลีปฏิเสธที่ดูเหมือนรอการอนุมัติว่าเป็นการปฏิเสธ เว้นแต่การทำงานที่ฝังอยู่จะให้เมทาดาทาการปฏิเสธแบบมีโครงสร้างด้วย เพื่อไม่ให้ข้อความทั่วไปของผู้ช่วยถูกมองว่าเป็นคำสั่งที่ถูกบล็อก

`cron list` และประวัติการทำงานจะแสดงเหตุผลที่ปฏิเสธ แทนที่จะรายงานคำสั่งที่ถูกบล็อกเป็น `ok`

## การเก็บรักษา

ลักษณะการเก็บรักษา:

- `cron.sessionRetention` (ค่าเริ่มต้น `24h` หรือ `false` เพื่อปิดใช้งาน) จะล้างเซสชันการทำงานแบบแยกที่เสร็จสมบูรณ์แล้ว
- ประวัติการทำงานจะเก็บแถวสถานะสิ้นสุดล่าสุด 2000 แถวต่องาน Cron แต่ละงาน แถวที่สูญหายยังคงใช้ช่วงเวลาล้างงานที่สูญหายมาตรฐาน 24 ชั่วโมง

## การย้ายงานรุ่นเก่า

<Note>
หากมีงาน Cron จากก่อนรูปแบบการส่งมอบและการจัดเก็บปัจจุบัน ให้เรียกใช้ `openclaw doctor --fix` Doctor จะปรับฟิลด์ Cron แบบเดิมให้เป็นมาตรฐาน (`jobId`, `schedule.cron`, ฟิลด์การส่งมอบระดับบนสุด รวมถึง `threadId` แบบเดิม และนามแฝงการส่งมอบ `provider` ในเพย์โหลด) และย้ายงาน Webhook สำรอง `notify: true` จากค่า `cron.webhook` แบบดิบที่เลิกใช้แล้วไปเป็นการส่งมอบผ่าน Webhook อย่างชัดเจน ก่อนลบคีย์การกำหนดค่านั้น งานที่ประกาศไปยังแชตอยู่แล้วจะคงการส่งมอบนั้นไว้และได้รับปลายทาง Webhook เมื่อเสร็จสมบูรณ์ หากไม่มี Webhook แบบเดิม เครื่องหมาย `notify` ระดับบนสุดที่ไม่ทำงานจะถูกลบออกจากงานที่ไม่มีเป้าหมายสำหรับการย้าย (การส่งมอบที่มีอยู่จะถูกเก็บไว้โดยไม่เปลี่ยนแปลง) เพื่อให้ `doctor --fix` ไม่แจ้งเตือนซ้ำเกี่ยวกับงานเหล่านั้นอีก
</Note>

## การแก้ไขทั่วไป

อัปเดตการตั้งค่าการส่งมอบโดยไม่เปลี่ยนข้อความ:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

ปิดใช้งานการส่งมอบสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --no-deliver
```

เปิดใช้งานบริบทเริ่มต้นแบบเบาสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --light-context
```

ประกาศไปยังช่องทางที่ระบุ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

ประกาศไปยังหัวข้อฟอรัม Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

สร้างงานแบบแยกพร้อมบริบทเริ่มต้นแบบเบา:

```bash
openclaw cron create "0 7 * * *" \
  "สรุปการอัปเดตในช่วงข้ามคืน" \
  --name "สรุปยามเช้าแบบเบา" \
  --session isolated \
  --light-context \
  --no-deliver
```

`--light-context` มีผลเฉพาะกับงานแบบรอบการทำงานของเอเจนต์ที่แยกเท่านั้น สำหรับการทำงาน Cron โหมดแบบเบาจะปล่อยบริบทเริ่มต้นให้ว่าง แทนที่จะแทรกชุดบริบทเริ่มต้นของพื้นที่ทำงานทั้งหมด

สร้างงานคำสั่งที่ระบุ argv, cwd, env, stdin และขีดจำกัดเอาต์พุตอย่างแม่นยำ:

```bash
openclaw cron create "*/30 * * * *" \
  --name "ส่งออกตำแหน่ง" \
  --command-argv '["node","scripts/export-position.mjs"]' \
  --command-cwd "/srv/app" \
  --command-env "NODE_ENV=production" \
  --command-input '{"mode":"summary"}' \
  --timeout-seconds 120 \
  --no-output-timeout-seconds 30 \
  --output-max-bytes 65536 \
  --webhook "https://example.invalid/openclaw/cron"
```

## คำสั่งผู้ดูแลระบบทั่วไป

การเรียกใช้และการตรวจสอบด้วยตนเอง:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron get <job-id>
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron run <job-id> --wait --wait-timeout 10m
openclaw cron run <job-id> --wait --wait-timeout 10m --poll-interval 2s
openclaw cron runs --id <job-id> --limit 50
openclaw cron runs --id <job-id> --run-id <run-id>
```

`openclaw cron list` จะแสดงงานที่ตรงกันทั้งหมดโดยค่าเริ่มต้น ส่ง `--agent <id>` เพื่อแสดงเฉพาะงานที่ ID เอเจนต์มาตรฐานที่มีผลตรงกัน งานที่ไม่มี ID เอเจนต์ที่จัดเก็บไว้จะนับว่าใช้เอเจนต์เริ่มต้นที่กำหนดค่าไว้

`openclaw cron get <job-id>` จะส่งคืน JSON ของงานที่จัดเก็บไว้โดยตรง ใช้ `cron show <job-id>` เมื่อต้องการมุมมองที่มนุษย์อ่านได้พร้อมตัวอย่างเส้นทางการส่งมอบ

`cron list --json` และ `cron show <job-id> --json` จะมีฟิลด์ระดับบนสุด `status` ในแต่ละงาน ซึ่งคำนวณจาก `enabled`, `state.runningAtMs` และ `state.lastRunStatus` ค่าที่เป็นไปได้: `disabled`, `running`, `ok`, `error`, `skipped` หรือ `idle` สถานะ JSON ยังคงเป็นรูปแบบมาตรฐานและไม่มีการตกแต่ง เพื่อให้เครื่องมือภายนอกอ่านสถานะงานได้โดยไม่ต้องคำนวณซ้ำ ส่วนเอาต์พุตสำหรับมนุษย์อาจตกแต่งสถานะ `error` ที่เกิดซ้ำด้วยจำนวนความล้มเหลว

รายการ `cron runs` มีข้อมูลวินิจฉัยการส่งมอบ ซึ่งรวมถึงเป้าหมาย Cron ที่ตั้งใจไว้ เป้าหมายที่แก้ค่าแล้ว การส่งด้วยเครื่องมือข้อความ การใช้ทางเลือกสำรอง และสถานะการส่งมอบ

การกำหนดเป้าหมายเอเจนต์และเซสชันใหม่:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` จะแจ้งเตือนเมื่อไม่ได้ระบุ `--agent` ในงานแบบรอบการทำงานของเอเจนต์ และจะย้อนกลับไปใช้เอเจนต์เริ่มต้น (`main`) ส่ง `--agent <id>` ขณะสร้างเพื่อปักหมุดเอเจนต์ที่ระบุ

การปรับแต่งการส่งมอบ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --webhook "https://example.invalid/openclaw/cron"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
