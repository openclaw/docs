# `openclaw cron`

จัดการงาน Cron สำหรับตัวจัดตารางเวลาของ Gateway

<Tip>
เรียกใช้ `openclaw cron --help` เพื่อดูพื้นผิวคำสั่งทั้งหมด ดู [งาน Cron](/th/automation/cron-jobs) สำหรับคู่มือแนวคิด
</Tip>

## เซสชัน

`--session` รับ `main`, `isolated`, `current` หรือ `session:<id>`

<AccordionGroup>
  <Accordion title="คีย์เซสชัน">
    - `main` ผูกกับเซสชันหลักของเอเจนต์
    - `isolated` สร้างทรานสคริปต์และรหัสเซสชันใหม่สำหรับการรันแต่ละครั้ง
    - `current` ผูกกับเซสชันที่ใช้งานอยู่ ณ เวลาที่สร้าง
    - `session:<id>` ปักหมุดกับคีย์เซสชันถาวรที่ระบุชัดเจน

  </Accordion>
  <Accordion title="ความหมายของเซสชันแยก">
    การรันแบบแยกจะรีเซ็ตบริบทการสนทนาแวดล้อม การกำหนดเส้นทางช่องและกลุ่ม นโยบายส่ง/เข้าคิว การยกระดับ ต้นทาง และการผูก ACP runtime จะถูกรีเซ็ตสำหรับการรันใหม่ การตั้งค่าที่ปลอดภัยและการแทนที่โมเดลหรือการตรวจสอบสิทธิ์ที่ผู้ใช้เลือกอย่างชัดเจนสามารถส่งต่อข้ามการรันได้
  </Accordion>
</AccordionGroup>

## การส่ง

`openclaw cron list` และ `openclaw cron show <job-id>` แสดงตัวอย่างเส้นทางการส่งที่แก้ไขแล้ว สำหรับ `channel: "last"` ตัวอย่างจะแสดงว่าเส้นทางแก้ไขจากเซสชันหลักหรือเซสชันปัจจุบัน หรือจะล้มเหลวแบบปิด

เป้าหมายที่มีคำนำหน้าผู้ให้บริการสามารถแยกความกำกวมของช่องประกาศที่ยังแก้ไขไม่ได้ ตัวอย่างเช่น `to: "telegram:123"` เลือก Telegram เมื่อ `delivery.channel` ถูกละไว้หรือเป็น `last` เฉพาะคำนำหน้าที่ Plugin ที่โหลดประกาศไว้เท่านั้นจึงเป็นตัวเลือกผู้ให้บริการ หาก `delivery.channel` ระบุชัดเจน คำนำหน้าต้องตรงกับช่องนั้น; `channel: "whatsapp"` พร้อม `to: "telegram:123"` จะถูกปฏิเสธ คำนำหน้าบริการ เช่น `imessage:` และ `sms:` ยังคงเป็นไวยากรณ์เป้าหมายที่ช่องเป็นเจ้าของ

<Note>
งาน `cron add` แบบแยกมีค่าเริ่มต้นเป็นการส่งแบบ `--announce` ใช้ `--no-deliver` เพื่อเก็บเอาต์พุตไว้ภายใน `--deliver` ยังคงอยู่ในฐานะนามแฝงที่เลิกใช้แล้วของ `--announce`
</Note>

### ความเป็นเจ้าของการส่ง

การส่งแชท Cron แบบแยกเป็นความรับผิดชอบร่วมกันระหว่างเอเจนต์และตัวรัน:

- เอเจนต์สามารถส่งโดยตรงโดยใช้เครื่องมือ `message` เมื่อมีเส้นทางแชทพร้อมใช้งาน
- `announce` จะส่งสำรองเฉพาะคำตอบสุดท้ายเมื่อเอเจนต์ไม่ได้ส่งโดยตรงไปยังเป้าหมายที่แก้ไขแล้ว
- `webhook` โพสต์ payload ที่เสร็จแล้วไปยัง URL
- `none` ปิดใช้งานการส่งสำรองของตัวรัน

`--announce` คือการส่งสำรองของตัวรันสำหรับคำตอบสุดท้าย `--no-deliver` ปิดใช้งานการส่งสำรองนั้น แต่ไม่ได้ลบเครื่องมือ `message` ของเอเจนต์เมื่อมีเส้นทางแชทพร้อมใช้งาน

การเตือนความจำที่สร้างจากแชทที่ใช้งานอยู่จะรักษาเป้าหมายการส่งแชทสดไว้สำหรับการส่งประกาศสำรอง คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็ก อย่าใช้เป็นแหล่งข้อมูลจริงสำหรับรหัสผู้ให้บริการที่ไวต่อตัวพิมพ์ใหญ่เล็ก เช่น รหัสห้อง Matrix

### การส่งเมื่อเกิดความล้มเหลว

การแจ้งเตือนความล้มเหลวจะแก้ไขตามลำดับนี้:

1. `delivery.failureDestination` บนงาน
2. `cron.failureDestination` ส่วนกลาง
3. เป้าหมายประกาศหลักของงาน (เมื่อไม่ได้ตั้งปลายทางความล้มเหลวไว้อย่างชัดเจน)

<Note>
งานเซสชันหลักสามารถใช้ `delivery.failureDestination` ได้เฉพาะเมื่อโหมดการส่งหลักเป็น `webhook` งานแบบแยกรับค่าได้ในทุกโหมด
</Note>

หมายเหตุ: การรัน Cron แบบแยกจะถือว่าความล้มเหลวระดับการรันของเอเจนต์เป็นข้อผิดพลาดของงานแม้เมื่อ
ไม่มี payload คำตอบถูกสร้างขึ้น ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการยังคงเพิ่ม
ตัวนับข้อผิดพลาดและทริกเกอร์การแจ้งเตือนความล้มเหลว

หากการรันแบบแยกหมดเวลาก่อนคำขอโมเดลแรก `openclaw cron show`
และ `openclaw cron runs` จะรวมข้อผิดพลาดเฉพาะเฟส เช่น
`setup timed out before runner start` หรือ
`stalled before first model call (last phase: context-engine)`
สำหรับผู้ให้บริการที่รองรับด้วย CLI ตัวเฝ้าระวังก่อนโมเดลจะยังคงทำงานจนกว่าเทิร์น CLI
ภายนอกจะเริ่ม ดังนั้นการค้นหาเซสชัน hook การตรวจสอบสิทธิ์ prompt และการค้างของการตั้งค่า CLI
จะถูกรายงานเป็นความล้มเหลว Cron ก่อนโมเดล

## การจัดตารางเวลา

### งานครั้งเดียว

`--at <datetime>` จัดตารางการรันครั้งเดียว วันที่และเวลาที่ไม่มีออฟเซ็ตจะถือเป็น UTC เว้นแต่คุณจะส่ง `--tz <iana>` ด้วย ซึ่งจะตีความเวลาตามนาฬิกาในเขตเวลาที่กำหนด

<Note>
งานครั้งเดียวจะลบหลังสำเร็จตามค่าเริ่มต้น ใช้ `--keep-after-run` เพื่อเก็บไว้
</Note>

### งานที่เกิดซ้ำ

งานที่เกิดซ้ำใช้การถอยหลังเพื่อ retry แบบเอ็กซ์โพเนนเชียลหลังข้อผิดพลาดต่อเนื่อง: 30s, 1m, 5m, 15m, 60m ตารางเวลาจะกลับสู่ปกติหลังจากการรันที่สำเร็จครั้งถัดไป

การรันที่ถูกข้ามจะถูกติดตามแยกจากข้อผิดพลาดในการดำเนินการ การรันเหล่านี้ไม่ส่งผลต่อการถอยหลังเพื่อ retry แต่ `openclaw cron edit <job-id> --failure-alert-include-skipped` สามารถเลือกให้การแจ้งเตือนความล้มเหลวรวมการแจ้งเตือนการรันที่ถูกข้ามซ้ำได้

สำหรับงานแบบแยกที่กำหนดเป้าหมายผู้ให้บริการโมเดลที่กำหนดค่าไว้ภายในเครื่อง Cron จะรัน preflight ผู้ให้บริการแบบเบาก่อนเริ่มเทิร์นเอเจนต์ ผู้ให้บริการ `api: "ollama"` แบบ Loopback, private-network และ `.local` จะถูกตรวจสอบที่ `/api/tags`; ผู้ให้บริการที่เข้ากันได้กับ OpenAI ภายในเครื่อง เช่น vLLM, SGLang และ LM Studio จะถูกตรวจสอบที่ `/models` หากเข้าถึง endpoint ไม่ได้ การรันจะถูกบันทึกเป็น `skipped` และ retry ในตารางเวลาภายหลัง; endpoint ที่ตายแล้วซึ่งตรงกันจะถูกแคชไว้ 5 นาทีเพื่อหลีกเลี่ยงไม่ให้หลายงานถล่มเซิร์ฟเวอร์ภายในเครื่องเดียวกัน

หมายเหตุ: นิยามงาน Cron อยู่ใน `jobs.json` ขณะที่สถานะ runtime ที่ค้างอยู่จะอยู่ใน `jobs-state.json` หาก `jobs.json` ถูกแก้ไขจากภายนอก Gateway จะโหลดตารางเวลาที่เปลี่ยนแปลงใหม่และล้างสล็อตที่ค้างเก่า การเขียนใหม่เฉพาะรูปแบบจะไม่ล้างสล็อตที่ค้างอยู่

### การรันด้วยตนเอง

`openclaw cron run` จะคืนค่าทันทีที่การรันด้วยตนเองถูกเข้าคิว การตอบสนองที่สำเร็จจะรวม `{ ok: true, enqueued: true, runId }` ใช้ `openclaw cron runs --id <job-id>` เพื่อติดตามผลลัพธ์สุดท้าย

<Note>
`openclaw cron run <job-id>` บังคับรันตามค่าเริ่มต้น ใช้ `--due` เพื่อรักษาพฤติกรรมแบบเก่า “รันเฉพาะเมื่อถึงกำหนด” ไว้
</Note>

## โมเดล

`cron add|edit --model <ref>` เลือกโมเดลที่อนุญาตสำหรับงาน

<Warning>
หากโมเดลไม่ได้รับอนุญาตหรือไม่สามารถแก้ไขได้ Cron จะทำให้การรันล้มเหลวด้วยข้อผิดพลาดการตรวจสอบความถูกต้องที่ชัดเจน แทนที่จะ fallback ไปยังการเลือกเอเจนต์ของงานหรือโมเดลเริ่มต้น
</Warning>

Cron `--model` เป็น **โมเดลหลักของงาน** ไม่ใช่การแทนที่ `/model` ของเซสชันแชท ซึ่งหมายความว่า:

- fallback ของโมเดลที่กำหนดค่าไว้ยังคงมีผลเมื่อโมเดลงานที่เลือกไว้ล้มเหลว
- `fallbacks` ใน payload รายงานจะแทนที่รายการ fallback ที่กำหนดค่าไว้เมื่อมีอยู่
- รายการ fallback ต่อหนึ่งงานที่ว่างเปล่า (`fallbacks: []` ใน payload/API ของงาน) ทำให้การรัน Cron เข้มงวด
- เมื่องานมี `--model` แต่ไม่ได้กำหนดค่ารายการ fallback ไว้ OpenClaw จะส่งการแทนที่ fallback ว่างเปล่าอย่างชัดเจน เพื่อไม่ให้โมเดลหลักของเอเจนต์ถูกต่อท้ายเป็นเป้าหมาย retry ที่ซ่อนอยู่

### ลำดับความสำคัญของโมเดล Cron แบบแยก

Cron แบบแยกจะแก้ไขโมเดลที่ใช้งานอยู่ตามลำดับนี้:

1. การแทนที่จาก Gmail-hook
2. `--model` ต่อหนึ่งงาน
3. การแทนที่โมเดลของเซสชัน Cron ที่เก็บไว้ (เมื่อผู้ใช้เลือกไว้)
4. การเลือกเอเจนต์หรือโมเดลเริ่มต้น

### โหมดเร็ว

โหมดเร็วของ Cron แบบแยกจะตามการเลือกโมเดลสดที่แก้ไขแล้ว ค่า config ของโมเดล `params.fastMode` มีผลตามค่าเริ่มต้น แต่การแทนที่ `fastMode` ของเซสชันที่เก็บไว้ยังคงชนะ config

### การ retry เมื่อสลับโมเดลสด

หากการรันแบบแยกโยน `LiveSessionModelSwitchError` Cron จะคงค่าผู้ให้บริการและโมเดลที่สลับแล้ว (และการแทนที่โปรไฟล์การตรวจสอบสิทธิ์ที่สลับแล้วเมื่อมีอยู่) สำหรับการรันที่ใช้งานอยู่ก่อน retry ลูป retry ชั้นนอกถูกจำกัดไว้ที่การ retry การสลับสองครั้งหลังความพยายามเริ่มต้น จากนั้นจะยกเลิกแทนที่จะวนซ้ำตลอดไป

## เอาต์พุตการรันและการปฏิเสธ

### การระงับการยืนยันรับที่เก่า

เทิร์น Cron แบบแยกจะระงับคำตอบที่เป็นเพียงการยืนยันรับที่เก่า หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะชั่วคราวและไม่มีการรัน subagent ลูกหลานรับผิดชอบคำตอบสุดท้าย Cron จะ prompt อีกครั้งหนึ่งเพื่อขอผลลัพธ์จริงก่อนการส่ง

### การระงับโทเค็นเงียบ

หากการรัน Cron แบบแยกคืนเฉพาะโทเค็นเงียบ (`NO_REPLY` หรือ `no_reply`) Cron จะระงับทั้งการส่งออกโดยตรงและเส้นทางสรุปที่เข้าคิวสำรอง ดังนั้นจะไม่มีสิ่งใดถูกโพสต์กลับไปยังแชท

### การปฏิเสธแบบมีโครงสร้าง

การรัน Cron แบบแยกจะเลือก metadata การปฏิเสธการดำเนินการแบบมีโครงสร้างจากการรันที่ฝังอยู่ก่อน จากนั้นจึง fallback ไปยังตัวบ่งชี้การปฏิเสธที่รู้จักในเอาต์พุตสุดท้าย เช่น `SYSTEM_RUN_DENIED`, `INVALID_REQUEST` และวลีปฏิเสธการผูกการอนุมัติ

`cron list` และประวัติการรันจะแสดงเหตุผลการปฏิเสธแทนการรายงานคำสั่งที่ถูกบล็อกเป็น `ok`

## การเก็บรักษา

การเก็บรักษาและการตัดแต่งถูกควบคุมใน config:

- `cron.sessionRetention` (ค่าเริ่มต้น `24h`) ตัดแต่งเซสชันการรันแบบแยกที่เสร็จแล้ว
- `cron.runLog.maxBytes` และ `cron.runLog.keepLines` ตัดแต่ง `~/.openclaw/cron/runs/<jobId>.jsonl`

## การย้ายงานเก่า

<Note>
หากคุณมีงาน Cron จากก่อนรูปแบบการส่งและการจัดเก็บปัจจุบัน ให้รัน `openclaw doctor --fix` Doctor จะทำให้ฟิลด์ Cron แบบ legacy เป็นมาตรฐาน (`jobId`, `schedule.cron`, ฟิลด์การส่งระดับบนรวมถึง `threadId` แบบ legacy, นามแฝงการส่ง `provider` ของ payload) และย้ายงาน fallback webhook แบบง่ายที่เป็น `notify: true` ไปเป็นการส่ง webhook ที่ชัดเจนเมื่อมีการกำหนดค่า `cron.webhook`
</Note>

## การแก้ไขทั่วไป

อัปเดตการตั้งค่าการส่งโดยไม่เปลี่ยนข้อความ:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

ปิดใช้งานการส่งสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --no-deliver
```

เปิดใช้งานบริบท bootstrap แบบเบาสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --light-context
```

ประกาศไปยังช่องเฉพาะ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

ประกาศไปยังหัวข้อฟอรัม Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

สร้างงานแบบแยกพร้อมบริบท bootstrap แบบเบา:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` มีผลเฉพาะกับงานเทิร์นเอเจนต์แบบแยกเท่านั้น สำหรับการรัน Cron โหมดเบาจะคงบริบท bootstrap ให้ว่างแทนการฉีดชุด bootstrap ของ workspace ทั้งหมด

## คำสั่งผู้ดูแลทั่วไป

การรันด้วยตนเองและการตรวจสอบ:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

`openclaw cron list` แสดงงานที่ตรงกันทั้งหมดตามค่าเริ่มต้น ส่ง `--agent <id>` เพื่อแสดงเฉพาะงานที่รหัสเอเจนต์ที่ normalize แล้วซึ่งมีผลตรงกัน; งานที่ไม่มีรหัสเอเจนต์ที่เก็บไว้จะนับเป็นเอเจนต์เริ่มต้นที่กำหนดค่าไว้

`cron list --json` และ `cron show <job-id> --json` รวมฟิลด์ `status` ระดับบนในแต่ละงาน ซึ่งคำนวณจาก `enabled`, `state.runningAtMs` และ `state.lastRunStatus` ค่า: `disabled`, `running`, `ok`, `error`, `skipped` หรือ `idle` ฟิลด์นี้สะท้อนคอลัมน์สถานะที่มนุษย์อ่านได้ เพื่อให้เครื่องมือภายนอกอ่านสถานะงานได้โดยไม่ต้องคำนวณใหม่

รายการ `cron runs` รวม diagnostics การส่งพร้อมเป้าหมาย Cron ที่ตั้งใจ เป้าหมายที่แก้ไขแล้ว การส่งด้วยเครื่องมือ message การใช้ fallback และสถานะการส่งสำเร็จ

การเปลี่ยนเป้าหมายเอเจนต์และเซสชัน:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` เตือนเมื่อ `--agent` ถูกละไว้ในงานเทิร์นเอเจนต์และ fallback ไปยังเอเจนต์เริ่มต้น (`main`) ส่ง `--agent <id>` ตอนสร้างเพื่อปักหมุดเอเจนต์เฉพาะ

การปรับแต่งการส่ง:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
