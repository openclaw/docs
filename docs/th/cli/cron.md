# `openclaw cron`

จัดการงาน Cron สำหรับตัวกำหนดเวลาของ Gateway

<Tip>
เรียกใช้ `openclaw cron --help` เพื่อดูพื้นผิวคำสั่งทั้งหมด ดู [งาน Cron](/th/automation/cron-jobs) สำหรับคู่มือเชิงแนวคิด
</Tip>

## เซสชัน

`--session` รับค่า `main`, `isolated`, `current` หรือ `session:<id>`

<AccordionGroup>
  <Accordion title="คีย์เซสชัน">
    - `main` ผูกกับเซสชันหลักของเอเจนต์
    - `isolated` สร้างทรานสคริปต์ใหม่และรหัสเซสชันใหม่สำหรับแต่ละรัน
    - `current` ผูกกับเซสชันที่ใช้งานอยู่ ณ เวลาที่สร้าง
    - `session:<id>` ปักหมุดกับคีย์เซสชันถาวรที่ระบุอย่างชัดเจน

  </Accordion>
  <Accordion title="ความหมายของเซสชันแบบแยก">
    การรันแบบแยกจะรีเซ็ตบริบทการสนทนาแวดล้อม การกำหนดเส้นทางช่องและกลุ่ม, นโยบายส่ง/คิว, การยกระดับ, ต้นทาง และการผูก ACP runtime จะถูกรีเซ็ตสำหรับการรันใหม่ การตั้งค่าที่ปลอดภัยและการแทนที่โมเดลหรือการยืนยันตัวตนที่ผู้ใช้เลือกไว้อย่างชัดเจนสามารถส่งต่อข้ามการรันได้
  </Accordion>
</AccordionGroup>

## การจัดส่ง

`openclaw cron list` และ `openclaw cron show <job-id>` แสดงตัวอย่างเส้นทางการจัดส่งที่ resolve แล้ว สำหรับ `channel: "last"` ตัวอย่างจะแสดงว่าเส้นทาง resolve จากเซสชันหลักหรือเซสชันปัจจุบัน หรือจะ fail closed

เป้าหมายที่มีคำนำหน้าผู้ให้บริการสามารถแยกแยะช่องประกาศที่ยัง resolve ไม่ได้ ตัวอย่างเช่น `to: "telegram:123"` จะเลือก Telegram เมื่อ `delivery.channel` ถูกละไว้หรือเป็น `last` เฉพาะคำนำหน้าที่ Plugin ที่โหลดอยู่ประกาศไว้เท่านั้นที่เป็นตัวเลือกผู้ให้บริการ หาก `delivery.channel` ระบุชัดเจน คำนำหน้าต้องตรงกับช่องนั้น; `channel: "whatsapp"` พร้อม `to: "telegram:123"` จะถูกปฏิเสธ คำนำหน้าบริการ เช่น `imessage:` และ `sms:` ยังคงเป็นไวยากรณ์เป้าหมายที่ช่องเป็นเจ้าของ

<Note>
งาน `cron add` แบบแยกจะใช้การจัดส่งแบบ `--announce` เป็นค่าเริ่มต้น ใช้ `--no-deliver` เพื่อเก็บเอาต์พุตไว้ภายใน `--deliver` ยังคงเป็นนามแฝงที่เลิกใช้แล้วของ `--announce`
</Note>

### ความเป็นเจ้าของการจัดส่ง

การจัดส่งแชต Cron แบบแยกใช้ร่วมกันระหว่างเอเจนต์และตัวรัน:

- เอเจนต์สามารถส่งโดยตรงโดยใช้เครื่องมือ `message` เมื่อมีเส้นทางแชตพร้อมใช้งาน
- `announce` จะจัดส่งคำตอบสุดท้ายแบบ fallback เฉพาะเมื่อเอเจนต์ไม่ได้ส่งโดยตรงไปยังเป้าหมายที่ resolve แล้ว
- `webhook` โพสต์ payload ที่เสร็จแล้วไปยัง URL
- `none` ปิดการจัดส่ง fallback ของตัวรัน

`--announce` คือการจัดส่ง fallback ของตัวรันสำหรับคำตอบสุดท้าย `--no-deliver` ปิด fallback นั้น แต่ไม่ลบเครื่องมือ `message` ของเอเจนต์เมื่อมีเส้นทางแชตพร้อมใช้งาน

การเตือนความจำที่สร้างจากแชตที่ใช้งานอยู่จะรักษาเป้าหมายการจัดส่งแชตสดไว้สำหรับการจัดส่งประกาศ fallback คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็ก อย่าใช้เป็นแหล่งความจริงสำหรับรหัสผู้ให้บริการที่แยกแยะตัวพิมพ์เล็กใหญ่ เช่น รหัสห้อง Matrix

### การจัดส่งเมื่อเกิดข้อผิดพลาด

การแจ้งเตือนข้อผิดพลาด resolve ตามลำดับนี้:

1. `delivery.failureDestination` ในงาน
2. `cron.failureDestination` ส่วนกลาง
3. เป้าหมายประกาศหลักของงาน (เมื่อไม่ได้ตั้งค่าปลายทางข้อผิดพลาดไว้อย่างชัดเจน)

<Note>
งานเซสชันหลักสามารถใช้ `delivery.failureDestination` ได้เฉพาะเมื่อโหมดการจัดส่งหลักเป็น `webhook` งานแบบแยกรับค่านี้ได้ในทุกโหมด
</Note>

หมายเหตุ: การรัน Cron แบบแยกจะถือความล้มเหลวของเอเจนต์ในระดับการรันเป็นข้อผิดพลาดของงาน แม้เมื่อ
ไม่มี payload คำตอบถูกสร้างขึ้น ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการยังคงเพิ่มตัวนับข้อผิดพลาด
และทริกเกอร์การแจ้งเตือนข้อผิดพลาด

## การจัดตารางเวลา

### งานครั้งเดียว

`--at <datetime>` กำหนดเวลาการรันครั้งเดียว datetime ที่ไม่มี offset จะถูกถือเป็น UTC เว้นแต่คุณจะส่ง `--tz <iana>` ด้วย ซึ่งจะตีความเวลา wall-clock ในเขตเวลาที่กำหนด

<Note>
งานครั้งเดียวจะลบหลังสำเร็จตามค่าเริ่มต้น ใช้ `--keep-after-run` เพื่อเก็บไว้
</Note>

### งานที่เกิดซ้ำ

งานที่เกิดซ้ำใช้ retry backoff แบบเอ็กซ์โปเนนเชียลหลังเกิดข้อผิดพลาดต่อเนื่อง: 30s, 1m, 5m, 15m, 60m ตารางเวลาจะกลับสู่ปกติหลังการรันที่สำเร็จครั้งถัดไป

การรันที่ถูกข้ามจะถูกติดตามแยกจากข้อผิดพลาดในการดำเนินการ การรันเหล่านี้ไม่กระทบ retry backoff แต่ `openclaw cron edit <job-id> --failure-alert-include-skipped` สามารถเลือกให้การแจ้งเตือนข้อผิดพลาดรวมการแจ้งเตือนการรันที่ถูกข้ามซ้ำได้

สำหรับงานแบบแยกที่กำหนดเป้าหมายผู้ให้บริการโมเดลที่กำหนดค่าไว้ภายในเครื่อง Cron จะรัน preflight ผู้ให้บริการแบบเบาก่อนเริ่มเทิร์นของเอเจนต์ ผู้ให้บริการ `api: "ollama"` แบบ Loopback, เครือข่ายส่วนตัว และ `.local` จะถูก probe ที่ `/api/tags`; ผู้ให้บริการที่เข้ากันได้กับ OpenAI ภายในเครื่อง เช่น vLLM, SGLang และ LM Studio จะถูก probe ที่ `/models` หากเข้าถึง endpoint ไม่ได้ การรันจะถูกบันทึกเป็น `skipped` และลองใหม่ตามตารางเวลาภายหลัง; endpoint ที่ตายและตรงกันจะถูกแคชไว้ 5 นาทีเพื่อหลีกเลี่ยงไม่ให้หลายงาน hammer เซิร์ฟเวอร์ภายในเครื่องเดียวกัน

หมายเหตุ: นิยามงาน Cron อยู่ใน `jobs.json` ขณะที่สถานะ runtime ที่ค้างอยู่จะอยู่ใน `jobs-state.json` หากมีการแก้ไข `jobs.json` จากภายนอก Gateway จะโหลดตารางเวลาที่เปลี่ยนแล้วใหม่และล้างสล็อตที่ค้างและล้าสมัย; การเขียนใหม่ที่เปลี่ยนเฉพาะรูปแบบจะไม่ล้างสล็อตที่ค้างอยู่

### การรันด้วยตนเอง

`openclaw cron run` ส่งคืนทันทีที่การรันด้วยตนเองถูกเข้าคิว การตอบกลับที่สำเร็จจะมี `{ ok: true, enqueued: true, runId }` ใช้ `openclaw cron runs --id <job-id>` เพื่อติดตามผลลัพธ์ในท้ายที่สุด

<Note>
`openclaw cron run <job-id>` จะ force-run ตามค่าเริ่มต้น ใช้ `--due` เพื่อคงพฤติกรรมแบบเก่า "รันเฉพาะเมื่อถึงกำหนด" ไว้
</Note>

## โมเดล

`cron add|edit --model <ref>` เลือกโมเดลที่อนุญาตสำหรับงาน

<Warning>
หากโมเดลไม่ได้รับอนุญาตหรือไม่สามารถ resolve ได้ Cron จะทำให้การรันล้มเหลวด้วยข้อผิดพลาดการตรวจสอบที่ชัดเจน แทนที่จะ fallback ไปยังการเลือกโมเดลของเอเจนต์ของงานหรือโมเดลเริ่มต้น
</Warning>

Cron `--model` เป็น **ค่าหลักของงาน** ไม่ใช่การแทนที่ `/model` ของเซสชันแชต ซึ่งหมายความว่า:

- fallback ของโมเดลที่กำหนดค่าไว้ยังคงใช้เมื่อโมเดลของงานที่เลือกไว้ล้มเหลว
- payload รายงาน `fallbacks` จะแทนที่รายการ fallback ที่กำหนดค่าไว้เมื่อมีอยู่
- รายการ fallback รายงานที่ว่าง (`fallbacks: []` ใน payload/API ของงาน) ทำให้การรัน Cron เป็นแบบ strict
- เมื่องานมี `--model` แต่ไม่ได้กำหนดค่ารายการ fallback OpenClaw จะส่งการแทนที่ fallback ว่างอย่างชัดเจนเพื่อไม่ให้ค่าหลักของเอเจนต์ถูกเพิ่มเป็นเป้าหมายลองใหม่ที่ซ่อนอยู่

### ลำดับความสำคัญของโมเดล Cron แบบแยก

Cron แบบแยก resolve โมเดลที่ใช้งานอยู่ตามลำดับนี้:

1. การแทนที่จาก Gmail-hook
2. `--model` รายงาน
3. การแทนที่โมเดลของเซสชัน Cron ที่จัดเก็บไว้ (เมื่อผู้ใช้เลือกไว้)
4. การเลือกโมเดลของเอเจนต์หรือค่าเริ่มต้น

### โหมดเร็ว

โหมดเร็วของ Cron แบบแยกตามการเลือกโมเดลสดที่ resolve แล้ว การกำหนดค่าโมเดล `params.fastMode` ใช้ตามค่าเริ่มต้น แต่การแทนที่ `fastMode` ของเซสชันที่จัดเก็บไว้ยังคงชนะการกำหนดค่า

### การลองใหม่เมื่อสลับโมเดลสด

หากการรันแบบแยกโยน `LiveSessionModelSwitchError` Cron จะ persist ผู้ให้บริการและโมเดลที่สลับแล้ว (และการแทนที่โปรไฟล์การยืนยันตัวตนที่สลับแล้วเมื่อมี) สำหรับการรันที่ใช้งานอยู่ก่อนลองใหม่ วงลองใหม่ชั้นนอกถูกจำกัดไว้ที่การลองสลับสองครั้งหลังจากความพยายามเริ่มต้น จากนั้นจะยกเลิกแทนที่จะวนซ้ำตลอดไป

## เอาต์พุตการรันและการปฏิเสธ

### การระงับการตอบรับที่ล้าสมัย

เทิร์น Cron แบบแยกจะระงับคำตอบที่เป็นเพียงการตอบรับที่ล้าสมัย หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะชั่วคราวและไม่มีการรัน subagent ลูกหลานที่รับผิดชอบคำตอบในท้ายที่สุด Cron จะ prompt ใหม่หนึ่งครั้งเพื่อขอผลลัพธ์จริงก่อนจัดส่ง

### การระงับโทเค็นเงียบ

หากการรัน Cron แบบแยกส่งคืนเฉพาะโทเค็นเงียบ (`NO_REPLY` หรือ `no_reply`) Cron จะระงับทั้งการจัดส่งขาออกโดยตรงและเส้นทางสรุปที่เข้าคิวแบบ fallback ดังนั้นจะไม่มีอะไรถูกโพสต์กลับไปยังแชต

### การปฏิเสธแบบมีโครงสร้าง

การรัน Cron แบบแยกจะให้ความสำคัญกับ metadata การปฏิเสธการดำเนินการแบบมีโครงสร้างจากการรันที่ฝังอยู่ จากนั้น fallback ไปยัง marker การปฏิเสธที่รู้จักในเอาต์พุตสุดท้าย เช่น `SYSTEM_RUN_DENIED`, `INVALID_REQUEST` และวลีปฏิเสธการผูกการอนุมัติ

`cron list` และประวัติการรันจะแสดงเหตุผลการปฏิเสธ แทนที่จะรายงานคำสั่งที่ถูกบล็อกเป็น `ok`

## การเก็บรักษา

การเก็บรักษาและการตัดทอนถูกควบคุมใน config:

- `cron.sessionRetention` (ค่าเริ่มต้น `24h`) ตัดเซสชันการรันแบบแยกที่เสร็จสมบูรณ์แล้ว
- `cron.runLog.maxBytes` และ `cron.runLog.keepLines` ตัด `~/.openclaw/cron/runs/<jobId>.jsonl`

## การย้ายงานเก่า

<Note>
หากคุณมีงาน Cron จากก่อนรูปแบบการจัดส่งและ store ปัจจุบัน ให้เรียกใช้ `openclaw doctor --fix` Doctor จะปรับ field Cron แบบ legacy ให้เป็นมาตรฐาน (`jobId`, `schedule.cron`, field การจัดส่งระดับบนรวมถึง `threadId` แบบ legacy, alias การจัดส่ง `provider` ใน payload) และย้ายงาน fallback webhook แบบ `notify: true` ที่เรียบง่ายไปเป็นการจัดส่ง webhook อย่างชัดเจนเมื่อกำหนดค่า `cron.webhook` ไว้
</Note>

## การแก้ไขทั่วไป

อัปเดตการตั้งค่าการจัดส่งโดยไม่เปลี่ยนข้อความ:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

ปิดการจัดส่งสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --no-deliver
```

เปิดใช้บริบท bootstrap แบบเบาสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --light-context
```

ประกาศไปยังช่องเฉพาะ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

ประกาศไปยังหัวข้อฟอรัม Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

สร้างงานแบบแยกพร้อมบริบท bootstrap แบบเบา:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` ใช้กับงานเทิร์นเอเจนต์แบบแยกเท่านั้น สำหรับการรัน Cron โหมดเบาจะทำให้บริบท bootstrap ว่างแทนที่จะฉีดชุด bootstrap ของ workspace ทั้งหมด

```bash
openclaw cron list --agent ops
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

`openclaw cron list` แสดงงานที่ตรงทั้งหมดตามค่าเริ่มต้น ส่ง `--agent <id>` เพื่อแสดงเฉพาะงานที่รหัสเอเจนต์ normalized ที่มีผลตรงกัน; งานที่ไม่มีรหัสเอเจนต์ที่จัดเก็บไว้จะนับเป็นเอเจนต์เริ่มต้นที่กำหนดค่าไว้

รายการ `cron runs` มี diagnostics การจัดส่งพร้อมเป้าหมาย Cron ที่ตั้งใจไว้, เป้าหมายที่ resolve แล้ว, การส่งผ่าน message-tool, การใช้ fallback และสถานะการจัดส่ง

การกำหนดเป้าหมายเอเจนต์และเซสชันใหม่:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` เตือนเมื่อ `--agent` ถูกละไว้ในงานเทิร์นเอเจนต์และ fallback ไปยังเอเจนต์เริ่มต้น (`main`) ส่ง `--agent <id>` ตอนสร้างเพื่อปักหมุดเอเจนต์เฉพาะ

การปรับแต่งการจัดส่ง:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
