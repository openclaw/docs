# `openclaw cron`

จัดการงาน Cron สำหรับตัวจัดตารางเวลาของ Gateway

<Tip>
เรียกใช้ `openclaw cron --help` เพื่อดูพื้นผิวคำสั่งทั้งหมด ดู [งาน Cron](/th/automation/cron-jobs) สำหรับคู่มือแนวคิด
</Tip>

## เซสชัน

`--session` รับค่า `main`, `isolated`, `current` หรือ `session:<id>`

<AccordionGroup>
  <Accordion title="Session keys">
    - `main` ผูกกับเซสชันหลักของเอเจนต์
    - `isolated` สร้างทรานสคริปต์และรหัสเซสชันใหม่สำหรับการเรียกใช้แต่ละครั้ง
    - `current` ผูกกับเซสชันที่ใช้งานอยู่ ณ เวลาที่สร้าง
    - `session:<id>` ปักหมุดกับคีย์เซสชันแบบคงอยู่ที่ระบุอย่างชัดเจน

  </Accordion>
  <Accordion title="Isolated session semantics">
    การเรียกใช้แบบแยกจะรีเซ็ตบริบทการสนทนาแวดล้อม การกำหนดเส้นทางช่องทางและกลุ่ม นโยบายส่ง/เข้าคิว การยกระดับ ต้นทาง และการผูก ACP runtime จะถูกรีเซ็ตสำหรับการเรียกใช้ใหม่ ค่ากำหนดที่ปลอดภัยและการแทนที่โมเดลหรือการยืนยันตัวตนที่ผู้ใช้เลือกไว้อย่างชัดเจนสามารถส่งต่อข้ามการเรียกใช้ได้
  </Accordion>
</AccordionGroup>

## การส่งมอบ

`openclaw cron list` และ `openclaw cron show <job-id>` แสดงตัวอย่างเส้นทางการส่งมอบที่แก้ค่าแล้ว สำหรับ `channel: "last"` ตัวอย่างจะแสดงว่าเส้นทางแก้ค่าจากเซสชันหลักหรือเซสชันปัจจุบัน หรือจะล้มเหลวแบบปิด

<Note>
งาน `cron add` แบบแยกมีค่าเริ่มต้นเป็นการส่งมอบแบบ `--announce` ใช้ `--no-deliver` เพื่อเก็บเอาต์พุตไว้ภายใน `--deliver` ยังอยู่ในฐานะ alias ที่เลิกแนะนำแล้วของ `--announce`
</Note>

### ความเป็นเจ้าของการส่งมอบ

การส่งมอบแชต Cron แบบแยกใช้ร่วมกันระหว่างเอเจนต์และรันเนอร์:

- เอเจนต์สามารถส่งโดยตรงโดยใช้เครื่องมือ `message` เมื่อมีเส้นทางแชตพร้อมใช้งาน
- `announce` ส่งมอบคำตอบสุดท้ายแบบสำรองเฉพาะเมื่อเอเจนต์ไม่ได้ส่งโดยตรงไปยังเป้าหมายที่แก้ค่าแล้ว
- `webhook` โพสต์ payload ที่เสร็จแล้วไปยัง URL
- `none` ปิดการส่งมอบสำรองของรันเนอร์

`--announce` คือการส่งมอบสำรองของรันเนอร์สำหรับคำตอบสุดท้าย `--no-deliver` ปิดการสำรองนั้น แต่ไม่ได้ลบเครื่องมือ `message` ของเอเจนต์เมื่อมีเส้นทางแชตพร้อมใช้งาน

การเตือนความจำที่สร้างจากแชตที่ใช้งานอยู่จะเก็บเป้าหมายการส่งมอบแชตสดไว้สำหรับการส่งมอบ announce สำรอง คีย์เซสชันภายในอาจเป็นตัวพิมพ์เล็กทั้งหมด อย่าใช้คีย์เหล่านี้เป็นแหล่งความจริงสำหรับ ID ผู้ให้บริการที่ไวต่ออักษรตัวพิมพ์ใหญ่เล็ก เช่น ID ห้อง Matrix

### การส่งมอบเมื่อเกิดความล้มเหลว

การแจ้งเตือนความล้มเหลวจะแก้ค่าตามลำดับนี้:

1. `delivery.failureDestination` บนงาน
2. `cron.failureDestination` ส่วนกลาง
3. เป้าหมาย announce หลักของงาน เมื่อไม่ได้ตั้งค่าปลายทางความล้มเหลวอย่างชัดเจน

<Note>
งานเซสชันหลักสามารถใช้ `delivery.failureDestination` ได้เฉพาะเมื่อโหมดการส่งมอบหลักเป็น `webhook` งานแบบแยกรับค่านี้ได้ในทุกโหมด
</Note>

หมายเหตุ: การเรียกใช้ Cron แบบแยกถือว่าความล้มเหลวของเอเจนต์ระดับการเรียกใช้เป็นข้อผิดพลาดของงาน แม้เมื่อไม่มี payload คำตอบถูกสร้างขึ้น ดังนั้นความล้มเหลวของโมเดล/ผู้ให้บริการยังคงเพิ่มตัวนับข้อผิดพลาดและทริกเกอร์การแจ้งเตือนความล้มเหลว

## การจัดตารางเวลา

### งานแบบครั้งเดียว

`--at <datetime>` จัดตารางการเรียกใช้แบบครั้งเดียว วันที่และเวลาที่ไม่มี offset จะถือเป็น UTC เว้นแต่คุณจะส่ง `--tz <iana>` ด้วย ซึ่งจะแปลความหมายเวลา wall-clock ในเขตเวลาที่กำหนด

<Note>
งานแบบครั้งเดียวจะถูกลบหลังจากสำเร็จโดยค่าเริ่มต้น ใช้ `--keep-after-run` เพื่อเก็บงานไว้
</Note>

### งานที่เกิดซ้ำ

งานที่เกิดซ้ำใช้การหน่วง retry แบบ exponential หลังจากเกิดข้อผิดพลาดต่อเนื่อง: 30s, 1m, 5m, 15m, 60m ตารางเวลาจะกลับสู่ปกติหลังจากการเรียกใช้ครั้งถัดไปสำเร็จ

การเรียกใช้ที่ถูกข้ามจะถูกติดตามแยกจากข้อผิดพลาดในการดำเนินการ การข้ามเหล่านี้ไม่ส่งผลต่อการหน่วง retry แต่ `openclaw cron edit <job-id> --failure-alert-include-skipped` สามารถเลือกให้การแจ้งเตือนความล้มเหลวรวมการแจ้งเตือนการเรียกใช้ที่ถูกข้ามซ้ำได้

สำหรับงานแบบแยกที่กำหนดเป้าหมายผู้ให้บริการโมเดลที่กำหนดค่าไว้ภายในเครื่อง Cron จะเรียก preflight ผู้ให้บริการแบบเบาก่อนเริ่มเทิร์นของเอเจนต์ ผู้ให้บริการ `api: "ollama"` แบบ loopback เครือข่ายส่วนตัว และ `.local` จะถูก probe ที่ `/api/tags`; ผู้ให้บริการที่เข้ากันได้กับ OpenAI ภายในเครื่อง เช่น vLLM, SGLang และ LM Studio จะถูก probe ที่ `/models` หากเข้าถึง endpoint ไม่ได้ การเรียกใช้จะถูกบันทึกเป็น `skipped` และ retry ในตารางเวลาภายหลัง endpoint ที่ตายและตรงกันจะถูกแคชไว้ 5 นาทีเพื่อหลีกเลี่ยงงานจำนวนมากถล่มเซิร์ฟเวอร์ภายในเครื่องเดียวกัน

หมายเหตุ: นิยามงาน Cron อยู่ใน `jobs.json` ขณะที่สถานะ runtime ที่รอดำเนินการอยู่ใน `jobs-state.json` หากมีการแก้ไข `jobs.json` จากภายนอก Gateway จะโหลดตารางเวลาที่เปลี่ยนใหม่และล้างสล็อตที่รอดำเนินการซึ่งค้างอยู่ การเขียนใหม่ที่เปลี่ยนเฉพาะการจัดรูปแบบจะไม่ล้างสล็อตที่รอดำเนินการ

### การเรียกใช้ด้วยตนเอง

`openclaw cron run` คืนค่าทันทีเมื่อการเรียกใช้ด้วยตนเองถูกเข้าคิวแล้ว การตอบกลับที่สำเร็จมี `{ ok: true, enqueued: true, runId }` ใช้ `openclaw cron runs --id <job-id>` เพื่อติดตามผลลัพธ์สุดท้าย

<Note>
`openclaw cron run <job-id>` บังคับเรียกใช้โดยค่าเริ่มต้น ใช้ `--due` เพื่อคงพฤติกรรมเดิมแบบ "เรียกใช้เฉพาะเมื่อถึงกำหนด"
</Note>

## โมเดล

`cron add|edit --model <ref>` เลือกโมเดลที่อนุญาตสำหรับงาน

<Warning>
หากโมเดลไม่ได้รับอนุญาตหรือแก้ค่าไม่ได้ Cron จะทำให้การเรียกใช้ล้มเหลวพร้อมข้อผิดพลาดการตรวจสอบความถูกต้องที่ชัดเจน แทนที่จะ fallback ไปใช้เอเจนต์ของงานหรือการเลือกโมเดลเริ่มต้น
</Warning>

Cron `--model` เป็น **โมเดลหลักของงาน** ไม่ใช่การแทนที่ `/model` ของเซสชันแชต นั่นหมายความว่า:

- fallback ของโมเดลที่กำหนดค่าไว้ยังคงมีผลเมื่อโมเดลของงานที่เลือกไว้ล้มเหลว
- payload ต่อหนึ่งงาน `fallbacks` จะแทนที่รายการ fallback ที่กำหนดค่าไว้เมื่อมีอยู่
- รายการ fallback ต่อหนึ่งงานที่ว่างเปล่า (`fallbacks: []` ใน payload/API ของงาน) ทำให้การเรียกใช้ Cron เป็นแบบเข้มงวด
- เมื่องานมี `--model` แต่ไม่ได้กำหนดค่ารายการ fallback OpenClaw จะส่งการแทนที่ fallback ว่างอย่างชัดเจนเพื่อไม่ให้โมเดลหลักของเอเจนต์ถูกต่อท้ายเป็นเป้าหมาย retry ที่ซ่อนอยู่

### ลำดับความสำคัญของโมเดล Cron แบบแยก

Cron แบบแยกจะแก้ค่าโมเดลที่ใช้งานอยู่ตามลำดับนี้:

1. การแทนที่ของ Gmail-hook
2. `--model` ต่อหนึ่งงาน
3. การแทนที่โมเดลของเซสชัน Cron ที่จัดเก็บไว้ เมื่อผู้ใช้เลือกไว้
4. การเลือกโมเดลของเอเจนต์หรือค่าเริ่มต้น

### โหมดเร็ว

โหมดเร็วของ Cron แบบแยกจะตามการเลือกโมเดลสดที่แก้ค่าแล้ว การกำหนดค่าโมเดล `params.fastMode` มีผลโดยค่าเริ่มต้น แต่การแทนที่ `fastMode` ของเซสชันที่จัดเก็บไว้ยังคงชนะการกำหนดค่า

### การ retry เมื่อสลับโมเดลสด

หากการเรียกใช้แบบแยกโยน `LiveSessionModelSwitchError` Cron จะคงผู้ให้บริการและโมเดลที่สลับแล้วไว้ รวมถึงการแทนที่โปรไฟล์การยืนยันตัวตนที่สลับแล้วเมื่อมี สำหรับการเรียกใช้ที่ใช้งานอยู่ก่อน retry ลูป retry ภายนอกถูกจำกัดไว้ที่การ retry จากการสลับสองครั้งหลังจากความพยายามครั้งแรก จากนั้นจะยกเลิกแทนที่จะวนซ้ำตลอดไป

## เอาต์พุตการเรียกใช้และการปฏิเสธ

### การระงับ acknowledgement ที่ค้างเก่า

เทิร์น Cron แบบแยกจะระงับคำตอบที่เป็นเพียง acknowledgement ที่ค้างเก่า หากผลลัพธ์แรกเป็นเพียงการอัปเดตสถานะชั่วคราว และไม่มีการเรียกใช้ subagent ลูกหลานที่รับผิดชอบคำตอบสุดท้าย Cron จะ prompt ซ้ำหนึ่งครั้งเพื่อขอผลลัพธ์จริงก่อนส่งมอบ

### การระงับ token เงียบ

หากการเรียกใช้ Cron แบบแยกคืนค่าเฉพาะ token เงียบ (`NO_REPLY` หรือ `no_reply`) Cron จะระงับทั้งการส่งมอบขาออกโดยตรงและเส้นทางสรุปที่เข้าคิวแบบสำรอง ดังนั้นจะไม่มีสิ่งใดถูกโพสต์กลับไปยังแชต

### การปฏิเสธแบบมีโครงสร้าง

การเรียกใช้ Cron แบบแยกจะเลือกใช้ metadata การปฏิเสธการดำเนินการแบบมีโครงสร้างจากการเรียกใช้ที่ฝังอยู่ก่อน แล้วจึง fallback ไปยัง marker การปฏิเสธที่รู้จักในเอาต์พุตสุดท้าย เช่น `SYSTEM_RUN_DENIED`, `INVALID_REQUEST` และวลีปฏิเสธการผูกกับการอนุมัติ

`cron list` และประวัติการเรียกใช้จะแสดงเหตุผลการปฏิเสธแทนที่จะรายงานคำสั่งที่ถูกบล็อกเป็น `ok`

## การเก็บรักษา

การเก็บรักษาและการ prune ควบคุมได้ในการกำหนดค่า:

- `cron.sessionRetention` ค่าเริ่มต้น `24h` prune เซสชันการเรียกใช้แบบแยกที่เสร็จสมบูรณ์แล้ว
- `cron.runLog.maxBytes` และ `cron.runLog.keepLines` prune `~/.openclaw/cron/runs/<jobId>.jsonl`

## การย้ายงานเก่า

<Note>
หากคุณมีงาน Cron จากก่อนรูปแบบการส่งมอบและ store ปัจจุบัน ให้เรียกใช้ `openclaw doctor --fix` Doctor จะปรับฟิลด์ Cron ดั้งเดิม (`jobId`, `schedule.cron`, ฟิลด์การส่งมอบระดับบนรวมถึง `threadId` ดั้งเดิม และ alias การส่งมอบ `provider` ใน payload) ให้เป็นรูปแบบปกติ และย้ายงาน webhook fallback แบบง่ายที่เป็น `notify: true` ไปเป็นการส่งมอบ webhook อย่างชัดเจนเมื่อกำหนดค่า `cron.webhook` แล้ว
</Note>

## การแก้ไขทั่วไป

อัปเดตการตั้งค่าการส่งมอบโดยไม่เปลี่ยนข้อความ:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

ปิดการส่งมอบสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --no-deliver
```

เปิดใช้บริบท bootstrap แบบเบาสำหรับงานแบบแยก:

```bash
openclaw cron edit <job-id> --light-context
```

Announce ไปยังช่องทางที่ระบุ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

Announce ไปยังหัวข้อฟอรัม Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

สร้างงานแบบแยกพร้อมบริบท bootstrap แบบเบา:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` มีผลเฉพาะกับงานเทิร์นเอเจนต์แบบแยกเท่านั้น สำหรับการเรียกใช้ Cron โหมดเบาจะคงบริบท bootstrap ให้ว่างเปล่าแทนการฉีดชุด bootstrap ของ workspace ทั้งหมด

## คำสั่งผู้ดูแลทั่วไป

การเรียกใช้ด้วยตนเองและการตรวจสอบ:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

รายการ `cron runs` มีการวินิจฉัยการส่งมอบพร้อมเป้าหมาย Cron ที่ตั้งใจไว้ เป้าหมายที่แก้ค่าแล้ว การส่งด้วยเครื่องมือข้อความ การใช้ fallback และสถานะที่ส่งมอบแล้ว

การเปลี่ยนเป้าหมายเอเจนต์และเซสชัน:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` เตือนเมื่อไม่ได้ระบุ `--agent` บนงานเทิร์นเอเจนต์ และ fallback ไปยังเอเจนต์เริ่มต้น (`main`) ส่ง `--agent <id>` ตอนสร้างเพื่อปักหมุดเอเจนต์ที่ระบุ

การปรับแต่งการส่งมอบ:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
