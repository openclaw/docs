# `openclaw daemon`

นามแฝงเดิมสำหรับการจัดการบริการ Gateway โดย `openclaw daemon ...` จะเชื่อมโยงไปยังคำสั่งควบคุมบริการชุดเดียวกับ `openclaw gateway ...` สำหรับเอกสารและตัวอย่างปัจจุบัน โปรดใช้ [`openclaw gateway`](/th/cli/gateway)

## การใช้งาน

```bash
openclaw daemon status
openclaw daemon install
openclaw daemon start
openclaw daemon stop
openclaw daemon restart
openclaw daemon uninstall
```

## คำสั่งย่อยและตัวเลือก

| คำสั่งย่อย  | ตัวเลือก                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------ |
| `status`    | `--url`, `--token`, `--password`, `--timeout`, `--no-probe`, `--require-rpc`, `--deep`, `--json` |
| `install`   | `--port`, `--runtime <node>`, `--token`, `--wrapper <path>`, `--force`, `--json`                 |
| `uninstall` | `--json`                                                                                         |
| `start`     | `--json`                                                                                         |
| `stop`      | `--json`, `--disable` (เฉพาะ launchd: ระงับ KeepAlive/RunAtLoad อย่างถาวรจนกว่าจะเริ่มทำงานครั้งถัดไป) |
| `restart`   | `--force`, `--safe`, `--skip-deferral`, `--wait <duration>`, `--json`                            |

- `status`: แสดงสถานะการติดตั้งบริการ (launchd/systemd/schtasks) และตรวจสอบสถานะการทำงานของ Gateway
- `install`: ติดตั้งบริการ โดย `--force` จะติดตั้งซ้ำ/เขียนทับการติดตั้งที่มีอยู่
- `restart --safe`: ขอให้ Gateway ที่กำลังทำงานตรวจสอบงานที่กำลังดำเนินอยู่ล่วงหน้า และกำหนดเวลาการรีสตาร์ตแบบรวมหนึ่งครั้งหลังจากงานเสร็จสิ้น โดยจำกัดด้วย `gateway.reload.deferralTimeoutMs` (ค่าเริ่มต้น 300000ms/5 นาที; ตั้งเป็น `0` เพื่อรอโดยไม่มีกำหนด) เมื่อระยะเวลานี้หมดลง ระบบจะบังคับรีสตาร์ตไม่ว่าอย่างไรก็ตาม ส่วน `restart` แบบปกติจะใช้ตัวจัดการบริการโดยตรง และ `--force` เป็นตัวเลือกแทนที่ให้ดำเนินการทันที
- `restart --safe --skip-deferral`: ข้ามกลไกเลื่อนการทำงานเนื่องจากมีงานที่กำลังดำเนินอยู่ เพื่อให้ Gateway รีสตาร์ตทันทีแม้มีการรายงานตัวขัดขวาง ต้องใช้ร่วมกับ `--safe`

## หมายเหตุ

- `status` จะแก้ไข SecretRefs สำหรับการยืนยันตัวตนที่กำหนดค่าไว้ เพื่อนำมาใช้ยืนยันตัวตนในการตรวจสอบเมื่อทำได้ หาก SecretRef ที่จำเป็นไม่ได้รับการแก้ไข `status --json` จะรายงาน `rpc.authWarning`; ให้ระบุ `--token`/`--password` อย่างชัดเจน หรือแก้ไขแหล่งที่มาของข้อมูลลับก่อน คำเตือนเกี่ยวกับการยืนยันตัวตนที่ไม่ได้รับการแก้ไขจะถูกระงับเมื่อการตรวจสอบส่วนอื่นสำเร็จ
- `status --deep` เพิ่มการสแกนระดับระบบแบบพยายามอย่างเต็มที่เพื่อค้นหาบริการอื่นที่คล้าย Gateway (แสดงคำแนะนำในการล้างข้อมูล โดยยังคงแนะนำให้มี Gateway หนึ่งรายการต่อเครื่อง) และตรวจสอบความถูกต้องของการกำหนดค่าในโหมดที่รับรู้ Plugin พร้อมแสดงคำเตือนจากไฟล์ manifest ของ Plugin ซึ่งเส้นทางเริ่มต้นแบบรวดเร็วจะข้ามไป
- สำหรับการติดตั้ง systemd บน Linux การตรวจสอบความคลาดเคลื่อนของโทเค็นจะตรวจสอบแหล่งที่มาของยูนิตทั้ง `Environment=` และ `EnvironmentFile=`
- การตรวจสอบความคลาดเคลื่อนของโทเค็นจะแก้ไข SecretRefs ของ `gateway.auth.token` โดยใช้สภาพแวดล้อมรันไทม์ที่ผสานแล้ว (สภาพแวดล้อมคำสั่งของบริการก่อน จากนั้นจึงเป็นสภาพแวดล้อมของโพรเซส) หากการยืนยันตัวตนด้วยโทเค็นไม่ได้เปิดใช้งานอย่างมีผล (`gateway.auth.mode` เป็น `password`/`none`/`trusted-proxy` หรือไม่ได้ตั้งค่าโดยที่รหัสผ่านสามารถมีผลเหนือกว่าได้) ระบบจะข้ามการแก้ไขโทเค็นจากการกำหนดค่า
- `install` จะตรวจสอบว่า `gateway.auth.token` ที่จัดการด้วย SecretRef สามารถแก้ไขได้ แต่จะไม่บันทึกค่าที่แก้ไขแล้วลงในข้อมูลเมตาสภาพแวดล้อมของบริการ หากไม่สามารถแก้ไขได้ การติดตั้งจะล้มเหลวแบบปิดกั้น
- หากกำหนดค่าทั้ง `gateway.auth.token` และ `gateway.auth.password` แต่ไม่ได้ตั้งค่า `gateway.auth.mode` ไว้ `install` จะบล็อกจนกว่าจะตั้งค่าโหมดอย่างชัดเจน
- บน macOS `install` จะจำกัดสิทธิ์ไฟล์ plist ของ LaunchAgent รวมถึงไฟล์สภาพแวดล้อม/ตัวห่อหุ้มที่สร้างขึ้นให้เฉพาะเจ้าของเท่านั้น (โหมด `0600`/`0700`) แทนการฝังข้อมูลลับไว้ใน `EnvironmentVariables`
- การเรียกใช้ Gateway หลายรายการบนโฮสต์เดียว: แยกพอร์ต การกำหนดค่า/สถานะ และพื้นที่ทำงานออกจากกัน ดู [Gateway หลายรายการ](/th/gateway#multiple-gateways-same-host)
