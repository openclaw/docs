# `openclaw daemon`

นามแฝงแบบเดิมสำหรับการจัดการบริการ Gateway คำสั่ง `openclaw daemon ...` จะเชื่อมโยงไปยังคำสั่งควบคุมบริการเดียวกับ `openclaw gateway ...` สำหรับเอกสารและตัวอย่างปัจจุบัน ควรใช้ [`openclaw gateway`](/th/cli/gateway)

## การใช้งาน

```bash
openclaw daemon status
openclaw daemon install
openclaw daemon start
openclaw daemon stop
openclaw daemon restart
openclaw daemon uninstall
```

## คำสั่งย่อยและตัวเลือก

| คำสั่งย่อย  | ตัวเลือก                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------ |
| `status`    | `--url`, `--token`, `--password`, `--timeout`, `--no-probe`, `--require-rpc`, `--deep`, `--json` |
| `install`   | `--port`, `--runtime <node\|bun>`, `--token`, `--wrapper <path>`, `--force`, `--json`            |
| `uninstall` | `--json`                                                                                         |
| `start`     | `--json`                                                                                         |
| `stop`      | `--json`, `--disable` (เฉพาะ launchd: ระงับ KeepAlive/RunAtLoad อย่างถาวรจนกว่าจะเริ่มทำงานครั้งถัดไป) |
| `restart`   | `--force`, `--safe`, `--skip-deferral`, `--wait <duration>`, `--json`                            |

- `status`: แสดงสถานะการติดตั้งบริการ (launchd/systemd/schtasks) และตรวจสอบสถานะการทำงานของ Gateway
- `install`: ติดตั้งบริการ โดย `--force` จะติดตั้งซ้ำหรือเขียนทับการติดตั้งที่มีอยู่
- `restart --safe`: ขอให้ Gateway ที่กำลังทำงานตรวจสอบงานที่ใช้งานอยู่ล่วงหน้า และกำหนดเวลาการเริ่มใหม่แบบรวมเป็นครั้งเดียวหลังจากงานทั้งหมดเสร็จสิ้น โดยจำกัดเวลาตาม `gateway.reload.deferralTimeoutMs` (ค่าเริ่มต้น 300000 มิลลิวินาที/5 นาที ตั้งเป็น `0` เพื่อรอโดยไม่มีกำหนด) เมื่อหมดเวลาที่กำหนด ระบบจะบังคับเริ่มใหม่ไม่ว่าอย่างไรก็ตาม ส่วน `restart` ธรรมดาจะใช้ตัวจัดการบริการโดยตรง และ `--force` ใช้สำหรับบังคับให้ดำเนินการทันที
- `restart --safe --skip-deferral`: ข้ามด่านการเลื่อนเวลาสำหรับงานที่กำลังดำเนินอยู่ เพื่อให้ Gateway เริ่มใหม่ทันทีแม้จะมีการรายงานสิ่งกีดขวาง ต้องใช้ร่วมกับ `--safe`

## หมายเหตุ

- `status` จะแก้ไข SecretRef สำหรับการยืนยันตัวตนที่กำหนดค่าไว้ เพื่อใช้ยืนยันตัวตนระหว่างการตรวจสอบเมื่อทำได้ หากไม่สามารถแก้ไข SecretRef ที่จำเป็นได้ `status --json` จะรายงาน `rpc.authWarning` ให้ส่ง `--token`/`--password` อย่างชัดเจน หรือแก้ไขแหล่งข้อมูลลับก่อน คำเตือนการยืนยันตัวตนที่ยังแก้ไขไม่ได้จะถูกระงับเมื่อการตรวจสอบสำเร็จในด้านอื่นแล้ว
- `status --deep` เพิ่มการสแกนระดับระบบแบบพยายามให้ดีที่สุดเพื่อค้นหาบริการอื่นที่คล้าย Gateway (แสดงคำแนะนำในการล้างข้อมูล โดยยังคงแนะนำให้มี Gateway หนึ่งรายการต่อเครื่อง) และเรียกใช้การตรวจสอบความถูกต้องของการกำหนดค่าในโหมดที่รับรู้ Plugin เพื่อแสดงคำเตือนจากไฟล์ manifest ของ Plugin ซึ่งเส้นทางเริ่มต้นแบบรวดเร็วจะข้ามไป
- สำหรับการติดตั้ง systemd บน Linux การตรวจสอบความคลาดเคลื่อนของโทเค็นจะตรวจสอบทั้งแหล่งข้อมูลยูนิต `Environment=` และ `EnvironmentFile=`
- การตรวจสอบความคลาดเคลื่อนของโทเค็นจะแก้ไข SecretRef ของ `gateway.auth.token` โดยใช้สภาพแวดล้อมรันไทม์ที่ผสานแล้ว (สภาพแวดล้อมของคำสั่งบริการก่อน ตามด้วยสภาพแวดล้อมของโปรเซส) หากการยืนยันตัวตนด้วยโทเค็นไม่ได้เปิดใช้งานจริง (`gateway.auth.mode` เป็น `password`/`none`/`trusted-proxy` หรือไม่ได้ตั้งค่าและรหัสผ่านสามารถมีลำดับความสำคัญเหนือกว่า) ระบบจะข้ามการแก้ไขโทเค็นจากการกำหนดค่า
- `install` จะตรวจสอบว่าสามารถแก้ไข `gateway.auth.token` ที่จัดการด้วย SecretRef ได้ แต่จะไม่บันทึกค่าที่แก้ไขแล้วลงในข้อมูลเมตาสภาพแวดล้อมของบริการ หากแก้ไขไม่ได้ การติดตั้งจะล้มเหลวแบบปิดกั้น
- หากกำหนดค่าทั้ง `gateway.auth.token` และ `gateway.auth.password` แต่ไม่ได้ตั้งค่า `gateway.auth.mode` คำสั่ง `install` จะถูกระงับจนกว่าคุณจะตั้งค่าโหมดอย่างชัดเจน
- บน macOS คำสั่ง `install` จะจำกัดให้เฉพาะเจ้าของเข้าถึง plist ของ LaunchAgent และไฟล์สภาพแวดล้อม/ตัวห่อหุ้มที่สร้างขึ้น (โหมด `0600`/`0700`) แทนการฝังข้อมูลลับไว้ใน `EnvironmentVariables`
- การเรียกใช้ Gateway หลายรายการบนโฮสต์เดียวกัน: แยกพอร์ต การกำหนดค่า/สถานะ และพื้นที่ทำงานออกจากกัน ดู [Gateway หลายรายการ](/th/gateway#multiple-gateways-same-host)
