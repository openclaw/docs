# `openclaw daemon`

นามแฝงเดิมสำหรับการจัดการบริการ Gateway โดย `openclaw daemon ...` จะแมปไปยังคำสั่งควบคุมบริการชุดเดียวกับ `openclaw gateway ...` โปรดใช้ [`openclaw gateway`](/th/cli/gateway) สำหรับเอกสารและตัวอย่างปัจจุบัน

## การใช้งาน

```bash
openclaw daemon status
openclaw daemon install
openclaw daemon start
openclaw daemon stop
openclaw daemon restart
openclaw daemon uninstall
```

## คำสั่งย่อยและตัวเลือก

| คำสั่งย่อย  | ตัวเลือก                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------ |
| `status`    | `--url`, `--token`, `--password`, `--timeout`, `--no-probe`, `--require-rpc`, `--deep`, `--json` |
| `install`   | `--port`, `--runtime <node>`, `--token`, `--wrapper <path>`, `--force`, `--json`                 |
| `uninstall` | `--json`                                                                                         |
| `start`     | `--json`                                                                                         |
| `stop`      | `--json`, `--disable` (เฉพาะ launchd: ระงับ KeepAlive/RunAtLoad อย่างถาวรจนกว่าจะเริ่มทำงานครั้งถัดไป) |
| `restart`   | `--force`, `--safe`, `--skip-deferral`, `--wait <duration>`, `--json`                            |

- `status`: แสดงสถานะการติดตั้งบริการ (launchd/systemd/schtasks) และตรวจสอบสถานะการทำงานของ Gateway
- `install`: ติดตั้งบริการ ส่วน `--force` จะติดตั้งซ้ำ/เขียนทับการติดตั้งที่มีอยู่
- `restart --safe`: ขอให้ Gateway ที่กำลังทำงานตรวจสอบงานที่ยังดำเนินอยู่ล่วงหน้า และกำหนดเวลารีสตาร์ตแบบรวมเป็นครั้งเดียวหลังจากงานทั้งหมดเสร็จสิ้น โดยจำกัดเวลาไว้ที่ 5 นาที เมื่อครบกำหนดเวลาดังกล่าว ระบบจะบังคับรีสตาร์ตไม่ว่าอย่างไรก็ตาม ส่วน `restart` แบบปกติจะใช้ตัวจัดการบริการโดยตรง และ `--force` ใช้สำหรับสั่งให้ดำเนินการทันทีแทน
- `restart --safe --skip-deferral`: ข้ามเงื่อนไขการเลื่อนเวลาเนื่องจากมีงานที่ยังดำเนินอยู่ เพื่อให้ Gateway รีสตาร์ตทันทีแม้มีรายงานตัวขัดขวาง ต้องใช้ร่วมกับ `--safe`

## หมายเหตุ

- `status` จะพยายามแก้ค่า SecretRefs สำหรับการยืนยันตัวตนที่กำหนดค่าไว้ เพื่อใช้ในการยืนยันตัวตนของการตรวจสอบ หากไม่สามารถแก้ค่า SecretRef ที่จำเป็นได้ `status --json` จะรายงาน `rpc.authWarning` ให้ส่ง `--token`/`--password` อย่างชัดเจน หรือแก้ไขแหล่งที่มาของข้อมูลลับก่อน คำเตือนเกี่ยวกับการยืนยันตัวตนที่ยังแก้ค่าไม่ได้จะถูกระงับเมื่อการตรวจสอบส่วนอื่นสำเร็จ
- `status --deep` เพิ่มการสแกนระดับระบบแบบพยายามให้ดีที่สุด เพื่อค้นหาบริการอื่นที่มีลักษณะคล้าย Gateway (แสดงคำแนะนำในการล้างข้อมูล โดยยังคงแนะนำให้มี Gateway หนึ่งรายการต่อเครื่อง) และเรียกใช้การตรวจสอบความถูกต้องของการกำหนดค่าในโหมดที่รับรู้ Plugin ซึ่งจะแสดงคำเตือนจากไฟล์ manifest ของ Plugin ที่เส้นทางเริ่มต้นแบบรวดเร็วข้ามไป
- สำหรับการติดตั้งด้วย systemd บน Linux การตรวจสอบความคลาดเคลื่อนของโทเค็นจะตรวจสอบแหล่งที่มาของยูนิตทั้ง `Environment=` และ `EnvironmentFile=`
- การตรวจสอบความคลาดเคลื่อนของโทเค็นจะแก้ค่า SecretRefs ของ `gateway.auth.token` โดยใช้สภาพแวดล้อมรันไทม์ที่ผสานรวมกัน (สภาพแวดล้อมของคำสั่งบริการก่อน แล้วตามด้วยสภาพแวดล้อมของกระบวนการ) หากการยืนยันตัวตนด้วยโทเค็นไม่ได้เปิดใช้งานจริง (`gateway.auth.mode` เป็น `password`/`none`/`trusted-proxy` หรือไม่ได้ตั้งค่าไว้และรหัสผ่านสามารถมีลำดับความสำคัญเหนือกว่าได้) ระบบจะข้ามการแก้ค่าโทเค็นในการกำหนดค่า
- `install` ตรวจสอบว่าสามารถแก้ค่า `gateway.auth.token` ที่จัดการด้วย SecretRef ได้ แต่จะไม่บันทึกค่าที่แก้แล้วลงในข้อมูลเมตาของสภาพแวดล้อมบริการ หากไม่สามารถแก้ค่าได้ การติดตั้งจะล้มเหลวแบบปิดกั้น
- หากกำหนดค่าทั้ง `gateway.auth.token` และ `gateway.auth.password` แต่ไม่ได้ตั้งค่า `gateway.auth.mode` ไว้ `install` จะบล็อกจนกว่าจะตั้งค่าโหมดอย่างชัดเจน
- บน macOS `install` จะจำกัดให้เฉพาะเจ้าของเข้าถึงไฟล์ plist ของ LaunchAgent รวมถึงไฟล์สภาพแวดล้อม/แรปเปอร์ที่สร้างขึ้น (โหมด `0600`/`0700`) แทนการฝังข้อมูลลับไว้ใน `EnvironmentVariables`
- การเรียกใช้ Gateway หลายรายการบนโฮสต์เดียว: แยกพอร์ต การกำหนดค่า/สถานะ และพื้นที่ทำงานออกจากกัน ดู [Gateway หลายรายการ](/th/gateway#multiple-gateways-same-host)
