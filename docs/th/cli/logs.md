# `openclaw logs`

ติดตามบันทึกไฟล์ของ Gateway ผ่าน RPC (ทำงานในโหมดรีโมต)

ที่เกี่ยวข้อง:

- ภาพรวมการบันทึก: [การบันทึก](/th/logging)
- CLI ของ Gateway: [gateway](/th/cli/gateway)

## ตัวเลือก

- `--limit <n>`: จำนวนบรรทัดบันทึกสูงสุดที่จะส่งคืน (ค่าเริ่มต้น `200`)
- `--max-bytes <n>`: จำนวนไบต์สูงสุดที่จะอ่านจากไฟล์บันทึก (ค่าเริ่มต้น `250000`)
- `--follow`: ติดตามสตรีมบันทึก
- `--interval <ms>`: ช่วงเวลาการโพลระหว่างการติดตาม (ค่าเริ่มต้น `1000`)
- `--json`: ส่งออกเหตุการณ์ JSON แบบหนึ่งรายการต่อบรรทัด
- `--plain`: เอาต์พุตข้อความธรรมดาโดยไม่มีการจัดรูปแบบสไตล์
- `--no-color`: ปิดใช้สี ANSI
- `--local-time`: แสดงเวลาประทับในเขตเวลาท้องถิ่นของคุณ (ค่าเริ่มต้น)
- `--utc`: แสดงเวลาประทับเป็น UTC

## ตัวเลือก RPC ของ Gateway ที่ใช้ร่วมกัน

`openclaw logs` ยังรับแฟล็กไคลเอนต์ Gateway มาตรฐานด้วย:

- `--url <url>`: URL WebSocket ของ Gateway
- `--token <token>`: โทเค็น Gateway
- `--timeout <ms>`: ระยะหมดเวลาเป็นมิลลิวินาที (ค่าเริ่มต้น `30000`)
- `--expect-final`: รอการตอบกลับสุดท้ายเมื่อการเรียก Gateway มีเอเจนต์รองรับ

เมื่อคุณส่ง `--url` CLI จะไม่นำการกำหนดค่าหรือข้อมูลรับรองจากสภาพแวดล้อมมาใช้โดยอัตโนมัติ ระบุ `--token` อย่างชัดเจนหาก Gateway ปลายทางต้องใช้การยืนยันตัวตน

## ตัวอย่าง

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --limit 500
openclaw logs --local-time
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## หมายเหตุ

- โดยค่าเริ่มต้น เวลาประทับจะแสดงในเขตเวลาท้องถิ่นของคุณ ใช้ `--utc` สำหรับเอาต์พุต UTC
- หาก Gateway แบบ local loopback โดยปริยายขอให้จับคู่ ปิดระหว่างเชื่อมต่อ หรือหมดเวลาก่อนที่ `logs.tail` จะตอบ `openclaw logs` จะถอยกลับไปใช้บันทึกไฟล์ของ Gateway ที่กำหนดค่าไว้โดยอัตโนมัติ เป้าหมาย `--url` ที่ระบุอย่างชัดเจนจะไม่ใช้การถอยกลับนี้
- `openclaw logs --follow` จะไม่ติดตามการถอยกลับไปยังไฟล์ที่กำหนดค่าไว้หลังจาก RPC ของ Gateway ภายในเครื่องโดยปริยายล้มเหลว บน Linux จะใช้เจอร์นัล Gateway ของ user-systemd ที่ใช้งานอยู่ตาม PID เมื่อมี และพิมพ์แหล่งบันทึกที่เลือก มิฉะนั้นจะลอง Gateway แบบสดซ้ำต่อไปแทนการติดตามไฟล์ข้างเคียงที่อาจล้าสมัย
- เมื่อใช้ `--follow` การตัดการเชื่อมต่อชั่วคราวของ Gateway (WebSocket ปิด หมดเวลา การเชื่อมต่อหลุด) จะกระตุ้นการเชื่อมต่อใหม่อัตโนมัติพร้อม exponential backoff (สูงสุด 8 ครั้ง จำกัดไม่เกิน 30 วินาทีระหว่างความพยายาม) จะพิมพ์คำเตือนไปยัง stderr ในแต่ละครั้งที่ลองซ้ำ และพิมพ์ประกาศ `[logs] gateway reconnected` เมื่อการโพลสำเร็จ ในโหมด `--json` ทั้งคำเตือนการลองซ้ำและการเปลี่ยนผ่านการเชื่อมต่อใหม่จะถูกส่งออกเป็นระเบียน `{"type":"notice"}` บน stderr ข้อผิดพลาดที่กู้คืนไม่ได้ (การยืนยันตัวตนล้มเหลว การกำหนดค่าไม่ถูกต้อง) จะออกทันทีเหมือนเดิม
- ในโหมด `--follow --json` การเปลี่ยนแหล่งบันทึกจะถูกส่งออกเป็นระเบียน `{"type":"meta"}` ผู้บริโภคควรติดตามเคอร์เซอร์แยกตาม `sourceKind`: สตรีมสามารถย้ายจากเอาต์พุตไฟล์ Gateway (`sourceKind: "file"`) ไปยังการถอยกลับเป็นเจอร์นัลภายในเครื่อง (`sourceKind: "journal"`, `localFallback: true`, พร้อม `service.pid`/`service.unit`) และกลับไปยังเอาต์พุตไฟล์ Gateway หลังจากกู้คืนแล้ว อย่าสันนิษฐานว่ามีแหล่งเดียวหรือเคอร์เซอร์เดียวที่คงที่ตลอดทั้งเซสชันการติดตาม และรองรับบรรทัดที่ซ้ำซ้อนเมื่อการกู้คืนเล่นซ้ำเคอร์เซอร์ไฟล์ Gateway
