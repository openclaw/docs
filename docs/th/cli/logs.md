# `openclaw logs`

ติดตามบันทึกไฟล์ของ Gateway ผ่าน RPC รองรับโหมดระยะไกล

## ตัวเลือก

- `--limit <n>`: จำนวนบรรทัดบันทึกสูงสุดที่จะส่งคืน (ค่าเริ่มต้น `200`)
- `--max-bytes <n>`: จำนวนไบต์สูงสุดที่จะอ่านจากไฟล์บันทึก (ค่าเริ่มต้น `250000`)
- `--follow`: ติดตามสตรีมบันทึก
- `--interval <ms>`: ช่วงเวลาการโพลขณะติดตาม (ค่าเริ่มต้น `1000`)
- `--json`: ส่งออกเหตุการณ์ JSON โดยแยกแต่ละรายการเป็นหนึ่งบรรทัด
- `--plain`: ส่งออกเป็นข้อความธรรมดาโดยไม่มีการจัดรูปแบบ
- `--no-color`: ปิดใช้สี ANSI
- `--local-time`: แสดงการประทับเวลาตามเขตเวลาท้องถิ่นของคุณ (ค่าเริ่มต้น)
- `--utc`: แสดงการประทับเวลาเป็น UTC

## ตัวเลือก RPC ที่ใช้ร่วมกันของ Gateway

- `--url <url>`: URL ของ WebSocket สำหรับ Gateway
- `--token <token>`: โทเค็นของ Gateway
- `--timeout <ms>`: ระยะหมดเวลาเป็นมิลลิวินาที (ค่าเริ่มต้น `30000`)
- `--expect-final`: รอการตอบกลับสุดท้ายเมื่อการเรียก Gateway มีเอเจนต์รองรับ

การระบุ `--url` จะข้ามข้อมูลรับรองจากการกำหนดค่าที่ระบบนำมาใช้โดยอัตโนมัติ หาก Gateway เป้าหมายกำหนดให้ต้องยืนยันตัวตน ให้ระบุ `--token` อย่างชัดเจน

## ตัวอย่าง

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## ลักษณะการทำงานเมื่อใช้ทางเลือกสำรองและการกู้คืน

- หาก Gateway แบบ local loopback โดยนัยร้องขอการจับคู่ ปิดการเชื่อมต่อระหว่างกำลังเชื่อมต่อ หรือหมดเวลาก่อนที่ `logs.tail` จะตอบกลับ `openclaw logs` จะเปลี่ยนไปใช้ไฟล์บันทึกของ Gateway ที่กำหนดค่าไว้โดยอัตโนมัติ เป้าหมายที่ระบุ `--url` อย่างชัดเจนจะไม่ใช้ทางเลือกสำรองนี้
- `--follow` จะไม่เปลี่ยนไปใช้ไฟล์ที่กำหนดค่าไว้นั้นหลังจาก RPC ของ Gateway ภายในเครื่องโดยนัยล้มเหลว เนื่องจากไฟล์เก่าที่อยู่คู่กันอาจทำให้เข้าใจผิดขณะติดตามข้อมูลสด บน Linux คำสั่งจะใช้เจอร์นัล Gateway ของ user-systemd ที่กำลังทำงานตาม PID แทนเมื่อพร้อมใช้งาน (พร้อมแสดงแหล่งข้อมูลที่เลือก) มิฉะนั้นจะพยายามเชื่อมต่อ Gateway ที่ทำงานอยู่ต่อไป
- ระหว่างใช้ `--follow` การตัดการเชื่อมต่อชั่วคราว (WebSocket ปิด หมดเวลา หรือการเชื่อมต่อหลุด) จะทำให้เชื่อมต่อใหม่โดยอัตโนมัติด้วยการหน่วงเวลาแบบทวีคูณ โดยลองใหม่สูงสุด 8 ครั้ง และจำกัดช่วงเวลาระหว่างความพยายามไว้ไม่เกิน 30 วินาที ระบบจะแสดงคำเตือนไปยัง stderr ทุกครั้งที่ลองใหม่ และแสดงข้อความแจ้ง `[logs] gateway reconnected` หนึ่งครั้งเมื่อการโพลสำเร็จ ในโหมด `--json` ทั้งสองรายการจะถูกส่งออกเป็นระเบียน `{"type":"notice"}` ไปยัง stderr ข้อผิดพลาดที่กู้คืนไม่ได้ (การยืนยันตัวตนล้มเหลวหรือการกำหนดค่าไม่ถูกต้อง) จะยังคงทำให้โปรแกรมออกทันที
- ในโหมด `--follow --json` การเปลี่ยนแหล่งบันทึกจะถูกส่งออกเป็นระเบียน `{"type":"meta"}` ให้ติดตามเคอร์เซอร์แยกตาม `sourceKind`: สตรีมสามารถเปลี่ยนจากเอาต์พุตไฟล์ของ Gateway (`sourceKind: "file"`) ไปเป็นทางเลือกสำรองจากเจอร์นัลภายในเครื่อง (`sourceKind: "journal"`, `localFallback: true` พร้อม `service.pid`/`service.unit`) และกลับไปเป็นเอาต์พุตไฟล์ของ Gateway หลังจากกู้คืนได้ อย่าคาดว่าแหล่งข้อมูลหรือเคอร์เซอร์จะคงที่เพียงรายการเดียวตลอดทั้งเซสชัน และควรรองรับบรรทัดที่ซ้ำซ้อนกันเมื่อการกู้คืนเล่นข้อมูลจากเคอร์เซอร์ไฟล์ของ Gateway ซ้ำ
