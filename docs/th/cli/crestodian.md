# `openclaw crestodian`

Crestodian แบบสนทนาคือเอเจนต์สำหรับการตั้งค่า การซ่อมแซม และการกำหนดค่า OpenClaw ภายในเครื่อง โดยจะเริ่มทำงานหลังจากโมเดลเริ่มต้นที่มีผลใช้งานดำเนินการโต้ตอบจริงสำเร็จแล้วเท่านั้น การติดตั้งใหม่จะตั้งค่าการอนุมานก่อน ส่วนการกำหนดค่าที่มีรูปแบบไม่ถูกต้องจะยังคงใช้กระบวนการ doctor แบบดั้งเดิม

## เริ่มทำงานเมื่อใด

การเรียกใช้ `openclaw` โดยไม่มีคำสั่งย่อยจะเลือกเส้นทางตามสถานะการกำหนดค่า:

- ไม่มีการกำหนดค่า หรือมีอยู่แต่ไม่มีการตั้งค่าที่ผู้ใช้กำหนด (ว่างเปล่า หรือมีเฉพาะคีย์ `$schema`/`meta`): เริ่มการเริ่มต้นใช้งานแบบมีคำแนะนำพร้อมการตรวจสอบด้วย AI แบบสด
- มีการกำหนดค่าแต่ตรวจสอบความถูกต้องไม่ผ่าน: เริ่มการเริ่มต้นใช้งานแบบดั้งเดิม ซึ่งจะรายงานปัญหาและแนะนำให้คุณใช้ `openclaw doctor`
- มีการกำหนดค่าและถูกต้อง: เปิด TUI ของเอเจนต์ตามปกติ Gateway ที่กำหนดค่าไว้และเข้าถึงได้ ซึ่งเอเจนต์เริ่มต้นมีโมเดล จะไปยัง UI นั้นโดยตรงโดยไม่ผ่านการเริ่มต้นใช้งานหรือ Crestodian หากต้องการเข้าสู่ Crestodian ภายหลัง ให้ใช้ `/crestodian` ภายใน TUI หรือเรียกใช้ `openclaw crestodian` โดยตรง

การเรียกใช้ `openclaw crestodian` จะทดสอบโมเดลเริ่มต้นที่กำหนดค่าไว้แบบสดก่อน หากการโต้ตอบผ่าน จะเริ่ม Crestodian หากล้มเหลวในโหมดโต้ตอบ ระบบจะเปิดการตั้งค่าการอนุมานแบบมีคำแนะนำ และส่งต่อไปยัง Crestodian หลังจากตัวเลือกหนึ่งผ่านการทดสอบ คำขอแบบครั้งเดียว, JSON และคำขอแบบไม่โต้ตอบอื่น ๆ จะล้มเหลวพร้อมคำแนะนำให้เรียกใช้ `openclaw onboard` เมื่อการอนุมานไม่พร้อมใช้งาน `openclaw --help` และ `openclaw --version` ยังคงใช้เส้นทางด่วนตามปกติ

`openclaw` เปล่าในโหมดไม่โต้ตอบ (ไม่มี TTY) จะออกพร้อมข้อความสั้น ๆ แทนการแสดงวิธีใช้ระดับราก โดยจะแนะนำไปยังการเริ่มต้นใช้งานแบบไม่โต้ตอบสำหรับการติดตั้งใหม่หรือไม่ถูกต้อง หรือไปยัง `openclaw agent --local ...` เมื่อการกำหนดค่าถูกต้อง

`openclaw onboard --modern` ยังคงเป็นนามแฝงเพื่อความเข้ากันได้สำหรับ Crestodian แต่ใช้ด่านตรวจการอนุมานเดียวกัน: การอนุมานที่ทำงานได้จะเปิดการสนทนา ความล้มเหลวในโหมดโต้ตอบจะเริ่มการตั้งค่าการอนุมานแบบมีคำแนะนำ และความล้มเหลวในโหมดไม่โต้ตอบจะออกพร้อมคำแนะนำการเริ่มต้นใช้งาน `openclaw onboard --classic` จะเปิดตัวช่วยแบบทีละขั้นตอนฉบับเต็ม

## สิ่งที่ Crestodian แสดง

Crestodian แบบโต้ตอบจะเปิดเชลล์ TUI เดียวกับ `openclaw tui` โดยใช้แบ็กเอนด์การสนทนาของ Crestodian คำทักทายเมื่อเริ่มต้นครอบคลุม:

- ความถูกต้องของการกำหนดค่าและเอเจนต์เริ่มต้น
- โมเดลที่ผ่านการตรวจสอบซึ่ง Crestodian กำลังใช้งาน
- การเข้าถึง Gateway จากการตรวจสอบครั้งแรกเมื่อเริ่มต้น
- การดำเนินการแก้จุดบกพร่องที่แนะนำเป็นลำดับถัดไป

ระบบจะไม่แสดงข้อมูลลับทั้งหมดหรือโหลดคำสั่ง CLI ของ Plugin เพียงเพื่อเริ่มทำงาน

ใช้ `status` เพื่อดูรายการโดยละเอียด ได้แก่ พาธการกำหนดค่า พาธเอกสาร/ซอร์ส การตรวจสอบ CLI ภายในเครื่อง สถานะการมีอยู่ของคีย์/โทเค็น เอเจนต์ โมเดล และรายละเอียด Gateway

Crestodian ใช้การค้นหาแหล่งอ้างอิงแบบเดียวกับเอเจนต์ทั่วไป: ในการเช็กเอาต์ Git ระบบจะชี้ไปยัง `docs/` ภายในเครื่องและโครงสร้างซอร์ส ส่วนในการติดตั้งผ่าน npm ระบบจะใช้เอกสารที่รวมมาให้และลิงก์ไปยัง [https://github.com/openclaw/openclaw](https://github.com/openclaw/openclaw) พร้อมคำแนะนำให้ตรวจสอบซอร์สเมื่อเอกสารยังไม่เพียงพอ

## การดำเนินการและการอนุมัติ

Crestodian ใช้การดำเนินการแบบมีชนิดแทนการแก้ไขการกำหนดค่าเฉพาะหน้า

การดำเนินการแบบอ่านอย่างเดียวจะทำงานทันที ได้แก่ แสดงภาพรวม แสดงรายการเอเจนต์ แสดงรายการ Plugin ที่ติดตั้ง ค้นหา Plugin ใน ClawHub แสดงสถานะโมเดล/แบ็กเอนด์ เรียกใช้การตรวจสอบสถานะ/สุขภาพ ตรวจสอบการเข้าถึง Gateway เรียกใช้ doctor โดยไม่มีการแก้ไขแบบโต้ตอบ ตรวจสอบความถูกต้องของการกำหนดค่า และแสดงพาธบันทึกการตรวจสอบ

การเริ่มตั้งค่าช่องทางแบบมีคำแนะนำ (`connect telegram`) จะทำงานทันทีเช่นกัน ตัวช่วยจะรวบรวมคำตอบอย่างชัดเจนและรับผิดชอบการเขียนค่าที่เกิดขึ้น

การดำเนินการที่คงอยู่ถาวรต้องได้รับการอนุมัติผ่านการสนทนา (หรือใช้ `--yes` สำหรับคำสั่งโดยตรง): เขียนการกำหนดค่า, `config set`, `config set-ref`, การตั้งค่า/บูตสแตรปการเริ่มต้นใช้งาน, เปลี่ยนโมเดลเริ่มต้น, เริ่ม/หยุด/เริ่ม Gateway ใหม่, สร้างเอเจนต์ และติดตั้ง Plugin

การซ่อมแซมด้วย doctor ไม่พร้อมใช้งานภายใน Crestodian เพราะอาจเขียนเส้นทางผู้ให้บริการ การยืนยันตัวตน หรือการอนุมานของเอเจนต์เริ่มต้นที่ขับเคลื่อนเซสชันใหม่ ให้ออกจาก Crestodian แล้วเรียกใช้ `openclaw doctor --fix` ในเทอร์มินัล `doctor` แบบอ่านอย่างเดียวยังคงพร้อมใช้งานภายใน Crestodian

เอเจนต์ใหม่จะสืบทอดเส้นทางการอนุมานเริ่มต้นที่ผ่านการตรวจสอบแบบสด รหัสเอเจนต์ `crestodian` สงวนไว้สำหรับผู้ดูแลเสมือนที่มีสิทธิ์พิเศษ และไม่สามารถสร้างเป็นเอเจนต์ทั่วไปได้

`config set` และ `config set-ref` ไม่สามารถเปลี่ยนสถานะเส้นทางการอนุมานได้ รวมถึงข้อมูลประจำตัวของผู้ให้บริการการอนุมาน, `auth.*` ระดับบนสุด, แค็ตตาล็อกโมเดล, แบ็กเอนด์ CLI, เส้นทางโมเดลเริ่มต้น/รายเอเจนต์, พารามิเตอร์/เครื่องมือของเอเจนต์ หรือ `tools.*` ระดับราก การเขียนค่าโดยตรงภายใต้ `env.*`, `secrets.*`, `plugins.*` และ `$include` จะถูกปฏิเสธเช่นกัน เพราะอาจแทนที่การแก้ไขข้อมูลประจำตัวหรือการเปิดใช้งานผู้ให้บริการ การยืนยันตัวตนของ Gateway และช่องทางยังคงเป็นพื้นผิวการกำหนดค่าตามปกติ ใช้เวิร์กโฟลว์ Plugin/ช่องทางแบบมีชนิด และ `set default model <provider/model>` สำหรับเส้นทางที่กำหนดค่าไว้แล้ว โดยระบบจะทดสอบเส้นทางแบบสดก่อนบันทึก หากต้องการกำหนดค่าหรือซ่อมแซมการเข้าถึงผู้ให้บริการ/การยืนยันตัวตน ให้ออกจาก Crestodian แล้วเรียกใช้ `openclaw onboard`

การถอนการติดตั้ง Plugin จะถูกปฏิเสธภายใน Crestodian เพราะการนำ Plugin ของผู้ให้บริการออกอาจปิดใช้งานเส้นทางการอนุมานที่ขับเคลื่อนเซสชัน ให้ออกจาก Crestodian แล้วเรียกใช้ `openclaw plugins uninstall <id>` จากเทอร์มินัล

การอนุมัติทำได้ด้วยถ้อยคำของคุณเอง: คำตอบที่ไม่กำกวม ("ใช่", "แน่นอน", "ดำเนินการเลย", "ยังไม่ใช่ตอนนี้") จะได้รับการตีความจากรายการแบบปิดที่กำหนดไว้แน่นอน เมื่อเส้นทางที่กำหนดค่ารองรับการเรียกการเติมข้อความแยกต่างหาก คำตอบอื่น ๆ สามารถจำแนกได้จากเฉพาะข้อความของคุณและข้อเสนอที่รอดำเนินการเท่านั้น — ไม่ใช้โมเดลการสนทนาเองเด็ดขาด เพราะโมเดลไม่สามารถอนุมัติตนเองได้ คำตอบที่จำแนกไม่ได้หรือกำกวมจะทำให้ข้อเสนอยังคงรอดำเนินการ และการสนทนาจะถามอีกครั้ง

การเขียนค่าที่นำไปใช้แล้วจะถูกบันทึกใน `~/.openclaw/audit/crestodian.jsonl` การค้นหาไม่มีการบันทึกตรวจสอบ โดยจะบันทึกเฉพาะการดำเนินการและการเขียนค่าที่นำไปใช้แล้วเท่านั้น

การตั้งค่าช่องทางสามารถดำเนินเป็นการสนทนาที่โฮสต์ไว้ได้จนกว่าจะถึงขั้นตอนข้อมูลลับ TUI ของ Crestodian ภายในเครื่องไม่รับคำตอบที่ละเอียดอ่อนในตัวช่วย เพราะข้อมูลการสนทนาที่ป้อนในเทอร์มินัลสามารถมองเห็นได้ ระบบจะเสนอ `open channel wizard` ทันที โดยส่งต่อช่องทางที่เลือกไปยังตัวช่วยในเทอร์มินัลที่ปกปิดข้อมูล หรือคุณสามารถเรียกใช้ `openclaw channels add --channel <channel>` ภายหลังก็ได้

### การสลับไปยังการตั้งค่าช่องทางแบบปกปิดข้อมูล

การสนทนาภายในเครื่องสามารถส่งต่อการควบคุมไปยังตัวช่วยช่องทางแบบปกปิดข้อมูลได้:

```text
open channel wizard for slack
channel info slack
```

`open channel wizard for <channel>` จะเปิดการตั้งค่าช่องทางแบบปกปิดข้อมูลหลังจาก TUI การสนทนาปิดลง ใช้ `channel info <channel>` ก่อนเพื่อดูป้ายชื่อช่องทาง สถานะการตั้งค่า สรุปข้อกำหนดเบื้องต้น และลิงก์เอกสาร

Crestodian จะไม่เปลี่ยนการเข้าถึงผู้ให้บริการ/การยืนยันตัวตนจากภายในเซสชันของตนเองเด็ดขาด เพราะเซสชันพึ่งพาเส้นทางการอนุมานนั้นอยู่แล้ว สำหรับการตั้งค่าหรือซ่อมแซมผู้ให้บริการโมเดล `configure model provider` จะส่งคืนคำแนะนำให้ออกและเริ่มต้นใช้งาน โดยไม่เริ่มตัวช่วยหรือเขียนการกำหนดค่า ให้ออกจาก Crestodian แล้วเรียกใช้ `openclaw onboard` การเริ่มต้นใช้งานจะจัดเตรียมข้อมูลประจำตัวและบันทึกเฉพาะเส้นทางที่ดำเนินการโต้ตอบจริงแบบสดสำเร็จ เริ่ม Crestodian อีกครั้งหลังจากการเริ่มต้นใช้งานสำเร็จ

## บูตสแตรปการตั้งค่า

`setup` กำหนดค่าสถานะพื้นที่ทำงานและ Gateway ที่เหลือ หลังจากการเริ่มต้นใช้งานแบบมีคำแนะนำได้ตั้งค่าการอนุมานเรียบร้อยแล้ว ระบบจะเขียนค่าผ่านการดำเนินการกำหนดค่าแบบมีชนิดเท่านั้นและขออนุมัติก่อน

```text
setup
setup workspace ~/Projects/work
```

`setup` จะคงโมเดลที่มีผลใช้งานซึ่งผ่านการตรวจสอบไว้ โดยจะไม่กำหนดค่าหรือแทนที่การอนุมาน

หากไม่มีการอนุมานหรือการตรวจสอบแบบสดล้มเหลว ให้ออกจาก Crestodian แล้วเรียกใช้ `openclaw onboard` การเริ่มต้นใช้งานแบบมีคำแนะนำจะตรวจหาโมเดลที่กำหนดค่าไว้ คีย์ API และ CLI ภายในเครื่องที่ผ่านการยืนยันตัวตน ขอให้แต่ละตัวเลือกตอบกลับจริง และคงอยู่เฉพาะเส้นทางที่ผ่านการทดสอบ Crestodian จะเริ่มทันทีหลังจากผ่านขอบเขตนั้น และจากนั้นจึงสามารถกำหนดค่าพื้นที่ทำงาน, Gateway, ช่องทาง, เอเจนต์, Plugin และคุณสมบัติเสริมอื่น ๆ ได้

แอป macOS จะข้ามลำดับนี้ทั้งหมดเมื่อเข้าถึง Gateway ที่กำหนดค่าไว้ ซึ่งเอเจนต์เริ่มต้นมีโมเดลที่กำหนดค่าไว้แล้ว โดยจะเปิด UI ของเอเจนต์ตามปกติ
สำหรับ Gateway ใหม่หรือยังไม่สมบูรณ์ แอปจะดำเนินลำดับการอนุมานผ่านเมธอด Gateway `crestodian.setup.detect` และ `crestodian.setup.activate`: detect จะแสดงรายการแบ็กเอนด์ตัวเลือกทั้งหมดที่พบ ส่วน activate จะทดสอบตัวเลือกหนึ่งแบบสด (การเติมข้อความ "reply with OK" จริง) และจะคงอยู่เฉพาะสถานะโมเดล ข้อมูลประจำตัว และผู้ให้บริการ/รันไทม์ที่จำเป็นสำหรับเส้นทางนั้น หลังจากการทดสอบผ่านแล้วเท่านั้น ค่าเริ่มต้นของพื้นที่ทำงานและ Gateway ยังคงเป็นหน้าที่ของ Crestodian ตัวเลือกที่ล้มเหลวจะไม่เปลี่ยนการกำหนดค่า แอปจะไล่ลงตามลำดับโดยอัตโนมัติ และสุดท้ายจะเสนอขั้นตอนคีย์/โทเค็นแบบกำหนดเอง ซึ่งสร้างรายการจาก Plugin ผู้ให้บริการการอนุมานข้อความที่ใช้งานอยู่ของ Gateway ผู้ให้บริการที่เลือกจะเป็นเจ้าของโมเดลเริ่มต้นและการกำหนดค่าของตน และข้อมูลประจำตัวจะได้รับการตรวจสอบด้วยวิธีเดียวกันก่อนบันทึก

การควบคุมดูแล Codex และคุณสมบัติเสริมอื่น ๆ ของ Plugin จะอยู่นอกธุรกรรมเปิดใช้งานการอนุมานนี้ กำหนดค่าคุณสมบัติเหล่านั้นหลังจากการอนุมานทำงานและ Crestodian เริ่มต้นแล้วเท่านั้น นโยบาย Plugin ที่มีอยู่และการยกเว้นการควบคุมดูแลอย่างชัดเจนจะไม่ถูกเปลี่ยนแปลงระหว่างการตั้งค่าการอนุมาน

## การสนทนาด้วย AI

การสนทนาแบบอิสระของ Crestodian แบบโต้ตอบจะทำงานผ่านลูปเอเจนต์เดียวกับเอเจนต์ OpenClaw ทั่วไป โดยจำกัดไว้ที่เครื่องมือสิทธิ์ระดับศูนย์ของ OpenClaw เพียงหนึ่งรายการคือ `crestodian` ซึ่งครอบการดำเนินการแบบมีชนิด การดำเนินการอ่านทำงานได้โดยอิสระ ส่วนการเปลี่ยนแปลงต้องได้รับการอนุมัติผ่านการสนทนาของคุณสำหรับการดำเนินการนั้นโดยเฉพาะ (ดูการดำเนินการและการอนุมัติ) และทุกการเขียนค่าที่นำไปใช้จะถูกบันทึกตรวจสอบและตรวจสอบความถูกต้องอีกครั้ง เซสชันเอเจนต์จะคงอยู่ ดังนั้น Crestodian จึงมีหน่วยความจำหลายรอบอย่างแท้จริง หากเส้นทางการอนุมานที่ผ่านการตรวจสอบหยุดทำงานในภายหลัง ให้กลับไปที่ `openclaw onboard` และซ่อมแซมก่อนดำเนินการต่อ

โฮสต์จะไม่แยกวิเคราะห์คำขอภาษาธรรมชาติให้เป็นการดำเนินการ ข้อความแบบอิสระ — รวมถึงข้อความที่ดูเหมือนคำสั่งและคำถาม เช่น "เหตุใด gateway ของฉันจึงหยุดทำงาน?" — จะถูกส่งไปยัง AI ซึ่งสามารถแมปคำขอเป็นการดำเนินการแบบมีชนิดผ่านเครื่องมือ `crestodian`

เมื่อมีการเปลี่ยนแปลงรอดำเนินการ เฉพาะวลีอนุมัติหรือปฏิเสธที่ไม่กำกวมจากรายการแบบปิดเท่านั้นที่จะได้รับการตีความโดยไม่ใช้การอนุมาน ความยินยอมที่กำกวมจะถูกส่งไปยังการเรียกการเติมข้อความที่กำหนดค่าไว้แยกต่างหาก และหากไม่สำเร็จจะปฏิเสธโดยค่าเริ่มต้น ฟิลด์ตัวช่วยแบบมีโครงสร้างและการนำทางโฮสต์แบบตรงตัวเป็นตัวควบคุม UI ไม่ใช่การแยกวิเคราะห์การดำเนินการจากภาษาธรรมชาติ ข้อยกเว้นด้านสุขอนามัยของข้อมูลลับที่สำคัญเป็นพิเศษคือ: `config set` แบบตรงตัวบนพาธที่ละเอียดอ่อน (โทเค็น คีย์ รหัสผ่าน) จะไม่ถูกส่งถึงโมเดลเด็ดขาด โฮสต์จะสร้างข้อเสนอที่ปกปิดข้อมูล และค่าจะถูกปกปิดในประวัติที่ AI มองเห็น สำหรับข้อมูลลับ ควรใช้ `config set-ref <path> env <ENV_VAR>`

โหมดกู้คืนผ่านช่องทางข้อความจะไม่ใช้ตัววางแผนที่มีโมเดลช่วยเหลือเด็ดขาด การกู้คืนระยะไกลยังคงทำงานแบบกำหนดผลแน่นอน เพื่อไม่ให้เส้นทางเอเจนต์ปกติที่เสียหายหรือถูกเจาะระบบถูกใช้เป็นตัวแก้ไขการกำหนดค่า

### โมเดลความไว้วางใจของชุดทดสอบ CLI

รันไทม์แบบฝังและชุดทดสอบแอปเซิร์ฟเวอร์ Codex บังคับใช้ข้อจำกัดสิทธิ์ระดับศูนย์โดยตรง: การเรียกใช้จะมีรายการอนุญาตเครื่องมือ OpenClaw ซึ่งมีเฉพาะเครื่องมือ `crestodian` สำหรับ Codex นั้น OpenClaw ยังปิดใช้งานสภาพแวดล้อม การดำเนินการแบบเนทีฟ หลายเอเจนต์ เป้าหมาย แอป/Plugin, skill/MCP, การค้นหาเว็บ และพื้นผิว `request_user_input` สำหรับการเรียกใช้นั้น Codex ยังคงแทรกยูทิลิตีเนทีฟ `update_plan` ที่ไม่ก่อให้เกิดการเปลี่ยนแปลง ซึ่งสามารถอัปเดตรายการตรวจสอบชั่วคราวของโมเดลได้ แต่ไม่สามารถเขียนไฟล์หรือการกำหนดค่า OpenClaw ชุดทดสอบ CLI ไม่ใช้รายการอนุญาตของ OpenClaw ดังนั้น Crestodian จึงยอมรับเฉพาะแบ็กเอนด์ที่สัญญาการเลือกเครื่องมือของตนสามารถพิสูจน์ข้อจำกัดเดียวกันได้:

- แบ็กเอนด์ที่เลือกได้ รวมถึง Claude Code จะเริ่มทำงานโดยไม่มีการเลือกเครื่องมือเนทีฟ
  และมีเครื่องมือ MCP หนึ่งรายการคือ `crestodian` การกำหนดค่า MCP ที่ Claude สร้างขึ้นจะถูก
  นำไปใช้พร้อม `--strict-mcp-config` ดังนั้นจึงไม่มีการโหลดเซิร์ฟเวอร์ MCP อื่น
- แบ็กเอนด์ที่ประกาศว่าไม่มีเครื่องมือเนทีฟจะได้รับเซิร์ฟเวอร์ MCP สำหรับ Crestodian
  โดยเฉพาะเช่นเดียวกัน
- แบ็กเอนด์เครื่องมือเนทีฟแบบเปิดตลอดเวลาหรือไม่ทราบประเภทจะปฏิเสธการทำงานอย่างปลอดภัยก่อนการอนุมาน
  จึงไม่สามารถโฮสต์เซสชัน Crestodian ได้

เฉพาะเซสชัน Crestodian เท่านั้นที่ได้รับเซิร์ฟเวอร์ MCP ของ crestodian ส่วนการเรียกใช้เอเจนต์ตามปกติ
จะไม่เห็นเครื่องมือนี้เลย ดังนั้นแบ็กเอนด์ CLI แบบเลือกได้/ไม่มีเครื่องมือเนทีฟและโมเดลที่ใช้คีย์ API
จึงบังคับใช้ลูปที่มีเครื่องมือเดียวตามตัวอักษร โมเดลแอปเซิร์ฟเวอร์ Codex บังคับใช้
เครื่องมืออำนาจควบคุมของ OpenClaw เพียงรายการเดียวร่วมกับยูทิลิตีวางแผนเนทีฟที่ไม่ทำงาน ในทั้ง
สามกรณี การเขียนระหว่างการตั้งค่ายังคงจำกัดอยู่ภายในสัญญาการอนุมัติที่ผ่านการตรวจสอบของ Crestodian

Gemini CLI ยังคงพร้อมใช้งานสำหรับเอเจนต์ทั่วไป แต่ไม่สามารถบังคับใช้
การตรวจสอบแบบไม่ใช้เครื่องมือที่เกตการอนุมานกำหนด จึงไม่สามารถโฮสต์ Crestodian ได้

## การสลับไปยังเอเจนต์

ใช้ตัวเลือกภาษาธรรมชาติเพื่อออกจาก Crestodian และเปิด TUI ตามปกติ:

```text
คุยกับเอเจนต์
คุยกับเอเจนต์ work
สลับไปยังเอเจนต์ main
```

`openclaw tui`, `openclaw chat` และ `openclaw terminal` จะเปิด TUI ของเอเจนต์ตามปกติโดยตรง โดยไม่เริ่ม Crestodian หลังจากสลับเข้าสู่ TUI ตามปกติแล้ว `/crestodian` จะกลับไปยัง Crestodian และอาจระบุคำขอต่อเนื่องเพิ่มเติมได้:

```text
/crestodian
/crestodian restart gateway
```

## โหมดกู้คืนข้อความ

โหมดกู้คืนข้อความคือจุดเข้าสู่ Crestodian ผ่านช่องทางข้อความ ใช้เมื่อเอเจนต์ตามปกติของคุณหยุดทำงาน แต่ช่องทางที่เชื่อถือได้ (เช่น WhatsApp) ยังคงรับคำสั่งได้

นี่คือตัวจัดการคำสั่งฉุกเฉินแบบกำหนดผลลัพธ์ตายตัว ไม่ใช่เอเจนต์ Crestodian
แบบสนทนา โดยจะไม่เริ่มต้นการตั้งค่าใหม่หรือผ่อนปรนเกตการอนุมาน
สำหรับแชต Crestodian

คำสั่งที่รองรับ: `/crestodian <request>` การกู้คืนยอมรับเฉพาะไวยากรณ์คำสั่งที่พิมพ์ตรงตามรูปแบบเท่านั้น — ภาษาธรรมชาติจะถูกปฏิเสธพร้อมคำแนะนำ จะไม่มีการคาดเดาให้กลายเป็นการดำเนินการ และจะไม่มีการเรียกใช้โมเดลโดยเด็ดขาด

```text
คุณ ใน DM เจ้าของที่เชื่อถือได้: /crestodian status
OpenClaw: โหมดกู้คืน Crestodian เข้าถึง Gateway ได้: ไม่ การกำหนดค่าถูกต้อง: ไม่
คุณ: /crestodian restart gateway
OpenClaw: แผน: เริ่ม Gateway ใหม่ ตอบกลับ /crestodian yes เพื่อดำเนินการ
คุณ: /crestodian yes
OpenClaw: ดำเนินการแล้ว เขียนรายการบันทึกการตรวจสอบแล้ว
```

สามารถจัดคิวการสร้างเอเจนต์ภายในเครื่องหรือผ่านการกู้คืนได้เช่นกัน:

```text
create agent work workspace ~/Projects/work model openai/gpt-5.6-sol
/crestodian create agent work workspace ~/Projects/work
```

การสร้างเอเจนต์ระบุชื่อได้เฉพาะโมเดลเริ่มต้นปัจจุบันที่ผ่านการตรวจสอบสดแล้วเท่านั้น หากไม่ระบุ
โมเดล ระบบจะสืบทอดเส้นทางนั้น

การกู้คืนระยะไกลเป็นพื้นผิวผู้ดูแลระบบและต้องได้รับการปฏิบัติเสมือนการซ่อมแซมการกำหนดค่าระยะไกล ไม่ใช่แชตตามปกติ

สัญญาความปลอดภัยสำหรับการกู้คืนระยะไกล:

- ปิดใช้งานเมื่อแซนด์บ็อกซ์ทำงานอยู่สำหรับเอเจนต์/เซสชัน โดย Crestodian จะปฏิเสธการกู้คืนระยะไกลและแนะนำให้ซ่อมแซมผ่าน CLI ภายในเครื่อง
- สถานะที่มีผลเริ่มต้นคือ `auto`: อนุญาตการกู้คืนระยะไกลเฉพาะในการทำงานแบบ YOLO ที่เชื่อถือได้ ซึ่งรันไทม์มีอำนาจภายในเครื่องแบบไม่ใช้แซนด์บ็อกซ์อยู่แล้ว (`tools.exec.security` ประเมินค่าเป็น `full` และ `tools.exec.ask` ประเมินค่าเป็น `off` โดยโหมดแซนด์บ็อกซ์เป็น `off`)
- ต้องมีข้อมูลประจำตัวเจ้าของที่ระบุอย่างชัดเจน ห้ามใช้กฎผู้ส่งแบบไวลด์การ์ด นโยบายกลุ่มแบบเปิด Webhook ที่ไม่ผ่านการยืนยันตัวตน หรือช่องทางนิรนาม
- ตามค่าเริ่มต้นอนุญาตเฉพาะ DM ของเจ้าของ การกู้คืนผ่านกลุ่ม/ช่องทางต้องเลือกเปิดใช้งานอย่างชัดเจน
- การค้นหาและแสดงรายการ Plugin เป็นแบบอ่านอย่างเดียว การติดตั้ง Plugin ทำได้ภายในเครื่องเท่านั้นเสมอ (ถูกบล็อกในการกู้คืน แม้จะเปิดใช้งานไว้ในกรณีอื่น) เนื่องจากเป็นการดาวน์โหลดโค้ดที่เรียกใช้งานได้ การถอนการติดตั้ง Plugin จะถูกปฏิเสธทั้งใน Crestodian ภายในเครื่องและการกู้คืน ให้เรียกใช้ `openclaw plugins uninstall <id>` จากเทอร์มินัล
- การกู้คืนระยะไกลไม่สามารถเปิด TUI ภายในเครื่องหรือสลับเข้าสู่เซสชันเอเจนต์แบบโต้ตอบได้ ให้ใช้ `openclaw` ภายในเครื่องเพื่อส่งมอบงานให้เอเจนต์
- การเขียนแบบถาวรยังคงต้องได้รับการอนุมัติ แม้อยู่ในโหมดกู้คืน
- การดำเนินการกู้คืนทุกครั้งที่นำไปใช้จะถูกบันทึกเพื่อตรวจสอบ การกู้คืนผ่านช่องทางข้อความจะบันทึกช่องทาง บัญชี ผู้ส่ง และข้อมูลเมตาที่อยู่ต้นทาง ส่วนการดำเนินการที่แก้ไขการกำหนดค่าจะบันทึกแฮชการกำหนดค่าทั้งก่อนและหลังด้วย
- จะไม่มีการแสดงข้อมูลลับกลับมาโดยเด็ดขาด การตรวจสอบ SecretRef จะรายงานความพร้อมใช้งาน ไม่ใช่ค่า
- หาก Gateway ยังทำงานอยู่ การกู้คืนจะเลือกใช้การดำเนินการแบบมีชนิดของ Gateway หาก Gateway หยุดทำงาน การกู้คืนจะใช้เฉพาะพื้นผิวการซ่อมแซมภายในเครื่องขั้นต่ำที่ไม่พึ่งพาลูปเอเจนต์ตามปกติ

รูปแบบการกำหนดค่า:

```jsonc
{
  "crestodian": {
    "rescue": {
      "enabled": "auto",
      "ownerDmOnly": true,
      "pendingTtlMinutes": 15,
    },
  },
}
```

- `enabled`: `"auto"` (ค่าเริ่มต้น) อนุญาตการกู้คืนเฉพาะเมื่อรันไทม์ที่มีผลเป็น YOLO และปิดแซนด์บ็อกซ์อยู่ `false` จะไม่อนุญาตการกู้คืนผ่านช่องทางข้อความโดยเด็ดขาด ส่วน `true` จะอนุญาตการกู้คืนอย่างชัดเจนเมื่อผ่านการตรวจสอบเจ้าของ/ช่องทาง (แต่ยังคงอยู่ภายใต้การปฏิเสธเมื่อใช้แซนด์บ็อกซ์)
- `ownerDmOnly`: จำกัดการกู้คืนไว้เฉพาะข้อความโดยตรงจากเจ้าของ ค่าเริ่มต้นคือ `true`
- `pendingTtlMinutes`: ระยะเวลาที่การเขียนเพื่อกู้คืนซึ่งรออนุมัติจะยังเปิดไว้สำหรับการอนุมัติด้วย `/crestodian yes` ก่อนหมดอายุ ค่าเริ่มต้นคือ `15`

การกู้คืนระยะไกลครอบคลุมโดยเลน Docker:

```bash
pnpm test:docker:crestodian-rescue
```

การทดสอบควันของพื้นผิวคำสั่งช่องทางสดแบบเลือกเปิดใช้งานจะตรวจสอบ `/crestodian status` พร้อมการทดสอบไป-กลับของการอนุมัติแบบถาวรผ่านตัวจัดการการกู้คืน:

```bash
pnpm test:live:crestodian-rescue-channel
```

การตั้งค่าแบบครั้งเดียวของแพ็กเกจที่มีเกตการอนุมานครอบคลุมโดย:

```bash
pnpm test:docker:crestodian-first-run
```

เลน CLI แบบแพ็กเกจนี้เริ่มต้นด้วยไดเรกทอรีสถานะว่างและพิสูจน์ว่า Crestodian
ปฏิเสธการทำงานอย่างปลอดภัยเมื่อไม่มีการอนุมาน จากนั้นจะทดสอบและเปิดใช้งาน Claude จำลองผ่าน
โมดูลเปิดใช้งานแบบแพ็กเกจ หลังจากนั้นเท่านั้นคำขอแบบคลุมเครือจึงจะเข้าถึง
ตัววางแผนและแปลงเป็นการตั้งค่าแบบมีชนิด ตามด้วยคำสั่งครั้งเดียวที่สร้าง
เอเจนต์เพิ่มเติม กำหนดค่า Discord ผ่านการเปิดใช้งาน Plugin ร่วมกับโทเค็น
SecretRef ตรวจสอบความถูกต้องของการกำหนดค่า และตรวจสอบบันทึกการตรวจสอบ เลนนี้เป็นหลักฐานสนับสนุน
สำหรับเกต/การดำเนินการ แต่ไม่ได้ทดสอบการเริ่มต้นใช้งานแบบโต้ตอบหรือบทสนทนา
ของเอเจนต์/เครื่องมือ/การอนุมัติ Crestodian สถานการณ์ QA Lab ด้านล่างจะเปลี่ยนเส้นทาง
ไปยังเลน Docker เดียวกัน:

```bash
pnpm openclaw qa suite --scenario crestodian-ring-zero-setup
```
