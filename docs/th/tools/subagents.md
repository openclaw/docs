เอเจนต์ย่อยคือการรันเอเจนต์เบื้องหลังที่สร้างขึ้นจากการรันเอเจนต์ที่มีอยู่
แต่ละตัวทำงานในเซสชันของตนเอง (`agent:<agentId>:subagent:<uuid>`) และ
เมื่อเสร็จสิ้น จะ **ประกาศ** ผลลัพธ์กลับไปยังช่องแชทของผู้ร้องขอ
การรันเอเจนต์ย่อยทุกรายการจะถูกติดตามเป็น [งานเบื้องหลัง](/th/automation/tasks)

เป้าหมาย:

- ทำงานวิจัย งานที่ใช้เวลานาน และงานเครื่องมือที่ทำงานช้าแบบขนาน โดยไม่ปิดกั้นการรันหลัก
- แยกเอเจนต์ย่อยออกจากกันโดยค่าเริ่มต้น (แยกเซสชัน และเลือกใช้แซนด์บ็อกซ์ได้)
- ทำให้ชุดเครื่องมือใช้งานผิดได้ยาก: โดยค่าเริ่มต้น เอเจนต์ย่อยจะ **ไม่ได้รับ** เครื่องมือเซสชันหรือข้อความ
- รองรับความลึกของการซ้อนที่กำหนดค่าได้สำหรับรูปแบบออร์เคสเตรเตอร์

<Note>
**หมายเหตุด้านค่าใช้จ่าย:** โดยค่าเริ่มต้น เอเจนต์ย่อยแต่ละตัวมีบริบทและการใช้โทเค็น
แยกเป็นของตนเอง สำหรับงานหนักหรืองานซ้ำ ให้ตั้งค่าโมเดลที่มีราคาถูกกว่าแก่เอเจนต์ย่อย
และคงเอเจนต์หลักไว้บนโมเดลคุณภาพสูงกว่าผ่าน
`agents.defaults.subagents.model` หรือการกำหนดค่าเฉพาะเอเจนต์ เมื่อเอเจนต์ลูก
จำเป็นต้องใช้บทสนทนาปัจจุบันของผู้ร้องขอจริง ๆ ให้สร้างโดยใช้
`context: "fork"` เซสชันเอเจนต์ย่อยที่ผูกกับเธรดมีค่าเริ่มต้นเป็น
`context: "fork"` เนื่องจากเซสชันเหล่านี้แตกแขนงบทสนทนาปัจจุบันไปเป็น
เธรดติดตามผล
</Note>

## คำสั่งเครื่องหมายทับ

`/subagents` ใช้ตรวจสอบการรันเอเจนต์ย่อยสำหรับ **เซสชันปัจจุบัน**:

```text
/subagents list
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
```

`/subagents info` แสดงข้อมูลเมตาของการรัน (สถานะ การประทับเวลา รหัสเซสชัน
พาธบทสนทนา การล้างข้อมูล) `/subagents log` แสดงช่วงการสนทนาล่าสุดของ
การรัน เพิ่มโทเค็น `tools` เพื่อรวมข้อความการเรียกเครื่องมือ/ผลลัพธ์ (ค่าเริ่มต้น
จะไม่รวม) ใช้ `sessions_history` เพื่อเรียกดูย้อนหลังแบบจำกัดขอบเขตและกรองเพื่อความปลอดภัย
จากภายในรอบการทำงานของเอเจนต์ หรือตรวจสอบพาธบทสนทนาบนดิสก์เพื่อดู
บทสนทนาฉบับเต็มแบบดิบ

ใน Control UI เซสชันแม่ที่มีการรันลูกเมื่อไม่นานมานี้จะมีแถวแถบด้านข้าง
ที่ขยายได้ แถวที่ซ้อนอยู่จะแสดงสถานะและระยะเวลาการทำงานของลูก และเมื่อเลือกแถวใด
จะแชทของลูกนั้นโดยยังคงลำดับชั้นของเซสชันแม่ไว้

### การควบคุมการผูกเธรด

คำสั่งเหล่านี้ทำงานบนช่องที่รองรับการผูกเธรดแบบถาวร ดู
[ช่องที่รองรับเธรด](#thread-supporting-channels) ด้านล่าง

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### พฤติกรรมการสร้าง

เอเจนต์เริ่มเอเจนต์ย่อยเบื้องหลังด้วยเครื่องมือ `sessions_spawn`
ผลลัพธ์เมื่อเสร็จสิ้นจะกลับมาเป็นเหตุการณ์ภายในเซสชันแม่ โดยเอเจนต์แม่/ผู้ร้องขอ
จะตัดสินใจว่าจำเป็นต้องอัปเดตข้อมูลที่ผู้ใช้มองเห็นหรือไม่

## โหมดบริบท

เอเจนต์ย่อยแบบเนทีฟเริ่มต้นโดยแยกออกจากกัน เว้นแต่ผู้เรียกจะร้องขออย่างชัดเจนให้แตกแขนง
บทสนทนาปัจจุบัน

| โหมด       | ควรใช้เมื่อใด                                                                                                                         | พฤติกรรม                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | การวิจัยใหม่ การใช้งานที่เป็นอิสระ งานเครื่องมือที่ทำงานช้า หรือสิ่งใดก็ตามที่สามารถอธิบายอย่างย่อในข้อความงานได้                           | สร้างบทสนทนาของเอเจนต์ลูกที่สะอาด นี่คือค่าเริ่มต้นและช่วยลดการใช้โทเค็น  |
| `fork`     | งานที่ขึ้นอยู่กับบทสนทนาปัจจุบัน ผลลัพธ์เครื่องมือก่อนหน้า หรือคำสั่งที่มีรายละเอียดซับซ้อนซึ่งมีอยู่แล้วในบทสนทนาของผู้ร้องขอ | แตกแขนงบทสนทนาของผู้ร้องขอไปยังเซสชันเอเจนต์ลูกก่อนที่เอเจนต์ลูกจะเริ่มทำงาน |

ใช้ `fork` เท่าที่จำเป็น มีไว้สำหรับการมอบหมายงานที่ไวต่อบริบท ไม่ใช่
สิ่งทดแทนการเขียนพรอมต์งานที่ชัดเจน

## เครื่องมือ: `sessions_spawn`

เริ่มการรันเอเจนต์ย่อยด้วย `deliver: false` บนเลน `subagent` ส่วนกลาง
จากนั้นเรียกใช้ขั้นตอนการประกาศและโพสต์การตอบกลับการประกาศไปยังช่องแชทของผู้ร้องขอ

ความพร้อมใช้งานขึ้นอยู่กับนโยบายเครื่องมือที่มีผลของผู้เรียก โปรไฟล์ในตัว
`coding` และ `messaging` มี `sessions_spawn`,
`sessions_yield` และ `subagents`; ส่วน `minimal` ไม่มี `full` อนุญาต
เครื่องมือทุกตัว เพิ่มเครื่องมือเหล่านั้นด้วย `tools.alsoAllow` หรือใช้หนึ่งในโปรไฟล์
ข้างต้น สำหรับเอเจนต์ที่ใช้โปรไฟล์กำหนดเองซึ่งแคบกว่าแต่ยังควร
มอบหมายงานได้
นโยบายอนุญาต/ปฏิเสธของช่อง/กลุ่ม ผู้ให้บริการ แซนด์บ็อกซ์ และเฉพาะเอเจนต์
ยังคงนำเครื่องมือออกได้หลังขั้นตอนโปรไฟล์ ใช้ `/tools` จากเซสชันเดียวกัน
เพื่อยืนยันรายการเครื่องมือที่มีผล

**ค่าเริ่มต้น:**

- **โมเดล:** เอเจนต์ย่อยแบบเนทีฟสืบทอดจากผู้เรียก เว้นแต่จะตั้งค่า `agents.defaults.subagents.model` (หรือ `agents.list[].subagents.model` เฉพาะเอเจนต์) การสร้างผ่านรันไทม์ ACP ใช้โมเดลเอเจนต์ย่อยที่กำหนดค่าไว้เช่นกันเมื่อมี มิฉะนั้นชุดทดสอบ ACP จะคงค่าเริ่มต้นของตนเองไว้ `sessions_spawn.model` ที่ระบุอย่างชัดเจนยังคงมีลำดับความสำคัญสูงกว่า
- **การคิด:** เอเจนต์ย่อยแบบเนทีฟสืบทอดจากผู้เรียก เว้นแต่จะตั้งค่า `agents.defaults.subagents.thinking` (หรือ `agents.list[].subagents.thinking` เฉพาะเอเจนต์) การสร้างผ่านรันไทม์ ACP จะใช้ `agents.defaults.models["provider/model"].params.thinking` กับโมเดลที่เลือกด้วย `sessions_spawn.thinking` ที่ระบุอย่างชัดเจนยังคงมีลำดับความสำคัญสูงกว่า
- **ระยะหมดเวลาการรัน:** OpenClaw ใช้ `agents.defaults.subagents.runTimeoutSeconds` เมื่อตั้งค่าไว้ มิฉะนั้นจะย้อนกลับไปใช้ `0` (ไม่มีการหมดเวลา) `sessions_spawn` ไม่รับการกำหนดระยะหมดเวลาเฉพาะการเรียก
- **อายุการทำงานของโพรเซส:** เอเจนต์ย่อย OpenClaw ที่แยกออกมามีวงจรชีวิตการรันเป็นของตนเอง งานเบื้องหลังที่สร้างภายในแบ็กเอนด์ CLI ภายนอกนั้นแตกต่างออกไป โดยใช้โพรเซสย่อย CLI ร่วมกับเอเจนต์แม่และจะหยุดหากเอเจนต์แม่เข้าสู่ `agents.defaults.timeoutSeconds`
- **การส่งมอบงาน:** เอเจนต์ย่อยแบบเนทีฟได้รับงานที่มอบหมายในข้อความ `[Subagent Task]` แรกที่มองเห็นได้ พรอมต์ระบบของเอเจนต์ย่อยมีข้อกำหนดรันไทม์และบริบทการกำหนดเส้นทาง ไม่ใช่สำเนางานที่ซ่อนไว้

การสร้างเอเจนต์ย่อยแบบเนทีฟที่ยอมรับแล้วจะรวมข้อมูลเมตาของโมเดลเอเจนต์ลูกที่แก้ไขแล้ว
ไว้ในผลลัพธ์เครื่องมือ: `resolvedModel` มีการอ้างอิงโมเดลที่ใช้ และ
`resolvedProvider` มีคำนำหน้าผู้ให้บริการเมื่อการอ้างอิงมีคำนำหน้าดังกล่าว

### โหมดพรอมต์การมอบหมายงาน

`agents.defaults.subagents.delegationMode` ควบคุมเฉพาะคำแนะนำในพรอมต์ โดยไม่เปลี่ยนนโยบายเครื่องมือหรือบังคับใช้การมอบหมายงาน

- `suggest` (ค่าเริ่มต้น): คงคำแนะนำมาตรฐานในพรอมต์ให้ใช้เอเจนต์ย่อยสำหรับงานที่ใหญ่ขึ้นหรือช้าลง
- `prefer`: บอกเอเจนต์หลักให้ตอบสนองอย่างต่อเนื่อง และมอบหมายสิ่งที่ซับซ้อนกว่าการตอบกลับโดยตรงผ่าน `sessions_spawn`

การแทนที่เฉพาะเอเจนต์: `agents.list[].subagents.delegationMode`

```json5
{
  agents: {
    defaults: {
      subagents: {
        delegationMode: "prefer",
        maxConcurrent: 4,
      },
    },
    list: [
      {
        id: "coordinator",
        subagents: { delegationMode: "prefer" },
      },
    ],
  },
}
```

### ชื่องานและการกำหนดเป้าหมาย

`taskName` เป็นชื่ออ้างอิงสำหรับโมเดลเพื่อใช้จัดการลำดับงาน ไม่ใช่คีย์เซสชัน
ใช้สำหรับชื่อ child แบบคงที่ เช่น `review_subagents`,
`linux_validation` หรือ `docs_update` เมื่อผู้ประสานงานอาจต้องตรวจสอบ
child นั้นภายหลัง

การแก้ไขเป้าหมายยอมรับทั้งค่าที่ตรงกับ `taskName` ทุกประการและ
คำนำหน้าที่ไม่กำกวม การจับคู่ถูกจำกัดให้อยู่ในหน้าต่างเป้าหมายที่ใช้งานอยู่/ล่าสุดเดียวกับ
ที่เป้าหมายแบบมีหมายเลข `/subagents` ใช้ ดังนั้น child ที่เสร็จสิ้นแล้วและเก่าเกินไปจะไม่ทำให้
ชื่ออ้างอิงที่นำกลับมาใช้ใหม่เกิดความกำกวม หาก child ที่ใช้งานอยู่หรือล่าสุดสองรายการใช้
`taskName` เดียวกัน เป้าหมายจะกำกวม ให้ใช้ดัชนีรายการ คีย์เซสชัน หรือ
run id แทน

เป้าหมายสงวน `last` และ `all` ไม่ใช่ค่า `taskName` ที่ถูกต้อง
เนื่องจากมีความหมายด้านการควบคุมอยู่แล้ว

## เครื่องมือ: `sessions_yield`

สิ้นสุดเทิร์นโมเดลปัจจุบันและรอให้เหตุการณ์จากรันไทม์ โดยหลักคือ
เหตุการณ์ที่ sub-agent ทำงานเสร็จ มาถึงในฐานะข้อความถัดไป ใช้หลังจาก
สร้างงาน child ที่จำเป็น เมื่อผู้ร้องขอยังไม่สามารถให้คำตอบสุดท้ายได้
จนกว่างานเหล่านั้นจะเสร็จสิ้น

`sessions_yield` เป็นกลไกพื้นฐานสำหรับการรอ อย่าแทนที่ด้วยลูปโพล
ผ่าน `subagents`, `sessions_list`, `sessions_history`, `sleep` ของเชลล์
หรือการโพลโพรเซสเพียงเพื่อตรวจหาว่า child ทำงานเสร็จแล้ว

ใช้ `sessions_yield` เฉพาะเมื่อรายการเครื่องมือที่มีผลของเซสชันมีเครื่องมือนี้
โปรไฟล์เครื่องมือแบบขั้นต่ำหรือกำหนดเองบางรายการอาจเปิดให้ใช้ `sessions_spawn` และ
`subagents` โดยไม่เปิดให้ใช้ `sessions_yield`; ในกรณีนั้น อย่าสร้าง
ลูปโพลขึ้นมาเพียงเพื่อรอให้งานเสร็จ

เมื่อมี child ที่ใช้งานอยู่ OpenClaw จะแทรกบล็อกพรอมต์
`Active Subagents` แบบกะทัดรัดที่สร้างโดยรันไทม์ลงในเทิร์นปกติ เพื่อให้ผู้ร้องขอเห็น
เซสชัน child ปัจจุบัน, run id, สถานะ, ป้ายกำกับ, งาน และ
นามแฝง `taskName` ได้โดยไม่ต้องโพล ฟิลด์งานและป้ายกำกับในบล็อกนั้น
ถูกใส่เครื่องหมายคำพูดในฐานะข้อมูล ไม่ใช่คำสั่ง เนื่องจากอาจมาจาก
อาร์กิวเมนต์การสร้างที่ผู้ใช้/โมเดลระบุ

## เครื่องมือ: `subagents`

แสดงรายการการรัน sub-agent ที่สร้างไว้และระเบียนงานเบื้องหลังซึ่งเป็นของ
ทรีเซสชันผู้ร้องขอ แถวงานครอบคลุม sub-agent แบบเนทีฟ การรัน ACP
งาน Gateway CLI/สื่อ และการดำเนินการ Cron ขอบเขตถูกจำกัดไว้ที่ผู้ร้องขอปัจจุบัน
โดย child จะมองเห็นได้เฉพาะ child ที่ตนควบคุมเอง

ใช้ `subagents` สำหรับดูสถานะและแก้ไขจุดบกพร่องตามต้องการ ใช้ `sessions_yield` เพื่อ
รอเหตุการณ์งานเสร็จ

ใช้ `action: "cancel"` ร่วมกับ `taskId` ที่ `action: "list"` ส่งคืนเพื่อหยุด
งาน การยกเลิกถูกจำกัดอยู่ภายในทรีเซสชันที่ควบคุม โดย sub-agent ปลายสุด
ไม่สามารถยกเลิกงานที่เป็นของเซสชันอื่นได้

## เซสชันที่ผูกกับเธรด

เมื่อเปิดใช้การผูกเธรดสำหรับช่องทาง sub-agent สามารถผูกอยู่กับ
เธรดต่อไปได้ เพื่อให้ข้อความติดตามผลของผู้ใช้ในเธรดนั้นยังคงถูกกำหนดเส้นทางไปยัง
เซสชัน sub-agent เดิม

### ช่องทางที่รองรับเธรด

ช่องทางรองรับเซสชัน subagent แบบถาวรที่ผูกกับเธรด
(`sessions_spawn` ร่วมกับ `thread: true`) เมื่อช่องทางนั้นลงทะเบียนอะแดปเตอร์
การผูกการสนทนา ช่องทางที่มาพร้อมระบบและรองรับ ได้แก่ **Discord**,
**iMessage**, **Matrix** และ **Telegram** โดยค่าเริ่มต้น Discord และ Matrix
จะสร้างเธรด child ส่วน Telegram และ iMessage จะผูก
การสนทนาปัจจุบัน ใช้คีย์การกำหนดค่า `threadBindings` ของแต่ละช่องทางสำหรับ
การเปิดใช้ การหมดเวลา และ `spawnSessions`

### ขั้นตอนแบบย่อ

<Steps>
  <Step title="สร้าง">
    `sessions_spawn` ร่วมกับ `thread: true` (และ `mode: "session"` ซึ่งระบุหรือไม่ก็ได้)
  </Step>
  <Step title="ผูก">
    OpenClaw สร้างหรือผูกเธรดเข้ากับเป้าหมายเซสชันนั้นในช่องทางที่ใช้งานอยู่
  </Step>
  <Step title="กำหนดเส้นทางข้อความติดตามผล">
    การตอบกลับและข้อความติดตามผลในเธรดนั้นจะถูกกำหนดเส้นทางไปยังเซสชันที่ผูกไว้
  </Step>
  <Step title="ตรวจสอบการหมดเวลา">
    ใช้ `/session idle` เพื่อตรวจสอบ/อัปเดตการยกเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน และ
    `/session max-age` เพื่อควบคุมขีดจำกัดสูงสุดแบบตายตัว
  </Step>
  <Step title="ยกเลิกการเชื่อม">
    ใช้ `/unfocus` เพื่อยกเลิกการเชื่อมด้วยตนเอง
  </Step>
</Steps>

### การควบคุมด้วยตนเอง

| คำสั่ง            | ผล                                                                                       |
| ------------------ | ----------------------------------------------------------------------------------------- |
| `/focus <target>`  | ผูกเธรดปัจจุบัน (หรือสร้างเธรด) เข้ากับเป้าหมาย sub-agent/เซสชัน                     |
| `/unfocus`         | ลบการผูกสำหรับเธรดปัจจุบันที่ผูกไว้                                           |
| `/agents`          | แสดงรายการการรันที่ใช้งานอยู่และสถานะการผูก (`binding:<id>`, `unbound` หรือ `bindings unavailable`) |
| `/session idle`    | ตรวจสอบ/อัปเดตการยกเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน (เฉพาะเธรดที่ผูกและอยู่ในโฟกัส)                             |
| `/session max-age` | ตรวจสอบ/อัปเดตขีดจำกัดสูงสุดแบบตายตัว (เฉพาะเธรดที่ผูกและอยู่ในโฟกัส)                                      |

### สวิตช์การกำหนดค่า

- **ค่าเริ่มต้นส่วนกลาง:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`
- **คีย์การแทนที่สำหรับช่องทางและการผูกอัตโนมัติเมื่อสร้าง** เป็นค่าเฉพาะของอะแดปเตอร์ ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านบน

ดูรายละเอียดอะแดปเตอร์ปัจจุบันได้ที่ [เอกสารอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference) และ
[คำสั่ง Slash](/th/tools/slash-commands)

### รายการอนุญาต

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  รายการ id ของ agent ที่กำหนดค่าไว้ซึ่งสามารถกำหนดเป็นเป้าหมายผ่าน `agentId` ที่ระบุอย่างชัดเจนได้ (`["*"]` อนุญาตเป้าหมายที่กำหนดค่าไว้ทั้งหมด) ค่าเริ่มต้น: เฉพาะ agent ผู้ร้องขอ หากกำหนดรายการและยังต้องการให้ผู้ร้องขอสร้างตัวเองด้วย `agentId` ให้รวม id ของผู้ร้องขอไว้ในรายการ
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  รายการอนุญาตเริ่มต้นของ agent เป้าหมายที่กำหนดค่าไว้ ซึ่งใช้เมื่อ agent ผู้ร้องขอไม่ได้กำหนด `subagents.allowAgents` ของตนเอง
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  บล็อกการเรียก `sessions_spawn` ที่ไม่ได้ระบุ `agentId` (บังคับให้เลือกโปรไฟล์อย่างชัดเจน) การแทนที่ราย agent: `agents.list[].subagents.requireAgentId`
</ParamField>
<ParamField path="agents.defaults.subagents.announceTimeoutMs" type="number" default="120000">
  การหมดเวลาต่อการเรียกสำหรับความพยายามส่งประกาศ `agent` ของ Gateway ค่าเป็นจำนวนเต็มบวกในหน่วยมิลลิวินาทีและจะถูกจำกัดไว้ที่ค่าสูงสุดของตัวจับเวลาที่ปลอดภัยสำหรับแพลตฟอร์ม การลองใหม่จากข้อผิดพลาดชั่วคราวอาจทำให้เวลารอประกาศรวมยาวนานกว่าค่าการหมดเวลาที่กำหนดไว้หนึ่งครั้ง
</ParamField>

หากเซสชันผู้ร้องขออยู่ในแซนด์บ็อกซ์ `sessions_spawn` จะปฏิเสธเป้าหมาย
ที่จะทำงานนอกแซนด์บ็อกซ์

### การค้นหา

ใช้ `agents_list` เพื่อดูว่า id ของเอเจนต์ใดได้รับอนุญาตสำหรับ
`sessions_spawn` อยู่ในขณะนี้ การตอบกลับประกอบด้วยโมเดลที่มีผลจริงและ
ข้อมูลเมตารันไทม์แบบฝังของเอเจนต์แต่ละรายการ เพื่อให้ผู้เรียกแยกแยะ OpenClaw, app-server ของ Codex
และรันไทม์เนทีฟอื่นๆ ที่กำหนดค่าไว้ได้

รายการ `allowAgents` ต้องชี้ไปยัง id ของเอเจนต์ที่กำหนดค่าไว้ใน `agents.list[]`
`["*"]` หมายถึงเอเจนต์เป้าหมายที่กำหนดค่าไว้ทั้งหมดรวมถึงผู้ร้องขอ หากลบการกำหนดค่าเอเจนต์
แต่ id ของเอเจนต์นั้นยังคงอยู่ใน `allowAgents` ระบบ `sessions_spawn` จะปฏิเสธ id นั้น
และ `agents_list` จะไม่รวม id ดังกล่าว เรียกใช้ `openclaw doctor --fix` เพื่อล้าง
รายการ allowlist ที่ล้าสมัย หรือเพิ่มรายการ `agents.list[]` ขั้นต่ำเมื่อเป้าหมายควร
ยังคงสามารถถูกสร้างได้โดยสืบทอดค่าเริ่มต้น

### การเก็บถาวรอัตโนมัติ

- เซสชันของเอเจนต์ย่อยจะถูกเก็บถาวรโดยอัตโนมัติหลังจาก `agents.defaults.subagents.archiveAfterMinutes` (ค่าเริ่มต้น `60`)
- การเก็บถาวรใช้ `sessions.delete` และเปลี่ยนชื่อทรานสคริปต์เป็น `*.deleted.<timestamp>` (โฟลเดอร์เดียวกัน)
- `cleanup: "delete"` จะเก็บถาวรทันทีหลังการประกาศ (ยังคงเก็บทรานสคริปต์ไว้ด้วยการเปลี่ยนชื่อ)
- การเก็บถาวรอัตโนมัติเป็นแบบพยายามให้ดีที่สุด ตัวจับเวลาที่ยังค้างอยู่จะสูญหายหาก Gateway เริ่มทำงานใหม่
- ระยะหมดเวลาการทำงานที่กำหนดค่าไว้จะ **ไม่** เก็บถาวรโดยอัตโนมัติ แต่จะหยุดเฉพาะการทำงานเท่านั้น เซสชันจะคงอยู่จนกว่าจะมีการเก็บถาวรอัตโนมัติ
- การเก็บถาวรอัตโนมัติใช้กับทั้งเซสชันระดับความลึก 1 และระดับความลึก 2 อย่างเท่าเทียมกัน
- การล้างเบราว์เซอร์แยกจากการล้างข้อมูลเก็บถาวร: ระบบจะพยายามปิดแท็บ/โพรเซสของเบราว์เซอร์ที่ติดตามไว้เมื่อการทำงานเสร็จสิ้น แม้ว่าจะยังเก็บทรานสคริปต์/ระเบียนเซสชันไว้ก็ตาม

## เอเจนต์ย่อยแบบซ้อน

ตามค่าเริ่มต้น เอเจนต์ย่อยไม่สามารถสร้างเอเจนต์ย่อยของตนเองได้
(`maxSpawnDepth: 1`) ตั้งค่า `maxSpawnDepth: 2` เพื่อเปิดใช้งานการซ้อนหนึ่งระดับ
ซึ่งเป็น **รูปแบบออร์เคสเตรเตอร์**: เอเจนต์หลัก → เอเจนต์ย่อยออร์เคสเตรเตอร์ →
เอเจนต์ย่อยระดับถัดไปที่เป็นเวิร์กเกอร์

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // อนุญาตให้เอเจนต์ย่อยสร้างลูก (ค่าเริ่มต้น: 1, ช่วง 1-5)
        maxChildrenPerAgent: 5, // จำนวนลูกที่ทำงานพร้อมกันสูงสุดต่อเซสชันเอเจนต์ (ค่าเริ่มต้น: 5, ช่วง 1-20)
        maxConcurrent: 8, // ขีดจำกัดเลนการทำงานพร้อมกันทั่วระบบ (ค่าเริ่มต้น: 8)
        runTimeoutSeconds: 900, // ระยะหมดเวลาเริ่มต้นสำหรับ sessions_spawn (0 = ไม่หมดเวลา)
        announceTimeoutMs: 120000, // ระยะหมดเวลาการประกาศของ Gateway ต่อการเรียก
      },
    },
  },
}
```

### ระดับความลึก

| ความลึก | รูปแบบคีย์เซสชัน                            | บทบาท                                          | สร้างลูกได้หรือไม่                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | เอเจนต์หลัก                                    | ได้เสมอ                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | เอเจนต์ย่อย (เป็นออร์เคสเตรเตอร์เมื่ออนุญาตระดับความลึก 2) | เฉพาะเมื่อ `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | เอเจนต์ย่อยระดับถัดไป (เวิร์กเกอร์ปลายทาง)                   | ไม่ได้                        |

### ลำดับการประกาศ

ผลลัพธ์จะไหลย้อนกลับขึ้นตามลำดับ:

1. เวิร์กเกอร์ระดับความลึก 2 ทำงานเสร็จ → ประกาศไปยังเอเจนต์แม่ (ออร์เคสเตรเตอร์ระดับความลึก 1)
2. ออร์เคสเตรเตอร์ระดับความลึก 1 รับประกาศ สังเคราะห์ผลลัพธ์ และทำงานเสร็จ → ประกาศไปยังเอเจนต์หลัก
3. เอเจนต์หลักรับประกาศและส่งมอบให้ผู้ใช้

แต่ละระดับจะเห็นเฉพาะประกาศจากลูกโดยตรงของตนเท่านั้น

<Note>
**คำแนะนำด้านการปฏิบัติงาน:** เริ่มงานของลูกหนึ่งครั้งแล้วรอเหตุการณ์
การทำงานเสร็จ แทนการสร้างลูปโพลรอบ `sessions_list`,
`sessions_history`, `/subagents list` หรือคำสั่งพัก `exec`
`sessions_list` และ `/subagents list` ทำให้ความสัมพันธ์ของเซสชันลูก
มุ่งเน้นที่งานที่กำลังดำเนินอยู่ โดยลูกที่ยังทำงานจะยังคงเชื่อมโยงอยู่ ลูกที่สิ้นสุดแล้วจะยัง
มองเห็นได้ในช่วงเวลาล่าสุดระยะสั้น และลิงก์ลูกที่ล้าสมัยซึ่งมีอยู่เฉพาะในที่เก็บ
จะถูกเพิกเฉยหลังพ้นช่วงความสดใหม่ วิธีนี้ป้องกันไม่ให้ข้อมูลเมตา `spawnedBy` /
`parentSessionKey` เก่าทำให้ลูกเงากลับมาปรากฏอีกหลัง
เริ่มทำงานใหม่ หากเหตุการณ์การทำงานเสร็จของลูกมาถึงหลังจากส่ง
คำตอบสุดท้ายไปแล้ว การติดตามผลที่ถูกต้องคือโทเค็นแบบเงียบที่ตรงกันทุกประการ
`NO_REPLY` / `no_reply`
</Note>

### นโยบายเครื่องมือตามระดับความลึก

- ระบบจะเขียนบทบาทและขอบเขตการควบคุมลงในข้อมูลเมตาของเซสชันขณะสร้าง ซึ่งป้องกันไม่ให้คีย์เซสชันแบบแบนหรือที่กู้คืนแล้วได้รับสิทธิ์ออร์เคสเตรเตอร์กลับคืนโดยไม่ตั้งใจ
- **ระดับความลึก 1 (ออร์เคสเตรเตอร์ เมื่อ `maxSpawnDepth >= 2`):** ได้รับ `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` เพื่อให้สามารถสร้างลูกและตรวจสอบสถานะได้ เครื่องมือเซสชัน/ระบบอื่นๆ ยังคงถูกปฏิเสธ
- **ระดับความลึก 1 (ปลายทาง เมื่อ `maxSpawnDepth == 1`):** ไม่มีเครื่องมือเซสชัน (พฤติกรรมเริ่มต้นในปัจจุบัน)
- **ระดับความลึก 2 (เวิร์กเกอร์ปลายทาง):** ไม่มีเครื่องมือเซสชัน โดย `sessions_spawn` จะถูกปฏิเสธเสมอที่ระดับความลึก 2 และไม่สามารถสร้างลูกเพิ่มเติมได้

### ขีดจำกัดการสร้างต่อเอเจนต์

แต่ละเซสชันเอเจนต์ (ที่ระดับความลึกใดๆ) สามารถมีลูกที่ทำงานพร้อมกันได้ไม่เกิน `maxChildrenPerAgent`
(ค่าเริ่มต้น `5`) วิธีนี้ป้องกันการแตกแขนงที่ควบคุมไม่ได้
จากออร์เคสเตรเตอร์ตัวเดียว

### การหยุดแบบต่อเนื่อง

การหยุดออร์เคสเตรเตอร์ระดับความลึก 1 จะหยุดลูกระดับความลึก 2
ทั้งหมดโดยอัตโนมัติ:

- `/stop` ในแชตหลักจะหยุดเอเจนต์ระดับความลึก 1 ทั้งหมดและส่งต่อการหยุดไปยังลูกระดับความลึก 2

## การยืนยันตัวตน

การยืนยันตัวตนของเอเจนต์ย่อยจะได้รับการแก้ไขตาม **id ของเอเจนต์** ไม่ใช่ประเภทเซสชัน:

- คีย์เซสชันของเอเจนต์ย่อยคือ `agent:<agentId>:subagent:<uuid>`
- ระบบโหลดที่เก็บข้อมูลยืนยันตัวตนจาก `agentDir` ของเอเจนต์นั้น
- ระบบจะผสานโปรไฟล์การยืนยันตัวตนของเอเจนต์หลักเข้ามาเป็น **ทางเลือกสำรอง** โดยโปรไฟล์ของเอเจนต์จะมีสิทธิ์เหนือโปรไฟล์หลักเมื่อเกิดข้อขัดแย้ง

การผสานเป็นแบบเพิ่มเติม ดังนั้นโปรไฟล์หลักจึงพร้อมใช้งานเป็น
ทางเลือกสำรองเสมอ ขณะนี้ยังไม่รองรับการแยกการยืนยันตัวตนอย่างสมบูรณ์ต่อเอเจนต์

## การประกาศ

เอเจนต์ย่อยรายงานกลับผ่านขั้นตอนการประกาศ:

- ขั้นตอนการประกาศทำงานภายในเซสชันเอเจนต์ย่อย (ไม่ใช่เซสชันของผู้ร้องขอ)
- หากเอเจนต์ย่อยตอบกลับตรงกับ `ANNOUNCE_SKIP` ทุกประการ ระบบจะไม่โพสต์สิ่งใด
- หากข้อความล่าสุดของผู้ช่วยเป็นโทเค็นแบบเงียบที่ตรงกันทุกประการ `NO_REPLY` / `no_reply` ผลลัพธ์การประกาศจะถูกระงับ แม้ว่าก่อนหน้านั้นจะมีความคืบหน้าที่มองเห็นได้ก็ตาม

การส่งมอบขึ้นอยู่กับระดับความลึกของผู้ร้องขอ:

- เซสชันผู้ร้องขอระดับบนสุดใช้การเรียก `agent` เพื่อติดตามผลพร้อมการส่งมอบภายนอก (`deliver=true`)
- เซสชันเอเจนต์ย่อยของผู้ร้องขอแบบซ้อนจะได้รับการแทรกข้อมูลติดตามผลภายใน (`deliver=false`) เพื่อให้ออร์เคสเตรเตอร์สามารถสังเคราะห์ผลลัพธ์ของลูกภายในเซสชันได้
- หากเซสชันเอเจนต์ย่อยของผู้ร้องขอแบบซ้อนไม่มีอยู่แล้ว OpenClaw จะใช้ผู้ร้องขอของเซสชันนั้นเป็นทางเลือกสำรองเมื่อมีข้อมูล

สำหรับเซสชันผู้ร้องขอระดับบนสุด การส่งมอบโดยตรงในโหมดเสร็จสิ้นจะ
แก้ไขเส้นทางการสนทนา/เธรดที่ผูกไว้และการแทนที่ด้วยฮุกก่อน จากนั้นจึงเติม
ฟิลด์ช่องทาง-เป้าหมายที่ขาดหายจากเส้นทางที่จัดเก็บไว้ของเซสชันผู้ร้องขอ
วิธีนี้ทำให้การทำงานเสร็จสิ้นยังคงอยู่ในแชต/หัวข้อที่ถูกต้อง แม้ว่าต้นทางของการทำงานเสร็จสิ้น
จะระบุเพียงช่องทางเท่านั้น

การรวมผลการทำงานเสร็จของลูกจะจำกัดขอบเขตไว้ที่การทำงานปัจจุบันของผู้ร้องขอเมื่อ
สร้างข้อค้นพบการทำงานเสร็จแบบซ้อน เพื่อป้องกันไม่ให้ผลลัพธ์ของลูกจากการทำงานก่อนหน้า
ที่ล้าสมัยรั่วไหลเข้าสู่การประกาศปัจจุบัน การตอบกลับการประกาศจะรักษา
การกำหนดเส้นทางเธรด/หัวข้อไว้เมื่ออะแดปเตอร์ช่องทางรองรับ

### บริบทการประกาศ

บริบทการประกาศจะถูกปรับให้อยู่ในบล็อกเหตุการณ์ภายในที่เสถียร:

| ฟิลด์          | แหล่งที่มา                                                                                                   |
| -------------- | -------------------------------------------------------------------------------------------------------- |
| แหล่งที่มา         | `subagent` หรือ `cron`                                                                                     |
| id เซสชัน    | คีย์/id เซสชันลูก                                                                                     |
| ประเภท           | ประเภทการประกาศ + ป้ายกำกับงาน                                                                               |
| สถานะ         | ได้มาจากผลลัพธ์รันไทม์ (`ok`, `error`, `timeout` หรือ `unknown`) — **ไม่ได้** อนุมานจากข้อความของโมเดล |
| เนื้อหาผลลัพธ์ | ข้อความล่าสุดที่มองเห็นได้ของผู้ช่วยจากลูก                                                             |
| การติดตามผล      | คำสั่งที่อธิบายว่าเมื่อใดควรตอบกลับหรือคงความเงียบ                                                      |

การทำงานที่ล้มเหลวในสถานะสิ้นสุดจะรายงานสถานะล้มเหลวโดยไม่เล่นซ้ำ
ข้อความตอบกลับที่บันทึกไว้ ผลลัพธ์ tool/toolResult จะไม่ถูกเลื่อนขึ้นเป็นข้อความผลลัพธ์ของลูก

### บรรทัดสถิติ

เพย์โหลดการประกาศมีบรรทัดสถิติที่ส่วนท้าย (แม้เมื่อถูกตัดบรรทัด):

- รันไทม์ (เช่น `runtime 5m12s`)
- การใช้โทเค็น (อินพุต/เอาต์พุต/รวม)
- ค่าใช้จ่ายโดยประมาณเมื่อกำหนดราคาของโมเดลไว้ (`models.providers.*.models[].cost`)
- `sessionKey`, `sessionId` และพาธของทรานสคริปต์ เพื่อให้เอเจนต์หลักสามารถดึงประวัติผ่าน `sessions_history` หรือตรวจสอบไฟล์บนดิสก์ได้

ข้อมูลเมตาภายในมีไว้สำหรับการประสานงานเท่านั้น การตอบกลับที่ผู้ใช้มองเห็น
ควรเขียนใหม่ด้วยน้ำเสียงปกติของผู้ช่วย

### เหตุผลที่ควรใช้ `sessions_history`

`sessions_history` เป็นเส้นทางการประสานงานที่ปลอดภัยกว่าสำหรับการอ่าน
ทรานสคริปต์ของลูกจากภายในเทิร์นของเอเจนต์:

- ปกปิดข้อความที่มีลักษณะคล้ายข้อมูลประจำตัว/โทเค็น แม้จะปิดใช้งานการปกปิดบันทึกทั่วไป
- ตัดบล็อกข้อความยาวให้สั้นลง (4000 อักขระต่อบล็อก) และละทิ้งลายเซ็นการคิด เพย์โหลดการเล่นซ้ำเหตุผล และข้อมูลรูปภาพแบบอินไลน์
- บังคับใช้ขีดจำกัดการตอบกลับ 80 KB โดยแทนที่แถวที่มีขนาดเกินด้วย `[sessions_history omitted: message too large]`
- ใช้ `nextOffset` เมื่อมี เพื่อเลื่อนย้อนกลับไปยังช่วงทรานสคริปต์ที่เก่ากว่า
- `sessions_history` **ไม่** ลบแท็กการให้เหตุผล โครงร่าง `<relevant-memories>` หรือ XML การเรียกเครื่องมือออกจากข้อความ โดยจะส่งคืนบล็อกเนื้อหาที่มีโครงสร้างใกล้เคียงกับรูปแบบทรานสคริปต์ดิบ เพียงแต่ผ่านการปกปิดและจำกัดขนาดแล้ว ส่วน `/subagents log` ใช้ตัวทำความสะอาดร้อยแก้วที่เข้มงวดกว่า (ลบแท็กการให้เหตุผล โครงร่างหน่วยความจำ และ XML การเรียกเครื่องมือ) เนื่องจากแสดงผลเป็นบรรทัดแชตข้อความธรรมดาแทนบล็อกที่มีโครงสร้าง
- การตรวจสอบทรานสคริปต์ดิบบนดิสก์เป็นทางเลือกสำรองเมื่อจำเป็นต้องใช้ทรานสคริปต์ฉบับเต็มที่ตรงกันทุกไบต์

## นโยบายเครื่องมือ

เอเจนต์ย่อยใช้โปรไฟล์และไปป์ไลน์นโยบายเครื่องมือเดียวกับเอเจนต์แม่หรือ
เอเจนต์เป้าหมายก่อน จากนั้น OpenClaw จึงใช้ชั้นข้อจำกัดของเอเจนต์ย่อย

เอเจนต์ย่อยจะสูญเสีย `gateway`, `agents_list`, `session_status` และ
`cron` เสมอโดยไม่คำนึงถึงระดับความลึกหรือบทบาท (เครื่องมือระดับระบบ/แบบโต้ตอบ หรือ
เครื่องมือที่เอเจนต์หลักควรประสานงาน) เอเจนต์ย่อยปลายทาง (พฤติกรรมเริ่มต้นที่ระดับความลึก 1
และเสมอที่ระดับความลึก 2) จะสูญเสีย `subagents`,
`sessions_list`, `sessions_history` และ `sessions_spawn` เพิ่มเติม เอเจนต์ย่อยจะไม่ได้รับ
เครื่องมือ `message` เลย โดยเครื่องมือนี้ถูกปิดใช้งานขณะสร้าง ไม่ได้ถูกกรองด้วย
รายการปฏิเสธนี้ และ `sessions_send` ยังคงถูกปฏิเสธเพื่อให้เอเจนต์ย่อย
สื่อสารผ่านลำดับการประกาศเท่านั้น

`sessions_history` ยังคงเป็นมุมมองการเรียกคืนที่มีขอบเขตและผ่านการทำความสะอาดในที่นี้ด้วย
ไม่ใช่การถ่ายโอนทรานสคริปต์ดิบ

เมื่อ `maxSpawnDepth >= 2` เอเจนต์ย่อยออร์เคสเตรเตอร์ระดับความลึก 1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ
`sessions_history` เพิ่มเติม เพื่อให้สามารถจัดการลูกของตนได้

### แทนที่ผ่านการกำหนดค่า

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // การปฏิเสธมีสิทธิ์เหนือกว่า
        deny: ["gateway", "cron"],
        // หากตั้งค่า allow รายการนี้จะกลายเป็นอนุญาตเฉพาะที่ระบุ (การปฏิเสธยังคงมีสิทธิ์เหนือกว่า)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` เป็นตัวกรองขั้นสุดท้ายที่อนุญาตเฉพาะรายการที่กำหนด ตัวกรองนี้สามารถจำกัด
ชุดเครื่องมือที่ได้รับการกำหนดแล้วให้แคบลงได้ แต่ไม่สามารถ **เพิ่มกลับ** เครื่องมือที่
`tools.profile` นำออกไปแล้วได้ ตัวอย่างเช่น `tools.profile: "coding"` มี
`web_search`/`web_fetch` แต่ไม่มีเครื่องมือ `browser` หากต้องการให้
เอเจนต์ย่อยของโปรไฟล์การเขียนโค้ดใช้ระบบอัตโนมัติของเบราว์เซอร์ได้ ให้เพิ่ม browser ใน
ขั้นตอนโปรไฟล์:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

ใช้ `agents.list[].tools.alsoAllow: ["browser"]` แบบต่อเอเจนต์ เมื่อต้องการให้มีเพียงเอเจนต์เดียว
ที่ได้รับระบบอัตโนมัติของเบราว์เซอร์

## การทำงานพร้อมกัน

เอเจนต์ย่อยใช้เลนคิวเฉพาะภายในโปรเซส:

- **ชื่อเลน:** `subagent`
- **จำนวนงานพร้อมกัน:** `agents.defaults.subagents.maxConcurrent` (ค่าเริ่มต้น `8`)

## การตรวจสอบสถานะการทำงานและการกู้คืน

OpenClaw ไม่ถือว่าการไม่มี `endedAt` เป็นหลักฐานถาวรว่า
เอเจนต์ย่อยยังคงทำงานอยู่ งานที่ยังไม่สิ้นสุดและเก่ากว่าช่วงเวลาที่ถือว่างานค้าง
(2 ชั่วโมง หรือระยะหมดเวลาของงานที่กำหนดค่าไว้บวกช่วงผ่อนผันสั้น ๆ
แล้วแต่ว่าค่าใดนานกว่า) จะหยุดถูกนับว่ากำลังทำงาน/รอดำเนินการใน `/subagents list`,
สรุปสถานะ การควบคุมการเสร็จสมบูรณ์ของเอเจนต์สืบทอด และการตรวจสอบ
จำนวนงานพร้อมกันต่อเซสชัน

หลังจาก Gateway เริ่มทำงานใหม่ งานที่กู้คืนซึ่งยังไม่สิ้นสุดและค้างอยู่จะถูกล้างออก เว้นแต่
เซสชันลูกของงานนั้นจะมีเครื่องหมาย `abortedLastRun: true` งานที่ถูกยกเลิกเนื่องจาก
การเริ่มทำงานใหม่จะยังคงลงทะเบียนไว้สำหรับขั้นตอนการกู้คืนเอเจนต์ย่อยที่กำพร้า โดยงานที่ค้าง
จะถูกทำให้สิ้นสุดโดยไม่มีการทำงานต่อ ส่วนเซสชันลูกที่ยังใหม่จะได้รับข้อความจำลองให้ทำงานต่อ
ก่อนที่เครื่องหมายยกเลิกจะถูกล้าง

การกู้คืนอัตโนมัติหลังการเริ่มทำงานใหม่มีขีดจำกัดต่อเซสชันลูก หากเอเจนต์ย่อยลูกเดียวกัน
ได้รับการยอมรับให้กู้คืนสถานะกำพร้าซ้ำ ๆ ภายในช่วงเวลาที่เกิดการค้างซ้ำอย่างรวดเร็ว
OpenClaw จะบันทึก recovery tombstone ไว้ในเซสชันนั้น และหยุดทำงานต่อโดยอัตโนมัติ
ในการเริ่มทำงานใหม่ครั้งถัดไป เรียกใช้ `openclaw tasks maintenance --apply` เพื่อปรับระเบียนงานให้สอดคล้องกัน หรือ
`openclaw doctor --fix` เพื่อล้างแฟล็กการกู้คืนที่ถูกยกเลิกและค้างอยู่ใน
เซสชันที่มี tombstone

<Note>
หากการสร้างเอเจนต์ย่อยล้มเหลวพร้อม Gateway `PAIRING_REQUIRED` /
`scope-upgrade` ให้ตรวจสอบผู้เรียก RPC ก่อนแก้ไขสถานะการจับคู่
การส่งงานประสานงานภายใน `sessions_spawn` จะดำเนินการภายในโปรเซส เมื่อ
ผู้เรียกกำลังทำงานอยู่ในบริบทคำขอของ Gateway อยู่แล้ว จึงไม่
เปิด WebSocket แบบ loopback หรือพึ่งพาขอบเขตพื้นฐานของอุปกรณ์ที่จับคู่ไว้ของ CLI
ผู้เรียกที่อยู่นอกโปรเซส Gateway ยังคงใช้ WebSocket
สำรองเป็น `client.id: "gateway-client"` พร้อม `client.mode: "backend"`
ผ่านการยืนยันตัวตนด้วยโทเค็นร่วม/รหัสผ่านแบบ loopback โดยตรง ผู้เรียกระยะไกล การระบุ
`deviceIdentity` อย่างชัดเจน เส้นทางโทเค็นอุปกรณ์ที่ระบุอย่างชัดเจน และไคลเอนต์ browser/node
ยังคงต้องได้รับการอนุมัติอุปกรณ์ตามปกติสำหรับการเพิ่มขอบเขต
</Note>

## การหยุด

- การส่ง `/stop` ในแชตของผู้ร้องขอจะยกเลิกเซสชันของผู้ร้องขอ และหยุดงานเอเจนต์ย่อยที่กำลังทำงานทั้งหมดซึ่งสร้างจากเซสชันนั้น โดยส่งผลต่อเนื่องไปยังเอเจนต์ลูกที่ซ้อนกัน

## ข้อจำกัด

- การประกาศของเอเจนต์ย่อยดำเนินการแบบ **พยายามให้ดีที่สุด** หาก Gateway เริ่มทำงานใหม่ งาน "announce back" ที่รอดำเนินการจะสูญหาย
- เอเจนต์ย่อยยังคงใช้ทรัพยากรโปรเซส Gateway เดียวกันร่วมกัน ให้ถือว่า `maxConcurrent` เป็นกลไกป้องกันความเสียหาย
- `sessions_spawn` จะไม่บล็อกเสมอ โดยส่งคืน `{ status: "accepted", runId, childSessionKey }` ทันที
- บริบทของเอเจนต์ย่อยจะแทรกเฉพาะ `AGENTS.md` และ `TOOLS.md` (ไม่มี `SOUL.md`, `IDENTITY.md`, `USER.md`, `MEMORY.md`, `HEARTBEAT.md` หรือ `BOOTSTRAP.md`) เอเจนต์ย่อยแบบเนทีฟของ Codex ใช้ขอบเขตเดียวกัน โดย `TOOLS.md` จะยังคงอยู่ในคำสั่งของเธรด Codex ที่สืบทอดมา ส่วนไฟล์บุคลิก ตัวตน และผู้ใช้ที่มีเฉพาะในเอเจนต์แม่จะถูกแทรกเป็นคำสั่งการทำงานร่วมกันที่มีขอบเขตเฉพาะเทิร์น เพื่อไม่ให้เอเจนต์ลูกทำสำเนาไฟล์เหล่านั้น
- ความลึกสูงสุดของการซ้อนคือ 5 (ช่วงของ `maxSpawnDepth`: 1-5) แนะนำให้ใช้ความลึก 2 สำหรับกรณีใช้งานส่วนใหญ่
- `maxChildrenPerAgent` จำกัดจำนวนเอเจนต์ลูกที่ทำงานอยู่ต่อเซสชัน (ค่าเริ่มต้น `5`, ช่วง `1-20`)
