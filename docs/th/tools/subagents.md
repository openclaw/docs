เอเจนต์ย่อยคือการรันเอเจนต์เบื้องหลังที่ถูกสร้างจากการรันเอเจนต์ที่มีอยู่
เอเจนต์เหล่านี้รันในเซสชันของตัวเอง (`agent:<agentId>:subagent:<uuid>`) และ
เมื่อเสร็จแล้วจะ**ประกาศ**ผลลัพธ์กลับไปยังช่องทางแชตของผู้ร้องขอ
การรันเอเจนต์ย่อยแต่ละครั้งจะถูกติดตามเป็น
[งานเบื้องหลัง](/th/automation/tasks)

เป้าหมายหลัก:

- ทำให้งาน "วิจัย / งานยาว / เครื่องมือที่ช้า" ทำงานขนานกันได้โดยไม่บล็อกการรันหลัก
- แยกเอเจนต์ย่อยออกจากกันตามค่าเริ่มต้น (แยกเซสชัน + เลือกใช้แซนด์บ็อกซ์ได้)
- ทำให้พื้นที่เครื่องมือใช้งานผิดพลาดได้ยาก: เอเจนต์ย่อยจะไม่ได้รับเครื่องมือเซสชันตามค่าเริ่มต้น
- รองรับความลึกของการซ้อนที่กำหนดค่าได้สำหรับรูปแบบ orchestrator

<Note>
**หมายเหตุเรื่องค่าใช้จ่าย:** เอเจนต์ย่อยแต่ละตัวมีบริบทและการใช้โทเคนของตัวเองตาม
ค่าเริ่มต้น สำหรับงานหนักหรืองานซ้ำ ๆ ให้ตั้งค่าโมเดลที่ถูกลงสำหรับเอเจนต์ย่อย
และให้เอเจนต์หลักใช้โมเดลคุณภาพสูงกว่า กำหนดค่าผ่าน
`agents.defaults.subagents.model` หรือการแทนที่รายเอเจนต์ เมื่อลูก
    จำเป็นต้องใช้ทรานสคริปต์ปัจจุบันของผู้ร้องขอจริง ๆ เอเจนต์สามารถขอ
    `context: "fork"` ในการสร้างครั้งนั้นได้ เซสชันเอเจนต์ย่อยที่ผูกกับเธรดมีค่าเริ่มต้นเป็น
    `context: "fork"` เพราะจะแตกบทสนทนาปัจจุบันออกไปเป็น
    เธรดติดตามผล
</Note>

## คำสั่ง Slash

ใช้ `/subagents` เพื่อตรวจสอบหรือควบคุมการรันเอเจนต์ย่อยสำหรับ**เซสชัน
ปัจจุบัน**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

ใช้ [`/steer <message>`](/th/tools/steer) ระดับบนสุดเพื่อควบคุมการรันที่ใช้งานอยู่ของเซสชันผู้ร้องขอปัจจุบัน ใช้ `/subagents steer <id|#> <message>` เมื่อเป้าหมายคือการรันลูก

`/subagents info` แสดงเมทาดาทาของการรัน (สถานะ, เวลา, ไอดีเซสชัน,
พาธทรานสคริปต์, การล้างข้อมูล) ใช้ `sessions_history` สำหรับมุมมองเรียกคืนแบบจำกัดขอบเขต
และผ่านตัวกรองความปลอดภัยแล้ว; ตรวจสอบพาธทรานสคริปต์บนดิสก์เมื่อคุณ
ต้องการทรานสคริปต์ฉบับเต็มแบบดิบ

### การควบคุมการผูกเธรด

คำสั่งเหล่านี้ทำงานกับช่องทางที่รองรับการผูกเธรดแบบถาวร
ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านล่าง

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### พฤติกรรมการสร้าง

`/subagents spawn` เริ่มเอเจนต์ย่อยเบื้องหลังเป็นคำสั่งของผู้ใช้ (ไม่ใช่
รีเลย์ภายใน) และส่งการอัปเดตการเสร็จสิ้นครั้งสุดท้ายกลับไปยังแชตของ
ผู้ร้องขอเมื่อการรันเสร็จ

<AccordionGroup>
  <Accordion title="Non-blocking, push-based completion">
    - คำสั่งสร้างไม่บล็อก; จะส่งคืนไอดีการรันทันที
    - เมื่อเสร็จ เอเจนต์ย่อยจะประกาศข้อความสรุป/ผลลัพธ์กลับไปยังช่องทางแชตของผู้ร้องขอ
    - การเสร็จสิ้นเป็นแบบ push-based เมื่อสร้างแล้ว อย่าวน poll `/subagents list`, `sessions_list`, หรือ `sessions_history` เพียงเพื่อรอให้เสร็จ; ตรวจสอบสถานะเฉพาะเมื่อต้องการเพื่อดีบักหรือแทรกแซงเท่านั้น
    - เมื่อเสร็จ OpenClaw จะพยายามอย่างดีที่สุดเพื่อปิดแท็บเบราว์เซอร์/โปรเซสที่ติดตามไว้ซึ่งเปิดโดยเซสชันเอเจนต์ย่อยนั้น ก่อนที่โฟลว์ล้างข้อมูลการประกาศจะดำเนินต่อ

  </Accordion>
  <Accordion title="Manual-spawn delivery resilience">
    - OpenClaw ส่งการเสร็จสิ้นกลับไปยังเซสชันผู้ร้องขอผ่านเทิร์น `agent` ด้วยคีย์ idempotency ที่เสถียร
    - หากการรันของผู้ร้องขอยังใช้งานอยู่ OpenClaw จะพยายามปลุก/ควบคุมการรันนั้นก่อน แทนที่จะเริ่มเส้นทางตอบกลับที่มองเห็นได้เส้นที่สอง
    - หากการส่งต่อการเสร็จสิ้นไปยังเอเจนต์ผู้ร้องขอล้มเหลวหรือไม่มีเอาต์พุตที่มองเห็นได้ OpenClaw จะถือว่าการส่งมอบล้มเหลวและถอยกลับไปใช้การกำหนดเส้นทางคิว/ลองใหม่ โดยจะไม่ส่งผลลัพธ์ดิบของลูกไปยังแชตภายนอกโดยตรง
    - หากใช้การส่งต่อโดยตรงไม่ได้ จะถอยกลับไปใช้การกำหนดเส้นทางคิว
    - หากการกำหนดเส้นทางคิวยังไม่พร้อมใช้งาน การประกาศจะถูกลองใหม่ด้วย exponential backoff สั้น ๆ ก่อนยอมแพ้สุดท้าย
    - การส่งมอบการเสร็จสิ้นจะเก็บเส้นทางผู้ร้องขอที่ resolve แล้วไว้: เส้นทางการเสร็จสิ้นแบบผูกกับเธรดหรือผูกกับบทสนทนาจะชนะเมื่อพร้อมใช้งาน; หากต้นทางการเสร็จสิ้นให้มาเฉพาะช่องทาง OpenClaw จะเติมเป้าหมาย/บัญชีที่ขาดหายจากเส้นทางที่ resolve แล้วของเซสชันผู้ร้องขอ (`lastChannel` / `lastTo` / `lastAccountId`) เพื่อให้การส่งมอบโดยตรงยังทำงานได้

    - ใช้ `info`/`log` เพื่อตรวจสอบรายละเอียดและเอาต์พุตหลังจากเสร็จ
    - `/subagents spawn` เป็นโหมดครั้งเดียว (`mode: "run"`) สำหรับเซสชันถาวรที่ผูกกับเธรด ให้ใช้ `sessions_spawn` พร้อม `thread: true` และ `mode: "session"`

  </Accordion>
</AccordionGroup>

## โหมดบริบท

เอเจนต์ย่อยแบบเนทีฟจะเริ่มแบบแยกเดี่ยว เว้นแต่ผู้เรียกจะขอ fork
ทรานสคริปต์ปัจจุบันอย่างชัดเจน

| โหมด       | ควรใช้เมื่อใด                                                                                                                         | พฤติกรรม                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | การวิจัยใหม่, การทำงานอิสระ, งานเครื่องมือที่ช้า, หรือสิ่งใดก็ตามที่สามารถสรุปในข้อความงานได้                           | สร้างทรานสคริปต์ลูกใหม่สะอาด นี่คือค่าเริ่มต้นและช่วยลดการใช้โทเคน  |
| `fork`     | งานที่ขึ้นกับบทสนทนาปัจจุบัน, ผลลัพธ์เครื่องมือก่อนหน้า, หรือคำสั่งที่ละเอียดอ่อนซึ่งมีอยู่แล้วในทรานสคริปต์ของผู้ร้องขอ | แตกทรานสคริปต์ของผู้ร้องขอไปยังเซสชันลูกก่อนที่ลูกจะเริ่ม |

ใช้ `fork` อย่างประหยัด มีไว้สำหรับการมอบหมายงานที่ไวต่อบริบท ไม่ใช่
สิ่งทดแทนการเขียนพรอมป์งานให้ชัดเจน

## เครื่องมือ: `sessions_spawn`

เริ่มการรันเอเจนต์ย่อยด้วย `deliver: false` บนเลน `subagent` ทั่วโลก
จากนั้นรันขั้นตอนประกาศและโพสต์การตอบกลับประกาศไปยังช่องทางแชตของผู้ร้องขอ

ความพร้อมใช้งานขึ้นกับนโยบายเครื่องมือที่มีผลของผู้เรียก โปรไฟล์ `coding` และ
`full` เปิดเผย `sessions_spawn` ตามค่าเริ่มต้น โปรไฟล์ `messaging`
ไม่เปิดเผย; เพิ่ม `tools.alsoAllow: ["sessions_spawn", "sessions_yield",
"subagents"]` หรือใช้ `tools.profile: "coding"` สำหรับเอเจนต์ที่ควรมอบหมาย
งานได้ นโยบายอนุญาต/ปฏิเสธของช่องทาง/กลุ่ม, ผู้ให้บริการ, แซนด์บ็อกซ์, และรายเอเจนต์
ยังสามารถถอดเครื่องมือออกหลังขั้นโปรไฟล์ได้ ใช้ `/tools` จาก
เซสชันเดียวกันเพื่อยืนยันรายการเครื่องมือที่มีผล

**ค่าเริ่มต้น:**

- **โมเดล:** สืบทอดจากผู้เรียก เว้นแต่คุณตั้งค่า `agents.defaults.subagents.model` (หรือ `agents.list[].subagents.model` รายเอเจนต์); `sessions_spawn.model` ที่ระบุชัดเจนยังคงชนะ
- **Thinking:** สืบทอดจากผู้เรียก เว้นแต่คุณตั้งค่า `agents.defaults.subagents.thinking` (หรือ `agents.list[].subagents.thinking` รายเอเจนต์); `sessions_spawn.thinking` ที่ระบุชัดเจนยังคงชนะ
- **เวลาหมดของการรัน:** หากละ `sessions_spawn.runTimeoutSeconds` ไว้ OpenClaw จะใช้ `agents.defaults.subagents.runTimeoutSeconds` เมื่อถูกตั้งค่า; มิฉะนั้นจะถอยกลับไปเป็น `0` (ไม่มีเวลาหมด)

  ค่าเริ่มต้นเป็น `agents.defaults.subagents.runTimeoutSeconds` เมื่อถูกตั้งค่า มิฉะนั้นเป็น `0` เมื่อตั้งค่าไว้ การรันเอเจนต์ย่อยจะถูกยกเลิกหลังจาก N วินาที
</ParamField>
<ParamField path="thread" type="boolean" default="false">
  เมื่อเป็น `true` จะร้องขอการผูกเธรดของช่องทางสำหรับเซสชันเอเจนต์ย่อยนี้
</ParamField>
<ParamField path="mode" type='"run" | "session"' default="run">
  หาก `thread: true` และละ `mode` ไว้ ค่าเริ่มต้นจะเป็น `session` `mode: "session"` ต้องใช้ `thread: true`
</ParamField>
<ParamField path="cleanup" type='"delete" | "keep"' default="keep">
  `"delete"` เก็บถาวรทันทีหลังประกาศ (ยังคงเก็บทรานสคริปต์ไว้ผ่านการเปลี่ยนชื่อ)
</ParamField>
<ParamField path="sandbox" type='"inherit" | "require"' default="inherit">
  `require` ปฏิเสธการสร้าง เว้นแต่รันไทม์ลูกเป้าหมายอยู่ในแซนด์บ็อกซ์
</ParamField>
<ParamField path="context" type='"isolated" | "fork"' default="isolated">
  `fork` แตกทรานสคริปต์ปัจจุบันของผู้ร้องขอเข้าไปในเซสชันลูก เอเจนต์ย่อยแบบเนทีฟเท่านั้น การสร้างแบบผูกกับเธรดมีค่าเริ่มต้นเป็น `fork`; การสร้างที่ไม่ใช่เธรดมีค่าเริ่มต้นเป็น `isolated`
</ParamField>

<Warning>
`sessions_spawn` ไม่รับพารามิเตอร์การส่งมอบผ่านช่องทาง (`target`,
`channel`, `to`, `threadId`, `replyTo`, `transport`) สำหรับการส่งมอบ ให้ใช้
`message`/`sessions_send` จากการรันที่ถูกสร้าง
</Warning>

## เซสชันที่ผูกกับเธรด

เมื่อเปิดใช้การผูกเธรดสำหรับช่องทาง เอเจนต์ย่อยสามารถคงการผูกกับ
เธรดไว้ได้ เพื่อให้ข้อความติดตามผลของผู้ใช้ในเธรดนั้นยังคงถูกกำหนดเส้นทางไปยัง
เซสชันเอเจนต์ย่อยเดียวกัน

### ช่องทางที่รองรับเธรด

**Discord** เป็นช่องทางเดียวที่รองรับในปัจจุบัน รองรับ
เซสชันเอเจนต์ย่อยแบบผูกกับเธรดถาวร (`sessions_spawn` พร้อม
`thread: true`), การควบคุมเธรดแบบแมนนวล (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`), และคีย์อะแดปเตอร์
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours`, และ
`channels.discord.threadBindings.spawnSessions`

### โฟลว์แบบเร็ว

<Steps>
  <Step title="สร้าง">
    `sessions_spawn` พร้อม `thread: true` (และเลือกใช้ `mode: "session"` ได้)
  </Step>
  <Step title="ผูก">
    OpenClaw สร้างหรือผูกเธรดกับเป้าหมาย session นั้นในแชนเนลที่ใช้งานอยู่
  </Step>
  <Step title="กำหนดเส้นทางข้อความต่อเนื่อง">
    การตอบกลับและข้อความต่อเนื่องในเธรดนั้นจะถูกกำหนดเส้นทางไปยัง session ที่ผูกไว้
  </Step>
  <Step title="ตรวจสอบการหมดเวลา">
    ใช้ `/session idle` เพื่อตรวจสอบ/อัปเดตการเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน และ
    `/session max-age` เพื่อควบคุมเพดานเวลาสูงสุดแบบตายตัว
  </Step>
  <Step title="แยกออก">
    ใช้ `/unfocus` เพื่อแยกออกด้วยตนเอง
  </Step>
</Steps>

### การควบคุมด้วยตนเอง

| คำสั่ง             | ผลลัพธ์                                                              |
| ------------------ | --------------------------------------------------------------------- |
| `/focus <target>`  | ผูกเธรดปัจจุบัน (หรือสร้างเธรดใหม่) กับเป้าหมายเอเจนต์ย่อย/session |
| `/unfocus`         | ลบการผูกสำหรับเธรดที่ผูกอยู่ปัจจุบัน                                |
| `/agents`          | แสดงรายการงานที่กำลังทำงานและสถานะการผูก (`thread:<id>` หรือ `unbound`) |
| `/session idle`    | ตรวจสอบ/อัปเดตการเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน (เฉพาะเธรดที่ผูกและโฟกัสอยู่) |
| `/session max-age` | ตรวจสอบ/อัปเดตเพดานเวลาสูงสุดแบบตายตัว (เฉพาะเธรดที่ผูกและโฟกัสอยู่) |

### สวิตช์การกำหนดค่า

- **ค่าเริ่มต้นส่วนกลาง:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **คีย์สำหรับการแทนที่ตามแชนเนลและการผูกอัตโนมัติเมื่อสร้าง** เป็นค่าเฉพาะของ adapter ดู [แชนเนลที่รองรับเธรด](#thread-supporting-channels) ด้านบน

ดู [ข้อมูลอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference) และ
[คำสั่ง Slash](/th/tools/slash-commands) สำหรับรายละเอียด adapter ปัจจุบัน

### Allowlist

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  รายการรหัสเอเจนต์ที่สามารถกำหนดเป็นเป้าหมายผ่าน `agentId` แบบระบุชัดเจน (`["*"]` อนุญาตทุกตัว) ค่าเริ่มต้น: เฉพาะเอเจนต์ผู้ร้องขอ หากคุณตั้งค่ารายการและยังต้องการให้ผู้ร้องขอสร้างตัวเองด้วย `agentId` ให้ใส่รหัสผู้ร้องขอไว้ในรายการด้วย
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  Allowlist เอเจนต์เป้าหมายเริ่มต้นที่ใช้เมื่อเอเจนต์ผู้ร้องขอไม่ได้ตั้งค่า `subagents.allowAgents` ของตัวเอง
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  บล็อกการเรียก `sessions_spawn` ที่ละ `agentId` (บังคับให้เลือกโปรไฟล์แบบระบุชัดเจน) การแทนที่ต่อเอเจนต์: `agents.list[].subagents.requireAgentId`
</ParamField>

หาก session ของผู้ร้องขอถูก sandbox, `sessions_spawn` จะปฏิเสธเป้าหมาย
ที่จะทำงานแบบไม่ถูก sandbox

### การค้นหา

ใช้ `agents_list` เพื่อดูว่ารหัสเอเจนต์ใดที่ได้รับอนุญาตให้ใช้กับ
`sessions_spawn` ในปัจจุบัน การตอบกลับจะรวม model ที่มีผลจริงของเอเจนต์แต่ละตัวในรายการ
และ metadata runtime ที่ฝังไว้ เพื่อให้ผู้เรียกสามารถแยกความแตกต่างระหว่าง PI, Codex
app-server และ runtime แบบ native อื่นที่กำหนดค่าไว้ได้

### การเก็บถาวรอัตโนมัติ

- Session ของเอเจนต์ย่อยจะถูกเก็บถาวรโดยอัตโนมัติหลังจาก `agents.defaults.subagents.archiveAfterMinutes` (ค่าเริ่มต้น `60`)
- การเก็บถาวรใช้ `sessions.delete` และเปลี่ยนชื่อ transcript เป็น `*.deleted.<timestamp>` (โฟลเดอร์เดียวกัน)
- `cleanup: "delete"` จะเก็บถาวรทันทีหลัง announce (ยังคงเก็บ transcript ไว้ผ่านการเปลี่ยนชื่อ)
- การเก็บถาวรอัตโนมัติเป็นแบบ best-effort; timer ที่ค้างอยู่จะหายไปหาก Gateway รีสตาร์ท
- `runTimeoutSeconds` จะไม่เก็บถาวรอัตโนมัติ; เพียงหยุดการรันเท่านั้น session จะยังคงอยู่จนกว่าจะเก็บถาวรอัตโนมัติ
- การเก็บถาวรอัตโนมัติใช้กับ session ระดับความลึก 1 และระดับความลึก 2 อย่างเท่าเทียมกัน
- การล้างข้อมูลเบราว์เซอร์แยกจากการล้างข้อมูลเก็บถาวร: แท็บ/โปรเซสเบราว์เซอร์ที่ติดตามไว้จะถูกปิดแบบ best-effort เมื่อการรันเสร็จ แม้ว่า transcript/ระเบียน session จะยังถูกเก็บไว้ก็ตาม

## เอเจนต์ย่อยแบบซ้อน

ตามค่าเริ่มต้น เอเจนต์ย่อยไม่สามารถสร้างเอเจนต์ย่อยของตัวเองได้
(`maxSpawnDepth: 1`) ตั้งค่า `maxSpawnDepth: 2` เพื่อเปิดใช้การซ้อนหนึ่งระดับ
ซึ่งคือ **รูปแบบ orchestrator**: หลัก → เอเจนต์ย่อย orchestrator →
เอเจนต์ย่อยของเอเจนต์ย่อยแบบ worker

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // allow sub-agents to spawn children (default: 1)
        maxChildrenPerAgent: 5, // max active children per agent session (default: 5)
        maxConcurrent: 8, // global concurrency lane cap (default: 8)
        runTimeoutSeconds: 900, // default timeout for sessions_spawn when omitted (0 = no timeout)
      },
    },
  },
}
```

### ระดับความลึก

| ความลึก | รูปแบบคีย์ session                         | บทบาท                                        | สร้างต่อได้หรือไม่          |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | เอเจนต์หลัก                                  | ได้เสมอ                      |
| 1     | `agent:<id>:subagent:<uuid>`                 | เอเจนต์ย่อย (orchestrator เมื่ออนุญาตความลึก 2) | เฉพาะเมื่อ `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | เอเจนต์ย่อยของเอเจนต์ย่อย (leaf worker)     | ไม่ได้เลย                    |

### ลำดับ announce

ผลลัพธ์จะไหลย้อนขึ้นไปตามลำดับ:

1. Worker ระดับความลึก 2 เสร็จสิ้น → announce ไปยัง parent ของตน (orchestrator ระดับความลึก 1)
2. Orchestrator ระดับความลึก 1 ได้รับ announce, สังเคราะห์ผลลัพธ์, เสร็จสิ้น → announce ไปยังหลัก
3. เอเจนต์หลักได้รับ announce และส่งมอบให้ผู้ใช้

แต่ละระดับจะเห็นเฉพาะ announce จากลูกโดยตรงของตัวเองเท่านั้น

<Note>
**คำแนะนำด้านปฏิบัติการ:** เริ่มงานลูกหนึ่งครั้งและรอเหตุการณ์เสร็จสิ้น
แทนการสร้าง poll loop รอบ `sessions_list`,
`sessions_history`, `/subagents list` หรือคำสั่ง `exec` sleep
`sessions_list` และ `/subagents list` ทำให้ความสัมพันธ์ session ลูก
โฟกัสอยู่กับงานสด — ลูกที่ยังทำงานอยู่จะยังแนบอยู่, ลูกที่จบแล้วจะยังมองเห็นได้
ในหน้าต่างเวลาล่าสุดช่วงสั้น ๆ และลิงก์ลูกที่มีอยู่เฉพาะใน store และค้างเก่าจะถูก
ละเว้นหลังพ้นหน้าต่างความสดใหม่ วิธีนี้ป้องกันไม่ให้ metadata `spawnedBy` /
`parentSessionKey` เก่าฟื้นลูกที่ไม่มีอยู่จริงหลัง
รีสตาร์ท หากเหตุการณ์เสร็จสิ้นของลูกมาถึงหลังจากคุณส่ง
คำตอบสุดท้ายไปแล้ว การติดตามผลที่ถูกต้องคือโทเคนเงียบแบบตรงตัว
`NO_REPLY` / `no_reply`
</Note>

### นโยบายเครื่องมือตามความลึก

- บทบาทและขอบเขตการควบคุมถูกเขียนลงใน metadata ของ session ตอนสร้าง วิธีนี้ป้องกันไม่ให้คีย์ session แบบแบนหรือที่กู้คืนกลับมาได้รับสิทธิ์ orchestrator โดยไม่ตั้งใจ
- **ความลึก 1 (orchestrator, เมื่อ `maxSpawnDepth >= 2`):** ได้รับ `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` เพื่อให้จัดการลูกของตัวเองได้ เครื่องมือ session/system อื่นยังคงถูกปฏิเสธ
- **ความลึก 1 (leaf, เมื่อ `maxSpawnDepth == 1`):** ไม่มีเครื่องมือ session (พฤติกรรมเริ่มต้นปัจจุบัน)
- **ความลึก 2 (leaf worker):** ไม่มีเครื่องมือ session — `sessions_spawn` ถูกปฏิเสธเสมอที่ความลึก 2 ไม่สามารถสร้างลูกเพิ่มได้

### ขีดจำกัดการสร้างต่อเอเจนต์

แต่ละ session ของเอเจนต์ (ไม่ว่าที่ความลึกใด) สามารถมีลูกที่ใช้งานอยู่ได้สูงสุด `maxChildrenPerAgent`
(ค่าเริ่มต้น `5`) พร้อมกัน วิธีนี้ป้องกันการกระจายงานเกินควบคุม
จาก orchestrator ตัวเดียว

### การหยุดแบบ cascade

การหยุด orchestrator ระดับความลึก 1 จะหยุดลูกระดับความลึก 2
ทั้งหมดโดยอัตโนมัติ:

- `/stop` ในแชตหลักจะหยุดเอเจนต์ระดับความลึก 1 ทั้งหมดและ cascade ไปยังลูกระดับความลึก 2 ของพวกเขา
- `/subagents kill <id>` หยุดเอเจนต์ย่อยเฉพาะตัวและ cascade ไปยังลูกของมัน
- `/subagents kill all` หยุดเอเจนต์ย่อยทั้งหมดของผู้ร้องขอและ cascade

## การยืนยันตัวตน

Auth ของเอเจนต์ย่อยจะถูก resolve ตาม **รหัสเอเจนต์** ไม่ใช่ตามชนิด session:

- คีย์ session ของเอเจนต์ย่อยคือ `agent:<agentId>:subagent:<uuid>`
- Auth store ถูกโหลดจาก `agentDir` ของเอเจนต์นั้น
- Auth profile ของเอเจนต์หลักจะถูก merge เข้ามาเป็น **fallback**; profile ของเอเจนต์จะแทนที่ profile หลักเมื่อมี conflict

การ merge เป็นแบบเพิ่มเท่านั้น ดังนั้น profile หลักจึงพร้อมใช้งานเป็น
fallback เสมอ ยังไม่รองรับ auth ที่แยกโดดเดี่ยวต่อเอเจนต์อย่างสมบูรณ์

## Announce

เอเจนต์ย่อยรายงานกลับผ่านขั้นตอน announce:

- ขั้นตอน announce ทำงานภายใน session ของเอเจนต์ย่อย (ไม่ใช่ session ของผู้ร้องขอ)
- หากเอเจนต์ย่อยตอบกลับตรงตัวว่า `ANNOUNCE_SKIP` จะไม่มีการโพสต์อะไร
- หากข้อความ assistant ล่าสุดเป็นโทเคนเงียบแบบตรงตัว `NO_REPLY` / `no_reply` เอาต์พุต announce จะถูกระงับแม้เคยมีความคืบหน้าที่มองเห็นได้ก่อนหน้า

การส่งมอบขึ้นอยู่กับความลึกของผู้ร้องขอ:

- Session ผู้ร้องขอระดับบนสุดใช้การเรียก `agent` แบบติดตามผลพร้อมการส่งมอบภายนอก (`deliver=true`)
- Session เอเจนต์ย่อยของผู้ร้องขอที่ซ้อนอยู่จะได้รับการฉีดข้อมูลติดตามผลภายใน (`deliver=false`) เพื่อให้ orchestrator สังเคราะห์ผลลัพธ์ลูกใน session ได้
- หาก session เอเจนต์ย่อยของผู้ร้องขอที่ซ้อนอยู่หายไป OpenClaw จะ fallback ไปยังผู้ร้องขอของ session นั้นเมื่อพร้อมใช้งาน

สำหรับ session ผู้ร้องขอระดับบนสุด การส่งมอบโดยตรงใน completion-mode จะ
resolve เส้นทาง conversation/thread ที่ผูกไว้และ hook override ก่อน จากนั้นเติม
ฟิลด์เป้าหมายแชนเนลที่ขาดหายจากเส้นทางที่เก็บไว้ใน session ของผู้ร้องขอ
วิธีนี้ทำให้ completion อยู่ในแชต/topic ที่ถูกต้อง แม้ต้นทาง completion
จะระบุเฉพาะแชนเนลก็ตาม

การรวม completion ของลูกถูกจำกัดขอบเขตไว้ที่การรันผู้ร้องขอปัจจุบันเมื่อ
สร้าง finding ของ completion แบบซ้อน เพื่อป้องกันไม่ให้เอาต์พุตลูกจากการรันก่อนหน้าที่ค้างเก่า
รั่วไหลเข้ามาใน announce ปัจจุบัน การตอบกลับ announce จะรักษา
เส้นทาง thread/topic เมื่อพร้อมใช้งานบน adapter ของแชนเนล

### บริบท announce

บริบท announce ถูก normalize เป็นบล็อกเหตุการณ์ภายในที่เสถียร:

| ฟิลด์          | แหล่งที่มา                                                                                                      |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| แหล่งที่มา     | `subagent` หรือ `cron`                                                                                         |
| รหัส session   | คีย์/id ของ session ลูก                                                                                        |
| ชนิด           | ชนิด announce + ป้ายกำกับงาน                                                                                   |
| สถานะ          | อนุมานจากผลลัพธ์ runtime (`success`, `error`, `timeout` หรือ `unknown`) — **ไม่ใช่** อนุมานจากข้อความ model |
| เนื้อหาผลลัพธ์ | ข้อความ assistant ล่าสุดที่มองเห็นได้ ไม่เช่นนั้นเป็นข้อความ tool/toolResult ล่าสุดที่ผ่านการ sanitize แล้ว |
| การติดตามผล    | คำสั่งที่อธิบายว่าเมื่อใดควรตอบกลับหรือควรเงียบ                                                            |

การรันที่ล้มเหลวขั้นสุดท้ายจะรายงานสถานะล้มเหลวโดยไม่เล่นซ้ำข้อความ
ตอบกลับที่จับไว้ เมื่อ timeout หากลูกไปได้เพียงถึงการเรียก tool, announce
สามารถยุบประวัตินั้นเป็นสรุปความคืบหน้าบางส่วนแบบสั้น แทน
การเล่นซ้ำเอาต์พุต tool ดิบ

### บรรทัดสถิติ

Payload ของ announce รวมบรรทัดสถิติไว้ท้ายสุด (แม้เมื่อถูก wrap):

- Runtime (เช่น `runtime 5m12s`)
- การใช้ token (input/output/total)
- ค่าใช้จ่ายโดยประมาณเมื่อกำหนดราคา model ไว้ (`models.providers.*.models[].cost`)
- `sessionKey`, `sessionId` และเส้นทาง transcript เพื่อให้เอเจนต์หลักดึงประวัติผ่าน `sessions_history` หรือตรวจสอบไฟล์บนดิสก์ได้

Metadata ภายในมีไว้สำหรับ orchestration เท่านั้น; การตอบกลับที่แสดงต่อผู้ใช้
ควรถูกเขียนใหม่ด้วยน้ำเสียง assistant ปกติ

### เหตุผลที่ควรใช้ `sessions_history`

`sessions_history` เป็นเส้นทาง orchestration ที่ปลอดภัยกว่า:

- การ recall ของ assistant ถูก normalize ก่อน: ลบ thinking tag; ลบ scaffolding `<relevant-memories>` / `<relevant_memories>`; ลบบล็อก payload XML ของการเรียก tool แบบ plain-text (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) รวมถึง payload ที่ถูกตัดทอนซึ่งไม่เคยปิดอย่างสมบูรณ์; ลบ scaffolding การเรียก/ผลลัพธ์ tool ที่ถูกลดระดับและ marker ของ historical-context; ลบ token ควบคุม model ที่รั่ว (`<|assistant|>`, ASCII `<|...|>` อื่น, full-width `<｜...｜>`); ลบ XML การเรียก tool ของ MiniMax ที่ malformed
- ข้อความที่ดูเหมือน credential/token ถูก redact
- บล็อกยาวอาจถูกตัดทอน
- ประวัติขนาดใหญ่มากอาจตัดแถวเก่าทิ้งหรือแทนที่แถวที่ใหญ่เกินด้วย `[sessions_history omitted: message too large]`
- การตรวจสอบ transcript ดิบบนดิสก์เป็น fallback เมื่อคุณต้องการ transcript แบบ byte-for-byte เต็มรูปแบบ

## นโยบายเครื่องมือ

ซับเอเจนต์ใช้โปรไฟล์และไปป์ไลน์นโยบายเครื่องมือเดียวกับเอเจนต์แม่หรือ
เอเจนต์เป้าหมายก่อน หลังจากนั้น OpenClaw จะใช้ชั้นข้อจำกัดของซับเอเจนต์

เมื่อไม่มี `tools.profile` ที่จำกัด ซับเอเจนต์จะได้รับ **เครื่องมือทั้งหมด ยกเว้น
เครื่องมือเซสชัน** และเครื่องมือระบบ:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

`sessions_history` ยังคงเป็นมุมมองเรียกคืนที่จำกัดขอบเขตและผ่านการทำความสะอาดที่นี่ด้วย —
ไม่ใช่การทิ้งทรานสคริปต์ดิบ

เมื่อ `maxSpawnDepth >= 2` ซับเอเจนต์ตัวประสานงาน depth-1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ
`sessions_history` เพิ่มเติม เพื่อให้จัดการลูกของตนได้

### แทนที่ผ่านการกำหนดค่า

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny wins
        deny: ["gateway", "cron"],
        // if allow is set, it becomes allow-only (deny still wins)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` เป็นตัวกรองแบบ allow-only ขั้นสุดท้าย โดยสามารถจำกัด
ชุดเครื่องมือที่แก้ไขเรียบร้อยแล้วให้แคบลงได้ แต่ไม่สามารถ **เพิ่มกลับ** เครื่องมือที่ถูกลบออก
โดย `tools.profile` ได้ ตัวอย่างเช่น `tools.profile: "coding"` รวม
`web_search`/`web_fetch` แต่ไม่รวมเครื่องมือ `browser` หากต้องการให้
ซับเอเจนต์โปรไฟล์ coding ใช้ browser automation ได้ ให้เพิ่ม browser ใน
ขั้นตอนโปรไฟล์:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

ใช้ `agents.list[].tools.alsoAllow: ["browser"]` แบบต่อเอเจนต์ เมื่อมีเพียงเอเจนต์เดียว
ที่ควรได้รับ browser automation

## การทำงานพร้อมกัน

ซับเอเจนต์ใช้เลนคิวในกระบวนการเฉพาะ:

- **ชื่อเลน:** `subagent`
- **การทำงานพร้อมกัน:** `agents.defaults.subagents.maxConcurrent` (ค่าเริ่มต้น `8`)

## สถานะมีชีวิตและการกู้คืน

OpenClaw ไม่ถือว่าการไม่มี `endedAt` เป็นหลักฐานถาวรว่า
ซับเอเจนต์ยังมีชีวิตอยู่ รันที่ยังไม่สิ้นสุดซึ่งเก่ากว่าหน้าต่าง stale-run
จะหยุดนับเป็น active/pending ใน `/subagents list`, สรุปสถานะ,
การกั้นการเสร็จสิ้นของลูกหลาน และการตรวจสอบการทำงานพร้อมกันต่อเซสชัน

หลังจาก Gateway รีสตาร์ต รันที่กู้คืนมาและยังไม่สิ้นสุดซึ่งค้างเก่าจะถูกตัดออก เว้นแต่
เซสชันลูกของรันนั้นถูกทำเครื่องหมาย `abortedLastRun: true` เซสชันลูกที่ถูกยกเลิกจาก
การรีสตาร์ตเหล่านั้นยังคงกู้คืนได้ผ่านโฟลว์การกู้คืนซับเอเจนต์ที่กลายเป็นกำพร้า ซึ่งจะส่งข้อความ resume สังเคราะห์ก่อน
ล้างเครื่องหมาย aborted

การกู้คืนอัตโนมัติหลังรีสตาร์ตมีขอบเขตจำกัดต่อเซสชันลูก หากลูกของซับเอเจนต์เดียวกัน
ถูกยอมรับสำหรับการกู้คืนกำพร้าซ้ำ ๆ ภายในหน้าต่าง rapid re-wedge,
OpenClaw จะบันทึก tombstone การกู้คืนไว้บนเซสชันนั้น และหยุด auto-resume ให้ในการรีสตาร์ตครั้งถัดไป รัน
`openclaw tasks maintenance --apply` เพื่อปรับข้อมูลระเบียนงานให้สอดคล้องกัน หรือ
`openclaw doctor --fix` เพื่อล้างแฟล็กการกู้คืน aborted ที่ค้างอยู่บน
เซสชันที่ถูก tombstone

<Note>
หากการ spawn ซับเอเจนต์ล้มเหลวด้วย Gateway `PAIRING_REQUIRED` /
`scope-upgrade` ให้ตรวจสอบ RPC caller ก่อนแก้ไขสถานะการจับคู่
การประสานงาน `sessions_spawn` ภายในควรเชื่อมต่อเป็น
`client.id: "gateway-client"` พร้อม `client.mode: "backend"` ผ่าน
การตรวจสอบสิทธิ์ shared-token/password แบบ direct loopback เส้นทางดังกล่าวไม่ขึ้นกับ
ขอบเขตพื้นฐาน paired-device ของ CLI ผู้เรียกระยะไกล,
`deviceIdentity` แบบระบุชัดเจน, เส้นทาง device-token แบบระบุชัดเจน และไคลเอนต์ browser/node
ยังต้องได้รับการอนุมัติอุปกรณ์ตามปกติสำหรับการอัปเกรดขอบเขต
</Note>

## การหยุด

- การส่ง `/stop` ในแชตผู้ร้องขอจะยกเลิกเซสชันผู้ร้องขอ และหยุดรันของซับเอเจนต์ที่ใช้งานอยู่ทั้งหมดซึ่ง spawn จากเซสชันนั้น โดย cascade ไปยังลูกที่ซ้อนอยู่
- `/subagents kill <id>` หยุดซับเอเจนต์ที่ระบุ และ cascade ไปยังลูกของมัน

## ข้อจำกัด

- การประกาศซับเอเจนต์เป็นแบบ **best-effort** หาก Gateway รีสตาร์ต งาน "announce back" ที่ค้างอยู่จะสูญหาย
- ซับเอเจนต์ยังคงใช้ทรัพยากรกระบวนการ Gateway เดียวกันร่วมกัน ให้ถือว่า `maxConcurrent` เป็นวาล์วนิรภัย
- `sessions_spawn` เป็นแบบไม่บล็อกเสมอ: จะคืนค่า `{ status: "accepted", runId, childSessionKey }` ทันที
- บริบทซับเอเจนต์ฉีดเฉพาะ `AGENTS.md` + `TOOLS.md` (ไม่มี `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` หรือ `BOOTSTRAP.md`)
- ความลึกการซ้อนสูงสุดคือ 5 (ช่วง `maxSpawnDepth`: 1–5) แนะนำให้ใช้ความลึก 2 สำหรับกรณีใช้งานส่วนใหญ่
- `maxChildrenPerAgent` จำกัดจำนวนลูกที่ใช้งานอยู่ต่อเซสชัน (ค่าเริ่มต้น `5`, ช่วง `1–20`)
