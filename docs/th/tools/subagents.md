เอเจนต์ย่อยคือการรันเอเจนต์เบื้องหลังที่ถูกสร้างจากการรันเอเจนต์ที่มีอยู่
เอเจนต์เหล่านี้รันในเซสชันของตนเอง (`agent:<agentId>:subagent:<uuid>`) และ
เมื่อเสร็จแล้วจะ **ประกาศ** ผลลัพธ์กลับไปยังช่องแชตของผู้ร้องขอ
การรันเอเจนต์ย่อยแต่ละครั้งถูกติดตามเป็น
[งานพื้นหลัง](/th/automation/tasks)

เป้าหมายหลัก:

- ทำให้งาน "ค้นคว้า / งานยาว / เครื่องมือช้า" ทำงานขนานกันได้โดยไม่บล็อกการรันหลัก
- แยกเอเจนต์ย่อยออกจากกันตามค่าเริ่มต้น (แยกเซสชัน + แซนด์บ็อกซ์แบบเลือกได้)
- ทำให้พื้นผิวเครื่องมือใช้งานผิดได้ยาก: เอเจนต์ย่อยจะ **ไม่ได้** เครื่องมือของเซสชันตามค่าเริ่มต้น
- รองรับความลึกของการซ้อนที่กำหนดค่าได้สำหรับรูปแบบตัวจัดการงาน

<Note>
**หมายเหตุเรื่องค่าใช้จ่าย:** เอเจนต์ย่อยแต่ละตัวมีคอนเท็กซ์และการใช้โทเค็นของตนเองตามค่าเริ่มต้น สำหรับงานหนักหรืองานซ้ำ ให้ตั้งโมเดลที่ถูกกว่าสำหรับเอเจนต์ย่อย และคงเอเจนต์หลักไว้บนโมเดลคุณภาพสูงกว่า กำหนดค่าผ่าน `agents.defaults.subagents.model` หรือการ override รายเอเจนต์ เมื่อเอเจนต์ลูกจำเป็นต้องใช้ transcript ปัจจุบันของผู้ร้องขอจริง ๆ เอเจนต์สามารถขอ `context: "fork"` สำหรับการ spawn ครั้งนั้นได้
</Note>

## คำสั่ง Slash

ใช้ `/subagents` เพื่อตรวจสอบหรือควบคุมการรันเอเจนต์ย่อยสำหรับ **เซสชันปัจจุบัน**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

`/subagents info` แสดงเมตาดาต้าการรัน (สถานะ, timestamps, session id,
transcript path, cleanup) ใช้ `sessions_history` สำหรับมุมมองการเรียกคืนที่จำกัดขอบเขตและกรองเพื่อความปลอดภัย ตรวจสอบ transcript path บนดิสก์เมื่อคุณต้องการ transcript แบบดิบทั้งหมด

### การควบคุมการผูกเธรด

คำสั่งเหล่านี้ทำงานบนช่องทางที่รองรับการผูกเธรดแบบถาวร
ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านล่าง

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### พฤติกรรมการ Spawn

`/subagents spawn` เริ่มเอเจนต์ย่อยเบื้องหลังเป็นคำสั่งของผู้ใช้ (ไม่ใช่ relay ภายใน) และส่งอัปเดตการเสร็จสิ้นสุดท้ายกลับไปยังแชตของผู้ร้องขอเมื่อการรันเสร็จ

<AccordionGroup>
  <Accordion title="Non-blocking, push-based completion">
    - คำสั่ง spawn ไม่บล็อก; จะคืน run id ทันที
    - เมื่อเสร็จ เอเจนต์ย่อยจะประกาศข้อความสรุป/ผลลัพธ์กลับไปยังช่องแชตของผู้ร้องขอ
    - การเสร็จสิ้นเป็นแบบ push-based เมื่อ spawn แล้ว อย่า poll `/subagents list`, `sessions_list`, หรือ `sessions_history` เป็น loop เพียงเพื่อรอให้เสร็จ; ตรวจสอบสถานะเฉพาะเมื่อต้องการสำหรับการดีบักหรือการแทรกแซง
    - เมื่อเสร็จ OpenClaw จะพยายามปิดแท็บเบราว์เซอร์/โปรเซสที่ติดตามซึ่งเปิดโดยเซสชันเอเจนต์ย่อยนั้น ก่อนที่ flow cleanup ของการประกาศจะดำเนินต่อ

  </Accordion>
  <Accordion title="Manual-spawn delivery resilience">
    - OpenClaw ลองส่งแบบ `agent` โดยตรงก่อนด้วย idempotency key ที่เสถียร
    - หากการส่งโดยตรงล้มเหลว จะ fallback ไปใช้ queue routing
    - หาก queue routing ยังไม่พร้อมใช้งาน การประกาศจะ retry ด้วย exponential backoff สั้น ๆ ก่อนยอมแพ้ขั้นสุดท้าย
    - การส่งมอบเมื่อเสร็จจะคงเส้นทางผู้ร้องขอที่ resolve แล้วไว้: เส้นทางการเสร็จสิ้นแบบ thread-bound หรือ conversation-bound จะชนะเมื่อพร้อมใช้งาน; หาก origin ของการเสร็จสิ้นให้มาแค่ channel, OpenClaw จะเติม target/account ที่ขาดจากเส้นทางที่ resolve แล้วของเซสชันผู้ร้องขอ (`lastChannel` / `lastTo` / `lastAccountId`) เพื่อให้การส่งโดยตรงยังทำงานได้

  </Accordion>
  <Accordion title="Completion handoff metadata">
    การส่งต่อการเสร็จสิ้นไปยังเซสชันผู้ร้องขอเป็นคอนเท็กซ์ภายในที่ runtime สร้างขึ้น
    (ไม่ใช่ข้อความที่ผู้ใช้เขียน) และประกอบด้วย:

    - `Result` — ข้อความตอบกลับ `assistant` ล่าสุดที่มองเห็นได้ มิฉะนั้นเป็นข้อความ tool/toolResult ล่าสุดที่ sanitize แล้ว การรันที่จบแบบล้มเหลวจะไม่นำข้อความตอบกลับที่จับไว้กลับมาใช้
    - `Status` — `completed successfully` / `failed` / `timed out` / `unknown`
    - สถิติ runtime/token แบบกระชับ
    - คำสั่งการส่งมอบที่บอกเอเจนต์ผู้ร้องขอให้เขียนใหม่ด้วยน้ำเสียง assistant ปกติ (ไม่ส่งต่อเมตาดาต้าภายในแบบดิบ)

  </Accordion>
  <Accordion title="Modes and ACP runtime">
    - `--model` และ `--thinking` override ค่าเริ่มต้นสำหรับการรันนั้นโดยเฉพาะ
    - ใช้ `info`/`log` เพื่อตรวจสอบรายละเอียดและผลลัพธ์หลังเสร็จสิ้น
    - `/subagents spawn` เป็นโหมดครั้งเดียว (`mode: "run"`) สำหรับเซสชันแบบ thread-bound ถาวร ให้ใช้ `sessions_spawn` พร้อม `thread: true` และ `mode: "session"`
    - สำหรับเซสชัน ACP harness (Claude Code, Gemini CLI, OpenCode, หรือ Codex ACP/acpx ที่ระบุอย่างชัดเจน) ให้ใช้ `sessions_spawn` พร้อม `runtime: "acp"` เมื่อเครื่องมือประกาศ runtime นั้น ดู [โมเดลการส่งมอบ ACP](/th/tools/acp-agents#delivery-model) เมื่อดีบักการเสร็จสิ้นหรือ loop เอเจนต์ถึงเอเจนต์ เมื่อเปิดใช้งาน Plugin `codex` แล้ว การควบคุมแชต/เธรด Codex ควรเลือกใช้ `/codex ...` แทน ACP เว้นแต่ผู้ใช้จะขอ ACP/acpx อย่างชัดเจน
    - OpenClaw ซ่อน `runtime: "acp"` จนกว่า ACP จะถูกเปิดใช้งาน ผู้ร้องขอไม่ได้ถูก sandbox และมี Plugin backend เช่น `acpx` โหลดอยู่ `runtime: "acp"` คาดหวัง external ACP harness id หรือรายการ `agents.list[]` ที่มี `runtime.type="acp"`; ใช้ runtime เอเจนต์ย่อยค่าเริ่มต้นสำหรับเอเจนต์ config ปกติของ OpenClaw จาก `agents_list`

  </Accordion>
</AccordionGroup>

## โหมดคอนเท็กซ์

เอเจนต์ย่อยแบบ native จะเริ่มแบบแยกเดี่ยว เว้นแต่ caller จะขอ fork
transcript ปัจจุบันอย่างชัดเจน

| โหมด       | ควรใช้เมื่อใด                                                                                                                         | พฤติกรรม                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | การค้นคว้าใหม่, การ implement แบบอิสระ, งานเครื่องมือช้า, หรือสิ่งใดก็ตามที่สามารถสรุปโจทย์ไว้ในข้อความงานได้                           | สร้าง transcript ลูกที่สะอาด นี่คือค่าเริ่มต้นและช่วยลดการใช้โทเค็น  |
| `fork`     | งานที่ขึ้นกับบทสนทนาปัจจุบัน, ผลลัพธ์เครื่องมือก่อนหน้า, หรือคำสั่งที่ละเอียดอ่อนซึ่งมีอยู่แล้วใน transcript ของผู้ร้องขอ | แตกกิ่ง transcript ของผู้ร้องขอเข้าไปในเซสชันลูกก่อนที่ลูกจะเริ่ม |

ใช้ `fork` อย่างประหยัด มีไว้สำหรับการมอบหมายงานที่ไวต่อคอนเท็กซ์ ไม่ใช่สิ่งทดแทนการเขียน prompt ของงานให้ชัดเจน

## เครื่องมือ: `sessions_spawn`

เริ่มการรันเอเจนต์ย่อยด้วย `deliver: false` บน lane `subagent` ทั่วโลก
จากนั้นรันขั้นตอนประกาศและโพสต์ข้อความตอบกลับการประกาศไปยังช่องแชตของผู้ร้องขอ

ความพร้อมใช้งานขึ้นกับนโยบายเครื่องมือที่มีผลของ caller โปรไฟล์ `coding` และ
`full` เปิดเผย `sessions_spawn` ตามค่าเริ่มต้น โปรไฟล์ `messaging`
ไม่เปิดเผย; เพิ่ม `tools.alsoAllow: ["sessions_spawn", "sessions_yield",
"subagents"]` หรือใช้ `tools.profile: "coding"` สำหรับเอเจนต์ที่ควรมอบหมายงาน
นโยบาย allow/deny ของ channel/group, provider, sandbox และรายเอเจนต์ยังสามารถลบเครื่องมือหลังขั้นตอนโปรไฟล์ได้ ใช้ `/tools` จากเซสชันเดียวกันเพื่อยืนยันรายการเครื่องมือที่มีผล

**ค่าเริ่มต้น:**

- **โมเดล:** สืบทอดจาก caller เว้นแต่คุณตั้ง `agents.defaults.subagents.model` (หรือ `agents.list[].subagents.model` รายเอเจนต์); `sessions_spawn.model` ที่ระบุชัดเจนยังคงชนะ
- **Thinking:** สืบทอดจาก caller เว้นแต่คุณตั้ง `agents.defaults.subagents.thinking` (หรือ `agents.list[].subagents.thinking` รายเอเจนต์); `sessions_spawn.thinking` ที่ระบุชัดเจนยังคงชนะ
- **Run timeout:** หากละ `sessions_spawn.runTimeoutSeconds` ไว้ OpenClaw จะใช้ `agents.defaults.subagents.runTimeoutSeconds` เมื่อกำหนดไว้; มิฉะนั้นจะ fallback เป็น `0` (ไม่มี timeout)

## เซสชันแบบ Thread-bound

เมื่อเปิดใช้งานการผูกเธรดสำหรับ channel เอเจนต์ย่อยสามารถยังคงผูกกับเธรดได้ เพื่อให้ข้อความติดตามผลของผู้ใช้ในเธรดนั้นยัง route ไปยังเซสชันเอเจนต์ย่อยเดียวกัน

### ช่องทางที่รองรับเธรด

ปัจจุบัน **Discord** เป็นช่องทางเดียวที่รองรับ รองรับ
เซสชัน subagent แบบ thread-bound ถาวร (`sessions_spawn` พร้อม
`thread: true`), การควบคุมเธรดด้วยตนเอง (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`) และ adapter keys
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours`, และ
`channels.discord.threadBindings.spawnSubagentSessions`

### Flow แบบเร็ว

<Steps>
  <Step title="Spawn">
    `sessions_spawn` พร้อม `thread: true` (และเลือกใส่ `mode: "session"` ได้)
  </Step>
  <Step title="Bind">
    OpenClaw สร้างหรือผูกเธรดกับ session target นั้นใน channel ที่ใช้งานอยู่
  </Step>
  <Step title="Route follow-ups">
    ข้อความตอบกลับและข้อความติดตามผลในเธรดนั้น route ไปยังเซสชันที่ผูกไว้
  </Step>
  <Step title="Inspect timeouts">
    ใช้ `/session idle` เพื่อตรวจสอบ/อัปเดต auto-unfocus เมื่อไม่มีการใช้งาน และ
    `/session max-age` เพื่อควบคุมเพดานแข็ง
  </Step>
  <Step title="Detach">
    ใช้ `/unfocus` เพื่อ detach ด้วยตนเอง
  </Step>
</Steps>

### การควบคุมด้วยตนเอง

| คำสั่ง            | ผลลัพธ์                                                                |
| ------------------ | --------------------------------------------------------------------- |
| `/focus <target>`  | ผูกเธรดปัจจุบัน (หรือสร้างใหม่) กับเป้าหมายเอเจนต์ย่อย/เซสชัน |
| `/unfocus`         | นำการผูกสำหรับเธรดที่ผูกไว้ในปัจจุบันออก                       |
| `/agents`          | แสดงรายการรันที่ใช้งานอยู่และสถานะการผูก (`thread:<id>` หรือ `unbound`)       |
| `/session idle`    | ตรวจสอบ/อัปเดตการยกเลิกโฟกัสอัตโนมัติเมื่อไม่ได้ใช้งาน (เฉพาะเธรดที่ผูกไว้ซึ่งถูกโฟกัส)         |
| `/session max-age` | ตรวจสอบ/อัปเดตเพดานจำกัดสูงสุดแบบเด็ดขาด (เฉพาะเธรดที่ผูกไว้ซึ่งถูกโฟกัส)                  |

### สวิตช์การกำหนดค่า

- **ค่าเริ่มต้นส่วนกลาง:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **คีย์สำหรับการแทนที่ช่องทางและการผูกอัตโนมัติเมื่อสปอว์น** เป็นแบบเฉพาะอะแดปเตอร์ ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านบน

ดู [ข้อมูลอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference) และ
[คำสั่ง Slash](/th/tools/slash-commands) สำหรับรายละเอียดอะแดปเตอร์ปัจจุบัน

### รายการอนุญาต

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  รายการ id ของเอเจนต์ที่สามารถกำหนดเป็นเป้าหมายผ่าน `agentId` แบบชัดเจน (`["*"]` อนุญาตทุกตัว) ค่าเริ่มต้น: เฉพาะเอเจนต์ผู้ร้องขอเท่านั้น หากคุณตั้งค่ารายการและยังต้องการให้ผู้ร้องขอสปอว์นตัวเองด้วย `agentId` ให้รวม id ของผู้ร้องขอไว้ในรายการ
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  รายการอนุญาตของเอเจนต์เป้าหมายเริ่มต้นที่ใช้เมื่อเอเจนต์ผู้ร้องขอไม่ได้ตั้งค่า `subagents.allowAgents` ของตัวเอง
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  บล็อกการเรียก `sessions_spawn` ที่ละเว้น `agentId` (บังคับให้เลือกโปรไฟล์แบบชัดเจน) การแทนที่รายเอเจนต์: `agents.list[].subagents.requireAgentId`
</ParamField>

หากเซสชันผู้ร้องขออยู่ในแซนด์บ็อกซ์ `sessions_spawn` จะปฏิเสธเป้าหมาย
ที่จะรันโดยไม่อยู่ในแซนด์บ็อกซ์

### การค้นพบ

ใช้ `agents_list` เพื่อดูว่า id ของเอเจนต์ใดบ้างที่ได้รับอนุญาตสำหรับ
`sessions_spawn` อยู่ในขณะนี้ การตอบกลับจะรวมโมเดลที่มีผลจริงของเอเจนต์แต่ละตัวในรายการ
และเมตาดาต้ารันไทม์ที่ฝังไว้ เพื่อให้ผู้เรียกแยกแยะ PI, Codex
app-server และรันไทม์เนทีฟอื่นๆ ที่กำหนดค่าไว้ได้

### การเก็บถาวรอัตโนมัติ

- เซสชันเอเจนต์ย่อยจะถูกเก็บถาวรโดยอัตโนมัติหลังจาก `agents.defaults.subagents.archiveAfterMinutes` (ค่าเริ่มต้น `60`)
- การเก็บถาวรใช้ `sessions.delete` และเปลี่ยนชื่อบันทึกการถอดความเป็น `*.deleted.<timestamp>` (โฟลเดอร์เดียวกัน)
- `cleanup: "delete"` จะเก็บถาวรทันทีหลังจากประกาศ (ยังคงเก็บบันทึกการถอดความไว้ผ่านการเปลี่ยนชื่อ)
- การเก็บถาวรอัตโนมัติเป็นแบบพยายามให้ดีที่สุด ตัวจับเวลาที่ค้างอยู่จะสูญหายหาก Gateway รีสตาร์ท
- `runTimeoutSeconds` ไม่ได้เก็บถาวรอัตโนมัติ แต่เพียงหยุดการรันเท่านั้น เซสชันจะยังคงอยู่จนกว่าจะถูกเก็บถาวรอัตโนมัติ
- การเก็บถาวรอัตโนมัติใช้กับเซสชันความลึก 1 และความลึก 2 เท่าๆ กัน
- การล้างเบราว์เซอร์แยกจากการล้างการเก็บถาวร: แท็บ/โปรเซสของเบราว์เซอร์ที่ติดตามไว้จะถูกปิดแบบพยายามให้ดีที่สุดเมื่อการรันเสร็จสิ้น แม้ว่าจะเก็บบันทึกการถอดความ/ระเบียนเซสชันไว้ก็ตาม

## เอเจนต์ย่อยแบบซ้อน

โดยค่าเริ่มต้น เอเจนต์ย่อยไม่สามารถสปอว์นเอเจนต์ย่อยของตัวเองได้
(`maxSpawnDepth: 1`) ตั้งค่า `maxSpawnDepth: 2` เพื่อเปิดใช้งานการซ้อนหนึ่งระดับ
ซึ่งคือ **รูปแบบออร์เคสเตรเตอร์**: หลัก → เอเจนต์ย่อยออร์เคสเตรเตอร์ →
เอเจนต์ย่อยชั้นย่อยแบบเวิร์กเกอร์

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // allow sub-agents to spawn children (default: 1)
        maxChildrenPerAgent: 5, // max active children per agent session (default: 5)
        maxConcurrent: 8, // global concurrency lane cap (default: 8)
        runTimeoutSeconds: 900, // default timeout for sessions_spawn when omitted (0 = no timeout)
      },
    },
  },
}
```

### ระดับความลึก

| ความลึก | รูปแบบคีย์เซสชัน                            | บทบาท                                          | สปอว์นได้หรือไม่                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | เอเจนต์หลัก                                    | เสมอ                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | เอเจนต์ย่อย (ออร์เคสเตรเตอร์เมื่ออนุญาตความลึก 2) | เฉพาะเมื่อ `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | เอเจนต์ย่อยชั้นย่อย (เวิร์กเกอร์ปลายทาง)                   | ไม่เคย                        |

### เชนการประกาศ

ผลลัพธ์จะไหลย้อนกลับขึ้นไปตามเชน:

1. เวิร์กเกอร์ความลึก 2 เสร็จสิ้น → ประกาศไปยังพาเรนต์ของตน (ออร์เคสเตรเตอร์ความลึก 1)
2. ออร์เคสเตรเตอร์ความลึก 1 รับประกาศ สังเคราะห์ผลลัพธ์ เสร็จสิ้น → ประกาศไปยังตัวหลัก
3. เอเจนต์หลักรับประกาศและส่งต่อให้ผู้ใช้

แต่ละระดับจะเห็นเฉพาะประกาศจากลูกโดยตรงของตัวเองเท่านั้น

<Note>
**แนวทางการปฏิบัติงาน:** เริ่มงานของลูกหนึ่งครั้งและรอเหตุการณ์การเสร็จสิ้น
แทนการสร้างลูปโพลรอบ `sessions_list`,
`sessions_history`, `/subagents list` หรือคำสั่งพักของ `exec`
`sessions_list` และ `/subagents list` คงความสัมพันธ์ของเซสชันลูก
ให้โฟกัสกับงานที่กำลังทำอยู่ โดยลูกที่ยังทำงานอยู่จะยังแนบอยู่ ลูกที่จบแล้วจะยัง
มองเห็นได้ในหน้าต่างล่าสุดช่วงสั้นๆ และลิงก์ลูกที่มีเพียงในสโตร์และล้าสมัยจะถูก
เพิกเฉยหลังจากพ้นหน้าต่างความสดใหม่ วิธีนี้ป้องกันไม่ให้เมตาดาต้า `spawnedBy` /
`parentSessionKey` เก่าปลุกลูกหลอนกลับมาหลัง
รีสตาร์ท หากเหตุการณ์เสร็จสิ้นของลูกมาถึงหลังจากคุณส่ง
คำตอบสุดท้ายไปแล้ว การติดตามผลที่ถูกต้องคือโทเคนเงียบตรงตัว
`NO_REPLY` / `no_reply`
</Note>

### นโยบายเครื่องมือตามความลึก

- บทบาทและขอบเขตการควบคุมจะถูกเขียนลงในเมตาดาต้าเซสชันตอนสปอว์น ซึ่งช่วยป้องกันไม่ให้คีย์เซสชันแบบแบนหรือที่กู้คืนมากลับได้สิทธิ์ออร์เคสเตรเตอร์โดยไม่ตั้งใจ
- **ความลึก 1 (ออร์เคสเตรเตอร์ เมื่อ `maxSpawnDepth >= 2`):** ได้รับ `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` เพื่อให้จัดการลูกของตัวเองได้ เครื่องมือเซสชัน/ระบบอื่นๆ ยังคงถูกปฏิเสธ
- **ความลึก 1 (ปลายทาง เมื่อ `maxSpawnDepth == 1`):** ไม่มีเครื่องมือเซสชัน (พฤติกรรมเริ่มต้นปัจจุบัน)
- **ความลึก 2 (เวิร์กเกอร์ปลายทาง):** ไม่มีเครื่องมือเซสชัน โดย `sessions_spawn` จะถูกปฏิเสธเสมอที่ความลึก 2 ไม่สามารถสปอว์นลูกต่อได้

### ขีดจำกัดการสปอว์นรายเอเจนต์

เซสชันเอเจนต์แต่ละรายการ (ที่ความลึกใดก็ได้) สามารถมีลูกที่ใช้งานอยู่ได้ไม่เกิน `maxChildrenPerAgent`
(ค่าเริ่มต้น `5`) ในแต่ละครั้ง วิธีนี้ป้องกันการแตกแขนงเกินควบคุม
จากออร์เคสเตรเตอร์ตัวเดียว

### การหยุดแบบลูกโซ่

การหยุดออร์เคสเตรเตอร์ความลึก 1 จะหยุดลูกความลึก 2 ทั้งหมดของมันโดยอัตโนมัติ:

- `/stop` ในแชตหลักจะหยุดเอเจนต์ความลึก 1 ทั้งหมดและส่งผลต่อเนื่องไปยังลูกความลึก 2 ของพวกมัน
- `/subagents kill <id>` หยุดเอเจนต์ย่อยที่ระบุและส่งผลต่อเนื่องไปยังลูกของมัน
- `/subagents kill all` หยุดเอเจนต์ย่อยทั้งหมดสำหรับผู้ร้องขอและส่งผลต่อเนื่อง

## การรับรองความถูกต้อง

การรับรองความถูกต้องของเอเจนต์ย่อยจะถูกแก้ไขตาม **id ของเอเจนต์** ไม่ใช่ตามประเภทเซสชัน:

- คีย์เซสชันเอเจนต์ย่อยคือ `agent:<agentId>:subagent:<uuid>`
- สโตร์การรับรองความถูกต้องถูกโหลดจาก `agentDir` ของเอเจนต์นั้น
- โปรไฟล์การรับรองความถูกต้องของเอเจนต์หลักจะถูกรวมเข้ามาเป็น **ตัวสำรอง** โปรไฟล์ของเอเจนต์จะแทนที่โปรไฟล์หลักเมื่อมีข้อขัดแย้ง

การรวมเป็นแบบเพิ่มเติม ดังนั้นโปรไฟล์หลักจึงพร้อมใช้งานเป็น
ตัวสำรองเสมอ ยังไม่รองรับการรับรองความถูกต้องที่แยกอย่างสมบูรณ์ต่อเอเจนต์

## การประกาศ

เอเจนต์ย่อยรายงานกลับผ่านขั้นตอนการประกาศ:

- ขั้นตอนการประกาศรันภายในเซสชันเอเจนต์ย่อย (ไม่ใช่เซสชันผู้ร้องขอ)
- หากเอเจนต์ย่อยตอบกลับตรงตัวว่า `ANNOUNCE_SKIP` จะไม่มีการโพสต์สิ่งใด
- หากข้อความผู้ช่วยล่าสุดเป็นโทเคนเงียบตรงตัว `NO_REPLY` / `no_reply` เอาต์พุตการประกาศจะถูกระงับ แม้ว่าก่อนหน้านั้นจะมีความคืบหน้าที่มองเห็นได้ก็ตาม

การส่งมอบขึ้นอยู่กับความลึกของผู้ร้องขอ:

- เซสชันผู้ร้องขอระดับบนสุดใช้การเรียกติดตามผล `agent` พร้อมการส่งมอบภายนอก (`deliver=true`)
- เซสชันเอเจนต์ย่อยผู้ร้องขอแบบซ้อนจะได้รับการฉีดติดตามผลภายใน (`deliver=false`) เพื่อให้ออร์เคสเตรเตอร์สังเคราะห์ผลลัพธ์ของลูกภายในเซสชันได้
- หากเซสชันเอเจนต์ย่อยผู้ร้องขอแบบซ้อนหายไป OpenClaw จะถอยกลับไปยังผู้ร้องขอของเซสชันนั้นเมื่อพร้อมใช้งาน

สำหรับเซสชันผู้ร้องขอระดับบนสุด การส่งมอบโดยตรงในโหมดเสร็จสิ้นจะ
แก้ไขเส้นทางบทสนทนา/เธรดที่ผูกไว้และการแทนที่ฮุคก่อน จากนั้นเติม
ฟิลด์เป้าหมายช่องทางที่ขาดหายจากเส้นทางที่จัดเก็บไว้ของเซสชันผู้ร้องขอ
วิธีนี้ทำให้การเสร็จสิ้นอยู่ในแชต/หัวข้อที่ถูกต้อง แม้ว่าต้นทางการเสร็จสิ้น
จะระบุเพียงช่องทางก็ตาม

การรวมผลการเสร็จสิ้นของลูกถูกจำกัดขอบเขตไว้ที่การรันของผู้ร้องขอปัจจุบันเมื่อ
สร้างข้อค้นพบการเสร็จสิ้นแบบซ้อน เพื่อป้องกันไม่ให้เอาต์พุตลูกจากการรันก่อนหน้า
ที่ล้าสมัยรั่วเข้ามาในประกาศปัจจุบัน การตอบกลับประกาศจะรักษา
เส้นทางเธรด/หัวข้อเมื่อมีให้ใช้บนอะแดปเตอร์ช่องทาง

### บริบทการประกาศ

บริบทการประกาศถูกทำให้เป็นปกติเป็นบล็อกเหตุการณ์ภายในที่เสถียร:

| ฟิลด์          | แหล่งที่มา                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| แหล่งที่มา         | `subagent` หรือ `cron`                                                                                          |
| id เซสชัน    | คีย์/id เซสชันลูก                                                                                          |
| ประเภท           | ประเภทการประกาศ + ป้ายกำกับงาน                                                                                    |
| สถานะ         | อนุมานจากผลลัพธ์รันไทม์ (`success`, `error`, `timeout` หรือ `unknown`) — **ไม่ใช่** อนุมานจากข้อความโมเดล |
| เนื้อหาผลลัพธ์ | ข้อความผู้ช่วยล่าสุดที่มองเห็นได้ มิฉะนั้นใช้ข้อความเครื่องมือ/toolResult ล่าสุดที่ผ่านการทำให้ปลอดภัย                                |
| การติดตามผล      | คำสั่งที่อธิบายว่าเมื่อใดควรตอบกลับเทียบกับนิ่งเงียบ                                                           |

การรันที่ล้มเหลวแบบปลายทางจะรายงานสถานะล้มเหลวโดยไม่เล่นซ้ำ
ข้อความตอบกลับที่จับไว้ เมื่อหมดเวลา หากลูกทำได้เพียงผ่านการเรียกเครื่องมือ
ประกาศสามารถยุบประวัตินั้นเป็นสรุปความคืบหน้าบางส่วนสั้นๆ แทน
การเล่นเอาต์พุตเครื่องมือดิบซ้ำ

### บรรทัดสถิติ

เพย์โหลดการประกาศรวมบรรทัดสถิติไว้ท้ายสุด (แม้เมื่อมีการห่อบรรทัด):

- รันไทม์ (เช่น `runtime 5m12s`)
- การใช้โทเคน (อินพุต/เอาต์พุต/รวม)
- ค่าใช้จ่ายโดยประมาณเมื่อกำหนดค่าราคาโมเดลไว้ (`models.providers.*.models[].cost`)
- `sessionKey`, `sessionId` และพาธบันทึกการถอดความ เพื่อให้เอเจนต์หลักดึงประวัติผ่าน `sessions_history` หรือตรวจสอบไฟล์บนดิสก์ได้

เมตาดาต้าภายในมีไว้สำหรับการออร์เคสเตรตเท่านั้น การตอบกลับที่ผู้ใช้เห็น
ควรถูกเขียนใหม่ด้วยน้ำเสียงผู้ช่วยตามปกติ

### เหตุผลที่ควรใช้ `sessions_history`

`sessions_history` เป็นเส้นทางการออร์เคสเตรตที่ปลอดภัยกว่า:

- การเรียกคืนของผู้ช่วยถูกทำให้เป็นปกติก่อน: ลบแท็กการคิด; ลบโครง `<relevant-memories>` / `<relevant_memories>`; ลบบล็อกเพย์โหลด XML การเรียกเครื่องมือแบบข้อความล้วน (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) รวมถึงเพย์โหลดที่ถูกตัดทอนและไม่เคยปิดอย่างสมบูรณ์; ลดระดับและลบโครงเรียกเครื่องมือ/ผลลัพธ์และตัวทำเครื่องหมายบริบทประวัติ; ลบโทเคนควบคุมโมเดลที่รั่ว (`<|assistant|>`, ASCII `<|...|>` อื่นๆ, แบบเต็มความกว้าง `<｜...｜>`); ลบ XML การเรียกเครื่องมือ MiniMax ที่ผิดรูป
- ข้อความที่มีลักษณะเหมือนข้อมูลประจำตัว/โทเคนถูกปกปิด
- บล็อกยาวสามารถถูกตัดทอนได้
- ประวัติที่ใหญ่มากสามารถทิ้งแถวเก่ากว่าหรือแทนที่แถวที่ใหญ่เกินด้วย `[sessions_history omitted: message too large]`
- การตรวจสอบบันทึกการถอดความดิบบนดิสก์เป็นตัวสำรองเมื่อคุณต้องการบันทึกการถอดความเต็มแบบตรงทุกไบต์

## นโยบายเครื่องมือ

เอเจนต์ย่อยใช้โปรไฟล์และไปป์ไลน์นโยบายเครื่องมือเดียวกับพาเรนต์หรือ
เอเจนต์เป้าหมายก่อน หลังจากนั้น OpenClaw จะใช้ชั้นข้อจำกัด
ของเอเจนต์ย่อย

เมื่อไม่มี `tools.profile` ที่จำกัด เอเจนต์ย่อยจะได้รับ **เครื่องมือทั้งหมด ยกเว้น
เครื่องมือเซสชัน** และเครื่องมือระบบ:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

`sessions_history` ยังคงเป็นมุมมองการเรียกคืนที่มีขอบเขตและผ่านการทำให้ปลอดภัยที่นี่เช่นกัน โดย
ไม่ใช่ดัมป์บันทึกการถอดความดิบ

เมื่อ `maxSpawnDepth >= 2` เอเจนต์ย่อยออร์เคสเตรเตอร์ความลึก 1 จะ
ได้รับ `sessions_spawn`, `subagents`, `sessions_list` และ
`sessions_history` เพิ่มเติม เพื่อให้จัดการลูกของตัวเองได้

### แทนที่ผ่านการกำหนดค่า

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny wins
        deny: ["gateway", "cron"],
        // if allow is set, it becomes allow-only (deny still wins)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` เป็นตัวกรองแบบ allow-only ขั้นสุดท้าย ตัวกรองนี้สามารถจำกัด
ชุดเครื่องมือที่แก้ไขแล้วให้แคบลงได้ แต่ไม่สามารถ **เพิ่มกลับ** เครื่องมือที่ถูกลบออก
โดย `tools.profile` ได้ ตัวอย่างเช่น `tools.profile: "coding"` รวม
`web_search`/`web_fetch` แต่ไม่รวมเครื่องมือ `browser` หากต้องการให้
เอเจนต์ย่อยที่ใช้โปรไฟล์ coding ใช้ระบบอัตโนมัติของเบราว์เซอร์ได้ ให้เพิ่ม browser ใน
ขั้นตอนโปรไฟล์:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

ใช้ `agents.list[].tools.alsoAllow: ["browser"]` ต่อเอเจนต์เมื่อมีเพียงเอเจนต์เดียว
ที่ควรได้รับระบบอัตโนมัติของเบราว์เซอร์

## การทำงานพร้อมกัน

เอเจนต์ย่อยใช้เลนคิวในโปรเซสเฉพาะ:

- **ชื่อเลน:** `subagent`
- **การทำงานพร้อมกัน:** `agents.defaults.subagents.maxConcurrent` (ค่าเริ่มต้น `8`)

## ความมีชีวิตและการกู้คืน

OpenClaw ไม่ถือว่าการไม่มี `endedAt` เป็นหลักฐานถาวรว่า
เอเจนต์ย่อยยังมีชีวิตอยู่ รันที่ยังไม่สิ้นสุดซึ่งเก่ากว่าหน้าต่างรันค้าง
จะหยุดถูกนับเป็น active/pending ใน `/subagents list`, สรุปสถานะ,
การกั้นการเสร็จสิ้นของลูกหลาน และการตรวจสอบการทำงานพร้อมกันต่อเซสชัน

หลังจาก Gateway รีสตาร์ท รันที่กู้คืนกลับมาและยังไม่สิ้นสุดซึ่งค้างจะถูกตัดออก เว้นแต่
เซสชันลูกของรันนั้นถูกทำเครื่องหมาย `abortedLastRun: true` เซสชันลูกที่
ถูกยกเลิกจากการรีสตาร์ทเหล่านั้นยังคงกู้คืนได้ผ่านโฟลว์กู้คืนเอเจนต์ย่อยกำพร้า
ซึ่งจะส่งข้อความกลับมาทำงานต่อสังเคราะห์ก่อน
ล้างเครื่องหมาย aborted

การกู้คืนอัตโนมัติหลังรีสตาร์ทมีขอบเขตต่อเซสชันลูก หาก
ลูกของเอเจนต์ย่อยเดียวกันถูกรับเข้ากระบวนการกู้คืนกำพร้าซ้ำๆ ภายใน
หน้าต่าง rapid re-wedge OpenClaw จะบันทึก tombstone การกู้คืนไว้บน
เซสชันนั้นและหยุด auto-resume ในการรีสตาร์ทครั้งถัดไป รัน
`openclaw tasks maintenance --apply` เพื่อปรับระเบียนงานให้ตรงกัน หรือ
`openclaw doctor --fix` เพื่อล้างธงการกู้คืน aborted ที่ค้างอยู่บน
เซสชันที่ถูก tombstone

<Note>
หากการ spawn เอเจนต์ย่อยล้มเหลวด้วย Gateway `PAIRING_REQUIRED` /
`scope-upgrade` ให้ตรวจสอบตัวเรียก RPC ก่อนแก้ไขสถานะการจับคู่
การประสานงาน `sessions_spawn` ภายในควรเชื่อมต่อเป็น
`client.id: "gateway-client"` พร้อม `client.mode: "backend"` ผ่านการยืนยันตัวตน
shared-token/password แบบ direct loopback; เส้นทางนั้นไม่ขึ้นกับ
scope baseline ของอุปกรณ์ที่จับคู่แล้วของ CLI ผู้เรียกระยะไกล,
`deviceIdentity` ที่ระบุชัดเจน, เส้นทาง device-token ที่ระบุชัดเจน และไคลเอนต์ browser/node
ยังต้องได้รับการอนุมัติอุปกรณ์ตามปกติสำหรับการอัปเกรด scope
</Note>

## การหยุด

- การส่ง `/stop` ในแชตของผู้ร้องขอจะยกเลิกเซสชันของผู้ร้องขอและหยุดรันของเอเจนต์ย่อยที่ใช้งานอยู่ซึ่ง spawn จากเซสชันนั้น โดย cascade ไปยังลูกที่ซ้อนกัน
- `/subagents kill <id>` หยุดเอเจนต์ย่อยที่ระบุและ cascade ไปยังลูกของเอเจนต์นั้น

## ข้อจำกัด

- การประกาศของเอเจนต์ย่อยเป็นแบบ **best-effort** หาก Gateway รีสตาร์ท งาน "announce back" ที่ค้างอยู่จะหายไป
- เอเจนต์ย่อยยังคงใช้ทรัพยากรโปรเซส Gateway เดียวกันร่วมกัน ให้ถือว่า `maxConcurrent` เป็นวาล์วนิรภัย
- `sessions_spawn` เป็น non-blocking เสมอ: จะส่งคืน `{ status: "accepted", runId, childSessionKey }` ทันที
- บริบทของเอเจนต์ย่อยจะฉีดเฉพาะ `AGENTS.md` + `TOOLS.md` (ไม่มี `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` หรือ `BOOTSTRAP.md`)
- ความลึกสูงสุดของการซ้อนคือ 5 (ช่วง `maxSpawnDepth`: 1–5) แนะนำให้ใช้ความลึก 2 สำหรับกรณีใช้งานส่วนใหญ่
- `maxChildrenPerAgent` จำกัดจำนวนลูกที่ใช้งานอยู่ต่อเซสชัน (ค่าเริ่มต้น `5`, ช่วง `1–20`)
