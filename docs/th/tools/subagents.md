เอเจนต์ย่อยคือการรันเอเจนต์เบื้องหลังที่ถูกสร้างจากการรันเอเจนต์ที่มีอยู่
เอเจนต์เหล่านี้รันในเซสชันของตนเอง (`agent:<agentId>:subagent:<uuid>`) และ
เมื่อเสร็จแล้วจะ **ประกาศ** ผลลัพธ์กลับไปยังช่องทางแชตของผู้ร้องขอ
การรันเอเจนต์ย่อยแต่ละครั้งจะถูกติดตามเป็น
[งานเบื้องหลัง](/th/automation/tasks)

เป้าหมายหลัก:

- ทำให้งาน "ค้นคว้า / งานยาว / เครื่องมือช้า" ทำงานขนานกันได้โดยไม่บล็อกการรันหลัก
- แยกเอเจนต์ย่อยออกจากกันโดยค่าเริ่มต้น (การแยกเซสชัน + sandboxing แบบเลือกได้)
- ทำให้พื้นผิวเครื่องมือใช้งานผิดได้ยาก: เอเจนต์ย่อยจะไม่ได้รับเครื่องมือเซสชันโดยค่าเริ่มต้น
- รองรับความลึกการซ้อนที่กำหนดค่าได้สำหรับรูปแบบ orchestrator

<Note>
**หมายเหตุเรื่องค่าใช้จ่าย:** เอเจนต์ย่อยแต่ละตัวมีบริบทและการใช้โทเคนของตนเองโดย
ค่าเริ่มต้น สำหรับงานหนักหรืองานที่ทำซ้ำ ให้ตั้งโมเดลที่ถูกกว่าสำหรับเอเจนต์ย่อย
และคงเอเจนต์หลักไว้บนโมเดลคุณภาพสูงกว่า กำหนดค่าผ่าน
`agents.defaults.subagents.model` หรือ override รายเอเจนต์ เมื่อ child
    จำเป็นต้องใช้ transcript ปัจจุบันของผู้ร้องขอจริง ๆ เอเจนต์สามารถขอ
    `context: "fork"` ในการ spawn ครั้งนั้นได้ เซสชัน subagent ที่ผูกกับ thread มีค่าเริ่มต้น
    เป็น `context: "fork"` เพราะจะแตกแขนงบทสนทนาปัจจุบันไปเป็น
    thread ติดตามผล
</Note>

## คำสั่ง slash

ใช้ `/subagents` เพื่อตรวจสอบหรือควบคุมการรันเอเจนต์ย่อยสำหรับ **เซสชันปัจจุบัน**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

ใช้ [`/steer <message>`](/th/tools/steer) ระดับบนสุดเพื่อ steer การรันที่ใช้งานอยู่ของเซสชันผู้ร้องขอปัจจุบัน ใช้ `/subagents steer <id|#> <message>` เมื่อเป้าหมายคือการรัน child

`/subagents info` แสดง metadata ของการรัน (สถานะ, timestamp, session id,
พาธ transcript, cleanup) ใช้ `sessions_history` สำหรับมุมมอง recall ที่จำกัดขอบเขตและ
ผ่านตัวกรองความปลอดภัยแล้ว; ตรวจสอบพาธ transcript บนดิสก์เมื่อคุณ
ต้องใช้ transcript เต็มแบบดิบ

### การควบคุมการผูก thread

คำสั่งเหล่านี้ทำงานบนช่องทางที่รองรับการผูก thread แบบถาวร
ดู [ช่องทางที่รองรับ thread](#thread-supporting-channels) ด้านล่าง

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### พฤติกรรมการ spawn

`/subagents spawn` เริ่มเอเจนต์ย่อยเบื้องหลังเป็นคำสั่งของผู้ใช้ (ไม่ใช่
relay ภายใน) และส่งการอัปเดตการเสร็จสิ้นสุดท้ายหนึ่งครั้งกลับไปยัง
แชตของผู้ร้องขอเมื่อการรันเสร็จสิ้น

<AccordionGroup>
  <Accordion title="Non-blocking, push-based completion">
    - คำสั่ง spawn ไม่บล็อก; จะคืน run id ทันที
    - เมื่อเสร็จสิ้น เอเจนต์ย่อยจะประกาศข้อความสรุป/ผลลัพธ์กลับไปยังช่องทางแชตของผู้ร้องขอ
    - เทิร์นของเอเจนต์ที่ต้องใช้ผลลัพธ์จาก child ควรเรียก `sessions_yield` หลังจาก spawn งานที่จำเป็นแล้ว การทำเช่นนี้จะจบเทิร์นปัจจุบันและให้ event การเสร็จสิ้นเข้ามาเป็นข้อความถัดไปที่โมเดลมองเห็นได้
    - การเสร็จสิ้นเป็นแบบ push-based เมื่อ spawn แล้ว อย่า poll `/subagents list`, `sessions_list` หรือ `sessions_history` วนซ้ำเพียงเพื่อรอให้เสร็จ; ตรวจสอบสถานะเฉพาะเมื่อจำเป็นสำหรับการ debug หรือการแทรกแซง
    - เอาต์พุตของ child เป็นรายงาน/หลักฐานสำหรับเอเจนต์ผู้ร้องขอใช้สังเคราะห์ ไม่ใช่ข้อความคำสั่งที่ผู้ใช้เขียน และไม่สามารถ override นโยบายของ system, developer หรือ user ได้
    - เมื่อเสร็จสิ้น OpenClaw จะพยายามปิดแท็บเบราว์เซอร์/โปรเซสที่ติดตามได้ซึ่งเปิดโดยเซสชันเอเจนต์ย่อยนั้น ก่อนที่ flow cleanup สำหรับการประกาศจะดำเนินต่อ

  </Accordion>
  <Accordion title="Manual-spawn delivery resilience">
    - OpenClaw ส่งการเสร็จสิ้นกลับไปยังเซสชันผู้ร้องขอผ่านเทิร์น `agent` พร้อม idempotency key ที่เสถียร
    - หากการรันของผู้ร้องขอยังใช้งานอยู่ OpenClaw จะพยายาม wake/steer การรันนั้นก่อน แทนที่จะเริ่มเส้นทางตอบกลับที่มองเห็นได้เส้นที่สอง
    - หากการ handoff การเสร็จสิ้นไปยังเอเจนต์ผู้ร้องขอล้มเหลวหรือไม่สร้างเอาต์พุตที่มองเห็นได้ OpenClaw จะถือว่าการส่งล้มเหลวและ fallback ไปที่ queue routing/retry โดยจะไม่ส่งผลลัพธ์ child แบบ raw ไปยังแชตภายนอกโดยตรง
    - หากใช้ direct handoff ไม่ได้ จะ fallback ไปที่ queue routing
    - หาก queue routing ยังไม่พร้อมใช้งาน การประกาศจะถูก retry ด้วย exponential backoff สั้น ๆ ก่อนยอมแพ้ขั้นสุดท้าย
    - การส่งการเสร็จสิ้นจะคง route ของผู้ร้องขอที่ resolve แล้วไว้: route การเสร็จสิ้นที่ผูกกับ thread หรือผูกกับ conversation จะชนะเมื่อพร้อมใช้งาน; หาก origin ของการเสร็จสิ้นให้มาเพียง channel, OpenClaw จะเติม target/account ที่ขาดจาก route ที่ resolve แล้วของเซสชันผู้ร้องขอ (`lastChannel` / `lastTo` / `lastAccountId`) เพื่อให้การส่งโดยตรงยังทำงานได้

  </Accordion>

    - `Status` — `completed successfully` / `failed` / `timed out` / `unknown`

## โหมดบริบท

เอเจนต์ย่อย native จะเริ่มแบบแยกเดี่ยว เว้นแต่ผู้เรียกจะขอ fork
transcript ปัจจุบันอย่างชัดเจน

| โหมด       | ควรใช้เมื่อใด                                                                                                                         | พฤติกรรม                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | การค้นคว้าใหม่, การ implement อิสระ, งานเครื่องมือที่ช้า หรือสิ่งใดก็ตามที่สามารถสรุปในข้อความงานได้                           | สร้าง transcript child ที่สะอาด นี่คือค่าเริ่มต้นและช่วยให้ใช้โทเคนน้อยลง  |
| `fork`     | งานที่ขึ้นกับบทสนทนาปัจจุบัน, ผลลัพธ์เครื่องมือก่อนหน้า หรือคำสั่งที่ละเอียดอ่อนซึ่งมีอยู่แล้วใน transcript ของผู้ร้องขอ | แตกแขนง transcript ของผู้ร้องขอเข้าไปในเซสชัน child ก่อนที่ child จะเริ่ม |

ใช้ `fork` อย่างประหยัด มีไว้สำหรับการมอบหมายงานที่ไวต่อบริบท ไม่ใช่
สิ่งทดแทนการเขียน prompt งานให้ชัดเจน

## เครื่องมือ: `sessions_spawn`

เริ่มการรันเอเจนต์ย่อยด้วย `deliver: false` บน lane `subagent` ระดับ global
จากนั้นรันขั้นตอน announce และโพสต์การตอบกลับ announce ไปยังช่องทางแชต
ของผู้ร้องขอ

ความพร้อมใช้งานขึ้นกับนโยบายเครื่องมือที่มีผลของผู้เรียก โปรไฟล์ `coding` และ
`full` เปิดเผย `sessions_spawn` โดยค่าเริ่มต้น โปรไฟล์ `messaging`
ไม่เปิดเผย; เพิ่ม `tools.alsoAllow: ["sessions_spawn", "sessions_yield",
"subagents"]` หรือใช้ `tools.profile: "coding"` สำหรับเอเจนต์ที่ควรมอบหมาย
งาน นโยบาย allow/deny ของ channel/group, provider, sandbox และรายเอเจนต์ยังสามารถ
นำเครื่องมือออกหลังจากขั้นตอนโปรไฟล์ได้ ใช้ `/tools` จากเซสชันเดียวกัน
เพื่อยืนยันรายการเครื่องมือที่มีผล

**ค่าเริ่มต้น:**

- **โมเดล:** สืบทอดจากผู้เรียก เว้นแต่คุณตั้ง `agents.defaults.subagents.model` (หรือ `agents.list[].subagents.model` รายเอเจนต์); `sessions_spawn.model` ที่ระบุชัดเจนยังคงมีสิทธิ์เหนือกว่า
- **Thinking:** สืบทอดจากผู้เรียก เว้นแต่คุณตั้ง `agents.defaults.subagents.thinking` (หรือ `agents.list[].subagents.thinking` รายเอเจนต์); `sessions_spawn.thinking` ที่ระบุชัดเจนยังคงมีสิทธิ์เหนือกว่า
- **timeout การรัน:** หากละ `sessions_spawn.runTimeoutSeconds` ไว้ OpenClaw จะใช้ `agents.defaults.subagents.runTimeoutSeconds` เมื่อมีการตั้งค่า; มิฉะนั้นจะ fallback ไปที่ `0` (ไม่มี timeout)

### โหมด prompt สำหรับการมอบหมายงาน

`agents.defaults.subagents.delegationMode` ควบคุมคำแนะนำใน prompt เท่านั้น; ไม่เปลี่ยนนโยบายเครื่องมือหรือบังคับการมอบหมายงาน

- `suggest` (ค่าเริ่มต้น): คง prompt nudge มาตรฐานให้ใช้เอเจนต์ย่อยสำหรับงานที่ใหญ่กว่าหรือช้ากว่า
- `prefer`: บอกเอเจนต์หลักให้ยังตอบสนองได้ดีและมอบหมายสิ่งที่ซับซ้อนกว่าการตอบตรงผ่าน `sessions_spawn`

override รายเอเจนต์ใช้ `agents.list[].subagents.delegationMode`

```json5
{
  agents: {
    defaults: {
      subagents: {
        delegationMode: "prefer",
        maxConcurrent: 4,
      },
    },
    list: [
      {
        id: "coordinator",
        subagents: { delegationMode: "prefer" },
      },
    ],
  },
}
```

</ParamField>
<ParamField path="streamTo" type='"parent"'>

</ParamField>
<ParamField path="model" type="string">

</ParamField>
<ParamField path="thinking" type="string">

<ParamField path="runTimeoutSeconds" type="number">

<ParamField path="thread" type="boolean" default="false">

### ชื่องานและการกำหนดเป้าหมาย

`taskName` เป็นตัวระบุสำหรับการประสานงานที่โมเดลมองเห็น ไม่ใช่คีย์เซสชัน
ใช้สำหรับชื่อลูกที่เสถียร เช่น `review_subagents`,
`linux_validation` หรือ `docs_update` เมื่อผู้ประสานงานอาจต้องกำกับ
หรือหยุดลูกนั้นในภายหลัง

การแปลงเป้าหมายรองรับการตรงกับ `taskName` แบบเป๊ะ และคำนำหน้า
ที่ไม่กำกวม การจับคู่ถูกจำกัดขอบเขตไว้ในหน้าต่างเป้าหมายที่ใช้งานอยู่/ล่าสุดเดียวกันกับที่ใช้
โดยเป้าหมาย `/subagents` แบบลำดับเลข ดังนั้นลูกที่เสร็จสิ้นไปนานแล้วจะไม่ทำให้
ตัวระบุที่นำกลับมาใช้ใหม่กำกวม หากลูกที่ใช้งานอยู่หรือล่าสุดสองรายใช้
`taskName` เดียวกัน เป้าหมายจะกำกวม ให้ใช้ดัชนีรายการ คีย์เซสชัน หรือ
run id แทน

เป้าหมายที่สงวนไว้ `last` และ `all` ไม่ใช่ค่า `taskName` ที่ถูกต้อง
เพราะมีความหมายในการควบคุมอยู่แล้ว

## เครื่องมือ: `sessions_yield`

จบเทิร์นโมเดลปัจจุบันและรอให้เหตุการณ์รันไทม์ โดยหลักคือ
เหตุการณ์การเสร็จสิ้นของซับเอเจนต์ เข้ามาเป็นข้อความถัดไป ใช้หลังจาก
สร้างงานลูกที่จำเป็นเมื่อผู้ร้องขอไม่สามารถสร้างคำตอบสุดท้าย
ได้จนกว่าการเสร็จสิ้นเหล่านั้นจะมาถึง

`sessions_yield` เป็น primitive สำหรับการรอ อย่าแทนที่ด้วยลูป polling
เหนือ `subagents`, `sessions_list`, `sessions_history`, เชลล์
`sleep` หรือการ polling โปรเซสเพียงเพื่อตรวจจับการเสร็จสิ้นของลูก

ใช้ `sessions_yield` เฉพาะเมื่อรายการเครื่องมือที่มีผลของเซสชันมี
เครื่องมือนี้อยู่ โปรไฟล์เครื่องมือแบบขั้นต่ำหรือกำหนดเองบางรายการอาจเปิดให้ใช้ `sessions_spawn` และ
`subagents` โดยไม่เปิดให้ใช้ `sessions_yield`; ในกรณีนั้น อย่าประดิษฐ์
ลูป polling เพียงเพื่อรอการเสร็จสิ้น

เมื่อมีลูกที่ใช้งานอยู่ OpenClaw จะแทรกบล็อกพรอมป์
`Active Subagents` ขนาดกะทัดรัดที่รันไทม์สร้างขึ้นเข้าในเทิร์นปกติ เพื่อให้ผู้ร้องขอเห็น
เซสชันลูกปัจจุบัน, run ids, สถานะ, ป้ายกำกับ, งาน และ
นามแฝง `taskName` ได้โดยไม่ต้อง polling ฟิลด์งานและป้ายกำกับใน
บล็อกนั้นถูกใส่เครื่องหมายคำพูดเป็นข้อมูล ไม่ใช่คำสั่ง เพราะอาจมีต้นกำเนิด
จากอาร์กิวเมนต์การสร้างที่ผู้ใช้/โมเดลให้มา

## เครื่องมือ: `subagents`

แสดงรายการ กำกับ หรือหยุดการรันซับเอเจนต์ที่ถูกสร้างและเป็นของเซสชัน
ผู้ร้องขอ ขอบเขตถูกจำกัดไว้ที่ผู้ร้องขอปัจจุบัน; ลูกจะเห็น/ควบคุมได้เฉพาะ
ลูกที่อยู่ภายใต้การควบคุมของตนเองเท่านั้น

ใช้ `subagents` สำหรับสถานะแบบตามต้องการ การดีบัก การกำกับ หรือการหยุด
ใช้ `sessions_yield` เพื่อรอเหตุการณ์การเสร็จสิ้น

## เซสชันที่ผูกกับเธรด

เมื่อเปิดใช้การผูกเธรดสำหรับช่องทาง ซับเอเจนต์สามารถผูกอยู่กับ
เธรดได้ เพื่อให้ข้อความติดตามผลของผู้ใช้ในเธรดนั้นยังคงถูกส่งไปยัง
เซสชันซับเอเจนต์เดิม

### ช่องทางที่รองรับเธรด

**Discord** เป็นช่องทางเดียวที่รองรับในขณะนี้ รองรับ
เซสชันซับเอเจนต์ที่ผูกกับเธรดแบบถาวร (`sessions_spawn` พร้อม
`thread: true`), การควบคุมเธรดด้วยตนเอง (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`) และคีย์ adapter
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours` และ
`channels.discord.threadBindings.spawnSessions`

### โฟลว์ด่วน

<Steps>
  <Step title="สร้าง">
    `sessions_spawn` พร้อม `thread: true` (และ `mode: "session"` แบบไม่บังคับ)
  </Step>
  <Step title="ผูก">
    OpenClaw สร้างหรือผูกเธรดกับเป้าหมายเซสชันนั้นในช่องทางที่ใช้งานอยู่
  </Step>
  <Step title="กำหนดเส้นทางการติดตามผล">
    การตอบกลับและข้อความติดตามผลในเธรดนั้นจะถูกส่งไปยังเซสชันที่ผูกไว้
  </Step>
  <Step title="ตรวจสอบ timeout">
    ใช้ `/session idle` เพื่อตรวจสอบ/อัปเดตการเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน และ
    `/session max-age` เพื่อควบคุมขีดจำกัดสูงสุดแบบตายตัว
  </Step>
  <Step title="แยกออก">
    ใช้ `/unfocus` เพื่อแยกออกด้วยตนเอง
  </Step>
</Steps>

### การควบคุมด้วยตนเอง

| คำสั่ง            | ผลลัพธ์                                                                |
| ------------------ | --------------------------------------------------------------------- |
| `/focus <target>`  | ผูกเธรดปัจจุบัน (หรือสร้างใหม่) กับเป้าหมายซับเอเจนต์/เซสชัน |
| `/unfocus`         | ลบการผูกสำหรับเธรดที่ผูกอยู่ในปัจจุบัน                       |
| `/agents`          | แสดงรายการการรันที่ใช้งานอยู่และสถานะการผูก (`thread:<id>` หรือ `unbound`)       |
| `/session idle`    | ตรวจสอบ/อัปเดตการเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน (เฉพาะเธรดที่ผูกและโฟกัสอยู่)         |
| `/session max-age` | ตรวจสอบ/อัปเดตขีดจำกัดสูงสุดแบบตายตัว (เฉพาะเธรดที่ผูกและโฟกัสอยู่)                  |

### สวิตช์การกำหนดค่า

- **ค่าเริ่มต้นส่วนกลาง:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`
- **คีย์ override ของช่องทางและการผูกอัตโนมัติเมื่อสร้าง** เป็นแบบเฉพาะ adapter ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านบน

ดู [ข้อมูลอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference) และ
[คำสั่ง Slash](/th/tools/slash-commands) สำหรับรายละเอียด adapter ปัจจุบัน

### Allowlist

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  รายการ id ของเอเจนต์ที่สามารถกำหนดเป้าหมายผ่าน `agentId` แบบชัดเจน (`["*"]` อนุญาตทุกตัว) ค่าเริ่มต้น: เฉพาะเอเจนต์ผู้ร้องขอ หากคุณตั้งค่ารายการและยังต้องการให้ผู้ร้องขอสร้างตัวเองด้วย `agentId` ให้รวม id ของผู้ร้องขอไว้ในรายการ
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  allowlist ของเอเจนต์เป้าหมายเริ่มต้นที่ใช้เมื่อเอเจนต์ผู้ร้องขอไม่ได้ตั้งค่า `subagents.allowAgents` ของตนเอง
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  บล็อกการเรียก `sessions_spawn` ที่ละ `agentId` (บังคับให้เลือกโปรไฟล์อย่างชัดเจน) การ override รายเอเจนต์: `agents.list[].subagents.requireAgentId`
</ParamField>

หากเซสชันผู้ร้องขออยู่ในแซนด์บ็อกซ์ `sessions_spawn` จะปฏิเสธเป้าหมาย
ที่จะรันโดยไม่อยู่ในแซนด์บ็อกซ์

### การค้นหา

ใช้ `agents_list` เพื่อดูว่า id ของเอเจนต์ใดที่ได้รับอนุญาตสำหรับ
`sessions_spawn` อยู่ในขณะนี้ การตอบกลับมีโมเดลที่มีผลของเอเจนต์แต่ละตัวในรายการ
และเมตาดาต้ารันไทม์ที่ฝังไว้ เพื่อให้ผู้เรียกสามารถแยกความแตกต่างระหว่าง PI, Codex
app-server และรันไทม์เนทีฟอื่นที่กำหนดค่าไว้

### การเก็บถาวรอัตโนมัติ

- เซสชันซับเอเจนต์จะถูกเก็บถาวรโดยอัตโนมัติหลังจาก `agents.defaults.subagents.archiveAfterMinutes` (ค่าเริ่มต้น `60`)
- การเก็บถาวรใช้ `sessions.delete` และเปลี่ยนชื่อทรานสคริปต์เป็น `*.deleted.<timestamp>` (โฟลเดอร์เดียวกัน)
- `cleanup: "delete"` เก็บถาวรทันทีหลังประกาศ (ยังคงเก็บทรานสคริปต์ไว้ผ่านการเปลี่ยนชื่อ)
- การเก็บถาวรอัตโนมัติเป็นแบบ best-effort; timer ที่ค้างอยู่จะสูญหายหาก gateway รีสตาร์ต
- `runTimeoutSeconds` **ไม่** เก็บถาวรอัตโนมัติ; เพียงหยุดการรันเท่านั้น เซสชันยังคงอยู่จนกว่าจะถูกเก็บถาวรอัตโนมัติ
- การเก็บถาวรอัตโนมัติมีผลเท่ากันกับเซสชันระดับความลึก 1 และระดับความลึก 2
- การล้างเบราว์เซอร์แยกจากการล้างการเก็บถาวร: แท็บ/โปรเซสเบราว์เซอร์ที่ติดตามไว้จะถูกปิดแบบ best-effort เมื่อการรันเสร็จสิ้น แม้จะเก็บทรานสคริปต์/เรคคอร์ดเซสชันไว้ก็ตาม

## ซับเอเจนต์ซ้อนกัน

โดยค่าเริ่มต้น ซับเอเจนต์ไม่สามารถสร้างซับเอเจนต์ของตัวเองได้
(`maxSpawnDepth: 1`) ตั้งค่า `maxSpawnDepth: 2` เพื่อเปิดใช้การซ้อนกันหนึ่งระดับ
ซึ่งเป็น **รูปแบบ orchestrator**: หลัก → ซับเอเจนต์ orchestrator →
ซับเอเจนต์ย่อย worker

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // allow sub-agents to spawn children (default: 1)
        maxChildrenPerAgent: 5, // max active children per agent session (default: 5)
        maxConcurrent: 8, // global concurrency lane cap (default: 8)
        runTimeoutSeconds: 900, // default timeout for sessions_spawn when omitted (0 = no timeout)
      },
    },
  },
}
```

### ระดับความลึก

| ความลึก | รูปแบบคีย์เซสชัน                            | บทบาท                                          | สร้างได้หรือไม่                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | เอเจนต์หลัก                                    | เสมอ                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | ซับเอเจนต์ (orchestrator เมื่ออนุญาตความลึก 2) | เฉพาะเมื่อ `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | ซับเอเจนต์ย่อย (leaf worker)                   | ไม่เคย                        |

### สายการประกาศ

ผลลัพธ์ไหลย้อนกลับขึ้นตามสาย:

1. worker ระดับความลึก 2 เสร็จสิ้น → ประกาศไปยังพาเรนต์ของตน (orchestrator ระดับความลึก 1)
2. orchestrator ระดับความลึก 1 รับประกาศ สังเคราะห์ผลลัพธ์ เสร็จสิ้น → ประกาศไปยังตัวหลัก
3. เอเจนต์หลักรับประกาศและส่งให้ผู้ใช้

แต่ละระดับจะเห็นเฉพาะประกาศจากลูกโดยตรงของตนเท่านั้น

<Note>
**แนวทางการปฏิบัติงาน:** เริ่มงานลูกหนึ่งครั้งและรอเหตุการณ์การเสร็จสิ้น
แทนการสร้างลูป polling รอบ `sessions_list`,
`sessions_history`, `/subagents list` หรือคำสั่ง sleep ของ `exec`
`sessions_list` และ `/subagents list` คงความสัมพันธ์ของเซสชันลูก
ให้โฟกัสกับงานที่ยังมีชีวิตอยู่ ลูกที่ยังมีชีวิตอยู่จะยังคงแนบอยู่ ลูกที่จบแล้วจะยัง
มองเห็นได้ในหน้าต่างล่าสุดช่วงสั้น ๆ และลิงก์ลูกที่มีอยู่เฉพาะใน store และเก่าแล้วจะถูก
ละเว้นหลังพ้นหน้าต่างความสดใหม่ วิธีนี้ป้องกันไม่ให้เมตาดาต้า `spawnedBy` /
`parentSessionKey` เก่าปลุกลูกปลอมขึ้นมาหลัง
รีสตาร์ต หากเหตุการณ์การเสร็จสิ้นของลูกมาถึงหลังจากคุณส่ง
คำตอบสุดท้ายไปแล้ว การติดตามผลที่ถูกต้องคือโทเคนเงียบที่ตรงเป๊ะ
`NO_REPLY` / `no_reply`
</Note>

### นโยบายเครื่องมือตามความลึก

- บทบาทและขอบเขตการควบคุมถูกเขียนลงในเมทาดาทาของเซสชันเมื่อ spawn เซสชัน ซึ่งช่วยป้องกันไม่ให้คีย์เซสชันแบบแบนหรือเซสชันที่กู้คืนมากลับได้รับสิทธิ์ของตัวประสานงานโดยไม่ตั้งใจ
- **ความลึก 1 (ตัวประสานงาน เมื่อ `maxSpawnDepth >= 2`):** ได้รับ `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` เพื่อให้จัดการลูกของตนได้ เครื่องมือเซสชัน/ระบบอื่นยังคงถูกปฏิเสธ
- **ความลึก 1 (ปลายทาง เมื่อ `maxSpawnDepth == 1`):** ไม่มีเครื่องมือเซสชัน (พฤติกรรมค่าเริ่มต้นปัจจุบัน)
- **ความลึก 2 (เวิร์กเกอร์ปลายทาง):** ไม่มีเครื่องมือเซสชัน — `sessions_spawn` ถูกปฏิเสธเสมอที่ความลึก 2 ไม่สามารถ spawn ลูกเพิ่มเติมได้

### ขีดจำกัดการ spawn ต่อเอเจนต์

แต่ละเซสชันของเอเจนต์ (ที่ความลึกใดก็ตาม) มีลูกที่กำลังทำงานได้สูงสุด `maxChildrenPerAgent`
(ค่าเริ่มต้น `5`) ในแต่ละครั้ง ซึ่งป้องกันการแตกกิ่งมากเกินไป
จากตัวประสานงานตัวเดียว

### การหยุดแบบทอดต่อ

การหยุดตัวประสานงานที่ความลึก 1 จะหยุดลูกที่ความลึก 2
ทั้งหมดของตัวนั้นโดยอัตโนมัติ:

- `/stop` ในแชตหลักจะหยุดเอเจนต์ความลึก 1 ทั้งหมดและทอดต่อไปยังลูกความลึก 2 ของเอเจนต์เหล่านั้น
- `/subagents kill <id>` หยุดเอเจนต์ย่อยที่ระบุและทอดต่อไปยังลูกของเอเจนต์นั้น
- `/subagents kill all` หยุดเอเจนต์ย่อยทั้งหมดของผู้ร้องขอและทอดต่อ

## การยืนยันตัวตน

การยืนยันตัวตนของเอเจนต์ย่อยถูกแก้ค่าโดยใช้ **รหัสเอเจนต์** ไม่ใช่ชนิดเซสชัน:

- คีย์เซสชันของเอเจนต์ย่อยคือ `agent:<agentId>:subagent:<uuid>`
- สโตร์การยืนยันตัวตนถูกโหลดจาก `agentDir` ของเอเจนต์นั้น
- โปรไฟล์การยืนยันตัวตนของเอเจนต์หลักจะถูกรวมเข้ามาเป็น **fallback**; โปรไฟล์เอเจนต์จะเขียนทับโปรไฟล์หลักเมื่อมีการชนกัน

การรวมเป็นแบบเพิ่มเข้าไป ดังนั้นโปรไฟล์หลักจะพร้อมใช้งานเป็น
fallback เสมอ ยังไม่รองรับการยืนยันตัวตนที่แยกขาดจากกันอย่างสมบูรณ์ต่อเอเจนต์

## การประกาศ

เอเจนต์ย่อยรายงานกลับผ่านขั้นตอนประกาศ:

- ขั้นตอนประกาศทำงานภายในเซสชันของเอเจนต์ย่อย (ไม่ใช่เซสชันของผู้ร้องขอ)
- ถ้าเอเจนต์ย่อยตอบกลับว่า `ANNOUNCE_SKIP` ตรงตัว จะไม่มีการโพสต์อะไร
- ถ้าข้อความผู้ช่วยล่าสุดเป็นโทเค็นเงียบตรงตัว `NO_REPLY` / `no_reply` เอาต์พุตประกาศจะถูกระงับ แม้เคยมีความคืบหน้าที่มองเห็นได้ก่อนหน้า

การส่งมอบขึ้นกับความลึกของผู้ร้องขอ:

- เซสชันผู้ร้องขอระดับบนสุดใช้การเรียก `agent` ตามผลลัพธ์พร้อมการส่งมอบภายนอก (`deliver=true`)
- เซสชันเอเจนต์ย่อยของผู้ร้องขอที่ซ้อนอยู่จะได้รับการฉีดตามผลลัพธ์ภายใน (`deliver=false`) เพื่อให้ตัวประสานงานสังเคราะห์ผลลัพธ์ของลูกภายในเซสชันได้
- ถ้าเซสชันเอเจนต์ย่อยของผู้ร้องขอที่ซ้อนอยู่หายไป OpenClaw จะ fallback ไปยังผู้ร้องขอของเซสชันนั้นเมื่อมี

สำหรับเซสชันผู้ร้องขอระดับบนสุด การส่งมอบโดยตรงในโหมดเสร็จสิ้นจะ
แก้ค่าเส้นทางบทสนทนา/เธรดที่ผูกไว้และการแทนที่ hook ก่อน จากนั้นเติม
ฟิลด์ช่องทาง-เป้าหมายที่ขาดไปจากเส้นทางที่เก็บไว้ของเซสชันผู้ร้องขอ
ซึ่งทำให้การเสร็จสิ้นอยู่ในแชต/หัวข้อที่ถูกต้อง แม้จุดกำเนิดการเสร็จสิ้น
จะระบุเฉพาะช่องทางเท่านั้น

การรวมผลการเสร็จสิ้นของลูกถูกจำกัดขอบเขตไว้ที่รันของผู้ร้องขอปัจจุบันเมื่อ
สร้างผลการค้นพบการเสร็จสิ้นแบบซ้อน เพื่อป้องกันไม่ให้เอาต์พุตลูกจากรันก่อนหน้า
ที่ค้างอยู่รั่วเข้าสู่ประกาศปัจจุบัน การตอบประกาศจะรักษาเส้นทาง
เธรด/หัวข้อเมื่อมีอยู่บนอะแดปเตอร์ช่องทาง

### บริบทประกาศ

บริบทประกาศถูกทำให้เป็นบล็อกเหตุการณ์ภายในที่เสถียร:

| ฟิลด์          | แหล่งที่มา                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| แหล่งที่มา         | `subagent` หรือ `cron`                                                                                          |
| รหัสเซสชัน    | คีย์/รหัสเซสชันลูก                                                                                          |
| ชนิด           | ชนิดประกาศ + ป้ายชื่องาน                                                                                    |
| สถานะ         | ได้มาจากผลลัพธ์รันไทม์ (`success`, `error`, `timeout` หรือ `unknown`) — **ไม่ได้** อนุมานจากข้อความโมเดล |
| เนื้อหาผลลัพธ์ | ข้อความผู้ช่วยที่มองเห็นได้ล่าสุด มิฉะนั้นเป็นข้อความ tool/toolResult ล่าสุดที่ผ่านการทำให้ปลอดภัย                                |
| การติดตามผล      | คำสั่งที่อธิบายว่าเมื่อใดควรตอบกลับและเมื่อใดควรเงียบ                                                           |

รันปลายทางที่ล้มเหลวรายงานสถานะล้มเหลวโดยไม่เล่นซ้ำ
ข้อความตอบกลับที่จับไว้ เมื่อหมดเวลา ถ้าลูกไปได้เพียงผ่านการเรียกเครื่องมือ ประกาศ
สามารถยุบประวัตินั้นเป็นสรุปความคืบหน้าบางส่วนแบบสั้นแทน
การเล่นเอาต์พุตดิบของเครื่องมือซ้ำ

### บรรทัดสถิติ

เพย์โหลดประกาศมีบรรทัดสถิติที่ท้ายสุด (แม้ถูกห่อ):

- รันไทม์ (เช่น `runtime 5m12s`)
- การใช้โทเค็น (อินพุต/เอาต์พุต/รวม)
- ต้นทุนประมาณการเมื่อกำหนดราคาโมเดลไว้ (`models.providers.*.models[].cost`)
- `sessionKey`, `sessionId` และเส้นทาง transcript เพื่อให้เอเจนต์หลักดึงประวัติผ่าน `sessions_history` หรือตรวจสอบไฟล์บนดิสก์ได้

เมทาดาทาภายในมีไว้สำหรับการประสานงานเท่านั้น; การตอบกลับที่ผู้ใช้เห็น
ควรถูกเขียนใหม่ด้วยน้ำเสียงผู้ช่วยตามปกติ

### เหตุผลที่ควรใช้ `sessions_history`

`sessions_history` เป็นเส้นทางการประสานงานที่ปลอดภัยกว่า:

- การระลึกของผู้ช่วยถูกทำให้เป็นมาตรฐานก่อน: ตัดแท็กการคิดออก; ตัดโครง `<relevant-memories>` / `<relevant_memories>` ออก; ตัดบล็อกเพย์โหลด XML การเรียกเครื่องมือแบบข้อความล้วน (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) ออก รวมถึงเพย์โหลดที่ถูกตัดทอนและไม่เคยปิดอย่างสมบูรณ์; ตัดโครง scaffolding ของการเรียก/ผลลัพธ์เครื่องมือที่ถูกลดระดับและตัวทำเครื่องหมายบริบทประวัติออก; ตัดโทเค็นควบคุมโมเดลที่รั่ว (`<|assistant|>`, ASCII อื่นที่เป็น `<|...|>`, แบบเต็มความกว้าง `<｜...｜>`) ออก; ตัด XML การเรียกเครื่องมือ MiniMax ที่ผิดรูปออก
- ข้อความที่ดูเหมือนข้อมูลรับรอง/โทเค็นจะถูกปกปิด
- บล็อกยาวสามารถถูกตัดทอนได้
- ประวัติที่ใหญ่มากสามารถทิ้งแถวเก่ากว่า หรือแทนที่แถวที่ใหญ่เกินด้วย `[sessions_history omitted: message too large]`
- การตรวจสอบ transcript ดิบบนดิสก์เป็น fallback เมื่อคุณต้องการ transcript แบบ byte-for-byte ครบถ้วน

## นโยบายเครื่องมือ

เอเจนต์ย่อยใช้โปรไฟล์และไปป์ไลน์นโยบายเครื่องมือเดียวกับเอเจนต์แม่หรือ
เอเจนต์เป้าหมายก่อน หลังจากนั้น OpenClaw จะใช้ชั้นข้อจำกัด
ของเอเจนต์ย่อย

เมื่อไม่มี `tools.profile` ที่จำกัด เอเจนต์ย่อยจะได้รับ **เครื่องมือทั้งหมด ยกเว้น
เครื่องมือเซสชัน** และเครื่องมือระบบ:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

`sessions_history` ยังคงเป็นมุมมองการระลึกที่ถูกจำกัดขอบเขตและทำให้ปลอดภัยที่นี่ด้วย —
ไม่ใช่การ dump transcript ดิบ

เมื่อ `maxSpawnDepth >= 2` เอเจนต์ย่อยตัวประสานงานที่ความลึก 1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ
`sessions_history` เพิ่มเติม เพื่อให้จัดการลูกของตนได้

### แทนที่ผ่านการกำหนดค่า

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny wins
        deny: ["gateway", "cron"],
        // if allow is set, it becomes allow-only (deny still wins)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` เป็นตัวกรอง allow-only ขั้นสุดท้าย สามารถจำกัด
ชุดเครื่องมือที่แก้ค่าแล้วให้แคบลง แต่ไม่สามารถ **เพิ่มกลับ** เครื่องมือที่ถูกนำออก
โดย `tools.profile` ได้ ตัวอย่างเช่น `tools.profile: "coding"` รวม
`web_search`/`web_fetch` แต่ไม่รวมเครื่องมือ `browser` เพื่อให้
เอเจนต์ย่อยโปรไฟล์ coding ใช้ browser automation ได้ ให้เพิ่ม browser ที่
ขั้นโปรไฟล์:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

ใช้ `agents.list[].tools.alsoAllow: ["browser"]` ต่อเอเจนต์เมื่อมีเพียงเอเจนต์เดียว
ที่ควรได้รับ browser automation

## ภาวะพร้อมกัน

เอเจนต์ย่อยใช้เลนคิวในกระบวนการเฉพาะ:

- **ชื่อเลน:** `subagent`
- **ภาวะพร้อมกัน:** `agents.defaults.subagents.maxConcurrent` (ค่าเริ่มต้น `8`)

## ความมีชีวิตและการกู้คืน

OpenClaw ไม่ถือว่าการไม่มี `endedAt` เป็นหลักฐานถาวรว่า
เอเจนต์ย่อยยังมีชีวิตอยู่ รันที่ยังไม่จบซึ่งเก่ากว่าหน้าต่างรันค้าง
จะเลิกถูกนับว่า active/pending ใน `/subagents list`, สรุปสถานะ,
ตัวกั้นการเสร็จสิ้นของลูกหลาน และการตรวจสอบภาวะพร้อมกันต่อเซสชัน

หลังการรีสตาร์ต Gateway รันที่กู้คืนมาและยังไม่จบซึ่งค้างอยู่จะถูกตัดออก เว้นแต่
เซสชันลูกของรันนั้นถูกทำเครื่องหมาย `abortedLastRun: true` เซสชันลูกที่ถูกยกเลิกโดย
การรีสตาร์ตเหล่านั้นยังคงกู้คืนได้ผ่านโฟลว์กู้คืนเอเจนต์ย่อยกำพร้า ซึ่งส่งข้อความ resume
สังเคราะห์ก่อนล้างเครื่องหมายถูกยกเลิก

การกู้คืนหลังรีสตาร์ตอัตโนมัติถูกจำกัดต่อเซสชันลูก ถ้าเอเจนต์ย่อยลูกตัวเดิม
ถูกยอมรับสำหรับการกู้คืนกำพร้าซ้ำๆ ภายในหน้าต่าง rapid re-wedge,
OpenClaw จะบันทึก tombstone การกู้คืนไว้บนเซสชันนั้นและหยุด resume อัตโนมัติ
ให้ในการรีสตาร์ตครั้งถัดไป เรียกใช้
`openclaw tasks maintenance --apply` เพื่อปรับระเบียนงานให้สอดคล้องกัน หรือ
`openclaw doctor --fix` เพื่อล้างแฟล็กการกู้คืนที่ถูกยกเลิกและค้างอยู่บน
เซสชันที่มี tombstone

<Note>
ถ้าการ spawn เอเจนต์ย่อยล้มเหลวด้วย Gateway `PAIRING_REQUIRED` /
`scope-upgrade` ให้ตรวจสอบตัวเรียก RPC ก่อนแก้ไขสถานะการจับคู่
การประสานงาน `sessions_spawn` ภายในควรเชื่อมต่อเป็น
`client.id: "gateway-client"` พร้อม `client.mode: "backend"` ผ่าน

## การหยุด

- การส่ง `/stop` ในแชตของผู้ร้องขอจะยกเลิกเซสชันผู้ร้องขอและหยุดรันเอเจนต์ย่อยที่กำลังทำงานซึ่งถูก spawn จากเซสชันนั้น โดยทอดต่อไปยังลูกที่ซ้อนอยู่
- `/subagents kill <id>` หยุดเอเจนต์ย่อยที่ระบุและทอดต่อไปยังลูกของเอเจนต์นั้น

## ข้อจำกัด

- การประกาศของเอเจนต์ย่อยเป็นแบบ **พยายามให้ดีที่สุด** ถ้า Gateway รีสตาร์ต งาน "ประกาศกลับ" ที่ค้างอยู่จะสูญหาย
- เอเจนต์ย่อยยังคงใช้ทรัพยากรกระบวนการ Gateway เดียวกัน; ให้มอง `maxConcurrent` เป็นวาล์วนิรภัย
- `sessions_spawn` ไม่บล็อกเสมอ: จะส่งคืน `{ status: "accepted", runId, childSessionKey }` ทันที
- บริบทเอเจนต์ย่อยฉีดเฉพาะ `AGENTS.md`, `TOOLS.md`, `SOUL.md`, `IDENTITY.md` และ `USER.md` (ไม่มี `MEMORY.md`, `HEARTBEAT.md` หรือ `BOOTSTRAP.md`)
- ความลึกการซ้อนสูงสุดคือ 5 (ช่วง `maxSpawnDepth`: 1–5) แนะนำให้ใช้ความลึก 2 สำหรับกรณีใช้งานส่วนใหญ่
- `maxChildrenPerAgent` จำกัดจำนวนลูกที่กำลังทำงานต่อเซสชัน (ค่าเริ่มต้น `5`, ช่วง `1–20`)
