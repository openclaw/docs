Sub-agent คือการทำงานของเอเจนต์เบื้องหลังที่เริ่มจากการทำงานของเอเจนต์ที่มีอยู่
แต่ละรายการทำงานในเซสชันของตนเอง (`agent:<agentId>:subagent:<uuid>`) และ
เมื่อเสร็จแล้วจะ **ประกาศ** ผลลัพธ์กลับไปยังช่องแชทของผู้ร้องขอ
การทำงานของ sub-agent ทุกรายการจะถูกติดตามเป็น [งานเบื้องหลัง](/th/automation/tasks)

เป้าหมาย:

- ทำงานวิจัย งานที่ใช้เวลานาน และงานเครื่องมือที่ทำงานช้าแบบขนาน โดยไม่ขัดขวางการทำงานหลัก
- แยก sub-agent ออกจากกันโดยค่าเริ่มต้น (แยกเซสชัน และเลือกใช้ sandbox ได้)
- ทำให้ชุดเครื่องมือใช้งานผิดได้ยาก: โดยค่าเริ่มต้น sub-agent จะ **ไม่ได้รับ** เครื่องมือเซสชันหรือข้อความ
- รองรับความลึกของการซ้อนที่กำหนดค่าได้สำหรับรูปแบบตัวประสานงาน

<Note>
**หมายเหตุเรื่องค่าใช้จ่าย:** โดยค่าเริ่มต้น sub-agent แต่ละรายการมีบริบทและการใช้โทเค็น
ของตนเอง สำหรับงานหนักหรืองานซ้ำ ให้ตั้งค่าโมเดลที่มีต้นทุนต่ำกว่าสำหรับ sub-agent
และคงเอเจนต์หลักไว้บนโมเดลคุณภาพสูงกว่าผ่าน
`agents.defaults.subagents.model` หรือการเขียนทับรายเอเจนต์ เมื่อเอเจนต์ลูก
จำเป็นต้องใช้บทสนทนาปัจจุบันของผู้ร้องขอจริง ๆ ให้เริ่มเอเจนต์ด้วย
`context: "fork"` เซสชัน sub-agent ที่ผูกกับเธรดใช้ค่าเริ่มต้นเป็น
`context: "fork"` เนื่องจากจะแยกแขนงบทสนทนาปัจจุบันไปยัง
เธรดติดตามผล
</Note>

## คำสั่ง Slash

`/subagents` ใช้ตรวจสอบการทำงานของ sub-agent สำหรับ **เซสชันปัจจุบัน**:

```text
/subagents list
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
```

`/subagents info` แสดงข้อมูลเมตาของการทำงาน (สถานะ การประทับเวลา รหัสเซสชัน
พาธบทสนทนา การล้างข้อมูล) `/subagents log` แสดงช่วงการสนทนาล่าสุดสำหรับ
การทำงานนั้น เพิ่มโทเค็น `tools` เพื่อรวมข้อความการเรียกใช้เครื่องมือ/ผลลัพธ์ (ละเว้น
โดยค่าเริ่มต้น) ใช้ `sessions_history` เพื่อดูข้อมูลย้อนหลังแบบจำกัดขอบเขตและกรองด้านความปลอดภัย
จากภายในรอบการทำงานของเอเจนต์ หรือตรวจสอบพาธบทสนทนาบนดิสก์เพื่อดู
บทสนทนาดิบฉบับเต็ม

ใน Control UI เซสชันแม่ที่มีการทำงานของเอเจนต์ลูกเมื่อไม่นานมานี้จะมีแถวในแถบด้านข้าง
ที่ขยายได้ แถวซ้อนจะแสดงสถานะและเวลาทำงานของเอเจนต์ลูก และเมื่อเลือกรายการใด
จะเปิดแชทของเอเจนต์ลูกนั้นโดยยังคงลำดับชั้นของเซสชันแม่ไว้

### การควบคุมการผูกเธรด

คำสั่งเหล่านี้ทำงานบนช่องที่รองรับการผูกเธรดแบบถาวร ดู
[ช่องที่รองรับเธรด](#thread-supporting-channels) ด้านล่าง

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### พฤติกรรมการเริ่มทำงาน

เอเจนต์เริ่ม sub-agent เบื้องหลังด้วยเครื่องมือ `sessions_spawn`
ผลลัพธ์เมื่อเสร็จสิ้นจะส่งกลับเป็นเหตุการณ์ภายในเซสชันแม่ เอเจนต์แม่/ผู้ร้องขอ
เป็นผู้ตัดสินใจว่าจำเป็นต้องแจ้งข้อมูลอัปเดตแก่ผู้ใช้หรือไม่

## โหมดบริบท

sub-agent แบบเนทีฟจะเริ่มโดยแยกออกจากกัน เว้นแต่ผู้เรียกจะร้องขอให้แยกแขนง
บทสนทนาปัจจุบันอย่างชัดเจน

| โหมด       | ควรใช้เมื่อ                                                                                                                         | พฤติกรรม                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | การวิจัยใหม่ การนำไปใช้โดยอิสระ งานเครื่องมือที่ทำงานช้า หรือสิ่งใดก็ตามที่อธิบายโดยย่อในข้อความงานได้                           | สร้างบทสนทนาเอเจนต์ลูกใหม่ที่สะอาด นี่คือค่าเริ่มต้นและช่วยลดการใช้โทเค็น  |
| `fork`     | งานที่ต้องอาศัยบทสนทนาปัจจุบัน ผลลัพธ์เครื่องมือก่อนหน้า หรือคำสั่งที่มีรายละเอียดอ่อนละเอียดซึ่งมีอยู่แล้วในบทสนทนาของผู้ร้องขอ | แยกแขนงบทสนทนาของผู้ร้องขอไปยังเซสชันเอเจนต์ลูกก่อนที่เอเจนต์ลูกจะเริ่มทำงาน |

ใช้ `fork` เท่าที่จำเป็น มีไว้สำหรับการมอบหมายงานที่อ่อนไหวต่อบริบท ไม่ใช่
สิ่งทดแทนการเขียนพรอมต์งานที่ชัดเจน

## เครื่องมือ: `sessions_spawn`

เริ่มการทำงานของ sub-agent ด้วย `deliver: false` บนเลน `subagent` ส่วนกลาง
จากนั้นดำเนินขั้นตอนประกาศและโพสต์ข้อความตอบกลับการประกาศไปยังช่องแชท
ของผู้ร้องขอ

ความพร้อมใช้งานขึ้นอยู่กับนโยบายเครื่องมือที่มีผลของผู้เรียก โปรไฟล์ในตัว
`coding` และ `messaging` มี `sessions_spawn`,
`sessions_yield` และ `subagents` ส่วน `minimal` ไม่มี `full` อนุญาต
เครื่องมือทั้งหมด เพิ่มเครื่องมือเหล่านั้นด้วย `tools.alsoAllow` หรือใช้โปรไฟล์ใดโปรไฟล์หนึ่ง
ข้างต้น สำหรับเอเจนต์ที่ใช้โปรไฟล์กำหนดเองซึ่งแคบกว่าแต่ยังควร
มอบหมายงานได้
นโยบายอนุญาต/ปฏิเสธของช่อง/กลุ่ม ผู้ให้บริการ sandbox และรายเอเจนต์
ยังสามารถนำเครื่องมือออกได้หลังขั้นตอนโปรไฟล์ ใช้ `/tools` จากเซสชันเดียวกัน
เพื่อยืนยันรายการเครื่องมือที่มีผล

**ค่าเริ่มต้น:**

- **โมเดล:** sub-agent แบบเนทีฟสืบทอดจากผู้เรียก เว้นแต่จะตั้งค่า `agents.defaults.subagents.model` (หรือ `agents.list[].subagents.model` รายเอเจนต์) การเริ่มรันไทม์ ACP ใช้โมเดล sub-agent ที่กำหนดค่าไว้เดียวกันเมื่อมี มิฉะนั้นชุดทดสอบ ACP จะคงค่าเริ่มต้นของตนเองไว้ ค่า `sessions_spawn.model` ที่ระบุอย่างชัดเจนยังคงมีสิทธิ์เหนือกว่า
- **การคิด:** sub-agent แบบเนทีฟสืบทอดจากผู้เรียก เว้นแต่จะตั้งค่า `agents.defaults.subagents.thinking` (หรือ `agents.list[].subagents.thinking` รายเอเจนต์) การเริ่มรันไทม์ ACP จะใช้ `agents.defaults.models["provider/model"].params.thinking` กับโมเดลที่เลือกด้วย ค่า `sessions_spawn.thinking` ที่ระบุอย่างชัดเจนยังคงมีสิทธิ์เหนือกว่า
- **หมดเวลาการทำงาน:** OpenClaw ใช้ `agents.defaults.subagents.runTimeoutSeconds` เมื่อตั้งค่าไว้ มิฉะนั้นจะย้อนกลับไปใช้ `0` (ไม่หมดเวลา) `sessions_spawn` ไม่ยอมรับการเขียนทับเวลาหมดอายุรายครั้ง
- **อายุการทำงานของกระบวนการ:** sub-agent ของ OpenClaw ที่แยกออกมามีวงจรชีวิตการทำงานของตนเอง งานเบื้องหลังที่สร้างภายในแบ็กเอนด์ CLI ภายนอกแตกต่างออกไป โดยใช้กระบวนการย่อย CLI ร่วมกับเอเจนต์แม่และหยุดทำงานหากเอเจนต์แม่นั้นถึง `agents.defaults.timeoutSeconds`
- **การส่งมอบงาน:** sub-agent แบบเนทีฟได้รับงานที่มอบหมายในข้อความ `[Subagent Task]` แรกที่มองเห็นได้ พรอมต์ระบบของ sub-agent มีข้อกำหนดรันไทม์และบริบทการกำหนดเส้นทาง ไม่ใช่สำเนางานที่ซ่อนและซ้ำกัน

การเริ่ม sub-agent แบบเนทีฟที่ได้รับการยอมรับจะมีข้อมูลเมตาโมเดลเอเจนต์ลูกที่แก้ไขแล้ว
ในผลลัพธ์เครื่องมือ: `resolvedModel` มีการอ้างอิงโมเดลที่นำไปใช้ และ
`resolvedProvider` มีคำนำหน้าผู้ให้บริการเมื่อการอ้างอิงมีคำนำหน้าดังกล่าว

### โหมดพรอมต์การมอบหมายงาน

`agents.defaults.subagents.delegationMode` ควบคุมเฉพาะคำแนะนำในพรอมต์ โดยไม่เปลี่ยนนโยบายเครื่องมือหรือบังคับใช้การมอบหมายงาน

- `suggest` (ค่าเริ่มต้น): คงคำแนะนำมาตรฐานในพรอมต์ให้ใช้ sub-agent สำหรับงานที่ใหญ่กว่าหรือช้ากว่า
- `prefer`: บอกเอเจนต์หลักให้ตอบสนองได้ต่อเนื่องและมอบหมายทุกสิ่งที่ซับซ้อนกว่าการตอบกลับโดยตรงผ่าน `sessions_spawn`

การเขียนทับรายเอเจนต์: `agents.list[].subagents.delegationMode`

```json5
{
  agents: {
    defaults: {
      subagents: {
        delegationMode: "prefer",
        maxConcurrent: 4,
      },
    },
    list: [
      {
        id: "coordinator",
        subagents: { delegationMode: "prefer" },
      },
    ],
  },
}
```

### ชื่องานและการกำหนดเป้าหมาย

`taskName` เป็นชื่ออ้างอิงสำหรับโมเดลเพื่อใช้ในการควบคุมการทำงาน ไม่ใช่คีย์เซสชัน
ใช้สำหรับชื่อเอเจนต์ลูกแบบคงที่ เช่น `review_subagents`,
`linux_validation` หรือ `docs_update` เมื่อผู้ประสานงานอาจต้องตรวจสอบ
เอเจนต์ลูกนั้นในภายหลัง

การแก้ไขเป้าหมายรองรับการจับคู่ `taskName` แบบตรงทั้งหมดและคำนำหน้าที่ไม่กำกวม
การจับคู่จำกัดขอบเขตอยู่ในหน้าต่างเป้าหมายที่ใช้งานอยู่/ล่าสุดเดียวกับที่ใช้
โดยเป้าหมาย `/subagents` แบบมีหมายเลข ดังนั้นเอเจนต์ลูกที่เสร็จสิ้นไปนานแล้วจะไม่ทำให้
ชื่ออ้างอิงที่นำกลับมาใช้ใหม่กำกวม หากเอเจนต์ลูกที่ใช้งานอยู่หรือล่าสุดสองตัวใช้
`taskName` เดียวกัน เป้าหมายจะกำกวม ให้ใช้ดัชนีรายการ คีย์เซสชัน หรือ
รหัสการรันแทน

เป้าหมายสงวน `last` และ `all` ใช้เป็นค่า `taskName` ไม่ได้
เนื่องจากมีความหมายสำหรับการควบคุมอยู่แล้ว

## เครื่องมือ: `sessions_yield`

จบเทิร์นปัจจุบันของโมเดลและรอให้เหตุการณ์รันไทม์ ซึ่งหลัก ๆ คือ
เหตุการณ์การทำงานเสร็จสิ้นของเอเจนต์ย่อย มาถึงเป็นข้อความถัดไป ใช้หลังจาก
สร้างงานของเอเจนต์ลูกที่จำเป็นแล้ว เมื่อผู้ร้องขอยังไม่สามารถให้คำตอบสุดท้าย
จนกว่างานเหล่านั้นจะเสร็จสิ้น

`sessions_yield` เป็นกลไกพื้นฐานสำหรับการรอ อย่าแทนที่ด้วยลูปการตรวจสอบซ้ำ
ผ่าน `subagents`, `sessions_list`, `sessions_history`, เชลล์
`sleep` หรือการตรวจสอบกระบวนการ เพียงเพื่อตรวจจับว่าเอเจนต์ลูกทำงานเสร็จแล้วหรือไม่

ใช้ `sessions_yield` เฉพาะเมื่อรายการเครื่องมือที่มีผลของเซสชันมีเครื่องมือนี้
โปรไฟล์เครื่องมือแบบขั้นต่ำหรือแบบกำหนดเองบางรายการอาจเปิดให้ใช้ `sessions_spawn` และ
`subagents` โดยไม่เปิดให้ใช้ `sessions_yield`; ในกรณีนั้น อย่าสร้าง
ลูปการตรวจสอบซ้ำขึ้นมาเพียงเพื่อรอให้งานเสร็จ

เมื่อมีเอเจนต์ลูกที่ใช้งานอยู่ OpenClaw จะแทรกบล็อกพรอมต์ `Active Subagents`
แบบกะทัดรัดที่สร้างโดยรันไทม์ลงในเทิร์นปกติ เพื่อให้ผู้ร้องขอเห็น
เซสชันเอเจนต์ลูกปัจจุบัน รหัสการรัน สถานะ ป้ายกำกับ งาน และ
นามแฝง `taskName` โดยไม่ต้องตรวจสอบซ้ำ ฟิลด์งานและป้ายกำกับใน
บล็อกนั้นจะอยู่ในเครื่องหมายอัญประกาศในฐานะข้อมูล ไม่ใช่คำสั่ง เนื่องจากอาจมาจาก
อาร์กิวเมนต์การสร้างที่ผู้ใช้/โมเดลระบุ

## เครื่องมือ: `subagents`

แสดงรายการการรันเอเจนต์ย่อยที่สร้างขึ้นและระเบียนงานเบื้องหลังซึ่งเป็นของ
โครงสร้างเซสชันของผู้ร้องขอ แถวงานครอบคลุมเอเจนต์ย่อยแบบเนทีฟ การรัน ACP
งาน CLI/สื่อของ Gateway และการดำเนินการ Cron ขอบเขตจำกัดอยู่ที่ผู้ร้องขอปัจจุบัน
เอเจนต์ลูกจะเห็นได้เฉพาะเอเจนต์ลูกภายใต้การควบคุมของตนเอง

ใช้ `subagents` สำหรับตรวจสอบสถานะและแก้ไขข้อบกพร่องตามต้องการ ใช้ `sessions_yield` เพื่อ
รอเหตุการณ์การทำงานเสร็จสิ้น

ใช้ `action: "cancel"` พร้อม `taskId` ที่ `action: "list"` ส่งคืนเพื่อหยุด
งาน การยกเลิกจำกัดอยู่ในโครงสร้างเซสชันภายใต้การควบคุม เอเจนต์ย่อยปลายทาง
ไม่สามารถยกเลิกงานที่เป็นของเซสชันอื่นได้

## เซสชันที่ผูกกับเธรด

เมื่อเปิดใช้งานการผูกเธรดสำหรับช่องทาง เอเจนต์ย่อยสามารถคงการผูก
กับเธรดไว้ เพื่อให้ข้อความติดตามผลของผู้ใช้ในเธรดนั้นยังคงถูกส่งต่อไปยัง
เซสชันเอเจนต์ย่อยเดิม

### ช่องทางที่รองรับเธรด

ช่องทางรองรับเซสชันเอเจนต์ย่อยแบบถาวรที่ผูกกับเธรด
(`sessions_spawn` พร้อม `thread: true`) เมื่อลงทะเบียนอะแดปเตอร์
การผูกบทสนทนา ช่องทางที่รวมมาและรองรับคุณสมบัตินี้ ได้แก่ **Discord**,
**iMessage**, **Matrix** และ **Telegram** โดยค่าเริ่มต้น Discord และ Matrix
จะสร้างเธรดลูก ส่วน Telegram และ iMessage จะผูกกับ
บทสนทนาปัจจุบัน ใช้คีย์การกำหนดค่า `threadBindings` ของแต่ละช่องทางสำหรับ
การเปิดใช้งาน การหมดเวลา และ `spawnSessions`

### ขั้นตอนแบบย่อ

<Steps>
  <Step title="สร้าง">
    `sessions_spawn` พร้อม `thread: true` (และ `mode: "session"` หากต้องการ)
  </Step>
  <Step title="ผูก">
    OpenClaw สร้างหรือผูกเธรดกับเป้าหมายเซสชันนั้นในช่องทางที่ใช้งานอยู่
  </Step>
  <Step title="กำหนดเส้นทางข้อความติดตามผล">
    การตอบกลับและข้อความติดตามผลในเธรดนั้นจะถูกส่งไปยังเซสชันที่ผูกไว้
  </Step>
  <Step title="ตรวจสอบการหมดเวลา">
    ใช้ `/session idle` เพื่อตรวจสอบ/อัปเดตการยกเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน และ
    `/session max-age` เพื่อควบคุมขีดจำกัดสูงสุดแบบตายตัว
  </Step>
  <Step title="ยกเลิกการผูก">
    ใช้ `/unfocus` เพื่อยกเลิกการผูกด้วยตนเอง
  </Step>
</Steps>

### การควบคุมด้วยตนเอง

| คำสั่ง            | ผลลัพธ์                                                                                    |
| ------------------ | ----------------------------------------------------------------------------------------- |
| `/focus <target>`  | ผูกเธรดปัจจุบัน (หรือสร้างเธรด) กับเป้าหมายเอเจนต์ย่อย/เซสชัน                     |
| `/unfocus`         | ลบการผูกสำหรับเธรดที่ผูกอยู่ในปัจจุบัน                                           |
| `/agents`          | แสดงรายการการรันที่ใช้งานอยู่และสถานะการผูก (`binding:<id>`, `unbound` หรือ `bindings unavailable`) |
| `/session idle`    | ตรวจสอบ/อัปเดตการยกเลิกโฟกัสอัตโนมัติเมื่อไม่มีการใช้งาน (เฉพาะเธรดที่ผูกและมีโฟกัส)                             |
| `/session max-age` | ตรวจสอบ/อัปเดตขีดจำกัดสูงสุดแบบตายตัว (เฉพาะเธรดที่ผูกและมีโฟกัส)                                      |

### สวิตช์การกำหนดค่า

- **ค่าเริ่มต้นส่วนกลาง:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`
- **คีย์การแทนที่ของช่องทางและการผูกอัตโนมัติเมื่อสร้าง** เป็นค่าเฉพาะของอะแดปเตอร์ ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านบน

ดูรายละเอียดอะแดปเตอร์ปัจจุบันได้ที่ [ข้อมูลอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference) และ
[คำสั่ง Slash](/th/tools/slash-commands)

### รายการที่อนุญาต

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  รายการรหัสเอเจนต์ที่กำหนดค่าไว้ซึ่งสามารถกำหนดเป็นเป้าหมายผ่าน `agentId` อย่างชัดเจน (`["*"]` อนุญาตเป้าหมายที่กำหนดค่าไว้ทั้งหมด) ค่าเริ่มต้น: เฉพาะเอเจนต์ผู้ร้องขอ หากกำหนดรายการและยังต้องการให้ผู้ร้องขอสร้างตัวเองด้วย `agentId` ให้รวมรหัสผู้ร้องขอไว้ในรายการ
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  รายการที่อนุญาตของเอเจนต์เป้าหมายที่กำหนดค่าไว้ตามค่าเริ่มต้น ซึ่งใช้เมื่อเอเจนต์ผู้ร้องขอไม่ได้กำหนด `subagents.allowAgents` ของตนเอง
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  บล็อกการเรียก `sessions_spawn` ที่ละ `agentId` ไว้ (บังคับให้เลือกโปรไฟล์อย่างชัดเจน) การแทนที่รายเอเจนต์: `agents.list[].subagents.requireAgentId`
</ParamField>
<ParamField path="agents.defaults.subagents.announceTimeoutMs" type="number" default="120000">
  การหมดเวลาต่อการเรียกสำหรับความพยายามส่งประกาศ `agent` ของ Gateway ค่าเป็นจำนวนเต็มบวกในหน่วยมิลลิวินาทีและจะถูกจำกัดไว้ที่ค่าสูงสุดของตัวจับเวลาที่ปลอดภัยสำหรับแพลตฟอร์ม การลองใหม่จากข้อผิดพลาดชั่วคราวอาจทำให้เวลารอประกาศทั้งหมดนานกว่าค่าการหมดเวลาที่กำหนดไว้หนึ่งครั้ง
</ParamField>

หากเซสชันผู้ร้องขออยู่ในแซนด์บ็อกซ์ `sessions_spawn` จะปฏิเสธเป้าหมาย
ที่จะทำงานนอกแซนด์บ็อกซ์

### การค้นหา

ใช้ `agents_list` เพื่อดูว่ารหัสเอเจนต์ใดได้รับอนุญาตในปัจจุบันสำหรับ
`sessions_spawn` การตอบกลับประกอบด้วยโมเดลที่มีผลและข้อมูลเมตารันไทม์แบบฝัง
ของเอเจนต์แต่ละรายการ เพื่อให้ผู้เรียกสามารถแยกแยะ OpenClaw, เซิร์ฟเวอร์แอป Codex
และรันไทม์แบบเนทีฟอื่น ๆ ที่กำหนดค่าไว้ได้

รายการ `allowAgents` ต้องชี้ไปยัง ID ของเอเจนต์ที่กำหนดค่าไว้ใน `agents.list[]`
`["*"]` หมายถึงเอเจนต์เป้าหมายใดๆ ที่กำหนดค่าไว้ รวมถึงผู้ร้องขอ หากลบการกำหนดค่าเอเจนต์
แต่ ID ของเอเจนต์นั้นยังคงอยู่ใน `allowAgents` ระบบ `sessions_spawn` จะปฏิเสธ ID นั้น
และ `agents_list` จะไม่นำ ID ดังกล่าวมารวม เรียกใช้ `openclaw doctor --fix` เพื่อล้าง
รายการ allowlist ที่ล้าสมัย หรือเพิ่มรายการ `agents.list[]` ขั้นต่ำเมื่อเป้าหมายควร
ยังคงสามารถถูกสร้างได้โดยสืบทอดค่าเริ่มต้น

### การเก็บถาวรอัตโนมัติ

- เซสชันของเอเจนต์ย่อยจะถูกเก็บถาวรโดยอัตโนมัติหลังจาก `agents.defaults.subagents.archiveAfterMinutes` (ค่าเริ่มต้น `60`)
- การเก็บถาวรใช้ `sessions.delete` และเปลี่ยนชื่อทรานสคริปต์เป็น `*.deleted.<timestamp>` (อยู่ในโฟลเดอร์เดิม)
- `cleanup: "delete"` จะเก็บถาวรทันทีหลังจากประกาศ (ยังคงเก็บทรานสคริปต์ไว้ด้วยการเปลี่ยนชื่อ)
- การเก็บถาวรอัตโนมัติเป็นแบบพยายามให้ดีที่สุด ตัวจับเวลาที่ยังค้างอยู่จะสูญหายหาก Gateway เริ่มทำงานใหม่
- การหมดเวลาการทำงานที่กำหนดค่าไว้จะ **ไม่** เก็บถาวรโดยอัตโนมัติ แต่จะหยุดเฉพาะการทำงานเท่านั้น เซสชันจะยังคงอยู่จนกว่าจะมีการเก็บถาวรอัตโนมัติ
- การเก็บถาวรอัตโนมัติใช้กับทั้งเซสชันระดับความลึก 1 และระดับความลึก 2 อย่างเท่าเทียมกัน
- การล้างเบราว์เซอร์แยกจากการล้างข้อมูลเก็บถาวร โดยระบบจะพยายามปิดแท็บ/โปรเซสของเบราว์เซอร์ที่ติดตามไว้เมื่อการทำงานเสร็จสิ้น แม้ว่าจะเก็บทรานสคริปต์/ระเบียนเซสชันไว้ก็ตาม

## เอเจนต์ย่อยแบบซ้อน

ตามค่าเริ่มต้น เอเจนต์ย่อยไม่สามารถสร้างเอเจนต์ย่อยของตนเองได้
(`maxSpawnDepth: 1`) ตั้งค่า `maxSpawnDepth: 2` เพื่อเปิดใช้งานการซ้อนหนึ่งระดับ
ซึ่งเป็น **รูปแบบออร์เคสเตรเตอร์**: เอเจนต์หลัก → เอเจนต์ย่อยออร์เคสเตรเตอร์ →
เอเจนต์ย่อยซ้อนระดับผู้ปฏิบัติงาน

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // อนุญาตให้เอเจนต์ย่อยสร้างเอเจนต์ลูก (ค่าเริ่มต้น: 1, ช่วง 1-5)
        maxChildrenPerAgent: 5, // จำนวนเอเจนต์ลูกที่ทำงานอยู่สูงสุดต่อเซสชันเอเจนต์ (ค่าเริ่มต้น: 5, ช่วง 1-20)
        maxConcurrent: 8, // ขีดจำกัดเลนการทำงานพร้อมกันโดยรวม (ค่าเริ่มต้น: 8)
        runTimeoutSeconds: 900, // เวลาหมดอายุเริ่มต้นสำหรับ sessions_spawn (0 = ไม่หมดเวลา)
        announceTimeoutMs: 120000, // เวลาหมดอายุการประกาศของ Gateway ต่อการเรียก
      },
    },
  },
}
```

### ระดับความลึก

| ความลึก | รูปแบบคีย์เซสชัน                            | บทบาท                                          | สร้างได้หรือไม่                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | เอเจนต์หลัก                                    | ได้เสมอ                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | เอเจนต์ย่อย (ออร์เคสเตรเตอร์เมื่ออนุญาตระดับความลึก 2) | เฉพาะเมื่อ `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | เอเจนต์ย่อยซ้อน (ผู้ปฏิบัติงานปลายทาง)                   | ไม่ได้                        |

### ลำดับการประกาศ

ผลลัพธ์ไหลย้อนกลับขึ้นไปตามลำดับ:

1. ผู้ปฏิบัติงานระดับความลึก 2 ทำงานเสร็จ → ประกาศไปยังเอเจนต์แม่ (ออร์เคสเตรเตอร์ระดับความลึก 1)
2. ออร์เคสเตรเตอร์ระดับความลึก 1 ได้รับประกาศ สังเคราะห์ผลลัพธ์ และทำงานเสร็จ → ประกาศไปยังเอเจนต์หลัก
3. เอเจนต์หลักได้รับประกาศและส่งมอบให้ผู้ใช้

แต่ละระดับจะเห็นเฉพาะประกาศจากเอเจนต์ลูกโดยตรงเท่านั้น

<Note>
**คำแนะนำด้านการปฏิบัติงาน:** เริ่มงานของเอเจนต์ลูกหนึ่งครั้งแล้วรอเหตุการณ์
การทำงานเสร็จ แทนการสร้างลูปสำรวจสถานะด้วย `sessions_list`,
`sessions_history`, `/subagents list` หรือคำสั่งพัก `exec`
`sessions_list` และ `/subagents list` ทำให้ความสัมพันธ์ของเซสชันลูก
มุ่งเน้นงานที่ยังทำอยู่ โดยเอเจนต์ลูกที่ยังทำงานจะยังคงเชื่อมโยงอยู่ เอเจนต์ลูกที่สิ้นสุดแล้วจะยัง
มองเห็นได้ในช่วงเวลาล่าสุดระยะสั้น และลิงก์เอเจนต์ลูกที่ล้าสมัยซึ่งมีเฉพาะในพื้นที่จัดเก็บจะ
ถูกละเว้นหลังพ้นช่วงความสดใหม่ วิธีนี้ป้องกันไม่ให้ข้อมูลเมตา `spawnedBy` /
`parentSessionKey` เก่าปลุกเอเจนต์ลูกเงาขึ้นมาอีกหลัง
เริ่มทำงานใหม่ หากเหตุการณ์การทำงานเสร็จของเอเจนต์ลูกมาถึงหลังจากส่ง
คำตอบสุดท้ายไปแล้ว การติดตามผลที่ถูกต้องคือโทเค็นเงียบแบบตรงตัว
`NO_REPLY` / `no_reply`
</Note>

### นโยบายเครื่องมือตามระดับความลึก

- บทบาทและขอบเขตการควบคุมจะถูกเขียนลงในข้อมูลเมตาของเซสชันขณะสร้าง ซึ่งช่วยป้องกันไม่ให้คีย์เซสชันแบบแบนหรือที่กู้คืนแล้วได้รับสิทธิ์ของออร์เคสเตรเตอร์กลับคืนโดยไม่ตั้งใจ
- **ระดับความลึก 1 (ออร์เคสเตรเตอร์ เมื่อ `maxSpawnDepth >= 2`):** ได้รับ `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` เพื่อให้สามารถสร้างเอเจนต์ลูกและตรวจสอบสถานะได้ เครื่องมือเซสชัน/ระบบอื่นๆ ยังคงถูกปฏิเสธ
- **ระดับความลึก 1 (ปลายทาง เมื่อ `maxSpawnDepth == 1`):** ไม่มีเครื่องมือเซสชัน (พฤติกรรมเริ่มต้นในปัจจุบัน)
- **ระดับความลึก 2 (ผู้ปฏิบัติงานปลายทาง):** ไม่มีเครื่องมือเซสชัน โดย `sessions_spawn` จะถูกปฏิเสธเสมอที่ระดับความลึก 2 และไม่สามารถสร้างเอเจนต์ลูกเพิ่มเติมได้

### ขีดจำกัดการสร้างต่อเอเจนต์

แต่ละเซสชันเอเจนต์ (ที่ระดับความลึกใดๆ) สามารถมีเอเจนต์ลูกที่ทำงานอยู่พร้อมกันได้สูงสุด `maxChildrenPerAgent`
(ค่าเริ่มต้น `5`) ซึ่งป้องกันการกระจายตัวที่ควบคุมไม่ได้
จากออร์เคสเตรเตอร์เพียงตัวเดียว

### การหยุดแบบลูกโซ่

การหยุดออร์เคสเตรเตอร์ระดับความลึก 1 จะหยุดเอเจนต์ลูกระดับความลึก 2
ทั้งหมดโดยอัตโนมัติ:

- `/stop` ในแชตหลักจะหยุดเอเจนต์ระดับความลึก 1 ทั้งหมด และส่งต่อการหยุดไปยังเอเจนต์ลูกระดับความลึก 2

## การยืนยันตัวตน

การยืนยันตัวตนของเอเจนต์ย่อยจะกำหนดตาม **ID ของเอเจนต์** ไม่ใช่ตามประเภทเซสชัน:

- คีย์เซสชันของเอเจนต์ย่อยคือ `agent:<agentId>:subagent:<uuid>`
- พื้นที่จัดเก็บการยืนยันตัวตนจะโหลดจาก `agentDir` ของเอเจนต์นั้น
- โปรไฟล์การยืนยันตัวตนของเอเจนต์หลักจะถูกผสานเข้ามาเป็น **ทางเลือกสำรอง** โดยโปรไฟล์ของเอเจนต์จะมีผลเหนือกว่าโปรไฟล์หลักเมื่อเกิดความขัดแย้ง

การผสานเป็นแบบเพิ่มเติม ดังนั้นโปรไฟล์หลักจึงพร้อมใช้งานเป็น
ทางเลือกสำรองเสมอ ขณะนี้ยังไม่รองรับการยืนยันตัวตนที่แยกอย่างสมบูรณ์สำหรับแต่ละเอเจนต์

## การประกาศ

เอเจนต์ย่อยรายงานกลับผ่านขั้นตอนการประกาศ:

- ขั้นตอนการประกาศทำงานภายในเซสชันเอเจนต์ย่อย (ไม่ใช่เซสชันผู้ร้องขอ)
- หากเอเจนต์ย่อยตอบกลับตรงกับ `ANNOUNCE_SKIP` ทุกประการ ระบบจะไม่โพสต์สิ่งใด
- หากข้อความล่าสุดของผู้ช่วยเป็นโทเค็นเงียบแบบตรงตัว `NO_REPLY` / `no_reply` ผลลัพธ์การประกาศจะถูกระงับ แม้ว่าก่อนหน้านั้นจะมีความคืบหน้าที่มองเห็นได้ก็ตาม

การส่งมอบขึ้นอยู่กับระดับความลึกของผู้ร้องขอ:

- เซสชันผู้ร้องขอระดับบนสุดใช้การเรียก `agent` เพื่อติดตามผลพร้อมการส่งมอบภายนอก (`deliver=true`)
- เซสชันเอเจนต์ย่อยผู้ร้องขอแบบซ้อนจะได้รับการแทรกข้อมูลติดตามผลภายใน (`deliver=false`) เพื่อให้ออร์เคสเตรเตอร์สามารถสังเคราะห์ผลลัพธ์ของเอเจนต์ลูกภายในเซสชันได้
- หากเซสชันเอเจนต์ย่อยผู้ร้องขอแบบซ้อนไม่มีอยู่แล้ว OpenClaw จะย้อนกลับไปยังผู้ร้องขอของเซสชันนั้นเมื่อมีข้อมูล

สำหรับเซสชันผู้ร้องขอระดับบนสุด การส่งมอบโดยตรงในโหมดการทำงานเสร็จ
จะระบุเส้นทางบทสนทนา/เธรดที่ผูกไว้และการแทนที่ฮุกก่อน จากนั้นจึงเติม
ฟิลด์ช่องทาง-เป้าหมายที่ขาดหายจากเส้นทางที่จัดเก็บไว้ในเซสชันผู้ร้องขอ
วิธีนี้ทำให้ผลการทำงานเสร็จอยู่ในแชต/หัวข้อที่ถูกต้อง แม้ว่าต้นทางของผลการทำงานเสร็จ
จะระบุเพียงช่องทางก็ตาม

การรวมผลการทำงานเสร็จของเอเจนต์ลูกจะจำกัดอยู่ภายในรอบการทำงานปัจจุบันของผู้ร้องขอเมื่อ
สร้างข้อค้นพบจากการทำงานเสร็จแบบซ้อน เพื่อป้องกันไม่ให้ผลลัพธ์ของเอเจนต์ลูกจากรอบก่อน
ที่ล้าสมัยรั่วไหลเข้าสู่ประกาศปัจจุบัน การตอบกลับประกาศจะรักษา
เส้นทางเธรด/หัวข้อไว้เมื่ออะแดปเตอร์ช่องทางรองรับ

### บริบทการประกาศ

บริบทการประกาศจะถูกปรับให้อยู่ในบล็อกเหตุการณ์ภายในที่เสถียร:

| ฟิลด์          | แหล่งที่มา                                                                                                   |
| -------------- | -------------------------------------------------------------------------------------------------------- |
| แหล่งที่มา         | `subagent` หรือ `cron`                                                                                     |
| ID เซสชัน    | คีย์/ID เซสชันของเอเจนต์ลูก                                                                                     |
| ประเภท           | ประเภทการประกาศ + ป้ายกำกับงาน                                                                               |
| สถานะ         | ได้มาจากผลลัพธ์รันไทม์ (`ok`, `error`, `timeout` หรือ `unknown`) — **ไม่ได้** อนุมานจากข้อความของโมเดล |
| เนื้อหาผลลัพธ์ | ข้อความล่าสุดของผู้ช่วยจากเอเจนต์ลูกที่มองเห็นได้                                                             |
| การติดตามผล      | คำสั่งที่อธิบายว่าเมื่อใดควรตอบกลับหรือควรเงียบ                                                      |

การทำงานที่สิ้นสุดด้วยความล้มเหลวจะรายงานสถานะล้มเหลวโดยไม่เล่นซ้ำข้อความ
ตอบกลับที่บันทึกไว้ ผลลัพธ์ tool/toolResult จะไม่ถูกยกระดับเป็นข้อความผลลัพธ์ของเอเจนต์ลูก

### บรรทัดสถิติ

เพย์โหลดการประกาศจะมีบรรทัดสถิติที่ส่วนท้าย (แม้ว่าจะถูกตัดบรรทัด):

- รันไทม์ (เช่น `runtime 5m12s`)
- การใช้โทเค็น (อินพุต/เอาต์พุต/รวม)
- ค่าใช้จ่ายโดยประมาณเมื่อกำหนดราคาของโมเดลไว้ (`models.providers.*.models[].cost`)
- `sessionKey`, `sessionId` และพาธทรานสคริปต์ เพื่อให้เอเจนต์หลักสามารถดึงประวัติผ่าน `sessions_history` หรือตรวจสอบไฟล์บนดิสก์ได้

ข้อมูลเมตาภายในมีไว้สำหรับการควบคุมประสานงานเท่านั้น คำตอบที่แสดงต่อผู้ใช้
ควรถูกเขียนใหม่ด้วยน้ำเสียงปกติของผู้ช่วย

### เหตุผลที่ควรเลือกใช้ `sessions_history`

`sessions_history` เป็นเส้นทางการควบคุมประสานงานที่ปลอดภัยกว่าสำหรับการอ่าน
ทรานสคริปต์ของเอเจนต์ลูกจากภายในรอบการทำงานของเอเจนต์:

- ปกปิดข้อความที่มีลักษณะคล้ายข้อมูลรับรอง/โทเค็น แม้ว่าการปกปิดข้อมูลในบันทึกทั่วไปจะถูกปิดใช้งาน
- ตัดบล็อกข้อความยาวให้สั้นลง (4000 อักขระต่อบล็อก) และทิ้งลายเซ็นกระบวนการคิด เพย์โหลดสำหรับเล่นซ้ำการให้เหตุผล และข้อมูลรูปภาพแบบอินไลน์
- บังคับใช้ขีดจำกัดการตอบกลับ 80 KB โดยแถวที่มีขนาดใหญ่เกินไปจะถูกแทนที่ด้วย `[sessions_history omitted: message too large]`
- ใช้ `nextOffset` เมื่อมี เพื่อแบ่งหน้าย้อนกลับไปยังช่วงทรานสคริปต์ที่เก่ากว่า
- `sessions_history` **ไม่** ลบแท็กการให้เหตุผล โครงร่าง `<relevant-memories>` หรือ XML การเรียกเครื่องมือออกจากข้อความ แต่จะส่งคืนบล็อกเนื้อหาที่มีโครงสร้างใกล้เคียงรูปแบบทรานสคริปต์ดิบ เพียงปกปิดข้อมูลและจำกัดขนาดเท่านั้น ส่วน `/subagents log` ใช้ตัวทำความสะอาดข้อความร้อยแก้วที่เข้มงวดกว่า (ลบแท็กการให้เหตุผล โครงร่างหน่วยความจำ และ XML การเรียกเครื่องมือ) เพราะแสดงผลเป็นบรรทัดแชตข้อความธรรมดาแทนบล็อกที่มีโครงสร้าง
- การตรวจสอบทรานสคริปต์ดิบบนดิสก์เป็นทางเลือกสำรองเมื่อต้องการทรานสคริปต์ฉบับเต็มที่ตรงกันทุกไบต์

## นโยบายเครื่องมือ

เอเจนต์ย่อยใช้โปรไฟล์และไปป์ไลน์นโยบายเครื่องมือเดียวกับเอเจนต์แม่หรือ
เอเจนต์เป้าหมายก่อน จากนั้น OpenClaw จึงใช้ชั้นข้อจำกัดสำหรับเอเจนต์ย่อย

เอเจนต์ย่อยจะสูญเสีย `gateway`, `agents_list`, `session_status` และ
`cron` เสมอโดยไม่คำนึงถึงระดับความลึกหรือบทบาท (เครื่องมือระดับระบบ/แบบโต้ตอบ หรือ
เครื่องมือที่เอเจนต์หลักควรเป็นผู้ควบคุมประสานงาน) เอเจนต์ย่อยปลายทาง (พฤติกรรมเริ่มต้นที่ระดับความลึก 1
และที่ระดับความลึก 2 เสมอ) จะสูญเสีย `subagents`,
`sessions_list`, `sessions_history` และ `sessions_spawn` เพิ่มเติม เอเจนต์ย่อยจะไม่มีวัน
ได้รับเครื่องมือ `message` ซึ่งถูกปิดใช้งานในขณะสร้าง ไม่ได้ถูกกรองด้วย
รายการปฏิเสธนี้ และ `sessions_send` ยังคงถูกปฏิเสธเพื่อให้เอเจนต์ย่อย
สื่อสารผ่านลำดับการประกาศเท่านั้น

`sessions_history` ยังคงเป็นมุมมองเรียกคืนข้อมูลที่จำกัดขอบเขตและผ่านการทำความสะอาดแล้วในที่นี้เช่นกัน
ไม่ใช่การดัมป์ทรานสคริปต์ดิบ

เมื่อ `maxSpawnDepth >= 2` เอเจนต์ย่อยออร์เคสเตรเตอร์ระดับความลึก 1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ
`sessions_history` เพิ่มเติม เพื่อให้สามารถจัดการเอเจนต์ลูกได้

### การแทนที่ผ่านการกำหนดค่า

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny มีผลเหนือกว่า
        deny: ["gateway", "cron"],
        // หากตั้งค่า allow จะกลายเป็นรายการอนุญาตเท่านั้น (deny ยังคงมีผลเหนือกว่า)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` เป็นตัวกรองรายการอนุญาตเท่านั้นขั้นสุดท้าย ซึ่งสามารถจำกัด
ชุดเครื่องมือที่ผ่านการกำหนดแล้วให้แคบลง แต่ไม่สามารถ **เพิ่มกลับ**
เครื่องมือที่ `tools.profile` ลบออกไป ตัวอย่างเช่น `tools.profile: "coding"` มี
`web_search`/`web_fetch` แต่ไม่มีเครื่องมือ `browser` หากต้องการให้
เอเจนต์ย่อยที่ใช้โปรไฟล์การเขียนโค้ดใช้ระบบอัตโนมัติของเบราว์เซอร์ได้ ให้เพิ่มเบราว์เซอร์ใน
ขั้นตอนโปรไฟล์:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

ใช้ `agents.list[].tools.alsoAllow: ["browser"]` แบบต่อเอเจนต์ เมื่อมีเพียงเอเจนต์เดียวที่ควรได้รับความสามารถในการทำงานอัตโนมัติผ่านเบราว์เซอร์

## การทำงานพร้อมกัน

เอเจนต์ย่อยใช้เลนคิวเฉพาะภายในโพรเซส:

- **ชื่อเลน:** `subagent`
- **จำนวนงานพร้อมกัน:** `agents.defaults.subagents.maxConcurrent` (ค่าเริ่มต้น `8`)

## ความพร้อมทำงานและการกู้คืน

OpenClaw ไม่ถือว่าการไม่มี `endedAt` เป็นหลักฐานถาวรว่าเอเจนต์ย่อยยังคงทำงานอยู่ การทำงานที่ยังไม่สิ้นสุดและเก่ากว่าช่วงเวลาที่ถือว่าการทำงานค้าง (2 ชั่วโมง หรือระยะหมดเวลาการทำงานที่กำหนดไว้บวกช่วงผ่อนผันสั้นๆ แล้วแต่ว่าค่าใดนานกว่า) จะไม่นับเป็นกำลังทำงาน/รอดำเนินการใน `/subagents list` สรุปสถานะ การกั้นการเสร็จสิ้นของรายการสืบทอด และการตรวจสอบจำนวนงานพร้อมกันต่อเซสชันอีกต่อไป

หลังจาก Gateway เริ่มทำงานใหม่ การทำงานที่กู้คืนมาแต่ยังไม่สิ้นสุดและค้างอยู่จะถูกลบออก เว้นแต่เซสชันลูกของการทำงานนั้นจะถูกทำเครื่องหมายเป็น `abortedLastRun: true` การทำงานที่ถูกยุติเพราะการเริ่มทำงานใหม่จะยังคงลงทะเบียนไว้สำหรับขั้นตอนการกู้คืนเอเจนต์ย่อยที่ขาดการเชื่อมโยง โดยการทำงานที่ค้างจะถูกทำให้สิ้นสุดโดยไม่กลับมาทำต่อ ส่วนเซสชันลูกที่ยังใหม่จะได้รับข้อความสังเคราะห์ให้กลับมาทำต่อก่อนล้างเครื่องหมายว่าถูกยุติ

การกู้คืนอัตโนมัติหลังเริ่มทำงานใหม่มีขีดจำกัดต่อเซสชันลูก หากเอเจนต์ย่อยลูกเดิมได้รับการยอมรับให้กู้คืนจากการขาดการเชื่อมโยงซ้ำๆ ภายในช่วงเวลาที่เกิดการค้างซ้ำอย่างรวดเร็ว OpenClaw จะบันทึก recovery tombstone ไว้ในเซสชันนั้นและหยุดกลับมาทำต่อโดยอัตโนมัติในการเริ่มทำงานใหม่ครั้งถัดไป เรียกใช้
`openclaw tasks maintenance --apply` เพื่อกระทบยอดระเบียนงาน หรือ
`openclaw doctor --fix` เพื่อล้างแฟล็กการกู้คืนที่ถูกยุติและค้างอยู่ในเซสชันที่มี tombstone

<Note>
หากการสร้างเอเจนต์ย่อยล้มเหลวโดยมี Gateway `PAIRING_REQUIRED` /
`scope-upgrade` ให้ตรวจสอบผู้เรียก RPC ก่อนแก้ไขสถานะการจับคู่ การส่งต่อเพื่อประสานงานภายใน `sessions_spawn` จะเกิดขึ้นภายในโพรเซสเมื่อผู้เรียกกำลังทำงานอยู่ในบริบทคำขอของ Gateway อยู่แล้ว จึงไม่เปิด WebSocket แบบลูปแบ็กหรือขึ้นอยู่กับขอบเขตพื้นฐานของอุปกรณ์ที่จับคู่ไว้ของ CLI ผู้เรียกที่อยู่นอกโพรเซส Gateway ยังคงใช้ WebSocket สำรองในฐานะ `client.id: "gateway-client"` พร้อม `client.mode: "backend"`
ผ่านการยืนยันตัวตนด้วยโทเค็นที่ใช้ร่วมกัน/รหัสผ่านผ่านลูปแบ็กโดยตรง ผู้เรียกจากระยะไกล, `deviceIdentity` ที่ระบุอย่างชัดเจน, เส้นทางโทเค็นอุปกรณ์ที่ระบุอย่างชัดเจน และไคลเอนต์เบราว์เซอร์/Node ยังคงต้องได้รับการอนุมัติอุปกรณ์ตามปกติเพื่ออัปเกรดขอบเขต
</Note>

## การหยุด

- การส่ง `/stop` ในแชตของผู้ร้องขอจะยุติเซสชันของผู้ร้องขอและหยุดการทำงานของเอเจนต์ย่อยที่กำลังทำงานทั้งหมดซึ่งสร้างจากเซสชันนั้น โดยส่งผลต่อเนื่องไปยังลูกที่ซ้อนกัน

## ข้อจำกัด

- การประกาศของเอเจนต์ย่อยเป็นแบบ **พยายามอย่างดีที่สุด** หาก Gateway เริ่มทำงานใหม่ งาน "ประกาศกลับ" ที่รอดำเนินการจะสูญหาย
- เอเจนต์ย่อยยังคงใช้ทรัพยากรของโพรเซส Gateway เดียวกันร่วมกัน ให้ถือว่า `maxConcurrent` เป็นกลไกนิรภัย
- `sessions_spawn` จะไม่บล็อกเสมอ โดยส่งคืน `{ status: "accepted", runId, childSessionKey }` ทันที
- บริบทของเอเจนต์ย่อยจะแทรกเฉพาะ `AGENTS.md` และ `TOOLS.md` (ไม่มี `SOUL.md`, `IDENTITY.md`, `USER.md`, `MEMORY.md`, `HEARTBEAT.md` หรือ `BOOTSTRAP.md`) เอเจนต์ย่อยแบบเนทีฟของ Codex ใช้ขอบเขตเดียวกัน โดย `TOOLS.md` จะยังคงอยู่ในคำสั่งของเธรด Codex ที่สืบทอดมา ส่วนไฟล์บุคลิก ตัวตน และผู้ใช้สำหรับแม่เท่านั้นจะถูกแทรกเป็นคำสั่งการทำงานร่วมกันที่มีขอบเขตเฉพาะเทิร์น เพื่อไม่ให้ลูกโคลนไฟล์เหล่านั้น
- ความลึกสูงสุดของการซ้อนคือ 5 (ช่วงของ `maxSpawnDepth`: 1-5) แนะนำให้ใช้ความลึก 2 สำหรับกรณีการใช้งานส่วนใหญ่
- `maxChildrenPerAgent` จำกัดจำนวนลูกที่กำลังทำงานต่อเซสชัน (ค่าเริ่มต้น `5`, ช่วง `1-20`)
