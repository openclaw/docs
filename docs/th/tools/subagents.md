เอเจนต์ย่อยคือการรันเอเจนต์เบื้องหลังที่ถูกสร้างจากการรันเอเจนต์ที่มีอยู่
เอเจนต์เหล่านี้รันในเซสชันของตัวเอง (`agent:<agentId>:subagent:<uuid>`) และ
เมื่อเสร็จสิ้น จะ**ประกาศ**ผลลัพธ์กลับไปยังช่องแชตของผู้ร้องขอ
การรันเอเจนต์ย่อยแต่ละครั้งจะถูกติดตามเป็น
[งานเบื้องหลัง](/th/automation/tasks)

เป้าหมายหลัก:

- ทำให้งาน "ค้นคว้า / งานยาว / เครื่องมือที่ช้า" ทำงานแบบขนานได้โดยไม่บล็อกการรันหลัก
- แยกเอเจนต์ย่อยออกจากกันเป็นค่าเริ่มต้น (การแยกเซสชัน + การทำ sandbox แบบเลือกได้)
- ทำให้พื้นผิวเครื่องมือใช้งานผิดได้ยาก: เอเจนต์ย่อยจะ**ไม่ได้**เครื่องมือของเซสชันเป็นค่าเริ่มต้น
- รองรับความลึกของการซ้อนที่กำหนดค่าได้สำหรับรูปแบบ orchestrator

<Note>
**หมายเหตุค่าใช้จ่าย:** เอเจนต์ย่อยแต่ละตัวมีบริบทและการใช้โทเค็นของตัวเองตามค่าเริ่มต้น
สำหรับงานหนักหรืองานซ้ำ ๆ ให้ตั้งโมเดลที่ถูกกว่าสำหรับเอเจนต์ย่อย
และคงเอเจนต์หลักไว้บนโมเดลคุณภาพสูงกว่า กำหนดค่าผ่าน
`agents.defaults.subagents.model` หรือการเขียนทับรายเอเจนต์ เมื่อ child
จำเป็นต้องใช้ทรานสคริปต์ปัจจุบันของผู้ร้องขอจริง ๆ เอเจนต์สามารถขอ
`context: "fork"` ในการสร้างครั้งนั้นครั้งเดียวได้
</Note>

## คำสั่งสแลช

ใช้ `/subagents` เพื่อตรวจสอบหรือควบคุมการรันเอเจนต์ย่อยสำหรับ**เซสชันปัจจุบัน**:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

`/subagents info` แสดงเมทาดาทาของการรัน (สถานะ, ประทับเวลา, id เซสชัน,
พาธทรานสคริปต์, การล้างข้อมูล) ใช้ `sessions_history` สำหรับมุมมองการเรียกคืนแบบจำกัดขอบเขต
และกรองความปลอดภัยแล้ว ตรวจสอบพาธทรานสคริปต์บนดิสก์เมื่อคุณ
ต้องการทรานสคริปต์ฉบับเต็มแบบดิบ

### การควบคุมการผูกเธรด

คำสั่งเหล่านี้ทำงานบนช่องที่รองรับการผูกเธรดแบบคงอยู่
ดู [ช่องที่รองรับเธรด](#thread-supporting-channels) ด้านล่าง

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### พฤติกรรมการสร้าง

`/subagents spawn` เริ่มเอเจนต์ย่อยเบื้องหลังเป็นคำสั่งผู้ใช้ (ไม่ใช่
รีเลย์ภายใน) และส่งการอัปเดตการเสร็จสิ้นสุดท้ายหนึ่งครั้งกลับไปยัง
แชตของผู้ร้องขอเมื่อการรันเสร็จสิ้น

<AccordionGroup>
  <Accordion title="Non-blocking, push-based completion">
    - คำสั่งสร้างไม่บล็อก โดยจะคืน id การรันทันที
    - เมื่อเสร็จสิ้น เอเจนต์ย่อยจะประกาศข้อความสรุป/ผลลัพธ์กลับไปยังช่องแชตของผู้ร้องขอ
    - การเสร็จสิ้นเป็นแบบพุช เมื่อสร้างแล้ว อย่าวนโพล `/subagents list`, `sessions_list`, หรือ `sessions_history` เพียงเพื่อรอให้เสร็จสิ้น ให้ตรวจสอบสถานะเฉพาะเมื่อจำเป็นสำหรับการดีบักหรือการแทรกแซงเท่านั้น
    - เมื่อเสร็จสิ้น OpenClaw จะพยายามปิดแท็บเบราว์เซอร์/โปรเซสที่ติดตามไว้ซึ่งเปิดโดยเซสชันเอเจนต์ย่อยนั้น ก่อนที่โฟลว์ล้างข้อมูลของการประกาศจะดำเนินต่อ

  </Accordion>
  <Accordion title="Manual-spawn delivery resilience">
    - OpenClaw ลองส่งแบบตรงด้วย `agent` ก่อนโดยใช้คีย์ idempotency ที่เสถียร
    - หากการส่งตรงล้มเหลว จะถอยกลับไปใช้การกำหนดเส้นทางผ่านคิว
    - หากการกำหนดเส้นทางผ่านคิวยังไม่พร้อมใช้งาน การประกาศจะถูกลองใหม่ด้วย exponential backoff สั้น ๆ ก่อนยอมแพ้ในขั้นสุดท้าย
    - การส่งมอบการเสร็จสิ้นจะคงเส้นทางผู้ร้องขอที่ resolve แล้วไว้: เส้นทางการเสร็จสิ้นที่ผูกกับเธรดหรือผูกกับการสนทนาจะชนะเมื่อพร้อมใช้งาน หากจุดกำเนิดการเสร็จสิ้นให้เฉพาะช่อง OpenClaw จะเติม target/account ที่ขาดหายไปจากเส้นทางที่ resolve แล้วของเซสชันผู้ร้องขอ (`lastChannel` / `lastTo` / `lastAccountId`) เพื่อให้การส่งตรงยังทำงานได้

  </Accordion>
  <Accordion title="Completion handoff metadata">
    การส่งมอบการเสร็จสิ้นไปยังเซสชันผู้ร้องขอเป็นบริบทภายในที่สร้างโดยรันไทม์
    (ไม่ใช่ข้อความที่ผู้ใช้เขียน) และประกอบด้วย:

    - `Result` — ข้อความตอบกลับ `assistant` ล่าสุดที่มองเห็นได้ มิฉะนั้นเป็นข้อความ tool/toolResult ล่าสุดที่ผ่านการ sanitize แล้ว การรันที่ล้มเหลวแบบ terminal จะไม่นำข้อความตอบกลับที่จับไว้มาใช้ซ้ำ
    - `Status` — `completed successfully` / `failed` / `timed out` / `unknown`
    - สถิติรันไทม์/โทเค็นแบบกระชับ
    - คำสั่งการส่งที่บอกเอเจนต์ผู้ร้องขอให้เขียนใหม่ด้วยเสียงผู้ช่วยตามปกติ (ไม่ส่งต่อเมทาดาทาภายในแบบดิบ)

  </Accordion>
  <Accordion title="Modes and ACP runtime">
    - `--model` และ `--thinking` เขียนทับค่าเริ่มต้นสำหรับการรันเฉพาะครั้งนั้น
    - ใช้ `info`/`log` เพื่อตรวจสอบรายละเอียดและเอาต์พุตหลังเสร็จสิ้น
    - `/subagents spawn` เป็นโหมดครั้งเดียว (`mode: "run"`) สำหรับเซสชันแบบคงอยู่ที่ผูกกับเธรด ให้ใช้ `sessions_spawn` พร้อม `thread: true` และ `mode: "session"`
    - สำหรับเซสชัน ACP harness (Claude Code, Gemini CLI, OpenCode, หรือ Codex ACP/acpx ที่ระบุชัดเจน) ให้ใช้ `sessions_spawn` พร้อม `runtime: "acp"` เมื่อเครื่องมือประกาศรันไทม์นั้น ดู [โมเดลการส่งของ ACP](/th/tools/acp-agents#delivery-model) เมื่อดีบักการเสร็จสิ้นหรือลูปเอเจนต์ต่อเอเจนต์ เมื่อเปิดใช้ Plugin `codex` การควบคุมแชต/เธรดของ Codex ควรใช้ `/codex ...` แทน ACP เว้นแต่ผู้ใช้จะขอ ACP/acpx อย่างชัดเจน
    - OpenClaw ซ่อน `runtime: "acp"` จนกว่า ACP จะเปิดใช้ ผู้ร้องขอไม่ได้อยู่ใน sandbox และโหลด Plugin backend เช่น `acpx` แล้ว `runtime: "acp"` คาดหวัง id ของ ACP harness ภายนอก หรือรายการ `agents.list[]` ที่มี `runtime.type="acp"`; ใช้รันไทม์เอเจนต์ย่อยค่าเริ่มต้นสำหรับเอเจนต์ config ของ OpenClaw ปกติจาก `agents_list`

  </Accordion>
</AccordionGroup>

## โหมดบริบท

เอเจนต์ย่อยแบบ native จะเริ่มแบบแยกออกจากกัน เว้นแต่ผู้เรียกจะขอ fork
ทรานสคริปต์ปัจจุบันอย่างชัดเจน

| โหมด       | เมื่อใดควรใช้                                                                                                                         | พฤติกรรม                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | การค้นคว้าใหม่, การใช้งานแบบอิสระ, งานเครื่องมือที่ช้า, หรืออะไรก็ตามที่สรุปโจทย์ได้ในข้อความงาน                           | สร้างทรานสคริปต์ child ที่สะอาด นี่คือค่าเริ่มต้นและช่วยลดการใช้โทเค็น  |
| `fork`     | งานที่ขึ้นกับการสนทนาปัจจุบัน, ผลลัพธ์เครื่องมือก่อนหน้า, หรือคำสั่งที่มีความละเอียดอ่อนซึ่งมีอยู่แล้วในทรานสคริปต์ของผู้ร้องขอ | แตกแขนงทรานสคริปต์ของผู้ร้องขอเข้าไปในเซสชัน child ก่อนที่ child จะเริ่ม |

ใช้ `fork` อย่างประหยัด มีไว้สำหรับการมอบหมายที่ไวต่อบริบท ไม่ใช่
สิ่งทดแทนการเขียน prompt งานที่ชัดเจน

## เครื่องมือ: `sessions_spawn`

เริ่มการรันเอเจนต์ย่อยด้วย `deliver: false` บนเลน `subagent` ส่วนกลาง
จากนั้นรันขั้นตอนประกาศและโพสต์การตอบกลับประกาศไปยังช่องแชตของผู้ร้องขอ

ความพร้อมใช้งานขึ้นกับนโยบายเครื่องมือที่มีผลของผู้เรียก โปรไฟล์ `coding` และ
`full` เปิดเผย `sessions_spawn` เป็นค่าเริ่มต้น โปรไฟล์ `messaging`
ไม่เปิดเผย ให้เพิ่ม `tools.alsoAllow: ["sessions_spawn", "sessions_yield",
"subagents"]` หรือใช้ `tools.profile: "coding"` สำหรับเอเจนต์ที่ควรมอบหมายงาน
นโยบาย allow/deny ของช่อง/กลุ่ม, provider, sandbox และรายเอเจนต์
ยังสามารถนำเครื่องมือออกหลังจากขั้นโปรไฟล์ได้ ใช้ `/tools` จาก
เซสชันเดียวกันเพื่อยืนยันรายการเครื่องมือที่มีผล

**ค่าเริ่มต้น:**

- **โมเดล:** สืบทอดจากผู้เรียก เว้นแต่คุณจะตั้ง `agents.defaults.subagents.model` (หรือ `agents.list[].subagents.model` รายเอเจนต์); `sessions_spawn.model` ที่ระบุชัดเจนยังชนะเสมอ
- **Thinking:** สืบทอดจากผู้เรียก เว้นแต่คุณจะตั้ง `agents.defaults.subagents.thinking` (หรือ `agents.list[].subagents.thinking` รายเอเจนต์); `sessions_spawn.thinking` ที่ระบุชัดเจนยังชนะเสมอ
- **ไทม์เอาต์การรัน:** หากละ `sessions_spawn.runTimeoutSeconds` ไว้ OpenClaw จะใช้ `agents.defaults.subagents.runTimeoutSeconds` เมื่อมีการตั้งค่า มิฉะนั้นจะถอยกลับไปเป็น `0` (ไม่มีไทม์เอาต์)

</ParamField>

<Warning>
`sessions_spawn` **ไม่**รับพารามิเตอร์การส่งผ่านช่อง (`target`,
`channel`, `to`, `threadId`, `replyTo`, `transport`) สำหรับการส่ง ให้ใช้
`message`/`sessions_send` จากการรันที่ถูกสร้าง
</Warning>

## เซสชันที่ผูกกับเธรด

เมื่อเปิดใช้การผูกเธรดสำหรับช่อง เอเจนต์ย่อยสามารถคงการผูก
กับเธรดไว้ได้ เพื่อให้ข้อความติดตามผลจากผู้ใช้ในเธรดนั้นยังคงถูกกำหนดเส้นทางไปยัง
เซสชันเอเจนต์ย่อยเดิม

### ช่องที่รองรับเธรด

**Discord** เป็นช่องเดียวที่รองรับในขณะนี้ โดยรองรับ
เซสชันเอเจนต์ย่อยที่ผูกกับเธรดแบบคงอยู่ (`sessions_spawn` พร้อม
`thread: true`), การควบคุมเธรดด้วยตนเอง (`/focus`, `/unfocus`, `/agents`,
`/session idle`, `/session max-age`) และคีย์ adapter
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours`, และ
`channels.discord.threadBindings.spawnSubagentSessions`

### โฟลว์แบบเร็ว

<Steps>
  <Step title="Spawn">
    `sessions_spawn` พร้อม `thread: true` (และเลือกใส่ `mode: "session"` ได้)
  </Step>
  <Step title="Bind">
    OpenClaw สร้างหรือผูกเธรดกับเป้าหมายเซสชันนั้นในช่องที่ใช้งานอยู่
  </Step>
  <Step title="Route follow-ups">
    การตอบกลับและข้อความติดตามผลในเธรดนั้นถูกกำหนดเส้นทางไปยังเซสชันที่ผูกไว้
  </Step>
  <Step title="Inspect timeouts">
    ใช้ `/session idle` เพื่อตรวจสอบ/อัปเดตการเลิกโฟกัสอัตโนมัติเมื่อไม่มีความเคลื่อนไหว และ
    `/session max-age` เพื่อควบคุมเพดานสูงสุดแบบแข็ง
  </Step>
  <Step title="Detach">
    ใช้ `/unfocus` เพื่อถอดออกด้วยตนเอง
  </Step>
</Steps>

### การควบคุมด้วยตนเอง

| คำสั่ง            | ผลลัพธ์                                                                |
| ------------------ | --------------------------------------------------------------------- |
| `/focus <target>`  | ผูกเธรดปัจจุบัน (หรือสร้างเธรดใหม่) กับเป้าหมายเอเจนต์ย่อย/เซสชัน |
| `/unfocus`         | ลบการผูกสำหรับเธรดปัจจุบันที่ถูกผูกไว้                       |
| `/agents`          | แสดงรายการการรันที่ใช้งานอยู่และสถานะการผูก (`thread:<id>` หรือ `unbound`)       |
| `/session idle`    | ตรวจสอบ/อัปเดตการยกเลิกโฟกัสอัตโนมัติเมื่อไม่ได้ใช้งาน (เฉพาะเธรดที่ถูกผูกและอยู่ในโฟกัส)         |
| `/session max-age` | ตรวจสอบ/อัปเดตขีดจำกัดสูงสุดแบบตายตัว (เฉพาะเธรดที่ถูกผูกและอยู่ในโฟกัส)                  |

### สวิตช์การกำหนดค่า

- **ค่าเริ่มต้นส่วนกลาง:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **คีย์สำหรับการเขียนทับตามช่องทางและการผูกอัตโนมัติเมื่อสร้าง** ขึ้นกับอะแดปเตอร์ ดู [ช่องทางที่รองรับเธรด](#thread-supporting-channels) ด้านบน

ดู [ข้อมูลอ้างอิงการกำหนดค่า](/th/gateway/configuration-reference) และ
[คำสั่งสแลช](/th/tools/slash-commands) สำหรับรายละเอียดอะแดปเตอร์ปัจจุบัน

### รายการอนุญาต

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  รายการรหัสเอเจนต์ที่สามารถกำหนดเป็นเป้าหมายผ่าน `agentId` อย่างชัดเจน (`["*"]` อนุญาตทุกตัว) ค่าเริ่มต้น: เฉพาะเอเจนต์ผู้ร้องขอเท่านั้น หากคุณตั้งค่ารายการและยังต้องการให้ผู้ร้องขอสร้างตัวเองด้วย `agentId` ให้รวมรหัสผู้ร้องขอไว้ในรายการด้วย
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  รายการอนุญาตเอเจนต์เป้าหมายเริ่มต้นที่ใช้เมื่อเอเจนต์ผู้ร้องขอไม่ได้ตั้งค่า `subagents.allowAgents` ของตัวเอง
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  บล็อกการเรียก `sessions_spawn` ที่ละ `agentId` (บังคับให้เลือกโปรไฟล์อย่างชัดเจน) การเขียนทับรายเอเจนต์: `agents.list[].subagents.requireAgentId`
</ParamField>

หากเซสชันผู้ร้องขออยู่ในแซนด์บ็อกซ์ `sessions_spawn` จะปฏิเสธเป้าหมาย
ที่จะรันแบบไม่อยู่ในแซนด์บ็อกซ์

### การค้นพบ

ใช้ `agents_list` เพื่อดูว่ารหัสเอเจนต์ใดได้รับอนุญาตสำหรับ
`sessions_spawn` ในปัจจุบัน การตอบกลับมีโมเดลที่มีผลจริงของแต่ละเอเจนต์ที่แสดงรายการ
และเมตาดาตา runtime ที่ฝังไว้ เพื่อให้ผู้เรียกแยกแยะ PI, Codex
app-server และ runtime ดั้งเดิมอื่น ๆ ที่กำหนดค่าไว้ได้

### การเก็บถาวรอัตโนมัติ

- เซสชันเอเจนต์ย่อยจะถูกเก็บถาวรโดยอัตโนมัติหลังจาก `agents.defaults.subagents.archiveAfterMinutes` (ค่าเริ่มต้น `60`)
- การเก็บถาวรใช้ `sessions.delete` และเปลี่ยนชื่อทรานสคริปต์เป็น `*.deleted.<timestamp>` (โฟลเดอร์เดียวกัน)
- `cleanup: "delete"` เก็บถาวรทันทีหลังจากประกาศ (ยังคงเก็บทรานสคริปต์ไว้ผ่านการเปลี่ยนชื่อ)
- การเก็บถาวรอัตโนมัติเป็นแบบพยายามอย่างดีที่สุด; ตัวจับเวลาที่ค้างอยู่จะหายไปหาก Gateway เริ่มใหม่
- `runTimeoutSeconds` **ไม่** เก็บถาวรอัตโนมัติ; มันหยุดเฉพาะการรันเท่านั้น เซสชันยังคงอยู่จนกว่าจะถึงการเก็บถาวรอัตโนมัติ
- การเก็บถาวรอัตโนมัติใช้เท่า ๆ กันกับเซสชันความลึก 1 และความลึก 2
- การล้างเบราว์เซอร์แยกจากการล้างการเก็บถาวร: แท็บ/โปรเซสเบราว์เซอร์ที่ติดตามจะถูกปิดแบบพยายามอย่างดีที่สุดเมื่อการรันเสร็จสิ้น แม้ว่าทรานสคริปต์/ระเบียนเซสชันจะถูกเก็บไว้ก็ตาม

## เอเจนต์ย่อยแบบซ้อนกัน

ตามค่าเริ่มต้น เอเจนต์ย่อยไม่สามารถสร้างเอเจนต์ย่อยของตัวเองได้
(`maxSpawnDepth: 1`) ตั้งค่า `maxSpawnDepth: 2` เพื่อเปิดใช้การซ้อนหนึ่งระดับ
ซึ่งคือ **รูปแบบตัวประสานงาน**: ตัวหลัก → เอเจนต์ย่อยตัวประสานงาน →
เอเจนต์ย่อยย่อยผู้ปฏิบัติงาน

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // allow sub-agents to spawn children (default: 1)
        maxChildrenPerAgent: 5, // max active children per agent session (default: 5)
        maxConcurrent: 8, // global concurrency lane cap (default: 8)
        runTimeoutSeconds: 900, // default timeout for sessions_spawn when omitted (0 = no timeout)
      },
    },
  },
}
```

### ระดับความลึก

| ความลึก | รูปแบบคีย์เซสชัน                            | บทบาท                                          | สร้างได้หรือไม่                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | เอเจนต์หลัก                                    | เสมอ                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | เอเจนต์ย่อย (ตัวประสานงานเมื่ออนุญาตความลึก 2) | เฉพาะเมื่อ `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | เอเจนต์ย่อยย่อย (ผู้ปฏิบัติงานปลายทาง)                   | ไม่เคย                        |

### สายการประกาศ

ผลลัพธ์ไหลย้อนกลับขึ้นไปตามสาย:

1. ผู้ปฏิบัติงานความลึก 2 เสร็จสิ้น → ประกาศไปยังพาเรนต์ของตน (ตัวประสานงานความลึก 1)
2. ตัวประสานงานความลึก 1 รับประกาศ สังเคราะห์ผลลัพธ์ เสร็จสิ้น → ประกาศไปยังตัวหลัก
3. เอเจนต์หลักรับประกาศและส่งต่อให้ผู้ใช้

แต่ละระดับเห็นเฉพาะประกาศจากลูกโดยตรงของตนเท่านั้น

<Note>
**คำแนะนำด้านการปฏิบัติการ:** เริ่มงานลูกหนึ่งครั้งและรอเหตุการณ์เสร็จสิ้น
แทนการสร้างลูปโพลรอบ `sessions_list`,
`sessions_history`, `/subagents list` หรือคำสั่ง `exec` sleep
`sessions_list` และ `/subagents list` รักษาความสัมพันธ์ของเซสชันลูก
ให้เน้นงานที่ยังทำงานอยู่ — ลูกรันสดยังคงแนบอยู่ ลูกที่จบแล้วจะยัง
มองเห็นได้ในหน้าต่างล่าสุดช่วงสั้น ๆ และลิงก์ลูกที่เก่าและมีเฉพาะในที่เก็บ
จะถูกละเว้นหลังจากหน้าต่างความสดของมัน วิธีนี้ป้องกันไม่ให้เมตาดาตา `spawnedBy` /
`parentSessionKey` เก่าทำให้ลูกเงากลับมาอีกหลังจาก
เริ่มใหม่ หากเหตุการณ์เสร็จสิ้นของลูกมาถึงหลังจากคุณส่ง
คำตอบสุดท้ายไปแล้ว การติดตามผลที่ถูกต้องคือโทเค็นเงียบที่ตรงทุกตัวอักษร
`NO_REPLY` / `no_reply`
</Note>

### นโยบายเครื่องมือตามความลึก

- บทบาทและขอบเขตการควบคุมถูกเขียนลงในเมตาดาตาเซสชันตอนสร้าง วิธีนี้ทำให้คีย์เซสชันแบบแบนหรือที่กู้คืนมาไม่สามารถได้สิทธิ์ตัวประสานงานคืนโดยไม่ตั้งใจ
- **ความลึก 1 (ตัวประสานงาน เมื่อ `maxSpawnDepth >= 2`):** ได้รับ `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` เพื่อให้จัดการลูกของตนได้ เครื่องมือเซสชัน/ระบบอื่น ๆ ยังคงถูกปฏิเสธ
- **ความลึก 1 (ปลายทาง เมื่อ `maxSpawnDepth == 1`):** ไม่มีเครื่องมือเซสชัน (พฤติกรรมเริ่มต้นปัจจุบัน)
- **ความลึก 2 (ผู้ปฏิบัติงานปลายทาง):** ไม่มีเครื่องมือเซสชัน — `sessions_spawn` ถูกปฏิเสธเสมอที่ความลึก 2 ไม่สามารถสร้างลูกเพิ่มได้

### ขีดจำกัดการสร้างต่อเอเจนต์

แต่ละเซสชันเอเจนต์ (ที่ความลึกใดก็ได้) มีลูกที่ใช้งานอยู่ได้สูงสุด `maxChildrenPerAgent`
(ค่าเริ่มต้น `5`) ในเวลาเดียวกัน วิธีนี้ป้องกันการแตกแขนงแบบควบคุมไม่ได้
จากตัวประสานงานตัวเดียว

### การหยุดแบบต่อเนื่อง

การหยุดตัวประสานงานความลึก 1 จะหยุดลูกความลึก 2
ทั้งหมดโดยอัตโนมัติ:

- `/stop` ในแชตหลักจะหยุดเอเจนต์ความลึก 1 ทั้งหมดและหยุดต่อเนื่องไปยังลูกความลึก 2 ของพวกมัน
- `/subagents kill <id>` หยุดเอเจนต์ย่อยที่ระบุและหยุดต่อเนื่องไปยังลูกของมัน
- `/subagents kill all` หยุดเอเจนต์ย่อยทั้งหมดสำหรับผู้ร้องขอและหยุดต่อเนื่อง

## การยืนยันตัวตน

การยืนยันตัวตนของเอเจนต์ย่อยถูกแก้ตาม **รหัสเอเจนต์** ไม่ใช่ตามชนิดเซสชัน:

- คีย์เซสชันเอเจนต์ย่อยคือ `agent:<agentId>:subagent:<uuid>`
- ที่เก็บการยืนยันตัวตนถูกโหลดจาก `agentDir` ของเอเจนต์นั้น
- โปรไฟล์การยืนยันตัวตนของเอเจนต์หลักถูกรวมเข้าเป็น **fallback**; โปรไฟล์เอเจนต์จะเขียนทับโปรไฟล์หลักเมื่อมีความขัดแย้ง

การรวมเป็นแบบเพิ่มเข้าไป ดังนั้นโปรไฟล์หลักจึงพร้อมใช้งานเป็น
fallback เสมอ ยังไม่รองรับการยืนยันตัวตนที่แยกอย่างสมบูรณ์ต่อเอเจนต์

## ประกาศ

เอเจนต์ย่อยรายงานกลับผ่านขั้นตอนประกาศ:

- ขั้นตอนประกาศรันภายในเซสชันเอเจนต์ย่อย (ไม่ใช่เซสชันผู้ร้องขอ)
- หากเอเจนต์ย่อยตอบกลับตรงว่า `ANNOUNCE_SKIP` จะไม่มีการโพสต์อะไร
- หากข้อความผู้ช่วยล่าสุดเป็นโทเค็นเงียบที่ตรงทุกตัวอักษร `NO_REPLY` / `no_reply` ผลลัพธ์ประกาศจะถูกระงับ แม้ว่าก่อนหน้านั้นจะมีความคืบหน้าที่มองเห็นได้

การส่งขึ้นกับความลึกของผู้ร้องขอ:

- เซสชันผู้ร้องขอระดับบนสุดใช้การเรียก `agent` เพื่อติดตามผลพร้อมการส่งภายนอก (`deliver=true`)
- เซสชันเอเจนต์ย่อยผู้ร้องขอแบบซ้อนรับการฉีดติดตามผลภายใน (`deliver=false`) เพื่อให้ตัวประสานงานสังเคราะห์ผลลัพธ์ลูกภายในเซสชันได้
- หากเซสชันเอเจนต์ย่อยผู้ร้องขอแบบซ้อนหายไป OpenClaw จะ fallback ไปยังผู้ร้องขอของเซสชันนั้นเมื่อมี

สำหรับเซสชันผู้ร้องขอระดับบนสุด การส่งตรงในโหมดเสร็จสิ้นจะ
แก้เส้นทางการสนทนา/เธรดที่ถูกผูกและการเขียนทับ hook ก่อน แล้วเติม
ฟิลด์เป้าหมายช่องทางที่ขาดจากเส้นทางที่เก็บไว้ของเซสชันผู้ร้องขอ
วิธีนี้ทำให้การเสร็จสิ้นไปยังแชต/หัวข้อที่ถูกต้อง แม้เมื่อจุดเริ่มต้นการเสร็จสิ้น
ระบุเฉพาะช่องทางเท่านั้น

การรวมการเสร็จสิ้นของลูกถูกจำกัดขอบเขตไว้กับการรันของผู้ร้องขอปัจจุบันเมื่อ
สร้างข้อค้นพบการเสร็จสิ้นแบบซ้อน เพื่อป้องกันไม่ให้ผลลัพธ์ลูกจากการรันก่อนหน้า
ที่เก่ารั่วเข้าสู่ประกาศปัจจุบัน การตอบกลับประกาศรักษา
การกำหนดเส้นทางเธรด/หัวข้อเมื่อมีบนอะแดปเตอร์ช่องทาง

### บริบทประกาศ

บริบทประกาศถูกทำให้เป็นมาตรฐานเป็นบล็อกเหตุการณ์ภายในที่เสถียร:

| ฟิลด์          | แหล่งที่มา                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| แหล่งที่มา         | `subagent` หรือ `cron`                                                                                          |
| รหัสเซสชัน    | คีย์/รหัสเซสชันลูก                                                                                          |
| ชนิด           | ชนิดประกาศ + ป้ายกำกับงาน                                                                                    |
| สถานะ         | ได้มาจากผลลัพธ์ runtime (`success`, `error`, `timeout` หรือ `unknown`) — **ไม่ใช่** อนุมานจากข้อความโมเดล |
| เนื้อหาผลลัพธ์ | ข้อความผู้ช่วยล่าสุดที่มองเห็นได้ มิฉะนั้นใช้ข้อความเครื่องมือ/toolResult ล่าสุดที่ผ่านการทำความสะอาด                                |
| การติดตามผล      | คำสั่งที่อธิบายว่าเมื่อใดควรตอบกลับเทียบกับอยู่นิ่ง                                                           |

การรันที่ล้มเหลวในขั้นสุดท้ายรายงานสถานะล้มเหลวโดยไม่เล่นซ้ำ
ข้อความตอบกลับที่จับไว้ เมื่อหมดเวลา หากลูกทำได้เพียงเรียกเครื่องมือ
ประกาศสามารถยุบประวัตินั้นเป็นสรุปความคืบหน้าบางส่วนสั้น ๆ แทน
การเล่นซ้ำผลลัพธ์เครื่องมือดิบ

### บรรทัดสถิติ

เพย์โหลดประกาศมีบรรทัดสถิติที่ท้ายข้อความ (แม้เมื่อถูกห่อ):

- Runtime (เช่น `runtime 5m12s`)
- การใช้งานโทเค็น (อินพุต/เอาต์พุต/รวม)
- ค่าใช้จ่ายโดยประมาณเมื่อกำหนดราคาของโมเดลไว้ (`models.providers.*.models[].cost`)
- `sessionKey`, `sessionId` และเส้นทางทรานสคริปต์ เพื่อให้เอเจนต์หลักดึงประวัติผ่าน `sessions_history` หรือตรวจสอบไฟล์บนดิสก์ได้

เมตาดาตาภายในมีไว้สำหรับการประสานงานเท่านั้น; คำตอบที่แสดงต่อผู้ใช้
ควรถูกเขียนใหม่ด้วยน้ำเสียงผู้ช่วยปกติ

### ทำไมจึงควรใช้ `sessions_history`

`sessions_history` เป็นเส้นทางการประสานงานที่ปลอดภัยกว่า:

- การเรียกคืนของผู้ช่วยถูกทำให้เป็นมาตรฐานก่อน: ลบแท็กการคิด; ลบโครง `<relevant-memories>` / `<relevant_memories>`; ลบบล็อกเพย์โหลด XML การเรียกเครื่องมือแบบข้อความล้วน (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`) รวมถึงเพย์โหลดที่ถูกตัดซึ่งไม่เคยปิดสมบูรณ์; ลบโครงสร้างการเรียกเครื่องมือ/ผลลัพธ์ที่ถูกลดระดับและเครื่องหมายบริบทเชิงประวัติ; ลบโทเค็นควบคุมโมเดลที่รั่ว (`<|assistant|>`, ASCII `<|...|>` อื่น ๆ, แบบเต็มความกว้าง `<｜...｜>`); ลบ XML การเรียกเครื่องมือ MiniMax ที่ผิดรูป
- ข้อความที่คล้ายข้อมูลรับรอง/โทเค็นถูกปกปิด
- บล็อกยาวสามารถถูกตัดให้สั้นลงได้
- ประวัติขนาดใหญ่มากสามารถทิ้งแถวเก่าหรือแทนที่แถวที่ใหญ่เกินด้วย `[sessions_history omitted: message too large]`
- การตรวจสอบทรานสคริปต์ดิบบนดิสก์เป็น fallback เมื่อคุณต้องการทรานสคริปต์แบบตรงไบต์ต่อไบต์เต็มรูปแบบ

## นโยบายเครื่องมือ

เอเจนต์ย่อยใช้โปรไฟล์และไปป์ไลน์นโยบายเครื่องมือเดียวกับพาเรนต์หรือ
เอเจนต์เป้าหมายก่อน หลังจากนั้น OpenClaw ใช้เลเยอร์ข้อจำกัด
เอเจนต์ย่อย

เมื่อไม่มี `tools.profile` ที่จำกัด เอเจนต์ย่อยจะได้ **เครื่องมือทั้งหมด ยกเว้น
เครื่องมือเซสชัน** และเครื่องมือระบบ:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

`sessions_history` ยังคงเป็นมุมมองการเรียกคืนที่มีขอบเขตและผ่านการทำความสะอาดที่นี่เช่นกัน —
ไม่ใช่การดัมพ์ทรานสคริปต์ดิบ

เมื่อ `maxSpawnDepth >= 2` เอเจนต์ย่อยตัวประสานงานความลึก 1 จะได้รับ
`sessions_spawn`, `subagents`, `sessions_list` และ
`sessions_history` เพิ่มเติม เพื่อให้จัดการลูกของตนได้

### เขียนทับผ่านการกำหนดค่า

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny wins
        deny: ["gateway", "cron"],
        // if allow is set, it becomes allow-only (deny still wins)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` เป็นตัวกรองขั้นสุดท้ายแบบอนุญาตเท่านั้น ตัวกรองนี้สามารถจำกัด
ชุดเครื่องมือที่แก้ไขแล้วให้แคบลงได้ แต่ไม่สามารถ **เพิ่มกลับ** เครื่องมือที่ถูกลบ
โดย `tools.profile` ได้ ตัวอย่างเช่น `tools.profile: "coding"` รวม
`web_search`/`web_fetch` แต่ไม่รวมเครื่องมือ `browser` หากต้องการให้
เอเจนต์ย่อยโปรไฟล์การเขียนโค้ดใช้การทำงานอัตโนมัติของเบราว์เซอร์ได้ ให้เพิ่ม browser ใน
ขั้นตอนโปรไฟล์:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

ใช้ `agents.list[].tools.alsoAllow: ["browser"]` แบบรายเอเจนต์ เมื่อมีเพียง
เอเจนต์เดียวที่ควรได้รับการทำงานอัตโนมัติของเบราว์เซอร์

## ภาวะพร้อมกัน

เอเจนต์ย่อยใช้เลนคิวเฉพาะภายในโปรเซส:

- **ชื่อเลน:** `subagent`
- **ภาวะพร้อมกัน:** `agents.defaults.subagents.maxConcurrent` (ค่าเริ่มต้น `8`)

## ความมีชีวิตและการกู้คืน

OpenClaw ไม่ถือว่าการไม่มี `endedAt` เป็นหลักฐานถาวรว่า
เอเจนต์ย่อยยังมีชีวิตอยู่ การรันที่ยังไม่สิ้นสุดซึ่งเก่ากว่าหน้าต่างการรันค้าง
จะหยุดถูกนับเป็น active/pending ใน `/subagents list`, สรุปสถานะ,
การควบคุมการเสร็จสิ้นของลูกหลาน และการตรวจสอบภาวะพร้อมกันรายเซสชัน

หลังจาก Gateway รีสตาร์ท การรันที่กู้คืนมาแล้วแต่ค้างและยังไม่สิ้นสุดจะถูกตัดออก เว้นแต่
เซสชันลูกของการรันนั้นถูกทำเครื่องหมาย `abortedLastRun: true` เซสชันลูกที่ถูกยกเลิกจาก
การรีสตาร์ทเหล่านั้นยังคงกู้คืนได้ผ่านโฟลว์กู้คืนเอเจนต์ย่อยกำพร้า ซึ่งส่งข้อความ resume สังเคราะห์ก่อน
ล้างเครื่องหมาย aborted

<Note>
หากการสร้างเอเจนต์ย่อยล้มเหลวด้วย Gateway `PAIRING_REQUIRED` /
`scope-upgrade` ให้ตรวจสอบตัวเรียก RPC ก่อนแก้ไขสถานะการจับคู่
การประสานงาน `sessions_spawn` ภายในควรเชื่อมต่อเป็น
`client.id: "gateway-client"` พร้อม `client.mode: "backend"` ผ่านการยืนยันตัวตนด้วยโทเค็น/รหัสผ่านที่แชร์กันบน loopback โดยตรง เส้นทางนั้นไม่ได้ขึ้นกับ
ขอบเขตพื้นฐานของอุปกรณ์ที่จับคู่ของ CLI ตัวเรียกระยะไกล,
`deviceIdentity` ที่ระบุชัดเจน, เส้นทาง device-token ที่ระบุชัดเจน และไคลเอนต์ browser/node
ยังคงต้องได้รับการอนุมัติอุปกรณ์ตามปกติสำหรับการอัปเกรดขอบเขต
</Note>

## การหยุด

- การส่ง `/stop` ในแชตผู้ร้องขอจะยกเลิกเซสชันผู้ร้องขอและหยุดการรันเอเจนต์ย่อยที่ใช้งานอยู่ใดๆ ที่สร้างจากเซสชันนั้น โดยส่งผลต่อเนื่องไปยังลูกที่ซ้อนกัน
- `/subagents kill <id>` หยุดเอเจนต์ย่อยที่ระบุ และส่งผลต่อเนื่องไปยังลูกของเอเจนต์นั้น

## ข้อจำกัด

- การประกาศของเอเจนต์ย่อยเป็นแบบ **พยายามให้ดีที่สุด** หาก gateway รีสตาร์ท งาน "announce back" ที่ค้างอยู่จะหายไป
- เอเจนต์ย่อยยังคงใช้ทรัพยากรโปรเซส gateway เดียวกันร่วมกัน ให้ถือว่า `maxConcurrent` เป็นวาล์วนิรภัย
- `sessions_spawn` เป็นแบบไม่บล็อกเสมอ: จะคืนค่า `{ status: "accepted", runId, childSessionKey }` ทันที
- บริบทเอเจนต์ย่อยจะแทรกเฉพาะ `AGENTS.md` + `TOOLS.md` (ไม่มี `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md` หรือ `BOOTSTRAP.md`)
- ความลึกการซ้อนสูงสุดคือ 5 (ช่วง `maxSpawnDepth`: 1–5) แนะนำให้ใช้ความลึก 2 สำหรับกรณีใช้งานส่วนใหญ่
- `maxChildrenPerAgent` จำกัดจำนวนลูกที่ใช้งานอยู่ต่อเซสชัน (ค่าเริ่มต้น `5`, ช่วง `1–20`)
