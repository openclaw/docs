[เซสชัน Agent Client Protocol (ACP)](https://agentclientprotocol.com/) ช่วยให้
OpenClaw เรียกใช้ชุดเครื่องมือเขียนโค้ดภายนอก (Claude Code, Cursor, Copilot, Droid,
OpenClaw ACP, OpenCode, Gemini CLI และชุดเครื่องมือ ACPX อื่นๆ ที่รองรับ)
ผ่าน Plugin แบ็กเอนด์ ACP การสร้างแต่ละครั้งจะถูกติดตามเป็น
[งานเบื้องหลัง](/th/automation/tasks)

<Note>
**ACP เป็นเส้นทางสำหรับชุดเครื่องมือภายนอก ไม่ใช่เส้นทาง Codex เริ่มต้น** Plugin
Codex app-server แบบเนทีฟเป็นเจ้าของการควบคุม `/codex ...` และรันไทม์แบบฝัง
`openai/gpt-*` เริ่มต้นสำหรับรอบการทำงานของเอเจนต์ ส่วน ACP เป็นเจ้าของการควบคุม `/acp ...`
และเซสชัน `sessions_spawn({ runtime: "acp" })`

หากต้องการให้ Codex หรือ Claude Code เชื่อมต่อโดยตรงในฐานะไคลเอนต์ MCP ภายนอก
กับบทสนทนาในช่องทาง OpenClaw ที่มีอยู่ ให้ใช้
[`openclaw mcp serve`](/th/cli/mcp) แทน ACP
</Note>

## ใช้งานได้ทันทีหรือไม่?

ได้ หลังจากติดตั้ง Plugin รันไทม์ ACP อย่างเป็นทางการ:

```bash
openclaw plugins install @openclaw/acpx
openclaw config set plugins.entries.acpx.enabled true
```

การเช็กเอาต์ซอร์สสามารถใช้ Plugin เวิร์กสเปซ `extensions/acpx` ภายในเครื่องได้หลังจาก
`pnpm install` เรียกใช้ `/acp doctor` เพื่อตรวจสอบความพร้อม

OpenClaw จะแนะนำเอเจนต์เกี่ยวกับการสร้าง ACP เฉพาะเมื่อ ACP **ใช้งานได้จริง** เท่านั้น:
ต้องเปิดใช้งาน ACP, ห้ามปิดใช้งานการส่งต่อ, เซสชันปัจจุบันต้องไม่ถูกแซนด์บ็อกซ์บล็อก
และต้องโหลดแบ็กเอนด์รันไทม์ที่ทำงานเป็นปกติ หากเงื่อนไขใดไม่ผ่าน Skills ของ ACP
และคำแนะนำ ACP `sessions_spawn` จะยังคงถูกซ่อนไว้
เพื่อไม่ให้เอเจนต์แนะนำแบ็กเอนด์ที่ใช้งานไม่ได้

    - อะแดปเตอร์ Codex ACP มาพร้อมกับ Plugin `acpx` และจะเปิดใช้ภายในเครื่องเมื่อทำได้
    - Codex ACP ทำงานด้วย `CODEX_HOME` ที่แยกออกมา OpenClaw จะคัดลอกรายการความเชื่อถือของโปรเจกต์ที่เชื่อถือได้ พร้อมการกำหนดค่าเส้นทางโมเดล/ผู้ให้บริการที่ปลอดภัย (`model`, `model_provider`, `model_reasoning_effort`, `sandbox_mode` และฟิลด์ `model_providers.<name>` ที่ปลอดภัย) จากการกำหนดค่า Codex ของโฮสต์ ส่วนการยืนยันตัวตน การแจ้งเตือน และฮุกจะคงอยู่ในการกำหนดค่าของโฮสต์เท่านั้น
    - อะแดปเตอร์ชุดเครื่องมือเป้าหมายอื่นอาจถูกดึงเมื่อต้องการด้วย `npx` ในการใช้งานครั้งแรก
    - การยืนยันตัวตนของผู้จำหน่ายต้องมีอยู่บนโฮสต์สำหรับชุดเครื่องมือนั้นแล้ว
    - หากโฮสต์ไม่มี npm หรือการเข้าถึงเครือข่าย การดึงอะแดปเตอร์ครั้งแรกจะล้มเหลวจนกว่าจะอุ่นแคชล่วงหน้าหรือติดตั้งอะแดปเตอร์ด้วยวิธีอื่น

  </Accordion>

    - โมเดลที่เลือกมีอยู่สำหรับชุดเครื่องมือนั้น - รหัสโมเดลไม่สามารถใช้ข้ามชุดเครื่องมือได้
    - `cwd` ที่ร้องขอมีอยู่และเข้าถึงได้ หรือไม่ต้องระบุ `cwd` แล้วให้แบ็กเอนด์ใช้ค่าเริ่มต้น
    - โหมดสิทธิ์ตรงกับงาน เซสชันที่ไม่โต้ตอบไม่สามารถคลิกข้อความแจ้งสิทธิ์แบบเนทีฟได้ ดังนั้นการเขียนโค้ดที่ใช้การเขียน/เรียกใช้คำสั่งจำนวนมากมักต้องใช้โปรไฟล์สิทธิ์ ACPX ที่ดำเนินการต่อได้โดยไม่ต้องมีส่วนติดต่อผู้ใช้

เครื่องมือ Plugin ของ OpenClaw และเครื่องมือ OpenClaw ในตัวจะ **ไม่** เปิดเผยแก่
ชุดเครื่องมือ ACP โดยค่าเริ่มต้น เปิดใช้งานบริดจ์ MCP ที่ระบุอย่างชัดเจนใน
[เอเจนต์ ACP - การตั้งค่า](/th/tools/acp-agents-setup) เฉพาะเมื่อชุดเครื่องมือควร
เรียกเครื่องมือเหล่านั้นโดยตรง

## เป้าหมายชุดเครื่องมือที่รองรับ

เมื่อใช้แบ็กเอนด์ `acpx` ให้ใช้รหัสเหล่านี้เป็นเป้าหมาย `/acp spawn <id>` หรือ
`sessions_spawn({ runtime: "acp", agentId: "<id>" })`:

| รหัสชุดเครื่องมือ   | แบ็กเอนด์ทั่วไป                                | หมายเหตุ                                                                               |
| ------------ | ---------------------------------------------- | ----------------------------------------------------------------------------------- |
| `claude`     | อะแดปเตอร์ Claude Code ACP                        | ต้องมีการยืนยันตัวตน Claude Code บนโฮสต์                                              |
| `codex`      | อะแดปเตอร์ Codex ACP                              | เป็นทางเลือกสำรอง ACP ที่ระบุอย่างชัดเจน เฉพาะเมื่อ `/codex` แบบเนทีฟใช้งานไม่ได้หรือมีการร้องขอ ACP |
| `copilot`    | อะแดปเตอร์ GitHub Copilot ACP                     | ต้องมีการยืนยันตัวตนของ Copilot CLI/รันไทม์                                                  |
| `cursor`     | Cursor CLI ACP (`cursor-agent acp`)            | แทนที่คำสั่ง acpx หากการติดตั้งภายในเครื่องเปิดเผยจุดเริ่มต้น ACP ที่ต่างออกไป    |
| `droid`      | Factory Droid CLI                              | ต้องมีการยืนยันตัวตน Factory/Droid หรือ `FACTORY_API_KEY` ในสภาพแวดล้อมชุดเครื่องมือ        |
| `fast-agent` | อะแดปเตอร์ fast-agent-mcp ACP                     | ดึงเมื่อต้องการด้วย `uvx`                                                       |
| `gemini`     | อะแดปเตอร์ Gemini CLI ACP                         | ต้องมีการยืนยันตัวตน Gemini CLI หรือการตั้งค่าคีย์ API                                          |
| `iflow`      | iFlow CLI                                      | ความพร้อมใช้งานของอะแดปเตอร์และการควบคุมโมเดลขึ้นอยู่กับ CLI ที่ติดตั้ง                 |
| `kilocode`   | Kilo Code CLI                                  | ความพร้อมใช้งานของอะแดปเตอร์และการควบคุมโมเดลขึ้นอยู่กับ CLI ที่ติดตั้ง                 |
| `kimi`       | Kimi/Moonshot CLI                              | ต้องมีการยืนยันตัวตน Kimi/Moonshot บนโฮสต์                                            |
| `kiro`       | Kiro CLI                                       | ความพร้อมใช้งานของอะแดปเตอร์และการควบคุมโมเดลขึ้นอยู่กับ CLI ที่ติดตั้ง                 |
| `mux`        | อะแดปเตอร์ Mux CLI ACP                            | ดึงเมื่อต้องการด้วย `npx`                                                       |
| `opencode`   | อะแดปเตอร์ OpenCode ACP                           | ต้องมีการยืนยันตัวตนของ OpenCode CLI/ผู้ให้บริการ                                                |
| `openclaw`   | บริดจ์ OpenClaw Gateway ผ่าน `openclaw acp` | ช่วยให้ชุดเครื่องมือที่รองรับ ACP สื่อสารกลับไปยังเซสชัน OpenClaw Gateway                 |
| `qoder`      | Qoder CLI                                      | ความพร้อมใช้งานของอะแดปเตอร์และการควบคุมโมเดลขึ้นอยู่กับ CLI ที่ติดตั้ง                 |

สามารถกำหนดค่านามแฝงของเอเจนต์ acpx แบบกำหนดเองใน acpx ได้ แต่ OpenClaw
ยังคงตรวจสอบนโยบาย `acp.allowedAgents` และการแมป
`agents.list[].runtime.acp.agent` ก่อนส่งต่อ

## คู่มือปฏิบัติงานสำหรับผู้ดูแลระบบ

ขั้นตอน `/acp` แบบย่อจากแชต:

<Steps>
  <Step title="สร้าง">
    `/acp spawn claude --bind here`,
    `/acp spawn gemini --mode persistent --thread auto` หรือ
    `/acp spawn codex --bind here` ที่ระบุอย่างชัดเจน
  </Step>
  <Step title="ทำงาน">
    ดำเนินการต่อในบทสนทนาหรือเธรดที่ผูกไว้ (หรือระบุคีย์เซสชัน
    เป็นเป้าหมายอย่างชัดเจน)
  </Step>
  <Step title="ตรวจสอบสถานะ">
    `/acp status`
  </Step>
  <Step title="ปรับแต่ง">
    `/acp model <provider/model>`, `/acp permissions <profile>`,
    `/acp timeout <seconds>`
  </Step>
  <Step title="ชี้นำ">
    โดยไม่แทนที่บริบท: `/acp steer tighten logging and continue`
  </Step>
  <Step title="หยุด">
    `/acp cancel` (รอบปัจจุบัน) หรือ `/acp close` (เซสชัน + การผูก)
  </Step>
</Steps>

    - ข้อความติดตามผลที่ผูกไว้จะส่งตรงไปยังเซสชัน ACP จนกว่าการผูกจะถูกปิด เลิกโฟกัส รีเซ็ต หรือหมดอายุ
    - คำสั่ง Gateway จะยังคงประมวลผลภายในระบบ `/acp ...`, `/status` และ `/unfocus` จะไม่ถูกส่งเป็นข้อความพรอมต์ปกติไปยัง ACP harness ที่ผูกไว้
    - `cancel` จะยกเลิกรอบที่กำลังทำงานเมื่อแบ็กเอนด์รองรับการยกเลิก แต่จะไม่ลบการผูกหรือเมทาดาทาเซสชัน

## ACP เทียบกับเอเจนต์ย่อย

ใช้ ACP เมื่อต้องการรันไทม์ harness ภายนอก ใช้ **app-server Codex
แบบเนทีฟ** สำหรับการผูก/ควบคุมการสนทนา Codex เมื่อเปิดใช้ Plugin `codex`
ใช้ **เอเจนต์ย่อย** เมื่อต้องการการเรียกใช้แบบมอบหมายที่เป็นเนทีฟของ OpenClaw

| พื้นที่          | เซสชัน ACP                           | การเรียกใช้เอเจนต์ย่อย                      |
| ------------- | ------------------------------------- | ---------------------------------- |
| รันไทม์       | Plugin แบ็กเอนด์ ACP (เช่น acpx) | รันไทม์เอเจนต์ย่อยแบบเนทีฟของ OpenClaw  |
| คีย์เซสชัน   | `agent:<agentId>:acp:<uuid>`          | `agent:<agentId>:subagent:<uuid>`  |
| คำสั่งหลัก | `/acp ...`                            | `/subagents ...`                   |
| เครื่องมือสร้าง    | `sessions_spawn` พร้อม `runtime:"acp"` | `sessions_spawn` (รันไทม์เริ่มต้น) |

ดูเพิ่มเติมที่ [เอเจนต์ย่อย](/th/tools/subagents)

## วิธีที่ ACP เรียกใช้ Claude Code

สำหรับ Claude Code ผ่าน ACP สแต็กประกอบด้วย:

1. ระนาบควบคุมเซสชัน ACP ของ OpenClaw
2. Plugin รันไทม์ `@openclaw/acpx` อย่างเป็นทางการ
3. อะแดปเตอร์ Claude ACP
4. กลไกรันไทม์/เซสชันฝั่ง Claude

ACP Claude เป็น **เซสชัน harness** ที่มีการควบคุม ACP การดำเนินเซสชันต่อ
การติดตามงานเบื้องหลัง และการผูกการสนทนา/เธรดที่เลือกใช้ได้

แบ็กเอนด์ CLI เป็นรันไทม์สำรองภายในเครื่องแบบข้อความเท่านั้นที่แยกจากกัน - ดู
[แบ็กเอนด์ CLI](/th/gateway/cli-backends)

สำหรับผู้ปฏิบัติงาน กฎที่ใช้จริงคือ:

- **ต้องการ `/acp spawn`, เซสชันที่ผูกได้, การควบคุมรันไทม์ หรืองาน harness แบบถาวรใช่หรือไม่** ใช้ ACP
- **ต้องการทางเลือกสำรองแบบข้อความภายในเครื่องอย่างง่ายผ่าน CLI โดยตรงใช่หรือไม่** ใช้แบ็กเอนด์ CLI

## เซสชันที่ผูกไว้

### แบบจำลองทางความคิด

- **พื้นผิวแชต** - ตำแหน่งที่ผู้คนสนทนาต่อเนื่อง (ช่อง Discord, หัวข้อ Telegram, แชต iMessage)
- **เซสชัน ACP** - สถานะรันไทม์ Codex/Claude/Gemini แบบคงทนที่ OpenClaw กำหนดเส้นทางไปยัง
- **เธรด/หัวข้อย่อย** - พื้นผิวการส่งข้อความเพิ่มเติมที่เลือกใช้ได้ ซึ่งสร้างโดย `--thread ...` เท่านั้น
- **พื้นที่ทำงานรันไทม์** - ตำแหน่งระบบไฟล์ (`cwd`, เช็กเอาต์รีโพ, พื้นที่ทำงานแบ็กเอนด์) ที่ harness ทำงาน โดยไม่ขึ้นกับพื้นผิวแชต

### การผูกกับการสนทนาปัจจุบัน

`/acp spawn <harness> --bind here` จะตรึงการสนทนาปัจจุบันไว้กับ
เซสชัน ACP ที่สร้างขึ้น โดยไม่มีเธรดย่อยและใช้พื้นผิวแชตเดิม OpenClaw ยังคง
เป็นเจ้าของการขนส่ง การยืนยันตัวตน ความปลอดภัย และการส่ง ข้อความติดตามผลในการ
สนทนานั้นจะถูกกำหนดเส้นทางไปยังเซสชันเดิม ส่วน `/new` และ `/reset` จะรีเซ็ตเซสชัน
ในตำแหน่งเดิม และ `/acp close` จะนำการผูกออก

ตัวอย่าง:

```text
/codex bind                                              # ผูก Codex แบบเนทีฟ กำหนดเส้นทางข้อความในอนาคตมาที่นี่
/codex model gpt-5.4                                     # ปรับแต่งเธรด Codex แบบเนทีฟที่ผูกไว้
/codex stop                                              # ควบคุมรอบ Codex แบบเนทีฟที่กำลังทำงาน
/acp spawn codex --bind here                             # ทางเลือกสำรอง ACP อย่างชัดเจนสำหรับ Codex
/acp spawn codex --thread auto                           # อาจสร้างเธรด/หัวข้อย่อยและผูกไว้ที่นั่น
/acp spawn codex --bind here --cwd /workspace/repo       # ผูกกับแชตเดิม โดย Codex ทำงานใน /workspace/repo
```

    การรองรับการผูกเธรดขึ้นอยู่กับอะแดปเตอร์ หากอะแดปเตอร์ช่องที่ใช้งานอยู่
    ไม่รองรับการผูกเธรด OpenClaw จะส่งข้อความที่ชัดเจนว่า
    ไม่รองรับ/ไม่พร้อมใช้งาน

  </Accordion>
  <Accordion title="ช่องที่รองรับเธรด">
    - อะแดปเตอร์ช่องใด ๆ ที่เปิดเผยความสามารถในการผูกเซสชัน/เธรด
    - การรองรับในตัวปัจจุบัน: เธรด/ช่อง **Discord**, หัวข้อ **Telegram** (หัวข้อฟอรัมในกลุ่ม/ซูเปอร์กรุ๊ปและหัวข้อ DM)
    - ช่อง Plugin สามารถเพิ่มการรองรับผ่านอินเทอร์เฟซการผูกเดียวกัน

  </Accordion>
</AccordionGroup>

## การผูกช่องแบบถาวร

สำหรับเวิร์กโฟลว์ที่ไม่ใช่แบบชั่วคราว ให้กำหนดค่าการผูก ACP แบบถาวรในรายการ
`bindings[]` ระดับบนสุด

### โมเดลการผูก

<ParamField path="bindings[].type" type='"acp"'>
  ระบุการผูกการสนทนา ACP แบบถาวร
</ParamField>
<ParamField path="bindings[].match" type="object">
  ระบุการสนทนาเป้าหมาย รูปแบบสำหรับแต่ละช่อง:

- **ช่อง/เธรด Discord:** `match.channel="discord"` + `match.peer.id="<channelOrThreadId>"`
- **ช่อง/DM Slack:** `match.channel="slack"` + `match.peer.id="<channelId|channel:<channelId>|#<channelId>|userId|user:<userId>|slack:<userId>|<@userId>>"` ควรใช้ ID ของ Slack ที่คงที่ การผูกช่องจะจับคู่การตอบกลับภายในเธรดของช่องนั้นด้วย
- **หัวข้อฟอรัม Telegram:** `match.channel="telegram"` + `match.peer.id="<chatId>:topic:<topicId>"`
- **DM/กลุ่ม WhatsApp:** `match.channel="whatsapp"` + `match.peer.id="<E.164|group JID>"` ใช้หมายเลข E.164 เช่น `+15555550123` สำหรับแชตโดยตรง และใช้ JID ของกลุ่ม WhatsApp เช่น `120363424282127706@g.us` สำหรับกลุ่ม
- **DM/กลุ่ม iMessage:** `match.channel="imessage"` + `match.peer.id="<handle|chat_id:*|chat_guid:*|chat_identifier:*>"` ควรใช้ `chat_id:*` เพื่อให้การผูกกลุ่มมีความคงที่

</ParamField>
<ParamField path="bindings[].agentId" type="string">
  ID ของเอเจนต์ OpenClaw ที่เป็นเจ้าของ
</ParamField>
<ParamField path="bindings[].acp.mode" type='"persistent" | "oneshot"'>
  การแทนที่ ACP ที่ระบุหรือไม่ก็ได้
</ParamField>
<ParamField path="bindings[].acp.label" type="string">
  ป้ายกำกับสำหรับผู้ปฏิบัติงานที่ระบุหรือไม่ก็ได้
</ParamField>
<ParamField path="bindings[].acp.cwd" type="string">
  ไดเรกทอรีทำงานของรันไทม์ที่ระบุหรือไม่ก็ได้
</ParamField>
<ParamField path="bindings[].acp.backend" type="string">
  การแทนที่แบ็กเอนด์ที่ระบุหรือไม่ก็ได้
</ParamField>

### ค่าเริ่มต้นของรันไทม์ต่อเอเจนต์

ใช้ `agents.list[].runtime` เพื่อกำหนดค่าเริ่มต้นของ ACP ครั้งเดียวต่อเอเจนต์:

- `agents.list[].runtime.type="acp"`
- `agents.list[].runtime.acp.agent` (ID ของระบบดำเนินการ เช่น `codex` หรือ `claude`)
- `agents.list[].runtime.acp.backend`
- `agents.list[].runtime.acp.mode`
- `agents.list[].runtime.acp.cwd`

**ลำดับความสำคัญของการแทนที่สำหรับเซสชันที่ผูกกับ ACP:**

1. `bindings[].acp.*`
2. `agents.list[].runtime.acp.*`
3. ค่าเริ่มต้นส่วนกลางของ ACP (เช่น `acp.backend`)

### ตัวอย่าง

```json5
{
  agents: {
    list: [
      {
        id: "codex",
        runtime: {
          type: "acp",
          acp: {
            agent: "codex",
            backend: "acpx",
            mode: "persistent",
            cwd: "/workspace/openclaw",
          },
        },
      },
      {
        id: "claude",
        runtime: {
          type: "acp",
          acp: { agent: "claude", backend: "acpx", mode: "persistent" },
        },
      },
    ],
  },
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "discord",
        accountId: "default",
        peer: { kind: "channel", id: "222222222222222222" },
      },
      acp: { label: "codex-main" },
    },
    {
      type: "acp",
      agentId: "claude",
      match: {
        channel: "telegram",
        accountId: "default",
        peer: { kind: "group", id: "-1001234567890:topic:42" },
      },
      acp: { cwd: "/workspace/repo-b" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "discord", accountId: "default" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "telegram", accountId: "default" },
    },
  ],
  channels: {
    discord: {
      guilds: {
        "111111111111111111": {
          channels: {
            "222222222222222222": { requireMention: false },
          },
        },
      },
    },
    telegram: {
      groups: {
        "-1001234567890": {
          topics: { "42": { requireMention: false } },
        },
      },
    },
  },
}
```

### ลักษณะการทำงาน

- OpenClaw ตรวจสอบว่าเซสชัน ACP ที่กำหนดค่าไว้มีอยู่หลังจากผ่านการอนุญาตเฉพาะช่องและก่อนใช้งาน
- ข้อความในช่อง หัวข้อ หรือแชตนั้นจะกำหนดเส้นทางไปยังเซสชัน ACP ที่กำหนดค่าไว้
- การผูก ACP ที่กำหนดค่าไว้เป็นเจ้าของเส้นทางเซสชันของตนเอง การกระจายข้อความของช่องแบบ fan-out จะไม่แทนที่เซสชัน ACP ที่กำหนดค่าไว้สำหรับการผูกที่ตรงกัน
- ในการสนทนาที่ผูกไว้ `/new` และ `/reset` จะรีเซ็ตคีย์เซสชัน ACP เดิมโดยตรง
- การผูกรันไทม์ชั่วคราว (เช่น ที่สร้างโดยขั้นตอนการโฟกัสเธรด) ยังคงมีผลเมื่อมีอยู่
- สำหรับการสร้าง ACP ข้ามเอเจนต์โดยไม่มี `cwd` ที่ระบุอย่างชัดเจน OpenClaw จะสืบทอดพื้นที่ทำงานของเอเจนต์เป้าหมายจากการกำหนดค่าเอเจนต์
- เส้นทางพื้นที่ทำงานที่สืบทอดมาซึ่งไม่มีอยู่จะย้อนกลับไปใช้ cwd เริ่มต้นของแบ็กเอนด์ ส่วนความล้มเหลวในการเข้าถึงเส้นทางที่มีอยู่จะแสดงเป็นข้อผิดพลาดในการสร้าง

## เริ่มเซสชัน ACP

มีสองวิธีในการเริ่มเซสชัน ACP:

<Tabs>
  <Tab title="จาก sessions_spawn">
    ใช้ `runtime: "acp"` เพื่อเริ่มเซสชัน ACP จากรอบการทำงานของเอเจนต์หรือการเรียกใช้
    เครื่องมือ

    ```json
    {
      "task": "เปิด repo และสรุปการทดสอบที่ล้มเหลว",
      "runtime": "acp",
      "agentId": "codex",
      "thread": true,
      "mode": "session"
    }
    ```

    <Note>
    `runtime` มีค่าเริ่มต้นเป็น `subagent` ดังนั้นให้กำหนด `runtime: "acp"` อย่างชัดเจนสำหรับ
    เซสชัน ACP หากละ `agentId` ไว้ OpenClaw จะใช้ `acp.defaultAgent`
    เมื่อมีการกำหนดค่าไว้ `mode: "session"` ต้องใช้ `thread: true` เพื่อรักษา
    การสนทนาที่ผูกไว้อย่างถาวร
    </Note>

  </Tab>
  <Tab title="จากคำสั่ง /acp">
    ใช้ `/acp spawn` เพื่อให้ผู้ปฏิบัติงานควบคุมจากแชตอย่างชัดเจน

    ```text
    /acp spawn codex --mode persistent --thread auto
    /acp spawn codex --mode oneshot --thread off
    /acp spawn codex --bind here
    /acp spawn codex --thread here
    ```

    แฟล็กหลัก:

    - `--mode persistent|oneshot`
    - `--bind here|off`
    - `--thread auto|here|off`
    - `--cwd <absolute-path>`
    - `--label <name>`

    ดู [คำสั่ง Slash](/th/tools/slash-commands)

  </Tab>
</Tabs>

</ParamField>
<ParamField path="label" type="string">

## โหมดการผูกและเธรดสำหรับการสร้าง

<Tabs>
  <Tab title="--bind here|off">
    | โหมด   | ลักษณะการทำงาน                                                               |
    | ------ | ----------------------------------------------------------------------- |
    | `here` | ผูกการสนทนาที่ใช้งานอยู่ในปัจจุบันโดยตรง และล้มเหลวหากไม่มีการสนทนาที่ใช้งานอยู่ |
    | `off`  | ไม่สร้างการผูกกับการสนทนาปัจจุบัน                          |

    หมายเหตุ:

    - `--bind here` เป็นเส้นทางที่ง่ายที่สุดสำหรับผู้ปฏิบัติงานในการ "ทำให้ช่องหรือแชตนี้ใช้ Codex เป็นแบ็กเอนด์"
    - `--bind here` ไม่สร้างเธรดลูก
    - `--bind here` ใช้ได้เฉพาะกับช่องที่รองรับการผูกการสนทนาปัจจุบัน
    - `--bind` และ `--thread` ไม่สามารถใช้ร่วมกันในการเรียก `/acp spawn` ครั้งเดียวกัน

  </Tab>
  <Tab title="--thread auto|here|off">
    | โหมด   | ลักษณะการทำงาน                                                                                            |
    | ------ | ------------------------------------------------------------------------------------------------- |
    | `auto` | เมื่ออยู่ในเธรดที่ใช้งานอยู่: ผูกเธรดนั้น เมื่ออยู่นอกเธรด: สร้าง/ผูกเธรดลูกหากรองรับ |
    | `here` | ต้องมีเธรดที่ใช้งานอยู่ในปัจจุบัน และล้มเหลวหากไม่ได้อยู่ในเธรด                                                  |
    | `off`  | ไม่มีการผูก เซสชันจะเริ่มต้นโดยไม่ผูก                                                                 |

    หมายเหตุ:

    - บนพื้นผิวการผูกที่ไม่ใช่เธรด ลักษณะการทำงานเริ่มต้นจะเทียบเท่ากับ `off`
    - การสร้างแบบผูกกับเธรดต้องอาศัยการรองรับจากนโยบายช่อง:
      - Discord: `channels.discord.threadBindings.spawnSessions=true`
      - Telegram: `channels.telegram.threadBindings.spawnSessions=true`
    - ใช้ `--bind here` เมื่อต้องการตรึงการสนทนาปัจจุบันโดยไม่สร้างเธรดลูก

  </Tab>
</Tabs>

## รูปแบบการส่งมอบ

เซสชัน ACP อาจเป็นพื้นที่ทำงานแบบโต้ตอบหรืองานเบื้องหลังที่เซสชันแม่เป็นเจ้าของ
เส้นทางการส่งมอบขึ้นอยู่กับรูปแบบนั้น

<AccordionGroup>
  <Accordion title="เซสชัน ACP แบบโต้ตอบ">
    เซสชันแบบโต้ตอบมีไว้เพื่อสนทนาต่อบนพื้นผิวแชตที่มองเห็นได้:

    - `/acp spawn ... --bind here` ผูกการสนทนาปัจจุบันกับเซสชัน ACP
    - `/acp spawn ... --thread ...` ผูกเธรด/หัวข้อของช่องกับเซสชัน ACP
    - `bindings[].type="acp"` แบบถาวรที่กำหนดค่าไว้จะกำหนดเส้นทางการสนทนาที่ตรงกันไปยังเซสชัน ACP เดิม

    ข้อความติดตามผลในการสนทนาที่ผูกไว้จะกำหนดเส้นทางไปยังเซสชัน ACP
    โดยตรง และผลลัพธ์ ACP จะถูกส่งกลับไปยัง
    ช่อง/เธรด/หัวข้อเดิมนั้น

    สิ่งที่ OpenClaw ส่งไปยังระบบดำเนินการ:

    - การติดตามผลแบบผูกตามปกติจะถูกส่งเป็นข้อความพรอมต์ รวมถึงไฟล์แนบเฉพาะเมื่อฮาร์เนส/แบ็กเอนด์รองรับเท่านั้น
    - คำสั่งจัดการ `/acp` และคำสั่ง Gateway ภายในเครื่องจะถูกสกัดไว้ก่อนส่งต่อไปยัง ACP
    - เหตุการณ์การเสร็จสิ้นที่รันไทม์สร้างขึ้นจะถูกทำให้เป็นรูปธรรมแยกตามเป้าหมาย เอเจนต์ OpenClaw จะได้รับเอนเวโลปบริบทรันไทม์ภายในของ OpenClaw ส่วนฮาร์เนส ACP ภายนอกจะได้รับพรอมต์แบบข้อความธรรมดาที่มีผลลัพธ์ของลูกและคำสั่ง ไม่ควรส่งเอนเวโลป `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>` แบบดิบไปยังฮาร์เนสภายนอกหรือเก็บถาวรเป็นข้อความทรานสคริปต์ผู้ใช้ ACP
    - รายการทรานสคริปต์ ACP ใช้ข้อความทริกเกอร์ที่ผู้ใช้มองเห็นหรือพรอมต์การเสร็จสิ้นแบบข้อความธรรมดา เมทาดาทาเหตุการณ์ภายในจะยังคงมีโครงสร้างอยู่ใน OpenClaw เมื่อทำได้ และไม่ถือเป็นเนื้อหาแชตที่ผู้ใช้เขียน

  </Accordion>
  <Accordion title="เซสชัน ACP แบบครั้งเดียวที่พาเรนต์เป็นเจ้าของ">

    คล้ายกับเอเจนต์ย่อย:

    - พาเรนต์ร้องขอให้ทำงานด้วย `sessions_spawn({ runtime: "acp", mode: "run" })`
    - ลูกทำงานในเซสชันฮาร์เนส ACP ของตนเอง
    - เทิร์นของลูกทำงานบนเลนเบื้องหลังเดียวกับที่ใช้สร้างเอเจนต์ย่อยแบบเนทีฟ ดังนั้นฮาร์เนส ACP ที่ช้าจึงไม่บล็อกงานในเซสชันหลักที่ไม่เกี่ยวข้อง
    - การเสร็จสิ้นจะรายงานกลับผ่านเส้นทางประกาศการเสร็จสิ้นของงาน OpenClaw จะแปลงเมทาดาทาการเสร็จสิ้นภายในเป็นพรอมต์ ACP แบบข้อความธรรมดาก่อนส่งไปยังฮาร์เนสภายนอก ดังนั้นฮาร์เนสจะไม่เห็นมาร์กเกอร์บริบทรันไทม์ที่มีเฉพาะใน OpenClaw
    - พาเรนต์จะเขียนผลลัพธ์ของลูกใหม่ด้วยน้ำเสียงผู้ช่วยตามปกติเมื่อการตอบกลับที่ผู้ใช้มองเห็นมีประโยชน์

    **อย่า** ถือว่าเส้นทางนี้เป็นการแชตแบบเพียร์ทูเพียร์ระหว่างพาเรนต์กับ
    ลูก ลูกมีช่องทางส่งผลการเสร็จสิ้นกลับไปยังพาเรนต์อยู่แล้ว

  </Accordion>
  <Accordion title="sessions_send และการส่งแบบ A2A">

    - ขอให้เป้าหมายสร้างข้อความประกาศ
    - ส่งประกาศนั้นไปยังช่องหรือเธรดที่มองเห็นได้

    เส้นทาง A2A ดังกล่าวเป็นทางเลือกสำรองสำหรับการส่งแบบเพียร์ที่ผู้ส่งต้องการ
    การติดตามผลที่มองเห็นได้ เส้นทางนี้ยังคงเปิดใช้งานเมื่อเซสชันที่ไม่เกี่ยวข้องสามารถมองเห็นและ
    ส่งข้อความถึงเป้าหมาย ACP ได้ ตัวอย่างเช่น ภายใต้การตั้งค่า `tools.sessions.visibility`
    แบบกว้าง

    OpenClaw จะข้ามการติดตามผลแบบ A2A เฉพาะเมื่อผู้ร้องขอเป็นพาเรนต์ของ
    ลูก ACP แบบครั้งเดียวที่พาเรนต์เป็นเจ้าของของตนเอง ในกรณีนั้น การรัน A2A ซ้อนบน
    การเสร็จสิ้นของงานอาจปลุกพาเรนต์ด้วยผลลัพธ์ของลูก ส่งต่อ
    คำตอบของพาเรนต์กลับเข้าไปในลูก และสร้างลูปเสียงสะท้อนระหว่างพาเรนต์/ลูก
    ผลลัพธ์ `sessions_send` จะรายงาน `delivery.status="skipped"` สำหรับ
    กรณีลูกที่เป็นเจ้าของนั้น เนื่องจากเส้นทางการเสร็จสิ้นมีหน้าที่รับผิดชอบ
    ผลลัพธ์อยู่แล้ว

  </Accordion>

    ```json
    {
      "task": "ดำเนินการต่อจากจุดที่เราค้างไว้ - แก้ไขการทดสอบที่ยังล้มเหลืออยู่",
      "runtime": "acp",
      "agentId": "codex",
      "resumeSessionId": "<previous-session-id>"
    }
    ```

    กรณีใช้งานทั่วไป:

    - ส่งต่อเซสชัน Codex จากแล็ปท็อปไปยังโทรศัพท์ โดยบอกเอเจนต์ให้ดำเนินการต่อจากจุดที่ค้างไว้
    - ดำเนินเซสชันเขียนโค้ดที่เริ่มแบบโต้ตอบใน CLI ต่อ โดยตอนนี้ทำงานแบบไม่มีส่วนติดต่อผ่านเอเจนต์
    - ดำเนินงานที่หยุดชะงักจากการรีสตาร์ต Gateway หรือหมดเวลาเนื่องจากไม่มีการใช้งานต่อ

    หมายเหตุ:

    - `resumeSessionId` ใช้ได้เฉพาะเมื่อ `runtime: "acp"`; รันไทม์เอเจนต์ย่อยเริ่มต้นจะไม่สนใจฟิลด์ที่มีเฉพาะใน ACP นี้
    - `streamTo` ใช้ได้เฉพาะเมื่อ `runtime: "acp"`; รันไทม์เอเจนต์ย่อยเริ่มต้นจะไม่สนใจฟิลด์ที่มีเฉพาะใน ACP นี้
    - `resumeSessionId` เป็น ID สำหรับดำเนินการต่อของ ACP/ฮาร์เนสภายในโฮสต์ ไม่ใช่คีย์เซสชันช่องของ OpenClaw; OpenClaw จะยังคงตรวจสอบนโยบายการสร้าง ACP และนโยบายเอเจนต์เป้าหมายก่อนส่งต่อ ขณะที่แบ็กเอนด์ ACP หรือฮาร์เนสเป็นผู้รับผิดชอบการอนุญาตให้โหลด ID ต้นทางนั้น
    - `resumeSessionId` จะกู้คืนประวัติการสนทนา ACP ต้นทาง; `thread` และ `mode` ยังคงใช้ตามปกติกับเซสชัน OpenClaw ใหม่ที่กำลังสร้าง ดังนั้น `mode: "session"` ยังคงต้องใช้ `thread: true`
    - เอเจนต์เป้าหมายต้องรองรับ `session/load` (Codex และ Claude Code รองรับ)
    - หากไม่พบ ID เซสชัน การสร้างจะล้มเหลวพร้อมข้อผิดพลาดที่ชัดเจน โดยไม่มีการถอยกลับไปสร้างเซสชันใหม่แบบเงียบ ๆ

  </Accordion>
  <Accordion title="การทดสอบควันหลังการปรับใช้">
    หลังจากปรับใช้ Gateway ให้เรียกใช้การตรวจสอบแบบต้นทางถึงปลายทางจริงแทนการเชื่อถือ
    การทดสอบหน่วย:

    1. ตรวจสอบเวอร์ชันและคอมมิตของ Gateway ที่ปรับใช้บนโฮสต์เป้าหมาย
    2. เปิดเซสชันบริดจ์ ACPX ชั่วคราวไปยังเอเจนต์จริง
    3. ขอให้เอเจนต์นั้นเรียก `sessions_spawn` ด้วย `runtime: "acp"`, `agentId: "codex"`, `mode: "run"` และงาน `Reply with exactly LIVE-ACP-SPAWN-OK`
    4. ตรวจสอบ `accepted=yes`, `childSessionKey` จริง และไม่มีข้อผิดพลาดจากตัวตรวจสอบ
    5. ล้างเซสชันบริดจ์ชั่วคราว

    ให้เกตคงอยู่ที่ `mode: "run"` และข้าม `streamTo: "parent"` -
    `mode: "session"` ที่ผูกกับเธรดและเส้นทางรีเลย์สตรีมเป็นรอบการผสานรวม
    ที่สมบูรณ์กว่าและแยกจากกัน

  </Accordion>
</AccordionGroup>

## ความเข้ากันได้กับแซนด์บ็อกซ์

ปัจจุบันเซสชัน ACP ทำงานบนรันไทม์ของโฮสต์ **ไม่ใช่** ภายในแซนด์บ็อกซ์
ของ OpenClaw

<Warning>
**ขอบเขตความปลอดภัย:**

- ฮาร์เนสภายนอกสามารถอ่าน/เขียนตามสิทธิ์ CLI ของตนเองและ `cwd` ที่เลือก
- นโยบายแซนด์บ็อกซ์ของ OpenClaw **ไม่** ครอบการทำงานของฮาร์เนส ACP
- OpenClaw ยังคงบังคับใช้เกตฟีเจอร์ ACP เอเจนต์ที่อนุญาต ความเป็นเจ้าของเซสชัน การผูกช่อง และนโยบายการส่งของ Gateway
- ใช้ `runtime: "subagent"` สำหรับงานแบบเนทีฟของ OpenClaw ที่บังคับใช้แซนด์บ็อกซ์

</Warning>

ข้อจำกัดปัจจุบัน:

- หากเซสชันผู้ร้องขออยู่ในแซนด์บ็อกซ์ การสร้าง ACP จะถูกบล็อกทั้งสำหรับ `sessions_spawn({ runtime: "acp" })` และ `/acp spawn`
- `sessions_spawn` พร้อม `runtime: "acp"` ไม่รองรับ `sandbox: "require"`

## การแก้ไขเป้าหมายเซสชัน

การดำเนินการ `/acp` ส่วนใหญ่ยอมรับเป้าหมายเซสชันที่ระบุหรือไม่ก็ได้ (`session-key`,
`session-id` หรือ `session-label`)

**ลำดับการแก้ไข:**

1. อาร์กิวเมนต์เป้าหมายที่ระบุอย่างชัดเจน (หรือ `--session` สำหรับ `/acp steer`)
   - ลองใช้คีย์
   - จากนั้นใช้ ID เซสชันที่มีรูปแบบ UUID
   - จากนั้นใช้ป้ายกำกับ
2. การผูกเธรดปัจจุบัน (หากการสนทนา/เธรดนี้ผูกกับเซสชัน ACP)
3. การถอยกลับไปยังเซสชันผู้ร้องขอปัจจุบัน

ทั้งการผูกการสนทนาปัจจุบันและการผูกเธรดมีส่วนร่วมในขั้นตอนที่ 2

หากไม่สามารถแก้ไขเป้าหมายได้ OpenClaw จะส่งคืนข้อผิดพลาดที่ชัดเจน
(`Unable to resolve session target: ...`)

| `/acp install`       | พิมพ์ขั้นตอนการติดตั้งและเปิดใช้งานแบบกำหนดแน่นอน             | `/acp install`                                                |

การควบคุมรันไทม์ (`spawn`, `cancel`, `steer`, `close`, `status`, `set-mode`,
`set`, `cwd`, `permissions`, `timeout`, `model` และ `reset-options`) ต้องใช้
ข้อมูลระบุตัวตนของเจ้าของจากช่องภายนอกและ `operator.admin` จากไคลเอนต์
Gateway ภายใน ผู้ส่งที่ได้รับอนุญาตแต่ไม่ใช่เจ้าของยังคงใช้ `sessions`,
`doctor`, `install` และ `help` ได้ สำหรับผู้ส่งที่ไม่ใช่เจ้าของ `/acp sessions`
จะแสดงเฉพาะเซสชันที่ผูกอยู่ในปัจจุบันหรือเซสชันผู้ร้องขอ; ข้อมูลระบุตัวตนของเจ้าของและ
ไคลเอนต์ `operator.admin` จะเห็นเซสชันล่าสุดทั้งหมด

`/acp status` แสดงตัวเลือกรันไทม์ที่มีผล รวมถึงตัวระบุเซสชันระดับรันไทม์และ
ระดับแบ็กเอนด์ ข้อผิดพลาดของการควบคุมที่ไม่รองรับจะแสดงอย่าง
ชัดเจนเมื่อแบ็กเอนด์ไม่มีความสามารถ คำสั่งที่ยอมรับโทเค็นเป้าหมาย
(`session-key`, `session-id` หรือ `session-label`) จะแก้ไขโทเค็นเหล่านั้นผ่านการค้นหาเซสชันของ Gateway
รวมถึงรูท `session.store` แบบกำหนดเองต่อเอเจนต์ `/acp sessions`
ไม่ยอมรับโทเค็นเป้าหมาย

### การแมปตัวเลือกรันไทม์

`/acp` มีคำสั่งอำนวยความสะดวกและตัวตั้งค่าทั่วไป การดำเนินการที่เทียบเท่ากัน:

| คำสั่ง                      | แมปไปยัง                              | หมายเหตุ                                                                                                                                                                                                      |
| ---------------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/acp model <id>`            | คีย์การกำหนดค่ารันไทม์ `model`           | สำหรับ Codex ACP นั้น OpenClaw จะปรับ `openai/<model>` ให้เป็นรหัสโมเดลของอะแดปเตอร์ และแมปส่วนต่อท้ายการให้เหตุผลที่คั่นด้วยเครื่องหมายทับ เช่น `openai/gpt-5.4/high` ไปยัง `reasoning_effort`                                         |
| `/acp set thinking <level>`  | ตัวเลือกมาตรฐาน `thinking`          | OpenClaw จะส่งค่าที่เทียบเท่าซึ่งแบ็กเอนด์ประกาศไว้เมื่อมี โดยเลือก `thinking` ก่อน ตามด้วย `effort`, `reasoning_effort` หรือ `thought_level` สำหรับ Codex ACP อะแดปเตอร์จะแมปค่าไปยัง `reasoning_effort` |
| `/acp permissions <profile>` | ตัวเลือกมาตรฐาน `permissionProfile` | OpenClaw จะส่งค่าที่เทียบเท่าซึ่งแบ็กเอนด์ประกาศไว้เมื่อมี เช่น `approval_policy`, `permission_profile`, `permissions` หรือ `permission_mode`                                                       |
| `/acp timeout <seconds>`     | ตัวเลือกมาตรฐาน `timeoutSeconds`    | OpenClaw จะส่งค่าที่เทียบเท่าซึ่งแบ็กเอนด์ประกาศไว้เมื่อมี เช่น `timeout` หรือ `timeout_seconds`                                                                                                     |
| `/acp cwd <path>`            | การแทนที่ cwd ของรันไทม์                 | อัปเดตโดยตรง                                                                                                                                                                                             |
| `/acp set <key> <value>`     | ทั่วไป                              | `key=cwd` ใช้พาธแทนที่ cwd                                                                                                                                                                      |
| `/acp reset-options`         | ล้างค่าที่แทนที่ของรันไทม์ทั้งหมด         | -                                                                                                                                                                                                          |

## ฮาร์เนส acpx การตั้งค่า Plugin และสิทธิ์

สำหรับการกำหนดค่าฮาร์เนส acpx (นามแฝง Claude Code / Codex / Gemini CLI)
บริดจ์ MCP ของ plugin-tools และ OpenClaw-tools รวมถึงโหมดสิทธิ์ของ ACP
โปรดดู [เอเจนต์ ACP - การตั้งค่า](/th/tools/acp-agents-setup)

| ไม่พบคำสั่งฮาร์เนส                                                                 | ไม่ได้ติดตั้ง CLI ของอะแดปเตอร์, Plugin ภายนอกขาดหาย หรือการดึงข้อมูล `npx` ในการเรียกใช้ครั้งแรกล้มเหลวสำหรับอะแดปเตอร์ที่ไม่ใช่ Codex | เรียกใช้ `/acp doctor`, ติดตั้ง/เตรียมอะแดปเตอร์ล่วงหน้าบนโฮสต์ Gateway หรือกำหนดค่าคำสั่งเอเจนต์ acpx อย่างชัดเจน                                                      |

| ฮาร์เนสเห็น `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>`                                      | ซองหุ้มเหตุการณ์ภายในรั่วไหลข้ามขอบเขต ACP                                                                | อัปเดต OpenClaw แล้วเรียกขั้นตอนการทำงานให้เสร็จสมบูรณ์อีกครั้ง ฮาร์เนสภายนอกควรได้รับเฉพาะพรอมต์การเสร็จสมบูรณ์แบบข้อความธรรมดา                                                          |
