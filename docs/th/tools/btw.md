`/btw` ช่วยให้คุณถามคำถามข้างเคียงแบบรวดเร็วเกี่ยวกับ **เซสชันปัจจุบัน** ได้โดยไม่
เปลี่ยนคำถามนั้นให้เป็นประวัติการสนทนาปกติ `/side` เป็นนามแฝง

พฤติกรรมนี้อิงตาม `/btw` ของ Claude Code แต่ปรับให้เข้ากับ Gateway
และสถาปัตยกรรมหลายช่องทางของ OpenClaw

## ทำอะไรได้บ้าง

เมื่อคุณส่ง:

```text
/btw what changed?
```

OpenClaw จะ:

1. snapshot บริบทของเซสชันปัจจุบัน
2. เรียกโมเดลแยกต่างหากแบบ **ไม่มีเครื่องมือ**
3. ตอบเฉพาะคำถามข้างเคียง
4. ปล่อยให้การรันหลักดำเนินไปตามเดิม
5. **ไม่** เขียนคำถามหรือคำตอบ BTW ลงในประวัติเซสชัน
6. ส่งคำตอบออกมาเป็น **ผลลัพธ์ข้างเคียงแบบสด** แทนข้อความผู้ช่วยปกติ

โมเดลทางความคิดที่สำคัญคือ:

- บริบทเซสชันเดียวกัน
- คำถามข้างเคียงแบบครั้งเดียวที่แยกต่างหาก
- ไม่มีการเรียกเครื่องมือ
- ไม่มีการปนเปื้อนบริบทในอนาคต
- ไม่มีการคงอยู่ใน transcript

## ไม่ทำอะไรบ้าง

`/btw` **ไม่**:

- สร้างเซสชันถาวรใหม่
- ดำเนินงานหลักที่ยังไม่เสร็จต่อ
- เรียกใช้เครื่องมือหรือ agent tool loops
- เขียนข้อมูลคำถาม/คำตอบ BTW ลงในประวัติ transcript
- ปรากฏใน `chat.history`
- อยู่รอดหลังรีโหลด

ตั้งใจให้เป็นแบบ **ชั่วคราว** โดยเฉพาะ

## บริบททำงานอย่างไร

BTW ใช้เซสชันปัจจุบันเป็น **บริบทพื้นหลังเท่านั้น**

หากการรันหลักกำลังทำงานอยู่ OpenClaw จะ snapshot สถานะข้อความปัจจุบัน
และรวมพรอมป์หลักที่กำลังดำเนินอยู่เป็นบริบทพื้นหลัง พร้อมทั้ง
บอกโมเดลอย่างชัดเจนว่า:

- ตอบเฉพาะคำถามข้างเคียง
- อย่ากลับไปทำต่อหรือทำงานหลักที่ยังไม่เสร็จให้เสร็จ
- อย่าส่งการเรียกเครื่องมือหรือการเรียกเครื่องมือจำลอง

สิ่งนี้ทำให้ BTW แยกออกจากการรันหลัก ในขณะที่ยังรับรู้ว่า
เซสชันนี้เกี่ยวกับอะไร

## รูปแบบการส่งมอบ

BTW **ไม่ได้** ถูกส่งเป็นข้อความ transcript ของผู้ช่วยแบบปกติ

ในระดับโปรโตคอล Gateway:

- แชตผู้ช่วยปกติใช้เหตุการณ์ `chat`
- BTW ใช้เหตุการณ์ `chat.side_result`

การแยกนี้เป็นไปโดยตั้งใจ หาก BTW ใช้เส้นทางเหตุการณ์ `chat` ปกติซ้ำ
ไคลเอนต์จะถือว่ามันเป็นประวัติการสนทนาปกติ

เพราะ BTW ใช้เหตุการณ์สดที่แยกต่างหากและไม่ได้เล่นซ้ำจาก
`chat.history` มันจึงหายไปหลังรีโหลด

## พฤติกรรมบนพื้นผิวต่างๆ

### TUI

ใน TUI, BTW จะแสดงแบบ inline ในมุมมองเซสชันปัจจุบัน แต่ยังคงเป็น
แบบชั่วคราว:

- มองเห็นได้ว่าแตกต่างจากคำตอบผู้ช่วยปกติ
- ปิดได้ด้วย `Enter` หรือ `Esc`
- ไม่ถูกเล่นซ้ำเมื่อรีโหลด

### ช่องทางภายนอก

บนช่องทางอย่าง Telegram, WhatsApp และ Discord, BTW จะถูกส่งเป็น
คำตอบครั้งเดียวที่ติดป้ายกำกับชัดเจน เพราะพื้นผิวเหล่านั้นไม่มีแนวคิด
overlay ชั่วคราวแบบ local

คำตอบยังคงถูกปฏิบัติเป็นผลลัพธ์ข้างเคียง ไม่ใช่ประวัติเซสชันปกติ

### Control UI / เว็บ

Gateway ส่ง BTW อย่างถูกต้องเป็น `chat.side_result` และ BTW ไม่ถูกรวม
อยู่ใน `chat.history` ดังนั้นสัญญาการคงอยู่จึงถูกต้องสำหรับเว็บแล้ว

Control UI ปัจจุบันยังต้องมี consumer เฉพาะสำหรับ `chat.side_result`
เพื่อแสดง BTW แบบสดในเบราว์เซอร์ จนกว่าการรองรับฝั่งไคลเอนต์นั้นจะพร้อมใช้งาน BTW จึงเป็น
ฟีเจอร์ระดับ Gateway ที่มีพฤติกรรมครบถ้วนใน TUI และช่องทางภายนอก
แต่ยังไม่ใช่ UX บนเบราว์เซอร์ที่สมบูรณ์

## ควรใช้ BTW เมื่อใด

ใช้ `/btw` เมื่อคุณต้องการ:

- คำชี้แจงแบบรวดเร็วเกี่ยวกับงานปัจจุบัน
- คำตอบเชิงข้อเท็จจริงข้างเคียงในขณะที่การรันยาวยังดำเนินอยู่
- คำตอบชั่วคราวที่ไม่ควรกลายเป็นส่วนหนึ่งของบริบทเซสชันในอนาคต

ตัวอย่าง:

```text
/btw what file are we editing?
/side what changed while the main run continued?
/btw what does this error mean?
/btw summarize the current task in one sentence
/btw what is 17 * 19?
```

## ไม่ควรใช้ BTW เมื่อใด

อย่าใช้ `/btw` เมื่อคุณต้องการให้คำตอบกลายเป็นส่วนหนึ่งของ
บริบทการทำงานในอนาคตของเซสชัน

ในกรณีนั้น ให้ถามตามปกติในเซสชันหลักแทนการใช้ BTW
