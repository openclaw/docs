Plugin ช่วยขยาย OpenClaw ด้วยช่องทาง ผู้ให้บริการโมเดล เฟรมเวิร์กควบคุมเอเจนต์ เครื่องมือ
Skills การพูด การถอดเสียงแบบเรียลไทม์ เสียง การทำความเข้าใจสื่อ การสร้างเนื้อหา
การดึงข้อมูลจากเว็บ การค้นหาเว็บ และความสามารถอื่นๆ ของรันไทม์

ใช้หน้านี้เพื่อติดตั้ง Plugin รีสตาร์ท Gateway ตรวจสอบว่ารันไทม์
โหลด Plugin แล้ว และแก้ไขความล้มเหลวทั่วไปในการตั้งค่า สำหรับตัวอย่างที่มีเฉพาะคำสั่ง โปรดดู
[จัดการ Plugin](/th/plugins/manage-plugins) สำหรับรายการที่สร้างขึ้นของ
Plugin แบบรวมมากับระบบ แบบภายนอกอย่างเป็นทางการ และแบบมีเฉพาะซอร์ส โปรดดู
[รายการ Plugin](/th/plugins/plugin-inventory)

## ข้อกำหนด

- เช็กเอาต์หรือการติดตั้ง OpenClaw ที่มี CLI `openclaw` พร้อมใช้งาน
- การเข้าถึงเครือข่ายไปยังแหล่งที่เลือก (ClawHub, npm หรือโฮสต์ git)
- ข้อมูลประจำตัว คีย์การกำหนดค่า หรือเครื่องมือระบบปฏิบัติการเฉพาะ Plugin ที่ระบุไว้ใน
  เอกสารการตั้งค่าของ Plugin นั้น
- สิทธิ์ให้ Gateway ที่ให้บริการช่องทางของคุณโหลดใหม่หรือรีสตาร์ท

## เริ่มต้นอย่างรวดเร็ว

<Steps>
  <Step title="ค้นหา Plugin">
    ค้นหาแพ็กเกจ Plugin สาธารณะใน [ClawHub](/clawhub):

    ```bash
    openclaw plugins search "calendar"
    ```

    ClawHub เป็นช่องทางหลักในการค้นหา Plugin จากชุมชน ระหว่างช่วง
    เปลี่ยนผ่านการเปิดตัว ข้อกำหนดแพ็กเกจเปล่าทั่วไปยังคงติดตั้งจาก npm เว้นแต่
    จะตรงกับ id ของ Plugin อย่างเป็นทางการ ข้อกำหนด `@openclaw/*` แบบดิบที่ตรงกับ
    Plugin แบบรวมมากับระบบจะถูกแก้ไขไปยังสำเนาที่รวมมากับระบบนั้น ใช้คำนำหน้าแหล่งที่มาอย่างชัดเจน
    เมื่อต้องการระบุแหล่งใดแหล่งหนึ่งโดยเฉพาะ

  </Step>

  <Step title="ติดตั้ง Plugin">
    ```bash
    # จาก ClawHub
    openclaw plugins install clawhub:<package>

    # จาก npm
    openclaw plugins install npm:<package>

    # จาก git
    openclaw plugins install git:github.com/<owner>/<repo>@<ref>

    # จากเช็กเอาต์สำหรับการพัฒนาในเครื่อง
    openclaw plugins install ./my-plugin
    openclaw plugins install --link ./my-plugin
    ```

    ให้ถือว่าการติดตั้ง Plugin เป็นเสมือนการเรียกใช้โค้ด ควรใช้เวอร์ชันที่ตรึงไว้เพื่อให้
    การติดตั้งในระบบใช้งานจริงทำซ้ำได้ แพ็กเกจ ClawHub และแค็ตตาล็อก
    แบบรวมมากับระบบ/อย่างเป็นทางการของ OpenClaw เป็นแหล่งที่เชื่อถือได้ แหล่ง npm, git,
    พาธ/ไฟล์เก็บถาวรภายในเครื่อง, `npm-pack:` หรือตลาดกลางที่กำหนดเองใหม่
    ต้องใช้ `--force` ในการติดตั้งแบบไม่โต้ตอบ หลังจากที่คุณ
    ตรวจสอบและเชื่อถือแหล่งนั้นแล้ว

  </Step>

  <Step title="กำหนดค่าและเปิดใช้งาน">
    กำหนดการตั้งค่าเฉพาะ Plugin ภายใต้ `plugins.entries.<id>.config`
    เปิดใช้งาน Plugin หากยังไม่ได้เปิดใช้งาน:

    ```bash
    openclaw plugins enable <plugin-id>
    ```

    หากตั้งค่า `plugins.allow` ไว้ id ของ Plugin ที่ติดตั้งต้องอยู่ในรายการนั้น
    ก่อนที่ Plugin จะโหลดได้ `openclaw plugins install` จะเพิ่ม
    id ที่ติดตั้งลงในรายการ `plugins.allow` ที่มีอยู่ และนำ id เดียวกันออกจาก
    `plugins.deny` เพื่อให้การติดตั้งที่ระบุอย่างชัดเจนโหลดได้หลังรีสตาร์ท

  </Step>

  <Step title="ให้ Gateway โหลดใหม่">
    การติดตั้ง อัปเดต หรือถอนการติดตั้งโค้ด Plugin ต้องรีสตาร์ท Gateway
    Gateway ที่ได้รับการจัดการและเปิดใช้การโหลดการกำหนดค่าใหม่จะตรวจพบระเบียน
    การติดตั้ง Plugin ที่เปลี่ยนแปลงและรีสตาร์ทโดยอัตโนมัติ มิฉะนั้นให้รีสตาร์ท
    ด้วยตนเอง:

    ```bash
    openclaw gateway restart
    ```

    การเปิด/ปิดใช้งานจะอัปเดตการกำหนดค่าและรีจิสทรีแบบเย็น การตรวจสอบรันไทม์
    ยังคงเป็นหลักฐานที่ชัดเจนที่สุดของพื้นผิวรันไทม์ที่ทำงานอยู่

  </Step>

  <Step title="ตรวจสอบการลงทะเบียนรันไทม์">
    ```bash
    openclaw plugins inspect <plugin-id> --runtime --json
    ```

    ใช้ `--runtime` เพื่อยืนยันเครื่องมือ ฮุก บริการ เมธอดของ Gateway
    หรือคำสั่ง CLI ที่ Plugin เป็นเจ้าของและได้ลงทะเบียนไว้ ส่วน `inspect` แบบธรรมดาเป็นเพียง
    การตรวจสอบไฟล์กำกับและรีจิสทรีแบบเย็น

  </Step>
</Steps>

## การกำหนดค่า

### เลือกแหล่งติดตั้ง

| แหล่งที่มา   | ใช้เมื่อ                                                                         | ตัวอย่าง                                                       |
| ----------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| ClawHub     | ต้องการการค้นหาแบบเฉพาะ OpenClaw การสแกน ข้อมูลเมตาเวอร์ชัน และคำแนะนำการติดตั้ง | `openclaw plugins install clawhub:<package>`                   |
| npm         | ต้องใช้รีจิสทรี npm โดยตรงหรือเวิร์กโฟลว์ dist-tag                              | `openclaw plugins install npm:<package>`                       |
| git         | ต้องใช้แบรนช์ แท็ก หรือคอมมิตจากที่เก็บ                                         | `openclaw plugins install git:github.com/<owner>/<repo>@<ref>` |
| พาธภายในเครื่อง | กำลังพัฒนาหรือทดสอบ Plugin บนเครื่องเดียวกัน                                  | `openclaw plugins install --link ./my-plugin`                  |
| ตลาดกลาง    | กำลังติดตั้ง Plugin จากตลาดกลางที่เข้ากันได้กับ Claude                           | `openclaw plugins install <plugin> --marketplace <source>`     |

ข้อกำหนดแพ็กเกจเปล่ามีพฤติกรรมความเข้ากันได้พิเศษ: ชื่อเปล่าที่
ตรงกับ id ของ Plugin แบบรวมมากับระบบจะใช้แหล่งที่รวมมากับระบบนั้น ชื่อเปล่าที่ตรงกับ
id ของ Plugin ภายนอกอย่างเป็นทางการจะใช้แค็ตตาล็อกแพ็กเกจอย่างเป็นทางการ ส่วนข้อกำหนดเปล่าอื่นๆ
จะติดตั้งผ่าน npm ระหว่างช่วงเปลี่ยนผ่านการเปิดตัว ข้อกำหนด `@openclaw/*` แบบดิบ
ที่ตรงกับ Plugin แบบรวมมากับระบบจะถูกแก้ไขไปยังสำเนาที่รวมมากับระบบก่อนใช้
npm เป็นทางเลือกสำรอง ใช้ `npm:@openclaw/<plugin>@<version>` เพื่อติดตั้ง
แพ็กเกจ npm ภายนอกแทนสำเนาที่รวมมากับระบบโดยเจตนา ใช้ `clawhub:`, `npm:`,
`git:` หรือ `npm-pack:` เพื่อเลือกแหล่งที่มาแบบกำหนดแน่นอน โปรดดู
[`openclaw plugins`](/th/cli/plugins#install) สำหรับสัญญาคำสั่งฉบับเต็ม

สำหรับการติดตั้งจาก npm ข้อกำหนดที่ไม่ได้ตรึงและ `@latest` จะเลือกแพ็กเกจ
เสถียรล่าสุดที่ประกาศความเข้ากันได้กับบิลด์ OpenClaw นี้ หากรุ่นล่าสุดปัจจุบันของ npm
ประกาศ `openclaw.compat.pluginApi` หรือ `openclaw.install.minHostVersion` ที่ใหม่กว่า
ที่บิลด์นี้รองรับ OpenClaw จะสแกนเวอร์ชันเสถียรรุ่นเก่ากว่า
และติดตั้งเวอร์ชันใหม่ที่สุดที่เข้ากันได้ เวอร์ชันที่ระบุแน่นอน
และแท็กช่องทางที่ระบุชัดเจน เช่น `@beta` จะยังคงตรึงกับแพ็กเกจที่เลือก
และล้มเหลวเมื่อเข้ากันไม่ได้

### นโยบายการติดตั้งของผู้ดำเนินการ

กำหนดค่า `security.installPolicy` เพื่อเรียกใช้คำสั่งนโยบายภายในเครื่องที่เชื่อถือได้
ก่อนดำเนินการติดตั้งหรืออัปเดต Plugin นโยบายจะได้รับข้อมูลเมตาพร้อม
พาธแหล่งที่มาที่จัดเตรียมไว้ และสามารถอนุญาตหรือบล็อกการติดตั้ง ครอบคลุมทั้งพาธ
การติดตั้ง/อัปเดตผ่าน CLI และ Gateway ฮุก `before_install` ของ Plugin จะทำงาน
ภายหลัง และเฉพาะในกระบวนการ OpenClaw ที่โหลดฮุกของ Plugin เท่านั้น ดังนั้นให้ใช้
`security.installPolicy` สำหรับการตัดสินใจติดตั้งที่ผู้ดำเนินการเป็นเจ้าของแทน แฟล็ก
`--dangerously-force-unsafe-install` ที่เลิกใช้แล้วได้รับการยอมรับเพื่อ
ความเข้ากันได้ แต่ไม่มีผลใดๆ กล่าวคือไม่ข้ามนโยบายการติดตั้งหรือรายการปฏิเสธ
การพึ่งพาของ Plugin ที่มีใน OpenClaw

โปรดดู [การกำหนดค่า Skills](/th/tools/skills-config#operator-install-policy-securityinstallpolicy)
สำหรับสคีมา exec `security.installPolicy` ที่ใช้ร่วมกันทั้ง Skills และ

### กำหนดนโยบาย Plugin

รูปแบบการกำหนดค่าทั่วไปของ Plugin คือ:

```json5
{
  plugins: {
    enabled: true,
    allow: ["voice-call"],
    deny: ["untrusted-plugin"],
    load: { paths: ["~/Projects/oss/voice-call-plugin"] },
    slots: { memory: "memory-core" },
    entries: {
      "voice-call": { enabled: true, config: { provider: "twilio" } },
    },
  },
}
```

กฎนโยบายหลัก:

- `plugins.enabled: false` ปิดใช้งาน Plugin ทั้งหมดและข้ามงานค้นหา/โหลด
  การอ้างอิง Plugin ที่ล้าสมัยจะยังคงไม่มีผลขณะเปิดใช้งานค่านี้ ให้เปิดใช้งาน
  Plugin อีกครั้งก่อนเรียกใช้การล้างข้อมูลของ doctor หากต้องการลบ id ที่ล้าสมัย
- `plugins.deny` มีลำดับความสำคัญเหนือรายการอนุญาตและการเปิดใช้งานราย Plugin
- `plugins.allow` เป็นรายการอนุญาตแบบเฉพาะเท่านั้น เครื่องมือที่ Plugin เป็นเจ้าของซึ่งอยู่นอก
  รายการอนุญาตจะยังคงใช้ไม่ได้ แม้ว่า `tools.allow` จะมี `"*"`
- `plugins.entries.<id>.enabled: false` ปิดใช้งาน Plugin หนึ่งรายการโดยยังคง
  การกำหนดค่าไว้
- `plugins.load.paths` เพิ่มไฟล์หรือไดเรกทอรี Plugin ภายในเครื่องอย่างชัดเจน
  พาธภายในเครื่อง `plugins install` ที่ได้รับการจัดการต้องเป็นไดเรกทอรีหรือ
  ไฟล์เก็บถาวรของ Plugin ใช้ `plugins.load.paths` สำหรับไฟล์ Plugin แบบเดี่ยว
- Plugin ที่มาจากพื้นที่ทำงานจะปิดใช้งานโดยค่าเริ่มต้น ให้เปิดใช้งานอย่างชัดเจนหรือ
  เพิ่มลงในรายการอนุญาตก่อนใช้โค้ดพื้นที่ทำงานภายในเครื่อง
- Plugin แบบรวมมากับระบบจะทำตามข้อมูลเมตาเริ่มต้นแบบเปิด/ปิดที่มีอยู่ในตัว
  เว้นแต่การกำหนดค่าจะแทนที่อย่างชัดเจน
- `plugins.slots.<slot>` (`memory` หรือ `contextEngine`) เลือก Plugin หนึ่งรายการสำหรับ
  หมวดหมู่แบบเฉพาะ การเลือกสล็อตนับเป็นการเปิดใช้งานอย่างชัดเจนและ
  บังคับเปิดใช้งาน Plugin ที่เลือกสำหรับสล็อตนั้น แม้ว่าในกรณีอื่น
  จะต้องเลือกเข้าร่วมก็ตาม `plugins.deny` และ `plugins.entries.<id>.enabled: false` ยังคง
  บล็อก Plugin นั้น
- Plugin แบบรวมมากับระบบที่ต้องเลือกเข้าร่วมสามารถเปิดใช้งานโดยอัตโนมัติเมื่อการกำหนดค่าระบุหนึ่งใน
  พื้นผิวที่ Plugin เป็นเจ้าของ เช่น การอ้างอิงผู้ให้บริการ/โมเดล การกำหนดค่าช่องทาง แบ็กเอนด์ CLI
  หรือรันไทม์เฟรมเวิร์กควบคุมเอเจนต์
- การกำหนดเส้นทาง Codex ในตระกูล OpenAI จะแยกขอบเขตของผู้ให้บริการและ Plugin รันไทม์
  ออกจากกัน: การอ้างอิงโมเดล Codex แบบเดิมเป็นการกำหนดค่าเดิมที่ doctor ซ่อมแซม
  ส่วน Plugin `codex` แบบรวมมากับระบบเป็นเจ้าของรันไทม์แอปเซิร์ฟเวอร์ Codex สำหรับ
  การอ้างอิงเอเจนต์ `openai/*` แบบมาตรฐาน, `agentRuntime.id: "codex"` ที่ระบุชัดเจน และ
  การอ้างอิง `codex/*` แบบเดิม

เมื่อไม่ได้ตั้งค่า `plugins.allow` และมีการค้นพบ Plugin ที่ไม่ได้รวมมากับระบบโดยอัตโนมัติจาก
พื้นที่ทำงานหรือรูท Plugin ส่วนกลาง บันทึกการเริ่มต้นจะแสดง
`plugins.allow is empty; discovered non-bundled plugins may auto-load: ...`
พร้อม id ของ Plugin ที่ค้นพบ และสำหรับรายการสั้นๆ จะแสดงตัวอย่าง `plugins.allow`
แบบขั้นต่ำ เรียกใช้ [`openclaw plugins list --enabled --verbose`](/th/cli/plugins#list)
หรือ [`openclaw plugins inspect <id>`](/th/cli/plugins#inspect) กับ id ของ
Plugin ที่แสดงไว้ ก่อนคัดลอก Plugin ที่เชื่อถือได้ไปยัง `openclaw.json` การตรึง
ความเชื่อถือลักษณะเดียวกันนี้ใช้เมื่อการวินิจฉัยแจ้งว่า Plugin โหลด
`without install/load-path provenance`: ให้ตรวจสอบ id ของ Plugin นั้น จากนั้นตรึงไว้ใน
`plugins.allow` หรือติดตั้งใหม่จากแหล่งที่เชื่อถือได้ เพื่อให้ OpenClaw บันทึก
ที่มาของการติดตั้ง

เรียกใช้ `openclaw doctor` หรือ `openclaw doctor --fix` เมื่อการตรวจสอบ
การกำหนดค่ารายงาน id ของ Plugin ที่ล้าสมัย รายการอนุญาต/เครื่องมือไม่ตรงกัน หรือพาธ
Plugin แบบรวมมากับระบบรุ่นเดิม

## ทำความเข้าใจรูปแบบ Plugin

OpenClaw รองรับ Plugin สองรูปแบบ:

| รูปแบบ                  | วิธีโหลด                                                                      | ใช้เมื่อ                                                                  |
| ---------------------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Plugin OpenClaw แบบเนทีฟ | `openclaw.plugin.json` พร้อมโมดูลรันไทม์ที่โหลดภายในกระบวนการ                     | กำลังติดตั้งหรือสร้างความสามารถรันไทม์เฉพาะ OpenClaw                     |
| บันเดิลที่เข้ากันได้      | เลย์เอาต์ Plugin ของ Codex, Claude หรือ Cursor ที่แมปเข้าในรายการ Plugin ของ OpenClaw | กำลังนำ Skills คำสั่ง ฮุก หรือข้อมูลเมตาของบันเดิลที่เข้ากันได้มาใช้ซ้ำ |

ทั้งสองรูปแบบปรากฏใน `openclaw plugins list`, `openclaw plugins inspect`,
`openclaw plugins enable` และ `openclaw plugins disable` โปรดดู
[บันเดิล Plugin](/th/plugins/bundles) สำหรับขอบเขตความเข้ากันได้ของบันเดิล และ
[การสร้าง Plugin](/th/plugins/building-plugins) สำหรับการสร้าง Plugin แบบเนทีฟ

## ฮุกของ Plugin

Plugin สามารถลงทะเบียนฮุกในรันไทม์ผ่าน API สองแบบ:

- `api.on(...)` ฮุกแบบมีชนิดสำหรับเหตุการณ์วงจรชีวิตของรันไทม์ นี่คือ
  พื้นผิวที่แนะนำสำหรับมิดเดิลแวร์ นโยบาย การเขียนข้อความใหม่ การปรับรูปแบบพรอมต์
  และการควบคุมเครื่องมือ
- `api.registerHook(...)` สำหรับระบบฮุกภายในที่อธิบายไว้ใน
  [ฮุก](/th/automation/hooks) ใช้เป็นหลักสำหรับผลข้างเคียงแบบหยาบของคำสั่ง/วงจรชีวิต
  และความเข้ากันได้กับระบบอัตโนมัติรูปแบบ HOOK ที่มีอยู่

กฎโดยย่อ: หากตัวจัดการต้องใช้ลำดับความสำคัญ ความหมายของการผสาน หรือ
พฤติกรรมบล็อก/ยกเลิก ให้ใช้ฮุกแบบมีชนิด หากเพียงตอบสนองต่อ `command:new`,
`command:reset`, `message:sent` หรือเหตุการณ์แบบหยาบที่คล้ายกัน `api.registerHook`
ก็เพียงพอ

ฮุกภายในที่ Plugin จัดการจะแสดงใน `openclaw hooks list` พร้อม
`plugin:<id>` คุณไม่สามารถเปิดหรือปิดใช้งานฮุกเหล่านี้ผ่าน `openclaw hooks`;
ให้เปิดหรือปิดใช้งาน Plugin แทน

## ตรวจสอบ Gateway ที่ใช้งานอยู่

`openclaw plugins list` และ `openclaw plugins inspect` แบบธรรมดาจะอ่านสถานะการกำหนดค่าแบบเย็น
manifest และรีจิสทรี ซึ่งไม่ได้พิสูจน์ว่า Gateway ที่กำลังทำงานอยู่
ได้นำเข้าโค้ด Plugin เดียวกันแล้ว

เมื่อ Plugin แสดงว่าติดตั้งแล้ว แต่ทราฟฟิกแชตสดไม่ได้ใช้งาน:

```bash
openclaw gateway status --deep --require-rpc
openclaw plugins inspect <plugin-id> --runtime --json
openclaw gateway restart
```

Gateway ที่มีการจัดการจะรีสตาร์ตโดยอัตโนมัติหลังการติดตั้ง อัปเดต และ
ถอนการติดตั้ง Plugin ซึ่งเปลี่ยนแปลงซอร์สของ Plugin สำหรับการติดตั้งบน VPS หรือคอนเทนเนอร์
ตรวจสอบให้แน่ใจว่าการรีสตาร์ตด้วยตนเองกำหนดเป้าหมายไปยังโปรเซสลูก `openclaw gateway run` จริง
ที่ให้บริการช่องทางของคุณ ไม่ใช่เพียงตัวห่อหุ้มหรือซูเปอร์ไวเซอร์

## การแก้ไขปัญหา

| อาการ                                                        | การตรวจสอบ                                                                                                                                      | วิธีแก้ไข                                                                                                     |
| -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------- |
| Plugin ปรากฏใน `plugins list` แต่ฮุกขณะรันไทม์ไม่ทำงาน  | ใช้ `openclaw plugins inspect <id> --runtime --json` และยืนยัน Gateway ที่ใช้งานอยู่ด้วย `gateway status --deep --require-rpc`             | รีสตาร์ต Gateway ที่ทำงานอยู่หลังการติดตั้ง อัปเดต เปลี่ยนการกำหนดค่า หรือเปลี่ยนซอร์ส                               |
| มีข้อมูลวินิจฉัยเกี่ยวกับการเป็นเจ้าของช่องทางหรือเครื่องมือซ้ำ         | เรียกใช้ `openclaw plugins list --enabled --verbose` ตรวจสอบ Plugin แต่ละรายการที่สงสัยด้วย `--runtime --json` และเปรียบเทียบความเป็นเจ้าของช่องทาง/เครื่องมือ | ปิดใช้งานเจ้าของรายหนึ่ง ลบการติดตั้งที่ล้าสมัย หรือใช้ `preferOver` ใน manifest สำหรับการแทนที่โดยเจตนา      |
| การกำหนดค่าระบุว่าไม่มี Plugin                                | ตรวจสอบ [รายการ Plugin](/th/plugins/plugin-inventory) ว่าเป็นรายการที่รวมมาให้แล้ว รายการภายนอกอย่างเป็นทางการ หรือมีเฉพาะซอร์ส                           | ติดตั้งแพ็กเกจภายนอก เปิดใช้งาน Plugin ที่รวมมาให้แล้ว หรือลบการกำหนดค่าที่ล้าสมัย                         |
| การกำหนดค่าไม่ถูกต้องระหว่างการติดตั้ง                               | อ่านข้อความตรวจสอบความถูกต้อง และเรียกใช้ `openclaw doctor --fix` หากข้อความชี้ไปยังสถานะ Plugin ที่ล้าสมัย                                             | Doctor สามารถกักกันการกำหนดค่า Plugin ที่ไม่ถูกต้องได้ โดยปิดใช้งานรายการและลบเพย์โหลดที่ไม่ถูกต้อง     |
| พาธ Plugin ถูกบล็อกเนื่องจากความเป็นเจ้าของหรือสิทธิ์ที่น่าสงสัย | ตรวจสอบข้อมูลวินิจฉัยก่อนข้อผิดพลาดการกำหนดค่า                                                                                             | แก้ไขความเป็นเจ้าของ/สิทธิ์ของระบบไฟล์ แล้วเรียกใช้ `openclaw plugins registry --refresh`                    |
| `OPENCLAW_NIX_MODE=1` บล็อกคำสั่งวงจรชีวิต                | ยืนยันว่าการติดตั้งได้รับการจัดการโดย Nix                                                                                                      | เปลี่ยนการเลือก Plugin ในซอร์ส Nix แทนการใช้คำสั่งแก้ไข Plugin                      |
| การนำเข้าดีเพนเดนซีล้มเหลวขณะรันไทม์                             | ตรวจสอบว่า Plugin ติดตั้งผ่าน npm/git/ClawHub หรือโหลดจากพาธภายในเครื่อง                                                 | เรียกใช้ `openclaw plugins update <id>` ติดตั้งซอร์สใหม่ หรือติดตั้งดีเพนเดนซีของ Plugin ภายในเครื่องด้วยตนเอง |

เมื่อการกำหนดค่า Plugin ที่ล้าสมัยยังคงระบุ Plugin ช่องทางที่ไม่สามารถค้นพบได้อีก
การตรวจสอบความถูกต้องของการกำหนดค่าจะลดระดับคีย์ช่องทางนั้นเป็นคำเตือนแทนความล้มเหลวร้ายแรง
เพื่อให้การเริ่มต้น Gateway ยังสามารถให้บริการช่องทางอื่นทั้งหมดได้ เรียกใช้
`openclaw doctor --fix` เพื่อลบรายการ Plugin และช่องทางที่ล้าสมัย คีย์
ช่องทางที่ไม่รู้จักและไม่มีหลักฐานว่าเกิดจาก Plugin ที่ล้าสมัยจะยังคงทำให้การตรวจสอบความถูกต้องล้มเหลว เพื่อให้การพิมพ์ผิด
ยังคงมองเห็นได้

สำหรับการแทนที่ช่องทางโดยเจตนา Plugin ที่ต้องการควรประกาศ
`channelConfigs.<channel-id>.preferOver` ด้วยรหัส Plugin เดิมหรือที่มีลำดับความสำคัญต่ำกว่า
หากเปิดใช้งาน Plugin ทั้งสองอย่างชัดเจน OpenClaw จะคงคำขอนั้นไว้
และรายงานข้อมูลวินิจฉัยช่องทาง/เครื่องมือซ้ำ แทนที่จะเลือก
เจ้าของรายหนึ่งโดยไม่แจ้งให้ทราบ

หากแพ็กเกจที่ติดตั้งรายงานว่า `requires compiled runtime output for
TypeScript entry ...` แสดงว่าแพ็กเกจถูกเผยแพร่โดยไม่มีไฟล์ JavaScript
ที่ OpenClaw ต้องใช้ขณะรันไทม์ ให้อัปเดตหรือติดตั้งใหม่หลังจากผู้เผยแพร่จัดส่ง
JavaScript ที่คอมไพล์แล้ว หรือปิดใช้งาน/ถอนการติดตั้ง Plugin จนกว่าจะพร้อม

### ความเป็นเจ้าของพาธ Plugin ที่ถูกบล็อก

หากข้อมูลวินิจฉัยระบุว่า
`blocked plugin candidate: suspicious ownership (... uid=1000, expected uid=0 or root)`
และตามด้วยการตรวจสอบความถูกต้องที่ระบุว่า `plugin present but blocked` แสดงว่า OpenClaw พบ
ไฟล์ Plugin ที่มีผู้ใช้ Unix ซึ่งไม่ใช่ผู้ใช้เดียวกับโปรเซสที่โหลดไฟล์นั้นเป็นเจ้าของ
คงการกำหนดค่า Plugin ไว้ แก้ไขความเป็นเจ้าของของระบบไฟล์ หรือเรียกใช้ OpenClaw
ด้วยผู้ใช้เดียวกับที่เป็นเจ้าของไดเรกทอรีสถานะ

สำหรับการติดตั้ง Docker อิมเมจอย่างเป็นทางการจะทำงานเป็น `node` (uid `1000`) ดังนั้น
ไดเรกทอรีการกำหนดค่าและพื้นที่ทำงานของ OpenClaw ที่ผูกเมานต์จากโฮสต์ โดยปกติควรมี
uid `1000` เป็นเจ้าของ:

```bash
sudo chown -R 1000:1000 /path/to/openclaw-config /path/to/openclaw-workspace
```

หากคุณตั้งใจเรียกใช้ OpenClaw เป็น root ให้ซ่อมแซมรูทของ Plugin ที่มีการจัดการให้
root เป็นเจ้าของแทน:

```bash
sudo chown -R root:root /path/to/openclaw-config/npm
```

หลังจากแก้ไขความเป็นเจ้าของแล้ว ให้เรียกใช้ `openclaw doctor --fix` หรือ
`openclaw plugins registry --refresh` อีกครั้ง เพื่อให้รีจิสทรี Plugin ที่บันทึกถาวร
ตรงกับไฟล์ที่ซ่อมแซมแล้ว

### การตั้งค่าเครื่องมือ Plugin ที่ช้า

หากรอบการทำงานของเอเจนต์ดูเหมือนหยุดค้างขณะเตรียมเครื่องมือ ให้เปิดใช้การบันทึกล็อกระดับ trace
และตรวจสอบบรรทัดการจับเวลาของแฟกทอรีเครื่องมือ Plugin:

```bash
openclaw config set logging.level trace
openclaw logs --follow
```

มองหา:

```text
[trace:plugin-tools] factory timings ...
```

ข้อมูลสรุปจะแสดงเวลารวมของแฟกทอรีและแฟกทอรีเครื่องมือ Plugin ที่ช้าที่สุด
รวมถึงรหัส Plugin ชื่อเครื่องมือที่ประกาศ รูปแบบผลลัพธ์ และระบุว่าเครื่องมือนั้น
เป็นตัวเลือกหรือไม่ บรรทัดที่ช้าจะถูกยกระดับเป็นคำเตือนเมื่อแฟกทอรีรายการเดียวใช้เวลา
อย่างน้อย 1s หรือการเตรียมแฟกทอรีเครื่องมือ Plugin ทั้งหมดใช้เวลาอย่างน้อย 5s

OpenClaw แคชผลลัพธ์แฟกทอรีเครื่องมือ Plugin ที่สำเร็จสำหรับการแก้ไขซ้ำ
ด้วยบริบทคำขอที่มีผลเหมือนกัน คีย์แคชประกอบด้วย
การกำหนดค่ารันไทม์ที่มีผล พื้นที่ทำงานและรหัสเอเจนต์ นโยบาย sandbox การตั้งค่าเบราว์เซอร์
บริบทการส่งมอบ ตัวตนของผู้ร้องขอ และสถานะความเป็นเจ้าของ ดังนั้น
แฟกทอรีที่ขึ้นอยู่กับฟิลด์ที่เชื่อถือได้เหล่านั้นจะทำงานใหม่เมื่อบริบท
เปลี่ยนแปลง หากเวลายังคงสูง Plugin อาจกำลังทำงานที่ใช้ทรัพยากรมากก่อน
ส่งคืนคำจำกัดความเครื่องมือ

หาก Plugin หนึ่งใช้เวลาส่วนใหญ่ ให้ตรวจสอบการลงทะเบียนขณะรันไทม์:

```bash
openclaw plugins inspect <plugin-id> --runtime --json
```

จากนั้นอัปเดต ติดตั้งใหม่ หรือปิดใช้งาน Plugin นั้น ผู้เขียน Plugin ควรย้าย
การโหลดดีเพนเดนซีที่ใช้ทรัพยากรมากไปไว้หลังเส้นทางการเรียกใช้เครื่องมือ แทนการทำ
ภายในแฟกทอรีเครื่องมือ

สำหรับรูทของดีเพนเดนซี การตรวจสอบความถูกต้องของเมทาดาทาแพ็กเกจ ระเบียนรีจิสทรี พฤติกรรม
การโหลดใหม่เมื่อเริ่มต้น และการล้างข้อมูลแบบเดิม โปรดดู
[การแก้ไขดีเพนเดนซีของ Plugin](/th/plugins/dependency-resolution)
