การค้นหาเครื่องมือช่วยให้เอเจนต์ PI มีวิธีแบบกะทัดรัดหนึ่งวิธีในการค้นพบและเรียกใช้แค็ตตาล็อกเครื่องมือขนาดใหญ่ มีประโยชน์เมื่อการรันมีเครื่องมือที่พร้อมใช้งานจำนวนมาก แต่โมเดลน่าจะต้องใช้เพียงไม่กี่รายการเท่านั้น

เมื่อเปิดใช้งานสำหรับ PI โมเดลจะได้รับเครื่องมือ `tool_search_code` หนึ่งรายการเป็นค่าเริ่มต้น เครื่องมือนั้นรันเนื้อหา JavaScript สั้น ๆ ใน subprocess ของ Node ที่แยกออกมา พร้อมบริดจ์ `openclaw.tools`:

```js
const hits = await openclaw.tools.search("create a GitHub issue");
const tool = await openclaw.tools.describe(hits[0].id);
return await openclaw.tools.call(tool.id, {
  title: "Crash on startup",
  body: "Steps to reproduce...",
});
```

แค็ตตาล็อกสามารถรวมเครื่องมือของ OpenClaw เครื่องมือของ Plugin เครื่องมือ MCP และเครื่องมือที่ไคลเอนต์จัดหาให้ โมเดลจะไม่เห็น schema แบบเต็มทั้งหมดตั้งแต่ต้น แต่จะค้นหาตัวอธิบายแบบกะทัดรัด อธิบายเครื่องมือที่เลือกหนึ่งรายการเมื่อต้องการ schema ที่แน่นอน และเรียกใช้เครื่องมือนั้นผ่าน OpenClaw

การรันด้วย Codex harness จะไม่ได้รับตัวควบคุมการค้นหาเครื่องมือของ OpenClaw เหล่านี้ OpenClaw ส่งความสามารถของผลิตภัณฑ์ให้ Codex เป็นเครื่องมือแบบไดนามิก และ Codex เป็นเจ้าของโหมดโค้ดแบบเนทีฟ การค้นหาเครื่องมือแบบเนทีฟ เครื่องมือแบบไดนามิกที่เลื่อนการใช้งาน และการเรียกเครื่องมือแบบซ้อนกัน

## วิธีที่หนึ่ง turn ทำงาน

ในช่วงวางแผน ตัวรัน PI แบบฝังจะสร้างแค็ตตาล็อกที่มีผลสำหรับการรัน:

1. แก้ไขนโยบายเครื่องมือที่ใช้งานอยู่สำหรับเอเจนต์ โปรไฟล์ sandbox และ session
2. แสดงรายการเครื่องมือ OpenClaw และ Plugin ที่เข้าเกณฑ์
3. แสดงรายการเครื่องมือ MCP ที่เข้าเกณฑ์ผ่าน runtime MCP ของ session
4. เพิ่มเครื่องมือไคลเอนต์ที่เข้าเกณฑ์ซึ่งจัดหาให้สำหรับการรันปัจจุบัน
5. ทำดัชนีตัวอธิบายแบบกะทัดรัดสำหรับการค้นหา
6. เปิดเผยบริดจ์โค้ด PI หรือเครื่องมือ fallback แบบมีโครงสร้างให้โมเดล

ในช่วงดำเนินการ การเรียกเครื่องมือจริงทุกครั้งจะกลับไปที่ OpenClaw runtime ของ Node ที่แยกออกมาไม่ได้ถือ implementation ของ Plugin, อ็อบเจ็กต์ไคลเอนต์ MCP หรือความลับ `openclaw.tools.call(...)` ข้ามบริดจ์กลับเข้าสู่ Gateway ซึ่งยังคงใช้นโยบาย การอนุมัติ hook การบันทึก log และการจัดการผลลัพธ์ตามปกติ

## โหมด

`tools.toolSearch` มีสองโหมดที่โมเดลมองเห็น:

- `code`: เปิดเผย `tool_search_code` ซึ่งเป็นบริดจ์ JavaScript แบบกะทัดรัดตามค่าเริ่มต้น
- `tools`: เปิดเผย `tool_search`, `tool_describe` และ `tool_call` เป็นเครื่องมือแบบมีโครงสร้างธรรมดาสำหรับผู้ให้บริการที่ไม่ควรได้รับโค้ด

ทั้งสองโหมดใช้แค็ตตาล็อกและเส้นทางการดำเนินการเดียวกัน ความแตกต่างเพียงอย่างเดียวคือรูปแบบที่โมเดลเห็น หาก runtime ปัจจุบันไม่สามารถเปิด subprocess โหมดโค้ดของ Node ที่แยกออกมาได้ โหมด `code` ตามค่าเริ่มต้นจะ fallback ไปที่ `tools` ก่อนการย่อแค็ตตาล็อก

ไม่มี config สำหรับการเลือกแหล่งที่มาแยกต่างหาก เมื่อเปิดใช้งานการค้นหาเครื่องมือ แค็ตตาล็อกจะรวมเครื่องมือ OpenClaw, MCP และไคลเอนต์ที่เข้าเกณฑ์หลังผ่านการกรองนโยบายปกติ

## เหตุผลที่มีสิ่งนี้

แค็ตตาล็อกขนาดใหญ่มีประโยชน์แต่มีต้นทุนสูง การส่ง schema ของเครื่องมือทุกตัวไปยังโมเดลทำให้ request ใหญ่ขึ้น ทำให้การวางแผนช้าลง และเพิ่มโอกาสเลือกเครื่องมือโดยไม่ตั้งใจ

การค้นหาเครื่องมือเปลี่ยนรูปแบบดังนี้:

- เครื่องมือโดยตรง: โมเดลเห็น schema ที่เลือกทั้งหมดก่อน token แรก
- โหมดโค้ดของการค้นหาเครื่องมือ: โมเดลเห็นเครื่องมือโค้ดแบบกะทัดรัดหนึ่งรายการและสัญญา API สั้น ๆ
- โหมดเครื่องมือของการค้นหาเครื่องมือ: โมเดลเห็นเครื่องมือ fallback แบบมีโครงสร้างขนาดกะทัดรัดสามรายการ
- ระหว่าง turn: โมเดลโหลดเฉพาะ schema ของเครื่องมือที่จำเป็นจริง ๆ

การเปิดเผยเครื่องมือโดยตรงยังคงเป็นค่าเริ่มต้นที่เหมาะสมสำหรับแค็ตตาล็อกขนาดเล็ก การค้นหาเครื่องมือเหมาะที่สุดเมื่อการรันหนึ่งครั้งสามารถเห็นเครื่องมือจำนวนมาก โดยเฉพาะจากเซิร์ฟเวอร์ MCP หรือเครื่องมือแอปที่ไคลเอนต์จัดหาให้

## API

`openclaw.tools.search(query, options?)`

ค้นหาแค็ตตาล็อกที่มีผลสำหรับการรันปัจจุบัน ผลลัพธ์มีขนาดกะทัดรัดและปลอดภัยสำหรับใส่กลับเข้าไปใน context ของ prompt

```js
const hits = await openclaw.tools.search("calendar event", { limit: 5 });
```

`openclaw.tools.describe(id)`

โหลด metadata แบบเต็มสำหรับผลการค้นหาหนึ่งรายการ รวมถึง input schema ที่แน่นอน

```js
const calendarCreate = await openclaw.tools.describe("mcp:calendar:create_event");
```

`openclaw.tools.call(id, args)`

เรียกเครื่องมือที่เลือกผ่าน OpenClaw

```js
await openclaw.tools.call(calendarCreate.id, {
  summary: "Planning",
  start: "2026-05-09T14:00:00Z",
});
```

โหมด fallback แบบมีโครงสร้างเปิดเผยการดำเนินการเดียวกันเป็นเครื่องมือ:

- `tool_search`
- `tool_describe`
- `tool_call`

## ขอบเขต runtime

บริดจ์โค้ดรันใน subprocess ของ Node อายุสั้น subprocess เริ่มต้นโดยเปิดใช้งานโหมด permission ของ Node มี environment ว่างเปล่า ไม่มีสิทธิ์ filesystem หรือ network และไม่มีสิทธิ์ child-process หรือ worker OpenClaw บังคับใช้ timeout แบบ wall-clock ใน parent process และ kill subprocess เมื่อ timeout รวมถึงหลัง async continuations

runtime เปิดเผยเฉพาะ:

- `console.log`, `console.warn` และ `console.error`
- `openclaw.tools.search`
- `openclaw.tools.describe`
- `openclaw.tools.call`

พฤติกรรมปกติของ OpenClaw ยังคงใช้กับการเรียกสุดท้าย:

- นโยบายอนุญาตและปฏิเสธเครื่องมือ
- ข้อจำกัดเครื่องมือต่อเอเจนต์และต่อ sandbox
- การกั้นสำหรับเจ้าของเท่านั้น
- hook การอนุมัติ
- hook `before_tool_call` ของ Plugin
- ข้อมูลระบุตัวตนของ session, log และ telemetry

## Config

เปิดใช้งานการค้นหาเครื่องมือสำหรับการรัน PI ด้วยบริดจ์โค้ดตามค่าเริ่มต้น:

```bash
openclaw config set tools.toolSearch true
```

JSON ที่เทียบเท่า:

```json5
{
  tools: {
    toolSearch: true,
  },
}
```

ใช้เครื่องมือ fallback แบบมีโครงสร้างแทนสำหรับการรัน PI:

```json5
{
  tools: {
    toolSearch: {
      mode: "tools",
    },
  },
}
```

ปรับ timeout ของโหมดโค้ดและขีดจำกัดผลการค้นหา:

```json5
{
  tools: {
    toolSearch: {
      mode: "code",
      codeTimeoutMs: 10000,
      searchDefaultLimit: 8,
      maxSearchLimit: 20,
    },
  },
}
```

ปิดใช้งาน:

```json5
{
  tools: {
    toolSearch: false,
  },
}
```

## Prompt และ telemetry

การค้นหาเครื่องมือบันทึก telemetry เพียงพอสำหรับเปรียบเทียบกับการเปิดเผยเครื่องมือโดยตรง:

- จำนวนไบต์ทั้งหมดของเครื่องมือและ prompt ที่ serialize แล้วซึ่งส่งไปยัง harness
- ขนาดแค็ตตาล็อกและการแจกแจงตามแหล่งที่มา
- จำนวนครั้งของ search, describe และ call
- การเรียกเครื่องมือสุดท้ายที่ดำเนินการผ่าน OpenClaw
- id และแหล่งที่มาของเครื่องมือที่เลือก

log ของ session ควรทำให้ตอบได้ว่า:

- โมเดลเห็น schema ของเครื่องมือกี่รายการตั้งแต่ต้น
- โมเดลดำเนินการ search และ describe กี่ครั้ง
- เครื่องมือสุดท้ายใดถูกเรียก
- ผลลัพธ์มาจาก OpenClaw, MCP หรือเครื่องมือไคลเอนต์

## การตรวจสอบ E2E

ตัวรัน E2E ของ Gateway พิสูจน์ทั้งสองเส้นทางด้วย PI harness:

```bash
node --import tsx scripts/tool-search-gateway-e2e.ts
```

มันสร้าง Plugin ปลอมชั่วคราวพร้อมแค็ตตาล็อกเครื่องมือขนาดใหญ่ เริ่มผู้ให้บริการ OpenAI จำลอง เริ่ม Gateway หนึ่งครั้งในโหมดตรงและอีกหนึ่งครั้งโดยเปิดใช้งานการค้นหาเครื่องมือ จากนั้นเปรียบเทียบ payload ของ request ผู้ให้บริการและ log ของ session

regression พิสูจน์ว่า:

1. โหมดตรงสามารถเรียกเครื่องมือ Plugin ปลอมได้
2. การค้นหาเครื่องมือสามารถเรียกเครื่องมือ Plugin ปลอมเดียวกันได้
3. โหมดตรงเปิดเผย schema ของเครื่องมือ Plugin ปลอมให้ผู้ให้บริการโดยตรง
4. การค้นหาเครื่องมือเปิดเผยเฉพาะบริดจ์แบบกะทัดรัด
5. payload ของ request การค้นหาเครื่องมือมีขนาดเล็กกว่าสำหรับแค็ตตาล็อกปลอมขนาดใหญ่
6. log ของ session แสดงจำนวนการเรียกเครื่องมือที่คาดไว้และ telemetry ของการเรียกที่ผ่านบริดจ์

## พฤติกรรมเมื่อเกิดความล้มเหลว

การค้นหาเครื่องมือควรล้มเหลวแบบปิด:

- หากเครื่องมือไม่อยู่ในนโยบายที่มีผล search ไม่ควรส่งคืนนั้น
- หากเครื่องมือที่เลือกไม่พร้อมใช้งาน `tool_call` ควรล้มเหลว
- หากนโยบายหรือการอนุมัติบล็อกการดำเนินการ ผลลัพธ์ของการเรียกควรรายงานการบล็อกนั้นแทนการข้ามมัน
- หากบริดจ์โค้ดไม่สามารถสร้าง runtime ที่แยกออกมาได้ ให้ใช้ `mode: "tools"` หรือปิดใช้งานการค้นหาเครื่องมือสำหรับ deployment นั้น
