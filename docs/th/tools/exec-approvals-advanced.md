และการส่งต่อการอนุมัติไปยังช่องทางแชต (รวมถึงการส่งแบบ native)
สำหรับนโยบายหลักและลำดับการอนุมัติ โปรดดู [การอนุมัติ exec](/th/tools/exec-approvals)

## ไบนารีที่ปลอดภัย (เฉพาะ stdin)

`tools.exec.safeBins` กำหนดรายการขนาดเล็กของไบนารีแบบ **เฉพาะ stdin** (เช่น
`cut`) ที่สามารถรันในโหมดรายการอนุญาตได้ **โดยไม่ต้อง** มีรายการอนุญาตอย่างชัดเจน
ไบนารีที่ปลอดภัยจะปฏิเสธอาร์กิวเมนต์ไฟล์แบบตำแหน่งและโทเคนที่มีลักษณะเป็นพาธ ดังนั้นจึงทำงานได้เฉพาะกับสตรีมขาเข้าเท่านั้น ให้ถือว่านี่เป็น fast-path แบบแคบสำหรับตัวกรองสตรีม ไม่ใช่รายการความเชื่อถือทั่วไป

<Warning>
**อย่า** เพิ่มไบนารี interpreter หรือ runtime (เช่น `python3`, `node`,
`ruby`, `bash`, `sh`, `zsh`) ลงใน `safeBins` หากคำสั่งสามารถประเมินโค้ด
เรียกใช้คำสั่งย่อย หรืออ่านไฟล์ได้ตามการออกแบบ ให้ใช้รายการอนุญาตอย่างชัดเจน
และเปิดพรอมป์การอนุมัติไว้ ไบนารีที่ปลอดภัยแบบกำหนดเองต้องกำหนดโปรไฟล์อย่างชัดเจน
ใน `tools.exec.safeBinProfiles.<bin>`
</Warning>

ไบนารีที่ปลอดภัยเริ่มต้น:

[//]: # "SAFE_BIN_DEFAULTS:START"

`cut`, `uniq`, `head`, `tail`, `tr`, `wc`

[//]: # "SAFE_BIN_DEFAULTS:END"

`grep` และ `sort` ไม่อยู่ในรายการเริ่มต้น หากคุณเลือกใช้ ให้คงรายการอนุญาตอย่างชัดเจน
สำหรับเวิร์กโฟลว์ที่ไม่ใช่ stdin ของคำสั่งเหล่านี้ไว้ สำหรับ `grep` ในโหมด safe-bin
ให้ระบุแพตเทิร์นด้วย `-e`/`--regexp`; รูปแบบแพตเทิร์นแบบตำแหน่งจะถูกปฏิเสธ
เพื่อไม่ให้สามารถแฝงตัวดำเนินการไฟล์เข้ามาเป็นอาร์กิวเมนต์ตำแหน่งที่กำกวมได้

### การตรวจสอบ argv และแฟล็กที่ถูกปฏิเสธ

การตรวจสอบเป็นแบบกำหนดผลแน่นอนจากรูปร่างของ argv เท่านั้น (ไม่มีการตรวจสอบว่าไฟล์มีอยู่จริงในระบบโฮสต์)
ซึ่งป้องกันพฤติกรรมแบบ oracle จากความแตกต่างระหว่าง allow/deny
ตัวเลือกที่มุ่งเน้นไฟล์จะถูกปฏิเสธสำหรับไบนารีที่ปลอดภัยเริ่มต้น; ตัวเลือกแบบยาว
ตรวจสอบแบบ fail-closed (แฟล็กที่ไม่รู้จักและตัวย่อที่กำกวมจะถูกปฏิเสธ)

แฟล็กที่ถูกปฏิเสธตามโปรไฟล์ safe-bin:

[//]: # "SAFE_BIN_DENIED_FLAGS:START"

- `grep`: `--dereference-recursive`, `--directories`, `--exclude-from`, `--file`, `--recursive`, `-R`, `-d`, `-f`, `-r`
- `jq`: `--argfile`, `--from-file`, `--library-path`, `--rawfile`, `--slurpfile`, `-L`, `-f`
- `sort`: `--compress-program`, `--files0-from`, `--output`, `--random-source`, `--temporary-directory`, `-T`, `-o`
- `wc`: `--files0-from`

[//]: # "SAFE_BIN_DENIED_FLAGS:END"

ไบนารีที่ปลอดภัยยังบังคับให้โทเคน argv ถูกปฏิบัติเป็น **ข้อความลิเทอรัล** ตอนเรียกใช้
(ไม่มี globbing และไม่มีการขยาย `$VARS`) สำหรับเซกเมนต์แบบเฉพาะ stdin ดังนั้นแพตเทิร์น
เช่น `*` หรือ `$HOME/...` จึงไม่สามารถใช้แฝงการอ่านไฟล์ได้

### ไดเรกทอรีไบนารีที่เชื่อถือได้

ไบนารีที่ปลอดภัยต้อง resolve จากไดเรกทอรีไบนารีที่เชื่อถือได้ (ค่าเริ่มต้นของระบบ รวมกับ
`tools.exec.safeBinTrustedDirs` แบบเลือกเพิ่มได้) รายการใน `PATH` จะไม่ถูกเชื่อถือโดยอัตโนมัติ
ไดเรกทอรีที่เชื่อถือได้เริ่มต้นตั้งใจให้มีน้อยที่สุด: `/bin`, `/usr/bin` หาก
ไฟล์ executable แบบ safe-bin ของคุณอยู่ในพาธของ package-manager/user (เช่น
`/opt/homebrew/bin`, `/usr/local/bin`, `/opt/local/bin`, `/snap/bin`) ให้เพิ่มพาธเหล่านั้น
อย่างชัดเจนใน `tools.exec.safeBinTrustedDirs`

### การเชน shell, wrapper และ multiplexer

การเชน shell (`&&`, `||`, `;`) อนุญาตเมื่อทุกเซกเมนต์ระดับบนสุด
เป็นไปตามรายการอนุญาต (รวมถึงไบนารีที่ปลอดภัยหรือการอนุญาตอัตโนมัติของ skill) Redirection
ยังไม่รองรับในโหมดรายการอนุญาต Command substitution (`$()` / backticks) จะถูก
ปฏิเสธระหว่างการแยกรายการอนุญาต รวมถึงเมื่ออยู่ในเครื่องหมายคำพูดคู่; ใช้เครื่องหมายคำพูดเดี่ยว
หากคุณต้องการข้อความ `$()` แบบลิเทอรัล

ในการอนุมัติผ่าน companion app บน macOS ข้อความ shell ดิบที่มีไวยากรณ์ควบคุมหรือ
ขยายของ shell (`&&`, `||`, `;`, `|`, `` ` ``, `$`, `<`, `>`, `(`, `)`) จะถูก
ถือว่าไม่ตรงกับรายการอนุญาต เว้นแต่ตัวไบนารี shell เองจะอยู่ในรายการอนุญาต

สำหรับ shell wrapper (`bash|sh|zsh ... -c/-lc`) การ override env แบบจำกัดตามคำขอ
จะถูกลดให้เหลือรายการอนุญาตขนาดเล็กอย่างชัดเจน (`TERM`, `LANG`, `LC_*`, `COLORTERM`,
`NO_COLOR`, `FORCE_COLOR`)

สำหรับการตัดสินใจแบบ `allow-always` ในโหมดรายการอนุญาต wrapper สำหรับ dispatch ที่รู้จัก (`env`,
`nice`, `nohup`, `stdbuf`, `timeout`) จะบันทึกพาธ executable ภายในแทน
พาธของ wrapper Shell multiplexer (`busybox`, `toybox`) จะถูก unwrap สำหรับ
applet ของ shell (`sh`, `ash` เป็นต้น) ด้วยวิธีเดียวกัน หากไม่สามารถ unwrap wrapper หรือ multiplexer
ได้อย่างปลอดภัย จะไม่มีการบันทึกรายการอนุญาตโดยอัตโนมัติ

หากคุณเพิ่ม interpreter อย่าง `python3` หรือ `node` ลงในรายการอนุญาต ให้ใช้
`tools.exec.strictInlineEval=true` เพื่อให้ inline eval ยังต้องมี
การอนุมัติอย่างชัดเจน ในโหมด strict `allow-always` ยังสามารถบันทึกการเรียกใช้
interpreter/script ที่ไม่เป็นอันตรายได้ แต่ตัวนำพา inline-eval จะไม่ถูกบันทึก
โดยอัตโนมัติ

## คำสั่ง interpreter/runtime

การรัน interpreter/runtime ที่รองรับด้วยการอนุมัติถูกตั้งใจให้ออกแบบอย่างระมัดระวัง:

- บริบท argv/cwd/env แบบตรงทั้งหมดจะถูกผูกไว้เสมอ
- รูปแบบ shell script โดยตรงและไฟล์ runtime โดยตรงจะถูกผูกแบบ best-effort กับ snapshot ไฟล์ local ที่เป็นรูปธรรมเพียงไฟล์เดียว
- รูปแบบ wrapper ของ package-manager ทั่วไปที่ยัง resolve ไปยังไฟล์ local โดยตรงหนึ่งไฟล์ (เช่น
  `pnpm exec`, `pnpm node`, `npm exec`, `npx`) จะถูก unwrap ก่อนการผูก
- หาก OpenClaw ไม่สามารถระบุไฟล์ local ที่เป็นรูปธรรมหนึ่งไฟล์พอดีสำหรับคำสั่ง interpreter/runtime
  ได้ (เช่น package scripts, รูปแบบ eval, เชน loader เฉพาะ runtime หรือรูปแบบหลายไฟล์ที่กำกวม)
  การดำเนินการที่รองรับด้วยการอนุมัติจะถูกปฏิเสธ แทนที่จะอ้างว่าครอบคลุมเชิง semantic ทั้งที่ไม่ได้ครอบคลุม
- สำหรับเวิร์กโฟลว์เหล่านั้น ให้ใช้ sandboxing, ขอบเขตโฮสต์แยกต่างหาก หรือรายการอนุญาต/เวิร์กโฟลว์เต็มที่เชื่อถืออย่างชัดเจน
  ซึ่ง operator ยอมรับ semantic ของ runtime ที่กว้างกว่า

เมื่อจำเป็นต้องมีการอนุมัติ เครื่องมือ exec จะคืนค่าทันทีพร้อม id การอนุมัติ ใช้ id นั้นเพื่อ
จับคู่กับเหตุการณ์ระบบภายหลัง (`Exec finished` / `Exec denied`) หากไม่มีการตัดสินใจมาถึงก่อน
timeout คำขอจะถูกถือว่า timeout การอนุมัติและแสดงเป็นเหตุผลการปฏิเสธ

### พฤติกรรมการส่ง followup

หลังจาก async exec ที่ได้รับอนุมัติเสร็จสิ้น OpenClaw จะส่ง turn `agent` แบบ followup ไปยัง session เดิม

- หากมี target การส่งภายนอกที่ถูกต้อง (ช่องทางที่ส่งได้พร้อม target `to`) การส่ง followup จะใช้ช่องทางนั้น
- ในโฟลว์ webchat-only หรือ internal-session ที่ไม่มี target ภายนอก การส่ง followup จะอยู่เฉพาะใน session (`deliver: false`)
- หาก caller ขอการส่งภายนอกแบบ strict อย่างชัดเจนแต่ไม่มีช่องทางภายนอกที่ resolve ได้ คำขอจะล้มเหลวด้วย `INVALID_REQUEST`
- หากเปิดใช้ `bestEffortDeliver` และไม่สามารถ resolve ช่องทางภายนอกได้ การส่งจะถูกลดระดับเป็นเฉพาะใน session แทนที่จะล้มเหลว

ด้วย `/approve` การทำงานนี้ใช้ pipeline การส่งออกปกติ

### การส่งต่อการอนุมัติ Plugin

การส่งต่อการอนุมัติ Plugin ใช้ pipeline การส่งเดียวกับการอนุมัติ exec แต่มี
config แยกของตัวเองใต้ `approvals.plugin` การเปิดหรือปิดอย่างใดอย่างหนึ่งไม่กระทบอีกอย่าง
__OC_I18N_900003__
รูปร่าง config เหมือนกับ `approvals.exec`: `enabled`, `mode`, `agentFilter`,
`sessionFilter` และ `targets` ทำงานแบบเดียวกัน

ช่องทางที่รองรับการตอบกลับแบบโต้ตอบร่วมกันจะแสดงปุ่มอนุมัติแบบเดียวกันสำหรับทั้งการอนุมัติ exec และ
Plugin ช่องทางที่ไม่มี UI โต้ตอบร่วมกันจะ fallback เป็นข้อความธรรมดาพร้อมคำแนะนำ `/approve`

### การอนุมัติในแชตเดียวกันบนทุกช่องทาง

เมื่อคำขออนุมัติ exec หรือ Plugin เริ่มจากพื้นผิวแชตที่ส่งได้ แชตเดียวกัน
สามารถอนุมัติด้วย `/approve` ได้ตามค่าเริ่มต้นแล้ว สิ่งนี้ใช้กับช่องทางอย่าง Slack, Matrix และ
Microsoft Teams นอกเหนือจากโฟลว์ Web UI และ terminal UI ที่มีอยู่

เส้นทางคำสั่งข้อความร่วมนี้ใช้โมเดล auth ของช่องทางปกติสำหรับบทสนทนานั้น หาก
แชตต้นทางสามารถส่งคำสั่งและรับการตอบกลับได้อยู่แล้ว คำขออนุมัติไม่จำเป็นต้องมี
adapter การส่งแบบ native แยกต่างหากเพียงเพื่อให้ค้างอยู่

Discord และ Telegram ยังรองรับ `/approve` ในแชตเดียวกันด้วย แต่ช่องทางเหล่านั้นยังคงใช้
รายการ approver ที่ resolve แล้วสำหรับการอนุญาต แม้เมื่อปิดการส่งการอนุมัติแบบ native

สำหรับ Telegram และ client การอนุมัติแบบ native อื่นๆ ที่เรียก Gateway โดยตรง
fallback นี้ถูกจำกัดโดยตั้งใจไว้เฉพาะกรณีล้มเหลวแบบ "ไม่พบการอนุมัติ" เท่านั้น การปฏิเสธ/ข้อผิดพลาด
การอนุมัติ exec จริงจะไม่ retry แบบเงียบๆ เป็นการอนุมัติ Plugin

### การส่งการอนุมัติแบบ native

บางช่องทางยังสามารถทำหน้าที่เป็น client การอนุมัติแบบ native ได้ด้วย client แบบ native จะเพิ่ม DM ถึง approver, fanout ไปยังแชตต้นทาง
และ UX การอนุมัติแบบโต้ตอบเฉพาะช่องทางไว้บนโฟลว์ `/approve` ในแชตเดียวกันที่ใช้ร่วมกัน

เมื่อมีการ์ด/ปุ่มอนุมัติแบบเนทีฟ UI แบบเนทีฟนั้นจะเป็นเส้นทางหลักที่เอเจนต์ใช้ เอเจนต์ไม่ควร echo คำสั่งแชตแบบข้อความธรรมดา `/approve` ซ้ำอีก เว้นแต่ผลลัพธ์ของเครื่องมือจะระบุว่าการอนุมัติผ่านแชตไม่พร้อมใช้งาน หรือการอนุมัติด้วยตนเองเป็นเส้นทางเดียวที่เหลืออยู่

หากมีการกำหนดค่าไคลเอนต์อนุมัติแบบเนทีฟไว้ แต่ไม่มีรันไทม์แบบเนทีฟที่ทำงานอยู่สำหรับช่องทางต้นทาง OpenClaw จะยังแสดงพรอมป์ `/approve` แบบกำหนดได้แน่นอนในเครื่องไว้ หากรันไทม์แบบเนทีฟทำงานอยู่และพยายามส่งแล้วแต่ไม่มีเป้าหมายใดได้รับการ์ด OpenClaw จะส่งประกาศสำรองในแชตเดียวกันพร้อมคำสั่ง `/approve <id> <decision>` ที่ถูกต้องครบถ้วน เพื่อให้คำขอยังสามารถถูกจัดการได้

โมเดลทั่วไป:

- นโยบาย exec ของโฮสต์ยังคงตัดสินว่าจำเป็นต้องมีการอนุมัติ exec หรือไม่
- `approvals.exec` ควบคุมการส่งต่อพรอมป์อนุมัติไปยังปลายทางแชตอื่น
- `channels.<channel>.execApprovals` ควบคุมว่าช่องทางนั้นทำหน้าที่เป็นไคลเอนต์อนุมัติแบบเนทีฟหรือไม่

ไคลเอนต์อนุมัติแบบเนทีฟจะเปิดใช้การส่งแบบ DM ก่อนโดยอัตโนมัติเมื่อเงื่อนไขทั้งหมดนี้เป็นจริง:

- ช่องทางรองรับการส่งการอนุมัติแบบเนทีฟ
- สามารถระบุผู้อนุมัติได้จาก `execApprovals.approvers` ที่กำหนดไว้อย่างชัดเจน หรือตัวตนของเจ้าของ
  เช่น `commands.ownerAllowFrom`
- `channels.<channel>.execApprovals.enabled` ไม่ได้ตั้งค่าไว้ หรือเป็น `"auto"`

ตั้งค่า `enabled: false` เพื่อปิดใช้ไคลเอนต์อนุมัติแบบเนทีฟอย่างชัดเจน ตั้งค่า `enabled: true` เพื่อบังคับ
เปิดใช้เมื่อระบุผู้อนุมัติได้ การส่งไปยังแชตต้นทางแบบสาธารณะยังคงต้องระบุอย่างชัดเจนผ่าน
`channels.<channel>.execApprovals.target`

FAQ: [ทำไมจึงมีการกำหนดค่าการอนุมัติ exec สองรายการสำหรับการอนุมัติผ่านแชต?](/help/faq-first-run#why-are-there-two-exec-approval-configs-for-chat-approvals)

- Discord: `channels.discord.execApprovals.*`
- Slack: `channels.slack.execApprovals.*`
- Telegram: `channels.telegram.execApprovals.*`

ไคลเอนต์อนุมัติแบบเนทีฟเหล่านี้เพิ่มการกำหนดเส้นทาง DM และการกระจายไปยังช่องทางแบบเลือกได้บน flow
`/approve` ในแชตเดียวกันที่ใช้ร่วมกัน และปุ่มอนุมัติที่ใช้ร่วมกัน

พฤติกรรมที่ใช้ร่วมกัน:

- Slack, Matrix, Microsoft Teams และแชตที่ส่งข้อความได้ลักษณะคล้ายกัน ใช้โมเดลการยืนยันตัวตนของช่องทางตามปกติ
  สำหรับ `/approve` ในแชตเดียวกัน
- เมื่อไคลเอนต์อนุมัติแบบเนทีฟเปิดใช้โดยอัตโนมัติ เป้าหมายการส่งแบบเนทีฟเริ่มต้นคือ DM ของผู้อนุมัติ
- สำหรับ Discord และ Telegram เฉพาะผู้อนุมัติที่ระบุได้เท่านั้นที่สามารถอนุมัติหรือปฏิเสธได้
- ผู้อนุมัติของ Discord สามารถกำหนดอย่างชัดเจน (`execApprovals.approvers`) หรืออนุมานจาก `commands.ownerAllowFrom`
- ผู้อนุมัติของ Telegram สามารถกำหนดอย่างชัดเจน (`execApprovals.approvers`) หรืออนุมานจาก `commands.ownerAllowFrom`
- ผู้อนุมัติของ Slack สามารถกำหนดอย่างชัดเจน (`execApprovals.approvers`) หรืออนุมานจาก `commands.ownerAllowFrom`
- ปุ่มแบบเนทีฟของ Slack จะคงชนิด id การอนุมัติไว้ ดังนั้น id แบบ `plugin:` จึงสามารถ resolve การอนุมัติ Plugin
  ได้โดยไม่ต้องมีชั้น fallback เฉพาะ Slack ชั้นที่สอง
- การกำหนดเส้นทาง DM/ช่องทางแบบเนทีฟของ Matrix และทางลัด reaction รองรับทั้งการอนุมัติ exec และ Plugin;
  การอนุญาต Plugin ยังคงมาจาก `channels.matrix.dm.allowFrom`
- พรอมป์แบบเนทีฟของ Matrix มีเนื้อหาอีเวนต์กำหนดเอง `com.openclaw.approval` ในอีเวนต์พรอมป์แรก
  เพื่อให้ไคลเอนต์ Matrix ที่รองรับ OpenClaw อ่านสถานะการอนุมัติแบบมีโครงสร้างได้ ในขณะที่ไคลเอนต์มาตรฐาน
  ยังคงใช้ fallback `/approve` แบบข้อความธรรมดา
- ผู้ร้องขอไม่จำเป็นต้องเป็นผู้อนุมัติ
- แชตต้นทางสามารถอนุมัติโดยตรงด้วย `/approve` เมื่อแชตนั้นรองรับคำสั่งและการตอบกลับอยู่แล้ว
- ปุ่มอนุมัติ Discord แบบเนทีฟกำหนดเส้นทางตามชนิด id การอนุมัติ: id แบบ `plugin:` จะไป
  ยังการอนุมัติ Plugin โดยตรง ส่วนที่เหลือทั้งหมดจะไปยังการอนุมัติ exec
- ปุ่มอนุมัติ Telegram แบบเนทีฟใช้ fallback จาก exec ไปยัง Plugin แบบมีขอบเขตเดียวกับ `/approve`
- เมื่อ `target` แบบเนทีฟเปิดใช้การส่งไปยังแชตต้นทาง พรอมป์อนุมัติจะมีข้อความคำสั่งรวมอยู่ด้วย
- การอนุมัติ exec ที่ค้างอยู่จะหมดอายุหลัง 30 นาทีโดยค่าเริ่มต้น
- หากไม่มี UI สำหรับ operator หรือไคลเอนต์อนุมัติที่กำหนดค่าไว้ที่สามารถรับคำขอได้ พรอมป์จะ fallback ไปยัง `askFallback`

คำสั่งกลุ่มที่ละเอียดอ่อนและใช้ได้เฉพาะเจ้าของ เช่น `/diagnostics` และ `/export-trajectory` ใช้การกำหนดเส้นทาง
ส่วนตัวของเจ้าของสำหรับพรอมป์อนุมัติและผลลัพธ์สุดท้าย OpenClaw จะลองใช้เส้นทางส่วนตัวบนพื้นผิวเดียวกัน
ที่เจ้าของเรียกใช้คำสั่งก่อน หากพื้นผิวนั้นไม่มีเส้นทางส่วนตัวของเจ้าของ ระบบจะ fallback ไปยังเส้นทางเจ้าของแรก
ที่พร้อมใช้งานจาก `commands.ownerAllowFrom` ดังนั้นคำสั่งกลุ่มใน Discord ยังคงสามารถส่งการอนุมัติและผลลัพธ์
ไปยัง DM ของเจ้าของใน Telegram ได้ เมื่อ Telegram เป็นอินเทอร์เฟซส่วนตัวหลักที่กำหนดค่าไว้ แชตกลุ่มจะได้รับเพียง
การตอบรับสั้นๆ

Telegram ใช้ค่าเริ่มต้นเป็น DM ของผู้อนุมัติ (`target: "dm"`) คุณสามารถเปลี่ยนเป็น `channel` หรือ `both` ได้เมื่อคุณ
ต้องการให้พรอมป์อนุมัติปรากฏในแชต/หัวข้อ Telegram ต้นทางด้วย สำหรับหัวข้อ forum ของ Telegram
OpenClaw จะรักษาหัวข้อนั้นไว้สำหรับพรอมป์อนุมัติและการติดตามผลหลังการอนุมัติ
