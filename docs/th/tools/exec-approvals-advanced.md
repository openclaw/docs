หัวข้อขั้นสูงของการอนุมัติ exec: เส้นทางลัด `safeBins`, การผูกอินเทอร์พรีเตอร์/รันไทม์
และการส่งต่อการอนุมัติไปยังช่องทางแชต (รวมถึงการนำส่งแบบเนทีฟ)
สำหรับนโยบายหลักและโฟลว์การอนุมัติ โปรดดู [การอนุมัติ Exec](/th/tools/exec-approvals)

## ไบนารีที่ปลอดภัย (เฉพาะ stdin)

`tools.exec.safeBins` ระบุชื่อไบนารีแบบ **เฉพาะ stdin** (ตัวอย่างเช่น `cut`) ซึ่ง
ทำงานในโหมดรายการอนุญาต **โดยไม่ต้องมี** รายการอนุญาตที่ระบุไว้อย่างชัดเจน ไบนารีที่ปลอดภัยจะปฏิเสธ
อาร์กิวเมนต์ไฟล์แบบระบุตำแหน่งและโทเค็นที่มีลักษณะคล้ายพาธ จึงทำงานได้เฉพาะกับ
สตรีมขาเข้าเท่านั้น ให้ถือว่านี่เป็นเส้นทางลัดแบบจำกัดสำหรับตัวกรองสตรีม ไม่ใช่
รายการความเชื่อถือทั่วไป

<Warning>
**ห้าม** เพิ่มไบนารีอินเทอร์พรีเตอร์หรือรันไทม์ (ตัวอย่างเช่น `python3`, `node`,
`ruby`, `bash`, `sh`, `zsh`) ลงใน `safeBins` หากคำสั่งสามารถประเมินโค้ด
เรียกใช้คำสั่งย่อย หรืออ่านไฟล์ได้โดยการออกแบบ ให้เลือกใช้รายการอนุญาตที่ระบุไว้อย่างชัดเจน
และเปิดพรอมต์การอนุมัติไว้ ไบนารีที่ปลอดภัยแบบกำหนดเองต้องกำหนด
โปรไฟล์ไว้อย่างชัดเจนใน `tools.exec.safeBinProfiles.<bin>`
</Warning>

ไบนารีที่ปลอดภัยเริ่มต้น:

[//]: # "SAFE_BIN_DEFAULTS:START"

`cut`, `uniq`, `head`, `tail`, `tr`, `wc`

[//]: # "SAFE_BIN_DEFAULTS:END"

`grep` และ `sort` ไม่อยู่ในรายการเริ่มต้น หากเลือกเปิดใช้ ให้คง
รายการอนุญาตที่ระบุไว้อย่างชัดเจนสำหรับเวิร์กโฟลว์ที่ไม่ได้ใช้ stdin สำหรับ `grep` ในโหมดไบนารีที่ปลอดภัย
ให้ระบุรูปแบบด้วย `-e`/`--regexp`; ระบบจะปฏิเสธรูปแบบการระบุแพตเทิร์นแบบระบุตำแหน่ง
เพื่อไม่ให้สามารถลอบส่งโอเปอแรนด์ไฟล์เป็นอาร์กิวเมนต์ระบุตำแหน่งที่กำกวมได้

### การตรวจสอบ argv และแฟล็กที่ถูกปฏิเสธ

การตรวจสอบมีผลแน่นอนจากรูปแบบของ argv เท่านั้น (ไม่มีการตรวจสอบว่าไฟล์มีอยู่จริง
ในระบบไฟล์ของโฮสต์) ซึ่งป้องกันพฤติกรรมออราเคิลตรวจสอบการมีอยู่ของไฟล์จากความแตกต่าง
ระหว่างการอนุญาต/ปฏิเสธ ตัวเลือกที่เกี่ยวข้องกับไฟล์จะถูกปฏิเสธสำหรับไบนารีที่ปลอดภัยเริ่มต้น ส่วน
ตัวเลือกแบบยาวจะตรวจสอบแบบ fail-closed (ปฏิเสธแฟล็กที่ไม่รู้จักและตัวย่อที่กำกวม)
แฟล็กบูลีนแบบอ่านอย่างเดียวที่รู้จักของไบนารีเริ่มต้น (ตัวอย่างเช่น
`wc -l`, `tr -d`, `uniq -c`) จะได้รับการยอมรับ ส่วนแฟล็กแบบสั้นที่ไม่รู้จักจะยังคง
เป็นแบบ fail-closed และส่งต่อไปยังการอนุมัติด้วยตนเอง

แฟล็กที่ถูกปฏิเสธตามโปรไฟล์ไบนารีที่ปลอดภัย:

[//]: # "SAFE_BIN_DENIED_FLAGS:START"

- `grep`: `--dereference-recursive`, `--directories`, `--exclude-from`, `--file`, `--recursive`, `-R`, `-d`, `-f`, `-r`
- `jq`: `--argfile`, `--from-file`, `--library-path`, `--rawfile`, `--slurpfile`, `-L`, `-f`
- `sort`: `--compress-program`, `--files0-from`, `--output`, `--random-source`, `--temporary-directory`, `-T`, `-o`
- `tail`: `--follow`, `--retry`, `-F`, `-f`
- `wc`: `--files0-from`

[//]: # "SAFE_BIN_DENIED_FLAGS:END"

ไบนารีที่ปลอดภัยยังบังคับให้โทเค็น argv ถูกปฏิบัติเป็น **ข้อความตามตัวอักษร** ขณะดำเนินการ
(ไม่มีการขยาย glob และไม่มีการขยาย `$VARS`) สำหรับเซกเมนต์เฉพาะ stdin ดังนั้น
จึงไม่สามารถใช้รูปแบบอย่าง `*` หรือ `$HOME/...` เพื่อลอบอ่านไฟล์ได้ `awk`,
`sed` และ `jq` จะถูกปฏิเสธไม่ให้เป็นไบนารีที่ปลอดภัยเสมอ เนื่องจากไม่สามารถ
ตรวจสอบได้ว่าเซแมนติกของไบนารีเหล่านี้จำกัดอยู่เฉพาะ stdin: `jq` สามารถอ่านข้อมูลสภาพแวดล้อมและโหลดโค้ด jq จาก
โมดูลหรือไฟล์เริ่มต้น ให้ใช้รายการอนุญาตที่ระบุไว้อย่างชัดเจนหรือพรอมต์การอนุมัติสำหรับ
เครื่องมือเหล่านั้นแทน `safeBins`

### ไดเรกทอรีไบนารีที่เชื่อถือได้

ไบนารีที่ปลอดภัยต้องได้รับการแก้ไขตำแหน่งจากไดเรกทอรีไบนารีที่เชื่อถือได้ (ค่าเริ่มต้นของระบบและ
`tools.exec.safeBinTrustedDirs` ที่เลือกเพิ่มได้) รายการ `PATH` จะไม่ได้รับความเชื่อถือโดยอัตโนมัติ
ไดเรกทอรีที่เชื่อถือได้เริ่มต้นถูกจำกัดไว้ให้น้อยโดยตั้งใจ: `/bin`, `/usr/bin` หาก
ไฟล์ปฏิบัติการไบนารีที่ปลอดภัยอยู่ในพาธของตัวจัดการแพ็กเกจ/ผู้ใช้ (ตัวอย่างเช่น
`/opt/homebrew/bin`, `/usr/local/bin`, `/opt/local/bin`, `/snap/bin`) ให้เพิ่มพาธเหล่านั้น
ลงใน `tools.exec.safeBinTrustedDirs` อย่างชัดเจน

### การเชื่อมคำสั่งเชลล์ แรปเปอร์ และมัลติเพล็กเซอร์

อนุญาตให้เชื่อมคำสั่งเชลล์ (`&&`, `||`, `;`) เมื่อทุกเซกเมนต์ระดับบนสุด
เป็นไปตามรายการอนุญาต (รวมถึงไบนารีที่ปลอดภัยหรือการอนุญาต Skills โดยอัตโนมัติ) การเปลี่ยนทิศทาง
ยังคงไม่รองรับในโหมดรายการอนุญาต การแทนที่คำสั่ง (`$()` / แบ็กทิก)
จะถูกปฏิเสธระหว่างการแยกวิเคราะห์รายการอนุญาต รวมถึงภายในอัญประกาศคู่ ให้ใช้
อัญประกาศเดี่ยวหากต้องการข้อความ `$()` ตามตัวอักษร

สำหรับการอนุมัติผ่านแอปคู่หูบน macOS ข้อความเชลล์ดิบที่มีไวยากรณ์ควบคุมเชลล์หรือ
ไวยากรณ์การขยาย (`&&`, `||`, `;`, `|`, `` ` ``, `$`, `<`, `>`, `(`, `)`) จะ
ถือว่าไม่ตรงกับรายการอนุญาต เว้นแต่ไบนารีเชลล์เองจะอยู่ในรายการอนุญาต

สำหรับแรปเปอร์เชลล์ (`bash|sh|zsh ... -c/-lc`) การแทนที่ตัวแปรสภาพแวดล้อมที่จำกัดขอบเขตตามคำขอจะ
ถูกลดเหลือรายการอนุญาตขนาดเล็กที่ระบุไว้อย่างชัดเจน (`TERM`, `LANG`, `LC_*`, `COLORTERM`,
`NO_COLOR`, `FORCE_COLOR`)

สำหรับการตัดสินใจ `allow-always` ในโหมดรายการอนุญาต แรปเปอร์การส่งต่อแบบโปร่งใส
(ตัวอย่างเช่น `env`, `flock`, `nice`, `nohup`, `stdbuf`, `timeout`) จะจัดเก็บ
พาธของไฟล์ปฏิบัติการภายในแทนพาธของแรปเปอร์ มัลติเพล็กเซอร์เชลล์
(`busybox`, `toybox`) จะถูกแกะแรปสำหรับแอปเพล็ตเชลล์ (`sh`, `ash` เป็นต้น) ด้วย
วิธีเดียวกัน หากไม่สามารถแกะแรปแรปเปอร์หรือมัลติเพล็กเซอร์ได้อย่างปลอดภัย ระบบจะไม่จัดเก็บ
รายการอนุญาตโดยอัตโนมัติ

หากเพิ่มอินเทอร์พรีเตอร์อย่าง `python3` หรือ `node` ลงในรายการอนุญาต ให้เลือกใช้
`tools.exec.strictInlineEval=true` เพื่อให้การประเมินแบบอินไลน์ยังคงต้องได้รับ
การอนุมัติอย่างชัดเจน ในโหมดเข้มงวด `allow-always` ยังคงสามารถจัดเก็บ
การเรียกใช้อินเทอร์พรีเตอร์/สคริปต์ที่ไม่เป็นอันตรายได้ แต่ตัวนำการประเมินแบบอินไลน์จะไม่ถูกจัดเก็บ
โดยอัตโนมัติ

### ไบนารีที่ปลอดภัยเทียบกับรายการอนุญาต

| หัวข้อ            | `tools.exec.safeBins`                                  | รายการอนุญาต (`exec-approvals.json`)                                                  |
| ---------------- | ------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| เป้าหมาย             | อนุญาตตัวกรอง stdin ที่มีขอบเขตจำกัดโดยอัตโนมัติ                        | เชื่อถือไฟล์ปฏิบัติการที่ระบุไว้อย่างชัดเจน                                              |
| ประเภทการจับคู่       | ชื่อไฟล์ปฏิบัติการ + นโยบาย argv ของไบนารีที่ปลอดภัย                 | glob พาธไฟล์ปฏิบัติการที่แก้ไขแล้ว หรือ glob ชื่อคำสั่งเปล่าสำหรับคำสั่งที่เรียกผ่าน PATH |
| ขอบเขตอาร์กิวเมนต์   | จำกัดด้วยโปรไฟล์ไบนารีที่ปลอดภัยและกฎโทเค็นตามตัวอักษร | จับคู่พาธโดยค่าเริ่มต้น; `argPattern` ที่เลือกเพิ่มได้สามารถจำกัด argv ที่แยกวิเคราะห์แล้ว              |
| ตัวอย่างทั่วไป | `head`, `tail`, `tr`, `wc`                             | `jq`, `python3`, `node`, `ffmpeg`, CLI แบบกำหนดเอง                                     |
| การใช้งานที่เหมาะที่สุด         | การแปลงข้อความความเสี่ยงต่ำในไปป์ไลน์                  | เครื่องมือใดก็ตามที่มีพฤติกรรมหรือผลข้างเคียงกว้างกว่า                                     |

ตำแหน่งการกำหนดค่า:

- `safeBins` มาจากการกำหนดค่า (`tools.exec.safeBins` หรือ `agents.list[].tools.exec.safeBins` ต่อเอเจนต์)
- `safeBinTrustedDirs` มาจากการกำหนดค่า (`tools.exec.safeBinTrustedDirs` หรือ `agents.list[].tools.exec.safeBinTrustedDirs` ต่อเอเจนต์)
- `safeBinProfiles` มาจากการกำหนดค่า (`tools.exec.safeBinProfiles` หรือ `agents.list[].tools.exec.safeBinProfiles` ต่อเอเจนต์) คีย์โปรไฟล์ต่อเอเจนต์จะแทนที่คีย์ส่วนกลาง
- รายการอนุญาตอยู่ในไฟล์การอนุมัติเฉพาะโฮสต์ภายใต้ `agents.<id>.allowlist` (หรือผ่าน Control UI / `openclaw approvals allowlist ...`)
- `openclaw security audit` จะแจ้งเตือนด้วย `tools.exec.safe_bins_interpreter_unprofiled` เมื่อไบนารีอินเทอร์พรีเตอร์/รันไทม์ปรากฏใน `safeBins` โดยไม่มีโปรไฟล์ที่ระบุไว้อย่างชัดเจน
- `openclaw doctor --fix` สามารถสร้างโครงรายการ `safeBinProfiles.<bin>` แบบกำหนดเองที่ขาดหายไปเป็น `{}` (ตรวจสอบและปรับให้เข้มงวดขึ้นภายหลัง) ระบบจะไม่สร้างโครงไบนารีอินเทอร์พรีเตอร์/รันไทม์โดยอัตโนมัติ

ตัวอย่างโปรไฟล์แบบกำหนดเอง:

```json5
{
  tools: {
    exec: {
      safeBins: ["myfilter"],
      safeBinProfiles: {
        myfilter: {
          minPositional: 0,
          maxPositional: 0,
          allowedValueFlags: ["-n", "--limit"],
          deniedFlags: ["-f", "--file", "-c", "--command"],
        },
      },
    },
  },
}
```

## คำสั่งอินเทอร์พรีเตอร์/รันไทม์

การเรียกใช้อินเทอร์พรีเตอร์/รันไทม์ที่มีการอนุมัติรองรับถูกออกแบบให้ระมัดระวัง:

- บริบท argv/cwd/env ที่ตรงกันทุกประการจะถูกผูกไว้เสมอ
- รูปแบบไฟล์สคริปต์เชลล์โดยตรงและไฟล์รันไทม์โดยตรงจะถูกผูกกับสแนปช็อตไฟล์ภายในเครื่องที่เฉพาะเจาะจงหนึ่งไฟล์
  แบบพยายามให้ดีที่สุด
- รูปแบบแรปเปอร์ตัวจัดการแพ็กเกจทั่วไปที่ยังคงแก้ไขตำแหน่งไปยังไฟล์ภายในเครื่องโดยตรงหนึ่งไฟล์ (ตัวอย่างเช่น
  `pnpm exec`, `pnpm node`, `npm exec`, `npx`) จะถูกแกะแรปก่อนการผูก
- หาก OpenClaw ไม่สามารถระบุไฟล์ภายในเครื่องที่เฉพาะเจาะจงเพียงหนึ่งไฟล์สำหรับคำสั่งอินเทอร์พรีเตอร์/รันไทม์
  ได้อย่างแน่นอน (ตัวอย่างเช่น สคริปต์แพ็กเกจ รูปแบบการประเมิน เชนตัวโหลดเฉพาะรันไทม์ หรือรูปแบบหลายไฟล์
  ที่กำกวม) การดำเนินการที่มีการอนุมัติรองรับจะถูกปฏิเสธ แทนที่จะอ้างว่าครอบคลุมเซแมนติกที่ระบบ
  ไม่ได้ครอบคลุม
- สำหรับเวิร์กโฟลว์เหล่านั้น ให้เลือกใช้แซนด์บ็อกซ์ ขอบเขตโฮสต์แยกต่างหาก หรือเวิร์กโฟลว์
  รายการอนุญาต/เต็มรูปแบบที่เชื่อถืออย่างชัดเจน ซึ่งผู้ปฏิบัติงานยอมรับเซแมนติกของรันไทม์ที่กว้างกว่า

เมื่อจำเป็นต้องมีการอนุมัติ เครื่องมือ exec จะส่งคืนทันทีพร้อมรหัสการอนุมัติ ใช้รหัสดังกล่าวเพื่อ
เชื่อมโยงกับเหตุการณ์ระบบของการเรียกใช้ที่ได้รับอนุมัติในภายหลัง (`Exec finished` และ `Exec running` เมื่อกำหนดค่าไว้)
หากไม่มีการตัดสินใจก่อนหมดเวลา คำขอจะถูกถือว่าหมดเวลารอการอนุมัติและ
แสดงเป็นการปฏิเสธคำสั่งโฮสต์ขั้นสุดท้าย สำหรับการอนุมัติแบบอะซิงโครนัสของเอเจนต์หลักที่มี
เซสชันต้นทาง OpenClaw จะทำให้เซสชันนั้นดำเนินต่อด้วยการติดตามผลภายใน เพื่อให้เอเจนต์รับรู้ว่า
คำสั่งไม่ได้ทำงาน แทนที่จะมาซ่อมแซมผลลัพธ์ที่หายไปในภายหลัง การอนุมัติ exec ที่รอดำเนินการจะหมดอายุ
หลังจาก 30 นาทีโดยค่าเริ่มต้น

### พฤติกรรมการนำส่งการติดตามผล

หลังจาก exec แบบอะซิงโครนัสที่ได้รับอนุมัติทำงานเสร็จ OpenClaw จะส่งเทิร์นติดตามผล `agent` ไปยังเซสชันเดียวกัน
การอนุมัติแบบอะซิงโครนัสที่ถูกปฏิเสธใช้เส้นทางติดตามผลของเซสชันหลักเดียวกันเพื่อรายงานสถานะการปฏิเสธ แต่
จะไม่ลงทะเบียนการส่งต่อรันไทม์แบบยกระดับและจะไม่เรียกใช้คำสั่ง การปฏิเสธที่ไม่มี
เซสชันหลักซึ่งดำเนินต่อได้จะถูกระงับหรือรายงานผ่านเส้นทางตรงที่ปลอดภัยหากมี

- หากมีเป้าหมายการนำส่งภายนอกที่ถูกต้อง (ช่องทางที่นำส่งได้พร้อม `to` เป้าหมาย) การนำส่งการติดตามผลจะใช้ช่องทางนั้น
- ในโฟลว์เฉพาะเว็บแชตหรือเซสชันภายในที่ไม่มีเป้าหมายภายนอก การนำส่งการติดตามผลจะอยู่เฉพาะในเซสชัน (`deliver: false`)
- หากผู้เรียกขอการนำส่งภายนอกแบบเข้มงวดอย่างชัดเจนโดยไม่มีช่องทางภายนอกที่แก้ไขตำแหน่งได้ คำขอจะล้มเหลวด้วย `INVALID_REQUEST`
- หากเปิดใช้ `bestEffortDeliver` และไม่สามารถแก้ไขช่องทางภายนอกได้ การนำส่งจะถูกลดระดับเป็นเฉพาะเซสชันแทนที่จะล้มเหลว

## การส่งต่อการอนุมัติไปยังช่องทางแชต

สามารถส่งต่อพรอมต์การอนุมัติ exec ไปยังช่องทางแชตใดก็ได้ (รวมถึงช่องทาง Plugin) และอนุมัติ
ด้วย `/approve` การดำเนินการนี้ใช้ไปป์ไลน์การนำส่งขาออกตามปกติ

การกำหนดค่า:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "session", // "session" | "targets" | "both"
      agentFilter: ["main"],
      sessionFilter: ["discord"], // substring or regex
      targets: [
        { channel: "slack", to: "U12345678" },
        { channel: "telegram", to: "123456789" },
      ],
    },
  },
}
```

ตอบกลับในแชต:

```
/approve <id> allow-once
/approve <id> allow-always
/approve <id> deny
```

คำสั่ง `/approve` จัดการทั้งการอนุมัติ exec และการอนุมัติ Plugin หาก ID ไม่ตรงกับการอนุมัติ exec ที่รอดำเนินการ ระบบจะตรวจสอบการอนุมัติ Plugin แทนโดยอัตโนมัติ กลไกสำรองนี้จำกัดเฉพาะความล้มเหลวประเภท "ไม่พบการอนุมัติ" เท่านั้น การปฏิเสธ/ข้อผิดพลาดจริงจากการอนุมัติ exec จะไม่ลองใหม่โดยไม่แจ้งให้ทราบในฐานะการอนุมัติ Plugin

### การส่งต่อการอนุมัติ Plugin

การส่งต่อการอนุมัติ Plugin ใช้ไปป์ไลน์การส่งเดียวกับการอนุมัติ exec แต่มี
การกำหนดค่าแยกต่างหากภายใต้ `approvals.plugin` การเปิดหรือปิดรายการหนึ่งจะไม่ส่งผลต่ออีกรายการ
สำหรับพฤติกรรมการเขียน Plugin ฟิลด์คำขอ และความหมายของการตัดสินใจ โปรดดู
[คำขอสิทธิ์ของ Plugin](/plugins/plugin-permission-requests)

```json5
{
  approvals: {
    plugin: {
      enabled: true,
      mode: "targets",
      agentFilter: ["main"],
      targets: [
        { channel: "slack", to: "U12345678" },
        { channel: "telegram", to: "123456789" },
      ],
    },
  },
}
```

รูปแบบการกำหนดค่าเหมือนกับ `approvals.exec`: `enabled`, `mode`, `agentFilter`,
`sessionFilter` และ `targets` ทำงานในลักษณะเดียวกัน

ช่องทางที่รองรับการตอบกลับแบบโต้ตอบร่วมกันจะแสดงปุ่มอนุมัติแบบเดียวกันสำหรับทั้งการอนุมัติ exec และ
การอนุมัติ Plugin ช่องทางที่ไม่มี UI แบบโต้ตอบร่วมกันจะใช้ข้อความธรรมดาพร้อมคำแนะนำ
`/approve` แทน คำขออนุมัติ Plugin อาจจำกัดตัวเลือกการตัดสินใจที่ใช้ได้ โดยพื้นผิวการอนุมัติจะใช้
ชุดการตัดสินใจที่ประกาศไว้ในคำขอ และ Gateway จะปฏิเสธความพยายามส่งการตัดสินใจที่
ไม่ได้เสนอไว้

### การอนุมัติในแชตเดียวกันบนทุกช่องทาง

เมื่อคำขออนุมัติ exec หรือ Plugin มาจากพื้นผิวแชตที่ส่งข้อความได้ แชตเดียวกันนั้น
สามารถอนุมัติด้วย `/approve` ได้ตามค่าเริ่มต้น ซึ่งใช้ได้กับ Slack, Matrix, Microsoft Teams และ
แชตที่ส่งข้อความได้ในลักษณะเดียวกัน นอกเหนือจากโฟลว์ Web UI และ UI เทอร์มินัลที่มีอยู่ โดยใช้
โมเดลการยืนยันตัวตนตามปกติของช่องทางสำหรับการสนทนานั้น หากแชตต้นทางส่งคำสั่ง
และรับการตอบกลับได้อยู่แล้ว คำขออนุมัติก็ไม่จำเป็นต้องมีอะแดปเตอร์การส่งแบบเนทีฟแยกต่างหากเพียงเพื่อ
คงสถานะรอดำเนินการอีกต่อไป

Discord, Telegram และ QQ bot รองรับ `/approve` ในแชตเดียวกันด้วย แต่ช่องทางเหล่านั้นยังคงใช้
รายการผู้อนุมัติที่แก้ไขแล้วเพื่อการให้สิทธิ์ แม้จะปิดใช้งานการส่งการอนุมัติแบบเนทีฟก็ตาม

### การส่งการอนุมัติแบบเนทีฟ

บางช่องทางสามารถทำหน้าที่เป็นไคลเอนต์การอนุมัติแบบเนทีฟได้ด้วย ได้แก่ Discord, Slack, Telegram, Matrix และ QQ bot
ไคลเอนต์แบบเนทีฟเพิ่ม DM ถึงผู้อนุมัติ การกระจายไปยังแชตต้นทาง และประสบการณ์การอนุมัติแบบโต้ตอบเฉพาะช่องทาง
ต่อยอดจากโฟลว์ `/approve` ในแชตเดียวกันที่ใช้ร่วมกัน

เมื่อมีการ์ด/ปุ่มอนุมัติแบบเนทีฟ UI แบบเนทีฟนั้นจะเป็นเส้นทางหลักที่เอเจนต์มองเห็น
เอเจนต์ไม่ควรส่งคำสั่ง `/approve` แบบข้อความธรรมดาซ้ำในแชตด้วย เว้นแต่ผลลัพธ์ของเครื่องมือจะระบุว่า
การอนุมัติผ่านแชตใช้ไม่ได้ หรือการอนุมัติด้วยตนเองเป็นเส้นทางเดียวที่เหลืออยู่

หากกำหนดค่าไคลเอนต์การอนุมัติแบบเนทีฟไว้ แต่ไม่มีรันไทม์แบบเนทีฟที่ทำงานอยู่สำหรับช่องทาง
ต้นทาง OpenClaw จะคงพรอมต์ `/approve` แบบกำหนดแน่นอนภายในเครื่องให้มองเห็นได้ หากรันไทม์แบบเนทีฟ
ทำงานอยู่และพยายามส่ง แต่ไม่มีเป้าหมายใดได้รับการ์ด OpenClaw จะส่งการแจ้งเตือนสำรองในแชตเดียวกัน
พร้อมคำสั่ง `/approve <id> <decision>` ที่ตรงทุกประการ เพื่อให้ยังสามารถดำเนินการคำขอให้เสร็จสิ้นได้

โมเดลทั่วไป:

- นโยบาย exec ของโฮสต์ยังคงเป็นตัวตัดสินว่าจำเป็นต้องอนุมัติ exec หรือไม่
- `approvals.exec` ควบคุมการส่งต่อพรอมต์การอนุมัติไปยังปลายทางแชตอื่น
- `channels.<channel>.execApprovals` ควบคุมว่าจะเปิดใช้งานไคลเอนต์แบบเนทีฟเฉพาะช่องทางของ Discord, Slack, Telegram, QQ bot และ
  ช่องทางที่คล้ายกันหรือไม่
- การอนุมัติ Plugin ของ Slack สามารถใช้ไคลเอนต์การอนุมัติแบบเนทีฟของ Slack เมื่อคำขอมาจาก Slack
  และสามารถแก้ไขผู้อนุมัติ Plugin ของ Slack ได้ นอกจากนี้ `approvals.plugin` ยังสามารถกำหนดเส้นทางการอนุมัติ Plugin ไปยังเซสชัน
  หรือเป้าหมายของ Slack ได้ แม้จะปิดใช้งานการอนุมัติ exec ของ Slack
- การ์ดอนุมัติแบบเนทีฟของ Google Chat จัดการการอนุมัติ exec และ Plugin ที่มาจากสเปซ
  หรือเธรดของ Google Chat เมื่อสามารถแก้ไขผู้อนุมัติ `users/<id>` ที่เสถียรจาก `dm.allowFrom` หรือ
  `defaultTo` ได้ โดยไม่ใช้เหตุการณ์รีแอ็กชันสำหรับการตัดสินใจ
- การส่งการอนุมัติด้วยรีแอ็กชันของ WhatsApp และ Signal ถูกควบคุมโดย `approvals.exec` และ
  `approvals.plugin` โดยไม่มีบล็อก `channels.<channel>.execApprovals`

ไคลเอนต์การอนุมัติแบบเนทีฟจะเปิดใช้การส่งแบบ DM ก่อนโดยอัตโนมัติเมื่อเงื่อนไขทั้งหมดต่อไปนี้เป็นจริง:

- ช่องทางรองรับการส่งการอนุมัติแบบเนทีฟ
- สามารถแก้ไขผู้อนุมัติจาก `execApprovals.approvers` ที่ระบุอย่างชัดเจน หรือข้อมูลประจำตัว
  ของเจ้าของ เช่น `commands.ownerAllowFrom`
- `channels.<channel>.execApprovals.enabled` ไม่ได้ตั้งค่าไว้ หรือเป็น `"auto"`

ตั้งค่า `enabled: false` เพื่อปิดใช้งานไคลเอนต์การอนุมัติแบบเนทีฟอย่างชัดเจน ตั้งค่า `enabled: true` เพื่อบังคับ
เปิดใช้งานเมื่อสามารถแก้ไขผู้อนุมัติได้ การส่งไปยังแชตต้นทางแบบสาธารณะยังคงต้องระบุอย่างชัดเจนผ่าน
`channels.<channel>.execApprovals.target` เมื่อ `target` แบบเนทีฟเปิดใช้การส่งไปยังแชตต้นทาง
พรอมต์การอนุมัติจะมีข้อความคำสั่งรวมอยู่ด้วย

คำถามที่พบบ่อย: [เหตุใดจึงมีการกำหนดค่าการอนุมัติ exec สองรายการสำหรับการอนุมัติผ่านแชต](/help/faq-first-run)

- Discord: `channels.discord.execApprovals.*`
- Slack: `channels.slack.execApprovals.*`
- Telegram: `channels.telegram.execApprovals.*`
- QQ bot: `channels.qqbot.execApprovals.*`
- Google Chat: กำหนดค่าผู้อนุมัติที่เสถียรด้วย `channels.googlechat.dm.allowFrom` หรือ
  `channels.googlechat.defaultTo` โดยไม่จำเป็นต้องมีบล็อก `execApprovals`
- WhatsApp: ใช้ `approvals.exec` และ `approvals.plugin` เพื่อกำหนดเส้นทางพรอมต์การอนุมัติไปยัง WhatsApp
- Signal: ใช้ `approvals.exec` และ `approvals.plugin` เพื่อกำหนดเส้นทางพรอมต์การอนุมัติไปยัง Signal

การกำหนดเส้นทางเฉพาะไคลเอนต์แบบเนทีฟ:

- Telegram ใช้ DM ถึงผู้อนุมัติเป็นค่าเริ่มต้น (`target: "dm"`) เปลี่ยนเป็น `channel` หรือ `both` เพื่อแสดง
  พรอมต์การอนุมัติในแชต/หัวข้อ Telegram ต้นทางด้วย สำหรับหัวข้อฟอรัม Telegram นั้น OpenClaw
  จะคงหัวข้อไว้สำหรับพรอมต์การอนุมัติและการติดตามผลหลังอนุมัติ
- ผู้อนุมัติของ Discord และ Telegram สามารถระบุอย่างชัดเจน (`execApprovals.approvers`) หรืออนุมานจาก
  `commands.ownerAllowFrom` ได้ เฉพาะผู้อนุมัติที่แก้ไขแล้วเท่านั้นที่สามารถอนุมัติหรือปฏิเสธ
- ผู้อนุมัติของ Slack สามารถระบุอย่างชัดเจน (`execApprovals.approvers`) หรืออนุมานจาก
  `commands.ownerAllowFrom` ได้ DM การอนุมัติ Plugin ของ Slack ใช้ผู้อนุมัติ Plugin ของ Slack จาก `allowFrom`
  และการกำหนดเส้นทางเริ่มต้นของบัญชี ไม่ใช่ผู้อนุมัติ exec ของ Slack ปุ่มแบบเนทีฟของ Slack จะเก็บชนิด ID การอนุมัติไว้
  เพื่อให้ ID `plugin:` ดำเนินการการอนุมัติ Plugin ให้เสร็จสิ้นได้โดยไม่ต้องมีชั้นสำรองภายใน Slack ชั้นที่สอง
- การ์ดแบบเนทีฟของ Google Chat จะคงทางเลือกสำรองแบบดำเนินการด้วยตนเอง `/approve` ไว้ในข้อความ แต่คอลแบ็ก
  ของปุ่มการ์ดจะส่งเฉพาะโทเค็นการดำเนินการแบบทึบเท่านั้น โดยระบบจะกู้คืน ID การอนุมัติและการตัดสินใจจาก
  สถานะรอดำเนินการฝั่งเซิร์ฟเวอร์
- การอนุมัติด้วยอีโมจิของ WhatsApp จัดการทั้งพรอมต์ exec และ Plugin เมื่อกลุ่มการส่งต่อระดับบนสุด
  ที่ตรงกันกำหนดเส้นทางไปยัง WhatsApp พรอมต์ที่มาจากระบบแบบเนทีฟจะผูกโดยตรง ส่วนการส่งในโหมดเป้าหมายที่ใช้ร่วมกัน
  จะผูกข้อมูลเมตาการอนุมัติแบบระบุชนิดเดียวกันเข้ากับใบตอบรับข้อความ WhatsApp ที่ยอมรับแล้ว
- การอนุมัติด้วยรีแอ็กชันของ Signal จัดการทั้งพรอมต์ exec และ Plugin เฉพาะเมื่อกลุ่มการส่งต่อระดับบนสุด
  ที่ตรงกันเปิดใช้งานและกำหนดเส้นทางไปยัง Signal การอนุมัติ exec ของ Signal โดยตรงในแชตเดียวกันสามารถ
  ระงับทางเลือกสำรอง `/approve` ภายในเครื่องได้โดยไม่ต้องระบุผู้อนุมัติอย่างชัดเจน แต่การแก้ไขการอนุมัติด้วยรีแอ็กชันของ Signal
  ยังคงต้องมีผู้อนุมัติ Signal ที่ระบุอย่างชัดเจนจาก `channels.signal.allowFrom` หรือ `defaultTo`
- การกำหนดเส้นทาง DM/ช่องทางแบบเนทีฟและทางลัดด้วยรีแอ็กชันของ Matrix จัดการทั้งการอนุมัติ exec และ Plugin
  แต่การให้สิทธิ์ Plugin ยังคงมาจาก `channels.matrix.dm.allowFrom` พรอมต์แบบเนทีฟของ Matrix
  มีเนื้อหาเหตุการณ์แบบกำหนดเอง `com.openclaw.approval` ในเหตุการณ์พรอมต์แรก เพื่อให้ไคลเอนต์ Matrix
  ที่รู้จัก OpenClaw อ่านสถานะการอนุมัติแบบมีโครงสร้างได้ ขณะที่ไคลเอนต์ทั่วไปยังคงใช้ทางเลือกสำรอง
  `/approve` แบบข้อความธรรมดา
- ปุ่มอนุมัติแบบเนทีฟของ Discord และ Telegram มีชนิดเจ้าของแบบ exec หรือ Plugin ที่ระบุชัดเจนใน
  ข้อมูลคอลแบ็กส่วนตัวของการรับส่ง และดำเนินการเฉพาะเจ้าของชนิดนั้น ส่วนตัวควบคุม `/approve` รุ่นเก่าที่ไม่มี
  ชนิดจะยังคงเป็นเส้นทางความเข้ากันได้ที่มีขอบเขตจำกัด โดยจะลองเฉพาะชนิดเจ้าของที่ผู้ดำเนินการมีสิทธิ์อนุมัติ
  ดำเนินการต่อเฉพาะหลังจากผลลัพธ์ไม่พบการอนุมัติ และไม่อนุมานความเป็นเจ้าของจาก ID การอนุมัติ
- ผู้ส่งคำขอไม่จำเป็นต้องเป็นผู้อนุมัติ
- หากไม่มี UI ของผู้ปฏิบัติงานหรือไคลเอนต์การอนุมัติที่กำหนดค่าไว้ซึ่งสามารถรับคำขอได้ พรอมต์จะใช้
  `askFallback` เป็นทางเลือกสำรอง

คำสั่งกลุ่มที่ละเอียดอ่อนและจำกัดเฉพาะเจ้าของ เช่น `/diagnostics` และ `/export-trajectory` ใช้การกำหนดเส้นทาง
ส่วนตัวของเจ้าของสำหรับพรอมต์การอนุมัติและผลลัพธ์สุดท้าย OpenClaw จะลองใช้เส้นทางส่วนตัวบน
พื้นผิวเดียวกับที่เจ้าของเรียกใช้คำสั่งก่อน หากพื้นผิวนั้นไม่มีเส้นทางส่วนตัวของเจ้าของ ระบบจะ
ใช้เส้นทางเจ้าของรายการแรกที่ใช้ได้จาก `commands.ownerAllowFrom` เป็นทางเลือกสำรอง เพื่อให้คำสั่งกลุ่ม Discord
ยังคงส่งการอนุมัติและผลลัพธ์ไปยัง DM Telegram ของเจ้าของได้ เมื่อกำหนดให้ Telegram เป็น
อินเทอร์เฟซส่วนตัวหลัก แชตกลุ่มจะได้รับเพียงข้อความตอบรับสั้น ๆ

### แอปผู้ปฏิบัติงานบนอุปกรณ์เคลื่อนที่อย่างเป็นทางการ

แอป iOS และ Android อย่างเป็นทางการสามารถตรวจสอบการอนุมัติ exec ที่รอดำเนินการและมี Gateway
เป็นเจ้าของได้เช่นกัน เมื่อใช้การเชื่อมต่อ `operator.admin` หรือเมื่ออุปกรณ์
`operator.approvals` ที่จับคู่ไว้ถูกกำหนดเป็นเป้าหมายของคำขออย่างชัดเจน แอปจะอ่าน
ระเบียนถาวรที่ผ่านการล้างข้อมูลชุดเดียวกับที่ Control UI ใช้
ส่งการตัดสินใจที่รับรู้ชนิด และแสดงผลลัพธ์คำตอบแรกตามรูปแบบมาตรฐานของ Gateway
Apple Watch จะแสดงพรอมต์การอนุมัติเหล่านี้ผ่าน iPhone ที่จับคู่ไว้
พร้อมการดำเนินการอนุญาตครั้งเดียวและปฏิเสธ โหมด Gateway โดยตรงของ Watch
จะไม่ตรวจสอบการอนุมัติ

การสูญหายของข้อความตอบรับการดำเนินการไม่ทำให้ตัวเลือกที่ส่งไปมีอำนาจเด็ดขาด:
แอปจะปิดใช้งานตัวควบคุมและอ่านระเบียนอีกครั้ง หากพื้นผิวอื่น
ดำเนินการสำเร็จก่อน แอปจะแสดงการตัดสินใจที่บันทึกไว้นั้น พรอมต์ที่รอดำเนินการยังคงผูกกับ
Gateway ที่ออกพรอมต์ ดังนั้นการสลับ Gateway ที่ใช้งานอยู่จึงไม่สามารถเปลี่ยนเส้นทาง
ID การอนุมัติเก่าได้

### โฟลว์ IPC ของ macOS

```
Gateway -> บริการ Node (WS)
                 |  IPC (UDS + โทเค็น + HMAC + TTL)
                 v
             แอป Mac (UI + การอนุมัติ + system.run)
```

หมายเหตุด้านความปลอดภัย:

- โหมดซ็อกเก็ต Unix `0600` โดยจัดเก็บโทเค็นไว้ใน `exec-approvals.json`
- การตรวจสอบเพียร์ที่มี UID เดียวกัน
- กลไกถามตอบ (nonce + โทเค็น HMAC + แฮชคำขอ) + TTL ระยะสั้น

## คำถามที่พบบ่อย

### เมื่อใดจึงควรใช้ `accountId` และ `threadId` กับเป้าหมายการอนุมัติ

ใช้ `accountId` เมื่อช่องทางมีข้อมูลประจำตัวที่กำหนดค่าไว้หลายรายการและพรอมต์การอนุมัติต้อง
ส่งออกผ่านบัญชีใดบัญชีหนึ่งโดยเฉพาะ ใช้ `threadId` เมื่อปลายทางรองรับหัวข้อหรือ
เธรด และควรให้พรอมต์อยู่ภายในเธรดนั้นแทนแชตระดับบนสุด

กรณีตัวอย่างของ Telegram คือซูเปอร์กรุ๊ปปฏิบัติการที่มีหัวข้อฟอรัมและบัญชีบอต Telegram
สองบัญชี ค่า `to` ระบุชื่อซูเปอร์กรุ๊ป `accountId` เลือกบัญชีบอต และ `threadId`
เลือกหัวข้อฟอรัม:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "targets",
      targets: [
        {
          channel: "telegram",
          to: "-1001234567890",
          accountId: "ops-bot",
          threadId: "77",
        },
      ],
    },
  },
  channels: {
    telegram: {
      accounts: {
        default: {
          name: "บอตหลัก",
          botToken: "env:TELEGRAM_PRIMARY_BOT_TOKEN",
        },
        "ops-bot": {
          name: "บอตปฏิบัติการ",
          botToken: "env:TELEGRAM_OPS_BOT_TOKEN",
        },
      },
    },
  },
}
```

ด้วยการตั้งค่านี้ การอนุมัติ exec ที่ส่งต่อจะถูกโพสต์โดยบัญชี Telegram `ops-bot` ไปยังหัวข้อ
`77` ของแชต `-1001234567890` เป้าหมายที่ไม่มี `accountId` จะใช้บัญชีเริ่มต้นของช่องทาง และ
เป้าหมายที่ไม่มี `threadId` จะโพสต์ไปยังปลายทางระดับบนสุด

### เมื่อส่งคำขออนุมัติไปยังเซสชัน ทุกคนในเซสชันนั้นสามารถอนุมัติได้หรือไม่?

ไม่ได้ การส่งไปยังเซสชันควบคุมเพียงตำแหน่งที่พรอมต์ปรากฏเท่านั้น โดยตัวมันเองไม่ได้ให้อำนาจ
ผู้เข้าร่วมทุกคนในแชตนั้นในการอนุมัติ

สำหรับ `/approve` ทั่วไปภายในแชตเดียวกัน ผู้ส่งต้องได้รับอนุญาตให้ใช้คำสั่งใน
เซสชันของช่องนั้นอยู่แล้ว หากช่องระบุผู้มีสิทธิ์อนุมัติอย่างชัดเจน ผู้มีสิทธิ์เหล่านั้นสามารถอนุมัติ
การดำเนินการ `/approve` ได้ แม้จะไม่ได้รับอนุญาตให้ใช้คำสั่งในเซสชันนั้นก็ตาม

บางช่องมีข้อกำหนดเข้มงวดกว่า Discord, Telegram, Matrix, DM การอนุมัติแบบเนทีฟของ Slack และ
ไคลเอนต์การอนุมัติแบบเนทีฟที่คล้ายกันจะใช้รายชื่อผู้มีสิทธิ์อนุมัติที่ระบบตรวจหาได้เพื่อกำหนดสิทธิ์การอนุมัติ ตัวอย่างเช่น
พรอมต์ขออนุมัติในหัวข้อฟอรัม Telegram อาจปรากฏให้ทุกคนในหัวข้อนั้นเห็น แต่มีเพียง ID ผู้ใช้ Telegram
ที่เป็นตัวเลขซึ่งตรวจหาได้จาก `channels.telegram.execApprovals.approvers` หรือ
`commands.ownerAllowFrom` เท่านั้นที่สามารถอนุมัติหรือปฏิเสธได้
