OpenClaw สามารถช่วยป้องกันไม่ให้ agent ติดอยู่ในรูปแบบการเรียกเครื่องมือซ้ำ ๆ ได้
ตัวป้องกันนี้**ปิดใช้งานโดยค่าเริ่มต้น**

เปิดใช้เฉพาะจุดที่จำเป็น เพราะเมื่อตั้งค่าเข้มงวด อาจบล็อกการเรียกซ้ำที่ถูกต้องตามปกติได้

## เหตุผลที่มีฟีเจอร์นี้

- ตรวจจับลำดับซ้ำ ๆ ที่ไม่ทำให้เกิดความคืบหน้า
- ตรวจจับลูปที่ไม่มีผลลัพธ์ด้วยความถี่สูง (เครื่องมือเดิม อินพุตเดิม ข้อผิดพลาดซ้ำ)
- ตรวจจับรูปแบบการเรียกซ้ำเฉพาะสำหรับเครื่องมือ polling ที่รู้จัก

## บล็อกการกำหนดค่า

ค่าเริ่มต้นแบบสากล:

```json5
{
  tools: {
    loopDetection: {
      enabled: false,
      historySize: 30,
      warningThreshold: 10,
      criticalThreshold: 20,
      globalCircuitBreakerThreshold: 30,
      detectors: {
        genericRepeat: true,
        knownPollNoProgress: true,
        pingPong: true,
      },
    },
  },
}
```

การแทนที่ค่าต่อ agent (ไม่บังคับ):

```json5
{
  agents: {
    list: [
      {
        id: "safe-runner",
        tools: {
          loopDetection: {
            enabled: true,
            warningThreshold: 8,
            criticalThreshold: 16,
          },
        },
      },
    ],
  },
}
```

### พฤติกรรมของฟิลด์

- `enabled`: สวิตช์หลัก `false` หมายถึงไม่มีการตรวจจับลูป
- `historySize`: จำนวนการเรียกเครื่องมือล่าสุดที่เก็บไว้สำหรับการวิเคราะห์
- `warningThreshold`: เกณฑ์ก่อนจัดประเภทรูปแบบเป็นระดับเตือนเท่านั้น
- `criticalThreshold`: เกณฑ์สำหรับบล็อกรูปแบบลูปซ้ำ
- `globalCircuitBreakerThreshold`: เกณฑ์ตัวตัดวงจรแบบสากลเมื่อไม่มีความคืบหน้า
- `detectors.genericRepeat`: ตรวจจับรูปแบบเครื่องมือเดิม + พารามิเตอร์เดิมซ้ำ
- `detectors.knownPollNoProgress`: ตรวจจับรูปแบบที่คล้าย polling ที่รู้จักซึ่งไม่มีการเปลี่ยนสถานะ
- `detectors.pingPong`: ตรวจจับรูปแบบ ping-pong แบบสลับไปมา

สำหรับ `exec` การตรวจสอบว่าไม่มีความคืบหน้าจะเปรียบเทียบผลลัพธ์คำสั่งที่เสถียร และละเว้นเมตาดาต้ารันไทม์ที่เปลี่ยนแปลงได้ เช่น ระยะเวลา, PID, session ID และไดเรกทอรีทำงาน
เมื่อมี run id ประวัติการเรียกเครื่องมือล่าสุดจะถูกประเมินเฉพาะภายในรันนั้น เพื่อให้รอบ Heartbeat ที่ตั้งเวลาไว้และรันใหม่ไม่สืบทอดจำนวนลูปเก่าจากรันก่อนหน้า

## การตั้งค่าที่แนะนำ

- สำหรับโมเดลขนาดเล็ก ให้เริ่มด้วย `enabled: true` และไม่เปลี่ยนค่าเริ่มต้น โมเดลเรือธงมักแทบไม่ต้องใช้การตรวจจับลูป และสามารถปล่อยให้ปิดใช้งานไว้ได้
- รักษาลำดับเกณฑ์เป็น `warningThreshold < criticalThreshold < globalCircuitBreakerThreshold`
- หากเกิด false positive:
  - เพิ่ม `warningThreshold` และ/หรือ `criticalThreshold`
  - (ไม่บังคับ) เพิ่ม `globalCircuitBreakerThreshold`
  - ปิดใช้เฉพาะตัวตรวจจับที่ทำให้เกิดปัญหา
  - ลด `historySize` เพื่อให้บริบทประวัติเข้มงวดน้อยลง

## ตัวป้องกันหลัง Compaction

เมื่อ runner ทำ auto-compaction-retry เสร็จแล้ว (หลัง context-overflow) ระบบจะติดอาวุธตัวป้องกันช่วงสั้น ๆ ที่เฝ้าดูการเรียกเครื่องมือถัดไปไม่กี่ครั้ง หาก agent ส่งทริปเปิล `(toolName, args, result)` _เดียวกัน_ หลายครั้งภายในช่วงนั้น ตัวป้องกันจะสรุปว่า Compaction ไม่ได้หยุดลูป และยกเลิกรันด้วยข้อผิดพลาด `compaction_loop_persisted`

นี่เป็นเส้นทางโค้ดที่แยกจากตัวตรวจจับ `tools.loopDetection` แบบสากล และกำหนดค่าได้อย่างอิสระ:

```json5
{
  tools: {
    loopDetection: {
      enabled: true, // existing master switch; set false to disable loop guards
      postCompactionGuard: {
        windowSize: 3, // default: 3
      },
    },
  },
}
```

- `windowSize`: จำนวนการเรียกเครื่องมือหลัง Compaction ที่ตัวป้องกันยังติดอาวุธอยู่ _และ_ จำนวนทริปเปิล (tool, args, result) ที่เหมือนกันซึ่งจะทริกเกอร์การยกเลิก

ตัวป้องกันจะไม่ยกเลิกเมื่อผลลัพธ์กำลังเปลี่ยนแปลง แต่จะยกเลิกเฉพาะเมื่อผลลัพธ์เหมือนกันระดับไบต์ตลอดทั้งช่วงเท่านั้น โดยตั้งใจให้มีขอบเขตแคบ: จะทำงานเฉพาะทันทีหลัง compaction-retry

## ล็อกและพฤติกรรมที่คาดหวัง

เมื่อตรวจพบลูป OpenClaw จะรายงานเหตุการณ์ลูป และบล็อกหรือลดแรงรอบเครื่องมือถัดไปตามระดับความรุนแรง
สิ่งนี้ช่วยปกป้องผู้ใช้จากการใช้โทเค็นอย่างควบคุมไม่ได้และการค้าง ขณะยังรักษาการเข้าถึงเครื่องมือตามปกติไว้

- ควรใช้การเตือนและการระงับชั่วคราวก่อน
- ยกระดับเฉพาะเมื่อมีหลักฐานซ้ำสะสมมากขึ้น

## หมายเหตุ

- `tools.loopDetection` จะถูกผสานกับการแทนที่ค่าระดับ agent
- การกำหนดค่าต่อ agent จะแทนที่หรือขยายค่าระดับสากลอย่างเต็มรูปแบบ
- หากไม่มีการกำหนดค่า guardrail จะยังปิดอยู่
