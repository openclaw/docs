เรียกใช้คำสั่งเชลล์ในพื้นที่ทำงาน `exec` เป็นพื้นผิวเชลล์ที่แก้ไขข้อมูลได้: คำสั่งสามารถสร้าง แก้ไข หรือลบไฟล์ได้ทุกตำแหน่งที่ระบบไฟล์ของโฮสต์หรือแซนด์บ็อกซ์ที่เลือกอนุญาต การปิดใช้งานเครื่องมือระบบไฟล์ของ OpenClaw เช่น `write`, `edit` หรือ `apply_patch` ไม่ได้ทำให้ `exec` เป็นแบบอ่านอย่างเดียว

รองรับการดำเนินการทั้งเบื้องหน้าและเบื้องหลังผ่าน `process` หากไม่อนุญาต `process` ระบบจะเรียกใช้ `exec` แบบซิงโครนัสและไม่สนใจ `yieldMs`/`background` เซสชันเบื้องหลังมีขอบเขตแยกตามเอเจนต์ โดย `process` จะเห็นเฉพาะเซสชันจากเอเจนต์เดียวกัน

## พารามิเตอร์

<ParamField path="command" type="string" required>
คำสั่งเชลล์ที่จะเรียกใช้
</ParamField>

<ParamField path="workdir" type="string" default="cwd">
ไดเรกทอรีทำงานสำหรับคำสั่ง
</ParamField>

<ParamField path="env" type="object">
ค่าทดแทนสภาพแวดล้อมแบบคีย์/ค่าที่ผสานทับสภาพแวดล้อมที่สืบทอดมา
</ParamField>

<ParamField path="yieldMs" type="number" default="10000">
ย้ายคำสั่งไปทำงานเบื้องหลังโดยอัตโนมัติหลังจากระยะเวลาหน่วงนี้ (ms)
</ParamField>

<ParamField path="background" type="boolean" default="false">
ย้ายคำสั่งไปทำงานเบื้องหลังทันทีแทนการรอ `yieldMs`
</ParamField>

<ParamField path="timeout" type="number" default="tools.exec.timeoutSec">
แทนที่ระยะหมดเวลาของ exec ที่กำหนดค่าไว้สำหรับการเรียกครั้งนี้ หน่วยเป็นวินาที มีผลกับการดำเนินการเบื้องหน้า เบื้องหลัง `yieldMs`, gateway, แซนด์บ็อกซ์ และ Node `system.run` โดย `timeout: 0` จะปิดใช้งานระยะหมดเวลาของกระบวนการ exec สำหรับการเรียกครั้งนั้น
</ParamField>

<ParamField path="pty" type="boolean" default="false">
เรียกใช้ในเทอร์มินัลจำลองเมื่อพร้อมใช้งาน ใช้สำหรับ CLI ที่ทำงานเฉพาะกับ TTY, เอเจนต์เขียนโค้ด และ UI เทอร์มินัล
</ParamField>

<ParamField path="host" type="'auto' | 'sandbox' | 'gateway' | 'node'" default="auto">
ตำแหน่งที่จะดำเนินการ `auto` จะถูกแก้ค่าเป็น `sandbox` เมื่อรันไทม์แซนด์บ็อกซ์ทำงานอยู่ และเป็น `gateway` ในกรณีอื่น
</ParamField>

<ParamField path="security" type="'deny' | 'allowlist' | 'full'">
ไม่มีผลสำหรับการเรียกเครื่องมือตามปกติ ความปลอดภัยของ `gateway`/`node` ถูกควบคุมโดย `tools.exec.security` และไฟล์การอนุมัติของโฮสต์ โหมดยกระดับสามารถบังคับใช้ `security=full` ได้เฉพาะเมื่อผู้ดำเนินการให้สิทธิ์การเข้าถึงแบบยกระดับอย่างชัดเจน
</ParamField>

<ParamField path="ask" type="'off' | 'on-miss' | 'always'">
โหมดการขออนุมัติพื้นฐานมาจาก `tools.exec.ask` และการอนุมัติของโฮสต์ สำหรับการเรียกโมเดลที่มาจากช่องทาง ค่า `ask` รายการเรียกจะถูกละเว้นเมื่อการขออนุมัติที่มีผลของโฮสต์เป็น `off` มิฉะนั้น ค่านี้ทำได้เพียงเพิ่มความเข้มงวดเป็นโหมดที่เข้มกว่า ผู้เรียกภายใน/API ที่เชื่อถือได้ซึ่งสร้างเครื่องมือ exec ด้วยค่า `ask` ที่ระบุไว้อย่างชัดเจนจะไม่เปลี่ยนแปลง
</ParamField>

<ParamField path="node" type="string">
รหัส/ชื่อ Node เมื่อ `host=node`
</ParamField>

<ParamField path="elevated" type="boolean" default="false">
ขอโหมดยกระดับ: ออกจากแซนด์บ็อกซ์ไปยังพาธโฮสต์ที่กำหนดค่าไว้ `security=full` จะถูกบังคับใช้เฉพาะเมื่อค่ายกระดับถูกแก้เป็น `full`
</ParamField>

หมายเหตุ:

- `host` ยอมรับเฉพาะ `auto`, `sandbox`, `gateway` หรือ `node` เท่านั้น ค่านี้ไม่ใช่ตัวเลือกชื่อโฮสต์ และค่าที่มีลักษณะเหมือนชื่อโฮสต์จะถูกปฏิเสธก่อนเรียกใช้คำสั่ง
- อนุญาต `host=node` ต่อการเรียกจาก `auto` ส่วน `host=gateway` ต่อการเรียกจะได้รับอนุญาตเฉพาะเมื่อไม่มีรันไทม์แซนด์บ็อกซ์ทำงานอยู่
- แม้ไม่มีการกำหนดค่าเพิ่มเติม `host=auto` ก็ยัง “ใช้งานได้ทันที”: หากไม่มีแซนด์บ็อกซ์ ค่าจะถูกแก้เป็น `gateway` ส่วนเมื่อมีแซนด์บ็อกซ์ที่ทำงานอยู่ ค่าจะยังคงอยู่ในแซนด์บ็อกซ์
- `elevated` ออกจากแซนด์บ็อกซ์ไปยังพาธโฮสต์ที่กำหนดค่าไว้ ซึ่งโดยค่าเริ่มต้นคือ `gateway` หรือเป็น `node` เมื่อ `tools.exec.host=node` (หรือค่าเริ่มต้นของเซสชันเป็น `host=node`) ตัวเลือกนี้พร้อมใช้งานเฉพาะเมื่อเปิดใช้การเข้าถึงแบบยกระดับสำหรับเซสชัน/ผู้ให้บริการปัจจุบัน
- การอนุมัติ `gateway`/`node` ถูกควบคุมโดยไฟล์การอนุมัติของโฮสต์
- `node` ต้องใช้ Node ที่จับคู่แล้ว (แอปคู่หูหรือโฮสต์ Node แบบไม่มีส่วนติดต่อผู้ใช้) หากมีหลาย Node ให้ตั้งค่า `exec.node` หรือ `tools.exec.node` เพื่อเลือกหนึ่งรายการ
- `exec host=node` เป็นพาธเดียวสำหรับการเรียกใช้เชลล์บน Node โดยนำ wrapper รุ่นเก่า `nodes.run` ออกแล้ว
- บนโฮสต์ที่ไม่ใช่ Windows exec จะใช้ `SHELL` เมื่อตั้งค่าไว้ หาก `SHELL` เป็น `fish` ระบบจะเลือกใช้ `bash` (หรือ `sh`) จาก `PATH` ก่อน เพื่อหลีกเลี่ยงไวยากรณ์เฉพาะ bash ที่ใช้กับ fish ไม่ได้ จากนั้นจึงถอยกลับไปใช้ `SHELL` หากไม่พบทั้งสองรายการ
- บนโฮสต์ Windows exec จะค้นหาและเลือกใช้ PowerShell 7 (`pwsh`) ก่อน (Program Files, ProgramW6432 แล้วจึง PATH) จากนั้นจึงถอยกลับไปใช้ Windows PowerShell 5.1
- บนโฮสต์ gateway ที่ไม่ใช่ Windows คำสั่ง exec ของ bash และ zsh จะใช้สแนปช็อตเริ่มต้น OpenClaw จะบันทึก alias/ฟังก์ชันที่นำเข้าได้และชุดสภาพแวดล้อมขนาดเล็กที่ปลอดภัยจากไฟล์เริ่มต้นของเชลล์ลงใน `$OPENCLAW_STATE_DIR/cache/shell-snapshots/` จากนั้นจะนำเข้าสแนปช็อตดังกล่าวก่อนคำสั่ง exec แต่ละคำสั่ง ตัวแปรที่ดูเหมือนมีข้อมูลลับจะถูกตัดออก การเรียก exec ในแซนด์บ็อกซ์และ Node จะไม่ใช้สแนปช็อตนี้ ตั้งค่า `OPENCLAW_EXEC_SHELL_SNAPSHOT=0` ในสภาพแวดล้อมของกระบวนการ Gateway เพื่อปิดใช้งานพาธสแนปช็อตนี้
- การดำเนินการบนโฮสต์ (`gateway`/`node`) จะปฏิเสธ `env.PATH` และค่าทดแทนของ loader (`LD_*`/`DYLD_*`) เพื่อป้องกันการยึดไบนารีหรือการแทรกโค้ด
- OpenClaw ตั้งค่า `OPENCLAW_SHELL=exec` ในสภาพแวดล้อมของคำสั่งที่สร้างขึ้น (รวมถึงการดำเนินการผ่าน PTY และแซนด์บ็อกซ์) เพื่อให้กฎของเชลล์/โปรไฟล์ตรวจจับบริบทของเครื่องมือ exec ได้
- สำหรับการเรียกใช้ที่มาจากช่องทาง OpenClaw ยังเปิดเผยเพย์โหลด JSON แบบจำกัดที่ระบุตัวตนของผู้ส่ง/แชตไว้ใน `OPENCLAW_CHANNEL_CONTEXT` เมื่อช่องทางให้รหัสเหล่านั้นมา
- `exec` ไม่สามารถเรียกใช้คำสั่งเชลล์ `openclaw channels login` หรือ `/approve` ได้: `openclaw channels login` เป็นโฟลว์การยืนยันตัวตนช่องทางแบบโต้ตอบ และ `/approve` ต้องดำเนินการผ่านตัวจัดการคำสั่งอนุมัติ ไม่ใช่เชลล์ ให้เข้าสู่ระบบช่องทางในเทอร์มินัลบนโฮสต์ gateway หรือใช้เครื่องมือเอเจนต์สำหรับเข้าสู่ระบบของช่องทางนั้นโดยเฉพาะเมื่อมีให้ใช้ (เช่น `whatsapp_login`)
- สำคัญ: การใช้แซนด์บ็อกซ์ **ปิดอยู่โดยค่าเริ่มต้น** หากปิดการใช้แซนด์บ็อกซ์ ค่า `host=auto` โดยนัยจะถูกแก้เป็น `gateway` ส่วน `host=sandbox` ที่ระบุอย่างชัดเจนจะยังคงล้มเหลวแบบปิด แทนที่จะเรียกใช้บนโฮสต์ gateway โดยเงียบ ๆ เปิดใช้แซนด์บ็อกซ์หรือใช้ `host=gateway` พร้อมการอนุมัติ
- การตรวจสอบล่วงหน้าของสคริปต์ (สำหรับข้อผิดพลาดทั่วไปในไวยากรณ์เชลล์ของ Python/Node) จะตรวจสอบเฉพาะไฟล์ภายในขอบเขต `workdir` ที่มีผล หากพาธสคริปต์ถูกแก้ไปยังตำแหน่งนอก `workdir` ระบบจะข้ามการตรวจสอบล่วงหน้าสำหรับไฟล์นั้น และจะข้ามการตรวจสอบล่วงหน้าทั้งหมดเมื่อ `host=gateway` และนโยบายที่มีผลเป็น `security=full` พร้อม `ask=off`
- สำหรับงานที่ใช้เวลานานและเริ่มตอนนี้ ให้เริ่มเพียงครั้งเดียวและอาศัยการปลุกเมื่อเสร็จสิ้นโดยอัตโนมัติเมื่อเปิดใช้และคำสั่งมีเอาต์พุตหรือล้มเหลว ใช้ `process` สำหรับบันทึก สถานะ อินพุต หรือการแทรกแซง อย่าจำลองการจัดกำหนดการด้วยลูป sleep, ลูป timeout หรือการตรวจสอบซ้ำ ๆ
- สำหรับงานที่ควรเกิดขึ้นภายหลังหรือตามกำหนดเวลา ให้ใช้ Cron แทนรูปแบบ sleep/หน่วงเวลาของ `exec`

| `tools.exec.safeBins`                | ไม่ได้ตั้งค่า                                                  | ไบนารีที่ปลอดภัยและรับข้อมูลผ่าน stdin เท่านั้น ซึ่งเรียกใช้ได้โดยไม่ต้องมีรายการ allowlist ที่ระบุชัดเจน ดู[ไบนารีที่ปลอดภัย](/th/tools/exec-approvals-advanced#safe-bins-stdin-only)         |
| `tools.exec.safeBinTrustedDirs`      | `/bin`, `/usr/bin`                                     | ไดเรกทอรีเพิ่มเติมที่เชื่อถืออย่างชัดเจนสำหรับการตรวจสอบพาธ `safeBins` รายการ `PATH` จะไม่ได้รับความเชื่อถือโดยอัตโนมัติ                                              |
| `tools.exec.safeBinProfiles`         | ไม่ได้ตั้งค่า                                                  | นโยบาย argv แบบกำหนดเองต่อไบนารีที่ปลอดภัย (`minPositional`, `maxPositional`, `allowedValueFlags`, `deniedFlags`)                                        |

การเรียกใช้ exec บนโฮสต์โดยไม่ต้องอนุมัติเป็นค่าเริ่มต้นสำหรับ Gateway และ Node (`security=full`, `ask=off`) ซึ่งมาจากค่าเริ่มต้นของนโยบายโฮสต์ ไม่ใช่จาก `host=auto` หากต้องการพฤติกรรมการอนุมัติ/allowlist ให้เพิ่มความเข้มงวดทั้ง `tools.exec.*` และไฟล์การอนุมัติของโฮสต์ ดู[การอนุมัติ Exec](/th/tools/exec-approvals#yolo-mode-no-approval) หากต้องการบังคับการกำหนดเส้นทางผ่าน Gateway หรือ Node โดยไม่ขึ้นกับสถานะแซนด์บ็อกซ์ ให้ตั้งค่า `tools.exec.host` หรือใช้ `/exec host=...`

ตัวอย่าง:

```json5
{
  tools: {
    exec: {
      pathPrepend: ["~/bin", "/opt/oss/bin"],
    },
  },
}
```

### โหมด

`tools.exec.mode` คือตัวควบคุมนโยบายที่ปรับให้อยู่ในรูปมาตรฐาน การตั้งค่านี้จะกำหนด `security`/`ask` และไม่สามารถใช้ร่วมกับการกำหนด `tools.exec.security`/`tools.exec.ask` อย่างชัดเจน

| โหมด        | ความปลอดภัย    | การถาม       | พฤติกรรม                                                                                                                       |
| ----------- | ----------- | --------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `deny`      | `deny`      | `off`     | ปฏิเสธ Exec                                                                                                                |
| `allowlist` | `allowlist` | `off`     | เรียกใช้เฉพาะคำสั่งที่อยู่ใน allowlist/เป็นไบนารีที่ปลอดภัย โดยไม่ถามสำหรับคำสั่งอื่น                                                                 |
| `ask`       | `allowlist` | `on-miss` | รายการที่ตรงกับ allowlist จะทำงานโดยตรง ส่วนรายการอื่นทั้งหมดจะถามมนุษย์                                                                  |
| `auto`      | `allowlist` | `on-miss` | รายการที่ตรงกับ allowlist/ไบนารีที่ปลอดภัยจะทำงานโดยตรง ส่วนรายการอื่นทั้งหมดจะผ่านผู้ตรวจสอบอัตโนมัติแบบเนทีฟของ OpenClaw ก่อนถามมนุษย์ |
| `full`      | `full`      | `off`     | ไม่มีด่านอนุมัติ                                                                                                              |

`ask`/`ask=always` ยังคงถามมนุษย์ทุกครั้งโดยไม่ขึ้นกับโหมด

การอนุมัติจากการตรวจสอบอัตโนมัติใช้ได้ครั้งเดียว บน Gateway นั้น OpenClaw จะส่งพาธไฟล์ปฏิบัติการที่แก้ค่าแล้วให้ผู้ตรวจสอบและตรึงการเรียกใช้ไว้กับพาธเดียวกันนั้น คำสั่งที่ไม่สามารถย่อเป็นแผนการเรียกใช้เพียงแผนเดียวที่บังคับใช้ได้ เช่น heredoc, การขยายค่าของเชลล์ หรือการใส่เครื่องหมายคำพูดของตัวห่อหุ้มที่ไม่รองรับ จะเปลี่ยนไปใช้การอนุมัติจากมนุษย์ แม้ว่าในกรณีอื่นโมเดลจะอนุญาตก็ตาม

การอนุมัติคำสั่งของแอปเซิร์ฟเวอร์ Codex ที่ยังไม่ได้รับการตัดสินโดยรันไทม์ที่ระบุชัดเจนหรือนโยบายแบบเนทีฟจะใช้เส้นทางการอนุมัติจากมนุษย์ OpenClaw จะไม่เรียกใช้ผู้ตรวจสอบ exec ที่กำหนดค่าไว้สำหรับคำขอเหล่านี้ เนื่องจาก Codex ไม่เปิดเผยไฟล์ปฏิบัติการที่แก้ค่าแล้วและบังคับใช้ได้ ซึ่งสามารถผูกการตัดสินใจของผู้ตรวจสอบเข้ากับคำสั่งที่ Codex เรียกใช้

### การประเมินค่าแบบอินไลน์ (`strictInlineEval`)

เมื่อ `tools.exec.strictInlineEval` เป็น `true` รูปแบบการประเมินค่าของอินเทอร์พรีเตอร์แบบอินไลน์ต้องผ่านผู้ตรวจสอบหรือการอนุมัติอย่างชัดเจน ได้แก่ `python -c`, `node -e`, `ruby -e`, `perl -e`, `php -r`, `lua -e`, `osascript -e` และรูปแบบที่คล้ายกันในอินเทอร์พรีเตอร์และตัวนำส่งคำสั่งอื่นที่รองรับ (`awk`, `find -exec`, `make`, `sed`, `xargs` และอื่น ๆ) ใน `mode=auto` เส้นทางการอนุมัติ exec ปกติอาจเปิดทางให้ผู้ตรวจสอบอัตโนมัติแบบเนทีฟอนุญาตคำสั่งแบบครั้งเดียวที่เห็นได้ชัดว่ามีความเสี่ยงต่ำ ส่วนการเรียก `system.run` โดยตรงบนโฮสต์ Node ยังคงต้องได้รับการอนุมัติอย่างชัดเจน เพราะไม่สามารถส่งคำสั่งไปยังเส้นทางการอนุมัติจากมนุษย์ได้ หากผู้ตรวจสอบร้องขอ คำขอจะถูกส่งให้มนุษย์ `allow-always` ยังคงสามารถบันทึกการเรียกอินเทอร์พรีเตอร์/สคริปต์ที่ไม่เป็นอันตรายอย่างถาวรได้ แต่รูปแบบการประเมินค่าแบบอินไลน์จะไม่กลายเป็นกฎอนุญาตแบบถาวร

### การจัดการ PATH

- `host=gateway`: ผสาน `PATH` ของล็อกอินเชลล์เข้ากับสภาพแวดล้อม exec การแทนที่ `env.PATH` จะถูกปฏิเสธสำหรับการเรียกใช้บนโฮสต์ ตัวดีมอนเองยังคงทำงานด้วย `PATH` ขั้นต่ำ:
  - macOS: `/opt/homebrew/bin`, `/usr/local/bin`, `/usr/bin`, `/bin`
  - Linux: `/usr/local/bin`, `/usr/bin`, `/bin`
  - เพื่อป้องกันไม่ให้การกำหนดค่าเชลล์ของผู้ใช้ (เช่น `~/.zshenv` หรือ `/etc/zshenv`) แทนที่พาธที่มีลำดับความสำคัญระหว่างการเริ่มต้น ระบบจะเติมรายการ `tools.exec.pathPrepend` ไว้หน้า `PATH` สุดท้ายภายในคำสั่งเชลล์อย่างปลอดภัยก่อนเรียกใช้ทันที
- `host=sandbox`: เรียกใช้ `sh -lc` (ล็อกอินเชลล์) ภายในคอนเทนเนอร์ ดังนั้น `/etc/profile` อาจรีเซ็ต `PATH` OpenClaw จะเติม `env.PATH` ไว้หน้าหลังโหลดโปรไฟล์ผ่านตัวแปรสภาพแวดล้อมภายใน (ไม่มีการแทนค่าของเชลล์) และ `tools.exec.pathPrepend` ใช้กับกรณีนี้ด้วย
- `host=node`: ระบบจะส่งเฉพาะการแทนที่สภาพแวดล้อมที่ไม่ถูกบล็อกซึ่งส่งเข้ามาไปยัง Node การแทนที่ `env.PATH` จะถูกปฏิเสธสำหรับการเรียกใช้บนโฮสต์และถูกโฮสต์ Node เพิกเฉย หากต้องการรายการ PATH เพิ่มเติมบน Node ให้กำหนดค่าสภาพแวดล้อมของบริการโฮสต์ Node (systemd/launchd) หรือติดตั้งเครื่องมือในตำแหน่งมาตรฐาน

การผูก Node ต่อเอเจนต์ (ใช้ดัชนีรายการเอเจนต์ในการกำหนดค่า):

```bash
openclaw config get agents.list
openclaw config set 'agents.list[0].tools.exec.node' "node-id-or-name"
```

Control UI: หน้า **อุปกรณ์** มีแผง "การผูก Node สำหรับ Exec" ขนาดเล็กสำหรับการตั้งค่าเดียวกัน

## การแทนที่ระดับเซสชัน (`/exec`)

ใช้ `/exec` เพื่อตั้งค่าเริ่มต้น **ต่อเซสชัน** สำหรับ `host`, `security`, `ask` และ `node` ส่ง `/exec` โดยไม่มีอาร์กิวเมนต์เพื่อแสดงค่าปัจจุบัน

ตัวอย่าง:

```text
/exec host=auto security=allowlist ask=on-miss node=mac-1
```

ระบบจะยอมรับ `/exec` เฉพาะจาก **ผู้ส่งที่ได้รับอนุญาต** (allowlist/การจับคู่ของช่องทาง รวมถึง `commands.useAccessGroups`) เท่านั้น โดยจะอัปเดตเฉพาะ **สถานะเซสชัน** และไม่เขียนการกำหนดค่า ผู้ส่งจากช่องทางภายนอกที่ได้รับอนุญาตสามารถตั้งค่าเริ่มต้นของเซสชันเหล่านี้ได้ ส่วนไคลเอนต์ Gateway/Webchat ภายในต้องมี `operator.admin` จึงจะบันทึกค่าเหล่านี้อย่างถาวรได้

หากต้องการปิดใช้ exec อย่างเด็ดขาด ให้ปฏิเสธผ่านนโยบายเครื่องมือ (`tools.deny: ["exec"]` หรือต่อเอเจนต์) การอนุมัติของโฮสต์ยังคงมีผล เว้นแต่จะตั้งค่า `security=full` และ `ask=off` อย่างชัดเจน

## การอนุมัติ Exec (แอปคู่หู/โฮสต์ Node)

เอเจนต์ในแซนด์บ็อกซ์สามารถกำหนดให้ต้องอนุมัติต่อคำขอก่อนที่ `exec` จะทำงานบน Gateway หรือโฮสต์ Node ดูนโยบาย, allowlist และลำดับงาน UI ที่[การอนุมัติ Exec](/th/tools/exec-approvals)

เมื่อต้องได้รับการอนุมัติจากมนุษย์ ขั้นตอนของโฮสต์ Node และ Gateway ที่ไม่ใช่แบบเนทีฟจะส่งคืน `status: "approval-pending"` พร้อมรหัสการอนุมัติทันที ส่วนขั้นตอน Gateway ของแชตแบบเนทีฟและ Web UI สามารถรอในบรรทัดและส่งคืนผลลัพธ์สุดท้ายของคำสั่งหลังได้รับการอนุมัติได้ ผลลัพธ์ `approval-pending` หมายความว่าคำสั่งยังไม่เริ่มทำงาน ดังนั้นคำเตือนการเปลี่ยนไปทำงานเบื้องหน้าจะปรากฏเฉพาะเมื่อคำสั่งที่ได้รับอนุมัติทำงานในบรรทัดจริง การทำงานแบบอะซิงโครนัสที่ได้รับอนุมัติจะส่งเหตุการณ์ระบบเกี่ยวกับความคืบหน้าและการเสร็จสิ้นของคำสั่ง (`Exec running` / `Exec finished`) ส่วนการอนุมัติที่ถูกปฏิเสธหรือหมดเวลาถือเป็นสถานะสิ้นสุด และจะไม่ปลุกเซสชันเอเจนต์ด้วยเหตุการณ์ระบบแจ้งการปฏิเสธ

ในช่องทางที่มีการ์ด/ปุ่มอนุมัติแบบเนทีฟ เอเจนต์ควรใช้ UI แบบเนทีฟนั้นก่อน และใส่คำสั่ง `/approve` สำหรับดำเนินการด้วยตนเองเฉพาะเมื่อผลลัพธ์ของเครื่องมือระบุอย่างชัดเจนว่าไม่สามารถอนุมัติผ่านแชตได้ หรือการอนุมัติด้วยตนเองเป็นวิธีเดียวเท่านั้น

## รายการที่อนุญาต + ไบนารีที่ปลอดภัย

การบังคับใช้รายการที่อนุญาตด้วยตนเองจะจับคู่กับรูปแบบ glob ของพาธไบนารีที่แก้ไขแล้วและรูปแบบ glob ของชื่อคำสั่งเปล่า ชื่อเปล่าจะจับคู่เฉพาะคำสั่งที่เรียกผ่าน PATH ดังนั้น `rg` จึงจับคู่กับ `/opt/homebrew/bin/rg` ได้เมื่อคำสั่งคือ `rg` แต่ไม่จับคู่กับ `./rg` หรือ `/tmp/rg`

เมื่อ `security=allowlist` คำสั่งเชลล์จะได้รับอนุญาตโดยอัตโนมัติเฉพาะเมื่อทุกส่วนของไปป์ไลน์อยู่ในรายการที่อนุญาตหรือเป็นไบนารีที่ปลอดภัย การเชื่อมคำสั่ง (`;`, `&&`, `||`) และการเปลี่ยนทิศทางจะถูกปฏิเสธในโหมดรายการที่อนุญาต เว้นแต่ทุกส่วนระดับบนสุดจะเป็นไปตามรายการที่อนุญาต (รวมถึงไบนารีที่ปลอดภัย) การเปลี่ยนทิศทางยังคงไม่รองรับ ความเชื่อถือ `allow-always` แบบถาวรไม่ข้ามกฎนี้ กล่าวคือคำสั่งที่เชื่อมกันยังคงกำหนดให้ทุกส่วนระดับบนสุดต้องจับคู่

`autoAllowSkills` เป็นช่องทางอำนวยความสะดวกแยกต่างหากในการอนุมัติ exec ไม่ใช่สิ่งเดียวกับรายการพาธที่อนุญาตด้วยตนเอง สำหรับความเชื่อถือแบบชัดแจ้งที่เคร่งครัด ให้ปิดใช้งาน `autoAllowSkills` ไว้

ใช้ตัวควบคุมทั้งสองสำหรับงานที่ต่างกัน:

- `tools.exec.safeBins`: ตัวกรองสตรีมขนาดเล็กที่รับข้อมูลผ่าน stdin เท่านั้น
- `tools.exec.safeBinTrustedDirs`: ไดเรกทอรีที่เชื่อถือเพิ่มเติมซึ่งระบุอย่างชัดเจนสำหรับพาธไฟล์ปฏิบัติการของไบนารีที่ปลอดภัย
- `tools.exec.safeBinProfiles`: นโยบาย argv ที่ระบุอย่างชัดเจนสำหรับไบนารีที่ปลอดภัยแบบกำหนดเอง
- allowlist: ความเชื่อถือที่ระบุอย่างชัดเจนสำหรับพาธไฟล์ปฏิบัติการ

อย่าถือว่า `safeBins` เป็นรายการที่อนุญาตทั่วไป และอย่าเพิ่มไบนารีอินเทอร์พรีเตอร์/รันไทม์ (เช่น `python3`, `node`, `ruby`, `bash`) หากจำเป็นต้องใช้ ให้ใช้รายการที่อนุญาตแบบชัดแจ้งและเปิดพรอมต์การอนุมัติไว้

`openclaw security audit` จะแจ้งเตือนเมื่อรายการ `safeBins` ของอินเทอร์พรีเตอร์/รันไทม์ไม่มีโปรไฟล์ที่ระบุอย่างชัดเจน และ `openclaw doctor --fix` สามารถสร้างโครงรายการ `safeBinProfiles` แบบกำหนดเองที่ขาดหายไปได้ `openclaw security audit` และ `openclaw doctor` จะแจ้งเตือนด้วยเมื่อเพิ่มไบนารีที่มีพฤติกรรมกว้าง เช่น `jq` กลับเข้าไปใน `safeBins` อย่างชัดเจน (`jq` สามารถอ่านข้อมูลสภาพแวดล้อมและโหลดโค้ด jq จากโมดูลหรือไฟล์เริ่มต้นได้ ดังนั้นควรใช้รายการที่อนุญาตแบบชัดแจ้งหรือการเรียกใช้ที่มีการอนุมัติกำกับแทน) `jq` จะถูกปฏิเสธไม่ให้เป็นไบนารีที่ปลอดภัย แม้ว่าจะระบุไว้โดยชัดแจ้งก็ตาม หากเพิ่มอินเทอร์พรีเตอร์ลงในรายการที่อนุญาตอย่างชัดแจ้ง ให้เปิดใช้ `tools.exec.strictInlineEval` เพื่อให้รูปแบบการประเมินโค้ดแบบอินไลน์ยังคงต้องได้รับการอนุมัติจากผู้ตรวจสอบหรือการอนุมัติอย่างชัดแจ้ง

สำหรับรายละเอียดและตัวอย่างนโยบายทั้งหมด โปรดดู [การอนุมัติ Exec](/th/tools/exec-approvals-advanced#safe-bins-stdin-only) และ [ไบนารีที่ปลอดภัยเทียบกับรายการที่อนุญาต](/th/tools/exec-approvals-advanced#safe-bins-versus-allowlist)

## ตัวอย่าง

เบื้องหน้า:

```json
{ "tool": "exec", "command": "ls -la" }
```

เบื้องหลัง + การสำรวจสถานะ:

```json
{"tool":"exec","command":"npm run build","yieldMs":1000}
{"tool":"process","action":"poll","sessionId":"<id>"}
```

การสำรวจสถานะมีไว้สำหรับตรวจสอบสถานะตามต้องการ ไม่ใช่สำหรับลูปรอ หากเปิดใช้การปลุกเมื่อเสร็จสิ้นโดยอัตโนมัติ คำสั่งสามารถปลุกเซสชันเมื่อมีเอาต์พุตหรือเมื่อทำงานล้มเหลว

ส่งปุ่มกด (รูปแบบ tmux):

```json
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["Enter"]}
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["C-c"]}
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["Up","Up","Enter"]}
```

ส่งข้อมูล (ส่งเฉพาะ CR):

```json
{ "tool": "process", "action": "submit", "sessionId": "<id>" }
```

วาง (ใช้โหมด bracketed โดยค่าเริ่มต้น):

```json
{ "tool": "process", "action": "paste", "sessionId": "<id>", "text": "line1\nline2\n" }
```

## apply_patch

`apply_patch` เป็นเครื่องมือย่อยของ `exec` สำหรับการแก้ไขหลายไฟล์แบบมีโครงสร้าง โดยเปิดใช้เป็นค่าเริ่มต้นและพร้อมใช้งานกับผู้ให้บริการโมเดลทุกราย ส่วน `allowModels` สามารถจำกัดการใช้งานได้ ใช้การกำหนดค่าเฉพาะเมื่อต้องการปิดใช้งานหรือจำกัดให้ใช้ได้กับโมเดลที่ระบุ:

```json5
{
  tools: {
    exec: {
      applyPatch: { workspaceOnly: true, allowModels: ["gpt-5.6-sol"] },
    },
  },
}
```

หมายเหตุ:

- นโยบายเครื่องมือยังคงมีผล โดย `allow: ["write"]` จะอนุญาต `apply_patch` โดยปริยาย
- `deny: ["write"]` ไม่ได้ปฏิเสธ `apply_patch` ให้ปฏิเสธ `apply_patch` อย่างชัดแจ้ง หรือใช้ `deny: ["group:fs"]` เมื่อต้องการบล็อกการเขียนแพตช์ด้วย
- การกำหนดค่าอยู่ภายใต้ `tools.exec.applyPatch`
- `tools.exec.applyPatch.enabled` มีค่าเริ่มต้นเป็น `true` ให้ตั้งค่าเป็น `false` เพื่อปิดใช้งานเครื่องมือ
- `tools.exec.applyPatch.workspaceOnly` มีค่าเริ่มต้นเป็น `true` (จำกัดอยู่ภายในเวิร์กสเปซ) ตั้งค่าเป็น `false` เฉพาะเมื่อตั้งใจให้ `apply_patch` เขียน/ลบข้อมูลนอกไดเรกทอรีเวิร์กสเปซ
- `tools.exec.applyPatch.allowModels` เป็นรายการรหัสโมเดลที่อนุญาตซึ่งเป็นตัวเลือก (แบบดิบ เช่น `gpt-5.4` หรือแบบเต็ม เช่น `openai/gpt-5.4`) เมื่อตั้งค่าแล้ว เฉพาะโมเดลที่ตรงกันเท่านั้นที่จะได้รับเครื่องมือ เมื่อไม่ได้ตั้งค่า โมเดลทั้งหมดจะได้รับเครื่องมือ
