`show_widget` เป็นเครื่องมือหลักที่แสดงวิดเจ็ต HTML แบบสมบูรณ์ในตัวบนพื้นผิวปัจจุบันของผู้ใช้ OpenClaw แสดงผลแบบอินไลน์ในบันทึกการสนทนาของ Control UI, iOS, Android และ macOS ส่วน Linux ใช้ Control UI บนเบราว์เซอร์ ในเซสชัน Discord ที่เปิดใช้ [Activities](/channels/discord-activities) Plugin Discord จะโพสต์ปุ่ม **Open widget** ซึ่งเปิดวิดเจ็ตเป็น Activity

## วิธีการทำงานของวิดเจ็ต

เมื่อเอเจนต์เรียก `show_widget` แกนหลักของ OpenClaw จะห่อ `widget_code` ไว้ในเอกสาร HTML ขั้นต่ำ จัดเก็บเป็นเอกสาร Canvas และส่งคืนแฮนเดิลตัวอย่าง Control UI จะแสดงแฮนเดิลนั้นเป็น iframe แบบแซนด์บ็อกซ์โดยตรงใต้การเรียกเครื่องมือ ส่วนแอปเนทีฟจะใช้เว็บวิวแบบแยก ทั้งสองแบบจะกู้คืนวิดเจ็ตหลังโหลดประวัติใหม่

สำหรับการฝังในเบราว์เซอร์ เอกสารตัวห่อจะแทรกบริดจ์ขนาดเล็กของโฮสต์สี่รายการไว้รอบโค้ดวิดเจ็ต:

- ตัวรายงานขนาดจะโพสต์ความสูงของเนื้อหาที่แสดงผลไปยังแชตที่ฝัง ซึ่งจะจำกัดและปรับ iframe ให้พอดี (160 ถึง 1200 พิกเซล)
- บริดจ์พรอมต์จะกำหนดฟังก์ชันส่วนกลาง `sendPrompt(text)` ซึ่งสคริปต์วิดเจ็ตเรียกใช้เพื่อส่งข้อความติดตามผลเข้าแชตได้ บริดจ์จะสร้างช่องทางข้อความส่วนตัวและเสนอปลายทางหนึ่งรายการให้แชตก่อนโค้ดวิดเจ็ตใด ๆ ทำงาน โดยแชตจะรับเฉพาะข้อเสนอแรกเท่านั้น ดู[วิดเจ็ตแบบโต้ตอบ](#interactive-widgets)
- บริดจ์ธีมจะรอรับโทเค็นการออกแบบปัจจุบันของ Control UI และนำไปใช้เป็นตัวแปร CSS เมื่อโหลดและทุกครั้งที่ธีมเปลี่ยน
- บริดจ์สแนปชอตจะแสดงผลเอกสารวิดเจ็ตปัจจุบันเป็น PNG เมื่อแชตที่ฝังร้องขอให้ส่งออก

ส่วนอื่นทั้งหมดจะอยู่ภายในเฟรม เอกสารทำงานในต้นทางแบบทึบพร้อม Content Security Policy ที่เข้มงวด ดังนั้นสคริปต์วิดเจ็ตจึงเข้าถึง Control UI, Gateway หรือเครือข่ายไม่ได้

การทำงานของแกนหลักจะพร้อมใช้งานเฉพาะเมื่อไคลเอนต์ Gateway ต้นทางประกาศความสามารถ `inline-widgets` เท่านั้น Control UI และแอปเนทีฟที่รองรับจะประกาศความสามารถนี้โดยอัตโนมัติ การทำงานของ Discord จะพร้อมใช้งานเฉพาะในเซสชัน Discord ที่กำหนดค่า Activities แล้ว การทำงานในช่องทางอื่นจะไม่ได้รับ `show_widget`

การขนส่งความสามารถครอบคลุมแบ็กเอนด์โมเดลแบบฝัง, Codex app-server และแบบใช้ CLI ผู้เรียก MCP ที่ยืนยันตัวตนด้วยสิทธิ์อนุญาตและผู้เรียกใช้เครื่องมือผ่าน HTTP โดยตรงยังคงปิดกั้นเมื่อไม่ผ่านเงื่อนไข เนื่องจากไม่ได้ประกาศความสามารถของไคลเอนต์

## ระบบการออกแบบ

วิดเจ็ต Canvas ทุกตัวมีสไตล์ชีตพื้นฐานแบบไม่ใช้คลาสและชุดโทเค็นขนาดเล็ก:

| โทเค็น                                                                                 | วัตถุประสงค์                               |
| ------------------------------------------------------------------------------------- | ------------------------------------- |
| `--surface`                                                                           | สีพื้นผิวระดับหน้า              |
| `--card`                                                                              | พื้นหลังของการ์ด ปุ่ม และโค้ด     |
| `--elevated`                                                                          | พื้นหลังแบบยกระดับของตัวควบคุมฟอร์ม      |
| `--text`                                                                              | ข้อความเริ่มต้นของเนื้อหาและตัวควบคุม         |
| `--text-strong`                                                                       | หัวเรื่องและค่าที่โดดเด่น         |
| `--muted`                                                                             | ข้อความรองและเส้นขอบที่ไม่เด่น     |
| `--border`                                                                            | เส้นแบ่งมาตรฐานและเส้นขอบการ์ด  |
| `--border-strong`                                                                     | เส้นขอบตัวควบคุมแบบชัดเจน                |
| `--accent`                                                                            | ลิงก์และวงแหวนโฟกัส                 |
| `--accent-fill`                                                                       | สีเติมของการดำเนินการหลัก                   |
| `--accent-fg`                                                                         | ข้อความบนการดำเนินการหลัก              |
| `--ok`                                                                                | สถานะสำเร็จ                         |
| `--warn`                                                                              | สถานะคำเตือน                         |
| `--danger`                                                                            | สถานะข้อผิดพลาดหรือการทำลาย            |
| `--info`                                                                              | สถานะให้ข้อมูล                   |
| `--radius`                                                                            | รัศมีมุมร่วมของตัวควบคุมและการ์ด |
| `--font-body`                                                                         | ชุดฟอนต์เนื้อหาของโฮสต์                  |
| `--font-mono`                                                                         | ชุดฟอนต์ความกว้างคงที่ของโฮสต์             |
| `--accent-subtle`, `--ok-subtle`, `--warn-subtle`, `--danger-subtle`, `--info-subtle` | พื้นหลังสถานะแบบโปร่งแสงที่คำนวณต่อยอด |

หัวเรื่อง ย่อหน้า ลิงก์ ปุ่ม ช่องป้อนข้อมูล ช่องเลือก พื้นที่ข้อความ ตาราง และบล็อกโค้ดที่ไม่ระบุคลาสจะได้รับสไตล์พื้นฐาน คลาสตัวช่วยมีรูปแบบทั่วไปดังนี้:

- `.card` สำหรับพื้นผิวเนื้อหาที่มีเส้นขอบ
- `.badge` ร่วมกับ `.ok`, `.warn`, `.danger` หรือ `.info` สำหรับป้ายสถานะแบบกะทัดรัด
- `.metric` สำหรับค่าตัวเลขที่โดดเด่น
- `.muted` สำหรับข้อความรอง
- `.row` สำหรับเค้าโครงแนวนอนที่ตัดบรรทัดได้
- `button.primary` สำหรับการดำเนินการหลัก

Control UI จะโพสต์ข้อความ `openclaw:widget-theme` พร้อมค่าธีมที่ใช้งานอยู่เมื่อวิดเจ็ตโหลดและทุกครั้งที่ธีมเปลี่ยน ดังนั้นวิดเจ็ตจึงติดตามธีมทุกตระกูล รวมถึง Claw, Knot, Dash และธีมกำหนดเองโดยไม่ต้องโหลดใหม่ ภายนอก Control UI รวมถึงแอปเนทีฟและการเปิดโดยตรง วิดเจ็ตจะใช้ชุดสีสว่างหรือมืดที่ฝังไว้ซึ่งเลือกโดย `prefers-color-scheme`

สร้างวิดเจ็ตโดยยึดหลักสามข้อ:

1. ใช้ตัวแปรการออกแบบสำหรับทุกสีและพื้นหลัง ห้ามกำหนดค่าสีตายตัว
2. ทำให้พื้นหลังของหน้าโปร่งใส เพื่อให้วิดเจ็ตกลมกลืนกับพื้นผิวโฮสต์
3. สงวน `--accent-fill` ไว้สำหรับการดำเนินการหลักไม่เกินหนึ่งรายการ

**การส่งออก:** ในเว็บแชต ให้เปิดเมนูการ์ดวิดเจ็ตเพื่อคัดลอกวิดเจ็ตที่แสดงผลไปยังคลิปบอร์ดหรือดาวน์โหลดเป็น PNG เอกสารวิดเจ็ตรุ่นเก่าที่ไม่มีบริดจ์สแนปชอตจะเปลี่ยนไปดาวน์โหลดไฟล์ HTML แทน

## ใช้เครื่องมือ

การทำงานทั้งสองแบบใช้ฟิลด์บังคับเหมือนกัน:

<ParamField path="title" type="string" required>
  ชื่อเรื่องสั้นที่แสดงพร้อมตัวอย่างแบบอินไลน์และในชื่อเอกสารที่โฮสต์
</ParamField>

<ParamField path="widget_code" type="string" required>
  HTML หรือ SVG ที่สมบูรณ์ในตัว สำหรับไคลเอนต์วิดเจ็ตแบบอินไลน์ อินพุตที่ขึ้นต้นด้วย `<svg` หลังตัดช่องว่างจะแสดงผลในโหมด SVG ความยาวสูงสุดคือ 262,144 อักขระ Discord รองรับเอกสาร HTML แบบสมบูรณ์หรือส่วนเนื้อหาขนาดสูงสุด 48 KiB
</ParamField>

Discord ยังรองรับข้อความ `button_label` แบบไม่บังคับสำหรับปุ่มเปิด Activity สคีมา Canvas ตั้งใจละเว้นฟิลด์เฉพาะ Discord นี้

ผลลัพธ์จากแกนหลักประกอบด้วยแฮนเดิลตัวอย่าง Canvas เพื่อให้ Control UI และแอปเนทีฟที่รองรับแสดงวิดเจ็ตได้โดยตรงจากการเรียกเครื่องมือ และกู้คืนได้หลังโหลดประวัติใหม่ Discord จะส่งคืนตัวระบุวิดเจ็ตที่จัดเก็บและข้อความที่โพสต์

`discord_widget` ยังคงลงทะเบียนเป็นนามแฝงที่เลิกใช้แล้วเป็นเวลาหนึ่งรุ่น การเรียกใหม่จากเอเจนต์ควรใช้ `show_widget`

## วิดเจ็ตแบบโต้ตอบ

ใน Control UI สคริปต์วิดเจ็ตสามารถควบคุมการสนทนาได้ เอกสารตัวห่อจะกำหนดฟังก์ชันส่วนกลาง `sendPrompt(text)` การเรียกฟังก์ชันนี้จะส่ง `text` ไปยังแชตเสมือนว่าผู้ใช้พิมพ์และส่งข้อความนั้นเอง เชื่อมฟังก์ชันนี้กับปุ่มหรือตัวควบคุมอื่น ๆ เพื่อสร้างขั้นตอนแบบโต้ตอบ เช่น ตัวเลือก แบบทดสอบ หรือแดชบอร์ดแบบเจาะลึก แอปเนทีฟแสดงผลโค้ดวิดเจ็ตแบบโต้ตอบ แต่ไม่เปิดเผยบริดจ์พรอมต์แชตนี้

```html
<button onclick="sendPrompt('แสดงรายละเอียดการทดสอบที่ล้มเหลว')">การทดสอบที่ล้มเหลว</button>
```

ทุกพรอมต์จะได้รับการตรวจสอบทั้งสองด้านของขอบเขตเฟรม:

- `sendPrompt` ต้องมี[การเปิดใช้งานโดยผู้ใช้ชั่วคราว](https://developer.mozilla.org/en-US/docs/Web/Security/User_activation) ภายในวิดเจ็ต โดยจะทำงานเฉพาะในช่วงไม่กี่วินาทีหลังผู้ใช้คลิกหรือกดแป้นในวิดเจ็ต ดังนั้นให้เชื่อมกับปุ่มและเป้าหมายการคลิกอื่น ๆ การเรียกโดยอัตโนมัติขณะโหลดจะไม่เกิดผลใด ๆ บริดจ์จะเก็บปลายทางการส่งไว้เป็นส่วนตัว และปิดกั้นเมื่อไม่ผ่านเงื่อนไขในเบราว์เซอร์ที่ไม่เปิดเผยการเปิดใช้งานโดยผู้ใช้ เพื่อให้โค้ดวิดเจ็ตข้ามการตรวจสอบไม่ได้
- สิทธิ์ในการส่งพรอมต์เป็นของเอกสารวิดเจ็ตต้นฉบับเท่านั้น บริดจ์ที่เชื่อถือได้จะเสนอปลายทางของช่องทางให้แชตก่อนโค้ดวิดเจ็ตทำงานหรือนำทางเฟรมได้ แชตจะรับเฉพาะข้อเสนอแรก และช่องทางจะสิ้นสุดพร้อมเอกสารเมื่อมีการนำทาง URL สำหรับฝังที่อนุญาตจากภายนอกจะไม่ได้รับการยอมรับ
- เฟรมวิดเจ็ตต้องมองเห็นได้ในบันทึกการสนทนาและได้รับโฟกัส ซึ่งเป็นสัญญาณเพิ่มเติมที่โฮสต์สังเกตได้ว่าผู้ใช้กำลังโต้ตอบกับวิดเจ็ตนี้จริง
- ข้อความต้องไม่ว่างหลังตัดช่องว่างและมีความยาวไม่เกิน 4,000 อักขระ
- พรอมต์ที่ขึ้นต้นด้วย `/` จะถูกปฏิเสธ เพื่อให้โค้ดวิดเจ็ตเรียกคำสั่งแชต เช่น `/approve` หรือ `/stop` ไม่ได้
- เอกสารวิดเจ็ตแต่ละรายการส่งพรอมต์ได้สูงสุด 10 รายการต่อช่วงเวลาหนึ่งนาทีแบบเลื่อน พรอมต์ส่วนเกินจะถูกทิ้งโดยไม่มีการแจ้งเตือน

พรอมต์ที่ยอมรับจะแสดงในบันทึกการสนทนาเป็นข้อความผู้ใช้ตามปกติ และเริ่มรอบการทำงานปกติของเอเจนต์ในเซสชันที่เป็นเจ้าของวิดเจ็ต ไม่มีช่องทางส่งผลตอบกลับเข้าสู่วิดเจ็ต พรอมต์ที่ถูกทิ้งจะล้มเหลวโดยไม่มีการแจ้งเตือน และวิดเจ็ตไม่สามารถอ่านข้อความตอบกลับของเอเจนต์ได้

## ความปลอดภัยและการจัดเก็บ

เอกสารวิดเจ็ตใช้ Content Security Policy ที่เข้มงวด อนุญาตสไตล์และสคริปต์แบบอินไลน์ แต่บล็อกการดึงข้อมูลและการโหลดทรัพยากรภายนอก เก็บมาร์กอัป สไตล์ สคริปต์ และข้อมูลรูปภาพทั้งหมดไว้ภายใน `widget_code`

iframe ของ Control UI จะไม่ระบุ `allow-same-origin` เสมอ แม้โหมดการฝังส่วนกลางจะเป็น `trusted` เพื่อให้สคริปต์วิดเจ็ตอ่านต้นทางของแอปพลิเคชันแม่ไม่ได้ ไคลเอนต์เนทีฟใช้เว็บวิวแบบแยกที่ไม่เก็บสถานะ และบล็อกการนำทางออกจากวิดเจ็ตที่โฮสต์ โฮสต์เอกสารของแกนหลักยังให้บริการวิดเจ็ตพร้อมส่วนหัวการตอบกลับ `Content-Security-Policy: sandbox allow-scripts` ดังนั้นการแสดงผลโดยตรงยังคงเรียกใช้วิดเจ็ตในต้นทางแบบทึบแทนต้นทางของแอปพลิเคชัน แสดงผลเฉพาะโค้ดวิดเจ็ตที่ยอมรับให้ดำเนินการในเฟรมแบบแยกนั้นได้เท่านั้น

iframe ยังเป็นไปตาม [`gateway.controlUi.embedSandbox`](/th/web/control-ui#hosted-embeds) ระดับเริ่มต้น `scripts` รองรับวิดเจ็ตแบบโต้ตอบพร้อมคงการแยกต้นทางไว้

Canvas เก็บวิดเจ็ตได้สูงสุด 32 รายการต่อเซสชัน (หรือต่อเอเจนต์เมื่อไม่มีเซสชัน) การสร้างวิดเจ็ตเพิ่มจะลบเอกสารที่เก่าที่สุดในขอบเขตนั้น
