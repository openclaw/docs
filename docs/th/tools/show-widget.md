`show_widget` เป็นเครื่องมือหลักที่แสดงวิดเจ็ต HTML แบบครบจบในตัวบนพื้นผิวปัจจุบันของผู้ใช้ OpenClaw จะแสดงผลแบบอินไลน์ในบันทึกการสนทนาของ Control UI, iOS, Android และ macOS ส่วน Linux ใช้ Control UI บนเบราว์เซอร์ ในเซสชัน Discord ที่เปิดใช้ [Activities](/th/channels/discord-activities) Plugin ของ Discord จะโพสต์ปุ่ม **เปิดวิดเจ็ต** ซึ่งเปิดวิดเจ็ตเป็น Activity

## วิธีการทำงานของวิดเจ็ต

เมื่อเอเจนต์เรียก `show_widget` แกนหลักของ OpenClaw จะห่อ `widget_code` ไว้ในเอกสาร HTML ขั้นต่ำ จัดเก็บเป็นเอกสาร Canvas และส่งคืนแฮนเดิลตัวอย่าง Control UI จะแสดงแฮนเดิลนั้นเป็น iframe แบบแซนด์บ็อกซ์ใต้การเรียกเครื่องมือโดยตรง ส่วนแอปเนทีฟจะใช้มุมมองเว็บแบบแยกส่วน ทั้งสองแบบจะคืนค่าวิดเจ็ตหลังโหลดประวัติใหม่

ในเซสชัน Control UI วิดเจ็ต Canvas ยังสามารถปักหมุดไว้บนแดชบอร์ดของเซสชันได้อีกด้วย ตั้งค่า `pin: true` ในการเรียกเครื่องมือ หรือใช้ **ปักหมุดบนแดชบอร์ด** บนวิดเจ็ตที่มีอยู่ในบันทึกการสนทนา การปักหมุดจะใช้เอกสารที่โฮสต์อยู่เดิมทุกประการ โดยไม่ดึง HTML ของวิดเจ็ตผ่านเบราว์เซอร์

สำหรับการฝังในเบราว์เซอร์ เอกสารตัวห่อจะแทรกบริดจ์โฮสต์ขนาดเล็กสี่รายการรอบโค้ดวิดเจ็ต:

- ตัวรายงานขนาดจะส่งความสูงของเนื้อหาที่แสดงผลไปยังแชตที่ฝัง ซึ่งจะจำกัดค่าและปรับ iframe ให้พอดี (160 ถึง 1200 พิกเซล)
- บริดจ์พรอมต์จะกำหนดฟังก์ชันส่วนกลาง `sendPrompt(text)` ซึ่งสคริปต์วิดเจ็ตเรียกใช้เพื่อส่งข้อความติดตามผลเข้าสู่แชตได้ บริดจ์จะสร้างช่องทางข้อความส่วนตัวและเสนอปลายทางหนึ่งรายการให้แชตก่อนที่โค้ดวิดเจ็ตใด ๆ จะทำงาน โดยแชตจะรับเฉพาะข้อเสนอแรกนั้น ดู [วิดเจ็ตแบบโต้ตอบ](#interactive-widgets)
- บริดจ์ธีมจะคอยรับโทเค็นการออกแบบปัจจุบันของ Control UI และนำไปใช้เป็นตัวแปร CSS ทั้งเมื่อโหลดและทุกครั้งที่เปลี่ยนธีม
- บริดจ์สแนปชอตจะแสดงผลเอกสารวิดเจ็ตปัจจุบันเป็น PNG เมื่อแชตที่ฝังร้องขอการส่งออก

ส่วนอื่นทั้งหมดจะอยู่ภายในเฟรม เอกสารทำงานในออริจินแบบทึบภายใต้นโยบาย Content Security Policy ที่เข้มงวด ดังนั้นสคริปต์วิดเจ็ตจึงไม่สามารถเข้าถึง Control UI, Gateway หรือเครือข่ายได้

การทำงานของแกนหลักพร้อมใช้งานเฉพาะเมื่อไคลเอนต์ Gateway ต้นทางประกาศความสามารถ `inline-widgets` เท่านั้น Control UI และแอปเนทีฟที่รองรับจะประกาศความสามารถนี้โดยอัตโนมัติ การทำงานบน Discord พร้อมใช้งานเฉพาะในเซสชัน Discord ที่กำหนดค่า Activities แล้ว การทำงานผ่านช่องทางอื่นจะไม่ได้รับ `show_widget`

การส่งผ่านความสามารถครอบคลุมแบ็กเอนด์โมเดลแบบฝัง, Codex app-server และแบบที่รองรับด้วย CLI ส่วนผู้เรียก MCP ที่ยืนยันตัวตนด้วย grant และผู้เรียกใช้เครื่องมือผ่าน HTTP โดยตรงยังคงปฏิเสธโดยค่าเริ่มต้น เนื่องจากไม่ได้ประกาศความสามารถของไคลเอนต์

## ระบบการออกแบบ

วิดเจ็ต Canvas ทุกตัวมีสไตล์ชีตพื้นฐานที่ไม่ใช้คลาสและชุดโทเค็นขนาดเล็ก:

| โทเค็น                                                                                 | วัตถุประสงค์                               |
| ------------------------------------------------------------------------------------- | ------------------------------------- |
| `--surface`                                                                           | สีพื้นผิวระดับหน้า              |
| `--card`                                                                              | พื้นหลังของการ์ด ปุ่ม และโค้ด     |
| `--elevated`                                                                          | พื้นหลังส่วนควบคุมฟอร์มแบบยกระดับ      |
| `--text`                                                                              | ข้อความเนื้อหาและส่วนควบคุมเริ่มต้น         |
| `--text-strong`                                                                       | หัวข้อและค่าที่โดดเด่น         |
| `--muted`                                                                             | ข้อความรองและเส้นขอบแบบบาง     |
| `--border`                                                                            | เส้นแบ่งมาตรฐานและเส้นขอบการ์ด  |
| `--border-strong`                                                                     | เส้นขอบส่วนควบคุมแบบเด่น                |
| `--accent`                                                                            | ลิงก์และวงแหวนโฟกัส                 |
| `--accent-fill`                                                                       | สีเติมของการดำเนินการหลัก                   |
| `--accent-fg`                                                                         | ข้อความบนการดำเนินการหลัก              |
| `--ok`                                                                                | สถานะสำเร็จ                         |
| `--warn`                                                                              | สถานะคำเตือน                         |
| `--danger`                                                                            | สถานะข้อผิดพลาดหรือการดำเนินการที่เป็นอันตราย            |
| `--info`                                                                              | สถานะข้อมูล                   |
| `--radius`                                                                            | รัศมีมุมร่วมสำหรับส่วนควบคุมและการ์ด |
| `--font-body`                                                                         | ชุดฟอนต์เนื้อหาของโฮสต์                  |
| `--font-mono`                                                                         | ชุดฟอนต์ความกว้างคงที่ของโฮสต์             |
| `--accent-subtle`, `--ok-subtle`, `--warn-subtle`, `--danger-subtle`, `--info-subtle` | พื้นหลังสถานะแบบโปร่งแสงที่คำนวณจากค่าอื่น |

หัวข้อ ย่อหน้า ลิงก์ ปุ่ม อินพุต ตัวเลือก พื้นที่ข้อความ ตาราง และบล็อกโค้ดที่ไม่มีคลาสจะได้รับสไตล์พื้นฐาน คลาสตัวช่วยมีรูปแบบที่ใช้บ่อยดังนี้:

- `.card` สำหรับพื้นผิวเนื้อหาที่มีเส้นขอบ
- `.badge` ร่วมกับ `.ok`, `.warn`, `.danger` หรือ `.info` สำหรับป้ายสถานะแบบกะทัดรัด
- `.metric` สำหรับค่าตัวเลขที่โดดเด่น
- `.muted` สำหรับข้อความรอง
- `.row` สำหรับเลย์เอาต์แนวนอนที่ตัดขึ้นบรรทัดใหม่ได้
- `button.primary` สำหรับการดำเนินการหลัก

Control UI จะโพสต์ข้อความ `openclaw:widget-theme` พร้อมค่าธีมที่ใช้งานอยู่เมื่อวิดเจ็ตโหลดและทุกครั้งที่ธีมเปลี่ยน ดังนั้นวิดเจ็ตจึงติดตามทุกตระกูลธีม รวมถึง Claw, Knot, Dash และธีมแบบกำหนดเองได้โดยไม่ต้องโหลดใหม่ เมื่ออยู่นอก Control UI รวมถึงแอปเนทีฟและการเปิดโดยตรง วิดเจ็ตจะใช้ชุดสีสว่างหรือมืดที่ฝังไว้ตามค่าที่ `prefers-color-scheme` เลือก

สร้างวิดเจ็ตตามกฎสามข้อต่อไปนี้:

1. ใช้ตัวแปรการออกแบบสำหรับสีและพื้นหลังทั้งหมด ห้ามกำหนดค่าสีตายตัว
2. ทำให้พื้นหลังของหน้าโปร่งใสเพื่อให้วิดเจ็ตกลมกลืนกับพื้นผิวโฮสต์
3. สงวน `--accent-fill` ไว้สำหรับการดำเนินการหลักไม่เกินหนึ่งรายการ

**การส่งออก:** ในเว็บแชต ให้เปิดเมนูการ์ดวิดเจ็ตเพื่อคัดลอกวิดเจ็ตที่แสดงผลไปยังคลิปบอร์ดหรือดาวน์โหลดเป็น PNG เอกสารวิดเจ็ตรุ่นเก่าที่ไม่มีบริดจ์สแนปชอตจะใช้การดาวน์โหลดเป็นไฟล์ HTML แทน

## ใช้เครื่องมือ

การทำงานทั้งสองแบบใช้ฟิลด์บังคับเดียวกัน:

<ParamField path="title" type="string" required>
  ชื่อสั้นที่แสดงพร้อมตัวอย่างแบบอินไลน์และใช้เป็นชื่อเอกสารที่โฮสต์
</ParamField>

<ParamField path="widget_code" type="string" required>
  HTML หรือ SVG แบบครบจบในตัว สำหรับไคลเอนต์วิดเจ็ตแบบอินไลน์ อินพุตที่ขึ้นต้นด้วย `<svg` หลังตัดช่องว่างจะแสดงผลในโหมด SVG โดยมีความยาวสูงสุด 262,144 อักขระ Discord รองรับเอกสาร HTML แบบสมบูรณ์หรือส่วนเนื้อหาที่มีขนาดสูงสุด 48 KiB
</ParamField>

Discord ยังรองรับข้อความ `button_label` แบบไม่บังคับสำหรับปุ่มเปิด Activity สคีมา Canvas จงใจไม่รวมฟิลด์ที่ใช้เฉพาะกับ Discord นี้

เครื่องมือ Canvas หลักรองรับฟิลด์การจัดวางบนแดชบอร์ดแบบไม่บังคับต่อไปนี้:

- `pin`: วางวิดเจ็ตบนแดชบอร์ดของเซสชันด้วย
- `name`: ชื่อวิดเจ็ตแบบคงที่ ค่าเริ่มต้นเป็น slug ของ `title`
- `tab`: slug ของแท็บปลายทาง
- `size`: ค่าใดค่าหนึ่งจาก `sm`, `md`, `lg`, `xl` หรือ `full`
- `after`: ชื่อวิดเจ็ตพี่น้องที่จะวางวิดเจ็ตนี้ไว้ต่อจากรายการนั้น

ผลลัพธ์จากแกนหลักมีแฮนเดิลตัวอย่าง Canvas เพื่อให้ Control UI และแอปเนทีฟที่รองรับแสดงวิดเจ็ตจากการเรียกเครื่องมือโดยตรงและคืนค่าหลังโหลดประวัติใหม่ ผลลัพธ์ที่ปักหมุดยังเก็บชื่อวิดเจ็ตบนบอร์ดไว้ เพื่อไม่ให้ Control UI เสนอการปักหมุดซ้ำหลังโหลดบันทึกการสนทนาใหม่ Discord จะส่งคืนตัวระบุของวิดเจ็ตที่จัดเก็บและข้อความที่โพสต์

`discord_widget` ยังคงลงทะเบียนเป็นนามแฝงที่เลิกใช้แล้วเป็นเวลาหนึ่งรุ่น การเรียกใช้ใหม่จากเอเจนต์ควรใช้ `show_widget`

## วิดเจ็ตแบบโต้ตอบ

ใน Control UI สคริปต์วิดเจ็ตสามารถควบคุมการสนทนาได้ เอกสารตัวห่อจะกำหนดฟังก์ชันส่วนกลาง `sendPrompt(text)` เมื่อเรียกใช้ ฟังก์ชันนี้จะส่ง `text` ไปยังแชตเสมือนว่าผู้ใช้พิมพ์และส่งข้อความนั้น เชื่อมฟังก์ชันนี้กับปุ่มหรือส่วนควบคุมอื่นเพื่อสร้างขั้นตอนแบบโต้ตอบ เช่น ตัวเลือก แบบทดสอบ หรือแดชบอร์ดแบบเจาะลึก แอปเนทีฟจะแสดงโค้ดวิดเจ็ตแบบโต้ตอบ แต่ไม่เปิดให้ใช้บริดจ์พรอมต์แชตนี้

```html
<button onclick="sendPrompt('แสดงรายละเอียดการทดสอบที่ล้มเหลว')">การทดสอบที่ล้มเหลว</button>
```

ทุกพรอมต์จะได้รับการตรวจสอบทั้งสองฝั่งของขอบเขตเฟรม:

- `sendPrompt` ต้องมี [การเปิดใช้งานชั่วคราวโดยผู้ใช้](https://developer.mozilla.org/en-US/docs/Web/Security/User_activation) ภายในวิดเจ็ต โดยจะทำงานเฉพาะในช่วงไม่กี่วินาทีหลังจากผู้ใช้คลิกหรือกดแป้นในวิดเจ็ต ดังนั้นให้เชื่อมกับปุ่มและเป้าหมายการคลิกอื่น ๆ การเรียกโดยอัตโนมัติขณะโหลดจะไม่เกิดผลใด ๆ บริดจ์จะเก็บปลายทางการส่งไว้เป็นส่วนตัวและปฏิเสธโดยค่าเริ่มต้นในเบราว์เซอร์ที่ไม่เปิดเผยการเปิดใช้งานโดยผู้ใช้ ดังนั้นโค้ดวิดเจ็ตจึงไม่สามารถหลีกเลี่ยงการตรวจสอบนี้ได้
- สิทธิ์ในการส่งพรอมต์เป็นของเอกสารวิดเจ็ตต้นฉบับเท่านั้น บริดจ์ที่เชื่อถือได้จะเสนอปลายทางช่องทางให้แชตก่อนที่โค้ดวิดเจ็ตจะทำงานหรือนำทางเฟรม แชตจะรับเฉพาะข้อเสนอแรกนั้น และช่องทางจะสิ้นสุดพร้อมเอกสารเมื่อมีการนำทาง URL สำหรับฝังจากภายนอกที่ได้รับอนุญาตจะไม่ถูกรับ
- เฟรมวิดเจ็ตต้องมองเห็นได้ในบันทึกการสนทนาและต้องมีโฟกัส ซึ่งเป็นสัญญาณเพิ่มเติมที่โฮสต์สังเกตได้ว่าผู้ใช้กำลังโต้ตอบกับวิดเจ็ตนี้จริง
- ข้อความต้องไม่ว่างหลังตัดช่องว่างและมีความยาวไม่เกิน 4,000 อักขระ
- พรอมต์ที่ขึ้นต้นด้วย `/` จะถูกปฏิเสธ ดังนั้นโค้ดวิดเจ็ตจึงไม่สามารถเรียกคำสั่งแชต เช่น `/approve` หรือ `/stop`
- เอกสารวิดเจ็ตแต่ละรายการส่งพรอมต์ได้สูงสุด 10 รายการต่อช่วงเวลาหนึ่งนาทีแบบเลื่อน พรอมต์ที่เกินจะถูกละทิ้งโดยไม่มีการแจ้งเตือน

พรอมต์ที่ยอมรับจะแสดงในบันทึกการสนทนาเป็นข้อความผู้ใช้ทั่วไป และเริ่มรอบการทำงานปกติของเอเจนต์ในเซสชันที่เป็นเจ้าของวิดเจ็ต ไม่มีช่องทางป้อนกลับเข้าสู่วิดเจ็ต พรอมต์ที่ถูกละทิ้งจะล้มเหลวโดยไม่มีการแจ้งเตือน และวิดเจ็ตไม่สามารถอ่านคำตอบของเอเจนต์ได้

## ความปลอดภัยและพื้นที่จัดเก็บ

เอกสารวิดเจ็ตใช้นโยบาย Content Security Policy ที่เข้มงวด อนุญาตสไตล์และสคริปต์แบบอินไลน์ แต่บล็อกการดึงข้อมูลและการโหลดทรัพยากรจากภายนอก เก็บมาร์กอัป สไตล์ สคริปต์ และข้อมูลรูปภาพทั้งหมดไว้ภายใน `widget_code`

iframe ของ Control UI จะละเว้น `allow-same-origin` เสมอ แม้ว่าโหมดฝังส่วนกลางจะเป็น `trusted` ดังนั้นสคริปต์วิดเจ็ตจึงไม่สามารถอ่าน origin ของแอปพลิเคชันแม่ได้ ไคลเอนต์เนทีฟใช้ web view ที่แยกโดดเดี่ยวและไม่คงอยู่ พร้อมทั้งบล็อกการนำทางออกจากวิดเจ็ตที่โฮสต์ไว้ โฮสต์เอกสารหลักยังให้บริการวิดเจ็ตพร้อมส่วนหัวการตอบกลับ `Content-Security-Policy: sandbox allow-scripts` ดังนั้นแม้เรนเดอร์โดยตรง วิดเจ็ตก็ยังทำงานใน origin แบบทึบแทน origin ของแอปพลิเคชัน ให้เรนเดอร์เฉพาะโค้ดวิดเจ็ตที่ยินยอมให้เรียกใช้ในเฟรมที่แยกโดดเดี่ยวนั้นเท่านั้น

iframe ยังเป็นไปตาม [`gateway.controlUi.embedSandbox`](/th/web/control-ui#hosted-embeds) ระดับเริ่มต้น `scripts` รองรับวิดเจ็ตแบบโต้ตอบได้ ขณะเดียวกันก็ยังคงการแยก origin ไว้

Canvas เก็บวิดเจ็ตได้สูงสุด 32 รายการต่อเซสชัน (หรือต่อเอเจนต์เมื่อไม่มีเซสชัน) การสร้างวิดเจ็ตเพิ่มจะลบเอกสารที่เก่าที่สุดในขอบเขตนั้น
