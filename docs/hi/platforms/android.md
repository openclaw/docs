## सहायता का संक्षिप्त विवरण

- भूमिका: सहयोगी Node ऐप (Android Gateway को होस्ट नहीं करता)।
- Gateway आवश्यक: हाँ (इसे macOS, Linux या WSL2 के माध्यम से Windows पर चलाएँ)।
- इंस्टॉल करें: [Google Play](https://play.google.com/store/apps/details?id=ai.openclaw.app&hl=en_IN) या किसी समर्थित [GitHub Release](https://github.com/openclaw/openclaw/releases) से `OpenClaw-Android.apk`, Gateway के लिए [आरंभ करना](/hi/start/getting-started), फिर [पेयरिंग](/hi/channels/pairing)।
- Gateway: [रनबुक](/hi/gateway) + [कॉन्फ़िगरेशन](/hi/gateway/configuration)।
  - प्रोटोकॉल: [Gateway प्रोटोकॉल](/hi/gateway/protocol) (Node + नियंत्रण प्लेन)।

सिस्टम नियंत्रण (launchd/systemd) Gateway होस्ट पर होता है — [Gateway](/hi/gateway) देखें।

## एक साथ Gateway सत्र

प्रत्येक Gateway को एक बार पेयर करें, फिर **सेटिंग्स → Gateway** खोलें। सही का निशान
फ़ोकस किए गए Gateway को दर्शाता है और प्रत्येक स्विच नियंत्रित करता है कि फ़ोकस न किए गए Gateway का
ऑपरेटर सत्र कनेक्टेड रहे या नहीं। ऐप के फ़ोरग्राउंड में रहने के दौरान सक्षम Gateway स्वतंत्र रूप से
दोबारा कनेक्ट होते हैं, इसलिए फ़ोकस बदलने पर अन्य कनेक्शन बंद नहीं होते।
केवल फ़ोकस किया गया Gateway Android Node सत्र और डिवाइस की
क्षमताओं को नियंत्रित करता है; यह एक साथ कई Gateway को उसी फ़ोन पर कैमरा,
स्थान, स्क्रीन या सूचना कमांड जारी करने से रोकता है। ऐप के फ़ोरग्राउंड से बाहर जाने के बाद Android
द्वितीयक कनेक्शन निलंबित कर सकता है।

## Wear OS सहयोगी

Wear OS सहयोगी पेयर किए गए Android फ़ोन के प्रमाणीकृत Gateway कनेक्शन का उपयोग करता है; घड़ी कभी भी Gateway क्रेडेंशियल प्राप्त या संग्रहीत नहीं करती। यह एजेंट और सत्र चुन सकता है, सीमित ट्रांसक्रिप्ट पढ़ सकता है, टेक्स्ट या बोलकर दिए गए उत्तर भेज सकता है, सक्रिय रन रोक सकता है, चुने गए सत्र में रियल-टाइम वार्ता शुरू कर सकता है और पेयर किए गए फ़ोन के Gateway को कनेक्ट या डिस्कनेक्ट कर सकता है। यह स्थानीय उत्तर सूचनाएँ, गहरी या हल्की दिखावट और उत्तरों के लिए वैकल्पिक स्वचालित वाचन भी प्रदान करता है। फ़ोन और घड़ी के अलग-अलग समय पर होने वाले अपडेट के लिए एजेंट और Gateway नियंत्रणों पर क्षमता के अनुसार सहमति होती है। रियल-टाइम वार्ता एक अस्थायी Wear OS Data Layer चैनल पर माइक्रोफ़ोन और प्लेबैक ऑडियो स्ट्रीम करती है और चयनित फ़ोन, Gateway कनेक्शन या ऑडियो चैनल खोने पर रुक जाती है।

ऐसी [रिलीज़](https://github.com/openclaw/openclaw/releases) चुनें जिसमें दोनों एसेट सूचीबद्ध हों, फिर साइडलोड करने से पहले वही टैग डाउनलोड और सत्यापित करें:

```bash
  --repo openclaw/openclaw \

  --repo openclaw/openclaw \
  --signer-workflow openclaw/openclaw/.github/workflows/android-release.yml \
  --source-ref "refs/tags/${release_tag}" \
  --deny-self-hosted-runners
```

<Warning>
Google Play और स्टैंडअलोन APK इंस्टॉलेशन अलग-अलग अपडेट चैनलों का उपयोग करते हैं और उनकी हस्ताक्षर पहचान अलग हो सकती है। चैनल बदलने से पहले Android को मौजूदा ऐप अनइंस्टॉल करने की आवश्यकता हो सकती है, जिससे उसका स्थानीय ऐप डेटा हट जाता है। सामान्य अपडेट के लिए एक ही चैनल पर बने रहें।
</Warning>

## रिमोट Mac से Android को मिरर और नियंत्रित करें

[scrcpy](https://github.com/Genymobile/scrcpy) Android स्क्रीन को macOS विंडो में मिरर करता है और
Android Debug Bridge (ADB) के माध्यम से कीबोर्ड और पॉइंटर इनपुट अग्रेषित करता है। यह ऑपरेटर की ओर से किया जाने वाला
कार्यप्रवाह है, जो OpenClaw Node कनेक्शन से अलग है। यह तब उपयोगी है जब Android डिवाइस और
Mac अलग-अलग स्थानों पर हों, लेकिन एक निजी Tailscale नेटवर्क साझा करते हों।

### शुरू करने से पहले

- Android डिवाइस और Mac पर Tailscale इंस्टॉल करें और दोनों को एक ही tailnet से कनेक्ट करें।
- Android पर **Developer options** और **USB debugging** सक्षम करें। Android 16 में **Wireless
  debugging**, **Settings > System > Developer options** के अंतर्गत होता है। [Android डेवलपर
  विकल्प](https://developer.android.com/studio/debug/dev-options) देखें।
- Mac पर scrcpy और ADB इंस्टॉल करें:

  ```bash
  brew install scrcpy
  brew install --cask android-platform-tools
  ```

- पहले कनेक्शन के लिए Android डिवाइस को उपलब्ध रखें। Mac द्वारा डिवाइस नियंत्रित किए जाने से पहले Android को प्रत्येक Mac की ADB
  कुंजी स्वीकृत करनी होगी।

### TCP पर ADB सक्षम करें

आरंभिक सेटअप के लिए Android डिवाइस को USB द्वारा किसी विश्वसनीय कंप्यूटर से कनेक्ट करें और उसके
डीबगिंग संकेत को स्वीकृति दें। फिर चलाएँ:

```bash
adb devices
adb tcpip 5555
```

अब आप USB डिस्कनेक्ट कर सकते हैं। यदि डिवाइस रीबूट या डीबगिंग रीसेट के बाद पोर्ट 5555 सुनना बंद कर देता है,
तो इस स्थानीय सेटअप चरण को दोहराएँ। Android 11 और उसके बाद के संस्करण
**Wireless debugging > Pair device with pairing code** और `adb pair` के साथ भी आरंभिक विश्वास स्थापित कर सकते हैं।

### केवल नियंत्रक Mac को अनुमति दें

प्रतिबंधात्मक अनुमतियों वाले tailnet को नियंत्रक Mac को Android डिवाइस के TCP पोर्ट 5555 तक पहुँचने की
स्पष्ट रूप से अनुमति देनी होगी। उदाहरण पतों को दोनों डिवाइस के स्थिर Tailscale IP से बदलते हुए
tailnet नीति में एक सीमित नियम जोड़ें:

```json5
{
  grants: [
    {
      src: ["<remote-mac-tailnet-ip>"],
      dst: ["<android-tailnet-ip>"],
      ip: ["tcp:5555"],
    },
  ],
}
```

होस्ट उपनामों और अन्य चयनकर्ताओं के लिए [Tailscale अनुदान](https://tailscale.com/docs/reference/syntax/grants) देखें।
इस पोर्ट को सार्वजनिक इंटरनेट की अनुमति न दें और न ही इसे Funnel के साथ उजागर करें: अधिकृत ADB
क्लाइंट के पास डिवाइस का व्यापक नियंत्रण होता है।

### कनेक्ट करें और मिररिंग शुरू करें

रिमोट Mac पर:

```bash
adb connect <android-tailnet-ip>:5555
adb devices
scrcpy --serial <android-tailnet-ip>:5555
```

इस Mac से पहला `adb connect` Android पर प्राधिकरण संवाद दिखाता है। डिवाइस अनलॉक करें,
कुंजी फ़िंगरप्रिंट की पुष्टि करें और केवल Mac के विश्वसनीय होने पर **Always allow from this computer** चुनें।
सफल `adb devices` प्रविष्टि के अंत में `device` होता है; `unauthorized` का अर्थ है कि डिवाइस पर दिखाया गया संकेत
स्वीकृत नहीं हुआ है।

scrcpy विंडो खुलने के बाद इसका सीधे उपयोग करें या इसे किसी macOS स्क्रीन-ऑटोमेशन टूल, जैसे
[Peekaboo](https://peekaboo.sh/), से लक्षित करें। scrcpy डिस्प्ले और इनपुट संभालता है; Tailscale केवल
निजी नेटवर्क मार्ग प्रदान करता है।

### समस्या निवारण

- `Connection timed out`: TCP 5555 के लिए tailnet अनुमति सत्यापित करें। सफल `tailscale ping`
  पीयर पहुँच की पुष्टि करता है, यह नहीं कि नीति इस TCP पोर्ट की अनुमति देती है। Mac से
  `nc -vz <android-tailnet-ip> 5555` का उपयोग करके परीक्षण करें।
- `unauthorized`: Android अनलॉक करें और रिमोट Mac की ADB कुंजी स्वीकृत करें या **Wireless debugging > Paired devices**
  के अंतर्गत पुराने वर्कस्टेशन को हटाकर उसे फिर से पेयर करें।
- `Connection refused`: स्थानीय रूप से पुनः कनेक्ट करें और `adb tcpip 5555` फिर से चलाएँ।
- एक से अधिक डिवाइस सूचीबद्ध हैं: स्पष्ट `--serial <android-tailnet-ip>:5555` आर्ग्युमेंट बनाए रखें।

पूरा होने पर scrcpy बंद करें और ADB डिस्कनेक्ट करें:

```bash
adb disconnect <android-tailnet-ip>:5555
```

## कनेक्शन रनबुक

Android Node ऐप ⇄ (mDNS/NSD + WebSocket) ⇄ **Gateway**

Android सीधे Gateway WebSocket से कनेक्ट होता है और डिवाइस पेयरिंग (`role: node`) का उपयोग करता है।

Tailscale या सार्वजनिक होस्ट के लिए Android को सुरक्षित एंडपॉइंट चाहिए:

- वरीय: `https://<magicdns>` / `wss://<magicdns>` के साथ Tailscale Serve / Funnel
- यह भी समर्थित है: वास्तविक TLS एंडपॉइंट वाला कोई अन्य `wss://` Gateway URL
- सादे टेक्स्ट वाला `ws://` निजी LAN पतों / `.local` होस्ट, साथ ही `localhost`, `127.0.0.1` और Android एमुलेटर ब्रिज (`10.0.2.2`) पर समर्थित रहता है; गैर-लूपबैक सेटअप स्वचालित रूप से सीमित ऑपरेटर पहुँच का उपयोग करता है

### पूर्वापेक्षाएँ

- Gateway किसी अन्य मशीन पर चल रहा हो (या SSH के माध्यम से पहुँच योग्य हो)।
- Android डिवाइस/एमुलेटर Gateway WebSocket तक पहुँच सकता हो:
  - mDNS/NSD के साथ समान LAN, **या**
  - Wide-Area Bonjour / यूनिकास्ट DNS-SD का उपयोग करने वाला समान Tailscale tailnet (नीचे देखें), **या**
  - मैन्युअल Gateway होस्ट/पोर्ट (फ़ॉलबैक)
- tailnet/सार्वजनिक मोबाइल पेयरिंग कच्चे tailnet IP `ws://` एंडपॉइंट का उपयोग **नहीं** करती। इसके बजाय Tailscale Serve या किसी अन्य `wss://` URL का उपयोग करें।
- पेयरिंग अनुरोधों को स्वीकृति देने के लिए Gateway मशीन पर (या SSH के माध्यम से) `openclaw` CLI उपलब्ध हो।

### 1. Gateway शुरू करें

```bash
openclaw gateway --port 18789 --verbose
```

पुष्टि करें कि लॉग में आपको कुछ ऐसा दिखाई देता है:

- `listening on ws://0.0.0.0:18789`

Tailscale पर रिमोट Android पहुँच के लिए कच्ची tailnet बाइंडिंग के बजाय Serve/Funnel को प्राथमिकता दें:

```bash
openclaw gateway --tailscale serve
```

इससे Android को सुरक्षित `wss://` / `https://` एंडपॉइंट मिलता है। पहली बार रिमोट Android पेयरिंग के लिए केवल `gateway.bind: "tailnet"` सेटअप पर्याप्त नहीं है, जब तक कि आप TLS को अलग से समाप्त न करें।

### 2. खोज सत्यापित करें (वैकल्पिक)

Gateway मशीन से:

```bash
dns-sd -B _openclaw-gw._tcp local.
```

डीबगिंग के अधिक नोट: [Bonjour](/hi/gateway/bonjour)।

यदि आपने वाइड-एरिया खोज डोमेन भी कॉन्फ़िगर किया है, तो इससे तुलना करें:

```bash
openclaw gateway discover --json
```

यह केवल TXT संकेतों के बजाय समाधान किए गए सेवा एंडपॉइंट का उपयोग करते हुए, एक ही बार में `local.` और कॉन्फ़िगर किया गया वाइड-एरिया डोमेन दिखाता है।

#### यूनिकास्ट DNS-SD के माध्यम से विभिन्न नेटवर्क में खोज

Android NSD/mDNS खोज नेटवर्क की सीमा पार नहीं करती। यदि Android Node और Gateway अलग-अलग नेटवर्क पर हैं, लेकिन Tailscale के माध्यम से कनेक्टेड हैं, तो इसके बजाय Wide-Area Bonjour / यूनिकास्ट DNS-SD का उपयोग करें। tailnet/सार्वजनिक Android पेयरिंग के लिए केवल खोज पर्याप्त नहीं है — खोजे गए मार्ग को फिर भी एक सुरक्षित एंडपॉइंट (`wss://` या Tailscale Serve) चाहिए:

1. Gateway होस्ट पर DNS-SD ज़ोन (उदाहरण `openclaw.internal.`) सेट अप करें और `_openclaw-gw._tcp` रिकॉर्ड प्रकाशित करें।
2. चुने गए डोमेन के लिए उस DNS सर्वर की ओर इंगित करने वाला Tailscale स्प्लिट DNS कॉन्फ़िगर करें।

विवरण और CoreDNS कॉन्फ़िगरेशन का उदाहरण: [Bonjour](/hi/gateway/bonjour)।

### 3. Android से कनेक्ट करें

Android ऐप में:

- ऐप **foreground service** (स्थायी सूचना) के माध्यम से अपने Gateway कनेक्शन को सक्रिय रखता है।
- **Connect** टैब खोलें।
- **Setup Code** या **Manual** मोड का उपयोग करें।
- यदि खोज अवरुद्ध है, तो **Advanced controls** में मैन्युअल होस्ट/पोर्ट का उपयोग करें। निजी LAN होस्ट के लिए `ws://` अभी भी काम करता है। Tailscale/सार्वजनिक होस्ट के लिए TLS चालू करें और `wss://` / Tailscale Serve एंडपॉइंट का उपयोग करें।

पहली सफल पेयरिंग के बाद Android ऐप शुरू होने पर सक्रिय पेयर किए गए Gateway से स्वचालित रूप से पुनः कनेक्ट होता है (खोजे गए Gateway के लिए सर्वोत्तम प्रयास के आधार पर, जिन्हें नेटवर्क पर दिखाई देना चाहिए)।

आधिकारिक सेटअप कोड Android को Node के रूप में कनेक्ट करते हैं और डिफ़ॉल्ट रूप से `wss://` पर पूर्ण Gateway ऑपरेटर
पहुँच देते हैं। सादे टेक्स्ट वाला गैर-लूपबैक `ws://` सेटअप
बेयरर-टोकन सुरक्षा के लिए स्वचालित रूप से सीमित पहुँच का उपयोग करता है। **Settings → Gateway**
**Full** या **Limited** पहुँच दिखाता है। सीमित कनेक्शन के लिए
`wss://` या Tailscale Serve कॉन्फ़िगर करें, Control UI में या
`openclaw qr` से नया पूर्ण-पहुँच कोड जनरेट करें, फिर उसे उस पृष्ठ पर स्कैन या पेस्ट करके पुनः कनेक्ट करें। कम किए गए
प्रोफ़ाइल का उपयोग करने के इच्छुक ऑपरेटर Control UI में **Limited access** चुन सकते हैं या
`openclaw qr --limited` चला सकते हैं।

### पेयर किए गए Gateway प्रबंधित करें

ऐप अपने द्वारा पेयर किए गए प्रत्येक Gateway की रजिस्ट्री रखता है, ताकि आप ऑपरेटर सत्रों को कनेक्टेड रख सकें और दोबारा पेयर किए बिना फ़ोकस बदल सकें:

- **Settings → Gateway** में जोड़े गए Gateway सूचीबद्ध होते हैं और फ़ोकस किए गए Gateway को चिह्नित किया जाता है। किसी प्रविष्टि पर टैप करके उसे फ़ोकस करें; अन्य सक्षम ऑपरेटर सत्र कनेक्टेड रहते हैं।
- प्रत्येक स्विच नियंत्रित करता है कि ऐप के फ़ोरग्राउंड में रहने के दौरान वह गैर-फ़ोकस Gateway कनेक्टेड रहे या नहीं। फ़ोकस किया गया Gateway सक्षम रहता है और फ़ोन के Node कनेक्शन तथा डिवाइस क्षमताओं को नियंत्रित करता है।
- एक से अधिक Gateway जोड़े जाने पर **Connect** टैब त्वरित स्विचर दिखाता है।
- क्रेडेंशियल, डिवाइस टोकन, TLS विश्वास, चैट इतिहास और कतारबद्ध ऑफ़लाइन संदेश प्रत्येक Gateway के लिए अलग-अलग संग्रहीत किए जाते हैं। फ़ोकस बदलने से Gateways के बीच स्थिति कभी मिश्रित नहीं होती, और ऑफ़लाइन रहते हुए कतारबद्ध संदेश केवल उसी Gateway को डिलीवर किए जाते हैं जिसके लिए वे लिखे गए थे।
- **Forget** किसी Gateway की रजिस्ट्री प्रविष्टि को उसके क्रेडेंशियल, डिवाइस टोकन, TLS पिन और कैश की गई चैट सहित हटा देता है।

### उपस्थिति-सक्रियता बीकन

प्रमाणीकृत Node सत्र कनेक्ट होने के बाद, और फ़ोरग्राउंड सेवा के कनेक्टेड रहते हुए ऐप के बैकग्राउंड में जाने पर, Android `event: "node.presence.alive"` के साथ `node.event` को कॉल करता है। प्रमाणीकृत Node डिवाइस पहचान ज्ञात होने के बाद ही Gateway इसे जोड़े गए Node/डिवाइस मेटाडेटा पर `lastSeenAtMs`/`lastSeenReason` के रूप में रिकॉर्ड करता है।

ऐप बीकन को सफलतापूर्वक रिकॉर्ड हुआ तभी मानता है जब Gateway प्रतिक्रिया में `handled: true` शामिल हो। पुराने Gateways `{ "ok": true }` के साथ `node.event` को स्वीकार कर सकते हैं; यह प्रतिक्रिया संगत है, लेकिन इसे टिकाऊ अंतिम-दर्शन अपडेट नहीं माना जाता।

### 4. जोड़ने की स्वीकृति दें (CLI)

Gateway मशीन पर:

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

जोड़ने का विवरण: [जोड़ना](/hi/channels/pairing)।

वैकल्पिक: यदि Android Node हमेशा किसी कड़ाई से नियंत्रित सबनेट से कनेक्ट होता है, तो स्पष्ट CIDR या सटीक IP के साथ पहली बार के Node की स्वतः-स्वीकृति चुन सकते हैं:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

यह डिफ़ॉल्ट रूप से अक्षम है। यह केवल बिना अनुरोधित स्कोप वाले नए `role: node` जोड़ने पर लागू होता है। ऑपरेटर/ब्राउज़र जोड़ने तथा भूमिका, स्कोप, मेटाडेटा या सार्वजनिक-कुंजी में किसी भी बदलाव के लिए अब भी मैन्युअल स्वीकृति आवश्यक है।

### 6. चैट + इतिहास

Android का Chat टैब सत्र चयन का समर्थन करता है (डिफ़ॉल्ट `main`, साथ ही अन्य मौजूदा सत्र):

- इतिहास: `chat.history` (प्रदर्शन के लिए सामान्यीकृत — इनलाइन निर्देश टैग, सादे-पाठ वाले टूल-कॉल XML पेलोड (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>` और संक्षिप्त किए गए प्रकार), तथा लीक हुए ASCII/पूर्ण-चौड़ाई मॉडल नियंत्रण टोकन हटा दिए जाते हैं; सटीक `NO_REPLY` / `no_reply` जैसी मौन-टोकन सहायक पंक्तियाँ छोड़ दी जाती हैं; अत्यधिक बड़ी पंक्तियों को प्लेसहोल्डर से बदला जा सकता है)
- भेजें: `chat.send`
- टिकाऊ प्रेषण: प्रत्येक प्रेषण (पाठ, चुनी गई छवियाँ और वॉइस नोट) को किसी भी नेटवर्क प्रयास से पहले प्रत्येक Gateway के लिए डिवाइस पर मौजूद आउटबॉक्स में दर्ज किया जाता है, इसलिए ऐप बंद होने पर भी सबमिट किया गया इनपुट नहीं खोता। ऑफ़लाइन रहते हुए कतारबद्ध प्रेषण दोबारा कनेक्ट होने पर स्थिर इडेम्पोटेंसी कुंजियों के साथ क्रम से डिलीवर होते हैं, और किसी प्रेषण को तभी हटाया जाता है जब टर्न कैनोनिकल `chat.history` में दिखाई दे — केवल अभिस्वीकृति को डिलीवरी का प्रमाण नहीं माना जाता। अस्पष्ट परिणाम (अभिस्वीकृति खोना, प्रेषण के बीच ऐप बंद होना, ट्रांसक्रिप्ट लिखे जाने से पहले Gateway पुनः आरंभ होना) स्वतः दोबारा भेजने के बजाय स्पष्ट **Retry**/**Delete** वाली दृश्यमान पंक्तियों के रूप में दिखते हैं। स्लैश कमांड दोबारा कनेक्ट होने पर कभी स्वतः नहीं दोहराए जाते; उन्हें स्पष्ट पुनः प्रयास के लिए रोका जाता है। कतार सीमित है (प्रत्येक Gateway के लिए 50 संदेश और 48 MB अटैचमेंट बाइट), और न भेजी गई पंक्तियाँ 48 घंटे बाद समाप्त हो जाती हैं। कभी सबमिट न किए गए कंपोज़र ड्राफ़्ट प्रक्रिया-टिकाऊ नहीं होते।
- पुश अपडेट (सर्वोत्तम प्रयास): `chat.subscribe` -> `event:"chat"`
- सुनें: किसी सहायक संदेश को देर तक दबाएँ और उसे सुनने के लिए **Listen** चुनें; ऑडियो कॉन्फ़िगर की गई TTS प्रदाता शृंखला के साथ Gateway `tts.speak` के माध्यम से रेंडर होता है, और जब Gateway ऑडियो रेंडर नहीं कर पाता तब डिवाइस की सिस्टम TTS का उपयोग किया जाता है। सत्र बदलने, नई चैट शुरू करने, ऐप के बैकग्राउंड में जाने या चैट बंद होने पर प्लेबैक रुक जाता है।

### 7. कैनवस + कैमरा

#### Gateway कैनवस होस्ट (वेब सामग्री के लिए अनुशंसित)

Node को वास्तविक HTML/CSS/JS दिखाने के लिए, जिसे एजेंट डिस्क पर संपादित कर सके, Node को Gateway कैनवस होस्ट पर निर्देशित करें।

<Note>
Nodes, Gateway HTTP सर्वर से कैनवस लोड करते हैं (`gateway.port` वाला ही पोर्ट, डिफ़ॉल्ट `18789`)।
</Note>

1. Gateway होस्ट पर `~/.openclaw/workspace/canvas/index.html` बनाएँ।
2. Node को वहाँ ले जाएँ (LAN):

```bash
openclaw nodes invoke --node "<Android Node>" --command canvas.navigate --params '{"url":"http://<gateway-hostname>.local:18789/__openclaw__/canvas/"}'
```

टेलनेट (वैकल्पिक): यदि दोनों डिवाइस Tailscale पर हैं, तो `.local` के बजाय MagicDNS नाम या टेलनेट IP का उपयोग करें, उदाहरण के लिए `http://<gateway-magicdns>:18789/__openclaw__/canvas/`।

यह सर्वर HTML में लाइव-रीलोड क्लाइंट इंजेक्ट करता है और फ़ाइल बदलने पर पुनः लोड करता है। Gateway `/__openclaw__/a2ui/` भी सर्व करता है, लेकिन Android ऐप दूरस्थ A2UI पृष्ठों को केवल रेंडर करने योग्य मानता है। क्रिया-सक्षम A2UI कमांड बंडल किए गए, ऐप-स्वामित्व वाले A2UI पृष्ठ का उपयोग करते हैं।

कैनवस कमांड (केवल फ़ोरग्राउंड में):

- `canvas.eval`, `canvas.snapshot`, `canvas.navigate` (डिफ़ॉल्ट स्कैफ़ोल्ड पर लौटने के लिए `{"url":""}` या `{"url":"/"}` का उपयोग करें)। `canvas.snapshot`, `{ format, base64 }` लौटाता है (डिफ़ॉल्ट `format="jpeg"`)।
- A2UI: `canvas.a2ui.push`, `canvas.a2ui.reset` (`canvas.a2ui.pushJSONL` पुराना उपनाम)। ये क्रिया-सक्षम रेंडरिंग के लिए बंडल किए गए, ऐप-स्वामित्व वाले A2UI पृष्ठ का उपयोग करते हैं।

कैमरा कमांड (केवल फ़ोरग्राउंड में; अनुमति-नियंत्रित): `camera.snap` (jpg), `camera.clip` (mp4)। पैरामीटर और CLI सहायकों के लिए [कैमरा Node](/hi/nodes/camera) देखें।

### 8. वॉइस + विस्तारित Android कमांड सतह

- Android का शेल नेविगेशन **Home**, **Chat** और **Settings** है। वॉइस इनपुट
  Chat कंपोज़र का भाग है; कोई अलग Voice टैब नहीं है।
- डिवाइस पर वाक् पहचान के लिए कंपोज़र माइक्रोफ़ोन पर टैप करें, जो
  ड्राफ़्ट में ट्रांसक्रिप्ट डालता है। वॉइस-नोट अटैचमेंट रिकॉर्ड करने के लिए माइक्रोफ़ोन
  को देर तक दबाएँ। प्रयास को चुपचाप छोड़ने के बजाय UI अनुपलब्ध पहचान, अनुपलब्ध अनुमति,
  व्यस्तता/नेटवर्क विफलताओं और वाक् न मिलने के परिणामों की सूचना देता
  है।
- Chat वेवफ़ॉर्म से निरंतर **Talk** शुरू करें। डिक्टेशन, वॉइस-नोट
  रिकॉर्डिंग और Talk परस्पर अनन्य माइक्रोफ़ोन पथ हैं।
- Talk Mode कैप्चर शुरू होने से पहले मौजूदा फ़ोरग्राउंड सेवा को `connectedDevice` से `connectedDevice|microphone` में उन्नत करता है, फिर Talk Mode रुकने पर उसे अवनत कर देता है। Node सेवा `CHANGE_NETWORK_STATE` के साथ `FOREGROUND_SERVICE_CONNECTED_DEVICE` घोषित करती है; Android 14+ में `FOREGROUND_SERVICE_MICROPHONE` घोषणा, `RECORD_AUDIO` रनटाइम अनुदान और रनटाइम पर माइक्रोफ़ोन सेवा प्रकार भी आवश्यक हैं।
- डिफ़ॉल्ट रूप से Android Talk, कॉन्फ़िगर किए गए Gateway Talk प्रदाता के माध्यम से मूल वाक् पहचान, Gateway चैट और `talk.speak` का उपयोग करता है। स्थानीय सिस्टम TTS का उपयोग केवल `talk.speak` के अनुपलब्ध होने पर किया जाता है।
- Android Talk रियलटाइम Gateway रिले का उपयोग केवल तब करता है जब `talk.realtime.mode`, `realtime` हो और `talk.realtime.transport`, `gateway-relay` हो।
- Android `voiceWake` क्षमता का विज्ञापन नहीं करता। वॉइस इनपुट के लिए Chat डिक्टेशन,
  वॉइस नोट या Talk का उपयोग करें।
- अतिरिक्त Android कमांड परिवार (उपलब्धता डिवाइस, अनुमतियों और उपयोगकर्ता सेटिंग पर निर्भर करती है):
  - `device.status`, `device.info`, `device.permissions`, `device.health`
  - `device.apps` केवल तब, जब **Settings > Phone Capabilities > Installed Apps** सक्षम हो; डिफ़ॉल्ट रूप से यह लॉन्चर में दिखाई देने वाले ऐप सूचीबद्ध करता है (पूरी सूची के लिए `includeNonLaunchable` पास करें)।
  - `notifications.list`, `notifications.actions` (नीचे [सूचना अग्रेषण](#notification-forwarding) देखें)
  - `photos.latest`
  - `contacts.search`, `contacts.add`
  - `calendar.events`, `calendar.add`
  - `callLog.search`
  - `sms.search`
  - `motion.activity`, `motion.pedometer`

### 9. कार्यस्थान फ़ाइलें (केवल पढ़ने योग्य)

Home अवलोकन में एक **Files** कार्ड होता है, जो केवल पढ़ने योग्य `agents.workspace.list` / `agents.workspace.get` Gateway RPC के माध्यम से सक्रिय एजेंट के कार्यस्थान को ब्राउज़ करता है: डायरेक्टरी में क्रमशः भीतर जाना, पाठ और छवि पूर्वावलोकन तथा Android शेयर शीट के माध्यम से निर्यात। कोई लेखन कार्रवाई नहीं है, और पूर्वावलोकन का आकार Gateway द्वारा सीमित किया जाता है।

## समीक्षा कमांड की स्वीकृतियाँ

`operator.admin` वाला ऑपरेटर कनेक्शन, या Gateway द्वारा स्पष्ट रूप से लक्षित जोड़ा गया
`operator.approvals` कनेक्शन, **Settings -> Approvals** के अंतर्गत लंबित exec
अनुरोधों की समीक्षा कर सकता है। ऐप अपने बटन सक्षम करने से पहले Gateway का
सैनिटाइज़ किया गया स्वीकृति रिकॉर्ड लोड करता है, किसी भी सुरक्षा चेतावनी और
उस अनुरोध द्वारा दिए गए सटीक निर्णय दिखाता है, और स्वीकृति ID तथा स्वामी प्रकार
वापस Gateway को सबमिट करता है।

स्वीकृति स्थिति Control UI और समर्थित चैट सतहों के साथ साझा की जाती है। पहला
प्रतिबद्ध उत्तर मान्य होता है; किसी अन्य सतह द्वारा पहले उत्तर दिए जाने पर भी Android
वही कैनोनिकल परिणाम दिखाता है। यदि समाधान प्रतिक्रिया खो जाती है या Gateway
डिस्कनेक्ट हो जाता है, तो ऐप कार्रवाई को लॉक रखता है और दूसरा निर्णय प्रस्तुत करने
से पहले स्वीकृति को फिर से पढ़ता है।

एकीकृत स्वीकृति विधियों से पुराने Gateways, भेजी गई exec-विशिष्ट विधियों का
फ़ॉलबैक के रूप में उपयोग करते हैं। लंबित समीक्षा तब भी काम करती है, लेकिन बनाए रखी गई टर्मिनल स्थिति
और अधिक समृद्ध क्रॉस-सतह परिणाम के लिए अपडेट किया गया Gateway आवश्यक है।

## एजेंट के प्रश्नों का उत्तर दें

Chat, `operator.questions` (या `operator.admin`) वाले ऑपरेटर कनेक्शन के लिए
लंबित Gateway प्रश्नों को मूल कार्ड के रूप में दिखाता है। कार्ड एकल और
बहु-चयन विकल्पों, विकल्प विवरणों, मुक्त-पाठ **Other** उत्तरों और
समाप्ति उलटी गिनती का समर्थन करते हैं। दोबारा कनेक्ट होने पर लंबित प्रश्न Gateway से पुनः लोड होते हैं। जब
यह डिवाइस कार्ड का उत्तर देता है, कोई अन्य सतह पहले उत्तर देती है, या
प्रश्न समाप्त या रद्द हो जाता है, तब कार्ड लॉक हो जाता है।

## सहायक प्रवेश-बिंदु

Android, सिस्टम सहायक ट्रिगर (Google Assistant) से OpenClaw लॉन्च करने का समर्थन करता है। होम बटन दबाए रखने (या किसी अन्य `ACTION_ASSIST` ट्रिगर) पर ऐप खुलता है; "Hey Google, ask OpenClaw `<prompt>`" कहने पर ऐप का घोषित App Actions क्वेरी पैटर्न मेल खाता है और प्रॉम्प्ट को स्वतः भेजे बिना चैट कंपोज़र में डाल देता है।

यह ऐप मेनिफ़ेस्ट में घोषित Android **App Actions** (`shortcuts.xml` क्षमता) का उपयोग करता है। Gateway की ओर किसी कॉन्फ़िगरेशन की आवश्यकता नहीं है — सहायक इंटेंट पूरी तरह Android ऐप द्वारा संभाला जाता है।

<Note>
App Actions की उपलब्धता डिवाइस, Google Play Services संस्करण और इस पर निर्भर करती है कि उपयोगकर्ता ने OpenClaw को डिफ़ॉल्ट सहायक ऐप के रूप में सेट किया है या नहीं।
</Note>

## सूचना अग्रेषण

Android डिवाइस सूचनाओं को `node.event` आइटम के रूप में Gateway को अग्रेषित कर सकता है। इसे ऐप की Settings शीट में **डिवाइस पर** कॉन्फ़िगर किया जाता है — Gateway/`openclaw.json` कॉन्फ़िगरेशन में नहीं।

| सेटिंग                     | विवरण                                                                                                                                                                                            |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| सूचना ईवेंट अग्रेषित करें | मुख्य टॉगल। डिफ़ॉल्ट रूप से बंद; पहले Notification Listener Access प्रदान करना आवश्यक है।                                                                                                              |
| पैकेज फ़िल्टर              | **अनुमति-सूची** (केवल सूचीबद्ध पैकेज ID अग्रेषित किए जाते हैं) या **अवरोध-सूची** (डिफ़ॉल्ट: सूचीबद्ध ID को छोड़कर सभी पैकेज)। अग्रेषण लूप रोकने के लिए अवरोध-सूची मोड में OpenClaw का अपना पैकेज हमेशा बाहर रखा जाता है। |
| शांत समय                 | स्थानीय HH:mm आरंभ/समाप्ति समयावधि, जिसमें अग्रेषण रोक दिया जाता है। डिफ़ॉल्ट रूप से अक्षम; सक्षम होने पर डिफ़ॉल्ट मान `22:00`-`07:00` होता है।                                                                                |
| अधिकतम ईवेंट / मिनट         | अग्रेषित सूचनाओं पर प्रति-डिवाइस दर सीमा। डिफ़ॉल्ट 20।                                                                                                                                          |
| रूट सत्र कुंजी           | वैकल्पिक। अग्रेषित सूचना ईवेंट को डिवाइस के डिफ़ॉल्ट सूचना रूट के बजाय किसी विशिष्ट सत्र में पिन करता है।                                                                               |

<Note>
सूचना अग्रेषण के लिए Android की Notification Listener अनुमति आवश्यक है। ऐप सेटअप के दौरान इसके लिए संकेत देता है।
</Note>

WhatsApp, WhatsApp Business, Telegram, Telegram X, Discord और Signal की सूचनाएँ हमेशा बाहर रखी जाती हैं। उनके संदेश पहले से ही मूल OpenClaw चैनल सत्रों के स्वामित्व में होते हैं; Android सूचना को एक अलग Node ईवेंट के रूप में अग्रेषित करने से उत्तर गलत वार्तालाप के माध्यम से रूट हो सकता है।
