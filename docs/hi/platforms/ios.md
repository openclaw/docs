उपलब्धता: किसी रिलीज़ के लिए सक्षम होने पर iPhone ऐप बिल्ड Apple चैनलों के माध्यम से वितरित किए जाते हैं। स्थानीय डेवलपमेंट बिल्ड स्रोत से भी चलाए जा सकते हैं।

## यह क्या करता है

- WebSocket (LAN या tailnet) पर Gateway से कनेक्ट करता है।
- Node क्षमताएँ उपलब्ध कराता है: Canvas, स्क्रीन स्नैपशॉट, कैमरा कैप्चर, स्थान, Talk मोड, वॉइस वेक और ऑप्ट-इन स्वास्थ्य सारांश।
- `node.invoke` कमांड प्राप्त करता है और Node स्थिति ईवेंट की रिपोर्ट करता है।
- Agents सतह (Files) से चयनित एजेंट के कार्यक्षेत्र को केवल-पढ़ने योग्य रूप में ब्राउज़ करता है: डायरेक्टरी में क्रमिक नेविगेशन, सिंटैक्स-हाइलाइट किए गए टेक्स्ट पूर्वावलोकन, इमेज पूर्वावलोकन और शेयर शीट निर्यात। कोई लेखन कार्रवाई नहीं; पूर्वावलोकनों का आकार Gateway द्वारा सीमित होता है।
- प्रत्येक युग्मित Gateway के लिए हाल के चैट सत्रों और ट्रांसक्रिप्ट का एक छोटा केवल-पढ़ने योग्य ऑफ़लाइन कैश रखता है: कोल्ड ओपन पर अंतिम ज्ञात ट्रांसक्रिप्ट तुरंत दिखाई देता है और Gateway से प्रतिक्रिया मिलते ही रीफ़्रेश होता है, डिस्कनेक्ट रहने के दौरान हाल की चैट ब्राउज़ की जा सकती हैं, और रीसेट/फ़ॉरगेट करने पर संरक्षित स्थानीय कैश मिट जाता है।
- डिस्कनेक्ट रहने के दौरान भेजे गए टेक्स्ट संदेशों को एक टिकाऊ प्रति-Gateway आउटबॉक्स (अधिकतम 50) में कतारबद्ध करता है: कतारबद्ध बबल ट्रांसक्रिप्ट में दिखाई देते हैं, दोबारा कनेक्ट होने पर आइडेम्पोटेंट पुनःप्रयासों के साथ क्रम से भेजे जाते हैं, कैनोनिकल इतिहास द्वारा भेजे जाने की पुष्टि तक टिकाऊ बने रहते हैं, पुनःप्रयास/हटाने की कार्रवाई दिखाने से पहले बैकऑफ़ के साथ पुनःप्रयास करते हैं, और 48 घंटे ऑफ़लाइन रहने के बाद भेजे जाने के बजाय समाप्त हो जाते हैं; रीसेट/फ़ॉरगेट करने पर कैश के साथ कतार भी साफ़ हो जाती है।
- Chat टेक्स्ट और वॉइस की एकमात्र सतह है। Chat की कार्रवाइयाँ Chat से बाहर निकले बिना पूर्ण Sessions स्क्रीन खोल सकती हैं और सहायक के तर्क तथा टूल गतिविधि को दिखा या छिपा सकती हैं। ड्राफ़्ट डिक्टेशन के लिए माइक्रोफ़ोन पर टैप करें, वॉइस नोट रिकॉर्ड करने के लिए उसका मेन्यू खोलें, या रियलटाइम वॉइस के लिए इनलाइन Talk नियंत्रण का उपयोग करें; सुनते या बोलते समय Talk नियंत्रण लाइव माइक्रोफ़ोन या प्लेबैक स्तर के अनुसार एनिमेट होता है।
- माँग पर सहायक के संदेश बोलकर सुनाता है: Chat में किसी संदेश को देर तक दबाएँ और **सुनें** चुनें। ऐप कॉन्फ़िगर किए गए TTS प्रदाता के साथ समर्थित Gateway `tts.speak` क्लिप चलाता है और Gateway ऑडियो अनुपलब्ध या चलाने योग्य न होने पर डिवाइस पर उपलब्ध वाक् सुविधा का उपयोग करता है। सत्र बदलने या ऐप के बैकग्राउंड में जाने पर प्लेबैक रुक जाता है।

## आवश्यकताएँ

- किसी अन्य डिवाइस पर चलता हुआ Gateway (macOS, Linux या WSL2 के माध्यम से Windows)।
- नेटवर्क पथ:
  - Bonjour के माध्यम से वही LAN, **या**
  - यूनिकास्ट DNS-SD के माध्यम से Tailnet (उदाहरण डोमेन: `openclaw.internal.`), **या**
  - मैन्युअल होस्ट/पोर्ट (फ़ॉलबैक)।

## तुरंत शुरू करें (युग्मित करें + कनेक्ट करें)

पहली बार लॉन्च करने पर ऐप युग्मन की संक्षिप्त व्याख्या और एक
अनुमतियों वाले पेज (सूचनाएँ, कैमरा, माइक्रोफ़ोन, फ़ोटो, संपर्क,
कैलेंडर, रिमाइंडर, स्थान) से मार्गदर्शन करता है। प्रत्येक अनुमति वैकल्पिक है और इसे
बाद में **Settings** -> **Permissions** या iOS Settings ऐप में बदला जा सकता है।

1. ऐसे मार्ग के साथ प्रमाणीकृत Gateway शुरू करें जिस तक आपका फ़ोन पहुँच सके। दूरस्थ पथ के लिए Tailscale
   Serve अनुशंसित है:

```bash
openclaw gateway --port 18789 --tailscale serve
```

विश्वसनीय समान-LAN सेटअप के लिए इसके बजाय प्रमाणीकृत `gateway.bind: "lan"`
का उपयोग करें। डिफ़ॉल्ट लूपबैक बाइंड तक फ़ोन से नहीं पहुँचा जा सकता। यदि
Gateway अभी तक कॉन्फ़िगर नहीं किया गया है, तो पहले `openclaw onboard` चलाएँ ताकि सेटअप-कोड
बनाने के लिए टोकन या पासवर्ड प्रमाणीकरण पथ उपलब्ध हो।

2. [Control UI](/hi/web/control-ui) खोलें, **Nodes** चुनें और
   **Devices** पेज पर **Pair mobile device** पर क्लिक करें। पूर्ण पहुँच अनुशंसित है
   और डिफ़ॉल्ट रूप से चयनित होती है; केवल तभी Limited access चुनें जब आप
   प्रशासनिक Gateway नियंत्रणों को हटाना चाहते हों, फिर **Create setup code** पर क्लिक करें।

3. iOS ऐप में **Settings** -> **Gateway** खोलें, QR कोड स्कैन करें (या
   सेटअप कोड पेस्ट करें) और कनेक्ट करें।

   यदि सेटअप कोड में LAN और Tailscale Serve दोनों मार्ग हैं, तो ऐप
   उनका क्रम से परीक्षण करता है और पहले पहुँच योग्य एंडपॉइंट को सहेजता है।

4. आधिकारिक ऐप स्वचालित रूप से कनेक्ट होता है। यदि **Pending approval** में कोई
   अनुरोध दिखाई देता है, तो उसे स्वीकृत करने से पहले उसकी भूमिका और स्कोप की समीक्षा करें।

   **Settings → Gateway** दिखाता है कि सहेजे गए ऑपरेटर कनेक्शन के पास
   **Full** या **Limited** पहुँच है। बेयरर-टोकन सुरक्षा के लिए प्लेनटेक्स्ट LAN `ws://` सेटअप
   स्वचालित रूप से सीमित होता है। यदि यह सीमित है, तो `wss://` या
   Tailscale Serve कॉन्फ़िगर करें, Control UI या `openclaw qr` से नया पूर्ण-पहुँच कोड स्कैन करें,
   फिर सेटिंग और अपग्रेड सक्षम करने के लिए दोबारा कनेक्ट करें।

Control UI बटन के लिए `operator.admin` वाला पहले से युग्मित सत्र आवश्यक है।
टर्मिनल फ़ॉलबैक के रूप में iOS ऐप में खोजा गया Gateway चुनें (या
Manual Host सक्षम करके होस्ट/पोर्ट दर्ज करें), फिर Gateway होस्ट पर अनुरोध स्वीकृत करें:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

यदि ऐप बदले हुए प्रमाणीकरण विवरण (भूमिका/स्कोप/सार्वजनिक कुंजी) के साथ युग्मन का पुनःप्रयास करता है, तो पिछले लंबित अनुरोध को प्रतिस्थापित कर दिया जाता है और नया `requestId` बनाया जाता है। स्वीकृति से पहले `openclaw devices list` फिर से चलाएँ।

वैकल्पिक: यदि iOS Node हमेशा कड़े नियंत्रण वाले सबनेट से कनेक्ट होता है, तो स्पष्ट CIDR या सटीक IP के साथ पहली बार Node की स्वचालित स्वीकृति के लिए ऑप्ट-इन किया जा सकता है:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

यह डिफ़ॉल्ट रूप से अक्षम है। यह केवल बिना अनुरोधित स्कोप वाले नए `role: node` युग्मन पर लागू होता है। ऑपरेटर/ब्राउज़र युग्मन और भूमिका, स्कोप, मेटाडेटा या सार्वजनिक-कुंजी में किसी भी बदलाव के लिए अब भी मैन्युअल स्वीकृति आवश्यक है।

5. कनेक्शन सत्यापित करें:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## स्वास्थ्य सारांश

iOS Node वर्तमान कैलेंडर दिन के लिए ऑप्ट-इन, केवल-पढ़ने योग्य HealthKit समुच्चय लौटा सकता है। iOS डिवाइस की सहमति और स्पष्ट Gateway कमांड प्राधिकरण
स्वतंत्र नियंत्रण हैं। सेटअप, आह्वान, पेलोड फ़ील्ड, गोपनीयता व्यवहार और समस्या निवारण के लिए
[HealthKit सारांश](/hi/platforms/ios-healthkit) देखें।

डिफ़ॉल्ट रूप से Apple Watch सहयोगी मौजूदा iPhone रिले का उपयोग करता रहता है और
उसे अलग Gateway युग्मन की आवश्यकता नहीं होती। Apple के Watch ऐप में Watch को iPhone के साथ युग्मित करें,
**Watch app -> My Watch -> Available
Apps** से OpenClaw इंस्टॉल करें, फिर दोनों डिवाइस पर OpenClaw को एक बार खोलें।

## कमांड स्वीकृतियों की समीक्षा करें

`operator.admin` वाला ऑपरेटर कनेक्शन, या Gateway द्वारा स्पष्ट रूप से लक्षित
युग्मित `operator.approvals` कनेक्शन, iPhone पर
लंबित exec अनुरोधों की समीक्षा कर सकता है। स्वीकृति कार्ड Gateway का
साफ़ किया गया कमांड पूर्वावलोकन, चेतावनी, होस्ट संदर्भ, समाप्ति और केवल उस
अनुरोध द्वारा दिए गए निर्णय दिखाता है। युग्मित Apple Watch मौजूदा
iPhone रिले के माध्यम से वही समीक्षक-सुरक्षित प्रॉम्प्ट प्राप्त करता है और संक्षिप्त
एक-बार-अनुमति/अस्वीकार निर्णय उपसमुच्चय देता है। प्रत्यक्ष Watch Gateway मोड में
स्वीकृति प्रॉम्प्ट नहीं आते।

स्वीकृति स्थिति Control UI और समर्थित चैट सतहों के साथ साझा होती है।
पहला प्रतिबद्ध उत्तर मान्य होता है। किसी अन्य सतह द्वारा अनुरोध हल करने के बाद, दूरस्थ
समाधान सूचना के बाद और जब भी समाधान की अभिस्वीकृति खो जाने की संभावना हो,
iPhone और Watch Gateway का कैनोनिकल
टर्मिनल रिकॉर्ड प्राप्त करते हैं। जब तक वह पुनर्पठन यह पुष्टि नहीं करता कि
अनुरोध अभी भी लंबित है या नहीं, कार्रवाइयाँ अनुपलब्ध रहती हैं।

स्वीकृति का स्वामित्व चयनित Gateway से बँधा होता है। Gateway बदलने पर
पुराना प्रॉम्प्ट प्रतिस्थापन कनेक्शन पर लागू नहीं किया जा सकता। एकीकृत
स्वीकृति विधियों से पुराने Gateway भेजी गई exec-विशिष्ट विधियों पर वापस जाते हैं;
संरक्षित टर्मिनल स्थिति और अधिक समृद्ध क्रॉस-सतह परिणामों के लिए अपडेट किया हुआ
Gateway आवश्यक है।

## एजेंट के प्रश्नों का उत्तर दें

Chat, `operator.questions` (या `operator.admin`) वाले ऑपरेटर कनेक्शन के लिए
लंबित Gateway प्रश्नों को मूल कार्ड के रूप में दिखाता है। कार्ड एकल और
बहु-चयन विकल्पों, विकल्प विवरणों, मुक्त-पाठ **अन्य** उत्तरों और
समाप्ति उलटी गिनती का समर्थन करते हैं। दोबारा कनेक्ट होने पर लंबित प्रश्न Gateway से पुनः लोड होते हैं। जब
यह डिवाइस उत्तर देता है, कोई अन्य सतह पहले उत्तर देती है, या
प्रश्न की अवधि समाप्त होती है या उसे रद्द किया जाता है, तब कार्ड लॉक हो जाता है।

## वैकल्पिक प्रत्यक्ष Apple Watch Node

प्रत्यक्ष मोड Watch को अपनी हस्ताक्षरित Node पहचान और Gateway कनेक्शन देता है।
जब OpenClaw सक्रिय होता है, तब युग्मित iPhone अनुपलब्ध होने पर भी
समर्थित Node कमांड Watch Wi-Fi या सेलुलर पर काम करते रहते हैं।

आवश्यकताएँ:

- iPhone, `operator.admin` स्कोप के साथ Gateway से कनेक्ट है।
- सेटअप कोड ऐसे `wss://` Gateway एंडपॉइंट का विज्ञापन करता है जिसका प्रमाणपत्र
  watchOS द्वारा विश्वसनीय है; Watch संबंधित `https://` मूल को पोल करता है। सादा HTTP और
  स्व-हस्ताक्षरित या केवल-फ़िंगरप्रिंट विश्वास समर्थित नहीं हैं। एंडपॉइंट कॉन्फ़िगरेशन के लिए [Gateway-स्वामित्व वाला
  युग्मन](/hi/gateway/pairing) देखें। लूपबैक, केवल-iPhone और
  केवल-tailnet मार्गों तक Watch स्वतंत्र रूप से नहीं पहुँच सकता।
- सेलुलर उपयोग के लिए सक्रिय सेवा वाला सेलुलर-सक्षम Apple Watch आवश्यक है।
- Watch पर OpenClaw सक्रिय है। Apple सामान्य watchOS ऐप्स को
  सामान्य WebSocket/TCP कनेक्शन बनाए रखने की अनुमति नहीं देता, इसलिए प्रत्यक्ष Node छोटे HTTPS
  पोल का उपयोग करता है और ऐप के दोबारा फ़ोरग्राउंड में आने पर फिर से कनेक्ट होता है। Apple का
  [watchOS निम्न-स्तरीय नेटवर्किंग मार्गदर्शन](https://developer.apple.com/documentation/technotes/tn3135-low-level-networking-on-watchOS) देखें।

सेटअप:

1. iPhone पर **Settings -> Apple Watch** खोलें।
2. __Enable Direct Gateway Connection** पर टैप करें।
3. कम अवधि वाले सेटअप कोड की समय-सीमा समाप्त होने से पहले Watch पर OpenClaw खोलें।
4. `openclaw nodes status` के साथ अलग Apple Watch पंक्ति सत्यापित करें।

सेटअप कोड में कम अवधि वाला, केवल-Node बूटस्ट्रैप क्रेडेंशियल होता है; इसकी समय-सीमा समाप्त होने तक इसे
पासवर्ड की तरह मानें। इसमें कभी भी iPhone का सहेजा हुआ Gateway
पासवर्ड या टोकन नहीं होता। युग्मन के बाद Watch अपना डिवाइस टोकन सहेजता है और
बूटस्ट्रैप क्रेडेंशियल हटा देता है। प्रत्यक्ष मोड केवल नीचे दिए गए कमांड को कवर करता है।
Chat, Talk, स्वीकृतियाँ और मौजूदा `watch.*` सूचना प्रवाह
iPhone-रिले सुविधाएँ बने रहते हैं और इनके लिए अब भी युग्मित iPhone आवश्यक है।

प्रत्यक्ष watchOS Node कमांड:

| सतह           | कमांड                          | टिप्पणियाँ                                              |
| ------------- | ------------------------------ | ------------------------------------------------------- |
| डिवाइस        | `device.info`, `device.status` | Watch की पहचान, बैटरी, तापीय स्थिति, स्टोरेज और नेटवर्क। |
| सूचनाएँ       | `system.notify`                | ऐप सक्रिय रहने के दौरान; Watch की अनुमति आवश्यक है।    |

watchOS तीसरे पक्ष के ऐप्स को WebKit उपलब्ध नहीं कराता, इसलिए प्रत्यक्ष Watch Node
Canvas कमांड का विज्ञापन नहीं करता।

## आधिकारिक बिल्ड के लिए रिले-समर्थित पुश

आधिकारिक रूप से वितरित iOS बिल्ड कच्चे APNs टोकन को Gateway पर प्रकाशित करने के बजाय बाहरी पुश रिले का उपयोग करते हैं। सार्वजनिक रिलीज़ लेन के आधिकारिक App Store बिल्ड `https://ios-push-relay.openclaw.ai` पर होस्ट किए गए रिले का उपयोग करते हैं; यह बेस URL App Store वितरण के लिए हार्डकोड किया गया है और किसी ओवरराइड को नहीं पढ़ता।

कस्टम रिले डिप्लॉयमेंट के लिए जानबूझकर अलग iOS बिल्ड/डिप्लॉयमेंट पथ आवश्यक है, जिसका रिले URL Gateway के रिले URL से मेल खाता हो। App Store रिलीज़ लेन कभी भी कस्टम रिले URL स्वीकार नहीं करती। यदि आप कस्टम रिले बिल्ड का उपयोग कर रहे हैं, तो मेल खाने वाला Gateway रिले URL सेट करें:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

प्रवाह इस प्रकार काम करता है:

- iOS ऐप App Attest और StoreKit ऐप ट्रांज़ैक्शन JWS का उपयोग करके रिले के साथ पंजीकरण करता है।
- रिले एक अपारदर्शी रिले हैंडल और पंजीकरण-स्कोप वाला सेंड ग्रांट लौटाता है।
- iOS ऐप युग्मित Gateway पहचान (`gateway.identity.get`) प्राप्त करता है और उसे रिले पंजीकरण में शामिल करता है, ताकि रिले-समर्थित पंजीकरण उस विशिष्ट Gateway को प्रत्यायोजित हो।
- ऐप उस रिले-समर्थित पंजीकरण को `push.apns.register` के साथ युग्मित Gateway को अग्रेषित करता है।
- Gateway उस संग्रहीत रिले हैंडल का उपयोग `push.test`, बैकग्राउंड वेक और वेक नज के लिए करता है।
- यदि ऐप बाद में किसी भिन्न Gateway या भिन्न रिले बेस URL वाले बिल्ड से कनेक्ट होता है, तो वह पुराने बाइंडिंग का पुनः उपयोग करने के बजाय रिले पंजीकरण रीफ़्रेश करता है।

इस पथ के लिए Gateway को **क्या नहीं** चाहिए: कोई डिप्लॉयमेंट-व्यापी रिले टोकन नहीं, आधिकारिक App Store रिले-समर्थित सेंड के लिए कोई प्रत्यक्ष APNs कुंजी नहीं।

अपेक्षित ऑपरेटर प्रवाह:

1. आधिकारिक iOS ऐप इंस्टॉल करें।
2. वैकल्पिक: जानबूझकर अलग कस्टम रिले बिल्ड का उपयोग करते समय ही Gateway पर `gateway.push.apns.relay.baseUrl` सेट करें।
3. ऐप को Gateway से युग्मित करें और उसे कनेक्शन पूरा करने दें।
4. APNs टोकन मिलने, ऑपरेटर सत्र कनेक्ट होने और रिले पंजीकरण सफल होने के बाद ऐप `push.apns.register` प्रकाशित करता है।
5. उसके बाद, `push.test`, रीकनेक्ट वेक और वेक नज संग्रहीत रिले-समर्थित पंजीकरण का उपयोग कर सकते हैं।

## बैकग्राउंड सक्रियता बीकन

जब iOS किसी साइलेंट पुश, बैकग्राउंड रीफ़्रेश या महत्वपूर्ण-स्थान इवेंट के लिए ऐप को जगाता है, तो ऐप संक्षिप्त Node रीकनेक्ट का प्रयास करता है और फिर `event: "node.presence.alive"` के साथ `node.event` को कॉल करता है। प्रमाणित Node डिवाइस पहचान ज्ञात होने के बाद ही Gateway इसे युग्मित Node/डिवाइस मेटाडेटा पर `lastSeenAtMs`/`lastSeenReason` के रूप में दर्ज करता है।

ऐप किसी बैकग्राउंड वेक को सफलतापूर्वक दर्ज हुआ तभी मानता है, जब Gateway प्रतिक्रिया में `handled: true` शामिल हो। पुराने Gateway `{ "ok": true }` के साथ `node.event` को स्वीकार कर सकते हैं; वह प्रतिक्रिया संगत है, लेकिन उसे स्थायी अंतिम-दृश्य अपडेट नहीं माना जाता।

संगतता संबंधी नोट:

- `OPENCLAW_APNS_RELAY_BASE_URL` अब भी Gateway के लिए अस्थायी एनवायरनमेंट ओवरराइड के रूप में काम करता है (`gateway.push.apns.relay.baseUrl` कॉन्फ़िगरेशन-प्रथम पथ है)।
- App Store रिलीज़ बिल्ड का पुश मोड होस्टेड रिले होस्ट को हार्डकोड करता है और कभी भी रिले-URL ओवरराइड नहीं पढ़ता — `OPENCLAW_PUSH_RELAY_BASE_URL` बिल्ड-टाइम एनवायरनमेंट वेरिएबल केवल स्थानीय/सैंडबॉक्स iOS बिल्ड मोड को प्रभावित करता है।

## प्रमाणीकरण और विश्वास प्रवाह

रिले उन दो प्रतिबंधों को लागू करने के लिए मौजूद है जिन्हें Gateway पर प्रत्यक्ष APNs आधिकारिक iOS बिल्ड के लिए उपलब्ध नहीं करा सकता:

- केवल Apple के माध्यम से वितरित वास्तविक OpenClaw iOS बिल्ड ही होस्टेड रिले का उपयोग कर सकते हैं।
- कोई Gateway केवल उन iOS डिवाइस के लिए रिले-समर्थित पुश भेज सकता है, जो उसी विशिष्ट Gateway से युग्मित हुए हैं।

प्रत्येक चरण में:

1. `iOS app -> gateway`: ऐप सामान्य Gateway प्रमाणीकरण प्रवाह के माध्यम से Gateway से युग्मित होता है, जिससे उसे एक प्रमाणित Node सत्र और एक प्रमाणित ऑपरेटर सत्र मिलता है। ऑपरेटर सत्र `gateway.identity.get` को कॉल करता है।
2. `iOS app -> relay`: ऐप App Attest प्रमाण और StoreKit ऐप ट्रांज़ैक्शन JWS के साथ HTTPS पर रिले पंजीकरण एंडपॉइंट को कॉल करता है। रिले बंडल ID, App Attest प्रमाण और Apple वितरण प्रमाण को सत्यापित करता है तथा आधिकारिक/प्रोडक्शन वितरण पथ की आवश्यकता रखता है — यही स्थानीय Xcode/डेवलपमेंट बिल्ड को होस्टेड रिले का उपयोग करने से रोकता है, क्योंकि कोई स्थानीय बिल्ड आधिकारिक Apple वितरण प्रमाण को पूरा नहीं कर सकता।
3. `gateway identity delegation`: रिले पंजीकरण से पहले ऐप `gateway.identity.get` से युग्मित Gateway पहचान प्राप्त करता है और उसे रिले पंजीकरण पेलोड में शामिल करता है। रिले एक रिले हैंडल और उस Gateway पहचान को प्रत्यायोजित पंजीकरण-स्कोप वाला सेंड ग्रांट लौटाता है।
4. `gateway -> relay`: Gateway `push.apns.register` से मिले रिले हैंडल और सेंड ग्रांट को संग्रहीत करता है। `push.test`, रीकनेक्ट वेक और वेक नज पर Gateway अपनी डिवाइस पहचान से सेंड अनुरोध पर हस्ताक्षर करता है; रिले पंजीकरण से मिली प्रत्यायोजित Gateway पहचान के विरुद्ध संग्रहीत सेंड ग्रांट और Gateway हस्ताक्षर, दोनों को सत्यापित करता है। कोई अन्य Gateway उस संग्रहीत पंजीकरण का पुनः उपयोग नहीं कर सकता, भले ही उसे किसी तरह हैंडल मिल जाए।
5. `relay -> APNs`: आधिकारिक बिल्ड के प्रोडक्शन APNs क्रेडेंशियल और अपरिष्कृत APNs टोकन का स्वामित्व रिले के पास होता है। रिले-समर्थित आधिकारिक बिल्ड के लिए Gateway कभी भी अपरिष्कृत APNs टोकन संग्रहीत नहीं करता; रिले युग्मित Gateway की ओर से अंतिम पुश APNs को भेजता है।

यह डिज़ाइन क्यों बनाया गया: प्रोडक्शन APNs क्रेडेंशियल को उपयोगकर्ता Gateway से बाहर रखने, आधिकारिक बिल्ड के अपरिष्कृत APNs टोकन को Gateway पर संग्रहीत करने से बचने, केवल आधिकारिक OpenClaw iOS बिल्ड को होस्टेड रिले उपयोग की अनुमति देने और किसी Gateway को किसी भिन्न Gateway के स्वामित्व वाले iOS डिवाइस पर वेक पुश भेजने से रोकने के लिए।

स्थानीय/मैन्युअल बिल्ड प्रत्यक्ष APNs पर बने रहते हैं। यदि रिले के बिना उन बिल्ड का परीक्षण किया जा रहा है, तो Gateway को अब भी प्रत्यक्ष APNs क्रेडेंशियल चाहिए:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

ये Gateway होस्ट रनटाइम एनवायरनमेंट वेरिएबल हैं, Fastlane सेटिंग नहीं। `apps/ios/fastlane/.env` केवल `APP_STORE_CONNECT_KEY_ID` और `APP_STORE_CONNECT_ISSUER_ID` जैसे App Store Connect प्रमाणीकरण को संग्रहीत करता है; यह स्थानीय iOS बिल्ड के लिए प्रत्यक्ष APNs डिलीवरी कॉन्फ़िगर नहीं करता।

`~/.openclaw/credentials/` के अंतर्गत अन्य प्रोवाइडर क्रेडेंशियल के अनुरूप अनुशंसित Gateway होस्ट स्टोरेज:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

`.p8` फ़ाइल को कमिट न करें या उसे रिपॉज़िटरी चेकआउट के अंतर्गत न रखें।

## डिस्कवरी पथ

### Bonjour (LAN)

iOS ऐप `local.` पर `_openclaw-gw._tcp` ब्राउज़ करता है और कॉन्फ़िगर होने पर उसी वाइड-एरिया DNS-SD डिस्कवरी डोमेन को भी ब्राउज़ करता है। समान LAN वाले Gateway `local.` से स्वतः दिखाई देते हैं; क्रॉस-नेटवर्क डिस्कवरी बीकन प्रकार बदले बिना कॉन्फ़िगर किए गए वाइड-एरिया डोमेन का उपयोग कर सकती है।

### Tailnet (क्रॉस-नेटवर्क)

यदि mDNS अवरुद्ध है, तो यूनिकास्ट DNS-SD ज़ोन (कोई डोमेन चुनें; उदाहरण: `openclaw.internal.`) और Tailscale स्प्लिट DNS का उपयोग करें। CoreDNS उदाहरण के लिए [Bonjour](/hi/gateway/bonjour) देखें।

### मैन्युअल होस्ट/पोर्ट

Settings में **Manual Host** सक्षम करें और Gateway होस्ट + पोर्ट (डिफ़ॉल्ट `18789`) दर्ज करें।

## एकाधिक Gateway

ऐप अपने साथ युग्मित प्रत्येक Gateway की रजिस्ट्री रखता है, ताकि आप दोबारा युग्मित किए बिना उनके बीच स्विच कर सकें:

- **Settings -> Gateway** सक्रिय Gateway को चिह्नित करते हुए **Paired Gateways** सूची दिखाता है। स्विच करने के लिए किसी प्रविष्टि पर टैप करें; ऐप वर्तमान सत्रों को समाप्त करके चयनित Gateway से दोबारा कनेक्ट होता है। एक से अधिक Gateway युग्मित होने पर कनेक्शन पंक्ति के आगे क्विक-स्विच मेन्यू दिखाई देता है।
- क्रेडेंशियल, TLS विश्वास निर्णय, प्रत्येक Gateway की प्राथमिकताएँ और कैश किया गया चैट इतिहास प्रत्येक Gateway के लिए अलग-अलग संग्रहीत होते हैं। स्विच करने पर अलग-अलग Gateway की स्थिति कभी मिश्रित नहीं होती और पुश पंजीकरण सक्रिय Gateway का अनुसरण करता है।
- किसी युग्मित Gateway को स्वाइप करें (या उसके संदर्भ मेन्यू का उपयोग करें) और उसे **Forget** करें, जिससे उसके क्रेडेंशियल, डिवाइस टोकन, TLS पिन और कैश किए गए चैट हट जाते हैं।
- डिस्कवर किए गए Gateway पर स्विच करने के लिए उनका नेटवर्क पर दिखाई देना आवश्यक है; मैन्युअल Gateway सहेजे गए होस्ट और पोर्ट से दोबारा कनेक्ट होते हैं।

## Canvas + A2UI

iOS Node एक WKWebView कैनवास रेंडर करता है। उसे नियंत्रित करने के लिए `node.invoke` का उपयोग करें:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

नोट:

- Gateway कैनवास होस्ट, Gateway HTTP सर्वर से `/__openclaw__/canvas/` और `/__openclaw__/a2ui/` प्रदान करता है (`gateway.port` वाला ही पोर्ट, डिफ़ॉल्ट `18789`)।
- iOS Node अंतर्निहित स्कैफ़ोल्ड को कनेक्टेड डिफ़ॉल्ट दृश्य के रूप में रखता है। `canvas.a2ui.push` और `canvas.a2ui.reset` बंडल किए गए ऐप-स्वामित्व वाले A2UI पेज का उपयोग करते हैं।
- रिमोट Gateway A2UI पेज iOS पर केवल रेंडर किए जा सकते हैं; नेटिव A2UI बटन कार्रवाइयाँ केवल बंडल किए गए ऐप-स्वामित्व वाले पेज से स्वीकार की जाती हैं।
- `canvas.navigate` और `{"url":""}` से अंतर्निहित स्कैफ़ोल्ड पर लौटें।

## Computer Use से संबंध

iOS ऐप एक मोबाइल Node सतह है, Codex Computer Use बैकएंड नहीं। Codex Computer Use और `cua-driver mcp` MCP टूल के माध्यम से स्थानीय macOS डेस्कटॉप को नियंत्रित करते हैं; iOS ऐप `canvas.*`, `camera.*`, `screen.*`, `location.*` और `talk.*` जैसे OpenClaw Node कमांड के माध्यम से iPhone क्षमताएँ उपलब्ध कराता है।

एजेंट Node कमांड लागू करके OpenClaw के माध्यम से अब भी iOS ऐप संचालित कर सकते हैं, लेकिन वे कॉल Gateway Node प्रोटोकॉल से होकर जाते हैं और iOS की फ़ोरग्राउंड/बैकग्राउंड सीमाओं का पालन करते हैं। स्थानीय डेस्कटॉप नियंत्रण के लिए [Codex Computer Use](/hi/plugins/codex-computer-use) और iOS Node क्षमताओं के लिए इस पेज का उपयोग करें।

### Canvas eval / स्नैपशॉट

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.eval --params '{"javaScript":"(() => { const {ctx} = window.__openclaw; ctx.clearRect(0,0,innerWidth,innerHeight); ctx.lineWidth=6; ctx.strokeStyle=\"#ff2d55\"; ctx.beginPath(); ctx.moveTo(40,40); ctx.lineTo(innerWidth-40, innerHeight-40); ctx.stroke(); return \"ok\"; })()"}'
```

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.snapshot --params '{"maxWidth":900,"format":"jpeg"}'
```

## वॉइस वेक + टॉक मोड

- वॉइस वेक और टॉक मोड Settings में उपलब्ध हैं।
- `talk.realtime.transport` के `webrtc` होने पर OpenAI रीयलटाइम टॉक क्लाइंट-स्वामित्व वाले WebRTC का उपयोग करता है; स्पष्ट `gateway-relay` कॉन्फ़िगरेशन का स्वामित्व Gateway के पास बना रहता है। [टॉक मोड](/hi/nodes/talk) देखें।
- टॉक-सक्षम iOS Node `talk` क्षमता का विज्ञापन करते हैं और `talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` तथा `talk.ptt.once` घोषित कर सकते हैं; Gateway विश्वसनीय टॉक-सक्षम Node के लिए उन पुश-टू-टॉक कमांड को डिफ़ॉल्ट रूप से अनुमति देता है।
- iOS बैकग्राउंड ऑडियो को निलंबित कर सकता है; ऐप सक्रिय न होने पर वॉइस सुविधाओं को सर्वोत्तम-प्रयास मानें।

## सामान्य त्रुटियाँ

- `NODE_BACKGROUND_UNAVAILABLE`: iOS ऐप को फ़ोरग्राउंड में लाएँ (कैनवास/कैमरा/स्क्रीन कमांड के लिए यह आवश्यक है)।
- `A2UI_HOST_UNAVAILABLE`: बंडल किया गया A2UI पेज ऐप WebView में पहुँच योग्य नहीं था; ऐप को Screen टैब पर फ़ोरग्राउंड में रखें और दोबारा प्रयास करें।
- युग्मन प्रॉम्प्ट कभी दिखाई नहीं देता: `openclaw devices list` चलाएँ और मैन्युअल रूप से स्वीकृति दें।
- Watch पर iPhone की स्थिति दिखाई नहीं देती: पुष्टि करें कि iPhone `watch.status` में `watchPaired: true`
  और `watchAppInstalled: true` रिपोर्ट करता है। यदि युग्मन गलत है, तो Apple के Watch ऐप में
  Watch को युग्मित करें। यदि इंस्टॉलेशन गलत है, तो **My Watch -> Available Apps**
  से सहायक ऐप इंस्टॉल करें। किसी भी बदलाव के बाद Watch पर OpenClaw को एक बार
  खोलें; तत्काल पहुँच के लिए अब भी दोनों ऐप का चलना आवश्यक है,
  जबकि कतारबद्ध अपडेट बाद में बैकग्राउंड में पहुँच सकते हैं।
- दोबारा इंस्टॉल करने के बाद रीकनेक्ट विफल होता है: Keychain युग्मन टोकन साफ़ कर दिया गया था; Node को दोबारा युग्मित करें।
