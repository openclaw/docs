उपलब्धता: किसी रिलीज़ के लिए सक्षम होने पर iPhone ऐप बिल्ड Apple चैनलों के माध्यम से वितरित किए जाते हैं। स्थानीय डेवलपमेंट बिल्ड स्रोत से भी चलाए जा सकते हैं।

## यह क्या करता है

- WebSocket (LAN या tailnet) के माध्यम से Gateway से कनेक्ट करता है।
- Node क्षमताएँ उपलब्ध कराता है: Canvas, स्क्रीन स्नैपशॉट, कैमरा कैप्चर, स्थान, Talk मोड, Voice wake और ऑप्ट-इन स्वास्थ्य सारांश।
- `node.invoke` कमांड प्राप्त करता है और Node स्थिति ईवेंट की रिपोर्ट करता है।
- Agents सतह (Files) से चयनित एजेंट के वर्कस्पेस को केवल-पढ़ने के लिए ब्राउज़ करता है: डायरेक्टरी में क्रमिक नेविगेशन, सिंटैक्स-हाइलाइट किए गए टेक्स्ट पूर्वावलोकन, इमेज पूर्वावलोकन और शेयर शीट से निर्यात। कोई लेखन ऑपरेशन नहीं; Gateway पूर्वावलोकनों के आकार को सीमित करता है।
- प्रत्येक युग्मित Gateway के लिए हाल के चैट सत्रों और ट्रांसक्रिप्ट का छोटा, केवल-पढ़ने योग्य ऑफ़लाइन कैश रखता है: कोल्ड ओपन पर अंतिम ज्ञात ट्रांसक्रिप्ट तुरंत दिखाई देता है और Gateway से प्रतिक्रिया मिलने पर रीफ़्रेश होता है, डिस्कनेक्ट रहने के दौरान हाल की चैट ब्राउज़ की जा सकती हैं और रीसेट/फ़ॉरगेट करने पर सुरक्षित स्थानीय कैश मिट जाता है।
- डिस्कनेक्ट रहने के दौरान भेजे गए टेक्स्ट संदेशों को प्रति-Gateway टिकाऊ आउटबॉक्स (अधिकतम 50) में कतारबद्ध करता है: कतारबद्ध बबल ट्रांसक्रिप्ट में दिखाई देते हैं, दोबारा कनेक्ट होने पर आइडेम्पोटेंट पुनः प्रयासों के साथ क्रम से भेजे जाते हैं, कैनोनिकल इतिहास द्वारा प्रेषण की पुष्टि होने तक टिकाऊ रहते हैं, पुनः प्रयास/हटाने की कार्रवाई दिखाने से पहले बैकऑफ़ के साथ पुनः प्रयास करते हैं और 48 घंटे ऑफ़लाइन रहने के बाद भेजे जाने के बजाय समाप्त हो जाते हैं; रीसेट/फ़ॉरगेट करने पर कैश के साथ कतार भी साफ़ हो जाती है।
- माँग पर सहायक के संदेश बोलकर सुनाता है: Chat में किसी संदेश को देर तक दबाएँ और **Listen** चुनें। ऐप कॉन्फ़िगर किए गए TTS प्रदाता के साथ समर्थित Gateway `tts.speak` क्लिप चलाता है और Gateway ऑडियो उपलब्ध न होने या चलाए न जा सकने पर डिवाइस पर उपलब्ध वाक्-सुविधा का उपयोग करता है। सत्र बदलने या ऐप के बैकग्राउंड में जाने पर प्लेबैक रुक जाता है।

## आवश्यकताएँ

- किसी अन्य डिवाइस पर चलता हुआ Gateway (macOS, Linux या WSL2 के माध्यम से Windows)।
- नेटवर्क पथ:
  - Bonjour के माध्यम से समान LAN, **या**
  - यूनिकास्ट DNS-SD के माध्यम से Tailnet (उदाहरण डोमेन: `openclaw.internal.`), **या**
  - मैन्युअल होस्ट/पोर्ट (फ़ॉलबैक)।

## त्वरित शुरुआत (युग्मित करें + कनेक्ट करें)

पहली बार लॉन्च करने पर ऐप युग्मन की एक संक्षिप्त व्याख्या और
अनुमतियों के पृष्ठ (सूचनाएँ, कैमरा, माइक्रोफ़ोन, फ़ोटो, संपर्क,
कैलेंडर, रिमाइंडर, स्थान) से आपका मार्गदर्शन करता है। हर अनुमति वैकल्पिक है और उसे
बाद में **Settings** -> **Permissions** या iOS Settings ऐप में बदला जा सकता है।

1. ऐसे रूट के साथ प्रमाणीकृत Gateway शुरू करें जिस तक आपका फ़ोन पहुँच सकता हो। Tailscale
   Serve अनुशंसित रिमोट पथ है:

```bash
openclaw gateway --port 18789 --tailscale serve
```

विश्वसनीय समान-LAN सेटअप के लिए इसके बजाय प्रमाणीकृत `gateway.bind: "lan"`
का उपयोग करें। डिफ़ॉल्ट लूपबैक बाइंड तक फ़ोन से नहीं पहुँचा जा सकता। यदि
Gateway अभी तक कॉन्फ़िगर नहीं किया गया है, तो पहले `openclaw onboard` चलाएँ, ताकि सेटअप-कोड
बनाने के लिए टोकन या पासवर्ड प्रमाणीकरण पथ उपलब्ध हो।

2. [Control UI](/hi/web/control-ui) खोलें, **Nodes** चुनें और
   **Devices** पृष्ठ पर **Pair mobile device** क्लिक करें। पूर्ण पहुँच अनुशंसित है
   और डिफ़ॉल्ट रूप से चयनित होती है; **Limited access** केवल तभी चुनें जब आप
   प्रशासनिक Gateway नियंत्रणों को छोड़ना चाहते हों, फिर **Create setup code** क्लिक करें।

3. iOS ऐप में **Settings** -> **Gateway** खोलें, QR कोड स्कैन करें (या
   सेटअप कोड पेस्ट करें) और कनेक्ट करें।

   यदि सेटअप कोड में LAN और Tailscale Serve दोनों रूट हैं, तो ऐप
   उन्हें क्रम से जाँचता है और पहले पहुँच योग्य एंडपॉइंट को सहेजता है।

4. आधिकारिक ऐप स्वचालित रूप से कनेक्ट होता है। यदि **Pending approval** में कोई
   अनुरोध दिखाई देता है, तो उसे स्वीकृत करने से पहले उसकी भूमिका और स्कोप की समीक्षा करें।

   **Settings → Gateway** दिखाता है कि सहेजे गए ऑपरेटर कनेक्शन के पास
   **Full** या **Limited** पहुँच है। प्लेनटेक्स्ट LAN `ws://` सेटअप
   बेयरर-टोकन सुरक्षा के लिए स्वचालित रूप से सीमित रहता है। यदि यह सीमित है, तो `wss://` या
   Tailscale Serve कॉन्फ़िगर करें, Control UI या `openclaw qr` से नया पूर्ण-पहुँच कोड स्कैन करें,
   फिर सेटिंग्स और अपग्रेड सक्षम करने के लिए दोबारा कनेक्ट करें।

Control UI बटन के लिए `operator.admin` वाला पहले से युग्मित सत्र आवश्यक है।
टर्मिनल फ़ॉलबैक के रूप में iOS ऐप में खोजा गया Gateway चुनें (या
Manual Host सक्षम करके होस्ट/पोर्ट दर्ज करें), फिर Gateway होस्ट पर अनुरोध स्वीकृत करें:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

यदि ऐप बदले हुए प्रमाणीकरण विवरण (भूमिका/स्कोप/सार्वजनिक कुंजी) के साथ युग्मन का पुनः प्रयास करता है, तो पिछला लंबित अनुरोध प्रतिस्थापित हो जाता है और नया `requestId` बनाया जाता है। स्वीकृति से पहले `openclaw devices list` दोबारा चलाएँ।

वैकल्पिक: यदि iOS Node हमेशा कड़े नियंत्रण वाले सबनेट से कनेक्ट होता है, तो आप स्पष्ट CIDR या सटीक IP के साथ पहली बार होने वाले Node स्वतः-अनुमोदन को ऑप्ट-इन कर सकते हैं:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

यह डिफ़ॉल्ट रूप से अक्षम है। यह केवल बिना अनुरोधित स्कोप वाले नए `role: node` युग्मन पर लागू होता है। ऑपरेटर/ब्राउज़र युग्मन और भूमिका, स्कोप, मेटाडेटा या सार्वजनिक-कुंजी में किसी भी बदलाव के लिए अब भी मैन्युअल स्वीकृति आवश्यक है।

5. कनेक्शन सत्यापित करें:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## स्वास्थ्य सारांश

iOS Node वर्तमान कैलेंडर दिन के लिए ऑप्ट-इन, केवल-पढ़ने योग्य HealthKit
समग्र डेटा लौटा सकता है। iPhone की सहमति और स्पष्ट Gateway कमांड प्राधिकरण
स्वतंत्र नियंत्रण हैं। सेटअप, आह्वान, पेलोड फ़ील्ड, गोपनीयता व्यवहार और समस्या निवारण के लिए
[HealthKit सारांश](/platforms/ios-healthkit) देखें।

डिफ़ॉल्ट रूप से Apple Watch सहयोगी मौजूदा iPhone रिले का उपयोग करता रहता है और
उसे अलग Gateway युग्मन की आवश्यकता नहीं होती। Apple के Watch ऐप में Watch को iPhone से युग्मित करें,
**Watch app -> My Watch -> Available
Apps** से OpenClaw इंस्टॉल करें, फिर दोनों डिवाइस पर OpenClaw एक बार खोलें।

## कमांड स्वीकृतियों की समीक्षा करें

`operator.admin` वाला ऑपरेटर कनेक्शन, या Gateway द्वारा स्पष्ट रूप से लक्षित
युग्मित `operator.approvals` कनेक्शन, iPhone पर
लंबित निष्पादन अनुरोधों की समीक्षा कर सकता है। स्वीकृति कार्ड Gateway का
साफ़ किया गया कमांड पूर्वावलोकन, चेतावनी, होस्ट संदर्भ, समाप्ति और केवल उस
अनुरोध द्वारा प्रस्तुत निर्णय दिखाता है। युग्मित Apple Watch को मौजूदा
iPhone रिले के माध्यम से वही समीक्षक-सुरक्षित प्रॉम्प्ट मिलता है और वह संक्षिप्त
एक-बार-अनुमति/अस्वीकृति निर्णय उपसमुच्चय प्रस्तुत करती है। सीधे Watch Gateway मोड में
स्वीकृति प्रॉम्प्ट नहीं पहुँचते।

स्वीकृति स्थिति Control UI और समर्थित चैट सतहों के साथ साझा होती है।
पहला प्रतिबद्ध उत्तर मान्य होता है। किसी अन्य सतह द्वारा अनुरोध हल किए जाने के बाद, रिमोट
समाधान सूचना के बाद और जब भी समाधान की अभिस्वीकृति खो गई होने की संभावना हो,
iPhone और Watch Gateway का कैनोनिकल
टर्मिनल रिकॉर्ड प्राप्त करते हैं। जब तक वह रीडबैक पुष्टि नहीं करता कि
अनुरोध अभी भी लंबित है या नहीं, तब तक कार्रवाइयाँ अनुपलब्ध रहती हैं।

स्वीकृति का स्वामित्व चयनित Gateway से बँधा है। Gateway बदलने पर
पुराना प्रॉम्प्ट प्रतिस्थापन कनेक्शन पर लागू नहीं किया जा सकता। एकीकृत
स्वीकृति विधियों से पुराने Gateway, जारी किए गए निष्पादन-विशिष्ट तरीकों का फ़ॉलबैक उपयोग करते हैं;
सुरक्षित रखी गई अंतिम स्थिति और अधिक समृद्ध क्रॉस-सतह परिणामों के लिए अपडेट किया हुआ
Gateway आवश्यक है।

## वैकल्पिक प्रत्यक्ष Apple Watch Node

प्रत्यक्ष मोड Watch को उसकी अपनी हस्ताक्षरित Node पहचान और Gateway कनेक्शन देता है।
OpenClaw सक्रिय रहने के दौरान समर्थित Node कमांड Watch के Wi-Fi या सेल्युलर पर
काम करते रहते हैं, भले ही युग्मित iPhone उपलब्ध न हो।

आवश्यकताएँ:

- iPhone `operator.admin` स्कोप के साथ Gateway से कनेक्ट है।
- सेटअप कोड ऐसे `wss://` Gateway एंडपॉइंट का विज्ञापन करता है, जिसका प्रमाणपत्र watchOS द्वारा विश्वसनीय है;
  Watch संबंधित `https://` ओरिजिन को पोल करती है। सामान्य HTTP और
  स्व-हस्ताक्षरित या केवल-फ़िंगरप्रिंट विश्वास समर्थित नहीं हैं। एंडपॉइंट कॉन्फ़िगरेशन के लिए [Gateway-स्वामित्व वाला
  युग्मन](/hi/gateway/pairing) देखें। लूपबैक, केवल-iPhone और
  केवल-tailnet रूट तक Watch स्वतंत्र रूप से नहीं पहुँच सकती।
- सेल्युलर उपयोग के लिए सक्रिय सेवा वाली सेल्युलर-समर्थ Apple Watch आवश्यक है।
- OpenClaw Watch पर सक्रिय है। Apple सामान्य watchOS ऐप्स को
  जेनेरिक WebSocket/TCP कनेक्शन बनाए रखने की अनुमति नहीं देता, इसलिए प्रत्यक्ष Node छोटे HTTPS
  पोल का उपयोग करता है और ऐप के फिर से फ़ोरग्राउंड में आने पर दोबारा कनेक्ट होता है। Apple का
  [watchOS निम्न-स्तरीय नेटवर्किंग मार्गदर्शन](https://developer.apple.com/documentation/technotes/tn3135-low-level-networking-on-watchOS) देखें।

सेटअप:

1. iPhone पर **Settings -> Apple Watch** खोलें।
2. __Enable Direct Gateway Connection** पर टैप करें।
3. अल्पकालिक सेटअप कोड की समय-सीमा समाप्त होने से पहले Watch पर OpenClaw खोलें।
4. `openclaw nodes status` वाली अलग Apple Watch पंक्ति सत्यापित करें।

सेटअप कोड में अल्पकालिक, केवल-Node बूटस्ट्रैप क्रेडेंशियल होता है; इसकी
समय-सीमा समाप्त होने तक इसे पासवर्ड जैसा मानें। इसमें iPhone का सहेजा गया Gateway
पासवर्ड या टोकन कभी नहीं होता। युग्मन के बाद Watch अपना स्वयं का डिवाइस टोकन
संग्रहीत करती है और बूटस्ट्रैप क्रेडेंशियल हटा देती है। प्रत्यक्ष मोड केवल नीचे दिए गए कमांडों को कवर करता है।
Chat, Talk, स्वीकृतियाँ और मौजूदा `watch.*` सूचना प्रवाह
iPhone-रिले सुविधाएँ बने रहते हैं और उनके लिए अब भी युग्मित iPhone आवश्यक है।

प्रत्यक्ष watchOS Node कमांड:

| सतह          | कमांड                          | टिप्पणियाँ                                               |
| ------------- | ------------------------------ | ------------------------------------------------------- |
| डिवाइस        | `device.info`, `device.status` | Watch की पहचान, बैटरी, तापमान, स्टोरेज और नेटवर्क। |
| सूचनाएँ       | `system.notify`                | ऐप सक्रिय रहने के दौरान; Watch की अनुमति आवश्यक है।     |

watchOS तृतीय-पक्ष ऐप्स को WebKit उपलब्ध नहीं कराता, इसलिए प्रत्यक्ष Watch Node
Canvas कमांड का विज्ञापन नहीं करता।

## आधिकारिक बिल्ड के लिए रिले-समर्थित पुश

आधिकारिक रूप से वितरित iOS बिल्ड, Gateway पर अपरिष्कृत APNs टोकन प्रकाशित करने के बजाय बाहरी पुश रिले का उपयोग करते हैं। सार्वजनिक रिलीज़ लेन के आधिकारिक App Store बिल्ड `https://ios-push-relay.openclaw.ai` पर होस्ट किए गए रिले का उपयोग करते हैं; यह आधार URL App Store वितरण के लिए हार्डकोड किया गया है और किसी ओवरराइड को नहीं पढ़ता।

कस्टम रिले परिनियोजन के लिए जानबूझकर अलग iOS बिल्ड/परिनियोजन पथ आवश्यक है, जिसका रिले URL Gateway के रिले URL से मेल खाता हो। App Store रिलीज़ लेन कभी भी कस्टम रिले URL स्वीकार नहीं करती। यदि आप कस्टम रिले बिल्ड का उपयोग कर रहे हैं, तो मेल खाता Gateway रिले URL सेट करें:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

प्रवाह इस प्रकार काम करता है:

- iOS ऐप App Attest और StoreKit ऐप लेन-देन JWS का उपयोग करके रिले के साथ पंजीकृत होता है।
- रिले एक अपारदर्शी रिले हैंडल और पंजीकरण-स्कोप वाला प्रेषण अनुदान लौटाता है।
- iOS ऐप युग्मित Gateway पहचान (`gateway.identity.get`) प्राप्त करता है और उसे रिले पंजीकरण में शामिल करता है, ताकि रिले-समर्थित पंजीकरण उस विशिष्ट Gateway को प्रत्यायोजित हो।
- ऐप उस रिले-समर्थित पंजीकरण को `push.apns.register` के साथ युग्मित Gateway को अग्रेषित करता है।
- Gateway उस संग्रहीत रिले हैंडल का उपयोग `push.test`, बैकग्राउंड वेक और वेक संकेतों के लिए करता है।
- यदि ऐप बाद में किसी अलग Gateway या अलग रिले आधार URL वाले बिल्ड से कनेक्ट होता है, तो वह पुराने बाइंडिंग का पुनः उपयोग करने के बजाय रिले पंजीकरण को रीफ़्रेश करता है।

इस पथ के लिए Gateway को क्या **आवश्यक नहीं** है: कोई परिनियोजन-व्यापी रिले टोकन नहीं, आधिकारिक App Store रिले-समर्थित प्रेषणों के लिए कोई प्रत्यक्ष APNs कुंजी नहीं।

अपेक्षित ऑपरेटर प्रवाह:

1. आधिकारिक iOS ऐप इंस्टॉल करें।
2. वैकल्पिक: जानबूझकर अलग कस्टम रिले बिल्ड का उपयोग करते समय ही Gateway पर `gateway.push.apns.relay.baseUrl` सेट करें।
3. ऐप को Gateway से युग्मित करें और उसे कनेक्ट होना पूरा करने दें।
4. APNs टोकन मिलने, ऑपरेटर सत्र कनेक्ट होने और रिले पंजीकरण सफल होने के बाद ऐप `push.apns.register` प्रकाशित करता है।
5. इसके बाद `push.test`, पुनः कनेक्शन वेक और वेक संकेत संग्रहीत रिले-समर्थित पंजीकरण का उपयोग कर सकते हैं।

## बैकग्राउंड सक्रियता बीकन

जब iOS किसी साइलेंट पुश, बैकग्राउंड रीफ़्रेश या महत्वपूर्ण-स्थान इवेंट के लिए ऐप को सक्रिय करता है, तो ऐप Node को संक्षिप्त रूप से दोबारा कनेक्ट करने का प्रयास करता है और फिर `event: "node.presence.alive"` के साथ `node.event` को कॉल करता है। प्रमाणित Node डिवाइस की पहचान ज्ञात होने के बाद ही Gateway इसे युग्मित Node/डिवाइस मेटाडेटा पर `lastSeenAtMs`/`lastSeenReason` के रूप में दर्ज करता है।

ऐप बैकग्राउंड सक्रियण को सफलतापूर्वक दर्ज हुआ तभी मानता है, जब Gateway की प्रतिक्रिया में `handled: true` शामिल हो। पुराने Gateway, `node.event` को `{ "ok": true }` के साथ स्वीकार कर सकते हैं; वह प्रतिक्रिया संगत है, लेकिन उसे स्थायी अंतिम-दृश्य अपडेट नहीं माना जाता।

संगतता नोट:

- `OPENCLAW_APNS_RELAY_BASE_URL` अभी भी Gateway के लिए अस्थायी env ओवरराइड के रूप में काम करता है (`gateway.push.apns.relay.baseUrl` कॉन्फ़िगरेशन-प्रथम पथ है)।
- App Store रिलीज़ बिल्ड का पुश मोड होस्ट किए गए रिले होस्ट को हार्डकोड करता है और रिले-URL ओवरराइड को कभी नहीं पढ़ता — `OPENCLAW_PUSH_RELAY_BASE_URL` बिल्ड-टाइम env var केवल स्थानीय/सैंडबॉक्स iOS बिल्ड मोड को प्रभावित करता है।

## प्रमाणीकरण और विश्वास प्रवाह

रिले उन दो प्रतिबंधों को लागू करने के लिए मौजूद है जिन्हें आधिकारिक iOS बिल्ड के लिए सीधे Gateway से APNs प्रदान नहीं कर सकता:

- केवल Apple के माध्यम से वितरित वास्तविक OpenClaw iOS बिल्ड ही होस्ट किए गए रिले का उपयोग कर सकते हैं।
- कोई Gateway केवल उन iOS डिवाइसों के लिए रिले-समर्थित पुश भेज सकता है जिन्हें उसी विशिष्ट Gateway के साथ युग्मित किया गया हो।

प्रत्येक चरण:

1. `iOS app -> gateway`: ऐप सामान्य Gateway प्रमाणीकरण प्रवाह के माध्यम से Gateway के साथ युग्मित होता है, जिससे उसे एक प्रमाणित Node सत्र और एक प्रमाणित ऑपरेटर सत्र मिलता है। ऑपरेटर सत्र `gateway.identity.get` को कॉल करता है।
2. `iOS app -> relay`: ऐप, App Attest प्रमाण और StoreKit ऐप ट्रांज़ैक्शन JWS के साथ HTTPS पर रिले पंजीकरण एंडपॉइंट को कॉल करता है। रिले बंडल ID, App Attest प्रमाण और Apple वितरण प्रमाण को सत्यापित करता है तथा आधिकारिक/प्रोडक्शन वितरण पथ की आवश्यकता रखता है — यही स्थानीय Xcode/डेवलपमेंट बिल्ड को होस्ट किए गए रिले का उपयोग करने से रोकता है, क्योंकि स्थानीय बिल्ड आधिकारिक Apple वितरण प्रमाण को पूरा नहीं कर सकता।
3. `gateway identity delegation`: रिले पंजीकरण से पहले, ऐप `gateway.identity.get` से युग्मित Gateway की पहचान प्राप्त करता है और उसे रिले पंजीकरण पेलोड में शामिल करता है। रिले एक रिले हैंडल और उस Gateway पहचान को सौंपा गया पंजीकरण-स्कोप वाला प्रेषण अनुदान लौटाता है।
4. `gateway -> relay`: Gateway, `push.apns.register` से मिले रिले हैंडल और प्रेषण अनुदान को संग्रहीत करता है। `push.test`, पुनः-कनेक्ट सक्रियणों और सक्रियण संकेतों पर, Gateway अपनी डिवाइस पहचान से प्रेषण अनुरोध पर हस्ताक्षर करता है; रिले पंजीकरण से सौंपी गई Gateway पहचान के विरुद्ध संग्रहीत प्रेषण अनुदान और Gateway हस्ताक्षर, दोनों को सत्यापित करता है। कोई अन्य Gateway उस संग्रहीत पंजीकरण का दोबारा उपयोग नहीं कर सकता, भले ही उसे किसी तरह हैंडल मिल जाए।
5. `relay -> APNs`: रिले आधिकारिक बिल्ड के प्रोडक्शन APNs क्रेडेंशियल और अपरिष्कृत APNs टोकन का स्वामी होता है। रिले-समर्थित आधिकारिक बिल्ड के लिए Gateway कभी भी अपरिष्कृत APNs टोकन संग्रहीत नहीं करता; रिले युग्मित Gateway की ओर से अंतिम पुश APNs को भेजता है।

यह डिज़ाइन इसलिए बनाया गया था: प्रोडक्शन APNs क्रेडेंशियल को उपयोगकर्ता Gateway से बाहर रखने, आधिकारिक बिल्ड के अपरिष्कृत APNs टोकन को Gateway पर संग्रहीत करने से बचने, केवल आधिकारिक OpenClaw iOS बिल्ड को होस्ट किए गए रिले का उपयोग करने देने और किसी Gateway को दूसरे Gateway के स्वामित्व वाले iOS डिवाइसों पर सक्रियण पुश भेजने से रोकने के लिए।

स्थानीय/मैन्युअल बिल्ड सीधे APNs पर बने रहते हैं। यदि आप रिले के बिना उन बिल्ड का परीक्षण कर रहे हैं, तो Gateway को अभी भी सीधे APNs क्रेडेंशियल की आवश्यकता होती है:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

ये Gateway होस्ट के रनटाइम env var हैं, Fastlane सेटिंग नहीं। `apps/ios/fastlane/.env` केवल App Store Connect प्रमाणीकरण, जैसे `APP_STORE_CONNECT_KEY_ID` और `APP_STORE_CONNECT_ISSUER_ID`, संग्रहीत करता है; यह स्थानीय iOS बिल्ड के लिए सीधे APNs वितरण को कॉन्फ़िगर नहीं करता।

अन्य प्रदाता क्रेडेंशियल के लिए `~/.openclaw/credentials/` के अंतर्गत उपयोग किए जाने वाले संग्रहण के अनुरूप, अनुशंसित Gateway होस्ट संग्रहण:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

`.p8` फ़ाइल को कमिट न करें या उसे रिपॉज़िटरी चेकआउट के अंतर्गत न रखें।

## खोज पथ

### Bonjour (LAN)

iOS ऐप `local.` पर `_openclaw-gw._tcp` को खोजता है और कॉन्फ़िगर होने पर उसी वाइड-एरिया DNS-SD खोज डोमेन को भी खोजता है। समान LAN पर मौजूद Gateway, `local.` से स्वचालित रूप से दिखाई देते हैं; क्रॉस-नेटवर्क खोज बीकन प्रकार बदले बिना कॉन्फ़िगर किए गए वाइड-एरिया डोमेन का उपयोग कर सकती है।

### Tailnet (क्रॉस-नेटवर्क)

यदि mDNS अवरुद्ध है, तो यूनिकास्ट DNS-SD ज़ोन (कोई डोमेन चुनें; उदाहरण: `openclaw.internal.`) और Tailscale स्प्लिट DNS का उपयोग करें। CoreDNS उदाहरण के लिए [Bonjour](/hi/gateway/bonjour) देखें।

### मैन्युअल होस्ट/पोर्ट

Settings में, **Manual Host** सक्षम करें और Gateway होस्ट + पोर्ट (डिफ़ॉल्ट `18789`) दर्ज करें।

## एकाधिक Gateway

ऐप अपने साथ युग्मित प्रत्येक Gateway की रजिस्ट्री रखता है, ताकि आप दोबारा युग्मित किए बिना उनके बीच स्विच कर सकें:

- **Settings -> Gateway** सक्रिय Gateway को चिह्नित करते हुए **Paired Gateways** सूची दिखाता है। स्विच करने के लिए किसी प्रविष्टि पर टैप करें; ऐप वर्तमान सत्र समाप्त करता है और चयनित Gateway से दोबारा कनेक्ट होता है। एक से अधिक Gateway युग्मित होने पर कनेक्शन पंक्ति के पास त्वरित-स्विच मेन्यू दिखाई देता है।
- क्रेडेंशियल, TLS विश्वास निर्णय, प्रत्येक Gateway की प्राथमिकताएँ और कैश किया गया चैट इतिहास हर Gateway के लिए अलग-अलग संग्रहीत होते हैं। स्विच करने पर अलग-अलग Gateway की स्थिति कभी मिश्रित नहीं होती और पुश पंजीकरण सक्रिय Gateway का अनुसरण करता है।
- किसी युग्मित Gateway को स्वाइप करें (या उसका संदर्भ मेन्यू उपयोग करें) और उसे **Forget** करें, जिससे उसके क्रेडेंशियल, डिवाइस टोकन, TLS पिन और कैश की गई चैट हट जाती हैं।
- खोजे गए Gateway पर स्विच करने के लिए उनका नेटवर्क पर दिखाई देना आवश्यक है; मैन्युअल Gateway सहेजे गए होस्ट और पोर्ट से दोबारा कनेक्ट होते हैं।

## कैनवास + A2UI

iOS Node, WKWebView कैनवास रेंडर करता है। इसे नियंत्रित करने के लिए `node.invoke` का उपयोग करें:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

नोट:

- Gateway कैनवास होस्ट, Gateway HTTP सर्वर से `/__openclaw__/canvas/` और `/__openclaw__/a2ui/` उपलब्ध कराता है (`gateway.port` के समान पोर्ट, डिफ़ॉल्ट `18789`)।
- iOS Node अंतर्निहित स्कैफ़ोल्ड को कनेक्टेड डिफ़ॉल्ट दृश्य के रूप में बनाए रखता है। `canvas.a2ui.push` और `canvas.a2ui.reset` बंडल किए गए, ऐप-स्वामित्व वाले A2UI पेज का उपयोग करते हैं।
- दूरस्थ Gateway A2UI पेज iOS पर केवल रेंडर किए जाते हैं; नेटिव A2UI बटन क्रियाएँ केवल बंडल किए गए, ऐप-स्वामित्व वाले पेजों से स्वीकार की जाती हैं।
- `canvas.navigate` और `{"url":""}` के साथ अंतर्निहित स्कैफ़ोल्ड पर लौटें।

## Computer Use से संबंध

iOS ऐप एक मोबाइल Node सतह है, Codex Computer Use बैकएंड नहीं। Codex Computer Use और `cua-driver mcp`, MCP टूल के माध्यम से स्थानीय macOS डेस्कटॉप नियंत्रित करते हैं; iOS ऐप `canvas.*`, `camera.*`, `screen.*`, `location.*` और `talk.*` जैसे OpenClaw Node कमांड के माध्यम से iPhone क्षमताएँ उपलब्ध कराता है।

एजेंट अब भी Node कमांड लागू करके OpenClaw के माध्यम से iOS ऐप संचालित कर सकते हैं, लेकिन वे कॉल Gateway Node प्रोटोकॉल से गुजरती हैं और iOS की फ़ोरग्राउंड/बैकग्राउंड सीमाओं का पालन करती हैं। स्थानीय डेस्कटॉप नियंत्रण के लिए [Codex Computer Use](/hi/plugins/codex-computer-use) और iOS Node क्षमताओं के लिए इस पेज का उपयोग करें।

### कैनवास मूल्यांकन / स्नैपशॉट

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.eval --params '{"javaScript":"(() => { const {ctx} = window.__openclaw; ctx.clearRect(0,0,innerWidth,innerHeight); ctx.lineWidth=6; ctx.strokeStyle=\"#ff2d55\"; ctx.beginPath(); ctx.moveTo(40,40); ctx.lineTo(innerWidth-40, innerHeight-40); ctx.stroke(); return \"ok\"; })()"}'
```

```bash

```

## वॉइस सक्रियण + वार्ता मोड

- वॉइस सक्रियण और वार्ता मोड Settings में उपलब्ध हैं।
- जब `talk.realtime.transport`, `webrtc` होता है, तब OpenAI रीयलटाइम वार्ता क्लाइंट-स्वामित्व वाले WebRTC का उपयोग करती है; स्पष्ट `gateway-relay` कॉन्फ़िगरेशन का स्वामित्व Gateway के पास रहता है। [वार्ता मोड](/hi/nodes/talk) देखें।
- वार्ता-सक्षम iOS Node, `talk` क्षमता घोषित करते हैं और `talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` तथा `talk.ptt.once` घोषित कर सकते हैं; Gateway विश्वसनीय वार्ता-सक्षम Node के लिए उन पुश-टू-टॉक कमांड को डिफ़ॉल्ट रूप से अनुमति देता है।
- iOS बैकग्राउंड ऑडियो निलंबित कर सकता है; ऐप के सक्रिय न होने पर वॉइस सुविधाओं को सर्वोत्तम-प्रयास मानें।

## सामान्य त्रुटियाँ

- `NODE_BACKGROUND_UNAVAILABLE`: iOS ऐप को फ़ोरग्राउंड में लाएँ (कैनवास/कैमरा/स्क्रीन कमांड के लिए यह आवश्यक है)।
- `A2UI_HOST_UNAVAILABLE`: बंडल किया गया A2UI पेज ऐप WebView में उपलब्ध नहीं था; ऐप को Screen टैब पर फ़ोरग्राउंड में रखें और पुनः प्रयास करें।
- युग्मन संकेत कभी दिखाई नहीं देता: `openclaw devices list` चलाएँ और मैन्युअल रूप से अनुमोदन करें।
- Watch पर iPhone की स्थिति दिखाई नहीं देती: पुष्टि करें कि iPhone, `watch.status` में `watchPaired: true`
  और `watchAppInstalled: true` रिपोर्ट करता है। यदि युग्मन false है, तो Apple के Watch ऐप में
  Watch को युग्मित करें। यदि इंस्टॉलेशन false है, तो **My Watch -> Available Apps**
  से सहायक ऐप इंस्टॉल करें। किसी भी बदलाव के बाद, Watch पर OpenClaw को एक बार
  खोलें; तत्काल उपलब्धता के लिए दोनों ऐप का चलना अब भी आवश्यक है,
  जबकि कतारबद्ध अपडेट बाद में बैकग्राउंड में पहुँच सकते हैं।
- दोबारा इंस्टॉल करने के बाद पुनः-कनेक्शन विफल होता है: Keychain युग्मन टोकन मिटा दिया गया था; Node को फिर से युग्मित करें।
