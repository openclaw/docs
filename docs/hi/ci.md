OpenClaw CI `main` पर पुश होने पर (ट्रिगर पर Markdown और `docs/**` पथों को अनदेखा किया जाता है),
हर गैर-ड्राफ्ट पुल रिक्वेस्ट पर और मैन्युअल डिस्पैच पर चलता है।
कैनोनिकल `main` पुश एकल-उड़ान होते हैं: `CI` समवर्ती समूह एक
पूर्ण एकीकरण चक्र को चलने देता है, जबकि GitHub केवल नवीनतम लंबित पुश रखता है।
नए मर्ज उस लंबित रन को बदल देते हैं, बजाय उस कार्य को रद्द करने के जिसने पहले ही
Blacksmith मैट्रिक्स पंजीकृत कर लिया है। पुल रिक्वेस्ट अब भी अधिक्रमित हेड रद्द करती हैं,
और मैन्युअल डिस्पैच पृथक समूहों का उपयोग करते हैं। `preflight` डिफ़ को वर्गीकृत करता है और
केवल असंबंधित क्षेत्रों में बदलाव होने पर महंगे लेन बंद कर देता है। मैन्युअल
`workflow_dispatch` रन जानबूझकर स्मार्ट स्कोपिंग को बायपास करते हैं और रिलीज़ कैंडिडेट तथा व्यापक
सत्यापन के लिए पूरे ग्राफ़ को फैला देते हैं। Android लेन
`include_android` (या `release_gate` इनपुट) के माध्यम से ऑप्ट-इन रहते हैं। केवल-रिलीज़
Plugin कवरेज अलग
[`Plugin Prerelease`](#plugin-prerelease) वर्कफ़्लो में रहता है और केवल
[`Full Release Validation`](#full-release-validation) या स्पष्ट मैन्युअल
डिस्पैच से चलता है।

## शीघ्र-विफलता क्रम

1. `preflight` तय करता है कि कौन-से लेन अस्तित्व में होंगे। `docs-scope` और `changed-scope` तर्क इस जॉब के अंदर चरण हैं, स्वतंत्र जॉब नहीं। कैनोनिकल `main` तुरंत शुरू होता है, लेकिन उसका समवर्ती समूह केवल एक पूर्ण रन को प्रवेश देता है और बाद के पुश को एक नवीनतम लंबित रन में समेकित करता है। Node-संबंधित मुख्य पुश एकमात्र निर्भरता-डिस्क राइटर और उसके आकार के अनुरक्षण को भी यहीं क्रमबद्ध करते हैं, इससे पहले कि डाउनस्ट्रीम जॉब कुंजी माउंट कर सकें; Blacksmith किसी नए कमिट को केवल बाद के वर्कफ़्लो रन के लिए उपलब्ध करा सकता है, इसलिए उसी रन के उपभोक्ता मार्कर-जाँचा हुआ स्थानीय फ़ॉलबैक बनाए रखते हैं।
2. `security-fast`, `check-*`, `check-additional-*`, `check-docs` और `skills-python` भारी आर्टिफ़ैक्ट और प्लेटफ़ॉर्म मैट्रिक्स जॉब की प्रतीक्षा किए बिना शीघ्र विफल होते हैं।
3. `build-artifacts` और लोकेल जाँच तेज़ Linux लेन के साथ समानांतर चलती हैं। Control UI और नेटिव ऐप स्रोत PR जनरेट किए गए लोकेल स्नैपशॉट/संसाधनों को बाहर रखते हैं; उनके क्रमबद्ध रीफ़्रेश वर्कफ़्लो पृष्ठभूमि में पृथक जनरेट किए गए PR की मरम्मत और स्वचालित मर्ज करते हैं। स्रोत CI अब भी पुराने स्रोत इन्वेंटरी और असुरक्षित स्थानीयकरण कॉल को अवरुद्ध करता है। जनरेट किए गए PR, मैन्युअल CI और रिलीज़ तैयारी पूर्ण अनुवादित/प्लेटफ़ॉर्म-जनित समानता लागू करते हैं। कैनोनिकल `release/YYYY.M.PATCH` शाखाएँ अन्य जनरेट किए गए रिलीज़ आउटपुट के साथ रिलीज़-तैयारी लोकेल मरम्मत शामिल कर सकती हैं।
4. इसके बाद भारी प्लेटफ़ॉर्म और रनटाइम लेन फैलते हैं: `checks-fast-core`, `checks-fast-contracts-plugins-*`, `checks-fast-contracts-channels-*`, `checks-node-*`, `checks-windows`, `macos-node`, `macos-swift`, `ios-build` और `android`।
5. `openclaw/ci-gate` प्रत्येक चयनित लेन की प्रतीक्षा करता है। प्रीफ़्लाइट और सुरक्षा सफल होने चाहिए; डाउनस्ट्रीम जॉब केवल तभी स्किप हो सकते हैं जब मेनिफ़ेस्ट ने उन्हें न चुना हो। कोई विफल या रद्द किया गया चयनित लेन समुच्चय को विफल कर देता है।

मर्ज समन्वयक समान पुल-रिक्वेस्ट हेड के लिए प्रमाणीकृत सफल `openclaw/ci-gate` का
24 घंटे तक पुनः उपयोग कर सकता है। इससे असंबंधित `main` बदलावों के बाद
योगदानकर्ता शाखा को दोबारा लिखने से बचा जाता है। पुनः उपयोग योग्य परिणाम वर्तमान `main` के विरुद्ध
अलग, सख्त, ऐप-स्वामित्व वाली परीक्षण-मर्ज जाँच को प्रतिस्थापित नहीं करता।
बाद का लंबित या विफल पुनः रन ताज़गी अवधि के दौरान उस अपरिवर्तित हेड के
पहले के सफल परिणाम को नहीं मिटाता।

डिफ़ॉल्ट-ब्रांच नियम-समुच्चय के लिए GitHub Actions के स्वामित्व वाली `openclaw/ci-gate` जाँच आवश्यक है। रिपॉज़िटरी अनुरक्षकों और व्यवस्थापकों के पास ऑडिट किया गया आपातकालीन बायपास है, जो केवल हस्ताक्षरित प्रत्यक्ष फ़ास्ट-फ़ॉरवर्ड लैंडिंग के लिए अभिप्रेत है; संगठन का नियम-समुच्चय अब भी हटाने और नॉन-फ़ास्ट-फ़ॉरवर्ड अपडेट को अवरुद्ध करता है। सामान्य पुल रिक्वेस्ट मर्ज में विफल CI को बायपास करने के बजाय गेट का उपयोग जारी रहना चाहिए। अलग, सख्त App-स्वामित्व वाली टेस्ट-मर्ज जाँच अब भी हेड को वर्तमान `main` से बाँधती है।

जब कोई नया हेड लैंड होता है, तो GitHub प्रतिस्थापित पुल रिक्वेस्ट जॉब को `cancelled` के रूप में चिह्नित कर सकता है। जब तक उसी PR का नवीनतम रन भी विफल न हो, इसे CI शोर मानें। प्रवेश के बाद कैनोनिकल `main` रन रद्द नहीं किए जाते; मर्ज ट्रैफ़िक आने पर GitHub केवल पुराने लंबित रन को नवीनतम टिप से बदलता है। Matrix जॉब `fail-fast: false` का उपयोग करते हैं, और `build-artifacts` छोटे सत्यापनकर्ता जॉब को कतारबद्ध करने के बजाय एम्बेडेड चैनल, कोर-सपोर्ट-बाउंडरी और gateway-watch विफलताओं की सीधे रिपोर्ट करता है। स्वचालित CI समवर्तीता कुंजी संस्करणित है (`CI-v7-*`), ताकि किसी पुराने कतार समूह में GitHub-पक्ष का ज़ॉम्बी नए main रन को अनिश्चितकाल तक अवरुद्ध न कर सके। मैन्युअल पूर्ण-सुइट रन `CI-manual-v1-*` का उपयोग करते हैं और प्रगति पर चल रहे रन को रद्द नहीं करते। plugin-list स्टार्टअप-मेमोरी गार्ड स्वयं-होस्ट किए गए Blacksmith Linux पर 350 MiB की अधिकतम सीमा रखता है और GitHub-होस्ट किए गए Linux पर 425 MiB की अनुमति देता है, जहाँ समान निर्मित CLI के लिए RSS आधार-रेखा अधिक होती है।

GitHub Actions से कुल बीता समय, कतार समय, सबसे धीमे जॉब, विफलताओं और `pnpm-store-warmup` फ़ैनआउट अवरोध को संक्षेपित करने के लिए `pnpm ci:timings`, `pnpm ci:timings:recent`, या `node scripts/ci-run-timings.mjs <run-id>` का उपयोग करें। वर्कफ़्लो के भीतर `ci-timings-summary` जॉब `ci.yml` में मौजूद है, लेकिन फ़िलहाल अक्षम है (`if: false`); इसके बजाय टाइमिंग सहायक को स्थानीय रूप से चलाएँ। बिल्ड टाइमिंग के लिए, `build-artifacts` जॉब का `Build dist` चरण देखें: `pnpm build:ci-artifacts`, `[build-all] phase timings:` प्रिंट करता है और उसमें `ui:build` शामिल होता है; जॉब `startup-memory` आर्टिफ़ैक्ट भी अपलोड करता है।

## PR संदर्भ और साक्ष्य

बाहरी योगदानकर्ताओं के PR, यहाँ से PR संदर्भ और साक्ष्य गेट चलाते हैं:
`.github/workflows/real-behavior-proof.yml`। वर्कफ़्लो विश्वसनीय वर्कफ़्लो संशोधन
(`github.workflow_sha`) को चेक आउट करता है और केवल PR बॉडी का मूल्यांकन
करता है; यह योगदानकर्ता ब्रांच से कोड निष्पादित नहीं करता।

यह गेट उन PR लेखकों पर लागू होता है जो रिपॉज़िटरी के स्वामी, सदस्य,
सहयोगी या बॉट नहीं हैं। जब PR बॉडी में लेखक द्वारा लिखे गए
`What Problem This Solves` और `Evidence` अनुभाग होते हैं, तो यह पास हो जाता है। साक्ष्य कोई केंद्रित
टेस्ट, CI परिणाम, स्क्रीनशॉट, रिकॉर्डिंग, टर्मिनल आउटपुट, प्रत्यक्ष अवलोकन,
संशोधित लॉग या आर्टिफ़ैक्ट लिंक हो सकता है। बॉडी आशय और उपयोगी सत्यापन प्रदान करती है;
समीक्षक शुद्धता का आकलन करने के लिए कोड, टेस्ट और CI का निरीक्षण करते हैं।

जाँच विफल होने पर एक और कोड कमिट पुश करने के बजाय PR बॉडी अपडेट करें।

## दायरा और रूटिंग

दायरा तर्क `scripts/ci-changed-scope.mjs` में स्थित है और `src/scripts/ci-changed-scope.test.ts` में यूनिट टेस्ट द्वारा कवर किया गया है। मैन्युअल डिस्पैच परिवर्तित-दायरा पहचान छोड़ देता है और प्रीफ़्लाइट मैनिफ़ेस्ट को ऐसे कार्य कराता है मानो प्रत्येक दायरा-निर्धारित क्षेत्र बदल गया हो।

अलग-अलग iOS और macOS Periphery वर्कफ़्लो शून्य-निष्कर्ष डेड-कोड नीति लागू करते हैं। प्रत्येक केवल तभी चलता है जब कोई गैर-ड्राफ़्ट पुल रिक्वेस्ट उसके नेटिव स्कैन दायरे को छूती है, या जब उसे मैन्युअल रूप से डिस्पैच किया जाता है।

- **CI वर्कफ़्लो संपादन** Node CI ग्राफ़, वर्कफ़्लो लिंटिंग और Windows लेन को सत्यापित करते हैं (`ci.yml` इसे निष्पादित करता है), लेकिन स्वयं iOS, Android या macOS नेटिव बिल्ड को बाध्य नहीं करते; वे प्लेटफ़ॉर्म लेन प्लेटफ़ॉर्म स्रोत परिवर्तनों तक सीमित रहते हैं।
- **वर्कफ़्लो शुद्धता** सभी वर्कफ़्लो YAML फ़ाइलों पर `actionlint`, `zizmor`, कंपोज़िट-एक्शन इंटरपोलेशन गार्ड और कॉन्फ़्लिक्ट-मार्कर गार्ड चलाती है। PR-दायरा-निर्धारित `security-fast` जॉब परिवर्तित वर्कफ़्लो फ़ाइलों पर `zizmor` भी चलाता है, ताकि वर्कफ़्लो सुरक्षा निष्कर्ष मुख्य CI ग्राफ़ में जल्दी विफल हों।
- **`main` पुश पर दस्तावेज़** को स्टैंडअलोन `Docs` वर्कफ़्लो द्वारा CI में उपयोग किए जाने वाले समान ClawHub दस्तावेज़ मिरर के साथ जाँचा जाता है, इसलिए मिश्रित कोड+दस्तावेज़ पुश CI के `check-docs` शार्ड को भी कतारबद्ध नहीं करते। दस्तावेज़ बदलने पर पुल रिक्वेस्ट और मैन्युअल CI अब भी CI से `check-docs` चलाते हैं।
- **TUI PTY**, TUI परिवर्तनों के लिए `checks-node-core-runtime-tui-pty` Linux Node शार्ड में चलता है। शार्ड `OPENCLAW_TUI_PTY_INCLUDE_LOCAL=1` के साथ `test/vitest/vitest.tui-pty.config.ts` चलाता है, इसलिए यह नियतात्मक `TuiBackend` फ़िक्स्चर लेन और केवल बाहरी मॉडल एंडपॉइंट को मॉक करने वाले धीमे `tui --local` स्मोक—दोनों को कवर करता है।
- **केवल CI रूटिंग संपादन, फ़ास्ट टास्क द्वारा सीधे चलाए जाने वाले कोर-टेस्ट फ़िक्स्चर का छोटा समुच्चय और सीमित plugin अनुबंध सहायक संपादन** एक तेज़, केवल-Node मैनिफ़ेस्ट पथ का उपयोग करते हैं: `preflight`, `security-fast`, और केवल वे फ़ास्ट लेन जिन्हें परिवर्तन छूता है—एकल `checks-fast-core` CI-रूटिंग टास्क, दो plugin अनुबंध शार्ड, या दोनों। यह पथ बिल्ड आर्टिफ़ैक्ट, Node 22 संगतता, चैनल अनुबंध, पूर्ण कोर शार्ड, बंडल किए गए-plugin शार्ड और अतिरिक्त गार्ड Matrix को छोड़ देता है।
- **Windows Node जाँचें** Windows-विशिष्ट प्रोसेस/पथ रैपर, npm/pnpm/UI रनर सहायक, पैकेज मैनेजर कॉन्फ़िगरेशन और उस लेन को निष्पादित करने वाली CI वर्कफ़्लो सतहों तक सीमित हैं; असंबंधित स्रोत, plugin, इंस्टॉल-स्मोक और केवल-टेस्ट परिवर्तन Linux Node लेन पर बने रहते हैं।

सबसे धीमे Node टेस्ट परिवारों को विभाजित या संतुलित किया जाता है, ताकि प्रत्येक जॉब रनर को आवश्यकता से अधिक आरक्षित किए बिना छोटा बना रहे:

- Plugin अनुबंध और चैनल अनुबंध, प्रत्येक मानक GitHub रनर फ़ॉलबैक के साथ दो भारित Blacksmith-समर्थित शार्ड के रूप में चलते हैं।
- कोर यूनिट फ़ास्ट/सपोर्ट लेन अलग-अलग चलती हैं; कोर रनटाइम इन्फ़्रा प्रोसेस, साझा, हुक, सीक्रेट और तीन cron डोमेन शार्ड में विभाजित होता है।
- ऑटो-रिप्लाई संतुलित वर्कर के रूप में चलता है, जिसमें रिप्लाई सबट्री एजेंट-रनर, कमांड, डिस्पैच, सेशन और स्टेट-रूटिंग शार्ड में विभाजित होती है।
- एजेंटिक Gateway/सर्वर (कंट्रोल-प्लेन) कॉन्फ़िगरेशन, निर्मित आर्टिफ़ैक्ट की प्रतीक्षा करने के बजाय चैट, प्रमाणीकरण, मॉडल, HTTP/Plugin, रनटाइम और स्टार्टअप लेन में विभाजित होते हैं।
- सामान्य CI केवल पृथक इन्फ़्रा इन्क्लूड-पैटर्न शार्ड को अधिकतम 64 टेस्ट फ़ाइलों के नियतात्मक बंडलों में पैक करती है, जिससे गैर-पृथक कमांड/cron, स्टेटफ़ुल एजेंट्स-कोर या Gateway/सर्वर सुइट को मर्ज किए बिना Node मैट्रिक्स कम होता है। भारी निश्चित सुइट 8 vCPU पर बने रहते हैं, जबकि बंडल की गई और कम भार वाली लेन 4 vCPU का उपयोग करती हैं।
- कैनोनिकल रिपॉज़िटरी पर पुल रिक्वेस्ट, सिंथेटिक मर्ज्ड-ट्री डिफ़ के विरुद्ध परिवर्तित-टेस्ट रिज़ॉल्वर का पुनः उपयोग करती हैं। सटीक परिवर्तनों के लिए एक लक्षित Node जॉब चलता है; प्रत्येक चयनित टेस्ट फ़ाइल को अपनी प्रक्रिया मिलती है, ताकि स्टेटफ़ुल सुइट पृथक्करण अक्षुण्ण रहे। प्लानर सिबलिंग टेस्ट को इम्पोर्ट-ग्राफ़ आश्रितों के साथ जोड़ता है और वर्कस्पेस पैकेज, पैकेज/लॉकफ़ाइल, साझा हार्नेस, स्प्लिट-कॉन्फ़िग, नाम बदले या हटाए गए परिवर्तन, सार्वजनिक एक्सटेंशन-अनुबंध परिवर्तन, विशेष शार्ड सेटअप वाले टेस्ट, आंशिक रूप से रिज़ॉल्व किए गए या खाली लक्ष्य, अत्यधिक बड़े पाथ या लक्ष्य प्लान और प्लानर त्रुटियों के लिए मौजूदा 14-जॉब कॉम्पैक्ट पूर्ण-सुइट प्लान पर फ़ॉलबैक करता है। लक्षित प्लान हमेशा पूर्ण निर्मित-आर्टिफ़ैक्ट सीमा गेट बनाए रखते हैं, क्योंकि इसके रिपॉज़िटरी स्कैनर इम्पोर्ट से व्युत्पन्न नहीं किए जा सकते। `main` पुश समान पूर्ण कॉम्पैक्ट सुइट चलाते हैं: लंबित मध्यवर्ती पुश इवेंट एकत्रित किए जा सकते हैं, इसलिए नवीनतम शेष रन को केवल उसके अंतिम एकल-पुश डिफ़ के बजाय संपूर्ण एकीकरण ट्री को सत्यापित करना चाहिए। मैन्युअल डिस्पैच और रिलीज़ गेट पूर्ण नामित प्रति-शार्ड मैट्रिक्स बनाए रखते हैं।
- पूर्ण Node मैट्रिक्स लगातार धीमे सीरियल टूलिंग, ऑटो-रिप्लाई कमांड शार्ड और व्यापक कोर-फ़ास्ट कैश राइटर को पहले प्रवेश देता है। इससे 28-जॉब की सीमा बनी रहती है और क्रिटिकल-पाथ कार्य तथा अगले रन के ट्रांसफ़ॉर्म सीड को बाद की वेव में खिसकने से रोका जाता है।
- व्यापक ब्राउज़र, QA, मीडिया और विविध Plugin टेस्ट साझा Plugin कैच-ऑल के बजाय अपने समर्पित Vitest कॉन्फ़िगरेशन का उपयोग करते हैं। इन्क्लूड-पैटर्न शार्ड CI शार्ड नाम का उपयोग करके टाइमिंग प्रविष्टियाँ रिकॉर्ड करते हैं, ताकि `.artifacts/vitest-shard-timings.json` संपूर्ण कॉन्फ़िगरेशन को फ़िल्टर किए गए शार्ड से अलग पहचान सके।
- Linux Node शार्ड जॉब Vitest के प्रयोगात्मक फ़ाइलसिस्टम मॉड्यूल कैश को अपस्ट्रीम Actions कैश API के माध्यम से बनाए रखते हैं, जिसे Blacksmith अपने रनर पर पारदर्शी रूप से तेज़ करता है। प्रत्येक CI शार्ड केवल-रिस्टोर है और संरक्षित सीड को अपने रनर-स्थानीय रूट में अनपैक करता है; इसके बाद शार्ड रैपर समवर्ती Vitest प्रक्रियाओं को अलग-अलग सक्रिय सबडायरेक्टरी देता है। केवल रद्द न होने वाला दैनिक या स्पष्ट रूप से डिस्पैच किया गया वार्मर नया अपरिवर्तनीय आर्काइव सहेजता है, इसलिए पुल रिक्वेस्ट ट्रांसफ़ॉर्म प्रकाशित नहीं कर सकतीं या प्रति-PR कैश फ़ैमिली नहीं बना सकतीं। ट्रांसफ़ॉर्म-इनपुट फ़िंगरप्रिंट असंगत लॉकफ़ाइल, पैकेज, tsconfig और Vitest-कॉन्फ़िग पीढ़ियों को साफ़ करता है। संरक्षित राइटर अपने रिस्टोर किए गए कैश के 2 GiB से अधिक होने के बाद उसे स्कैन और प्रून करके 75% तक घटाता है। Vitest मॉड्यूल id, स्रोत सामग्री, परिवेश और रिज़ॉल्व किए गए ट्रांसफ़ॉर्म कॉन्फ़िगरेशन को हैश करता है, इसलिए सामान्य आंशिक स्रोत परिवर्तन अपरिवर्तित प्रविष्टियों को वार्म रखते हैं, जबकि परिवर्तित मॉड्यूल सुरक्षित रूप से मिस होते हैं। मोटे रिस्टोर प्रीफ़िक्स वर्कफ़्लो रन के बीच सेतु बनाते हैं; सामान्य Actions कैश LRU और निष्क्रियता निष्कासन पुराने अपरिवर्तनीय आर्काइव को सीमित करते हैं।
- विश्वसनीय Linux Node जॉब pnpm स्टोर और `node_modules` को प्रत्येक समर्थित Node लाइन के लिए एक संरक्षित डिपेंडेंसी डिस्क से भी बाइंड करते हैं। पैकेज मैनिफ़ेस्ट, इंस्टॉल सेटिंग, रनर प्लेटफ़ॉर्म और सटीक Node पैच डिस्क कुंजी से बाहर रहते हैं; एक सटीक रनटाइम और इंस्टॉल-इनपुट फ़िंगरप्रिंट तय करता है कि कोई जॉब ट्री का पुनः उपयोग करेगा या पुनः इंस्टॉल करके उसी डिस्क को रीफ़्रेश करेगा। हैश करने से पहले मैनिफ़ेस्ट को कैनोनिकल बनाया जाता है। ऑडिट किए गए प्रत्यक्ष रूट हुक केवल pnpm की इंस्टॉल लाइफ़साइकल स्क्रिप्ट बनाए रखते हैं, इसलिए फ़ॉर्मैटिंग और सामान्य टेस्ट/बिल्ड स्क्रिप्ट संपादन वार्म डिपेंडेंसी ट्री को बनाए रखते हैं; बिना ऑडिट वाला लाइफ़साइकल-हुक ड्रिफ़्ट तब तक फ़ेल-क्लोज़्ड रहता है, जब तक उसके स्रोत इनपुट फ़िंगरप्रिंट अनुबंध में शामिल न हो जाएँ। डिपेंडेंसी, पैकेज-मैनेजर, हुक-स्रोत और लॉकफ़ाइल परिवर्तन हमेशा स्नैपशॉट को अमान्य करते हैं। मेल खाता फ़िंगरप्रिंट आवश्यक है, पर पर्याप्त नहीं: सेटअप इम्पोर्टर आर्काइव और मैनिफ़ेस्ट चेकसम भी जाँचता है, फिर पोस्टइंस्टॉल द्वारा बनाए रखी गई रजिस्ट्री-समर्थित लॉकफ़ाइल डिपेंडेंसी को उन पैकेज मैनिफ़ेस्ट के विरुद्ध सत्यापित करता है जिन्हें Node उनके इम्पोर्टर से रिज़ॉल्व करता है। अनुपस्थित या पुरानी इम्पोर्टर सामग्री रूट होइस्ट देने के बजाय नए इंस्टॉल पर फ़ॉलबैक करती है। ऐसी पुल रिक्वेस्ट जिसका केवल-पठन स्नैपशॉट अनुपयोगी है, वर्कस्पेस बाइंड को अलग करके रनर-स्थानीय स्टोरेज में इंस्टॉल करती है, जिससे ऐसे क्लोन पर धीमे लेखन से बचा जाता है जिसे वह प्रकाशित नहीं कर सकती। स्टिकी कोल्ड इंस्टॉल pnpm के आंतरिक फ़ेच पुनः प्रयासों को अक्षम करते हैं और क्रमशः वार्म होते स्टोर से अधिकतम तीन सीमित पूर्ण-इंस्टॉल प्रयास करते हैं; टाइमआउट फिर भी विफलता रहता है। सामग्री-सत्यापित रिस्टोर या फ़्रोज़न-लॉकफ़ाइल इंस्टॉल के बाद, सेटअप pnpm की अनावश्यक प्री-रन डिपेंडेंसी जाँच अक्षम करता है: रिपॉज़िटरी जानबूझकर Plugin-स्थानीय `node_modules` को प्रून करती है, जिसे pnpm अन्यथा पुराना मानकर शार्ड फ़ैनआउट के दौरान असुरक्षित समवर्ती अप्रत्यक्ष इंस्टॉल के माध्यम से सुधारता है। कैनोनिकल मुख्य प्रीफ़्लाइट एकमात्र राइटर है और प्रत्येक रीफ़्रेश पर स्टोर मापता है, तथा सेवानिवृत्त पैकेज संस्करणों द्वारा इसे 8 GiB से ऊपर पहुँचाने के बाद ही `pnpm store prune` चलाता है। राइटर जॉब पूर्ण होने के बाद भी Blacksmith स्नैपशॉट प्रकाशन असिंक्रोनस होता है, इसलिए नई कुंजी या फ़िंगरप्रिंट के बाद पहला रन कोल्ड रह सकता है; बाद के सामग्री-सत्यापित सटीक-मार्कर रिस्टोर रोलआउट का प्रमाण होते हैं। आवश्यक CI जॉब और पुल रिक्वेस्ट को डिस्पोज़ेबल क्लोन मिलते हैं, इसलिए डिपेंडेंसी परिवर्तन नई डिस्क, प्रतिस्पर्धी स्नैपशॉट या बिल्ड रद्द कर सकने वाला कैश लॉक नहीं बनाते।
- Node शार्ड और बिल्ड-आर्टिफ़ैक्ट जॉब अपरिवर्तनीय Actions कैश के माध्यम से Node के पोर्टेबल ऑन-डिस्क कंपाइल कैश को भी रिस्टोर करते हैं। स्वतंत्र `test` और `build` नेमस्पेस उनके राइटर को एक-दूसरे के आर्काइव बदलने से रोकते हैं: निर्धारित टेस्ट वार्मर संरक्षित टेस्ट सीड का स्वामी है, जबकि `build-artifacts` विश्वसनीय `main` पुश से प्रति UTC दिन अधिकतम एक संरक्षित बिल्ड आर्काइव प्रकाशित कर सकता है। PR और सामान्य टेस्ट जॉब केवल संरक्षित स्नैपशॉट पढ़ते हैं, इसलिए फ़ीचर-ब्रांच बाइटकोड कभी साझा सीड में प्रवेश नहीं करता और PR ट्रैफ़िक कोई कैश आर्काइव नहीं बनाता। इससे अलग-अलग चेकआउट पाथ पर Node द्वारा लोड किए गए ऑर्केस्ट्रेशन, बिल्ड टूलिंग और बाहरी डिपेंडेंसी के लिए V8 बाइटकोड का पुनः उपयोग होता है, जिसमें स्रोत ग्राफ़ का केवल एक भाग बदलने की स्थिति भी शामिल है। Vitest चाइल्ड प्रक्रियाएँ इनहेरिट किया गया कंपाइल कैश अक्षम करती हैं, क्योंकि डायनेमिक कॉन्फ़िगरेशन के भीतर कवरेज सक्षम किया जा सकता है और बाइटकोड से स्क्रिप्ट डीसीरियलाइज़ किए जाने पर V8 कवरेज स्रोत-स्थिति की सटीकता खो सकता है।
- बिल्ड-आर्टिफ़ैक्ट जॉब सामग्री-फ़िंगरप्रिंट किए गए `build-all` चरण आउटपुट भी बनाए रखता है। CI की स्वयं निर्मित Plugin SDK घोषणाएँ रिपॉज़िटरी के स्वामित्व वाले संपूर्ण TypeScript/JSON स्रोत ग्राफ़ को हैश करती हैं, इंस्टॉल की गई और जनरेट की गई डायरेक्टरी को बाहर रखती हैं, और `tsdown` द्वारा `dist` साफ़ करने के बाद फ़्लैट घोषणाओं तथा पैकेज ब्रिज दोनों को रिस्टोर करती हैं। उस ग्राफ़ से बाहर के दस्तावेज़ीकरण, वर्कफ़्लो, Plugin और अन्य परिवर्तन घोषणा स्नैपशॉट का पुनः उपयोग कर सकते हैं; स्रोत परिवर्तन एक्सपोर्ट गेट चलने से पहले इसे पुनः बनाते हैं।
- पूर्ण घोषणा बिल्ड `tsdown` को AI, वर्कस्पेस-पैकेज और एकीकृत समूहों में विभाजित करते हैं। प्रत्येक समूह केवल घोषणाएँ कैश करता है, फिर उन घोषणाओं को रिस्टोर करने से पहले रनटाइम JavaScript को फिर भी पुनः बनाता है। इसलिए कोर या Plugin परिवर्तन केवल बड़े एकीकृत ग्राफ़ को अमान्य करते हैं, जबकि वर्कस्पेस-पैकेज परिवर्तन सावधानीपूर्वक प्रत्येक आश्रित घोषणा समूह को अमान्य करते हैं। सार्वजनिक पूर्ण बिल्ड सामान्यतः अपरिवर्तनीय Actions कैश का उपयोग करते हैं; मोटी रिस्टोर कुंजियाँ आंशिक परिवर्तनों को सीड करती हैं, प्रति-समूह सामग्री फ़िंगरप्रिंट पुराने डेटा को अस्वीकार करते हैं, और GitHub का कैश कोटा पुरानी पीढ़ियों को निष्कासित करता है। इसके बजाय साप्ताहिक Node 22 लेन सफल `main` रन के बाद 14-दिवसीय आर्टिफ़ैक्ट प्रकाशित करती है और केवल उन आर्टिफ़ैक्ट को रिस्टोर करती है जिनकी अपरिवर्तनीय निर्माता पहचान `main` पर उस वर्कफ़्लो को रिज़ॉल्व होती है, जिससे PR कोड को साझा कैश में लिखने की अनुमति दिए बिना कोटा चर्न से बचा जाता है। निजी-QA घोषणाएँ Actions कैश में कभी बनाए नहीं रखी जातीं, क्योंकि कैश नेमस्पेस गोपनीयता सीमाएँ नहीं हैं।
- `check-additional-*` पूरक सीमा गार्ड सूची (`scripts/run-additional-boundary-checks.mjs`) को एक प्रॉम्प्ट-भारी शार्ड (`check-additional-boundaries-a`, जिसमें Codex प्रॉम्प्ट स्नैपशॉट ड्रिफ़्ट जाँच शामिल है) और शेष स्ट्राइप के लिए एक संयुक्त शार्ड (`check-additional-boundaries-bcd`) में स्ट्राइप करता है, जिनमें से प्रत्येक स्वतंत्र गार्ड को समवर्ती रूप से चलाता और प्रति-जाँच टाइमिंग प्रिंट करता है। पैकेज-सीमा कंपाइल/कैनरी कार्य साथ रहता है, और रनटाइम टोपोलॉजी आर्किटेक्चर `build-artifacts` में अंतर्निहित Gateway वॉच कवरेज से अलग चलता है।
- 32-vCPU स्व-होस्टेड बिल्ड रनर पर, Gateway वॉच, चैनल टेस्ट और कोर सपोर्ट-बाउंड्री शार्ड `dist/` तथा `dist-runtime/` के पहले से निर्मित हो जाने के बाद `build-artifacts` के भीतर एक साथ शुरू होते हैं। GitHub-होस्टेड फ़ॉलबैक रन Gateway वॉच को सीरियल रखते हैं, ताकि कम-कोर प्रतिस्पर्धा उसकी तत्परता समय-सीमा का उपभोग न कर सके।

जॉब बजट साझा करते हैं।

Android CI `testPlayDebugUnitTest` और `testThirdPartyDebugUnitTest` दोनों चलाती है और फिर Play डीबग APK बनाती है। तृतीय-पक्ष फ़्लेवर का कोई अलग स्रोत सेट या मैनिफ़ेस्ट नहीं है; उसकी यूनिट-टेस्ट लेन फिर भी SMS/कॉल-लॉग BuildConfig फ़्लैग के साथ फ़्लेवर को कंपाइल करती है, जबकि प्रत्येक Android-संबंधित पुश पर डुप्लिकेट डीबग APK पैकेजिंग जॉब से बचती है। प्रत्येक वर्तमान Gradle टास्क की एक संरक्षित स्टिकी डिस्क है; PR जॉब डिस्पोज़ेबल क्लोन का उपयोग करते हैं, जबकि संरक्षित रन सामग्री-संबोधित Gradle प्रविष्टियों को उसी स्थान पर रीफ़्रेश करते हैं।

Blacksmith स्टिकी-डिस्क कुंजियाँ जानबूझकर समर्थित रनटाइम या टास्क आयामों तक सीमित होती हैं, कभी भी PR संख्या, कमिट, रन, ब्रांच या डिपेंडेंसी हैश तक नहीं। रनटाइम ट्रांसफ़ॉर्म और कंपाइल कैश स्टिकी डिस्क के बजाय Actions कैश का उपयोग करते हैं, क्योंकि अपरिवर्तनीय आर्काइव सत्यापन योग्य रिस्टोर/सेव परिणाम दिखाते हैं और परिवर्तनीय स्नैपशॉट-प्रमोशन विफलताओं से बचते हैं। स्टिकी कुंजी-संस्करण माइग्रेशन के बाद, `.github/retired-sticky-disks.json` में केवल सटीक अप्रचलित कुंजी, आर्किटेक्चर और क्षेत्र पहचान जोड़ें, समान आयामों और पुष्टिकरण के साथ `main` से `Sticky Disk Cleanup` डिस्पैच करें, विलोपन सत्यापित करें और फिर उन प्रविष्टियों को हटा दें। वर्कफ़्लो ARM पहचानों को ARM रनर पर रूट करता है, रनर-क्षेत्र बेमेल को अस्वीकार करता है, Blacksmith की सटीक-कुंजी विलोपन कार्रवाई का उपयोग करता है और Docker बिल्डर कैश या वाइल्डकार्ड प्रीफ़िक्स कभी नहीं हटाता। Actions कैश आर्काइव सामान्य LRU और निष्क्रियता निष्कासन का उपयोग करते हैं।

`check-dependencies` शार्ड उत्पादन Knip डिपेंडेंसी, अप्रयुक्त-फ़ाइल और अप्रयुक्त-एक्सपोर्ट जाँच चलाता है। अप्रयुक्त-फ़ाइल गार्ड तब विफल होता है जब कोई PR नई, बिना समीक्षा वाली अप्रयुक्त फ़ाइल जोड़ता है या पुरानी अलाउलिस्ट प्रविष्टि छोड़ता है, जबकि उन अभिप्रेत डायनेमिक Plugin, जनरेट किए गए, बिल्ड, लाइव-टेस्ट और पैकेज ब्रिज सतहों को बनाए रखता है जिन्हें Knip स्थैतिक रूप से रिज़ॉल्व नहीं कर सकता। अप्रयुक्त-एक्सपोर्ट गार्ड टेस्ट-सपोर्ट फ़ाइलों को बाहर रखता है और प्रत्येक अप्रयुक्त उत्पादन एक्सपोर्ट पर विफल होता है; अभिप्रेत डायनेमिक उपभोक्ताओं को `config/knip.config.ts` में मॉडल किया जाना चाहिए। ऐतिहासिक लक्ष्य उपलब्ध होने पर एक्सपोर्ट गार्ड चलाते हैं और अन्यथा अपना पुराना डेड-कोड फ़ॉलबैक बनाए रखते हैं।

## ClawSweeper गतिविधि फ़ॉरवर्डिंग

`.github/workflows/clawsweeper-dispatch.yml` OpenClaw रिपॉज़िटरी गतिविधि से ClawSweeper तक लक्ष्य-पक्षीय ब्रिज है। यह अविश्वसनीय पुल रिक्वेस्ट कोड को चेक आउट या निष्पादित नहीं करता। वर्कफ़्लो `CLAWSWEEPER_APP_PRIVATE_KEY` से एक GitHub App टोकन बनाता है, फिर संक्षिप्त `repository_dispatch` पेलोड को `openclaw/clawsweeper` पर डिस्पैच करता है।

वर्कफ़्लो में चार लेन हैं:

- `clawsweeper_item` सटीक इश्यू और पुल रिक्वेस्ट समीक्षा अनुरोधों के लिए;
- `clawsweeper_comment` इश्यू टिप्पणियों में स्पष्ट ClawSweeper कमांड के लिए;
- `clawsweeper_commit_review` `main` पुश पर कमिट-स्तरीय समीक्षा अनुरोधों के लिए;
- `github_activity` सामान्य GitHub गतिविधि के लिए, जिसका ClawSweeper एजेंट निरीक्षण कर सकता है।

`github_activity` लेन केवल सामान्यीकृत मेटाडेटा अग्रेषित करती है: इवेंट प्रकार, कार्रवाई, अभिनेता, रिपॉज़िटरी, आइटम संख्या, URL, शीर्षक, स्थिति, और उपलब्ध होने पर टिप्पणियों या समीक्षाओं के छोटे अंश। यह जानबूझकर संपूर्ण Webhook बॉडी अग्रेषित करने से बचती है। `openclaw/clawsweeper` में प्राप्तकर्ता वर्कफ़्लो `.github/workflows/github-activity.yml` है, जो सामान्यीकृत इवेंट को ClawSweeper एजेंट के लिए OpenClaw Gateway हुक पर पोस्ट करता है।

सामान्य गतिविधि अवलोकन है, डिफ़ॉल्ट रूप से डिलीवरी नहीं। ClawSweeper एजेंट को उसके प्रॉम्प्ट में Discord लक्ष्य मिलता है और उसे `#clawsweeper` पर केवल तभी पोस्ट करना चाहिए जब इवेंट अप्रत्याशित, कार्रवाई योग्य, जोखिमपूर्ण या संचालन की दृष्टि से उपयोगी हो। नियमित रूप से खोलने, संपादन, बॉट गतिविधि, डुप्लिकेट Webhook शोर और सामान्य समीक्षा ट्रैफ़िक का परिणाम `NO_REPLY` होना चाहिए।

इस पूरे पथ में GitHub शीर्षकों, टिप्पणियों, बॉडी, समीक्षा टेक्स्ट, ब्रांच नामों और कमिट संदेशों को अविश्वसनीय डेटा मानें। वे सारांश और ट्रायेज के लिए इनपुट हैं, वर्कफ़्लो या एजेंट रनटाइम के निर्देश नहीं।

## मैन्युअल डिस्पैच

मैन्युअल CI डिस्पैच सामान्य CI वाला ही जॉब ग्राफ़ चलाते हैं, लेकिन Android के अलावा प्रत्येक दायरा-निर्धारित लेन को अनिवार्य रूप से चालू करते हैं: Linux Node शार्ड, बंडल किए गए Plugin शार्ड, Plugin और चैनल अनुबंध शार्ड, Node 22 संगतता, `check-*`, `check-additional-*`, निर्मित आर्टिफ़ैक्ट स्मोक जाँच, दस्तावेज़ जाँच, Python Skills, Windows, macOS, iOS बिल्ड और Control UI/नेटिव ऐप i18n। स्वचालित स्रोत PR समान PR में अनुवादित या प्लेटफ़ॉर्म-जनित आउटपुट की आवश्यकता के बिना नेटिव निष्कर्षण इन्वेंट्री और Android/Apple स्थानीयकरण सुरक्षा सत्यापित करते हैं। क्रमबद्ध Native App Locale Refresh वर्कफ़्लो उन आर्टिफ़ैक्ट को एक पृथक PR में दोबारा बनाता है और आवश्यक जाँच सफल होने के बाद सटीक हेड का स्वतः-मर्ज सक्षम करता है। जनित-आर्टिफ़ैक्ट PR, मैन्युअल CI, Full Release Validation और रिलीज़ की तैयारी के लिए पूर्ण नेटिव समतुल्यता अवरोधक बनी रहती है। स्वचालित PR और `main` रन पर Control UI लोकेल समतुल्यता परामर्शात्मक और मैन्युअल/रिलीज़ CI पर अवरोधक बनी रहती है। स्वतंत्र मैन्युअल CI डिस्पैच Android को केवल `include_android=true` के साथ चलाते हैं (`release_gate` इनपुट भी Android को अनिवार्य करता है); पूर्ण रिलीज़ अम्ब्रेला `include_android=true` पास करके Android सक्षम करता है। Plugin प्रीरिलीज़ स्थैतिक जाँच, केवल-रिलीज़ `agentic-plugins` शार्ड, संपूर्ण एक्सटेंशन बैच स्वीप और Plugin प्रीरिलीज़ Docker लेन CI से बाहर रखे गए हैं। Docker प्रीरिलीज़ सुइट केवल तभी चलता है जब `Full Release Validation` रिलीज़-सत्यापन गेट सक्षम करके अलग `Plugin Prerelease` वर्कफ़्लो डिस्पैच करता है।

PR अधिकतम-पंक्ति जाँच चेक आउट किए गए सिंथेटिक मर्ज ट्री से बेसलाइन प्राप्त करती हैं और इवेंट हेड के विरुद्ध उसके हेड पैरेंट को सत्यापित करती हैं। मैन्युअल रन एक अद्वितीय समवर्ती समूह का उपयोग करते हैं, ताकि रिलीज़-कैंडिडेट पूर्ण सुइट उसी रेफ़ पर किसी अन्य पुश या PR रन के कारण रद्द न हो। वैकल्पिक `target_ref` इनपुट किसी विश्वसनीय कॉलर को चयनित डिस्पैच रेफ़ की वर्कफ़्लो फ़ाइल का उपयोग करते हुए किसी ब्रांच, टैग या पूर्ण कमिट SHA के विरुद्ध वह ग्राफ़ चलाने देता है; अधिकतम-पंक्ति बेसलाइन की तुलना उस रन के लिए निर्धारित डिफ़ॉल्ट-ब्रांच हेड के विरुद्ध लक्ष्य के मर्ज बेस से की जाती है। `release_gate` इनपुट क्षमता के कारण रुके हुए PR CI के लिए सटीक-SHA मेंटेनर फ़ॉलबैक है: इसके लिए आवश्यक है कि `target_ref` एक पूर्ण कमिट SHA हो, जो डिस्पैच की गई ब्रांच के हेड से मेल खाए, और `pull_request_number` उस खुले PR की पहचान करे जिसके मर्ज ट्री को सत्यापित किया जाता है।

```bash
gh workflow run ci.yml --ref release/YYYY.M.PATCH
gh workflow run ci.yml --ref main -f target_ref=<branch-or-sha> -f include_android=true
gh workflow run full-release-validation.yml --ref main -f ref=<branch-or-sha>
```

मासिक केवल-npm विस्तारित-स्थिर पथ इसका अपवाद है: सटीक
`extended-stable/YYYY.M.33` ब्रांच से `OpenClaw NPM
Release` प्रीफ़्लाइट और `Full Release Validation` दोनों डिस्पैच करें, उनकी रन ID सुरक्षित रखें और दोनों ID
प्रत्यक्ष npm प्रकाशन रन को पास करें। कमांड, सटीक पहचान आवश्यकताओं, रजिस्ट्री रीडबैक और सिलेक्टर
सुधार प्रक्रिया के लिए [मासिक केवल-npm विस्तारित-स्थिर
प्रकाशन](/hi/reference/RELEASING#monthly-npm-only-extended-stable-publication) देखें। यह पथ Plugin, macOS, Windows, GitHub
Release, निजी dist-tag या अन्य प्लेटफ़ॉर्म प्रकाशन डिस्पैच नहीं करता।

## रनर पंजीकरण बजट

`ghx api rate_limit` में OpenClaw का वर्तमान GitHub रनर-पंजीकरण बकेट प्रति 5 मिनट में 10,000 स्व-होस्टेड
रनर पंजीकरण रिपोर्ट करता है। प्रत्येक ट्यूनिंग चरण से पहले
`actions_runner_registration` दोबारा जाँचें, क्योंकि GitHub इस बकेट को बदल सकता है।
यह सीमा `openclaw` संगठन के सभी Blacksmith रनर पंजीकरणों द्वारा साझा की जाती है, इसलिए कोई अन्य Blacksmith इंस्टॉलेशन जोड़ने से
नया बकेट नहीं जुड़ता।

बर्स्ट नियंत्रण के लिए Blacksmith लेबल को दुर्लभ संसाधन मानें। जो जॉब
केवल रूट, सूचित, सारांशित या शार्ड चुनते हैं, अथवा छोटे CodeQL स्कैन चलाते हैं, उन्हें
GitHub-होस्टेड रनर पर ही रहना चाहिए, जब तक उनकी मापी गई Blacksmith-विशिष्ट
आवश्यकताएँ न हों। किसी भी नए Blacksmith मैट्रिक्स, बड़े `max-parallel` या उच्च-आवृत्ति
वर्कफ़्लो को अपनी सर्वाधिक खराब स्थिति की पंजीकरण संख्या दिखानी होगी और संगठन-स्तरीय
लक्ष्य को लाइव बकेट के लगभग 60% से नीचे रखना होगा। वर्तमान 10,000-पंजीकरण
बकेट के साथ इसका अर्थ 6,000-पंजीकरण संचालन लक्ष्य है, जिससे
समवर्ती रिपॉज़िटरी, पुनः प्रयास और बर्स्ट ओवरलैप के लिए गुंजाइश बचती है।

परिवर्तित-लक्ष्य PR योजना सामान्य Node परीक्षण बर्स्ट को 14 Blacksmith पंजीकरण से घटाकर एक करती है। व्यापक-जोखिम वाले PR 14-पंजीकरण वाला संक्षिप्त फ़ॉलबैक बनाए रखते हैं, इसलिए सर्वाधिक खराब स्थिति नहीं बढ़ती।

## OpenClaw प्रदर्शन

`OpenClaw Performance` उत्पाद/रनटाइम प्रदर्शन वर्कफ़्लो है। यह `main` पर प्रतिदिन चलता है और इसे मैन्युअल रूप से डिस्पैच किया जा सकता है:

```bash
gh workflow run openclaw-performance.yml --ref main -f profile=diagnostic -f repeat=3
gh workflow run openclaw-performance.yml --ref main -f profile=smoke -f repeat=1 -f deep_profile=true -f live_openai_candidate=true
gh workflow run openclaw-performance.yml --ref main -f target_ref=v2026.5.2 -f profile=diagnostic -f repeat=3
```

मैन्युअल डिस्पैच सामान्यतः वर्कफ़्लो रेफ़ का बेंचमार्क करता है। वर्तमान वर्कफ़्लो कार्यान्वयन के साथ किसी रिलीज़ टैग या अन्य ब्रांच का बेंचमार्क करने के लिए `target_ref` सेट करें। प्रकाशित रिपोर्ट पाथ और नवीनतम पॉइंटर परीक्षण किए गए रेफ़ के अनुसार कुंजीबद्ध होते हैं, और प्रत्येक `index.md` परीक्षण किए गए रेफ़/SHA, वर्कफ़्लो रेफ़/SHA, Kova रेफ़, प्रोफ़ाइल, लेन प्रमाणीकरण मोड, मॉडल, पुनरावृत्ति संख्या और परिदृश्य फ़िल्टर दर्ज करता है।

वर्कफ़्लो OCM को पिन किए गए रिलीज़ से और Kova को पिन किए गए `kova_ref` इनपुट पर `openclaw/Kova` से इंस्टॉल करता है, फिर तीन लेन चलाता है:

- `mock-provider`: नियतात्मक नकली OpenAI-संगत प्रमाणीकरण वाले लोकल-बिल्ड रनटाइम के विरुद्ध Kova नैदानिक परिदृश्य।
- `mock-deep-profile`: स्टार्टअप, Gateway और एजेंट-टर्न हॉटस्पॉट के लिए CPU/हीप/ट्रेस प्रोफ़ाइलिंग। शेड्यूल पर, या `deep_profile=true` के साथ डिस्पैच पर चलता है।
- `live-openai-candidate`: एक वास्तविक OpenAI `openai/gpt-5.6-luna` एजेंट टर्न, जो `OPENAI_API_KEY` अनुपलब्ध होने पर छोड़ दिया जाता है। शेड्यूल पर, या `live_openai_candidate=true` के साथ डिस्पैच पर चलता है।

मॉक-प्रोवाइडर लेन Kova पास के बाद OpenClaw-नेटिव स्रोत प्रोब भी चलाती है: डिफ़ॉल्ट, छोड़े गए चैनल, आंतरिक हुक और पचास-Plugin स्टार्टअप मामलों में Gateway बूट समय और मेमोरी; बंडल किए गए Plugin इंपोर्ट RSS, बार-बार चलने वाले मॉक-OpenAI `channel-chat-baseline` हेलो लूप, बूट किए गए Gateway के विरुद्ध CLI स्टार्टअप कमांड और SQLite स्टेट स्मोक प्रदर्शन प्रोब। जब परीक्षण किए गए रेफ़ के लिए पिछली प्रकाशित मॉक-प्रोवाइडर स्रोत रिपोर्ट उपलब्ध होती है, तो स्रोत सारांश मौजूदा RSS और हीप मानों की उस बेसलाइन से तुलना करता है और RSS में बड़ी वृद्धियों को `watch` के रूप में चिह्नित करता है। स्रोत प्रोब Markdown सारांश रिपोर्ट बंडल में `source/index.md` पर रहता है और उसके साथ रॉ JSON होता है।

प्रत्येक लेन CPU, हीप, ट्रेस और संपीड़ित नैदानिक बंडल सहित अपना पूर्ण GitHub आर्टिफ़ैक्ट अपलोड करती है। एक अलग प्रकाशक जॉब उन आर्टिफ़ैक्ट को डाउनलोड और सत्यापित करता है, फिर केवल `openclaw/clawgrit-reports` सामग्री तक सीमित अल्पकालिक ClawSweeper GitHub App टोकन बनाता है और उसे केवल Git पुश चरण को देता है। यह `report.json`, `report.md`, `index.md`, स्रोत-प्रोब आर्टिफ़ैक्ट और बंडल मेटाडेटा/चेकसम को `openclaw-performance/<tested-ref>/<run-id>-<attempt>/<lane>/` के अंतर्गत कमिट करता है; पूर्ण नैदानिक आर्काइव लिंक किए गए Actions आर्टिफ़ैक्ट में रहता है। प्रकाशक पुश का प्रयास करने से पहले 50 MB से बड़ी किसी भी रिपोर्ट फ़ाइल को अस्वीकार कर देता है। परीक्षण किए गए मौजूदा रेफ़ का पॉइंटर `openclaw-performance/<tested-ref>/latest-<lane>.json` है। यदि ऐप-टोकन निर्माण या रिपोर्ट प्रकाशन विफल होता है, तो शेड्यूल किए गए रन और `profile=release` डिस्पैच विफल हो जाते हैं। प्रमाणीकरण या प्रकाशन विफल होने पर मैन्युअल गैर-रिलीज़ डिस्पैच प्रकाशन को परामर्शात्मक रखते हैं और GitHub आर्टिफ़ैक्ट बनाए रखते हैं। पिछली स्रोत बेसलाइन सार्वजनिक रिपोर्ट रिपॉज़िटरी से अनाम रूप से प्राप्त की जाती है, इसलिए बेसलाइन का सफलतापूर्वक प्राप्त होना प्रकाशक प्रमाणीकरण को प्रमाणित नहीं करता।

## पूर्ण रिलीज़ सत्यापन

`Full Release Validation` "रिलीज़ से पहले सब कुछ चलाएँ" के लिए मैन्युअल अंब्रेला वर्कफ़्लो है। यह किसी ब्रांच, टैग या पूर्ण कमिट SHA को स्वीकार करता है, उस लक्ष्य के साथ मैन्युअल `CI` वर्कफ़्लो (Android सहित) डिस्पैच करता है, केवल-रिलीज़ Plugin/पैकेज/स्टैटिक/Docker प्रमाण के लिए `Plugin Prerelease` डिस्पैच करता है, लक्ष्य SHA के विरुद्ध `OpenClaw Performance` डिस्पैच करता है और इंस्टॉल स्मोक, पैकेज स्वीकृति, क्रॉस-OS पैकेज जाँच, QA Lab समानता, Matrix, Telegram और गेट की गई Discord, WhatsApp तथा Slack लेन के लिए `OpenClaw Release Checks` डिस्पैच करता है (परामर्शात्मक परिपक्वता स्कोरकार्ड रेंडरिंग `run_maturity_scorecard` के माध्यम से ऑप्ट-इन है)। स्थिर और पूर्ण प्रोफ़ाइल में हमेशा व्यापक लाइव/E2E और Docker रिलीज़-पाथ सोक कवरेज शामिल होता है; बीटा प्रोफ़ाइल `run_release_soak=true` के साथ ऑप्ट इन कर सकती है। मानक पैकेज Telegram E2E पैकेज स्वीकृति के भीतर चलता है, इसलिए पूर्ण उम्मीदवार कोई डुप्लिकेट लाइव पोलर शुरू नहीं करता। प्रकाशन के बाद रिलीज़ जाँच, पैकेज स्वीकृति, Docker, क्रॉस-OS और Telegram में भेजे गए npm पैकेज को पुनः उपयोग करने के लिए `release_package_spec` दें, ताकि दोबारा बिल्ड न करना पड़े। केवल केंद्रित प्रकाशित-पैकेज Telegram पुनः रन के लिए `npm_telegram_package_spec` का उपयोग करें। Codex Plugin लाइव पैकेज लेन डिफ़ॉल्ट रूप से उसी चयनित स्थिति का उपयोग करती है: प्रकाशित `release_package_spec=openclaw@<tag>`, `codex_plugin_spec=npm:@openclaw/codex@<tag>` व्युत्पन्न करता है, जबकि SHA/आर्टिफ़ैक्ट रन चयनित रेफ़ से `extensions/codex` पैक करते हैं। `npm:`, `npm-pack:` या `git:` स्पेक जैसे कस्टम Plugin स्रोतों के लिए `codex_plugin_spec` स्पष्ट रूप से सेट करें। इसका लाइव एजेंट प्रमाण दृश्यमान प्रगति भेजता है, यादृच्छिक वर्कस्पेस रीड और सटीक आर्टिफ़ैक्ट लेखन से होते हुए जारी रहता है, फिर पूर्णता भेजता है।

चरण मैट्रिक्स, सटीक वर्कफ़्लो जॉब नाम, प्रोफ़ाइल अंतर, आर्टिफ़ैक्ट और
केंद्रित पुनः रन हैंडल के लिए [पूर्ण रिलीज़ सत्यापन](/hi/reference/full-release-validation) देखें।

`OpenClaw Release Publish` मैन्युअल परिवर्तनकारी रिलीज़ वर्कफ़्लो है। रिलीज़ टैग
मौजूद होने और OpenClaw npm प्रीफ़्लाइट सफल होने के बाद विश्वसनीय `main` से
नियमित बीटा और स्थिर प्रकाशन डिस्पैच करें (प्रीफ़्लाइट अपनी जाँचों में
`pnpm plugins:sync:check` चलाता है)। टैग अब भी सटीक
रिलीज़ कमिट चुनता है, जिसमें `release/YYYY.M.PATCH` पर मौजूद कमिट भी शामिल है; Tideclaw अल्फ़ा
प्रकाशन अपनी संबंधित अल्फ़ा ब्रांच का उपयोग जारी रखते हैं। इसके लिए सहेजा गया
`preflight_run_id`, सफल
`full_release_validation_run_id` और उसका सटीक
`full_release_validation_run_attempt` आवश्यक है, यह सभी
प्रकाशन-योग्य Plugin पैकेजों के लिए `Plugin NPM Release` डिस्पैच करता है, उसी
रिलीज़ SHA के लिए `Plugin ClawHub Release` डिस्पैच करता है और उसके बाद ही `OpenClaw NPM Release` डिस्पैच करता है। स्थिर प्रकाशन के लिए
सटीक `windows_node_tag` भी आवश्यक है; वर्कफ़्लो किसी भी चाइल्ड प्रकाशन से पहले Windows स्रोत
रिलीज़ सत्यापित करता है और उसके x64/ARM64 इंस्टॉलर की उम्मीदवार द्वारा स्वीकृत
`windows_node_installer_digests` इनपुट से तुलना करता है, फिर GitHub रिलीज़ ड्राफ़्ट प्रकाशित करने से पहले
उन्हीं पिन किए गए इंस्टॉलर डाइजेस्ट तथा सटीक सहयोगी एसेट
और चेकसम अनुबंध को प्रोमोट और सत्यापित करता है।
केंद्रित केवल-Plugin सुधार एक गैर-रिक्त
पैकेज सूची के साथ `plugin_publish_scope=selected` का उपयोग करते हैं। केवल-Plugin `all-publishable` रन के लिए कोर प्रकाशन के समान अपरिवर्तनीय npm
प्रीफ़्लाइट और पूर्ण रिलीज़ सत्यापन प्रमाण आवश्यक है।

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH-beta.N \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

तेज़ी से बदलती ब्रांच पर पिन किए गए कमिट प्रमाण के लिए
`gh workflow run ... --ref main -f ref=<sha>` के बजाय हेल्पर का उपयोग करें:

```bash
pnpm ci:full-release --sha <full-sha>
```

GitHub वर्कफ़्लो डिस्पैच रेफ़ ब्रांच या टैग होने चाहिए, रॉ कमिट SHA नहीं। यह
हेल्पर विश्वसनीय `main`
वर्कफ़्लो SHA पर अस्थायी `release-ci/<sha>-...` ब्रांच पुश करता है, अनुरोधित लक्ष्य SHA को वर्कफ़्लो `ref` इनपुट के माध्यम से
पास करता है, उपलब्ध होने पर सख्त सटीक-लक्ष्य प्रमाण का पुनः उपयोग करता है, सत्यापित करता है कि प्रत्येक चाइल्ड
वर्कफ़्लो `headSha` विश्वसनीय वर्कफ़्लो SHA से मेल खाता है और रन पूर्ण होने पर अस्थायी
ब्रांच हटा देता है। नया
सत्यापन बाध्य करने के लिए `-f reuse_evidence=false` दें। यदि कोई चाइल्ड वर्कफ़्लो अलग
वर्कफ़्लो SHA पर चला हो, तो अंब्रेला सत्यापक भी विफल हो जाता है।

`release_profile` रिलीज़ जाँचों में भेजी जाने वाली लाइव/प्रोवाइडर व्यापकता को नियंत्रित करता है।
मैन्युअल रिलीज़ वर्कफ़्लो डिफ़ॉल्ट रूप से `stable` का उपयोग करते हैं; `full` का उपयोग केवल तभी करें जब आप
जानबूझकर व्यापक परामर्शात्मक प्रोवाइडर/मीडिया मैट्रिक्स चाहते हों। स्थिर और पूर्ण
रिलीज़ जाँच हमेशा व्यापक लाइव/E2E और Docker रिलीज़-पाथ सोक चलाती हैं;
बीटा प्रोफ़ाइल `run_release_soak=true` के साथ ऑप्ट इन कर सकती है।

- `beta` सबसे तेज़ OpenAI/कोर रिलीज़-महत्वपूर्ण लेन बनाए रखता है।
- `stable` स्थिर प्रोवाइडर/बैकएंड सेट जोड़ता है।
- `full` व्यापक परामर्शात्मक प्रोवाइडर/मीडिया मैट्रिक्स चलाता है।

अंब्रेला डिस्पैच किए गए चाइल्ड रन आईडी दर्ज करता है और अंतिम `Verify full validation` जॉब मौजूदा चाइल्ड रन निष्कर्षों की दोबारा जाँच करता है तथा प्रत्येक चाइल्ड रन के लिए सबसे धीमे जॉब की तालिकाएँ जोड़ता है। यदि कोई चाइल्ड वर्कफ़्लो दोबारा चलाकर सफल हो जाता है, तो अंब्रेला परिणाम और समय सारांश ताज़ा करने के लिए केवल पैरेंट सत्यापक जॉब दोबारा चलाएँ।

पुनर्प्राप्ति के लिए `Full Release Validation` और `OpenClaw Release Checks`, दोनों `rerun_group` स्वीकार करते हैं। किसी रिलीज़ उम्मीदवार के लिए `all`, केवल सामान्य पूर्ण CI चाइल्ड के लिए `ci`, केवल Plugin प्रीरिलीज़ चाइल्ड के लिए `plugin-prerelease`, केवल OpenClaw प्रदर्शन चाइल्ड के लिए `performance`, प्रत्येक रिलीज़ चाइल्ड के लिए `release-checks`, या अंब्रेला पर कोई अधिक संकीर्ण समूह उपयोग करें: `install-smoke`, `cross-os`, `live-e2e`, `package`, `qa`, `qa-parity`, `qa-live` या `npm-telegram`। यह केंद्रित सुधार के बाद विफल रिलीज़ बॉक्स के पुनः रन को सीमित रखता है। किसी एक विफल क्रॉस-OS लेन के लिए `rerun_group=cross-os` को `cross_os_suite_filter` के साथ संयोजित करें, उदाहरण के लिए `windows/packaged-upgrade`; लंबे क्रॉस-OS कमांड Heartbeat पंक्तियाँ उत्सर्जित करते हैं और पैकेज्ड-अपग्रेड सारांश में प्रत्येक चरण का समय शामिल होता है। चयनित Matrix और Telegram QA लेन सामान्य रिलीज़ सत्यापन को अवरुद्ध करती हैं, और मानक रनटाइम टूल कवरेज गेट भी ऐसा ही करता है। QA समानता, रनटाइम समानता और गेट की गई Discord, WhatsApp तथा Slack लाइव लेन परामर्शात्मक हैं।

`OpenClaw Release Checks` चयनित रेफ़ को एक बार `release-package-under-test` टारबॉल में रिज़ॉल्व करने के लिए विश्वसनीय वर्कफ़्लो रेफ़ का उपयोग करता है, फिर उस आर्टिफ़ैक्ट को क्रॉस-OS जाँच और पैकेज स्वीकृति तथा सोक कवरेज चलने पर लाइव/E2E रिलीज़-पाथ Docker वर्कफ़्लो में भेजता है। इससे रिलीज़ बॉक्सों में पैकेज बाइट समान रहती हैं और कई चाइल्ड जॉब में एक ही उम्मीदवार को दोबारा पैक करने से बचा जाता है। Codex npm-Plugin लाइव लेन के लिए रिलीज़ जाँच या तो `release_package_spec` से व्युत्पन्न मेल खाती प्रकाशित Plugin स्पेक भेजती हैं, ऑपरेटर द्वारा दी गई `codex_plugin_spec` भेजती हैं, या इनपुट रिक्त छोड़ती हैं ताकि Docker स्क्रिप्ट चयनित चेकआउट का Codex Plugin पैक करे।

`ref=main` और `rerun_group=all` के लिए डुप्लिकेट `Full Release Validation` रन
पुराने अंब्रेला को प्रतिस्थापित कर देते हैं। पैरेंट रद्द होने पर पैरेंट मॉनिटर पहले से
डिस्पैच किए गए किसी भी चाइल्ड वर्कफ़्लो को रद्द कर देता है, ताकि नया main सत्यापन
पुराने दो-घंटे के रिलीज़-जाँच रन के पीछे प्रतीक्षा न करे। रिलीज़ ब्रांच/टैग
सत्यापन और केंद्रित पुनः रन समूह `cancel-in-progress: false` बनाए रखते हैं।

## लाइव और E2E शार्ड

रिलीज़ लाइव/E2E चाइल्ड व्यापक नेटिव `pnpm test:live` कवरेज बनाए रखता है, लेकिन उसे एक सीरियल जॉब के बजाय `scripts/test-live-shard.mjs` के माध्यम से नामित शार्ड के रूप में चलाता है:

- `native-live-src-agents` और `native-live-src-agents-zai-coding`
- `native-live-src-gateway-core`
- प्रोवाइडर-फ़िल्टर किए गए `native-live-src-gateway-profiles` जॉब
- `native-live-src-gateway-backends`
- `native-live-src-infra`
- `native-live-test`
- `native-live-extensions-a-k`
- `native-live-extensions-l-n`
- `native-live-extensions-moonshot`
- `native-live-extensions-openai`
- `native-live-extensions-o-z-other`
- `native-live-extensions-xai`
- अलग किए गए मीडिया ऑडियो/वीडियो शार्ड और प्रोवाइडर-फ़िल्टर किए गए संगीत शार्ड

यह समान फ़ाइल कवरेज बनाए रखता है, साथ ही धीमी लाइव प्रोवाइडर विफलताओं को दोबारा चलाना और उनका निदान करना आसान बनाता है। समेकित `native-live-src-gateway`, `native-live-extensions-o-z`, `native-live-extensions-media` और `native-live-extensions-media-music` शार्ड नाम मैन्युअल एकल पुनः रन के लिए मान्य बने रहते हैं।

नेटिव लाइव मीडिया शार्ड `ghcr.io/openclaw/openclaw-live-media-runner:ubuntu-24.04` में चलते हैं, जिसे `Live Media Runner Image` वर्कफ़्लो बनाता है। वह इमेज `ffmpeg` और `ffprobe` को पहले से इंस्टॉल करती है; मीडिया जॉब सेटअप से पहले केवल बाइनरी सत्यापित करते हैं। Docker-समर्थित लाइव सुइट को सामान्य Blacksmith रनर पर रखें — नेस्टेड Docker परीक्षण शुरू करने के लिए कंटेनर जॉब उचित स्थान नहीं हैं।

Docker-समर्थित लाइव मॉडल/बैकएंड शार्ड प्रत्येक चयनित कमिट के लिए एक अलग साझा `ghcr.io/openclaw/openclaw-live-test:<sha>-<extensions>` इमेज का उपयोग करते हैं। लाइव रिलीज़ वर्कफ़्लो उस इमेज को एक बार बिल्ड और पुश करता है, फिर Docker लाइव मॉडल, प्रदाता-शार्डेड Gateway, CLI बैकएंड, ACP बाइंड और Codex हार्नेस शार्ड `OPENCLAW_SKIP_DOCKER_BUILD=1` के साथ चलते हैं। Gateway Docker शार्ड में वर्कफ़्लो जॉब टाइमआउट से कम स्पष्ट स्क्रिप्ट-स्तरीय `timeout` सीमाएँ होती हैं, ताकि अटका हुआ कंटेनर या क्लीनअप पथ पूरे रिलीज़-जाँच बजट का उपयोग करने के बजाय शीघ्र विफल हो जाए। यदि वे शार्ड पूर्ण स्रोत Docker लक्ष्य को स्वतंत्र रूप से दोबारा बिल्ड करते हैं, तो रिलीज़ रन गलत तरीके से कॉन्फ़िगर किया गया है और डुप्लिकेट इमेज बिल्ड पर वास्तविक समय व्यर्थ करेगा।

## पैकेज स्वीकृति

जब प्रश्न हो, "क्या यह इंस्टॉल करने योग्य OpenClaw पैकेज एक उत्पाद के रूप में काम करता है?", तब `Package Acceptance` का उपयोग करें। यह सामान्य CI से अलग है: सामान्य CI स्रोत ट्री को सत्यापित करता है, जबकि पैकेज स्वीकृति एकल टारबॉल को उसी Docker E2E हार्नेस के माध्यम से सत्यापित करती है जिसका उपयोग उपयोगकर्ता इंस्टॉल या अपडेट के बाद करते हैं।

### जॉब

1. `resolve_package`, `workflow_ref` को चेक आउट करता है, एक पैकेज उम्मीदवार का समाधान करता है, `.artifacts/docker-e2e-package/openclaw-current.tgz` लिखता है, `.artifacts/docker-e2e-package/package-candidate.json` लिखता है, दोनों को `package-under-test` आर्टिफ़ैक्ट के रूप में अपलोड करता है और GitHub चरण सारांश में स्रोत, वर्कफ़्लो रेफ़, पैकेज रेफ़, संस्करण, SHA-256 और प्रोफ़ाइल प्रिंट करता है।
2. `package_integrity`, `package-under-test` आर्टिफ़ैक्ट डाउनलोड करता है और `scripts/check-openclaw-package-tarball.mjs` के साथ सार्वजनिक पैकेज टारबॉल अनुबंध लागू करता है।
3. `docker_acceptance`, समाधान किए गए पैकेज स्रोत SHA (`workflow_ref` पर फ़ॉलबैक करते हुए) और `package_artifact_name=package-under-test` के साथ `openclaw-live-and-e2e-checks-reusable.yml` को कॉल करता है। पुन: प्रयोज्य वर्कफ़्लो उस आर्टिफ़ैक्ट को डाउनलोड करता है, टारबॉल इन्वेंट्री को सत्यापित करता है, आवश्यकता होने पर पैकेज-डाइजेस्ट Docker इमेज तैयार करता है और वर्कफ़्लो चेकआउट को पैक करने के बजाय उस पैकेज के विरुद्ध चयनित Docker लेन चलाता है। जब कोई प्रोफ़ाइल अनेक लक्षित `docker_lanes` चुनती है, तो पुन: प्रयोज्य वर्कफ़्लो पैकेज और साझा इमेज को एक बार तैयार करता है, फिर उन लेन को अद्वितीय आर्टिफ़ैक्ट वाले समानांतर लक्षित Docker जॉब के रूप में वितरित करता है।
4. `package_telegram`, वैकल्पिक रूप से `NPM Telegram Beta E2E` को कॉल करता है। यह तब चलता है जब `telegram_mode`, `none` नहीं होता और यदि पैकेज स्वीकृति ने किसी का समाधान किया हो, तो वही `package-under-test` आर्टिफ़ैक्ट इंस्टॉल करता है; स्वतंत्र Telegram डिस्पैच अभी भी प्रकाशित npm स्पेक इंस्टॉल कर सकता है।
5. `summary`, पैकेज समाधान, अखंडता, Docker स्वीकृति या वैकल्पिक Telegram लेन के विफल होने पर वर्कफ़्लो को विफल करता है। `advisory` इनपुट सलाहकारी कॉलर के लिए स्वीकृति विफलताओं को चेतावनियों में बदल देता है।

### उम्मीदवार स्रोत

- `source=npm`, केवल `openclaw@extended-stable`, `openclaw@beta`, `openclaw@latest` या `openclaw@2026.4.27-beta.2` जैसा सटीक OpenClaw रिलीज़ संस्करण स्वीकार करता है। प्रकाशित एक्सटेंडेड-स्टेबल, प्रीरिलीज़ या स्टेबल स्वीकृति के लिए इसका उपयोग करें।
- `source=ref`, किसी विश्वसनीय `package_ref` ब्रांच, टैग या पूर्ण कमिट SHA को पैक करता है। रिज़ॉल्वर OpenClaw ब्रांच/टैग फ़ेच करता है, सत्यापित करता है कि चयनित कमिट रिपॉज़िटरी ब्रांच इतिहास या किसी रिलीज़ टैग से पहुँच योग्य है, डिटैच्ड वर्कट्री में निर्भरताएँ इंस्टॉल करता है और इसे `scripts/package-openclaw-for-docker.mjs` के साथ पैक करता है।
- `source=url`, एक सार्वजनिक HTTPS `.tgz` डाउनलोड करता है; `package_sha256` आवश्यक है। यह पथ URL क्रेडेंशियल, गैर-डिफ़ॉल्ट HTTPS पोर्ट, निजी/आंतरिक/विशेष-उपयोग होस्टनेम या समाधान किए गए IP और समान सार्वजनिक सुरक्षा नीति के बाहर के रीडायरेक्ट अस्वीकार करता है।
- `source=trusted-url`, `.github/package-trusted-sources.json` में नामित विश्वसनीय-स्रोत नीति से HTTPS `.tgz` डाउनलोड करता है; `package_sha256` और `trusted_source_id` आवश्यक हैं। इसका उपयोग केवल अनुरक्षक-स्वामित्व वाले एंटरप्राइज़ मिरर या निजी पैकेज रिपॉज़िटरी के लिए करें, जिन्हें कॉन्फ़िगर किए गए होस्ट, पोर्ट, पथ प्रीफ़िक्स, रीडायरेक्ट होस्ट या निजी-नेटवर्क समाधान की आवश्यकता होती है। यदि नीति बेयरर प्रमाणीकरण घोषित करती है, तो वर्कफ़्लो निश्चित `OPENCLAW_TRUSTED_PACKAGE_TOKEN` सीक्रेट का उपयोग करता है; URL में अंतर्निहित क्रेडेंशियल फिर भी अस्वीकार किए जाते हैं।
- `source=artifact`, `artifact_run_id` और `artifact_name` से एक `.tgz` डाउनलोड करता है; `package_sha256` वैकल्पिक है, लेकिन बाहरी रूप से साझा किए गए आर्टिफ़ैक्ट के लिए इसे प्रदान किया जाना चाहिए।

`workflow_ref` और `package_ref` को अलग रखें। `workflow_ref` वह विश्वसनीय वर्कफ़्लो/हार्नेस कोड है जो परीक्षण चलाता है। `package_ref` वह स्रोत कमिट है जिसे `source=ref` होने पर पैक किया जाता है। इससे वर्तमान परीक्षण हार्नेस पुराने वर्कफ़्लो तर्क को चलाए बिना पुराने विश्वसनीय स्रोत कमिट को सत्यापित कर सकता है।

### सुइट प्रोफ़ाइल

- `smoke` — `npm-onboard-channel-agent`, `gateway-network`, `config-reload`
- `package` — `npm-onboard-channel-agent`, `doctor-switch`, `update-channel-switch`, `skill-install`, `update-corrupt-plugin`, `upgrade-survivor`, `published-upgrade-survivor`, `root-managed-vps-upgrade`, `update-restart-auth`, `plugins-offline`, `plugin-update`
- `product` — `plugins-offline` के बजाय लाइव `plugins` कवरेज वाला `package` सेट, साथ में `mcp-channels`, `cron-mcp-cleanup`, `openai-web-search-minimal`, `openwebui`
- `full` — OpenWebUI के साथ पूर्ण Docker रिलीज़-पथ खंड
- `custom` — सटीक `docker_lanes`; `suite_profile=custom` होने पर आवश्यक

`package` प्रोफ़ाइल ऑफ़लाइन Plugin कवरेज का उपयोग करती है, ताकि प्रकाशित-पैकेज सत्यापन लाइव ClawHub उपलब्धता पर निर्भर न हो। वैकल्पिक Telegram लेन `NPM Telegram Beta E2E` में `package-under-test` आर्टिफ़ैक्ट का पुन: उपयोग करती है, जबकि स्वतंत्र डिस्पैच के लिए प्रकाशित npm स्पेक पथ रखा जाता है।

स्थानीय कमांड, Docker लेन, पैकेज स्वीकृति इनपुट, रिलीज़ डिफ़ॉल्ट और विफलता ट्राइएज सहित समर्पित अपडेट और Plugin परीक्षण नीति के लिए,
[अपडेट और Plugin का परीक्षण](/hi/help/testing-updates-plugins) देखें।

रिलीज़ जाँच `source=artifact`, तैयार रिलीज़ पैकेज आर्टिफ़ैक्ट, `suite_profile=custom`, `docker_lanes='doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor root-managed-vps-upgrade update-restart-auth plugins-offline plugin-update plugin-binding-command-escape'` और `telegram_mode=mock-openai` के साथ पैकेज स्वीकृति को कॉल करती हैं। इससे पैकेज माइग्रेशन, अपडेट, लाइव ClawHub स्किल इंस्टॉल, पुराने Plugin निर्भरता क्लीनअप, कॉन्फ़िगर किए गए Plugin इंस्टॉल की मरम्मत, ऑफ़लाइन Plugin, Plugin अपडेट और Telegram प्रमाण समान समाधान किए गए पैकेज टारबॉल पर बने रहते हैं। प्रकाशित बीटा के विरुद्ध पुनर्निर्माण किए बिना उसी मैट्रिक्स को शिप किए गए npm पैकेज पर चलाने के लिए पूर्ण रिलीज़ सत्यापन या OpenClaw रिलीज़ जाँच पर `release_package_spec` सेट करें; केवल तभी `package_acceptance_package_spec` सेट करें जब पैकेज स्वीकृति को शेष रिलीज़ सत्यापन से अलग पैकेज चाहिए। क्रॉस-OS रिलीज़ जाँच अभी भी OS-विशिष्ट ऑनबोर्डिंग, इंस्टॉलर और प्लेटफ़ॉर्म व्यवहार को कवर करती हैं; पैकेज/अपडेट उत्पाद सत्यापन पैकेज स्वीकृति से शुरू होना चाहिए।

`published-upgrade-survivor` Docker लेन अवरोधक रिलीज़ पथ में प्रति रन एक प्रकाशित पैकेज बेसलाइन सत्यापित करती है। पैकेज स्वीकृति में, समाधान किया गया `package-under-test` टारबॉल हमेशा उम्मीदवार होता है और `published_upgrade_survivor_baseline` फ़ॉलबैक प्रकाशित बेसलाइन चुनता है, जिसका डिफ़ॉल्ट `openclaw@latest` है; विफल-लेन पुनः-रन कमांड उस बेसलाइन को बनाए रखते हैं। `run_release_soak=true` या `release_profile=full` वाला पूर्ण रिलीज़ सत्यापन `published_upgrade_survivor_baselines='last-stable-4 2026.4.23 2026.5.2 2026.4.15'` और `published_upgrade_survivor_scenarios=reported-issues` सेट करता है, ताकि चार नवीनतम स्टेबल npm रिलीज़ के साथ पिन किए गए Plugin-संगतता सीमा रिलीज़ और Feishu कॉन्फ़िगरेशन, संरक्षित बूटस्ट्रैप/पर्सोना फ़ाइलों, कॉन्फ़िगर किए गए OpenClaw Plugin इंस्टॉल, टिल्ड लॉग पथ तथा पुराने विरासती Plugin निर्भरता रूट के लिए समस्या-आकार वाले फ़िक्स्चर तक विस्तार किया जा सके। बहु-बेसलाइन प्रकाशित-अपग्रेड सर्वाइवर चयन को बेसलाइन के अनुसार अलग-अलग लक्षित Docker रनर जॉब में शार्ड किया जाता है। जब प्रश्न सामान्य पूर्ण रिलीज़ CI विस्तार नहीं, बल्कि प्रकाशित अपडेट के संपूर्ण क्लीनअप का हो, तब अलग `Update Migration` वर्कफ़्लो `all-since-2026.4.23` बेसलाइन और `plugin-deps-cleanup` परिदृश्यों के साथ `update-migration` Docker लेन का उपयोग करता है। स्थानीय एग्रीगेट रन `OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPECS` के साथ सटीक पैकेज स्पेक पास कर सकते हैं, `openclaw@2026.4.15` जैसे `OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPEC` के साथ एकल लेन रख सकते हैं या परिदृश्य मैट्रिक्स के लिए `OPENCLAW_UPGRADE_SURVIVOR_SCENARIOS` सेट कर सकते हैं। प्रकाशित लेन बेक की गई `openclaw config set` कमांड रेसिपी के साथ बेसलाइन कॉन्फ़िगर करती है, रेसिपी चरणों को `summary.json` में रिकॉर्ड करती है और Gateway शुरू होने के बाद `/healthz`, `/readyz` तथा RPC स्थिति की जाँच करती है। Windows पैकेज्ड और इंस्टॉलर फ़्रेश लेन यह भी सत्यापित करती हैं कि इंस्टॉल किया गया पैकेज किसी मूल एब्सोल्यूट Windows पथ से ब्राउज़र-कंट्रोल ओवरराइड इंपोर्ट कर सकता है। OpenAI क्रॉस-OS एजेंट-टर्न स्मोक सेट होने पर डिफ़ॉल्ट रूप से `OPENCLAW_CROSS_OS_OPENAI_MODEL` का, अन्यथा `openai/gpt-5.6-luna` का उपयोग करता है, ताकि इंस्टॉल और Gateway प्रमाण कम-लागत वाले GPT-5.6 परीक्षण टियर का उपयोग करे।

### विरासती संगतता अवधियाँ

पहले से प्रकाशित पैकेजों के लिए पैकेज स्वीकृति में सीमित विरासती-संगतता अवधियाँ हैं। `2026.4.25-beta.*` सहित `2026.4.25` तक के पैकेज संगतता पथ का उपयोग कर सकते हैं:

- `dist/postinstall-inventory.json` में ज्ञात निजी QA प्रविष्टियाँ टारबॉल से छोड़ी गई फ़ाइलों की ओर संकेत कर सकती हैं;
- जब पैकेज उस फ़्लैग को उजागर नहीं करता, तब `doctor-switch`, `gateway install --wrapper` पर्सिस्टेंस उप-मामला छोड़ सकता है;
- `update-channel-switch`, टारबॉल से प्राप्त नकली git फ़िक्स्चर से अनुपलब्ध pnpm `patchedDependencies` हटा सकता है और अनुपलब्ध पर्सिस्टेड `update.channel` को लॉग कर सकता है;
- Plugin स्मोक विरासती इंस्टॉल-रिकॉर्ड स्थान पढ़ सकते हैं या अनुपलब्ध मार्केटप्लेस इंस्टॉल-रिकॉर्ड पर्सिस्टेंस स्वीकार कर सकते हैं;
- `plugin-update`, इंस्टॉल रिकॉर्ड और पुनः-इंस्टॉल न करने के व्यवहार को अपरिवर्तित रखना आवश्यक बनाते हुए भी कॉन्फ़िगरेशन मेटाडेटा माइग्रेशन की अनुमति दे सकता है।

प्रकाशित `2026.4.26` पैकेज पहले से शिप की गई स्थानीय बिल्ड मेटाडेटा स्टैम्प फ़ाइलों के लिए चेतावनी भी दे सकता है और `2026.5.20` तक के पैकेज `npm-shrinkwrap.json` अनुपलब्ध होने पर विफल होने के बजाय चेतावनी दे सकते हैं। बाद के पैकेजों को आधुनिक अनुबंधों को पूरा करना आवश्यक है; समान स्थितियाँ चेतावनी देने या छोड़ने के बजाय विफल होती हैं।

### उदाहरण

```bash
# उत्पाद-स्तरीय कवरेज के साथ वर्तमान बीटा पैकेज सत्यापित करें।
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f telegram_mode=mock-openai

# पैकेज कवरेज के साथ प्रकाशित एक्सटेंडेड-स्टेबल पैकेज सत्यापित करें।
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@extended-stable \
  -f suite_profile=package \
  -f telegram_mode=mock-openai

# वर्तमान हार्नेस के साथ रिलीज़ ब्रांच पैक और सत्यापित करें।
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=ref \
  -f package_ref=release/YYYY.M.PATCH \
  -f suite_profile=package \
  -f telegram_mode=mock-openai

# टारबॉल URL सत्यापित करें। source=url के लिए SHA-256 अनिवार्य है।
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=url \
  -f package_url=https://example.com/openclaw-current.tgz \
  -f package_sha256=<64-char-sha256> \
  -f suite_profile=smoke

# नामित विश्वसनीय निजी मिरर नीति से टारबॉल सत्यापित करें।
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=trusted-url \
  -f trusted_source_id=enterprise-artifactory \
  -f package_url=https://packages.example.internal:8443/artifactory/openclaw/openclaw-current.tgz \
  -f package_sha256=<64-char-sha256> \
  -f suite_profile=smoke

# किसी अन्य Actions रन द्वारा अपलोड किए गए टारबॉल का पुन: उपयोग करें।
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=artifact \
  -f artifact_run_id=<run-id> \
  -f artifact_name=package-under-test \
  -f suite_profile=custom \
  -f docker_lanes='install-e2e plugin-update'
```

विफल पैकेज स्वीकृति रन को डीबग करते समय, पैकेज स्रोत, संस्करण और SHA-256 की पुष्टि करने के लिए `resolve_package` सारांश से शुरू करें। फिर `docker_acceptance` चाइल्ड रन और उसके Docker आर्टिफ़ैक्ट देखें: `.artifacts/docker-tests/**/summary.json`, `failures.json`, लेन लॉग, चरण समय और पुनः-रन कमांड। पूर्ण रिलीज़ सत्यापन को दोबारा चलाने के बजाय विफल पैकेज प्रोफ़ाइल या सटीक Docker लेन को दोबारा चलाना बेहतर है।

## इंस्टॉल स्मोक

`Install Smoke` वर्कफ़्लो अब पुल रिक्वेस्ट या `main` पुश पर नहीं चलता। इसका रात्रिकालीन/मैन्युअल रैपर और रिलीज़ सत्यापन, दोनों केवल-पढ़ने योग्य `install-smoke-reusable.yml` कोर को कॉल करते हैं, और प्रत्येक रन GitHub-होस्टेड रनर पर पूर्ण इंस्टॉल-स्मोक पथ अपनाता है:

- रूट Dockerfile स्मोक इमेज प्रत्येक लक्ष्य SHA के लिए एक बार बनाई जाती है, वर्कफ़्लो संशोधन और उत्पादक प्रयास से एक अपरिवर्तनीय आर्टिफ़ैक्ट में बाँधी जाती है, फिर CLI स्मोक, एजेंट द्वारा साझा-वर्कस्पेस हटाने वाले CLI स्मोक, कंटेनर Gateway-नेटवर्क E2E और बंडल किए गए `matrix` Plugin बिल्ड-आर्ग स्मोक द्वारा लोड की जाती है। Plugin स्मोक रनटाइम निर्भरता इंस्टॉल मिररिंग और यह सत्यापित करता है कि Plugin एंट्री-एस्केप डायग्नोस्टिक्स के बिना लोड होता है।
- QR पैकेज इंस्टॉल और इंस्टॉलर/अपडेट Docker स्मोक (Rocky Linux इंस्टॉलर लेन और कॉन्फ़िगर किए जा सकने वाले `update_baseline_version` npm बेसलाइन के विरुद्ध एक अपडेट लेन सहित) अलग-अलग जॉब के रूप में चलते हैं, ताकि इंस्टॉलर कार्य को रूट इमेज स्मोक के पीछे प्रतीक्षा न करनी पड़े।

धीमा Bun ग्लोबल इंस्टॉल इमेज-प्रोवाइडर स्मोक अलग से `run_bun_global_install_smoke` द्वारा गेट किया जाता है। यह रात्रिकालीन शेड्यूल पर चलता है, रिलीज़ जाँचों से आने वाली वर्कफ़्लो कॉल के लिए डिफ़ॉल्ट रूप से चालू रहता है, और मैन्युअल `Install Smoke` डिस्पैच इसे चुन सकते हैं। सामान्य PR CI अब भी Node-संबंधित परिवर्तनों के लिए तेज़ Bun लॉन्चर रिग्रेशन लेन चलाता है। QR और इंस्टॉलर Docker परीक्षण अपने इंस्टॉल-केंद्रित Dockerfile बनाए रखते हैं।

## स्थानीय Docker E2E

`pnpm test:docker:all` एक साझा लाइव-परीक्षण इमेज पहले से बनाता है, OpenClaw को एक बार npm टारबॉल के रूप में पैक करता है और दो साझा `scripts/e2e/Dockerfile` इमेज बनाता है:

- इंस्टॉलर/अपडेट/Plugin-निर्भरता लेन के लिए एक मूलभूत Node/Git रनर;
- सामान्य कार्यक्षमता लेन के लिए उसी टारबॉल को `/app` में इंस्टॉल करने वाली एक कार्यात्मक इमेज।

Docker लेन परिभाषाएँ `scripts/lib/docker-e2e-scenarios.mjs` में, प्लानर लॉजिक `scripts/lib/docker-e2e-plan.mjs` में रहती हैं और रनर केवल चयनित योजना निष्पादित करता है। शेड्यूलर `OPENCLAW_DOCKER_E2E_BARE_IMAGE` और `OPENCLAW_DOCKER_E2E_FUNCTIONAL_IMAGE` के साथ प्रत्येक लेन के लिए इमेज चुनता है, फिर `OPENCLAW_SKIP_DOCKER_BUILD=1` के साथ लेन चलाता है।

### पुनः उपयोग योग्य लाइव/E2E वर्कफ़्लो

पुनः उपयोग योग्य लाइव/E2E वर्कफ़्लो `scripts/test-docker-all.mjs --plan-json` से पूछता है कि कौन-सा पैकेज, इमेज प्रकार, लाइव इमेज, लेन और क्रेडेंशियल कवरेज आवश्यक है। फिर `scripts/docker-e2e.mjs` उस योजना को GitHub आउटपुट और सारांशों में बदलता है। यह या तो `scripts/package-openclaw-for-docker.mjs` के माध्यम से OpenClaw पैक करता है, वर्तमान रन का पैकेज आर्टिफ़ैक्ट डाउनलोड करता है या `package_artifact_run_id` से पैकेज आर्टिफ़ैक्ट डाउनलोड करता है, फिर टारबॉल इन्वेंटरी सत्यापित करता है। डिफ़ॉल्ट `no-push-artifact` पथ Blacksmith के Docker लेयर कैश के माध्यम से पैकेज-डाइजेस्ट-टैग की गई मूलभूत/कार्यात्मक इमेज बनाता है, इमेज के सटीक बाइट को एक अपरिवर्तनीय वर्कफ़्लो आर्टिफ़ैक्ट में पैक करता है और प्रत्येक उपभोक्ता से उस आर्टिफ़ैक्ट को सत्यापित और लोड करवाता है। इसके बजाय `existing-only` को स्पष्ट `docker_e2e_bare_image`/`docker_e2e_functional_image` GHCR संदर्भों की आवश्यकता होती है और यह कभी निर्माण या पुश नहीं करता। वे रजिस्ट्री पुल प्रति प्रयास 180-सेकंड का सीमित टाइमआउट उपयोग करते हैं, ताकि अटकी हुई स्ट्रीम CI पाइपलाइन के महत्वपूर्ण पथ का अधिकांश भाग लेने के बजाय शीघ्र पुनः प्रयास करे। सफल निर्धारित सत्यापन के बाद, `openclaw-scheduled-live-checks.yml` अपरिवर्तनीय परीक्षित-इमेज मैनिफ़ेस्ट अलग पैकेज-लेखन प्रकाशक को देता है; केवल-पढ़ने वाले रिलीज़ और प्रीरिलीज़ कॉलर कभी उस लेखक से होकर नहीं जाते।

### रिलीज़-पथ खंड

रिलीज़ Docker कवरेज `OPENCLAW_SKIP_DOCKER_BUILD=1` के साथ छोटे खंडित जॉब चलाता है, ताकि प्रत्येक खंड केवल अपने लिए आवश्यक आर्टिफ़ैक्ट-समर्थित इमेज प्रकार को सत्यापित और लोड करे (या स्पष्ट `existing-only` पुनः उपयोग के अंतर्गत उसे पुल करे) और उसी भारित शेड्यूलर के माध्यम से कई लेन निष्पादित करे:

- `OPENCLAW_DOCKER_ALL_PROFILE=release-path`
- `OPENCLAW_DOCKER_ALL_CHUNK=core | package-update-openai | package-update-anthropic | package-update-core | plugins-runtime-plugins | plugins-runtime-services | plugins-runtime-install-a..h | openwebui`

वर्तमान रिलीज़ Docker खंड `core`, `package-update-openai`, `package-update-anthropic`, `package-update-core`, `plugins-runtime-plugins`, `plugins-runtime-services`, `plugins-runtime-install-a` से `plugins-runtime-install-h` तक और `openwebui` हैं। `package-update-openai` में लाइव Codex Plugin पैकेज लेन शामिल है, जो प्रत्याशी OpenClaw पैकेज इंस्टॉल करती है, स्पष्ट Codex CLI इंस्टॉल अनुमोदन के साथ `codex_plugin_spec` या समान-संदर्भ टारबॉल से Codex Plugin इंस्टॉल करती है, Codex CLI पूर्व-जाँच और समान-सत्र एजेंट टर्न चलाती है, फिर शून्य-पुनः-प्रयास वाला मध्यम-विचार टर्न चलाती है, जो प्रगति भेजता है, यादृच्छिक वर्कस्पेस इनपुट पढ़ता है, उनका सटीक आर्टिफ़ैक्ट लिखता है और पूर्णता भेजता है। `plugins-runtime-core`, `plugins-runtime` और `plugins-integrations` समुच्चय Plugin/रनटाइम उपनाम बने रहते हैं। `install-e2e` लेन उपनाम दोनों प्रोवाइडर इंस्टॉलर लेन के लिए समुच्चय मैन्युअल पुनः-रन उपनाम बना रहता है।

जब भी स्थिर या पूर्ण रिलीज़-पथ कवरेज इसका अनुरोध करता है, OpenWebUI समर्पित बड़ी-डिस्क वाले Blacksmith रनर पर एक स्वतंत्र `openwebui` खंड के रूप में चलता है, भले ही पुनः उपयोग योग्य वर्कफ़्लो समर्थित जॉब को GitHub-होस्टेड रनर की ओर भेजे। बाहरी इमेज पुल को अलग रखने से बड़ी इमेज `plugins-runtime-services` में साझा पैकेज और Plugin इमेज से प्रतिस्पर्धा नहीं करती; पुराने समुच्चय Plugin/रनटाइम खंड अब भी संगत मैन्युअल पुनः-रन के लिए OpenWebUI शामिल करते हैं। बंडल किए गए चैनल की अपडेट लेन क्षणिक npm नेटवर्क विफलताओं के लिए एक बार पुनः प्रयास करती हैं।

प्रत्येक खंड लेन लॉग, समय, `summary.json`, `failures.json`, चरण समय, शेड्यूलर योजना JSON, धीमी-लेन तालिकाओं और प्रति-लेन पुनः-रन कमांड सहित `.artifacts/docker-tests/` अपलोड करता है। वर्कफ़्लो का `docker_lanes` इनपुट खंड जॉब के बजाय उस रन के लिए तैयार इमेज के विरुद्ध चयनित लेन चलाता है, जिससे विफल-लेन डीबगिंग एक लक्षित Docker जॉब तक सीमित रहती है; यदि चयनित लेन लाइव Docker लेन है, तो लक्षित जॉब उस पुनः-रन के लिए लाइव-परीक्षण इमेज स्थानीय रूप से बनाता है। पुनः-रन सहायक विफलता आर्टिफ़ैक्ट के सटीक चयनित लक्ष्य SHA को सत्यापित करता है और मैन्युअल डिस्पैच उस संदर्भ को दोबारा पैक करता है, क्योंकि आंतरिक पुनः उपयोग योग्य वर्कफ़्लो पैकेज ट्यूपल `workflow_dispatch` स्कीमा का हिस्सा नहीं है। जेनरेट किए गए कमांड तैयार इमेज इनपुट और `shared_image_policy=existing-only` केवल तब शामिल करते हैं, जब वे इनपुट GHCR-समर्थित हों; रनर-स्थानीय आर्टिफ़ैक्ट टैग छोड़ दिए जाते हैं, ताकि नया रनर उन्हें दोबारा बनाए। स्पष्ट लक्ष्य ओवरराइड पुनर्प्राप्त GHCR इमेज संदर्भ हटा देता है, जब तक आर्टिफ़ैक्ट यह सिद्ध न करे कि वे ओवरराइड से मेल खाते हैं। आर्टिफ़ैक्ट द्वारा जेनरेट किए गए वर्कफ़्लो-परिभाषा संदर्भ भी छोड़ दिए जाते हैं, क्योंकि पूर्ण-रिलीज़ अस्थायी शाखाएँ हटा दी जाती हैं; जब तक ऑपरेटर स्पष्ट रूप से ओवरराइड न करे, डिस्पैच रिपॉज़िटरी की डिफ़ॉल्ट शाखा उपयोग करता है।

```bash
pnpm test:docker:rerun <run-id>      # Docker आर्टिफ़ैक्ट डाउनलोड करें और संयुक्त/प्रति-लेन लक्षित पुनः-रन कमांड प्रिंट करें
pnpm test:docker:timings <summary>   # धीमी-लेन और चरण महत्वपूर्ण-पथ सारांश
```

निर्धारित लाइव/E2E वर्कफ़्लो प्रतिदिन पूर्ण रिलीज़-पथ Docker सुइट चलाता है और सफल होने के बाद सटीक परीक्षित इमेज आर्टिफ़ैक्ट के लिए स्पष्ट प्रकाशक को आमंत्रित करता है।

## Plugin प्रीरिलीज़

`Plugin Prerelease` अधिक महँगा उत्पाद/पैकेज कवरेज है, इसलिए यह `Full Release Validation` या किसी स्पष्ट ऑपरेटर द्वारा डिस्पैच किया जाने वाला अलग वर्कफ़्लो है। सामान्य पुल रिक्वेस्ट, `main` पुश और स्वतंत्र मैन्युअल CI डिस्पैच उस सुइट को बंद रखते हैं। यह बंडल किए गए Plugin परीक्षणों को आठ एक्सटेंशन वर्कर में संतुलित करता है; वे एक्सटेंशन शार्ड जॉब एक समय में अधिकतम दो Plugin कॉन्फ़िग समूह चलाते हैं, प्रत्येक समूह के लिए एक Vitest वर्कर और बड़ा Node हीप उपयोग करते हैं, ताकि आयात-भारी Plugin बैच अतिरिक्त CI जॉब न बनाएँ। केवल-रिलीज़ Docker प्रीरिलीज़ पथ (`full_release_validation` इनपुट द्वारा सक्षम) लक्षित Docker लेन को चार-चार के समूह में बैच करता है, ताकि एक-से-तीन-मिनट के जॉब के लिए दर्जनों रनर आरक्षित न हों। वर्कफ़्लो `@openclaw/plugin-inspector` से एक सूचनात्मक `plugin-inspector-advisory` आर्टिफ़ैक्ट भी अपलोड करता है; इंस्पेक्टर निष्कर्ष ट्रायेज इनपुट हैं और अवरोधक Plugin प्रीरिलीज़ गेट को नहीं बदलते।

## QA Lab

QA Lab के पास मुख्य स्मार्ट-स्कोप्ड वर्कफ़्लो के बाहर समर्पित CI लेन हैं। एजेंटिक समानता व्यापक QA और रिलीज़ हार्नेस के अंतर्गत नेस्ट की गई है, यह स्वतंत्र PR वर्कफ़्लो नहीं है। जब समानता को व्यापक सत्यापन रन के साथ चलना चाहिए, तब `rerun_group=qa-parity` के साथ `Full Release Validation` उपयोग करें।

- `QA-Lab - All Lanes` वर्कफ़्लो `main` पर रात्रिकालीन और मैन्युअल डिस्पैच पर चलता है; यह मॉक समानता के साथ लाइव Matrix, Telegram, Discord, WhatsApp और Slack जॉब में विभाजित होता है। लाइव जॉब `qa-live-shared` वातावरण उपयोग करते हैं; Telegram, Discord, WhatsApp और Slack Convex लीज़ उपयोग करते हैं, जबकि Matrix अस्थायी स्थानीय क्रेडेंशियल प्रावधान करता है।

रिलीज़ जाँचें नियतात्मक मॉक प्रोवाइडर और मॉक-योग्य मॉडल (`mock-openai/gpt-5.6-luna` और `mock-openai/gpt-5.6-luna-alt`) के साथ Matrix और Telegram लाइव ट्रांसपोर्ट लेन चलाती हैं, ताकि चैनल अनुबंध लाइव मॉडल विलंबता और सामान्य प्रोवाइडर-Plugin स्टार्टअप से अलग रहे। लाइव ट्रांसपोर्ट Gateway मेमोरी खोज अक्षम करता है, क्योंकि QA समानता मेमोरी व्यवहार को अलग से कवर करती है; प्रोवाइडर कनेक्टिविटी अलग लाइव मॉडल, नेटिव प्रोवाइडर और Docker प्रोवाइडर सुइट द्वारा कवर की जाती है।

निर्धारित और रिलीज़ Matrix गेट रिलीज़ परिदृश्यों के साथ साझा QA Lab सुइट होस्ट और लाइव एडाप्टर उपयोग करते हैं। CLI डिफ़ॉल्ट और मैन्युअल वर्कफ़्लो इनपुट `all` बने रहते हैं; मैन्युअल `all` डिस्पैच `transport`, `media`, `e2ee-smoke`, `e2ee-deep` और `e2ee-cli` प्रोफ़ाइल में विभाजित होते हैं, ताकि 93-परिदृश्य प्रमाण प्रति-जॉब टाइमआउट के भीतर रहे। केंद्रित मैन्युअल डिस्पैच एक जॉब में `fast`, `release` या `transport` चुनते हैं।

`OpenClaw Release Checks` रिलीज़ अनुमोदन से पहले रिलीज़-महत्वपूर्ण QA Lab लेन भी चलाता है; इसका QA समानता गेट प्रत्याशी और बेसलाइन पैक को समानांतर लेन जॉब के रूप में चलाता है, फिर अंतिम समानता तुलना के लिए दोनों आर्टिफ़ैक्ट एक छोटे रिपोर्ट जॉब में डाउनलोड करता है।

सामान्य PR के लिए समानता को आवश्यक स्थिति मानने के बजाय स्कोप किए गए CI/जाँच प्रमाण का पालन करें।

## CodeQL

`CodeQL` वर्कफ़्लो जानबूझकर संकीर्ण प्रथम-चरण सुरक्षा स्कैनर है, पूर्ण रिपॉज़िटरी स्वीप नहीं। दैनिक, मैन्युअल, `main` पुश और गैर-ड्राफ़्ट पुल रिक्वेस्ट गार्ड रन, Actions वर्कफ़्लो कोड तथा सर्वाधिक जोखिम वाली JavaScript/TypeScript सतहों को उच्च/गंभीर `security-severity` तक फ़िल्टर की गई उच्च-विश्वसनीयता सुरक्षा क्वेरी के साथ स्कैन करते हैं।

पुल रिक्वेस्ट गार्ड हल्का रहता है: यह केवल `.github/actions`, `.github/codeql`, `.github/workflows`, `packages`, `scripts`, `src`, या प्रक्रिया का स्वामित्व रखने वाले बंडल किए गए Plugin रनटाइम पाथ के अंतर्गत बदलावों के लिए शुरू होता है, और यह शेड्यूल किए गए वर्कफ़्लो वाला वही उच्च-विश्वसनीयता सुरक्षा मैट्रिक्स चलाता है। Android और macOS CodeQL को PR डिफ़ॉल्ट से बाहर रखा जाता है।

### सुरक्षा श्रेणियाँ

| ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |

### प्लेटफ़ॉर्म-विशिष्ट सुरक्षा शार्ड

- `CodeQL Android Critical Security` — शेड्यूल किया गया Android सुरक्षा शार्ड। वर्कफ़्लो सैनिटी द्वारा स्वीकार किए गए सबसे छोटे Blacksmith Linux रनर पर CodeQL के लिए Android ऐप को मैन्युअल रूप से बिल्ड करता है। `/codeql-critical-security/android` के अंतर्गत अपलोड करता है।
- `CodeQL macOS Critical Security` — साप्ताहिक/मैन्युअल macOS सुरक्षा शार्ड। Blacksmith macOS पर CodeQL के लिए macOS ऐप को मैन्युअल रूप से बिल्ड करता है, अपलोड किए गए SARIF से निर्भरता बिल्ड परिणामों को फ़िल्टर करता है और `/codeql-critical-security/macos` के अंतर्गत अपलोड करता है। इसे दैनिक डिफ़ॉल्ट से बाहर रखा गया है, क्योंकि सब कुछ साफ़ होने पर भी macOS बिल्ड रनटाइम पर हावी रहता है।

### गंभीर गुणवत्ता श्रेणियाँ

`CodeQL Critical Quality` इसका समकक्ष गैर-सुरक्षा शार्ड है। यह GitHub द्वारा होस्ट किए गए Linux रनर पर संकीर्ण, उच्च-मूल्य सतहों के लिए केवल त्रुटि-गंभीरता वाली, गैर-सुरक्षा JavaScript/TypeScript गुणवत्ता क्वेरी चलाता है, ताकि गुणवत्ता स्कैन Blacksmith रनर-पंजीकरण बजट खर्च न करें। इसका पुल रिक्वेस्ट गार्ड जानबूझकर शेड्यूल किए गए प्रोफ़ाइल से छोटा है: गैर-ड्राफ़्ट PR केवल उन सतहों से मेल खाने वाले शार्ड चलाते हैं जिन्हें वे स्पर्श करते हैं—PR द्वारा रूट किए जा सकने वाले तेरह शार्ड में से: `agent-runtime-boundary`, `channel-runtime-boundary`, `config-boundary`, `core-auth-secrets`, `gateway-runtime-boundary`, `mcp-process-runtime-boundary`, `memory-runtime-boundary`, `network-runtime-boundary`, `plugin-boundary`, `plugin-sdk-package-contract`, `plugin-sdk-reply-runtime`, `provider-runtime-boundary`, और `session-diagnostics-boundary`। `ui-control-plane` और `web-media-runtime-boundary` को PR रन से बाहर रखा जाता है। CodeQL कॉन्फ़िगरेशन और गुणवत्ता वर्कफ़्लो में बदलाव पूरे PR शार्ड सेट को चलाते हैं (नेटवर्क रनटाइम शार्ड अपनी CodeQL कॉन्फ़िगरेशन फ़ाइलों और नेटवर्क का स्वामित्व रखने वाले स्रोत पाथ के आधार पर सक्रिय होता है)।

मैन्युअल डिस्पैच निम्न को स्वीकार करता है:

```text

```

संकीर्ण प्रोफ़ाइल किसी एक गुणवत्ता शार्ड को अलग से चलाने के लिए शिक्षण/पुनरावृत्ति हुक हैं।

गुणवत्ता को सुरक्षा से अलग रखा जाता है, ताकि सुरक्षा संकेत को अस्पष्ट किए बिना गुणवत्ता निष्कर्षों को शेड्यूल, मापा, अक्षम या विस्तारित किया जा सके। Swift, Python और बंडल किए गए Plugin के CodeQL विस्तार को केवल तभी सीमित या शार्ड किए गए अनुवर्ती कार्य के रूप में दोबारा जोड़ा जाना चाहिए, जब संकीर्ण प्रोफ़ाइल का रनटाइम और संकेत स्थिर हो जाए।

## रखरखाव वर्कफ़्लो

### दस्तावेज़ एजेंट

`Docs Agent` वर्कफ़्लो हाल ही में मर्ज किए गए बदलावों के अनुरूप मौजूदा दस्तावेज़ों को बनाए रखने के लिए एक इवेंट-संचालित Codex रखरखाव लेन है। इसका कोई विशुद्ध शेड्यूल नहीं है: `main` पर सफल गैर-बॉट पुश CI रन इसे ट्रिगर कर सकता है और मैन्युअल डिस्पैच इसे सीधे चला सकता है। वर्कफ़्लो-रन आह्वान तब छोड़ दिए जाते हैं जब `main` आगे बढ़ चुका हो या पिछले एक घंटे में कोई अन्य न छोड़ा गया दस्तावेज़ एजेंट रन बनाया गया हो। चलने पर यह पिछले न छोड़े गए दस्तावेज़ एजेंट स्रोत SHA से वर्तमान `main` तक की कमिट सीमा की समीक्षा करता है, इसलिए प्रति घंटे का एक रन पिछले दस्तावेज़ पास के बाद संचित सभी मुख्य बदलावों को समेट सकता है।

### परीक्षण प्रदर्शन एजेंट

`Test Performance Agent` वर्कफ़्लो धीमे परीक्षणों के लिए एक इवेंट-संचालित Codex रखरखाव लेन है। इसका कोई विशुद्ध शेड्यूल नहीं है: `main` पर सफल गैर-बॉट पुश CI रन इसे ट्रिगर कर सकता है, लेकिन यदि उस UTC दिन कोई अन्य वर्कफ़्लो-रन आह्वान पहले ही चल चुका हो या चल रहा हो, तो इसे छोड़ दिया जाता है। मैन्युअल डिस्पैच उस दैनिक गतिविधि गेट को दरकिनार करता है। यह लेन एक पूर्ण-सुइट समूहीकृत Vitest प्रदर्शन रिपोर्ट बनाती है, Codex को व्यापक रिफ़ैक्टर के बजाय केवल छोटे कवरेज-संरक्षणकारी परीक्षण प्रदर्शन सुधार करने देती है, फिर पूर्ण-सुइट रिपोर्ट दोबारा चलाती है और उन बदलावों को अस्वीकार करती है जो उत्तीर्ण बेसलाइन परीक्षण संख्या घटाते हैं। समूहीकृत रिपोर्ट Linux और macOS पर प्रति-कॉन्फ़िगरेशन वॉल टाइम और अधिकतम RSS दर्ज करती है, इसलिए पहले/बाद की तुलना अवधि के अंतर के साथ परीक्षण मेमोरी के अंतर भी दिखाती है। यदि बेसलाइन में विफल परीक्षण हैं, तो Codex केवल स्पष्ट विफलताओं को ठीक कर सकता है और कुछ भी कमिट होने से पहले एजेंट के बाद की पूर्ण-सुइट रिपोर्ट का उत्तीर्ण होना आवश्यक है। जब बॉट पुश मर्ज होने से पहले `main` आगे बढ़ता है, तो लेन सत्यापित पैच को रीबेस करती है, `pnpm check:changed` दोबारा चलाती है और पुश का पुनः प्रयास करती है; विरोध वाले पुराने पैच छोड़ दिए जाते हैं। यह GitHub द्वारा होस्ट किए गए Ubuntu का उपयोग करती है, ताकि Codex एक्शन दस्तावेज़ एजेंट जैसी ही drop-sudo सुरक्षा मुद्रा बनाए रख सके।

### मर्ज के बाद डुप्लिकेट PR

`Duplicate PRs After Merge` वर्कफ़्लो मर्ज के बाद डुप्लिकेट सफ़ाई के लिए एक मैन्युअल अनुरक्षक वर्कफ़्लो है। यह डिफ़ॉल्ट रूप से ड्राई-रन करता है और `apply=true` होने पर केवल स्पष्ट रूप से सूचीबद्ध PR बंद करता है। GitHub में बदलाव करने से पहले यह सत्यापित करता है कि मर्ज किया गया PR वास्तव में मर्ज हो चुका है और प्रत्येक डुप्लिकेट में या तो साझा संदर्भित समस्या है या बदले गए हंक ओवरलैप करते हैं।

```bash
gh workflow run duplicate-after-merge.yml \
  -f landed_pr=70532 \
  -f duplicate_prs='70530,70592' \
  -f apply=true
```

## स्थानीय जाँच गेट और बदलाव रूटिंग

स्थानीय बदली हुई लेन का तर्क `scripts/changed-lanes.mjs` में स्थित है और `scripts/check-changed.mjs` द्वारा निष्पादित किया जाता है। यह स्थानीय जाँच गेट व्यापक CI प्लेटफ़ॉर्म दायरे की तुलना में आर्किटेक्चर सीमाओं को लेकर अधिक सख्त है:

- मुख्य उत्पादन बदलाव मुख्य उत्पादन और मुख्य परीक्षण टाइपचेक के साथ मुख्य लिंट/गार्ड चलाते हैं;
- केवल मुख्य परीक्षण वाले बदलाव केवल मुख्य परीक्षण टाइपचेक और मुख्य लिंट चलाते हैं;
- एक्सटेंशन उत्पादन बदलाव एक्सटेंशन उत्पादन और एक्सटेंशन परीक्षण टाइपचेक के साथ एक्सटेंशन लिंट चलाते हैं;
- केवल एक्सटेंशन परीक्षण वाले बदलाव एक्सटेंशन परीक्षण टाइपचेक और एक्सटेंशन लिंट चलाते हैं;
- सार्वजनिक Plugin SDK या Plugin-अनुबंध बदलाव एक्सटेंशन टाइपचेक तक विस्तृत होते हैं, क्योंकि एक्सटेंशन उन मुख्य अनुबंधों पर निर्भर करते हैं (Vitest एक्सटेंशन स्वीप स्पष्ट परीक्षण कार्य बने रहते हैं);
- केवल रिलीज़ मेटाडेटा वाले संस्करण वृद्धि बदलाव लक्षित संस्करण/कॉन्फ़िगरेशन/रूट-निर्भरता जाँच चलाते हैं;
- अज्ञात रूट/कॉन्फ़िगरेशन बदलाव सुरक्षित रूप से सभी जाँच लेन पर विफल होते हैं।

स्थानीय बदले हुए परीक्षणों की रूटिंग `scripts/test-projects.test-support.mjs` में स्थित है और जानबूझकर `check:changed` से कम खर्चीली है: सीधे परीक्षण संपादन स्वयं को चलाते हैं, स्रोत संपादन पहले स्पष्ट मैपिंग, फिर सहोदर परीक्षण और इम्पोर्ट-ग्राफ़ आश्रितों को प्राथमिकता देते हैं। साझा समूह-कक्ष डिलीवरी कॉन्फ़िगरेशन स्पष्ट मैपिंग में से एक है: समूह के दृश्य-उत्तर कॉन्फ़िगरेशन, स्रोत उत्तर डिलीवरी मोड या संदेश-टूल सिस्टम प्रॉम्प्ट में बदलाव मुख्य उत्तर परीक्षणों के साथ Discord और Slack डिलीवरी रिग्रेशन से होकर गुजरते हैं, ताकि साझा डिफ़ॉल्ट में बदलाव पहले PR पुश से पहले विफल हो जाए। `OPENCLAW_TEST_CHANGED_BROAD=1 pnpm test:changed` का उपयोग केवल तब करें, जब बदलाव पूरे हार्नेस में इतना व्यापक हो कि सस्ता मैप किया गया सेट विश्वसनीय प्रतिनिधि न हो।

## Testbox सत्यापन

Crabbox मेंटेनर Linux प्रमाण के लिए रेपो-स्वामित्व वाला रिमोट-बॉक्स रैपर है। एजेंट
सत्र एक/कुछ केंद्रित परीक्षण और सस्ते स्थिर जाँच स्थानीय रूप से केवल
विश्वसनीय स्रोत के लिए रखते हैं, जब मौजूदा निर्भरता इंस्टॉलेशन तैयार हो। वे बड़े सुइट और
गणना की दृष्टि से गहन कार्य के लिए Crabbox का उपयोग करते हैं, जिनमें बिल्ड, टाइपचेक, लिंट फ़ैन-आउट,
Docker, पैकेज लेन, E2E, लाइव प्रमाण और CI समानता शामिल हैं। विश्वसनीय मेंटेनर का भारी
प्रमाण डिफ़ॉल्ट रूप से `blacksmith-testbox` का उपयोग करता है, और `.crabbox.yaml` भी अब डिफ़ॉल्ट रूप से इसका उपयोग करता है। इसका कॉन्फ़िगर किया गया
वर्कफ़्लो प्रोवाइडर और एजेंट क्रेडेंशियल हाइड्रेट करता है, इसलिए अविश्वसनीय योगदानकर्ता या
फ़ोर्क कोड को इसके बजाय सीक्रेट-रहित फ़ोर्क CI या सैनिटाइज़ किया हुआ प्रत्यक्ष AWS Crabbox उपयोग करना होगा।
सैनिटाइज़ किए गए AWS रन `CRABBOX_ENV_ALLOW=CI` सेट करते हैं,
`--no-hydrate` पास करते हैं, और एक नया अस्थायी रिमोट `HOME` उपयोग करते हैं; यह रेपो की
`OPENCLAW_*` अनुमति-सूची और मौजूदा प्रमाणीकरण प्रोफ़ाइल को अविश्वसनीय कोड तक पहुँचने से रोकता है।
वे उस अविश्वसनीय स्रोत के लिए समर्पित नई वार्म की गई लीज़ उपयोग करते हैं, कभी भी
विश्वसनीय या पहले हाइड्रेट की गई लीज़ नहीं। स्वच्छ विश्वसनीय `main` चेकआउट से इंस्टॉल किया हुआ विश्वसनीय Crabbox
बाइनरी लॉन्च करें और केवल रिमोट PR को
`--fresh-pr` के साथ फ़ेच करें; अविश्वसनीय चेकआउट का रैपर या कॉन्फ़िगरेशन कभी भी स्थानीय रूप से निष्पादित न करें।
`CRABBOX_AWS_INSTANCE_PROFILE` को अनसेट करें और जब तक समाधान किया गया
`aws.instanceProfile` खाली न हो, तब तक बंद स्थिति में विफल हों। किसी भी इंस्टॉल/परीक्षण से पहले, विश्वसनीय
निरपेक्ष-पथ टूल से IMDSv2 टोकन आवश्यक बनाएँ, प्रमाणित करें कि IAM क्रेडेंशियल
एंडपॉइंट 404 लौटाता है, और रिमोट `git rev-parse HEAD` की तुलना पूर्ण
समीक्षित PR हेड SHA से करें। लीज़ को उस SHA से बाँधें और हेड बदलने पर रोककर फिर से वार्म करें।
स्वच्छ `main` से विश्वसनीय `scripts/crabbox-untrusted-bootstrap.sh` को
`--fresh-pr` के साथ अपलोड करें; यह पिन किए गए Node/pnpm इंस्टॉल करता है, SHA और
पैकेज-मैनेजर पिन सत्यापित करता है, `HOME` को पृथक करता है, निर्भरताएँ इंस्टॉल करता है, फिर अनुरोधित
परीक्षण निष्पादित करता है।
सभी `CRABBOX_TAILSCALE*` ओवरराइड अनसेट करें, `--network public
--tailscale=false` लागू करें, एग्ज़िट-नोड/LAN फ़्लैग साफ़ करें, और कोई भी स्क्रिप्ट अपलोड करने से पहले `crabbox inspect` से
बिना किसी Tailscale स्थिति वाली सार्वजनिक नेटवर्किंग रिपोर्ट करना आवश्यक बनाएँ।
स्वामित्व वाली AWS/Hetzner क्षमता Blacksmith आउटेज,
कोटा समस्याओं या स्पष्ट स्वामित्व-क्षमता परीक्षण के लिए फ़ॉलबैक भी बनी रहती है।

एजेंट अनुमानित कार्य के लिए पहले से वार्म नहीं करते। पहला भारी कमांड तैयार होने पर
Testbox को विलंबित रूप से प्राप्त करें, बाद के भारी
कमांड के लिए लौटाई गई `tbx_...` आईडी का पुनः उपयोग करें, प्रत्येक रन पर वर्तमान चेकआउट सिंक करें, और हैंडऑफ़ से पहले इसे रोकें।

Crabbox-समर्थित Blacksmith रन एकबारगी Testbox को वार्म, क्लेम, सिंक, रन, रिपोर्ट और साफ़
करते हैं। अंतर्निहित सिंक सैनिटी जाँच तब तुरंत विफल हो जाती है जब
सिंक किए गए बॉक्स पर `git status --short` कम-से-कम 200 ट्रैक किए गए विलोपन दिखाता है,
जिससे `pnpm-lock.yaml` जैसी गायब होती रूट फ़ाइलें पकड़ी जाती हैं। जानबूझकर
बड़े-विलोपन वाले PR के लिए, रिमोट कमांड हेतु `CRABBOX_ALLOW_MASS_DELETIONS=1` सेट करें।

Crabbox ऐसी स्थानीय Blacksmith CLI इनवोकेशन को भी समाप्त करता है जो
सिंक के बाद कोई आउटपुट दिए बिना पाँच मिनट से अधिक समय तक सिंक चरण में रहती है।
उस गार्ड को अक्षम करने के लिए `CRABBOX_BLACKSMITH_SYNC_TIMEOUT_MS=0` सेट करें, या असामान्य रूप से बड़े स्थानीय अंतर के लिए अधिक
मिलीसेकंड मान उपयोग करें।

पहले रन से पहले, रेपो रूट से रैपर जाँचें:

```bash
pnpm crabbox:run -- --help | sed -n '1,120p'
```

रेपो रैपर ऐसे पुराने Crabbox बाइनरी को अस्वीकार करता है जो चयनित प्रोवाइडर का विज्ञापन नहीं करता, और Blacksmith-समर्थित रन के लिए Crabbox 0.22.0 या नया आवश्यक है, ताकि रैपर को वर्तमान Testbox सिंक, कतार और क्लीनअप व्यवहार मिले। Codex वर्कट्री या लिंक किए गए/स्पार्स चेकआउट में, स्थानीय `pnpm crabbox:run` स्क्रिप्ट से बचें क्योंकि Crabbox शुरू होने से पहले pnpm निर्भरताओं का मिलान कर सकता है; इसके बजाय node रैपर को सीधे इनवोक करें:

```bash
node scripts/crabbox-wrapper.mjs run --provider blacksmith-testbox --timing-json --shell -- "pnpm test <path-or-filter>"
```

सिबलिंग चेकआउट का उपयोग करते समय, टाइमिंग या प्रमाण कार्य से पहले अनदेखा किया गया स्थानीय बाइनरी फिर से बिल्ड करें:

```bash
version="$(git -C ../crabbox describe --tags --always --dirty | sed 's/^v//')" \
  && go build -C ../crabbox -trimpath -ldflags "-s -w -X github.com/openclaw/crabbox/internal/cli.version=${version}" -o bin/crabbox ./cmd/crabbox
```

`.crabbox.yaml` में `blacksmith:` ब्लॉक पहले से संगठन, वर्कफ़्लो, जॉब और रेफ़ के डिफ़ॉल्ट पिन करता है, इसलिए नीचे दिए गए स्पष्ट फ़्लैग वैकल्पिक हैं। परिवर्तित गेट:

```bash
pnpm crabbox:run -- --provider blacksmith-testbox \
  --blacksmith-org openclaw \
  --blacksmith-workflow .github/workflows/ci-check-testbox.yml \
  --blacksmith-job check \
  --blacksmith-ref main \
  --idle-timeout 90m \
  --ttl 240m \
  --timing-json \
  --shell -- \
  "corepack pnpm check:changed"
```

स्थानीय निर्भरताएँ अनुपलब्ध होने या लक्ष्य के फ़ैन-आउट होने पर Testbox पर केंद्रित परीक्षण पुनः चलाएँ:

```bash
pnpm crabbox:run -- --provider blacksmith-testbox \
  --idle-timeout 90m \
  --ttl 240m \
  --timing-json \
  --shell -- \
  "corepack pnpm test <path-or-filter>"
```

पूर्ण सुइट:

```bash
pnpm crabbox:run -- --provider blacksmith-testbox \
  --idle-timeout 90m \
  --ttl 240m \
  --timing-json \
  --shell -- \
  "corepack pnpm test"
```

अंतिम JSON सारांश पढ़ें। उपयोगी फ़ील्ड `provider`, `leaseId`,
`syncDelegated`, `exitCode`, `commandMs` और `totalMs` हैं। प्रत्यायोजित
Blacksmith Testbox रन के लिए, Crabbox रैपर एग्ज़िट कोड और JSON सारांश ही
कमांड परिणाम हैं। लिंक किया गया GitHub Actions रन हाइड्रेशन और कीपअलाइव का स्वामी है; SSH
कमांड पहले ही लौट जाने के बाद Testbox को बाहरी रूप से रोकने पर यह
`cancelled` के रूप में समाप्त हो सकता है। इसे क्लीनअप/स्थिति आर्टिफ़ैक्ट मानें, जब तक
रैपर `exitCode` शून्येतर न हो या कमांड आउटपुट विफल परीक्षण न दिखाए।
एकबारगी Blacksmith-समर्थित Crabbox रन को Testbox स्वचालित रूप से रोकना चाहिए;
यदि कोई रन बाधित हो या क्लीनअप अस्पष्ट हो, तो लाइव बॉक्स का निरीक्षण करें और केवल
अपने बनाए बॉक्स रोकें:

```bash
blacksmith testbox list --all
blacksmith testbox status --id <tbx_id>
blacksmith testbox stop --id <tbx_id>
```

पुनः उपयोग केवल तब करें जब आपको जानबूझकर उसी हाइड्रेट किए गए बॉक्स पर कई कमांड चाहिए:

```bash
node scripts/crabbox-wrapper.mjs run --provider blacksmith-testbox --id <tbx_id> --timing-json --shell -- "corepack pnpm test <path-or-filter>"
pnpm crabbox:stop -- <tbx_id>
```

लीज़ का पुनः उपयोग करें, पुराने स्रोत का नहीं। `--no-sync` छोड़ दें ताकि प्रत्येक रन
वर्तमान चेकआउट अपलोड करे; इसका उपयोग केवल अपरिवर्तित, पहले से सिंक किए गए ट्री को
जानबूझकर फिर से चलाने के लिए करें। अविश्वसनीय योगदानकर्ता/फ़ोर्क कोड को प्रत्येक कमांड के लिए
`CRABBOX_ENV_ALLOW=CI`, `--provider aws --no-hydrate` और एक नया
अस्थायी रिमोट `HOME` उपयोग करना होगा; परीक्षण से पहले उस
सैनिटाइज़ किए गए कमांड के अंदर निर्भरताएँ इंस्टॉल करें। केवल उसी अविश्वसनीय स्रोत के लिए
समर्पित नई वार्म की गई लीज़ का पुनः उपयोग करें; किसी विश्वसनीय या पहले हाइड्रेट की गई लीज़ का कभी नहीं। अविश्वसनीय
चेकआउट का रैपर या कॉन्फ़िगरेशन कभी स्थानीय रूप से निष्पादित न करें: स्वच्छ विश्वसनीय `main` से इंस्टॉल किया हुआ
विश्वसनीय Crabbox बाइनरी लॉन्च करें और प्रत्येक
रन पर `--fresh-pr` पास करें। `CRABBOX_AWS_INSTANCE_PROFILE` को अनसेट रखें, समाधान की गई गैर-रिक्त
इंस्टेंस प्रोफ़ाइल अस्वीकार करें, विश्वसनीय रिमोट IMDS नो-रोल प्रमाण आवश्यक बनाएँ, और
इंस्टॉल/परीक्षण से पहले समीक्षित हेड SHA सत्यापित करें। लीज़ को उस SHA से बाँधें; किसी भी हेड परिवर्तन के बाद रोकें और
फिर से वार्म करें। यदि कोई रिमोट PR मौजूद नहीं है, तो सीक्रेट-रहित फ़ोर्क CI उपयोग करें।
अविश्वसनीय स्रोत के लिए कभी भी `hydrate-github` या क्रेडेंशियल-हाइड्रेटेड Blacksmith वर्कफ़्लो
न चुनें।

यदि Crabbox टूटी हुई परत है लेकिन Blacksmith स्वयं काम करता है, तो प्रत्यक्ष
Blacksmith का उपयोग केवल `list`, `status` और क्लीनअप जैसे निदान के लिए करें। प्रत्यक्ष Blacksmith रन को मेंटेनर प्रमाण मानने से पहले
Crabbox पथ ठीक करें।

यदि `blacksmith testbox list --all` और `blacksmith testbox status` काम करते हैं लेकिन नए
वार्मअप कुछ मिनट बाद भी बिना IP या Actions रन URL के `queued` बने रहते हैं,
तो इसे Blacksmith प्रोवाइडर, कतार, बिलिंग या संगठन-सीमा दबाव मानें। अपने बनाए
कतारबद्ध आईडी रोकें, अधिक Testbox शुरू करने से बचें, और जब कोई Blacksmith डैशबोर्ड,
बिलिंग और संगठन सीमाएँ जाँच रहा हो, तब प्रमाण को नीचे दिए गए
स्वामित्व वाले Crabbox क्षमता पथ पर ले जाएँ।

स्वामित्व वाली Crabbox क्षमता पर केवल तब एस्केलेट करें जब Blacksmith बंद हो, कोटा-सीमित हो, आवश्यक परिवेश अनुपलब्ध हो, या स्वामित्व वाली क्षमता स्पष्ट रूप से लक्ष्य हो:

```bash
CRABBOX_CAPACITY_REGIONS=eu-west-1,eu-west-2,eu-central-1,us-east-1,us-west-2 \
  pnpm crabbox:warmup -- --provider aws --class standard --market on-demand --idle-timeout 90m
pnpm crabbox:hydrate -- --provider aws --id <cbx_id-or-slug>
pnpm crabbox:run -- --provider aws --id <cbx_id-or-slug> --timing-json --shell -- "pnpm check:changed"
pnpm crabbox:stop -- --provider aws <cbx_id-or-slug>
```

AWS दबाव में, जब तक कार्य को वास्तव में 48xlarge-श्रेणी CPU की आवश्यकता न हो, `class=beast` से बचें। `beast` अनुरोध 192 vCPU से शुरू होता है और क्षेत्रीय EC2 Spot या On-Demand Standard कोटा पार करने का सबसे आसान तरीका है। रेपो-स्वामित्व वाला `.crabbox.yaml` डिफ़ॉल्ट रूप से `class: standard`, ऑन-डिमांड मार्केट और `capacity.hints: true` का उपयोग करता है ताकि ब्रोकर किए गए AWS लीज़ चयनित क्षेत्र/मार्केट, कोटा दबाव, Spot फ़ॉलबैक और उच्च-दबाव श्रेणी चेतावनियाँ प्रिंट करें। अधिक भारी व्यापक जाँच के लिए `fast`, standard/fast पर्याप्त न होने के बाद ही `large`, और केवल पूर्ण-सुइट या सभी-Plugin Docker मैट्रिक्स, स्पष्ट रिलीज़/ब्लॉकर सत्यापन या हाई-कोर प्रदर्शन प्रोफ़ाइलिंग जैसी असाधारण CPU-बाउंड लेन के लिए `beast` उपयोग करें। `pnpm check:changed`, केंद्रित परीक्षण, केवल-दस्तावेज़ कार्य, सामान्य लिंट/टाइपचेक, छोटे E2E पुनरुत्पादन या Blacksmith आउटेज ट्रायेज के लिए `beast` उपयोग न करें। क्षमता निदान के लिए `--market on-demand` उपयोग करें ताकि Spot मार्केट अस्थिरता सिग्नल में न मिले।

`.crabbox.yaml` प्रोवाइडर, सिंक और GitHub Actions हाइड्रेशन डिफ़ॉल्ट का स्वामी है। Crabbox सिंक कभी भी `.git` स्थानांतरित नहीं करता, इसलिए हाइड्रेट किया हुआ Actions चेकआउट मेंटेनर-स्थानीय रिमोट और ऑब्जेक्ट स्टोर सिंक करने के बजाय अपना रिमोट Git मेटाडेटा बनाए रखता है, और रेपो कॉन्फ़िगरेशन स्थानीय रनटाइम/बिल्ड आर्टिफ़ैक्ट (जैसे `.artifacts` और परीक्षण रिपोर्ट) भी बाहर रखता है जिन्हें कभी स्थानांतरित नहीं किया जाना चाहिए। `.github/workflows/crabbox-hydrate.yml` चेकआउट, Node/pnpm सेटअप, `origin/main` फ़ेच और स्वामित्व-क्लाउड `crabbox run --id <cbx_id>` कमांड के लिए गैर-गोपनीय परिवेश हैंडऑफ़ का स्वामी है।
