स्थिति: macOS/iOS SwiftUI चैट UI सीधे Gateway WebSocket से संचार करता है। कोई एम्बेडेड ब्राउज़र नहीं, कोई स्थानीय स्थिर सर्वर नहीं।

## यह क्या है

- Gateway के लिए एक नेटिव चैट UI।
- अन्य चैनलों के समान सत्रों और रूटिंग नियमों का उपयोग करता है।
- निर्धारित रूटिंग: उत्तर हमेशा WebChat पर वापस जाते हैं।
- इतिहास हमेशा Gateway से प्राप्त किया जाता है (स्थानीय फ़ाइल की निगरानी नहीं होती)। यदि Gateway पहुँच योग्य नहीं है, तो WebChat केवल-पढ़ने योग्य होता है।

## त्वरित शुरुआत

1. Gateway शुरू करें।
2. WebChat UI (macOS/iOS ऐप) या Control UI चैट टैब खोलें।
3. सुनिश्चित करें कि एक मान्य Gateway प्रमाणीकरण पथ कॉन्फ़िगर किया गया है (डिफ़ॉल्ट रूप से shared-secret, loopback पर भी)।

## यह कैसे काम करता है

- UI Gateway WebSocket से कनेक्ट होता है और `chat.history`, `chat.send`, `chat.inject`, तथा `chat.message.get` RPC विधियों का उपयोग करता है।
- `chat.history` स्थिरता के लिए सीमित है: Gateway लंबे टेक्स्ट फ़ील्ड को छोटा कर सकता है, भारी मेटाडेटा छोड़ सकता है और अत्यधिक बड़े प्रविष्टियों को `[chat.history omitted: message too large]` से बदल सकता है। API क्लाइंट एक कॉल के लिए डिफ़ॉल्ट सीमा को ओवरराइड करने हेतु प्रति-अनुरोध `maxChars` भेज सकते हैं।
- जब कोई दृश्यमान सहायक संदेश `chat.history` में छोटा किया गया हो, तो Control UI एक पार्श्व रीडर खोल सकता है और डिफ़ॉल्ट इतिहास पेलोड बढ़ाए बिना `chat.message.get` के माध्यम से माँग पर पूर्ण, प्रदर्शन-सामान्यीकृत प्रविष्टि प्राप्त कर सकता है। `chat.message.get`, `chat.history` के समान ट्रांसक्रिप्ट शाखा और प्रदर्शन नियमों का उपयोग करता है, लेकिन `messageId` द्वारा एक प्रविष्टि को लक्षित करता है और जब पूर्ण सामग्री अब लौटाई नहीं जा सकती, तब उपलब्ध न होने का वास्तविक कारण लौटाता है।
- `chat.history` केवल-जोड़ने योग्य सत्र फ़ाइलों के लिए सक्रिय ट्रांसक्रिप्ट शाखा का अनुसरण करता है, इसलिए छोड़ी गई पुनर्लेखन शाखाएँ और प्रतिस्थापित प्रॉम्प्ट प्रतियाँ WebChat में रेंडर नहीं होतीं।
- Compaction प्रविष्टियाँ "संक्षिप्त किया गया इतिहास" विभाजक के रूप में रेंडर होती हैं, जो बताता है कि संक्षिप्त ट्रांसक्रिप्ट को चेकपॉइंट के रूप में संरक्षित किया गया है, साथ ही सत्र चेकपॉइंट खोलने की क्रिया भी होती है (अनुमतियाँ मिलने पर शाखा बनाना या पुनर्स्थापित करना)।
- Control UI, `chat.history` द्वारा लौटाए गए आधारभूत Gateway `sessionId` को याद रखता है और उसे बाद की `chat.send` कॉल में शामिल करता है, ताकि पुनः कनेक्शन और पृष्ठ रीफ़्रेश उसी संग्रहीत वार्तालाप को जारी रखें, जब तक उपयोगकर्ता कोई सत्र शुरू या रीसेट न करे।
- `chat.send` एक आइडेम्पोटेंसी कुंजी लेता है (Control UI रन आईडी का उपयोग करता है); Gateway समान कुंजी का पुनः उपयोग करने वाले दोहराए गए अनुरोधों को डीडुप्लिकेट करता है, इसलिए समान सत्र/संदेश/अटैचमेंट के लिए पुनः प्रयास किए गए या डुप्लिकेट प्रगतिशील सबमिशन दूसरा रन नहीं बनाते।
- किसी विशिष्ट संदेश का उत्तर देना (राइट-क्लिक → Reply) लक्ष्य की ट्रांसक्रिप्ट आईडी को `chat.send` पर `replyToId` के रूप में भेजता है। Gateway सत्र इतिहास से उस संदेश को खोजता है और उसी चैनल-अज्ञेय उत्तर-संदर्भ मेटाडेटा को भरता है जिसका उपयोग Discord उत्तर करते हैं: एजेंट `has_reply_context` के साथ प्रेषक लेबल और मुख्य पाठ वाला अविश्वसनीय "वर्तमान उपयोगकर्ता संदेश का उत्तर लक्ष्य" ब्लॉक देखते हैं। (प्रत्यक्ष WebChat सत्रों के लिए मौजूदा बाइट-स्थिर प्रॉम्प्ट नीति के अनुसार, WebChat प्रॉम्प्ट `reply_to_id` जैसी अस्थायी वार्तालाप आईडी को दबाए रखते हैं।) स्थायी ट्रांसक्रिप्ट आईडी के बिना उत्तर लक्ष्य (उदाहरण के लिए लंबित प्रेषण) संदेश के मुख्य पाठ में इनलाइन उद्धरण पर फ़ॉलबैक करते हैं।
- कार्यस्थान स्टार्टअप फ़ाइलें और लंबित `BOOTSTRAP.md` निर्देश एजेंट सिस्टम प्रॉम्प्ट के `# Project Context` अनुभाग के माध्यम से दिए जाते हैं, WebChat उपयोगकर्ता संदेश में कॉपी नहीं किए जाते। यदि बूटस्ट्रैप सामग्री छोटी कर दी जाती है, तो सिस्टम प्रॉम्प्ट को इसके बजाय एक छोटा "बूटस्ट्रैप संदर्भ सूचना" मिलता है; विस्तृत गणनाएँ और कॉन्फ़िगरेशन नियंत्रण निदान सतहों पर रहते हैं।
- `chat.history` पर प्रदर्शन सामान्यीकरण इन्हें हटाता है: केवल-रनटाइम OpenClaw संदर्भ, इनबाउंड एनवेलप रैपर, `[[reply_to_current]]`, `[[reply_to:<id>]]`, और `[[audio_as_voice]]` जैसे इनलाइन डिलीवरी निर्देश टैग, सादा-टेक्स्ट टूल-कॉल XML पेलोड (`<tool_call>`, `<function_call>`, `<tool_calls>`, `<function_calls>`, छोटे किए गए ब्लॉक सहित), तथा लीक हुए ASCII/पूर्ण-चौड़ाई मॉडल नियंत्रण टोकन। वे सहायक प्रविष्टियाँ छोड़ दी जाती हैं जिनका पूरा दृश्यमान टेक्स्ट केवल मौन टोकन `NO_REPLY` होता है (अक्षर-आकार निरपेक्ष)।
- रीज़निंग-फ़्लैग वाले उत्तर पेलोड (`isReasoning: true`) WebChat सहायक सामग्री, ट्रांसक्रिप्ट रीप्ले टेक्स्ट और ऑडियो सामग्री ब्लॉक से बाहर रखे जाते हैं, ताकि केवल-विचार वाले पेलोड दृश्यमान सहायक संदेश या चलाने योग्य ऑडियो के रूप में दिखाई न दें।
- `chat.inject` सीधे ट्रांसक्रिप्ट में एक सहायक नोट जोड़ता है और उसे UI पर प्रसारित करता है (कोई एजेंट रन नहीं)।
- निरस्त रन आंशिक सहायक आउटपुट को UI में दृश्यमान रख सकते हैं। बफ़र किया गया आउटपुट मौजूद होने पर Gateway उस आंशिक टेक्स्ट को ट्रांसक्रिप्ट इतिहास में स्थायी करता है और प्रविष्टि को निरस्तीकरण मेटाडेटा से चिह्नित करता है।

### ट्रांसक्रिप्ट और डिलीवरी मॉडल

WebChat में दो अलग डेटा पथ हैं:

- SQLite ट्रांसक्रिप्ट पंक्तियाँ स्थायी मॉडल/रनटाइम ट्रांसक्रिप्ट हैं। सामान्य एजेंट रन के लिए, एम्बेडेड OpenClaw रनटाइम सत्र एक्सेसर के माध्यम से मॉडल को दिखाई देने वाले `user`, `assistant`, और `toolResult` संदेशों को स्थायी करता है। WebChat उस ट्रांसक्रिप्ट में मनमाना डिलीवरी, स्थिति या सहायक टेक्स्ट नहीं लिखता।
- Gateway `ReplyPayload` इवेंट लाइव डिलीवरी प्रक्षेपण हैं: WebChat/चैनल प्रदर्शन, ब्लॉक स्ट्रीमिंग, निर्देश टैग, मीडिया एम्बेडिंग, TTS/ऑडियो फ़्लैग और UI फ़ॉलबैक व्यवहार के लिए सामान्यीकृत। वे स्वयं कैनोनिकल सत्र लॉग नहीं हैं।
- वे हार्नेस जिन्हें `tools.message` के माध्यम से दृश्यमान उत्तर चाहिए, अभी भी WebChat को वर्तमान रन के आंतरिक स्रोत-उत्तर सिंक के रूप में उपयोग करते हैं। उस सक्रिय WebChat रन से लक्ष्य-रहित `message.send` को उसी चैट में प्रक्षेपित और सत्र ट्रांसक्रिप्ट में प्रतिबिंबित किया जाता है; WebChat पुनः उपयोग योग्य आउटबाउंड चैनल नहीं बनता और कभी भी `lastChannel` विरासत में नहीं लेता।
- WebChat सहायक ट्रांसक्रिप्ट प्रविष्टियाँ केवल तब इंजेक्ट करता है जब Gateway सामान्य एम्बेडेड एजेंट टर्न के बाहर प्रदर्शित संदेश का स्वामी होता है: `chat.inject`, गैर-एजेंट कमांड उत्तर, निरस्त आंशिक आउटपुट और WebChat-प्रबंधित मीडिया ट्रांसक्रिप्ट पूरक।
- यदि रन के दौरान लाइव सहायक टेक्स्ट दिखाई देता है लेकिन इतिहास पुनः लोड होने के बाद गायब हो जाता है, तो इस क्रम में जाँचें: क्या SQLite ट्रांसक्रिप्ट में सहायक टेक्स्ट है, क्या `chat.history` प्रदर्शन प्रक्षेपण ने उसे हटा दिया, फिर क्या Control UI आशावादी-टेल मर्ज ने स्थानीय डिलीवरी स्थिति को स्थायी स्नैपशॉट से बदल दिया।

सामान्य एजेंट-रन अंतिम उत्तर स्थायी होने चाहिए क्योंकि एम्बेडेड रनटाइम सहायक `message_end` लिखता है। डिलीवर किए गए अंतिम पेलोड को ट्रांसक्रिप्ट में प्रतिबिंबित करने वाले किसी भी फ़ॉलबैक को पहले उस सहायक टर्न के दोहराव से बचना चाहिए जिसे एम्बेडेड रनटाइम पहले ही लिख चुका है।

## Control UI एजेंट टूल पैनल

- Control UI `/agents` Tools पैनल में `tools.effective(sessionKey=...)` द्वारा समर्थित "अभी उपलब्ध" दृश्य है: वर्तमान सत्र की टूल सूची का सर्वर-व्युत्पन्न, केवल-पढ़ने योग्य प्रक्षेपण, जिसमें कोर, Plugin, चैनल-स्वामित्व वाले और पहले से खोजे गए MCP सर्वर टूल शामिल हैं।
- एक अलग कॉन्फ़िगरेशन-संपादन दृश्य (`tools.catalog` द्वारा समर्थित) प्रोफ़ाइल, प्रति-एजेंट ओवरराइड और कैटलॉग सिमेंटिक्स को शामिल करता है।
- रनटाइम उपलब्धता सत्र-स्कोप वाली है। समान एजेंट पर सत्र बदलने से "अभी उपलब्ध" सूची बदल सकती है। यदि कॉन्फ़िगर किए गए MCP सर्वर अंतिम खोज के बाद से कनेक्ट या परिवर्तित नहीं हुए हैं, तो पैनल रीड पथ से MCP ट्रांसपोर्ट चुपचाप शुरू करने के बजाय एक सूचना दिखाता है।
- कॉन्फ़िगरेशन संपादक रनटाइम उपलब्धता का संकेत नहीं देता; प्रभावी पहुँच अब भी नीति प्राथमिकता (`allow`/`deny`, प्रति-एजेंट और प्रदाता/चैनल ओवरराइड) का अनुसरण करती है।

## दूरस्थ उपयोग

- दूरस्थ मोड SSH/Tailscale के माध्यम से Gateway WebSocket की टनल बनाता है।
- आपको अलग WebChat सर्वर चलाने की आवश्यकता नहीं है।

## कॉन्फ़िगरेशन संदर्भ (WebChat)

पूर्ण कॉन्फ़िगरेशन: [कॉन्फ़िगरेशन](/hi/gateway/configuration)

WebChat में कोई स्थायी कॉन्फ़िगरेशन अनुभाग नहीं है। Gateway अंतर्निहित `chat.history` प्रदर्शन सीमा का उपयोग करता है; API क्लाइंट एक कॉल के लिए इसे ओवरराइड करने हेतु प्रति-अनुरोध `maxChars` भेज सकते हैं। विरासती `channels.webchat` और `gateway.webchat` कॉन्फ़िगरेशन सेवानिवृत्त हैं; उन्हें हटाने के लिए `openclaw doctor --fix` चलाएँ।

संबंधित वैश्विक विकल्प:

- `gateway.port`, `gateway.bind`: WebSocket होस्ट/पोर्ट।
- `gateway.auth.mode`, `gateway.auth.token`, `gateway.auth.password`:
  shared-secret WebSocket प्रमाणीकरण।
- `gateway.auth.allowTailscale`: सक्षम होने पर ब्राउज़र Control UI चैट टैब Tailscale
  Serve पहचान हेडर का उपयोग कर सकता है।
- `gateway.auth.mode: "trusted-proxy"`: पहचान-जागरूक **non-loopback** प्रॉक्सी स्रोत के पीछे ब्राउज़र क्लाइंट के लिए रिवर्स-प्रॉक्सी प्रमाणीकरण ([विश्वसनीय प्रॉक्सी प्रमाणीकरण](/hi/gateway/trusted-proxy-auth) देखें)।
- `gateway.remote.url`, `gateway.remote.token`, `gateway.remote.password`: दूरस्थ Gateway लक्ष्य।
- `session.*`: सत्र संग्रहण और मुख्य कुंजी के डिफ़ॉल्ट।
