Control UI, Gateway द्वारा सर्व किया जाने वाला एक छोटा **Vite + Lit** सिंगल-पेज ऐप है:

- डिफ़ॉल्ट: `http://<host>:18789/`
- वैकल्पिक प्रीफ़िक्स: `gateway.controlUi.basePath` सेट करें (उदा. `/openclaw`)

यह उसी पोर्ट पर **सीधे Gateway WebSocket से** संचार करता है।

## तुरंत खोलें (स्थानीय)

यदि Gateway उसी कंप्यूटर पर चल रहा है, तो [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (या [http://localhost:18789/](http://localhost:18789/)) खोलें।

यदि पेज लोड नहीं होता, तो पहले Gateway शुरू करें: `openclaw gateway`।

<Note>
नेटिव Windows LAN बाइंड पर, Windows Firewall या संगठन-प्रबंधित Group Policy विज्ञापित LAN URL को तब भी ब्लॉक कर सकती है, जब Gateway होस्ट पर `127.0.0.1` काम करता हो। Windows होस्ट पर `openclaw gateway status --deep` चलाएँ; यह संभावित रूप से ब्लॉक किए गए पोर्ट, प्रोफ़ाइल विसंगतियों और ऐसी स्थानीय फ़ायरवॉल नियमों की रिपोर्ट करता है जिन्हें नीति अनदेखा कर सकती है।
</Note>

WebSocket हैंडशेक के दौरान प्रमाणीकरण इस माध्यम से दिया जाता है:

- `connect.params.auth.token`
- `connect.params.auth.password`
- Tailscale Serve पहचान हेडर, जब `gateway.auth.allowTailscale: true`
- विश्वसनीय-प्रॉक्सी पहचान हेडर, जब `gateway.auth.mode: "trusted-proxy"`

डैशबोर्ड सेटिंग पैनल वर्तमान ब्राउज़र टैब सत्र और चयनित Gateway URL के लिए एक टोकन रखता है; पासवर्ड सहेजे नहीं जाते। ऑनबोर्डिंग आम तौर पर पहले कनेक्शन पर साझा-गुप्त प्रमाणीकरण के लिए Gateway टोकन बनाती है, लेकिन जब `gateway.auth.mode`, `"password"` हो, तो पासवर्ड प्रमाणीकरण भी काम करता है।

## डिवाइस पेयरिंग (पहला कनेक्शन)

किसी नए ब्राउज़र या डिवाइस से कनेक्ट करने के लिए आम तौर पर **एक बार की पेयरिंग स्वीकृति** आवश्यक होती है, जो `disconnected (1008): pairing required` के रूप में दिखाई जाती है।

<Steps>
  <Step title="लंबित अनुरोधों की सूची देखें">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="अनुरोध ID द्वारा स्वीकृत करें">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

यदि ब्राउज़र बदले हुए प्रमाणीकरण विवरण (भूमिका/स्कोप/सार्वजनिक कुंजी) के साथ पेयरिंग का पुनः प्रयास करता है, तो पिछला लंबित अनुरोध अप्रचलित हो जाता है और नया `requestId` बनाया जाता है; स्वीकृति देने से पहले `openclaw devices list` फिर से चलाएँ।

पहले से पेयर किए गए ब्राउज़र को पठन पहुँच से लेखन/व्यवस्थापक पहुँच पर स्विच करना मौन पुनः कनेक्शन नहीं, बल्कि स्वीकृति अपग्रेड माना जाता है: OpenClaw पुरानी स्वीकृति को सक्रिय रखता है, व्यापक पुनः कनेक्शन को ब्लॉक करता है और आपसे नए स्कोप सेट को स्पष्ट रूप से स्वीकृत करने के लिए कहता है।

स्वीकृत होने के बाद डिवाइस याद रखा जाता है और तब तक पुनः स्वीकृति की आवश्यकता नहीं होगी, जब तक आप `openclaw devices revoke --device <id> --role <role>` से उसकी स्वीकृति रद्द नहीं करते। टोकन रोटेशन, निरस्तीकरण और Paperclip / `openclaw_gateway` के प्रथम-रन स्वीकृति प्रवाह के लिए [डिवाइस CLI](/hi/cli/devices) देखें।

<Note>
- सीधे स्थानीय लूपबैक ब्राउज़र कनेक्शन (`127.0.0.1` / `localhost`) स्वतः स्वीकृत होते हैं।
- जब `gateway.auth.allowTailscale: true` हो, Tailscale पहचान सत्यापित हो और ब्राउज़र अपनी डिवाइस पहचान प्रस्तुत करे, तो Tailscale Serve Control UI ऑपरेटर सत्रों के लिए पेयरिंग राउंड ट्रिप छोड़ सकता है। डिवाइस पहचान रहित ब्राउज़र और Node-भूमिका कनेक्शन अब भी सामान्य डिवाइस जाँच का पालन करते हैं।
- सीधे Tailnet बाइंड, LAN ब्राउज़र कनेक्शन और डिवाइस पहचान रहित ब्राउज़र प्रोफ़ाइल के लिए अब भी स्पष्ट स्वीकृति आवश्यक है।
- हर ब्राउज़र प्रोफ़ाइल एक विशिष्ट डिवाइस ID बनाती है, इसलिए ब्राउज़र बदलने या ब्राउज़र डेटा साफ़ करने पर फिर से पेयरिंग करनी पड़ती है।

</Note>

## मोबाइल डिवाइस पेयर करें

पहले से पेयर किया हुआ व्यवस्थापक टर्मिनल खोले बिना iOS/Android कनेक्शन QR बना सकता है:

<Steps>
  <Step title="मोबाइल पेयरिंग खोलें">
    **डिवाइस** चुनें, फिर **डिवाइस** कार्ड में **मोबाइल डिवाइस पेयर करें** पर क्लिक करें।
  </Step>
  <Step title="फ़ोन कनेक्ट करें">
    OpenClaw मोबाइल ऐप में **सेटिंग्स** → **Gateway** खोलें और QR कोड स्कैन करें। इसके बजाय आप सेटअप कोड कॉपी करके पेस्ट कर सकते हैं।
  </Step>
  <Step title="कनेक्शन की पुष्टि करें">
    आधिकारिक iOS/Android ऐप स्वचालित रूप से कनेक्ट हो जाता है। यदि **स्वीकृति लंबित** में कोई अनुरोध दिखाई देता है, तो उसे स्वीकृत करने से पहले उसकी भूमिका और स्कोप की समीक्षा करें।
  </Step>
</Steps>

सेटअप कोड बनाने के लिए `operator.admin` आवश्यक है; इसके बिना सत्रों के लिए बटन अक्षम रहता है। सेटअप कोड में अल्पकालिक बूटस्ट्रैप क्रेडेंशियल होता है, इसलिए जब तक QR और कॉपी किया गया कोड मान्य हों, उनके साथ पासवर्ड जैसा व्यवहार करें। रिमोट पेयरिंग के लिए Gateway का `wss://` पर रिज़ॉल्व होना आवश्यक है (उदाहरण के लिए, Tailscale Serve/Funnel के माध्यम से); साधारण `ws://` केवल लूपबैक और निजी LAN पतों तक सीमित है। सुरक्षा और फ़ॉलबैक के पूरे विवरण के लिए [पेयरिंग](/hi/channels/pairing#pair-from-the-control-ui-recommended) देखें।

## व्यक्तिगत पहचान (ब्राउज़र-स्थानीय)

Control UI साझा सत्रों में श्रेय देने के लिए आउटगोइंग संदेशों से जुड़ी प्रति-ब्राउज़र व्यक्तिगत पहचान (प्रदर्शन नाम और अवतार) का समर्थन करता है। यह वर्तमान ब्राउज़र प्रोफ़ाइल के दायरे वाले ब्राउज़र स्टोरेज में रहती है और न तो अन्य डिवाइसों से सिंक होती है, न ही आपके भेजे गए संदेशों पर सामान्य ट्रांसक्रिप्ट लेखकत्व मेटाडेटा से आगे सर्वर पर सहेजी जाती है। साइट डेटा साफ़ करने या ब्राउज़र बदलने पर यह फिर से खाली हो जाती है।

सहायक अवतार ओवरराइड भी इसी ब्राउज़र-स्थानीय पैटर्न का पालन करता है: अपलोड किए गए ओवरराइड स्थानीय रूप से Gateway द्वारा रिज़ॉल्व की गई पहचान पर दिखाई देते हैं और `config.patch` से होकर कभी राउंड ट्रिप नहीं करते। फ़ील्ड को सीधे लिखने वाले गैर-UI क्लाइंट के लिए साझा `ui.assistant.avatar` कॉन्फ़िग फ़ील्ड अब भी उपलब्ध है।

## रनटाइम कॉन्फ़िग एंडपॉइंट

Control UI अपनी रनटाइम सेटिंग्स `/control-ui-config.json` से प्राप्त करता है, जिसे Gateway के Control UI बेस पाथ के सापेक्ष रिज़ॉल्व किया जाता है (उदाहरण के लिए, बेस पाथ `/__openclaw__/` के अंतर्गत `/__openclaw__/control-ui-config.json`)। यह एंडपॉइंट भी शेष HTTP सतह के समान Gateway प्रमाणीकरण द्वारा नियंत्रित है: अप्रमाणित ब्राउज़र इसे प्राप्त नहीं कर सकते, और सफल अनुरोध के लिए मान्य Gateway टोकन/पासवर्ड, Tailscale Serve पहचान या विश्वसनीय-प्रॉक्सी पहचान आवश्यक है।

## Gateway होस्ट की स्थिति

Gateway मशीन, LAN पता, ऑपरेटिंग सिस्टम, रनटाइम, अपटाइम, CPU लोड, मेमोरी और स्टेट-वॉल्यूम डिस्क स्थान वाला **Gateway Host** कार्ड देखने के लिए **Settings → General** खोलें। दिखाई देने के दौरान कार्ड `system.info` Gateway RPC के माध्यम से हर 10 सेकंड में रीफ़्रेश होता है, जिसके लिए `operator.read` स्कोप आवश्यक है। पुराने Gateway और उस स्कोप के बिना कनेक्शन कार्ड को प्रदर्शित नहीं करते।

## भाषा समर्थन

Control UI पहली बार लोड होने पर आपके ब्राउज़र लोकेल के आधार पर स्वयं को स्थानीयकृत करता है। बाद में इसे बदलने के लिए **Settings -> General -> Language** खोलें (चयनकर्ता General पेज पर है, Appearance के अंतर्गत नहीं)।

- समर्थित लोकेल: `en`, `ar`, `de`, `es`, `fa`, `fr`, `hi`, `id`, `it`, `ja-JP`, `ko`, `nl`, `pl`, `pt-BR`, `ru`, `th`, `tr`, `uk`, `vi`, `zh-CN`, `zh-TW`
- गैर-अंग्रेज़ी अनुवाद ब्राउज़र में लेज़ी-लोड किए जाते हैं।
- चयनित लोकेल ब्राउज़र स्टोरेज में सहेजा जाता है और भविष्य की विज़िट पर पुनः उपयोग होता है।
- अनुपलब्ध अनुवाद कुंजियाँ अंग्रेज़ी पर फ़ॉलबैक करती हैं।

दस्तावेज़ों के अनुवाद भी गैर-अंग्रेज़ी लोकेल के इसी सेट के लिए बनाए जाते हैं, लेकिन दस्तावेज़ साइट का अंतर्निहित Mintlify भाषा चयनकर्ता केवल Mintlify द्वारा स्वीकार किए जाने वाले लोकेल कोड सूचीबद्ध करता है। थाई (`th`) और फ़ारसी (`fa`) दस्तावेज़ अब भी प्रकाशन रेपो में बनाए जाते हैं; जब तक Mintlify उन कोड का समर्थन नहीं करता, वे उस चयनकर्ता में दिखाई नहीं दे सकते।

## स्वरूप थीम

Appearance पैनल में अंतर्निहित Claw, Knot और Dash थीम (Claw डिफ़ॉल्ट है), साथ ही एक ब्राउज़र-स्थानीय tweakcn आयात स्लॉट होता है। थीम आयात करने के लिए [tweakcn संपादक](https://tweakcn.com/editor/theme) खोलें, कोई थीम चुनें या बनाएँ, **Share** पर क्लिक करें और कॉपी किया गया लिंक Appearance में पेस्ट करें। आयातक `https://tweakcn.com/r/themes/<id>` रजिस्ट्री URL, `https://tweakcn.com/editor/theme?theme=amethyst-haze` जैसे संपादक URL, सापेक्ष `/themes/<id>` पाथ, रॉ थीम ID और `amethyst-haze` जैसे डिफ़ॉल्ट थीम नाम भी स्वीकार करता है।

आयातित थीम केवल वर्तमान ब्राउज़र प्रोफ़ाइल में संग्रहीत होती हैं; वे Gateway कॉन्फ़िग में नहीं लिखी जातीं और डिवाइसों के बीच सिंक नहीं होतीं। आयातित थीम बदलने पर एक स्थानीय स्लॉट अपडेट होता है; यदि आयातित थीम सक्रिय थी, तो उसे साफ़ करने पर Claw फिर से सक्रिय हो जाता है।

Appearance में Text size सेटिंग भी होती है। यह चैट टेक्स्ट, कंपोज़र टेक्स्ट, टूल कार्ड और चैट साइडबार पर लागू होती है, और टेक्स्ट इनपुट को कम-से-कम 16px रखती है ताकि मोबाइल Safari फ़ोकस होने पर स्वतः ज़ूम न करे।

थीम, थीम मोड, टेक्स्ट आकार, भाषा और चैट प्रदर्शन प्राथमिकताएँ Gateway कॉन्फ़िग (`ui.prefs`) के माध्यम से सिंक होती हैं, इसलिए वे सभी डिवाइसों पर आपके साथ रहती हैं और एजेंट उन्हें स्वीकृति गेट के माध्यम से बदल सकते हैं — कनेक्टेड क्लाइंट Gateway की `config.changed` सूचना के माध्यम से बदलाव तुरंत लागू करते हैं। प्रत्येक ब्राउज़र तत्काल बूट के लिए स्थानीय प्रतिलिपि रखता है; जो क्लाइंट कॉन्फ़िग नहीं लिख सकते (दर्शक स्कोप, ऑफ़लाइन), वे बदलावों को डिवाइस-स्थानीय रखते हैं। [कॉन्फ़िगरेशन संदर्भ](/hi/gateway/configuration-reference#ui) देखें।

## OpenClaw सिस्टम रखरखाव

सिस्टम सेटअप और मरम्मत एजेंट से बात करने के लिए **Settings → Ask OpenClaw** खोलें। ऑनबोर्डिंग के बाहर यह पेज प्रत्येक विज़िट में अधिकतम एक खारिज किया जा सकने वाला इवेंट चिप दिखा सकता है। यह नियमित Gateway ट्रैफ़िक के लिए मौन रहता है और केवल उन स्वास्थ्य स्नैपशॉट पर प्रतिक्रिया करता है जो अक्षम कॉन्फ़िगरेशन रीलोडर, कॉन्फ़िगर किए गए चैनल के डिस्कनेक्शन/गिरावट, विफल चैनल जाँच या अनुपलब्ध चैनल क्रेडेंशियल की रिपोर्ट करते हैं। कोई नया इवेंट लंबित चिप को तभी बदलता है जब वह अधिक गंभीर हो; चिप को खारिज करने या उपयोग करने से उस विज़िट के लिए इवेंट प्रॉम्प्ट मौन हो जाते हैं। चिप पर क्लिक करने से उसका निदान प्रश्न वास्तविक `openclaw.chat` संदेश के रूप में भेजा जाता है, इसलिए ट्रांसक्रिप्ट अनुरोध दर्ज करता है और OpenClaw निदान करता है। ऑनबोर्डिंग में ये इवेंट चिप कभी नहीं दिखाई देते।

## Plugin प्रबंधित करें

Control UI छोड़े बिना Plugin ब्राउज़ और प्रबंधित करने के लिए साइडबार में **Plugins** खोलें, या कॉन्फ़िगर किए गए Control UI बेस पाथ के सापेक्ष `/settings/plugins` का उपयोग करें। उदाहरण के लिए, `/openclaw` बेस पाथ `/openclaw/settings/plugins` का उपयोग करता है। प्रत्येक वैकल्पिक Plugin अक्षम होने पर भी यह पेज हमेशा उपलब्ध रहता है।

Plugins चार टैब वाला एक हब है: **Installed** और **Discover**, `/settings/plugins` पर Plugin कोड प्रबंधित करते हैं, **Skills**, `/skills` पर प्रति-एजेंट Skills प्रबंधक होस्ट करता है, और **Workshop**, `/skills/workshop` पर Skill Workshop प्रस्ताव समीक्षा होस्ट करता है। प्रत्येक टैब अपना URL बनाए रखता है और साइडबार उन सभी के लिए एक ही Plugins प्रविष्टि दिखाता है।

**Installed** टैब श्रेणी के अनुसार समूहीकृत पूरी स्थानीय इन्वेंट्री और अवलोकन संख्याएँ दिखाता है। प्रत्येक पंक्ति विस्तृत दृश्य खोलती है; उसका ओवरफ़्लो (`…`) मेनू Plugin को सक्षम या अक्षम करता है और बाहरी रूप से इंस्टॉल किए गए Plugin के लिए **Remove** प्रदान करता है। यह कॉन्फ़िगर किए गए [MCP सर्वर](/hi/cli/mcp) भी सूचीबद्ध करता है और उन्हें वहीं जोड़ने, अक्षम करने और हटाने का समर्थन करता है। यही सर्वर नियंत्रण **Settings → MCP** पर उपलब्ध हैं। **Discover** टैब स्टोर है: OpenClaw में शामिल फ़ीचर्ड Plugin, आधिकारिक बाहरी Plugin और लोकप्रिय सेवाओं के लिए एक-क्लिक MCP कनेक्टर। खोज बॉक्स में टाइप करने पर [ClawHub](https://clawhub.ai/plugins) में वहीं क्वेरी की जाती है और डाउनलोड संख्याओं तथा स्रोत-सत्यापन बैज वाला **From ClawHub** अनुभाग जोड़ा जाता है। डीप लिंक `/settings/plugins?tab=discover` से सीधे स्टोर को लक्षित कर सकते हैं।

**Skills** टैब चयनित एजेंट के दायरे में Skills स्थिति रिपोर्ट, सक्षम/अक्षम टॉगल, API कुंजी प्रविष्टि और इनलाइन ClawHub Skills खोज रखता है। **Workshop** टैब [Skill प्रस्तावों](/hi/tools/skill-workshop) के लिए Skill Workshop बोर्ड और Today समीक्षा प्रवाह रखता है। **Skill विचार खोजें** पर्याप्त सत्रों की सीमित विंडो की नवीनतम से सबसे पुराने क्रम में समीक्षा करता है और किसी भी परिणाम को लंबित प्रस्तावों के रूप में छोड़ता है। पैनल संचयी कवरेज दिखाता है; **पहले के काम को स्कैन करें** स्थायी कर्सर से जारी रखता है, फिर पुराना इतिहास समाप्त होने के बाद **नए काम को स्कैन करें** बन जाता है। स्वायत्त स्व-अधिगम अक्षम होने पर भी मैन्युअल इतिहास समीक्षा काम करती है और चयनित एजेंट के कॉन्फ़िगर किए गए मॉडल का उपयोग करती है।

शामिल Plugin पहले से ही Gateway पर मौजूद होते हैं और **Install** के बजाय **Enable** या **Disable** दिखाते हैं। उदाहरण के लिए, Workboard OpenClaw में शामिल है लेकिन डिफ़ॉल्ट रूप से अक्षम है, इसलिए उसकी कार्रवाई **Enable** है। बंडल किए गए Plugin हटाए नहीं जा सकते, केवल अक्षम किए जा सकते हैं।

कैटलॉग पढ़ने और ClawHub में खोजने के लिए `operator.read` आवश्यक है। किसी Plugin को इंस्टॉल,
सक्षम, अक्षम या हटाने और MCP सर्वर बदलने के लिए
`operator.admin` आवश्यक है; केवल-पढ़ने वाले ऑपरेटरों के लिए ये कार्रवाइयाँ अक्षम रहती हैं।

ClawHub इंस्टॉल Gateway के माध्यम से चलते हैं और अन्य Gateway-मध्यस्थ इंस्टॉल की तरह ही विश्वास, अखंडता
और Plugin-इंस्टॉल नीति जाँच बनाए रखते हैं। Plugin कोड इंस्टॉल
करने या हटाने के लिए Gateway को पुनः आरंभ करना आवश्यक है। इंस्टॉल किए गए Plugin को सक्षम या अक्षम करना
बिना पुनः आरंभ किए लागू हो सकता है, जब Plugin और वर्तमान
Gateway रनटाइम इसका समर्थन करते हों; अन्यथा UI बताता है कि पुनः आरंभ करना
आवश्यक है। OAuth-समर्थित MCP कनेक्टर जोड़े जाने के बाद CLI से एक बार
`openclaw mcp login <name>` की आवश्यकता होती है।

यह पृष्ठ जानबूझकर इन्वेंट्री, खोज, इंस्टॉल, सक्षमता
और हटाने पर केंद्रित है। मनमाने npm, git या
स्थानीय-पथ स्रोतों, अपडेट और उन्नत Plugin कॉन्फ़िगरेशन के लिए [`openclaw plugins`](/hi/cli/plugins) का उपयोग करें।

## साइडबार नेविगेशन

साइडबार सब कुछ एजेंट के इर्द-गिर्द व्यवस्थित करता है। शीर्ष पर पहचान पंक्ति सक्रिय एजेंट है; उसके नीचे **Pages** अनुभाग **Home** से शुरू होता है — एजेंट का निरंतर मुख्य सत्र, जिस पर उसकी अपठित या चालू स्थिति का बैज होता है — और उसके बाद पिन किए गए गंतव्य (डिफ़ॉल्ट रूप से **Usage**, **Automations** और **Plugins**) आते हैं। Pages हेडर का अनुकूलन नियंत्रण ऐसा मेनू खोलता है जिसमें Plugin द्वारा प्रदान किए गए टैब सहित हर अन्य गंतव्य और **Edit pinned items** शामिल होते हैं; नेविगेशन क्षेत्र पर राइट-क्लिक करने से पिन संपादक सीधे खुलता है। नीचे दी गई सत्र सूची क्षेत्रों में विभाजित होती है: एजेंट के चैट सत्रों के लिए **Threads** (मुख्य सत्र Home के पीछे रहता है; उसके द्वारा बनाए गए सत्र यहाँ शीर्ष-स्तरीय थ्रेड के रूप में दिखाई देते हैं और नामित थ्रेड बिना प्रकार उपसर्ग के दिखते हैं), समूह और रूम वार्तालापों के लिए **Groups**, और प्रबंधित वर्कट्री या exec Node से बंधे सत्रों के लिए **Coding** (पंक्तियाँ Node होस्ट के साथ एक `repo ⎇ branch` पंक्ति दिखाती हैं), ACP-समर्थित हार्नेस सत्र और Codex/Claude CLI कैटलॉग। पहली बार चलाने पर Coding संकुचित अवस्था में शुरू होता है और आपकी पसंद याद रखता है; उसके संकुचित हेडर में वास्तविक संख्या बनी रहती है और उसमें मौजूद सत्रों के काम करते समय चालू होने का संकेतक दिखता है। कस्टम समूह (सत्र `category`) और **Pinned** पंक्तियाँ Threads के ऊपर रहती हैं, और किसी सत्र को कस्टम समूह में असाइन करना हमेशा स्वचालित क्षेत्र वर्गीकरण पर प्राथमिकता पाता है। Threads हेडर में क्रम नियंत्रण (Created या Last updated, साथ में Group by टॉगल) और नया सत्र पृष्ठ खोलने वाला **+** होता है। किसी सत्र को खोलने पर पंक्तियों का क्रम बदले बिना चयन हाइलाइट स्थानांतरित होता है। हाल में चाइल्ड रन वाले पैरेंट सत्र प्रकटीकरण नियंत्रण और चाइल्ड संख्या दिखाते हैं; साइडबार छोड़े बिना नेस्टेड चाइल्ड सत्रों, लाइव या अंतिम स्थिति और रनटाइम का निरीक्षण करने के लिए इसे विस्तृत करें। किसी चाइल्ड को चुनने पर उसकी चैट खुलती है और उसका पूर्वज पथ स्वतः दिखाई देता है। चाइल्ड पंक्तियाँ रूट समूहीकरण, पिनिंग, ड्रैगिंग, बहु-चयन और पृष्ठांकन से बाहर रहती हैं; संकुचित क्षेत्र दृश्यमान पृष्ठ बजट का उपयोग नहीं करते। पिछली बार पढ़े जाने के बाद नई गतिविधि वाले सत्र अपठित बिंदु दिखाते हैं और उन्हें खोलने पर वे पढ़े हुए चिह्नित हो जाते हैं। एजेंट छोटी, समय-सीमित स्थिति पंक्ति भी प्रकाशित कर सकता है और वैकल्पिक रूप से चुने हुए एम्बर आइकन के साथ ध्यान देने का अनुरोध कर सकता है; सत्र खोलने, अगला संदेश भेजने, उसे स्पष्ट रूप से साफ़ करने या उसका TTL समाप्त होने पर यह घोषणा साफ़ हो जाती है। क्लाउड-वर्कर जीवनचक्र स्थितियाँ ग्लोब बैज का उपयोग करती हैं; स्थानीय और पुनः प्राप्त सत्र प्लेसमेंट बैज नहीं दिखाते, क्योंकि स्थानीय निष्पादन डिफ़ॉल्ट है। प्रत्येक रूट सत्र पंक्ति में संदर्भ मेनू (कबाब बटन या राइट-क्लिक) होता है, जिसमें Pin/Unpin, Mark as unread/read, Rename, Fork, Move to group (New group और Remove from group सहित), Archive और Delete शामिल हैं; टच लेआउट प्रत्यक्ष पिन और मेनू नियंत्रणों को दृश्यमान रखते हैं। Cmd/Ctrl-क्लिक रूट पंक्तियों को बहु-चयन में टॉगल करता है और Shift-क्लिक उसे दृश्यमान क्रम में विस्तृत करता है; फिर किसी चयनित पंक्ति पर मेनू खोलने से बैच कार्रवाइयाँ (Mark N as unread/read, Move N to group, Archive N, Delete N) मिलती हैं, जो प्रत्येक चयनित सत्र पर लागू होती हैं और बैच डिलीट के लिए एक ही पुष्टि माँगती हैं। किसी रूट सत्र को पिन करने के लिए उसे **Pinned** पर खींचें या स्थानांतरित करने के लिए किसी कस्टम समूह पर खींचें। कस्टम समूह हेडर संकुचित, विस्तृत या क्रम बदलने के लिए खींचे जा सकते हैं; समूह के नाम और उनका क्रम Gateway (`sessions.groups.*`) में रहते हैं, इसलिए वे विभिन्न ब्राउज़र में आपके साथ बने रहते हैं, जबकि संकुचित स्थिति ब्राउज़र प्रोफ़ाइल में रहती है। समूह हेडर में संदर्भ मेनू (कबाब बटन या राइट-क्लिक) भी होता है, जिसमें Rename group, New group और Delete group शामिल हैं; किसी समूह का नाम बदलने या उसे हटाने से आर्काइव किए गए सत्रों सहित प्रत्येक सदस्य सत्र सर्वर-साइड अपडेट होता है, और समूह हटाने पर उसके सत्र बने रहते हैं तथा वापस Threads में चले जाते हैं।

## नया सत्र पृष्ठ

साइडबार सत्र-सूची हेडर में **+** `/new` पर पूर्ण-पृष्ठ ड्राफ़्ट खोलता है: पहला संदेश भेजने तक कुछ भी नहीं बनाया जाता। संदेश बॉक्स के ऊपर लक्ष्य पंक्ति चुनती है कि सत्र कहाँ काम करेगा: एजेंट (बहु-एजेंट सेटअप), exec कहाँ चलता है (**Gateway · local** या `system.run` उजागर करने वाला युग्मित Node; `operator.admin` आवश्यक), फ़ोल्डर (डिफ़ॉल्ट रूप से एजेंट कार्यक्षेत्र; अन्य पूर्ण Gateway पथों के लिए `operator.admin` और एक वर्कट्री आवश्यक है), और आधार-ब्रांच चयनकर्ता वाला वैकल्पिक **Worktree** टॉगल (`worktrees.branches` द्वारा समर्थित, इसलिए कोई fetch नहीं होता) तथा वैकल्पिक वर्कट्री नाम (ब्रांच `openclaw/<name>` बन जाती है)। कंपोज़र फ़ुटर नए सत्र का मॉडल और रीजनिंग स्तर चुनता है, और क्लाउड प्रारंभ सत्र को उसके वर्कर तक भेजने से पहले दोनों विकल्पों को स्थायी करता है। फ़ोल्डर चिप का ब्राउज़ बटन केवल-एडमिन `fs.listDir` विधि द्वारा समर्थित इनलाइन डायरेक्टरी पिकर खोलता है। उसका शीर्ष स्तर Gateway और प्रत्येक ज्ञात Node दिखाता है; ऑफ़लाइन Node और डायरेक्टरी-ब्राउज़िंग समर्थन के बिना Node दृश्यमान लेकिन अक्षम रहते हैं। Gateway चुनने पर वर्तमान फ़ोल्डर या Gateway होम से शुरुआत होती है। सक्षम Node चुनने पर उस Node का होस्ट फ़ाइल सिस्टम ब्राउज़ होता है, exec उससे बंध जाता है और चयनित पूर्ण Node पथ का सीधे उपयोग होता है (प्रबंधित वर्कट्री केवल Gateway पर रहते हैं)। सबमिट करने पर पहले संदेश के साथ `sessions.create` कॉल होता है, इसलिए रन उसी राउंड-ट्रिप में शुरू होता है और UI नए सत्र की चैट पर चला जाता है। यदि Gateway सत्र बना देता है लेकिन पहला प्रेषण अस्वीकार कर देता है, तो चैट पुनः लोड होने पर भी प्रॉम्प्ट और त्रुटि सुरक्षित रखती है; **Retry** दूसरा सत्र बनाने के बजाय उसे पहले से बने सत्र के माध्यम से भेजता है।

**Settings** के भीतर समर्पित साइडबार में **Ask OpenClaw** शामिल है और सेटिंग अनुभागों को तेज़ी से खोजने के लिए **Search settings** फ़ील्ड से शुरुआत होती है।

साइडबार के शीर्ष पर **Search** फ़ील्ड कमांड पैलेट (⌘K) खोलता है। साइडबार के शीर्ष पर एजेंट पहचान पंक्ति क्लिक करने से एजेंट मेनू खुलता है; **Home** मुख्य सत्र खोलता है। जब किसी चीज़ पर कार्रवाई आवश्यक होती है — विफल या विलंबित Cron जॉब, समाप्त होने वाला या समाप्त हो चुका मॉडल प्रमाणीकरण — तो साइडबार फ़ुटर के ऊपर छोटे ध्यान चिप दिखाई देते हैं और क्लिक करने पर संबंधित पृष्ठ खुलता है। पहचान पंक्ति एजेंट का अवतार (पहचान चित्र या इमोजी), नाम, कनेक्शन बिंदु और लाइव उपशीर्षक दिखाती है। इसे क्लिक करने पर एजेंट मेनू खुलता है: एजेंट स्विचर (बहु-एजेंट सेटअप), "यह एजेंट क्या कर सकता है?", **Agent settings**, **Settings**, मोबाइल पेयरिंग, **Docs**, बिल्ड चिप और रंग-मोड टॉगल। दस से अधिक एजेंटों वाली सूचियों में फ़िल्टर फ़ील्ड होता है और पिन किए गए एजेंट पहले सूचीबद्ध होते हैं; Agents सेटिंग पृष्ठ से एजेंटों को पिन या अनपिन करें, जहाँ पिन किया गया सेट ब्राउज़र प्रोफ़ाइल में संग्रहीत होता है। एजेंट चुनने पर Chat के साथ Usage, Automations, Tasks, Workboard और Sessions उस एजेंट तक सीमित हो जाते हैं। प्रत्येक सीमित पृष्ठ **All agents** के साथ निकास विकल्प के रूप में **Agent** नियंत्रण देता है; इससे ठोस चैट एजेंट बदले बिना साझा पृष्ठ का दायरा विस्तृत होता है, जबकि प्रत्यक्ष सत्र लिंक फिर भी अपना लक्ष्य खोलते हैं। Agents सेटिंग पृष्ठ अपना `?agent=` चयन बनाए रखता है और साझा पृष्ठ दायरे का अनुसरण नहीं करता। फ़ुटर बार में उत्पाद लोगो, बिल्ड चिप, Gateway कनेक्शन बिंदु और Settings शॉर्टकट होते हैं। जब Gateway `main` के अतिरिक्त किसी अन्य ब्रांच पर स्रोत चेकआउट से चलता है, तो फ़ुटर उस ब्रांच का नाम भी लाल रंग में दिखाता है, ताकि गैर-रिलीज़ Gateway एक नज़र में स्पष्ट हो (रिलीज़ इंस्टॉल इसे कभी नहीं दिखाते)। Shift-Command-Comma ब्राउज़र के Command-Comma शॉर्टकट को ओवरराइड किए बिना **Settings** खोलता है। साइडबार हेडर में संकुचन टॉगल (⌘B) भी होता है; संकुचित करने पर पूर्ण-चौड़ाई कार्यक्षेत्र के लिए साइडबार पूरी तरह छिप जाता है और फ़्लोटिंग विस्तार नियंत्रण (या ⌘B) उसे वापस लाता है; इसके बजाय macOS ऐप उस टॉगल को मूल रूप से टाइटलबार में होस्ट करता है। डेस्कटॉप पर साइडबार ही एकमात्र नेविगेशन क्रोम है और कोई शीर्ष बार नहीं है। संकरे व्यूपोर्ट में साइडबार की जगह कॉम्पैक्ट हेडर पंक्ति के पीछे स्लाइड-ओवर ड्रॉअर आता है, जिसमें ड्रॉअर टॉगल, ब्रांड और कमांड-पैलेट खोज होती है; फ़ोन पर Chat उस नेविगेशन पंक्ति को अपनी शीर्षक पट्टी में समाहित कर लेता है, जहाँ मेनू और खोज नियंत्रण सत्र शीर्षक के बगल में होते हैं। macOS ऐप में अलग हेडर पंक्ति टाइटलबार की खाली जगह को विंडो नियंत्रणों के पास एकल कॉम्पैक्ट पट्टी में समेटती है। नेविगेशन नियमित ब्राउज़र इतिहास का उपयोग करता है, इसलिए ब्राउज़र के पीछे/आगे बटन उसमें आवागमन करते हैं; macOS ऐप विंडो नियंत्रणों के पास मूल साइडबार टॉगल और ट्रैकपैड स्वाइप जेस्चर जोड़ता है, तथा साइडबार विस्तृत होने पर उसके दाएँ किनारे पर पीछे/आगे बटन और संकुचित होने पर मूल खोज (कमांड पैलेट) तथा नया-सत्र बटन दिखाता है।

लंबित अनुमोदन भी साइडबार फ़ुटर के ऊपर ध्यान चिप जोड़ते हैं;
संबंधित Approvals पृष्ठ खोलने के लिए उसे चुनें।

    - थ्रेड्स (`/sessions` पर एक वर्कस्पेस पेज, जिसके साथ एक **Worktrees** टैब है): डिफ़ॉल्ट रूप से कॉन्फ़िगर किए गए एजेंट के सत्र सूचीबद्ध करें, बार-बार उपयोग होने वाले सत्र पिन करें, उनका नाम बदलें, निष्क्रिय सत्रों को आर्काइव या पुनर्स्थापित करें, पुराने अनकॉन्फ़िगर किए गए एजेंट सत्र कुंजियों से फ़ॉलबैक करें और प्रति-सत्र मॉडल/थिंकिंग/फ़ास्ट/वर्बोज़/ट्रेस/रीज़निंग ओवरराइड लागू करें (`sessions.list`, `sessions.patch`)। पिन किए गए सत्र हाल के अनपिन किए गए सत्रों से ऊपर क्रमबद्ध होते हैं; आर्काइव किए गए सत्र Threads पेज के आर्काइव दृश्य में रहते हैं और अपनी ट्रांसक्रिप्ट बनाए रखते हैं। जिन सत्रों में अंतिम बार पढ़े जाने के बाद गतिविधि हुई हो, उनकी पंक्तियाँ अपठित बिंदु दिखाती हैं और उनके साथ अपठित-चिह्नित/पठित-चिह्नित क्रियाएँ होती हैं (`sessions.patch { unread }`), साथ ही एक Fork क्रिया होती है जो ट्रांसक्रिप्ट को एक नए सत्र में शाखित करती है (`sessions.create { parentSessionKey, fork: true }`)। तालिका के ऊपर के अवलोकन टाइल लोड किए गए रोस्टर का सारांश देते हैं (सत्र संख्या, लाइव रन, अपठित सत्र, कुल टोकन), प्रत्येक पंक्ति में लाइव-रन बिंदु के साथ प्रकार का ग्लिफ़ होता है, स्थिति एक साधारण बिंदु और लेबल के रूप में रेंडर होती है, और जब सत्र टोकन तथा कॉन्टेक्स्ट आकार रिपोर्ट करता है तो Tokens कॉलम कॉन्टेक्स्ट-विंडो उपयोग मीटर दिखाता है। पंक्ति प्रबंधन क्रियाएँ प्रति-पंक्ति मेनू (कबाब बटन या राइट-क्लिक) में होती हैं, जो साइडबार के सत्र मेनू को प्रतिबिंबित करता है, और पंक्ति ड्रॉअर अन्य सत्र विवरणों के साथ एजेंट रनटाइम और रन अवधि रखता है।
    - नेटिव Claude और Codex साइडबार कैटलॉग एक समय में एक होस्ट को स्ट्रीम करते हैं, फिर Node कनेक्टिविटी बदलने पर, पेज फ़ोकस होने पर और दृश्यमान रहते हुए अधिकतम प्रत्येक 30 सेकंड में समन्वय करते हैं। कैटलॉग बदलाव अधिक तेज़ फ़ॉलो-अप पास ट्रिगर करते हैं, ताकि नेटिव टूल में बनाए गए सत्र Control UI को पुनः लोड किए बिना दिखाई दें। Claude Desktop की पंक्तियाँ मौजूद होने पर अपना स्थानीय कस्टम-समूह लेबल भी बनाए रखती हैं; OpenClaw उस मैपिंग को Desktop के स्थानीय स्टोर से पढ़ता है और उसमें कभी लिखता नहीं है।
    - सत्र समूहीकरण: Group by नियंत्रण सत्र तालिका को कस्टम समूह, चैनल, प्रकार, एजेंट या तारीख के आधार पर अनुभागों में व्यवस्थित करता है। कस्टम समूह `sessions.patch` (`category`) के माध्यम से प्रति सत्र बने रहते हैं, इसलिए संदेश चैनलों (Discord, Telegram, WhatsApp, ...) से शुरू किए गए सत्रों को भी वर्गीकृत किया जा सकता है; पंक्तियों को किसी अनुभाग पर खींचकर या प्रति-पंक्ति समूह चयनकर्ता से समूह निर्दिष्ट करें और New group क्रिया से समूह बनाएँ।
    - मेमोरी (Agents पेज पर एक टैब, चयनित एजेंट तक सीमित): Dreaming स्थिति, सक्षम/अक्षम टॉगल और Dream Diary रीडर (`doctor.memory.status`, `doctor.memory.dreamDiary`, `config.patch`)।
    - मेमोरी आयात करें (`/memory-import`, Agents पेज के Memory टैब से पहुँचा जाता है): स्थानीय Claude Code ऑटो-मेमोरी, Codex समेकित मेमोरी या Hermes मेमोरी फ़ाइलों का पूर्वावलोकन करें और उन्हें चयनित एजेंट वर्कस्पेस में कॉपी करें (`migrations.memory.plan`, `migrations.memory.apply`)।
    - ऑनबोर्डिंग मेमोरी प्रस्ताव: जब Control UI ऑनबोर्डिंग मोड में खुलता है (`?onboarding=1`, जिसे Linux सहयोगी ऐप पहली बार इंस्टॉल होने के बाद उपयोग करता है), तो एक-पेज का डायलॉग उसी योजना/लागू प्रवाह के साथ पहचानी गई मेमोरी आयात करने की पेशकश करता है; इसे छोड़ने पर बाद में प्रवेश बिंदु के रूप में सेटिंग्स पेज उपलब्ध रहता है।

  <Accordion title="Cron, कार्य, plugins, Skills, डिवाइस, exec अनुमोदन">
    - ऑटोमेशन (cron जॉब): Automations/Run history टैब स्विच के ऊपर आँकड़ा कार्ड (ऑटोमेशन संख्या, विफल संख्या, शेड्यूलर स्थिति, अगला वेक); Automations टैब फ़िल्टर करने योग्य तालिका में जॉब सूचीबद्ध करता है (All/Active/Paused, खोज, शेड्यूल और अंतिम-रन फ़िल्टर, प्रति-पंक्ति क्रिया मेनू), जिसके नीचे आरंभिक सुझाव होते हैं, और Run history टैब सभी ऑटोमेशन के हाल के रन दिखाता है (`cron.*`)।
    - कार्य: लिंक किए गए सत्रों और रद्द करने की सुविधा के साथ सक्रिय तथा हाल के बैकग्राउंड कार्यों का लाइव लेजर (`tasks.*`)। Chat की Background tasks रेल चल रहे और पूर्ण हुए कार्यों को समूहित करती है; किसी पंक्ति का सीमित प्रॉम्प्ट और आउटपुट या त्रुटि सारांश देखने के लिए उसे चुनें।
    - Plugins: इंस्टॉल की गई इन्वेंटरी और क्यूरेट किए गए स्टोर को ब्राउज़ करें, ClawHub में खोजें, plugin कोड इंस्टॉल करें और हटाएँ तथा इंस्टॉल किए गए plugins को सक्षम या अक्षम करें (`plugins.*`); MCP सर्वर पंक्तियाँ कॉन्फ़िगरेशन विधियों के माध्यम से `mcp.servers` संपादित करती हैं।
    - Skills: स्थिति, सक्षम/अक्षम, इंस्टॉल करना, API कुंजी अपडेट (`skills.*`)।
    - डिवाइस: एक इन्वेंटरी युग्मित डिवाइस रिकॉर्ड, Node कैटलॉग और लाइव उपस्थिति को जोड़ती है (`device.pair.list`, `node.list`, `system-presence`)। Gateway होस्ट सबसे पहले पिन होता है; युग्मित क्लाइंट कनेक्शन स्थिति, भूमिकाएँ, टोकन, क्षमताएँ और कमांड दिखाते हैं। डुप्लिकेट पेयरिंग एक विस्तृत किए जा सकने वाले समूह में संक्षिप्त हो जाती हैं और **Clean up N stale** एडमिन द्वारा पुष्टि किए गए उन ऑफ़लाइन डुप्लिकेट को एक साथ हटाता है, जिन्हें स्वतः अनुमोदित किया गया था (साइलेंट लोकल, विश्वसनीय-CIDR या SSH-सत्यापित) या जो अनुमोदन उद्गम से पहले के हैं। प्रविष्टियाँ हटाई जा सकती हैं (`node.pair.remove`, `device.pair.remove`), डिवाइस पेयरिंग और Node के पुनः-अनुमोदन इनलाइन सँभाले जाते हैं (`device.pair.*`, `node.pair.approve`/`reject`), और मोबाइल सेटअप कोड उसी कार्ड से बनाए जाते हैं।
    - Exec अनुमोदन: `exec host=gateway/node` के लिए Gateway या Node की अनुमतिसूचियाँ और पूछने की नीति संपादित करें (`exec.approvals.*`)।

  <Accordion title="कॉन्फ़िगरेशन">
    - `~/.openclaw/openclaw.json` देखें/संपादित करें (`config.get`, `config.set`)।
    - सेटिंग्स नेविगेशन Ask OpenClaw से शुरू होता है, फिर पृष्ठों को ध्यान की आवश्यकता के अनुसार समूहित करता है: सबसे ऊपर सामान्य, स्वरूप और सूचनाएँ; कनेक्शन (कनेक्शन, चैनल, संचार, डिवाइस); एजेंट और टूल (एजेंट, AI और एजेंट, मॉडल प्रदाता, MCP, स्वचालन, लैब्स); गोपनीयता और सुरक्षा (सुरक्षा, अनुमोदन); और सिस्टम (इन्फ़्रास्ट्रक्चर, उन्नत, डीबग, लॉग, परिचय)। सामान्य एक संक्षिप्त हब है जिसमें मॉडल डिफ़ॉल्ट, भाषा और Gateway होस्ट आँकड़े होते हैं; प्रत्येक अन्य सेटिंग ठीक एक पृष्ठ पर होती है।
    - गोपनीयता और सुरक्षा: स्कीमा-समर्थित `security`/`approvals` अनुभागों के ऊपर Gateway प्रमाणीकरण, निष्पादन नीति, ब्राउज़र सक्षमता, टूल प्रोफ़ाइल, डिवाइस प्रमाणीकरण और मोबाइल पेयरिंग के लिए व्यवस्थित पंक्तियाँ।

    - लैब्स जारी किए गए प्रयोगात्मक स्विच दिखाता है। Code Mode वर्तमान प्रविष्टि है और `tools.codeMode.enabled` को तुरंत सहेजता है; जारी न किए गए प्रयोग दिखाई नहीं देते या अनुमानाधारित कॉन्फ़िगरेशन कुंजियाँ नहीं लिखते।
    - सूचनाएँ: ब्राउज़र वेब-पुश स्थिति, सदस्यता लेना/हटाना और एक परीक्षण प्रेषण।

    - प्रोफ़ाइल: सर्वकालिक उपयोग आँकड़ों—कुल जीवनकाल टोकन, सर्वाधिक उपयोग वाला दिन, सबसे लंबा सत्र, गतिविधि क्रम, वर्ष-भर का टोकन हीटमैप, शीर्ष टूल और चैनल की मुख्य बातें (`usage.cost`, `sessions.usage`)—के साथ डिफ़ॉल्ट एजेंट की पहचान दिखाने वाला सेटिंग्स पृष्ठ।
    - MCP का एक समर्पित सेटिंग्स पृष्ठ है जिसमें सर्वर पंक्तियाँ (ट्रांसपोर्ट, सक्षमता, OAuth/फ़िल्टर/समानांतरता सारांश), सीधे जोड़ने/सक्षम करने/अक्षम करने/हटाने के नियंत्रण, सामान्य ऑपरेटर कमांड और सीमित-दायरे वाला `mcp` कॉन्फ़िगरेशन संपादक है। एक-क्लिक कनेक्टर और खोज के लिए Plugins पृष्ठ मुख्य स्थान बना रहता है।
    - मॉडल प्रदाता: प्रत्येक कॉन्फ़िगर किए गए मॉडल प्रदाता को उसके ब्रांड आइकन, प्रमाणीकरण स्थिति (`models.authStatus`), मॉडल उपलब्धता (`models.list`), जहाँ प्रदाता रिपोर्ट करता है वहाँ लाइव प्लान/कोटा/बिलिंग डेटा (`usage.status`), और पिछले 30 दिनों के स्थानीय सत्र व्यय (`sessions.usage`) के साथ सूचीबद्ध करने वाला सेटिंग्स पृष्ठ। रीफ़्रेश क्रिया क्रेडेंशियल स्थिति और प्रदाता उपयोग को दोबारा पढ़ती है।
    - कनेक्शन: **कनेक्शन** के अंतर्गत एक सेटिंग्स पृष्ठ, जो डैशबोर्ड के अपने Gateway लिंक—WebSocket URL, Gateway टोकन, पासवर्ड और डिफ़ॉल्ट सत्र कुंजी—के साथ नवीनतम हैंडशेक स्नैपशॉट (स्थिति, अपटाइम, टिक अंतराल, चैनलों का अंतिम रीफ़्रेश) को नियंत्रित करता है। ऑफ़लाइन लॉगिन गेट डिस्कनेक्टेड स्थिति संभालता है; यह पृष्ठ कनेक्ट रहते हुए कनेक्शन संपादित करता है।
    - सत्यापन के साथ लागू करें और पुनः आरंभ करें (`config.apply`), फिर अंतिम सक्रिय सत्र को जगाएँ।
    - समवर्ती संपादनों को अधिलेखित होने से रोकने के लिए लेखन में बेस-हैश गार्ड शामिल होता है।
    - लेखन (`config.set`/`config.apply`/`config.patch`) प्रस्तुत कॉन्फ़िगरेशन पेलोड में रेफ़रेंस के लिए सक्रिय SecretRef समाधान की पूर्व-जाँच करते हैं; प्रस्तुत किए गए ऐसे सक्रिय रेफ़रेंस जिनका समाधान नहीं हुआ है, लेखन से पहले अस्वीकार कर दिए जाते हैं।
    - फ़ॉर्म सहेजने पर पुराने संशोधित प्लेसहोल्डर हटा दिए जाते हैं जिन्हें सहेजे गए कॉन्फ़िगरेशन से पुनर्स्थापित नहीं किया जा सकता, जबकि ऐसे संशोधित मान संरक्षित रहते हैं जो अब भी सहेजे गए सीक्रेट से मैप होते हैं।
    - स्कीमा और फ़ॉर्म रेंडरिंग `config.schema` / `config.schema.lookup` से आती है, जिसमें फ़ील्ड `title`/`description`, मेल खाते UI संकेत, तत्काल चाइल्ड सारांश, नेस्टेड ऑब्जेक्ट/वाइल्डकार्ड/ऐरे/कंपोज़िशन नोड पर दस्तावेज़ मेटाडेटा, और उपलब्ध होने पर Plugin तथा चैनल स्कीमा शामिल हैं। अपरिष्कृत JSON संपादक केवल तभी उपलब्ध होता है जब स्नैपशॉट का अपरिष्कृत राउंड-ट्रिप सुरक्षित हो; अन्यथा Control UI फ़ॉर्म मोड लागू करता है।
    - अपरिष्कृत JSON संपादक में "सहेजे गए पर रीसेट करें" समतल किए गए स्नैपशॉट को दोबारा रेंडर करने के बजाय अपरिष्कृत रूप से लिखी आकृति (फ़ॉर्मेटिंग, टिप्पणियाँ, `$include` लेआउट) को संरक्षित करता है, ताकि जब स्नैपशॉट सुरक्षित रूप से राउंड-ट्रिप कर सके तब बाहरी संपादन रीसेट के बाद भी बने रहें।
    - संरचित SecretRef ऑब्जेक्ट मान फ़ॉर्म टेक्स्ट इनपुट में केवल-पढ़ने योग्य रूप में रेंडर होते हैं, ताकि ऑब्जेक्ट से स्ट्रिंग में आकस्मिक विकृति रोकी जा सके।

## सहायक मेमोरी आयात करें

स्थानीय Codex या Claude Code मेमोरी को OpenClaw एजेंट में लाने के लिए **सेटिंग्स** → **मेमोरी आयात करें** खोलें। Gateway अपने ही होस्ट पर समर्थित स्थानीय मेमोरी का पता लगाता है, इसलिए दूरस्थ Control UI ब्राउज़र कंप्यूटर के बजाय Gateway कंप्यूटर से आयात करता है।

1. गंतव्य एजेंट चुनें।
2. पता लगाए गए स्रोत संग्रहों और Markdown फ़ाइल नामों की समीक्षा करें। फ़ाइल की सामग्री
   योजना प्रतिक्रिया में नहीं भेजी जाती या पृष्ठ पर प्रदर्शित नहीं की जाती।
3. आयात करने के लिए संग्रह चुनें और पुष्टि करें। लागू करना लिखने से पहले योजना को फिर से बनाता है,
   ताकि पुराने चयन सुरक्षित रूप से विफल हों।
4. यदि फ़ाइलें पहले से मौजूद हैं, तो **मौजूदा आयात बदलें** सक्षम करें, पूर्वावलोकन
   रीफ़्रेश करें और प्रतिस्थापन की पुष्टि करें।

Codex केवल अपने समेकित `MEMORY.md` और `memory_summary.md` को आयात करता है। Claude
Code प्रोजेक्ट की स्वतः-मेमोरी निर्देशिकाओं और कॉन्फ़िगर किए गए
`autoMemoryDirectory` से Markdown आयात करता है; यह इस पृष्ठ के माध्यम से सत्र, सेटिंग्स, निर्देश, या
क्रेडेंशियल आयात नहीं करता। फ़ाइलें चयनित कार्यस्थान में `memory/imports/` के नीचे कॉपी की जाती हैं,
जहाँ सक्रिय मेमोरी Plugin उन्हें अनुक्रमित कर सकता है। स्रोतों को
कभी नहीं बदला जाता।

योजना बनाने और लागू करने के लिए `operator.admin` आवश्यक है। स्थिति मौजूद होने पर प्रत्येक लागू प्रक्रिया सत्यापित
OpenClaw बैकअप बनाती है, संशोधित माइग्रेशन रिपोर्ट लिखती है और
मौजूदा गंतव्य फ़ाइलों को बदलने से पहले आइटम-स्तरीय बैकअप रखती है। पथों और
स्मरण व्यवहार के लिए [मेमोरी अवलोकन](/hi/concepts/memory#import-from-coding-assistants) देखें।

## MCP पृष्ठ

समर्पित MCP पृष्ठ `mcp.servers` के अंतर्गत OpenClaw द्वारा प्रबंधित MCP सर्वरों के लिए ऑपरेटर दृश्य है। यह स्वयं MCP ट्रांसपोर्ट आरंभ नहीं करता; सहेजे गए कॉन्फ़िगरेशन का निरीक्षण और संपादन करने के लिए इसका उपयोग करें, फिर लाइव सर्वर प्रमाण की आवश्यकता होने पर `openclaw mcp doctor --probe` का उपयोग करें।

सामान्य कार्यप्रवाह:

1. साइडबार से **MCP** खोलें।
2. कुल, सक्षम, OAuth और फ़िल्टर किए गए सर्वर की संख्या के लिए सारांश कार्ड जाँचें।
3. ट्रांसपोर्ट, सक्षमता, प्रमाणीकरण, फ़िल्टर, टाइमआउट और कमांड संकेतों के लिए प्रत्येक सर्वर पंक्ति की समीक्षा करें।
4. सर्वरों को सीधे MCP पृष्ठ पर जोड़ें, सक्षम करें, अक्षम करें या हटाएँ। Streamable HTTP, SSE या stdio स्पष्ट रूप से चुनें; stdio कमांड लाइनें रिक्त स्थान वाले पथ जैसे उद्धृत तर्क स्वीकार करती हैं। एक-क्लिक कनेक्टर और खोज के लिए **Plugins** पृष्ठ का उपयोग करें।
5. पर्यावरण चर, कार्यशील निर्देशिकाएँ, शीर्षलेख, TLS/mTLS पथ, OAuth मेटाडेटा, टूल फ़िल्टर और Codex प्रोजेक्शन मेटाडेटा जैसे उन्नत सर्वर फ़ील्ड के लिए सीमित-दायरे वाले `mcp` कॉन्फ़िगरेशन अनुभाग को संपादित करें।
6. कॉन्फ़िगरेशन लिखने के लिए **सहेजें** या जब चल रहे Gateway को बदला हुआ कॉन्फ़िगरेशन लागू करना हो तब **सहेजें और प्रकाशित करें** का उपयोग करें।
7. स्थिर निदान, लाइव प्रमाण या कैश किए गए रनटाइम को हटाने के लिए टर्मिनल से `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` या `openclaw mcp reload` चलाएँ।

पृष्ठ रेंडरिंग से पहले क्रेडेंशियल वाले URL-सदृश मानों को संशोधित करता है और कमांड स्निपेट में सर्वर नामों को उद्धृत करता है, ताकि कॉपी किए गए कमांड रिक्त स्थान या शेल मेटाकैरेक्टर के साथ भी काम करें। पूर्ण CLI और कॉन्फ़िगरेशन संदर्भ: [MCP](/hi/cli/mcp)।

## गतिविधि टैब

गतिविधि टैब **सेटिंग्स › सिस्टम** में, लॉग और डीबग के पास होता है। यह लाइव टूल गतिविधि के लिए अस्थायी ब्राउज़र-स्थानीय पर्यवेक्षक है, जो उसी Gateway `session.tool` / टूल इवेंट स्ट्रीम से प्राप्त होता है जो चैट टूल कार्ड को संचालित करती है। यह कोई अन्य Gateway इवेंट परिवार, एंडपॉइंट, स्थायी गतिविधि स्टोर, मेट्रिक्स फ़ीड या बाहरी पर्यवेक्षक स्ट्रीम नहीं जोड़ता।

गतिविधि प्रविष्टियाँ केवल स्वच्छ किए गए सारांश और संशोधित, छोटे किए गए आउटपुट पूर्वावलोकन रखती हैं। टूल तर्क मान गतिविधि स्थिति में संग्रहीत नहीं होते; UI दिखाता है कि तर्क छिपे हुए हैं और केवल तर्क फ़ील्ड की संख्या दर्ज करता है। इन-मेमोरी सूची वर्तमान ब्राउज़र टैब के साथ चलती है, Control UI के भीतर नेविगेशन के दौरान बनी रहती है और पृष्ठ पुनः लोड होने, सत्र बदलने या **साफ़ करें** पर रीसेट हो जाती है।

## ऑपरेटर टर्मिनल

डॉक किया जा सकने वाला ऑपरेटर टर्मिनल डिफ़ॉल्ट रूप से अक्षम है। इसे सक्षम करने के लिए `gateway.terminal.enabled: true` सेट करें और Gateway पुनः आरंभ करें। टर्मिनल को `operator.admin` कनेक्शन की आवश्यकता होती है और सक्रिय एजेंट कार्यस्थान में होस्ट PTY खोलता है। नए टैब वर्तमान में चयनित चैट एजेंट का अनुसरण करते हैं।

<Warning>
टर्मिनल एक अप्रतिबंधित होस्ट शेल है और Gateway प्रक्रिया का परिवेश प्राप्त करता है। इसे केवल विश्वसनीय ऑपरेटर परिनियोजनों के लिए सक्षम करें। OpenClaw `sandbox.mode: "all"` वाले एजेंटों के लिए टर्मिनल सत्र अस्वीकार करता है; किसी सक्रिय एजेंट को उस मोड में बदलने पर उसके मौजूदा और प्रगति में चल रहे टर्मिनल सत्र बंद हो जाते हैं।
</Warning>

डॉक को टॉगल करने के लिए **Ctrl + backtick** का उपयोग करें। लेआउट नीचे और दाईं ओर डॉकिंग का समर्थन करता है, ब्राउज़र व्यूपोर्ट के साथ आकार बदलता है और कई शेल टैब बनाए रखता है। `gateway.terminal.enabled` और वैकल्पिक `gateway.terminal.shell` ओवरराइड के लिए [Gateway कॉन्फ़िगरेशन](/hi/gateway/configuration-reference#gateway) देखें।

स्वामी द्वारा अधिकृत, सैंडबॉक्स-रहित एजेंट लंबे या इंटरैक्टिव कार्य के लिए `terminal` टूल का उपयोग कर सकते हैं, जिसे ऑपरेटर को देखना चाहिए। प्रत्येक टूल कॉल एजेंट के अपने gateway PTY को खोल, पढ़, लिख, आकार बदल, बंद या सूचीबद्ध कर सकती है। नए सत्र डिफ़ॉल्ट रूप से सह-संलग्न Control UI टैब खोलते हैं, इसलिए एजेंट और ऑपरेटर आउटपुट साझा करते हैं और दोनों में से कोई भी टाइप कर सकता है या आकार बदल सकता है। एजेंट की पहुँच सटीक-सत्र तक सीमित होती है: कोई एजेंट ऑपरेटर द्वारा बनाए गए टर्मिनलों या किसी अन्य एजेंट सत्र द्वारा खोले गए टर्मिनलों को पढ़ या नियंत्रित नहीं कर सकता।

एक या अधिक फ़ाइलों को सक्रिय टर्मिनल पर खींचें, या फ़ाइलें चुनने के लिए पेपरक्लिप बटन का उपयोग करें। OpenClaw प्रत्येक फ़ाइल को PTY के स्वामी वाली मशीन पर स्टेज करता है और कर्सर पर शेल-कोट किए गए निरपेक्ष पथ चिपकाता है; यह कभी Enter नहीं दबाता या इनपुट निष्पादित नहीं करता। एक संक्षिप्त बैच संकेतक वर्तमान फ़ाइल और पूर्ण हो चुकी फ़ाइलों की संख्या दिखाता है। रद्द करने पर पथ चिपकाए बिना शेष बैच रुक जाता है; विफल स्थानांतरण दृश्यमान रहता है, ताकि पूर्ण हो चुकी फ़ाइलों को फिर से अपलोड किए बिना उसी फ़ाइल से पुनः प्रयास किया जा सके। चित्र, PDF, आर्काइव और अन्य फ़ाइल प्रकार प्रति फ़ाइल 16 MiB तक स्वीकार किए जाते हैं। स्टेज की गई फ़ाइलें POSIX होस्ट पर एक निजी सिस्टम-अस्थायी निर्देशिका (निर्देशिका मोड `0700`, फ़ाइल मोड `0600`) या Windows पर उपयोगकर्ता-प्रोफ़ाइल ACL सीमा के अंतर्गत एक निर्देशिका और साथ में 24-घंटे का क्लीनअप टाइमर उपयोग करती हैं, इसलिए जिसे बनाए रखना आवश्यक हो उसे स्थानांतरित या कॉपी कर लें।

पथ प्रविष्टि PowerShell, `cmd.exe` और पहचाने गए POSIX शेल (`sh`, Bash, Dash, Ash, Ksh, Zsh और Fish) का समर्थन करती है, जिसमें Windows पर Git Bash भी शामिल है। अन्य शेल ओवरराइड अस्वीकार किए जाते हैं, क्योंकि उनके कोटिंग नियम सुरक्षित रूप से निर्धारित नहीं किए जा सकते; मूल WSL टर्मिनल और Linux अपलोड पथों के लिए Gateway को WSL के अंदर चलाएँ। `%` या `!` वाले `cmd.exe` पथ भी अस्वीकार किए जाते हैं, क्योंकि वह शेल दोहरे उद्धरण चिह्नों के अंदर भी उन वर्णों का विस्तार करता है।

सत्र साइडबार में मिले Codex और Claude Code सत्र उसी टर्मिनल पैनल के अंदर अपने मूल CLI में खोले जा सकते हैं। **Settings › Chat** में **Open Codex/Claude threads in** को **Terminal** पर सेट करें, ताकि सामान्य पंक्ति पर क्लिक करने से `codex resume` या `claude --resume` खुले; डिफ़ॉल्ट केवल-पढ़ने योग्य OpenClaw व्यूअर ही रहता है। किसी पंक्ति का राइट-क्लिक या कबाब मेनू हमेशा दोनों विकल्प देता है, और सत्र के पात्र होने पर व्यूअर हेडर में **Open in terminal** शामिल होता है।

पात्रता प्रत्येक सत्र और प्रत्येक होस्ट के अनुसार निर्धारित होती है। Gateway-स्थानीय सत्र Gateway होस्ट पर प्रदाता-स्वामित्व वाली रिज़्यूम कमांड शुरू करते हैं। युग्मित-नोड सत्र स्वामी नोड पर अनुमत-सूची में शामिल प्रदाता कमांड शुरू करते हैं और केवल उस PTY के आउटपुट, इनपुट और आकार-बदलाव इवेंट रिले करते हैं; इससे सामान्य नोड शेल उजागर नहीं होता और न ही ब्राउज़र द्वारा दी गई कमांड स्वीकार की जाती हैं। फ़ाइल अपलोड अलग, आकार-सीमित `terminal.upload` नोड कमांड का उपयोग करते हैं और पहले से खुले टर्मिनल सत्र से बँधे रहते हैं। यह कमांड पहली बार दिखाई देने पर नोड युग्मन अपग्रेड को स्वीकृति दें। जो नोड मिलती-जुलती टर्मिनल-रिज़्यूम कमांड का विज्ञापन नहीं करते, जिनमें डुप्लेक्स स्ट्रीमिंग के बिना एम्बेडेड वर्कर ब्रिज भी शामिल हैं, वे व्यूअर उपलब्ध रखते हैं और टर्मिनल खोलने को अनुपलब्ध दिखाते हैं; पुराने नोड अब भी टर्मिनल चला सकते हैं, लेकिन खींची गई फ़ाइलें प्राप्त नहीं कर सकते।

कनेक्शन-स्वामित्व वाले सत्र डिस्कनेक्ट के बाद भी बने रहते हैं: पेज रीलोड, लैपटॉप स्लीप या नेटवर्क में थोड़ी रुकावट सत्र को समाप्त करने के बजाय Gateway पर अलग कर देती है, और वही ब्राउज़र टैब दोबारा कनेक्ट होने पर हाल का आउटपुट पुनः चलाते हुए फिर से जुड़ जाता है। अलग किए गए कनेक्शन-स्वामित्व वाले सत्र `gateway.terminal.detachedSessionTimeoutSeconds` के बाद समाप्त कर दिए जाते हैं (डिफ़ॉल्ट 300 सेकंड; `0` डिस्कनेक्ट होने पर समाप्त करने का व्यवहार पुनर्स्थापित करता है)। इनमें से किसी सत्र को संलग्न करना tmux-शैली का नियंत्रण-अधिग्रहण बना रहता है।

एजेंट-स्वामित्व वाले सत्र किसी ब्राउज़र कनेक्शन से बँधे नहीं होते। `terminal.attach` स्वामित्व लिए बिना प्रत्येक ब्राउज़र को व्यूअर के रूप में जोड़ता है, और व्यूअर टैब बंद करने पर केवल वही ब्राउज़र अलग होता है। PTY तब तक बना रहता है, जब तक स्वामी एजेंट उसे बंद नहीं करता, उसकी प्रक्रिया समाप्त नहीं होती, नीति उसे अक्षम नहीं करती या Gateway बंद नहीं होता। `terminal.list` प्रत्येक प्रविष्टि को कनेक्शन- या एजेंट-स्वामित्व वाली चिह्नित करता है, और `terminal.text` किसी एडमिन कनेक्शन को संलग्न हुए बिना हाल का सादा-पाठ आउटपुट पढ़ने देता है।

टर्मिनल `/?view=terminal` पर पूर्ण-स्क्रीन, केवल-टर्मिनल दस्तावेज़ के रूप में भी उपलब्ध है। iOS और Android ऐप अपने Terminal स्क्रीन में इस पेज को एम्बेड करते हैं और संग्रहीत gateway क्रेडेंशियल का पुनः उपयोग करते हैं; उपलब्धता समान `gateway.terminal.enabled` और `operator.admin` गेट का पालन करती है, और कनेक्ट किया गया Gateway टर्मिनल प्रदान न करने पर पेज एक सूचना दिखाता है।

## ब्राउज़र पैनल

Control UI में एक डॉक करने योग्य ब्राउज़र पैनल शामिल है, जो किसी भी सामान्य वेब ब्राउज़र में Gateway-नियंत्रित ब्राउज़र (वही जिसे एजेंट [ब्राउज़र टूल](/hi/tools/browser-control) के माध्यम से संचालित करते हैं) रेंडर करता है—किसी मूल वेबव्यू की आवश्यकता नहीं होती। यह तब दिखाई देता है, जब कनेक्ट किया गया Gateway किसी `operator.admin` कनेक्शन को `browser.request` का विज्ञापन करता है; थ्रेड वर्कस्पेस रेल में ग्लोब बटन इसे टॉगल करता है। पैनल टैब, संपादन योग्य URL बार, पीछे/आगे/रीलोड और अपने ब्राउज़र में खोलने के विकल्प के साथ लाइव पेज स्नैपशॉट दिखाता है, दाईं या नीचे डॉक होता है और क्लिक, माउस-व्हील स्क्रॉलिंग तथा मूल टाइपिंग को रिमोट पेज पर अग्रेषित करता है।

दो कैप्चर मोड एजेंट के लिए पेज संदर्भ पैकेज करते हैं:

- **एनोटेट (पेंसिल)**: पेज के ऊपर मुक्तहस्त मार्कअप बनाएँ। **चैट में भेजें** स्ट्रोक को स्क्रीनशॉट में संयोजित करता है, चित्र को सक्रिय चैट कंपोज़र से संलग्न करता है और पेज के URL, शीर्षक तथा प्रत्येक चिह्नित क्षेत्र का वर्णन करने वाला प्रॉम्प्ट पहले से भरता है, ताकि एजेंट को ठीक-ठीक पता हो कि आपने किस पर घेरा बनाया है।
- **निरीक्षण (पॉइंटर)**: कर्सर के नीचे मौजूद एलिमेंट (सिलेक्टर, सुलभ नाम, भूमिका, आकार) देखने के लिए उस पर होवर करें; उसी कंपोज़र प्रवाह के माध्यम से उस एलिमेंट का विवरण और हाइलाइट किया गया स्क्रीनशॉट भेजने के लिए क्लिक करें। निरीक्षण, माउस-व्हील स्क्रॉलिंग और पीछे/आगे जाने के लिए `browser.evaluateEnabled` आवश्यक है (डिफ़ॉल्ट रूप से चालू)।

macOS ऐप डैशबोर्ड में क्लिक किए गए लिंक के लिए अपना मूल लिंक-ब्राउज़र साइडबार बनाए रखता है; ब्राउज़र पैनल वहाँ भी काम करता है और प्रत्येक अन्य प्लेटफ़ॉर्म पर पेजों को एनोटेट करने का तरीका है।

    - उपयोगकर्ता-संदेश बबल में ट्रांसक्रिप्ट क्रियाएँ होती हैं: होवर करने पर रिवाइंड बटन ("फिर न पूछें" विकल्प वाले पुष्टि पॉपओवर के साथ), तथा राइट-क्लिक पर **यहाँ तक रिवाइंड करें** और **यहाँ से फ़ोर्क करें**। रिवाइंड सत्र को उस संदेश से ठीक पहले की स्थिति पर पुनः इंगित करता है और उसका टेक्स्ट संपादन तथा पुनः भेजने के लिए कंपोज़र में लौटाता है (`sessions.rewind`, `operator.admin`); फ़ोर्क संदेश से पहले सक्रिय-पथ प्रीफ़िक्स से नया सत्र बनाता है, उसे खोलता है और उसके कंपोज़र में वही टेक्स्ट भरता है (`sessions.fork`, `operator.write`)। एजेंट के काम करते समय दोनों क्रियाएँ व्याख्यात्मक टूलटिप के साथ अक्षम रहती हैं, केवल सुरक्षित रखे गए उपयोगकर्ता संदेशों पर लागू होती हैं, और उन सत्रों के लिए अस्वीकार कर दी जाती हैं जिनकी बातचीत किसी बाहरी एजेंट हार्नेस के स्वामित्व में हो। रिवाइंड केवल चैट संदर्भ को पीछे ले जाता है — फ़ाइलें और टूल के अन्य दुष्प्रभाव पूर्ववत नहीं होते — और रिवाइंड से पहले का ट्रांसक्रिप्ट केवल-जोड़ने योग्य सत्र स्टोर में सुरक्षित रहता है। जब उस स्टोर में कई ट्रांसक्रिप्ट शाखाएँ होती हैं, तो चैट टाइटल बार प्रत्येक शाखा के नवीनतम संदेश, संदेश संख्या और हालियापन वाला शाखा मेन्यू दिखाता है; निष्क्रिय शाखा चुनने पर वर्तमान सत्र उस सुरक्षित पथ पर वापस चला जाता है (`sessions.branches.list`, `operator.read`; `sessions.branches.switch`, `operator.admin`)। एजेंट के काम करते समय शाखा बदलना भी अनुपलब्ध रहता है, और पहले से सक्रिय शाखा चुनना RPC सीमा पर टाइप की गई नो-ऑप त्रुटि है। उपयोगकर्ता बबल पर अलग छिपाने की क्रिया केवल वर्तमान ब्राउज़र में संदेश छिपाती है; संदेश ट्रांसक्रिप्ट में बना रहता है और एजेंट उसे फिर भी देखता है।
    - जब किसी सत्र का चेकआउट GitHub रिपॉज़िटरी की गैर-डिफ़ॉल्ट शाखा पर होता है, तो चैट दृश्य कंपोज़र के ऊपर पुल रिक्वेस्ट चिप पिन करता है: PR संख्या, रिपॉज़िटरी, शाखा, डिफ़ गणना, CI पिल और ड्राफ़्ट/मर्ज/बंद स्थिति; प्रत्येक चिप PR से लिंक होती है। पंक्ति अधिकतम दो चिप दिखाती है — पहले लाइव (खुले/ड्राफ़्ट) PR — और "और दिखाएँ" बटन संकुचित मर्ज/बंद इतिहास दिखाता है। CI पिल एक छोटा CI निगरानी पॉपओवर खोलता है, जिसमें सफल/विफल/चल रही/छोड़ी गई जाँचों की संख्या और PR के जाँच पृष्ठ का लिंक होता है। पहचान सर्वर-साइड पर `controlUi.sessionPullRequests` के माध्यम से चलती है, जो सेट होने पर Gateway के `GH_TOKEN`/`GITHUB_TOKEN` का पुनः उपयोग करता है। GitHub API की दर सीमा पूरी होने पर, चिप अंतिम ज्ञात स्थिति बनाए रखती हैं और चेतावनी दिखाती हैं कि स्थिति पुरानी हो सकती है; किसी चिप को बंद करने पर वह वर्तमान ब्राउज़र प्रोफ़ाइल में उस सत्र के लिए छिप जाती है। कोई PR मौजूद होने से पहले, पंक्ति स्वयं शाखा दिखाती है — रिपॉज़िटरी, शाखा का नाम और डिफ़ॉल्ट-शाखा मर्ज बेस के विरुद्ध डिफ़ का +/− आकार (कमिट किया गया और बिना कमिट वाला कार्य)। पुश की गई शाखा पर तुलना के लिए कमिट उपलब्ध होने पर, पंक्ति Create PR बटन जोड़ती है जो GitHub का नया-पुल-रिक्वेस्ट पृष्ठ खोलता है; उससे पहले भी बदली हुई फ़ाइलों (कमिट की गई, बिना कमिट वाली या अनट्रैक की गई) वाले सत्र को बटन के बिना यह पंक्ति मिलती है। कोई खुला या ड्राफ़्ट PR मौजूद होने पर पंक्ति स्वयं को छिपा लेती है। शाखा पंक्ति केवल स्थानीय git से आती है, इसलिए GitHub की दर सीमा लागू होने पर भी उपलब्ध रहती है और वही पुरानी-स्थिति चेतावनी दिखाती है, क्योंकि सीमा रीसेट होने तक "कोई PR नहीं मिला" पर भरोसा नहीं किया जा सकता।
    - सत्र डिफ़ पैनल दिखाता है कि किसी सत्र के चेकआउट ने वास्तव में क्या बदला: वर्कस्पेस रेल या चैट टाइटल बार का शाखा बटन विवरण पैनल खोलता है, जिसमें चेकआउट के डिफ़ॉल्ट-शाखा मर्ज बेस के विरुद्ध शाखा, बिना कमिट वाले और अनट्रैक किए गए कार्य का प्रति-फ़ाइल डिफ़ होता है — स्थिति बिंदु, नाम-बदलाव तीर, प्रति-फ़ाइल +/− गणना, संकुचित की जा सकने वाली फ़ाइलें और हंक के बीच "N अपरिवर्तित पंक्तियाँ" चिह्न। डिफ़ की गणना सर्वर-साइड पर `sessions.diff` Gateway विधि (`operator.read` स्कोप) के माध्यम से होती है; बाइनरी और अत्यधिक बड़ी फ़ाइलें केवल आँकड़ों वाली प्रविष्टियों में सीमित हो जाती हैं, और बटन केवल तब दिखाई देता है जब कनेक्ट किया गया Gateway `sessions.diff` घोषित करता है।
    - प्रत्येक चैट पेन में टाइटल बार होता है। सत्र का नाम बदलने के लिए उसके शीर्षक पर क्लिक करें; वर्कस्पेस चिप चेकआउट पथ या शाखा कॉपी करती है और होस्ट फ़ाइल प्रबंधक में स्थानीय Gateway वर्कस्पेस दिखा सकती है। रिमोट और exec-node सत्र कॉपी क्रियाएँ बनाए रखते हैं, लेकिन दिखाने की क्रिया छिपाते हैं।
    - प्रत्येक चैट पेन की थ्रेड वर्कस्पेस रेल थ्रेड फ़ाइलें, प्रोजेक्ट फ़ाइलें और आर्टिफ़ैक्ट सूचीबद्ध करती है। डिफ़ॉल्ट रूप से यह पेन के दाएँ किनारे से डॉक रहती है; इसे नीचे ले जाने के लिए इसका हेडर खींचें (या डॉक बटन का उपयोग करें), और यह विकल्प वर्तमान ब्राउज़र प्रोफ़ाइल में संग्रहीत होता है। संकुचित रेल कोई स्थान नहीं लेती: इसे ⇧⌘B या टाइटल बार के फ़ाइल टॉगल से दोबारा खोलें, जिस पर बदली हुई फ़ाइलों की संख्या का बैज होता है। अलग फ़ाइल, टूल और कैनवास विवरण पैनल पर इसका प्रभाव नहीं पड़ता।
    - चैट में किसी फ़ाइल संदर्भ, विस्तृत पढ़ने/संपादित करने/लिखने वाले टूल कार्ड में फ़ाइल पथ या वर्कस्पेस रेल में फ़ाइल पंक्ति पर क्लिक करने से फ़ाइल विवरण पैनल खुलता है: CodeMirror-आधारित कोड दृश्य, जिसमें सिंटैक्स हाइलाइटिंग, पंक्ति संख्याएँ, पंक्ति पर जाएँ, फ़ाइल में खोज, कॉपी क्रियाएँ और बाहरी एडिटर में खोलने का मेन्यू होता है। जब Gateway किसी `operator.admin` कनेक्शन को `sessions.files.set` घोषित करता है, तो पैनल में डर्टी ट्रैकिंग और Cmd/Ctrl-S से सहेजने वाला संपादन मोड जुड़ जाता है; बिना सहेजे ड्राफ़्ट वर्तमान ब्राउज़र टैब में फ़ाइल, पैनल और सत्र नेविगेशन के दौरान तब तक बने रहते हैं, जब तक उन्हें स्पष्ट रूप से सहेजा या त्यागा न जाए। सहेजने की क्रिया `sessions.files.get` द्वारा लौटाए गए सामग्री हैश पर तुलना-और-अदला-बदली के रूप में होती है: यदि फ़ाइल लोड होने के बाद डिस्क पर बदल गई हो (उदाहरण के लिए, क्योंकि एजेंट काम करता रहा), तो पैनल पुनः लोड करें (नवीनतम सामग्री लें) और अधिलेखित करें (स्थानीय संपादन बनाए रखें) क्रियाओं के साथ विरोध सूचना दिखाता है। लेखन उन्हीं fs-सुरक्षित वर्कस्पेस गार्ड से होकर जाता है जिनसे पठन होता है — पथ सीमाबद्धता, सिमलिंक/हार्डलिंक अस्वीकृति और 256 KB UTF-8 सीमा — और केवल मौजूदा फ़ाइलों को अधिलेखित करता है; एडिटर कभी उन्हें बनाता या हटाता नहीं है।
    - प्रत्येक चैट पेन की पृष्ठभूमि कार्य रेल वर्तमान एजेंट के पृष्ठभूमि कार्य और उप-एजेंट सूचीबद्ध करती है (एजेंट के अनुसार स्कोप किया गया `tasks.list`, जिसे `task` इवेंट लाइव रखते हैं): चल रहे कार्य में लाइव बीता हुआ टाइमर, टूल-उपयोग संख्या, वर्तमान में उपयोग हो रहा टूल और रोकने का नियंत्रण दिखता है; संकुचित किया जा सकने वाला पूर्ण अनुभाग रन अवधि जोड़ता है; और ट्रांसक्रिप्ट देखें लिंक कार्य का चाइल्ड सत्र पेन में खोलता है। इसे टाइटल-बार गतिविधि टॉगल से खोलें; कार्य स्नैपशॉट पहले ही उत्सुकता से लोड हो जाता है, इसलिए रेल खोले बिना भी उस पर चल रहे कार्यों की संख्या का बैज दिखाई देता है। कार्य पृष्ठ सभी एजेंटों का संपूर्ण लेखा बना रहता है।
    - वर्कस्पेस रेल, बैकग्राउंड टास्क रेल और विवरण पैनल विंडो के बजाय प्रत्येक पेन की अपनी चौड़ाई के अनुसार अनुकूलित होते हैं: संकरे पेन या कॉम्पैक्ट विंडो में दोनों रेल नीचे की पट्टियों के रूप में दिखाई देती हैं (पेन के चौड़ा होने तक साइड-डॉक नियंत्रण छिपे रहते हैं; जब केवल एक कॉलम समा सकता है, तो साइड स्लॉट पर पहला अधिकार वर्कस्पेस रेल का रहता है), और विवरण पैनल थ्रेड के साथ एक ही पंक्ति साझा करने के बजाय क्षैतिज आकार-परिवर्तन हैंडल के साथ उसके नीचे स्टैक हो जाता है। फ़ोन के आकार के व्यूपोर्ट में विवरण पैनल अब भी पूर्ण-स्क्रीन खुलता है।
    - चैट हेडर के मॉडल और थिंकिंग पिकर `sessions.patch` के माध्यम से सक्रिय सत्र को तुरंत पैच करते हैं; वे स्थायी सत्र ओवरराइड हैं, केवल एक टर्न के लिए भेजने के विकल्प नहीं।
    - **स्प्लिट व्यू:** इसे चैट शीर्षक बार से खोलें (थ्रेड डिफ़, बैकग्राउंड टास्क और थ्रेड फ़ाइल टॉगल के पास), फिर जितने पेन समा सकें, उनके लिए सक्रिय पेन को दाईं ओर या नीचे विभाजित करें। प्रत्येक पेन का अपना थ्रेड, ट्रांसक्रिप्ट, कंपोज़र और टूल स्ट्रीम होता है।
    - `screen` टूल वाले एजेंट, किसी सक्षम Control UI के कनेक्ट रहने के दौरान, पेन, साइडबार, टर्मिनल, ब्राउज़र, फ़ोकस और नेविगेशन में वही बदलाव करने का अनुरोध कर सकते हैं। प्रोटोकॉल v1 कमांड को कनेक्ट किए गए प्रत्येक सक्षम Control UI पर लागू करता है; [स्क्रीन](/hi/tools/screen) देखें।
    - किसी सत्र को पेन में खोलने के लिए उसे साइडबार से खींचकर चैट में छोड़ें। एक एनिमेटेड ड्रॉप पूर्वावलोकन ज़ोन के बीच सहजता से सरकता है और परिणाम को लेबल करता है—नए पेन द्वारा घेरे जाने वाले सटीक आधे भाग पर "Split", पूरे पेन पर "Open here"—और ड्रॉप करना एकल-पेन मोड में भी काम करता है।
    - सक्रिय स्प्लिट पेन साइडबार चयन और URL को नियंत्रित करता है। उसका शीर्षक बार स्प्लिट और बंद करने के नियंत्रण जोड़ता है; डिवाइडर कॉलम और स्टैक किए गए पेन का आकार बदलते हैं, और ब्राउज़र रीलोड के बाद भी लेआउट को स्थानीय रूप से संग्रहीत रखता है।
    - संकरी स्क्रीन पर स्प्लिट व्यू लेआउट को बनाए रखता है, लेकिन केवल सक्रिय पेन को रेंडर करता है, जिसमें बंद करने के नियंत्रण वाला उसका हेडर भी शामिल है।
    - यदि आप उसी सत्र के लिए मॉडल पिकर में किया गया बदलाव सहेजे जाते समय संदेश भेजते हैं, तो कंपोज़र `chat.send` को कॉल करने से पहले उस सत्र पैच की प्रतीक्षा करता है, ताकि भेजे जाने वाले संदेश में चयनित मॉडल का उपयोग हो।
    - `/new` टाइप करने पर New Chat के समान एक नया डैशबोर्ड सत्र बनता है और उसी पर स्विच हो जाता है, सिवाय इसके कि जब `session.dmScope: "main"` कॉन्फ़िगर हो और वर्तमान पैरेंट एजेंट का मुख्य सत्र हो; तब यह मुख्य सत्र को उसी स्थान पर रीसेट करता है। `/reset` टाइप करने पर वर्तमान सत्र के लिए Gateway का स्पष्ट इन-प्लेस रीसेट बना रहता है।
    - चैट मॉडल पिकर Gateway के कॉन्फ़िगर किए गए मॉडल व्यू का अनुरोध करता है। यदि `agents.defaults.modelPolicy.allow` रिक्त नहीं है, तो वही नीति पिकर को संचालित करती है, जिसमें वे `provider/*` प्रविष्टियाँ भी शामिल हैं जो प्रदाता-स्कोप वाले कैटलॉग को डायनेमिक बनाए रखती हैं। अन्यथा पिकर कॉन्फ़िगर की गई प्रविष्टियों के साथ उन प्रदाताओं को दिखाता है जिनके पास उपयोग योग्य प्रमाणीकरण है; `agents.defaults.models` के अंतर्गत उपनाम और सेटिंग्स इसे प्रतिबंधित नहीं करते। पूरा कैटलॉग `view: "all"` के साथ डीबग `models.list` RPC के माध्यम से उपलब्ध रहता है।
    - जब Gateway की नई सत्र-उपयोग रिपोर्ट में वर्तमान संदर्भ टोकन शामिल होते हैं, तो चैट कंपोज़र टूलबार उपयोग किए गए प्रतिशत के साथ संदर्भ उपयोग का एक छोटा रिंग दिखाता है। वर्तमान संदर्भ विंडो, नवीनतम रन के टोकन की संख्या और अनुमानित कुल लागत, प्रदाता/मॉडल की पहचान तथा रिपोर्ट किए जाने पर नवीनतम प्रदाता प्रतिक्रिया की इनपुट/आउटपुट/कैश लागत का विवरण देखने के लिए रिंग खोलें। संदर्भ पर अधिक दबाव होने पर रिंग चेतावनी शैली में बदल जाता है और सुझाए गए Compaction स्तरों पर एक कॉम्पैक्ट बटन दिखाता है, जो सामान्य सत्र Compaction पथ चलाता है। Gateway द्वारा फिर से नया उपयोग रिपोर्ट किए जाने तक पुराने टोकन स्नैपशॉट छिपे रहते हैं।

    Talk मोड एक पंजीकृत रियलटाइम वॉइस प्रदाता का उपयोग करता है। OpenAI को `talk.realtime.provider: "openai"` के साथ किसी `openai` API-कुंजी प्रोफ़ाइल, `talk.realtime.providers.openai.apiKey`, या `OPENAI_API_KEY` से कॉन्फ़िगर करें। OpenAI Realtime सार्वजनिक Platform API का उपयोग करता है और इसके लिए Platform API कुंजी आवश्यक है; Codex OAuth लॉगिन इस सतह की आवश्यकता पूरी नहीं करता। Google को `talk.realtime.provider: "google"` के साथ `talk.realtime.providers.google.apiKey` से कॉन्फ़िगर करें। ब्राउज़र को कभी भी कोई मानक प्रदाता API कुंजी प्राप्त नहीं होती: OpenAI को WebRTC के लिए एक अल्पकालिक Realtime क्लाइंट सीक्रेट मिलता है, और Google Live को ब्राउज़र WebSocket सत्र के लिए एक बार उपयोग होने वाला सीमित Live API प्रमाणीकरण टोकन मिलता है, जिसमें निर्देश और टूल घोषणाएँ Gateway द्वारा टोकन में लॉक कर दी जाती हैं। जो प्रदाता केवल बैकएंड रियलटाइम ब्रिज उपलब्ध कराते हैं, वे Gateway रिले ट्रांसपोर्ट के माध्यम से चलते हैं, इसलिए क्रेडेंशियल और विक्रेता सॉकेट सर्वर-साइड रहते हैं, जबकि ब्राउज़र ऑडियो प्रमाणीकृत Gateway RPC के माध्यम से जाता है। Realtime सत्र प्रॉम्प्ट Gateway द्वारा संयोजित किया जाता है; `talk.client.create` कॉलर द्वारा दिए गए निर्देश ओवरराइड स्वीकार नहीं करता।

    स्थायी प्रदाता, मॉडल, वॉइस, ट्रांसपोर्ट, रीजनिंग प्रयास, सटीक VAD थ्रेशोल्ड, मौन अवधि और प्रीफ़िक्स पैडिंग के डिफ़ॉल्ट **Settings → Communications → Talk** में होते हैं; उन्हें बदलने के लिए `operator.admin` एक्सेस आवश्यक है। Gateway रिले कॉन्फ़िगर करने से बैकएंड रिले पथ अनिवार्य हो जाता है; WebRTC कॉन्फ़िगर करने पर सत्र क्लाइंट के स्वामित्व में रहता है और यदि प्रदाता ब्राउज़र सत्र नहीं बना सकता, तो चुपचाप रिले पर फ़ॉलबैक करने के बजाय विफल हो जाता है।

## कनेक्शन टूटना और पुनः कनेक्ट होना

सत्र स्थापित हो जाने के बाद, Gateway कनेक्शन टूटने से आपका लॉगआउट नहीं होता। क्लाइंट द्वारा बैकऑफ़ (800 ms से 15 s तक) के साथ अपने-आप पुनः प्रयास करते समय डैशबोर्ड दिखाई देता रहता है और शीर्ष बार के नीचे एक तैरता हुआ एम्बर "Gateway connection lost — Reconnecting…" पिल दिखाई देता है। कनेक्शन वापस आने तक लाइव अपडेट और रियलटाइम/सत्र कार्रवाइयाँ रुक जाती हैं; पिल में **Retry now** तत्काल प्रयास कराता है। चैट संपादन योग्य रहती है: सामान्य टेक्स्ट और अटैचमेंट प्रेषण वर्तमान टैब के Gateway/सत्र-स्कोप वाले ब्राउज़र स्टोरेज में रखे जाते हैं, पुनः कनेक्शन की प्रतीक्षा में दिखाए जाते हैं और Gateway वापस आते ही अपने-आप भेज दिए जाते हैं। ऑफ़लाइन रहते समय लाइव नियंत्रण और स्लैश कमांड उपलब्ध नहीं रहते।

जब इस ब्राउज़र में पहले से क्रेडेंशियल मौजूद हों (कॉन्फ़िगर किया गया टोकन/पासवर्ड या स्वीकृत डिवाइस टोकन), तो पहली बार खोलने और पुनः लोड करने पर लॉगिन गेट क्षणभर दिखने के बजाय कनेक्शन स्थापित होते समय एक छोटा एनिमेटेड OpenClaw चिह्न दिखाई देता है। लॉगिन गेट केवल तभी दिखाई देता है जब अभी कोई क्रेडेंशियल संग्रहीत न हो या Gateway उन्हें सक्रिय रूप से अस्वीकार कर दे (गलत टोकन/पासवर्ड, निरस्त पेयरिंग)—ये ऐसी स्थितियाँ हैं जिनमें प्रतीक्षा के बजाय आपके इनपुट की आवश्यकता होती है।

## PWA इंस्टॉलेशन और वेब पुश

Control UI में एक `manifest.webmanifest` और सर्विस वर्कर शामिल हैं, इसलिए आधुनिक ब्राउज़र इसे स्वतंत्र PWA के रूप में इंस्टॉल कर सकते हैं। Web Push की सहायता से Gateway, टैब या ब्राउज़र विंडो खुली न होने पर भी इंस्टॉल किए गए PWA को सूचनाओं के साथ सक्रिय कर सकता है।

macOS ऐप के भीतर Notifications सेटिंग पृष्ठ ब्राउज़र पुश के बजाय ऐप की मूल सूचना अनुमति दिखाता है, क्योंकि ऐप सूचनाएँ मूल रूप से वितरित करता है।

यदि OpenClaw अपडेट के ठीक बाद पृष्ठ **Protocol mismatch** दिखाता है, तो पहले डैशबोर्ड को `openclaw dashboard` के साथ फिर से खोलें और हार्ड-रिफ़्रेश करें। यदि यह फिर भी विफल हो, तो डैशबोर्ड ओरिजिन का साइट डेटा साफ़ करें या निजी ब्राउज़र विंडो में परीक्षण करें; कोई पुराना टैब या ब्राउज़र सर्विस-वर्कर कैश नए Gateway के विरुद्ध अपडेट-पूर्व Control UI बंडल चलाता रह सकता है।

| सतह                                               | यह क्या करता है                                                               |
| -------------------------------------------------- | ---------------------------------------------------------------------------- |
| `ui/public/manifest.webmanifest`                   | PWA मैनिफ़ेस्ट। इसके पहुँच योग्य होते ही ब्राउज़र "Install app" का विकल्प देते हैं।             |
| `ui/public/sw.js`                                  | `push` इवेंट और सूचना क्लिक संभालने वाला सर्विस वर्कर।           |
| `state/openclaw.sqlite` → `web_push_vapid_keys`    | Web Push पेलोड पर हस्ताक्षर करने के लिए उपयोग की जाने वाली स्वतः जनरेट हुई VAPID कुंजी-जोड़ी।                 |
| `state/openclaw.sqlite` → `web_push_subscriptions` | बनाए रखे गए ब्राउज़र सदस्यता एंडपॉइंट, कुंजियाँ और पंजीकरण टाइमस्टैम्प। |

सेवानिवृत्त `push/vapid-keys.json` और `push/web-push-subscriptions.json` स्टोर से अपग्रेड `openclaw doctor --fix` द्वारा आयात किए जाते हैं। वह सुधार चलाने से पहले Gateway रोकें, ताकि कोई पुरानी प्रक्रिया आयात के दौरान सेवानिवृत्त स्थिति को फिर से न बना सके। अपग्रेड के बाद Web Push का उपयोग करने से पहले सुधार चलाएँ; जब तक कोई सेवानिवृत्त स्रोत या बाधित Doctor दावा मौजूद है, तब तक पंजीकरण, वितरण, विलोपन और कुंजी समाधान आगे बढ़ने से इनकार करते हैं। Gateway रनटाइम केवल SQLite पढ़ता और लिखता है।

जब आप कुंजियों को पिन करना चाहते हों (मल्टी-होस्ट डिप्लॉयमेंट, सीक्रेट रोटेशन या परीक्षण), तब Gateway प्रक्रिया पर एनवायरनमेंट वेरिएबल के माध्यम से VAPID कुंजी-जोड़ी ओवरराइड करें:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (डिफ़ॉल्ट `https://openclaw.ai`)

Control UI ब्राउज़र सदस्यताओं को पंजीकृत और परीक्षण करने के लिए इन स्कोप-गेटेड Gateway विधियों का उपयोग करता है:

- `push.web.vapidPublicKey` सक्रिय VAPID सार्वजनिक कुंजी प्राप्त करता है।
- `push.web.subscribe` किसी `endpoint` को `keys.p256dh`/`keys.auth` के साथ पंजीकृत करता है।
- `push.web.unsubscribe` किसी पंजीकृत एंडपॉइंट को हटाता है।
- `push.web.test` कॉलर की सदस्यता को एक परीक्षण सूचना भेजता है।

<Note>
Web Push, iOS APNS रिले पथ (रिले-समर्थित पुश के लिए [कॉन्फ़िगरेशन](/hi/gateway/configuration) देखें) और मूल मोबाइल पेयरिंग को लक्षित करने वाली `push.test` विधि से स्वतंत्र है।
</Note>

## होस्ट किए गए एम्बेड

सहायक संदेश `[embed ...]` शॉर्टकोड के साथ होस्ट की गई वेब सामग्री इनलाइन रेंडर कर सकते हैं। iframe सैंडबॉक्स नीति `gateway.controlUi.embedSandbox` द्वारा नियंत्रित होती है:

मुख्य [`show_widget`](/hi/tools/show-widget) टूल किसी टूल कॉल से सीधे स्व-निहित SVG या HTML रेंडर करता है। ब्राउज़र और समर्थित मूल चैट क्लाइंट `inline-widgets` Gateway क्षमता घोषित करते हैं और चैट इतिहास पुनः लोड होने पर परिणामी Canvas दस्तावेज़ उपलब्ध रहता है। Discord Activities, Discord पर यही टूल नाम उपलब्ध कराती हैं; अन्य चैनलों से शुरू हुए रन को यह प्राप्त नहीं होता।

<Tabs>
  <Tab title="सख्त">
    होस्ट किए गए एम्बेड के भीतर स्क्रिप्ट निष्पादन अक्षम करता है।
  </Tab>
  <Tab title="स्क्रिप्ट (डिफ़ॉल्ट)">
    ओरिजिन पृथक्करण बनाए रखते हुए इंटरैक्टिव एम्बेड की अनुमति देता है; सामान्यतः स्व-निहित ब्राउज़र गेम/विजेट के लिए पर्याप्त है।
  </Tab>
  <Tab title="विश्वसनीय">
    समान-साइट दस्तावेज़ों के लिए, जिन्हें जानबूझकर अधिक शक्तिशाली विशेषाधिकारों की आवश्यकता होती है, `allow-scripts` के ऊपर `allow-same-origin` जोड़ता है।
  </Tab>
</Tabs>

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
`trusted` का उपयोग केवल तभी करें जब एम्बेड किए गए दस्तावेज़ को वास्तव में समान-ओरिजिन व्यवहार की आवश्यकता हो। एजेंट द्वारा जनरेट किए गए अधिकांश गेम और इंटरैक्टिव कैनवास के लिए `scripts` अधिक सुरक्षित विकल्प है।
</Warning>

निरपेक्ष बाहरी `http(s)` एम्बेड URL डिफ़ॉल्ट रूप से अवरुद्ध रहते हैं। `[embed url="https://..."]` को तृतीय-पक्ष पृष्ठ लोड करने देने के लिए `gateway.controlUi.allowExternalEmbedUrls: true` सेट करें।

## चैट संदेश की चौड़ाई

चैट ट्रांसक्रिप्ट कंपोज़र के साथ संरेखित, बीच में स्थित, पठनीय फ़्रेम का उपयोग करता है। उस फ़्रेम के भीतर सहायक और टूल आउटपुट बाईं ओर संरेखित रहते हैं, जबकि उपयोगकर्ता बबल दाईं ओर संरेखित रहते हैं। वाइड-मॉनिटर डिप्लॉयमेंट बंडल किए गए CSS में पैच किए बिना `gateway.controlUi.chatMessageMaxWidth` सेट करके ट्रांसक्रिप्ट की चौड़ाई ओवरराइड कर सकते हैं:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

मान को ब्राउज़र तक पहुँचने से पहले सत्यापित किया जाता है। समर्थित रूपों में `960px` या `82%` जैसी साधारण लंबाइयाँ और प्रतिशत, साथ ही सीमित `min(...)`, `max(...)`, `clamp(...)`, `calc(...)`, और `fit-content(...)` चौड़ाई एक्सप्रेशन शामिल हैं।

## Tailnet एक्सेस (अनुशंसित)

<Tabs>
  <Tab title="एकीकृत Tailscale Serve (वरीय)">
    Gateway को लूपबैक पर रखें और Tailscale Serve को HTTPS के साथ उसका प्रॉक्सी बनने दें:

    ```bash
    openclaw gateway --tailscale serve
    ```

    `https://<magicdns>/` (या अपना कॉन्फ़िगर किया हुआ `gateway.controlUi.basePath`) खोलें।

    डिफ़ॉल्ट रूप से, Control UI/WebSocket Serve अनुरोध Tailscale पहचान हेडर (`tailscale-user-login`) के माध्यम से प्रमाणित हो सकते हैं, जब `gateway.auth.allowTailscale`, `true` हो। OpenClaw, `x-forwarded-for` पते को `tailscale whois` से हल करके और उसे हेडर से मिलाकर पहचान सत्यापित करता है, और इन्हें केवल तभी स्वीकार करता है जब अनुरोध Tailscale के `x-forwarded-*` हेडर के साथ लूपबैक पर पहुँचता है। ब्राउज़र डिवाइस पहचान वाले Control UI ऑपरेटर सत्रों के लिए, यह सत्यापित Serve पथ डिवाइस-पेयरिंग की आने-जाने वाली प्रक्रिया को भी छोड़ देता है; डिवाइस-रहित ब्राउज़र और नोड-भूमिका कनेक्शन अब भी सामान्य डिवाइस जाँच का पालन करते हैं। यदि आप Serve ट्रैफ़िक के लिए भी स्पष्ट साझा-गुप्त क्रेडेंशियल अनिवार्य करना चाहते हैं, तो `gateway.auth.allowTailscale: false` सेट करें, फिर `gateway.auth.mode: "token"` या `"password"` का उपयोग करें।

    उस एसिंक्रोनस Serve पहचान पथ के लिए, समान क्लाइंट IP और प्रमाणीकरण दायरे के विफल प्रमाणीकरण प्रयासों को दर-सीमा लेखन से पहले क्रमबद्ध किया जाता है। इसलिए समान ब्राउज़र से समवर्ती गलत पुनः प्रयासों में समानांतर रूप से प्रतिस्पर्धा करते दो साधारण बेमेल के बजाय दूसरे अनुरोध पर `retry later` दिखाई दे सकता है।

    <Warning>
    टोकन-रहित Serve प्रमाणीकरण यह मानता है कि Gateway होस्ट विश्वसनीय है। यदि उस होस्ट पर अविश्वसनीय स्थानीय कोड चल सकता है, तो टोकन/पासवर्ड प्रमाणीकरण अनिवार्य करें।
    </Warning>

  </Tab>
  <Tab title="टेलनेट से बाइंड करें + टोकन">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    `http://<tailscale-ip>:18789/` (या अपना कॉन्फ़िगर किया हुआ `gateway.controlUi.basePath`) खोलें।

    मेल खाता साझा गुप्त UI सेटिंग्स में पेस्ट करें (`connect.params.auth.token` या `connect.params.auth.password` के रूप में भेजा जाता है)।

  </Tab>
</Tabs>

## असुरक्षित HTTP

यदि आप डैशबोर्ड को सामान्य HTTP (`http://<lan-ip>` या `http://<tailscale-ip>`) पर खोलते हैं, तो ब्राउज़र **असुरक्षित संदर्भ** में चलता है और WebCrypto को अवरुद्ध करता है। डिफ़ॉल्ट रूप से, OpenClaw डिवाइस पहचान के बिना Control UI कनेक्शन **अवरुद्ध करता है**।

दस्तावेज़ीकृत अपवाद:

- केवल-localhost असुरक्षित HTTP संगतता, `gateway.controlUi.allowInsecureAuth=true` के साथ
- `gateway.auth.mode: "trusted-proxy"` के माध्यम से सफल ऑपरेटर Control UI प्रमाणीकरण
- आपातकालीन `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**अनुशंसित समाधान:** HTTPS (Tailscale Serve) का उपयोग करें या UI को स्थानीय रूप से `https://<magicdns>/` (Serve) अथवा `http://127.0.0.1:18789/` (Gateway होस्ट पर) खोलें।

<AccordionGroup>
  <Accordion title="असुरक्षित-प्रमाणीकरण टॉगल का व्यवहार">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` केवल एक स्थानीय संगतता टॉगल है:

    - यह localhost Control UI सत्रों को असुरक्षित HTTP संदर्भों में डिवाइस पहचान के बिना आगे बढ़ने देता है।
    - यह पेयरिंग जाँच को बायपास नहीं करता।
    - यह दूरस्थ (गैर-localhost) डिवाइस पहचान आवश्यकताओं को शिथिल नहीं करता।

  </Accordion>
  <Accordion title="केवल आपातकालीन उपयोग">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` Control UI डिवाइस पहचान जाँच को अक्षम करता है और सुरक्षा को गंभीर रूप से कम करता है। आपातकालीन उपयोग के बाद इसे शीघ्र वापस करें।
    </Warning>

  </Accordion>
  <Accordion title="विश्वसनीय-प्रॉक्सी संबंधी टिप्पणी">
    - सफल विश्वसनीय-प्रॉक्सी प्रमाणीकरण डिवाइस पहचान के बिना **ऑपरेटर** Control UI सत्रों को प्रवेश दे सकता है।
    - यह नोड-भूमिका वाले Control UI सत्रों तक विस्तारित **नहीं** होता।
    - समान-होस्ट लूपबैक रिवर्स प्रॉक्सी फिर भी विश्वसनीय-प्रॉक्सी प्रमाणीकरण को पूरा नहीं करते; [विश्वसनीय प्रॉक्सी प्रमाणीकरण](/hi/gateway/trusted-proxy-auth) देखें।

  </Accordion>
</AccordionGroup>

## सामग्री सुरक्षा नीति

Control UI एक सख्त `img-src` नीति के साथ आता है: केवल **समान-ऑरिजिन** एसेट, `data:` URL और स्थानीय रूप से जनरेट किए गए `blob:` URL की अनुमति है। दूरस्थ `http(s)` और प्रोटोकॉल-सापेक्ष इमेज URL ब्राउज़र द्वारा अस्वीकार कर दिए जाते हैं और कभी नेटवर्क फ़ेच जारी नहीं करते।

व्यवहार में:

- सापेक्ष पथों के अंतर्गत दिए गए अवतार और इमेज (उदाहरण के लिए `/avatars/<id>`) अब भी रेंडर होते हैं, जिनमें वे प्रमाणित अवतार रूट भी शामिल हैं जिन्हें UI फ़ेच करके स्थानीय `blob:` URL में बदलता है।
- इनलाइन `data:image/...` URL अब भी रेंडर होते हैं।
- Control UI द्वारा बनाए गए स्थानीय `blob:` URL अब भी रेंडर होते हैं।
- GitHub लिंक पूर्वावलोकन अवतारों को Gateway, GitHub के निश्चित अवतार होस्ट से फ़ेच करता है और सीमित `data:` URL के रूप में लौटाता है; ऑपरेटर का ब्राउज़र कभी दूरस्थ अवतार होस्ट से संपर्क नहीं करता।
- चैनल मेटाडेटा से मिले दूरस्थ अवतार URL को Control UI के अवतार सहायक हटा देते हैं और उनकी जगह अंतर्निहित लोगो/बैज लगा देते हैं, ताकि कोई समझौता किया हुआ या दुर्भावनापूर्ण चैनल ऑपरेटर के ब्राउज़र को मनमाने दूरस्थ इमेज फ़ेच करने के लिए बाध्य न कर सके।

यह हमेशा चालू रहता है और कॉन्फ़िगर करने योग्य नहीं है।

## अवतार रूट प्रमाणीकरण

Gateway प्रमाणीकरण कॉन्फ़िगर होने पर, Control UI अवतार एंडपॉइंट को API के शेष भाग के समान Gateway टोकन की आवश्यकता होती है:

- `GET /avatar/<agentId>` केवल प्रमाणित कॉलर को अवतार इमेज लौटाता है। `GET /avatar/<agentId>?meta=1` उसी नियम के अंतर्गत अवतार मेटाडेटा लौटाता है।
- दोनों में से किसी भी रूट के अप्रमाणित अनुरोध अस्वीकार कर दिए जाते हैं (समान सहायक-मीडिया रूट के अनुरूप), इसलिए अवतार रूट उन होस्ट पर एजेंट पहचान लीक नहीं कर सकता जो अन्यथा सुरक्षित हैं।
- अवतार फ़ेच करते समय Control UI, Gateway टोकन को बेयरर हेडर के रूप में अग्रेषित करता है और प्रमाणित ब्लॉब URL का उपयोग करता है, ताकि इमेज डैशबोर्ड में फिर भी रेंडर हो।

यदि आप Gateway प्रमाणीकरण अक्षम करते हैं (साझा होस्ट पर अनुशंसित नहीं), तो Gateway के शेष भाग की तरह अवतार रूट भी अप्रमाणित हो जाता है।

## सहायक मीडिया रूट प्रमाणीकरण

Gateway प्रमाणीकरण कॉन्फ़िगर होने पर, सहायक के स्थानीय-मीडिया पूर्वावलोकन दो-चरणीय रूट का उपयोग करते हैं:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` को सामान्य Control UI ऑपरेटर प्रमाणीकरण की आवश्यकता होती है; उपलब्धता जाँचते समय ब्राउज़र Gateway टोकन को बेयरर हेडर के रूप में भेजता है।
- सफल मेटाडेटा प्रतिक्रियाओं में एक अल्पकालिक `mediaTicket` शामिल होता है, जिसका दायरा उसी सटीक स्रोत पथ तक सीमित होता है।
- ब्राउज़र में रेंडर किए जाने वाले इमेज, ऑडियो, वीडियो और दस्तावेज़ URL सक्रिय Gateway टोकन या पासवर्ड के बजाय `mediaTicket=<ticket>` का उपयोग करते हैं। टिकट शीघ्र समाप्त हो जाता है और किसी भिन्न स्रोत को अधिकृत नहीं कर सकता।

इससे पुनः उपयोग योग्य Gateway क्रेडेंशियल को दृश्यमान मीडिया URL में डाले बिना मीडिया रेंडरिंग ब्राउज़र के मूल मीडिया तत्वों के साथ संगत रहती है।

## अनुमोदन लिंक

ऑपरेटर अनुमोदन सूचनाएँ आरक्षित `${controlUiBasePath}/approve/{approvalId}` नेमस्पेस के अंतर्गत दिए गए एक स्वतंत्र अनुमोदन दस्तावेज़ से डीप-लिंक कर सकती हैं (उदाहरण के लिए `/approve/<approvalId>`, या कॉन्फ़िगर किए गए बेस पथ के साथ `/openclaw/approve/<approvalId>`)। URL अनुमोदन के जीवनकाल तक स्थिर रहता है और आपके अपने डिवाइसों के बीच अग्रेषित करना सुरक्षित है: यह अनुमोदन की पहचान करता है, उसे कभी अधिकृत नहीं करता।

- एक-खंड वाला `/approve/<approvalId>` नेमस्पेस Gateway द्वारा Plugin HTTP रूट से पहले **सभी** HTTP विधियों के लिए आरक्षित किया जाता है, इसलिए कोई Plugin रूट कभी अनुमोदन दस्तावेज़ को ओवरराइड या इंटरसेप्ट नहीं कर सकता।
- अनुमोदन दस्तावेज़ खोलने के लिए Control UI के शेष भाग के समान Gateway प्रमाणीकरण (टोकन/पासवर्ड, Tailscale Serve पहचान या विश्वसनीय-प्रॉक्सी पहचान) आवश्यक है; क्रेडेंशियल कभी अनुमोदन URL का हिस्सा नहीं होते।
- Control UI सेवा अक्षम होने पर, नेमस्पेस के अनुरोध Plugin हैंडलर तक पहुँचने के बजाय `404` लौटाते हैं।
- अनुमोदन दस्तावेज़ पर साइन इन करना केवल उस पृष्ठ के लिए अस्थायी होता है: यह उसी ब्राउज़र में पूर्ण Control UI द्वारा सहेजे गए Gateway चयन या सेटिंग्स को ओवरराइट नहीं करता।

Gateway, `dist/control-ui` से स्थिर फ़ाइलें प्रदान करता है:

```bash
pnpm ui:build
```

वैकल्पिक निरपेक्ष बेस (निश्चित एसेट URL):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

स्थानीय डेवलपमेंट (अलग डेवलपमेंट सर्वर):

```bash
pnpm ui:dev
```

फिर UI को अपने Gateway WS URL पर इंगित करें (उदा. `ws://127.0.0.1:18789`)।

## रिक्त Control UI पृष्ठ

यदि ब्राउज़र रिक्त डैशबोर्ड लोड करता है और DevTools कोई उपयोगी त्रुटि नहीं दिखाता, तो संभव है कि किसी एक्सटेंशन या शुरुआती सामग्री स्क्रिप्ट ने JavaScript मॉड्यूल ऐप का मूल्यांकन होने से रोक दिया हो। स्थिर पृष्ठ में एक साधारण HTML पुनर्प्राप्ति पैनल शामिल है, जो स्टार्टअप के बाद `<openclaw-app>` पंजीकृत न होने पर दिखाई देता है।

ब्राउज़र परिवेश बदलने के बाद पैनल की **Try again** क्रिया का उपयोग करें, या इन जाँचों के बाद मैन्युअल रूप से पुनः लोड करें:

- सभी पृष्ठों में इंजेक्ट होने वाले एक्सटेंशन अक्षम करें, विशेष रूप से `<all_urls>` सामग्री स्क्रिप्ट वाले एक्सटेंशन।
- निजी विंडो, साफ़ ब्राउज़र प्रोफ़ाइल या कोई अन्य ब्राउज़र आज़माएँ।
- Gateway को चालू रखें और ब्राउज़र बदलने के बाद उसी डैशबोर्ड URL को सत्यापित करें।

## डीबगिंग/परीक्षण: डेवलपमेंट सर्वर + दूरस्थ Gateway

Control UI स्थिर फ़ाइलों से बना है; WebSocket लक्ष्य कॉन्फ़िगर करने योग्य है और HTTP ऑरिजिन से भिन्न हो सकता है। यह तब उपयोगी है जब आप Vite डेवलपमेंट सर्वर स्थानीय रूप से चलाना चाहते हैं, लेकिन Gateway कहीं और चलता है।

    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` Host-header ऑरिजिन फ़ॉलबैक मोड सक्षम करता है, लेकिन यह एक खतरनाक सुरक्षा मोड है।
