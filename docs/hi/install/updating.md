OpenClaw को अद्यतित रखें।

Docker, Podman और Kubernetes इमेज प्रतिस्थापनों के लिए,
[कंटेनर इमेज अपग्रेड करना](/hi/install/docker#upgrading-container-images) देखें। Gateway तैयार होने से पहले
स्टार्टअप-सुरक्षित अपग्रेड कार्य चलाता है और यदि माउंट की गई
स्थिति को मैन्युअल मरम्मत की आवश्यकता हो तो बाहर निकल जाता है।

## अनुशंसित: `openclaw update`

आपके इंस्टॉल प्रकार (npm, pnpm, Bun या git) का पता लगाता है, नवीनतम संस्करण प्राप्त करता है, `openclaw doctor` चलाता है और Gateway को पुनः आरंभ करता है।

```bash
openclaw update
```

चैनल बदलें या किसी विशिष्ट संस्करण को लक्षित करें:

```bash
openclaw update --channel beta
openclaw update --channel extended-stable
openclaw update --channel dev
openclaw update --dry-run   # लागू किए बिना पूर्वावलोकन
```

`openclaw update` में `--verbose` फ़्लैग नहीं है (इंस्टॉलर में है)। निदान के लिए
नियोजित कार्रवाइयों का पूर्वावलोकन करने हेतु `--dry-run`, संरचित परिणामों के लिए `--json`, या
चैनल और उपलब्धता स्थिति की जाँच करने के लिए `openclaw update status --json` का उपयोग करें।

`--channel beta` बीटा npm dist-tag को प्राथमिकता देता है, लेकिन बीटा टैग अनुपलब्ध होने या
उसका संस्करण नवीनतम स्थिर रिलीज़ से पुराना होने पर stable/latest पर वापस जाता है।
इसके बजाय मूल npm बीटा dist-tag पर पिन किए गए एकबारगी पैकेज अपडेट के लिए
`--tag beta` का उपयोग करें।

`--channel extended-stable` केवल पैकेज के लिए है और इंस्टॉलेशन केवल
फ़ोरग्राउंड में ही रहता है। OpenClaw सार्वजनिक npm `extended-stable` चयनकर्ता को पढ़ता है,
चुने गए सटीक पैकेज का सत्यापन करता है और वही सटीक संस्करण इंस्टॉल करता है। अनुपलब्ध
या असंगत रजिस्ट्री डेटा पर प्रक्रिया सुरक्षित रूप से विफल होती है; यह कभी भी `latest` पर वापस नहीं जाती।
यदि चुना गया संस्करण इंस्टॉल किए गए संस्करण से पुराना है, तो सामान्य
डाउनग्रेड पुष्टि अब भी लागू होती है। सफल कोर अपडेट के बाद CLI चैनल को
स्थायी करता है; प्रत्यक्ष `npm install -g openclaw@extended-stable`
`update.channel` को अपडेट नहीं करता।
कोर बदलने के बाद, bare/default या
`latest` अभिप्राय वाले पात्र आधिकारिक npm plugins उसी सटीक कोर संस्करण पर अभिसरित होते हैं। सटीक पिन और स्पष्ट
गैर-`latest` टैग, तृतीय-पक्ष plugins और गैर-npm स्रोत अपरिवर्तित रहते हैं।
वर्तमान OpenClaw संस्करणों द्वारा बनाए गए कैटलॉग इंस्टॉल उस डिफ़ॉल्ट
अभिप्राय को बनाए रखते हैं। केवल सटीक संस्करण वाले पुराने रिकॉर्ड पिन ही रहते हैं क्योंकि
OpenClaw किसी पुराने स्वचालित पिन को उपयोगकर्ता पिन से सुरक्षित रूप से अलग नहीं कर सकता; उस plugin को
सटीक-कोर ट्रैकिंग में फिर से शामिल करने के लिए extended-stable चैनल पर
`openclaw plugins update @openclaw/name` एक बार चलाएँ।

`--channel dev` एक स्थायी, बदलता रहने वाला GitHub `main` चेकआउट देता है। एकबारगी
पैकेज अपडेट के लिए, `--tag main` को `github:openclaw/openclaw#main` पैकेज
विनिर्देश से मैप किया जाता है और लक्ष्य पैकेज मैनेजर (npm/pnpm/bun) के माध्यम से सीधे इंस्टॉल किया जाता है।

प्रबंधित plugins के लिए, बीटा रिलीज़ का अनुपलब्ध होना चेतावनी है, विफलता नहीं:
कोर अपडेट फिर भी सफल हो सकता है, जबकि कोई plugin अपनी दर्ज की गई
default/latest रिलीज़ पर वापस जा सकता है।

चैनल के अर्थ-विज्ञान के लिए [रिलीज़ चैनल](/hi/install/development-channels) देखें।

## npm और git इंस्टॉल के बीच स्विच करें

इंस्टॉल प्रकार बदलने के लिए चैनलों का उपयोग करें। अपडेटर आपकी स्थिति, कॉन्फ़िगरेशन,
क्रेडेंशियल और कार्यक्षेत्र को `~/.openclaw` में बनाए रखता है; यह केवल उस OpenClaw
कोड इंस्टॉल को बदलता है जिसका उपयोग CLI और Gateway करते हैं।

```bash
# npm पैकेज इंस्टॉल -> संपादन योग्य git चेकआउट
openclaw update --channel dev

# git चेकआउट -> npm पैकेज इंस्टॉल
openclaw update --channel stable
```

पहले इंस्टॉल-मोड परिवर्तन का पूर्वावलोकन करें:

```bash
openclaw update --channel dev --dry-run
openclaw update --channel stable --dry-run
```

`dev` git चेकआउट सुनिश्चित करता है, उसे बिल्ड करता है और उसी
चेकआउट से वैश्विक CLI इंस्टॉल करता है। `stable`, `extended-stable` और `beta` चैनल पैकेज
इंस्टॉल का उपयोग करते हैं। git चेकआउट पर extended-stable को उसमें कोई बदलाव या
रूपांतरण किए बिना अस्वीकार कर दिया जाता है। यदि Gateway पहले से इंस्टॉल है, तो `openclaw update`
सेवा मेटाडेटा को रीफ़्रेश करता है और उसे पुनः आरंभ करता है, जब तक कि आप `--no-restart` न दें।

प्रबंधित Gateway सेवा वाले पैकेज इंस्टॉल के लिए, `openclaw update` उस सेवा द्वारा
उपयोग किए जाने वाले पैकेज रूट को लक्षित करता है। यदि शेल का `openclaw` कमांड
किसी अलग इंस्टॉल से आता है, तो अपडेटर दोनों रूट और प्रबंधित
सेवा का Node पथ प्रिंट करता है तथा पैकेज बदलने से पहले उस Node संस्करण को लक्ष्य रिलीज़ की
`engines.node` आवश्यकता के विरुद्ध जाँचता है।

## विकल्प: इंस्टॉलर फिर से चलाएँ

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
```

ऑनबोर्डिंग छोड़ने के लिए `--no-onboard` जोड़ें। किसी विशिष्ट इंस्टॉल प्रकार को बाध्य करने के लिए,
`--install-method git --no-onboard` या `--install-method npm --no-onboard` दें।

यदि npm पैकेज इंस्टॉल चरण के बाद `openclaw update` विफल हो जाए, तो इसके बजाय
इंस्टॉलर फिर से चलाएँ। यह अपडेटर को कॉल नहीं करता; यह वैश्विक पैकेज
इंस्टॉल सीधे चलाता है और आंशिक रूप से अपडेट हुए npm इंस्टॉल को पुनर्प्राप्त कर सकता है।

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method npm
```

`--version` के साथ पुनर्प्राप्ति को किसी विशिष्ट संस्करण या dist-tag पर पिन करें:

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method npm --version <version-or-dist-tag>
```

## विकल्प: मैन्युअल npm, pnpm या bun

```bash
npm i -g openclaw@latest
```

पर्यवेक्षित इंस्टॉल के लिए `openclaw update` को प्राथमिकता दें: यह चल रही Gateway सेवा के साथ
पैकेज परिवर्तन का समन्वय कर सकता है। यदि आप किसी पर्यवेक्षित
इंस्टॉल पर मैन्युअल रूप से अपडेट करते हैं, तो पहले प्रबंधित Gateway को रोकें। पैकेज मैनेजर फ़ाइलों को
उसी स्थान पर बदलते हैं और अन्यथा चल रहा Gateway परिवर्तन के बीच में कोर या plugin फ़ाइलें
लोड करने का प्रयास कर सकता है। पैकेज मैनेजर का कार्य पूरा होने के बाद Gateway को पुनः आरंभ करें,
ताकि वह नया इंस्टॉल अपना सके।

रूट-स्वामित्व वाले Linux सिस्टम-व्यापी इंस्टॉल के लिए, यदि `openclaw update`
`EACCES` के साथ विफल हो जाए, तो मैन्युअल प्रतिस्थापन के दौरान Gateway को रोके रखते हुए
सिस्टम npm से पुनर्प्राप्त करें। वही प्रोफ़ाइल फ़्लैग/पर्यावरण उपयोग करें जिनका सामान्यतः उस
Gateway के लिए उपयोग किया जाता है। `/usr/bin/npm` को उस सिस्टम npm से बदलें जो आपके
होस्ट पर रूट-स्वामित्व वाले वैश्विक प्रीफ़िक्स का स्वामी है:

```bash
openclaw gateway stop
sudo /usr/bin/npm i -g openclaw@latest
openclaw gateway install --force
openclaw gateway restart
```

फिर सत्यापित करें:

```bash
openclaw --version
curl -fsS http://127.0.0.1:18789/readyz
openclaw plugins list --json
openclaw gateway status --deep --json
openclaw doctor --lint --json
```

जब `openclaw update` किसी वैश्विक npm इंस्टॉल को प्रबंधित करता है, तो वह पहले लक्ष्य को
एक अस्थायी npm प्रीफ़िक्स में इंस्टॉल करता है। उम्मीदवार पैकेज `preinstall` के दौरान होस्ट
Node संस्करण का सत्यापन करता है; उसके बाद ही OpenClaw पैकेज की गई
`dist` इन्वेंटरी को सत्यापित करता है और स्वच्छ पैकेज ट्री को वास्तविक वैश्विक प्रीफ़िक्स में बदलता है।
पैक किया गया पूर्णता गार्ड अपेक्षित इन्वेंटरी से छोड़ा जाता है और केवल
`preinstall` के सफल होने के बाद हटाया जाता है, इसलिए छोड़ी गई जीवनचक्र स्क्रिप्ट भी
परिवर्तन से पहले विफल हो जाती हैं। npm 12 और नए संस्करणों पर, अपडेटर केवल उम्मीदवार OpenClaw
जीवनचक्र को स्वीकृति देता है; ट्रांज़िटिव निर्भरता स्क्रिप्ट अवरुद्ध रहती हैं। इससे npm द्वारा
पुराने पैकेज की बासी फ़ाइलों के ऊपर नया पैकेज चढ़ाने से बचाव होता है। यदि इंस्टॉल
कमांड विफल होता है, तो OpenClaw `--omit=optional` के साथ एक बार पुनः प्रयास करता है, जो उन
होस्ट पर सहायक होता है जहाँ नेटिव वैकल्पिक निर्भरताएँ कंपाइल नहीं हो सकतीं।

OpenClaw द्वारा प्रबंधित npm अपडेट और plugin-अपडेट कमांड चाइल्ड npm प्रक्रिया के लिए npm की
`min-release-age` सप्लाई-चेन क्वारंटीन (या पुरानी `before` कॉन्फ़िगरेशन कुंजी)
को भी साफ़ करते हैं। वह नीति सामान्य सुरक्षा के लिए मौजूद है, लेकिन स्पष्ट
OpenClaw अपडेट का अर्थ है, "चुनी गई रिलीज़ अभी इंस्टॉल करें।"

```bash
pnpm add -g openclaw@latest
```

यदि pnpm 11 ने OpenClaw 2026.7.1 इंस्टॉल किया है, तो उस मैन्युअल कमांड को एक बार चलाएँ। वह
रिलीज़ pnpm 11 के पृथक वैश्विक-पैकेज लेआउट से पहले की है, इसलिए उसका अपडेटर
किसी अन्य npm इंस्टॉलेशन को चल रहा CLI समझ सकता है। बाद की रिलीज़
pnpm स्वामित्व बनाए रखती हैं और अपडेट के दौरान प्रतिस्थापन पैकेज रूट का अनुसरण करती हैं। वे
स्वामी मैनेजर द्वारा रिपोर्ट की गई वैश्विक bin डायरेक्टरी का भी उपयोग करती हैं और
जब उपलब्ध pnpm कमांड किसी अन्य वैश्विक रूट या प्रमुख संस्करण की सूचना देता है,
या जब आह्वान करने वाला पैकेज अनाथ हो या वहाँ एकमात्र सक्रिय OpenClaw
इंस्टॉल न हो, तो बदलाव से पहले रुक जाती हैं।

यदि OpenClaw किसी अन्य पैकेज के साथ pnpm 11 वैश्विक इंस्टॉल समूह साझा करता है, तो
स्वचालित अपडेटर समूह को बदलने से पहले रुक जाता है। मूल
कॉमा-विभाजित समूह को मैन्युअल रूप से अपडेट करें, ताकि उसके सहोदर पैकेज और बिल्ड नीति
अक्षुण्ण रहें।

```bash
bun add -g openclaw@latest
```

## स्वचालित अपडेटर

डिफ़ॉल्ट रूप से बंद। इसे `~/.openclaw/openclaw.json` में सक्षम करें:

```json5
{
  update: {
    channel: "stable",
    auto: {
      enabled: true,
    },
  },
}
```

| चैनल           | व्यवहार                                                                                                                      |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `stable`          | विस्तृत रोलआउट के लिए नियतात्मक जिटर सहित अंतर्निहित विलंब के बाद लागू होता है।                                                |
| `extended-stable` | `checkOnStart` सक्षम होने पर स्टार्टअप और प्रत्येक 24 घंटे में केवल-पठन अपडेट संकेत की जाँच करता है। कभी भी स्वचालित रूप से लागू नहीं होता। |
| `beta`            | अंतर्निहित अंतराल पर जाँच करता है और तुरंत लागू होता है।                                                                        |
| `dev`             | कोई स्वचालित अनुप्रयोग नहीं। `openclaw update` का मैन्युअल रूप से उपयोग करें।                                                                           |

Gateway स्टार्टअप पर एक अपडेट संकेत भी लॉग करता है (इसे
`update.checkOnStart: false` से अक्षम करें)। संग्रहीत extended-stable चयन इस
केवल-पठन संकेत पथ और मौजूदा 24-घंटे के संकेत अंतराल का उपयोग करते हैं, लेकिन कभी भी
स्वचालित इंस्टॉलेशन, हैंडऑफ़, पुनः आरंभ, स्थिर विलंब/जिटर या बीटा पोलिंग नहीं चलाते।
डाउनग्रेड या घटना-पुनर्प्राप्ति के लिए, स्वचालित अनुप्रयोगों को अवरुद्ध करने हेतु Gateway परिवेश में `OPENCLAW_NO_AUTO_UPDATE=1` सेट करें, भले ही `update.auto.enabled` कॉन्फ़िगर किया गया हो। स्टार्टअप अपडेट संकेत तब तक चल सकते हैं, जब तक `update.checkOnStart` भी अक्षम न हो।

लाइव Gateway कंट्रोल-प्लेन
(`update.run`) के माध्यम से अनुरोधित पैकेज-मैनेजर अपडेट चल रही Gateway
प्रक्रिया के भीतर पैकेज ट्री को नहीं बदलते। प्रबंधित सेवा इंस्टॉल पर, Gateway एक पृथक हैंडऑफ़ शुरू करता है,
बाहर निकलता है और सामान्य `openclaw update --yes --json` CLI पथ को
सेवा रोकने, पैकेज बदलने, सेवा मेटाडेटा रीफ़्रेश करने, पुनः आरंभ करने, Gateway संस्करण और पहुँच का सत्यापन करने
और जहाँ संभव हो, इंस्टॉल किए गए लेकिन लोड न हुए macOS
LaunchAgent को पुनर्प्राप्त करने देता है। यदि Gateway यह हैंडऑफ़ सुरक्षित रूप से नहीं कर सकता,
तो `update.run` पैकेज मैनेजर को उसी प्रक्रिया में चलाने के बजाय एक सुरक्षित शेल कमांड की सूचना देता है।

जब Control UI साइडबार का अपडेट कार्ड सीधे इस `update.run` प्रवाह को शुरू करेगा,
तो वह **Gateway अपडेट करें** दिखाता है। इसमें ब्राउज़र-होस्टेड Control UI, दूरस्थ
Gateways और मैन्युअल रूप से प्रबंधित स्थानीय Gateways शामिल हैं।

साइन की हुई macOS ऐप में, ऐप के स्वामित्व वाला स्थानीय Gateway उस कार्ड को
**Mac ऐप + Gateway अपडेट करें** में बदल देता है। Sparkle पहले ऐप को अपडेट करता है; दोबारा लॉन्च होने के बाद,
ऐप `openclaw update --tag <app-version> --json` चलाता है, अपने Gateway को पुनः आरंभ करता है,
और सेटअप-जैसी प्रगति विंडो में स्वास्थ्य की पुष्टि करता है। यह विंडो केवल तभी दिखाई देती है
जब उस प्रबंधित Gateway को अपडेट, मरम्मत या इंस्टॉलेशन की आवश्यकता हो; केवल-ऐप अपडेट
सीधे ऐप में दोबारा लॉन्च होते हैं। विफलता का विवरण Retry, [अपडेट मार्गदर्शिका](/hi/install/updating), और
[Discord](https://discord.gg/clawd) कार्रवाइयों के साथ दिखाई देता रहता है। ऐप किसी दूरस्थ या बाहरी रूप से प्रबंधित Gateway के लिए
इस समन्वित पथ का कभी उपयोग नहीं करता, किसी नए Gateway को कभी डाउनग्रेड नहीं करता,
और किसी `extended-stable` चैनल पिन को कभी ओवरराइड नहीं करता।

अपडेट सफल होने पर, ऐप वास्तविक उपयोगकर्ता/चैनल इंटरैक्शन वाले सबसे हाल के
शीर्ष-स्तरीय प्रत्यक्ष सत्र के लिए एक बार का स्वागत इवेंट कतारबद्ध करता है। Cron रन,
Heartbeat और केवल-पृष्ठभूमि सत्र अपडेट उस चयन को नहीं बदलते। दूरस्थ मोड में,
ऐप केवल अपने स्थानीय Mac Node रनटाइम को अपडेट करता है और इवेंट केवल तभी भेजता है
जब कनेक्ट किया गया दूरस्थ Gateway कम-से-कम ऐप जितना नया हो।

## रोलबैक

रोलबैक की दो परतें हैं:

1. वर्तमान स्थिति बनाए रखते हुए पुराना OpenClaw कोड फिर से इंस्टॉल करें।
2. अपडेट से पहले की स्थिति केवल तभी पुनर्स्थापित करें, जब पुराना कोड माइग्रेट किए गए
   कॉन्फ़िग या डेटाबेस का उपयोग न कर सके।

पहले केवल-कोड रोलबैक करें। स्थिति पुनर्स्थापित करने से बैकअप के बाद किए गए
परिवर्तन हट जाते हैं।

### अपडेट करने से पहले: सत्यापित बैकअप बनाएँ

`openclaw update` अपडेट से पहले की स्वचालित कॉन्फ़िग प्रति सुरक्षित रखता है, लेकिन यह
स्थिति की पूर्ण पुनर्प्राप्ति का बिंदु नहीं बनाता। किसी महत्वपूर्ण अपडेट से पहले, इसे
स्पष्ट रूप से बनाएँ:

```bash
mkdir -p ~/Backups/openclaw
openclaw backup create --output ~/Backups/openclaw --verify
```

आर्काइव मैनिफ़ेस्ट OpenClaw संस्करण और बैकअप में शामिल स्रोत पथों को दर्ज करता है।
आर्काइव में क्रेडेंशियल, प्रमाणीकरण प्रोफ़ाइल और चैनल स्थिति हो सकती है, इसलिए इसे
केवल-स्वामी अनुमतियों और लाइव स्थिति डायरेक्टरी जैसी ही सुरक्षा के साथ संग्रहीत करें।
शामिल और जानबूझकर छोड़ी गई फ़ाइलों के लिए [बैकअप](/hi/cli/backup) देखें।

पोर्टेबल आर्काइव द्वारा छोड़ी गई अस्थिर कलाकृतियों सहित बाइट-दर-बाइट पुनर्प्राप्ति बिंदु के लिए,
Gateway रोकें और अपने प्लेटफ़ॉर्म द्वारा उपलब्ध कराया गया फ़ाइल-सिस्टम, वॉल्यूम या VM
स्नैपशॉट उपयोग करें।

### पैकेज इंस्टॉलेशन को रोलबैक करें

प्रकाशित संस्करणों की सूची देखें, फिर ज्ञात रूप से सही संस्करण का पूर्वावलोकन करके उसे इंस्टॉल करें:

```bash
npm view openclaw versions --json
openclaw update --tag <known-good-version> --dry-run
openclaw update --tag <known-good-version>
```

प्रत्यक्ष पैकेज-मैनेजर इंस्टॉलेशन की तुलना में `openclaw update --tag` को प्राथमिकता दी जाती है। यह
डाउनग्रेड का पता लगाता है, पुष्टि माँगता है, इंस्टॉल किए गए लक्ष्य के विरुद्ध प्रबंधित Plugin अभिसरण
और संगतता जाँच चलाता है, सेवा मेटाडेटा रीफ़्रेश करता है, Gateway को पुनः आरंभ करता है
और चल रहे संस्करण की पुष्टि करता है। यदि संग्रहीत चैनल `extended-stable` है, तो
`--channel stable --tag <known-good-version>` उपयोग करें, क्योंकि सटीक एक-बार टैग को
`extended-stable` चयनकर्ता के साथ संयोजित नहीं किया जा सकता।

पैकेज अपडेट सक्रियण से पहले उम्मीदवार को स्टेज और सत्यापित करते हैं। यदि
फ़ाइल-सिस्टम स्वैप या कमांड-शिम प्रतिस्थापन विफल होता है, तो OpenClaw पुराने
पैकेज को स्वचालित रूप से पुनर्स्थापित कर देता है। सफल स्वैप के बाद, Gateway की बाद की स्वास्थ्य विफलता
पैकेज को दोबारा स्वचालित रूप से बदलने के बजाय पिछले संस्करण और मैन्युअल रोलबैक
निर्देशों की रिपोर्ट करती है।

यदि CLI अपडेट पथ उपलब्ध नहीं है, तो उसी पैकेज मैनेजर और इंस्टॉलेशन
स्कोप का उपयोग करें जो वर्तमान Gateway के स्वामी हैं:

```bash
openclaw gateway stop
npm i -g openclaw@<known-good-version>
openclaw gateway install --force
openclaw gateway restart
```

जब वह मैनेजर इंस्टॉलेशन का स्वामी हो, तो `npm` को `pnpm` या `bun` से बदलें। घटना
पुनर्प्राप्ति के दौरान, Gateway परिवेश में `OPENCLAW_NO_AUTO_UPDATE=1` सेट करके किसी सक्षम स्वचालित अपडेटर को तुरंत
नया रिलीज़ लागू करने से रोकें।

### स्रोत चेकआउट को रोलबैक करें

साफ़ चेकआउट का उपयोग करें और ज्ञात रूप से सही टैग या कमिट चुनें:

```bash
git fetch --all --tags
git checkout --detach <known-good-tag-or-commit>
pnpm install && pnpm build
openclaw gateway restart
```

नवीनतम पर लौटने के लिए: `git checkout main && git pull`।

गिट अपडेट शुरू होने के बाद निर्भरता इंस्टॉलेशन, बिल्ड, UI बिल्ड या डॉक्टर विफल होने पर,
अपडेटर गिट चेकआउट को स्वचालित रूप से उसकी पिछली ब्रांच और
SHA पर लौटा देता है। जब आप जानबूझकर कोई पुराना कमिट चुनते हैं, तब भी मैन्युअल चेकआउट
आवश्यक है।

### सत्र SQLite माइग्रेशन के पार डाउनग्रेड करना

पुरानी फ़ाइल-समर्थित OpenClaw रिलीज़ शुरू करने से पहले, आर्काइव की गई लीगेसी ट्रांसक्रिप्ट
कलाकृतियाँ पुनर्स्थापित करने के लिए वर्तमान CLI का उपयोग करें:

```bash
openclaw gateway stop
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

यह SQLite डेटा नहीं हटाता। SQLite माइग्रेशन के बाद बनाए गए सत्र
केवल SQLite में मौजूद होते हैं और पुराने रनटाइम में दिखाई नहीं देंगे। देखें:
[सत्र SQLite माइग्रेशन के बाद डाउनग्रेड करना](/hi/cli/doctor#downgrading-after-session-sqlite-migration)।

### स्थिति केवल आवश्यक होने पर पुनर्स्थापित करें

यदि पुराना कोड किसी नए कॉन्फ़िग या डेटाबेस स्कीमा को नहीं पढ़ सकता, तो
Gateway रोकें और अपडेट से पहले का सत्यापित फ़ाइल-सिस्टम, वॉल्यूम या VM स्नैपशॉट पुनर्स्थापित करें।
पुनर्स्थापित करने से पहले वर्तमान स्थिति को अलग से सुरक्षित रखें, क्योंकि इससे
स्नैपशॉट के बाद किए गए परिवर्तन हट जाते हैं।

व्यापक `openclaw backup create` आर्काइव निर्माण और सत्यापन का समर्थन करते हैं, लेकिन
स्थान पर पूरे आर्काइव का सक्रियण नहीं। किसी व्यापक आर्काइव को स्टेजिंग
डायरेक्टरी में एक्सट्रैक्ट करें और ऑफ़लाइन पुनर्स्थापना के लिए उसकी `manifest.json` स्रोत-से-आर्काइव मैपिंग का उपयोग करें।
`openclaw backup sqlite restore` भी एक सत्यापित डेटाबेस
नए लक्ष्य पर लिखता है; उस लक्ष्य को सक्रिय करना स्पष्ट ऑफ़लाइन ऑपरेटर
चरण बना रहता है।

- स्रोत चेकआउट पर `openclaw update --channel dev` के लिए, आवश्यकता पड़ने पर अपडेटर `pnpm` को स्वचालित रूप से बूटस्ट्रैप करता है। यदि आपको pnpm/corepack बूटस्ट्रैप त्रुटि दिखाई दे, तो `pnpm` को मैन्युअल रूप से इंस्टॉल करें (या `corepack` को फिर से सक्षम करें) और अपडेट दोबारा चलाएँ।
- देखें: [समस्या निवारण](/hi/gateway/troubleshooting)
- Discord में पूछें: [https://discord.gg/clawd](https://discord.gg/clawd)
