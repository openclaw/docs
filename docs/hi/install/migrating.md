OpenClaw तीन माइग्रेशन पथों का समर्थन करता है: किसी अन्य एजेंट सिस्टम से आयात करना, मौजूदा इंस्टॉलेशन को नई मशीन पर ले जाना, और किसी Plugin को उसी स्थान पर अपग्रेड करना।

## किसी अन्य एजेंट सिस्टम से आयात करें

बंडल किए गए माइग्रेशन प्रदाता निर्देशों, MCP सर्वरों, Skills, मॉडल कॉन्फ़िगरेशन और (ऑप्ट-इन) API कुंजियों को OpenClaw में लाते हैं। कोई भी बदलाव करने से पहले योजनाओं का पूर्वावलोकन दिखाया जाता है, रिपोर्ट में सीक्रेट छिपाए जाते हैं और लागू करने की प्रक्रिया सत्यापित बैकअप द्वारा सुरक्षित होती है।

CLI प्रवेश बिंदु [`openclaw migrate`](/hi/cli/migrate) है। ज्ञात स्रोत (`openclaw onboard --flow import`) का पता लगने पर ऑनबोर्डिंग भी माइग्रेशन का विकल्प दे सकती है।

## OpenClaw को नई मशीन पर ले जाएँ

इन चीज़ों को सुरक्षित रखने के लिए **स्टेट डायरेक्टरी** (डिफ़ॉल्ट रूप से `~/.openclaw/`) और अपना **वर्कस्पेस** कॉपी करें:

- **कॉन्फ़िगरेशन** — `openclaw.json` और सभी Gateway सेटिंग।
- **प्रमाणीकरण** — प्रति-एजेंट `auth-profiles.json` (API कुंजियाँ और OAuth), साथ ही `credentials/` के अंतर्गत मौजूद कोई भी चैनल या प्रदाता स्थिति।
- **सेशन** — बातचीत का इतिहास और एजेंट की स्थिति।
- **चैनल स्थिति** — WhatsApp लॉगिन, Telegram सेशन और इसी तरह की अन्य स्थिति।
- **वर्कस्पेस फ़ाइलें** — `MEMORY.md`, `USER.md`, Skills और प्रॉम्प्ट।

<Tip>
अपनी स्टेट डायरेक्टरी के पथ की पुष्टि करने के लिए पुरानी मशीन पर `openclaw status` चलाएँ। कस्टम प्रोफ़ाइल `~/.openclaw-<profile>/` या `OPENCLAW_STATE_DIR` के माध्यम से निर्धारित पथ का उपयोग करती हैं।
</Tip>

### माइग्रेशन के चरण

<Steps>
  <Step title="Gateway रोकें और बैकअप लें">
    कॉपी करने के दौरान फ़ाइलों में बदलाव न हो, इसलिए **पुरानी** मशीन पर Gateway रोकें और फिर संग्रह बनाएँ:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    यदि आप कई प्रोफ़ाइल का उपयोग करते हैं (उदाहरण के लिए `~/.openclaw-work`), तो प्रत्येक को अलग से संग्रहित करें।

  </Step>

  <Step title="नई मशीन पर OpenClaw इंस्टॉल करें">
    नई मशीन पर CLI (और आवश्यकता होने पर Node) [इंस्टॉल करें](/hi/install)। यदि ऑनबोर्डिंग नया `~/.openclaw/` बनाती है तो कोई समस्या नहीं है—अगले चरण में आप इसे अधिलेखित कर देंगे।
  </Step>

  <Step title="स्टेट डायरेक्टरी और वर्कस्पेस कॉपी करें">
    संग्रह को `scp`, `rsync -a` या किसी बाहरी ड्राइव के माध्यम से स्थानांतरित करें और फिर उसे निकालें:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    पुष्टि करें कि छिपी हुई डायरेक्टरी शामिल की गई थीं और फ़ाइलों का स्वामित्व उस उपयोगकर्ता से मेल खाता है जो Gateway चलाएगा।

  </Step>

  <Step title="Doctor चलाएँ और सत्यापित करें">
    कॉन्फ़िगरेशन माइग्रेशन लागू करने और सेवाओं की मरम्मत करने के लिए नई मशीन पर [Doctor](/hi/gateway/doctor) चलाएँ:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

यदि Telegram या Discord डिफ़ॉल्ट एनवायरनमेंट फ़ॉलबैक (`TELEGRAM_BOT_TOKEN` या `DISCORD_BOT_TOKEN`) का उपयोग करता है, तो सीक्रेट मान प्रिंट किए बिना सत्यापित करें कि माइग्रेट की गई स्टेट डायरेक्टरी की `.env` में वे कुंजियाँ मौजूद हैं:

```bash
awk -F= '/^(TELEGRAM_BOT_TOKEN|DISCORD_BOT_TOKEN)=/ { print $1 "=present" }' ~/.openclaw/.env
```

जब किसी सक्षम डिफ़ॉल्ट Telegram या Discord खाते के लिए कोई टोकन कॉन्फ़िगर नहीं होता और संबंधित एनवायरनमेंट वेरिएबल Doctor प्रक्रिया के लिए उपलब्ध नहीं होता, तब `openclaw doctor` चेतावनी भी देता है।

### सामान्य समस्याएँ

<AccordionGroup>
  <Accordion title="प्रोफ़ाइल या स्टेट डायरेक्टरी का मेल न होना">
    यदि पुराना Gateway `--profile` या `OPENCLAW_STATE_DIR` का उपयोग करता था और नया Gateway ऐसा नहीं करता, तो चैनल लॉग आउट दिखाई देंगे और सेशन खाली होंगे। Gateway को माइग्रेट की गई **उसी** प्रोफ़ाइल या स्टेट डायरेक्टरी के साथ शुरू करें और फिर `openclaw doctor` दोबारा चलाएँ।
  </Accordion>

  <Accordion title="केवल openclaw.json कॉपी करना">
    केवल कॉन्फ़िगरेशन फ़ाइल पर्याप्त नहीं है। मॉडल प्रमाणीकरण प्रोफ़ाइल `agents/<agentId>/agent/auth-profiles.json` के अंतर्गत होती हैं और चैनल तथा प्रदाता की स्थिति `credentials/` के अंतर्गत होती है। हमेशा **पूरी** स्टेट डायरेक्टरी माइग्रेट करें।
  </Accordion>

  <Accordion title="अनुमतियाँ और स्वामित्व">
    यदि आपने root के रूप में कॉपी किया है या उपयोगकर्ता बदला है, तो Gateway क्रेडेंशियल पढ़ने में विफल हो सकता है। सुनिश्चित करें कि स्टेट डायरेक्टरी और वर्कस्पेस का स्वामित्व Gateway चलाने वाले उपयोगकर्ता के पास है।
  </Accordion>

  <Accordion title="रिमोट मोड">
    यदि आपका UI किसी **रिमोट** Gateway की ओर इंगित करता है, तो सेशन और वर्कस्पेस का स्वामित्व रिमोट होस्ट के पास होता है। अपने स्थानीय लैपटॉप के बजाय स्वयं Gateway होस्ट को माइग्रेट करें। [अक्सर पूछे जाने वाले प्रश्न](/hi/help/faq#where-things-live-on-disk) देखें।
  </Accordion>

  <Accordion title="बैकअप में सीक्रेट">
    स्टेट डायरेक्टरी में प्रमाणीकरण प्रोफ़ाइल, चैनल क्रेडेंशियल और अन्य प्रदाता स्थितियाँ होती हैं। बैकअप को एन्क्रिप्ट करके रखें, असुरक्षित स्थानांतरण चैनलों से बचें और यदि आपको डेटा उजागर होने का संदेह हो तो कुंजियाँ बदलें।
  </Accordion>
</AccordionGroup>

### सत्यापन चेकलिस्ट

नई मशीन पर पुष्टि करें:

- [ ] `openclaw status` दिखाता है कि Gateway चल रहा है।
- [ ] चैनल अब भी कनेक्ट हैं (दोबारा पेयर करने की आवश्यकता नहीं है)।
- [ ] डैशबोर्ड खुलता है और मौजूदा सेशन दिखाता है।
- [ ] वर्कस्पेस फ़ाइलें (मेमोरी, कॉन्फ़िगरेशन) मौजूद हैं।

## Plugin को उसी स्थान पर अपग्रेड करें

उसी स्थान पर किए जाने वाले Plugin अपग्रेड समान Plugin आईडी और कॉन्फ़िगरेशन कुंजियों को सुरक्षित रखते हैं, लेकिन डिस्क पर मौजूद स्थिति को वर्तमान लेआउट में ले जा सकते हैं। Plugin-विशिष्ट अपग्रेड मार्गदर्शिकाएँ उनके चैनलों के साथ उपलब्ध हैं:

- [Matrix माइग्रेशन](/hi/channels/matrix-migration): एन्क्रिप्ट की गई स्थिति की पुनर्प्राप्ति सीमाएँ, स्वचालित स्नैपशॉट व्यवहार और मैन्युअल पुनर्प्राप्ति कमांड।
