ClickClack प्रथम-श्रेणी ClickClack बॉट टोकन के माध्यम से OpenClaw को स्वयं-होस्ट किए गए ClickClack वर्कस्पेस से जोड़ता है।

इसका उपयोग तब करें जब आप चाहते हैं कि कोई OpenClaw एजेंट ClickClack बॉट उपयोगकर्ता के रूप में दिखाई दे। ClickClack स्वतंत्र सेवा बॉट और उपयोगकर्ता-स्वामित्व वाले बॉट का समर्थन करता है; उपयोगकर्ता-स्वामित्व वाले बॉट एक `owner_user_id` बनाए रखते हैं और केवल आपके द्वारा प्रदान किए गए टोकन स्कोप प्राप्त करते हैं।

## त्वरित सेटअप

ClickClack में **Workspace settings → Integrations → OpenClaw** खोलें, **Setup code (recommended)** का उपयोग करके एक
बॉट बनाएँ और जनरेट किया गया कमांड कॉपी करें:

```bash
openclaw channels add clickclack --code 'https://clickclack.example.com/#XXXX-XXXX-XXXX'
```

सेटअप कोड केवल एक बार उपयोग किया जा सकता है और 10 मिनट बाद समाप्त हो जाता है। OpenClaw उस पर दावा करता है,
नया बनाया गया बॉट टोकन और वर्कस्पेस सेटिंग प्राप्त करता है, अकाउंट सहेजता है,
कनेक्शन सत्यापित करता है और बताता है कि चालू Gateway ने उसे अपनाया या नहीं।
सेटअप कोड स्वयं OpenClaw कॉन्फ़िगरेशन में संग्रहीत नहीं होता।

सार्वजनिक सर्वर के लिए सेटअप-कोड दावे HTTPS का उपयोग करते हैं। लूपबैक या निजी नेटवर्क पर
स्थानीय इंस्टॉलेशन के लिए सादा HTTP भी समर्थित है, जिसमें `localhost`,
निजी IP पते और केवल निजी पतों पर रिज़ॉल्व होने वाले आंतरिक होस्टनाम
शामिल हैं।

यदि OpenClaw पहले से चल रहा है, तो ClickClack स्वतः कनेक्ट हो जाता है और किसी दूसरे
कमांड की आवश्यकता नहीं होती। अन्यथा, इसे इससे शुरू करें:

```bash
openclaw gateway
```

आप कोड को सर्वर URL से अलग भी दे सकते हैं:

```bash
openclaw channels add clickclack --code XXXX-XXXX-XXXX --base-url https://clickclack.example.com
```

निर्देशित सेटअप के लिए चलाएँ:

```bash
openclaw onboard
```

ClickClack चुनें, फिर संकेत मिलने पर सर्वर URL, बॉट टोकन और वर्कस्पेस दर्ज
करें। निर्देशित सेटअप सहेजने के बाद सर्वर, टोकन और वर्कस्पेस की जाँच करता है; जाँच
विफल होने पर कॉन्फ़िगरेशन हटाया नहीं जाता।

### विकल्प: मैन्युअल टोकन

गैर-OpenClaw क्लाइंट को कॉन्फ़िगर करते समय या जब आपको स्पष्ट रूप से टोकन स्वयं
प्रबंधित करना हो, तब ClickClack में **Manual token** चुनें:

```bash
openclaw channels add clickclack --base-url https://clickclack.example.com --token ccb_... --workspace default
```

`workspace` वर्कस्पेस आईडी (`wsp_...`), स्लग या प्रदर्शन नाम स्वीकार करता है।
`--code` को `--token`, `--token-file` या `--use-env` के साथ संयोजित नहीं किया जा सकता।

### विकल्प: परिवेश-आधारित टोकन

डिफ़ॉल्ट अकाउंट कॉन्फ़िगरेशन में टोकन संग्रहीत करने के बजाय `CLICKCLACK_BOT_TOKEN` को पढ़
सकता है:

```bash
export CLICKCLACK_BOT_TOKEN="ccb_..."
openclaw channels add clickclack --base-url https://clickclack.example.com --workspace default --use-env
openclaw gateway
```

नामित अकाउंट को कॉन्फ़िगर किए गए टोकन या टोकन फ़ाइल का उपयोग करना आवश्यक है; साझा परिवेश
वैरिएबल को जानबूझकर केवल डिफ़ॉल्ट अकाउंट तक सीमित रखा गया है।

### JSON5 संदर्भ

समतुल्य कॉन्फ़िगरेशन संरचना यह है:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      defaultTo: "channel:general",
    },
  },
}
```

किसी अकाउंट को तभी कॉन्फ़िगर किया हुआ माना जाता है, जब `baseUrl`, एक टोकन स्रोत और
`workspace` सभी सेट हों। डिफ़ॉल्ट अकाउंट के लिए टोकन स्रोत `token`, `tokenFile` या
`CLICKCLACK_BOT_TOKEN` हो सकता है। `workspace` वर्कस्पेस
आईडी (`wsp_...`), स्लग या नाम स्वीकार करता है; Gateway स्टार्टअप पर इसे आईडी में रिज़ॉल्व करता है।

### अकाउंट कॉन्फ़िगरेशन कुंजियाँ

| कुंजी                     | डिफ़ॉल्ट             | टिप्पणियाँ                                                                                   |
| ----------------------- | ------------------- | --------------------------------------------------------------------------------------- |
| `baseUrl`               | कोई नहीं (आवश्यक)     | ClickClack सर्वर URL।                                                                  |
| `token`                 | कोई नहीं                | साधारण स्ट्रिंग या सीक्रेट संदर्भ (`source: "env" \| "file" \| "exec"`) के रूप में बॉट टोकन।        |
| `tokenFile`             | कोई नहीं                | बॉट-टोकन फ़ाइल का पथ; `token` पर प्राथमिकता लेता है।                                |
| `workspace`             | कोई नहीं (आवश्यक)     | वर्कस्पेस आईडी, स्लग या नाम।                                                            |
| `replyMode`             | `"agent"`           | `"agent"` पूर्ण एजेंट पाइपलाइन चलाता है; `"model"` संक्षिप्त प्रत्यक्ष मॉडल पूर्णताएँ भेजता है। |
| `defaultTo`             | `"channel:general"` | उस समय उपयोग किया जाने वाला लक्ष्य, जब आउटबाउंड पथ कोई लक्ष्य नहीं देता।                                      |
| `allowFrom`             | `["*"]`             | इनबाउंड DM और चैनल संदेशों के लिए उपयोगकर्ता-आईडी अनुमति-सूची।                                 |
| `botUserId`             | स्वतः पहचाना गया       | स्टार्टअप पर बॉट टोकन पहचान से रिज़ॉल्व किया जाता है।                                        |
| `agentId`               | रूट डिफ़ॉल्ट       | इस अकाउंट के इनबाउंड संदेशों को एक एजेंट से पिन करें।                                       |
| `toolsAllow`            | कोई नहीं                | इस अकाउंट से एजेंट के उत्तरों के लिए टूल अनुमति-सूची।                                     |
| `model`, `systemPrompt` | कोई नहीं                | `replyMode: "model"` पूर्णताओं द्वारा उपयोग किए जाते हैं।                                               |
| `commandMenu`           | `true`              | ClickClack कंपोज़र स्वतःपूर्णता में मूल कमांड प्रकाशित करें।                            |
| `reconnectMs`           | `1500`              | रीयलटाइम पुनःकनेक्शन विलंब (100 से 60000)।                                                |

यदि `plugins.allow` एक गैर-रिक्त प्रतिबंधात्मक सूची है, तो चैनल सेटअप में
ClickClack को स्पष्ट रूप से चुनने या `openclaw plugins enable clickclack` चलाने पर
उस सूची में `clickclack` जोड़ दिया जाता है। ऑनबोर्डिंग इंस्टॉलेशन समान
स्पष्ट-चयन व्यवहार का उपयोग करता है। ये पथ `plugins.deny` या किसी
वैश्विक `plugins.enabled: false` सेटिंग को ओवरराइड नहीं करते। प्रत्यक्ष
`openclaw plugins install @openclaw/clickclack` सामान्य
Plugin-इंस्टॉल नीति का पालन करता है और मौजूदा अनुमति-सूची में ClickClack को भी दर्ज करता है।

## एकाधिक बॉट

प्रत्येक अकाउंट अपना ClickClack रीयलटाइम कनेक्शन खोलता है और अपने बॉट टोकन का उपयोग करता है।

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      defaultAccount: "service",
      accounts: {
        service: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SERVICE_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "channel:general",
          agentId: "service-bot",
        },
        support: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SUPPORT_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "dm:usr_...",
          agentId: "support-bot",
        },
      },
    },
  },
}
```

## उत्तर मोड

- `replyMode: "agent"` (डिफ़ॉल्ट) सत्र रिकॉर्डिंग और टूल नीति सहित सामान्य एजेंट पाइपलाइन के माध्यम से इनबाउंड संदेश भेजता है।
- `replyMode: "model"` एजेंट पाइपलाइन को छोड़ देता है और प्रत्यक्ष बॉट उत्तरों के लिए Plugin रनटाइम के `llm.complete` का उपयोग करता है, जिसे वैकल्पिक रूप से `model` और `systemPrompt` आकार देते हैं। चयनित प्रदाता और मॉडल पूर्णता बजट के स्वामी होते हैं।

मॉडल मोड रिज़ॉल्व की गई बॉट एजेंट आईडी के विरुद्ध पूर्णताएँ चलाता है, जिसके लिए
स्पष्ट `plugins.entries.clickclack.llm.allowAgentIdOverride: true` विश्वास
बिट आवश्यक है:

```json5
{
  plugins: {
    entries: {
      clickclack: {
        llm: {
          allowAgentIdOverride: true,
        },
      },
    },
  },
}
```

यदि आप केवल डिफ़ॉल्ट `agent` उत्तर मोड का उपयोग करते हैं, तो विश्वास बिट बंद रखें; वहाँ इसकी
आवश्यकता नहीं है।

## कमांड मेनू

Gateway स्टार्टअप पर प्रत्येक कॉन्फ़िगर किया गया अकाउंट OpenClaw के मूल
कमांड ClickClack पर प्रकाशित करता है। वे कंपोज़र स्वतःपूर्णता में बॉट के
हैंडल से लेबल किए हुए दिखाई देते हैं। प्रत्येक स्टार्टअप पर प्रकाशित सेट को पूरी तरह बदल दिया जाता है,
जिसमें मूल कमांड कैटलॉग खाली होने पर पुराने मेनू को साफ़ करना भी शामिल है।

कमांड-मेनू सिंक डिफ़ॉल्ट रूप से सक्षम है। इससे बाहर रहने के लिए किसी अकाउंट पर `commandMenu: false`
सेट करें:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      commandMenu: false,
    },
  },
}
```

टोकन को `commands:write` की आवश्यकता होती है। वर्तमान ClickClack `bot:write` और
`bot:admin` बंडल में वह स्कोप शामिल है और उसे अलग से भी प्रदान किया जा सकता है।
कमांड मेनू प्रस्तुत किए जाने से पहले बनाए गए टोकन में स्कोप जोड़ने या टोकन
बदलने की आवश्यकता हो सकती है।

सिंक सर्वोत्तम प्रयास के आधार पर किया जाता है और प्रत्येक Gateway प्रारंभ पर एक बार चलता है। अनुपलब्ध स्कोप या नेटवर्क
विफलता पर चेतावनी लॉग होती है; एंडपॉइंट रहित पुराने ClickClack सर्वर पर
डीबग स्तर पर लॉग होता है। इनमें से कोई भी विफलता रीयलटाइम स्टार्टअप को अवरुद्ध नहीं करती। एजेंट के ऑफ़लाइन रहने पर भी मेनू
उपलब्ध रहते हैं और बॉट के वर्कस्पेस छोड़ने पर हटा दिए जाते हैं।

यह रिलीज़ केवल मूल कमांड विनिर्देश प्रकाशित करती है। उपनाम और
Skills-, Plugin- या कस्टम-कमांड कैटलॉग मेनू में नहीं जोड़े जाते। यदि कोई
नाम HTTP स्लैश कमांड के रूप में भी पंजीकृत है, तो ClickClack पहले उस
पंजीकरण को डिस्पैच करता है; अन्य मेनू कमांड सामान्य संदेश
डिलीवरी के माध्यम से जारी रहते हैं।

क्रॉस-सर्विस सहसंबंध प्रमाण के लिए `agent` मोड का उपयोग करें। उसके प्रामाणिक
`msg_<ulid>` आकार में आधिकारिक ClickClack संदेश आईडी के लिए, चैनल
नियतात्मक OpenClaw रन आईडी `clickclack:<message-id>` प्राप्त करता है। इसके बाद प्रत्येक मॉडल कॉल
निदान में `clickclack:<message-id>:model:<n>` के रूप में दिखाई देती है; जब वह
टर्न ClawRouter का उपयोग करता है, तो वही मॉडल-कॉल आईडी `X-Request-ID` के रूप में भेजी जाती है।
`model` मोड सामान्य एजेंट रन/सत्र निदान को बायपास करता है और इसलिए
इस प्रमाण पथ के लिए उपयुक्त नहीं है।

जब किसी रीयलटाइम इवेंट में सत्यापित `payload.correlation_id` होता है, तो
चैनल उसे प्रामाणिक संदेश फ़ेच और
उससे बनने वाले ClickClack उत्तर अनुरोधों पर `X-Correlation-ID` के रूप में आगे ले जाता है। मान ClickClack के सुरक्षित
128-वर्ण सेट (`A-Z`, `a-z`, `0-9`, `.`, `_`, `:` और `-`) का उपयोग करते हैं; अमान्य मान
छोड़ दिए जाते हैं। इन संयोजनों में केवल पहचानकर्ता होते हैं, संदेश का मुख्य भाग,
प्रॉम्प्ट, पूर्णताएँ, क्रेडेंशियल या टूल आउटपुट कभी नहीं।

## टिकाऊ मीडिया डिलीवरी

मीडिया वाले एजेंट उत्तर आवश्यक टिकाऊ डिलीवरी का उपयोग करते हैं। OpenClaw पहले
ClickClack लेखन से पहले प्रत्येक भाग के लिए स्थिर संदेश और अपलोड नॉन्स निर्धारित करता है, ताकि
पुनःप्रयास स्टोरेज कोटा खर्च करने या डुप्लिकेट प्रकाशित करने के बजाय समान अपलोड और संदेश का
पुनः उपयोग करे। यदि पुनःप्रारंभ के बाद कोई अपलोड पहले से मौजूद है,
तो OpenClaw मूल स्थानीय पथ या रिमोट मीडिया URL को दोबारा नहीं पढ़ता।

इस पुनर्प्राप्ति अनुबंध के लिए ऐसे ClickClack सर्वर की आवश्यकता होती है, जो इसका समर्थन करे:

- `GET /api/uploads/by-nonce`, जिसमें
  प्राप्त और अनुपलब्ध परिणामों पर `X-ClickClack-Upload-Nonce: supported` हो।
- `GET /api/messages/by-nonce`, जिसमें
  प्राप्त और अनुपलब्ध परिणामों पर `X-ClickClack-Message-Nonce: supported` हो।
- समान
  स्वामी-स्कोप नॉन्स और अपलोड के लिए इडेम्पोटेंट संदेश निर्माण और अटैचमेंट संबद्धता।

पुराने सर्वर की सामान्य 404 प्रतिक्रिया को इस प्रमाण के रूप में नहीं माना जाता कि कोई प्रेषण अनुपस्थित है।
डुप्लिकेट का जोखिम लेने के बजाय OpenClaw डिलीवरी को अनिर्णीत छोड़ देता है; मीडिया उत्पन्न करने वाले
एजेंट उत्तरों को सक्षम करने से पहले ClickClack अपडेट करें।

## एजेंट गतिविधि पंक्तियाँ

डिफ़ॉल्ट रूप से, एजेंट टर्न के चलने के दौरान ClickClack चैनल कुछ नहीं दिखाता; केवल अंतिम उत्तर पहुँचता है। टर्न जारी रहने के दौरान टिकाऊ `agent_commentary` और `agent_tool` संदेश पंक्तियाँ प्रकाशित करने के लिए किसी अकाउंट पर `agentActivity: true` सेट करें:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      agentActivity: true,
    },
  },
}
```

आवश्यकताएँ और व्यवहार:

- **डिफ़ॉल्ट रूप से बंद।** मानक सेटअप और पुराने ClickClack सर्वर अप्रभावित रहते हैं।
- **इसके लिए `agent_activity:write` टोकन स्कोप आवश्यक है।** यह स्कोप `bot:write` से अलग है और इससे इनहेरिट नहीं होता; विकल्प सक्षम करने से पहले `--scopes bot:write,agent_activity:write` के साथ बॉट टोकन बनाएँ (या किसी मौजूदा टोकन को यह स्कोप प्रदान करें)।
- **यथासंभव सुचारु अवनयन।** यदि टोकन में `agent_activity:write` नहीं है या सर्वर गतिविधि लेखन अस्वीकार करता है, तो विफलताएँ लॉग की जाती हैं और अंतिम उत्तर फिर भी सामान्य रूप से डिलीवर होता है; कोई गतिविधि पंक्ति दिखाई नहीं देती।
- पंक्तियाँ प्रत्येक टर्न (`turn_id`) के अनुसार समूहीकृत और इस तरह संयोजित की जाती हैं कि एक तार्किक चरण एक पंक्ति हो, तथा टूल पंक्तियाँ Discord/Slack/Telegram जैसी ही प्रगति फ़ॉर्मैटिंग (टूल का नाम और कमांड का विवरण) उपयोग करती हैं।
- **एट्रिब्यूशन मेटाडेटा।** एजेंट द्वारा लिखी गई पोस्ट (गतिविधि पंक्तियाँ और अंतिम उत्तर) में `author_model` और `author_thinking` फ़ील्ड होते हैं, जिन्हें उस टर्न के लिए वास्तव में उपयोग किए गए मॉडल से निर्धारित किया जाता है (फ़ॉलबैक के बाद भी)। जो सर्वर इन कॉलम को परिभाषित नहीं करते, वे अज्ञात JSON फ़ील्ड को अनदेखा कर देते हैं; जो सर्वर इन्हें बनाए रखते हैं, वे प्रत्येक संदेश के लिए "यह पंक्ति किस मॉडल ने और किस चिंतन स्तर पर कही" का उत्तर दे सकते हैं।

## लक्ष्य

- `channel:<name-or-id>` किसी कार्यस्थान चैनल पर भेजता है। बिना उपसर्ग वाले लक्ष्य डिफ़ॉल्ट रूप से `channel:` होते हैं।
- `dm:<user_id>` उस उपयोगकर्ता के साथ सीधी बातचीत बनाता है या मौजूदा बातचीत का पुनः उपयोग करता है।
- `thread:<message_id>` उस संदेश से शुरू होने वाले थ्रेड में उत्तर देता है।

स्पष्ट आउटबाउंड लक्ष्यों में `clickclack:` या `cc:` प्रदाता उपसर्ग भी हो सकता है।

आउटबाउंड मीडिया ClickClack के अपलोड API का उपयोग करता है और फिर स्थायी अपलोड को
बनाए गए चैनल संदेश, थ्रेड उत्तर या DM से संलग्न करता है। स्थानीय फ़ाइलें और समर्थित
दूरस्थ मीडिया URL, OpenClaw की सामान्य मीडिया-पहुँच नीति का पालन करते हैं और प्रति फ़ाइल
सीमा 64 MiB है। स्थायी कतारबद्ध प्रेषण प्रत्येक अपलोड और संदेश भाग के लिए अलग-अलग
स्वामी-स्कोप वाले नॉन्स का उपयोग करते हैं, फिर उन्हीं ऑब्जेक्ट के साथ अनुलग्नक संबंध का
पुनः प्रयास करते हैं। सर्वर अनुबंध और पुनर्प्राप्ति व्यवहार के लिए
[स्थायी मीडिया डिलीवरी](#durable-media-delivery) देखें।

उदाहरण:

```bash
openclaw message send --channel clickclack --target channel:general --message "hello"
openclaw message send --channel clickclack --target dm:usr_123 --message "hello"
openclaw message send --channel clickclack --target thread:msg_123 --message "following up"
```

## अनुमतियाँ

ClickClack टोकन स्कोप ClickClack API द्वारा लागू किए जाते हैं।

- `bot:read`: कार्यस्थान/चैनल/संदेश/थ्रेड/DM/रीयल-टाइम/प्रोफ़ाइल डेटा पढ़ना।
- `bot:write`: `bot:read` के साथ चैनल संदेश, थ्रेड उत्तर, DM, अपलोड और कमांड-मेन्यू प्रकाशन।
- `bot:admin`: `bot:write` के साथ चैनल निर्माण।
- `commands:write`: बॉट का कमांड मेन्यू प्रकाशित करना। वर्तमान `bot:write` और `bot:admin` बंडल में शामिल है और अलग से भी प्रदान किया जा सकता है।
- `agent_activity:write`: स्थायी एजेंट गतिविधि पंक्तियाँ (`agent_commentary` / `agent_tool`)। यह `bot:write` या `bot:admin` से इनहेरिट नहीं होता; केवल `agentActivity: true` सेट होने पर आवश्यक है।

सामान्य एजेंट चैट और कमांड-मेन्यू सिंक के लिए OpenClaw को केवल वर्तमान `bot:write` की आवश्यकता होती है। [एजेंट गतिविधि पंक्तियाँ](#agent-activity-rows) सक्षम करते समय `agent_activity:write` जोड़ें।

## समस्या निवारण

- `ClickClack is not configured for account "<id>"`: उस खाते के लिए `baseUrl`, `token` (उदाहरण के लिए `CLICKCLACK_BOT_TOKEN` के माध्यम से), और `workspace` सेट करें।
- `ClickClack workspace not found: <value>`: `workspace` को ClickClack द्वारा लौटाई गई कार्यस्थान आईडी, स्लग या नाम पर सेट करें।
- कोई इनबाउंड उत्तर नहीं: पुष्टि करें कि टोकन के पास रीयल-टाइम पढ़ने की पहुँच है और ध्यान दें कि बॉट अपने संदेशों तथा अन्य बॉट के संदेशों को अनदेखा करता है।
- चैनल पर प्रेषण विफल होते हैं: सत्यापित करें कि बॉट कार्यस्थान का सदस्य है और उसके पास `bot:write` है।
- कोई कमांड मेन्यू नहीं: पुष्टि करें कि `commandMenu`, `false` नहीं है, ClickClack सर्वर `PUT /api/bots/self/commands` का समर्थन करता है, और टोकन के पास `commands:write` है।
