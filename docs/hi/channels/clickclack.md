ClickClack प्रथम-श्रेणी ClickClack बॉट टोकन के माध्यम से OpenClaw को स्वयं-होस्ट किए गए ClickClack कार्यस्थान से जोड़ता है।

इसका उपयोग तब करें जब आप चाहते हों कि कोई OpenClaw एजेंट ClickClack बॉट उपयोगकर्ता के रूप में दिखाई दे। ClickClack स्वतंत्र सेवा बॉट और उपयोगकर्ता-स्वामित्व वाले बॉट का समर्थन करता है; उपयोगकर्ता-स्वामित्व वाले बॉट एक `owner_user_id` बनाए रखते हैं और केवल आपके द्वारा प्रदान किए गए टोकन स्कोप प्राप्त करते हैं।

## त्वरित सेटअप

ClickClack में, **Workspace settings → Integrations → OpenClaw** खोलें, **Setup code (recommended)** का उपयोग करके एक
बॉट बनाएँ और जनरेट की गई कमांड कॉपी करें:

```bash
openclaw channels add clickclack --code 'https://clickclack.example.com/#XXXX-XXXX-XXXX'
```

अलग फ्रंटएंड और API ओरिजिन या पथ पर माउंट किए गए API के लिए, ClickClack इसके बजाय एक
सटीक क्लेम एंडपॉइंट देता है:

```bash
openclaw channels add clickclack --code 'https://api.example.com/services/clickclack/api/bot-setup-codes/claim#XXXX-XXXX-XXXX'
```

सेटअप कोड केवल एक बार उपयोग किया जा सकता है और 10 मिनट बाद समाप्त हो जाता है। OpenClaw इसे क्लेम करता है,
नया बनाया गया बॉट टोकन और कार्यस्थान सेटिंग प्राप्त करता है, खाता सहेजता है,
कनेक्शन सत्यापित करता है और बताता है कि चल रहे Gateway ने उसे ग्रहण किया या नहीं।
संस्करणयुक्त सटीक एंडपॉइंट के लिए, OpenClaw ClickClack द्वारा लौटाए गए प्रामाणिक API
बेस को सत्यापित और सहेजता है, जिसमें कोई भी पथ प्रीफ़िक्स शामिल होता है। सेटअप कोड स्वयं
OpenClaw कॉन्फ़िगरेशन में संग्रहीत नहीं होता।

सार्वजनिक सर्वर के लिए सेटअप-कोड क्लेम HTTPS का उपयोग करते हैं। `localhost` और `127.0.0.1` जैसे
लूपबैक पतों पर स्थानीय इंस्टॉलेशन के लिए साधारण HTTP भी समर्थित है।

यदि OpenClaw पहले से चल रहा है, तो ClickClack स्वचालित रूप से कनेक्ट हो जाता है और दूसरी
कमांड की आवश्यकता नहीं होती। अन्यथा, इसे इससे शुरू करें:

```bash
openclaw gateway
```

आप कोड को सर्वर URL से अलग भी दे सकते हैं:

```bash
openclaw channels add clickclack --code XXXX-XXXX-XXXX --base-url https://clickclack.example.com
```

निर्देशित सेटअप के लिए, चलाएँ:

```bash
openclaw onboard
```

ClickClack चुनें, फिर संकेत मिलने पर सर्वर URL, बॉट टोकन और कार्यस्थान दर्ज करें।
निर्देशित सेटअप सहेजने के बाद सर्वर, टोकन और कार्यस्थान की जाँच करता है; जाँच विफल होने पर
कॉन्फ़िगरेशन हटाया नहीं जाता।

### विकल्प: मैन्युअल टोकन

गैर-OpenClaw क्लाइंट कॉन्फ़िगर करते समय या जब आपको स्पष्ट रूप से टोकन स्वयं प्रबंधित करना हो,
ClickClack में **Manual token** चुनें:

```bash
openclaw channels add clickclack --base-url https://clickclack.example.com --token ccb_... --workspace default
```

`workspace` कार्यस्थान आईडी (`wsp_...`), स्लग या प्रदर्शन नाम स्वीकार करता है।
`--code` को `--token`, `--token-file` या `--use-env` के साथ संयोजित नहीं किया जा सकता।

### विकल्प: परिवेश-आधारित टोकन

डिफ़ॉल्ट खाता कॉन्फ़िगरेशन में टोकन संग्रहीत करने के बजाय `CLICKCLACK_BOT_TOKEN` पढ़ सकता है:

```bash
export CLICKCLACK_BOT_TOKEN="ccb_..."
openclaw channels add clickclack --base-url https://clickclack.example.com --workspace default --use-env
openclaw gateway
```

नामित खातों को कॉन्फ़िगर किया गया टोकन या टोकन फ़ाइल उपयोग करनी होगी; साझा परिवेश
चर जानबूझकर केवल डिफ़ॉल्ट खाते तक सीमित है।

### JSON5 संदर्भ

समतुल्य कॉन्फ़िगरेशन संरचना यह है:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      defaultTo: "channel:general",
    },
  },
}
```

किसी खाते को केवल तभी कॉन्फ़िगर किया हुआ माना जाता है जब `baseUrl`, एक टोकन स्रोत और
`workspace` सभी सेट हों। डिफ़ॉल्ट खाते के लिए टोकन स्रोत `token`, `tokenFile` या
`CLICKCLACK_BOT_TOKEN` हो सकता है। `workspace` कार्यस्थान
आईडी (`wsp_...`), स्लग या नाम स्वीकार करता है; Gateway स्टार्टअप पर इसे आईडी में बदलता है।

### खाता कॉन्फ़िगरेशन कुंजियाँ

| कुंजी                     | डिफ़ॉल्ट             | टिप्पणियाँ                                                                                   |
| ----------------------- | ------------------- | --------------------------------------------------------------------------------------- |
| `baseUrl`               | कोई नहीं (आवश्यक)     | ब्राउज़र में दिखाई देने वाले लिंक के लिए प्रयुक्त सार्वजनिक ClickClack URL।                                    |
| `apiBaseUrl`            | `baseUrl`           | REST और रियलटाइम WebSocket ट्रैफ़िक के लिए वैकल्पिक सर्वर-से-सर्वर एंडपॉइंट।             |
| `token`                 | कोई नहीं                | साधारण स्ट्रिंग या सीक्रेट संदर्भ (`source: "env" \| "file" \| "exec"`) के रूप में बॉट टोकन।        |
| `tokenFile`             | कोई नहीं                | बॉट-टोकन फ़ाइल का पथ; `token` पर प्राथमिकता लेता है।                                |
| `workspace`             | कोई नहीं (आवश्यक)     | कार्यस्थान आईडी, स्लग या नाम।                                                            |
| `replyMode`             | `"agent"`           | `"agent"` पूर्ण एजेंट पाइपलाइन चलाता है; `"model"` छोटे प्रत्यक्ष मॉडल कम्प्लीशन भेजता है। |
| `defaultTo`             | `"channel:general"` | जब आउटबाउंड पथ कोई लक्ष्य नहीं देता, तब प्रयुक्त लक्ष्य।                                      |
| `allowFrom`             | `["*"]`             | इनबाउंड DM और चैनल संदेशों के लिए उपयोगकर्ता-आईडी अनुमति-सूची।                                 |
| `botUserId`             | स्वतः पहचाना गया       | स्टार्टअप पर बॉट टोकन पहचान से निर्धारित।                                        |
| `agentId`               | रूट डिफ़ॉल्ट       | इस खाते के इनबाउंड संदेशों को एक एजेंट पर पिन करें।                                       |
| `toolsAllow`            | कोई नहीं                | इस खाते से एजेंट उत्तरों के लिए टूल अनुमति-सूची।                                     |
| `model`, `systemPrompt` | कोई नहीं                | `replyMode: "model"` कम्प्लीशन द्वारा प्रयुक्त।                                               |
| `commandMenu`           | `true`              | ClickClack कंपोज़र स्वतःपूर्ण में मूल कमांड प्रकाशित करें।                            |
| `reconnectMs`           | `1500`              | रियलटाइम पुनःकनेक्शन विलंब (100 से 60000)।                                                |
| `discussions`           | अक्षम            | प्रति-सत्र प्रबंधित चैनल सेटिंग; [सत्र चर्चाएँ](#session-discussions) देखें।  |

### प्रमाणीकरण-नियंत्रित सार्वजनिक होस्टनाम बनाए रखें

जब ClickClack और OpenClaw Gateway एक ही होस्ट पर चलें, लेकिन सार्वजनिक ClickClack होस्टनाम
Cloudflare Access जैसे प्रमाणीकरण Gateway द्वारा सुरक्षित हो, तब `apiBaseUrl` का उपयोग करें:

```json5
{
  channels: {
    clickclack: {
      baseUrl: "https://clack.openclaw.ai",
      apiBaseUrl: "http://127.0.0.1:8484",
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
    },
  },
}
```

ब्राउज़र उपयोगकर्ताओं के लिए सार्वजनिक होस्टनाम पूरी तरह प्रमाणीकरण-नियंत्रित रह सकता है। OpenClaw
REST अनुरोधों, सेटअप सत्यापन और रियलटाइम WebSocket के लिए लूपबैक एंडपॉइंट
का उपयोग करता है, जबकि चर्चा के `embedUrl` और `openUrl` लिंक सार्वजनिक
`baseUrl` का उपयोग जारी रखते हैं। यदि `apiBaseUrl` छोड़ दिया जाता है, तो सारा ट्रैफ़िक
`baseUrl` का उपयोग करता है, जिससे मौजूदा व्यवहार बना रहता है।

यदि `plugins.allow` एक गैर-रिक्त प्रतिबंधात्मक सूची है, तो चैनल सेटअप में ClickClack को स्पष्ट रूप से चुनने या
`openclaw plugins enable clickclack` चलाने पर
उस सूची में `clickclack` जोड़ दिया जाता है। ऑनबोर्डिंग इंस्टॉलेशन भी इसी
स्पष्ट-चयन व्यवहार का उपयोग करता है। ये पथ `plugins.deny` या वैश्विक
`plugins.enabled: false` सेटिंग को ओवरराइड नहीं करते। प्रत्यक्ष
`openclaw plugins install @openclaw/clickclack` सामान्य
Plugin-इंस्टॉल नीति का पालन करता है और मौजूदा अनुमति-सूची में ClickClack को दर्ज भी करता है।

## एकाधिक बॉट

प्रत्येक खाता अपना ClickClack रियलटाइम कनेक्शन खोलता है और अपने बॉट टोकन का उपयोग करता है।

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      defaultAccount: "service",
      accounts: {
        service: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SERVICE_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "channel:general",
          agentId: "service-bot",
        },
        support: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SUPPORT_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "dm:usr_...",
          agentId: "support-bot",
        },
      },
    },
  },
}
```

## सत्र चर्चाएँ

प्रत्येक OpenClaw सत्र को समर्पित ClickClack चैनल देने के लिए एक ClickClack खाते पर चर्चाएँ
सक्षम करें। खाते के टोकन में
`channels:write` शामिल होना चाहिए (`bot:admin` बंडल में यह शामिल है); सामान्य `bot:write`
सेटअप टोकन चैनल बना या सिंक्रनाइज़ नहीं कर सकता।

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      discussions: {
        enabled: true,
        workspace: "default",
        controlUrlBase: "https://team.openclaw.ai",
        section: "Sessions",
      },
    },
  },
}
```

`discussions.workspace` खाता-स्तरीय `workspace` के समान कार्यस्थान आईडी, स्लग या प्रदर्शन नाम
स्वीकार करता है और डिफ़ॉल्ट रूप से उसी मान का उपयोग करता है। `section`
ClickClack साइडबार अनुभाग को नियंत्रित करता है और इसका डिफ़ॉल्ट `Sessions` है। जब
`controlUrlBase` सेट हो, तो प्रबंधित चैनल वास्तविक Control UI
सत्र रूट, `/chat?session=<encoded-session-key>`, से वापस लिंक होता है।

चर्चाएँ ठीक एक ClickClack खाते पर सक्षम करें। Gateway प्रदाता में
खाता चयनकर्ता नहीं है, इसलिए कॉन्फ़िगरेशन क्रम से किसी एक को चुनने के बजाय
एकाधिक सक्षम चर्चा खातों को अस्वीकार कर दिया जाता है।

चर्चा खोलने पर बाहरी रूप से प्रबंधित के रूप में चिह्नित एक सार्वजनिक ClickClack चैनल बनता है।
Plugin सत्र लेबल, श्रेणी और संग्रह स्थिति को
सिंक्रनाइज़ रखता है। सत्र पुनर्स्थापित करने पर उसका चैनल पुनर्स्थापित होता है; सत्र श्रेणी
हटाने पर चैनल वापस कॉन्फ़िगर किए गए डिफ़ॉल्ट अनुभाग में चला जाता है। किसी
OpenClaw सत्र को हटाने पर ClickClack चैनल हटाने के बजाय संग्रहित किया जाता है, इसलिए उसका
इतिहास उपलब्ध रहता है। चर्चा RPC का उपयोग होने पर और किसी भी बाइंडिंग के मौजूद रहने के दौरान
लगभग हर मिनट Plugin बाइंडिंग का सामंजस्य करता है।

प्रबंधित चैनल में इनबाउंड संदेश, संलग्न मुख्य सत्र के समान
एजेंट आईडी के अंतर्गत एक नियतात्मक सहायक सत्र का उपयोग करते हैं। सहायक एजेंट को बताया जाता है कि किस
मुख्य सत्र का अवलोकन करना है और वह `sessions_history` तथा `session_status`
का उपयोग कर सकता है (`changesSince` वृद्धिशील जाँचों के लिए उपयोगी है)। वह `sessions_send` का उपयोग केवल
तभी करता है जब चर्चा में लोग उससे मुख्य सत्र को संदेश भेजने या निर्देशित करने के लिए कहते हैं।
बाइंडिंग, प्रबंधित स्वामित्व संदर्भ और सहायक-सत्र पीयर पहचान में
पिन किए गए ClickClack सर्वर और चैनल के साथ ठोस OpenClaw सत्र आईडी शामिल होती है।
पुन: उपयोग योग्य सत्र कुंजी रीसेट करने या खाते का लक्ष्य बदलने पर पुराना चैनल स्थानीय रूप से निरस्त हो जाता है,
पुराने क्रेडेंशियल के उपयोग योग्य रहने पर संग्रहित हो जाता है और उसकी सहायक ट्रांसक्रिप्ट का पुन: उपयोग
नहीं किया जा सकता। संग्रहित, रीसेट, अक्षम या पुनर्लक्षित बाइंडिंग के माध्यम से आने वाले संदेश
खाते की सामान्य चैनल रूटिंग पर वापस जाने के बजाय छोड़ दिए जाते हैं।
मुक्त की गई बाइंडिंग एक स्थायी निरस्त-चैनल मार्कर छोड़ती हैं, ताकि विलंबित रियलटाइम घटनाएँ
सुरक्षित रूप से बंद रहें। दूरस्थ स्वामित्व ClickClack सर्वर और चैनल आईडी द्वारा कुंजीबद्ध होता है,
इसलिए स्थानीय खाते का नाम बदलने से प्रबंधित चैनल सामान्य चैनल में नहीं बदल सकता।

`tools.sessions.visibility` को इसके अधिक सुरक्षित डिफ़ॉल्ट `tree` पर रखें। Plugin
प्रत्येक सहायक सत्र और उससे संलग्न मुख्य सत्र के बीच केवल एक होस्ट-स्कोप वाला अनुदान
स्थापित करता है, साथ ही एक टूल-नीति हुक जो सत्र खोज और
क्रॉस-सत्र लक्ष्यों को अवरुद्ध करता है। यह `sessions_history`, `session_status` और
`sessions_send` को केवल संलग्न मुख्य सत्र के लिए अनुमति देता है और स्थिति कॉल को
उस सत्र का मॉडल बदलने से रोकता है। वे टूल अभी भी एजेंट की प्रभावी टूल अनुमति-सूची में
मौजूद होने चाहिए। सिस्टम प्रॉम्प्ट मार्गदर्शन है; होस्ट अनुदान
और हुक प्राधिकरण सीमा हैं।

ClickClack सर्वर को चैनल बनाने और अपडेट करने पर managed-channel फ़ील्ड (`external_managed`,
`external_ref`, `external_url`, और `sidebar_section`) का समर्थन करना और
उन्हें चैनल प्रतिक्रियाओं में लौटाना आवश्यक है। OpenClaw किसी बाइंडिंग को स्थायी रूप से सहेजने से
पहले उस अनुबंध की पुष्टि करता है। यदि निर्माण की प्रतिक्रिया खो जाती है, तो अगली बार खोलने पर
दूसरा चैनल बनाने के बजाय सर्वर द्वारा लागू `external_ref` के आधार पर चैनल अपना लिया जाता है।
जब तक इस परिणाम का मिलान नहीं हो जाता, लंबित आरक्षण गंतव्य वर्कस्पेस में
अन्यथा अनबाउंड इवेंट को क्वारंटीन में रखता है। मोटे स्तर का रिकंसाइलर
उसी सेशन के अब भी सक्रिय होने पर चैनल अपना लेता है या रीसेट के बाद उसे
आर्काइव कर देता है; कोई रिमोट चैनल न बनाए जाने पर यह आरक्षण हटा देता है।
उस संदर्भ में प्रति-OpenClaw-इंस्टॉलेशन एक स्थायी नेमस्पेस और
सेशन कुंजी, ठोस सेशन आईडी, ClickClack गंतव्य तथा स्थायी
बाइंडिंग जनरेशन का हैश शामिल होता है। अलग-अलग Gateway एक-दूसरे के चैनल नहीं अपना सकते,
रीसेट किए गए सेशन पुराने चैनल इतिहास को इनहेरिट नहीं कर सकते, और किसी अकाउंट या वर्कस्पेस की
राउंड ट्रिप पिछले चैनल को दोबारा नहीं अपना सकती। बाइंडिंग को कॉन्फ़िगर किए गए
ClickClack सर्वर URL से भी पिन किया जाता है और अकाउंट को
पुनः लक्षित करने पर वे अमान्य हो जाती हैं। `controlUrlBase` को बदलने या हटाने से अगले
रिकंसिलिएशन पास में प्रबंधित चैनल लिंक अपडेट या साफ़ हो जाता है।
`discussions.workspace` बदलने पर, पुराने वर्कस्पेस का क्रेडेंशियल कॉन्फ़िगर रहने की स्थिति में,
नए वर्कस्पेस में चैनल खोले जाने से पहले पुरानी बाइंडिंग आर्काइव और रिलीज़ कर दी जाती है।
यदि टोकन को ऐसे वर्कस्पेस-स्कोप्ड क्रेडेंशियल से बदल दिया गया है जो
पुराने वर्कस्पेस को एक्सेस नहीं कर सकता, तो OpenClaw पुराने चैनल को निरस्त के रूप में दर्ज करता है और
प्रतिस्थापन टोकन का प्रयास किए बिना बाइंडिंग रिलीज़ कर देता है; उस बचे हुए
चैनल को ClickClack से आर्काइव करें।

संलग्न मुख्य सेशन को केवल-पुल `discussion` टूल भी मिलता है। यह
नवीनतम संदेशों और हाल के थ्रेड उत्तरों को प्रति संदेश एक एस्केप किए गए, स्रोत-निर्दिष्ट रिकॉर्ड के रूप में
पढ़ता है और इसके कोई लेखन या लाइफ़साइकल दुष्प्रभाव नहीं हैं। चैनल-रूट और थ्रेड
लुकअप के लिए निश्चित अनुरोध बजट होते हैं; परिणाम स्पष्ट रूप से चेतावनी देता है कि
यह सुरक्षा सीमा किसी पुराने सक्रिय थ्रेड को छोड़ सकती है।

## उत्तर मोड

- `replyMode: "agent"` (डिफ़ॉल्ट) इनबाउंड संदेशों को सामान्य एजेंट पाइपलाइन के माध्यम से भेजता है, जिसमें सेशन रिकॉर्डिंग और टूल नीति शामिल हैं।
- `replyMode: "model"` एजेंट पाइपलाइन को छोड़ देता है और सीधे बॉट उत्तरों के लिए Plugin रनटाइम के `llm.complete` का उपयोग करता है, जिन्हें वैकल्पिक रूप से `model` और `systemPrompt` द्वारा आकार दिया जाता है। चुना गया प्रदाता और मॉडल कम्प्लीशन बजट नियंत्रित करते हैं।

मॉडल मोड समाधान किए गए बॉट एजेंट आईडी के विरुद्ध कम्प्लीशन चलाता है, जिसके लिए
स्पष्ट `plugins.entries.clickclack.llm.allowAgentIdOverride: true` ट्रस्ट
बिट आवश्यक है:

```json5
{
  plugins: {
    entries: {
      clickclack: {
        llm: {
          allowAgentIdOverride: true,
        },
      },
    },
  },
}
```

यदि केवल डिफ़ॉल्ट `agent` उत्तर मोड का उपयोग किया जाता है, तो ट्रस्ट बिट बंद रखें;
वहाँ इसकी आवश्यकता नहीं है।

## कमांड मेन्यू

Gateway के स्टार्टअप पर, प्रत्येक कॉन्फ़िगर किया गया अकाउंट OpenClaw के नेटिव
कमांड ClickClack पर प्रकाशित करता है। वे बॉट के हैंडल से लेबल किए हुए
कम्पोज़र ऑटोकम्प्लीट में दिखाई देते हैं। प्रत्येक स्टार्टअप पर प्रकाशित सेट को पूरी तरह
बदल दिया जाता है, जिसमें नेटिव कमांड कैटलॉग खाली होने पर पुराने मेन्यू को साफ़ करना भी शामिल है।

कमांड-मेन्यू सिंक डिफ़ॉल्ट रूप से सक्षम है। ऑप्ट आउट करने के लिए अकाउंट पर
`commandMenu: false` सेट करें:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      commandMenu: false,
    },
  },
}
```

टोकन को `commands:write` की आवश्यकता होती है। मौजूदा ClickClack `bot:write` और
`bot:admin` बंडल में यह स्कोप शामिल है और इसे अलग से भी प्रदान किया जा सकता है।
कमांड मेन्यू आने से पहले बनाए गए टोकन में यह स्कोप जोड़ने या
प्रतिस्थापन टोकन की आवश्यकता हो सकती है।

सिंक सर्वोत्तम प्रयास के आधार पर होता है और प्रत्येक Gateway स्टार्ट पर एक बार चलता है। स्कोप न होना या नेटवर्क
विफलता चेतावनी लॉग करती है; एंडपॉइंट रहित पुराने ClickClack सर्वर के लिए
डीबग स्तर पर लॉग होता है। इनमें से कोई भी विफलता रीयलटाइम स्टार्टअप को अवरुद्ध नहीं करती।
एजेंट के ऑफ़लाइन रहने पर भी मेन्यू उपलब्ध रहते हैं और बॉट के
वर्कस्पेस छोड़ने पर हटा दिए जाते हैं।

यह रिलीज़ केवल नेटिव कमांड स्पेक प्रकाशित करती है। एलियास और
स्किल-, Plugin-, या कस्टम-कमांड कैटलॉग मेन्यू में नहीं जोड़े जाते। यदि किसी
नाम को HTTP स्लैश कमांड के रूप में भी पंजीकृत किया गया है, तो ClickClack पहले उस
पंजीकरण को डिस्पैच करता है; अन्य मेन्यू कमांड सामान्य संदेश
डिलीवरी से गुजरते रहते हैं।

क्रॉस-सर्विस सहसंबंध प्रमाण के लिए `agent` मोड का उपयोग करें। ClickClack संदेश की उसकी कैनोनिकल
`msg_<ulid>` आकृति में आधिकारिक आईडी से, चैनल
नियतात्मक OpenClaw रन आईडी `clickclack:<message-id>` निकालता है। इसके बाद प्रत्येक मॉडल कॉल
डायग्नोस्टिक्स में `clickclack:<message-id>:model:<n>` के रूप में दिखाई देती है; जब वह
टर्न ClawRouter का उपयोग करता है, तो वही मॉडल-कॉल आईडी `X-Request-ID` के रूप में भेजी जाती है।
`model` मोड सामान्य एजेंट रन/सेशन डायग्नोस्टिक्स को बायपास करता है और इसलिए
इस प्रमाण पथ के लिए उपयुक्त नहीं है।

जब किसी रीयलटाइम इवेंट में सत्यापित `payload.correlation_id` होता है, तो
चैनल इसे आधिकारिक संदेश फ़ेच और परिणामस्वरूप मिलने वाले
ClickClack उत्तर अनुरोधों में `X-Correlation-ID` के रूप में रखता है। मान ClickClack के सुरक्षित
128-वर्ण सेट (`A-Z`, `a-z`, `0-9`, `.`, `_`, `:`, और `-`) का उपयोग करते हैं; अमान्य मान
छोड़ दिए जाते हैं। इन जॉइन में केवल आइडेंटिफ़ायर होते हैं, संदेश बॉडी,
प्रॉम्प्ट, कम्प्लीशन, क्रेडेंशियल या टूल आउटपुट कभी नहीं।

## स्थायी मीडिया डिलीवरी

मीडिया वाले एजेंट उत्तरों के लिए स्थायी डिलीवरी आवश्यक है। OpenClaw पहले
ClickClack लेखन से पहले प्रत्येक भाग के लिए स्थिर संदेश और अपलोड नॉन्स निर्धारित करता है, ताकि
पुनः प्रयास उसी अपलोड और संदेश का पुनः उपयोग करे और स्टोरेज कोटा खर्च न करे
या डुप्लिकेट प्रकाशित न करे। यदि रीस्टार्ट के बाद अपलोड पहले से मौजूद है,
तो OpenClaw मूल स्थानीय पथ या रिमोट मीडिया URL को दोबारा नहीं पढ़ता।

इस पुनर्प्राप्ति अनुबंध के लिए ऐसा ClickClack सर्वर आवश्यक है जो इनका समर्थन करता हो:

- `GET /api/uploads/by-nonce`, जिसमें
  मिले और अनुपलब्ध परिणामों पर `X-ClickClack-Upload-Nonce: supported` हो।
- `GET /api/messages/by-nonce`, जिसमें
  मिले और अनुपलब्ध परिणामों पर `X-ClickClack-Message-Nonce: supported` हो।
- समान स्वामी-स्कोप्ड नॉन्स और अपलोड के लिए आइडेम्पोटेंट संदेश निर्माण और अटैचमेंट संबद्धता।

पुराने सर्वर की सामान्य 404 प्रतिक्रिया को भेजे गए संदेश की अनुपस्थिति का प्रमाण नहीं माना जाता।
डुप्लिकेट के जोखिम के बजाय OpenClaw डिलीवरी को अनसुलझा छोड़ देता है; मीडिया उत्पन्न करने वाले
एजेंट उत्तर सक्षम करने से पहले ClickClack को अपडेट करें।

## एजेंट गतिविधि पंक्तियाँ

डिफ़ॉल्ट रूप से, एजेंट टर्न चलने के दौरान ClickClack चैनल कुछ नहीं दिखाता; केवल अंतिम उत्तर पहुँचता है। टर्न की प्रगति के दौरान स्थायी `agent_commentary` और `agent_tool` संदेश पंक्तियाँ प्रकाशित करने के लिए अकाउंट पर `agentActivity: true` सेट करें:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      agentActivity: true,
    },
  },
}
```

आवश्यकताएँ और व्यवहार:

- **डिफ़ॉल्ट रूप से बंद।** मानक सेटअप और पुराने ClickClack सर्वर अप्रभावित रहते हैं।
- **`agent_activity:write` टोकन स्कोप आवश्यक है।** यह स्कोप `bot:write` से अलग है और उससे इनहेरिट नहीं होता; विकल्प सक्षम करने से पहले बॉट टोकन को `--scopes bot:write,agent_activity:write` के साथ बनाएँ (या किसी मौजूदा टोकन को यह स्कोप प्रदान करें)।
- **सर्वोत्तम-प्रयास डिग्रेडेशन।** यदि टोकन में `agent_activity:write` नहीं है या सर्वर गतिविधि लेखन अस्वीकार करता है, तो विफलताएँ लॉग की जाती हैं और अंतिम उत्तर फिर भी सामान्य रूप से डिलीवर होता है; कोई गतिविधि पंक्ति दिखाई नहीं देती।
- पंक्तियों को प्रति टर्न (`turn_id`) समूहित किया जाता है और इस प्रकार संयोजित किया जाता है कि एक तार्किक चरण एक पंक्ति हो; टूल पंक्तियाँ Discord/Slack/Telegram जैसा ही प्रगति फ़ॉर्मेट उपयोग करती हैं (टूल का नाम और कमांड विवरण)।
- **एट्रिब्यूशन मेटाडेटा।** एजेंट द्वारा लिखी गई पोस्ट (गतिविधि पंक्तियाँ और अंतिम उत्तर) में टर्न के लिए वास्तव में उपयोग किए गए मॉडल से समाधान किए गए `author_model` और `author_thinking` फ़ील्ड होते हैं (फ़ॉलबैक के बाद भी)। जो सर्वर इन कॉलम को परिभाषित नहीं करते, वे अज्ञात JSON फ़ील्ड को अनदेखा करते हैं; जो सर्वर इन्हें स्थायी रूप से सहेजते हैं, वे प्रत्येक संदेश के लिए "यह पंक्ति किस मॉडल ने, किस चिंतन स्तर पर कही" का उत्तर दे सकते हैं।

## लक्ष्य

- `channel:<name-or-id>` वर्कस्पेस चैनल को भेजता है। बिना प्रीफ़िक्स वाले लक्ष्य डिफ़ॉल्ट रूप से `channel:` होते हैं।
- `dm:<user_id>` उस उपयोगकर्ता के साथ सीधी बातचीत बनाता है या उसका पुनः उपयोग करता है।
- `thread:<message_id>` उस संदेश से शुरू होने वाले थ्रेड में उत्तर देता है।

स्पष्ट आउटबाउंड लक्ष्यों में `clickclack:` या `cc:` प्रदाता प्रीफ़िक्स भी हो सकता है।

आउटबाउंड मीडिया ClickClack के अपलोड API का उपयोग करता है और फिर स्थायी अपलोड को
बनाए गए चैनल संदेश, थ्रेड उत्तर या DM से अटैच करता है। स्थानीय फ़ाइलें और समर्थित
रिमोट मीडिया URL OpenClaw की सामान्य मीडिया-एक्सेस नीति का पालन करते हैं, जिसमें प्रति फ़ाइल
64 MiB की सीमा है। स्थायी कतारबद्ध प्रेषण प्रत्येक अपलोड और संदेश भाग के लिए अलग-अलग
स्वामी-स्कोप्ड नॉन्स उपयोग करते हैं, फिर उन्हीं ऑब्जेक्ट के साथ अटैचमेंट संबद्धता का
पुनः प्रयास करते हैं। सर्वर अनुबंध और पुनर्प्राप्ति व्यवहार के लिए
[स्थायी मीडिया डिलीवरी](#durable-media-delivery) देखें।

उदाहरण:

```bash
openclaw message send --channel clickclack --target channel:general --message "hello"
openclaw message send --channel clickclack --target dm:usr_123 --message "hello"
openclaw message send --channel clickclack --target thread:msg_123 --message "following up"
```

## अनुमतियाँ

ClickClack टोकन स्कोप ClickClack API द्वारा लागू किए जाते हैं।

- `bot:read`: वर्कस्पेस/चैनल/संदेश/थ्रेड/DM/रीयलटाइम/प्रोफ़ाइल डेटा पढ़ें।
- `bot:write`: `bot:read` के साथ चैनल संदेश, थ्रेड उत्तर, DM, अपलोड और कमांड-मेन्यू प्रकाशन।
- `bot:admin`: `bot:write` के साथ चैनल निर्माण।
- `commands:write`: बॉट का कमांड मेन्यू प्रकाशित करें। मौजूदा `bot:write` और `bot:admin` बंडल में शामिल है और अलग से प्रदान किया जा सकता है।
- `agent_activity:write`: स्थायी एजेंट गतिविधि पंक्तियाँ (`agent_commentary` / `agent_tool`)। `bot:write` या `bot:admin` से इनहेरिट नहीं होता; केवल `agentActivity: true` सेट होने पर आवश्यक है।

सामान्य एजेंट चैट और कमांड-मेन्यू सिंक के लिए OpenClaw को केवल मौजूदा `bot:write` की आवश्यकता है। [एजेंट गतिविधि पंक्तियाँ](#agent-activity-rows) सक्षम करते समय `agent_activity:write` जोड़ें।

## समस्या निवारण

- `ClickClack is not configured for account "<id>"`: उस अकाउंट के लिए `baseUrl`, `token` (उदाहरण के लिए `CLICKCLACK_BOT_TOKEN` के माध्यम से), और `workspace` सेट करें।
- `ClickClack workspace not found: <value>`: `workspace` को ClickClack द्वारा लौटाई गई वर्कस्पेस आईडी, स्लग या नाम पर सेट करें।
- कोई इनबाउंड उत्तर नहीं: पुष्टि करें कि टोकन के पास रीयलटाइम पढ़ने की पहुँच है और ध्यान दें कि बॉट अपने संदेशों तथा अन्य बॉट के संदेशों को अनदेखा करता है।
- चैनल प्रेषण विफल होते हैं: सत्यापित करें कि बॉट वर्कस्पेस का सदस्य है और उसके पास `bot:write` है।
- कोई कमांड मेन्यू नहीं: पुष्टि करें कि `commandMenu`, `false` नहीं है, ClickClack सर्वर `PUT /api/bots/self/commands` का समर्थन करता है, और टोकन के पास `commands:write` है।
