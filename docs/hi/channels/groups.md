OpenClaw समूह-सक्षम चैनलों में समान समूह नियम लागू करता है, जिनमें Discord, iMessage, Matrix, Microsoft Teams, QQBot, Signal, Slack, Telegram, WhatsApp और Zalo शामिल हैं।

हमेशा सक्रिय रहने वाले उन कक्षों के लिए, जिन्हें एजेंट द्वारा स्पष्ट रूप से दृश्य संदेश भेजे जाने तक शांत संदर्भ प्रदान करना चाहिए, [परिवेशी कक्ष इवेंट](/hi/channels/ambient-room-events) देखें।

## शुरुआती परिचय (2 मिनट)

OpenClaw आपके अपने मैसेजिंग खातों पर "रहता" है। कोई अलग WhatsApp बॉट उपयोगकर्ता नहीं होता: यदि **आप** किसी समूह में हैं, तो OpenClaw उस समूह को देख सकता है और वहीं उत्तर दे सकता है।

डिफ़ॉल्ट व्यवहार:

- समूह प्रतिबंधित हैं (`groupPolicy: "allowlist"`); अनुमति-सूची में जोड़े जाने तक समूह प्रेषक अवरुद्ध रहते हैं।
- जब तक आप किसी समूह के लिए उल्लेख गेटिंग अक्षम नहीं करते, उत्तरों के लिए उल्लेख आवश्यक है।
- अंतिम उत्तर का टेक्स्ट स्वचालित रूप से कक्ष में पोस्ट होता है (`visibleReplies: "automatic"`)।

अर्थात: अनुमति-सूची में शामिल प्रेषक OpenClaw का उल्लेख करके उसे सक्रिय कर सकते हैं।

<Note>
**संक्षेप में**

- **DM पहुँच** को `*.allowFrom` नियंत्रित करता है।
- **समूह पहुँच** को `*.groupPolicy` + अनुमति-सूचियाँ (`*.groups`, `*.groupAllowFrom`) नियंत्रित करती हैं।
- **उत्तर सक्रियण** को उल्लेख गेटिंग (`requireMention`, `/activation`) नियंत्रित करती है।

</Note>

त्वरित प्रवाह (समूह संदेश के साथ क्या होता है):

```text
groupPolicy? disabled -> छोड़ें
groupPolicy? allowlist -> समूह अनुमत है? नहीं -> छोड़ें
requireMention? yes -> उल्लेख हुआ? नहीं -> केवल संदर्भ के लिए संग्रहीत करें
उल्लेख/उत्तर/कमांड/DM -> उपयोगकर्ता अनुरोध
हमेशा सक्रिय समूह वार्तालाप -> उपयोगकर्ता अनुरोध, या कॉन्फ़िगर होने पर कक्ष इवेंट
```

## दृश्य उत्तर

सामान्य समूह/चैनल अनुरोधों के लिए, OpenClaw डिफ़ॉल्ट रूप से `messages.groupChat.visibleReplies: "automatic"` का उपयोग करता है: अंतिम सहायक टेक्स्ट दृश्य उत्तर के रूप में कक्ष में पोस्ट होता है।

जब किसी साझा कक्ष में एजेंट को `message(action=send)` कॉल करके यह तय करने देना हो कि कब बोलना है, तब `messages.groupChat.visibleReplies: "message_tool"` का उपयोग करें। यह टूल का विश्वसनीय रूप से उपयोग करने वाले मॉडल के साथ सबसे अच्छा काम करता है (उदाहरण के लिए GPT-5.6 Sol)। यदि मॉडल टूल का उपयोग करना भूल जाता है और सार्थक अंतिम टेक्स्ट लौटाता है, तो OpenClaw उसे कक्ष में पोस्ट करने के बजाय निजी रखता है।

उन मॉडल या रनटाइम के लिए `"automatic"` का उपयोग करें जो केवल-टूल डिलीवरी का विश्वसनीय रूप से पालन नहीं करते: सामान्य अंतिम टेक्स्ट सीधे कक्ष में पोस्ट होता है और एजेंट फिर भी ऐसी फ़ाइलों, इमेज या अन्य अटैचमेंट के लिए `message(action=send)` कॉल कर सकता है जिन्हें अंतिम टेक्स्ट के साथ नहीं भेजा जा सकता।

यदि सक्रिय टूल नीति के अंतर्गत संदेश टूल अनुपलब्ध है, तो OpenClaw उत्तर को चुपचाप दबाने के बजाय स्वचालित दृश्य उत्तरों पर वापस चला जाता है। `openclaw doctor` इस असंगति के बारे में चेतावनी देता है।

प्रत्यक्ष चैट और किसी भी अन्य स्रोत इवेंट के लिए, `messages.visibleReplies: "message_tool"` समान केवल-टूल व्यवहार वैश्विक रूप से लागू करता है; समूह/चैनल कक्षों के लिए `messages.groupChat.visibleReplies` अधिक विशिष्ट ओवरराइड बना रहता है। आंतरिक WebChat के प्रत्यक्ष टर्न डिफ़ॉल्ट रूप से स्वचालित अंतिम-उत्तर डिलीवरी का उपयोग करते हैं, ताकि Pi और Codex को समान दृश्य-उत्तर अनुबंध मिले।

केवल-टूल मोड अधिकांश निष्क्रिय-निगरानी मोड टर्न के लिए मॉडल को `NO_REPLY` उत्तर देने के लिए बाध्य करने वाले पुराने पैटर्न को प्रतिस्थापित करता है। केवल-टूल मोड में प्रॉम्प्ट किसी `NO_REPLY` अनुबंध को परिभाषित नहीं करता; कुछ भी दृश्य न करने का अर्थ केवल संदेश टूल को कॉल न करना है।

Plugin के स्वामित्व वाली वार्तालाप बाइंडिंग इसका अपवाद हैं। जब कोई Plugin किसी थ्रेड को बाइंड करके इनबाउंड टर्न का दावा कर लेता है, तो Plugin द्वारा लौटाया गया उत्तर दृश्य बाइंडिंग उत्तर होता है; उसे `message(action=send)` की आवश्यकता नहीं होती। वह उत्तर Plugin रनटाइम आउटपुट है, निजी मॉडल का अंतिम टेक्स्ट नहीं।

प्रत्यक्ष समूह अनुरोधों के लिए टाइपिंग संकेतक अब भी भेजे जाते हैं। सक्षम होने पर परिवेशी, हमेशा सक्रिय रहने वाले कक्ष इवेंट तब तक सख्त और शांत रहते हैं, जब तक एजेंट संदेश टूल को कॉल नहीं करता।

सेशन डिफ़ॉल्ट रूप से विस्तृत टूल/प्रगति सारांश दबाते हैं। डीबगिंग के दौरान उन्हें वर्तमान सेशन में दिखाने के लिए `/verbose on` (या `/verbose full`) और केवल-अंतिम-उत्तर व्यवहार पर लौटने के लिए `/verbose off` का उपयोग करें। विस्तृत स्थिति प्रत्येक सेशन के लिए अलग होती है और प्रत्यक्ष चैट, समूहों, चैनलों तथा फ़ोरम विषयों में समान रूप से काम करती है।

बिना उल्लेख वाले, हमेशा सक्रिय समूह वार्तालाप को उपयोगकर्ता अनुरोधों के बजाय शांत कक्ष संदर्भ के रूप में सबमिट करने के लिए [परिवेशी कक्ष इवेंट](/hi/channels/ambient-room-events) का उपयोग करें:

```json5
{
  messages: {
    groupChat: {
      unmentionedInbound: "room_event",
    },
  },
}
```

डिफ़ॉल्ट `unmentionedInbound: "user_request"` है। उल्लेख किए गए संदेश, कमांड, निरस्त करने के अनुरोध और DM उपयोगकर्ता अनुरोध बने रहते हैं।

समूह/चैनल अनुरोधों के दृश्य आउटपुट को संदेश टूल से भेजना अनिवार्य करने के लिए:

```json5
{
  messages: {
    groupChat: {
      visibleReplies: "message_tool",
    },
  },
}
```

प्रत्येक स्रोत चैट के लिए इसे अनिवार्य करने हेतु:

```json5
{
  messages: {
    visibleReplies: "message_tool",
  },
}
```

फ़ाइल सहेजे जाने के बाद Gateway बिना पुनः आरंभ किए `messages` कॉन्फ़िगरेशन बदलावों को अपना लेता है। केवल तब पुनः आरंभ करें, जब कॉन्फ़िगरेशन रीलोड अक्षम हो (`gateway.reload.mode: "off"`)।

कमांड टर्न `visibleReplies: "message_tool"` को बायपास करते हैं और हमेशा दृश्य रूप से उत्तर देते हैं: नेटिव स्लैश कमांड (Discord, Telegram और नेटिव कमांड समर्थन वाले अन्य इंटरफ़ेस) तथा अधिकृत टेक्स्ट `/...` कमांड, दोनों अपना उत्तर स्रोत चैट में पोस्ट करते हैं। समूहों में अनधिकृत टेक्स्ट `/...` टर्न केवल-संदेश-टूल बने रहते हैं; सामान्य चैट टर्न कॉन्फ़िगर किए गए डिफ़ॉल्ट का पालन करते हैं।

## संदर्भ दृश्यता और अनुमति-सूचियाँ

समूह सुरक्षा में दो अलग-अलग नियंत्रण शामिल होते हैं:

- **सक्रियण प्राधिकरण**: एजेंट को कौन सक्रिय कर सकता है (`groupPolicy`, `groups`, `groupAllowFrom`, चैनल-विशिष्ट अनुमति-सूचियाँ)।
- **संदर्भ दृश्यता**: मॉडल में कौन-सा पूरक संदर्भ डाला जाता है (उत्तर/उद्धरण टेक्स्ट, थ्रेड इतिहास, अग्रेषित मेटाडेटा)।

डिफ़ॉल्ट रूप से OpenClaw संदर्भ को प्राप्त रूप में ही रखता है: अनुमति-सूचियाँ यह निर्धारित करती हैं कि कौन कार्रवाइयाँ सक्रिय कर सकता है, न कि मॉडल कौन-से उद्धृत या ऐतिहासिक अंश देखता है। पूरक संदर्भ भी फ़िल्टर करने के लिए `contextVisibility` सेट करें:

| मोड                | व्यवहार                                                                         |
| ------------------- | -------------------------------------------------------------------------------- |
| `"all"` (डिफ़ॉल्ट)   | पूरक संदर्भ को प्राप्त रूप में रखें।                                           |
| `"allowlist"`       | केवल अनुमति-सूची में शामिल प्रेषकों का इतिहास/थ्रेड/उद्धरण/अग्रेषित संदर्भ डालें।     |
| `"allowlist_quote"` | `allowlist`, साथ ही किसी भी प्रेषक के स्पष्ट रूप से उद्धृत/उत्तर दिए गए संदेश को रखें। |

इसे प्रति चैनल (`channels.<channel>.contextVisibility`), प्रति खाता (`channels.<channel>.accounts.<accountId>.contextVisibility`) या वैश्विक रूप से (`channels.defaults.contextVisibility`) सेट करें। पूरक संदर्भ प्राप्त करने वाले चैनल (Discord, Feishu, iMessage, Matrix, Microsoft Teams, Signal, Slack, Telegram, WhatsApp) इनबाउंड संदर्भ बनाते समय नीति लागू करते हैं; अज्ञात नीति संयोजन सुरक्षित रूप से बंद होते हैं और संदर्भ को छोड़ देते हैं।

यदि आप चाहते हैं...

| लक्ष्य                                         | क्या सेट करें                                                |
| -------------------------------------------- | ---------------------------------------------------------- |
| सभी समूहों को अनुमति दें, लेकिन केवल @mentions पर उत्तर दें | `groups: { "*": { requireMention: true } }`                |
| सभी समूह उत्तर अक्षम करें                    | `groupPolicy: "disabled"`                                  |
| केवल विशिष्ट समूह                         | `groups: { "<group-id>": { ... } }` (कोई `"*"` कुंजी नहीं)         |
| समूहों में केवल आप सक्रिय कर सकें               | `groupPolicy: "allowlist"`, `groupAllowFrom: ["+1555..."]` |
| सभी चैनलों में एक विश्वसनीय प्रेषक समूह का पुनः उपयोग करें | `groupAllowFrom: ["accessGroup:operators"]`                |

पुनः उपयोग योग्य प्रेषक अनुमति-सूचियों के लिए [पहुँच समूह](/hi/channels/access-groups) देखें।

## सेशन कुंजियाँ

- समूह सेशन `agent:<agentId>:<channel>:group:<id>` सेशन कुंजियों का उपयोग करते हैं (कक्ष/चैनल `agent:<agentId>:<channel>:channel:<id>` का उपयोग करते हैं)।
- Telegram फ़ोरम विषय समूह आईडी में `:topic:<threadId>` जोड़ते हैं, ताकि प्रत्येक विषय का अपना सेशन हो।
- प्रत्यक्ष चैट मुख्य सेशन का उपयोग करती हैं (या यदि `session.dmScope` कॉन्फ़िगर किया गया हो, तो प्रति-प्रेषक सेशन का)।
- Heartbeats कॉन्फ़िगर किए गए Heartbeat सेशन में चलते हैं (डिफ़ॉल्ट: एजेंट का मुख्य सेशन); समूह सेशन अपने Heartbeats नहीं चलाते।

## पैटर्न: व्यक्तिगत DM + सार्वजनिक समूह (एकल एजेंट)

हाँ—यदि आपका "व्यक्तिगत" ट्रैफ़िक **DM** है और "सार्वजनिक" ट्रैफ़िक **समूह** है, तो यह अच्छी तरह काम करता है।

कारण: एकल-एजेंट मोड में DM सामान्यतः **मुख्य** सेशन कुंजी (`agent:main:main`) में पहुँचते हैं, जबकि समूह हमेशा **गैर-मुख्य** सेशन कुंजियों (`agent:main:<channel>:group:<id>`) का उपयोग करते हैं। यदि आप `mode: "non-main"` के साथ सैंडबॉक्सिंग सक्षम करते हैं, तो वे समूह सेशन कॉन्फ़िगर किए गए सैंडबॉक्स बैकएंड में चलते हैं, जबकि आपका मुख्य DM सेशन होस्ट पर रहता है। यदि आप कोई बैकएंड नहीं चुनते, तो Docker डिफ़ॉल्ट बैकएंड है।

इससे आपको एक एजेंट "मस्तिष्क" (साझा वर्कस्पेस + मेमोरी), लेकिन निष्पादन की दो स्थितियाँ मिलती हैं:

- **DM**: पूर्ण टूल (होस्ट)
- **समूह**: सैंडबॉक्स + प्रतिबंधित टूल

<Note>
यदि आपको वास्तव में अलग-अलग वर्कस्पेस/व्यक्तित्व चाहिए ("व्यक्तिगत" और "सार्वजनिक" कभी मिश्रित नहीं होने चाहिए), तो दूसरे एजेंट + बाइंडिंग का उपयोग करें। [मल्टी-एजेंट रूटिंग](/hi/concepts/multi-agent) देखें।
</Note>

            mode: "non-main", // समूह/चैनल गैर-मुख्य हैं -> सैंडबॉक्स में
            scope: "session", // सबसे मजबूत पृथक्करण (प्रति समूह/चैनल एक कंटेनर)
            workspaceAccess: "none",
          },
        },
      },
      tools: {
        sandbox: {
          tools: {
            // यदि अनुमति-सूची खाली नहीं है, तो बाकी सब अवरुद्ध है (अस्वीकृति फिर भी प्राथमिक है)।
            allow: ["group:messaging", "group:sessions"],

## समूह नीति

प्रति चैनल समूह/कक्ष संदेशों के प्रबंधन का तरीका नियंत्रित करें:

```json5
{
  channels: {
    whatsapp: {
      groupPolicy: "disabled", // "open" | "disabled" | "allowlist"
      groupAllowFrom: ["+15551234567"],
    },
    telegram: {
      groupPolicy: "disabled",
      groupAllowFrom: ["123456789"], // संख्यात्मक Telegram उपयोगकर्ता आईडी (सेटअप @username का समाधान करता है)
    },
    signal: {
      groupPolicy: "disabled",
      groupAllowFrom: ["+15551234567"],
    },
    imessage: {
      groupPolicy: "disabled",
      groupAllowFrom: ["chat_id:123"],
    },
    msteams: {
      groupPolicy: "disabled",
      groupAllowFrom: ["user@org.com"],
    },
    discord: {
      groupPolicy: "allowlist",
      guilds: {
        GUILD_ID: { channels: { help: { enabled: true } } },
      },
    },
    slack: {
      groupPolicy: "allowlist",
      channels: { "#general": { enabled: true } },
    },
    matrix: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["@owner:example.org"],
      groups: {
        "!roomId:example.org": { enabled: true },
        "#alias:example.org": { enabled: true },
      },
    },
  },
}
```

| नीति        | व्यवहार                                                     |
| ------------- | ------------------------------------------------------------ |
| `"open"`      | समूह अनुमति-सूचियों को दरकिनार करते हैं; उल्लेख-गेटिंग फिर भी लागू रहती है।      |
| `"disabled"`  | सभी समूह संदेशों को पूरी तरह अवरुद्ध करें।                           |
| `"allowlist"` | केवल कॉन्फ़िगर की गई अनुमति-सूची से मेल खाने वाले समूहों/कक्षों को अनुमति दें। |

    - WhatsApp/Telegram/Signal/iMessage/Microsoft Teams/Zalo: `groupAllowFrom` का उपयोग करें (फ़ॉलबैक: स्पष्ट `allowFrom`)।
    - Signal: `groupAllowFrom` आने वाले Signal समूह आईडी या प्रेषक के फ़ोन/UUID में से किसी से भी मेल खा सकता है।
    - DM पेयरिंग स्वीकृतियाँ (`*-allowFrom` स्टोर प्रविष्टियाँ) केवल DM पहुँच पर लागू होती हैं; समूह प्रेषक प्राधिकरण समूह अनुमति-सूचियों में स्पष्ट ही रहता है।
    - Discord: अनुमति-सूची `channels.discord.guilds.<id>.channels` का उपयोग करती है।
    - Slack: अनुमति-सूची `channels.slack.channels` का उपयोग करती है।
    - Matrix: अनुमति-सूची `channels.matrix.groups` का उपयोग करती है। कक्ष आईडी (`!room:server`) या उपनाम (`#alias:server`) का उपयोग करें; कक्ष-नाम कुंजियाँ केवल `channels.matrix.dangerouslyAllowNameMatching: true` के साथ मेल खाती हैं, और अनसुलझी प्रविष्टियों को रनटाइम पर अनदेखा किया जाता है। प्रेषकों को प्रतिबंधित करने के लिए `channels.matrix.groupAllowFrom` का उपयोग करें; प्रति-कक्ष `users` अनुमति-सूचियाँ भी समर्थित हैं।
    - समूह DM अलग से नियंत्रित किए जाते हैं (`channels.discord.dm.*`, `channels.slack.dm.*`: `groupEnabled`, `groupChannels`)।
    - Telegram: प्रेषक अनुमति-सूचियाँ केवल संख्यात्मक उपयोगकर्ता आईडी स्वीकार करती हैं (`"123456789"`; `telegram:`/`tg:` उपसर्गों को अक्षर-स्थिति की परवाह किए बिना हटा दिया जाता है)। `@username` प्रविष्टियाँ रनटाइम पर मेल नहीं खातीं और चेतावनी लॉग करती हैं; सेटअप `@username` को आईडी में बदलता है। ऋणात्मक चैट आईडी प्रेषक अनुमति-सूचियों में नहीं, बल्कि `channels.telegram.groups` के अंतर्गत होनी चाहिए।
    - डिफ़ॉल्ट `groupPolicy: "allowlist"` है; यदि आपकी समूह अनुमति-सूची खाली है, तो समूह संदेश अवरुद्ध कर दिए जाते हैं।
    - रनटाइम सुरक्षा: जब कोई प्रदाता ब्लॉक पूरी तरह अनुपस्थित हो (`channels.<provider>` अनुपस्थित), तो समूह नीति `channels.defaults.groupPolicy` को विरासत में लेने के बजाय सुरक्षित रूप से `allowlist` पर बंद हो जाती है, और Gateway प्रति खाते एक बार फ़ॉलबैक लॉग करता है।

## उल्लेख-गेटिंग (डिफ़ॉल्ट)

समूह संदेशों के लिए उल्लेख आवश्यक है, जब तक कि इसे प्रति समूह ओवरराइड न किया गया हो। डिफ़ॉल्ट प्रत्येक उपतंत्र में `*.groups."*"` के अंतर्गत होते हैं।

समर्थित अप्रत्यक्ष उल्लेख तथ्य चैनल-विशिष्ट हैं:

| तथ्य                  | वर्तमान अंतर्निहित उत्पादक                       |
| --------------------- | ------------------------------------------------ |
| बॉट को उत्तर      | Discord, Microsoft Teams, QQBot, Slack, Telegram |
| बॉट का उद्धरण      | WhatsApp, Zalo व्यक्तिगत                          |
| बॉट थ्रेड में शामिल हुआ | Mattermost, Slack, Tlon                          |

जब चैनल किसी तथ्य को उत्पन्न करता है, तो वह तथ्य डिफ़ॉल्ट रूप से सक्षम होता है। उस तथ्य को उल्लेख-गेटिंग दरकिनार करने से रोकने के लिए संबंधित `implicitMentions` फ़्लैग को `false` पर सेट करें; मूल स्पष्ट उल्लेख अप्रभावित रहते हैं। जो चैनल वह तथ्य उत्पन्न नहीं करते, उन पर फ़्लैग का कोई प्रभाव नहीं पड़ता।

```json5
{
  channels: {
    whatsapp: {
      groups: {
        "*": { requireMention: true },
        "123@g.us": { requireMention: false },
      },
    },
    telegram: {
      groups: {
        "*": { requireMention: true },
        "123456789": { requireMention: false },
      },
    },
    imessage: {
      groups: {
        "*": { requireMention: true },
        "123": { requireMention: false },
      },
    },
  },
  agents: {
    list: [
      {
        id: "main",
        groupChat: {
          mentionPatterns: ["@openclaw", "openclaw", "\\+15555550123"],
          historyLimit: 50,
        },
      },
    ],
  },
}
```

## कॉन्फ़िगर किए गए उल्लेख पैटर्न का दायरा

कॉन्फ़िगर किए गए `mentionPatterns` रेगेक्स फ़ॉलबैक ट्रिगर हैं। उनका उपयोग तब करें जब
प्लेटफ़ॉर्म मूल बॉट उल्लेख उपलब्ध नहीं कराता, या जब आप चाहते हैं कि `openclaw:`
जैसा सादा पाठ उल्लेख माना जाए। मूल प्लेटफ़ॉर्म उल्लेख अलग होते हैं:
जब Discord, Slack, Telegram, Matrix, Signal या कोई अन्य चैनल प्रमाणित कर सकता है कि संदेश में
बॉट का स्पष्ट रूप से उल्लेख किया गया है, तो कॉन्फ़िगर किए गए रेगेक्स पैटर्न अस्वीकृत होने पर भी
वह मूल उल्लेख ट्रिगर होता है।

डिफ़ॉल्ट रूप से, कॉन्फ़िगर किए गए उल्लेख पैटर्न हर उस स्थान पर लागू होते हैं जहाँ चैनल प्रदाता और वार्तालाप तथ्यों को उल्लेख पहचान में भेजता है। व्यापक पैटर्न को प्रत्येक समूह में एजेंट सक्रिय करने से रोकने के लिए, `channels.<channel>.mentionPatterns` के साथ प्रति चैनल उनका दायरा निर्धारित करें।

जब किसी चैनल के लिए रेगेक्स उल्लेख पैटर्न डिफ़ॉल्ट रूप से बंद होने चाहिए, तब `mode: "deny"` का उपयोग करें, फिर `allowIn` के साथ विशिष्ट कक्षों को स्पष्ट रूप से शामिल करें:

```json5
{
  messages: {
    groupChat: {
      mentionPatterns: ["\\bopenclaw\\b", "\\bops bot\\b"],
    },
  },
  channels: {
    slack: {
      mentionPatterns: {
        mode: "deny",
        allowIn: ["C0123OPS"],
      },
    },
  },
}
```

जब रेगेक्स उल्लेख पैटर्न व्यापक रूप से लागू होने चाहिए, तब डिफ़ॉल्ट `mode: "allow"` का उपयोग करें (या `mode` को छोड़ दें), फिर `denyIn` के साथ शोरयुक्त कक्षों में उन्हें बंद करें:

```json5
{
  messages: {
    groupChat: {
      mentionPatterns: ["\\bopenclaw\\b"],
    },
  },
  channels: {
    telegram: {
      mentionPatterns: {
        denyIn: ["-1001234567890", "-1001234567890:topic:42"],
      },
    },
  },
}
```

नीति समाधान:

| फ़ील्ड           | प्रभाव                                                                                                                |
| --------------- | --------------------------------------------------------------------------------------------------------------------- |
| `mode: "allow"` | रेगेक्स उल्लेख पैटर्न तब तक सक्षम रहते हैं, जब तक वार्तालाप आईडी `denyIn` में न हो। यह डिफ़ॉल्ट है।                    |
| `mode: "deny"`  | रेगेक्स उल्लेख पैटर्न तब तक अक्षम रहते हैं, जब तक वार्तालाप आईडी `allowIn` में न हो।                                       |
| `allowIn`       | वे वार्तालाप आईडी जहाँ अस्वीकार मोड में रेगेक्स उल्लेख पैटर्न सक्षम हैं।                                               |
| `denyIn`        | वे वार्तालाप आईडी जहाँ रेगेक्स उल्लेख पैटर्न अक्षम हैं। यदि दोनों में समान आईडी शामिल हो, तो `denyIn`, `allowIn` पर प्राथमिकता पाता है। |

वर्तमान में समर्थित दायरा-बद्ध रेगेक्स नीति:

| चैनल  | `allowIn` / `denyIn` में उपयोग किए गए आईडी                             |
| -------- | ------------------------------------------------------------ |
| Discord  | Discord चैनल आईडी।                                         |
| Matrix   | Matrix कक्ष आईडी।                                             |
| Slack    | Slack चैनल आईडी।                                           |
| Telegram | समूह चैट आईडी, या फ़ोरम विषयों के लिए `chatId:topic:threadId`। |
| WhatsApp | `123@g.us` जैसे WhatsApp वार्तालाप आईडी।                |

जब कोई चैनल एकाधिक खातों का समर्थन करता है, तब खाता-स्तरीय चैनल कॉन्फ़िगरेशन `channels.<channel>.accounts.<accountId>.mentionPatterns` के अंतर्गत वही नीति सेट कर सकते हैं। उस खाते के लिए खाता नीति शीर्ष-स्तरीय चैनल नीति पर प्राथमिकता पाती है।

<AccordionGroup>
  <Accordion title="उल्लेख-गेटिंग टिप्पणियाँ">
    - `mentionPatterns` अक्षर-स्थिति से स्वतंत्र सुरक्षित रेगेक्स पैटर्न हैं; अमान्य पैटर्न और असुरक्षित नेस्टेड-पुनरावृत्ति रूपों को चेतावनी के साथ अनदेखा कर दिया जाता है।
    - पैटर्न प्राथमिकता: `agents.list[].groupChat.mentionPatterns` (जब कई एजेंट एक समूह साझा करते हैं तब उपयोगी) `messages.groupChat.mentionPatterns` को ओवरराइड करता है; जब दोनों में से कोई सेट न हो, तो पैटर्न एजेंट पहचान के नाम/इमोजी से निकाले जाते हैं।
    - उल्लेख-गेटिंग केवल तभी लागू की जाती है जब उल्लेख पहचान संभव हो (मूल उल्लेख या `mentionPatterns` कॉन्फ़िगर किए गए हों)।
    - किसी समूह या प्रेषक को अनुमति-सूची में रखने से उल्लेख-गेटिंग अक्षम नहीं होती; जब सभी संदेशों को ट्रिगर होना चाहिए, तब उस समूह के `requireMention` को `false` पर सेट करें।
    - स्वचालित समूह चैट प्रॉम्प्ट संदर्भ हर चरण में समाधान किया गया मौन-उत्तर निर्देश रखता है; वर्कस्पेस फ़ाइलों को `NO_REPLY` तंत्र की नकल नहीं करनी चाहिए।
    - वे समूह जहाँ स्वचालित मौन उत्तरों की अनुमति है, साफ़ खाली या केवल तर्क वाले मॉडल चरणों को `NO_REPLY` के समकक्ष मौन मानते हैं। प्रत्यक्ष चैट को कभी भी `NO_REPLY` मार्गदर्शन नहीं मिलता, और केवल संदेश-टूल वाले समूह उत्तर `message(action=send)` को कॉल न करके मौन रहते हैं।
    - परिवेशी सदैव-सक्रिय समूह वार्तालाप डिफ़ॉल्ट रूप से उपयोगकर्ता-अनुरोध शब्दार्थ का उपयोग करता है। इसके बजाय इसे शांत संदर्भ के रूप में सबमिट करने के लिए `messages.groupChat.unmentionedInbound: "room_event"` सेट करें। सेटअप उदाहरणों के लिए [परिवेशी कक्ष घटनाएँ](/hi/channels/ambient-room-events) देखें।
    - कक्ष घटनाओं को नकली उपयोगकर्ता अनुरोधों के रूप में संग्रहीत नहीं किया जाता, और संदेश-टूल रहित कक्ष घटनाओं का निजी सहायक पाठ चैट इतिहास के रूप में पुनः नहीं चलाया जाता।
    - Discord डिफ़ॉल्ट `channels.discord.guilds."*"` में होते हैं (प्रति गिल्ड/चैनल ओवरराइड किए जा सकते हैं)।
    - समूह इतिहास संदर्भ सभी चैनलों में एकसमान रूप से लपेटा जाता है। उल्लेख-गेट वाले समूह लंबित छोड़े गए संदेश बनाए रखते हैं; सदैव-सक्रिय समूह, चैनल द्वारा समर्थित होने पर, हाल ही में संसाधित कक्ष संदेश भी बनाए रख सकते हैं। वैश्विक डिफ़ॉल्ट के लिए `messages.groupChat.historyLimit` और ओवरराइड के लिए `channels.<channel>.historyLimit` (या `channels.<channel>.accounts.*.historyLimit`) का उपयोग करें। अक्षम करने के लिए `0` सेट करें।

  </Accordion>
</AccordionGroup>

## समूह/चैनल टूल प्रतिबंध (वैकल्पिक)

कुछ चैनल कॉन्फ़िगरेशन यह सीमित करने का समर्थन करते हैं कि **किसी विशिष्ट समूह/कक्ष/चैनल के भीतर** कौन-से टूल उपलब्ध हैं।

- `tools`: पूरे समूह के लिए टूल को अनुमति दें/अस्वीकार करें (`allow`, `alsoAllow`, `deny`; अस्वीकार को प्राथमिकता मिलती है)।
- `toolsBySender`: समूह के भीतर प्रति-प्रेषक ओवरराइड। स्पष्ट कुंजी उपसर्गों का उपयोग करें: `channel:<channelId>:<senderId>`, `id:<senderId>`, `e164:<phone>`, `username:<handle>`, `name:<displayName>`, और `"*"` वाइल्डकार्ड। चैनल आईडी मानक OpenClaw चैनल आईडी का उपयोग करते हैं; `teams` जैसे उपनाम `msteams` में सामान्यीकृत होते हैं। पुराने उपसर्ग-रहित कुंजी अभी भी स्वीकार किए जाते हैं, केवल `id:` के रूप में मिलान किए जाते हैं, और अप्रचलन चेतावनी लॉग करते हैं।

समाधान क्रम (सबसे विशिष्ट को प्राथमिकता):

<Steps>
  <Step title="समूह toolsBySender">
    समूह/चैनल `toolsBySender` मिलान।
  </Step>
  <Step title="समूह tools">
    समूह/चैनल `tools`।
  </Step>
  <Step title="डिफ़ॉल्ट toolsBySender">
    डिफ़ॉल्ट (`"*"`) `toolsBySender` मिलान।
  </Step>
  <Step title="डिफ़ॉल्ट tools">
    डिफ़ॉल्ट (`"*"`) `tools`।
  </Step>
</Steps>

उदाहरण (Telegram):

```json5
{
  channels: {
    telegram: {
      groups: {
        "*": { tools: { deny: ["exec"] } },
        "-1001234567890": {
          tools: { deny: ["exec", "read", "write"] },
          toolsBySender: {
            "id:123456789": { alsoAllow: ["exec"] },
          },
        },
      },
    },
  },
}
```

<Note>
समूह/चैनल टूल प्रतिबंध वैश्विक/एजेंट टूल नीति के अतिरिक्त लागू किए जाते हैं (अस्वीकृति की प्राथमिकता बनी रहती है)। कुछ चैनल कक्षों/चैनलों के लिए अलग नेस्टिंग का उपयोग करते हैं (उदा., Discord `guilds.*.channels.*`, Slack `channels.*`, Microsoft Teams `teams.*.channels.*`)।
</Note>

## समूह अनुमति-सूचियाँ

जब `channels.whatsapp.groups`, `channels.telegram.groups`, या `channels.imessage.groups` कॉन्फ़िगर किया जाता है, तो कुंजियाँ समूह अनुमति-सूची के रूप में कार्य करती हैं। डिफ़ॉल्ट उल्लेख व्यवहार सेट करते हुए भी सभी समूहों को अनुमति देने के लिए `"*"` का उपयोग करें।

<Warning>
सामान्य भ्रम: DM पेयरिंग की स्वीकृति समूह प्राधिकरण के समान नहीं है। DM पेयरिंग का समर्थन करने वाले चैनलों में, पेयरिंग स्टोर केवल DM को अनलॉक करता है। समूह कमांड के लिए अब भी `groupAllowFrom` जैसी कॉन्फ़िगरेशन अनुमति-सूचियों या उस चैनल के दस्तावेज़ीकृत कॉन्फ़िगरेशन फ़ॉलबैक से स्पष्ट समूह प्रेषक प्राधिकरण आवश्यक है।
</Warning>

सामान्य उद्देश्य (कॉपी/पेस्ट करें):

<Tabs>
  <Tab title="समूह के सभी उत्तर अक्षम करें">
    ```json5
    {
      channels: { whatsapp: { groupPolicy: "disabled" } },
    }
    ```
  </Tab>
  <Tab title="केवल विशिष्ट समूहों को अनुमति दें (WhatsApp)">
    ```json5
    {
      channels: {
        whatsapp: {
          groups: {
            "123@g.us": { requireMention: true },
            "456@g.us": { requireMention: false },
          },
        },
      },
    }
    ```
  </Tab>
  <Tab title="सभी समूहों को अनुमति दें, लेकिन उल्लेख आवश्यक रखें">
    ```json5
    {
      channels: {
        whatsapp: {
          groups: { "*": { requireMention: true } },
        },
      },
    }
    ```
  </Tab>
  <Tab title="केवल स्वामी के ट्रिगर (WhatsApp)">
    ```json5
    {
      channels: {
        whatsapp: {
          groupPolicy: "allowlist",
          groupAllowFrom: ["+15551234567"],
          groups: { "*": { requireMention: true } },
        },
      },
    }
    ```
  </Tab>
</Tabs>

## सक्रियण (केवल स्वामी)

समूह के स्वामी एक स्वतंत्र संदेश से प्रत्येक समूह का सक्रियण टॉगल कर सकते हैं:

- `/activation mention`
- `/activation always`

`/activation` एक मुख्य, स्वामी-नियंत्रित कमांड है और केवल समूह चैट में लागू होता है। स्वामी का अर्थ है कि प्रेषक `commands.ownerAllowFrom` से मेल खाता है; चैनल की `allowFrom` सूचियाँ केवल सामान्य चैनल और कमांड पहुँच नियंत्रित करती हैं। जो चैनल संग्रहीत मोड का उपयोग करते हैं (Google Chat, QQBot, Telegram, WhatsApp), उनमें यह मोड उस समूह के `requireMention` को ओवरराइड करता है, और समूह के सिस्टम-प्रॉम्प्ट का परिचय हर जगह सक्रिय मोड दर्शाता है।

## संदर्भ फ़ील्ड

समूह के इनबाउंड पेलोड ये सेट करते हैं:

- `ChatType=group`
- `GroupSubject` (यदि ज्ञात हो)
- `GroupMembers` (यदि ज्ञात हो)
- `WasMentioned` (उल्लेख गेटिंग का परिणाम)
- Telegram फ़ोरम विषयों में `MessageThreadId` और `IsForum` भी शामिल होते हैं।

एजेंट सिस्टम प्रॉम्प्ट में नए समूह सत्र की पहली बारी पर (और `/activation` बदलने के बाद) समूह का परिचय शामिल होता है। यह मॉडल को मनुष्य की तरह उत्तर देने, खाली पंक्तियाँ कम रखने और सामान्य चैट अंतराल का पालन करने तथा शाब्दिक `\n` अनुक्रम टाइप करने से बचने की याद दिलाता है। जिन चैनलों का घोषित तालिका मोड नेटिव या रॉ तालिकाओं को संरक्षित नहीं करता, वे Markdown तालिकाओं को भी हतोत्साहित करते हैं। चैनल से प्राप्त समूह नाम और प्रतिभागी लेबल इनलाइन सिस्टम निर्देशों के बजाय फ़ेंस किए गए अविश्वसनीय मेटाडेटा के रूप में रेंडर किए जाते हैं।

## iMessage की विशिष्टताएँ

- रूटिंग या अनुमति-सूची बनाते समय `chat_id:<id>` को प्राथमिकता दें।
- चैट सूचीबद्ध करें: `imsg chats --limit 20`।
- समूह के उत्तर हमेशा उसी `chat_id` पर वापस जाते हैं।
