स्थिति: WhatsApp Web (Baileys) के माध्यम से उत्पादन के लिए तैयार। Gateway लिंक किए गए सत्र का स्वामी होता है; कोई अलग Twilio WhatsApp चैनल नहीं है।

## इंस्टॉल करना

`openclaw onboard` और `openclaw channels add --channel whatsapp` पहली बार Plugin चुनने पर उसे इंस्टॉल करने का संकेत देते हैं; Plugin अनुपलब्ध होने पर `openclaw channels login --channel whatsapp` वही इंस्टॉल प्रक्रिया प्रदान करता है। डेवलपमेंट चेकआउट स्थानीय Plugin पथ का उपयोग करते हैं; स्थिर/बीटा इंस्टॉल पहले ClawHub से `@openclaw/whatsapp` इंस्टॉल करते हैं और असफल होने पर npm का उपयोग करते हैं। WhatsApp रनटाइम मुख्य OpenClaw npm पैकेज के बाहर वितरित होता है, इसलिए इसकी रनटाइम निर्भरताएँ बाहरी Plugin के साथ रहती हैं। मैन्युअल इंस्टॉल:

```bash
openclaw plugins install clawhub:@openclaw/whatsapp
```

साधारण npm पैकेज (`@openclaw/whatsapp`) का उपयोग केवल रजिस्ट्री फ़ॉलबैक के लिए करें; पुनरुत्पाद्य इंस्टॉल के लिए ही सटीक संस्करण पिन करें।

## रनटाइम मॉडल

- Gateway WhatsApp सॉकेट और पुनः कनेक्ट करने वाले लूप का स्वामी होता है।
- एक वॉचडॉग दो संकेतों को स्वतंत्र रूप से ट्रैक करता है: कच्ची WhatsApp Web ट्रांसपोर्ट गतिविधि और एप्लिकेशन-संदेश गतिविधि। किसी शांत लेकिन कनेक्टेड सत्र को केवल इसलिए पुनः शुरू नहीं किया जाता कि हाल ही में कोई संदेश नहीं आया; यह केवल तभी पुनः कनेक्शन बाध्य करता है जब एक निश्चित आंतरिक अवधि (उपयोगकर्ता द्वारा कॉन्फ़िगर करने योग्य नहीं) तक ट्रांसपोर्ट फ़्रेम आना बंद हो जाएँ या एप्लिकेशन संदेश सामान्य संदेश टाइमआउट की 4x अवधि तक शांत रहें। हाल ही में सक्रिय रहे सत्र के पुनः कनेक्ट होने के तुरंत बाद, पहली अवधि 4x अवधि के बजाय छोटे सामान्य संदेश टाइमआउट का उपयोग करती है। उस पुनः कनेक्शन के आरंभ में Baileys द्वारा दिए गए ऑफ़लाइन संदेशों का OpenClaw स्वतः उत्तर दे सकता है, जिसकी सीमा आने वाले संदेश-ID की डीडुप्लिकेशन अवधि से निर्धारित होती है; प्रारंभिक स्टार्टअप में छोटी पुराने-इतिहास सुरक्षा बनी रहती है।
- Baileys सॉकेट समय-सीमाएँ `web.whatsapp.*` के अंतर्गत स्पष्ट रूप से निर्धारित हैं: `keepAliveIntervalMs` (एप्लिकेशन पिंग अंतराल), `connectTimeoutMs` (आरंभिक हैंडशेक टाइमआउट), `defaultQueryTimeoutMs` (Baileys क्वेरी प्रतीक्षा, साथ ही OpenClaw के आउटबाउंड प्रेषण/उपस्थिति और इनबाउंड पठन-प्राप्ति टाइमआउट)।
- आउटबाउंड प्रेषण के लिए लक्षित खाते का सक्रिय WhatsApp लिसनर आवश्यक है; अन्यथा प्रेषण तुरंत विफल हो जाता है।
- समूह में भेजे जाने वाले संदेश `@+<digits>` और `@<digits>` टोकन के लिए मूल उल्लेख मेटाडेटा जोड़ते हैं (टेक्स्ट और मीडिया कैप्शन में), जब टोकन वर्तमान प्रतिभागी मेटाडेटा से मेल खाता हो; इसमें LID-समर्थित समूह भी शामिल हैं।
- स्थिति और ब्रॉडकास्ट चैट (`@status`, `@broadcast`) को अनदेखा किया जाता है।
- प्रत्यक्ष चैट DM सत्र नियमों का उपयोग करती हैं (`session.dmScope`; डिफ़ॉल्ट `main` DM को एजेंट के मुख्य सत्र में समेट देता है)। समूह सत्रों को प्रत्येक JID के अनुसार अलग रखा जाता है (`agent:<agentId>:whatsapp:group:<jid>`)।
- WhatsApp चैनल/न्यूज़लेटर अपने मूल `@newsletter` JID के माध्यम से स्पष्ट आउटबाउंड लक्ष्य हो सकते हैं, जिनमें DM अर्थविज्ञान के बजाय चैनल सत्र मेटाडेटा (`agent:<agentId>:whatsapp:channel:<jid>`) का उपयोग होता है।
- WhatsApp Web ट्रांसपोर्ट Gateway होस्ट पर मानक प्रॉक्सी पर्यावरण चर (`HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY`, छोटे अक्षरों वाले रूप) का पालन करता है। प्रत्येक चैनल की सेटिंग के बजाय होस्ट-स्तरीय प्रॉक्सी कॉन्फ़िगरेशन को प्राथमिकता दें।
- `messages.removeAckAfterReply` सक्षम होने पर, दृश्य उत्तर वितरित हो जाने के बाद OpenClaw स्वीकृति प्रतिक्रिया हटा देता है।

## MeowCaller से वर्तमान अनुरोधकर्ता को कॉल करें (प्रयोगात्मक)

Plugin WhatsApp से शुरू हुए एजेंट टर्न में `whatsapp_call` उपलब्ध करा सकता है। यह वर्तमान अधिकृत अनुरोधकर्ता को WhatsApp वॉइस कॉल करने और उनके उत्तर देने के बाद OpenClaw TTS संदेश चलाने के लिए [MeowCaller](https://github.com/purpshell/meowcaller) का उपयोग करता है। टूल में गंतव्य नंबर का कोई पैरामीटर नहीं है, इसलिए कोई प्रॉम्प्ट कॉल को पुनर्निर्देशित नहीं कर सकता। डिफ़ॉल्ट रूप से अक्षम।

<Warning>
MeowCaller प्रयोगात्मक है, इसका कोई टैग किया गया रिलीज़ नहीं है और यह अलग से पेयर किए गए whatsmeow लिंक्ड-डिवाइस सत्र का उपयोग करता है—यह Plugin के Baileys क्रेडेंशियल्स का पुनः उपयोग नहीं कर सकता। पेयरिंग उसी WhatsApp खाते में एक और लिंक्ड डिवाइस जोड़ती है; OpenClaw द्वारा उपयोग की जाने वाली पहचान से स्कैन करें। व्यक्तिगत नंबर/स्वयं से चैट मोड स्वयं को कॉल नहीं कर सकता; अपने व्यक्तिगत नंबर पर कॉल करने के लिए समर्पित OpenClaw नंबर का उपयोग करें।
</Warning>

<Steps>
  <Step title="प्रयोगात्मक कॉल सक्षम करें">

    WhatsApp चैनल कॉन्फ़िगरेशन में `actions.calls: true` जोड़ें और Gateway पुनः शुरू करें:

```json
{
  "channels": {
    "whatsapp": {
      "actions": {
        "calls": true
      }
    }
  }
}
```

    अनुपस्थित या `false` होने पर, OpenClaw `whatsapp_call` टूल उपलब्ध नहीं कराता।

  </Step>

  <Step title="समीक्षित MeowCaller CLI इंस्टॉल करें">

    अडैप्टर को Gateway होस्ट के `PATH` में `meowcaller` एक्ज़ीक्यूटेबल अपेक्षित है। [MeowCaller PR #7](https://github.com/purpshell/meowcaller/pull/7) के मर्ज होने तक समीक्षित ब्रांच बनाएँ:

```bash
git clone --branch feat/send-only-notify https://github.com/steipete/meowcaller.git
cd meowcaller
git checkout 752050471fc2bf7a8cdfbf7dbd3cd4e865d85d3f
mkdir -p "$HOME/.local/bin"
go build -o "$HOME/.local/bin/meowcaller" ./cmd/meowcaller
```

    सुनिश्चित करें कि `$HOME/.local/bin` Gateway सेवा के `PATH` में है। इस संशोधन में स्पष्ट `pair` और केवल-प्रेषण `notify` कमांड हैं; `notify` कोई माइक्रोफ़ोन, स्पीकर, वीडियो डिवाइस या नैदानिक कैप्चर नहीं खोलता। अपस्ट्रीम उदाहरण CLI के `play` कमांड का विकल्प के रूप में उपयोग न करें।

  </Step>

  <Step title="MeowCaller लिंक्ड डिवाइस को पेयर करें">

    WhatsApp एजेंट से कॉल सेटअप की जाँच करने को कहें (`whatsapp_call` स्थिति कार्रवाई खाते के लिए विशिष्ट स्टेट डायरेक्टरी और पेयरिंग कमांड बताती है)। डिफ़ॉल्ट खाते के लिए:

```bash
state_dir="$HOME/.openclaw/credentials/whatsapp-calls/default"
mkdir -p "$state_dir"
chmod 700 "$state_dir"
meowcaller pair --store "$state_dir/wa-voip.db"
```

    इसे इंटरैक्टिव रूप से चलाएँ, **WhatsApp > Linked devices** से QR स्कैन करें और `MeowCaller linked device ready` की प्रतीक्षा करें। `wa-voip.db` को निजी रखें—यह MeowCaller सत्र है। गैर-डिफ़ॉल्ट खातों को स्थिति कार्रवाई से अपना स्टोर पथ मिलता है; Windows पर उसका PowerShell कमांड चलाएँ।

  </Step>

  <Step title="TTS कॉन्फ़िगर करें और WhatsApp से कॉल करें">

    टेलीफ़ोनी-सक्षम [TTS प्रदाता](/hi/tools/tts) कॉन्फ़िगर करें, Gateway पुनः शुरू करें, फिर `Call me and say the build finished.` जैसा अनुरोध भेजें। टूल विश्वसनीय इनबाउंड संदर्भ से प्रेषक का निर्धारण करता है, एक अस्थायी निजी WAV फ़ाइल संश्लेषित करता है, सीमित कॉल अवधि के लिए MeowCaller चलाता है और बाद में ऑडियो फ़ाइल हटा देता है। OpenClaw खाते का स्टोर स्पष्ट रूप से पास करता है, उत्तर/प्लेबैक/कॉल समाप्ति के बाद शून्य एग्ज़िट स्थिति की प्रतीक्षा करता है और टाइमआउट या गैर-शून्य एग्ज़िट को विफल टूल कॉल मानता है।

  </Step>
</Steps>

सीमाएँ: केवल एक-से-एक आउटबाउंड ऑडियो कॉल, कोई मनमाना गंतव्य नंबर नहीं, चैट कनेक्शन के साथ कोई साझा प्रमाणीकरण नहीं, व्यक्तिगत नंबर/स्वयं से चैट मोड से स्वयं को कॉल नहीं, संश्लेषित ऑडियो की सीमा 60 सेकंड, MeowCaller के उत्तर/प्लेबैक/कॉल समाप्ति पूर्ण होने के अतिरिक्त हैंडसेट पर सुनाई देने की कोई प्राप्ति पुष्टि नहीं, और OpenClaw सहायक प्रक्रिया को सीमित 115-175 सेकंड की अवधि के बाद रोक देता है (जिसमें MeowCaller के कनेक्शन, उत्तर, प्लेबैक और शटडाउन चरण शामिल हैं)।

## स्वीकृति प्रॉम्प्ट

WhatsApp exec और Plugin स्वीकृति प्रॉम्प्ट को `👍`/`👎` प्रतिक्रियाओं के रूप में रेंडर कर सकता है, जिन्हें शीर्ष-स्तरीय स्वीकृति फ़ॉरवर्डिंग कॉन्फ़िगरेशन नियंत्रित करता है:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "session",
    },
    plugin: {
      enabled: true,
      mode: "targets",
      targets: [{ channel: "whatsapp", to: "+15551234567" }],
    },
  },
}
```

`approvals.exec` और `approvals.plugin` स्वतंत्र हैं; WhatsApp को चैनल के रूप में सक्षम करने पर केवल ट्रांसपोर्ट लिंक होता है और तब तक कुछ नहीं भेजा जाता जब तक संबंधित स्वीकृति परिवार सक्षम न हो और वहाँ रूट न किया गया हो। सत्र मोड केवल WhatsApp से उत्पन्न स्वीकृतियों के लिए मूल इमोजी स्वीकृतियाँ वितरित करता है। लक्ष्य मोड स्पष्ट लक्ष्यों के लिए साझा फ़ॉरवर्डिंग पाइपलाइन का उपयोग करता है और अलग स्वीकर्ता-DM फ़ैनआउट नहीं बनाता।

WhatsApp स्वीकृति प्रतिक्रियाओं के लिए `allowFrom` (या `"*"`) में स्पष्ट स्वीकर्ता आवश्यक हैं। `defaultTo` सामान्य डिफ़ॉल्ट संदेश लक्ष्य निर्धारित करता है, स्वीकर्ताओं की सूची नहीं। मैन्युअल `/approve` कमांड स्वीकृति निर्धारण से पहले भी सामान्य WhatsApp प्रेषक-प्राधिकरण पथ से गुजरते हैं।

## Plugin हुक और गोपनीयता

इनबाउंड WhatsApp संदेशों में व्यक्तिगत सामग्री, फ़ोन नंबर, समूह पहचानकर्ता, प्रेषक के नाम और सत्र सहसंबंध फ़ील्ड हो सकते हैं। जब तक आप स्पष्ट रूप से अनुमति नहीं देते, WhatsApp इनबाउंड `message_received` हुक पेलोड को Plugin में प्रसारित नहीं करता:

```json5
{
  channels: {
    whatsapp: {
      pluginHooks: {
        messageReceived: true,
      },
    },
  },
}
```

इस अनुमति को `channels.whatsapp.accounts.<id>.pluginHooks.messageReceived` के अंतर्गत एक खाते तक सीमित करें। इसे केवल उन Plugin के लिए सक्षम करें जिन पर आप इनबाउंड WhatsApp सामग्री और पहचानकर्ताओं के संबंध में भरोसा करते हैं।

## पहुँच नियंत्रण और सक्रियण

<Tabs>
  <Tab title="DM नीति">
    `channels.whatsapp.dmPolicy`:

    | मान | व्यवहार |
    | --- | --- |
    | `pairing` (डिफ़ॉल्ट) | अज्ञात प्रेषक पेयरिंग का अनुरोध करते हैं; स्वामी स्वीकृति देता है |
    | `allowlist` | केवल `allowFrom` प्रेषकों को प्रवेश मिलता है |
    | `open` | `allowFrom` में `"*"` शामिल होना आवश्यक है |
    | `disabled` | सभी DM अवरुद्ध करें |

    `allowFrom` E.164-शैली के नंबर स्वीकार करता है (आंतरिक रूप से सामान्यीकृत)। यह केवल DM प्रेषक की पहुँच-नियंत्रण सूची है—यह समूह JID या `@newsletter` चैनल JID को स्पष्ट आउटबाउंड प्रेषण से नहीं रोकती।

    बहु-खाता ओवरराइड: `channels.whatsapp.accounts.<id>.dmPolicy` (और `.allowFrom`) उस खाते के लिए चैनल-स्तरीय डिफ़ॉल्ट पर प्राथमिकता लेते हैं।

    रनटाइम टिप्पणियाँ:

    - पेयरिंग चैनल की अनुमति-स्टोर में बनी रहती हैं और कॉन्फ़िगर किए गए `allowFrom` के साथ मर्ज होती हैं
    - शेड्यूल किया गया ऑटोमेशन और Heartbeat प्राप्तकर्ता फ़ॉलबैक स्पष्ट डिलीवरी लक्ष्यों या कॉन्फ़िगर किए गए `allowFrom` का उपयोग करते हैं; DM पेयरिंग स्वीकृतियाँ अंतर्निहित Cron/Heartbeat प्राप्तकर्ता नहीं होतीं
    - यदि कोई अनुमति-सूची कॉन्फ़िगर नहीं है, तो लिंक किया गया स्वयं का नंबर डिफ़ॉल्ट रूप से अनुमत होता है
    - OpenClaw आउटबाउंड `fromMe` DM (लिंक किए गए डिवाइस से स्वयं को भेजे गए संदेश) को कभी स्वचालित रूप से पेयर नहीं करता

  </Tab>

  <Tab title="समूह नीति और अनुमति-सूचियाँ">
    समूह एक्सेस की दो परतें हैं:

    1. **समूह सदस्यता अनुमति-सूची** (`channels.whatsapp.groups`): यदि `groups` छोड़ दिया गया है, तो सभी समूह पात्र होते हैं; यदि मौजूद है, तो यह समूह अनुमति-सूची के रूप में कार्य करता है (`"*"` सभी को प्रवेश देता है)।
    2. **समूह प्रेषक नीति** (`channels.whatsapp.groupPolicy` + `groupAllowFrom`): `open` प्रेषक अनुमति-सूची को बायपास करता है, `allowlist` को `groupAllowFrom` (या `*`) से मिलान आवश्यक है, `disabled` समूह से आने वाले सभी संदेशों को अवरुद्ध करता है।

    यदि `groupAllowFrom` सेट नहीं है, तो प्रविष्टियाँ होने पर प्रेषक जाँच `allowFrom` पर फ़ॉलबैक करती है। उल्लेख/उत्तर सक्रियण से पहले प्रेषक अनुमति-सूचियों का मूल्यांकन किया जाता है।

    यदि कोई `channels.whatsapp` ब्लॉक बिल्कुल मौजूद नहीं है, तो रनटाइम `groupPolicy: "allowlist"` पर फ़ॉलबैक करता है (चेतावनी लॉग के साथ), भले ही `channels.defaults.groupPolicy` किसी अन्य मान पर सेट हो।

    <Note>
    समूह-सदस्यता समाधान में एकल-अकाउंट सुरक्षा-व्यवस्था है: यदि केवल एक WhatsApp अकाउंट कॉन्फ़िगर है और उसका `accounts.<id>.groups` एक स्पष्ट रिक्त ऑब्जेक्ट (`{}`) है, तो उसे "सेट नहीं" माना जाता है और प्रत्येक समूह को चुपचाप अवरुद्ध करने के बजाय रूट `channels.whatsapp.groups` मैप पर फ़ॉलबैक किया जाता है। 2+ अकाउंट कॉन्फ़िगर होने पर, स्पष्ट रिक्त अकाउंट मैप रिक्त ही रहता है और फ़ॉलबैक नहीं करता — इससे एक अकाउंट संबंधित अन्य अकाउंट को प्रभावित किए बिना जानबूझकर सभी समूह अक्षम कर सकता है।
    </Note>

  </Tab>

  <Tab title="उल्लेख और /activation">
    समूह उत्तरों के लिए डिफ़ॉल्ट रूप से उल्लेख आवश्यक है। उल्लेख पहचान में ये शामिल हैं:

    - बॉट पहचान के स्पष्ट WhatsApp उल्लेख
    - कॉन्फ़िगर किए गए उल्लेख रेगेक्स पैटर्न (`agents.list[].groupChat.mentionPatterns`, फ़ॉलबैक `messages.groupChat.mentionPatterns`)
    - अधिकृत समूह संदेशों के इनबाउंड वॉइस-नोट ट्रांसक्रिप्ट
    - अंतर्निहित बॉट-को-उत्तर पहचान (उत्तर प्रेषक बॉट पहचान से मेल खाता है)

    सुरक्षा: उद्धरण/उत्तर केवल उल्लेख गेटिंग को पूरा करता है — यह प्रेषक प्राधिकरण **नहीं** देता। `groupPolicy: "allowlist"` के साथ, अनुमति-सूची में शामिल न होने वाले प्रेषक किसी अनुमत उपयोगकर्ता के संदेश का उत्तर देते समय भी अवरुद्ध रहते हैं।

    सत्र-स्तरीय सक्रियण कमांड: `/activation mention` या `/activation always`। यह सत्र स्थिति को अपडेट करता है (वैश्विक कॉन्फ़िगरेशन को नहीं) और स्वामी द्वारा नियंत्रित है।

  </Tab>
</Tabs>

## कॉन्फ़िगर की गई ACP बाइंडिंग

WhatsApp शीर्ष-स्तरीय `bindings[]` के माध्यम से स्थायी ACP बाइंडिंग का समर्थन करता है:

```json5
{
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "direct", id: "+15555550123" },
      },
    },
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "group", id: "120363424282127706@g.us" },
      },
    },
  ],
}
```

प्रत्यक्ष चैट E.164 नंबरों से मेल खाती हैं; समूह WhatsApp समूह JID से मेल खाते हैं। OpenClaw द्वारा बाउंड ACP सत्र का अस्तित्व सुनिश्चित करने से पहले समूह अनुमति-सूचियाँ, प्रेषक नीति और उल्लेख/सक्रियण गेटिंग चलती हैं। मिलान की गई बाइंडिंग रूट की स्वामी होती है — ब्रॉडकास्ट समूह उस टर्न को सामान्य WhatsApp सत्रों में वितरित नहीं करते।

## व्यक्तिगत नंबर और स्वयं से चैट का व्यवहार

जब लिंक किया गया स्वयं का नंबर `allowFrom` में भी मौजूद हो, तो स्वयं से चैट की सुरक्षा-व्यवस्थाएँ सक्रिय हो जाती हैं: स्वयं से चैट वाले टर्न के लिए पढ़ने की रसीदें छोड़ना, स्वयं को पिंग करने वाले उल्लेख-JID स्वचालित-ट्रिगर व्यवहार को अनदेखा करना, और `messages.responsePrefix` सेट न होने पर उत्तरों को डिफ़ॉल्ट रूप से `[{identity.name}]` (या `[openclaw]`) पर भेजना।

## संदेश सामान्यीकरण और संदर्भ

<AccordionGroup>
  <Accordion title="इनबाउंड एनवेलप और उत्तर संदर्भ">
    आने वाले संदेश साझा इनबाउंड एनवेलप में रैप किए जाते हैं। उद्धृत उत्तर इस रूप में संदर्भ जोड़ता है:

    ```text
    [Replying to <sender> id:<stanzaId>]
    <quoted body or media placeholder>
    [/Replying]
    ```

    उपलब्ध होने पर उत्तर मेटाडेटा (`ReplyToId`, `ReplyToBody`, `ReplyToSender`, प्रेषक JID/E.164) भरा जाता है। यदि उद्धृत लक्ष्य डाउनलोड करने योग्य मीडिया है, तो OpenClaw उसे सामान्य इनबाउंड मीडिया स्टोर के माध्यम से सहेजता है और `MediaPath`/`MediaType` उपलब्ध कराता है, ताकि एजेंट केवल `<media:image>` देखने के बजाय उसका सीधे निरीक्षण कर सके।

  </Accordion>

  <Accordion title="मीडिया प्लेसहोल्डर और स्थान/संपर्क निष्कर्षण">
    केवल-मीडिया संदेश इन प्लेसहोल्डर में सामान्यीकृत होते हैं: `<media:image>`, `<media:video>`, `<media:audio>`, `<media:document>`, `<media:sticker>`।

    जब बॉडी केवल `<media:audio>` हो, तो अधिकृत समूह वॉइस नोट का उल्लेख गेटिंग से पहले ट्रांसक्रिप्शन किया जाता है, ताकि वॉइस नोट में बॉट का उल्लेख बोलने से उत्तर ट्रिगर हो सके। यदि ट्रांसक्रिप्ट में फिर भी बॉट का उल्लेख नहीं है, तो वह कच्चे प्लेसहोल्डर के बजाय लंबित समूह इतिहास में रहता है।

    स्थान की बॉडी संक्षिप्त निर्देशांक टेक्स्ट के रूप में रेंडर होती है। स्थान लेबल/टिप्पणियाँ और संपर्क/vCard विवरण इनलाइन प्रॉम्प्ट टेक्स्ट के बजाय फ़ेंस किए गए अविश्वसनीय मेटाडेटा के रूप में रेंडर होते हैं।

  </Accordion>

  <Accordion title="लंबित समूह इतिहास का अंतःक्षेपण">
    असंसाधित समूह संदेश बफ़र होते हैं और अंततः बॉट ट्रिगर होने पर संदर्भ के रूप में अंतःक्षेपित किए जाते हैं।

    - डिफ़ॉल्ट सीमा: `50`
    - कॉन्फ़िगरेशन: `channels.whatsapp.historyLimit`, फ़ॉलबैक `messages.groupChat.historyLimit`
    - `0` अक्षम करता है

    अंतःक्षेपण मार्कर: `[Chat messages since your last reply - for context]` और `[Current message - respond to this]`।

  </Accordion>

  <Accordion title="पढ़ने की रसीदें">
    स्वीकृत इनबाउंड संदेशों के लिए डिफ़ॉल्ट रूप से सक्षम। वैश्विक रूप से अक्षम करें:

    ```json5
    { channels: { whatsapp: { sendReadReceipts: false } } }
    ```

    प्रति-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.sendReadReceipts`। वैश्विक रूप से सक्षम होने पर भी स्वयं से चैट वाले टर्न पढ़ने की रसीदें छोड़ देते हैं।

  </Accordion>

    - ऑडियो को `ptt: true` के साथ Baileys `audio` पेलोड के रूप में भेजा जाता है, जो पुश-टू-टॉक वॉइस नोट के रूप में रेंडर होता है; उत्तर पेलोड में `audioAsVoice` संरक्षित रहता है, ताकि प्रदाता के स्रोत प्रारूप की परवाह किए बिना TTS वॉइस-नोट आउटपुट इसी पथ पर रहे
    - नेटिव Ogg/Opus ऑडियो `audio/ogg; codecs=opus` के रूप में भेजा जाता है; अन्य सभी प्रारूपों (Microsoft Edge TTS MP3/WebM आउटपुट सहित) को PTT डिलीवरी से पहले `ffmpeg` से 48 kHz मोनो Ogg/Opus में ट्रांसकोड किया जाता है
    - `/tts latest` नवीनतम सहायक उत्तर को एक वॉइस नोट के रूप में भेजता है और उसी उत्तर के दोबारा प्रेषण को रोकता है; `/tts chat on|off|default` वर्तमान चैट के लिए स्वचालित-TTS नियंत्रित करता है
    - वीडियो पर `gifPlayback: true` भेजने से एनिमेटेड GIF प्लेबैक सक्षम होता है
    - `forceDocument`/`asDocument` WhatsApp के मीडिया संपीड़न से बचने के लिए आउटबाउंड इमेज, GIF और वीडियो को Baileys दस्तावेज़ पेलोड के माध्यम से रूट करता है, जिससे समाधान किया गया फ़ाइल नाम और MIME प्रकार संरक्षित रहता है
    - बहु-मीडिया उत्तर में कैप्शन पहले मीडिया आइटम पर लागू होता है, PTT वॉइस नोट को छोड़कर: ऑडियो पहले बिना कैप्शन भेजा जाता है, फिर कैप्शन अलग टेक्स्ट संदेश के रूप में भेजा जाता है (WhatsApp क्लाइंट वॉइस-नोट कैप्शन को सुसंगत रूप से रेंडर नहीं करते)
    - मीडिया स्रोत HTTP(S), `file://`, या स्थानीय पथ हो सकता है

  </Accordion>

  <Accordion title="मीडिया आकार सीमाएँ और फ़ॉलबैक व्यवहार">
    - इनबाउंड सहेजने की सीमा और आउटबाउंड भेजने की सीमा: `channels.whatsapp.mediaMaxMb` (डिफ़ॉल्ट `50`)
    - प्रति-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.mediaMaxMb`

    - मीडिया भेजने में विफलता होने पर, पहला-आइटम फ़ॉलबैक प्रतिक्रिया को चुपचाप छोड़ने के बजाय टेक्स्ट चेतावनी भेजता है

  </Accordion>

## उत्तर उद्धरण

`channels.whatsapp.replyToMode` नेटिव उत्तर उद्धरण नियंत्रित करता है (आउटबाउंड उत्तर इनबाउंड संदेश को स्पष्ट रूप से उद्धृत करते हैं):

| मान             | व्यवहार                                                       |
| ----------------- | -------------------------------------------------------------- |
| `"off"` (डिफ़ॉल्ट) | कभी उद्धृत न करें; सामान्य संदेश के रूप में भेजें                           |
| `"first"`         | केवल पहले आउटबाउंड उत्तर खंड को उद्धृत करें                      |
| `"all"`           | प्रत्येक आउटबाउंड उत्तर खंड को उद्धृत करें                               |
| `"batched"`       | कतारबद्ध बैच उत्तरों को उद्धृत करें; तत्काल उत्तरों को बिना उद्धरण छोड़ें |

प्रति-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.replyToMode`।

```json5
{ channels: { whatsapp: { replyToMode: "first" } } }
```

## प्रतिक्रिया स्तर

`channels.whatsapp.reactionLevel` नियंत्रित करता है कि एजेंट इमोजी प्रतिक्रियाओं का कितने व्यापक रूप से उपयोग करता है:

| स्तर                 | पावती प्रतिक्रियाएँ | एजेंट द्वारा आरंभ की गई प्रतिक्रियाएँ  |
| --------------------- | ------------- | -------------------------- |
| `"off"`               | नहीं            | नहीं                         |
| `"ack"`               | हाँ           | नहीं                         |
| `"minimal"` (डिफ़ॉल्ट) | हाँ           | हाँ, सीमित मार्गदर्शन |
| `"extensive"`         | हाँ           | हाँ, प्रोत्साहित मार्गदर्शन   |

प्रति-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.reactionLevel`।

```json5
{ channels: { whatsapp: { reactionLevel: "ack" } } }
```

## पावती प्रतिक्रियाएँ

`channels.whatsapp.ackReaction` इनबाउंड प्राप्ति पर तत्काल प्रतिक्रिया भेजता है, जिसे `reactionLevel` नियंत्रित करता है (`"off"` होने पर दबाया जाता है):

```json5
{
  channels: {
    whatsapp: {
      ackReaction: {
        emoji: "👀",
        direct: true,
        group: "mentions", // always | mentions | never
      },
    },
  },
}
```

टिप्पणियाँ: इनबाउंड स्वीकार होने के तुरंत बाद भेजा जाता है (उत्तर से पहले); यदि `ackReaction`, `emoji` के बिना मौजूद है, तो WhatsApp रूट किए गए एजेंट की पहचान इमोजी का उपयोग करता है और उपलब्ध न होने पर "👀" का उपयोग करता है (कोई पावती न भेजने के लिए `ackReaction` छोड़ दें या `emoji: ""` सेट करें); विफलताएँ लॉग की जाती हैं, लेकिन उत्तर डिलीवरी को अवरुद्ध नहीं करतीं; समूह मोड `mentions` केवल उल्लेख से ट्रिगर हुए टर्न पर प्रतिक्रिया देता है, जबकि समूह सक्रियण `always` उस जाँच को बायपास करता है; WhatsApp केवल `channels.whatsapp.ackReaction` का उपयोग करता है (लीगेसी `messages.ackReaction` यहाँ लागू नहीं होता)।

## जीवनचक्र स्थिति प्रतिक्रियाएँ

`messages.statusReactions.enabled: true` सेट करें, ताकि WhatsApp किसी टर्न के दौरान स्थिर प्राप्ति इमोजी छोड़ने के बजाय पावती प्रतिक्रिया को बदलता रहे और कतारबद्ध, विचाररत, टूल गतिविधि, Compaction, पूर्ण और त्रुटि जैसी स्थितियों से गुज़रे:

```json5
{
  messages: {
    statusReactions: {
      enabled: true,
      emojis: {
        deploy: "🛫",
        build: "🏗️",
        concierge: "💁",
      },
    },
  },
}
```

टिप्पणियाँ: `channels.whatsapp.ackReaction` अब भी प्रत्यक्ष संदेशों और समूहों की पात्रता नियंत्रित करता है; कतारबद्ध स्थिति सामान्य पावती प्रतिक्रियाओं के समान प्रभावी इमोजी का उपयोग करती है; WhatsApp में प्रति संदेश बॉट प्रतिक्रिया का एक स्लॉट होता है, इसलिए जीवनचक्र अपडेट मौजूदा प्रतिक्रिया को उसी स्थान पर बदलते हैं; `messages.removeAckAfterReply: true` कॉन्फ़िगर किए गए पूर्ण/त्रुटि होल्ड के बाद अंतिम स्थिति प्रतिक्रिया हटा देता है; टूल इमोजी श्रेणियों में `tool`, `coding`, `web`, `deploy`, `build`, और `concierge` शामिल हैं।

## टूल, कार्रवाइयाँ और कॉन्फ़िगरेशन लेखन

- एजेंट टूल समर्थन में WhatsApp प्रतिक्रिया कार्रवाई (`react`) शामिल है।
- कार्रवाई गेट: `channels.whatsapp.actions.reactions`, `channels.whatsapp.actions.polls` (मौजूदा कार्रवाइयों का डिफ़ॉल्ट `true` है), `channels.whatsapp.actions.calls` (डिफ़ॉल्ट `false`, ऊपर MeowCaller देखें)।
- चैनल द्वारा आरंभ किए गए कॉन्फ़िगरेशन लेखन डिफ़ॉल्ट रूप से सक्षम हैं; `channels.whatsapp.configWrites: false` के माध्यम से अक्षम करें।

    लक्षण: लिंक किए गए खाते में बार-बार कनेक्शन टूटता है या पुनः कनेक्ट करने के प्रयास होते हैं।

    कम सक्रिय खाते सामान्य संदेश टाइमआउट के बाद भी कनेक्ट रह सकते हैं; वॉचडॉग केवल तब पुनः शुरू करता है, जब WhatsApp Web ट्रांसपोर्ट गतिविधि रुक जाती है, सॉकेट बंद हो जाता है या एप्लिकेशन-स्तरीय गतिविधि लंबी सुरक्षा अवधि से अधिक समय तक निष्क्रिय रहती है (ऊपर रनटाइम मॉडल देखें)।

    यदि लॉग में बार-बार `status=408 Request Time-out Connection was lost` दिखाई दे, तो `web.whatsapp` के अंतर्गत Baileys सॉकेट टाइमिंग समायोजित करें। पहले `keepAliveIntervalMs` को अपने नेटवर्क के निष्क्रियता टाइमआउट से कम करें और धीमे या अधिक पैकेट-हानि वाले लिंक पर `connectTimeoutMs` बढ़ाएँ:

    ```json5
    {
      web: {
        whatsapp: {
          keepAliveIntervalMs: 15000,
          connectTimeoutMs: 60000,
          defaultQueryTimeoutMs: 60000,
        },
      },
    }
    ```

    समाधान:

    ```bash
    openclaw channels status --probe
    openclaw doctor
    openclaw logs --follow
    openclaw gateway status
    ```

    यदि होस्ट कनेक्टिविटी और टाइमिंग ठीक करने के बाद भी लूप बना रहता है, तो खाते की प्रमाणीकरण डायरेक्टरी का बैकअप लें और दोबारा लिंक करें:

    ```bash
    cp -a ~/.openclaw/credentials/whatsapp/<accountId> \
      ~/.openclaw/credentials/whatsapp/<accountId>.bak
    openclaw channels logout --channel whatsapp --account <accountId>
    openclaw channels login --channel whatsapp --account <accountId>
    ```

    यदि `~/.openclaw/logs/whatsapp-health.log` में `Gateway inactive` लिखा है, लेकिन `openclaw gateway status` और `openclaw channels status --probe` दोनों स्वस्थ स्थिति दिखाते हैं, तो `openclaw doctor` चलाएँ। Linux पर, doctor सेवानिवृत्त `~/.openclaw/bin/ensure-whatsapp.sh` स्क्रिप्ट को चलाने वाली लेगेसी crontab प्रविष्टियों के बारे में चेतावनी देता है; उन प्रविष्टियों को `crontab -e` से हटाएँ — cron में systemd यूज़र-बस परिवेश नहीं हो सकता, जिससे वह पुरानी स्क्रिप्ट Gateway की स्थिति की गलत रिपोर्ट कर सकती है।

  </Accordion>

  </Accordion>

  <Accordion title="समूह संदेश अनपेक्षित रूप से अनदेखे किए जाते हैं">
    इस क्रम में जाँचें: `groupPolicy`, `groupAllowFrom`/`allowFrom`, `groups` अनुमति-सूची प्रविष्टियाँ, उल्लेख गेटिंग (`requireMention` + उल्लेख पैटर्न) और `openclaw.json` में डुप्लिकेट कुंजियाँ (JSON5 में बाद की प्रविष्टियाँ पहले वाली को ओवरराइड करती हैं — प्रत्येक दायरे में केवल एक `groupPolicy` रखें)।

    यदि `channels.whatsapp.groups` मौजूद है, तो WhatsApp अब भी अन्य समूहों के संदेश देख सकता है, लेकिन OpenClaw उन्हें सत्र रूटिंग से पहले हटा देता है। समूह JID को `channels.whatsapp.groups` में जोड़ें, या प्रेषक प्राधिकरण को `groupPolicy`/`groupAllowFrom` के अंतर्गत रखते हुए सभी समूहों को अनुमति देने के लिए `groups["*"]` जोड़ें।

  </Accordion>

  <Accordion title="Bun रनटाइम चेतावनी">
    OpenClaw Gateway के लिए Node आवश्यक है। Bun, कैनोनिकल स्थिति स्टोर द्वारा उपयोग किया जाने वाला `node:sqlite` API उपलब्ध नहीं कराता और doctor लेगेसी Bun सेवाओं को Node पर माइग्रेट करता है।
  </Accordion>
</AccordionGroup>

## सिस्टम प्रॉम्प्ट

WhatsApp, `groups` और `direct` मैप के माध्यम से समूहों तथा डायरेक्ट चैट के लिए Telegram-शैली के सिस्टम प्रॉम्प्ट का समर्थन करता है।

समूह संदेशों का रिज़ॉल्यूशन: पहले प्रभावी `groups` मैप निर्धारित होता है — यदि खाता अपनी `groups` कुंजी किसी भी रूप में परिभाषित करता है, तो वह रूट `groups` मैप को पूरी तरह बदल देता है (कोई डीप मर्ज नहीं)। इसके बाद प्रॉम्प्ट लुकअप केवल उसी परिणामी मैप पर चलता है:

1. **समूह-विशिष्ट प्रॉम्प्ट** (`groups["<groupId>"].systemPrompt`): इसका उपयोग तब होता है, जब समूह प्रविष्टि मौजूद हो **और** उसकी `systemPrompt` कुंजी परिभाषित हो। खाली स्ट्रिंग (`""`) वाइल्डकार्ड को दबा देती है और कोई प्रॉम्प्ट लागू नहीं करती।
2. **समूह वाइल्डकार्ड प्रॉम्प्ट** (`groups["*"].systemPrompt`): इसका उपयोग तब होता है, जब विशिष्ट समूह प्रविष्टि अनुपस्थित हो या वह `systemPrompt` कुंजी के बिना मौजूद हो।

डायरेक्ट संदेशों का रिज़ॉल्यूशन `direct` मैप और `direct["*"]` पर इसी समान पैटर्न का पालन करता है।

<Note>
`dms`, प्रति-DM इतिहास ओवरराइड के हल्के बकेट (`dms.<id>.historyLimit`) के रूप में बना रहता है। प्रॉम्प्ट ओवरराइड `direct` के अंतर्गत रहते हैं।
</Note>

<Note>
प्रॉम्प्ट रिज़ॉल्यूशन के लिए खाते द्वारा रूट को बदलने का यह व्यवहार एक सामान्य शैलो ओवरराइड है: स्पष्ट खाली ऑब्जेक्ट सहित खाते की कोई भी `groups`/`direct` कुंजी रूट मैप को बदल देती है। यह ऊपर वर्णित समूह-सदस्यता अनुमति-सूची जाँच से अलग है, जिसमें गलती से खाली `groups: {}` के लिए एकल-खाता सुरक्षा-व्यवस्था है।
</Note>

**Telegram से अंतर:** Telegram बहु-खाता सेटअप में प्रत्येक खाते के लिए रूट `groups` को दबा देता है (उन खातों के लिए भी, जिनका अपना कोई `groups` नहीं है), ताकि बॉट उन समूहों के संदेश प्राप्त न करे जिनका वह सदस्य नहीं है। WhatsApp यह सुरक्षा लागू नहीं करता — जिन खातों का अपना ओवरराइड नहीं है, वे खाता संख्या की परवाह किए बिना रूट `groups`/`direct` इनहेरिट करते हैं। बहु-खाता WhatsApp सेटअप में, यदि आप प्रति-खाता प्रॉम्प्ट चाहते हैं, तो प्रत्येक खाते के अंतर्गत पूरा मैप स्पष्ट रूप से परिभाषित करें।

महत्वपूर्ण व्यवहार:

- `channels.whatsapp.groups` प्रति-समूह कॉन्फ़िगरेशन मैप और चैट-स्तरीय समूह अनुमति-सूची दोनों है। रूट या खाता दायरे में, `groups["*"]` का अर्थ उस दायरे के लिए "सभी समूहों को अनुमति है" होता है।
- वाइल्डकार्ड `systemPrompt` केवल तभी जोड़ें, जब आप पहले से उस दायरे में सभी समूहों को अनुमति देना चाहते हों। केवल समूह आईडी के निश्चित सेट को पात्र रखने के लिए, `groups["*"]` का उपयोग करने के बजाय प्रत्येक स्पष्ट रूप से अनुमति-सूचीबद्ध प्रविष्टि पर प्रॉम्प्ट दोहराएँ।
- समूह प्रवेश और प्रेषक प्राधिकरण अलग-अलग जाँचें हैं। `groups["*"]` उन समूहों का दायरा बढ़ाता है जो समूह हैंडलिंग तक पहुँचते हैं; यह उन समूहों के प्रत्येक प्रेषक को प्राधिकृत नहीं करता — वह `groupPolicy`/`groupAllowFrom` द्वारा नियंत्रित रहता है।
- `channels.whatsapp.direct` का DM के लिए कोई समकक्ष पार्श्व प्रभाव नहीं है: `direct["*"]` केवल तब डिफ़ॉल्ट कॉन्फ़िगरेशन देता है, जब DM को `dmPolicy` के साथ `allowFrom` या पेयरिंग-स्टोर नियमों द्वारा पहले ही अनुमति मिल चुकी हो।

उदाहरण:

```json5
{
  channels: {
    whatsapp: {
      groups: {
        // केवल तभी उपयोग करें, जब रूट दायरे में सभी समूहों को अनुमति मिलनी चाहिए।
        // उन सभी खातों पर लागू होता है जो अपना groups मैप परिभाषित नहीं करते।
        "*": { systemPrompt: "सभी समूहों के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
      },
      direct: {
        // उन सभी खातों पर लागू होता है जो अपना direct मैप परिभाषित नहीं करते।
        "*": { systemPrompt: "सभी डायरेक्ट चैट के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
      },
      accounts: {
        work: {
          groups: {
            // यह खाता अपना groups परिभाषित करता है, इसलिए रूट groups पूरी तरह
            // बदल दिए जाते हैं। वाइल्डकार्ड रखने के लिए यहाँ भी "*" स्पष्ट रूप से परिभाषित करें।
            "120363406415684625@g.us": {
              requireMention: false,
              systemPrompt: "परियोजना प्रबंधन पर ध्यान केंद्रित करें।",
            },
            // केवल तभी उपयोग करें, जब इस खाते में सभी समूहों को अनुमति मिलनी चाहिए।
            "*": { systemPrompt: "कार्य समूहों के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
          },
          direct: {
            // यह खाता अपना direct मैप परिभाषित करता है, इसलिए रूट direct प्रविष्टियाँ
            // पूरी तरह बदल दी जाती हैं। वाइल्डकार्ड रखने के लिए यहाँ भी "*" स्पष्ट रूप से परिभाषित करें।
            "+15551234567": { systemPrompt: "किसी विशिष्ट कार्य-संबंधी डायरेक्ट चैट के लिए प्रॉम्प्ट।" },
            "*": { systemPrompt: "कार्य-संबंधी डायरेक्ट चैट के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
          },
        },
      },
    },
  },
}
```
