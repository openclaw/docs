स्थिति: WhatsApp Web (Baileys) के माध्यम से प्रोडक्शन के लिए तैयार। लिंक किए गए सत्रों का स्वामित्व Gateway के पास होता है; कोई अलग Twilio WhatsApp चैनल नहीं है।

## इंस्टॉल करें

`openclaw onboard` और `openclaw channels add --channel whatsapp` पहली बार इसे चुनने पर Plugin इंस्टॉल करने का संकेत देते हैं; यदि Plugin उपलब्ध नहीं है, तो `openclaw channels login --channel whatsapp` वही इंस्टॉल प्रवाह प्रस्तुत करता है। डेवलपमेंट चेकआउट स्थानीय Plugin पथ का उपयोग करते हैं; स्थिर/बीटा इंस्टॉल पहले ClawHub से `@openclaw/whatsapp` इंस्टॉल करते हैं और विफल होने पर npm का उपयोग करते हैं। WhatsApp रनटाइम मुख्य OpenClaw npm पैकेज के बाहर वितरित होता है, इसलिए इसकी रनटाइम निर्भरताएँ बाहरी Plugin के साथ रहती हैं। मैन्युअल इंस्टॉल:

```bash
openclaw plugins install clawhub:@openclaw/whatsapp
```

साधारण npm पैकेज (`@openclaw/whatsapp`) का उपयोग केवल रजिस्ट्री फ़ॉलबैक के लिए करें; पुनरुत्पाद्य इंस्टॉल के लिए ही सटीक संस्करण पिन करें।

## रनटाइम मॉडल

- WhatsApp सॉकेट और पुनः कनेक्शन लूप का स्वामित्व Gateway के पास होता है।
- एक वॉचडॉग दो संकेतों को स्वतंत्र रूप से ट्रैक करता है: अपरिष्कृत WhatsApp Web ट्रांसपोर्ट गतिविधि और एप्लिकेशन-संदेश गतिविधि। शांत लेकिन कनेक्टेड सत्र को केवल इसलिए पुनः शुरू नहीं किया जाता कि हाल में कोई संदेश नहीं आया; यह पुनः कनेक्शन केवल तभी बाध्य करता है जब एक निश्चित आंतरिक अवधि (उपयोगकर्ता द्वारा कॉन्फ़िगर करने योग्य नहीं) तक ट्रांसपोर्ट फ़्रेम आना बंद हो जाएँ या एप्लिकेशन संदेश सामान्य संदेश टाइमआउट के 4x से अधिक समय तक शांत रहें। हाल में सक्रिय रहे सत्र के पुनः कनेक्ट होने के तुरंत बाद, पहली अवधि 4x अवधि के बजाय छोटे सामान्य संदेश टाइमआउट का उपयोग करती है। उस पुनः कनेक्शन में Baileys द्वारा जल्दी वितरित किए गए ऑफ़लाइन संदेशों का OpenClaw स्वतः उत्तर दे सकता है, जिसकी सीमा इनबाउंड संदेश-ID डीडुप्लिकेशन जीवनकाल से निर्धारित होती है; प्रारंभिक स्टार्टअप में छोटी पुराना-इतिहास सुरक्षा बनी रहती है।
- आउटबाउंड प्रेषण के लिए लक्षित खाते पर सक्रिय WhatsApp लिसनर आवश्यक है; अन्यथा प्रेषण तुरंत विफल हो जाता है।
- जब टोकन मौजूदा प्रतिभागी मेटाडेटा से मेल खाता है, तो समूह प्रेषण `@+<digits>` और `@<digits>` टोकन के लिए (टेक्स्ट और मीडिया कैप्शन में) नेटिव उल्लेख मेटाडेटा जोड़ते हैं; इसमें LID-समर्थित समूह भी शामिल हैं।
- स्थिति और ब्रॉडकास्ट चैट (`@status`, `@broadcast`) अनदेखी की जाती हैं।
- सीधी चैट DM सत्र नियमों का उपयोग करती हैं (`session.dmScope`; डिफ़ॉल्ट `main` DM को एजेंट के मुख्य सत्र में समेकित करता है)। समूह सत्र प्रत्येक JID के अनुसार अलग रखे जाते हैं (`agent:<agentId>:whatsapp:group:<jid>`)।
- WhatsApp Channels/Newsletters अपने नेटिव `@newsletter` JID के माध्यम से स्पष्ट आउटबाउंड लक्ष्य हो सकते हैं और DM अर्थ-विज्ञान के बजाय चैनल सत्र मेटाडेटा (`agent:<agentId>:whatsapp:channel:<jid>`) का उपयोग करते हैं।
- WhatsApp Web ट्रांसपोर्ट Gateway होस्ट पर मानक प्रॉक्सी पर्यावरण चर (`HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY`, छोटे अक्षरों वाले रूप) का पालन करता है। प्रति-चैनल सेटिंग्स के बजाय होस्ट-स्तरीय प्रॉक्सी कॉन्फ़िगरेशन को प्राथमिकता दें।
- `messages.removeAckAfterReply` सक्षम होने पर, दृश्यमान उत्तर वितरित हो जाने के बाद OpenClaw पावती प्रतिक्रिया हटा देता है।

## MeowCaller से वर्तमान अनुरोधकर्ता को कॉल करें (प्रयोगात्मक)

WhatsApp से शुरू होने वाले एजेंट टर्न में Plugin `whatsapp_call` उपलब्ध करा सकता है। यह वर्तमान अधिकृत अनुरोधकर्ता को WhatsApp वॉइस कॉल करने और उनके उत्तर देने के बाद OpenClaw TTS संदेश चलाने के लिए [MeowCaller](https://github.com/purpshell/meowcaller) का उपयोग करता है। टूल में गंतव्य नंबर का कोई पैरामीटर नहीं है, इसलिए कोई प्रॉम्प्ट कॉल को पुनर्निर्देशित नहीं कर सकता। डिफ़ॉल्ट रूप से अक्षम है।

<Warning>
MeowCaller प्रयोगात्मक है, इसका कोई टैग किया गया रिलीज़ नहीं है और यह अलग से पेयर किए गए whatsmeow लिंक्ड-डिवाइस सत्र का उपयोग करता है—यह Plugin के Baileys क्रेडेंशियल्स का पुनः उपयोग नहीं कर सकता। पेयरिंग उसी WhatsApp खाते में एक और लिंक्ड डिवाइस जोड़ती है; OpenClaw द्वारा उपयोग की जाने वाली पहचान से स्कैन करें। व्यक्तिगत नंबर/स्वयं से चैट मोड स्वयं को कॉल नहीं कर सकता; अपने व्यक्तिगत नंबर पर कॉल करने के लिए समर्पित OpenClaw नंबर का उपयोग करें।
</Warning>

<Steps>
  <Step title="प्रयोगात्मक कॉल सक्षम करें">

    WhatsApp चैनल कॉन्फ़िगरेशन में `actions.calls: true` जोड़ें और Gateway पुनः शुरू करें:

```json
{
  "channels": {
    "whatsapp": {
      "actions": {
        "calls": true
      }
    }
  }
}
```

    अनुपस्थित या `false` होने पर, OpenClaw `whatsapp_call` टूल उपलब्ध नहीं कराता।

  </Step>

  <Step title="समीक्षित MeowCaller CLI इंस्टॉल करें">

    अडैप्टर Gateway होस्ट के `PATH` पर `meowcaller` एक्ज़ीक्यूटेबल की अपेक्षा करता है। [MeowCaller PR #7](https://github.com/purpshell/meowcaller/pull/7) के मर्ज होने तक समीक्षित ब्रांच बनाएँ:

```bash
git clone --branch feat/send-only-notify https://github.com/steipete/meowcaller.git
cd meowcaller
git checkout 752050471fc2bf7a8cdfbf7dbd3cd4e865d85d3f
mkdir -p "$HOME/.local/bin"
go build -o "$HOME/.local/bin/meowcaller" ./cmd/meowcaller
```

    सुनिश्चित करें कि `$HOME/.local/bin`, Gateway सेवा के `PATH` पर है। इस संशोधन में स्पष्ट `pair` और केवल-प्रेषण `notify` कमांड हैं; `notify` कोई माइक्रोफ़ोन, स्पीकर, वीडियो डिवाइस या निदान कैप्चर नहीं खोलता। अपस्ट्रीम उदाहरण CLI की `play` कमांड का विकल्प के रूप में उपयोग न करें।

  </Step>

  <Step title="MeowCaller लिंक्ड डिवाइस पेयर करें">

    WhatsApp एजेंट से कॉल सेटअप जाँचने को कहें (`whatsapp_call` स्थिति क्रिया खाते-विशिष्ट स्टेट डायरेक्टरी और पेयरिंग कमांड की सूचना देती है)। डिफ़ॉल्ट खाते के लिए:

```bash
state_dir="$HOME/.openclaw/credentials/whatsapp-calls/default"
mkdir -p "$state_dir"
chmod 700 "$state_dir"
meowcaller pair --store "$state_dir/wa-voip.db"
```

    इसे इंटरैक्टिव रूप से चलाएँ, **WhatsApp > Linked devices** से QR स्कैन करें और `MeowCaller linked device ready` की प्रतीक्षा करें। `wa-voip.db` को निजी रखें—यह MeowCaller सत्र है। गैर-डिफ़ॉल्ट खातों को स्थिति क्रिया से अपना अलग स्टोर पथ मिलता है; Windows पर इसकी PowerShell कमांड चलाएँ।

  </Step>

  <Step title="TTS कॉन्फ़िगर करें और WhatsApp से कॉल करें">

    टेलीफ़ोनी-सक्षम [TTS प्रदाता](/hi/tools/tts) कॉन्फ़िगर करें, Gateway पुनः शुरू करें, फिर `Call me and say the build finished.` जैसा अनुरोध भेजें। टूल विश्वसनीय इनबाउंड संदर्भ से प्रेषक निर्धारित करता है, अस्थायी निजी WAV फ़ाइल संश्लेषित करता है, सीमित कॉल अवधि के लिए MeowCaller चलाता है और बाद में ऑडियो फ़ाइल हटा देता है। OpenClaw खाते का स्टोर स्पष्ट रूप से पास करता है, उत्तर/प्लेबैक/कॉल समाप्ति के बाद शून्य निकास स्थिति की प्रतीक्षा करता है और टाइमआउट या गैर-शून्य निकास को विफल टूल कॉल मानता है।

  </Step>
</Steps>

सीमाएँ: केवल एक-से-एक आउटबाउंड ऑडियो कॉल, कोई मनमाना गंतव्य नंबर नहीं, चैट कनेक्शन के साथ कोई साझा प्रमाणीकरण नहीं, व्यक्तिगत नंबर/स्वयं से चैट मोड से स्वयं को कॉल नहीं, संश्लेषित ऑडियो अधिकतम 60 सेकंड, MeowCaller के उत्तर/प्लेबैक/कॉल समाप्ति पूर्ण होने के अतिरिक्त हैंडसेट पर सुनाई देने की कोई रसीद नहीं, और OpenClaw सहायक प्रक्रिया को सीमित 115-175 सेकंड की अवधि के बाद रोक देता है (जिसमें MeowCaller के कनेक्शन, उत्तर, प्लेबैक और शटडाउन चरण शामिल हैं)।

## अनुमोदन प्रॉम्प्ट

WhatsApp शीर्ष-स्तरीय अनुमोदन फ़ॉरवर्डिंग कॉन्फ़िगरेशन द्वारा नियंत्रित, exec और Plugin अनुमोदन प्रॉम्प्ट को `👍`/`👎` प्रतिक्रियाओं के रूप में प्रदर्शित कर सकता है:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "session",
    },
    plugin: {
      enabled: true,
      mode: "targets",
      targets: [{ channel: "whatsapp", to: "+15551234567" }],
    },
  },
}
```

`approvals.exec` और `approvals.plugin` स्वतंत्र हैं; WhatsApp को चैनल के रूप में सक्षम करना केवल ट्रांसपोर्ट को लिंक करता है और तब तक कुछ नहीं भेजता जब तक संबंधित अनुमोदन वर्ग सक्षम करके वहाँ रूट न किया गया हो। सत्र मोड केवल WhatsApp से उत्पन्न अनुमोदनों के लिए नेटिव इमोजी अनुमोदन वितरित करता है। लक्ष्य मोड स्पष्ट लक्ष्यों के लिए साझा फ़ॉरवर्डिंग पाइपलाइन का उपयोग करता है और अलग अनुमोदक-DM फ़ैनआउट नहीं बनाता।

WhatsApp अनुमोदन प्रतिक्रियाओं के लिए `allowFrom` (या `"*"`) में स्पष्ट अनुमोदक आवश्यक हैं। `defaultTo` सामान्य डिफ़ॉल्ट संदेश लक्ष्य निर्धारित करता है, अनुमोदकों की सूची नहीं। मैन्युअल `/approve` कमांड अनुमोदन समाधान से पहले भी सामान्य WhatsApp प्रेषक-प्राधिकरण पथ से गुजरती हैं।

## प्रश्न प्रतिक्रियाएँ

एक गैर-गोपनीय, एकल-चयन प्रश्न और एक से चार विकल्पों वाले `ask_user` प्रॉम्प्ट के लिए, WhatsApp विकल्प लेबल के पास `1️⃣` से `4️⃣` तक दिखाता है। उत्तर देने के लिए वितरित प्रॉम्प्ट पर मेल खाते नंबर से प्रतिक्रिया दें। OpenClaw Gateway के माध्यम से नंबर को कैनॉनिकल विकल्प पर मैप करता है; पुराने या डुप्लिकेट टैप अनदेखे किए जाते हैं। बहु-प्रश्न, बहु-चयन और मुक्त-पाठ प्रॉम्प्ट केवल टेक्स्ट उत्तर वाले बने रहते हैं। सामान्य WhatsApp DM/समूह प्रवेश नियम प्रतिक्रिया देने वाले प्रेषक को अधिकृत करते हैं।

## Plugin हुक और गोपनीयता

इनबाउंड WhatsApp संदेशों में व्यक्तिगत सामग्री, फ़ोन नंबर, समूह पहचानकर्ता, प्रेषक नाम और सत्र सहसंबंध फ़ील्ड हो सकते हैं। जब तक आप ऑप्ट इन नहीं करते, WhatsApp इनबाउंड `message_received` हुक पेलोड को Plugins में ब्रॉडकास्ट नहीं करता:

```json5
{
  channels: {
    whatsapp: {
      pluginHooks: {
        messageReceived: true,
      },
    },
  },
}
```

ऑप्ट-इन का दायरा `channels.whatsapp.accounts.<id>.pluginHooks.messageReceived` के अंतर्गत एक खाते तक सीमित करें। इसे केवल उन Plugins के लिए सक्षम करें जिन पर आप इनबाउंड WhatsApp सामग्री और पहचानकर्ताओं के संबंध में भरोसा करते हैं।

## एक्सेस नियंत्रण और सक्रियण

<Tabs>
  <Tab title="DM नीति">
    `channels.whatsapp.dmPolicy`:

    | मान | व्यवहार |
    | --- | --- |
    | `pairing` (डिफ़ॉल्ट) | अज्ञात प्रेषक पेयरिंग का अनुरोध करते हैं; स्वामी स्वीकृत करता है |
    | `allowlist` | केवल `allowFrom` प्रेषकों को प्रवेश मिलता है |
    | `open` | `allowFrom` में `"*"` का शामिल होना आवश्यक है |
    | `disabled` | सभी DM अवरुद्ध करें |

    `allowFrom` E.164-शैली के नंबर स्वीकार करता है (आंतरिक रूप से सामान्यीकृत)। यह केवल DM प्रेषक अभिगम-नियंत्रण सूची है — यह समूह JID या `@newsletter` चैनल JID पर स्पष्ट आउटबाउंड प्रेषण को नियंत्रित नहीं करता।

    बहु-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.dmPolicy` (और `.allowFrom`) उस अकाउंट के लिए चैनल-स्तरीय डिफ़ॉल्ट पर प्राथमिकता लेते हैं।

    रनटाइम टिप्पणियाँ:

    - पेयरिंग चैनल अनुमति-स्टोर में बनी रहती हैं और कॉन्फ़िगर किए गए `allowFrom` के साथ मर्ज होती हैं
    - शेड्यूल किया गया स्वचालन और Heartbeat प्राप्तकर्ता फ़ॉलबैक स्पष्ट डिलीवरी लक्ष्य या कॉन्फ़िगर किए गए `allowFrom` का उपयोग करते हैं; DM पेयरिंग स्वीकृतियाँ अंतर्निहित Cron/Heartbeat प्राप्तकर्ता नहीं हैं
    - यदि कोई अनुमति-सूची कॉन्फ़िगर नहीं है, तो लिंक किया गया स्वयं का नंबर डिफ़ॉल्ट रूप से अनुमत होता है
    - OpenClaw आउटबाउंड `fromMe` DM को कभी स्वतः पेयर नहीं करता (वे संदेश जो आप लिंक किए गए डिवाइस से स्वयं को भेजते हैं)

  </Tab>

  <Tab title="समूह नीति और अनुमति-सूचियाँ">
    समूह अभिगम की दो परतें हैं:

    1. **समूह सदस्यता अनुमति-सूची** (`channels.whatsapp.groups`): यदि `groups` छोड़ा गया है, तो सभी समूह पात्र हैं; यदि मौजूद है, तो यह समूह अनुमति-सूची की तरह कार्य करता है (`"*"` सभी को अनुमति देता है)।
    2. **समूह प्रेषक नीति** (`channels.whatsapp.groupPolicy` + `groupAllowFrom`): `open` प्रेषक अनुमति-सूची को बायपास करता है, `allowlist` को `groupAllowFrom` (या `*`) से मिलान आवश्यक है, `disabled` सभी इनबाउंड समूह संदेशों को अवरुद्ध करता है।

    यदि `groupAllowFrom` सेट नहीं है, तो प्रविष्टियाँ होने पर प्रेषक जाँच `allowFrom` पर फ़ॉलबैक होती है। प्रेषक अनुमति-सूचियों का मूल्यांकन उल्लेख/उत्तर सक्रियण से पहले किया जाता है।

    यदि कोई `channels.whatsapp` ब्लॉक बिल्कुल मौजूद नहीं है, तो रनटाइम `groupPolicy: "allowlist"` पर फ़ॉलबैक होता है (चेतावनी लॉग के साथ), भले ही `channels.defaults.groupPolicy` किसी अन्य मान पर सेट हो।

    <Note>
    समूह-सदस्यता समाधान में एकल-अकाउंट सुरक्षा व्यवस्था है: यदि केवल एक WhatsApp अकाउंट कॉन्फ़िगर किया गया है और उसका `accounts.<id>.groups` एक स्पष्ट खाली ऑब्जेक्ट (`{}`) है, तो उसे "सेट नहीं" माना जाता है और प्रत्येक समूह को चुपचाप अवरुद्ध करने के बजाय रूट `channels.whatsapp.groups` मैप पर फ़ॉलबैक किया जाता है। 2+ अकाउंट कॉन्फ़िगर होने पर स्पष्ट खाली अकाउंट मैप खाली ही रहता है और फ़ॉलबैक नहीं करता — इससे एक अकाउंट अन्य अकाउंट को प्रभावित किए बिना जानबूझकर सभी समूहों को अक्षम कर सकता है।
    </Note>

  </Tab>

  <Tab title="उल्लेख और /activation">
    समूह उत्तरों के लिए डिफ़ॉल्ट रूप से उल्लेख आवश्यक है। उल्लेख पहचान में शामिल हैं:

    - बॉट पहचान के स्पष्ट WhatsApp उल्लेख
    - कॉन्फ़िगर किए गए उल्लेख रेगेक्स पैटर्न (`agents.list[].groupChat.mentionPatterns`, फ़ॉलबैक `messages.groupChat.mentionPatterns`)
    - अधिकृत समूह संदेशों के इनबाउंड वॉइस-नोट प्रतिलेख
    - अंतर्निहित बॉट-को-उत्तर पहचान (उत्तर प्रेषक बॉट पहचान से मेल खाता है)

    सुरक्षा: उद्धरण/उत्तर केवल उल्लेख नियंत्रण को संतुष्ट करता है — यह प्रेषक प्राधिकरण **नहीं** देता। `groupPolicy: "allowlist"` के साथ, अनुमति-सूची में न होने वाले प्रेषक किसी अनुमत उपयोगकर्ता के संदेश का उत्तर देते समय भी अवरुद्ध रहते हैं।

    सत्र-स्तरीय सक्रियण कमांड: `/activation mention` या `/activation always`। यह सत्र स्थिति को अपडेट करता है (वैश्विक कॉन्फ़िगरेशन को नहीं) और स्वामी-नियंत्रित है।

  </Tab>
</Tabs>

## कॉन्फ़िगर किए गए ACP बाइंडिंग

WhatsApp शीर्ष-स्तरीय `bindings[]` के माध्यम से स्थायी ACP बाइंडिंग का समर्थन करता है:

```json5
{
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "direct", id: "+15555550123" },
      },
    },
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "group", id: "120363424282127706@g.us" },
      },
    },
  ],
}
```

प्रत्यक्ष चैट E.164 नंबरों से मेल खाते हैं; समूह WhatsApp समूह JID से मेल खाते हैं। OpenClaw द्वारा बाउंड ACP सत्र का अस्तित्व सुनिश्चित करने से पहले समूह अनुमति-सूचियाँ, प्रेषक नीति और उल्लेख/सक्रियण नियंत्रण चलते हैं। मेल खाया बाइंडिंग रूट का स्वामी होता है — प्रसारण समूह उस टर्न को सामान्य WhatsApp सत्रों में वितरित नहीं करते।

## व्यक्तिगत नंबर और स्वयं-चैट व्यवहार

जब लिंक किया गया स्वयं का नंबर `allowFrom` में भी मौजूद हो, तो स्वयं-चैट सुरक्षा उपाय सक्रिय हो जाते हैं: स्वयं-चैट टर्न के लिए पठन रसीदें छोड़ना, स्वयं को पिंग करने वाले उल्लेख-JID स्वतः-ट्रिगर व्यवहार को अनदेखा करना, और `messages.responsePrefix` सेट न होने पर उत्तरों को डिफ़ॉल्ट रूप से `[{identity.name}]` (या `[openclaw]`) पर भेजना।

## संदेश सामान्यीकरण और संदर्भ

<AccordionGroup>

    आने वाले संदेश साझा इनबाउंड एनवेलप में लपेटे जाते हैं। उद्धृत उत्तर इस रूप में संदर्भ जोड़ता है:

    ```text
    [Replying to <sender> id:<stanzaId>]
    <quoted body or media placeholder>
    [/Replying]
    ```

  </Accordion>

  <Accordion title="मीडिया प्लेसहोल्डर और स्थान/संपर्क निष्कर्षण">
    केवल-मीडिया संदेश प्लेसहोल्डर में सामान्यीकृत होते हैं: `<media:image>`, `<media:video>`, `<media:audio>`, `<media:document>`, `<media:sticker>`।

    जब बॉडी केवल `<media:audio>` हो, तो अधिकृत समूह वॉइस नोट का उल्लेख नियंत्रण से पहले प्रतिलेखन किया जाता है, इसलिए वॉइस नोट में बॉट का उल्लेख कहने से उत्तर ट्रिगर हो सकता है। यदि प्रतिलेख में फिर भी बॉट का उल्लेख नहीं है, तो यह कच्चे प्लेसहोल्डर के बजाय लंबित समूह इतिहास में रहता है।

    स्थान बॉडी संक्षिप्त निर्देशांक टेक्स्ट के रूप में रेंडर होती हैं। स्थान लेबल/टिप्पणियाँ और संपर्क/vCard विवरण इनलाइन प्रॉम्प्ट टेक्स्ट के बजाय फ़ेन्स किए गए अविश्वसनीय मेटाडेटा के रूप में रेंडर होते हैं।

  </Accordion>

  <Accordion title="लंबित समूह इतिहास अंतःक्षेपण">
    अप्रसंस्कृत समूह संदेश बफ़र होते हैं और अंततः बॉट ट्रिगर होने पर संदर्भ के रूप में अंतःक्षेपित किए जाते हैं।

    - डिफ़ॉल्ट सीमा: `50`
    - कॉन्फ़िगरेशन: `channels.whatsapp.historyLimit`, फ़ॉलबैक `messages.groupChat.historyLimit`
    - `0` अक्षम करता है

    अंतःक्षेपण मार्कर: `[Chat messages since your last reply - for context]` और `[Current message - respond to this]`।

    - ऑडियो को `ptt: true` के साथ Baileys `audio` पेलोड के रूप में भेजा जाता है, जो पुश-टू-टॉक वॉइस नोट के रूप में रेंडर होता है; उत्तर पेलोड पर `audioAsVoice` संरक्षित रहता है, ताकि प्रदाता के स्रोत प्रारूप की परवाह किए बिना TTS वॉइस-नोट आउटपुट इसी पथ पर रहे
    - मूल Ogg/Opus ऑडियो `audio/ogg; codecs=opus` के रूप में भेजा जाता है; अन्य सभी चीज़ों (Microsoft Edge TTS MP3/WebM आउटपुट सहित) को PTT डिलीवरी से पहले `ffmpeg` से 48 kHz मोनो Ogg/Opus में ट्रांसकोड किया जाता है
    - `/tts latest` नवीनतम सहायक उत्तर को एक वॉइस नोट के रूप में भेजता है और उसी उत्तर के दोहराए गए प्रेषण को रोकता है; `/tts chat on|off|default` वर्तमान चैट के लिए स्वतः-TTS नियंत्रित करता है
    - वीडियो प्रेषण पर `gifPlayback: true` एनिमेटेड GIF प्लेबैक सक्षम करता है
    - `forceDocument`/`asDocument` आउटबाउंड छवियों, GIF और वीडियो को Baileys दस्तावेज़ पेलोड के माध्यम से रूट करता है, ताकि WhatsApp के मीडिया संपीड़न से बचते हुए निर्धारित फ़ाइल नाम और MIME प्रकार संरक्षित रहें
    - बहु-मीडिया उत्तर में कैप्शन पहले मीडिया आइटम पर लागू होते हैं, PTT वॉइस नोट को छोड़कर: ऑडियो पहले बिना कैप्शन भेजा जाता है, फिर कैप्शन अलग टेक्स्ट संदेश के रूप में भेजा जाता है (WhatsApp क्लाइंट वॉइस-नोट कैप्शन को सुसंगत रूप से रेंडर नहीं करते)
    - मीडिया स्रोत HTTP(S), `file://` या स्थानीय पथ हो सकता है

  </Accordion>

  <Accordion title="मीडिया आकार सीमाएँ और फ़ॉलबैक व्यवहार">
    - इनबाउंड सहेजने की अधिकतम सीमा और आउटबाउंड भेजने की अधिकतम सीमा: `channels.whatsapp.mediaMaxMb` (डिफ़ॉल्ट `50`)

    - मीडिया प्रेषण विफल होने पर, पहले आइटम का फ़ॉलबैक प्रतिक्रिया को चुपचाप छोड़ने के बजाय एक टेक्स्ट चेतावनी भेजता है

  </Accordion>

## उत्तर उद्धरण

`channels.whatsapp.replyToMode` मूल उत्तर उद्धरण नियंत्रित करता है (आउटबाउंड उत्तर इनबाउंड संदेश को स्पष्ट रूप से उद्धृत करते हैं):

| मान             | व्यवहार                                                       |
| ----------------- | -------------------------------------------------------------- |
| `"off"` (डिफ़ॉल्ट) | कभी उद्धृत न करें; सामान्य संदेश के रूप में भेजें                           |
| `"first"`         | केवल पहले आउटबाउंड उत्तर खंड को उद्धृत करें                      |
| `"all"`           | प्रत्येक आउटबाउंड उत्तर खंड को उद्धृत करें                               |
| `"batched"`       | कतारबद्ध बैच उत्तरों को उद्धृत करें; तत्काल उत्तरों को बिना उद्धरण छोड़ें |

प्रति-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.replyToMode`।

```json5
{ channels: { whatsapp: { replyToMode: "first" } } }
```

## प्रतिक्रिया स्तर

`channels.whatsapp.reactionLevel` नियंत्रित करता है कि एजेंट इमोजी प्रतिक्रियाओं का कितने व्यापक रूप से उपयोग करता है:

| स्तर                 | अभिस्वीकृति प्रतिक्रियाएँ | एजेंट द्वारा आरंभ की गई प्रतिक्रियाएँ  |
| --------------------- | ------------- | -------------------------- |
| `"off"`               | नहीं            | नहीं                         |
| `"ack"`               | हाँ           | नहीं                         |
| `"minimal"` (डिफ़ॉल्ट) | हाँ           | हाँ, संयमित मार्गदर्शन |
| `"extensive"`         | हाँ           | हाँ, प्रोत्साहित मार्गदर्शन   |

प्रति-अकाउंट ओवरराइड: `channels.whatsapp.accounts.<id>.reactionLevel`।

```json5
{ channels: { whatsapp: { reactionLevel: "ack" } } }
```

## अभिस्वीकृति प्रतिक्रियाएँ

`channels.whatsapp.ackReaction` इनबाउंड प्राप्ति पर तत्काल प्रतिक्रिया भेजता है, जिसे `reactionLevel` नियंत्रित करता है (`"off"` होने पर दबा दिया जाता है):

```json5
{
  channels: {
    whatsapp: {
      ackReaction: {
        emoji: "👀",
        direct: true,
        group: "mentions", // हमेशा | उल्लेख | कभी नहीं
      },
    },
  },
}
```

टिप्पणियाँ: इनबाउंड स्वीकार होने के तुरंत बाद (उत्तर से पहले) भेजा जाता है; यदि `ackReaction` बिना `emoji` के मौजूद है, तो WhatsApp रूट किए गए एजेंट की पहचान इमोजी का उपयोग करता है और उपलब्ध न होने पर "👀" का फ़ॉलबैक लेता है (अभिस्वीकृति न भेजने के लिए `ackReaction` छोड़ें या `emoji: ""` सेट करें); विफलताएँ लॉग होती हैं लेकिन उत्तर डिलीवरी को अवरुद्ध नहीं करतीं; समूह मोड `mentions` केवल उल्लेख-ट्रिगर किए गए टर्न पर प्रतिक्रिया करता है, जबकि समूह सक्रियण `always` उस जाँच को बायपास करता है; WhatsApp केवल `channels.whatsapp.ackReaction` का उपयोग करता है (पुराना `messages.ackReaction` यहाँ लागू नहीं होता)।

## जीवनचक्र स्थिति प्रतिक्रियाएँ

`messages.statusReactions.enabled: true` सेट करें ताकि WhatsApp किसी टर्न के दौरान स्थिर प्राप्ति इमोजी छोड़ने के बजाय अभिस्वीकृति प्रतिक्रिया को बदल सके और कतारबद्ध, विचाररत, टूल गतिविधि, Compaction, पूर्ण और त्रुटि जैसी स्थितियों से क्रमशः गुज़रे:

```json5
{
  messages: {
    statusReactions: {
      enabled: true,
      emojis: {
        deploy: "🛫",
        build: "🏗️",
        concierge: "💁",
      },
    },
  },
}
```

टिप्पणियाँ: `channels.whatsapp.ackReaction` अभी भी सीधे संदेशों और समूहों के लिए पात्रता नियंत्रित करता है; कतारबद्ध स्थिति सामान्य ack प्रतिक्रियाओं के समान प्रभावी इमोजी का उपयोग करती है; WhatsApp में प्रति संदेश एक bot प्रतिक्रिया स्लॉट होता है, इसलिए जीवनचक्र अपडेट वर्तमान प्रतिक्रिया को उसी स्थान पर बदल देते हैं; `messages.removeAckAfterReply: true` कॉन्फ़िगर किए गए पूर्ण/त्रुटि होल्ड के बाद अंतिम स्थिति प्रतिक्रिया हटा देता है; टूल इमोजी श्रेणियों में `tool`, `coding`, `web`, `deploy`, `build`, और `concierge` शामिल हैं।

## टूल, कार्रवाइयाँ और कॉन्फ़िगरेशन लेखन

- एजेंट टूल समर्थन में WhatsApp प्रतिक्रिया कार्रवाई (`react`) शामिल है।
- कार्रवाई गेट: `channels.whatsapp.actions.reactions`, `channels.whatsapp.actions.polls` (मौजूदा कार्रवाइयों का डिफ़ॉल्ट `true` है), `channels.whatsapp.actions.calls` (डिफ़ॉल्ट `false`, ऊपर MeowCaller देखें)।
- चैनल द्वारा आरंभ किए गए कॉन्फ़िगरेशन लेखन डिफ़ॉल्ट रूप से सक्षम होते हैं; `channels.whatsapp.configWrites: false` के माध्यम से अक्षम करें।

  <Accordion title="समूह संदेश अप्रत्याशित रूप से अनदेखे किए जाते हैं">
    इस क्रम में जाँचें: `groupPolicy`, `groupAllowFrom`/`allowFrom`, `groups` अनुमति-सूची प्रविष्टियाँ, उल्लेख गेटिंग (`requireMention` + उल्लेख पैटर्न), और `openclaw.json` में डुप्लिकेट कुंजियाँ (JSON5 की बाद की प्रविष्टियाँ पहले वाली प्रविष्टियों को ओवरराइड करती हैं — प्रत्येक स्कोप में केवल एक `groupPolicy` रखें)।

    यदि `channels.whatsapp.groups` मौजूद है, तो WhatsApp अब भी अन्य समूहों के संदेश देख सकता है, लेकिन OpenClaw उन्हें सत्र रूटिंग से पहले हटा देता है। समूह JID को `channels.whatsapp.groups` में जोड़ें, या प्रेषक प्राधिकरण को `groupPolicy`/`groupAllowFrom` के अधीन रखते हुए सभी समूहों को अनुमति देने के लिए `groups["*"]` जोड़ें।

  </Accordion>

  <Accordion title="Bun रनटाइम चेतावनी">
    OpenClaw Gateway को Node की आवश्यकता होती है। Bun, कैनोनिकल स्थिति स्टोर द्वारा उपयोग किया जाने वाला `node:sqlite` API उपलब्ध नहीं कराता, और doctor विरासत Bun सेवाओं को Node पर माइग्रेट करता है।
  </Accordion>
</AccordionGroup>

## सिस्टम प्रॉम्प्ट

WhatsApp, `groups` और `direct` मैप के माध्यम से समूहों और सीधी चैट के लिए Telegram-शैली के सिस्टम प्रॉम्प्ट का समर्थन करता है।

समूह संदेशों का समाधान: प्रभावी `groups` मैप पहले निर्धारित किया जाता है — यदि खाता अपनी `groups` कुंजी किसी भी रूप में परिभाषित करता है, तो वह रूट `groups` मैप को पूरी तरह बदल देता है (कोई डीप मर्ज नहीं)। इसके बाद प्रॉम्प्ट लुकअप उसी एक परिणामी मैप पर चलता है:

1. **समूह-विशिष्ट प्रॉम्प्ट** (`groups["<groupId>"].systemPrompt`): इसका उपयोग तब होता है जब समूह प्रविष्टि मौजूद हो **और** उसकी `systemPrompt` कुंजी परिभाषित हो। रिक्त स्ट्रिंग (`""`) वाइल्डकार्ड को दबाती है और कोई प्रॉम्प्ट लागू नहीं करती।
2. **समूह वाइल्डकार्ड प्रॉम्प्ट** (`groups["*"].systemPrompt`): इसका उपयोग तब होता है जब विशिष्ट समूह प्रविष्टि अनुपस्थित हो, या `systemPrompt` कुंजी के बिना मौजूद हो।

सीधे संदेशों का समाधान `direct` मैप और `direct["*"]` पर इसी समान पैटर्न का अनुसरण करता है।

<Note>
`dms` हल्का प्रति-DM इतिहास ओवरराइड बकेट (`dms.<id>.historyLimit`) बना रहता है। प्रॉम्प्ट ओवरराइड `direct` के अंतर्गत रहते हैं।
</Note>

<Note>
प्रॉम्प्ट समाधान के लिए खाते द्वारा रूट बदलने का यह व्यवहार एक सामान्य शैलो ओवरराइड है: स्पष्ट रिक्त ऑब्जेक्ट सहित खाते की कोई भी `groups`/`direct` कुंजी रूट मैप को बदल देती है। यह ऊपर वर्णित समूह-सदस्यता अनुमति-सूची जाँच से अलग है, जिसमें गलती से रिक्त `groups: {}` के लिए एकल-खाता सुरक्षा व्यवस्था है।
</Note>

**Telegram से अंतर:** Telegram, एकाधिक-खाता सेटअप में प्रत्येक खाते के लिए रूट `groups` को दबाता है (उन खातों के लिए भी जिनका अपना कोई `groups` नहीं है), ताकि bot उन समूहों के संदेश प्राप्त न करे जिनका वह सदस्य नहीं है। WhatsApp यह सुरक्षा लागू नहीं करता — अपनी ओवरराइड के बिना कोई भी खाता, खातों की संख्या चाहे जो हो, रूट `groups`/`direct` को इनहेरिट करता है। एकाधिक-खाता WhatsApp सेटअप में, यदि आप प्रति-खाता प्रॉम्प्ट चाहते हैं, तो प्रत्येक खाते के अंतर्गत पूरा मैप स्पष्ट रूप से परिभाषित करें।

महत्वपूर्ण व्यवहार:

- `channels.whatsapp.groups` प्रति-समूह कॉन्फ़िगरेशन मैप और चैट-स्तरीय समूह अनुमति-सूची दोनों है। रूट या खाता स्कोप में, `groups["*"]` का अर्थ उस स्कोप के लिए "सभी समूह अनुमत हैं" होता है।
- वाइल्डकार्ड `systemPrompt` केवल तभी जोड़ें जब आप पहले से ही उस स्कोप में सभी समूहों को अनुमति देना चाहते हों। केवल समूह आईडी के निश्चित सेट को पात्र रखने के लिए, `groups["*"]` का उपयोग करने के बजाय प्रत्येक स्पष्ट रूप से अनुमति-सूचीबद्ध प्रविष्टि पर प्रॉम्प्ट दोहराएँ।
- समूह प्रवेश और प्रेषक प्राधिकरण अलग-अलग जाँचें हैं। `groups["*"]` यह विस्तार करता है कि कौन-से समूह समूह हैंडलिंग तक पहुँचते हैं; यह उन समूहों के प्रत्येक प्रेषक को अधिकृत नहीं करता — वह अब भी `groupPolicy`/`groupAllowFrom` द्वारा नियंत्रित होता है।
- DM के लिए `channels.whatsapp.direct` का कोई समकक्ष दुष्प्रभाव नहीं है: `direct["*"]` केवल तब डिफ़ॉल्ट कॉन्फ़िगरेशन प्रदान करता है जब DM को `dmPolicy` के साथ `allowFrom` या पेयरिंग-स्टोर नियमों द्वारा पहले ही अनुमति मिल चुकी हो।

उदाहरण:

```json5
{
  channels: {
    whatsapp: {
      groups: {
        // केवल तभी उपयोग करें जब रूट स्कोप में सभी समूहों को अनुमति मिलनी चाहिए।
        // उन सभी खातों पर लागू होता है जो अपना groups मैप परिभाषित नहीं करते।
        "*": { systemPrompt: "सभी समूहों के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
      },
      direct: {
        // उन सभी खातों पर लागू होता है जो अपना direct मैप परिभाषित नहीं करते।
        "*": { systemPrompt: "सभी सीधी चैट के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
      },
      accounts: {
        work: {
          groups: {
            // यह खाता अपना groups परिभाषित करता है, इसलिए रूट groups पूरी तरह
            // बदल दिए जाते हैं। वाइल्डकार्ड रखने के लिए, यहाँ भी "*" स्पष्ट रूप से परिभाषित करें।
            "120363406415684625@g.us": {
              requireMention: false,
              systemPrompt: "परियोजना प्रबंधन पर ध्यान केंद्रित करें।",
            },
            // केवल तभी उपयोग करें जब इस खाते में सभी समूहों को अनुमति मिलनी चाहिए।
            "*": { systemPrompt: "कार्य समूहों के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
          },
          direct: {
            // यह खाता अपना direct मैप परिभाषित करता है, इसलिए रूट direct प्रविष्टियाँ
            // पूरी तरह बदल दी जाती हैं। वाइल्डकार्ड रखने के लिए, यहाँ भी "*" स्पष्ट रूप से परिभाषित करें।
            "+15551234567": { systemPrompt: "किसी विशिष्ट कार्य-संबंधी सीधी चैट के लिए प्रॉम्प्ट।" },
            "*": { systemPrompt: "कार्य-संबंधी सीधी चैट के लिए डिफ़ॉल्ट प्रॉम्प्ट।" },
          },
        },
      },
    },
  },
}
```
