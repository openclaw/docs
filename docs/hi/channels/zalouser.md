स्थिति: प्रायोगिक। यह एकीकरण मूल `zca-js` के माध्यम से, प्रक्रिया के भीतर और किसी बाहरी CLI बाइनरी के बिना, एक **व्यक्तिगत Zalo खाते** को स्वचालित करता है।

<Warning>
यह एक अनौपचारिक एकीकरण है और इसके परिणामस्वरूप खाता निलंबित या प्रतिबंधित हो सकता है। इसका उपयोग अपने जोखिम पर करें।
</Warning>

## इंस्टॉल करें

Zalo Personal एक आधिकारिक बाहरी Plugin है, जो कोर के साथ बंडल नहीं है। उपयोग से पहले इसे इंस्टॉल करें:

```bash
openclaw plugins install @openclaw/zalouser
```

- किसी संस्करण को पिन करें: `openclaw plugins install @openclaw/zalouser@<version>`
- स्रोत चेकआउट से: `openclaw plugins install ./path/to/local/zalouser-plugin`
- विवरण: [Plugins](/hi/tools/plugin)

## त्वरित सेटअप

1. Plugin इंस्टॉल करें (ऊपर)।
2. लॉग इन करें (QR, Gateway मशीन पर):
   - `openclaw channels login --channel zalouser`
   - Zalo मोबाइल ऐप से QR कोड स्कैन करें।
3. चैनल सक्षम करें:

```json5
{
  channels: {
    zalouser: {
      enabled: true,
      dmPolicy: "pairing",
    },
  },
}
```

4. Gateway को पुनः आरंभ करें (या सेटअप पूरा करें)।
5. DM पहुँच डिफ़ॉल्ट रूप से पेयरिंग का उपयोग करती है; पहले संपर्क पर पेयरिंग कोड अनुमोदित करें।

## यह क्या है

- `zca-js` लाइब्रेरी के माध्यम से पूरी तरह प्रक्रिया के भीतर चलता है (कोई बाहरी `zca`/`openzca` बाइनरी नहीं)।
- आने वाले संदेश प्राप्त करने के लिए मूल इवेंट लिसनर (`message`, `error`) का उपयोग करता है।
- JS API के माध्यम से सीधे उत्तर भेजता है (टेक्स्ट/मीडिया/लिंक)।
- उन "व्यक्तिगत खाते" वाले उपयोग के मामलों के लिए बनाया गया है, जहाँ Zalo Bot API उपलब्ध नहीं है।

## नामकरण

चैनल आईडी `zalouser` है, ताकि यह स्पष्ट रहे कि यह एक **व्यक्तिगत Zalo उपयोगकर्ता खाते** को स्वचालित करता है (अनौपचारिक)। `zalo` संभावित भावी आधिकारिक Zalo API एकीकरण के लिए आरक्षित है।

## आईडी ढूँढना (डायरेक्टरी)

```bash
openclaw directory self --channel zalouser
openclaw directory peers list --channel zalouser --query "name"

```

## सीमाएँ

- बाहर भेजे जाने वाले टेक्स्ट को 2000 वर्णों के खंडों में बाँटा जाता है (Zalo क्लाइंट सीमा)।
- स्ट्रीमिंग समर्थित नहीं है।
- पूर्ण हो चुके आने वाले संदेशों की आईडी 30 दिनों तक रखी जाती हैं और प्रति खाते सबसे हाल की 1000 प्रविष्टियों तक सीमित रहती हैं।

## आने वाले संदेशों की टिकाऊता

OpenClaw प्रत्येक कच्चे `zca-js` संदेश कॉलबैक को संसाधित करने से पहले संग्रहीत करता है। Gateway के पुनः आरंभ होने के बाद लंबित संदेश खाते की कतार से फिर शुरू होते हैं और प्रत्येक सीधे चैट या समूह के लिए संसाधन क्रमिक बना रहता है।

`zca-js` सॉकेट लिसनर डिलीवरी अभिस्वीकृति उपलब्ध नहीं कराता और पुनः कनेक्ट होने के बाद पुराने संदेशों को स्वचालित रूप से दोबारा नहीं चलाता। इसलिए टिकाऊ कतार किसी कॉलबैक के OpenClaw तक पहुँचने के बाद स्थानीय क्रैश अवधि से सुरक्षा देती है; वह उस संदेश को पुनर्प्राप्त नहीं कर सकती जिसे सॉकेट ने कभी डिलीवर ही नहीं किया। रीप्ले टूम्बस्टोन मुख्यतः समान Zalo संदेश आईडी वाले दोहराए गए कॉलबैक से सुरक्षा देते हैं।

## पहुँच नियंत्रण (DM)

`channels.zalouser.dmPolicy`: `pairing | allowlist | open | disabled` (डिफ़ॉल्ट: `pairing`)।

`channels.zalouser.allowFrom` में स्थिर Zalo उपयोगकर्ता आईडी का उपयोग किया जाना चाहिए। यह स्थिर प्रेषक पहुँच समूहों (`accessGroup:<name>`) को भी संदर्भित कर सकता है। इंटरैक्टिव सेटअप के दौरान, दर्ज किए गए नामों को Plugin की प्रक्रिया-अंतर्गत संपर्क खोज का उपयोग करके आईडी में बदला जा सकता है।

यदि कॉन्फ़िगरेशन में कोई कच्चा नाम रह जाता है, तो स्टार्टअप उसे केवल तभी हल करता है जब `channels.zalouser.dangerouslyAllowNameMatching: true` सक्षम हो। इस वैकल्पिक अनुमति के बिना, रनटाइम प्रेषक जाँच केवल आईडी पर आधारित होती है और प्राधिकरण के लिए कच्चे नामों को अनदेखा किया जाता है।

इसके माध्यम से अनुमोदित करें:

- `openclaw pairing list zalouser`
- `openclaw pairing approve zalouser <code>`

## समूह पहुँच (वैकल्पिक)

- डिफ़ॉल्ट: `channels.zalouser.groupPolicy = "allowlist"` (समूहों के लिए स्पष्ट अनुमत-सूची प्रविष्टि आवश्यक है)।
- सभी समूह खोलें: `channels.zalouser.groupPolicy = "open"`।
- सभी समूह अवरुद्ध करें: `channels.zalouser.groupPolicy = "disabled"`।
- `groupPolicy = "allowlist"` के साथ:
  - `channels.zalouser.groups` कुंजियाँ स्थिर समूह आईडी होनी चाहिए; नामों को स्टार्टअप पर केवल तभी आईडी में बदला जाता है जब `channels.zalouser.dangerouslyAllowNameMatching: true` सक्षम हो।
  - `channels.zalouser.groupAllowFrom` नियंत्रित करता है कि अनुमत समूहों में कौन-से प्रेषक बॉट को सक्रिय कर सकते हैं; स्थिर प्रेषक पहुँच समूहों को `accessGroup:<name>` से संदर्भित किया जा सकता है।
- कॉन्फ़िगरेशन विज़ार्ड समूह अनुमत-सूचियों के लिए संकेत दे सकता है।
- समूह अनुमत-सूची मिलान डिफ़ॉल्ट रूप से केवल आईडी पर आधारित है। जब तक `channels.zalouser.dangerouslyAllowNameMatching: true` सक्षम न हो, अनसुलझे नामों को प्रमाणीकरण के लिए अनदेखा किया जाता है।
- `channels.zalouser.dangerouslyAllowNameMatching: true` एक आपातकालीन संगतता मोड है, जो स्टार्टअप पर परिवर्तनशील नाम समाधान और रनटाइम समूह-नाम मिलान को फिर से सक्षम करता है।
- `groupAllowFrom` सामान्य समूह संदेशों के लिए `allowFrom` पर **वापस नहीं जाता**: किसी अनुमत-सूचीबद्ध समूह में इसे खाली छोड़ने पर वह समूह किसी भी प्रेषक के लिए खुल जाता है। अधिकृत नियंत्रण कमांड (उदाहरण के लिए `/new`) अपवाद हैं; `groupAllowFrom` खाली होने पर कमांड प्रेषक जाँच `allowFrom` पर वापस जाती है।

उदाहरण:

```json5
{
  channels: {
    zalouser: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["1471383327500481391"],
      groups: {
        "123456789": { enabled: true },
        "Work Chat": { enabled: true },
      },
    },
  },
}
```

<Note>
`channels.zalouser.groups.<id>.allow` एक पुराना फ़ील्ड नाम है; वर्तमान कॉन्फ़िगरेशन `enabled` का उपयोग करता है। `openclaw doctor --fix`, `allow` को स्वचालित रूप से `enabled` में माइग्रेट करता है।
</Note>

### समूह उल्लेख गेटिंग

- `channels.zalouser.groups.<group>.requireMention` नियंत्रित करता है कि समूह उत्तरों के लिए उल्लेख आवश्यक है या नहीं।
- समाधान क्रम: समूह आईडी -> `group:<id>` उपनाम -> समूह नाम/स्लग (नाम-आधारित उम्मीदवार केवल तभी लागू होते हैं जब `dangerouslyAllowNameMatching: true`) -> `*` -> डिफ़ॉल्ट (`true`)।
- यह अनुमत-सूचीबद्ध समूहों और खुले समूह मोड, दोनों पर लागू होता है।
- किसी बॉट संदेश को उद्धृत करना समूह सक्रियण के लिए निहित उल्लेख माना जाता है।
- अधिकृत नियंत्रण कमांड (उदाहरण के लिए `/new`) उल्लेख गेटिंग को बायपास कर सकते हैं।
- जब उल्लेख आवश्यक होने के कारण किसी समूह संदेश को छोड़ दिया जाता है, तो OpenClaw उसे लंबित समूह इतिहास के रूप में संग्रहीत करता है और अगले संसाधित समूह संदेश में शामिल करता है।
- समूह इतिहास सीमा: `channels.zalouser.historyLimit`, फिर `messages.groupChat.historyLimit`, फिर `50` का फ़ॉलबैक।

उदाहरण:

```json5
{
  channels: {
    zalouser: {
      groupPolicy: "allowlist",
      groups: {
        "*": { enabled: true, requireMention: true },
        "Work Chat": { enabled: true, requireMention: false },
      },
    },
  },
}
```

## बहु-खाता

OpenClaw स्थिति में खाते `zalouser` प्रोफ़ाइल से मैप होते हैं। उदाहरण:

```json5
{
  channels: {
    zalouser: {
      enabled: true,
      defaultAccount: "default",
      accounts: {
        work: { enabled: true, profile: "work" },
      },
    },
  },
}
```

## पर्यावरण चर

प्रोफ़ाइल चयन पर्यावरण चरों से भी आ सकता है:

| चर                | उद्देश्य                                                                    |
| ------------------ | -------------------------------------------------------------------------- |
| `ZALOUSER_PROFILE` | चैनल या खाता कॉन्फ़िगरेशन में कोई `profile` सेट न होने पर उपयोग किया जाने वाला प्रोफ़ाइल नाम। |
| `ZCA_PROFILE`      | पुराना फ़ॉलबैक, केवल तब उपयोग किया जाता है जब `ZALOUSER_PROFILE` सेट न हो।             |

प्रोफ़ाइल नाम OpenClaw स्थिति में सहेजे गए Zalo लॉगिन क्रेडेंशियल चुनते हैं। समाधान क्रम:

1. कॉन्फ़िगरेशन में स्पष्ट `profile`।
2. `ZALOUSER_PROFILE`।
3. `ZCA_PROFILE`।
4. गैर-डिफ़ॉल्ट खातों के लिए खाता आईडी, या डिफ़ॉल्ट खाते के लिए `default`।

बहु-खाता सेटअप के लिए, कॉन्फ़िगरेशन में प्रत्येक खाते पर `profile` सेट करना बेहतर है, ताकि एक पर्यावरण चर के कारण कई खाते समान लॉगिन सत्र साझा न करें।

## समस्या निवारण

**लॉगिन कायम नहीं रहता:**

- `openclaw channels status --probe`
- फिर से लॉग इन करें: `openclaw channels logout --channel zalouser && openclaw channels login --channel zalouser`

**अनुमत-सूची/समूह नाम हल नहीं हुआ:**

- `allowFrom`/`groupAllowFrom` में संख्यात्मक आईडी और `groups` में स्थिर समूह आईडी का उपयोग करें। यदि आपको जानबूझकर सटीक मित्र/समूह नामों की आवश्यकता है, तो `channels.zalouser.dangerouslyAllowNameMatching: true` सक्षम करें।

**किसी पुराने बाहरी `zca`/CLI-आधारित सेटअप से अपग्रेड किया है:**

- बाहरी `zca` प्रक्रिया से संबंधित सभी धारणाएँ हटाएँ; चैनल अब `zca-js` के माध्यम से, बिना किसी बाहरी CLI बाइनरी के, पूरी तरह प्रक्रिया के भीतर चलता है।
