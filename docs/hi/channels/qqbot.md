QQ Bot आधिकारिक QQ Bot API (WebSocket gateway) के माध्यम से OpenClaw से जुड़ता है।
C2C निजी चैट और समूह `@`-मेंशन प्राथमिक चैट प्रकार हैं, जिनमें समृद्ध
मीडिया (चित्र, वॉइस, वीडियो, फ़ाइलें) उपलब्ध है। गिल्ड चैनल संदेशों में केवल
टेक्स्ट और रिमोट-URL चित्र समर्थित हैं; वॉइस, वीडियो, फ़ाइल अपलोड और स्थानीय/Base64
चित्र गिल्ड चैनलों में उपलब्ध नहीं हैं। रिएक्शन और थ्रेड कहीं भी
समर्थित नहीं हैं।

स्थिति: आधिकारिक डाउनलोड योग्य plugin।

## इंस्टॉल करें

```bash
openclaw plugins install @openclaw/qqbot
```

## सेटअप

1. [QQ Open Platform](https://q.qq.com/) पर जाएँ और पंजीकरण / लॉग इन करने के लिए अपने
   फ़ोन के QQ से QR कोड स्कैन करें।
2. नया QQ bot बनाने के लिए **Create Bot** पर क्लिक करें।
3. bot के सेटिंग पेज पर **AppID** और **AppSecret** खोजें और उन्हें कॉपी करें।

<Note>
AppSecret सादे टेक्स्ट में संग्रहीत नहीं होता। यदि आप इसे सहेजे बिना पेज छोड़ देते हैं, तो आपको नया AppSecret फिर से जनरेट करना होगा।
</Note>

4. चैनल जोड़ें:

```bash
openclaw channels add --channel qqbot --token "AppID:AppSecret"
```

5. Gateway पुनः आरंभ करें।

इंटरैक्टिव सेटअप:

```bash
openclaw channels add
```

विज़ार्ड में AppID/AppSecret को मैन्युअल रूप से टाइप करने के विकल्प के रूप में QR-कोड
बाइंडिंग भी उपलब्ध है: बाइंडिंग पूरी करने के लिए लक्षित QQ Bot से संबद्ध फ़ोन ऐप से
कोड स्कैन करें। OpenClaw लौटाए गए क्रेडेंशियल को खाते के कॉन्फ़िगरेशन
स्कोप में स्थायी रूप से सहेजता है।

## कॉन्फ़िगर करें

न्यूनतम कॉन्फ़िगरेशन:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "YOUR_APP_ID",
      clientSecret: "YOUR_APP_SECRET",
    },
  },
}
```

डिफ़ॉल्ट खाते के एनवायरनमेंट वेरिएबल (केवल शीर्ष-स्तरीय खाता):

- `QQBOT_APP_ID`
- `QQBOT_CLIENT_SECRET`

फ़ाइल-समर्थित AppSecret:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "YOUR_APP_ID",
      clientSecretFile: "/path/to/qqbot-secret.txt",
    },
  },
}
```

एनवायरनमेंट SecretRef AppSecret:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "YOUR_APP_ID",
      clientSecret: { source: "env", provider: "default", id: "QQBOT_CLIENT_SECRET" },
    },
  },
}
```

नोट:

- `openclaw channels add --channel qqbot --token-file ...` केवल AppSecret सेट करता है;
  `appId` पहले से कॉन्फ़िगरेशन या `QQBOT_APP_ID` में सेट होना चाहिए।
- `clientSecret` सादे टेक्स्ट की स्ट्रिंग, फ़ाइल पथ (`clientSecretFile`),
  या संरचित SecretRef ऑब्जेक्ट स्वीकार करता है।
- पुराने `secretref:...` / `secretref-env:...` मार्कर स्ट्रिंग
  `clientSecret` के लिए अस्वीकार किए जाते हैं; इसके बजाय संरचित SecretRef ऑब्जेक्ट का उपयोग करें।

### स्ट्रीमिंग

```json5
{
  channels: {
    qqbot: {
      streaming: {
        mode: "partial", // ब्लॉक स्ट्रीमिंग: "partial" (डिफ़ॉल्ट) या "off"
        nativeTransport: true, // DM के लिए QQ के आधिकारिक C2C stream_messages API का उपयोग करें
      },
    },
  },
}
```

- `streaming.mode: "off"` खाते के लिए ब्लॉक स्ट्रीमिंग अक्षम करता है।
- `streaming.nativeTransport: true` C2C (DM) उत्तरों को QQ के
  आधिकारिक `stream_messages` API के माध्यम से स्ट्रीम करता है; समूह/चैनल लक्ष्य अप्रभावित रहते हैं।
- पुराने `streaming: true|false` स्केलर और `streaming.c2cStreamApi` कुंजी
  `openclaw doctor --fix` के माध्यम से इस संरचना में माइग्रेट होते हैं।
- `/bot-streaming on|off` DM से इसी कॉन्फ़िगरेशन को टॉगल करता है।

### पहुँच नीति

- `allowFrom` / `groupAllowFrom` नियंत्रित करते हैं कि C2C /
  समूह संदर्भों में bot से कौन चैट कर सकता है। `dmPolicy` / `groupPolicy` (`open` | `allowlist` | `disabled`)
  प्रवर्तन मोड नियंत्रित करते हैं। `allowFrom` में कोई ठोस (गैर-वाइल्डकार्ड)
  प्रविष्टि होने पर `dmPolicy` का डिफ़ॉल्ट `allowlist` होता है,
  अन्यथा `open`। `groupAllowFrom` या
  `allowFrom` में से किसी में ठोस प्रविष्टि होने पर `groupPolicy` का
  डिफ़ॉल्ट `allowlist` होता है, अन्यथा `open`।
- "Auth: allowlist" स्लैश कमांड के लिए
  `allowFrom` (या समूह से आह्वान के लिए `groupAllowFrom`) में स्पष्ट गैर-वाइल्डकार्ड प्रविष्टि आवश्यक है,
  चाहे `dmPolicy` / `groupPolicy` कुछ भी हों — [स्लैश कमांड](#slash-commands) देखें।

### बहु-खाता सेटअप

एक OpenClaw इंस्टेंस के अंतर्गत एकाधिक QQ bot चलाएँ:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "111111111",
      clientSecret: "secret-of-bot-1",
      accounts: {
        bot2: {
          enabled: true,
          appId: "222222222",
          clientSecret: "secret-of-bot-2",
        },
      },
    },
  },
}
```

प्रत्येक खाते का एक पृथक WebSocket कनेक्शन, API क्लाइंट और टोकन
कैश होता है, जिसकी कुंजी `appId` होती है। लॉग पंक्तियों को स्वामी खाते की id से टैग किया जाता है, ताकि
एक Gateway के अंतर्गत कई bot चलाने पर निदान अलग-अलग बना रहे।

CLI के माध्यम से दूसरा bot जोड़ें:

```bash
openclaw channels add --channel qqbot --account bot2 --token "222222222:secret-of-bot-2"
```

### समूह चैट

समूह समर्थन डिस्प्ले नामों के बजाय QQ समूह OpenID का उपयोग करता है। bot को किसी
समूह में जोड़ें, फिर उसका उल्लेख करें या बिना उल्लेख के चलने के लिए समूह को कॉन्फ़िगर करें।

```json5
{
  channels: {
    qqbot: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["member_openid"],
      groups: {
        "*": {
          requireMention: true,
          commandLevel: "all",
          historyLimit: 50,
          tools: { deny: ["exec", "read", "write"] },
        },
        GROUP_OPENID: {
          name: "Release room",
          requireMention: false,
          ignoreOtherMentions: true,
          commandLevel: "safety",
          historyLimit: 20,
          prompt: "Keep replies short and operational.",
        },
      },
    },
  },
}
```

`groups["*"]` प्रत्येक समूह के लिए डिफ़ॉल्ट सेट करता है; कोई ठोस `groups.GROUP_OPENID`
प्रविष्टि एक समूह के लिए उन डिफ़ॉल्ट को ओवरराइड करती है। समूह सेटिंग:

| फ़ील्ड                 | डिफ़ॉल्ट          | विवरण                                                                                        |
| --------------------- | ---------------- | -------------------------------------------------------------------------------------------------- |
| `requireMention`      | `true`           | bot के उत्तर देने से पहले `@`-मेंशन आवश्यक बनाएँ।                                                     |
| `commandLevel`        | `all`            | समूह में कौन-से अंतर्निर्मित स्लैश कमांड चल सकते हैं (नीचे देखें)।                                    |
| `ignoreOtherMentions` | `false`          | ऐसे संदेश हटा दें जिनमें किसी अन्य व्यक्ति का उल्लेख हो, लेकिन bot का नहीं।                                           |
| `historyLimit`        | `50`             | अगली उल्लेख वाली बारी के संदर्भ के रूप में रखे गए हाल के गैर-मेंशन संदेश। `0` इतिहास अक्षम करता है।     |
| `tools`               | —                | पूरे समूह के लिए टूल को अनुमति दें/अस्वीकार करें।                                                              |
| `toolsBySender`       | —                | प्रत्येक प्रेषक के लिए टूल ओवरराइड; [समूह](/hi/channels/groups#groupchannel-tool-restrictions-optional) देखें। |
| `name`                | openid उपसर्ग    | लॉग और समूह संदर्भ में उपयोग किया जाने वाला सुविधाजनक लेबल।                                                     |
| `prompt`              | अंतर्निर्मित डिफ़ॉल्ट | एजेंट संदर्भ में जोड़ा गया प्रति-समूह व्यवहार प्रॉम्प्ट।                                           |

`commandLevel` स्वीकार करता है:

| स्तर    | व्यवहार                                                                                                                                      |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `all`    | मौजूदा अंतर्निर्मित कमांड उपलब्ध रहते हैं। कुछ मेनू से छिपे रहते हैं, लेकिन अधिकृत उपयोगकर्ता उन्हें समूह में फिर भी चला सकते हैं।                  |
| `safety` | `/help`, `/btw`, `/stop` समूह में दिखाई देते रहते हैं; संवेदनशील कमांड (`/config`, `/tools`, `/bash`, आदि) निजी चैट में चलाने आवश्यक हैं।      |
| `strict` | केवल कड़े संचालन के लिए आवश्यक समूह-सत्र नियंत्रणों की अनुमति है। `/stop` फिर भी काम करता है, ताकि कोई अधिकृत प्रेषक सक्रिय रन को बाधित कर सके। |

पुरानी QQBot `toolPolicy` प्रविष्टियाँ बंद कर दी गई हैं। उन्हें `tools` में माइग्रेट करने के लिए `openclaw doctor --fix` चलाएँ।

सक्रियण मोड `mention` और `always` हैं। `requireMention: true`,
`mention` से मैप होता है; `requireMention: false`, `always` से मैप होता है। सत्र-स्तरीय सक्रियण
ओवरराइड मौजूद होने पर कॉन्फ़िगरेशन पर प्राथमिकता लेता है।

इनबाउंड क्यू प्रत्येक पीयर के लिए अलग होती है। समूह पीयर को अधिक बड़ी क्यू सीमा मिलती है (प्रत्यक्ष
पीयर के लिए 20 की तुलना में 50), क्यू भरने पर मानव द्वारा लिखे संदेशों से पहले bot द्वारा लिखे संदेशों को
हटाया जाता है, और सामान्य समूह संदेशों के बर्स्ट को एक एट्रिब्यूटेड बारी में मर्ज किया जाता है। स्लैश
कमांड किसी भी मर्ज बैच से स्वतंत्र होकर एक-एक करके चलते हैं।

### वॉइस (STT / TTS)

STT और TTS प्राथमिकता फ़ॉलबैक के साथ दो-स्तरीय कॉन्फ़िगरेशन का समर्थन करते हैं:

| सेटिंग | Plugin-विशिष्ट                                          | फ़्रेमवर्क फ़ॉलबैक            |
| ------- | -------------------------------------------------------- | ----------------------------- |
| STT     | `channels.qqbot.stt`                                     | `tools.media.audio.models[0]` |
| TTS     | `channels.qqbot.tts`, `channels.qqbot.accounts.<id>.tts` | `messages.tts`                |

```json5
{
  channels: {
    qqbot: {
      stt: {
        provider: "your-provider",
        model: "your-stt-model",
      },
      tts: {
        provider: "your-provider",
        model: "your-tts-model",
        voice: "your-voice",
      },
      accounts: {
        "qq-main": {
          tts: {
            providers: {
              openai: { voice: "shimmer" },
            },
          },
        },
      },
    },
  },
}
```

अक्षम करने के लिए इनमें से किसी पर `enabled: false` सेट करें। खाता-स्तरीय TTS ओवरराइड
`messages.tts` जैसी ही संरचना का उपयोग करते हैं और चैनल/वैश्विक TTS कॉन्फ़िगरेशन पर डीप-मर्ज होते हैं।

STT अनुरोध डिफ़ॉल्ट रूप से 60 सेकंड बाद टाइम आउट हो जाते हैं। Plugin-विशिष्ट STT चयनित
`models.providers.<id>.timeoutSeconds` ओवरराइड का उपयोग करता है। फ़्रेमवर्क ऑडियो STT
पहले `tools.media.audio.models[0].timeoutSeconds`, फिर
`tools.media.audio.timeoutSeconds`, और फिर चयनित प्रदाता ओवरराइड का उपयोग करता है।

इनबाउंड QQ वॉइस अटैचमेंट एजेंटों को ऑडियो मीडिया मेटाडेटा के रूप में उपलब्ध कराए जाते हैं,
जबकि रॉ वॉइस फ़ाइलों को सामान्य `MediaPaths` से बाहर रखा जाता है। सादे-टेक्स्ट उत्तर में `[[audio_as_voice]]`
TTS संश्लेषित करता है और TTS कॉन्फ़िगर होने पर मूल QQ वॉइस संदेश भेजता है।

आउटबाउंड ऑडियो अपलोड/ट्रांसकोड व्यवहार को
`channels.qqbot.audioFormatPolicy` से भी समायोजित किया जा सकता है:

- `sttDirectFormats`
- `uploadDirectFormats`
- `transcodeEnabled`

## लक्ष्य प्रारूप

| प्रारूप                     | विवरण        |
| -------------------------- | ------------------ |
| `qqbot:c2c:OPENID`         | निजी चैट (C2C) |
| `qqbot:group:GROUP_OPENID` | समूह चैट         |
| `qqbot:channel:CHANNEL_ID` | गिल्ड चैनल      |

<Note>
प्रत्येक bot के पास उपयोगकर्ता OpenID का अपना सेट होता है। Bot A द्वारा प्राप्त OpenID का उपयोग Bot B के माध्यम से संदेश भेजने के लिए **नहीं** किया जा सकता।
</Note>

| `/bot-approve`       | अनुमति-सूची | केवल निजी | कमांड-निष्पादन अनुमोदन कॉन्फ़िगरेशन प्रबंधित करें (चालू / बंद / हमेशा / रीसेट / स्थिति)  |
| `/bot-logs`          | अनुमति-सूची | केवल निजी | हाल के gateway लॉग फ़ाइल के रूप में निर्यात करें                                           |
| `/bot-clear-storage` | अनुमति-सूची | केवल निजी | QQBot मीडिया डायरेक्टरी के अंतर्गत कैश किए गए डाउनलोड हटाएँ                        |
| `/bot-streaming`     | अनुमति-सूची | केवल निजी | C2C स्ट्रीमिंग उत्तरों को चालू या बंद करें                                                   |
| `/bot-group-allways` | अनुमति-सूची | केवल निजी | डिफ़ॉल्ट समूह सक्रियण मोड को बदलें (उल्लेख-आवश्यक बनाम हमेशा-चालू)      |

उपयोग सहायता के लिए किसी भी कमांड के अंत में `?` जोड़ें (उदाहरण के लिए `/bot-upgrade ?`)।

"प्रमाणीकरण: अनुमति-सूची" वाले कमांड के लिए अतिरिक्त रूप से प्रेषक का openid
एक स्पष्ट गैर-वाइल्डकार्ड `allowFrom` सूची में होना आवश्यक है (समूह से जारी
कमांड के लिए `groupAllowFrom` को प्राथमिकता मिलती है और उपलब्ध न होने पर
`allowFrom` का उपयोग होता है)। वाइल्डकार्ड `allowFrom: ["*"]` चैट की अनुमति
देता है, लेकिन इन कमांड की नहीं। इनमें से किसी कमांड को निजी चैट के बाहर या
बिना प्राधिकरण के चलाने पर संदेश को चुपचाप छोड़ने के बजाय एक संकेत लौटाया जाता है।

`/bot-me`, `/bot-version`, और `/bot-upgrade` केवल निजी चैट के लिए हैं, लेकिन इनके लिए
अनुमति-सूची आवश्यक नहीं है — कोई भी C2C प्रेषक इन्हें चला सकता है।

जब QQ Bot निष्पादन अनुमोदन डिफ़ॉल्ट समान-चैट फ़ॉलबैक का उपयोग करते हैं, तो मूल अनुमोदन
बटन पर क्लिक भी उसी स्पष्ट गैर-वाइल्डकार्ड कमांड अनुमति-सूची का पालन करते हैं। व्यापक
कमांड पहुँच दिए बिना केवल अनुमोदन की पहुँच प्रदान करने के लिए
`channels.qqbot.execApprovals.approvers` कॉन्फ़िगर करें। मूल निष्पादन अनुमोदन डिफ़ॉल्ट रूप से
सक्षम होते हैं।

## मीडिया और संग्रहण

- आवक, जावक और Gateway-ब्रिज मीडिया `~/.openclaw/media/qqbot` के अंतर्गत एक ही पेलोड रूट साझा करते हैं
  (`OPENCLAW_HOME` सेट होने पर उसका पालन करते हुए), ताकि अपलोड,
  डाउनलोड और ट्रांसकोड कैश एक ही संरक्षित डायरेक्टरी में रहें।
- C2C और समूह लक्ष्यों के लिए रिच मीडिया डिलीवरी एक ही `sendMedia`
  पथ से होती है। 5&nbsp;MiB या उससे अधिक की स्थानीय फ़ाइलें और इन-मेमोरी बफ़र QQ के
  खंडित अपलोड एंडपॉइंट का उपयोग करते हैं; छोटे पेलोड और रिमोट-URL/Base64 स्रोत
  एकल-प्रयास अपलोड API का उपयोग करते हैं।
- यदि हॉट अपग्रेड `openclaw.json` का लेखन पूरा होने से पहले Gateway को बाधित कर देता है,
  तो Plugin अगली शुरुआत पर उस खाते के लिए अंतिम ज्ञात `appId` / `clientSecret`
  को आंतरिक स्नैपशॉट से पुनर्स्थापित करता है (जानबूझकर किए गए कॉन्फ़िगरेशन परिवर्तन को
  कभी अधिलेखित नहीं करता), इसलिए QR कोड को दोबारा स्कैन करना आवश्यक नहीं
  होता।

## समस्या निवारण

- **Gateway शुरू नहीं होता / कोई आवक संदेश नहीं:** सत्यापित करें कि `appId` और
  `clientSecret` सही हैं और QQ Open Platform पर बॉट सक्षम है।
  अनुपलब्ध क्रेडेंशियल "QQBot कॉन्फ़िगर नहीं है (appId या
  clientSecret अनुपलब्ध है)" के रूप में दिखाई देता है।
- **`--token-file` से सेटअप करने पर भी कॉन्फ़िगर नहीं दिखता:** `--token-file` केवल
  AppSecret सेट करता है। `appId` को अभी भी कॉन्फ़िगरेशन या `QQBOT_APP_ID` में सेट करना होगा।
- **तेज़ समूह उत्तर आपस में टकराते हैं:** किसी सहकर्मी की कतार भरने पर आवक कतार
  मानव-लिखित संदेशों से पहले बॉट-लिखित संदेशों को हटाती है और सामान्य
  (गैर-कमांड) समूह संदेशों के झुंड को प्रेषक-निर्दिष्ट एक टर्न में मिला देती है, ताकि
  बॉट संदेशों की बाढ़ मानव संदेशों को वंचित न करे।
- **सक्रिय रूप से भेजे गए संदेश नहीं पहुँच रहे:** यदि उपयोगकर्ता ने हाल में इंटरैक्ट नहीं किया है,
  तो QQ बॉट द्वारा आरंभ किए गए संदेशों को अवरुद्ध कर सकता है।
- **आवाज़ का लिप्यंतरण नहीं हुआ:** सुनिश्चित करें कि STT कॉन्फ़िगर है और प्रदाता तक
  पहुँचा जा सकता है।
