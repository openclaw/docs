"पेयरिंग" OpenClaw का स्पष्ट पहुँच-अनुमोदन चरण है।
इसका उपयोग दो स्थानों पर होता है:

1. **DM पेयरिंग** (बॉट से बात करने की अनुमति किसे है)
2. **Node पेयरिंग** (किन डिवाइस/Node को Gateway नेटवर्क में शामिल होने की अनुमति है)

सुरक्षा संदर्भ: [सुरक्षा](/hi/gateway/security)

## 1) DM पेयरिंग (इनबाउंड चैट पहुँच)

जब किसी चैनल को DM नीति `pairing` के साथ कॉन्फ़िगर किया जाता है, तो अज्ञात प्रेषकों को एक छोटा कोड मिलता है और आपके अनुमोदन तक उनका संदेश **प्रोसेस नहीं किया जाता**।

डिफ़ॉल्ट DM नीतियों का दस्तावेज़ यहाँ उपलब्ध है: [सुरक्षा](/hi/gateway/security)

`dmPolicy: "open"` केवल तभी सार्वजनिक होता है, जब प्रभावी DM अनुमति-सूची में `"*"` शामिल हो।
सार्वजनिक रूप से खुले कॉन्फ़िगरेशन के लिए सेटअप और सत्यापन में उस वाइल्डकार्ड की आवश्यकता होती है। यदि मौजूदा
स्थिति में ठोस `allowFrom` प्रविष्टियों के साथ `open` मौजूद है, तो रनटाइम फिर भी
केवल उन्हीं प्रेषकों को अनुमति देता है और पेयरिंग-स्टोर के अनुमोदन `open` पहुँच का विस्तार नहीं करते।

पेयरिंग कोड:

- 8 वर्ण, अपरकेस, कोई भ्रमित करने वाला वर्ण नहीं (`0O1I`)।
- **1 घंटे बाद समाप्त हो जाते हैं**। बॉट पेयरिंग संदेश केवल नया अनुरोध बनने पर भेजता है (लगभग प्रति प्रेषक प्रति घंटे एक बार)।
- लंबित DM पेयरिंग अनुरोधों की सीमा **प्रति चैनल खाते 3** है; अतिरिक्त अनुरोधों को तब तक अनदेखा किया जाता है, जब तक कोई अनुरोध समाप्त या अनुमोदित नहीं हो जाता।

### किसी प्रेषक को अनुमोदित करें

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

उसी चैनल पर अनुरोधकर्ता को सूचित करने के लिए अनुमोदन कमांड में `--notify` जोड़ें। एकाधिक-खाता चैनल `--account <id>` लेते हैं।

यदि अभी तक कोई कमांड स्वामी कॉन्फ़िगर नहीं है, तो DM पेयरिंग कोड को अनुमोदित करने से
`commands.ownerAllowFrom` भी अनुमोदित प्रेषक के रूप में बूटस्ट्रैप हो जाता है, जैसे `telegram:123456789`।
इससे पहली बार किए जाने वाले सेटअप को विशेषाधिकार प्राप्त कमांड और exec
अनुमोदन प्रॉम्प्ट के लिए एक स्पष्ट स्वामी मिलता है। स्वामी मौजूद होने के बाद, बाद के पेयरिंग अनुमोदन केवल DM
पहुँच देते हैं; वे और स्वामी नहीं जोड़ते।

समर्थित चैनल (कोई भी इंस्टॉल किया गया चैनल Plugin जो पेयरिंग घोषित करता है; `openclaw-weixin` जैसे बाहरी Plugin और जोड़ सकते हैं): `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `signal`, `slack`, `sms`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`।

### पुनः उपयोग योग्य प्रेषक समूह

जब विश्वसनीय प्रेषकों का वही समूह कई संदेश चैनलों या DM और समूह दोनों की
अनुमति-सूचियों पर लागू होना चाहिए, तो शीर्ष-स्तरीय `accessGroups` का उपयोग करें।

स्थिर समूह `type: "message.senders"` का उपयोग करते हैं और चैनल अनुमति-सूचियों से
`accessGroup:<name>` द्वारा संदर्भित किए जाते हैं:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

पहुँच समूहों का विस्तृत दस्तावेज़ यहाँ उपलब्ध है: [पहुँच समूह](/hi/channels/access-groups)

### स्थिति कहाँ रहती है

साझा SQLite स्थिति डेटाबेस में
`~/.openclaw/state/openclaw.sqlite` पर संग्रहीत:

- `channel_pairing_requests` में लंबित अनुरोध
- `channel_pairing_allow_entries` में अनुमोदित प्रेषक

खाता-स्कोपिंग व्यवहार:

- प्रत्येक अनुरोध और अनुमोदित प्रेषक को चैनल और खाते के अनुसार कुंजीबद्ध किया जाता है
- रनटाइम केवल प्रामाणिक SQLite पंक्तियाँ पढ़ता है; यह लेगेसी फ़ाइलों को मर्ज नहीं करता

पुराने Gateway ने `~/.openclaw/credentials/` के अंतर्गत `<channel>-pairing.json` और
`<channel>-<accountId>-allowFrom.json` लिखे थे।
स्टार्टअप माइग्रेशन और `openclaw doctor --fix` उन फ़ाइलों को SQLite में आयात करते हैं और
सफल आयात के बाद प्रत्येक स्रोत को हटा देते हैं। SQLite डेटाबेस को
संवेदनशील मानें, क्योंकि ये पंक्तियाँ आपके सहायक की पहुँच नियंत्रित करती हैं।

<Note>
पेयरिंग अनुमति-सूची स्टोर DM पहुँच के लिए है। समूह प्राधिकरण अलग है।
DM पेयरिंग कोड अनुमोदित करने से उस प्रेषक को समूह
कमांड चलाने या समूहों में बॉट नियंत्रित करने की अनुमति स्वतः नहीं मिलती। प्रथम-स्वामी बूटस्ट्रैप `commands.ownerAllowFrom` में अलग कॉन्फ़िगरेशन
स्थिति है और समूह चैट डिलीवरी अब भी चैनल की
समूह अनुमति-सूचियों का पालन करती है (उदाहरण के लिए `groupAllowFrom`, `groups`, या चैनल के आधार पर प्रति-समूह
या प्रति-विषय ओवरराइड)।
</Note>

## 2) Node डिवाइस पेयरिंग (iOS/Android/macOS/हेडलेस Node)

Node `role: node` वाले **डिवाइस** के रूप में Gateway से कनेक्ट होते हैं। Gateway
एक डिवाइस पेयरिंग अनुरोध बनाता है, जिसे अनुमोदित करना आवश्यक है।

### Control UI से पेयर करें (अनुशंसित)

`operator.admin` पहुँच वाले पहले से कनेक्ट Control UI सत्र का उपयोग करें:

1. Control UI खोलें और **Settings → Devices** पर जाएँ।
2. **Devices** पृष्ठ पर **Pair mobile device** पर क्लिक करें।
3. **Full access (recommended)** बनाए रखें या प्रशासनिक Gateway नियंत्रण हटाने के लिए **Limited access** चुनें।
4. **Create setup code** पर क्लिक करें।
5. अपने फ़ोन पर OpenClaw ऐप खोलें → **Settings** → **Gateway**।
6. QR कोड स्कैन करें या सेटअप कोड पेस्ट करें, फिर कनेक्ट करें।

आधिकारिक OpenClaw iOS और Android ऐप के सेटअप-कोड मेटाडेटा का मिलान होने पर उन्हें
स्वतः अनुमोदित कर दिया जाता है। यदि **Pending approval** में कोई अनुरोध दिखाई देता है (उदाहरण के लिए,
किसी गैर-आधिकारिक क्लाइंट या बेमेल मेटाडेटा के लिए), तो उसे अनुमोदित करने से पहले उसकी भूमिका और
स्कोप की समीक्षा करें।

जब वर्तमान Control UI सत्र के पास
व्यवस्थापक पहुँच नहीं होती, तब बटन अक्षम रहता है। ऐसी स्थिति में Gateway होस्ट से नीचे दिए गए CLI अनुमोदन प्रवाह का
उपयोग करें।

### Telegram के माध्यम से पेयर करें

यदि आप `device-pair` Plugin का उपयोग करते हैं, तो पहली बार डिवाइस पेयरिंग पूरी तरह Telegram से कर सकते हैं:

1. Telegram में अपने बॉट को यह संदेश भेजें: `/pair`
2. बॉट दो संदेशों से उत्तर देता है: एक निर्देश संदेश और एक अलग **सेटअप कोड** संदेश (Telegram में आसानी से कॉपी/पेस्ट करने के लिए)।
3. अपने फ़ोन पर OpenClaw iOS ऐप खोलें → Settings → Gateway।
4. QR कोड (`/pair qr`) स्कैन करें या सेटअप कोड पेस्ट करके कनेक्ट करें।
5. आधिकारिक मोबाइल ऐप स्वतः कनेक्ट हो जाता है। यदि `/pair pending` में कोई
   अनुरोध दिखाई देता है, तो उसे अनुमोदित करने से पहले उसकी भूमिका और स्कोप की समीक्षा करें।

सेटअप कोड एक base64-एन्कोडेड JSON पेलोड है, जिसमें ये शामिल होते हैं:

- `url`: Gateway WebSocket URL (`ws://...` या `wss://...`)
- `urls`: उपलब्ध होने पर, क्रमबद्ध LAN/Tailnet रूट जिन्हें मोबाइल ऐप आज़मा सकता है
- `bootstrapToken`: शुरुआती पेयरिंग हैंडशेक के लिए एकल-उपयोग बूटस्ट्रैप टोकन; Gateway इसे 10 मिनट बाद समाप्त कर देता है

पेयरिंग पूरी होने पर अप्रयुक्त सेटअप कोड अमान्य करने के लिए `/pair cleanup` चलाएँ।

वह बूटस्ट्रैप टोकन अंतर्निहित पेयरिंग बूटस्ट्रैप प्रोफ़ाइल रखता है:

- एक सुरक्षित `wss://` सेटअप (या समान-होस्ट लूपबैक) डिफ़ॉल्ट रूप से `node` और पूर्ण
  नेटिव-मोबाइल `operator` पहुँच देता है
- हस्तांतरित `node` टोकन `scopes: []` ही रहता है
- डिफ़ॉल्ट हस्तांतरित `operator` टोकन में `operator.admin`,
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, और
  `operator.write` शामिल होते हैं
- Control UI **Limited access** और `openclaw qr --limited`, अन्य ऑपरेटर स्कोप बनाए रखते हुए
  `operator.admin` को हटा देते हैं
- प्लेनटेक्स्ट LAN `ws://` सेटअप स्वतः वही सीमित प्रोफ़ाइल उपयोग करता है;
  पूर्ण पहुँच के लिए `wss://` या Tailscale Serve कॉन्फ़िगर करें और नया कोड जनरेट करें
- बाद का टोकन रोटेशन/निरस्तीकरण, डिवाइस के अनुमोदित
  भूमिका अनुबंध और कॉलर सत्र के ऑपरेटर स्कोप—दोनों से सीमित रहता है

सेटअप कोड के मान्य रहने तक उसे पासवर्ड की तरह सुरक्षित रखें।

iOS और Android के **Settings → Gateway** पृष्ठ **Full** या **Limited**
पहुँच दिखाते हैं। किसी सीमित फ़ोन को अपग्रेड करने के लिए पहले सुरक्षित `wss://` या
Tailscale Serve रूट कॉन्फ़िगर करें, फिर नया पूर्ण-पहुँच सेटअप कोड जनरेट करें, उसे उस सेटिंग पृष्ठ में स्कैन या पेस्ट
करें और पुनः कनेक्ट करें।

Tailscale, सार्वजनिक या अन्य रिमोट मोबाइल पेयरिंग के लिए Tailscale Serve/Funnel
या किसी अन्य `wss://` Gateway URL का उपयोग करें। प्लेनटेक्स्ट `ws://` सेटअप कोड केवल
लूपबैक, निजी LAN पतों, `.local` Bonjour होस्ट और Android
एमुलेटर होस्ट के लिए स्वीकार किए जाते हैं। गैर-लूपबैक प्लेनटेक्स्ट रूट को सीमित पहुँच मिलती है। Tailnet
CGNAT पते, `.ts.net` नाम और सार्वजनिक होस्ट, QR/सेटअप-कोड जारी होने से पहले अब भी सुरक्षित रूप से
अस्वीकृत हो जाते हैं।

`gateway.bind=lan` सेटअप URL के लिए OpenClaw, सक्रिय Gateway के लूपबैक पोर्ट को प्रॉक्सी करने वाले
स्थायी Tailscale Serve HTTPS रूट का पता लगाता है और LAN रूट के साथ उनका
प्रचार करता है। सेटअप कमांड यह फ़ॉलबैक केवल
`lan` के लिए जोड़ता है; `custom` और `tailnet` अपने स्पष्ट रूप से प्रचारित रूट बनाए रखते हैं। 
iOS ऐप प्रचारित रूटों की क्रम से जाँच करता है और पहले पहुँच योग्य
एंडपॉइंट को सहेजता है।

### किसी Node डिवाइस को अनुमोदित करें

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

जब स्पष्ट अनुमोदन इसलिए अस्वीकार हो जाता है क्योंकि अनुमोदन करने वाला पेयर्ड-डिवाइस सत्र
केवल-पेयरिंग स्कोप के साथ खोला गया था, तो CLI उसी अनुरोध को
`operator.admin` के साथ पुनः प्रयास करता है। इससे मौजूदा व्यवस्थापक-सक्षम पेयर्ड डिवाइस, पेयरिंग स्टोर को हाथ से संपादित किए बिना नए
Control UI/ब्राउज़र की पेयरिंग पुनर्प्राप्त कर सकता है। Gateway फिर भी पुनः प्रयास किए गए कनेक्शन को सत्यापित करता है; जो टोकन
`operator.admin` से प्रमाणीकरण नहीं कर सकते, वे अवरुद्ध रहते हैं।

यदि वही डिवाइस अलग प्रमाणीकरण विवरण (उदाहरण के लिए अलग
भूमिका/स्कोप/सार्वजनिक कुंजी) के साथ पुनः प्रयास करता है, तो पिछला लंबित अनुरोध अधिक्रमित हो जाता है और नया
`requestId` बनाया जाता है।

<Note>
पहले से पेयर्ड डिवाइस को बिना सूचना के व्यापक पहुँच नहीं मिलती। यदि वह अधिक स्कोप या व्यापक भूमिका माँगते हुए पुनः कनेक्ट होता है, तो OpenClaw मौजूदा अनुमोदन को यथावत रखता है और एक नया लंबित अपग्रेड अनुरोध बनाता है। अनुमोदन करने से पहले वर्तमान में अनुमोदित पहुँच की नई अनुरोधित पहुँच से तुलना करने के लिए `openclaw devices list` का उपयोग करें।
</Note>

### वैकल्पिक विश्वसनीय-CIDR Node स्वतः-अनुमोदन

डिवाइस पेयरिंग डिफ़ॉल्ट रूप से मैन्युअल रहती है। कड़े नियंत्रण वाले Node नेटवर्कों के लिए,
आप स्पष्ट CIDR या सटीक IP के साथ पहली बार Node के स्वतः-अनुमोदन को चुन सकते हैं:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

यह केवल नए `role: node` पेयरिंग अनुरोधों पर लागू होता है, जिनमें कोई अनुरोधित
स्कोप नहीं होता। ऑपरेटर, ब्राउज़र, Control UI और WebChat क्लाइंट को अब भी मैन्युअल
अनुमोदन की आवश्यकता होती है। भूमिका, स्कोप, मेटाडेटा और सार्वजनिक-कुंजी में बदलावों के लिए अब भी मैन्युअल
अनुमोदन आवश्यक है।

### Node पेयरिंग स्थिति का संग्रहण

साझा SQLite स्थिति डेटाबेस में `~/.openclaw/state/openclaw.sqlite` पर संग्रहीत:

- लंबित डिवाइस पेयरिंग अनुरोध (अल्पकालिक; वे 5 मिनट बाद समाप्त हो जाते हैं)
- पेयर्ड डिवाइस + टोकन

पुराने Gateway इस स्थिति को `~/.openclaw/devices/*.json` में रखते थे; उन फ़ाइलों को
Gateway स्टार्टअप पर SQLite में आयात किया जाता है और `.migrated` प्रत्यय के साथ संग्रहीत किया जाता है।

### टिप्पणियाँ

- `node.pair.*` API (CLI: `openclaw nodes pending|approve|reject|remove|rename`) उसी पेयर्ड डिवाइस रिकॉर्ड पर संग्रहीत
  Node क्षमता अनुमोदनों को प्रबंधित करता है। WS Node को
  अब भी डिवाइस पेयरिंग की आवश्यकता होती है; [Node पेयरिंग](/hi/gateway/pairing) देखें।
- पेयरिंग रिकॉर्ड अनुमोदित भूमिकाओं के लिए स्थायी प्रामाणिक स्रोत है। सक्रिय
  डिवाइस टोकन उसी अनुमोदित भूमिका समूह तक सीमित रहते हैं; अनुमोदित भूमिकाओं से
  बाहर कोई अलग-थलग टोकन प्रविष्टि नई पहुँच नहीं बनाती।
