Nostr एक डाउनलोड करने योग्य चैनल plugin (`@openclaw/nostr`) है, जो OpenClaw को Nostr रिले पर NIP-04 एन्क्रिप्ट किए गए प्रत्यक्ष संदेश प्राप्त करने और उनका उत्तर देने देता है। प्रति Gateway एक खाता; केवल DM।

## इंस्टॉल करना

```bash
openclaw plugins install @openclaw/nostr
```

वर्तमान आधिकारिक रिलीज़ टैग का अनुसरण करने के लिए केवल पैकेज स्पेक का उपयोग करें। सटीक संस्करण केवल तभी पिन करें, जब आपको पुनरुत्पाद्य इंस्टॉलेशन चाहिए।

स्थानीय चेकआउट से (डेवलपमेंट वर्कफ़्लो):

```bash
openclaw plugins install --link <path-to-local-nostr-plugin>
```

plugins इंस्टॉल या सक्षम करने के बाद Gateway पुनः आरंभ करें। Plugin इंस्टॉल होने के बाद ऑनबोर्डिंग (`openclaw onboard`) और `openclaw channels add`, साझा चैनल कैटलॉग से Nostr उपलब्ध कराते हैं।

### गैर-इंटरैक्टिव सेटअप

```bash
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY"
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY" --relay-urls "wss://relay.damus.io,wss://relay.primal.net"
```

कुंजी को कॉन्फ़िगरेशन में संग्रहीत करने के बजाय `NOSTR_PRIVATE_KEY` को परिवेश में रखने के लिए `--use-env` का उपयोग करें (केवल डिफ़ॉल्ट खाता)।

## त्वरित सेटअप

1. Nostr कुंजी-युग्म जनरेट करें (यदि आवश्यक हो):

```bash
# nak का उपयोग करके
nak key generate
```

2. कॉन्फ़िगरेशन में जोड़ें:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
    },
  },
}
```

3. कुंजी एक्सपोर्ट करें:

```bash
export NOSTR_PRIVATE_KEY="nsec1..."
```

4. Gateway पुनः आरंभ करें।

## कॉन्फ़िगरेशन संदर्भ

| कुंजी          | प्रकार     | डिफ़ॉल्ट                                     | विवरण                                              |
| ------------ | -------- | ------------------------------------------- | -------------------------------------------------------- |
| `privateKey` | स्ट्रिंग   | आवश्यक                                    | `nsec` या हेक्स प्रारूप में निजी कुंजी; सीक्रेट संदर्भ अनुमत हैं |
| `relays`     | स्ट्रिंग[] | `['wss://relay.damus.io', 'wss://nos.lol']` | रिले URL (WebSocket)                                   |
| `dmPolicy`   | स्ट्रिंग   | `pairing`                                   | DM पहुँच नीति                                         |
| `allowFrom`  | स्ट्रिंग[] | `[]`                                        | अनुमत प्रेषक सार्वजनिक कुंजियाँ                                   |
| `enabled`    | बूलियन  | `true`                                      | चैनल सक्षम/अक्षम करें                                   |
| `name`       | स्ट्रिंग   | -                                           | प्रदर्शन नाम                                             |
| `profile`    | ऑब्जेक्ट   | -                                           | NIP-01 प्रोफ़ाइल मेटाडेटा                                  |

## प्रोफ़ाइल मेटाडेटा

प्रोफ़ाइल डेटा NIP-01 `kind:0` इवेंट के रूप में प्रकाशित होता है। आप इसे Control UI (Channels -> Nostr -> Profile) से प्रबंधित कर सकते हैं या सीधे कॉन्फ़िगरेशन में सेट कर सकते हैं।

उदाहरण:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      profile: {
        name: "openclaw",
        displayName: "OpenClaw",
        about: "निजी सहायक DM बॉट",
        picture: "https://example.com/avatar.png",
        banner: "https://example.com/banner.png",
        website: "https://example.com",
        nip05: "openclaw@example.com",
        lud16: "openclaw@example.com",
      },
    },
  },
}
```

टिप्पणियाँ:

- प्रोफ़ाइल URL में `https://` का उपयोग होना आवश्यक है।
- रिले से इंपोर्ट करने पर फ़ील्ड मर्ज होते हैं और स्थानीय ओवरराइड सुरक्षित रहते हैं।

## पहुँच नियंत्रण

### DM नीतियाँ

- **पेयरिंग** (डिफ़ॉल्ट): अज्ञात प्रेषकों को पेयरिंग कोड मिलता है।
- **अनुमति-सूची**: केवल `allowFrom` में मौजूद सार्वजनिक कुंजियाँ DM कर सकती हैं।
- **खुला**: सार्वजनिक इनबाउंड DM (`allowFrom: ["*"]` आवश्यक है)।
- **अक्षम**: इनबाउंड DM अनदेखा करें।

प्रवर्तन संबंधी टिप्पणियाँ:

- प्रेषक नीति और NIP-04 डिक्रिप्शन से पहले इनबाउंड इवेंट हस्ताक्षर सत्यापित किए जाते हैं, इसलिए जाली इवेंट आरंभ में ही अस्वीकार कर दिए जाते हैं।
- मूल DM का मुख्य भाग डिक्रिप्ट या प्रोसेस किए बिना पेयरिंग उत्तर भेजे जाते हैं।
- इनबाउंड DM पर दर-सीमा लागू होती है (वैश्विक रूप से और प्रति प्रेषक), और अत्यधिक बड़े पेलोड डिक्रिप्ट करने से पहले हटा दिए जाते हैं।

### अनुमति-सूची का उदाहरण

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      dmPolicy: "allowlist",
      allowFrom: ["npub1abc...", "npub1xyz..."],
    },
  },
}
```

## कुंजी प्रारूप

स्वीकृत प्रारूप:

- **निजी कुंजी:** `nsec...` या 64-वर्ण हेक्स
- **सार्वजनिक कुंजियाँ (`allowFrom`):** `npub...` या हेक्स

## रिले

डिफ़ॉल्ट: `relay.damus.io` और `nos.lol`।

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["wss://relay.damus.io", "wss://relay.primal.net", "wss://nostr.wine"],
    },
  },
}
```

सुझाव:

- अतिरेक के लिए 2-3 रिले का उपयोग करें।
- बहुत अधिक रिले से बचें (विलंबता, दोहराव)।
- सशुल्क रिले विश्वसनीयता बेहतर कर सकते हैं।
- परीक्षण के लिए स्थानीय रिले उपयुक्त हैं (`ws://localhost:7777`)।

## प्रोटोकॉल समर्थन

| NIP    | स्थिति    | विवरण                           |
| ------ | --------- | ------------------------------------- |
| NIP-01 | समर्थित | मूल इवेंट प्रारूप + प्रोफ़ाइल मेटाडेटा |
| NIP-04 | समर्थित | एन्क्रिप्ट किए गए DM (`kind:4`)              |
| NIP-17 | नियोजित   | गिफ़्ट-रैप किए गए DM                      |
| NIP-44 | नियोजित   | संस्करणयुक्त एन्क्रिप्शन                  |

# strfry आरंभ करें
docker run -p 7777:7777 ghcr.io/hoytech/strfry

```json5

```

### मैन्युअल परीक्षण

1. Gateway लॉग या `openclaw channels status` से बॉट की सार्वजनिक कुंजी नोट करें (हेक्स; आवश्यकता होने पर अपने क्लाइंट में npub में बदलें)।
2. Nostr क्लाइंट खोलें (Amethyst, Damus आदि)।
3. बॉट की सार्वजनिक कुंजी को DM भेजें।
4. उत्तर सत्यापित करें।

## समस्या निवारण

### संदेश प्राप्त नहीं हो रहे हैं

- सत्यापित करें कि निजी कुंजी मान्य है।
- सुनिश्चित करें कि रिले URL पहुँच योग्य हैं और `wss://` (या स्थानीय के लिए `ws://`) का उपयोग करते हैं।
- पुष्टि करें कि `enabled`, `false` नहीं है।
- रिले कनेक्शन त्रुटियों के लिए Gateway लॉग जाँचें।

### उत्तर नहीं भेजे जा रहे हैं

- जाँचें कि रिले लेखन स्वीकार करता है।
- आउटबाउंड कनेक्टिविटी सत्यापित करें।
- रिले की दर-सीमाओं पर नज़र रखें।

### डुप्लिकेट उत्तर

- एकाधिक रिले का उपयोग करते समय यह अपेक्षित है।
- संदेशों से इवेंट ID के आधार पर डुप्लिकेट हटाए जाते हैं; केवल पहली डिलीवरी उत्तर ट्रिगर करती है।

## सुरक्षा

- निजी कुंजियाँ कभी कमिट न करें।
- कुंजियों के लिए परिवेश चर का उपयोग करें।
- प्रोडक्शन बॉट के लिए `allowlist` पर विचार करें।
- प्रेषक नीति से पहले हस्ताक्षर सत्यापित किए जाते हैं और डिक्रिप्शन से पहले प्रेषक नीति लागू की जाती है, इसलिए जाली इवेंट आरंभ में ही अस्वीकार कर दिए जाते हैं और अज्ञात प्रेषक पूर्ण क्रिप्टोग्राफ़िक कार्य को बाध्य नहीं कर सकते।
