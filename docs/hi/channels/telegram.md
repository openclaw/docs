## डैशबोर्ड Mini App

Telegram के भीतर OpenClaw डैशबोर्ड खोलने के लिए बॉट के साथ DM में `/dashboard` चलाएँ।

आवश्यकताएँ:

- प्रकाशित HTTPS Mini App URL के लिए `gateway.tailscale.mode: "serve"` या `"funnel"`।
- आपकी संख्यात्मक Telegram उपयोगकर्ता ID, चुने गए अकाउंट के प्रभावी `allowFrom` या `commands.ownerAllowFrom` में होनी चाहिए।
- DM का उपयोग करें। समूहों में, `/dashboard`, `open this in a DM with the bot` के साथ उत्तर देता है और कोई बटन नहीं भेजता।
- Docker इंस्टॉलेशन: Serve/Funnel मोड के लिए Gateway को `tailscaled` के पास लूपबैक से बाइंड करना आवश्यक है, जिसे प्रकाशित पोर्ट वाली ब्रिज नेटवर्किंग पूरा नहीं कर सकती। Gateway कंटेनर को `network_mode: host` के साथ चलाएँ और होस्ट `tailscaled` सॉकेट (`/var/run/tailscale`) के साथ `tailscale` CLI को कंटेनर में माउंट करें।

Mini App केवल Tailscale वाला v1 पथ है और Telegram Web iframe का समर्थन नहीं करता।

## एक्सेस नियंत्रण और सक्रियण

### समूह बॉट पहचान

समूहों और फ़ोरम विषयों में, कॉन्फ़िगर किए गए बॉट हैंडल का स्पष्ट उल्लेख (उदाहरण के लिए `@my_bot`) चुने गए OpenClaw एजेंट को संबोधित करता है, भले ही एजेंट के व्यक्तित्व का नाम Telegram उपयोगकर्ता नाम से अलग हो। असंबंधित ट्रैफ़िक पर समूह मौन नीति अब भी लागू होती है, लेकिन बॉट हैंडल स्वयं कभी भी "कोई और" नहीं होता।

<Tabs>
  <Tab title="DM नीति">
    `channels.telegram.dmPolicy` प्रत्यक्ष संदेश की पहुँच नियंत्रित करता है:

    - `pairing` (डिफ़ॉल्ट)
    - `allowlist` (`allowFrom` में कम-से-कम एक प्रेषक ID आवश्यक है)
    - `open` (`allowFrom` में `"*"` शामिल होना आवश्यक है)
    - `disabled`

    `allowFrom: ["*"]` के साथ `dmPolicy: "open"` किसी भी ऐसे Telegram खाते को बॉट को आदेश देने देता है जो बॉट उपयोगकर्ता नाम खोज या अनुमान कर ले। इसका उपयोग केवल जानबूझकर सार्वजनिक रखे गए और अत्यधिक प्रतिबंधित टूल वाले बॉट के लिए करें; एक स्वामी वाले बॉट को संख्यात्मक उपयोगकर्ता ID के साथ `allowlist` का उपयोग करना चाहिए।

    `channels.telegram.allowFrom` संख्यात्मक Telegram उपयोगकर्ता ID स्वीकार करता है। `telegram:` / `tg:` उपसर्ग स्वीकार और सामान्यीकृत किए जाते हैं।
    बहु-खाता कॉन्फ़िगरेशन में, प्रतिबंधात्मक शीर्ष-स्तरीय `channels.telegram.allowFrom` एक सुरक्षा सीमा है: खाता-स्तरीय `allowFrom: ["*"]` उस खाते को तब तक सार्वजनिक नहीं बनाता, जब तक मर्ज की गई प्रभावी अनुमत-सूची में स्पष्ट वाइल्डकार्ड अब भी मौजूद न हो।
    रिक्त `allowFrom` वाला `dmPolicy: "allowlist"` सभी DM अवरुद्ध करता है और कॉन्फ़िगरेशन सत्यापन द्वारा अस्वीकार कर दिया जाता है।
    सेटअप केवल संख्यात्मक उपयोगकर्ता ID माँगता है। यदि आपके कॉन्फ़िगरेशन में पुराने सेटअप की `@username` अनुमत-सूची प्रविष्टियाँ हैं, तो उन्हें संख्यात्मक ID में बदलने के लिए `openclaw doctor --fix` चलाएँ (यथासंभव प्रयास; Telegram बॉट टोकन आवश्यक है)।
    यदि आप पहले पेयरिंग-स्टोर अनुमत-सूची फ़ाइलों पर निर्भर थे, तो `openclaw doctor --fix` अनुमत-सूची प्रवाहों के लिए प्रविष्टियों को `channels.telegram.allowFrom` में पुनर्प्राप्त कर सकता है (उदाहरण के लिए, जब `dmPolicy: "allowlist"` में अभी कोई स्पष्ट ID न हो)।

    एक स्वामी वाले बॉट के लिए, पिछले पेयरिंग अनुमोदनों पर निर्भर रहने के बजाय स्पष्ट संख्यात्मक `allowFrom` ID के साथ `dmPolicy: "allowlist"` को प्राथमिकता दें।

    सामान्य भ्रम: DM पेयरिंग अनुमोदन का अर्थ यह नहीं है कि "यह प्रेषक हर जगह अधिकृत है।" पेयरिंग केवल DM पहुँच देती है। यदि अभी तक कोई कमांड स्वामी मौजूद नहीं है, तो पहली अनुमोदित पेयरिंग `commands.ownerAllowFrom` भी सेट करती है, जिससे केवल-स्वामी कमांड और निष्पादन अनुमोदनों को एक स्पष्ट ऑपरेटर खाता मिलता है। समूह प्रेषक प्राधिकरण अब भी स्पष्ट कॉन्फ़िगरेशन अनुमत-सूचियों से आता है।
    एक ही पहचान से DM और समूह कमांड, दोनों के लिए अधिकृत होने हेतु: अपनी संख्यात्मक Telegram उपयोगकर्ता ID को `channels.telegram.allowFrom` में रखें, और केवल-स्वामी कमांड के लिए सुनिश्चित करें कि `commands.ownerAllowFrom` में `telegram:<your user id>` शामिल हो।

    ### अपनी Telegram उपयोगकर्ता ID ढूँढना

    अधिक सुरक्षित (कोई तृतीय-पक्ष बॉट नहीं): अपने बॉट को DM भेजें, `openclaw logs --follow` चलाएँ, `from.id` पढ़ें।

    आधिकारिक Bot API विधि:

```bash
curl "https://api.telegram.org/bot<bot_token>/getUpdates"
```

    तृतीय-पक्ष (कम निजी): `@userinfobot` या `@getidsbot`।

  </Tab>

  <Tab title="समूह नीति और अनुमत-सूचियाँ">
    दो नियंत्रण एक साथ लागू होते हैं:

    1. **कौन-से समूह अनुमत हैं** (`channels.telegram.groups`)
       - कोई `groups` कॉन्फ़िगरेशन नहीं, `groupPolicy: "open"`: कोई भी समूह समूह-ID जाँचों में सफल होता है
       - कोई `groups` कॉन्फ़िगरेशन नहीं, `groupPolicy: "allowlist"` (डिफ़ॉल्ट): जब तक आप `groups` प्रविष्टियाँ (या `"*"`) नहीं जोड़ते, सभी समूह अवरुद्ध रहते हैं
       - `groups` कॉन्फ़िगर है: अनुमत-सूची के रूप में कार्य करता है (स्पष्ट ID या `"*"`)

    2. **समूहों में कौन-से प्रेषक अनुमत हैं** (`channels.telegram.groupPolicy`)
       - `open` / `allowlist` (डिफ़ॉल्ट) / `disabled`

    `groupAllowFrom` समूह प्रेषकों को फ़िल्टर करता है; यदि यह सेट नहीं है, तो Telegram `allowFrom` पर वापस जाता है (पेयरिंग स्टोर पर नहीं — समूह प्रेषक प्रमाणीकरण कभी भी DM पेयरिंग-स्टोर अनुमोदनों को इनहेरिट नहीं करता, यह `2026.2.25` से एक सुरक्षा सीमा है)।
    `groupAllowFrom` प्रविष्टियाँ संख्यात्मक Telegram उपयोगकर्ता ID होनी चाहिए (`telegram:` / `tg:` उपसर्ग सामान्यीकृत किए जाते हैं); गैर-संख्यात्मक प्रविष्टियाँ अनदेखी की जाती हैं। समूह या सुपरग्रुप चैट ID यहाँ न रखें — ऋणात्मक चैट ID `channels.telegram.groups` के अंतर्गत आते हैं।
    एक स्वामी वाले बॉट के लिए व्यावहारिक प्रतिरूप: अपनी उपयोगकर्ता ID को `channels.telegram.allowFrom` में सेट करें, `groupAllowFrom` को सेट न करें, और लक्षित समूहों को `channels.telegram.groups` के अंतर्गत अनुमति दें।
    यदि कॉन्फ़िगरेशन में `channels.telegram` पूरी तरह अनुपस्थित है, तो रनटाइम डिफ़ॉल्ट रूप से बंद-विफल `groupPolicy="allowlist"` का उपयोग करता है, जब तक कि `channels.defaults.groupPolicy` स्पष्ट रूप से सेट न हो।

    केवल-स्वामी समूह सेटअप:

```json5
{
  channels: {
    telegram: {
      enabled: true,
      dmPolicy: "pairing",
      allowFrom: ["<YOUR_TELEGRAM_USER_ID>"],
      groupPolicy: "allowlist",
      groups: {
        "<GROUP_CHAT_ID>": {
          requireMention: true,
        },
      },
    },
  },
}
```

    समूह से `@<bot_username> ping` के साथ परीक्षण करें। `requireMention: true` रहने तक सामान्य समूह संदेश बॉट को सक्रिय नहीं करते।

    किसी एक विशिष्ट समूह में किसी भी सदस्य को अनुमति दें:

```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          groupPolicy: "open",
          requireMention: false,
        },
      },
    },
  },
}
```

    किसी एक विशिष्ट समूह में केवल विशिष्ट उपयोगकर्ताओं को अनुमति दें:

```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          requireMention: true,
          allowFrom: ["8734062810", "745123456"],
        },
      },
    },
  },
}
```

    <Warning>
      सामान्य गलती: `groupAllowFrom` समूह अनुमत-सूची नहीं है।

      - ऋणात्मक Telegram समूह/सुपरग्रुप चैट ID (`-1001234567890`) `channels.telegram.groups` के अंतर्गत आते हैं।
      - Telegram उपयोगकर्ता ID (`8734062810`) `groupAllowFrom` के अंतर्गत आते हैं, ताकि यह सीमित किया जा सके कि अनुमत समूह के भीतर कौन-से लोग बॉट को सक्रिय कर सकते हैं।
      - अनुमत समूह के किसी भी सदस्य को बॉट से बात करने देने के लिए ही `groupAllowFrom: ["*"]` का उपयोग करें।

    </Warning>

  </Tab>

  <Tab title="उल्लेख व्यवहार">
    समूह उत्तरों के लिए डिफ़ॉल्ट रूप से उल्लेख आवश्यक है। उल्लेख इनमें से आ सकता है:

    - मूल `@botusername` उल्लेख, या
    - `agents.list[].groupChat.mentionPatterns` या `messages.groupChat.mentionPatterns` में उल्लेख प्रतिरूप

    सत्र-स्तरीय टॉगल (केवल स्थिति, स्थायी नहीं): `/activation always`, `/activation mention`। स्थायित्व के लिए कॉन्फ़िगरेशन का उपयोग करें:

```json5
{
  channels: {
    telegram: {
      groups: {
        "*": { requireMention: false },
      },
    },
  },
}
```

    समूह इतिहास संदर्भ हमेशा चालू रहता है और `historyLimit` द्वारा सीमित होता है। समूह इतिहास विंडो अक्षम करने के लिए `channels.telegram.historyLimit: 0` सेट करें। `openclaw doctor --fix` सेवानिवृत्त `includeGroupHistoryContext` कुंजी को हटा देता है।

    समूह चैट ID प्राप्त करना: समूह संदेश को `@userinfobot` / `@getidsbot` पर अग्रेषित करें, `openclaw logs --follow` से `chat.id` पढ़ें, Bot API `getUpdates` का निरीक्षण करें, या (समूह को अनुमति मिलने के बाद) `/whoami@<bot_username>` चलाएँ।

  </Tab>

## रनटाइम व्यवहार

- Telegram Gateway प्रक्रिया के भीतर चलता है।
- रूटिंग नियतात्मक है: Telegram से आने वाले संदेशों के उत्तर Telegram पर ही वापस जाते हैं (मॉडल चैनल नहीं चुनता)।
- इनबाउंड संदेशों को उत्तर मेटाडेटा, मीडिया प्लेसहोल्डर और Gateway द्वारा देखे गए उत्तरों के लिए संग्रहीत उत्तर-श्रृंखला संदर्भ सहित साझा चैनल एनवलप में सामान्यीकृत किया जाता है।
- समूह सत्रों को समूह ID के आधार पर अलग रखा जाता है। फ़ोरम विषयों में `:topic:<threadId>` जोड़ा जाता है।
- DM संदेशों में `message_thread_id` हो सकता है; OpenClaw इसे उत्तरों के लिए सुरक्षित रखता है। DM विषय सत्र केवल तभी विभाजित होते हैं, जब Telegram `getMe` बॉट के लिए `has_topics_enabled: true` रिपोर्ट करता है; अन्यथा DM समतल सत्र पर बने रहते हैं।
- लॉन्ग पोलिंग प्रति-चैट/प्रति-थ्रेड अनुक्रमण के साथ grammY रनर का उपयोग करती है। रनर सिंक समवर्तीता `agents.defaults.maxConcurrent` का उपयोग करती है।
- बहु-अकाउंट स्टार्टअप समवर्ती `getMe` जाँचों को सीमित करता है, ताकि बड़े बॉट समूह हर अकाउंट की जाँच एक साथ आरंभ न करें।
- प्रत्येक Gateway प्रक्रिया लॉन्ग पोलिंग को सुरक्षित करती है, ताकि एक समय में केवल एक सक्रिय पोलर बॉट टोकन का उपयोग कर सके। लगातार बने रहने वाले `getUpdates` 409 विरोध उसी टोकन का उपयोग करने वाले किसी अन्य OpenClaw Gateway, स्क्रिप्ट या बाहरी पोलर की ओर संकेत करते हैं।
- डिफ़ॉल्ट रूप से, `getUpdates` की जीवंतता 120 सेकंड तक पूरी न होने पर पोलिंग वॉचडॉग पुनः आरंभ होता है। `channels.telegram.pollingStallThresholdMs` (30000-600000, प्रति-अकाउंट ओवरराइड समर्थित हैं) केवल तभी बढ़ाएँ, जब आपके परिनियोजन में लंबे समय तक चलने वाले कार्य के दौरान पोलिंग रुकने के गलत पुनरारंभ दिखाई देते हों।
- Telegram Bot API में पढ़े जाने की पुष्टि का समर्थन नहीं है (`sendReadReceipts` लागू नहीं होता)।

<Note>
  `channels.telegram.dm.threadReplies` और `channels.telegram.direct.<chatId>.threadReplies` हटा दिए गए हैं। अपग्रेड करने के बाद, यदि आपके कॉन्फ़िगरेशन में अभी भी वे कुंजियाँ हैं, तो `openclaw doctor --fix` चलाएँ। DM विषय रूटिंग अब Telegram `getMe.has_topics_enabled` (BotFather के थ्रेडेड मोड द्वारा नियंत्रित) का पालन करती है: विषय-सक्षम बॉट तब थ्रेड-स्कोप वाले DM सत्रों का उपयोग करते हैं, जब Telegram `message_thread_id` भेजता है; अन्य DM समतल सत्र पर बने रहते हैं।
</Note>

## सुविधा संदर्भ

<AccordionGroup>
  <Accordion title="लाइव स्ट्रीम पूर्वावलोकन (संदेश संपादन)">
    OpenClaw सीधे चैट, समूहों और विषयों में आंशिक उत्तरों को रीयल टाइम में स्ट्रीम करता है: एक पूर्वावलोकन संदेश भेजें, फिर `editMessageText` बार-बार करें और उसी स्थान पर अंतिम रूप दें।

    - `channels.telegram.streaming` का मान `off | partial | block | progress` है (डिफ़ॉल्ट: `partial`)
    - छोटे आरंभिक उत्तर पूर्वावलोकनों को डिबाउंस किया जाता है, फिर यदि रन अभी भी सक्रिय हो तो सीमित विलंब के बाद उन्हें मूर्त रूप दिया जाता है
    - `progress` टूल की प्रगति के लिए एक संपादन योग्य स्थिति ड्राफ़्ट रखता है, टूल की प्रगति से पहले उत्तर गतिविधि आने पर स्थिर स्थिति लेबल दिखाता है, पूर्ण होने पर उसे साफ़ करता है और अंतिम उत्तर को सामान्य संदेश के रूप में भेजता है
    - `streaming.preview.toolProgress` नियंत्रित करता है कि टूल/प्रगति अपडेट उसी संपादित पूर्वावलोकन संदेश का पुनः उपयोग करें या नहीं (डिफ़ॉल्ट: पूर्वावलोकन स्ट्रीमिंग सक्रिय होने पर `true`)
    - `streaming.preview.commandText` उन पंक्तियों के भीतर कमांड/निष्पादन विवरण को नियंत्रित करता है: `raw` (डिफ़ॉल्ट) या `status` (केवल टूल लेबल)
    - `streaming.progress.commentary` (डिफ़ॉल्ट: `false`) अस्थायी प्रगति ड्राफ़्ट में सहायक की टिप्पणी/प्रस्तावना टेक्स्ट को वैकल्पिक रूप से सक्षम करता है
    - पुराने `channels.telegram.streamMode`, बूलियन `streaming` मानों और सेवानिवृत्त मूल ड्राफ़्ट पूर्वावलोकन कुंजियों का पता लगाया जाता है; उन्हें माइग्रेट करने के लिए `openclaw doctor --fix` चलाएँ

    टूल-प्रगति पंक्तियाँ वे छोटे स्थिति अपडेट हैं, जो टूल चलने के दौरान दिखाए जाते हैं (कमांड निष्पादन, फ़ाइल पढ़ना, योजना अपडेट, पैच सारांश, ऐप-सर्वर मोड में Codex प्रस्तावना/टिप्पणी)। Telegram इन्हें डिफ़ॉल्ट रूप से चालू रखता है (`v2026.4.22`+ से जारी व्यवहार के अनुरूप)।

    उत्तर-पूर्वावलोकन संपादन बनाए रखें, लेकिन टूल-प्रगति पंक्तियाँ छिपाएँ:

    ```json
    {
      "channels": {
        "telegram": {
          "streaming": {
            "mode": "partial",
            "preview": { "toolProgress": false }
          }
        }
      }
    }
    ```

    टूल-प्रगति दृश्यमान रखें, लेकिन कमांड/निष्पादन टेक्स्ट छिपाएँ:

    ```json
    {
      "channels": {
        "telegram": {
          "streaming": {
            "mode": "partial",
            "preview": { "commandText": "status" }
          }
        }
      }
    }
    ```

    `progress` मोड अंतिम उत्तर को उस संदेश में संपादित किए बिना टूल की प्रगति दिखाता है। कमांड-टेक्स्ट नीति को `streaming.progress` के अंतर्गत रखें:

    ```json
    {
      "channels": {
        "telegram": {
          "streaming": {
            "mode": "progress",
            "progress": {
              "toolProgress": true,
              "commandText": "status"
            }
          }
        }
      }
    }
    ```

    `streaming.mode: "off"` पूर्वावलोकन संपादन अक्षम करता है और सामान्य टूल/प्रगति वार्तालाप को अलग स्थिति संदेशों के रूप में भेजने के बजाय दबा देता है; अनुमोदन संकेत, मीडिया और त्रुटियाँ फिर भी सामान्य अंतिम वितरण के माध्यम से भेजी जाती हैं। `streaming.preview.toolProgress: false` केवल उत्तर-पूर्वावलोकन संपादन बनाए रखता है।

    <Note>
      चयनित उद्धरण वाले उत्तर इसका अपवाद हैं। जब `replyToMode` का मान `first`, `all` या `batched` हो और इनबाउंड संदेश में चयनित उद्धरण टेक्स्ट हो, तब OpenClaw उत्तर पूर्वावलोकन संपादित करने के बजाय Telegram के मूल उद्धरण-उत्तर पथ से अंतिम उत्तर भेजता है, इसलिए उस क्रम में `streaming.preview.toolProgress` स्थिति पंक्तियाँ नहीं दिखा सकता। चयनित उद्धरण टेक्स्ट के बिना वर्तमान-संदेश उत्तर अब भी स्ट्रीम होते हैं। जब टूल-प्रगति की दृश्यता मूल उद्धरण उत्तरों से अधिक महत्वपूर्ण हो, तो `replyToMode: "off"` सेट करें, या इस समझौते को स्वीकार करने के लिए `streaming.preview.toolProgress: false` सेट करें।
    </Note>

    केवल-टेक्स्ट उत्तरों के लिए: छोटे पूर्वावलोकनों का अंतिम संपादन उसी स्थान पर होता है; कई संदेशों में विभाजित होने वाले लंबे अंतिम उत्तर पूर्वावलोकन को पहले खंड के रूप में पुनः उपयोग करते हैं और फिर केवल शेष भाग भेजते हैं; प्रगति-मोड के अंतिम उत्तर स्थिति ड्राफ़्ट साफ़ करते हैं और सामान्य अंतिम वितरण का उपयोग करते हैं; यदि पूर्णता की पुष्टि से पहले अंतिम संपादन विफल हो जाए, तो OpenClaw सामान्य अंतिम वितरण का सहारा लेता है और पुराने पूर्वावलोकन को साफ़ करता है। जटिल उत्तरों (मीडिया पेलोड) के लिए, OpenClaw हमेशा सामान्य अंतिम वितरण का सहारा लेता है और पूर्वावलोकन साफ़ करता है।

    पूर्वावलोकन स्ट्रीमिंग और ब्लॉक स्ट्रीमिंग परस्पर अनन्य हैं — जब ब्लॉक स्ट्रीमिंग स्पष्ट रूप से सक्षम होती है, तो OpenClaw दोहरी स्ट्रीमिंग से बचने के लिए पूर्वावलोकन स्ट्रीम छोड़ देता है।

```json5
{
  channels: {

      richMessages: true,
    },
  },
}
```

    इससे मॉडल टेक्स्ट Telegram के समृद्ध-Markdown चिह्नों से दूर रहता है, इसलिए `$400-600K` जैसी मुद्रा को गणित के रूप में पार्स नहीं किया जाता। लंबा समृद्ध टेक्स्ट Telegram की सीमाओं के अनुसार स्वचालित रूप से विभाजित हो जाता है। 20-कॉलम सीमा से बड़ी तालिकाएँ कोड ब्लॉक का सहारा लेती हैं।

    डिफ़ॉल्ट: क्लाइंट संगतता के लिए बंद — कुछ वर्तमान Desktop, Web, Android और तृतीय-पक्ष क्लाइंट स्वीकृत समृद्ध संदेशों को असमर्थित रूप में रेंडर करते हैं। इसे तब तक बंद रखें, जब तक बॉट के साथ उपयोग होने वाला हर क्लाइंट उन्हें रेंडर न कर सके। `/status` दिखाता है कि वर्तमान सत्र में समृद्ध संदेश चालू हैं या बंद।

    लिंक पूर्वावलोकन डिफ़ॉल्ट रूप से चालू हैं। `channels.telegram.linkPreview: false` समृद्ध टेक्स्ट के लिए स्वचालित एंटिटी पहचान अक्षम करता है।

  </Accordion>

  <Accordion title="मूल कमांड और कस्टम कमांड">
    Telegram का कमांड मेनू स्टार्टअप पर `setMyCommands` के साथ पंजीकृत होता है। `commands.native: "auto"` Telegram के लिए मूल कमांड सक्षम करता है।

    कस्टम कमांड मेनू प्रविष्टियाँ जोड़ें:

```json5
{
  channels: {
    telegram: {
      customCommands: [
        { command: "backup", description: "Git बैकअप" },
        { command: "generate", description: "एक छवि बनाएँ" },
      ],
    },
  },
}
```

    नियम: नामों को सामान्यीकृत किया जाता है (आरंभिक `/` हटाना, लोअरकेस); मान्य पैटर्न `a-z`, `0-9`, `_`, लंबाई 1-32; कस्टम कमांड मूल कमांड को ओवरराइड नहीं कर सकते; विरोध/डुप्लिकेट छोड़ दिए जाते हैं और लॉग किए जाते हैं।

    कस्टम कमांड केवल मेनू प्रविष्टियाँ हैं — वे व्यवहार को स्वचालित रूप से लागू नहीं करते। Plugin/Skill कमांड टाइप किए जाने पर अब भी काम कर सकते हैं, भले ही वे Telegram मेनू में न दिखें। यदि मूल कमांड अक्षम हैं, तो अंतर्निहित कमांड हटा दिए जाते हैं; कॉन्फ़िगर किए जाने पर कस्टम/Plugin कमांड फिर भी पंजीकृत हो सकते हैं।

    सामान्य सेटअप विफलताएँ:

    - ट्रिम के पुनः प्रयास के बाद `BOT_COMMANDS_TOO_MUCH` के साथ `setMyCommands failed` का अर्थ है कि मेनू अब भी सीमा से बड़ा है; Plugin/Skill/कस्टम कमांड कम करें या `channels.telegram.commands.native` अक्षम करें।
    - प्रत्यक्ष Bot API curl कमांड के काम करने के बावजूद `deleteWebhook`, `deleteMyCommands` या `setMyCommands` का `404: Not Found` के साथ विफल होना आमतौर पर दर्शाता है कि `channels.telegram.apiRoot` को पूर्ण `/bot<TOKEN>` एंडपॉइंट पर सेट किया गया था। `apiRoot` केवल Bot API रूट होना चाहिए; `openclaw doctor --fix` गलती से लगे अंतिम `/bot<TOKEN>` को हटा देता है।
    - `getMe returned 401` का अर्थ है कि Telegram ने कॉन्फ़िगर किए गए बॉट टोकन को अस्वीकार कर दिया। `botToken`, `tokenFile` या `TELEGRAM_BOT_TOKEN` (डिफ़ॉल्ट अकाउंट) को वर्तमान BotFather टोकन से अपडेट करें; OpenClaw पोलिंग से पहले रुक जाता है, इसलिए इसे Webhook सफ़ाई विफलता के रूप में रिपोर्ट नहीं किया जाता।
    - नेटवर्क/फ़ेच त्रुटियों के साथ `setMyCommands failed` का आमतौर पर अर्थ है कि `api.telegram.org` तक आउटबाउंड DNS/HTTPS अवरुद्ध है।

    ### डिवाइस पेयरिंग कमांड (`device-pair` Plugin)

    इंस्टॉल होने पर:

    1. `/pair` एक सेटअप कोड बनाता है
    2. कोड को iOS ऐप में पेस्ट करें
    3. `/pair pending` लंबित अनुरोधों की सूची दिखाता है (भूमिका/स्कोप सहित)
    4. स्वीकृत करें: `/pair approve <requestId>`, `/pair approve` (केवल लंबित अनुरोध) या `/pair approve latest`

    यदि कोई डिवाइस बदले हुए प्रमाणीकरण विवरण (भूमिका, स्कोप, सार्वजनिक कुंजी) के साथ पुनः प्रयास करता है, तो पिछले लंबित अनुरोध को नए `requestId` द्वारा प्रतिस्थापित कर दिया जाता है; स्वीकृति देने से पहले `/pair pending` फिर से चलाएँ।

    अधिक विवरण: [पेयरिंग](/hi/channels/pairing#pair-via-telegram)।

  <Accordion title="फ़ोरम विषय और थ्रेड व्यवहार">
    फ़ोरम सुपरग्रुप: विषय सेशन कुंजियाँ `:topic:<threadId>` जोड़ती हैं; उत्तर और टाइपिंग विषय थ्रेड को लक्षित करते हैं; विषय कॉन्फ़िगरेशन पथ `channels.telegram.groups.<chatId>.topics.<threadId>` है।

    सामान्य विषय (`threadId=1`) एक विशेष मामला है: संदेश प्रेषण `message_thread_id` को छोड़ देते हैं (Telegram `sendMessage(...thread_id=1)` को "थ्रेड नहीं मिला" के साथ अस्वीकार करता है), लेकिन टाइपिंग कार्रवाइयों में फिर भी `message_thread_id` शामिल होता है (टाइपिंग संकेतक दिखने के लिए अनुभवजन्य रूप से आवश्यक)।

    विषय प्रविष्टियाँ ओवरराइड न होने पर समूह सेटिंग्स इनहेरिट करती हैं (`requireMention`, `allowFrom`, `skills`, `systemPrompt`, `enabled`, `groupPolicy`)। `agentId` केवल विषय के लिए है और समूह डिफ़ॉल्ट से इनहेरिट नहीं होता। `topics."*"` उस समूह के प्रत्येक विषय के लिए डिफ़ॉल्ट सेट करता है; सटीक विषय ID फिर भी `"*"` पर प्राथमिकता रखते हैं।

    **प्रति-विषय एजेंट रूटिंग**: प्रत्येक विषय, विषय कॉन्फ़िगरेशन में `agentId` के माध्यम से किसी अलग एजेंट को रूट कर सकता है, जिससे उसे अपना कार्यक्षेत्र, मेमोरी और सेशन मिलता है:

    ```json5
    {
      channels: {
        telegram: {
          groups: {
            "-1001234567890": {
              topics: {
                "1": { agentId: "main" },      // सामान्य विषय -> मुख्य एजेंट
                "3": { agentId: "zu" },        // डेवलपमेंट विषय -> zu एजेंट
                "5": { agentId: "coder" }      // कोड समीक्षा -> coder एजेंट
              }
            }
          }
        }
      }
    }
    ```

    इसके बाद प्रत्येक विषय की अपनी सेशन कुंजी होती है, उदाहरण के लिए `agent:zu:telegram:group:-1001234567890:topic:3`.

    **स्थायी ACP विषय बाइंडिंग**: फ़ोरम विषय शीर्ष-स्तरीय टाइप्ड बाइंडिंग के माध्यम से ACP हार्नेस सेशन पिन कर सकते हैं (`bindings[]`, जिसमें `type: "acp"`, `match.channel: "telegram"`, `peer.kind: "group"`, और `-1001234567890:topic:42` जैसा विषय-योग्य ID हो)। फ़िलहाल यह समूहों/सुपरग्रुप के फ़ोरम विषयों तक सीमित है। [ACP एजेंट](/hi/tools/acp-agents) देखें।

    **चैट से थ्रेड-बाउंड ACP स्पॉन**: `/acp spawn <agent> --thread here|auto` वर्तमान विषय को नए ACP सेशन से बाँधता है; आगे के संदेश सीधे वहाँ रूट होते हैं, और OpenClaw स्पॉन पुष्टि को विषय में पिन करता है। इसके लिए `channels.telegram.threadBindings.spawnSessions` आवश्यक है (डिफ़ॉल्ट: `true`)।

    टेम्पलेट संदर्भ `MessageThreadId` और `IsForum` उपलब्ध कराता है। `message_thread_id` वाली DM चैट रिप्लाई मेटाडेटा बनाए रखती हैं, लेकिन थ्रेड-जागरूक सेशन कुंजियों का उपयोग केवल तब करती हैं जब Telegram `getMe`, `has_topics_enabled: true` रिपोर्ट करता है।
    सेवानिवृत्त `dm.threadReplies` और `direct.*.threadReplies` ओवरराइड हटा दिए गए हैं; BotFather का थ्रेडेड मोड ही सत्य का एकमात्र स्रोत है। पुराने कॉन्फ़िगरेशन कुंजियाँ हटाने के लिए `openclaw doctor --fix` चलाएँ।

  </Accordion>

    ### ऑडियो संदेश

    Telegram वॉइस नोट और ऑडियो फ़ाइलों में अंतर करता है। डिफ़ॉल्ट: ऑडियो-फ़ाइल व्यवहार; वॉइस-नोट प्रेषण बाध्य करने के लिए एजेंट के उत्तर में `[[audio_as_voice]]` टैग करें। इनबाउंड वॉइस-नोट ट्रांसक्रिप्ट को एजेंट संदर्भ में मशीन-जनित, अविश्वसनीय टेक्स्ट के रूप में फ़्रेम किया जाता है, लेकिन उल्लेख पहचान अब भी कच्चे ट्रांसक्रिप्ट का उपयोग करती है, ताकि उल्लेख-गेटेड वॉइस संदेश काम करते रहें।

```json5
{
  action: "send",
  channel: "telegram",
  to: "123456789",
  media: "https://example.com/voice.ogg",
  asVoice: true,
}
```

    ### वीडियो संदेश

    Telegram वीडियो फ़ाइलों और वीडियो नोट में अंतर करता है। वीडियो नोट कैप्शन का समर्थन नहीं करते; दिया गया संदेश टेक्स्ट अलग से भेजा जाता है।

```json5
{
  action: "send",
  channel: "telegram",
  to: "123456789",
  media: "https://example.com/video.mp4",
  asVideoNote: true,
}
```

    ### स्थान और स्थल

    एक स्वतंत्र `location` ऑब्जेक्ट के साथ मौजूदा `send` कार्रवाई का उपयोग करें। निर्देशांक एक नेटिव पिन भेजते हैं; `name` और `address` दोनों जोड़ने पर एक नेटिव स्थल कार्ड भेजा जाता है। स्थान प्रेषण को संदेश टेक्स्ट या मीडिया के साथ संयोजित नहीं किया जा सकता।

```json5
{
  action: "send",
  channel: "telegram",
  to: "123456789",
  location: {
    latitude: 48.858844,
    longitude: 2.294351,
    accuracy: 12,
    name: "एफ़िल टॉवर",
    address: "शाँ द मार्स, पेरिस",
  },
}
```

    ### स्टिकर

  <Accordion title="रिएक्शन सूचनाएँ">
    Telegram रिएक्शन संदेश पेलोड से अलग `message_reaction` अपडेट के रूप में आते हैं। सक्षम होने पर, OpenClaw `Telegram reaction added: 👍 by Alice (@alice) on msg 42` जैसे सिस्टम इवेंट कतार में डालता है।

    - `channels.telegram.reactionNotifications`: `off | own | all` (डिफ़ॉल्ट: `own`)
    - `channels.telegram.reactionLevel`: `off | ack | minimal | extensive` (डिफ़ॉल्ट: `minimal`)

    `own` का अर्थ केवल बॉट द्वारा भेजे गए संदेशों पर उपयोगकर्ता रिएक्शन है (भेजे गए संदेशों के कैश के माध्यम से यथासंभव)। रिएक्शन इवेंट फिर भी Telegram एक्सेस नियंत्रणों (`dmPolicy`, `allowFrom`, `groupPolicy`, `groupAllowFrom`) का पालन करते हैं; अनधिकृत प्रेषकों को छोड़ दिया जाता है।

    Telegram रिएक्शन अपडेट में थ्रेड ID उपलब्ध नहीं कराता: गैर-फ़ोरम समूह, समूह चैट सेशन को रूट होते हैं; फ़ोरम समूह सटीक मूल विषय के बजाय सामान्य-विषय सेशन (`:topic:1`) को रूट होते हैं।

    पोलिंग/Webhook के लिए `allowed_updates` स्वचालित रूप से `message_reaction` शामिल करता है।

    जब OpenClaw किसी इनबाउंड संदेश को प्रोसेस करता है, तब `ackReaction` एक स्वीकृति इमोजी भेजता है। `messages.ackReactionScope` तय करता है कि इसे *कब* भेजा जाए।

    - `channels.telegram.accounts.<accountId>.ackReaction`
    - `channels.telegram.ackReaction`
    - `messages.ackReaction`
    - एजेंट पहचान इमोजी फ़ॉलबैक (`agents.list[].identity.emoji`, अन्यथा "👀")

    Telegram एक यूनिकोड इमोजी की अपेक्षा करता है (उदाहरण के लिए "👀"); किसी चैनल या खाते के लिए रिएक्शन अक्षम करने हेतु `""` का उपयोग करें।

    </Note>

  <Accordion title="लॉन्ग पोलिंग बनाम Webhook">
    डिफ़ॉल्ट लॉन्ग पोलिंग है। Webhook मोड के लिए `channels.telegram.webhookUrl` और `channels.telegram.webhookSecret` सेट करें; वैकल्पिक `webhookPath` (डिफ़ॉल्ट `/telegram-webhook`), `webhookHost` (डिफ़ॉल्ट `127.0.0.1`), `webhookPort` (डिफ़ॉल्ट `8787`), `webhookCertPath` (डायरेक्ट-IP या बिना-डोमेन सेटअप के लिए स्वयं-हस्ताक्षरित प्रमाणपत्र PEM)।

    लॉन्ग-पोलिंग मोड में, OpenClaw अपना पुनः प्रारंभ वॉटरमार्क केवल अपडेट के सफलतापूर्वक डिस्पैच होने के बाद स्थायी करता है; विफल हैंडलर उस अपडेट को पूर्ण चिह्नित करने के बजाय उसी प्रक्रिया में पुनः प्रयास योग्य छोड़ देता है।

    स्थानीय लिसनर डिफ़ॉल्ट रूप से `127.0.0.1:8787` से बाइंड होता है। सार्वजनिक इनग्रेस के लिए स्थानीय पोर्ट के आगे रिवर्स प्रॉक्सी रखें, या जानबूझकर `webhookHost: "0.0.0.0"` सेट करें।

    Webhook मोड अनुरोध गार्ड, Telegram सीक्रेट टोकन और JSON बॉडी को सत्यापित करता है, फिर खाली `200` लौटाने से पहले अपडेट को अपनी टिकाऊ इनग्रेस कतार में कमिट करता है। सफल टिकाऊ अंगीकरण में `x-openclaw-delivery-accepted: durable` शामिल होता है; स्वास्थ्य, रूटिंग, प्रमाणीकरण, सत्यापन और स्टोरेज-त्रुटि प्रतिक्रियाएँ इस हेडर को छोड़ देती हैं। रिवर्स प्रॉक्सी और होस्ट नियंत्रक इस हेडर की आवश्यकता रख सकते हैं, ताकि प्रतिक्रिया समय से स्वीकृति का अनुमान लगाए बिना OpenClaw अंगीकरण को सामान्य खाली `200` से अलग किया जा सके।

    इसके बाद OpenClaw अपडेट को लॉन्ग पोलिंग द्वारा उपयोग की जाने वाली उन्हीं प्रति-चैट/प्रति-विषय बॉट लेन के माध्यम से एसिंक्रोनस रूप से प्रोसेस करता है, इसलिए धीमे एजेंट टर्न Telegram के डिलीवरी ACK को रोके नहीं रखते।

  </Accordion>

  <Accordion title="सीमाएँ, पुनः प्रयास, और CLI लक्ष्य">
    - `channels.telegram.textChunkLimit` डिफ़ॉल्ट रूप से 4000 है; `streaming.chunkMode="newline"` लंबाई के आधार पर विभाजन से पहले अनुच्छेद सीमाओं (रिक्त पंक्तियों) को प्राथमिकता देता है।
    - `channels.telegram.mediaMaxMb` (डिफ़ॉल्ट 100) इनबाउंड और आउटबाउंड मीडिया आकार को सीमित करता है।
    - `channels.telegram.mediaGroupFlushMs` (डिफ़ॉल्ट 500, सीमा 10-60000) नियंत्रित करता है कि OpenClaw द्वारा एल्बम/मीडिया समूहों को एक इनबाउंड संदेश के रूप में भेजने से पहले उन्हें कितनी देर बफ़र किया जाता है। यदि एल्बम के भाग देर से आते हैं, तो इसे बढ़ाएँ; एल्बम के उत्तर की विलंबता कम करने के लिए इसे घटाएँ।
    - `channels.telegram.timeoutSeconds` API क्लाइंट टाइमआउट को ओवरराइड करता है (सेट न होने पर grammY का डिफ़ॉल्ट लागू होता है)। बॉट क्लाइंट कॉन्फ़िगर किए गए 60-सेकंड के आउटबाउंड टेक्स्ट/टाइपिंग अनुरोध गार्ड से कम मानों को उसी सीमा तक बढ़ा देते हैं, ताकि OpenClaw का ट्रांसपोर्ट गार्ड और फ़ॉलबैक चलने से पहले grammY दृश्यमान उत्तर डिलीवरी को निरस्त न करे। लॉन्ग पोलिंग अब भी 45-सेकंड के `getUpdates` अनुरोध गार्ड का उपयोग करता है, ताकि निष्क्रिय पोल अनिश्चित काल तक छोड़े न रहें।
    - `channels.telegram.pollingStallThresholdMs` डिफ़ॉल्ट रूप से 120000 है; इसे 30000 और 600000 के बीच केवल झूठे-सकारात्मक पोलिंग-स्टॉल पुनरारंभों के लिए समायोजित करें।
    - समूह संदर्भ इतिहास `channels.telegram.historyLimit` या `messages.groupChat.historyLimit` (डिफ़ॉल्ट 50) का उपयोग करता है; `0` इसे अक्षम करता है।
    - जब Gateway ने मूल संदेशों को देखा हो, तो उत्तर/उद्धरण/फ़ॉरवर्ड का पूरक संदर्भ एक चयनित वार्तालाप संदर्भ विंडो में सामान्यीकृत होता है; देखे गए संदेशों की कैश OpenClaw SQLite Plugin स्थिति में रहती है, और `openclaw doctor --fix` पुरानी साइडकार फ़ाइलें आयात करता है। Telegram प्रत्येक अपडेट में केवल एक सतही `reply_to_message` शामिल करता है, इसलिए कैश से पुरानी शृंखलाएँ उसी पेलोड तक सीमित रहती हैं।
    - Telegram की अनुमतिसूचियाँ मुख्य रूप से यह नियंत्रित करती हैं कि एजेंट को कौन ट्रिगर कर सकता है; वे पूरक संदर्भ को पूरी तरह संपादित करने की सीमा नहीं हैं।
    - DM इतिहास: `channels.telegram.dmHistoryLimit`, `channels.telegram.dms["<user_id>"].historyLimit`।
    - `channels.telegram.retry` पुनर्प्राप्ति योग्य आउटबाउंड API त्रुटियों के लिए Telegram प्रेषण सहायकों (CLI/टूल/क्रियाओं) पर लागू होता है। इनबाउंड अंतिम-उत्तर डिलीवरी कनेक्शन से पहले की विफलताओं के लिए सीमित सुरक्षित-प्रेषण पुनः प्रयास का उपयोग करती है, लेकिन प्रेषण के बाद के अस्पष्ट नेटवर्क एनवेलप पर पुनः प्रयास नहीं करती, क्योंकि इससे दृश्यमान संदेश दोहराए जा सकते हैं।

    CLI और संदेश-टूल प्रेषण लक्ष्य संख्यात्मक चैट ID, उपयोगकर्ता नाम या फ़ोरम विषय लक्ष्य स्वीकार करते हैं:

```bash
openclaw message send --channel telegram --target 123456789 --message "नमस्ते"
openclaw message send --channel telegram --target @name --message "नमस्ते"
openclaw message send --channel telegram --target -1001234567890:topic:42 --message "नमस्ते विषय"
```

    पोल `openclaw message poll` का उपयोग करते हैं और फ़ोरम विषयों का समर्थन करते हैं:

```bash
openclaw message poll --channel telegram --target 123456789 \
  --poll-question "इसे जारी करें?" --poll-option "हाँ" --poll-option "नहीं"
openclaw message poll --channel telegram --target -1001234567890:topic:42 \
  --poll-question "समय चुनें" --poll-option "सुबह 10 बजे" --poll-option "दोपहर 2 बजे" \
  --poll-duration-seconds 300 --poll-public
```

    केवल Telegram के पोल फ़्लैग: `--poll-duration-seconds` (5-600), `--poll-anonymous`, `--poll-public`, `--thread-id` (या एक `:topic:` लक्ष्य)। `--poll-option` 2-12 बार दोहराता है (Telegram की विकल्प सीमा)।

    Telegram प्रेषण इनलाइन कीबोर्ड के लिए `buttons` ब्लॉक के साथ `--presentation` का भी समर्थन करता है (जब `channels.telegram.capabilities.inlineButtons` इसकी अनुमति देता है), बॉट के उस चैट में पिन कर सकने पर पिन की गई डिलीवरी का अनुरोध करने के लिए `--pin` या `--delivery '{"pin":true}'`, और आउटबाउंड चित्रों, GIF और वीडियो को संपीड़ित/एनिमेटेड/वीडियो अपलोड के बजाय दस्तावेज़ों के रूप में भेजने के लिए `--force-document`।

    क्रिया नियंत्रण: `channels.telegram.actions.sendMessage=false` पोल सहित सभी आउटबाउंड संदेशों को अक्षम करता है; `channels.telegram.actions.poll=false` नियमित प्रेषण सक्षम रखते हुए पोल निर्माण को अक्षम करता है।

  </Accordion>

  <Accordion title="Telegram में निष्पादन अनुमोदन">
    Telegram अनुमोदक DM में निष्पादन अनुमोदन का समर्थन करता है और वैकल्पिक रूप से मूल चैट या विषय में संकेत पोस्ट कर सकता है। अनुमोदक संख्यात्मक Telegram उपयोगकर्ता ID होने चाहिए।

    - `channels.telegram.execApprovals.enabled` (`"auto"` कम-से-कम एक अनुमोदक का समाधान संभव होने पर सक्षम करता है)
    - `channels.telegram.execApprovals.approvers` (`commands.ownerAllowFrom` से संख्यात्मक स्वामी ID पर वापस जाता है)
    - `channels.telegram.execApprovals.target`: `dm` (डिफ़ॉल्ट) | `channel` | `both`
    - `agentFilter`, `sessionFilter`

    `channels.telegram.allowFrom`, `groupAllowFrom`, और `defaultTo` नियंत्रित करते हैं कि बॉट से कौन बात कर सकता है और वह सामान्य उत्तर कहाँ भेजता है — वे किसी को निष्पादन अनुमोदक नहीं बनाते। जब अभी कोई कमांड स्वामी मौजूद न हो, तो पहली अनुमोदित DM पेयरिंग `commands.ownerAllowFrom` को आरंभ करती है, ताकि एक-स्वामी सेटअप में `execApprovals.approvers` के अंतर्गत ID दोहराने की आवश्यकता न हो।

    चैनल डिलीवरी चैट में कमांड टेक्स्ट दिखाती है; `channel` या `both` को केवल विश्वसनीय समूहों/विषयों में सक्षम करें। जब संकेत किसी फ़ोरम विषय में पहुँचता है, तो OpenClaw अनुमोदन संकेत और अनुवर्ती कार्रवाई के लिए विषय बनाए रखता है। निष्पादन अनुमोदन डिफ़ॉल्ट रूप से 30 मिनट बाद समाप्त हो जाते हैं।

    इनलाइन अनुमोदन बटनों के लिए यह भी आवश्यक है कि `channels.telegram.capabilities.inlineButtons` लक्ष्य सतह (`dm`, `group`, या `all`) को अनुमति दे। `plugin:` से उपसर्गित अनुमोदन ID Plugin अनुमोदनों के माध्यम से हल होती हैं; अन्य पहले निष्पादन अनुमोदनों के माध्यम से हल होती हैं।

    [निष्पादन अनुमोदन](/hi/tools/exec-approvals) देखें।

  </Accordion>
</AccordionGroup>

## त्रुटि उत्तर नियंत्रण

जब एजेंट को डिलीवरी या प्रदाता त्रुटि मिलती है, तो त्रुटि नीति नियंत्रित करती है कि त्रुटि संदेश Telegram चैट तक पहुँचते हैं या नहीं:

| कुंजी                                 | मान                     | डिफ़ॉल्ट         | विवरण                                                                                                                                                                                              |
| ----------------------------------- | -------------------------- | --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `channels.telegram.errorPolicy`     | `always`, `once`, `silent` | `always`        | `always` प्रत्येक त्रुटि संदेश चैट में भेजता है। `once` प्रत्येक विशिष्ट त्रुटि संदेश को प्रति कूलडाउन विंडो एक बार भेजता है (बार-बार आने वाली समान त्रुटियों को रोकता है)। `silent` चैट में कभी त्रुटि संदेश नहीं भेजता। |
| `channels.telegram.errorCooldownMs` | संख्या (ms)                | `14400000` (4h) | `once` नीति के लिए कूलडाउन विंडो। त्रुटि भेजे जाने के बाद, इस अंतराल के बीतने तक वही संदेश रोक दिया जाता है। सेवा बाधाओं के दौरान त्रुटि स्पैम को रोकता है।                                           |

प्रति-खाता, प्रति-समूह और प्रति-विषय ओवरराइड समर्थित हैं (अन्य Telegram कॉन्फ़िगरेशन कुंजियों जैसी ही इनहेरिटेंस)।

```json5
{
  channels: {
    telegram: {
      errorPolicy: "always",
      errorCooldownMs: 120000,
      groups: {
        "-1001234567890": {
          errorPolicy: "silent", // इस समूह में त्रुटियाँ रोकें
        },
      },
    },
  },
}
```

## समस्या निवारण

<AccordionGroup>
  <Accordion title="बॉट उल्लेख-रहित समूह संदेशों का उत्तर नहीं देता">

    - यदि `requireMention=false`, तो Telegram गोपनीयता मोड को पूर्ण दृश्यता की अनुमति देनी होगी: BotFather `/setprivacy` -> Disable, फिर बॉट को समूह से हटाकर दोबारा जोड़ें।
    - जब कॉन्फ़िगरेशन उल्लेख-रहित समूह संदेशों की अपेक्षा करता है, तो `openclaw channels status` चेतावनी देता है।
    - `openclaw channels status --probe` स्पष्ट संख्यात्मक समूह ID की जाँच करता है; वाइल्डकार्ड `"*"` की सदस्यता की जाँच नहीं की जा सकती।
    - त्वरित सत्र परीक्षण: `/activation always`।

  </Accordion>

  <Accordion title="बॉट को समूह संदेश बिल्कुल दिखाई नहीं देते">

    - जब `channels.telegram.groups` मौजूद हो, तो समूह सूचीबद्ध होना चाहिए (या `"*"` शामिल करें)।
    - समूह में बॉट की सदस्यता सत्यापित करें।
    - छोड़ने के कारणों के लिए `openclaw logs --follow` की समीक्षा करें।

  </Accordion>

  <Accordion title="कमांड आंशिक रूप से काम करते हैं या बिल्कुल नहीं करते">

    - अपनी प्रेषक पहचान को अधिकृत करें (पेयरिंग और/या संख्यात्मक `allowFrom`); समूह नीति `open` होने पर भी कमांड प्राधिकरण लागू होता है।
    - `setMyCommands failed` के साथ `BOT_COMMANDS_TOO_MUCH` का अर्थ है कि नेटिव मेन्यू में बहुत अधिक प्रविष्टियाँ हैं; Plugin/Skill/कस्टम कमांड घटाएँ या नेटिव मेन्यू अक्षम करें।
    - `deleteMyCommands` / `setMyCommands` स्टार्टअप कॉल और `sendChatAction` टाइपिंग कॉल सीमित हैं तथा अनुरोध टाइमआउट पर Telegram के ट्रांसपोर्ट फ़ॉलबैक के माध्यम से एक बार पुनः प्रयास करते हैं। लगातार नेटवर्क/फ़ेच त्रुटियों का सामान्यतः अर्थ है कि `api.telegram.org` तक DNS/HTTPS नहीं पहुँच पा रहा है।

  </Accordion>

  <Accordion title="स्टार्टअप अनधिकृत टोकन की सूचना देता है">

    - `getMe returned 401` कॉन्फ़िगर किए गए बॉट टोकन के लिए Telegram प्रमाणीकरण विफलता है। BotFather में टोकन को फिर से कॉपी करें या पुनः जनरेट करें, फिर `channels.telegram.botToken`, `tokenFile`, `accounts.<id>.botToken`, या `TELEGRAM_BOT_TOKEN` (डिफ़ॉल्ट खाता) अपडेट करें।
    - स्टार्टअप के दौरान `deleteWebhook 401 Unauthorized` भी प्रमाणीकरण विफलता है; इसे "कोई Webhook मौजूद नहीं है" मानने से वही अमान्य-टोकन विफलता केवल बाद की API कॉल तक टलेगी।

  </Accordion>

  <Accordion title="पोलिंग या नेटवर्क अस्थिरता">

    - यदि `AbortSignal` प्रकार मेल नहीं खाते, तो कस्टम फ़ेच/प्रॉक्सी के साथ Node 22+ तत्काल निरसन व्यवहार ट्रिगर कर सकता है।
    - कुछ होस्ट `api.telegram.org` को पहले IPv6 में रिज़ॉल्व करते हैं; खराब IPv6 निर्गमन के कारण रुक-रुककर API विफलताएँ होती हैं।
    - `TypeError: fetch failed` या `Network request for 'getUpdates' failed!` वाले लॉग पर पुनर्प्राप्ति योग्य नेटवर्क त्रुटियों के रूप में पुनः प्रयास किया जाता है।
    - पोलिंग स्टार्टअप के दौरान, OpenClaw grammY के लिए सफल स्टार्टअप `getMe` प्रोब का पुनः उपयोग करता है, ताकि रनर को पहले `getUpdates` से पहले दूसरे `getMe` की आवश्यकता न पड़े।
    - यदि पोलिंग स्टार्टअप के दौरान `deleteWebhook` क्षणिक नेटवर्क त्रुटि के साथ विफल होता है, तो OpenClaw एक और प्री-पोल कंट्रोल-प्लेन कॉल करने के बजाय लॉन्ग पोलिंग में आगे बढ़ता है। तब भी सक्रिय Webhook `getUpdates` टकराव के रूप में सामने आता है; OpenClaw ट्रांसपोर्ट को फिर बनाता है और Webhook सफ़ाई का पुनः प्रयास करता है।
    - यदि Telegram सॉकेट छोटे निश्चित अंतराल पर पुनर्चक्रित होते हैं, तो कम `channels.telegram.timeoutSeconds` की जाँच करें — बॉट क्लाइंट कॉन्फ़िगर किए गए आउटबाउंड और `getUpdates` अनुरोध गार्ड से कम मानों को उन्हीं सीमाओं तक बढ़ाते हैं, लेकिन पुराने रिलीज़ में इसे उन गार्ड से कम सेट करने पर प्रत्येक पोल या उत्तर निरस्त हो सकता था।
    - लॉग में `Polling stall detected` का अर्थ है कि OpenClaw डिफ़ॉल्ट रूप से 120 सेकंड तक पूर्ण लॉन्ग-पोल सक्रियता न मिलने के बाद पोलिंग को पुनरारंभ करता और ट्रांसपोर्ट को फिर बनाता है।
    - `openclaw channels status --probe` और `openclaw doctor` तब चेतावनी देते हैं, जब चल रहे पोलिंग खाते ने स्टार्टअप छूट अवधि के बाद `getUpdates` पूरा नहीं किया हो, चल रहे Webhook खाते ने स्टार्टअप छूट अवधि के बाद `setWebhook` पूरा नहीं किया हो, या पिछली सफल पोलिंग ट्रांसपोर्ट गतिविधि पुरानी हो।
    - `channels.telegram.pollingStallThresholdMs` केवल तब बढ़ाएँ जब लंबे समय तक चलने वाली `getUpdates` कॉल स्वस्थ हों, लेकिन आपका होस्ट फिर भी झूठे पोलिंग-स्टॉल पुनरारंभों की सूचना देता हो। लगातार स्टॉल सामान्यतः `api.telegram.org` तक प्रॉक्सी, DNS, IPv6, या TLS निर्गमन समस्याओं की ओर संकेत करते हैं।
    - Telegram Bot API ट्रांसपोर्ट के लिए प्रक्रिया के प्रॉक्सी परिवेश चर स्वीकार करता है: `HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY`, और छोटे अक्षरों वाले प्रकार। `NO_PROXY` / `no_proxy` फिर भी `api.telegram.org` को बायपास कर सकते हैं।
    - यदि सेवा परिवेश के लिए `OPENCLAW_PROXY_URL` सेट है और कोई मानक प्रॉक्सी परिवेश चर मौजूद नहीं है, तो Telegram Bot API ट्रांसपोर्ट के लिए भी उस URL का उपयोग करता है।
    - अस्थिर प्रत्यक्ष निर्गमन/TLS वाले VPS होस्ट पर, Telegram API कॉल को प्रॉक्सी के माध्यम से रूट करें:

```yaml
channels:
  telegram:
    proxy: socks5://<user>:<password>@proxy-host:1080
```

    - Node 22+ में डिफ़ॉल्ट रूप से `autoSelectFamily=true` होता है (WSL2 को छोड़कर)। Telegram DNS परिणाम क्रम पहले `OPENCLAW_TELEGRAM_DNS_RESULT_ORDER`, फिर `channels.telegram.network.dnsResultOrder`, और फिर प्रक्रिया के डिफ़ॉल्ट (उदाहरण के लिए `NODE_OPTIONS=--dns-result-order=ipv4first`) का पालन करता है; यदि इनमें से कोई लागू न हो, तो Node 22+ पर `ipv4first` पर वापस जाता है।
    - WSL2 पर, या जब केवल-IPv4 व्यवहार बेहतर काम करता हो, तो फ़ैमिली चयन को बाध्य करें:

```yaml
channels:
  telegram:
    network:
      autoSelectFamily: false
```

    - RFC 2544 बेंचमार्क-रेंज उत्तर (`198.18.0.0/15`) Telegram मीडिया डाउनलोड के लिए पहले से ही डिफ़ॉल्ट रूप से अनुमत हैं। यदि कोई विश्वसनीय फ़ेक-IP या पारदर्शी प्रॉक्सी मीडिया डाउनलोड के दौरान `api.telegram.org` को किसी अन्य निजी/आंतरिक/विशेष-उपयोग पते में फिर से लिखता है, तो केवल-Telegram बायपास को सक्षम करें:

```yaml
channels:
  telegram:
    network:
      dangerouslyAllowPrivateNetwork: true
```

    - यही वैकल्पिक सक्षमकरण प्रति अकाउंट `channels.telegram.accounts.<accountId>.network.dangerouslyAllowPrivateNetwork` पर उपलब्ध है।
    - यदि आपका प्रॉक्सी Telegram मीडिया होस्ट को `198.18.x.x` में रिज़ॉल्व करता है, तो पहले खतरनाक फ़्लैग को बंद ही रखें — वह रेंज पहले से ही डिफ़ॉल्ट रूप से अनुमत है।

    <Warning>
      `channels.telegram.network.dangerouslyAllowPrivateNetwork` Telegram मीडिया की SSRF सुरक्षा को कमज़ोर करता है। इसका उपयोग केवल विश्वसनीय, ऑपरेटर-नियंत्रित प्रॉक्सी परिवेशों (Clash, Mihomo, Surge फ़ेक-IP रूटिंग) के लिए करें, जो RFC 2544 बेंचमार्क रेंज के बाहर निजी या विशेष-उपयोग उत्तर बनाते हैं। सामान्य सार्वजनिक इंटरनेट से Telegram एक्सेस के लिए इसे बंद रखें।
    </Warning>

    - अस्थायी परिवेश ओवरराइड: `OPENCLAW_TELEGRAM_DISABLE_AUTO_SELECT_FAMILY=1`, `OPENCLAW_TELEGRAM_ENABLE_AUTO_SELECT_FAMILY=1`, `OPENCLAW_TELEGRAM_DNS_RESULT_ORDER=ipv4first`।
    - DNS उत्तरों को सत्यापित करें:

```bash
dig +short api.telegram.org A
dig +short api.telegram.org AAAA
```

  </Accordion>
</AccordionGroup>

</Accordion>

<Note>
बहु-अकाउंट प्राथमिकता: दो या अधिक अकाउंट ID कॉन्फ़िगर होने पर, डिफ़ॉल्ट रूटिंग को स्पष्ट बनाने के लिए `channels.telegram.defaultAccount` सेट करें (या `channels.telegram.accounts.default` शामिल करें)। अन्यथा OpenClaw पहले सामान्यीकृत अकाउंट ID पर वापस जाता है और `openclaw doctor` चेतावनी देता है। नामित अकाउंट `channels.telegram.allowFrom` / `groupAllowFrom` को इनहेरिट करते हैं, लेकिन `accounts.default.*` मानों को नहीं।
</Note>
