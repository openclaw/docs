क्लासिक चैनलों (`#room`) और डायरेक्ट मैसेज में OpenClaw का उपयोग करने के लिए IRC का उपयोग करें।
आधिकारिक IRC plugin इंस्टॉल करें, फिर उसे `channels.irc` के अंतर्गत कॉन्फ़िगर करें।

## त्वरित शुरुआत

1. Plugin इंस्टॉल करें:

```bash
openclaw plugins install @openclaw/irc
```

2. `~/.openclaw/openclaw.json` में कम-से-कम होस्ट, निक और शामिल होने वाले चैनल सेट करें:

```json5
{
  channels: {
    irc: {
      enabled: true,
      host: "irc.example.com",
      port: 6697,
      tls: true,
      nick: "openclaw-bot",
      channels: ["#openclaw"],
    },
  },
}
```

3. Gateway शुरू/पुनः शुरू करें:

```bash
openclaw gateway run
```

बॉट समन्वय के लिए निजी IRC सर्वर को प्राथमिकता दें। यदि आप जानबूझकर किसी सार्वजनिक IRC नेटवर्क का उपयोग करते हैं, तो सामान्य विकल्पों में Libera.Chat, OFTC और Snoonet शामिल हैं। बॉट या स्वार्म के बैकचैनल ट्रैफ़िक के लिए आसानी से अनुमान लगाए जा सकने वाले सार्वजनिक चैनलों से बचें।

## इनबाउंड स्थायित्व

OpenClaw प्रत्येक स्वीकृत IRC `PRIVMSG` को सामान्य नीति जाँच और एजेंट डिस्पैच से पहले अपनी टिकाऊ इनग्रेस क्यू में लिखता है। लंबित या पुनः प्रयास योग्य मैसेज Gateway के पुनः शुरू होने के बाद भी बने रहते हैं और प्रत्येक चैनल या डायरेक्ट-मैसेज पीयर के अनुसार क्रमबद्ध रहते हैं।

IRC पुनः चलाने योग्य डिलीवरी ID प्रदान नहीं करता और डिस्कनेक्ट हुए क्लाइंट से छूटे मैसेज दोबारा नहीं भेजता। इसलिए OpenClaw एक स्थानीय ID निर्दिष्ट करता है, जो केवल वर्तमान TCP कनेक्शन के भीतर स्थिर रहती है। क्यू स्थानीय स्वीकृति-से-डिस्पैच अवधि की सुरक्षा करती है; यह ऐसे मैसेज को पुनर्प्राप्त नहीं कर सकती जो OpenClaw तक कभी पहुँचा ही नहीं, न ही कनेक्शनों के बीच सर्वर द्वारा दोबारा भेजे गए मैसेज की डुप्लिकेट प्रतियों को हटा सकती है।

| ----------------------------- | ----------------------------- | ----------------------------------------------------------- |
| `host`                        | कोई नहीं (आवश्यक)               | IRC सर्वर होस्टनेम                                         |
| `port`                        | TLS के साथ `6697`, प्लेन `6667` | 1-65535                                                     |
| `tls`                         | `true`                        | केवल जानबूझकर प्लेनटेक्स्ट उपयोग करने के लिए `false` सेट करें                  |
| `nick`                        | कोई नहीं (आवश्यक)               | बॉट निक                                                    |
| `username`                    | निक, अन्यथा `openclaw`         | IRC उपयोगकर्ता नाम                                                |
| `realname`                    | `OpenClaw`                    | वास्तविक नाम/GECOS फ़ील्ड                                        |
| `password` / `passwordFile`   | कोई नहीं                          | सर्वर पासवर्ड; फ़ाइल का सामान्य फ़ाइल होना आवश्यक है                |
| `channels`                    | कोई नहीं                          | शामिल होने वाले चैनल (`["#openclaw"]`)                          |
| `accounts` / `defaultAccount` | कोई नहीं                          | बहु-अकाउंट सेटअप; एनवायरनमेंट वेरिएबल केवल डिफ़ॉल्ट अकाउंट को भरते हैं |

## सुरक्षा डिफ़ॉल्ट

- IRC, OpenClaw ऑपरेटर द्वारा प्रबंधित फ़ॉरवर्ड प्रॉक्सी रूटिंग के बाहर रॉ TCP/TLS सॉकेट का उपयोग करता है। जिन डिप्लॉयमेंट में सभी आउटगोइंग ट्रैफ़िक को उस फ़ॉरवर्ड प्रॉक्सी से भेजना आवश्यक है, उनमें सीधे IRC आउटगोइंग ट्रैफ़िक की स्पष्ट स्वीकृति न होने पर `channels.irc.enabled=false` सेट करें।
- `channels.irc.dmPolicy` का डिफ़ॉल्ट `"pairing"` है: अज्ञात DM प्रेषकों को एक पेयरिंग कोड मिलता है, जिसे आप `openclaw pairing approve irc <code>` से स्वीकृत करते हैं।
- `channels.irc.groupPolicy` का डिफ़ॉल्ट `"allowlist"` है।
- `groupPolicy="allowlist"` के साथ अनुमत चैनल निर्धारित करने के लिए `channels.irc.groups` सेट करें।
- TLS (`channels.irc.tls=true`) का उपयोग करें, जब तक कि आप जानबूझकर प्लेनटेक्स्ट ट्रांसपोर्ट स्वीकार नहीं करते।

## एक्सेस नियंत्रण

IRC चैनलों के लिए दो अलग-अलग "गेट" हैं:

1. **चैनल एक्सेस** (`groupPolicy` + `groups`): बॉट किसी चैनल से मैसेज स्वीकार करता है या नहीं।
2. **प्रेषक एक्सेस** (`groupAllowFrom` / प्रति-चैनल `groups["#channel"].allowFrom`): उस चैनल के भीतर बॉट को ट्रिगर करने की अनुमति किसे है।

कॉन्फ़िगरेशन कुंजियाँ:

- DM अनुमत-सूची (DM प्रेषक एक्सेस): `channels.irc.allowFrom`
- समूह प्रेषक अनुमत-सूची (चैनल प्रेषक एक्सेस): `channels.irc.groupAllowFrom`
- प्रति-चैनल नियंत्रण (चैनल + प्रेषक + मेंशन नियम): `channels.irc.groups["#channel"]` के साथ `requireMention`, `allowFrom`, `enabled`, `tools`, `toolsBySender`, `skills`, और `systemPrompt`
- `channels.irc.groupPolicy="open"` बिना कॉन्फ़िगर किए गए चैनलों को अनुमति देता है (**डिफ़ॉल्ट रूप से अब भी मेंशन-गेट लागू होता है**)

अनुमत-सूची की प्रविष्टियों में स्थिर प्रेषक पहचानों (`nick!user@host`) का उपयोग किया जाना चाहिए।
केवल निक से मिलान परिवर्तनशील है और तभी सक्षम होता है जब `channels.irc.dangerouslyAllowNameMatching: true`।

### सामान्य समस्या: `allowFrom` DM के लिए है, चैनलों के लिए नहीं

यदि आपको इस तरह के लॉग दिखाई दें:

- `irc: drop group sender alice!ident@host (policy=allowlist)`

...तो इसका अर्थ है कि प्रेषक को **समूह/चैनल** मैसेज के लिए अनुमति नहीं थी। इसे इनमें से किसी तरीके से ठीक करें:

- `channels.irc.groupAllowFrom` सेट करके (सभी चैनलों के लिए वैश्विक), या
- प्रति-चैनल प्रेषक अनुमत-सूचियाँ सेट करके: `channels.irc.groups["#channel"].allowFrom`

उदाहरण (`#openclaw` में किसी को भी बॉट से बात करने की अनुमति दें):

```json5
{
  channels: {
    irc: {
      groupPolicy: "allowlist",
      groups: {
        "#openclaw": { allowFrom: ["*"] },
      },
    },
  },
}
```

## जवाब ट्रिगर करना (मेंशन)

भले ही किसी चैनल की अनुमति हो (`groupPolicy` + `groups` के माध्यम से) और प्रेषक को भी अनुमति हो, OpenClaw समूह संदर्भों में डिफ़ॉल्ट रूप से **मेंशन-गेटिंग** लागू करता है। जब मैसेज में कनेक्टेड बॉट का निक शामिल हो या वह आपके कॉन्फ़िगर किए गए मेंशन पैटर्न से मेल खाए, तो बॉट इसे अपना मेंशन मानता है।

इसका अर्थ है कि जब तक मैसेज में बॉट से मेल खाने वाला मेंशन पैटर्न शामिल न हो, आपको `drop channel … (missing-mention)` जैसे लॉग दिखाई दे सकते हैं।

किसी IRC चैनल में बॉट से **मेंशन की आवश्यकता के बिना** जवाब दिलाने के लिए उस चैनल की मेंशन-गेटिंग अक्षम करें:

```json5
{
  channels: {
    irc: {
      groupPolicy: "allowlist",
      groups: {
        "#openclaw": {
          requireMention: false,
          allowFrom: ["*"],
        },
      },
    },
  },
}
```

या **सभी** IRC चैनलों को अनुमति देने (प्रति-चैनल अनुमत-सूची के बिना) और फिर भी मेंशन के बिना जवाब पाने के लिए:

```json5
{
  channels: {
    irc: {
      groupPolicy: "open",
      groups: {
        "*": { requireMention: false, allowFrom: ["*"] },
      },
    },
  },
}
```

## सुरक्षा टिप्पणी (सार्वजनिक चैनलों के लिए अनुशंसित)

यदि आप किसी सार्वजनिक चैनल में `allowFrom: ["*"]` की अनुमति देते हैं, तो कोई भी बॉट को प्रॉम्प्ट दे सकता है।
जोखिम कम करने के लिए उस चैनल के टूल प्रतिबंधित करें।

### चैनल में सभी के लिए समान टूल

```json5
{
  channels: {
    irc: {
      groups: {
        "#openclaw": {
          allowFrom: ["*"],
          tools: {
            deny: ["group:runtime", "group:fs", "gateway", "nodes", "cron", "browser"],
          },
        },
      },
    },
  },
}
```

### प्रत्येक प्रेषक के लिए अलग टूल (स्वामी को अधिक अधिकार मिलते हैं)

`"*"` पर अधिक कठोर और अपने निक पर अधिक उदार नीति लागू करने के लिए `toolsBySender` का उपयोग करें:

```json5
{
  channels: {
    irc: {
      groups: {
        "#openclaw": {
          allowFrom: ["*"],
          toolsBySender: {
            "*": {
              deny: ["group:runtime", "group:fs", "gateway", "nodes", "cron", "browser"],
            },
            "id:alice": {
              deny: ["gateway", "nodes", "cron"],
            },
          },
        },
      },
    },
  },
}
```

टिप्पणियाँ:

- `toolsBySender` कुंजियों में स्पष्ट उपसर्गों (`channel:`, `id:`, `e164:`, `username:`, `name:`) का उपयोग किया जाना चाहिए। IRC के लिए प्रेषक पहचान मान के साथ `id:` का उपयोग करें: अधिक सुदृढ़ मिलान के लिए `id:alice` या `id:alice!~alice@203.0.113.7`।
- उपसर्ग रहित पुरानी कुंजियाँ अब भी स्वीकार की जाती हैं, केवल `id:` के रूप में मिलाई जाती हैं और एक डिप्रिकेशन चेतावनी देती हैं।
- पहली मेल खाने वाली प्रेषक नीति प्रभावी होती है; `"*"` वाइल्डकार्ड फ़ॉलबैक है।

समूह एक्सेस और मेंशन-गेटिंग (और उनकी पारस्परिक क्रिया) के बारे में अधिक जानकारी के लिए देखें: [/channels/groups](/hi/channels/groups)।

## NickServ

कनेक्ट होने के बाद NickServ से पहचान सत्यापित करने के लिए:

```json5
{
  channels: {
    irc: {
      nickserv: {
        enabled: true,
        service: "NickServ",
        password: "your-nickserv-password",
      },
    },
  },
}
```

जब भी पासवर्ड सेट होता है, NickServ पहचान डिफ़ॉल्ट रूप से चलती है (इससे बाहर निकलने के लिए केवल `enabled` को `false` करना आवश्यक है)। `service` का डिफ़ॉल्ट `NickServ` है; `passwordFile`, इनलाइन `password` का एक विकल्प है।

कनेक्ट होने पर वैकल्पिक एक-बार पंजीकरण (`register: true` के लिए `registerEmail` आवश्यक है):

```json5
{
  channels: {
    irc: {
      nickserv: {
        register: true,
        registerEmail: "bot@example.com",
      },
    },
  },
}
```

निक पंजीकृत होने के बाद बार-बार REGISTER प्रयासों से बचने के लिए `register` अक्षम करें।

## एनवायरनमेंट वेरिएबल

डिफ़ॉल्ट अकाउंट इनका समर्थन करता है:

- `IRC_HOST`
- `IRC_PORT`
- `IRC_TLS`
- `IRC_NICK`
- `IRC_USERNAME`
- `IRC_REALNAME`
- `IRC_PASSWORD`
- `IRC_CHANNELS` (कॉमा से अलग किए गए)
- `IRC_NICKSERV_PASSWORD`
- `IRC_NICKSERV_REGISTER_EMAIL`

`IRC_HOST` को वर्कस्पेस `.env` से सेट नहीं किया जा सकता; [वर्कस्पेस `.env` फ़ाइलें](/hi/gateway/security) देखें।

## समस्या निवारण

- यदि बॉट कनेक्ट होता है लेकिन चैनलों में कभी जवाब नहीं देता, तो `channels.irc.groups` **और** यह सत्यापित करें कि मेंशन-गेटिंग मैसेज को छोड़ तो नहीं रही है (`missing-mention`)। यदि आप चाहते हैं कि वह पिंग के बिना जवाब दे, तो चैनल के लिए `requireMention:false` सेट करें।
- यदि लॉगिन विफल होता है, तो निक की उपलब्धता और सर्वर पासवर्ड सत्यापित करें।
- यदि किसी कस्टम नेटवर्क पर TLS विफल होता है, तो होस्ट/पोर्ट और सर्टिफ़िकेट सेटअप सत्यापित करें।
