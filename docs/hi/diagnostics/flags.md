डायग्नोस्टिक्स फ़्लैग वैश्विक रूप से
`logging.level` बढ़ाए बिना किसी एक सबसिस्टम के लिए अतिरिक्त लॉगिंग चालू करते हैं। जब तक कोई सबसिस्टम किसी फ़्लैग की जाँच न करे, उसका कोई प्रभाव नहीं होता।

## यह कैसे काम करता है

- फ़्लैग केस-असंवेदी स्ट्रिंग होते हैं, जिन्हें कॉन्फ़िगरेशन में `diagnostics.flags` और
  `OPENCLAW_DIAGNOSTICS` एन्वायरनमेंट ओवरराइड से रिज़ॉल्व करके, डुप्लिकेट हटाकर लोअरकेस में बदला जाता है।
- `name.*` स्वयं `name` और `name.` के अंतर्गत आने वाली हर चीज़ से मेल खाता है (उदाहरण के लिए,
  `telegram.*`, `telegram.http` से मेल खाता है)।
- `*` या `all` प्रत्येक फ़्लैग को सक्षम करता है।
- कॉन्फ़िगरेशन में `diagnostics.flags` बदलने के बाद Gateway को पुनः आरंभ करें; इसे
  हॉट-रीलोड नहीं किया जाता।

## ज्ञात फ़्लैग

| फ़्लैग                  | सक्षम करता है                                                   |
| --------------------- | --------------------------------------------------------- |
| `telegram.http`       | Telegram Bot API की HTTP त्रुटि लॉगिंग                       |
| `brave.http`          | Brave Search अनुरोध/प्रतिक्रिया/कैश लॉगिंग               |
| `profiler`            | उत्तर-चरण प्रोफ़ाइलर और Codex ऐप-सर्वर प्रोफ़ाइलर (दोनों) |
| `reply.profiler`      | केवल उत्तर-चरण प्रोफ़ाइलर                                 |
| `codex.profiler`      | केवल Codex ऐप-सर्वर प्रोफ़ाइलर                            |
| `health`              | Gateway स्वास्थ्य जाँच/खाता/बाइंडिंग के डीबग विवरण        |
| `ingress.timing`      | सेशन लोड, मॉडल चयन और मॉडल कैटलॉग की टाइमिंग              |
| `plugin.load-profile` | सिंक्रोनस Plugin मॉड्यूल-लोड की टाइमिंग                    |
| `timeline`            | संरचित JSONL टाइमलाइन आर्टिफ़ैक्ट (नीचे देखें)            |

## कॉन्फ़िगरेशन के माध्यम से सक्षम करें

```json
{
  "diagnostics": {
    "flags": ["telegram.http"]
  }
}
```

एकाधिक फ़्लैग:

```json
{
  "diagnostics": {
    "flags": ["telegram.http", "brave.http", "gateway.*"]
  }
}
```

## एन्वायरनमेंट ओवरराइड (एक बार के लिए)

```bash
OPENCLAW_DIAGNOSTICS=telegram.http,brave.http
```

मानों को कॉमा या व्हाइटस्पेस पर विभाजित किया जाता है। विशेष मान:

| मान                       | प्रभाव                                   |
| --------------------------- | ---------------------------------------- |
| `0`, `false`, `off`, `none` | कॉन्फ़िगरेशन को भी ओवरराइड करते हुए सभी फ़्लैग अक्षम करें |
| `1`, `true`, `all`, `*`     | प्रत्येक फ़्लैग सक्षम करें                        |

`OPENCLAW_DIAGNOSTICS=0` उस प्रक्रिया के लिए एन्वायरनमेंट और कॉन्फ़िगरेशन, दोनों से प्राप्त फ़्लैग अक्षम करता है,
जो फ़ाइल संपादित किए बिना कॉन्फ़िगरेशन में चालू छोड़े गए प्रोफ़ाइलर फ़्लैग को
अस्थायी रूप से शांत करने के लिए उपयोगी है।

## प्रोफ़ाइलर फ़्लैग

प्रोफ़ाइलर फ़्लैग हल्के टाइमिंग स्पैन को नियंत्रित करते हैं; बंद होने पर वे कोई ओवरहेड नहीं जोड़ते।

Gateway के एक रन के लिए प्रोफ़ाइलर द्वारा नियंत्रित सभी स्पैन सक्षम करें:

```bash
OPENCLAW_DIAGNOSTICS=profiler openclaw gateway run
```

केवल उत्तर-डिस्पैच प्रोफ़ाइलर स्पैन सक्षम करें:

```bash
OPENCLAW_DIAGNOSTICS=reply.profiler openclaw gateway run
```

केवल Codex ऐप-सर्वर स्टार्टअप/टूल/थ्रेड प्रोफ़ाइलर स्पैन सक्षम करें:

```bash
OPENCLAW_DIAGNOSTICS=codex.profiler openclaw gateway run
```

`profiler` उत्तर प्रोफ़ाइलर और Codex प्रोफ़ाइलर, दोनों को सक्षम करता है; केवल एक को सक्षम करने के लिए
स्कोप किए गए फ़्लैग नामों का उपयोग करें।

या इसे कॉन्फ़िगरेशन में सेट करें:

```json
{
  "diagnostics": {
    "flags": ["reply.profiler", "codex.profiler"]
  }
}
```

कॉन्फ़िगरेशन फ़्लैग बदलने के बाद Gateway को पुनः आरंभ करें। किसी प्रोफ़ाइलर फ़्लैग को अक्षम करने के लिए,
उसे `diagnostics.flags` से हटाकर पुनः आरंभ करें, या उस रन के लिए प्रत्येक डायग्नोस्टिक्स फ़्लैग को
ओवरराइड करने हेतु प्रक्रिया को `OPENCLAW_DIAGNOSTICS=0` के साथ आरंभ करें।

## टाइमलाइन आर्टिफ़ैक्ट

`timeline` फ़्लैग (उपनाम: `diagnostics.timeline`) बाहरी QA हार्नेस के लिए संरचित स्टार्टअप
और रनटाइम टाइमिंग इवेंट को JSONL के रूप में लिखता है:

```bash
OPENCLAW_DIAGNOSTICS=timeline \
OPENCLAW_DIAGNOSTICS_TIMELINE_PATH=/tmp/openclaw-timeline.jsonl \
openclaw gateway run
```

या इसे कॉन्फ़िगरेशन में सक्षम करें:

```json
{
  "diagnostics": {
    "flags": ["timeline"]
  }
}
```

आउटपुट पथ हमेशा `OPENCLAW_DIAGNOSTICS_TIMELINE_PATH` से आता है, भले ही
फ़्लैग स्वयं कॉन्फ़िगरेशन में सेट हो; पथ के लिए कोई कॉन्फ़िगरेशन कुंजी नहीं है।
जब `timeline` केवल कॉन्फ़िगरेशन से सक्षम होता है, तो सबसे शुरुआती कॉन्फ़िगरेशन-लोडिंग स्पैन
अनुपस्थित रहते हैं, क्योंकि OpenClaw ने अभी तक कॉन्फ़िगरेशन नहीं पढ़ा होता; बाद के स्टार्टअप स्पैन
सामान्य रूप से कैप्चर किए जाते हैं।

`OPENCLAW_DIAGNOSTICS=1`, `=all`, और `=*` भी टाइमलाइन सक्षम करते हैं, क्योंकि वे
प्रत्येक फ़्लैग सक्षम करते हैं। जब आपको केवल JSONL आर्टिफ़ैक्ट चाहिए और अन्य सभी
डायग्नोस्टिक्स फ़्लैग नहीं चाहिए, तो स्कोप किए गए `timeline` फ़्लैग को प्राथमिकता दें।

टाइमलाइन में इवेंट-लूप विलंब के नमूनों के लिए
`timeline` के अतिरिक्त एक और ऑप्ट-इन आवश्यक है: टाइमलाइन सक्षम करने के साथ
`OPENCLAW_DIAGNOSTICS_EVENT_LOOP=1` (या `on`/`true`/`yes`) सेट करें।

टाइमलाइन रिकॉर्ड `openclaw.diagnostics.v1` एनवेलप का उपयोग करते हैं और उनमें
प्रक्रिया आईडी, चरण नाम, स्पैन नाम, अवधियाँ, Plugin आईडी, निर्भरता
गणनाएँ, इवेंट-लूप विलंब के नमूने, प्रोवाइडर ऑपरेशन नाम, चाइल्ड-प्रोसेस की निकास
स्थिति और स्टार्टअप त्रुटि के नाम/संदेश शामिल हो सकते हैं। टाइमलाइन फ़ाइलों को स्थानीय
डायग्नोस्टिक्स आर्टिफ़ैक्ट मानें; उन्हें अपनी मशीन के बाहर साझा करने से पहले उनकी समीक्षा करें।

## लॉग कहाँ जाते हैं

फ़्लैग मानक डायग्नोस्टिक्स लॉग फ़ाइल में लॉग उत्सर्जित करते हैं। डिफ़ॉल्ट रूप से:

```
/tmp/openclaw/openclaw-YYYY-MM-DD.log
```

यदि आप `logging.file` सेट करते हैं, तो इसके बजाय उस पथ का उपयोग करें। लॉग JSONL होते हैं (प्रति पंक्ति एक JSON
ऑब्जेक्ट)। `logging.redactSensitive` के आधार पर संपादन अभी भी लागू होता है।
लॉग-पथ रिज़ॉल्यूशन, रोटेशन और संपादन के पूर्ण मॉडल के लिए [लॉगिंग](/hi/logging) देखें।

## लॉग निकालें

नवीनतम लॉग फ़ाइल चुनें:

```bash
ls -t /tmp/openclaw/openclaw-*.log | head -n 1
```

Telegram HTTP डायग्नोस्टिक्स के लिए फ़िल्टर करें:

```bash
rg "telegram http error" /tmp/openclaw/openclaw-*.log
```

Brave Search HTTP डायग्नोस्टिक्स के लिए फ़िल्टर करें:

```bash
rg "brave http" /tmp/openclaw/openclaw-*.log
```

या पुनरुत्पादन करते समय टेल करें:

```bash
tail -f /tmp/openclaw/openclaw-$(date +%F).log | rg "telegram http error"
```

रिमोट Gateway के लिए इसके बजाय `openclaw logs --follow` का उपयोग करें (देखें
[/cli/logs](/hi/cli/logs))।

## टिप्पणियाँ

- यदि `logging.level`, `warn` से अधिक सेट है, तो फ़्लैग-नियंत्रित लॉग
  दबाए जा सकते हैं। डिफ़ॉल्ट `info` उपयुक्त है।
- `brave.http` Brave Search अनुरोध URL/क्वेरी पैरामीटर, प्रतिक्रिया
  स्थिति/टाइमिंग और कैश हिट/मिस/लेखन इवेंट लॉग करता है। यह API कुंजी
  (जो अनुरोध हेडर के रूप में भेजी जाती है) या प्रतिक्रिया बॉडी लॉग नहीं करता, लेकिन खोज क्वेरी
  संवेदनशील हो सकती हैं।
- फ़्लैग को सक्षम छोड़ना सुरक्षित है; वे केवल विशिष्ट सबसिस्टम के लिए
  लॉग की मात्रा को प्रभावित करते हैं।
- लॉग गंतव्य, स्तर और संपादन बदलने के लिए [/logging](/hi/logging) का उपयोग करें।
