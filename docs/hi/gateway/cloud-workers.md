क्लाउड वर्कर किसी सत्र को अपना एजेंट लूप एक अस्थायी क्लाउड मशीन पर चलाने देते हैं, जबकि सत्र से संबंधित सब कुछ वहीं रहता है जहाँ वह हमेशा था: साइडबार में दिखाई देता है, लाइव स्ट्रीम होता है, और ट्रांसक्रिप्ट का स्वामित्व Gateway के पास रहता है। Gateway एक बॉक्स लीज़ पर लेता है, उस पर OpenClaw की पिन की गई प्रति इंस्टॉल करता है, सत्र के वर्कस्पेस को वहाँ सिंक करता है, और टर्न लूप को एक प्रतिबंधित `openclaw worker` प्रक्रिया को सौंप देता है। मॉडल कॉल Gateway के माध्यम से वापस प्रॉक्सी किए जाते हैं, इसलिए प्रदाता क्रेडेंशियल कभी आपकी मशीन से बाहर नहीं जाते, और प्रॉम्प्ट कैशिंग काम करती रहती है क्योंकि प्रदाता को एक निरंतर स्ट्रीम दिखाई देती है।

काम पूरा होने पर (या बॉक्स बंद हो जाने पर), मशीन हटा दी जाती है। स्थायी स्थिति — ट्रांसक्रिप्ट, वर्कस्पेस कमिट, प्लेसमेंट रिकॉर्ड — Gateway के पास रहती है।

<Note>
क्लाउड वर्कर वैकल्पिक हैं और प्रोफ़ाइल कॉन्फ़िगर करने तक दिखाई नहीं देते। बिना कॉन्फ़िगरेशन वाले इंस्टॉल में कोई नया RPC, कॉन्फ़िगरेशन या UI दिखाई नहीं देता।
</Note>

## क्या कहाँ चलता है

| विषय                                                 | स्थान                                                                         |
| ------------------------------------------------------- | -------------------------------------------------------------------------------- |
| एजेंट लूप + टूल (`exec`, `read`, `write`, `edit`, …) | क्लाउड वर्कर बॉक्स                                                                 |
| मॉडल इन्फ़रेंस और प्रदाता क्रेडेंशियल                | Gateway (`{provider, model}` संदर्भ द्वारा प्रॉक्सी किया गया)                               |
| ट्रांसक्रिप्ट (स्थायी, सत्र स्टोर)                     | Gateway                                                                          |
| साइडबार में लाइव स्ट्रीमिंग                         | Gateway फ़ैनआउट, जिसे वर्कर की पुनः चलाने योग्य इवेंट स्ट्रीम से डेटा मिलता है                      |
| वर्कस्पेस Git इतिहास                                   | बॉक्स पर बिना क्रेडेंशियल के बनाया जाता है; Gateway कमिट अपनाता है और पुश/PR का स्वामित्व रखता है |

बॉक्स को `sshd` के अतिरिक्त किसी इनबाउंड पोर्ट की आवश्यकता नहीं होती: Gateway पिन किए गए SSH के माध्यम से आउटबाउंड कनेक्शन बनाता है, और एक रिवर्स टनल वर्कर के WebSocket को वापस ले जाती है। बंडल किया गया Crabbox प्रदाता सार्वजनिक SSH मार्ग को अनिवार्य करता है और प्रबंधित Tailscale नामांकन अक्षम करता है। आउटबाउंड इंटरनेट पहुँच प्रदाता की नीति पर निर्भर है; डिफ़ॉल्ट AWS प्रोफ़ाइल इंटरनेट तक पहुँच सकती है, जब तक कि आप उसके नेटवर्क या सुरक्षा समूह को प्रतिबंधित न करें।

## आवश्यकताएँ

- एक वर्कर प्रदाता Plugin। बंडल किया गया `crabbox` Plugin [Crabbox](https://github.com/openclaw/crabbox) CLI को संचालित करता है, जो क्लाउड बैकएंड (AWS, Hetzner और अन्य) में लीज़ की मध्यस्थता करता है। `crabbox` बाइनरी `PATH` पर उपलब्ध होनी चाहिए (या `settings.binary` सेट करें), और प्रदाता क्रेडेंशियल पहले से कॉन्फ़िगर होने चाहिए। AWS प्रवेश के लिए Crabbox 0.38.1 या नया संस्करण आवश्यक है।
- Crabbox AWS वर्कर के लिए प्रभावी `aws.instanceProfile` खाली होना चाहिए। प्रदाता आवंटन से पहले `crabbox config show --json` की जाँच करता है, फिर आवश्यक करता है कि `crabbox inspect --json`, EC2 `DescribeInstances` से `providerMetadata.instanceProfileAttached: false` रिपोर्ट करे। इंस्टेंस भूमिका वाली या आधिकारिक मेटाडेटा के बिना लीज़ रोककर अस्वीकार कर दी जाती हैं।
- लीज़ पर ली गई मशीन पर Node.js। सामान्य क्लाउड इमेज में प्रायः यह उपलब्ध नहीं होता — इसे प्रोफ़ाइल की `setup` कमांड में इंस्टॉल करें।
- सत्र के स्वामित्व वाले प्रबंधित वर्कट्री वाला सत्र (`worktree: true` से बनाएँ)। डिस्पैच उस वर्कट्री की सामग्री को स्थानांतरित करता है; साधारण डायरेक्टरी मेनिफ़ेस्ट मिरर के रूप में सिंक होती हैं।

## कॉन्फ़िगरेशन

`openclaw.json` में `cloudWorkers.profiles` के अंतर्गत एक प्रोफ़ाइल जोड़ें:

```json
{
  "cloudWorkers": {
    "profiles": {
      "aws": {
        "provider": "crabbox",
        "install": "bundle",
        "settings": {
          "provider": "aws",
          "class": "standard",
          "ttl": "8h",
          "idleTimeout": "45m",
          "setup": "test -x /usr/bin/node || (curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash - && sudo apt-get install -y nodejs)"
        }
      }
    }
  }
}
```

प्रोफ़ाइल फ़ील्ड:

| कुंजी        | अर्थ                                                                                                                                                                                                                                        |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `provider` | Plugin द्वारा पंजीकृत वर्कर प्रदाता आईडी (बंडल किए गए Plugin के लिए `crabbox`)।                                                                                                                                                                  |
| `install`  | `bundle` (डिफ़ॉल्ट) चल रहे Gateway का बिल्ड भेजता है; `npm` पिन की गई अखंडता के साथ ठीक वही रिलीज़ किया गया Gateway संस्करण इंस्टॉल करता है। `npm` के लिए आवश्यक है कि Gateway पैकेज की गई रिलीज़ से चले।                                                      |
| `settings` | प्रदाता के स्वामित्व वाला JSON। crabbox के लिए: `provider` (बैकएंड), `class` (मशीन श्रेणी), `ttl`, `idleTimeout` (Go अवधियाँ), वैकल्पिक `setup` और निरपेक्ष `binary` पथ। OpenClaw इन लीज़ के लिए सार्वजनिक SSH को अनिवार्य करता है और प्रबंधित Tailscale अक्षम करता है। |
| `lifetime` | वैकल्पिक संग्रहीत नीति (`idleTimeoutMinutes`, `maxLifetimeMinutes`)।                                                                                                                                                                           |

### सेटअप कमांड

`settings.setup` लीज़ पर लिए गए बॉक्स पर उसके SSH के लिए तैयार होने के बाद और OpenClaw इंस्टॉल होने से पहले चलता है। यह प्रावधान के **प्रत्येक** प्रयास पर चलता है (बाधित डिस्पैच के बाद पुनः चलाने सहित), इसलिए इसे आइडेम्पोटेंट होना चाहिए — उदाहरण की तरह इंस्टॉल को `command -v`/`test -x` जाँच से सुरक्षित रखें। सेटअप विफल होने पर प्रदाता लीज़ रोक देता है और डिस्पैच सुरक्षित रूप से बंद होकर विफल हो जाता है; कोई आधा-कॉन्फ़िगर किया हुआ बॉक्स चलता नहीं छोड़ा जाता।

### इंस्टॉल चैनल

- **`bundle`** चल रहे Gateway के `dist`, छँटे हुए `package.json`, और बिल्ड द्वारा संदर्भित किसी भी वर्कस्पेस पैकेज को पैक करता है; ये सभी एक सामग्री हैश द्वारा सुरक्षित होते हैं। बॉक्स उस हैश से मूल बंडल का सत्यापन करता है, फिर उत्पादन npm निर्भरताएँ इंस्टॉल करता है (स्क्रिप्ट अक्षम रहती हैं)। वर्कर पर डेवलपमेंट बिल्ड इसी तरह चलाया जाता है।
- **`npm`** प्रमाणित करता है कि रिलीज़ सार्वजनिक रजिस्ट्री में मौजूद है, उसकी SHA-512 अखंडता पिन करता है, और Gateway से पूरी तरह मेल खाने वाला `openclaw@<version>` इंस्टॉल करता है।

## सत्र डिस्पैच करना

Control UI में, **नया सत्र** खोलें, ऐसा एजेंट चुनें जिसका कॉन्फ़िगर किया गया रनटाइम OpenClaw हो, **कहाँ** मेनू से कॉन्फ़िगर किया गया **क्लाउड · प्रोफ़ाइल** लक्ष्य चुनें, और कार्य शुरू करें। क्लाउड चयन आवश्यक प्रबंधित वर्कट्री को स्वचालित रूप से सक्षम करता है; Gateway सत्र बनाता है, डिस्पैच पूरा करता है, और उसके बाद ही पहला टर्न भेजता है। सत्र साइडबार में सर्वर बैज स्थायी प्लेसमेंट स्थिति दिखाता है। बाहरी CLI सत्र कैटलॉग के लिए क्लाउड लक्ष्य उपलब्ध नहीं कराए जाते।

समतुल्य RPC प्रवाह यह है:

प्रबंधित वर्कट्री के साथ सत्र बनाएँ, फिर उसे डिस्पैच करें (RPC के लिए `operator.admin` आवश्यक है और यह केवल प्रोफ़ाइल कॉन्फ़िगर होने पर उपलब्ध होता है):

क्लाउड वर्कर OpenClaw एजेंट रनटाइम चलाते हैं। ऐसा `openai/*` या अन्य मॉडल चुनें जो उस रनटाइम में रिज़ॉल्व होता हो; `claude-cli` जैसे बाहरी CLI रनटाइम के लिए कॉन्फ़िगर किए गए सत्र डिस्पैच नहीं किए जा सकते।

```bash
openclaw gateway call sessions.create \
  --params '{"key":"agent:main:big-refactor","worktree":true,"cwd":"/path/to/repo","worktreeName":"big-refactor"}'

openclaw gateway call sessions.dispatch \
  --timeout 1500000 \
  --params '{"key":"agent:main:big-refactor","profileId":"aws"}'
```

`sessions.dispatch` स्थानीय टर्न प्रवेश बंद करता है, सक्रिय कार्य समाप्त होने की प्रतीक्षा करता है, लीज़ का प्रावधान करता है, सेटअप चलाता है, OpenClaw बूटस्ट्रैप करता है, वर्कस्पेस सिंक करता है, और प्लेसमेंट के `active` वर्कर स्वामित्व तक पहुँचने पर लौटता है। पहले डिस्पैच के लिए कई मिनट रखें; जहाँ प्रदाता समर्थन करता है, वहाँ लीज़ और इंस्टॉल कैश किए जाते हैं। इसके बाद सत्र से सामान्य रूप से संवाद करें — टर्न स्वचालित रूप से वर्कर को भेजे जाते हैं।

पूर्ण हुए वर्कर टर्न, टर्न का दावा मुक्त होने से पहले, पात्र और आकार-सीमित वर्कस्पेस फ़ाइलों को सत्र के प्रबंधित वर्कट्री में वापस समन्वित करते हैं। टर्मिनल वर्कर इवेंट की पावती देने से पहले एक स्थायी लंबित-परिणाम फ़ेंस बनाया जाता है। फिर Gateway पूर्ण क्लाउड परिणाम को लागू करने से पहले `refs/openclaw/worker-results/` के अंतर्गत Git रेफ़ के रूप में स्टेज करता है, जिससे लागू करते समय Gateway रुक जाने पर भी क्लाउड संस्करण पुनर्प्राप्त किया जा सकता है। वर्कस्पेस परिणाम Git फ़ाइल अर्थ-विज्ञान का उपयोग करते हैं: नियमित फ़ाइलें, निष्पादन योग्य बिट, सिमलिंक, जोड़, परिवर्तन और विलोपन बनाए रखे जाते हैं, जबकि खाली डायरेक्टरी और अन्य डायरेक्टरी मोड नहीं रखे जाते। परिणामस्वरूप फ़ाइल परिवर्तन सामान्य समीक्षा और कमिट के लिए प्रबंधित वर्कट्री में बने रहते हैं।

लागू करने की प्रक्रिया डिस्पैच-समय मेनिफ़ेस्ट को मर्ज आधार के रूप में उपयोग करती है। केवल क्लाउड वाले परिवर्तन लागू किए जाते हैं, केवल स्थानीय परिवर्तन यथास्थान रहते हैं, और दोनों ओर बदले गए पथ त्रि-पक्षीय स्थानीय-रखें नीति का उपयोग करते हैं। विरोध वाला टर्न भी पूरा हो जाता है: ट्रांसक्रिप्ट सीमित पथ सारांश और स्टेज किया गया परिणाम रेफ़ रिपोर्ट करता है, प्लेसमेंट Control UI के लिए वही विरोध दिखाता है, और विरोध-रहित क्लाउड परिवर्तन लागू रहते हैं। सूचना में मौजूद क्लाउड फ़ाइल का निरीक्षण करने के लिए `git show <ref>:<path>` और उसे किसी भी वर्कस्पेस डायरेक्टरी से लेने के लिए शीर्ष-स्तरीय लिटरल-पाथस्पेक `git checkout <ref> -- <path>` कमांड शामिल होती है। कमांड को Bash या zsh (Windows पर Git Bash) में चलाएँ। यदि निरीक्षण बताता है कि पथ मौजूद नहीं है, तो क्लाउड परिणाम ने उसे हटा दिया है; सत्यापन करें और बचा हुआ स्थानीय पथ मैन्युअल रूप से हटाएँ। यदि चेकआउट फ़ाइल/डायरेक्टरी अवरोध रिपोर्ट करता है, तो बाधक स्थानीय पथ को स्थानांतरित करें या हटाएँ और पुनः प्रयास करें। यदि स्टेज किया गया रेफ़ ही उपलब्ध नहीं है, तो सूचना को बासी मानें और स्थानीय पथ न बदलें। विरोध वाले स्टेज किए गए रेफ़ सामान्य टर्न फ़ेंस मुक्त होने के बाद भी उपलब्ध रहते हैं; बाद का स्वच्छ परिणाम सूचना हटाता है और पुराने रेफ़ को निवृत्त करता है, जबकि स्पष्ट फ़ेंस निष्कासन अंतिम सफ़ाई सीमा है।

जब फ़ेंस किया गया परिणाम अभी भी समन्वित हो रहा हो, तो नया टर्न पूर्व दावे के मुक्त होने के लिए अधिकतम 15 सेकंड प्रतीक्षा करता है। यदि वह अब भी व्यस्त है, तो टर्न कार्रवाई योग्य “पिछले क्लाउड टर्न का वर्कस्पेस परिणाम अभी भी समन्वित हो रहा है” संदेश के साथ विफल होता है और शीघ्र ही पुनः प्रयास किया जा सकता है। पुनरारंभ होने पर, पुनर्प्राप्ति बासी दावे साफ़ करने से पहले लंबित और स्टेज किए गए परिणाम खोजती है, उनके स्थानीय अनुप्रयोग को पूरा करती है या पुनः प्रयास करती है, और परिणाम सुरक्षित रखने के बाद ही निष्क्रिय एनवायरनमेंट वापस लेती है। सीमित SQLite रोलबैक जर्नल पहले से स्वीकार किए गए म्यूटेशन को दोबारा चलाए बिना बाधित फ़ाइलसिस्टम अनुप्रयोग को पुनर्प्राप्त करने योग्य बनाता है।

जब काम पूरा हो जाए और कोई टर्न न चल रहा हो, तो सत्र मेनू खोलें और **क्लाउड वर्कर रोकें…** चुनें। Gateway एनवायरनमेंट नष्ट करने से पहले एक अंतिम वर्कस्पेस समन्वयन करता है। जो प्लेसमेंट पहले से `draining` या `reconciling` में है, वह विघटन पूरा कर रहा है; सत्र हटाने से पहले उसके बैज के `reclaimed` बनने तक प्रतीक्षा करें।

टूटे हुए या नियंत्रण से बाहर संलग्न वर्कर के लिए, ऑपरेटर अंतिम उपाय के रूप में `{ "force": true }` के साथ `environments.destroy` कॉल कर सकता है। बलपूर्वक विघटन प्लेसमेंट को स्थायी रूप से विफल चिह्नित करता है और एनवायरनमेंट नष्ट करने से पहले किसी भी असमन्वित रिमोट परिणाम को छोड़ देता है।

समतुल्य प्रशासनिक RPC यह है:

```bash
openclaw gateway call sessions.reclaim \
  --timeout 600000 \
  --params '{"key":"agent:main:big-refactor"}'
```

प्लेसमेंट एक टिकाऊ स्टेट मशीन (`local → requested → provisioning → syncing → starting → active`) से होकर आगे बढ़ता है, इसलिए डिस्पैच के बीच Gateway के पुनः आरंभ होने पर मशीनें लीक होने के बजाय उनका समाधान किया जाता है। मॉडल टर्न विफल होने पर सक्रिय प्लेसमेंट पुनः प्रयास के लिए उपलब्ध रहता है। वर्कस्पेस पाथ के टकरावों में स्थानीय संस्करण रखा जाता है, क्लाउड परिणाम का शेष भाग लागू किया जाता है, और निरीक्षण के लिए स्टेज किया गया क्लाउड रेफ़ सुरक्षित रखा जाता है; अन्य समाधान या लाइफ़साइकल विफलताओं में उनकी टिकाऊ रिकवरी फ़ेंस और डायग्नोस्टिक टेल तब तक बनी रहती है, जब तक रिकवरी सुरक्षित रूप से पुनः प्रयास या परिवेश को पुनः प्राप्त नहीं कर सकती।

## सुरक्षा मॉडल

- **बंद वर्कर इनग्रेस।** वर्कर टनल किए गए सॉकेट पर, विधियों की बंद अनुमत-सूची वाले एक समर्पित प्रोटोकॉल से संचार करते हैं — कोई वर्कर ऑपरेटर RPC को कॉल नहीं कर सकता।
- **निर्मित क्रेडेंशियल, संग्रहीत अवस्था में हैश किए गए।** प्रत्येक डिस्पैच एक वर्कर क्रेडेंशियल बनाता है; Gateway केवल उसका हैश संग्रहीत करता है। क्रेडेंशियल रोटेशन और ओनर-एपॉक फ़ेंसिंग यह सुनिश्चित करते हैं कि प्रत्येक सत्र के लिए अधिकतम एक सक्रिय ओनर हो — दोबारा कनेक्ट होने वाले पुराने वर्कर को फ़ेंस किया जाता है, कभी मर्ज नहीं किया जाता।
- **होस्ट-की पिनिंग।** प्रोविज़निंग के समय प्रोवाइडर को बॉक्स की SSH होस्ट की उपलब्ध करानी होगी; बूटस्ट्रैप सख़्त पिनिंग के साथ कनेक्ट होता है और इसके बिना सुरक्षित रूप से विफल हो जाता है।
- **बॉक्स पर स्थायी मॉडल, फ़ोर्ज या क्लाउड क्रेडेंशियल नहीं।** मॉडल प्रमाणीकरण Gateway पर रहता है (इनफ़रेंस `{provider, model}` संदर्भ द्वारा जाता है), वर्कस्पेस git कमिट फ़ोर्ज क्रेडेंशियल के बिना लिखे जाते हैं, और सेटअप से पहले इंस्टेंस रोल के लिए Crabbox AWS लीज़ मेटाडेटा की आधिकारिक रूप से जाँच की जाती है। सेटअप कमांड को भी क्रेडेंशियल-मुक्त रखें।
- **प्रोवाइडर-स्वामित्व वाला इग्रेस।** रिवर्स टनल प्रत्यक्ष मॉडल एक्सेस की OpenClaw की आवश्यकता समाप्त कर देती है, लेकिन OpenClaw प्रोवाइडर फ़ायरवॉल को दोबारा नहीं लिखता। कार्य की आवश्यकता होने पर वर्कर प्रोवाइडर में आउटबाउंड ट्रैफ़िक प्रतिबंधित करें।
- **टिकाऊ, ठीक-एक-बार ट्रांसक्रिप्ट।** वर्कर सत्र के लीफ़ के विरुद्ध कंपेयर-एंड-स्वैप प्रोटोकॉल के माध्यम से ट्रांसक्रिप्ट बैच कमिट करता है; पुराना बेस भुगतान किए गए आउटपुट को डुप्लिकेट या रीबेस करने के बजाय रन को फ़ेल-स्टॉप कर देता है।

## समस्या निवारण

- **`sessions.dispatch` एक अज्ञात विधि है** — कोई `cloudWorkers.profiles` कॉन्फ़िगर नहीं है, या कॉलर के पास `operator.admin` नहीं है।
- **"क्लाउड वर्कर टर्न के लिए OpenClaw रनटाइम आवश्यक है"** — ऐसा मॉडल चुनें जिसका कॉन्फ़िगर किया गया रनटाइम OpenClaw हो। `claude-cli` जैसे बाहरी CLI रनटाइम वर्कर इनफ़रेंस का समर्थन नहीं करते।
- **"वर्कर बूटस्ट्रैप के लिए लीज़ किए गए होस्ट पर Node.js आवश्यक है"** — `settings.setup` में Node इंस्टॉल जोड़ें (ऊपर देखें)।
- **AWS इंस्टेंस-रोल सत्यापन विफल होता है** — `aws.instanceProfile` (और सेट होने पर `CRABBOX_AWS_INSTANCE_PROFILE`) साफ़ करें। Crabbox 0.38.1 या नया संस्करण इंस्टॉल करें; पुराने बाइनरी AWS प्रवेश के लिए आवश्यक आधिकारिक `providerMetadata.instanceProfileAttached` अनुबंध उपलब्ध नहीं कराते।
- **प्रोवाइडर त्रुटि के साथ डिस्पैच विफल होता है** — प्लेसमेंट रिकॉर्ड और `environments.list` अंतिम त्रुटि रखते हैं, जिसमें सेटअप/बूटस्ट्रैप stderr टेल शामिल है। विफलता पर बॉक्स नष्ट कर दिए जाते हैं, इसलिए वह टेल प्राथमिक फ़ोरेंसिक साक्ष्य है।
- **डिस्पैच करते समय क्लाइंट टाइमआउट** — `openclaw gateway call` का डिफ़ॉल्ट टाइमआउट 10s है; `--timeout` के लिए पर्याप्त बड़ा मान दें (दोनों ही स्थितियों में डिस्पैच सर्वर की ओर चलता रहता है, और प्रोविज़निंग के दौरान पुनः प्रयास को `session cannot dispatch from placement provisioning` के साथ अस्वीकार कर दिया जाता है)।
- **क्लाउड वर्कस्पेस टकराव सूचना** — टर्न पूरा हो गया और प्रत्येक सूचीबद्ध पाथ का स्थानीय संस्करण रखा गया। क्लाउड संस्करण का निरीक्षण करने या उसे अपनाने के लिए सूचना में दिए गए स्टेज्ड-रेफ़ कमांड का उपयोग करें; बिना टकराव वाले परिवर्तनों के लिए पुनः प्रयास आवश्यक नहीं है, क्योंकि वे पहले ही लागू हो चुके हैं।
- **“पिछले क्लाउड टर्न के वर्कस्पेस परिणाम का समाधान अभी भी जारी है”** — Gateway ने पिछले परिणाम की टिकाऊ फ़ेंस के लिए थोड़ी देर प्रतीक्षा की और सत्र क्लेम प्राप्त नहीं कर सका। समाधान पूरा होने की प्रतीक्षा करें, फिर टर्न का पुनः प्रयास करें; Gateway को पुनः आरंभ करना सुरक्षित है, क्योंकि रिकवरी मृत वर्कर को पुनः प्राप्त करने से पहले स्टेज किए गए परिणामों को सुरक्षित रखती है।
- **लीज़ रखरखाव** — `crabbox list --provider <backend>` सक्रिय लीज़ दिखाता है; `crabbox stop --provider <backend> --id <lease>` किसी एक को मैन्युअल रूप से रिलीज़ करता है। निष्क्रिय लीज़ प्रोफ़ाइल के `idleTimeout` पर समाप्त हो जाती हैं।
