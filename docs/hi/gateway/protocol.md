OpenClaw के लिए Gateway WS प्रोटोकॉल एकमात्र नियंत्रण तल और नोड परिवहन है।
ऑपरेटर और नोड क्लाइंट (CLI, वेब UI, macOS ऐप, iOS/Android नोड,
हेडलेस नोड) WebSocket के माध्यम से कनेक्ट होते हैं और हैंडशेक के समय एक
**भूमिका** और **स्कोप** घोषित करते हैं।

## परिवहन और फ़्रेमिंग

- WebSocket, टेक्स्ट फ़्रेम, JSON पेलोड।
- पहला फ़्रेम `connect` अनुरोध **होना अनिवार्य है**।
- कनेक्शन से पहले के फ़्रेम अधिकतम 64 KiB (`MAX_PREAUTH_PAYLOAD_BYTES`) तक सीमित हैं। हैंडशेक के बाद,
  `hello-ok.policy.maxPayload` और
  `hello-ok.policy.maxBufferedBytes` का पालन करें। डायग्नोस्टिक्स सक्षम होने पर, अत्यधिक बड़े
  इनबाउंड फ़्रेम और धीमे आउटबाउंड बफ़र, Gateway द्वारा फ़्रेम बंद करने या
  छोड़ने से पहले `payload.large` इवेंट उत्सर्जित करते हैं। इन इवेंट में `surface`, बाइट
  आकार, सीमाएँ और एक सुरक्षित कारण कोड होता है; संदेश बॉडी, अटैचमेंट
  सामग्री, कच्चे फ़्रेम बाइट, टोकन, कुकी या सीक्रेट कभी नहीं होते।

फ़्रेम के आकार:

- अनुरोध: `{type:"req", id, method, params}`
- प्रतिक्रिया: `{type:"res", id, ok, payload|error}`
- इवेंट: `{type:"event", event, payload, seq?, stateVersion?}`

साइड इफ़ेक्ट उत्पन्न करने वाली विधियों के लिए आइडेम्पोटेंसी कुंजियाँ आवश्यक हैं (स्कीमा देखें)।

## हैंडशेक

Gateway कनेक्शन से पहले एक चुनौती भेजता है:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

क्लाइंट `connect` के साथ उत्तर देता है:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway `hello-ok` के साथ प्रतिक्रिया देता है:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`, `features`, `snapshot`, `policy`, और `auth`, सभी
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) के लिए आवश्यक हैं। डिवाइस टोकन जारी न होने पर भी
`auth` तय की गई भूमिका/स्कोप की रिपोर्ट करता है (ऊपर दिया
आकार)। `pluginSurfaceUrls` वैकल्पिक है और Plugin सरफ़ेस नामों (जैसे
`canvas`) को स्कोप किए गए होस्टेड URL से मैप करता है; इसकी समय-सीमा समाप्त हो सकती है, इसलिए नोड
नई प्रविष्टि के लिए `{ "surface": "canvas" }` के साथ `node.pluginSurface.refresh` को कॉल करते हैं।
बहिष्कृत `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
पथ समर्थित नहीं है; Plugin सरफ़ेस का उपयोग करें।
स्नैपशॉट का वैकल्पिक `appliedConfigHash`, सक्रिय Gateway रनटाइम द्वारा स्वीकार किया गया
रिज़ॉल्व किया हुआ स्रोत-कॉन्फ़िग संशोधन है। क्लाइंट इसकी तुलना
`config.get.configRevisionHash` से करके निर्धारित कर सकते हैं कि नए सहेजे गए कॉन्फ़िग के लिए अभी भी
पुनः आरंभ करना आवश्यक है या नहीं। `config.get.hash` कॉन्फ़िग लेखन टकराव रक्षकों द्वारा उपयोग किया जाने वाला
कच्चा रूट-फ़ाइल संशोधन बना रहता है।

जब Gateway अभी स्टार्टअप साइडकार पूर्ण कर रहा हो, तब `connect`,
`details.reason: "startup-sidecars"` और `retryAfterMs` के साथ पुनः प्रयास योग्य
`UNAVAILABLE` त्रुटि लौटा सकता है। इसे अंतिम हैंडशेक विफलता मानने के बजाय
अपनी कनेक्शन समय-सीमा के भीतर पुनः प्रयास करें।

डिवाइस टोकन जारी होने पर, `hello-ok.auth` उसे जोड़ता है:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

अंतर्निहित QR/सेटअप-कोड बूटस्ट्रैप एक मोबाइल हस्तांतरण पथ है। सफल
बेसलाइन सेटअप-कोड कनेक्शन एक प्राथमिक नोड टोकन और एक सीमित
ऑपरेटर टोकन लौटाता है:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

यह ऑपरेटर हस्तांतरण जानबूझकर सीमित है: मोबाइल ऑपरेटर लूप और
नेटिव सेटअप शुरू करने के लिए पर्याप्त, जिसमें Talk कॉन्फ़िग पढ़ने हेतु `operator.talk.secrets`
शामिल है, लेकिन पेयरिंग-म्यूटेशन स्कोप और `operator.admin` नहीं। अधिक व्यापक
पेयरिंग/एडमिन एक्सेस के लिए अलग से स्वीकृत पेयरिंग या टोकन प्रवाह आवश्यक है।
`hello-ok.auth.deviceTokens` को केवल तभी स्थायी बनाएँ जब बूटस्ट्रैप प्रमाणीकरण किसी विश्वसनीय
परिवहन (`wss://` या लूपबैक/स्थानीय पेयरिंग) पर चला हो।

विश्वसनीय समान-प्रोसेस बैकएंड क्लाइंट (`client.id: "gateway-client"`,
`client.mode: "backend"`) साझा Gateway टोकन/पासवर्ड से प्रमाणीकरण करते समय
प्रत्यक्ष लूपबैक कनेक्शन पर `device` को छोड़ सकते हैं। यह पथ आंतरिक
नियंत्रण-तल RPC (जैसे सबएजेंट सत्र अपडेट) के लिए आरक्षित है और पुरानी
CLI/डिवाइस पेयरिंग बेसलाइन को स्थानीय बैकएंड कार्य अवरुद्ध करने से रोकता है। रिमोट,
ब्राउज़र-मूल, नोड और स्पष्ट डिवाइस-टोकन/डिवाइस-पहचान क्लाइंट अब भी
सामान्य पेयरिंग और स्कोप-अपग्रेड जाँच से गुजरते हैं।

### वर्कर भूमिका और बंद प्रोटोकॉल

क्लाउड वर्कर, Gateway के स्वामित्व वाली और होस्ट-कुंजी-पिन की गई SSH टनल के माध्यम से
समर्पित लूपबैक प्रवेश का उपयोग करते हैं। यह केवल वर्कर पहचान स्वीकार करता है और कभी भी
सामान्य प्रमाणीकरण, नोड इवेंट, ऑपरेटर RPC या Plugin विधियाँ प्रेषित नहीं करता। एक सख्त
`connect` स्थिर-अवस्था में हैश किए गए, अल्पकालिक क्रेडेंशियल को सत्यापित करता है, जो
परिवेश, बंडल हैश, स्वामी युग, RPC-सेट संस्करण, समाप्ति और एक नल-योग्य सत्र से बँधा होता है;
यह वर्तमान संस्करण और फ़ीचर सेट की अलग से जाँच करता है। सफलता पर न्यूनतम
`worker-hello-ok` लौटता है; फ़ीचर समझौता सामान्य प्रोटोकॉल संस्करण से स्वतंत्र है।
फ़्रेम 64 KiB से कम रहते हैं, सिवाय इसके कि समझौता किया गया `worker.inference.start`
फ़्रेम अधिकतम 25 MiB का हो सकता है। बंद अनुमति-सूची में `worker.heartbeat`,
`worker.transcript.commit`, `worker.live-event`, `worker.inference.start`, और
`worker.inference.cancel` शामिल हैं।

ट्रांसक्रिप्ट कमिट स्वामी-युग फ़ेंसिंग, Gateway के स्वामित्व वाली सत्र बाइंडिंग,
बेस-लीफ़ तुलना-और-स्वैप तथा टिकाऊ अनुक्रम रीप्ले का उपयोग करते हैं; Gateway सामान्य
सत्र राइटर के माध्यम से ट्रांसक्रिप्ट प्रविष्टि और पैरेंट ID उत्पन्न करता है। प्रत्येक RPC पर
स्वामित्व और समाप्ति की पुनः जाँच की जाती है।

### क्लाइंट क्षमताएँ

ऑपरेटर क्लाइंट `connect.params.caps` में वैकल्पिक क्षमताएँ घोषित कर सकते हैं:

- `tool-events`: संरचित टूल जीवनचक्र इवेंट स्वीकार करता है।
- `inline-widgets`: होस्टेड इनलाइन विजेट टूल परिणाम रेंडर कर सकता है।

क्लाइंट क्षमताएँ कनेक्ट किए गए क्लाइंट का वर्णन करती हैं, प्राधिकरण का नहीं। एजेंट टूल आवश्यक क्षमताएँ घोषित कर सकते हैं; जब तक प्रत्येक आवश्यकता मूल क्लाइंट के `caps` में मौजूद न हो, Gateway उन टूल को छोड़ देता है। चैनल से शुरू किए गए रन में Gateway क्लाइंट क्षमताएँ नहीं होतीं, इसलिए क्षमता-नियंत्रित टूल तब भी अनुपलब्ध रहते हैं जब टूल नीति उन्हें स्पष्ट रूप से अनुमति देती हो।

## भूमिकाएँ और स्कोप

संपूर्ण ऑपरेटर स्कोप मॉडल, स्वीकृति-समय जाँच और साझा-सीक्रेट
अर्थविज्ञान के लिए [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) देखें।

भूमिकाएँ:

- `operator`: नियंत्रण-तल क्लाइंट (CLI/UI/ऑटोमेशन)।
- `node`: क्षमता होस्ट (कैमरा/स्क्रीन/कैनवास/system.run)।
- `worker`: समर्पित, बंद वर्कर प्रोटोकॉल पर क्लाउड निष्पादन होस्ट।

ऑपरेटर स्कोप (`src/gateway/operator-scopes.ts`), संपूर्ण बंद सेट:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`includeSecrets: true` के साथ `talk.config` के लिए `operator.talk.secrets` (या
`operator.admin`) आवश्यक है। सीक्रेट शामिल होने पर, सक्रिय Talk प्रदाता
क्रेडेंशियल को `talk.resolved.config.apiKey` से पढ़ें; `talk.providers.<id>.apiKey`
स्रोत के आकार में बना रहता है और SecretRef ऑब्जेक्ट या संशोधित स्ट्रिंग हो सकता है।

Plugin द्वारा पंजीकृत Gateway RPC विधियाँ अपना स्वयं का ऑपरेटर स्कोप माँग सकती हैं,
लेकिन ये आरक्षित कोर उपसर्ग हमेशा `operator.admin`
(`src/shared/gateway-method-policy.ts`) में रिज़ॉल्व होते हैं: `config.*`, `exec.approvals.*`,
`wizard.*`, `update.*`।

विधि स्कोप केवल पहला द्वार है। `chat.send` के माध्यम से पहुँची कुछ स्लैश कमांड
अधिक सख्त कमांड-स्तरीय जाँच लागू करती हैं: स्थायी `/config set` और
`/config unset` लेखन के लिए `operator.admin` आवश्यक है, यहाँ तक कि उन Gateway क्लाइंट के लिए भी
जिनके पास पहले से कमतर ऑपरेटर स्कोप है।

`node.pair.approve` में आधार विधि स्कोप (`operator.pairing`) के अतिरिक्त,
लंबित अनुरोध के घोषित `commands` (`src/infra/node-pairing-authz.ts`) के आधार पर
स्वीकृति-समय की एक अतिरिक्त स्कोप जाँच होती है:

| घोषित कमांड                                                                                                                   | आवश्यक स्कोप                          |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| कोई नहीं                                                                                                                      | `operator.pairing`                    |
| सामान्य कमांड                                                                                                                 | `operator.pairing` + `operator.write` |
| इसमें `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir`, या `system.execApprovals.get/set` शामिल है | `operator.pairing` + `operator.admin` |

### क्षमताएँ/कमांड/अनुमतियाँ (नोड)

नोड कनेक्शन के समय क्षमता दावे घोषित करते हैं:

- `caps`: `camera`, `canvas`, `screen`,
  `location`, `voice`, और `talk` जैसी उच्च-स्तरीय क्षमता श्रेणियाँ।
- `commands`: आह्वान के लिए कमांड अनुमति-सूची।
- `permissions`: सूक्ष्म टॉगल (जैसे `screen.record`, `camera.capture`)।

Gateway इन्हें **दावों** के रूप में मानता है और सर्वर-साइड अनुमति-सूचियाँ लागू करता है।
सफल कनेक्ट या पुनः कनेक्ट होने के बाद, कनेक्टेड Node वैकल्पिक, एजेंट को दिखाई देने वाले Plugin या MCP टूल
डिस्क्रिप्टर को `node.pluginTools.update` के साथ प्रकाशित कर सकते हैं।
घोषणात्मक MCP इन्वेंट्री परिवर्तनों को लागू करने के लिए हेडलेस Node होस्ट पुनः आरंभ होते हैं।
यह अपडेट विधि ही प्रकाशन का एकमात्र पथ है; Plugin टूल डिस्क्रिप्टर
`connect` पैरामीटर में स्वीकार नहीं किए जाते। प्रत्येक डिस्क्रिप्टर को प्रदाता-सुरक्षित टूल `name` का उपयोग करना और
Node की वर्तमान कमांड अनुमति-सूची में मौजूद किसी `command` का नाम देना आवश्यक है। Gateway युग्मित Node से प्राप्त डिस्क्रिप्टर
मेटाडेटा पर भरोसा करता है, स्वीकृत कमांड
सतह के बाहर के डिस्क्रिप्टर को फ़िल्टर करता है, Node के डिस्कनेक्ट होने पर उन्हें हटा देता है और किसी अन्य Node के कैटलॉग में
बदलाव करने के ऑपरेटर प्रयासों को अस्वीकार करता है। Node द्वारा प्रकाशित डिस्क्रिप्टर को अनदेखा करने के लिए `gateway.nodes.pluginTools.enabled: false`
सेट करें।

कनेक्टेड Node होस्ट अपनी पूरी Skills प्रतिस्थापन कैटलॉग
`node.skills.update` के साथ प्रकाशित करते हैं। यह Node-भूमिका विधि ही Node Skills प्रकाशन का एकमात्र
पथ है; Skills को `connect` पैरामीटर में स्वीकार नहीं किया जाता। प्रत्येक डिस्क्रिप्टर में एक
सुरक्षित नाम, विवरण और सीमाबद्ध `SKILL.md` सामग्री होती है। Gateway उस
सामग्री को सामान्य Skills लोडर से पार्स करता है, Node के कनेक्ट रहने के दौरान उसे एजेंट Skills स्नैपशॉट में
शामिल करता है और डिस्कनेक्ट होने पर हटा देता है। Node द्वारा प्रकाशित Skills को अनदेखा करने के लिए
`gateway.nodes.skills.enabled: false` सेट करें।

## उपस्थिति

- `system-presence` डिवाइस पहचान के अनुसार कुंजीबद्ध प्रविष्टियाँ लौटाता है, जिनमें
  `deviceId`, `roles` और `scopes` शामिल हैं, ताकि UI प्रत्येक डिवाइस के लिए एक पंक्ति दिखा सकें, भले ही
  वह ऑपरेटर और Node दोनों के रूप में कनेक्ट हो।
- `node.list` में वैकल्पिक `lastSeenAtMs` और `lastSeenReason` शामिल होते हैं। कनेक्टेड
  Node वर्तमान कनेक्शन समय को `connect` कारण के साथ रिपोर्ट करते हैं; युग्मित Node
  किसी विश्वसनीय Node इवेंट के माध्यम से स्थायी पृष्ठभूमि उपस्थिति भी रिपोर्ट कर सकते हैं।

नेटिव macOS Node सीमाबद्ध इनपुट निष्क्रिय समय के साथ प्रमाणित `node.presence.activity` इवेंट भी
भेज सकते हैं। Gateway अपनी घड़ी पर गतिविधि टाइमस्टैम्प निर्धारित करता है,
`node.list` और `node.describe` के माध्यम से सबसे हाल में सक्रिय कनेक्टेड Mac को प्रदर्शित करता है और
रीड-स्कोप वाले क्लाइंट को `node.presence` अपडेट प्रसारित करता है।
चयन, गोपनीयता, मॉडल संदर्भ और सूचना-रूटिंग व्यवहार के लिए [सक्रिय कंप्यूटर उपस्थिति](/nodes/presence) देखें।

### Node पृष्ठभूमि सक्रिय इवेंट

पृष्ठभूमि वेक के दौरान युग्मित Node के सक्रिय रहने को रिकॉर्ड करने के लिए Node,
उसे कनेक्टेड चिह्नित किए बिना, `event: "node.presence.alive"` के साथ `node.event` को कॉल करते हैं:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` एक बंद एनम है: `background`, `silent_push`, `bg_app_refresh`,
`significant_location`, `manual`, `connect`। अज्ञात मान
`background` (`src/shared/node-presence.ts`) में सामान्यीकृत होते हैं। इवेंट केवल
प्रमाणित Node डिवाइस सत्रों के लिए स्थायी रूप से सहेजा जाता है; डिवाइस-रहित या अयुग्मित सत्र
`handled: false` लौटाते हैं।

सफल Gateway एक संरचित परिणाम लौटाते हैं:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

पुराने Gateway, `node.event` के लिए केवल `{ "ok": true }` लौटा सकते हैं; इसे
स्वीकृत RPC मानें, स्थायी उपस्थिति का स्थायी संग्रहण नहीं।

## प्रसारण इवेंट का स्कोप निर्धारण

सर्वर द्वारा पुश किए गए प्रसारण इवेंट स्कोप-नियंत्रित होते हैं, ताकि युग्मन-स्कोप या केवल-Node
सत्र निष्क्रिय रूप से सत्र सामग्री प्राप्त न करें
(`src/gateway/server-broadcast.ts`):

- चैट, एजेंट और टूल-परिणाम फ़्रेम (स्ट्रीम किए गए `agent` इवेंट, टूल-परिणाम
  इवेंट) के लिए कम-से-कम `operator.read` आवश्यक है। इसके बिना सत्र इन
  फ़्रेम को पूरी तरह छोड़ देते हैं।
- Plugin द्वारा परिभाषित `plugin.*` प्रसारण डिफ़ॉल्ट रूप से `operator.write` या
  `operator.admin` तक सीमित होते हैं; `plugin.approval.requested` / `plugin.approval.resolved` जैसी स्पष्ट प्रविष्टियाँ
  इसके बजाय `operator.approvals` का उपयोग करती हैं।
- स्थिति/ट्रांसपोर्ट इवेंट (`heartbeat`, `presence`, `tick`, कनेक्ट/डिस्कनेक्ट
  जीवनचक्र) अप्रतिबंधित रहते हैं, ताकि प्रत्येक प्रमाणित सत्र ट्रांसपोर्ट की स्थिति देख सके।
- अज्ञात प्रसारण इवेंट परिवार डिफ़ॉल्ट रूप से स्कोप-नियंत्रित (विफलता पर बंद)
  होते हैं, जब तक कि कोई पंजीकृत हैंडलर उन्हें स्पष्ट रूप से शिथिल न करे।

प्रत्येक क्लाइंट कनेक्शन अपनी अलग प्रति-क्लाइंट क्रम संख्या रखता है, इसलिए अलग-अलग क्लाइंट द्वारा
इवेंट स्ट्रीम के अलग-अलग स्कोप-फ़िल्टर किए गए उपसमुच्चय देखने पर भी उस सॉकेट पर प्रसारण
एकसमान रूप से बढ़ते क्रम में बने रहते हैं।

    - `plugins.install` (`operator.admin`), `{ source: "official", pluginId }` वाली आधिकारिक कैटलॉग प्रविष्टि या `{ source: "clawhub", packageName, version?, acknowledgeClawHubRisk? }` वाला ClawHub पैकेज इंस्टॉल करता है। ClawHub इंस्टॉल Gateway के विश्वास, अखंडता और इंस्टॉल-नीति जाँचों को बनाए रखते हैं। सफल इंस्टॉल के लिए Gateway पुनः आरंभ करना आवश्यक है।
    - `plugins.setEnabled` (`operator.admin`), `{ pluginId, enabled }` के साथ एक स्थापित Plugin की सक्षम नीति बदलता है। प्रतिक्रिया में अपडेट की गई कैटलॉग प्रविष्टि, पुनः आरंभ मेटाडेटा और स्लॉट-चयन की सभी चेतावनियाँ शामिल होती हैं।
    - `plugins.uninstall` (`operator.admin`), `{ pluginId }` वाले बाहरी रूप से स्थापित Plugin को हटाता है: कॉन्फ़िगरेशन संदर्भ, इंस्टॉल रिकॉर्ड और प्रबंधित फ़ाइलें। बंडल किए गए Plugin अनइंस्टॉल नहीं किए जा सकते, केवल अक्षम किए जा सकते हैं। प्रतिक्रिया हटाने की कार्रवाइयाँ सूचीबद्ध करती है और हमेशा Gateway पुनः आरंभ करना आवश्यक बनाती है।

  </Accordion>

  <Accordion title="संदेश और लॉग">
    - `send` चैट रनर के बाहर चैनल/खाता/थ्रेड-लक्षित प्रेषण के लिए प्रत्यक्ष आउटबाउंड-वितरण RPC है।
    - `logs.tail` कर्सर/सीमा और अधिकतम-बाइट नियंत्रणों के साथ कॉन्फ़िगर किए गए Gateway फ़ाइल-लॉग का अंतिम भाग लौटाता है।

  </Accordion>

  <Accordion title="ऑपरेटर टर्मिनल">
    - `terminal.open` किसी स्पष्ट `agentId` या डिफ़ॉल्ट एजेंट के लिए होस्ट PTY शुरू करता है और निर्धारित एजेंट, कार्यशील डायरेक्टरी, शेल और परिसीमन स्थिति लौटाता है।
    - `terminal.input`, `terminal.resize`, और `terminal.close` केवल कॉल करने वाले कनेक्शन के स्वामित्व वाले सत्रों पर काम करते हैं।
    - `terminal.upload` अधिकतम 16 MiB की एक base64 फ़ाइल स्वीकार करता है, उसे सत्र के Gateway या युग्मित-Node होस्ट पर एक निजी 24-घंटे की अस्थायी डायरेक्टरी में रखता है और निरपेक्ष पथ लौटाता है। कॉलर को अभी भी वह पथ पेस्ट करना या किसी अन्य तरीके से उपयोग करना होगा; RPC कभी भी टर्मिनल इनपुट नहीं लिखता या कोई कमांड निष्पादित नहीं करता।
    - `terminal.data` और `terminal.exit` ईवेंट केवल उस कनेक्शन पर स्ट्रीम होते हैं जिसके स्वामित्व में सत्र है।
    - जिन सत्रों का कनेक्शन टूट जाता है, वे समाप्त नहीं बल्कि अलग किए जाते हैं: वे `gateway.terminal.detachedSessionTimeoutSeconds` तक दोबारा जोड़े जा सकते हैं (डिफ़ॉल्ट 300; `0` डिस्कनेक्ट होने पर समाप्त करने का व्यवहार पुनर्स्थापित करता है), जबकि हालिया आउटपुट सीमित सर्वर-साइड बफ़र में एकत्र होता रहता है।
    - `terminal.list` जोड़े जा सकने वाले सत्र लौटाता है; `terminal.attach` किसी सक्रिय या अलग किए गए सत्र को कॉल करने वाले कनेक्शन से फिर बाँधता है और रीप्ले बफ़र लौटाता है (tmux-शैली का अधिग्रहण—पिछले सक्रिय स्वामी को कारण `detached` के साथ `terminal.exit` प्राप्त होता है); `terminal.text` बिना जुड़े बफ़र को सादे टेक्स्ट के रूप में पढ़ता है।
    - हर टर्मिनल विधि के लिए `operator.admin` आवश्यक है; `gateway.terminal.enabled` को स्पष्ट रूप से true होना चाहिए। पूर्णतः सैंडबॉक्स किए गए एजेंट अस्वीकार कर दिए जाते हैं और एजेंट नीति में बदलाव मौजूदा तथा प्रगतिरत PTY बंद कर देता है, जिनमें अलग किए गए PTY भी शामिल हैं।

  </Accordion>

  <Accordion title="Talk और TTS">
    - `talk.catalog` वाक्, स्ट्रीमिंग ट्रांसक्रिप्शन और रीयलटाइम वॉइस के लिए केवल-पढ़ने योग्य Talk प्रदाता कैटलॉग लौटाता है: कैनॉनिकल प्रदाता आईडी, रजिस्ट्री उपनाम, लेबल, कॉन्फ़िगर की गई स्थिति, वैकल्पिक समूह-स्तरीय `ready` परिणाम, प्रदर्शित मॉडल/वॉइस आईडी, कैनॉनिकल मोड, ट्रांसपोर्ट, ब्रेन रणनीतियाँ और रीयलटाइम ऑडियो/क्षमता फ़्लैग; इसमें प्रदाता के सीक्रेट नहीं लौटाए जाते और वैश्विक कॉन्फ़िगरेशन में बदलाव नहीं किया जाता। वर्तमान Gateway रनटाइम प्रदाता चयन लागू करने के बाद `ready` सेट करते हैं; पुराने Gateway पर इसकी अनुपस्थिति को असत्यापित मानें।
    - `talk.config` प्रभावी Talk कॉन्फ़िगरेशन पेलोड लौटाता है; `includeSecrets` के लिए `operator.talk.secrets` (या `operator.admin`) आवश्यक है।
    - `talk.session.create`, `realtime/gateway-relay`, `transcription/gateway-relay`, या `stt-tts/managed-room` के लिए Gateway-स्वामित्व वाला Talk सत्र बनाता है। `stt-tts/managed-room` के लिए, `sessionKey` देने वाले `operator.write` कॉलर को सीमित सत्र-कुंजी दृश्यता के लिए `spawnedBy` भी देना होगा; बिना स्कोप वाले `sessionKey` निर्माण और `brain: "direct-tools"` के लिए `operator.admin` आवश्यक है।
    - `talk.session.join` प्रबंधित-रूम सत्र टोकन को सत्यापित करता है, आवश्यकतानुसार `session.ready` या `session.replaced` उत्सर्जित करता है और रूम/सत्र मेटाडेटा के साथ हालिया Talk ईवेंट लौटाता है, लेकिन कभी भी सादा-टेक्स्ट टोकन या उसका हैश नहीं लौटाता।
    - `talk.session.appendAudio` Gateway-स्वामित्व वाले रीयलटाइम रिले और ट्रांसक्रिप्शन सत्रों में base64 PCM इनपुट ऑडियो जोड़ता है।
    - `talk.session.startTurn`, `talk.session.endTurn`, और `talk.session.cancelTurn` स्थिति साफ़ होने से पहले पुराने टर्न अस्वीकार करते हुए प्रबंधित-रूम टर्न जीवनचक्र संचालित करते हैं।
    - `talk.session.cancelOutput` सहायक ऑडियो आउटपुट रोकता है, मुख्यतः Gateway रिले सत्रों में VAD-नियंत्रित हस्तक्षेप के लिए।
    - `talk.session.submitToolResult` Gateway-स्वामित्व वाले रीयलटाइम रिले सत्र द्वारा उत्सर्जित प्रदाता टूल कॉल पूरा करता है। अनुरोध प्रदाता ब्रिज द्वारा प्रदर्शित किसी भी एसिंक्रोनस पूर्णता संकेत की प्रतीक्षा करता है; विफल सबमिशन संबद्ध रन को सक्रिय रखते हैं और सफल टूल-परिणाम ईवेंट उत्सर्जित नहीं करते। अंतरिम टूल आउटपुट के लिए `options: { willContinue: true }` दें या जब प्रदाता ब्रिज दमन समर्थन घोषित करता हो और परिणाम से दूसरी प्रतिक्रिया शुरू नहीं होनी चाहिए, तब `options: { suppressResponse: true }` दें।
    - `talk.session.steer` Gateway-स्वामित्व वाले एजेंट-समर्थित Talk सत्र में सक्रिय-रन वॉइस नियंत्रण भेजता है: `{ sessionId, text, mode? }`, जहाँ `mode`, `status`, `steer`, `cancel`, या `followup` होता है; मोड न दिए जाने पर उसका वर्गीकरण बोले गए टेक्स्ट से किया जाता है।
    - `talk.session.close` Gateway-स्वामित्व वाला रिले, ट्रांसक्रिप्शन या प्रबंधित-रूम सत्र बंद करता है और अंतिम Talk ईवेंट उत्सर्जित करता है।
    - `talk.mode` WebChat/Control UI क्लाइंट के लिए वर्तमान Talk मोड स्थिति सेट/प्रसारित करता है।
    - `talk.client.create`, `webrtc` या `provider-websocket` का उपयोग करके क्लाइंट-स्वामित्व वाला रीयलटाइम प्रदाता सत्र बनाता है, जबकि Gateway कॉन्फ़िगरेशन, क्रेडेंशियल, निर्देश और टूल नीति का स्वामी रहता है।
    - `talk.client.toolCall` क्लाइंट-स्वामित्व वाले रीयलटाइम ट्रांसपोर्ट को प्रदाता टूल कॉल Gateway नीति तक अग्रेषित करने देता है। पहला समर्थित टूल `openclaw_agent_consult` है; क्लाइंट को रन आईडी मिलती है और प्रदाता-विशिष्ट टूल परिणाम सबमिट करने से पहले वह सामान्य चैट जीवनचक्र ईवेंट की प्रतीक्षा करता है।
    - `talk.client.steer` क्लाइंट-स्वामित्व वाले रीयलटाइम ट्रांसपोर्ट के लिए सक्रिय-रन वॉइस नियंत्रण भेजता है। Gateway, `sessionKey` से सक्रिय एम्बेडेड रन निर्धारित करता है और संचालन को चुपचाप छोड़ने के बजाय संरचित स्वीकृत/अस्वीकृत परिणाम लौटाता है।
    - `talk.event` रीयलटाइम, ट्रांसक्रिप्शन, STT/TTS, प्रबंधित-रूम, टेलीफ़ोनी और मीटिंग अडैप्टर के लिए एकमात्र Talk ईवेंट चैनल है।
    - `talk.speak` सक्रिय Talk वाक् प्रदाता के माध्यम से वाक् संश्लेषित करता है।
    - `tts.status` TTS सक्षम स्थिति, सक्रिय प्रदाता, फ़ॉलबैक प्रदाता और प्रदाता कॉन्फ़िगरेशन स्थिति लौटाता है।

    - `tts.enable` और `tts.disable` TTS प्राथमिकता स्थिति को टॉगल करते हैं।
    - `tts.setProvider` पसंदीदा TTS प्रदाता अपडेट करता है।

  <Accordion title="सीक्रेट, कॉन्फ़िगरेशन, अपडेट और विज़ार्ड">
    - `secrets.reload` सक्रिय SecretRefs को फिर से निर्धारित करता है और केवल पूर्ण सफलता पर रनटाइम सीक्रेट स्थिति बदलता है।
    - `secrets.resolve` किसी विशिष्ट कमांड/लक्ष्य समूह के लिए कमांड-लक्ष्य सीक्रेट असाइनमेंट निर्धारित करता है।
    - `config.get` वर्तमान ऑन-डिस्क कॉन्फ़िगरेशन स्नैपशॉट, अपरिष्कृत रूट-फ़ाइल `hash`, निर्धारित `configRevisionHash` और सक्रिय Gateway रनटाइम द्वारा स्वीकृत निर्धारित संशोधन के लिए वैकल्पिक `appliedConfigHash` लौटाता है।
    - `config.set` सत्यापित कॉन्फ़िगरेशन पेलोड लिखता है।
    - `config.patch` आंशिक कॉन्फ़िगरेशन अपडेट को मर्ज करता है। विनाशकारी ऐरे प्रतिस्थापन के लिए प्रभावित पथ का `replacePaths` में होना आवश्यक है; ऐरे प्रविष्टियों के अंतर्गत नेस्टेड ऐरे, `agents.list[].skills` जैसे `[]` पथों का उपयोग करते हैं।
    - `config.apply` पूरे कॉन्फ़िगरेशन पेलोड को सत्यापित करके प्रतिस्थापित करता है।
    - `config.schema` Control UI और CLI टूलिंग द्वारा उपयोग किया जाने वाला लाइव कॉन्फ़िगरेशन स्कीमा पेलोड लौटाता है: स्कीमा, `uiHints`, संस्करण, जनरेशन मेटाडेटा और लोड किए जा सकने पर Plugin + चैनल स्कीमा मेटाडेटा। इसमें UI के समान लेबल/सहायता टेक्स्ट से `title` / `description` मेटाडेटा शामिल होता है, जिसमें मेल खाने वाले फ़ील्ड दस्तावेज़ उपलब्ध होने पर नेस्टेड ऑब्जेक्ट, वाइल्डकार्ड, ऐरे-आइटम और `anyOf` / `oneOf` / `allOf` संयोजन शाखाएँ भी शामिल हैं।
    - `config.schema.lookup` एक कॉन्फ़िगरेशन पथ के लिए पथ-सीमित लुकअप पेलोड लौटाता है: सामान्यीकृत पथ, उथला स्कीमा Node, मेल खाता संकेत + `hintPath`, वैकल्पिक `reloadKind` और UI/CLI ड्रिल-डाउन के लिए तात्कालिक चाइल्ड सारांश। `reloadKind`, `restart`, `hot`, या `none` (`src/config/schema.ts`) में से एक होता है और अनुरोधित पथ के लिए Gateway कॉन्फ़िगरेशन रीलोड प्लानर को प्रतिबिंबित करता है। लुकअप स्कीमा Node उपयोगकर्ता-सामना दस्तावेज़ और सामान्य सत्यापन फ़ील्ड (`title`, `description`, `type`, `enum`, `const`, `format`, `pattern`, संख्यात्मक/स्ट्रिंग/ऐरे/ऑब्जेक्ट सीमाएँ, `additionalProperties`, `deprecated`, `readOnly`, `writeOnly`) बनाए रखते हैं। चाइल्ड सारांश `key`, सामान्यीकृत `path`, `type`, `required`, `hasChildren`, वैकल्पिक `reloadKind`, साथ ही मेल खाते `hint` / `hintPath` प्रदर्शित करते हैं।
    - `update.run` Gateway अपडेट प्रवाह चलाता है और केवल अपडेट सफल होने पर रीस्टार्ट निर्धारित करता है; सत्र वाले कॉलर `continuationMessage` शामिल कर सकते हैं, ताकि स्टार्टअप रीस्टार्ट निरंतरता कतार के माध्यम से एजेंट का एक अनुवर्ती टर्न फिर शुरू करे। कंट्रोल प्लेन से पैकेज-मैनेजर अपडेट और पर्यवेक्षित git-चेकआउट अपडेट, सक्रिय Gateway के भीतर पैकेज ट्री बदलने या चेकआउट/बिल्ड आउटपुट में बदलाव करने के बजाय अलग प्रबंधित-सेवा हैंडऑफ़ का उपयोग करते हैं। शुरू किया गया हैंडऑफ़ `result.reason: "managed-service-handoff-started"` और `handoff.status: "started"` के साथ `ok: true` लौटाता है; अनुपलब्ध या विफल हैंडऑफ़ `managed-service-handoff-unavailable` या `managed-service-handoff-failed` के साथ `ok: false`, और मैन्युअल शेल अपडेट आवश्यक होने पर `handoff.command` लौटाते हैं। अनुपलब्ध का अर्थ है कि OpenClaw के पास सुरक्षित पर्यवेक्षक सीमा या स्थायी सेवा पहचान नहीं है, जैसे systemd के लिए `OPENCLAW_SYSTEMD_UNIT`। शुरू किए गए हैंडऑफ़ के दौरान, रीस्टार्ट सेंटिनल थोड़े समय के लिए `stats.reason: "restart-health-pending"` रिपोर्ट कर सकता है; निरंतरता तब तक विलंबित रहती है जब तक CLI रीस्टार्ट किए गए Gateway को सत्यापित करके अंतिम `ok` सेंटिनल नहीं लिख देता।
    - `update.status` नवीनतम अपडेट रीस्टार्ट सेंटिनल को रीफ़्रेश करके लौटाता है, जिसमें उपलब्ध होने पर रीस्टार्ट के बाद चल रहा संस्करण भी शामिल होता है।
    - `wizard.start`, `wizard.next`, `wizard.status`, और `wizard.cancel` WS RPC पर ऑनबोर्डिंग विज़ार्ड उपलब्ध कराते हैं।

  </Accordion>

  <Accordion title="एजेंट और वर्कस्पेस सहायक">
    - `agents.list` कॉन्फ़िगर की गई एजेंट प्रविष्टियाँ लौटाता है, जिनमें प्रभावी मॉडल और रनटाइम मेटाडेटा शामिल होते हैं।
    - `agents.create`, `agents.update`, और `agents.delete` एजेंट रिकॉर्ड और वर्कस्पेस वायरिंग प्रबंधित करते हैं।
    - `agents.files.list`, `agents.files.get`, और `agents.files.set` किसी एजेंट के लिए उपलब्ध कराई गई बूटस्ट्रैप वर्कस्पेस फ़ाइलें प्रबंधित करते हैं।
    - `audit.activity.list` संस्करणित, केवल-मेटाडेटा गतिविधि लेजर लौटाता है; `audit.list` संगतता-सुरक्षित रन/टूल RPC बना रहता है।
    - `agents.workspace.list` और `agents.workspace.get` (`operator.read`) [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) में वर्णित विश्वसनीय ऑपरेटर डोमेन के क्लाइंटों के लिए एजेंट की वर्कस्पेस डायरेक्टरी की केवल-पठन, पृष्ठांकित ब्राउज़िंग उपलब्ध कराते हैं। अनुरोध केवल वर्कस्पेस-सापेक्ष पथ स्वीकार करते हैं; पठन वास्तविक पथ में बदले गए वर्कस्पेस रूट तक सीमित रहता है (सिमलिंक और हार्डलिंक के माध्यम से बाहर निकलना अस्वीकार किया जाता है), आकार-सीमित होता है, और UTF-8 टेक्स्ट तथा सामान्य इमेज प्रकारों (base64) तक सीमित होता है। प्रत्युत्तर होस्ट वर्कस्पेस पथ उजागर नहीं करते। इस नेमस्पेस में कोई लेखन ऑपरेशन नहीं है।
    - `tasks.list`, `tasks.get`, और `tasks.cancel` SDK और ऑपरेटर क्लाइंटों को Gateway टास्क लेजर उपलब्ध कराते हैं। नीचे [टास्क लेजर RPC](#task-ledger-rpcs) देखें।
    - `artifacts.list`, `artifacts.get`, और `artifacts.download` स्पष्ट `sessionKey`, `runId`, या `taskId` स्कोप के लिए ट्रांसक्रिप्ट से प्राप्त आर्टिफ़ैक्ट सारांश और डाउनलोड उपलब्ध कराते हैं। रन और टास्क क्वेरी स्वामित्व वाले सेशन को सर्वर-साइड पर निर्धारित करती हैं और केवल मेल खाते उद्गम वाली ट्रांसक्रिप्ट मीडिया लौटाती हैं; असुरक्षित या स्थानीय URL स्रोतों को सर्वर-साइड से फ़ेच करने के बजाय असमर्थित डाउनलोड लौटाए जाते हैं।
    - `environments.list` और `environments.status` Gateway-स्थानीय और Node परिवेश खोज को बनाए रखते हैं। कॉन्फ़िगर किए गए क्लाउड वर्कर और पहले के प्रोफ़ाइलों द्वारा छोड़े गए स्थायी रिकॉर्ड `providerId`, वैकल्पिक `leaseId`, `state`, `ageMs`, वैकल्पिक `idleMs`, और `attachedSessionIds` वाला `worker` मेटाडेटा जोड़ते हैं। वर्कर जीवनचक्र स्थितियाँ `requested`, `provisioning`, `bootstrapping`, `ready`, `attached`, `idle`, `draining`, `destroying`, `destroyed`, `failed`, और `orphaned` हैं।
    - `environments.create` (`{ profileId, idempotencyKey }`) कॉन्फ़िगर किए गए Plugin प्रदाता प्रोफ़ाइल से एक वर्कर प्रावधानित करता है; समान कुंजी से दोबारा प्रयास करने पर स्थायी ऑपरेशन का पुनः उपयोग होता है। `environments.destroy` (`{ environmentId }`) स्थायी वर्कर परिवेश को पुनरावृत्ति-सुरक्षित ढंग से हटाने का अनुरोध करता है। दोनों के लिए `operator.admin` आवश्यक है, दोनों कंट्रोल-प्लेन लेखन हैं, और दोनों स्थिति प्रत्युत्तरों में प्रयुक्त समान परिवेश सारांश संरचना लौटाते हैं।
    - `agent.identity.get` किसी एजेंट या सेशन के लिए प्रभावी सहायक पहचान लौटाता है।
    - `agent.wait` रन समाप्त होने की प्रतीक्षा करता है और उपलब्ध होने पर अंतिम स्नैपशॉट लौटाता है।

  <Accordion title="सेशन नियंत्रण">
    - `sessions.list` वर्तमान सेशन इंडेक्स लौटाता है, जिसमें एजेंट रनटाइम बैकएंड कॉन्फ़िगर होने पर प्रति-पंक्ति `agentRuntime` मेटाडेटा शामिल होता है। क्लाउड-वर्कर प्लेसमेंट सक्षम होने या स्थायी पुनर्प्राप्ति स्थिति मौजूद होने पर, सेशन पंक्तियों में एक बंद `placement` स्थिति (`local`, `requested`, `provisioning`, `syncing`, `starting`, `active`, `draining`, `reconciling`, `reclaimed`, या `failed`) के साथ स्थिति-विशिष्ट परिवेश, स्वामी-युग, वर्कस्पेस, बंडल, ACK-कर्सर, या पुनर्प्राप्ति फ़ील्ड भी शामिल होते हैं।
    - `sessions.subscribe` और `sessions.unsubscribe` वर्तमान WS क्लाइंट के लिए सेशन परिवर्तन इवेंट सदस्यताएँ चालू या बंद करते हैं।
    - `sessions.messages.subscribe` और `sessions.messages.unsubscribe` एक सेशन के लिए ट्रांसक्रिप्ट/संदेश इवेंट सदस्यताएँ चालू या बंद करते हैं। उन अनुमोदनों के लिए स्वच्छ किए गए `session.approval` जीवनचक्र इवेंट भी प्राप्त करने हेतु `includeApprovals: true` पास करें, जिनके सहेजे गए दर्शक-वर्ग में ठीक वही सेशन शामिल है और जिनकी समीक्षक बाइंडिंग सदस्यता लेने वाले क्लाइंट को अधिकृत करती है। इसके बाद सदस्यता प्रत्युत्तर में सीमित लंबित `approvalReplay` शामिल होता है; `truncated` के false होने पर यह प्रामाणिक होता है। यह ऑप्ट-इन प्रत्येक सदस्यता कॉल के लिए है, स्थायी नहीं: `includeApprovals: true` के बिना उसी सेशन की पुनः सदस्यता लेने पर मौजूदा अनुमोदन सदस्यता हट जाती है। सामान्य सेशन-पठन अधिकार के अतिरिक्त, इस ऑप्ट-इन के लिए `operator.admin`, या युग्मित डिवाइस पर `operator.approvals` आवश्यक है।
    - `sessions.preview` विशिष्ट सेशन कुंजियों के लिए सीमित ट्रांसक्रिप्ट पूर्वावलोकन लौटाता है।
    - `sessions.describe` सटीक सेशन कुंजी के लिए एक Gateway सेशन पंक्ति लौटाता है।
    - `sessions.resolve` किसी सेशन लक्ष्य को निर्धारित या मानकीकृत करता है।
    - `sessions.create` एक नई सेशन प्रविष्टि बनाता है। वैकल्पिक `model` और `thinkingLevel` मान प्रारंभिक मॉडल और रीजनिंग ओवरराइड को परमाण्विक रूप से सहेजते हैं। `worktree: true` एक प्रबंधित वर्कट्री प्रावधानित करता है; वैकल्पिक `worktreeBaseRef`/`worktreeName` आधार रेफ़रेंस और ब्रांच नाम चुनते हैं, और `execNode` (`operator.admin`) सेशन निष्पादन को Node होस्ट से बाँधता है। बनाया गया वर्कट्री परिणाम में वापस दिया जाता है और सेशन पंक्ति (`worktree: { id, branch, repoRoot }`) में सहेजा जाता है। जब प्रविष्टि बन जाती है, लेकिन उसका नेस्टेड प्रारंभिक `chat.send` अस्वीकार हो जाता है, तो सफल परिणाम में `runStarted: false` और `runError` शामिल होते हैं; क्लाइंट प्रॉम्प्ट सुरक्षित रख सकते हैं और लौटाई गई सेशन कुंजी पर फिर प्रयास कर सकते हैं।
    - `sessions.dispatch` (`operator.admin`) सेशन के स्वामित्व वाले प्रबंधित वर्कट्री सहित किसी मौजूदा स्थानीय OpenClaw सेशन को कॉन्फ़िगर किए गए क्लाउड-वर्कर प्रोफ़ाइल पर ले जाता है। `{ key, profileId, agentId? }` पास करें। कोई वर्कर प्रोफ़ाइल कॉन्फ़िगर न होने पर यह विधि उपलब्ध नहीं होती, सक्रिय कार्य समाप्त होने की प्रतीक्षा से पहले स्थानीय टर्न प्रवेश बंद करती है, और प्लेसमेंट के `active` वर्कर स्वामित्व तक पहुँचने के बाद ही लौटती है। प्रेषण एकतरफ़ा है; वर्कर से स्थानीय परिवेश में वापस खींचना इस RPC का हिस्सा नहीं है।
    - `sessions.groups.list`, `sessions.groups.put`, `sessions.groups.rename`, और `sessions.groups.delete` Gateway के स्वामित्व वाली कस्टम सेशन समूह सूची (नाम + प्रदर्शन क्रम) प्रबंधित करते हैं। सदस्यता प्रत्येक सेशन के `category` फ़ील्ड में रहती है; नाम बदलने और हटाने पर सदस्य सेशन सर्वर-साइड पर अपडेट होते हैं।
    - `sessions.send` किसी मौजूदा सेशन में संदेश भेजता है।
    - `sessions.steer` किसी सक्रिय सेशन के लिए व्यवधान डालकर दिशा देने वाला प्रकार है।
    - `sessions.abort` किसी सेशन का सक्रिय कार्य निरस्त करता है। `key` के साथ वैकल्पिक `runId` पास करें, या उन सक्रिय रनों के लिए केवल `runId` पास करें जिन्हें Gateway किसी सेशन से संबद्ध कर सकता है।
    - `sessions.patch` सेशन मेटाडेटा/ओवरराइड अपडेट करता है और निर्धारित मानक मॉडल के साथ प्रभावी `agentRuntime` की रिपोर्ट करता है।
    - `sessions.reset`, `sessions.delete`, और `sessions.compact` सेशन रखरखाव करते हैं।
    - `sessions.get` पूरी सहेजी गई सेशन पंक्ति लौटाता है।
    - चैट निष्पादन अब भी `chat.history`, `chat.send`, `chat.abort`, और `chat.inject` का उपयोग करता है। UI क्लाइंटों के लिए `chat.history` को प्रदर्शन हेतु सामान्यीकृत किया जाता है: इनलाइन डायरेक्टिव टैग दृश्यमान टेक्स्ट से हटा दिए जाते हैं, सादे-टेक्स्ट वाले टूल-कॉल XML पेलोड (`<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>`, और काटे गए टूल-कॉल ब्लॉक) तथा लीक हुए ASCII/पूर्ण-चौड़ाई मॉडल नियंत्रण टोकन हटा दिए जाते हैं, पूर्णतः मौन-टोकन वाली सहायक पंक्तियाँ (सटीक `NO_REPLY` / `no_reply`) छोड़ दी जाती हैं, और अत्यधिक बड़ी पंक्तियों को प्लेसहोल्डर से बदला जा सकता है।
    - `chat.message.get` एकल दृश्यमान ट्रांसक्रिप्ट प्रविष्टि के लिए योगात्मक, सीमित पूर्ण-संदेश रीडर है। `sessionKey`, सेशन चयन के एजेंट-स्कोप वाला होने पर वैकल्पिक `agentId`, और `chat.history` के माध्यम से पहले उपलब्ध कराया गया ट्रांसक्रिप्ट `messageId` पास करें; सहेजी गई प्रविष्टि अब भी उपलब्ध और अत्यधिक बड़ी न होने पर Gateway हल्की हिस्ट्री काट-छाँट सीमा के बिना वही प्रदर्शन-सामान्यीकृत प्रक्षेपण लौटाता है।
    - `chat.toolTitles` Control UI में रेंडर किए गए टूल कॉल के लिए छोटे उद्देश्य शीर्षक लौटाता है (बैच में, सीमित इनपुट के साथ अधिकतम 24 आइटम)। यह सुविधा `gateway.controlUi.toolTitles` के माध्यम से ऑप्ट-इन है (डिफ़ॉल्ट रूप से बंद); अक्षम Gateway बिना मॉडल कॉल के `{ titles: {}, disabled: true }` प्रत्युत्तर देते हैं, ताकि क्लाइंट पूछना बंद कर दें। सक्षम होने पर शीर्षक मानक यूटिलिटी-मॉडल रूटिंग का उपयोग करते हैं: स्पष्ट रूप से कॉन्फ़िगर किया गया `utilityModel` (एक ऑपरेटर निर्णय, जो सभी यूटिलिटी टास्क की तरह सीमित टास्क सामग्री चुने गए प्रदाता को भेज सकता है), अन्यथा सेशन प्रदाता का घोषित छोटे-मॉडल का डिफ़ॉल्ट, ताकि कोई नया बहिर्गमन गंतव्य अप्रत्यक्ष रूप से न जुड़े; खाली `utilityModel` उन्हें पूरी तरह अक्षम कर देता है। शीर्षक कभी भी प्राथमिक मॉडल पर वापस नहीं जाते। परिणाम टूल नाम + इनपुट के आधार पर प्रति-एजेंट स्थिति डेटाबेस में कैश होते हैं, इसलिए बार-बार देखने पर समान कॉल के लिए दोबारा शुल्क नहीं लगता।
    - `chat.send` स्वतः कटऑफ़ से पहले शुरू की गई मॉडल कॉल के लिए फ़ास्ट मोड का उपयोग करने हेतु एक-टर्न `fastMode: "auto"` स्वीकार करता है, फिर बाद की पुनः प्रयास, फ़ॉलबैक, टूल-परिणाम, या निरंतरता कॉल फ़ास्ट मोड के बिना शुरू करता है। कटऑफ़ डिफ़ॉल्ट रूप से 60 सेकंड (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) है और इसे `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` से प्रति मॉडल कॉन्फ़िगर किया जा सकता है। कोई `chat.send` कॉलर उस अनुरोध के लिए कटऑफ़ ओवरराइड करने हेतु एक-टर्न `fastAutoOnSeconds` पास कर सकता है। केवल इस अनुरोध के लिए सहेजे गए कतार मोड को ओवरराइड करने हेतु `queueMode` (`steer`, `followup`, `collect`, या `interrupt`) पास करें; स्पष्ट Control UI दिशा-परिवर्तन कार्रवाइयाँ `queueMode: "steer"` का उपयोग करती हैं।

## ऑडिट लेजर RPC

`audit.activity.list` ऑपरेटर क्लाइंट को एजेंट रन, टूल कार्रवाई, और ऑप्ट-इन संदेश
जीवनचक्र मेटाडेटा का स्थिर, नवीनतम-प्रथम दृश्य देता है। इसके लिए
`operator.read` आवश्यक है। क्वेरी 30 दिनों से पुराने रिकॉर्ड बाहर रखती हैं, और साझा
SQLite लेजर 100,000 रिकॉर्ड तक सीमित है। समय-सीमा समाप्त पंक्तियाँ
Gateway स्टार्टअप, प्रति घंटे रखरखाव, और बाद के लेखन के दौरान हटाई जाती हैं। डेटा मॉडल और गोपनीयता अर्थविज्ञान के लिए
[ऑडिट इतिहास](/hi/gateway/audit) देखें।

- पैरामीटर: वैकल्पिक सटीक `agentId`, `sessionKey`, या `runId`; वैकल्पिक `kind`
  (`"agent_run"`, `"tool_action"`, या `"message"`); वैकल्पिक `status`
  (`"started"`, `"succeeded"`, `"failed"`, `"cancelled"`, `"timed_out"`,
  `"blocked"`, या `"unknown"`); वैकल्पिक संदेश `direction` (`"inbound"` या
  `"outbound"`) और सटीक `channel`; वैकल्पिक समावेशी `after` / `before`
  Unix-मिलीसेकंड सीमाएँ; `1` से `500` तक वैकल्पिक `limit`; और पिछले पृष्ठ से वैकल्पिक
  स्ट्रिंग `cursor`।
- परिणाम: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`।

नामित V1 परिणाम यूनियन में एजेंट-रन, टूल-कार्रवाई, इनबाउंड-संदेश,
और आउटबाउंड-संदेश की अलग-अलग स्कीमा हैं। `eventType` विभेदक क्रमशः
`agent_run`, `tool_action`, `inbound_message`, या `outbound_message` है; `kind` और
संदेश `direction` फ़िल्टरिंग और प्रदर्शन के लिए उपलब्ध रहते हैं। प्रत्येक इवेंट में
पूर्णांक `schemaVersion: 1` होता है। संदेश पहचान संदर्भ सटीक
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` प्रारूप का उपयोग करते हैं; चैनल-प्रेषक अभिनेता
ID भी इसी प्रारूप का उपयोग करता है।

सभी वेरिएंट में `eventType`, `schemaVersion`, `eventId`, `sequence`,
`sourceSequence`, `occurredAt`, `kind`, `action`, `status`, `actor`, और
`redaction` आवश्यक हैं। वेरिएंट फ़ील्ड हैं:

| `eventType`        | आवश्यक फ़ील्ड                                                   | वैकल्पिक फ़ील्ड                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`, `runId`; `kind: "agent_run"`                           | `sessionKey`, `sessionId`, `errorCode`                                                                                          |
| `tool_action`      | `agentId`, `runId`; `kind: "tool_action"`                         | `sessionKey`, `sessionId`, `toolCallId`, `toolName`, `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`, `channel`, `conversationKind`, `outcome`  | `agentId`, `runId`, `durationMs`, `resultCount`, पहचान संदर्भ, `reasonCode`, `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`, `channel`, `conversationKind`, `outcome` | `agentId`, `runId`, `durationMs`, `resultCount`, पहचान संदर्भ, `reasonCode`, `deliveryKind`, `failureStage`, `errorCode` |

बंद संदेश एनम हैं:

- `conversationKind`: `direct`, `group`, `channel`, या `unknown`।
- इनबाउंड `outcome`: `completed`, `skipped`, या `failed`; वैकल्पिक
  `reasonCode`: `duplicate`, `reply_operation_active`,
  `reply_operation_aborted`, `fast_abort`, `plugin_bound_handled`,
  `plugin_bound_unavailable`, `plugin_bound_declined`, `plugin_bound_error`,
  `before_dispatch_handled`, `acp_dispatch_completed`, `acp_dispatch_failed`,
  `acp_dispatch_empty`, या `acp_dispatch_aborted`।
- आउटबाउंड `outcome`: `sent`, `suppressed`, `failed`, या `unknown`; वैकल्पिक
  `reasonCode`: `cancelled_by_message_sending_hook`,
  `cancelled_by_reply_payload_sending_hook`,
  `empty_after_message_sending_hook`, `empty_after_reply_payload_sending_hook`,
  या `no_visible_payload`। कोई एडाप्टर जो प्लेटफ़ॉर्म पहचान नहीं लौटाता,
  `unknown` है, क्योंकि बाहरी दुष्प्रभाव को असिद्ध नहीं किया जा सकता।
- `deliveryKind`: `text`, `media`, या `other`; `failureStage`:
  `platform_send`, `queue`, या `unknown`।

टर्मिनल फ़ील्ड सहसंबद्ध हैं, स्वतंत्र रूप से वैकल्पिक नहीं:

| वेरिएंट          | टर्मिनल मैपिंग                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| एजेंट रन        | `started` में `errorCode` नहीं होता; प्रत्येक गैर-सफल समाप्त स्थिति के लिए उसका संगत `run_*` कोड आवश्यक है।                                                                 |
| टूल कार्रवाई      | `started` और सफल स्थिति में `errorCode` नहीं होता; प्रत्येक अन्य समाप्त स्थिति के लिए उसका संगत `tool_*` कोड आवश्यक है।                                                       |
| इनबाउंड संदेश  | सफल = `completed`; अवरुद्ध = `skipped`; विफल = `failed` और `message_processing_failed`। `reasonCode`, मौजूद होने पर, उसी टर्मिनल समूह से संबंधित होना चाहिए। |
| आउटबाउंड संदेश | सफल = `sent`; अवरुद्ध = `suppressed` और `reasonCode`; विफल = `failed`, `errorCode`, और `failureStage`; अज्ञात = `unknown` और `failureStage`।      |

प्रत्येक गतिविधि इवेंट में एक स्थिर इवेंट आईडी, एकदिश रूप से बढ़ता लेजर अनुक्रम,
स्रोत इवेंट अनुक्रम, टाइमस्टैम्प, कर्ता, कार्रवाई, स्थिति, पूर्णांक
`schemaVersion: 1`, और `redaction: "metadata_only"` शामिल होते हैं। रन और टूल रिकॉर्ड के लिए
एजेंट और रन की उत्पत्ति आवश्यक है तथा इनमें सत्र की उत्पत्ति शामिल हो सकती है। संदेश
रिकॉर्ड में एजेंट और रन आईडी शामिल हो सकते हैं, लेकिन जानबूझकर कभी भी
`sessionKey` या `sessionId` शामिल नहीं होते; इसलिए `sessionKey` क्वेरी फ़िल्टर केवल
रन और टूल पंक्तियों पर लागू होता है। टूल इवेंट में टूल कॉल आईडी और टूल का नाम शामिल हो सकता है।

संदेश रिकॉर्ड `message.inbound.processed` या
`message.outbound.finished` का उपयोग करते हैं और दिशा, चैनल, वार्तालाप प्रकार,
सामान्यीकृत परिणाम, तथा वैकल्पिक डिलीवरी प्रकार, विफलता चरण, अवधि,
परिणाम संख्या, कारण कोड, और इंस्टॉलेशन-स्थानीय कुंजीबद्ध
खाता/वार्तालाप/संदेश/लक्ष्य छद्मनाम जोड़ते हैं। ये छद्मनाम
सहसंबंध स्थापित करने में सहायता करते हैं, लेकिन ये अनामीकरण नहीं हैं: स्टेट डेटाबेस में उनकी कुंजी होती है,
जबकि RPC और CLI निर्यातों में नहीं होती। लेजर प्रॉम्प्ट, संदेश
सामग्री, टूल आर्ग्युमेंट, टूल परिणाम, कमांड आउटपुट या अपरिष्कृत त्रुटि टेक्स्ट संग्रहीत नहीं करता।
रन/टूल के `sessionKey` मान अपरिष्कृत सहसंबंध मेटाडेटा बने रहते हैं और उनमें
प्लेटफ़ॉर्म खाता या पीयर आईडी अंतर्निहित हो सकती हैं; संदेश रिकॉर्ड सत्र कुंजियाँ छोड़ देते हैं।

इनबाउंड पंक्तियों के लिए, `durationMs` कोर डिस्पैच से उसके अंतिम बिंदु तक का मापन करता है और
`resultCount` अंतिम रूप दिए गए कतारबद्ध टूल, ब्लॉक और उत्तर पेलोड की संख्या गिनता है। आउटबाउंड
पंक्तियों के लिए, `durationMs` डिलीवरी स्वामित्व से पावती,
डेड लेटर या समन्वयन तक की अवधि समेटता है (जिसमें कतारबद्ध प्रतीक्षा समय शामिल है), और `resultCount`
पहचाने गए वास्तविक प्लेटफ़ॉर्म प्रेषणों की संख्या गिनता है। `deliveryKind`, मौजूद होने पर,
हुक और रेंडरिंग के बाद प्रभावी पेलोड का वर्णन करता है; दबाई गई या
क्रैश-संबंधी अस्पष्ट पंक्तियाँ इसे छोड़ देती हैं।

वर्तमान संदेश कवरेज में वे स्वीकृत इनबाउंड संदेश शामिल हैं जो कोर
डिस्पैच तक पहुँचते हैं, जिनमें कोर के डुप्लिकेट/अंतिम परिणाम भी शामिल हैं। आउटबाउंड कवरेज
साझा टिकाऊ डिलीवरी तक पहुँचने वाले प्रत्येक मूल तार्किक उत्तर पेलोड के लिए
एक अंतिम पंक्ति लिखता है; चंकिंग और एडाप्टर फ़ैन-आउट को `resultCount` में एकत्रित किया जाता है। कतारबद्ध
पुनःप्रयास-योग्य या अस्पष्ट प्रेषण केवल पावती, डेड
लेटर या समन्वयन के बाद रिकॉर्ड किए जाते हैं। इन साझा सीमाओं को बायपास करने वाले
Plugin-स्थानीय और प्रत्यक्ष-प्रेषण पथ अभी कवर नहीं किए गए हैं। सीमित वर्कर कतार सर्वोत्तम-प्रयास
आधारित है और विफलता या संतृप्ति पर रिकॉर्ड छोड़ सकती है, इसलिए यह सतह
हानिरहित पूर्ण अनुपालन अभिलेखागार नहीं है।

रिकॉर्डिंग डिफ़ॉल्ट रूप से चालू रहती है और
[`audit.enabled`](/hi/gateway/configuration-reference#audit) द्वारा नियंत्रित होती है। संदेश रिकॉर्डिंग
अलग से `audit.messages` द्वारा नियंत्रित होती है और इसका डिफ़ॉल्ट `"off"` है। जब
रिकॉर्डिंग अक्षम होती है, तो `audit.activity.list` पहले लिखे गए रिकॉर्ड की
समाप्ति तक उन्हें उपलब्ध कराता रहता है।

शिप किए गए `audit.list` अनुरोध, परिणाम और `AuditEvent` स्कीमा
अपरिवर्तित रहते हैं और केवल एजेंट-रन तथा टूल-कार्रवाई रिकॉर्ड लौटाते हैं। नए ऑपरेटर
क्लाइंट को Gateway द्वारा इसका विज्ञापन किए जाने पर `audit.activity.list` कॉल करना चाहिए। पुराने
Gateway या तो `unknown method: audit.activity.list` रिपोर्ट कर सकते हैं या, क्योंकि
शिप किए गए संस्करणों में विधि खोज से पहले प्राधिकरण होता था, केवल-पठन स्कोप वाले अनुरोध को `missing scope:
operator.admin` रिपोर्ट कर सकते हैं। बाद वाले को विधि की अनुपस्थिति
केवल तभी मानें जब विधि का विज्ञापन नहीं किया गया था। इसके बाद क्लाइंट `audit.list` का पुनःप्रयास
केवल तभी कर सकता है जब उसके फ़िल्टर को संदेश प्रकार, दिशा या चैनल
समर्थन की आवश्यकता न हो।

टेक्स्ट क्वेरी और सीमित JSON निर्यातों के लिए [`openclaw audit`](/hi/cli/audit) का उपयोग करें।

## टास्क लेजर RPC

ऑपरेटर क्लाइंट टास्क लेजर RPC (`packages/gateway-protocol/src/schema/tasks.ts`) के माध्यम से
Gateway के पृष्ठभूमि टास्क रिकॉर्ड का निरीक्षण और उन्हें रद्द करते हैं। ये
स्वच्छ किए गए टास्क सारांश लौटाते हैं, अपरिष्कृत रनटाइम स्थिति नहीं।

- `tasks.list` के लिए `operator.read` आवश्यक है।
  - पैरामीटर: वैकल्पिक `status` (`"queued"`, `"running"`, `"completed"`,
    `"failed"`, `"cancelled"`, या `"timed_out"`) या उन स्थितियों की एक सरणी,
    वैकल्पिक `agentId`, वैकल्पिक `sessionKey`, वैकल्पिक `limit` जो `1` से
    `500` तक हो, और वैकल्पिक स्ट्रिंग `cursor`।
  - परिणाम: `{ "tasks": TaskSummary[], "nextCursor"?: string }`।
- `tasks.get` के लिए `operator.read` आवश्यक है।
  - पैरामीटर: `{ "taskId": string }`।
  - परिणाम: `{ "task": TaskSummary }`।
  - अनुपलब्ध टास्क आईडी Gateway की नॉट-फ़ाउंड त्रुटि संरचना लौटाती हैं।
- `tasks.cancel` के लिए `operator.write` आवश्यक है।
  - पैरामीटर: `{ "taskId": string, "reason"?: string }`।
  - परिणाम: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`।
  - `found` बताता है कि लेजर में कोई मेल खाता टास्क था या नहीं। `cancelled`
    बताता है कि रनटाइम ने रद्दीकरण स्वीकार या रिकॉर्ड किया था या नहीं।

`TaskSummary` में `id`, `status`, और वैकल्पिक मेटाडेटा शामिल हैं: `kind`,
`runtime`, `title`, `agentId`, `sessionKey`, `childSessionKey`, `ownerKey`,
`runId`, `taskId`, `flowId`, `parentTaskId`, `sourceId`, टाइमस्टैम्प, प्रगति,
अंतिम सारांश और स्वच्छ किया गया त्रुटि टेक्स्ट। `agentId` टास्क निष्पादित करने वाले
एजेंट की पहचान करता है; `sessionKey` और `ownerKey` अनुरोधकर्ता तथा नियंत्रण
संदर्भ बनाए रखते हैं।

## ऑपरेटर सहायक विधियाँ

- `commands.list` (`operator.read`) किसी एजेंट के लिए रनटाइम कमांड सूची प्राप्त करता है।
  - `agentId` वैकल्पिक है; डिफ़ॉल्ट एजेंट कार्यस्थान पढ़ने के लिए इसे छोड़ दें।
  - `scope` नियंत्रित करता है कि प्राथमिक `name` किस सतह को लक्षित करता है: `text`
    अग्रणी `/` के बिना प्राथमिक टेक्स्ट कमांड टोकन लौटाता है; `native` और
    डिफ़ॉल्ट `both` पथ उपलब्ध होने पर प्रदाता-जागरूक नेटिव नाम लौटाते हैं।
  - `textAliases` में `/model` और `/m` जैसे सटीक स्लैश उपनाम होते हैं।
  - `nativeName` में प्रदाता-जागरूक नेटिव कमांड नाम होता है, यदि वह
    मौजूद हो।
  - `provider` वैकल्पिक है और केवल नेटिव नामकरण तथा नेटिव Plugin
    कमांड उपलब्धता को प्रभावित करता है।
  - `includeArgs=false` प्रतिक्रिया से क्रमबद्ध आर्ग्युमेंट मेटाडेटा छोड़ देता है।
- `tools.catalog` (`operator.read`) किसी
  एजेंट के लिए रनटाइम टूल कैटलॉग प्राप्त करता है। प्रतिक्रिया में समूहीकृत टूल और उत्पत्ति मेटाडेटा शामिल होते हैं:
  - `source`: `core` या `plugin`
  - `pluginId`: `source="plugin"` होने पर Plugin स्वामी
  - `optional`: कोई Plugin टूल वैकल्पिक है या नहीं
- `tools.effective` (`operator.read`) किसी सत्र के लिए रनटाइम-प्रभावी टूल
  सूची प्राप्त करता है।
  - `sessionKey` आवश्यक है।
  - Gateway कॉलर द्वारा दिए गए प्रमाणीकरण या डिलीवरी संदर्भ को स्वीकार करने के बजाय
    सर्वर-साइड सत्र से विश्वसनीय रनटाइम संदर्भ प्राप्त करता है।
  - प्रतिक्रिया सक्रिय सूची का सत्र-स्कोप वाला, सर्वर-व्युत्पन्न प्रक्षेपण है,
    जिसमें कोर, Plugin, चैनल और पहले से खोजे गए MCP
    सर्वर टूल शामिल हैं।
  - `tools.effective` MCP के लिए केवल-पठन है: यह अंतिम टूल नीति के माध्यम से किसी वार्म सत्र के MCP
    कैटलॉग को प्रक्षेपित कर सकता है, लेकिन MCP रनटाइम नहीं बनाता,
    ट्रांसपोर्ट से कनेक्ट नहीं करता या `tools/list` जारी नहीं करता। यदि कोई मेल खाता वार्म कैटलॉग
    मौजूद नहीं है, तो प्रतिक्रिया में `mcp-not-yet-connected`,
    `mcp-not-yet-listed`, या `mcp-stale-catalog` जैसी सूचना शामिल हो सकती है।
  - प्रभावी टूल प्रविष्टियाँ `source="core"`, `source="plugin"`,
    `source="channel"`, या `source="mcp"` का उपयोग करती हैं।
- `tools.invoke` (`operator.write`) `/tools/invoke` के समान
  Gateway नीति पथ के माध्यम से एक उपलब्ध टूल का आह्वान करता है।
  - `name` आवश्यक है। `args`, `sessionKey`, `agentId`, `confirm`, और
    `idempotencyKey` वैकल्पिक हैं।
  - यदि `sessionKey` और `agentId` दोनों मौजूद हैं, तो समाधान किया गया सत्र एजेंट
    `agentId` से मेल खाना चाहिए।
  - केवल-स्वामी कोर रैपर, जैसे `cron`, `gateway`, और `nodes`, के लिए
    स्वामी/व्यवस्थापक पहचान (`operator.admin`) आवश्यक है, भले ही `tools.invoke` स्वयं
    `operator.write` हो।
  - प्रतिक्रिया SDK-उन्मुख आवरण है, जिसमें `ok`, `toolName`, वैकल्पिक
    `output`, और टाइप किए गए `error` फ़ील्ड होते हैं। अनुमोदन या नीति अस्वीकृतियाँ
    Gateway टूल नीति पाइपलाइन को बायपास करने के बजाय पेलोड में
    `ok:false` लौटाती हैं।
- `skills.status` (`operator.read`) किसी
  एजेंट के लिए दृश्यमान Skills सूची प्राप्त करता है।
  - `agentId` वैकल्पिक है; डिफ़ॉल्ट एजेंट कार्यस्थान पढ़ने के लिए इसे छोड़ दें।
  - प्रतिक्रिया में पात्रता, अनुपलब्ध आवश्यकताएँ, कॉन्फ़िगरेशन जाँच,
    और अपरिष्कृत सीक्रेट मान उजागर किए बिना स्वच्छ किए गए इंस्टॉल विकल्प शामिल होते हैं।
- `skills.search` और `skills.detail` (`operator.read`) ClawHub
  खोज मेटाडेटा लौटाते हैं।
- `skills.upload.begin`, `skills.upload.chunk`, और `skills.upload.commit`
  (`operator.admin`) किसी निजी Skill संग्रह को इंस्टॉल करने से पहले स्टेज करते हैं। यह
  विश्वसनीय क्लाइंट के लिए एक अलग व्यवस्थापक अपलोड पथ है, सामान्य ClawHub
  Skill इंस्टॉल प्रवाह नहीं, और डिफ़ॉल्ट रूप से अक्षम रहता है जब तक
  `skills.install.allowUploadedArchives` सक्षम न हो।
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })`
    उस स्लग और फ़ोर्स मान से बंधा अपलोड बनाता है।
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })`
    सटीक डीकोड किए गए ऑफ़सेट पर बाइट जोड़ता है।
  - `skills.upload.commit({ uploadId, sha256? })` अंतिम आकार और
    SHA-256 सत्यापित करता है। कमिट केवल अपलोड को अंतिम रूप देता है; यह Skill इंस्टॉल नहीं करता।
  - अपलोड किए गए Skill संग्रह ऐसे zip संग्रह होते हैं जिनमें `SKILL.md` रूट होता है। संग्रह का
    आंतरिक डायरेक्टरी नाम कभी भी इंस्टॉल लक्ष्य का चयन नहीं करता।
- `skills.install` (`operator.admin`) के तीन मोड हैं:
  - ClawHub मोड: `{ source: "clawhub", slug, version?, force? }` डिफ़ॉल्ट एजेंट कार्यस्थान की `skills/`
    डायरेक्टरी में एक Skill फ़ोल्डर इंस्टॉल करता है।
  - अपलोड मोड: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }`
    कमिट किए गए अपलोड को डिफ़ॉल्ट एजेंट कार्यस्थान की
    `skills/<slug>` डायरेक्टरी में इंस्टॉल करता है। स्लग और फ़ोर्स मान मूल
    `skills.upload.begin` अनुरोध से मेल खाने चाहिए। जब तक
    `skills.install.allowUploadedArchives` सक्षम न हो, इसे अस्वीकार कर दिया जाता है; यह सेटिंग
    ClawHub इंस्टॉल को प्रभावित नहीं करती।
  - Gateway इंस्टॉलर मोड: `{ name, installId, timeoutMs? }` Gateway होस्ट पर घोषित
    `metadata.openclaw.install` कार्रवाई चलाता है। पुराने क्लाइंट अभी भी
    `dangerouslyForceUnsafeInstall` भेज सकते हैं; यह फ़ील्ड अप्रचलित है,
    केवल प्रोटोकॉल संगतता के लिए स्वीकार किया जाता है और अनदेखा कर दिया जाता है। ऑपरेटर-स्वामित्व वाले
    इंस्टॉल निर्णयों के लिए `security.installPolicy` का उपयोग करें।
- `skills.update` (`operator.admin`) के दो मोड हैं:
  - ClawHub मोड डिफ़ॉल्ट एजेंट कार्यस्थान में एक ट्रैक किए गए स्लग या सभी ट्रैक किए गए ClawHub इंस्टॉल को
    अपडेट करता है।
  - कॉन्फ़िगरेशन मोड `skills.entries.<skillKey>` मानों, जैसे `enabled`,
    `apiKey`, और `env`, को पैच करता है।

### `models.list` दृश्य

`models.list` एक वैकल्पिक `view` पैरामीटर
(`src/agents/model-catalog-visibility.ts`) स्वीकार करता है:

- हटाया गया या `"default"`: यदि `agents.defaults.models` कॉन्फ़िगर किया गया है, तो
  प्रतिक्रिया अनुमत कैटलॉग होती है, जिसमें `provider/*` प्रविष्टियों के लिए
  गतिशील रूप से खोजे गए मॉडल शामिल होते हैं। अन्यथा प्रतिक्रिया संपूर्ण Gateway
  कैटलॉग होती है।
- `"configured"`: पिकर के अनुरूप व्यवहार। यदि `agents.defaults.models`
  कॉन्फ़िगर किया गया है, तो उसे ही प्राथमिकता मिलती है, जिसमें
  `provider/*` प्रविष्टियों के लिए प्रदाता-स्कोप वाली खोज शामिल है। अनुमत-सूची के बिना, प्रतिक्रिया स्पष्ट
  `models.providers.<provider>.models` प्रविष्टियों का उपयोग करती है और केवल तभी संपूर्ण
  कैटलॉग पर वापस जाती है, जब कोई कॉन्फ़िगर की गई मॉडल पंक्ति मौजूद न हो।
- `"provider-config"`: स्रोत द्वारा बनाई गई `models.providers.*.models` सूची,
  जो पिकर की अनुमत-सूचियों से स्वतंत्र है। पंक्तियों में सार्वजनिक मॉडल क्षमताएँ और
  रूट-जागरूक उपलब्धता शामिल होती हैं, लेकिन प्रदाता एंडपॉइंट, प्रमाणीकरण सामग्री और
  रनटाइम अनुरोध कॉन्फ़िगरेशन शामिल नहीं होते।
- `"all"`: संपूर्ण Gateway कैटलॉग, जो `agents.defaults.models` को बायपास करता है। इसका उपयोग
  निदान/खोज UI के लिए करें, सामान्य मॉडल पिकर के लिए नहीं।

## निष्पादन अनुमोदन

- जब किसी निष्पादन अनुरोध को अनुमोदन की आवश्यकता होती है, तो Gateway
  `exec.approval.requested` प्रसारित करता है।
- ऑपरेटर क्लाइंट `exec.approval.resolve` को कॉल करके समाधान करते हैं (इसके लिए
  `operator.approvals` आवश्यक है)।
- `host=node` के लिए, `exec.approval.request` में `systemRunPlan`
  (कैनोनिकल `argv`/`cwd`/`rawCommand`/सत्र मेटाडेटा) शामिल होना चाहिए। `systemRunPlan` के बिना अनुरोध
  अस्वीकार कर दिए जाते हैं।
- अनुमोदन के बाद, अग्रेषित `node.invoke system.run` कॉल उसी
  कैनोनिकल `systemRunPlan` को आधिकारिक कमांड/cwd/सत्र संदर्भ के रूप में पुनः उपयोग करती हैं।
- यदि कोई कॉलर तैयारी और अंतिम अनुमोदित `system.run` अग्रेषण के बीच
  `command`, `rawCommand`, `cwd`, `agentId`, या
  `sessionKey` को बदलता है, तो Gateway बदले हुए पेलोड पर विश्वास करने के बजाय रन को अस्वीकार कर देता है।

## एजेंट डिलीवरी फ़ॉलबैक

- `agent` अनुरोध आउटबाउंड डिलीवरी का अनुरोध करने के लिए `deliver=true` शामिल कर सकते हैं।
- `bestEffortDeliver=false` (डिफ़ॉल्ट) सख्त व्यवहार बनाए रखता है: अनसुलझे या
  केवल-आंतरिक डिलीवरी लक्ष्य `INVALID_REQUEST` लौटाते हैं।
- जब किसी बाहरी डिलीवरी-योग्य रूट का समाधान नहीं किया जा सकता (उदाहरण के लिए आंतरिक/वेबचैट
  सत्र या अस्पष्ट मल्टी-चैनल कॉन्फ़िगरेशन), तब `bestEffortDeliver=true` केवल-सत्र निष्पादन पर फ़ॉलबैक की अनुमति देता है।
- जब डिलीवरी का अनुरोध किया गया हो, तो अंतिम `agent` परिणामों में `result.deliveryStatus` शामिल हो सकता है,
  जिसमें [`openclaw agent --json --deliver`](/hi/cli/agent#json-delivery-status) के लिए दस्तावेज़ित समान
  `sent`, `suppressed`, `partial_failed`, और
  `failed` स्थितियों का उपयोग किया जाता है।

## संस्करण प्रबंधन

- `PROTOCOL_VERSION`, `MIN_CLIENT_PROTOCOL_VERSION`,
  `MIN_NODE_PROTOCOL_VERSION`, और `MIN_PROBE_PROTOCOL_VERSION`
  `packages/gateway-protocol/src/version.ts` में स्थित हैं।
- क्लाइंट `minProtocol` + `maxProtocol` भेजते हैं। ऑपरेटर और UI क्लाइंट को
  उस सीमा में वर्तमान प्रोटोकॉल शामिल करना आवश्यक है; वर्तमान क्लाइंट और सर्वर
  प्रोटोकॉल v4 चलाते हैं।
- प्रमाणित क्लाइंट, जिनमें `role: "node"` और `client.mode: "node"` दोनों हैं,
  N-1 Node प्रोटोकॉल (वर्तमान में v3) का उपयोग कर सकते हैं। हल्के पुनरारंभ प्रोब
  इसी N-1 विंडो का उपयोग करते हैं। इस संगतता विंडो से डिवाइस प्रमाणीकरण, पेयरिंग,
  स्कोप, कमांड नीति और निष्पादन अनुमोदन अपरिवर्तित रहते हैं। Plugin के स्वामित्व वाली Node
  क्षमताएँ और कमांड तब तक उपलब्ध नहीं कराए जाते, जब तक Node वर्तमान
  प्रोटोकॉल में अपग्रेड नहीं हो जाता, क्योंकि उनकी होस्ट की गई सतहें N-1 अनुबंध का हिस्सा नहीं हैं।
- स्कीमा और मॉडल TypeBox परिभाषाओं से जनरेट किए जाते हैं:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

### क्लाइंट स्थिरांक

संदर्भ क्लाइंट कार्यान्वयन `packages/gateway-client/src/` में स्थित है
(OpenClaw इसे पतले `src/gateway/client.ts` फ़साड के माध्यम से आवृत करता है)। ये
डिफ़ॉल्ट प्रोटोकॉल v4 में स्थिर हैं और तृतीय-पक्ष क्लाइंट के लिए अपेक्षित आधार-रेखा हैं।

| स्थिरांक                                  | डिफ़ॉल्ट                                               | स्रोत                                                                                                                    |
| ----------------------------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `PROTOCOL_VERSION`                        | `4`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| `MIN_CLIENT_PROTOCOL_VERSION`             | `4`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| `MIN_NODE_PROTOCOL_VERSION`               | `3`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| `MIN_PROBE_PROTOCOL_VERSION`              | `3`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| अनुरोध टाइमआउट (प्रति RPC)                 | `30_000` ms                                           | `packages/gateway-client/src/client.ts` (`requestTimeoutMs`)                                                              |
| पूर्व-प्रमाणीकरण / कनेक्ट-चैलेंज टाइमआउट       | `15_000` ms                                           | `packages/gateway-client/src/timeouts.ts` (`OPENCLAW_HANDSHAKE_TIMEOUT_MS` env युग्मित सर्वर/क्लाइंट बजट बढ़ा सकता है) |
| आरंभिक पुनः कनेक्ट बैकऑफ़                 | `1_000` ms                                            | `packages/gateway-client/src/client.ts` (`GATEWAY_RECONNECT_POLICY`)                                                      |
| अधिकतम पुनः कनेक्ट बैकऑफ़                     | `30_000` ms                                           | `packages/gateway-client/src/client.ts` (`GATEWAY_RECONNECT_POLICY`)                                                      |
| डिवाइस-टोकन बंद होने के बाद तेज़-पुनःप्रयास सीमा | `250` ms                                              | `packages/gateway-client/src/client.ts`                                                                                   |
| `terminate()` से पहले बलपूर्वक रोकने की अनुग्रह अवधि     | `250` ms                                              | `FORCE_STOP_TERMINATE_GRACE_MS`                                                                                           |
| `stopAndWait()` डिफ़ॉल्ट टाइमआउट           | `1_000` ms                                            | `STOP_AND_WAIT_TIMEOUT_MS`                                                                                                |
| डिफ़ॉल्ट टिक अंतराल (`hello-ok` से पहले)    | `30_000` ms                                           | `packages/gateway-client/src/client.ts`                                                                                   |
| टिक-टाइमआउट पर बंद होना                        | मौन अवधि `tickIntervalMs * 2` से अधिक होने पर कोड `4000` | `packages/gateway-client/src/client.ts`                                                                                   |
| `MAX_PAYLOAD_BYTES`                       | `25 * 1024 * 1024` (25 MB)                            | `src/gateway/server-constants.ts`                                                                                         |

सर्वर प्रभावी `policy.tickIntervalMs`,
`policy.maxPayload`, और `policy.maxBufferedBytes` को `hello-ok` में घोषित करता है; क्लाइंट को
पूर्व-हैंडशेक डिफ़ॉल्ट के बजाय उन मानों का पालन करना चाहिए।

जब प्रत्येक लंबित अनुरोध की एक समय-सीमा होती है, तब संदर्भ क्लाइंट सीमित अनुरोधों को उनकी कॉन्फ़िगर की गई समय-सीमा नियंत्रित करने देता है।
सीमित `timeoutMs` के बिना कोई `expectFinal` अनुरोध,
`timeoutMs: null` वाला कोई भी अनुरोध, या सीमित और
असीमित अनुरोधों का मिश्रण टिक वॉचडॉग को सक्रिय रखता है। यदि इनबाउंड इवेंट और
प्रतिक्रियाएँ टिक-टाइमआउट सीमा से आगे तक मौन रहती हैं, तो क्लाइंट
सॉकेट को कोड `4000` के साथ बंद करता है, प्रत्येक लंबित अनुरोध को अस्वीकार करता है और पुनः कनेक्ट करता है। पुनः कनेक्ट होने के बाद वह
अस्वीकृत अनुरोधों को दोबारा नहीं चलाता।

## प्रमाणीकरण

- साझा-सीक्रेट Gateway प्रमाणीकरण कॉन्फ़िगर किए गए
  `gateway.auth.mode` (`"none" | "token" | "password" | "trusted-proxy"`) के आधार पर `connect.params.auth.token` या
  `connect.params.auth.password` का उपयोग करता है।
- Tailscale Serve (`gateway.auth.allowTailscale: true`) या गैर-लूपबैक
  `gateway.auth.mode: "trusted-proxy"` जैसे पहचान-धारक मोड, `connect.params.auth.*` के बजाय अनुरोध
  हेडर से कनेक्ट प्रमाणीकरण जाँच पूरी करते हैं।
- निजी-इनग्रेस `gateway.auth.mode: "none"` साझा-सीक्रेट कनेक्ट प्रमाणीकरण को
  पूरी तरह छोड़ देता है; इस मोड को सार्वजनिक/अविश्वसनीय इनग्रेस पर उजागर न करें।
- पेयरिंग के बाद, Gateway कनेक्शन भूमिका + कार्यक्षेत्रों तक सीमित एक डिवाइस
  टोकन जारी करता है, जो `hello-ok.auth.deviceToken` में लौटाया जाता है। किसी भी सफल
  कनेक्ट के बाद क्लाइंट को इसे सहेजना चाहिए।
- उस सहेजे गए डिवाइस टोकन से दोबारा कनेक्ट करते समय उस टोकन के लिए सहेजे गए
  स्वीकृत कार्यक्षेत्र सेट का भी पुनः उपयोग किया जाना चाहिए। यह पहले से दी गई
  पठन/जाँच/स्थिति पहुँच बनाए रखता है और पुनः कनेक्शन को चुपचाप अधिक संकीर्ण
  अंतर्निहित केवल-व्यवस्थापक कार्यक्षेत्र में सीमित होने से बचाता है।
- क्लाइंट-साइड कनेक्ट प्रमाणीकरण संयोजन (`packages/gateway-client/src/client.ts` में
  `selectConnectAuth`):
  - `auth.password` स्वतंत्र है और सेट होने पर हमेशा अग्रेषित किया जाता है।
  - `auth.token` को प्राथमिकता क्रम में भरा जाता है: पहले स्पष्ट साझा टोकन,
    फिर स्पष्ट `deviceToken`, फिर सहेजा गया प्रति-डिवाइस टोकन
    (`deviceId` + `role` द्वारा कुंजीबद्ध)।
  - `auth.bootstrapToken` केवल तभी भेजा जाता है जब उपर्युक्त में से किसी ने
    `auth.token` को निर्धारित न किया हो। साझा टोकन या कोई भी निर्धारित
    डिवाइस टोकन इसे रोक देता है।
  - एक-बार वाले `AUTH_TOKEN_MISMATCH` पुनः प्रयास पर सहेजे गए डिवाइस टोकन का
    स्वतः-प्रोमोशन केवल विश्वसनीय एंडपॉइंट तक सीमित है: लूपबैक, या पिन किए गए
    `tlsFingerprint` वाला `wss://`। पिनिंग के बिना सार्वजनिक
    `wss://` पात्र नहीं है।
- अंतर्निहित सेटअप-कोड बूटस्ट्रैप विश्वसनीय मोबाइल हस्तांतरण के लिए प्राथमिक
  Node `hello-ok.auth.deviceToken` और `hello-ok.auth.deviceTokens` में एक सीमित ऑपरेटर टोकन
  लौटाता है। ऑपरेटर टोकन में नेटिव Talk कॉन्फ़िगरेशन पठन के लिए
  `operator.talk.secrets` शामिल है, लेकिन पेयरिंग-परिवर्तन कार्यक्षेत्र और
  `operator.admin` शामिल नहीं हैं।
- जब कोई गैर-बेसलाइन सेटअप-कोड बूटस्ट्रैप स्वीकृति की प्रतीक्षा करता है,
  `PAIRING_REQUIRED` विवरण में `recommendedNextStep: "wait_then_retry"`,
  `retryable: true`, और `pauseReconnect: false` शामिल होते हैं। अनुरोध स्वीकृत
  होने या टोकन अमान्य होने तक उसी बूटस्ट्रैप टोकन से दोबारा कनेक्ट करते रहें।
- `hello-ok.auth.deviceTokens` को केवल तभी सहेजें जब कनेक्ट ने
  `wss://` या लूपबैक/स्थानीय पेयरिंग जैसे विश्वसनीय ट्रांसपोर्ट पर
  बूटस्ट्रैप प्रमाणीकरण का उपयोग किया हो।
- यदि कोई क्लाइंट स्पष्ट `deviceToken` या स्पष्ट `scopes` देता है, तो
  कॉलर द्वारा अनुरोधित वह कार्यक्षेत्र सेट प्रामाणिक बना रहता है; कैश किए गए
  कार्यक्षेत्रों का पुनः उपयोग केवल तब होता है जब क्लाइंट सहेजे गए प्रति-डिवाइस
  टोकन का पुनः उपयोग कर रहा हो।
- डिवाइस टोकन को `device.token.rotate` और `device.token.revoke` के माध्यम
  से रोटेट/निरस्त किया जा सकता है (`operator.pairing` आवश्यक है)। किसी Node
  या अन्य गैर-ऑपरेटर भूमिका को रोटेट या निरस्त करने के लिए
  `operator.admin` भी आवश्यक है।
- `device.token.rotate` रोटेशन मेटाडेटा लौटाता है। यह प्रतिस्थापन
  बेयरर टोकन केवल उसी डिवाइस की उन कॉल के लिए लौटाता है जो पहले से उस डिवाइस
  टोकन से प्रमाणित हैं, ताकि केवल-टोकन क्लाइंट दोबारा कनेक्ट करने से पहले अपना
  प्रतिस्थापन सहेज सकें। साझा/व्यवस्थापक रोटेशन बेयरर टोकन वापस नहीं करते।
- टोकन जारी करना, रोटेशन और निरस्तीकरण उस डिवाइस की पेयरिंग प्रविष्टि में दर्ज
  स्वीकृत भूमिका सेट तक सीमित रहते हैं; टोकन परिवर्तन किसी ऐसी डिवाइस भूमिका
  का विस्तार या उसे लक्षित नहीं कर सकता जिसे पेयरिंग स्वीकृति ने कभी प्रदान
  नहीं किया।
- पेयर किए गए डिवाइस टोकन सत्रों के लिए, डिवाइस प्रबंधन स्वयं तक सीमित होता है,
  जब तक कॉलर के पास `operator.admin` भी न हो: गैर-व्यवस्थापक कॉलर केवल
  अपनी डिवाइस प्रविष्टि के ऑपरेटर टोकन को प्रबंधित कर सकते हैं। Node और अन्य
  गैर-ऑपरेटर टोकन प्रबंधन केवल व्यवस्थापक के लिए है, यहाँ तक कि कॉलर के अपने
  डिवाइस के लिए भी।
- `device.token.rotate` और `device.token.revoke` लक्षित ऑपरेटर टोकन
  कार्यक्षेत्र सेट की जाँच भी कॉलर के वर्तमान सत्र कार्यक्षेत्रों के विरुद्ध करते
  हैं। गैर-व्यवस्थापक कॉलर अपने पास पहले से मौजूद कार्यक्षेत्र से अधिक व्यापक
  ऑपरेटर टोकन को रोटेट या निरस्त नहीं कर सकते।
- प्रमाणीकरण विफलताओं में `error.details.code` और पुनर्प्राप्ति संकेत शामिल हैं:
  - `error.details.canRetryWithDeviceToken` (बूलियन)
  - `error.details.recommendedNextStep`: `retry_with_device_token`,
    `update_auth_configuration`, `update_auth_credentials`,
    `wait_then_retry`, `review_auth_configuration` में से एक
    (`packages/gateway-protocol/src/connect-error-details.ts`)।
- `AUTH_TOKEN_MISMATCH` के लिए क्लाइंट व्यवहार:
  - विश्वसनीय क्लाइंट कैश किए गए प्रति-डिवाइस टोकन से एक सीमित पुनः प्रयास कर
    सकते हैं।
  - यदि वह पुनः प्रयास विफल होता है, तो स्वचालित पुनः कनेक्ट लूप रोकें और
    ऑपरेटर कार्रवाई मार्गदर्शन दिखाएँ।
- `AUTH_SCOPE_MISMATCH` का अर्थ है कि डिवाइस टोकन पहचाना गया था, लेकिन वह
  अनुरोधित भूमिका/कार्यक्षेत्रों को समाहित नहीं करता। इसे खराब टोकन के रूप में
  प्रस्तुत न करें; ऑपरेटर को दोबारा पेयर करने या अधिक संकीर्ण/व्यापक कार्यक्षेत्र
  अनुबंध स्वीकृत करने के लिए कहें।

## डिवाइस पहचान और पेयरिंग

- Nodes में की-पेयर फ़िंगरप्रिंट से प्राप्त स्थिर डिवाइस पहचान
  (`device.id`) शामिल होनी चाहिए।
- Gateways प्रति डिवाइस + भूमिका टोकन जारी करते हैं।
- नई डिवाइस ID के लिए पेयरिंग स्वीकृति आवश्यक है, जब तक स्थानीय
  स्वतः-स्वीकृति सक्षम न हो।
- पेयरिंग स्वतः-स्वीकृति प्रत्यक्ष स्थानीय लूपबैक कनेक्शन पर केंद्रित है।
- OpenClaw में विश्वसनीय साझा-सीक्रेट सहायक प्रवाहों के लिए एक सीमित
  बैकएंड/कंटेनर-स्थानीय स्व-कनेक्ट पथ भी है।
- समान-होस्ट टेलनेट या LAN कनेक्शन को पेयरिंग के लिए फिर भी रिमोट माना जाता है
  और स्वीकृति आवश्यक होती है।
- WS क्लाइंट सामान्यतः `connect` के दौरान `device`
  पहचान शामिल करते हैं (ऑपरेटर + Node)। बिना-डिवाइस ऑपरेटर के एकमात्र अपवाद
  स्पष्ट विश्वसनीयता पथ हैं:
  - केवल-लोकलहोस्ट असुरक्षित HTTP संगतता के लिए
    `gateway.controlUi.allowInsecureAuth=true`।
  - सफल `gateway.auth.mode: "trusted-proxy"` ऑपरेटर Control UI प्रमाणीकरण।
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (आपातकालीन उपाय, गंभीर
    सुरक्षा अवनयन)।
  - आरक्षित आंतरिक सहायक पथ पर प्रत्यक्ष-लूपबैक `gateway-client` बैकएंड
    RPC।
- डिवाइस पहचान न देने से कार्यक्षेत्र-संबंधी परिणाम होते हैं। जब बिना-डिवाइस
  ऑपरेटर कनेक्शन को किसी स्पष्ट विश्वसनीयता पथ से अनुमति दी जाती है, तब भी
  OpenClaw स्वयं घोषित कार्यक्षेत्रों को खाली सेट कर देता है, जब तक उस पथ के
  लिए नामित कार्यक्षेत्र-संरक्षण अपवाद न हो। इसके बाद कार्यक्षेत्र-नियंत्रित
  विधियाँ `missing scope` के साथ विफल होती हैं।
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` एक Control UI आपातकालीन
  कार्यक्षेत्र-संरक्षण पथ है। यह मनमाने कस्टम बैकएंड या CLI-रूपी WebSocket
  क्लाइंट को कार्यक्षेत्र प्रदान नहीं करता।
- आरक्षित प्रत्यक्ष-लूपबैक `gateway-client` बैकएंड सहायक पथ केवल आंतरिक
  स्थानीय कंट्रोल-प्लेन RPC के लिए कार्यक्षेत्र संरक्षित रखता है; कस्टम बैकएंड
  ID को यह अपवाद नहीं मिलता।
- सभी कनेक्शनों को सर्वर द्वारा दिए गए `connect.challenge` नॉन्स पर हस्ताक्षर
  करना होगा।

### डिवाइस प्रमाणीकरण माइग्रेशन निदान

उन पुराने क्लाइंट के लिए जो अभी भी चुनौती-पूर्व हस्ताक्षर व्यवहार का उपयोग करते हैं, `connect`
स्थिर `error.details.reason` के साथ `error.details.code` के अंतर्गत
`DEVICE_AUTH_*` विवरण कोड लौटाता है।

सामान्य माइग्रेशन विफलताएँ:

| संदेश                     | details.code                     | details.reason           | अर्थ                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | क्लाइंट ने `device.nonce` नहीं दिया (या रिक्त भेजा)।     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | क्लाइंट ने पुराने/गलत नॉन्स से हस्ताक्षर किया।            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | हस्ताक्षर पेलोड v2 पेलोड से मेल नहीं खाता।       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | हस्ताक्षरित टाइमस्टैम्प अनुमत विचलन से बाहर है।          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` सार्वजनिक कुंजी फ़िंगरप्रिंट से मेल नहीं खाता। |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | सार्वजनिक कुंजी प्रारूप/कैननिकलाइज़ेशन विफल हुआ।         |

माइग्रेशन लक्ष्य:

- हमेशा `connect.challenge` की प्रतीक्षा करें।
- सर्वर नॉन्स शामिल करने वाले v2 पेलोड पर हस्ताक्षर करें।
- `connect.params.device.nonce` में वही नॉन्स भेजें।
- वरीय हस्ताक्षर पेलोड `v3`
  (`packages/gateway-client/src/device-auth.ts` में `buildDeviceAuthPayloadV3`) है,
  जो डिवाइस/क्लाइंट/भूमिका/कार्यक्षेत्र/टोकन/नॉन्स फ़ील्ड के अतिरिक्त
  `platform` और `deviceFamily` को बाँधता है।
- संगतता के लिए पुराने `v2` हस्ताक्षर स्वीकार किए जाते हैं, लेकिन
  पेयर किए गए डिवाइस की मेटाडेटा पिनिंग पुनः कनेक्शन पर कमांड नीति को नियंत्रित
  करती रहती है।

## TLS और पिनिंग

- WS कनेक्शन के लिए TLS समर्थित है (`gateway.tls` कॉन्फ़िगरेशन)।
- क्लाइंट वैकल्पिक रूप से `gateway.remote.tlsFingerprint` या CLI
  `--tls-fingerprint` के माध्यम से Gateway प्रमाणपत्र फ़िंगरप्रिंट पिन कर सकते हैं।
