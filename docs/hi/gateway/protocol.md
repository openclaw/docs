Gateway WS प्रोटोकॉल OpenClaw के लिए एकल नियंत्रण प्लेन और Node ट्रांसपोर्ट है।
ऑपरेटर और Node क्लाइंट (CLI, वेब UI, macOS ऐप, iOS/Android Node,
हेडलेस Node) WebSocket पर कनेक्ट होते हैं और हैंडशेक के समय एक **भूमिका** और **स्कोप** घोषित करते हैं।

## npm पैकेज

ये पैकेज OpenClaw रिलीज़ शृंखलाओं के साथ वितरित होते हैं। आरंभिक रोलआउट के दौरान,
पहली पैकेज-युक्त रिलीज़ प्रकाशित होने तक npm `E404` लौटा सकता है।

- [`@openclaw/gateway-protocol`](https://www.npmjs.com/package/@openclaw/gateway-protocol)
  स्कीमा, वैलिडेटर, TypeScript प्रकार, हल्के फ़्रेम और त्रुटि
  सहायक तथा संस्करण स्थिरांक प्रकाशित करता है। इसके टारबॉल में जनरेट किया गया
  [`protocol.schema.json`](https://unpkg.com/@openclaw/gateway-protocol/protocol.schema.json)
  मशीन-पठनीय अनुबंध शामिल है।
- [`@openclaw/gateway-client`](https://www.npmjs.com/package/@openclaw/gateway-client)
  संदर्भ Node क्लाइंट और
  `@openclaw/gateway-client/browser` पर ब्राउज़र-सुरक्षित एंट्री प्रकाशित करता है।

एप्लिकेशन जीवनचक्र मार्गदर्शन के लिए,
[Gateway क्लाइंट बनाना](https://docs.openclaw.ai/gateway/clients) देखें। Gateway को
चाइल्ड प्रोसेस के रूप में पर्यवेक्षित करने वाले ऐप्स के लिए,
[OpenClaw एम्बेड करना](https://docs.openclaw.ai/gateway/embedding) देखें।

## ट्रांसपोर्ट और फ़्रेमिंग

- WebSocket, टेक्स्ट फ़्रेम, JSON पेलोड।
- पहला फ़्रेम `connect` अनुरोध **होना अनिवार्य है**।
- प्री-कनेक्ट फ़्रेम 64 KiB (`MAX_PREAUTH_PAYLOAD_BYTES`) तक सीमित हैं। हैंडशेक के बाद,
  `hello-ok.policy.maxPayload` और
  `hello-ok.policy.maxBufferedBytes` का पालन करें। डायग्नोस्टिक्स सक्षम होने पर, अत्यधिक बड़े
  इनबाउंड फ़्रेम और धीमे आउटबाउंड बफ़र, Gateway द्वारा फ़्रेम बंद करने या
  छोड़ने से पहले `payload.large` इवेंट उत्सर्जित करते हैं। इन इवेंट में `surface`, बाइट
  आकार, सीमाएँ और एक सुरक्षित कारण कोड होते हैं; संदेश बॉडी, अटैचमेंट
  सामग्री, रॉ फ़्रेम बाइट, टोकन, कुकी या सीक्रेट कभी नहीं होते।

फ़्रेम के आकार:

- अनुरोध: `{type:"req", id, method, params}`
- प्रतिक्रिया: `{type:"res", id, ok, payload|error}`
- इवेंट: `{type:"event", event, payload, seq?, stateVersion?}`

प्रतिक्रिया त्रुटियाँ `{ code, message, details?, retryable?, retryAfterMs? }` का उपयोग करती हैं।
क्लाइंट को `code` और `details.code` के आधार पर शाखा चुननी चाहिए; `message` मानव-पठनीय रहता है
और जहाँ संगतता नोट अन्यथा न कहे, वहाँ बदल सकता है। विधि-स्तरीय
प्राधिकरण विफलताएँ संरचित अनुपलब्ध-स्कोप विवरण के साथ शीर्ष-स्तरीय `code: "FORBIDDEN"` का उपयोग करती हैं:

- अनुपलब्ध स्कोप: `{ code: "MISSING_SCOPE", missingScope, requiredScopes }`।
  `requiredScopes` अनुरोधित ऑपरेशन के लिए ज्ञात स्कोप का संपूर्ण सेट है।
  पुराने क्लाइंट के लिए विरासती `missing scope: <scope>` संदेश बनाए रखा गया है।

क्लाइंट को पहले `details` पढ़ना चाहिए और विरासती संदेश का उपयोग केवल संगतता
फ़ॉलबैक के रूप में करना चाहिए। `readMissingScopeError` और `readMissingScopeErrorDetails` को
`@openclaw/gateway-protocol/gateway-error-details` से एक्सपोर्ट किया जाता है; ब्राउज़र-सुरक्षित Gateway क्लाइंट
उन्हें `@openclaw/gateway-client/browser` से पुनः एक्सपोर्ट करता है।

स्कीमा को `@openclaw/gateway-protocol/schema` से `GatewayErrorDetailsSchema`,
`MissingScopeErrorDetailsSchema` के रूप में एक्सपोर्ट किया जाता है।
HTTP स्कोप विफलताएँ `error.details` के अंतर्गत `MISSING_SCOPE` ऑब्जेक्ट को प्रतिबिंबित करती हैं और
HTTP स्थिति `403` का उपयोग करती हैं।

दुष्प्रभाव उत्पन्न करने वाली विधियों के लिए आइडेम्पोटेंसी कुंजियाँ आवश्यक हैं (स्कीमा देखें)।

## हैंडशेक

Gateway एक प्री-कनेक्ट चुनौती भेजता है:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

क्लाइंट `connect` से उत्तर देता है:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway `hello-ok` से प्रतिक्रिया देता है:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`, `features`, `snapshot`, `policy` और `auth`, सभी
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) द्वारा आवश्यक हैं। कोई डिवाइस टोकन जारी न होने पर भी `auth`
तय की गई भूमिका/स्कोप की रिपोर्ट करता है (ऊपर दिया गया आकार)। `pluginSurfaceUrls` वैकल्पिक है और Plugin सतह नामों (उदा.
`canvas`) को स्कोप किए गए होस्टेड URL से मैप करता है; इसकी समय-सीमा समाप्त हो सकती है, इसलिए Node
नई एंट्री के लिए `{ "surface": "canvas" }` के साथ `node.pluginSurface.refresh` कॉल करते हैं।
बहिष्कृत `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
पथ समर्थित नहीं है; Plugin सतहों का उपयोग करें।
स्नैपशॉट का वैकल्पिक `appliedConfigHash`, सक्रिय Gateway रनटाइम द्वारा स्वीकृत
समाधानित स्रोत-कॉन्फ़िग संशोधन है। क्लाइंट इसकी तुलना
`config.get.configRevisionHash` से करके यह निर्धारित कर सकते हैं कि क्या किसी अधिक नए सहेजे गए कॉन्फ़िग को अभी भी
पुनरारंभ की आवश्यकता है। `config.get.hash`, कॉन्फ़िग लेखन विरोध गार्ड द्वारा उपयोग किया जाने वाला
रॉ रूट-फ़ाइल संशोधन बना रहता है।

जब Gateway अभी भी स्टार्टअप साइडकार पूर्ण कर रहा हो, तब `connect`,
`details.reason: "startup-sidecars"` और `retryAfterMs` के साथ पुनः प्रयास योग्य
`UNAVAILABLE` त्रुटि लौटा सकता है। इसे अंतिम हैंडशेक विफलता मानने के बजाय
अपनी कनेक्शन समय-सीमा के भीतर पुनः प्रयास करें।

डिवाइस टोकन जारी होने पर, `hello-ok.auth` उसे जोड़ता है:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

अंतर्निहित QR/सेटअप-कोड बूटस्ट्रैप एक मोबाइल हैंडऑफ़ पथ है। सफल
बेसलाइन सेटअप-कोड कनेक्शन एक प्राथमिक Node टोकन और एक सीमित
ऑपरेटर टोकन लौटाता है:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

यह ऑपरेटर हैंडऑफ़ जानबूझकर सीमित है: मोबाइल
ऑपरेटर लूप और नेटिव सेटअप आरंभ करने के लिए पर्याप्त है, जिसमें Talk
कॉन्फ़िग पढ़ने हेतु `operator.talk.secrets` शामिल है, लेकिन इसमें कोई पेयरिंग-म्यूटेशन स्कोप और कोई `operator.admin` नहीं है। व्यापक
पेयरिंग/एडमिन एक्सेस के लिए अलग अनुमोदित पेयरिंग या टोकन प्रवाह आवश्यक है। `hello-ok.auth.deviceTokens` को केवल तभी बनाए रखें जब बूटस्ट्रैप प्रमाणीकरण किसी विश्वसनीय
ट्रांसपोर्ट (`wss://` या लूपबैक/स्थानीय पेयरिंग) पर चला हो।

विश्वसनीय समान-प्रोसेस बैकएंड क्लाइंट (`client.id: "gateway-client"`,
`client.mode: "backend"`), साझा Gateway टोकन/पासवर्ड से प्रमाणीकरण करते समय
प्रत्यक्ष लूपबैक कनेक्शन पर `device` छोड़ सकते हैं। यह पथ आंतरिक
नियंत्रण-प्लेन RPC (उदा. सबएजेंट सत्र अपडेट) के लिए आरक्षित है और पुराने
CLI/डिवाइस पेयरिंग बेसलाइन को स्थानीय बैकएंड कार्य अवरुद्ध करने से बचाता है। रिमोट,
ब्राउज़र-मूल, Node और स्पष्ट डिवाइस-टोकन/डिवाइस-पहचान क्लाइंट अब भी
सामान्य पेयरिंग और स्कोप-अपग्रेड जाँच से गुजरते हैं।

### वर्कर भूमिका और बंद प्रोटोकॉल

क्लाउड वर्कर, Gateway-स्वामित्व वाली,
होस्ट-कुंजी-पिन की गई SSH टनल के माध्यम से एक समर्पित लूपबैक इंग्रेस का उपयोग करते हैं। यह केवल वर्कर पहचान स्वीकार करता है और कभी भी
सामान्य प्रमाणीकरण, Node इवेंट, ऑपरेटर RPC या Plugin विधियाँ डिस्पैच नहीं करता। एक सख्त `connect`
परिवेश, बंडल हैश, स्वामी एपॉक, RPC-सेट संस्करण, समाप्ति और एक शून्य-योग्य सत्र से बँधे
स्थिर-अवस्था में हैश किए गए, अल्पकालिक क्रेडेंशियल को सत्यापित करता है; यह
वर्तमान संस्करण और सुविधा सेट की अलग से जाँच करता है। सफलता न्यूनतम
`worker-hello-ok` लौटाती है; सुविधा नेगोशिएशन सामान्य प्रोटोकॉल
संस्करण से स्वतंत्र है। फ़्रेम 64 KiB से कम रहते हैं, सिवाय तय किए गए `worker.inference.start`
फ़्रेम के, जो 25 MiB तक हो सकता है। बंद अनुमतिसूची में `worker.heartbeat`,
`worker.transcript.commit`, `worker.live-event`, `worker.inference.start` और
`worker.inference.cancel` शामिल हैं।

ट्रांसक्रिप्ट कमिट स्वामी-एपॉक फ़ेंसिंग, Gateway-स्वामित्व वाली सत्र बाइंडिंग,
बेस-लीफ़ कम्पेयर-एंड-स्वैप और टिकाऊ अनुक्रम रीप्ले का उपयोग करते हैं; Gateway सामान्य
सत्र राइटर के माध्यम से ट्रांसक्रिप्ट एंट्री और पैरेंट ID जनरेट करता है। प्रत्येक RPC पर स्वामित्व और
समाप्ति की पुनः जाँच की जाती है।

### क्लाइंट क्षमताएँ

ऑपरेटर क्लाइंट `connect.params.caps` में वैकल्पिक क्षमताएँ विज्ञापित कर सकते हैं:

- `tool-events`: संरचित टूल जीवनचक्र इवेंट स्वीकार करता है।
- `inline-widgets`: होस्ट किए गए इनलाइन विजेट टूल परिणाम रेंडर कर सकता है।

क्लाइंट क्षमताएँ कनेक्टेड क्लाइंट का वर्णन करती हैं, प्राधिकरण का नहीं। एजेंट टूल आवश्यक क्षमताएँ घोषित कर सकते हैं; जब तक प्रत्येक आवश्यकता मूल क्लाइंट के `caps` में उपस्थित न हो, Gateway उन टूल को छोड़ देता है। चैनल से आरंभ हुए रन में कोई Gateway क्लाइंट क्षमता नहीं होती, इसलिए क्षमता-सीमित टूल तब भी अनुपलब्ध होते हैं जब टूल नीति उन्हें स्पष्ट रूप से अनुमति देती है।

### Node कनेक्ट उदाहरण

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "ios-node",
      "version": "1.2.3",
      "platform": "ios",
      "mode": "node"
    },
    "role": "node",
    "scopes": [],
    "caps": ["camera", "canvas", "screen", "location", "voice"],
    "commands": ["camera.snap", "canvas.navigate", "screen.record", "location.get"],
    "permissions": { "camera.capture": true, "screen.record": false },
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-ios/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Node कनेक्ट के समय क्षमता दावे घोषित करते हैं:

- `caps`: `camera`, `canvas`, `screen`,
  `location`, `voice`, `talk` जैसी उच्च-स्तरीय श्रेणियाँ।
- `commands`: इनवोक के लिए कमांड अनुमतिसूची।
- `permissions`: सूक्ष्म टॉगल (उदा. `screen.record`, `camera.capture`)।

Gateway इन्हें दावे मानता है और सर्वर-साइड अनुमतिसूचियाँ लागू करता है।

## भूमिकाएँ और स्कोप

संपूर्ण ऑपरेटर स्कोप मॉडल, अनुमोदन-समय जाँच और साझा-सीक्रेट
सिमेंटिक्स के लिए, [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) देखें।

भूमिकाएँ:

- `operator`: नियंत्रण-प्लेन क्लाइंट (CLI/UI/ऑटोमेशन)।
- `node`: क्षमता होस्ट (camera/screen/canvas/system.run)।
- `worker`: समर्पित, बंद वर्कर प्रोटोकॉल पर क्लाउड निष्पादन होस्ट।

ऑपरेटर स्कोप (`src/gateway/operator-scopes.ts`), संपूर्ण बंद सेट:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`includeSecrets: true` के साथ `talk.config` को `operator.talk.secrets` (या
`operator.admin`) की आवश्यकता होती है। सीक्रेट शामिल होने पर, सक्रिय Talk प्रदाता
क्रेडेंशियल को `talk.resolved.config.apiKey` से पढ़ें; `talk.providers.<id>.apiKey`
स्रोत-आकार में रहता है और SecretRef ऑब्जेक्ट या संशोधित स्ट्रिंग हो सकता है।

Plugin-पंजीकृत Gateway RPC विधियाँ अपने ऑपरेटर स्कोप का अनुरोध कर सकती हैं,
लेकिन ये आरक्षित कोर उपसर्ग हमेशा `operator.admin`
(`src/shared/gateway-method-policy.ts`) में समाधानित होते हैं: `config.*`, `exec.approvals.*`,
`wizard.*`, `update.*`।

विधि स्कोप केवल पहला गेट है। `chat.send` के माध्यम से पहुँचे कुछ स्लैश कमांड
अधिक सख्त कमांड-स्तरीय जाँच लागू करते हैं: स्थायी `/config set` और
`/config unset` लेखन के लिए `operator.admin` आवश्यक है, उन Gateway क्लाइंट के लिए भी
जिनके पास पहले से निम्न ऑपरेटर स्कोप है।

`node.pair.approve` में आधार
विधि स्कोप (`operator.pairing`) के अतिरिक्त, लंबित अनुरोध के घोषित
`commands` (`src/infra/node-pairing-authz.ts`) पर आधारित अनुमोदन-समय की एक अतिरिक्त स्कोप जाँच होती है:

| घोषित कमांड                                                                                                             | आवश्यक स्कोप                       |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| कोई नहीं                                                                                                                          | `operator.pairing`                    |
| सामान्य कमांड                                                                                                             | `operator.pairing` + `operator.write` |
| इसमें `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir`, या `system.execApprovals.get/set` शामिल हैं | `operator.pairing` + `operator.admin` |

### क्षमताएँ/कमांड/अनुमतियाँ (Node)

Nodes कनेक्ट होते समय क्षमता-दावे घोषित करते हैं:

- `caps`: उच्च-स्तरीय क्षमता श्रेणियाँ, जैसे `camera`, `canvas`, `screen`,
  `location`, `voice`, और `talk`।
- `commands`: आह्वान के लिए कमांड अनुमतिसूची।
- `permissions`: सूक्ष्म टॉगल (उदा. `screen.record`, `camera.capture`)।

Gateway इन्हें **दावे** मानता है और सर्वर-साइड अनुमतिसूचियाँ लागू करता है।
सफल कनेक्शन या पुनः कनेक्शन के बाद, कनेक्टेड Nodes `node.pluginTools.update` के साथ वैकल्पिक, एजेंट को दिखाई देने वाले Plugin या MCP टूल
विवरण प्रकाशित कर सकते हैं। हेडलेस Node होस्ट घोषणात्मक MCP इन्वेंटरी
परिवर्तन लागू करने के लिए पुनः आरंभ होते हैं। यह अपडेट विधि ही एकमात्र प्रकाशन पथ है; Plugin टूल विवरण
`connect` पैरामीटर में स्वीकार नहीं किए जाते। प्रत्येक विवरण में प्रदाता-सुरक्षित टूल `name` का उपयोग और
Node की वर्तमान कमांड अनुमतिसूची में मौजूद `command` का नाम होना आवश्यक है। Gateway युग्मित Node से प्राप्त विवरण
मेटाडेटा पर भरोसा करता है, स्वीकृत कमांड
सतह से बाहर के विवरणों को फ़िल्टर करता है, Node के डिस्कनेक्ट होने पर उन्हें हटा देता है और किसी अन्य Node के कैटलॉग को
बदलने के ऑपरेटर प्रयासों को अस्वीकार करता है। Node द्वारा प्रकाशित विवरणों को अनदेखा करने के लिए `gateway.nodes.pluginTools.enabled: false`
सेट करें।

कनेक्टेड Node होस्ट अपनी पूरी Skills प्रतिस्थापन कैटलॉग
`node.skills.update` के साथ प्रकाशित करते हैं। यह Node-भूमिका विधि ही Node Skills के प्रकाशन का एकमात्र
पथ है; Skills को `connect` पैरामीटर में स्वीकार नहीं किया जाता। प्रत्येक विवरण में एक
सुरक्षित नाम, वर्णन और सीमित `SKILL.md` सामग्री होती है। Gateway उस
सामग्री को सामान्य Skills लोडर से पार्स करता है, Node के कनेक्ट रहने तक उसे एजेंट Skills स्नैपशॉट में
शामिल करता है और डिस्कनेक्ट होने पर हटा देता है। Node द्वारा प्रकाशित Skills को अनदेखा करने के लिए
`gateway.nodes.skills.enabled: false` सेट करें।

## उपस्थिति

- `system-presence` डिवाइस पहचान द्वारा कुंजीबद्ध प्रविष्टियाँ लौटाता है, जिनमें
  `deviceId`, `roles`, और `scopes` शामिल हैं, ताकि UI प्रत्येक डिवाइस के लिए एक पंक्ति दिखा सकें, भले ही
  वह ऑपरेटर और Node दोनों के रूप में कनेक्ट हो।
- `node.list` में वैकल्पिक `lastSeenAtMs` और `lastSeenReason` शामिल हैं। कनेक्टेड
  Nodes वर्तमान कनेक्शन समय को `connect` कारण के साथ रिपोर्ट करते हैं; युग्मित Nodes
  किसी विश्वसनीय Node इवेंट के माध्यम से स्थायी पृष्ठभूमि उपस्थिति भी रिपोर्ट कर सकते हैं।

नेटिव macOS Nodes सीमित इनपुट निष्क्रियता समय के साथ प्रमाणित `node.presence.activity` इवेंट भी भेज सकते हैं।
Gateway अपनी घड़ी पर गतिविधि टाइमस्टैम्प प्राप्त करता है, सबसे नए कनेक्टेड Mac को `node.list` और
`node.describe` के माध्यम से उपलब्ध कराता है और रीड-स्कोप वाले क्लाइंट को `node.presence` अपडेट प्रसारित करता है।
चयन, गोपनीयता, मॉडल संदर्भ और सूचना-रूटिंग व्यवहार के लिए
[सक्रिय कंप्यूटर उपस्थिति](/hi/nodes/presence) देखें।

### Node की पृष्ठभूमि सक्रियता का इवेंट

Nodes `event: "node.presence.alive"` के साथ `node.event` को कॉल करके दर्ज करते हैं कि कोई
युग्मित Node पृष्ठभूमि वेक के दौरान सक्रिय था, लेकिन उसे कनेक्टेड के रूप में चिह्नित नहीं करते:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` एक बंद एनम है: `background`, `silent_push`, `bg_app_refresh`,
`significant_location`, `manual`, `connect`। अज्ञात मान
`background` (`src/shared/node-presence.ts`) में सामान्यीकृत होते हैं। इवेंट केवल
प्रमाणित Node डिवाइस सत्रों के लिए स्थायी रूप से सहेजा जाता है; डिवाइस-रहित या अयुग्मित सत्र
`handled: false` लौटाते हैं।

सफल Gateways एक संरचित परिणाम लौटाते हैं:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

पुराने Gateways `node.event` के लिए केवल `{ "ok": true }` लौटा सकते हैं; इसे
स्वीकृत RPC मानें, स्थायी उपस्थिति का सहेजा जाना नहीं।

## प्रसारण इवेंट का स्कोप निर्धारण

सर्वर द्वारा पुश किए गए प्रसारण इवेंट स्कोप द्वारा नियंत्रित होते हैं, ताकि युग्मन-स्कोप वाले या केवल Node
सत्र निष्क्रिय रूप से सत्र सामग्री
(`src/gateway/server-broadcast.ts`) प्राप्त न करें:

- चैट, एजेंट और टूल-परिणाम फ़्रेम (स्ट्रीम किए गए `agent` इवेंट, टूल-परिणाम
  इवेंट) के लिए कम-से-कम `operator.read` आवश्यक है। इसके बिना सत्र इन
  फ़्रेम को पूरी तरह छोड़ देते हैं।
- Plugin-परिभाषित `plugin.*` प्रसारण डिफ़ॉल्ट रूप से `operator.write` या
  `operator.admin` तक सीमित होते हैं; `plugin.approval.requested` / `plugin.approval.resolved` जैसी स्पष्ट प्रविष्टियाँ
  इसके बजाय `operator.approvals` का उपयोग करती हैं।
- स्थिति/ट्रांसपोर्ट इवेंट (`heartbeat`, `presence`, `tick`, कनेक्ट/डिस्कनेक्ट
  जीवनचक्र) अप्रतिबंधित रहते हैं, ताकि प्रत्येक प्रमाणित सत्र ट्रांसपोर्ट की स्थिति देख सके।
- अज्ञात प्रसारण इवेंट परिवार डिफ़ॉल्ट रूप से स्कोप द्वारा नियंत्रित (विफलता पर बंद)
  होते हैं, जब तक कोई पंजीकृत हैंडलर स्पष्ट रूप से उन्हें शिथिल न करे।

प्रत्येक क्लाइंट कनेक्शन अपनी अलग प्रति-क्लाइंट अनुक्रम संख्या रखता है, इसलिए जब अलग-अलग क्लाइंट
इवेंट स्ट्रीम के अलग-अलग स्कोप-फ़िल्टर किए गए उपसमुच्चय देखते हैं, तब भी उस सॉकेट पर प्रसारण
एकदिश बढ़ते क्रम में रहते हैं।

      दोनों उपयोग विधियाँ DST-जागरूक कैलेंडर-दिवस सीमाओं और बकेट के लिए IANA `timeZone` वाला `mode: "specific"` स्वीकार करती हैं। पुराने क्लाइंट के लिए और जब Gateway रनटाइम अनुरोधित ज़ोन को नहीं पहचानता, तब फ़ॉलबैक के रूप में `utcOffset` समर्थित रहता है।

    - `sessions.usage.logs` एक सत्र के लिए उपयोग लॉग प्रविष्टियाँ लौटाता है।

  <Accordion title="Plugin प्रबंधन">
    - `plugins.list` (`operator.read`) इंस्टॉल किए गए Plugin की इन्वेंटरी के साथ स्थानीय रूप से चुने गए आधिकारिक विकल्प, निदान और यह जानकारी लौटाता है कि वर्तमान इंस्टॉल मोड परिवर्तन की अनुमति देता है या नहीं।
    - `plugins.search` (`operator.read`) इंस्टॉल किए जा सकने वाले ClawHub कोड-Plugin और बंडल-Plugin परिवार खोजता है। गैर-रिक्त `query` और 1 से 100 तक वैकल्पिक `limit` पास करें।
    - `plugins.install` (`operator.admin`) या तो `{ source: "official", pluginId }` वाली आधिकारिक कैटलॉग प्रविष्टि या `{ source: "clawhub", packageName, version?, acknowledgeClawHubRisk? }` वाला ClawHub पैकेज इंस्टॉल करता है। ClawHub इंस्टॉलेशन Gateway के विश्वास, अखंडता और इंस्टॉल-नीति जाँचों को बनाए रखते हैं। सफल इंस्टॉलेशन के लिए Gateway को पुनः आरंभ करना आवश्यक है।
    - `plugins.setEnabled` (`operator.admin`) `{ pluginId, enabled }` के साथ किसी एक इंस्टॉल किए गए Plugin की सक्षम नीति बदलता है। प्रतिक्रिया में अपडेट की गई कैटलॉग प्रविष्टि, पुनः आरंभ मेटाडेटा और स्लॉट-चयन संबंधी चेतावनियाँ शामिल होती हैं।
    - `plugins.uninstall` (`operator.admin`) `{ pluginId }` के साथ बाहरी रूप से इंस्टॉल किए गए एक Plugin को हटाता है: कॉन्फ़िगरेशन संदर्भ, इंस्टॉल रिकॉर्ड और प्रबंधित फ़ाइलें। बंडल किए गए Plugins को अनइंस्टॉल नहीं किया जा सकता, केवल अक्षम किया जा सकता है। प्रतिक्रिया हटाने की कार्रवाइयाँ सूचीबद्ध करती है और हमेशा Gateway पुनः आरंभ करना आवश्यक बनाती है।

  </Accordion>

  <Accordion title="ऑपरेटर टर्मिनल">
    - `terminal.open` किसी स्पष्ट `agentId` या डिफ़ॉल्ट एजेंट के लिए होस्ट PTY शुरू करता है और निर्धारित एजेंट, कार्यशील डायरेक्टरी, शेल और परिसीमन स्थिति लौटाता है।
    - `terminal.input`, `terminal.resize`, और `terminal.close` केवल कॉल करने वाले कनेक्शन के स्वामित्व वाले सत्रों पर काम करते हैं।
    - `terminal.upload` अधिकतम 16 MiB की एक base64 फ़ाइल स्वीकार करता है, उसे सत्र के Gateway या युग्मित-Node होस्ट पर निजी 24-घंटे की अस्थायी डायरेक्टरी में रखता है और निरपेक्ष पथ लौटाता है। कॉलर को अब भी उस पथ को पेस्ट करना या किसी अन्य तरीके से उपयोग करना होगा; RPC कभी भी टर्मिनल इनपुट नहीं लिखता या कमांड निष्पादित नहीं करता।
    - `terminal.data` और `terminal.exit` इवेंट केवल सत्र के स्वामी कनेक्शन पर स्ट्रीम होते हैं।
    - जिन सत्रों का कनेक्शन टूट जाता है, वे समाप्त नहीं बल्कि अलग किए जाते हैं: वे `gateway.terminal.detachedSessionTimeoutSeconds` (डिफ़ॉल्ट 300; `0` डिस्कनेक्ट होने पर समाप्त करने का व्यवहार पुनर्स्थापित करता है) तक पुनः अटैच किए जा सकते हैं, जबकि हालिया आउटपुट सीमित सर्वर-साइड बफ़र में जमा होता रहता है।
    - `terminal.list` अटैच किए जा सकने वाले सत्र लौटाता है; `terminal.attach` किसी सक्रिय या अलग किए गए सत्र को कॉल करने वाले कनेक्शन से दोबारा बाँधता है और रीप्ले बफ़र लौटाता है (tmux-शैली अधिग्रहण—पिछले सक्रिय स्वामी को कारण `detached` सहित `terminal.exit` मिलता है); `terminal.text` बिना अटैच किए बफ़र को सादे टेक्स्ट के रूप में पढ़ता है।
    - प्रत्येक टर्मिनल विधि के लिए `operator.admin` आवश्यक है; `gateway.terminal.enabled` स्पष्ट रूप से true होना चाहिए। पूरी तरह सैंडबॉक्स किए गए एजेंट अस्वीकार कर दिए जाते हैं और एजेंट नीति में बदलाव मौजूदा तथा प्रगतिरत PTY को बंद कर देता है, जिनमें अलग किए गए PTY भी शामिल हैं।

  </Accordion>

    - `tts.setProvider` पसंदीदा TTS प्रदाता अपडेट करता है।
    - `tts.convert` एकबारगी टेक्स्ट-टू-स्पीच रूपांतरण चलाता है।
    - `tts.speak` (`operator.write`) कॉन्फ़िगर की गई सामान्य TTS प्रदाता शृंखला के साथ गैर-रिक्त `text` को रेंडर करता है और एक संपूर्ण क्लिप को इनलाइन `audioBase64` के रूप में लौटाता है, साथ ही `provider` और वैकल्पिक `outputFormat`, `mimeType`, तथा `fileExtension` मेटाडेटा भी लौटाता है। `tts.convert` के विपरीत, यह Gateway-स्थानीय पथ नहीं लौटाता; `talk.speak` के विपरीत, इसके लिए Talk प्रदाता आवश्यक नहीं है। `messages.tts.maxTextLength` से अधिक टेक्स्ट पर `INVALID_REQUEST` लौटता है; संश्लेषण विफलताओं पर `UNAVAILABLE` लौटता है।

  </Accordion>

  <Accordion title="सीक्रेट, कॉन्फ़िगरेशन, अपडेट और विज़ार्ड">
    - `secrets.reload` सक्रिय SecretRefs का पुनः निर्धारण करता है और स्वामी-जागरूक रनटाइम स्थिति को परमाण्विक रूप से प्रकाशित करता है। पात्र स्वामी विफलताओं को `warningCount` के साथ कोल्ड या स्टेल डिग्रेडेशन के रूप में प्रकाशित किया जा सकता है; सख़्त या अमैप्ड विफलताएँ रीलोड अस्वीकार करती हैं और सक्रिय स्नैपशॉट सुरक्षित रखती हैं।
    - `secrets.resolve` किसी विशिष्ट कमांड/लक्ष्य सेट के लिए कमांड-लक्ष्य सीक्रेट असाइनमेंट निर्धारित करता है।
    - `config.get` वर्तमान ऑन-डिस्क कॉन्फ़िगरेशन स्नैपशॉट, रॉ रूट-फ़ाइल `hash`, निर्धारित `configRevisionHash`, और सक्रिय Gateway रनटाइम द्वारा स्वीकृत निर्धारित संशोधन के लिए वैकल्पिक `appliedConfigHash` लौटाता है।
    - `config.set` सत्यापित कॉन्फ़िगरेशन पेलोड लिखता है।
    - `config.patch` आंशिक कॉन्फ़िगरेशन अपडेट को मर्ज करता है। विनाशकारी ऐरे प्रतिस्थापन के लिए प्रभावित पथ का `replacePaths` में होना आवश्यक है; ऐरे प्रविष्टियों के अंतर्गत नेस्टेड ऐरे `agents.list[].skills` जैसे `[]` पथों का उपयोग करते हैं।
    - `config.apply` संपूर्ण कॉन्फ़िगरेशन पेलोड को सत्यापित करके प्रतिस्थापित करता है।
    - `config.schema` Control UI और CLI टूलिंग द्वारा उपयोग किया जाने वाला लाइव कॉन्फ़िगरेशन स्कीमा पेलोड लौटाता है: स्कीमा, `uiHints`, संस्करण, जनरेशन मेटाडेटा और लोड किए जा सकने पर Plugin + चैनल स्कीमा मेटाडेटा। इसमें UI के समान लेबल/सहायता टेक्स्ट से `title` / `description` मेटाडेटा शामिल होता है, जिसमें मेल खाने वाला फ़ील्ड दस्तावेज़ उपलब्ध होने पर नेस्टेड ऑब्जेक्ट, वाइल्डकार्ड, ऐरे-आइटम और `anyOf` / `oneOf` / `allOf` संयोजन शाखाएँ भी शामिल होती हैं।
    - `config.schema.lookup` एक कॉन्फ़िगरेशन पथ के लिए पथ-दायरे वाला लुकअप पेलोड लौटाता है: सामान्यीकृत पथ, उथला स्कीमा Node, मेल खाता संकेत + `hintPath`, वैकल्पिक `reloadKind` और UI/CLI ड्रिल-डाउन के लिए निकटतम चाइल्ड सारांश। `reloadKind`, `restart`, `hot`, या `none` (`src/config/schema.ts`) में से एक है और अनुरोधित पथ के लिए Gateway कॉन्फ़िगरेशन रीलोड प्लानर को प्रतिबिंबित करता है। लुकअप स्कीमा Node उपयोगकर्ता-दृश्य दस्तावेज़ और सामान्य सत्यापन फ़ील्ड (`title`, `description`, `type`, `enum`, `const`, `format`, `pattern`, संख्यात्मक/स्ट्रिंग/ऐरे/ऑब्जेक्ट सीमाएँ, `additionalProperties`, `deprecated`, `readOnly`, `writeOnly`) बनाए रखते हैं। चाइल्ड सारांश `key`, सामान्यीकृत `path`, `type`, `required`, `hasChildren`, वैकल्पिक `reloadKind`, और मेल खाते `hint` / `hintPath` प्रदर्शित करते हैं।
    - `update.run` Gateway अपडेट प्रवाह चलाता है और केवल अपडेट सफल होने पर पुनः आरंभ शेड्यूल करता है; सत्र वाले कॉलर `continuationMessage` शामिल कर सकते हैं, ताकि स्टार्टअप पुनः आरंभ निरंतरता कतार के माध्यम से एक फ़ॉलो-अप एजेंट टर्न फिर से शुरू करे। कंट्रोल प्लेन से पैकेज-मैनेजर अपडेट और पर्यवेक्षित git-checkout अपडेट, सक्रिय Gateway के भीतर पैकेज ट्री बदलने या checkout/build आउटपुट संशोधित करने के बजाय अलग की गई प्रबंधित-सेवा हैंडऑफ़ का उपयोग करते हैं। शुरू किया गया हैंडऑफ़ `result.reason: "managed-service-handoff-started"` और `handoff.status: "started"` के साथ `ok: true` लौटाता है। उसी Gateway प्रक्रिया द्वारा संभाला गया दूसरा समवर्ती `update.run`, `result.reason: "managed-service-handoff-already-running"` और `handoff.status: "already-running"` के साथ `ok: false` लौटाता है; उसकी निरंतरता स्वीकार नहीं की जाती, इसलिए सक्रिय अपडेट पूरा होने के बाद कॉलर पुनः प्रयास कर सकता है। स्टैंडअलोन CLI अपडेटर और प्रतिस्थापन Gateway प्रक्रियाएँ इस प्रक्रिया-स्थानीय गार्ड से बाहर हैं। अनुपलब्ध या विफल हैंडऑफ़, `managed-service-handoff-unavailable` या `managed-service-handoff-failed` के साथ `ok: false` लौटाते हैं, और मैन्युअल शेल अपडेट आवश्यक होने पर `handoff.command` भी लौटाते हैं। अनुपलब्ध का अर्थ है कि OpenClaw में सुरक्षित सुपरवाइज़र सीमा या स्थायी सेवा पहचान नहीं है, जैसे systemd के लिए `OPENCLAW_SYSTEMD_UNIT`। शुरू किए गए हैंडऑफ़ के दौरान पुनः आरंभ सेंटिनल थोड़े समय के लिए `stats.reason: "restart-health-pending"` रिपोर्ट कर सकता है; निरंतरता तब तक विलंबित रहती है, जब तक CLI पुनः आरंभ किए गए Gateway को सत्यापित करके अंतिम `ok` सेंटिनल नहीं लिखता।
    - `update.status` नवीनतम अपडेट पुनः आरंभ सेंटिनल को रीफ़्रेश करके लौटाता है, जिसमें उपलब्ध होने पर पुनः आरंभ के बाद चल रहा संस्करण भी शामिल होता है।
    - `wizard.start`, `wizard.next`, `wizard.status`, और `wizard.cancel` WS RPC पर ऑनबोर्डिंग विज़ार्ड उपलब्ध कराते हैं।

  </Accordion>

  <Accordion title="एजेंट और वर्कस्पेस सहायक">
    - `agents.list` प्रभावी मॉडल/रनटाइम मेटाडेटा और वैकल्पिक सिमैंटिक `kind` (`agent` या `system`) सहित Gateway को दिखाई देने वाली एजेंट प्रविष्टियाँ लौटाता है। पूर्ण टाइपयुक्त सूची प्राप्त करने के लिए क्लाइंट `agent-kind` हैंडशेक क्षमता घोषित करते हैं; इसके बिना क्लाइंट सिस्टम पंक्तियों से रहित पुरानी चयनकर्ता-सुरक्षित सूची बनाए रखते हैं। प्रकार-सजग क्लाइंट `system` पंक्तियों को सामान्य चयनकर्ताओं से बाहर रखते हैं, जबकि उन्हें निदान दृश्यों में बनाए रखते हैं। पुराने v4 Gateway ऐसी पंक्तियाँ लौटा सकते हैं जिनमें `kind` न हो।
    - `agents.create`, `agents.update`, और `agents.delete` एजेंट रिकॉर्ड और वर्कस्पेस वायरिंग प्रबंधित करते हैं।
    - `agents.files.list`, `agents.files.get`, और `agents.files.set` किसी एजेंट के लिए उपलब्ध कराई गई बूटस्ट्रैप वर्कस्पेस फ़ाइलें प्रबंधित करते हैं।
    - `audit.activity.list` संस्करणित, केवल-मेटाडेटा गतिविधि लेजर लौटाता है; `audit.list` संगतता-सुरक्षित रन/टूल RPC बना रहता है।
    - `agents.workspace.list` और `agents.workspace.get` (`operator.read`) [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) में वर्णित विश्वसनीय ऑपरेटर डोमेन के क्लाइंट के लिए एजेंट की वर्कस्पेस डायरेक्टरी का केवल-पढ़ने योग्य, पृष्ठांकित ब्राउज़िंग उपलब्ध कराते हैं। अनुरोध केवल वर्कस्पेस-सापेक्ष पथ स्वीकार करते हैं; पढ़ना realpath किए गए वर्कस्पेस रूट तक सीमित रहता है (सिमलिंक और हार्डलिंक से बाहर निकलने के प्रयास अस्वीकार किए जाते हैं), आकार-सीमित होता है, और UTF-8 टेक्स्ट तथा सामान्य इमेज प्रकारों (base64) तक सीमित रहता है। प्रतिक्रियाएँ होस्ट वर्कस्पेस पथ प्रकट नहीं करतीं। इस नेमस्पेस में कोई लेखन ऑपरेशन नहीं है।
    - `tasks.list`, `tasks.get`, और `tasks.cancel` Gateway टास्क लेजर को SDK और ऑपरेटर क्लाइंट के लिए उपलब्ध कराते हैं। नीचे [टास्क लेजर RPC](#task-ledger-rpcs) देखें।
    - `artifacts.list`, `artifacts.get`, और `artifacts.download` स्पष्ट `sessionKey`, `runId`, या `taskId` स्कोप के लिए ट्रांसक्रिप्ट से प्राप्त आर्टिफ़ैक्ट सारांश और डाउनलोड उपलब्ध कराते हैं। रन और टास्क क्वेरी स्वामित्व वाले सेशन को सर्वर-साइड पर हल करती हैं और केवल मेल खाते उद्गम वाला ट्रांसक्रिप्ट मीडिया लौटाती हैं; असुरक्षित या स्थानीय URL स्रोतों को सर्वर-साइड से प्राप्त करने के बजाय असमर्थित डाउनलोड लौटाए जाते हैं।
    - `environments.list` और `environments.status` Gateway-स्थानीय और Node परिवेश खोज को बनाए रखते हैं। कॉन्फ़िगर किए गए क्लाउड वर्कर और पुराने प्रोफ़ाइल द्वारा छोड़े गए स्थायी रिकॉर्ड `providerId`, वैकल्पिक `leaseId`, `state`, `ageMs`, वैकल्पिक `idleMs`, और `attachedSessionIds` वाला `worker` मेटाडेटा जोड़ते हैं। वर्कर जीवनचक्र स्थितियाँ `requested`, `provisioning`, `bootstrapping`, `ready`, `attached`, `idle`, `draining`, `destroying`, `destroyed`, `failed`, और `orphaned` हैं।
    - `environments.create` (`{ profileId, idempotencyKey }`) किसी कॉन्फ़िगर किए गए Plugin प्रदाता प्रोफ़ाइल से वर्कर का प्रावधान करता है; समान कुंजी वाले पुनः प्रयास उसी स्थायी ऑपरेशन का पुनः उपयोग करते हैं। `environments.destroy` (`{ environmentId }`) स्थायी वर्कर परिवेश को इडेम्पोटेंट रूप से हटाने का अनुरोध करता है। दोनों के लिए `operator.admin` आवश्यक है, ये कंट्रोल-प्लेन लेखन हैं, और स्थिति प्रतिक्रियाओं द्वारा प्रयुक्त समान परिवेश सारांश संरचना लौटाते हैं।
    - `agent.identity.get` किसी एजेंट या सेशन की प्रभावी सहायक पहचान लौटाता है।
    - `agent.wait` रन समाप्त होने की प्रतीक्षा करता है और उपलब्ध होने पर टर्मिनल स्नैपशॉट लौटाता है।

  </Accordion>

  <Accordion title="सेशन नियंत्रण">
    - `sessions.list` वर्तमान सेशन इंडेक्स लौटाता है, जिसमें एजेंट रनटाइम बैकएंड कॉन्फ़िगर होने पर प्रत्येक पंक्ति का `agentRuntime` मेटाडेटा शामिल होता है। क्लाउड-वर्कर प्लेसमेंट सक्षम होने या स्थायी पुनर्प्राप्ति स्थिति मौजूद होने पर सेशन पंक्तियों में बंद `placement` स्थिति (`local`, `requested`, `provisioning`, `syncing`, `starting`, `active`, `draining`, `reconciling`, `reclaimed`, या `failed`) के साथ स्थिति-विशिष्ट परिवेश, स्वामी-युग, वर्कस्पेस, बंडल, ACK-कर्सर, या पुनर्प्राप्ति फ़ील्ड भी शामिल होते हैं।
    - `sessions.subscribe` और `sessions.unsubscribe` वर्तमान WS क्लाइंट के लिए सेशन परिवर्तन इवेंट सदस्यताएँ चालू या बंद करते हैं।
    - `sessions.messages.subscribe` और `sessions.messages.unsubscribe` एक सेशन के लिए ट्रांसक्रिप्ट/संदेश इवेंट सदस्यताएँ चालू या बंद करते हैं। उन अनुमोदनों के स्वच्छ किए गए `session.approval` जीवनचक्र इवेंट भी प्राप्त करने के लिए `includeApprovals: true` दें, जिनकी स्थायी ऑडियंस में ठीक वही सेशन शामिल है और जिनकी समीक्षक बाइंडिंग सदस्यता लेने वाले क्लाइंट को अधिकृत करती है। इसके बाद सदस्यता प्रतिक्रिया में सीमित लंबित `approvalReplay` शामिल होता है; `truncated` के false होने पर यह प्रामाणिक होता है। ऑप्ट-इन प्रत्येक सदस्यता कॉल के लिए है, स्थायी नहीं: `includeApprovals: true` के बिना उसी सेशन की फिर से सदस्यता लेने पर मौजूदा अनुमोदन सदस्यता हटा दी जाती है। सामान्य सेशन-पठन प्राधिकार के अतिरिक्त, इस ऑप्ट-इन के लिए `operator.admin`, या युग्मित डिवाइस पर `operator.approvals` आवश्यक है।
    - `sessions.preview` विशिष्ट सेशन कुंजियों के लिए सीमित ट्रांसक्रिप्ट पूर्वावलोकन लौटाता है।
    - `sessions.describe` सटीक सेशन कुंजी के लिए एक Gateway सेशन पंक्ति लौटाता है।
    - `sessions.resolve` सेशन लक्ष्य को हल या कैनोनिकलाइज़ करता है।
    - `sessions.create` नई सेशन प्रविष्टि बनाता है। वैकल्पिक `model` और `thinkingLevel` मान आरंभिक मॉडल और रीजनिंग ओवरराइड को परमाण्विक रूप से स्थायी बनाते हैं। `worktree: true` प्रबंधित वर्कट्री का प्रावधान करता है; वैकल्पिक `worktreeBaseRef`/`worktreeName` आधार रेफ़ और ब्रांच नाम चुनते हैं, और `execNode` (`operator.admin`) सेशन निष्पादन को Node होस्ट से बाँधता है। बनाया गया वर्कट्री परिणाम में लौटाया जाता है और सेशन पंक्ति (`worktree: { id, branch, repoRoot }`) में स्थायी किया जाता है। जब प्रविष्टि बन जाती है, लेकिन उसका नेस्टेड आरंभिक `chat.send` अस्वीकार हो जाता है, तो सफल परिणाम में `runStarted: false` और `runError` शामिल होते हैं; क्लाइंट प्रॉम्प्ट बनाए रखकर लौटाई गई सेशन कुंजी के विरुद्ध पुनः प्रयास कर सकते हैं। `emitCommandHooks: true` के साथ `parentSessionKey` देने वाले कॉलर को भिन्न चाइल्ड के जीवनचक्र निपटान की घोषणा भी करनी चाहिए: `succeedsParent: true` पैरेंट को `session_end` के साथ समाप्त करता है, जबकि `false` पैरेंट को सक्रिय रखता है और केवल चाइल्ड का `session_start` उत्सर्जित करता है। `succeedsParent` छोड़ने पर मौजूदा क्लाइंट के लिए पुराना पैरेंट-रोलओवर व्यवहार बना रहता है। इस निपटान के लिए पैरेंट लिंकेज और कमांड हुक दोनों आवश्यक हैं; फ़ोर्क अपने पैरेंट को सफल घोषित नहीं कर सकता। मुख्य सेशन का उसी स्थान पर रीसेट व्यवहार अपरिवर्तित रहता है, क्योंकि कोई अलग चाइल्ड नहीं बनाया जाता।
    - `sessions.dispatch` (`operator.admin`) सेशन-स्वामित्व वाले प्रबंधित वर्कट्री सहित मौजूदा स्थानीय OpenClaw सेशन को कॉन्फ़िगर किए गए क्लाउड-वर्कर प्रोफ़ाइल में ले जाता है। `{ key, profileId, agentId? }` दें। कोई वर्कर प्रोफ़ाइल कॉन्फ़िगर न होने पर यह विधि अनुपस्थित रहती है, सक्रिय कार्य को समाप्त होने देने से पहले स्थानीय टर्न प्रवेश बंद करती है, और प्लेसमेंट के `active` वर्कर स्वामित्व तक पहुँचने के बाद ही लौटती है। डिस्पैच एकतरफ़ा है; वर्कर से स्थानीय परिवेश में वापस खींचना इस RPC का भाग नहीं है।
    - `sessions.groups.list`, `sessions.groups.put`, `sessions.groups.rename`, और `sessions.groups.delete` Gateway-स्वामित्व वाला कस्टम सेशन समूह कैटलॉग (नाम + प्रदर्शन क्रम) प्रबंधित करते हैं। सदस्यता प्रत्येक सेशन के `category` फ़ील्ड पर रहती है; नाम बदलने और हटाने पर सदस्य सेशन सर्वर-साइड अपडेट होते हैं।
    - `sessions.send` मौजूदा सेशन में संदेश भेजता है।
    - `sessions.steer` सक्रिय सेशन के लिए बाधित करके दिशा बदलने वाला संस्करण है।
    - `sessions.abort` किसी सेशन का सक्रिय कार्य निरस्त करता है। `key` के साथ वैकल्पिक `runId` दें, या उन सक्रिय रन के लिए केवल `runId` दें जिन्हें Gateway किसी सेशन से संबद्ध कर सकता है।
    - `sessions.patch` सेशन मेटाडेटा/ओवरराइड अपडेट करता है और हल किया गया कैनोनिकल मॉडल तथा प्रभावी `agentRuntime` रिपोर्ट करता है।
    - `sessions.reset`, `sessions.delete`, और `sessions.compact` सेशन रखरखाव करते हैं।
    - `sessions.get` पूर्ण संग्रहीत सेशन पंक्ति लौटाता है।
    - चैट निष्पादन अब भी `chat.history`, `chat.send`, `chat.abort`, और `chat.inject` का उपयोग करता है। `chat.history` UI क्लाइंट के लिए प्रदर्शन-अनुकूल रूप से सामान्यीकृत होता है: इनलाइन डायरेक्टिव टैग दृश्यमान टेक्स्ट से हटा दिए जाते हैं, सादा-टेक्स्ट टूल-कॉल XML पेलोड (`<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>`, और काटे गए टूल-कॉल ब्लॉक) तथा लीक हुए ASCII/पूर्ण-चौड़ाई मॉडल नियंत्रण टोकन हटा दिए जाते हैं, केवल मौन-टोकन वाली सहायक पंक्तियाँ (सटीक `NO_REPLY` / `no_reply`) छोड़ दी जाती हैं, और अत्यधिक बड़ी पंक्तियों को प्लेसहोल्डर से बदला जा सकता है।
    - `chat.message.get` किसी एक दृश्यमान ट्रांसक्रिप्ट प्रविष्टि के लिए योगात्मक, सीमित पूर्ण-संदेश रीडर है। `sessionKey`, एजेंट-स्कोप वाले सेशन चयन के समय वैकल्पिक `agentId`, और `chat.history` के माध्यम से पहले प्रदर्शित किया गया ट्रांसक्रिप्ट `messageId` दें; यदि संग्रहीत प्रविष्टि अब भी उपलब्ध है और अत्यधिक बड़ी नहीं है, तो Gateway हल्की हिस्ट्री कटौती सीमा के बिना वही प्रदर्शन-अनुकूल सामान्यीकृत प्रक्षेपण लौटाता है।
    - `chat.toolTitles` Control UI में रेंडर किए गए टूल कॉल के संक्षिप्त उद्देश्य शीर्षक लौटाता है (बैच में, सीमित इनपुट के साथ अधिकतम 24 आइटम)। यह सुविधा `gateway.controlUi.toolTitles` के माध्यम से ऑप्ट-इन है (डिफ़ॉल्ट रूप से बंद); अक्षम Gateway बिना मॉडल कॉल के `{ titles: {}, disabled: true }` उत्तर देते हैं, ताकि क्लाइंट अनुरोध करना बंद कर दें। सक्षम होने पर शीर्षक मानक यूटिलिटी-मॉडल रूटिंग का उपयोग करते हैं: स्पष्ट रूप से कॉन्फ़िगर किया गया `utilityModel` (एक ऑपरेटर निर्णय, जो सभी यूटिलिटी कार्यों की तरह सीमित कार्य सामग्री चुने गए प्रदाता को भेज सकता है), अन्यथा सेशन प्रदाता का घोषित लघु-मॉडल डिफ़ॉल्ट, ताकि कोई नया निर्गमन गंतव्य अप्रत्यक्ष रूप से न जोड़ा जाए; खाली `utilityModel` इन्हें पूरी तरह अक्षम कर देता है। शीर्षक कभी भी प्राथमिक मॉडल पर फ़ॉलबैक नहीं करते। परिणाम टूल नाम + इनपुट से कुंजीकृत प्रति-एजेंट स्थिति डेटाबेस में कैश होते हैं, इसलिए बार-बार देखने पर समान कॉल के लिए दोबारा शुल्क नहीं लगता।
    - `chat.send` स्वचालित कटऑफ़ से पहले शुरू हुई मॉडल कॉल के लिए फ़ास्ट मोड का उपयोग करने हेतु एक-टर्न `fastMode: "auto"` स्वीकार करता है, फिर बाद के पुनः प्रयास, फ़ॉलबैक, टूल-परिणाम, या निरंतरता कॉल फ़ास्ट मोड के बिना शुरू करता है। कटऑफ़ का डिफ़ॉल्ट 60 सेकंड (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) है और इसे प्रत्येक मॉडल के लिए `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` से कॉन्फ़िगर किया जा सकता है। `chat.send` कॉलर उस अनुरोध के लिए कटऑफ़ ओवरराइड करने हेतु एक-टर्न `fastAutoOnSeconds` दे सकता है। केवल इस अनुरोध के लिए संग्रहीत कतार मोड को ओवरराइड करने हेतु `queueMode` (`steer`, `followup`, `collect`, या `interrupt`) दें; स्पष्ट Control UI दिशा-परिवर्तन क्रियाएँ `queueMode: "steer"` का उपयोग करती हैं।

  </Accordion>

  </Accordion>

  <Accordion title="ऑटोमेशन, Skills और टूल">
    - ऑटोमेशन: `wake` तत्काल या अगले-Heartbeat वेक टेक्स्ट इंजेक्शन को शेड्यूल करता है; `cron.get`, `cron.list`, `cron.status`, `cron.add`, `cron.update`, `cron.remove`, `cron.run`, `cron.runs` शेड्यूल किए गए कार्य को प्रबंधित करते हैं।

    - Skills और टूल: `commands.list`, `skills.*`, `tools.catalog`, `tools.effective`, `tools.invoke`। नीचे [ऑपरेटर सहायक विधियाँ](#operator-helper-methods) देखें।

## ऑडिट लेजर RPC

`audit.activity.list` ऑपरेटर क्लाइंट को एजेंट रन, टूल कार्रवाई और ऑप्ट-इन संदेश जीवनचक्र मेटाडेटा का स्थिर नवीनतम-प्रथम दृश्य देता है। इसके लिए
`operator.read` आवश्यक है। क्वेरी 30 दिनों से पुराने रिकॉर्ड को बाहर रखती हैं, और साझा
SQLite लेजर की सीमा 100,000 रिकॉर्ड है। समय-सीमा समाप्त पंक्तियाँ
Gateway स्टार्टअप, प्रति घंटे रखरखाव और बाद के राइट के दौरान हटाई जाती हैं। डेटा मॉडल और गोपनीयता सिमेंटिक्स के लिए
[ऑडिट इतिहास](/hi/gateway/audit) देखें।

- पैरामीटर: वैकल्पिक सटीक `agentId`, `sessionKey`, या `runId`; वैकल्पिक `kind`
  (`"agent_run"`, `"tool_action"`, या `"message"`); वैकल्पिक `status`
  (`"started"`, `"succeeded"`, `"failed"`, `"cancelled"`, `"timed_out"`,
  `"blocked"`, या `"unknown"`); वैकल्पिक संदेश `direction` (`"inbound"` या
  `"outbound"`) और सटीक `channel`; वैकल्पिक समावेशी `after` / `before`
  यूनिक्स-मिलीसेकंड सीमाएँ; `1` से `500` तक वैकल्पिक `limit`; और पिछले पृष्ठ का वैकल्पिक
  स्ट्रिंग `cursor`।
- परिणाम: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`।

नामित V1 परिणाम यूनियन में एजेंट-रन, टूल-कार्रवाई, इनबाउंड-संदेश और आउटबाउंड-संदेश के लिए अलग-अलग स्कीमा हैं। `eventType` डिस्क्रिमिनेटर क्रमशः
`agent_run`, `tool_action`, `inbound_message`, या `outbound_message` है; `kind` और
संदेश `direction` फ़िल्टरिंग तथा प्रदर्शन के लिए उपलब्ध रहते हैं। हर इवेंट में
पूर्णांक `schemaVersion: 1` होता है। संदेश पहचान संदर्भ सटीक
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` प्रारूप का उपयोग करते हैं; चैनल-प्रेषक एक्टर
आईडी भी इसी प्रारूप का उपयोग करती है।

सभी वेरिएंट के लिए `eventType`, `schemaVersion`, `eventId`, `sequence`,
`sourceSequence`, `occurredAt`, `kind`, `action`, `status`, `actor`, और
`redaction` आवश्यक हैं। वेरिएंट फ़ील्ड ये हैं:

| `eventType`        | आवश्यक फ़ील्ड                                                   | वैकल्पिक फ़ील्ड                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`, `runId`; `kind: "agent_run"`                           | `sessionKey`, `sessionId`, `errorCode`                                                                                          |
| `tool_action`      | `agentId`, `runId`; `kind: "tool_action"`                         | `sessionKey`, `sessionId`, `toolCallId`, `toolName`, `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`, `channel`, `conversationKind`, `outcome`  | `agentId`, `runId`, `durationMs`, `resultCount`, पहचान संदर्भ, `reasonCode`, `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`, `channel`, `conversationKind`, `outcome` | `agentId`, `runId`, `durationMs`, `resultCount`, पहचान संदर्भ, `reasonCode`, `deliveryKind`, `failureStage`, `errorCode` |

बंद संदेश एनम ये हैं:

- `conversationKind`: `direct`, `group`, `channel`, या `unknown`।
- इनबाउंड `outcome`: `completed`, `skipped`, या `failed`; वैकल्पिक
  `reasonCode`: `duplicate`, `reply_operation_active`,
  `reply_operation_aborted`, `fast_abort`, `plugin_bound_handled`,
  `plugin_bound_unavailable`, `plugin_bound_declined`, `plugin_bound_error`,
  `before_dispatch_handled`, `acp_dispatch_completed`, `acp_dispatch_failed`,
  `acp_dispatch_empty`, या `acp_dispatch_aborted`।
- आउटबाउंड `outcome`: `sent`, `suppressed`, `failed`, या `unknown`; वैकल्पिक
  `reasonCode`: `cancelled_by_message_sending_hook`,
  `cancelled_by_reply_payload_sending_hook`,
  `empty_after_message_sending_hook`, `empty_after_reply_payload_sending_hook`,
  या `no_visible_payload`। कोई एडाप्टर जो प्लेटफ़ॉर्म पहचान नहीं लौटाता,
  `unknown` होता है, क्योंकि बाहरी दुष्प्रभाव को असिद्ध नहीं किया जा सकता।
- `deliveryKind`: `text`, `media`, या `other`; `failureStage`:
  `platform_send`, `queue`, या `unknown`।

अंतिम फ़ील्ड सहसंबद्ध हैं, स्वतंत्र रूप से वैकल्पिक नहीं:

| प्रकार          | टर्मिनल मैपिंग                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| एजेंट रन        | `started` में कोई `errorCode` नहीं है; प्रत्येक गैर-सफल पूर्ण स्थिति के लिए उसका संगत `run_*` कोड आवश्यक है।                                                                 |
| टूल कार्रवाई      | `started` और सफल स्थिति में कोई `errorCode` नहीं है; प्रत्येक अन्य पूर्ण स्थिति के लिए उसका संगत `tool_*` कोड आवश्यक है।                                                       |
| इनबाउंड संदेश  | सफल = `completed`; अवरुद्ध = `skipped`; विफल = `failed` के साथ `message_processing_failed`। `reasonCode`, मौजूद होने पर, उसी टर्मिनल परिवार से संबंधित होना चाहिए। |
| आउटबाउंड संदेश | सफल = `sent`; अवरुद्ध = `suppressed` के साथ `reasonCode`; विफल = `failed` के साथ `errorCode` और `failureStage`; अज्ञात = `unknown` के साथ `failureStage`।      |

प्रत्येक गतिविधि इवेंट में एक स्थिर इवेंट आईडी, एकदिश रूप से बढ़ता लेजर अनुक्रम,
स्रोत इवेंट अनुक्रम, टाइमस्टैम्प, कर्ता, कार्रवाई, स्थिति, पूर्णांक
`schemaVersion: 1`, और `redaction: "metadata_only"` शामिल होते हैं। रन और टूल रिकॉर्ड के लिए
एजेंट और रन उद्गम आवश्यक हैं और उनमें सत्र उद्गम शामिल हो सकता है। संदेश
रिकॉर्ड में एजेंट और रन आईडी शामिल हो सकती हैं, लेकिन जानबूझकर कभी भी
`sessionKey` या `sessionId` शामिल नहीं होते; इसलिए `sessionKey` क्वेरी फ़िल्टर
केवल रन और टूल पंक्तियों पर लागू होता है। टूल इवेंट में टूल कॉल आईडी और टूल नाम शामिल हो सकते हैं।

संदेश रिकॉर्ड `message.inbound.processed` या
`message.outbound.finished` का उपयोग करते हैं और दिशा, चैनल, वार्तालाप प्रकार,
सामान्यीकृत परिणाम, तथा वैकल्पिक डिलीवरी प्रकार, विफलता चरण, अवधि,
परिणाम संख्या, कारण कोड, और इंस्टॉलेशन-स्थानीय कुंजीबद्ध
खाता/वार्तालाप/संदेश/लक्ष्य छद्मनाम जोड़ते हैं। ये छद्मनाम
सहसंबंध में सहायता करते हैं, लेकिन अनामीकरण नहीं हैं: स्थिति डेटाबेस में उनकी कुंजी होती है,
जबकि RPC और CLI निर्यात में नहीं होती। लेजर प्रॉम्प्ट, संदेश
मुख्य भाग, टूल तर्क, टूल परिणाम, कमांड आउटपुट, या अपरिष्कृत त्रुटि टेक्स्ट संग्रहीत नहीं करता।
रन/टूल `sessionKey` मान अपरिष्कृत सहसंबंध मेटाडेटा बने रहते हैं और उनमें
प्लेटफ़ॉर्म खाता या पीयर आईडी अंतर्निहित हो सकती हैं; संदेश रिकॉर्ड सत्र कुंजियाँ छोड़ देते हैं।

इनबाउंड पंक्तियों के लिए, `durationMs` कोर डिस्पैच से उसके टर्मिनल तक का मापन करता है और
`resultCount` अंतिम रूप दिए गए कतारबद्ध टूल, ब्लॉक, और उत्तर पेलोड की गणना करता है। 
आउटबाउंड पंक्तियों के लिए, `durationMs` डिलीवरी स्वामित्व से अभिस्वीकृति,
डेड लेटर, या समाधान तक की अवधि को समेटता है (जिसमें कतारबद्ध प्रतीक्षा समय शामिल है), और `resultCount`
पहचाने गए भौतिक प्लेटफ़ॉर्म प्रेषणों की गणना करता है। `deliveryKind`, मौजूद होने पर,
हुक और रेंडरिंग के बाद प्रभावी पेलोड का वर्णन करता है; दबाई गई या
क्रैश-अस्पष्ट पंक्तियाँ इसे छोड़ देती हैं।

वर्तमान संदेश कवरेज में कोर
डिस्पैच तक पहुँचने वाले स्वीकृत इनबाउंड संदेश शामिल हैं, जिनमें कोर डुप्लिकेट/टर्मिनल परिणाम भी शामिल हैं। आउटबाउंड कवरेज
साझा टिकाऊ डिलीवरी तक पहुँचने वाले प्रत्येक मूल तार्किक उत्तर पेलोड के लिए
एक टर्मिनल पंक्ति लिखता है; खंडन और अडैप्टर फ़ैन-आउट को `resultCount` में समेकित किया जाता है। कतारबद्ध
पुनः प्रयास योग्य या अस्पष्ट प्रेषण केवल अभिस्वीकृति, डेड
लेटर, या समाधान के बाद रिकॉर्ड किए जाते हैं। उन साझा सीमाओं को बायपास करने वाले
Plugin-स्थानीय और प्रत्यक्ष-प्रेषण पथ अभी शामिल नहीं हैं। सीमित वर्कर कतार सर्वोत्तम-प्रयास
आधारित है और विफलता या संतृप्ति पर रिकॉर्ड छोड़ सकती है, इसलिए यह सतह
हानिरहित अनुपालन संग्रह नहीं है।

रिकॉर्डिंग डिफ़ॉल्ट रूप से चालू रहती है और
[`audit.enabled`](/hi/gateway/configuration-reference#audit) द्वारा नियंत्रित होती है। संदेश रिकॉर्डिंग
अलग से `audit.messages` द्वारा नियंत्रित होती है और इसका डिफ़ॉल्ट `"off"` है। जब
रिकॉर्डिंग अक्षम होती है, तब भी `audit.activity.list` पहले लिखे गए रिकॉर्ड की
समाप्ति तक उन्हें प्रस्तुत करता रहता है।

शिप किए गए `audit.list` अनुरोध, परिणाम, और `AuditEvent` स्कीमा
अपरिवर्तित रहते हैं और केवल एजेंट-रन तथा टूल-कार्रवाई रिकॉर्ड लौटाते हैं। नए ऑपरेटर
क्लाइंट को Gateway द्वारा इसका विज्ञापन किए जाने पर `audit.activity.list` कॉल करना चाहिए। पुराने
Gateway या तो `unknown method: audit.activity.list`, या फिर शिप किए गए संस्करणों में
विधि लुकअप से पहले प्राधिकरण होने के कारण, रीड-स्कोप अनुरोध को `missing scope:
operator.admin` रिपोर्ट कर सकते हैं। बाद वाले को केवल तभी विधि की अनुपस्थिति मानें
जब विधि का विज्ञापन नहीं किया गया हो। इसके बाद क्लाइंट केवल तभी `audit.list`
का पुनः प्रयास कर सकता है, जब उसके फ़िल्टर को संदेश प्रकार, दिशा, या चैनल
समर्थन की आवश्यकता न हो।

टेक्स्ट क्वेरी और सीमित JSON निर्यात के लिए [`openclaw audit`](/hi/cli/audit) का उपयोग करें।

## कार्य लेजर RPC

ऑपरेटर क्लाइंट कार्य लेजर RPC (`packages/gateway-protocol/src/schema/tasks.ts`) के माध्यम से
Gateway पृष्ठभूमि कार्य रिकॉर्ड का निरीक्षण और निरस्तीकरण करते हैं। ये
परिशोधित कार्य सारांश लौटाते हैं, अपरिष्कृत रनटाइम स्थिति नहीं।

- `tasks.list` के लिए `operator.read` आवश्यक है।
  - पैरामीटर: वैकल्पिक `status` (`"queued"`, `"running"`, `"completed"`,
    `"failed"`, `"cancelled"`, या `"timed_out"`) या उन स्थितियों की एक सरणी,
    वैकल्पिक `agentId`, वैकल्पिक `sessionKey`, `1` से
    `500` तक वैकल्पिक `limit`, और वैकल्पिक स्ट्रिंग `cursor`।
  - परिणाम: `{ "tasks": TaskSummary[], "nextCursor"?: string }`।
- `tasks.get` के लिए `operator.read` आवश्यक है।
  - पैरामीटर: `{ "taskId": string }`।
  - परिणाम: `{ "task": TaskSummary }`।
  - अनुपलब्ध कार्य आईडी Gateway का नॉट-फाउंड त्रुटि प्रारूप लौटाती हैं।
- `tasks.cancel` के लिए `operator.write` आवश्यक है।
  - पैरामीटर: `{ "taskId": string, "reason"?: string }`।
  - परिणाम: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`।
  - `found` बताता है कि लेजर में मेल खाता कार्य था या नहीं। `cancelled`
    बताता है कि रनटाइम ने निरस्तीकरण स्वीकार या रिकॉर्ड किया या नहीं।

`TaskSummary` में `id`, `status`, और वैकल्पिक मेटाडेटा शामिल हैं: `kind`,
`runtime`, `title`, `agentId`, `sessionKey`, `childSessionKey`, `ownerKey`,
`runId`, `taskId`, `flowId`, `parentTaskId`, `sourceId`, टाइमस्टैम्प, प्रगति,
टर्मिनल सारांश, और परिशोधित त्रुटि टेक्स्ट। `agentId` कार्य निष्पादित करने वाले एजेंट की
पहचान करता है; `sessionKey` और `ownerKey` अनुरोधकर्ता तथा नियंत्रण
संदर्भ को संरक्षित रखते हैं।

## ऑपरेटर सहायक विधियाँ

- `commands.list` (`operator.read`) किसी एजेंट के लिए रनटाइम कमांड इन्वेंट्री प्राप्त करता है।
  - `agentId` वैकल्पिक है; डिफ़ॉल्ट एजेंट कार्यस्थान पढ़ने के लिए इसे छोड़ दें।
  - `scope` नियंत्रित करता है कि प्राथमिक `name` किस सतह को लक्षित करता है: `text` आरंभिक `/` के बिना प्राथमिक टेक्स्ट कमांड टोकन लौटाता है; उपलब्ध होने पर `native` और डिफ़ॉल्ट `both` पथ प्रदाता-जागरूक नेटिव नाम लौटाते हैं।
  - `textAliases` में `/model` और `/m` जैसे सटीक स्लैश उपनाम होते हैं।
  - `nativeName` में प्रदाता-जागरूक नेटिव कमांड नाम होता है, यदि कोई मौजूद हो।
  - `provider` वैकल्पिक है और केवल नेटिव नामकरण तथा नेटिव Plugin कमांड की उपलब्धता को प्रभावित करता है।
  - `includeArgs=false` प्रतिक्रिया से क्रमबद्ध आर्ग्युमेंट मेटाडेटा हटा देता है।
- `tools.catalog` (`operator.read`) किसी एजेंट के लिए रनटाइम टूल कैटलॉग प्राप्त करता है। प्रतिक्रिया में समूहीकृत टूल और उद्गम मेटाडेटा शामिल होते हैं:
  - `source`: `core` या `plugin`
  - `pluginId`: `source="plugin"` होने पर Plugin स्वामी
  - `optional`: क्या कोई Plugin टूल वैकल्पिक है
- `tools.effective` (`operator.read`) किसी सत्र के लिए रनटाइम-प्रभावी टूल इन्वेंट्री प्राप्त करता है।
  - `sessionKey` आवश्यक है।
  - Gateway कॉलर द्वारा दी गई प्रमाणीकरण या डिलीवरी संदर्भ स्वीकार करने के बजाय सत्र से सर्वर-साइड पर विश्वसनीय रनटाइम संदर्भ प्राप्त करता है।
  - प्रतिक्रिया सक्रिय इन्वेंट्री का सत्र-स्कोप वाला, सर्वर-व्युत्पन्न प्रक्षेपण है, जिसमें कोर, Plugin, चैनल और पहले से खोजे गए MCP सर्वर टूल शामिल होते हैं।
  - `tools.effective` MCP के लिए केवल-पढ़ने योग्य है: यह अंतिम टूल नीति के माध्यम से किसी सक्रिय सत्र के MCP कैटलॉग को प्रक्षेपित कर सकता है, लेकिन MCP रनटाइम नहीं बनाता, ट्रांसपोर्ट से कनेक्ट नहीं करता या `tools/list` जारी नहीं करता। यदि कोई मेल खाता सक्रिय कैटलॉग मौजूद नहीं है, तो प्रतिक्रिया में `mcp-not-yet-connected`, `mcp-not-yet-listed` या `mcp-stale-catalog` जैसी सूचना शामिल हो सकती है।
  - प्रभावी टूल प्रविष्टियाँ `source="core"`, `source="plugin"`, `source="channel"` या `source="mcp"` का उपयोग करती हैं।
- `tools.invoke` (`operator.write`) `/tools/invoke` के समान Gateway नीति पथ से एक उपलब्ध टूल को आह्वान करता है।
  - `name` आवश्यक है। `args`, `sessionKey`, `agentId`, `confirm` और `idempotencyKey` वैकल्पिक हैं।
  - यदि `sessionKey` और `agentId` दोनों मौजूद हैं, तो समाधान किया गया सत्र एजेंट `agentId` से मेल खाना चाहिए।
  - `cron`, `gateway` और `nodes` जैसे केवल-स्वामी कोर रैपर को स्वामी/व्यवस्थापक पहचान (`operator.admin`) की आवश्यकता होती है, भले ही `tools.invoke` स्वयं `operator.write` हो।
  - प्रतिक्रिया SDK-सामना करने वाला एनवेलप है, जिसमें `ok`, `toolName`, वैकल्पिक `output` और टाइप किए गए `error` फ़ील्ड होते हैं। अनुमोदन या नीति अस्वीकृतियाँ Gateway टूल नीति पाइपलाइन को बायपास करने के बजाय पेलोड में `ok:false` लौटाती हैं।
- `skills.status` (`operator.read`) किसी एजेंट के लिए दृश्यमान स्किल इन्वेंट्री प्राप्त करता है।
  - `agentId` वैकल्पिक है; डिफ़ॉल्ट एजेंट कार्यस्थान पढ़ने के लिए इसे छोड़ दें।
  - प्रतिक्रिया में पात्रता, अनुपलब्ध आवश्यकताएँ, कॉन्फ़िगरेशन जाँच और कच्चे गुप्त मान उजागर किए बिना स्वच्छ किए गए इंस्टॉल विकल्प शामिल होते हैं।
- `skills.search` और `skills.detail` (`operator.read`) ClawHub खोज मेटाडेटा लौटाते हैं।
- `skills.upload.begin`, `skills.upload.chunk` और `skills.upload.commit` (`operator.admin`) किसी निजी स्किल संग्रह को इंस्टॉल करने से पहले तैयार करते हैं। यह विश्वसनीय क्लाइंट के लिए एक अलग व्यवस्थापक अपलोड पथ है, सामान्य ClawHub स्किल इंस्टॉल प्रवाह नहीं, और डिफ़ॉल्ट रूप से अक्षम रहता है जब तक कि `skills.install.allowUploadedArchives` सक्षम न हो।
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })` उस स्लग और फ़ोर्स मान से बंधा अपलोड बनाता है।
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })` सटीक डीकोड किए गए ऑफ़सेट पर बाइट जोड़ता है।
  - `skills.upload.commit({ uploadId, sha256? })` अंतिम आकार और SHA-256 सत्यापित करता है। कमिट केवल अपलोड को अंतिम रूप देता है; यह स्किल इंस्टॉल नहीं करता।
  - अपलोड किए गए स्किल संग्रह ऐसे ज़िप संग्रह होते हैं जिनमें `SKILL.md` रूट होता है। संग्रह का आंतरिक डायरेक्टरी नाम कभी भी इंस्टॉल लक्ष्य का चयन नहीं करता।
- `skills.install` (`operator.admin`) के तीन मोड हैं:
  - ClawHub मोड: `{ source: "clawhub", slug, version?, force? }` किसी स्किल फ़ोल्डर को डिफ़ॉल्ट एजेंट कार्यस्थान की `skills/` डायरेक्टरी में इंस्टॉल करता है।
  - अपलोड मोड: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }` किसी कमिट किए गए अपलोड को डिफ़ॉल्ट एजेंट कार्यस्थान की `skills/<slug>` डायरेक्टरी में इंस्टॉल करता है। स्लग और फ़ोर्स मान मूल `skills.upload.begin` अनुरोध से मेल खाने चाहिए। `skills.install.allowUploadedArchives` सक्षम न होने पर अस्वीकृत; यह सेटिंग ClawHub इंस्टॉल को प्रभावित नहीं करती।
  - Gateway इंस्टॉलर मोड: `{ name, installId, timeoutMs? }` Gateway होस्ट पर घोषित `metadata.openclaw.install` क्रिया चलाता है। पुराने क्लाइंट अभी भी `dangerouslyForceUnsafeInstall` भेज सकते हैं; यह फ़ील्ड अप्रचलित है, केवल प्रोटोकॉल संगतता के लिए स्वीकार किया जाता है और अनदेखा किया जाता है। ऑपरेटर-स्वामित्व वाले इंस्टॉल निर्णयों के लिए `security.installPolicy` का उपयोग करें।
- `skills.update` (`operator.admin`) के दो मोड हैं:
  - ClawHub मोड डिफ़ॉल्ट एजेंट कार्यस्थान में एक ट्रैक किए गए स्लग या सभी ट्रैक किए गए ClawHub इंस्टॉल को अपडेट करता है।
  - कॉन्फ़िगरेशन मोड `skills.entries.<skillKey>` मानों को पैच करता है, जैसे `enabled`, `apiKey` और `env`।

### `models.list` दृश्य

`models.list` एक वैकल्पिक `view` पैरामीटर स्वीकार करता है
(`src/agents/model-catalog-visibility.ts`):

- छोड़ा गया या `"default"`: यदि `agents.defaults.modelPolicy.allow` कॉन्फ़िगर किया गया है, तो प्रतिक्रिया अनुमत कैटलॉग होती है, जिसमें `provider/*` प्रविष्टियों के लिए गतिशील रूप से खोजे गए मॉडल शामिल होते हैं। अन्यथा प्रतिक्रिया संपूर्ण Gateway कैटलॉग होती है।
- `"configured"`: पिकर-आकार का व्यवहार। यदि `agents.defaults.modelPolicy.allow` कॉन्फ़िगर किया गया है, तो वही प्रभावी रहता है, जिसमें `provider/*` प्रविष्टियों के लिए प्रदाता-स्कोप वाली खोज शामिल है। अनुमति-सूची के बिना, प्रतिक्रिया स्पष्ट `models.providers.<provider>.models` प्रविष्टियों का उपयोग करती है और केवल तब संपूर्ण कैटलॉग पर वापस जाती है जब कोई कॉन्फ़िगर की गई मॉडल पंक्ति मौजूद न हो।
- `"provider-config"`: स्रोत-निर्मित `models.providers.*.models` इन्वेंट्री, जो पिकर अनुमति-सूचियों से स्वतंत्र है। पंक्तियों में सार्वजनिक मॉडल क्षमताएँ और रूट-जागरूक उपलब्धता शामिल होती हैं, लेकिन प्रदाता एंडपॉइंट, प्रमाणीकरण सामग्री और रनटाइम अनुरोध कॉन्फ़िगरेशन शामिल नहीं होते।
- `"all"`: संपूर्ण Gateway कैटलॉग, जो `agents.defaults.modelPolicy.allow` को बायपास करता है। इसका उपयोग निदान/खोज UI के लिए करें, सामान्य मॉडल पिकर के लिए नहीं।

## निष्पादन अनुमोदन

- जब किसी निष्पादन अनुरोध को अनुमोदन की आवश्यकता होती है, तो Gateway `exec.approval.requested` प्रसारित करता है।
- ऑपरेटर क्लाइंट `exec.approval.resolve` को कॉल करके समाधान करते हैं (`operator.approvals` आवश्यक है)।
- `host=node` के लिए, `exec.approval.request` में `systemRunPlan` (प्रामाणिक `argv`/`cwd`/`rawCommand`/सत्र मेटाडेटा) शामिल होना चाहिए। `systemRunPlan` से रहित अनुरोध अस्वीकृत कर दिए जाते हैं।
- अनुमोदन के बाद, अग्रेषित `node.invoke system.run` कॉल उसी प्रामाणिक `systemRunPlan` को आधिकारिक कमांड/cwd/सत्र संदर्भ के रूप में पुनः उपयोग करते हैं।
- यदि कोई कॉलर तैयारी और अंतिम अनुमोदित `system.run` अग्रेषण के बीच `command`, `rawCommand`, `cwd`, `agentId` या `sessionKey` को बदलता है, तो Gateway बदले हुए पेलोड पर भरोसा करने के बजाय रन को अस्वीकार कर देता है।

## एजेंट डिलीवरी फ़ॉलबैक

- `agent` अनुरोधों में आउटबाउंड डिलीवरी का अनुरोध करने के लिए `deliver=true` शामिल हो सकता है।
- `bestEffortDeliver=false` (डिफ़ॉल्ट) सख्त व्यवहार बनाए रखता है: अनसुलझे या केवल-आंतरिक डिलीवरी लक्ष्य `INVALID_REQUEST` लौटाते हैं।
- `bestEffortDeliver=true` तब केवल-सत्र निष्पादन पर फ़ॉलबैक की अनुमति देता है जब किसी बाहरी रूप से डिलीवर किए जा सकने वाले रूट का समाधान न हो सके (उदाहरण के लिए, आंतरिक/वेबचैट सत्र या अस्पष्ट बहु-चैनल कॉन्फ़िगरेशन)।
- जब डिलीवरी का अनुरोध किया गया हो, तो अंतिम `agent` परिणामों में `result.deliveryStatus` शामिल हो सकता है, जो [`openclaw agent --json --deliver`](/hi/cli/agent#json-delivery-status) के लिए प्रलेखित समान `sent`, `suppressed`, `partial_failed` और `failed` स्थितियों का उपयोग करता है।

## संस्करण निर्धारण

- `PROTOCOL_VERSION`, `MIN_CLIENT_PROTOCOL_VERSION`, `MIN_NODE_PROTOCOL_VERSION` और `MIN_PROBE_PROTOCOL_VERSION`, `packages/gateway-protocol/src/version.ts` में स्थित हैं।
- क्लाइंट `minProtocol` + `maxProtocol` भेजते हैं। ऑपरेटर और UI क्लाइंट को उस सीमा में वर्तमान प्रोटोकॉल शामिल करना चाहिए; वर्तमान क्लाइंट और सर्वर प्रोटोकॉल v4 चलाते हैं।
- `role: "node"` और `client.mode: "node"` दोनों वाले प्रमाणित क्लाइंट N-1 Node प्रोटोकॉल (वर्तमान में v3) का उपयोग कर सकते हैं। हल्के पुनरारंभ प्रोब समान N-1 विंडो का उपयोग करते हैं। इस संगतता विंडो से डिवाइस प्रमाणीकरण, पेयरिंग, स्कोप, कमांड नीति और निष्पादन अनुमोदन अपरिवर्तित रहते हैं। Plugin-स्वामित्व वाली Node क्षमताएँ और कमांड तब तक रोककर रखे जाते हैं जब तक Node वर्तमान प्रोटोकॉल में अपग्रेड न हो जाए, क्योंकि उनकी होस्ट की गई सतहें N-1 अनुबंध का हिस्सा नहीं हैं।
- स्कीमा और मॉडल TypeBox परिभाषाओं से जनरेट किए जाते हैं:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

प्रतिक्रियाएँ टिक-टाइमआउट सीमा के बाद भी मौन रहती हैं, तो क्लाइंट
सॉकेट को कोड `4000` के साथ बंद करता है, प्रत्येक लंबित अनुरोध को अस्वीकार करता है, और पुनः कनेक्ट करता है। यह
पुनः कनेक्ट होने के बाद अस्वीकृत अनुरोधों को दोबारा नहीं चलाता।

## प्रमाणीकरण

- साझा-सीक्रेट Gateway प्रमाणीकरण कॉन्फ़िगर किए गए
  `gateway.auth.mode` (`"none" | "token" | "password" | "trusted-proxy"`) के आधार पर `connect.params.auth.token` या
  `connect.params.auth.password` का उपयोग करता है।
- Tailscale Serve (`gateway.auth.allowTailscale: true`) या नॉन-लूपबैक `gateway.auth.mode: "trusted-proxy"`
  जैसे पहचान-युक्त मोड `connect.params.auth.*` के बजाय अनुरोध हेडर से कनेक्ट
  प्रमाणीकरण जाँच पूरी करते हैं।
- निजी-इनग्रेस `gateway.auth.mode: "none"` साझा-सीक्रेट कनेक्ट प्रमाणीकरण को
  पूरी तरह छोड़ देता है; इस मोड को सार्वजनिक/अविश्वसनीय इनग्रेस पर उपलब्ध न कराएँ।
- पेयरिंग के बाद, Gateway कनेक्शन की भूमिका + स्कोप तक सीमित एक डिवाइस टोकन जारी करता है,
  जो `hello-ok.auth.deviceToken` में लौटाया जाता है। क्लाइंटों को
  किसी भी सफल कनेक्ट के बाद इसे स्थायी रूप से सहेजना चाहिए।
- उस संग्रहीत डिवाइस टोकन से पुनः कनेक्ट करते समय उस टोकन के लिए संग्रहीत
  स्वीकृत स्कोप सेट का भी पुनः उपयोग होना चाहिए। यह पहले से दी गई रीड/प्रोब/स्टेटस पहुँच
  को बनाए रखता है और पुनः कनेक्शन को चुपचाप केवल-एडमिन के अधिक संकीर्ण
  अंतर्निहित स्कोप तक सीमित होने से बचाता है।
- क्लाइंट-साइड कनेक्ट प्रमाणीकरण संयोजन (`packages/gateway-client/src/client.ts` में
  `selectConnectAuth`):
  - `auth.password` स्वतंत्र है और सेट होने पर हमेशा अग्रेषित किया जाता है।
  - `auth.token` को प्राथमिकता क्रम में भरा जाता है: पहले स्पष्ट साझा टोकन,
    फिर स्पष्ट `deviceToken`, फिर प्रति-डिवाइस संग्रहीत टोकन
    (`deviceId` + `role` द्वारा कुंजीबद्ध)।
  - `auth.bootstrapToken` केवल तभी भेजा जाता है जब उपर्युक्त में से किसी ने भी
    `auth.token` का समाधान न किया हो। साझा टोकन या कोई भी समाधान किया गया डिवाइस टोकन इसे रोक देता है।
  - एकबारगी `AUTH_TOKEN_MISMATCH` पुनः प्रयास पर संग्रहीत डिवाइस टोकन का स्वतः-प्रमोशन
    केवल विश्वसनीय एंडपॉइंट तक सीमित है: लूपबैक,
    या पिन किए गए `tlsFingerprint` वाला `wss://`। बिना पिनिंग वाला सार्वजनिक `wss://`
    पात्र नहीं होता।
- अंतर्निर्मित सेटअप-कोड बूटस्ट्रैप विश्वसनीय मोबाइल हैंडऑफ़ के लिए प्राथमिक Node
  `hello-ok.auth.deviceToken` और `hello-ok.auth.deviceTokens` में एक सीमित ऑपरेटर टोकन
  लौटाता है। ऑपरेटर टोकन में नेटिव Talk कॉन्फ़िगरेशन रीड के लिए
  `operator.talk.secrets` शामिल होता है, लेकिन पेयरिंग-म्यूटेशन स्कोप और
  `operator.admin` शामिल नहीं होते।
- जब कोई गैर-बेसलाइन सेटअप-कोड बूटस्ट्रैप स्वीकृति की प्रतीक्षा करता है,
  `PAIRING_REQUIRED` विवरणों में `recommendedNextStep: "wait_then_retry"`,
  `retryable: true`, और `pauseReconnect: false` शामिल होते हैं। अनुरोध स्वीकृत होने या टोकन
  अमान्य होने तक उसी बूटस्ट्रैप टोकन से पुनः कनेक्ट करते रहें।
- `hello-ok.auth.deviceTokens` को केवल तभी स्थायी रूप से सहेजें जब कनेक्ट ने
  `wss://` या लूपबैक/स्थानीय पेयरिंग जैसे विश्वसनीय ट्रांसपोर्ट पर बूटस्ट्रैप
  प्रमाणीकरण का उपयोग किया हो।
- यदि कोई क्लाइंट स्पष्ट `deviceToken` या स्पष्ट `scopes` देता है, तो
  कॉलर द्वारा अनुरोधित वह स्कोप सेट प्रामाणिक बना रहता है; कैश किए गए स्कोप का पुनः उपयोग केवल
  तभी होता है जब क्लाइंट संग्रहीत प्रति-डिवाइस टोकन का पुनः उपयोग कर रहा हो।
- डिवाइस टोकन को `device.token.rotate` और
  `device.token.revoke` के माध्यम से घुमाया/निरस्त किया जा सकता है (`operator.pairing` आवश्यक है)। किसी
  Node या अन्य गैर-ऑपरेटर भूमिका को घुमाने या निरस्त करने के लिए `operator.admin` भी आवश्यक है।
- `device.token.rotate` रोटेशन मेटाडेटा लौटाता है। यह प्रतिस्थापन
  बेयरर टोकन केवल उसी डिवाइस से किए गए उन कॉल के लिए वापस देता है जो पहले ही उस
  डिवाइस टोकन से प्रमाणित हैं, ताकि केवल-टोकन क्लाइंट पुनः कनेक्ट करने से पहले
  अपना प्रतिस्थापन स्थायी रूप से सहेज सकें। साझा/एडमिन रोटेशन बेयरर टोकन वापस नहीं देते।
- टोकन जारी करना, रोटेशन और निरस्तीकरण उस डिवाइस की पेयरिंग प्रविष्टि में दर्ज स्वीकृत भूमिका
  सेट तक सीमित रहते हैं; टोकन म्यूटेशन ऐसी डिवाइस भूमिका का विस्तार या लक्ष्य नहीं बना सकता
  जिसे पेयरिंग स्वीकृति ने कभी प्रदान नहीं किया।
- पेयर किए गए डिवाइस-टोकन सत्रों में, डिवाइस प्रबंधन स्वयं तक सीमित होता है, जब तक कि
  कॉलर के पास `operator.admin` भी न हो: गैर-एडमिन कॉलर केवल अपनी
  डिवाइस प्रविष्टि के ऑपरेटर टोकन को प्रबंधित कर सकते हैं। Node और अन्य गैर-ऑपरेटर टोकन
  प्रबंधन केवल एडमिन के लिए है, भले ही वह कॉलर के अपने डिवाइस का हो।
- `device.token.rotate` और `device.token.revoke` लक्ष्य
  ऑपरेटर टोकन स्कोप सेट की जाँच कॉलर के वर्तमान सत्र स्कोप के विरुद्ध भी करते हैं।
  गैर-एडमिन कॉलर अपने मौजूदा स्कोप से अधिक व्यापक ऑपरेटर टोकन को घुमा या निरस्त नहीं कर सकते।
- प्रमाणीकरण विफलताओं में `error.details.code` और पुनर्प्राप्ति संकेत शामिल होते हैं:
  - `error.details.canRetryWithDeviceToken` (बूलियन)
  - `error.details.recommendedNextStep`: `retry_with_device_token`,
    `update_auth_configuration`, `update_auth_credentials`,
    `wait_then_retry`, `review_auth_configuration`
    (`packages/gateway-protocol/src/connect-error-details.ts`) में से एक।
- `AUTH_TOKEN_MISMATCH` के लिए क्लाइंट व्यवहार:
  - विश्वसनीय क्लाइंट कैश किए गए प्रति-डिवाइस
    टोकन से एक सीमित पुनः प्रयास कर सकते हैं।
  - यदि वह पुनः प्रयास विफल हो, तो स्वचालित पुनः कनेक्ट लूप रोकें और ऑपरेटर
    कार्रवाई का मार्गदर्शन दिखाएँ।
- `AUTH_SCOPE_MISMATCH` का अर्थ है कि डिवाइस टोकन पहचाना गया था, लेकिन वह
  अनुरोधित भूमिका/स्कोप को कवर नहीं करता। इसे खराब टोकन के रूप में प्रस्तुत न करें; ऑपरेटर को
  दोबारा पेयर करने या अधिक संकीर्ण/व्यापक स्कोप अनुबंध स्वीकृत करने के लिए कहें।

## डिवाइस पहचान और पेयरिंग

- Nodes को कीपेयर फ़िंगरप्रिंट से प्राप्त एक स्थिर डिवाइस पहचान (`device.id`)
  शामिल करनी चाहिए।
- Gateways प्रति डिवाइस + भूमिका टोकन जारी करते हैं।
- नई डिवाइस ID के लिए पेयरिंग स्वीकृति आवश्यक है, जब तक कि स्थानीय
  स्वतः-स्वीकृति सक्षम न हो।
- पेयरिंग स्वतः-स्वीकृति प्रत्यक्ष स्थानीय लूपबैक कनेक्ट पर केंद्रित है।
- OpenClaw में विश्वसनीय साझा-सीक्रेट सहायक प्रवाहों के लिए एक संकीर्ण बैकएंड/कंटेनर-स्थानीय
  स्व-कनेक्ट पथ भी है।
- समान-होस्ट टेलनेट या LAN कनेक्ट को अब भी पेयरिंग के लिए रिमोट माना जाता है
  और उन्हें स्वीकृति आवश्यक होती है।
- WS क्लाइंट सामान्यतः `connect` के दौरान `device` पहचान शामिल करते हैं (ऑपरेटर +
  Node)। डिवाइस-रहित ऑपरेटर के एकमात्र अपवाद स्पष्ट विश्वसनीय पथ हैं:
  - केवल-लोकलहोस्ट असुरक्षित
    HTTP संगतता के लिए `gateway.controlUi.allowInsecureAuth=true`।
  - सफल `gateway.auth.mode: "trusted-proxy"` ऑपरेटर Control UI प्रमाणीकरण।
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (आपातकालीन उपाय, गंभीर
    सुरक्षा अवनयन)।
  - आरक्षित आंतरिक सहायक पथ पर प्रत्यक्ष-लूपबैक `gateway-client` बैकएंड RPC।
- डिवाइस पहचान छोड़ने के स्कोप-संबंधी परिणाम होते हैं। जब किसी डिवाइस-रहित
  ऑपरेटर कनेक्शन को स्पष्ट विश्वसनीय पथ से अनुमति दी जाती है, तब भी OpenClaw
  स्वयं-घोषित स्कोप को खाली सेट में साफ़ कर देता है, जब तक कि उस पथ के पास
  नामित स्कोप-संरक्षण अपवाद न हो। इसके बाद स्कोप द्वारा नियंत्रित विधियाँ
  `missing scope` के साथ विफल होती हैं।
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` एक Control UI
  आपातकालीन स्कोप-संरक्षण पथ है। यह मनमाने
  कस्टम बैकएंड या CLI-आकृति वाले WebSocket क्लाइंटों को स्कोप प्रदान नहीं करता।
- आरक्षित प्रत्यक्ष-लूपबैक `gateway-client` बैकएंड सहायक पथ
  केवल आंतरिक स्थानीय कंट्रोल-प्लेन RPC के लिए स्कोप संरक्षित करता है; कस्टम बैकएंड ID को
  यह अपवाद नहीं मिलता।
- सभी कनेक्शनों को सर्वर द्वारा प्रदान किए गए `connect.challenge` नॉन्स पर हस्ताक्षर करना आवश्यक है।

### डिवाइस प्रमाणीकरण माइग्रेशन निदान

उन लेगेसी क्लाइंटों के लिए जो अब भी चैलेंज-पूर्व हस्ताक्षर व्यवहार का उपयोग करते हैं, `connect`
स्थिर `error.details.reason` के साथ `error.details.code` के अंतर्गत `DEVICE_AUTH_*`
विवरण कोड लौटाता है।

सामान्य माइग्रेशन विफलताएँ:

| संदेश                     | details.code                     | details.reason           | अर्थ                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | क्लाइंट ने `device.nonce` नहीं भेजा (या रिक्त भेजा)।     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | क्लाइंट ने पुराने/गलत नॉन्स के साथ हस्ताक्षर किया।            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | हस्ताक्षर पेलोड v2 पेलोड से मेल नहीं खाता।       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | हस्ताक्षरित टाइमस्टैम्प अनुमत विचलन से बाहर है।          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` सार्वजनिक कुंजी के फ़िंगरप्रिंट से मेल नहीं खाता। |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | सार्वजनिक कुंजी का प्रारूप/कैनोनिकलाइज़ेशन विफल हुआ।         |

माइग्रेशन लक्ष्य:

- हमेशा `connect.challenge` की प्रतीक्षा करें।
- सर्वर नॉन्स शामिल करने वाले v2 पेलोड पर हस्ताक्षर करें।
- `connect.params.device.nonce` में वही नॉन्स भेजें।
- वरीय हस्ताक्षर पेलोड `v3` है
  (`packages/gateway-client/src/device-auth.ts` में `buildDeviceAuthPayloadV3`),
  जो डिवाइस/क्लाइंट/भूमिका/स्कोप/टोकन/नॉन्स फ़ील्ड के अतिरिक्त
  `platform` और `deviceFamily` को भी संबद्ध करता है।
- संगतता के लिए पुराने `v2` हस्ताक्षर अब भी स्वीकार किए जाते हैं, लेकिन दोबारा कनेक्ट होने पर
  युग्मित-डिवाइस मेटाडेटा पिनिंग अब भी कमांड नीति नियंत्रित करती है।

## TLS और पिनिंग

- WS कनेक्शन के लिए TLS समर्थित है (`gateway.tls` कॉन्फ़िगरेशन)।
- क्लाइंट वैकल्पिक रूप से `gateway.remote.tlsFingerprint` या CLI
  `--tls-fingerprint` के माध्यम से Gateway प्रमाणपत्र का फ़िंगरप्रिंट पिन कर सकते हैं।
