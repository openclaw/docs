Gateway WS प्रोटोकॉल OpenClaw के लिए एकमात्र नियंत्रण प्लेन और Node ट्रांसपोर्ट है।
ऑपरेटर और Node क्लाइंट (CLI, वेब UI, macOS ऐप, iOS/Android Node,
हेडलेस Node) WebSocket पर कनेक्ट होते हैं और हैंडशेक के समय एक **भूमिका** और **स्कोप** घोषित करते हैं।

## ट्रांसपोर्ट और फ़्रेमिंग

- WebSocket, टेक्स्ट फ़्रेम, JSON पेलोड।
- पहला फ़्रेम **अनिवार्य रूप से** एक `connect` अनुरोध होना चाहिए।
- प्री-कनेक्ट फ़्रेम 64 KiB (`MAX_PREAUTH_PAYLOAD_BYTES`) तक सीमित हैं। हैंडशेक के बाद,
  `hello-ok.policy.maxPayload` और
  `hello-ok.policy.maxBufferedBytes` का पालन करें। डायग्नोस्टिक्स सक्षम होने पर, अत्यधिक बड़े
  इनबाउंड फ़्रेम और धीमे आउटबाउंड बफ़र, Gateway द्वारा फ़्रेम बंद करने या
  छोड़ने से पहले `payload.large` इवेंट उत्सर्जित करते हैं। इन इवेंट में `surface`, बाइट
  आकार, सीमाएँ और एक सुरक्षित कारण कोड होता है; संदेश बॉडी, अटैचमेंट
  सामग्री, रॉ फ़्रेम बाइट, टोकन, कुकी या सीक्रेट कभी नहीं होते।

फ़्रेम के आकार:

- अनुरोध: `{type:"req", id, method, params}`
- प्रतिक्रिया: `{type:"res", id, ok, payload|error}`
- इवेंट: `{type:"event", event, payload, seq?, stateVersion?}`

प्रतिक्रिया त्रुटियाँ `{ code, message, details?, retryable?, retryAfterMs? }` का उपयोग करती हैं।
क्लाइंट को `code` और `details.code` के आधार पर शाखा बनानी चाहिए; `message` मानव-पठनीय रहता है
और जहाँ संगतता नोट अन्यथा न कहे, वहाँ बदल सकता है। विधि-स्तरीय
प्राधिकरण विफलताएँ संरचित गुम-स्कोप विवरण के साथ शीर्ष-स्तरीय `code: "FORBIDDEN"` का उपयोग करती हैं:

- गुम स्कोप: `{ code: "MISSING_SCOPE", missingScope, requiredScopes }`।
  `requiredScopes` अनुरोधित ऑपरेशन के लिए ज्ञात स्कोप का पूरा सेट है।
  पुराने क्लाइंट के लिए लीगेसी `missing scope: <scope>` संदेश बनाए रखा गया है।

क्लाइंट को पहले `details` पढ़ना चाहिए और लीगेसी संदेश का उपयोग केवल संगतता
फ़ॉलबैक के रूप में करना चाहिए। `readMissingScopeError` और `readMissingScopeErrorDetails` को
`@openclaw/gateway-protocol/gateway-error-details` से एक्सपोर्ट किया जाता है; ब्राउज़र-सुरक्षित Gateway क्लाइंट
उन्हें `@openclaw/gateway-client/browser` से पुनः एक्सपोर्ट करता है।

स्कीमा को `@openclaw/gateway-protocol/schema` से `GatewayErrorDetailsSchema`,
`MissingScopeErrorDetailsSchema` के रूप में एक्सपोर्ट किया जाता है।
HTTP स्कोप विफलताएँ `error.details` के अंतर्गत `MISSING_SCOPE` ऑब्जेक्ट को प्रतिबिंबित करती हैं और
HTTP स्थिति `403` का उपयोग करती हैं।

दुष्प्रभाव उत्पन्न करने वाली विधियों के लिए आइडेम्पोटेंसी कुंजियाँ आवश्यक हैं (स्कीमा देखें)।

## हैंडशेक

Gateway एक प्री-कनेक्ट चुनौती भेजता है:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

क्लाइंट `connect` के साथ उत्तर देता है:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway `hello-ok` के साथ प्रतिक्रिया देता है:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`, `features`, `snapshot`, `policy`, और `auth`, सभी
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) के लिए आवश्यक हैं। `auth`
तय की गई भूमिका/स्कोप की रिपोर्ट करता है, भले ही कोई डिवाइस टोकन जारी न किया गया हो (ऊपर दिया
आकार)। `pluginSurfaceUrls` वैकल्पिक है और Plugin सतह नामों (उदा.
`canvas`) को स्कोप किए गए होस्टेड URL से मैप करता है; इसकी समय-सीमा समाप्त हो सकती है, इसलिए Node
नई प्रविष्टि के लिए `{ "surface": "canvas" }` के साथ `node.pluginSurface.refresh` को कॉल करते हैं।
बहिष्कृत `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
पथ समर्थित नहीं है; Plugin सतहों का उपयोग करें।
स्नैपशॉट का वैकल्पिक `appliedConfigHash`, सक्रिय Gateway रनटाइम द्वारा स्वीकार किया गया
रिज़ॉल्व किया हुआ स्रोत-कॉन्फ़िग संशोधन है। क्लाइंट इसकी तुलना
`config.get.configRevisionHash` से करके निर्धारित कर सकते हैं कि किसी नए सहेजे गए कॉन्फ़िग को अभी भी
पुनरारंभ की आवश्यकता है या नहीं। `config.get.hash` कॉन्फ़िग लेखन टकराव गार्ड द्वारा उपयोग किया जाने वाला
रॉ रूट-फ़ाइल संशोधन बना रहता है।

जब Gateway अभी भी स्टार्टअप साइडकार को पूरा कर रहा हो, तब `connect`
`details.reason: "startup-sidecars"` और `retryAfterMs` के साथ पुनः प्रयास योग्य
`UNAVAILABLE` त्रुटि लौटा सकता है। इसे अंतिम हैंडशेक विफलता मानने के बजाय
अपने कनेक्शन बजट के भीतर पुनः प्रयास करें।

डिवाइस टोकन जारी होने पर, `hello-ok.auth` उसे जोड़ता है:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

बिल्ट-इन QR/सेटअप-कोड बूटस्ट्रैप एक मोबाइल हैंडऑफ़ पथ है। एक सफल
बेसलाइन सेटअप-कोड कनेक्शन एक प्राथमिक Node टोकन और एक सीमित
ऑपरेटर टोकन लौटाता है:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

यह ऑपरेटर हैंडऑफ़ जानबूझकर सीमित है: मोबाइल
ऑपरेटर लूप और नेटिव सेटअप शुरू करने के लिए पर्याप्त, जिसमें Talk
कॉन्फ़िग रीड के लिए `operator.talk.secrets` शामिल है, लेकिन कोई पेयरिंग-म्यूटेशन स्कोप और कोई `operator.admin` नहीं। अधिक व्यापक
पेयरिंग/एडमिन पहुँच के लिए अलग स्वीकृत पेयरिंग या टोकन प्रवाह आवश्यक है। `hello-ok.auth.deviceTokens` को
केवल तभी स्थायी रखें जब बूटस्ट्रैप प्रमाणीकरण किसी विश्वसनीय
ट्रांसपोर्ट (`wss://` या लूपबैक/स्थानीय पेयरिंग) पर चला हो।

विश्वसनीय समान-प्रोसेस बैकएंड क्लाइंट (`client.id: "gateway-client"`,
`client.mode: "backend"`) साझा Gateway टोकन/पासवर्ड से प्रमाणीकरण करते समय
प्रत्यक्ष लूपबैक कनेक्शन पर `device` को छोड़ सकते हैं। यह पथ आंतरिक
नियंत्रण-प्लेन RPC (उदा. सबएजेंट सत्र अपडेट) के लिए आरक्षित है और
पुरानी CLI/डिवाइस पेयरिंग बेसलाइन को स्थानीय बैकएंड कार्य अवरुद्ध करने से रोकता है। रिमोट,
ब्राउज़र-उत्पत्ति, Node और स्पष्ट डिवाइस-टोकन/डिवाइस-पहचान क्लाइंट अब भी
सामान्य पेयरिंग और स्कोप-अपग्रेड जाँच से गुजरते हैं।

### वर्कर भूमिका और बंद प्रोटोकॉल

क्लाउड वर्कर Gateway के स्वामित्व वाली,
होस्ट-कुंजी-पिन की गई SSH टनल के माध्यम से एक समर्पित लूपबैक इनग्रेस का उपयोग करते हैं। यह केवल वर्कर पहचान स्वीकार करता है और
सामान्य प्रमाणीकरण, Node इवेंट, ऑपरेटर RPC या Plugin विधियाँ कभी डिस्पैच नहीं करता। एक सख्त `connect`
पर्यावरण, बंडल हैश, स्वामी युग, RPC-सेट संस्करण, समाप्ति और एक नलेबल सत्र से बंधे
स्थिर-अवस्था में हैश किए गए अल्पकालिक क्रेडेंशियल को सत्यापित करता है; यह
वर्तमान संस्करण और फ़ीचर सेट की अलग से जाँच करता है। सफलता पर न्यूनतम
`worker-hello-ok` लौटता है; फ़ीचर नेगोशिएशन सामान्य प्रोटोकॉल
संस्करण से स्वतंत्र है। फ़्रेम 64 KiB से कम रहते हैं, सिवाय तय किए गए `worker.inference.start`
फ़्रेम के, जो 25 MiB तक हो सकता है। बंद अनुमति-सूची में `worker.heartbeat`,
`worker.transcript.commit`, `worker.live-event`, `worker.inference.start`, और
`worker.inference.cancel` शामिल हैं।

ट्रांसक्रिप्ट कमिट स्वामी-युग फ़ेंसिंग, Gateway के स्वामित्व वाली सत्र बाइंडिंग,
बेस-लीफ़ कम्पेयर-एंड-स्वैप और टिकाऊ अनुक्रम रीप्ले का उपयोग करते हैं; Gateway सामान्य
सत्र राइटर के माध्यम से ट्रांसक्रिप्ट प्रविष्टि और पैरेंट ID जनरेट करता है। प्रत्येक RPC पर स्वामित्व और
समाप्ति की पुनः जाँच की जाती है।

### क्लाइंट क्षमताएँ

ऑपरेटर क्लाइंट `connect.params.caps` में वैकल्पिक क्षमताएँ विज्ञापित कर सकते हैं:

- `tool-events`: संरचित टूल जीवनचक्र इवेंट स्वीकार करता है।
- `inline-widgets`: होस्टेड इनलाइन विजेट टूल परिणाम रेंडर कर सकता है।

क्लाइंट क्षमताएँ कनेक्ट किए गए क्लाइंट का वर्णन करती हैं, प्राधिकरण का नहीं। एजेंट टूल आवश्यक क्षमताएँ घोषित कर सकते हैं; Gateway उन टूल को तब तक छोड़ देता है जब तक प्रत्येक आवश्यकता मूल क्लाइंट के `caps` में उपस्थित न हो। चैनल से उत्पन्न रन में कोई Gateway क्लाइंट क्षमता नहीं होती, इसलिए क्षमता-गेटेड टूल तब भी अनुपलब्ध रहते हैं जब टूल नीति उन्हें स्पष्ट रूप से अनुमति देती हो।

### Node कनेक्ट उदाहरण

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "ios-node",
      "version": "1.2.3",
      "platform": "ios",
      "mode": "node"
    },
    "role": "node",
    "scopes": [],
    "caps": ["camera", "canvas", "screen", "location", "voice"],
    "commands": ["camera.snap", "canvas.navigate", "screen.record", "location.get"],
    "permissions": { "camera.capture": true, "screen.record": false },
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-ios/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }

```

Node कनेक्ट करते समय क्षमता दावे घोषित करते हैं:

- `caps`: `camera`, `canvas`, `screen`,
  `location`, `voice`, `talk` जैसी उच्च-स्तरीय श्रेणियाँ।
- `commands`: इनवोक के लिए कमांड अनुमति-सूची।
- `permissions`: सूक्ष्म टॉगल (उदा. `screen.record`, `camera.capture`)।

Gateway इन्हें दावों के रूप में मानता है और सर्वर-साइड अनुमति-सूचियाँ लागू करता है।

## भूमिकाएँ और स्कोप

पूर्ण ऑपरेटर स्कोप मॉडल, अनुमोदन-समय जाँच और साझा-सीक्रेट
अर्थविज्ञान के लिए, [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) देखें।

भूमिकाएँ:

- `operator`: नियंत्रण-प्लेन क्लाइंट (CLI/UI/ऑटोमेशन)।
- `node`: क्षमता होस्ट (कैमरा/स्क्रीन/कैनवास/system.run)।
- `worker`: समर्पित, बंद वर्कर प्रोटोकॉल पर क्लाउड निष्पादन होस्ट।

ऑपरेटर स्कोप (`src/gateway/operator-scopes.ts`), पूर्ण बंद सेट:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`includeSecrets: true` के साथ `talk.config` के लिए `operator.talk.secrets` (या
`operator.admin`) आवश्यक है। सीक्रेट शामिल होने पर, सक्रिय Talk प्रदाता
क्रेडेंशियल को `talk.resolved.config.apiKey` से पढ़ें; `talk.providers.<id>.apiKey`
स्रोत-आकार में रहता है और एक SecretRef ऑब्जेक्ट या संशोधित स्ट्रिंग हो सकता है।

Plugin-पंजीकृत Gateway RPC विधियाँ अपना ऑपरेटर स्कोप माँग सकती हैं,
लेकिन ये आरक्षित कोर प्रीफ़िक्स हमेशा `operator.admin`
(`src/shared/gateway-method-policy.ts`) में रिज़ॉल्व होते हैं: `config.*`, `exec.approvals.*`,
`wizard.*`, `update.*`।

विधि स्कोप केवल पहला गेट है। `chat.send` के माध्यम से पहुँचे कुछ स्लैश कमांड
अधिक सख्त कमांड-स्तरीय जाँच लागू करते हैं: स्थायी `/config set` और
`/config unset` लेखन के लिए `operator.admin` आवश्यक है, यहाँ तक कि उन Gateway क्लाइंट के लिए भी
जिनके पास पहले से निम्न ऑपरेटर स्कोप है।

`node.pair.approve` में आधार
विधि स्कोप (`operator.pairing`) के अतिरिक्त, लंबित अनुरोध के घोषित
`commands` (`src/infra/node-pairing-authz.ts`) के आधार पर एक अतिरिक्त अनुमोदन-समय स्कोप जाँच होती है:

| घोषित कमांड                                                                                                             | आवश्यक स्कोप                       |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| कोई नहीं                                                                                                                          | `operator.pairing`                    |
| सामान्य कमांड                                                                                                             | `operator.pairing` + `operator.write` |
| इसमें `system.run`, `system.run.prepare`, `system.which`, `browser.proxy`, `fs.listDir`, या `system.execApprovals.get/set` शामिल हैं | `operator.pairing` + `operator.admin` |

### क्षमताएँ/कमांड/अनुमतियाँ (Node)

Node कनेक्ट करते समय क्षमता दावे घोषित करते हैं:

- `caps`: उच्च-स्तरीय क्षमता श्रेणियाँ, जैसे `camera`, `canvas`, `screen`,
  `location`, `voice`, और `talk`।
- `commands`: आह्वान के लिए कमांड अनुमति-सूची।
- `permissions`: सूक्ष्म टॉगल (उदा. `screen.record`, `camera.capture`)।

Gateway इन्हें **दावों** के रूप में मानता है और सर्वर-साइड अनुमति-सूचियाँ लागू करता है।
कनेक्ट किए गए Node सफल कनेक्शन या पुनः कनेक्शन के बाद `node.pluginTools.update` के साथ वैकल्पिक एजेंट-दृश्य Plugin या MCP टूल
विवरण प्रकाशित कर सकते हैं। हेडलेस Node होस्ट घोषणात्मक MCP इन्वेंटरी
परिवर्तन लागू करने के लिए पुनः आरंभ होते हैं। यह अपडेट विधि प्रकाशन का एकमात्र मार्ग है; Plugin टूल विवरण
`connect` पैरामीटर में स्वीकार नहीं किए जाते। प्रत्येक विवरण को प्रदाता-सुरक्षित टूल `name` का उपयोग करना और
Node की वर्तमान कमांड अनुमति-सूची में एक `command` का नाम देना आवश्यक है। Gateway युग्मित Node से मिले विवरण
मेटाडेटा पर भरोसा करता है, स्वीकृत कमांड
सतह से बाहर के विवरणों को फ़िल्टर करता है, Node के डिस्कनेक्ट होने पर उन्हें हटाता है, और किसी अन्य Node के कैटलॉग को
बदलने के ऑपरेटर प्रयासों को अस्वीकार करता है। Node द्वारा प्रकाशित विवरणों को अनदेखा करने के लिए `gateway.nodes.pluginTools.enabled: false`
सेट करें।

कनेक्ट किए गए Node होस्ट अपनी संपूर्ण Skills प्रतिस्थापन कैटलॉग
`node.skills.update` के साथ प्रकाशित करते हैं। यह Node-भूमिका विधि Node Skills प्रकाशन का एकमात्र
मार्ग है; Skills को `connect` पैरामीटर में स्वीकार नहीं किया जाता। प्रत्येक विवरण में एक
सुरक्षित नाम, विवरण और सीमित `SKILL.md` सामग्री होती है। Gateway उस
सामग्री को सामान्य Skills लोडर से पार्स करता है, Node के कनेक्ट रहने के दौरान उसे एजेंट Skills स्नैपशॉट में
शामिल करता है, और डिस्कनेक्ट होने पर हटा देता है। Node द्वारा प्रकाशित Skills को अनदेखा करने के लिए
`gateway.nodes.skills.enabled: false` सेट करें।

## उपस्थिति

- `system-presence` डिवाइस पहचान के आधार पर कुंजीबद्ध प्रविष्टियाँ लौटाता है, जिनमें
  `deviceId`, `roles`, और `scopes` शामिल हैं, ताकि UI प्रत्येक डिवाइस के लिए एक पंक्ति दिखा सकें, भले ही
  वह ऑपरेटर और Node दोनों के रूप में कनेक्ट हो।
- `node.list` में वैकल्पिक `lastSeenAtMs` और `lastSeenReason` शामिल होते हैं। कनेक्ट किए गए
  Node कारण `connect` के साथ वर्तमान कनेक्शन समय की रिपोर्ट करते हैं; युग्मित Node
  विश्वसनीय Node ईवेंट के माध्यम से स्थायी पृष्ठभूमि उपस्थिति की रिपोर्ट भी कर सकते हैं।

मूल macOS Node सीमित इनपुट निष्क्रियता समय वाले प्रमाणित `node.presence.activity` ईवेंट भी
भेज सकते हैं। Gateway अपनी घड़ी पर गतिविधि टाइमस्टैम्प प्राप्त करता है,
`node.list` और `node.describe` के माध्यम से सबसे नवीन कनेक्टेड Mac को उजागर करता है,
और पढ़ने के दायरे वाले क्लाइंट को `node.presence` अपडेट प्रसारित करता है।
चयन, गोपनीयता, मॉडल संदर्भ और सूचना-रूटिंग व्यवहार के लिए
[सक्रिय कंप्यूटर उपस्थिति](/hi/nodes/presence) देखें।

### Node पृष्ठभूमि सक्रियता ईवेंट

Node यह दर्ज करने के लिए `event: "node.presence.alive"` के साथ `node.event` कॉल करते हैं कि
पृष्ठभूमि वेक के दौरान कोई युग्मित Node सक्रिय था, उसे कनेक्टेड चिह्नित किए बिना:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` एक बंद एनम है: `background`, `silent_push`, `bg_app_refresh`,
`significant_location`, `manual`, `connect`। अज्ञात मान
`background` (`src/shared/node-presence.ts`) में सामान्यीकृत होते हैं। ईवेंट केवल
प्रमाणित Node डिवाइस सत्रों के लिए स्थायी रहता है; डिवाइस-रहित या अयुग्मित सत्र
`handled: false` लौटाते हैं।

सफल Gateway एक संरचित परिणाम लौटाते हैं:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

पुराने Gateway `node.event` के लिए केवल `{ "ok": true }` लौटा सकते हैं; इसे
स्वीकृत RPC मानें, स्थायी उपस्थिति संग्रहण नहीं।

## प्रसारण ईवेंट का दायरा

सर्वर से पुश किए गए प्रसारण ईवेंट दायरे द्वारा नियंत्रित होते हैं, ताकि युग्मन-दायरे वाले या केवल-Node
सत्र निष्क्रिय रूप से सत्र सामग्री प्राप्त न करें
(`src/gateway/server-broadcast.ts`):

- चैट, एजेंट और टूल-परिणाम फ़्रेम (स्ट्रीम किए गए `agent` ईवेंट, टूल-परिणाम
  ईवेंट) के लिए कम-से-कम `operator.read` आवश्यक है। इसके बिना सत्र इन
  फ़्रेम को पूरी तरह छोड़ देते हैं।
- Plugin-परिभाषित `plugin.*` प्रसारण डिफ़ॉल्ट रूप से `operator.write` या
  `operator.admin` तक सीमित होते हैं; स्पष्ट प्रविष्टियाँ, जैसे
  `plugin.approval.requested` / `plugin.approval.resolved`, इसके बजाय
  `operator.approvals` का उपयोग करती हैं।
- स्थिति/ट्रांसपोर्ट ईवेंट (`heartbeat`, `presence`, `tick`, कनेक्ट/डिस्कनेक्ट
  जीवनचक्र) अप्रतिबंधित रहते हैं, ताकि प्रत्येक प्रमाणित सत्र
  ट्रांसपोर्ट की स्थिति देख सके।
- अज्ञात प्रसारण ईवेंट परिवार डिफ़ॉल्ट रूप से दायरे द्वारा नियंत्रित (विफलता पर बंद)
  होते हैं, जब तक कोई पंजीकृत हैंडलर उन्हें स्पष्ट रूप से शिथिल न करे।

प्रत्येक क्लाइंट कनेक्शन अपनी प्रति-क्लाइंट अनुक्रम संख्या बनाए रखता है, इसलिए अलग-अलग क्लाइंट द्वारा
ईवेंट स्ट्रीम के अलग-अलग दायरा-फ़िल्टर किए गए उपसमुच्चय देखने पर भी उस सॉकेट पर प्रसारण
एकदिश बढ़ते क्रम में रहते हैं।

    - `web.login.wait` उस प्रवाह के पूर्ण होने की प्रतीक्षा करता है और सफलता पर चैनल आरंभ करता है।
    - `push.test` किसी पंजीकृत iOS Node को परीक्षण APNs पुश भेजता है।
    - `voicewake.get` संग्रहीत वेक-वर्ड ट्रिगर लौटाता है।
    - `voicewake.set` वेक-वर्ड ट्रिगर अपडेट करता है और परिवर्तन प्रसारित करता है।

  </Accordion>

  <Accordion title="Plugin प्रबंधन">
    - `plugins.list` (`operator.read`) स्थापित Plugin इन्वेंटरी के साथ स्थानीय रूप से चयनित आधिकारिक विकल्प, निदान और यह जानकारी लौटाता है कि वर्तमान इंस्टॉल मोड परिवर्तन की अनुमति देता है या नहीं।
    - `plugins.search` (`operator.read`) इंस्टॉल किए जा सकने वाले ClawHub कोड-Plugin और बंडल-Plugin परिवार खोजता है। गैर-रिक्त `query` और 1 से 100 तक वैकल्पिक `limit` पास करें।
    - `plugins.install` (`operator.admin`) या तो `{ source: "official", pluginId }` वाली आधिकारिक कैटलॉग प्रविष्टि या `{ source: "clawhub", packageName, version?, acknowledgeClawHubRisk? }` वाला ClawHub पैकेज इंस्टॉल करता है। ClawHub इंस्टॉल Gateway विश्वास, अखंडता और इंस्टॉल-नीति जाँच बनाए रखते हैं। सफल इंस्टॉल के लिए Gateway पुनः आरंभ आवश्यक है।
    - `plugins.setEnabled` (`operator.admin`) `{ pluginId, enabled }` के साथ एक स्थापित Plugin की सक्षम नीति बदलता है। प्रतिक्रिया में अपडेट की गई कैटलॉग प्रविष्टि, पुनः आरंभ मेटाडेटा और स्लॉट-चयन संबंधी चेतावनियाँ शामिल होती हैं।
    - `plugins.uninstall` (`operator.admin`) `{ pluginId }` के साथ बाहरी रूप से स्थापित एक Plugin हटाता है: कॉन्फ़िगरेशन संदर्भ, इंस्टॉल रिकॉर्ड और प्रबंधित फ़ाइलें। बंडल किए गए Plugin अनइंस्टॉल नहीं किए जा सकते, केवल अक्षम किए जा सकते हैं। प्रतिक्रिया निष्कासन कार्रवाइयाँ सूचीबद्ध करती है और हमेशा Gateway पुनः आरंभ आवश्यक होता है।

  </Accordion>

  <Accordion title="संदेश और लॉग">
    - `send` चैट रनर के बाहर चैनल/खाता/थ्रेड-लक्षित प्रेषण के लिए प्रत्यक्ष आउटबाउंड-डिलीवरी RPC है।
    - `logs.tail` कर्सर/सीमा और अधिकतम-बाइट नियंत्रणों के साथ कॉन्फ़िगर किया गया Gateway फ़ाइल-लॉग टेल लौटाता है।

  </Accordion>

  <Accordion title="ऑपरेटर टर्मिनल">
    - `terminal.open` किसी स्पष्ट `agentId` या डिफ़ॉल्ट एजेंट के लिए होस्ट PTY शुरू करता है और निर्धारित एजेंट, कार्यशील डायरेक्टरी, शेल तथा परिसीमन स्थिति लौटाता है।
    - `terminal.input`, `terminal.resize`, और `terminal.close` केवल कॉल करने वाले कनेक्शन के स्वामित्व वाले सत्रों पर काम करते हैं।
    - `terminal.upload` अधिकतम 16 MiB की एक base64 फ़ाइल स्वीकार करता है, उसे सत्र के Gateway या युग्मित-Node होस्ट पर एक निजी 24-घंटे की अस्थायी डायरेक्टरी में रखता है और निरपेक्ष पथ लौटाता है। कॉलर को फिर भी उस पथ को पेस्ट करना या किसी अन्य तरीके से उपयोग करना आवश्यक है; RPC कभी भी टर्मिनल इनपुट नहीं लिखता या कोई कमांड निष्पादित नहीं करता।
    - `terminal.data` और `terminal.exit` ईवेंट केवल सत्र के स्वामी कनेक्शन पर स्ट्रीम होते हैं।
    - जिन सत्रों का कनेक्शन टूट जाता है, उन्हें समाप्त नहीं बल्कि अलग किया जाता है: वे `gateway.terminal.detachedSessionTimeoutSeconds` (डिफ़ॉल्ट 300; `0` डिस्कनेक्ट होने पर समाप्त करने की सुविधा पुनर्स्थापित करता है) तक दोबारा संलग्न किए जा सकते हैं, जबकि हालिया आउटपुट सीमित सर्वर-साइड बफ़र में संचित होता है।
    - `terminal.list` संलग्न किए जा सकने वाले सत्र लौटाता है; `terminal.attach` किसी सक्रिय या अलग किए गए सत्र को कॉल करने वाले कनेक्शन से दोबारा बाँधता है और रीप्ले बफ़र लौटाता है (tmux-शैली का अधिग्रहण — पिछले सक्रिय स्वामी को कारण `detached` सहित `terminal.exit` प्राप्त होता है); `terminal.text` बिना संलग्न हुए बफ़र को सादे टेक्स्ट के रूप में पढ़ता है।
    - प्रत्येक टर्मिनल विधि के लिए `operator.admin` आवश्यक है; `gateway.terminal.enabled` स्पष्ट रूप से true होना चाहिए। पूरी तरह सैंडबॉक्स किए गए एजेंट अस्वीकार किए जाते हैं और एजेंट नीति में बदलाव मौजूदा तथा प्रगतिरत PTY को बंद कर देता है, जिसमें अलग किए गए PTY भी शामिल हैं।

  </Accordion>

  <Accordion title="वार्तालाप और TTS">
    - `talk.catalog` वाक्, स्ट्रीमिंग ट्रांसक्रिप्शन और रियलटाइम वॉइस के लिए केवल-पठन Talk प्रदाता कैटलॉग लौटाता है: कैनोनिकल प्रदाता आईडी, रजिस्ट्री उपनाम, लेबल, कॉन्फ़िगर की गई स्थिति, वैकल्पिक समूह-स्तरीय `ready` परिणाम, उपलब्ध मॉडल/वॉइस आईडी, कैनोनिकल मोड, ट्रांसपोर्ट, ब्रेन रणनीतियाँ और रियलटाइम ऑडियो/क्षमता फ़्लैग; यह प्रदाता सीक्रेट लौटाए या वैश्विक कॉन्फ़िगरेशन बदले बिना ऐसा करता है। मौजूदा Gateway रनटाइम प्रदाता चयन लागू करने के बाद `ready` सेट करते हैं; पुराने Gateway में इसकी अनुपस्थिति को असत्यापित मानें।
    - `talk.config` प्रभावी Talk कॉन्फ़िगरेशन पेलोड लौटाता है; `includeSecrets` के लिए `operator.talk.secrets` (या `operator.admin`) आवश्यक है।
    - `talk.session.create`, `realtime/gateway-relay`, `transcription/gateway-relay`, या `stt-tts/managed-room` के लिए Gateway-स्वामित्व वाला Talk सत्र बनाता है। `stt-tts/managed-room` के लिए, `sessionKey` देने वाले `operator.write` कॉलर को सीमित सत्र-कुंजी दृश्यता हेतु `spawnedBy` भी देना होगा; बिना दायरे के `sessionKey` निर्माण और `brain: "direct-tools"` के लिए `operator.admin` आवश्यक है।
    - `talk.session.join` प्रबंधित-कक्ष सत्र टोकन को सत्यापित करता है, आवश्यकता के अनुसार `session.ready` या `session.replaced` उत्सर्जित करता है और कक्ष/सत्र मेटाडेटा के साथ हालिया Talk ईवेंट लौटाता है, लेकिन कभी भी प्लेनटेक्स्ट टोकन या उसका हैश नहीं लौटाता।
    - `talk.session.appendAudio` Gateway-स्वामित्व वाले रियलटाइम रिले और ट्रांसक्रिप्शन सत्रों में base64 PCM इनपुट ऑडियो जोड़ता है।
    - `talk.session.startTurn`, `talk.session.endTurn`, और `talk.session.cancelTurn` स्थिति साफ़ होने से पहले पुराने टर्न को अस्वीकार करते हुए प्रबंधित-कक्ष टर्न जीवनचक्र संचालित करते हैं।
    - `talk.session.cancelOutput` सहायक का ऑडियो आउटपुट रोकता है, मुख्यतः Gateway रिले सत्रों में VAD-नियंत्रित हस्तक्षेप के लिए।
    - `talk.session.submitToolResult` Gateway-स्वामित्व वाले रियलटाइम रिले सत्र द्वारा उत्सर्जित प्रदाता टूल कॉल को पूरा करता है। अनुरोध प्रदाता ब्रिज द्वारा उपलब्ध कराए गए किसी भी एसिंक्रोनस पूर्णता संकेत की प्रतीक्षा करता है; विफल सबमिशन संबद्ध रन को सक्रिय रखते हैं और सफल टूल-परिणाम ईवेंट उत्सर्जित नहीं करते। अंतरिम टूल आउटपुट के लिए `options: { willContinue: true }` दें, या जब प्रदाता ब्रिज दमन समर्थन घोषित करता हो और परिणाम से दूसरी प्रतिक्रिया शुरू नहीं होनी चाहिए, तब `options: { suppressResponse: true }` दें।
    - `talk.session.steer` Gateway-स्वामित्व वाले एजेंट-समर्थित Talk सत्र में सक्रिय-रन वॉइस नियंत्रण भेजता है: `{ sessionId, text, mode? }`, जहाँ `mode`, `status`, `steer`, `cancel`, या `followup` होता है; मोड न देने पर उसे बोले गए टेक्स्ट से वर्गीकृत किया जाता है।
    - `talk.session.close` Gateway-स्वामित्व वाले रिले, ट्रांसक्रिप्शन या प्रबंधित-कक्ष सत्र को बंद करता है और अंतिम Talk ईवेंट उत्सर्जित करता है।
    - `talk.mode` WebChat/Control UI क्लाइंट के लिए मौजूदा Talk मोड स्थिति सेट/प्रसारित करता है।
    - `talk.client.create`, `webrtc` या `provider-websocket` का उपयोग करके क्लाइंट-स्वामित्व वाला रियलटाइम प्रदाता सत्र बनाता या फिर से शुरू करता है, जबकि Gateway क्रेडेंशियल, निर्देश, टूल नीति और लौटाए गए `voiceSessionId` का स्वामी रहता है। क्लाइंट एक कॉल के दौरान प्रदाता ट्रांसपोर्ट बदलते समय `sessionKey` देते हैं और `voiceSessionId` का पुनः उपयोग करते हैं।
    - `talk.client.transcript` सामान्य एजेंट सत्र में एक अंतिम रूप दिया गया `{ role, text }` आइटम जोड़ता है। आवश्यक `entryId`, `voiceSessionId` के भीतर आइडेम्पोटेंट है; पुनः प्रयास ट्रांसक्रिप्ट संदेशों की नकल नहीं करते।
    - `talk.client.close` लंबित ट्रांसक्रिप्ट लेखन के बाद तार्किक वॉइस सत्र बंद करता है। बंद करना आइडेम्पोटेंट है और यह सत्र के अंतिम गैर-WebChat चैनल को केवल-म्यूटेशन कॉल सारांश भेज सकता है।
    - `talk.client.toolCall` क्लाइंट-स्वामित्व वाले रियलटाइम ट्रांसपोर्ट को प्रदाता टूल कॉल Gateway नीति को अग्रेषित करने देता है। पहला समर्थित टूल `openclaw_agent_consult` है; क्लाइंट को रन आईडी मिलती है और वे प्रदाता-विशिष्ट टूल परिणाम सबमिट करने से पहले सामान्य चैट जीवनचक्र ईवेंट की प्रतीक्षा करते हैं। वॉइस से जुड़ी उच्च-प्रभाव वाली कार्रवाइयाँ तब तक `VOICE_CONFIRMATION_REQUIRED:<id>` लौटाती हैं, जब तक बाद में अंतिम रूप दिया गया उपयोगकर्ता कथन ठीक उसी कार्रवाई की स्पष्ट पुष्टि न करे और अगला परामर्श `confirmationId` उपलब्ध न कराए।
    - `talk.client.steer` क्लाइंट-स्वामित्व वाले रियलटाइम ट्रांसपोर्ट के लिए सक्रिय-रन वॉइस नियंत्रण भेजता है। Gateway `sessionKey` से सक्रिय एम्बेडेड रन निर्धारित करता है और निर्देशन को चुपचाप छोड़ने के बजाय संरचित स्वीकृत/अस्वीकृत परिणाम लौटाता है।
    - `talk.event` रियलटाइम, ट्रांसक्रिप्शन, STT/TTS, प्रबंधित-कक्ष, टेलीफ़ोनी और मीटिंग अडैप्टर के लिए एकमात्र Talk ईवेंट चैनल है।
    - `talk.speak` सक्रिय Talk वाक् प्रदाता के माध्यम से वाक् संश्लेषित करता है।
    - `tts.status` TTS सक्षम स्थिति, सक्रिय प्रदाता, फ़ॉलबैक प्रदाता और प्रदाता कॉन्फ़िगरेशन स्थिति लौटाता है।
    - `tts.providers` दृश्यमान TTS प्रदाता सूची लौटाता है।
    - `tts.enable` और `tts.disable` TTS प्राथमिकता स्थिति को टॉगल करते हैं।
    - `tts.setProvider` पसंदीदा TTS प्रदाता को अपडेट करता है।
    - `tts.convert` एक-बार टेक्स्ट-टू-स्पीच रूपांतरण चलाता है।
    - `tts.speak` (`operator.write`) कॉन्फ़िगर की गई सामान्य TTS प्रदाता शृंखला से गैर-रिक्त `text` को रेंडर करता है और एक संपूर्ण क्लिप को `audioBase64` के रूप में इनलाइन लौटाता है, साथ ही `provider` तथा वैकल्पिक `outputFormat`, `mimeType`, और `fileExtension` मेटाडेटा भी लौटाता है। `tts.convert` के विपरीत, यह Gateway-स्थानीय पथ नहीं लौटाता; `talk.speak` के विपरीत, इसे Talk प्रदाता की आवश्यकता नहीं होती। `messages.tts.maxTextLength` से अधिक टेक्स्ट `INVALID_REQUEST` लौटाता है; संश्लेषण विफलताएँ `UNAVAILABLE` लौटाती हैं।

  </Accordion>

  <Accordion title="सीक्रेट, कॉन्फ़िगरेशन, अपडेट और विज़ार्ड">
    - `secrets.reload` सक्रिय SecretRefs को फिर से निर्धारित करता है और स्वामी-सजग रनटाइम स्थिति को परमाण्विक रूप से प्रकाशित करता है। योग्य स्वामी विफलताएँ `warningCount` सहित शीत या पुराने अवक्रमण के रूप में प्रकाशित हो सकती हैं; सख्त या अमैप की गई विफलताएँ रीलोड को अस्वीकार करती हैं और सक्रिय स्नैपशॉट सुरक्षित रखती हैं।
    - `secrets.resolve` किसी विशिष्ट कमांड/लक्ष्य समूह के लिए कमांड-लक्ष्य सीक्रेट असाइनमेंट निर्धारित करता है।
    - `config.get` मौजूदा ऑन-डिस्क कॉन्फ़िगरेशन स्नैपशॉट, रॉ रूट-फ़ाइल `hash`, निर्धारित `configRevisionHash`, और सक्रिय Gateway रनटाइम द्वारा स्वीकृत निर्धारित संशोधन के लिए वैकल्पिक `appliedConfigHash` लौटाता है।
    - `config.set` सत्यापित कॉन्फ़िगरेशन पेलोड लिखता है।
    - `config.patch` आंशिक कॉन्फ़िगरेशन अपडेट को मर्ज करता है। विनाशकारी ऐरे प्रतिस्थापन के लिए प्रभावित पथ का `replacePaths` में होना आवश्यक है; ऐरे प्रविष्टियों के अंतर्गत नेस्टेड ऐरे, `agents.list[].skills` जैसे `[]` पथों का उपयोग करते हैं।
    - `config.apply` पूरे कॉन्फ़िगरेशन पेलोड को सत्यापित करके प्रतिस्थापित करता है।
    - `config.schema` Control UI और CLI टूलिंग द्वारा उपयोग किया जाने वाला सक्रिय कॉन्फ़िगरेशन स्कीमा पेलोड लौटाता है: स्कीमा, `uiHints`, संस्करण, जनरेशन मेटाडेटा तथा लोड किए जा सकने पर Plugin + चैनल स्कीमा मेटाडेटा। इसमें UI के समान लेबल/सहायता टेक्स्ट से `title` / `description` मेटाडेटा शामिल होता है, जिसमें मेल खाने वाला फ़ील्ड दस्तावेज़ उपलब्ध होने पर नेस्टेड ऑब्जेक्ट, वाइल्डकार्ड, ऐरे-आइटम और `anyOf` / `oneOf` / `allOf` संयोजन शाखाएँ भी शामिल हैं।
    - `config.schema.lookup` एक कॉन्फ़िगरेशन पथ के लिए पथ-सीमित लुकअप पेलोड लौटाता है: सामान्यीकृत पथ, उथला स्कीमा Node, मेल खाता संकेत + `hintPath`, वैकल्पिक `reloadKind`, और UI/CLI ड्रिल-डाउन के लिए निकटतम चाइल्ड सारांश। `reloadKind`, `restart`, `hot`, या `none` (`src/config/schema.ts`) में से एक है और अनुरोधित पथ के लिए Gateway कॉन्फ़िगरेशन रीलोड प्लानर को प्रतिबिंबित करता है। लुकअप स्कीमा Node उपयोगकर्ता-दृश्यमान दस्तावेज़ और सामान्य सत्यापन फ़ील्ड (`title`, `description`, `type`, `enum`, `const`, `format`, `pattern`, संख्यात्मक/स्ट्रिंग/ऐरे/ऑब्जेक्ट सीमाएँ, `additionalProperties`, `deprecated`, `readOnly`, `writeOnly`) बनाए रखते हैं। चाइल्ड सारांश `key`, सामान्यीकृत `path`, `type`, `required`, `hasChildren`, वैकल्पिक `reloadKind`, और मेल खाते `hint` / `hintPath` उपलब्ध कराते हैं।
    - `update.run` Gateway अपडेट प्रवाह चलाता है और केवल अपडेट सफल होने पर पुनः आरंभ निर्धारित करता है; सत्र वाले कॉलर `continuationMessage` शामिल कर सकते हैं, ताकि स्टार्टअप पुनः आरंभ निरंतरता कतार के माध्यम से एजेंट का एक अनुवर्ती टर्न फिर से शुरू करे। नियंत्रण तल से पैकेज-मैनेजर अपडेट और पर्यवेक्षित git-checkout अपडेट, सक्रिय Gateway के भीतर पैकेज ट्री बदलने या checkout/build आउटपुट संशोधित करने के बजाय अलग किए गए प्रबंधित-सेवा हैंडऑफ़ का उपयोग करते हैं। शुरू किया गया हैंडऑफ़ `result.reason: "managed-service-handoff-started"` और `handoff.status: "started"` सहित `ok: true` लौटाता है। उसी Gateway प्रक्रिया द्वारा संभाला गया दूसरा समवर्ती `update.run`, `result.reason: "managed-service-handoff-already-running"` और `handoff.status: "already-running"` सहित `ok: false` लौटाता है; उसकी निरंतरता स्वीकार नहीं की जाती, इसलिए सक्रिय अपडेट पूरा होने के बाद कॉलर पुनः प्रयास कर सकता है। स्टैंडअलोन CLI अपडेटर और प्रतिस्थापन Gateway प्रक्रियाएँ इस प्रक्रिया-स्थानीय सुरक्षा से बाहर हैं। अनुपलब्ध या विफल हैंडऑफ़ `managed-service-handoff-unavailable` या `managed-service-handoff-failed` सहित `ok: false` लौटाते हैं, साथ ही मैन्युअल शेल अपडेट आवश्यक होने पर `handoff.command` भी लौटाते हैं। अनुपलब्ध का अर्थ है कि OpenClaw के पास सुरक्षित सुपरवाइज़र सीमा या टिकाऊ सेवा पहचान नहीं है, जैसे systemd के लिए `OPENCLAW_SYSTEMD_UNIT`। शुरू किए गए हैंडऑफ़ के दौरान पुनः आरंभ सेंटिनल कुछ समय के लिए `stats.reason: "restart-health-pending"` रिपोर्ट कर सकता है; निरंतरता तब तक विलंबित रहती है जब तक CLI पुनः आरंभ किए गए Gateway को सत्यापित करके अंतिम `ok` सेंटिनल नहीं लिख देता।
    - `update.status` नवीनतम अपडेट पुनः आरंभ सेंटिनल को रीफ़्रेश करके लौटाता है, जिसमें उपलब्ध होने पर पुनः आरंभ के बाद चल रहा संस्करण भी शामिल होता है।
    - `wizard.start`, `wizard.next`, `wizard.status`, और `wizard.cancel` WS RPC पर ऑनबोर्डिंग विज़ार्ड उपलब्ध कराते हैं।

  </Accordion>

  <Accordion title="एजेंट और वर्कस्पेस सहायक">
    - `agents.list` कॉन्फ़िगर की गई एजेंट प्रविष्टियाँ लौटाता है, जिनमें प्रभावी मॉडल और रनटाइम मेटाडेटा शामिल हैं।
    - `agents.create`, `agents.update`, और `agents.delete` एजेंट रिकॉर्ड और वर्कस्पेस वायरिंग प्रबंधित करते हैं।
    - `agents.files.list`, `agents.files.get`, और `agents.files.set` किसी एजेंट के लिए उपलब्ध कराई गई बूटस्ट्रैप वर्कस्पेस फ़ाइलें प्रबंधित करते हैं।
    - `audit.activity.list` संस्करणित, केवल-मेटाडेटा गतिविधि लेज़र लौटाता है; `audit.list` संगतता-सुरक्षित रन/टूल RPC बना रहता है।
    - `agents.workspace.list` और `agents.workspace.get` (`operator.read`) [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) में वर्णित विश्वसनीय ऑपरेटर डोमेन के क्लाइंट के लिए एजेंट की वर्कस्पेस डायरेक्टरी की केवल-पढ़ने योग्य, पृष्ठांकित ब्राउज़िंग उपलब्ध कराते हैं। अनुरोध केवल वर्कस्पेस-सापेक्ष पथ स्वीकार करते हैं; पठन रियलपाथ किए गए वर्कस्पेस रूट तक सीमित रहता है (सिमलिंक और हार्डलिंक पलायन अस्वीकार किए जाते हैं), आकार-सीमित होता है, और UTF-8 टेक्स्ट तथा सामान्य छवि प्रकारों (base64) तक सीमित होता है। प्रतिक्रियाएँ होस्ट वर्कस्पेस पथ उजागर नहीं करतीं। इस नेमस्पेस में कोई लेखन कार्रवाई नहीं है।
    - `tasks.list`, `tasks.get`, और `tasks.cancel` SDK और ऑपरेटर क्लाइंट के लिए Gateway कार्य लेज़र उपलब्ध कराते हैं। नीचे [कार्य लेज़र RPC](#task-ledger-rpcs) देखें।
    - `artifacts.list`, `artifacts.get`, और `artifacts.download` स्पष्ट `sessionKey`, `runId`, या `taskId` स्कोप के लिए ट्रांस्क्रिप्ट से प्राप्त आर्टिफ़ैक्ट सारांश और डाउनलोड उपलब्ध कराते हैं। रन और कार्य क्वेरी सर्वर-साइड पर स्वामी सत्र का समाधान करती हैं और केवल मेल खाते उद्गम वाला ट्रांस्क्रिप्ट मीडिया लौटाती हैं; असुरक्षित या स्थानीय URL स्रोत सर्वर-साइड से फ़ेच करने के बजाय असमर्थित डाउनलोड लौटाते हैं।
    - `environments.list` और `environments.status` Gateway-स्थानीय और Node परिवेश खोज को बनाए रखते हैं। कॉन्फ़िगर किए गए क्लाउड वर्कर और पुराने प्रोफ़ाइलों द्वारा छोड़े गए स्थायी रिकॉर्ड `worker` मेटाडेटा जोड़ते हैं, जिसमें `providerId`, वैकल्पिक `leaseId`, `state`, `ageMs`, वैकल्पिक `idleMs`, और `attachedSessionIds` शामिल हैं। वर्कर जीवनचक्र स्थितियाँ `requested`, `provisioning`, `bootstrapping`, `ready`, `attached`, `idle`, `draining`, `destroying`, `destroyed`, `failed`, और `orphaned` हैं।
    - `environments.create` (`{ profileId, idempotencyKey }`) किसी कॉन्फ़िगर किए गए Plugin प्रदाता प्रोफ़ाइल से वर्कर का प्रावधान करता है; समान कुंजी वाले पुनः प्रयास स्थायी कार्रवाई का पुनः उपयोग करते हैं। `environments.destroy` (`{ environmentId }`) स्थायी वर्कर परिवेश को समान परिणाम देने वाले ढंग से हटाने का अनुरोध करता है। दोनों के लिए `operator.admin` आवश्यक है, ये कंट्रोल-प्लेन लेखन हैं, और स्थिति प्रतिक्रियाओं में प्रयुक्त समान परिवेश सारांश संरचना लौटाते हैं।
    - `agent.identity.get` किसी एजेंट या सत्र के लिए प्रभावी सहायक पहचान लौटाता है।
    - `agent.wait` रन समाप्त होने की प्रतीक्षा करता है और उपलब्ध होने पर अंतिम स्नैपशॉट लौटाता है।

  <Accordion title="सत्र नियंत्रण">
    - `sessions.list` वर्तमान सत्र इंडेक्स लौटाता है, जिसमें एजेंट रनटाइम बैकएंड कॉन्फ़िगर होने पर प्रत्येक पंक्ति का `agentRuntime` मेटाडेटा शामिल होता है। जब क्लाउड-वर्कर प्लेसमेंट सक्षम हो या स्थायी पुनर्प्राप्ति स्थिति मौजूद हो, तो सत्र पंक्तियों में बंद `placement` स्थिति (`local`, `requested`, `provisioning`, `syncing`, `starting`, `active`, `draining`, `reconciling`, `reclaimed`, या `failed`) के साथ स्थिति-विशिष्ट परिवेश, स्वामी-युग, वर्कस्पेस, बंडल, ACK-कर्सर, या पुनर्प्राप्ति फ़ील्ड भी शामिल होते हैं।
    - `sessions.subscribe` और `sessions.unsubscribe` वर्तमान WS क्लाइंट के लिए सत्र परिवर्तन इवेंट सदस्यताएँ चालू या बंद करते हैं।
    - `sessions.messages.subscribe` और `sessions.messages.unsubscribe` एक सत्र के लिए ट्रांस्क्रिप्ट/संदेश इवेंट सदस्यताएँ चालू या बंद करते हैं। उन अनुमोदनों के स्वच्छ किए गए `session.approval` जीवनचक्र इवेंट भी प्राप्त करने के लिए `includeApprovals: true` पास करें, जिनकी स्थायी ऑडियंस में वही सटीक सत्र शामिल है और जिनका समीक्षक बंधन सदस्यता लेने वाले क्लाइंट को अधिकृत करता है। इसके बाद सदस्यता प्रतिक्रिया में सीमित लंबित `approvalReplay` शामिल होता है; जब `truncated` false हो, तब यह प्रामाणिक होता है। ऑप्ट-इन प्रत्येक सदस्यता कॉल के लिए है, स्थायी नहीं: `includeApprovals: true` के बिना उसी सत्र की फिर से सदस्यता लेने पर मौजूदा अनुमोदन सदस्यता हट जाती है। सामान्य सत्र-पठन प्राधिकार के अतिरिक्त, इस ऑप्ट-इन के लिए `operator.admin`, या युग्मित डिवाइस पर `operator.approvals` आवश्यक है।
    - `sessions.preview` विशिष्ट सत्र कुंजियों के लिए सीमित ट्रांस्क्रिप्ट पूर्वावलोकन लौटाता है।
    - `sessions.describe` सटीक सत्र कुंजी के लिए एक Gateway सत्र पंक्ति लौटाता है।
    - `sessions.resolve` सत्र लक्ष्य का समाधान या मानकीकरण करता है।
    - `sessions.create` नई सत्र प्रविष्टि बनाता है। वैकल्पिक `model` और `thinkingLevel` मान प्रारंभिक मॉडल और रीजनिंग ओवरराइड को परमाण्विक रूप से स्थायी बनाते हैं। `worktree: true` प्रबंधित वर्कट्री का प्रावधान करता है; वैकल्पिक `worktreeBaseRef`/`worktreeName` आधार रेफ़ और शाखा नाम चुनते हैं, और `execNode` (`operator.admin`) सत्र निष्पादन को Node होस्ट से बाँधता है। बनाया गया वर्कट्री परिणाम में प्रतिध्वनित होता है और सत्र पंक्ति (`worktree: { id, branch, repoRoot }`) पर स्थायी बनाया जाता है। जब प्रविष्टि बन जाती है लेकिन उसका नेस्टेड प्रारंभिक `chat.send` अस्वीकार हो जाता है, तो सफल परिणाम में `runStarted: false` और `runError` शामिल होते हैं; क्लाइंट प्रॉम्प्ट सुरक्षित रख सकते हैं और लौटाई गई सत्र कुंजी के विरुद्ध पुनः प्रयास कर सकते हैं। `emitCommandHooks: true` के साथ `parentSessionKey` पास करने वाले कॉलर को किसी अलग चाइल्ड के जीवनचक्र निपटान की भी घोषणा करनी चाहिए: `succeedsParent: true` पैरेंट को `session_end` के साथ समाप्त करता है, जबकि `false` पैरेंट को सक्रिय रखता है और केवल चाइल्ड का `session_start` उत्सर्जित करता है। `succeedsParent` छोड़ने पर मौजूदा क्लाइंट के लिए पुराना पैरेंट-रोलओवर व्यवहार सुरक्षित रहता है। इस निपटान के लिए पैरेंट लिंक और कमांड हुक दोनों आवश्यक हैं; कोई फ़ोर्क अपने पैरेंट को सफल नहीं कर सकता। मुख्य-सत्र का उसी स्थान पर रीसेट व्यवहार अपरिवर्तित है, क्योंकि कोई अलग चाइल्ड नहीं बनाया जाता।
    - `sessions.dispatch` (`operator.admin`) सत्र-स्वामित्व वाले प्रबंधित वर्कट्री सहित किसी मौजूदा स्थानीय OpenClaw सत्र को कॉन्फ़िगर किए गए क्लाउड-वर्कर प्रोफ़ाइल में ले जाता है। `{ key, profileId, agentId? }` पास करें। कोई वर्कर प्रोफ़ाइल कॉन्फ़िगर न होने पर यह विधि उपलब्ध नहीं होती, सक्रिय कार्य को समाप्त होने देने से पहले स्थानीय टर्न प्रवेश बंद करती है, और प्लेसमेंट के `active` वर्कर स्वामित्व तक पहुँचने के बाद ही लौटती है। प्रेषण एकतरफ़ा है; वर्कर से स्थानीय में वापस खींचना इस RPC का हिस्सा नहीं है।
    - `sessions.groups.list`, `sessions.groups.put`, `sessions.groups.rename`, और `sessions.groups.delete` Gateway-स्वामित्व वाली कस्टम सत्र समूह सूची (नाम + प्रदर्शन क्रम) प्रबंधित करते हैं। सदस्यता प्रत्येक सत्र के `category` फ़ील्ड में रहती है; नाम बदलना और हटाना सर्वर-साइड पर सदस्य सत्रों को अपडेट करता है।
    - `sessions.send` मौजूदा सत्र में संदेश भेजता है।
    - `sessions.steer` सक्रिय सत्र के लिए बाधित करके दिशा देने वाला प्रकार है।
    - `sessions.abort` किसी सत्र का सक्रिय कार्य निरस्त करता है। `key` के साथ वैकल्पिक `runId` पास करें, या उन सक्रिय रनों के लिए केवल `runId` पास करें जिन्हें Gateway किसी सत्र से संबद्ध कर सकता है।
    - `sessions.patch` सत्र मेटाडेटा/ओवरराइड अपडेट करता है और समाधान किए गए मानक मॉडल के साथ प्रभावी `agentRuntime` की रिपोर्ट करता है।
    - `sessions.reset`, `sessions.delete`, और `sessions.compact` सत्र रखरखाव करते हैं।
    - `sessions.get` पूरी संग्रहीत सत्र पंक्ति लौटाता है।
    - चैट निष्पादन अब भी `chat.history`, `chat.send`, `chat.abort`, और `chat.inject` का उपयोग करता है। `chat.history` UI क्लाइंट के लिए प्रदर्शन-मानकीकृत है: इनलाइन डायरेक्टिव टैग दृश्य टेक्स्ट से हटा दिए जाते हैं, सादे-टेक्स्ट वाले टूल-कॉल XML पेलोड (`<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>`, और काटे गए टूल-कॉल ब्लॉक) तथा लीक हुए ASCII/पूर्ण-चौड़ाई मॉडल नियंत्रण टोकन हटा दिए जाते हैं, केवल मौन-टोकन वाली सहायक पंक्तियाँ (सटीक `NO_REPLY` / `no_reply`) छोड़ दी जाती हैं, और अत्यधिक बड़ी पंक्तियों को प्लेसहोल्डर से बदला जा सकता है।
    - `chat.message.get` एकल दृश्य ट्रांस्क्रिप्ट प्रविष्टि के लिए योगात्मक, सीमित पूर्ण-संदेश रीडर है। `sessionKey`, एजेंट-स्कोप वाले सत्र चयन के समय वैकल्पिक `agentId`, और `chat.history` के माध्यम से पहले प्रदर्शित ट्रांस्क्रिप्ट `messageId` पास करें; यदि संग्रहीत प्रविष्टि अब भी उपलब्ध और अत्यधिक बड़ी नहीं है, तो Gateway हल्की हिस्ट्री कटौती सीमा के बिना वही प्रदर्शन-मानकीकृत प्रक्षेपण लौटाता है।
    - `chat.toolTitles` Control UI में रेंडर किए गए टूल कॉल के लिए संक्षिप्त उद्देश्य शीर्षक लौटाता है (बैच में, सीमित इनपुट के साथ अधिकतम 24 आइटम)। यह सुविधा `gateway.controlUi.toolTitles` के माध्यम से ऑप्ट-इन है (डिफ़ॉल्ट रूप से बंद); अक्षम Gateway बिना मॉडल कॉल के `{ titles: {}, disabled: true }` का उत्तर देते हैं, ताकि क्लाइंट पूछना बंद कर दें। सक्षम होने पर शीर्षक मानक उपयोगिता-मॉडल रूटिंग का उपयोग करते हैं: स्पष्ट रूप से कॉन्फ़िगर किया गया `utilityModel` (एक ऑपरेटर निर्णय, जो सभी उपयोगिता कार्यों की तरह सीमित कार्य सामग्री को चुने गए प्रदाता को भेज सकता है), अन्यथा सत्र प्रदाता का घोषित लघु-मॉडल डिफ़ॉल्ट, ताकि कोई नया निकास गंतव्य अप्रत्यक्ष रूप से प्रकट न हो; खाली `utilityModel` इन्हें पूरी तरह अक्षम कर देता है। शीर्षक कभी भी प्राथमिक मॉडल पर फ़ॉलबैक नहीं करते। परिणाम प्रति-एजेंट स्थिति डेटाबेस में टूल नाम + इनपुट के आधार पर कैश होते हैं, इसलिए बार-बार देखने पर समान कॉल का दोबारा शुल्क नहीं लगता।
    - `chat.send` स्वतः कटऑफ़ से पहले शुरू किए गए मॉडल कॉल के लिए तेज़ मोड का उपयोग करने हेतु एक-टर्न `fastMode: "auto"` स्वीकार करता है, फिर बाद के पुनः प्रयास, फ़ॉलबैक, टूल-परिणाम, या निरंतरता कॉल को तेज़ मोड के बिना शुरू करता है। कटऑफ़ डिफ़ॉल्ट रूप से 60 सेकंड (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) है और इसे प्रत्येक मॉडल के लिए `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` से कॉन्फ़िगर किया जा सकता है। `chat.send` कॉलर उस अनुरोध के लिए कटऑफ़ ओवरराइड करने हेतु एक-टर्न `fastAutoOnSeconds` पास कर सकता है। केवल इस अनुरोध के लिए संग्रहीत कतार मोड ओवरराइड करने हेतु `queueMode` (`steer`, `followup`, `collect`, या `interrupt`) पास करें; स्पष्ट Control UI दिशा-परिवर्तन कार्रवाइयाँ `queueMode: "steer"` का उपयोग करती हैं।

## ऑडिट लेजर RPC

`audit.activity.list` ऑपरेटर क्लाइंट को एजेंट रन, टूल कार्रवाई और ऑप्ट-इन संदेश जीवनचक्र मेटाडेटा का स्थिर नवीनतम-प्रथम दृश्य देता है। इसके लिए
`operator.read` आवश्यक है। क्वेरी 30 दिनों से पुराने रिकॉर्ड को शामिल नहीं करतीं और साझा
SQLite लेजर की सीमा 100,000 रिकॉर्ड है। समय-सीमा समाप्त पंक्तियाँ
Gateway स्टार्टअप, प्रति घंटे रखरखाव और बाद के लेखनों के दौरान हटा दी जाती हैं। डेटा मॉडल और गोपनीयता सिमेंटिक्स के लिए
[ऑडिट इतिहास](/hi/gateway/audit) देखें।

- पैरामीटर: वैकल्पिक सटीक `agentId`, `sessionKey`, या `runId`; वैकल्पिक `kind`
  (`"agent_run"`, `"tool_action"`, या `"message"`); वैकल्पिक `status`
  (`"started"`, `"succeeded"`, `"failed"`, `"cancelled"`, `"timed_out"`,
  `"blocked"`, या `"unknown"`); वैकल्पिक संदेश `direction` (`"inbound"` या
  `"outbound"`) और सटीक `channel`; वैकल्पिक समावेशी `after` / `before`
  Unix-मिलीसेकंड सीमाएँ; `1` से `500` तक वैकल्पिक `limit`; और पिछले पृष्ठ का वैकल्पिक
  स्ट्रिंग `cursor`।
- परिणाम: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`।

नामित V1 परिणाम यूनियन में एजेंट-रन, टूल-कार्रवाई, इनबाउंड-संदेश
और आउटबाउंड-संदेश के लिए अलग-अलग स्कीमा हैं। `eventType` डिस्क्रिमिनेटर क्रमशः
`agent_run`, `tool_action`, `inbound_message`, या `outbound_message` है; `kind` और
संदेश `direction` फ़िल्टरिंग और प्रदर्शन के लिए उपलब्ध रहते हैं। प्रत्येक इवेंट में
पूर्णांक `schemaVersion: 1` होता है। संदेश पहचान संदर्भ सटीक
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` प्रारूप का उपयोग करते हैं; चैनल-प्रेषक ऐक्टर
ID भी उसी प्रारूप का उपयोग करता है।

सभी वेरिएंट के लिए `eventType`, `schemaVersion`, `eventId`, `sequence`,
`sourceSequence`, `occurredAt`, `kind`, `action`, `status`, `actor`, और
`redaction` आवश्यक हैं। वेरिएंट फ़ील्ड ये हैं:

| `eventType`        | आवश्यक फ़ील्ड                                                   | वैकल्पिक फ़ील्ड                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`, `runId`; `kind: "agent_run"`                           | `sessionKey`, `sessionId`, `errorCode`                                                                                          |
| `tool_action`      | `agentId`, `runId`; `kind: "tool_action"`                         | `sessionKey`, `sessionId`, `toolCallId`, `toolName`, `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`, `channel`, `conversationKind`, `outcome`  | `agentId`, `runId`, `durationMs`, `resultCount`, पहचान संदर्भ, `reasonCode`, `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`, `channel`, `conversationKind`, `outcome` | `agentId`, `runId`, `durationMs`, `resultCount`, पहचान संदर्भ, `reasonCode`, `deliveryKind`, `failureStage`, `errorCode` |

बंद संदेश एनम ये हैं:

- `conversationKind`: `direct`, `group`, `channel`, या `unknown`।
- इनबाउंड `outcome`: `completed`, `skipped`, या `failed`; वैकल्पिक
  `reasonCode`: `duplicate`, `reply_operation_active`,
  `reply_operation_aborted`, `fast_abort`, `plugin_bound_handled`,
  `plugin_bound_unavailable`, `plugin_bound_declined`, `plugin_bound_error`,
  `before_dispatch_handled`, `acp_dispatch_completed`, `acp_dispatch_failed`,
  `acp_dispatch_empty`, या `acp_dispatch_aborted`।
- आउटबाउंड `outcome`: `sent`, `suppressed`, `failed`, या `unknown`; वैकल्पिक
  `reasonCode`: `cancelled_by_message_sending_hook`,
  `cancelled_by_reply_payload_sending_hook`,
  `empty_after_message_sending_hook`, `empty_after_reply_payload_sending_hook`,
  या `no_visible_payload`। ऐसा एडाप्टर जो कोई प्लेटफ़ॉर्म पहचान नहीं लौटाता,
  `unknown` होता है, क्योंकि बाहरी दुष्प्रभाव का न होना सिद्ध नहीं किया जा सकता।
- `deliveryKind`: `text`, `media`, या `other`; `failureStage`:
  `platform_send`, `queue`, या `unknown`।

टर्मिनल फ़ील्ड सहसंबद्ध हैं, स्वतंत्र रूप से वैकल्पिक नहीं:

| प्रकार          | टर्मिनल मैपिंग                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| एजेंट रन        | `started` में कोई `errorCode` नहीं होता; प्रत्येक गैर-सफल समाप्त स्थिति के लिए उसका संगत `run_*` कोड आवश्यक है।                                                                 |
| टूल कार्रवाई      | `started` और सफल स्थिति में कोई `errorCode` नहीं होता; प्रत्येक अन्य समाप्त स्थिति के लिए उसका संगत `tool_*` कोड आवश्यक है।                                                       |
| इनबाउंड संदेश  | सफल = `completed`; अवरुद्ध = `skipped`; विफल = `failed` के साथ `message_processing_failed`। `reasonCode`, मौजूद होने पर, उसी टर्मिनल परिवार से संबंधित होना चाहिए। |
| आउटबाउंड संदेश | सफल = `sent`; अवरुद्ध = `suppressed` के साथ `reasonCode`; विफल = `failed` के साथ `errorCode` और `failureStage`; अज्ञात = `unknown` के साथ `failureStage`।      |

प्रत्येक गतिविधि इवेंट में एक स्थिर इवेंट आईडी, एकदिश रूप से बढ़ता लेजर अनुक्रम,
स्रोत इवेंट अनुक्रम, टाइमस्टैम्प, कर्ता, कार्रवाई, स्थिति, पूर्णांक
`schemaVersion: 1`, और `redaction: "metadata_only"` शामिल होते हैं। रन और टूल रिकॉर्ड के
लिए एजेंट तथा रन उद्गम आवश्यक हैं और इनमें सत्र उद्गम शामिल हो सकता है। संदेश
रिकॉर्ड में एजेंट और रन आईडी शामिल हो सकते हैं, लेकिन जानबूझकर कभी
`sessionKey` या `sessionId` शामिल नहीं होते; इसलिए `sessionKey` क्वेरी फ़िल्टर केवल
रन और टूल पंक्तियों पर लागू होता है। टूल इवेंट में टूल कॉल आईडी और टूल का नाम शामिल हो सकता है।

संदेश रिकॉर्ड `message.inbound.processed` या
`message.outbound.finished` का उपयोग करते हैं और दिशा, चैनल, वार्तालाप प्रकार,
सामान्यीकृत परिणाम तथा वैकल्पिक डिलीवरी प्रकार, विफलता चरण, अवधि,
परिणाम संख्या, कारण कोड और इंस्टॉलेशन-स्थानीय कुंजीबद्ध
खाता/वार्तालाप/संदेश/लक्ष्य छद्मनाम जोड़ते हैं। ये छद्मनाम
सहसंबंध स्थापित करने में सहायता करते हैं, लेकिन अनामीकरण नहीं हैं: स्टेट डेटाबेस में उनकी कुंजी होती है,
जबकि RPC और CLI निर्यात में नहीं होती। लेजर प्रॉम्प्ट, संदेश
के मुख्य भाग, टूल आर्ग्युमेंट, टूल परिणाम, कमांड आउटपुट या अपरिष्कृत त्रुटि टेक्स्ट संग्रहीत नहीं करता।
रन/टूल के `sessionKey` मान अपरिष्कृत सहसंबंध मेटाडेटा बने रहते हैं और इनमें
प्लेटफ़ॉर्म खाता या पीयर आईडी अंतर्निहित हो सकते हैं; संदेश रिकॉर्ड सत्र कुंजियाँ छोड़ देते हैं।

इनबाउंड पंक्तियों के लिए, `durationMs` कोर डिस्पैच से उसके टर्मिनल तक का मापन करता है और
`resultCount` अंतिम रूप दिए गए कतारबद्ध टूल, ब्लॉक और उत्तर पेलोड की गणना करता है। 
आउटबाउंड पंक्तियों के लिए, `durationMs` डिलीवरी स्वामित्व से अभिस्वीकृति,
डेड लेटर या समाधान तक की अवधि को समाहित करता है (कतार में प्रतीक्षा समय सहित), और `resultCount`
पहचाने गए भौतिक प्लेटफ़ॉर्म प्रेषणों की गणना करता है। `deliveryKind`, मौजूद होने पर,
हुक और रेंडरिंग के बाद प्रभावी पेलोड का वर्णन करता है; दबाई गई या
क्रैश-अस्पष्ट पंक्तियों में इसे छोड़ दिया जाता है।

वर्तमान संदेश कवरेज में वे स्वीकृत इनबाउंड संदेश शामिल हैं जो कोर
डिस्पैच तक पहुँचते हैं, जिनमें कोर डुप्लिकेट/टर्मिनल परिणाम भी शामिल हैं। आउटबाउंड कवरेज
साझा टिकाऊ डिलीवरी तक पहुँचने वाले प्रत्येक मूल तार्किक उत्तर पेलोड के लिए
एक टर्मिनल पंक्ति लिखता है; खंडन और अडैप्टर फ़ैन-आउट को `resultCount` में एकत्रित किया जाता है। कतारबद्ध
पुनः प्रयास योग्य या अस्पष्ट प्रेषण केवल अभिस्वीकृति, डेड
लेटर या समाधान के बाद रिकॉर्ड किए जाते हैं। उन साझा
सीमाओं को बाइपास करने वाले Plugin-स्थानीय और प्रत्यक्ष-प्रेषण पथ अभी तक शामिल नहीं हैं। सीमित वर्कर कतार सर्वोत्तम-प्रयास
आधारित है और विफलता या संतृप्ति पर रिकॉर्ड छोड़ सकती है, इसलिए यह सतह
हानिरहित अनुपालन अभिलेख नहीं है।

रिकॉर्डिंग डिफ़ॉल्ट रूप से चालू होती है और
[`audit.enabled`](/hi/gateway/configuration-reference#audit) द्वारा नियंत्रित होती है। संदेश रिकॉर्डिंग
अलग से `audit.messages` द्वारा नियंत्रित होती है और डिफ़ॉल्ट रूप से `"off"` होती है। जब
रिकॉर्डिंग अक्षम होती है, तब `audit.activity.list` पहले लिखे गए रिकॉर्ड की
समाप्ति तक उन्हें उपलब्ध कराता रहता है।

शिप किए गए `audit.list` अनुरोध, परिणाम और `AuditEvent` स्कीमा
अपरिवर्तित रहते हैं और केवल एजेंट-रन तथा टूल-कार्रवाई रिकॉर्ड लौटाते हैं। नए ऑपरेटर
क्लाइंट को Gateway द्वारा इसका विज्ञापन किए जाने पर `audit.activity.list` कॉल करना चाहिए। पुराने
Gateway या तो `unknown method: audit.activity.list` रिपोर्ट कर सकते हैं या, क्योंकि
शिप किए गए संस्करणों में प्राधिकरण मेथड लुकअप से पहले होता था, रीड-स्कोप वाले अनुरोध को `missing scope:
operator.admin` रिपोर्ट कर सकते हैं। बाद वाले को केवल तभी मेथड की अनुपस्थिति मानें
जब मेथड का विज्ञापन नहीं किया गया हो। इसके बाद क्लाइंट केवल तभी `audit.list`
का पुनः प्रयास कर सकता है जब उसके फ़िल्टर के लिए संदेश प्रकार, दिशा या चैनल
समर्थन आवश्यक न हो।

टेक्स्ट क्वेरी और सीमित JSON निर्यात के लिए [`openclaw audit`](/hi/cli/audit) का उपयोग करें।

## टास्क लेजर RPC

ऑपरेटर क्लाइंट टास्क लेजर RPC (`packages/gateway-protocol/src/schema/tasks.ts`) के माध्यम से
Gateway की पृष्ठभूमि टास्क के रिकॉर्ड का निरीक्षण और उन्हें रद्द करते हैं। ये
अपरिष्कृत रनटाइम स्थिति नहीं, बल्कि स्वच्छीकृत टास्क सारांश लौटाते हैं।

- `tasks.list` के लिए `operator.read` आवश्यक है।
  - पैरामीटर: वैकल्पिक `status` (`"queued"`, `"running"`, `"completed"`,
    `"failed"`, `"cancelled"`, या `"timed_out"`) या इन स्थितियों की एक सरणी,
    वैकल्पिक `agentId`, वैकल्पिक `sessionKey`, वैकल्पिक `limit` जिसका मान `1` से
    `500` तक हो, और वैकल्पिक स्ट्रिंग `cursor`।
  - परिणाम: `{ "tasks": TaskSummary[], "nextCursor"?: string }`।
- `tasks.get` के लिए `operator.read` आवश्यक है।
  - पैरामीटर: `{ "taskId": string }`।
  - परिणाम: `{ "task": TaskSummary }`।
  - अनुपलब्ध टास्क आईडी Gateway का नॉट-फाउंड त्रुटि स्वरूप लौटाती हैं।
- `tasks.cancel` के लिए `operator.write` आवश्यक है।
  - पैरामीटर: `{ "taskId": string, "reason"?: string }`।
  - परिणाम: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`।
  - `found` बताता है कि लेजर में कोई मेल खाता टास्क था या नहीं। `cancelled`
    बताता है कि रनटाइम ने रद्दीकरण स्वीकार किया या रिकॉर्ड किया या नहीं।

`TaskSummary` में `id`, `status` और वैकल्पिक मेटाडेटा शामिल हैं: `kind`,
`runtime`, `title`, `agentId`, `sessionKey`, `childSessionKey`, `ownerKey`,
`runId`, `taskId`, `flowId`, `parentTaskId`, `sourceId`, टाइमस्टैम्प, प्रगति,
टर्मिनल सारांश और स्वच्छीकृत त्रुटि टेक्स्ट। `agentId` टास्क निष्पादित करने वाले
एजेंट की पहचान करता है; `sessionKey` और `ownerKey` अनुरोधकर्ता तथा नियंत्रण
संदर्भ को संरक्षित रखते हैं।

## ऑपरेटर सहायक मेथड

- `commands.list` (`operator.read`) किसी एजेंट के लिए रनटाइम कमांड इन्वेंटरी प्राप्त करता है।
  - `agentId` वैकल्पिक है; डिफ़ॉल्ट एजेंट वर्कस्पेस पढ़ने के लिए इसे छोड़ दें।
  - `scope` नियंत्रित करता है कि प्राथमिक `name` किस सतह को लक्षित करता है: `text` अग्रणी `/` के बिना प्राथमिक टेक्स्ट कमांड टोकन लौटाता है; उपलब्ध होने पर `native` और डिफ़ॉल्ट `both` पथ प्रोवाइडर-सजग नेटिव नाम लौटाते हैं।
  - `textAliases` में `/model` और `/m` जैसे सटीक स्लैश उपनाम होते हैं।
  - `nativeName` में, मौजूद होने पर, प्रोवाइडर-सजग नेटिव कमांड नाम होता है।
  - `provider` वैकल्पिक है और केवल नेटिव नामकरण तथा नेटिव Plugin कमांड की उपलब्धता को प्रभावित करता है।
  - `includeArgs=false` प्रतिक्रिया से सीरियलाइज़ किए गए आर्ग्युमेंट मेटाडेटा को हटा देता है।
- `tools.catalog` (`operator.read`) किसी एजेंट के लिए रनटाइम टूल कैटलॉग प्राप्त करता है। प्रतिक्रिया में समूहीकृत टूल और उद्गम मेटाडेटा शामिल होते हैं:
  - `source`: `core` या `plugin`
  - `pluginId`: `source="plugin"` होने पर Plugin स्वामी
  - `optional`: क्या कोई Plugin टूल वैकल्पिक है
- `tools.effective` (`operator.read`) किसी सत्र के लिए रनटाइम-प्रभावी टूल इन्वेंटरी प्राप्त करता है।
  - `sessionKey` आवश्यक है।
  - Gateway कॉलर द्वारा दी गई प्रमाणीकरण या डिलीवरी संदर्भ को स्वीकार करने के बजाय सत्र से सर्वर-साइड विश्वसनीय रनटाइम संदर्भ प्राप्त करता है।
  - प्रतिक्रिया सक्रिय इन्वेंटरी का सत्र-स्कोप वाला, सर्वर से प्राप्त प्रक्षेपण है, जिसमें कोर, Plugin, चैनल और पहले से खोजे गए MCP सर्वर टूल शामिल हैं।
  - `tools.effective` MCP के लिए केवल-पढ़ने योग्य है: यह वार्म सत्र MCP कैटलॉग को अंतिम टूल नीति के माध्यम से प्रक्षेपित कर सकता है, लेकिन MCP रनटाइम नहीं बनाता, ट्रांसपोर्ट से कनेक्ट नहीं करता, या `tools/list` जारी नहीं करता। यदि कोई मेल खाता वार्म कैटलॉग मौजूद नहीं है, तो प्रतिक्रिया में `mcp-not-yet-connected`, `mcp-not-yet-listed`, या `mcp-stale-catalog` जैसी सूचना शामिल हो सकती है।
  - प्रभावी टूल प्रविष्टियाँ `source="core"`, `source="plugin"`, `source="channel"`, या `source="mcp"` का उपयोग करती हैं।
- `tools.invoke` (`operator.write`) `/tools/invoke` के समान Gateway नीति पथ के माध्यम से किसी एक उपलब्ध टूल को आमंत्रित करता है।
  - `name` आवश्यक है। `args`, `sessionKey`, `agentId`, `confirm`, और `idempotencyKey` वैकल्पिक हैं।
  - यदि `sessionKey` और `agentId` दोनों मौजूद हैं, तो रिज़ॉल्व किया गया सत्र एजेंट `agentId` से मेल खाना चाहिए।
  - `cron`, `gateway`, और `nodes` जैसे केवल-स्वामी कोर रैपरों के लिए स्वामी/व्यवस्थापक पहचान (`operator.admin`) आवश्यक है, भले ही `tools.invoke` स्वयं `operator.write` हो।
  - प्रतिक्रिया SDK के लिए एक एन्वेलप है, जिसमें `ok`, `toolName`, वैकल्पिक `output`, और टाइप किए गए `error` फ़ील्ड होते हैं। अनुमोदन या नीति अस्वीकृतियाँ Gateway टूल नीति पाइपलाइन को बायपास करने के बजाय पेलोड में `ok:false` लौटाती हैं।
- `skills.status` (`operator.read`) किसी एजेंट के लिए दृश्यमान स्किल इन्वेंटरी प्राप्त करता है।
  - `agentId` वैकल्पिक है; डिफ़ॉल्ट एजेंट वर्कस्पेस पढ़ने के लिए इसे छोड़ दें।
  - प्रतिक्रिया में पात्रता, अनुपस्थित आवश्यकताएँ, कॉन्फ़िगरेशन जाँच और कच्चे सीक्रेट मान उजागर किए बिना सैनिटाइज़ किए गए इंस्टॉल विकल्प शामिल होते हैं।
- `skills.search` और `skills.detail` (`operator.read`) ClawHub खोज मेटाडेटा लौटाते हैं।
- `skills.upload.begin`, `skills.upload.chunk`, और `skills.upload.commit` (`operator.admin`) निजी स्किल आर्काइव को इंस्टॉल करने से पहले स्टेज करते हैं। यह विश्वसनीय क्लाइंट के लिए एक अलग व्यवस्थापक अपलोड पथ है, सामान्य ClawHub स्किल इंस्टॉल प्रवाह नहीं, और जब तक `skills.install.allowUploadedArchives` सक्षम न हो तब तक डिफ़ॉल्ट रूप से अक्षम रहता है।
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })` उस स्लग और फ़ोर्स मान से बँधा अपलोड बनाता है।
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })` सटीक डीकोड किए गए ऑफ़सेट पर बाइट जोड़ता है।
  - `skills.upload.commit({ uploadId, sha256? })` अंतिम आकार और SHA-256 सत्यापित करता है। कमिट केवल अपलोड को अंतिम रूप देता है; यह स्किल इंस्टॉल नहीं करता।
  - अपलोड किए गए स्किल आर्काइव ऐसे ज़िप आर्काइव होते हैं जिनमें `SKILL.md` रूट होता है। आर्काइव की आंतरिक डायरेक्टरी का नाम कभी भी इंस्टॉल लक्ष्य नहीं चुनता।
- `skills.install` (`operator.admin`) के तीन मोड हैं:
  - ClawHub मोड: `{ source: "clawhub", slug, version?, force? }` डिफ़ॉल्ट एजेंट वर्कस्पेस की `skills/` डायरेक्टरी में स्किल फ़ोल्डर इंस्टॉल करता है।
  - अपलोड मोड: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }` कमिट किया गया अपलोड डिफ़ॉल्ट एजेंट वर्कस्पेस की `skills/<slug>` डायरेक्टरी में इंस्टॉल करता है। स्लग और फ़ोर्स मान मूल `skills.upload.begin` अनुरोध से मेल खाने चाहिए। जब तक `skills.install.allowUploadedArchives` सक्षम न हो, इसे अस्वीकार किया जाता है; यह सेटिंग ClawHub इंस्टॉल को प्रभावित नहीं करती।
  - Gateway इंस्टॉलर मोड: `{ name, installId, timeoutMs? }` Gateway होस्ट पर घोषित `metadata.openclaw.install` क्रिया चलाता है। पुराने क्लाइंट अभी भी `dangerouslyForceUnsafeInstall` भेज सकते हैं; यह फ़ील्ड अप्रचलित है, केवल प्रोटोकॉल संगतता के लिए स्वीकार किया जाता है और अनदेखा किया जाता है। ऑपरेटर-स्वामित्व वाले इंस्टॉल निर्णयों के लिए `security.installPolicy` का उपयोग करें।
- `skills.update` (`operator.admin`) के दो मोड हैं:
  - ClawHub मोड डिफ़ॉल्ट एजेंट वर्कस्पेस में ट्रैक किए गए किसी एक स्लग या सभी ट्रैक किए गए ClawHub इंस्टॉल को अपडेट करता है।
  - कॉन्फ़िगरेशन मोड `skills.entries.<skillKey>` मानों को पैच करता है, जैसे `enabled`, `apiKey`, और `env`।

### `models.list` दृश्य

`models.list` एक वैकल्पिक `view` पैरामीटर स्वीकार करता है
(`src/agents/model-catalog-visibility.ts`):

- छोड़े जाने पर या `"default"`: यदि `agents.defaults.modelPolicy.allow` कॉन्फ़िगर किया गया है, तो प्रतिक्रिया अनुमत कैटलॉग होती है, जिसमें `provider/*` प्रविष्टियों के लिए गतिशील रूप से खोजे गए मॉडल शामिल होते हैं। अन्यथा प्रतिक्रिया पूर्ण Gateway कैटलॉग होती है।
- `"configured"`: पिकर-आकार का व्यवहार। यदि `agents.defaults.modelPolicy.allow` कॉन्फ़िगर किया गया है, तो वही प्रभावी रहता है, जिसमें `provider/*` प्रविष्टियों के लिए प्रोवाइडर-स्कोप वाली खोज भी शामिल है। अनुमति-सूची के बिना, प्रतिक्रिया स्पष्ट `models.providers.<provider>.models` प्रविष्टियों का उपयोग करती है और केवल तब पूर्ण कैटलॉग पर लौटती है जब कोई कॉन्फ़िगर की गई मॉडल पंक्ति मौजूद न हो।
- `"provider-config"`: स्रोत द्वारा लिखी गई `models.providers.*.models` इन्वेंटरी, जो पिकर अनुमति-सूचियों से स्वतंत्र है। पंक्तियों में सार्वजनिक मॉडल क्षमताएँ और रूट-सजग उपलब्धता शामिल होती हैं, लेकिन प्रोवाइडर एंडपॉइंट, प्रमाणीकरण सामग्री और रनटाइम अनुरोध कॉन्फ़िगरेशन शामिल नहीं होते।
- `"all"`: पूर्ण Gateway कैटलॉग, जो `agents.defaults.modelPolicy.allow` को बायपास करता है। इसका उपयोग निदान/खोज UI के लिए करें, सामान्य मॉडल पिकर के लिए नहीं।

## निष्पादन अनुमोदन

- जब किसी निष्पादन अनुरोध को अनुमोदन चाहिए, तो Gateway `exec.approval.requested` प्रसारित करता है।
- ऑपरेटर क्लाइंट `exec.approval.resolve` को कॉल करके समाधान करते हैं (`operator.approvals` आवश्यक है)।
- `host=node` के लिए, `exec.approval.request` में `systemRunPlan` (कैनोनिकल `argv`/`cwd`/`rawCommand`/सत्र मेटाडेटा) शामिल होना चाहिए। `systemRunPlan` के बिना अनुरोध अस्वीकार कर दिए जाते हैं।
- अनुमोदन के बाद, अग्रेषित `node.invoke system.run` कॉल उस कैनोनिकल `systemRunPlan` को प्रामाणिक कमांड/cwd/सत्र संदर्भ के रूप में पुनः उपयोग करती हैं।
- यदि कोई कॉलर तैयारी और अंतिम अनुमोदित `system.run` अग्रेषण के बीच `command`, `rawCommand`, `cwd`, `agentId`, या `sessionKey` बदलता है, तो Gateway बदले गए पेलोड पर भरोसा करने के बजाय रन को अस्वीकार कर देता है।

## एजेंट डिलीवरी फ़ॉलबैक

- `agent` अनुरोधों में आउटबाउंड डिलीवरी का अनुरोध करने के लिए `deliver=true` शामिल किया जा सकता है।
- `bestEffortDeliver=false` (डिफ़ॉल्ट) सख्त व्यवहार बनाए रखता है: अनसुलझे या केवल-आंतरिक डिलीवरी लक्ष्य `INVALID_REQUEST` लौटाते हैं।
- `bestEffortDeliver=true` किसी बाहरी डिलीवरी-योग्य रूट को रिज़ॉल्व न किए जा सकने पर केवल-सत्र निष्पादन पर फ़ॉलबैक की अनुमति देता है (उदाहरण के लिए, आंतरिक/वेबचैट सत्र या अस्पष्ट मल्टी-चैनल कॉन्फ़िगरेशन)।
- डिलीवरी का अनुरोध किए जाने पर अंतिम `agent` परिणामों में `result.deliveryStatus` शामिल हो सकता है, जो [`openclaw agent --json --deliver`](/hi/cli/agent#json-delivery-status) के लिए दस्तावेजीकृत समान `sent`, `suppressed`, `partial_failed`, और `failed` स्थितियों का उपयोग करता है।

## संस्करण निर्धारण

- `PROTOCOL_VERSION`, `MIN_CLIENT_PROTOCOL_VERSION`, `MIN_NODE_PROTOCOL_VERSION`, और `MIN_PROBE_PROTOCOL_VERSION`, `packages/gateway-protocol/src/version.ts` में स्थित हैं।
- क्लाइंट `minProtocol` + `maxProtocol` भेजते हैं। ऑपरेटर और UI क्लाइंट को उस सीमा में वर्तमान प्रोटोकॉल शामिल करना चाहिए; वर्तमान क्लाइंट और सर्वर प्रोटोकॉल v4 चलाते हैं।
- `role: "node"` और `client.mode: "node"` दोनों वाले प्रमाणित क्लाइंट N-1 Node प्रोटोकॉल (वर्तमान में v3) का उपयोग कर सकते हैं। हल्के रीस्टार्ट प्रोब समान N-1 विंडो का उपयोग करते हैं। इस संगतता विंडो से डिवाइस प्रमाणीकरण, पेयरिंग, स्कोप, कमांड नीति और निष्पादन अनुमोदन अपरिवर्तित रहते हैं। Plugin-स्वामित्व वाली Node क्षमताएँ और कमांड तब तक रोके जाते हैं, जब तक Node वर्तमान प्रोटोकॉल पर अपग्रेड नहीं हो जाता, क्योंकि उनकी होस्ट की गई सतहें N-1 अनुबंध का हिस्सा नहीं हैं।
- स्कीमा और मॉडल TypeBox परिभाषाओं से जनरेट किए जाते हैं:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

`policy.maxPayload`, और `policy.maxBufferedBytes` को `hello-ok` में घोषित करता है; क्लाइंटों
को हैंडशेक-पूर्व डिफ़ॉल्ट के बजाय इन मानों का पालन करना चाहिए।

जब प्रत्येक लंबित अनुरोध की एक समयसीमा होती है, तब संदर्भ क्लाइंट सीमित अनुरोधों को
उनकी कॉन्फ़िगर की गई समयसीमा का स्वामित्व लेने देता है। सीमित
`timeoutMs` के बिना कोई `expectFinal` अनुरोध, `timeoutMs: null` वाला कोई भी अनुरोध,
या सीमित और असीमित अनुरोधों का मिश्रण टिक वॉचडॉग को सक्रिय रखता है। यदि इनबाउंड ईवेंट और
प्रतिक्रियाएँ टिक-टाइमआउट सीमा के बाद भी मौन रहती हैं, तो क्लाइंट
सॉकेट को कोड `4000` के साथ बंद करता है, प्रत्येक लंबित अनुरोध को अस्वीकार करता है, और पुनः कनेक्ट करता है। यह
पुनः कनेक्ट होने के बाद अस्वीकृत अनुरोधों को फिर से नहीं चलाता।

## प्रमाणीकरण

- साझा-गोपनीय Gateway प्रमाणीकरण, कॉन्फ़िगर किए गए
  `gateway.auth.mode` (`"none" | "token" | "password" | "trusted-proxy"`) के आधार पर `connect.params.auth.token` या
  `connect.params.auth.password` का उपयोग करता है।
- Tailscale Serve (`gateway.auth.allowTailscale: true`) या गैर-लूपबैक `gateway.auth.mode: "trusted-proxy"` जैसे पहचान-युक्त मोड,
  `connect.params.auth.*` के बजाय अनुरोध हेडर से कनेक्ट
  प्रमाणीकरण जाँच को पूरा करते हैं।
- निजी-इनग्रेस `gateway.auth.mode: "none"` साझा-गोपनीय कनेक्ट प्रमाणीकरण को
  पूरी तरह छोड़ देता है; इस मोड को सार्वजनिक/अविश्वसनीय इनग्रेस पर उजागर न करें।
- पेयरिंग के बाद, Gateway कनेक्शन
  भूमिका + स्कोप तक सीमित एक डिवाइस टोकन जारी करता है, जो `hello-ok.auth.deviceToken` में लौटाया जाता है। क्लाइंटों को
  किसी भी सफल कनेक्ट के बाद इसे स्थायी रूप से सहेजना चाहिए।
- उस संग्रहीत डिवाइस टोकन से पुनः कनेक्ट करते समय उस टोकन के लिए संग्रहीत
  अनुमोदित स्कोप सेट का भी पुनः उपयोग करना चाहिए। यह पहले से प्रदान की गई
  रीड/प्रोब/स्टेटस पहुँच सुरक्षित रखता है और पुनः कनेक्शन को चुपचाप केवल-एडमिन के संकरे
  अंतर्निहित स्कोप में सिमटने से बचाता है।
- क्लाइंट-पक्षीय कनेक्ट प्रमाणीकरण संयोजन (`packages/gateway-client/src/client.ts` में
  `selectConnectAuth`):
  - `auth.password` स्वतंत्र है और सेट होने पर हमेशा अग्रेषित किया जाता है।
  - `auth.token` को प्राथमिकता क्रम में भरा जाता है: पहले स्पष्ट साझा टोकन,
    फिर स्पष्ट `deviceToken`, और फिर संग्रहीत प्रति-डिवाइस टोकन (जिसकी कुंजी
    `deviceId` + `role` है)।
  - `auth.bootstrapToken` केवल तभी भेजा जाता है जब उपर्युक्त में से किसी ने
    `auth.token` को निर्धारित न किया हो। साझा टोकन या कोई भी निर्धारित डिवाइस टोकन इसे रोक देता है।
  - एक-बार वाले `AUTH_TOKEN_MISMATCH` पुनःप्रयास पर संग्रहीत डिवाइस टोकन का स्वतः-प्रचार
    केवल विश्वसनीय एंडपॉइंट तक सीमित है: लूपबैक,
    या पिन किए गए `tlsFingerprint` वाला `wss://`। बिना पिनिंग वाला सार्वजनिक `wss://`
    पात्र नहीं है।
- अंतर्निर्मित सेटअप-कोड बूटस्ट्रैप विश्वसनीय मोबाइल हैंडऑफ़ के लिए प्राथमिक Node
  `hello-ok.auth.deviceToken` और `hello-ok.auth.deviceTokens` में एक सीमित ऑपरेटर टोकन
  लौटाता है। ऑपरेटर टोकन में मूल Talk कॉन्फ़िगरेशन रीड के लिए
  `operator.talk.secrets` शामिल है, लेकिन पेयरिंग-परिवर्तन स्कोप और `operator.admin` शामिल नहीं हैं।
- जब कोई गैर-बेसलाइन सेटअप-कोड बूटस्ट्रैप अनुमोदन की प्रतीक्षा करता है,
  `PAIRING_REQUIRED` विवरण में `recommendedNextStep: "wait_then_retry"`,
  `retryable: true`, और `pauseReconnect: false` शामिल होते हैं। अनुरोध अनुमोदित होने या टोकन
  अमान्य होने तक उसी बूटस्ट्रैप टोकन से पुनः कनेक्ट करते रहें।
- `hello-ok.auth.deviceTokens` को केवल तभी स्थायी रूप से सहेजें जब कनेक्ट ने
  `wss://` या लूपबैक/स्थानीय पेयरिंग जैसे विश्वसनीय ट्रांसपोर्ट पर बूटस्ट्रैप
  प्रमाणीकरण का उपयोग किया हो।
- यदि कोई क्लाइंट स्पष्ट `deviceToken` या स्पष्ट `scopes` देता है, तो
  कॉलर द्वारा अनुरोधित वह स्कोप सेट प्रामाणिक बना रहता है; कैश किए गए स्कोप का पुनः उपयोग केवल
  तब होता है जब क्लाइंट संग्रहीत प्रति-डिवाइस टोकन का पुनः उपयोग कर रहा हो।
- डिवाइस टोकन को `device.token.rotate` और
  `device.token.revoke` के माध्यम से घुमाया/रद्द किया जा सकता है (`operator.pairing` आवश्यक है)। किसी
  Node या अन्य गैर-ऑपरेटर भूमिका को घुमाने या रद्द करने के लिए `operator.admin` भी आवश्यक है।
- `device.token.rotate` रोटेशन मेटाडेटा लौटाता है। यह प्रतिस्थापन
  बेयरर टोकन को केवल उसी डिवाइस की उन कॉल के लिए प्रतिध्वनित करता है जो उस
  डिवाइस टोकन से पहले ही प्रमाणीकृत हैं, ताकि केवल-टोकन क्लाइंट पुनः कनेक्ट होने से पहले
  अपना प्रतिस्थापन स्थायी रूप से सहेज सकें। साझा/एडमिन रोटेशन बेयरर टोकन को प्रतिध्वनित नहीं करते।
- टोकन जारी करना, घुमाना और रद्द करना उस डिवाइस की पेयरिंग प्रविष्टि में दर्ज
  अनुमोदित भूमिका सेट तक सीमित रहता है; टोकन परिवर्तन ऐसी डिवाइस भूमिका का विस्तार या
  लक्ष्य नहीं बना सकता जिसे पेयरिंग अनुमोदन ने कभी प्रदान नहीं किया।
- पेयर किए गए डिवाइस टोकन सत्रों के लिए, डिवाइस प्रबंधन स्वयं तक सीमित होता है, जब तक
  कॉलर के पास `operator.admin` भी न हो: गैर-एडमिन कॉलर केवल अपनी
  डिवाइस प्रविष्टि का ऑपरेटर टोकन प्रबंधित कर सकते हैं। Node और अन्य गैर-ऑपरेटर टोकन
  प्रबंधन केवल एडमिन के लिए है, भले ही वह कॉलर का अपना डिवाइस हो।
- `device.token.rotate` और `device.token.revoke` लक्षित
  ऑपरेटर टोकन स्कोप सेट की जाँच कॉलर के वर्तमान सत्र स्कोप के विरुद्ध भी करते हैं।
  गैर-एडमिन कॉलर अपने पास पहले से मौजूद स्कोप से अधिक व्यापक ऑपरेटर टोकन को घुमा या रद्द नहीं सकते।
- प्रमाणीकरण विफलताओं में `error.details.code` और पुनर्प्राप्ति संकेत शामिल होते हैं:
  - `error.details.canRetryWithDeviceToken` (बूलियन)
  - `error.details.recommendedNextStep`: `retry_with_device_token`,
    `update_auth_configuration`, `update_auth_credentials`,
    `wait_then_retry`, `review_auth_configuration`
    में से एक (`packages/gateway-protocol/src/connect-error-details.ts`)।
- `AUTH_TOKEN_MISMATCH` के लिए क्लाइंट व्यवहार:
  - विश्वसनीय क्लाइंट कैश किए गए प्रति-डिवाइस
    टोकन से एक सीमित पुनःप्रयास कर सकते हैं।
  - यदि वह पुनःप्रयास विफल हो जाए, तो स्वचालित पुनःकनेक्ट लूप रोकें और ऑपरेटर
    कार्रवाई संबंधी मार्गदर्शन प्रदर्शित करें।
- `AUTH_SCOPE_MISMATCH` का अर्थ है कि डिवाइस टोकन पहचाना गया था, लेकिन वह
  अनुरोधित भूमिका/स्कोप को कवर नहीं करता। इसे खराब टोकन के रूप में प्रस्तुत न करें; ऑपरेटर को
  पुनः पेयर करने या संकरे/व्यापक स्कोप अनुबंध को अनुमोदित करने के लिए संकेत दें।

## डिवाइस पहचान और पेयरिंग

- Node में की-पेयर फ़िंगरप्रिंट से प्राप्त एक स्थिर डिवाइस पहचान
  (`device.id`) शामिल होनी चाहिए।
- Gateway प्रति डिवाइस + भूमिका टोकन जारी करते हैं।
- नई डिवाइस ID के लिए पेयरिंग अनुमोदन आवश्यक हैं, जब तक स्थानीय
  स्वतः-अनुमोदन सक्षम न हो।
- पेयरिंग स्वतः-अनुमोदन प्रत्यक्ष स्थानीय लूपबैक कनेक्शन पर केंद्रित है।
- OpenClaw में विश्वसनीय साझा-गोपनीय सहायक प्रवाहों के लिए एक संकरा
  बैकएंड/कंटेनर-स्थानीय स्व-कनेक्ट पथ भी है।
- समान-होस्ट टेलनेट या LAN कनेक्शन को पेयरिंग के लिए अब भी दूरस्थ माना जाता है
  और उनके लिए अनुमोदन आवश्यक है।
- WS क्लाइंट सामान्यतः `connect` (ऑपरेटर +
  Node) के दौरान `device` पहचान शामिल करते हैं। डिवाइस-रहित ऑपरेटर के एकमात्र अपवाद स्पष्ट विश्वसनीय पथ हैं:
  - केवल-लोकलहोस्ट असुरक्षित
    HTTP संगतता के लिए `gateway.controlUi.allowInsecureAuth=true`।
  - सफल `gateway.auth.mode: "trusted-proxy"` ऑपरेटर Control UI प्रमाणीकरण।
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (आपातकालीन उपाय, गंभीर
    सुरक्षा अवनयन)।
  - आरक्षित आंतरिक
    सहायक पथ पर प्रत्यक्ष-लूपबैक `gateway-client` बैकएंड RPC।
- डिवाइस पहचान छोड़ने के स्कोप-संबंधी परिणाम होते हैं। जब किसी डिवाइस-रहित
  ऑपरेटर कनेक्शन को स्पष्ट विश्वसनीय पथ से अनुमति दी जाती है, तब भी OpenClaw
  स्वयं घोषित स्कोप को रिक्त सेट में साफ़ कर देता है, जब तक उस पथ में कोई
  नामित स्कोप-संरक्षण अपवाद न हो। तब स्कोप-नियंत्रित विधियाँ
  `missing scope` के साथ विफल होती हैं।
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` एक Control UI
  आपातकालीन स्कोप-संरक्षण पथ है। यह मनमाने
  कस्टम बैकएंड या CLI-आकृति वाले WebSocket क्लाइंटों को स्कोप प्रदान नहीं करता।
- आरक्षित प्रत्यक्ष-लूपबैक `gateway-client` बैकएंड सहायक पथ
  केवल आंतरिक स्थानीय कंट्रोल-प्लेन RPC के लिए स्कोप सुरक्षित रखता है; कस्टम बैकएंड ID को
  यह अपवाद नहीं मिलता।
- सभी कनेक्शनों को सर्वर द्वारा प्रदान किए गए `connect.challenge` नॉन्स पर हस्ताक्षर करने होंगे।

### डिवाइस प्रमाणीकरण माइग्रेशन निदान

उन लेगेसी क्लाइंटों के लिए जो अब भी प्री-चैलेंज हस्ताक्षर व्यवहार का उपयोग करते हैं, `connect`
स्थिर `error.details.reason` के साथ `error.details.code` के अंतर्गत `DEVICE_AUTH_*` विवरण कोड
लौटाता है।

सामान्य माइग्रेशन विफलताएँ:

| संदेश                     | details.code                     | details.reason           | अर्थ                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | क्लाइंट ने `device.nonce` छोड़ दिया (या रिक्त भेजा)।     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | क्लाइंट ने पुराने/गलत नॉन्स से हस्ताक्षर किया।            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | हस्ताक्षर पेलोड v2 पेलोड से मेल नहीं खाता।       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | हस्ताक्षरित टाइमस्टैम्प अनुमत विचलन से बाहर है।          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` सार्वजनिक कुंजी फ़िंगरप्रिंट से मेल नहीं खाता। |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | सार्वजनिक कुंजी प्रारूप/कैनोनिकलाइज़ेशन विफल रहा।         |

माइग्रेशन लक्ष्य:

- हमेशा `connect.challenge` की प्रतीक्षा करें।
- सर्वर नॉन्स शामिल करने वाले v2 पेलोड पर हस्ताक्षर करें।
- वही नॉन्स `connect.params.device.nonce` में भेजें।
- पसंदीदा हस्ताक्षर पेलोड `v3` है
  (`packages/gateway-client/src/device-auth.ts` में `buildDeviceAuthPayloadV3`),
  जो डिवाइस/क्लाइंट/भूमिका/स्कोप/टोकन/नॉन्स फ़ील्ड के अतिरिक्त
  `platform` और `deviceFamily` को भी संबद्ध करता है।
- संगतता के लिए पुराने `v2` हस्ताक्षर अभी भी स्वीकार किए जाते हैं, लेकिन पुनः कनेक्ट होने पर
  युग्मित-डिवाइस मेटाडेटा पिनिंग अब भी कमांड नीति को नियंत्रित करती है।

## TLS और पिनिंग

- WS कनेक्शन के लिए TLS समर्थित है (`gateway.tls` कॉन्फ़िगरेशन)।
- क्लाइंट वैकल्पिक रूप से `gateway.remote.tlsFingerprint` या CLI
  `--tls-fingerprint` के माध्यम से Gateway प्रमाणपत्र फ़िंगरप्रिंट पिन कर सकते हैं।
