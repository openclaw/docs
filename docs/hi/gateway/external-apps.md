बाहरी ऐप्स Gateway प्रोटोकॉल के माध्यम से OpenClaw से संचार करते हैं: WebSocket
ट्रांसपोर्ट और RPC विधियाँ। इसका उपयोग तब करें जब कोई स्क्रिप्ट, डैशबोर्ड, CI जॉब, IDE
एक्सटेंशन या कोई अन्य प्रक्रिया एजेंट रन शुरू करना, इवेंट स्ट्रीम करना, परिणामों की प्रतीक्षा
करना, कार्य रद्द करना या Gateway संसाधनों का निरीक्षण करना चाहती हो।

<Note>
  npm पैकेज, डिवाइस पेयरिंग, पुनः कनेक्शन रिकवरी, इतिहास, सदस्यताओं
  और अनुमोदनों के लिए
  [Gateway क्लाइंट बनाना](https://docs.openclaw.ai/gateway/clients) से शुरुआत करें। यदि आपका
  ऐप Gateway को चाइल्ड प्रोसेस के रूप में संचालित करता है, तो
  [OpenClaw एम्बेड करना](https://docs.openclaw.ai/gateway/embedding) भी पढ़ें। आरंभिक
  पैकेज रोलआउट के दौरान, पहला पैकेज-युक्त OpenClaw रिलीज़ प्रकाशित होने तक npm
  `E404` लौटा सकता है।
</Note>

<Note>
  यह पृष्ठ OpenClaw प्रक्रिया के बाहर के कोड के लिए है। OpenClaw के
  अंदर चलने वाले Plugin कोड को इसके बजाय दस्तावेज़ीकृत `openclaw/plugin-sdk/*` उप-पथों का उपयोग करना चाहिए।
</Note>

## अनुशंसित मार्ग

1. Gateway चलाएँ या खोजें।
2. [Gateway प्रोटोकॉल](/hi/gateway/protocol) के माध्यम से कनेक्ट करें।
3. [Gateway RPC संदर्भ](/hi/reference/rpc) में दस्तावेज़ीकृत RPC विधियों को कॉल करें।
4. जिस OpenClaw संस्करण पर आप परीक्षण करते हैं, उसे पिन करें।
5. OpenClaw अपग्रेड करते समय RPC संदर्भ दोबारा जाँचें।

एजेंट रन के लिए, `agent` RPC से शुरुआत करें और टर्मिनल परिणाम के लिए इसे
`agent.wait` के साथ जोड़ें। स्थायी वार्तालाप स्थिति के लिए `sessions.*` विधियों का उपयोग करें।
UI एकीकरणों के लिए, Gateway इवेंट की सदस्यता लें और केवल उन्हीं इवेंट
परिवारों को रेंडर करें जिन्हें आपका ऐप समझता है।

## सहयोगात्मक होस्ट निलंबन

चलती हुई प्रक्रिया को फ़्रीज़ या स्नैपशॉट करने वाले होस्टिंग नियंत्रक
होस्ट-निरपेक्ष निलंबन हैंडशेक का उपयोग कर सकते हैं:

1. होस्ट द्वारा नियंत्रित बाहरी इनग्रेस स्वीकार करना बंद करें।
2. स्थिर, अद्वितीय `requestId` के साथ `gateway.suspend.prepare` को कॉल करें।
3. यदि प्रतिक्रिया `busy` है, तो प्रक्रिया चलती रहने दें और बाद में पुनः प्रयास करें।
4. यदि यह `ready` है, तो लौटाया गया `suspensionId` सहेजें, फिर
   `expiresAtMs` से पहले प्रक्रिया को फ़्रीज़ या स्नैपशॉट करें।
5. अनफ़्रीज़ करने के बाद, या यदि निलंबन छोड़ दिया गया हो, तो मौजूदा WebSocket या
   Admin HTTP नियंत्रण पथ के माध्यम से उसी `suspensionId` के साथ
   `gateway.suspend.resume` को कॉल करें।

तैयार किया गया Gateway नए WebSocket हैंडशेक अस्वीकार करता है। WebSocket नियंत्रक
को होस्ट कार्रवाई के दौरान अपना प्रमाणित कनेक्शन खुला रखना होगा। यदि इसकी
गारंटी नहीं दी जा सकती, तो तैयारी करने से पहले
[Admin HTTP RPC Plugin](/hi/plugins/admin-http-rpc) सक्षम करके उसका उपयोग करें। यदि
नियंत्रण पथ खो जाता है, तो पुनः कनेक्ट करने से पहले दो मिनट की लीज़ समाप्त होने की
प्रतीक्षा करें; समाप्ति स्वतः प्रवेश दोबारा खोल देती है।

RPC अनुबंध यह है:

- `gateway.suspend.prepare` — `operator.admin`; पैरामीटर
  `{ "requestId": "stable-host-operation-id" }`
- `gateway.suspend.status` — `operator.read`; पैरामीटर
  `{ "suspensionId": "id-from-prepare" }`
- `gateway.suspend.resume` — `operator.admin`; पैरामीटर
  `{ "suspensionId": "id-from-prepare" }`

ID ट्रिम किए जाते हैं, उनमें कोई गैर-व्हाइटस्पेस वर्ण होना आवश्यक है और उनकी सीमा
128 वर्ण है। व्यस्त तैयारी परिणाम में `status: "busy"`, `reason`,
`retryAfterMs`, `activeCount` और `blockers` होते हैं। तैयार परिणाम का स्वरूप यह है:

```json
{
  "status": "ready",
  "suspensionId": "2c3f...",
  "expiresAtMs": 1770000000000,
  "activeCount": 0,
  "blockers": []
}
```

स्थिति `{"status":"running"}` या `expiresAtMs` वाला तैयार परिणाम लौटाती है।
पुनः आरंभ `{"ok":true,"status":"running","resumed":true}` लौटाता है; सफल पुनः आरंभ के बाद इसे
दोहराने पर `resumed: false` लौटता है।

प्रतिस्पर्धी अनुरोध ID या अस्थायी शेड्यूलर-पुनः आरंभ विफलता
`retryAfterMs` के साथ पुनः प्रयास योग्य `UNAVAILABLE` लौटाती है। शेड्यूलर रिकवरी के दौरान तैयारी, स्थिति
और पुनः आरंभ सभी वही त्रुटि लौटाते हैं, Gateway तैयार नहीं और
विफलता पर बंद रहता है, और होस्ट को इसे फ़्रीज़ या स्नैपशॉट नहीं करना चाहिए। OpenClaw
शेड्यूलर के लिए स्वतः पुनः प्रयास करता है और रिकवरी सफल होने के बाद ही प्रवेश
दोबारा खोलता है। असंगत पुनः आरंभ ID `INVALID_REQUEST` लौटाता है। तैयारी Gateway के
प्रति मिनट तीन प्रयासों वाले नियंत्रण-प्लेन लेखन बजट को साझा करती है; लौटाई गई
पुनः प्रयास देरी का पालन करें। WebSocket क्लाइंट डिवाइस और IP के अनुसार बकेट किए जाते हैं। Admin HTTP
नियंत्रक हल किए गए क्लाइंट IP के अनुसार बकेट किए जाते हैं, इसलिए एक
प्रॉक्सी के पीछे के नियंत्रक बजट साझा कर सकते हैं।

तैयारी केवल अस्वीकार करने वाली है: OpenClaw नए रूट/सत्र/कमांड प्रवेश को बंद करता है,
स्वचालित Cron टिक रोकता है और कार्य का समकालिक रूप से निरीक्षण करता है। यदि कुछ भी
सक्रिय है, तो यह शेड्यूलर को पुनः आरंभ करता है और `busy` लौटाने से पहले
प्रवेश दोबारा खोलता है; यह उस कार्य को बाधित या समाप्त होने तक रिक्त नहीं करता। तैयार लीज़ दो
मिनट तक चलती है। समान `requestId` के साथ `prepare` दोहराने पर इसका नवीनीकरण होता है; समाप्ति
प्रवेश दोबारा खोलने से पहले शेड्यूलर को पुनः आरंभ करती है।
तैयार लीज़ के दौरान देय होने वाला पुनः आरंभ उत्सर्जन लीज़ के पुनः आरंभ होने तक प्रतीक्षा
करता है; प्रगति पर मौजूद पुनः आरंभ के कारण तैयारी `busy` लौटाती है।

तैयार अवस्था में, `/healthz` सक्रिय रहता है और `/readyz` `503` लौटाता है। स्थानीय या
प्रमाणित तत्परता प्रतिक्रियाओं में `gateway-draining` शामिल होता है; अप्रमाणित
दूरस्थ प्रोब को केवल `{ "ready": false }` मिलता है। HTTP स्वास्थ्य प्रोब,
मौजूदा WebSocket कनेक्शन पर निलंबन विधियाँ और पहले से सक्षम
Admin HTTP RPC रूट उपलब्ध रहते हैं। अन्य RPC पुनः प्रयास योग्य
`UNAVAILABLE` लौटाते हैं। अंतर्निहित HTTP उपयोगकर्ता-कार्य रूट और सामान्य Plugin HTTP रूट,
जिनमें OpenAI-संगत API, टूल/सत्र कार्रवाइयाँ, Node वॉच और
कॉन्फ़िगर किए गए हुक शामिल हैं, `error.code: "gateway_unavailable"` के साथ `503` लौटाते हैं। नए
Plugin-स्वामित्व वाले WebSocket अपग्रेड भी `503` लौटाते हैं; यह अपग्रेड
स्वामित्व को कवर करता है, स्थापित Plugin सॉकेट पर बाद में किए गए कार्य को नहीं।

यह हैंडशेक आने वाले संदेशों को स्थायी नहीं करता, तृतीय-पक्ष चैनल
ट्रांसपोर्ट को नहीं रोकता या होस्टिंग प्लेटफ़ॉर्म को नियंत्रित नहीं करता। होस्ट को तैयारी से
पहले अपने इनग्रेस को अवरुद्ध करना होगा और वेक, स्नैपशॉट/फ़्रीज़ तथा
रोकने के लिए वही उत्तरदायी रहता है। `activeCount` कुल ट्रैक किए गए कार्य की संख्या है, जबकि `blockers`
में गैर-शून्य श्रेणी संख्याएँ और सीमित कार्य विवरण होते हैं। यह
सामान्य प्रक्रिया-निष्क्रियता अवरोध नहीं है। `background-exec` अवरोधक केवल समग्र
होता है: कमांड टेक्स्ट, प्रक्रिया ID, आउटपुट और सत्र या स्कोप पहचानकर्ता कभी
प्रोटोकॉल पार नहीं करते। चैनल स्वास्थ्य, रखरखाव, कैश रीफ़्रेश, स्थापित
Plugin WebSocket सत्र और अपंजीकृत Plugin-स्वामित्व वाला पृष्ठभूमि कार्य
सक्रिय रह सकते हैं।
होस्टिंग प्लेटफ़ॉर्म को पूरी प्रक्रिया ट्री और उसके
फ़ाइल सिस्टम को सुसंगत रूप से फ़्रीज़ या स्नैपशॉट करना होगा; इस पहले अनुबंध से अपंजीकृत कार्य के निष्क्रिय होने को
सिद्ध नहीं किया जा सकता।

<Tip>
  होस्ट वेक शेड्यूलिंग के लिए, OpenClaw-संबंधी भाग को इन-प्रोसेस
  Plugin में रखें और आइडेम्पोटेंट पूर्ण स्नैपशॉट को बाहरी होस्ट अडैप्टर पर प्रक्षेपित करें।
  होस्टिंग नियंत्रक को Plugin SDK आयात नहीं करना चाहिए या इवेंट डेल्टा से Cron
  स्थिति का पुनर्निर्माण नहीं करना चाहिए। [सुरक्षित बाहरी Cron
  प्रक्षेपण](/hi/plugins/hooks#safe-external-cron-projection) देखें।
</Tip>

## ऐप कोड बनाम Plugin कोड

जब कोड OpenClaw के बाहर हो, तब Gateway RPC का उपयोग करें:

- एजेंट रन शुरू करने या देखने वाली Node स्क्रिप्ट
- Gateway को कॉल करने वाले CI जॉब
- डैशबोर्ड और एडमिन पैनल
- IDE एक्सटेंशन
- ऐसे बाहरी ब्रिज जिन्हें चैनल Plugin बनने की आवश्यकता नहीं है
- नकली या वास्तविक Gateway ट्रांसपोर्ट वाले एकीकरण परीक्षण

जब कोड OpenClaw के अंदर चले, तब Plugin SDK का उपयोग करें:

- प्रदाता Plugin
- चैनल Plugin
- टूल या जीवनचक्र हुक
- एजेंट हार्नेस Plugin
- विश्वसनीय रनटाइम सहायक

बाहरी ऐप्स को `openclaw/plugin-sdk/*` आयात नहीं करना चाहिए; वे उप-पथ
OpenClaw द्वारा लोड किए गए Plugin के लिए हैं।
