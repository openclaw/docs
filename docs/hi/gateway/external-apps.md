बाहरी ऐप्स Gateway प्रोटोकॉल के माध्यम से OpenClaw से संवाद करते हैं: WebSocket
ट्रांसपोर्ट और RPC विधियाँ। इसका उपयोग तब करें जब कोई स्क्रिप्ट, डैशबोर्ड, CI जॉब, IDE
एक्सटेंशन या कोई अन्य प्रक्रिया एजेंट रन शुरू करना, इवेंट स्ट्रीम करना, परिणामों की प्रतीक्षा
करना, कार्य रद्द करना या Gateway संसाधनों का निरीक्षण करना चाहती हो।

<Warning>
  अभी कोई सार्वजनिक npm क्लाइंट पैकेज उपलब्ध नहीं है। जब तक रिलीज़ नोट्स किसी प्रकाशित
  पैकेज की घोषणा न करें और इस पृष्ठ पर इंस्टॉल करने के निर्देश शामिल न हों, तब तक OpenClaw
  क्लाइंट पैकेज के नामों को एप्लिकेशन डिपेंडेंसी के रूप में न जोड़ें।
</Warning>

<Note>
  यह पृष्ठ OpenClaw प्रक्रिया के बाहर के कोड के लिए है। OpenClaw के भीतर चलने वाले
  Plugin कोड को इसके बजाय दस्तावेज़ीकृत `openclaw/plugin-sdk/*` सबपाथ का उपयोग करना चाहिए।
</Note>

भविष्य का एक क्लाइंट लाइब्रेरी पैकेज आंतरिक रूप से प्रगति पर है, लेकिन यह अभी
सार्वजनिक इंस्टॉल सतह नहीं है। किसी रिलीज़ द्वारा प्रकाशित, वर्ज़नयुक्त पैकेज की घोषणा होने तक
इसे पूर्वावलोकन कार्यान्वयन विवरण मानें।

## अनुशंसित मार्ग

1. Gateway चलाएँ या खोजें।
2. [Gateway प्रोटोकॉल](/hi/gateway/protocol) के माध्यम से कनेक्ट करें।
3. [Gateway RPC संदर्भ](/hi/reference/rpc) में दस्तावेज़ीकृत RPC विधियों को कॉल करें।
4. जिस OpenClaw वर्ज़न के विरुद्ध आप परीक्षण करते हैं, उसे पिन करें।
5. OpenClaw अपग्रेड करते समय RPC संदर्भ दोबारा जाँचें।

एजेंट रन के लिए, `agent` RPC से शुरू करें और टर्मिनल परिणाम के लिए इसे
`agent.wait` के साथ जोड़ें। स्थायी वार्तालाप स्थिति के लिए, `sessions.*` विधियों का उपयोग करें।
UI इंटीग्रेशन के लिए, Gateway इवेंट की सदस्यता लें और केवल उन्हीं इवेंट
फ़ैमिली को रेंडर करें जिन्हें आपका ऐप समझता है।

## सहयोगात्मक होस्ट निलंबन

चलती हुई प्रक्रिया को फ़्रीज़ या स्नैपशॉट करने वाले होस्टिंग कंट्रोलर
होस्ट-न्यूट्रल निलंबन हैंडशेक का उपयोग कर सकते हैं:

1. होस्ट द्वारा नियंत्रित बाहरी इनग्रेस स्वीकार करना बंद करें।
2. स्थिर, अद्वितीय `requestId` के साथ `gateway.suspend.prepare` को कॉल करें।
3. यदि प्रतिक्रिया `busy` है, तो प्रक्रिया को चालू रखें और बाद में पुनः प्रयास करें।
4. यदि यह `ready` है, तो लौटाया गया `suspensionId` सहेजें, फिर
   `expiresAtMs` से पहले प्रक्रिया को फ़्रीज़ या स्नैपशॉट करें।
5. थॉ होने के बाद, या निलंबन छोड़ दिए जाने पर, मौजूदा WebSocket या
   Admin HTTP नियंत्रण पथ के माध्यम से उसी `suspensionId` के साथ
   `gateway.suspend.resume` को कॉल करें।

तैयार किया गया Gateway नए WebSocket हैंडशेक अस्वीकार करता है। WebSocket कंट्रोलर को
होस्ट कार्रवाई के दौरान अपना प्रमाणीकृत कनेक्शन खुला रखना होगा। यदि इसकी
गारंटी नहीं दी जा सकती, तो तैयारी से पहले
[Admin HTTP RPC Plugin](/hi/plugins/admin-http-rpc) सक्षम करें और उसका उपयोग करें। यदि
नियंत्रण पथ खो जाए, तो दो-मिनट की लीज़ समाप्त होने की प्रतीक्षा करने के बाद
पुनः कनेक्ट करें; अवधि समाप्त होने पर प्रवेश स्वचालित रूप से फिर खुल जाता है।

RPC अनुबंध यह है:

- `gateway.suspend.prepare` — `operator.admin`; पैरामीटर
  `{ "requestId": "stable-host-operation-id" }`
- `gateway.suspend.status` — `operator.read`; पैरामीटर
  `{ "suspensionId": "id-from-prepare" }`
- `gateway.suspend.resume` — `operator.admin`; पैरामीटर
  `{ "suspensionId": "id-from-prepare" }`

ID के किनारों की रिक्त जगह हटाई जाती है, उनमें कोई गैर-व्हाइटस्पेस वर्ण होना आवश्यक है और वे
128 वर्णों तक सीमित हैं। व्यस्त तैयारी परिणाम में `status: "busy"`, `reason`,
`retryAfterMs`, `activeCount` और `blockers` होते हैं। तैयार परिणाम का स्वरूप यह है:

```json
{
  "status": "ready",
  "suspensionId": "2c3f...",
  "expiresAtMs": 1770000000000,
  "activeCount": 0,
  "blockers": []
}
```

स्थिति `{"status":"running"}` या `expiresAtMs` वाला तैयार परिणाम लौटाती है।
फिर से शुरू करना `{"ok":true,"status":"running","resumed":true}` लौटाता है; सफलतापूर्वक फिर से शुरू करने के बाद
इसे दोहराने पर `resumed: false` लौटता है।

प्रतिस्पर्धी अनुरोध ID या अस्थायी शेड्यूलर-पुनःआरंभ विफलता
`retryAfterMs` के साथ पुनः प्रयास योग्य `UNAVAILABLE` लौटाती है। शेड्यूलर पुनर्प्राप्ति के दौरान,
तैयारी, स्थिति और पुनःआरंभ सभी वही त्रुटि लौटाते हैं, Gateway तैयार नहीं और
फेल-क्लोज़्ड रहता है, और होस्ट को इसे फ़्रीज़ या स्नैपशॉट नहीं करना चाहिए। OpenClaw
शेड्यूलर का स्वचालित रूप से पुनः प्रयास करता है और पुनर्प्राप्ति सफल होने के बाद ही
प्रवेश फिर खोलता है। मेल न खाने वाला पुनःआरंभ ID `INVALID_REQUEST` लौटाता है। तैयारी में Gateway का
प्रति मिनट तीन प्रयासों वाला नियंत्रण-प्लेन लेखन बजट साझा होता है; लौटाई गई
पुनः प्रयास देरी का पालन करें। WebSocket क्लाइंट को डिवाइस और IP के अनुसार बकेट किया जाता है। Admin HTTP
कंट्रोलर को समाधान किए गए क्लाइंट IP के अनुसार बकेट किया जाता है, इसलिए एक ही
प्रॉक्सी के पीछे मौजूद कंट्रोलर बजट साझा कर सकते हैं।

तैयारी केवल-अस्वीकार प्रकार की है: OpenClaw नए रूट/सेशन/कमांड प्रवेश बंद करता है,
स्वचालित Cron टिक रोकता है और कार्य का समकालिक रूप से निरीक्षण करता है। यदि कुछ भी
सक्रिय है, तो यह `busy` लौटाने से पहले शेड्यूलर फिर शुरू करके प्रवेश
फिर खोल देता है; यह उस कार्य को बाधित या ड्रेन नहीं करता। तैयार लीज़ दो
मिनट तक रहती है। उसी `requestId` के साथ `prepare` दोहराने पर उसका नवीनीकरण होता है; अवधि समाप्त होने पर
प्रवेश फिर खोलने से पहले शेड्यूलर पुनः शुरू होता है।
तैयार लीज़ के दौरान नियत हुआ पुनःआरंभ उत्सर्जन लीज़ के पुनःआरंभ होने तक प्रतीक्षा करता है;
चल रहा पुनःआरंभ तैयारी से `busy` लौटवाता है।

तैयार होने के दौरान, `/healthz` सक्रिय रहता है और `/readyz`, `503` लौटाता है। स्थानीय या
प्रमाणीकृत तत्परता प्रतिक्रियाओं में `gateway-draining` शामिल होता है; अप्रमाणीकृत
रिमोट प्रोब को केवल `{ "ready": false }` मिलता है। HTTP स्वास्थ्य प्रोब,
मौजूदा WebSocket कनेक्शन पर निलंबन विधियाँ और पहले से सक्षम
Admin HTTP RPC रूट उपलब्ध रहते हैं। अन्य RPC पुनः प्रयास योग्य
`UNAVAILABLE` लौटाते हैं। अंतर्निहित HTTP उपयोगकर्ता-कार्य रूट और सामान्य Plugin HTTP रूट,
जिनमें OpenAI-संगत API, टूल/सेशन कार्रवाइयाँ, Node वॉच और
कॉन्फ़िगर किए गए हुक शामिल हैं, `error.code: "gateway_unavailable"` के साथ `503` लौटाते हैं। नए
Plugin-स्वामित्व वाले WebSocket अपग्रेड भी `503` लौटाते हैं; इसमें अपग्रेड
स्वामित्व शामिल है, स्थापित Plugin सॉकेट पर बाद में किया गया कार्य नहीं।

यह हैंडशेक आने वाले संदेशों को स्थायी नहीं करता, तृतीय-पक्ष चैनल
ट्रांसपोर्ट को नहीं रोकता और होस्टिंग प्लेटफ़ॉर्म को नियंत्रित नहीं करता। होस्ट को तैयारी से पहले
अपने इनग्रेस की घेराबंदी करनी होगी और वह वेक, स्नैपशॉट/फ़्रीज़ तथा
स्टॉप के लिए उत्तरदायी रहता है। `activeCount` समेकित ट्रैक किए गए कार्य की संख्या है, जबकि `blockers`
में गैर-शून्य श्रेणी संख्याएँ और सीमित टास्क विवरण होते हैं। यह
सामान्य प्रक्रिया-निष्क्रियता अवरोध नहीं है। `background-exec` ब्लॉकर केवल समेकित होता है:
कमांड टेक्स्ट, प्रक्रिया ID, आउटपुट और सेशन या स्कोप पहचानकर्ता कभी
प्रोटोकॉल पार नहीं करते। चैनल स्वास्थ्य, रखरखाव, कैश रीफ़्रेश, स्थापित
Plugin WebSocket सेशन और अपंजीकृत Plugin-स्वामित्व वाला पृष्ठभूमि कार्य
सक्रिय रह सकता है।
होस्टिंग प्लेटफ़ॉर्म को संपूर्ण प्रक्रिया ट्री और उसके फ़ाइल सिस्टम को
सुसंगत रूप से फ़्रीज़ या स्नैपशॉट करना होगा; इस पहले अनुबंध द्वारा अपंजीकृत कार्य के निष्क्रिय होने को
सिद्ध नहीं किया जा सकता।

<Tip>
  होस्ट वेक शेड्यूलिंग के लिए, OpenClaw-सामना करने वाले भाग को इन-प्रोसेस
  Plugin में रखें और आइडेम्पोटेंट पूर्ण स्नैपशॉट को बाहरी होस्ट अडैप्टर में प्रोजेक्ट करें।
  होस्टिंग कंट्रोलर को Plugin SDK इंपोर्ट नहीं करना चाहिए या इवेंट डेल्टा से Cron
  स्थिति का पुनर्निर्माण नहीं करना चाहिए। [सुरक्षित बाहरी Cron
  प्रोजेक्शन](/hi/plugins/hooks#safe-external-cron-projection) देखें।
</Tip>

## ऐप कोड बनाम Plugin कोड

जब कोड OpenClaw के बाहर रहता है, तो Gateway RPC का उपयोग करें:

- एजेंट रन शुरू करने या उनका निरीक्षण करने वाली Node स्क्रिप्ट
- Gateway को कॉल करने वाले CI जॉब
- डैशबोर्ड और एडमिन पैनल
- IDE एक्सटेंशन
- ऐसे बाहरी ब्रिज जिन्हें चैनल Plugin बनने की आवश्यकता नहीं है
- नकली या वास्तविक Gateway ट्रांसपोर्ट वाले इंटीग्रेशन परीक्षण

जब कोड OpenClaw के भीतर चलता है, तो Plugin SDK का उपयोग करें:

- प्रोवाइडर Plugin
- चैनल Plugin
- टूल या लाइफ़साइकल हुक
- एजेंट हार्नेस Plugin
- विश्वसनीय रनटाइम सहायक

बाहरी ऐप्स को `openclaw/plugin-sdk/*` इंपोर्ट नहीं करना चाहिए; वे सबपाथ
OpenClaw द्वारा लोड किए गए Plugin के लिए हैं।
