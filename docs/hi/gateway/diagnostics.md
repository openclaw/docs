OpenClaw बग रिपोर्ट के लिए एक स्थानीय डायग्नोस्टिक्स `.zip` बना सकता है: सैनिटाइज़ किया गया Gateway
स्टेटस, हेल्थ, लॉग, कॉन्फ़िग आकार और हाल की पेलोड-रहित स्थिरता घटनाएँ।

समीक्षा होने तक डायग्नोस्टिक्स बंडल को सीक्रेट की तरह संभालें। पेलोड और क्रेडेंशियल
डिज़ाइन के अनुसार रिडैक्ट किए जाते हैं, लेकिन बंडल फिर भी स्थानीय Gateway लॉग और
होस्ट-स्तरीय रनटाइम स्थिति का सारांश देता है।

## त्वरित शुरुआत

```bash
openclaw gateway diagnostics export
```

लिखी गई zip का पथ प्रिंट करता है। आउटपुट पथ चुनें:

```bash
openclaw gateway diagnostics export --output openclaw-diagnostics.zip
```

ऑटोमेशन के लिए:

```bash
openclaw gateway diagnostics export --json
```

## चैट कमांड

स्थानीय Gateway एक्सपोर्ट को कॉपी-पेस्ट किए जा सकने वाले एक सपोर्ट रिपोर्ट के रूप में
अनुरोध करने के लिए स्वामी किसी भी बातचीत में `/diagnostics [note]` चला सकते हैं:

1. `/diagnostics` भेजें, वैकल्पिक रूप से एक संक्षिप्त नोट के साथ (`/diagnostics bad tool choice`)।
2. OpenClaw एक प्रस्तावना भेजता है और एक स्पष्ट exec अनुमोदन माँगता है, जो
   `openclaw gateway diagnostics export --json` चलाता है। किसी
   allow-all नियम के माध्यम से डायग्नोस्टिक्स को अनुमोदित न करें।
3. अनुमोदन के बाद, OpenClaw स्थानीय बंडल पथ, मैनिफ़ेस्ट
   सारांश, गोपनीयता नोट और प्रासंगिक सेशन आईडी के साथ उत्तर देता है।

समूह चैट में स्वामी अब भी `/diagnostics` चला सकते हैं, लेकिन OpenClaw
एक्सपोर्ट परिणाम, अनुमोदन प्रॉम्प्ट और Codex सेशन/थ्रेड विवरण
स्वामी को निजी रूप से भेजता है। समूह को केवल एक संक्षिप्त सूचना दिखाई देती है कि डायग्नोस्टिक्स
निजी रूप से भेजे गए हैं। यदि स्वामी के लिए कोई निजी रूट मौजूद नहीं है, तो कमांड सुरक्षित रूप से विफल होता है और
स्वामी से इसे DM से चलाने के लिए कहता है।

जब सक्रिय सेशन नेटिव OpenAI Codex हार्नेस का उपयोग करता है, तो वही exec
अनुमोदन उन Codex थ्रेड के लिए OpenAI फ़ीडबैक अपलोड को भी कवर करता है जिनके बारे में OpenClaw
जानता है। वह अपलोड स्थानीय Gateway zip से अलग है और केवल
Codex हार्नेस सेशन के लिए होता है। अनुमोदन प्रॉम्प्ट बताता है कि अनुमोदन करने पर
Codex फ़ीडबैक भी भेजा जाता है, लेकिन Codex सेशन या थ्रेड आईडी सूचीबद्ध नहीं करता। अनुमोदन के
बाद, उत्तर OpenAI को भेजे गए थ्रेड के लिए चैनल, OpenClaw सेशन आईडी, Codex थ्रेड आईडी और
स्थानीय रिज़्यूम कमांड सूचीबद्ध करता है। अनुमोदन अस्वीकार करने या
अनदेखा करने पर एक्सपोर्ट, Codex फ़ीडबैक अपलोड और
Codex आईडी सूची छोड़ दी जाती है।

इससे Codex डीबगिंग लूप छोटा हो जाता है: किसी चैनल में खराब व्यवहार पर ध्यान दें,
`/diagnostics` चलाएँ, एक बार अनुमोदन करें, रिपोर्ट साझा करें, फिर यदि आप स्वयं थ्रेड का निरीक्षण
करना चाहते हैं तो प्रिंट किया गया `codex resume <thread-id>` कमांड स्थानीय रूप से चलाएँ।
[Codex हार्नेस](/hi/plugins/codex-harness#inspect-codex-threads-locally) देखें।

## एक्सपोर्ट में क्या शामिल होता है

- `summary.md`: सपोर्ट के लिए मानव-पठनीय अवलोकन।
- `diagnostics.json`: कॉन्फ़िग, लॉग, स्टेटस, हेल्थ
  और स्थिरता डेटा का मशीन-पठनीय सारांश।
- `manifest.json`: एक्सपोर्ट मेटाडेटा और फ़ाइल सूची।
- सैनिटाइज़ किया गया कॉन्फ़िग आकार और गैर-गोपनीय कॉन्फ़िग विवरण।
- सैनिटाइज़ किए गए लॉग सारांश और हाल की रिडैक्ट की गई लॉग पंक्तियाँ।
- सर्वोत्तम-प्रयास Gateway स्टेटस और हेल्थ स्नैपशॉट।
- `stability/latest.json`: उपलब्ध होने पर नवीनतम स्थायी स्थिरता बंडल।

Gateway के अस्वस्थ होने पर भी एक्सपोर्ट उपयोगी रहता है: यदि स्टेटस/हेल्थ
अनुरोध विफल होते हैं, तो उपलब्ध होने पर स्थानीय लॉग, कॉन्फ़िग आकार और नवीनतम स्थिरता बंडल
फिर भी एकत्र किए जाते हैं।

## गोपनीयता मॉडल

रखा जाता है: सबसिस्टम नाम, Plugin आईडी, प्रोवाइडर आईडी, चैनल आईडी, कॉन्फ़िगर किए गए
मोड, स्टेटस कोड, अवधियाँ, बाइट गणनाएँ, क्यू स्थिति, मेमोरी रीडिंग,
सैनिटाइज़ किया गया लॉग मेटाडेटा, रिडैक्ट किए गए संचालन संदेश, कॉन्फ़िग आकार और
गैर-गोपनीय फ़ीचर सेटिंग।

हटाया या रिडैक्ट किया जाता है: चैट टेक्स्ट, प्रॉम्प्ट, निर्देश, Webhook बॉडी, टूल
आउटपुट, क्रेडेंशियल, API कुंजियाँ, टोकन, कुकी, सीक्रेट मान, रॉ
अनुरोध/प्रतिक्रिया बॉडी, अकाउंट आईडी, संदेश आईडी, रॉ सेशन आईडी,
होस्टनाम और स्थानीय उपयोगकर्ता नाम।

जब कोई लॉग संदेश उपयोगकर्ता, चैट, प्रॉम्प्ट या टूल पेलोड टेक्स्ट जैसा दिखता है, तो
एक्सपोर्ट केवल यह जानकारी रखता है कि संदेश हटाया गया था और उसकी बाइट गणना क्या थी।

## स्थिरता रिकॉर्डर

डायग्नोस्टिक्स सक्षम होने पर Gateway डिफ़ॉल्ट रूप से एक सीमित, पेलोड-रहित स्थिरता स्ट्रीम
रिकॉर्ड करता है। यह संचालन संबंधी तथ्य कैप्चर करता है, सामग्री नहीं।

जब इवेंट लूप या CPU संतृप्त दिखाई देता है, तो वही Heartbeat सक्रियता का भी नमूना लेता है,
और इवेंट-लूप विलंब, इवेंट-लूप उपयोग, CPU-कोर अनुपात, सक्रिय/प्रतीक्षारत/कतारबद्ध सेशन गणना,
वर्तमान स्टार्टअप/रनटाइम चरण (ज्ञात होने पर), हाल के चरण विस्तार और
सीमित कार्य लेबल के साथ `diagnostic.liveness.warning` घटनाएँ उत्सर्जित करता है।
ये Gateway की `warn`-स्तरीय लॉग पंक्तियाँ केवल तभी बनती हैं जब
कार्य प्रतीक्षारत या कतारबद्ध हो, या सक्रिय कार्य निरंतर इवेंट-लूप
विलंब के साथ ओवरलैप करे; अन्यथा वे `debug` पर लॉग होती हैं। निष्क्रिय सक्रियता नमूने फिर भी
डायग्नोस्टिक घटनाओं के रूप में रिकॉर्ड होते हैं, लेकिन अपने आप कभी चेतावनी तक नहीं बढ़ते।

स्टार्टअप चरण वॉल-क्लॉक और
CPU समय के साथ `diagnostic.phase.completed` घटनाएँ उत्सर्जित करते हैं। रुके हुए एम्बेडेड-रन डायग्नोस्टिक्स `terminalProgressStale=true` चिह्नित करते हैं,
जब अंतिम ब्रिज प्रगति टर्मिनल जैसी दिखाई देती थी (उदाहरण के लिए रॉ प्रतिक्रिया
आइटम या प्रतिक्रिया-पूर्णता घटना), लेकिन Gateway अब भी
एम्बेडेड रन को सक्रिय मानता है।

लाइव रिकॉर्डर का निरीक्षण करें:

```bash
openclaw gateway stability
openclaw gateway stability --type payload.large
openclaw gateway stability --json
```

घातक निकास, शटडाउन टाइमआउट या
रीस्टार्ट स्टार्टअप विफलता के बाद नवीनतम स्थायी बंडल का निरीक्षण करें:

```bash
openclaw gateway stability --bundle latest
```

नवीनतम स्थायी बंडल से डायग्नोस्टिक्स zip बनाएँ:

```bash
openclaw gateway stability --bundle latest --export
```

घटनाएँ मौजूद होने पर स्थायी बंडल `~/.openclaw/logs/stability/` के अंतर्गत रहते हैं।

## उपयोगी विकल्प

```bash
openclaw gateway diagnostics export \
  --output openclaw-diagnostics.zip \
  --log-lines 5000 \
  --log-bytes 1000000
```

| फ़्लैग                    | डिफ़ॉल्ट                                                                       | विवरण                                        |
| ----------------------- | ----------------------------------------------------------------------------- | -------------------------------------------------- |
| `--output <path>`       | `$OPENCLAW_STATE_DIR/logs/support/openclaw-diagnostics-<timestamp>-<pid>.zip` | किसी विशिष्ट zip पथ (या डायरेक्टरी) में लिखें।       |
| `--log-lines <count>`   | `5000`                                                                        | शामिल करने के लिए सैनिटाइज़ की गई लॉग पंक्तियों की अधिकतम संख्या।            |
| `--log-bytes <bytes>`   | `1000000`                                                                     | निरीक्षण के लिए लॉग बाइट की अधिकतम संख्या।                      |
| `--url <url>`           | -                                                                             | स्टेटस/हेल्थ स्नैपशॉट के लिए Gateway WebSocket URL। |
| `--token <token>`       | -                                                                             | स्टेटस/हेल्थ स्नैपशॉट के लिए Gateway टोकन।         |
| `--password <password>` | -                                                                             | स्टेटस/हेल्थ स्नैपशॉट के लिए Gateway पासवर्ड।      |
| `--timeout <ms>`        | `3000`                                                                        | स्टेटस/हेल्थ स्नैपशॉट टाइमआउट।                    |
| `--no-stability-bundle` | बंद                                                                           | स्थायी स्थिरता बंडल की खोज छोड़ें।            |
| `--json`                | बंद                                                                           | मशीन-पठनीय एक्सपोर्ट मेटाडेटा प्रिंट करें।            |

## डायग्नोस्टिक्स अक्षम करें

डायग्नोस्टिक्स डिफ़ॉल्ट रूप से सक्षम हैं। स्थिरता रिकॉर्डर और
डायग्नोस्टिक घटना संग्रह को अक्षम करने के लिए:

```json5
{
  diagnostics: {
    enabled: false,
  },
}
```

डायग्नोस्टिक्स अक्षम करने से बग रिपोर्ट का विवरण कम हो जाता है; इससे सामान्य
Gateway लॉगिंग प्रभावित नहीं होती।

गंभीर मेमोरी दबाव स्नैपशॉट डिफ़ॉल्ट रूप से बंद हैं। सामान्य डायग्नोस्टिक्स घटनाओं के अतिरिक्त
OOM-पूर्व स्थिरता स्नैपशॉट कैप्चर करने के लिए:

```json5
{
  diagnostics: {
    memoryPressureSnapshot: true,
  },
}
```

इसका उपयोग केवल उन होस्ट पर करें जो गंभीर मेमोरी दबाव के दौरान अतिरिक्त फ़ाइल-सिस्टम स्कैन और
स्नैपशॉट लेखन सहन कर सकते हैं। स्नैपशॉट बंद होने पर भी सामान्य मेमोरी दबाव घटनाएँ
RSS, हीप, थ्रेशोल्ड और वृद्धि संबंधी तथ्य (`rss_threshold`,
`heap_threshold`, `rss_growth`) रिकॉर्ड करती हैं।
