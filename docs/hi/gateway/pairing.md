Node पेयरिंग की दो परतें हैं, और दोनों पेयर किए गए डिवाइस रिकॉर्ड पर
Gateway के SQLite स्टेट डेटाबेस में संग्रहीत होती हैं:

- **डिवाइस पेयरिंग** (भूमिका `node`) `connect` हैंडशेक को नियंत्रित करती है। नीचे
  [विश्वसनीय-CIDR डिवाइस स्वतः-अनुमोदन](#trusted-cidr-device-auto-approval)
  और [चैनल पेयरिंग](/hi/channels/pairing) देखें।
- **Node क्षमता अनुमोदन** (`node.pair.*`) यह नियंत्रित करता है कि कनेक्ट किया गया Node किन घोषित
  क्षमताओं/कमांड को उपलब्ध करा सकता है। Gateway सत्य का
  स्रोत है; UI (macOS ऐप, Control UI) ऐसे फ़्रंटएंड हैं जो लंबित अनुरोधों को अनुमोदित या
  अस्वीकार करते हैं।

पहले का स्वतंत्र Node पेयरिंग स्टोर (`nodes/paired.json`, जिसमें प्रत्येक Node के लिए
टोकन था और जिसे जनवरी 2026 में कनेक्ट पथ से हटा दिया गया था) अब समाप्त हो चुका है: Gateway
स्टार्टअप पर एक बार शेष सभी पंक्तियों को डिवाइस रिकॉर्ड में समाहित करते हैं और
पुरानी फ़ाइलों को `.migrated` प्रत्यय के साथ संग्रहित करते हैं। पुराने TCP ब्रिज का समर्थन
हटा दिया गया है।

## क्षमता अनुमोदन कैसे काम करता है

1. एक Node Gateway WS से कनेक्ट होता है (डिवाइस पेयरिंग इस चरण को नियंत्रित करती है)।
2. Gateway घोषित क्षमता/कमांड सतह की तुलना
   अनुमोदित सतह से करता है; नई या विस्तृत सतहें डिवाइस रिकॉर्ड पर एक **लंबित अनुरोध** संग्रहीत करती हैं
   और `node.pair.requested` उत्सर्जित करती हैं।
3. आप अनुरोध को अनुमोदित या अस्वीकार करते हैं (CLI या UI)।
4. अनुमोदन होने तक Node कमांड फ़िल्टर किए जाते हैं; अनुमोदन सामान्य कमांड नीति के
   अधीन घोषित सतह को उपलब्ध कराता है।

लंबित अनुरोध Node के अंतिम
पुनः प्रयास के **5 मिनट बाद** अपने-आप समाप्त हो जाते हैं—सक्रिय रूप से पुनः कनेक्ट हो रहा Node
हर प्रयास पर नया अनुरोध (और अनुमोदन संकेत) बनाने के बजाय अपने एक लंबित अनुरोध को सक्रिय रखता है।

## CLI कार्यप्रवाह (हेडलेस-अनुकूल)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` पेयर किए गए/कनेक्टेड Node और उनकी क्षमताएँ दिखाता है।

## API सतह (Gateway प्रोटोकॉल)

इवेंट:

- `node.pair.requested` - नया लंबित अनुरोध बनाए जाने पर उत्सर्जित होता है।
- `node.pair.resolved` - अनुरोध अनुमोदित, अस्वीकृत या
  समाप्त होने पर उत्सर्जित होता है।

विधियाँ:

- `node.pair.list` - लंबित और पेयर किए गए Node सूचीबद्ध करता है (`operator.pairing`)।
- `node.pair.approve` - लंबित अनुरोध को अनुमोदित करता है।
- `node.pair.reject` - लंबित अनुरोध को अस्वीकार करता है।
- `node.pair.remove` - पेयर किए गए Node को हटाता है। यह पेयर-डिवाइस स्टोर में डिवाइस की `node`
  भूमिका को रद्द करता है, उसके साथ अनुमोदित Node सतह को हटाता है और
  उस डिवाइस के Node-भूमिका सत्रों को अमान्य/डिस्कनेक्ट करता है। एक **मिश्रित-भूमिका**
  वाला डिवाइस (उदाहरण के लिए, जिसके पास `operator` भी है) अपनी पंक्ति बनाए रखता है और केवल
  `node` भूमिका खोता है; केवल-Node डिवाइस की पंक्ति हटा दी जाती है। प्राधिकरण:
  `operator.pairing` गैर-ऑपरेटर Node पंक्तियाँ हटा सकता है; मिश्रित-भूमिका वाले डिवाइस पर
  अपनी **स्वयं की** Node भूमिका रद्द करने वाले डिवाइस-टोकन कॉलर को अतिरिक्त रूप से
  `operator.admin` की आवश्यकता होती है।
- `node.rename` - पेयर किए गए Node का ऑपरेटर को दिखाई देने वाला प्रदर्शन नाम बदलता है।

2026.7 में हटाए गए: `node.pair.request` और `node.pair.verify`। लंबित
अनुरोध Node कनेक्शन के दौरान स्वयं Gateway द्वारा बनाए जाते हैं, और
उनके लिए प्रयुक्त स्वतंत्र प्रति-Node टोकन अब मौजूद नहीं है; Node प्रमाणीकरण
डिवाइस पेयरिंग टोकन है।

टिप्पणियाँ:

- अपरिवर्तित सतह वाले पुनः कनेक्शन लंबित अनुरोध का पुनः उपयोग करते हैं; बार-बार किए गए
  अनुरोध ऑपरेटर की दृश्यता के लिए संग्रहीत Node मेटाडेटा और अनुमत-सूची में शामिल
  नवीनतम घोषित कमांड स्नैपशॉट को रीफ़्रेश करते हैं।
- ऑपरेटर स्कोप स्तरों और अनुमोदन-समय जाँचों का सारांश
  [ऑपरेटर स्कोप](/hi/gateway/operator-scopes) में दिया गया है।
- `node.pair.approve` अतिरिक्त अनुमोदन स्कोप लागू करने के लिए लंबित अनुरोध के घोषित कमांड का उपयोग करता है:
  - कमांड-रहित अनुरोध: `operator.pairing`
  - सामान्य कमांड अनुरोध: `operator.pairing` + `operator.write`
  - `system.run`, `system.run.prepare`,
    `system.which`, `browser.proxy`, `fs.listDir`, या
    `system.execApprovals.get/set` वाला एडमिन-संवेदनशील अनुरोध: `operator.pairing` + `operator.admin`

<Warning>
Node पेयरिंग अनुमोदन विश्वसनीय क्षमता सतह को रिकॉर्ड करता है। यह प्रति Node लाइव Node कमांड सतह को **पिन नहीं करता**।

- लाइव Node कमांड कनेक्शन के समय Node द्वारा घोषित कमांड से आते हैं और
  Gateway की वैश्विक Node कमांड नीति (`gateway.nodes.allowCommands` और
  `denyCommands`) के अनुसार फ़िल्टर किए जाते हैं।
- प्रति-Node `system.run` अनुमति और पूछताछ नीति पेयरिंग रिकॉर्ड में नहीं,
  बल्कि `exec.approvals.node.*` में Node पर रहती है।

</Warning>

## Node कमांड नियंत्रण (2026.3.31+)

<Warning>
**ब्रेकिंग परिवर्तन:** `2026.3.31` से शुरू होकर, Node पेयरिंग अनुमोदित होने तक Node कमांड अक्षम रहते हैं। घोषित Node कमांड उपलब्ध कराने के लिए अब केवल डिवाइस पेयरिंग पर्याप्त नहीं है।
</Warning>

जब कोई Node पहली बार कनेक्ट होता है, तो पेयरिंग का अनुरोध अपने-आप किया जाता है।
उस अनुरोध के अनुमोदित होने तक, उस Node के सभी लंबित Node कमांड
फ़िल्टर किए जाते हैं और निष्पादित नहीं होंगे। पेयरिंग अनुमोदित होने के बाद, Node के घोषित
कमांड सामान्य कमांड नीति के अधीन उपलब्ध हो जाते हैं।

इसका अर्थ है:

- जो Node पहले कमांड उपलब्ध कराने के लिए केवल डिवाइस पेयरिंग पर निर्भर थे, उन्हें
  अब Node पेयरिंग भी पूरी करनी होगी।
- पेयरिंग अनुमोदन से पहले कतारबद्ध कमांड स्थगित नहीं किए जाते, बल्कि हटा दिए जाते हैं।

## Node इवेंट विश्वास सीमाएँ (2026.3.31+)

<Warning>
**ब्रेकिंग परिवर्तन:** Node से आरंभ किए गए रन अब सीमित विश्वसनीय सतह पर रहते हैं।
</Warning>

Node से आरंभ किए गए सारांश और संबंधित सत्र इवेंट निर्धारित
विश्वसनीय सतह तक सीमित हैं। सूचना-संचालित या Node-ट्रिगर किए गए ऐसे प्रवाह,
जो पहले व्यापक होस्ट या सत्र टूल पहुँच पर निर्भर थे, उनमें समायोजन की आवश्यकता हो सकती है।
यह सुदृढ़ीकरण Node इवेंट को Node की विश्वास सीमा द्वारा अनुमत सीमा से परे
होस्ट-स्तरीय टूल पहुँच तक विशेषाधिकार बढ़ाने से रोकता है।

स्थायी Node उपस्थिति अपडेट भी इसी पहचान सीमा का पालन करते हैं:
`node.presence.alive` इवेंट केवल प्रमाणित Node डिवाइस
सत्रों से स्वीकार किया जाता है और पेयरिंग मेटाडेटा को तभी अपडेट करता है जब डिवाइस/Node पहचान
पहले से पेयर हो। स्वयं घोषित `client.id` मान अंतिम-देखी गई स्थिति लिखने के लिए
पर्याप्त नहीं है।

## SSH-सत्यापित डिवाइस स्वतः-अनुमोदन (डिफ़ॉल्ट)

निजी/CGNAT पते से पहली बार की `role: node` डिवाइस पेयरिंग
तब स्वतः अनुमोदित होती है जब Gateway **SSH के माध्यम से मशीन के स्वामित्व को प्रमाणित** कर सके: यह
पेयरिंग होस्ट (`BatchMode`, `StrictHostKeyChecking=yes`) से वापस कनेक्ट होता है,
वहाँ `openclaw node identity --json` चलाता है और केवल तभी अनुमोदित करता है जब दूरस्थ
डिवाइस आईडी और सार्वजनिक कुंजी लंबित अनुरोध से हूबहू मेल खाते हों। कुंजी का मिलान ही
इसे सुरक्षित बनाता है: केवल पहुँच-योग्यता कभी अनुमोदन नहीं करती, इसलिए NAT के सह-उपयोगकर्ता,
साझा होस्ट के अन्य उपयोगकर्ता और LAN स्पूफ़िंग—सभी सामान्य
संकेत प्रवाह पर वापस जाते हैं।

डिफ़ॉल्ट रूप से सक्षम। इसके सक्रिय होने की आवश्यकताएँ:

- Gateway प्रक्रिया उपयोगकर्ता (या `sshVerify.user`) Node होस्ट में
  गैर-इंटरैक्टिव रूप से SSH कर सकता है (कुंजियाँ/एजेंट; Tailscale SSH भी काम करता है), और होस्ट कुंजी
  पहले से विश्वसनीय है।
- `openclaw` गैर-इंटरैक्टिव `sh -lc` के लिए दूरस्थ `PATH` पर रिज़ॉल्व होता है।
- कनेक्ट हो रहा IP एक प्रत्यक्ष (गैर-प्रॉक्सी, गैर-लूपबैक) निजी, ULA,
  लिंक-लोकल या CGNAT पता है, या सेट होने पर `sshVerify.cidrs` से मेल खाता है।
- विश्वसनीय-CIDR अनुमोदन के समान पात्रता आधार: केवल नया, स्कोप-रहित Node
  पेयरिंग; अपग्रेड, ब्राउज़र, Control UI और WebChat हमेशा संकेत दिखाते हैं।

जाँच चलने के दौरान, Node क्लाइंट को मैन्युअल अनुमोदन के लिए रुकने के बजाय
पुनः प्रयास जारी रखने (`wait_then_retry`) को कहा जाता है; यदि जाँच
विफल होती है, तो अगला प्रयास सामान्य संकेत प्रवाह पर वापस चला जाता है। विफल लक्ष्यों
पर थोड़ी प्रतीक्षा अवधि लागू होती है (कुंजी असंगति के बाद 5 मिनट)।

अनुमोदित डिवाइस `approvedVia: "ssh-verified"` रिकॉर्ड करते हैं और उनकी पहली घोषित
क्षमता सतह उसी चरण में अनुमोदित हो जाती है—कुंजी का मिलान पहले ही यह प्रमाणित करता है कि
Node ऑपरेटर के खाते के अंतर्गत उनकी स्वामित्व वाली मशीन पर चल रहा है, जो वही
दावा है जिसे मैन्युअल क्षमता अनुमोदन सत्यापित करता है। बाद के सतह अपग्रेड फिर भी
संकेत दिखाते हैं।

सुदृढ़ करें या अक्षम करें:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        // पूरी तरह अक्षम करें:
        sshVerify: false,
        // ...या जाँच का दायरा/समायोजन निर्धारित करें:
        // sshVerify: { user: "me", identity: "~/.ssh/probe", timeoutMs: 7000, cidrs: ["10.0.0.0/8"] },
      },
    },
  },
}
```

## स्वतः-अनुमोदन (macOS ऐप)

macOS ऐप निम्न स्थितियों में Node क्षमता अनुरोधों का **मौन अनुमोदन** करने का प्रयास कर सकता है:

- अनुरोध को `silent` के रूप में चिह्नित किया गया हो (जब डिवाइस पेयरिंग
  गैर-इंटरैक्टिव रूप से अनुमोदित हुई थी, तब Gateway पहली क्षमता सतह को मौन चिह्नित करता है), और
- ऐप उसी उपयोगकर्ता का उपयोग करके Gateway होस्ट से SSH कनेक्शन सत्यापित कर सके।

यदि मौन अनुमोदन विफल होता है, तो यह सामान्य Approve/Reject संकेत पर वापस चला जाता है।

## विश्वसनीय-CIDR डिवाइस स्वतः-अनुमोदन

`role: node` के लिए WS डिवाइस पेयरिंग डिफ़ॉल्ट रूप से मैन्युअल रहती है। उन निजी Node
नेटवर्क के लिए, जहाँ Gateway पहले से नेटवर्क पथ पर विश्वास करता है, ऑपरेटर स्पष्ट CIDR या सटीक IP के साथ
इसे चुन सकते हैं:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

सुरक्षा सीमा:

- `gateway.nodes.pairing.autoApproveCidrs` सेट न होने पर अक्षम।
- व्यापक LAN या निजी-नेटवर्क स्वतः-अनुमोदन मोड मौजूद नहीं है; SSH-सत्यापित
  स्वतः-अनुमोदन (ऊपर) के लिए क्रिप्टोग्राफ़िक डिवाइस-कुंजी मिलान आवश्यक है, केवल
  नेटवर्क निकटता कभी पर्याप्त नहीं होती।
- केवल बिना किसी अनुरोधित स्कोप वाला नया `role: node` डिवाइस पेयरिंग अनुरोध
  पात्र है।
- ऑपरेटर, ब्राउज़र, Control UI और WebChat क्लाइंट मैन्युअल रहते हैं।
- भूमिका, स्कोप, मेटाडेटा और सार्वजनिक-कुंजी अपग्रेड मैन्युअल रहते हैं।
- एक ही होस्ट के लूपबैक विश्वसनीय-प्रॉक्सी हेडर पथ पात्र नहीं हैं, क्योंकि स्थानीय
  कॉलर उस पथ की स्पूफ़िंग कर सकते हैं।

## मौन पेयरिंग प्रतिस्थापन की सफ़ाई

गैर-इंटरैक्टिव अनुमोदन पेयर-डिवाइस पंक्ति पर अपना उद्गम रिकॉर्ड करते हैं:
एक ही होस्ट की स्थानीय नीति के अनुमोदन `silent` के रूप में, विश्वसनीय-CIDR Node अनुमोदन
`trusted-cidr` के रूप में और SSH-सत्यापित Node अनुमोदन `ssh-verified` के रूप में। जिन क्लाइंट की स्टेट डायरेक्टरी अस्थायी होती है (अस्थायी होम,
कंटेनर, प्रति-रन सैंडबॉक्स), वे प्रत्येक रन के लिए नई डिवाइस कुंजी-जोड़ी बनाते हैं और हर
रन चुपचाप एक बिल्कुल नए डिवाइस के रूप में फिर से पेयर होता है—सफ़ाई के बिना पेयर की गई सूची में
प्रति रन एक निष्क्रिय पंक्ति बढ़ती जाती है।

जब Gateway किसी **स्थानीय** डिवाइस पेयरिंग को मौन रूप से अनुमोदित करता है, तो वह
उसी क्लाइंट क्लस्टर से संबंधित पुराने `silent`-अनुमोदित रिकॉर्ड को सेवानिवृत्त करता है
(जहाँ `clientId`, `clientMode` और प्रदर्शन नाम मेल खाते हों) और जो वर्तमान में
कनेक्ट न हों। स्थानीय क्लाइंट स्वयं Gateway होस्ट पर चलते हैं, इसलिए क्लस्टर कुंजी
किसी दूसरी मशीन से मेल नहीं खा सकती। सेवानिवृत्त पंक्तियाँ तुरंत अपने टोकन खो देती हैं;
मेल खाती कोई भी पुरानी Node पेयरिंग प्रविष्टि साफ़ कर दी जाती है और एक `node.pair.resolved`
हटाने का इवेंट प्रसारित किया जाता है।

सीमाएँ:

- केवल वे रिकॉर्ड जिनकी नवीनतम स्वीकृति उसी होस्ट पर स्थानीय (`silent`) थी,
  ट्रिगर और लक्ष्य के रूप में पात्र हैं। विश्वसनीय-CIDR और SSH-सत्यापित पेयरिंग
  ऐसे होस्ट के बीच होती हैं जहाँ डिस्प्ले मेटाडेटा मशीन की पहचान नहीं होता, इसलिए उन्हें
  कभी भी स्वचालित रूप से नहीं हटाया जाता — उनके लिए Control UI क्लीनअप या
  `openclaw nodes remove` का उपयोग करें।
- स्वामी द्वारा स्वीकृत और QR/सेटअप-कोड (बूटस्ट्रैप) पेयरिंग कभी भी
  स्वचालित रूप से नहीं हटाई जातीं। उद्गम जानकारी उपलब्ध होने से पहले स्वीकृत रिकॉर्ड सुरक्षित रहते हैं,
  उसी डिवाइस आईडी की बाद में मौन पुनः-स्वीकृति होने के बाद भी।
- वर्तमान में कनेक्टेड डिवाइस छोड़ दिए जाते हैं, ताकि अलग-अलग स्थिति डायरेक्टरी वाले
  समवर्ती स्थानीय सत्र सक्रिय रहने के दौरान अपने टोकन बनाए रखें। पिछले एक मिनट के भीतर स्वीकृत रिकॉर्ड
  भी छोड़ दिए जाते हैं, ताकि एक साथ होने वाले पेयरिंग हैंडशेक अपने कनेक्शन पंजीकृत होने से पहले
  एक-दूसरे को निष्क्रिय न कर सकें।
- प्रभावित क्लाइंट संरचना के अनुसार स्थानीय होते हैं, इसलिए वे अपने
  अगले कनेक्शन पर मौन रूप से फिर से पेयर हो जाते हैं।

## मेटाडेटा-अपग्रेड की स्वचालित स्वीकृति

जब पहले से पेयर किया गया डिवाइस केवल गैर-संवेदनशील मेटाडेटा
परिवर्तनों (उदाहरण के लिए डिस्प्ले नाम या क्लाइंट प्लेटफ़ॉर्म संकेत) के साथ दोबारा कनेक्ट होता है, तो OpenClaw
इसे `metadata-upgrade` मानता है। मौन स्वचालित स्वीकृति का दायरा सीमित है: यह केवल
विश्वसनीय गैर-ब्राउज़र स्थानीय पुनः-कनेक्शन पर लागू होती है, जो पहले ही
स्थानीय या साझा क्रेडेंशियल के अधिकार को प्रमाणित कर चुके हैं, जिसमें
OS संस्करण मेटाडेटा बदलने के बाद उसी होस्ट पर नेटिव ऐप का पुनः-कनेक्शन शामिल है।
ब्राउज़र/Control UI क्लाइंट और दूरस्थ क्लाइंट अब भी स्पष्ट पुनः-स्वीकृति प्रवाह
का उपयोग करते हैं। दायरा अपग्रेड (रीड से राइट/एडमिन) और सार्वजनिक कुंजी परिवर्तन
मेटाडेटा-अपग्रेड की स्वचालित स्वीकृति के लिए **पात्र नहीं** हैं; वे स्पष्ट
पुनः-स्वीकृति अनुरोध बने रहते हैं।

## QR पेयरिंग सहायक

`/pair qr` पेयरिंग पेलोड को संरचित मीडिया के रूप में रेंडर करता है, ताकि मोबाइल और
ब्राउज़र क्लाइंट इसे सीधे स्कैन कर सकें।

किसी डिवाइस को हटाने पर उस डिवाइस आईडी के सभी पुराने लंबित पेयरिंग अनुरोध भी
हटा दिए जाते हैं, ताकि निरस्त करने के बाद `nodes pending` अनाथ पंक्तियाँ न दिखाए।

## स्थानीयता और फ़ॉरवर्ड किए गए हेडर

Gateway पेयरिंग किसी कनेक्शन को केवल तभी लूपबैक मानती है, जब रॉ सॉकेट
और किसी भी अपस्ट्रीम प्रॉक्सी के प्रमाण दोनों सहमत हों। यदि कोई अनुरोध लूपबैक पर आता है, लेकिन
उसमें `Forwarded`, कोई `X-Forwarded-*`, या `X-Real-IP` हेडर प्रमाण मौजूद है, तो वह
फ़ॉरवर्डेड-हेडर प्रमाण लूपबैक स्थानीयता के दावे को अमान्य कर देता है, और
पेयरिंग पथ अनुरोध को उसी होस्ट का कनेक्शन मानकर मौन रूप से स्वीकार करने के बजाय स्पष्ट स्वीकृति
की माँग करता है। ऑपरेटर प्रमाणीकरण पर समकक्ष नियम के लिए
[विश्वसनीय प्रॉक्सी प्रमाणीकरण](/hi/gateway/trusted-proxy-auth) देखें।

## संग्रहण (स्थानीय, निजी)

पेयरिंग स्थिति, Gateway स्थिति डायरेक्टरी के अंतर्गत साझा SQLite स्थिति
डेटाबेस में पेयर किए गए डिवाइस रिकॉर्ड पर रहती है (डिफ़ॉल्ट `~/.openclaw`):

- `~/.openclaw/state/openclaw.sqlite` (डिवाइस प्रमाणीकरण वाले पेयर किए गए डिवाइस,
  स्वीकृत Node सतहें, लंबित सतह अनुरोध, लंबित डिवाइस पेयरिंग
  अनुरोध और बूटस्ट्रैप टोकन)

यदि आप `OPENCLAW_STATE_DIR` को ओवरराइड करते हैं, तो डेटाबेस भी उसके साथ स्थानांतरित हो जाता है। JSON स्टोर वाले
रिलीज़ से अपग्रेड किए गए Gateway उन्हें स्टार्टअप पर आयात करते हैं और
`devices/*.json.migrated` तथा `nodes/*.json.migrated` संग्रह पीछे छोड़ देते हैं।

सुरक्षा संबंधी टिप्पणियाँ:

- डिवाइस टोकन सीक्रेट होते हैं; स्थिति डेटाबेस को संवेदनशील मानें।
- डिवाइस टोकन को रोटेट करने के लिए `openclaw devices rotate` /
  `device.token.rotate` का उपयोग किया जाता है।

## ट्रांसपोर्ट व्यवहार

- ट्रांसपोर्ट **स्टेटलेस** है; यह सदस्यता संग्रहीत नहीं करता।
- यदि Gateway ऑफ़लाइन है या पेयरिंग अक्षम है, तो Node पेयर नहीं कर सकते।
- दूरस्थ मोड में, पेयरिंग दूरस्थ Gateway के स्टोर के विरुद्ध होती है।
