OpenClaw में दो स्वतंत्र स्ट्रीमिंग परतें हैं, और आज चैनल संदेशों के लिए **कोई वास्तविक
टोकन-डेल्टा स्ट्रीमिंग नहीं है**:

- **ब्लॉक स्ट्रीमिंग (चैनल):** सहायक के लिखते समय पूर्ण हुए **ब्लॉक** भेजें।
  ये सामान्य चैनल संदेश हैं, टोकन डेल्टा नहीं।
- **पूर्वावलोकन स्ट्रीमिंग (Telegram/Discord/Slack/Matrix/Mattermost/MS Teams):**
  जनरेट करते समय एक अस्थायी **पूर्वावलोकन संदेश** अपडेट करें (भेजना + संपादन/जोड़ना)।

## ब्लॉक स्ट्रीमिंग (चैनल संदेश)

ब्लॉक स्ट्रीमिंग उपलब्ध होते ही सहायक का आउटपुट मोटे खंडों में भेजती है।

```text
मॉडल आउटपुट
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ बफ़र बढ़ने पर चंकर ब्लॉक भेजता है
       └─ (blockStreamingBreak=message_end)
            └─ चंकर message_end पर फ़्लश करता है
                   └─ चैनल पर भेजना (ब्लॉक उत्तर)
```

- `text_delta/events`: मॉडल स्ट्रीम इवेंट (गैर-स्ट्रीमिंग मॉडल के लिए विरल हो सकते हैं)।
- `chunker`: न्यूनतम/अधिकतम सीमाएँ + ब्रेक प्राथमिकता लागू करने वाला `EmbeddedBlockChunker`।
- `channel send`: वास्तविक आउटबाउंड संदेश (ब्लॉक उत्तर)।

**नियंत्रण** (जब तक उल्लेख न हो, सभी `agents.defaults` के अंतर्गत):

| कुंजी                                                          | मान / स्वरूप                                                          | डिफ़ॉल्ट    |
| ------------------------------------------------------------ | ----------------------------------------------------------------------- | ---------- |
| `blockStreamingDefault`                                      | `"on"` / `"off"`                                                        | `"off"`    |
| `blockStreamingBreak`                                        | `"text_end"` / `"message_end"`                                          | -          |
| `blockStreamingChunk`                                        | `{ minChars, maxChars, breakPreference? }`                              | -          |
| `blockStreamingCoalesce`                                     | `{ minChars?, maxChars?, idleMs? }` (भेजने से पहले स्ट्रीम किए गए ब्लॉक मर्ज करें) | -          |
| `*.streaming.block.enabled` (चैनल ओवरराइड)               | `true` / `false`, प्रत्येक चैनल (और प्रत्येक खाते) के लिए ब्लॉक स्ट्रीमिंग बाध्य करता है  | -          |
| `*.textChunkLimit` (उदा. `channels.whatsapp.textChunkLimit`) | संख्या, कठोर सीमा                                                        | 4000       |
| `*.streaming.chunkMode`                                      | `"length"` / `"newline"`                                                | `"length"` |
| `channels.discord.maxLinesPerMessage`                        | संख्या, UI क्लिपिंग से बचने के लिए लंबे उत्तरों को विभाजित करने वाली लचीली पंक्ति सीमा     | 17         |

`streaming.chunkMode: "newline"` प्रत्येक नई पंक्ति पर नहीं, बल्कि रिक्त पंक्तियों (अनुच्छेद सीमाओं) पर
विभाजित करता है; पाठ के सीमा पार करने के बाद ही लंबाई-आधारित खंडीकरण का
सहारा लेता है।

बंडल किए गए चैनल इन ओवरराइड को
`channels.<id>.streaming.{chunkMode,block.enabled,block.coalesce}` के रूप में लिखते हैं। समतल
`*.chunkMode` / `*.blockStreaming` / `*.blockStreamingCoalesce` वर्तनी
प्रत्येक बंडल किए गए चैनल पर लेगेसी हैं: `openclaw doctor --fix` उन्हें नेस्टेड
स्वरूप में माइग्रेट करता है और चैनल स्कीमा उन्हें अस्वीकार करते हैं। अब भी समतल
वर्तनी का उपयोग करने वाले बाहरी SDK Plugin कॉन्फ़िगरेशन अगले रिलीज़ चक्र तक
बहिष्कृत फ़ॉलबैक (रनटाइम चेतावनी सहित) के माध्यम से काम करते रहेंगे।

`blockStreamingBreak` के लिए **सीमा अर्थविज्ञान**:

- `text_end`: चंकर के भेजते ही ब्लॉक स्ट्रीम करें; प्रत्येक `text_end` पर फ़्लश करें।
- `message_end`: सहायक संदेश पूरा होने तक प्रतीक्षा करें, फिर बफ़र किया हुआ
  आउटपुट फ़्लश करें। यदि बफ़र किया हुआ पाठ `maxChars` से अधिक हो, तब भी चंकर का
  उपयोग होता है, इसलिए यह अंत में कई खंड भेज सकता है।

### ब्लॉक स्ट्रीमिंग के साथ मीडिया डिलीवरी

स्ट्रीमिंग मीडिया को `mediaUrl` या
`mediaUrls` जैसे संरचित पेलोड फ़ील्ड का उपयोग करना चाहिए; स्ट्रीम किए गए पाठ को अटैचमेंट कमांड के रूप में पार्स नहीं किया जाता। जब ब्लॉक
स्ट्रीमिंग मीडिया को जल्दी भेजती है, OpenClaw उस टर्न के लिए उस डिलीवरी को याद रखता है। यदि
अंतिम सहायक पेलोड उसी मीडिया URL को दोहराता है, तो अंतिम डिलीवरी अटैचमेंट को फिर से
भेजने के बजाय डुप्लिकेट मीडिया को हटा देती है।

पूरी तरह समान अंतिम पेलोड दबा दिए जाते हैं। यदि अंतिम पेलोड पहले से स्ट्रीम किए गए
मीडिया के आस-पास अलग पाठ जोड़ता है, तो OpenClaw मीडिया को केवल एक बार भेजते हुए भी
नया पाठ भेजता है। इससे Telegram जैसे चैनलों पर वॉइस
नोट या फ़ाइलें डुप्लिकेट होने से बचती हैं।

## खंडीकरण एल्गोरिदम (निम्न/उच्च सीमाएँ)

ब्लॉक खंडीकरण `EmbeddedBlockChunker` द्वारा कार्यान्वित होता है:

- **निम्न सीमा:** बफ़र >= `minChars` होने तक न भेजें (बाध्य किए जाने को छोड़कर)।
- **उच्च सीमा:** `maxChars` से पहले विभाजन को प्राथमिकता दें; बाध्य होने पर `maxChars` पर विभाजित करें।
- **ब्रेक प्राथमिकता शृंखला:** `paragraph` -> `newline` -> `sentence` ->
  रिक्त स्थान -> कठोर ब्रेक।
- **कोड फ़ेंस:** फ़ेंस के अंदर कभी विभाजित न करें; `maxChars` पर बाध्य होने पर
  Markdown को मान्य रखने के लिए फ़ेंस बंद करके फिर से खोलें।

`maxChars` को चैनल के `textChunkLimit` तक सीमित किया जाता है, इसलिए आप
प्रति-चैनल सीमाओं से अधिक नहीं जा सकते।

## सहयोजन (स्ट्रीम किए गए ब्लॉक मर्ज करना)

ब्लॉक स्ट्रीमिंग सक्षम होने पर, OpenClaw लगातार आने वाले **ब्लॉक
खंडों को मर्ज** करके भेज सकता है, जिससे प्रगतिशील आउटपुट देते हुए भी
एकल-पंक्ति स्पैम कम होता है।

- सहयोजन फ़्लश करने से पहले **निष्क्रिय अंतराल** (`idleMs`) की प्रतीक्षा करता है।
- बफ़र `maxChars` द्वारा सीमित होते हैं और इसे पार करने पर फ़्लश हो जाते हैं।
- `minChars` पर्याप्त पाठ जमा होने तक छोटे खंडों को भेजे जाने से रोकता है
  (अंतिम फ़्लश हमेशा शेष पाठ भेजता है)।
- जोड़क `blockStreamingChunk.breakPreference` से व्युत्पन्न होता है: `paragraph` ->
  `\n\n`, `newline` -> `\n`, `sentence` -> स्पेस।
- चैनल ओवरराइड `*.streaming.block.coalesce` के माध्यम से उपलब्ध हैं (प्रति-खाता
  कॉन्फ़िगरेशन सहित)।
- ओवरराइड न किए जाने पर Discord, Signal और Slack का डिफ़ॉल्ट सहयोजन `{ minChars: 1500, idleMs: 1000 }`
  होता है।

## ब्लॉक के बीच मानव-जैसी गति

ब्लॉक स्ट्रीमिंग सक्षम होने पर, पहले ब्लॉक के बाद ब्लॉक
उत्तरों के बीच एक **यादृच्छिक विराम** जोड़ें, ताकि कई बबल वाले उत्तर अधिक स्वाभाविक लगें।

| `agents.defaults.humanDelay.mode` | व्यवहार                |
| --------------------------------- | ----------------------- |
| `off` (डिफ़ॉल्ट)                   | कोई विराम नहीं                |
| `natural`                         | 800-2500ms का यादृच्छिक विराम |
| `custom`                          | `minMs`/`maxMs`         |

प्रत्येक एजेंट के लिए `agents.list[].humanDelay` के माध्यम से ओवरराइड करें। यह केवल **ब्लॉक
उत्तरों** पर लागू होता है, अंतिम उत्तरों या टूल सारांशों पर नहीं।

## "खंड स्ट्रीम करें या सब कुछ"

- **खंड स्ट्रीम करें:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"`
  (बनते ही भेजें)। गैर-Telegram चैनलों को
  `*.streaming.block.enabled: true` भी चाहिए।
- **अंत में सब कुछ स्ट्रीम करें:** `blockStreamingBreak: "message_end"` (एक बार फ़्लश करें,
  बहुत लंबा होने पर संभवतः कई खंडों में)।
- **कोई ब्लॉक स्ट्रीमिंग नहीं:** `blockStreamingDefault: "off"` (केवल अंतिम उत्तर)।

ब्लॉक स्ट्रीमिंग तब तक **बंद रहती है जब तक** `*.streaming.block.enabled` को स्पष्ट रूप से
`true` पर सेट न किया जाए (अपवाद: QQ Bot में कोई `streaming.block` कुंजी नहीं है और
जब तक `channels.qqbot.streaming.mode`, `"off"` न हो, यह ब्लॉक उत्तर स्ट्रीम करता है)। चैनल ब्लॉक
उत्तरों के बिना लाइव पूर्वावलोकन (`channels.<channel>.streaming.mode`) स्ट्रीम कर सकते हैं।
`blockStreaming*` डिफ़ॉल्ट कॉन्फ़िगरेशन रूट में नहीं, बल्कि `agents.defaults` के अंतर्गत होते हैं।

## पूर्वावलोकन स्ट्रीमिंग मोड

कैनोनिकल कुंजी: `channels.<channel>.streaming` (नेस्टेड `{ mode, ... }`; लेगेसी
शीर्ष-स्तरीय बूलियन/स्ट्रिंग वर्तनी को `openclaw doctor --fix` द्वारा फिर से लिखा जाता है)।

| मोड       | व्यवहार                                                              |
| ---------- | --------------------------------------------------------------------- |
| `off`      | पूर्वावलोकन स्ट्रीमिंग अक्षम करें                                             |
| `partial`  | एकल पूर्वावलोकन को नवीनतम पाठ से बदलें                              |
| `block`    | खंडित/जोड़े गए चरणों में पूर्वावलोकन अपडेट करें                             |
| `progress` | जनरेशन के दौरान प्रगति/स्थिति पूर्वावलोकन, पूर्ण होने पर अंतिम उत्तर |

`streaming.mode: "block"`, Discord और Telegram जैसे संपादन-सक्षम
चैनलों के लिए एक पूर्वावलोकन-स्ट्रीमिंग मोड है; यह वहाँ अपने आप चैनल
ब्लॉक डिलीवरी सक्षम नहीं करता। सामान्य ब्लॉक उत्तरों के लिए `streaming.block.enabled` का उपयोग करें।
Microsoft Teams इसका
अपवाद है: इसमें कोई ड्राफ़्ट-पूर्वावलोकन ब्लॉक ट्रांसपोर्ट नहीं है, इसलिए `streaming.mode:
"block"` नेटिव स्ट्रीमिंग को पूरी तरह अक्षम कर देता है और उत्तर नेटिव आंशिक/प्रगति स्ट्रीमिंग के बजाय नियमित
ब्लॉक डिलीवरी के रूप में पहुँचता है। Mattermost भी
अलग है: `block` मोड में यह पूर्वावलोकन को पूर्ण पाठ और
टूल-गतिविधि ब्लॉक के बीच बदलता है, इसलिए पहले के ब्लॉक एक संपादन-योग्य ड्राफ़्ट में
ओवरराइट होने के बजाय अलग पोस्ट के रूप में दिखाई देते रहते हैं।

### चैनल मैपिंग

| चैनल    | `off` | `partial` | `block` | `progress`              |
| ---------- | ----- | --------- | ------- | ----------------------- |
| Telegram   | हाँ   | हाँ       | हाँ     | संपादन-योग्य प्रगति ड्राफ़्ट |
| Discord    | हाँ   | हाँ       | हाँ     | संपादन-योग्य प्रगति ड्राफ़्ट |
| Slack      | हाँ   | हाँ       | हाँ     | हाँ                     |
| Mattermost | हाँ   | हाँ       | हाँ     | हाँ                     |
| MS Teams   | हाँ   | हाँ       | हाँ     | नेटिव प्रगति स्ट्रीम  |

पूर्वावलोकन खंड कॉन्फ़िगरेशन (`streaming.preview.chunk.*`, उदाहरण के लिए
`channels.discord.streaming` या `channels.telegram.streaming` के अंतर्गत) का डिफ़ॉल्ट
`minChars: 200`, `maxChars: 800` (चैनल के `textChunkLimit` तक सीमित), और
`breakPreference: "paragraph"` होता है।

केवल Slack:

- `channels.slack.streaming.nativeTransport`, `channels.slack.streaming.mode="partial"` होने पर
  Slack नेटिव स्ट्रीमिंग API कॉल
  (`chat.startStream`/`chat.appendStream`/`chat.stopStream`) को टॉगल करता है
  (डिफ़ॉल्ट: `true`)।
- Slack नेटिव स्ट्रीमिंग और Slack सहायक थ्रेड स्थिति के लिए उत्तर
  थ्रेड लक्ष्य आवश्यक है। शीर्ष-स्तरीय DM वह थ्रेड-शैली पूर्वावलोकन नहीं दिखाते, लेकिन
  फिर भी Slack ड्राफ़्ट पूर्वावलोकन पोस्ट और संपादन का उपयोग कर सकते हैं।

## रनटाइम व्यवहार

### Telegram

- DM और
  समूह/विषयों में पूर्वावलोकन अपडेट के लिए `sendMessage` + `editMessageText` का उपयोग करता है; अंतिम टेक्स्ट सक्रिय पूर्वावलोकन को उसी स्थान पर संपादित करता है। Telegram
  के अस्थायी 30-सेकंड वाले "टाइपिंग" ड्राफ़्ट (`sendMessageDraft`) का उपयोग
  उत्तर स्ट्रीमिंग के लिए नहीं किया जाता।
- छोटे आरंभिक पूर्वावलोकनों को अब भी पुश-नोटिफ़िकेशन UX के लिए डिबाउंस किया जाता है, लेकिन
  वे एक सीमित विलंब के बाद दिखाई देते हैं, ताकि सक्रिय रन दृश्य रूप से मौन न रहें।
- लंबे अंतिम उत्तर पहले खंड के लिए पूर्वावलोकन संदेश का पुनः उपयोग करते हैं और केवल
  शेष खंड भेजते हैं।
- `block` मोड पूर्वावलोकन को
  `streaming.preview.chunk.maxChars` पर एक नए संदेश में बदलता है (डिफ़ॉल्ट 800, Telegram की 4096
  संपादन सीमा तक सीमित); अन्य मोड एक पूर्वावलोकन को 4096 वर्णों तक बढ़ाते हैं।
- `progress` मोड टूल की प्रगति को संपादन योग्य स्थिति ड्राफ़्ट में रखता है, उत्तर स्ट्रीमिंग सक्रिय होने पर लेकिन अभी कोई टूल पंक्ति
  उपलब्ध न होने पर स्थिति लेबल दिखाता है, पूर्ण होने पर ड्राफ़्ट साफ़ करता है और अंतिम उत्तर
  सामान्य डिलीवरी के माध्यम से भेजता है।
- यदि पूर्ण टेक्स्ट की पुष्टि होने से पहले अंतिम संपादन विफल हो जाता है, तो OpenClaw
  सामान्य अंतिम डिलीवरी का उपयोग करता है और पुराने पूर्वावलोकन को साफ़ कर देता है।
- दोहरी स्ट्रीमिंग से बचने के लिए, Telegram ब्लॉक स्ट्रीमिंग स्पष्ट रूप से
  सक्षम होने पर पूर्वावलोकन स्ट्रीमिंग छोड़ दी जाती है।
- `/reasoning stream` तर्क-विचार को एक अस्थायी पूर्वावलोकन में लिख सकता है, जिसे
  अंतिम डिलीवरी के बाद हटा दिया जाता है।
- Telegram के चयनित उद्धरण वाले उत्तर अपवाद हैं: जब `replyToMode`
  `"off"` नहीं है और चयनित उद्धरण टेक्स्ट मौजूद है, तब OpenClaw उस टर्न के लिए उत्तर की पूर्वावलोकन
  स्ट्रीम छोड़ देता है (अंतिम उत्तर को मूल उद्धरण-उत्तर
  पथ से जाना आवश्यक है), इसलिए टूल-प्रगति की पूर्वावलोकन पंक्तियाँ रेंडर नहीं हो सकतीं। चयनित उद्धरण टेक्स्ट के बिना
  वर्तमान-संदेश के उत्तरों में पूर्वावलोकन स्ट्रीमिंग जारी रहती है। विवरण के लिए
  [Telegram चैनल दस्तावेज़](/hi/channels/telegram) देखें।

### Discord

- भेजें + संपादित करें वाले पूर्वावलोकन संदेशों का उपयोग करता है।
- `block` मोड ड्राफ़्ट खंडीकरण (`draftChunk`) का उपयोग करता है।
- Discord ब्लॉक स्ट्रीमिंग स्पष्ट रूप से
  सक्षम होने पर पूर्वावलोकन स्ट्रीमिंग छोड़ दी जाती है।
- `progress` मोड अंतिम उत्तर में एक छोटी `-#` गतिविधि रसीद (विचार/टूल-कॉल
  की संख्या और बीता समय) जोड़ता है और उत्तर पहुँचने के बाद स्थिति ड्राफ़्ट
  हटा देता है, ताकि व्यस्त चैनलों में उत्तर के ऊपर कोई अनाथ टूल लॉग
  न रहे। त्रुटि वाले अंतिम उत्तर विफल टर्न के रिकॉर्ड के रूप में ड्राफ़्ट बनाए रखते हैं।
- अंतिम मीडिया, त्रुटि और स्पष्ट-उत्तर पेलोड नया ड्राफ़्ट फ़्लश किए बिना
  लंबित पूर्वावलोकन रद्द करते हैं, फिर सामान्य डिलीवरी का उपयोग करते हैं।

### Slack

- `partial` उपलब्ध होने पर Slack की मूल स्ट्रीमिंग (`chat.startStream`/`append`/`stop`)
  का उपयोग कर सकता है।
- `block` जोड़ने-शैली वाले ड्राफ़्ट पूर्वावलोकन का उपयोग करता है।
- `progress` स्थिति पूर्वावलोकन टेक्स्ट और फिर अंतिम उत्तर का उपयोग करता है।
- उत्तर थ्रेड के बिना शीर्ष-स्तरीय DM, Slack की मूल स्ट्रीमिंग के बजाय
  ड्राफ़्ट पूर्वावलोकन पोस्ट और संपादन का उपयोग करते हैं।
- मूल और ड्राफ़्ट पूर्वावलोकन स्ट्रीमिंग उस टर्न के लिए ब्लॉक उत्तरों को दबाती है, ताकि
  Slack उत्तर केवल एक डिलीवरी पथ से स्ट्रीम हो।
- अंतिम मीडिया/त्रुटि पेलोड और प्रगति वाले अंतिम संदेश अस्थायी ड्राफ़्ट
  संदेश नहीं बनाते; केवल वे टेक्स्ट/ब्लॉक अंतिम संदेश, जो पूर्वावलोकन संपादित कर सकते हैं, लंबित
  ड्राफ़्ट टेक्स्ट को फ़्लश करते हैं।

### Mattermost

- `partial` मोड में, सोच-विचार और आंशिक उत्तर टेक्स्ट को एकल ड्राफ़्ट
  पूर्वावलोकन पोस्ट में स्ट्रीम करता है, जो अंतिम उत्तर सुरक्षित रूप से भेजे जा सकने पर उसी स्थान पर अंतिम रूप लेता है।
- `progress` मोड में, सोच-विचार और टूल गतिविधि को एकल स्थिति
  पूर्वावलोकन में स्ट्रीम करता है, जो अंतिम उत्तर सुरक्षित रूप से भेजे जा सकने पर उसी स्थान पर अंतिम रूप लेता है।
- `block` मोड में, पूर्ण टेक्स्ट और टूल-गतिविधि पोस्ट के बीच अदला-बदली करता है;
  समानांतर और लगातार टूल अपडेट वर्तमान टूल-गतिविधि पोस्ट साझा करते हैं।
- यदि अंतिम रूप देते समय पूर्वावलोकन पोस्ट हटा दी गई हो या
  अन्यथा अनुपलब्ध हो, तो नया अंतिम पोस्ट भेजता है।
- अंतिम मीडिया/त्रुटि पेलोड अस्थायी पूर्वावलोकन पोस्ट फ़्लश करने के बजाय सामान्य
  डिलीवरी से पहले लंबित पूर्वावलोकन अपडेट रद्द करते हैं।

### Matrix

- जब अंतिम टेक्स्ट पूर्वावलोकन इवेंट का पुनः उपयोग कर सकता है, तब ड्राफ़्ट पूर्वावलोकन
  उसी स्थान पर अंतिम रूप लेते हैं।
- केवल-मीडिया, त्रुटि और उत्तर-लक्ष्य-बेमेल वाले अंतिम संदेश सामान्य डिलीवरी से पहले लंबित पूर्वावलोकन
  अपडेट रद्द करते हैं; पहले से दिखाई देने वाला पुराना पूर्वावलोकन संशोधित करके हटा दिया जाता है।

## टूल-प्रगति पूर्वावलोकन अपडेट

पूर्वावलोकन स्ट्रीमिंग में **टूल-प्रगति** अपडेट भी शामिल हो सकते हैं: "वेब पर खोज हो रही है", "फ़ाइल पढ़ी जा रही है" या "टूल कॉल किया जा रहा है" जैसी छोटी स्थिति
पंक्तियाँ, जो टूल चलने के दौरान अंतिम उत्तर से पहले
उसी पूर्वावलोकन संदेश में दिखाई देती हैं।
Codex ऐप-सर्वर मोड में, Codex की प्रस्तावना/टिप्पणी संदेश इसी
पूर्वावलोकन पथ का उपयोग करते हैं, इसलिए "मैं जाँच कर रहा हूँ..." जैसी छोटी प्रगति सूचनाएँ
अंतिम उत्तर का हिस्सा बने बिना संपादन योग्य ड्राफ़्ट में स्ट्रीम हो सकती हैं। इससे
बहु-चरणीय टूल टर्न पहले सोच-विचार पूर्वावलोकन और अंतिम उत्तर के बीच मौन रहने के बजाय
दृश्य रूप से सक्रिय रहते हैं।

लंबे समय तक चलने वाले टूल लौटने से पहले टाइप की गई प्रगति उत्सर्जित कर सकते हैं। उदाहरण के लिए,
`web_fetch` शुरू होने पर पाँच-सेकंड का टाइमर सक्रिय करता है: यदि फ़ेच अब भी
लंबित है, तो पूर्वावलोकन `Fetching page content...` दिखाता है; यदि फ़ेच इससे पहले पूर्ण हो जाता है या
रद्द हो जाता है, तो कोई प्रगति पंक्ति उत्सर्जित नहीं होती। बाद का अंतिम टूल
परिणाम फिर भी मॉडल को सामान्य रूप से दिया जाता है।

समर्थित सतहें:

- पूर्वावलोकन स्ट्रीमिंग सक्रिय होने पर **Discord**, **Slack**, **Telegram** और **Matrix** डिफ़ॉल्ट रूप से टूल-प्रगति और
  Codex प्रस्तावना अपडेट को लाइव पूर्वावलोकन संपादन में स्ट्रीम करते हैं। Microsoft Teams व्यक्तिगत चैट में
  अपनी मूल प्रगति स्ट्रीम का उपयोग करता है।
- Telegram में टूल-प्रगति पूर्वावलोकन अपडेट
  `v2026.4.22` से सक्षम रूप में जारी किए गए हैं; उन्हें सक्षम रखने से जारी किया गया वह व्यवहार संरक्षित रहता है।
- **Mattermost**, `partial` और
  `progress` मोड में टूल गतिविधि को एक पूर्वावलोकन पोस्ट में, या `block`
  मोड में टेक्स्ट ब्लॉकों के बीच एक टूल-गतिविधि पोस्ट में समाहित करता है (ऊपर देखें)।
- टूल-प्रगति संपादन सक्रिय पूर्वावलोकन स्ट्रीमिंग मोड का अनुसरण करते हैं; पूर्वावलोकन स्ट्रीमिंग
  `off` होने पर या ब्लॉक स्ट्रीमिंग द्वारा संदेश का नियंत्रण ले लेने पर इन्हें
  छोड़ दिया जाता है। Telegram पर, `streaming.mode: "off"` केवल-अंतिम है: सामान्य
  प्रगति चर्चा भी अलग स्थिति संदेशों के रूप में दिए जाने के बजाय दबा दी जाती है, जबकि स्वीकृति संकेत, मीडिया पेलोड और त्रुटियाँ फिर भी
  सामान्य रूप से रूट होती हैं।
- पूर्वावलोकन स्ट्रीमिंग बनाए रखते हुए टूल-प्रगति पंक्तियाँ छिपाने के लिए, उस चैनल हेतु
  `streaming.preview.toolProgress` को `false` पर सेट करें (डिफ़ॉल्ट
  `true`)। कमांड/एक्ज़ीक्यूशन टेक्स्ट छिपाते हुए टूल-प्रगति पंक्तियाँ दृश्यमान रखने के लिए,
  `streaming.preview.commandText` को `"status"` या
  `streaming.progress.commandText` को `"status"` पर सेट करें; जारी किया गया व्यवहार
  बनाए रखने के लिए डिफ़ॉल्ट `"raw"` है। यह नीति उन ड्राफ़्ट/प्रगति चैनलों द्वारा साझा की जाती है,
  जो OpenClaw के संक्षिप्त प्रगति रेंडरर का उपयोग करते हैं, जिनमें Discord, Matrix,
  Microsoft Teams, Mattermost, Slack ड्राफ़्ट पूर्वावलोकन और Telegram शामिल हैं। पूर्वावलोकन
  संपादन पूरी तरह अक्षम करने के लिए, `streaming.mode` को `off` पर सेट करें।

## प्रगति ड्राफ़्ट रेंडरिंग

प्रगति-मोड ड्राफ़्ट (`streaming.progress.*`) सीमित होते हैं और प्रत्येक
चैनल के लिए कॉन्फ़िगर किए जा सकते हैं:

| कुंजी                               | डिफ़ॉल्ट       | व्यवहार                                                       |
| --------------------------------- | ------------- | -------------------------------------------------------------- |
| `streaming.progress.maxLines`     | `8`           | ड्राफ़्ट लेबल के नीचे रखी जाने वाली संक्षिप्त प्रगति पंक्तियों की अधिकतम संख्या          |
| `streaming.progress.maxLineChars` | `120`         | काटे जाने से पहले प्रत्येक संक्षिप्त पंक्ति में अधिकतम वर्ण (शब्द-सचेत) |
| `streaming.progress.label`        | `"auto"`      | ड्राफ़्ट शीर्षक; एक कस्टम स्ट्रिंग, या इसे छिपाने के लिए `false`            |
| `streaming.progress.labels`       | अंतर्निहित समूह | `label: "auto"` होने पर उपयोग किए जाने वाले संभावित लेबल                     |

### टिप्पणी प्रगति लेन

टूल-प्रगति के अतिरिक्त, संक्षिप्त प्रगति रेंडरर ड्राफ़्ट में एक और लेन
दिखा सकता है:

- **`streaming.progress.commentary`** - मॉडल की टूल-पूर्व
  **टिप्पणी** (एक छोटा "मैं जाँच करूँगा... फिर..." वर्णन) प्रगति ड्राफ़्ट में
  टूल पंक्तियों के साथ बीच-बीच में रेंडर करें। प्रगति मोड में Discord और Telegram पर,
  यह वैकल्पिक लेन बंद होने पर भी वही प्रस्तावना स्थिति शीर्षक प्रदान करती है;
  अन्य चैनल अपना मौजूदा प्रगति व्यवहार बनाए रखते हैं। देखें
  [प्रगति ड्राफ़्ट](/hi/concepts/progress-drafts#status-headline)।

```json
{
  "channels": {
    "discord": {
      "streaming": { "mode": "progress", "progress": { "commentary": true } }
    }
  }
}
```

प्रगति पंक्तियाँ दृश्यमान रखें, लेकिन कच्चा कमांड/एक्ज़ीक्यूशन टेक्स्ट छिपाएँ:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```

किसी अन्य संक्षिप्त प्रगति चैनल कुंजी के अंतर्गत समान संरचना का उपयोग करें, उदाहरण के लिए
`channels.discord`, `channels.matrix`, `channels.msteams`,
`channels.mattermost`, या Slack ड्राफ़्ट पूर्वावलोकन। प्रगति-ड्राफ़्ट मोड के लिए,
समान नीति `streaming.progress` के अंतर्गत रखें:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "progress",
        "progress": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```
