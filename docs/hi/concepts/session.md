OpenClaw प्रत्येक इनबाउंड संदेश को उसके स्रोत—DM, समूह चैट, Cron जॉब आदि—के आधार पर एक **सत्र** में रूट करता है। सत्र की पूरी स्थिति का स्वामित्व **Gateway** के पास होता है; UI क्लाइंट सत्र डेटा के लिए Gateway से क्वेरी करते हैं।

व्यक्तिगत एजेंट के डिफ़ॉल्ट—आपके सभी DM चैनलों द्वारा साझा की जाने वाली एक निरंतर बातचीत, जिसमें समूह गतिविधि और पृष्ठभूमि कार्य भी प्रवाहित होते हैं—के लिए [मुख्य सत्र](/concepts/main-session) देखें।

## संदेशों को कैसे रूट किया जाता है

| स्रोत          | व्यवहार                  |
| --------------- | ------------------------- |
| सीधे संदेश | डिफ़ॉल्ट रूप से साझा सत्र |
| समूह चैट     | प्रत्येक समूह के लिए पृथक        |
| रूम/चैनल  | प्रत्येक रूम के लिए पृथक         |
| Cron जॉब       | प्रत्येक रन के लिए नया सत्र     |
| Webhook        | प्रत्येक हुक के लिए पृथक         |

## DM पृथक्करण

डिफ़ॉल्ट रूप से, निरंतरता के लिए सभी DM एक सत्र साझा करते हैं, जो एकल-उपयोगकर्ता सेटअप के लिए उपयुक्त है।

<Warning>
यदि कई लोग आपके एजेंट को संदेश भेज सकते हैं, तो DM पृथक्करण सक्षम करें। इसके बिना, सभी उपयोगकर्ता समान बातचीत संदर्भ साझा करते हैं, इसलिए Alice के निजी संदेश Bob को दिखाई देंगे।
</Warning>

```json5
{
  session: {
    dmScope: "per-channel-peer", // चैनल + प्रेषक के आधार पर पृथक करें
  },
}
```

`session.dmScope` विकल्प:

| मान                      | व्यवहार                                                 |
| -------------------------- | -------------------------------------------------------- |
| `main` (डिफ़ॉल्ट)           | सभी DM [मुख्य सत्र](/concepts/main-session) साझा करते हैं |
| `per-peer`                 | सभी चैनलों में प्रेषक के आधार पर पृथक करें                       |
| `per-channel-peer`         | चैनल + प्रेषक के आधार पर पृथक करें (अनुशंसित)                |
| `per-account-channel-peer` | खाता + चैनल + प्रेषक के आधार पर पृथक करें                    |

<Tip>
यदि वही व्यक्ति कई चैनलों से आपसे संपर्क करता है, तो उसकी पहचानों को एक मानक पीयर आईडी से मैप करने के लिए `session.identityLinks` का उपयोग करें, ताकि वे एक सत्र साझा करें।
</Tip>

### लिंक किए गए चैनलों को डॉक करना

डॉक कमांड नया सत्र शुरू किए बिना वर्तमान सीधे-चैट सत्र के उत्तर रूट को किसी अन्य लिंक किए गए चैनल पर ले जाते हैं। उदाहरणों, कॉन्फ़िगरेशन और समस्या निवारण के लिए [चैनल डॉकिंग](/hi/concepts/channel-docking) देखें।

`openclaw security audit` से अपना सेटअप सत्यापित करें।

## अलग-अलग बातचीत में याद रखना

अलग ट्रांसक्रिप्ट प्रत्येक बातचीत के स्थानीय इतिहास को नियंत्रित करते हैं। किसी व्यक्तिगत या पूर्णतः विश्वसनीय एजेंट के लिए, `memorySearch.rememberAcrossConversations: true` उस एजेंट की अन्य निजी बातचीत में एक वैकल्पिक पुनर्प्राप्ति चरण जोड़ता है; यह उनके ट्रांसक्रिप्ट को संयोजित नहीं करता।

निजी प्रत्यक्ष और स्थायी स्पष्ट UI बातचीत एक-दूसरे को प्रासंगिक संदर्भ प्रदान कर सकती हैं। समूह और चैनल दोनों दिशाओं में अलग रहते हैं: उनके ट्रांसक्रिप्ट निजी स्मरण स्रोत नहीं होते और उन बातचीत के उत्तरों को निजी ट्रांसक्रिप्ट संदर्भ नहीं मिलता। वर्तमान बातचीत भी शामिल नहीं होती क्योंकि उसका इतिहास पहले से लोड है।

यह सेटिंग सत्र कुंजियां, DM स्कोप, रूटिंग, डिलीवरी या `tools.sessions.visibility` नहीं बदलती। `MEMORY.md` और `memory/*.md` में साझा कार्यस्थान मेमोरी भी अपना मौजूदा व्यवहार बनाए रखती है। वर्तमान मेमोरी प्रदाता को संरक्षित निजी ट्रांसक्रिप्ट स्मरण का समर्थन करना आवश्यक है; Lossless Claw जैसे संदर्भ इंजन स्वतंत्र रहते हैं और इसके साथ चल सकते हैं। सेटअप और रनटाइम विवरण के लिए [Active Memory](/hi/concepts/active-memory#remember-across-conversations) देखें।

## सत्र जीवनचक्र

सत्रों का पुनः उपयोग तब तक होता है, जब तक आप उन्हें मैन्युअल रूप से रीसेट नहीं करते या स्वचालित रीसेट नीति नहीं चुनते:

- **कोई स्वचालित रीसेट नहीं** (डिफ़ॉल्ट `mode: "none"`) - सत्र समान
  `sessionId` बनाए रखते हैं; बातचीत बढ़ने पर Compaction सक्रिय संदर्भ को प्रबंधित करता है।
- **दैनिक रीसेट** (`mode: "daily"`) - Gateway होस्ट पर कॉन्फ़िगर किए गए स्थानीय
  घंटे (`session.reset.atHour`, डिफ़ॉल्ट `4`, 0-23) पर नया सत्र चुनें। दैनिक
  ताज़गी इस पर आधारित होती है कि वर्तमान `sessionId` कब शुरू हुआ, न कि बाद में
  मेटाडेटा कब लिखा गया।
- **निष्क्रियता रीसेट** (`mode: "idle"`) - `session.reset.idleMinutes`
  की निष्क्रियता के बाद नया सत्र चुनें। निष्क्रियता की ताज़गी अंतिम वास्तविक उपयोगकर्ता/चैनल
  इंटरैक्शन पर आधारित होती है, इसलिए Heartbeat, Cron और exec सिस्टम इवेंट
  सत्र को सक्रिय नहीं रखते।
- **मैन्युअल रीसेट** - चैट में `/new` या `/reset` टाइप करें। `/new <model>`
  मॉडल भी बदलता है।

जब दैनिक और निष्क्रियता, दोनों रीसेट कॉन्फ़िगर किए जाते हैं, तो जो पहले समाप्त होता है वही लागू होता है। Heartbeat, Cron, exec और अन्य सिस्टम-इवेंट टर्न सत्र मेटाडेटा लिख सकते हैं, लेकिन वे लेखन दैनिक या निष्क्रियता रीसेट की ताज़गी नहीं बढ़ाते। जब रीसेट सत्र को बदलता है, तो पुराने सत्र के लिए कतारबद्ध सिस्टम-इवेंट सूचनाएं हटा दी जाती हैं, ताकि पुरानी पृष्ठभूमि अपडेट नए सत्र के पहले प्रॉम्प्ट के आरंभ में न जोड़ी जाएं।

सक्रिय प्रदाता-स्वामित्व वाले CLI सत्र वाले सत्र भी समान कोई-स्वचालित-रीसेट-नहीं डिफ़ॉल्ट का पालन करते हैं। जब ऐसे सत्रों को टाइमर पर समाप्त होना चाहिए, तो `/reset` का उपयोग करें या `session.reset` को स्पष्ट रूप से कॉन्फ़िगर करें।

स्वचालित रीसेट को वैश्विक रूप से चुनें, फिर उन्हें प्रत्येक चैट प्रकार या चैनल के अनुसार ओवरराइड करें:

```json5
{
  session: {
    reset: { mode: "daily", atHour: 4 },
    resetByType: {
      group: { mode: "idle", idleMinutes: 120 },
      thread: { mode: "daily", atHour: 6 },
    },
    resetByChannel: {
      discord: { mode: "idle", idleMinutes: 10080 },
    },
  },
}
```

`resetByType` `direct` (पुराना उपनाम `dm`), `group` और `thread` का समर्थन करता है।
जब कोई `session.reset`/`resetByType` ब्लॉक सेट न हो, तब पुराना शीर्ष-स्तरीय `session.idleMinutes` निष्क्रियता-मोड डिफ़ॉल्ट के लिए संगतता उपनाम के रूप में अब भी काम करता है।

## स्थिति कहां रहती है

- **रनटाइम सत्र पंक्तियां:** `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- **संग्रहीत ट्रांसक्रिप्ट फ़ाइलें:** `~/.openclaw/agents/<agentId>/sessions/`
- **पुरानी पंक्ति माइग्रेशन स्रोत:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`

प्रति-एजेंट SQLite डेटाबेस में सत्र पंक्तियां अलग-अलग जीवनचक्र टाइमस्टैंप रखती हैं:

- `sessionStartedAt`: वर्तमान `sessionId` कब शुरू हुआ; दैनिक रीसेट इसका उपयोग करता है।
- `lastInteractionAt`: अंतिम उपयोगकर्ता/चैनल इंटरैक्शन जो निष्क्रियता अवधि बढ़ाता है।
- `updatedAt`: स्टोर-पंक्ति में अंतिम परिवर्तन; सूचीकरण और छंटाई के लिए उपयोगी, लेकिन
  दैनिक/निष्क्रियता रीसेट की ताज़गी के लिए प्रामाणिक नहीं।

पुराने इंस्टॉलेशन से माइग्रेशन के दौरान, Gateway स्टार्टअप और `openclaw doctor
--fix` पुरानी `sessions.json` पंक्तियों और हालिया ट्रांसक्रिप्ट JSONL इतिहास को स्वचालित रूप से SQLite में आयात करते हैं। `sessionStartedAt` के बिना पंक्तियों का समाधान उपलब्ध होने पर पुराने ट्रांसक्रिप्ट JSONL सत्र हेडर से किया जाता है। यदि किसी पुरानी पंक्ति में `lastInteractionAt` भी नहीं है, तो निष्क्रियता की ताज़गी बाद के हिसाब-किताब लेखन के बजाय उस सत्र के प्रारंभ समय पर वापस जाती है। जब स्पष्ट निरीक्षण या सत्यापन प्रमाण चाहिए, तो `openclaw doctor --session-sqlite inspect
--session-sqlite-all-agents` और [Doctor माइग्रेशन अनुक्रम](/hi/cli/doctor#session-sqlite-migration) का उपयोग करें।

## सत्र रखरखाव

OpenClaw समय के साथ सत्र स्टोरेज को `session.maintenance` के माध्यम से सीमित करता है; डिफ़ॉल्ट नीचे दिखाए गए हैं:

```json5
{
  session: {
    maintenance: {
      mode: "enforce", // "enforce" सफ़ाई लागू करता है; "warn" केवल रिपोर्ट करता है
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

उत्पादन-स्तरीय `maxEntries` सीमाओं के लिए, Gateway रनटाइम लेखन एक छोटे उच्च-सीमा बफ़र का उपयोग करता है और बैच में कॉन्फ़िगर की गई सीमा तक वापस सफ़ाई करता है। Gateway स्टार्टअप के दौरान सत्र स्टोर पठन प्रविष्टियों की छंटाई या सीमा लागू नहीं करता, इसलिए स्टार्टअप और पृथक Cron सत्रों को पूरे स्टोर की सफ़ाई का भार नहीं उठाना पड़ता। `openclaw sessions cleanup --enforce` सीमा को तुरंत लागू करता है।

Gateway मॉडल-रन जांच सत्र डिफ़ॉल्ट रूप से अल्पकालिक होते हैं। `agent:*:explicit:model-run-<uuid>` से मेल खाने वाली पंक्तियां निश्चित `24h` प्रतिधारण का उपयोग करती हैं, लेकिन सफ़ाई दबाव-नियंत्रित होती है: यह पुरानी जांच पंक्तियों को केवल तब हटाती है जब सत्र-प्रविष्टि रखरखाव/सीमा दबाव उत्पन्न होता है, और व्यापक पुरानी-प्रविष्टि आयु कटऑफ़ तथा प्रविष्टि सीमा से पहले चलती है। सामान्य प्रत्यक्ष, समूह, थ्रेड, Cron, हुक, Heartbeat, ACP और उप-एजेंट सत्रों पर यह 24h प्रतिधारण लागू नहीं होता।

रखरखाव स्थायी बाहरी बातचीत पॉइंटरों को संरक्षित रखता है, जिनमें समूह सत्र और थ्रेड-स्कोप चैट सत्र शामिल हैं, जबकि कृत्रिम Cron, हुक, Heartbeat, ACP और उप-एजेंट प्रविष्टियों को समय के साथ समाप्त होने देता है।

यदि आपने पहले DM पृथक्करण का उपयोग किया था और बाद में `session.dmScope` को `main` पर वापस कर दिया, तो `openclaw sessions cleanup --dry-run --fix-dm-scope` से पुरानी पीयर-कुंजीयुक्त DM पंक्तियों का पूर्वावलोकन करें। वही फ़्लैग लागू करने पर वे पुरानी प्रत्यक्ष-DM पंक्तियां सेवानिवृत्त हो जाती हैं और उनके ट्रांसक्रिप्ट हटाए गए संग्रह के रूप में रखे जाते हैं।

किसी भी रखरखाव रन का पूर्वावलोकन `openclaw sessions cleanup --dry-run` से करें।
