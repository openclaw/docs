एक **कॉन्टेक्स्ट इंजन** यह नियंत्रित करता है कि OpenClaw प्रत्येक रन के लिए मॉडल कॉन्टेक्स्ट कैसे बनाता है: किन संदेशों को शामिल करना है, पुराने इतिहास का सारांश कैसे बनाना है, और सबएजेंट सीमाओं के आर-पार कॉन्टेक्स्ट कैसे प्रबंधित करना है।

OpenClaw में एक अंतर्निहित `legacy` इंजन शामिल है और वह डिफ़ॉल्ट रूप से इसका उपयोग करता है। Plugin इंजन केवल तभी इंस्टॉल और चुनें, जब आपको अलग संयोजन, Compaction या क्रॉस-सेशन रिकॉल व्यवहार चाहिए।

## यह कैसे काम करता है

जब भी OpenClaw कोई मॉडल प्रॉम्प्ट चलाता है, कॉन्टेक्स्ट इंजन जीवनचक्र के चार बिंदुओं पर भाग लेता है:

<AccordionGroup>
  <Accordion title="1. अंतर्ग्रहण">
    सेशन में नया संदेश जोड़े जाने पर कॉल किया जाता है। इंजन संदेश को अपने डेटा स्टोर में संग्रहित या अनुक्रमित कर सकता है।
  </Accordion>
  <Accordion title="2. संयोजन">
    प्रत्येक मॉडल रन से पहले कॉल किया जाता है। इंजन टोकन बजट में समाने वाले संदेशों का क्रमबद्ध सेट (और एक वैकल्पिक `systemPromptAddition`) लौटाता है।
  </Accordion>
  <Accordion title="3. संक्षिप्तीकरण">
    कॉन्टेक्स्ट विंडो भर जाने पर या उपयोगकर्ता द्वारा `/compact` चलाने पर कॉल किया जाता है। इंजन स्थान खाली करने के लिए पुराने इतिहास का सारांश बनाता है।
  </Accordion>
  <Accordion title="4. टर्न के बाद">
    रन पूरा होने के बाद कॉल किया जाता है। इंजन स्थिति स्थायी कर सकता है, पृष्ठभूमि Compaction ट्रिगर कर सकता है या इंडेक्स अपडेट कर सकता है।
  </Accordion>
</AccordionGroup>

इंजन बूटस्ट्रैप, सफल टर्न या Compaction के बाद ट्रांसक्रिप्ट रखरखाव (`runtimeContext.rewriteTranscriptEntries()` के माध्यम से सुरक्षित पुनर्लेखन) के लिए वैकल्पिक `maintain()` विधि भी लागू कर सकते हैं। उत्तर को अवरुद्ध करने के बजाय इसे स्थगित कार्य के रूप में चलाने के लिए `info.turnMaintenanceMode: "background"` सेट करें।

बंडल किए गए गैर-ACP Codex हार्नेस के लिए, OpenClaw संयोजित कॉन्टेक्स्ट को Codex डेवलपर निर्देशों और वर्तमान टर्न प्रॉम्प्ट में प्रक्षेपित करके यही जीवनचक्र लागू करता है। Codex अभी भी अपने मूल थ्रेड इतिहास और मूल कॉम्पैक्टर का स्वामी रहता है।

### सबएजेंट जीवनचक्र (वैकल्पिक)

OpenClaw दो वैकल्पिक सबएजेंट जीवनचक्र हुक कॉल करता है:

<ParamField path="prepareSubagentSpawn" type="method">
  चाइल्ड रन शुरू होने से पहले साझा कॉन्टेक्स्ट स्थिति तैयार करें। हुक को पैरेंट/चाइल्ड सेशन कुंजियाँ, `contextMode` (`isolated` या `fork`), उपलब्ध ट्रांसक्रिप्ट आईडी/फ़ाइलें और वैकल्पिक TTL प्राप्त होते हैं। यदि यह रोलबैक हैंडल लौटाता है, तो तैयारी सफल होने के बाद स्पॉन विफल होने पर OpenClaw उसे कॉल करता है। `lightContext` का अनुरोध करने वाले और `contextMode="isolated"` में रिज़ॉल्व होने वाले मूल सबएजेंट स्पॉन जानबूझकर इस हुक को छोड़ देते हैं, ताकि चाइल्ड कॉन्टेक्स्ट-इंजन-प्रबंधित प्री-स्पॉन स्थिति के बिना हल्के बूटस्ट्रैप कॉन्टेक्स्ट से शुरू हो।
</ParamField>
<ParamField path="onSubagentEnded" type="method">
  सबएजेंट सेशन पूरा होने या हटाए जाने पर सफ़ाई करें।
</ParamField>

### सिस्टम प्रॉम्प्ट में जोड़

`assemble` विधि एक `systemPromptAddition` स्ट्रिंग लौटा सकती है। OpenClaw इसे रन के सिस्टम प्रॉम्प्ट से पहले जोड़ता है। इससे इंजन स्थिर वर्कस्पेस फ़ाइलों की आवश्यकता के बिना डायनेमिक रिकॉल मार्गदर्शन, पुनर्प्राप्ति निर्देश या कॉन्टेक्स्ट-जागरूक संकेत अंतःक्षेपित कर सकते हैं।

## विरासत इंजन

अंतर्निहित `legacy` इंजन OpenClaw के मूल व्यवहार को बनाए रखता है:

- **अंतर्ग्रहण**: कोई कार्रवाई नहीं (सेशन प्रबंधक संदेश स्थायित्व को सीधे संभालता है)।
- **संयोजन**: पास-थ्रू (रनटाइम में मौजूदा सैनिटाइज़ → वैलिडेट → लिमिट पाइपलाइन कॉन्टेक्स्ट संयोजन संभालती है)।
- **संक्षिप्तीकरण**: अंतर्निहित सारांशीकरण Compaction को सौंपता है, जो पुराने संदेशों का एक सारांश बनाता है और हाल के संदेशों को अक्षुण्ण रखता है।
- **टर्न के बाद**: कोई कार्रवाई नहीं।

विरासत इंजन टूल पंजीकृत नहीं करता या `systemPromptAddition` प्रदान नहीं करता।

जब कोई `plugins.slots.contextEngine` सेट नहीं होता (या उसे `"legacy"` पर सेट किया जाता है), तो इस इंजन का स्वचालित रूप से उपयोग किया जाता है।

## Plugin इंजन

कोई Plugin, Plugin API का उपयोग करके कॉन्टेक्स्ट इंजन पंजीकृत कर सकता है:

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function register(api) {
  api.registerContextEngine("my-engine", (ctx) => ({
    info: {
      id: "my-engine",
      name: "My Context Engine",
      ownsCompaction: true,
    },

    async ingest({ sessionId, message, isHeartbeat }) {
      // संदेश को अपने डेटा स्टोर में संग्रहित करें
      return { ingested: true };
    },

    async assemble({
      sessionId,
      sessionKey,
      messages,
      tokenBudget,
      availableTools,
      citationsMode,
    }) {
      // बजट में समाने वाले संदेश लौटाएँ
      return {
        messages: buildContext(messages, tokenBudget),
        estimatedTokens: countTokens(messages),
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentSessionKey: sessionKey,
        }),
      };
    },

    async compact({ sessionId, force }) {
      // पुराने कॉन्टेक्स्ट का सारांश बनाएँ
      return { ok: true, compacted: true };
    },
  }));
}
```

फ़ैक्टरी `ctx` में वैकल्पिक `config`, `agentDir` और `workspaceDir`
मान शामिल होते हैं, ताकि Plugin पहले जीवनचक्र कॉल से पहले प्रति-एजेंट या प्रति-वर्कस्पेस स्थिति आरंभ कर सकें। गैर-विरासत `assemble()` कॉल से पहले, होस्ट
पंजीकृत एसिंक मेमोरी प्रॉम्प्ट तैयारी पूरी करता है। सिंक्रोनस
`buildMemorySystemPromptAddition(...)` सहायक उस अपरिवर्तनीय रन स्नैपशॉट को पढ़ता है;
प्रदान किए गए टूल, उद्धरण, एजेंट और सेशन कॉन्टेक्स्ट को बिना बदलाव के आगे भेजें।

फिर इसे कॉन्फ़िगरेशन में सक्षम करें:

```json5
{
  plugins: {
    slots: {
      contextEngine: "my-engine",
    },
    entries: {
      "my-engine": {
        enabled: true,
      },
    },
  },
}
```

### ContextEngine इंटरफ़ेस

आवश्यक सदस्य:

| सदस्य             | प्रकार     | उद्देश्य                                                  |
| ------------------ | -------- | -------------------------------------------------------- |
| `info`             | प्रॉपर्टी | इंजन आईडी, नाम, संस्करण और यह कि क्या वह Compaction का स्वामी है |
| `ingest(params)`   | विधि   | एक संदेश संग्रहित करना                                   |
| `assemble(params)` | विधि   | मॉडल रन के लिए कॉन्टेक्स्ट बनाना (`AssembleResult` लौटाता है) |
| `compact(params)`  | विधि   | कॉन्टेक्स्ट को सारांशित/कम करना                                 |

`assemble`, निम्नलिखित के साथ एक `AssembleResult` लौटाता है:

<ParamField path="messages" type="Message[]" required>
  मॉडल को भेजने के लिए क्रमबद्ध संदेश।
</ParamField>
<ParamField path="estimatedTokens" type="number" required>
  संयोजित कॉन्टेक्स्ट में कुल टोकन का इंजन का अनुमान। OpenClaw इसका उपयोग Compaction थ्रेशोल्ड निर्णयों और डायग्नोस्टिक रिपोर्टिंग के लिए करता है।
</ParamField>
<ParamField path="systemPromptAddition" type="string">
  सिस्टम प्रॉम्प्ट से पहले जोड़ा जाता है।
</ParamField>
<ParamField path="promptAuthority" type='"assembled" | "preassembly_may_overflow"'>
  नियंत्रित करता है कि रनर अग्रसक्रिय ओवरफ़्लो
  पूर्व-जाँच के लिए किस टोकन अनुमान का उपयोग करता है। डिफ़ॉल्ट `"assembled"` है, जिसका अर्थ है कि Compaction का स्वामित्व न रखने वाले इंजनों के लिए केवल संयोजित
  प्रॉम्प्ट के अनुमान की जाँच की जाती है।
  `ownsCompaction: true` सेट करने वाले इंजन अपना प्रॉम्प्ट प्रवेश स्वयं प्रबंधित करते हैं,
  इसलिए OpenClaw डिफ़ॉल्ट रूप से सामान्य प्री-प्रॉम्प्ट पूर्व-जाँच छोड़ देता है। `"preassembly_may_overflow"` केवल तभी सेट करें, जब आपका संयोजित दृश्य
  अंतर्निहित ट्रांसक्रिप्ट में ओवरफ़्लो
  जोखिम छिपा सकता हो; तब रनर सामान्य
  पूर्व-जाँच सक्रिय रखता है और अग्रसक्रिय रूप से
  कॉम्पैक्ट करने का निर्णय लेते समय संयोजित अनुमान और
  प्री-असेंबली (अनविंडोड) सेशन-इतिहास अनुमान में से अधिकतम लेता है। दोनों ही स्थितियों में, आपके द्वारा लौटाए गए संदेश ही मॉडल को
  दिखाई देते हैं—`promptAuthority` केवल पूर्व-जाँच को प्रभावित करता है।
</ParamField>
<ParamField path="contextProjection" type="ContextEngineProjection">
  स्थायी बैकएंड थ्रेड वाले होस्ट (उदाहरण के लिए Codex app-server) के लिए वैकल्पिक प्रक्षेपण जीवनचक्र। स्थिर `epoch` के साथ `mode: "thread_bootstrap"` होस्ट से संयोजित कॉन्टेक्स्ट को प्रत्येक एपॉक में एक बार अंतःक्षेपित करने और एपॉक बदलने तक बैकएंड थ्रेड का पुनः उपयोग करने को कहता है, बजाय प्रत्येक टर्न पर दोबारा प्रक्षेपित करने के। सामान्य प्रति-टर्न प्रक्षेपण के लिए यह फ़ील्ड छोड़ दें।
</ParamField>

`compact` एक `CompactResult` लौटाता है। जब Compaction सक्रिय सेशन
पहचान बदलता है, तो `result.sessionTarget` (सेशन पहचान और स्टोर स्कोप रखने वाला एक टाइप्ड `ContextEngineSessionTarget`)
उस उत्तराधिकारी सेशन की पहचान करता है जिसका उपयोग अगली
पुनः कोशिश या टर्न को करना आवश्यक है; `result.sessionId` उत्तराधिकारी आईडी को प्रतिबिंबित करता है।

वैकल्पिक सदस्य:

| सदस्य                         | प्रकार   | उद्देश्य                                                                                                                                      |
| ------------------------------ | ------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `bootstrap(params)`            | विधि | किसी सेशन के लिए इंजन स्थिति आरंभ करना। इंजन द्वारा किसी सेशन को पहली बार देखने पर एक बार कॉल किया जाता है (उदाहरण के लिए, इतिहास आयात करना)।                              |
| `maintain(params)`             | विधि | बूटस्ट्रैप, सफल टर्न या Compaction के बाद ट्रांसक्रिप्ट रखरखाव। सुरक्षित पुनर्लेखन के लिए `runtimeContext.rewriteTranscriptEntries()` का उपयोग करें। |
| `ingestBatch(params)`          | विधि | पूर्ण टर्न को बैच के रूप में अंतर्ग्रहण करना। रन पूरा होने के बाद उस टर्न के सभी संदेशों के साथ एक साथ कॉल किया जाता है।                                  |
| `afterTurn(params)`            | विधि | रन-पश्चात जीवनचक्र कार्य (स्थिति स्थायी करना, पृष्ठभूमि Compaction ट्रिगर करना)।                                                                      |
| `prepareSubagentSpawn(params)` | विधि | चाइल्ड सेशन शुरू होने से पहले उसके लिए साझा स्थिति सेट अप करना।                                                                                    |
| `onSubagentEnded(params)`      | विधि | सबएजेंट समाप्त होने के बाद सफ़ाई करना।                                                                                                              |
| `dispose()`                    | विधि | संसाधन मुक्त करना। Gateway बंद होने या Plugin पुनः लोड होने के दौरान कॉल किया जाता है—प्रति-सेशन नहीं।                                                        |

### रनटाइम सेटिंग्स

OpenClaw के भीतर चलने वाले जीवनचक्र हुक को एक वैकल्पिक
`runtimeSettings` ऑब्जेक्ट प्राप्त होता है। यह संस्करणित, केवल-पढ़ने योग्य आंतरिक
प्रोड्यूसर/कंज़्यूमर API सतह है: OpenClaw इसे चयनित कॉन्टेक्स्ट
इंजन के लिए उत्पन्न करता है, और कॉन्टेक्स्ट इंजन जीवनचक्र हुक के भीतर इसका उपयोग करता है। इसे
उपयोगकर्ताओं के लिए सीधे रेंडर नहीं किया जाता और यह समर्पित रिपोर्टिंग सतह नहीं बनाता।

- `schemaVersion`: वर्तमान में `1`
- `runtime`: OpenClaw होस्ट, रनटाइम मोड (`normal`, `fallback`, या
  `degraded`), और वैकल्पिक हार्नेस/रनटाइम आईडी
- `contextEngineSelection`: चयनित कॉन्टेक्स्ट इंजन आईडी और चयन स्रोत
- `executionHost`: हुक को लागू करने वाले सरफ़ेस का होस्ट आईडी और लेबल
- `model`: अनुरोधित मॉडल, समाधान किया गया मॉडल, प्रदाता, और वैकल्पिक मॉडल परिवार
- `limits`: ज्ञात होने पर प्रॉम्प्ट टोकन बजट और अधिकतम आउटपुट टोकन
- `diagnostics`: ज्ञात होने पर बंद फ़ॉलबैक और अवनत होने के कारण के कोड

जो फ़ील्ड अज्ञात हो सकते हैं, उन्हें `null` के रूप में दर्शाया जाता है; रनटाइम मोड और चयन स्रोत जैसे
डिस्क्रिमिनेटर फ़ील्ड नॉन-नलेबल रहते हैं। पुराने इंजन संगत रहते हैं:
यदि कोई सख़्त लेगेसी इंजन `runtimeSettings` को अज्ञात प्रॉपर्टी मानकर अस्वीकार करता है,
तो OpenClaw इंजन को क्वारंटीन करने के बजाय उसके बिना लाइफ़साइकल कॉल का पुनः प्रयास
करता है।

### होस्ट आवश्यकताएँ

कॉन्टेक्स्ट इंजन `info.hostRequirements` पर होस्ट क्षमता आवश्यकताएँ घोषित कर सकते हैं।
OpenClaw ऑपरेशन शुरू करने से पहले इन आवश्यकताओं की जाँच करता है और चयनित
रनटाइम द्वारा इन्हें पूरा न कर पाने पर वर्णनात्मक त्रुटि के साथ बंद रूप से विफल होता है।

एजेंट रन के लिए, जब इंजन को `assemble()` के माध्यम से वास्तविक
मॉडल प्रॉम्प्ट नियंत्रित करना आवश्यक हो, तब `assemble-before-prompt` घोषित करें:

```ts
info: {
  id: "my-context-engine",
  name: "My Context Engine",
  hostRequirements: {
    "agent-run": {
      requiredCapabilities: ["assemble-before-prompt"],
      unsupportedMessage:
        "मूल Codex या OpenClaw एम्बेडेड रनटाइम का उपयोग करें, या लेगेसी कॉन्टेक्स्ट इंजन चुनें।",
    },
  },
}
```

मूल Codex और OpenClaw एम्बेडेड एजेंट रन `assemble-before-prompt` को पूरा करते हैं।
सामान्य CLI बैकएंड ऐसा नहीं करते, इसलिए इसकी आवश्यकता वाले इंजनों को CLI
प्रोसेस शुरू होने से पहले अस्वीकार कर दिया जाता है।

### विफलता पृथक्करण

OpenClaw चयनित Plugin इंजन को मुख्य उत्तर पथ से अलग रखता है। यदि कोई
गैर-लेगेसी इंजन अनुपलब्ध है, अनुबंध सत्यापन में विफल होता है, फ़ैक्टरी
निर्माण के दौरान अपवाद उत्पन्न करता है, या किसी लाइफ़साइकल विधि से अपवाद उत्पन्न करता है,
तो OpenClaw वर्तमान Gateway प्रोसेस के लिए उस इंजन को क्वारंटीन करता है और
कॉन्टेक्स्ट-इंजन कार्य को अंतर्निहित `legacy` इंजन पर अवनत कर देता है।
त्रुटि को विफल ऑपरेशन के साथ लॉग किया जाता है, ताकि ऑपरेटर एजेंट को
निःशब्द किए बिना Plugin की मरम्मत, अपडेट या उसे अक्षम कर सके।

होस्ट आवश्यकता विफलताएँ अलग होती हैं: जब कोई इंजन घोषित करता है कि रनटाइम में
आवश्यक क्षमता नहीं है, तो OpenClaw रन शुरू करने से पहले बंद रूप से विफल होता है।
यह उन इंजनों की सुरक्षा करता है जो असमर्थित होस्ट में चलने पर स्थिति को दूषित कर सकते हैं।

### ownsCompaction

`ownsCompaction` नियंत्रित करता है कि रन के लिए OpenClaw रनटाइम का अंतर्निहित प्रयास-अंतर्गत स्वचालित Compaction सक्षम रहे या नहीं:

<AccordionGroup>
  <Accordion title="ownsCompaction: true">
    इंजन Compaction व्यवहार का स्वामी होता है। OpenClaw उस रन के लिए OpenClaw रनटाइम के अंतर्निहित स्वचालित Compaction और सामान्य प्री-प्रॉम्प्ट ओवरफ़्लो पूर्व-जाँच को अक्षम करता है, और इंजन का `compact()` कार्यान्वयन `/compact`, प्रदाता ओवरफ़्लो रिकवरी Compaction, और `afterTurn()` में किए जाने वाले किसी भी सक्रिय Compaction के लिए उत्तरदायी होता है। जब इंजन `assemble()` से `promptAuthority: "preassembly_may_overflow"` लौटाता है, तब भी OpenClaw प्री-प्रॉम्प्ट ओवरफ़्लो सुरक्षा उपाय चलाता है।
  </Accordion>
  <Accordion title="ownsCompaction: false या सेट नहीं">
    प्रॉम्प्ट निष्पादन के दौरान OpenClaw रनटाइम का अंतर्निहित स्वचालित Compaction अब भी चल सकता है, लेकिन सक्रिय इंजन की `compact()` विधि को `/compact` और ओवरफ़्लो रिकवरी के लिए फिर भी कॉल किया जाता है।
  </Accordion>
</AccordionGroup>

<Warning>
`ownsCompaction: false` का अर्थ यह **नहीं** है कि OpenClaw स्वचालित रूप से लेगेसी इंजन के Compaction पथ पर फ़ॉलबैक करता है।
</Warning>

इसका अर्थ है कि Plugin के दो वैध पैटर्न हैं:

<Tabs>
  <Tab title="स्वामित्व मोड">
    अपना Compaction एल्गोरिदम लागू करें और `ownsCompaction: true` सेट करें।
  </Tab>
  <Tab title="प्रत्यायोजन मोड">
    `ownsCompaction: false` सेट करें और OpenClaw के अंतर्निहित Compaction व्यवहार का उपयोग करने के लिए `compact()` से `delegateCompactionToRuntime(...)` को `openclaw/plugin-sdk/core` कॉल करवाएँ।
  </Tab>
</Tabs>

सक्रिय गैर-स्वामित्व वाले इंजन के लिए निष्क्रिय `compact()` असुरक्षित है, क्योंकि यह उस इंजन स्लॉट के सामान्य `/compact` और ओवरफ़्लो-रिकवरी Compaction पथ को अक्षम कर देता है।

## कॉन्फ़िगरेशन संदर्भ

```json5
{
  plugins: {
    slots: {
      // सक्रिय कॉन्टेक्स्ट इंजन चुनें। डिफ़ॉल्ट: "legacy"।
      // Plugin इंजन का उपयोग करने के लिए इसे Plugin आईडी पर सेट करें।
      contextEngine: "legacy",
    },
  },
}
```

<Note>
रनटाइम पर स्लॉट अनन्य होता है—किसी दिए गए रन या Compaction ऑपरेशन के लिए केवल एक पंजीकृत कॉन्टेक्स्ट इंजन का समाधान किया जाता है। अन्य सक्षम `kind: "context-engine"` Plugin अब भी लोड हो सकते हैं और अपना पंजीकरण कोड चला सकते हैं; `plugins.slots.contextEngine` केवल यह चुनता है कि कॉन्टेक्स्ट इंजन की आवश्यकता होने पर OpenClaw किस पंजीकृत इंजन आईडी का समाधान करता है।
</Note>

<Note>
**Plugin अनइंस्टॉल:** जब आप वर्तमान में `plugins.slots.contextEngine` के रूप में चयनित Plugin को अनइंस्टॉल करते हैं, तो OpenClaw स्लॉट को वापस डिफ़ॉल्ट (`legacy`) पर रीसेट कर देता है। यही रीसेट व्यवहार `plugins.slots.memory` पर भी लागू होता है। मैन्युअल कॉन्फ़िगरेशन संपादन आवश्यक नहीं है।
</Note>

## सुझाव

- यह सत्यापित करने के लिए `openclaw doctor` का उपयोग करें कि आपका इंजन सही ढंग से लोड हो रहा है।
- इंजन बदलने पर मौजूदा सेशन अपने वर्तमान इतिहास के साथ जारी रहते हैं। नया इंजन भावी रन के लिए नियंत्रण संभालता है।
- इंजन त्रुटियाँ लॉग की जाती हैं और चयनित Plugin इंजन को वर्तमान Gateway प्रोसेस के लिए क्वारंटीन किया जाता है। OpenClaw उपयोगकर्ता टर्न के लिए `legacy` पर फ़ॉलबैक करता है, ताकि उत्तर जारी रह सकें, लेकिन फिर भी आपको खराब Plugin की मरम्मत या उसे अपडेट, अक्षम अथवा अनइंस्टॉल करना चाहिए।
- विकास के लिए, स्थानीय Plugin डायरेक्टरी को कॉपी किए बिना लिंक करने हेतु `openclaw plugins install -l ./my-engine` का उपयोग करें।
