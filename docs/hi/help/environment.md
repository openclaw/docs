OpenClaw कई स्रोतों से पर्यावरण चर प्राप्त करता है। नियम है: **मौजूदा मानों को कभी ओवरराइड न करें**।
वर्कस्पेस `.env` फ़ाइलें कम विश्वसनीय स्रोत हैं: वरीयता लागू करने से पहले OpenClaw वर्कस्पेस `.env` से प्रदाता क्रेडेंशियल और संरक्षित रनटाइम नियंत्रणों को अनदेखा करता है।

## वरीयता (उच्चतम से निम्नतम)

1. **प्रोसेस पर्यावरण** (Gateway प्रोसेस को पैरेंट शेल/डेमन से पहले से जो मिला हुआ है)।
2. **वर्तमान कार्यशील डायरेक्टरी में `.env`** (dotenv डिफ़ॉल्ट; ओवरराइड नहीं करता; प्रदाता क्रेडेंशियल और संरक्षित रनटाइम नियंत्रण अनदेखे किए जाते हैं)।
3. `~/.openclaw/.env` पर **वैश्विक `.env`** (जिसे `$OPENCLAW_STATE_DIR/.env` भी कहा जाता है; प्रदाता API कुंजियों के लिए अनुशंसित; ओवरराइड नहीं करता)।
4. `~/.openclaw/openclaw.json` में **कॉन्फ़िगरेशन `env` ब्लॉक** (केवल अनुपस्थित होने पर लागू होता है)।
5. **वैकल्पिक लॉगिन-शेल आयात** (`env.shellEnv.enabled` या `OPENCLAW_LOAD_SHELL_ENV=1`), केवल अनुपस्थित अपेक्षित कुंजियों के लिए लागू होता है।

डिफ़ॉल्ट स्टेट डायरेक्टरी का उपयोग करने वाले नए Ubuntu इंस्टॉलेशन पर, OpenClaw वैश्विक `.env` के बाद `~/.config/openclaw/gateway.env` को संगतता फ़ॉलबैक भी मानता है। यदि दोनों फ़ाइलें मौजूद हों और उनमें अंतर हो, तो OpenClaw `~/.openclaw/.env` को बनाए रखता है और चेतावनी प्रिंट करता है।

यदि कॉन्फ़िगरेशन फ़ाइल पूरी तरह अनुपस्थित हो, तो चरण 4 छोड़ दिया जाता है; सक्षम होने पर शेल आयात फिर भी चलता है।

## प्रदाता क्रेडेंशियल और वर्कस्पेस `.env`

प्रदाता API कुंजियाँ केवल किसी वर्कस्पेस `.env` में न रखें। OpenClaw वर्कस्पेस `.env` फ़ाइलों से प्रदाता क्रेडेंशियल और एंडपॉइंट-रीडायरेक्ट कुंजियों के एक बड़े समूह को ब्लॉक करता है, जिसमें प्रत्येक ज्ञात प्रदाता प्रमाणीकरण पर्यावरण चर (उदाहरण के लिए `GEMINI_API_KEY`, `GOOGLE_API_KEY`, `XAI_API_KEY`, `MISTRAL_API_KEY`, `GROQ_API_KEY`, `DEEPSEEK_API_KEY`, `PERPLEXITY_API_KEY`, `BRAVE_API_KEY`, `TAVILY_API_KEY`, `EXA_API_KEY`, `FIRECRAWL_API_KEY`), साथ ही `_API_HOST`, `_BASE_URL`, `_ENDPOINT`, या `_HOMESERVER` में समाप्त होने वाली कोई भी कुंजी और संपूर्ण `OPENCLAW_*`, `CLAWHUB_*`, `ANTHROPIC_API_KEY_*`, तथा `OPENAI_API_KEY_*` नेमस्पेस शामिल हैं।

इसके बजाय प्रदाता क्रेडेंशियल के लिए इन विश्वसनीय स्रोतों में से किसी एक का उपयोग करें:

- Gateway प्रोसेस पर्यावरण, जैसे शेल, launchd/systemd यूनिट, कंटेनर सीक्रेट या CI सीक्रेट।
- `~/.openclaw/.env` या `$OPENCLAW_STATE_DIR/.env` पर वैश्विक रनटाइम dotenv फ़ाइल।
- `~/.openclaw/openclaw.json` में कॉन्फ़िगरेशन `env` ब्लॉक।
- `env.shellEnv.enabled` या `OPENCLAW_LOAD_SHELL_ENV=1` सक्षम होने पर वैकल्पिक लॉगिन-शेल आयात।

यदि आपने पहले प्रदाता कुंजियाँ या एंडपॉइंट रूटिंग मान केवल किसी वर्कस्पेस `.env` में संग्रहीत किए थे, तो उन्हें ऊपर दिए गए विश्वसनीय स्रोतों में से किसी एक में ले जाएँ। वर्कस्पेस `.env` अब भी ऐसे सामान्य प्रोजेक्ट चर प्रदान कर सकता है जो क्रेडेंशियल, एंडपॉइंट रीडायरेक्ट, होस्ट ओवरराइड या `OPENCLAW_*` रनटाइम नियंत्रण नहीं हैं।

सुरक्षा संबंधी तर्क के लिए [वर्कस्पेस `.env` फ़ाइलें](/hi/gateway/security#workspace-env-files) देखें।

## कॉन्फ़िगरेशन `env` ब्लॉक

इनलाइन पर्यावरण चर सेट करने के दो समकक्ष तरीके (दोनों ओवरराइड नहीं करते):

```json5
{
  env: {
    OPENROUTER_API_KEY: "sk-or-...",
    vars: {
      GROQ_API_KEY: "gsk-...",
    },
  },
}
```

कॉन्फ़िगरेशन `env` ब्लॉक केवल शाब्दिक स्ट्रिंग मान स्वीकार करता है। यह
`file:...` मानों का विस्तार नहीं करता; उदाहरण के लिए, `XAI_API_KEY: "file:secrets/xai-api-key.txt"`
प्रदाताओं को ठीक उसी स्ट्रिंग के रूप में भेजा जाता है।

फ़ाइल-समर्थित प्रदाता कुंजियों के लिए, इसका समर्थन करने वाले क्रेडेंशियल फ़ील्ड पर SecretRef का उपयोग करें:

```json5
{
  secrets: {
    providers: {
      xai_key_file: {
        source: "file",
        path: "~/.openclaw/secrets/xai-api-key.txt",
        mode: "singleValue",
      },
    },
  },
  models: {
    providers: {
      xai: {
        apiKey: { source: "file", provider: "xai_key_file", id: "value" },
      },
    },
  },
}
```

समर्थित फ़ील्ड के लिए [सीक्रेट प्रबंधन](/hi/gateway/secrets) और
[SecretRef क्रेडेंशियल सतह](/hi/reference/secretref-credential-surface)
देखें।

## शेल पर्यावरण आयात

`env.shellEnv` आपका लॉगिन शेल चलाता है और केवल **अनुपस्थित** अपेक्षित कुंजियों को आयात करता है:

```json5
{
  env: {
    shellEnv: {
      enabled: true,
      timeoutMs: 15000,
    },
  },
}
```

समकक्ष पर्यावरण चर:

- `OPENCLAW_LOAD_SHELL_ENV=1`
- `OPENCLAW_SHELL_ENV_TIMEOUT_MS=15000` (डिफ़ॉल्ट `15000`)

## Exec शेल स्नैपशॉट

गैर-Windows Gateway होस्ट पर, bash और zsh `exec` कमांड डिफ़ॉल्ट रूप से स्टार्टअप स्नैपशॉट का उपयोग करते हैं।
इस पथ को अक्षम करने के लिए Gateway प्रोसेस पर्यावरण में `OPENCLAW_EXEC_SHELL_SNAPSHOT=0` सेट करें।
`false`, `no`, और `off` मान भी इसे अक्षम करते हैं। प्रति-कॉल `exec.env` मान
स्नैपशॉट को टॉगल या स्नैपशॉट कैश को रीडायरेक्ट नहीं कर सकते।

## रनटाइम द्वारा प्रविष्ट पर्यावरण चर

OpenClaw उत्पन्न की गई चाइल्ड प्रोसेस में संदर्भ मार्कर भी प्रविष्ट करता है:

- `OPENCLAW_SHELL=exec`: `exec` टूल के माध्यम से चलाए गए कमांड के लिए सेट होता है।
- `OPENCLAW_SHELL=acp-client`: ACP ब्रिज प्रोसेस उत्पन्न करने पर `openclaw acp client` के लिए सेट होता है।
- `OPENCLAW_SHELL=tui-local`: स्थानीय TUI `!` शेल कमांड के लिए सेट होता है।
- `OPENCLAW_CLI=1`: CLI एंट्री पॉइंट द्वारा उत्पन्न चाइल्ड प्रोसेस के लिए सेट होता है।

ये रनटाइम मार्कर हैं (आवश्यक उपयोगकर्ता कॉन्फ़िगरेशन नहीं)। संदर्भ-विशिष्ट नियम लागू करने के लिए
इनका उपयोग शेल/प्रोफ़ाइल लॉजिक में किया जा सकता है।

## UI पर्यावरण चर

- `OPENCLAW_THEME=light`: जब आपके टर्मिनल की पृष्ठभूमि हल्की हो, तब हल्के TUI पैलेट को बाध्य करता है।
- `OPENCLAW_THEME=dark`: गहरे TUI पैलेट को बाध्य करता है।
- `COLORFGBG`: यदि आपका टर्मिनल इसे निर्यात करता है, तो OpenClaw TUI पैलेट स्वतः चुनने के लिए पृष्ठभूमि रंग संकेत का उपयोग करता है।

## कॉन्फ़िगरेशन में पर्यावरण चर प्रतिस्थापन

आप `${VAR_NAME}` सिंटैक्स का उपयोग करके कॉन्फ़िगरेशन स्ट्रिंग मानों में पर्यावरण चर को सीधे संदर्भित कर सकते हैं:

```json5
{
  models: {
    providers: {
      "vercel-gateway": {
        apiKey: "${VERCEL_GATEWAY_API_KEY}",
      },
    },
  },
}
```

पूर्ण विवरण के लिए [कॉन्फ़िगरेशन: पर्यावरण चर प्रतिस्थापन](/hi/gateway/configuration-reference#env-var-substitution) देखें।

## सीक्रेट संदर्भ बनाम `${ENV}` स्ट्रिंग

OpenClaw पर्यावरण-संचालित दो पैटर्न का समर्थन करता है:

- कॉन्फ़िगरेशन मानों में `${VAR}` स्ट्रिंग प्रतिस्थापन।
- सीक्रेट संदर्भों का समर्थन करने वाले फ़ील्ड के लिए SecretRef ऑब्जेक्ट (`{ source: "env", provider: "default", id: "VAR" }`)।

दोनों सक्रियण के समय प्रोसेस पर्यावरण से मान हल करते हैं। SecretRef का विवरण [सीक्रेट प्रबंधन](/hi/gateway/secrets) में दर्ज है।
कॉन्फ़िगरेशन `env` ब्लॉक स्वयं SecretRefs या `file:...`
शॉर्टहैंड मानों को हल नहीं करता।

## एजेंट सहायक टूल डाउनलोड

OpenClaw को अपने पिन किए हुए `fd`
और `ripgrep` सहायक बाइनरी डाउनलोड करने से रोकने के लिए `OPENCLAW_OFFLINE=1` सेट करें। OpenClaw टूल
डायरेक्टरी में मौजूदा सहायक और कार्यशील सिस्टम बाइनरी उपयोग के योग्य बने रहते हैं; अनुपस्थित सहायक
नेटवर्क अनुरोध ट्रिगर करने के बजाय अनुपलब्ध रहता है।

### `OPENCLAW_HOME`

सेट होने पर, आंतरिक OpenClaw पथ डिफ़ॉल्ट के लिए `OPENCLAW_HOME` सिस्टम होम डायरेक्टरी (`$HOME` / `os.homedir()`) को प्रतिस्थापित करता है। इसमें डिफ़ॉल्ट स्टेट डायरेक्टरी, कॉन्फ़िगरेशन पथ, एजेंट डायरेक्टरी, क्रेडेंशियल, इंस्टॉलर ऑनबोर्डिंग वर्कस्पेस और `openclaw update --channel dev` द्वारा उपयोग किया जाने वाला डिफ़ॉल्ट डेवलपमेंट चेकआउट शामिल है।

**वरीयता:** `OPENCLAW_HOME` > `$HOME` > `USERPROFILE` > Android पर Termux `PREFIX` होम फ़ॉलबैक > `os.homedir()`

**उदाहरण** (macOS LaunchDaemon):

```xml
<key>EnvironmentVariables</key>
<dict>
  <key>OPENCLAW_HOME</key>
  <string>/Users/user</string>
</dict>
```

`OPENCLAW_HOME` को टिल्ड पथ (उदाहरण: `~/svc`) पर भी सेट किया जा सकता है, जिसे उपयोग से पहले उसी OS होम फ़ॉलबैक शृंखला का उपयोग करके विस्तारित किया जाता है।

`OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`, और `OPENCLAW_GIT_DIR` जैसे स्पष्ट पथ चर अब भी वरीयता लेते हैं। शेल स्टार्टअप फ़ाइल पहचान, पैकेज-मैनेजर सेटअप और होस्ट `~` विस्तार जैसे OS-अकाउंट कार्य अब भी वास्तविक सिस्टम होम का उपयोग कर सकते हैं।

## nvm उपयोगकर्ता: web_fetch TLS विफलताएँ

यदि Node.js को सिस्टम पैकेज मैनेजर के बजाय **nvm** के माध्यम से इंस्टॉल किया गया था, तो अंतर्निहित `fetch()`
nvm के बंडल किए हुए CA स्टोर का उपयोग करता है, जिसमें आधुनिक रूट CA (Let's Encrypt के लिए ISRG Root X1/X2,
DigiCert Global Root G2 आदि) अनुपस्थित हो सकते हैं। इसके कारण अधिकांश HTTPS साइटों पर `web_fetch` `"fetch failed"` के साथ विफल हो जाता है।

Linux पर, OpenClaw स्वचालित रूप से nvm का पता लगाता है और वास्तविक स्टार्टअप पर्यावरण में सुधार लागू करता है:

- `openclaw gateway install` सिस्टमड सेवा पर्यावरण में `NODE_EXTRA_CA_CERTS` लिखता है
- `openclaw` CLI एंट्रीपॉइंट Node स्टार्टअप से पहले `NODE_EXTRA_CA_CERTS` सेट करके स्वयं को फिर से निष्पादित करता है

**मैन्युअल सुधार (पुराने संस्करणों या सीधे `node ...` लॉन्च के लिए):**

OpenClaw शुरू करने से पहले वेरिएबल एक्सपोर्ट करें:

```bash
export NODE_EXTRA_CA_CERTS=/etc/ssl/certs/ca-certificates.crt
openclaw gateway run
```

इस वेरिएबल के लिए केवल `~/.openclaw/.env` में लिखने पर निर्भर न रहें; Node प्रक्रिया शुरू होते समय
`NODE_EXTRA_CA_CERTS` को पढ़ता है।

## पुराने एनवायरनमेंट वेरिएबल

OpenClaw केवल `OPENCLAW_*` एनवायरनमेंट वेरिएबल पढ़ता है। पुराने रिलीज़ के लीगेसी
`CLAWDBOT_*` और `MOLTBOT_*` प्रीफ़िक्स को बिना किसी सूचना के
अनदेखा कर दिया जाता है।

यदि स्टार्टअप के समय Gateway प्रक्रिया पर इनमें से कोई अब भी सेट है, तो OpenClaw
एक Node डिप्रिकेशन चेतावनी (`OPENCLAW_LEGACY_ENV_VARS`) जारी करता है, जिसमें
पहचाने गए प्रीफ़िक्स और उनकी कुल संख्या सूचीबद्ध होती है। लीगेसी प्रीफ़िक्स को
`OPENCLAW_` से बदलकर प्रत्येक मान का नाम बदलें (उदाहरण के लिए `CLAWDBOT_GATEWAY_TOKEN` को
`OPENCLAW_GATEWAY_TOKEN`); पुराने नामों का कोई प्रभाव नहीं पड़ता।
