# संगठन और नेमस्पेस दावे

ClawHub स्वामी हैंडल, संगठन हैंडल, skill स्लग, plugin पैकेज नाम और
पैकेज स्कोप को सार्वजनिक नेमस्पेस के रूप में उपयोग करता है। यदि कोई नेमस्पेस
किसी वास्तविक परियोजना, ब्रांड, पैकेज इकोसिस्टम या संगठन का प्रतीत होता है, लेकिन
ClawHub पर उस पर पहले से दावा किया गया है, वह आरक्षित या भ्रामक है, या उसे लेकर विवाद है,
तो कर्मचारियों से
[संगठन / नेमस्पेस दावा समस्या फ़ॉर्म](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के माध्यम से उसकी समीक्षा करने को कहें।

सार्वजनिक और गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस माध्यम का उपयोग करें। नेमस्पेस
दावों के लिए उत्पाद के अंदर उपलब्ध रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

जब आपको लगे कि वास्तविक दुनिया के स्वामित्व के कारण किसी नेमस्पेस को आरक्षित,
हस्तांतरित, पुनर्नामित, छिपाया, क्वारंटीन, उपनामित या किसी अन्य रूप में बदला जाना
चाहिए या नहीं, इसकी समीक्षा ClawHub कर्मचारियों को करनी चाहिए, तब नेमस्पेस दावा खोलें।

उदाहरणों में शामिल हैं:

- आपके GitHub संगठन, परियोजना, कंपनी या समुदाय से मेल खाने वाला संगठन हैंडल
- `@example-org/*` जैसा पैकेज स्कोप, जिसे केवल मेल खाने वाले
  ClawHub स्वामी के अंतर्गत प्रकाशित किया जाना चाहिए
- ऐसा skill स्लग या plugin पैकेज नाम, जो किसी परियोजना का प्रतिरूपण करता प्रतीत हो
- ब्रांड, ट्रेडमार्क, परियोजना के पुनर्नामकरण या पैकेज इतिहास से संबंधित विवाद
- हटाया गया, निष्क्रिय या पहुँच से बाहर स्वामी, जो नेमस्पेस के वास्तविक
  स्वामी को अवरुद्ध करता हो

यदि सूची स्वामित्व विवाद से परे असुरक्षित, दुर्भावनापूर्ण या भ्रामक है, तो
संबंधित मॉडरेशन या सुरक्षा मार्गदर्शन का भी पालन करें। नेमस्पेस दावा
फ़ॉर्म स्वामित्व समीक्षा के लिए है, आपातकालीन भेद्यता प्रकटीकरण के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उस स्वामी के साथ प्रकाशित कर रहे हैं जो नेमस्पेस से मेल खाता है।
plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे स्कोप वाले नाम
मेल खाने वाले `example-org` स्वामी के रूप में प्रकाशित किए जाने चाहिए।

यदि आप वर्तमान स्वामी को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
पुनर्नामित, हस्तांतरित, छिपाकर या हटाकर नेमस्पेस को सीधे ठीक करें। दावा तब
करें, जब आप वर्तमान स्वामी को प्रबंधित नहीं कर सकते या किसी विवाद का समाधान
कर्मचारियों द्वारा किया जाना आवश्यक हो।

## शामिल किए जाने वाले साक्ष्य

सार्वजनिक और गैर-संवेदनशील साक्ष्य का उपयोग करें। उपयोगी प्रमाणों में शामिल हैं:

- GitHub संगठन, रिपॉज़िटरी, रिलीज़ या अनुरक्षक इतिहास
- नेमस्पेस का उल्लेख करने वाले आधिकारिक परियोजना दस्तावेज़
- डोमेन या आधिकारिक ईमेल-डोमेन का प्रमाण
- npm, PyPI, crates.io या अन्य पैकेज-रजिस्ट्री स्कोप पर नियंत्रण
- ट्रेडमार्क, ब्रांड या परियोजना के स्वामित्व का ऐसा साक्ष्य, जिस पर सार्वजनिक
  रूप से चर्चा करना सुरक्षित हो
- स्रोत रिपॉज़िटरी इतिहास, पैकेज इतिहास या सार्वजनिक पुनर्नामकरण सूचनाएँ
- विवादित ClawHub स्वामी, skill, plugin, पैकेज या समस्या के लिंक

समझाएँ कि प्रत्येक लिंक क्या प्रमाणित करता है। कर्मचारियों को निजी क्रेडेंशियल
या रहस्यों की आवश्यकता के बिना संबंध समझने में सक्षम होना चाहिए।

## क्या शामिल न करें

सार्वजनिक GitHub समस्या में रहस्य या निजी प्रमाण न डालें। इनमें शामिल न करें:

- API टोकन, हस्ताक्षर कुंजियाँ या क्रेडेंशियल
- DNS चुनौती टोकन
- निजी कानूनी फ़ाइलें या अनुबंध
- व्यक्तिगत पहचान दस्तावेज़
- निजी ईमेल, निजी सुरक्षा रिपोर्ट या गोपनीय ग्राहक डेटा

दावा फ़ॉर्म पूछता है कि क्या संवेदनशील साक्ष्य के लिए निजी कर्मचारी चैनल की
आवश्यकता है। संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प
का उपयोग करें।

## संभावित परिणाम

साक्ष्य और जोखिम के आधार पर, ClawHub कर्मचारी किसी नेमस्पेस को आरक्षित कर सकते हैं,
स्वामित्व हस्तांतरित कर सकते हैं, किसी संसाधन को पुनर्नामित कर सकते हैं, मौजूदा सूची को
छिपा या क्वारंटीन कर सकते हैं, उपनाम या रीडायरेक्ट जोड़ सकते हैं, अधिक प्रमाण माँग सकते हैं
या अनुरोध अस्वीकार कर सकते हैं।

नेमस्पेस समीक्षा यह गारंटी नहीं देती कि हर मेल खाने वाला नाम हस्तांतरित किया जाएगा।
कर्मचारी सार्वजनिक साक्ष्य, मौजूदा उपयोग, सुरक्षा जोखिम और उपयोगकर्ता प्रभाव का मूल्यांकन करते हैं।
