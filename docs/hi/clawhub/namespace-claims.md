# संगठन और नेमस्पेस दावे

ClawHub स्वामी हैंडल, संगठन हैंडल, skill स्लग, Plugin पैकेज नाम, और
पैकेज स्कोप को सार्वजनिक नेमस्पेस के रूप में उपयोग करता है। यदि कोई नेमस्पेस
किसी वास्तविक परियोजना, ब्रांड, पैकेज इकोसिस्टम, या संगठन से संबंधित लगता है
लेकिन ClawHub पर पहले से दावा किया गया, आरक्षित, भ्रामक, या विवादित है, तो
कर्मचारियों से
[संगठन / नेमस्पेस दावा issue फ़ॉर्म](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के साथ इसकी समीक्षा करने को कहें।

सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस पथ का उपयोग करें। नेमस्पेस
दावों के लिए उत्पाद के भीतर की रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

जब आपको लगे कि ClawHub कर्मचारियों को यह समीक्षा करनी चाहिए कि वास्तविक
स्वामित्व के कारण किसी नेमस्पेस को आरक्षित, स्थानांतरित, नाम बदला, छिपाया,
क्वारंटीन किया, उपनाम दिया, या अन्यथा बदला जाना चाहिए, तब नेमस्पेस दावा खोलें।

उदाहरणों में शामिल हैं:

- ऐसा संगठन हैंडल जो आपके GitHub संगठन, परियोजना, कंपनी, या समुदाय से मेल खाता हो
- `@example-org/*` जैसा पैकेज स्कोप, जिसे केवल मेल खाते ClawHub स्वामी के अंतर्गत
  प्रकाशित होना चाहिए
- ऐसा skill स्लग या Plugin पैकेज नाम जो किसी परियोजना का प्रतिरूपण करता हुआ लगे
- ब्रांड, ट्रेडमार्क, परियोजना नाम बदलने, या पैकेज इतिहास से जुड़ा विवाद
- हटाया गया, निष्क्रिय, या पहुंच से बाहर स्वामी जो वैध नेमस्पेस स्वामी को रोकता हो

यदि सूचीकरण स्वामित्व विवाद से आगे unsafe, दुर्भावनापूर्ण, या भ्रामक है, तो
संबंधित moderation या security मार्गदर्शन का भी पालन करें। नेमस्पेस दावा फ़ॉर्म
स्वामित्व समीक्षा के लिए है, आपातकालीन vulnerability disclosure के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उसी स्वामी के साथ प्रकाशित कर रहे हैं जो नेमस्पेस से मेल
खाता है। Plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे scoped नाम
मेल खाते `example-org` स्वामी के रूप में प्रकाशित होने चाहिए।

यदि आप मौजूदा स्वामी को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
नाम बदलकर, स्थानांतरित, छिपाकर, या हटाकर सीधे नेमस्पेस ठीक करें। दावा तब उपयोग
करें जब आप मौजूदा स्वामी को प्रबंधित नहीं कर सकते या जब कर्मचारियों को कोई विवाद
सुलझाना हो।

## शामिल करने के लिए साक्ष्य

सार्वजनिक, गैर-संवेदनशील साक्ष्य का उपयोग करें। उपयोगी प्रमाण में शामिल हैं:

- GitHub संगठन, repo, release, या maintainer इतिहास
- आधिकारिक परियोजना दस्तावेज़ जो नेमस्पेस का नाम देते हों
- domain या आधिकारिक email-domain प्रमाण
- npm, PyPI, crates.io, या अन्य package-registry scope नियंत्रण
- trademark, brand, या project ownership साक्ष्य जिसे सार्वजनिक रूप से चर्चा करना सुरक्षित हो
- स्रोत repository इतिहास, package इतिहास, या सार्वजनिक rename notices
- विवादित ClawHub स्वामी, skill, Plugin, package, या issue के लिंक

बताएं कि प्रत्येक लिंक क्या प्रमाणित करता है। कर्मचारियों को निजी credentials या
secrets की आवश्यकता के बिना संबंध समझने में सक्षम होना चाहिए।

## क्या शामिल न करें

सार्वजनिक GitHub issue में secrets या निजी प्रमाण न डालें। शामिल न करें:

- API tokens, signing keys, या credentials
- DNS challenge tokens
- निजी legal files या contracts
- व्यक्तिगत identity documents
- निजी emails, निजी security reports, या confidential customer data

दावा फ़ॉर्म पूछता है कि क्या संवेदनशील साक्ष्य के लिए निजी staff channel की
आवश्यकता है। संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प
का उपयोग करें।

## संभावित परिणाम

साक्ष्य और जोखिम के आधार पर, ClawHub कर्मचारी किसी नेमस्पेस को आरक्षित कर सकते
हैं, स्वामित्व स्थानांतरित कर सकते हैं, किसी संसाधन का नाम बदल सकते हैं, मौजूदा
listing को छिपा या quarantine कर सकते हैं, alias या redirect जोड़ सकते हैं,
अधिक प्रमाण मांग सकते हैं, या अनुरोध अस्वीकार कर सकते हैं।

नेमस्पेस समीक्षा यह गारंटी नहीं देती कि हर मेल खाता नाम स्थानांतरित कर दिया
जाएगा। कर्मचारी सार्वजनिक साक्ष्य, मौजूदा उपयोग, security risk, और user impact
को तौलते हैं।
