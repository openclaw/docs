# संगठन और नेमस्पेस दावे

ClawHub, owner हैंडल, संगठन हैंडल, स्किल स्लग, Plugin पैकेज नाम, और
पैकेज स्कोप को सार्वजनिक नेमस्पेस के रूप में उपयोग करता है। यदि कोई नेमस्पेस
किसी वास्तविक प्रोजेक्ट, ब्रांड, पैकेज इकोसिस्टम, या संगठन से संबंधित लगता है,
लेकिन ClawHub पर पहले से दावा किया गया, आरक्षित, भ्रामक, या विवादित है, तो स्टाफ
से इसकी समीक्षा करने को
[संगठन / नेमस्पेस दावा इश्यू फ़ॉर्म](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के साथ कहें।

सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस पथ का उपयोग करें। नेमस्पेस
दावों के लिए इन-प्रोडक्ट रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

जब आपको लगता है कि ClawHub स्टाफ को यह समीक्षा करनी चाहिए कि वास्तविक स्वामित्व
के कारण किसी नेमस्पेस को आरक्षित, स्थानांतरित, नाम बदला, छिपाया, क्वारंटीन,
उपनामित, या अन्यथा बदला जाना चाहिए या नहीं, तब नेमस्पेस दावा खोलें।

उदाहरणों में शामिल हैं:

- ऐसा संगठन हैंडल जो आपके GitHub संगठन, प्रोजेक्ट, कंपनी, या समुदाय से मेल खाता हो
- `@example-org/*` जैसा पैकेज स्कोप, जिसे केवल मिलते-जुलते ClawHub owner के अंतर्गत
  प्रकाशित होना चाहिए
- ऐसा स्किल स्लग या Plugin पैकेज नाम जो किसी प्रोजेक्ट की नक़ल करता हुआ लगता हो
- कोई ब्रांड, ट्रेडमार्क, प्रोजेक्ट नाम-परिवर्तन, या पैकेज इतिहास विवाद
- हटाया गया, निष्क्रिय, या पहुंच से बाहर owner जो वैध नेमस्पेस owner को अवरुद्ध
  करता हो

यदि लिस्टिंग स्वामित्व विवाद से आगे असुरक्षित, दुर्भावनापूर्ण, या भ्रामक है, तो
संबंधित मॉडरेशन या सुरक्षा मार्गदर्शन का भी पालन करें। नेमस्पेस दावा फ़ॉर्म
स्वामित्व समीक्षा के लिए है, आपातकालीन भेद्यता प्रकटीकरण के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उस owner के साथ प्रकाशित कर रहे हैं जो नेमस्पेस से मेल
खाता है। Plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे स्कोप वाले
नाम मिलते-जुलते `example-org` owner के रूप में प्रकाशित होने चाहिए।

यदि आप मौजूदा owner को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
नाम बदलकर, स्थानांतरित करके, छिपाकर, या हटाकर नेमस्पेस को सीधे ठीक करें। दावा
तब उपयोग करें जब आप मौजूदा owner को प्रबंधित नहीं कर सकते या जब विवाद हल करने
के लिए स्टाफ की आवश्यकता हो।

## शामिल किए जाने वाले प्रमाण

सार्वजनिक, गैर-संवेदनशील प्रमाण का उपयोग करें। सहायक प्रमाण में शामिल हैं:

- GitHub संगठन, रिपो, रिलीज़, या मेंटेनर इतिहास
- आधिकारिक प्रोजेक्ट दस्तावेज़ जो नेमस्पेस का नाम बताते हों
- डोमेन या आधिकारिक ईमेल-डोमेन प्रमाण
- npm, PyPI, crates.io, या अन्य पैकेज-रजिस्ट्री स्कोप नियंत्रण
- ट्रेडमार्क, ब्रांड, या प्रोजेक्ट स्वामित्व प्रमाण जिसे सार्वजनिक रूप से
  चर्चा करना सुरक्षित हो
- स्रोत रिपॉज़िटरी इतिहास, पैकेज इतिहास, या सार्वजनिक नाम-परिवर्तन नोटिस
- विवादित ClawHub owner, स्किल, Plugin, पैकेज, या इश्यू के लिंक

बताएं कि प्रत्येक लिंक क्या साबित करता है। स्टाफ को निजी क्रेडेंशियल या रहस्यों
की आवश्यकता के बिना संबंध समझ में आना चाहिए।

## क्या शामिल न करें

सार्वजनिक GitHub इश्यू में रहस्य या निजी प्रमाण न डालें। शामिल न करें:

- API टोकन, साइनिंग कुंजियां, या क्रेडेंशियल
- DNS चुनौती टोकन
- निजी कानूनी फ़ाइलें या अनुबंध
- व्यक्तिगत पहचान दस्तावेज़
- निजी ईमेल, निजी सुरक्षा रिपोर्ट, या गोपनीय ग्राहक डेटा

दावा फ़ॉर्म पूछता है कि क्या संवेदनशील प्रमाण के लिए निजी स्टाफ चैनल की
आवश्यकता है। संवेदनशील सामग्री सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प
का उपयोग करें।

## संभावित परिणाम

प्रमाण और जोखिम के आधार पर, ClawHub स्टाफ किसी नेमस्पेस को आरक्षित कर सकता है,
स्वामित्व स्थानांतरित कर सकता है, किसी संसाधन का नाम बदल सकता है, मौजूदा लिस्टिंग
को छिपा या क्वारंटीन कर सकता है, उपनाम या पुनर्निर्देशन जोड़ सकता है, अधिक
प्रमाण मांग सकता है, या अनुरोध अस्वीकार कर सकता है।

नेमस्पेस समीक्षा यह गारंटी नहीं देती कि हर मेल खाने वाला नाम स्थानांतरित किया
जाएगा। स्टाफ सार्वजनिक प्रमाण, मौजूदा उपयोग, सुरक्षा जोखिम, और उपयोगकर्ता प्रभाव
का मूल्यांकन करता है।
