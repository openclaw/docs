# संगठन और नामस्थान दावे

ClawHub मालिक हैंडल, संगठन हैंडल, skill slugs, plugin पैकेज नामों और
पैकेज scopes को सार्वजनिक नामस्थानों के रूप में उपयोग करता है। यदि कोई नामस्थान
किसी वास्तविक परियोजना, ब्रांड, पैकेज पारिस्थितिकी तंत्र या संगठन से संबंधित
लगता है, लेकिन ClawHub पर पहले से दावा किया गया, आरक्षित, भ्रामक या विवादित है,
तो स्टाफ से
[संगठन / नामस्थान दावा issue फ़ॉर्म](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के साथ इसकी समीक्षा करने को कहें।

सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस मार्ग का उपयोग करें। नामस्थान
दावों के लिए इन-प्रोडक्ट रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

जब आपको लगता है कि ClawHub स्टाफ को यह समीक्षा करनी चाहिए कि वास्तविक स्वामित्व
के कारण किसी नामस्थान को आरक्षित, स्थानांतरित, पुनर्नामित, छिपाया, क्वारंटीन,
alias किया या अन्यथा बदला जाना चाहिए या नहीं, तब नामस्थान दावा खोलें।

उदाहरणों में शामिल हैं:

- ऐसा संगठन हैंडल जो आपके GitHub संगठन, परियोजना, कंपनी या समुदाय से मेल खाता हो
- `@example-org/*` जैसा पैकेज scope, जिसे केवल मेल खाने वाले ClawHub मालिक के
  अंतर्गत प्रकाशित होना चाहिए
- ऐसा skill slug या plugin पैकेज नाम जो किसी परियोजना का impersonation करता हुआ लगे
- ब्रांड, ट्रेडमार्क, परियोजना पुनर्नामकरण या पैकेज इतिहास विवाद
- हटाया गया, निष्क्रिय या पहुंच से बाहर मालिक जो वैध नामस्थान मालिक को रोकता हो

यदि listing स्वामित्व विवाद से आगे जाकर असुरक्षित, दुर्भावनापूर्ण या भ्रामक है,
तो संबंधित मॉडरेशन या सुरक्षा मार्गदर्शन का भी पालन करें। नामस्थान दावा फ़ॉर्म
स्वामित्व समीक्षा के लिए है, आपातकालीन vulnerability disclosure के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उसी मालिक के साथ प्रकाशित कर रहे हैं जो नामस्थान से मेल
खाता है। plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे scoped नामों
को मेल खाने वाले `example-org` मालिक के रूप में प्रकाशित किया जाना चाहिए।

यदि आप वर्तमान मालिक को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
पुनर्नामित, स्थानांतरित, छिपाकर या हटाकर सीधे नामस्थान ठीक करें। जब आप वर्तमान
मालिक को प्रबंधित नहीं कर सकते या जब स्टाफ को किसी विवाद का समाधान करना हो,
तब दावा उपयोग करें।

## शामिल करने के लिए प्रमाण

सार्वजनिक, गैर-संवेदनशील प्रमाण उपयोग करें। उपयोगी प्रमाणों में शामिल हैं:

- GitHub संगठन, repo, release या maintainer इतिहास
- आधिकारिक परियोजना docs जो नामस्थान का नाम बताते हों
- domain या आधिकारिक email-domain प्रमाण
- npm, PyPI, crates.io या अन्य package-registry scope नियंत्रण
- ट्रेडमार्क, ब्रांड या परियोजना स्वामित्व प्रमाण, जिस पर सार्वजनिक रूप से
  चर्चा करना सुरक्षित हो
- स्रोत repository इतिहास, पैकेज इतिहास या सार्वजनिक पुनर्नामकरण सूचनाएं
- विवादित ClawHub मालिक, skill, plugin, पैकेज या issue के links

समझाएं कि प्रत्येक link क्या साबित करता है। स्टाफ को निजी credentials या secrets
की आवश्यकता के बिना संबंध समझ में आना चाहिए।

## क्या शामिल न करें

किसी सार्वजनिक GitHub issue में secrets या निजी प्रमाण न डालें। शामिल न करें:

- API tokens, signing keys या credentials
- DNS challenge tokens
- निजी कानूनी फ़ाइलें या contracts
- व्यक्तिगत पहचान दस्तावेज
- निजी emails, निजी सुरक्षा reports या गोपनीय customer data

दावा फ़ॉर्म पूछता है कि क्या संवेदनशील प्रमाण के लिए निजी staff channel की
आवश्यकता है। संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प
का उपयोग करें।

## संभावित परिणाम

प्रमाण और जोखिम के आधार पर, ClawHub स्टाफ किसी नामस्थान को आरक्षित कर सकता है,
स्वामित्व स्थानांतरित कर सकता है, किसी संसाधन का पुनर्नामकरण कर सकता है, मौजूदा
listing को छिपा या क्वारंटीन कर सकता है, alias या redirect जोड़ सकता है, अधिक
प्रमाण मांग सकता है या अनुरोध अस्वीकार कर सकता है।

नामस्थान समीक्षा यह गारंटी नहीं देती कि हर मेल खाता नाम स्थानांतरित किया जाएगा।
स्टाफ सार्वजनिक प्रमाण, मौजूदा उपयोग, सुरक्षा जोखिम और उपयोगकर्ता प्रभाव को
तौलता है।
