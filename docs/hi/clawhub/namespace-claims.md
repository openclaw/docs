# संगठन और नेमस्पेस दावे

ClawHub स्वामी हैंडल, संगठन हैंडल, skill स्लग, Plugin पैकेज नाम, और
पैकेज स्कोप को सार्वजनिक नेमस्पेस के रूप में उपयोग करता है। यदि कोई नेमस्पेस
किसी वास्तविक परियोजना, ब्रांड, पैकेज इकोसिस्टम, या संगठन से संबंधित लगता है
लेकिन ClawHub पर पहले से दावा किया हुआ, आरक्षित, भ्रामक, या विवादित है, तो स्टाफ से
[संगठन / नेमस्पेस दावा issue फ़ॉर्म](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के साथ इसकी समीक्षा करने को कहें।

इस पथ का उपयोग सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए करें। नेमस्पेस
दावों के लिए इन-प्रोडक्ट रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

नेमस्पेस दावा तब खोलें जब आपको लगता हो कि ClawHub स्टाफ को समीक्षा करनी चाहिए कि
वास्तविक स्वामित्व के कारण किसी नेमस्पेस को आरक्षित, स्थानांतरित, नाम बदला,
छिपाया, क्वारंटीन, उपनामित, या अन्यथा बदला जाना चाहिए या नहीं।

उदाहरणों में शामिल हैं:

- ऐसा संगठन हैंडल जो आपके GitHub संगठन, परियोजना, कंपनी, या समुदाय से मेल खाता हो
- `@example-org/*` जैसा पैकेज स्कोप जिसे केवल मेल खाने वाले ClawHub स्वामी के
  अंतर्गत प्रकाशित होना चाहिए
- ऐसा skill स्लग या Plugin पैकेज नाम जो किसी परियोजना का प्रतिरूपण करता हुआ लगता हो
- ब्रांड, ट्रेडमार्क, परियोजना नाम बदलने, या पैकेज इतिहास से जुड़ा विवाद
- हटाया गया, निष्क्रिय, या पहुंच से बाहर स्वामी जो सही नेमस्पेस स्वामी को रोकता हो

यदि लिस्टिंग स्वामित्व विवाद से परे असुरक्षित, दुर्भावनापूर्ण, या भ्रामक है,
तो संबंधित मॉडरेशन या सुरक्षा मार्गदर्शन का भी पालन करें। नेमस्पेस दावा
फ़ॉर्म स्वामित्व समीक्षा के लिए है, आपातकालीन कमजोरी प्रकटीकरण के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उस स्वामी के साथ प्रकाशित कर रहे हैं जो नेमस्पेस से मेल खाता है।
Plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे स्कोप किए गए नामों को
मेल खाने वाले `example-org` स्वामी के रूप में प्रकाशित किया जाना चाहिए।

यदि आप वर्तमान स्वामी को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
नाम बदलकर, स्थानांतरित करके, छिपाकर, या हटाकर नेमस्पेस को सीधे ठीक करें। दावा
तब उपयोग करें जब आप वर्तमान स्वामी को प्रबंधित नहीं कर सकते या जब स्टाफ को
किसी विवाद का समाधान करना हो।

## शामिल करने योग्य प्रमाण

सार्वजनिक, गैर-संवेदनशील प्रमाण का उपयोग करें। उपयोगी प्रमाण में शामिल हैं:

- GitHub संगठन, repo, रिलीज़, या मेंटेनर इतिहास
- आधिकारिक परियोजना दस्तावेज़ जो नेमस्पेस का नाम बताते हों
- डोमेन या आधिकारिक ईमेल-डोमेन प्रमाण
- npm, PyPI, crates.io, या अन्य पैकेज-रजिस्ट्री स्कोप नियंत्रण
- ट्रेडमार्क, ब्रांड, या परियोजना स्वामित्व प्रमाण जिसे सार्वजनिक रूप से चर्चा करना सुरक्षित हो
- स्रोत repository इतिहास, पैकेज इतिहास, या सार्वजनिक नाम-बदलाव सूचनाएं
- विवादित ClawHub स्वामी, skill, Plugin, पैकेज, या issue के लिंक

बताएं कि प्रत्येक लिंक क्या साबित करता है। स्टाफ को निजी क्रेडेंशियल या रहस्यों
की आवश्यकता के बिना संबंध समझ में आना चाहिए।

## क्या शामिल न करें

किसी सार्वजनिक GitHub issue में रहस्य या निजी प्रमाण न डालें। शामिल न करें:

- API tokens, signing keys, या credentials
- DNS challenge tokens
- निजी कानूनी फ़ाइलें या अनुबंध
- व्यक्तिगत पहचान दस्तावेज़
- निजी ईमेल, निजी सुरक्षा रिपोर्ट, या गोपनीय ग्राहक डेटा

दावा फ़ॉर्म पूछता है कि क्या संवेदनशील प्रमाण के लिए निजी स्टाफ चैनल की आवश्यकता है।
संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प का उपयोग करें।

## संभावित परिणाम

प्रमाण और जोखिम के आधार पर, ClawHub स्टाफ किसी नेमस्पेस को आरक्षित कर सकता है,
स्वामित्व स्थानांतरित कर सकता है, संसाधन का नाम बदल सकता है, मौजूदा लिस्टिंग को
छिपा या क्वारंटीन कर सकता है, उपनाम या रीडायरेक्ट जोड़ सकता है, अधिक प्रमाण मांग
सकता है, या अनुरोध अस्वीकार कर सकता है।

नेमस्पेस समीक्षा यह गारंटी नहीं देती कि हर मेल खाने वाला नाम स्थानांतरित किया जाएगा।
स्टाफ सार्वजनिक प्रमाण, मौजूदा उपयोग, सुरक्षा जोखिम, और उपयोगकर्ता प्रभाव को तौलता है।
