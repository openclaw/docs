# Org और Namespace दावे

ClawHub मालिक हैंडल, org हैंडल, skill slugs, Plugin पैकेज नाम, और
पैकेज scopes को सार्वजनिक namespaces के रूप में उपयोग करता है। यदि कोई namespace
किसी वास्तविक दुनिया के प्रोजेक्ट, ब्रांड, पैकेज इकोसिस्टम, या संगठन से संबंधित
लगता है लेकिन ClawHub पर पहले से दावा किया गया, आरक्षित, भ्रामक, या विवादित है,
तो स्टाफ से
[Org / Namespace दावा issue form](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के साथ इसकी समीक्षा करने को कहें।

सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस पथ का उपयोग करें।
namespace दावों के लिए इन-प्रोडक्ट रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

जब आपको लगे कि ClawHub स्टाफ को यह समीक्षा करनी चाहिए कि वास्तविक दुनिया के
स्वामित्व के कारण किसी namespace को आरक्षित, स्थानांतरित, पुनर्नामित, छिपाया,
क्वारंटीन, उपनामित, या अन्यथा बदला जाना चाहिए या नहीं, तब namespace दावा खोलें।

उदाहरणों में शामिल हैं:

- कोई org हैंडल जो आपके GitHub org, प्रोजेक्ट, कंपनी, या समुदाय से मेल खाता हो
- `@example-org/*` जैसा कोई पैकेज scope जिसे केवल मिलते-जुलते ClawHub मालिक के
  अंतर्गत ही प्रकाशित होना चाहिए
- कोई skill slug या Plugin पैकेज नाम जो किसी प्रोजेक्ट का प्रतिरूपण करता हुआ लगे
- कोई ब्रांड, ट्रेडमार्क, प्रोजेक्ट पुनर्नामकरण, या पैकेज इतिहास विवाद
- कोई हटाया गया, निष्क्रिय, या पहुंच से बाहर मालिक जो उचित namespace मालिक को
  रोकता हो

यदि लिस्टिंग स्वामित्व विवाद से परे असुरक्षित, दुर्भावनापूर्ण, या भ्रामक है,
तो संबंधित मॉडरेशन या सुरक्षा मार्गदर्शन का भी पालन करें। namespace दावा फ़ॉर्म
स्वामित्व समीक्षा के लिए है, आपातकालीन भेद्यता प्रकटीकरण के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उस मालिक के साथ प्रकाशित कर रहे हैं जो namespace से मेल
खाता है। Plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे scoped नामों
को मिलते-जुलते `example-org` मालिक के रूप में प्रकाशित किया जाना चाहिए।

यदि आप मौजूदा मालिक को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
पुनर्नामित, स्थानांतरित, छिपाकर, या हटाकर namespace को सीधे ठीक करें। दावा तब
उपयोग करें जब आप मौजूदा मालिक को प्रबंधित नहीं कर सकते या जब स्टाफ को कोई विवाद
हल करना हो।

## शामिल करने योग्य प्रमाण

सार्वजनिक, गैर-संवेदनशील प्रमाण का उपयोग करें। उपयोगी प्रमाणों में शामिल हैं:

- GitHub org, repo, release, या maintainer इतिहास
- आधिकारिक प्रोजेक्ट दस्तावेज़ जो namespace का नाम बताते हों
- डोमेन या आधिकारिक ईमेल-डोमेन प्रमाण
- npm, PyPI, crates.io, या अन्य पैकेज-रजिस्ट्री scope नियंत्रण
- ट्रेडमार्क, ब्रांड, या प्रोजेक्ट स्वामित्व प्रमाण जिसे सार्वजनिक रूप से चर्चा
  करना सुरक्षित हो
- स्रोत repository इतिहास, पैकेज इतिहास, या सार्वजनिक पुनर्नामकरण सूचनाएं
- विवादित ClawHub मालिक, skill, Plugin, पैकेज, या issue के लिंक

समझाएं कि प्रत्येक लिंक क्या साबित करता है। स्टाफ को निजी credentials या secrets
की आवश्यकता के बिना संबंध समझ में आना चाहिए।

## क्या शामिल न करें

किसी सार्वजनिक GitHub issue में secrets या निजी प्रमाण न डालें। शामिल न करें:

- API tokens, signing keys, या credentials
- DNS challenge tokens
- निजी कानूनी फ़ाइलें या अनुबंध
- व्यक्तिगत पहचान दस्तावेज़
- निजी ईमेल, निजी सुरक्षा रिपोर्ट, या गोपनीय ग्राहक डेटा

दावा फ़ॉर्म पूछता है कि संवेदनशील प्रमाण के लिए निजी स्टाफ चैनल की आवश्यकता है
या नहीं। संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प का
उपयोग करें।

## संभावित परिणाम

प्रमाण और जोखिम के आधार पर, ClawHub स्टाफ किसी namespace को आरक्षित कर सकता है,
स्वामित्व स्थानांतरित कर सकता है, किसी संसाधन का पुनर्नामकरण कर सकता है, मौजूदा
लिस्टिंग को छिपा या क्वारंटीन कर सकता है, कोई उपनाम या redirect जोड़ सकता है,
अधिक प्रमाण मांग सकता है, या अनुरोध अस्वीकार कर सकता है।

namespace समीक्षा यह गारंटी नहीं देती कि हर मिलते-जुलते नाम को स्थानांतरित कर
दिया जाएगा। स्टाफ सार्वजनिक प्रमाण, मौजूदा उपयोग, सुरक्षा जोखिम, और उपयोगकर्ता
प्रभाव को तौलता है।
