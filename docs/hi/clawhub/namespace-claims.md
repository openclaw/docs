# संगठन और नेमस्पेस दावे

ClawHub स्वामी हैंडल, संगठन हैंडल, skill slugs, Plugin पैकेज नामों और
पैकेज scopes को सार्वजनिक नेमस्पेस के रूप में उपयोग करता है। यदि कोई नेमस्पेस
किसी वास्तविक परियोजना, ब्रांड, पैकेज ecosystem या संगठन से संबंधित लगता है,
लेकिन ClawHub पर पहले से दावा किया गया, आरक्षित, भ्रामक या विवादित है, तो
कर्मचारियों से
[संगठन / नेमस्पेस दावा issue form](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
के साथ इसकी समीक्षा करने के लिए कहें।

सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस पथ का उपयोग करें।
नेमस्पेस दावों के लिए in-product रिपोर्ट या खाता appeal form का उपयोग न करें।

## दावा कब खोलें

जब आपको लगे कि ClawHub कर्मचारियों को यह समीक्षा करनी चाहिए कि वास्तविक
स्वामित्व के कारण किसी नेमस्पेस को आरक्षित, हस्तांतरित, नाम बदला, छिपाया,
quarantine, alias किया जाना चाहिए या किसी और तरह बदला जाना चाहिए, तब
नेमस्पेस दावा खोलें।

उदाहरणों में शामिल हैं:

- ऐसा संगठन हैंडल जो आपके GitHub संगठन, परियोजना, कंपनी या समुदाय से मेल खाता हो
- `@example-org/*` जैसा पैकेज scope, जिसे केवल मेल खाने वाले ClawHub स्वामी के
  तहत प्रकाशित होना चाहिए
- कोई skill slug या Plugin पैकेज नाम जो किसी परियोजना का प्रतिरूपण करता हुआ लगे
- ब्रांड, ट्रेडमार्क, परियोजना नाम-परिवर्तन या पैकेज इतिहास विवाद
- हटाया गया, निष्क्रिय या अनुपलब्ध स्वामी जो सही नेमस्पेस स्वामी को रोकता है

यदि listing स्वामित्व विवाद से आगे असुरक्षित, दुर्भावनापूर्ण या भ्रामक है,
तो संबंधित moderation या security मार्गदर्शन का भी पालन करें। नेमस्पेस दावा
form स्वामित्व समीक्षा के लिए है, emergency vulnerability disclosure के लिए नहीं।

## फ़ाइल करने से पहले

पहले पुष्टि करें कि आप उस स्वामी के साथ प्रकाशित कर रहे हैं जो नेमस्पेस से
मेल खाता है। Plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे scoped
नामों को मेल खाने वाले `example-org` स्वामी के रूप में प्रकाशित किया जाना चाहिए।

यदि आप मौजूदा स्वामी को प्रबंधित कर सकते हैं, तो प्रभावित resource को प्रकाशित,
नाम बदलकर, हस्तांतरित, छिपाकर या हटाकर नेमस्पेस को सीधे ठीक करें। दावा तब
उपयोग करें जब आप मौजूदा स्वामी को प्रबंधित नहीं कर सकते या जब कर्मचारियों को
किसी विवाद को हल करना हो।

## शामिल करने के लिए साक्ष्य

सार्वजनिक, गैर-संवेदनशील साक्ष्य का उपयोग करें। उपयोगी प्रमाण में शामिल हैं:

- GitHub संगठन, repo, release या maintainer इतिहास
- आधिकारिक परियोजना docs जो नेमस्पेस का नाम बताते हैं
- domain या आधिकारिक email-domain प्रमाण
- npm, PyPI, crates.io या अन्य package-registry scope नियंत्रण
- ट्रेडमार्क, ब्रांड या परियोजना स्वामित्व साक्ष्य, जिस पर सार्वजनिक रूप से
  चर्चा करना सुरक्षित हो
- स्रोत repository इतिहास, पैकेज इतिहास या सार्वजनिक नाम-परिवर्तन सूचनाएं
- विवादित ClawHub स्वामी, skill, Plugin, पैकेज या issue के links

समझाएं कि हर link क्या साबित करता है। कर्मचारियों को निजी credentials या
secrets की आवश्यकता के बिना संबंध समझ में आना चाहिए।

## क्या शामिल न करें

किसी सार्वजनिक GitHub issue में secrets या निजी प्रमाण न डालें। शामिल न करें:

- API tokens, signing keys या credentials
- DNS challenge tokens
- निजी कानूनी फ़ाइलें या अनुबंध
- व्यक्तिगत पहचान दस्तावेज़
- निजी emails, निजी security reports या गोपनीय customer data

दावा form पूछता है कि संवेदनशील साक्ष्य के लिए निजी staff channel की आवश्यकता
है या नहीं। संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प
का उपयोग करें।

## संभावित परिणाम

साक्ष्य और जोखिम के आधार पर, ClawHub कर्मचारी किसी नेमस्पेस को आरक्षित कर सकते
हैं, स्वामित्व हस्तांतरित कर सकते हैं, किसी resource का नाम बदल सकते हैं, किसी
मौजूदा listing को छिपा या quarantine कर सकते हैं, alias या redirect जोड़ सकते
हैं, अधिक प्रमाण मांग सकते हैं या अनुरोध अस्वीकार कर सकते हैं।

नेमस्पेस समीक्षा यह गारंटी नहीं देती कि हर मेल खाता नाम हस्तांतरित किया जाएगा।
कर्मचारी सार्वजनिक साक्ष्य, मौजूदा उपयोग, security जोखिम और उपयोगकर्ता प्रभाव
को तौलते हैं।
