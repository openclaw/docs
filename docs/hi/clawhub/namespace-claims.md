# संगठन और नेमस्पेस दावे

ClawHub सार्वजनिक नेमस्पेस के रूप में स्वामी हैंडल, संगठन हैंडल, skill स्लग, Plugin पैकेज नाम और
पैकेज स्कोप का उपयोग करता है। यदि कोई नेमस्पेस किसी
वास्तविक परियोजना, ब्रांड, पैकेज पारिस्थितिकी तंत्र या संगठन से संबंधित प्रतीत होता है, लेकिन ClawHub पर पहले से
दावा किया गया, आरक्षित, भ्रामक या विवादित है, तो कर्मचारियों से
[संगठन / नेमस्पेस दावा समस्या फ़ॉर्म](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml) के माध्यम से उसकी समीक्षा करने के लिए कहें।

सार्वजनिक, गैर-संवेदनशील स्वामित्व समीक्षा के लिए इस माध्यम का उपयोग करें। नेमस्पेस दावों के लिए उत्पाद के भीतर की
रिपोर्ट या खाता अपील फ़ॉर्म का उपयोग न करें।

## दावा कब खोलें

जब आपको लगता है कि वास्तविक स्वामित्व के कारण किसी
नेमस्पेस को आरक्षित, हस्तांतरित, पुनर्नामित, छिपाया, क्वारंटीन, उपनामित
या अन्यथा बदला जाना चाहिए या नहीं, इसकी समीक्षा ClawHub कर्मचारियों को करनी चाहिए, तब नेमस्पेस दावा खोलें।

उदाहरणों में शामिल हैं:

- कोई संगठन हैंडल जो आपके GitHub संगठन, परियोजना, कंपनी या समुदाय से मेल खाता हो
- `@example-org/*` जैसा कोई पैकेज स्कोप, जिसे केवल मेल खाने वाले
  ClawHub स्वामी के अंतर्गत प्रकाशित होना चाहिए
- कोई skill स्लग या Plugin पैकेज नाम जो किसी परियोजना का प्रतिरूपण करता प्रतीत हो
- कोई ब्रांड, ट्रेडमार्क, परियोजना का नाम बदलने या पैकेज इतिहास संबंधी विवाद
- कोई हटाया गया, निष्क्रिय या पहुँच से बाहर स्वामी जो उचित नेमस्पेस
  स्वामी को अवरुद्ध करता हो

यदि सूची स्वामित्व विवाद के अतिरिक्त असुरक्षित, दुर्भावनापूर्ण या भ्रामक है,
तो संबंधित मॉडरेशन या सुरक्षा मार्गदर्शन का भी पालन करें। नेमस्पेस दावा
फ़ॉर्म स्वामित्व समीक्षा के लिए है, आपातकालीन भेद्यता प्रकटीकरण के लिए नहीं।

## दाखिल करने से पहले

पहले पुष्टि करें कि आप ऐसे स्वामी के साथ प्रकाशन कर रहे हैं जो नेमस्पेस से मेल खाता है।
Plugin पैकेजों के लिए, `@example-org/example-plugin` जैसे स्कोप वाले नाम
मेल खाने वाले `example-org` स्वामी के रूप में प्रकाशित किए जाने चाहिए।

यदि आप वर्तमान स्वामी को प्रबंधित कर सकते हैं, तो प्रभावित संसाधन को प्रकाशित,
पुनर्नामित, हस्तांतरित, छिपाकर या हटाकर नेमस्पेस को सीधे ठीक करें। जब आप वर्तमान स्वामी को
प्रबंधित नहीं कर सकते या कर्मचारियों को किसी विवाद का समाधान करना आवश्यक हो, तब दावे का
उपयोग करें।

## शामिल किए जाने वाले साक्ष्य

सार्वजनिक, गैर-संवेदनशील साक्ष्य का उपयोग करें। उपयोगी प्रमाणों में शामिल हैं:

- GitHub संगठन, रिपॉज़िटरी, रिलीज़ या अनुरक्षक इतिहास
- नेमस्पेस का नाम बताने वाले आधिकारिक परियोजना दस्तावेज़
- डोमेन या आधिकारिक ईमेल-डोमेन प्रमाण
- npm, PyPI, crates.io या अन्य पैकेज-रजिस्ट्री स्कोप का नियंत्रण
- ट्रेडमार्क, ब्रांड या परियोजना के स्वामित्व का ऐसा साक्ष्य जिस पर सार्वजनिक रूप से
  सुरक्षित ढंग से चर्चा की जा सके
- स्रोत रिपॉज़िटरी इतिहास, पैकेज इतिहास या नाम बदलने की सार्वजनिक सूचनाएँ
- विवादित ClawHub स्वामी, skill, Plugin, पैकेज या समस्या के लिंक

समझाएँ कि प्रत्येक लिंक क्या प्रमाणित करता है। कर्मचारियों को निजी क्रेडेंशियल या सीक्रेट की
आवश्यकता के बिना संबंध को समझने में सक्षम होना चाहिए।

## क्या शामिल न करें

किसी सार्वजनिक GitHub समस्या में सीक्रेट या निजी प्रमाण न डालें। निम्नलिखित शामिल न करें:

- API टोकन, हस्ताक्षर कुंजियाँ या क्रेडेंशियल
- DNS चुनौती टोकन
- निजी कानूनी फ़ाइलें या अनुबंध
- व्यक्तिगत पहचान दस्तावेज़
- निजी ईमेल, निजी सुरक्षा रिपोर्ट या गोपनीय ग्राहक डेटा

दावा फ़ॉर्म पूछता है कि क्या संवेदनशील साक्ष्य के लिए कर्मचारियों के निजी चैनल की आवश्यकता है।
संवेदनशील सामग्री को सार्वजनिक रूप से पोस्ट करने के बजाय उस विकल्प का उपयोग करें।

## संभावित परिणाम

साक्ष्य और जोखिम के आधार पर, ClawHub कर्मचारी किसी नेमस्पेस को आरक्षित कर सकते हैं,
स्वामित्व हस्तांतरित कर सकते हैं, किसी संसाधन का नाम बदल सकते हैं, किसी मौजूदा सूची को छिपा या क्वारंटीन कर सकते हैं,
कोई उपनाम या रीडायरेक्ट जोड़ सकते हैं, अधिक प्रमाण माँग सकते हैं या अनुरोध अस्वीकार कर सकते हैं।

नेमस्पेस समीक्षा इस बात की गारंटी नहीं देती कि प्रत्येक मेल खाने वाला नाम हस्तांतरित कर दिया जाएगा।
कर्मचारी सार्वजनिक साक्ष्य, मौजूदा उपयोग, सुरक्षा जोखिम और उपयोगकर्ता प्रभाव का मूल्यांकन करते हैं।
