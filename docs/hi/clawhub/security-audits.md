# सुरक्षा ऑडिट

ClawHub सुरक्षा ऑडिट आपको यह तय करने में मदद करते हैं कि कोई स्किल या Plugin
इंस्टॉल करने के लिए पर्याप्त सुरक्षित है या नहीं। वे दिखाते हैं कि कोई रिलीज़ क्या करती है, वह कौन-सी अधिकारिता मांगती है, और
क्या फ़ाइलों, खातों, क्रेडेंशियल, कोड, या बाहरी सेवाओं तक पहुंच पाने से पहले किसी चीज़ पर अतिरिक्त ध्यान देने की ज़रूरत है।

ऑडिट मज़बूत सुरक्षा संकेत हैं, लेकिन वे यह गारंटी नहीं देते कि कोई रिलीज़
जोखिम-मुक्त है। संवेदनशील पहुंच देने से पहले हमेशा अपने विवेक का उपयोग करें।

यह भी देखें [सुरक्षा](/clawhub/security), [स्वीकार्य उपयोग](/clawhub/acceptable-usage),
और [मॉडरेशन और खाता सुरक्षा](/clawhub/moderation)।

## इंस्टॉल करने से पहले क्या जांचें

इंस्टॉल करने से पहले, समीक्षा करें:

- समग्र ऑडिट स्थिति
- जोखिम स्तर
- सूचीबद्ध कोई भी निष्कर्ष
- आवश्यक क्रेडेंशियल, अनुमतियां, या पर्यावरण चर
- मालिक, स्रोत, संस्करण, चेंजलॉग, डाउनलोड, स्टार, और अन्य भरोसे के संकेत

केवल वही सामग्री इंस्टॉल करें जिसे आप समझते हैं और जिस पर भरोसा करते हैं।

## ऑडिट स्थिति

ऑडिट स्थिति बताती है कि ऑडिट परिणाम पर आपको कैसे प्रतिक्रिया देनी चाहिए:

| स्थिति      | अर्थ                                                                   |
| ----------- | ------------------------------------------------------------------------- |
| `Pass`      | कम जोखिम से ऊपर की कोई दिखाई देने वाली समस्या नहीं मिली।                                |
| `Review`    | इंस्टॉल करने से पहले निष्कर्ष पढ़ें। रिलीज़ फिर भी वैध हो सकती है। |
| `Warn`      | अतिरिक्त सावधानी बरतें। ClawHub को उच्च-प्रभाव वाली चिंता या चेतावनी संकेत मिला। |
| `Malicious` | इंस्टॉल न करें।                                                           |
| `Pending`   | ऑडिट अभी तक समाप्त नहीं हुए हैं।                                             |
| `Error`     | ऑडिट पूरा नहीं किया जा सका।                                         |

`Pass` आश्वस्त करने वाला है, लेकिन यह आपके अपने विवेक की जगह नहीं लेता। यह उन टूल के लिए
सबसे अधिक मायने रखता है जो सामग्री प्रकाशित कर सकते हैं, डेटा संपादित कर सकते हैं, कमांड चला सकते हैं, फ़ाइलें पढ़ सकते हैं, या
प्रोडक्शन सिस्टम तक पहुंच सकते हैं।

## जोखिम स्तर

जोखिम स्तर प्रभाव-क्षेत्र बताता है: यदि आप रिलीज़ का उपयोग अपेक्षित तरीके से करते हैं, तो उसके पास
कितनी शक्ति प्रतीत होती है।

| जोखिम स्तर | अर्थ                                                                       |
| ---------- | ----------------------------------------------------------------------------- |
| `Low`      | कम संवेदनशील अधिकारिता या उपयोगकर्ता प्रभाव मिला।                          |
| `Medium`   | रिलीज़ के पास सार्थक अधिकारिता है, जैसे खाता पहुंच या डेटा परिवर्तन। |
| `High`     | रिलीज़ के पास उच्च-प्रभाव वाली अधिकारिता, गंभीर निष्कर्ष, या दुर्भावनापूर्ण संकेत हैं। |

जोखिम स्तर और ऑडिट स्थिति अलग-अलग प्रश्नों का उत्तर देते हैं:

- जोखिम स्तर पूछता है: "यहां कितनी शक्ति है?"
- ऑडिट स्थिति पूछती है: "मुझे इस परिणाम के साथ क्या करना चाहिए?"

उदाहरण के लिए, कोई प्रकाशन स्किल `Medium` जोखिम के साथ `Review` दिखा सकती है। इसका
मतलब यह नहीं है कि वह दुर्भावनापूर्ण है। इसका मतलब है कि स्किल उद्देश्य के अनुरूप प्रतीत होती है, लेकिन
सार्थक खाता अधिकारिता के साथ कार्य कर सकती है।

## निष्कर्ष

निष्कर्ष बताते हैं कि कोई ऑडिट परिणाम क्यों दिखाया गया। प्रत्येक निष्कर्ष में आमतौर पर शामिल होता है:

- इसका क्या अर्थ है
- इसे क्यों चिह्नित किया गया
- संबंधित स्किल या Plugin सामग्री
- एक अनुशंसा

निष्कर्षों को `Info`, `Low`, `Medium`, `High`, या `Critical` लेबल दिया जा सकता है। अधिक
गंभीरता वाले निष्कर्ष जोखिम स्तर और ऑडिट स्थिति में अधिक मज़बूती से योगदान देते हैं।

कम-विश्वास वाले निष्कर्ष सार्वजनिक ऑडिट रोलअप से छिपाए जाते हैं ताकि पृष्ठ
उपयोगी साक्ष्य पर केंद्रित रहे।

## ClawHub क्या जांचता है

ClawHub सबमिट किए गए रिलीज़ आर्टिफैक्ट्स का ऑडिट करता है, जिनमें शामिल हैं:

- skill निर्देश या Plugin metadata
- घोषित environment variables और permissions
- install निर्देश और package metadata
- शामिल files और file manifests
- compatibility और capability metadata

मुख्य प्रश्न संगति का है: क्या नाम, सारांश, metadata, अनुरोधित
authority, और वास्तविक सामग्री उस चीज़ से मेल खाते हैं जिसकी उपयोगकर्ता उचित रूप से अपेक्षा करेंगे?

शक्तिशाली व्यवहार अपने आप बुरा नहीं होता। कई उपयोगी tools को credentials,
local commands, provider APIs, या package installs की आवश्यकता होती है। ऑडिट जांचता है कि क्या वह
शक्ति अपेक्षित, प्रकट, और अनुपातिक है।

Artifact pages पूर्ण ऑडिट से यहां लिंक करते हैं:

```text
/<owner>/skills/<slug>/security-audit
```

ऑडिट पृष्ठ इन्हें जोड़ता है:

1. SkillSpector
2. VirusTotal
3. जोखिम विश्लेषण

## VirusTotal

ClawHub ऑडिट स्टैक में VirusTotal को malware telemetry के रूप में उपयोग करता है। VirusTotal
file reputation और malware scanning के लिए एक भरोसेमंद उद्योग मानक है, और हमारी
partnership ClawHub को skill और Plugin review में व्यापक security intelligence जोड़ने देती है।

VirusTotal ज्ञात malicious artifacts, engine hits, और
reputation signals के लिए विशेष रूप से उपयोगी है, जो ClawHub की agent-aware review को पूरक करते हैं। जब vendor
engine counts उपलब्ध होते हैं, तो ऑडिट उन्हें सरल भाषा में सारांशित करता है, जैसे:

```text
62/62 vendors flagged this skill as clean.
```

या:

```text
2/64 vendors flagged this skill as malicious, 1/64 flagged it as suspicious, and 61/64 flagged it as clean.
```

जब ClawHub के पास सारांशित करने के लिए कोई vendor-count telemetry नहीं होती, तो ऑडिट कहता है:

```text
No VirusTotal findings
```

VirusTotal telemetry ही रहता है। यह ClawHub के अपने artifact-aware
जोखिम विश्लेषण की जगह नहीं लेता।

## जोखिम विश्लेषण

जोखिम विश्लेषण आंतरिक रूप से ClawScan, ClawHub की अपनी security audit
system, द्वारा संचालित होता है। यह हर रिलीज़ को agent-facing artifact के रूप में review करता है: instructions,
metadata, declared permissions, files, capability signals, static scan signals,
SkillSpector findings, VirusTotal telemetry, और publisher-provided context।
Static scan signals इस review के लिए आंतरिक context हैं; वे कोई
standalone public audit section या install-blocking verdict नहीं हैं।

जोखिम विश्लेषण
[OWASP Agentic Skills Top 10](https://owasp.org/www-project-agentic-skills-top-10/)
का उपयोग prompt injection, tool misuse, credential exposure,
unsafe execution, memory या context poisoning, और excessive agency जैसे जोखिमों के लिए एक lens के रूप में करता है।

ClawScan किसी डरावनी दिखने वाली capability को अपने आप malicious नहीं मानता।
यह पूछता है कि क्या capability disclosed, purpose-aligned, और
release के stated use case द्वारा समर्थित है।
