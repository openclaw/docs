# सुरक्षा ऑडिट

ClawHub सुरक्षा ऑडिट आपको यह तय करने में मदद करते हैं कि कोई skill या Plugin इंस्टॉल करने के लिए पर्याप्त सुरक्षित है या नहीं। वे दिखाते हैं कि कोई रिलीज़ क्या करती है, वह कौन-से अधिकार मांगती है, और क्या फ़ाइलों, खातों, क्रेडेंशियल्स, कोड, या बाहरी सेवाओं तक पहुँचने से पहले किसी चीज़ पर अतिरिक्त ध्यान देने की ज़रूरत है।

ऑडिट मज़बूत सुरक्षा संकेत होते हैं, लेकिन वे यह गारंटी नहीं देते कि कोई रिलीज़ जोखिम-मुक्त है। संवेदनशील पहुँच देने से पहले हमेशा अपना विवेक इस्तेमाल करें।

यह भी देखें [सुरक्षा](/clawhub/security), [स्वीकार्य उपयोग](/hi/clawhub/acceptable-usage),
और [मॉडरेशन और खाता सुरक्षा](/clawhub/moderation).

## इंस्टॉल करने से पहले क्या जांचें

इंस्टॉल करने से पहले, समीक्षा करें:

- समग्र ऑडिट स्थिति
- जोखिम स्तर
- सूचीबद्ध कोई भी निष्कर्ष
- आवश्यक क्रेडेंशियल्स, अनुमतियाँ, या पर्यावरण वेरिएबल्स
- मालिक, स्रोत, संस्करण, changelog, डाउनलोड, stars, और अन्य भरोसे के संकेत

केवल वही सामग्री इंस्टॉल करें जिसे आप समझते और भरोसा करते हैं।

## ऑडिट स्थिति

ऑडिट स्थिति बताती है कि आपको ऑडिट परिणाम पर कैसे प्रतिक्रिया देनी चाहिए:

| स्थिति      | अर्थ                                                                   |
| ----------- | ------------------------------------------------------------------------- |
| `Pass`      | कम जोखिम से ऊपर कोई दिखाई देने वाली समस्या नहीं मिली।                                |
| `Review`    | इंस्टॉल करने से पहले निष्कर्ष पढ़ें। रिलीज़ फिर भी वैध हो सकती है। |
| `Warn`      | अतिरिक्त सावधानी बरतें। ClawHub को कोई उच्च-प्रभाव वाली चिंता या चेतावनी संकेत मिला। |
| `Malicious` | इंस्टॉल न करें।                                                           |
| `Pending`   | ऑडिट अभी पूरा नहीं हुआ है।                                             |
| `Error`     | ऑडिट पूरा नहीं किया जा सका।                                         |

`Pass` आश्वस्त करने वाला है, लेकिन यह आपके अपने विवेक की जगह नहीं लेता। यह उन टूल्स के लिए सबसे ज़्यादा मायने रखता है जो सामग्री प्रकाशित कर सकते हैं, डेटा संपादित कर सकते हैं, कमांड चला सकते हैं, फ़ाइलें पढ़ सकते हैं, या उत्पादन प्रणालियों तक पहुँच सकते हैं।

## जोखिम स्तर

जोखिम स्तर blast radius का वर्णन करता है: रिलीज़ को इच्छित तरीके से उपयोग करने पर उसके पास कितनी शक्ति दिखाई देती है।

| जोखिम स्तर | अर्थ                                                                       |
| ---------- | ----------------------------------------------------------------------------- |
| `Low`      | बहुत कम संवेदनशील अधिकार या उपयोगकर्ता प्रभाव मिला।                          |
| `Medium`   | रिलीज़ के पास अर्थपूर्ण अधिकार हैं, जैसे खाता पहुँच या डेटा बदलाव। |
| `High`     | रिलीज़ के पास उच्च-प्रभाव वाले अधिकार, गंभीर निष्कर्ष, या malicious संकेत हैं। |

जोखिम स्तर और ऑडिट स्थिति अलग-अलग सवालों का जवाब देते हैं:

- जोखिम स्तर पूछता है: "यहाँ कितनी शक्ति है?"
- ऑडिट स्थिति पूछती है: "मुझे इस परिणाम के साथ क्या करना चाहिए?"

उदाहरण के लिए, कोई प्रकाशन skill `Medium` जोखिम के साथ `Review` दिखा सकती है। इसका मतलब यह नहीं है कि वह malicious है। इसका मतलब है कि skill उद्देश्य के अनुरूप दिखती है, लेकिन अर्थपूर्ण खाता अधिकार के साथ कार्रवाई कर सकती है।

## निष्कर्ष

निष्कर्ष बताते हैं कि कोई ऑडिट परिणाम क्यों दिखाया गया। हर निष्कर्ष में आमतौर पर ये शामिल होते हैं:

- इसका क्या अर्थ है
- इसे क्यों फ़्लैग किया गया
- संबंधित skill या Plugin सामग्री
- एक अनुशंसा

निष्कर्षों को `Info`, `Low`, `Medium`, `High`, या `Critical` लेबल किया जा सकता है। अधिक गंभीरता वाले निष्कर्ष जोखिम स्तर और ऑडिट स्थिति में अधिक मज़बूती से योगदान देते हैं।

कम-विश्वास वाले निष्कर्ष सार्वजनिक ऑडिट रोलअप से छिपाए जाते हैं ताकि पेज उपयोगी प्रमाण पर केंद्रित रहे।

## ClawHub क्या जांचता है

ClawHub सबमिट किए गए रिलीज़ आर्टिफैक्ट्स का ऑडिट करता है, जिनमें शामिल हैं:

- skill निर्देश या Plugin मेटाडेटा
- घोषित पर्यावरण वेरिएबल्स और अनुमतियाँ
- इंस्टॉल निर्देश और पैकेज मेटाडेटा
- शामिल फ़ाइलें और फ़ाइल manifests
- संगतता और क्षमता मेटाडेटा

मुख्य सवाल coherence है: क्या नाम, सारांश, मेटाडेटा, मांगे गए अधिकार, और वास्तविक सामग्री उस चीज़ के साथ मेल खाते हैं जिसकी उपयोगकर्ता उचित रूप से अपेक्षा करेंगे?

शक्तिशाली व्यवहार अपने-आप बुरा नहीं होता। कई उपयोगी टूल्स को क्रेडेंशियल्स, स्थानीय कमांड, provider APIs, या पैकेज इंस्टॉल की ज़रूरत होती है। ऑडिट जांचता है कि वह शक्ति अपेक्षित, घोषित, और अनुपातिक है या नहीं।

आर्टिफैक्ट पेज पूर्ण ऑडिट से यहाँ लिंक करते हैं:

```text
/<owner>/skills/<slug>/security-audit
```

ऑडिट पेज इनका संयोजन करता है:

1. SkillSpector
2. VirusTotal
3. जोखिम विश्लेषण

## VirusTotal

ClawHub ऑडिट stack में VirusTotal को malware telemetry के रूप में उपयोग करता है। VirusTotal फ़ाइल प्रतिष्ठा और malware scanning के लिए एक भरोसेमंद उद्योग मानक है, और हमारी साझेदारी ClawHub को skill और Plugin समीक्षा में व्यापक सुरक्षा intelligence जोड़ने देती है।

VirusTotal खास तौर पर ज्ञात malicious आर्टिफैक्ट्स, engine hits, और reputation signals के लिए उपयोगी है, जो ClawHub की agent-aware समीक्षा को पूरक करते हैं। जब vendor engine counts उपलब्ध होते हैं, तो ऑडिट उन्हें सरल भाषा में सारांशित करता है, जैसे:

```text
62/62 vendors flagged this skill as clean.
```

या:

```text
2/64 vendors flagged this skill as malicious, 1/64 flagged it as suspicious, and 61/64 flagged it as clean.
```

जब ClawHub के पास सारांशित करने के लिए कोई vendor-count telemetry नहीं होती, तो ऑडिट कहता है:

```text
No VirusTotal findings
```

VirusTotal telemetry ही रहता है। यह ClawHub के अपने आर्टिफैक्ट-aware जोखिम विश्लेषण की जगह नहीं लेता।

## जोखिम विश्लेषण

जोखिम विश्लेषण आंतरिक रूप से ClawScan, ClawHub की अपनी सुरक्षा ऑडिट प्रणाली, द्वारा संचालित होता है। यह हर रिलीज़ को agent-facing आर्टिफैक्ट के रूप में समीक्षा करता है: निर्देश, मेटाडेटा, घोषित अनुमतियाँ, फ़ाइलें, क्षमता संकेत, static scan signals, SkillSpector निष्कर्ष, VirusTotal telemetry, और publisher-provided context। Static scan signals इस समीक्षा के लिए आंतरिक संदर्भ हैं; वे कोई standalone सार्वजनिक ऑडिट अनुभाग या install-blocking verdict नहीं हैं।

जोखिम विश्लेषण
[OWASP Agentic Skills Top 10](https://owasp.org/www-project-agentic-skills-top-10/)
को prompt injection, tool misuse, credential exposure, unsafe execution, memory या context poisoning, और excessive agency जैसे जोखिमों के लिए एक lens के रूप में उपयोग करता है।

ClawScan किसी डरावनी दिखने वाली क्षमता को अपने-आप malicious नहीं मानता। यह पूछता है कि क्या क्षमता घोषित है, उद्देश्य के अनुरूप है, और रिलीज़ के बताए गए use case द्वारा समर्थित है।
