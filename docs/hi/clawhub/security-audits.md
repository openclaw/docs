# सुरक्षा ऑडिट

ClawHub सुरक्षा ऑडिट आपको यह तय करने में मदद करते हैं कि कोई कौशल या Plugin इंस्टॉल करने के लिए पर्याप्त सुरक्षित है या नहीं। वे दिखाते हैं कि कोई रिलीज़ क्या करती है, वह किस अधिकार की मांग करती है, और क्या फ़ाइलों, खातों, क्रेडेंशियल्स, कोड, या बाहरी सेवाओं तक पहुंच पाने से पहले किसी चीज़ पर अतिरिक्त ध्यान देने की ज़रूरत है।

ऑडिट मजबूत सुरक्षा संकेत हैं, लेकिन वे यह गारंटी नहीं देते कि कोई रिलीज़ जोखिम-मुक्त है। संवेदनशील पहुंच देने से पहले हमेशा विवेक का उपयोग करें।

यह भी देखें [सुरक्षा](/clawhub/security), [स्वीकार्य उपयोग](/clawhub/acceptable-usage),
और [मॉडरेशन और खाता सुरक्षा](/clawhub/moderation).

## इंस्टॉल करने से पहले क्या जांचें

इंस्टॉल करने से पहले, समीक्षा करें:

- समग्र ऑडिट स्थिति
- जोखिम स्तर
- सूचीबद्ध कोई भी निष्कर्ष
- आवश्यक क्रेडेंशियल्स, अनुमतियां, या environment variables
- स्वामी, स्रोत, संस्करण, changelog, डाउनलोड, stars, और अन्य भरोसे के संकेत

केवल वही सामग्री इंस्टॉल करें जिसे आप समझते और भरोसा करते हैं।

## ऑडिट स्थिति

ऑडिट स्थिति बताती है कि आपको ऑडिट परिणाम पर कैसे प्रतिक्रिया देनी चाहिए:

| स्थिति      | अर्थ                                                                   |
| ----------- | ------------------------------------------------------------------------- |
| `Pass`      | कम जोखिम से ऊपर कोई दिखाई देने वाली समस्या नहीं मिली।                                |
| `Review`    | इंस्टॉल करने से पहले निष्कर्ष पढ़ें। रिलीज़ फिर भी वैध हो सकती है। |
| `Warn`      | अतिरिक्त सावधानी बरतें। ClawHub को उच्च-प्रभाव वाली चिंता या चेतावनी संकेत मिला। |
| `Malicious` | इंस्टॉल न करें।                                                           |
| `Pending`   | ऑडिट अभी पूरे नहीं हुए हैं।                                             |
| `Error`     | ऑडिट पूरा नहीं किया जा सका।                                         |

`Pass` आश्वस्त करने वाला है, लेकिन यह आपके अपने विवेक की जगह नहीं लेता। यह उन टूल्स के लिए सबसे अधिक मायने रखता है जो सामग्री प्रकाशित कर सकते हैं, डेटा संपादित कर सकते हैं, commands चला सकते हैं, फ़ाइलें पढ़ सकते हैं, या production systems तक पहुंच सकते हैं।

## जोखिम स्तर

जोखिम स्तर blast radius का वर्णन करता है: यदि आप रिलीज़ का उपयोग उसके इच्छित तरीके से करते हैं, तो उसके पास कितनी शक्ति प्रतीत होती है।

| जोखिम स्तर | अर्थ                                                                       |
| ---------- | ----------------------------------------------------------------------------- |
| `Low`      | बहुत कम संवेदनशील अधिकार या उपयोगकर्ता प्रभाव मिला।                          |
| `Medium`   | रिलीज़ के पास महत्वपूर्ण अधिकार हैं, जैसे खाते तक पहुंच या डेटा परिवर्तन। |
| `High`     | रिलीज़ के पास उच्च-प्रभाव वाले अधिकार, गंभीर निष्कर्ष, या दुर्भावनापूर्ण संकेत हैं। |

जोखिम स्तर और ऑडिट स्थिति अलग-अलग प्रश्नों का उत्तर देते हैं:

- जोखिम स्तर पूछता है: "यहां कितनी शक्ति है?"
- ऑडिट स्थिति पूछती है: "मुझे इस परिणाम के साथ क्या करना चाहिए?"

उदाहरण के लिए, कोई publishing कौशल `Medium` जोखिम के साथ `Review` दिखा सकता है। इसका मतलब यह नहीं है कि वह दुर्भावनापूर्ण है। इसका मतलब है कि कौशल अपने उद्देश्य के अनुरूप प्रतीत होता है, लेकिन महत्वपूर्ण खाता अधिकार के साथ कार्य कर सकता है।

## निष्कर्ष

निष्कर्ष बताते हैं कि कोई ऑडिट परिणाम क्यों दिखाया गया। हर निष्कर्ष में आम तौर पर शामिल होता है:

- इसका क्या अर्थ है
- इसे क्यों चिह्नित किया गया
- संबंधित कौशल या Plugin सामग्री
- एक सिफारिश

निष्कर्षों को `Info`, `Low`, `Medium`, `High`, या `Critical` लेबल किया जा सकता है। अधिक गंभीर निष्कर्ष जोखिम स्तर और ऑडिट स्थिति में अधिक मजबूत योगदान देते हैं।

कम-विश्वास वाले निष्कर्ष public audit rollup से छिपाए जाते हैं ताकि पेज उपयोगी साक्ष्य पर केंद्रित रहे।

## ClawHub क्या जांचता है

ClawHub सबमिट किए गए रिलीज़ artifacts का ऑडिट करता है, जिनमें शामिल हैं:

- कौशल निर्देश या Plugin metadata
- घोषित environment variables और अनुमतियां
- इंस्टॉल निर्देश और package metadata
- शामिल फ़ाइलें और file manifests
- compatibility और capability metadata

मुख्य प्रश्न coherence है: क्या नाम, सारांश, metadata, मांगा गया अधिकार, और वास्तविक सामग्री उस चीज़ से मेल खाते हैं जिसकी उपयोगकर्ता उचित रूप से अपेक्षा करेंगे?

शक्तिशाली व्यवहार अपने आप बुरा नहीं होता। कई उपयोगी टूल्स को credentials, local commands, provider APIs, या package installs की आवश्यकता होती है। ऑडिट जांचता है कि क्या वह शक्ति अपेक्षित, प्रकट, और अनुपातिक है।

Artifact pages पूर्ण ऑडिट से यहां लिंक करते हैं:

```text
/<owner>/skills/<slug>/security-audit
```

ऑडिट पेज मिलाता है:

1. SkillSpector
2. VirusTotal
3. जोखिम विश्लेषण

## VirusTotal

ClawHub ऑडिट stack में malware telemetry के रूप में VirusTotal का उपयोग करता है। VirusTotal file reputation और malware scanning के लिए एक भरोसेमंद industry standard है, और हमारी partnership ClawHub को कौशल और Plugin समीक्षा में व्यापक सुरक्षा intelligence जोड़ने देती है।

VirusTotal ज्ञात दुर्भावनापूर्ण artifacts, engine hits, और reputation signals के लिए विशेष रूप से उपयोगी है, जो ClawHub की agent-aware समीक्षा को पूरक करते हैं। जब vendor engine counts उपलब्ध होते हैं, तो ऑडिट उन्हें सरल भाषा में संक्षेपित करता है, जैसे:

```text
62/62 vendors flagged this skill as clean.
```

या:

```text
2/64 vendors flagged this skill as malicious, 1/64 flagged it as suspicious, and 61/64 flagged it as clean.
```

जब ClawHub के पास संक्षेपित करने के लिए कोई vendor-count telemetry नहीं होती, तो ऑडिट कहता है:

```text
No VirusTotal findings
```

VirusTotal telemetry ही रहता है। यह ClawHub के अपने artifact-aware जोखिम विश्लेषण की जगह नहीं लेता।

## जोखिम विश्लेषण

जोखिम विश्लेषण आंतरिक रूप से ClawScan द्वारा संचालित है, जो ClawHub की अपनी सुरक्षा ऑडिट प्रणाली है। यह हर रिलीज़ की agent-facing artifact के रूप में समीक्षा करता है: निर्देश, metadata, घोषित अनुमतियां, फ़ाइलें, capability signals, static scan signals, SkillSpector findings, VirusTotal telemetry, और publisher-provided context। Static scan signals इस समीक्षा के लिए आंतरिक context हैं; वे कोई standalone public audit section या install-blocking verdict नहीं हैं।

जोखिम विश्लेषण
[OWASP Agentic Skills Top 10](https://owasp.org/www-project-agentic-skills-top-10/)
का उपयोग prompt injection, tool misuse, credential exposure, unsafe execution, memory या context poisoning, और excessive agency जैसे जोखिमों को देखने के लिए एक lens के रूप में करता है।

ClawScan डरावनी दिखने वाली capability को अपने आप दुर्भावनापूर्ण नहीं मानता। यह पूछता है कि क्या capability प्रकट की गई है, उद्देश्य के अनुरूप है, और रिलीज़ के बताए गए use case द्वारा समर्थित है।
