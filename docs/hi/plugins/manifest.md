यह पृष्ठ **मूल OpenClaw Plugin मैनिफ़ेस्ट**, `openclaw.plugin.json`, के बारे में बताता है। संगत बंडल लेआउट (Codex, Claude, Cursor) के लिए, [Plugin बंडल](/hi/plugins/bundles) देखें।

संगत बंडल प्रारूप इसके बजाय अपनी मैनिफ़ेस्ट फ़ाइलों का उपयोग करते हैं:

- Codex बंडल: `.codex-plugin/plugin.json`
- Claude बंडल: `.claude-plugin/plugin.json`, या बिना मैनिफ़ेस्ट वाला डिफ़ॉल्ट Claude घटक लेआउट
- Cursor बंडल: `.cursor-plugin/plugin.json`

OpenClaw उन लेआउट का स्वतः पता लगाता है, लेकिन नीचे दिए गए `openclaw.plugin.json` स्कीमा के अनुसार उन्हें सत्यापित नहीं करता। किसी संगत बंडल के लिए, जब लेआउट OpenClaw की रनटाइम अपेक्षाओं से मेल खाता है, तब OpenClaw बंडल मेटाडेटा, घोषित स्किल रूट, Claude कमांड रूट, Claude `settings.json` डिफ़ॉल्ट, Claude LSP डिफ़ॉल्ट और समर्थित हुक पैक पढ़ता है।

प्रत्येक मूल OpenClaw Plugin के **Plugin रूट** में `openclaw.plugin.json` **अवश्य** शामिल होना चाहिए। OpenClaw इसका उपयोग **Plugin कोड निष्पादित किए बिना** कॉन्फ़िगरेशन सत्यापित करने के लिए करता है। अनुपस्थित या अमान्य मैनिफ़ेस्ट कॉन्फ़िगरेशन सत्यापन को रोक देता है और इसे Plugin त्रुटि माना जाता है।

संपूर्ण Plugin सिस्टम मार्गदर्शिका के लिए [Plugins](/hi/tools/plugin) और मूल क्षमता मॉडल तथा वर्तमान बाहरी-संगतता मार्गदर्शन के लिए [क्षमता मॉडल](/hi/plugins/architecture#public-capability-model) देखें।

## यह फ़ाइल क्या करती है

`openclaw.plugin.json` वह मेटाडेटा है जिसे OpenClaw **आपका Plugin कोड लोड करने से पहले** पढ़ता है। इसमें मौजूद हर चीज़ इतनी कम खर्चीली होनी चाहिए कि Plugin रनटाइम प्रारंभ किए बिना उसका निरीक्षण किया जा सके।

**इसका उपयोग इनके लिए करें:**

- Plugin पहचान, कॉन्फ़िगरेशन सत्यापन और कॉन्फ़िगरेशन UI संकेत
- प्रमाणीकरण, ऑनबोर्डिंग और सेटअप मेटाडेटा (उपनाम, स्वतः सक्षम करना, प्रदाता एनवायरनमेंट वेरिएबल और प्रमाणीकरण विकल्प)
- कंट्रोल-प्लेन सतहों के लिए सक्रियण संकेत
- संक्षिप्त मॉडल-फ़ैमिली स्वामित्व
- स्थिर क्षमता-स्वामित्व स्नैपशॉट (`contracts`)
- QA रनर मेटाडेटा, जिसका साझा `openclaw qa` होस्ट निरीक्षण कर सकता है
- कैटलॉग और सत्यापन सतहों में मर्ज किया गया चैनल-विशिष्ट कॉन्फ़िगरेशन मेटाडेटा

**इसका उपयोग इनके लिए न करें:** रनटाइम व्यवहार पंजीकृत करना, कोड एंट्रीपॉइंट घोषित करना या npm इंस्टॉल मेटाडेटा। ये आपके Plugin कोड और `package.json` में होने चाहिए।

| `contracts`                          | नहीं       | `object`                     | बाहरी प्रमाणीकरण हुक, एम्बेडिंग, वाक्, रीयलटाइम ट्रांसक्रिप्शन, रीयलटाइम वॉइस, मीडिया-समझ, छवि/वीडियो/संगीत जनरेशन, वेब फ़ेच, वेब खोज, वर्कर प्रदाताओं, दस्तावेज़/वेब-सामग्री निष्कर्षण और टूल स्वामित्व के लिए स्थिर क्षमता-स्वामित्व स्नैपशॉट। |
| `configContracts`                    | नहीं       | `object`                     | मैनिफ़ेस्ट-स्वामित्व वाला कॉन्फ़िग व्यवहार, जिसका उपयोग सामान्य कोर हेल्पर करते हैं: खतरनाक फ़्लैग की पहचान, SecretRef माइग्रेशन लक्ष्य और लेगेसी कॉन्फ़िग-पथ का संकुचन। [configContracts संदर्भ](#configcontracts-reference) देखें।                                                     |
| `mediaUnderstandingProviderMetadata` | नहीं       | `Record<string, object>`     | `contracts.mediaUnderstandingProviders` में घोषित प्रोवाइडर आईडी के लिए कम लागत वाली मीडिया-समझ डिफ़ॉल्ट सेटिंग।                                                                                                                                                                   |
| `imageGenerationProviderMetadata`    | नहीं       | `Record<string, object>`     | `contracts.imageGenerationProviders` में घोषित प्रोवाइडर आईडी के लिए कम लागत वाला इमेज-जनरेशन प्रमाणीकरण मेटाडेटा, जिसमें प्रोवाइडर-स्वामित्व वाले प्रमाणीकरण उपनाम और बेस-URL गार्ड शामिल हैं।                                                                                                         |
| `videoGenerationProviderMetadata`    | नहीं       | `Record<string, object>`     | `contracts.videoGenerationProviders` में घोषित प्रोवाइडर आईडी के लिए कम लागत वाला वीडियो-जनरेशन प्रमाणीकरण मेटाडेटा, जिसमें प्रोवाइडर-स्वामित्व वाले प्रमाणीकरण उपनाम और बेस-URL गार्ड शामिल हैं।                                                                                                         |
| `musicGenerationProviderMetadata`    | नहीं       | `Record<string, object>`     | `contracts.musicGenerationProviders` में घोषित प्रोवाइडर आईडी के लिए कम लागत वाला संगीत-जनरेशन प्रमाणीकरण मेटाडेटा, जिसमें प्रोवाइडर-स्वामित्व वाले प्रमाणीकरण उपनाम और बेस-URL गार्ड शामिल हैं।                                                                                                         |

| `version`                            | नहीं       | `string`                     | जानकारीपरक Plugin संस्करण।                                                                                                                                                                                                                                              |
| `uiHints`                            | नहीं       | `Record<string, object>`     | कॉन्फ़िग फ़ील्ड के लिए UI लेबल, प्लेसहोल्डर और संवेदनशीलता संकेत।                                                                                                                                                                                                          |

## कैटलॉग संदर्भ

`catalog` Plugin ब्राउज़र के लिए वैकल्पिक प्रदर्शन संकेत प्रदान करता है। होस्ट इन संकेतों को अनदेखा कर सकते हैं। ये कभी भी Plugin को इंस्टॉल या सक्षम नहीं करते, और न ही उसके रनटाइम व्यवहार या विश्वास स्तर को बदलते हैं।

```json
{
  "catalog": {
    "featured": true,
    "order": 10
  }
}
```

| फ़ील्ड      | प्रकार      | इसका अर्थ                                                              |
| ---------- | --------- | -------------------------------------------------------------------------- |
| `featured` | `boolean` | क्या कैटलॉग सतहों पर इस Plugin को प्रमुखता से दिखाया जाना चाहिए।                       |
| `order`    | `number`  | चुने गए Plugins के बीच आरोही प्रदर्शन संकेत; कम मान पहले दिखाई देते हैं। |

## जनरेशन प्रदाता मेटाडेटा संदर्भ

जनरेशन प्रदाता मेटाडेटा फ़ील्ड मेल खाने वाली `contracts.*GenerationProviders` सूची में घोषित प्रदाताओं के लिए स्थिर प्रमाणीकरण संकेतों का वर्णन करते हैं। OpenClaw इन फ़ील्ड को प्रदाता रनटाइम लोड होने से पहले पढ़ता है, ताकि मुख्य टूल हर प्रदाता Plugin को आयात किए बिना तय कर सकें कि कोई जनरेशन प्रदाता उपलब्ध है या नहीं।

इन फ़ील्ड का उपयोग केवल कम लागत वाले, घोषणात्मक तथ्यों के लिए करें। ट्रांसपोर्ट, अनुरोध रूपांतरण, टोकन रीफ़्रेश, क्रेडेंशियल सत्यापन और वास्तविक जनरेशन व्यवहार Plugin रनटाइम में ही रहते हैं।

```json
{
  "contracts": {
    "imageGenerationProviders": ["example-image"]
  },
  "imageGenerationProviderMetadata": {
    "example-image": {
      "aliases": ["example-image-oauth"],
      "authProviders": ["example-image"],
      "configSignals": [
        {
          "rootPath": "plugins.entries.example-image.config",
          "overlayPath": "image",
          "mode": {
            "path": "mode",
            "default": "local",
            "allowed": ["local"]
          },
          "requiredAny": ["workflow", "workflowPath"],
          "required": ["promptNodeId"]
        }
      ],
      "authSignals": [
        {
          "provider": "example-image"
        },
        {
          "provider": "example-image-oauth",
          "providerBaseUrl": {
            "provider": "example-image",
            "defaultBaseUrl": "https://api.example.com/v1",
            "allowedBaseUrls": ["https://api.example.com/v1"]
          }
        }
      ]
    }
  }
}
```

प्रत्येक मेटाडेटा प्रविष्टि निम्न का समर्थन करती है:

| फ़ील्ड                  | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                       |
| ---------------------- | -------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `aliases`              | नहीं       | `string[]` | अतिरिक्त प्रदाता आईडी, जिन्हें जनरेशन प्रदाता के लिए स्थिर प्रमाणीकरण उपनाम माना जाना चाहिए।                                                       |
| `authProviders`        | नहीं       | `string[]` | वे प्रदाता आईडी जिनकी कॉन्फ़िगर की गई प्रमाणीकरण प्रोफ़ाइल को इस जनरेशन प्रदाता का प्रमाणीकरण माना जाना चाहिए।                                                      |
| `configSignals`        | नहीं       | `object[]` | स्थानीय या स्वयं होस्ट किए गए प्रदाताओं के लिए कम लागत वाले केवल-कॉन्फ़िग उपलब्धता संकेत, जिन्हें प्रमाणीकरण प्रोफ़ाइल या पर्यावरण चर के बिना कॉन्फ़िगर किया जा सकता है।                 |
| `authSignals`          | नहीं       | `object[]` | स्पष्ट प्रमाणीकरण संकेत। मौजूद होने पर, ये प्रदाता आईडी, `aliases` और `authProviders` से प्राप्त डिफ़ॉल्ट संकेत समूह को प्रतिस्थापित करते हैं।                     |
| `referenceAudioInputs` | नहीं       | `boolean`  | केवल वीडियो जनरेशन। जब प्रदाता संदर्भ ऑडियो एसेट स्वीकार करता हो, तब `true` पर सेट करें; अन्यथा `video_generate` ऑडियो संदर्भ पैरामीटर छिपा देता है। |

प्रत्येक `configSignals` प्रविष्टि निम्न का समर्थन करती है:

| फ़ील्ड            | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                                                             |
| ---------------- | -------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `rootPath`       | हाँ      | `string`   | निरीक्षण किए जाने वाले Plugin-स्वामित्व वाले कॉन्फ़िग ऑब्जेक्ट का डॉट पथ, उदाहरण के लिए `plugins.entries.example.config`।                                                                                      |
| `overlayPath`    | नहीं       | `string`   | रूट कॉन्फ़िग के भीतर वह डॉट पथ, जिसका ऑब्जेक्ट संकेत का मूल्यांकन करने से पहले रूट ऑब्जेक्ट पर ओवरले होना चाहिए। इसका उपयोग `image`, `video` या `music` जैसे क्षमता-विशिष्ट कॉन्फ़िग के लिए करें।   |
| `overlayMapPath` | नहीं       | `string`   | रूट कॉन्फ़िग के भीतर वह डॉट पथ, जिसके प्रत्येक ऑब्जेक्ट मान को रूट ऑब्जेक्ट पर ओवरले होना चाहिए। इसका उपयोग `accounts` जैसे नामित खाता मैप के लिए करें, जहाँ कोई भी कॉन्फ़िगर किया गया खाता पात्र होना चाहिए। |
| `required`       | नहीं       | `string[]` | प्रभावी कॉन्फ़िग के भीतर वे डॉट पथ जिनमें कॉन्फ़िगर किए गए मान होना आवश्यक है। स्ट्रिंग खाली नहीं होनी चाहिए; ऑब्जेक्ट और सरणियाँ खाली नहीं होनी चाहिए।                                                  |
| `requiredAny`    | नहीं       | `string[]` | प्रभावी कॉन्फ़िग के भीतर वे डॉट पथ जिनमें से कम-से-कम एक में कॉन्फ़िगर किया गया मान होना आवश्यक है।                                                                                                    |
| `mode`           | नहीं       | `object`   | प्रभावी कॉन्फ़िग के भीतर वैकल्पिक स्ट्रिंग मोड गार्ड। इसका उपयोग तब करें जब केवल-कॉन्फ़िग उपलब्धता केवल एक मोड पर लागू होती हो।                                                                  |

प्रत्येक `mode` गार्ड निम्न का समर्थन करता है:

| फ़ील्ड        | आवश्यक | प्रकार       | इसका अर्थ                                                                      |
| ------------ | -------- | ---------- | ---------------------------------------------------------------------------------- |
| `path`       | नहीं       | `string`   | प्रभावी कॉन्फ़िग के भीतर डॉट पथ। डिफ़ॉल्ट `mode` है।                          |
| `default`    | नहीं       | `string`   | कॉन्फ़िग द्वारा पथ छोड़े जाने पर उपयोग किया जाने वाला मोड मान।                                  |
| `allowed`    | नहीं       | `string[]` | मौजूद होने पर, संकेत केवल तभी सफल होता है जब प्रभावी मोड इनमें से कोई एक मान हो। |
| `disallowed` | नहीं       | `string[]` | मौजूद होने पर, प्रभावी मोड के इनमें से कोई एक मान होने पर संकेत विफल हो जाता है।       |

प्रत्येक `authSignals` प्रविष्टि निम्न का समर्थन करती है:

| फ़ील्ड             | आवश्यक | प्रकार     | इसका अर्थ                                                                                                                                                                 |
| ----------------- | -------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `provider`        | हाँ      | `string` | कॉन्फ़िगर की गई प्रमाणीकरण प्रोफ़ाइल में जाँचने के लिए प्रदाता आईडी।                                                                                                                             |
| `providerBaseUrl` | नहीं       | `object` | वैकल्पिक गार्ड, जो संकेत को केवल तभी मान्य बनाता है जब संदर्भित कॉन्फ़िगर किया गया प्रदाता अनुमत बेस URL का उपयोग करता हो। इसका उपयोग तब करें जब कोई प्रमाणीकरण उपनाम केवल कुछ API के लिए मान्य हो। |

प्रत्येक `providerBaseUrl` गार्ड निम्न का समर्थन करता है:

| फ़ील्ड             | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                        |
| ----------------- | -------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `provider`        | हाँ      | `string`   | वह प्रदाता कॉन्फ़िग आईडी, जिसके `baseUrl` की जाँच की जानी चाहिए।                                                                                                |
| `defaultBaseUrl`  | नहीं       | `string`   | प्रदाता कॉन्फ़िग द्वारा `baseUrl` छोड़े जाने पर मान लिया जाने वाला बेस URL।                                                                                         |
| `allowedBaseUrls` | हाँ      | `string[]` | इस प्रमाणीकरण संकेत के लिए अनुमत बेस URL। जब कॉन्फ़िगर किया गया या डिफ़ॉल्ट बेस URL इन सामान्यीकृत मानों में से किसी से मेल नहीं खाता, तो संकेत अनदेखा कर दिया जाता है। |

## टूल मेटाडेटा संदर्भ

`toolMetadata` टूल नाम द्वारा कुंजीबद्ध, जनरेशन प्रदाता मेटाडेटा के समान `configSignals` और `authSignals` आकृतियों का उपयोग करता है। `contracts.tools` स्वामित्व घोषित करता है। `toolMetadata` कम लागत वाला उपलब्धता प्रमाण घोषित करता है, ताकि OpenClaw केवल उसके टूल फ़ैक्टरी से `null` वापस पाने के लिए Plugin रनटाइम आयात करने से बच सके।

```json
{
  "setup": {
    "providers": [
      {
        "id": "example",
        "envVars": ["EXAMPLE_API_KEY"]
      }
    ]
  },
  "contracts": {
    "tools": ["example_search"]
  },
  "toolMetadata": {
    "example_search": {
      "authSignals": [
        {
          "provider": "example"
        }
      ],
      "configSignals": [
        {
          "rootPath": "plugins.entries.example.config",
          "overlayPath": "search",
          "required": ["apiKey"]
        }
      ]
    }
  }
}
```

`toolMetadata` प्रविष्टियाँ ऊपर दिए गए साझा `configSignals`/`authSignals` फ़ील्ड के अतिरिक्त `optional` (टूल को Plugin सक्रियण के लिए अनावश्यक चिह्नित करता है) और `replaySafe` (अधूरे मॉडल टर्न के बाद टूल निष्पादन को दोहराने के लिए सुरक्षित चिह्नित करता है) भी स्वीकार करती हैं।

यदि किसी टूल में `toolMetadata` नहीं है, तो OpenClaw मौजूदा व्यवहार बनाए रखता है और टूल अनुबंध के नीति से मेल खाने पर स्वामी Plugin को लोड करता है। जिन हॉट-पाथ टूल की फ़ैक्टरी प्रमाणीकरण/कॉन्फ़िग पर निर्भर करती है, उनके लिए Plugin लेखकों को पूछने हेतु मुख्य भाग से रनटाइम आयात करवाने के बजाय `toolMetadata` घोषित करना चाहिए।

चुने गए सटीक मॉडल की पुनः जाँच की जाती है और एक कॉन्फ़िगरेशन प्रस्ताव लौटाया जाता है; OpenClaw उस
प्रस्ताव का अलगाव में लाइव परीक्षण करता है और सफलता के बाद ही उसे कमिट करता है।

## commandAliases संदर्भ

जब किसी Plugin के स्वामित्व वाला कोई रनटाइम कमांड नाम हो, जिसे उपयोगकर्ता भूलवश `plugins.allow` में डाल सकते हैं या रूट CLI कमांड के रूप में चलाने का प्रयास कर सकते हैं, तब `commandAliases` का उपयोग करें। OpenClaw, Plugin रनटाइम कोड आयात किए बिना निदान के लिए इस मेटाडेटा का उपयोग करता है।

```json
{
  "commandAliases": [
    {
      "name": "dreaming",
      "kind": "runtime-slash",
      "cliCommand": "memory"
    }
  ]
}
```

| फ़ील्ड        | आवश्यक | प्रकार              | इसका अर्थ                                                           |
| ------------ | -------- | ----------------- | ----------------------------------------------------------------------- |
| `name`       | हाँ      | `string`          | इस Plugin से संबंधित कमांड नाम।                               |
| `kind`       | नहीं       | `"runtime-slash"` | उपनाम को रूट CLI कमांड के बजाय चैट स्लैश कमांड के रूप में चिह्नित करता है। |
| `cliCommand` | नहीं       | `string`          | CLI कार्रवाइयों के लिए सुझाया जाने वाला संबंधित रूट CLI कमांड, यदि कोई हो।  |

## activation संदर्भ

जब Plugin कम लागत में यह घोषित कर सकता हो कि किन नियंत्रण-प्लेन घटनाओं के कारण उसे सक्रियण/लोड योजना में शामिल किया जाना चाहिए, तब `activation` का उपयोग करें।

यह ब्लॉक प्लानर मेटाडेटा है, जीवनचक्र API नहीं। यह रनटाइम व्यवहार पंजीकृत नहीं करता, `register(...)` को प्रतिस्थापित नहीं करता और यह आश्वासन नहीं देता कि Plugin कोड पहले ही निष्पादित हो चुका है। सक्रियण प्लानर, `providers`, `channels`, `commandAliases`, `setup.providers`, `contracts.tools` और हुक जैसे मौजूदा मैनिफ़ेस्ट स्वामित्व मेटाडेटा पर वापस जाने से पहले, उम्मीदवार Plugin को सीमित करने के लिए इन फ़ील्ड का उपयोग करता है।

पहले से स्वामित्व का वर्णन करने वाले सबसे संकीर्ण मेटाडेटा को प्राथमिकता दें। जब `providers`, `channels`, `commandAliases`, सेटअप वर्णनकर्ता या `contracts` संबंध व्यक्त करते हों, तो उनका उपयोग करें। उन अतिरिक्त प्लानर संकेतों के लिए `activation` का उपयोग करें, जिन्हें इन स्वामित्व फ़ील्ड से दर्शाया नहीं जा सकता। `claude-cli`, `my-cli` या `google-gemini-cli` जैसे CLI रनटाइम उपनामों के लिए शीर्ष-स्तरीय `cliBackends` का उपयोग करें; `activation.onAgentHarnesses` केवल उन एम्बेडेड एजेंट हार्नेस आईडी के लिए है, जिनका पहले से कोई स्वामित्व फ़ील्ड नहीं है।

प्रत्येक Plugin को `activation.onStartup` जानबूझकर सेट करना चाहिए। इसे `true` पर केवल तभी सेट करें, जब Plugin को Gateway स्टार्टअप के दौरान चलना आवश्यक हो। जब Plugin स्टार्टअप पर निष्क्रिय हो और उसे केवल अधिक संकीर्ण ट्रिगर से लोड किया जाना चाहिए, तब इसे `false` पर सेट करें। `onStartup` को छोड़ देने पर अब Plugin स्टार्टअप के समय अंतर्निहित रूप से लोड नहीं होता; स्टार्टअप, चैनल, कॉन्फ़िगरेशन, एजेंट हार्नेस, मेमोरी या अन्य अधिक संकीर्ण सक्रियण ट्रिगर के लिए स्पष्ट सक्रियण मेटाडेटा का उपयोग करें।

```json
{
  "activation": {
    "onStartup": false,
    "onProviders": ["openai"],
    "onCommands": ["models"],
    "onChannels": ["web"],
    "onRoutes": ["gateway-webhook"],
    "onConfigPaths": ["browser"],
    "onCapabilities": ["provider", "tool"]
  }
}
```

| फ़ील्ड              | आवश्यक | प्रकार                                                 | इसका अर्थ                                                                                                                                                                               |
| ------------------ | -------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `onStartup`        | नहीं       | `boolean`                                            | स्पष्ट Gateway स्टार्टअप सक्रियण। प्रत्येक Plugin को इसे सेट करना चाहिए। `true` स्टार्टअप के दौरान Plugin आयात करता है; जब तक किसी अन्य संगत ट्रिगर को लोडिंग की आवश्यकता न हो, `false` इसे स्टार्टअप पर विलंबित रखता है। |
| `onProviders`      | नहीं       | `string[]`                                           | वे प्रदाता आईडी, जिनके कारण इस Plugin को सक्रियण/लोड योजनाओं में शामिल किया जाना चाहिए।                                                                                                                      |
| `onAgentHarnesses` | नहीं       | `string[]`                                           | वे एम्बेडेड एजेंट हार्नेस रनटाइम आईडी, जिनके कारण इस Plugin को सक्रियण/लोड योजनाओं में शामिल किया जाना चाहिए। CLI बैकएंड उपनामों के लिए शीर्ष-स्तरीय `cliBackends` का उपयोग करें।                                           |
| `onCommands`       | नहीं       | `string[]`                                           | वे कमांड आईडी, जिनके कारण इस Plugin को सक्रियण/लोड योजनाओं में शामिल किया जाना चाहिए।                                                                                                                       |
| `onChannels`       | नहीं       | `string[]`                                           | वे चैनल आईडी, जिनके कारण इस Plugin को सक्रियण/लोड योजनाओं में शामिल किया जाना चाहिए।                                                                                                                       |
| `onRoutes`         | नहीं       | `string[]`                                           | वे रूट प्रकार, जिनके कारण इस Plugin को सक्रियण/लोड योजनाओं में शामिल किया जाना चाहिए।                                                                                                                       |
| `onConfigPaths`    | नहीं       | `string[]`                                           | रूट-सापेक्ष कॉन्फ़िगरेशन पथ, जिनके मौजूद और स्पष्ट रूप से अक्षम न होने पर इस Plugin को स्टार्टअप/लोड योजनाओं में शामिल किया जाना चाहिए।                                                      |
| `onCapabilities`   | नहीं       | `Array<"provider" \| "channel" \| "tool" \| "hook">` | नियंत्रण-प्लेन सक्रियण योजना द्वारा उपयोग किए जाने वाले व्यापक क्षमता संकेत। संभव होने पर अधिक संकीर्ण फ़ील्ड को प्राथमिकता दें।                                                                                     |

वर्तमान लाइव उपभोक्ता:

- Gateway स्टार्टअप योजना स्पष्ट स्टार्टअप इम्पोर्ट के लिए `activation.onStartup` का उपयोग करती है।
- कमांड से ट्रिगर होने वाली CLI योजना लेगेसी `commandAliases[].cliCommand` या `commandAliases[].name` पर फ़ॉलबैक करती है।
- एजेंट-रनटाइम स्टार्टअप योजना एम्बेडेड हार्नेस के लिए `activation.onAgentHarnesses` और CLI रनटाइम उपनामों के लिए शीर्ष-स्तरीय `cliBackends[]` का उपयोग करती है।
- स्पष्ट चैनल सक्रियण मेटाडेटा न होने पर चैनल से ट्रिगर होने वाली सेटअप/चैनल योजना लेगेसी `channels[]` स्वामित्व पर फ़ॉलबैक करती है।
- स्टार्टअप Plugin योजना गैर-चैनल रूट कॉन्फ़िगरेशन सतहों के लिए `activation.onConfigPaths` का उपयोग करती है, जैसे बंडल किए गए ब्राउज़र Plugin का `browser` ब्लॉक।
- स्पष्ट प्रोवाइडर सक्रियण मेटाडेटा न होने पर प्रोवाइडर से ट्रिगर होने वाली सेटअप/रनटाइम योजना लेगेसी `providers[]` और शीर्ष-स्तरीय `cliBackends[]` स्वामित्व पर फ़ॉलबैक करती है।

प्लानर डायग्नोस्टिक्स स्पष्ट सक्रियण संकेतों को मैनिफ़ेस्ट स्वामित्व फ़ॉलबैक से अलग कर सकते हैं। उदाहरण के लिए, `activation-command-hint` का अर्थ है कि `activation.onCommands` मेल खाता है, जबकि `manifest-command-alias` का अर्थ है कि प्लानर ने इसके बजाय `commandAliases` स्वामित्व का उपयोग किया। ये कारण लेबल होस्ट डायग्नोस्टिक्स और परीक्षणों के लिए हैं; Plugin लेखकों को स्वामित्व का सर्वोत्तम वर्णन करने वाला मेटाडेटा घोषित करते रहना चाहिए।

## qaRunners संदर्भ

जब कोई Plugin साझा `openclaw qa` रूट के नीचे एक या अधिक ट्रांसपोर्ट रनर प्रदान करता है, तब `qaRunners` का उपयोग करें। इस मेटाडेटा को हल्का और स्थिर रखें; Plugin
रनटाइम अब भी वास्तविक CLI पंजीकरण का स्वामी एक हल्की
`runtime-api.ts` सतह के माध्यम से है, जो मेल खाते `qaRunnerCliRegistrations` निर्यात करती है। एक
वैकल्पिक `adapterFactory` पंजीकृत कमांड के रनर को बदले बिना ट्रांसपोर्ट को साझा QA परिदृश्यों के लिए उपलब्ध कराता है।

```json
{
  "qaRunners": [
    {
      "commandName": "matrix",
      "description": "डिस्पोज़ेबल होमसर्वर के विरुद्ध Docker-समर्थित Matrix लाइव QA लेन चलाएँ"
    }
  ]
}
```

| फ़ील्ड         | आवश्यक | प्रकार     | इसका अर्थ                                                      |
| ------------- | -------- | -------- | ------------------------------------------------------------------ |
| `commandName` | हाँ      | `string` | `openclaw qa` के नीचे माउंट किया गया सबकमांड, उदाहरण के लिए `matrix`।    |
| `description` | नहीं       | `string` | साझा होस्ट को स्टब कमांड की आवश्यकता होने पर उपयोग किया जाने वाला फ़ॉलबैक सहायता टेक्स्ट। |

`adapterFactory` आईडी का `commandName` से मेल खाना आवश्यक है। मैनिफ़ेस्ट में अनुपस्थित कमांड के लिए पंजीकरण निर्यात न करें।

## सेटअप संदर्भ

जब सेटअप और ऑनबोर्डिंग सतहों को रनटाइम लोड होने से पहले हल्के Plugin-स्वामित्व वाले मेटाडेटा की आवश्यकता हो, तब `setup` का उपयोग करें।

```json
{
  "setup": {
    "providers": [
      {
        "id": "openai",
        "authMethods": ["api-key"],
        "envVars": ["OPENAI_API_KEY"],
        "authEvidence": [
          {
            "type": "local-file-with-env",
            "fileEnvVar": "OPENAI_CREDENTIALS_FILE",
            "requiresAllEnv": ["OPENAI_PROJECT"],
            "credentialMarker": "openai-local-credentials",
            "source": "OpenAI स्थानीय क्रेडेंशियल"
          }
        ]
      }
    ],
    "cliBackends": ["openai-cli"],
    "configMigrations": ["legacy-openai-auth"],
    "requiresRuntime": false
  }
}
```

शीर्ष-स्तरीय `cliBackends` मान्य रहता है और CLI अनुमान बैकएंड का वर्णन करना जारी रखता है। `setup.cliBackends` उन कंट्रोल-प्लेन/सेटअप प्रवाहों के लिए सेटअप-विशिष्ट डिस्क्रिप्टर सतह है, जिन्हें केवल मेटाडेटा तक सीमित रहना चाहिए।

मौजूद होने पर, `setup.providers` और `setup.cliBackends` सेटअप खोज के लिए पसंदीदा डिस्क्रिप्टर-प्रथम लुकअप सतह हैं। यदि डिस्क्रिप्टर केवल संभावित Plugin को सीमित करता है और सेटअप को अब भी अधिक समृद्ध सेटअप-समय रनटाइम हुक की आवश्यकता है, तो `requiresRuntime: true` सेट करें और फ़ॉलबैक निष्पादन पथ के रूप में `setup-api` बनाए रखें।

OpenClaw सामान्य प्रोवाइडर प्रमाणीकरण और एनवायरनमेंट-वेरिएबल लुकअप में `setup.providers[].envVars` को भी शामिल करता है। अप्रचलन अवधि के दौरान `providerAuthEnvVars` एक संगतता अडैप्टर के माध्यम से समर्थित रहता है, लेकिन अब भी इसका उपयोग करने वाले गैर-बंडल Plugin को मैनिफ़ेस्ट डायग्नोस्टिक मिलता है। नए Plugin को सेटअप/स्थिति एनवायरनमेंट मेटाडेटा `setup.providers[].envVars` पर रखना चाहिए।

जब किसी बिलिंग या संगठन-स्तरीय क्रेडेंशियल को अनुमान क्रेडेंशियल बने बिना `resolveUsageAuth` सक्रिय करना आवश्यक हो, तब `providerUsageAuthEnvVars` का उपयोग करें। ये नाम वर्कस्पेस dotenv अवरोधन, ACP चाइल्ड-प्रोसेस स्ट्रिपिंग, सैंडबॉक्स सीक्रेट फ़िल्टरिंग और व्यापक सीक्रेट स्क्रबिंग में शामिल होते हैं। प्रोवाइडर रनटाइम अब भी `resolveUsageAuth` के भीतर मान को पढ़ता और वर्गीकृत करता है।

जब कोई सेटअप प्रविष्टि उपलब्ध न हो, या जब `setup.requiresRuntime: false` सेटअप रनटाइम को अनावश्यक घोषित करे, तब OpenClaw `setup.providers[].authMethods` से सरल सेटअप विकल्प भी प्राप्त कर सकता है। कस्टम लेबल, CLI फ़्लैग, ऑनबोर्डिंग दायरे और सहायक मेटाडेटा के लिए स्पष्ट `providerAuthChoices` प्रविष्टियाँ पसंदीदा रहती हैं।

`requiresRuntime: false` केवल तभी सेट करें, जब वे डिस्क्रिप्टर सेटअप सतह के लिए पर्याप्त हों। OpenClaw स्पष्ट `false` को केवल-डिस्क्रिप्टर अनुबंध मानता है और सेटअप लुकअप के लिए `setup-api` या `openclaw.setupEntry` निष्पादित नहीं करेगा। यदि कोई केवल-डिस्क्रिप्टर Plugin फिर भी उन सेटअप रनटाइम प्रविष्टियों में से एक प्रदान करता है, तो OpenClaw एक अतिरिक्त डायग्नोस्टिक रिपोर्ट करता है और उसे अनदेखा करना जारी रखता है। `requiresRuntime` को छोड़ देने पर लेगेसी फ़ॉलबैक व्यवहार बना रहता है, ताकि फ़्लैग के बिना डिस्क्रिप्टर जोड़ने वाले मौजूदा Plugin न टूटें।

चूँकि सेटअप लुकअप Plugin-स्वामित्व वाला `setup-api` कोड निष्पादित कर सकता है, सामान्यीकृत `setup.providers[].id` और `setup.cliBackends[]` मान खोजे गए सभी Plugin में अद्वितीय रहने चाहिए। अस्पष्ट स्वामित्व खोज क्रम से विजेता चुनने के बजाय सुरक्षित रूप से विफल होता है।

जब सेटअप रनटाइम निष्पादित होता है, तब यदि `setup-api` ऐसे प्रोवाइडर या CLI बैकएंड को पंजीकृत करता है जिसे मैनिफ़ेस्ट डिस्क्रिप्टर घोषित नहीं करते, या यदि किसी डिस्क्रिप्टर के लिए मेल खाता रनटाइम पंजीकरण नहीं है, तो सेटअप रजिस्ट्री डायग्नोस्टिक्स डिस्क्रिप्टर विचलन की रिपोर्ट करते हैं। ये डायग्नोस्टिक्स अतिरिक्त हैं और लेगेसी Plugin को अस्वीकार नहीं करते।

### setup.providers संदर्भ

| फ़ील्ड          | आवश्यक | प्रकार       | इसका अर्थ                                                                                    |
| -------------- | -------- | ---------- | ------------------------------------------------------------------------------------------------ |
| `id`           | हाँ      | `string`   | सेटअप या ऑनबोर्डिंग के दौरान प्रदर्शित प्रोवाइडर आईडी। सामान्यीकृत आईडी को वैश्विक रूप से अद्वितीय रखें।             |
| `authMethods`  | नहीं       | `string[]` | वे सेटअप/प्रमाणीकरण विधि आईडी जिन्हें यह प्रोवाइडर पूरा रनटाइम लोड किए बिना समर्थित करता है।                       |
| `envVars`      | नहीं       | `string[]` | वे एनवायरनमेंट वेरिएबल जिन्हें सामान्य सेटअप/स्थिति सतहें Plugin रनटाइम लोड होने से पहले जाँच सकती हैं।               |
| `authEvidence` | नहीं       | `object[]` | उन प्रोवाइडर के लिए हल्की स्थानीय प्रमाणीकरण प्रमाण जाँचें, जो गैर-सीक्रेट मार्कर के माध्यम से प्रमाणीकरण कर सकते हैं। |

`authEvidence` प्रोवाइडर-स्वामित्व वाले स्थानीय क्रेडेंशियल मार्कर के लिए है, जिन्हें रनटाइम कोड लोड किए बिना सत्यापित किया जा सकता है। ये जाँच हल्की और स्थानीय रहनी चाहिए: कोई नेटवर्क कॉल नहीं, कीचेन या सीक्रेट-मैनेजर रीड नहीं, शेल कमांड नहीं और प्रोवाइडर API प्रोब नहीं।

समर्थित प्रमाण प्रविष्टियाँ:

| फ़ील्ड              | आवश्यक | प्रकार       | इसका अर्थ                                                                                                  |
| ------------------ | -------- | ---------- | -------------------------------------------------------------------------------------------------------------- |
| `type`             | हाँ      | `string`   | वर्तमान में `local-file-with-env`।                                                                               |
| `fileEnvVar`       | नहीं       | `string`   | स्पष्ट क्रेडेंशियल फ़ाइल पथ वाला एनवायरनमेंट वेरिएबल।                                                           |
| `fallbackPaths`    | नहीं       | `string[]` | `fileEnvVar` के अनुपस्थित या खाली होने पर जाँचे जाने वाले स्थानीय क्रेडेंशियल फ़ाइल पथ। `${HOME}` और `${APPDATA}` समर्थित हैं। |
| `requiresAnyEnv`   | नहीं       | `string[]` | प्रमाण के मान्य होने से पहले सूचीबद्ध एनवायरनमेंट वेरिएबल में से कम-से-कम एक गैर-रिक्त होना आवश्यक है।                                    |
| `requiresAllEnv`   | नहीं       | `string[]` | प्रमाण के मान्य होने से पहले प्रत्येक सूचीबद्ध एनवायरनमेंट वेरिएबल का गैर-रिक्त होना आवश्यक है।                                           |
| `credentialMarker` | हाँ      | `string`   | प्रमाण मौजूद होने पर लौटाया जाने वाला गैर-सीक्रेट मार्कर।                                                       |
| `source`           | नहीं       | `string`   | प्रमाणीकरण/स्थिति आउटपुट के लिए उपयोगकर्ता-दृश्य स्रोत लेबल।                                                               |

### सेटअप फ़ील्ड

| फ़ील्ड              | आवश्यक | प्रकार       | इसका अर्थ                                                                                       |
| ------------------ | -------- | ---------- | --------------------------------------------------------------------------------------------------- |
| `providers`        | नहीं       | `object[]` | सेटअप और ऑनबोर्डिंग के दौरान प्रदर्शित प्रोवाइडर सेटअप डिस्क्रिप्टर।                                     |
| `cliBackends`      | नहीं       | `string[]` | डिस्क्रिप्टर-प्रथम सेटअप लुकअप के लिए उपयोग किए जाने वाले सेटअप-समय बैकएंड आईडी। सामान्यीकृत आईडी को वैश्विक रूप से अद्वितीय रखें। |
| `configMigrations` | नहीं       | `string[]` | इस Plugin की सेटअप सतह के स्वामित्व वाली कॉन्फ़िगरेशन माइग्रेशन आईडी।                                          |
| `requiresRuntime`  | नहीं       | `boolean`  | क्या डिस्क्रिप्टर लुकअप के बाद भी सेटअप को `setup-api` निष्पादन की आवश्यकता है।                            |

## uiHints संदर्भ

`uiHints` कॉन्फ़िगरेशन फ़ील्ड नामों से छोटे रेंडरिंग संकेतों तक का मैप है। नेस्टेड कॉन्फ़िगरेशन फ़ील्ड के लिए कुंजियाँ डॉट का उपयोग कर सकती हैं, लेकिन कोई भी पथ खंड `__proto__`, `constructor` या `prototype` नहीं हो सकता; सेटअप इन नामों को अस्वीकार करता है।

```json
{
  "uiHints": {
    "apiKey": {
      "label": "API कुंजी",
      "help": "OpenRouter अनुरोधों के लिए उपयोग की जाती है",
      "placeholder": "sk-or-v1-...",
      "sensitive": true
    }
  }
}
```

प्रत्येक फ़ील्ड संकेत में ये शामिल हो सकते हैं:

| फ़ील्ड         | प्रकार       | इसका अर्थ                           |
| ------------- | ---------- | --------------------------------------- |
| `label`       | `string`   | उपयोगकर्ता-दृश्य फ़ील्ड लेबल।                |
| `help`        | `string`   | संक्षिप्त सहायक टेक्स्ट।                      |
| `tags`        | `string[]` | वैकल्पिक UI टैग।                       |
| `advanced`    | `boolean`  | फ़ील्ड को उन्नत के रूप में चिह्नित करता है।            |
| `sensitive`   | `boolean`  | फ़ील्ड को सीक्रेट या संवेदनशील के रूप में चिह्नित करता है। |
| `placeholder` | `string`   | फ़ॉर्म इनपुट के लिए प्लेसहोल्डर टेक्स्ट।       |

## अनुबंध संदर्भ

`contracts` का उपयोग केवल स्थिर क्षमता स्वामित्व मेटाडेटा के लिए करें, जिसे OpenClaw Plugin रनटाइम इम्पोर्ट किए बिना पढ़ सकता है।

```json
{
  "contracts": {
    "agentToolResultMiddleware": ["openclaw", "codex"],
    "trustedToolPolicies": ["workflow-budget"],
    "externalAuthProviders": ["acme-ai"],
    "embeddingProviders": ["openai-compatible"],
    "speechProviders": ["openai"],
    "realtimeTranscriptionProviders": ["openai"],
    "realtimeVoiceProviders": ["openai"],
    "memoryEmbeddingProviders": ["local"],
    "mediaUnderstandingProviders": ["openai"],
    "imageGenerationProviders": ["openai"],
    "videoGenerationProviders": ["qwen"],
    "musicGenerationProviders": ["stability-audio"],
    "documentExtractors": ["example-docs"],
    "webContentExtractors": ["firecrawl"],
    "webFetchProviders": ["firecrawl"],
    "webSearchProviders": ["gemini"],
    "workerProviders": ["example-worker"],
    "usageProviders": ["acme-ai"],
    "migrationProviders": ["hermes"],
    "gatewayMethodDispatch": ["authenticated-request"],
    "tools": ["firecrawl_search", "firecrawl_scrape"]
  }
}
```

प्रत्येक सूची वैकल्पिक है:

| फ़ील्ड                            | प्रकार       | इसका अर्थ                                                                                                                        |
| -------------------------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `embeddedExtensionFactories`     | `string[]` | Codex app-server एक्सटेंशन फ़ैक्टरी आईडी, वर्तमान में `codex-app-server`।                                                                |
| `agentToolResultMiddleware`      | `string[]` | वे रनटाइम आईडी जिनके लिए यह Plugin टूल-परिणाम मिडलवेयर पंजीकृत कर सकता है।                                                                     |
| `trustedToolPolicies`            | `string[]` | Plugin-स्थानीय विश्वसनीय प्री-टूल नीति आईडी जिन्हें कोई इंस्टॉल किया गया Plugin पंजीकृत कर सकता है। बंडल किए गए Plugin इस फ़ील्ड के बिना नीतियाँ पंजीकृत कर सकते हैं। |
| `externalAuthProviders`          | `string[]` | वे प्रदाता आईडी जिनके बाहरी प्रमाणीकरण प्रोफ़ाइल हुक का स्वामित्व इस Plugin के पास है।                                                                      |
| `embeddingProviders`             | `string[]` | सामान्य एम्बेडिंग प्रदाता आईडी जिनका स्वामित्व पुनः उपयोग योग्य वेक्टर एम्बेडिंग के लिए इस Plugin के पास है, जिसमें मेमोरी भी शामिल है।                                 |
| `speechProviders`                | `string[]` | वाक् प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                                |
| `realtimeTranscriptionProviders` | `string[]` | रीयलटाइम-ट्रांसक्रिप्शन प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                |
| `realtimeVoiceProviders`         | `string[]` | रीयलटाइम-वॉइस प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                        |
| `memoryEmbeddingProviders`       | `string[]` | अप्रचलित मेमोरी-विशिष्ट एम्बेडिंग प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                  |
| `mediaUnderstandingProviders`    | `string[]` | मीडिया-समझ प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                   |
| `transcriptSourceProviders`      | `string[]` | ट्रांसक्रिप्ट स्रोत प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                     |
| `documentExtractors`             | `string[]` | दस्तावेज़ (उदाहरण के लिए PDF) निष्कर्षक प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                  |
| `imageGenerationProviders`       | `string[]` | छवि-निर्माण प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                      |
| `videoGenerationProviders`       | `string[]` | वीडियो-निर्माण प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                      |
| `musicGenerationProviders`       | `string[]` | संगीत-निर्माण प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                      |
| `webContentExtractors`           | `string[]` | वेब-पृष्ठ सामग्री-निष्कर्षण प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                           |
| `webFetchProviders`              | `string[]` | वेब-फ़ेच प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                             |
| `webSearchProviders`             | `string[]` | वेब-खोज प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                                            |
| `workerProviders`                | `string[]` | प्रावधान और प्रोफ़ाइल-समर्थित लीज़ जीवनचक्र के लिए क्लाउड-वर्कर प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                      |
| `usageProviders`                 | `string[]` | वे प्रदाता आईडी जिनके उपयोग-प्रमाणीकरण और उपयोग-स्नैपशॉट हुक का स्वामित्व इस Plugin के पास है।                                                             |
| `migrationProviders`             | `string[]` | `openclaw migrate` के लिए आयात प्रदाता आईडी जिनका स्वामित्व इस Plugin के पास है।                                                                         |
| `gatewayMethodDispatch`          | `string[]` | उन प्रमाणीकृत Plugin HTTP रूटों के लिए आरक्षित अधिकार, जो प्रक्रिया के भीतर Gateway विधियाँ डिस्पैच करते हैं।                                  |
| `tools`                          | `string[]` | एजेंट टूल नाम जिनका स्वामित्व इस Plugin के पास है।                                                                                                   |

`contracts.embeddedExtensionFactories` को केवल बंडल किए गए Codex app-server एक्सटेंशन फ़ैक्टरी के लिए बनाए रखा गया है। इसके बजाय बंडल किए गए टूल-परिणाम रूपांतरणों को `contracts.agentToolResultMiddleware` घोषित करना और `api.registerAgentToolResultMiddleware(...)` के साथ पंजीकृत होना चाहिए। इंस्टॉल किए गए Plugin उसी मिडलवेयर सीम का उपयोग केवल स्पष्ट रूप से सक्षम होने पर और केवल उन रनटाइम के लिए कर सकते हैं जिन्हें वे `contracts.agentToolResultMiddleware` में घोषित करते हैं।

जिन इंस्टॉल किए गए Plugin को होस्ट-विश्वसनीय प्री-टूल नीति स्तर की आवश्यकता है, उन्हें प्रत्येक पंजीकृत स्थानीय आईडी को `contracts.trustedToolPolicies` में घोषित करना और स्पष्ट रूप से सक्षम होना आवश्यक है। बंडल किए गए Plugin मौजूदा विश्वसनीय-नीति पथ बनाए रखते हैं, लेकिन अघोषित नीति आईडी वाले इंस्टॉल किए गए Plugin को पंजीकरण से पहले अस्वीकार कर दिया जाता है। नीति आईडी पंजीकरण करने वाले Plugin के दायरे में होते हैं, इसलिए दो Plugin दोनों `workflow-budget` घोषित और पंजीकृत कर सकते हैं; कोई एक Plugin उसी स्थानीय आईडी को दो बार पंजीकृत नहीं कर सकता।

रनटाइम `api.registerTool(...)` पंजीकरणों का `contracts.tools` से मेल खाना आवश्यक है। टूल खोज इस सूची का उपयोग केवल उन Plugin रनटाइम को लोड करने के लिए करती है जो अनुरोधित टूल के स्वामी हो सकते हैं।

`resolveExternalAuthProfiles` लागू करने वाले प्रदाता Plugin को `contracts.externalAuthProviders` घोषित करना चाहिए; अघोषित बाहरी-प्रमाणीकरण हुक अनदेखे कर दिए जाते हैं।

`resolveUsageAuth` और `fetchUsageSnapshot` दोनों लागू करने वाले प्रदाता Plugin को प्रत्येक स्वतः खोजे गए प्रदाता आईडी को `contracts.usageProviders` में घोषित करना चाहिए। उपयोग खोज रनटाइम कोड लोड करने से पहले इस अनुबंध को पढ़ती है, फिर केवल घोषित स्वामियों को लोड करने के बाद दोनों हुक सत्यापित करती है।

सामान्य एम्बेडिंग प्रदाताओं को `api.registerEmbeddingProvider(...)` के साथ पंजीकृत प्रत्येक अडैप्टर के लिए `contracts.embeddingProviders` घोषित करना चाहिए। पुनः उपयोग योग्य वेक्टर निर्माण के लिए सामान्य अनुबंध का उपयोग करें, जिसमें मेमोरी खोज द्वारा उपयोग किए जाने वाले प्रदाता शामिल हैं। `contracts.memoryEmbeddingProviders` अप्रचलित मेमोरी-विशिष्ट संगतता है और केवल तब तक बनी रहती है जब तक मौजूदा प्रदाता सामान्य एम्बेडिंग प्रदाता सीम पर माइग्रेट नहीं हो जाते।

वर्कर प्रदाताओं को प्रत्येक `api.registerWorkerProvider(...)` आईडी को `contracts.workerProviders` में घोषित करना आवश्यक है। कोर `provision` को कॉल करने से पहले स्थायी अभिप्राय को सहेजता है; प्रदाता बाहरी आवंटन से पहले अपनी सेटिंग्स सत्यापित करते हैं, और समान ऑपरेशन आईडी वाली दोहराई गई कॉल को उसी लीज़ को अपनाना आवश्यक है। कोर उस सत्यापित सेटिंग्स स्नैपशॉट को भी सहेजता है और उसे `leaseId` के साथ `inspect({ leaseId, profile })` तथा `destroy({ leaseId, profile })` को भेजता है, जिसमें नामित प्रोफ़ाइल को बदले या हटाए जाने के बाद की स्थिति भी शामिल है। विनाश आइडेम्पोटेंट है, निरीक्षण बंद `active` / `destroyed` / `unknown` स्थिति यूनियन लौटाता है, और SSH निजी-कुंजी सामग्री को केवल `SecretRef` के माध्यम से संदर्भित किया जाता है। प्रावधान किए गए SSH एंडपॉइंट में विश्वसनीय प्रावधान आउटपुट से सार्वजनिक `hostKey` भी ठीक `algorithm base64` के रूप में शामिल होना आवश्यक है, बिना होस्टनाम या टिप्पणी के, ताकि कोर कनेक्ट करने से पहले होस्ट को पिन कर सके। डायनेमिक पहचान संदर्भ जारी करने वाले प्रदाता प्रामाणिक `resolveSshIdentity({ leaseId, profile, keyRef })` लागू कर सकते हैं; इसके बिना प्रदाता कोर के सामान्य सीक्रेट रिज़ॉल्वर का उपयोग करते हैं। प्रामाणिक `unknown` किसी सक्रिय स्थानीय रिकॉर्ड को अनाथ बनाता है; सहेजे गए विनाश अनुरोध के बाद यह विघटन की पुष्टि करता है।

`contracts.gatewayMethodDispatch` वर्तमान में `"authenticated-request"` स्वीकार करता है। यह उन नेटिव Plugin HTTP रूटों के लिए एक API स्वच्छता गेट है जो जानबूझकर प्रक्रिया के भीतर Gateway नियंत्रण-प्लेन विधियाँ डिस्पैच करते हैं, न कि दुर्भावनापूर्ण नेटिव Plugin के विरुद्ध कोई सैंडबॉक्स। इसका उपयोग केवल कड़ी समीक्षा वाले बंडल/ऑपरेटर सतहों के लिए करें जिन्हें पहले से Gateway HTTP प्रमाणीकरण की आवश्यकता होती है। जब Gateway रूट-वर्क प्रवेश बंद हो, तब कोई अधिकार-प्राप्त रूट केवल तभी पहुँच योग्य रहता है जब वह `auth: "gateway"` और रूट-विशिष्ट `gatewayRuntimeScopeSurface: "trusted-operator"` भी घोषित करता हो; उसी Plugin के सामान्य सहोदर रूट प्रवेश सीमा के पीछे रहते हैं। इससे पूरे Plugin को प्रवेश बाइपास दिए बिना निलंबन स्थिति और पुनः आरंभ तक पहुँच बनी रहती है। पार्सिंग और प्रतिक्रिया आकार निर्धारण को डिस्पैच के बाहर सीमित रखें; सारभूत या परिवर्तनकारी कार्य को Gateway विधि डिस्पैच से होकर जाना आवश्यक है, जो प्रवेश और दायरा प्रवर्तन का स्वामी है।

## configContracts संदर्भ

मैनिफ़ेस्ट-स्वामित्व वाले उस कॉन्फ़िग व्यवहार के लिए `configContracts` का उपयोग करें जिसकी सामान्य कोर सहायकों को Plugin रनटाइम आयात किए बिना आवश्यकता होती है: खतरनाक-फ़्लैग पहचान, SecretRef माइग्रेशन लक्ष्य और विरासत कॉन्फ़िग-पथ संकुचन।

```json
{
  "configContracts": {
    "compatibilityMigrationPaths": ["legacyProvider"],
    "compatibilityRuntimePaths": ["legacyProvider.webhook"],
    "dangerousFlags": [
      {
        "path": "accounts.*.allowUnverifiedSenders",
        "equals": true
      }
    ],
    "secretInputs": {
      "bundledDefaultEnabled": false,
      "paths": [
        {
          "path": "routes.*.secret",
          "expected": "string",
          "ownerKind": "route"
        }
      ]
    }
  }
}
```

| फ़ील्ड                         | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                                                                                                          |
| ----------------------------- | -------- | ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `compatibilityMigrationPaths` | नहीं       | `string[]` | रूट-सापेक्ष कॉन्फ़िग पथ जो संकेत देते हैं कि इस Plugin के सेटअप-समय संगतता माइग्रेशन लागू हो सकते हैं। जब कॉन्फ़िग में Plugin का कोई संदर्भ न हो, तब सामान्य रनटाइम कॉन्फ़िग पठन प्रत्येक Plugin सेटअप सतह को छोड़ सकता है।                 |
| `compatibilityRuntimePaths`   | नहीं       | `string[]` | रूट-सापेक्ष संगतता पथ जिन्हें यह Plugin, Plugin कोड के पूरी तरह सक्रिय होने से पहले रनटाइम के दौरान सेवा दे सकता है। इसका उपयोग उन विरासत सतहों के लिए करें जिन्हें प्रत्येक संगत Plugin रनटाइम आयात किए बिना बंडल किए गए उम्मीदवार समूहों को संकुचित करना चाहिए। |
| `dangerousFlags`              | नहीं       | `object[]` | वे कॉन्फ़िग लिटरल जिन्हें सक्षम होने पर `openclaw doctor` को असुरक्षित या खतरनाक के रूप में चिह्नित करना चाहिए। नीचे देखें।                                                                                                                                   |
| `secretInputs`                | नहीं       | `object`   | SecretRef माइग्रेशन, ऑडिट, स्टार्टअप मटेरियलाइज़ेशन और वैकल्पिक रनटाइम स्वामी पृथक्करण के लिए `plugins.entries.<id>.config` के अंतर्गत कॉन्फ़िग पथ। नीचे देखें।                                                                             |

प्रत्येक `dangerousFlags` प्रविष्टि इसका समर्थन करती है:

| फ़ील्ड    | आवश्यक | प्रकार                                  | इसका अर्थ                                                                                                       |
| -------- | -------- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `path`   | हाँ      | `string`                              | `plugins.entries.<id>.config` के सापेक्ष डॉट से अलग किया गया कॉन्फ़िग पथ। मैप/ऐरे खंडों के लिए `*` वाइल्डकार्ड का समर्थन करता है। |
| `equals` | हाँ      | `string \| number \| boolean \| null` | वह सटीक लिटरल जो इस कॉन्फ़िग मान को खतरनाक के रूप में चिह्नित करता है।                                                            |

`secretInputs` इसका समर्थन करता है:

| फ़ील्ड                   | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                                                                                                                                                                                                                              |
| ----------------------- | -------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bundledDefaultEnabled` | नहीं       | `boolean`  | यह तय करते समय कि यह SecretRef सतह सक्रिय है या नहीं, बंडल किए गए Plugin की डिफ़ॉल्ट सक्षमता को ओवरराइड करें। इसका उपयोग तब करें जब Plugin बंडल किया गया हो, लेकिन कॉन्फ़िगरेशन में स्पष्ट रूप से सक्षम किए जाने तक सतह निष्क्रिय रहनी चाहिए।                                                                                                                                            |
| `paths`                 | हाँ      | `object[]` | सीक्रेट के स्वरूप वाले कॉन्फ़िगरेशन पथ, जिनमें प्रत्येक के साथ `path` (डॉट से अलग किया गया, `plugins.entries.<id>.config` के सापेक्ष, `*` वाइल्डकार्ड का समर्थन करता है), वैकल्पिक `expected` (वर्तमान में केवल `"string"`), और वैकल्पिक `ownerKind` (वर्तमान में केवल `"route"`) होता है। समाधान विफल होने पर घोषित स्वामी केवल उसी सटीक मिलान वाले पथ को अलग करता है; उसकी स्वामी आईडी पूर्ण कॉन्फ़िगरेशन पथ होती है। |

## mediaUnderstandingProviderMetadata संदर्भ

जब किसी मीडिया-समझ प्रदाता के पास डिफ़ॉल्ट मॉडल, स्वचालित प्रमाणीकरण फ़ॉलबैक प्राथमिकता, या मूल दस्तावेज़ समर्थन हो, जिसकी सामान्य कोर सहायकों को रनटाइम लोड होने से पहले आवश्यकता होती है, तब `mediaUnderstandingProviderMetadata` का उपयोग करें। कुंजियाँ `contracts.mediaUnderstandingProviders` में भी घोषित होनी चाहिए।

```json
{
  "contracts": {
    "mediaUnderstandingProviders": ["example"]
  },
  "mediaUnderstandingProviderMetadata": {
    "example": {
      "capabilities": ["image", "audio"],
      "defaultModels": {
        "image": "example-vision-latest",
        "audio": "example-transcribe-latest"
      },
      "autoPriority": {
        "image": 40
      },
      "nativeDocumentInputs": ["pdf"],
      "documentModels": {
        "pdf": {
          "textExtraction": "example-doc-text-latest",
          "image": "example-doc-vision-latest"
        }
      }
    }
  }
}
```

प्रत्येक प्रदाता प्रविष्टि में ये शामिल हो सकते हैं:

| फ़ील्ड                  | प्रकार                                                             | इसका अर्थ                                                                                                   |
| ---------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `capabilities`         | `("image" \| "audio" \| "video")[]`                              | इस प्रदाता द्वारा उपलब्ध कराई गई मीडिया क्षमताएँ।                                                                    |
| `defaultModels`        | `Record<string, string>`                                         | जब कॉन्फ़िगरेशन कोई मॉडल निर्दिष्ट नहीं करता, तब उपयोग किए जाने वाले क्षमता-से-मॉडल डिफ़ॉल्ट।                                         |
| `autoPriority`         | `Record<string, number>`                                         | स्वचालित क्रेडेंशियल-आधारित प्रदाता फ़ॉलबैक के लिए छोटी संख्याएँ पहले क्रमबद्ध होती हैं।                                    |
| `nativeDocumentInputs` | `"pdf"[]`                                                        | प्रदाता द्वारा समर्थित मूल दस्तावेज़ इनपुट।                                                               |
| `documentModels`       | `{ pdf?: { textExtraction?: string; image?: string \| false } }` | प्रत्येक दस्तावेज़ प्रकार के लिए मॉडल ओवरराइड। उस दस्तावेज़ प्रकार हेतु छवि-आधारित निष्कर्षण अक्षम करने के लिए `image: false` सेट करें। |

## channelConfigs संदर्भ

जब किसी चैनल Plugin को रनटाइम लोड होने से पहले हल्के कॉन्फ़िगरेशन मेटाडेटा की आवश्यकता हो, तब `channelConfigs` का उपयोग करें। केवल-पठन चैनल सेटअप/स्थिति खोज इस मेटाडेटा का सीधे कॉन्फ़िगर किए गए बाहरी चैनलों के लिए उपयोग कर सकती है, जब कोई सेटअप प्रविष्टि उपलब्ध न हो, या जब `setup.requiresRuntime: false` यह घोषित करे कि सेटअप रनटाइम अनावश्यक है।

`channelConfigs` Plugin मैनिफ़ेस्ट मेटाडेटा है, कोई नया शीर्ष-स्तरीय उपयोगकर्ता कॉन्फ़िगरेशन अनुभाग नहीं। उपयोगकर्ता अब भी चैनल इंस्टेंस को `channels.<channel-id>` के अंतर्गत कॉन्फ़िगर करते हैं। Plugin रनटाइम कोड निष्पादित होने से पहले यह तय करने के लिए कि उस कॉन्फ़िगर किए गए चैनल का स्वामी कौन-सा Plugin है, OpenClaw मैनिफ़ेस्ट मेटाडेटा पढ़ता है।

किसी चैनल Plugin के लिए, `configSchema` और `channelConfigs` अलग-अलग पथों का वर्णन करते हैं:

- `configSchema`, `plugins.entries.<plugin-id>.config` को सत्यापित करता है
- `channelConfigs.<channel-id>.schema`, `channels.<channel-id>` को सत्यापित करता है

`channels[]` घोषित करने वाले गैर-बंडल Plugin को मेल खाती `channelConfigs` प्रविष्टियाँ भी घोषित करनी चाहिए। इनके बिना, OpenClaw अब भी Plugin लोड कर सकता है, लेकिन कोल्ड-पाथ कॉन्फ़िगरेशन स्कीमा, सेटअप और Control UI सतहें Plugin रनटाइम निष्पादित होने तक चैनल-स्वामित्व वाले विकल्प का स्वरूप नहीं जान सकतीं।

`channelConfigs.<channel-id>.commands.nativeCommandsAutoEnabled` और `nativeSkillsAutoEnabled`, चैनल रनटाइम लोड होने से पहले चलने वाली कमांड कॉन्फ़िगरेशन जाँचों के लिए स्थिर `auto` डिफ़ॉल्ट घोषित कर सकते हैं। बंडल किए गए चैनल अपने अन्य पैकेज-स्वामित्व वाले चैनल कैटलॉग मेटाडेटा के साथ `package.json#openclaw.channel.commands` के माध्यम से भी वही डिफ़ॉल्ट प्रकाशित कर सकते हैं।

```json
{
  "channelConfigs": {
    "matrix": {
      "schema": {
        "type": "object",
        "additionalProperties": false,
        "properties": {
          "homeserverUrl": { "type": "string" }
        }
      },
      "uiHints": {
        "homeserverUrl": {
          "label": "होमसर्वर URL",
          "placeholder": "https://matrix.example.com"
        }
      },
      "label": "Matrix",
      "description": "Matrix होमसर्वर कनेक्शन",
      "commands": {
        "nativeCommandsAutoEnabled": true,
        "nativeSkillsAutoEnabled": true
      },
      "preferOver": ["matrix-legacy"]
    }
  }
}
```

प्रत्येक चैनल प्रविष्टि में ये शामिल हो सकते हैं:

| फ़ील्ड         | प्रकार                     | इसका अर्थ                                                                             |
| ------------- | ------------------------ | ----------------------------------------------------------------------------------------- |
| `schema`      | `object`                 | `channels.<id>` के लिए JSON स्कीमा। प्रत्येक घोषित चैनल कॉन्फ़िगरेशन प्रविष्टि के लिए आवश्यक।         |
| `uiHints`     | `Record<string, object>` | उस चैनल कॉन्फ़िगरेशन अनुभाग के लिए वैकल्पिक UI लेबल/प्लेसहोल्डर/संवेदनशीलता संकेत।          |
| `label`       | `string`                 | रनटाइम मेटाडेटा तैयार न होने पर चयनकर्ता और निरीक्षण सतहों में मर्ज किया गया चैनल लेबल। |
| `description` | `string`                 | निरीक्षण और कैटलॉग सतहों के लिए संक्षिप्त चैनल विवरण।                               |
| `commands`    | `object`                 | रनटाइम-पूर्व कॉन्फ़िगरेशन जाँचों के लिए मूल कमांड और मूल Skill के स्थिर स्वचालित डिफ़ॉल्ट।       |
| `preferOver`  | `string[]`               | पुराने या कम प्राथमिकता वाले Plugin आईडी, जिनसे इस चैनल को चयन सतहों में उच्च स्थान मिलना चाहिए।    |

### किसी अन्य चैनल Plugin को प्रतिस्थापित करना

जब आपका Plugin ऐसे चैनल आईडी का पसंदीदा स्वामी हो जिसे कोई अन्य Plugin भी प्रदान कर सकता है, तब `preferOver` का उपयोग करें। सामान्य स्थितियों में बदला हुआ Plugin आईडी, बंडल किए गए Plugin का स्थान लेने वाला स्वतंत्र Plugin, या कॉन्फ़िगरेशन संगतता के लिए वही चैनल आईडी बनाए रखने वाला अनुरक्षित फ़ोर्क शामिल हैं।

```json
{
  "id": "acme-chat",
  "channels": ["chat"],
  "channelConfigs": {
    "chat": {
      "schema": {
        "type": "object",
        "additionalProperties": false,
        "properties": {
          "webhookUrl": { "type": "string" }
        }
      },
      "preferOver": ["chat"]
    }
  }
}
```

जब `channels.chat` कॉन्फ़िगर होता है, OpenClaw चैनल आईडी और पसंदीदा Plugin आईडी दोनों पर विचार करता है। यदि कम प्राथमिकता वाला Plugin केवल इसलिए चुना गया था क्योंकि वह बंडल किया गया है या डिफ़ॉल्ट रूप से सक्षम है, तो OpenClaw उसे प्रभावी रनटाइम कॉन्फ़िगरेशन में अक्षम कर देता है, ताकि चैनल और उसके टूल का स्वामी एक ही Plugin हो। स्पष्ट उपयोगकर्ता चयन फिर भी प्रभावी रहता है: यदि उपयोगकर्ता दोनों Plugin को स्पष्ट रूप से सक्षम करता है (`plugins.allow` या किसी वास्तविक `plugins.entries` कॉन्फ़िगरेशन के माध्यम से), तो OpenClaw उस चयन को बनाए रखता है और अनुरोधित Plugin सेट को चुपचाप बदलने के बजाय डुप्लिकेट चैनल/टूल निदान रिपोर्ट करता है।

`preferOver` को केवल उन Plugin आईडी तक सीमित रखें जो वास्तव में वही चैनल प्रदान कर सकते हैं। यह कोई सामान्य प्राथमिकता फ़ील्ड नहीं है और उपयोगकर्ता कॉन्फ़िगरेशन कुंजियों का नाम नहीं बदलता।

## modelSupport संदर्भ

जब OpenClaw को Plugin रनटाइम लोड होने से पहले `gpt-5.6-sol` या `claude-sonnet-4.6` जैसी संक्षिप्त मॉडल आईडी से आपके प्रदाता Plugin का अनुमान लगाना चाहिए, तब `modelSupport` का उपयोग करें।

```json
{
  "modelSupport": {
    "modelPrefixes": ["gpt-", "o1", "o3", "o4"],
    "modelPatterns": ["^computer-use-preview"]
  }
}
```

OpenClaw यह वरीयता लागू करता है:

- स्पष्ट `provider/model` संदर्भ स्वामित्व वाले `providers` मैनिफ़ेस्ट मेटाडेटा का उपयोग करते हैं
- `modelPatterns`, `modelPrefixes` पर प्राथमिकता पाते हैं
- यदि एक गैर-बंडल Plugin और एक बंडल Plugin दोनों मेल खाते हैं, तो गैर-बंडल Plugin को प्राथमिकता मिलती है
- शेष अस्पष्टता को तब तक अनदेखा किया जाता है, जब तक उपयोगकर्ता या कॉन्फ़िगरेशन कोई प्रदाता निर्दिष्ट नहीं करता

फ़ील्ड:

| फ़ील्ड           | प्रकार       | इसका अर्थ                                                                   |
| --------------- | ---------- | ------------------------------------------------------------------------------- |
| `modelPrefixes` | `string[]` | संक्षिप्त मॉडल आईडी के विरुद्ध `startsWith` के साथ मिलाए जाने वाले उपसर्ग।                 |
| `modelPatterns` | `string[]` | प्रोफ़ाइल प्रत्यय हटाने के बाद संक्षिप्त मॉडल आईडी से मिलाए जाने वाले रेगेक्स स्रोत। |

`modelPatterns` प्रविष्टियों को `compileSafeRegex` के माध्यम से संकलित किया जाता है, जो नेस्टेड पुनरावृत्ति वाले पैटर्न (उदाहरण के लिए `(a+)+$`) अस्वीकार करता है। सुरक्षा जाँच में विफल पैटर्न को सिंटैक्टिक रूप से अमान्य रेगेक्स की तरह चुपचाप छोड़ दिया जाता है। पैटर्न सरल रखें और नेस्टेड क्वांटिफ़ायर से बचें।

## modelCatalog संदर्भ

जब OpenClaw को Plugin रनटाइम लोड करने से पहले प्रदाता मॉडल मेटाडेटा की जानकारी होनी चाहिए, तब `modelCatalog` का उपयोग करें। यह स्थिर कैटलॉग पंक्तियों, प्रदाता उपनामों, दमन नियमों और खोज मोड के लिए मैनिफ़ेस्ट-स्वामित्व वाला स्रोत है। रनटाइम रीफ़्रेश अब भी प्रदाता रनटाइम कोड की ज़िम्मेदारी है, लेकिन मैनिफ़ेस्ट कोर को बताता है कि रनटाइम कब आवश्यक है।

```json
{
  "providers": ["openai"],
  "modelCatalog": {
    "providers": {
      "openai": {
        "baseUrl": "https://api.openai.com/v1",
        "api": "openai-responses",
        "models": [
          {
            "id": "gpt-5.4",
            "name": "GPT-5.4",
            "input": ["text", "image"],
            "reasoning": true,
            "contextWindow": 256000,
            "maxTokens": 128000,
            "cost": {
              "input": 1.25,
              "output": 10,
              "cacheRead": 0.125
            },
            "status": "available",
            "tags": ["default"]
          }
        ]
      }
    },
    "aliases": {
      "azure-openai-responses": {
        "provider": "openai",
        "api": "azure-openai-responses"
      }
    },
    "suppressions": [
      {
        "provider": "azure-openai-responses",
        "model": "gpt-5.3-codex-spark",
        "reason": "Azure OpenAI Responses पर उपलब्ध नहीं है"
      }
    ],
    "discovery": {
      "openai": "static"
    }
  }
}
```

शीर्ष-स्तरीय फ़ील्ड:

| फ़ील्ड            | प्रकार                                                     | इसका अर्थ                                                                                               |
| ---------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `providers`      | `Record<string, object>`                                 | इस Plugin के स्वामित्व वाली provider ids के लिए कैटलॉग पंक्तियाँ। कुंजियाँ शीर्ष-स्तरीय `providers` में भी दिखाई देनी चाहिए।       |
| `aliases`        | `Record<string, object>`                                 | ऐसे provider उपनाम जिन्हें कैटलॉग या दमन योजना के लिए किसी स्वामित्व वाले provider में रिज़ॉल्व होना चाहिए।              |
| `suppressions`   | `object[]`                                               | किसी अन्य स्रोत की मॉडल पंक्तियाँ, जिन्हें यह Plugin provider-विशिष्ट कारण से दबाता है।                  |
| `discovery`      | `Record<string, "static" \| "refreshable" \| "runtime">` | क्या provider कैटलॉग को मैनिफ़ेस्ट मेटाडेटा से पढ़ा जा सकता है, कैश में रीफ़्रेश किया जा सकता है, या इसके लिए रनटाइम आवश्यक है। |
| `runtimeAugment` | `boolean`                                                | केवल तभी `true` पर सेट करें, जब provider रनटाइम को मैनिफ़ेस्ट/कॉन्फ़िग योजना के बाद कैटलॉग पंक्तियाँ जोड़नी हों।       |

`aliases` मॉडल-कैटलॉग योजना के लिए provider स्वामित्व लुकअप में भाग लेता है। उपनाम लक्ष्य उसी Plugin के स्वामित्व वाले शीर्ष-स्तरीय provider होने चाहिए। जब provider-फ़िल्टर की गई सूची किसी उपनाम का उपयोग करती है, तो OpenClaw provider रनटाइम लोड किए बिना स्वामी मैनिफ़ेस्ट पढ़ सकता है और उपनाम API/आधार URL ओवरराइड लागू कर सकता है। उपनाम बिना फ़िल्टर वाली कैटलॉग सूचियों का विस्तार नहीं करते; व्यापक सूचियाँ केवल स्वामी के कैनोनिकल provider की पंक्तियाँ उत्सर्जित करती हैं।

`suppressions` पुराने provider रनटाइम `suppressBuiltInModel` हुक को प्रतिस्थापित करता है। दमन प्रविष्टियों का पालन केवल तभी किया जाता है, जब provider Plugin के स्वामित्व में हो या स्वामित्व वाले provider को लक्षित करने वाली `modelCatalog.aliases` कुंजी के रूप में घोषित हो। मॉडल रिज़ॉल्यूशन के दौरान रनटाइम दमन हुक अब कॉल नहीं किए जाते।

Provider फ़ील्ड:

| फ़ील्ड                 | प्रकार                     | इसका अर्थ                                                                                                                                                                                                     |
| --------------------- | ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `baseUrl`             | `string`                 | इस provider कैटलॉग के मॉडलों के लिए वैकल्पिक डिफ़ॉल्ट आधार URL।                                                                                                                                                    |
| `api`                 | `ModelApi`               | इस provider कैटलॉग के मॉडलों के लिए वैकल्पिक डिफ़ॉल्ट API अडैप्टर।                                                                                                                                                 |
| `headers`             | `Record<string, string>` | इस provider कैटलॉग पर लागू होने वाले वैकल्पिक स्थिर हेडर।                                                                                                                                                      |
| `defaultUtilityModel` | `string`                 | छोटे आंतरिक उपयोगिता कार्यों (शीर्षक, प्रगति विवरण) के लिए provider द्वारा अनुशंसित वैकल्पिक छोटा मॉडल id। इसका उपयोग तब होता है, जब `agents.defaults.utilityModel` सेट न हो और यह provider एजेंट का प्राथमिक मॉडल प्रदान करता हो। |
| `models`              | `object[]`               | आवश्यक मॉडल पंक्तियाँ। `id` के बिना पंक्तियों को अनदेखा किया जाता है।                                                                                                                                                            |

मॉडल फ़ील्ड:

| फ़ील्ड              | प्रकार                                                           | इसका अर्थ                                                               |
| ------------------ | -------------------------------------------------------------- | --------------------------------------------------------------------------- |
| `id`               | `string`                                                       | `provider/` उपसर्ग के बिना provider-स्थानीय मॉडल id।                    |
| `name`             | `string`                                                       | वैकल्पिक प्रदर्शन नाम।                                                      |
| `api`              | `ModelApi`                                                     | वैकल्पिक प्रति-मॉडल API ओवरराइड।                                            |
| `baseUrl`          | `string`                                                       | वैकल्पिक प्रति-मॉडल आधार URL ओवरराइड।                                       |

| `reasoning`        | `boolean`                                                      | क्या मॉडल तर्क व्यवहार उपलब्ध कराता है।                               |

| `mediaInput`       | `object`                                                       | वैकल्पिक प्रति-मोडैलिटी इनपुट कॉन्फ़िग, फ़िलहाल केवल इमेज।                   |
| `status`           | `"available"` \| `"preview"` \| `"deprecated"` \| `"disabled"` | सूचीकरण स्थिति। केवल तभी दबाएँ, जब पंक्ति बिल्कुल दिखाई नहीं देनी चाहिए।          |

दमन फ़ील्ड:

| फ़ील्ड                      | प्रकार       | इसका अर्थ                                                                                             |
| -------------------------- | ---------- | --------------------------------------------------------------------------------------------------------- |
| `provider`                 | `string`   | दबाई जाने वाली अपस्ट्रीम पंक्ति का provider id। यह इस Plugin के स्वामित्व में होना चाहिए या स्वामित्व वाले उपनाम के रूप में घोषित होना चाहिए। |
| `model`                    | `string`   | दबाया जाने वाला provider-स्थानीय मॉडल id।                                                                      |
| `reason`                   | `string`   | दबाई गई पंक्ति का सीधे अनुरोध किए जाने पर दिखाया जाने वाला वैकल्पिक संदेश।                                     |
| `when.baseUrlHosts`        | `string[]` | दमन लागू होने से पहले आवश्यक प्रभावी provider आधार URL होस्ट की वैकल्पिक सूची।               |
| `when.providerConfigApiIn` | `string[]` | दमन लागू होने से पहले आवश्यक सटीक provider-कॉन्फ़िग `api` मानों की वैकल्पिक सूची।              |

`modelCatalog` में केवल-रनटाइम डेटा न रखें। `static` का उपयोग केवल तभी करें, जब मैनिफ़ेस्ट पंक्तियाँ इतनी पूर्ण हों कि provider-फ़िल्टर की गई सूची और चयनकर्ता सतहें रजिस्ट्री/रनटाइम खोज को छोड़ सकें। `refreshable` का उपयोग तब करें, जब मैनिफ़ेस्ट पंक्तियाँ उपयोगी सूचीबद्ध करने योग्य बीज या पूरक हों, लेकिन बाद में रीफ़्रेश/कैश और पंक्तियाँ जोड़ सके; रीफ़्रेश की जा सकने वाली पंक्तियाँ अपने आप में प्रामाणिक नहीं होतीं। `runtime` का उपयोग तब करें, जब सूची जानने के लिए OpenClaw को provider रनटाइम लोड करना आवश्यक हो।

## modelIdNormalization संदर्भ

provider रनटाइम लोड होने से पहले आवश्यक कम लागत वाली, provider-स्वामित्व वाली मॉडल-id सफ़ाई के लिए `modelIdNormalization` का उपयोग करें। यह छोटे मॉडल नामों, provider-स्थानीय लेगेसी ids और प्रॉक्सी उपसर्ग नियमों जैसे उपनामों को मुख्य मॉडल-चयन तालिकाओं के बजाय स्वामी Plugin मैनिफ़ेस्ट में रखता है।

```json
{
  "providers": ["anthropic", "openrouter"],
  "modelIdNormalization": {
    "providers": {
      "anthropic": {
        "aliases": {
          "sonnet-4.6": "claude-sonnet-4-6"
        }
      },
      "openrouter": {
        "prefixWhenBare": "openrouter"
      }
    }
  }
}
```

Provider फ़ील्ड:

| फ़ील्ड                                | प्रकार                    | इसका अर्थ                                                                             |
| ------------------------------------ | ----------------------- | ----------------------------------------------------------------------------------------- |
| `aliases`                            | `Record<string,string>` | केस-असंवेदी सटीक मॉडल-id उपनाम। मान लिखे गए रूप में ही लौटाए जाते हैं।                  |
| `stripPrefixes`                      | `string[]`              | उपनाम लुकअप से पहले हटाए जाने वाले उपसर्ग, लेगेसी provider/मॉडल दोहराव के लिए उपयोगी।     |
| `prefixWhenBare`                     | `string`                | जब सामान्यीकृत मॉडल id में पहले से `/` न हो, तब जोड़ा जाने वाला उपसर्ग।                  |
| `prefixWhenBareAfterAliasStartsWith` | `object[]`              | उपनाम लुकअप के बाद सशर्त बेयर-id उपसर्ग नियम, जिनकी कुंजी `modelPrefix` और `prefix` होती है। |

## providerEndpoints संदर्भ

ऐसे एंडपॉइंट वर्गीकरण के लिए `providerEndpoints` का उपयोग करें, जिसे सामान्य अनुरोध नीति को provider रनटाइम लोड होने से पहले जानना आवश्यक है। प्रत्येक `endpointClass` का अर्थ अब भी कोर के स्वामित्व में है; Plugin मैनिफ़ेस्ट होस्ट और आधार URL मेटाडेटा के स्वामी हैं।

आधिकारिक रूप से बाह्यीकृत provider Plugin कोर वितरण से बाहर रखे जाते हैं, इसलिए
इंस्टॉल होने तक उनके मैनिफ़ेस्ट दिखाई नहीं देते। उनके `providerEndpoints` को
`scripts/lib/official-external-provider-catalog.json` में भी प्रतिबिंबित किया जाना चाहिए, ताकि
Plugin के बिना भी एंडपॉइंट वर्गीकरण काम करता रहे; एक अनुबंध परीक्षण
इस प्रतिबिंब को लागू करता है।

एंडपॉइंट फ़ील्ड:

| फ़ील्ड                          | प्रकार       | इसका अर्थ                                                                                  |
| ------------------------------ | ---------- | ---------------------------------------------------------------------------------------------- |
| `endpointClass`                | `string`   | ज्ञात कोर एंडपॉइंट वर्ग, जैसे `openrouter`, `moonshot-native`, या `google-vertex`।        |
| `hosts`                        | `string[]` | वे सटीक होस्टनाम जो एंडपॉइंट वर्ग से मैप होते हैं।                                                |
| `hostSuffixes`                 | `string[]` | वे होस्ट प्रत्यय जो एंडपॉइंट वर्ग से मैप होते हैं। केवल डोमेन-प्रत्यय मिलान के लिए `.` उपसर्ग लगाएँ। |
| `baseUrls`                     | `string[]` | वे सटीक सामान्यीकृत HTTP(S) बेस URL जो एंडपॉइंट वर्ग से मैप होते हैं।                             |
| `googleVertexRegion`           | `string`   | सटीक वैश्विक होस्ट के लिए स्थिर Google Vertex क्षेत्र।                                            |
| `googleVertexRegionHostSuffix` | `string`   | Google Vertex क्षेत्र उपसर्ग प्रकट करने के लिए मेल खाने वाले होस्ट से हटाया जाने वाला प्रत्यय।                 |

```json
        }
      }
    }
  }
}
```

| `compatibilityFamily` | `"moonshot"` | साझा अनुरोध सहायकों के लिए वैकल्पिक प्रदाता-परिवार संगतता बकेट।              |
| `openAICompletions`   | `object`     | OpenAI-संगत पूर्णता अनुरोध फ़्लैग, वर्तमान में `supportsStreamingUsage`।       |

## secretProviderIntegrations संदर्भ

जब कोई plugin पुनः उपयोग योग्य SecretRef exec प्रदाता प्रीसेट प्रकाशित कर सकता है, तब `secretProviderIntegrations` का उपयोग करें। OpenClaw plugin रनटाइम लोड होने से पहले इस मेटाडेटा को पढ़ता है, plugin स्वामित्व को `secrets.providers.<alias>.pluginIntegration` में संग्रहीत करता है, और वास्तविक सीक्रेट समाधान को SecretRef रनटाइम पर छोड़ देता है। प्रीसेट केवल बंडल किए गए plugins और प्रबंधित plugin इंस्टॉल रूट से खोजे गए इंस्टॉल किए गए plugins, जैसे git और ClawHub इंस्टॉल, के लिए उपलब्ध कराए जाते हैं।

```json
{
  "secretProviderIntegrations": {
    "secret-store": {
      "providerAlias": "team-secrets",
      "displayName": "Team secrets",
      "source": "exec",
      "command": "${node}",
      "args": ["./bin/resolve-secrets.mjs"]
    }
  }
}
```

मैप कुंजी इंटीग्रेशन आईडी है। यदि `providerAlias` छोड़ा गया है, तो OpenClaw इंटीग्रेशन आईडी को SecretRef प्रदाता उपनाम के रूप में उपयोग करता है। प्रदाता उपनामों को सामान्य SecretRef प्रदाता उपनाम पैटर्न से मेल खाना चाहिए, उदाहरण के लिए `team-secrets` या `onepassword-work`।

जब कोई ऑपरेटर प्रीसेट चुनता है, तो OpenClaw इस प्रकार का प्रदाता संदर्भ लिखता है:

```json
{
  "secrets": {
    "providers": {
      "team-secrets": {
        "source": "exec",
        "pluginIntegration": {
          "pluginId": "acme-secrets",
          "integrationId": "secret-store"
        }
      }
    }
  }
}
```

स्टार्टअप/रीलोड पर, OpenClaw वर्तमान plugin मैनिफ़ेस्ट मेटाडेटा लोड करके, स्वामी plugin के इंस्टॉल और सक्रिय होने की जाँच करके, तथा मैनिफ़ेस्ट से exec कमांड को साकार करके उस प्रदाता का समाधान करता है। plugin को अक्षम करने या हटाने से सक्रिय SecretRefs के लिए प्रदाता निरस्त हो जाता है। जो ऑपरेटर स्वतंत्र exec कॉन्फ़िगरेशन चाहते हैं, वे अब भी मैन्युअल `command`/`args` प्रदाता सीधे लिख सकते हैं।

वर्तमान में केवल `source: "exec"` प्रीसेट समर्थित हैं। `command` को `${node}` होना चाहिए, और `args[0]` को `./` plugin-रूट-सापेक्ष रिज़ॉल्वर स्क्रिप्ट होना चाहिए। OpenClaw स्टार्टअप/रीलोड पर इसे वर्तमान Node निष्पादन योग्य फ़ाइल और plugin के भीतर की निरपेक्ष स्क्रिप्ट पाथ के रूप में साकार करता है। `--require`, `--import`, `--loader`, `--env-file`, `--eval`, और `--print` जैसे Node विकल्प मैनिफ़ेस्ट प्रीसेट अनुबंध का हिस्सा नहीं हैं। जिन ऑपरेटरों को गैर-Node कमांड चाहिए, वे स्वतंत्र मैन्युअल exec प्रदाता सीधे कॉन्फ़िगर कर सकते हैं।

OpenClaw मैनिफ़ेस्ट प्रीसेट के लिए `trustedDirs` को plugin रूट से और `${node}` प्रीसेट के लिए वर्तमान Node निष्पादन योग्य फ़ाइल की डायरेक्टरी से प्राप्त करता है। मैनिफ़ेस्ट में लिखे गए `trustedDirs` अनदेखे किए जाते हैं। `timeoutMs`, `noOutputTimeoutMs`, `maxOutputBytes`, `jsonOnly`, `env`, `passEnv`, और `allowInsecurePath` जैसे अन्य exec प्रदाता विकल्प सामान्य SecretRef exec प्रदाता कॉन्फ़िगरेशन तक यथावत पहुँचते हैं।

## modelPricing संदर्भ

जब किसी प्रदाता को रनटाइम लोड होने से पहले कंट्रोल-प्लेन मूल्य निर्धारण व्यवहार पर नियंत्रण चाहिए, तब `modelPricing` का उपयोग करें। Gateway मूल्य निर्धारण कैश प्रदाता रनटाइम कोड आयात किए बिना इस मेटाडेटा को पढ़ता है।

```json
{
  "providers": ["ollama", "openrouter"],
  "modelPricing": {
    "providers": {
      "ollama": {
        "external": false
      },
      "openrouter": {
        "openRouter": {
          "passthroughProviderModel": true
        },
        "liteLLM": false
      }
    }
  }
}
```

प्रदाता फ़ील्ड:

| फ़ील्ड        | प्रकार              | इसका अर्थ                                                                                      |
| ------------ | ----------------- | -------------------------------------------------------------------------------------------------- |
| `external`   | `boolean`         | स्थानीय/स्वयं-होस्ट किए गए प्रदाताओं के लिए `false` सेट करें, जिन्हें कभी भी OpenRouter या LiteLLM मूल्य निर्धारण प्राप्त नहीं करना चाहिए। |
| `openRouter` | `false \| object` | OpenRouter मूल्य निर्धारण लुकअप मैपिंग। `false` इस प्रदाता के लिए OpenRouter लुकअप अक्षम करता है।           |
| `liteLLM`    | `false \| object` | LiteLLM मूल्य निर्धारण लुकअप मैपिंग। `false` इस प्रदाता के लिए LiteLLM लुकअप अक्षम करता है।                 |

स्रोत फ़ील्ड:

| फ़ील्ड                      | प्रकार               | इसका अर्थ                                                                                                        |
| -------------------------- | ------------------ | -------------------------------------------------------------------------------------------------------------------- |
| `provider`                 | `string`           | जब बाहरी कैटलॉग प्रदाता आईडी OpenClaw प्रदाता आईडी से भिन्न हो, तब उसकी आईडी, उदाहरण के लिए किसी `zai` प्रदाता के लिए `z-ai`। |
| `passthroughProviderModel` | `boolean`          | स्लैश वाले मॉडल आईडी को नेस्टेड प्रदाता/मॉडल संदर्भ मानें, जो OpenRouter जैसे प्रॉक्सी प्रदाताओं के लिए उपयोगी है।       |
| `modelIdTransforms`        | `"version-dots"[]` | अतिरिक्त बाहरी कैटलॉग मॉडल-आईडी प्रकार। `version-dots`, `claude-opus-4.6` जैसे बिंदुयुक्त संस्करण आईडी आज़माता है।            |

### OpenClaw प्रदाता इंडेक्स

OpenClaw प्रदाता इंडेक्स उन प्रदाताओं के लिए OpenClaw-स्वामित्व वाला पूर्वावलोकन मेटाडेटा है, जिनके plugins अभी इंस्टॉल नहीं किए गए हों। यह plugin मैनिफ़ेस्ट का हिस्सा नहीं है। plugin मैनिफ़ेस्ट इंस्टॉल किए गए plugin के लिए प्रामाणिक स्रोत बने रहते हैं। प्रदाता इंडेक्स वह आंतरिक फ़ॉलबैक अनुबंध है जिसका उपयोग भविष्य में इंस्टॉल योग्य प्रदाता और इंस्टॉल-पूर्व मॉडल पिकर सतहें तब करेंगी, जब कोई प्रदाता plugin इंस्टॉल न हो।

कैटलॉग प्रामाणिकता क्रम:

1. उपयोगकर्ता कॉन्फ़िगरेशन।
2. इंस्टॉल किया गया plugin मैनिफ़ेस्ट `modelCatalog`।
3. स्पष्ट रीफ़्रेश से मॉडल कैटलॉग कैश।
4. OpenClaw प्रदाता इंडेक्स पूर्वावलोकन पंक्तियाँ।

प्रदाता इंडेक्स में सीक्रेट, सक्षम स्थिति, रनटाइम हुक, या लाइव खाता-विशिष्ट मॉडल डेटा नहीं होना चाहिए। इसके पूर्वावलोकन कैटलॉग plugin मैनिफ़ेस्ट के समान `modelCatalog` प्रदाता पंक्ति आकार का उपयोग करते हैं, लेकिन इन्हें स्थिर प्रदर्शन मेटाडेटा तक सीमित रहना चाहिए, जब तक कि `api`, `baseUrl`, मूल्य निर्धारण, या संगतता फ़्लैग जैसे रनटाइम अडैप्टर फ़ील्ड जानबूझकर इंस्टॉल किए गए plugin मैनिफ़ेस्ट के साथ संरेखित न रखे जाएँ। लाइव `/models` खोज वाले प्रदाताओं को सामान्य सूचीकरण या ऑनबोर्डिंग में प्रदाता API कॉल करने के बजाय स्पष्ट मॉडल कैटलॉग कैश पाथ के माध्यम से रीफ़्रेश की गई पंक्तियाँ लिखनी चाहिए।

प्रदाता इंडेक्स प्रविष्टियों में उन प्रदाताओं के लिए इंस्टॉल योग्य-plugin मेटाडेटा भी हो सकता है, जिनका plugin कोर से बाहर स्थानांतरित हो गया है या अन्यथा अभी इंस्टॉल नहीं है। यह मेटाडेटा चैनल कैटलॉग पैटर्न को प्रतिबिंबित करता है: पैकेज नाम, npm इंस्टॉल विनिर्देश, अपेक्षित इंटीग्रिटी, और कम लागत वाले प्रमाणीकरण-विकल्प लेबल किसी इंस्टॉल योग्य सेटअप विकल्प को दिखाने के लिए पर्याप्त हैं। plugin इंस्टॉल हो जाने के बाद, उसका मैनिफ़ेस्ट प्रभावी होता है और उस प्रदाता के लिए प्रदाता इंडेक्स प्रविष्टि अनदेखी की जाती है।

`openclaw doctor --fix`, पुराने शीर्ष-स्तरीय मैनिफ़ेस्ट क्षमता कुंजियों के एक छोटे, बंद समूह को `contracts.*` में माइग्रेट करता है: `speechProviders`, `mediaUnderstandingProviders`, `imageGenerationProviders`, और `tools`। इनमें से कोई भी (या कोई अन्य क्षमता सूची) अब शीर्ष-स्तरीय मैनिफ़ेस्ट फ़ील्ड के रूप में नहीं पढ़ी जाती; सामान्य मैनिफ़ेस्ट लोडिंग उन्हें केवल `contracts` के अंतर्गत पहचानती है।

## मैनिफ़ेस्ट बनाम package.json

दोनों फ़ाइलें अलग-अलग कार्य करती हैं:

| फ़ाइल                   | इसका उपयोग करें                                                                                                                       |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `openclaw.plugin.json` | खोज, कॉन्फ़िगरेशन सत्यापन, प्रमाणीकरण-विकल्प मेटाडेटा, और ऐसे UI संकेतों के लिए जो plugin कोड चलने से पहले मौजूद होने चाहिए                         |
| `package.json`         | npm मेटाडेटा, निर्भरता इंस्टॉलेशन, और एंट्रीपॉइंट, इंस्टॉल गेटिंग, सेटअप, या कैटलॉग मेटाडेटा के लिए उपयोग किया जाने वाला `openclaw` ब्लॉक |

यदि यह स्पष्ट न हो कि कोई मेटाडेटा कहाँ होना चाहिए, तो इस नियम का उपयोग करें:

- यदि OpenClaw को plugin कोड लोड करने से पहले इसकी जानकारी होनी चाहिए, तो इसे `openclaw.plugin.json` में रखें
- यदि यह पैकेजिंग, एंट्री फ़ाइलों, या npm इंस्टॉल व्यवहार से संबंधित है, तो इसे `package.json` में रखें

### खोज को प्रभावित करने वाले package.json फ़ील्ड

कुछ प्री-रनटाइम plugin मेटाडेटा जानबूझकर `openclaw.plugin.json` के बजाय `package.json` में `openclaw` ब्लॉक के अंतर्गत रहता है। `openclaw.bundle` और `openclaw.bundle.json`, OpenClaw plugin अनुबंध नहीं हैं; नेटिव plugins को `openclaw.plugin.json` और नीचे दिए गए समर्थित `package.json#openclaw` फ़ील्ड का उपयोग करना चाहिए।

महत्वपूर्ण उदाहरण:

| फ़ील्ड                                                                                      | इसका अर्थ                                                                                                                                                                        |
| ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `openclaw.extensions`                                                                      | नेटिव Plugin एंट्रीपॉइंट घोषित करता है। इसे Plugin पैकेज डायरेक्टरी के भीतर ही रहना चाहिए।                                                                                                   |
| `openclaw.runtimeExtensions`                                                               | इंस्टॉल किए गए पैकेजों के लिए बिल्ट JavaScript रनटाइम एंट्रीपॉइंट घोषित करता है। इसे Plugin पैकेज डायरेक्टरी के भीतर ही रहना चाहिए।                                                                 |
| `openclaw.setupEntry`                                                                      | ऑनबोर्डिंग, स्थगित चैनल स्टार्टअप और केवल-पढ़ने योग्य चैनल स्थिति/SecretRef खोज के दौरान उपयोग किया जाने वाला हल्का, केवल-सेटअप एंट्रीपॉइंट। इसे Plugin पैकेज डायरेक्टरी के भीतर ही रहना चाहिए। |
| `openclaw.runtimeSetupEntry`                                                               | इंस्टॉल किए गए पैकेजों के लिए बिल्ट JavaScript सेटअप एंट्रीपॉइंट घोषित करता है। इसके लिए `setupEntry` आवश्यक है, यह मौजूद होना चाहिए और इसे Plugin पैकेज डायरेक्टरी के भीतर ही रहना चाहिए।                         |
| `openclaw.channel`                                                                         | लेबल, दस्तावेज़ पाथ, उपनाम और चयन संबंधी पाठ जैसा हल्का चैनल कैटलॉग मेटाडेटा।                                                                                                 |
| `openclaw.channel.approvalFlags`                                                           | रनटाइम लोड होने से पहले उपलब्ध क्लोज़्ड अनुमोदन व्यवहार फ़्लैग। `native` का अर्थ है कि चैनल नेटिव अनुमोदन UI और उसी टर्न में समाधान का स्वामी है।                                           |
| `openclaw.channel.commands`                                                                | चैनल रनटाइम लोड होने से पहले कॉन्फ़िगरेशन, ऑडिट और कमांड-सूची सतहों द्वारा उपयोग किया जाने वाला स्थिर नेटिव कमांड और नेटिव skill ऑटो-डिफ़ॉल्ट मेटाडेटा।                                          |
| `openclaw.channel.configuredState`                                                         | हल्का कॉन्फ़िगर की गई स्थिति जाँचकर्ता मेटाडेटा, जो पूरा चैनल रनटाइम लोड किए बिना उत्तर दे सकता है कि "क्या केवल-env सेटअप पहले से मौजूद है?"                                         |
| `openclaw.channel.persistedAuthState`                                                      | हल्का स्थायी-प्रमाणीकरण जाँचकर्ता मेटाडेटा, जो पूरा चैनल रनटाइम लोड किए बिना उत्तर दे सकता है कि "क्या पहले से किसी चीज़ में साइन इन किया गया है?"                                               |
| `openclaw.install.clawhubSpec` / `openclaw.install.npmSpec` / `openclaw.install.localPath` | बंडल किए गए और बाहरी रूप से प्रकाशित plugins के लिए इंस्टॉल/अपडेट संकेत।                                                                                                                   |
| `openclaw.install.defaultChoice`                                                           | एकाधिक इंस्टॉल स्रोत उपलब्ध होने पर पसंदीदा इंस्टॉल पाथ।                                                                                                                  |
| `openclaw.install.minHostVersion`                                                          | न्यूनतम समर्थित OpenClaw होस्ट संस्करण, जिसमें `>=2026.3.22` या `>=2026.5.1-beta.1` जैसा semver न्यूनतम मान उपयोग होता है।                                                                             |
| `openclaw.compat.pluginApi`                                                                | इस पैकेज के लिए आवश्यक न्यूनतम OpenClaw Plugin API रेंज, जिसमें `>=2026.5.27` जैसा semver न्यूनतम मान उपयोग होता है।                                                                                 |
| `openclaw.install.expectedIntegrity`                                                       | `sha512-...` जैसी अपेक्षित npm dist अखंडता स्ट्रिंग; इंस्टॉल और अपडेट प्रवाह प्राप्त आर्टिफ़ैक्ट को इसके विरुद्ध सत्यापित करते हैं।                                                            |
| `openclaw.install.allowInvalidConfigRecovery`                                              | कॉन्फ़िगरेशन अमान्य होने पर बंडल किए गए Plugin को फिर से इंस्टॉल करने के सीमित पुनर्प्राप्ति पाथ की अनुमति देता है।                                                                                                       |
| `openclaw.install.requiredPlatformPackages`                                                | npm पैकेज उपनाम, जिन्हें उनकी lockfile प्लेटफ़ॉर्म बाधाएँ वर्तमान होस्ट से मेल खाने पर साकार होना चाहिए।                                                                           |
| `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`                          | सुनना शुरू होने से पहले सेटअप-रनटाइम चैनल सतहों को लोड होने देता है, फिर पूरे कॉन्फ़िगर किए गए चैनल Plugin को सुनना शुरू होने के बाद सक्रियण तक स्थगित करता है।                                                 |

मैनिफ़ेस्ट मेटाडेटा तय करता है कि रनटाइम लोड होने से पहले ऑनबोर्डिंग में कौन-से प्रदाता/चैनल/सेटअप विकल्प दिखाई दें। जब उपयोगकर्ता इनमें से कोई विकल्प चुनता है, तो `package.json#openclaw.install` ऑनबोर्डिंग को बताता है कि उस Plugin को कैसे प्राप्त या सक्षम करना है। इंस्टॉल संकेतों को `openclaw.plugin.json` में न ले जाएँ।

गैर-बंडल Plugin स्रोतों के लिए इंस्टॉल और मैनिफ़ेस्ट रजिस्ट्री लोडिंग के दौरान `openclaw.install.minHostVersion` लागू किया जाता है। अमान्य मान अस्वीकार कर दिए जाते हैं; अधिक नए लेकिन मान्य मान पुराने होस्ट पर बाहरी plugins को छोड़ देते हैं। बंडल किए गए स्रोत plugins को होस्ट चेकआउट के साथ समान संस्करण वाला माना जाता है।

`openclaw.install.requiredPlatformPackages` उन npm पैकेजों के लिए है जो वैकल्पिक, प्लेटफ़ॉर्म-विशिष्ट उपनामों के माध्यम से आवश्यक नेटिव बाइनरी उपलब्ध कराते हैं। प्रत्येक समर्थित प्लेटफ़ॉर्म उपनाम के लिए केवल npm पैकेज का नाम सूचीबद्ध करें। npm इंस्टॉल के दौरान OpenClaw केवल उस घोषित उपनाम को सत्यापित करता है जिसकी lockfile बाधाएँ वर्तमान होस्ट से मेल खाती हैं। यदि npm सफलता की सूचना देता है लेकिन उस उपनाम को छोड़ देता है, तो OpenClaw नए कैश के साथ एक बार पुनः प्रयास करता है और उपनाम फिर भी अनुपस्थित होने पर इंस्टॉल को वापस ले लेता है।

गैर-बंडल Plugin स्रोतों के लिए पैकेज इंस्टॉल के दौरान `openclaw.compat.pluginApi` लागू किया जाता है। इसे उस OpenClaw Plugin SDK/रनटाइम API के न्यूनतम मान के लिए उपयोग करें जिसके विरुद्ध पैकेज बनाया गया था। जब किसी Plugin पैकेज को नया API चाहिए लेकिन वह अन्य प्रवाहों के लिए कम इंस्टॉल संकेत बनाए रखता है, तो यह `minHostVersion` से अधिक सख्त हो सकता है। आधिकारिक OpenClaw रिलीज़ सिंक डिफ़ॉल्ट रूप से मौजूदा आधिकारिक Plugin API न्यूनतम मानों को OpenClaw रिलीज़ संस्करण तक बढ़ाता है, लेकिन यदि पैकेज जानबूझकर पुराने होस्ट का समर्थन करता है, तो केवल-Plugin रिलीज़ कम न्यूनतम मान बनाए रख सकती हैं। केवल पैकेज संस्करण को संगतता अनुबंध के रूप में उपयोग न करें। `peerDependencies.openclaw` npm पैकेज मेटाडेटा ही रहता है; OpenClaw इंस्टॉल संगतता निर्णयों के लिए `openclaw.compat.pluginApi` अनुबंध का उपयोग करता है।

जब Plugin ClawHub पर प्रकाशित हो, तो आधिकारिक माँग-पर-इंस्टॉल मेटाडेटा में `clawhubSpec` का उपयोग होना चाहिए; ऑनबोर्डिंग इसे पसंदीदा दूरस्थ स्रोत मानता है और इंस्टॉल के बाद ClawHub आर्टिफ़ैक्ट तथ्य दर्ज करता है। जो पैकेज अभी तक ClawHub पर नहीं आए हैं, उनके लिए `npmSpec` संगतता फ़ॉलबैक बना रहता है।

सटीक npm संस्करण पिनिंग पहले से `npmSpec` में रहती है, उदाहरण के लिए `"npmSpec": "@wecom/wecom-openclaw-plugin@1.2.3"`। आधिकारिक बाहरी कैटलॉग प्रविष्टियों को सटीक स्पेसिफ़िकेशन के साथ `expectedIntegrity` जोड़ना चाहिए, ताकि प्राप्त npm आर्टिफ़ैक्ट पिन की गई रिलीज़ से मेल न खाने पर अपडेट प्रवाह विफल होकर बंद हो जाएँ। संगतता के लिए इंटरैक्टिव ऑनबोर्डिंग अभी भी विश्वसनीय रजिस्ट्री npm स्पेसिफ़िकेशन उपलब्ध कराता है, जिनमें केवल पैकेज नाम और dist-tags शामिल हैं। कैटलॉग निदान सटीक, फ़्लोटिंग, अखंडता-पिन किए गए, अखंडता-विहीन, पैकेज-नाम बेमेल और अमान्य डिफ़ॉल्ट-विकल्प स्रोतों में अंतर कर सकता है। जब `expectedIntegrity` मौजूद हो लेकिन ऐसा कोई मान्य npm स्रोत न हो जिसे वह पिन कर सके, तब भी वे चेतावनी देते हैं। जब `expectedIntegrity` मौजूद हो, तो इंस्टॉल/अपडेट प्रवाह इसे लागू करते हैं; इसे छोड़ दिए जाने पर रजिस्ट्री रिज़ॉल्यूशन बिना अखंडता पिन के दर्ज किया जाता है।

जब स्थिति, चैनल सूची या SecretRef स्कैन को पूरा रनटाइम लोड किए बिना कॉन्फ़िगर किए गए खातों की पहचान करनी हो, तो चैनल plugins को `openclaw.setupEntry` प्रदान करना चाहिए। सेटअप एंट्री को चैनल मेटाडेटा के साथ सेटअप-सुरक्षित कॉन्फ़िगरेशन, स्थिति और सीक्रेट अडैप्टर उजागर करने चाहिए; नेटवर्क क्लाइंट, Gateway लिसनर और ट्रांसपोर्ट रनटाइम को मुख्य एक्सटेंशन एंट्रीपॉइंट में रखें।

रनटाइम एंट्रीपॉइंट फ़ील्ड, स्रोत एंट्रीपॉइंट फ़ील्ड के लिए पैकेज-सीमा जाँचों को ओवरराइड नहीं करते। उदाहरण के लिए, `openclaw.runtimeExtensions` सीमा से बाहर जाने वाले `openclaw.extensions` पाथ को लोड करने योग्य नहीं बना सकता।

`openclaw.install.allowInvalidConfigRecovery` जानबूझकर सीमित है। यह किसी भी मनमाने टूटे हुए कॉन्फ़िगरेशन को इंस्टॉल करने योग्य नहीं बनाता। वर्तमान में यह इंस्टॉल प्रवाहों को केवल बंडल किए गए Plugin के कुछ विशिष्ट पुराने अपग्रेड विफलताओं से उबरने देता है, जैसे बंडल किए गए Plugin का अनुपस्थित पाथ या उसी बंडल किए गए Plugin के लिए पुरानी `channels.<id>` प्रविष्टि। असंबंधित कॉन्फ़िगरेशन त्रुटियाँ अब भी इंस्टॉल को रोकती हैं और ऑपरेटरों को `openclaw doctor --fix` पर भेजती हैं।

`openclaw.channel.persistedAuthState` एक छोटे जाँचकर्ता मॉड्यूल के लिए पैकेज मेटाडेटा है:

```json
{
  "openclaw": {
    "channel": {
      "id": "whatsapp",
      "persistedAuthState": {
        "specifier": "./auth-presence",
        "exportName": "hasAnyWhatsAppAuth"
      }
    }
  }
}
```

जब पूरा चैनल Plugin लोड होने से पहले सेटअप, doctor, स्थिति या केवल-पढ़ने योग्य उपस्थिति प्रवाहों को सस्ती हाँ/नहीं प्रमाणीकरण जाँच चाहिए, तब इसका उपयोग करें। स्थायी प्रमाणीकरण स्थिति, कॉन्फ़िगर की गई चैनल स्थिति नहीं है: plugins को स्वतः सक्षम करने, रनटाइम निर्भरताओं की मरम्मत करने या यह तय करने के लिए कि चैनल रनटाइम लोड होना चाहिए या नहीं, इस मेटाडेटा का उपयोग न करें। लक्ष्य एक्सपोर्ट एक छोटा फ़ंक्शन होना चाहिए जो केवल स्थायी स्थिति पढ़े; इसे पूरे चैनल रनटाइम बैरल के माध्यम से रूट न करें।

`openclaw.channel.configuredState` सस्ती कॉन्फ़िगर की गई जाँचों का समर्थन करता है। जब पर्यावरण चर पर्याप्त हों, तो घोषणात्मक env मेटाडेटा को प्राथमिकता दें:

```json
{
  "openclaw": {
    "channel": {
      "id": "telegram",
      "configuredState": {
        "env": {
          "allOf": ["TELEGRAM_BOT_TOKEN"]
        }
      }
    }
  }
}
```

जब प्रत्येक सूचीबद्ध चर आवश्यक हो, तब `env.allOf` का उपयोग करें और जब कोई भी एक गैर-रिक्त चर पर्याप्त हो, तब `env.anyOf` का उपयोग करें। यदि किसी छोटे गैर-रनटाइम जाँच को पर्यावरण मेटाडेटा से अधिक चाहिए, तो `persistedAuthState` के लिए दिखाए अनुसार `specifier` के साथ `exportName` का उपयोग करें; जब `env` मौजूद हो, तो OpenClaw उस मॉड्यूल को लोड किए बिना इसका उपयोग करता है। यदि जाँच को पूरे कॉन्फ़िगरेशन रिज़ॉल्यूशन या वास्तविक चैनल रनटाइम की आवश्यकता हो, तो उस लॉजिक को Plugin के `config.hasConfiguredState` हुक में ही रखें।

## खोज प्राथमिकता (डुप्लिकेट Plugin आईडी)

OpenClaw तीन रूट से plugins खोजता है, जिनकी जाँच इस क्रम में होती है: OpenClaw के साथ भेजे गए बंडल plugins, वैश्विक इंस्टॉल रूट (`~/.openclaw/extensions`) और वर्तमान वर्कस्पेस रूट (`<workspace>/.openclaw/extensions`), साथ ही कोई भी स्पष्ट `plugins.load.paths` प्रविष्टियाँ।

यदि दो खोजों का `id` समान हो, तो केवल **सर्वोच्च-प्राथमिकता** वाला मैनिफ़ेस्ट रखा जाता है; कम प्राथमिकता वाले डुप्लिकेट को उसके साथ लोड करने के बजाय हटा दिया जाता है। सर्वोच्च से निम्नतम प्राथमिकता:

1. **कॉन्फ़िगरेशन द्वारा चयनित** — `plugins.entries.<id>` में स्पष्ट रूप से पिन किया गया पाथ
2. **ट्रैक किए गए इंस्टॉल रिकॉर्ड से मेल खाने वाला वैश्विक इंस्टॉल** — `openclaw plugin install`/`openclaw plugin update` के माध्यम से इंस्टॉल किया गया ऐसा Plugin, जिसे OpenClaw की इंस्टॉल ट्रैकिंग उसी आईडी के लिए पहचानती है, भले ही वह आईडी किसी बंडल किए गए Plugin की भी हो
3. **बंडल किया गया** — OpenClaw के साथ भेजे गए plugins
4. **वर्कस्पेस** — वर्तमान वर्कस्पेस के सापेक्ष खोजे गए plugins
5. कोई अन्य खोजा गया उम्मीदवार

निहितार्थ:

- वर्कस्पेस या ग्लोबल रूट में अनट्रैक्ड पड़ी किसी बंडल किए गए Plugin की फोर्क की हुई या पुरानी प्रति, बंडल किए गए बिल्ड को शैडो नहीं करेगी।
- किसी बंडल किए गए Plugin को ओवरराइड करने के लिए, या तो उस आईडी के लिए `openclaw plugin install` चलाएँ ताकि ट्रैक किया गया ग्लोबल इंस्टॉल बंडल की गई प्रति से उच्च प्राथमिकता पाए, या `plugins.entries.<id>` के माध्यम से कोई विशिष्ट पाथ पिन करें ताकि कॉन्फ़िगरेशन द्वारा चुनी गई प्राथमिकता के कारण उसे वरीयता मिले।
- डुप्लिकेट प्रतियों को हटाए जाने की जानकारी लॉग की जाती है, ताकि Doctor और स्टार्टअप डायग्नोस्टिक्स त्यागी गई प्रति की ओर संकेत कर सकें।
- कॉन्फ़िगरेशन द्वारा चुने गए डुप्लिकेट ओवरराइड को डायग्नोस्टिक्स में स्पष्ट ओवरराइड के रूप में लिखा जाता है, लेकिन चेतावनी फिर भी दी जाती है ताकि पुराने फोर्क और अनजाने शैडो दिखाई देते रहें।

## JSON Schema की आवश्यकताएँ

- **हर Plugin के साथ JSON Schema देना आवश्यक है**, भले ही वह कोई कॉन्फ़िगरेशन स्वीकार न करता हो।
- खाली स्कीमा स्वीकार्य है (उदाहरण के लिए, `{ "type": "object", "additionalProperties": false }`)।
- स्कीमा का सत्यापन कॉन्फ़िगरेशन पढ़ते/लिखते समय किया जाता है, रनटाइम पर नहीं।
- किसी बंडल किए गए Plugin को नई कॉन्फ़िगरेशन कुंजियों के साथ विस्तारित या फोर्क करते समय, उसी समय उस Plugin के `openclaw.plugin.json` `configSchema` को भी अपडेट करें। बंडल किए गए Plugin के स्कीमा सख्त होते हैं, इसलिए `configSchema.properties` में `myNewKey` जोड़े बिना उपयोगकर्ता कॉन्फ़िगरेशन में `plugins.entries.<id>.config.myNewKey` जोड़ने पर Plugin रनटाइम लोड होने से पहले ही उसे अस्वीकार कर दिया जाएगा।

स्कीमा विस्तार का उदाहरण:

```json
{
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "myNewKey": {
        "type": "string"
      }
    }
  }
}
```

## सत्यापन व्यवहार

- अज्ञात `channels.*` कुंजियाँ **त्रुटियाँ** हैं, जब तक कि चैनल आईडी किसी Plugin मैनिफ़ेस्ट द्वारा घोषित न हो। यदि वही आईडी `plugins.allow`, `plugins.entries`, या `plugins.installs` (ऐसा Plugin जिसका संदर्भ मौजूद है, लेकिन जो फ़िलहाल खोजने योग्य नहीं है) में भी दिखाई देती है, तो OpenClaw इसे घटाकर **चेतावनी** कर देता है।
- अज्ञात Plugin आईडी का संदर्भ देने वाले `plugins.entries.<id>`, `plugins.allow`, और `plugins.deny` **चेतावनियाँ** ("पुरानी कॉन्फ़िगरेशन प्रविष्टि को अनदेखा किया गया") हैं, त्रुटियाँ नहीं, ताकि अपग्रेड और हटाए गए/पुनर्नामित Plugin Gateway का स्टार्टअप न रोकें।
- अज्ञात Plugin आईडी का संदर्भ देने वाला `plugins.slots.memory` एक **त्रुटि** है, सिवाय ज्ञात आधिकारिक बाहरी Plugin `memory-lancedb` के, जिसके लिए इसके बजाय चेतावनी दी जाती है।
- यदि कोई Plugin इंस्टॉल है, लेकिन उसका मैनिफ़ेस्ट या स्कीमा खराब अथवा अनुपलब्ध है, तो सत्यापन विफल हो जाता है और Doctor Plugin की त्रुटि की रिपोर्ट करता है।
- यदि Plugin का कॉन्फ़िगरेशन मौजूद है, लेकिन Plugin **अक्षम** है, तो कॉन्फ़िगरेशन रखा जाता है और Doctor + लॉग में **चेतावनी** दिखाई जाती है।

पूर्ण `plugins.*` स्कीमा के लिए [कॉन्फ़िगरेशन संदर्भ](/hi/gateway/configuration) देखें।

## टिप्पणियाँ

- स्थानीय फ़ाइल-सिस्टम लोड सहित, **नेटिव OpenClaw plugins के लिए मैनिफ़ेस्ट आवश्यक है**। रनटाइम अब भी Plugin मॉड्यूल को अलग से लोड करता है; मैनिफ़ेस्ट केवल खोज + सत्यापन के लिए है।
- नेटिव मैनिफ़ेस्ट को JSON5 से पार्स किया जाता है, इसलिए टिप्पणियाँ, अंतिम अल्पविराम और उद्धरण-चिह्न रहित कुंजियाँ स्वीकार की जाती हैं, बशर्ते अंतिम मान फिर भी एक ऑब्जेक्ट हो।
- मैनिफ़ेस्ट लोडर केवल दस्तावेज़ीकृत मैनिफ़ेस्ट फ़ील्ड पढ़ता है। कस्टम शीर्ष-स्तरीय कुंजियों से बचें।
- यदि किसी Plugin को उनकी आवश्यकता नहीं है, तो `channels`, `providers`, `cliBackends`, और `skills` सभी छोड़े जा सकते हैं।
- `providerCatalogEntry` को हल्का रखना आवश्यक है और इसे व्यापक रनटाइम कोड आयात नहीं करना चाहिए; इसका उपयोग स्थिर प्रोवाइडर कैटलॉग मेटाडेटा या सीमित डिस्कवरी डिस्क्रिप्टर के लिए करें, अनुरोध-समय निष्पादन के लिए नहीं।
- विशिष्ट Plugin प्रकारों का चयन `plugins.slots.*` के माध्यम से किया जाता है: `plugins.slots.memory` के माध्यम से `kind: "memory"` (डिफ़ॉल्ट `memory-core`), `plugins.slots.contextEngine` के माध्यम से `kind: "context-engine"` (डिफ़ॉल्ट `legacy`)।
- इस मैनिफ़ेस्ट में विशिष्ट Plugin प्रकार घोषित करें। रनटाइम-एंट्री `OpenClawPluginDefinition.kind` अप्रचलित है और केवल पुराने plugins के लिए संगतता फ़ॉलबैक के रूप में बनी हुई है।
- एनवायरनमेंट-वेरिएबल मेटाडेटा (`setup.providers[].envVars`, अप्रचलित `providerAuthEnvVars`, और `channelEnvVars`) केवल घोषणात्मक है। किसी एनवायरनमेंट वेरिएबल को कॉन्फ़िगर किया हुआ मानने से पहले, स्थिति, ऑडिट, Cron डिलीवरी सत्यापन और अन्य केवल-पढ़ने योग्य सतहों पर भी Plugin की विश्वसनीयता और प्रभावी सक्रियण नीति लागू होती है।
- प्रोवाइडर कोड की आवश्यकता वाले रनटाइम विज़ार्ड मेटाडेटा के लिए, [प्रोवाइडर रनटाइम हुक](/hi/plugins/architecture-internals#provider-runtime-hooks) देखें।
- यदि आपका Plugin नेटिव मॉड्यूल पर निर्भर है, तो बिल्ड चरणों और पैकेज-मैनेजर की किसी भी अनुमत-सूची आवश्यकता को दस्तावेज़ीकृत करें (उदाहरण के लिए, pnpm `allow-build-scripts` + `pnpm rebuild <package>`)।
