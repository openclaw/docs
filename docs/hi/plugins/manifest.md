यह पृष्ठ **मूल OpenClaw plugin मैनिफ़ेस्ट**, `openclaw.plugin.json`, को कवर करता है। संगत बंडल लेआउट (Codex, Claude, Cursor) के लिए, [Plugin बंडल](/hi/plugins/bundles) देखें।

संगत बंडल प्रारूप इसके बजाय अपनी स्वयं की मैनिफ़ेस्ट फ़ाइलों का उपयोग करते हैं:

- Codex बंडल: `.codex-plugin/plugin.json`
- Claude बंडल: `.claude-plugin/plugin.json`, या बिना मैनिफ़ेस्ट वाला डिफ़ॉल्ट Claude कंपोनेंट लेआउट
- Cursor बंडल: `.cursor-plugin/plugin.json`

OpenClaw उन लेआउट का स्वतः पता लगाता है, लेकिन नीचे दिए गए `openclaw.plugin.json` स्कीमा के अनुसार उन्हें सत्यापित नहीं करता। किसी संगत बंडल के लिए, जब लेआउट OpenClaw की रनटाइम अपेक्षाओं से मेल खाता है, तब OpenClaw बंडल मेटाडेटा, घोषित स्किल रूट, Claude कमांड रूट, Claude `settings.json` डिफ़ॉल्ट, Claude LSP डिफ़ॉल्ट और समर्थित हुक पैक पढ़ता है।

प्रत्येक मूल OpenClaw plugin को **plugin रूट** में `openclaw.plugin.json` **अवश्य** शामिल करना चाहिए। OpenClaw इसे **plugin कोड निष्पादित किए बिना** कॉन्फ़िगरेशन सत्यापित करने के लिए पढ़ता है। अनुपस्थित या अमान्य मैनिफ़ेस्ट कॉन्फ़िगरेशन सत्यापन को अवरुद्ध करता है और इसे plugin त्रुटि माना जाता है।

संपूर्ण plugin सिस्टम गाइड के लिए [Plugins](/hi/tools/plugin) और मूल क्षमता मॉडल तथा वर्तमान बाहरी-संगतता मार्गदर्शन के लिए [क्षमता मॉडल](/hi/plugins/architecture#public-capability-model) देखें।

## यह फ़ाइल क्या करती है

`openclaw.plugin.json` वह मेटाडेटा है जिसे OpenClaw **आपका plugin कोड लोड करने से पहले** पढ़ता है। इसमें मौजूद हर चीज़ इतनी हल्की होनी चाहिए कि plugin रनटाइम शुरू किए बिना उसका निरीक्षण किया जा सके।

**इसका उपयोग इनके लिए करें:**

- plugin पहचान, कॉन्फ़िगरेशन सत्यापन और कॉन्फ़िगरेशन UI संकेत
- प्रमाणीकरण, ऑनबोर्डिंग और सेटअप मेटाडेटा (उपनाम, स्वतः-सक्षम करना, प्रोवाइडर एनवायरनमेंट वेरिएबल और प्रमाणीकरण विकल्प)
- कंट्रोल-प्लेन सतहों के लिए सक्रियण संकेत
- संक्षिप्त मॉडल-फ़ैमिली स्वामित्व
- स्थिर क्षमता-स्वामित्व स्नैपशॉट (`contracts`)
- QA रनर मेटाडेटा जिसका साझा `openclaw qa` होस्ट निरीक्षण कर सकता है
- चैनल-विशिष्ट कॉन्फ़िगरेशन मेटाडेटा, जिसे कैटलॉग और सत्यापन सतहों में मर्ज किया जाता है

**इसका उपयोग इनके लिए न करें:** रनटाइम व्यवहार पंजीकृत करना, कोड एंट्रीपॉइंट घोषित करना या npm इंस्टॉल मेटाडेटा। ये आपके plugin कोड और `package.json` में होने चाहिए।

## कैटलॉग संदर्भ

`catalog` Plugin ब्राउज़र को वैकल्पिक प्रदर्शन संकेत प्रदान करता है। होस्ट इन संकेतों को अनदेखा कर सकते हैं। ये कभी भी Plugin को इंस्टॉल या सक्षम नहीं करते और न ही इसके रनटाइम व्यवहार या विश्वास स्तर को बदलते हैं।

```json
{
  "catalog": {
    "featured": true,
    "order": 10
  }
}
```

| फ़ील्ड      | प्रकार      | इसका अर्थ                                                              |
| ---------- | --------- | -------------------------------------------------------------------------- |
| `featured` | `boolean` | क्या कैटलॉग सतहों पर इस Plugin को प्रमुखता से दिखाया जाना चाहिए।                       |
| `order`    | `number`  | चयनित Plugins के बीच आरोही प्रदर्शन संकेत; कम मान पहले दिखाई देते हैं। |

## जनरेशन प्रदाता मेटाडेटा संदर्भ

जनरेशन प्रदाता मेटाडेटा फ़ील्ड, संबंधित `contracts.*GenerationProviders` सूची में घोषित प्रदाताओं के स्थिर प्रमाणीकरण संकेतों का वर्णन करते हैं। OpenClaw इन फ़ील्ड को प्रदाता रनटाइम लोड होने से पहले पढ़ता है, ताकि मुख्य टूल प्रत्येक प्रदाता Plugin को आयात किए बिना यह तय कर सकें कि कोई जनरेशन प्रदाता उपलब्ध है या नहीं।

इन फ़ील्ड का उपयोग केवल कम लागत वाले, घोषणात्मक तथ्यों के लिए करें। ट्रांसपोर्ट, अनुरोध रूपांतरण, टोकन रीफ़्रेश, क्रेडेंशियल सत्यापन और वास्तविक जनरेशन व्यवहार Plugin रनटाइम में रहते हैं।

```json
{
  "contracts": {
    "imageGenerationProviders": ["example-image"]
  },
  "imageGenerationProviderMetadata": {
    "example-image": {
      "aliases": ["example-image-oauth"],
      "authProviders": ["example-image"],
      "configSignals": [
        {
          "rootPath": "plugins.entries.example-image.config",
          "overlayPath": "image",
          "mode": {
            "path": "mode",
            "default": "local",
            "allowed": ["local"]
          },
          "requiredAny": ["workflow", "workflowPath"],
          "required": ["promptNodeId"]
        }
      ],
      "authSignals": [
        {
          "provider": "example-image"
        },
        {
          "provider": "example-image-oauth",
          "providerBaseUrl": {
            "provider": "example-image",
            "defaultBaseUrl": "https://api.example.com/v1",
            "allowedBaseUrls": ["https://api.example.com/v1"]
          }
        }
      ]
    }
  }
}
```

प्रत्येक मेटाडेटा प्रविष्टि निम्न का समर्थन करती है:

| फ़ील्ड                  | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                       |
| ---------------------- | -------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `aliases`              | नहीं       | `string[]` | अतिरिक्त प्रदाता आईडी, जिन्हें जनरेशन प्रदाता के लिए स्थिर प्रमाणीकरण उपनाम माना जाना चाहिए।                                                       |
| `authProviders`        | नहीं       | `string[]` | वे प्रदाता आईडी, जिनकी कॉन्फ़िगर की गई प्रमाणीकरण प्रोफ़ाइल को इस जनरेशन प्रदाता के लिए प्रमाणीकरण माना जाना चाहिए।                                                      |
| `configSignals`        | नहीं       | `object[]` | स्थानीय या स्वयं-होस्ट किए गए प्रदाताओं के लिए कम लागत वाले, केवल-कॉन्फ़िग उपलब्धता संकेत, जिन्हें प्रमाणीकरण प्रोफ़ाइल या पर्यावरण चर के बिना कॉन्फ़िगर किया जा सकता है।                 |
| `authSignals`          | नहीं       | `object[]` | स्पष्ट प्रमाणीकरण संकेत। इनके मौजूद होने पर ये प्रदाता आईडी, `aliases` और `authProviders` से प्राप्त डिफ़ॉल्ट संकेत समूह को प्रतिस्थापित करते हैं।                     |
| `referenceAudioInputs` | नहीं       | `boolean`  | केवल वीडियो जनरेशन। जब प्रदाता संदर्भ ऑडियो एसेट स्वीकार करता हो, तो इसे `true` पर सेट करें; अन्यथा `video_generate` ऑडियो संदर्भ पैरामीटर छिपा देता है। |

प्रत्येक `configSignals` प्रविष्टि निम्न का समर्थन करती है:

| फ़ील्ड            | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                                                             |
| ---------------- | -------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `rootPath`       | हाँ      | `string`   | जाँच किए जाने वाले Plugin-स्वामित्व वाले कॉन्फ़िग ऑब्जेक्ट का डॉट पथ, उदाहरण के लिए `plugins.entries.example.config`।                                                                                      |
| `overlayPath`    | नहीं       | `string`   | रूट कॉन्फ़िग के भीतर डॉट पथ, जिसका ऑब्जेक्ट संकेत का मूल्यांकन करने से पहले रूट ऑब्जेक्ट पर ओवरले होना चाहिए। इसका उपयोग क्षमता-विशिष्ट कॉन्फ़िग, जैसे `image`, `video` या `music`, के लिए करें।   |
| `overlayMapPath` | नहीं       | `string`   | रूट कॉन्फ़िग के भीतर डॉट पथ, जिसके प्रत्येक ऑब्जेक्ट मान को रूट ऑब्जेक्ट पर ओवरले होना चाहिए। इसका उपयोग `accounts` जैसे नामित खाता मैप के लिए करें, जहाँ कोई भी कॉन्फ़िगर किया गया खाता पात्र होना चाहिए। |
| `required`       | नहीं       | `string[]` | प्रभावी कॉन्फ़िग के भीतर वे डॉट पथ, जिनमें कॉन्फ़िगर किए गए मान होना आवश्यक है। स्ट्रिंग रिक्त नहीं होनी चाहिए; ऑब्जेक्ट और ऐरे खाली नहीं होने चाहिए।                                                  |
| `requiredAny`    | नहीं       | `string[]` | प्रभावी कॉन्फ़िग के भीतर वे डॉट पथ, जिनमें से कम-से-कम एक में कॉन्फ़िगर किया गया मान होना आवश्यक है।                                                                                                    |
| `mode`           | नहीं       | `object`   | प्रभावी कॉन्फ़िग के भीतर वैकल्पिक स्ट्रिंग मोड गार्ड। इसका उपयोग तब करें, जब केवल-कॉन्फ़िग उपलब्धता केवल एक मोड पर लागू होती हो।                                                                  |

प्रत्येक `mode` गार्ड निम्न का समर्थन करता है:

| फ़ील्ड        | आवश्यक | प्रकार       | इसका अर्थ                                                                      |
| ------------ | -------- | ---------- | ---------------------------------------------------------------------------------- |
| `path`       | नहीं       | `string`   | प्रभावी कॉन्फ़िग के भीतर डॉट पथ। डिफ़ॉल्ट मान `mode` है।                          |
| `default`    | नहीं       | `string`   | कॉन्फ़िग द्वारा पथ छोड़े जाने पर उपयोग किया जाने वाला मोड मान।                                  |
| `allowed`    | नहीं       | `string[]` | यदि मौजूद हो, तो संकेत केवल तभी सफल होता है, जब प्रभावी मोड इन मानों में से एक हो। |
| `disallowed` | नहीं       | `string[]` | यदि मौजूद हो, तो प्रभावी मोड इन मानों में से एक होने पर संकेत विफल हो जाता है।       |

प्रत्येक `authSignals` प्रविष्टि निम्न का समर्थन करती है:

| फ़ील्ड             | आवश्यक | प्रकार     | इसका अर्थ                                                                                                                                                                 |
| ----------------- | -------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `provider`        | हाँ      | `string` | कॉन्फ़िगर की गई प्रमाणीकरण प्रोफ़ाइल में जाँचने के लिए प्रदाता आईडी।                                                                                                                             |
| `providerBaseUrl` | नहीं       | `object` | वैकल्पिक गार्ड, जिसके कारण संकेत केवल तभी गिना जाता है, जब संदर्भित कॉन्फ़िगर किया गया प्रदाता अनुमत बेस URL का उपयोग करता हो। इसका उपयोग तब करें, जब कोई प्रमाणीकरण उपनाम केवल कुछ API के लिए मान्य हो। |

प्रत्येक `providerBaseUrl` गार्ड निम्न का समर्थन करता है:

| फ़ील्ड             | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                        |
| ----------------- | -------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `provider`        | हाँ      | `string`   | वह प्रदाता कॉन्फ़िग आईडी, जिसके `baseUrl` की जाँच की जानी चाहिए।                                                                                                |
| `defaultBaseUrl`  | नहीं       | `string`   | प्रदाता कॉन्फ़िग द्वारा `baseUrl` छोड़े जाने पर मान लिया जाने वाला बेस URL।                                                                                         |
| `allowedBaseUrls` | हाँ      | `string[]` | इस प्रमाणीकरण संकेत के लिए अनुमत बेस URL। कॉन्फ़िगर किया गया या डिफ़ॉल्ट बेस URL इन सामान्यीकृत मानों में से किसी से मेल न खाने पर संकेत को अनदेखा कर दिया जाता है। |

## टूल मेटाडेटा संदर्भ

`toolMetadata` जनरेशन प्रदाता मेटाडेटा के समान `configSignals` और `authSignals` आकारों का उपयोग करता है, जिन्हें टूल नाम से कुंजीबद्ध किया जाता है। `contracts.tools` स्वामित्व घोषित करता है। `toolMetadata` कम लागत वाला उपलब्धता प्रमाण घोषित करता है, ताकि OpenClaw केवल उसके टूल फ़ैक्टरी से `null` लौटवाने के लिए Plugin रनटाइम आयात करने से बच सके।

```json
{
  "setup": {
    "providers": [
      {
        "id": "example",
        "envVars": ["EXAMPLE_API_KEY"]
      }
    ]
  },
  "contracts": {
    "tools": ["example_search"]
  },
  "toolMetadata": {
    "example_search": {
      "authSignals": [
        {
          "provider": "example"
        }
      ],
      "configSignals": [
        {
          "rootPath": "plugins.entries.example.config",
          "overlayPath": "search",
          "required": ["apiKey"]
        }
      ]
    }
  }
}
```

`toolMetadata` प्रविष्टियाँ ऊपर दिए गए साझा `configSignals`/`authSignals` फ़ील्ड के अतिरिक्त `optional` (Plugin सक्रियण के लिए टूल को गैर-आवश्यक चिह्नित करता है) और `replaySafe` (अपूर्ण मॉडल टर्न के बाद टूल निष्पादन को दोहराना सुरक्षित चिह्नित करता है) भी स्वीकार करती हैं।

यदि किसी टूल में `toolMetadata` नहीं है, तो OpenClaw मौजूदा व्यवहार बनाए रखता है और टूल अनुबंध के नीति से मेल खाने पर स्वामी Plugin को लोड करता है। जिन हॉट-पाथ टूल की फ़ैक्टरी प्रमाणीकरण/कॉन्फ़िग पर निर्भर करती है, उनके लिए Plugin लेखकों को पूछने हेतु कोर से रनटाइम आयात करवाने के बजाय `toolMetadata` घोषित करना चाहिए।

## providerAuthChoices संदर्भ

केवल-पठन होनी चाहिए: कोई लॉगिन, मॉडल पुल, डाउनलोड या कॉन्फ़िग लेखन नहीं। तैयारी
ठीक चयनित मॉडल की दोबारा जाँच करती है और एक कॉन्फ़िग प्रस्ताव लौटाती है; OpenClaw उस
प्रस्ताव का पृथक रूप से लाइव परीक्षण करता है और सफलता के बाद ही उसे कमिट करता है।

## commandAliases संदर्भ

जब कोई Plugin ऐसे रनटाइम कमांड नाम का स्वामी हो जिसे उपयोगकर्ता गलती से `plugins.allow` में डाल सकते हैं या रूट CLI कमांड के रूप में चलाने का प्रयास कर सकते हैं, तब `commandAliases` का उपयोग करें। OpenClaw, Plugin रनटाइम कोड आयात किए बिना निदान के लिए इस मेटाडेटा का उपयोग करता है।

```json
{
  "commandAliases": [
    {
      "name": "dreaming",
      "kind": "runtime-slash",
      "cliCommand": "memory"
    }
  ]
}
```

| फ़ील्ड        | आवश्यक | प्रकार              | इसका अर्थ                                                           |
| ------------ | -------- | ----------------- | ----------------------------------------------------------------------- |
| `name`       | हाँ      | `string`          | इस Plugin से संबंधित कमांड नाम।                               |
| `kind`       | नहीं       | `"runtime-slash"` | उपनाम को रूट CLI कमांड के बजाय चैट स्लैश कमांड के रूप में चिह्नित करता है। |
| `cliCommand` | नहीं       | `string`          | CLI संचालनों के लिए सुझाया जाने वाला संबंधित रूट CLI कमांड, यदि कोई हो।  |

## activation संदर्भ

जब Plugin कम लागत में यह घोषित कर सके कि किन नियंत्रण-प्लेन घटनाओं को उसे सक्रियण/लोड योजना में शामिल करना चाहिए, तब `activation` का उपयोग करें।

यह ब्लॉक प्लानर मेटाडेटा है, जीवनचक्र API नहीं। यह रनटाइम व्यवहार पंजीकृत नहीं करता, `register(...)` को प्रतिस्थापित नहीं करता और यह आश्वासन नहीं देता कि Plugin कोड पहले ही निष्पादित हो चुका है। सक्रियण प्लानर, `providers`, `channels`, `commandAliases`, `setup.providers`, `contracts.tools` और हुक जैसे मौजूदा मैनिफ़ेस्ट स्वामित्व मेटाडेटा पर वापस जाने से पहले, संभावित Plugin को सीमित करने के लिए इन फ़ील्ड का उपयोग करता है।

स्वामित्व का पहले से वर्णन करने वाला सबसे संकीर्ण मेटाडेटा चुनें। जब `providers`, `channels`, `commandAliases`, सेटअप वर्णनकर्ता या `contracts` उस संबंध को व्यक्त करते हों, तो उनका उपयोग करें। ऐसे अतिरिक्त प्लानर संकेतों के लिए `activation` का उपयोग करें जिन्हें उन स्वामित्व फ़ील्ड द्वारा प्रदर्शित नहीं किया जा सकता। `claude-cli`, `my-cli` या `google-gemini-cli` जैसे CLI रनटाइम उपनामों के लिए शीर्ष-स्तरीय `cliBackends` का उपयोग करें; `activation.onAgentHarnesses` केवल उन एम्बेडेड एजेंट हार्नेस आईडी के लिए है जिनके पास पहले से कोई स्वामित्व फ़ील्ड नहीं है।

प्रत्येक Plugin को `activation.onStartup` जानबूझकर सेट करना चाहिए। इसे केवल तभी `true` पर सेट करें जब Plugin को Gateway स्टार्टअप के दौरान चलना आवश्यक हो। जब Plugin स्टार्टअप पर निष्क्रिय हो और केवल अधिक संकीर्ण ट्रिगर से लोड होना चाहिए, तो इसे `false` पर सेट करें। `onStartup` को छोड़ देने पर अब Plugin स्टार्टअप पर अंतर्निहित रूप से लोड नहीं होता; स्टार्टअप, चैनल, कॉन्फ़िग, एजेंट हार्नेस, मेमोरी या अन्य अधिक संकीर्ण सक्रियण ट्रिगर के लिए स्पष्ट सक्रियण मेटाडेटा का उपयोग करें।

```json
{
  "activation": {
    "onStartup": false,
    "onProviders": ["openai"],
    "onCommands": ["models"],
    "onChannels": ["web"],
    "onRoutes": ["gateway-webhook"],
    "onConfigPaths": ["browser"],
    "onCapabilities": ["provider", "tool"]
  }
}
```

| फ़ील्ड              | आवश्यक | प्रकार                                                 | इसका अर्थ                                                                                                                                                                               |
| ------------------ | -------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `onStartup`        | नहीं       | `boolean`                                            | स्पष्ट Gateway स्टार्टअप सक्रियण। प्रत्येक Plugin को इसे सेट करना चाहिए। `true` स्टार्टअप के दौरान Plugin को आयात करता है; `false` उसे स्टार्टअप पर लेज़ी रखता है, जब तक कोई अन्य मेल खाता ट्रिगर लोड करने की आवश्यकता न पैदा करे। |
| `onProviders`      | नहीं       | `string[]`                                           | वे प्रदाता आईडी, जिन्हें इस Plugin को सक्रियण/लोड योजनाओं में शामिल करना चाहिए।                                                                                                                      |
| `onAgentHarnesses` | नहीं       | `string[]`                                           | वे एम्बेडेड एजेंट हार्नेस रनटाइम आईडी, जिन्हें इस Plugin को सक्रियण/लोड योजनाओं में शामिल करना चाहिए। CLI बैकएंड उपनामों के लिए शीर्ष-स्तरीय `cliBackends` का उपयोग करें।                                           |
| `onCommands`       | नहीं       | `string[]`                                           | वे कमांड आईडी, जिन्हें इस Plugin को सक्रियण/लोड योजनाओं में शामिल करना चाहिए।                                                                                                                       |
| `onChannels`       | नहीं       | `string[]`                                           | वे चैनल आईडी, जिन्हें इस Plugin को सक्रियण/लोड योजनाओं में शामिल करना चाहिए।                                                                                                                       |
| `onRoutes`         | नहीं       | `string[]`                                           | वे रूट प्रकार, जिन्हें इस Plugin को सक्रियण/लोड योजनाओं में शामिल करना चाहिए।                                                                                                                       |
| `onConfigPaths`    | नहीं       | `string[]`                                           | रूट-सापेक्ष कॉन्फ़िग पथ, जो मौजूद होने और स्पष्ट रूप से अक्षम न किए जाने पर इस Plugin को स्टार्टअप/लोड योजनाओं में शामिल करें।                                                      |
| `onCapabilities`   | नहीं       | `Array<"provider" \| "channel" \| "tool" \| "hook">` | नियंत्रण-प्लेन सक्रियण योजना द्वारा उपयोग किए जाने वाले व्यापक क्षमता संकेत। जब संभव हो, अधिक संकीर्ण फ़ील्ड को प्राथमिकता दें।                                                                                     |

वर्तमान लाइव उपभोक्ता:

- Gateway स्टार्टअप योजना स्पष्ट स्टार्टअप इम्पोर्ट के लिए `activation.onStartup` का उपयोग करती है।
- कमांड द्वारा ट्रिगर की गई CLI योजना लेगेसी `commandAliases[].cliCommand` या `commandAliases[].name` पर फ़ॉलबैक करती है।
- एजेंट-रनटाइम स्टार्टअप योजना एम्बेडेड हार्नेस के लिए `activation.onAgentHarnesses` और CLI रनटाइम एलियास के लिए शीर्ष-स्तरीय `cliBackends[]` का उपयोग करती है।
- स्पष्ट चैनल सक्रियण मेटाडेटा अनुपस्थित होने पर चैनल द्वारा ट्रिगर की गई सेटअप/चैनल योजना लेगेसी `channels[]` स्वामित्व पर फ़ॉलबैक करती है।
- स्टार्टअप Plugin योजना बंडल किए गए ब्राउज़र Plugin के `browser` ब्लॉक जैसी गैर-चैनल रूट कॉन्फ़िगरेशन सतहों के लिए `activation.onConfigPaths` का उपयोग करती है।
- स्पष्ट प्रोवाइडर सक्रियण मेटाडेटा अनुपस्थित होने पर प्रोवाइडर द्वारा ट्रिगर की गई सेटअप/रनटाइम योजना लेगेसी `providers[]` और शीर्ष-स्तरीय `cliBackends[]` स्वामित्व पर फ़ॉलबैक करती है।

प्लानर डायग्नोस्टिक्स स्पष्ट सक्रियण संकेतों को मैनिफ़ेस्ट स्वामित्व फ़ॉलबैक से अलग कर सकते हैं। उदाहरण के लिए, `activation-command-hint` का अर्थ है कि `activation.onCommands` मेल खाता है, जबकि `manifest-command-alias` का अर्थ है कि प्लानर ने इसके बजाय `commandAliases` स्वामित्व का उपयोग किया। ये कारण लेबल होस्ट डायग्नोस्टिक्स और परीक्षणों के लिए हैं; Plugin लेखकों को उस मेटाडेटा की घोषणा जारी रखनी चाहिए जो स्वामित्व का सर्वोत्तम वर्णन करता है।

## qaRunners संदर्भ

जब कोई Plugin साझा `openclaw qa` रूट के नीचे एक या अधिक ट्रांसपोर्ट रनर प्रदान करता है, तब `qaRunners` का उपयोग करें। इस मेटाडेटा को हल्का और स्थिर रखें; Plugin रनटाइम अब भी वास्तविक CLI पंजीकरण का स्वामी एक हल्की `runtime-api.ts` सतह के माध्यम से होता है, जो मेल खाने वाले `qaRunnerCliRegistrations` को एक्सपोर्ट करती है। एक वैकल्पिक `adapterFactory` पंजीकृत कमांड के रनर को बदले बिना ट्रांसपोर्ट को साझा QA परिदृश्यों के लिए उपलब्ध कराता है।

```json
{
  "qaRunners": [
    {
      "commandName": "matrix",
      "description": "डिस्पोज़ेबल होमसर्वर के विरुद्ध Docker-समर्थित Matrix लाइव QA लेन चलाएँ"
    }
  ]
}
```

| फ़ील्ड         | आवश्यक | प्रकार     | इसका अर्थ                                                      |
| ------------- | -------- | -------- | ------------------------------------------------------------------ |
| `commandName` | हाँ      | `string` | `openclaw qa` के नीचे माउंट किया गया सबकमांड, उदाहरण के लिए `matrix`।    |
| `description` | नहीं       | `string` | साझा होस्ट को स्टब कमांड की आवश्यकता होने पर उपयोग किया जाने वाला फ़ॉलबैक सहायता टेक्स्ट। |

`adapterFactory` आईडी का `commandName` से मेल खाना आवश्यक है। मैनिफ़ेस्ट में अनुपस्थित कमांड के लिए पंजीकरण एक्सपोर्ट न करें।

## सेटअप संदर्भ

जब सेटअप और ऑनबोर्डिंग सतहों को रनटाइम लोड होने से पहले हल्के Plugin-स्वामित्व वाले मेटाडेटा की आवश्यकता हो, तब `setup` का उपयोग करें।

```json
{
  "setup": {
    "providers": [
      {
        "id": "openai",
        "authMethods": ["api-key"],
        "envVars": ["OPENAI_API_KEY"],
        "authEvidence": [
          {
            "type": "local-file-with-env",
            "fileEnvVar": "OPENAI_CREDENTIALS_FILE",
            "requiresAllEnv": ["OPENAI_PROJECT"],
            "credentialMarker": "openai-local-credentials",
            "source": "OpenAI स्थानीय क्रेडेंशियल"
          }
        ]
      }
    ],
    "cliBackends": ["openai-cli"],
    "configMigrations": ["legacy-openai-auth"],
    "requiresRuntime": false
  }
}
```

शीर्ष-स्तरीय `cliBackends` वैध रहता है और CLI अनुमान बैकएंड का वर्णन करना जारी रखता है। `setup.cliBackends` कंट्रोल-प्लेन/सेटअप प्रवाहों के लिए सेटअप-विशिष्ट डिस्क्रिप्टर सतह है, जिसे केवल मेटाडेटा तक सीमित रहना चाहिए।

मौजूद होने पर, `setup.providers` और `setup.cliBackends` सेटअप खोज के लिए पसंदीदा डिस्क्रिप्टर-प्रथम लुकअप सतह हैं। यदि डिस्क्रिप्टर केवल उम्मीदवार Plugin का दायरा सीमित करता है और सेटअप को फिर भी अधिक समृद्ध सेटअप-समय रनटाइम हुक की आवश्यकता है, तो `requiresRuntime: true` सेट करें और फ़ॉलबैक निष्पादन पथ के रूप में `setup-api` को यथास्थान रखें।

OpenClaw सामान्य प्रोवाइडर प्रमाणीकरण और एनवायरनमेंट-वेरिएबल लुकअप में `setup.providers[].envVars` शामिल करता है। सेटअप और स्थिति का एनवायरनमेंट मेटाडेटा वहाँ रखें।

जब किसी बिलिंग या संगठन-स्तरीय क्रेडेंशियल को अनुमान क्रेडेंशियल बने बिना `resolveUsageAuth` सक्रिय करना आवश्यक हो, तब `providerUsageAuthEnvVars` का उपयोग करें। ये नाम वर्कस्पेस dotenv ब्लॉकिंग, ACP चाइल्ड-प्रोसेस स्ट्रिपिंग, सैंडबॉक्स सीक्रेट फ़िल्टरिंग और व्यापक सीक्रेट स्क्रबिंग में शामिल होते हैं। प्रोवाइडर रनटाइम अब भी `resolveUsageAuth` के भीतर मान को पढ़ता और वर्गीकृत करता है।

जब कोई सेटअप प्रविष्टि उपलब्ध न हो, या जब `setup.requiresRuntime: false` सेटअप रनटाइम को अनावश्यक घोषित करे, तब OpenClaw `setup.providers[].authMethods` से सरल सेटअप विकल्प भी प्राप्त कर सकता है। कस्टम लेबल, CLI फ़्लैग, ऑनबोर्डिंग दायरे और सहायक मेटाडेटा के लिए स्पष्ट `providerAuthChoices` प्रविष्टियाँ पसंदीदा रहती हैं।

`requiresRuntime: false` केवल तभी सेट करें जब वे डिस्क्रिप्टर सेटअप सतह के लिए पर्याप्त हों। OpenClaw स्पष्ट `false` को केवल-डिस्क्रिप्टर अनुबंध मानता है और सेटअप लुकअप के लिए `setup-api` या `openclaw.setupEntry` निष्पादित नहीं करेगा। यदि केवल-डिस्क्रिप्टर Plugin फिर भी उन सेटअप रनटाइम प्रविष्टियों में से किसी को शिप करता है, तो OpenClaw एक अतिरिक्त डायग्नोस्टिक रिपोर्ट करता है और उसे अनदेखा करना जारी रखता है। `requiresRuntime` को छोड़ देने पर लेगेसी फ़ॉलबैक व्यवहार बना रहता है, ताकि फ़्लैग के बिना डिस्क्रिप्टर जोड़ने वाले मौजूदा Plugin न टूटें।

चूँकि सेटअप लुकअप Plugin-स्वामित्व वाला `setup-api` कोड निष्पादित कर सकता है, इसलिए नॉर्मलाइज़ किए गए `setup.providers[].id` और `setup.cliBackends[]` मान खोजे गए सभी Plugin में अद्वितीय रहने चाहिए। अस्पष्ट स्वामित्व खोज क्रम से किसी विजेता को चुनने के बजाय बंद स्थिति में विफल होता है।

जब सेटअप रनटाइम निष्पादित होता है, तब यदि `setup-api` ऐसे प्रोवाइडर या CLI बैकएंड को पंजीकृत करता है जिसकी घोषणा मैनिफ़ेस्ट डिस्क्रिप्टर नहीं करते, या यदि किसी डिस्क्रिप्टर का कोई मेल खाता रनटाइम पंजीकरण नहीं है, तो सेटअप रजिस्ट्री डायग्नोस्टिक्स डिस्क्रिप्टर विचलन रिपोर्ट करते हैं। ये डायग्नोस्टिक्स अतिरिक्त हैं और लेगेसी Plugin को अस्वीकार नहीं करते।

### setup.providers संदर्भ

| फ़ील्ड          | आवश्यक | प्रकार       | इसका अर्थ                                                                                    |
| -------------- | -------- | ---------- | ------------------------------------------------------------------------------------------------ |
| `id`           | हाँ      | `string`   | सेटअप या ऑनबोर्डिंग के दौरान उपलब्ध कराया गया प्रोवाइडर आईडी। नॉर्मलाइज़ किए गए आईडी को वैश्विक रूप से अद्वितीय रखें।             |
| `authMethods`  | नहीं       | `string[]` | पूर्ण रनटाइम लोड किए बिना इस प्रोवाइडर द्वारा समर्थित सेटअप/प्रमाणीकरण विधि आईडी।                       |
| `envVars`      | नहीं       | `string[]` | वे एनवायरनमेंट वेरिएबल जिन्हें सामान्य सेटअप/स्थिति सतहें Plugin रनटाइम लोड होने से पहले जाँच सकती हैं।               |
| `authEvidence` | नहीं       | `object[]` | उन प्रोवाइडर के लिए हल्की स्थानीय प्रमाणीकरण प्रमाण जाँचें जो गैर-सीक्रेट मार्कर के माध्यम से प्रमाणीकरण कर सकते हैं। |

`authEvidence` प्रोवाइडर-स्वामित्व वाले स्थानीय क्रेडेंशियल मार्कर के लिए है, जिन्हें रनटाइम कोड लोड किए बिना सत्यापित किया जा सकता है। ये जाँचें हल्की और स्थानीय रहनी चाहिए: कोई नेटवर्क कॉल नहीं, कोई कीचेन या सीक्रेट-मैनेजर रीड नहीं, कोई शेल कमांड नहीं और कोई प्रोवाइडर API प्रोब नहीं।

समर्थित प्रमाण प्रविष्टियाँ:

| फ़ील्ड              | आवश्यक | प्रकार       | इसका अर्थ                                                                                                  |
| ------------------ | -------- | ---------- | -------------------------------------------------------------------------------------------------------------- |
| `type`             | हाँ      | `string`   | वर्तमान में `local-file-with-env`।                                                                               |
| `fileEnvVar`       | नहीं       | `string`   | स्पष्ट क्रेडेंशियल फ़ाइल पथ वाला एनवायरनमेंट वेरिएबल।                                                           |
| `fallbackPaths`    | नहीं       | `string[]` | `fileEnvVar` के अनुपस्थित या खाली होने पर जाँचे जाने वाले स्थानीय क्रेडेंशियल फ़ाइल पथ। `${HOME}` और `${APPDATA}` का समर्थन करता है। |
| `requiresAnyEnv`   | नहीं       | `string[]` | प्रमाण के वैध होने से पहले सूचीबद्ध एनवायरनमेंट वेरिएबल में से कम-से-कम एक गैर-खाली होना आवश्यक है।                                    |
| `requiresAllEnv`   | नहीं       | `string[]` | प्रमाण के वैध होने से पहले प्रत्येक सूचीबद्ध एनवायरनमेंट वेरिएबल का गैर-खाली होना आवश्यक है।                                           |
| `credentialMarker` | हाँ      | `string`   | प्रमाण मौजूद होने पर लौटाया जाने वाला गैर-सीक्रेट मार्कर।                                                       |
| `source`           | नहीं       | `string`   | प्रमाणीकरण/स्थिति आउटपुट के लिए उपयोगकर्ता-दृश्य स्रोत लेबल।                                                               |

### सेटअप फ़ील्ड

| फ़ील्ड              | आवश्यक | प्रकार       | इसका अर्थ                                                                                       |
| ------------------ | -------- | ---------- | --------------------------------------------------------------------------------------------------- |
| `providers`        | नहीं       | `object[]` | सेटअप और ऑनबोर्डिंग के दौरान उपलब्ध कराए गए प्रोवाइडर सेटअप डिस्क्रिप्टर।                                     |
| `cliBackends`      | नहीं       | `string[]` | डिस्क्रिप्टर-प्रथम सेटअप लुकअप के लिए उपयोग किए जाने वाले सेटअप-समय बैकएंड आईडी। नॉर्मलाइज़ किए गए आईडी को वैश्विक रूप से अद्वितीय रखें। |
| `configMigrations` | नहीं       | `string[]` | इस Plugin की सेटअप सतह के स्वामित्व वाले कॉन्फ़िगरेशन माइग्रेशन आईडी।                                          |
| `requiresRuntime`  | नहीं       | `boolean`  | क्या सेटअप को डिस्क्रिप्टर लुकअप के बाद भी `setup-api` निष्पादन की आवश्यकता है।                            |

## uiHints संदर्भ

`uiHints` कॉन्फ़िगरेशन फ़ील्ड नामों से छोटे रेंडरिंग संकेतों तक का मैप है। नेस्टेड कॉन्फ़िगरेशन फ़ील्ड के लिए कुंजियाँ डॉट का उपयोग कर सकती हैं, लेकिन कोई भी पथ खंड `__proto__`, `constructor` या `prototype` नहीं हो सकता; सेटअप इन नामों को अस्वीकार करता है।

```json
{
  "uiHints": {
    "apiKey": {
      "label": "API कुंजी",
      "help": "OpenRouter अनुरोधों के लिए उपयोग किया जाता है",
      "placeholder": "sk-or-v1-...",
      "sensitive": true
    }
  }
}
```

प्रत्येक फ़ील्ड संकेत में ये शामिल हो सकते हैं:

| फ़ील्ड         | प्रकार       | इसका अर्थ                           |
| ------------- | ---------- | --------------------------------------- |
| `label`       | `string`   | उपयोगकर्ता-दृश्य फ़ील्ड लेबल।                |
| `help`        | `string`   | संक्षिप्त सहायक टेक्स्ट।                      |
| `tags`        | `string[]` | वैकल्पिक UI टैग।                       |
| `advanced`    | `boolean`  | फ़ील्ड को उन्नत के रूप में चिह्नित करता है।            |
| `sensitive`   | `boolean`  | फ़ील्ड को सीक्रेट या संवेदनशील के रूप में चिह्नित करता है। |
| `placeholder` | `string`   | फ़ॉर्म इनपुट के लिए प्लेसहोल्डर टेक्स्ट।       |

## अनुबंध संदर्भ

`contracts` का उपयोग केवल ऐसे स्थिर क्षमता-स्वामित्व मेटाडेटा के लिए करें, जिसे OpenClaw Plugin रनटाइम इम्पोर्ट किए बिना पढ़ सके।

```json
{
  "contracts": {
    "agentToolResultMiddleware": ["openclaw", "codex"],
    "trustedToolPolicies": ["workflow-budget"],
    "externalAuthProviders": ["acme-ai"],
    "embeddingProviders": ["openai-compatible"],
    "speechProviders": ["openai"],
    "realtimeTranscriptionProviders": ["openai"],
    "realtimeVoiceProviders": ["openai"],
    "memoryEmbeddingProviders": ["local"],
    "mediaUnderstandingProviders": ["openai"],
    "imageGenerationProviders": ["openai"],
    "videoGenerationProviders": ["qwen"],
    "musicGenerationProviders": ["stability-audio"],
    "documentExtractors": ["example-docs"],
    "webContentExtractors": ["firecrawl"],
    "webFetchProviders": ["firecrawl"],
    "webSearchProviders": ["gemini"],
    "workerProviders": ["example-worker"],
    "usageProviders": ["acme-ai"],
    "migrationProviders": ["hermes"],
    "gatewayMethodDispatch": ["authenticated-request"],
    "tools": ["firecrawl_search", "firecrawl_scrape"]
  }
}
```

प्रत्येक सूची वैकल्पिक है:

| फ़ील्ड                            | प्रकार       | इसका अर्थ                                                                                                                        |
| -------------------------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `embeddedExtensionFactories`     | `string[]` | Codex ऐप-सर्वर एक्सटेंशन फ़ैक्टरी आईडी, वर्तमान में `codex-app-server`।                                                                |
| `agentToolResultMiddleware`      | `string[]` | वे रनटाइम आईडी जिनके लिए यह Plugin टूल-परिणाम मिडलवेयर पंजीकृत कर सकता है।                                                                     |
| `trustedToolPolicies`            | `string[]` | Plugin-स्थानीय विश्वसनीय प्री-टूल नीति आईडी, जिन्हें कोई इंस्टॉल किया गया Plugin पंजीकृत कर सकता है। बंडल किए गए Plugin इस फ़ील्ड के बिना नीतियाँ पंजीकृत कर सकते हैं। |

`contracts.embeddedExtensionFactories` को केवल बंडल किए गए Codex ऐप-सर्वर एक्सटेंशन फ़ैक्टरियों के लिए बनाए रखा गया है। बंडल किए गए टूल-परिणाम रूपांतरणों को इसके बजाय `contracts.agentToolResultMiddleware` घोषित करना और `api.registerAgentToolResultMiddleware(...)` के साथ पंजीकृत होना चाहिए। इंस्टॉल किए गए Plugin उसी मिडलवेयर सीम का उपयोग केवल तभी कर सकते हैं जब उसे स्पष्ट रूप से सक्षम किया गया हो और केवल उन रनटाइम के लिए जिन्हें वे `contracts.agentToolResultMiddleware` में घोषित करते हैं।

जिन इंस्टॉल किए गए Plugin को होस्ट-विश्वसनीय प्री-टूल नीति स्तर की आवश्यकता है, उन्हें प्रत्येक पंजीकृत स्थानीय आईडी को `contracts.trustedToolPolicies` में घोषित करना और स्पष्ट रूप से सक्षम होना आवश्यक है। बंडल किए गए Plugin मौजूदा विश्वसनीय-नीति पथ बनाए रखते हैं, लेकिन अघोषित नीति आईडी वाले इंस्टॉल किए गए Plugin को पंजीकरण से पहले अस्वीकार कर दिया जाता है। नीति आईडी पंजीकरण करने वाले Plugin के दायरे में होती हैं, इसलिए दो Plugin `workflow-budget` को घोषित और पंजीकृत कर सकते हैं; एक ही Plugin उसी स्थानीय आईडी को दो बार पंजीकृत नहीं कर सकता।

रनटाइम `api.registerTool(...)` पंजीकरणों का `contracts.tools` से मेल खाना आवश्यक है। टूल खोज इस सूची का उपयोग केवल उन Plugin रनटाइम को लोड करने के लिए करती है जो अनुरोधित टूल के स्वामी हो सकते हैं।

`resolveExternalAuthProfiles` लागू करने वाले प्रदाता Plugin को `contracts.externalAuthProviders` घोषित करना चाहिए; अघोषित बाहरी-प्रमाणीकरण हुक अनदेखे कर दिए जाते हैं।

`resolveUsageAuth` और `fetchUsageSnapshot` दोनों लागू करने वाले प्रदाता Plugin को प्रत्येक स्वतः खोजे गए प्रदाता आईडी को `contracts.usageProviders` में घोषित करना चाहिए। उपयोग खोज रनटाइम कोड लोड करने से पहले इस अनुबंध को पढ़ती है, फिर केवल घोषित स्वामियों को लोड करने के बाद दोनों हुक सत्यापित करती है।

सामान्य एम्बेडिंग प्रदाताओं को `api.registerEmbeddingProvider(...)` के साथ पंजीकृत प्रत्येक अडैप्टर के लिए `contracts.embeddingProviders` घोषित करना चाहिए। मेमोरी खोज द्वारा उपयोग किए जाने वाले प्रदाताओं सहित, पुन: प्रयोज्य वेक्टर निर्माण के लिए सामान्य अनुबंध का उपयोग करें। `contracts.memoryEmbeddingProviders` अप्रचलित मेमोरी-विशिष्ट संगतता है और केवल तब तक बनी रहती है जब तक मौजूदा प्रदाता जेनेरिक एम्बेडिंग प्रदाता सीम पर माइग्रेट नहीं हो जाते।

वर्कर प्रदाताओं को प्रत्येक `api.registerWorkerProvider(...)` आईडी को `contracts.workerProviders` में घोषित करना आवश्यक है। कोर `provision` को कॉल करने से पहले टिकाऊ आशय सहेजता है; प्रदाता बाहरी आवंटन से पहले अपनी सेटिंग्स सत्यापित करते हैं, और समान ऑपरेशन आईडी वाली बार-बार की गई कॉल को उसी लीज़ को अपनाना आवश्यक है। कोर उस सत्यापित सेटिंग्स स्नैपशॉट को भी सहेजता है और उसे `leaseId` के साथ `inspect({ leaseId, profile })` और `destroy({ leaseId, profile })` को भेजता है, जिसमें नामित प्रोफ़ाइल को बदलने या हटाने के बाद की स्थिति भी शामिल है। विनाश समघाती है, निरीक्षण बंद `active` / `destroyed` / `unknown` स्थिति यूनियन लौटाता है, और SSH निजी-कुंजी सामग्री को केवल `SecretRef` के माध्यम से संदर्भित किया जाता है। प्रोविज़न किए गए SSH एंडपॉइंट में विश्वसनीय प्रोविज़निंग आउटपुट से एक सार्वजनिक `hostKey` भी ठीक `algorithm base64` के रूप में शामिल होना आवश्यक है, बिना किसी होस्टनाम या टिप्पणी के, ताकि कोर कनेक्ट करने से पहले होस्ट को पिन कर सके। डायनेमिक पहचान संदर्भ बनाने वाले प्रदाता आधिकारिक `resolveSshIdentity({ leaseId, profile, keyRef })` लागू कर सकते हैं; इसके बिना प्रदाता कोर के जेनेरिक सीक्रेट रिज़ॉल्वर का उपयोग करते हैं। आधिकारिक `unknown` किसी सक्रिय स्थानीय रिकॉर्ड को अनाथ कर देता है; स्थायी विनाश अनुरोध के बाद यह विघटन की पुष्टि करता है।

`contracts.gatewayMethodDispatch` वर्तमान में `"authenticated-request"` स्वीकार करता है। यह उन नेटिव Plugin HTTP रूट के लिए एक API स्वच्छता गेट है जो जानबूझकर Gateway नियंत्रण-प्लेन विधियों को इन-प्रोसेस डिस्पैच करते हैं, न कि दुर्भावनापूर्ण नेटिव Plugin के विरुद्ध कोई सैंडबॉक्स। इसका उपयोग केवल कड़ाई से समीक्षित बंडल किए गए/ऑपरेटर सतहों के लिए करें, जिन्हें पहले से Gateway HTTP प्रमाणीकरण की आवश्यकता होती है। Gateway रूट-कार्य प्रवेश बंद होने पर कोई अधिकृत रूट केवल तभी पहुँच योग्य रहता है जब वह `auth: "gateway"` और रूट-विशिष्ट `gatewayRuntimeScopeSurface: "trusted-operator"` भी घोषित करता है; उसी Plugin के सामान्य सहोदर रूट प्रवेश सीमा के पीछे रहते हैं। इससे पूरे Plugin को प्रवेश बायपास दिए बिना निलंबन स्थिति और पुनरारंभ पहुँच योग्य रहते हैं। पार्सिंग और प्रतिक्रिया संरचना को डिस्पैच के बाहर सीमित रखें; सारभूत या परिवर्तनकारी कार्य को Gateway विधि डिस्पैच के माध्यम से जाना आवश्यक है, जिसके पास प्रवेश और दायरा प्रवर्तन का स्वामित्व है।

## configContracts संदर्भ

मैनिफ़ेस्ट-स्वामित्व वाले उस कॉन्फ़िग व्यवहार के लिए `configContracts` का उपयोग करें जिसकी आवश्यकता जेनेरिक कोर सहायकों को Plugin रनटाइम आयात किए बिना होती है: खतरनाक-फ़्लैग पहचान, SecretRef माइग्रेशन लक्ष्य, और विरासत कॉन्फ़िग-पथ संकुचन।

```json
{
  "configContracts": {
    "compatibilityMigrationPaths": ["legacyProvider"],
    "compatibilityRuntimePaths": ["legacyProvider.webhook"],
    "dangerousFlags": [
      {
        "path": "accounts.*.allowUnverifiedSenders",
        "equals": true
      }
    ],
    "secretInputs": {
      "bundledDefaultEnabled": false,
      "paths": [
        {
          "path": "routes.*.secret",
          "expected": "string",
          "ownerKind": "route"
        }
      ]
    }
  }
}
```

| फ़ील्ड                         | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                                                                                                          |
| ----------------------------- | -------- | ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `compatibilityMigrationPaths` | नहीं       | `string[]` | रूट-सापेक्ष कॉन्फ़िग पथ जो दर्शाते हैं कि इस Plugin के सेटअप-समय संगतता माइग्रेशन लागू हो सकते हैं। जब कॉन्फ़िग में Plugin का कभी संदर्भ नहीं होता, तब जेनेरिक रनटाइम कॉन्फ़िग पठन को हर Plugin सेटअप सतह छोड़ने देता है।                 |
| `compatibilityRuntimePaths`   | नहीं       | `string[]` | रूट-सापेक्ष संगतता पथ जिन्हें यह Plugin, Plugin कोड के पूरी तरह सक्रिय होने से पहले रनटाइम के दौरान संभाल सकता है। इसका उपयोग उन विरासत सतहों के लिए करें जिन्हें प्रत्येक संगत Plugin रनटाइम आयात किए बिना बंडल किए गए उम्मीदवार समूहों को सीमित करना चाहिए। |
| `dangerousFlags`              | नहीं       | `object[]` | वे कॉन्फ़िग लिटरल जिन्हें सक्षम होने पर `openclaw doctor` को असुरक्षित या खतरनाक के रूप में चिह्नित करना चाहिए। नीचे देखें।                                                                                                                                   |
| `secretInputs`                | नहीं       | `object`   | SecretRef माइग्रेशन, ऑडिट, स्टार्टअप मैटेरियलाइज़ेशन और वैकल्पिक रनटाइम स्वामी पृथक्करण के लिए `plugins.entries.<id>.config` के अंतर्गत कॉन्फ़िग पथ। नीचे देखें।                                                                             |

प्रत्येक `dangerousFlags` प्रविष्टि इसका समर्थन करती है:

| फ़ील्ड    | आवश्यक | प्रकार                                  | इसका अर्थ                                                                                                       |
| -------- | -------- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `path`   | हाँ      | `string`                              | `plugins.entries.<id>.config` के सापेक्ष डॉट से अलग किया गया कॉन्फ़िग पथ। मैप/ऐरे खंडों के लिए `*` वाइल्डकार्ड का समर्थन करता है। |
| `equals` | हाँ      | `string \| number \| boolean \| null` | वह सटीक लिटरल जो इस कॉन्फ़िग मान को खतरनाक के रूप में चिह्नित करता है।                                                            |

`secretInputs` इसका समर्थन करता है:

| फ़ील्ड                   | आवश्यक | प्रकार       | इसका अर्थ                                                                                                                                                                                                                                                                                                                                              |
| ----------------------- | -------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bundledDefaultEnabled` | नहीं       | `boolean`  | यह तय करते समय कि यह SecretRef सतह सक्रिय है या नहीं, बंडल किए गए Plugin की डिफ़ॉल्ट सक्षमता को ओवरराइड करें। इसका उपयोग तब करें जब Plugin बंडल किया गया हो, लेकिन कॉन्फ़िगरेशन में स्पष्ट रूप से सक्षम किए जाने तक सतह निष्क्रिय रहनी चाहिए।                                                                                                                                            |
| `paths`                 | हाँ      | `object[]` | सीक्रेट के स्वरूप वाले कॉन्फ़िगरेशन पथ, जिनमें प्रत्येक के साथ `path` (डॉट से अलग किया हुआ, `plugins.entries.<id>.config` के सापेक्ष, `*` वाइल्डकार्ड समर्थित), वैकल्पिक `expected` (वर्तमान में केवल `"string"`), और वैकल्पिक `ownerKind` (वर्तमान में केवल `"route"`) होता है। रिज़ॉल्यूशन विफल होने पर घोषित स्वामी केवल ठीक उसी मेल खाने वाले पथ को अलग करता है; उसकी स्वामी आईडी पूरा कॉन्फ़िगरेशन पथ होती है। |

## mediaUnderstandingProviderMetadata संदर्भ

जब किसी मीडिया-समझ प्रदाता के पास डिफ़ॉल्ट मॉडल, स्वतः-प्रमाणीकरण फ़ॉलबैक प्राथमिकता, या मूल दस्तावेज़ समर्थन हो जिसकी जेनेरिक कोर सहायकों को रनटाइम लोड होने से पहले आवश्यकता हो, तब `mediaUnderstandingProviderMetadata` का उपयोग करें। कुंजियाँ `contracts.mediaUnderstandingProviders` में भी घोषित होनी चाहिए।

```json
{
  "contracts": {
    "mediaUnderstandingProviders": ["example"]
  },
  "mediaUnderstandingProviderMetadata": {
    "example": {
      "capabilities": ["image", "audio"],
      "defaultModels": {
        "image": "example-vision-latest",
        "audio": "example-transcribe-latest"
      },
      "autoPriority": {
        "image": 40
      },
      "nativeDocumentInputs": ["pdf"],
      "documentModels": {
        "pdf": {
          "textExtraction": "example-doc-text-latest",
          "image": "example-doc-vision-latest"
        }
      }
    }
  }
}
```

प्रत्येक प्रदाता प्रविष्टि में ये शामिल हो सकते हैं:

| फ़ील्ड                  | प्रकार                                                             | इसका अर्थ                                                                                                   |
| ---------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `capabilities`         | `("image" \| "audio" \| "video")[]`                              | इस प्रदाता द्वारा उपलब्ध कराई गई मीडिया क्षमताएँ।                                                                    |
| `defaultModels`        | `Record<string, string>`                                         | जब कॉन्फ़िगरेशन कोई मॉडल निर्दिष्ट नहीं करता, तब उपयोग किए जाने वाले क्षमता-से-मॉडल डिफ़ॉल्ट।                                         |
| `autoPriority`         | `Record<string, number>`                                         | स्वचालित क्रेडेंशियल-आधारित प्रदाता फ़ॉलबैक के लिए छोटी संख्याएँ पहले क्रमबद्ध होती हैं।                                    |
| `nativeDocumentInputs` | `"pdf"[]`                                                        | प्रदाता द्वारा समर्थित मूल दस्तावेज़ इनपुट।                                                               |
| `documentModels`       | `{ pdf?: { textExtraction?: string; image?: string \| false } }` | प्रत्येक दस्तावेज़ प्रकार के लिए मॉडल ओवरराइड। उस दस्तावेज़ प्रकार के लिए छवि-आधारित निष्कर्षण अक्षम करने हेतु `image: false` सेट करें। |

## channelConfigs संदर्भ

जब किसी चैनल Plugin को रनटाइम लोड होने से पहले हल्के कॉन्फ़िगरेशन मेटाडेटा की आवश्यकता हो, तब `channelConfigs` का उपयोग करें। जब कोई सेटअप प्रविष्टि उपलब्ध न हो, या जब `setup.requiresRuntime: false` सेटअप रनटाइम को अनावश्यक घोषित करे, तब केवल-पढ़ने योग्य चैनल सेटअप/स्थिति खोज कॉन्फ़िगर किए गए बाहरी चैनलों के लिए इस मेटाडेटा का सीधे उपयोग कर सकती है।

`channelConfigs` Plugin मैनिफ़ेस्ट मेटाडेटा है, कोई नया शीर्ष-स्तरीय उपयोगकर्ता कॉन्फ़िगरेशन अनुभाग नहीं। उपयोगकर्ता अब भी चैनल इंस्टेंस को `channels.<channel-id>` के अंतर्गत कॉन्फ़िगर करते हैं। Plugin रनटाइम कोड निष्पादित होने से पहले OpenClaw यह तय करने के लिए मैनिफ़ेस्ट मेटाडेटा पढ़ता है कि उस कॉन्फ़िगर किए गए चैनल का स्वामी कौन-सा Plugin है।

किसी चैनल Plugin के लिए, `configSchema` और `channelConfigs` अलग-अलग पथों का वर्णन करते हैं:

- `configSchema`, `plugins.entries.<plugin-id>.config` को सत्यापित करता है
- `channelConfigs.<channel-id>.schema`, `channels.<channel-id>` को सत्यापित करता है

`channels[]` घोषित करने वाले गैर-बंडल Plugin को मेल खाने वाली `channelConfigs` प्रविष्टियाँ भी घोषित करनी चाहिए। इनके बिना OpenClaw अब भी Plugin लोड कर सकता है, लेकिन कोल्ड-पाथ कॉन्फ़िगरेशन स्कीमा, सेटअप और Control UI सतहें Plugin रनटाइम निष्पादित होने तक चैनल के स्वामित्व वाले विकल्प का स्वरूप नहीं जान सकतीं।

`channelConfigs.<channel-id>.commands.nativeCommandsAutoEnabled` और `nativeSkillsAutoEnabled`, चैनल रनटाइम लोड होने से पहले चलने वाली कमांड कॉन्फ़िगरेशन जाँचों के लिए स्थिर `auto` डिफ़ॉल्ट घोषित कर सकते हैं। बंडल किए गए चैनल अपने अन्य पैकेज-स्वामित्व वाले चैनल कैटलॉग मेटाडेटा के साथ `package.json#openclaw.channel.commands` के माध्यम से भी वही डिफ़ॉल्ट प्रकाशित कर सकते हैं।

```json
{
  "channelConfigs": {
    "matrix": {
      "schema": {
        "type": "object",
        "additionalProperties": false,
        "properties": {
          "homeserverUrl": { "type": "string" }
        }
      },
      "uiHints": {
        "homeserverUrl": {
          "label": "होमसर्वर URL",
          "placeholder": "https://matrix.example.com"
        }
      },
      "label": "Matrix",
      "description": "Matrix होमसर्वर कनेक्शन",
      "commands": {
        "nativeCommandsAutoEnabled": true,
        "nativeSkillsAutoEnabled": true
      },
      "preferOver": ["matrix-legacy"]
    }
  }
}
```

प्रत्येक चैनल प्रविष्टि में ये शामिल हो सकते हैं:

| फ़ील्ड         | प्रकार                     | इसका अर्थ                                                                             |
| ------------- | ------------------------ | ----------------------------------------------------------------------------------------- |
| `schema`      | `object`                 | `channels.<id>` के लिए JSON स्कीमा। प्रत्येक घोषित चैनल कॉन्फ़िगरेशन प्रविष्टि के लिए आवश्यक।         |
| `uiHints`     | `Record<string, object>` | उस चैनल कॉन्फ़िगरेशन अनुभाग के लिए वैकल्पिक UI लेबल/प्लेसहोल्डर/संवेदनशीलता संकेत।          |
| `label`       | `string`                 | रनटाइम मेटाडेटा तैयार न होने पर पिकर और निरीक्षण सतहों में मर्ज किया जाने वाला चैनल लेबल। |
| `description` | `string`                 | निरीक्षण और कैटलॉग सतहों के लिए चैनल का संक्षिप्त विवरण।                               |
| `commands`    | `object`                 | रनटाइम-पूर्व कॉन्फ़िगरेशन जाँचों के लिए मूल कमांड और मूल स्किल के स्थिर स्वचालित डिफ़ॉल्ट।       |
| `preferOver`  | `string[]`               | पुराने या निम्न-प्राथमिकता वाले Plugin आईडी, जिनसे इस चैनल को चयन सतहों में ऊपर रहना चाहिए।    |

### किसी अन्य चैनल Plugin को प्रतिस्थापित करना

जब आपका Plugin किसी ऐसे चैनल आईडी का पसंदीदा स्वामी हो जिसे कोई अन्य Plugin भी उपलब्ध करा सकता है, तब `preferOver` का उपयोग करें। सामान्य मामलों में बदली हुई Plugin आईडी, किसी बंडल किए गए Plugin का स्थान लेने वाला स्टैंडअलोन Plugin, या कॉन्फ़िगरेशन संगतता के लिए समान चैनल आईडी बनाए रखने वाला अनुरक्षित फ़ोर्क शामिल हैं।

```json
{
  "id": "acme-chat",
  "channels": ["chat"],
  "channelConfigs": {
    "chat": {
      "schema": {
        "type": "object",
        "additionalProperties": false,
        "properties": {
          "webhookUrl": { "type": "string" }
        }
      },
      "preferOver": ["chat"]
    }
  }
}
```

जब `channels.chat` कॉन्फ़िगर किया जाता है, तब OpenClaw चैनल आईडी और पसंदीदा Plugin आईडी, दोनों पर विचार करता है। यदि निम्न-प्राथमिकता वाला Plugin केवल इसलिए चुना गया था क्योंकि वह बंडल किया गया है या डिफ़ॉल्ट रूप से सक्षम है, तो OpenClaw उसे प्रभावी रनटाइम कॉन्फ़िगरेशन में अक्षम कर देता है, ताकि एक ही Plugin चैनल और उसके टूल का स्वामी हो। उपयोगकर्ता का स्पष्ट चयन फिर भी प्रभावी रहता है: यदि उपयोगकर्ता दोनों Plugin को स्पष्ट रूप से सक्षम करता है (`plugins.allow` या किसी सार्थक `plugins.entries` कॉन्फ़िगरेशन के माध्यम से), तो OpenClaw उस चयन को बनाए रखता है और अनुरोधित Plugin सेट को चुपचाप बदलने के बजाय डुप्लिकेट चैनल/टूल निदान रिपोर्ट करता है।

`preferOver` को केवल उन Plugin आईडी तक सीमित रखें जो वास्तव में वही चैनल उपलब्ध करा सकती हैं। यह सामान्य प्राथमिकता फ़ील्ड नहीं है और यह उपयोगकर्ता कॉन्फ़िगरेशन कुंजियों का नाम नहीं बदलता।

## modelSupport संदर्भ

जब OpenClaw को Plugin रनटाइम लोड होने से पहले `gpt-5.6-sol` या `claude-sonnet-4.6` जैसी संक्षिप्त मॉडल आईडी से आपके प्रदाता Plugin का अनुमान लगाना चाहिए, तब `modelSupport` का उपयोग करें।

```json
{
  "modelSupport": {
    "modelPrefixes": ["gpt-", "o1", "o3", "o4"],
    "modelPatterns": ["^computer-use-preview"]
  }
}
```

OpenClaw यह वरीयता क्रम लागू करता है:

- स्पष्ट `provider/model` संदर्भ स्वामित्व वाले `providers` मैनिफ़ेस्ट मेटाडेटा का उपयोग करते हैं
- `modelPatterns`, `modelPrefixes` से अधिक प्राथमिकता पाते हैं
- यदि एक गैर-बंडल Plugin और एक बंडल किया गया Plugin दोनों मेल खाते हैं, तो गैर-बंडल Plugin को प्राथमिकता मिलती है
- शेष अस्पष्टता को तब तक अनदेखा किया जाता है जब तक उपयोगकर्ता या कॉन्फ़िगरेशन कोई प्रदाता निर्दिष्ट न करे

फ़ील्ड:

| फ़ील्ड           | प्रकार       | इसका अर्थ                                                                   |
| --------------- | ---------- | ------------------------------------------------------------------------------- |
| `modelPrefixes` | `string[]` | संक्षिप्त मॉडल आईडी के विरुद्ध `startsWith` से मिलान किए गए उपसर्ग।                 |
| `modelPatterns` | `string[]` | प्रोफ़ाइल प्रत्यय हटाने के बाद संक्षिप्त मॉडल आईडी के विरुद्ध मिलान किए गए रेगेक्स स्रोत। |

`modelPatterns` प्रविष्टियाँ `compileSafeRegex` के माध्यम से संकलित की जाती हैं, जो नेस्टेड पुनरावृत्ति वाले पैटर्न (उदाहरण के लिए `(a+)+$`) अस्वीकार करता है। सुरक्षा जाँच में विफल पैटर्न को वाक्यविन्यास की दृष्टि से अमान्य रेगेक्स की तरह चुपचाप छोड़ दिया जाता है। पैटर्न सरल रखें और नेस्टेड क्वांटिफ़ायर से बचें।

## modelCatalog संदर्भ

जब OpenClaw को Plugin रनटाइम लोड करने से पहले प्रदाता मॉडल मेटाडेटा ज्ञात होना चाहिए, तब `modelCatalog` का उपयोग करें। यह निश्चित कैटलॉग पंक्तियों, प्रदाता उपनामों, दमन नियमों और खोज मोड के लिए मैनिफ़ेस्ट-स्वामित्व वाला स्रोत है। रनटाइम रीफ़्रेश अब भी प्रदाता रनटाइम कोड के स्वामित्व में रहता है, लेकिन मैनिफ़ेस्ट कोर को बताता है कि रनटाइम कब आवश्यक है।

```json
{
  "providers": ["openai"],
  "modelCatalog": {
    "providers": {
      "openai": {
        "baseUrl": "https://api.openai.com/v1",
        "api": "openai-responses",
        "models": [
          {
            "id": "gpt-5.4",
            "name": "GPT-5.4",
            "input": ["text", "image"],
            "reasoning": true,
            "contextWindow": 256000,
            "maxTokens": 128000,
            "cost": {
              "input": 1.25,
              "output": 10,
              "cacheRead": 0.125
            },
            "status": "available",
            "tags": ["default"]
          }
        ]
      }
    },
    "aliases": {
      "azure-openai-responses": {
        "provider": "openai",
        "api": "azure-openai-responses"
      }
    },
    "suppressions": [
      {
        "provider": "azure-openai-responses",
        "model": "gpt-5.3-codex-spark",
        "reason": "Azure OpenAI Responses पर उपलब्ध नहीं"
      }
    ],
    "discovery": {
      "openai": "static"
    }
  }
}
```

शीर्ष-स्तरीय फ़ील्ड:

| फ़ील्ड            | प्रकार                                                     | इसका अर्थ                                                                                               |
| ---------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `providers`      | `Record<string, object>`                                 | इस Plugin के स्वामित्व वाली प्रदाता आईडी के लिए कैटलॉग पंक्तियाँ। कुंजियाँ शीर्ष-स्तरीय `providers` में भी होनी चाहिए।       |
| `aliases`        | `Record<string, object>`                                 | प्रदाता उपनाम, जिन्हें कैटलॉग या दमन योजना के लिए किसी स्वामित्व वाले प्रदाता में रिज़ॉल्व होना चाहिए।              |
| `suppressions`   | `object[]`                                               | किसी अन्य स्रोत की मॉडल पंक्तियाँ, जिन्हें यह Plugin प्रदाता-विशिष्ट कारण से दबाता है।                  |
| `discovery`      | `Record<string, "static" \| "refreshable" \| "runtime">` | क्या प्रदाता कैटलॉग को मैनिफ़ेस्ट मेटाडेटा से पढ़ा जा सकता है, कैश में रीफ़्रेश किया जा सकता है, या इसके लिए रनटाइम आवश्यक है। |
| `runtimeAugment` | `boolean`                                                | केवल तभी `true` पर सेट करें, जब प्रदाता रनटाइम को मैनिफ़ेस्ट/कॉन्फ़िग योजना के बाद कैटलॉग पंक्तियाँ जोड़नी हों।       |

`aliases` मॉडल-कैटलॉग योजना के लिए प्रदाता स्वामित्व लुकअप में भाग लेता है। उपनाम लक्ष्य उसी Plugin के स्वामित्व वाले शीर्ष-स्तरीय प्रदाता होने चाहिए। जब प्रदाता-फ़िल्टर की गई सूची किसी उपनाम का उपयोग करती है, तो OpenClaw स्वामी मैनिफ़ेस्ट को पढ़ सकता है और प्रदाता रनटाइम लोड किए बिना उपनाम API/बेस URL ओवरराइड लागू कर सकता है। उपनाम बिना फ़िल्टर वाली कैटलॉग सूचियों का विस्तार नहीं करते; व्यापक सूचियाँ केवल स्वामी की कैनोनिकल प्रदाता पंक्तियाँ उत्सर्जित करती हैं।

`suppressions` पुराने प्रदाता रनटाइम `suppressBuiltInModel` हुक को प्रतिस्थापित करता है। दमन प्रविष्टियों का पालन केवल तभी किया जाता है, जब प्रदाता Plugin के स्वामित्व में हो या किसी स्वामित्व वाले प्रदाता को लक्षित करने वाली `modelCatalog.aliases` कुंजी के रूप में घोषित हो। मॉडल रिज़ॉल्यूशन के दौरान रनटाइम दमन हुक अब कॉल नहीं किए जाते।

प्रदाता फ़ील्ड:

| फ़ील्ड                 | प्रकार                     | इसका अर्थ                                                                                                                                                                                                     |
| --------------------- | ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `baseUrl`             | `string`                 | इस प्रदाता कैटलॉग के मॉडलों के लिए वैकल्पिक डिफ़ॉल्ट बेस URL।                                                                                                                                                    |
| `api`                 | `ModelApi`               | इस प्रदाता कैटलॉग के मॉडलों के लिए वैकल्पिक डिफ़ॉल्ट API अडैप्टर।                                                                                                                                                 |
| `headers`             | `Record<string, string>` | इस प्रदाता कैटलॉग पर लागू होने वाले वैकल्पिक स्थिर हेडर।                                                                                                                                                      |

## modelIdNormalization संदर्भ

प्रदाता रनटाइम लोड होने से पहले होने वाली सस्ती, प्रदाता-स्वामित्व वाली मॉडल-आईडी सफ़ाई के लिए `modelIdNormalization` का उपयोग करें। इससे छोटे मॉडल नामों, प्रदाता-स्थानीय लेगेसी आईडी और प्रॉक्सी उपसर्ग नियमों जैसे उपनाम कोर मॉडल-चयन तालिकाओं के बजाय स्वामी Plugin मैनिफ़ेस्ट में रहते हैं।

```json
{
  "providers": ["anthropic", "openrouter"],
  "modelIdNormalization": {
    "providers": {
      "anthropic": {
        "aliases": {
          "sonnet-4.6": "claude-sonnet-4-6"
        }
      },
      "openrouter": {
        "prefixWhenBare": "openrouter"
      }
    }
  }
}
```

प्रदाता फ़ील्ड:

| फ़ील्ड                                | प्रकार                    | इसका अर्थ                                                                             |
| ------------------------------------ | ----------------------- | ----------------------------------------------------------------------------------------- |
| `aliases`                            | `Record<string,string>` | केस-असंवेदी सटीक मॉडल-आईडी उपनाम। मान उसी रूप में लौटाए जाते हैं, जिसमें वे लिखे गए हैं।                  |
| `stripPrefixes`                      | `string[]`              | उपनाम लुकअप से पहले हटाए जाने वाले उपसर्ग, जो लेगेसी प्रदाता/मॉडल दोहराव के लिए उपयोगी हैं।     |
| `prefixWhenBare`                     | `string`                | जब सामान्यीकृत मॉडल आईडी में पहले से `/` न हो, तब जोड़ा जाने वाला उपसर्ग।                  |
| `prefixWhenBareAfterAliasStartsWith` | `object[]`              | उपनाम लुकअप के बाद सशर्त बेयर-आईडी उपसर्ग नियम, जिन्हें `modelPrefix` और `prefix` के आधार पर कुंजीबद्ध किया गया है। |

## providerEndpoints संदर्भ

प्रदाता रनटाइम लोड होने से पहले जेनेरिक अनुरोध नीति को ज्ञात होने वाले एंडपॉइंट वर्गीकरण के लिए `providerEndpoints` का उपयोग करें। प्रत्येक `endpointClass` का अर्थ अब भी कोर के स्वामित्व में है; Plugin मैनिफ़ेस्ट होस्ट और बेस URL मेटाडेटा के स्वामी हैं।

आधिकारिक रूप से बाह्यीकृत प्रदाता Plugin कोर डिस्ट्रिब्यूशन से बाहर रखे जाते हैं, इसलिए
इंस्टॉल होने तक उनके मैनिफ़ेस्ट दिखाई नहीं देते। उनके `providerEndpoints` को
`scripts/lib/official-external-provider-catalog.json` में भी प्रतिबिंबित किया जाना चाहिए, ताकि
Plugin के बिना एंडपॉइंट वर्गीकरण काम करता रहे; एक अनुबंध परीक्षण
इस प्रतिबिंब को लागू करता है।

| `endpointClass`                | `string`   | ज्ञात कोर एंडपॉइंट वर्ग, जैसे `openrouter`, `moonshot-native`, या `google-vertex`।        |

| `baseUrls`                     | `string[]` | सटीक सामान्यीकृत HTTP(S) आधार URL जो एंडपॉइंट वर्ग से मैप होते हैं।                             |

## secretProviderIntegrations संदर्भ

जब कोई Plugin पुन: प्रयोज्य SecretRef exec प्रदाता प्रीसेट प्रकाशित कर सकता है, तब `secretProviderIntegrations` का उपयोग करें। OpenClaw इस मेटाडेटा को Plugin रनटाइम लोड होने से पहले पढ़ता है, Plugin स्वामित्व को `secrets.providers.<alias>.pluginIntegration` में संग्रहीत करता है, और वास्तविक सीक्रेट समाधान SecretRef रनटाइम पर छोड़ता है। प्रीसेट केवल बंडल किए गए Plugins और प्रबंधित Plugin इंस्टॉल रूट से खोजे गए इंस्टॉल किए गए Plugins के लिए उपलब्ध कराए जाते हैं, जैसे git और ClawHub इंस्टॉल।

```json
{
  "secretProviderIntegrations": {
    "secret-store": {
      "providerAlias": "team-secrets",
      "displayName": "Team secrets",
      "source": "exec",
      "command": "${node}",
      "args": ["./bin/resolve-secrets.mjs"]
    }
  }
}
```

मैप कुंजी इंटीग्रेशन आईडी है। यदि `providerAlias` छोड़ा गया है, तो OpenClaw इंटीग्रेशन आईडी को SecretRef प्रदाता उपनाम के रूप में उपयोग करता है। प्रदाता उपनामों को सामान्य SecretRef प्रदाता उपनाम पैटर्न से मेल खाना चाहिए, उदाहरण के लिए `team-secrets` या `onepassword-work`।

जब कोई ऑपरेटर प्रीसेट चुनता है, तो OpenClaw इस प्रकार का प्रदाता संदर्भ लिखता है:

```json
{
  "secrets": {
    "providers": {
      "team-secrets": {
        "source": "exec",
        "pluginIntegration": {
          "pluginId": "acme-secrets",
          "integrationId": "secret-store"
        }
      }
    }
  }
}
```

स्टार्टअप/रीलोड पर, OpenClaw वर्तमान Plugin मैनिफ़ेस्ट मेटाडेटा लोड करके, स्वामी Plugin के इंस्टॉल और सक्रिय होने की जाँच करके, तथा मैनिफ़ेस्ट से exec कमांड निर्मित करके उस प्रदाता को हल करता है। Plugin को अक्षम करने या हटाने से सक्रिय SecretRefs के लिए प्रदाता निरस्त हो जाता है। जो ऑपरेटर स्वतंत्र exec कॉन्फ़िगरेशन चाहते हैं, वे अब भी मैन्युअल `command`/`args` प्रदाता सीधे लिख सकते हैं।

वर्तमान में केवल `source: "exec"` प्रीसेट समर्थित हैं। `command` को `${node}` होना चाहिए, और `args[0]` को `./` Plugin-रूट-सापेक्ष रिज़ॉल्वर स्क्रिप्ट होना चाहिए। OpenClaw स्टार्टअप/रीलोड पर इसे वर्तमान Node निष्पादन योग्य फ़ाइल और Plugin के भीतर स्क्रिप्ट के निरपेक्ष पथ में रूपांतरित करता है। `--require`, `--import`, `--loader`, `--env-file`, `--eval`, और `--print` जैसे Node विकल्प मैनिफ़ेस्ट प्रीसेट अनुबंध का भाग नहीं हैं। जिन ऑपरेटरों को गैर-Node कमांड चाहिए, वे स्वतंत्र मैन्युअल exec प्रदाताओं को सीधे कॉन्फ़िगर कर सकते हैं।

OpenClaw मैनिफ़ेस्ट प्रीसेट के लिए `trustedDirs` को Plugin रूट से और `${node}` प्रीसेट के लिए वर्तमान Node निष्पादन योग्य फ़ाइल की डायरेक्टरी से प्राप्त करता है। मैनिफ़ेस्ट में लिखे गए `trustedDirs` अनदेखे किए जाते हैं। `timeoutMs`, `noOutputTimeoutMs`, `maxOutputBytes`, `jsonOnly`, `env`, `passEnv`, और `allowInsecurePath` जैसे अन्य exec प्रदाता विकल्प सामान्य SecretRef exec प्रदाता कॉन्फ़िगरेशन में यथावत भेजे जाते हैं।

## modelPricing संदर्भ

जब किसी प्रदाता को रनटाइम लोड होने से पहले नियंत्रण-प्लेन मूल्य निर्धारण व्यवहार की आवश्यकता हो, तब `modelPricing` का उपयोग करें। Gateway मूल्य निर्धारण कैश प्रदाता रनटाइम कोड आयात किए बिना यह मेटाडेटा पढ़ता है।

```json
    "providers": {
      "ollama": {
        "external": false
      },
      "openrouter": {
        "openRouter": {
          "passthroughProviderModel": true
        },
        "liteLLM": false
      }
    }
  }
}
```

प्रदाता फ़ील्ड:

| फ़ील्ड        | प्रकार              | इसका अर्थ                                                                                      |
| ------------ | ----------------- | -------------------------------------------------------------------------------------------------- |

| `liteLLM`    | `false \| object` | LiteLLM मूल्य निर्धारण लुकअप मैपिंग। `false` इस प्रदाता के लिए LiteLLM लुकअप अक्षम करता है।                 |

### OpenClaw प्रदाता सूचकांक

OpenClaw प्रदाता सूचकांक उन प्रदाताओं के लिए OpenClaw-स्वामित्व वाला पूर्वावलोकन मेटाडेटा है जिनके Plugins अभी इंस्टॉल न हुए हों। यह Plugin मैनिफ़ेस्ट का भाग नहीं है। Plugin मैनिफ़ेस्ट इंस्टॉल किए गए Plugin का प्रामाणिक स्रोत बने रहते हैं। प्रदाता सूचकांक आंतरिक फ़ॉलबैक अनुबंध है, जिसका उपयोग भविष्य के इंस्टॉल-योग्य प्रदाता और इंस्टॉल-पूर्व मॉडल चयनकर्ता सतहें तब करेंगी जब कोई प्रदाता Plugin इंस्टॉल न हो।

कैटलॉग प्रामाणिकता क्रम:

1. उपयोगकर्ता कॉन्फ़िगरेशन।
2. इंस्टॉल किए गए Plugin का मैनिफ़ेस्ट `modelCatalog`।
3. स्पष्ट रीफ़्रेश से प्राप्त मॉडल कैटलॉग कैश।
4. OpenClaw प्रदाता सूचकांक की पूर्वावलोकन पंक्तियाँ।

प्रदाता सूचकांक में सीक्रेट, सक्षम स्थिति, रनटाइम हुक, या सक्रिय खाता-विशिष्ट मॉडल डेटा नहीं होना चाहिए। इसके पूर्वावलोकन कैटलॉग Plugin मैनिफ़ेस्ट के समान `modelCatalog` प्रदाता पंक्ति आकार का उपयोग करते हैं, लेकिन इन्हें स्थिर प्रदर्शन मेटाडेटा तक सीमित रहना चाहिए, जब तक कि `api`, `baseUrl`, मूल्य निर्धारण, या संगतता फ़्लैग जैसे रनटाइम अडैप्टर फ़ील्ड को जानबूझकर इंस्टॉल किए गए Plugin मैनिफ़ेस्ट के अनुरूप न रखा जाए। सक्रिय `/models` खोज वाले प्रदाताओं को सामान्य सूचीकरण या ऑनबोर्डिंग के दौरान प्रदाता API कॉल करने के बजाय स्पष्ट मॉडल कैटलॉग कैश पथ के माध्यम से रीफ़्रेश की गई पंक्तियाँ लिखनी चाहिए।

प्रदाता सूचकांक प्रविष्टियों में उन प्रदाताओं के लिए इंस्टॉल-योग्य Plugin मेटाडेटा भी हो सकता है जिनका Plugin कोर से बाहर जा चुका है या अन्यथा अभी इंस्टॉल नहीं हुआ है। यह मेटाडेटा चैनल कैटलॉग पैटर्न को प्रतिबिंबित करता है: पैकेज नाम, npm इंस्टॉल विनिर्देश, अपेक्षित अखंडता, और कम-लागत प्रमाणीकरण-विकल्प लेबल किसी इंस्टॉल-योग्य सेटअप विकल्प को दिखाने के लिए पर्याप्त हैं। Plugin इंस्टॉल हो जाने पर, उसका मैनिफ़ेस्ट प्राथमिकता लेता है और उस प्रदाता के लिए प्रदाता सूचकांक प्रविष्टि अनदेखी कर दी जाती है।

`openclaw doctor --fix` पुराने शीर्ष-स्तरीय मैनिफ़ेस्ट क्षमता कुंजियों के एक छोटे, बंद समूह को `contracts.*` में माइग्रेट करता है: `speechProviders`, `mediaUnderstandingProviders`, `imageGenerationProviders`, और `tools`। इनमें से कोई भी (या कोई अन्य क्षमता सूची) अब शीर्ष-स्तरीय मैनिफ़ेस्ट फ़ील्ड के रूप में नहीं पढ़ी जाती; सामान्य मैनिफ़ेस्ट लोडिंग उन्हें केवल `contracts` के अंतर्गत पहचानती है।

## मैनिफ़ेस्ट बनाम package.json

दोनों फ़ाइलें अलग-अलग कार्य करती हैं:

| फ़ाइल                   | इसका उपयोग                                                                                                                       |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `openclaw.plugin.json` | खोज, कॉन्फ़िगरेशन सत्यापन, प्रमाणीकरण-विकल्प मेटाडेटा, और ऐसे UI संकेतों के लिए जो Plugin कोड चलने से पहले मौजूद होने चाहिए                         |
| `package.json`         | npm मेटाडेटा, निर्भरता इंस्टॉलेशन, और एंट्रीपॉइंट, इंस्टॉल गेटिंग, सेटअप, या कैटलॉग मेटाडेटा के लिए प्रयुक्त `openclaw` ब्लॉक |

यदि यह स्पष्ट न हो कि मेटाडेटा का कोई अंश कहाँ होना चाहिए, तो इस नियम का उपयोग करें:

- यदि OpenClaw को Plugin कोड लोड करने से पहले इसे जानना आवश्यक है, तो इसे `openclaw.plugin.json` में रखें
- यदि यह पैकेजिंग, एंट्री फ़ाइलों, या npm इंस्टॉल व्यवहार के बारे में है, तो इसे `package.json` में रखें

### खोज को प्रभावित करने वाले package.json फ़ील्ड

कुछ पूर्व-रनटाइम Plugin मेटाडेटा जानबूझकर `openclaw.plugin.json` के बजाय `package.json` में `openclaw` ब्लॉक के अंतर्गत रहता है। `openclaw.bundle` और `openclaw.bundle.json` OpenClaw Plugin अनुबंध नहीं हैं; नेटिव Plugins को `openclaw.plugin.json` तथा नीचे दिए गए समर्थित `package.json#openclaw` फ़ील्ड का उपयोग करना चाहिए।

महत्वपूर्ण उदाहरण:

| फ़ील्ड                                                                                      | इसका अर्थ                                                                                                                                                                        |

| `openclaw.runtimeExtensions`                                                               | इंस्टॉल किए गए पैकेजों के लिए बिल्ट JavaScript रनटाइम एंट्रीपॉइंट घोषित करता है। इसे Plugin पैकेज डायरेक्टरी के अंदर ही रहना चाहिए।                                                                 |
| `openclaw.setupEntry`                                                                      | ऑनबोर्डिंग, स्थगित चैनल स्टार्टअप और केवल-पढ़ने योग्य चैनल स्थिति/SecretRef खोज के दौरान उपयोग किया जाने वाला हल्का, केवल सेटअप एंट्रीपॉइंट। इसे Plugin पैकेज डायरेक्टरी के अंदर ही रहना चाहिए। |
| `openclaw.runtimeSetupEntry`                                                               | इंस्टॉल किए गए पैकेजों के लिए बिल्ट JavaScript सेटअप एंट्रीपॉइंट घोषित करता है। इसके लिए `setupEntry` आवश्यक है, इसका मौजूद होना अनिवार्य है और इसे Plugin पैकेज डायरेक्टरी के अंदर ही रहना चाहिए।                         |

| `openclaw.channel.persistedAuthState`                                                      | हल्का स्थायी-प्रमाणीकरण जाँचकर्ता मेटाडेटा, जो पूरा चैनल रनटाइम लोड किए बिना यह बता सकता है कि "क्या किसी चीज़ में पहले से साइन इन किया गया है?"                                               |

| `openclaw.install.allowInvalidConfigRecovery`                                              | कॉन्फ़िगरेशन अमान्य होने पर सीमित बंडल-Plugin पुनः इंस्टॉल पुनर्प्राप्ति पाथ की अनुमति देता है।                                                                                                       |

मैनिफ़ेस्ट मेटाडेटा तय करता है कि रनटाइम लोड होने से पहले ऑनबोर्डिंग में कौन-से प्रदाता/चैनल/सेटअप विकल्प दिखाई देंगे। जब उपयोगकर्ता उन विकल्पों में से किसी एक को चुनता है, तो `package.json#openclaw.install` ऑनबोर्डिंग को बताता है कि उस Plugin को कैसे प्राप्त या सक्षम करना है। इंस्टॉल संकेतों को `openclaw.plugin.json` में न ले जाएँ।

गैर-बंडल Plugin स्रोतों के लिए इंस्टॉल और मैनिफ़ेस्ट रजिस्ट्री लोडिंग के दौरान `openclaw.install.minHostVersion` लागू किया जाता है। अमान्य मान अस्वीकार कर दिए जाते हैं; अधिक नए लेकिन मान्य मान पुराने होस्ट पर बाहरी Plugins को छोड़ देते हैं। बंडल स्रोत Plugins को होस्ट चेकआउट के साथ समान संस्करण वाला माना जाता है।

`openclaw.install.requiredPlatformPackages` उन npm पैकेजों के लिए है जो वैकल्पिक, प्लेटफ़ॉर्म-विशिष्ट उपनामों के माध्यम से आवश्यक मूल बाइनरी उपलब्ध कराते हैं। प्रत्येक समर्थित प्लेटफ़ॉर्म उपनाम के लिए केवल npm पैकेज नाम सूचीबद्ध करें। npm इंस्टॉल के दौरान, OpenClaw केवल उस घोषित उपनाम को सत्यापित करता है जिसकी lockfile बाधाएँ वर्तमान होस्ट से मेल खाती हैं। यदि npm सफलता की सूचना देता है लेकिन उस उपनाम को छोड़ देता है, तो OpenClaw नए कैश के साथ एक बार पुनः प्रयास करता है और उपनाम फिर भी अनुपस्थित रहने पर इंस्टॉल वापस ले लेता है।

गैर-बंडल Plugin स्रोतों के लिए पैकेज इंस्टॉल के दौरान `openclaw.compat.pluginApi` लागू किया जाता है। इसे उस OpenClaw Plugin SDK/रनटाइम API न्यूनतम सीमा के लिए उपयोग करें जिसके विरुद्ध पैकेज बनाया गया था। जब किसी Plugin पैकेज को अधिक नया API चाहिए, लेकिन वह अन्य प्रवाहों के लिए कम इंस्टॉल संकेत बनाए रखता है, तब यह `minHostVersion` से अधिक कठोर हो सकता है। आधिकारिक OpenClaw रिलीज़ सिंक मौजूदा आधिकारिक Plugin API न्यूनतम सीमाओं को डिफ़ॉल्ट रूप से OpenClaw रिलीज़ संस्करण तक बढ़ाता है, लेकिन जब पैकेज जानबूझकर पुराने होस्ट का समर्थन करता है, तब केवल-Plugin रिलीज़ कम न्यूनतम सीमा बनाए रख सकती हैं। केवल पैकेज संस्करण को संगतता अनुबंध के रूप में उपयोग न करें। `peerDependencies.openclaw` npm पैकेज मेटाडेटा ही रहता है; इंस्टॉल संगतता निर्णयों के लिए OpenClaw `openclaw.compat.pluginApi` अनुबंध का उपयोग करता है।

जब Plugin ClawHub पर प्रकाशित हो, तो आधिकारिक माँग-पर-इंस्टॉल मेटाडेटा में `clawhubSpec` का उपयोग होना चाहिए; ऑनबोर्डिंग इसे पसंदीदा रिमोट स्रोत मानता है और इंस्टॉल के बाद ClawHub आर्टिफ़ैक्ट तथ्य दर्ज करता है। जो पैकेज अभी तक ClawHub पर स्थानांतरित नहीं हुए हैं, उनके लिए `npmSpec` संगतता फ़ॉलबैक बना रहता है।

सटीक npm संस्करण पिनिंग पहले से `npmSpec` में रहती है, उदाहरण के लिए `"npmSpec": "@wecom/wecom-openclaw-plugin@1.2.3"`। आधिकारिक बाहरी कैटलॉग प्रविष्टियों को सटीक विनिर्देशों के साथ `expectedIntegrity` जोड़ना चाहिए, ताकि प्राप्त npm आर्टिफ़ैक्ट के पिन किए गए रिलीज़ से मेल न खाने पर अपडेट प्रवाह विफल होकर बंद हो जाएँ। संगतता के लिए इंटरैक्टिव ऑनबोर्डिंग अब भी विश्वसनीय रजिस्ट्री npm विनिर्देश प्रदान करता है, जिनमें केवल पैकेज नाम और dist-tags शामिल हैं। कैटलॉग निदान सटीक, परिवर्तनशील, अखंडता-पिन किए गए, अखंडता-विहीन, पैकेज-नाम बेमेल और अमान्य डिफ़ॉल्ट-विकल्प स्रोतों के बीच अंतर कर सकता है। जब `expectedIntegrity` मौजूद हो लेकिन कोई मान्य npm स्रोत न हो जिसे वह पिन कर सके, तब यह चेतावनी भी देता है। जब `expectedIntegrity` मौजूद हो, तो इंस्टॉल/अपडेट प्रवाह इसे लागू करते हैं; इसे छोड़ देने पर रजिस्ट्री रिज़ॉल्यूशन बिना अखंडता पिन के दर्ज किया जाता है।

जब स्थिति, चैनल सूची या SecretRef स्कैन को पूरा रनटाइम लोड किए बिना कॉन्फ़िगर किए गए खातों की पहचान करनी हो, तब चैनल Plugins को `openclaw.setupEntry` प्रदान करना चाहिए। सेटअप प्रविष्टि को चैनल मेटाडेटा के साथ सेटअप-सुरक्षित कॉन्फ़िगरेशन, स्थिति और सीक्रेट एडाप्टर उपलब्ध कराने चाहिए; नेटवर्क क्लाइंट, Gateway लिसनर और ट्रांसपोर्ट रनटाइम को मुख्य एक्सटेंशन एंट्रीपॉइंट में रखें।

रनटाइम एंट्रीपॉइंट फ़ील्ड, स्रोत एंट्रीपॉइंट फ़ील्ड की पैकेज-सीमा जाँचों को ओवरराइड नहीं करते। उदाहरण के लिए, `openclaw.runtimeExtensions` पैकेज से बाहर निकलने वाले `openclaw.extensions` पाथ को लोड करने योग्य नहीं बना सकता।

`openclaw.install.allowInvalidConfigRecovery` जानबूझकर सीमित है। यह मनमाने ढंग से खराब कॉन्फ़िगरेशन को इंस्टॉल करने योग्य नहीं बनाता। वर्तमान में यह इंस्टॉल प्रवाहों को केवल विशिष्ट पुराने बंडल-Plugin अपग्रेड विफलताओं से उबरने देता है, जैसे अनुपस्थित बंडल Plugin पाथ या उसी बंडल Plugin के लिए पुरानी `channels.<id>` प्रविष्टि। असंबंधित कॉन्फ़िगरेशन त्रुटियाँ अब भी इंस्टॉल को रोकती हैं और ऑपरेटरों को `openclaw doctor --fix` पर भेजती हैं।

`openclaw.channel.persistedAuthState` एक छोटे जाँचकर्ता मॉड्यूल के लिए पैकेज मेटाडेटा है:

```json
{
  "openclaw": {
    "channel": {
      "id": "whatsapp",
      "persistedAuthState": {
        "specifier": "./auth-presence",
        "exportName": "hasAnyWhatsAppAuth"
      }
    }
  }
}
```

जब पूरा चैनल Plugin लोड होने से पहले सेटअप, डॉक्टर, स्थिति या केवल-पढ़ने योग्य उपस्थिति प्रवाहों को कम लागत वाली हाँ/नहीं प्रमाणीकरण जाँच चाहिए, तब इसका उपयोग करें। स्थायी प्रमाणीकरण अवस्था, कॉन्फ़िगर की गई चैनल अवस्था नहीं है: Plugins को स्वतः सक्षम करने, रनटाइम निर्भरताएँ सुधारने या यह तय करने के लिए कि चैनल रनटाइम लोड होना चाहिए या नहीं, इस मेटाडेटा का उपयोग न करें। लक्षित एक्सपोर्ट एक छोटा फ़ंक्शन होना चाहिए जो केवल स्थायी अवस्था पढ़ता हो; इसे पूरे चैनल रनटाइम बैरल से न गुजारें।

`openclaw.channel.configuredState` कम लागत वाली कॉन्फ़िगर-जाँचों का समर्थन करता है। जब एनवायरनमेंट वेरिएबल पर्याप्त हों, तो घोषणात्मक एनवायरनमेंट मेटाडेटा को प्राथमिकता दें:

```json
{
  "openclaw": {
    "channel": {
      "id": "telegram",
      "configuredState": {
        "env": {
          "allOf": ["TELEGRAM_BOT_TOKEN"]
        }
      }
    }
  }
}
```

जब सूचीबद्ध प्रत्येक वेरिएबल आवश्यक हो, तब `env.allOf` का उपयोग करें और जब कोई भी एक गैर-रिक्त वेरिएबल पर्याप्त हो, तब `env.anyOf` का उपयोग करें। यदि किसी छोटे गैर-रनटाइम जाँच को एनवायरनमेंट मेटाडेटा से अधिक की आवश्यकता हो, तो `persistedAuthState` के लिए दिखाए अनुसार `specifier` के साथ `exportName` का उपयोग करें; जब `env` मौजूद हो, तो OpenClaw उस मॉड्यूल को लोड किए बिना इसका उपयोग करता है। यदि जाँच को पूर्ण कॉन्फ़िगरेशन रिज़ॉल्यूशन या वास्तविक चैनल रनटाइम चाहिए, तो उस लॉजिक को इसके बजाय Plugin के `config.hasConfiguredState` हुक में रखें।

## खोज प्राथमिकता (डुप्लिकेट Plugin आईडी)

OpenClaw तीन मूल स्थानों से Plugins खोजता है, जिनकी जाँच इस क्रम में होती है: OpenClaw के साथ भेजे गए बंडल Plugins, वैश्विक इंस्टॉल रूट (`~/.openclaw/extensions`) और वर्तमान वर्कस्पेस रूट (`<workspace>/.openclaw/extensions`), साथ ही कोई भी स्पष्ट `plugins.load.paths` प्रविष्टियाँ।

यदि दो खोजों में समान `id` हो, तो केवल **सर्वोच्च-प्राथमिकता** वाला मैनिफ़ेस्ट रखा जाता है; कम प्राथमिकता वाले डुप्लिकेट को उसके साथ लोड करने के बजाय हटा दिया जाता है। सर्वोच्च से निम्नतम तक प्राथमिकता:

1. **कॉन्फ़िगरेशन द्वारा चयनित** — `plugins.entries.<id>` में स्पष्ट रूप से पिन किया गया पाथ
2. **ट्रैक किए गए इंस्टॉल रिकॉर्ड से मेल खाता वैश्विक इंस्टॉल** — `openclaw plugin install`/`openclaw plugin update` के माध्यम से इंस्टॉल किया गया ऐसा Plugin, जिसे OpenClaw की इंस्टॉल ट्रैकिंग उसी आईडी के लिए पहचानती है, भले ही वह आईडी किसी बंडल Plugin की भी हो
3. **बंडल** — OpenClaw के साथ भेजे गए Plugins
4. **वर्कस्पेस** — वर्तमान वर्कस्पेस के सापेक्ष खोजे गए Plugins
5. कोई अन्य खोजा गया उम्मीदवार

निहितार्थ:

- वर्कस्पेस या ग्लोबल रूट में अनट्रैक्ड पड़ी किसी बंडल किए गए Plugin की फ़ोर्क की गई या पुरानी प्रति, बंडल किए गए बिल्ड को शैडो नहीं करेगी।
- किसी बंडल किए गए Plugin को ओवरराइड करने के लिए, उस आईडी हेतु या तो `openclaw plugin install` चलाएँ, ताकि ट्रैक किया गया ग्लोबल इंस्टॉलेशन बंडल की गई प्रति से उच्च प्राथमिकता पाए, या `plugins.entries.<id>` के माध्यम से किसी विशिष्ट पथ को पिन करें, ताकि वह कॉन्फ़िगरेशन द्वारा चुनी गई प्राथमिकता के आधार पर प्रभावी हो।
- डुप्लिकेट प्रतियों को हटाए जाने की जानकारी लॉग की जाती है, ताकि Doctor और स्टार्टअप डायग्नोस्टिक्स त्यागी गई प्रति की ओर संकेत कर सकें।
- कॉन्फ़िगरेशन द्वारा चुने गए डुप्लिकेट ओवरराइड को डायग्नोस्टिक्स में स्पष्ट ओवरराइड के रूप में वर्णित किया जाता है, लेकिन चेतावनी फिर भी दी जाती है, ताकि पुराने फ़ोर्क और अनजाने शैडो दिखाई देते रहें।

## JSON Schema की आवश्यकताएँ

- **प्रत्येक Plugin के साथ JSON Schema अवश्य दिया जाना चाहिए**, भले ही वह कोई कॉन्फ़िगरेशन स्वीकार न करता हो।
- एक खाली स्कीमा स्वीकार्य है (उदाहरण के लिए, `{ "type": "object", "additionalProperties": false }`)।
- स्कीमा का सत्यापन कॉन्फ़िगरेशन पढ़ते/लिखते समय किया जाता है, रनटाइम पर नहीं।
- किसी बंडल किए गए Plugin को नई कॉन्फ़िगरेशन कुंजियों के साथ विस्तारित या फ़ोर्क करते समय, उसी समय उस Plugin के `openclaw.plugin.json` `configSchema` को भी अपडेट करें। बंडल किए गए Plugin के स्कीमा सख़्त होते हैं, इसलिए `myNewKey` को `configSchema.properties` में जोड़े बिना उपयोगकर्ता कॉन्फ़िगरेशन में `plugins.entries.<id>.config.myNewKey` जोड़ना, Plugin रनटाइम लोड होने से पहले ही अस्वीकार कर दिया जाएगा।

स्कीमा विस्तार का उदाहरण:

```json
{
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "myNewKey": {
        "type": "string"
      }
    }
  }
}
```

## सत्यापन व्यवहार

- अज्ञात `channels.*` कुंजियाँ **त्रुटियाँ** हैं, जब तक कि चैनल आईडी किसी Plugin मैनिफ़ेस्ट द्वारा घोषित न की गई हो। यदि वही आईडी `plugins.allow`, `plugins.entries`, या `plugins.installs` में भी दिखाई देती है (एक ऐसा Plugin जिसका संदर्भ दिया गया है, लेकिन जो वर्तमान में खोजने योग्य नहीं है), तो OpenClaw इसे घटाकर **चेतावनी** कर देता है।
- अज्ञात Plugin आईडी का संदर्भ देने वाले `plugins.entries.<id>`, `plugins.allow`, और `plugins.deny` **चेतावनियाँ** ("पुरानी कॉन्फ़िगरेशन प्रविष्टि अनदेखी की गई") हैं, त्रुटियाँ नहीं, ताकि अपग्रेड और हटाए गए/पुनर्नामित Plugin Gateway स्टार्टअप को अवरुद्ध न करें।
- अज्ञात Plugin आईडी का संदर्भ देने वाला `plugins.slots.memory` एक **त्रुटि** है, ज्ञात `memory-lancedb` आधिकारिक बाहरी Plugin को छोड़कर, जिसके लिए इसके बजाय चेतावनी दी जाती है।
- यदि कोई Plugin इंस्टॉल है लेकिन उसका मैनिफ़ेस्ट या स्कीमा खराब अथवा अनुपस्थित है, तो सत्यापन विफल हो जाता है और Doctor Plugin त्रुटि की रिपोर्ट करता है।
- यदि Plugin कॉन्फ़िगरेशन मौजूद है लेकिन Plugin **अक्षम** है, तो कॉन्फ़िगरेशन रखा जाता है और Doctor तथा लॉग में एक **चेतावनी** दिखाई जाती है।

पूर्ण `plugins.*` स्कीमा के लिए [कॉन्फ़िगरेशन संदर्भ](/hi/gateway/configuration) देखें।

## टिप्पणियाँ

- स्थानीय फ़ाइल सिस्टम से लोड किए जाने वाले Plugin सहित, **मूल OpenClaw Plugin के लिए मैनिफ़ेस्ट आवश्यक है**। रनटाइम अब भी Plugin मॉड्यूल को अलग से लोड करता है; मैनिफ़ेस्ट केवल खोज और सत्यापन के लिए है।
- मूल मैनिफ़ेस्ट JSON5 के साथ पार्स किए जाते हैं, इसलिए टिप्पणियाँ, अंतिम कॉमा और उद्धरण-रहित कुंजियाँ स्वीकार की जाती हैं, बशर्ते अंतिम मान अब भी एक ऑब्जेक्ट हो।
- मैनिफ़ेस्ट लोडर केवल दस्तावेज़ीकृत मैनिफ़ेस्ट फ़ील्ड पढ़ता है। कस्टम शीर्ष-स्तरीय कुंजियों से बचें।
- यदि किसी Plugin को उनकी आवश्यकता न हो, तो `channels`, `providers`, `cliBackends`, और `skills` सभी को छोड़ा जा सकता है।
- `providerCatalogEntry` को हल्का रखना आवश्यक है और इसे व्यापक रनटाइम कोड आयात नहीं करना चाहिए; इसका उपयोग स्थिर प्रोवाइडर कैटलॉग मेटाडेटा या सीमित खोज डिस्क्रिप्टर के लिए करें, अनुरोध-समय निष्पादन के लिए नहीं।
- विशिष्ट Plugin प्रकार `plugins.slots.*` के माध्यम से चुने जाते हैं: `kind: "memory"` को `plugins.slots.memory` के माध्यम से (डिफ़ॉल्ट `memory-core`), और `kind: "context-engine"` को `plugins.slots.contextEngine` के माध्यम से (डिफ़ॉल्ट `legacy`)।
- इस मैनिफ़ेस्ट में विशिष्ट Plugin प्रकार घोषित करें। रनटाइम-एंट्री `OpenClawPluginDefinition.kind` अप्रचलित है और केवल पुराने Plugin के लिए संगतता फ़ॉलबैक के रूप में बनी हुई है।
- `setup.providers[].envVars` में एनवायरनमेंट-वेरिएबल मेटाडेटा केवल घोषणात्मक है। स्टेटस, ऑडिट, Cron डिलीवरी सत्यापन और अन्य केवल-पठन सतहें किसी एनवायरनमेंट वेरिएबल को कॉन्फ़िगर मानने से पहले अब भी Plugin विश्वास और प्रभावी सक्रियण नीति लागू करती हैं।
- प्रोवाइडर कोड की आवश्यकता वाले रनटाइम विज़ार्ड मेटाडेटा के लिए, [प्रोवाइडर रनटाइम हुक](/hi/plugins/architecture-internals#provider-runtime-hooks) देखें।
- यदि आपका Plugin मूल मॉड्यूल पर निर्भर करता है, तो बिल्ड चरणों और पैकेज-मैनेजर की किसी भी अनुमति-सूची संबंधी आवश्यकता का दस्तावेज़ीकरण करें (उदाहरण के लिए, pnpm `allow-build-scripts` + `pnpm rebuild <package>`)।
