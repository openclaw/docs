## सार्वजनिक क्षमता मॉडल

क्षमताएँ OpenClaw के भीतर सार्वजनिक **नेटिव Plugin** मॉडल हैं। प्रत्येक नेटिव OpenClaw Plugin एक या अधिक क्षमता प्रकारों के लिए पंजीकरण करता है:

| क्षमता                 | पंजीकरण विधि                                     | उदाहरण Plugin                  |
| ---------------------- | ------------------------------------------------ | ------------------------------ |
| टेक्स्ट अनुमान         | `api.registerProvider(...)`                      | `anthropic`, `openai`          |
| CLI अनुमान बैकएंड      | `api.registerCliBackend(...)`                    | `anthropic`, `openai`          |
| एम्बेडिंग              | `api.registerEmbeddingProvider(...)`             | प्रोवाइडर-स्वामित्व वाले वेक्टर Plugin |
| वाक्                   | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`      |
| रीयलटाइम ट्रांसक्रिप्शन | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                       |
| रीयलटाइम वॉइस          | `api.registerRealtimeVoiceProvider(...)`         | `google`, `openai`             |
| मीडिया की समझ          | `api.registerMediaUnderstandingProvider(...)`    | `google`, `openai`             |
| ट्रांसक्रिप्ट स्रोत     | `api.registerTranscriptSourceProvider(...)`      | `discord`                      |
| इमेज जनरेशन            | `api.registerImageGenerationProvider(...)`       | `fal`, `google`, `openai`      |
| संगीत जनरेशन           | `api.registerMusicGenerationProvider(...)`       | `fal`, `google`, `minimax`     |
| वीडियो जनरेशन          | `api.registerVideoGenerationProvider(...)`       | `fal`, `google`, `qwen`        |
| वेब फ़ेच               | `api.registerWebFetchProvider(...)`              | `firecrawl`                    |
| वेब खोज                | `api.registerWebSearchProvider(...)`             | `brave`, `firecrawl`, `google` |
| चैनल / मैसेजिंग        | `api.registerChannel(...)`                       | `matrix`, `msteams`            |
| Gateway डिस्कवरी       | `api.registerGatewayDiscoveryService(...)`       | `bonjour`                      |

<Note>
कोई ऐसा Plugin जो शून्य क्षमताएँ पंजीकृत करता है, लेकिन हुक, टूल, डिस्कवरी सेवाएँ या बैकग्राउंड सेवाएँ प्रदान करता है, **केवल-लेगेसी-हुक** Plugin है। यह पैटर्न अभी भी पूरी तरह समर्थित है।
</Note>

### बाहरी संगतता का दृष्टिकोण

क्षमता मॉडल को कोर में शामिल किया जा चुका है और आज बंडल किए गए/नेटिव Plugin इसका उपयोग करते हैं, लेकिन बाहरी Plugin संगतता के लिए अब भी "यह एक्सपोर्ट किया गया है, इसलिए यह स्थिर है" की तुलना में अधिक कठोर मानदंड की आवश्यकता है।

| Plugin की स्थिति                                | मार्गदर्शन                                                                                                    |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| मौजूदा बाहरी Plugin                              | हुक-आधारित इंटीग्रेशन को कार्यशील रखें; यह संगतता की आधाररेखा है।                                               |
| नए बंडल किए गए/नेटिव Plugin                     | विक्रेता-विशिष्ट आंतरिक पहुँच या नए केवल-हुक डिज़ाइन के बजाय स्पष्ट क्षमता पंजीकरण को प्राथमिकता दें।              |
| क्षमता पंजीकरण अपनाने वाले बाहरी Plugin          | अनुमति है, लेकिन जब तक दस्तावेज़ उन्हें स्थिर न बताएँ, क्षमता-विशिष्ट सहायक सतहों को विकसित होती हुई मानें।        |

क्षमता पंजीकरण अभीष्ट दिशा है। संक्रमण के दौरान बाहरी Plugin के लिए लेगेसी हुक बिना किसी टूट-फूट वाला सबसे सुरक्षित मार्ग बने हुए हैं। सभी एक्सपोर्ट किए गए सहायक सबपाथ समान नहीं हैं — आकस्मिक सहायक एक्सपोर्ट के बजाय सीमित, दस्तावेज़ीकृत अनुबंधों को प्राथमिकता दें।

### Plugin के आकार

OpenClaw प्रत्येक लोड किए गए Plugin को उसके वास्तविक पंजीकरण व्यवहार के आधार पर एक आकार में वर्गीकृत करता है (केवल स्थिर मेटाडेटा के आधार पर नहीं):

<AccordionGroup>
  <Accordion title="साधारण-क्षमता">
    ठीक एक क्षमता प्रकार पंजीकृत करता है (उदाहरण के लिए `arcee` या `chutes` जैसा केवल-प्रोवाइडर Plugin)।
  </Accordion>
  <Accordion title="हाइब्रिड-क्षमता">
    कई क्षमता प्रकार पंजीकृत करता है (उदाहरण के लिए `openai` टेक्स्ट अनुमान, वाक्, मीडिया की समझ और इमेज जनरेशन का स्वामी है)।
  </Accordion>
  <Accordion title="केवल-हुक">
    केवल हुक (टाइप किए गए या कस्टम) पंजीकृत करता है, कोई क्षमता, टूल, कमांड या सेवा नहीं।
  </Accordion>
  <Accordion title="गैर-क्षमता">
    टूल, कमांड, सेवाएँ या रूट पंजीकृत करता है, लेकिन कोई क्षमता नहीं।
  </Accordion>
</AccordionGroup>

किसी Plugin का आकार और उसकी क्षमताओं का विवरण देखने के लिए `openclaw plugins inspect <id>` का उपयोग करें। विवरण के लिए [CLI संदर्भ](/hi/cli/plugins#inspect) देखें।

### संगतता संकेत

`openclaw doctor`, `openclaw plugins inspect <id>`, `openclaw status --all`, और `openclaw plugins doctor` ये संगतता सूचनाएँ प्रदर्शित करते हैं:

| संकेत                                       | अर्थ                                                                                                                   |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **कॉन्फ़िग मान्य**                         | कॉन्फ़िग ठीक से पार्स होता है और Plugin रिज़ॉल्व होते हैं                                                               |
| **केवल-हुक** (जानकारी)                    | Plugin केवल हुक पंजीकृत करता है; यह समर्थित मार्ग है, लेकिन इसे अभी क्षमता पंजीकरण में माइग्रेट नहीं किया गया है          |
| **बहिष्कृत मेमोरी-एम्बेडिंग API** (चेतावनी) | गैर-बंडल Plugin `registerEmbeddingProvider` के बजाय पुराने मेमोरी-विशिष्ट एम्बेडिंग प्रोवाइडर API का उपयोग करता है |
| **गंभीर त्रुटि**                           | कॉन्फ़िग अमान्य है या Plugin लोड नहीं हो सका                                                                             |

इनमें से कोई भी परामर्श/चेतावनी संकेत आज आपके Plugin को नहीं तोड़ता। ये संकेत `openclaw status --all` और `openclaw plugins doctor` में भी दिखाई देते हैं।

## आर्किटेक्चर का अवलोकन

OpenClaw के Plugin सिस्टम की चार परतें हैं:

<Steps>
  <Step title="मैनिफ़ेस्ट + डिस्कवरी">
    OpenClaw कॉन्फ़िगर किए गए पाथ, वर्कस्पेस रूट, वैश्विक Plugin रूट और बंडल किए गए Plugin से संभावित Plugin खोजता है। डिस्कवरी पहले नेटिव `openclaw.plugin.json` मैनिफ़ेस्ट और समर्थित बंडल मैनिफ़ेस्ट पढ़ती है।
  </Step>
  <Step title="सक्षमीकरण + सत्यापन">
    कोर तय करता है कि खोजा गया Plugin सक्षम, अक्षम, अवरुद्ध है या मेमोरी जैसे किसी विशिष्ट स्लॉट के लिए चुना गया है।
  </Step>
  <Step title="रनटाइम लोडिंग">
    नेटिव OpenClaw Plugin प्रक्रिया के भीतर लोड होते हैं और क्षमताओं को एक केंद्रीय रजिस्ट्री में पंजीकृत करते हैं। पैकेज किया गया JavaScript नेटिव `require` के माध्यम से लोड होता है; तृतीय-पक्ष स्थानीय स्रोत TypeScript के लिए Jiti आपातकालीन फ़ॉलबैक है। संगत बंडल को रनटाइम कोड इंपोर्ट किए बिना रजिस्ट्री रिकॉर्ड में सामान्यीकृत किया जाता है।
  </Step>
  <Step title="सतह का उपभोग">
    OpenClaw का शेष भाग टूल, चैनल, प्रोवाइडर सेटअप, हुक, HTTP रूट, CLI कमांड और सेवाएँ उपलब्ध कराने के लिए रजिस्ट्री को पढ़ता है।
  </Step>
</Steps>

विशेष रूप से Plugin CLI के लिए, रूट कमांड डिस्कवरी दो चरणों में विभाजित है:

- पार्स-समय मेटाडेटा `registerCli(..., { descriptors: [...] })` से आता है
- वास्तविक Plugin CLI मॉड्यूल लेज़ी रह सकता है और पहली बार लागू किए जाने पर पंजीकरण कर सकता है

इससे Plugin के स्वामित्व वाला CLI कोड Plugin के भीतर रहता है, जबकि OpenClaw पार्सिंग से पहले रूट कमांड नाम आरक्षित कर सकता है।

महत्वपूर्ण डिज़ाइन सीमा:

- मैनिफ़ेस्ट/कॉन्फ़िग सत्यापन Plugin कोड निष्पादित किए बिना **मैनिफ़ेस्ट/स्कीमा मेटाडेटा** से कार्य करना चाहिए
- नेटिव क्षमता डिस्कवरी गैर-सक्रिय रजिस्ट्री स्नैपशॉट बनाने के लिए विश्वसनीय Plugin एंट्री कोड लोड कर सकती है
- नेटिव रनटाइम व्यवहार `api.registrationMode === "full"` के साथ Plugin मॉड्यूल के `register(api)` पाथ से आता है

यह विभाजन OpenClaw को पूर्ण रनटाइम सक्रिय होने से पहले कॉन्फ़िग सत्यापित करने, अनुपलब्ध/अक्षम Plugin की व्याख्या करने और UI/स्कीमा संकेत बनाने देता है।

### Plugin मेटाडेटा स्नैपशॉट और लुकअप तालिका

Gateway स्टार्टअप वर्तमान कॉन्फ़िग स्नैपशॉट के लिए एक `PluginMetadataSnapshot` बनाता है। स्नैपशॉट केवल मेटाडेटा है: यह इंस्टॉल किए गए Plugin इंडेक्स, मैनिफ़ेस्ट रजिस्ट्री, मैनिफ़ेस्ट डायग्नोस्टिक्स, स्वामी मैप, Plugin आईडी नॉर्मलाइज़र और मैनिफ़ेस्ट रिकॉर्ड संग्रहीत करता है। इसमें लोड किए गए Plugin मॉड्यूल, प्रोवाइडर SDK, पैकेज सामग्री या रनटाइम एक्सपोर्ट नहीं होते।

Plugin-जागरूक कॉन्फ़िग सत्यापन, स्टार्टअप स्वतः-सक्षमीकरण और Gateway Plugin बूटस्ट्रैप स्वतंत्र रूप से मैनिफ़ेस्ट/इंडेक्स मेटाडेटा दोबारा बनाने के बजाय उस स्नैपशॉट का उपयोग करते हैं। `PluginLookUpTable` उसी स्नैपशॉट से प्राप्त होता है और वर्तमान रनटाइम कॉन्फ़िग के लिए स्टार्टअप Plugin योजना जोड़ता है।

स्टार्टअप के बाद, Gateway वर्तमान मेटाडेटा स्नैपशॉट को प्रतिस्थापित किए जा सकने वाले रनटाइम उत्पाद के रूप में रखता है। प्रत्येक प्रोवाइडर-कैटलॉग पास के लिए इंस्टॉल किया गया इंडेक्स और मैनिफ़ेस्ट रजिस्ट्री दोबारा बनाने के बजाय, बार-बार होने वाली रनटाइम प्रोवाइडर डिस्कवरी उस स्नैपशॉट का उपयोग कर सकती है। Gateway बंद होने, कॉन्फ़िग/Plugin इन्वेंट्री में बदलाव और इंस्टॉल किए गए इंडेक्स में लेखन होने पर स्नैपशॉट साफ़ या प्रतिस्थापित किया जाता है; जब कोई संगत वर्तमान स्नैपशॉट मौजूद न हो, तो कॉलर कोल्ड मैनिफ़ेस्ट/इंडेक्स पाथ पर वापस जाते हैं। संगतता जाँच में `plugins.load.paths` और डिफ़ॉल्ट एजेंट वर्कस्पेस जैसे Plugin डिस्कवरी रूट शामिल होने चाहिए, क्योंकि वर्कस्पेस Plugin मेटाडेटा के दायरे का हिस्सा हैं।

स्नैपशॉट और लुकअप तालिका बार-बार लिए जाने वाले स्टार्टअप निर्णयों को फ़ास्ट पाथ पर रखते हैं:

- चैनल स्वामित्व
- स्थगित चैनल स्टार्टअप
- स्टार्टअप Plugin आईडी
- प्रोवाइडर और CLI बैकएंड स्वामित्व
- सेटअप प्रोवाइडर, कमांड उपनाम, मॉडल कैटलॉग प्रोवाइडर और मैनिफ़ेस्ट अनुबंध स्वामित्व
- Plugin कॉन्फ़िग स्कीमा और चैनल कॉन्फ़िग स्कीमा सत्यापन
- स्टार्टअप स्वतः-सक्षमीकरण निर्णय

सुरक्षा सीमा स्नैपशॉट का प्रतिस्थापन है, उत्परिवर्तन नहीं। कॉन्फ़िग, Plugin इन्वेंट्री, इंस्टॉल रिकॉर्ड या स्थायी इंडेक्स नीति बदलने पर स्नैपशॉट दोबारा बनाएँ। इसे व्यापक परिवर्तनशील वैश्विक रजिस्ट्री न मानें और असीमित ऐतिहासिक स्नैपशॉट न रखें। रनटाइम Plugin लोडिंग मेटाडेटा स्नैपशॉट से अलग रहती है, ताकि पुराने रनटाइम स्टेट को मेटाडेटा कैश के पीछे छिपाया न जा सके।

कैश नियम [Plugin आर्किटेक्चर के आंतरिक विवरण](/hi/plugins/architecture-internals#plugin-cache-boundary) में दस्तावेज़ीकृत है: मैनिफ़ेस्ट और डिस्कवरी मेटाडेटा तब तक नवीन रहते हैं, जब तक कोई कॉलर वर्तमान प्रवाह के लिए स्पष्ट स्नैपशॉट, लुकअप तालिका या मैनिफ़ेस्ट रजिस्ट्री नहीं रखता। छिपे हुए मेटाडेटा कैश और वॉल-क्लॉक TTL, Plugin लोडिंग का हिस्सा नहीं हैं। केवल रनटाइम लोडर, मॉड्यूल और डिपेंडेंसी-आर्टिफ़ैक्ट कैश ही कोड या इंस्टॉल किए गए आर्टिफ़ैक्ट वास्तव में लोड होने के बाद बने रह सकते हैं।

कुछ कोल्ड-पाथ कॉलर अब भी Gateway `PluginLookUpTable` प्राप्त करने के बजाय स्थायी इंस्टॉल किए गए Plugin इंडेक्स से सीधे मैनिफ़ेस्ट रजिस्ट्रियाँ दोबारा बनाते हैं। वह पाथ अब माँग पर रजिस्ट्री दोबारा बनाता है; यदि किसी कॉलर के पास पहले से वर्तमान लुकअप तालिका या स्पष्ट मैनिफ़ेस्ट रजिस्ट्री है, तो रनटाइम प्रवाहों में उसे पास करना बेहतर है।

### सक्रियण योजना

सक्रियण योजना कंट्रोल प्लेन का हिस्सा है। व्यापक रनटाइम रजिस्ट्रियाँ लोड करने से पहले कॉलर पूछ सकते हैं कि किसी ठोस कमांड, प्रोवाइडर, चैनल, रूट, एजेंट हार्नेस या क्षमता के लिए कौन-से Plugin प्रासंगिक हैं।

प्लानर वर्तमान मैनिफ़ेस्ट व्यवहार को संगत रखता है:

- `activation.*` फ़ील्ड स्पष्ट प्लानर संकेत हैं
- `providers`, `channels`, `commandAliases`, `setup.providers`, `contracts.tools`, और हुक मैनिफ़ेस्ट स्वामित्व फ़ॉलबैक बने रहते हैं
- केवल-आईडी प्लानर API मौजूदा कॉलर के लिए उपलब्ध रहता है
- प्लान API कारण लेबल की रिपोर्ट करता है, ताकि निदान स्पष्ट संकेतों और स्वामित्व फ़ॉलबैक में अंतर कर सके

<Warning>
`activation` को लाइफ़साइकल हुक या `register(...)` का प्रतिस्थापन न मानें। यह लोडिंग को सीमित करने के लिए उपयोग किया जाने वाला मेटाडेटा है। जब स्वामित्व फ़ील्ड पहले से संबंध का वर्णन करते हों, तो उन्हें प्राथमिकता दें; अतिरिक्त प्लानर संकेतों के लिए ही `activation` का उपयोग करें।
</Warning>

### चैनल Plugin और साझा संदेश टूल

सामान्य चैट क्रियाओं के लिए चैनल Plugin को अलग से भेजने/संपादित करने/प्रतिक्रिया देने वाला टूल पंजीकृत करने की आवश्यकता नहीं है। OpenClaw कोर में एक साझा `message` टूल रखता है, और चैनल Plugin उसके पीछे चैनल-विशिष्ट खोज और निष्पादन के स्वामी होते हैं।

वर्तमान सीमा यह है:

- कोर साझा `message` टूल होस्ट, प्रॉम्प्ट वायरिंग, सत्र/थ्रेड अभिलेख-प्रबंधन और निष्पादन डिस्पैच का स्वामी है
- चैनल Plugin सीमित क्रिया खोज, क्षमता खोज और किसी भी चैनल-विशिष्ट स्कीमा खंड के स्वामी हैं
- चैनल Plugin प्रदाता-विशिष्ट सत्र वार्तालाप व्याकरण के स्वामी हैं, जैसे वार्तालाप आईडी थ्रेड आईडी को कैसे एन्कोड करते हैं या मूल वार्तालापों से कैसे प्राप्त होते हैं
- चैनल Plugin अपने क्रिया अडैप्टर के माध्यम से अंतिम क्रिया निष्पादित करते हैं

चैनल Plugin के लिए SDK सतह `ChannelMessageActionAdapter.describeMessageTool(...)` है। वह एकीकृत खोज कॉल किसी Plugin को उसकी दृश्यमान क्रियाएँ, क्षमताएँ और स्कीमा योगदान एक साथ लौटाने देता है, ताकि ये भाग एक-दूसरे से अलग न हो जाएँ।

संदेश क्रिया नाम जानबूझकर बंद, कोर-स्वामित्व वाली शब्दावली का उपयोग करते हैं, ताकि प्रत्येक ट्रांसपोर्ट हर क्रिया को रेंडर कर सके। Plugin कोर PR के माध्यम से क्रिया नाम जोड़ते हैं; रनटाइम पंजीकरण जानबूझकर समर्थित नहीं है।

जब कोई चैनल-विशिष्ट संदेश-टूल पैरामीटर स्थानीय पथ या दूरस्थ मीडिया URL जैसे मीडिया स्रोत को वहन करता है, तो Plugin को `describeMessageTool(...)` से `mediaSourceParams` भी लौटाना चाहिए। कोर उस स्पष्ट सूची का उपयोग सैंडबॉक्स पथ सामान्यीकरण और आउटबाउंड मीडिया-पहुँच संकेत लागू करने के लिए करता है, बिना Plugin-स्वामित्व वाले पैरामीटर नामों को हार्डकोड किए। वहाँ एक चैनल-व्यापी सपाट सूची के बजाय क्रिया-सीमित मैप को प्राथमिकता दें, ताकि केवल-प्रोफ़ाइल मीडिया पैरामीटर `send` जैसी असंबंधित क्रियाओं पर सामान्यीकृत न हो।

कोर उस खोज चरण में रनटाइम स्कोप भेजता है। महत्वपूर्ण फ़ील्ड में शामिल हैं:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- विश्वसनीय इनबाउंड `requesterSenderId`

यह संदर्भ-संवेदी Plugin के लिए महत्वपूर्ण है। कोई चैनल सक्रिय खाते, वर्तमान रूम/थ्रेड/संदेश या विश्वसनीय अनुरोधकर्ता पहचान के आधार पर संदेश क्रियाओं को छिपा या दिखा सकता है, और इसके लिए कोर `message` टूल में चैनल-विशिष्ट शाखाएँ हार्डकोड करने की आवश्यकता नहीं होती।

इसी कारण एम्बेडेड-रनर रूटिंग परिवर्तन अब भी Plugin का कार्य हैं: रनर वर्तमान चैट/सत्र पहचान को Plugin खोज सीमा तक अग्रेषित करने के लिए उत्तरदायी है, ताकि साझा `message` टूल वर्तमान टर्न के लिए सही चैनल-स्वामित्व वाली सतह दिखाए।

चैनल-स्वामित्व वाले निष्पादन सहायकों के लिए, बंडल किए गए Plugin को निष्पादन रनटाइम अपने स्वयं के Plugin मॉड्यूल में रखना चाहिए। कोर अब `src/agents/tools` के अंतर्गत Discord, Slack, Telegram या WhatsApp संदेश-क्रिया रनटाइम का स्वामी नहीं है। हम अलग `plugin-sdk/*-action-runtime` उपपथ प्रकाशित नहीं करते, और बंडल किए गए Plugin को अपने Plugin-स्वामित्व वाले मॉड्यूल से अपना स्थानीय रनटाइम कोड सीधे आयात करना चाहिए।

यही सीमा सामान्यतः प्रदाता-नामित SDK सीमों पर भी लागू होती है: कोर को Discord, Signal, Slack, WhatsApp या समान Plugin के लिए चैनल-विशिष्ट सुविधाजनक बैरल आयात नहीं करने चाहिए। यदि कोर को किसी व्यवहार की आवश्यकता है, तो या तो बंडल किए गए Plugin के अपने `api.ts` / `runtime-api.ts` बैरल का उपयोग करें या आवश्यकता को साझा SDK में एक सीमित सामान्य क्षमता के रूप में उन्नत करें।

बंडल किए गए Plugin भी इसी नियम का पालन करते हैं। किसी बंडल किए गए Plugin के `runtime-api.ts` को अपना ब्रांडेड `openclaw/plugin-sdk/<plugin-id>` फ़साड पुनः निर्यात नहीं करना चाहिए। वे ब्रांडेड फ़साड बाहरी Plugin और पुराने उपभोक्ताओं के लिए संगतता शिम बने रहते हैं, लेकिन बंडल किए गए Plugin को स्थानीय निर्यात के साथ `openclaw/plugin-sdk/channel-policy`, `openclaw/plugin-sdk/runtime-store`, या `openclaw/plugin-sdk/webhook-ingress` जैसे सीमित सामान्य SDK उपपथों का उपयोग करना चाहिए। नए कोड में Plugin-आईडी-विशिष्ट SDK फ़साड नहीं जोड़े जाने चाहिए, जब तक किसी मौजूदा बाहरी पारिस्थितिकी तंत्र की संगतता सीमा के लिए उनकी आवश्यकता न हो।

विशेष रूप से पोल के लिए, दो निष्पादन पथ हैं:

- `outbound.sendPoll` उन चैनलों के लिए साझा आधाररेखा है जो सामान्य पोल मॉडल के अनुकूल हैं
- `actions.handleAction("poll")` चैनल-विशिष्ट पोल अर्थविज्ञान या अतिरिक्त पोल पैरामीटर के लिए प्राथमिक पथ है

कोर अब साझा पोल पार्सिंग को तब तक स्थगित करता है जब तक Plugin पोल डिस्पैच क्रिया को अस्वीकार न कर दे, ताकि Plugin-स्वामित्व वाले पोल हैंडलर पहले सामान्य पोल पार्सर द्वारा अवरुद्ध हुए बिना चैनल-विशिष्ट पोल फ़ील्ड स्वीकार कर सकें।

संपूर्ण स्टार्टअप अनुक्रम के लिए [Plugin आर्किटेक्चर की आंतरिक कार्यप्रणाली](/hi/plugins/architecture-internals) देखें।

## क्षमता स्वामित्व मॉडल

OpenClaw किसी नेटिव Plugin को असंबंधित एकीकरणों के संग्रह के रूप में नहीं, बल्कि किसी **कंपनी** या **फ़ीचर** की स्वामित्व सीमा के रूप में देखता है।

इसका अर्थ है:

- किसी कंपनी के Plugin को सामान्यतः उस कंपनी की सभी OpenClaw-संबंधी सतहों का स्वामी होना चाहिए
- किसी फ़ीचर Plugin को सामान्यतः उसके द्वारा प्रस्तुत संपूर्ण फ़ीचर सतह का स्वामी होना चाहिए
- चैनलों को प्रदाता व्यवहार को तदर्थ रूप से पुनः लागू करने के बजाय साझा कोर क्षमताओं का उपयोग करना चाहिए

<AccordionGroup>
  <Accordion title="विक्रेता बहु-क्षमता">
    `google` टेक्स्ट इन्फ़रेंस, CLI बैकएंड, एम्बेडिंग, स्पीच, रियलटाइम वॉइस, मीडिया समझ, छवि/संगीत/वीडियो जनरेशन और वेब खोज का स्वामी है। `openai` टेक्स्ट इन्फ़रेंस, एम्बेडिंग, स्पीच, रियलटाइम ट्रांसक्रिप्शन, रियलटाइम वॉइस, मीडिया समझ और छवि/वीडियो जनरेशन का स्वामी है। `minimax` टेक्स्ट इन्फ़रेंस के साथ मीडिया समझ, स्पीच, छवि/संगीत/वीडियो जनरेशन और वेब खोज का स्वामी है।
  </Accordion>
  <Accordion title="विक्रेता एकल-क्षमता">
    `arcee` और `chutes` केवल टेक्स्ट इन्फ़रेंस के स्वामी हैं; `microsoft` केवल स्पीच का स्वामी है। किसी विक्रेता का Plugin तब तक इतना सीमित रह सकता है, जब तक उसे उस विक्रेता की और अधिक सतह को शामिल करने की आवश्यकता न हो।
  </Accordion>
  <Accordion title="फ़ीचर Plugin">
    `voice-call` कॉल ट्रांसपोर्ट, टूल, CLI, रूट और Twilio मीडिया-स्ट्रीम ब्रिजिंग का स्वामी है, लेकिन विक्रेता Plugin को सीधे आयात करने के बजाय साझा स्पीच, रियलटाइम ट्रांसक्रिप्शन और रियलटाइम वॉइस क्षमताओं का उपयोग करता है।
  </Accordion>
</AccordionGroup>

अभीष्ट अंतिम स्थिति यह है:

- किसी विक्रेता की OpenClaw-संबंधी सतह एक Plugin में रहती है, भले ही वह टेक्स्ट मॉडल, स्पीच, छवियों और वीडियो तक विस्तृत हो
- अन्य विक्रेता अपने सतह क्षेत्र के लिए भी ऐसा ही कर सकते हैं
- चैनलों को इससे कोई सरोकार नहीं होता कि प्रदाता का स्वामी कौन-सा विक्रेता Plugin है; वे कोर द्वारा उजागर साझा क्षमता अनुबंध का उपयोग करते हैं

मुख्य अंतर यह है:

- **Plugin** = स्वामित्व सीमा
- **क्षमता** = कोर अनुबंध जिसे कई Plugin लागू कर सकते हैं या जिसका उपयोग कर सकते हैं

इसलिए यदि OpenClaw वीडियो जैसा कोई नया डोमेन जोड़ता है, तो पहला प्रश्न यह नहीं है कि "किस प्रदाता को वीडियो प्रबंधन हार्डकोड करना चाहिए?" पहला प्रश्न है कि "कोर वीडियो क्षमता अनुबंध क्या है?" वह अनुबंध उपलब्ध होने के बाद विक्रेता Plugin उसके साथ पंजीकृत हो सकते हैं और चैनल/फ़ीचर Plugin उसका उपयोग कर सकते हैं।

यदि क्षमता अभी उपलब्ध नहीं है, तो सामान्यतः सही कदम हैं:

    कोर में अनुपलब्ध क्षमता परिभाषित करें।
  </Step>
  <Step title="SDK के माध्यम से उजागर करें">
    इसे टाइप-सुरक्षित रूप से Plugin API/रनटाइम के माध्यम से उजागर करें।
  </Step>
  <Step title="उपभोक्ताओं को जोड़ें">
    चैनलों/फ़ीचर को उस क्षमता से जोड़ें।
  </Step>
  <Step title="विक्रेता कार्यान्वयन">
    विक्रेता Plugin को कार्यान्वयन पंजीकृत करने दें।
  </Step>
</Steps>

यह स्वामित्व को स्पष्ट रखता है और साथ ही ऐसे कोर व्यवहार से बचाता है जो किसी एक विक्रेता या एकबारगी Plugin-विशिष्ट कोड पथ पर निर्भर हो।

### क्षमता स्तरीकरण

कोड कहाँ होना चाहिए, यह तय करते समय इस मानसिक मॉडल का उपयोग करें:

<Tabs>

  <Tab title="विक्रेता Plugin स्तर">
    विक्रेता-विशिष्ट API, प्रमाणीकरण, मॉडल कैटलॉग, स्पीच सिंथेसिस, छवि जनरेशन, वीडियो बैकएंड और उपयोग एंडपॉइंट।
  </Tab>
  <Tab title="चैनल/फ़ीचर Plugin स्तर">

उदाहरण के लिए, TTS इस संरचना का अनुसरण करता है:

- कोर उत्तर-समय TTS नीति, फ़ॉलबैक क्रम, प्राथमिकताओं और चैनल डिलीवरी का स्वामी है
- `elevenlabs`, `google`, `microsoft`, और `openai` सिंथेसिस कार्यान्वयनों के स्वामी हैं
- `voice-call` टेलीफ़ोनी TTS रनटाइम सहायक का उपयोग करता है

भविष्य की क्षमताओं के लिए भी इसी पैटर्न को प्राथमिकता दी जानी चाहिए।

### बहु-क्षमता कंपनी Plugin का उदाहरण

किसी कंपनी के Plugin को बाहर से सुसंगत अनुभव होना चाहिए। यदि OpenClaw के पास मॉडल, स्पीच, रियलटाइम ट्रांसक्रिप्शन, रियलटाइम वॉइस, मीडिया समझ, छवि जनरेशन, वीडियो जनरेशन, वेब फ़ेच और वेब खोज के लिए साझा अनुबंध हैं, तो कोई विक्रेता अपनी सभी सतहों का स्वामित्व एक ही स्थान पर रख सकता है:

```ts
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { exampleAiMedia } from "./exampleai-media.js";

export default definePluginEntry({
  id: "exampleai",
  name: "ExampleAI",
  description: "ExampleAI मॉडल और मीडिया क्षमताएँ।",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // प्रमाणीकरण/मॉडल कैटलॉग/रनटाइम हुक
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // विक्रेता स्पीच कॉन्फ़िग — SpeechProviderPlugin इंटरफ़ेस को सीधे लागू करें
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      describeImage: (req) => exampleAiMedia.describeImage(req),
      transcribeAudio: (req) => exampleAiMedia.transcribeAudio(req),
      describeVideo: (req) => exampleAiMedia.describeVideo(req),
    });

    api.registerWebSearchProvider({
      id: "exampleai-search",
      createTool() {
        // विक्रेता-स्वामित्व वाला वेब खोज टूल लौटाएँ।
      },
    });
  },
});
```

सटीक सहायक नाम महत्वपूर्ण नहीं हैं। संरचना महत्वपूर्ण है:

- एक Plugin विक्रेता सतह का स्वामी है
- कोर अब भी क्षमता अनुबंधों का स्वामी है
- प्रदाता अनुरोध रूपांतरण और HTTP सहायक विक्रेता Plugin में रहते हैं
- चैनल और फ़ीचर Plugin विक्रेता कोड के बजाय `api.runtime.*` सहायकों का उपयोग करते हैं
- अनुबंध परीक्षण यह सुनिश्चित कर सकते हैं कि Plugin ने उन क्षमताओं को पंजीकृत किया है जिनके स्वामित्व का वह दावा करता है

### क्षमता उदाहरण: वीडियो समझ

OpenClaw पहले से छवि/ऑडियो/वीडियो समझ को एक साझा क्षमता मानता है। वही स्वामित्व मॉडल वहाँ भी लागू होता है:

<Steps>
  <Step title="कोर अनुबंध परिभाषित करता है">
    कोर मीडिया-समझ अनुबंध परिभाषित करता है।
  </Step>
  <Step title="विक्रेता Plugin पंजीकरण करते हैं">
    विक्रेता Plugin उपयुक्ततानुसार `describeImage`, `transcribeAudio`, और `describeVideo` पंजीकृत करते हैं।
  </Step>
  <Step title="उपभोक्ता साझा व्यवहार का उपयोग करते हैं">
    चैनल और फ़ीचर Plugin सीधे विक्रेता कोड से जुड़ने के बजाय साझा कोर व्यवहार का उपयोग करते हैं।
  </Step>
</Steps>

इससे किसी एक प्रदाता की वीडियो संबंधी धारणाएँ कोर में अंतर्निहित नहीं होतीं। Plugin विक्रेता सतह का स्वामी होता है; कोर क्षमता अनुबंध और फ़ॉलबैक व्यवहार का स्वामी होता है।

वीडियो जनरेशन पहले से ही उसी क्रम का उपयोग करता है: कोर टाइप किए गए क्षमता अनुबंध और रनटाइम सहायक का स्वामी होता है, और वेंडर plugins उसके विरुद्ध `api.registerVideoGenerationProvider(...)` कार्यान्वयन पंजीकृत करते हैं।

ठोस रोलआउट चेकलिस्ट चाहिए? [क्षमता कुकबुक](/hi/plugins/adding-capabilities) देखें।

## अनुबंध और प्रवर्तन

Plugin API सतह को जानबूझकर `OpenClawPluginApi` में टाइप और केंद्रीकृत किया गया है। वह अनुबंध समर्थित पंजीकरण बिंदुओं और उन रनटाइम सहायकों को परिभाषित करता है जिन पर कोई plugin निर्भर हो सकता है।

यह क्यों महत्वपूर्ण है:

- plugin लेखकों को एक स्थिर आंतरिक मानक मिलता है
- कोर डुप्लिकेट स्वामित्व को अस्वीकार कर सकता है, जैसे एक ही प्रदाता आईडी को पंजीकृत करने वाले दो plugins
- स्टार्टअप विकृत पंजीकरण के लिए कार्रवाई योग्य निदान दिखा सकता है
- अनुबंध परीक्षण बंडल किए गए plugin के स्वामित्व को लागू कर सकते हैं और बिना सूचना के विचलन को रोक सकते हैं

प्रवर्तन की दो परतें हैं:

<AccordionGroup>
  <Accordion title="रनटाइम पंजीकरण प्रवर्तन">
    plugins लोड होते समय plugin रजिस्ट्री पंजीकरणों को मान्य करती है। उदाहरण: डुप्लिकेट प्रदाता आईडी, डुप्लिकेट स्पीच प्रदाता आईडी और विकृत पंजीकरण अपरिभाषित व्यवहार के बजाय plugin निदान उत्पन्न करते हैं।
  </Accordion>
  <Accordion title="अनुबंध परीक्षण">
    परीक्षण चलने के दौरान बंडल किए गए plugins को अनुबंध रजिस्ट्रियों में दर्ज किया जाता है, ताकि OpenClaw स्वामित्व का स्पष्ट रूप से सत्यापन कर सके। वर्तमान में इसका उपयोग मॉडल प्रदाताओं, स्पीच प्रदाताओं, वेब खोज प्रदाताओं और बंडल किए गए पंजीकरण के स्वामित्व के लिए किया जाता है।
  </Accordion>
</AccordionGroup>

इसका व्यावहारिक प्रभाव यह है कि OpenClaw को पहले से पता होता है कि कौन-सा plugin किस सतह का स्वामी है। इससे कोर और चैनल निर्बाध रूप से संयोजित हो सकते हैं, क्योंकि स्वामित्व निहित होने के बजाय घोषित, टाइप किया हुआ और परीक्षण योग्य होता है।

## निष्पादन मॉडल

मूल OpenClaw plugins Gateway के साथ **उसी प्रक्रिया में** चलते हैं। वे सैंडबॉक्स में नहीं होते। लोड किए गए मूल plugin की प्रक्रिया-स्तरीय विश्वास सीमा कोर कोड के समान होती है।

<Warning>
मूल plugin के निहितार्थ: कोई plugin टूल, नेटवर्क हैंडलर, हुक और सेवाएँ पंजीकृत कर सकता है; plugin की कोई त्रुटि gateway को क्रैश या अस्थिर कर सकती है; और कोई दुर्भावनापूर्ण मूल plugin OpenClaw प्रक्रिया के भीतर मनमाना कोड निष्पादन करने के समान है।
</Warning>

संगत बंडल डिफ़ॉल्ट रूप से अधिक सुरक्षित होते हैं, क्योंकि OpenClaw वर्तमान में उन्हें मेटाडेटा/कंटेंट पैक मानता है। मौजूदा रिलीज़ में इसका अर्थ मुख्यतः बंडल किए गए skills हैं।

बंडल नहीं किए गए plugins के लिए अनुमति-सूचियों और स्पष्ट इंस्टॉल/लोड पथों का उपयोग करें। वर्कस्पेस plugins को विकास-समय का कोड मानें, न कि उत्पादन डिफ़ॉल्ट।

बंडल किए गए वर्कस्पेस पैकेज नामों के लिए plugin आईडी को npm नाम में स्थिर रखें: डिफ़ॉल्ट रूप से `@openclaw/<id>`, या जब पैकेज जानबूझकर अधिक सीमित plugin भूमिका उजागर करता हो, तब `-provider`, `-plugin`, `-speech`, `-sandbox`, या `-media-understanding` जैसा अनुमोदित टाइप किया हुआ प्रत्यय।

<Note>
**विश्वास संबंधी टिप्पणी:** `plugins.allow` स्रोत के उद्गम पर नहीं, बल्कि **plugin आईडी** पर विश्वास करता है। बंडल किए गए plugin के समान आईडी वाला वर्कस्पेस plugin, सक्षम/अनुमति-सूचीबद्ध होने पर जानबूझकर बंडल की गई प्रति को ओवरराइड करता है। स्थानीय विकास, पैच परीक्षण और हॉटफ़िक्स के लिए यह सामान्य और उपयोगी है। बंडल किए गए plugin का विश्वास स्रोत स्नैपशॉट—लोड के समय डिस्क पर मौजूद मैनिफ़ेस्ट और कोड—से निर्धारित होता है, इंस्टॉल मेटाडेटा से नहीं। कोई दूषित या प्रतिस्थापित इंस्टॉल रिकॉर्ड बंडल किए गए plugin की विश्वास सतह को वास्तविक स्रोत के दावे से आगे चुपचाप विस्तृत नहीं कर सकता।
</Note>

## निर्यात सीमा

OpenClaw क्षमताओं को निर्यात करता है, कार्यान्वयन की सुविधाओं को नहीं।

क्षमता पंजीकरण को सार्वजनिक रखें। गैर-अनुबंध सहायक निर्यातों को हटाएँ:

- बंडल किए गए plugin-विशिष्ट सहायक उपपथ
- सार्वजनिक API के रूप में अभिप्रेत नहीं रनटाइम प्लंबिंग उपपथ
- वेंडर-विशिष्ट सुविधा सहायक
- कार्यान्वयन विवरण वाले सेटअप/ऑनबोर्डिंग सहायक

आरक्षित बंडल किए गए plugin सहायक उपपथों को जनरेट किए गए SDK निर्यात मैप से हटा दिया गया है। स्वामी-विशिष्ट सहायकों को स्वामी plugin पैकेज के भीतर रखें; केवल पुन: उपयोग योग्य होस्ट व्यवहार को `plugin-sdk/gateway-runtime`, `plugin-sdk/security-runtime` और इंजेक्ट की गई plugin API क्षमताओं जैसे सामान्य SDK अनुबंधों में पदोन्नत करें।

## आंतरिक विवरण और संदर्भ

लोड पाइपलाइन, रजिस्ट्री मॉडल, प्रदाता रनटाइम हुक, Gateway HTTP रूट, संदेश टूल स्कीमा, चैनल लक्ष्य समाधान, प्रदाता कैटलॉग, संदर्भ इंजन plugins और नई क्षमता जोड़ने की मार्गदर्शिका के लिए [Plugin आर्किटेक्चर के आंतरिक विवरण](/hi/plugins/architecture-internals) देखें।
