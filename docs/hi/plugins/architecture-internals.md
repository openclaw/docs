सार्वजनिक क्षमता मॉडल, Plugin संरचनाओं और स्वामित्व/निष्पादन
अनुबंधों के लिए, [Plugin आर्किटेक्चर](/hi/plugins/architecture) देखें। यह पृष्ठ
आंतरिक कार्यप्रणाली को कवर करता है: लोड पाइपलाइन, रजिस्ट्री, रनटाइम हुक, Gateway HTTP
रूट, इंपोर्ट पथ और स्कीमा तालिकाएँ।

## लोड पाइपलाइन

स्टार्टअप पर, OpenClaw मोटे तौर पर यह करता है:

1. संभावित Plugin रूट खोजता है
2. नेटिव या संगत बंडल मैनिफ़ेस्ट और पैकेज मेटाडेटा पढ़ता है
3. असुरक्षित संभावित Plugin अस्वीकार करता है
4. Plugin कॉन्फ़िगरेशन को सामान्यीकृत करता है (`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. प्रत्येक संभावित Plugin को सक्षम करने का निर्णय लेता है
6. सक्षम नेटिव मॉड्यूल लोड करता है: निर्मित बंडल मॉड्यूल नेटिव लोडर का उपयोग करते हैं;
   तृतीय-पक्ष का स्थानीय स्रोत TypeScript आपातकालीन Jiti फ़ॉलबैक का उपयोग करता है
7. नेटिव `register(api)` हुक को कॉल करता है और पंजीकरणों को Plugin रजिस्ट्री में एकत्र करता है
8. रजिस्ट्री को कमांड/रनटाइम सतहों के लिए उपलब्ध कराता है

सुरक्षा जाँचें रनटाइम निष्पादन से **पहले** चलती हैं। खोज किसी संभावित Plugin को
तब अवरुद्ध करती है जब:

- उसकी रिज़ॉल्व की गई एंट्री Plugin रूट से बाहर निकलती है
- उसका पथ (या उसकी रूट डायरेक्टरी) सभी उपयोगकर्ताओं के लिए लिखने योग्य है
- गैर-बंडल Plugin के लिए, पथ का स्वामित्व वर्तमान uid (या root) से मेल नहीं खाता

सभी उपयोगकर्ताओं के लिए लिखने योग्य बंडल डायरेक्टरी पर गेट की
दोबारा जाँच से पहले उसी स्थान पर `chmod` सुधार का प्रयास किया जाता है
(npm/वैश्विक इंस्टॉल पैकेज डायरेक्टरी को `0777` पर भेज सकते हैं);
बंडल मूल के लिए स्वामित्व जाँच पूरी तरह छोड़ दी जाती है।

अवरुद्ध संभावित Plugin के उत्सर्जित निदान में ज्ञात होने पर उसका Plugin id फिर भी शामिल रहता है
(इसमें अन्यथा अस्वीकृत डायरेक्टरी के भीतर मैनिफ़ेस्ट से रिज़ॉल्व किए गए id भी शामिल हैं),
इसलिए उस id का संदर्भ देने वाले कॉन्फ़िगरेशन को असंबंधित "अज्ञात Plugin"
त्रुटि के बजाय पथ-सुरक्षा चेतावनी से जुड़ा अवरुद्ध
Plugin दिखाई देता है।

### मैनिफ़ेस्ट-प्रथम व्यवहार

मैनिफ़ेस्ट नियंत्रण-प्लेन के सत्य का स्रोत है। OpenClaw इसका उपयोग इनके लिए करता है:

- Plugin की पहचान करना
- घोषित चैनल/Skills/कॉन्फ़िगरेशन स्कीमा या बंडल क्षमताएँ खोजना
- `plugins.entries.<id>.config` को सत्यापित करना
- Control UI लेबल/प्लेसहोल्डर को समृद्ध करना
- इंस्टॉल/कैटलॉग मेटाडेटा दिखाना
- Plugin रनटाइम लोड किए बिना हल्के सक्रियण और सेटअप वर्णनकर्ता बनाए रखना

नेटिव Plugin के लिए, रनटाइम मॉड्यूल डेटा-प्लेन का भाग है। यह
हुक, टूल, कमांड या प्रदाता प्रवाह जैसे वास्तविक व्यवहार पंजीकृत करता है।

वैकल्पिक मैनिफ़ेस्ट `activation` और `setup` ब्लॉक नियंत्रण प्लेन पर रहते हैं।
वे सक्रियण योजना और सेटअप खोज के लिए केवल-मेटाडेटा वर्णनकर्ता हैं;
वे रनटाइम पंजीकरण, `register(...)` या `setupEntry` को प्रतिस्थापित नहीं करते।
लाइव सक्रियण उपभोक्ता व्यापक रजिस्ट्री मूर्त रूप देने से पहले Plugin लोडिंग को
सीमित करने के लिए मैनिफ़ेस्ट के कमांड, चैनल और प्रदाता संकेतों का उपयोग करते हैं:

- CLI लोडिंग उन Plugin तक सीमित होती है जो अनुरोधित प्राथमिक कमांड के स्वामी हैं
- चैनल सेटअप/Plugin रिज़ॉल्यूशन उन Plugin तक सीमित होता है जो अनुरोधित
  चैनल id के स्वामी हैं
- स्पष्ट प्रदाता सेटअप/रनटाइम रिज़ॉल्यूशन उन Plugin तक सीमित होता है जो
  अनुरोधित प्रदाता id के स्वामी हैं
- Gateway स्टार्टअप योजना स्पष्ट स्टार्टअप
  इंपोर्ट के लिए `activation.onStartup` का उपयोग करती है; स्टार्टअप मेटाडेटा के बिना Plugin केवल अधिक सीमित
  सक्रियण ट्रिगर के माध्यम से लोड होते हैं

सक्रियण योजनाकार मौजूदा कॉलर के लिए केवल-id API और निदान के लिए
योजना API, दोनों उपलब्ध कराता है। योजना प्रविष्टियाँ बताती हैं कि कोई Plugin क्यों चुना गया,
और स्पष्ट `activation.*` संकेतों को मैनिफ़ेस्ट-स्वामित्व फ़ॉलबैक से अलग करती हैं:

| कारण (`activation.*` संकेतों से)   | कारण (मैनिफ़ेस्ट स्वामित्व से)                                                             |
| ------------------------------------ | -------------------------------------------------------------------------------------------- |
| `activation-agent-harness-hint`      | —                                                                                            |
| `activation-capability-hint`         | —                                                                                            |
| `activation-channel-hint`            | `manifest-channel-owner` (`channels`)                                                        |
| `activation-command-hint`            | `manifest-command-alias` (`commandAliases`)                                                  |
| `activation-provider-hint`           | `manifest-provider-owner` (`providers`), `manifest-setup-provider-owner` (`setup.providers`) |
| `activation-route-hint`              | —                                                                                            |
| — (हुक ट्रिगर का कोई संकेत प्रकार नहीं है) | `manifest-hook-owner` (`hooks`), `manifest-tool-contract` (`contracts.tools`)                |

कारणों का यह विभाजन संगतता सीमा है: मौजूदा Plugin मेटाडेटा
काम करता रहता है, जबकि नया कोड रनटाइम लोडिंग के अर्थ को बदले बिना
व्यापक संकेतों या फ़ॉलबैक व्यवहार का पता लगा सकता है।

अनुरोध-समय के रनटाइम प्रीलोड, जो व्यापक `all` दायरे की माँग करते हैं, फिर भी
कॉन्फ़िगरेशन, स्टार्टअप योजना, कॉन्फ़िगर किए गए चैनलों, स्लॉट और स्वतः-सक्षम नियमों से
स्पष्ट प्रभावी Plugin id सेट प्राप्त करते हैं
(`src/plugins/effective-plugin-ids.ts` में `resolveEffectivePluginIds`)। यदि वह
प्राप्त सेट खाली है, तो OpenClaw प्रत्येक खोज योग्य Plugin तक विस्तार करने के बजाय
दायरे को खाली रखता है।

सेटअप खोज, `setup-api` पर फ़ॉलबैक करने से पहले संभावित Plugin को
सीमित करने के लिए `setup.providers` और
`setup.cliBackends` जैसे वर्णनकर्ता-स्वामित्व वाले id को प्राथमिकता देती है,
उन Plugin के लिए जिन्हें अभी भी सेटअप-समय रनटाइम हुक चाहिए। प्रदाता
सेटअप सूचियाँ प्रदाता रनटाइम लोड किए बिना मैनिफ़ेस्ट `providerAuthChoices`, वर्णनकर्ता से प्राप्त सेटअप
विकल्पों और इंस्टॉल-कैटलॉग मेटाडेटा का उपयोग करती हैं। स्पष्ट
`setup.requiresRuntime: false` केवल-वर्णनकर्ता कटऑफ़ है; छोड़ा गया
`requiresRuntime` संगतता के लिए पुराने setup-api फ़ॉलबैक को बनाए रखता है। यदि
एक से अधिक खोजे गए Plugin समान सामान्यीकृत सेटअप प्रदाता या
CLI बैकएंड id पर दावा करते हैं, तो सेटअप लुकअप खोज क्रम पर निर्भर होने के बजाय
अस्पष्ट स्वामी को अस्वीकार कर देता है। जब सेटअप रनटाइम निष्पादित होता है, तो रजिस्ट्री निदान
`setup.providers` / `setup.cliBackends` और setup-api द्वारा वास्तव में पंजीकृत प्रदाताओं या CLI
बैकएंड के बीच अंतर की रिपोर्ट करता है, पर पुराने Plugin को अवरुद्ध नहीं करता।

### Plugin कैश सीमा

OpenClaw Plugin खोज परिणामों या प्रत्यक्ष मैनिफ़ेस्ट रजिस्ट्री
डेटा को वास्तविक-समय विंडो के पीछे कैश नहीं करता। इंस्टॉल, मैनिफ़ेस्ट संपादन और लोड-पथ परिवर्तन
अगली स्पष्ट मेटाडेटा रीड या स्नैपशॉट पुनर्निर्माण में दिखाई देने चाहिए।
मैनिफ़ेस्ट फ़ाइल पार्सर खोले गए मैनिफ़ेस्ट पथ और डिवाइस/inode, आकार तथा mtime/ctime के आधार पर
एक सीमित फ़ाइल-हस्ताक्षर कैश रखता है; वह कैश केवल
अपरिवर्तित बाइट को दोबारा पार्स करने से बचाता है और उसे खोज, रजिस्ट्री,
स्वामी या नीति उत्तरों को कैश नहीं करना चाहिए।

सुरक्षित मेटाडेटा तेज़ पथ स्पष्ट ऑब्जेक्ट स्वामित्व है, कोई छिपा हुआ कैश नहीं।
Gateway स्टार्टअप के हॉट पथ को वर्तमान `PluginMetadataSnapshot`,
प्राप्त `PluginLookUpTable` या स्पष्ट मैनिफ़ेस्ट रजिस्ट्री को कॉल
श्रृंखला के माध्यम से भेजना चाहिए। कॉन्फ़िगरेशन सत्यापन, स्टार्टअप स्वतः-सक्षमता, Plugin बूटस्ट्रैप और प्रदाता
चयन उन ऑब्जेक्ट का पुनः उपयोग कर सकते हैं, जब तक वे वर्तमान कॉन्फ़िगरेशन और
Plugin इन्वेंटरी का प्रतिनिधित्व करते हैं। सेटअप लुकअप अभी भी माँग पर मैनिफ़ेस्ट मेटाडेटा का पुनर्निर्माण करता है,
जब तक विशिष्ट सेटअप पथ को स्पष्ट मैनिफ़ेस्ट रजिस्ट्री न मिले; छिपे हुए लुकअप कैश जोड़ने के बजाय
इसे कोल्ड-पथ फ़ॉलबैक ही रखें। इनपुट बदलने पर, स्नैपशॉट को परिवर्तित करने या
ऐतिहासिक प्रतियाँ रखने के बजाय उसका पुनर्निर्माण करके उसे बदलें। सक्रिय Plugin रजिस्ट्री के दृश्य और बंडल
चैनल बूटस्ट्रैप सहायक वर्तमान
रजिस्ट्री/रूट से दोबारा परिकलित किए जाने चाहिए। काम की पुनरावृत्ति हटाने या पुनःप्रवेश से बचाने के लिए एक कॉल के भीतर
अल्पकालिक मैप उचित हैं; उन्हें प्रक्रिया मेटाडेटा कैश नहीं बनना चाहिए।

Plugin लोडिंग के लिए, स्थायी कैश परत रनटाइम लोडिंग है। कोड या इंस्टॉल किए गए आर्टिफ़ैक्ट
वास्तव में लोड होने पर यह लोडर स्थिति का पुनः उपयोग कर सकती है, जैसे:

- `PluginLoaderCacheState` और संगत सक्रिय रनटाइम रजिस्ट्री
- एक ही रनटाइम सतह को बार-बार इंपोर्ट करने से बचने के लिए उपयोग किए जाने वाले jiti/मॉड्यूल कैश और सार्वजनिक-सतह लोडर कैश
- इंस्टॉल किए गए Plugin आर्टिफ़ैक्ट के लिए फ़ाइलसिस्टम कैश
- पथ सामान्यीकरण या डुप्लिकेट रिज़ॉल्यूशन के लिए अल्पकालिक प्रति-कॉल मैप

वे कैश डेटा-प्लेन के कार्यान्वयन विवरण हैं। उन्हें
"इस प्रदाता का स्वामी कौन-सा Plugin है?" जैसे नियंत्रण-प्लेन प्रश्नों का उत्तर नहीं देना चाहिए,
जब तक कॉलर ने जानबूझकर रनटाइम लोडिंग का अनुरोध न किया हो।

इनके लिए स्थायी या वास्तविक-समय कैश न जोड़ें:

- खोज परिणाम
- प्रत्यक्ष मैनिफ़ेस्ट रजिस्ट्री
- इंस्टॉल किए गए Plugin इंडेक्स से पुनर्निर्मित मैनिफ़ेस्ट रजिस्ट्री
- प्रदाता स्वामी लुकअप, मॉडल दमन, प्रदाता नीति या सार्वजनिक-आर्टिफ़ैक्ट
  मेटाडेटा
- मैनिफ़ेस्ट से प्राप्त कोई अन्य उत्तर, जहाँ बदला हुआ मैनिफ़ेस्ट, इंस्टॉल किया गया इंडेक्स
  या लोड पथ अगली मेटाडेटा रीड में दिखाई देना चाहिए

स्थायी इंस्टॉल किए गए Plugin इंडेक्स से मैनिफ़ेस्ट मेटाडेटा का पुनर्निर्माण करने वाले
कॉलर माँग पर उस रजिस्ट्री का पुनर्निर्माण करते हैं। इंस्टॉल किया गया इंडेक्स टिकाऊ
स्रोत-प्लेन स्थिति है; यह कोई छिपा हुआ प्रक्रिया-अंतर्गत मेटाडेटा कैश नहीं है।

## रजिस्ट्री मॉडल

लोड किए गए Plugin सीधे अनियमित कोर ग्लोबल को परिवर्तित नहीं करते। वे एक
केंद्रीय Plugin रजिस्ट्री (`src/plugins/registry-types.ts` में `PluginRegistry`) में पंजीकरण करते हैं,
जो Plugin रिकॉर्ड (पहचान, स्रोत, मूल, स्थिति, निदान)
के साथ प्रत्येक क्षमता के लिए ऐरे ट्रैक करती है: टूल, पुराने हुक और टाइप किए गए हुक,
चैनल, प्रदाता, Gateway RPC हैंडलर, HTTP रूट, CLI रजिस्ट्रार,
पृष्ठभूमि सेवाएँ, Plugin-स्वामित्व वाले कमांड और दर्जनों अन्य टाइप किए गए प्रदाता
परिवार (वाक्, एम्बेडिंग, छवि/वीडियो/संगीत जनरेशन, वेब
फ़ेच/खोज, एजेंट हार्नेस, सत्र क्रियाएँ आदि)।

इसके बाद कोर सुविधाएँ Plugin मॉड्यूल से सीधे बात करने के बजाय उस रजिस्ट्री से पढ़ती हैं।
इससे लोडिंग एक-दिशीय रहती है:

- Plugin मॉड्यूल -> रजिस्ट्री पंजीकरण
- कोर रनटाइम -> रजिस्ट्री उपभोग

यह पृथक्करण रखरखाव के लिए महत्वपूर्ण है। इसका अर्थ है कि अधिकांश कोर सतहों को केवल
एक एकीकरण बिंदु चाहिए: "रजिस्ट्री पढ़ें", न कि "प्रत्येक
Plugin मॉड्यूल को विशेष रूप से संभालें"।

## वार्तालाप बाइंडिंग कॉलबैक

वार्तालाप को बाइंड करने वाले Plugin किसी अनुमोदन का समाधान होने पर प्रतिक्रिया दे सकते हैं।

किसी बाइंड अनुरोध को स्वीकृत या अस्वीकृत किए जाने के बाद कॉलबैक प्राप्त करने के लिए
`api.onConversationBindingResolved(...)` का उपयोग करें:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // इस Plugin + वार्तालाप के लिए अब एक बाइंडिंग मौजूद है।
        console.log(event.binding?.conversationId);
        return;
      }

      // अनुरोध अस्वीकृत हुआ; कोई भी स्थानीय लंबित स्थिति साफ़ करें।
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

कॉलबैक पेलोड फ़ील्ड:

- `status`: `"approved"` या `"denied"`
- `decision`: `"allow-once"`, `"allow-always"` या `"deny"`
- `binding`: स्वीकृत अनुरोधों के लिए समाधान की गई बाइंडिंग
- `request`: मूल अनुरोध सारांश, डिटैच संकेत, प्रेषक id और
  वार्तालाप मेटाडेटा

यह कॉलबैक केवल सूचना के लिए है। यह यह नहीं बदलता कि वार्तालाप को बाइंड करने की अनुमति किसे है,
और यह कोर अनुमोदन प्रबंधन पूरा होने के बाद चलता है।

## प्रदाता रनटाइम हुक

प्रदाता Plugin की तीन परतें होती हैं:

- **मैनिफ़ेस्ट मेटाडेटा** सस्ते पूर्व-रनटाइम लुकअप के लिए:
  `setup.providers[].envVars`, `providerAuthAliases`, `providerAuthChoices`
  और `channelConfigs`।
- **कॉन्फ़िगरेशन-समय हुक**: `catalog` और `applyConfigDefaults`।
- **रनटाइम हुक**: प्रमाणीकरण, मॉडल रिज़ॉल्यूशन,
  स्ट्रीम रैपिंग, चिंतन स्तर, रीप्ले नीति और उपयोग एंडपॉइंट को कवर करने वाले 40+ वैकल्पिक हुक। देखें
  [हुक क्रम और उपयोग](#hook-order-and-usage)।

OpenClaw अब भी सामान्य एजेंट लूप, फ़ेलओवर, ट्रांसक्रिप्ट प्रबंधन और
टूल नीति का स्वामी है। ये हुक संपूर्ण कस्टम इन्फ़रेंस ट्रांसपोर्ट की आवश्यकता के बिना प्रदाता-विशिष्ट
व्यवहार के लिए विस्तार सतह हैं।

जब प्रदाता के पास पर्यावरण-आधारित क्रेडेंशियल हों, जिन्हें सामान्य प्रमाणीकरण/स्थिति/मॉडल-चयनकर्ता पथों को
Plugin रनटाइम लोड किए बिना देखना चाहिए, तब मैनिफ़ेस्ट `setup.providers[].envVars` का उपयोग करें।
जब एक प्रदाता आईडी को किसी अन्य प्रदाता आईडी के पर्यावरण चर, प्रमाणीकरण प्रोफ़ाइल,
कॉन्फ़िगरेशन-समर्थित प्रमाणीकरण और API-कुंजी ऑनबोर्डिंग विकल्प का पुनः उपयोग करना हो, तब मैनिफ़ेस्ट `providerAuthAliases`
का उपयोग करें। जब ऑनबोर्डिंग/प्रमाणीकरण-विकल्प CLI सतहों को
प्रदाता रनटाइम लोड किए बिना प्रदाता की विकल्प आईडी, समूह लेबल और सरल एक-फ़्लैग प्रमाणीकरण वायरिंग की जानकारी होनी चाहिए,
तब मैनिफ़ेस्ट `providerAuthChoices` का उपयोग करें। ऑनबोर्डिंग लेबल या OAuth
क्लाइंट-आईडी/क्लाइंट-सीक्रेट सेटअप चर जैसे ऑपरेटर-सामना संकेतों के लिए प्रदाता रनटाइम
`envVars` बनाए रखें।

पर्यावरण-संचालित चैनल सेटअप और प्रमाणीकरण का वर्णन स्वामी
`channelConfigs.<id>.schema` और सेटअप वर्णनकर्ताओं के माध्यम से करें।

| `buildAuthDoctorHint`             | OAuth रिफ़्रेश विफल होने पर मरम्मत संकेत जोड़ता है                                                                  | रिफ़्रेश विफलता के बाद प्रोवाइडर को प्रोवाइडर-स्वामित्व वाला प्रमाणीकरण मरम्मत मार्गदर्शन चाहिए                                                                      |
| `matchesContextOverflowError`     | प्रोवाइडर-स्वामित्व वाला कॉन्टेक्स्ट-विंडो ओवरफ़्लो मैचर                                                                 | प्रोवाइडर में ऐसी कच्ची ओवरफ़्लो त्रुटियाँ हैं जिन्हें जेनेरिक ह्यूरिस्टिक्स पहचान नहीं पाएँगे                                                                                |
| `classifyFailoverReason`          | प्रोवाइडर-स्वामित्व वाला फ़ेलओवर कारण वर्गीकरण                                                                  | प्रोवाइडर कच्ची API/ट्रांसपोर्ट त्रुटियों को रेट-लिमिट/ओवरलोड/आदि में मैप कर सकता है                                                                          |
| `isCacheTtlEligible`              | प्रॉक्सी/बैकहॉल प्रोवाइडर के लिए प्रॉम्प्ट-कैश नीति                                                               | प्रोवाइडर को प्रॉक्सी-विशिष्ट कैश TTL गेटिंग चाहिए                                                                                                |
| `buildMissingAuthMessage`         | जेनेरिक अनुपलब्ध-प्रमाणीकरण रिकवरी संदेश का प्रतिस्थापन                                                      | प्रोवाइडर को प्रोवाइडर-विशिष्ट अनुपलब्ध-प्रमाणीकरण रिकवरी संकेत चाहिए                                                                                 |
| `augmentModelCatalog`             | डिस्कवरी के बाद जोड़ी गई सिंथेटिक/अंतिम कैटलॉग पंक्तियाँ (अप्रचलित, नीचे देखें)                                  | प्रोवाइडर को `models list` और पिकर में सिंथेटिक फ़ॉरवर्ड-संगतता पंक्तियाँ चाहिए                                                                     |
| `resolveThinkingProfile`          | मॉडल-विशिष्ट `/think` स्तर सेट, प्रदर्शन लेबल और डिफ़ॉल्ट                                                 | प्रोवाइडर चयनित मॉडलों के लिए कस्टम थिंकिंग लैडर या बाइनरी लेबल उपलब्ध कराता है                                                                 |
| `isBinaryThinking`                | ऑन/ऑफ़ रीजनिंग टॉगल संगतता हुक                                                                     | प्रोवाइडर केवल बाइनरी थिंकिंग ऑन/ऑफ़ उपलब्ध कराता है                                                                                                  |
| `supportsXHighThinking`           | `xhigh` रीजनिंग समर्थन संगतता हुक                                                                   | प्रोवाइडर केवल मॉडलों के एक उपसमुच्चय पर `xhigh` चाहता है                                                                                             |
| `resolveDefaultThinkingLevel`     | डिफ़ॉल्ट `/think` स्तर संगतता हुक                                                                      | किसी मॉडल फ़ैमिली के लिए डिफ़ॉल्ट `/think` नीति का स्वामी प्रोवाइडर है                                                                                      |
| `isModernModelRef`                | लाइव प्रोफ़ाइल फ़िल्टर और स्मोक चयन के लिए आधुनिक-मॉडल मैचर                                              | लाइव/स्मोक पसंदीदा-मॉडल मिलान का स्वामी प्रोवाइडर है                                                                                             |
| `prepareRuntimeAuth`              | इन्फ़रेंस से ठीक पहले कॉन्फ़िगर किए गए क्रेडेंशियल को वास्तविक रनटाइम टोकन/कुंजी में बदलता है                       | प्रोवाइडर को टोकन एक्सचेंज या अल्पकालिक अनुरोध क्रेडेंशियल चाहिए                                                                             |
| `resolveUsageAuth`                | `/usage` और संबंधित स्थिति सतहों के लिए उपयोग/बिलिंग क्रेडेंशियल रिज़ॉल्व करता है                                     | प्रोवाइडर को कस्टम उपयोग/कोटा टोकन पार्सिंग या किसी अलग उपयोग क्रेडेंशियल की आवश्यकता है                                                               |
| `fetchUsageSnapshot`              | प्रमाणीकरण रिज़ॉल्व होने के बाद प्रोवाइडर-विशिष्ट उपयोग/कोटा स्नैपशॉट फ़ेच और नॉर्मलाइज़ करता है                             | प्रोवाइडर को प्रोवाइडर-विशिष्ट उपयोग एंडपॉइंट या पेलोड पार्सर चाहिए                                                                           |
| `createEmbeddingProvider`         | मेमोरी/खोज के लिए प्रदाता-स्वामित्व वाला एम्बेडिंग अडैप्टर बनाएँ                                                     | मेमोरी एम्बेडिंग व्यवहार प्रदाता Plugin के अंतर्गत होना चाहिए                                                                                    |
| `buildReplayPolicy`               | प्रदाता के लिए ट्रांसक्रिप्ट प्रबंधन नियंत्रित करने वाली रीप्ले नीति लौटाएँ                                        | प्रदाता को कस्टम ट्रांसक्रिप्ट नीति चाहिए (उदाहरण के लिए, थिंकिंग-ब्लॉक हटाना)                                                               |
| `sanitizeReplayHistory`           | सामान्य ट्रांसक्रिप्ट सफ़ाई के बाद रीप्ले इतिहास फिर से लिखें                                                        | प्रदाता को साझा Compaction सहायकों से परे प्रदाता-विशिष्ट रीप्ले पुनर्लेखन चाहिए                                                             |
| `validateReplayTurns`             | एम्बेडेड रनर से पहले अंतिम रीप्ले-टर्न सत्यापन या पुनर्रचना करें                                           | सामान्य सैनिटाइज़ेशन के बाद प्रदाता ट्रांसपोर्ट को अधिक सख़्त टर्न सत्यापन चाहिए                                                                    |
| `onModelSelected`                 | प्रदाता-स्वामित्व वाले चयन-पश्चात दुष्प्रभाव चलाएँ                                                                 | मॉडल के सक्रिय होने पर प्रदाता को टेलीमेट्री या प्रदाता-स्वामित्व वाली स्थिति चाहिए                                                                  |

`normalizeModelId`, `normalizeTransport`, और `normalizeConfig` पहले मिलान किए गए
प्रदाता Plugin की जाँच करते हैं, फिर अन्य हुक-सक्षम प्रदाता Plugins पर आगे बढ़ते हैं
जब तक उनमें से कोई वास्तव में मॉडल id या ट्रांसपोर्ट/कॉन्फ़िगरेशन को बदल नहीं देता। इससे
कॉलर को यह जाने बिना कि पुनर्लेखन का स्वामी कौन-सा बंडल किया गया Plugin है,
उपनाम/संगतता प्रदाता शिम काम करते रहते हैं। यदि कोई प्रदाता हुक किसी समर्थित
Google-फ़ैमिली कॉन्फ़िगरेशन प्रविष्टि को पुनर्लेखित नहीं करता, तो बंडल किया गया Google कॉन्फ़िगरेशन नॉर्मलाइज़र फिर भी
वह संगतता क्लीनअप लागू करता है।

यदि प्रदाता को पूरी तरह कस्टम वायर प्रोटोकॉल या कस्टम अनुरोध निष्पादक चाहिए,
तो वह एक्सटेंशन की अलग श्रेणी है। ये हुक उस प्रदाता व्यवहार के लिए हैं
जो अब भी OpenClaw के सामान्य इन्फ़रेंस लूप पर चलता है।

`resolveUsageAuth` तय करता है कि OpenClaw को `fetchUsageSnapshot` कॉल करना चाहिए या
उपयोग/स्थिति सतहों के लिए सामान्य क्रेडेंशियल समाधान पर वापस जाना चाहिए। जब प्रदाता के पास
उपयोग क्रेडेंशियल हो, तो `{ token, accountId?, subscriptionType?, rateLimitTier? }` लौटाएँ
(वैकल्पिक प्लान मेटाडेटा `fetchUsageSnapshot` में प्रवाहित होता है), जब
प्रदाता-स्वामित्व वाले उपयोग प्रमाणीकरण ने अनुरोध संभाल लिया हो और उसे सामान्य
API-कुंजी/OAuth फ़ॉलबैक को रोकना आवश्यक हो, तो `{ handled: true }` लौटाएँ, और जब प्रदाता ने उपयोग
प्रमाणीकरण नहीं संभाला हो, तो `null` या `undefined` लौटाएँ।

संगठन या बिलिंग क्रेडेंशियल को मैनिफ़ेस्ट
`providerUsageAuthEnvVars` में घोषित करें। इससे सामान्य खोज और सीक्रेट-स्क्रबिंग
सतहें उन्हें इन्फ़रेंस प्रमाणीकरण उम्मीदवार बनाए बिना पहचान सकती हैं।

### प्रदाता उदाहरण

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

## रनटाइम सहायक

Plugins `api.runtime` के माध्यम से चुने हुए कोर सहायकों तक पहुँच सकते हैं। TTS के लिए:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

टिप्पणियाँ:

- `textToSpeech` फ़ाइल/वॉइस-नोट सतहों के लिए सामान्य कोर TTS आउटपुट पेलोड लौटाता है।
- कोर `messages.tts` कॉन्फ़िगरेशन और प्रदाता चयन का उपयोग करता है।
- PCM ऑडियो बफ़र + सैंपल दर लौटाता है। Plugins को प्रदाताओं के लिए फिर से सैंपल/एन्कोड करना आवश्यक है।
- `listVoices` प्रत्येक प्रदाता के लिए वैकल्पिक है। विक्रेता-स्वामित्व वाले वॉइस चयनकर्ता या सेटअप प्रवाह के लिए इसका उपयोग करें।
- कोर प्रदाता `listVoices` हुक को समाधान की गई अनुरोध समय-सीमा देता है; प्रदाता-विशिष्ट टाइमआउट सेटिंग इसे ओवरराइड कर सकती हैं।
- वॉइस सूचियों में प्रदाता-जागरूक चयनकर्ताओं के लिए लोकेल, लिंग और व्यक्तित्व टैग जैसे अधिक समृद्ध मेटाडेटा शामिल हो सकते हैं।
- OpenAI और ElevenLabs आज टेलीफ़ोनी का समर्थन करते हैं। Microsoft नहीं करता।

Plugins `api.registerSpeechProvider(...)` के माध्यम से स्पीच प्रदाता भी पंजीकृत कर सकते हैं।

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

टिप्पणियाँ:

- TTS नीति, फ़ॉलबैक और उत्तर वितरण को कोर में रखें।
- विक्रेता-स्वामित्व वाले संश्लेषण व्यवहार के लिए स्पीच प्रदाताओं का उपयोग करें।
- पुराने Microsoft `edge` इनपुट को `microsoft` प्रदाता id में सामान्यीकृत किया जाता है।
- वरीय स्वामित्व मॉडल कंपनी-उन्मुख है: जैसे-जैसे OpenClaw इन
  क्षमता अनुबंधों को जोड़ता है, एक विक्रेता Plugin टेक्स्ट, स्पीच, इमेज और भविष्य के
  मीडिया प्रदाताओं का स्वामी हो सकता है।

इमेज/ऑडियो/वीडियो समझ के लिए, Plugins किसी सामान्य कुंजी/मान बैग के बजाय एक टाइप किया हुआ
मीडिया-अंडरस्टैंडिंग प्रदाता पंजीकृत करते हैं:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

टिप्पणियाँ:

- ऑर्केस्ट्रेशन, फ़ॉलबैक, कॉन्फ़िगरेशन और चैनल वायरिंग को कोर में रखें।
- विक्रेता व्यवहार को प्रदाता Plugin में रखें।
- योगात्मक विस्तार टाइप किया हुआ रहना चाहिए: नई वैकल्पिक विधियाँ, नए वैकल्पिक
  परिणाम फ़ील्ड, नई वैकल्पिक क्षमताएँ।
- वीडियो जनरेशन पहले से इसी पैटर्न का पालन करता है:
  - कोर क्षमता अनुबंध और रनटाइम सहायक का स्वामी है
  - विक्रेता Plugins `api.registerVideoGenerationProvider(...)` पंजीकृत करते हैं
  - फ़ीचर/चैनल Plugins `api.runtime.videoGeneration.*` का उपयोग करते हैं

मीडिया-अंडरस्टैंडिंग रनटाइम सहायकों के लिए, Plugins कॉल कर सकते हैं:

```ts
const video = await api.runtime.mediaUnderstanding.describeVideoFile({

const extraction = await api.runtime.mediaUnderstanding.extractStructuredWithModel({
```

ऑडियो ट्रांसक्रिप्शन के लिए, Plugins मीडिया-अंडरस्टैंडिंग रनटाइम
या पुराने STT उपनाम में से किसी एक का उपयोग कर सकते हैं:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",

```

टिप्पणियाँ:

- `api.runtime.mediaUnderstanding.*` इमेज/ऑडियो/वीडियो समझ के लिए वरीय साझा सतह है।
- `extractStructuredWithModel(...)` सीमित
  प्रदाता-स्वामित्व वाले इमेज-प्रथम निष्कर्षण के लिए Plugin-सामना करने वाली सीमा है। कम-से-कम एक इमेज इनपुट शामिल करें;
  टेक्स्ट इनपुट पूरक संदर्भ हैं। उत्पाद Plugins अपने रूट और
  स्कीमा के स्वामी होते हैं, जबकि OpenClaw प्रदाता/रनटाइम सीमा का स्वामी होता है।
- कोर मीडिया-अंडरस्टैंडिंग ऑडियो कॉन्फ़िगरेशन (`tools.media.audio`) और प्रदाता फ़ॉलबैक क्रम का उपयोग करता है।
- जब कोई ट्रांसक्रिप्शन आउटपुट उत्पन्न नहीं होता (उदाहरण के लिए छोड़ा गया/असमर्थित इनपुट), तो `{ text: undefined }` लौटाता है।

Plugins `api.runtime.subagent` के माध्यम से पृष्ठभूमि सबएजेंट रन भी आरंभ कर सकते हैं:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  toolsAlsoAllow: ["my_plugin_progress"],
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

टिप्पणियाँ:

- `provider` और `model` वैकल्पिक प्रति-रन ओवरराइड हैं, स्थायी सत्र परिवर्तन नहीं।
- `toolsAlsoAllow` कॉल करने वाले Plugin द्वारा पंजीकृत सटीक, विशिष्ट-स्वामित्व वाले टूल नाम स्वीकार करता है। कोर और अस्पष्ट नाम अस्वीकार किए जाते हैं। यह सामान्य प्रोफ़ाइल में योगात्मक है, लेकिन ऑपरेटर अनुमति-सूचियाँ और निषेध प्रामाणिक बने रहते हैं।
- OpenClaw उन ओवरराइड फ़ील्ड का पालन केवल विश्वसनीय कॉलर के लिए करता है।
- Plugin-स्वामित्व वाले फ़ॉलबैक रन के लिए, ऑपरेटरों को `plugins.entries.<id>.subagent.allowModelOverride: true` से ऑप्ट इन करना आवश्यक है।
- विश्वसनीय Plugins को विशिष्ट कैनोनिकल `provider/model` लक्ष्यों तक सीमित करने के लिए `plugins.entries.<id>.subagent.allowedModels`, या किसी भी लक्ष्य को स्पष्ट रूप से अनुमति देने के लिए `"*"` का उपयोग करें।
- अविश्वसनीय Plugin सबएजेंट रन अब भी काम करते हैं, लेकिन ओवरराइड अनुरोध चुपचाप फ़ॉलबैक करने के बजाय अस्वीकार किए जाते हैं।
- Plugin द्वारा बनाए गए सबएजेंट सत्रों को बनाने वाले Plugin id से टैग किया जाता है। फ़ॉलबैक `api.runtime.subagent.deleteSession(...)` केवल उन स्वामित्व वाले सत्रों को हटा सकता है; मनमाने सत्र हटाने के लिए अब भी एडमिन-स्कोप वाला Gateway अनुरोध आवश्यक है।

वेब खोज के लिए, Plugins एजेंट टूल वायरिंग तक पहुँचने के बजाय
साझा रनटाइम सहायक का उपयोग कर सकते हैं:

```ts
const providers = api.runtime.webSearch.listProviders({

const result = await api.runtime.webSearch.search({
```

टिप्पणियाँ:

- प्रदाता चयन, क्रेडेंशियल समाधान और साझा अनुरोध सिमैंटिक्स को कोर में रखें।
- विक्रेता-विशिष्ट खोज ट्रांसपोर्ट के लिए वेब-खोज प्रदाताओं का उपयोग करें।
- `api.runtime.webSearch.*` उन फ़ीचर/चैनल Plugins के लिए वरीय साझा सतह है जिन्हें एजेंट टूल रैपर पर निर्भर हुए बिना खोज व्यवहार चाहिए।

## Gateway HTTP रूट

Plugin, `api.registerHttpRoute(...)` के साथ HTTP एंडपॉइंट उपलब्ध करा सकते हैं।

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

रूट फ़ील्ड:

- `path`: Gateway HTTP सर्वर के अंतर्गत रूट पथ।
- `auth`: आवश्यक, `"gateway"` या `"plugin"`। सामान्य Gateway प्रमाणीकरण आवश्यक करने के लिए `"gateway"`, या Plugin-प्रबंधित प्रमाणीकरण/Webhook सत्यापन के लिए `"plugin"` का उपयोग करें।
- `match`: वैकल्पिक। `"exact"` (डिफ़ॉल्ट) या `"prefix"`।
- `handleUpgrade`: उसी रूट पर WebSocket अपग्रेड अनुरोधों के लिए वैकल्पिक हैंडलर।
- `replaceExisting`: वैकल्पिक। उसी Plugin को अपना मौजूदा रूट पंजीकरण बदलने की अनुमति देता है।
- `handler`: जब रूट ने अनुरोध संभाल लिया हो, तब `true` लौटाएँ।

नोट:

- `api.registerHttpHandler(...)` हटा दिया गया है और इससे Plugin-लोड त्रुटि होगी। इसके बजाय `api.registerHttpRoute(...)` का उपयोग करें।
- Plugin रूट को `auth` स्पष्ट रूप से घोषित करना होगा।
- सटीक `path + match` टकराव अस्वीकार किए जाते हैं, जब तक कि `replaceExisting: true` न हो, और कोई Plugin दूसरे Plugin के रूट को नहीं बदल सकता।
- अलग-अलग `auth` स्तरों वाले ओवरलैपिंग रूट अस्वीकार किए जाते हैं। `exact`/`prefix` फ़ॉलथ्रू शृंखलाओं को केवल समान प्रमाणीकरण स्तर पर रखें।
- `auth: "plugin"` रूट को ऑपरेटर रनटाइम स्कोप स्वतः **नहीं** मिलते। वे Plugin-प्रबंधित Webhook/हस्ताक्षर सत्यापन के लिए हैं, विशेषाधिकार-प्राप्त Gateway सहायक कॉल के लिए नहीं।
- `auth: "gateway"` रूट Gateway अनुरोध रनटाइम स्कोप के भीतर चलते हैं। डिफ़ॉल्ट सतह (`gatewayRuntimeScopeSurface: "write-default"`) जानबूझकर सीमित रखी गई है:
  - साझा-सीक्रेट बेयरर प्रमाणीकरण (`gateway.auth.mode = "token"` / `"password"`) और किसी भी गैर-विश्वसनीय-प्रॉक्सी प्रमाणीकरण विधि को केवल एक `operator.write` स्कोप मिलता है, भले ही कॉलर `x-openclaw-scopes` भेजे
  - स्पष्ट `x-openclaw-scopes` हेडर के बिना `trusted-proxy` कॉलर भी पुरानी केवल-`operator.write` सतह बनाए रखते हैं
  - `x-openclaw-scopes` भेजने वाले `trusted-proxy` कॉलर को इसके बजाय घोषित स्कोप मिलते हैं
  - पहचान-युक्त प्रमाणीकरण मोड के लिए `x-openclaw-scopes` का हमेशा सम्मान करने हेतु कोई रूट `gatewayRuntimeScopeSurface: "trusted-operator"` चुन सकता है (हेडर अनुपस्थित होने पर पूर्ण CLI डिफ़ॉल्ट स्कोप सेट का उपयोग किया जाता है)
- `auth: "gateway"` रूट द्वारा समर्थित सैंडबॉक्स किए गए बाहरी Control UI टैब, केवल प्रमाणित बूटस्ट्रैप द्वारा जारी अल्पकालिक हस्ताक्षरित कुकी अनुदान का उपयोग करते हैं; Plugin-प्रमाणीकरण टैब अपना सीधा iframe पथ बनाए रखते हैं। माउंट करने से पहले, पैरेंट उसी अपारदर्शी सैंडबॉक्स के भीतर रूट-स्वामित्व वाली जाँच चलाता है और ब्राउज़र की गोपनीयता नीति द्वारा कुकी अवरुद्ध किए जाने पर बंद होकर विफल होता है। अनुदान स्वामी Plugin, मेल खाने वाले रूट मूल और वर्तमान प्रमाणीकरण जनरेशन से बँधा होता है; इसका प्रक्रिया-यादृच्छिक कुकी नाम समान होस्ट वाले विश्वसनीय Gateways को एक-दूसरे की कुकी अधिलेखित करने से रोकता है, लेकिन कुकी कभी भी TCP पोर्ट को अलग नहीं करती। इसलिए Gateway होस्टनाम एक क्रेडेंशियल सीमा है: उस होस्टनाम पर परस्पर अविश्वसनीय सेवाएँ, अन्य पोर्ट सहित, साथ में होस्ट न करें। रूट डिस्पैच किसी दूसरे Plugin के स्वामित्व वाले नेस्टेड रूट पर पुनः उपयोग अस्वीकार करता है। चूँकि कुकी के संदर्भ में सैंडबॉक्स के वंशज क्रॉस-साइट होते हैं, इसलिए अनुदान केवल `GET` और `HEAD` को `operator.read` के साथ स्वीकार करता है; बदलाव और WebSocket अपग्रेड स्पष्ट Gateway-प्रमाणित सतहों पर बने रहते हैं। कुकी जानबूझकर CHIPS का उपयोग नहीं कर सकती: वर्तमान ब्राउज़र विभाजन कुंजी में क्रॉस-साइट-पूर्वज बिट शामिल करते हैं, इसलिए नेस्टेड अपारदर्शी सैंडबॉक्स फ़्रेम समान-रूट एसेट की पहुँच खो देंगे। कुकी के लिए सुरक्षित संदर्भ और क्रॉस-साइट कुकी हेतु ब्राउज़र अनुमति आवश्यक है, इसलिए सादे-HTTP LAN मूल या पूर्ण तृतीय-पक्ष-कुकी अवरोधन के अंतर्गत Gateway-प्रमाणीकरण वाले बाहरी टैब उपलब्ध नहीं होते; HTTPS/Tailscale Serve या संगत कुकी नीति वाला ब्राउज़र-विश्वसनीय लूपबैक उपयोग करें।
- यह अनुदान Gateway बेयरर-टोकन के प्रकटीकरण और आकस्मिक रूट/स्कोप पुनः उपयोग को रोकता है; यह नेटिव Plugin के बीच सुरक्षा सीमा नहीं बनाता। नेटिव Plugin कोड और उसके द्वारा प्रस्तुत UI सामग्री समान विश्वसनीय इन-प्रोसेस Plugin सीमा का भाग बने रहते हैं।
- व्यावहारिक नियम: यह न मानें कि Gateway-प्रमाणीकरण वाला Plugin रूट अप्रत्यक्ष एडमिन सतह है। यदि आपके रूट को केवल-एडमिन व्यवहार चाहिए, तो `trusted-operator` स्कोप सतह चुनें, पहचान-युक्त प्रमाणीकरण मोड आवश्यक करें और स्पष्ट `x-openclaw-scopes` हेडर अनुबंध का दस्तावेज़ीकरण करें।
- रूट मिलान और प्रमाणीकरण के बाद, सामान्य हैंडलर Gateway रूट-वर्क प्रवेश में भाग लेते हैं। तैयार हो रहा या पुनः आरंभ हो रहा Gateway हैंडलर को चलाने से पहले `503` लौटाता है। सीमित अपवाद एक मैनिफ़ेस्ट-अधिकृत `auth: "gateway"` रूट है, जो रूट-विशिष्ट `trusted-operator` सतह भी चुनता है; यह पहुँच योग्य रहता है ताकि निलंबन नियंत्रण डिस्पैच अटक न जाए, जबकि उसी Plugin के सामान्य सहोदर रूट प्रवेश सीमा के पीछे रहते हैं। WebSocket `handleUpgrade` स्वामित्व समान परमाणु प्रवेश सीमा का उपयोग करता है; हैंडलर द्वारा सॉकेट स्वीकार करने के बाद, सॉकेट का आगामी जीवनकाल Plugin के स्वामित्व में होता है और इस सीमा द्वारा ट्रैक नहीं किया जाता।

## Plugin SDK इम्पोर्ट पथ

नए Plugin बनाते समय एकल `openclaw/plugin-sdk` रूट
बैरल के बजाय सीमित SDK उपपथों का उपयोग करें। मुख्य उपपथ:

| उपपथ                            | उद्देश्य                                      |
| ---------------------------------- | -------------------------------------------- |
| `openclaw/plugin-sdk/plugin-entry` | Plugin पंजीकरण प्रिमिटिव               |
| `openclaw/plugin-sdk/channel-core` | चैनल एंट्री/बिल्ड सहायक                  |
| `openclaw/plugin-sdk/core`         | सामान्य साझा सहायक और व्यापक अनुबंध |

चैनल Plugin सीमित सीमों के एक परिवार से चुनते हैं — `channel-setup`,
`setup-runtime`, `setup-tools`, `channel-pairing`,
`channel-contract`, `channel-feedback`, `channel-inbound`, `channel-outbound`,
`command-auth`, `secret-input`, `webhook-ingress`,
`channel-targets`, और `channel-actions`। अनुमोदन व्यवहार को असंबंधित
Plugin फ़ील्ड में मिलाने के बजाय एक `approvalCapability` अनुबंध पर समेकित होना
चाहिए। [चैनल Plugin](/hi/plugins/sdk-channel-plugins) देखें।

रनटाइम और कॉन्फ़िगरेशन सहायक मेल खाने वाले केंद्रित `*-runtime` उपपथों के अंतर्गत रहते हैं
(`approval-runtime`, `agent-runtime`, `lazy-runtime`, `directory-runtime`,
`text-runtime`, `runtime-store`, `system-event-runtime`, `heartbeat-runtime`,
`channel-activity-runtime`, आदि)। व्यापक `config-runtime` संगतता बैरल के बजाय
`config-contracts`, `plugin-config-runtime`, `runtime-config-snapshot`, और `config-mutation`
को प्राथमिकता दें।

<Info>
`openclaw/plugin-sdk/channel-lifecycle`, छोटे चैनल सहायक फ़साड,
`openclaw/plugin-sdk/config-runtime`, और `openclaw/plugin-sdk/infra-runtime`
पुराने Plugin के लिए बहिष्कृत संगतता शिम हैं। नए कोड को इसके बजाय
अधिक सीमित सामान्य प्रिमिटिव इम्पोर्ट करने चाहिए।
</Info>

रिपॉज़िटरी-आंतरिक एंट्री पॉइंट (प्रत्येक बंडल किए गए Plugin पैकेज रूट के लिए):

- `index.js` — बंडल किए गए Plugin की एंट्री
- `api.js` — सहायक/टाइप बैरल
- `runtime-api.js` — केवल-रनटाइम बैरल
- `setup-entry.js` — सेटअप Plugin एंट्री

बाहरी Plugin को केवल `openclaw/plugin-sdk/*` उपपथ इम्पोर्ट करने चाहिए। कोर या किसी अन्य Plugin से
किसी दूसरे Plugin पैकेज का `src/*` कभी इम्पोर्ट न करें।
फ़साड से लोड किए गए एंट्री पॉइंट उपलब्ध होने पर सक्रिय रनटाइम कॉन्फ़िगरेशन स्नैपशॉट को
प्राथमिकता देते हैं, फिर डिस्क पर हल की गई कॉन्फ़िगरेशन फ़ाइल का उपयोग करते हैं।

`image-generation`, `media-understanding`,
और `speech` जैसे क्षमता-विशिष्ट उपपथ मौजूद हैं क्योंकि बंडल किए गए Plugin आज उनका उपयोग करते हैं। वे
स्वतः दीर्घकालिक स्थिर बाहरी अनुबंध नहीं हैं — उन पर निर्भर होते समय संबंधित SDK
संदर्भ पृष्ठ देखें।

## संदेश टूल स्कीमा

प्रतिक्रियाओं, पढ़ने और पोल जैसे गैर-संदेश प्रिमिटिव के लिए चैनल-विशिष्ट `describeMessageTool(...)` स्कीमा
योगदान का स्वामित्व Plugin के पास होना चाहिए।
साझा प्रेषण प्रस्तुति को प्रोवाइडर-नेटिव बटन, कॉम्पोनेंट, ब्लॉक या कार्ड फ़ील्ड के बजाय
सामान्य `MessagePresentation` अनुबंध का उपयोग करना चाहिए।
अनुबंध, फ़ॉलबैक नियम, प्रोवाइडर मैपिंग और Plugin लेखक की चेकलिस्ट के लिए
[संदेश प्रस्तुति](/hi/plugins/message-presentation) देखें।

प्रेषण-सक्षम Plugin संदेश क्षमताओं के माध्यम से घोषित करते हैं कि वे क्या रेंडर कर सकते हैं:

- सिमेंटिक प्रस्तुति ब्लॉक के लिए `presentation` (`text`, `context`,
  `divider`, `chart`, `table`, `buttons`, `select`)
- पिन किए गए डिलीवरी अनुरोधों के लिए `delivery-pin`

कोर तय करता है कि प्रस्तुति को नेटिव रूप से रेंडर करना है या उसे टेक्स्ट में घटाना है।
सामान्य संदेश टूल से प्रोवाइडर-नेटिव UI बचाव मार्ग उपलब्ध न कराएँ।
पुरानी नेटिव स्कीमा के लिए बहिष्कृत SDK सहायक मौजूदा
तृतीय-पक्ष Plugin हेतु निर्यात होते रहते हैं, लेकिन नए Plugin को उनका उपयोग नहीं करना चाहिए।

## चैनल लक्ष्य समाधान

चैनल-विशिष्ट लक्ष्य सिमेंटिक्स का स्वामित्व चैनल Plugin के पास होना चाहिए। साझा
आउटबाउंड होस्ट को सामान्य रखें और प्रोवाइडर नियमों के लिए मैसेजिंग अडैप्टर सतह का उपयोग करें:

- `messaging.inferTargetChatType({ to })` तय करता है कि सामान्यीकृत लक्ष्य को डायरेक्टरी लुकअप से पहले
  `direct`, `group`, या `channel` के रूप में माना जाना चाहिए।
- `messaging.targetResolver.looksLikeId(raw, normalized)` कोर को बताता है कि किसी
  इनपुट को डायरेक्टरी खोज के बजाय सीधे ID-जैसे समाधान पर जाना चाहिए या नहीं।
- `messaging.targetResolver.reservedLiterals` उन स्वतंत्र शब्दों को सूचीबद्ध करता है जो
  उस प्रोवाइडर के लिए चैनल/सेशन संदर्भ हैं। समाधान आरक्षित लिटरल अस्वीकार करने से पहले कॉन्फ़िगर की गई
  डायरेक्टरी प्रविष्टियाँ बनाए रखता है, फिर डायरेक्टरी में मेल न मिलने पर बंद होकर विफल होता है।
- `messaging.targetResolver.resolveTarget(...)` तब Plugin फ़ॉलबैक होता है जब
  सामान्यीकरण के बाद या डायरेक्टरी में मेल न मिलने के बाद कोर को अंतिम प्रोवाइडर-स्वामित्व वाला समाधान चाहिए।
- `messaging.resolveOutboundSessionRoute(...)` लक्ष्य हल होने के बाद प्रोवाइडर-विशिष्ट सेशन
  रूट निर्माण का स्वामित्व रखता है।

अनुशंसित विभाजन:

- पीयर/समूह खोजने से पहले होने वाले श्रेणी निर्णयों के लिए `inferTargetChatType` का उपयोग करें।
- “इसे स्पष्ट/नेटिव लक्ष्य ID मानें” जाँच के लिए `looksLikeId` का उपयोग करें।
- व्यापक डायरेक्टरी खोज के लिए नहीं, बल्कि प्रोवाइडर-विशिष्ट सामान्यीकरण फ़ॉलबैक के लिए
  `resolveTarget` का उपयोग करें।
- चैट ID, थ्रेड ID, JID, हैंडल और रूम ID जैसे प्रोवाइडर-नेटिव ID को
  सामान्य SDK फ़ील्ड में नहीं, बल्कि `target` मानों या प्रोवाइडर-विशिष्ट पैरामीटर में रखें।

## कॉन्फ़िगरेशन-समर्थित डायरेक्टरियाँ

कॉन्फ़िगरेशन से डायरेक्टरी प्रविष्टियाँ प्राप्त करने वाले Plugin को यह तर्क
Plugin में रखना चाहिए और
`openclaw/plugin-sdk/directory-runtime` के साझा सहायकों का पुनः उपयोग करना चाहिए।

इसका उपयोग तब करें जब किसी चैनल को निम्न जैसे कॉन्फ़िगरेशन-समर्थित पीयर/समूह चाहिए:

- अनुमति-सूची से संचालित DM पीयर
- कॉन्फ़िगर किए गए चैनल/समूह मैप
- खाता-स्कोप वाले स्थिर डायरेक्टरी फ़ॉलबैक

`directory-runtime` के साझा सहायक केवल सामान्य संचालन संभालते हैं:

- क्वेरी फ़िल्टरिंग
- सीमा लागू करना
- डुप्लिकेट हटाने/सामान्यीकरण के सहायक
- `ChannelDirectoryEntry[]` बनाना

चैनल-विशिष्ट खाता निरीक्षण और ID सामान्यीकरण को
Plugin कार्यान्वयन में ही रहना चाहिए।

## प्रोवाइडर कैटलॉग

प्रोवाइडर Plugin, `registerProvider({ catalog: { run(...) { ... } } })` के साथ अनुमान के लिए
मॉडल कैटलॉग परिभाषित कर सकते हैं।

`catalog.run(...)` वही संरचना लौटाता है जिसे OpenClaw
`models.providers` में लिखता है:

- `{ provider }` एक प्रदाता प्रविष्टि के लिए
- `{ providers }` एकाधिक प्रदाता प्रविष्टियों के लिए

जब Plugin प्रदाता-विशिष्ट मॉडल आईडी, आधार URL डिफ़ॉल्ट,
या प्रमाणीकरण-नियंत्रित मॉडल मेटाडेटा का स्वामी हो, तब `catalog` का उपयोग करें।

`catalog.order` यह नियंत्रित करता है कि Plugin का कैटलॉग OpenClaw के
अंतर्निहित अप्रत्यक्ष प्रदाताओं के सापेक्ष कब मर्ज होता है:

- `simple`: सामान्य API-कुंजी या परिवेश-संचालित प्रदाता
- `profile`: प्रमाणीकरण प्रोफ़ाइल मौजूद होने पर दिखाई देने वाले प्रदाता
- `paired`: कई संबंधित प्रदाता प्रविष्टियाँ संश्लेषित करने वाले प्रदाता
- `late`: अन्य अप्रत्यक्ष प्रदाताओं के बाद अंतिम चरण

कुंजी टकराव होने पर बाद वाले प्रदाता प्रभावी होते हैं, इसलिए plugins समान प्रदाता आईडी वाली
अंतर्निहित प्रदाता प्रविष्टि को जानबूझकर ओवरराइड कर सकते हैं।

Plugins `api.registerModelCatalogProvider({ provider, kinds, staticCatalog, liveCatalog
})` के माध्यम से केवल-पठन मॉडल पंक्तियाँ भी प्रकाशित कर सकते हैं।
यह सूची/सहायता/चयनकर्ता सतहों के लिए आगे का मार्ग है और `text`, `voice`,
`image_generation`, `video_generation`, और `music_generation` पंक्तियों का समर्थन करता है।
प्रदाता plugins अब भी लाइव एंडपॉइंट कॉल, टोकन विनिमय और
विक्रेता प्रतिक्रिया मैपिंग के स्वामी हैं; कोर सामान्य पंक्ति आकार, स्रोत लेबल और
मीडिया टूल सहायता प्रारूपण का स्वामी है। मीडिया-जनरेशन प्रदाता पंजीकरण
`defaultModel`, `models`, और `capabilities` से स्थिर कैटलॉग पंक्तियाँ
स्वचालित रूप से संश्लेषित करते हैं।

संगतता:

- `discovery` अब भी विरासती उपनाम के रूप में काम करता है, लेकिन अप्रचलन चेतावनी देता है
- यदि `catalog` और `discovery` दोनों पंजीकृत हैं, तो OpenClaw `catalog` का उपयोग करता है
  और चेतावनी देता है
- `augmentModelCatalog` अप्रचलित है; बंडल किए गए प्रदाताओं को
  `registerModelCatalogProvider` के माध्यम से पूरक पंक्तियाँ प्रकाशित करनी चाहिए

## केवल-पठन चैनल निरीक्षण

यदि आपका Plugin कोई चैनल पंजीकृत करता है, तो `resolveAccount(...)` के साथ
`plugin.config.inspectAccount(cfg, accountId)` लागू करना बेहतर है।

क्यों:

- `resolveAccount(...)` रनटाइम मार्ग है। इसे यह मानने की अनुमति है कि क्रेडेंशियल
  पूरी तरह मूर्त रूप ले चुके हैं और आवश्यक सीक्रेट अनुपलब्ध होने पर यह तुरंत विफल हो सकता है।
- `openclaw status`, `openclaw status --all`,
  `openclaw channels status`, `openclaw channels resolve`, और doctor/config
  सुधार प्रवाह जैसे केवल-पठन कमांड मार्गों को केवल कॉन्फ़िगरेशन का वर्णन करने के लिए
  रनटाइम क्रेडेंशियल मूर्त रूप देने की आवश्यकता नहीं होनी चाहिए।

अनुशंसित `inspectAccount(...)` व्यवहार:

- केवल वर्णनात्मक खाता स्थिति लौटाएँ।
- `enabled` और `configured` संरक्षित रखें।
- प्रासंगिक होने पर क्रेडेंशियल स्रोत/स्थिति फ़ील्ड शामिल करें, जैसे:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- केवल-पठन उपलब्धता की रिपोर्ट देने के लिए आपको अपरिष्कृत टोकन मान लौटाने की
  आवश्यकता नहीं है। स्थिति-शैली कमांडों के लिए `tokenStatus: "available"` (और मेल खाता स्रोत
  फ़ील्ड) लौटाना पर्याप्त है।
- जब कोई क्रेडेंशियल SecretRef के माध्यम से कॉन्फ़िगर किया गया हो लेकिन
  वर्तमान कमांड मार्ग में अनुपलब्ध हो, तब `configured_unavailable` का उपयोग करें।

इससे केवल-पठन कमांड क्रैश होने या खाते को गलत ढंग से गैर-कॉन्फ़िगर बताने के बजाय
"कॉन्फ़िगर किया गया है, लेकिन इस कमांड मार्ग में अनुपलब्ध है" रिपोर्ट कर सकते हैं।

## पैकेज पैक

किसी Plugin निर्देशिका में `openclaw.extensions` वाला `package.json` शामिल हो सकता है:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

प्रत्येक प्रविष्टि एक Plugin बनती है। यदि पैक में एकाधिक एक्सटेंशन सूचीबद्ध हैं, तो Plugin
आईडी `<manifestOrPackageName>/<fileBase>` बनती है (मौजूद होने पर मेनिफ़ेस्ट आईडी प्रभावी होती है;
अन्यथा बिना स्कोप वाला `package.json` नाम)।

यदि आपका Plugin npm निर्भरताएँ आयात करता है, तो उन्हें उसी निर्देशिका में इंस्टॉल करें ताकि
`node_modules` उपलब्ध हो (`npm install` / `pnpm install`)।

सुरक्षा नियंत्रण: प्रत्येक `openclaw.extensions` प्रविष्टि को सिमलिंक समाधान के बाद Plugin
निर्देशिका के भीतर ही रहना चाहिए। पैकेज निर्देशिका से बाहर निकलने वाली प्रविष्टियाँ
अस्वीकृत कर दी जाती हैं।

सुरक्षा टिप्पणी: `openclaw plugins install` प्रोजेक्ट-स्थानीय `npm install --omit=dev --ignore-scripts` के साथ
Plugin निर्भरताएँ इंस्टॉल करता है (कोई लाइफ़साइकल स्क्रिप्ट नहीं,
रनटाइम पर कोई डेवलपमेंट निर्भरता नहीं), और इनहेरिट की गई वैश्विक npm इंस्टॉल सेटिंग्स को अनदेखा करता है।
Plugin निर्भरता ट्री को "शुद्ध JS/TS" रखें और `postinstall` बिल्ड की आवश्यकता वाले
पैकेज से बचें।

वैकल्पिक: `openclaw.setupEntry` किसी हल्के, केवल-सेटअप मॉड्यूल की ओर संकेत कर सकता है।
जब OpenClaw को किसी अक्षम चैनल Plugin के लिए सेटअप सतहों की आवश्यकता होती है, या
जब कोई चैनल Plugin सक्षम लेकिन अभी भी गैर-कॉन्फ़िगर होता है, तो यह पूर्ण Plugin प्रविष्टि के बजाय
`setupEntry` लोड करता है। यदि आपकी मुख्य Plugin प्रविष्टि टूल, हुक या अन्य केवल-रनटाइम
कोड भी जोड़ती है, तो इससे स्टार्टअप और सेटअप हल्के रहते हैं।

वैकल्पिक: `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
किसी चैनल Plugin को Gateway के सुनना शुरू करने से पहले वाले स्टार्टअप चरण के दौरान उसी
`setupEntry` मार्ग का उपयोग करने दे सकता है, भले ही चैनल पहले से कॉन्फ़िगर हो।

इसका उपयोग केवल तभी करें जब `setupEntry` उस संपूर्ण स्टार्टअप सतह को समाहित करता हो
जिसका Gateway के सुनना शुरू करने से पहले मौजूद होना आवश्यक है। व्यवहार में इसका अर्थ है कि
सेटअप प्रविष्टि को स्टार्टअप पर निर्भर प्रत्येक चैनल-स्वामित्व वाली क्षमता पंजीकृत करनी होगी, जैसे:

- स्वयं चैनल पंजीकरण
- कोई भी HTTP रूट जो Gateway के सुनना शुरू करने से पहले उपलब्ध होना आवश्यक है
- कोई भी Gateway विधि, टूल या सेवा जिसका उसी अवधि में मौजूद होना आवश्यक है

यदि आपकी पूर्ण प्रविष्टि अब भी किसी आवश्यक स्टार्टअप क्षमता की स्वामी है, तो
इस फ़्लैग को सक्षम न करें। Plugin को डिफ़ॉल्ट व्यवहार पर रखें और OpenClaw को
स्टार्टअप के दौरान पूर्ण प्रविष्टि लोड करने दें।

बंडल किए गए चैनल केवल-सेटअप अनुबंध-सतह सहायक भी प्रकाशित कर सकते हैं, जिनका उपयोग कोर
पूर्ण चैनल रनटाइम लोड होने से पहले कर सकता है। वर्तमान सेटअप
प्रोन्नयन सतह है:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

जब कोर को पूर्ण Plugin प्रविष्टि लोड किए बिना किसी विरासती एकल-खाता चैनल
कॉन्फ़िगरेशन को `channels.<id>.accounts.*` में प्रोन्नत करना होता है, तब वह इस सतह का उपयोग करता है।
Matrix वर्तमान बंडल किया गया उदाहरण है: नामित खाते पहले से मौजूद होने पर यह केवल प्रमाणीकरण/बूटस्ट्रैप
कुंजियों को किसी नामित प्रोन्नत खाते में ले जाता है, और हमेशा `accounts.default` बनाने के बजाय
किसी कॉन्फ़िगर की गई गैर-कैनोनिकल डिफ़ॉल्ट-खाता कुंजी को संरक्षित रख सकता है।

वे सेटअप पैच एडाप्टर बंडल की गई अनुबंध-सतह खोज को आलसी बनाए रखते हैं। आयात
समय हल्का रहता है; प्रोन्नयन सतह को मॉड्यूल आयात पर बंडल किए गए चैनल स्टार्टअप में
दोबारा प्रवेश करने के बजाय केवल प्रथम उपयोग पर लोड किया जाता है।

जब उन स्टार्टअप सतहों में Gateway RPC विधियाँ शामिल हों, तो उन्हें
Plugin-विशिष्ट उपसर्ग पर रखें। कोर व्यवस्थापन नेमस्पेस (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) आरक्षित रहते हैं और हमेशा
`operator.admin` में समाधान होते हैं, भले ही कोई Plugin अधिक सीमित स्कोप का अनुरोध करे।

उदाहरण:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### चैनल कैटलॉग मेटाडेटा

चैनल plugins `openclaw.channel` के माध्यम से सेटअप/खोज मेटाडेटा और
`openclaw.install` के माध्यम से इंस्टॉल संकेत प्रकाशित कर सकते हैं। इससे कोर कैटलॉग डेटा-मुक्त रहता है।

उदाहरण:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (स्वयं-होस्टेड)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Nextcloud Talk webhook बॉट के माध्यम से स्वयं-होस्टेड चैट।",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

न्यूनतम उदाहरण से परे उपयोगी `openclaw.channel` फ़ील्ड:

- `detailLabel`: अधिक समृद्ध कैटलॉग/स्थिति सतहों के लिए द्वितीयक लेबल
- `docsLabel`: दस्तावेज़ लिंक के लिए लिंक टेक्स्ट ओवरराइड करें
- `preferOver`: कम प्राथमिकता वाली Plugin/चैनल आईडी जिन्हें यह कैटलॉग प्रविष्टि पीछे छोड़े
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: चयन-सतह पाठ नियंत्रण
- `markdownCapable`: आउटबाउंड प्रारूपण निर्णयों के लिए चैनल को Markdown-सक्षम चिह्नित करता है
- `exposure.configured`: `false` पर सेट होने पर चैनल को कॉन्फ़िगर किए गए चैनलों की सूची सतहों से छिपाता है
- `exposure.setup`: `false` पर सेट होने पर चैनल को इंटरैक्टिव सेटअप/कॉन्फ़िगर चयनकर्ताओं से छिपाता है
- `exposure.docs`: दस्तावेज़ नेविगेशन सतहों के लिए चैनल को आंतरिक/निजी चिह्नित करता है
- `quickstartAllowFrom`: चैनल को मानक त्वरित-आरंभ `allowFrom` प्रवाह में शामिल करता है
- `forceAccountBinding`: केवल एक खाता मौजूद होने पर भी स्पष्ट खाता बाइंडिंग आवश्यक करता है
- `preferSessionLookupForAnnounceTarget`: घोषणा लक्ष्य हल करते समय सत्र लुकअप को प्राथमिकता देता है

OpenClaw **बाहरी चैनल कैटलॉग** (उदाहरण के लिए, MPM
रजिस्ट्री निर्यात) भी मर्ज कर सकता है। इनमें से किसी स्थान पर JSON फ़ाइल रखें:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

या `OPENCLAW_PLUGIN_CATALOG_PATHS` (या `OPENCLAW_MPM_CATALOG_PATHS`) को
एक या अधिक JSON फ़ाइलों की ओर इंगित करें (अल्पविराम/अर्धविराम/`PATH`-सीमांकित)।
प्रत्येक फ़ाइल में `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }` होना चाहिए। पार्सर `"entries"` कुंजी के
विरासती उपनामों के रूप में `"packages"` या `"plugins"` भी स्वीकार करता है।

जनरेट की गई चैनल कैटलॉग प्रविष्टियाँ और प्रदाता इंस्टॉल कैटलॉग प्रविष्टियाँ
अपरिष्कृत `openclaw.install` ब्लॉक के साथ सामान्यीकृत इंस्टॉल-स्रोत तथ्य उजागर करती हैं।
सामान्यीकृत तथ्य बताते हैं कि npm विनिर्देश सटीक संस्करण है या परिवर्तनशील चयनकर्ता,
अपेक्षित अखंडता मेटाडेटा मौजूद है या नहीं, और स्थानीय स्रोत पथ भी उपलब्ध है या नहीं।
कैटलॉग/पैकेज पहचान ज्ञात होने पर, यदि पार्स किया गया npm पैकेज नाम उस पहचान से भटकता है
तो सामान्यीकृत तथ्य चेतावनी देते हैं। यदि `defaultChoice` अमान्य है या अनुपलब्ध स्रोत की
ओर संकेत करता है, और जब वैध npm स्रोत के बिना npm अखंडता मेटाडेटा मौजूद हो, तब भी वे
चेतावनी देते हैं। उपभोक्ताओं को `installSource` को एक योगात्मक वैकल्पिक फ़ील्ड मानना चाहिए,
ताकि हस्तनिर्मित प्रविष्टियों और कैटलॉग शिम को इसे संश्लेषित न करना पड़े।
इससे ऑनबोर्डिंग और निदान Plugin रनटाइम आयात किए बिना स्रोत-स्तर की स्थिति समझा सकते हैं।

आधिकारिक बाहरी npm प्रविष्टियों को सटीक `npmSpec` के साथ
`expectedIntegrity` को प्राथमिकता देनी चाहिए। केवल पैकेज नाम और dist-tags संगतता के लिए
अब भी काम करते हैं, लेकिन वे स्रोत-स्तर की चेतावनियाँ दिखाते हैं, ताकि कैटलॉग मौजूदा plugins को
तोड़े बिना पिन किए गए, अखंडता-जाँचे गए इंस्टॉल की ओर बढ़ सके।
जब ऑनबोर्डिंग किसी स्थानीय कैटलॉग पथ से इंस्टॉल करता है, तो वह `source: "path"` और
जहाँ संभव हो वहाँ कार्यस्थान-सापेक्ष `sourcePath` वाली प्रबंधित Plugin
Plugin इंडेक्स प्रविष्टि दर्ज करता है। निरपेक्ष परिचालन लोड पथ `plugins.load.paths` में रहता है;
इंस्टॉल रिकॉर्ड स्थानीय वर्कस्टेशन पथों को दीर्घजीवी कॉन्फ़िगरेशन में दोहराने से बचता है।
इससे स्थानीय विकास इंस्टॉल स्रोत-स्तर निदान में दिखाई देते रहते हैं और अपरिष्कृत फ़ाइल-सिस्टम
पथ प्रकट करने की दूसरी सतह नहीं जुड़ती। स्थायी `installed_plugin_index` SQLite तालिका इंस्टॉल
स्रोत की प्रामाणिक जानकारी है और Plugin रनटाइम मॉड्यूल लोड किए बिना ताज़ा की जा सकती है।
इसका `installRecords` मैप तब भी स्थायी रहता है जब कोई Plugin मेनिफ़ेस्ट अनुपलब्ध या
अमान्य हो; इसका `plugins` पेलोड पुनर्निर्माण योग्य मेनिफ़ेस्ट दृश्य है।

## संदर्भ इंजन plugins

संदर्भ इंजन plugins अंतर्ग्रहण, संयोजन और Compaction के लिए सत्र संदर्भ समन्वय के
स्वामी होते हैं। उन्हें अपने Plugin से `api.registerContextEngine(id, factory)` के साथ पंजीकृत करें,
फिर `plugins.slots.contextEngine` के साथ सक्रिय इंजन चुनें।

जब आपके Plugin को केवल मेमोरी खोज या हुक जोड़ने के बजाय डिफ़ॉल्ट संदर्भ
पाइपलाइन को बदलने या विस्तारित करने की आवश्यकता हो, तब इसका उपयोग करें।

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", (ctx) => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, sessionKey, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentSessionKey: sessionKey,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

फ़ैक्टरी `ctx` निर्माण-समय आरंभीकरण के लिए वैकल्पिक `config`, `agentDir`, और `workspaceDir`
मान उपलब्ध कराती है।

होस्ट किसी गैर-लेगेसी इंजन के `assemble()` को कॉल करने से पहले पंजीकृत एसिंक्रोनस मेमोरी प्रॉम्प्ट तैयारी पूरी करता है। `buildMemorySystemPromptAddition(...)`
सिंक्रोनस रहता है और `assemble()` के सक्रिय रहने के दौरान उस अपरिवर्तनीय रन स्नैपशॉट को पढ़ता है।
दिए गए टूल और उद्धरण संदर्भ को बिना बदले आगे भेजें, ताकि स्नैपशॉट
रन सीमाओं को पार न कर सके।

जब सक्रिय हार्नेस में एक स्थायी बैकएंड थ्रेड हो, तो `assemble()`, `contextProjection` लौटा सकता है।
लेगेसी प्रति-टर्न प्रोजेक्शन के लिए इसे छोड़ दें। जब संयोजित संदर्भ को
किसी बैकएंड थ्रेड में एक बार अंतःक्षेपित करके युग बदलने तक पुनः उपयोग किया जाना हो, तब
`{ mode: "thread_bootstrap", epoch }` लौटाएँ। इंजन का सिमैंटिक संदर्भ बदलने के बाद
युग बदलें, जैसे इंजन के स्वामित्व वाला Compaction पास पूरा होने के बाद।
होस्ट थ्रेड-बूटस्ट्रैप प्रोजेक्शन में टूल-कॉल मेटाडेटा, इनपुट
आकार और संशोधित टूल परिणाम सुरक्षित रख सकते हैं, ताकि नए
बैकएंड थ्रेड कच्चे, सीक्रेट-युक्त पेलोड कॉपी किए बिना टूल निरंतरता बनाए रखें।

यदि आपका इंजन Compaction एल्गोरिदम का स्वामी **नहीं** है, तो `compact()`
को लागू रखें और उसे स्पष्ट रूप से प्रत्यायोजित करें:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", (ctx) => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, sessionKey, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentSessionKey: sessionKey,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## नई क्षमता जोड़ना

जब किसी Plugin को ऐसे व्यवहार की आवश्यकता हो जो वर्तमान API में उपयुक्त न हो, तो
निजी आंतरिक पहुँच के माध्यम से Plugin प्रणाली को दरकिनार न करें। अनुपलब्ध क्षमता जोड़ें।

अनुशंसित क्रम:

1. **मुख्य अनुबंध परिभाषित करें।** तय करें कि साझा व्यवहार में से किसका स्वामित्व कोर के पास होना चाहिए:
   नीति, फ़ॉलबैक, कॉन्फ़िग मर्ज, जीवनचक्र, चैनल-संबंधी सिमैंटिक्स और
   रनटाइम हेल्पर का आकार।
2. **टाइप की गई Plugin पंजीकरण/रनटाइम सतहें जोड़ें।**
   सबसे छोटी उपयोगी टाइप की गई क्षमता सतह के साथ
   `OpenClawPluginApi` और/या `api.runtime` का विस्तार करें।
3. **कोर + चैनल/फ़ीचर उपभोक्ताओं को जोड़ें।** चैनलों और फ़ीचर Plugin को
   किसी वेंडर कार्यान्वयन को सीधे इंपोर्ट करने के बजाय कोर के माध्यम से नई क्षमता का उपयोग
   करना चाहिए।
4. **वेंडर कार्यान्वयन पंजीकृत करें।** इसके बाद वेंडर Plugin अपने
   बैकएंड को क्षमता के साथ पंजीकृत करते हैं।
5. **अनुबंध कवरेज जोड़ें।** परीक्षण जोड़ें, ताकि स्वामित्व और पंजीकरण का स्वरूप
   समय के साथ स्पष्ट बना रहे।

इसी तरह OpenClaw किसी एक प्रदाता के दृष्टिकोण में हार्डकोड हुए बिना
अपना स्पष्ट मत बनाए रखता है। ठोस फ़ाइल चेकलिस्ट और तैयार उदाहरण के लिए
[क्षमता कुकबुक](/hi/plugins/adding-capabilities) देखें।

### क्षमता चेकलिस्ट

नई क्षमता जोड़ते समय, कार्यान्वयन को सामान्यतः इन
सतहों को एक साथ छूना चाहिए:

- `src/<capability>/types.ts` में कोर अनुबंध प्रकार
- `src/<capability>/runtime.ts` में कोर रनर/रनटाइम हेल्पर
- `src/plugins/types.ts` में Plugin API पंजीकरण सतह
- `src/plugins/registry.ts` में Plugin रजिस्ट्री वायरिंग
- जब फ़ीचर/चैनल Plugin को इसका उपयोग करना हो, तब `src/plugins/runtime/*` में
  Plugin रनटाइम एक्सपोज़र
- `src/test-utils/plugin-registration.ts` में कैप्चर/परीक्षण हेल्पर
- `src/plugins/contracts/registry.ts` में स्वामित्व/अनुबंध अभिकथन
- `docs/` में ऑपरेटर/Plugin दस्तावेज़

यदि इनमें से कोई सतह अनुपस्थित है, तो यह सामान्यतः संकेत है कि क्षमता अभी
पूरी तरह एकीकृत नहीं हुई है।

### क्षमता टेम्पलेट

न्यूनतम पैटर्न:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

अनुबंध परीक्षण पैटर्न (`src/plugins/contracts/registry.ts`, `providerContractPluginIds` जैसी स्वामित्व
लुकअप उपलब्ध कराता है; परीक्षण अभिकथित करते हैं कि किसी Plugin की
`contracts.videoGenerationProviders` सूची उसके वास्तविक पंजीकरण से मेल खाती है):

```ts
expect(pluginManifest.contracts?.videoGenerationProviders).toEqual(["openai"]);
```

इससे नियम सरल रहता है:

- कोर क्षमता अनुबंध + ऑर्केस्ट्रेशन का स्वामी है
- वेंडर Plugin वेंडर कार्यान्वयनों के स्वामी हैं
- फ़ीचर/चैनल Plugin रनटाइम हेल्पर का उपयोग करते हैं
- अनुबंध परीक्षण स्वामित्व को स्पष्ट बनाए रखते हैं
