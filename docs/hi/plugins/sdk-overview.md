Plugin SDK, plugins और core के बीच टाइप किया गया अनुबंध है। यह पृष्ठ
**क्या इंपोर्ट करना है** और **क्या रजिस्टर किया जा सकता है**, इसका संदर्भ है।

<Note>
  यह पृष्ठ OpenClaw के भीतर `openclaw/plugin-sdk/*` का उपयोग करने वाले
  Plugin लेखकों के लिए है। Gateway के माध्यम से एजेंट चलाने वाले बाहरी ऐप्स, स्क्रिप्ट्स,
  डैशबोर्ड्स, CI जॉब्स और IDE एक्सटेंशंस के लिए इसके बजाय
  [बाहरी ऐप्स के लिए Gateway एकीकरण](/hi/gateway/external-apps) का उपयोग करें।
</Note>

<Tip>
इसके बजाय कोई उपयोग-विधि मार्गदर्शिका खोज रहे हैं? [Plugins बनाना](/hi/plugins/building-plugins) से शुरू करें। चैनलों के लिए [चैनल plugins](/hi/plugins/sdk-channel-plugins), मॉडल प्रदाताओं के लिए [प्रदाता plugins](/hi/plugins/sdk-provider-plugins), स्थानीय AI CLI बैकएंड्स के लिए [CLI बैकएंड plugins](/hi/plugins/cli-backend-plugins), नेटिव एजेंट निष्पादकों के लिए [एजेंट हार्नेस plugins](/hi/plugins/sdk-agent-harness), और टूल या जीवनचक्र हुक्स के लिए [Plugin हुक्स](/hi/plugins/hooks) का उपयोग करें।
</Tip>

## इंपोर्ट परिपाटी

हमेशा किसी विशिष्ट सबपाथ से इंपोर्ट करें:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { defineChannelPluginEntry } from "openclaw/plugin-sdk/channel-core";
```

हर सबपाथ एक छोटा, स्व-निहित मॉड्यूल है। इससे स्टार्टअप तेज़ रहता है और
चक्रीय निर्भरता संबंधी समस्याएँ रुकती हैं। चैनल-विशिष्ट एंट्री/बिल्ड सहायकों के लिए,
`openclaw/plugin-sdk/channel-core` को प्राथमिकता दें; व्यापक अम्ब्रेला सतह और
`buildChannelConfigSchema` जैसे साझा सहायकों के लिए `openclaw/plugin-sdk/core` रखें।

चैनल कॉन्फ़िगरेशन के लिए, चैनल के स्वामित्व वाला JSON Schema
`openclaw.plugin.json#channelConfigs` के माध्यम से प्रकाशित करें। `plugin-sdk/channel-config-schema`
सबपाथ साझा स्कीमा प्रिमिटिव्स और जेनेरिक बिल्डर के लिए है। OpenClaw के
बंडल किए गए plugins, बनाए रखे गए बंडल-चैनल स्कीमा के लिए
`plugin-sdk/bundled-channel-config-schema` का उपयोग करते हैं। वह बंडल किया गया स्कीमा सबपाथ नए
plugins के लिए पैटर्न नहीं है।

<Warning>
  प्रदाता- या चैनल-ब्रांडेड सुविधा सीमों (उदाहरण के लिए
  `openclaw/plugin-sdk/slack`, `.../discord`, `.../signal`, `.../whatsapp`) को इंपोर्ट न करें।
  बंडल किए गए plugins अपने स्वयं के `api.ts` /
  `runtime-api.ts` बैरल्स के भीतर जेनेरिक SDK सबपाथ्स संयोजित करते हैं; core उपभोक्ताओं
  को या तो उन Plugin-स्थानीय बैरल्स का उपयोग करना चाहिए या आवश्यकता वास्तव में
  सभी चैनलों पर लागू होने पर एक संकीर्ण जेनेरिक SDK अनुबंध जोड़ना चाहिए।

बंडल किए गए Plugin की सहायक सीमों का एक छोटा समूह ट्रैक किए गए स्वामी उपयोग के
समय अब भी जनरेट किए गए एक्सपोर्ट मैप में दिखाई देता है। वे केवल बंडल किए गए Plugin के
रखरखाव के लिए मौजूद हैं और नए तृतीय-पक्ष plugins के लिए अनुशंसित इंपोर्ट पथ नहीं हैं।

`openclaw/plugin-sdk/discord` और `openclaw/plugin-sdk/telegram-account` को भी
ट्रैक किए गए स्वामी उपयोग के लिए पदावनत संगतता फ़साड के रूप में रखा गया है। उन
इंपोर्ट पथों को नए plugins में कॉपी न करें; इसके बजाय इंजेक्ट किए गए रनटाइम सहायकों और
जेनेरिक चैनल SDK सबपाथ्स का उपयोग करें।
</Warning>

## सबपाथ संदर्भ

Plugin SDK को क्षेत्र के अनुसार समूहित संकीर्ण सबपाथ्स के समूह के रूप में प्रदर्शित किया जाता है (Plugin
एंट्री, चैनल, प्रदाता, प्रमाणीकरण, रनटाइम, क्षमता, मेमोरी और आरक्षित
बंडल किए गए Plugin सहायक)। समूहित और लिंक किए गए संपूर्ण कैटलॉग के लिए
[Plugin SDK सबपाथ्स](/hi/plugins/sdk-subpaths) देखें।

कंपाइलर एंट्रीपॉइंट इन्वेंट्री
`scripts/lib/plugin-sdk-entrypoints.json` में रहती है; पैकेज एक्सपोर्ट्स को
`scripts/lib/plugin-sdk-private-local-only-subpaths.json` में सूचीबद्ध रेपो-स्थानीय परीक्षण/आंतरिक सबपाथ्स घटाने के बाद
सार्वजनिक उपसमुच्चय से जनरेट किया जाता है।
सार्वजनिक एक्सपोर्ट संख्या का ऑडिट करने के लिए `pnpm plugin-sdk:surface` चलाएँ। पर्याप्त पुराने और
बंडल किए गए एक्सटेंशन के प्रोडक्शन कोड द्वारा अप्रयुक्त पदावनत सार्वजनिक सबपाथ्स को
`scripts/lib/plugin-sdk-deprecated-public-subpaths.json` में ट्रैक किया जाता है; व्यापक
पदावनत री-एक्सपोर्ट बैरल्स को
`scripts/lib/plugin-sdk-deprecated-barrel-subpaths.json` में ट्रैक किया जाता है।

## पंजीकरण API

`register(api)` कॉलबैक को इन विधियों वाला `OpenClawPluginApi` ऑब्जेक्ट मिलता है:

किसी सत्र के लिए बाहरी टीम-चैट सतह प्रदान करने वाले plugins,
`openclaw/plugin-sdk/session-discussion` द्वारा एक्सपोर्ट किए गए एकल प्रक्रिया-व्यापी प्रदाता को रजिस्टर कर सकते हैं।
इसकी `info({ sessionKey })` विधि बताती है कि चर्चा अनुपलब्ध है, खोलने के लिए तैयार है या
पहले से खुली है; `open({ sessionKey })` चर्चा बनाती या हल करती है और उसके एम्बेड तथा
बाहरी URLs लौटाती है। दूसरा प्रदाता रजिस्टर करने पर वर्तमान प्रदाता बदल जाता है।

### क्षमता पंजीकरण

| विधि                                             | यह क्या रजिस्टर करती है                                                           |
| ------------------------------------------------ | --------------------------------------------------------------------------------- |
| `api.registerProvider(...)`                      | टेक्स्ट अनुमान (LLM)                                                              |
| `api.registerWorkerProvider(...)`                | क्लाउड-वर्कर जीवनचक्र लीज़                                                        |
| `api.registerModelCatalogProvider(...)`          | टेक्स्ट और मीडिया जनरेशन के लिए मॉडल कैटलॉग पंक्तियाँ                             |
| `api.registerAgentHarness(...)`                  | [प्रायोगिक](/hi/plugins/sdk-agent-harness) नेटिव एजेंट निष्पादक (Codex, Copilot)     |
| `api.registerCliBackend(...)`                    | स्थानीय CLI अनुमान बैकएंड                                                         |
| `api.registerChannel(...)`                       | मैसेजिंग चैनल                                                                     |
| `api.registerEmbeddingProvider(...)`             | पुनः उपयोग योग्य वेक्टर एम्बेडिंग प्रदाता                                         |
| `api.registerSpeechProvider(...)`                | टेक्स्ट-टू-स्पीच / STT संश्लेषण                                                   |
| `api.registerRealtimeTranscriptionProvider(...)` | स्ट्रीमिंग रियलटाइम ट्रांसक्रिप्शन                                                 |
| `api.registerRealtimeVoiceProvider(...)`         | डुप्लेक्स रियलटाइम वॉइस सत्र                                                       |
| `api.registerMediaUnderstandingProvider(...)`    | छवि/ऑडियो/वीडियो विश्लेषण                                                         |
| `api.registerTranscriptSourceProvider(...)`      | लाइव या इंपोर्ट किया गया मीटिंग ट्रांसक्रिप्ट स्रोत                                |
| `api.registerImageGenerationProvider(...)`       | छवि जनरेशन                                                                        |
| `api.registerMusicGenerationProvider(...)`       | संगीत जनरेशन                                                                      |
| `api.registerVideoGenerationProvider(...)`       | वीडियो जनरेशन                                                                     |
| `api.registerWebFetchProvider(...)`              | वेब फ़ेच / स्क्रेप प्रदाता                                                        |
| `api.registerWebSearchProvider(...)`             | वेब खोज                                                                            |
| `api.registerCompactionProvider(...)`            | प्लग करने योग्य ट्रांसक्रिप्ट-Compaction बैकएंड                                   |

वर्कर प्रदाताओं को `contracts.workerProviders` में अपनी id भी घोषित करनी होगी।
Core, `provision(profile, operationId)` से पहले स्थायी अभिप्राय सहेजता है। प्रदाता बाहरी आवंटन से पहले सेटिंग्स सत्यापित करते हैं और स्थायी प्रोफ़ाइल अस्वीकृति के लिए `WorkerProviderError` थ्रो करते हैं। ऑपरेशन id दोहराए जाने पर `provision` को वही लीज़ अपनानी होगी।
Core सत्यापित प्रोफ़ाइल सेटिंग्स को लीज़ के साथ सहेजता है और वह स्नैपशॉट `destroy({ leaseId, profile })`, जिसे आइडेम्पोटेंट होना चाहिए, और `inspect({ leaseId, profile })`, जो `active`, `destroyed` या `unknown` लौटाता है, को देता है। इससे प्रदाता Gateway रीस्टार्ट या नामित-प्रोफ़ाइल हटने के बाद जीवनचक्र कॉल्स रूट कर सकते हैं। SSH एंडपॉइंट्स, `keyRef` के लिए `SecretRef` का उपयोग करते हैं, कभी इनलाइन कुंजी सामग्री का नहीं, और विश्वसनीय प्रोविज़निंग आउटपुट से `hostKey` को ठीक `algorithm base64` के रूप में शामिल करते हैं, बिना होस्टनेम या टिप्पणी के। Core, `hostKey` पिन करता है और पहले कनेक्शन की कुंजी पर कभी भरोसा नहीं करता। डायनेमिक `keyRef` बनाने वाला प्रदाता `resolveSshIdentity({ leaseId, profile, keyRef })` लागू कर सकता है; मौजूद होने पर वह रिज़ॉल्वर प्रामाणिक होता है, जबकि उसके बिना प्रदाता कॉन्फ़िगर किए गए जेनेरिक सीक्रेट रिज़ॉल्वर का उपयोग करते हैं।
नवीकरणीय लीज़ वाले प्रदाता `renew(leaseId)` भी लागू कर सकते हैं।
अस्थायी या अनिर्धारित विफलताओं पर `inspect` को थ्रो करना होगा; केवल प्रामाणिक अनुपस्थिति के लिए `unknown` लौटाएँ। Core किसी सक्रिय स्थानीय रिकॉर्ड को अनाथ चिह्नित करता है, या सहेजे गए नष्ट करने के अनुरोध के बाद अनुपस्थिति को टियरडाउन पूर्ण होना मानता है।

`api.registerEmbeddingProvider(...)` के साथ रजिस्टर किए गए एम्बेडिंग प्रदाताओं को
Plugin मैनिफ़ेस्ट में `contracts.embeddingProviders` में भी सूचीबद्ध करना होगा। यह
पुनः उपयोग योग्य वेक्टर जनरेशन के लिए जेनेरिक एम्बेडिंग सतह है। मेमोरी खोज
इस जेनेरिक प्रदाता सतह का उपयोग कर सकती है। पुरानी
`api.registerMemoryEmbeddingProvider(...)` और
`contracts.memoryEmbeddingProviders` सीमा पदावनत संगतता है, जबकि
मौजूदा मेमोरी-विशिष्ट प्रदाता माइग्रेट होते हैं।

जो मेमोरी-विशिष्ट प्रदाता अब भी रनटाइम `batchEmbed(...)` प्रदर्शित करते हैं, वे
मौजूदा प्रति-फ़ाइल बैचिंग अनुबंध पर बने रहते हैं, जब तक उनका रनटाइम स्पष्ट रूप से
`sourceWideBatchEmbed: true` सेट न करे। इस ऑप्ट-इन से मेमोरी होस्ट, होस्ट बैच सीमाओं तक
एक `batchEmbed(...)` कॉल में कई गंदी मेमोरी फ़ाइलों और सक्षम स्रोतों से चंक्स भेज सकता है।
JSONL अनुरोध फ़ाइलें अपलोड करने वाले बैच एडाप्टर्स को अपने अनुरोध-संख्या
कैप के साथ-साथ अपलोड-आकार कैप से पहले भी प्रदाता जॉब्स विभाजित करनी होंगी।
प्रदाता को `batch.chunks` के समान क्रम में प्रति इनपुट चंक एक एम्बेडिंग लौटानी होगी;
जब प्रदाता फ़ाइल-स्थानीय बैचों की अपेक्षा करता हो या बड़े स्रोत-व्यापी जॉब में
इनपुट क्रम सुरक्षित न रख सकता हो, तो फ़्लैग छोड़ दें।

### टूल्स और कमांड्स

निश्चित टूल नामों वाले सरल, केवल-टूल plugins के लिए
[`defineToolPlugin`](/hi/plugins/tool-plugins) का उपयोग करें।
मिश्रित plugins या पूरी तरह डायनेमिक टूल पंजीकरण के लिए सीधे
`api.registerTool(...)` का उपयोग करें।

| विधि                                   | यह क्या रजिस्टर करती है                                                                                                                   |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `api.registerTool(tool, opts?)`        | एजेंट टूल (आवश्यक या `{ optional: true }`)                                                                                                 |
| `api.registerCommand(def)`             | कस्टम कमांड (LLM को बायपास करती है)                                                                                                      |
| `api.registerNodeHostCommand(command)` | `openclaw node run` द्वारा संभाली जाने वाली कमांड; वैकल्पिक `agentTool` मेटाडेटा इसे Node कनेक्ट रहने के दौरान एजेंट-दृश्य टूल के रूप में प्रदर्शित कर सकता है |

जब एजेंट को संक्षिप्त, कमांड-स्वामित्व वाला रूटिंग संकेत चाहिए, तब Plugin कमांड्स
`agentPromptGuidance` सेट कर सकती हैं। उस टेक्स्ट को केवल कमांड के बारे में रखें;
core प्रॉम्प्ट बिल्डर्स में प्रदाता- या Plugin-विशिष्ट नीति न जोड़ें।

मार्गदर्शन प्रविष्टियाँ लीगेसी स्ट्रिंग्स हो सकती हैं, जो हर प्रॉम्प्ट सतह पर लागू होती हैं,
या संरचित प्रविष्टियाँ:

```ts
agentPromptGuidance: [
  "वैश्विक कमांड संकेत।",
  { text: "इसे केवल मुख्य OpenClaw प्रॉम्प्ट में दिखाएँ।", surfaces: ["openclaw_main"] },
];
```

संरचित `surfaces` में `openclaw_main`, `codex_app_server`,
`cli_backend`, `acp_backend` या `subagent` शामिल हो सकते हैं।
`pi_main`, `openclaw_main` का पदावनत उपनाम बना हुआ है।
सभी सतहों के लिए जानबूझकर दिए गए मार्गदर्शन में `surfaces` छोड़ दें।
खाली `surfaces` सरणी न दें; इसे अस्वीकार किया जाता है, ताकि आकस्मिक स्कोप हानि
वैश्विक प्रॉम्प्ट टेक्स्ट न बन जाए।

नेटिव Codex ऐप-सर्वर डेवलपर निर्देश अन्य प्रॉम्प्ट सतहों की तुलना में अधिक कठोर हैं:
केवल `codex_app_server` के लिए स्पष्ट रूप से स्कोप किया गया मार्गदर्शन उस
उच्च-प्राथमिकता लेन में पदोन्नत किया जाता है। लीगेसी स्ट्रिंग मार्गदर्शन और बिना स्कोप वाला
संरचित मार्गदर्शन संगतता के लिए गैर-Codex प्रॉम्प्ट सतहों पर उपलब्ध रहता है।

Node-host कमांड कनेक्ट किए गए Node होस्ट पर चलते हैं, Gateway
प्रक्रिया के भीतर नहीं। यदि `agentTool` मौजूद है, तो सफल
Gateway कनेक्शन के बाद Node एक वर्णक प्रकाशित करता है; Gateway इसे एजेंट रन के लिए
केवल तभी उपलब्ध कराता है जब वह Node कनेक्ट हो और केवल तब, जब वर्णक का
`command` Node की स्वीकृत कमांड सतह में हो। किसी
गैर-खतरनाक कमांड को डिफ़ॉल्ट Node कमांड अनुमतिसूची में शामिल करने के लिए `agentTool.defaultPlatforms` सेट करें;
अन्यथा स्पष्ट `gateway.nodes.allowCommands` या Node-आह्वान नीति आवश्यक है। `agentTool.name`
प्रदाता-सुरक्षित होना चाहिए: यह किसी अक्षर से शुरू हो, इसमें केवल अक्षर, अंक,
अंडरस्कोर या हाइफ़न हों और इसकी लंबाई 64 वर्णों के भीतर रहे। MCP-समर्थित Node टूल
`agentTool.mcp` मेटाडेटा सेट कर सकते हैं, ताकि कैटलॉग और टूल-खोज सतहें
दूरस्थ MCP सर्वर/टूल की पहचान दिखा सकें, लेकिन निष्पादन फिर भी
विज्ञापित Node कमांड के माध्यम से ही होता है।

#### स्वीकृति-पश्चात Webhook कार्य

जो Webhook रूट प्रसंस्करण पूरा होने से पहले अनुरोध को स्वीकार करते हैं, उन्हें
उस पृथक कार्य को उसके स्वयं के ट्रैक किए गए प्रवेश रूट पर ले जाना चाहिए:

```typescript
import { runDetachedWebhookWork } from "openclaw/plugin-sdk/webhook-request-guards";

void runDetachedWebhookWork(() => processWebhookEvent(event)).catch((error) => {
  runtime.error?.(`webhook dispatch failed: ${String(error)}`);
});
```

HTTP अनुरोध के अभी भी स्वीकृत रहते हुए `runDetachedWebhookWork(...)` को सिंक्रोनस रूप से
कॉल करें। सहायक तुरंत एक स्वतंत्र रूट आरक्षित करता है और फिर अगली
माइक्रोटास्क में कॉलबैक शुरू करता है, ताकि अनुरोध हैंडलर पहले अपनी
स्वीकृति लिख सके। लौटाया गया प्रॉमिस कॉलबैक परिणाम को अपनाता है; अस्वीकृति
प्रबंधन का दायित्व अब भी कॉलर का है। इससे स्वीकृति-पश्चात कतार कार्य स्वीकार होता रहता है और
रीस्टार्ट या निलंबन ड्रेन उसके पूरा होने की प्रतीक्षा करते हैं। जो हैंडलर लौटने से
पहले पूरे प्रसंस्करण की प्रतीक्षा करते हैं, उन्हें इस सहायक की आवश्यकता नहीं है।

#### अनुरोधकर्ता-स्कोप वाले MCP कनेक्शन

MCP सर्वर की **पहचान** (नाम, टूल फ़िल्टर) को `mcp.servers` या
बंडल मैनिफ़ेस्ट में स्थिर रखें। वैकल्पिक रूप से एक कनेक्शन रिज़ॉल्वर पंजीकृत करें, ताकि प्रत्येक विश्वसनीय
संदेश अनुरोधकर्ता को अपना अलग ट्रांसपोर्ट मिले:

```ts
api.registerMcpServerConnectionResolver({
  serverName: "user-email",
  resolve: async (ctx) => {
    // ctx.requesterSenderId is host-trusted; never invent sender identity here.
    const token = await lookupUserToken(ctx.requesterSenderId);
    if (!token) {
      return null; // omit this server for the current run
    }
    return {
      url: "https://mcp.example.com/email",
      headers: { Authorization: `Bearer ${token}` },
    };
  },
});
```

अनुबंध टिप्पणियाँ:

- रिज़ॉल्वर संदर्भ में केवल विश्वसनीय होस्ट पहचान होती है (`requesterSenderId`,
  वैकल्पिक `agentAccountId` / `messageChannel`)। भविष्य के विश्वसनीय फ़ील्ड (उदाहरण के लिए
  Cron/उप-एजेंट उपयोगकर्ता संदर्भ) योगात्मक रूप से जोड़े जा सकते हैं।
- एक Plugin एक सर्वर नाम का स्वामी होता है: किसी अन्य
  Plugin से उसी `serverName` के लिए डुप्लिकेट
  `registerMcpServerConnectionResolver` को त्रुटि निदान के साथ अस्वीकार कर दिया जाता है (पहला पंजीकरण मान्य रहता है), ताकि
  कनेक्शन स्वामित्व कभी भी Plugin लोड क्रम पर निर्भर न हो।
- टूल नाम पूरे घोषित सर्वर समूह से व्युत्पन्न होते हैं, इसलिए आंशिक रिज़ॉल्यूशन
  अनुरोधकर्ताओं या टर्न के बीच सुरक्षित सर्वर नामों को कभी नहीं बदलता। कोर यह
  सत्यापित नहीं करता कि अलग-अलग अनुरोधकर्ता एंडपॉइंट समान टूल स्कीमा प्रदान करते हैं; किसी
  रिज़ॉल्वर को प्रत्येक अनुरोधकर्ता को उसी तार्किक सेवा की ओर इंगित करना चाहिए, अन्यथा टूल
  स्कीमा (और प्रॉम्प्ट-कैश स्थिरता) प्रति अनुरोधकर्ता अलग हो जाते हैं।
- विश्वसनीय `requesterSenderId` के बिना रन (Cron, उप-एजेंट, Heartbeat, सार्वजनिक
  Gateway) अनुरोधकर्ता-स्कोप वाले सर्वर कभी साकार नहीं करते। कोई साझा
  फ़ॉलबैक कनेक्शन नहीं है।
- `resolve` प्रत्येक सर्वर के लिए 10 सेकंड तक सीमित है; टाइमआउट या अपवाद उस
  सर्वर को रन से हटा देता है, लेकिन स्थिर MCP को विफल नहीं करता।
- रिज़ॉल्व किए गए कनेक्शन प्रत्येक अनुरोधकर्ता के लिए अधिकतम हर 5 मिनट में पुनः सत्यापित होते हैं:
  रोटेशन नए क्रेडेंशियल के साथ ट्रांसपोर्ट को पुनर्निर्मित करता है और `null` परिणाम
  उसे निरस्त कर देता है (कैश किया हुआ रनटाइम सत्र के बीच में भी नष्ट कर दिया जाता है)। इसलिए निरस्त या
  रोटेट किया गया क्रेडेंशियल 5 मिनट तक उपयोग में रह सकता है।
- रिज़ॉल्व किए गए `headers` कभी लॉग या स्थायी रूप से संग्रहीत नहीं किए जाते; कोर क्रेडेंशियल
  रोटेशन का पता लगाने के लिए केवल एक अस्थायी इन-मेमोरी कुंजीबद्ध डाइजेस्ट (प्रक्रिया-स्थानीय HMAC) रखता है और
  रिज़ॉल्व किए गए हेडर/URL क्रेडेंशियल मानों को लॉग/डीबग-कैप्चर
  संशोधन रजिस्ट्री में पंजीकृत करता है।
- अनुरोधकर्ता-स्कोप वाले सर्वर MCP App दृश्य नहीं बनाते: कोई दृश्य
  अनुरोधकर्ता-प्रमाणित रन से अधिक समय तक बना रहता है और Gateway दृश्य सीमा में अनुरोधकर्ता
  पहचान नहीं होती, इसलिए इन सर्वरों के लिए ऐप पूर्वावलोकन विफलता-बंद रहते हैं। टूल परिणाम
  अप्रभावित रहते हैं।
- रिज़ॉल्वर के बिना स्थिर सर्वर मौजूदा सत्र-स्कोप वाला जीवनचक्र बनाए रखते हैं।
- **हार्नेस वितरण नियम:** अनुरोधकर्ता-स्कोप वाले सर्वर कभी भी हार्नेस-नेटिव
  MCP क्लाइंट कॉन्फ़िग (Codex थ्रेड `mcp_servers`, CLI `-c mcp_servers=…`, या किसी
  अन्य सत्र-साझा MCP प्रक्षेपण) में प्रवेश नहीं करते। इसके बजाय हार्नेस उन्हें रन-स्कोप वाले
  टूल के रूप में वितरित करते हैं:
  - एम्बेडेड रनर: सत्र MCP रनटाइम + बंडल टूल (स्थिर + स्कोप वाले)।
  - Codex ऐप-सर्वर: `materializeRequesterScopedMcpToolsForHarnessRun` के माध्यम से डायनेमिक टूल
    (केवल स्कोप वाले; स्थिर
    सर्वर Codex के नेटिव MCP क्लाइंट पर बने रहते हैं)।
- स्कोप वाले टूल **विनिर्देश** उस सत्र में पहले सफल रिज़ॉल्यूशन के बाद
  सत्र-स्थिर रहते हैं, इसलिए साझा-थ्रेड हार्नेस (Codex) प्रेषक बदलने पर
  थ्रेड रोटेट नहीं करते। किसी अनुरोधकर्ता के रिज़ॉल्व होने से पहले कोई स्कोप वाला विनिर्देश विज्ञापित नहीं होता।
- साझा-थ्रेड हार्नेस पर अप्रमाणित अनुरोधकर्ताओं को विज्ञापित
  स्कोप वाले टूल फिर भी दिखाई देते हैं; किसी टूल को कॉल करने पर उस अनुरोधकर्ता के लिए एक स्पष्ट कनेक्ट-नहीं
  टूल त्रुटि लौटती है। OpenClaw कभी किसी अन्य अनुरोधकर्ता के क्रेडेंशियल का फ़ॉलबैक उपयोग नहीं करता।

मेमोरी प्रॉम्प्ट पूरक बिल्डर को वैकल्पिक `agentId`,
`agentSessionKey`, और `sandboxed` संदर्भ प्राप्त होते हैं। मेमोरी कॉर्पस पूरक `search`
और `get` कॉल को वैकल्पिक `agentId` और `sandboxed` संदर्भ प्राप्त होते हैं। एजेंट-स्वामित्व
वाले भंडारण के साथ Plugins को पंजीकरण के दौरान एक वैश्विक पथ कैप्चर करने के बजाय
प्रत्येक कॉल के लिए उस भंडारण को रिज़ॉल्व करना चाहिए। यदि बहु-एजेंट संचालन में एजेंट id आवश्यक हो लेकिन
अनुपस्थित हो, तो किसी मनमाने एजेंट को चुनने के बजाय विफलता-बंद रहें।

जब प्रॉम्प्ट टेक्स्ट एसिंक Plugin स्थिति पर निर्भर हो, तो `registerMemoryPromptPreparation(...)` का उपयोग करें।
कॉलबैक प्रत्येक पूर्ण एजेंट प्रॉम्प्ट से पहले एक बार चलता है और
सिंक्रोनस मेमोरी प्रॉम्प्ट बिल्डर के समान टूल, एजेंट, सत्र और सैंडबॉक्स संदर्भ प्राप्त करता है।
स्थायी स्थिति लोड करने से पहले वर्तमान भंडारण-स्वामी इंस्टेंस को सत्यापित करें,
फिर केवल उस रन की पंक्तियाँ लौटाएँ। OpenClaw उन पंक्तियों को स्थिर करता है और
अपरिवर्तनीय परिणाम को सिंक्रोनस प्रॉम्प्ट संयोजन को सौंपता है। स्थायित्व,
परमाणु प्रतिस्थापन और स्वामी हटाए जाने पर विलोपन को स्वामी Plugin के भीतर रखें; प्रॉम्प्ट बिल्डर से
फ़ाइलों की पोलिंग या पठन न करें।

Telegram के सामान्य इनबाउंड एजेंट पथ से रूट करने के लिए `{ submitText }` लौटा सकते हैं। जब इनबाउंड नीति टेक्स्ट को छोड़ देती है या
प्रसंस्करण विफल होता है, तो OpenClaw कॉलबैक बटन बनाए रखता है, ताकि
अवरोधक स्थिति बदलने के बाद उपयोगकर्ता फिर से प्रयास कर सके। यह परिणाम फ़ील्ड
Telegram-विशिष्ट है; अन्य चैनल अपने स्वयं के इंटरैक्टिव परिणाम अनुबंध बनाए रखते हैं।

### वर्कफ़्लो Plugins के लिए होस्ट हुक

होस्ट हुक उन Plugins के लिए SDK सीमाएँ हैं जिन्हें केवल प्रदाता, चैनल या टूल जोड़ने के बजाय
होस्ट जीवनचक्र में भाग लेने की आवश्यकता होती है। ये
सामान्य अनुबंध हैं; Plan Mode उनका उपयोग कर सकता है, लेकिन स्वीकृति वर्कफ़्लो,
वर्कस्पेस नीति गेट, पृष्ठभूमि मॉनिटर, सेटअप विज़ार्ड और UI सहयोगी
Plugins भी उनका उपयोग कर सकते हैं।

| विधि                                                                                 | उसके स्वामित्व वाला अनुबंध                                                                                                                                 |
| ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `api.session.state.registerSessionExtension(...)`                                    | Plugin-स्वामित्व वाली, JSON-संगत सत्र स्थिति, जिसे Gateway सत्रों के माध्यम से प्रक्षेपित किया जाता है                                                     |
| `api.session.workflow.enqueueNextTurnInjection(...)`                                 | एक सत्र के लिए अगले एजेंट टर्न में अंतःक्षेपित टिकाऊ, ठीक-एक-बार संदर्भ                                                                                   |
| `api.registerTrustedToolPolicy(...)`                                                 | मैनिफ़ेस्ट-गेटेड विश्वसनीय प्री-Plugin टूल नीति, जो टूल पैरामीटरों को अवरुद्ध या पुनर्लिखित कर सकती है                                                     |
| `api.registerToolMetadata(...)`                                                      | टूल कार्यान्वयन को बदले बिना टूल कैटलॉग प्रदर्शन मेटाडेटा                                                                                                 |
| `api.registerCommand(...)`                                                           | सीमित-दायरे वाले Plugin कमांड; कमांड परिणाम `continueAgent: true` या `suppressReply: true` सेट कर सकते हैं; Discord नेटिव कमांड `descriptionLocalizations` का समर्थन करते हैं |
| `api.session.controls.registerControlUiDescriptor(...)`                              | सत्र, टूल, रन, सेटिंग्स या टैब सतहों के लिए Control UI योगदान वर्णनकर्ता                                                                                   |
| `api.lifecycle.registerRuntimeLifecycle(...)`                                        | रीसेट/हटाने/रीलोड पथों पर Plugin-स्वामित्व वाले रनटाइम संसाधनों के लिए क्लीनअप कॉलबैक                                                                      |
| `api.agent.events.registerAgentEventSubscription(...)`                               | वर्कफ़्लो स्थिति और मॉनिटर के लिए स्वच्छीकृत इवेंट सदस्यताएँ                                                                                              |
| `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`  | प्रति-रन Plugin स्क्रैच स्थिति, जिसे टर्मिनल रन जीवनचक्र पर साफ़ किया जाता है                                                                              |
| `api.session.workflow.registerSessionSchedulerJob(...)`                              | Plugin-स्वामित्व वाली शेड्यूलर जॉब के लिए क्लीनअप मेटाडेटा; यह कार्य शेड्यूल नहीं करता या कार्य रिकॉर्ड नहीं बनाता                                         |
| `api.session.workflow.sendSessionAttachment(...)`                                    | केवल-बंडल होस्ट-मध्यस्थ फ़ाइल अटैचमेंट डिलीवरी, सक्रिय सीधे-आउटबाउंड सत्र रूट पर                                                                           |
| `api.session.workflow.scheduleSessionTurn(...)` / `unscheduleSessionTurnsByTag(...)` | केवल-बंडल Cron-समर्थित शेड्यूल किए गए सत्र टर्न और टैग-आधारित क्लीनअप                                                                                      |
| `api.session.controls.registerSessionAction(...)`                                    | टाइपयुक्त सत्र क्रियाएँ, जिन्हें क्लाइंट Gateway के माध्यम से डिस्पैच कर सकते हैं                                                                          |

एक `surface: "tab"` वर्णनकर्ता Control UI में साइडबार टैब जोड़ता है। सक्रिय
plugins के टैब वर्णनकर्ता gateway
hello (`controlUiTabs`) में डैशबोर्ड क्लाइंट को विज्ञापित किए जाते हैं, इसलिए टैब केवल Plugin के सक्षम रहने पर दिखाई देता है।
बंडल किए गए plugins अपने टैब के लिए प्रथम-श्रेणी डैशबोर्ड दृश्य प्रदान कर सकते हैं; अन्य
plugins `path` को किसी Plugin HTTP रूट पर सेट कर सकते हैं (देखें
`api.registerHttpRoute(...)`), जिसे डैशबोर्ड सैंडबॉक्स किए गए फ़्रेम में रेंडर करता है।
`icon` डैशबोर्ड आइकन नाम का संकेत है, `group` साइडबार अनुभाग चुनता है
(`control` या `agent`), `order` Plugin टैब के बीच क्रम निर्धारित करता है, और `requiredScopes`
उन कनेक्शनों से टैब छिपाता है जिनके पास वे ऑपरेटर स्कोप नहीं हैं:

Gateway-संरक्षित बाहरी टैब के लिए, वर्णनकर्ता `path` को उसी
Plugin के `auth: "gateway"` HTTP रूट के अंतर्गत पंजीकृत करें। प्रमाणित बूटस्ट्रैप के बाद, ब्राउज़र को
उस Plugin और रूट मूल तक सीमित अल्पकालिक, HttpOnly अनुमति मिलती है, ताकि
सैंडबॉक्स किया गया फ़्रेम Gateway बियरर टोकन को उसके URL
या JavaScript में कॉपी किए बिना लोड हो सके। बाहरी टैब सक्रिय रहने पर और नेविगेशन या ब्राउज़र पुनः आरंभ होने के बाद उसे माउंट करने से पहले प्रमाणित पैरेंट अनुमति का नवीनीकरण करता है। वह
माउंट करने से पहले उसी अपारदर्शी सैंडबॉक्स से अनुमति की जाँच भी करता है, ताकि कुकी को
अवरुद्ध करने वाले ब्राउज़र गोपनीयता मोड अनुपलब्ध पैनल के साथ सुरक्षित रूप से विफल हों।
फ़्रेम अनुमति केवल `GET` और `HEAD` स्वीकार करती है और हमेशा
`operator.read` रखती है; `requiredScopes` टैब की दृश्यता नियंत्रित करता है, लेकिन
कुकी अनुमति का दायरा कभी नहीं बढ़ाता। म्यूटेशन स्पष्ट Gateway-प्रमाणित पैरेंट या
बियरर सतहों पर बने रहते हैं। बाहरी टैब के लिए HTTPS/Tailscale Serve या
ब्राउज़र-विश्वसनीय लूपबैक ओरिजिन आवश्यक है; LAN होस्ट पर सादा HTTP ऐसे पैनल को माउंट करने के बजाय
सुरक्षित-संदर्भ त्रुटि दिखाता है जो प्रमाणित नहीं हो सकता।
तृतीय-पक्ष कुकी को पूर्णतः अवरुद्ध करने पर भी Gateway-संरक्षित टैब अनुपलब्ध हो जाते हैं।
सभी नेटिव Plugin सतहों की तरह, फ़्रेम स्थापित
Plugin की विश्वास सीमा के भीतर रहता है; OpenClaw स्थापित plugins को परस्पर
पृथक ब्राउज़र सुरक्षा प्रिंसिपल नहीं मानता।
कुकी अनुमतियाँ ब्राउज़र की होस्टनाम सीमा का उपयोग करती हैं, पोर्ट सीमा का नहीं। परस्पर
अविश्वसनीय सेवाओं को Gateway होस्टनाम पर सह-होस्ट न करें, भले ही वे अन्य
पोर्ट पर हों।
Plugin-प्रबंधित प्रमाणीकरण द्वारा समर्थित टैब अपना सीधा iframe व्यवहार बनाए रखते हैं और
इस Gateway अनुमति का अनुरोध या इसकी आवश्यकता नहीं रखते।

```typescript
api.session.controls.registerControlUiDescriptor({
  surface: "tab",
  id: "logbook",
  label: "दैनंदिनी",
  description: "स्क्रीन स्नैपशॉट से निर्मित, समयरेखा के रूप में आपका दिन।",
  icon: "sun",
  group: "control",
  requiredScopes: ["operator.write"],
});
```

- `api.session.state.registerSessionExtension(...)`
- `api.session.workflow.enqueueNextTurnInjection(...)`
- `api.session.workflow.registerSessionSchedulerJob(...)`
- `api.session.workflow.sendSessionAttachment(...)`
- `api.session.workflow.scheduleSessionTurn(...)`
- `api.session.workflow.unscheduleSessionTurnsByTag(...)`
- `api.session.controls.registerSessionAction(...)`
- `api.session.controls.registerControlUiDescriptor(...)`

समतुल्य फ़्लैट विधियाँ मौजूदा plugins के लिए बहिष्कृत संगतता
उपनामों के रूप में उपलब्ध रहती हैं। ऐसा नया Plugin कोड न जोड़ें जो सीधे
`api.registerSessionExtension`, `api.enqueueNextTurnInjection`,
`api.registerControlUiDescriptor`, `api.registerRuntimeLifecycle`,
`api.registerAgentEventSubscription`, `api.emitAgentEvent`,
`api.setRunContext`, `api.getRunContext`, `api.clearRunContext`,
`api.registerSessionSchedulerJob`, `api.registerSessionAction`,
`api.sendSessionAttachment`, `api.scheduleSessionTurn`, या
`api.unscheduleSessionTurnsByTag` को कॉल करे।

`scheduleSessionTurn(...)` Gateway
Cron शेड्यूलर पर सत्र-सीमित सुविधा है। Cron समय-निर्धारण का स्वामी है और टर्न चलने पर
पृष्ठभूमि कार्य रिकॉर्ड बनाता है; Plugin SDK केवल लक्ष्य सत्र, Plugin-स्वामित्व वाले
नामकरण और क्लीनअप को सीमित करता है। जब कार्य को स्वयं टिकाऊ बहु-चरणीय Task Flow स्थिति चाहिए,
तब शेड्यूल किए गए टर्न के भीतर `api.runtime.tasks.managedFlows` का उपयोग करें।

अनुबंध जानबूझकर प्राधिकार को विभाजित करते हैं:

- बाहरी plugins सत्र एक्सटेंशन, UI वर्णनकर्ता, कमांड, टूल
  मेटाडेटा, अगले-टर्न के अंतःक्षेप और सामान्य हुक के स्वामी हो सकते हैं।
- विश्वसनीय टूल नीतियाँ सामान्य `before_tool_call` हुक से पहले चलती हैं और
  होस्ट-विश्वसनीय होती हैं। बंडल की गई नीतियाँ पहले चलती हैं; स्थापित-Plugin नीतियों के लिए
  स्पष्ट सक्षमता और
  `contracts.trustedToolPolicies` में उनके स्थानीय आईडी आवश्यक हैं, और वे Plugin-लोड क्रम में अगली चलती हैं। नीति आईडी
  पंजीकरण करने वाले Plugin तक सीमित होते हैं।
- आरक्षित कमांड स्वामित्व केवल-बंडल है। बाहरी plugins को अपने
  कमांड नामों या उपनामों का उपयोग करना चाहिए।
- `allowPromptInjection=false` प्रॉम्प्ट बदलने वाले हुक अक्षम करता है, जिनमें
  `agent_turn_prepare`, `before_prompt_build`, `heartbeat_prompt_contribution`,
  और `enqueueNextTurnInjection` शामिल हैं।

### Gateway खोज पंजीकरण

`api.registerGatewayDiscoveryService(...)` किसी Plugin को सक्रिय
Gateway का विज्ञापन mDNS/Bonjour जैसे स्थानीय खोज ट्रांसपोर्ट पर करने देता है। स्थानीय खोज सक्षम होने पर OpenClaw
Gateway स्टार्टअप के दौरान सेवा को कॉल करता है, वर्तमान Gateway पोर्ट और गैर-गोपनीय TXT संकेत डेटा
पास करता है, और Gateway शटडाउन के दौरान लौटाए गए
`stop` हैंडलर को कॉल करता है।

```typescript
api.registerGatewayDiscoveryService({
  id: "my-discovery",
  async advertise(ctx) {
    const handle = await startMyAdvertiser({
      gatewayPort: ctx.gatewayPort,
      tls: ctx.gatewayTlsEnabled,
      displayName: ctx.machineDisplayName,
    });
    return { stop: () => handle.stop() };
  },
});
```

Gateway खोज plugins को विज्ञापित TXT मानों को गोपनीय जानकारी या
प्रमाणीकरण नहीं मानना चाहिए। खोज एक रूटिंग संकेत है; विश्वास का स्वामित्व अब भी Gateway प्रमाणीकरण और TLS पिनिंग के पास है।

### CLI पंजीकरण मेटाडेटा

`api.registerCli(registrar, opts?)` दो प्रकार के कमांड मेटाडेटा स्वीकार करता है:

- `commands`: पंजीकरणकर्ता के स्वामित्व वाले स्पष्ट कमांड नाम
- `descriptors`: CLI सहायता,
  रूटिंग और लेज़ी Plugin CLI पंजीकरण के लिए उपयोग किए जाने वाले पार्स-टाइम कमांड वर्णनकर्ता
- `parentPath`: नेस्टेड कमांड समूहों के लिए वैकल्पिक पैरेंट कमांड पथ, जैसे
  `["nodes"]`

युग्मित-Node सुविधाओं के लिए,
`api.registerNodeCliFeature(registrar, opts?)` को प्राथमिकता दें। यह
`api.registerCli(..., { parentPath: ["nodes"] })` के चारों ओर एक छोटा रैपर है और
`openclaw nodes canvas` जैसे कमांड को स्पष्ट रूप से Plugin-स्वामित्व वाली Node सुविधाएँ बनाता है।

यदि आप चाहते हैं कि कोई Plugin कमांड सामान्य रूट CLI पथ में लेज़ी-लोडेड बना रहे,
तो ऐसे `descriptors` प्रदान करें जो उस
पंजीकरणकर्ता द्वारा उजागर किए गए प्रत्येक शीर्ष-स्तरीय कमांड रूट को कवर करें।

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerMatrixCli } = await import("./src/cli.js");
    registerMatrixCli({ program });
  },
  {
    descriptors: [
      {
        name: "matrix",
        description: "Manage Matrix accounts, verification, devices, and profile state",
        hasSubcommands: true,
      },
    ],
  },
);
```

नेस्टेड कमांड को समाधान किया गया पैरेंट कमांड `program` के रूप में मिलता है:

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerNodesCanvasCommands } = await import("./src/cli.js");
    registerNodesCanvasCommands(program);
  },
  {
    parentPath: ["nodes"],
    descriptors: [
      {
        name: "canvas",
        description: "Capture or render canvas content from a paired node",
        hasSubcommands: true,
      },
    ],
  },
);
```

`commands` का अकेले उपयोग केवल तब करें, जब आपको लेज़ी रूट CLI पंजीकरण की आवश्यकता न हो।
वह तत्पर संगतता पथ अब भी समर्थित है, लेकिन वह पार्स-समय लेज़ी लोडिंग के लिए
डिस्क्रिप्टर-समर्थित प्लेसहोल्डर इंस्टॉल नहीं करता।

### CLI बैकएंड पंजीकरण

`api.registerCliBackend(...)` किसी Plugin को `claude-cli` या `my-cli` जैसे स्थानीय
AI CLI बैकएंड के लिए डिफ़ॉल्ट कॉन्फ़िगरेशन का स्वामित्व लेने देता है।

- बैकएंड `id`, `my-cli/gpt-5` जैसे मॉडल रेफ़रेंस में प्रोवाइडर प्रीफ़िक्स बन जाता है।
- बैकएंड `config`, `agents.defaults.cliBackends.<id>` जैसी ही संरचना का उपयोग करता है।
- उपयोगकर्ता कॉन्फ़िगरेशन को अब भी प्राथमिकता मिलती है। CLI चलाने से पहले OpenClaw
  Plugin के डिफ़ॉल्ट पर `agents.defaults.cliBackends.<id>` को मर्ज करता है।
- जब किसी बैकएंड को मर्ज के बाद संगतता पुनर्लेखन की आवश्यकता हो
  (उदाहरण के लिए पुराने फ़्लैग की संरचनाओं को सामान्य बनाना), तब `normalizeConfig` का उपयोग करें।
- CLI बोली से संबंधित अनुरोध-स्कोप वाले argv पुनर्लेखनों के लिए `resolveExecutionArgs` का उपयोग करें,
  जैसे OpenClaw के विचार स्तरों को किसी नेटिव प्रयास फ़्लैग से मैप करना।
  हुक को `ctx.executionMode` मिलता है; क्षणिक `/btw` कॉल के लिए
  बैकएंड-नेटिव आइसोलेशन फ़्लैग जोड़ने हेतु `"side-question"` का उपयोग करें। यदि वे फ़्लैग
  अन्यथा हमेशा चालू रहने वाले CLI के नेटिव टूल विश्वसनीय रूप से अक्षम करते हैं, तो
  `sideQuestionToolMode: "disabled"` भी घोषित करें।
- बैकएंड के स्वामित्व वाले लॉन्च एनवायरनमेंट या अस्थायी
  प्रमाणीकरण/कॉन्फ़िगरेशन ब्रिज के लिए `prepareExecution` का उपयोग करें। इसका `ctx.contextTokenBudget`
  रन के लिए चुनी गई प्रभावी टोकन सीमा है, ताकि नेटिव-Compaction बैकएंड
  प्रोवाइडर-विशिष्ट कोर शाखाओं के बिना अपनी सीमा संरेखित कर सकें।
- जो बैकएंड किसी विशिष्ट रन के लिए सभी नेटिव टूल अक्षम कर सकते हैं, वे
  `nativeToolMode: "selectable"` घोषित कर सकते हैं। प्रतिबंधित कॉल एक खाली
  `ctx.toolAvailability.native` ट्यूपल और सटीक होस्ट-आइसोलेटेड MCP अनुमति-सूची पास करते हैं;
  `resolveExecutionArgs` को अंतिम नए या फिर से शुरू किए गए argv पर दोनों लागू करने होंगे।
  यदि बैकएंड ऐसा नहीं कर सकता, तो OpenClaw सुरक्षित रूप से विफल हो जाता है।

आरंभ से अंत तक की लेखन मार्गदर्शिका के लिए,
[CLI बैकएंड Plugin](/hi/plugins/cli-backend-plugins) देखें।

### अप्रचलित मेमोरी एम्बेडिंग अडैप्टर

| विधि                                         | यह क्या पंजीकृत करती है                              |

  किसी केंद्रित सार्वजनिक उपभोक्ता API के उपलब्ध होने तक, बनाए रखे गए
  `openclaw/plugin-sdk/memory-host-core` फ़साड से `listActiveMemoryPublicArtifacts(...)` का उपयोग जारी रखते हैं;
  उन्हें किसी अन्य Plugin की निजी संरचना तक नहीं पहुँचना चाहिए।
- `MemoryFlushPlan.model`, सक्रिय फ़ॉलबैक शृंखला को विरासत में लिए बिना,
  फ़्लश टर्न को `ollama/qwen3:8b` जैसे सटीक `provider/model` रेफ़रेंस पर पिन कर सकता है।
- `registerMemoryEmbeddingProvider` अप्रचलित है। नए एम्बेडिंग प्रोवाइडर को
  `api.registerEmbeddingProvider(...)` और
  `contracts.embeddingProviders` का उपयोग करना चाहिए।
- माइग्रेशन अवधि के दौरान मौजूदा मेमोरी-विशिष्ट प्रोवाइडर काम करना जारी रखते हैं,
  लेकिन Plugin निरीक्षण गैर-बंडल Plugin के लिए इसे संगतता ऋण के रूप में रिपोर्ट करता है।

### हुक निर्णय सिमेंटिक्स

`before_install` एक Plugin-रनटाइम लाइफ़साइकल हुक है, ऑपरेटर इंस्टॉल
नीति सतह नहीं। जब अनुमति/ब्लॉक निर्णय में CLI और Gateway-समर्थित इंस्टॉल
या अपडेट पथ शामिल होने चाहिए, तब `security.installPolicy` का उपयोग करें।

- `before_tool_call`: `{ block: true }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा इसे सेट करने के बाद कम-प्राथमिकता वाले हैंडलर छोड़ दिए जाते हैं।
- `before_tool_call`: `{ block: false }` लौटाना कोई निर्णय न होने के रूप में माना जाता है (`block` को छोड़ने के समान), ओवरराइड के रूप में नहीं।
- `before_install`: `{ block: true }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा इसे सेट करने के बाद कम-प्राथमिकता वाले हैंडलर छोड़ दिए जाते हैं।
- `before_install`: `{ block: false }` लौटाना कोई निर्णय न होने के रूप में माना जाता है (`block` को छोड़ने के समान), ओवरराइड के रूप में नहीं।
- `reply_dispatch`: `{ handled: true, ... }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा डिस्पैच का दावा करने के बाद कम-प्राथमिकता वाले हैंडलर और डिफ़ॉल्ट मॉडल डिस्पैच पथ छोड़ दिए जाते हैं।
- `message_sending`: `{ cancel: true }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा इसे सेट करने के बाद कम-प्राथमिकता वाले हैंडलर छोड़ दिए जाते हैं।
- `message_sending`: `{ cancel: false }` लौटाना कोई निर्णय न होने के रूप में माना जाता है (`cancel` को छोड़ने के समान), ओवरराइड के रूप में नहीं।
- `message_received`: जब आपको आने वाली थ्रेड/टॉपिक रूटिंग की आवश्यकता हो, तब टाइप किए हुए `threadId` फ़ील्ड का उपयोग करें। चैनल-विशिष्ट अतिरिक्त जानकारी के लिए `metadata` बनाए रखें।
- `message_sending`: चैनल-विशिष्ट `metadata` पर वापस जाने से पहले टाइप किए हुए `replyToId` / `threadId` रूटिंग फ़ील्ड का उपयोग करें।
- `gateway_start`: आंतरिक `gateway:startup` हुक पर निर्भर होने के बजाय Gateway के स्वामित्व वाली स्टार्टअप स्थिति के लिए `ctx.config`, `ctx.workspaceDir`, और `ctx.getCron?.()` का उपयोग करें। इस समय Cron अब भी लोड हो रहा हो सकता है।
- `cron_reconciled`: स्टार्टअप या शेड्यूलर पुनः लोड होने के बाद पूर्ण बाहरी Cron प्रोजेक्शन फिर से बनाएँ। इसमें `reason` और `enabled: false` सहित प्रभावी `enabled` स्थिति शामिल होती है, जबकि `ctx.getCron?.()` सटीक समाधान किया गया शेड्यूलर लौटाता है। टिकाऊ प्रोजेक्शन कार्य में `ctx.abortSignal` पास करें; वह शेड्यूलर स्नैपशॉट के प्रतिस्थापित होने या Gateway के बंद होने पर निरस्त हो जाता है।
- `cron_changed`: Gateway के स्वामित्व वाले Cron लाइफ़साइकल परिवर्तनों का अवलोकन करें। `scheduled` और `removed` इवेंट कमिट के बाद के समाधान संकेत हैं, क्रमबद्ध डेल्टा लॉग नहीं। यदि जॉब के लिए अगला जागरण नहीं है, तो शेड्यूल किए गए इवेंट का `event.nextRunAtMs` अनुपस्थित होता है; हटाया गया इवेंट अब भी मिटाए गए जॉब का स्नैपशॉट रखता है।

बाहरी वेक शेड्यूलर को `cron_changed` इवेंट को डीबाउंस या समेकित करना चाहिए,
फिर `cron_reconciled` द्वारा अंतिम बार कैप्चर किए गए शेड्यूलर से पूर्ण टिकाऊ दृश्य दोबारा पढ़ना चाहिए।
`cron_changed` कॉन्टेक्स्ट से शेड्यूलर न अपनाएँ: किसी पुराने शेड्यूलर का
अलग किया गया संकेत बाद के पुनः लोड से ओवरलैप कर सकता है।

Gateway स्टार्टअप या शेड्यूलर प्रतिस्थापन पर लोड की गई टिकाऊ स्थिति के लिए
पूर्ण-स्नैपशॉट ट्रिगर के रूप में `cron_reconciled` का उपयोग करें। इसे केवल Plugin के
हॉट रीलोड के लिए फिर से नहीं चलाया जाता। अवलोकन हैंडलर समानांतर चलते हैं और फ़ायर-एंड-फ़ॉरगेट
डिस्पैच ओवरलैप कर सकते हैं, इसलिए उपभोक्ताओं को इवेंट पूर्ण होने के क्रम पर निर्भर नहीं होना चाहिए।
नियत समय की जाँच और निष्पादन के लिए OpenClaw को सत्य का स्रोत बनाए रखें।

टिकाऊ प्रतिस्थापन, पुनः प्रयास/बैकऑफ़ और सुव्यवस्थित शटडाउन वाले सिंगल-फ़्लाइट अडैप्टर के लिए,
[सुरक्षित बाहरी Cron प्रोजेक्शन](/hi/plugins/hooks#safe-external-cron-projection) देखें।

| `api.logger`             | `PluginLogger`            | स्कोप किया हुआ लॉगर (`debug`, `info`, `warn`, `error`)                                            |
| `api.registrationMode`   | `PluginRegistrationMode`  | वर्तमान लोड मोड; `"setup-runtime"` पूर्ण एंट्री से पहले की हल्की स्टार्टअप/सेटअप अवधि है |
| `api.resolvePath(input)` | `(string) => string`      | Plugin रूट के सापेक्ष पथ समाधान करें                                                        |

## आंतरिक मॉड्यूल परिपाटी

अपने Plugin में आंतरिक इंपोर्ट के लिए स्थानीय बैरल फ़ाइलों का उपयोग करें:

```text
my-plugin/
  api.ts            # Public exports for external consumers
  runtime-api.ts    # Internal-only runtime exports
  index.ts          # Plugin entry point
  setup-entry.ts    # Lightweight setup-only entry (optional)
```

<Warning>
  प्रोडक्शन कोड से अपने Plugin को कभी भी `openclaw/plugin-sdk/<your-plugin>`
  के माध्यम से इंपोर्ट न करें। आंतरिक इंपोर्ट को `./api.ts` या
  `./runtime-api.ts` के माध्यम से रूट करें। SDK पथ केवल बाहरी अनुबंध है।
</Warning>

Facade से लोड किए गए बंडल्ड Plugin के सार्वजनिक सरफ़ेस (`api.ts`, `runtime-api.ts`,
`index.ts`, `setup-entry.ts`, और इसी तरह की सार्वजनिक एंट्री फ़ाइलें) उस समय
सक्रिय रनटाइम कॉन्फ़िग स्नैपशॉट को प्राथमिकता देते हैं, जब OpenClaw पहले से चल रहा हो। यदि अभी तक कोई रनटाइम
स्नैपशॉट मौजूद नहीं है, तो वे डिस्क पर मौजूद रिज़ॉल्व की गई कॉन्फ़िग फ़ाइल का उपयोग करते हैं।
पैकेज किए गए बंडल्ड Plugin Facade को OpenClaw के Plugin
Facade लोडर के माध्यम से लोड किया जाना चाहिए; `dist/extensions/...` से सीधे इंपोर्ट करने पर मेनिफ़ेस्ट
और रनटाइम साइडकार जाँच बायपास हो जाती हैं, जिनका उपयोग पैकेज किए गए इंस्टॉल Plugin के स्वामित्व वाले कोड के लिए करते हैं।

प्रोवाइडर Plugin एक सीमित Plugin-स्थानीय कॉन्ट्रैक्ट बैरल उपलब्ध करा सकते हैं, जब कोई
हेल्पर जानबूझकर किसी विशिष्ट प्रोवाइडर के लिए हो और अभी किसी सामान्य SDK
सबपाथ में उपयुक्त न हो। बंडल्ड उदाहरण:

- **Anthropic**: Claude
  बीटा-हेडर और `service_tier` स्ट्रीम हेल्पर के लिए सार्वजनिक `api.ts` / `contract-api.ts` सीम।
- **`@openclaw/openai-provider`**: `api.ts` प्रोवाइडर बिल्डर,
  डिफ़ॉल्ट-मॉडल हेल्पर और रीयलटाइम प्रोवाइडर बिल्डर एक्सपोर्ट करता है।
- **`@openclaw/openrouter-provider`**: `api.ts` प्रोवाइडर बिल्डर
  के साथ ऑनबोर्डिंग/कॉन्फ़िग हेल्पर एक्सपोर्ट करता है।

<Warning>
  एक्सटेंशन के प्रोडक्शन कोड को `openclaw/plugin-sdk/<other-plugin>`
  इंपोर्ट से भी बचना चाहिए। यदि कोई हेल्पर वास्तव में साझा है, तो दो Plugin को आपस में जोड़ने के बजाय उसे
  `openclaw/plugin-sdk/speech`, `.../provider-model-shared`, या किसी अन्य
  क्षमता-उन्मुख सरफ़ेस जैसे तटस्थ SDK सबपाथ में स्थानांतरित करें।
</Warning>
