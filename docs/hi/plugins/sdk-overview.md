Plugin SDK, plugins और core के बीच टाइप किया हुआ अनुबंध है। यह पृष्ठ
**क्या इंपोर्ट करना है** और **क्या पंजीकृत किया जा सकता है**, इसका संदर्भ है।

<Note>
  यह पृष्ठ OpenClaw के भीतर `openclaw/plugin-sdk/*` का उपयोग करने वाले Plugin
  लेखकों के लिए है। Gateway के माध्यम से एजेंट चलाने की इच्छा रखने वाले बाहरी ऐप्स,
  स्क्रिप्ट, डैशबोर्ड, CI जॉब और IDE एक्सटेंशन इसके बजाय
  [बाहरी ऐप्स के लिए Gateway एकीकरण](/hi/gateway/external-apps) का उपयोग करें।
</Note>

<Tip>
इसके बजाय कोई कार्यविधि मार्गदर्शिका खोज रहे हैं? [Plugins बनाना](/hi/plugins/building-plugins) से शुरू करें। चैनलों के लिए [चैनल plugins](/hi/plugins/sdk-channel-plugins), मॉडल प्रदाताओं के लिए [प्रदाता plugins](/hi/plugins/sdk-provider-plugins), स्थानीय AI CLI बैकएंड के लिए [CLI बैकएंड plugins](/hi/plugins/cli-backend-plugins), नेटिव एजेंट निष्पादकों के लिए [एजेंट हार्नेस plugins](/hi/plugins/sdk-agent-harness), और टूल या जीवनचक्र हुक के लिए [Plugin हुक](/hi/plugins/hooks) का उपयोग करें।
</Tip>

## इंपोर्ट परिपाटी

हमेशा किसी विशिष्ट सबपाथ से इंपोर्ट करें:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { defineChannelPluginEntry } from "openclaw/plugin-sdk/channel-core";
```

प्रत्येक सबपाथ एक छोटा, स्व-निहित मॉड्यूल है। इससे स्टार्टअप तेज़ रहता है और
चक्रीय निर्भरता संबंधी समस्याएँ रुकती हैं। चैनल-विशिष्ट एंट्री/बिल्ड सहायकों के लिए,
`openclaw/plugin-sdk/channel-core` को प्राथमिकता दें; व्यापक समग्र सतह और
`buildChannelConfigSchema` जैसे साझा सहायकों के लिए
`openclaw/plugin-sdk/core` रखें।

चैनल कॉन्फ़िगरेशन के लिए, चैनल के स्वामित्व वाला JSON Schema
`openclaw.plugin.json#channelConfigs` के माध्यम से प्रकाशित करें। `plugin-sdk/channel-config-schema`
सबपाथ साझा स्कीमा प्रिमिटिव और जेनेरिक बिल्डर के लिए है। OpenClaw के
बंडल किए गए plugins, बनाए रखे गए बंडल-चैनल स्कीमा के लिए
`plugin-sdk/bundled-channel-config-schema` का उपयोग करते हैं। बहिष्कृत संगतता एक्सपोर्ट
`plugin-sdk/channel-config-schema-legacy` पर बने रहते हैं; दोनों में से कोई भी बंडल स्कीमा
सबपाथ नए plugins के लिए प्रतिमान नहीं है।

<Warning>
  प्रदाता या चैनल ब्रांड वाले सुविधाजनक सीम इंपोर्ट न करें (उदाहरण के लिए
  `openclaw/plugin-sdk/slack`, `.../discord`, `.../signal`, `.../whatsapp`)।
  बंडल किए गए plugins अपने स्वयं के `api.ts` /
  `runtime-api.ts` बैरल के भीतर जेनेरिक SDK सबपाथ संयोजित करते हैं; core
  उपभोक्ताओं को या तो उन Plugin-स्थानीय बैरल का उपयोग करना चाहिए या आवश्यकता
  वास्तव में क्रॉस-चैनल होने पर एक संकीर्ण जेनेरिक SDK अनुबंध जोड़ना चाहिए।

बंडल किए गए Plugin के सहायक सीम का एक छोटा समूह अभी भी जनरेट किए गए एक्सपोर्ट
मैप में दिखाई देता है, जब उनके स्वामी का उपयोग ट्रैक किया जाता है। वे केवल
बंडल किए गए Plugin के रखरखाव के लिए मौजूद हैं और नए तृतीय-पक्ष plugins के लिए
अनुशंसित इंपोर्ट पाथ नहीं हैं।

`openclaw/plugin-sdk/discord` और `openclaw/plugin-sdk/telegram-account` को भी ट्रैक किए गए स्वामी उपयोग के
लिए बहिष्कृत संगतता फ़साड के रूप में रखा गया है। उन इंपोर्ट पाथ को नए plugins में
कॉपी न करें; इसके बजाय इंजेक्ट किए गए रनटाइम सहायक और जेनेरिक चैनल SDK सबपाथ
का उपयोग करें।
</Warning>

## सबपाथ संदर्भ

Plugin SDK को क्षेत्र के अनुसार समूहीकृत संकीर्ण सबपाथ के एक समूह के रूप में
प्रस्तुत किया जाता है (Plugin एंट्री, चैनल, प्रदाता, प्रमाणीकरण, रनटाइम, क्षमता,
मेमोरी और आरक्षित बंडल-Plugin सहायक)। समूहबद्ध और लिंक किए गए पूर्ण कैटलॉग के
लिए [Plugin SDK सबपाथ](/hi/plugins/sdk-subpaths) देखें।

कंपाइलर एंट्रीपॉइंट इन्वेंटरी
`scripts/lib/plugin-sdk-entrypoints.json` में रहती है; पैकेज एक्सपोर्ट, `scripts/lib/plugin-sdk-private-local-only-subpaths.json` में
सूचीबद्ध रेपो-स्थानीय परीक्षण/आंतरिक सबपाथ को घटाने के बाद सार्वजनिक उपसमूह से
जनरेट किए जाते हैं। सार्वजनिक एक्सपोर्ट संख्या का ऑडिट करने के लिए
`pnpm plugin-sdk:surface` चलाएँ। पर्याप्त पुराने और बंडल किए गए एक्सटेंशन के
प्रोडक्शन कोड द्वारा अप्रयुक्त बहिष्कृत सार्वजनिक सबपाथ को
`scripts/lib/plugin-sdk-deprecated-public-subpaths.json` में ट्रैक किया जाता है; व्यापक बहिष्कृत री-एक्सपोर्ट बैरल
`scripts/lib/plugin-sdk-deprecated-barrel-subpaths.json` में ट्रैक किए जाते हैं।

## पंजीकरण API

`register(api)` कॉलबैक को इन विधियों वाला एक `OpenClawPluginApi` ऑब्जेक्ट
प्राप्त होता है:

### क्षमता पंजीकरण

| विधि                                             | यह क्या पंजीकृत करती है                                                          |
| ------------------------------------------------ | --------------------------------------------------------------------------------- |
| `api.registerProvider(...)`                      | टेक्स्ट अनुमान (LLM)                                                              |
| `api.registerWorkerProvider(...)`                | क्लाउड-वर्कर जीवनचक्र लीज़                                                        |
| `api.registerModelCatalogProvider(...)`          | टेक्स्ट और मीडिया जनरेशन के लिए मॉडल कैटलॉग पंक्तियाँ                            |
| `api.registerAgentHarness(...)`                  | [प्रयोगात्मक](/hi/plugins/sdk-agent-harness) नेटिव एजेंट निष्पादक (Codex, Copilot) |
| `api.registerCliBackend(...)`                    | स्थानीय CLI अनुमान बैकएंड                                                        |
| `api.registerChannel(...)`                       | मैसेजिंग चैनल                                                                     |
| `api.registerEmbeddingProvider(...)`             | पुनः उपयोग योग्य वेक्टर एम्बेडिंग प्रदाता                                        |
| `api.registerSpeechProvider(...)`                | टेक्स्ट-टू-स्पीच / STT संश्लेषण                                                   |
| `api.registerRealtimeTranscriptionProvider(...)` | स्ट्रीमिंग रीयलटाइम ट्रांसक्रिप्शन                                                |
| `api.registerRealtimeVoiceProvider(...)`         | डुप्लेक्स रीयलटाइम वॉइस सत्र                                                      |
| `api.registerMediaUnderstandingProvider(...)`    | इमेज/ऑडियो/वीडियो विश्लेषण                                                       |
| `api.registerTranscriptSourceProvider(...)`      | लाइव या इंपोर्ट किया गया मीटिंग ट्रांसक्रिप्ट स्रोत                               |
| `api.registerImageGenerationProvider(...)`       | इमेज जनरेशन                                                                       |
| `api.registerMusicGenerationProvider(...)`       | संगीत जनरेशन                                                                      |
| `api.registerVideoGenerationProvider(...)`       | वीडियो जनरेशन                                                                     |
| `api.registerWebFetchProvider(...)`              | वेब फ़ेच / स्क्रैप प्रदाता                                                        |
| `api.registerWebSearchProvider(...)`             | वेब खोज                                                                           |
| `api.registerCompactionProvider(...)`            | प्लग करने योग्य ट्रांसक्रिप्ट-Compaction बैकएंड                                   |

वर्कर प्रदाताओं को `contracts.workerProviders` में अपनी ID भी घोषित करनी होगी।
core, `provision(profile, operationId)` से पहले स्थायी आशय को सहेजता है। प्रदाता बाहरी आवंटन
से पहले सेटिंग्स सत्यापित करते हैं और स्थायी प्रोफ़ाइल अस्वीकृति के लिए
`WorkerProviderError` थ्रो करते हैं। ऑपरेशन ID दोहराए जाने पर
`provision` को उसी लीज़ को अपनाना होगा।
core सत्यापित प्रोफ़ाइल सेटिंग्स को लीज़ के साथ सहेजता है और वह स्नैपशॉट
`destroy({ leaseId, profile })`, जिसे आइडेम्पोटेंट होना चाहिए, तथा `inspect({ leaseId, profile })` को
देता है, जो `active`, `destroyed` या
`unknown` लौटाता है। इससे प्रदाता Gateway के पुनः आरंभ होने या
नामित प्रोफ़ाइल हटाए जाने के बाद जीवनचक्र कॉल को रूट कर सकते हैं। SSH एंडपॉइंट
`keyRef` के लिए `SecretRef` का उपयोग करते हैं, कभी भी इनलाइन
कुंजी सामग्री का नहीं, और विश्वसनीय प्रोविज़निंग आउटपुट से प्राप्त
`hostKey` को ठीक `algorithm base64` के रूप में शामिल करते हैं, बिना
होस्टनाम या टिप्पणी के। core `hostKey` को पिन करता है और पहले
कनेक्शन से मिली कुंजी पर कभी भरोसा नहीं करता। डायनेमिक `keyRef`
बनाने वाला प्रदाता `resolveSshIdentity({ leaseId, profile, keyRef })` लागू कर सकता है; मौजूद होने पर वह
रिज़ॉल्वर प्रामाणिक होता है, जबकि इसके बिना प्रदाता कॉन्फ़िगर किए गए जेनेरिक
सीक्रेट रिज़ॉल्वर का उपयोग करते हैं।
नवीकरणीय लीज़ वाले प्रदाता `renew(leaseId)` भी लागू कर सकते हैं।
क्षणिक या अनिर्धारित विफलताओं पर `inspect` को थ्रो करना होगा;
`unknown` केवल प्रामाणिक अनुपस्थिति के लिए लौटाएँ। core किसी सक्रिय
स्थानीय रिकॉर्ड को अनाथ के रूप में चिह्नित करता है, या सहेजे गए नष्ट करने के
अनुरोध के बाद अनुपस्थिति को टियरडाउन पूर्ण होना मानता है।

`api.registerEmbeddingProvider(...)` के साथ पंजीकृत एम्बेडिंग प्रदाताओं को Plugin मेनिफ़ेस्ट में
`contracts.embeddingProviders` में भी सूचीबद्ध होना चाहिए। यह पुनः उपयोग योग्य वेक्टर
जनरेशन के लिए जेनेरिक एम्बेडिंग सतह है। मेमोरी खोज इस जेनेरिक प्रदाता सतह का
उपयोग कर सकती है। पुराना `api.registerMemoryEmbeddingProvider(...)` और
`contracts.memoryEmbeddingProviders` सीम, मौजूदा मेमोरी-विशिष्ट प्रदाताओं के माइग्रेट होने तक
बहिष्कृत संगतता है।

अब भी रनटाइम `batchEmbed(...)` प्रस्तुत करने वाले मेमोरी-विशिष्ट प्रदाता
मौजूदा प्रति-फ़ाइल बैचिंग अनुबंध पर बने रहते हैं, जब तक उनका रनटाइम स्पष्ट रूप
से `sourceWideBatchEmbed: true` सेट न करे। यह ऑप्ट-इन मेमोरी होस्ट को एक
`batchEmbed(...)` कॉल में, होस्ट बैच सीमाओं तक, कई डर्टी मेमोरी फ़ाइलों और
सक्षम स्रोतों के चंक सबमिट करने देता है। JSONL अनुरोध फ़ाइलें अपलोड करने वाले
बैच अडैप्टर को प्रदाता जॉब उनकी अपलोड-आकार सीमा के साथ-साथ उनकी अनुरोध-संख्या
सीमा से पहले विभाजित करने होंगे। प्रदाता को `batch.chunks` के समान क्रम
में प्रत्येक इनपुट चंक के लिए एक एम्बेडिंग लौटानी होगी; जब प्रदाता फ़ाइल-स्थानीय
बैच की अपेक्षा करता हो या बड़े स्रोत-व्यापी जॉब में इनपुट क्रम बनाए न रख सकता
हो, तो फ़्लैग छोड़ दें।

### टूल और कमांड

निश्चित टूल नामों वाले सरल, केवल-टूल plugins के लिए
[`defineToolPlugin`](/hi/plugins/tool-plugins) का उपयोग करें। मिश्रित plugins या
पूरी तरह डायनेमिक टूल पंजीकरण के लिए सीधे `api.registerTool(...)` का उपयोग करें।

| विधि                                   | यह क्या पंजीकृत करती है                                                                                                                   |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `api.registerTool(tool, opts?)`        | एजेंट टूल (आवश्यक या `{ optional: true }`)                                                                                              |
| `api.registerCommand(def)`             | कस्टम कमांड (LLM को बायपास करता है)                                                                                                      |
| `api.registerNodeHostCommand(command)` | `openclaw node run` द्वारा संभाला गया कमांड; वैकल्पिक `agentTool` मेटाडेटा, Node कनेक्ट होने के दौरान इसे एजेंट-दृश्यमान टूल के रूप में प्रस्तुत कर सकता है |

जब एजेंट को कमांड के स्वामित्व वाला संक्षिप्त रूटिंग संकेत चाहिए, तब Plugin
कमांड `agentPromptGuidance` सेट कर सकते हैं। उस टेक्स्ट को कमांड के बारे में ही
रखें; core प्रॉम्प्ट बिल्डर में प्रदाता या Plugin-विशिष्ट नीति न जोड़ें।

मार्गदर्शन प्रविष्टियाँ लेगेसी स्ट्रिंग हो सकती हैं, जो प्रत्येक प्रॉम्प्ट सतह
पर लागू होती हैं, या संरचित प्रविष्टियाँ हो सकती हैं:

```ts
agentPromptGuidance: [
  "सार्वभौमिक कमांड संकेत।",
  { text: "इसे केवल मुख्य OpenClaw प्रॉम्प्ट में दिखाएँ।", surfaces: ["openclaw_main"] },
];
```

संरचित `surfaces` में `openclaw_main`, `codex_app_server`,
`cli_backend`, `acp_backend` या `subagent` शामिल हो सकते
हैं। `pi_main`, `openclaw_main` का बहिष्कृत उपनाम बना हुआ है।
सभी सतहों के लिए जानबूझकर दिए गए मार्गदर्शन हेतु `surfaces` छोड़ दें।
खाली `surfaces` ऐरे पास न करें; इसे अस्वीकार कर दिया जाता है ताकि
अनजाने में स्कोप खोने से टेक्स्ट सार्वभौमिक प्रॉम्प्ट टेक्स्ट न बन जाए।

नेटिव Codex ऐप-सर्वर डेवलपर निर्देश अन्य प्रॉम्प्ट सतहों की तुलना में अधिक
कठोर हैं: केवल `codex_app_server` के लिए स्पष्ट रूप से स्कोप किया गया मार्गदर्शन
उस उच्च-प्राथमिकता लेन में पदोन्नत किया जाता है। लेगेसी स्ट्रिंग मार्गदर्शन और
बिना स्कोप वाला संरचित मार्गदर्शन संगतता के लिए गैर-Codex प्रॉम्प्ट सतहों पर
उपलब्ध रहता है।

Node-होस्ट कमांड कनेक्ट किए गए Node होस्ट पर चलते हैं, Gateway प्रक्रिया के
भीतर नहीं। यदि `agentTool` मौजूद है, तो सफल Gateway कनेक्शन के बाद Node
एक डिस्क्रिप्टर प्रकाशित करता है; Gateway इसे एजेंट रन के लिए केवल तब प्रस्तुत
करता है जब वह Node कनेक्ट हो और केवल तभी जब डिस्क्रिप्टर का
`command`, Node की स्वीकृत कमांड सतह में हो। किसी गैर-खतरनाक कमांड को
डिफ़ॉल्ट Node कमांड अनुमति-सूची में ऑप्ट-इन करने के लिए
`agentTool.defaultPlatforms` सेट करें; अन्यथा स्पष्ट `gateway.nodes.allowCommands` या Node-इनवोक
नीति आवश्यक करें। `agentTool.name` प्रदाता-सुरक्षित होना चाहिए: अक्षर से
शुरू हो, केवल अक्षर, अंक, अंडरस्कोर या हाइफ़न का उपयोग करे और 64 वर्णों के
भीतर रहे। MCP-समर्थित Node टूल `agentTool.mcp` मेटाडेटा सेट कर सकते हैं,
ताकि कैटलॉग और टूल-खोज सतहें रिमोट MCP सर्वर/टूल पहचान दिखा सकें, लेकिन
निष्पादन फिर भी विज्ञापित Node कमांड के माध्यम से ही होता है।

#### अभिस्वीकृति के बाद का Webhook कार्य

प्रोसेसिंग पूरी होने से पहले अनुरोध की अभिस्वीकृति देने वाले Webhook रूट्स को
उस अलग किए गए कार्य को उसके अपने ट्रैक किए गए प्रवेश रूट पर ले जाना आवश्यक है:

```typescript
import { runDetachedWebhookWork } from "openclaw/plugin-sdk/webhook-request-guards";

void runDetachedWebhookWork(() => processWebhookEvent(event)).catch((error) => {
  runtime.error?.(`Webhook प्रेषण विफल हुआ: ${String(error)}`);
});
```

HTTP अनुरोध के प्रवेश की स्थिति में रहते हुए `runDetachedWebhookWork(...)` को सिंक्रोनस रूप से
कॉल करें। हेल्पर तुरंत एक स्वतंत्र रूट आरक्षित करता है, फिर अगली माइक्रोटास्क में
कॉलबैक शुरू करता है ताकि अनुरोध हैंडलर पहले अपनी अभिस्वीकृति लिख सके।
लौटाया गया प्रॉमिस कॉलबैक के परिणाम को अपनाता है; अस्वीकृति प्रबंधन की ज़िम्मेदारी
अब भी कॉलर की होती है। इससे अभिस्वीकृति के बाद का क्यू कार्य स्वीकृत रहता है और
रीस्टार्ट या निलंबन के दौरान निकासी उसके लिए प्रतीक्षा करती है। लौटने से पहले सभी
प्रोसेसिंग की प्रतीक्षा करने वाले हैंडलरों को इस हेल्पर की आवश्यकता नहीं है।

#### अनुरोधकर्ता-सीमित MCP कनेक्शन

MCP सर्वर की **पहचान** (नाम, टूल फ़िल्टर) को `mcp.servers` या किसी
बंडल मैनिफ़ेस्ट में स्थिर रखें। वैकल्पिक रूप से एक कनेक्शन रिज़ॉल्वर पंजीकृत करें,
ताकि प्रत्येक विश्वसनीय संदेश अनुरोधकर्ता को अपना अलग ट्रांसपोर्ट मिले:

```ts
api.registerMcpServerConnectionResolver({
  serverName: "user-email",
  resolve: async (ctx) => {
    // ctx.requesterSenderId होस्ट द्वारा विश्वसनीय है; यहाँ प्रेषक की पहचान कभी न गढ़ें।
    const token = await lookupUserToken(ctx.requesterSenderId);
    if (!token) {
      return null; // वर्तमान रन के लिए इस सर्वर को छोड़ दें
    }
    return {
      url: "https://mcp.example.com/email",
      headers: { Authorization: `Bearer ${token}` },
    };
  },
});
```

अनुबंध संबंधी टिप्पणियाँ:

- रिज़ॉल्वर संदर्भ में केवल विश्वसनीय होस्ट पहचान होती है (`requesterSenderId`,
  वैकल्पिक `agentAccountId` / `messageChannel`)। भविष्य के विश्वसनीय फ़ील्ड
  (उदाहरण के लिए cron/subagent उपयोगकर्ता संदर्भ) योगात्मक रूप से जोड़े जा सकते हैं।
- एक सर्वर नाम का स्वामी एक ही Plugin होता है: किसी अन्य
  Plugin से समान `serverName` के लिए डुप्लिकेट
  `registerMcpServerConnectionResolver` को त्रुटि निदान के साथ अस्वीकार कर दिया जाता है
  (पहला पंजीकरण प्रभावी रहता है), इसलिए कनेक्शन स्वामित्व कभी भी Plugin लोड क्रम
  पर निर्भर नहीं करता।
- टूल नाम पूरे घोषित सर्वर सेट से व्युत्पन्न होते हैं, इसलिए आंशिक रिज़ॉल्यूशन
  अनुरोधकर्ताओं या टर्न के बीच सुरक्षित सर्वर नामों को कभी नहीं बदलता। कोर यह
  सत्यापित नहीं करता कि अलग-अलग अनुरोधकर्ता एंडपॉइंट समान टूल स्कीमा प्रदान करते
  हैं; रिज़ॉल्वर को प्रत्येक अनुरोधकर्ता को उसी तार्किक सेवा की ओर इंगित करना
  आवश्यक है, अन्यथा टूल स्कीमा (और प्रॉम्प्ट-कैश स्थिरता) प्रत्येक अनुरोधकर्ता के
  अनुसार भिन्न हो जाते हैं।
- विश्वसनीय `requesterSenderId` के बिना चलने वाले रन (cron, subagent, Heartbeat,
  सार्वजनिक Gateway) अनुरोधकर्ता-सीमित सर्वरों को कभी मूर्त रूप नहीं देते। कोई
  साझा फ़ॉलबैक कनेक्शन नहीं है।
- `resolve` प्रत्येक सर्वर के लिए 10 सेकंड तक सीमित है; टाइमआउट या थ्रो
  होने पर स्थिर MCP को विफल किए बिना उस रन के लिए वह सर्वर छोड़ दिया जाता है।
- रिज़ॉल्व किए गए कनेक्शन प्रत्येक अनुरोधकर्ता के लिए अधिकतम हर 5 मिनट में
  पुनः सत्यापित किए जाते हैं: रोटेशन नए क्रेडेंशियल के साथ ट्रांसपोर्ट को फिर से
  बनाता है, और `null` परिणाम उसे निरस्त कर देता है (कैश किया गया
  रनटाइम सत्र के बीच में भी डिस्पोज़ कर दिया जाता है)। इसलिए निरस्त या रोटेट किया
  गया क्रेडेंशियल अधिकतम 5 मिनट तक उपयोग में रह सकता है।
- रिज़ॉल्व किए गए `headers` को कभी लॉग या स्थायी रूप से संग्रहीत नहीं
  किया जाता; क्रेडेंशियल रोटेशन का पता लगाने के लिए कोर केवल एक अस्थायी
  इन-मेमोरी कुंजीबद्ध डाइजेस्ट (प्रक्रिया-स्थानीय HMAC) रखता है, और रिज़ॉल्व किए
  गए हेडर/URL क्रेडेंशियल मानों को लॉग/डीबग-कैप्चर रिडैक्शन रजिस्ट्री में
  पंजीकृत करता है।
- अनुरोधकर्ता-सीमित सर्वर MCP ऐप दृश्य उत्पन्न नहीं करते: कोई दृश्य
  अनुरोधकर्ता-प्रमाणित रन से अधिक समय तक बना रहता है और Gateway दृश्य सीमा में
  अनुरोधकर्ता पहचान नहीं होती, इसलिए इन सर्वरों के लिए ऐप पूर्वावलोकन
  विफलता-बंद रहते हैं। टूल परिणाम अप्रभावित रहते हैं।
- रिज़ॉल्वर के बिना स्थिर सर्वर मौजूदा सत्र-सीमित जीवनचक्र बनाए रखते हैं।
- **हार्नेस डिलीवरी नियम:** अनुरोधकर्ता-सीमित सर्वर कभी हार्नेस-नेटिव
  MCP क्लाइंट कॉन्फ़िग (Codex थ्रेड `mcp_servers`, CLI `-c mcp_servers=…`, या
  किसी अन्य सत्र-साझा MCP प्रोजेक्शन) में प्रवेश नहीं करते। इसके बजाय हार्नेस
  उन्हें रन-सीमित टूल के रूप में प्रदान करते हैं:
  - एम्बेडेड रनर: सत्र MCP रनटाइम + बंडल टूल (स्थिर + सीमित)।
  - Codex ऐप-सर्वर: `materializeRequesterScopedMcpToolsForHarnessRun` के माध्यम से
    डायनेमिक टूल (केवल सीमित; स्थिर सर्वर Codex के नेटिव MCP क्लाइंट पर बने
    रहते हैं)।
- सीमित टूल **विनिर्देश** उस सत्र में पहले सफल रिज़ॉल्यूशन के बाद
  सत्र-स्थिर रहते हैं, ताकि साझा-थ्रेड हार्नेस (Codex) प्रेषक बदलने पर थ्रेड
  रोटेट न करें। किसी अनुरोधकर्ता के रिज़ॉल्व होने से पहले कोई सीमित विनिर्देश
  विज्ञापित नहीं किया जाता।
- साझा-थ्रेड हार्नेस पर अप्रमाणित अनुरोधकर्ता फिर भी विज्ञापित सीमित टूल
  देखते हैं; किसी टूल को कॉल करने पर उस अनुरोधकर्ता के लिए साफ़
  कनेक्ट-नहीं-है टूल त्रुटि लौटती है। OpenClaw कभी किसी अन्य अनुरोधकर्ता के
  क्रेडेंशियल पर फ़ॉलबैक नहीं करता।

मेमोरी प्रॉम्प्ट पूरक बिल्डरों को वैकल्पिक `agentId`,
`agentSessionKey`, और `sandboxed` संदर्भ मिलता है। मेमोरी कॉर्पस पूरक `search`
और `get` कॉल को वैकल्पिक `agentId` और `sandboxed` संदर्भ मिलता है। एजेंट-स्वामित्व
वाले स्टोरेज वाले Plugins को पंजीकरण के दौरान एक वैश्विक पथ कैप्चर करने के बजाय
प्रत्येक कॉल के लिए उस स्टोरेज को रिज़ॉल्व करना चाहिए। यदि बहु-एजेंट संचालन में
एजेंट आईडी आवश्यक है लेकिन अनुपस्थित है, तो कोई मनमाना एजेंट चुनने के बजाय
विफलता-बंद रहें।

जब प्रॉम्प्ट टेक्स्ट एसिंक्रोनस Plugin स्थिति पर निर्भर हो, तब
`registerMemoryPromptPreparation(...)` का उपयोग करें। कॉलबैक प्रत्येक पूर्ण एजेंट प्रॉम्प्ट से पहले
एक बार चलता है और उसे सिंक्रोनस मेमोरी प्रॉम्प्ट बिल्डरों के समान टूल, एजेंट,
सत्र और सैंडबॉक्स संदर्भ मिलता है। स्थायी स्थिति लोड करने से पहले वर्तमान
स्टोरेज-स्वामी इंस्टेंस को सत्यापित करें, फिर केवल उस रन की पंक्तियाँ लौटाएँ।
OpenClaw उन पंक्तियों को फ़्रीज़ करता है और अपरिवर्तनीय परिणाम सिंक्रोनस प्रॉम्प्ट
असेंबली को सौंप देता है। स्थायित्व, एटॉमिक प्रतिस्थापन और स्वामी हटाने पर विलोपन
को स्वामी Plugin के भीतर रखें; प्रॉम्प्ट बिल्डर से फ़ाइलों को पोल या रीड न करें।

Telegram इंटरैक्टिव हैंडलर, हैंडलर के सफल होने के बाद टेक्स्ट को Telegram के
सामान्य इनबाउंड एजेंट पथ से रूट करने के लिए `{ submitText }` लौटा सकते हैं।
जब इनबाउंड नीति टेक्स्ट को छोड़ देती है या प्रोसेसिंग विफल होती है, तब OpenClaw
कॉलबैक बटन बनाए रखता है, ताकि अवरोधक स्थिति बदलने के बाद उपयोगकर्ता पुनः प्रयास
कर सके। यह परिणाम फ़ील्ड Telegram-विशिष्ट है; अन्य चैनल अपने स्वयं के इंटरैक्टिव
परिणाम अनुबंध बनाए रखते हैं।

### वर्कफ़्लो Plugins के लिए होस्ट हुक

होस्ट हुक उन Plugins के लिए SDK सीम हैं जिन्हें केवल कोई प्रोवाइडर, चैनल या टूल
जोड़ने के बजाय होस्ट जीवनचक्र में भाग लेने की आवश्यकता होती है। ये सामान्य
अनुबंध हैं; Plan Mode इनका उपयोग कर सकता है, लेकिन स्वीकृति वर्कफ़्लो,
वर्कस्पेस नीति गेट, पृष्ठभूमि मॉनिटर, सेटअप विज़ार्ड और UI सहयोगी Plugins भी
इनका उपयोग कर सकते हैं।

| विधि                                                                                 | वह अनुबंध जिसका यह स्वामी है                                                                                                                                        |
| ------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `api.session.state.registerSessionExtension(...)`                                    | Plugin-स्वामित्व वाली, JSON-संगत सत्र स्थिति, जिसे Gateway सत्रों के माध्यम से प्रक्षेपित किया जाता है                                                               |
| `api.session.workflow.enqueueNextTurnInjection(...)`                                 | एक सत्र के लिए अगले एजेंट टर्न में अंतःक्षेपित टिकाऊ, बिल्कुल-एक-बार संदर्भ                                                                                          |
| `api.registerTrustedToolPolicy(...)`                                                 | मैनिफ़ेस्ट-गेटेड विश्वसनीय प्री-Plugin टूल नीति, जो टूल पैरामीटरों को अवरुद्ध या पुनर्लेखित कर सकती है                                                               |
| `api.registerToolMetadata(...)`                                                      | टूल कार्यान्वयन बदले बिना टूल कैटलॉग प्रदर्शन मेटाडेटा                                                                                                              |
| `api.registerCommand(...)`                                                           | सीमित-दायरे वाले Plugin कमांड; कमांड परिणाम `continueAgent: true` या `suppressReply: true` सेट कर सकते हैं; Discord नेटिव कमांड `descriptionLocalizations` का समर्थन करते हैं |
| `api.session.controls.registerControlUiDescriptor(...)`                              | सत्र, टूल, रन, सेटिंग्स या टैब सतहों के लिए Control UI योगदान वर्णनकर्ता                                                                                             |
| `api.lifecycle.registerRuntimeLifecycle(...)`                                        | रीसेट/हटाने/रीलोड पथों पर Plugin-स्वामित्व वाले रनटाइम संसाधनों के लिए क्लीनअप कॉलबैक                                                                                |
| `api.agent.events.registerAgentEventSubscription(...)`                               | वर्कफ़्लो स्थिति और मॉनिटर के लिए स्वच्छीकृत इवेंट सदस्यताएँ                                                                                                        |
| `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`  | प्रत्येक रन की Plugin स्क्रैच स्थिति, जिसे टर्मिनल रन जीवनचक्र पर साफ़ किया जाता है                                                                                  |
| `api.session.workflow.registerSessionSchedulerJob(...)`                              | Plugin-स्वामित्व वाली शेड्यूलर जॉब के लिए क्लीनअप मेटाडेटा; यह कार्य शेड्यूल नहीं करता या टास्क रिकॉर्ड नहीं बनाता                                                    |
| `api.session.workflow.sendSessionAttachment(...)`                                    | केवल बंडल किए गए Plugin के लिए, होस्ट-मध्यस्थ फ़ाइल अटैचमेंट की सक्रिय प्रत्यक्ष-आउटबाउंड सत्र रूट पर डिलीवरी                                                         |
| `api.session.workflow.scheduleSessionTurn(...)` / `unscheduleSessionTurnsByTag(...)` | केवल बंडल किए गए Plugin के लिए, Cron-समर्थित शेड्यूल किए गए सत्र टर्न और टैग-आधारित क्लीनअप                                                                          |
| `api.session.controls.registerSessionAction(...)`                                    | टाइप किए गए सत्र एक्शन, जिन्हें क्लाइंट Gateway के माध्यम से डिस्पैच कर सकते हैं                                                                                      |

एक `surface: "tab"` वर्णनकर्ता Control UI में साइडबार टैब जोड़ता है। सक्रिय
Plugin के टैब वर्णनकर्ता Gateway
हेलो (`controlUiTabs`) में डैशबोर्ड क्लाइंट को विज्ञापित किए जाते हैं, इसलिए टैब केवल Plugin के सक्षम रहने पर दिखाई देता है।
बंडल किए गए Plugin अपने टैब के लिए प्रथम-श्रेणी डैशबोर्ड दृश्य प्रदान कर सकते हैं; अन्य
Plugin `path` को Plugin HTTP रूट पर सेट कर सकते हैं (देखें
`api.registerHttpRoute(...)`), जिसे डैशबोर्ड सैंडबॉक्स किए गए फ़्रेम में रेंडर करता है।
`icon` डैशबोर्ड आइकन नाम का संकेत है, `group` साइडबार अनुभाग चुनता है
(`control` या `agent`), `order` Plugin टैब के बीच क्रम निर्धारित करता है, और `requiredScopes`
उन कनेक्शन से टैब छिपाता है जिनमें वे ऑपरेटर स्कोप नहीं हैं:

Gateway-संरक्षित बाहरी टैब के लिए, वर्णनकर्ता `path` को उसी
Plugin के `auth: "gateway"` HTTP रूट के अंतर्गत पंजीकृत करें। प्रमाणीकृत बूटस्ट्रैप के बाद, ब्राउज़र को
उस Plugin और रूट मूल तक सीमित अल्पकालिक, HttpOnly अनुदान मिलता है, ताकि
सैंडबॉक्स किया गया फ़्रेम Gateway बियरर टोकन को अपने URL
या JavaScript में कॉपी किए बिना लोड हो सके। बाहरी टैब सक्रिय रहते समय और नेविगेशन या ब्राउज़र पुनः आरंभ होने के बाद उसे माउंट करने से पहले प्रमाणीकृत पैरेंट अनुदान का नवीनीकरण करता है। वह
माउंट करने से पहले उसी अपारदर्शी सैंडबॉक्स से अनुदान की जाँच भी करता है, ताकि कुकी को अवरुद्ध करने वाले ब्राउज़र
गोपनीयता मोड अनुपलब्ध पैनल के साथ सुरक्षित रूप से विफल हों।
फ़्रेम अनुदान केवल `GET` और `HEAD` स्वीकार करता है और हमेशा
`operator.read` रखता है; `requiredScopes` टैब की दृश्यता नियंत्रित करता है, लेकिन
कुकी अनुदान का दायरा कभी नहीं बढ़ाता। म्यूटेशन स्पष्ट Gateway-प्रमाणीकृत पैरेंट या
बियरर सतहों पर बने रहते हैं। बाहरी टैब के लिए HTTPS/Tailscale Serve या
ब्राउज़र-विश्वसनीय लूपबैक ओरिजिन आवश्यक है; LAN होस्ट पर सादा HTTP ऐसा पैनल माउंट करने के बजाय
सुरक्षित-संदर्भ त्रुटि दिखाता है, जो प्रमाणित नहीं हो सकता।
तृतीय-पक्ष कुकी का पूर्ण अवरोध भी Gateway-संरक्षित टैब को अनुपलब्ध बना देता है।
सभी नेटिव Plugin सतहों की तरह, फ़्रेम इंस्टॉल किए गए
Plugin की विश्वास सीमा के भीतर रहता है; OpenClaw इंस्टॉल किए गए Plugin को परस्पर
पृथक ब्राउज़र सुरक्षा प्रिंसिपल नहीं मानता।
कुकी अनुदान ब्राउज़र की होस्टनेम सीमा का उपयोग करते हैं, पोर्ट सीमा का नहीं। परस्पर
अविश्वसनीय सेवाओं को Gateway होस्टनेम पर, अन्य
पोर्ट पर भी, सह-होस्ट न करें।
Plugin-प्रबंधित प्रमाणीकरण द्वारा समर्थित टैब अपना प्रत्यक्ष iframe व्यवहार बनाए रखते हैं और इस
Gateway अनुदान का अनुरोध या आवश्यकता नहीं रखते।

```typescript
api.session.controls.registerControlUiDescriptor({
  surface: "tab",
  id: "logbook",
  label: "दिन-पुस्तिका",
  description: "स्क्रीन स्नैपशॉट से निर्मित, समयरेखा के रूप में आपका दिन।",
  icon: "sun",
  group: "control",
  requiredScopes: ["operator.write"],
});
```

नए Plugin कोड के लिए समूहित नेमस्पेस का उपयोग करें:

- `api.session.state.registerSessionExtension(...)`
- `api.session.workflow.enqueueNextTurnInjection(...)`
- `api.session.workflow.registerSessionSchedulerJob(...)`
- `api.session.workflow.sendSessionAttachment(...)`
- `api.session.workflow.scheduleSessionTurn(...)`
- `api.session.workflow.unscheduleSessionTurnsByTag(...)`
- `api.session.controls.registerSessionAction(...)`
- `api.session.controls.registerControlUiDescriptor(...)`
- `api.agent.events.registerAgentEventSubscription(...)`
- `api.agent.events.emitAgentEvent(...)`
- `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`
- `api.lifecycle.registerRuntimeLifecycle(...)`

समतुल्य सपाट विधियाँ मौजूदा Plugin के लिए पदावनत संगतता
उपनामों के रूप में उपलब्ध रहती हैं। ऐसा नया Plugin कोड न जोड़ें जो सीधे
`api.registerSessionExtension`, `api.enqueueNextTurnInjection`,
`api.registerControlUiDescriptor`, `api.registerRuntimeLifecycle`,
`api.registerAgentEventSubscription`, `api.emitAgentEvent`,
`api.setRunContext`, `api.getRunContext`, `api.clearRunContext`,
`api.registerSessionSchedulerJob`, `api.registerSessionAction`,
`api.sendSessionAttachment`, `api.scheduleSessionTurn`, या
`api.unscheduleSessionTurnsByTag` को कॉल करता हो।

`scheduleSessionTurn(...)`, Gateway
Cron शेड्यूलर के ऊपर सत्र-सीमित सुविधा है। Cron समय-निर्धारण का स्वामी है और
टर्न चलने पर पृष्ठभूमि टास्क रिकॉर्ड बनाता है; Plugin SDK केवल लक्ष्य सत्र, Plugin-स्वामित्व वाले
नामकरण और क्लीनअप को सीमित करता है। जब स्वयं कार्य को टिकाऊ बहु-चरणीय Task Flow स्थिति की आवश्यकता हो, तब शेड्यूल किए गए
टर्न के भीतर `api.runtime.tasks.managedFlows` का उपयोग करें।

अनुबंध जानबूझकर अधिकार को विभाजित करते हैं:

- बाहरी Plugin सत्र एक्सटेंशन, UI वर्णनकर्ता, कमांड, टूल
  मेटाडेटा, अगले-टर्न के अंतःक्षेप और सामान्य हुक के स्वामी हो सकते हैं।
- विश्वसनीय टूल नीतियाँ सामान्य `before_tool_call` हुक से पहले चलती हैं और
  होस्ट-विश्वसनीय होती हैं। बंडल की गई नीतियाँ पहले चलती हैं; इंस्टॉल किए गए Plugin की नीतियों को
  स्पष्ट सक्षमता और
  `contracts.trustedToolPolicies` में उनके स्थानीय id की आवश्यकता होती है, और वे Plugin-लोड क्रम में इसके बाद चलती हैं। नीति id
  पंजीकरण करने वाले Plugin तक सीमित होते हैं।
- आरक्षित कमांड स्वामित्व केवल बंडल किए गए Plugin के लिए है। बाहरी Plugin को अपने
  कमांड नामों या उपनामों का उपयोग करना चाहिए।
- `allowPromptInjection=false`, प्रॉम्प्ट को बदलने वाले हुक अक्षम करता है, जिनमें
  `agent_turn_prepare`, `before_prompt_build`, `heartbeat_prompt_contribution`,
  विरासती `before_agent_start` के प्रॉम्प्ट फ़ील्ड, और
  `enqueueNextTurnInjection` शामिल हैं।

गैर-Plan उपभोक्ताओं के उदाहरण:

| Plugin प्रारूप                   | उपयोग किए गए हुक                                                                                                                        |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| अनुमोदन वर्कफ़्लो               | सत्र एक्सटेंशन, कमांड निरंतरता, अगले-टर्न का अंतःक्षेप, UI वर्णनकर्ता                                                                   |
| बजट/वर्कस्पेस नीति गेट          | विश्वसनीय टूल नीति, टूल मेटाडेटा, सत्र प्रक्षेपण                                                                                       |
| पृष्ठभूमि जीवनचक्र मॉनिटर       | रनटाइम जीवनचक्र क्लीनअप, एजेंट इवेंट सदस्यता, सत्र शेड्यूलर स्वामित्व/क्लीनअप, Heartbeat प्रॉम्प्ट योगदान, UI वर्णनकर्ता                |
| सेटअप या ऑनबोर्डिंग विज़ार्ड    | सत्र एक्सटेंशन, सीमित-दायरे वाले कमांड, Control UI वर्णनकर्ता                                                                          |

<Note>
  आरक्षित कोर एडमिन नेमस्पेस (`config.*`, `exec.approvals.*`, `wizard.*`,
  `update.*`) हमेशा `operator.admin` रहते हैं, भले ही कोई Plugin अधिक
  संकीर्ण Gateway विधि स्कोप असाइन करने का प्रयास करे। Plugin-स्वामित्व वाली विधियों के लिए
  Plugin-विशिष्ट उपसर्गों को प्राथमिकता दें।
</Note>

<Accordion title="टूल-परिणाम मिडलवेयर का उपयोग कब करें">
  मिलते-जुलते
  मैनिफ़ेस्ट अनुबंध वाले बंडल किए गए Plugin और स्पष्ट रूप से सक्षम इंस्टॉल किए गए Plugin, `api.registerAgentToolResultMiddleware(...)` का उपयोग तब कर सकते हैं जब
  उन्हें निष्पादन के बाद और रनटाइम द्वारा परिणाम को
  मॉडल में वापस भेजने से पहले टूल परिणाम को पुनर्लेखित करना हो। यह tokenjuice जैसे असिंक्रोनस आउटपुट रिड्यूसर के लिए विश्वसनीय रनटाइम-निरपेक्ष
  सीम है।

Plugin को प्रत्येक लक्षित
रनटाइम के लिए `contracts.agentToolResultMiddleware` घोषित करना होगा, उदाहरण के लिए `["openclaw", "codex"]`। उस
अनुबंध के बिना, या स्पष्ट सक्षमता के बिना इंस्टॉल किए गए Plugin इस मिडलवेयर को पंजीकृत नहीं कर सकते; ऐसे कार्य के लिए
सामान्य OpenClaw Plugin हुक रखें, जिसे मॉडल-पूर्व टूल-परिणाम
समय-निर्धारण की आवश्यकता नहीं है। पुराना
केवल-एम्बेडेड-रनर एक्सटेंशन फ़ैक्टरी पंजीकरण पथ हटा दिया गया है।
</Accordion>

### Gateway डिस्कवरी पंजीकरण

`api.registerGatewayDiscoveryService(...)`, Plugin को mDNS/Bonjour जैसे स्थानीय डिस्कवरी ट्रांसपोर्ट पर सक्रिय
Gateway का विज्ञापन करने देता है। स्थानीय डिस्कवरी सक्षम होने पर OpenClaw
Gateway स्टार्टअप के दौरान सेवा को कॉल करता है, वर्तमान
Gateway पोर्ट और गैर-गोपनीय TXT संकेत डेटा पास करता है, और Gateway शटडाउन के दौरान लौटाए गए
`stop` हैंडलर को कॉल करता है।

```typescript
api.registerGatewayDiscoveryService({
  id: "my-discovery",
  async advertise(ctx) {
    const handle = await startMyAdvertiser({
      gatewayPort: ctx.gatewayPort,
      tls: ctx.gatewayTlsEnabled,
      displayName: ctx.machineDisplayName,
    });
    return { stop: () => handle.stop() };
  },
});
```

Gateway डिस्कवरी Plugin को विज्ञापित TXT मानों को गोपनीय या
प्रमाणीकरण नहीं मानना चाहिए। डिस्कवरी एक रूटिंग संकेत है; Gateway प्रमाणीकरण और TLS पिनिंग अब भी
विश्वास के स्वामी हैं।

### CLI पंजीकरण मेटाडेटा

`api.registerCli(registrar, opts?)` दो प्रकार के कमांड मेटाडेटा स्वीकार करता है:

- `commands`: पंजीकरणकर्ता के स्वामित्व वाले स्पष्ट कमांड नाम
- `descriptors`: CLI सहायता,
  रूटिंग और आलसी Plugin CLI पंजीकरण के लिए उपयोग किए जाने वाले पार्स-समय कमांड वर्णनकर्ता
- `parentPath`: नेस्टेड कमांड समूहों के लिए वैकल्पिक पैरेंट कमांड पथ, जैसे
  `["nodes"]`

युग्मित-Node सुविधाओं के लिए,
`api.registerNodeCliFeature(registrar, opts?)` को प्राथमिकता दें। यह
`api.registerCli(..., { parentPath: ["nodes"] })` के ऊपर एक छोटा रैपर है और
`openclaw nodes canvas` जैसे कमांड को स्पष्ट Plugin-स्वामित्व वाली Node सुविधाएँ बनाता है।

यदि आप चाहते हैं कि कोई Plugin कमांड सामान्य रूट CLI पथ में आलसी रूप से लोड रहे,
तो ऐसा `descriptors` प्रदान करें, जो उस
पंजीकरणकर्ता द्वारा उजागर किए गए प्रत्येक शीर्ष-स्तरीय कमांड रूट को कवर करे।

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerMatrixCli } = await import("./src/cli.js");
    registerMatrixCli({ program });
  },
  {
    descriptors: [
      {
        name: "matrix",
        description: "Matrix खातों, सत्यापन, डिवाइस और प्रोफ़ाइल स्थिति को प्रबंधित करें",
        hasSubcommands: true,
      },
    ],
  },
);
```

नेस्टेड कमांड को समाधान किया गया पैरेंट कमांड `program` के रूप में मिलता है:

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerNodesCanvasCommands } = await import("./src/cli.js");
    registerNodesCanvasCommands(program);
  },
  {
    parentPath: ["nodes"],
    descriptors: [
      {
        name: "canvas",
        description: "युग्मित नोड से कैनवास सामग्री कैप्चर या रेंडर करें",
        hasSubcommands: true,
      },
    ],
  },
);
```

`commands` का अकेले उपयोग केवल तब करें, जब आपको आलसी रूट CLI पंजीकरण की आवश्यकता न हो।
वह तत्पर संगतता पथ अब भी समर्थित है, लेकिन वह पार्स-समय आलसी लोडिंग के लिए
डिस्क्रिप्टर-समर्थित प्लेसहोल्डर इंस्टॉल नहीं करता।

### CLI बैकएंड पंजीकरण

`api.registerCliBackend(...)` किसी Plugin को `claude-cli` या `my-cli` जैसे स्थानीय
AI CLI बैकएंड के लिए डिफ़ॉल्ट कॉन्फ़िगरेशन का स्वामी बनने देता है।

- बैकएंड `id`, `my-cli/gpt-5` जैसे मॉडल संदर्भों में प्रोवाइडर प्रीफ़िक्स बन जाता है।
- बैकएंड `config`, `agents.defaults.cliBackends.<id>` के समान संरचना का उपयोग करता है।
- उपयोगकर्ता कॉन्फ़िगरेशन को अब भी प्राथमिकता मिलती है। CLI चलाने से पहले OpenClaw, `agents.defaults.cliBackends.<id>` को
  Plugin डिफ़ॉल्ट के ऊपर मर्ज करता है।
- जब किसी बैकएंड को मर्ज के बाद संगतता पुनर्लेखन की आवश्यकता हो
  (उदाहरण के लिए, पुराने फ़्लैग स्वरूपों को सामान्यीकृत करना), तब `normalizeConfig` का उपयोग करें।
- अनुरोध-स्कोप वाले argv पुनर्लेखनों के लिए `resolveExecutionArgs` का उपयोग करें, जो
  CLI बोली से संबंधित हों, जैसे OpenClaw के चिंतन स्तरों को किसी नेटिव प्रयास
  फ़्लैग से मैप करना। हुक को `ctx.executionMode` मिलता है; क्षणिक `/btw` कॉल के लिए
  बैकएंड-नेटिव पृथक्करण फ़्लैग जोड़ने हेतु `"side-question"` का उपयोग करें। यदि वे फ़्लैग
  अन्यथा हमेशा चालू रहने वाले CLI के नेटिव टूल को विश्वसनीय रूप से अक्षम करते हैं, तो
  `sideQuestionToolMode: "disabled"` भी घोषित करें।
- बैकएंड-स्वामित्व वाले लॉन्च परिवेश या अस्थायी
  प्रमाणीकरण/कॉन्फ़िगरेशन ब्रिज के लिए `prepareExecution` का उपयोग करें। इसका `ctx.contextTokenBudget`, रन के लिए
  चुनी गई प्रभावी टोकन सीमा है, ताकि नेटिव-Compaction बैकएंड प्रोवाइडर-विशिष्ट कोर शाखाओं के बिना
  अपनी सीमा को संरेखित कर सकें।
- जो बैकएंड किसी विशिष्ट रन के लिए सभी नेटिव टूल को अक्षम कर सकते हैं, वे
  `nativeToolMode: "selectable"` घोषित कर सकते हैं। प्रतिबंधित कॉल एक खाली
  `ctx.toolAvailability.native` टपल और एक सटीक होस्ट-पृथक MCP अनुमति-सूची पास करते हैं;
  `resolveExecutionArgs` को अंतिम नए या पुनःआरंभ argv पर दोनों लागू करने होंगे।
  यदि बैकएंड ऐसा नहीं कर सकता, तो OpenClaw सुरक्षित रूप से विफल हो जाता है।

आरंभ से अंत तक लेखन मार्गदर्शिका के लिए,
[CLI बैकएंड plugins](/hi/plugins/cli-backend-plugins) देखें।

### हुक निर्णय सिमेंटिक्स

`before_install` एक Plugin-रनटाइम जीवनचक्र हुक है, ऑपरेटर इंस्टॉल
नीति सतह नहीं। जब अनुमति/अवरोध निर्णय को CLI और Gateway-समर्थित इंस्टॉल या अपडेट पथों को
शामिल करना हो, तब `security.installPolicy` का उपयोग करें।

- `before_tool_call`: `{ block: true }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा इसे सेट करने के बाद, निम्न-प्राथमिकता हैंडलर छोड़ दिए जाते हैं।
- `before_tool_call`: `{ block: false }` लौटाना कोई निर्णय नहीं माना जाता (`block` को छोड़ देने के समान), ओवरराइड नहीं।
- `before_install`: `{ block: true }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा इसे सेट करने के बाद, निम्न-प्राथमिकता हैंडलर छोड़ दिए जाते हैं।
- `before_install`: `{ block: false }` लौटाना कोई निर्णय नहीं माना जाता (`block` को छोड़ देने के समान), ओवरराइड नहीं।
- `reply_dispatch`: `{ handled: true, ... }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा डिस्पैच का दावा करने के बाद, निम्न-प्राथमिकता हैंडलर और डिफ़ॉल्ट मॉडल डिस्पैच पथ छोड़ दिए जाते हैं।
- `message_sending`: `{ cancel: true }` लौटाना अंतिम है। किसी भी हैंडलर द्वारा इसे सेट करने के बाद, निम्न-प्राथमिकता हैंडलर छोड़ दिए जाते हैं।
- `message_sending`: `{ cancel: false }` लौटाना कोई निर्णय नहीं माना जाता (`cancel` को छोड़ देने के समान), ओवरराइड नहीं।
- `message_received`: जब आपको इनबाउंड थ्रेड/विषय रूटिंग की आवश्यकता हो, तब टाइप किए हुए `threadId` फ़ील्ड का उपयोग करें। चैनल-विशिष्ट अतिरिक्त डेटा के लिए `metadata` बनाए रखें।
- `message_sending`: चैनल-विशिष्ट `metadata` पर फ़ॉलबैक करने से पहले टाइप किए हुए `replyToId` / `threadId` रूटिंग फ़ील्ड का उपयोग करें।
- `gateway_start`: आंतरिक `gateway:startup` हुक पर निर्भर रहने के बजाय Gateway-स्वामित्व वाली स्टार्टअप स्थिति के लिए `ctx.config`, `ctx.workspaceDir`, और `ctx.getCron?.()` का उपयोग करें। इस बिंदु पर Cron अब भी लोड हो रहा हो सकता है।
- `cron_reconciled`: स्टार्टअप या शेड्यूलर रीलोड के बाद पूर्ण बाहरी Cron प्रोजेक्शन का पुनर्निर्माण करें। इसमें `reason` और प्रभावी `enabled` स्थिति, जिसमें `enabled: false` भी शामिल है, होती है; वहीं `ctx.getCron?.()` सटीक समन्वित शेड्यूलर लौटाता है। स्थायी प्रोजेक्शन कार्य में `ctx.abortSignal` पास करें; जब उस शेड्यूलर स्नैपशॉट को प्रतिस्थापित कर दिया जाता है या Gateway बंद हो जाता है, तो यह निरस्त हो जाता है।
- `cron_changed`: Gateway-स्वामित्व वाले Cron जीवनचक्र परिवर्तनों का अवलोकन करें। `scheduled` और `removed` इवेंट कमिट के बाद के समन्वयन संकेत हैं, क्रमबद्ध डेल्टा लॉग नहीं। यदि जॉब का अगला जागरण नहीं है, तो निर्धारित इवेंट का `event.nextRunAtMs` अनुपस्थित होता है; हटाए गए इवेंट में अब भी मिटाए गए जॉब का स्नैपशॉट होता है।

बाहरी वेक शेड्यूलरों को `cron_changed` इवेंट को डिबाउंस या समेकित करना चाहिए,
फिर `cron_reconciled` द्वारा अंतिम बार कैप्चर किए गए शेड्यूलर से पूर्ण स्थायी दृश्य को दोबारा पढ़ना चाहिए।
`cron_changed` संदर्भ से शेड्यूलर न अपनाएँ: किसी पुराने शेड्यूलर से अलग हुआ
संकेत बाद के रीलोड से ओवरलैप कर सकता है।

Gateway स्टार्टअप या शेड्यूलर प्रतिस्थापन पर लोड की गई स्थायी स्थिति के लिए
पूर्ण-स्नैपशॉट ट्रिगर के रूप में `cron_reconciled` का उपयोग करें। इसे केवल-Plugin
हॉट रीलोड के लिए दोबारा नहीं चलाया जाता। अवलोकन हैंडलर समानांतर रूप से चलते हैं, और फ़ायर-एंड-फ़ॉरगेट
डिस्पैच ओवरलैप हो सकते हैं, इसलिए उपभोक्ताओं को इवेंट पूर्णता क्रम पर निर्भर नहीं होना चाहिए।
नियत समय की जाँच और निष्पादन के लिए OpenClaw को सत्य का स्रोत बनाए रखें।

स्थायी प्रतिस्थापन, पुनःप्रयास/बैकऑफ़ और साफ़
शटडाउन वाले एकल-उड़ान अडैप्टर के लिए [सुरक्षित बाहरी Cron प्रोजेक्शन](/hi/plugins/hooks#safe-external-cron-projection) देखें।

| `api.rootDir`            | `string?`                 | Plugin रूट डायरेक्टरी (वैकल्पिक)                                                            |
| `api.config`             | `OpenClawConfig`          | वर्तमान कॉन्फ़िगरेशन स्नैपशॉट (उपलब्ध होने पर सक्रिय इन-मेमोरी रनटाइम स्नैपशॉट)                  |
| `api.pluginConfig`       | `Record<string, unknown>` | `plugins.entries.<id>.config` से Plugin-विशिष्ट कॉन्फ़िगरेशन                                   |
| `api.runtime`            | `PluginRuntime`           | [रनटाइम सहायक](/hi/plugins/sdk-runtime)                                                     |
| `api.logger`             | `PluginLogger`            | स्कोप किया हुआ लॉगर (`debug`, `info`, `warn`, `error`)                                            |
| `api.registrationMode`   | `PluginRegistrationMode`  | वर्तमान लोड मोड; `"setup-runtime"` पूर्ण-प्रविष्टि से पहले की हल्की स्टार्टअप/सेटअप अवधि है |
| `api.resolvePath(input)` | `(string) => string`      | Plugin रूट के सापेक्ष पथ का समाधान करें                                                        |

## आंतरिक मॉड्यूल परंपरा

अपने Plugin के भीतर, आंतरिक इंपोर्ट के लिए स्थानीय बैरल फ़ाइलों का उपयोग करें:

```text
my-plugin/
  api.ts            # बाहरी उपभोक्ताओं के लिए सार्वजनिक निर्यात
  runtime-api.ts    # केवल आंतरिक उपयोग के लिए रनटाइम निर्यात
  index.ts          # Plugin प्रवेश बिंदु
  setup-entry.ts    # हल्का, केवल सेटअप के लिए प्रवेश बिंदु (वैकल्पिक)
```

<Warning>
  प्रोडक्शन कोड से अपने Plugin को कभी भी `openclaw/plugin-sdk/<your-plugin>`
  के माध्यम से आयात न करें। आंतरिक आयातों को `./api.ts` या
  `./runtime-api.ts` के माध्यम से रूट करें। SDK पथ केवल बाहरी अनुबंध है।
</Warning>

फ़साड के माध्यम से लोड किए गए बंडल Plugin की सार्वजनिक सतहें (`api.ts`, `runtime-api.ts`,
`index.ts`, `setup-entry.ts`, और इसी तरह की सार्वजनिक प्रवेश फ़ाइलें), OpenClaw के पहले से
चल रहे होने पर सक्रिय रनटाइम कॉन्फ़िगरेशन स्नैपशॉट को प्राथमिकता देती हैं। यदि अभी तक कोई रनटाइम
स्नैपशॉट मौजूद नहीं है, तो वे डिस्क पर मौजूद समाधान की गई कॉन्फ़िगरेशन फ़ाइल का उपयोग करती हैं।
पैकेज किए गए बंडल Plugin फ़साड को OpenClaw के Plugin
फ़साड लोडर के माध्यम से लोड किया जाना चाहिए; `dist/extensions/...` से सीधे आयात करने पर उन मैनिफ़ेस्ट
और रनटाइम साइडकार जाँचों को बायपास कर दिया जाता है, जिनका उपयोग पैकेज किए गए इंस्टॉलेशन Plugin-स्वामित्व वाले कोड के लिए करते हैं।

जब कोई सहायक जानबूझकर प्रदाता-विशिष्ट हो और अभी किसी सामान्य SDK
उपपथ में उपयुक्त न हो, तो प्रदाता Plugin एक सीमित, Plugin-स्थानीय अनुबंध बैरल उपलब्ध करा सकते हैं।
बंडल किए गए उदाहरण:

- **Anthropic**: Claude
  बीटा-हेडर और `service_tier` स्ट्रीम सहायकों के लिए सार्वजनिक `api.ts` / `contract-api.ts` सीम।
- **`@openclaw/openai-provider`**: `api.ts` प्रदाता बिल्डर,
  डिफ़ॉल्ट-मॉडल सहायक और रियलटाइम प्रदाता बिल्डर निर्यात करता है।
- **`@openclaw/openrouter-provider`**: `api.ts` प्रदाता बिल्डर
  के साथ ऑनबोर्डिंग/कॉन्फ़िगरेशन सहायक निर्यात करता है।

<Warning>
  एक्सटेंशन प्रोडक्शन कोड को भी `openclaw/plugin-sdk/<other-plugin>`
  आयातों से बचना चाहिए। यदि कोई सहायक वास्तव में साझा है, तो दो Plugin को आपस में जोड़ने के बजाय
  उसे `openclaw/plugin-sdk/speech`, `.../provider-model-shared`, या किसी अन्य
  क्षमता-उन्मुख सतह जैसे तटस्थ SDK उपपथ में स्थानांतरित करें।
</Warning>
