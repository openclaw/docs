Plugin hooks, OpenClaw plugins के लिए इन-प्रोसेस विस्तार बिंदु हैं: एजेंट रन, टूल कॉल, संदेश प्रवाह, सत्र जीवनचक्र, सबएजेंट रूटिंग, इंस्टॉलेशन या Gateway स्टार्टअप का निरीक्षण करें या उन्हें बदलें।

इसके बजाय कमांड और Gateway इवेंट, जैसे `/new`, `/reset`, `/stop`, `agent:bootstrap` या `gateway:startup`, पर प्रतिक्रिया देने वाली ऑपरेटर द्वारा इंस्टॉल की गई छोटी `HOOK.md` स्क्रिप्ट के लिए [आंतरिक हुक](/hi/automation/hooks) का उपयोग करें।

## त्वरित शुरुआत

Plugin एंट्री से `api.on(...)` के साथ टाइप किए गए हुक पंजीकृत करें:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

export default definePluginEntry({
  id: "tool-preflight",
  name: "Tool Preflight",
  register(api) {
    api.on(
      "before_tool_call",
      async (event) => {
        if (event.toolName !== "web_search") {
          return;
        }

        return {
          requireApproval: {
            title: "Run web search",
            description: `Allow search query: ${String(event.params.query ?? "")}`,
            severity: "info",
            timeoutMs: 60_000,
          },
        };
      },
      { priority: 50 },
    );
  },
});
```

निर्णय या संशोधन लौटा सकने वाले हैंडलर घटते `priority` क्रम में अनुक्रमिक रूप से चलते हैं; समान प्राथमिकता वाले हैंडलर पंजीकरण क्रम बनाए रखते हैं। केवल-अवलोकन वाले हैंडलर समानांतर चलते हैं और फायर-एंड-फॉरगेट अवलोकन डिस्पैच बाद के इवेंट के साथ ओवरलैप हो सकते हैं। अवलोकन के साइड इफ़ेक्ट का क्रम निर्धारित करने के लिए प्राथमिकता का उपयोग न करें।

`api.on(name, handler, opts?)` निम्न को स्वीकार करता है:

| विकल्प      | प्रभाव                                                                                                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `priority`  | क्रम निर्धारण; अधिक मान वाला पहले चलता है।                                                                                                                                                                      |
| `timeoutMs` | प्रति-हुक प्रतीक्षा बजट। इसकी अवधि समाप्त होने पर OpenClaw उस हैंडलर की प्रतीक्षा बंद करके आगे बढ़ जाता है। यह हैंडलर या उसके साइड इफ़ेक्ट को रद्द नहीं करता। रनर के डिफ़ॉल्ट प्रति-हुक टाइमआउट का उपयोग करने के लिए इसे छोड़ दें। |

ऑपरेटर Plugin कोड में पैच किए बिना हुक बजट सेट कर सकते हैं:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "hooks": {
          "timeoutMs": 30000,
          "timeouts": {
            "before_prompt_build": 90000,
            "agent_end": 60000
          }
        }
      }
    }
  }
}
```

`hooks.timeouts.<hookName>`, `hooks.timeoutMs` को ओवरराइड करता है, जो Plugin द्वारा लिखे गए `api.on(..., { timeoutMs })` मान को ओवरराइड करता है। प्रत्येक मान 600000 ms तक का धनात्मक पूर्णांक होना चाहिए। ज्ञात रूप से धीमे हुक के लिए प्रति-हुक ओवरराइड को प्राथमिकता दें, ताकि किसी एक Plugin को हर जगह अधिक लंबा बजट न मिले।

समय-सीमा समाप्त हो चुके हैंडलर का प्रॉमिस चलता रहता है, क्योंकि हुक कॉलबैक को रद्दीकरण संकेत नहीं मिलता। Plugin का वह कार्य अभी भी जारी रहने के दौरान हुक डिस्पैच अपने Gateway प्रवेश को मुक्त कर सकता है। लंबे समय तक चलने वाले कार्य के स्वामी Plugins को अपना रद्दीकरण और शटडाउन जीवनचक्र उपलब्ध कराना होगा।

आउटबाउंड संशोधन हुक `message_sending` और `reply_payload_sending` प्रति हैंडलर 15-सेकंड का डिफ़ॉल्ट उपयोग करते हैं। यदि किसी का समय समाप्त हो जाता है, तो OpenClaw Plugin त्रुटि को लॉग करता है और नवीनतम पेलोड के साथ जारी रहता है, ताकि क्रमबद्ध डिलीवरी लेन स्थिर हो सके। डिलीवरी से पहले जानबूझकर धीमा कार्य करने वाले Plugins के लिए अधिक बड़ा प्रति-हुक बजट सेट करें।

`createReplyDispatcher` का उपयोग करने वाले चैनल Plugins इसी तरह `beforeDeliverOptions: { timeoutMs }` के साथ, या `dispatcher.appendBeforeDeliver(handler, { timeoutMs })` के माध्यम से कार्य जोड़ते समय, अधिक बड़ा धनात्मक प्रति-चरण बजट घोषित कर सकते हैं।
स्वामी द्वारा घोषित बजट के बिना, वे कॉलबैक वही 15-सेकंड का डिफ़ॉल्ट उपयोग करते हैं, ताकि कोई अटका हुआ कॉलबैक क्रमबद्ध डिलीवरी लेन को रोके न रख सके।

प्रत्येक हुक को `event.context.pluginConfig` मिलता है, जो उस हैंडलर को पंजीकृत करने वाले Plugin के लिए समाधान किया गया कॉन्फ़िगरेशन है। OpenClaw इसे प्रति हैंडलर इंजेक्ट करता है और अन्य Plugins को दिखने वाले साझा इवेंट ऑब्जेक्ट को परिवर्तित नहीं करता।

## हुक कैटलॉग

हुक को उनके द्वारा विस्तारित सतह के अनुसार समूहित किया गया है। **बोल्ड** नाम निर्णय परिणाम (ब्लॉक करना, रद्द करना, ओवरराइड करना या अनुमोदन की आवश्यकता रखना) स्वीकार करते हैं; शेष केवल-अवलोकन के लिए हैं।

**एजेंट टर्न**

| हुक                            | उद्देश्य                                                                                  |
| ------------------------------- | ---------------------------------------------------------------------------------------- |
| `before_model_resolve`          | सत्र संदेश लोड होने से पहले प्रोवाइडर या मॉडल को ओवरराइड करना                                  |
| `agent_turn_prepare`            | कतारबद्ध Plugin टर्न इंजेक्शन का उपयोग करना और प्रॉम्प्ट हुक से पहले उसी टर्न का संदर्भ जोड़ना      |
| `before_prompt_build`           | मॉडल कॉल से पहले डायनेमिक संदर्भ या सिस्टम-प्रॉम्प्ट टेक्स्ट जोड़ना                          |
| `before_agent_start`            | केवल संगतता के लिए संयुक्त चरण; ऊपर दिए गए दोनों हुक को प्राथमिकता दें                            |
| **`before_agent_run`**          | मॉडल सबमिशन से पहले अंतिम प्रॉम्प्ट और सत्र संदेशों का निरीक्षण करना; रन को ब्लॉक कर सकता है |
| **`before_agent_reply`**        | सिंथेटिक उत्तर या मौन के साथ मॉडल टर्न को बीच में समाप्त करना                           |
| **`before_agent_finalize`**     | स्वाभाविक अंतिम उत्तर का निरीक्षण करना और मॉडल से एक और पास का अनुरोध करना                         |
| `agent_end`                     | अंतिम संदेशों, सफलता की स्थिति और रन अवधि का अवलोकन करना                                  |
| `heartbeat_prompt_contribution` | बैकग्राउंड मॉनिटर और जीवनचक्र Plugins के लिए केवल Heartbeat संदर्भ जोड़ना                  |

**सत्र और Compaction**

| हुक                                     | उद्देश्य                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `session_start` / `session_end`          | सत्र जीवनचक्र सीमाओं को ट्रैक करना। `reason`, `new`, `reset`, `idle`, `daily`, `compaction`, `deleted`, `shutdown`, `restart` या `unknown` में से एक है। प्रक्रिया के सक्रिय सत्रों के साथ रुकने या पुनरारंभ होने पर `shutdown`/`restart`, Gateway शटडाउन फ़ाइनलाइज़र से सक्रिय होते हैं, ताकि Plugins (मेमोरी, ट्रांसक्रिप्ट स्टोर) पुनरारंभ के दौरान घोस्ट पंक्तियों को खुला छोड़ने के बजाय उन्हें अंतिम रूप दे सकें। फ़ाइनलाइज़र सीमित है, ताकि कोई धीमा Plugin SIGTERM/SIGINT को ब्लॉक न कर सके। |
| `before_compaction` / `after_compaction` | Compaction चक्रों का अवलोकन करना या उन पर टिप्पणी जोड़ना                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `before_reset`                           | सत्र-रीसेट इवेंट (`/reset`, प्रोग्रामेटिक रीसेट) का अवलोकन करना                                                                                                                                                                                                                                                                                                                                                                                                     |

`parentSessionKey` और `emitCommandHooks: true` वाले `sessions.create` कॉल के लिए, एक अलग चाइल्ड को हमेशा `session_start` मिलता है। कॉलर `succeedsParent` के माध्यम से घोषित करते हैं कि पैरेंट को टर्मिनल `session_end` भी मिलेगा या नहीं: `true` का अर्थ उत्तराधिकारी है, `false` का अर्थ समानांतर चाइल्ड है। इसे छोड़ने पर पुराना पैरेंट-रोलओवर व्यवहार संरक्षित रहता है। दोनों मामलों में `command:new` और `before_reset` हुक अनुरोधित `/new` कार्रवाई का वर्णन करते रहते हैं।

**सबएजेंट**

- `subagent_spawned` / `subagent_ended` - सबएजेंट के लॉन्च और पूर्णता का अवलोकन करें।
- `subagent_delivery_target` - जब कोई कोर सत्र बाइंडिंग किसी रूट को प्रोजेक्ट नहीं कर सकती, तब पूर्णता डिलीवरी के लिए संगतता हुक।
- `subagent_spawning` - अप्रचलित संगतता हुक। अब कोर, `subagent_spawned` के सक्रिय होने से पहले चैनल सत्र-बाइंडिंग अडैप्टर के माध्यम से `thread: true` सबएजेंट बाइंडिंग तैयार करता है।
- `subagent_spawned` में `resolvedModel` और `resolvedProvider` शामिल होते हैं, जब OpenClaw ने लॉन्च से पहले चाइल्ड सत्र का मूल मॉडल निर्धारित कर लिया हो।
- `subagent_ended` में `targetSessionKey` (पहचान - `subagent_spawned.childSessionKey` से मेल खाती है), `targetKind` (`"subagent"` या `"acp"`), `reason`, वैकल्पिक `outcome` (`"ok"`, `"error"`, `"timeout"`, `"killed"`, `"reset"`, या `"deleted"`), वैकल्पिक `error`, `runId`, `endedAt`, `accountId`, और `sendFarewell` होते हैं। इसमें `agentId` या `childSessionKey` शामिल **नहीं** होते; मेल खाने वाले `subagent_spawned` इवेंट से सहसंबंधित करने के लिए `targetSessionKey` का उपयोग करें।

| **`before_install`**             | लोड किए गए Plugin रनटाइम से स्टेज की गई skill या Plugin इंस्टॉल सामग्री का निरीक्षण करना                           |

### चैनल पेयरिंग अनुरोध

जब किसी अनपेयर किए गए DM प्रेषक द्वारा लंबित पेयरिंग अनुरोध बनाए जाने के बाद किसी Plugin को ऑपरेटर को सूचित करना हो या ऑडिट रिकॉर्ड लिखना हो, तब `channel_pairing_requested` का उपयोग करें। अनुरोध बनाए जाने पर हुक डिस्पैच होता है; धीमे या विफल हुक हैंडलर के कारण पेयरिंग उत्तर की चैनल डिलीवरी में देरी नहीं होती।

```typescript
api.on("channel_pairing_requested", async (event) => {
  await notifyOperator({
    text: `${event.channel} से नया पेयरिंग अनुरोध ${event.senderId} द्वारा: ${event.code}`,
  });
});
```

यह हुक केवल अवलोकन के लिए है। यह पेयरिंग उत्तर को स्वीकृत, अस्वीकृत, दबाता या पुनर्लिखित नहीं करता। पेलोड में चैनल, वैकल्पिक `accountId`, चैनल-स्कोप वाला `senderId`, पेयरिंग `code`, और चैनल मेटाडेटा शामिल हैं। पेयरिंग कोड को सक्रिय, एकल-उपयोग स्वीकृति क्रेडेंशियल मानें और इसे केवल विश्वसनीय ऑपरेटर सिंक तक पहुँचाएँ। `metadata` को प्रेषक द्वारा दी गई अविश्वसनीय पहचान टेक्स्ट मानें। हुक में इनबाउंड संदेश का मुख्य भाग या मीडिया शामिल नहीं होता।

## डीबग रनटाइम हुक

किसी एजेंट टर्न के लिए प्रदाता या मॉडल बदलने हेतु `before_model_resolve` का उपयोग करें - यह मॉडल रिज़ॉल्यूशन से पहले चलता है। `llm_output` केवल मॉडल प्रयास द्वारा सहायक आउटपुट उत्पन्न किए जाने के बाद चलता है।

प्रभावी सत्र मॉडल के प्रमाण के लिए रनटाइम पंजीकरणों का निरीक्षण करें, फिर `openclaw sessions` या Gateway सत्र/स्थिति सतहों का उपयोग करें। प्रदाता पेलोड डीबग करने के लिए Gateway को `--raw-stream` और `--raw-stream-path <path>` के साथ शुरू करें, ताकि अपरिष्कृत मॉडल स्ट्रीम इवेंट किसी jsonl फ़ाइल में लिखे जाएँ।

## टूल कॉल नीति

`before_tool_call` को प्राप्त होते हैं:

- `event.toolName`
- `event.params`
- वैकल्पिक `event.toolKind` और `event.toolInputKind`, जानबूझकर समान नाम साझा करने वाले टूल के लिए होस्ट-प्रामाणिक विभेदक; उदाहरण के लिए, बाहरी कोड-मोड `exec` कॉल `toolKind: "code_mode_exec"` का उपयोग करती हैं और इनपुट भाषा ज्ञात होने पर `toolInputKind: "javascript" | "typescript"` शामिल करती हैं
- वैकल्पिक `event.derivedPaths`, `apply_patch` जैसे जाने-पहचाने टूल एनवेलप के लिए होस्ट द्वारा सर्वोत्तम प्रयास से निकाले गए लक्ष्य पथ संकेत; ये पथ अधूरे हो सकते हैं या टूल वास्तव में जिन चीज़ों को प्रभावित करेगा उनका अत्यधिक अनुमान लगा सकते हैं (उदाहरण के लिए, विकृत या आंशिक इनपुट के साथ)
- वैकल्पिक `event.runId`
- वैकल्पिक `event.toolCallId`
- संदर्भ फ़ील्ड, जैसे `ctx.agentId`, `ctx.sessionKey`, `ctx.sessionId`, `ctx.runId`, `ctx.toolKind`, `ctx.toolInputKind`, और नैदानिक `ctx.trace`

यह निम्न लौटा सकता है:

```typescript
type BeforeToolCallResult = {
  params?: Record<string, unknown>;
  block?: boolean;
  blockReason?: string;
  requireApproval?: {
    title: string;
    description: string;
    severity?: "info" | "warning" | "critical";
    timeoutMs?: number;
    /** @deprecated अनसुलझी स्वीकृतियाँ हमेशा अस्वीकार होती हैं। */
    timeoutBehavior?: "allow" | "deny";
    allowedDecisions?: Array<"allow-once" | "allow-always" | "deny">;
    pluginId?: string;
    onResolution?: (
      decision: "allow-once" | "allow-always" | "deny" | "timeout" | "cancelled",
    ) => Promise<void> | void;
  };
};
```

टाइप किए गए जीवनचक्र हुक के लिए गार्ड व्यवहार:

- `block: true` अंतिम है और निम्न-प्राथमिकता वाले हैंडलर को छोड़ देता है।
- `block: false` को कोई निर्णय नहीं माना जाता।
- `params` निष्पादन के लिए टूल पैरामीटर पुनर्लिखता है।
- `requireApproval` एजेंट रन को रोकता है और Plugin स्वीकृतियों के माध्यम से उपयोगकर्ता से पूछता है। `/approve` exec और Plugin, दोनों स्वीकृतियों को मंज़ूरी दे सकता है। Codex app-server रिपोर्ट-मोड के मूल `PreToolUse` रिले में, यह मेल खाने वाले app-server स्वीकृति अनुरोध पर निर्भर करता है; [Codex हार्नेस रनटाइम](/hi/plugins/codex-harness-runtime#hook-boundaries) देखें।
- उच्च-प्राथमिकता वाले हुक द्वारा स्वीकृति का अनुरोध किए जाने के बाद भी निम्न-प्राथमिकता वाला `block: true` ब्लॉक कर सकता है।
- `onResolution` को निर्धारित निर्णय प्राप्त होता है: `allow-once`, `allow-always`, `deny`, `timeout`, या `cancelled`।

स्वीकृति रूटिंग, निर्णय व्यवहार, और वैकल्पिक टूल या exec स्वीकृतियों के बजाय `requireApproval` का उपयोग कब करना है, इसके लिए [Plugin अनुमति अनुरोध](/hi/plugins/plugin-permission-requests) देखें।

जिन plugins को होस्ट-स्तरीय नीति चाहिए, वे `api.registerTrustedToolPolicy(...)` के साथ विश्वसनीय टूल नीतियाँ पंजीकृत कर सकते हैं। ये सामान्य `before_tool_call` हुक और सामान्य हुक निर्णयों से पहले चलती हैं। बंडल की गई विश्वसनीय नीतियाँ पहले चलती हैं; इंस्टॉल किए गए Plugin की विश्वसनीय नीतियाँ Plugin-लोड क्रम में उसके बाद चलती हैं; सामान्य `before_tool_call` हुक उनके बाद चलते हैं। बंडल किए गए plugins मौजूदा विश्वसनीय-नीति पथ बनाए रखते हैं। इंस्टॉल किए गए plugins को स्पष्ट रूप से सक्षम किया जाना चाहिए और `contracts.trustedToolPolicies` में प्रत्येक नीति आईडी घोषित करनी चाहिए; अघोषित आईडी पंजीकरण से पहले अस्वीकृत कर दी जाती हैं। नीति आईडी पंजीकरण करने वाले Plugin के दायरे में होती हैं, इसलिए अलग-अलग plugins समान स्थानीय आईडी का पुनः उपयोग कर सकते हैं। इस स्तर का उपयोग केवल होस्ट-विश्वसनीय गेट के लिए करें, जैसे कार्यस्थान नीति, बजट प्रवर्तन, या आरक्षित कार्यप्रवाह सुरक्षा।

### Exec परिवेश हुक

`resolve_exec_env`, कमांड चलने से पहले plugins को `exec` टूल इनवोकेशन में परिवेश चर जोड़ने देता है। इसे प्राप्त होते हैं:

- `event.sessionKey`
- `event.toolName`, वर्तमान में हमेशा `"exec"`
- `event.host`, जो `"gateway"`, `"sandbox"`, या `"node"` में से एक है
- संदर्भ फ़ील्ड, जैसे `ctx.agentId`, `ctx.sessionKey`, `ctx.messageProvider`, और `ctx.channelId`

exec परिवेश में मर्ज करने के लिए `Record<string, string>` लौटाएँ। हैंडलर प्राथमिकता क्रम में चलते हैं; समान कुंजी के लिए बाद के परिणाम पहले के परिणामों को ओवरराइड करते हैं।

मर्ज करने से पहले हुक आउटपुट को होस्ट exec परिवेश कुंजी नीति के माध्यम से फ़िल्टर किया जाता है। `PATH` को हमेशा हटा दिया जाता है (कमांड रिज़ॉल्यूशन और सुरक्षित-बिन जाँच इस पर निर्भर करती हैं)। अमान्य कुंजियाँ और खतरनाक होस्ट ओवरराइड कुंजियाँ, जैसे `LD_*`, `DYLD_*`, `NODE_OPTIONS`, प्रॉक्सी चर (`HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY`, `NO_PROXY`), और TLS ओवरराइड चर (`NODE_TLS_REJECT_UNAUTHORIZED`, `SSL_CERT_FILE`, और समान) हटा दिए जाते हैं। फ़िल्टर किया गया Plugin env Gateway स्वीकृति/ऑडिट मेटाडेटा में शामिल किया जाता है और Node-होस्ट निष्पादन अनुरोधों को अग्रेषित किया जाता है।

### टूल परिणाम स्थायित्व

टूल परिणामों में UI रेंडरिंग, निदान, मीडिया रूटिंग, या Plugin-स्वामित्व वाले मेटाडेटा के लिए संरचित `details` शामिल हो सकता है। `details` को रनटाइम मेटाडेटा मानें, प्रॉम्प्ट सामग्री नहीं:

- OpenClaw प्रदाता रीप्ले और Compaction इनपुट से पहले `toolResult.details` हटा देता है, ताकि मेटाडेटा मॉडल संदर्भ न बने।
- स्थायी सत्र प्रविष्टियाँ केवल सीमित `details` रखती हैं। अत्यधिक बड़े विवरण एक संक्षिप्त सारांश और `persistedDetailsTruncated: true` से बदल दिए जाते हैं।
- `tool_result_persist` और `before_message_write` अंतिम स्थायित्व सीमा से पहले चलते हैं। लौटाए गए `details` को छोटा रखें और प्रॉम्प्ट-संबंधित टेक्स्ट केवल `details` में रखने से बचें; मॉडल को दिखाई देने वाला टूल आउटपुट `content` में रखें।

## प्रॉम्प्ट और मॉडल हुक

नए plugins के लिए चरण-विशिष्ट हुक का उपयोग करें:

- `before_model_resolve`: केवल वर्तमान प्रॉम्प्ट और अटैचमेंट मेटाडेटा प्राप्त करता है। `providerOverride` या `modelOverride` लौटाएँ।
- `agent_turn_prepare`: वर्तमान प्रॉम्प्ट, तैयार किए गए सत्र संदेश, और इस सत्र के लिए निकाले गए ठीक-एक-बार कतारबद्ध इंजेक्शन प्राप्त करता है। `prependContext` या `appendContext` लौटाएँ।
- `before_prompt_build`: वर्तमान प्रॉम्प्ट और सत्र संदेश प्राप्त करता है। `prependContext`, `appendContext`, `systemPrompt`, `prependSystemContext`, या `appendSystemContext` लौटाएँ।
- `heartbeat_prompt_contribution`: केवल Heartbeat टर्न के लिए चलता है और `prependContext` या `appendContext` लौटाता है। यह उन पृष्ठभूमि मॉनिटर के लिए है जिन्हें उपयोगकर्ता द्वारा आरंभ किए गए टर्न बदले बिना वर्तमान स्थिति का सारांश देना होता है।

`before_agent_start` संगतता के लिए बना हुआ है। ऊपर दिए गए स्पष्ट हुक को प्राथमिकता दें, ताकि Plugin किसी पुराने संयुक्त चरण पर निर्भर न हो।

`before_agent_run` प्रॉम्प्ट निर्माण के बाद और किसी भी मॉडल इनपुट से पहले चलता है, जिसमें प्रॉम्प्ट-स्थानीय छवि लोडिंग और `llm_input` अवलोकन शामिल हैं। इसे वर्तमान उपयोगकर्ता इनपुट `prompt` के रूप में, लोड किया गया सत्र इतिहास `messages` में, और सक्रिय सिस्टम प्रॉम्प्ट प्राप्त होता है। मॉडल द्वारा प्रॉम्प्ट पढ़े जाने से पहले रन रोकने के लिए `{ outcome: "block", reason, message? }` लौटाएँ। `reason` आंतरिक है; `message` उपयोगकर्ता-दृश्य प्रतिस्थापन है। केवल `pass` और `block` परिणाम समर्थित हैं; असमर्थित निर्णय आकार सुरक्षित रूप से विफल होते हैं।

जब कोई रन ब्लॉक किया जाता है, तो OpenClaw `message.content` में केवल प्रतिस्थापन टेक्स्ट और गैर-संवेदनशील ब्लॉक मेटाडेटा, जैसे ब्लॉक करने वाले Plugin की आईडी और टाइमस्टैम्प, संग्रहीत करता है। मूल उपयोगकर्ता टेक्स्ट ट्रांसक्रिप्ट या भविष्य के संदर्भ में नहीं रखा जाता। आंतरिक ब्लॉक कारणों को संवेदनशील माना जाता है और ट्रांसक्रिप्ट, इतिहास, प्रसारण, लॉग, और निदान पेलोड से बाहर रखा जाता है। अवलोकनीयता के लिए ब्लॉकर आईडी, परिणाम, टाइमस्टैम्प, या सुरक्षित श्रेणी जैसे स्वच्छ किए गए फ़ील्ड का उपयोग करना चाहिए।

जब OpenClaw सक्रिय रन की पहचान कर सकता है, तब `before_agent_start` और `agent_end` में `event.runId` शामिल होता है; यही मान `ctx.runId` पर भी होता है। Cron द्वारा संचालित रन एजेंट-टर्न संदर्भ पर `ctx.jobId` (आरंभ करने वाली cron जॉब आईडी) भी उजागर करते हैं, ताकि हुक मेट्रिक्स, दुष्प्रभाव, या स्थिति को किसी विशिष्ट शेड्यूल किए गए जॉब तक सीमित कर सकें। `ctx.jobId`, `before_tool_call` टूल संदर्भ का भाग नहीं है।

चैनल से शुरू किए गए रन के लिए, `ctx.channel` और `ctx.messageProvider`
`discord` या `telegram` जैसी प्रदाता सतह की पहचान करते हैं, जबकि `ctx.channelId`
वार्तालाप लक्ष्य पहचानकर्ता है, जब OpenClaw सत्र कुंजी या डिलीवरी मेटाडेटा से इसे
निकाल सकता है।

जब प्रेषक की पहचान उपलब्ध होती है, तब एजेंट हुक संदर्भों में ये भी शामिल होते हैं:

- `ctx.senderId` - चैनल-स्कोप वाला प्रेषक ID (जैसे Feishu `open_id`, Discord
  उपयोगकर्ता ID)। यह तब भरा जाता है, जब रन ज्ञात प्रेषक मेटाडेटा वाले
  उपयोगकर्ता संदेश से शुरू होता है।
- `ctx.chatId` - ट्रांसपोर्ट-मूल वार्तालाप पहचानकर्ता (जैसे Feishu
  `chat_id`, Telegram `chat_id`)। यह तब भरा जाता है, जब आरंभिक चैनल
  कोई मूल वार्तालाप ID प्रदान करता है।
- `ctx.channelContext.sender.id` - `ctx.senderId` के समान प्रेषक ID, एक
  चैनल-स्वामित्व वाले ऑब्जेक्ट के अंतर्गत, जिसे plugins चैनल-विशिष्ट फ़ील्ड से विस्तारित कर सकते हैं।
- `ctx.channelContext.chat.id` - `ctx.chatId` के समान वार्तालाप ID,
  एक चैनल-स्वामित्व वाले ऑब्जेक्ट के अंतर्गत, जिसे plugins चैनल-विशिष्ट
  फ़ील्ड से विस्तारित कर सकते हैं।

कोर केवल नेस्टेड `id` फ़ील्ड परिभाषित करता है। इनबाउंड हेल्पर के माध्यम से अधिक समृद्ध
प्रेषक या चैट मेटाडेटा भेजने वाले चैनल plugins
`openclaw/plugin-sdk/channel-inbound` से `PluginHookChannelSenderContext` या `PluginHookChannelChatContext` को
विस्तारित कर सकते हैं:

```ts
declare module "openclaw/plugin-sdk/channel-inbound" {
  interface PluginHookChannelSenderContext {
    unionId?: string;
    userId?: string;
  }
}
```

चैनल plugins उन फ़ील्ड को इनबाउंड SDK हेल्पर के माध्यम से भेजते हैं:

```ts
buildChannelInboundEventContext({
  // ...
  channelContext: {
    sender: { id: senderOpenId, unionId, userId },
    chat: { id: chatId },
  },
});
```

ये फ़ील्ड वैकल्पिक हैं और सिस्टम से शुरू किए गए रन (heartbeat,
cron, exec-event) में अनुपस्थित रहते हैं।

`ctx.senderExternalId` पुराने plugins के लिए पदावनत स्रोत-संगतता फ़ील्ड के रूप में
बना हुआ है। कोर इसे नहीं भरता; नई चैनल-विशिष्ट प्रेषक
पहचानें मॉड्यूल विस्तार के माध्यम से `ctx.channelContext.sender` के अंतर्गत
होनी चाहिए।

`agent_end` एक अवलोकन हुक है। Gateway और स्थायी हार्नेस पथ
इसे टर्न के बाद बिना प्रतीक्षा किए चलाते हैं, जबकि अल्पकालिक एकल-प्रयोग CLI पथ
प्रक्रिया सफ़ाई से पहले हुक प्रॉमिस की प्रतीक्षा करते हैं, ताकि विश्वसनीय plugins
टर्मिनल प्रेक्षणीयता को फ़्लश कर सकें या स्थिति कैप्चर कर सकें। हुक रनर 30 सेकंड की
समय-सीमा लागू करता है, ताकि अटका हुआ Plugin या एम्बेडिंग एंडपॉइंट हुक प्रॉमिस को
हमेशा के लिए लंबित न छोड़ सके। समय-सीमा समाप्त होने पर लॉग दर्ज किया जाता है और OpenClaw जारी रहता है;
यह Plugin के स्वामित्व वाले नेटवर्क कार्य को रद्द नहीं करता, जब तक कि Plugin अपना
अबॉर्ट सिग्नल भी उपयोग न करे।

ऐसी प्रदाता-कॉल टेलीमेट्री के लिए `model_call_started` और `model_call_ended` का उपयोग करें,
जिसे अपरिष्कृत प्रॉम्प्ट, इतिहास, प्रतिक्रियाएँ, हेडर, अनुरोध
बॉडी या प्रदाता अनुरोध ID प्राप्त नहीं होने चाहिए। इन हुक में
`runId`, `callId`, `provider`, `model`, वैकल्पिक `api`/`transport`, अंतिम
`durationMs`/`outcome`, और `upstreamRequestIdHash` जैसे स्थिर मेटाडेटा शामिल होते हैं, जब OpenClaw
सीमित प्रदाता अनुरोध-ID हैश निकाल सकता है। जब रनटाइम ने
संदर्भ-विंडो मेटाडेटा हल कर लिया हो, तब हुक इवेंट और संदर्भ में
`contextTokenBudget`, मॉडल/कॉन्फ़िगरेशन/एजेंट
सीमाओं के बाद प्रभावी टोकन बजट, साथ ही कम सीमा लागू होने पर
`contextWindowSource` और `contextWindowReferenceTokens` भी शामिल होते हैं।

`before_agent_finalize` केवल तब चलता है, जब कोई हार्नेस स्वाभाविक
अंतिम सहायक उत्तर स्वीकार करने वाला हो। यह `/stop` रद्दीकरण पथ नहीं है और
उपयोगकर्ता द्वारा किसी टर्न को निरस्त करने पर नहीं चलता। अंतिम रूप देने से पहले
हार्नेस से मॉडल का एक और पास माँगने के लिए `{ action: "revise", reason }` लौटाएँ,
अंतिम रूप देना बाध्य करने के लिए `{ action:
"finalize", reason? }` लौटाएँ, या जारी रखने के लिए परिणाम छोड़ दें।
हैंडलर का डिफ़ॉल्ट बजट 15s है; समय-सीमा समाप्त होने पर OpenClaw विफलता लॉग करता है और
मूल अंतिम उत्तर के साथ जारी रहता है।
Codex के मूल `Stop` हुक इस हुक में OpenClaw
`before_agent_finalize` निर्णयों के रूप में रिले किए जाते हैं।

`action: "revise"` लौटाते समय, plugins अतिरिक्त मॉडल पास को
सीमित और रीप्ले-सुरक्षित बनाने के लिए `retry` मेटाडेटा शामिल कर सकते हैं:

```typescript
type BeforeAgentFinalizeRetry = {
  instruction: string;
  idempotencyKey?: string;
  maxAttempts?: number;
};
```

`instruction` को हार्नेस को भेजे गए संशोधन कारण में जोड़ा जाता है।
`idempotencyKey` होस्ट को समतुल्य अंतिमीकरण निर्णयों में उसी Plugin अनुरोध के लिए
पुनः प्रयासों की गणना करने देता है, और `maxAttempts` यह सीमित करता है कि स्वाभाविक अंतिम उत्तर के साथ
जारी रखने से पहले होस्ट कितने अतिरिक्त पास की अनुमति देगा।

जिन गैर-बंडल plugins को अपरिष्कृत वार्तालाप हुक (`before_model_resolve`,
`before_agent_reply`, `llm_input`, `llm_output`, `before_agent_finalize`,
`agent_end`, या `before_agent_run`) चाहिए, उन्हें यह सेट करना होगा:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "hooks": {
          "allowConversationAccess": true
        }
      }
    }
  }
}
```

प्रॉम्प्ट बदलने वाले हुक और स्थायी अगले-टर्न इंजेक्शन को प्रति
Plugin `plugins.entries.<id>.hooks.allowPromptInjection=false` से अक्षम किया जा सकता है।

### सत्र विस्तार और अगले-टर्न इंजेक्शन

वर्कफ़्लो plugins `api.session.state.registerSessionExtension(...)` के साथ छोटी JSON-संगत सत्र स्थिति को
स्थायी रख सकते हैं और Gateway `sessions.pluginPatch` विधि के माध्यम से
इसे अपडेट कर सकते हैं। सत्र पंक्तियाँ पंजीकृत विस्तार स्थिति को
`pluginExtensions` के माध्यम से प्रक्षेपित करती हैं, जिससे Control UI और अन्य
क्लाइंट Plugin के आंतरिक विवरण जाने बिना Plugin-स्वामित्व वाली स्थिति रेंडर कर सकते हैं।
`api.registerSessionExtension(...)` अब भी काम करता है, लेकिन
`api.session.state` नेमस्पेस के पक्ष में पदावनत है।

जब किसी Plugin को अगले मॉडल टर्न तक ठीक एक बार
स्थायी संदर्भ पहुँचाने की आवश्यकता हो, तब `api.session.workflow.enqueueNextTurnInjection(...)` का उपयोग करें
(शीर्ष-स्तरीय `api.enqueueNextTurnInjection(...)` समान
व्यवहार वाला पदावनत उपनाम है)। OpenClaw प्रॉम्प्ट हुक से पहले कतारबद्ध इंजेक्शन निकालता है,
समाप्त हो चुके इंजेक्शन हटा देता है और प्रति Plugin `idempotencyKey` के अनुसार डुप्लिकेट हटाता है। यह
अनुमोदन बहाली, नीति सारांश, पृष्ठभूमि मॉनिटर
अंतर और कमांड निरंतरताओं के लिए सही सीम है, जिन्हें अगले टर्न पर मॉडल को दिखाई देना चाहिए,
लेकिन स्थायी सिस्टम प्रॉम्प्ट टेक्स्ट नहीं बनना चाहिए।

सफ़ाई का व्यवहार अनुबंध का हिस्सा है। सत्र विस्तार सफ़ाई और
रनटाइम जीवनचक्र सफ़ाई कॉलबैक `reset`, `delete`, `disable`, या
`restart` प्राप्त करते हैं। रीसेट/हटाने/अक्षम करने पर होस्ट स्वामी Plugin की स्थायी सत्र विस्तार
स्थिति और लंबित अगले-टर्न इंजेक्शन हटा देता है; पुनः आरंभ
स्थायी सत्र स्थिति बनाए रखता है, जबकि सफ़ाई कॉलबैक plugins को पुराने
रनटाइम जनरेशन के लिए शेड्यूलर कार्य, रन संदर्भ और अन्य आउट-ऑफ़-बैंड संसाधन
जारी करने देते हैं।

## संदेश हुक

चैनल-स्तरीय रूटिंग और डिलीवरी नीति के लिए संदेश हुक का उपयोग करें:

- `message_received`: इनबाउंड सामग्री, प्रेषक, `threadId`,
  `messageId`, `senderId`, वैकल्पिक रन/सत्र सहसंबंध और मेटाडेटा का अवलोकन करें।
- `message_sending`: `content` को फिर से लिखें या `{ cancel: true }` लौटाएँ।
- `reply_payload_sending`: सामान्यीकृत `ReplyPayload` ऑब्जेक्ट
  (`presentation`, `delivery`, मीडिया संदर्भ और टेक्स्ट सहित) को फिर से लिखें या
  `{ cancel: true }` लौटाएँ।
- `message_sent`: अंतिम सफलता या विफलता का अवलोकन करें।

केवल-ऑडियो TTS उत्तरों के लिए, चैनल पेलोड में कोई दृश्य टेक्स्ट/कैप्शन न होने पर भी
`content` में छिपा हुआ वाचित
प्रतिलेख हो सकता है।
उस `content` को फिर से लिखने पर केवल हुक को दिखाई देने वाला प्रतिलेख अपडेट होता है; इसे
मीडिया कैप्शन के रूप में रेंडर नहीं किया जाता।

`reply_payload_sending` इवेंट में `usageState`, हर टर्न का सर्वोत्तम-प्रयास वाला लाइव
मॉडल/उपयोग/संदर्भ स्नैपशॉट शामिल हो सकता है। स्थायी डिलीवरी, पुनर्प्राप्त रीप्ले और
सटीक रन सहसंबंध के बिना उत्तरों में यह शामिल नहीं होता।

उपलब्ध होने पर संदेश हुक संदर्भ स्थिर सहसंबंध फ़ील्ड उजागर करते हैं:
`ctx.sessionKey`, `ctx.runId`, `ctx.messageId`, `ctx.senderId`, `ctx.trace`,
`ctx.traceId`, `ctx.spanId`, `ctx.parentSpanId`, और `ctx.callDepth`। इनबाउंड
और `before_dispatch` संदर्भ उत्तर मेटाडेटा भी उजागर करते हैं, जब चैनल में
दृश्यता-फ़िल्टर किया गया उद्धृत संदेश डेटा हो: `replyToId`, `replyToIdFull`,
`replyToBody`, `replyToSender`, और `replyToIsQuote`। विरासत मेटाडेटा पढ़ने से पहले इन
प्रथम-श्रेणी फ़ील्ड को प्राथमिकता दें।

चैनल-विशिष्ट मेटाडेटा का उपयोग करने से पहले टाइप किए गए `threadId` और `replyToId` फ़ील्ड को प्राथमिकता दें।

निर्णय नियम:

- `message_sending` के साथ `cancel: true` अंतिम है।
- `message_sending` के साथ `cancel: false` को कोई निर्णय नहीं माना जाता।
- फिर से लिखा गया `content` निम्न-प्राथमिकता वाले हुक तक जारी रहता है, जब तक कोई बाद का हुक
  डिलीवरी रद्द न कर दे।
- `reply_payload_sending` पेलोड सामान्यीकरण के बाद और चैनल
  डिलीवरी से पहले चलता है, जिसमें मूल चैनल पर वापस भेजे गए उत्तर भी शामिल हैं।
  हैंडलर क्रमिक रूप से चलते हैं और प्रत्येक हैंडलर उच्च-प्राथमिकता वाले हैंडलर द्वारा बनाया गया नवीनतम पेलोड
  देखता है।
- `reply_payload_sending` पेलोड `trustedLocalMedia` जैसे रनटाइम विश्वास चिह्न उजागर नहीं करते;
  plugins पेलोड का आकार संपादित कर सकते हैं, लेकिन स्थानीय
  मीडिया विश्वास प्रदान नहीं कर सकते।
- `message_sending` रद्दीकरण के साथ `cancelReason` और सीमित `metadata` लौटा सकता है।
  नए संदेश जीवनचक्र API इसे `cancelled_by_message_sending_hook` कारण वाले दबाए गए
  डिलीवरी परिणाम के रूप में उजागर करते हैं; विरासत
  प्रत्यक्ष डिलीवरी संगतता के लिए खाली परिणाम सरणी लौटाती रहती है।
- `message_sent` केवल अवलोकन के लिए है। हैंडलर विफलताएँ लॉग की जाती हैं और
  डिलीवरी परिणाम नहीं बदलतीं।

## इंस्टॉल हुक

ऑपरेटर-स्वामित्व वाले अनुमति/अवरोध निर्णयों के लिए `security.installPolicy` का उपयोग करें। वह
नीति OpenClaw कॉन्फ़िगरेशन से चलती है, CLI इंस्टॉल और अपडेट पथों को कवर करती है और
सक्षम लेकिन अनुपलब्ध होने पर सुरक्षित रूप से अवरोधित करती है।

`before_install` एक Plugin-रनटाइम जीवनचक्र हुक है। यह
`security.installPolicy` के बाद केवल उस OpenClaw प्रक्रिया में चलता है, जहाँ Plugin हुक पहले ही
लोड हो चुके हों, जैसे Gateway-समर्थित इंस्टॉल प्रवाह। यह
Plugin-स्वामित्व वाले अवलोकनों, चेतावनियों और संगतता जाँचों के लिए उपयोगी है, लेकिन इंस्टॉल के लिए
प्राथमिक एंटरप्राइज़ या होस्ट सुरक्षा सीमा नहीं है। संगतता के लिए
`builtinScan` फ़ील्ड इवेंट पेलोड में बनी रहती है, लेकिन
OpenClaw अब अंतर्निहित इंस्टॉल-समय खतरनाक-कोड अवरोधन नहीं चलाता, इसलिए यह एक खाली
`ok` परिणाम है। उस प्रक्रिया में इंस्टॉल रोकने के लिए अतिरिक्त निष्कर्ष या
`{ block: true, blockReason }` लौटाएँ।

`block: true` अंतिम है। `block: false` को कोई निर्णय नहीं माना जाता। हैंडलर
विफलताएँ इंस्टॉल को सुरक्षित रूप से अवरोधित करती हैं।

## Gateway जीवनचक्र

सामान्य Plugin सेवाएँ शुरू करने के लिए `gateway_start` और
लंबे समय तक चलने वाले संसाधनों की सफ़ाई के लिए `gateway_stop` का उपयोग करें। जब
`gateway_start` चलता है, तब भी Cron शेड्यूलर लोड हो रहा हो सकता है, इसलिए इसे बाहरी
Cron प्रक्षेपण के लिए आधारभूत संकेत के रूप में उपयोग न करें।

Plugin-स्वामित्व वाली रनटाइम सेवाओं के लिए आंतरिक `gateway:startup` हुक पर
निर्भर न रहें।

`cron_reconciled` तब सक्रिय होता है, जब Gateway Cron शेड्यूलर और उसके निकास-समय
वॉचर अपनी स्थायी स्थिति का मिलान कर चुके होते हैं। यह प्रारंभिक
स्टार्टअप और कॉन्फ़िगरेशन रीलोड के दौरान शेड्यूलर प्रतिस्थापन, दोनों के लिए सक्रिय होता है। इवेंट
`reason` (`startup` या `reload`) और प्रभावी `enabled` स्थिति रिपोर्ट करता है। अक्षम
Cron भी `enabled: false` के साथ इवेंट उत्सर्जित करता है, जिससे कोई बाहरी प्रक्षेपण
पुराने वेक साफ़ कर सकता है। मिलान पूरा करने वाले सटीक शेड्यूलर इंस्टेंस के लिए
`ctx.getCron?.()` का उपयोग करें; बाद का रीलोड उस कॉलबैक को पुनः लक्षित नहीं करता।
`ctx.abortSignal` उसी शेड्यूलर स्नैपशॉट का स्वामी है। जैसे ही नया शेड्यूलर सक्रिय होता है
या शटडाउन शुरू होता है, Gateway इसे निरस्त कर देता है। इसे प्रत्येक
स्थायी दुष्प्रभाव में आगे भेजें और निरस्त होने के बाद स्नैपशॉट स्वीकार न करें।
यह शेड्यूलर जीवनचक्र संकेत है, Plugin-सक्रियण संकेत नहीं:
केवल-Plugin हॉट रीलोड इसे फिर से नहीं चलाता। नए सक्षम किए गए उपभोक्ता को
अगले शेड्यूलर प्रतिस्थापन या Gateway प्रारंभ पर उसका पहला आधारभूत मान मिलता है।

अन्य अवलोकन हुक की तरह, `gateway_start` और `cron_reconciled` कॉलबैक
एक-दूसरे के साथ ओवरलैप हो सकते हैं। यदि दोनों हैंडलर Plugin आरंभीकरण साझा करते हैं, तो कॉलबैक क्रम पर निर्भर रहने के बजाय
उन्हें Plugin-स्थानीय तत्परता प्रॉमिस के साथ समन्वित करें।

`cron_changed` Gateway के स्वामित्व वाले Cron जीवनचक्र इवेंट के लिए सक्रिय होता है, जिसमें `added`, `updated`, `removed`, `started`, `finished`,
और `scheduled` कारणों को समेटने वाला टाइप किया हुआ इवेंट पेलोड होता है। इवेंट में एक `PluginHookGatewayCronJob`
स्नैपशॉट (उपलब्ध होने पर `state.nextRunAtMs`, `state.lastRunStatus`, और
`state.lastError` सहित) के साथ `not-requested` | `delivered` | `not-delivered` | `unknown` में से एक `PluginHookGatewayCronDeliveryStatus`
होता है। हटाए गए इवेंट पोस्ट-कमिट होते हैं: वे केवल टिकाऊ विलोपन सफल होने के बाद सक्रिय होते हैं और हटाए गए जॉब का स्नैपशॉट फिर भी रखते हैं, ताकि बाहरी शेड्यूलर स्थिति का मिलान कर सकें।

एक `scheduled` इवेंट पोस्ट-कमिट होता है: यह केवल तभी सक्रिय होता है, जब कोई सफल टिकाऊ
लेखन किसी मौजूदा जॉब के प्रभावी `nextRunAtMs` को बदलता है; इसमें उस जॉब के स्पष्ट
`added`, `updated`, या `removed` जीवनचक्र इवेंट शामिल नहीं होते। शीर्ष-स्तरीय
`event.nextRunAtMs` कमिट किया गया अगला वेक है; इसके अनुपस्थित होने पर जॉब का
कोई अगला वेक नहीं होता। इन इवेंट को क्रमबद्ध डेल्टा लॉग नहीं, बल्कि मिलान संकेत
मानें। इन्हें संयोजित किए जा सकने वाले संकेतों के रूप में उपयोग करके
`cron_reconciled` द्वारा अंतिम बार कैप्चर किए गए शेड्यूलर को दोबारा पढ़ें; किसी `cron_changed` संदर्भ से शेड्यूलर न अपनाएँ।
नियत समय की जाँच और निष्पादन के लिए OpenClaw को सत्य का स्रोत बनाए रखें।

### सुरक्षित बाहरी Cron प्रक्षेपण

Cron इवेंट डेल्टा अग्रेषित करने के बजाय एक पूर्ण वेक स्नैपशॉट प्रक्षेपित करें।
बाहरी अडैप्टर का `replaceAll` ऑपरेशन परमाण्विक और आइडेम्पोटेंट होना चाहिए तथा इसे
केवल होस्ट द्वारा स्नैपशॉट को टिकाऊ रूप से स्वीकार किए जाने के बाद ही पूर्ण होना चाहिए। इसे
दिए गए एबॉर्ट सिग्नल का भी पालन करना चाहिए: यदि टिकाऊ स्वीकृति से पहले सिग्नल एबॉर्ट हो जाता है,
तो अडैप्टर को वह स्नैपशॉट स्वीकार नहीं करना चाहिए।

यह पैटर्न नवीनतम स्थिति वाला एक ही वर्कर सक्रिय रखता है। केवल `cron_reconciled`
शेड्यूलर इंस्टेंस अपनाता है; `cron_changed` केवल उस वर्कर से
प्रामाणिक इंस्टेंस दोबारा पढ़ने का अनुरोध करता है, इसलिए देर से आया संकेत किसी पुराने शेड्यूलर को पुनर्स्थापित नहीं कर सकता।
कोई नया संशोधन सक्रिय होस्ट प्रयास को पुराने स्नैपशॉट को स्वीकार करने से पहले एबॉर्ट कर देता है।

```typescript
import { setTimeout as sleep } from "node:timers/promises";
import type { OpenClawPluginApi } from "openclaw/plugin-sdk/plugin-entry";

type ExternalWake = { jobId: string; runAtMs: number };

type ExternalWakeHost = {
  replaceAll(wakes: readonly ExternalWake[], options: { signal: AbortSignal }): Promise<void>;
  close(): Promise<void>;
};

type CronReader = {
  list(options: { includeDisabled: true }): Promise<
    Array<{
      id: string;
      enabled?: boolean;
      state?: { nextRunAtMs?: number };
    }>
  >;
};

export function registerCronProjection(api: OpenClawPluginApi, host: ExternalWakeHost) {
  const lifecycle = new AbortController();
  let cron: CronReader | undefined;
  let enabled = false;
  let hasBaseline = false;
  let reconciliationSignal: AbortSignal | undefined;
  let requestedRevision = 0;
  let appliedRevision = 0;
  let worker = Promise.resolve();
  let activeAttempt: AbortController | undefined;

  const projectLatest = async () => {
    let retryMs = 1_000;

    while (!lifecycle.signal.aborted && appliedRevision < requestedRevision) {
      const ownerSignal = reconciliationSignal;
      if (!ownerSignal || ownerSignal.aborted) {
        return;
      }
      const targetRevision = requestedRevision;
      const attempt = new AbortController();
      const signal = AbortSignal.any([lifecycle.signal, ownerSignal, attempt.signal]);
      activeAttempt = attempt;

      try {
        const jobs = enabled && cron ? await cron.list({ includeDisabled: true }) : [];
        if (signal.aborted || targetRevision !== requestedRevision) {
          continue;
        }
        const wakes = jobs
          .flatMap((job): ExternalWake[] => {
            const runAtMs = job.enabled === false ? undefined : job.state?.nextRunAtMs;
            return runAtMs === undefined ? [] : [{ jobId: job.id, runAtMs }];
          })
          .sort((a, b) => a.runAtMs - b.runAtMs || a.jobId.localeCompare(b.jobId));

        await host.replaceAll(wakes, { signal });
        if (signal.aborted || targetRevision !== requestedRevision) {
          continue;
        }
        appliedRevision = targetRevision;
        retryMs = 1_000;
      } catch {
        if (lifecycle.signal.aborted || ownerSignal.aborted) {
          return;
        }
        if (attempt.signal.aborted) {
          continue;
        }
        api.logger.warn(`external cron projection failed; retrying in ${retryMs}ms`);
        try {
          await sleep(retryMs, undefined, { signal });
        } catch {
          if (lifecycle.signal.aborted) {
            return;
          }
          if (attempt.signal.aborted) {
            continue;
          }
        }
        retryMs = Math.min(retryMs * 2, 30_000);
      } finally {
        if (activeAttempt === attempt) {
          activeAttempt = undefined;
        }
      }
    }
  };

  const requestProjection = () => {
    const targetRevision = ++requestedRevision;
    activeAttempt?.abort();
    worker = worker.then(async () => {
      if (!lifecycle.signal.aborted && appliedRevision < targetRevision) {
        await projectLatest();
      }
    });
    return worker;
  };

  api.on("cron_reconciled", (event, ctx) => {
    const reconciledCron = ctx.getCron?.();
    if (event.enabled && !reconciledCron) {
      api.logger.warn("cron reconciliation did not expose a scheduler");
      return;
    }
    cron = reconciledCron;
    enabled = event.enabled;
    hasBaseline = true;
    reconciliationSignal = ctx.abortSignal;
    return requestProjection();
  });

  api.on("cron_changed", () => {
    if (hasBaseline) {
      return requestProjection();
    }
  });

  api.on("gateway_stop", async () => {
    lifecycle.abort();
    await worker;
    await host.close();
  });
}
```

जब `cron_reconciled`, `enabled: false` की रिपोर्ट करता है, तो वही पथ
`replaceAll([])` को कॉल करता है और पुराने बाहरी वेक साफ़ करता है। इस उदाहरण में पुनः प्रयास/बैकऑफ़
प्रक्रिया-स्थानीय है और रनटाइम अडैप्टर की विफलताओं को अस्थायी मानता है; पंजीकरण से पहले
पुनः प्रयास न की जा सकने वाली कॉन्फ़िगरेशन को सत्यापित करें। OpenClaw, Plugin हुक प्रभावों के लिए
आउटबॉक्स प्रदान नहीं करता। यदि टिकाऊ स्वीकृति से पहले प्रक्रिया बंद हो जाती है,
तो अगला Gateway प्रारंभ एक नया प्रामाणिक `cron_reconciled` स्नैपशॉट उत्सर्जित करता है।
`gateway_stop` चल रहे होस्ट कार्य को एबॉर्ट करता है, वर्कर के स्थिर होने की प्रतीक्षा करता है, फिर
अडैप्टर को बंद करता है।

## आगामी अप्रचलन

हुक से संबंधित कुछ सतहें अप्रचलित हैं, लेकिन अभी भी समर्थित हैं। अगले प्रमुख रिलीज़ से
पहले माइग्रेट करें:

- `inbound_claim` और `message_received`
  हैंडलर में **प्लेनटेक्स्ट चैनल एनवेलप**। सपाट एनवेलप टेक्स्ट को पार्स करने के बजाय
  `BodyForAgent` और संरचित उपयोगकर्ता-संदर्भ ब्लॉक पढ़ें। देखें
  [प्लेनटेक्स्ट चैनल एनवेलप → BodyForAgent](/hi/plugins/sdk-migration#active-deprecations)।
- **`before_agent_start`** संगतता के लिए बना हुआ है। नए Plugin को संयुक्त
  चरण के बजाय `before_model_resolve` और `before_prompt_build` का उपयोग करना चाहिए।
- **`subagent_spawning`** पुराने Plugin के साथ संगतता के लिए बना हुआ है, लेकिन
  नए Plugin को इससे थ्रेड रूटिंग वापस नहीं करनी चाहिए। कोर, `subagent_spawned` के सक्रिय होने से
  पहले चैनल सेशन-बाइंडिंग अडैप्टर के माध्यम से `thread: true` सबएजेंट बाइंडिंग तैयार करता है।
- **`deactivate`** 2026-08-16 के बाद तक अप्रचलित क्लीनअप संगतता उपनाम के रूप में
  बना हुआ है। नए Plugin को `gateway_stop` का उपयोग करना चाहिए।
- **`before_tool_call` में `onResolution`** अब मुक्त-रूप `string` के बजाय
  टाइप किए हुए `PluginApprovalResolution` यूनियन (`allow-once` / `allow-always` / `deny` /
  `timeout` / `cancelled`) का उपयोग करता है।
- **`api.registerSessionExtension` / `api.enqueueNextTurnInjection`** शीर्ष-स्तरीय
  संगतता उपनामों के रूप में बने हुए हैं। नए Plugin को
  `api.session.state.registerSessionExtension(...)` और
  `api.session.workflow.enqueueNextTurnInjection(...)` का उपयोग करना चाहिए।

पूरी सूची—मेमोरी क्षमता पंजीकरण, प्रदाता थिंकिंग प्रोफ़ाइल, बाहरी प्रमाणीकरण प्रदाता,
प्रदाता खोज प्रकार, टास्क रनटाइम एक्सेसर, और `command-auth` → `command-status`
नाम-परिवर्तन—के लिए देखें
[Plugin SDK माइग्रेशन → सक्रिय अप्रचलन](/hi/plugins/sdk-migration#active-deprecations)।
