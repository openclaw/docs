आधिकारिक `codex` Plugin, अंतर्निहित OpenClaw हार्नेस के बजाय Codex
app-server के माध्यम से एम्बेड किए गए OpenAI एजेंट टर्न चलाता है। निम्न-स्तरीय
एजेंट सत्र का स्वामित्व Codex के पास है: नेटिव थ्रेड पुनरारंभ, नेटिव टूल निरंतरता,
नेटिव Compaction और app-server निष्पादन। चैट चैनल, सत्र फ़ाइलें, मॉडल चयन,
OpenClaw डायनेमिक टूल, अनुमोदन, मीडिया डिलीवरी और दृश्यमान ट्रांसक्रिप्ट मिरर का
स्वामित्व अभी भी OpenClaw के पास है।

`openai/gpt-5.6-sol` जैसे कैनोनिकल OpenAI मॉडल रेफ़रेंस का उपयोग करें। पुराने
Codex GPT रेफ़रेंस कॉन्फ़िगर न करें; OpenAI एजेंट प्रमाणीकरण क्रम को
`auth.order.openai` के अंतर्गत रखें। पुराने Codex प्रमाणीकरण प्रोफ़ाइल आईडी और
पुरानी Codex प्रमाणीकरण क्रम प्रविष्टियों की मरम्मत `openclaw doctor --fix` द्वारा
की जाती है।

जब प्रदाता/मॉडल रनटाइम नीति सेट न हो या `auto` हो, तो अकेला `openai/*` प्रीफ़िक्स
इस हार्नेस को कभी नहीं चुनता। OpenAI केवल ऐसी सटीक आधिकारिक HTTPS Platform
Responses या ChatGPT Responses रूट के लिए Codex को अप्रत्यक्ष रूप से चुन सकता
है, जिसमें कोई उपयोगकर्ता-निर्धारित अनुरोध ओवरराइड न हो। देखें
[OpenAI अप्रत्यक्ष एजेंट रनटाइम](/hi/providers/openai#implicit-agent-runtime)।
यदि Platform बनाम ChatGPT रूटिंग ज्ञात होने से पहले प्रमाणीकरण का स्वामित्व
Codex के पास है, तब भी OpenClaw प्रत्येक संभावित रूट से Codex संगतता घोषित
करना आवश्यक बनाता है। केवल नेटिव प्रमाणीकरण स्वामित्व उस रूट जाँच को कभी
बायपास नहीं करता।

जब कोई OpenClaw सैंडबॉक्स सक्रिय नहीं होता, तो OpenClaw Codex app-server थ्रेड
को Codex नेटिव कोड मोड सक्षम करके शुरू करता है (डिफ़ॉल्ट रूप से केवल-कोड-मोड
बंद रहता है), ताकि नेटिव वर्कस्पेस/कोड क्षमताएँ app-server के `item/tool/call`
ब्रिज के माध्यम से रूट किए गए OpenClaw डायनेमिक टूल के साथ उपलब्ध रहें। सक्रिय
OpenClaw सैंडबॉक्स या प्रतिबंधित टूल नीति नेटिव कोड मोड को पूरी तरह अक्षम कर
देती है, जब तक कि आप प्रयोगात्मक सैंडबॉक्स exec-server पथ को न चुनें।

डिफ़ॉल्ट `tools.exec.host: "auto"` और कोई सक्रिय OpenClaw सैंडबॉक्स न होने पर,
Codex को युग्मित Node पर कमांड के लिए `node_exec` और `node_process` टूल भी मिलते हैं।
नेटिव शेल Codex app-server होस्ट और वर्कस्पेस पर रहता है (डिफ़ॉल्ट stdio
परिनियोजन के लिए Gateway-स्थानीय); `node_exec` नाम या आईडी से किसी Node को चुनता है
और OpenClaw की Node अनुमोदन नीति को लागू रखता है। यदि कोई सीमित रनटाइम अनुमतिसूची
नेटिव कोड मोड को अक्षम कर देती है और टर्न के पास कोई निष्पादन परिवेश नहीं बचता,
तो OpenClaw इसके बजाय सीधे, बिना सैंडबॉक्स वाले निष्पादन के लिए अपने नीति-फ़िल्टर
किए गए `exec` और `process` टूल उपलब्ध रखता है।

यह Codex-नेटिव सुविधा [OpenClaw कोड मोड](/hi/tools/code-mode) से अलग है, जो
सामान्य OpenClaw रन के लिए एक वैकल्पिक QuickJS-WASI रनटाइम है और जिसकी
`exec` इनपुट संरचना अलग है। व्यापक मॉडल/प्रदाता/रनटाइम विभाजन के लिए
[एजेंट रनटाइम](/hi/concepts/agent-runtimes) से शुरू करें: `openai/gpt-5.6-sol` मॉडल
रेफ़रेंस है, `codex` रनटाइम है और Telegram, Discord, Slack या कोई अन्य
चैनल संचार सतह है।

## आवश्यकताएँ

- आधिकारिक `@openclaw/codex` Plugin इंस्टॉल हो। यदि आपका कॉन्फ़िगरेशन अनुमतिसूची का उपयोग करता है, तो
  `plugins.allow` में `codex` शामिल करें।
- `0.143.0` से `0.144.6` तक का एक स्थिर Codex app-server। Plugin डिफ़ॉल्ट रूप से एक संगत
  बाइनरी प्रबंधित करता है, इसलिए `PATH` पर मौजूद `codex` कमांड सामान्य
  स्टार्टअप को प्रभावित नहीं करता।
- `openclaw models auth login --provider openai` के माध्यम से Codex प्रमाणीकरण, एजेंट के Codex होम में
  पहले से मौजूद app-server खाता या स्पष्ट Codex API-कुंजी प्रमाणीकरण प्रोफ़ाइल।

प्रमाणीकरण प्राथमिकता, परिवेश पृथक्करण, कस्टम app-server कमांड, मॉडल खोज और
कॉन्फ़िगरेशन फ़ील्ड की पूरी सूची के लिए
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference) देखें।

## त्वरित शुरुआत

आधिकारिक Plugin इंस्टॉल करें, फिर Codex OAuth से साइन इन करें:

```bash
openclaw plugins install @openclaw/codex
openclaw models auth login --provider openai
```

`codex` Plugin सक्षम करें और कोई OpenAI एजेंट मॉडल चुनें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

यदि आपका कॉन्फ़िगरेशन `plugins.allow` का उपयोग करता है, तो उसमें `codex` भी जोड़ें:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Plugin कॉन्फ़िगरेशन बदलने के बाद Gateway पुनः आरंभ करें। यदि किसी चैट में पहले
से सत्र है, तो पहले `/new` या `/reset` चलाएँ, ताकि अगला टर्न वर्तमान
कॉन्फ़िगरेशन से हार्नेस निर्धारित करे।

## Codex Desktop और CLI के साथ थ्रेड साझा करें

डिफ़ॉल्ट `appServer.homeScope: "agent"` प्रत्येक OpenClaw एजेंट को ऑपरेटर की
नेटिव Codex स्थिति से अलग रखता है। किसी स्वामी को Codex Desktop और Codex CLI
में दिखाए गए उन्हीं नेटिव थ्रेड का निरीक्षण और प्रबंधन करने देने के लिए,
उपयोगकर्ता Codex होम को चुनें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            homeScope: "user",
          },
        },
      },
    },
  },
}
```

उपयोगकर्ता-होम मोड स्थानीय प्रबंधित stdio प्रक्रिया या साझा Unix-socket
ट्रांसपोर्ट का समर्थन करता है। सेट होने पर यह `$CODEX_HOME` और अन्यथा
`~/.codex` का उपयोग करता है, जिसमें उस होम का नेटिव Codex प्रमाणीकरण,
कॉन्फ़िगरेशन, plugins और थ्रेड स्टोर शामिल हैं। OpenClaw इस app-server में
OpenClaw प्रमाणीकरण प्रोफ़ाइल इंजेक्ट नहीं करता।

स्वामी के टर्न को `codex_threads` टूल मिलता है: नेटिव थ्रेड को सूचीबद्ध करना,
खोजना, पढ़ना, फ़ोर्क करना, नाम बदलना, संग्रहित करना और पुनर्स्थापित करना।
किसी थ्रेड को OpenClaw में जारी रखने के लिए उसे फ़ोर्क करें; फ़ोर्क वर्तमान
OpenClaw सत्र से जुड़ता है और अन्य नेटिव Codex क्लाइंट को दिखाई देता रहता है।
संग्रहित करने के लिए स्पष्ट पुष्टि आवश्यक है कि थ्रेड कहीं और बंद है। जब
पर्यवेक्षण भी सक्षम हो, तो ट्रांसक्रिप्ट फ़ील्ड और परिवर्तनों के लिए संबंधित
`supervision.allowRawTranscripts` या `supervision.allowWriteControls` विकल्प चुनना आवश्यक है।

अलग-अलग प्रबंधित stdio App Servers के माध्यम से एक ही थ्रेड को समवर्ती रूप से
पुनरारंभ न करें या उसमें न लिखें। Codex एक App Server के भीतर सक्रिय लेखकों
का समन्वय करता है, अलग-अलग प्रक्रियाओं के बीच नहीं। सामान्य उपयोगकर्ता-होम
stdio सत्रों के लिए फ़ोर्क करना सुरक्षित सह-अस्तित्व पथ है।

अकेला `appServer.homeScope: "user"` फ़्लीट कैटलॉग को नियंत्रित नहीं करता। Plugin सक्रिय
रहने के दौरान नेटिव सत्र खोज सक्षम रहती है; Codex को अक्षम किए बिना उसे
OpenClaw साइडबार से हटाने के लिए `sessionCatalog.enabled: false` सेट करें। कैटलॉग एक अलग
पर्यवेक्षण कनेक्शन का उपयोग करता है; स्पष्ट `appServer` कनेक्शन सेटिंग न
होने पर, वह कनेक्शन डिफ़ॉल्ट रूप से प्रबंधित उपयोगकर्ता-होम stdio का उपयोग
करता है, जबकि सामान्य हार्नेस एजेंट-स्कोप्ड रहता है। स्पष्ट `appServer`
सेटिंग दोनों पथों द्वारा स्वीकार की जाती हैं। जब सामान्य हार्नेस को भी नेटिव
स्थिति साझा करनी हो, तो ऊपर की तरह `homeScope: "user"` स्पष्ट रूप से सेट करें।

## Codex सत्रों का पर्यवेक्षण करें

वही `codex` Plugin Gateway कंप्यूटर और विकल्प चुनकर शामिल किए गए
युग्मित Node से गैर-संग्रहित Codex सत्रों को सूचीबद्ध कर सकता है। कोई संग्रहीत
या निष्क्रिय Gateway-स्थानीय सत्र मॉडल-लॉक्ड चैट बना सकता है, जो उसके सीमित,
स्थायी उपयोगकर्ता और सहायक इतिहास को मिरर करता है। इसका निजी बाइंडिंग नेटिव
स्नैपशॉट, कैनोनिकल शाखा और बाद के टर्न के लिए पर्यवेक्षण कनेक्शन का उपयोग करता
है, जबकि सामान्य Codex सत्र एजेंट-स्कोप्ड रहते हैं। पहला कैनोनिकल प्रारंभ ठीक
उसी मॉडल और प्रदाता का उपयोग करता है जिसे Codex स्नैपशॉट फ़ोर्क के लिए लौटाता
है। बाद के पुनरारंभ चयन को Codex के नेटिव कॉन्फ़िगरेशन पर छोड़ देते हैं; बाहरी
OpenClaw मॉडल और फ़ॉलबैक श्रृंखला उसे कभी प्रतिस्थापित नहीं करते। स्पष्ट
कोई-अन्य-रनर-नहीं पुष्टि के बाद संग्रहीत और निष्क्रिय पंक्तियाँ संग्रहित की जा
सकती हैं। सक्रिय स्रोत न तो शाखा बना सकते हैं, न संग्रहित किए जा सकते हैं;
मौजूदा पर्यवेक्षित चैट फिर भी खोला जा सकता है। युग्मित-Node सत्र केवल
मेटाडेटा तक सीमित रहते हैं।

सेटअप, शाखाकरण नियमों, युग्मित-Node सीमाओं, मेटाडेटा प्रदर्शन और समस्या निवारण
के लिए [Codex सत्रों का पर्यवेक्षण करें](/hi/plugins/codex-supervision) देखें।

| Codex Computer Use सक्षम करें                       | `plugins.entries.codex.config.computerUse.*`                                                     | Codex Plugin कॉन्फ़िगरेशन          |

सदस्यता-प्रथम/API-कुंजी-बैकअप क्रम के लिए `auth.order.openai` को प्राथमिकता दें।
मौजूदा पुराने Codex प्रमाणीकरण प्रोफ़ाइल आईडी और पुराना Codex प्रमाणीकरण क्रम
केवल doctor द्वारा संभाली जाने वाली पुरानी स्थिति हैं; नए पुराने Codex GPT
रेफ़रेंस न लिखें।

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

Codex-संगत प्रभावी रूट के लिए, ऊपर की दोनों प्रोफ़ाइल एक ही Codex रन की
संभावित प्रोफ़ाइल बनी रहती हैं। प्रोफ़ाइल क्रम क्रेडेंशियल चुनता है, रनटाइम नहीं।
प्रमाणीकरण क्रम बदलने से कोई कस्टम, Completions, HTTP या अनुरोध-ओवरराइड वाला
रूट Codex-संगत नहीं बनता।

### Compaction

Codex-समर्थित एजेंट पर `compaction.model` या `compaction.provider` सेट न करें।
Codex अपनी नेटिव app-server थ्रेड स्थिति के माध्यम से Compaction करता है,
इसलिए OpenClaw रनटाइम पर उन स्थानीय सारांशकर्ता ओवरराइड को अनदेखा करता है और
एजेंट द्वारा Codex का उपयोग किए जाने पर `openclaw doctor --fix` उन्हें हटा देता है।

Codex टर्न के आसपास संयोजन, अंतर्ग्रहण और रखरखाव के लिए Lossless संदर्भ इंजन
के रूप में समर्थित रहता है। इसे `agents.defaults.compaction.provider` के माध्यम से नहीं, बल्कि
`plugins.slots.contextEngine: "lossless-claw"` और `plugins.entries.lossless-claw.config.summaryModel` के माध्यम से कॉन्फ़िगर किया जाता है।
Codex के सक्रिय रनटाइम होने पर `openclaw doctor --fix` पुराने `compaction.provider: "lossless-claw"`
आकार को Lossless संदर्भ-इंजन स्लॉट में माइग्रेट करता है, लेकिन Compaction का
स्वामित्व फिर भी नेटिव Codex के पास रहता है। नेटिव app-server हार्नेस उन
संदर्भ इंजनों का समर्थन करता है जिन्हें प्रॉम्प्ट-पूर्व संयोजन की आवश्यकता
होती है; `codex-cli` सहित सामान्य CLI बैकएंड वह होस्ट क्षमता प्रदान
नहीं करते।

Codex-समर्थित एजेंट के लिए, `/compact` बाउंड थ्रेड पर नेटिव Codex
app-server Compaction शुरू करता है। OpenClaw इसके पूरा होने की प्रतीक्षा नहीं
करता, OpenClaw टाइमआउट लागू नहीं करता, साझा app-server पुनः आरंभ नहीं करता और
किसी संदर्भ-इंजन या सार्वजनिक OpenAI सारांशकर्ता पर फ़ॉलबैक नहीं करता। यदि
नेटिव Codex थ्रेड बाइंडिंग अनुपस्थित या पुराना है, तो कमांड Compaction बैकएंड
को चुपचाप बदलने के बजाय बंद स्थिति में विफल होता है।

इस पृष्ठ का शेष भाग परिनियोजन संरचना, विफलता पर बंद होने वाली रूटिंग, Guardian
अनुमोदन नीति, नेटिव Codex plugins और कंप्यूटर उपयोग को कवर करता है। विकल्पों की पूरी
सूचियों, डिफ़ॉल्ट, enums, खोज, पर्यावरण पृथक्करण, टाइमआउट और
app-server ट्रांसपोर्ट फ़ील्ड के लिए
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference) देखें।

## Codex रनटाइम सत्यापित करें

उस चैट में `/status` का उपयोग करें जहाँ आप Codex की अपेक्षा करते हैं। Codex द्वारा समर्थित OpenAI
एजेंट टर्न यह दिखाता है:

```text
रनटाइम: OpenAI Codex
```

फिर Codex app-server की स्थिति जाँचें:

```text
/codex status
/codex models
```

`/codex status` app-server कनेक्टिविटी, अकाउंट, दर सीमाएँ, MCP
सर्वर और कौशल की रिपोर्ट करता है। `/codex models` हार्नेस और अकाउंट के लिए लाइव Codex app-server कैटलॉग
सूचीबद्ध करता है। यदि `/status` अप्रत्याशित है, तो
[समस्या निवारण](#troubleshooting) देखें।

## रूटिंग और मॉडल चयन

प्रदाता संदर्भों और रनटाइम नीति को अलग रखें:

- प्रामाणिक OpenAI मॉडल चयन के लिए `openai/gpt-*` का उपयोग करें। केवल उपसर्ग
  कभी भी Codex का चयन नहीं करता।
- रनटाइम अनिर्धारित या `auto` होने पर, केवल बिना किसी लिखित अनुरोध ओवरराइड वाला सटीक आधिकारिक HTTPS Platform Responses
  या ChatGPT Responses रूट ही Codex का
  अप्रत्यक्ष रूप से चयन कर सकता है।
- कॉन्फ़िगरेशन में पुराने Codex GPT संदर्भों का उपयोग न करें; पुराने संदर्भों और अप्रचलित सत्र रूट पिनों को
  सुधारने के लिए `openclaw doctor --fix` चलाएँ।
- `agentRuntime.id: "codex"` किसी संगत रूट के लिए Codex को विफलता पर बंद होने वाली आवश्यकता बनाता है।
  यह किसी असंगत प्रभावी रूट को संगत नहीं बनाता।
- `agentRuntime.id: "openclaw"` किसी प्रदाता या मॉडल को एम्बेडेड
  OpenClaw रनटाइम में शामिल करता है, जब ऐसा करना अभिप्रेत हो।
- `/codex ...` चैट से नेटिव Codex app-server वार्तालाप नियंत्रित करता है।
- ACP/acpx एक अलग बाहरी हार्नेस पथ है। इसका उपयोग केवल तब करें जब उपयोगकर्ता
  ACP/acpx या बाहरी हार्नेस अडैप्टर माँगे।

## परिनियोजन प्रतिरूप

### मूल Codex परिनियोजन

ऐसे OpenAI मॉडल के लिए क्विकस्टार्ट कॉन्फ़िगरेशन का उपयोग करें जिसका प्रभावी आधिकारिक HTTPS
रूट Codex का अप्रत्यक्ष रूप से चयन करने के योग्य हो:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

### मिश्रित प्रदाता परिनियोजन

Claude को डिफ़ॉल्ट एजेंट बनाए रखें और नामित Codex एजेंट जोड़ें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-6",
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.6-sol",
      },
    ],
  },
}
```

`main` एजेंट अपने सामान्य प्रदाता पथ का उपयोग करता है। `codex` एजेंट Codex
app-server का उपयोग करता है, जब उसका प्रभावी OpenAI रूट संगत रहता है; यदि इसे विफलता पर बंद होने वाली
आवश्यकता बनाना हो, तो स्पष्ट मॉडल-स्कोप वाला `agentRuntime.id: "codex"` जोड़ें।

### विफलता पर बंद होने वाला Codex परिनियोजन

बंडल किया गया plugin उपलब्ध होने पर कोई पात्र सटीक आधिकारिक HTTPS OpenAI रूट Codex में रिज़ॉल्व हो सकता है।
लिखित विफलता पर बंद होने वाले नियम के लिए स्पष्ट रनटाइम नीति जोड़ें:

```json5
{
  models: {
    providers: {
      openai: {
        agentRuntime: {
          id: "codex",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Codex बाध्य होने पर, यदि प्रभावी रूट Codex-संगत घोषित नहीं है,
plugin अक्षम है, app-server बहुत पुराना है या
app-server शुरू नहीं हो सकता, तो OpenClaw जल्दी विफल हो जाता है।

## App-server नीति

डिफ़ॉल्ट रूप से, plugin stdio ट्रांसपोर्ट के साथ OpenClaw की प्रबंधित Codex बाइनरी स्थानीय रूप से शुरू करता है।
केवल जानबूझकर कोई अलग निष्पादन योग्य फ़ाइल चलाने के लिए `appServer.command` सेट करें।
Codex WebSocket ट्रांसपोर्ट को प्रयोगात्मक और असमर्थित वर्गीकृत करता है;
इसका उपयोग केवल कहीं और पहले से चल रहे app-server के विरुद्ध गैर-उत्पादन परीक्षण के लिए करें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
          },
        },
      },
    },
  },
}
```

स्थानीय stdio app-server सत्र डिफ़ॉल्ट रूप से विश्वसनीय स्थानीय ऑपरेटर
रुख का उपयोग करते हैं: `approvalPolicy: "never"`, `approvalsReviewer: "user"` और
`sandbox: "danger-full-access"`। यदि स्थानीय Codex आवश्यकताएँ उस
अप्रत्यक्ष YOLO रुख की अनुमति नहीं देतीं, तो OpenClaw इसके बजाय अनुमत Guardian अनुमतियाँ चुनता है।
जब सत्र के लिए OpenClaw सैंडबॉक्स सक्रिय हो, तो OpenClaw
Codex के होस्ट-पक्षीय सैंडबॉक्स पर निर्भर रहने के बजाय उस टर्न के लिए Codex नेटिव Code Mode,
उपयोगकर्ता MCP सर्वर और ऐप-समर्थित plugin निष्पादन अक्षम कर देता है।
इसके बजाय शेल पहुँच OpenClaw सैंडबॉक्स-समर्थित डायनेमिक टूलों, जैसे
`sandbox_exec` और `sandbox_process`, के माध्यम से जाती है, जब सामान्य exec/process टूल
उपलब्ध हों।

सैंडबॉक्स से बाहर निकलने या अतिरिक्त अनुमतियाँ देने से पहले Codex नेटिव स्वतः-समीक्षा के लिए
सामान्यीकृत OpenClaw exec मोड का उपयोग करें:

```json5
{
  tools: {
    exec: {
      mode: "auto",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Codex app-server सत्रों के लिए, `tools.exec.mode: "auto"` Codex
Guardian-समीक्षित अनुमोदनों से मैप होता है: स्थानीय आवश्यकताएँ इन मानों की अनुमति दें तो सामान्यतः
`approvalPolicy: "on-request"`, `approvalsReviewer: "auto_review"` और `sandbox: "workspace-write"`।
`tools.exec.mode: "auto"` में,
OpenClaw पुराने असुरक्षित Codex `approvalPolicy: "never"` या
`sandbox: "danger-full-access"` ओवरराइड सुरक्षित नहीं रखता; जानबूझकर बिना-अनुमोदन वाले Codex रुख के लिए
`tools.exec.mode: "full"` का उपयोग करें। पुराना
`plugins.entries.codex.config.appServer.mode: "guardian"` प्रीसेट अब भी
काम करता है, लेकिन `tools.exec.mode: "auto"` सामान्यीकृत OpenClaw सतह है।

होस्ट exec अनुमोदनों और ACPX अनुमतियों के साथ मोड-स्तरीय तुलना के लिए
[अनुमति मोड](/hi/tools/permission-modes) देखें। प्रत्येक
app-server फ़ील्ड, प्रमाणीकरण क्रम, पर्यावरण पृथक्करण और टाइमआउट व्यवहार के लिए
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference) देखें।

## कमांड और निदान

`codex` plugin, OpenClaw टेक्स्ट कमांड का समर्थन करने वाले किसी भी चैनल पर `/codex` को स्लैश कमांड के रूप में पंजीकृत करता है।

नेटिव निष्पादन और नियंत्रण के लिए स्वामी या `operator.admin`
Gateway क्लाइंट आवश्यक है: थ्रेड आबद्ध करना या फिर से शुरू करना, टर्न भेजना या रोकना,
मॉडल, तेज़-मोड या अनुमति स्थिति बदलना, Compaction या समीक्षा करना और
आबद्धता अलग करना। अन्य अधिकृत प्रेषकों के पास केवल-पढ़ने वाली स्थिति, सहायता,
अकाउंट, मॉडल, थ्रेड, नेटिव लक्ष्य, MCP सर्वर, कौशल और आबद्धता निरीक्षण
कमांड रहते हैं।

सामान्य रूप:

- `/codex status` ऐप-सर्वर कनेक्टिविटी, मॉडल, खाता, दर
  सीमाएँ, MCP सर्वर और कौशल जाँचता है।
- `/codex models` सक्रिय Codex ऐप-सर्वर मॉडल सूचीबद्ध करता है।
- `/codex threads [filter]` हाल के Codex ऐप-सर्वर थ्रेड सूचीबद्ध करता है।
- `/codex goal` संलग्न थ्रेड के मूल Codex लक्ष्य को पढ़ता या अपडेट करता है। Codex का स्वचालित लक्ष्य जारी रखना अक्षम रहता है; OpenClaw अभी स्वायत्त अनुवर्ती टर्न का स्वामी नहीं है।
- `/codex resume <thread-id>` वर्तमान OpenClaw सत्र को किसी
  मौजूदा Codex थ्रेड से संलग्न करता है।
- `/codex bind [thread-id] [--cwd <path>] [--model <model>] [--provider <provider>]`
  वर्तमान चैट को संलग्न करता है।
- `/codex detach` (या `/codex unbind`) वर्तमान बाइंडिंग को अलग करता है।
- `/codex binding` वर्तमान बाइंडिंग का वर्णन करता है।
- `/codex stop` सक्रिय टर्न को रोकता है; `/codex steer <text>` उसका मार्गदर्शन करता है।
- `/codex model <model>`, `/codex fast [on|off|status]`, और
  `/codex permissions [default|yolo|status]` प्रत्येक वार्तालाप की स्थिति बदलते हैं।
- `/codex compact` Codex ऐप-सर्वर से संलग्न थ्रेड को संक्षिप्त करने का अनुरोध करता है।
- `/codex review` संलग्न थ्रेड के लिए Codex की मूल समीक्षा शुरू करता है।
- `/codex diagnostics [note]` संलग्न थ्रेड के लिए Codex फ़ीडबैक भेजने से
  पहले पूछता है।
- `/codex account` खाते और दर-सीमा की स्थिति दिखाता है।
- `/codex mcp` Codex ऐप-सर्वर के MCP सर्वर की स्थिति सूचीबद्ध करता है।
- `/codex skills` Codex ऐप-सर्वर के कौशल सूचीबद्ध करता है।
- `/codex plugins list`, `/codex plugins enable <name>`, और
  `/codex plugins disable <name>` कॉन्फ़िगर किए गए मूल Codex Plugin प्रबंधित करते हैं।
- `/codex computer-use [status|install]` Codex Computer Use प्रबंधित करता है।
- `/codex help` संपूर्ण कमांड ट्री सूचीबद्ध करता है।

अधिकांश सहायता रिपोर्ट के लिए, उस वार्तालाप में `/diagnostics [note]` से शुरू करें
जहाँ बग हुआ था। यह एक Gateway निदान रिपोर्ट बनाता है और Codex हार्नेस
सत्रों के लिए संबंधित Codex फ़ीडबैक बंडल भेजने की स्वीकृति माँगता है।
गोपनीयता मॉडल और समूह चैट व्यवहार के लिए
[निदान निर्यात](/hi/gateway/diagnostics) देखें। `/codex diagnostics [note]` का उपयोग केवल तभी करें जब आप
विशेष रूप से पूर्ण Gateway निदान बंडल के बिना वर्तमान में संलग्न थ्रेड के लिए
Codex फ़ीडबैक अपलोड करना चाहते हों।

### Codex थ्रेड का स्थानीय रूप से निरीक्षण करें

किसी खराब Codex रन का निरीक्षण करने का सबसे तेज़ तरीका अक्सर मूल
Codex थ्रेड को सीधे खोलना होता है:

```bash
codex resume <thread-id>
```

पूर्ण हो चुके `/diagnostics` उत्तर, `/codex binding`,
या `/codex threads [filter]` से थ्रेड आईडी प्राप्त करें।

अपलोड की कार्यप्रणाली और रनटाइम-स्तरीय निदान सीमाओं के लिए,
[Codex हार्नेस रनटाइम](/hi/plugins/codex-harness-runtime#codex-feedback-upload) देखें।

### प्रमाणीकरण क्रम

प्रत्येक एजेंट के डिफ़ॉल्ट होम में, प्रमाणीकरण इस क्रम में चुना जाता है:

1. एजेंट के लिए क्रमबद्ध OpenAI प्रमाणीकरण प्रोफ़ाइल, बेहतर है कि
   `auth.order.openai` के अंतर्गत हों। पुराने विरासत Codex प्रमाणीकरण प्रोफ़ाइल आईडी और
   विरासत Codex प्रमाणीकरण क्रम को माइग्रेट करने के लिए `openclaw doctor --fix` चलाएँ।
2. उस एजेंट के Codex होम में ऐप-सर्वर का मौजूदा खाता।
3. केवल स्थानीय stdio ऐप-सर्वर लॉन्च के लिए, `CODEX_API_KEY`, फिर
   `OPENAI_API_KEY`, जब कोई ऐप-सर्वर खाता मौजूद न हो और OpenAI प्रमाणीकरण
   अभी भी आवश्यक हो।

जब OpenClaw को ChatGPT सदस्यता-शैली की Codex प्रमाणीकरण प्रोफ़ाइल दिखाई देती है, तो वह
आरंभ की गई Codex चाइल्ड प्रक्रिया से `CODEX_API_KEY` और `OPENAI_API_KEY` हटा देता है।
इससे Gateway-स्तरीय API कुंजियाँ एम्बेडिंग या प्रत्यक्ष OpenAI मॉडल के लिए उपलब्ध रहती हैं,
और मूल Codex ऐप-सर्वर टर्न से अनजाने में API के माध्यम से बिल नहीं लिया जाता।
स्पष्ट Codex API-कुंजी प्रोफ़ाइल और स्थानीय stdio एनवायरनमेंट-कुंजी फ़ॉलबैक,
इनहेरिट किए गए चाइल्ड-प्रोसेस एनवायरनमेंट के बजाय ऐप-सर्वर लॉगिन का उपयोग करते हैं।
WebSocket ऐप-सर्वर कनेक्शन को Gateway एनवायरनमेंट API-कुंजी फ़ॉलबैक नहीं मिलता;
किसी स्पष्ट प्रमाणीकरण प्रोफ़ाइल या रिमोट ऐप-सर्वर के अपने खाते का उपयोग करें।

यदि कोई सदस्यता प्रोफ़ाइल Codex उपयोग सीमा तक पहुँचती है, तो Codex द्वारा रीसेट समय
रिपोर्ट किए जाने पर OpenClaw उसे दर्ज करता है और उसी Codex रन के लिए अगली
क्रमबद्ध प्रमाणीकरण प्रोफ़ाइल आज़माता है। रीसेट समय बीतने पर, चयनित
`openai/gpt-*` मॉडल या Codex रनटाइम बदले बिना सदस्यता प्रोफ़ाइल फिर से पात्र हो जाती है।

जब मूल Codex Plugin कॉन्फ़िगर किए जाते हैं, तो OpenClaw Plugin के स्वामित्व वाले
ऐप को Codex थ्रेड के समक्ष प्रस्तुत करने से पहले कनेक्टेड ऐप-सर्वर के माध्यम से
उन Plugin को इंस्टॉल या रीफ़्रेश करता है। ऐप आईडी, अभिगम्यता और मेटाडेटा के लिए
`app/list` सत्य का स्रोत बना रहता है, लेकिन प्रत्येक थ्रेड के लिए सक्षम करने के
निर्णय का स्वामी OpenClaw है: यदि नीति किसी सूचीबद्ध सुलभ ऐप की अनुमति देती है, तो
`app/list` द्वारा वर्तमान में उस ऐप को अक्षम बताए जाने पर भी OpenClaw
`thread/start.config.apps[appId].enabled = true` भेजता है। यह पथ अज्ञात आईडी के लिए ऐप इंस्टॉलेशन नहीं गढ़ता;
OpenClaw केवल `plugin/install` वाले मार्केटप्लेस Plugin सक्रिय करता है और फिर
इन्वेंटरी रीफ़्रेश करता है।

### एनवायरनमेंट पृथक्करण

स्थानीय stdio ऐप-सर्वर लॉन्च के लिए, OpenClaw `CODEX_HOME` को प्रत्येक एजेंट
की डायरेक्टरी पर सेट करता है, ताकि Codex कॉन्फ़िगरेशन, प्रमाणीकरण/खाता फ़ाइलें,
Plugin कैश/डेटा और मूल थ्रेड स्थिति डिफ़ॉल्ट रूप से ऑपरेटर के व्यक्तिगत
`~/.codex` को पढ़ें या उसमें लिखें नहीं। OpenClaw सामान्य प्रक्रिया
`HOME` को संरक्षित रखता है; Codex-रन उपप्रक्रियाएँ अब भी उपयोगकर्ता-होम
कॉन्फ़िगरेशन और टोकन ढूँढ सकती हैं, और Codex साझा `$HOME/.agents/skills` तथा
`$HOME/.agents/plugins/marketplace.json` प्रविष्टियाँ खोज सकता है।
`appServer.homeScope: "user"` के साथ, OpenClaw इसके बजाय मूल उपयोगकर्ता Codex होम और उसके
मौजूदा खाते का उपयोग करता है तथा OpenClaw प्रमाणीकरण प्रोफ़ाइल इंजेक्ट नहीं करता।

यदि किसी परिनियोजन को अतिरिक्त एनवायरनमेंट पृथक्करण की आवश्यकता हो, तो उन
वेरिएबल को `appServer.clearEnv` में जोड़ें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` केवल आरंभ की गई Codex ऐप-सर्वर चाइल्ड प्रक्रिया को
प्रभावित करता है। स्थानीय लॉन्च सामान्यीकरण के दौरान OpenClaw इस सूची से
`CODEX_HOME` और `HOME` हटा देता है: `CODEX_HOME` चयनित
एजेंट या उपयोगकर्ता स्कोप की ओर इंगित रहता है, और `HOME` इनहेरिट किया
रहता है ताकि उपप्रक्रियाएँ सामान्य उपयोगकर्ता-होम स्थिति का उपयोग कर सकें।

### डायनेमिक टूल और वेब खोज

Codex डायनेमिक टूल डिफ़ॉल्ट रूप से `searchable` लोडिंग का उपयोग करते हैं।
OpenClaw सामान्यतः उन डायनेमिक टूल को प्रस्तुत नहीं करता जो Codex के मूल
वर्कस्पेस संचालन की नकल करते हैं:
`read`, `write`, `edit`, `apply_patch`, `exec`, `process`, `update_plan`,
`get_goal`, `create_goal`, `update_goal`, `tool_call`, `tool_describe`,
`tool_search`, और `tool_search_code`। लक्ष्य संचालन Codex के मूल भाग बने रहते हैं,
इसलिए OpenClaw Codex टर्न में दूसरा लक्ष्य स्टोर प्रक्षेपित नहीं करता। अधिकांश
शेष OpenClaw एकीकरण टूल, जैसे मैसेजिंग, मीडिया, Cron, ब्राउज़र, नोड, Gateway और
`heartbeat_respond`, `openclaw` नेमस्पेस के अंतर्गत Codex टूल खोज के माध्यम से
उपलब्ध हैं, जिससे प्रारंभिक मॉडल संदर्भ छोटा रहता है। सीमित-टर्न शेल फ़ॉलबैक,
`exec` और `process` के लिए अपवाद है जब कोई सीमित अनुमति-सूची
मूल Code Mode को अक्षम करती है; रनटाइम अनुमति-सूचियाँ और `codexDynamicToolsExclude`
फिर भी लागू होते हैं।

`catalogMode: "direct-only"` चिह्नित टूल, जिनमें OpenClaw का `computer`
टूल शामिल है, इसके बजाय `openclaw_direct` नेमस्पेस का उपयोग करते हैं। Codex उस
नेमस्पेस को `DirectModelOnly` मानता है, इसलिए वे टूल नेस्टेड Code Mode
`tools.*` कॉल से गुज़रे बिना सामान्य और केवल-कोड-मोड थ्रेड में सीधे
मॉडल को दिखाई देते रहते हैं।

खोज सक्षम होने और कोई प्रबंधित प्रदाता चयनित न होने पर वेब खोज डिफ़ॉल्ट रूप से
Codex के होस्ट किए गए `web_search` टूल का उपयोग करती है। मूल होस्ट की गई खोज
और OpenClaw का प्रबंधित `web_search` डायनेमिक टूल परस्पर अनन्य हैं, ताकि
प्रबंधित खोज मूल डोमेन प्रतिबंधों को बायपास न कर सके। होस्ट की गई खोज अनुपलब्ध,
स्पष्ट रूप से अक्षम या किसी चयनित प्रबंधित प्रदाता द्वारा प्रतिस्थापित होने पर
OpenClaw प्रबंधित टूल का उपयोग करता है। OpenClaw Codex के स्वतंत्र
`web.run` एक्सटेंशन को अक्षम रखता है, क्योंकि प्रोडक्शन ऐप-सर्वर ट्रैफ़िक
उसके उपयोगकर्ता-परिभाषित `web` नेमस्पेस को अस्वीकार करता है।
`tools.web.search.enabled: false` दोनों पथ अक्षम करता है, और टूल-अक्षम केवल-LLM रन भी ऐसा ही करते हैं।
Codex `"cached"` को एक वरीयता मानता है और अप्रतिबंधित ऐप-सर्वर टर्न के लिए
उसे सक्रिय बाहरी पहुँच में बदलता है। मूल `allowedDomains` सेट होने पर स्वचालित
प्रबंधित फ़ॉलबैक बंद स्थिति में विफल होता है, ताकि अनुमति-सूची को बायपास न किया जा सके।
स्थायी प्रभावी खोज-नीति परिवर्तन अगले टर्न से पहले बाउंड Codex थ्रेड को बदलते हैं;
प्रत्येक टर्न के अस्थायी प्रतिबंध एक अस्थायी सीमित थ्रेड का उपयोग करते हैं और बाद में
फिर से शुरू करने के लिए मौजूदा बाइंडिंग संरक्षित रखते हैं।

`sessions_yield` और केवल-मैसेज-टूल स्रोत उत्तर सीधे रहते हैं, क्योंकि वे
टर्न-नियंत्रण अनुबंध हैं। `sessions_spawn` खोजने योग्य रहता है ताकि Codex का मूल
`spawn_agent` प्राथमिक Codex उप-एजेंट सतह बना रहे, जबकि स्पष्ट OpenClaw या ACP
प्रत्यायोजन अब भी `openclaw` डायनेमिक टूल नेमस्पेस के माध्यम से उपलब्ध है।
Heartbeat सहयोग निर्देश Codex को Heartbeat टर्न समाप्त करने से पहले
`heartbeat_respond` खोजने के लिए कहते हैं, जब टूल पहले से लोड न हो।

`codexDynamicToolsLoading: "direct"` केवल तभी सेट करें जब आप किसी ऐसे कस्टम
Codex ऐप-सर्वर से कनेक्ट कर रहे हों जो स्थगित डायनेमिक टूल खोज नहीं सकता, या
संपूर्ण टूल पेलोड को डीबग कर रहे हों।

`appServer.networkProxy` स्पष्ट है क्योंकि यह Codex sandbox
अनुबंध को बदलता है। सक्षम होने पर, OpenClaw Codex थ्रेड कॉन्फ़िगरेशन में
`features.network_proxy.enabled` और `default_permissions` भी सेट करता है, ताकि जनरेट की गई
अनुमति प्रोफ़ाइल Codex-प्रबंधित नेटवर्किंग शुरू कर सके। डिफ़ॉल्ट रूप से, OpenClaw
प्रोफ़ाइल बॉडी से टकराव-प्रतिरोधी `openclaw-network-<fingerprint>` प्रोफ़ाइल
नाम जनरेट करता है; `profileName` का उपयोग केवल तब करें जब एक स्थिर स्थानीय नाम
आवश्यक हो।

```json5
{
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            sandbox: "workspace-write",
            networkProxy: {
              enabled: true,
              domains: {
                "api.openai.com": "allow",
                "blocked.example.com": "deny",
              },
              unixSockets: {
                "/tmp/proxy.sock": "allow",
                "/tmp/blocked.sock": "none",
              },
              allowUpstreamProxy: true,
              proxyUrl: "http://127.0.0.1:3128",
            },
          },
        },
      },
    },
  },
}
```

यदि सामान्य app-server रनटाइम `danger-full-access` होता, तो
`networkProxy` सक्षम करने पर जनरेट की गई अनुमति प्रोफ़ाइल के लिए
वर्कस्पेस-शैली फ़ाइल सिस्टम पहुँच का उपयोग होता है: Codex-प्रबंधित नेटवर्क प्रवर्तन
sandbox की गई नेटवर्किंग है, इसलिए पूर्ण-पहुँच प्रोफ़ाइल आउटबाउंड ट्रैफ़िक की
सुरक्षा नहीं करेगी। डोमेन प्रविष्टियाँ `allow` या `deny` का उपयोग करती हैं;
Unix सॉकेट प्रविष्टियाँ Codex के `allow` या `none` मानों का उपयोग करती हैं।

### डायनेमिक टूल कॉल टाइमआउट

OpenClaw के स्वामित्व वाली डायनेमिक टूल कॉल
`appServer.requestTimeoutMs` से स्वतंत्र रूप से सीमित होती हैं: Codex `item/tool/call` अनुरोध
डिफ़ॉल्ट रूप से 90 सेकंड के OpenClaw वॉचडॉग का उपयोग करते हैं। प्रति-कॉल धनात्मक
`timeoutMs` आर्ग्युमेंट उस विशिष्ट टूल बजट को बढ़ाता या घटाता है, जिसकी
अधिकतम सीमा 600000 ms है। जब टूल कॉल अपना टाइमआउट प्रदान नहीं करती, तो
`image_generate` टूल `agents.defaults.imageGenerationModel.timeoutMs` का उपयोग करता है,
अन्यथा इमेज जनरेशन के लिए डिफ़ॉल्ट 120 सेकंड का उपयोग करता है।
मीडिया-अंडरस्टैंडिंग `image` टूल
`tools.media.image.timeoutSeconds` या अपने 60 सेकंड के मीडिया डिफ़ॉल्ट का उपयोग करता है;
इमेज अंडरस्टैंडिंग के लिए यह टाइमआउट स्वयं अनुरोध पर लागू होता है और पहले किए गए
तैयारी कार्य से कम नहीं होता। टाइमआउट होने पर, जहाँ समर्थित हो वहाँ OpenClaw टूल
सिग्नल को निरस्त करता है और Codex को विफल डायनेमिक-टूल प्रतिक्रिया लौटाता है,
ताकि सत्र को `processing` में छोड़ने के बजाय टर्न जारी रह सके।
यह वॉचडॉग बाहरी डायनेमिक `item/tool/call` बजट है; प्रदाता-विशिष्ट
अनुरोध टाइमआउट उस कॉल के भीतर चलते हैं और अपने टाइमआउट अर्थविज्ञान को बनाए रखते हैं।

Codex द्वारा टर्न स्वीकार किए जाने के बाद, और OpenClaw द्वारा टर्न-स्कोप वाले
app-server अनुरोध का उत्तर दिए जाने के बाद, हार्नेस अपेक्षा करता है कि Codex
वर्तमान टर्न में प्रगति करे और अंततः नेटिव टर्न को `turn/completed` के साथ
समाप्त करे। यदि app-server `appServer.turnCompletionIdleTimeoutMs` तक निष्क्रिय रहता है, तो OpenClaw
सर्वोत्तम प्रयास के आधार पर Codex टर्न को बाधित करता है, डायग्नोस्टिक टाइमआउट दर्ज
करता है और OpenClaw सत्र लेन को रिलीज़ करता है, ताकि आगे के चैट संदेश किसी पुराने
नेटिव टर्न के पीछे कतारबद्ध न हों। उसी टर्न की अधिकांश गैर-टर्मिनल सूचनाएँ उस
छोटे वॉचडॉग को निष्क्रिय कर देती हैं, क्योंकि Codex यह प्रमाणित कर चुका होता है कि
टर्न अब भी सक्रिय है।

टूल हैंडऑफ़ अधिक लंबे पोस्ट-टूल निष्क्रिय बजट का उपयोग करते हैं: OpenClaw द्वारा
`item/tool/call` प्रतिक्रिया लौटाने के बाद, `commandExecution` जैसे नेटिव
टूल आइटम पूरे होने के बाद, कच्चे `custom_tool_call_output` पूर्ण होने के बाद, और
पोस्ट-टूल कच्ची सहायक प्रगति, कच्चे रीजनिंग पूर्ण होने या रीजनिंग प्रगति के बाद।
कॉन्फ़िगर होने पर गार्ड `appServer.postToolRawAssistantCompletionIdleTimeoutMs` का उपयोग करता है और अन्यथा
डिफ़ॉल्ट रूप से पाँच मिनट का उपयोग करता है; यही बजट उस मौन संश्लेषण अवधि के लिए
प्रगति वॉचडॉग को भी बढ़ाता है, जिसमें Codex अगला वर्तमान-टर्न इवेंट उत्सर्जित करने
वाला होता है। दर-सीमा अपडेट जैसी ग्लोबल app-server सूचनाएँ टर्न-निष्क्रिय प्रगति
को रीसेट नहीं करतीं। रीजनिंग पूर्णताएँ, commentary `agentMessage` पूर्णताएँ,
और प्री-टूल कच्ची रीजनिंग या सहायक प्रगति के बाद स्वचालित अंतिम उत्तर आ सकता है,
इसलिए वे सत्र लेन को तुरंत रिलीज़ करने के बजाय पोस्ट-प्रोग्रेस उत्तर गार्ड का उपयोग
करती हैं।

केवल अंतिम/गैर-commentary पूर्ण `agentMessage` आइटम और प्री-टूल कच्ची
सहायक पूर्णताएँ सहायक-आउटपुट रिलीज़ को सक्रिय करती हैं: यदि Codex इसके बाद
`turn/completed` के बिना निष्क्रिय हो जाता है, तो OpenClaw सर्वोत्तम प्रयास के
आधार पर नेटिव टर्न को बाधित करता है और सत्र लेन को रिलीज़ करता है। यदि कोई अन्य
टर्न वॉच उस रिलीज़ रेस को जीत जाता है, तब भी OpenClaw पूर्ण अंतिम सहायक आइटम को
स्वीकार करता है, बशर्ते कोई नेटिव अनुरोध, आइटम या डायनेमिक टूल पूर्णता सक्रिय न हो,
सहायक-आउटपुट रिलीज़ अब भी नवीनतम पूर्ण आइटम से संबंधित हो और बाद में कोई आइटम
पूर्ण न हुआ हो। इससे पूर्ण हो चुके टूल कार्य के बाद टर्न को फिर से चलाए बिना अंतिम
उत्तर सुरक्षित रह सकता है। आंशिक सहायक डेल्टा, पहले के पुराने उत्तर और बाद की
खाली पूर्णताएँ पात्र नहीं होतीं।

रीप्ले-सुरक्षित stdio app-server विफलताएँ—जिनमें सहायक, टूल, सक्रिय-आइटम या
दुष्प्रभाव के प्रमाण के बिना टर्न-पूर्णता निष्क्रिय टाइमआउट शामिल हैं—एक नए
app-server प्रयास पर एक बार फिर आज़माई जाती हैं। असुरक्षित टाइमआउट फिर भी अटके
app-server क्लाइंट को सेवानिवृत्त करते हैं और OpenClaw सत्र लेन को रिलीज़ करते हैं;
वे स्वचालित रूप से रीप्ले करने के बजाय पुराने नेटिव थ्रेड बाइंडिंग को भी साफ़ करते
हैं। पूर्णता-वॉच टाइमआउट Codex-विशिष्ट टाइमआउट टेक्स्ट दिखाते हैं: रीप्ले-सुरक्षित
मामलों में कहा जाता है कि प्रतिक्रिया अधूरी हो सकती है, जबकि असुरक्षित मामलों में
उपयोगकर्ता को पुनः प्रयास करने से पहले वर्तमान स्थिति सत्यापित करने को कहा जाता
है। सार्वजनिक टाइमआउट डायग्नोस्टिक्स में अंतिम app-server सूचना विधि, कच्चे सहायक
प्रतिक्रिया आइटम की id/type/role, सक्रिय अनुरोध/आइटम गणना और सक्रिय वॉच स्थिति
जैसे संरचनात्मक फ़ील्ड शामिल होते हैं; जब अंतिम सूचना कच्चा सहायक प्रतिक्रिया आइटम
होती है, तो उनमें सीमित सहायक टेक्स्ट पूर्वावलोकन भी शामिल होता है। उनमें कच्चा
प्रॉम्प्ट या टूल कंटेंट शामिल नहीं होता।

### स्थानीय परीक्षण एनवायरनमेंट ओवरराइड

- `appServer.command` अनसेट होने पर `OPENCLAW_CODEX_APP_SERVER_BIN`
  प्रबंधित बाइनरी को बायपास करता है।
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

`OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1` हटा दिया गया था। इसके बजाय
`plugins.entries.codex.config.appServer.mode: "guardian"` का उपयोग करें, या एकबारगी स्थानीय परीक्षण के लिए
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` का उपयोग करें। दोहराए जा सकने वाले डिप्लॉयमेंट के लिए
कॉन्फ़िगरेशन को प्राथमिकता दी जाती है, क्योंकि यह Plugin व्यवहार को Codex हार्नेस
के शेष सेटअप वाली उसी समीक्षा की गई फ़ाइल में रखता है।

## नेटिव Codex plugins

नेटिव Codex plugin समर्थन, OpenClaw हार्नेस टर्न वाले उसी Codex थ्रेड में Codex
app-server की अपनी ऐप और plugin क्षमताओं का उपयोग करता है। OpenClaw Codex
plugins को कृत्रिम `codex_plugin_*` OpenClaw डायनेमिक टूल में अनुवाद नहीं करता।

`codexPlugins` केवल उन सत्रों को प्रभावित करता है जो नेटिव Codex हार्नेस
चुनते हैं। इसका बिल्ट-इन हार्नेस रन, सामान्य OpenAI प्रदाता रन, ACP वार्तालाप
बाइंडिंग या अन्य हार्नेस पर कोई प्रभाव नहीं पड़ता।

न्यूनतम माइग्रेट किया गया कॉन्फ़िगरेशन:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

थ्रेड ऐप कॉन्फ़िगरेशन की गणना तब की जाती है जब OpenClaw Codex हार्नेस सत्र
स्थापित करता है या पुराने Codex थ्रेड बाइंडिंग को बदलता है; हर टर्न पर इसकी पुनः
गणना नहीं होती। `codexPlugins` बदलने के बाद, `/new`,
`/reset` का उपयोग करें, या Gateway को पुनः शुरू करें, ताकि भविष्य के
Codex हार्नेस सत्र अपडेट किए गए ऐप सेट के साथ शुरू हों।

माइग्रेशन पात्रता, ऐप इन्वेंटरी, विनाशकारी कार्रवाई नीति, एलिसिटेशन और नेटिव
plugin डायग्नोस्टिक्स के लिए
[नेटिव Codex plugins](/hi/plugins/codex-native-plugins) देखें।

OpenAI-पक्षीय ऐप और plugin पहुँच को साइन-इन किए गए Codex खाते द्वारा, और
Business तथा Enterprise/Edu वर्कस्पेस के लिए वर्कस्पेस ऐप नियंत्रणों द्वारा,
नियंत्रित किया जाता है। OpenAI के खाते और वर्कस्पेस-नियंत्रण के अवलोकन के लिए
[अपने ChatGPT प्लान के साथ Codex का उपयोग करना](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan)
देखें।

## कंप्यूटर उपयोग

कंप्यूटर उपयोग की अपनी सेटअप गाइड है:
[Codex कंप्यूटर उपयोग](/hi/plugins/codex-computer-use)।

संक्षेप में: OpenClaw डेस्कटॉप-नियंत्रण ऐप को अपने साथ पैकेज नहीं करता और न ही
स्वयं डेस्कटॉप कार्रवाइयाँ निष्पादित करता है। यह Codex app-server तैयार करता है,
सत्यापित करता है कि `computer-use` MCP सर्वर उपलब्ध है, और फिर Codex-मोड
टर्न के दौरान नेटिव MCP टूल कॉल का स्वामित्व Codex को देता है।

## रनटाइम सीमाएँ

Codex हार्नेस केवल निम्न-स्तरीय एम्बेडेड एजेंट एक्ज़ीक्यूटर को बदलता है।

- OpenClaw डायनेमिक टूल समर्थित हैं। Codex OpenClaw से उन
  टूल को निष्पादित करने के लिए कहता है, इसलिए OpenClaw निष्पादन पथ में बना रहता है।
- Codex-नेटिव shell, patch, MCP और नेटिव ऐप टूल Codex के
  स्वामित्व में हैं। OpenClaw समर्थित रिले के माध्यम से चुने हुए नेटिव इवेंट देख
  या ब्लॉक कर सकता है, लेकिन यह नेटिव टूल आर्ग्युमेंट को फिर से नहीं लिखता।
- नेटिव Compaction का स्वामित्व Codex के पास है। OpenClaw
  चैनल इतिहास, खोज, `/new`, `/reset`, और भविष्य में मॉडल या हार्नेस
  स्विचिंग के लिए ट्रांसक्रिप्ट मिरर रखता है, लेकिन Codex Compaction को OpenClaw
  या कॉन्टेक्स्ट-इंजन सारांशकर्ता से नहीं बदलता।
- मीडिया जनरेशन, मीडिया अंडरस्टैंडिंग, TTS, अनुमोदन और
  मैसेजिंग-टूल आउटपुट संबंधित OpenClaw प्रदाता/मॉडल सेटिंग्स के माध्यम से जारी
  रहते हैं।
- `tool_result_persist` OpenClaw के स्वामित्व वाले ट्रांसक्रिप्ट
  टूल परिणामों पर लागू होता है, Codex-नेटिव टूल परिणाम रिकॉर्ड पर नहीं।

हुक परतों, समर्थित V1 सतहों, नेटिव अनुमति प्रबंधन, कतार निर्देशन, Codex फ़ीडबैक
अपलोड तंत्र और Compaction विवरणों के लिए
[Codex हार्नेस रनटाइम](/hi/plugins/codex-harness-runtime) देखें।

## समस्या निवारण

**Codex सामान्य `/model` प्रदाता के रूप में दिखाई नहीं देता:** नए
कॉन्फ़िगरेशन के लिए यह अपेक्षित है। कोई `openai/gpt-*` मॉडल चुनें,
`plugins.entries.codex.enabled` सक्षम करें और जाँचें कि क्या `plugins.allow`,
`codex` को बाहर करता है।

**OpenClaw Codex के बजाय बिल्ट-इन हार्नेस का उपयोग करता है:** पुष्टि करें कि
प्रभावी रूट सटीक आधिकारिक HTTPS Platform Responses या ChatGPT Responses रूट है,
उसमें कोई लिखित अनुरोध ओवरराइड नहीं है, और Codex Plugin इंस्टॉल तथा सक्षम है।
अकेला `openai/gpt-*` प्रीफ़िक्स पर्याप्त नहीं है। परीक्षण के दौरान कड़े प्रमाण
के लिए प्रदाता या मॉडल `agentRuntime.id: "codex"` सेट करें; रूट या हार्नेस असंगत होने पर
फ़ोर्स किया गया Codex फ़ॉलबैक करने के बजाय विफल होता है।

**OpenAI Codex रनटाइम API-कुंजी पथ पर फ़ॉलबैक करता है:** एक संशोधित Gateway
अंश एकत्र करें जो मॉडल, रनटाइम, चयनित प्रदाता और विफलता दिखाता हो। प्रभावित
सहयोगियों से उनके OpenClaw होस्ट पर यह केवल-पढ़ने वाला कमांड चलाने को कहें:

```bash
(
  pattern='openai/gpt-5\.[45]|openai[-]codex|agentRuntime(\.id)?|harnessRuntime|Runtime: OpenAI Codex|legacy OpenAI Codex prefix|resolveSelectedOpenAIRuntimeProvider|candidateProvider[": ]+openai|status[": ]+401|Incorrect API key|No API key|api-key path|API-key path|OAuth'

  if ls /tmp/openclaw/openclaw-*.log >/dev/null 2>&1; then
    grep -E -i -n "$pattern" /tmp/openclaw/openclaw-*.log 2>/dev/null || true
  else
    journalctl --user -u openclaw-gateway --since today --no-pager 2>/dev/null \
      | grep -E -i "$pattern" || true
  fi
) | sed -E \
    -e 's/(Authorization: Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(api[_ -]?key[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/(OPENAI_API_KEY[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/sk-[A-Za-z0-9_-]{12,}/sk-[REDACTED]/g' \
    -e 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL-REDACTED]/g' \
  | tail -200
```

उपयोगी अंशों में सामान्यतः `openai/gpt-5.6-sol` या `openai/gpt-5.6-luna`,
`Runtime: OpenAI Codex`, `agentRuntime.id` या `harnessRuntime`,
`candidateProvider: "openai"`, और कोई `401`, `Incorrect API key` या
`No API key` परिणाम शामिल होता है। सुधारे गए रन में सामान्य OpenAI
API-कुंजी विफलता के बजाय OpenAI OAuth पथ दिखाई देना चाहिए।

**लेगेसी Codex मॉडल रेफ़्स कॉन्फ़िगरेशन शेष है:** `openclaw doctor --fix` चलाएँ।
Doctor लेगेसी मॉडल रेफ़्स को `openai/*` में फिर से लिखता है, पुराने सेशन और
पूरे-एजेंट रनटाइम पिन हटाता है, और मौजूदा ऑथ-प्रोफ़ाइल ओवरराइड सुरक्षित रखता है।

**ऐप-सर्वर अस्वीकार कर दिया गया है:** बंडल किए गए `0.144.6` के माध्यम से
`0.143.0` का एक स्थिर Codex ऐप-सर्वर उपयोग करें। प्रीरिलीज़, बिल्ड-सफ़िक्स वाले संस्करण, और नए
अमान्यापित रिलीज़ अस्वीकार किए जाते हैं, क्योंकि OpenClaw जनरेट किए गए स्कीमा को
बंडल किए गए ऐप-सर्वर संस्करण के विरुद्ध सत्यापित करता है।

**`/codex status` कनेक्ट नहीं कर सकता:** जाँचें कि `codex` Plugin
सक्षम है, अलाउलिस्ट कॉन्फ़िगर होने पर `plugins.allow` में यह शामिल है,
और कोई भी कस्टम `appServer.command`, `url`, `authToken`, या
हेडर मान्य हैं।

**Codex ऐप-सर्वर बहुत अधिक मेमोरी उपयोग करता है:** पहले दोनों प्रक्रियाओं में
अंतर करें। OpenClaw स्थानीय Codex ऐप-सर्वर को एक अलग Rust चाइल्ड के रूप में चलाता है।
`NODE_OPTIONS=--max-old-space-size=...` केवल Gateway के Node.js V8
हीप को बदलता है; यह Codex को सीमित या विस्तारित नहीं करता। प्रबंधित Gateway इंस्टॉलेशन पहले से ही
एक अनुकूली V8 हीप चुनते हैं, और इसे बढ़ाने से Codex के लिए कम होस्ट मेमोरी बच सकती है।
Gateway पर दबाव के लिए
[Gateway मेमोरी समस्या-निवारण](/hi/gateway/troubleshooting#gateway-exits-during-high-memory-use)
का उपयोग करें, और Codex चाइल्ड के लिए होस्ट या कंटेनर मेमोरी की जाँच करें।

बंडल किए गए Codex की कोई हीप या RSS सीमा और कोई कॉन्फ़िगर करने योग्य निष्क्रिय-अनलोड
विलंब नहीं है। अंतिम क्लाइंट के अनसब्सक्राइब करने के बाद, कोई निष्क्रिय थ्रेड अधिकतम
30 मिनट तक लोड रह सकता है। सीमित होस्ट पर Gateway हीप बढ़ाने से पहले
नेटिव Codex सबएजेंट फ़ैन-आउट कम करें:

```json5
{
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            args: ["-c", "agents.max_threads=3", "app-server", "--listen", "stdio://"],
          },
        },
      },
    },
  },
}
```

वह सेटिंग बंडल किए गए Codex के डिफ़ॉल्ट
मल्टी-एजेंट बैकएंड के लिए नेटिव चाइल्ड थ्रेड सीमित करती है। यदि आप Codex मल्टी-एजेंट v2 को स्पष्ट रूप से सक्षम करते हैं, तो इसके बजाय
`features.multi_agent_v2.max_concurrent_threads_per_session=3` का उपयोग करें; v2
सीमा में रूट थ्रेड शामिल है और इसे `agents.max_threads` के साथ संयोजित नहीं किया जा सकता।
Codex के लिए अधिक क्षमता पाने हेतु होस्ट, कंटेनर, या cgroup मेमोरी
आवंटन बढ़ाएँ। OS की हार्ड सीमा बैकप्रेशर लागू करने के बजाय Codex को समाप्त कर सकती है।

**मॉडल खोज धीमी है:** `plugins.entries.codex.config.discovery.timeoutMs` कम करें
या खोज अक्षम करें।
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference#model-discovery) देखें।

**WebSocket ट्रांसपोर्ट तुरंत विफल हो जाता है:** `appServer.url`,
`authToken`, हेडर, और यह जाँचें कि रिमोट ऐप-सर्वर वही Codex
ऐप-सर्वर प्रोटोकॉल संस्करण उपयोग करता है। Codex WebSocket ट्रांसपोर्ट अभी भी प्रयोगात्मक
और असमर्थित है; प्रबंधित stdio या स्थानीय Unix नियंत्रण सॉकेट को प्राथमिकता दें।

**नेटिव शेल या पैच टूल `Native hook relay
unavailable` के साथ अवरुद्ध हैं:** Codex थ्रेड अभी भी ऐसे नेटिव हुक रिले
आईडी का उपयोग करने का प्रयास कर रहा है जिसे OpenClaw में अब पंजीकृत नहीं किया गया है। यह नेटिव Codex हुक
ट्रांसपोर्ट समस्या है, ACP बैकएंड, प्रदाता, GitHub, या शेल-कमांड
विफलता नहीं। प्रभावित चैट में `/new` या `/reset` के साथ नया सेशन शुरू करें,
फिर किसी अहानिकर कमांड को दोबारा आज़माएँ। यदि यह एक बार काम करता है लेकिन अगली नेटिव टूल
कॉल फिर विफल हो जाती है, तो `/new` को केवल अस्थायी समाधान मानें: Codex ऐप-सर्वर या
OpenClaw Gateway को पुनः आरंभ करने के बाद प्रॉम्प्ट को नए सेशन में कॉपी करें, ताकि पुराने थ्रेड हटा दिए जाएँ और नेटिव हुक पंजीकरण
फिर से बनाए जाएँ।

**Codex टूल कॉल बहुत अधिक अल्पजीवी हुक प्रक्रियाएँ बनाती हैं:** `plugins.entries.codex.config.appServer.loopDetectionPreToolUseRelay: false`
सेट करें
और Gateway को पुनः आरंभ करें। यह केवल OpenClaw लूप पहचान और उसके नो-पॉलिसी मार्कर के लिए उपयोग की जाने वाली Codex `PreToolUse` उपप्रक्रिया
को अक्षम करता है। आवश्यक
`before_tool_call` और विश्वसनीय-टूल पॉलिसी रिले सक्षम रहते हैं।

**एक गैर-Codex मॉडल बिल्ट-इन हार्नेस का उपयोग करता है:** यह अपेक्षित है, जब तक प्रदाता
या मॉडल रनटाइम पॉलिसी उसे किसी अन्य हार्नेस पर रूट न करे। सामान्य गैर-OpenAI
प्रदाता रेफ़्स `auto` मोड में अपने सामान्य प्रदाता पथ पर बने रहते हैं।

**Computer Use इंस्टॉल है लेकिन टूल नहीं चलते:** नए सेशन से
`/codex computer-use status` जाँचें। यदि कोई टूल
`Native hook relay unavailable` रिपोर्ट करता है, तो ऊपर दिए गए नेटिव हुक रिले पुनर्प्राप्ति चरणों का उपयोग करें।
[Codex Computer Use](/hi/plugins/codex-computer-use#troubleshooting) देखें।
