आधिकारिक `codex` Plugin, अंतर्निर्मित OpenClaw हार्नेस के बजाय Codex
app-server के माध्यम से एम्बेड किए गए OpenAI एजेंट टर्न चलाता है। निम्न-स्तरीय
एजेंट सत्र का स्वामित्व Codex के पास है: नेटिव थ्रेड पुनरारंभ, नेटिव टूल निरंतरता,
नेटिव Compaction और app-server निष्पादन। चैट चैनल, सत्र फ़ाइलें, मॉडल चयन,
OpenClaw डायनेमिक टूल, अनुमोदन, मीडिया डिलीवरी और दृश्यमान ट्रांस्क्रिप्ट मिरर
अब भी OpenClaw के स्वामित्व में हैं।

`openai/gpt-5.6-sol` जैसे कैनोनिकल OpenAI मॉडल संदर्भों का उपयोग करें। पुराने
Codex GPT संदर्भ कॉन्फ़िगर न करें; OpenAI एजेंट प्रमाणीकरण क्रम को
`auth.order.openai` के अंतर्गत रखें। पुराने Codex प्रमाणीकरण प्रोफ़ाइल आईडी और
पुरानी Codex प्रमाणीकरण क्रम प्रविष्टियों की मरम्मत `openclaw doctor --fix` द्वारा
की जाती है।

जब प्रदाता/मॉडल रनटाइम नीति सेट न हो या `auto` हो, तो केवल
`openai/*` उपसर्ग कभी भी इस हार्नेस का चयन नहीं करता। OpenAI केवल ऐसी
सटीक आधिकारिक HTTPS Platform Responses या ChatGPT Responses रूट के लिए Codex
का निहित रूप से चयन कर सकता है, जिसमें कोई लिखित अनुरोध ओवरराइड न हो। देखें
[OpenAI का निहित एजेंट रनटाइम](/hi/providers/openai#implicit-agent-runtime)।
यदि Platform बनाम ChatGPT रूटिंग ज्ञात होने से पहले प्रमाणीकरण का स्वामित्व
Codex के पास हो, तब भी OpenClaw के लिए प्रत्येक संभावित रूट द्वारा Codex
संगतता घोषित करना आवश्यक है। केवल नेटिव प्रमाणीकरण स्वामित्व उस रूट जाँच को
कभी बायपास नहीं करता।

जब कोई OpenClaw सैंडबॉक्स सक्रिय नहीं होता, तो OpenClaw नेटिव Codex कोड मोड
सक्षम करके Codex app-server थ्रेड शुरू करता है (केवल-कोड-मोड डिफ़ॉल्ट रूप से
बंद रहता है), इसलिए app-server `item/tool/call` ब्रिज के माध्यम से रूट किए
गए OpenClaw डायनेमिक टूल के साथ नेटिव कार्यस्थान/कोड क्षमताएँ उपलब्ध रहती हैं।
सक्रिय OpenClaw सैंडबॉक्स या प्रतिबंधित टूल नीति नेटिव कोड मोड को पूरी तरह
अक्षम कर देती है, जब तक कि आप प्रयोगात्मक सैंडबॉक्स exec-server पथ को स्पष्ट
रूप से न चुनें।

डिफ़ॉल्ट `tools.exec.host: "auto"` और कोई सक्रिय OpenClaw सैंडबॉक्स न होने पर,
Codex को युग्मित नोड पर कमांड के लिए `node_exec` और
`node_process` टूल भी मिलते हैं। नेटिव शेल Codex app-server होस्ट और
कार्यस्थान पर रहता है (डिफ़ॉल्ट stdio परिनियोजन के लिए Gateway-स्थानीय);
`node_exec` नाम या आईडी से किसी नोड का चयन करता है और OpenClaw की नोड
अनुमोदन नीति को लागू रखता है। यदि कोई सीमित रनटाइम अनुमति-सूची नेटिव कोड मोड
को अक्षम कर देती है और टर्न को निष्पादन परिवेश के बिना छोड़ देती है, तो
OpenClaw इसके बजाय प्रत्यक्ष, बिना सैंडबॉक्स वाले निष्पादन के लिए अपने
नीति-फ़िल्टर किए गए `exec` और `process` टूल उपलब्ध
रखता है।

यह Codex-नेटिव सुविधा [OpenClaw कोड मोड](/hi/reference/code-mode) से अलग है,
जो अलग `exec` इनपुट आकार वाले सामान्य OpenClaw रन के लिए
ऑप्ट-इन QuickJS-WASI रनटाइम है। व्यापक मॉडल/प्रदाता/रनटाइम विभाजन के लिए
[एजेंट रनटाइम](/hi/concepts/agent-runtimes) से आरंभ करें:
`openai/gpt-5.6-sol` मॉडल संदर्भ है, `codex` रनटाइम है और Telegram,
Discord, Slack या कोई अन्य चैनल संचार सतह है।

## आवश्यकताएँ

- आधिकारिक `@openclaw/codex` Plugin इंस्टॉल हो। यदि आपका
  कॉन्फ़िगरेशन अनुमति-सूची का उपयोग करता है, तो `plugins.allow` में
  `codex` शामिल करें।
- Codex app-server `0.143.0` या नया संस्करण। Plugin
  डिफ़ॉल्ट रूप से संगत बाइनरी का प्रबंधन करता है, इसलिए `PATH` पर
  मौजूद `codex` कमांड सामान्य स्टार्टअप को प्रभावित नहीं करता।
- `openclaw models auth login --provider openai` के माध्यम से Codex प्रमाणीकरण, एजेंट के
  Codex होम में पहले से मौजूद app-server खाता या स्पष्ट Codex API-कुंजी
  प्रमाणीकरण प्रोफ़ाइल।

प्रमाणीकरण प्राथमिकता, परिवेश पृथक्करण, कस्टम app-server कमांड, मॉडल खोज और
कॉन्फ़िगरेशन फ़ील्ड की पूरी सूची के लिए
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference) देखें।

## त्वरित शुरुआत

आधिकारिक Plugin इंस्टॉल करें, फिर Codex OAuth से साइन इन करें:

```bash
openclaw plugins install @openclaw/codex
openclaw models auth login --provider openai
```

`codex` Plugin सक्षम करें और OpenAI एजेंट मॉडल चुनें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

यदि आपका कॉन्फ़िगरेशन `plugins.allow` का उपयोग करता है, तो उसमें
`codex` भी जोड़ें:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Plugin कॉन्फ़िगरेशन बदलने के बाद Gateway पुनरारंभ करें। यदि किसी चैट में पहले
से सत्र है, तो पहले `/new` या `/reset` चलाएँ, ताकि
अगला टर्न वर्तमान कॉन्फ़िगरेशन से हार्नेस निर्धारित करे।

## Codex Desktop और CLI के साथ थ्रेड साझा करें

डिफ़ॉल्ट `appServer.homeScope: "agent"` प्रत्येक OpenClaw एजेंट को ऑपरेटर की नेटिव Codex
स्थिति से अलग रखता है। किसी स्वामी को Codex Desktop और Codex CLI में दिखने
वाले उन्हीं नेटिव थ्रेड का निरीक्षण और प्रबंधन करने देने के लिए उपयोगकर्ता
Codex होम को स्पष्ट रूप से चुनें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            homeScope: "user",
          },
        },
      },
    },
  },
}
```

उपयोगकर्ता-होम मोड स्थानीय प्रबंधित stdio प्रक्रिया या साझा Unix-socket
ट्रांसपोर्ट का समर्थन करता है। यह सेट होने पर `$CODEX_HOME` और अन्यथा
`~/.codex` का उपयोग करता है, जिसमें उस होम का नेटिव Codex
प्रमाणीकरण, कॉन्फ़िगरेशन, plugins और थ्रेड स्टोर शामिल हैं। OpenClaw इस
app-server में OpenClaw प्रमाणीकरण प्रोफ़ाइल इंजेक्ट नहीं करता।

स्वामी के टर्न को `codex_threads` टूल मिलता है: नेटिव थ्रेड सूचीबद्ध करना,
खोजना, पढ़ना, फोर्क करना, नाम बदलना, संग्रहित करना और पुनर्स्थापित करना। किसी
थ्रेड को OpenClaw में जारी रखने के लिए उसे फोर्क करें; फोर्क वर्तमान OpenClaw
सत्र से जुड़ता है और अन्य नेटिव Codex क्लाइंट को दिखाई देता रहता है। संग्रहित
करने के लिए स्पष्ट पुष्टि आवश्यक है कि थ्रेड अन्यत्र बंद है। जब पर्यवेक्षण भी
सक्षम हो, तो ट्रांस्क्रिप्ट फ़ील्ड और परिवर्तन के लिए संगत
`supervision.allowRawTranscripts` या `supervision.allowWriteControls` ऑप्ट-इन आवश्यक है।

स्वतंत्र प्रबंधित stdio App Servers के माध्यम से एक ही थ्रेड को समवर्ती रूप
से पुनरारंभ या उसमें लिखें नहीं। Codex लाइव लेखकों का समन्वय एक App Server
के भीतर करता है, अलग-अलग प्रक्रियाओं के बीच नहीं। सामान्य उपयोगकर्ता-होम
stdio सत्रों के लिए फोर्क करना सुरक्षित सह-अस्तित्व पथ है।

केवल `appServer.homeScope: "user"` फ़्लीट कैटलॉग को नियंत्रित नहीं करता। Plugin सक्रिय
रहने के दौरान नेटिव सत्र खोज सक्षम रहती है; Codex को अक्षम किए बिना इसे
OpenClaw साइडबार से हटाने के लिए `sessionCatalog.enabled: false` सेट करें। कैटलॉग एक अलग
पर्यवेक्षण कनेक्शन का उपयोग करता है; स्पष्ट `appServer` कनेक्शन
सेटिंग के बिना, वह कनेक्शन डिफ़ॉल्ट रूप से प्रबंधित उपयोगकर्ता-होम stdio का
उपयोग करता है, जबकि सामान्य हार्नेस एजेंट-स्कोप में रहता है। स्पष्ट
`appServer` सेटिंग दोनों पथों द्वारा मानी जाती हैं। जब सामान्य हार्नेस
को भी नेटिव स्थिति साझा करनी हो, तो ऊपर की तरह `homeScope: "user"` स्पष्ट
रूप से सेट करें।

## Codex सत्रों का पर्यवेक्षण करें

यही `codex` Plugin Gateway कंप्यूटर और ऑप्ट-इन किए गए युग्मित नोड
से गैर-संग्रहित Codex सत्रों को सूचीबद्ध कर सकता है। संग्रहित या निष्क्रिय
Gateway-स्थानीय सत्र एक मॉडल-लॉक चैट बना सकता है, जो उसके सीमित, स्थायी
उपयोगकर्ता और सहायक इतिहास को मिरर करता है। इसका निजी बाइंडिंग नेटिव स्नैपशॉट,
कैनोनिकल शाखा और बाद के टर्न के लिए पर्यवेक्षण कनेक्शन का उपयोग करता है, जबकि
सामान्य Codex सत्र एजेंट-स्कोप में रहते हैं। पहला कैनोनिकल प्रारंभ ठीक उसी
मॉडल और प्रदाता का उपयोग करता है जिसे Codex स्नैपशॉट फोर्क के लिए लौटाता है।
बाद के पुनरारंभ चयन को Codex के नेटिव कॉन्फ़िगरेशन पर छोड़ देते हैं; बाहरी
OpenClaw मॉडल और फ़ॉलबैक श्रृंखला इसे कभी प्रतिस्थापित नहीं करती। संग्रहित और
निष्क्रिय पंक्तियों को स्पष्ट रूप से यह पुष्टि करने के बाद संग्रहित किया जा
सकता है कि कोई अन्य रनर सक्रिय नहीं है। सक्रिय स्रोत न तो शाखा बना सकते हैं,
न संग्रहित किए जा सकते हैं; मौजूदा पर्यवेक्षित चैट फिर भी खोली जा सकती है।
युग्मित-नोड सत्र केवल मेटाडेटा तक सीमित रहते हैं।

सेटअप, ब्रांचिंग नियमों, युग्मित-नोड सीमाओं, मेटाडेटा प्रदर्शन और समस्या
निवारण के लिए [Codex सत्रों का पर्यवेक्षण करें](/plugins/codex-supervision)
देखें।

सदस्यता-प्रथम/API-कुंजी-बैकअप क्रम के लिए `auth.order.openai` को प्राथमिकता
दें। मौजूदा पुराने Codex प्रमाणीकरण प्रोफ़ाइल आईडी और पुराना Codex प्रमाणीकरण
क्रम केवल doctor द्वारा संभाली जाने वाली पुरानी स्थिति हैं; नए पुराने Codex
GPT संदर्भ न लिखें।

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

Codex-संगत प्रभावी रूट के लिए ऊपर दिए गए दोनों प्रोफ़ाइल उसी Codex रन के
संभावित विकल्प बने रहते हैं। प्रोफ़ाइल क्रम क्रेडेंशियल चुनता है, रनटाइम नहीं।
प्रमाणीकरण क्रम बदलने से कस्टम, Completions, HTTP या अनुरोध-ओवरराइड वाला रूट
Codex-संगत नहीं बनता।

### Compaction

Codex-समर्थित एजेंट पर `compaction.model` या `compaction.provider` सेट न करें।
Codex अपनी नेटिव app-server थ्रेड स्थिति के माध्यम से Compaction करता है,
इसलिए OpenClaw रनटाइम पर उन स्थानीय सारांशकर्ता ओवरराइड को अनदेखा करता है और
जब एजेंट Codex का उपयोग करता है, तो `openclaw doctor --fix` उन्हें हटा देता है।

Lossless, Codex टर्न के आसपास असेंबली, अंतर्ग्रहण और रखरखाव के लिए संदर्भ इंजन
के रूप में समर्थित रहता है। इसे `agents.defaults.compaction.provider` के बजाय
`plugins.slots.contextEngine: "lossless-claw"` और `plugins.entries.lossless-claw.config.summaryModel` के माध्यम से कॉन्फ़िगर किया जाता है।
Codex सक्रिय रनटाइम होने पर `openclaw doctor --fix` पुराने `compaction.provider: "lossless-claw"`
आकार को Lossless संदर्भ-इंजन स्लॉट में माइग्रेट करता है, लेकिन Compaction का
स्वामित्व फिर भी नेटिव Codex के पास रहता है। नेटिव app-server हार्नेस ऐसे
संदर्भ इंजनों का समर्थन करता है जिन्हें प्री-प्रॉम्प्ट असेंबली की आवश्यकता
होती है; `codex-cli` सहित सामान्य CLI बैकएंड वह होस्ट क्षमता प्रदान
नहीं करते।

Codex-समर्थित एजेंट के लिए `/compact` बाउंड थ्रेड पर नेटिव Codex
app-server Compaction शुरू करता है। OpenClaw पूर्ण होने की प्रतीक्षा नहीं
करता, OpenClaw टाइमआउट लागू नहीं करता, साझा app-server को पुनरारंभ नहीं करता
और किसी संदर्भ-इंजन या सार्वजनिक OpenAI सारांशकर्ता पर फ़ॉलबैक नहीं करता। यदि
नेटिव Codex थ्रेड बाइंडिंग अनुपलब्ध या पुरानी हो, तो कमांड चुपचाप Compaction
बैकएंड बदलने के बजाय बंद अवस्था में विफल होता है।

इस पृष्ठ का शेष भाग परिनियोजन संरचना, बंद अवस्था में विफल होने वाली रूटिंग,
गार्डियन अनुमोदन नीति, नेटिव Codex plugins और Computer Use को शामिल करता है।
विकल्पों की पूरी सूचियों, डिफ़ॉल्ट, enum, खोज, परिवेश पृथक्करण, टाइमआउट और
app-server ट्रांसपोर्ट फ़ील्ड के लिए
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference) देखें।

## Codex रनटाइम सत्यापित करें

उस चैट में `/status` का उपयोग करें जहाँ आप Codex की अपेक्षा करते हैं। Codex द्वारा समर्थित OpenAI
एजेंट टर्न यह दिखाता है:

```text
रनटाइम: OpenAI Codex
```

फिर Codex ऐप-सर्वर की स्थिति जाँचें:

```text
/codex status
/codex models
```

`/codex status` ऐप-सर्वर कनेक्टिविटी, अकाउंट, दर सीमाएँ, MCP
सर्वर और Skills की रिपोर्ट करता है। `/codex models` हार्नेस और अकाउंट के लिए लाइव Codex ऐप-सर्वर कैटलॉग
सूचीबद्ध करता है। यदि `/status` अप्रत्याशित है, तो
[समस्या निवारण](#troubleshooting) देखें।

## रूटिंग और मॉडल चयन

प्रदाता संदर्भों और रनटाइम नीति को अलग रखें:

- प्रामाणिक OpenAI मॉडल चयन के लिए `openai/gpt-*` का उपयोग करें। अकेला उपसर्ग
  कभी भी Codex का चयन नहीं करता।
- रनटाइम अनिर्धारित या `auto` होने पर, केवल बिना किसी लिखित अनुरोध ओवरराइड वाला सटीक आधिकारिक HTTPS Platform Responses
  या ChatGPT Responses रूट ही अंतर्निहित रूप से Codex का चयन कर सकता है।
- कॉन्फ़िगरेशन में पुराने Codex GPT संदर्भों का उपयोग न करें; पुराने संदर्भों और अप्रचलित सत्र रूट पिनों को
  सुधारने के लिए `openclaw doctor --fix` चलाएँ।
- `agentRuntime.id: "codex"` किसी संगत रूट के लिए Codex को विफलता पर बंद होने वाली अनिवार्य आवश्यकता बनाता है।
  यह किसी असंगत प्रभावी रूट को संगत नहीं बनाता।
- `agentRuntime.id: "openclaw"` किसी प्रदाता या मॉडल को एम्बेडेड
  OpenClaw रनटाइम में शामिल करता है, जब ऐसा करना अभिप्रेत हो।
- `/codex ...` चैट से मूल Codex ऐप-सर्वर वार्तालापों को नियंत्रित करता है।
- ACP/acpx एक अलग बाहरी हार्नेस पथ है। इसका उपयोग केवल तभी करें जब उपयोगकर्ता
  ACP/acpx या किसी बाहरी हार्नेस अडैप्टर का अनुरोध करे।

## परिनियोजन प्रतिरूप

### बुनियादी Codex परिनियोजन

ऐसे OpenAI मॉडल के लिए त्वरित-आरंभ कॉन्फ़िगरेशन का उपयोग करें जिसका प्रभावी आधिकारिक HTTPS
रूट अंतर्निहित रूप से Codex चुनने के योग्य हो:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

### मिश्रित प्रदाता परिनियोजन

Claude को डिफ़ॉल्ट एजेंट बनाए रखें और एक नामित Codex एजेंट जोड़ें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-6",
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.6-sol",
      },
    ],
  },
}
```

`main` एजेंट अपने सामान्य प्रदाता पथ का उपयोग करता है। `codex` एजेंट Codex
ऐप-सर्वर का उपयोग करता है, जब उसका प्रभावी OpenAI रूट संगत बना रहता है; जब इसे विफलता पर बंद होने वाली
अनिवार्य आवश्यकता बनाना हो, तो स्पष्ट मॉडल-स्कोप वाला `agentRuntime.id: "codex"` जोड़ें।

### विफलता पर बंद होने वाला Codex परिनियोजन

बंडल किया गया plugin उपलब्ध होने पर कोई योग्य सटीक आधिकारिक HTTPS OpenAI रूट Codex में
समाधान हो सकता है। लिखित विफलता-पर-बंद नियम के लिए स्पष्ट रनटाइम नीति
जोड़ें:

```json5
{
  models: {
    providers: {
      openai: {
        agentRuntime: {
          id: "codex",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Codex बाध्य होने पर, यदि प्रभावी रूट Codex-संगत घोषित नहीं है, plugin अक्षम है, ऐप-सर्वर बहुत पुराना है, या
ऐप-सर्वर प्रारंभ नहीं हो सकता, तो OpenClaw शीघ्र विफल हो जाता है।

## ऐप-सर्वर नीति

डिफ़ॉल्ट रूप से, plugin OpenClaw की प्रबंधित Codex बाइनरी को स्थानीय रूप से
stdio ट्रांसपोर्ट के साथ प्रारंभ करता है। किसी अलग निष्पादन योग्य फ़ाइल को जानबूझकर चलाने के लिए ही
`appServer.command` सेट करें। WebSocket ट्रांसपोर्ट का उपयोग केवल तभी करें जब कोई ऐप-सर्वर
पहले से कहीं और चल रहा हो:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
          },
        },
      },
    },
  },
}
```

स्थानीय stdio ऐप-सर्वर सत्र डिफ़ॉल्ट रूप से विश्वसनीय स्थानीय ऑपरेटर
स्थिति अपनाते हैं: `approvalPolicy: "never"`, `approvalsReviewer: "user"`, और
`sandbox: "danger-full-access"`। यदि स्थानीय Codex आवश्यकताएँ उस
अंतर्निहित YOLO स्थिति को अस्वीकार करती हैं, तो OpenClaw इसके बजाय अनुमत संरक्षक अनुमतियाँ
चुनता है। जब सत्र के लिए OpenClaw सैंडबॉक्स सक्रिय होता है, तो OpenClaw
उस टर्न के लिए Codex का मूल Code Mode, उपयोगकर्ता MCP सर्वर और ऐप-समर्थित plugin
निष्पादन अक्षम कर देता है, बजाय Codex की होस्ट-पक्षीय सैंडबॉक्सिंग पर निर्भर रहने के।
इसके बजाय शेल पहुँच OpenClaw सैंडबॉक्स-समर्थित डायनेमिक टूल्स, जैसे
`sandbox_exec` और `sandbox_process`, के माध्यम से होती है, जब सामान्य exec/process टूल
उपलब्ध हों।

सैंडबॉक्स से बाहर निकलने या अतिरिक्त अनुमतियों से पहले Codex की मूल स्वतः-समीक्षा के लिए
सामान्यीकृत OpenClaw exec मोड का उपयोग करें:

```json5
{
  tools: {
    exec: {
      mode: "auto",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Codex ऐप-सर्वर सत्रों के लिए, `tools.exec.mode: "auto"` Codex
Guardian-समीक्षित अनुमोदनों से मैप होता है: आम तौर पर `approvalPolicy: "on-request"`,
`approvalsReviewer: "auto_review"`, और `sandbox: "workspace-write"`, जब
स्थानीय आवश्यकताएँ इन मानों की अनुमति देती हैं। `tools.exec.mode: "auto"` में,
OpenClaw पुराने असुरक्षित Codex `approvalPolicy: "never"` या
`sandbox: "danger-full-access"` ओवरराइड संरक्षित नहीं रखता; जानबूझकर अनुमोदन-रहित Codex स्थिति के लिए
`tools.exec.mode: "full"` का उपयोग करें। पुराना
`plugins.entries.codex.config.appServer.mode: "guardian"` प्रीसेट अभी भी
काम करता है, लेकिन `tools.exec.mode: "auto"` सामान्यीकृत OpenClaw सतह है।

होस्ट exec अनुमोदनों और ACPX अनुमतियों के साथ मोड-स्तरीय तुलना के लिए,
[अनुमति मोड](/hi/tools/permission-modes) देखें। प्रत्येक ऐप-सर्वर फ़ील्ड,
प्रमाणीकरण क्रम, पर्यावरण पृथक्करण और समय-समाप्ति व्यवहार के लिए,
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference) देखें।

## कमांड और निदान

`codex` plugin, OpenClaw टेक्स्ट कमांड का समर्थन करने वाले किसी भी चैनल पर `/codex` को स्लैश कमांड के रूप में
पंजीकृत करता है।

मूल निष्पादन और नियंत्रण के लिए किसी मालिक या `operator.admin`
Gateway क्लाइंट की आवश्यकता होती है: थ्रेड बाँधना या फिर से शुरू करना, टर्न भेजना या रोकना,
मॉडल, तेज़-मोड या अनुमति स्थिति बदलना, संक्षिप्तीकरण या समीक्षा करना, और
बाइंडिंग अलग करना। अन्य अधिकृत प्रेषकों को केवल-पढ़ने योग्य स्थिति, सहायता,
अकाउंट, मॉडल, थ्रेड, MCP सर्वर, Skills और बाइंडिंग निरीक्षण कमांड मिलते हैं।

सामान्य रूप:

- `/codex status` ऐप-सर्वर कनेक्टिविटी, मॉडल, अकाउंट, दर
  सीमाएँ, MCP सर्वर और Skills जाँचता है।
- `/codex models` लाइव Codex ऐप-सर्वर मॉडल सूचीबद्ध करता है।
- `/codex threads [filter]` हाल के Codex ऐप-सर्वर थ्रेड सूचीबद्ध करता है।
- `/codex resume <thread-id>` वर्तमान OpenClaw सत्र को किसी
  मौजूदा Codex थ्रेड से संलग्न करता है।
- `/codex bind [thread-id] [--cwd <path>] [--model <model>] [--provider <provider>]`
  वर्तमान चैट संलग्न करता है।
- `/codex detach` (या `/codex unbind`) वर्तमान बाइंडिंग अलग करता है।
- `/codex binding` वर्तमान बाइंडिंग का वर्णन करता है।
- `/codex stop` सक्रिय टर्न रोकता है; `/codex steer <text>` इसे निर्देशित करता है।
- `/codex model <model>`, `/codex fast [on|off|status]`, और
  `/codex permissions [default|yolo|status]` प्रति-वार्तालाप स्थिति बदलते हैं।
- `/codex compact` Codex ऐप-सर्वर से संलग्न थ्रेड को संक्षिप्त करने के लिए कहता है।
- `/codex review` संलग्न थ्रेड के लिए Codex की मूल समीक्षा प्रारंभ करता है।
- `/codex diagnostics [note]` संलग्न थ्रेड के लिए Codex फ़ीडबैक भेजने से पहले
  पूछता है।
- `/codex account` अकाउंट और दर-सीमा स्थिति दिखाता है।
- `/codex mcp` Codex ऐप-सर्वर MCP सर्वर स्थिति सूचीबद्ध करता है।
- `/codex skills` Codex ऐप-सर्वर Skills सूचीबद्ध करता है।
- `/codex plugins list`, `/codex plugins enable <name>`, और
  `/codex plugins disable <name>` कॉन्फ़िगर किए गए मूल Codex plugins प्रबंधित करते हैं।
- `/codex computer-use [status|install]` Codex Computer Use प्रबंधित करता है।
- `/codex help` पूरा कमांड वृक्ष सूचीबद्ध करता है।

अधिकांश सहायता रिपोर्टों के लिए, उस बातचीत में `/diagnostics [note]` से शुरू करें
जहाँ बग हुआ था। यह एक Gateway निदान रिपोर्ट बनाता है और Codex हार्नेस
सत्रों के लिए, संबंधित Codex फ़ीडबैक बंडल भेजने की स्वीकृति माँगता है।
गोपनीयता मॉडल और समूह चैट व्यवहार के लिए
[निदान निर्यात](/hi/gateway/diagnostics) देखें। `/codex diagnostics [note]` का उपयोग केवल तभी करें जब आप विशेष रूप से
वर्तमान में संलग्न थ्रेड के लिए पूर्ण Gateway निदान बंडल के बिना
Codex फ़ीडबैक अपलोड चाहते हों।

### Codex थ्रेड का स्थानीय रूप से निरीक्षण करें

किसी खराब Codex रन का निरीक्षण करने का सबसे तेज़ तरीका अक्सर मूल
Codex थ्रेड को सीधे खोलना होता है:

```bash
codex resume <thread-id>
```

थ्रेड आईडी पूर्ण हो चुके `/diagnostics` उत्तर, `/codex binding`,
या `/codex threads [filter]` से प्राप्त करें।

अपलोड प्रक्रिया और रनटाइम-स्तरीय निदान सीमाओं के लिए,
[Codex हार्नेस रनटाइम](/hi/plugins/codex-harness-runtime#codex-feedback-upload) देखें।

### प्रमाणीकरण क्रम

डिफ़ॉल्ट प्रति-एजेंट होम में, प्रमाणीकरण इस क्रम में चुना जाता है:

1. एजेंट के लिए क्रमबद्ध OpenAI प्रमाणीकरण प्रोफ़ाइल, अधिमानतः
   `auth.order.openai` के अंतर्गत। पुराने लेगेसी
   Codex प्रमाणीकरण प्रोफ़ाइल आईडी और लेगेसी Codex प्रमाणीकरण क्रम को माइग्रेट करने के लिए `openclaw doctor --fix` चलाएँ।
2. उस एजेंट के Codex होम में app-server का मौजूदा खाता।
3. केवल स्थानीय stdio app-server लॉन्च के लिए, `CODEX_API_KEY`, फिर
   `OPENAI_API_KEY`, जब कोई app-server खाता मौजूद न हो और OpenAI प्रमाणीकरण
   अब भी आवश्यक हो।

जब OpenClaw को ChatGPT सदस्यता-शैली की Codex प्रमाणीकरण प्रोफ़ाइल मिलती है, तो यह
स्पॉन की गई Codex चाइल्ड प्रक्रिया से `CODEX_API_KEY` और `OPENAI_API_KEY` हटा देता है।
इससे Gateway-स्तरीय API कुंजियाँ एम्बेडिंग या
प्रत्यक्ष OpenAI मॉडलों के लिए उपलब्ध रहती हैं, जबकि मूल Codex app-server टर्न का
अनजाने में API के माध्यम से बिल बनना रुकता है। स्पष्ट Codex API-कुंजी प्रोफ़ाइल और स्थानीय
stdio एनव-कुंजी फ़ॉलबैक, इनहेरिट किए गए
चाइल्ड-प्रोसेस एनव के बजाय app-server लॉगिन का उपयोग करते हैं। WebSocket app-server कनेक्शन को Gateway
एनव API-कुंजी फ़ॉलबैक नहीं मिलता; एक स्पष्ट प्रमाणीकरण प्रोफ़ाइल या दूरस्थ
app-server के अपने खाते का उपयोग करें।

यदि कोई सदस्यता प्रोफ़ाइल Codex उपयोग सीमा तक पहुँचती है, तो OpenClaw
Codex द्वारा रीसेट समय बताए जाने पर उसे दर्ज करता है और उसी Codex रन के लिए
अगली क्रमबद्ध प्रमाणीकरण प्रोफ़ाइल आज़माता है। रीसेट समय बीतने पर, सदस्यता
प्रोफ़ाइल चयनित `openai/gpt-*`
मॉडल या Codex रनटाइम को बदले बिना फिर से पात्र हो जाती है।

जब मूल Codex Plugin कॉन्फ़िगर किए जाते हैं, तो OpenClaw Plugin के स्वामित्व वाले
ऐप को Codex थ्रेड के समक्ष प्रस्तुत करने से पहले कनेक्ट किए गए app-server के माध्यम से
उन Plugin को इंस्टॉल या रीफ़्रेश करता है। `app/list` ऐप
आईडी, अभिगम्यता और मेटाडेटा के लिए सत्य का स्रोत बना रहता है, लेकिन OpenClaw प्रति-थ्रेड
सक्षमता निर्णय का स्वामी है: यदि नीति किसी सूचीबद्ध अभिगम्य ऐप की अनुमति देती है, तो OpenClaw
`thread/start.config.apps[appId].enabled = true` भेजता है, भले ही `app/list`
वर्तमान में उस ऐप को अक्षम बताए। यह पथ अज्ञात आईडी के लिए ऐप
इंस्टॉलेशन की कल्पना नहीं करता; OpenClaw केवल `plugin/install` वाले मार्केटप्लेस Plugin
सक्रिय करता है और फिर इन्वेंट्री रीफ़्रेश करता है।

### परिवेश पृथक्करण

स्थानीय stdio app-server लॉन्च के लिए, OpenClaw `CODEX_HOME` को
प्रति-एजेंट निर्देशिका पर सेट करता है, ताकि Codex कॉन्फ़िगरेशन, प्रमाणीकरण/खाता फ़ाइलें, Plugin कैश/डेटा,
और मूल थ्रेड स्थिति डिफ़ॉल्ट रूप से ऑपरेटर के व्यक्तिगत
`~/.codex` को पढ़ें या लिखें नहीं। OpenClaw सामान्य प्रक्रिया `HOME` को सुरक्षित रखता है;
Codex-रन उपप्रक्रियाएँ अब भी उपयोगकर्ता-होम कॉन्फ़िगरेशन और टोकन खोज सकती हैं, और
Codex साझा `$HOME/.agents/skills` तथा
`$HOME/.agents/plugins/marketplace.json` प्रविष्टियाँ खोज सकता है। 
`appServer.homeScope: "user"` के साथ, OpenClaw इसके बजाय मूल उपयोगकर्ता Codex
होम और उसके मौजूदा खाते का उपयोग करता है तथा OpenClaw प्रमाणीकरण प्रोफ़ाइल इंजेक्ट नहीं करता।

यदि किसी परिनियोजन को अतिरिक्त परिवेश पृथक्करण चाहिए, तो उन
वेरिएबल को `appServer.clearEnv` में जोड़ें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` केवल स्पॉन की गई Codex app-server चाइल्ड
प्रक्रिया को प्रभावित करता है। OpenClaw स्थानीय लॉन्च सामान्यीकरण के दौरान इस सूची से
`CODEX_HOME` और `HOME` हटा देता है: `CODEX_HOME` चयनित
एजेंट या उपयोगकर्ता दायरे की ओर इंगित रहता है, और `HOME` इनहेरिट होता रहता है ताकि उपप्रक्रियाएँ
सामान्य उपयोगकर्ता-होम स्थिति का उपयोग कर सकें।

### डायनेमिक टूल और वेब खोज

Codex डायनेमिक टूल डिफ़ॉल्ट रूप से `searchable` लोडिंग का उपयोग करते हैं। OpenClaw सामान्यतः
Codex-मूल वर्कस्पेस संचालन की नकल करने वाले डायनेमिक टूल प्रस्तुत नहीं करता:
`read`, `write`, `edit`, `apply_patch`, `exec`, `process`, `update_plan`,
`tool_call`, `tool_describe`, `tool_search`, और `tool_search_code`। अधिकांश
शेष OpenClaw एकीकरण टूल, जैसे मैसेजिंग, मीडिया, cron,
ब्राउज़र, नोड, Gateway और `heartbeat_respond`, `openclaw` नेमस्पेस के अंतर्गत
Codex टूल खोज के माध्यम से उपलब्ध होते हैं, जिससे आरंभिक मॉडल
संदर्भ छोटा रहता है। जब कोई सीमित अनुमति-सूची मूल Code Mode को अक्षम करती है, तब
`exec` और `process` के लिए प्रतिबंधित-टर्न शेल फ़ॉलबैक अपवाद है;
रनटाइम अनुमति-सूचियाँ और `codexDynamicToolsExclude` अब भी लागू होते हैं।

OpenClaw `computer`
टूल सहित `catalogMode: "direct-only"` चिह्नित टूल इसके बजाय `openclaw_direct` नेमस्पेस का उपयोग करते हैं। Codex उस नेमस्पेस को
`DirectModelOnly` मानता है, इसलिए वे टूल नेस्टेड Code Mode `tools.*` कॉल से गुज़रने के बजाय
सामान्य और केवल-Code-Mode थ्रेड में सीधे मॉडल को दिखाई देते रहते हैं।

खोज सक्षम होने और कोई प्रबंधित प्रदाता न चुने जाने पर, वेब खोज डिफ़ॉल्ट रूप से Codex के होस्ट किए गए
`web_search` टूल का उपयोग करती है। मूल होस्ट की गई खोज और
OpenClaw का प्रबंधित `web_search` डायनेमिक टूल परस्पर अनन्य हैं, ताकि
प्रबंधित खोज मूल डोमेन प्रतिबंधों को बायपास न कर सके। होस्ट की गई खोज अनुपलब्ध होने, स्पष्ट रूप से अक्षम होने, या
चुने गए प्रबंधित प्रदाता द्वारा प्रतिस्थापित किए जाने पर OpenClaw
प्रबंधित टूल का उपयोग करता है। OpenClaw Codex के स्वतंत्र
`web.run` एक्सटेंशन को अक्षम रखता है, क्योंकि प्रोडक्शन app-server ट्रैफ़िक
इसके उपयोगकर्ता-परिभाषित `web` नेमस्पेस को अस्वीकार करता है। `tools.web.search.enabled: false`
दोनों पथों को अक्षम करता है, और टूल-अक्षम केवल-LLM रन भी ऐसा ही करते हैं। Codex
`"cached"` को वरीयता मानता है और अप्रतिबंधित app-server टर्न के लिए इसे लाइव बाहरी पहुँच में
रिज़ॉल्व करता है। मूल `allowedDomains` सेट होने पर स्वचालित प्रबंधित फ़ॉलबैक
बंद स्थिति में विफल होता है, ताकि अनुमति-सूची को बायपास न किया जा सके।
स्थायी प्रभावी खोज-नीति परिवर्तन अगले टर्न से पहले बाउंड Codex थ्रेड को
रोटेट करते हैं; अस्थायी प्रति-टर्न प्रतिबंध एक अस्थायी
प्रतिबंधित थ्रेड का उपयोग करते हैं और बाद में फिर शुरू करने के लिए मौजूदा बाइंडिंग सुरक्षित रखते हैं।

`sessions_yield` और केवल-संदेश-टूल स्रोत उत्तर सीधे रहते हैं, क्योंकि
वे टर्न-नियंत्रण अनुबंध हैं। `sessions_spawn` खोजने योग्य रहता है, ताकि
Codex का मूल `spawn_agent` प्राथमिक Codex उप-एजेंट सतह बना रहे,
जबकि स्पष्ट OpenClaw या ACP प्रत्यायोजन अब भी
`openclaw` डायनेमिक टूल नेमस्पेस के माध्यम से उपलब्ध है। Heartbeat सहयोग निर्देश
Codex को Heartbeat टर्न समाप्त करने से पहले `heartbeat_respond` खोजने को कहते हैं,
जब टूल पहले से लोड न हो।

`codexDynamicToolsLoading: "direct"` केवल तभी सेट करें जब आप ऐसे कस्टम
Codex app-server से कनेक्ट कर रहे हों जो स्थगित डायनेमिक टूल नहीं खोज सकता, या
पूर्ण टूल पेलोड को डीबग कर रहे हों।

`appServer.networkProxy` स्पष्ट है क्योंकि यह Codex सैंडबॉक्स
अनुबंध को बदलता है। सक्षम होने पर OpenClaw, Codex थ्रेड कॉन्फ़िगरेशन में
`features.network_proxy.enabled` और `default_permissions` भी सेट करता है, ताकि जनरेट की गई
अनुमति प्रोफ़ाइल Codex की प्रबंधित नेटवर्किंग प्रारंभ कर सके। डिफ़ॉल्ट रूप से OpenClaw
प्रोफ़ाइल बॉडी से टकराव-प्रतिरोधी `openclaw-network-<fingerprint>` प्रोफ़ाइल
नाम जनरेट करता है; `profileName` का उपयोग केवल तभी करें, जब एक स्थिर स्थानीय नाम
आवश्यक हो।

```json5
{
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            sandbox: "workspace-write",
            networkProxy: {
              enabled: true,
              domains: {
                "api.openai.com": "allow",
                "blocked.example.com": "deny",
              },
              unixSockets: {
                "/tmp/proxy.sock": "allow",
                "/tmp/blocked.sock": "none",
              },
              allowUpstreamProxy: true,
              proxyUrl: "http://127.0.0.1:3128",
            },
          },
        },
      },
    },
  },
}
```

यदि सामान्य app-server रनटाइम `danger-full-access` होता, तो
`networkProxy` को सक्षम करने से जनरेट की गई अनुमति प्रोफ़ाइल के लिए
वर्कस्पेस-शैली फ़ाइल सिस्टम पहुँच का उपयोग होता है: Codex द्वारा प्रबंधित नेटवर्क प्रवर्तन
सैंडबॉक्सयुक्त नेटवर्किंग है, इसलिए पूर्ण-पहुँच प्रोफ़ाइल आउटबाउंड ट्रैफ़िक को
सुरक्षित नहीं रखेगी। डोमेन प्रविष्टियाँ `allow` या `deny` का उपयोग करती हैं; Unix सॉकेट प्रविष्टियाँ Codex के
`allow` या `none` मानों का उपयोग करती हैं।

### डायनेमिक टूल कॉल टाइमआउट

OpenClaw के स्वामित्व वाली डायनेमिक टूल कॉल
`appServer.requestTimeoutMs` से स्वतंत्र रूप से सीमित होती हैं: Codex `item/tool/call` अनुरोध डिफ़ॉल्ट रूप से 90
सेकंड के OpenClaw वॉचडॉग का उपयोग करते हैं। प्रति-कॉल दिया गया धनात्मक `timeoutMs`
आर्ग्युमेंट उस विशिष्ट टूल बजट को बढ़ाता या घटाता है, जिसकी अधिकतम सीमा 600000 ms है।
जब टूल कॉल अपना टाइमआउट प्रदान नहीं करती, तब `image_generate` टूल
`agents.defaults.imageGenerationModel.timeoutMs` का उपयोग करता है, अन्यथा इमेज जनरेशन के लिए डिफ़ॉल्ट 120 सेकंड है।
मीडिया-अंडरस्टैंडिंग `image` टूल
`tools.media.image.timeoutSeconds` या मीडिया के लिए अपने डिफ़ॉल्ट 60 सेकंड का उपयोग करता है; इमेज
अंडरस्टैंडिंग के लिए यह टाइमआउट स्वयं अनुरोध पर लागू होता है और पहले किए गए
तैयारी कार्य से कम नहीं होता। टाइमआउट होने पर, जहाँ समर्थित हो वहाँ OpenClaw टूल
सिग्नल को निरस्त करता है और Codex को विफल डायनेमिक-टूल प्रतिक्रिया लौटाता है,
जिससे सत्र को `processing` में छोड़ने के बजाय टर्न जारी रह सके।
यह वॉचडॉग बाहरी डायनेमिक `item/tool/call` बजट है; प्रदाता-विशिष्ट
अनुरोध टाइमआउट उस कॉल के भीतर चलते हैं और अपने टाइमआउट अर्थ-विज्ञान को बनाए रखते हैं।

Codex द्वारा किसी टर्न को स्वीकार करने के बाद, और OpenClaw द्वारा टर्न-स्कोप वाले
app-server अनुरोध का उत्तर देने के बाद, हार्नेस अपेक्षा करता है कि Codex वर्तमान टर्न में
प्रगति करे और अंततः नेटिव टर्न को `turn/completed` के साथ पूरा करे। यदि
app-server `appServer.turnCompletionIdleTimeoutMs` तक निष्क्रिय रहता है, तो OpenClaw
सर्वोत्तम प्रयास से Codex टर्न को बाधित करता है, डायग्नोस्टिक टाइमआउट दर्ज करता है, और
OpenClaw सत्र लेन को मुक्त करता है ताकि अनुवर्ती चैट संदेश किसी पुराने
नेटिव टर्न के पीछे कतारबद्ध न हों। उसी टर्न की अधिकांश गैर-टर्मिनल सूचनाएँ
इस छोटे वॉचडॉग को निष्क्रिय कर देती हैं, क्योंकि Codex ने सिद्ध कर दिया है कि टर्न अभी
सक्रिय है।

टूल हैंडऑफ़ लंबे पोस्ट-टूल निष्क्रिय बजट का उपयोग करते हैं: OpenClaw द्वारा
`item/tool/call` प्रतिक्रिया लौटाने के बाद, `commandExecution` जैसे
नेटिव टूल आइटम पूर्ण होने के बाद, रॉ `custom_tool_call_output`
पूर्णताओं के बाद, और पोस्ट-टूल रॉ असिस्टेंट प्रगति, रॉ रीजनिंग
पूर्णताओं या रीजनिंग प्रगति के बाद। कॉन्फ़िगर होने पर गार्ड
`appServer.postToolRawAssistantCompletionIdleTimeoutMs` का उपयोग करता है और
अन्यथा डिफ़ॉल्ट रूप से पाँच मिनट का होता है; यही बजट उस मौन सिंथेसिस अवधि के लिए
प्रगति वॉचडॉग को भी बढ़ाता है जिसमें Codex अगला वर्तमान-टर्न इवेंट उत्सर्जित करने वाला होता है।
रेट-लिमिट अपडेट जैसी वैश्विक app-server सूचनाएँ टर्न-निष्क्रिय प्रगति को रीसेट नहीं करतीं।
रीजनिंग पूर्णताओं, कमेंट्री `agentMessage` पूर्णताओं, और प्री-टूल रॉ रीजनिंग या
असिस्टेंट प्रगति के बाद स्वचालित अंतिम उत्तर आ सकता है, इसलिए वे
सत्र लेन को तुरंत मुक्त करने के बजाय पोस्ट-प्रोग्रेस उत्तर गार्ड का उपयोग करते हैं।

केवल अंतिम/गैर-कमेंट्री पूर्ण `agentMessage` आइटम और प्री-टूल रॉ
असिस्टेंट पूर्णताएँ असिस्टेंट-आउटपुट रिलीज़ को सक्रिय करती हैं: यदि इसके बाद Codex
`turn/completed` के बिना निष्क्रिय हो जाता है, तो OpenClaw सर्वोत्तम प्रयास से नेटिव
टर्न को बाधित करता है और सत्र लेन को मुक्त करता है। यदि कोई अन्य टर्न वॉच उस रिलीज़
रेस को जीत लेता है, तो OpenClaw फिर भी पूर्ण हो चुके अंतिम असिस्टेंट आइटम को स्वीकार करता है,
जब कोई नेटिव अनुरोध, आइटम या डायनेमिक टूल पूर्णता सक्रिय न रह जाए और
असिस्टेंट-आउटपुट रिलीज़ अब भी नवीनतम पूर्ण आइटम से संबंधित हो तथा
बाद में कोई आइटम पूर्ण न हुआ हो। इससे टर्न को दोबारा चलाए बिना पूर्ण टूल कार्य के बाद
अंतिम उत्तर संरक्षित रह सकता है। आंशिक असिस्टेंट डेल्टा, पुराने पूर्ववर्ती उत्तर,
और बाद की खाली पूर्णताएँ योग्य नहीं होतीं।

रीप्ले-सुरक्षित stdio app-server विफलताएँ, जिनमें असिस्टेंट, टूल,
सक्रिय आइटम या साइड-इफ़ेक्ट प्रमाण के बिना टर्न-पूर्णता निष्क्रिय टाइमआउट शामिल हैं,
नए app-server प्रयास पर एक बार पुनः प्रयास की जाती हैं। असुरक्षित टाइमआउट फिर भी अटके हुए
app-server क्लाइंट को सेवानिवृत्त करते और OpenClaw सत्र लेन को मुक्त करते हैं; वे
स्वचालित रूप से रीप्ले करने के बजाय पुराने नेटिव थ्रेड बाइंडिंग को भी साफ़ करते हैं।
पूर्णता-वॉच टाइमआउट Codex-विशिष्ट टाइमआउट टेक्स्ट दिखाते हैं: रीप्ले-सुरक्षित मामलों में
कहा जाता है कि प्रतिक्रिया अपूर्ण हो सकती है, जबकि असुरक्षित मामलों में उपयोगकर्ता को
पुनः प्रयास करने से पहले वर्तमान स्थिति सत्यापित करने को कहा जाता है। सार्वजनिक टाइमआउट
डायग्नोस्टिक्स में अंतिम app-server सूचना विधि, रॉ असिस्टेंट प्रतिक्रिया आइटम
id/type/role, सक्रिय अनुरोध/आइटम गणना और सक्रिय वॉच स्थिति जैसे संरचनात्मक फ़ील्ड
शामिल होते हैं; जब अंतिम सूचना रॉ असिस्टेंट प्रतिक्रिया आइटम होती है, तो उनमें
सीमित असिस्टेंट टेक्स्ट पूर्वावलोकन भी शामिल होता है। इनमें रॉ प्रॉम्प्ट या टूल सामग्री
शामिल नहीं होती।

### स्थानीय परीक्षण env ओवरराइड

- `OPENCLAW_CODEX_APP_SERVER_BIN`, जब
  `appServer.command` सेट नहीं होता, तब प्रबंधित बाइनरी को बायपास करता है।
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

`OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1` हटा दिया गया था। इसके बजाय
`plugins.entries.codex.config.appServer.mode: "guardian"`, या एकबारगी स्थानीय परीक्षण के लिए
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` का उपयोग करें। दोहराए जा सकने वाले डिप्लॉयमेंट के लिए कॉन्फ़िग
बेहतर है, क्योंकि यह Plugin के व्यवहार को Codex हार्नेस सेटअप के शेष भाग वाली
उसी समीक्षित फ़ाइल में रखता है।

## नेटिव Codex Plugin

नेटिव Codex Plugin समर्थन, OpenClaw हार्नेस टर्न वाले उसी Codex थ्रेड में
Codex app-server की अपनी ऐप और Plugin क्षमताओं का उपयोग करता है। OpenClaw,
Codex Plugin को कृत्रिम `codex_plugin_*` OpenClaw
डायनेमिक टूल में रूपांतरित नहीं करता।

`codexPlugins` केवल नेटिव Codex हार्नेस चुनने वाले सत्रों को प्रभावित करता है।
इसका बिल्ट-इन हार्नेस रन, सामान्य OpenAI प्रदाता रन, ACP
वार्तालाप बाइंडिंग या अन्य हार्नेस पर कोई प्रभाव नहीं पड़ता।

न्यूनतम माइग्रेट किया गया कॉन्फ़िग:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

थ्रेड ऐप कॉन्फ़िग की गणना तब होती है जब OpenClaw कोई Codex हार्नेस
सत्र स्थापित करता है या पुराने Codex थ्रेड बाइंडिंग को बदलता है; प्रत्येक टर्न पर इसकी
पुनर्गणना नहीं होती। `codexPlugins` बदलने के बाद, `/new`, `/reset` का उपयोग करें, या
Gateway को पुनः आरंभ करें ताकि भविष्य के Codex हार्नेस सत्र अपडेट किए गए ऐप
सेट के साथ शुरू हों।

माइग्रेशन पात्रता, ऐप इन्वेंट्री, विध्वंसकारी कार्रवाई नीति,
एलिसिटेशन और नेटिव Plugin डायग्नोस्टिक्स के लिए,
[नेटिव Codex Plugin](/hi/plugins/codex-native-plugins) देखें।

OpenAI-पक्षीय ऐप और Plugin पहुँच, साइन-इन किए गए Codex
खाते द्वारा और Business तथा Enterprise/Edu वर्कस्पेस के लिए, वर्कस्पेस ऐप
नियंत्रणों द्वारा नियंत्रित होती है। OpenAI के खाते और वर्कस्पेस-नियंत्रण के अवलोकन के लिए
[अपने ChatGPT प्लान के साथ Codex का उपयोग करना](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan)
देखें।

## कंप्यूटर उपयोग

कंप्यूटर उपयोग की अपनी सेटअप गाइड है:
[Codex कंप्यूटर उपयोग](/hi/plugins/codex-computer-use)।

संक्षिप्त रूप: OpenClaw डेस्कटॉप-नियंत्रण ऐप को वेंडर नहीं करता और न ही
स्वयं डेस्कटॉप कार्रवाइयाँ निष्पादित करता है। यह Codex app-server तैयार करता है, सत्यापित करता है कि
`computer-use` MCP सर्वर उपलब्ध है, और फिर Codex-मोड टर्न के दौरान नेटिव
MCP टूल कॉल का स्वामित्व Codex को देता है।

## रनटाइम सीमाएँ

Codex हार्नेस केवल निम्न-स्तरीय एम्बेडेड एजेंट एक्ज़ीक्यूटर को बदलता है।

- OpenClaw डायनेमिक टूल समर्थित हैं। Codex, OpenClaw से उन टूल को निष्पादित करने के लिए कहता है,
  इसलिए OpenClaw निष्पादन पथ में बना रहता है।
- Codex-नेटिव शेल, पैच, MCP और नेटिव ऐप टूल का स्वामित्व Codex के पास है।
  OpenClaw समर्थित रिले के माध्यम से चयनित नेटिव इवेंट देख या अवरुद्ध कर सकता है,
  लेकिन यह नेटिव टूल आर्ग्युमेंट को फिर से नहीं लिखता।
- Codex नेटिव Compaction का स्वामी है। OpenClaw चैनल इतिहास,
  खोज, `/new`, `/reset`, और भविष्य में मॉडल या हार्नेस
  बदलने के लिए ट्रांस्क्रिप्ट मिरर रखता है, लेकिन Codex Compaction को OpenClaw या
  कॉन्टेक्स्ट-इंजन सारांशकर्ता से प्रतिस्थापित नहीं करता।
- मीडिया जनरेशन, मीडिया अंडरस्टैंडिंग, TTS, अनुमोदन और मैसेजिंग-टूल
  आउटपुट संबंधित OpenClaw प्रदाता/मॉडल सेटिंग के माध्यम से जारी रहते हैं।
- `tool_result_persist` OpenClaw के स्वामित्व वाले ट्रांस्क्रिप्ट टूल परिणामों पर लागू होता है,
  Codex-नेटिव टूल परिणाम रिकॉर्ड पर नहीं।

हुक लेयर, समर्थित V1 सतहों, नेटिव अनुमति प्रबंधन, कतार
संचालन, Codex फ़ीडबैक अपलोड कार्यप्रणाली और Compaction विवरण के लिए,
[Codex हार्नेस रनटाइम](/hi/plugins/codex-harness-runtime) देखें।

## समस्या निवारण

**Codex सामान्य `/model` प्रदाता के रूप में दिखाई नहीं देता:** नए
कॉन्फ़िग के लिए यह अपेक्षित है। कोई `openai/gpt-*` मॉडल चुनें,
`plugins.entries.codex.enabled` सक्षम करें, और जाँचें कि क्या `plugins.allow`,
`codex` को बाहर रखता है।

**OpenClaw, Codex के बजाय बिल्ट-इन हार्नेस का उपयोग करता है:** पुष्टि करें कि प्रभावी
रूट बिल्कुल आधिकारिक HTTPS Platform Responses या ChatGPT Responses रूट है,
इसमें कोई लिखित अनुरोध ओवरराइड नहीं है, और Codex Plugin इंस्टॉल तथा
सक्षम है। केवल `openai/gpt-*` प्रीफ़िक्स पर्याप्त नहीं है। परीक्षण के दौरान
सख्त प्रमाण के लिए, प्रदाता या मॉडल `agentRuntime.id: "codex"` सेट करें; रूट या हार्नेस
असंगत होने पर बाध्य Codex फ़ॉलबैक करने के बजाय विफल हो जाता है।

**OpenAI Codex रनटाइम API-key पथ पर फ़ॉलबैक करता है:** एक संशोधित
Gateway अंश एकत्र करें जो मॉडल, रनटाइम, चयनित प्रदाता और
विफलता दिखाता हो। प्रभावित सहयोगियों से उनके OpenClaw होस्ट पर यह केवल-पढ़ने योग्य कमांड चलाने को कहें:

```bash
(
  pattern='openai/gpt-5\.[45]|openai[-]codex|agentRuntime(\.id)?|harnessRuntime|Runtime: OpenAI Codex|legacy OpenAI Codex prefix|resolveSelectedOpenAIRuntimeProvider|candidateProvider[": ]+openai|status[": ]+401|Incorrect API key|No API key|api-key path|API-key path|OAuth'

  if ls /tmp/openclaw/openclaw-*.log >/dev/null 2>&1; then
    grep -E -i -n "$pattern" /tmp/openclaw/openclaw-*.log 2>/dev/null || true
  else
    journalctl --user -u openclaw-gateway --since today --no-pager 2>/dev/null \
      | grep -E -i "$pattern" || true
  fi
) | sed -E \
    -e 's/(Authorization: Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(api[_ -]?key[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/(OPENAI_API_KEY[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/sk-[A-Za-z0-9_-]{12,}/sk-[REDACTED]/g' \
    -e 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL-REDACTED]/g' \
  | tail -200
```

उपयोगी अंशों में सामान्यतः `openai/gpt-5.6-sol` या `openai/gpt-5.6-luna`,
`Runtime: OpenAI Codex`, `agentRuntime.id` या `harnessRuntime`,
`candidateProvider: "openai"`, और कोई `401`, `Incorrect API key`, या
`No API key` परिणाम शामिल होते हैं। सुधारे गए रन में सामान्य OpenAI API-key
विफलता के बजाय OpenAI OAuth पथ दिखाई देना चाहिए।

**पुराना Codex मॉडल रेफ़रेंस कॉन्फ़िग शेष है:** `openclaw doctor --fix` चलाएँ।
Doctor पुराने मॉडल रेफ़रेंस को `openai/*` में फिर से लिखता है, पुराने सत्र और
संपूर्ण-एजेंट रनटाइम पिन हटाता है, और मौजूदा auth-profile ओवरराइड संरक्षित रखता है।

**app-server अस्वीकार कर दिया जाता है:** Codex app-server `0.143.0` या नया उपयोग करें।
समान-संस्करण प्रीरिलीज़ या बिल्ड-सफ़िक्स वाले संस्करण, जैसे
`0.143.0-alpha.2` या `0.143.0+custom`, अस्वीकार किए जाते हैं क्योंकि OpenClaw
स्थिर `0.143.0` प्रोटोकॉल न्यूनतम सीमा का परीक्षण करता है।

**`/codex status` कनेक्ट नहीं हो सकता:** जाँचें कि `codex` Plugin
सक्षम है, allowlist कॉन्फ़िगर होने पर `plugins.allow` में यह शामिल है,
और कोई भी कस्टम `appServer.command`, `url`, `authToken`, या
हेडर मान्य हैं।

**मॉडल खोज धीमी है:** `plugins.entries.codex.config.discovery.timeoutMs` को कम करें
या खोज अक्षम करें।
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference#model-discovery) देखें।

**WebSocket ट्रांसपोर्ट तुरंत विफल हो जाता है:** `appServer.url`,
`authToken`, हेडर जाँचें और सुनिश्चित करें कि रिमोट app-server समान Codex
app-server प्रोटोकॉल संस्करण का उपयोग करता है।

**नेटिव शेल या पैच टूल `Native hook relay
unavailable` के साथ अवरुद्ध हैं:** Codex थ्रेड अब भी ऐसे नेटिव हुक रिले
आईडी का उपयोग करने का प्रयास कर रहा है जिसे OpenClaw में अब पंजीकृत नहीं किया गया है।
यह नेटिव Codex हुक ट्रांसपोर्ट की समस्या है, ACP बैकएंड, प्रदाता, GitHub या शेल-कमांड
की विफलता नहीं। प्रभावित चैट में `/new` या `/reset` के साथ नया सत्र शुरू करें,
फिर कोई अहानिकर कमांड दोबारा आज़माएँ। यदि यह एक बार काम करता है लेकिन अगली नेटिव टूल
कॉल फिर विफल हो जाती है, तो `/new` को केवल अस्थायी समाधान मानें: Codex app-server या
OpenClaw Gateway को पुनः आरंभ करने के बाद प्रॉम्प्ट को नए सत्र में कॉपी करें, ताकि पुराने थ्रेड
हटा दिए जाएँ और नेटिव हुक पंजीकरण फिर से बनाए जाएँ।

**Codex टूल कॉल बहुत अधिक अल्पकालिक हुक प्रक्रियाएँ बनाती हैं:** 
`plugins.entries.codex.config.appServer.loopDetectionPreToolUseRelay: false` सेट करें
और Gateway को पुनः आरंभ करें। इससे केवल OpenClaw लूप पहचान और उसके no-policy मार्कर के लिए
उपयोग की जाने वाली Codex `PreToolUse` उपप्रक्रिया अक्षम होती है। आवश्यक
`before_tool_call` और विश्वसनीय-टूल नीति रिले सक्षम रहते हैं।

**कोई गैर-Codex मॉडल अंतर्निहित हार्नेस का उपयोग करता है:** यह तब तक अपेक्षित है जब तक प्रदाता
या मॉडल रनटाइम नीति उसे किसी अन्य हार्नेस पर रूट न करे। सामान्य गैर-OpenAI
प्रदाता संदर्भ `auto` मोड में अपने सामान्य प्रदाता पथ पर बने रहते हैं।

**Computer Use इंस्टॉल है लेकिन टूल नहीं चलते:** नए सत्र से
`/codex computer-use status` जाँचें। यदि कोई टूल
`Native hook relay unavailable` रिपोर्ट करता है, तो ऊपर दिए गए नेटिव हुक रिले पुनर्प्राप्ति उपाय का उपयोग करें।
[Codex Computer Use](/hi/plugins/codex-computer-use#troubleshooting) देखें।
