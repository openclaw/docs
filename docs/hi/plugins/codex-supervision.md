Codex पर्यवेक्षण आधिकारिक `codex` plugin की एक वैकल्पिक क्षमता है। यह
Gateway कंप्यूटर और विकल्प सक्षम किए गए युग्मित कंप्यूटरों के गैर-संग्रहीत Codex CLI,
VS Code, Atlas और ChatGPT स्रोत सत्रों को सामान्य सत्र साइडबार और Chat फलक में
दिखाता है।

आरंभिक रिलीज़ जानबूझकर स्वामित्व को सीमित रखती है:

- संग्रहीत या निष्क्रिय स्थानीय सत्र अपने सीमित स्थायी उपयोगकर्ता और सहायक इतिहास से
  मॉडल-लॉक किया हुआ OpenClaw Chat बना सकता है। पहला संदेश एक मूल स्नैपशॉट फ़ोर्क
  बनाता है, फिर पूर्ण Codex हार्नेस थ्रेड को ठीक उसी मॉडल और प्रदाता के साथ शुरू करता है
  जिसे Codex App Server ने उस फ़ोर्क के लिए चुना था। बाद के टर्न कैनोनिकल मूल थ्रेड की
  स्थायी जोड़ी को पुनर्स्थापित करते हैं, जबकि पर्यवेक्षित बाइंडिंग OpenClaw को कोई अन्य
  रनटाइम, मॉडल या फ़ॉलबैक प्रतिस्थापित करने से रोकती है। एक अलग मूल Codex नियंत्रण
  फिर भी उस स्थायी जोड़ी को बदल सकता है। पहले से बनाई गई शाखा अपना मौजूदा Chat खोलती है।
- किसी अन्य Codex प्रक्रिया से खोजे गए संग्रहीत सत्र की लाइव गतिविधि अज्ञात होती है।
  उसकी शाखा बनाई जा सकती है, या उसे केवल तब संग्रहीत किया जा सकता है जब ऑपरेटर पुष्टि
  करे कि कोई अन्य Codex क्लाइंट उसका उपयोग नहीं कर रहा है।
- सक्रिय स्रोत दृश्यमान रहता है, लेकिन उसका वर्तमान टर्न पूरा होने तक उससे शाखा नहीं
  बनाई जा सकती और न ही उसे संग्रहीत किया जा सकता है। यदि उसका पर्यवेक्षित Chat पहले से
  मौजूद है, तो **Chat खोलें** उपलब्ध रहता है।
- युग्मित Node पर मौजूद सत्र अपनी स्थायी ट्रांसक्रिप्ट को सीमित, कर्सर-पृष्ठांकित
  App Server रीड के माध्यम से उपलब्ध कराता है। दूरस्थ निरंतरता के लिए भविष्य का
  स्ट्रीमिंग Node ब्रिज आवश्यक है; दूरस्थ संग्रहण के लिए अतिरिक्त रूप से रनर-स्वामित्व
  लीज़ या समकक्ष फ़ेंसिंग आवश्यक है।
- संग्रहीत सत्र सूचीबद्ध नहीं होते। संग्रहीत या निष्क्रिय स्थानीय सत्र को केवल तब
  संग्रहीत किया जा सकता है जब ऑपरेटर पुष्टि करे कि कोई अन्य Codex क्लाइंट उसका उपयोग
  नहीं कर रहा है।

## शुरू करने से पहले

- Gateway पर आधिकारिक `@openclaw/codex` plugin इंस्टॉल करें। Codex सुविधाएँ सक्षम
  करने पर OpenClaw macOS ऐप इसे इंस्टॉल कर सकता है; CLI इंस्टॉलेशन
  `openclaw plugins install @openclaw/codex` चला सकते हैं।
- जिस प्रत्येक कंप्यूटर के सत्र सूचीबद्ध करने हैं, उस पर Codex Desktop या Codex CLI
  इंस्टॉल करें और साइन इन करें।
- दूरस्थ कंप्यूटरों को OpenClaw Node के रूप में युग्मित करें। प्रत्येक कंप्यूटर पर
  स्थानीय रूप से विकल्प सक्षम होना चाहिए; केवल Gateway पर पर्यवेक्षण सक्षम करने से
  दूसरे Node को प्राधिकरण नहीं मिलता।
- स्वामी-नियंत्रित Gateway का उपयोग करें। सत्र शीर्षक, कार्यशील डायरेक्टरियाँ और Git
  शाखाएँ संवेदनशील परियोजना जानकारी प्रकट कर सकती हैं।

## पर्यवेक्षण सक्षम करें

निर्देशित `openclaw onboard` और macOS का प्रथम-रन सेटअप मूल Codex इंस्टॉलेशन का
पता लगाने और चयनित इन्फ़रेंस बैकएंड को सफलतापूर्वक सक्रिय करने के बाद Codex पर्यवेक्षण
इंस्टॉल और सक्षम करने का प्रयास करते हैं। Codex का प्राथमिक बैकएंड होना आवश्यक नहीं है।
जब यह अवसरानुकूल plugin सक्रियण सफल होता है, तब पर्यवेक्षण उपलब्ध हो जाता है।
पर्यवेक्षण के पहली बार कनेक्ट होने पर App Server की उपलब्धता जाँची जाती है। Codex
plugin को स्पष्ट रूप से अक्षम करना या नीति द्वारा अवरोधित करना अवसरानुकूल सक्रियण को
रोकता है, और मौजूदा स्पष्ट `supervision.enabled: false` एजेंट-सामना करने वाले पर्यवेक्षण टूल को
अक्षम करता है; जब भी Codex plugin सक्रिय हो, ऑपरेटर कैटलॉग पंजीकृत रहता है, जब तक कि
`sessionCatalog.enabled: false` उसे अक्षम न करे। यह अलग स्विच Codex प्रदाता, हार्नेस और एजेंट-सामना
करने वाली पर्यवेक्षण नीति को अपरिवर्तित रखता है, साथ ही इस होस्ट से युग्मित-Node कैटलॉग
सूची/रीड कमांड भी हटा देता है। मौजूदा इंस्टॉलेशन यही क्षमता मैन्युअल रूप से सक्षम कर सकते हैं:

`openclaw.json` में `codex` plugin और उसकी पर्यवेक्षण क्षमता सक्षम करें:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          supervision: {
            enabled: true,
          },
        },
      },
    },
  },
}
```

यदि `plugins.allow` मौजूद है, तो `codex` शामिल करें। plugin सक्रियण बदलने
के बाद Gateway पुनः आरंभ करें।

कोई स्पष्ट `appServer` कनेक्शन सेटिंग न होने पर, पर्यवेक्षण मूल उपयोगकर्ता Codex
होम के विरुद्ध एक अलग प्रबंधित stdio पर्यवेक्षण कनेक्शन का उपयोग करता है। सामान्य Codex
हार्नेस डिफ़ॉल्ट रूप से एजेंट-स्कोप्ड रहता है। इससे मूल सत्र दोनों ऐप में दृश्यमान होते हैं,
लेकिन सामान्य OpenClaw टर्न मूल Codex स्थिति साझा नहीं करते। यदि हार्नेस को भी वह स्थिति
साझा करनी चाहिए, तो `appServer.homeScope: "user"` को स्पष्ट रूप से सेट करें। पर्यवेक्षण स्पष्ट
`appServer` कनेक्शन सेटिंग को अपने स्थानीय उपयोगकर्ता-होम डिफ़ॉल्ट से बदलने के
बजाय उनका सम्मान करता है।

**Codex** साइडबार समूह से अपनाया गया Chat सामान्य हार्नेस सत्र नहीं होता। उसकी निजी
पर्यवेक्षण बाइंडिंग स्रोत रीड, कैनोनिकल शाखा निर्माण, इतिहास इंजेक्शन और बाद के प्रत्येक टर्न
के लिए पर्यवेक्षण कनेक्शन का उपयोग करती है। डिफ़ॉल्ट स्थानीय कनेक्शन के साथ, यह अन्य सत्रों
के डिफ़ॉल्ट को बदले बिना मूल उपयोगकर्ता Codex होम, प्रमाणीकरण और प्रदाता कॉन्फ़िगरेशन को
सुरक्षित रखता है। निगरानी किए गए अपनाए गए Chat [सत्र स्थिति जागरूकता](/hi/concepts/session-state)
में भी भाग लेते हैं।

डिफ़ॉल्ट स्थानीय पर्यवेक्षण कनेक्शन के लिए स्टोर मूल Codex क्लाइंट के साथ साझा होता है।
OpenClaw यह नहीं मानता कि कोई अन्य क्लाइंट वही लाइव App Server प्रक्रिया साझा करता है,
और मूल स्थिति का स्वामित्व प्रक्रिया-स्थानीय होता है। इसलिए जिस थ्रेड को उसका पर्यवेक्षण
App Server `notLoaded` के रूप में रिपोर्ट करता है, उसे वह निष्क्रिय नहीं, बल्कि
**संग्रहीत / गतिविधि अज्ञात** मानता है।

प्रत्येक ऐसे हेडलेस Node होस्ट पर यही विकल्प सक्षम करें जिसके सत्र दिखाई देने चाहिए।
मूल OpenClaw macOS ऐप युग्मित Gateway को अपना Codex कैटलॉग विज्ञापित करते समय यही
स्थानीय सेटिंग पढ़ता है। वह युग्मित मूल Mac कैटलॉग केवल अनसेट या स्पष्ट
`appServer.homeScope: "user"` के साथ डिफ़ॉल्ट या स्पष्ट `appServer.transport: "stdio"` का समर्थन करता है।
उस stdio प्रक्रिया के लिए `command`, `args` और
`clearEnv` का सम्मान किया जाता है। यदि Mac कॉन्फ़िगरेशन
`"unix"`, `"websocket"` या `homeScope: "agent"` चुनता है, तो ऐप कैटलॉग
क्षमता या कमांड विज्ञापित नहीं करता, और पुराना प्रत्यक्ष आह्वान उपयोगकर्ता Codex होम को
उजागर करने या कोई अलग स्थानीय stdio App Server शुरू करने के बजाय विफल हो जाता है।

नया विज्ञापित Node कमांड Node की स्वीकृत कमांड सतह बदलता है। Gateway होस्ट से अपडेट
स्वीकार करें:

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
```

गैर-संग्रहीत Codex सत्र मुख्य Control UI साइडबार में भी होस्ट के अनुसार समूहीकृत होकर
दिखाई देते हैं। किसी एक को चुनकर उसकी स्थायी ट्रांसक्रिप्ट पढ़ें। व्यूअर नवीनतम Codex
`thread/turns/list` API का `itemsView: "full"` के साथ उपयोग करता है और प्रति अनुरोध
अधिकतम 20 टर्न लोड करता है; **पुरानी ट्रांसक्रिप्ट प्रविष्टियाँ लोड करें** नवीनतम पृष्ठ से
अपारदर्शी App Server कर्सर का अनुसरण करता है। लोड किए गए पृष्ठ कालानुक्रमिक क्रम में
रेंडर होते हैं। व्यूअर कभी भी असीमित `thread/read` इतिहास लोड नहीं करता। 20 MiB
ट्रांसपोर्ट सुरक्षा सीमा से बड़ा पृष्ठ Node या Gateway कनेक्शन को जोखिम में डालने के बजाय
सुरक्षित रूप से विफल होता है।

सामान्य सत्र साइडबार में **Codex** समूह खोलें। यह उन्हीं सत्रों को होस्ट के अनुसार समूहीकृत
करके सूचीबद्ध करता है। **और सत्र लोड करें** पुराने पंक्ति वाले प्रत्येक होस्ट से अगला पृष्ठ
जोड़ता है, और जोड़ी गई वे पंक्तियाँ साइडबार के आवधिक रीफ़्रेश के बाद भी बनी रहती हैं।
प्रत्येक होस्ट अपनी मूल सूची तैयार होते ही दिखाई देता है। Node कनेक्टिविटी बदलने पर, फिर से
फ़ोकस मिलने पर और अधिकतम प्रत्येक 30 सेकंड में दृश्यमान पृष्ठ का मिलान किया जाता है;
परिणाम बदलने पर अगला मिलान अधिक शीघ्र होता है। इसलिए Codex Desktop, CLI या किसी अन्य
मूल क्लाइंट में बनाए गए सत्र पूर्ण पृष्ठ रीलोड के बिना दिखाई देते हैं। पहला पृष्ठ Codex के अपने
सबसे हाल में अपडेट किए गए क्रम का अनुसरण करता है, इसलिए नया बनाया गया मूल सत्र तुरंत
पात्र होता है।
प्रत्येक लौटाया गया खोज पृष्ठ क्वेरी को App Server पर भेजने के बजाय प्रति होस्ट मूल पृष्ठों
की सीमित संख्या स्कैन करता है, क्योंकि मूल खोज ट्रांसक्रिप्ट पूर्वावलोकनों से भी मिलान कर
सकती है।

होस्ट उपलब्धता और थ्रेड स्थिति अलग-अलग हैं। **ऑफ़लाइन** या **अनुपलब्ध** होस्ट रीफ़्रेश
का वर्णन करता है; अनुपलब्ध होस्ट कोई नई सत्र पंक्ति नहीं लौटाता और थ्रेड की मूल स्थिति को
`offline` में नहीं बदलता। सत्र पंक्तियाँ `idle`,
`active`, `notLoaded` या त्रुटि जैसी Codex स्थितियों का उपयोग करती
हैं। विफल होस्ट स्वस्थ होस्ट के परिणामों को नहीं छिपाता।

साइडबार चेतावनी में कैटलॉग त्रुटि कोड और सुरक्षित अंतर्निहित Gateway त्रुटि शामिल होती है।
Codex को अक्षम किए बिना खोज अक्षम करने के लिए **Settings > Automation > Plugins >
Codex > Native Session Discovery** खोलें। `NODE_LIST_FAILED` के लिए
`openclaw nodes list` और **Settings > Devices** की तुलना करें; विस्तृत कारण उस
पेयरिंग-स्टोर, Node-रजिस्ट्री, अनुमति या Gateway जीवनचक्र विफलता की पहचान करता है जिसे
सुधारने की आवश्यकता है।

## ऑपरेटर CLI का उपयोग करें

टर्मिनल CLI वही गैर-संग्रहीत कैटलॉग और Gateway-स्थानीय शाखा तथा संग्रहण कार्रवाइयाँ
उपलब्ध कराता है:

```bash
openclaw codex sessions [--search <text>] [--host <id>] [--limit <count>] [--cursor <cursor>] [--json] [--url <url>] [--token <token>] [--timeout <ms>] [--expect-final]
openclaw codex continue <thread-id> [--json] [--url <url>] [--token <token>] [--timeout <ms>] [--expect-final]
openclaw codex archive <thread-id> --confirm-no-other-runner [--json] [--url <url>] [--token <token>] [--timeout <ms>] [--expect-final]
```

`openclaw codex sessions` विकल्प:

- `--search <text>` सत्र शीर्षकों में अक्षर-आकार की परवाह किए बिना खोज करता है।
- `--host <id>` प्रतिक्रिया को एक स्थिर कैटलॉग होस्ट तक सीमित करता है, जैसे
  `gateway:local` या `node:<node-id>`।
- `--limit <count>` प्रति होस्ट 1 से 100 पंक्तियाँ सेट करता है; डिफ़ॉल्ट 50 है।
- `--cursor <cursor>` एक होस्ट पृष्ठ जारी रखता है और इसलिए
  `--host` आवश्यक है।
- `--json` संरचित Gateway प्रतिक्रिया प्रिंट करता है।

सभी तीन कमांड Gateway क्लाइंट से `--url`, `--token` और
`--timeout <ms>` प्राप्त करते हैं। सत्र सूचीकरण का डिफ़ॉल्ट 75,000 ms है, ताकि ठंडे
युग्मित-Node कैटलॉग पूरे हो सकें; जारी रखने और संग्रहण का डिफ़ॉल्ट 30,000 ms है। वे साझा
`--expect-final` स्विच भी उपलब्ध कराते हैं, जो इन यूनरी पर्यवेक्षण RPC को नहीं बदलता।
प्रत्येक कमांड के लिए `operator.write` Gateway स्कोप आवश्यक है।
प्रत्येक उपकमांड पर मानक `-h, --help` आउटपुट उपलब्ध है।
कोई संग्रहीत या संग्रहीत-को-शामिल-करें विकल्प नहीं है। `sessions` युग्मित होस्ट
सूचीबद्ध कर सकता है, लेकिन `continue` और `archive` हमेशा
`gateway:local` को लक्षित करते हैं; युग्मित पंक्तियाँ केवल सूचीकरण के लिए हैं। संग्रहण
के लिए हमेशा `--confirm-no-other-runner` आवश्यक है।

ये शेल कमांड Chat के भीतर मौजूद `/codex` रनटाइम कमांड से अलग हैं।
`/codex threads [filter]` वर्तमान वार्तालाप कनेक्शन के लिए उपलब्ध App Server थ्रेड सूचीबद्ध
करता है। `/codex sessions --host <node>` एक Node पर फिर से शुरू किए जा सकने वाले Codex CLI
सत्र फ़ाइलों को सूचीबद्ध करता है, पर्यवेक्षण फ़्लीट कैटलॉग को नहीं। `/codex
resume`
और `/codex bind` सुरक्षित पर्यवेक्षित शाखा बनाने के बजाय वर्तमान वार्तालाप को
संलग्न करते हैं, और मॉडल-लॉक किया हुआ पर्यवेक्षित Chat उन बाइंडिंग परिवर्तनों को अस्वीकार
करता है। कोई `/codex continue` या `/codex archive` रनटाइम कमांड नहीं है।

## स्थानीय सत्र से शाखा बनाएँ

Gateway कंप्यूटर की संग्रहीत या निष्क्रिय पंक्ति पर **शाखा के रूप में जारी रखें** चुनें।
OpenClaw एक सामान्य Chat प्रविष्टि बनाता है, स्रोत के अंतिम स्थायी टर्मिनल टर्न—पूर्ण,
बाधित या विफल—तक सीमित उपयोगकर्ता और सहायक इतिहास को मिरर करता है, लंबित हार्नेस
शाखा दर्ज करता है और Chat खोलता है। सामान्य मॉडल पिकर लॉक होता है, लेकिन अभी कोई
ठोस मॉडल या प्रदाता नहीं चुना गया होता। स्रोत पुनः शुरू नहीं होता और कैनोनिकल हार्नेस
थ्रेड भी अभी शुरू नहीं होता। कार्रवाई दोहराने पर दूसरी शाखा बनाने के बजाय मौजूदा Chat
खुलता है।

मिरर सबसे नया दृश्यमान अंतिम भाग रखता है जो इन तीनों सीमाओं में समाता है: अधिकतम 200
उपयोगकर्ता या सहायक संदेश, कुल 512 KiB UTF-8 टेक्स्ट और प्रति संदेश 64 KiB। बहुत बड़े
संदेशों को मार्कर के साथ छोटा किया जाता है और सीमा पूरी होने पर पुराने संदेश छोड़ दिए जाते
हैं। इमेज या स्थानीय-इमेज इनपुट शाब्दिक `[Image attachment]` प्लेसहोल्डर बन जाता है;
इमेज डेटा और स्थानीय पथ कॉपी नहीं किए जाते।

काम शुरू करने के लिए पहला सामान्य Chat संदेश भेजें। Codex हार्नेस वास्तविक अनुमोदन, जानकारी-अनुरोध, इवेंट और डिलीवरी हैंडलर इंस्टॉल करता है। यह मॉडल या प्रोवाइडर ओवरराइड दिए बिना स्रोत स्नैपशॉट को पिन करने के लिए पर्यवेक्षण कनेक्शन पर एक अस्थायी नेटिव फ़ोर्क का उपयोग करता है। Codex App Server अपने वर्तमान नेटिव कॉन्फ़िगरेशन से दोनों का चयन करता है और वास्तविक चयन लौटाता है। उसी कनेक्शन पर, OpenClaw ठीक उसी लौटाई गई जोड़ी के साथ अपने cwd और रनटाइम नीति के अंतर्गत कैनोनिकल `appServer`-स्रोत वाला पूर्ण हार्नेस थ्रेड शुरू करता है, सीमित दृश्यमान इतिहास इंजेक्ट करता है और अस्थायी फ़ोर्क को आर्काइव करता है। कैनोनिकल थ्रेड में OpenClaw हार्नेस के सभी टूल उपलब्ध होते हैं। यह दृश्यमान-इतिहास की ब्रांच है, पूर्ण नेटिव रोलआउट क्लोन नहीं: स्रोत का रीजनिंग, टूल कॉल और टूल परिणाम शामिल नहीं किए जाते। यह और इसके बाद का प्रत्येक टर्न किसी अन्य OpenClaw मॉडल रनटाइम या सामान्य एजेंट-होम हार्नेस के बजाय पर्यवेक्षित Codex कनेक्शन पर ही रहता है।

लौटाया गया चयन स्रोत के ऐतिहासिक मॉडल का प्रमाण नहीं है। यदि वर्तमान नेटिव कॉन्फ़िगरेशन स्रोत के अंतिम टर्न के लिए दर्ज मॉडल से अलग है, तो Codex अपनी सामान्य मॉडल-अंतर चेतावनी जारी करता है। OpenClaw कैनोनिकल थ्रेड शुरू करने के लिए लौटाई गई जोड़ी का उपयोग करता है। Codex उस कैनोनिकल थ्रेड के नेटिव मॉडल और प्रोवाइडर को बनाए रखता है और बाद के रिज़्यूम उन्हें सुरक्षित रखते हैं, क्योंकि OpenClaw मॉडल और प्रोवाइडर ओवरराइड नहीं देता। यदि कैनोनिकल थ्रेड को किसी अलग नेटिव Codex नियंत्रण के माध्यम से बदला जाता है, तो OpenClaw Codex के बनाए रखे गए चयन को स्वीकार करता है। OpenClaw कभी भी अपने बाहरी मॉडल या फ़ॉलबैक श्रृंखला को प्रतिस्थापित नहीं करता।

पर्यवेक्षित मॉडल-लॉक किया हुआ Chat हटाया नहीं जा सकता, मॉडल नहीं बदल सकता, `/new` या `/reset` का उपयोग नहीं कर सकता, Gateway सत्र-रीसेट कार्रवाई को लागू नहीं कर सकता या सामान्य **Fork session** कार्रवाई का उपयोग नहीं कर सकता। `/codex model <model>`, `/codex
bind`, `/codex resume` (जिसमें `--bind here` वाला Node सत्र भी शामिल है) और `/codex detach` या `/codex unbind` में परिवर्तन भी अस्वीकार कर दिए जाते हैं, क्योंकि वे लॉक की गई नेटिव बाइंडिंग को बदल देंगे या साफ़ कर देंगे। `/codex model` क्वेरी और `/codex fast`, `/codex permissions` तथा `/codex threads` उपलब्ध रहते हैं। जब आपको अलग मॉडल या नया थ्रेड चाहिए, तो कोई अन्य सामान्य सत्र शुरू करें।

इस Chat के लिए पर्यवेक्षण सक्षम रखें। यदि पर्यवेक्षण अक्षम कर दिया जाता है या उसकी संग्रहीत कनेक्शन बाइंडिंग अनुपलब्ध या असंगत हो जाती है, तो टर्न किसी सामान्य एजेंट-होम सत्र में जाने के बजाय सुरक्षित रूप से विफल हो जाता है।

`codex` Plugin को अक्षम या अनइंस्टॉल करने से उस स्वामित्व से मुक्ति नहीं मिलती और न ही Chat किसी अन्य मॉडल के लिए योग्य बनता है। लॉक किया हुआ Chat सुरक्षित रहता है, लेकिन अनुपलब्ध हो जाता है; उसे रिज़्यूम करने के लिए उसी Plugin को फिर से इंस्टॉल या सक्षम करें और Gateway को पुनः आरंभ करें। जानबूझकर अपनाया गया यह सुरक्षित-विफल व्यवहार रिटेंशन क्लीनअप या Plugin की अस्थायी अनुपलब्धता के कारण नेटिव बाइंडिंग को चुपचाप अनाथ होने से रोकता है।

`codex_threads` एजेंट टूल भी इसी सीमा का पालन करता है। यह कोई अलग फ़ोर्क संलग्न नहीं कर सकता या Chat के बाउंड नेटिव थ्रेड को आर्काइव नहीं कर सकता। सूची और केवल-मेटाडेटा रीड उपलब्ध रहते हैं। रॉ ट्रांसक्रिप्ट पढ़ने के लिए `allowRawTranscripts` आवश्यक है।
रॉ एक्सेस अक्षम होने पर `codex_threads` सूची खोज को भी अस्वीकार कर देता है, क्योंकि नेटिव खोज में ट्रांसक्रिप्ट पूर्वावलोकन शामिल होते हैं; Control UI और ऑपरेटर CLI अब भी सीमित, केवल-शीर्षक खोज प्रदान करते हैं। नाम बदलने, अनआर्काइव करने, अलग फ़ोर्क बनाने और किसी असंबंधित अस्वामित्व वाले थ्रेड को आर्काइव करने के लिए
`allowWriteControls` आवश्यक है। कोई भी विकल्प लॉक की गई बाइंडिंग को बायपास नहीं करता।

केवल स्रोत थ्रेड सूचीबद्ध करते समय या लंबित Chat दिखाते समय OpenClaw अनुमोदन अनुरोधों की सदस्यता नहीं लेता और न ही उनका उत्तर देता है। पहले टर्न पर एक अलग कैनोनिकल हार्नेस थ्रेड शुरू करने से कोई अन्य Codex प्रक्रिया प्रतिस्पर्धी रोलआउट राइटर बनाए बिना स्रोत का स्वामित्व बनाए रख सकती है।

मूल CLI, VS Code, Atlas या ChatGPT स्रोत नेटिव क्लाइंट और OpenClaw कैटलॉग में दृश्यमान रहता है। कैनोनिकल ब्रांच एक नेटिव Codex थ्रेड के रूप में संग्रहीत होती है, लेकिन उसका स्रोत प्रकार `appServer` होता है; Codex Desktop या कोई अन्य नेटिव क्लाइंट उस स्रोत प्रकार को फ़िल्टर कर सकता है, इसलिए यह गारंटी नहीं है कि ब्रांच स्वयं प्रत्येक नेटिव इतिहास दृश्य में दिखाई देगी।

OpenClaw के App Server द्वारा सक्रिय बताई गई पंक्ति नई ब्रांच शुरू नहीं कर सकती। वर्तमान टर्न समाप्त होने की प्रतीक्षा करें और कैटलॉग रीफ़्रेश करें। Codex App Server एक प्रक्रिया के भीतर परिवर्तनों को क्रमबद्ध करता है, लेकिन यह विशिष्ट क्रॉस-प्रोसेस रनर या अनुमोदन-स्वामी लीज़ प्रदान नहीं करता।

किसी **Stored / activity unknown** पंक्ति के लिए, Chat मिरर और पहले टर्न का स्नैपशॉट पिन अंतिम टर्मिनल रूप से बनाए रखे गए टर्न तक Codex की स्थिति का उपयोग करते हैं। स्रोत थ्रेड रिज़्यूम, बाधित या आर्काइव नहीं किया जाता। यदि किसी अन्य प्रक्रिया में कोई टर्न प्रगति पर है, तो उसका नवीनतम जारी कार्य ब्रांच में मौजूद नहीं हो सकता।

## स्थानीय सत्र आर्काइव करें

संग्रहीत या निष्क्रिय Gateway-स्थानीय पंक्ति पर **Archive** चुनें, फिर पुष्टि करें कि कोई अन्य Codex क्लाइंट या OpenClaw रनर उस थ्रेड या उससे उत्पन्न वंशजों का उपयोग नहीं कर रहा है। OpenClaw प्रक्रिया-स्थानीय स्थिति को नए सिरे से पढ़ता है, केवल `idle` या `notLoaded` के लिए आगे बढ़ता है, नेटिव Codex आर्काइव कार्रवाई को कॉल करता है और सत्र को गैर-आर्काइव सूची से हटा देता है। नेटिव Codex थ्रेड से उत्पन्न वंशजों को भी आर्काइव करने का प्रयास करता है।

जब नए रीड में सत्र सक्रिय या त्रुटि स्थिति में बताया जाता है, जब वह किसी युग्मित Node से संबंधित होता है या जब किसी नए बनाए गए पर्यवेक्षित Chat में अब भी उस स्रोत से लंबित ब्रांच होती है, तब आर्काइव उपलब्ध नहीं होता। स्रोत को आर्काइव करने से पहले उसकी कैनोनिकल ब्रांच को मूर्त रूप देने के लिए Chat का पहला संदेश भेजें। आर्काइव तब भी अवरुद्ध होता है जब OpenClaw को पता हो कि कोई सक्रिय बाइंडिंग ठीक लक्षित थ्रेड या आर्काइव न किए गए किसी उत्पन्न वंशज की स्वामी है। OpenClaw प्रत्येक पृष्ठ पर प्रयोगात्मक Codex वंशज क्वेरी का अनुसरण करता है; अमान्य प्रतिक्रिया, अनुरोध विफलता, दोहराया गया कर्सर या थ्रेड अथवा सुरक्षा-सीमा समाप्त होने पर आर्काइव अस्वीकार कर दिया जाता है।

रीड, वंशजों की गणना और आर्काइव अनुरोध एक ही सशर्त कार्रवाई नहीं हैं, इसलिए उनके बीच भी कोई टर्न शुरू हो सकता है। App Server की स्थिति स्वतंत्र प्रक्रियाओं के बीच साझा भी नहीं होती। इसलिए अज्ञात क्लाइंट और इस रेस के लिए पुष्टि ही सुरक्षा सीमा है: पुष्टि करने से पहले प्रत्येक अन्य क्लाइंट को बंद करें या अन्यथा सत्यापित करें। आर्काइव किए गए थ्रेड को Codex Desktop, Codex CLI या स्वामी-अधिकृत नेटिव थ्रेड-प्रबंधन प्रवाह से पुनर्स्थापित करें; अनआर्काइव होने के बाद वह फिर दिखाई देता है।

```bash
codex unarchive <thread-id>
```

## युग्मित-Node सीमाएँ समझें

युग्मित Node संस्करणयुक्त, केवल-पठन
`codex.appServer.threads.list.v1` और
`codex.appServer.thread.turns.list.v1` कमांड उपलब्ध कराते हैं। जिन नेटिव Node होस्ट पर
Codex CLI उपलब्ध है, वे अनुमति-सूचीबद्ध `codex.terminal.resume.v1`
कमांड भी उपलब्ध कराते हैं। Gateway को सामान्यीकृत
मेटाडेटा और स्पष्ट रूप से अनुरोध किए गए सीमित ट्रांसक्रिप्ट पृष्ठ मिलते हैं, रॉ App Server एंडपॉइंट कभी नहीं। ऑपरेटर टर्मिनल में किसी पंक्ति को खोलने पर स्वामी होस्ट में `codex resume <thread-id>`
चलता है और उस कमांड का PTY रिले होता है; इससे सामान्य
शेल या Gateway द्वारा दिया गया argv उपलब्ध नहीं होता।

टर्मिनल रिले हार्नेस निरंतरता या आर्काइव स्वामित्व
अनुबंध प्रदान नहीं करता। इसलिए रिमोट पंक्तियाँ दृश्यमान रहती हैं, लेकिन रिमोट थ्रेड निष्क्रिय होने पर भी **Continue** या
**Archive** उपलब्ध नहीं करातीं। उस कंप्यूटर पर **Open in terminal** के माध्यम से Codex का उपयोग करें या सुरक्षित
रनर-स्वामित्व सीमा वाले भावी निरंतरता प्रवाह का उपयोग करें।

## मेटाडेटा और अनुमतियाँ

कैटलॉग पंक्तियों में ये शामिल हो सकते हैं:

- थ्रेड और सत्र पहचानकर्ता
- शीर्षक और कार्यशील डायरेक्टरी
- वर्तमान स्थिति और सक्रिय प्रतीक्षा फ़्लैग
- निर्माण, अपडेट और गतिविधि टाइमस्टैम्प
- स्रोत, मॉडल प्रोवाइडर, Codex CLI संस्करण और Git ब्रांच

कैटलॉग प्रोजेक्शन में ट्रांसक्रिप्ट पूर्वावलोकन, टर्न, रोलआउट पथ,
Codex होम पथ, Git रिमोट, कमिट SHA और रॉ App Server त्रुटियाँ शामिल नहीं होतीं। कैटलॉग
एक्सेस और Control UI ट्रांसक्रिप्ट रीड के लिए `operator.write` Gateway
स्कोप आवश्यक है, क्योंकि फ़्लीट एकत्रीकरण मानक `node.invoke` पथ का उपयोग करता है, भले ही दोनों Node कमांड केवल-पठन हों।

`supervision.allowRawTranscripts` और `supervision.allowWriteControls`
स्वायत्त एजेंट और स्टैंडअलोन MCP टूल को नियंत्रित करते हैं। दोनों का डिफ़ॉल्ट `false` है। पर्यवेक्षण
सक्षम होने पर, यदि रॉ ट्रांसक्रिप्ट की अनुमति नहीं है तो `codex_threads` सूची और केवल-मेटाडेटा रीड परिणामों से ट्रांसक्रिप्ट पूर्वावलोकन और टर्न हटा देता है; टर्न-समावेशी रीड सुरक्षित रूप से विफल हो जाता है। प्रत्येक फ़ोर्क, नाम बदलने, आर्काइव और अनआर्काइव कार्रवाई के लिए राइट नियंत्रण आवश्यक हैं। ये विकल्प प्रमाणित Control UI
ट्रांसक्रिप्ट दृश्य को नियंत्रित नहीं करते और बाइंडिंग, होस्ट, स्थिति या पुष्टि जाँच को बायपास नहीं करते।

### संगतता टूल

आधिकारिक `codex` Plugin मौजूदा एजेंट और स्टैंडअलोन MCP क्लाइंट के लिए
रिलीज़ किए गए पाँच Supervisor टूल नाम बनाए रखता है:

- `codex_endpoint_probe`
- `codex_sessions_list`
- `codex_session_read`
- `codex_session_send`
- `codex_session_interrupt`

`codex_sessions_list` डिफ़ॉल्ट रूप से केवल लोड किए गए आइटम दिखाता है; कोई `loaded_only`
पैरामीटर नहीं है। Codex के स्थिति डेटाबेस से आर्काइव न की गई संग्रहीत पंक्तियाँ भी पढ़ने के लिए `include_stored: true` सेट करें। वैकल्पिक `max_stored_sessions` सीमा का डिफ़ॉल्ट 200
है और यह प्रति एंडपॉइंट 1 से 1,000 पंक्तियाँ स्वीकार करती है। यह लोड की गई पंक्तियों को सीमित नहीं करती।
रॉ-ट्रांसक्रिप्ट अनुमति के बिना, सूची परिणाम ट्रांसक्रिप्ट से प्राप्त नाम,
पूर्वावलोकन और विस्तृत एंडपॉइंट त्रुटियाँ शामिल नहीं करते।
`codex_session_read` के लिए `allowRawTranscripts` आवश्यक है; `include_turns: true`
इसके अतिरिक्त Codex से टर्न माँगता है।

`codex_session_send` और `codex_session_interrupt` के लिए
`allowWriteControls` आवश्यक है। Send `mode: "auto" | "start" | "steer"` स्वीकार करता है, लेकिन
`"start"` हमेशा अस्वीकार किया जाता है और `"auto"` तथा `"steer"` दोनों केवल पढ़े जा सकने वाले सक्रिय टर्न को निर्देशित कर सकते हैं। निष्क्रिय थ्रेड को **Codex
Sessions** का उपयोग करने के निर्देश के साथ अस्वीकार कर दिया जाता है, जहाँ पूर्ण हार्नेस निरंतरता से पहले अनुमोदन और टूल हैंडलर इंस्टॉल करता है। इसी तरह Interrupt के लिए भी पढ़े जा सकने वाला सक्रिय टर्न आवश्यक है। ये टूल निष्क्रिय स्रोत थ्रेड को रिज़्यूम या शुरू नहीं करते।

`openclaw doctor --fix` किसी सेवानिवृत्त `codex-supervisor` प्रविष्टि, उसके एंडपॉइंट
और अनुमति फ़ील्ड तथा Plugin अनुमति/अस्वीकृति नीति संदर्भों को स्पष्ट कैनोनिकल सेटिंग अधिलेखित किए बिना आधिकारिक
`codex` Plugin में स्थानांतरित करता है। स्टैंडअलोन
संगतता MCP अडैप्टर उसी Plugin से वही पाँच टूल लोड करना जारी रखता है; लेगेसी नीति एनवायरनमेंट वेरिएबल केवल उस विश्वसनीय
अडैप्टर के भीतर लागू होते हैं।

प्रत्येक पर्यवेक्षण कॉन्फ़िग फ़ील्ड के लिए
[Codex हार्नेस संदर्भ](/hi/plugins/codex-harness-reference#supervision) देखें।

## समस्या निवारण

**कोई सत्र दिखाई नहीं देता:** सत्यापित करें कि `@openclaw/codex` इंस्टॉल है, Plugin और
`supervision.enabled` दोनों true हैं, वर्तमान Plugin अनुमति-सूची
`codex` को अनुमति देती है और सत्र आर्काइव नहीं किए गए हैं। सक्रियण बदलने के बाद Gateway या Node को पुनः आरंभ करें।

**Continue अक्षम है:** कोई अमैप्ड पंक्ति सक्रिय है, किसी युग्मित Node से संबंधित है,
उसका होस्ट ऑफ़लाइन है या कोई अन्य कार्रवाई लंबित है। Gateway-स्थानीय संग्रहीत और निष्क्रिय
पंक्तियाँ असुरक्षित सटीक-थ्रेड अधिग्रहण के बजाय **Continue as branch** उपलब्ध कराती हैं। जिस पंक्ति में पहले से पर्यवेक्षित Chat है, वह **Open Chat** उपलब्ध कराती है।

**Archive अक्षम है:** कोई अन्य रनर न होने की पुष्टि के बाद संग्रहीत/गतिविधि-अज्ञात और
निष्क्रिय Gateway-स्थानीय पंक्तियों के लिए आर्काइव उपलब्ध होता है। सक्रिय, त्रुटिपूर्ण,
ऑफ़लाइन, युग्मित-Node, लंबित-ब्रांच और ज्ञात सटीक-बाइंडिंग-स्वामी पंक्तियाँ
आर्काइव के लिए केवल-पठन रहती हैं।

**आर्काइव किया गया सत्र गायब हो गया:** यह अपेक्षित है। पर्यवेक्षण पृष्ठ में
आर्काइव दृश्य नहीं है। उसे फिर दिखाने के लिए `codex unarchive <thread-id>` चलाएँ या Codex Desktop का उपयोग करें।

**पुराना `codex-supervisor` कॉन्फ़िग शेष है:** `openclaw doctor --fix` चलाएँ। Doctor
स्पष्ट Codex सेटिंग अधिलेखित किए बिना सेवानिवृत्त Plugin प्रविष्टि और संबंधित Plugin-नीति संदर्भों को
`plugins.entries.codex.config.supervision` में स्थानांतरित करता है।
