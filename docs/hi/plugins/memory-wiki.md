`memory-wiki` एक बंडल किया गया Plugin है, जो टिकाऊ ज्ञान को एक
नेविगेट करने योग्य विकी में संकलित करता है: नियतात्मक पृष्ठ, साक्ष्य सहित संरचित दावे,
उद्गम, डैशबोर्ड और मशीन-पठनीय सारांश।

यह Active Memory Plugin को प्रतिस्थापित नहीं करता। पुनःस्मरण, उन्नयन, अनुक्रमण और
Dreaming का स्वामित्व कॉन्फ़िगर किए गए मेमोरी बैकएंड
(`memory-core`, QMD, Honcho आदि) के पास ही रहता है। `memory-wiki` इसके साथ रहता है और
ज्ञान को एक अनुरक्षित विकी परत में संकलित करता है।

| परत                 | स्वामित्व                                                                           |
| -------------------- | --------------------------------------------------------------------------------- |
| Active Memory Plugin | पुनःस्मरण, सिमैंटिक खोज, उन्नयन, Dreaming, मेमोरी रनटाइम                           |
| `memory-wiki`        | संकलित विकी पृष्ठ, उद्गम-समृद्ध संश्लेषण, डैशबोर्ड, विकी खोज/प्राप्ति/लागू करना |

व्यावहारिक नियम:

- कॉन्फ़िगर किए गए सभी कॉर्पस में एक व्यापक पुनःस्मरण पास के लिए `memory_search`
- जब आपको विकी-विशिष्ट रैंकिंग, उद्गम या पृष्ठ-स्तरीय विश्वास संरचना चाहिए, तब `wiki_search` / `wiki_get`
- एक ही कॉल में दोनों परतों को समाहित करने के लिए `memory_search corpus=all`, जब Active Memory Plugin कॉर्पस चयन का समर्थन करता हो

एक सामान्य लोकल-फ़र्स्ट सेटअप: पुनःस्मरण के लिए Active Memory बैकएंड के रूप में QMD, और
टिकाऊ संश्लेषित पृष्ठों के लिए `bridge` मोड में
`memory-wiki`। [कॉन्फ़िगरेशन](#configuration) के अंतर्गत QMD + ब्रिज मोड का उदाहरण देखें।

यदि ब्रिज मोड शून्य निर्यातित आर्टिफ़ैक्ट रिपोर्ट करता है, तो Active Memory Plugin
वर्तमान में सार्वजनिक ब्रिज इनपुट उपलब्ध नहीं करा रहा है। पहले `openclaw wiki doctor` चलाएँ,
फिर पुष्टि करें कि Active Memory Plugin सार्वजनिक आर्टिफ़ैक्ट का समर्थन करता है।

## वॉल्ट मोड

- `isolated` (डिफ़ॉल्ट): अपना वॉल्ट, अपने स्रोत, Active Memory Plugin पर कोई निर्भरता नहीं। स्व-निहित, चयनित ज्ञान भंडार के लिए इसका उपयोग करें।
- `bridge`: सार्वजनिक Plugin SDK सीमाओं के माध्यम से Active Memory Plugin से सार्वजनिक मेमोरी आर्टिफ़ैक्ट और इवेंट लॉग पढ़ता है। निजी Plugin आंतरिक हिस्सों तक पहुँचे बिना मेमोरी Plugin के निर्यातित आर्टिफ़ैक्ट संकलित करने के लिए इसका उपयोग करें।
- `unsafe-local`: स्थानीय निजी पथों के लिए स्पष्ट समान-मशीन आपात विकल्प। जानबूझकर प्रयोगात्मक और गैर-पोर्टेबल; इसका उपयोग केवल तभी करें, जब आप विश्वास सीमा समझते हों और आपको विशेष रूप से ऐसा स्थानीय फ़ाइल सिस्टम एक्सेस चाहिए जिसे ब्रिज मोड प्रदान नहीं कर सकता।

वॉल्ट मोड और वॉल्ट स्कोप अलग-अलग विकल्प हैं:

- `vaultMode` चुनता है कि विकी इनपुट कहाँ से आते हैं।
- `vault.scope` चुनता है कि सभी एजेंट एक वॉल्ट का उपयोग करें या प्रत्येक एजेंट को एक चाइल्ड वॉल्ट मिले।

`vault.scope: "global"` डिफ़ॉल्ट है और मौजूदा एकल-वॉल्ट
व्यवहार को बनाए रखता है। जब एजेंटों को विकी पृष्ठ, संकलित सारांश, खोज परिणाम या लेखन
साझा नहीं करना चाहिए, तब `isolated` या `bridge` मोड के साथ `vault.scope: "agent"` का उपयोग करें।
एजेंट स्कोप को `unsafe-local` मोड के साथ संयोजित नहीं किया जा सकता, क्योंकि वे कॉन्फ़िगर किए गए
निजी पथ एजेंट-स्वामित्व वाले इनपुट नहीं हैं। कॉन्फ़िगरेशन सत्यापन इस
संयोजन को अस्वीकार करता है।

`bridge.*` कॉन्फ़िग टॉगल के अनुसार, ब्रिज मोड इन्हें अनुक्रमित कर सकता है:

- निर्यातित मेमोरी आर्टिफ़ैक्ट (`indexMemoryRoot`)
- दैनिक नोट्स (`indexDailyNotes`)
- Dreaming रिपोर्ट (`indexDreamReports`)
- मेमोरी इवेंट लॉग (`followMemoryEvents`)

जब ब्रिज मोड सक्रिय हो और `bridge.readMemoryArtifacts` सक्षम हो,
तो `openclaw wiki status`, `openclaw wiki doctor` और `openclaw wiki bridge
import` चालू Gateway के माध्यम से रूट होते हैं, ताकि उन्हें एजेंट/रनटाइम मेमोरी के समान Active Memory
Plugin संदर्भ दिखाई दे। यदि ब्रिज अक्षम है या आर्टिफ़ैक्ट
रीड बंद हैं, तो वे कमांड स्थानीय/ऑफ़लाइन व्यवहार बनाए रखते हैं।

```text
<vault>/
  AGENTS.md

  index.md
  inbox.md
  entities/
  concepts/
  syntheses/
  sources/
  reports/
  _attachments/
  _views/

```

प्रबंधित सामग्री जनरेट किए गए ब्लॉक के भीतर रहती है; मानव नोट ब्लॉक
पुनर्जनन के दौरान संरक्षित रहते हैं।

- `sources/`: आयातित कच्ची सामग्री और ब्रिज/असुरक्षित-स्थानीय-समर्थित पृष्ठ
- `entities/`: टिकाऊ वस्तुएँ, लोग, सिस्टम, प्रोजेक्ट, ऑब्जेक्ट
- `concepts/`: विचार, अमूर्तताएँ, पैटर्न, नीतियाँ (OKF आयातों का गंतव्य भी)
- `syntheses/`: संकलित सारांश और अनुरक्षित समेकित विवरण
- `reports/`: जनरेट किए गए डैशबोर्ड

## Open Knowledge Format आयात

```bash
openclaw wiki okf import ./bundles/ga4
```

अनपैक किए गए Open Knowledge Format बंडल को विकी अवधारणा पृष्ठों में आयात करें। यह तब
उपयुक्त है, जब कोई डेटा कैटलॉग, दस्तावेज़ीकरण क्रॉलर या संवर्धन एजेंट पहले से
OKF उत्पन्न करता हो: OKF को पोर्टेबल विनिमय आर्टिफ़ैक्ट बनाए रखें और `memory-wiki` को
इसे OpenClaw-मूल अवधारणा पृष्ठों और संकलित सारांशों में बदलने दें।

- गैर-आरक्षित `.md` फ़ाइलें अवधारणा दस्तावेज़ हैं
- प्रत्येक आयातित अवधारणा में गैर-रिक्त `type` फ्रंटमैटर फ़ील्ड आवश्यक है; `type` अनुपस्थित होने पर `missing-type` चेतावनी उत्पन्न होती है और फ़ाइल छोड़ दी जाती है
- अज्ञात `type` मान सामान्य अवधारणाओं के रूप में स्वीकार किए जाते हैं
- `index.md` और `log.md` आरक्षित हैं और कभी भी अवधारणाओं के रूप में आयात नहीं किए जाते
- टूटे हुए या बाहरी Markdown लिंक अपरिवर्तित छोड़े जाते हैं

आयातित पृष्ठ `concepts/` के अंतर्गत समतल हो जाते हैं, ताकि मौजूदा संकलन, खोज, प्राप्ति और
डैशबोर्ड प्रवाह उन्हें दूसरी विकी ट्री के बिना देख सकें। प्रत्येक पृष्ठ
मूल OKF अवधारणा ID, स्रोत पथ, `type`, `resource`, `tags`, टाइमस्टैम्प
और पूरा निर्माता फ्रंटमैटर बनाए रखता है। आंतरिक OKF लिंक जनरेट किए गए
विकी अवधारणा पृष्ठों के लिए पुनर्लिखित होते हैं और `kind: okf-link` के साथ
संरचित `relationships` प्रविष्टियाँ भी उत्सर्जित करते हैं।

## संरचित दावे और साक्ष्य

पृष्ठ केवल मुक्त-रूप पाठ ही नहीं, बल्कि संरचित `claims` फ्रंटमैटर रखते हैं। प्रत्येक
दावे में `id`, `text`, `status`, `confidence`, `evidence[]` और
`updatedAt` शामिल हो सकते हैं। प्रत्येक साक्ष्य प्रविष्टि में `kind`, `sourceId`, `path`,
`lines`, `weight`, `confidence`, `privacyTier`, `note` और `updatedAt` शामिल हो सकते हैं।

इससे विकी निष्क्रिय नोट संग्रह के बजाय विश्वास परत की तरह व्यवहार करता है।
दावों को ट्रैक किया जा सकता है, स्कोर दिया जा सकता है, चुनौती दी जा सकती है और स्रोतों के आधार पर सुलझाया जा सकता है।

## एजेंट-संबंधी एंटिटी मेटाडेटा

एंटिटी पृष्ठ लोगों, टीमों, सिस्टम, प्रोजेक्ट या किसी अन्य एंटिटी प्रकार के लिए
उपयोग योग्य सामान्य रूटिंग मेटाडेटा रखते हैं:

- `entityType`: उदाहरण के लिए `person`, `team`, `system`, `project`
- `canonicalId`: उपनामों और आयातों में स्थिर पहचान कुंजी
- `aliases`: नाम, हैंडल या लेबल, जो उसी पृष्ठ पर रिज़ॉल्व होते हैं
- `privacyTier`: मुक्त-रूप स्ट्रिंग; `public` को समीक्षा-रहित माना जाता है, अन्य सभी मान (उदाहरण के लिए `local-private`, `sensitive`, `confirm-before-use`) `reports/privacy-review.md` में चिह्नित किए जाते हैं
- `bestUsedFor` / `notEnoughFor`: संक्षिप्त रूटिंग संकेत
- `lastRefreshedAt`: स्रोत-रीफ़्रेश टाइमस्टैम्प, पृष्ठ संपादन समय से अलग
- `personCard`: वैकल्पिक व्यक्ति-विशिष्ट रूटिंग कार्ड (हैंडल, सोशल प्रोफ़ाइल, ईमेल, समय क्षेत्र, लेन, किसके लिए पूछें, किसके लिए न पूछें, विश्वसनीयता, गोपनीयता स्तर)
- `relationships`: संबंधित पृष्ठों के लिए टाइप किए गए किनारे (लक्ष्य, प्रकार, भार, विश्वसनीयता, साक्ष्य प्रकार, गोपनीयता स्तर, नोट)

लोगों की विकी के लिए `reports/person-agent-directory.md` से प्रारंभ करें, फिर संपर्क विवरण या अनुमानित
तथ्यों का उपयोग करने से पहले `wiki_get` से व्यक्ति का पृष्ठ खोलें।

## संकलन पाइपलाइन

संकलन विकी पृष्ठ पढ़ता है, सारांश सामान्यीकृत करता है और OpenClaw की साझा SQLite Plugin स्थिति में
मशीन-संबंधी स्नैपशॉट बनाए रखता है। रनटाइम कोड
असिंक्रोनस प्रॉम्प्ट तैयारी के दौरान SQLite लोड करने के लिए जीवनचक्र-स्वामित्व वाले स्वामी स्नैपशॉट का उपयोग करता है;
सिंक्रोनस प्रॉम्प्ट संयोजन कभी भी Markdown को स्क्रैप या कैश फ़ाइलों को नहीं पढ़ता।
संकलित आउटपुट खोज/प्राप्ति के लिए प्रथम-पास विकी अनुक्रमण, स्वामी पृष्ठों तक
दावा-ID लुकअप, संक्षिप्त प्रॉम्प्ट पूरक और रिपोर्ट
जनरेशन को भी संचालित करता है।

स्रोत संपादन और वॉल्ट पुनर्स्थापन अगले
संकलन के बाद ही मशीन-संबंधी बनते हैं। Plugin जीवनचक्र को पुनःप्रारंभ या रीफ़्रेश करने पर वॉल्ट के
कारणात्मक रूप से शृंखलाबद्ध संकलन प्रकाशन की SQLite से तुलना की जाती है और
अधिक नए, वापस किए गए स्थिति से आए स्नैपशॉट को अस्वीकार किया जाता है। रोलबैक से पहले प्रारंभ हुआ कंपाइलर
पुनर्स्थापित पूर्ववर्ती स्थिति पर प्रकाशन नहीं कर सकता। प्रॉम्प्ट तैयारी
वॉल्ट को पोल नहीं करती या फ़ाइल वॉचर इंस्टॉल नहीं करती।
रोलबैक क्वारंटीन के बाद, चालू प्रक्रिया में किया गया संकलन स्वामी को
तुरंत साफ़ कर देता है; अलग कंपाइलर प्रक्रिया के लिए Plugin जीवनचक्र रीफ़्रेश आवश्यक है, ताकि
डेमन नए टिकाऊ प्रकाशन की पुष्टि कर सके।
संकलित कैश पुनर्निर्माण योग्य हैं: प्रकाशन युगों से पहले की कैश पंक्तियों को
मिस माना जाता है और अगला संकलन उन्हें प्रतिस्थापित करता है; उन्हें माइग्रेट नहीं किया जाता।

## खोज और पुनर्प्राप्ति

दो खोज बैकएंड:

- `shared`: उपलब्ध होने पर साझा मेमोरी खोज प्रवाह का उपयोग करें
- `local`: विकी को स्थानीय रूप से खोजें

तीन कॉर्पस: `wiki`, `memory`, `all`।

- `wiki_search` / `wiki_get` संभव होने पर प्रथम पास के रूप में संकलित सारांशों का उपयोग करते हैं
- दावा ID स्वामी पृष्ठ तक वापस रिज़ॉल्व होते हैं
- विवादित/पुराने/ताज़ा दावे रैंकिंग को प्रभावित करते हैं
- उद्गम लेबल परिणामों में बने रहते हैं

खोज मोड (`--mode` / टूल `mode` पैरामीटर):

| मोड              | बेहतर करता है                                                         |
| ----------------- | -------------------------------------------------------------- |
| `auto`            | संतुलित डिफ़ॉल्ट                                               |
| `find-person`     | व्यक्ति-जैसी इकाइयाँ, उपनाम, हैंडल, सोशल प्रोफ़ाइल, कैनोनिकल आईडी |
| `route-question`  | एजेंट कार्ड, किससे पूछें/किसके लिए सर्वोत्तम संकेत, संबंध संदर्भ |
| `source-evidence` | स्रोत पृष्ठ और संरचित साक्ष्य मेटाडेटा                  |
| `raw-claim`       | मेल खाते संरचित दावे; दावा/साक्ष्य मेटाडेटा लौटाता है    |

जब कोई परिणाम किसी संरचित दावे से मेल खाता है, तो `wiki_search` अपने विवरण पेलोड में
`matchedClaimId`, `matchedClaimStatus`, `matchedClaimConfidence`,
`evidenceKinds`, और `evidenceSourceIds` लौटाता है। उपलब्ध होने पर टेक्स्ट आउटपुट में
संक्षिप्त `Claim:` और `Evidence:` पंक्तियाँ शामिल होती हैं।

## एजेंट टूल

| टूल          | उद्देश्य                                                                                                                                                       |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `wiki_status` | वर्तमान वॉल्ट मोड और दायरा, निर्धारित एजेंट, स्वास्थ्य, Obsidian CLI की उपलब्धता                                                                               |
| `wiki_search` | विकी पृष्ठों और, कॉन्फ़िगर होने पर, साझा मेमोरी कॉर्पस में खोज करता है; व्यक्ति खोज, प्रश्न रूटिंग, स्रोत साक्ष्य या सीधे दावे की विस्तृत जाँच के लिए `mode` स्वीकार करता है |
| `wiki_get`    | आईडी/पथ द्वारा विकी पृष्ठ पढ़ता है; साझा खोज सक्षम होने और लुकअप विफल होने पर साझा मेमोरी कॉर्पस का उपयोग करता है                                     |
| `wiki_apply`  | मुक्त-रूप पृष्ठ संशोधन के बिना सीमित संश्लेषण/मेटाडेटा परिवर्तन                                                                                             |
| `wiki_lint`   | संरचनात्मक जाँच, उद्गम में अंतराल, विरोधाभास, अनुत्तरित प्रश्न                                                                                            |

Plugin एक गैर-अनन्य मेमोरी कॉर्पस अनुपूरक भी पंजीकृत करता है, इसलिए सक्रिय मेमोरी
Plugin द्वारा कॉर्पस चयन समर्थित होने पर साझा `memory_search` और
`memory_get` विकी तक पहुँच सकते हैं।

## प्रॉम्प्ट और संदर्भ व्यवहार

जब `context.includeCompiledDigestPrompt` सक्षम होता है, तो मेमोरी प्रॉम्प्ट अनुभाग
Plugin की स्थिति से एक संक्षिप्त संकलित स्नैपशॉट जोड़ते हैं: केवल शीर्ष पृष्ठ,
केवल शीर्ष दावे, विरोधाभासों की संख्या, प्रश्नों की संख्या, विश्वास/नवीनता
निर्धारक। यह वैकल्पिक है क्योंकि इससे प्रॉम्प्ट का स्वरूप बदलता है; यह मुख्यतः
उन संदर्भ इंजनों या प्रॉम्प्ट संयोजन के लिए प्रासंगिक है जो स्पष्ट रूप से मेमोरी
अनुपूरकों का उपयोग करते हैं।

## कॉन्फ़िगरेशन

कॉन्फ़िगरेशन को `plugins.entries.memory-wiki.config` के अंतर्गत रखें:

```json5
{
  plugins: {
    entries: {
      "memory-wiki": {
        enabled: true,
        config: {
          vaultMode: "isolated",
          vault: {
            scope: "global",
            path: "~/.openclaw/wiki/main",
            renderMode: "obsidian",
          },
          obsidian: {
            enabled: true,
            useOfficialCli: true,
            vaultName: "OpenClaw Wiki",
            openAfterWrites: false,
          },
          bridge: {
            enabled: false,
            readMemoryArtifacts: true,
            indexDreamReports: true,
            indexDailyNotes: true,
            indexMemoryRoot: true,
            followMemoryEvents: true,
          },
          unsafeLocal: {
            allowPrivateMemoryCoreAccess: false,
            paths: [],
          },
          ingest: {
            autoCompile: true,
            maxConcurrentJobs: 1,
            allowUrlIngest: true,
          },
          search: {
            backend: "shared",
            corpus: "wiki",
          },
          context: {
            includeCompiledDigestPrompt: false,
          },
          render: {
            preserveHumanBlocks: true,
            createBacklinks: true,
            createDashboards: true,
          },
        },
      },
    },
  },
}
```

मुख्य टॉगल:

| कुंजी                                        | मान / डिफ़ॉल्ट                               | टिप्पणियाँ                                                                         |
| ------------------------------------------ | ---------------------------------------------- | ----------------------------------------------------------------------------- |
| `vaultMode`                                | `isolated` (डिफ़ॉल्ट), `bridge`, `unsafe-local` | इनपुट और एकीकरण व्यवहार चुनता है                                        |
| `vault.scope`                              | `global` (डिफ़ॉल्ट), `agent`                    | एक साझा वॉल्ट या प्रत्येक एजेंट के लिए एक चाइल्ड वॉल्ट                                 |
| `vault.path`                               | वैश्विक डिफ़ॉल्ट `~/.openclaw/wiki/main`         | वैश्विक रूप से सटीक वॉल्ट; एजेंट-दायरे के पैरेंट का डिफ़ॉल्ट `~/.openclaw/wiki` है       |
| `vault.renderMode`                         | `native` (डिफ़ॉल्ट), `obsidian`                 |                                                                               |
| `bridge.readMemoryArtifacts`               | डिफ़ॉल्ट `true`                                 | सक्रिय मेमोरी Plugin की सार्वजनिक कलाकृतियाँ आयात करता है                                  |
| `bridge.followMemoryEvents`                | डिफ़ॉल्ट `true`                                 | ब्रिज मोड में इवेंट लॉग शामिल करता है                                             |
| `unsafeLocal.allowPrivateMemoryCoreAccess` | डिफ़ॉल्ट `false`                                | `unsafe-local` आयात चलाने के लिए आवश्यक                                        |
| `unsafeLocal.paths`                        | डिफ़ॉल्ट `[]`                                   | `unsafe-local` मोड में आयात करने के लिए स्पष्ट स्थानीय पथ                         |
| `search.backend`                           | `shared` (डिफ़ॉल्ट), `local`                    |                                                                               |
| `search.corpus`                            | `wiki` (डिफ़ॉल्ट), `memory`, `all`              |                                                                               |
| `context.includeCompiledDigestPrompt`      | डिफ़ॉल्ट `false`                                | चयनित एजेंट का संक्षिप्त डाइजेस्ट स्नैपशॉट मेमोरी प्रॉम्प्ट अनुभागों में जोड़ता है |
| `render.createBacklinks`                   | डिफ़ॉल्ट `true`                                 | नियतात्मक संबंधित ब्लॉक बनाता है                                         |
| `render.createDashboards`                  | डिफ़ॉल्ट `true`                                 | डैशबोर्ड पृष्ठ बनाता है                                                      |

### प्रति-एजेंट वॉल्ट

प्रत्येक कॉन्फ़िगर किए गए एजेंट को अलग विकी देने के लिए `vault.scope` को
`agent` पर सेट करें। इस दायरे में, `vault.path` एक पैरेंट
डायरेक्टरी है और OpenClaw सामान्यीकृत एजेंट आईडी जोड़ता है:

```json5
{
  agents: {
    list: [{ id: "support" }, { id: "marketing" }],
  },
  plugins: {
    entries: {
      "memory-wiki": {
        enabled: true,
        config: {
          vaultMode: "bridge",
          vault: {
            scope: "agent",
            path: "~/.openclaw/wiki",
          },
          bridge: {
            enabled: true,
            readMemoryArtifacts: true,
          },
        },
      },
    },
  },
}
```

यह `~/.openclaw/wiki/support` और
`~/.openclaw/wiki/marketing` में निर्धारित होता है। यदि एजेंट दायरे में `vault.path`
नहीं दिया गया है, तो पैरेंट का डिफ़ॉल्ट `~/.openclaw/wiki` होता है। इसलिए डिफ़ॉल्ट
`main` एजेंट मौजूदा `~/.openclaw/wiki/main` पथ बनाए रखता है।

एजेंट टूल, संकलित प्रॉम्प्ट डाइजेस्ट, और `memory_search` /
`memory_get` के माध्यम से उपलब्ध कराया गया विकी अनुपूरक सक्रिय एजेंट संदर्भ
से वॉल्ट निर्धारित करते हैं। कई कॉन्फ़िगर किए गए एजेंट वाले सेटअप में CLI और Gateway
कॉल के लिए, `openclaw wiki --agent <agentId> ...` या Gateway अनुरोध के
`agentId` से एजेंट स्पष्ट रूप से प्रदान करें। कोई आईडी न दिए जाने पर एकमात्र
कॉन्फ़िगर किया गया एजेंट डिफ़ॉल्ट रहता है।

ब्रिज मोड में, एजेंट-दायरे वाले आयात किसी सार्वजनिक मेमोरी कलाकृति को केवल तभी
स्वीकार करते हैं जब उसके `agentIds` में चयनित एजेंट शामिल हो। किसी अन्य
एजेंट के स्वामित्व वाली, स्वामित्व मेटाडेटा से रहित या अज्ञात स्वामी वाली कलाकृतियाँ
छोड़ दी जाती हैं। वैश्विक दायरा मौजूदा साझा-कलाकृति व्यवहार बनाए रखता है।

<Warning>
`vault.scope` बदलने से मौजूदा वॉल्ट की प्रतिलिपि नहीं बनती या वह विभाजित
नहीं होता। एजेंट दायरे में, स्पष्ट रूप से कॉन्फ़िगर किया गया `vault.path` एक
पैरेंट डायरेक्टरी बन जाता है, इसलिए प्रोडक्शन एजेंट बदलने से पहले मौजूदा पृष्ठों को
सोच-समझकर स्थानांतरित या आयात करें। पहले वॉल्ट का बैकअप लें।

प्रति-एजेंट वॉल्ट समान-प्रोसेस ज्ञान सीमा हैं, ऑपरेटिंग-सिस्टम सुरक्षा सीमा नहीं।
होस्ट फ़ाइल सिस्टम तक पहुँच रखने वाले Plugin और सैंडबॉक्स-रहित टूल अब भी दूसरे
एजेंट की डायरेक्टरी पढ़ सकते हैं। जब एजेंट एक-दूसरे पर भरोसा न करते हों, तो
[सैंडबॉक्सिंग](/hi/gateway/sandboxing) या [अलग Gateway प्रोफ़ाइल](/hi/gateway/multiple-gateways)
का उपयोग करें।
</Warning>

### उदाहरण: QMD + ब्रिज मोड

जब आप रिकॉल के लिए QMD और अनुरक्षित ज्ञान परत के लिए `memory-wiki` चाहते
हैं, तब इसका उपयोग करें। प्रत्येक परत केंद्रित रहती है: QMD कच्चे नोट्स, सत्र
निर्यात और अतिरिक्त संग्रहों को खोजने योग्य रखता है, जबकि `memory-wiki`
स्थिर इकाइयों, दावों, डैशबोर्ड और स्रोत पृष्ठों को संकलित करता है।

```json5
{
  memory: {
    backend: "qmd",
  },
  plugins: {
    entries: {
      "memory-wiki": {
        enabled: true,
        config: {
          vaultMode: "bridge",
          bridge: {
            enabled: true,
            readMemoryArtifacts: true,
            indexDreamReports: true,
            indexDailyNotes: true,
            indexMemoryRoot: true,
            followMemoryEvents: true,
          },
          search: {
            backend: "shared",
            corpus: "all",
          },
          context: {
            includeCompiledDigestPrompt: false,
          },
        },
      },
    },
  },
}
```

इससे QMD सक्रिय मेमोरी रिकॉल का प्रभारी रहता है, `memory-wiki` संकलित पृष्ठों
और डैशबोर्ड पर केंद्रित रहता है, और जब तक आप जानबूझकर संकलित डाइजेस्ट प्रॉम्प्ट
सक्षम नहीं करते, तब तक प्रॉम्प्ट का स्वरूप अपरिवर्तित रहता है।

## Obsidian समर्थन

जब `vault.renderMode`, `obsidian` होता है, तो Plugin Obsidian-अनुकूल
Markdown लिखता है और स्थिति की जाँच, वॉल्ट खोज, पृष्ठ खोलने, कमांड लागू करने और
दैनिक नोट पर जाने के लिए वैकल्पिक रूप से आधिकारिक `obsidian` CLI का उपयोग
कर सकता है। यह वैकल्पिक है; Obsidian के बिना भी विकी नेटिव मोड में काम करता है।

एजेंट-दायरे वाले वॉल्ट अब भी Obsidian-अनुकूल Markdown का उपयोग कर सकते हैं, लेकिन
कॉन्फ़िगरेशन सत्यापन `obsidian.useOfficialCli: true` को `vault.scope: "agent"` के साथ अस्वीकार
करता है। वर्तमान `obsidian.vaultName` सेटिंग वैश्विक है और प्रत्येक एजेंट के लिए
अलग Obsidian वॉल्ट नहीं चुन सकती। इसके बजाय विकी टूल और CLI संचालन का उपयोग करें,
या Obsidian द्वारा संचालित विकी को वैश्विक दायरे में रखें।

## अनुशंसित कार्यप्रवाह

<Steps>
<Step title="स्मरण के लिए सक्रिय मेमोरी Plugin बनाए रखें">
स्मरण, प्रोमोशन और Dreaming कॉन्फ़िगर किए गए मेमोरी बैकएंड के स्वामित्व में रहते हैं।
</Step>
<Step title="memory-wiki सक्षम करें">
जब तक आप स्पष्ट रूप से ब्रिज मोड नहीं चाहते, `isolated` मोड से शुरू करें।
</Step>
<Step title="जब उद्गम महत्वपूर्ण हो, तब wiki_search / wiki_get का उपयोग करें">
जब आपको विकि-विशिष्ट रैंकिंग या पेज-स्तरीय विश्वास संरचना चाहिए, तब `memory_search` के बजाय इन्हें प्राथमिकता दें।
</Step>
<Step title="सीमित संश्लेषण या मेटाडेटा अपडेट के लिए wiki_apply का उपयोग करें">
प्रबंधित जनरेट किए गए ब्लॉक को हाथ से संपादित करने से बचें।
</Step>
<Step title="सार्थक बदलावों के बाद wiki_lint चलाएँ">
यह विरोधाभासों, अनसुलझे प्रश्नों और उद्गम की कमियों का पता लगाता है।
</Step>
<Step title="पुरानी जानकारी/विरोधाभासों की दृश्यता के लिए डैशबोर्ड चालू करें">
`render.createDashboards: true` (डिफ़ॉल्ट) सेट करें।
</Step>
</Steps>
