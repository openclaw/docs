## 1. पहचान और प्राथमिकता निर्धारण

सुरक्षा संकेत इन स्रोतों से आते हैं:

- GitHub Security Advisories (GHSA) और निजी भेद्यता रिपोर्ट।
- जब रिपोर्ट संवेदनशील न हों, तब सार्वजनिक GitHub इश्यू/चर्चाएँ।
- स्वचालित संकेत: Dependabot, CodeQL, npm एडवाइज़री और सीक्रेट स्कैनिंग।

प्रारंभिक प्राथमिकता निर्धारण:

1. प्रभावित घटक, संस्करण और विश्वास-सीमा पर प्रभाव की पुष्टि करें।
2. `SECURITY.md` के दायरे और दायरे से बाहर के नियमों का उपयोग करके इसे सुरक्षा समस्या या सुदृढ़ीकरण/कार्रवाई की आवश्यकता नहीं के रूप में वर्गीकृत करें।
3. घटना का स्वामी तदनुसार प्रतिक्रिया देता है।

## 2. गंभीरता

| गंभीरता | परिभाषा                                                                                                                                                                            |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| गंभीरतम | पैकेज/रिलीज़/रिपॉज़िटरी से समझौता, सक्रिय शोषण, या उच्च-प्रभावी नियंत्रण अथवा डेटा उजागर होने के साथ अनधिकृत विश्वास-सीमा बाइपास।                                       |

| निम्न      | बहुस्तरीय सुरक्षा संबंधी निष्कर्ष, सीमित दायरे वाली सेवा-अस्वीकृति, या बिना किसी प्रदर्शित विश्वास-सीमा बाइपास के सुदृढ़ीकरण/समानता की कमियाँ।                                                  |

## 3. प्रतिक्रिया

1. रिपोर्टकर्ता को रिपोर्ट प्राप्त होने की पुष्टि दें (संवेदनशील होने पर निजी रूप से)।
2. समर्थित रिलीज़ और नवीनतम `main` पर समस्या को पुनरुत्पादित करें, फिर रिग्रेशन कवरेज सहित पैच लागू और सत्यापित करें।
3. गंभीरतम/उच्च: व्यावहारिक रूप से यथासंभव शीघ्र पैच की गई रिलीज़ तैयार करें।
4. मध्यम/निम्न: सामान्य रिलीज़ प्रक्रिया में पैच करें और शमन मार्गदर्शन का दस्तावेज़ीकरण करें।

## 4. संचार और प्रकटीकरण

प्रभावित रिपॉज़िटरी में GitHub Security Advisories, सुधार वाले संस्करणों की रिलीज़ टिप्पणियों/चेंजलॉग प्रविष्टियों और स्थिति तथा समाधान के बारे में रिपोर्टकर्ता से सीधे अनुवर्ती संपर्क के माध्यम से संवाद करें।

गंभीरतम/उच्च घटनाओं का समन्वित प्रकटीकरण किया जाता है और उपयुक्त होने पर CVE जारी किया जाता है। कम जोखिम वाले सुदृढ़ीकरण निष्कर्षों को प्रभाव और उपयोगकर्ता जोखिम के आधार पर CVE के बिना रिलीज़ टिप्पणियों या एडवाइज़री में दर्ज किया जा सकता है।

## 5. पुनर्प्राप्ति और अनुवर्ती कार्रवाई

सुधार जारी करने के बाद:

1. CI और रिलीज़ आर्टिफ़ैक्ट में सुधारात्मक उपायों को सत्यापित करें।
2. घटना के बाद संक्षिप्त समीक्षा करें: समयरेखा, मूल कारण, पहचान की कमी और रोकथाम योजना।
3. अनुवर्ती सुदृढ़ीकरण/परीक्षण/दस्तावेज़ीकरण कार्य जोड़ें और पूर्ण होने तक उनकी प्रगति पर नज़र रखें।
