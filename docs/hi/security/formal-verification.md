OpenClaw के औपचारिक सुरक्षा मॉडल (वर्तमान में TLA+/TLC) मशीन द्वारा जाँचा गया तर्क प्रस्तुत करते हैं कि स्पष्ट रूप से बताई गई धारणाओं के अंतर्गत विशिष्ट सर्वाधिक-जोखिम वाले पथ — प्राधिकरण, सत्र पृथक्करण, टूल गेटिंग और गलत कॉन्फ़िगरेशन से सुरक्षा — अपनी निर्धारित नीति लागू करते हैं।

> नोट: कुछ पुराने लिंक पिछले प्रोजेक्ट नाम का उल्लेख कर सकते हैं।

## यह क्या है

हमलावर-संचालित, निष्पादन योग्य सुरक्षा रिग्रेशन सुइट:

- प्रत्येक दावे में सीमित अवस्था-स्थान पर चलाया जा सकने वाला मॉडल-जाँच परीक्षण है।
- कई दावों के साथ एक युग्मित नकारात्मक मॉडल होता है, जो किसी वास्तविक बग वर्ग के लिए प्रतिउदाहरण ट्रेस उत्पन्न करता है।

यह इस बात का प्रमाण **नहीं** है कि OpenClaw हर दृष्टि से सुरक्षित है, और यह संपूर्ण TypeScript कार्यान्वयन को सत्यापित नहीं करता।

## मॉडल कहाँ उपलब्ध हैं

मॉडल एक अलग रेपो में रखे जाते हैं: [vignesh07/openclaw-formal-models](https://github.com/vignesh07/openclaw-formal-models)।

<Note>
वर्तमान में वह रेपो उपलब्ध नहीं है (इसे लिखे जाने के समय GitHub "Repository not found" लौटाता है)। यदि आपके लिए भी यह अब तक अनुपलब्ध है, तो यह मानने से पहले कि मॉडल हटा दिए गए हैं, वर्तमान स्थान के बारे में OpenClaw मेंटेनर चैनलों में पूछें।
</Note>

## सीमाएँ

- ये मॉडल हैं, संपूर्ण TypeScript कार्यान्वयन नहीं — मॉडल और कोड के बीच अंतर संभव है।
- परिणाम उस अवस्था-स्थान तक सीमित हैं जिसे TLC खोजता है। हरा परिणाम मॉडल की गई धारणाओं और सीमाओं से परे सुरक्षा का संकेत नहीं देता।
- कुछ दावे स्पष्ट परिवेश धारणाओं पर निर्भर करते हैं (उदाहरण के लिए, सही डिप्लॉयमेंट और सही कॉन्फ़िगरेशन इनपुट)।

## परिणामों को पुनरुत्पादित करना

मॉडल रेपो को क्लोन करें और TLC चलाएँ:

```bash
git clone https://github.com/vignesh07/openclaw-formal-models
cd openclaw-formal-models

# Java 11+ आवश्यक है (TLC JVM पर चलता है)।
# रेपो में पिन किया गया tla2tools.jar शामिल है और यह bin/tlc तथा Make लक्ष्य प्रदान करता है।

make <target>
```

अभी इस रेपो में कोई CI एकीकरण नहीं है; भविष्य के किसी संस्करण में सार्वजनिक आर्टिफ़ैक्ट (प्रतिउदाहरण ट्रेस, रन लॉग) वाले CI-संचालित मॉडल या छोटे सीमित जाँच परीक्षणों के लिए होस्ट किया गया "यह मॉडल चलाएँ" वर्कफ़्लो जोड़ा जा सकता है।

## दावे और लक्ष्य

### Gateway का एक्सपोज़र और खुले Gateway का गलत कॉन्फ़िगरेशन

**दावा:** प्रमाणीकरण के बिना लूपबैक से परे बाइंड करने से रिमोट समझौता संभव हो सकता है और एक्सपोज़र बढ़ता है; मॉडल की धारणाओं के अनुसार, टोकन/पासवर्ड अप्रमाणित हमलावरों को रोकता है।

| परिणाम         | लक्ष्य                                                          |
| -------------- | ---------------------------------------------------------------- |
| हरा          | `make gateway-exposure-v2`, `make gateway-exposure-v2-protected` |
| लाल (अपेक्षित) | `make gateway-exposure-v2-negative`                              |

मॉडल रेपो में `docs/gateway-exposure-matrix.md` भी देखें।

### Node निष्पादन पाइपलाइन (सर्वाधिक-जोखिम वाली क्षमता)

**दावा:** मॉडल में `exec host=node` के लिए (a) Node कमांड की अनुमति-सूची और घोषित कमांड तथा (b) कॉन्फ़िगर होने पर लाइव अनुमोदन आवश्यक है; दोबारा उपयोग रोकने के लिए अनुमोदनों को टोकनयुक्त किया जाता है।

| परिणाम         | लक्ष्य                                                         |
| -------------- | --------------------------------------------------------------- |
| हरा          | `make nodes-pipeline`, `make approvals-token`                   |
| लाल (अपेक्षित) | `make nodes-pipeline-negative`, `make approvals-token-negative` |

### पेयरिंग स्टोर (DM गेटिंग)

**दावा:** पेयरिंग अनुरोध TTL और लंबित अनुरोध सीमाओं का पालन करते हैं।

| परिणाम         | लक्ष्य                                              |
| -------------- | ---------------------------------------------------- |
| हरा          | `make pairing`, `make pairing-cap`                   |
| लाल (अपेक्षित) | `make pairing-negative`, `make pairing-cap-negative` |

### इनग्रेस गेटिंग (उल्लेख और नियंत्रण-कमांड बायपास)

**दावा:** उल्लेख आवश्यक करने वाले समूह संदर्भों में, कोई अनधिकृत नियंत्रण कमांड उल्लेख गेटिंग को बायपास नहीं कर सकता।

| परिणाम         | लक्ष्य                        |
| -------------- | ------------------------------ |
| हरा          | `make ingress-gating`          |
| लाल (अपेक्षित) | `make ingress-gating-negative` |

### रूटिंग और सत्र-कुंजी पृथक्करण

**दावा:** अलग-अलग साथियों से प्राप्त DM एक ही सत्र में तब तक सम्मिलित नहीं होते, जब तक उन्हें स्पष्ट रूप से लिंक या कॉन्फ़िगर न किया गया हो।

| परिणाम         | लक्ष्य                           |
| -------------- | --------------------------------- |
| हरा          | `make routing-isolation`          |
| लाल (अपेक्षित) | `make routing-isolation-negative` |

## v1++ मॉडल: समवर्ती क्रियाएँ, पुनः प्रयास और ट्रेस की शुद्धता

अनुवर्ती मॉडल जो वास्तविक विफलता स्थितियों — गैर-परमाणु अपडेट, पुनः प्रयास और संदेश फ़ैन-आउट — के अनुरूप सटीकता बढ़ाते हैं।

### पेयरिंग स्टोर की समवर्ती क्रियाएँ और आइडेम्पोटेंसी

**दावा:** पेयरिंग स्टोर इंटरलीविंग के अंतर्गत भी `MaxPending` और आइडेम्पोटेंसी लागू करता है — जाँच-फिर-लेखन परमाणु/लॉक किया हुआ होना चाहिए और रीफ़्रेश को डुप्लिकेट नहीं बनाने चाहिए। ठोस रूप से: समवर्ती अनुरोध किसी चैनल के लिए `MaxPending` से अधिक नहीं हो सकते और समान `(channel, sender)` के लिए दोहराए गए अनुरोध/रीफ़्रेश डुप्लिकेट सक्रिय लंबित पंक्तियाँ नहीं बनाते।

| परिणाम         | लक्ष्य                                                                                                                                                                     |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| हरा          | `make pairing-race` (परमाणु/लॉक की गई सीमा जाँच), `make pairing-idempotency`, `make pairing-refresh`, `make pairing-refresh-race`                                              |
| लाल (अपेक्षित) | `make pairing-race-negative` (गैर-परमाणु आरंभ/कमिट सीमा रेस), `make pairing-idempotency-negative`, `make pairing-refresh-negative`, `make pairing-refresh-race-negative` |

### इनग्रेस ट्रेस सहसंबंध और आइडेम्पोटेंसी

**दावा:** अंतर्ग्रहण फ़ैन-आउट में ट्रेस सहसंबंध सुरक्षित रखता है और प्रदाता के पुनः प्रयासों के अंतर्गत आइडेम्पोटेंट रहता है। जब कोई बाहरी इवेंट कई आंतरिक संदेश बन जाता है, तो हर भाग समान ट्रेस/इवेंट पहचान बनाए रखता है; पुनः प्रयासों से दोहरा प्रसंस्करण नहीं होता; यदि प्रदाता इवेंट ID अनुपलब्ध हों, तो अलग-अलग इवेंट हटने से बचाने के लिए डीडुप्लिकेशन किसी सुरक्षित कुंजी (उदाहरण के लिए ट्रेस ID) का उपयोग करता है।

| परिणाम         | लक्ष्य                                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| हरा          | `make ingress-trace`, `make ingress-trace2`, `make ingress-idempotency`, `make ingress-dedupe-fallback`                                     |
| लाल (अपेक्षित) | `make ingress-trace-negative`, `make ingress-trace2-negative`, `make ingress-idempotency-negative`, `make ingress-dedupe-fallback-negative` |

### रूटिंग dmScope की प्राथमिकता और identityLinks

**दावा:** `dmScope` की प्राथमिकता और पहचान लिंक निर्धारक ढंग से व्यवहार करते हैं: डिफ़ॉल्ट `main` स्कोप एक ही स्वामी के DM में एक रोलिंग सत्र साझा करता है (व्यक्तिगत-एजेंट डिफ़ॉल्ट), जबकि कोई भी कॉन्फ़िगर किया हुआ पृथक्कारी स्कोप (`per-peer`, `per-channel-peer`, `per-account-channel-peer`) DM सत्रों को सख्ती से अलग रखता है। चैनल-विशिष्ट `dmScope` ओवरराइड वैश्विक डिफ़ॉल्ट पर प्राथमिकता पाते हैं; `identityLinks` सत्रों को केवल स्पष्ट रूप से लिंक किए गए समूहों के भीतर सम्मिलित करते हैं, असंबंधित साथियों के बीच नहीं। बहु-उपयोगकर्ता इनबॉक्स से पृथक्कारी स्कोप स्पष्ट रूप से चुनने की अपेक्षा की जाती है (रनटाइम सुरक्षा ऑडिट बहु-उपयोगकर्ता DM ट्रैफ़िक मिलने पर इसकी अनुशंसा करता है)।

| परिणाम         | लक्ष्य                                                                   |
| -------------- | ------------------------------------------------------------------------- |
| हरा          | `make routing-precedence`, `make routing-identitylinks`                   |
| लाल (अपेक्षित) | `make routing-precedence-negative`, `make routing-identitylinks-negative` |
