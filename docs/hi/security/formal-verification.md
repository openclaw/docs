OpenClaw के औपचारिक सुरक्षा मॉडल (वर्तमान में TLA+/TLC) मशीन द्वारा जाँचा गया तर्क प्रस्तुत करते हैं कि स्पष्ट रूप से बताई गई धारणाओं के अंतर्गत विशिष्ट सर्वाधिक जोखिम वाले पथ — प्राधिकरण, सत्र पृथक्करण, टूल गेटिंग और गलत कॉन्फ़िगरेशन से सुरक्षा — अपनी निर्धारित नीति लागू करते हैं।

> टिप्पणी: कुछ पुराने लिंक पिछले प्रोजेक्ट नाम का उल्लेख कर सकते हैं।

## यह क्या है

हमलावर-संचालित एक निष्पादन योग्य सुरक्षा रिग्रेशन सुइट:

- प्रत्येक दावे के लिए सीमित स्टेट स्पेस पर चलाया जा सकने वाला मॉडल-चेक उपलब्ध है।
- कई दावों के साथ एक युग्मित नकारात्मक मॉडल है, जो यथार्थवादी बग वर्ग के लिए काउंटरएग्ज़ाम्पल ट्रेस उत्पन्न करता है।

यह इस बात का प्रमाण **नहीं** है कि OpenClaw हर दृष्टि से सुरक्षित है, और यह संपूर्ण TypeScript कार्यान्वयन को सत्यापित नहीं करता।

## मॉडल कहाँ उपलब्ध हैं

मॉडल एक अलग रेपो में अनुरक्षित किए जाते हैं: [vignesh07/openclaw-formal-models](https://github.com/vignesh07/openclaw-formal-models)।

<Note>
वह रिपॉज़िटरी फ़िलहाल पहुँच योग्य नहीं है (यह लिखे जाने के समय GitHub "Repository not found" लौटाता है)। यदि आपके लिए भी यह उपलब्ध नहीं है, तो यह मानने से पहले कि मॉडल हटा दिए गए हैं, उनके वर्तमान स्थान के बारे में OpenClaw अनुरक्षक चैनलों में पूछें।
</Note>

## सीमाएँ

- ये मॉडल हैं, संपूर्ण TypeScript कार्यान्वयन नहीं — मॉडल और कोड के बीच अंतर आ सकता है।
- परिणाम उस स्टेट स्पेस तक सीमित हैं जिसे TLC जाँचता है। हरा परिणाम मॉडल की गई धारणाओं और सीमाओं से परे सुरक्षा का संकेत नहीं देता।
- कुछ दावे स्पष्ट पर्यावरणीय धारणाओं पर निर्भर हैं (उदाहरण के लिए, सही परिनियोजन और सही कॉन्फ़िगरेशन इनपुट)।

## परिणाम पुनरुत्पादित करना

मॉडल रेपो को क्लोन करें और TLC चलाएँ:

```bash
git clone https://github.com/vignesh07/openclaw-formal-models
cd openclaw-formal-models

# Java 11+ आवश्यक है (TLC JVM पर चलता है)।
# रेपो में एक पिन किया गया tla2tools.jar शामिल है और यह bin/tlc के साथ Make टार्गेट प्रदान करता है।

make <target>
```

इस रेपो में अभी कोई CI एकीकरण नहीं है; भविष्य का संस्करण सार्वजनिक आर्टिफ़ैक्ट (काउंटरएग्ज़ाम्पल ट्रेस, रन लॉग) के साथ CI द्वारा चलाए जाने वाले मॉडल या छोटे सीमित चेक के लिए होस्ट किया गया "यह मॉडल चलाएँ" वर्कफ़्लो जोड़ सकता है।

## दावे और टार्गेट

### Gateway एक्सपोज़र और खुले Gateway का गलत कॉन्फ़िगरेशन

**दावा:** मॉडल की धारणाओं के अनुसार, प्रमाणीकरण के बिना लूपबैक से बाहर बाइंड करने से दूरस्थ समझौता संभव हो सकता है और एक्सपोज़र बढ़ता है; टोकन/पासवर्ड अप्रमाणित हमलावरों को रोकता है।

| परिणाम         | टार्गेट                                                          |
| -------------- | ---------------------------------------------------------------- |
| हरा          | `make gateway-exposure-v2`, `make gateway-exposure-v2-protected` |
| लाल (अपेक्षित) | `make gateway-exposure-v2-negative`                              |

मॉडल रेपो में `docs/gateway-exposure-matrix.md` भी देखें।

### Node exec पाइपलाइन (सर्वाधिक जोखिम वाली क्षमता)

**दावा:** मॉडल में `exec host=node` के लिए (a) घोषित कमांड के साथ Node कमांड की अनुमति-सूची और (b) कॉन्फ़िगर होने पर लाइव अनुमोदन आवश्यक है; दोबारा उपयोग रोकने के लिए अनुमोदनों को टोकनाइज़ किया जाता है।

| परिणाम         | टार्गेट                                                         |
| -------------- | --------------------------------------------------------------- |
| हरा          | `make nodes-pipeline`, `make approvals-token`                   |
| लाल (अपेक्षित) | `make nodes-pipeline-negative`, `make approvals-token-negative` |

### पेयरिंग स्टोर (DM गेटिंग)

**दावा:** पेयरिंग अनुरोध TTL और लंबित अनुरोधों की सीमा का पालन करते हैं।

| परिणाम         | टार्गेट                                              |
| -------------- | ---------------------------------------------------- |
| हरा          | `make pairing`, `make pairing-cap`                   |
| लाल (अपेक्षित) | `make pairing-negative`, `make pairing-cap-negative` |

### इनग्रेस गेटिंग (उल्लेख और नियंत्रण-कमांड बायपास)

**दावा:** उल्लेख की आवश्यकता वाले समूह संदर्भों में, कोई अनधिकृत नियंत्रण कमांड उल्लेख गेटिंग को बायपास नहीं कर सकता।

| परिणाम         | टार्गेट                        |
| -------------- | ------------------------------ |
| हरा          | `make ingress-gating`          |
| लाल (अपेक्षित) | `make ingress-gating-negative` |

### रूटिंग और सत्र-कुंजी पृथक्करण

**दावा:** अलग-अलग पीयर से आने वाले DM तब तक एक ही सत्र में नहीं मिलते, जब तक उन्हें स्पष्ट रूप से लिंक या कॉन्फ़िगर न किया गया हो।

| परिणाम         | टार्गेट                           |
| -------------- | --------------------------------- |
| हरा          | `make routing-isolation`          |
| लाल (अपेक्षित) | `make routing-isolation-negative` |

## v1++ मॉडल: समवर्ती क्रियाएँ, पुनः प्रयास और ट्रेस की शुद्धता

वास्तविक दुनिया के विफलता प्रकारों के लिए सटीकता बढ़ाने वाले अनुवर्ती मॉडल: गैर-परमाण्विक अपडेट, पुनः प्रयास और संदेश फ़ैन-आउट।

### पेयरिंग स्टोर की समवर्ती क्रियाएँ और आइडेम्पोटेंसी

**दावा:** पेयरिंग स्टोर इंटरलीविंग के दौरान भी `MaxPending` और आइडेम्पोटेंसी लागू करता है — जाँच-फिर-लेखन परमाण्विक/लॉक किया हुआ होना चाहिए और रीफ़्रेश से डुप्लिकेट नहीं बनने चाहिए। विशेष रूप से: समवर्ती अनुरोध किसी चैनल के लिए `MaxPending` से अधिक नहीं हो सकते और उसी `(channel, sender)` के लिए दोहराए गए अनुरोध/रीफ़्रेश डुप्लिकेट लाइव लंबित पंक्तियाँ नहीं बनाते।

| परिणाम         | टार्गेट                                                                                                                                                                     |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| हरा          | `make pairing-race` (परमाण्विक/लॉक की गई सीमा जाँच), `make pairing-idempotency`, `make pairing-refresh`, `make pairing-refresh-race`                                              |
| लाल (अपेक्षित) | `make pairing-race-negative` (गैर-परमाण्विक आरंभ/कमिट सीमा रेस), `make pairing-idempotency-negative`, `make pairing-refresh-negative`, `make pairing-refresh-race-negative` |

### इनग्रेस ट्रेस सहसंबंध और आइडेम्पोटेंसी

**दावा:** इनजेशन फ़ैन-आउट के दौरान ट्रेस सहसंबंध बनाए रखता है और प्रदाता के पुनः प्रयासों में आइडेम्पोटेंट रहता है। जब एक बाहरी इवेंट कई आंतरिक संदेशों में बदलता है, तो प्रत्येक भाग समान ट्रेस/इवेंट पहचान बनाए रखता है; पुनः प्रयासों में दोहरा प्रसंस्करण नहीं होता; यदि प्रदाता इवेंट ID अनुपलब्ध हों, तो अलग-अलग इवेंट को हटने से बचाने के लिए डीडुप्लिकेशन किसी सुरक्षित कुंजी (उदाहरण के लिए ट्रेस ID) पर फ़ॉलबैक करता है।

| परिणाम         | टार्गेट                                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| हरा          | `make ingress-trace`, `make ingress-trace2`, `make ingress-idempotency`, `make ingress-dedupe-fallback`                                     |
| लाल (अपेक्षित) | `make ingress-trace-negative`, `make ingress-trace2-negative`, `make ingress-idempotency-negative`, `make ingress-dedupe-fallback-negative` |

### रूटिंग dmScope प्राथमिकता और identityLinks

**दावा:** रूटिंग डिफ़ॉल्ट रूप से DM सत्रों को पृथक रखती है और केवल स्पष्ट रूप से कॉन्फ़िगर किए जाने पर, चैनल प्राथमिकता और पहचान लिंक के माध्यम से सत्रों को मिलाती है। चैनल-विशिष्ट `dmScope` वैश्विक डिफ़ॉल्ट पर प्राथमिकता पाते हैं; `identityLinks` केवल स्पष्ट रूप से लिंक किए गए समूहों के भीतर सत्रों को मिलाते हैं, असंबंधित पीयर के बीच नहीं।

| परिणाम         | टार्गेट                                                                   |
| -------------- | ------------------------------------------------------------------------- |
| हरा          | `make routing-precedence`, `make routing-identitylinks`                   |
| लाल (अपेक्षित) | `make routing-precedence-negative`, `make routing-identitylinks-negative` |
