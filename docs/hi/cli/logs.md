# `openclaw logs`

RPC के माध्यम से Gateway फ़ाइल लॉग का टेल देखें। रिमोट मोड में काम करता है।

## विकल्प

- `--limit <n>`: लौटाई जाने वाली लॉग पंक्तियों की अधिकतम संख्या (डिफ़ॉल्ट `200`)
- `--max-bytes <n>`: लॉग फ़ाइल से पढ़े जाने वाले बाइट्स की अधिकतम संख्या (डिफ़ॉल्ट `250000`)
- `--follow`: लॉग स्ट्रीम को फ़ॉलो करें
- `--interval <ms>`: फ़ॉलो करते समय पोलिंग अंतराल (डिफ़ॉल्ट `1000`)
- `--json`: पंक्ति-विभाजित JSON इवेंट उत्सर्जित करें
- `--plain`: शैलीगत फ़ॉर्मैटिंग के बिना सादा टेक्स्ट आउटपुट
- `--no-color`: ANSI रंग अक्षम करें
- `--local-time`: टाइमस्टैम्प आपके स्थानीय समय क्षेत्र में रेंडर करें (डिफ़ॉल्ट)
- `--utc`: टाइमस्टैम्प UTC में रेंडर करें

## साझा Gateway RPC विकल्प

- `--url <url>`: Gateway WebSocket URL
- `--token <token>`: Gateway टोकन
- `--timeout <ms>`: मिलीसेकंड में टाइमआउट (डिफ़ॉल्ट `30000`)
- `--expect-final`: जब Gateway कॉल एजेंट-समर्थित हो, तो अंतिम प्रतिक्रिया की प्रतीक्षा करें

`--url` देने पर स्वतः लागू किए गए कॉन्फ़िगरेशन क्रेडेंशियल छोड़ दिए जाते हैं; यदि लक्षित Gateway को प्रमाणीकरण की आवश्यकता है, तो `--token` स्पष्ट रूप से शामिल करें।

## उदाहरण

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## फ़ॉलबैक और पुनर्प्राप्ति व्यवहार

- यदि निहित स्थानीय लूपबैक Gateway पेयरिंग माँगता है, कनेक्ट होते समय बंद हो जाता है, या `logs.tail` के उत्तर देने से पहले टाइमआउट हो जाता है, तो `openclaw logs` स्वतः कॉन्फ़िगर किए गए Gateway फ़ाइल लॉग पर फ़ॉलबैक करता है। स्पष्ट `--url` लक्ष्य इस फ़ॉलबैक का कभी उपयोग नहीं करते।
- निहित स्थानीय Gateway RPC विफलता के बाद `--follow` उस कॉन्फ़िगर की गई फ़ाइल पर फ़ॉलबैक नहीं करता—साथ रखी कोई पुरानी फ़ाइल लाइव टेल के बारे में भ्रमित कर सकती है। इसके बजाय Linux पर, उपलब्ध होने पर यह PID के आधार पर सक्रिय user-systemd Gateway जर्नल का उपयोग करता है (चुना गया स्रोत प्रिंट करता है); अन्यथा यह लाइव Gateway से दोबारा कनेक्ट करने का प्रयास जारी रखता है।
- `--follow` के दौरान, अस्थायी डिस्कनेक्शन (WebSocket बंद होना, टाइमआउट, कनेक्शन टूटना) एक्सपोनेंशियल बैकऑफ़ के साथ स्वतः पुनः कनेक्शन शुरू करते हैं: अधिकतम 8 पुनः प्रयास, और प्रयासों के बीच अधिकतम 30s। प्रत्येक पुनः प्रयास पर stderr में एक चेतावनी प्रिंट होती है, और किसी पोल के सफल होने पर `[logs] gateway reconnected` सूचना एक बार प्रिंट होती है। `--json` मोड में दोनों stderr पर `{"type":"notice"}` रिकॉर्ड के रूप में उत्सर्जित होते हैं। पुनर्प्राप्त न की जा सकने वाली त्रुटियाँ (प्रमाणीकरण विफलता, खराब कॉन्फ़िगरेशन) अब भी तुरंत बाहर निकलती हैं।
- `--follow --json` मोड में, लॉग-स्रोत संक्रमण `{"type":"meta"}` रिकॉर्ड के रूप में उत्सर्जित होते हैं। प्रत्येक `sourceKind` के लिए कर्सर ट्रैक करें: कोई स्ट्रीम Gateway फ़ाइल आउटपुट (`sourceKind: "file"`) से स्थानीय जर्नल फ़ॉलबैक (`sourceKind: "journal"`, `localFallback: true`, `service.pid`/`service.unit` सहित) पर जा सकती है और पुनर्प्राप्ति के बाद वापस Gateway फ़ाइल आउटपुट पर लौट सकती है। पूरे सत्र के लिए किसी एक स्थिर स्रोत या कर्सर को न मानें, और पुनर्प्राप्ति द्वारा Gateway फ़ाइल कर्सर दोबारा चलाए जाने पर ओवरलैप होती पंक्तियों को स्वीकार करें।
