# `openclaw agent`

Gateway के माध्यम से एक एजेंट टर्न चलाएँ। स्पष्ट `--local` फ़्लैग ही एकमात्र एम्बेडेड निष्पादन पथ है।

कम-से-कम एक सत्र चयनकर्ता दें: `--to`, `--session-key`, `--session-id`, या `--agent`।

संबंधित: [एजेंट प्रेषण टूल](/hi/tools/agent-send)

## विकल्प

- `-m, --message <text>`: संदेश का मुख्य भाग
- `--message-file <path>`: UTF-8 फ़ाइल से संदेश का मुख्य भाग पढ़ें
- `-t, --to <dest>`: सत्र कुंजी प्राप्त करने के लिए प्रयुक्त प्राप्तकर्ता
- `--session-key <key>`: रूटिंग के लिए उपयोग की जाने वाली स्पष्ट सत्र कुंजी
- `--session-id <id>`: स्पष्ट सत्र आईडी
- `--agent <id>`: एजेंट आईडी; रूटिंग बाइंडिंग को ओवरराइड करती है
- `--model <id>`: इस रन के लिए मॉडल ओवरराइड (`provider/model` या मॉडल आईडी)
- `--thinking <level>`: एजेंट का चिंतन स्तर (`off`, `minimal`, `low`, `medium`, `high`, और प्रदाता-समर्थित कस्टम स्तर जैसे `xhigh`, `adaptive`, या `max`)
- `--verbose <on|off>`: सत्र के लिए वर्बोज़ स्तर बनाए रखें
- `--channel <channel>`: डिलीवरी चैनल; मुख्य सत्र चैनल का उपयोग करने के लिए इसे छोड़ दें
- `--reply-to <target>`: डिलीवरी लक्ष्य ओवरराइड
- `--reply-channel <channel>`: डिलीवरी चैनल ओवरराइड
- `--reply-account <id>`: डिलीवरी खाता ओवरराइड
- `--local`: एम्बेडेड एजेंट को सीधे चलाएँ (Plugin रजिस्ट्री प्रीलोड के बाद)
- `--deliver`: उत्तर को चयनित चैनल/लक्ष्य पर वापस भेजें
- `--timeout <seconds>`: इस कमांड की एजेंट-टर्न समय-सीमा को ओवरराइड करें (डिफ़ॉल्ट 600, या `agents.defaults.timeoutSeconds`); `0` समग्र समय-सीमा को अक्षम करता है। 600-सेकंड का फ़ॉलबैक इस CLI कमांड से संबंधित है, सामान्य Gateway टर्न से नहीं, जिनका डिफ़ॉल्ट 48 घंटे है।
- `--json`: JSON आउटपुट करें

## उदाहरण

```bash
openclaw agent --to +15555550123 --message "स्थिति अपडेट" --deliver
openclaw agent --agent ops --message "लॉग का सारांश बनाएँ"
openclaw agent --agent ops --message-file ./task.md
openclaw agent --agent ops --model openai/gpt-5.4 --message "लॉग का सारांश बनाएँ"
openclaw agent --session-key agent:ops:incident-42 --message "स्थिति का सारांश बनाएँ"
openclaw agent --agent ops --session-key incident-42 --message "स्थिति का सारांश बनाएँ"
openclaw agent --session-id 1234 --message "इनबॉक्स का सारांश बनाएँ" --thinking medium
openclaw agent --to +15555550123 --message "लॉग ट्रेस करें" --verbose on --json
openclaw agent --agent ops --message "रिपोर्ट बनाएँ" --deliver --reply-channel slack --reply-to "#reports"
openclaw agent --agent ops --message "स्थानीय रूप से चलाएँ" --local
```

## टिप्पणियाँ

- `--message` या `--message-file` में से ठीक एक दें। `--message-file` आरंभिक UTF-8 BOM हटाता है और बहु-पंक्ति सामग्री बनाए रखता है; यह उन फ़ाइलों को अस्वीकार करता है जो मान्य UTF-8 नहीं हैं। 4 MiB से बड़ी फ़ाइलें डिस्पैच से पहले अस्वीकार कर दी जाती हैं।
- स्लैश कमांड (उदाहरण के लिए `/compact`) `--message` के माध्यम से नहीं चल सकते। CLI उन्हें अस्वीकार करता है और इसके बजाय आपको प्रथम-श्रेणी कमांड की ओर निर्देशित करता है (Compaction के लिए `openclaw sessions compact <key>`)।
- `--local` रन एकबारगी होते हैं: रन के लिए खोले गए बंडल किए हुए MCP लूपबैक संसाधन और सक्रिय Claude stdio सत्र उत्तर के बाद समाप्त कर दिए जाते हैं, इसलिए स्क्रिप्ट द्वारा किए गए आह्वान स्थानीय चाइल्ड प्रक्रियाओं को चालू नहीं छोड़ते। इसके बजाय, Gateway-समर्थित रन, Gateway के स्वामित्व वाले MCP लूपबैक संसाधनों को चालू Gateway प्रक्रिया के अंतर्गत बनाए रखते हैं।
- `--local` के साथ स्टैंडअलोन एम्बेडेड निष्पादन, पुनरारंभ पुनर्प्राप्ति लंबित होने पर किसी मौजूदा मुख्य सत्र का पुनः उपयोग करने से मना करता है। टर्न को किसी स्वस्थ Gateway के माध्यम से चलाएँ, या वहाँ `/new` अथवा `/reset` से उसे रीसेट करें; एक स्वतंत्र एम्बेडेड प्रक्रिया उस पुनर्प्राप्ति स्वामी का Gateway स्कैनर के साथ सुरक्षित रूप से समन्वय नहीं कर सकती।
- `--agent`, `--channel` और `--to` को एक साथ उपयोग करने पर, सत्र रूटिंग चैनल के प्रामाणिक प्राप्तकर्ता और `session.dmScope` का अनुसरण करती है। स्थिर केवल-आउटबाउंड प्राप्तकर्ता पहचान वाले चैनल, एजेंट के मुख्य सत्र से पृथक प्रदाता-स्वामित्व वाले सत्र का उपयोग करते हैं। `--reply-channel` और `--reply-account` केवल डिलीवरी को प्रभावित करते हैं।
- `--session-key` एक स्पष्ट सत्र कुंजी चुनता है। एजेंट-उपसर्ग वाली कुंजियों में `agent:<agent-id>:<session-key>` का उपयोग होना आवश्यक है, और दोनों दिए जाने पर `--agent` को कुंजी की एजेंट आईडी से मेल खाना आवश्यक है। बिना उपसर्ग वाली गैर-सेंटिनल कुंजियाँ, दिए जाने पर `--agent` तक सीमित होती हैं, अन्यथा कॉन्फ़िगर किए गए डिफ़ॉल्ट एजेंट तक; उदाहरण के लिए `--agent ops --session-key incident-42`, `agent:ops:incident-42` पर रूट करता है। शाब्दिक कुंजियाँ `global` और `unknown` केवल तभी बिना दायरे की रहती हैं, जब कोई `--agent` न दिया गया हो।
- `--json` JSON प्रतिक्रिया के लिए stdout आरक्षित करता है; Gateway, Plugin और `--local` निदान stderr पर जाते हैं, ताकि स्क्रिप्ट stdout को सीधे पार्स कर सकें।
- अस्थायी हैंडशेक पुनर्प्रयास समाप्त होने के बाद, Gateway की समय-सीमा समाप्त होने या कनेक्शन बंद होने पर कमांड विफल हो जाता है; CLI कभी भी टर्न को चुपचाप एम्बेडेड रूप में दोबारा नहीं चलाता। ट्रांसपोर्ट हानि अस्पष्ट होती है—हो सकता है Gateway ने टर्न स्वीकार कर लिया हो और अभी भी उसे पूरा कर रहा हो—इसलिए stderr संकेत, टर्न को दो बार निष्पादित होने से बचाने के लिए, पुनर्प्रयास करने या `--local` के साथ दोबारा चलाने से पहले `openclaw gateway status` और सत्र ट्रांस्क्रिप्ट जाँचने को कहता है।
- `SIGTERM`/`SIGINT` प्रतीक्षारत Gateway-समर्थित अनुरोध को बाधित करते हैं; यदि Gateway ने रन पहले ही स्वीकार कर लिया है, तो CLI बाहर निकलने से पहले उस रन आईडी के लिए `chat.abort` भी भेजता है। `--local` रन वही सिग्नल प्राप्त करते हैं, लेकिन `chat.abort` नहीं भेजते। पहले अग्रेषित `SIGINT` या `SIGTERM` से समाप्त होने वाली लॉन्चर चाइल्ड प्रक्रिया क्रमशः स्थिति 130 या 143 के साथ बाहर निकलती है। यदि आंतरिक रन-डीडुप कुंजी में इस सत्र के लिए पहले से कोई सक्रिय रन है, तो प्रतिक्रिया `status: "in_flight"` रिपोर्ट करती है और गैर-JSON CLI खाली उत्तर के बजाय stderr निदान प्रिंट करता है। बाहरी cron/systemd रैपरों के लिए, `timeout -k 60 600 openclaw agent ...` जैसा हार्ड-किल बैकस्टॉप रखें, ताकि शटडाउन समाप्त न हो पाने पर सुपरवाइज़र प्रक्रिया को समाप्त कर सके।
- जब यह कमांड `models.json` पुनर्जनन ट्रिगर करता है, तब SecretRef-प्रबंधित प्रदाता क्रेडेंशियल गैर-गोपनीय मार्कर के रूप में स्थायी किए जाते हैं (उदाहरण के लिए एनवायरनमेंट वेरिएबल नाम, `secretref-env:ENV_VAR_NAME`, या `secretref-managed`), कभी भी हल किए गए गोपनीय प्लेनटेक्स्ट के रूप में नहीं। मार्कर लेखन सक्रिय स्रोत कॉन्फ़िगरेशन स्नैपशॉट से आते हैं, हल किए गए रनटाइम गोपनीय मानों से नहीं।

## JSON डिलीवरी स्थिति

`--json --deliver` के साथ, CLI JSON प्रतिक्रिया में शीर्ष-स्तरीय `deliveryStatus` शामिल होता है, ताकि स्क्रिप्ट डिलीवर किए गए, रोके गए, आंशिक और विफल प्रेषणों में अंतर कर सकें:

```json
{
  "payloads": [{ "text": "रिपोर्ट तैयार है", "mediaUrl": null }],
  "meta": { "durationMs": 1200 },
  "deliveryStatus": {
    "requested": true,
    "attempted": true,
    "status": "sent",
    "succeeded": true,
    "resultCount": 1
  }
}
```

Gateway-समर्थित CLI प्रतिक्रियाएँ `result.deliveryStatus` पर अपरिष्कृत Gateway परिणाम संरचना भी बनाए रखती हैं।

`deliveryStatus.status` इनमें से एक है:

| स्थिति           | अर्थ                                                                                                                                    |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `sent`           | डिलीवरी पूरी हुई।                                                                                                                        |
| `suppressed`     | डिलीवरी जानबूझकर नहीं भेजी गई (उदाहरण के लिए संदेश-प्रेषण हुक ने इसे रद्द कर दिया, या कोई दृश्य परिणाम नहीं था)। अंतिम स्थिति, कोई पुनर्प्रयास नहीं। |
| `partial_failed` | बाद के पेलोड के विफल होने से पहले कम-से-कम एक पेलोड भेजा गया।                                                                                   |
| `failed`         | कोई स्थायी प्रेषण पूरा नहीं हुआ, या डिलीवरी प्रीफ़्लाइट विफल हुई।                                                                                   |

सामान्य फ़ील्ड:

- `requested`: ऑब्जेक्ट मौजूद होने पर हमेशा `true`।
- `attempted`: स्थायी प्रेषण पथ चलने के बाद `true`; प्रीफ़्लाइट विफलताओं या कोई दृश्य पेलोड न होने पर `false`।
- `succeeded`: `true`, `false`, या `"partial"`; `"partial"` को `status: "partial_failed"` के साथ जोड़ा जाता है।
- `reason`: स्थायी डिलीवरी या प्रीफ़्लाइट सत्यापन से प्राप्त लोअरकेस स्नेक-केस कारण। ज्ञात मानों में `cancelled_by_message_sending_hook`, `no_visible_payload`, `no_visible_result`, `channel_resolved_to_internal`, `unknown_channel`, `invalid_delivery_target`, और `no_delivery_target` शामिल हैं; विफल स्थायी प्रेषण विफल चरण की रिपोर्ट भी कर सकते हैं। अज्ञात मानों को अपारदर्शी मानें, क्योंकि यह समुच्चय विस्तृत हो सकता है।
- `resultCount`: उपलब्ध होने पर चैनल प्रेषण परिणामों की संख्या।
- `sentBeforeError`: जब किसी आंशिक विफलता में त्रुटि होने से पहले कम-से-कम एक पेलोड भेजा गया हो, तब `true`।
- `error`: विफल या आंशिक रूप से विफल प्रेषणों के लिए `true`।
- `errorMessage`: केवल तभी मौजूद होता है, जब अंतर्निहित डिलीवरी त्रुटि संदेश कैप्चर किया गया हो। प्रीफ़्लाइट विफलताओं में `error`/`reason` होते हैं, लेकिन `errorMessage` नहीं।
- `payloadOutcomes`: उपलब्ध होने पर `index`, `status`, `reason`, `resultCount`, `error`, `stage`, `sentBeforeError`, या हुक मेटाडेटा वाले वैकल्पिक प्रति-पेलोड परिणाम।
