# `openclaw path`

`oc://` एड्रेसिंग योजना तक शेल पहुँच: एड्रेस किए जा सकने वाले वर्कस्पेस फ़ाइलों (markdown, jsonc,
jsonl, yaml/yml/lobster) का निरीक्षण और संपादन करने के लिए प्रकार के अनुसार डिस्पैच होने वाला एक पथ सिंटैक्स।
स्वयं होस्ट करने वाले, Plugin लेखक और एडिटर एक्सटेंशन इसका उपयोग किसी संकीर्ण स्थान को पढ़ने, खोजने या
अपडेट करने के लिए करते हैं, बिना प्रत्येक फ़ाइल के लिए अलग पार्सर स्वयं बनाए।

`path` बंडल किए गए वैकल्पिक `oc-path` Plugin द्वारा प्रदान किया जाता है। पहले
उपयोग से पूर्व इसे सक्षम करें:

```bash
openclaw plugins enable oc-path
```

CLI क्रियाएँ एड्रेसिंग मॉडल के अनुरूप हैं:

- `resolve` ठोस है और केवल एक मिलान करता है।
- `find` वाइल्डकार्ड, यूनियन, प्रेडिकेट और
  स्थितिगत विस्तार के लिए बहु-मिलान क्रिया है।
- `set` केवल ठोस पथ या प्रविष्टि मार्कर स्वीकार करता है; वाइल्डकार्ड पैटर्न
  लिखने से पहले अस्वीकार कर दिए जाते हैं।
- `validate` फ़ाइल सिस्टम तक पहुँचे बिना पथ को पार्स करता है।
- `emit` किसी फ़ाइल को पार्स + उत्सर्जन के माध्यम से राउंड-ट्रिप करता है (बाइट-सटीकता निदान)।

## इसका उपयोग क्यों करें

OpenClaw की स्थिति मानव-संपादित markdown, टिप्पणियों वाले JSONC
कॉन्फ़िगरेशन, केवल-परिशिष्ट JSONL लॉग और YAML वर्कफ़्लो/विनिर्देश फ़ाइलों में फैली होती है। स्क्रिप्ट, हुक
और एजेंट को अक्सर उन फ़ाइलों से केवल एक छोटा मान चाहिए होता है: कोई फ्रंटमैटर कुंजी, कोई
Plugin सेटिंग, कोई लॉग रिकॉर्ड फ़ील्ड, कोई YAML चरण या किसी
नामित अनुभाग के अंतर्गत कोई बुलेट आइटम।

`openclaw path` उन कॉलर को प्रत्येक फ़ाइल प्रकार के लिए अलग-अलग बनाए गए
grep, regex या पार्सर के बजाय एक स्थिर पता देता है। उसी `oc://` पथ को टर्मिनल से सत्यापित,
रिज़ॉल्व, खोजा, ड्राई-रन और लिखा जा सकता है, जिससे संकीर्ण
ऑटोमेशन की समीक्षा और पुनरावृत्ति संभव रहती है। यह शेष फ़ाइल को सुरक्षित रखता है, इसलिए
एक लीफ़ लिखने से उसकी टिप्पणियाँ, पंक्ति-अंत या निकटवर्ती
फ़ॉर्मैटिंग प्रभावित नहीं होती।

जब इच्छित वस्तु का तार्किक पता हो, लेकिन फ़ाइल का स्वरूप
अलग-अलग हो, तब इसका उपयोग करें:

- कोई हुक टिप्पणियों वाले JSONC से एक सेटिंग पढ़ता है और
  मान वापस लिखते समय टिप्पणियाँ नहीं खोता।
- कोई रखरखाव स्क्रिप्ट पूरे लॉग को कस्टम पार्सर में
  लोड किए बिना JSONL लॉग में प्रत्येक मिलते हुए इवेंट फ़ील्ड को खोजती है।
- कोई एडिटर slug के आधार पर markdown अनुभाग या बुलेट आइटम पर जाता है, फिर
  रिज़ॉल्व हुई सटीक पंक्ति को रेंडर करता है।
- कोई एजेंट छोटा वर्कस्पेस संपादन लागू करने से पहले उसका ड्राई-रन करता है और
  समीक्षा में बदले हुए बाइट दिखाई देते हैं।

सामान्य संपूर्ण-फ़ाइल संपादनों, विस्तृत कॉन्फ़िगरेशन माइग्रेशन या
मेमोरी-विशिष्ट लेखन के लिए `openclaw path` का उपयोग न करें; इनके लिए स्वामी कमांड या Plugin का उपयोग किया जाना चाहिए। `path`
छोटे, एड्रेस किए जा सकने वाले फ़ाइल संचालन के लिए है, जहाँ पुनरावृत्त किया जा सकने वाला टर्मिनल कमांड
एक और विशेष रूप से बनाए गए पार्सर से बेहतर है।

## इसका उपयोग कैसे किया जाता है

मानव-संपादित कॉन्फ़िगरेशन फ़ाइल से एक मान पढ़ें:

```bash
openclaw path resolve 'oc://config.jsonc/plugins/github/enabled'
```

डिस्क को छुए बिना लेखन का पूर्वावलोकन करें:

```bash
openclaw path set 'oc://config.jsonc/plugins/github/enabled' 'true' --dry-run
```

केवल-परिशिष्ट JSONL लॉग में मिलते हुए रिकॉर्ड खोजें:

```bash
openclaw path find 'oc://session.jsonl/[event=tool_call]/name'
```

markdown में किसी निर्देश को पंक्ति संख्या के बजाय अनुभाग और आइटम के आधार पर एड्रेस करें:

```bash
openclaw path resolve 'oc://AGENTS.md/runtime-safety/openclaw-gateway'
```

स्क्रिप्ट द्वारा पढ़ने या लिखने से पहले CI या प्रीफ़्लाइट स्क्रिप्ट में पथ को सत्यापित करें:

```bash
openclaw path validate 'oc://AGENTS.md/tools/$last/risk'
```

ये कमांड शेल स्क्रिप्ट में कॉपी किए जा सकने के उद्देश्य से बनाए गए हैं। जब
किसी कॉलर को संरचित आउटपुट चाहिए तो `--json` और जब कोई व्यक्ति परिणाम का निरीक्षण कर रहा हो
तो `--human` का उपयोग करें।

## यह कैसे काम करता है

1. `oc://` पते को स्लॉट में पार्स करता है: फ़ाइल, अनुभाग, आइटम, फ़ील्ड और एक
   वैकल्पिक सत्र क्वेरी।
2. लक्ष्य एक्सटेंशन (`.md`, `.jsonc`,
   `.json`, `.jsonl`, `.ndjson`, `.yaml`, `.yml`, `.lobster`) से फ़ाइल-प्रकार एडाप्टर चुनता है।
3. स्लॉट को उस फ़ाइल प्रकार की संरचना के अनुसार रिज़ॉल्व करता है: markdown
   शीर्षक/आइटम, JSONC ऑब्जेक्ट कुंजियाँ/ऐरे इंडेक्स, JSONL पंक्ति रिकॉर्ड या
   YAML मैप/अनुक्रम Node।
4. `set` के लिए, उसी एडाप्टर के माध्यम से संपादित बाइट उत्सर्जित करता है, ताकि फ़ाइल के अछूते भागों
   की टिप्पणियाँ, पंक्ति-अंत और निकटवर्ती फ़ॉर्मैटिंग वहाँ सुरक्षित रहें
   जहाँ वह प्रकार इसका समर्थन करता है।

`resolve` और `set` के लिए एक ठोस लक्ष्य आवश्यक है। `find` अन्वेषणात्मक
क्रिया है: यह वाइल्डकार्ड, यूनियन, प्रेडिकेट और क्रमसूचकों को उन ठोस
मिलानों में विस्तारित करती है, जिनका आप लिखने के लिए कोई एक चुनने से पहले निरीक्षण कर सकते हैं।

## उपकमांड

| उपकमांड              | उद्देश्य                                                                     |
| ----------------------- | --------------------------------------------------------------------------- |
| `resolve <oc-path>`     | पथ पर ठोस मिलान (या "नहीं मिला") प्रिंट करें।                      |
| `find <pattern>`        | वाइल्डकार्ड / यूनियन / प्रेडिकेट पथ के मिलानों की गणना करें।                  |
| `set <oc-path> <value>` | किसी ठोस पथ पर लीफ़ या प्रविष्टि लक्ष्य लिखें। `--dry-run` समर्थित है।  |
| `validate <oc-path>`    | केवल पार्स करें; संरचनात्मक विभाजन (फ़ाइल / अनुभाग / आइटम / फ़ील्ड) प्रिंट करें। |
| `emit <file>`           | किसी फ़ाइल को पार्स + उत्सर्जन के माध्यम से राउंड-ट्रिप करें (बाइट-सटीकता निदान)।          |

## वैश्विक फ़्लैग

| फ़्लैग            | इन पर लागू                       | उद्देश्य                                                                  |
| --------------- | -------------------------------- | ------------------------------------------------------------------------ |
| `--cwd <dir>`   | `resolve`, `find`, `set`, `emit` | इस डायरेक्टरी के सापेक्ष फ़ाइल स्लॉट को रिज़ॉल्व करें (डिफ़ॉल्ट: `process.cwd()`)। |
| `--file <path>` | `resolve`, `find`, `set`, `emit` | फ़ाइल स्लॉट के रिज़ॉल्व किए गए पथ को ओवरराइड करें (निरपेक्ष पहुँच)।                |
| `--json`        | सभी                              | JSON आउटपुट बाध्य करें (जब stdout TTY न हो तब डिफ़ॉल्ट)।                    |
| `--human`       | सभी                              | मानव-पठनीय आउटपुट बाध्य करें (जब stdout TTY हो तब डिफ़ॉल्ट)।                       |
| `--value-json`  | `set`                            | JSON/JSONC/JSONL लीफ़ प्रतिस्थापन के लिए `<value>` को JSON के रूप में पार्स करें।           |
| `--dry-run`     | `set`                            | लिखे बिना वे बाइट प्रिंट करें जो लिखे जाते।                   |
| `--diff`        | `set` (`--dry-run` आवश्यक)     | पूर्ण बाइट के बजाय एकीकृत diff प्रिंट करें।                          |

`validate` केवल `--json` / `--human` लेता है; यह फ़ाइल सिस्टम तक नहीं पहुँचता, इसलिए
`--cwd` और `--file` लागू नहीं होते।

## `oc://` सिंटैक्स

```text
oc://FILE/SECTION/ITEM/FIELD?session=SCOPE
```

स्लॉट नियम: `field` के लिए `item` आवश्यक है और `item` के लिए `section` आवश्यक है। सभी
चार स्लॉट में:

- **उद्धृत खंड** — `"a/b.c"`, `/` और `.` विभाजकों के बावजूद अक्षुण्ण रहता है। सामग्री
  बाइट-शाब्दिक है; उद्धरणों के भीतर `"` और `\` अनुमत नहीं हैं। फ़ाइल स्लॉट भी
  उद्धरण-सजग है: `oc://"skills/email-drafter"/Tools/$last`,
  `skills/email-drafter` को एकल फ़ाइल पथ मानता है।
- **प्रेडिकेट** — `[k=v]`, `[k!=v]`, `[k<v]`, `[k<=v]`, `[k>v]`, `[k>=v]`।
  संख्यात्मक ऑपरेटरों के लिए दोनों पक्षों का परिमित संख्याओं में रूपांतरण संभव होना आवश्यक है।
- **यूनियन** — `{a,b,c}` किसी भी विकल्प से मेल खाता है।
- **वाइल्डकार्ड** — `*` (एकल उप-खंड) और `**` (शून्य या अधिक,
  पुनरावर्ती)। `find` इन्हें स्वीकार करता है; `resolve` और `set` इन्हें
  अस्पष्ट मानकर अस्वीकार करते हैं।
- **स्थितिगत** — `$first` / `$last` पहले / अंतिम इंडेक्स या
  घोषित कुंजी में रिज़ॉल्व होते हैं।
- **क्रमसूचक** — दस्तावेज़ क्रम के अनुसार Nवें मिलान के लिए `#N`।
- **प्रविष्टि मार्कर** — कुंजीबद्ध / इंडेक्स की गई प्रविष्टि के लिए `+`, `+key`, `+nnn`
  (`set` के साथ उपयोग करें)।
- **सत्र दायरा** — `?session=cron-daily` आदि। स्लॉट नेस्टिंग से स्वतंत्र।
  सत्र मान कच्चे होते हैं, प्रतिशत-डिकोड नहीं किए जाते; उनमें नियंत्रण
  वर्ण या आरक्षित क्वेरी सीमांकक (`?`, `&`, `%`) नहीं हो सकते।

उद्धृत, प्रेडिकेट या यूनियन खंडों के बाहर आरक्षित वर्ण (`?`, `&`, `%`)
अस्वीकार कर दिए जाते हैं। नियंत्रण वर्ण (U+0000-U+001F, U+007F)
`session` क्वेरी मान सहित कहीं भी अस्वीकार किए जाते हैं।

कैनोनिकल पथों के लिए `formatOcPath(parseOcPath(path)) === path` की गारंटी है।
पहले गैर-रिक्त `session=` मान को छोड़कर गैर-कैनोनिकल क्वेरी पैरामीटर
अनदेखे किए जाते हैं।

कठोर सीमाएँ: पथ अधिकतम 4096 बाइट, अधिकतम 4 स्लॉट (फ़ाइल/अनुभाग/आइटम/
फ़ील्ड), प्रति स्लॉट अधिकतम 64 डॉट-से-विभाजित उप-खंड और गहरे JSON पथों के लिए अधिकतम 256 नेस्टेड
ट्रैवर्सल स्तर तक सीमित है। अलग से, 16 MiB से अधिक का कोई भी JSONC/JSON फ़ाइल इनपुट,
उस फ़ाइल को लोड करने वाली किसी भी क्रिया के लिए पार्स किए जाने के बजाय पार्स निदान के साथ अस्वीकार कर दिया जाता है।

## फ़ाइल प्रकार के अनुसार एड्रेसिंग

| प्रकार          | फ़ाइल एक्सटेंशन             | एड्रेसिंग मॉडल                                                                                    |
| ------------- | --------------------------- | --------------------------------------------------------------------------------------------------- |
| Markdown      | `.md`                       | slug के अनुसार H2 अनुभाग, slug या `#N` के अनुसार बुलेट आइटम, `[frontmatter]` के माध्यम से फ्रंटमैटर।                 |
| JSONC/JSON    | `.jsonc`, `.json`           | ऑब्जेक्ट कुंजियाँ और ऐरे इंडेक्स; उद्धृत न होने पर डॉट नेस्टेड उप-खंडों को विभाजित करते हैं।                        |
| JSONL         | `.jsonl`, `.ndjson`         | शीर्ष-स्तरीय पंक्ति पते (`L1`, `L2`, `$first`, `$last`), फिर पंक्ति के भीतर JSONC-शैली का अवरोहण। |
| YAML/.lobster | `.yaml`, `.yml`, `.lobster` | मैप कुंजियाँ और अनुक्रम इंडेक्स; टिप्पणियाँ और फ़्लो शैली YAML दस्तावेज़ API द्वारा संभाली जाती हैं।        |

`resolve` एक संरचित मिलान लौटाता है: `root`, `node`, `leaf` या
`insertion-point`, जिसमें पंक्ति संख्या 1 से शुरू होती है। लीफ़ मानों को
टेक्स्ट और एक `leafType` के रूप में प्रस्तुत किया जाता है, ताकि Plugin लेखक
प्रत्येक प्रकार के AST स्वरूप पर निर्भर हुए बिना पूर्वावलोकन रेंडर कर सकें।

## परिवर्तन अनुबंध

`set` एक ठोस लक्ष्य लिखता है:

- Markdown frontmatter के मान और `- key: value` आइटम फ़ील्ड स्ट्रिंग
  लीफ़ हैं। Markdown इंसर्शन बदली गई फ़ाइल में सेक्शन, frontmatter कुंजियाँ या सेक्शन
  आइटम जोड़ते हैं और एक कैनोनिकल Markdown संरचना रेंडर करते हैं। सेक्शन
  बॉडी को `set` के माध्यम से समग्र रूप में नहीं लिखा जा सकता।
- JSONC लीफ़ राइट स्ट्रिंग मान को मौजूदा लीफ़ प्रकार में कोअर्स करते हैं
  (`string`, परिमित `number`, `true`/`false`, या `null`)। जब किसी JSONC/JSON/JSONL लीफ़ प्रतिस्थापन को `<value>` को JSON के रूप में पार्स करना हो और
  उसका आकार बदल सकता हो, तब `--value-json` का उपयोग करें, जैसे किसी स्ट्रिंग secret-ref शॉर्टहैंड को
  ऑब्जेक्ट से बदलना। JSONC ऑब्जेक्ट और ऐरे इंसर्शन `<value>` को JSON के रूप में पार्स करते हैं और सामान्य लीफ़ राइट के लिए
  `jsonc-parser` एडिट पाथ का उपयोग करते हैं, जिससे टिप्पणियाँ
  और आस-पास की फ़ॉर्मैटिंग संरक्षित रहती है।
- JSONL लीफ़ राइट किसी लाइन के भीतर JSONC की तरह कोअर्स करते हैं। पूरी लाइन का प्रतिस्थापन
  और अपेंड `<value>` को JSON के रूप में पार्स करते हैं। रेंडर किया गया JSONL फ़ाइल की
  प्रमुख LF/CRLF लाइन-एंडिंग परंपरा को संरक्षित रखता है (फ़ाइल की सभी
  नई लाइनों पर बहुमत के अनुसार, इसलिए कुछ इक्का-दुक्का LF वाली मुख्यतः CRLF फ़ाइल CRLF ही रहती है)।
- YAML लीफ़ राइट मौजूदा स्केलर प्रकार में कोअर्स करते हैं (`string`, परिमित
  `number`, `true`/`false`, या `null`)। YAML इंसर्शन मैप/सीक्वेंस अपडेट के लिए बंडल किए गए
  `yaml` पैकेज के डॉक्यूमेंट API का उपयोग करते हैं। पार्सर त्रुटियों वाले विकृत YAML
  डॉक्यूमेंट को म्यूटेशन से पहले
  `parse-error` के साथ अस्वीकार कर दिया जाता है।

जब सटीक बाइट्स महत्वपूर्ण हों, तब उपयोगकर्ता को दिखाई देने वाले राइट से पहले `--dry-run` का उपयोग करें। JSONC
और YAML एडिट मौजूदा डॉक्यूमेंट को पैच करते हैं (`jsonc-parser` या `yaml`
डॉक्यूमेंट API के माध्यम से), इसलिए अछूते बाइट्स सामान्यतः सुरक्षित रहते हैं; Markdown किसी भी एडिट पर
फ़ाइल को उसकी पार्स की गई संरचना से दोबारा बनाता है, जिससे बदले गए लीफ़ के बाहर की आकस्मिक
फ़ॉर्मैटिंग सामान्यीकृत हो सकती है। जब पूर्ण रेंडर की गई फ़ाइल के बजाय केंद्रित पहले/बाद का पैच
पूर्वावलोकन चाहिए, तब `--diff` जोड़ें।

## उदाहरण

```bash
# पाथ सत्यापित करें (फ़ाइलसिस्टम एक्सेस नहीं)
openclaw path validate 'oc://AGENTS.md/Tools/$last/risk'

# लीफ़ पढ़ें
openclaw path resolve 'oc://gateway.jsonc/version'

# वाइल्डकार्ड खोज
openclaw path find 'oc://session.jsonl/*/event' --file ./logs/session.jsonl

# राइट का ड्राई रन
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run

# यूनिफ़ाइड डिफ़ के रूप में राइट का ड्राई रन
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run --diff

# राइट लागू करें
openclaw path set 'oc://gateway.jsonc/version' '2.0'

# बाइट-फ़िडेलिटी राउंड-ट्रिप (डायग्नोस्टिक)
openclaw path emit ./AGENTS.md
```

व्याकरण के और उदाहरण:

```bash
# / या . वाली कुंजियों को उद्धृत करें
openclaw path resolve 'oc://config.jsonc/agents.defaults.models/"anthropic/claude-opus-4-7"/alias'

# गहरे JSON/JSONC पाथ स्लैश सेगमेंट का उपयोग कर सकते हैं; वे डॉटेड सबसेगमेंट में सामान्यीकृत होते हैं
openclaw path set 'oc://openclaw.json/agents/list/0/tools/exec/security' 'allowlist' --dry-run

# JSONC लीफ़ को पार्स किए गए ऑब्जेक्ट से बदलें
openclaw path set 'oc://openclaw.json/gateway/auth/token' '{"source":"file","provider":"secrets","id":"/test"}' --value-json --dry-run

# JSONC चिल्ड्रन पर प्रेडिकेट खोज
openclaw path find 'oc://config.jsonc/plugins/[enabled=true]/id'

# JSONC ऐरे में इंसर्ट करें
openclaw path set 'oc://config.jsonc/items/+1' '{"id":"new","enabled":true}' --dry-run

# JSONC ऑब्जेक्ट कुंजी इंसर्ट करें
openclaw path set 'oc://config.jsonc/plugins/+github' '{"enabled":true}' --dry-run

# JSONL इवेंट अपेंड करें
openclaw path set 'oc://session.jsonl/+' '{"event":"checkpoint","ok":true}' --file ./logs/session.jsonl

# अंतिम JSONL मान लाइन रिज़ॉल्व करें
openclaw path resolve 'oc://session.jsonl/$last/event' --file ./logs/session.jsonl

# YAML वर्कफ़्लो चरण रिज़ॉल्व करें
openclaw path resolve 'oc://workflow.yaml/steps/0/id'

# YAML स्केलर अपडेट करें
openclaw path set 'oc://workflow.yaml/steps/$last/id' 'classify-renamed' --dry-run

# Markdown frontmatter को संबोधित करें
openclaw path resolve 'oc://AGENTS.md/[frontmatter]/name'

# Markdown frontmatter इंसर्ट करें
openclaw path set 'oc://AGENTS.md/[frontmatter]/+description' 'Agent instructions' --dry-run

# Markdown आइटम फ़ील्ड खोजें
openclaw path find 'oc://SKILL.md/Tools/*/send_email'

# सेशन-स्कोप्ड पाथ सत्यापित करें
openclaw path validate 'oc://AGENTS.md/Tools/$last/risk?session=cron-daily'
```

## फ़ाइल प्रकार के अनुसार विधियाँ

सभी प्रकारों पर वही पाँच क्रियाएँ काम करती हैं; एड्रेसिंग स्कीम
फ़ाइल एक्सटेंशन के अनुसार डिस्पैच करती है।

### Markdown

```text
<!-- frontmatter.md -->
---
name: प्रारूपकार
description: ईमेल प्रारूपण एजेंट
tier: कोर
---
## टूल
- gh: GitHub CLI
- curl: HTTP क्लाइंट
- send_email: सक्षम
```

```bash
$ openclaw path resolve 'oc://x.md/[frontmatter]/tier' --file frontmatter.md --human
लीफ़ @ L4: "कोर" (स्ट्रिंग)

$ openclaw path resolve 'oc://x.md/tools/gh/gh' --file frontmatter.md --human
लीफ़ @ L9: "GitHub CLI" (स्ट्रिंग)

$ openclaw path find 'oc://x.md/tools/*' --file frontmatter.md --human
oc://x.md/tools/* के लिए 3 मिलान:
  oc://x.md/tools/gh           →  नोड @ L9 [md-item]
  oc://x.md/tools/curl         →  नोड @ L10 [md-item]
  oc://x.md/tools/send-email   →  नोड @ L11 [md-item]
```

`[frontmatter]` प्रेडिकेट YAML frontmatter ब्लॉक को संबोधित करता है; `tools`
स्लग के माध्यम से `## Tools` शीर्षक से मेल खाता है, और आइटम लीफ़ अपना स्लग रूप बनाए रखते हैं,
भले ही स्रोत अंडरस्कोर का उपयोग करता हो (`send_email` से `send-email` बनता है)।

### JSONC

```text
// config.jsonc
{
  "plugins": {
    "github": {"enabled": true, "role": "vcs"},
    "slack":  {"enabled": false, "role": "chat"}
  }
}
```

```bash
$ openclaw path resolve 'oc://config.jsonc/plugins/github/enabled' --file config.jsonc --human
लीफ़ @ L4: "true" (बूलियन)

$ openclaw path set 'oc://config.jsonc/plugins/slack/enabled' 'true' --file config.jsonc --dry-run
--dry-run: /…/config.jsonc में 142 बाइट्स लिखे जाएँगे
{
  "plugins": {
    "github": {"enabled": true, "role": "vcs"},
    "slack":  {"enabled": true, "role": "chat"}
  }
}
```

JSONC एडिट `jsonc-parser` से होकर जाते हैं, इसलिए `set` के बाद भी टिप्पणियाँ और व्हाइटस्पेस
सुरक्षित रहते हैं। कमिट करने से पहले बाइट्स की जाँच के लिए पहले `--dry-run` के साथ चलाएँ।
`.json` फ़ाइलें `.jsonc` के समान अडैप्टर और एडिट पाथ का उपयोग करती हैं।

### JSONL

```text
{"event":"start","userId":"u1","ts":1}
{"event":"action","userId":"u1","ts":2}
{"event":"end","userId":"u1","ts":3}
```

```bash
$ openclaw path find 'oc://session.jsonl/[event=action]/userId' --file session.jsonl --human
oc://session.jsonl/[event=action]/userId के लिए 1 मिलान:
  oc://session.jsonl/L2/userId  →  लीफ़ @ L2: "u1" (स्ट्रिंग)

$ openclaw path resolve 'oc://session.jsonl/L2/ts' --file session.jsonl --human
लीफ़ @ L2: "2" (संख्या)
```

प्रत्येक लाइन एक रिकॉर्ड है। जब लाइन संख्या ज्ञात न हो, तब प्रेडिकेट (`[event=action]`) से संबोधित करें,
और जब ज्ञात हो, तब कैनोनिकल `LN` सेगमेंट से।
`.ndjson` फ़ाइलें `.jsonl` के समान अडैप्टर का उपयोग करती हैं।

### YAML

```text
# workflow.yaml
name: inbox-triage
steps:
  - id: fetch
    command: gmail.search
  - id: classify
    command: openclaw.invoke
```

```bash
$ openclaw path resolve 'oc://workflow.yaml/steps/0/id' --file workflow.yaml --human
लीफ़ @ L3: "fetch" (स्ट्रिंग)

$ openclaw path set 'oc://workflow.yaml/steps/$last/id' 'classify-renamed' --file workflow.yaml --dry-run
--dry-run: /…/workflow.yaml में 99 बाइट्स लिखे जाएँगे
name: inbox-triage
steps:
  - id: fetch
    command: gmail.search
  - id: classify-renamed
    command: openclaw.invoke
```

YAML हाथ से बनाए गए पार्सर के बजाय `yaml` पैकेज के `Document` API का उपयोग करता है,
इसलिए सामान्य पार्स/एमिट राउंड-ट्रिप टिप्पणियाँ और लेखन संरचना को सुरक्षित रखते हैं,
जबकि रिज़ॉल्व किए गए पाथ JSONC के समान मैप-कुंजी / सीक्वेंस-इंडेक्स मॉडल का उपयोग करते हैं।
वही अडैप्टर `.yaml`, `.yml`, और `.lobster` फ़ाइलों को संभालता है।

## सबकमांड संदर्भ

### `resolve <oc-path>`

एक लीफ़ या नोड पढ़ें। वाइल्डकार्ड अस्वीकार किए जाते हैं — उनके लिए `find` का उपयोग करें।
मिलान होने पर `0`, स्पष्ट रूप से कोई मिलान न होने पर `1`, और पार्स त्रुटि या अस्वीकृत
पैटर्न पर `2` के साथ बाहर निकलता है।

```bash
openclaw path resolve 'oc://AGENTS.md/tools/gh/risk' --human
openclaw path resolve 'oc://gateway.jsonc/server/port' --json
```

### `find <pattern>`

वाइल्डकार्ड / प्रेडिकेट / यूनियन पैटर्न के प्रत्येक मिलान की गणना करें। कम-से-कम एक मिलान पर `0`
और शून्य मिलान पर `1` के साथ बाहर निकलता है। फ़ाइल-स्लॉट वाइल्डकार्ड
`OC_PATH_FILE_WILDCARD_UNSUPPORTED` के साथ अस्वीकार किए जाते हैं — एक ठोस फ़ाइल दें (बहु-फ़ाइल
ग्लॉबिंग भविष्य की सुविधा है)।

```bash
openclaw path find 'oc://AGENTS.md/tools/**/risk'
openclaw path find 'oc://session.jsonl/[event=action]/userId'
openclaw path find 'oc://config.jsonc/plugins/{github,slack}/enabled'
```

### `set <oc-path> <value>`

एक लीफ़ लिखें। फ़ाइल को छुए बिना लिखे जाने वाले बाइट्स का पूर्वावलोकन करने के लिए इसे
`--dry-run` के साथ जोड़ें। यूनिफ़ाइड डिफ़ पूर्वावलोकन के लिए `--diff` जोड़ें।
सफल राइट पर `0`, सब्सट्रेट के अस्वीकार करने पर `1` (उदाहरण के लिए,
सेंटिनल गार्ड हिट), और पार्स त्रुटियों पर `2` के साथ बाहर निकलता है।

```bash
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run --diff
openclaw path set 'oc://gateway.jsonc/version' '2.0'
openclaw path set 'oc://AGENTS.md/Tools/+gh/risk' 'low'
```

यदि नामित चाइल्ड पहले से मौजूद नहीं है, तो `+key` इंसर्शन मार्कर उसे बनाता है;
`+nnn` और केवल `+` क्रमशः इंडेक्स्ड और अपेंड इंसर्शन के लिए काम करते हैं।

### `validate <oc-path>`

केवल पार्स जाँच। कोई फ़ाइलसिस्टम एक्सेस नहीं। यह तब उपयोगी है जब वेरिएबल प्रतिस्थापित करने से पहले
यह पुष्टि करनी हो कि कोई टेम्पलेट पाथ सुगठित है, या डीबगिंग के लिए
संरचनात्मक विश्लेषण चाहिए:

```bash
$ openclaw path validate 'oc://AGENTS.md/tools/gh' --human
मान्य: oc://AGENTS.md/tools/gh
  फ़ाइल:    AGENTS.md
  सेक्शन: tools
  आइटम:    gh
```

मान्य होने पर `0`, अमान्य होने पर `1` (संरचित `code` और
`message` के साथ), तथा आर्ग्युमेंट त्रुटियों पर `2` के साथ बाहर निकलता है।

### `emit <file>`

किसी फ़ाइल को उसके प्रकार-विशिष्ट पार्सर और एमिटर के माध्यम से राउंड-ट्रिप करें। सही फ़ाइल पर आउटपुट
इनपुट के बाइट-समान होना चाहिए; अंतर पार्सर बग या सेंटिनल हिट का संकेत देता है।
वास्तविक इनपुट पर सब्सट्रेट व्यवहार की डीबगिंग के लिए उपयोगी।

```bash
openclaw path emit ./AGENTS.md
openclaw path emit ./gateway.jsonc --json
```

## निकास कोड

| कोड | अर्थ                                                                    |
| ---- | -------------------------------------------------------------------------- |
| `0`  | सफलता। (`resolve` / `find`: कम-से-कम एक मिलान। `set`: राइट सफल।) |
| `1`  | कोई मिलान नहीं, या `set` को सब्सट्रेट ने अस्वीकार किया (कोई सिस्टम-स्तरीय त्रुटि नहीं)।      |
| `2`  | आर्ग्युमेंट या पार्स त्रुटि।                                                   |

## आउटपुट मोड

`openclaw path` TTY-जागरूक है: टर्मिनल पर मानव-पठनीय आउटपुट, और stdout के पाइप या रीडायरेक्ट होने पर JSON।
`--json` और `--human` स्वतः-पहचान को ओवरराइड करते हैं।

## टिप्पणियाँ

- `set` सब्सट्रेट के emit पथ के माध्यम से बाइट्स लिखता है, जो
  रिडैक्शन-सेंटिनल सुरक्षा को स्वचालित रूप से लागू करता है। ऐसा लीफ़ जिसमें
  `__OPENCLAW_REDACTED__` (यथावत या सबस्ट्रिंग के रूप में) हो, लिखते
  समय अस्वीकार कर दिया जाता है।
- JSONC पार्सिंग और लीफ़ संपादन Plugin-स्थानीय `jsonc-parser`
  निर्भरता का उपयोग करते हैं, इसलिए सामान्य लीफ़ लेखन में टिप्पणियाँ और फ़ॉर्मैटिंग
  हाथ से बनाए गए पार्सर/पुनः-रेंडर पथ से गुज़रने के बजाय संरक्षित रहती हैं।
- `path` अंतिम-ज्ञात-सही (LKG) कॉन्फ़िगरेशन ट्रैकिंग या पुनर्प्राप्ति से अवगत नहीं है;
  उस जीवनचक्र का स्वामित्व कहीं और है। यदि `path` के माध्यम से संपादित की गई फ़ाइल
  भी LKG द्वारा ट्रैक की जाती है, तो अगला कॉन्फ़िगरेशन पठन तय करता है कि उसे प्रोमोट करना है या
  पुनर्प्राप्त करना है; `path` संपादन को उस फ़ाइल में किसी अन्य प्रत्यक्ष लेखन के
  समान मानें।
