# `openclaw cron`

Gateway शेड्यूलर के लिए Cron जॉब प्रबंधित करें।

<Tip>
संपूर्ण कमांड सतह के लिए `openclaw cron --help` चलाएँ। अवधारणात्मक मार्गदर्शिका के लिए [Cron जॉब](/hi/automation/cron-jobs) देखें।
</Tip>

<Note>
सभी Cron परिवर्तन (`add`/`create`, `update`/`edit`, `remove`, `run`) के लिए `operator.admin` आवश्यक है। कमांड-पेलोड रन सीधे Gateway प्रक्रिया में निष्पादित होते हैं, एजेंट के `tools.exec` टूल कॉल के रूप में नहीं; मॉडल को दिखाई देने वाले exec टूल अब भी `tools.exec.*` और exec अनुमोदनों द्वारा नियंत्रित होते हैं।
</Note>

## जॉब शीघ्र बनाएँ

`openclaw cron create`, `openclaw cron add` का उपनाम है। नए जॉब के लिए पहले शेड्यूल और फिर प्रॉम्प्ट रखें:

```bash
openclaw cron create "0 7 * * *" \
  "रात भर के अपडेट का सारांश दें।" \
  --name "सुबह का संक्षिप्त विवरण" \
  --agent ops
```

जब जॉब को किसी चैट लक्ष्य पर डिलीवर करने के बजाय पूर्ण पेलोड को POST करना हो, तब `--webhook <url>` का उपयोग करें:

```bash
openclaw cron create "0 18 * * 1-5" \
  "आज के डिप्लॉय का JSON के रूप में सारांश दें।" \
  --name "डिप्लॉय सारांश" \
  --webhook "https://example.invalid/openclaw/cron"
```

OpenClaw Cron के भीतर बिना कोई पृथक एजेंट/मॉडल रन शुरू किए चलने वाले नियतात्मक शेल-शैली जॉब के लिए `--command` का उपयोग करें:

```bash
openclaw cron create "*/15 * * * *" \
  --name "क्यू गहराई जाँच" \
  --command "scripts/check-queue.sh" \
  --command-cwd "/srv/app" \
  --announce \
  --channel telegram \
  --to "-1001234567890"
```

`--command <shell>`, `argv: ["sh", "-lc", <shell>]` संग्रहीत करता है। सटीक argv निष्पादन के लिए `--command-argv '["node","scripts/report.mjs"]'` का उपयोग करें। कमांड जॉब stdout/stderr कैप्चर करते हैं, सामान्य Cron इतिहास दर्ज करते हैं, और आउटपुट को पृथक जॉब के समान `announce`, `webhook`, या `none` डिलीवरी मोड के माध्यम से रूट करते हैं। केवल `NO_REPLY` प्रिंट करने वाला कमांड दबा दिया जाता है।

## सत्र

`--session`, `main`, `isolated`, `current`, या `session:<id>` स्वीकार करता है।

## डिलीवरी

`openclaw cron list` और `openclaw cron show <job-id>` समाधान किए गए डिलीवरी रूट का पूर्वावलोकन दिखाते हैं। `channel: "last"` के लिए पूर्वावलोकन दिखाता है कि रूट मुख्य या वर्तमान सत्र से समाधान किया गया है, अथवा सुरक्षित रूप से विफल होगा।

प्रदाता-उपसर्ग वाले लक्ष्य अनसुलझे घोषणा चैनलों का भेद स्पष्ट कर सकते हैं। उदाहरण के लिए, जब `delivery.channel` छोड़ा गया हो या `last` हो, तो `to: "telegram:123"` Telegram चुनता है। केवल लोड किए गए Plugin द्वारा घोषित उपसर्ग ही प्रदाता चयनकर्ता होते हैं। यदि `delivery.channel` स्पष्ट है, तो उपसर्ग उस चैनल से मेल खाना चाहिए; `to: "telegram:123"` के साथ `channel: "whatsapp"` अस्वीकार कर दिया जाता है। `imessage:` और `sms:` जैसे सेवा उपसर्ग चैनल-स्वामित्व वाला लक्ष्य सिंटैक्स बने रहते हैं।

<Note>
पृथक `cron add` जॉब में डिफ़ॉल्ट रूप से `--announce` डिलीवरी होती है। आउटपुट को आंतरिक रखने के लिए `--no-deliver` का उपयोग करें। `--deliver`, `--announce` के अप्रचलित उपनाम के रूप में बना हुआ है।
</Note>

### डिलीवरी का स्वामित्व

पृथक Cron चैट डिलीवरी एजेंट और रनर के बीच साझा होती है:

- चैट रूट उपलब्ध होने पर एजेंट `message` टूल का उपयोग करके सीधे भेज सकता है।
- `announce` अंतिम उत्तर को फ़ॉलबैक के रूप में केवल तब डिलीवर करता है, जब एजेंट ने समाधान किए गए लक्ष्य पर सीधे नहीं भेजा हो।
- `webhook` पूर्ण पेलोड को किसी URL पर पोस्ट करता है।
- `none` रनर फ़ॉलबैक डिलीवरी अक्षम करता है।

Webhook डिलीवरी सेट करने के लिए `cron add|create --webhook <url>` या `cron edit <job-id> --webhook <url>` का उपयोग करें। `--webhook` को `--announce`, `--no-deliver`, `--channel`, `--to`, `--thread-id`, या `--account` जैसे चैट डिलीवरी फ़्लैग के साथ संयोजित न करें।

`cron edit <job-id>`, `--clear-channel`, `--clear-to`, `--clear-thread-id`, और `--clear-account` के साथ अलग-अलग डिलीवरी रूटिंग फ़ील्ड अनसेट कर सकता है (प्रत्येक को उसके संगत सेट फ़्लैग के साथ संयोजित करने पर अस्वीकार कर दिया जाता है)। `--no-deliver` के विपरीत, जो केवल रनर फ़ॉलबैक डिलीवरी अक्षम करता है, ये संग्रहीत फ़ील्ड हटा देते हैं, ताकि जॉब अपने रूट के उस भाग को फिर से डिफ़ॉल्ट से समाधान करे।

`--announce` अंतिम उत्तर के लिए रनर फ़ॉलबैक डिलीवरी है। `--no-deliver` उस फ़ॉलबैक को अक्षम करता है, लेकिन चैट रूट उपलब्ध होने पर एजेंट का `message` टूल नहीं हटाता।

सक्रिय चैट से बनाए गए रिमाइंडर फ़ॉलबैक घोषणा डिलीवरी के लिए लाइव चैट डिलीवरी लक्ष्य सुरक्षित रखते हैं। आंतरिक सत्र कुंजियाँ लोअरकेस हो सकती हैं; Matrix रूम आईडी जैसे केस-संवेदी प्रदाता आईडी के लिए उन्हें सत्य के स्रोत के रूप में उपयोग न करें।

### विफलता डिलीवरी

विफलता सूचनाएँ इस क्रम में समाधान होती हैं:

1. जॉब पर `delivery.failureDestination`।
2. वैश्विक `cron.failureDestination`।
3. जॉब का प्राथमिक घोषणा लक्ष्य (जब ऊपर दिए गए दोनों में से कोई भी किसी ठोस गंतव्य में समाधान न हो)।

<Note>
मुख्य-सत्र जॉब `delivery.failureDestination` का उपयोग केवल तभी कर सकते हैं, जब प्राथमिक डिलीवरी मोड `webhook` हो। पृथक जॉब इसे सभी मोड में स्वीकार करते हैं।
</Note>

पृथक Cron रन, रन-स्तरीय एजेंट विफलताओं को जॉब त्रुटि मानते हैं, भले ही कोई उत्तर पेलोड उत्पन्न न हो; इसलिए मॉडल/प्रदाता विफलताएँ फिर भी त्रुटि काउंटर बढ़ाती हैं और विफलता सूचनाएँ ट्रिगर करती हैं।

कमांड Cron जॉब कोई पृथक एजेंट टर्न शुरू नहीं करते। शून्य एग्ज़िट कोड `ok` दर्ज करता है; गैर-शून्य एग्ज़िट, सिग्नल, टाइमआउट या बिना-आउटपुट टाइमआउट `error` दर्ज करता है और वही विफलता सूचना पथ ट्रिगर कर सकता है।

यदि कोई पृथक रन पहले मॉडल अनुरोध से पहले टाइमआउट हो जाता है, तो `openclaw cron show` और `openclaw cron runs` में चरण-विशिष्ट त्रुटि शामिल होती है, जैसे `setup timed out before runner start`, या अंतिम ज्ञात स्टार्टअप चरण बताने वाला रुकावट संदेश (उदाहरण के लिए `context-engine`)। CLI-समर्थित प्रदाताओं के लिए, बाहरी CLI टर्न शुरू होने तक प्री-मॉडल वॉचडॉग सक्रिय रहता है, इसलिए सत्र लुकअप, हुक, प्रमाणीकरण, प्रॉम्प्ट और CLI सेटअप की रुकावटें प्री-मॉडल Cron विफलताओं के रूप में रिपोर्ट की जाती हैं।

## शेड्यूलिंग

### एक-बार चलने वाले जॉब

`--at <datetime>` एक बार चलने वाला रन शेड्यूल करता है। ऑफ़सेट-रहित दिनांक-समय को UTC माना जाता है, जब तक कि आप `--tz <iana>` भी न दें, जो दी गई समय-क्षेत्र में वॉल-क्लॉक समय की व्याख्या करता है।

<Note>
एक-बार चलने वाले जॉब सफल होने के बाद डिफ़ॉल्ट रूप से मिट जाते हैं। उन्हें सुरक्षित रखने के लिए `--keep-after-run` का उपयोग करें।
</Note>

### आवर्ती जॉब

लगातार त्रुटियों के बाद आवर्ती जॉब एक्सपोनेंशियल पुनःप्रयास बैकऑफ़ का उपयोग करते हैं: 30s, 1m, 5m, 15m, 60m। अगले सफल रन के बाद शेड्यूल सामान्य हो जाता है।

छोड़े गए रन को निष्पादन त्रुटियों से अलग ट्रैक किया जाता है। वे पुनःप्रयास बैकऑफ़ को प्रभावित नहीं करते, लेकिन `openclaw cron edit <job-id> --failure-alert-include-skipped` विफलता अलर्ट को बार-बार छोड़े गए रन की सूचनाएँ भेजने के लिए सक्षम कर सकता है।

किसी स्थानीय कॉन्फ़िगर किए गए मॉडल प्रदाता (लूपबैक, निजी नेटवर्क या `.local` पर बेस URL) को लक्षित करने वाले पृथक जॉब के लिए, Cron एजेंट टर्न शुरू करने से पहले एक हल्का प्रदाता प्रीफ़्लाइट चलाता है: `api: "ollama"` प्रदाताओं की जाँच `/api/tags` पर होती है; अन्य स्थानीय OpenAI-संगत प्रदाताओं (`api: "openai-completions"`, जैसे vLLM, SGLang, LM Studio) की जाँच `/models` पर होती है। यदि एंडपॉइंट तक नहीं पहुँचा जा सकता, तो रन `skipped` के रूप में दर्ज होता है और बाद के शेड्यूल पर पुनः प्रयास किया जाता है; पहुँच-क्षमता परिणाम प्रत्येक एंडपॉइंट के लिए 5 मिनट तक कैश किया जाता है, ताकि एक ही स्थानीय सर्वर का उपयोग करने वाले कई जॉब बार-बार जाँच करके उस पर अत्यधिक भार न डालें।

Cron जॉब, लंबित रनटाइम स्थिति और रन इतिहास साझा SQLite स्थिति डेटाबेस में रहते हैं। पुराने `jobs.json`, `<name>-state.json`, और `runs/*.jsonl` फ़ाइलें एक बार आयात की जाती हैं और `.migrated` प्रत्यय के साथ उनका नाम बदल दिया जाता है। आयात के बाद JSON फ़ाइलें संपादित करने के बजाय `openclaw cron add|edit|remove` से शेड्यूल संपादित करें।

### मैन्युअल रन

`openclaw cron run <job-id>` डिफ़ॉल्ट रूप से बलपूर्वक रन करता है और मैन्युअल रन के क्यू में लगते ही लौट आता है। सफल प्रतिक्रियाओं में `{ ok: true, enqueued: true, runId }` शामिल होता है। बाद का परिणाम देखने के लिए लौटाए गए `runId` का उपयोग करें:

```bash
openclaw cron run <job-id>
openclaw cron runs --id <job-id> --run-id <run-id>
```

जब किसी स्क्रिप्ट को तब तक अवरुद्ध रहना चाहिए जब तक वही क्यू किया गया रन टर्मिनल स्थिति दर्ज न कर ले, तब `--wait` जोड़ें:

```bash
openclaw cron run <job-id> --wait --wait-timeout 10m --poll-interval 2s
```

`--wait` के साथ, CLI पहले फिर भी `cron.run` कॉल करता है, फिर लौटाए गए `runId` के लिए `cron.runs` को पोल करता है। कमांड केवल तभी `0` के साथ बाहर निकलता है, जब रन `ok` स्थिति में पूरा हो। जब रन `error` या `skipped` के साथ पूरा होता है, Gateway प्रतिक्रिया में `runId` शामिल नहीं होता, या `--wait-timeout` समाप्त हो जाता है (डिफ़ॉल्ट `10m`, और डिफ़ॉल्ट रूप से प्रत्येक `2s` पर पोल किया जाता है), तब यह गैर-शून्य के साथ बाहर निकलता है। `--poll-interval` शून्य से अधिक होना चाहिए।

<Note>
जब मैन्युअल कमांड को केवल तभी चलाना हो, जब जॉब वर्तमान में नियत हो, तब `--due` का उपयोग करें। यदि `--due --wait` किसी रन को क्यू में नहीं लगाता, तो कमांड पोल करने के बजाय सामान्य गैर-रन प्रतिक्रिया लौटाता है।
</Note>

## मॉडल

`cron add|edit --model <ref>` जॉब के लिए अनुमत मॉडल चुनता है। `cron add|edit --fallbacks <list>` प्रति-जॉब फ़ॉलबैक मॉडल सेट करता है, उदाहरण के लिए `--fallbacks openrouter/gpt-4.1-mini,openai/gpt-5`; बिना फ़ॉलबैक वाले सख्त रन के लिए `--fallbacks ""` दें। `cron edit <job-id> --clear-fallbacks` प्रति-जॉब फ़ॉलबैक ओवरराइड हटाता है। `cron edit <job-id> --clear-model` प्रति-जॉब मॉडल ओवरराइड हटाता है, ताकि जॉब सामान्य Cron मॉडल-चयन प्राथमिकता का अनुसरण करे (यदि मौजूद हो तो संग्रहीत Cron-सत्र ओवरराइड, अन्यथा एजेंट/डिफ़ॉल्ट मॉडल); इसे `--model` के साथ संयोजित नहीं किया जा सकता। `cron add|edit --thinking <level>` प्रति-जॉब विचार ओवरराइड सेट करता है; `cron edit <job-id> --clear-thinking` इसे हटाता है, ताकि जॉब सामान्य Cron विचार प्राथमिकता का अनुसरण करे, और इसे `--thinking` के साथ संयोजित नहीं किया जा सकता।

<Warning>
यदि मॉडल अनुमत नहीं है या उसका समाधान नहीं किया जा सकता, तो Cron जॉब के एजेंट या डिफ़ॉल्ट मॉडल चयन पर फ़ॉलबैक करने के बजाय स्पष्ट सत्यापन त्रुटि के साथ रन विफल करता है।
</Warning>

Cron `--model` एक **जॉब प्राथमिक मॉडल** है, चैट-सत्र `/model` ओवरराइड नहीं। इसका अर्थ है:

- चुना गया जॉब मॉडल विफल होने पर भी कॉन्फ़िगर किए गए मॉडल फ़ॉलबैक लागू होते हैं।
- प्रति-जॉब पेलोड `fallbacks` मौजूद होने पर कॉन्फ़िगर की गई फ़ॉलबैक सूची को प्रतिस्थापित करता है।
- रिक्त प्रति-जॉब फ़ॉलबैक सूची (जॉब पेलोड/API में `--fallbacks ""` या `fallbacks: []`) Cron रन को सख्त बनाती है।
- जब किसी जॉब में `--model` हो लेकिन कोई फ़ॉलबैक सूची कॉन्फ़िगर न हो, तो OpenClaw स्पष्ट रिक्त फ़ॉलबैक ओवरराइड देता है, ताकि एजेंट प्राथमिक मॉडल को छिपे हुए पुनःप्रयास लक्ष्य के रूप में न जोड़ा जाए।
- स्थानीय-प्रदाता प्रीफ़्लाइट जाँच, किसी Cron रन को `skipped` चिह्नित करने से पहले कॉन्फ़िगर किए गए फ़ॉलबैक की जाँच करती है।

`openclaw doctor` उन जॉब की रिपोर्ट करता है जिनमें पहले से `payload.model` सेट है, जिसमें प्रदाता नेमस्पेस की संख्याएँ और `agents.defaults.model` के विरुद्ध बेमेल शामिल हैं। जब लाइव चैट और शेड्यूल किए गए जॉब के बीच प्रमाणीकरण, प्रदाता या बिलिंग व्यवहार अलग दिखाई दे, तब इस जाँच का उपयोग करें।

### पृथक Cron मॉडल प्राथमिकता

पृथक Cron सक्रिय मॉडल को इस क्रम में समाधान करता है:

1. Gmail-हुक ओवरराइड।
2. प्रति-जॉब `--model`।
3. संग्रहीत Cron-सत्र मॉडल ओवरराइड (जब उपयोगकर्ता ने कोई चुना हो)।
4. एजेंट या डिफ़ॉल्ट मॉडल चयन।

### तेज़ मोड

अलग किया गया Cron तेज़ मोड, समाधान किए गए लाइव मॉडल चयन का अनुसरण करता है। मॉडल कॉन्फ़िगरेशन `params.fastMode` डिफ़ॉल्ट रूप से लागू होता है, लेकिन संग्रहीत सत्र का `fastMode` ओवरराइड अब भी कॉन्फ़िगरेशन पर प्राथमिकता पाता है। जब समाधान किया गया मोड `auto` होता है, तब कटऑफ़ चयनित मॉडल के `params.fastAutoOnSeconds` मान का उपयोग करता है, जिसका डिफ़ॉल्ट 60 सेकंड है।

### लाइव मॉडल स्विच के पुनः प्रयास

यदि कोई अलग किया गया रन `LiveSessionModelSwitchError` थ्रो करता है, तो Cron पुनः प्रयास करने से पहले सक्रिय रन के लिए स्विच किए गए प्रोवाइडर और मॉडल (और मौजूद होने पर स्विच किया गया प्रमाणीकरण प्रोफ़ाइल ओवरराइड) को सहेजता है। बाहरी पुनः प्रयास लूप प्रारंभिक प्रयास के बाद दो स्विच पुनः प्रयासों तक सीमित रहता है और फिर हमेशा लूप करते रहने के बजाय निरस्त हो जाता है।

## रन आउटपुट और अस्वीकृतियाँ

### पुराने अभिस्वीकृति संदेशों का दमन

अलग किए गए Cron टर्न केवल पुराने अभिस्वीकृति वाले उत्तरों को दबाते हैं। यदि पहला परिणाम केवल एक अंतरिम स्थिति अपडेट है और अंतिम उत्तर के लिए कोई वंशज सबएजेंट रन उत्तरदायी नहीं है, तो Cron डिलीवरी से पहले वास्तविक परिणाम के लिए एक बार फिर प्रॉम्प्ट करता है।

### मौन टोकन का दमन

यदि कोई अलग किया गया Cron रन केवल मौन टोकन (`NO_REPLY` या `no_reply`) लौटाता है, तो Cron प्रत्यक्ष आउटबाउंड डिलीवरी और फ़ॉलबैक कतारबद्ध सारांश पथ, दोनों को दबा देता है, इसलिए चैट में कुछ भी वापस पोस्ट नहीं किया जाता।

### संरचित अस्वीकृतियाँ

अलग किए गए Cron रन, एम्बेडेड रन से मिली संरचित निष्पादन-अस्वीकृति मेटाडेटा (`SYSTEM_RUN_DENIED` या `INVALID_REQUEST` कोड वाली घातक exec-tool त्रुटियाँ) को आधिकारिक अस्वीकृति संकेत के रूप में उपयोग करते हैं। वे नेस्ट की गई ऐसी संरचित त्रुटि के चारों ओर मौजूद node-host `UNAVAILABLE` रैपर का भी सम्मान करते हैं, जिसमें इनमें से कोई एक कोड हो।

जब तक एम्बेडेड रन संरचित अस्वीकृति मेटाडेटा भी उपलब्ध नहीं कराता, Cron अंतिम-आउटपुट गद्य या अनुमोदन-जैसे अस्वीकार वाक्यांशों को अस्वीकृति के रूप में वर्गीकृत नहीं करता, इसलिए सामान्य सहायक टेक्स्ट को अवरुद्ध कमांड नहीं माना जाता।

`cron list` और रन इतिहास अवरुद्ध कमांड को `ok` के रूप में रिपोर्ट करने के बजाय अस्वीकृति का कारण दिखाते हैं।

## अवधारण

अवधारण व्यवहार:

- `cron.sessionRetention` (डिफ़ॉल्ट `24h`, या अक्षम करने के लिए `false`) पूर्ण हो चुके अलग किए गए रन सत्रों को हटाता है।
- रन इतिहास प्रत्येक Cron जॉब के लिए नवीनतम 2000 टर्मिनल पंक्तियाँ रखता है। खोई हुई पंक्तियों के लिए मानक 24-घंटे की खोए-टास्क सफ़ाई अवधि बनी रहती है।

## पुराने जॉब माइग्रेट करना

<Note>
यदि आपके पास वर्तमान डिलीवरी और स्टोर प्रारूप से पहले के Cron जॉब हैं, तो `openclaw doctor --fix` चलाएँ। Doctor पुराने Cron फ़ील्ड (`jobId`, `schedule.cron`, पुराने `threadId` सहित शीर्ष-स्तरीय डिलीवरी फ़ील्ड, पेलोड `provider` डिलीवरी उपनाम) को सामान्यीकृत करता है और उस कॉन्फ़िगरेशन कुंजी को हटाने से पहले `notify: true` Webhook फ़ॉलबैक जॉब को सेवानिवृत्त कच्चे `cron.webhook` मान से स्पष्ट Webhook डिलीवरी में माइग्रेट करता है। जो जॉब पहले से किसी चैट में घोषणा करते हैं, वे उस डिलीवरी को बनाए रखते हैं और उन्हें पूर्णता Webhook गंतव्य मिलता है। किसी पुराने Webhook के बिना, माइग्रेशन लक्ष्य रहित जॉब से निष्क्रिय शीर्ष-स्तरीय `notify` मार्कर हटा दिया जाता है (मौजूदा डिलीवरी बिना बदलाव के सुरक्षित रहती है), इसलिए `doctor --fix` अब उनके बारे में बार-बार चेतावनी नहीं देता।
</Note>

## सामान्य संपादन

संदेश बदले बिना डिलीवरी सेटिंग अपडेट करें:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

अलग किए गए जॉब के लिए डिलीवरी अक्षम करें:

```bash
openclaw cron edit <job-id> --no-deliver
```

अलग किए गए जॉब के लिए हल्का बूटस्ट्रैप संदर्भ सक्षम करें:

```bash
openclaw cron edit <job-id> --light-context
```

किसी विशिष्ट चैनल पर घोषणा करें:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

Telegram फ़ोरम विषय पर घोषणा करें:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

हल्के बूटस्ट्रैप संदर्भ वाला अलग किया गया जॉब बनाएँ:

```bash
openclaw cron create "0 7 * * *" \
  "रातभर के अपडेट का सारांश दें।" \
  --name "हल्का सुबह का संक्षिप्त विवरण" \
  --session isolated \
  --light-context \
  --no-deliver
```

`--light-context` केवल अलग किए गए एजेंट-टर्न जॉब पर लागू होता है। Cron रन के लिए, हल्का मोड पूर्ण वर्कस्पेस बूटस्ट्रैप सेट इंजेक्ट करने के बजाय बूटस्ट्रैप संदर्भ को खाली रखता है।

सटीक argv, cwd, env, stdin और आउटपुट सीमाओं वाला कमांड जॉब बनाएँ:

```bash
openclaw cron create "*/30 * * * *" \
  --name "स्थिति निर्यात" \
  --command-argv '["node","scripts/export-position.mjs"]' \
  --command-cwd "/srv/app" \
  --command-env "NODE_ENV=production" \
  --command-input '{"mode":"summary"}' \
  --timeout-seconds 120 \
  --no-output-timeout-seconds 30 \
  --output-max-bytes 65536 \
  --webhook "https://example.invalid/openclaw/cron"
```

## सामान्य व्यवस्थापक कमांड

मैन्युअल रन और निरीक्षण:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron get <job-id>
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron run <job-id> --wait --wait-timeout 10m
openclaw cron run <job-id> --wait --wait-timeout 10m --poll-interval 2s
openclaw cron runs --id <job-id> --limit 50
openclaw cron runs --id <job-id> --run-id <run-id>
```

`openclaw cron list` डिफ़ॉल्ट रूप से सभी मेल खाने वाले जॉब दिखाता है। केवल वे जॉब दिखाने के लिए `--agent <id>` पास करें जिनकी प्रभावी सामान्यीकृत एजेंट आईडी मेल खाती है; बिना संग्रहीत एजेंट आईडी वाले जॉब को कॉन्फ़िगर किया गया डिफ़ॉल्ट एजेंट माना जाता है।

`openclaw cron get <job-id>` संग्रहीत जॉब JSON को सीधे लौटाता है। जब डिलीवरी-रूट पूर्वावलोकन सहित मानव-पठनीय दृश्य चाहिए, तब `cron show <job-id>` का उपयोग करें।

`cron list --json` और `cron show <job-id> --json` प्रत्येक जॉब में शीर्ष-स्तरीय `status` फ़ील्ड शामिल करते हैं, जिसकी गणना `enabled`, `state.runningAtMs` और `state.lastRunStatus` से की जाती है। मान: `disabled`, `running`, `ok`, `error`, `skipped` या `idle`। JSON स्थिति मानक और अलंकरण-रहित रहती है, ताकि बाहरी टूलिंग जॉब स्थिति को दोबारा निकाले बिना पढ़ सके; मानव-पठनीय आउटपुट दोहराई गई `error` स्थितियों को विफलता संख्या से सजा सकता है।

`cron runs` प्रविष्टियों में इच्छित Cron लक्ष्य, समाधान किया गया लक्ष्य, संदेश-टूल प्रेषण, फ़ॉलबैक उपयोग और डिलीवर की गई स्थिति सहित डिलीवरी निदान शामिल होते हैं।

एजेंट और सत्र को पुनः लक्षित करना:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

एजेंट-टर्न जॉब में `--agent` छोड़े जाने पर `openclaw cron add` चेतावनी देता है और डिफ़ॉल्ट एजेंट (`main`) का उपयोग करता है। किसी विशिष्ट एजेंट को स्थिर करने के लिए निर्माण के समय `--agent <id>` पास करें।

डिलीवरी समायोजन:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --webhook "https://example.invalid/openclaw/cron"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
