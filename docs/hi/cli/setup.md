# `openclaw setup`

`openclaw setup` सिस्टम एजेंट का प्रवेश बिंदु है। कॉन्फ़िगर किए गए सिस्टम पर, केवल
`openclaw setup` एक इंटरैक्टिव OpenClaw चैट खोलता है। नए सिस्टम पर, यह
निर्देशित ऑनबोर्डिंग शुरू करता है। एक अनुरोध के लिए `-m`/`--message` का उपयोग करें या
विज़ार्ड के बिना कॉन्फ़िगरेशन/वर्कस्पेस फ़ोल्डर आरंभ करने के लिए `--baseline` का उपयोग करें।

रूटिंग क्रम:

1. कोई भी ऑनबोर्डिंग विकल्प (`--wizard`, `--baseline`, वर्कस्पेस, रीसेट,
   नॉन-इंटरैक्टिव, प्रवाह, मोड, Gateway, डेमन, स्किप, इंपोर्ट, रिमोट या प्रमाणीकरण
   विकल्प) ऑनबोर्डिंग को ठीक उसी तरह चलाता है जैसे `openclaw onboard` चलाता है।
2. `-m`/`--message` या `--yes` सिस्टम एजेंट चलाता है।
3. कोई रूटिंग विकल्प न होने पर, कॉन्फ़िगर किया गया इंटरैक्टिव सिस्टम OpenClaw खोलता है। नया
   सिस्टम ऑनबोर्डिंग चलाता है। कॉन्फ़िगर किए गए सिस्टम पर, `--json` TTY के बिना भी
   सिस्टम अवलोकन प्रिंट करता है; ऑनबोर्डिंग विकल्प ऑनबोर्डिंग का
   JSON सारांश बनाए रखता है।

निर्देशित मोड में, `--workspace <dir>` OpenClaw को प्रस्तावित वर्कस्पेस है;
यह केवल आपके उस प्रस्ताव को स्वीकृति देने के बाद स्थायी किया जाता है। बेसलाइन, क्लासिक और
नॉन-इंटरैक्टिव सेटअप दिए गए वर्कस्पेस को अपने सामान्य प्रवाह के माध्यम से स्थायी करते हैं।

निर्देशित इन्फ़रेंस पहचान macOS या Linux पर Gateway होस्ट में चलती है। CLI
और macOS ऐप, Gateway के स्वामित्व वाले उसी डिटेक्टर को कॉल करते हैं, जो कॉन्फ़िगर किए गए
मॉडल, समर्थित CLI लॉगिन, API-कुंजी एनवायरनमेंट वेरिएबल और पहले से
इंस्टॉल किए गए Ollama या LM Studio मॉडल की जाँच करता है। इस स्वचालित
प्रक्रिया में स्थानीय मॉडल कभी डाउनलोड नहीं किए जाते। पहचाने गए स्थानीय रनटाइम का CLI और API-कुंजी
उम्मीदवारों के बाद स्वचालित परीक्षण किया जाता है; जब कई स्थानीय मॉडल उपलब्ध होते हैं, तो OpenClaw
सबसे सक्षम टूल-कॉलिंग इंस्ट्रक्ट परिवार को प्राथमिकता देता है। चुने गए उम्मीदवार को वास्तविक
कम्प्लीशन का उत्तर देना आवश्यक है, तभी उसका प्रोवाइडर और मॉडल कॉन्फ़िगरेशन सहेजा जाता है।
इंस्टॉल किए गए Gemini, Antigravity, Pi और OpenCode CLI की भी रिपोर्ट दी जाती है, जब
वे निर्देशित सेटअप के लिए पुनः उपयोग योग्य इन्फ़रेंस रूट के रूप में काम नहीं कर सकते।

`setup` वही ऑनबोर्डिंग फ़्लैग स्वीकार करता है जो `openclaw onboard` स्वीकार करता है, जिनमें
प्रमाणीकरण (`--auth-choice`, `--token`, प्रोवाइडर कुंजी फ़्लैग), Gateway
(`--gateway-port`, `--gateway-bind`, `--gateway-auth`, `--install-daemon`),
Tailscale (`--tailscale`), रीसेट (`--reset`, `--reset-scope`), प्रवाह
(`--flow quickstart|advanced|manual|import`) और स्किप फ़्लैग
(`--skip-channels`, `--skip-skills`, `--skip-bootstrap`, `--skip-search`,
`--skip-health`, `--skip-ui`, `--skip-hooks`) शामिल हैं। `openclaw onboard --tui` वाला ही
टर्मिनल हैच उपयोग करने के लिए `--tui` पास करें। संपूर्ण फ़्लैग संदर्भ और
नॉन-इंटरैक्टिव उदाहरणों के लिए [ऑनबोर्ड](/hi/cli/onboard) और
[CLI स्वचालन](/hi/start/wizard-cli-automation) देखें। उसी इन्फ़रेंस-गेटेड
OpenClaw सहायक के लिए `openclaw onboard --modern` संगतता
प्रवेश के रूप में बना हुआ है।

<Note>
`openclaw setup` परिवर्तनीय कॉन्फ़िगरेशन इंस्टॉलेशन के लिए है। Nix मोड (`OPENCLAW_NIX_MODE=1`) में OpenClaw सेटअप संबंधी लेखन अस्वीकार करता है, क्योंकि कॉन्फ़िगरेशन फ़ाइल Nix द्वारा प्रबंधित होती है। प्रथम-पक्ष [nix-openclaw त्वरित प्रारंभ](https://github.com/openclaw/nix-openclaw#quick-start) या किसी अन्य Nix पैकेज के समतुल्य स्रोत कॉन्फ़िगरेशन का उपयोग करें।
</Note>

## विकल्प

| फ़्लैग                       | विवरण                                                                                           |
| -------------------------- | ----------------------------------------------------------------------------------------------------- |
| `-m, --message <text>`     | एक OpenClaw अनुरोध चलाएँ।                                                                             |
| `--yes`                    | एक `--message` अनुरोध के लिए स्थायी कॉन्फ़िगरेशन लेखन स्वीकृत करें।                                         |
| `--workspace <dir>`        | निर्देशित मोड में वर्कस्पेस प्रस्ताव; बेसलाइन, क्लासिक और नॉन-इंटरैक्टिव सेटअप द्वारा सीधे स्थायी किया जाता है। |
| `--baseline`               | ऑनबोर्डिंग के बिना बेसलाइन कॉन्फ़िगरेशन/वर्कस्पेस/सेशन फ़ोल्डर बनाएँ।                                  |
| `--wizard`                 | इंटरैक्टिव ऑनबोर्डिंग को बाध्य करें।                                                                         |
| `--tui`                    | ब्राउज़र हैंडऑफ़ के बजाय टर्मिनल हैच का उपयोग करें।                                                |
| `--non-interactive`        | प्रॉम्प्ट के बिना ऑनबोर्डिंग चलाएँ।                                                                       |
| `--accept-risk`            | पूर्ण-सिस्टम एजेंट पहुँच जोखिम स्वीकार करें; `--non-interactive` के साथ आवश्यक।                         |
| `--mode <mode>`            | ऑनबोर्डिंग मोड: `local` या `remote`।                                                                 |
| `--flow <flow>`            | ऑनबोर्ड प्रवाह: `quickstart`, `advanced`, `manual` या `import`।                                        |
| `--reset`                  | ऑनबोर्डिंग से पहले कॉन्फ़िगरेशन + क्रेडेंशियल + सेशन रीसेट करें (वर्कस्पेस केवल `--reset-scope full` के साथ)।   |
| `--reset-scope <scope>`    | रीसेट का दायरा: `config`, `config+creds+sessions` या `full`।                                            |
| `--import-from <provider>` | ऑनबोर्डिंग के दौरान चलाया जाने वाला माइग्रेशन प्रोवाइडर।                                                          |
| `--import-source <path>`   | `--import-from` के लिए स्रोत एजेंट होम।                                                                |
| `--import-secrets`         | ऑनबोर्डिंग माइग्रेशन के दौरान समर्थित सीक्रेट इंपोर्ट करें।                                                 |
| `--remote-url <url>`       | रिमोट Gateway WebSocket URL।                                                                         |
| `--remote-token <token>`   | रिमोट Gateway टोकन (वैकल्पिक)।                                                                      |
| `--json`                   | कॉन्फ़िगर किया गया सिस्टम: OpenClaw अवलोकन। ऑनबोर्डिंग रूट: ऑनबोर्डिंग सारांश।                           |

`--classic` और `--non-interactive` परस्पर अनन्य हैं: क्लासिक
प्रॉम्प्ट वाला विज़ार्ड खोलता है, जबकि नॉन-इंटरैक्टिव सेटअप स्वचालन पथ का उपयोग करता है।
इंटरैक्टिव ऑनबोर्डिंग में, `--remote-url` और `--remote-token`
रिमोट Gateway चरण को पहले से भरते हैं और उस रन के लिए संग्रहित रिमोट मानों पर प्राथमिकता लेते हैं।
URL बदलने पर संग्रहित क्रेडेंशियल का पुनः उपयोग नहीं होता, जब तक कि आप टोकन भी पास न करें।
टोकन मास्क किया हुआ रहता है और विज़ार्ड द्वारा चुने गए प्लेनटेक्स्ट या SecretRef
स्टोरेज मोड का उपयोग करता है।

### बेसलाइन मोड

`openclaw setup --baseline` पुराने केवल-बेसलाइन व्यवहार को बनाए रखता है: यह
कॉन्फ़िगरेशन, वर्कस्पेस और सेशन डायरेक्टरी बनाता है, फिर ऑनबोर्डिंग चलाए बिना
बाहर निकल जाता है। यह `--workspace` और हानिरहित आउटपुट नियंत्रण स्वीकार करता है, लेकिन
स्पष्ट ऑनबोर्डिंग, Gateway, प्रमाणीकरण, रीसेट या डेमन विकल्पों को चुपचाप
अनदेखा करने के बजाय अस्वीकार करता है।

## उदाहरण

```bash
openclaw setup
openclaw setup -m "status"
openclaw setup -m "restart gateway" --yes
openclaw setup --json
openclaw setup --wizard
openclaw setup --baseline
openclaw setup --workspace ~/.openclaw/workspace
openclaw setup --import-from hermes --import-source ~/.hermes
openclaw setup --non-interactive --accept-risk --mode remote --remote-url wss://gateway-host:18789 --remote-token <token>
```

## टिप्पणियाँ

- बेसलाइन सेटअप के बाद, संपूर्ण निर्देशित प्रक्रिया के लिए `openclaw onboard`, लक्षित परिवर्तनों के लिए `openclaw configure` या चैनल खाते जोड़ने के लिए `openclaw channels add` चलाएँ।
- यदि Hermes की स्थिति मिलती है, तो इंटरैक्टिव ऑनबोर्डिंग स्वचालित रूप से माइग्रेशन का प्रस्ताव दे सकती है। इंपोर्ट ऑनबोर्डिंग के लिए नया सेटअप आवश्यक है; ऑनबोर्डिंग के बाहर ड्राई-रन योजनाओं, बैकअप और ओवरराइट मोड के लिए [माइग्रेट](/hi/cli/migrate) का उपयोग करें।
