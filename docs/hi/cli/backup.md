# `openclaw backup`

OpenClaw की स्थिति, कॉन्फ़िगरेशन, प्रमाणीकरण प्रोफ़ाइलों, चैनल/प्रदाता क्रेडेंशियल्स, सत्रों और वैकल्पिक रूप से कार्यस्थानों के लिए एक स्थानीय बैकअप आर्काइव बनाएँ।

```bash
openclaw backup create
openclaw backup create --output ~/Backups
openclaw backup create --dry-run --json
openclaw backup create --verify
openclaw backup create --no-include-workspace
openclaw backup create --only-config
openclaw backup verify ./2026-03-09T08-00-00.000+08-00-openclaw-backup.tar.gz
openclaw backup sqlite create --global --repository ~/Backups/openclaw-sqlite
openclaw backup sqlite create --agent main --repository ~/Backups/openclaw-sqlite
openclaw backup sqlite list --repository ~/Backups/openclaw-sqlite
openclaw backup sqlite verify ~/Backups/openclaw-sqlite/<snapshot-id>
openclaw backup sqlite verify ~/Backups/openclaw-sqlite/<snapshot-id> --scratch ~/Private/openclaw-scratch
openclaw backup sqlite restore ~/Backups/openclaw-sqlite/<snapshot-id> --target ./restored/openclaw.sqlite
```

## टिप्पणियाँ

- आर्काइव में समाधान किए गए स्रोत पथों और आर्काइव लेआउट सहित एक `manifest.json` अंतर्निहित होता है।
- डिफ़ॉल्ट आउटपुट वर्तमान कार्यशील डायरेक्टरी में समय-मुद्रित `.tar.gz` आर्काइव होता है। समय-मुद्रित फ़ाइल नाम आपकी मशीन के स्थानीय समय क्षेत्र का उपयोग करते हैं और उनमें UTC ऑफ़सेट शामिल होता है। यदि वर्तमान कार्यशील डायरेक्टरी बैकअप किए जा रहे स्रोत ट्री के भीतर है, तो OpenClaw डिफ़ॉल्ट आर्काइव स्थान के लिए आपकी होम डायरेक्टरी का उपयोग करता है।
- मौजूदा आर्काइव फ़ाइलें कभी अधिलेखित नहीं की जातीं। स्वयं को शामिल होने से बचाने के लिए स्रोत स्थिति/कार्यस्थान ट्री के भीतर के आउटपुट पथ अस्वीकार किए जाते हैं।
- `openclaw backup verify <archive>` जाँचता है कि आर्काइव में ठीक एक रूट मैनिफ़ेस्ट है, ट्रैवर्सल-शैली के आर्काइव पथों और SQLite साइडकारों को अस्वीकार करता है, पुष्टि करता है कि मैनिफ़ेस्ट में घोषित प्रत्येक पेलोड मौजूद है, प्रत्येक SQLite स्नैपशॉट की फ़ाइल संरचना सत्यापित करता है और मानक OpenClaw डेटाबेस पर पूर्ण अखंडता तथा भूमिका जाँच चलाता है। समर्पित Plugin स्कीमा अपारदर्शी रहते हैं, क्योंकि उन्हें स्वामी द्वारा परिभाषित SQLite क्षमताओं की आवश्यकता हो सकती है। `openclaw backup create --verify` आर्काइव लिखने के तुरंत बाद यह सत्यापन चलाता है।
- `openclaw backup create --only-config` केवल सक्रिय JSON कॉन्फ़िगरेशन फ़ाइल का बैकअप लेता है।

## SQLite स्नैपशॉट

जब व्यापक स्थिति आर्काइव के बजाय किसी एक OpenClaw-स्वामित्व वाले SQLite डेटाबेस के लिए पोर्टेबल आर्टिफ़ैक्ट चाहिए, तब `openclaw backup sqlite` का उपयोग करें।

स्नैपशॉट निर्माण ठीक एक नामित स्रोत स्वीकार करता है:

| कमांड                                                         | डेटाबेस               |
| --------------------------------------------------------------- | ---------------------- |
| `openclaw backup sqlite create --global --repository <dir>`     | साझा OpenClaw स्थिति  |
| `openclaw backup sqlite create --agent <id> --repository <dir>` | प्रति-एजेंट एक डेटाबेस |

रिपॉज़िटरी में प्रत्येक कमिट किए गए स्नैपशॉट के लिए एक डायरेक्टरी होती है। प्रत्येक स्नैपशॉट डायरेक्टरी में ठीक ये होते हैं:

- `manifest.json`
- `database.sqlite`

स्नैपशॉट निर्माण लाइव डेटाबेस को पढ़ने से पहले सत्यापित करता है, कमिट की गई WAL स्थिति को एक संक्षिप्त डेटाबेस में कैप्चर करने के लिए SQLite `VACUUM INTO` का उपयोग करता है, जनरेट किए गए डेटाबेस को दोबारा सत्यापित करता है और मौजूदा पथों को अधिलेखित किए बिना पूर्ण डायरेक्टरी प्रकाशित करता है। वैश्विक स्नैपशॉट क्षणिक डिलीवरी कतार पंक्तियाँ हटाते हैं और फिर से संक्षिप्त करते हैं, ताकि हटाए गए कतार पेलोड खाली पृष्ठों में बने न रहें।

लाइव `.sqlite`, `-wal`, `-shm` या `-journal` फ़ाइलों को पोर्टेबिलिटी आर्टिफ़ैक्ट के रूप में कॉपी न करें। केवल पूर्ण स्नैपशॉट डायरेक्टरियाँ कॉपी करें।

SQLite स्नैपशॉट में प्रमाणीकरण प्रोफ़ाइलें, सत्र स्थिति, Plugin स्थिति और अन्य संवेदनशील रिकॉर्ड हो सकते हैं। रिपॉज़िटरियों को लाइव OpenClaw स्थिति डायरेक्टरी के समान अनुमतियों, एन्क्रिप्शन, प्रतिधारण नीति और गंतव्य प्रतिबंधों से सुरक्षित रखें।

### सत्यापन और पुनर्स्थापन

```bash
openclaw backup sqlite verify <snapshot-directory>
openclaw backup sqlite restore <snapshot-directory> --target <new-database-path>
```

सत्यापन सख़्त मैनिफ़ेस्ट संरचना, आर्टिफ़ैक्ट आकार और SHA-256, SQLite अखंडता, फ़ॉरेन की, स्कीमा संस्करण, डेटाबेस भूमिका और स्वामी तथा OpenClaw-स्वामित्व वाली इंडेक्स परिभाषाओं की जाँच करता है।

सत्यापन एक निजी, सामग्री-पिन की गई प्रति को सत्यापित करता है, ताकि पथनाम रेस SQLite द्वारा जाँचे जाने वाले बाइट्स को बदल न सकें। डिफ़ॉल्ट रूप से, वह अस्थायी प्रति स्नैपशॉट रिपॉज़िटरी के पास बनाई जाती है और कमांड लौटने से पहले हटा दी जाती है। स्टेजिंग रूट और उसकी पूर्वज शृंखला को अन्य उपयोगकर्ताओं द्वारा उसके प्रतिस्थापन को रोकना चाहिए। POSIX रूट वर्तमान उपयोगकर्ता के स्वामित्व में होने चाहिए और समूह/विश्व द्वारा लिखने योग्य नहीं होने चाहिए; `/tmp` जैसे स्टिकी पूर्वज उपयोगकर्ता-स्वामित्व वाले चाइल्ड के लिए स्वीकार किए जाते हैं। स्टेजिंग को उजागर करने या प्रतिस्थापन योग्य बनाने वाले macOS ACL अनुदान अस्वीकार किए जाते हैं। Windows रूट और पूर्वज वर्तमान उपयोगकर्ता या किसी विश्वसनीय OS प्रिंसिपल के स्वामित्व में होने चाहिए और उनके ACL अविश्वसनीय स्टेजिंग पहुँच को अस्वीकार करने वाले होने चाहिए। केवल-पढ़ने योग्य माउंट या नेटवर्क शेयर के लिए, समकक्ष एन्क्रिप्शन और गंतव्य नियंत्रण वाले स्टोरेज पर `--scratch <existing-private-directory>` प्रदान करें।

स्नैपशॉट निर्माण डेटाबेस बाइट्स को स्टेज या प्रकाशित करने से पहले रिपॉज़िटरी पर समान स्वामी, ACL, पूर्वज और पथ-पहचान जाँच लागू करता है।

पुनर्स्थापन सत्यापन दोहराता है और केवल नए लक्ष्य में लिखता है। यह मौजूदा लक्ष्य, `-wal`, `-shm` या `-journal` साइडकार को अस्वीकार करता है और लाइव OpenClaw डेटाबेस का उसी स्थान पर प्रतिस्थापन कभी नहीं करता। लक्ष्य की पैरेंट डायरेक्टरी पर सत्यापन स्क्रैच जैसी ही पथ-सुरक्षा आवश्यकताएँ लागू होती हैं। पुनर्स्थापित डेटाबेस को सक्रिय करना एक स्पष्ट ऑफ़लाइन ऑपरेटर चरण बना रहता है।

स्नैपशॉट रिपॉज़िटरियाँ स्थानीय डायरेक्टरियाँ हैं। शेड्यूलिंग, अपलोड, प्रतिधारण, वृद्धिशील WAL बंडल, फ़ेलओवर और बूट पर पुनर्स्थापन व्यवहार जानबूझकर इस कमांड के दायरे से बाहर हैं।

## किसका बैकअप लिया जाता है

`openclaw backup create` आपके स्थानीय OpenClaw इंस्टॉलेशन से स्रोतों की योजना बनाता है:

- स्थिति डायरेक्टरी (आमतौर पर `~/.openclaw`)
- सक्रिय कॉन्फ़िगरेशन फ़ाइल पथ
- समाधान की गई `credentials/` डायरेक्टरी, जब वह स्थिति डायरेक्टरी के बाहर मौजूद हो
- वर्तमान कॉन्फ़िगरेशन से खोजी गई कार्यस्थान डायरेक्टरियाँ, जब तक कि आप `--no-include-workspace` प्रदान न करें

प्रमाणीकरण प्रोफ़ाइलें और अन्य प्रति-एजेंट रनटाइम स्थिति, स्थिति डायरेक्टरी (`agents/<agentId>/agent/openclaw-agent.sqlite`) के अंतर्गत SQLite में रहती हैं, इसलिए वे स्वचालित रूप से स्थिति बैकअप प्रविष्टि में शामिल होती हैं।

`--only-config` स्थिति, क्रेडेंशियल्स डायरेक्टरी और कार्यस्थान खोज को छोड़ देता है तथा केवल सक्रिय कॉन्फ़िगरेशन फ़ाइल पथ को आर्काइव करता है।

OpenClaw आर्काइव बनाने से पहले पथों को मानकीकृत करता है: यदि कॉन्फ़िगरेशन, क्रेडेंशियल्स डायरेक्टरी या कोई कार्यस्थान पहले से स्थिति डायरेक्टरी के भीतर है, तो उन्हें अलग शीर्ष-स्तरीय बैकअप स्रोतों के रूप में दोहराया नहीं जाता। अनुपलब्ध पथ छोड़ दिए जाते हैं।

आर्काइव निर्माण के दौरान, OpenClaw `tar` द्वारा पढ़े जाने से पहले ज्ञात लाइव-परिवर्तन पथों को बाहर कर देता है। इससे फ़ाइल के दर्ज आकार और समवर्ती लेखन के बीच रेस से बचा जाता है। फ़िल्टर प्रत्येक बैकअप की गई स्थिति डायरेक्टरी के अंतर्गत ये स्थिति-सापेक्ष नियम लागू करता है:

| स्थिति-सापेक्ष दायरा                         | छोड़े गए फ़ाइल प्रत्यय         |
| -------------------------------------------- | ----------------------------- |
| `sessions/**`                                | `.jsonl`, `.log`              |
| `agents/<agentId>/sessions/**`               | `.jsonl`, `.log`              |
| `cron/runs/**`                               | `.jsonl`, `.log`              |
| `logs/**`                                    | `.jsonl`, `.log`              |
| `delivery-queue/**`                          | `.json`, `.delivered`, `.tmp` |
| `session-delivery-queue/**`                  | `.json`, `.delivered`, `.tmp` |
| बैकअप की गई स्थिति डायरेक्टरी के अंतर्गत कोई भी पथ | `.sock`, `.pid`, `.tmp`       |

ये नियम स्थिति डायरेक्टरी के बाहर की कार्यस्थान फ़ाइलों को फ़िल्टर नहीं करते। वे तालिका से मेल खाने वाली पूर्ण ट्रांसक्रिप्ट और लॉग फ़ाइलें भी छोड़ देते हैं, इसलिए आवश्यकता होने पर उन रिकॉर्डों को अलग से बनाए रखें। JSON परिणाम का `skippedVolatileCount` बताता है कि कितनी फ़ाइलें जानबूझकर छोड़ी गईं।

स्थिति डायरेक्टरी के अंतर्गत SQLite डेटाबेस को `VACUUM INTO` से संक्षिप्त किया जाता है, ताकि हटाए गए पृष्ठों के अवशेष आर्काइव में न जाएँ और लाइव WAL/SHM फ़ाइलें कॉपी न हों। अनुपलब्ध स्वामी-परिभाषित SQLite क्षमताओं की आवश्यकता वाला Plugin-स्वामित्व वाला डेटाबेस कच्चे पृष्ठ की प्रति पर वापस जाने के बजाय बंद होकर विफल होता है। कार्यस्थान बैकअप के माध्यम से शामिल SQLite फ़ाइलें कार्यस्थान फ़ाइलों के रूप में कॉपी की जाती हैं और संक्षिप्तीकरण गारंटी में शामिल नहीं होतीं।

स्थिति डायरेक्टरी के `extensions/` ट्री के अंतर्गत इंस्टॉल किए गए Plugin स्रोत और मैनिफ़ेस्ट फ़ाइलें शामिल की जाती हैं, लेकिन उनके नेस्टेड `node_modules/` निर्भरता ट्री को पुनर्निर्माण योग्य इंस्टॉलेशन आर्टिफ़ैक्ट के रूप में छोड़ दिया जाता है। आर्काइव पुनर्स्थापित करने के बाद, यदि कोई पुनर्स्थापित Plugin अनुपलब्ध निर्भरताओं की रिपोर्ट करता है, तो `openclaw plugins update <id>` का उपयोग करें या `openclaw plugins install <spec> --force` से पुनः इंस्टॉल करें।

स्थिति डायरेक्टरी के अंतर्गत इंस्टॉलर-प्रबंधित और पुनर्निर्माण योग्य रनटाइम रूट भी छोड़ दिए जाते हैं: `dev/`, `git/`, `npm/`, लेगेसी `npm-runtime/` और `tools/`। इनमें प्रामाणिक उपयोगकर्ता स्थिति के बजाय प्रबंधित चेकआउट, पैकेज ट्री और डाउनलोड किए गए रनटाइम होते हैं; पुनर्स्थापन के बाद संबंधित रनटाइम या Plugin को पुनः इंस्टॉल या अपडेट करें। इनमें से किसी रूट के भीतर स्पष्ट रूप से कॉन्फ़िगर की गई कॉन्फ़िगरेशन फ़ाइल, क्रेडेंशियल्स डायरेक्टरी या कार्यस्थान शामिल रहता है।

## अमान्य कॉन्फ़िगरेशन व्यवहार

`openclaw backup` सामान्य कॉन्फ़िगरेशन पूर्व-जाँच को बायपास करता है, ताकि वह पुनर्प्राप्ति के दौरान भी सहायता कर सके। कार्यस्थान खोज एक मान्य कॉन्फ़िगरेशन पर निर्भर करती है, इसलिए जब कॉन्फ़िगरेशन फ़ाइल मौजूद हो लेकिन अमान्य हो और कार्यस्थान बैकअप अभी भी सक्षम हो, तब `openclaw backup create` तुरंत विफल हो जाता है।

उस स्थिति में आंशिक बैकअप के लिए `--no-include-workspace` के साथ दोबारा चलाएँ: यह कार्यस्थान खोज को पूरी तरह छोड़ते हुए स्थिति, कॉन्फ़िगरेशन और बाहरी क्रेडेंशियल्स डायरेक्टरी को दायरे में रखता है।

`--only-config` विकृत कॉन्फ़िगरेशन के साथ भी काम करता है, क्योंकि यह कार्यस्थान खोज के लिए कॉन्फ़िगरेशन को पार्स नहीं करता।

## आकार और प्रदर्शन

OpenClaw अंतर्निहित अधिकतम बैकअप आकार या प्रति-फ़ाइल आकार सीमा लागू नहीं करता। यदि कोई आर्काइव लेखन पाँच मिनट तक कोई डेटा उत्पन्न नहीं करता, तो वह अनिश्चित काल तक अटके रहने के बजाय विफल हो जाता है और अपनी आंशिक अस्थायी फ़ाइल हटा देता है। अन्य व्यावहारिक सीमाएँ इनसे आती हैं:

- अस्थायी आर्काइव लेखन और अंतिम आर्काइव के लिए उपलब्ध स्थान
- बड़े कार्यस्थान ट्री को स्कैन करने और उन्हें `.tar.gz` में संपीड़ित करने में लगने वाला समय
- `--verify` या `openclaw backup verify` से आर्काइव को दोबारा स्कैन करने में लगने वाला समय
- गंतव्य फ़ाइल सिस्टम का व्यवहार: OpenClaw बिना अधिलेखन वाले हार्ड-लिंक प्रकाशन चरण को प्राथमिकता देता है और हार्ड लिंक असमर्थित होने पर विशिष्ट कॉपी का उपयोग करता है

बड़े कार्यस्थान आमतौर पर आर्काइव आकार के मुख्य कारक होते हैं। छोटे/तेज़ बैकअप के लिए `--no-include-workspace` या सबसे छोटे आर्काइव के लिए `--only-config` का उपयोग करें।
