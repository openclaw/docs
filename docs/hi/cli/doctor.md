# `openclaw doctor`

Gateway, चैनलों, plugins, skills, मॉडल रूटिंग, स्थानीय स्थिति और कॉन्फ़िग माइग्रेशन के लिए स्वास्थ्य जाँच और त्वरित सुधार। जब भी कोई चीज़ अपेक्षा के अनुरूप काम न कर रही हो और आप चाहते हों कि एक कमांड बताए कि समस्या क्या है, इसका उपयोग करें।

जब Gateway की स्थिति निम्नीकृत SecretRef स्वामियों की रिपोर्ट करती है, तो doctor प्रत्येक कोल्ड या स्टेल स्वामी, प्रभावित कॉन्फ़िग पथ, संशोधित कारण और `openclaw secrets reload` पुनः प्रयास कमांड के साथ **सीक्रेट रनटाइम निम्नीकरण** चेतावनी दिखाता है।

जब चैनल इनग्रेस इवेंट डेड-लेटर किए जाते हैं, तो doctor प्रत्येक प्रभावित चैनल खाते का नाम बताता है और जाँच व पुनर्प्राप्ति के लिए [`openclaw channels dead-letters list`](/hi/cli/channels#inbound-dead-letters) की ओर इंगित करता है।

संबंधित:

- समस्या निवारण: [समस्या निवारण](/hi/gateway/troubleshooting)
- सुरक्षा ऑडिट: [सुरक्षा](/hi/gateway/security)

## कार्य-स्थितियाँ

Doctor की पाँच कार्य-स्थितियाँ हैं:

| कार्य-स्थिति                  | कमांड                                     | व्यवहार                                                                                       |
| ---------------------------- | ----------------------------------------- | --------------------------------------------------------------------------------------------- |
| निरीक्षण                     | `openclaw doctor`                         | मानव-केंद्रित जाँच और निर्देशित प्रॉम्प्ट।                                                    |
| सुधार                        | `openclaw doctor --fix`                   | समर्थित सुधार लागू करता है और गैर-इंटरैक्टिव सुधार सुरक्षित न होने पर प्रॉम्प्ट का उपयोग करता है। |
| लिंट                         | `openclaw doctor --lint`                  | CI, प्रीफ़्लाइट और समीक्षा गेट के लिए केवल-पठन संरचित निष्कर्ष।                               |
| साझा SQLite रखरखाव          | `openclaw doctor --state-sqlite compact`  | कैननिकल साझा स्थिति DB को स्पष्ट रूप से चेकपॉइंट, कॉम्पैक्ट और सत्यापित करता है।              |
| सत्र SQLite माइग्रेशन       | `openclaw doctor --session-sqlite <mode>` | सत्र स्थिति का निरीक्षण, आयात, सत्यापन, कॉम्पैक्शन, पुनर्प्राप्ति या पुनर्स्थापन करता है।     |

जब स्वचालन को स्थिर परिणाम चाहिए, तो `--lint` को प्राथमिकता दें। जब कोई मानव ऑपरेटर चाहता हो कि doctor कॉन्फ़िग या स्थिति को संपादित करे, तो `--fix` को प्राथमिकता दें।

## उदाहरण

```bash
openclaw doctor
openclaw doctor --lint
openclaw doctor --lint --json
openclaw doctor --lint --severity-min warning
openclaw doctor --lint --all
openclaw doctor --lint --allow-exec
openclaw doctor --deep
openclaw doctor --fix
openclaw doctor --fix --non-interactive
openclaw doctor --generate-gateway-token
openclaw doctor --post-upgrade
openclaw doctor --post-upgrade --json
openclaw doctor --state-sqlite compact
openclaw doctor --state-sqlite compact --json
openclaw doctor --session-sqlite inspect --session-sqlite-all-agents
openclaw doctor --session-sqlite dry-run --session-sqlite-agent main --json
openclaw doctor --session-sqlite import --session-sqlite-all-agents
openclaw doctor --session-sqlite validate --session-sqlite-all-agents --json
openclaw doctor --session-sqlite compact --session-sqlite-all-agents
openclaw doctor --session-sqlite recover --github-issue
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

चैनल-विशिष्ट अनुमतियों के लिए `doctor` के बजाय चैनल प्रोब का उपयोग करें:

```bash
openclaw channels capabilities --channel discord --target channel:<channel-id>
openclaw channels status --probe
```

`channels capabilities` किसी विशिष्ट चैनल लक्ष्य के लिए बॉट की प्रभावी अनुमतियों की रिपोर्ट करता है। `channels status --probe` सभी कॉन्फ़िगर किए गए चैनलों और वॉइस ऑटो-जॉइन लक्ष्यों का ऑडिट करता है।

## विकल्प

| विकल्प                          | प्रभाव                                                                                                                                                                                  |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--no-workspace-suggestions`    | वर्कस्पेस मेमोरी/खोज सुझाव अक्षम करें।                                                                                                                                                   |
| `--yes`                         | प्रॉम्प्ट किए बिना डिफ़ॉल्ट स्वीकार करें।                                                                                                                                                |
| `--repair` / `--fix`            | प्रॉम्प्ट किए बिना अनुशंसित गैर-सेवा सुधार लागू करें (`--fix` एक उपनाम है)। Gateway सेवा इंस्टॉल/पुनर्लेखन के लिए अब भी इंटरैक्टिव पुष्टि या स्पष्ट `gateway` कमांड आवश्यक हैं। |
| `--force`                       | कस्टम सेवा कॉन्फ़िग को अधिलेखित करने सहित आक्रामक सुधार लागू करें।                                                                                                                       |
| `--non-interactive`             | प्रॉम्प्ट के बिना चलाएँ; केवल सुरक्षित माइग्रेशन और गैर-सेवा सुधार।                                                                                                                      |
| `--generate-gateway-token`      | Gateway टोकन जनरेट और कॉन्फ़िगर करें।                                                                                                                                                    |
| `--allow-exec`                  | सीक्रेट सत्यापित करते समय doctor को कॉन्फ़िगर किए गए `exec` SecretRefs निष्पादित करने दें।                                                                                  |
| `--deep`                        | अतिरिक्त Gateway इंस्टॉल के लिए सिस्टम सेवाओं को स्कैन करें; हाल के Gateway सुपरवाइज़र रीस्टार्ट हैंडऑफ़ की रिपोर्ट करें।                                                               |
| `--lint`                        | आधुनिकीकृत स्वास्थ्य जाँच को केवल-पठन मोड में चलाएँ और नैदानिक निष्कर्ष प्रदर्शित करें।                                                                                                 |
| `--post-upgrade`                | अपग्रेड के बाद plugin संगतता प्रोब चलाएँ; निष्कर्ष stdout पर जाते हैं; कोई त्रुटि-स्तरीय निष्कर्ष मौजूद होने पर निकास कोड 1।                                                            |
| `--state-sqlite <mode>`         | स्पष्ट साझा स्थिति SQLite रखरखाव चलाएँ। एकमात्र मोड `compact` है।                                                                                                             |
| `--session-sqlite <mode>`       | लक्षित सत्र SQLite माइग्रेशन मोड चलाएँ: `inspect`, `dry-run`, `import`, `validate`, `compact`, `recover`, या `restore`। |
| `--session-sqlite-store <path>` | `--session-sqlite` के साथ: एक लीगेसी `sessions.json` स्टोर पथ चुनें।                                                                                                                  |
| `--session-sqlite-agent <id>`   | `--session-sqlite` के साथ: एक कॉन्फ़िगर किया गया एजेंट चुनें।                                                                                                                           |
| `--session-sqlite-all-agents`   | `--session-sqlite` के साथ: कॉन्फ़िगर किए गए और खोजे गए एजेंट स्टोर चुनें।                                                                                                                |
| `--github-issue`                | `--session-sqlite recover` के साथ: एक स्वच्छ किया हुआ openclaw/openclaw समस्या विवरण तैयार करें; `--yes` या इंटरैक्टिव पुष्टि के बाद doctor इसे `gh` से बनाता है।       |
| `--json`                        | `--lint` के साथ: JSON निष्कर्ष। `--post-upgrade` के साथ: `{ probesRun, findings }`। `--state-sqlite` या `--session-sqlite` के साथ: रखरखाव रिपोर्ट JSON के रूप में।                |
| `--severity-min <level>`        | `--lint` के साथ: `info`, `warning`, या `error` से नीचे के निष्कर्ष हटाएँ।                                                                     |
| `--all`                         | `--lint` के साथ: डिफ़ॉल्ट सेट से बाहर रखी गई ऑप्ट-इन जाँच सहित सभी पंजीकृत जाँच चलाएँ।                                                                                        |
| `--skip <id>`                   | `--lint` के साथ: एक जाँच आईडी छोड़ें। दोहराया जा सकता है।                                                                                                                     |
| `--only <id>`                   | `--lint` के साथ: केवल दी गई जाँच आईडी चलाएँ। दोहराया जा सकता है।                                                                                                              |

`--severity-min`, `--all`, `--only`, और `--skip` केवल `--lint` के साथ स्वीकार किए जाते हैं; `--json` को `--lint`, `--post-upgrade`, `--state-sqlite`, और `--session-sqlite` के साथ स्वीकार किया जाता है।

## लिंट मोड

`openclaw doctor --lint` केवल-पठन है: कोई प्रॉम्प्ट नहीं, कोई सुधार नहीं, कॉन्फ़िग/स्थिति का कोई पुनर्लेखन नहीं।

```bash
openclaw doctor --lint
openclaw doctor --lint --severity-min warning
openclaw doctor --lint --json
openclaw doctor --lint --all
openclaw doctor --lint --allow-exec
openclaw doctor --lint --only core/doctor/gateway-config --json
openclaw doctor --lint --only core/doctor/local-audio-acceleration --severity-min info
```

मानव-पठनीय आउटपुट संक्षिप्त होता है:

```text
doctor --lint: 6 जाँच चलाई गईं, 1 निष्कर्ष मिला
  [warning] core/doctor/gateway-config gateway.mode - gateway.mode सेट नहीं है; gateway का आरंभ अवरुद्ध होगा।
    सुधार: `openclaw configure` चलाएँ और Gateway मोड (local/remote) सेट करें, या `openclaw config set gateway.mode local` चलाएँ।
```

JSON आउटपुट स्क्रिप्टिंग सतह है:

```json
{
  "ok": false,
  "checksRun": 5,
  "checksSkipped": 0,
  "findings": [
    {
      "checkId": "core/doctor/gateway-config",
      "severity": "warning",
      "message": "gateway.mode सेट नहीं है; gateway का आरंभ अवरुद्ध होगा।",
      "path": "gateway.mode",
      "fixHint": "`openclaw configure` चलाएँ और Gateway मोड (local/remote) सेट करें, या `openclaw config set gateway.mode local` चलाएँ।"
    }
  ]
}
```

निकास कोड:

| कोड | अर्थ                                                           |
| ---- | ------------------------------------------------------------- |
| `0`  | चयनित गंभीरता सीमा पर या उससे ऊपर कोई निष्कर्ष नहीं।          |
| `1`  | कम-से-कम एक निष्कर्ष चयनित सीमा को पूरा करता है।              |
| `2`  | लिंट निष्कर्ष उत्पन्न होने से पहले कमांड/रनटाइम विफलता।       |

`--severity-min` यह नियंत्रित करता है कि कौन-से निष्कर्ष प्रिंट हों और निकास सीमा क्या हो: कम गंभीरता वाले `info`/`warning` निष्कर्ष मौजूद होने पर भी `openclaw doctor --lint --severity-min error` कुछ भी प्रिंट किए बिना `0` के साथ बाहर निकल सकता है।

`--all` यह नियंत्रित करता है कि गंभीरता फ़िल्टरिंग से पहले कौन-सी जाँच चुनी जाएँ। डिफ़ॉल्ट लिंट रन उन जाँचों को बाहर रखता है जो गहरी, ऐतिहासिक या सुधार योग्य लीगेसी अवशेष सामने लाने की अधिक संभावना वाली हैं; पूरी सूची के लिए `--all` का उपयोग करें। `--only <id>` सबसे सटीक चयनकर्ता है और आईडी द्वारा कोई भी पंजीकृत जाँच चला सकता है।

`core/doctor/local-audio-acceleration` स्पीच मॉडल लोड किए बिना स्वतः चयनित स्थानीय STT कमांड, सक्षम/अनुरोधित/अवलोकित बैकएंड के अलग-अलग प्रमाण और फ़ॉलबैक क्रम की रिपोर्ट करता है। यह एक सूचनात्मक निष्कर्ष प्रदर्शित करता है, इसलिए इसे दिखाने के लिए `--severity-min info` शामिल करें।

## संरचित स्वास्थ्य जाँच

आधुनिक doctor जाँच एक छोटे विभाजित अनुबंध का उपयोग करती हैं:

```ts
detect(ctx, scope?) -> HealthFinding[]
repair?(ctx, findings) -> HealthRepairResult
```

`detect()`, `doctor --lint` को संचालित करता है। `repair()` वैकल्पिक है और केवल `doctor --fix` / `doctor --repair` के अंतर्गत चलता है। जो जाँच अभी तक इस संरचना में माइग्रेट नहीं हुई हैं, वे अब भी लीगेसी doctor योगदान प्रवाह का उपयोग करती हैं।

मरम्मत संदर्भ `dryRun`/`diff` अनुरोध ले जा सकते हैं; मरम्मत परिणाम संरचित `diffs` (कॉन्फ़िगरेशन/फ़ाइल संपादन) और `effects` (सेवा, प्रक्रिया, पैकेज, स्थिति या अन्य दुष्प्रभाव) लौटा सकते हैं, ताकि रूपांतरित जाँचें म्यूटेशन योजना को `detect()` में स्थानांतरित किए बिना `doctor --fix --dry-run` की ओर विकसित हो सकें।

`repair()`, `status: "repaired" | "skipped" | "failed"` की रिपोर्ट करता है (स्थिति न होने का अर्थ `repaired` है)। जब मरम्मत `skipped` या `failed` लौटाती है, तो Doctor कारण की रिपोर्ट करता है और उस जाँच का सत्यापन छोड़ देता है। सफल मरम्मत के बाद, Doctor मरम्मत किए गए निष्कर्षों तक सीमित `detect()` को फिर से चलाता है; यदि निष्कर्ष अब भी मौजूद है, तो Doctor परिवर्तन को पूर्ण मानने के बजाय मरम्मत चेतावनी की रिपोर्ट करता है।

एक निष्कर्ष में ये शामिल होते हैं:

| फ़ील्ड             | उद्देश्य                                                |
| ----------------- | ------------------------------------------------------ |
| `checkId`         | स्किप/केवल फ़िल्टर और CI अनुमति-सूचियों के लिए स्थिर आईडी।     |
| `severity`        | `info`, `warning`, या `error`।                         |
| `message`         | मानव-पठनीय समस्या विवरण।                      |
| `path`            | उपलब्ध होने पर कॉन्फ़िगरेशन, फ़ाइल या तार्किक पथ।          |
| `line` / `column` | उपलब्ध होने पर स्रोत स्थान।                        |
| `ocPath`          | जब कोई जाँच किसी एक को इंगित कर सके, तब सटीक `oc://` पता। |
| `fixHint`         | सुझाई गई ऑपरेटर कार्रवाई या मरम्मत सारांश।           |

आधुनिकीकृत कोर Doctor जाँचें उस क्रमबद्ध Doctor योगदान से जुड़ी रहती हैं जो उनके मानवीय `doctor` / `doctor --fix` व्यवहार का स्वामी है। साझा संरचित स्वास्थ्य रजिस्ट्री विस्तार बिंदु है: बंडल और Plugin-समर्थित जाँचें कोर Doctor जाँचों के बाद तब चलती हैं, जब उनका स्वामी पैकेज उन्हें सक्रिय कमांड पथ में पंजीकृत कर देता है। `openclaw/plugin-sdk/health`, Plugin लेखकों के लिए वही अनुबंध उपलब्ध कराता है।

```bash
openclaw doctor --lint --all --skip core/doctor/session-locks
```

`--only` और `--skip` पूर्ण जाँच आईडी स्वीकार करते हैं और इन्हें दोहराया जा सकता है। यदि कोई `--only` आईडी पंजीकृत नहीं है, तो उस आईडी के लिए कोई जाँच नहीं चलती; यह पुष्टि करने के लिए कि केंद्रित गेट आपकी अपेक्षित जाँचों का चयन करता है, आउटपुट में `checksRun`/`checksSkipped` का उपयोग करें।

## अपग्रेड-पश्चात मोड

`openclaw doctor --post-upgrade`, बिल्ड या अपग्रेड के बाद श्रृंखलाबद्ध करने के लिए Plugin संगतता परीक्षण चलाता है। निष्कर्ष stdout पर जाते हैं; यदि किसी निष्कर्ष में `level: "error"` हो, तो निकास कोड 1 होता है। मशीन-पठनीय आवरण (`{ probesRun, findings }`) के लिए `--json` जोड़ें, जो CI, सामुदायिक `fork-upgrade` Skills और अन्य अपग्रेड-पश्चात स्मोक टूलिंग के लिए उपयुक्त है। यदि स्थापित Plugin इंडेक्स अनुपस्थित या विकृत है, तब भी JSON मोड `plugin.index_unavailable` त्रुटि निष्कर्ष के साथ आवरण उत्सर्जित करता है।

कंटेनर इमेज स्टार्टअप सामान्य "अपडेट करने के बाद Doctor चलाएँ"
प्रवाह का अपवाद है। जब `openclaw gateway run` किसी नए OpenClaw संस्करण पर शुरू होता है, तो यह
तैयार होने की रिपोर्ट करने से पहले सुरक्षित स्थिति और Plugin मरम्मत चलाता है। यदि मरम्मत
सुरक्षित रूप से पूरी नहीं हो सकती, तो स्टार्टअप बंद हो जाता है और आपको कंटेनर को
सामान्य रूप से पुनः आरंभ करने से पहले समान माउंट की गई स्थिति/कॉन्फ़िगरेशन पर
`openclaw doctor --fix` के साथ उसी इमेज को एक बार चलाने के लिए कहता है।

## लेगेसी स्थिति माइग्रेशन

`openclaw doctor --fix`, स्थायी फ़ाइल-से-SQLite माइग्रेशन का एकमात्र स्वामी है। यह प्रत्येक पहचाने गए स्रोत को सत्यापित करके उस पर स्वामित्व लेता है, कैनोनिकल पंक्तियाँ लिखकर सत्यापित करता है, माइग्रेशन रसीद दर्ज करता है और फिर सेवानिवृत्त स्रोत को हटा देता है। रनटाइम कोड लेज़ी इम्पोर्ट या फ़ॉलबैक रीड नहीं करता।

इसमें `<state-dir>/mcp-oauth/*.json` के अंतर्गत सेवानिवृत्त MCP OAuth फ़ाइलें शामिल हैं। मरम्मत से पहले Gateway रोकें। Doctor मान्य क्रेडेंशियल को `<state-dir>/state/openclaw.sqlite` में इम्पोर्ट करता है, दोनों स्टोर मौजूद होने पर मौजूदा कैनोनिकल SQLite सत्र को सुरक्षित रखता है, अप्रचलित स्थायी OAuth `state` मान हटाता है और अपनी रसीद का उपयोग करके दोबारा बनाई गई पुरानी फ़ाइल को लॉग-आउट किए गए क्रेडेंशियल पुनर्जीवित करने से रोकता है। सेवानिवृत्त `.lock` साइडकार विफलता पर बंद रहते हैं: यदि Doctor किसी पुराने स्वामी की रिपोर्ट करता है, तो सत्यापित करें कि कोई पुरानी OpenClaw प्रक्रिया नहीं चल रही है, उस साइडकार को हटाएँ और Doctor फिर से चलाएँ।

## साझा स्थिति SQLite Compaction

स्कीमा संस्करण निर्धारण, अखंडता जाँच और डाउनग्रेड पुनर्प्राप्ति के लिए [डेटाबेस स्कीमा](/hi/reference/database-schemas) देखें।

`openclaw doctor --state-sqlite compact`,
`<state-dir>/state/openclaw.sqlite` पर स्थित कैनोनिकल साझा स्थिति डेटाबेस के लिए स्पष्ट ऑफ़लाइन रखरखाव है।
यह कोई मनमाना डेटाबेस पथ स्वीकार नहीं करता, सामान्य Gateway संचालन द्वारा कभी लागू नहीं होता और
`openclaw doctor --fix` का भाग नहीं है। कमांड Gateway स्टार्टअप वाला ही स्थिति स्वामित्व लॉक प्राप्त करता है
और उसे सत्यापन, चेकपॉइंटिंग, `VACUUM` तथा अंतिम अखंडता जाँचों तक बनाए रखता है।
जब कोई Gateway या दूसरा SQLite रखरखाव कमांड उस लॉक का स्वामी हो, तो यह चलने से मना कर देता है।
जब `OPENCLAW_ALLOW_MULTI_GATEWAY=1` प्रति-कॉन्फ़िगरेशन Gateway सिंगलटन को छोड़ता है, तब भी स्थिति लॉक सक्रिय रहता है,
इसलिए रखरखाव द्वारा उसका पता लगाने के लिए ऑपरेटर शेल को Gateway सेवा का परिवेश इनहेरिट करने की
आवश्यकता नहीं होती।

पहले Gateway रोकें और सत्यापित बैकअप बनाएँ:

```bash
openclaw gateway stop
openclaw backup create --verify
openclaw doctor --state-sqlite compact --json
openclaw gateway start
```

कमांड:

1. कैनोनिकल साझा-स्थिति पथ पर एक नियमित फ़ाइल आवश्यक है। अनुपस्थित
   डेटाबेस की रिपोर्ट `skipped` के रूप में होती है और कमांड सफलतापूर्वक बंद हो जाता है।
2. चेकपॉइंटिंग या फ़ाइल बदलने से पहले वर्तमान समर्थित स्कीमा संस्करण और
   `schema_meta.role = "global"` को सत्यापित करता है।
3. एक गैर-व्यस्त `wal_checkpoint(TRUNCATE)` आवश्यक है। यदि चेकपॉइंट व्यस्त हो, तो बची हुई किसी भी OpenClaw
   प्रक्रिया को रोकें और पुनः प्रयास करें।
4. `auto_vacuum` को `INCREMENTAL` पर सेट करता है, पूर्ण `VACUUM` चलाता है और फिर से
   चेकपॉइंट करता है।
5. `quick_check`, `integrity_check` और `foreign_key_check` चलाता है, फिर
   डेटाबेस और SQLite साइडकार फ़ाइलों पर केवल-स्वामी अनुमतियाँ दोबारा लागू करता है।

JSON आउटपुट Compaction से पहले और बाद के डेटाबेस तथा WAL आकार, फ्रीलिस्ट पृष्ठ, पृष्ठ आकार और
`auto_vacuum` मान के साथ-साथ पुनः प्राप्त बाइट तथा `quick_check` और `integrity_check`
परिणामों की रिपोर्ट करता है। `foreign_key_check` विफलता पर बंद रहने के रूप में लागू होता है और
इसका कोई अलग सफलता फ़ील्ड नहीं है। SQLite, `auto_vacuum` की रिपोर्ट नहीं के लिए
`0`, पूर्ण के लिए `1` और वृद्धिशील के लिए `2` के रूप में करता है।

जब स्कीमा पुराना हो, चल रहे OpenClaw बिल्ड से नया हो या किसी एजेंट डेटाबेस का हो, तो
Compaction बिना म्यूटेशन के विफल हो जाता है। पुराने साझा-स्थिति स्कीमा के लिए पहले
`openclaw doctor --fix` चलाएँ। नए स्कीमा के लिए संगत बैकअप पुनर्स्थापित करें या OpenClaw अपग्रेड करें।

## सत्र SQLite माइग्रेशन

OpenClaw, Gateway स्टार्टअप के दौरान और `openclaw doctor --fix` के दौरान लेगेसी सत्र पंक्तियों और
ट्रांसक्रिप्ट इतिहास को प्रत्येक एजेंट के SQLite डेटाबेस में स्वचालित रूप से इम्पोर्ट करता है।
`openclaw doctor --session-sqlite <mode>` उस माइग्रेशन के लिए लक्षित निरीक्षण और सत्यापन उपकरण है। वर्तमान रनटाइम
सत्र पंक्तियाँ
`~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite` में रहती हैं। लेगेसी
`sessions.json` फ़ाइलें माइग्रेशन स्रोत हैं। हॉट ट्रांसक्रिप्ट JSONL फ़ाइलें सफल इम्पोर्ट के बाद
इम्पोर्ट होकर सक्रिय सत्र निर्देशिका से बाहर आर्काइव कर दी जाती हैं; आर्काइव-स्तरीय JSONL फ़ाइलें
सहायता आर्टिफ़ैक्ट बनी रहती हैं, रनटाइम फ़ॉलबैक नहीं।

मोड:

| मोड       | व्यवहार                                                                                                               |
| ---------- | ---------------------------------------------------------------------------------------------------------------------- |
| `inspect`  | इम्पोर्ट किए बिना लेगेसी और SQLite गणनाएँ तथा असंदर्भित JSONL फ़ाइलें पढ़ता है।                                       |
| `dry-run`  | लेगेसी प्रविष्टियों और ट्रांसक्रिप्ट JSONL फ़ाइलों को पार्स करता है, इम्पोर्ट योग्य पंक्तियाँ गिनता है और SQLite पंक्तियाँ लिखे बिना समस्याओं की रिपोर्ट करता है। |
| `import`   | चयनित लक्ष्यों के लिए लेगेसी प्रविष्टियों और ट्रांसक्रिप्ट घटनाओं को SQLite में इम्पोर्ट करता है।                                      |
| `validate` | चयनित लेगेसी स्रोतों की SQLite पंक्तियों और ट्रांसक्रिप्ट घटना गणनाओं से तुलना करता है।                                   |
| `compact`  | बड़े विलोपन या आर्काइव सफ़ाई के बाद खाली पृष्ठ पुनः प्राप्त करने के लिए चयनित एजेंट SQLite डेटाबेसों को चेकपॉइंट और VACUUM करता है।    |
| `recover`  | नवीनतम विफल माइग्रेशन रन पुनर्स्थापित करता है, उसके लक्ष्यों को सत्यापित करता है और एक स्वच्छ GitHub समस्या रिपोर्ट तैयार करता है।            |
| `restore`  | SQLite डेटा हटाए बिना दर्ज माइग्रेशन मैनिफ़ेस्ट से आर्काइव किए गए ट्रांसक्रिप्ट आर्टिफ़ैक्ट पुनर्स्थापित करता है।                  |

चयनकर्ता:

- डिफ़ॉल्ट: कॉन्फ़िगर किया गया डिफ़ॉल्ट एजेंट स्टोर, जब वह लेगेसी स्टोर फ़ाइल मौजूद हो।
- `--session-sqlite-agent <id>`: एक कॉन्फ़िगर किया गया एजेंट।
- `--session-sqlite-all-agents`: कॉन्फ़िगर किए गए एजेंट स्टोर और खोजे गए एजेंट स्टोर।
- `--session-sqlite-store <path>`: एक स्पष्ट लेगेसी `sessions.json` पथ।

मैन्युअल निरीक्षण क्रम:

```bash
openclaw doctor --session-sqlite inspect --session-sqlite-all-agents
openclaw doctor --session-sqlite dry-run --session-sqlite-all-agents --json
openclaw doctor --session-sqlite import --session-sqlite-all-agents
openclaw doctor --session-sqlite validate --session-sqlite-all-agents --json
openclaw doctor --session-sqlite compact --session-sqlite-all-agents
openclaw doctor --session-sqlite recover --github-issue
```

महत्वपूर्ण इतिहास वाली स्थापना पर `import` चलाने से पहले OpenClaw स्थिति निर्देशिका का
बैकअप लें। जब चयनित लेगेसी प्रविष्टि SQLite में अनुपस्थित हो, सत्र आईडी अलग हो या ट्रांसक्रिप्ट घटना
गणना अलग हो, तो `validate` गैर-शून्य निकास देता है। `--session-sqlite-store <path>` का उपयोग करते समय
जाँचें कि रिपोर्ट में अपेक्षित लक्ष्य गणना मौजूद है; कोई अस्तित्वहीन स्पष्ट स्टोर पथ किसी लक्ष्य का चयन नहीं करता।

SQLite विलोपन पहले डेटाबेस के अंदर पृष्ठ पुनः प्राप्त करते हैं; वे आवश्यक रूप से डेटाबेस फ़ाइल को
तुरंत छोटा नहीं करते। बड़े ट्रांसक्रिप्ट हटाने या आर्काइव करने के बाद, WAL फ़ाइलों को चेकपॉइंट करने,
`VACUUM` चलाने और डेटाबेस तथा WAL के पहले/बाद के आकारों की रिपोर्ट करने के लिए
`openclaw doctor --session-sqlite compact --session-sqlite-all-agents` चलाएँ। Compaction के लिए वर्तमान एजेंट स्कीमा वाली एक नियमित फ़ाइल,
चयनित एजेंट का स्थायी स्वामी मेटाडेटा और Doctor प्रक्रिया में कोई खुला हैंडल न होना आवश्यक है।
विनाशकारी `import`, `compact`, `recover` और `restore` मोड
अपने पूरे संचालन के दौरान Gateway स्टार्टअप वाला ही स्थिति स्वामित्व लॉक बनाए रखते हैं;
`inspect`, `dry-run` और `validate` केवल-पठन रहते हैं और उसे प्राप्त नहीं करते।
पहले Gateway रोकें। विनाशकारी मोड लाइव लेखन या किसी अन्य रखरखाव कमांड के साथ प्रतिस्पर्धा करने के
बजाय विफल हो जाते हैं। विनाशकारी `--session-sqlite-store` लक्ष्य सक्रिय स्थिति निर्देशिका के अंदर होना चाहिए;
किसी दूसरी स्थापना का रखरखाव करने से पहले `OPENCLAW_STATE_DIR` को उस स्टोर की स्वामी स्थिति निर्देशिका
पर सेट करें। मौजूदा हार्ड-लिंक किए गए लक्ष्य अस्वीकार कर दिए जाते हैं, क्योंकि कोई अन्य पथ लॉक की गई
स्थिति निर्देशिका के बाहर वही डेटाबेस inode साझा कर सकता है। यही स्वामित्व जाँचें SQLite WAL,
साझा-मेमोरी और रोलबैक-जर्नल साइडकार को भी कवर करती हैं।

प्रत्येक इम्पोर्ट ट्रांसक्रिप्ट आर्टिफ़ैक्ट को आर्काइव में स्थानांतरित करने से पहले
`~/.openclaw/session-sqlite-migration-runs/` के अंतर्गत एक मैनिफ़ेस्ट लिखता है।
यदि आर्टिफ़ैक्ट स्थानांतरित होने के बाद स्टार्टअप किसी विफल सत्र SQLite माइग्रेशन की रिपोर्ट करता है,
तो पुनर्प्राप्ति चलाएँ:

```bash
openclaw doctor --session-sqlite recover --github-issue
```

पुनर्प्राप्ति नवीनतम विफल माइग्रेशन मैनिफ़ेस्ट चुनती है, केवल मैनिफ़ेस्ट की
संग्रहीत कलाकृतियों को पुनर्स्थापित करती है, प्रभावित लक्ष्यों को सत्यापित करती है,
सैनिटाइज़ की गई `.failure.md` और `.failure.json` रिपोर्ट को रीफ़्रेश करती है, और एक GitHub समस्या
का मुख्य भाग तैयार करती है जिसमें ट्रांसक्रिप्ट की सामग्री, अपरिष्कृत परिवेश, सीक्रेट और असीमित
कॉन्फ़िग शामिल नहीं होते। जब कोई विफल माइग्रेशन मैनिफ़ेस्ट मौजूद नहीं होता, लेकिन चयनित एजेंट का SQLite
डेटाबेस दूषित होता है, डेटाबेस नहीं होता, या उसमें मुख्य डेटाबेस के बिना जर्नल साइडकार
होते हैं, तो पुनर्प्राप्ति फ़ाइलों के पूरे समूह को एक अस्थायी निरीक्षण
निर्देशिका में कॉपी करती है। SQLite उस अस्थायी प्रति में किसी मान्य हॉट जर्नल को
`quick_check`, `integrity_check`, और `foreign_key_check` के चलने से पहले रोल बैक कर सकता है, जबकि
मूल फ़ोरेंसिक फ़ाइलें अपरिवर्तित रहती हैं। विफल अखंडता जाँच या अनाथ
साइडकार, खोजे गए पूरे समूह का नाम एक `.corrupt-<timestamp>` प्रत्यय के साथ बदलकर
DB, WAL, SHM और रोलबैक-जर्नल फ़ाइलों को सुरक्षित रखते हैं। नाम बदलने की पकड़ी गई
विफलता की रिपोर्ट देने से पहले पहले से स्थानांतरित फ़ाइलें वापस कर दी जाती हैं, ताकि
पुनर्प्राप्ति योग्य फ़ाइल समूह चुपचाप विभाजित न हो। पुनर्प्राप्ति से पहले Gateway रोकें;
सक्रिय रूप से बदल रहे SQLite फ़ाइल समूह को कॉपी करना या उसका नाम बदलना असुरक्षित है और
विभिन्न ऑपरेटिंग सिस्टम पर अलग-अलग व्यवहार करता है। `--github-issue --yes` के साथ, डॉक्टर
GitHub CLI का उपयोग करके `openclaw/openclaw` में समस्या बनाता है; पुष्टि के बिना
यह स्थानीय सहायता रिपोर्ट लिखता है और पहले से भरा हुआ समस्या URL प्रिंट करता है।

`restore` निम्न-स्तरीय पूर्ववत कार्रवाई बनी रहती है। यह मैनिफ़ेस्ट के
`sourcePath -> archivePath` रिकॉर्ड का उपयोग करती है, संग्रहीत कलाकृतियों को केवल तभी वापस ले जाती है जब
मूल पथ अनुपस्थित हो, दोनों पथ मौजूद होने पर टकराव की रिपोर्ट करती है, और
SQLite डेटाबेस को यथास्थान छोड़ देती है।

### सत्र SQLite माइग्रेशन के बाद डाउनग्रेड करना

OpenClaw का कोई पुराना फ़ाइल-समर्थित संस्करण शुरू करने से पहले, संग्रहीत
पुरानी ट्रांसक्रिप्ट कलाकृतियों को पुनर्स्थापित करें:

```bash
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

पुराने संस्करण `sessions.json` प्रविष्टियाँ और उन प्रविष्टियों में दर्ज `sessionFile` पथ
पढ़ते हैं। SQLite माइग्रेशन के बाद, सफल आयात सक्रिय JSONL
ट्रांसक्रिप्ट को `session-sqlite-import-archive/` में स्थानांतरित कर देते हैं, इसलिए पुराना रनटाइम
उस इतिहास को तब तक नहीं देख सकता जब तक पुनर्स्थापना मैनिफ़ेस्ट में दर्ज उन कलाकृतियों को
उनके मूल पथों पर वापस नहीं ले जाती।

पुनर्स्थापना SQLite डेटा को नहीं मिटाती। SQLite पर स्विच करने के बाद बनाए गए सत्र
केवल SQLite में मौजूद होते हैं और पुराने रनटाइम में दिखाई नहीं देंगे। यदि आप बाद में
फिर अपग्रेड करते हैं, तो ऊपर दिया गया सामान्य माइग्रेशन सत्यापन क्रम चलाएँ, ताकि OpenClaw
आयात करने से पहले पुनर्स्थापित पुरानी कलाकृतियों की SQLite पंक्तियों से तुलना कर सके।

## टिप्पणियाँ

- Nix मोड (`OPENCLAW_NIX_MODE=1`) में, केवल-पढ़ने योग्य doctor जाँचें अब भी काम करती हैं, लेकिन `doctor --fix`, `doctor --repair`, `doctor --yes`, और `doctor --generate-gateway-token` अक्षम हैं क्योंकि `openclaw.json` अपरिवर्तनीय है। इसके बजाय इस इंस्टॉलेशन के लिए Nix स्रोत संपादित करें; nix-openclaw के लिए, एजेंट-प्रथम [त्वरित शुरुआत](https://github.com/openclaw/nix-openclaw#quick-start) का उपयोग करें।
- इंटरैक्टिव प्रॉम्प्ट (कीचेन/OAuth सुधार आदि) केवल तभी चलते हैं जब stdin एक TTY हो और `--non-interactive` सेट **नहीं** हो। हेडलेस रन (Cron, Telegram, कोई टर्मिनल नहीं) प्रॉम्प्ट छोड़ देते हैं।
- गैर-इंटरैक्टिव `doctor` रन अग्रिम Plugin लोडिंग छोड़ देते हैं, ताकि हेडलेस स्वास्थ्य जाँचें तेज़ रहें। इंटरैक्टिव सत्र अब भी पुराने स्वास्थ्य/सुधार प्रवाह के लिए आवश्यक Plugin सतहें लोड करते हैं।
- `--lint`, `--non-interactive` से अधिक सख्त है: हमेशा केवल-पढ़ने योग्य, कभी प्रॉम्प्ट नहीं करता और कभी सुरक्षित माइग्रेशन लागू नहीं करता। जब आप चाहते हों कि doctor बदलाव करे, तब `doctor --fix` या `doctor --repair` का उपयोग करें।
- डिफ़ॉल्ट रूप से रहस्यों की जाँच करते समय doctor `exec` SecretRefs निष्पादित नहीं करता। `--allow-exec` का उपयोग (`--lint` के साथ या उसके बिना) केवल तभी करें, जब आप जानबूझकर चाहते हों कि doctor उन कॉन्फ़िगर किए गए रहस्य रिज़ॉल्वर को चलाए।
- कोई भी कॉन्फ़िग लेखन (`--fix` सुधार सहित) बैकअप को `~/.openclaw/openclaw.json.bak` में घुमाता है (क्रमांकित `.bak.1`..`.bak.4` रिंग के साथ)। `--fix` स्कीमा सत्यापन द्वारा रिपोर्ट की गई अज्ञात कॉन्फ़िग कुंजियाँ भी हटाता है और हर हटाव को सूचीबद्ध करता है; अपडेट जारी होने के दौरान यह ऐसा नहीं करता, ताकि आंशिक रूप से लिखी गई अपग्रेड स्थिति को उसका माइग्रेशन पूरा होने से पहले न हटाया जाए।
- यदि `openclaw.json` को पार्स नहीं किया जा सकता और अंतिम ज्ञात-सही कॉन्फ़िग पुनर्प्राप्त नहीं किया जा सकता, तो `doctor --fix` मूल को `openclaw.json.clobbered.<timestamp>` के रूप में सुरक्षित रखता है, मौजूदा फ़ाइल को अपरिवर्तित छोड़ता है और आंशिक प्रतिस्थापन लिखने के बजाय त्रुटि के साथ बाहर निकलता है।
- जब कोई अन्य सुपरवाइज़र Gateway जीवनचक्र का स्वामी हो, तब `OPENCLAW_SERVICE_REPAIR_POLICY=external` सेट करें। Doctor फिर भी Gateway/सेवा स्वास्थ्य रिपोर्ट करता है और गैर-सेवा सुधार लागू करता है, लेकिन सेवा इंस्टॉल/प्रारंभ/पुनः प्रारंभ/बूटस्ट्रैप और पुरानी सेवा की सफ़ाई छोड़ देता है।
- Doctor प्रबंधित Gateway की लागू हीप सीमा और मौजूदा होस्ट या कंटेनर मेमोरी सीमा के लिए प्रयुक्त अनुकूली व्युत्पत्ति रिपोर्ट करता है। सुधार पास के बाहर यही रिपोर्ट पाने के लिए `openclaw gateway status` का उपयोग करें।
- Linux पर, doctor निष्क्रिय अतिरिक्त Gateway-जैसी systemd इकाइयों को अनदेखा करता है और सुधार के दौरान चल रही systemd Gateway सेवा के कमांड/एंट्रीपॉइंट मेटाडेटा को दोबारा नहीं लिखता। पहले सेवा रोकें, या सक्रिय लॉन्चर बदलने के लिए `openclaw gateway install --force` का उपयोग करें।
- `doctor --fix --non-interactive` अनुपलब्ध या पुराने Gateway सेवा निर्धारणों की रिपोर्ट करता है, लेकिन अपडेट सुधार मोड के बाहर उन्हें इंस्टॉल या दोबारा नहीं लिखता। अनुपलब्ध सेवा के लिए `openclaw gateway install`, या लॉन्चर बदलने के लिए `openclaw gateway install --force` चलाएँ।
- स्थिति अखंडता जाँचें सत्र निर्देशिका में अनाथ ट्रांसक्रिप्ट फ़ाइलों का पता लगाती हैं। उन्हें `.deleted.<timestamp>` के रूप में संग्रहित करने के लिए इंटरैक्टिव पुष्टि आवश्यक है; `--fix`, `--yes`, और हेडलेस रन उन्हें वहीं छोड़ देते हैं।
- Doctor पुराने Cron जॉब स्वरूपों के लिए `~/.openclaw/cron/jobs.json` (या `cron.store`) को स्कैन करता है और SQLite में कैनोनिकल पंक्तियाँ आयात करने से पहले उन्हें दोबारा लिखता है।
- Doctor स्पष्ट `payload.model` ओवरराइड वाले Cron जॉब की रिपोर्ट करता है, जिसमें प्रदाता-नेमस्पेस गणनाएँ और `agents.defaults.model` के विरुद्ध बेमेल शामिल हैं, ताकि डिफ़ॉल्ट मॉडल इनहेरिट न करने वाले निर्धारित जॉब प्रमाणीकरण या बिलिंग जाँच के दौरान दिखाई दें।
- Doctor अब भी प्रगति में चिह्नित (`state.runningAtMs`) Cron जॉब की रिपोर्ट करता है, जिसके कारण `openclaw cron list` उन्हें `running` के रूप में दिखा सकता है। यह जाँच केवल-पढ़ने योग्य है: यदि कोई Gateway वर्तमान में चिह्नित जॉब निष्पादित नहीं कर रहा है, तो अगला Cron सेवा स्टार्टअप बाधित रन दर्ज करता है और चिह्न हटा देता है।
- Linux पर, जब उपयोगकर्ता का crontab अब भी अनुरक्षण-रहित पुराना `~/.openclaw/bin/ensure-whatsapp.sh` चलाता है, तब doctor चेतावनी देता है; systemd उपयोगकर्ता-बस परिवेश न होने पर Cron इसके कारण `Gateway inactive` की गलत रिपोर्ट कर सकता है।
- WhatsApp सक्षम होने पर, doctor अब भी चल रहे स्थानीय `openclaw-tui` क्लाइंट के साथ खराब Gateway इवेंट लूप की जाँच करता है। `doctor --fix` केवल सत्यापित स्थानीय TUI क्लाइंट रोकता है, ताकि WhatsApp उत्तर पुराने TUI रीफ़्रेश लूप के पीछे कतारबद्ध न हों।
- जब HTTP(S) प्रॉक्सी परिवेश चर मौजूद हों लेकिन `tools.web.fetch.useTrustedEnvProxy` अक्षम हो, तो doctor समझाता है कि `web_fetch` अब भी प्रत्यक्ष रूटिंग का उपयोग करता है, एक छोटी प्रत्यक्ष TLS कनेक्टिविटी जाँच चलाता है और स्पष्ट ऑप्ट-इन का नाम बताता है। यह प्रॉक्सी विश्वास को कभी स्वचालित रूप से सक्षम नहीं करता।
- Doctor प्राथमिक मॉडल, फ़ॉलबैक, मॉडल अनुमति-सूचियों, छवि/वीडियो जनरेशन मॉडल, Heartbeat/सबएजेंट/Compaction ओवरराइड, हुक, चैनल मॉडल ओवरराइड, Cron पेलोड, और पुराने सत्र/ट्रांसक्रिप्ट रूट पिन में पुराने `codex/*` तथा `openai-codex/*` मॉडल संदर्भों को कैनोनिकल `openai/*` संदर्भों में दोबारा लिखता है। सुरक्षित होने पर `--fix` पुराने `models.providers.codex` और `models.providers.openai-codex` कॉन्फ़िग को भी मर्ज करता है, पुराने `openai-codex:*` प्रमाणीकरण प्रोफ़ाइल और `auth.order.openai-codex` प्रविष्टियों को `openai:*` में माइग्रेट करता है, Codex अभिप्राय को प्रदाता/मॉडल-स्कोप वाले `agentRuntime.id: "codex"` प्रविष्टियों में ले जाता है, पुराने संपूर्ण-एजेंट/सत्र रनटाइम पिन हटाता है, और सुधारे गए OpenAI एजेंट संदर्भों को प्रत्यक्ष OpenAI API-कुंजी प्रमाणीकरण के बजाय Codex प्रमाणीकरण रूटिंग पर रखता है।
- Doctor ऐसी गैर-रिक्त `auth.order.<provider>` सूचियों की रिपोर्ट करता है जिनकी संदर्भित सभी प्रोफ़ाइल हट चुकी हैं, जबकि संगत संग्रहीत क्रेडेंशियल मौजूद हैं। `doctor --fix` केवल उन्हीं पुराने ओवरराइड को हटाता है और स्वचालित प्रति-एजेंट क्रेडेंशियल चयन बहाल करता है; स्पष्ट रिक्त क्रम, आंशिक रूप से सक्रिय सूचियाँ, और संगत संग्रहीत क्रेडेंशियल के बिना क्रम अपरिवर्तित रहते हैं। यदि सक्रिय SQLite प्रमाणीकरण स्टोर अपठनीय या विकृत हो, तो doctor बताता है कि उसने यह सुधार क्यों छोड़ा। यदि चल रहे Gateway का कॉन्फ़िग रीलोड मोड लेखन को स्वचालित रूप से लागू नहीं करता, तो प्रमाणीकरण स्थिति की दोबारा जाँच से पहले उसे पुनः प्रारंभ करें।
- Doctor OpenClaw के पुराने संस्करणों की पुरानी Plugin निर्भरता स्टेजिंग स्थिति साफ़ करता है और प्रबंधित npm Plugins के लिए होस्ट `openclaw` पैकेज को दोबारा लिंक करता है, यदि वे इसे पीयर निर्भरता के रूप में घोषित करते हैं। यह कॉन्फ़िग द्वारा संदर्भित अनुपलब्ध डाउनलोड-योग्य Plugins (`plugins.entries`, कॉन्फ़िगर किए गए चैनल, कॉन्फ़िगर की गई प्रदाता/खोज सेटिंग्स, कॉन्फ़िगर किए गए एजेंट रनटाइम) की भी मरम्मत करता है। पैकेज अपडेट के दौरान, पैकेज अदला-बदली पूरी होने तक doctor पैकेज-मैनेजर Plugin मरम्मत छोड़ देता है; यदि किसी कॉन्फ़िगर किए गए Plugin को अब भी पुनर्प्राप्ति चाहिए, तो बाद में `openclaw doctor --fix` दोबारा चलाएँ। यदि डाउनलोड विफल होता है, तो doctor इंस्टॉल त्रुटि की रिपोर्ट करता है और अगले मरम्मत प्रयास के लिए कॉन्फ़िगर की गई Plugin प्रविष्टि सुरक्षित रखता है।
- Plugin खोज स्वस्थ होने पर, doctor `plugins.allow`/`plugins.deny`/`plugins.entries` से अनुपलब्ध Plugin आईडी, साथ ही मेल खाने वाला लटकता चैनल कॉन्फ़िग, Heartbeat लक्ष्य और चैनल मॉडल ओवरराइड हटाकर पुराने Plugin कॉन्फ़िग की मरम्मत करता है।
- Doctor प्रभावित `plugins.entries.<id>` प्रविष्टि को अक्षम करके और उसका अमान्य `config` पेलोड हटाकर अमान्य Plugin कॉन्फ़िग को क्वारंटीन करता है। Gateway स्टार्टअप पहले से ही केवल उस खराब Plugin को छोड़ता है, इसलिए अन्य Plugins और चैनल चलते रहते हैं।
- Doctor सेवानिवृत्त `plugins.entries.codex.config.codexDynamicToolsProfile` को हटाता है; Codex ऐप-सर्वर हमेशा Codex-मूल वर्कस्पेस टूल को मूल रूप में रखता है।
- Doctor पुराने समतल Talk कॉन्फ़िग (`talk.voiceId`, `talk.modelId`, और संबंधित प्रविष्टियों) को `talk.provider` + `talk.providers.<provider>` में स्वतः माइग्रेट करता है। बार-बार `doctor --fix` चलाने पर अब Talk सामान्यीकरण की रिपोर्ट/अनुप्रयोग नहीं होता, यदि एकमात्र अंतर ऑब्जेक्ट कुंजियों का क्रम हो।
- Doctor में मेमोरी-खोज तत्परता जाँच शामिल है और एम्बेडिंग क्रेडेंशियल अनुपलब्ध होने पर यह `openclaw configure --section model` की अनुशंसा कर सकता है।
- कोई कमांड स्वामी कॉन्फ़िगर न होने पर doctor चेतावनी देता है। कमांड स्वामी वह मानव ऑपरेटर खाता है जिसे केवल-स्वामी कमांड चलाने और खतरनाक कार्रवाइयाँ अनुमोदित करने की अनुमति है। DM पेयरिंग केवल किसी व्यक्ति को बॉट से बात करने देती है; यदि आपने प्रथम-स्वामी बूटस्ट्रैप के अस्तित्व में आने से पहले किसी प्रेषक को अनुमोदित किया था, तो `commands.ownerAllowFrom` स्पष्ट रूप से सेट करें।
- Codex-मोड एजेंट कॉन्फ़िगर होने और ऑपरेटर के Codex होम में व्यक्तिगत Codex CLI एसेट मौजूद होने पर doctor एक सूचनात्मक टिप्पणी रिपोर्ट करता है। स्थानीय Codex ऐप-सर्वर लॉन्च पृथक प्रति-एजेंट होम का उपयोग करते हैं; आवश्यकता हो तो पहले Codex Plugin इंस्टॉल करें, फिर उन एसेट की सूची बनाने के लिए `openclaw migrate plan codex` का उपयोग करें जिन्हें सोच-समझकर प्रोमोट किया जाना चाहिए।
- डिफ़ॉल्ट एजेंट के लिए अनुमत Skills मौजूदा रनटाइम परिवेश में अनुपलब्ध होने पर doctor चेतावनी देता है (अनुपलब्ध बाइनरी, परिवेश चर, कॉन्फ़िग, या OS आवश्यकताएँ)। `doctor --fix`, `skills.entries.<skill>.enabled=false` के साथ उन अनुपलब्ध Skills को अक्षम कर सकता है; यदि आप Skill सक्रिय रखना चाहते हैं, तो इसके बजाय अनुपलब्ध आवश्यकता को इंस्टॉल/कॉन्फ़िगर करें।
- यदि सैंडबॉक्स मोड सक्षम है लेकिन Docker अनुपलब्ध है, तो doctor सुधार उपाय (`install Docker` या `openclaw config set agents.defaults.sandbox.mode off`) के साथ एक स्पष्ट चेतावनी रिपोर्ट करता है।
- यदि पुरानी सैंडबॉक्स रजिस्ट्री फ़ाइलें या शार्ड निर्देशिकाएँ मौजूद हों (`~/.openclaw/sandbox/containers.json`, `~/.openclaw/sandbox/browsers.json`, `~/.openclaw/sandbox/containers/`, या `~/.openclaw/sandbox/browsers/`), तो doctor उनकी रिपोर्ट करता है; `--fix` वैध प्रविष्टियों को SQLite में माइग्रेट करता है और अमान्य पुरानी फ़ाइलों को क्वारंटीन करता है।
- यदि `gateway.auth.token`/`gateway.auth.password` SecretRef द्वारा प्रबंधित हों और मौजूदा कमांड पथ में अनुपलब्ध हों, तो doctor केवल-पढ़ने योग्य चेतावनी रिपोर्ट करता है और सादे-पाठ फ़ॉलबैक क्रेडेंशियल नहीं लिखता। exec-समर्थित SecretRefs के लिए, `--allow-exec` मौजूद न होने पर doctor निष्पादन छोड़ देता है।
- यदि सुधार पथ में चैनल SecretRef निरीक्षण विफल हो, तो doctor जल्दी बाहर निकलने के बजाय जारी रहता है और चेतावनी रिपोर्ट करता है।
- स्थिति-निर्देशिका माइग्रेशन के बाद, जब सक्षम डिफ़ॉल्ट Telegram या Discord खाते परिवेश फ़ॉलबैक पर निर्भर हों और `TELEGRAM_BOT_TOKEN` या `DISCORD_BOT_TOKEN` doctor प्रक्रिया के लिए अनुपलब्ध हो, तब doctor चेतावनी देता है।
- Telegram `allowFrom` उपयोगकर्ता नाम स्वतः-रिज़ॉल्यूशन (`doctor --fix`) के लिए मौजूदा कमांड पथ में रिज़ॉल्व किया जा सकने वाला Telegram टोकन आवश्यक है। यदि टोकन निरीक्षण अनुपलब्ध हो, तो doctor चेतावनी रिपोर्ट करता है और उस पास के लिए स्वतः-रिज़ॉल्यूशन छोड़ देता है।

## macOS: `launchctl` परिवेश ओवरराइड

यदि आपने पहले `launchctl setenv OPENCLAW_GATEWAY_TOKEN ...` (या `...PASSWORD`) चलाया था, तो वह मान आपकी कॉन्फ़िग फ़ाइल को ओवरराइड करता है और लगातार "अनधिकृत" त्रुटियाँ उत्पन्न कर सकता है।

```bash
launchctl getenv OPENCLAW_GATEWAY_TOKEN
launchctl getenv OPENCLAW_GATEWAY_PASSWORD

launchctl unsetenv OPENCLAW_GATEWAY_TOKEN
launchctl unsetenv OPENCLAW_GATEWAY_PASSWORD
```
