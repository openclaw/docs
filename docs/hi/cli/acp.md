उस [Agent Client Protocol (ACP)](https://agentclientprotocol.com/) ब्रिज को चलाएँ जो OpenClaw Gateway से संचार करता है।

`openclaw acp` IDE के लिए stdio पर ACP का उपयोग करता है और प्रॉम्प्ट को WebSocket पर Gateway को अग्रेषित करता है, साथ ही ACP सेशन को Gateway सेशन कुंजियों से मैप किए रखता है। यह Gateway-समर्थित ACP ब्रिज है, पूर्ण ACP-मूल एडिटर रनटाइम नहीं: यह सेशन रूटिंग, प्रॉम्प्ट डिलीवरी और स्ट्रीमिंग अपडेट पर केंद्रित है।

यदि आप चाहते हैं कि कोई बाहरी MCP क्लाइंट ACP हार्नेस सेशन होस्ट करने के बजाय सीधे OpenClaw चैनल वार्तालापों से संचार करे, तो इसके बजाय [`openclaw mcp serve`](/hi/cli/mcp) का उपयोग करें।

## यह क्या नहीं है

`openclaw acp` का अर्थ है कि OpenClaw ACP सर्वर के रूप में कार्य करता है: कोई IDE या ACP क्लाइंट OpenClaw से कनेक्ट होता है और OpenClaw उस कार्य को Gateway सेशन में अग्रेषित करता है।

यह [ACP एजेंट](/hi/tools/acp-agents) से अलग है, जहाँ OpenClaw `acpx` के माध्यम से Codex या Claude Code जैसे बाहरी हार्नेस को चलाता है।

त्वरित नियम:

- एडिटर/क्लाइंट ACP के माध्यम से OpenClaw से संचार करना चाहता है: `openclaw acp` का उपयोग करें
- OpenClaw को Codex/Claude/Gemini को ACP हार्नेस के रूप में लॉन्च करना चाहिए: `/acp spawn` और [ACP एजेंट](/hi/tools/acp-agents) का उपयोग करें

| प्रति-सेशन MCP सर्वर (`mcpServers`)                                | असमर्थित | ब्रिज मोड प्रति-सेशन MCP सर्वर अनुरोधों को अस्वीकार करता है। इसके बजाय MCP को OpenClaw Gateway या एजेंट पर कॉन्फ़िगर करें।                                                                                                                          |
| क्लाइंट फ़ाइल सिस्टम विधियाँ (`fs/read_text_file`, `fs/write_text_file`) | असमर्थित | ब्रिज ACP क्लाइंट फ़ाइल सिस्टम विधियों को कॉल नहीं करता है।                                                                                                                                                                               |

## ज्ञात सीमाएँ

- `loadSession` केवल ब्रिज द्वारा बनाए गए सेशन के लिए संपूर्ण ACP इवेंट-लेज़र इतिहास को दोबारा चलाता है। पुराने/लेज़र-रहित सेशन ट्रांसक्रिप्ट फ़ॉलबैक का उपयोग करते हैं और ऐतिहासिक टूल कॉल या सिस्टम सूचनाओं का पुनर्निर्माण नहीं करते।
- यदि कई ACP क्लाइंट समान Gateway सेशन कुंजी साझा करते हैं, तो इवेंट और रद्दीकरण रूटिंग प्रत्येक क्लाइंट के लिए सख्ती से पृथक होने के बजाय सर्वोत्तम-प्रयास होती है। जब आपको साफ़ एडिटर-स्थानीय टर्न चाहिए, तो डिफ़ॉल्ट पृथक `acp-bridge:<uuid>` सेशन को प्राथमिकता दें।
- Gateway की रुकने की स्थितियाँ ACP रुकने के कारणों में रूपांतरित होती हैं, लेकिन यह मैपिंग पूर्णतः ACP-मूल रनटाइम की तुलना में कम अभिव्यंजक है।
- सेशन नियंत्रण Gateway विकल्पों का एक केंद्रित उपसमुच्चय प्रस्तुत करते हैं: विचार स्तर, टूल वर्बोसिटी, रीजनिंग, उपयोग विवरण और उन्नत कार्रवाइयाँ। मॉडल चयन और निष्पादन-होस्ट नियंत्रण ACP कॉन्फ़िगरेशन विकल्पों के रूप में उपलब्ध नहीं हैं।
- `session_info_update` और `usage_update` लाइव ACP-मूल रनटाइम लेखांकन से नहीं, बल्कि Gateway सेशन स्नैपशॉट से प्राप्त होते हैं। उपयोग अनुमानित है, इसमें लागत डेटा नहीं होता और इसे केवल तभी उत्सर्जित किया जाता है जब Gateway कुल टोकन डेटा को नवीन चिह्नित करता है।
- टूल अनुवर्ती डेटा सर्वोत्तम-प्रयास है: ब्रिज ज्ञात टूल आर्ग्युमेंट/परिणामों में दिखाई देने वाले फ़ाइल पथ प्रस्तुत करता है, लेकिन ACP टर्मिनल या संरचित फ़ाइल डिफ़ उत्सर्जित नहीं करता।
- निष्पादन अनुमोदन रिले सक्रिय ACP प्रॉम्प्ट टर्न तक सीमित है; अन्य Gateway सेशन के अनुमोदन अनदेखे किए जाते हैं।

## उपयोग

```bash
openclaw acp

# रिमोट Gateway
openclaw acp --url wss://gateway-host:18789 --token <token>

# रिमोट Gateway (फ़ाइल से टोकन)
openclaw acp --url wss://gateway-host:18789 --token-file ~/.openclaw/gateway.token

# किसी मौजूदा सेशन कुंजी से जुड़ें
openclaw acp --session agent:main:main

# लेबल के अनुसार जुड़ें (पहले से मौजूद होना आवश्यक है)
openclaw acp --session-label "support inbox"

# पहले प्रॉम्प्ट से पहले सेशन कुंजी रीसेट करें
openclaw acp --session agent:main:main --reset-session
```

## ACP क्लाइंट (डीबग)

IDE के बिना ब्रिज की बुनियादी जाँच करने के लिए अंतर्निहित ACP क्लाइंट का उपयोग करें। यह ACP ब्रिज को स्पॉन करता है और आपको इंटरैक्टिव रूप से प्रॉम्प्ट टाइप करने देता है।

```bash
openclaw acp client

# स्पॉन किए गए ब्रिज को रिमोट Gateway की ओर इंगित करें
openclaw acp client --server-args --url wss://gateway-host:18789 --token-file ~/.openclaw/gateway.token

# सर्वर कमांड ओवरराइड करें (डिफ़ॉल्ट: openclaw)
openclaw acp client --server "node" --server-args openclaw.mjs acp --url ws://127.0.0.1:19001
```

अनुमति मॉडल (क्लाइंट डीबग मोड):

- स्वचालित अनुमोदन अनुमति-सूची पर आधारित है और केवल विश्वसनीय मुख्य टूल आईडी पर लागू होता है।
- `read` स्वचालित अनुमोदन वर्तमान कार्यशील डायरेक्टरी (`--cwd` सेट होने पर) तक सीमित है।
- ACP केवल सीमित रीड-ओनली वर्गों को स्वचालित रूप से अनुमोदित करता है: सक्रिय cwd के अंतर्गत सीमित `read` कॉल, साथ ही रीड-ओनली खोज टूल (`search`, `web_search`, `memory_search`)। अज्ञात/गैर-मुख्य टूल, दायरे से बाहर के रीड, निष्पादन-सक्षम टूल, नियंत्रण-प्लेन टूल, परिवर्तनकारी टूल और इंटरैक्टिव प्रवाहों के लिए हमेशा स्पष्ट प्रॉम्प्ट अनुमोदन आवश्यक होता है।
- सर्वर द्वारा प्रदान किए गए `toolCall.kind` को अविश्वसनीय मेटाडेटा माना जाता है, प्राधिकरण स्रोत नहीं।
- यह ACP ब्रिज नीति ACPX हार्नेस अनुमतियों से अलग है। यदि आप `acpx` बैकएंड के माध्यम से OpenClaw चलाते हैं, तो `plugins.entries.acpx.config.permissionMode=approve-all` उस हार्नेस सेशन के लिए आपातकालीन "yolo" स्विच है।

## प्रोटोकॉल स्मोक परीक्षण

प्रोटोकॉल-स्तरीय डीबगिंग के लिए, पृथक स्थिति वाला Gateway शुरू करें और ACP JSON-RPC क्लाइंट के साथ stdio पर `openclaw acp` चलाएँ। `initialize`, `session/new`, निरपेक्ष `cwd` के साथ `session/list`, `session/resume`, `session/close`, डुप्लिकेट बंद करना और अनुपस्थित पुनः आरंभ शामिल करें।

प्रमाण में घोषित जीवनचक्र क्षमताएँ, Gateway-समर्थित सेशन पंक्ति, अपडेट सूचनाएँ और Gateway `sessions.list` लॉग शामिल होने चाहिए:

```json
{
  "initialize": {
    "protocolVersion": 1,
    "agentCapabilities": {
      "sessionCapabilities": {
        "list": {},
        "resume": {},
        "close": {}
      }
    }
  },
  "listSessions": {
    "sessions": [
      {
        "sessionId": "agent:main:acp-smoke",
        "cwd": "/path/to/workspace",
        "_meta": {
          "sessionKey": "agent:main:acp-smoke",
          "kind": "direct"
        }
      }
    ],
    "nextCursor": null
  },
  "notifications": ["session_info_update", "available_commands_update", "usage_update"],
  "gatewayLogTail": ["[gateway] ready", "[ws] ⇄ res ✓ sessions.list 305ms"]
}
```

एकमात्र ACP प्रमाण के रूप में `openclaw gateway call sessions.list` का उपयोग करने से बचें। वह CLI पथ नवीन-टोकन ऑपरेटर स्कोप अपग्रेड का अनुरोध कर सकता है; ACP ब्रिज की शुद्धता ACP stdio फ़्रेम और Gateway `sessions.list` लॉग द्वारा सिद्ध होती है।

## इसका उपयोग कैसे करें

ACP का उपयोग तब करें जब कोई IDE (या अन्य क्लाइंट) Agent Client Protocol का उपयोग करता हो और आप चाहते हों कि वह OpenClaw Gateway सेशन चलाए।

1. सुनिश्चित करें कि Gateway चल रहा है (स्थानीय या रिमोट)।
2. Gateway लक्ष्य कॉन्फ़िगर करें (कॉन्फ़िगरेशन या फ़्लैग)।
3. अपने IDE को stdio पर `openclaw acp` चलाने के लिए इंगित करें।

उदाहरण कॉन्फ़िगरेशन (स्थायी):

```bash
openclaw config set gateway.remote.url wss://gateway-host:18789
openclaw config set gateway.remote.token <token>
```

सीधे चलाने का उदाहरण (कॉन्फ़िगरेशन में कुछ लिखे बिना):

```bash
openclaw acp --url wss://gateway-host:18789 --token <token>
# स्थानीय प्रक्रिया सुरक्षा के लिए प्राथमिकता दें
openclaw acp --url wss://gateway-host:18789 --token-file ~/.openclaw/gateway.token
```

## एजेंट चुनना

ACP सीधे एजेंट नहीं चुनता। यह Gateway सत्र कुंजी के अनुसार रूट करता है। किसी विशिष्ट एजेंट को लक्षित करने के लिए एजेंट-स्कोप वाली सत्र कुंजियों का उपयोग करें:

```bash
openclaw acp --session agent:main:main
openclaw acp --session agent:design:main
openclaw acp --session agent:qa:bug-123
```

प्रत्येक ACP सत्र एक ही Gateway सत्र कुंजी से मैप होता है। एक एजेंट के कई सत्र हो सकते हैं; जब तक आप कुंजी या लेबल को ओवरराइड नहीं करते, ACP डिफ़ॉल्ट रूप से एक पृथक `acp-bridge:<uuid>` सत्र का उपयोग करता है।

ब्रिज मोड में प्रति-सत्र `mcpServers` समर्थित नहीं हैं। यदि कोई ACP क्लाइंट उन्हें `newSession` या `loadSession` के दौरान भेजता है, तो ब्रिज उन्हें चुपचाप अनदेखा करने के बजाय एक स्पष्ट त्रुटि लौटाता है।

यदि आप चाहते हैं कि ACPX-समर्थित सत्र OpenClaw Plugin टूल या `cron` जैसे चुने हुए अंतर्निहित टूल देख सकें, तो प्रति-सत्र `mcpServers` पास करने का प्रयास करने के बजाय Gateway-साइड ACPX MCP ब्रिज सक्षम करें। [ACP एजेंट](/hi/tools/acp-agents-setup#plugin-tools-mcp-bridge) और [OpenClaw टूल MCP ब्रिज](/hi/tools/acp-agents-setup#openclaw-tools-mcp-bridge) देखें।

## `acpx` से उपयोग करें (Codex, Claude, अन्य ACP क्लाइंट)

यदि आप चाहते हैं कि Codex या Claude Code जैसा कोई कोडिंग एजेंट ACP के माध्यम से आपके OpenClaw बॉट से संवाद करे, तो इसके अंतर्निहित `openclaw` लक्ष्य के साथ `acpx` का उपयोग करें।

सामान्य प्रवाह:

1. Gateway चलाएँ और सुनिश्चित करें कि ACP ब्रिज उस तक पहुँच सकता है।
2. `acpx openclaw` को `openclaw acp` पर इंगित करें।
3. उस OpenClaw सत्र कुंजी को लक्षित करें जिसका उपयोग कोडिंग एजेंट से करवाना है।

उदाहरण:

```bash
# आपके डिफ़ॉल्ट OpenClaw ACP सत्र में एकबारगी अनुरोध
acpx openclaw exec "सक्रिय OpenClaw सत्र की स्थिति का सारांश दें।"

# आगामी संवादों के लिए स्थायी नामित सत्र
acpx openclaw sessions ensure --name codex-bridge
acpx openclaw -s codex-bridge --cwd /path/to/repo \
  "इस रेपो से संबंधित हालिया संदर्भ के लिए मेरे OpenClaw कार्य एजेंट से पूछें।"
```

यदि आप चाहते हैं कि `acpx openclaw` हर बार किसी विशिष्ट Gateway और सत्र कुंजी को लक्षित करे, तो `~/.acpx/config.json` में `openclaw` एजेंट कमांड को ओवरराइड करें:

```json
{
  "agents": {
    "openclaw": {
      "command": "env OPENCLAW_HIDE_BANNER=1 OPENCLAW_SUPPRESS_NOTES=1 openclaw acp --url ws://127.0.0.1:18789 --token-file ~/.openclaw/gateway.token --session agent:main:main"
    }
  }
}
```

रेपो-स्थानीय OpenClaw चेकआउट के लिए, डेवलपमेंट रनर के बजाय सीधे CLI एंट्रीपॉइंट का उपयोग करें, ताकि ACP स्ट्रीम साफ़ रहे:

```bash
env OPENCLAW_HIDE_BANNER=1 OPENCLAW_SUPPRESS_NOTES=1 node openclaw.mjs acp ...
```

टर्मिनल को स्क्रैप किए बिना Codex, Claude Code या किसी अन्य ACP-जागरूक क्लाइंट को OpenClaw एजेंट से प्रासंगिक जानकारी प्राप्त करने देने का यह सबसे आसान तरीका है।

## Zed एडिटर सेटअप

`~/.config/zed/settings.json` में एक कस्टम ACP एजेंट जोड़ें (या Zed के Settings UI का उपयोग करें):

```json
{
  "agent_servers": {
    "OpenClaw ACP": {
      "type": "custom",
      "command": "openclaw",
      "args": ["acp"],
      "env": {}
    }
  }
}
```

किसी विशिष्ट Gateway या एजेंट को लक्षित करने के लिए:

```json
{
  "agent_servers": {
    "OpenClaw ACP": {
      "type": "custom",
      "command": "openclaw",
      "args": [
        "acp",
        "--url",
        "wss://gateway-host:18789",
        "--token",
        "<token>",
        "--session",
        "agent:design:main"
      ],
      "env": {}
    }
  }
}
```

Zed में, Agent पैनल खोलें और थ्रेड शुरू करने के लिए "OpenClaw ACP" चुनें।

## सत्र मैपिंग

डिफ़ॉल्ट रूप से, ACP ब्रिज सत्रों को `acp-bridge:` उपसर्ग वाली एक पृथक Gateway सत्र कुंजी मिलती है। ये सामान्य-मॉडल ब्रिज सत्र कृत्रिम और अस्थायी हैं: इन पर पुराने प्रविष्टि-हटाने की प्रक्रिया लागू होती है और इन्हें संरक्षित मानवीय वार्तालाप सतहों के रूप में नहीं माना जाता। किसी ज्ञात सत्र का पुनः उपयोग करने के लिए, सत्र कुंजी या लेबल पास करें:

- `--session <key>`: किसी विशिष्ट Gateway सत्र कुंजी का उपयोग करें।
- `--session-label <label>`: लेबल द्वारा किसी मौजूदा सत्र को रिज़ॉल्व करें।
- `--reset-session`: उस कुंजी के लिए एक नया सत्र आईडी बनाएँ (वही कुंजी, नई ट्रांस्क्रिप्ट)।

यदि आपका ACP क्लाइंट मेटाडेटा का समर्थन करता है, तो आप इसे प्रति सत्र ओवरराइड कर सकते हैं:

```json
{
  "_meta": {
    "sessionKey": "agent:main:main",
    "sessionLabel": "support inbox",
    "resetSession": true
  }
}
```

सत्र कुंजियों के बारे में अधिक जानकारी के लिए [/concepts/session](/hi/concepts/session) देखें।

## विकल्प

- `--url <url>`: Gateway WebSocket URL (कॉन्फ़िगर होने पर डिफ़ॉल्ट: `gateway.remote.url`)।
- `--token <token>`: Gateway प्रमाणीकरण टोकन।
- `--token-file <path>`: फ़ाइल से Gateway प्रमाणीकरण टोकन पढ़ें।
- `--password <password>`: Gateway प्रमाणीकरण पासवर्ड।
- `--password-file <path>`: फ़ाइल से Gateway प्रमाणीकरण पासवर्ड पढ़ें।
- `--session <key>`: डिफ़ॉल्ट सत्र कुंजी।
- `--session-label <label>`: रिज़ॉल्व करने के लिए डिफ़ॉल्ट सत्र लेबल।
- `--require-existing`: यदि सत्र कुंजी/लेबल मौजूद नहीं है, तो विफल हों।
- `--reset-session`: पहले उपयोग से पूर्व सत्र कुंजी रीसेट करें।
- `--no-prefix-cwd`: प्रॉम्प्ट के आगे कार्यशील डायरेक्टरी न जोड़ें।
- `--provenance <off|meta|meta+receipt>`: ACP उद्गम मेटाडेटा या रसीदें शामिल करें।
- `--verbose, -v`: stderr पर विस्तृत लॉगिंग।

सुरक्षा टिप्पणी:

- `--token` और `--password` कुछ सिस्टम पर स्थानीय प्रोसेस सूचियों में दिखाई दे सकते हैं। `--token-file`/`--password-file` या एनवायरनमेंट वेरिएबल (`OPENCLAW_GATEWAY_TOKEN`, `OPENCLAW_GATEWAY_PASSWORD`) को प्राथमिकता दें।
- Gateway प्रमाणीकरण रिज़ॉल्यूशन अन्य Gateway क्लाइंट द्वारा उपयोग किए जाने वाले साझा अनुबंध का पालन करता है:
  - स्थानीय मोड: पहले env (`OPENCLAW_GATEWAY_*`), फिर `gateway.auth.*`; केवल `gateway.auth.*` अनसेट होने पर `gateway.remote.*` पर फ़ॉलबैक (कॉन्फ़िगर किया गया लेकिन अनरिज़ॉल्व स्थानीय SecretRef चुपचाप फ़ॉलबैक करने के बजाय सुरक्षित रूप से विफल होता है)
  - रिमोट मोड: रिमोट प्राथमिकता नियमों के अनुसार env/config फ़ॉलबैक के साथ `gateway.remote.*`
  - `--url` ओवरराइड-सुरक्षित है और निहित config/env क्रेडेंशियल का पुनः उपयोग नहीं करता; स्पष्ट `--token`/`--password` (या फ़ाइल वेरिएंट) पास करें

### `acp client` विकल्प

- `--cwd <dir>`: ACP सत्र की कार्यशील डायरेक्टरी।
- `--server <command>`: ACP सर्वर कमांड (डिफ़ॉल्ट: `openclaw`)।
- `--server-args <args...>`: ACP सर्वर को पास किए जाने वाले अतिरिक्त आर्ग्युमेंट।
- `--server-verbose`: ACP सर्वर पर विस्तृत लॉगिंग सक्षम करें।
- `--verbose, -v`: विस्तृत क्लाइंट लॉगिंग।
- `openclaw acp client` उत्पन्न ब्रिज प्रोसेस पर `OPENCLAW_SHELL=acp-client` सेट करता है, जिसका उपयोग संदर्भ-विशिष्ट शेल/प्रोफ़ाइल नियमों के लिए किया जा सकता है।
