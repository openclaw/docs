# `openclaw models`

मॉडल खोज, स्कैनिंग और कॉन्फ़िगरेशन (डिफ़ॉल्ट मॉडल, फ़ॉलबैक, प्रमाणीकरण प्रोफ़ाइलें)।

संबंधित:

- प्रदाता + मॉडल: [मॉडल](/hi/providers/models)
- मॉडल चयन की अवधारणाएँ + `/models` स्लैश कमांड: [मॉडल अवधारणा](/hi/concepts/models)
- प्रदाता प्रमाणीकरण सेटअप: [आरंभ करना](/hi/start/getting-started)

## सामान्य कमांड

```bash
openclaw models status
openclaw models list
openclaw models set <model-or-alias>
openclaw models set-image <model-or-alias>
openclaw models scan
```

`status` और `auth` उपकमांड किसी कॉन्फ़िगर किए गए एजेंट को लक्षित करने के लिए `--agent <id>` स्वीकार करते हैं; `list`, `scan`, `aliases`, और `fallbacks`/`image-fallbacks` हमेशा कॉन्फ़िगर किए गए डिफ़ॉल्ट एजेंट का उपयोग करते हैं, और `set`/`set-image` सीधे `--agent` को अस्वीकार करते हैं। इसे छोड़ने पर, `--agent`-सजग कमांड, यदि `OPENCLAW_AGENT_DIR` सेट है तो उसका, अन्यथा कॉन्फ़िगर किए गए डिफ़ॉल्ट एजेंट का उपयोग करते हैं।

### स्थिति

`openclaw models status` समाधान किए गए डिफ़ॉल्ट/फ़ॉलबैक के साथ प्रमाणीकरण का अवलोकन दिखाता है। Codex जैसे Plugin-स्वामित्व वाले एजेंट रनटाइम के लिए, यह भी जाँचता है कि स्वामी Plugin सक्षम है और स्टार्टअप पेलोड सत्यापन में सफल हुआ है या नहीं। मान्य क्रेडेंशियल लेकिन अनुपलब्ध रनटाइम वाला रूट `usable` के बजाय `status: unavailable` रिपोर्ट करता है; JSON आउटपुट में अलग-अलग `authStatus`, `runtimeStatus`, और सीमित रनटाइम निदान शामिल होते हैं। प्रदाता उपयोग स्नैपशॉट उपलब्ध होने पर, OAuth/API-कुंजी स्थिति अनुभाग में प्रदाता उपयोग विंडो और कोटा स्नैपशॉट शामिल होते हैं। वर्तमान उपयोग-विंडो प्रदाता: Anthropic, GitHub Copilot, Gemini CLI, OpenAI, MiniMax, Xiaomi, और z.ai। उपलब्ध होने पर उपयोग प्रमाणीकरण प्रदाता-विशिष्ट हुक से आता है; अन्यथा OpenClaw प्रमाणीकरण प्रोफ़ाइलों, परिवेश या कॉन्फ़िगरेशन से मेल खाते OAuth/API-कुंजी क्रेडेंशियल का उपयोग करता है।

`--json` आउटपुट में, `auth.providers` परिवेश/कॉन्फ़िगरेशन/स्टोर-सजग प्रदाता अवलोकन है, जबकि `auth.oauth` केवल प्रमाणीकरण-स्टोर प्रोफ़ाइल की स्थिति है।

विकल्प:

| फ़्लैग                      | प्रभाव                                                                                                                                   |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `--json`                  | JSON आउटपुट; प्रमाणीकरण-प्रोफ़ाइल, प्रदाता और स्टार्टअप निदान stderr पर जाते हैं, ताकि stdout को `jq` में पाइप किया जा सके।                            |
| `--plain`                 | सादा टेक्स्ट आउटपुट।                                                                                                                       |
| `--check`                 | प्रमाणीकरण की अवधि समाप्त होने वाली/समाप्त हो चुकी हो या चयनित एजेंट रनटाइम अनुपलब्ध हो, तो गैर-शून्य स्थिति के साथ बाहर निकलें: `1` = अनुपलब्ध/समाप्त/अनुपस्थित, `2` = अवधि समाप्त होने वाली। |
| `--probe`                 | कॉन्फ़िगर की गई प्रमाणीकरण प्रोफ़ाइलों की लाइव जाँच। वास्तविक अनुरोध; टोकन की खपत हो सकती है और दर सीमाएँ सक्रिय हो सकती हैं।                                       |
| `--probe-provider <name>` | केवल एक प्रदाता की जाँच करें।                                                                                                                 |
| `--probe-profile <id>`    | विशिष्ट प्रमाणीकरण प्रोफ़ाइल आईडी की जाँच करें (दोहराकर या अल्पविराम से अलग करके)।                                                                             |
| `--probe-timeout <ms>`    | प्रत्येक जाँच की समय-सीमा।                                                                                                                       |
| `--probe-concurrency <n>` | समवर्ती जाँचें।                                                                                                                       |
| `--probe-max-tokens <n>`  | जाँच के अधिकतम टोकन (सर्वोत्तम प्रयास)।                                                                                                          |
| `--agent <id>`            | कॉन्फ़िगर किया गया एजेंट आईडी; `OPENCLAW_AGENT_DIR` को ओवरराइड करता है।                                                                                     |

जाँच पंक्तियाँ प्रमाणीकरण प्रोफ़ाइलों, परिवेश क्रेडेंशियल या `models.json` से आ सकती हैं। जाँच स्थिति श्रेणियाँ: `ok`, `auth`, `rate_limit`, `billing`, `timeout`, `format`, `unknown`, `no_model`।

जब कोई जाँच मॉडल कॉल तक कभी नहीं पहुँचती, तब अपेक्षित जाँच विवरण/कारण कोड:

- `excluded_by_auth_order`: एक संग्रहीत प्रोफ़ाइल मौजूद है, लेकिन स्पष्ट `auth.order.<provider>` ने उसे छोड़ दिया है, इसलिए जाँच उसे आज़माने के बजाय अपवर्जन रिपोर्ट करती है।
- `missing_credential`, `invalid_expires`, `expired`, `unresolved_ref`: प्रोफ़ाइल मौजूद है लेकिन पात्र नहीं है या उसका समाधान नहीं किया जा सकता।
- `ineligible_profile`: किसी अन्य कारण से प्रोफ़ाइल प्रदाता कॉन्फ़िगरेशन के साथ असंगत है।
- `no_model`: प्रदाता प्रमाणीकरण मौजूद है, लेकिन OpenClaw उस प्रदाता के लिए जाँच योग्य मॉडल उम्मीदवार का समाधान नहीं कर सका।

OpenAI ChatGPT/Codex OAuth समस्या निवारण के लिए, `openclaw models status`, `openclaw models auth list --provider openai`, और `openclaw config get agents.defaults.model --json` यह पुष्टि करने का सबसे तेज़ तरीका हैं कि किसी एजेंट के पास मूल Codex रनटाइम के माध्यम से `openai/*` के लिए उपयोग योग्य `openai` OAuth प्रोफ़ाइल है या नहीं। [OpenAI प्रदाता सेटअप](/hi/providers/openai#check-and-recover-codex-oauth-routing) देखें।

### सूची

`openclaw models list` केवल-पढ़ने योग्य है: यह कॉन्फ़िगरेशन, प्रमाणीकरण प्रोफ़ाइलें, मौजूदा कैटलॉग स्थिति और प्रदाता-स्वामित्व वाली कैटलॉग पंक्तियाँ पढ़ता है, लेकिन `models.json` को कभी दोबारा नहीं लिखता।

विकल्प: `--all` (पूरा कैटलॉग), `--local` (केवल स्थानीय मॉडल फ़िल्टर करें), `--provider <id>`, `--json`, `--plain`।

टिप्पणियाँ:

- `Auth` कॉलम केवल-पढ़ने योग्य है। OpenAI जैसे प्रदाता-स्वामित्व वाले मॉडल रूट के लिए, यह प्रत्येक पंक्ति के API/आधार-URL रूट का प्रभावी `auth.order` में पात्र प्रोफ़ाइलों, परिवेश/कॉन्फ़िगरेशन क्रेडेंशियल और समाधान किए गए कमांड-स्कोप SecretRefs से मिलान करता है। रूट नीति अनुपलब्ध होने पर कोई ठोस OpenAI पंक्ति प्रदाता-स्तरीय प्रमाणीकरण उधार लेने के बजाय अज्ञात बनी रहती है; केवल-प्रदाता विरासती जाँचें और अन्य प्रदाता प्रदाता-स्तरीय व्यवहार बनाए रखते हैं। Plugin का कृत्रिम-प्रमाणीकरण मेटाडेटा केवल रनटाइम-क्षमता का संकेत है, मूल खाता प्रमाणीकरण का प्रमाण नहीं, इसलिए सकारात्मक रजिस्ट्री साक्ष्य के बिना खाता-निर्भर रूट अज्ञात रहते हैं। कमांड प्रदाता रनटाइम लोड नहीं करता, कीचेन सीक्रेट नहीं पढ़ता, प्रदाता API कॉल नहीं करता, और सटीक निष्पादन तत्परता प्रमाणित नहीं करता।
- `models list --all --provider <id>` में Plugin मैनिफ़ेस्ट या बंडल किए गए प्रदाता कैटलॉग मेटाडेटा की प्रदाता-स्वामित्व वाली स्थिर कैटलॉग पंक्तियाँ शामिल हो सकती हैं, भले ही आपने अभी तक उस प्रदाता से प्रमाणीकरण न किया हो। मेल खाता प्रमाणीकरण कॉन्फ़िगर होने तक वे पंक्तियाँ फिर भी अनुपलब्ध दिखाई देती हैं।
- जब प्रदाता कैटलॉग खोज धीमी हो, तब `models list` नियंत्रण तल को प्रतिक्रियाशील बनाए रखता है। डिफ़ॉल्ट और कॉन्फ़िगर किए गए दृश्य थोड़े इंतज़ार के बाद कॉन्फ़िगर की गई या कृत्रिम मॉडल पंक्तियों पर वापस आते हैं और खोज को पृष्ठभूमि में पूरा होने देते हैं। जब आपको सटीक पूर्ण खोजा गया कैटलॉग चाहिए और आप प्रदाता खोज की प्रतीक्षा करने को तैयार हों, तब `--all` का उपयोग करें।
- व्यापक `models list --all` प्रदाता रनटाइम पूरक हुक लोड किए बिना मैनिफ़ेस्ट कैटलॉग पंक्तियों को रजिस्ट्री पंक्तियों के ऊपर मर्ज करता है। प्रदाता-फ़िल्टर किए गए मैनिफ़ेस्ट त्वरित पथ केवल `static` चिह्नित प्रदाताओं का उपयोग करते हैं; `refreshable` चिह्नित प्रदाता रजिस्ट्री/कैश-समर्थित रहते हैं और मैनिफ़ेस्ट पंक्तियाँ पूरक के रूप में जोड़ते हैं, जबकि `runtime` चिह्नित प्रदाता रजिस्ट्री/रनटाइम खोज पर बने रहते हैं।
- `models list` मूल मॉडल मेटाडेटा और रनटाइम सीमाओं को अलग रखता है। तालिका आउटपुट में, जब प्रभावी रनटाइम सीमा मूल संदर्भ विंडो से अलग होती है, तब `Ctx` में `contextTokens/contextWindow` दिखाई देता है; प्रदाता द्वारा वह सीमा उपलब्ध कराने पर JSON पंक्तियों में `contextTokens` शामिल होता है।
- प्रदाता-स्वामित्व वाले रूट के लिए, `models list` एक तार्किक प्रदाता/मॉडल पंक्ति को चयनित रूट पर प्रक्षेपित करता है। `Input` और `Ctx` केवल सटीक भौतिक-रूट कैटलॉग पंक्ति से आते हैं, और स्पष्ट रूप से कॉन्फ़िगर किए गए तार्किक ओवरराइड अंत में लागू होते हैं; अनसुलझा रूट चयन सहोदर-रूट मेटाडेटा उधार लेने के बजाय अज्ञात क्षमता फ़ील्ड दिखाता है।
- `models list --provider <id>` प्रदाता आईडी के अनुसार फ़िल्टर करता है, जैसे `moonshot` या `openai`। यह संवादात्मक प्रदाता चयनकर्ताओं के प्रदर्शन लेबल, जैसे `Moonshot AI`, स्वीकार नहीं करता।
- मॉडल संदर्भों को **पहले** `/` पर विभाजित करके पार्स किया जाता है। यदि मॉडल आईडी में `/` (OpenRouter-शैली) शामिल है, तो प्रदाता उपसर्ग शामिल करें (उदाहरण: `openrouter/moonshotai/kimi-k2`)।
- यदि आप प्रदाता छोड़ देते हैं, तो OpenClaw पहले इनपुट का समाधान उपनाम के रूप में, फिर उस सटीक मॉडल आईडी के अद्वितीय कॉन्फ़िगर-प्रदाता मिलान के रूप में करता है, और केवल उसके बाद बहिष्करण चेतावनी के साथ कॉन्फ़िगर किए गए डिफ़ॉल्ट प्रदाता पर वापस आता है। यदि वह प्रदाता अब कॉन्फ़िगर किया गया डिफ़ॉल्ट मॉडल उपलब्ध नहीं कराता, तो OpenClaw किसी पुराने हटाए गए-प्रदाता डिफ़ॉल्ट को दिखाने के बजाय पहले कॉन्फ़िगर किए गए प्रदाता/मॉडल पर वापस आता है।
- `models status` गैर-गोपनीय प्लेसहोल्डर (उदाहरण के लिए `OPENAI_API_KEY`, `secretref-managed`, `minimax-oauth`, `oauth:chutes`, `ollama-local`) को सीक्रेट के रूप में छिपाने के बजाय प्रमाणीकरण आउटपुट में `marker(<value>)` दिखा सकता है।

### डिफ़ॉल्ट / छवि मॉडल सेट करें

```bash
openclaw models set <model-or-alias>
openclaw models set-image <model-or-alias>
```

`set`, `agents.defaults.model.primary` लिखता है; `set-image`, `agents.defaults.imageModel.primary` लिखता है। दोनों `provider/model` या कॉन्फ़िगर किया गया उपनाम स्वीकार करते हैं। जब नए चयनित मॉडल को इसकी आवश्यकता हो, तब `set` Codex/Copilot रनटाइम Plugin इंस्टॉलेशन की मरम्मत भी करता है; `set-image` ऐसा नहीं करता। कोई भी कमांड `--agent` स्वीकार नहीं करता; वे हमेशा एजेंट डिफ़ॉल्ट लिखते हैं।

### स्कैन

`models scan`, OpenRouter का सार्वजनिक `:free` कैटलॉग पढ़ता है और फ़ॉलबैक उपयोग के लिए उम्मीदवारों को रैंक करता है। कैटलॉग स्वयं सार्वजनिक है, इसलिए केवल-मेटाडेटा स्कैन के लिए OpenRouter कुंजी की आवश्यकता नहीं होती।

डिफ़ॉल्ट रूप से OpenClaw लाइव मॉडल कॉल के साथ टूल और छवि समर्थन की जाँच करने का प्रयास करता है। यदि कोई OpenRouter कुंजी कॉन्फ़िगर नहीं है, तो कमांड केवल-मेटाडेटा आउटपुट पर वापस आता है और बताता है कि `:free` मॉडल को जाँच और अनुमान के लिए फिर भी `OPENROUTER_API_KEY` की आवश्यकता होती है।

विकल्प:

- `--no-probe` (केवल मेटाडेटा; कोई कॉन्फ़िगरेशन/सीक्रेट खोज नहीं)
- `--min-params <b>`
- `--max-age-days <days>`
- `--provider <name>`
- `--max-candidates <n>`
- `--timeout <ms>` (कैटलॉग अनुरोध और प्रत्येक जाँच की समय-सीमा)
- `--concurrency <n>`
- `--yes`
- `--no-input`
- `--set-default`
- `--set-image`
- `--json`

`--set-default` और `--set-image` के लिए लाइव जाँच आवश्यक हैं; केवल-मेटाडेटा स्कैन परिणाम सूचनात्मक होते हैं और कॉन्फ़िगरेशन पर लागू नहीं किए जाते।

## उपनाम

```bash
openclaw models aliases list [--json] [--plain]
openclaw models aliases add <alias> <model-or-alias>
openclaw models aliases remove <alias>
```

उपनाम प्रत्येक मॉडल प्रविष्टि में `agents.defaults.models.<key>.alias` के रूप में संग्रहीत किए जाते हैं। `add`, पहले `<model-or-alias>` का समाधान प्रामाणिक प्रदाता/मॉडल कुंजी में करता है, इसलिए किसी उपनाम को उपनाम देने पर शृंखला बनाने के बजाय उसका लक्ष्य बदल जाता है।
उपनाम जोड़ने से `agents.defaults.modelPolicy.allow` नहीं बदलता या मॉडल ओवरराइड प्रतिबंधित नहीं होते।

## फ़ॉलबैक

```bash
openclaw models fallbacks list [--json] [--plain]
openclaw models fallbacks add <model-or-alias>
openclaw models fallbacks remove <model-or-alias>
openclaw models fallbacks clear
```

`agents.defaults.model.fallbacks` को प्रबंधित करता है। `openclaw models image-fallbacks list|add|remove|clear` समान उपकमांड संरचना वाली समानांतर `agents.defaults.imageModel.fallbacks` सूची को प्रबंधित करता है।

## प्रमाणीकरण प्रोफ़ाइलें

```bash
openclaw models auth add
openclaw models auth list [--provider <id>] [--json]
openclaw models auth login --provider <id>
openclaw models auth login --provider openai --profile-id openai:work
openclaw models auth login-github-copilot
openclaw models auth paste-api-key --provider <id>
openclaw models auth setup-token --provider <id>
openclaw models auth paste-token --provider <id>
openclaw models auth order get --provider <id>
openclaw models auth order set --provider <id> <profileIds...>
openclaw models auth order clear --provider <id>
```

`models auth add` इंटरैक्टिव प्रमाणीकरण सहायक है। यह आपके चुने हुए प्रदाता के आधार पर प्रदाता प्रमाणीकरण प्रवाह (OAuth/API कुंजी) शुरू कर सकता है या मैन्युअल रूप से टोकन पेस्ट करने में आपका मार्गदर्शन कर सकता है।

`models auth list` टोकन, API-कुंजी या OAuth की गुप्त सामग्री प्रिंट किए बिना चयनित एजेंट के सहेजे गए प्रमाणीकरण प्रोफ़ाइल सूचीबद्ध करता है। किसी एक प्रदाता, जैसे `openai`, के अनुसार फ़िल्टर करने के लिए `--provider <id>` और स्क्रिप्टिंग के लिए `--json` का उपयोग करें।

`models auth login` प्रदाता Plugin का प्रमाणीकरण प्रवाह (OAuth/API कुंजी) चलाता है। कौन-से प्रदाता इंस्टॉल हैं, यह देखने के लिए `openclaw plugins list` का उपयोग करें। लॉगिन के दौरान नामित प्रोफ़ाइल का समर्थन करने वाले प्रदाताओं के लिए `login` में `--profile-id <id>` स्वीकार किया जाता है (एक ही प्रदाता के कई लॉगिन अलग रखने के लिए इसका उपयोग करें), किसी विशिष्ट प्रमाणीकरण विधि को चुनने के लिए `--method <id>`, `--method device-code` के शॉर्टकट के रूप में `--device-code`, प्रदाता का अनुशंसित डिफ़ॉल्ट मॉडल लागू करने के लिए `--set-default`, और पहले उस प्रदाता की मौजूदा प्रोफ़ाइल हटाने के लिए `--force` स्वीकार किया जाता है (जब कैश किया गया OAuth प्रोफ़ाइल अटक गया हो या आप खाते बदलना चाहते हों, तब इसका उपयोग करें)।

`models auth login-github-copilot`, `models auth login --provider github-copilot --method device` (GitHub डिवाइस प्रवाह) का शॉर्टकट है; यह बिना संकेत दिए मौजूदा प्रोफ़ाइल को अधिलेखित करने के लिए `--yes` स्वीकार करता है।

प्रमाणीकरण परिणामों को किसी विशिष्ट कॉन्फ़िगर किए गए एजेंट स्टोर में लिखने के लिए `openclaw models auth --agent <id> <subcommand>` का उपयोग करें। मूल `--agent` फ़्लैग का पालन `add`, `list`, `login`, `paste-api-key`, `setup-token`, `paste-token`, `login-github-copilot`, और `order get`/`set`/`clear` द्वारा किया जाता है।

OpenAI मॉडल के लिए, `--provider openai` डिफ़ॉल्ट रूप से ChatGPT/Codex खाता लॉगिन का उपयोग करता है। `--method api-key` का उपयोग केवल तब करें, जब आप OpenAI API-कुंजी प्रोफ़ाइल जोड़ना चाहते हों, सामान्यतः Codex सदस्यता सीमाओं के लिए बैकअप के रूप में। पुराने विरासती OpenAI Codex उपसर्ग वाले प्रमाणीकरण/प्रोफ़ाइल स्टेट को `openai` में माइग्रेट करने के लिए `openclaw doctor --fix` चलाएँ।

उदाहरण:

```bash
openclaw models auth login --provider openai --set-default
openclaw models auth login --provider openai --method api-key
openclaw models auth paste-api-key --provider openai
openclaw models auth list --provider openai
```

टिप्पणियाँ:

- `paste-api-key` अन्यत्र जनरेट की गई API कुंजियाँ स्वीकार करता है, कुंजी मान के लिए संकेत देता है, और जब तक आप `--profile-id` पास नहीं करते, उसे डिफ़ॉल्ट प्रोफ़ाइल आईडी `<provider>:manual` में लिखता है। स्वचालन में कुंजी को stdin पर पाइप करें, उदाहरण के लिए `printf "%s\n" "$OPENAI_API_KEY" | openclaw models auth paste-api-key --provider openai`।
- `setup-token` और `paste-token` उन प्रदाताओं के लिए सामान्य टोकन कमांड बने रहते हैं, जो टोकन प्रमाणीकरण विधियाँ उपलब्ध कराते हैं।
- `setup-token` के लिए इंटरैक्टिव TTY आवश्यक है और यह प्रदाता की टोकन-प्रमाणीकरण विधि चलाता है (यदि प्रदाता `setup-token` विधि उपलब्ध कराता है, तो डिफ़ॉल्ट रूप से उसी का उपयोग होता है)।
- `paste-token` के लिए `--provider` आवश्यक है, यह डिफ़ॉल्ट रूप से टोकन मान के लिए संकेत देता है, और जब तक आप `--profile-id` पास नहीं करते, उसे डिफ़ॉल्ट प्रोफ़ाइल आईडी `<provider>:manual` में लिखता है। स्वचालन में टोकन को तर्क के रूप में पास करने के बजाय stdin पर पाइप करें, ताकि प्रदाता क्रेडेंशियल शेल इतिहास या प्रक्रिया सूचियों में दिखाई न दें।
- `paste-token --expires-in <duration>`, `365d` या `12h` जैसी सापेक्ष अवधि से टोकन की निरपेक्ष समाप्ति संग्रहीत करता है।
- `openai` के लिए, OpenAI API कुंजियाँ और ChatGPT/OAuth टोकन सामग्री अलग-अलग प्रमाणीकरण संरचनाएँ हैं। `sk-...` OpenAI API कुंजियों के लिए `paste-api-key` और केवल टोकन प्रमाणीकरण सामग्री के लिए `paste-token` का उपयोग करें।
- Anthropic: `anthropic` के लिए `setup-token`/`paste-token`, OpenClaw के समर्थित प्रमाणीकरण पथ हैं, लेकिन उपलब्ध होने पर OpenClaw होस्ट पर Claude CLI (`claude -p`) का पुनः उपयोग करना पसंद करता है।
- `auth order get/set/clear` किसी एक प्रदाता के लिए प्रति-एजेंट प्रमाणीकरण प्रोफ़ाइल क्रम ओवरराइड प्रबंधित करता है, जो `auth-state.json` में संग्रहीत होता है (`auth.order.<provider>` कॉन्फ़िगरेशन कुंजी से अलग)। `set` प्राथमिकता क्रम में एक या अधिक प्रोफ़ाइल आईडी लेता है; `clear` कॉन्फ़िगरेशन/राउंड-रॉबिन क्रम पर वापस जाता है।
