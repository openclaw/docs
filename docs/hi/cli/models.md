# `openclaw models`

मॉडल खोज, स्कैनिंग और कॉन्फ़िगरेशन (डिफ़ॉल्ट मॉडल, फ़ॉलबैक, प्रमाणीकरण प्रोफ़ाइल)।

संबंधित:

- प्रोवाइडर + मॉडल: [मॉडल](/hi/providers/models)
- मॉडल चयन की अवधारणाएँ + `/models` स्लैश कमांड: [मॉडल की अवधारणा](/hi/concepts/models)
- प्रोवाइडर प्रमाणीकरण सेटअप: [शुरू करना](/hi/start/getting-started)

## सामान्य कमांड

```bash
openclaw models status
openclaw models list
openclaw models set <model-or-alias>
openclaw models set-image <model-or-alias>
openclaw models scan
```

`status` और `auth` उपकमांड किसी कॉन्फ़िगर किए गए एजेंट को लक्षित करने के लिए `--agent <id>` स्वीकार करते हैं; `list`, `scan`, `aliases`, और `fallbacks`/`image-fallbacks` हमेशा कॉन्फ़िगर किए गए डिफ़ॉल्ट एजेंट का उपयोग करते हैं, और `set`/`set-image` सीधे तौर पर `--agent` को अस्वीकार करते हैं। इसे छोड़ने पर, `--agent`-सक्षम कमांड, यदि `OPENCLAW_AGENT_DIR` सेट है तो उसका उपयोग करते हैं, अन्यथा कॉन्फ़िगर किए गए डिफ़ॉल्ट एजेंट का।

### स्थिति

`openclaw models status` निर्धारित डिफ़ॉल्ट/फ़ॉलबैक के साथ प्रमाणीकरण का अवलोकन दिखाता है। प्रोवाइडर उपयोग स्नैपशॉट उपलब्ध होने पर, OAuth/API-कुंजी स्थिति अनुभाग में प्रोवाइडर उपयोग अवधियाँ और कोटा स्नैपशॉट शामिल होते हैं। वर्तमान उपयोग-अवधि प्रोवाइडर: Anthropic, GitHub Copilot, Gemini CLI, OpenAI, MiniMax, Xiaomi, और z.ai। उपलब्ध होने पर उपयोग प्रमाणीकरण प्रोवाइडर-विशिष्ट हुक से आता है; अन्यथा OpenClaw प्रमाणीकरण प्रोफ़ाइल, एनवायरनमेंट या कॉन्फ़िगरेशन से मेल खाने वाले OAuth/API-कुंजी क्रेडेंशियल पर फ़ॉलबैक करता है।

`--json` आउटपुट में, `auth.providers` एनवायरनमेंट/कॉन्फ़िगरेशन/स्टोर-सक्षम प्रोवाइडर अवलोकन है, जबकि `auth.oauth` केवल प्रमाणीकरण-स्टोर प्रोफ़ाइल की स्थिति है।

विकल्प:

| फ़्लैग                      | प्रभाव                                                                                                        |
| ------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `--json`                  | JSON आउटपुट; प्रमाणीकरण-प्रोफ़ाइल, प्रोवाइडर और स्टार्टअप निदान stderr पर जाते हैं, ताकि stdout को `jq` में पाइप किया जा सके। |
| `--plain`                 | सादा टेक्स्ट आउटपुट।                                                                                            |
| `--check`                 | प्रमाणीकरण की अवधि समाप्त होने वाली हो/समाप्त हो चुकी हो तो गैर-शून्य स्थिति के साथ बाहर निकलता है: `1` = समाप्त/अनुपलब्ध, `2` = अवधि समाप्त होने वाली।                             |
| `--probe`                 | कॉन्फ़िगर की गई प्रमाणीकरण प्रोफ़ाइल की लाइव जाँच। वास्तविक अनुरोध; टोकन का उपयोग हो सकता है और दर सीमाएँ सक्रिय हो सकती हैं।            |
| `--probe-provider <name>` | केवल एक प्रोवाइडर की जाँच।                                                                                      |
| `--probe-profile <id>`    | विशिष्ट प्रमाणीकरण प्रोफ़ाइल आईडी की जाँच (दोहराकर या अल्पविराम से अलग करके)।                                                  |
| `--probe-timeout <ms>`    | प्रत्येक जाँच की समय-सीमा।                                                                                            |
| `--probe-concurrency <n>` | समवर्ती जाँच।                                                                                            |
| `--probe-max-tokens <n>`  | जाँच के अधिकतम टोकन (सर्वोत्तम प्रयास)।                                                                               |
| `--agent <id>`            | कॉन्फ़िगर किया गया एजेंट आईडी; `OPENCLAW_AGENT_DIR` को ओवरराइड करता है।                                                          |

जाँच पंक्तियाँ प्रमाणीकरण प्रोफ़ाइल, एनवायरनमेंट क्रेडेंशियल या `models.json` से आ सकती हैं। जाँच स्थिति श्रेणियाँ: `ok`, `auth`, `rate_limit`, `billing`, `timeout`, `format`, `unknown`, `no_model`।

जब कोई जाँच मॉडल कॉल तक कभी नहीं पहुँचती, तब अपेक्षित जाँच विवरण/कारण कोड:

- `excluded_by_auth_order`: एक संग्रहीत प्रोफ़ाइल मौजूद है, लेकिन स्पष्ट `auth.order.<provider>` ने उसे छोड़ दिया है, इसलिए जाँच उसे आज़माने के बजाय बहिष्करण की सूचना देती है।
- `missing_credential`, `invalid_expires`, `expired`, `unresolved_ref`: प्रोफ़ाइल मौजूद है, लेकिन पात्र नहीं है या उसका निर्धारण नहीं किया जा सकता।
- `ineligible_profile`: प्रोफ़ाइल किसी अन्य कारण से प्रोवाइडर कॉन्फ़िगरेशन के साथ असंगत है।
- `no_model`: प्रोवाइडर प्रमाणीकरण मौजूद है, लेकिन OpenClaw उस प्रोवाइडर के लिए जाँच योग्य मॉडल उम्मीदवार निर्धारित नहीं कर सका।

OpenAI ChatGPT/Codex OAuth समस्या निवारण के लिए, `openclaw models status`, `openclaw models auth list --provider openai`, और `openclaw config get agents.defaults.model --json` यह पुष्टि करने के सबसे तेज़ तरीके हैं कि किसी एजेंट के पास नेटिव Codex रनटाइम के माध्यम से `openai/*` के लिए उपयोग योग्य `openai` OAuth प्रोफ़ाइल है या नहीं। [OpenAI प्रोवाइडर सेटअप](/hi/providers/openai#check-and-recover-codex-oauth-routing) देखें।

### सूची

`openclaw models list` केवल पढ़ने के लिए है: यह कॉन्फ़िगरेशन, प्रमाणीकरण प्रोफ़ाइल, मौजूदा कैटलॉग स्थिति और प्रोवाइडर-स्वामित्व वाली कैटलॉग पंक्तियाँ पढ़ता है, लेकिन `models.json` को कभी दोबारा नहीं लिखता।

विकल्प: `--all` (पूर्ण कैटलॉग), `--local` (स्थानीय मॉडल तक सीमित करें), `--provider <id>`, `--json`, `--plain`।

टिप्पणियाँ:

- `Auth` कॉलम केवल पढ़ने के लिए है। OpenAI जैसे प्रोवाइडर-स्वामित्व वाले मॉडल रूट के लिए, यह प्रत्येक पंक्ति के API/बेस-URL रूट का प्रभावी `auth.order`, एनवायरनमेंट/कॉन्फ़िगरेशन क्रेडेंशियल और निर्धारित कमांड-स्कोप वाले SecretRefs में पात्र प्रोफ़ाइल से मिलान करता है। किसी ठोस OpenAI पंक्ति की रूट नीति अनुपलब्ध होने पर वह प्रोवाइडर-स्तरीय प्रमाणीकरण उधार लेने के बजाय अज्ञात रहती है; केवल प्रोवाइडर वाली पुरानी जाँच और अन्य प्रोवाइडर, प्रोवाइडर-स्तरीय व्यवहार बनाए रखते हैं। Plugin का सिंथेटिक-प्रमाणीकरण मेटाडेटा केवल रनटाइम-क्षमता का संकेत है, नेटिव अकाउंट प्रमाणीकरण का प्रमाण नहीं, इसलिए सकारात्मक रजिस्ट्री प्रमाण के बिना अकाउंट-निर्भर रूट अज्ञात रहते हैं। यह कमांड प्रोवाइडर रनटाइम लोड नहीं करता, कीचेन सीक्रेट नहीं पढ़ता, प्रोवाइडर API को कॉल नहीं करता और सटीक निष्पादन तत्परता प्रमाणित नहीं करता।
- `models list --all --provider <id>` में Plugin मैनिफ़ेस्ट या बंडल किए गए प्रोवाइडर कैटलॉग मेटाडेटा की प्रोवाइडर-स्वामित्व वाली स्थिर कैटलॉग पंक्तियाँ शामिल हो सकती हैं, भले ही आपने अभी तक उस प्रोवाइडर के साथ प्रमाणीकरण न किया हो। मेल खाने वाला प्रमाणीकरण कॉन्फ़िगर होने तक वे पंक्तियाँ फिर भी अनुपलब्ध दिखाई देती हैं।
- प्रोवाइडर कैटलॉग खोज धीमी होने पर `models list` कंट्रोल प्लेन को प्रतिक्रियाशील बनाए रखता है। थोड़े समय तक प्रतीक्षा करने के बाद डिफ़ॉल्ट और कॉन्फ़िगर किए गए दृश्य कॉन्फ़िगर या सिंथेटिक मॉडल पंक्तियों पर फ़ॉलबैक करते हैं और खोज को पृष्ठभूमि में पूरा होने देते हैं। जब आपको सटीक पूर्ण खोजा गया कैटलॉग चाहिए और आप प्रोवाइडर खोज की प्रतीक्षा करने को तैयार हों, तब `--all` का उपयोग करें।
- व्यापक `models list --all` प्रोवाइडर रनटाइम पूरक हुक लोड किए बिना मैनिफ़ेस्ट कैटलॉग पंक्तियों को रजिस्ट्री पंक्तियों के ऊपर मर्ज करता है। प्रोवाइडर-फ़िल्टर किए गए मैनिफ़ेस्ट तेज़ पथ केवल `static` चिह्नित प्रोवाइडर का उपयोग करते हैं; `refreshable` चिह्नित प्रोवाइडर रजिस्ट्री/कैश-समर्थित रहते हैं और पूरक के रूप में मैनिफ़ेस्ट पंक्तियाँ जोड़ते हैं, जबकि `runtime` चिह्नित प्रोवाइडर रजिस्ट्री/रनटाइम खोज पर बने रहते हैं।
- `models list` नेटिव मॉडल मेटाडेटा और रनटाइम सीमाओं को अलग रखता है। तालिका आउटपुट में, जब प्रभावी रनटाइम सीमा नेटिव कॉन्टेक्स्ट विंडो से अलग होती है, तब `Ctx`, `contextTokens/contextWindow` दिखाता है; यदि कोई प्रोवाइडर वह सीमा उपलब्ध कराता है, तो JSON पंक्तियों में `contextTokens` शामिल होता है।
- प्रोवाइडर-स्वामित्व वाले रूट के लिए, `models list` एक तार्किक प्रोवाइडर/मॉडल पंक्ति को चुने गए रूट पर प्रक्षेपित करता है। `Input` और `Ctx` केवल सटीक भौतिक-रूट कैटलॉग पंक्ति से आते हैं, और स्पष्ट कॉन्फ़िगर किए गए तार्किक ओवरराइड अंत में लागू होते हैं; अनिर्धारित रूट चयन सहोदर-रूट मेटाडेटा उधार लेने के बजाय अज्ञात क्षमता फ़ील्ड दिखाता है।
- `models list --provider <id>` प्रोवाइडर आईडी के अनुसार फ़िल्टर करता है, जैसे `moonshot` या `openai`। यह इंटरैक्टिव प्रोवाइडर चयनकर्ताओं के प्रदर्शन लेबल स्वीकार नहीं करता, जैसे `Moonshot AI`।
- मॉडल संदर्भों को **पहले** `/` पर विभाजित करके पार्स किया जाता है। यदि मॉडल आईडी में `/` (OpenRouter-शैली) शामिल है, तो प्रोवाइडर उपसर्ग शामिल करें (उदाहरण: `openrouter/moonshotai/kimi-k2`)।
- यदि आप प्रोवाइडर छोड़ देते हैं, तो OpenClaw पहले इनपुट को उपनाम के रूप में, फिर उस सटीक मॉडल आईडी के लिए अद्वितीय कॉन्फ़िगर-प्रोवाइडर मिलान के रूप में निर्धारित करता है, और उसके बाद ही अप्रचलन चेतावनी के साथ कॉन्फ़िगर किए गए डिफ़ॉल्ट प्रोवाइडर पर फ़ॉलबैक करता है। यदि वह प्रोवाइडर अब कॉन्फ़िगर किया गया डिफ़ॉल्ट मॉडल उपलब्ध नहीं कराता, तो OpenClaw पुराने हटाए गए-प्रोवाइडर डिफ़ॉल्ट को दिखाने के बजाय पहले कॉन्फ़िगर किए गए प्रोवाइडर/मॉडल पर फ़ॉलबैक करता है।
- `models status` गैर-सीक्रेट प्लेसहोल्डर (उदाहरण के लिए `OPENAI_API_KEY`, `secretref-managed`, `minimax-oauth`, `oauth:chutes`, `ollama-local`) को सीक्रेट के रूप में छिपाने के बजाय प्रमाणीकरण आउटपुट में `marker(<value>)` दिखा सकता है।

### डिफ़ॉल्ट / इमेज मॉडल सेट करें

```bash
openclaw models set <model-or-alias>
openclaw models set-image <model-or-alias>
```

`set`, `agents.defaults.model.primary` लिखता है; `set-image`, `agents.defaults.imageModel.primary` लिखता है। दोनों `provider/model` या कॉन्फ़िगर किया गया उपनाम स्वीकार करते हैं। नया चयनित मॉडल आवश्यक होने पर `set` Codex/Copilot रनटाइम Plugin इंस्टॉलेशन की मरम्मत भी करता है; `set-image` ऐसा नहीं करता। कोई भी कमांड `--agent` स्वीकार नहीं करता; वे हमेशा एजेंट डिफ़ॉल्ट लिखते हैं।

### स्कैन

`models scan`, OpenRouter का सार्वजनिक `:free` कैटलॉग पढ़ता है और फ़ॉलबैक उपयोग के लिए उम्मीदवारों को क्रम देता है। कैटलॉग स्वयं सार्वजनिक है, इसलिए केवल-मेटाडेटा स्कैन के लिए OpenRouter कुंजी की आवश्यकता नहीं होती।

डिफ़ॉल्ट रूप से OpenClaw लाइव मॉडल कॉल के साथ टूल और इमेज समर्थन की जाँच करने का प्रयास करता है। यदि कोई OpenRouter कुंजी कॉन्फ़िगर नहीं है, तो कमांड केवल-मेटाडेटा आउटपुट पर फ़ॉलबैक करता है और समझाता है कि `:free` मॉडल को जाँच और अनुमान के लिए अभी भी `OPENROUTER_API_KEY` की आवश्यकता है।

विकल्प:

- `--no-probe` (केवल मेटाडेटा; कोई कॉन्फ़िगरेशन/सीक्रेट लुकअप नहीं)
- `--min-params <b>`
- `--max-age-days <days>`
- `--provider <name>`
- `--max-candidates <n>`
- `--timeout <ms>` (कैटलॉग अनुरोध और प्रत्येक जाँच की समय-सीमा)
- `--concurrency <n>`
- `--yes`
- `--no-input`
- `--set-default`
- `--set-image`
- `--json`

`--set-default` और `--set-image` के लिए लाइव जाँच आवश्यक है; केवल-मेटाडेटा स्कैन परिणाम सूचनात्मक होते हैं और कॉन्फ़िगरेशन पर लागू नहीं किए जाते।

## उपनाम

```bash
openclaw models aliases list [--json] [--plain]
openclaw models aliases add <alias> <model-or-alias>
openclaw models aliases remove <alias>
```

उपनाम प्रत्येक मॉडल प्रविष्टि के अनुसार `agents.defaults.models.<key>.alias` के रूप में संग्रहीत किए जाते हैं। `add` पहले `<model-or-alias>` को कैनोनिकल प्रोवाइडर/मॉडल कुंजी में निर्धारित करता है, इसलिए किसी उपनाम को उपनाम देने पर शृंखला बनाने के बजाय उसे पुनः इंगित किया जाता है।

## फ़ॉलबैक

```bash
openclaw models fallbacks list [--json] [--plain]
openclaw models fallbacks add <model-or-alias>
openclaw models fallbacks remove <model-or-alias>
openclaw models fallbacks clear
```

`agents.defaults.model.fallbacks` का प्रबंधन करता है। `openclaw models image-fallbacks list|add|remove|clear` समान उपकमांड संरचना वाली समानांतर `agents.defaults.imageModel.fallbacks` सूची का प्रबंधन करता है।

## प्रमाणीकरण प्रोफ़ाइल

```bash
openclaw models auth add
openclaw models auth list [--provider <id>] [--json]
openclaw models auth login --provider <id>
openclaw models auth login --provider openai --profile-id openai:work
openclaw models auth login-github-copilot
openclaw models auth paste-api-key --provider <id>
openclaw models auth setup-token --provider <id>
openclaw models auth paste-token --provider <id>
openclaw models auth order get --provider <id>
openclaw models auth order set --provider <id> <profileIds...>
openclaw models auth order clear --provider <id>
```

`models auth add` इंटरैक्टिव प्रमाणीकरण सहायक है। आपके चुने हुए प्रोवाइडर के आधार पर, यह प्रोवाइडर प्रमाणीकरण प्रवाह (OAuth/API कुंजी) शुरू कर सकता है या टोकन को मैन्युअल रूप से पेस्ट करने में आपका मार्गदर्शन कर सकता है।

`models auth list` चयनित एजेंट के सहेजे गए प्रमाणीकरण प्रोफ़ाइलों को टोकन, API-कुंजी या OAuth गोपनीय सामग्री प्रिंट किए बिना सूचीबद्ध करता है। केवल एक प्रदाता, जैसे `openai`, के अनुसार फ़िल्टर करने के लिए `--provider <id>` और स्क्रिप्टिंग के लिए `--json` का उपयोग करें।

`models auth login` किसी प्रदाता Plugin का प्रमाणीकरण प्रवाह (OAuth/API कुंजी) चलाता है। यह देखने के लिए कि कौन-से प्रदाता इंस्टॉल हैं, `openclaw plugins list` का उपयोग करें। `login` लॉगिन के दौरान नामित प्रोफ़ाइलों का समर्थन करने वाले प्रदाताओं के लिए `--profile-id <id>` स्वीकार करता है (एक ही प्रदाता के कई लॉगिन अलग रखने के लिए इसका उपयोग करें), किसी विशिष्ट प्रमाणीकरण विधि को चुनने के लिए `--method <id>`, `--method device-code` के शॉर्टकट के रूप में `--device-code`, प्रदाता का अनुशंसित डिफ़ॉल्ट मॉडल लागू करने के लिए `--set-default`, और पहले उस प्रदाता की मौजूदा प्रोफ़ाइलें हटाने के लिए `--force` स्वीकार करता है (जब कैश किया हुआ OAuth प्रोफ़ाइल अटक गया हो या आप खाता बदलना चाहते हों, तब इसका उपयोग करें)।

`models auth login-github-copilot`, `models auth login --provider github-copilot --method device` (GitHub डिवाइस प्रवाह) का शॉर्टकट है; यह बिना संकेत दिए मौजूदा प्रोफ़ाइल को अधिलेखित करने के लिए `--yes` स्वीकार करता है।

प्रमाणीकरण परिणामों को किसी विशिष्ट कॉन्फ़िगर किए गए एजेंट स्टोर में लिखने के लिए `openclaw models auth --agent <id> <subcommand>` का उपयोग करें। मूल `--agent` फ़्लैग का पालन `add`, `list`, `login`, `paste-api-key`, `setup-token`, `paste-token`, `login-github-copilot`, और `order get`/`set`/`clear` द्वारा किया जाता है।

OpenAI मॉडल के लिए, `--provider openai` डिफ़ॉल्ट रूप से ChatGPT/Codex खाता लॉगिन का उपयोग करता है। केवल तब `--method api-key` का उपयोग करें, जब आप OpenAI API-कुंजी प्रोफ़ाइल जोड़ना चाहते हों, आम तौर पर Codex सदस्यता सीमाओं के लिए बैकअप के रूप में। पुराने विरासती OpenAI Codex उपसर्ग वाले प्रमाणीकरण/प्रोफ़ाइल स्थिति को `openai` में माइग्रेट करने के लिए `openclaw doctor --fix` चलाएँ।

उदाहरण:

```bash
openclaw models auth login --provider openai --set-default
openclaw models auth login --provider openai --method api-key
openclaw models auth paste-api-key --provider openai
openclaw models auth list --provider openai
```

टिप्पणियाँ:

- `paste-api-key` कहीं और जनरेट की गई API कुंजियाँ स्वीकार करता है, कुंजी मान के लिए संकेत देता है, और जब तक आप `--profile-id` न दें, इसे डिफ़ॉल्ट प्रोफ़ाइल आईडी `<provider>:manual` में लिखता है। स्वचालन में, कुंजी को stdin पर पाइप करें, उदाहरण के लिए `printf "%s\n" "$OPENAI_API_KEY" | openclaw models auth paste-api-key --provider openai`।
- `setup-token` और `paste-token` उन प्रदाताओं के लिए सामान्य टोकन कमांड बने हुए हैं, जो टोकन प्रमाणीकरण विधियाँ उपलब्ध कराते हैं।
- `setup-token` के लिए इंटरैक्टिव TTY आवश्यक है और यह प्रदाता की टोकन-प्रमाणीकरण विधि चलाता है (यदि प्रदाता `setup-token` विधि उपलब्ध कराता है, तो डिफ़ॉल्ट रूप से उसी का उपयोग होता है)।
- `paste-token` के लिए `--provider` आवश्यक है, यह डिफ़ॉल्ट रूप से टोकन मान के लिए संकेत देता है, और जब तक आप `--profile-id` न दें, इसे डिफ़ॉल्ट प्रोफ़ाइल आईडी `<provider>:manual` में लिखता है। स्वचालन में, टोकन को तर्क के रूप में देने के बजाय stdin पर पाइप करें, ताकि प्रदाता क्रेडेंशियल शेल इतिहास या प्रक्रिया सूचियों में दिखाई न दें।
- `paste-token --expires-in <duration>`, `365d` या `12h` जैसी सापेक्ष अवधि से टोकन की निरपेक्ष समाप्ति संग्रहीत करता है।
- `openai` के लिए, OpenAI API कुंजियाँ और ChatGPT/OAuth टोकन सामग्री अलग-अलग प्रमाणीकरण संरचनाएँ हैं। `sk-...` OpenAI API कुंजियों के लिए `paste-api-key` और केवल टोकन प्रमाणीकरण सामग्री के लिए `paste-token` का उपयोग करें।
- Anthropic: `setup-token`/`paste-token`, `anthropic` के लिए समर्थित OpenClaw प्रमाणीकरण पथ हैं, लेकिन उपलब्ध होने पर OpenClaw होस्ट पर Claude CLI (`claude -p`) का पुनः उपयोग करना पसंद करता है।
- `auth order get/set/clear` एक प्रदाता के लिए प्रति-एजेंट प्रमाणीकरण प्रोफ़ाइल क्रम ओवरराइड प्रबंधित करता है, जिसे `auth-state.json` में संग्रहीत किया जाता है (`auth.order.<provider>` कॉन्फ़िगरेशन कुंजी से अलग)। `set` प्राथमिकता क्रम में एक या अधिक प्रोफ़ाइल आईडी लेता है; `clear` कॉन्फ़िगरेशन/राउंड-रॉबिन क्रम पर वापस जाता है।
