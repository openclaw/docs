Plugins, OpenClaw को चैनलों, मॉडल प्रदाताओं, एजेंट हार्नेस, टूल,
Skills, वाक्, रीयलटाइम ट्रांसक्रिप्शन, आवाज़, मीडिया की समझ, जनरेशन,
वेब फ़ेच, वेब खोज और अन्य रनटाइम क्षमताओं के साथ विस्तारित करते हैं।

Plugin इंस्टॉल करने, Gateway पुनः आरंभ करने, रनटाइम द्वारा उसे लोड किए जाने की
पुष्टि करने और सामान्य सेटअप विफलताओं का समाधान करने के लिए इस पृष्ठ का उपयोग करें।
केवल कमांड वाले उदाहरणों के लिए, [Plugins प्रबंधित करें](/hi/plugins/manage-plugins) देखें।
बंडल किए गए, आधिकारिक बाहरी और केवल-स्रोत Plugins की जनरेट की गई सूची के लिए,
[Plugin सूची](/hi/plugins/plugin-inventory) देखें।

## आवश्यकताएँ

- एक OpenClaw चेकआउट या इंस्टॉलेशन जिसमें `openclaw` CLI उपलब्ध हो
- चयनित स्रोत (ClawHub, npm या git होस्ट) तक नेटवर्क पहुँच
- उस Plugin के सेटअप दस्तावेज़ों में बताए गए Plugin-विशिष्ट क्रेडेंशियल,
  कॉन्फ़िगरेशन कुंजियाँ या OS टूल
- आपके चैनलों को सेवा देने वाले Gateway को पुनः लोड या पुनः आरंभ करने की अनुमति

## त्वरित शुरुआत

<Steps>
  <Step title="Plugin खोजें">
    सार्वजनिक Plugin पैकेजों के लिए [ClawHub](/clawhub) खोजें:

    ```bash
    openclaw plugins search "calendar"
    ```

    सामुदायिक Plugins खोजने के लिए ClawHub प्राथमिक माध्यम है। लॉन्च
    संक्रमण के दौरान, सामान्य उपसर्ग-रहित पैकेज विनिर्देश तब तक npm से इंस्टॉल
    होते हैं, जब तक वे किसी आधिकारिक Plugin id से मेल न खाएँ। बंडल किए गए
    Plugin से मेल खाने वाले अपरिष्कृत `@openclaw/*` विनिर्देश उसी बंडल की गई
    प्रति पर रिज़ॉल्व होते हैं। जब किसी एक स्रोत की विशेष रूप से आवश्यकता हो,
    तो स्पष्ट स्रोत उपसर्ग का उपयोग करें।

  </Step>

  <Step title="Plugin इंस्टॉल करें">
    ```bash
    # ClawHub से।
    openclaw plugins install clawhub:<package>

    # npm से।
    openclaw plugins install npm:<package>

    # git से।
    openclaw plugins install git:github.com/<owner>/<repo>@<ref>

    # स्थानीय डेवलपमेंट चेकआउट से।
    openclaw plugins install ./my-plugin
    openclaw plugins install --link ./my-plugin
    ```

    Plugin इंस्टॉलेशन को कोड चलाने जैसा मानें। पुनरुत्पाद्य प्रोडक्शन
    इंस्टॉलेशन के लिए पिन किए गए संस्करणों को प्राथमिकता दें। ClawHub पैकेज
    और OpenClaw का बंडल/आधिकारिक कैटलॉग विश्वसनीय स्रोत हैं। नए मनमाने npm,
    git, स्थानीय पथ/आर्काइव, `npm-pack:` या मार्केटप्लेस स्रोतों के लिए,
    स्रोत की समीक्षा करके उस पर भरोसा करने के बाद गैर-इंटरैक्टिव इंस्टॉलेशन में
    `--force` आवश्यक है।

  </Step>

  <Step title="इसे कॉन्फ़िगर और सक्षम करें">
    Plugin-विशिष्ट सेटिंग्स को `plugins.entries.<id>.config` के अंतर्गत कॉन्फ़िगर करें।
    यदि Plugin पहले से सक्षम नहीं है, तो उसे सक्षम करें:

    ```bash
    openclaw plugins enable <plugin-id>
    ```

    यदि `plugins.allow` सेट है, तो Plugin के लोड होने से पहले इंस्टॉल किया गया
    Plugin id उस सूची में होना चाहिए। `openclaw plugins install`, इंस्टॉल किए गए
    id को मौजूदा `plugins.allow` सूची में जोड़ता है और उसी id को
    `plugins.deny` से हटाता है, ताकि स्पष्ट इंस्टॉलेशन पुनः आरंभ करने के बाद
    लोड हो सके।

  </Step>

  <Step title="Gateway को पुनः लोड होने दें">
    Plugin कोड को इंस्टॉल, अपडेट या अनइंस्टॉल करने के लिए Gateway को
    पुनः आरंभ करना आवश्यक है। कॉन्फ़िगरेशन रीलोड सक्षम वाला प्रबंधित Gateway,
    बदले हुए Plugin इंस्टॉल रिकॉर्ड का पता लगाता है और स्वतः पुनः आरंभ हो जाता है।
    अन्यथा, इसे स्वयं पुनः आरंभ करें:

    ```bash
    openclaw gateway restart
    ```

    सक्षम/अक्षम करने से कॉन्फ़िगरेशन और कोल्ड रजिस्ट्री अपडेट होते हैं। सक्रिय
    रनटाइम सतहों का सबसे स्पष्ट प्रमाण अब भी रनटाइम निरीक्षण है।

  </Step>

  <Step title="रनटाइम पंजीकरण सत्यापित करें">
    ```bash
    openclaw plugins inspect <plugin-id> --runtime --json
    ```

    पंजीकृत टूल, हुक, सेवाएँ, Gateway विधियाँ या Plugin के स्वामित्व वाले CLI
    कमांड प्रमाणित करने के लिए `--runtime` का उपयोग करें। सामान्य
    `inspect` केवल कोल्ड मेनिफ़ेस्ट और रजिस्ट्री जाँच है।

  </Step>
</Steps>

## कॉन्फ़िगरेशन

### इंस्टॉल स्रोत चुनें

| स्रोत        | कब उपयोग करें                                                                  | उदाहरण                                                        |
| ----------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| ClawHub     | आपको OpenClaw-मूल खोज, स्कैन, संस्करण मेटाडेटा और इंस्टॉल संकेत चाहिए          | `openclaw plugins install clawhub:<package>`                   |
| npm         | आपको सीधे npm रजिस्ट्री या dist-tag वर्कफ़्लो चाहिए                            | `openclaw plugins install npm:<package>`                       |
| git         | आपको किसी रिपॉज़िटरी से ब्रांच, टैग या कमिट चाहिए                              | `openclaw plugins install git:github.com/<owner>/<repo>@<ref>` |
| स्थानीय पथ | आप उसी मशीन पर किसी Plugin को विकसित या परीक्षण कर रहे हैं                     | `openclaw plugins install --link ./my-plugin`                  |
| मार्केटप्लेस | आप Claude-संगत मार्केटप्लेस Plugin इंस्टॉल कर रहे हैं                          | `openclaw plugins install <plugin> --marketplace <source>`     |

उपसर्ग-रहित पैकेज विनिर्देशों में विशेष संगतता व्यवहार होता है: बंडल किए गए
Plugin id से मेल खाने वाला उपसर्ग-रहित नाम उस बंडल किए गए स्रोत का उपयोग करता है;
आधिकारिक बाहरी Plugin id से मेल खाने वाला उपसर्ग-रहित नाम आधिकारिक पैकेज
कैटलॉग का उपयोग करता है; लॉन्च संक्रमण के दौरान कोई भी अन्य उपसर्ग-रहित
विनिर्देश npm के माध्यम से इंस्टॉल होता है। बंडल किए गए Plugins से मेल खाने वाले
अपरिष्कृत `@openclaw/*` विनिर्देश भी npm फ़ॉलबैक से पहले बंडल की गई प्रति
पर रिज़ॉल्व होते हैं। बंडल की गई प्रति के बजाय जानबूझकर बाहरी npm पैकेज इंस्टॉल
करने के लिए `npm:@openclaw/<plugin>@<version>` का उपयोग करें। निर्धारक स्रोत चयन के लिए
`clawhub:`, `npm:`, `git:` या
`npm-pack:` का उपयोग करें। पूर्ण कमांड अनुबंध के लिए
[`openclaw plugins`](/hi/cli/plugins#install) देखें।

npm इंस्टॉलेशन के लिए, बिना पिन किए विनिर्देश और `@latest` उस नवीनतम
स्थिर पैकेज को चुनते हैं जो इस OpenClaw बिल्ड के साथ संगतता घोषित करता है।
यदि npm की वर्तमान नवीनतम रिलीज़ इस बिल्ड द्वारा समर्थित संस्करण से नया
`openclaw.compat.pluginApi` या `openclaw.install.minHostVersion` घोषित करती है, तो OpenClaw
पुराने स्थिर संस्करणों को स्कैन करता है और उपयुक्त नवीनतम संस्करण इंस्टॉल करता
है। सटीक संस्करण और `@beta` जैसे स्पष्ट चैनल टैग चयनित पैकेज पर
पिन रहते हैं और असंगत होने पर विफल होते हैं।

### ऑपरेटर इंस्टॉल नीति

Plugin इंस्टॉलेशन या अपडेट आगे बढ़ने से पहले किसी विश्वसनीय स्थानीय नीति
कमांड को चलाने के लिए `security.installPolicy` कॉन्फ़िगर करें। नीति को मेटाडेटा
और स्टेज किया गया स्रोत पथ मिलता है तथा वह इंस्टॉलेशन को अनुमति दे या रोक
सकती है। यह CLI और Gateway-समर्थित इंस्टॉल/अपडेट दोनों पथों को कवर करती है।
Plugin `before_install` हुक बाद में और केवल उन OpenClaw प्रक्रियाओं में चलते
हैं जहाँ Plugin हुक लोड होते हैं, इसलिए इसके बजाय ऑपरेटर-स्वामित्व वाले इंस्टॉल
निर्णयों के लिए `security.installPolicy` का उपयोग करें। अप्रचलित
`--dangerously-force-unsafe-install` फ़्लैग संगतता के लिए स्वीकार किया जाता है, लेकिन निष्प्रभावी
है: यह इंस्टॉल नीति या OpenClaw की अंतर्निहित Plugin निर्भरता निषेध-सूची को
बायपास नहीं करता।

Skills और Plugins दोनों द्वारा उपयोग की जाने वाली साझा `security.installPolicy`
exec स्कीमा के लिए
[Skills कॉन्फ़िगरेशन](/hi/tools/skills-config#operator-install-policy-securityinstallpolicy)
देखें।

### Plugin नीति कॉन्फ़िगर करें

सामान्य Plugin कॉन्फ़िगरेशन संरचना है:

```json5
{
  plugins: {
    enabled: true,
    allow: ["voice-call"],
    deny: ["untrusted-plugin"],
    load: { paths: ["~/Projects/oss/voice-call-plugin"] },
    slots: { memory: "memory-core" },
    entries: {
      "voice-call": { enabled: true, config: { provider: "twilio" } },
    },
  },
}
```

मुख्य नीति नियम:

- `plugins.enabled: false` सभी Plugins अक्षम करता है और खोज/लोड
  कार्य छोड़ देता है। इसके सक्रिय रहते पुराने Plugin संदर्भ निष्क्रिय रहते हैं;
  यदि पुराने ids हटवाने हैं, तो doctor क्लीनअप चलाने से पहले Plugins को पुनः
  सक्षम करें।
- `plugins.deny`, अनुमति और प्रति-Plugin सक्षमकरण पर
  प्राथमिकता पाता है।
- `plugins.allow` एक विशिष्ट अनुमति-सूची है। अनुमति-सूची
  से बाहर Plugin-स्वामित्व वाले टूल तब भी अनुपलब्ध रहते हैं, जब
  `tools.allow` में `"*"` शामिल हो।
- `plugins.entries.<id>.enabled: false` किसी एक Plugin का कॉन्फ़िगरेशन बनाए रखते
  हुए उसे अक्षम करता है।
- `plugins.load.paths` स्पष्ट स्थानीय Plugin फ़ाइलें या
  डायरेक्टरियाँ जोड़ता है। प्रबंधित `plugins install` स्थानीय पथ Plugin
  डायरेक्टरियाँ या आर्काइव होने चाहिए; स्वतंत्र Plugin फ़ाइलों के लिए
  `plugins.load.paths` का उपयोग करें।
- वर्कस्पेस-मूल Plugins डिफ़ॉल्ट रूप से अक्षम होते हैं; स्थानीय
  वर्कस्पेस कोड का उपयोग करने से पहले उन्हें स्पष्ट रूप से सक्षम करें या
  अनुमति-सूची में जोड़ें।
- बंडल किए गए Plugins अपने अंतर्निहित डिफ़ॉल्ट-चालू/डिफ़ॉल्ट-बंद
  मेटाडेटा का पालन करते हैं, जब तक कॉन्फ़िगरेशन उसे स्पष्ट रूप से ओवरराइड न करे।
- `plugins.slots.<slot>` (`memory` या
  `contextEngine`) किसी विशिष्ट श्रेणी के लिए एक Plugin चुनता है। स्लॉट चयन
  स्पष्ट सक्रियण माना जाता है और चयनित Plugin को उस स्लॉट के लिए बलपूर्वक सक्षम
  करता है, भले ही वह अन्यथा ऑप्ट-इन हो। `plugins.deny` और
  `plugins.entries.<id>.enabled: false` फिर भी उसे रोकते हैं।
- बंडल किए गए ऑप्ट-इन Plugins स्वतः सक्रिय हो सकते हैं, जब
  कॉन्फ़िगरेशन उनके स्वामित्व वाली किसी सतह का नाम देता है, जैसे प्रदाता/मॉडल
  संदर्भ, चैनल कॉन्फ़िगरेशन, CLI बैकएंड या एजेंट हार्नेस रनटाइम।
- OpenAI-परिवार का Codex रूटिंग प्रदाता और रनटाइम Plugin
  सीमाओं को अलग रखता है: पुराने Codex मॉडल संदर्भ पुराने कॉन्फ़िगरेशन हैं जिन्हें
  doctor सुधारता है, जबकि बंडल किया गया `codex` Plugin मानक
  `openai/*` एजेंट संदर्भों, स्पष्ट `agentRuntime.id: "codex"` और पुराने
  `codex/*` संदर्भों के लिए Codex ऐप-सर्वर रनटाइम का स्वामी है।

जब `plugins.allow` सेट नहीं होता और गैर-बंडल Plugins को वर्कस्पेस या वैश्विक
Plugin रूट से स्वतः खोजा जाता है, तो स्टार्टअप खोजे गए Plugin ids के साथ
`plugins.allow is empty; discovered non-bundled plugins may auto-load: ...` लॉग करता है और छोटी सूचियों के लिए न्यूनतम
`plugins.allow` स्निपेट भी देता है। विश्वसनीय Plugins को
`openclaw.json` में कॉपी करने से पहले सूचीबद्ध Plugin id पर
[`openclaw plugins list --enabled --verbose`](/hi/cli/plugins#list) या
[`openclaw plugins inspect <id>`](/hi/cli/plugins#inspect) चलाएँ। यही भरोसा-पिनिंग तब भी लागू
होती है, जब निदान बताता है कि कोई Plugin `without install/load-path provenance` लोड हुआ: उस Plugin
id का निरीक्षण करें, फिर उसे `plugins.allow` में पिन करें या किसी विश्वसनीय
स्रोत से पुनः इंस्टॉल करें, ताकि OpenClaw इंस्टॉल उद्गम रिकॉर्ड कर सके।

जब कॉन्फ़िगरेशन सत्यापन पुराने Plugin ids, अनुमति-सूची/टूल असंगतियाँ या पुराने
बंडल Plugin पथ रिपोर्ट करे, तब `openclaw doctor` या
`openclaw doctor --fix` चलाएँ।

## Plugin प्रारूप समझें

OpenClaw दो Plugin प्रारूपों को पहचानता है:

| प्रारूप                  | यह कैसे लोड होता है                                                        | कब उपयोग करें                                                          |
| ---------------------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| मूल OpenClaw Plugin    | `openclaw.plugin.json` और प्रक्रिया में लोड किया गया रनटाइम मॉड्यूल            | आप OpenClaw-विशिष्ट रनटाइम क्षमताएँ इंस्टॉल या निर्मित कर रहे हैं       |
| संगत बंडल              | OpenClaw Plugin सूची में मैप किया गया Codex, Claude या Cursor Plugin लेआउट | आप संगत Skills, कमांड, हुक या बंडल मेटाडेटा का पुनः उपयोग कर रहे हैं   |

दोनों प्रारूप `openclaw plugins list`, `openclaw plugins inspect`,
`openclaw plugins enable` और `openclaw plugins disable` में दिखाई देते हैं। बंडल संगतता सीमा
के लिए [Plugin बंडल](/hi/plugins/bundles) और मूल Plugin लेखन के लिए
[Plugins बनाना](/hi/plugins/building-plugins) देखें।

## Plugin हुक

Plugins दो अलग API के माध्यम से रनटाइम पर हुक पंजीकृत कर सकते हैं:

- `api.on(...)` रनटाइम जीवनचक्र घटनाओं के लिए टाइप किए
  गए हुक। मिडलवेयर, नीति, संदेश पुनर्लेखन, प्रॉम्प्ट आकार निर्धारण और टूल
  नियंत्रण के लिए यह पसंदीदा सतह है।
- [हुक](/hi/automation/hooks) में वर्णित आंतरिक हुक प्रणाली के लिए
  `api.registerHook(...)`। यह मुख्यतः मोटे कमांड/जीवनचक्र दुष्प्रभावों और मौजूदा
  HOOK-शैली ऑटोमेशन के साथ संगतता के लिए है।

त्वरित नियम: यदि हैंडलर को प्राथमिकता, मर्ज अर्थविज्ञान या रोकने/रद्द करने के
व्यवहार की आवश्यकता है, तो टाइप किए गए हुक का उपयोग करें। यदि वह केवल
`command:new`, `command:reset`, `message:sent` या ऐसी ही मोटी
घटनाओं पर प्रतिक्रिया करता है, तो `api.registerHook` उपयुक्त है।

Plugin-प्रबंधित आंतरिक हुक `openclaw hooks list` में
`plugin:<id>` के साथ दिखाई देते हैं। आप उन्हें `openclaw hooks` के
माध्यम से सक्षम या अक्षम नहीं कर सकते; इसके बजाय Plugin को सक्षम या अक्षम करें।

## सक्रिय Gateway सत्यापित करें

`openclaw plugins list` और सामान्य `openclaw plugins inspect` कोल्ड कॉन्फ़िगरेशन,
मैनिफ़ेस्ट और रजिस्ट्री स्थिति पढ़ते हैं। वे यह प्रमाणित नहीं करते कि पहले से चल रहे
Gateway ने वही Plugin कोड आयात किया है।

जब कोई Plugin इंस्टॉल किया हुआ दिखाई देता है, लेकिन लाइव चैट ट्रैफ़िक उसका उपयोग नहीं करता:

```bash
openclaw gateway status --deep --require-rpc
openclaw plugins inspect <plugin-id> --runtime --json
openclaw gateway restart
```

Plugin के इंस्टॉल, अपडेट और अनइंस्टॉल से Plugin स्रोत में बदलाव होने पर प्रबंधित Gateway
अपने आप रीस्टार्ट होते हैं। VPS या कंटेनर इंस्टॉलेशन पर सुनिश्चित करें कि कोई भी मैन्युअल
रीस्टार्ट केवल किसी रैपर या सुपरवाइज़र को नहीं, बल्कि आपके चैनलों को सेवा देने वाले वास्तविक
`openclaw gateway run` चाइल्ड को लक्षित करे।

| `OPENCLAW_NIX_MODE=1` लाइफ़साइकल कमांड अवरुद्ध करता है                | पुष्टि करें कि इंस्टॉलेशन Nix द्वारा प्रबंधित है                                                                                                      | Plugin परिवर्तक कमांड का उपयोग करने के बजाय Nix स्रोत में Plugin चयन बदलें                      |
| रनटाइम पर डिपेंडेंसी आयात विफल होता है                             | जाँचें कि Plugin npm/git/ClawHub के माध्यम से इंस्टॉल किया गया था या किसी स्थानीय पथ से लोड हुआ था                                                 | `openclaw plugins update <id>` चलाएँ, स्रोत को फिर से इंस्टॉल करें या स्थानीय Plugin डिपेंडेंसियाँ स्वयं इंस्टॉल करें |

जब पुराने Plugin कॉन्फ़िगरेशन में अब खोजे न जा सकने वाले चैनल Plugin का नाम मौजूद रहता है,
तो कॉन्फ़िगरेशन सत्यापन उस चैनल कुंजी को गंभीर विफलता के बजाय चेतावनी में बदल देता है,
जिससे Gateway स्टार्टअप अन्य सभी चैनलों को सेवा देना जारी रख सकता है। पुरानी Plugin और
चैनल प्रविष्टियाँ हटाने के लिए `openclaw doctor --fix` चलाएँ। पुराने Plugin के प्रमाण के बिना
अज्ञात चैनल कुंजियाँ सत्यापन में अब भी विफल होती हैं, ताकि टाइपो दिखाई देते रहें।

जानबूझकर चैनल प्रतिस्थापन के लिए, पसंदीदा Plugin को लीगेसी या निम्न-प्राथमिकता वाले
Plugin आईडी के साथ `channelConfigs.<channel-id>.preferOver` घोषित करना चाहिए।
यदि दोनों Plugin स्पष्ट रूप से सक्षम हैं, तो OpenClaw उस अनुरोध को बनाए रखता है और चुपचाप
एक स्वामी चुनने के बजाय डुप्लिकेट चैनल/टूल डायग्नोस्टिक्स की रिपोर्ट करता है।

यदि कोई इंस्टॉल किया गया पैकेज रिपोर्ट करता है कि वह `requires compiled runtime output for
TypeScript entry ...`, तो पैकेज को उन JavaScript फ़ाइलों के बिना प्रकाशित किया गया था
जिनकी OpenClaw को रनटाइम पर आवश्यकता होती है। प्रकाशक द्वारा संकलित JavaScript जारी करने
के बाद अपडेट या पुनः इंस्टॉल करें, या तब तक Plugin को अक्षम/अनइंस्टॉल करें।

### अवरुद्ध Plugin पथ स्वामित्व

यदि डायग्नोस्टिक्स में
`blocked plugin candidate: suspicious ownership (... uid=1000, expected uid=0 or root)`
कहा जाता है और उसके बाद सत्यापन में `plugin present but blocked` आता है, तो OpenClaw को
ऐसी Plugin फ़ाइलें मिली हैं जिनका स्वामी उन्हें लोड करने वाली प्रक्रिया से अलग Unix उपयोगकर्ता है।
Plugin कॉन्फ़िगरेशन को यथावत रखें; फ़ाइल सिस्टम स्वामित्व ठीक करें या OpenClaw को उसी
उपयोगकर्ता के रूप में चलाएँ जो स्थिति डायरेक्टरी का स्वामी है।

Docker इंस्टॉलेशन के लिए, आधिकारिक इमेज `node` (uid `1000`) के रूप में चलती है, इसलिए
होस्ट पर बाइंड-माउंट की गई OpenClaw कॉन्फ़िगरेशन और वर्कस्पेस डायरेक्टरियों का स्वामी
सामान्यतः uid `1000` होना चाहिए:

```bash
sudo chown -R 1000:1000 /path/to/openclaw-config /path/to/openclaw-workspace
```

यदि आप जानबूझकर OpenClaw को root के रूप में चलाते हैं, तो इसके बजाय प्रबंधित Plugin रूट का
स्वामित्व root पर सेट करें:

```bash
sudo chown -R root:root /path/to/openclaw-config/npm
```

स्वामित्व ठीक करने के बाद, `openclaw doctor --fix` या
`openclaw plugins registry --refresh` फिर से चलाएँ, ताकि स्थायी Plugin रजिस्ट्री
सुधारी गई फ़ाइलों से मेल खाए।

### धीमा Plugin टूल सेटअप

यदि टूल तैयार करते समय एजेंट टर्न रुके हुए दिखाई देते हैं, तो ट्रेस लॉगिंग सक्षम करें
और Plugin टूल फ़ैक्टरी की टाइमिंग पंक्तियाँ जाँचें:

```bash
openclaw config set logging.level trace
openclaw logs --follow
```

यह खोजें:

```text
[trace:plugin-tools] फ़ैक्टरी टाइमिंग ...
```

सारांश में कुल फ़ैक्टरी समय और सबसे धीमी Plugin टूल फ़ैक्टरियाँ सूचीबद्ध होती हैं,
जिनमें Plugin आईडी, घोषित टूल नाम, परिणाम का आकार और टूल के वैकल्पिक होने की जानकारी
शामिल होती है। जब किसी एक फ़ैक्टरी में कम से कम 1s लगता है या Plugin टूल फ़ैक्टरी की
कुल तैयारी में कम से कम 5s लगते हैं, तो धीमी पंक्तियों को चेतावनियों में पदोन्नत किया जाता है।

समान प्रभावी अनुरोध संदर्भ वाले दोहराए गए रिज़ॉल्यूशन के लिए OpenClaw सफल Plugin टूल
फ़ैक्टरी परिणामों को कैश करता है। कैश कुंजी में प्रभावी रनटाइम कॉन्फ़िगरेशन, वर्कस्पेस और
एजेंट आईडी, सैंडबॉक्स नीति, ब्राउज़र सेटिंग, डिलीवरी संदर्भ, अनुरोधकर्ता की पहचान और
स्वामित्व स्थिति शामिल हैं, इसलिए इन विश्वसनीय फ़ील्ड पर निर्भर फ़ैक्टरियाँ संदर्भ बदलने
पर फिर से चलती हैं। यदि टाइमिंग अधिक बनी रहती है, तो संभव है कि Plugin अपनी टूल
परिभाषाएँ लौटाने से पहले महँगा कार्य कर रहा हो।

यदि कोई एक Plugin अधिकांश समय ले रहा है, तो उसके रनटाइम पंजीकरणों का निरीक्षण करें:

```bash
openclaw plugins inspect <plugin-id> --runtime --json
```

फिर उस Plugin को अपडेट, पुनः इंस्टॉल या अक्षम करें। Plugin लेखकों को महँगी डिपेंडेंसी
लोडिंग टूल फ़ैक्टरी के भीतर करने के बजाय टूल निष्पादन पथ के पीछे ले जानी चाहिए।

डिपेंडेंसी रूट, पैकेज मेटाडेटा सत्यापन, रजिस्ट्री रिकॉर्ड, स्टार्टअप रीलोड व्यवहार और
लीगेसी क्लीनअप के लिए [Plugin डिपेंडेंसी रिज़ॉल्यूशन](/hi/plugins/dependency-resolution) देखें।
