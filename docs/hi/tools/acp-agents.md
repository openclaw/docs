[Agent Client Protocol (ACP)](https://agentclientprotocol.com/) सत्रों से
OpenClaw किसी ACP बैकएंड Plugin के माध्यम से बाहरी कोडिंग हार्नेस (Claude Code, Cursor, Copilot, Droid,
OpenClaw ACP, OpenCode, Gemini CLI और अन्य समर्थित ACPX हार्नेस)
चला सकता है। प्रत्येक स्पॉन को
[पृष्ठभूमि कार्य](/hi/automation/tasks) के रूप में ट्रैक किया जाता है।

<Note>
**ACP बाहरी-हार्नेस पथ है, डिफ़ॉल्ट Codex पथ नहीं।** नेटिव
Codex ऐप-सर्वर Plugin एजेंट टर्न के लिए `/codex ...` नियंत्रणों और डिफ़ॉल्ट
`openai/gpt-*` एम्बेडेड रनटाइम का स्वामी है; ACP `/acp ...` नियंत्रणों
और `sessions_spawn({ runtime: "acp" })` सत्रों का स्वामी है।

Codex या Claude Code को मौजूदा OpenClaw चैनल वार्तालापों से
सीधे बाहरी MCP क्लाइंट के रूप में कनेक्ट करने के लिए ACP के बजाय
[`openclaw mcp serve`](/hi/cli/mcp) का उपयोग करें।
</Note>

## क्या यह बिना अतिरिक्त सेटअप के काम करता है?

हाँ, आधिकारिक ACP रनटाइम Plugin इंस्टॉल करने के बाद:

```bash
openclaw plugins install @openclaw/acpx
openclaw config set plugins.entries.acpx.enabled true
```

सोर्स चेकआउट `pnpm install` के बाद स्थानीय `extensions/acpx` वर्कस्पेस Plugin का उपयोग कर सकते हैं।
तत्परता जाँच के लिए `/acp doctor` चलाएँ।

OpenClaw एजेंटों को ACP स्पॉनिंग के बारे में केवल तभी बताता है, जब ACP **वास्तव में उपयोग योग्य** हो:
ACP सक्षम होना चाहिए, डिस्पैच अक्षम नहीं होना चाहिए, वर्तमान सत्र
सैंडबॉक्स द्वारा अवरुद्ध नहीं होना चाहिए और कोई रनटाइम बैकएंड लोड तथा स्वस्थ होना चाहिए। यदि
कोई भी शर्त विफल होती है, तो ACP Skills और `sessions_spawn` ACP मार्गदर्शन छिपे रहते हैं,
ताकि एजेंट अनुपलब्ध बैकएंड का सुझाव न दे।

OpenClaw Plugin टूल और अंतर्निहित OpenClaw टूल डिफ़ॉल्ट रूप से ACP
हार्नेस को उपलब्ध **नहीं** कराए जाते। स्पष्ट MCP ब्रिज केवल तभी
[ACP एजेंट—सेटअप](/hi/tools/acp-agents-setup) में सक्षम करें, जब हार्नेस को
उन टूल को सीधे कॉल करना हो।

## समर्थित हार्नेस लक्ष्य

`acpx` बैकएंड के साथ, इन आईडी को `/acp spawn <id>` या
`sessions_spawn({ runtime: "acp", agentId: "<id>" })` लक्ष्य के रूप में उपयोग करें:

| हार्नेस आईडी  | सामान्य बैकएंड                                 | टिप्पणियाँ                                                                          |
| ------------ | ---------------------------------------------- | ----------------------------------------------------------------------------------- |
| `claude`     | Claude Code ACP अडैप्टर                        | होस्ट पर Claude Code प्रमाणीकरण आवश्यक है।                                           |
| `codex`      | Codex ACP अडैप्टर                              | केवल तब स्पष्ट ACP फ़ॉलबैक, जब नेटिव `/codex` अनुपलब्ध हो या ACP का अनुरोध किया गया हो। |
| `copilot`    | GitHub Copilot ACP अडैप्टर                     | Copilot CLI/रनटाइम प्रमाणीकरण आवश्यक है।                                             |
| `cursor`     | Cursor CLI ACP (`cursor-agent acp`)            | यदि स्थानीय इंस्टॉल कोई अलग ACP प्रवेश-बिंदु उपलब्ध कराता है, तो acpx कमांड को ओवरराइड करें। |
| `droid`      | Factory Droid CLI                              | हार्नेस परिवेश में Factory/Droid प्रमाणीकरण या `FACTORY_API_KEY` आवश्यक है।          |
| `fast-agent` | fast-agent-mcp ACP अडैप्टर                     | `uvx` के साथ माँग के अनुसार प्राप्त किया जाता है।                       |
| `gemini`     | Gemini CLI ACP अडैप्टर                         | Gemini CLI प्रमाणीकरण या API कुंजी सेटअप आवश्यक है।                                  |
| `iflow`      | iFlow CLI                                      | अडैप्टर की उपलब्धता और मॉडल नियंत्रण इंस्टॉल किए गए CLI पर निर्भर करते हैं।           |
| `kilocode`   | Kilo Code CLI                                  | अडैप्टर की उपलब्धता और मॉडल नियंत्रण इंस्टॉल किए गए CLI पर निर्भर करते हैं।           |
| `kimi`       | Kimi/Moonshot CLI                              | होस्ट पर Kimi/Moonshot प्रमाणीकरण आवश्यक है।                                         |
| `kiro`       | Kiro CLI                                       | अडैप्टर की उपलब्धता और मॉडल नियंत्रण इंस्टॉल किए गए CLI पर निर्भर करते हैं।           |
| `mux`        | Mux CLI ACP अडैप्टर                            | `npx` के साथ माँग के अनुसार प्राप्त किया जाता है।                       |
| `opencode`   | OpenCode ACP अडैप्टर                           | OpenCode CLI/प्रदाता प्रमाणीकरण आवश्यक है।                                           |
| `openclaw`   | `openclaw acp` के माध्यम से OpenClaw Gateway ब्रिज | ACP-सक्षम हार्नेस को OpenClaw Gateway सत्र से वापस संचार करने देता है।                |
| `qoder`      | Qoder CLI                                      | अडैप्टर की उपलब्धता और मॉडल नियंत्रण इंस्टॉल किए गए CLI पर निर्भर करते हैं।           |

| `trae`       | Trae CLI ACP अडैप्टर                           | अडैप्टर की उपलब्धता और मॉडल नियंत्रण इंस्टॉल किए गए CLI पर निर्भर करते हैं।           |

`pi` (pi-acp) भी acpx बैकएंड में पंजीकृत है, लेकिन यह ऊपर दिए गए अन्य हार्नेसों के समान अर्थ में कोडिंग
हार्नेस नहीं है।

कस्टम acpx एजेंट उपनाम acpx में ही कॉन्फ़िगर किए जा सकते हैं, लेकिन OpenClaw
नीति डिस्पैच से पहले फिर भी `acp.allowedAgents` और किसी भी
`agents.list[].runtime.acp.agent` मैपिंग की जाँच करती है।

## ऑपरेटर रनबुक

चैट से त्वरित `/acp` प्रवाह:

<Steps>
  <Step title="स्पॉन करें">
    `/acp spawn claude --bind here`,
    `/acp spawn gemini --mode persistent --thread auto`, या स्पष्ट
    `/acp spawn codex --bind here`।
  </Step>
  <Step title="काम करें">
    बाइंड की गई वार्तालाप या थ्रेड में जारी रखें (या सत्र कुंजी को
    स्पष्ट रूप से लक्ष्य बनाएँ)।
  </Step>
  <Step title="स्थिति जाँचें">
    `/acp status`
  </Step>
  <Step title="समायोजित करें">
    `/acp model <provider/model>`, `/acp permissions <profile>`,
    `/acp timeout <seconds>`।
  </Step>
  <Step title="दिशा दें">
    संदर्भ को बदले बिना: `/acp steer tighten logging and continue`।
  </Step>
  <Step title="रोकें">
    `/acp cancel` (वर्तमान टर्न) या `/acp close` (सत्र + बाइंडिंग)।
  </Step>
</Steps>

  <Accordion title="जीवनचक्र विवरण">
    - स्पॉन किसी ACP रनटाइम सत्र को बनाता या फिर से शुरू करता है, OpenClaw सत्र स्टोर में ACP मेटाडेटा दर्ज करता है, और जब रन पैरेंट-स्वामित्व वाला हो तो एक पृष्ठभूमि कार्य बना सकता है।
    - पैरेंट-स्वामित्व वाले ACP सत्रों को रनटाइम सत्र के स्थायी होने पर भी पृष्ठभूमि कार्य माना जाता है; पूर्णता और विभिन्न सतहों पर डिलीवरी सामान्य उपयोगकर्ता-दृश्य चैट सत्र की तरह होने के बजाय पैरेंट कार्य सूचक के माध्यम से होती है।
    - कार्य रखरखाव टर्मिनल या अनाथ पैरेंट-स्वामित्व वाले वन-शॉट ACP सत्रों को बंद करता है। सक्रिय वार्तालाप बाइंडिंग बने रहने तक स्थायी ACP सत्र संरक्षित रहते हैं; सक्रिय बाइंडिंग के बिना पुराने स्थायी सत्र बंद कर दिए जाते हैं, ताकि स्वामी कार्य पूरा होने या उसका कार्य रिकॉर्ड हट जाने के बाद उन्हें चुपचाप फिर से शुरू न किया जा सके।

## ACP बनाम सब-एजेंट

जब आपको बाहरी हार्नेस रनटाइम चाहिए, तब ACP का उपयोग करें। `codex` Plugin
सक्षम होने पर Codex वार्तालाप बाइंडिंग/नियंत्रण के लिए **नेटिव Codex
app-server** का उपयोग करें। जब आपको OpenClaw-नेटिव प्रत्यायोजित रन चाहिए, तब **सब-एजेंट** का उपयोग करें।

| क्षेत्र          | ACP सत्र                           | सब-एजेंट रन                      |
| ------------- | ------------------------------------- | ---------------------------------- |
| रनटाइम       | ACP बैकएंड Plugin (उदाहरण के लिए acpx) | OpenClaw नेटिव सब-एजेंट रनटाइम  |
| सत्र कुंजी   | `agent:<agentId>:acp:<uuid>`          | `agent:<agentId>:subagent:<uuid>`  |
| मुख्य कमांड | `/acp ...`                            | `/subagents ...`                   |
| स्पॉन टूल    | `sessions_spawn` के साथ `runtime:"acp"` | `sessions_spawn` (डिफ़ॉल्ट रनटाइम) |

[सब-एजेंट](/hi/tools/subagents) भी देखें।

## ACP Claude Code को कैसे चलाता है

ACP के माध्यम से Claude Code के लिए स्टैक यह है:

1. OpenClaw ACP सत्र नियंत्रण प्लेन।
2. आधिकारिक `@openclaw/acpx` रनटाइम Plugin।
3. Claude ACP अडैप्टर।
4. Claude-पक्षीय रनटाइम/सत्र तंत्र।

ACP Claude, ACP नियंत्रणों, सत्र पुनः आरंभ,
पृष्ठभूमि-कार्य ट्रैकिंग और वैकल्पिक वार्तालाप/थ्रेड बाइंडिंग वाला एक **हार्नेस सत्र** है।

CLI बैकएंड अलग, केवल-टेक्स्ट वाले स्थानीय फ़ॉलबैक रनटाइम हैं - देखें
[CLI बैकएंड](/hi/gateway/cli-backends)।

ऑपरेटरों के लिए व्यावहारिक नियम यह है:

- **`/acp spawn`, बाइंड किए जा सकने वाले सत्र, रनटाइम नियंत्रण या स्थायी हार्नेस कार्य चाहिए?** ACP का उपयोग करें।
- **रॉ CLI के माध्यम से साधारण स्थानीय टेक्स्ट फ़ॉलबैक चाहिए?** CLI बैकएंड का उपयोग करें।

## बाइंड किए गए सत्र

### मानसिक मॉडल

- **चैट सतह** - जहाँ लोग बातचीत जारी रखते हैं (Discord चैनल, Telegram टॉपिक, iMessage चैट)।
- **ACP सत्र** - स्थायी Codex/Claude/Gemini रनटाइम स्थिति, जिसकी ओर OpenClaw रूट करता है।
- **चाइल्ड थ्रेड/टॉपिक** - केवल `--thread ...` द्वारा बनाई जाने वाली एक वैकल्पिक अतिरिक्त मैसेजिंग सतह।
- **रनटाइम वर्कस्पेस** - फ़ाइलसिस्टम स्थान (`cwd`, रेपो चेकआउट, बैकएंड वर्कस्पेस), जहाँ हार्नेस चलता है। चैट सतह से स्वतंत्र।

### वर्तमान-वार्तालाप बाइंड

`/acp spawn <harness> --bind here` वर्तमान वार्तालाप को स्पॉन किए गए
ACP सत्र से पिन करता है - कोई चाइल्ड थ्रेड नहीं, वही चैट सतह। OpenClaw
ट्रांसपोर्ट, प्रमाणीकरण, सुरक्षा और डिलीवरी का स्वामित्व बनाए रखता है। उस
वार्तालाप के फ़ॉलो-अप संदेश उसी सत्र की ओर रूट होते हैं; `/new` और `/reset` सत्र को
उसी स्थान पर रीसेट करते हैं; `/acp close` बाइंडिंग हटाता है।

उदाहरण:

```text
/codex bind                                              # नेटिव Codex बाइंड, भविष्य के संदेश यहाँ रूट करें
/codex model gpt-5.4                                     # बाइंड किए गए नेटिव Codex थ्रेड को समायोजित करें
/codex stop                                              # सक्रिय नेटिव Codex टर्न नियंत्रित करें
/acp spawn codex --bind here                             # Codex के लिए स्पष्ट ACP फ़ॉलबैक
/acp spawn codex --thread auto                           # चाइल्ड थ्रेड/टॉपिक बना सकता है और वहाँ बाइंड कर सकता है
/acp spawn codex --bind here --cwd /workspace/repo       # वही चैट बाइंडिंग, Codex /workspace/repo में चलता है
```

## स्थायी चैनल बाइंडिंग

गैर-अल्पकालिक वर्कफ़्लो के लिए, शीर्ष-स्तरीय
`bindings[]` प्रविष्टियों में स्थायी ACP बाइंडिंग कॉन्फ़िगर करें।

### बाइंडिंग मॉडल

<ParamField path="bindings[].type" type='"acp"'>
  स्थायी ACP वार्तालाप बाइंडिंग चिह्नित करता है।
</ParamField>
<ParamField path="bindings[].match" type="object">
  लक्षित वार्तालाप की पहचान करता है। प्रत्येक चैनल के आकार:

- **Discord चैनल/थ्रेड:** `match.channel="discord"` + `match.peer.id="<channelOrThreadId>"`
- **Slack चैनल/DM:** `match.channel="slack"` + `match.peer.id="<channelId|channel:<channelId>|#<channelId>|userId|user:<userId>|slack:<userId>|<@userId>>"`. स्थिर Slack आईडी को प्राथमिकता दें; चैनल बाइंडिंग उस चैनल के थ्रेड में दिए गए उत्तरों से भी मेल खाती हैं।
- **Telegram फ़ोरम विषय:** `match.channel="telegram"` + `match.peer.id="<chatId>:topic:<topicId>"`
- **WhatsApp DM/समूह:** `match.channel="whatsapp"` + `match.peer.id="<E.164|group JID>"`. सीधे चैट के लिए `+15555550123` जैसी E.164 संख्याओं और समूहों के लिए `120363424282127706@g.us` जैसे WhatsApp समूह JID का उपयोग करें।
- **iMessage DM/समूह:** `match.channel="imessage"` + `match.peer.id="<handle|chat_id:*|chat_guid:*|chat_identifier:*>"`. स्थिर समूह बाइंडिंग के लिए `chat_id:*` को प्राथमिकता दें।

</ParamField>
<ParamField path="bindings[].agentId" type="string">
  स्वामी OpenClaw एजेंट आईडी।
</ParamField>
<ParamField path="bindings[].acp.mode" type='"persistent" | "oneshot"'>
  वैकल्पिक ACP ओवरराइड।
</ParamField>
<ParamField path="bindings[].acp.label" type="string">
  वैकल्पिक ऑपरेटर-दृश्य लेबल।
</ParamField>
<ParamField path="bindings[].acp.cwd" type="string">
  वैकल्पिक रनटाइम कार्यशील डायरेक्टरी।
</ParamField>
<ParamField path="bindings[].acp.backend" type="string">
  वैकल्पिक बैकएंड ओवरराइड।
</ParamField>

### प्रति एजेंट रनटाइम डिफ़ॉल्ट

प्रत्येक एजेंट के लिए ACP डिफ़ॉल्ट को एक बार परिभाषित करने हेतु `agents.list[].runtime` का उपयोग करें:

- `agents.list[].runtime.type="acp"`
- `agents.list[].runtime.acp.agent` (हार्नेस आईडी, जैसे `codex` या `claude`)
- `agents.list[].runtime.acp.backend`
- `agents.list[].runtime.acp.mode`
- `agents.list[].runtime.acp.cwd`

**ACP-बाउंड सत्रों के लिए ओवरराइड प्राथमिकता:**

1. `bindings[].acp.*`
2. `agents.list[].runtime.acp.*`
3. वैश्विक ACP डिफ़ॉल्ट (जैसे `acp.backend`)

### उदाहरण

```json5
{
  agents: {
    list: [
      {
        id: "codex",
        runtime: {
          type: "acp",
          acp: {
            agent: "codex",
            backend: "acpx",
            mode: "persistent",
            cwd: "/workspace/openclaw",
          },
        },
      },
      {
        id: "claude",
        runtime: {
          type: "acp",
          acp: { agent: "claude", backend: "acpx", mode: "persistent" },
        },
      },
    ],
  },
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "discord",
        accountId: "default",
        peer: { kind: "channel", id: "222222222222222222" },
      },
      acp: { label: "codex-main" },
    },
    {
      type: "acp",
      agentId: "claude",
      match: {
        channel: "telegram",
        accountId: "default",
        peer: { kind: "group", id: "-1001234567890:topic:42" },
      },
      acp: { cwd: "/workspace/repo-b" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "discord", accountId: "default" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "telegram", accountId: "default" },
    },
  ],
  channels: {
    discord: {
      guilds: {
        "111111111111111111": {
          channels: {
            "222222222222222222": { requireMention: false },
          },
        },
      },
    },
    telegram: {
      groups: {
        "-1001234567890": {
          topics: { "42": { requireMention: false } },
        },
      },
    },
  },
}
```

### व्यवहार

- OpenClaw सुनिश्चित करता है कि चैनल-विशिष्ट प्रवेश के बाद और उपयोग से पहले कॉन्फ़िगर किया गया ACP सत्र मौजूद हो।
- उस चैनल, विषय या चैट के संदेश कॉन्फ़िगर किए गए ACP सत्र पर रूट होते हैं।
- कॉन्फ़िगर की गई ACP बाइंडिंग अपने सत्र रूट की स्वामी होती हैं। मेल खाने वाली बाइंडिंग के लिए चैनल ब्रॉडकास्ट फ़ैन-आउट कॉन्फ़िगर किए गए ACP सत्र को प्रतिस्थापित नहीं करता।
- बाउंड वार्तालापों में, `/new` और `/reset` उसी ACP सत्र कुंजी को उसी स्थान पर रीसेट करते हैं।
- अस्थायी रनटाइम बाइंडिंग (उदाहरण के लिए, थ्रेड-फ़ोकस प्रवाहों द्वारा बनाई गई) मौजूद होने पर अब भी लागू होती हैं।
- स्पष्ट `cwd` के बिना क्रॉस-एजेंट ACP स्पॉन के लिए, OpenClaw एजेंट कॉन्फ़िगरेशन से लक्ष्य एजेंट कार्यस्थान इनहेरिट करता है।
- इनहेरिट किए गए अनुपलब्ध कार्यस्थान पथ बैकएंड के डिफ़ॉल्ट cwd पर वापस चले जाते हैं; मौजूद पथ पर पहुँच विफल होने पर स्पॉन त्रुटियाँ दिखाई जाती हैं।

## ACP सत्र शुरू करें

ACP सत्र शुरू करने के दो तरीके:

<Tabs>
  <Tab title="sessions_spawn से">
    किसी एजेंट टर्न या टूल कॉल से ACP सत्र शुरू करने के लिए
    `runtime: "acp"` का उपयोग करें।

    ```json
    {
      "task": "रिपॉज़िटरी खोलें और विफल परीक्षणों का सारांश दें",
      "runtime": "acp",
      "agentId": "codex",
      "thread": true,
      "mode": "session"
    }
    ```

    <Note>
    `runtime` का डिफ़ॉल्ट `subagent` है, इसलिए ACP सत्रों के लिए
    `runtime: "acp"` स्पष्ट रूप से सेट करें। यदि `agentId` छोड़ा गया है, तो कॉन्फ़िगर होने पर
    OpenClaw `acp.defaultAgent` का उपयोग करता है। एक स्थायी बाउंड वार्तालाप बनाए
    रखने के लिए `mode: "session"` को `thread: true` की आवश्यकता होती है।
    </Note>

  </Tab>
  <Tab title="/acp कमांड से">
    चैट से स्पष्ट ऑपरेटर नियंत्रण के लिए `/acp spawn` का उपयोग करें।

    ```text
    /acp spawn codex --mode persistent --thread auto
    /acp spawn codex --mode oneshot --thread off
    /acp spawn codex --bind here
    /acp spawn codex --thread here
    ```

    प्रमुख फ़्लैग:

    - `--mode persistent|oneshot`
    - `--bind here|off`
    - `--thread auto|here|off`
    - `--cwd <absolute-path>`
    - `--label <name>`

    [स्लैश कमांड](/hi/tools/slash-commands) देखें।

  </Tab>
</Tabs>

### `sessions_spawn` पैरामीटर

<ParamField path="task" type="string" required>
  ACP सत्र को भेजा गया प्रारंभिक प्रॉम्प्ट।
</ParamField>
<ParamField path="runtime" type='"acp"' required>
  ACP सत्रों के लिए `"acp"` होना आवश्यक है।
</ParamField>
<ParamField path="agentId" type="string">
  ACP लक्ष्य हार्नेस आईडी। सेट होने पर `acp.defaultAgent` का उपयोग करता है।
</ParamField>
<ParamField path="thread" type="boolean" default="false">
  जहाँ समर्थित हो, वहाँ थ्रेड बाइंडिंग प्रवाह का अनुरोध करें।
</ParamField>
<ParamField path="mode" type='"run" | "session"' default="run">
  `"run"` एक-बार चलने वाला है; `"session"` स्थायी है। यदि `thread: true` है और
  `mode` छोड़ा गया है, तो OpenClaw रनटाइम पथ के अनुसार स्थायी व्यवहार को
  डिफ़ॉल्ट बना सकता है। `mode: "session"` के लिए `thread: true` आवश्यक है।
</ParamField>
<ParamField path="cwd" type="string">
  अनुरोधित रनटाइम कार्यशील डायरेक्टरी (बैकएंड/रनटाइम नीति द्वारा सत्यापित)।
  यदि छोड़ा गया है, तो कॉन्फ़िगर होने पर ACP स्पॉन लक्ष्य एजेंट कार्यस्थान इनहेरिट करता है;
  इनहेरिट किए गए अनुपलब्ध पथ बैकएंड डिफ़ॉल्ट पर वापस चले जाते हैं, जबकि वास्तविक पहुँच
  त्रुटियाँ लौटाई जाती हैं।
</ParamField>
<ParamField path="label" type="string">
  सत्र/बैनर टेक्स्ट में उपयोग किया जाने वाला ऑपरेटर-दृश्य लेबल।
</ParamField>
<ParamField path="resumeSessionId" type="string">
  नया ACP सत्र बनाने के बजाय मौजूदा सत्र फिर से शुरू करें। एजेंट
  `session/load` के माध्यम से अपने वार्तालाप इतिहास को दोबारा चलाता है। इसके लिए
  `runtime: "acp"` आवश्यक है।
</ParamField>
<ParamField path="streamTo" type='"parent"'>
  `"parent"` प्रारंभिक ACP रन की प्रगति के सारांशों को सिस्टम इवेंट के रूप में अनुरोधकर्ता
  सत्र में वापस स्ट्रीम करता है। OpenClaw संपूर्ण रिले इतिहास को चाइल्ड एजेंट की
  SQLite स्थिति में रिकॉर्ड करता है और चाइल्ड सत्र के साथ उसे हटा देता है। पैरेंट
  प्रगति स्ट्रीम डिफ़ॉल्ट रूप से सहायक टिप्पणी और ACP स्थिति प्रगति दिखाती हैं, जब तक कि
  `streaming.progress.commentary=false` न हो। कोई स्ट्रीम मोड कॉन्फ़िगर न होने पर Discord भी पैरेंट
  पूर्वावलोकनों को डिफ़ॉल्ट रूप से प्रगति मोड में रखता है। स्थिति
  प्रगति अब भी `acp.stream.tagVisibility` का पालन करती है, इसलिए `plan` जैसे टैग
  स्पष्ट रूप से सक्षम किए बिना छिपे रहते हैं।
</ParamField>

ACP `sessions_spawn` रन अपने डिफ़ॉल्ट चाइल्ड टर्न की सीमा के लिए
`agents.defaults.subagents.runTimeoutSeconds` का उपयोग करते हैं। टूल प्रति-कॉल टाइमआउट ओवरराइड स्वीकार नहीं करता
(`runTimeoutSeconds`/`timeoutSeconds` को डिफ़ॉल्ट कॉन्फ़िगर करने की त्रुटि के साथ अस्वीकार किया जाता है)।

<ParamField path="model" type="string">
  ACP चाइल्ड सत्र के लिए स्पष्ट मॉडल ओवरराइड। Codex ACP स्पॉन
  `session/new` से पहले `openai/gpt-5.4` जैसे OpenAI संदर्भों को Codex ACP स्टार्टअप कॉन्फ़िगरेशन में
  सामान्यीकृत करते हैं; `openai/gpt-5.4/high` जैसे स्लैश रूप Codex ACP रीजनिंग प्रयास भी सेट
  करते हैं। छोड़े जाने पर, कॉन्फ़िगर होने की स्थिति में `sessions_spawn({ runtime: "acp" })`
  मौजूदा सबएजेंट मॉडल डिफ़ॉल्ट (`agents.defaults.subagents.model` या
  `agents.list[].subagents.model`) का उपयोग करता है; अन्यथा यह ACP
  हार्नेस को अपना डिफ़ॉल्ट मॉडल उपयोग करने देता है। अन्य हार्नेस को ACP
  `models` की घोषणा करनी और `session/set_model` का समर्थन करना आवश्यक है; अन्यथा OpenClaw/acpx
  लक्ष्य एजेंट के डिफ़ॉल्ट पर चुपचाप वापस जाने के बजाय स्पष्ट रूप से विफल होता है।
</ParamField>
<ParamField path="thinking" type="string">
  स्पष्ट चिंतन/रीजनिंग प्रयास। Codex ACP के लिए, `minimal` कम
  प्रयास से मैप होता है, `low`/`medium`/`high`/`xhigh` सीधे मैप होते हैं, और `off`
  रीजनिंग-प्रयास स्टार्टअप ओवरराइड को छोड़ देता है। छोड़े जाने पर, ACP स्पॉन मौजूदा
  सबएजेंट चिंतन डिफ़ॉल्ट और चयनित मॉडल के लिए प्रति-मॉडल
  `agents.defaults.models["provider/model"].params.thinking` का उपयोग करते हैं।
</ParamField>

## स्पॉन बाइंड और थ्रेड मोड

<Tabs>
  <Tab title="--bind here|off">
    | मोड   | व्यवहार                                                               |
    | ------ | ----------------------------------------------------------------------- |
    | `here` | मौजूदा सक्रिय वार्तालाप को उसी स्थान पर बाइंड करें; कोई सक्रिय वार्तालाप न होने पर विफल हों। |
    | `off`  | मौजूदा वार्तालाप की बाइंडिंग न बनाएँ।                          |

    टिप्पणियाँ:

    - `--bind here` "इस चैनल या चैट को Codex-समर्थित बनाएँ" के लिए सबसे सरल ऑपरेटर पथ है।
    - `--bind here` चाइल्ड थ्रेड नहीं बनाता।
    - `--bind here` केवल उन चैनलों पर उपलब्ध है जो मौजूदा वार्तालाप बाइंडिंग समर्थन प्रदान करते हैं।
    - `--bind` और `--thread` को एक ही `/acp spawn` कॉल में संयोजित नहीं किया जा सकता।

  </Tab>
  <Tab title="--thread auto|here|off">
    | मोड   | व्यवहार                                                                                            |
    | ------ | ------------------------------------------------------------------------------------------------- |
    | `auto` | सक्रिय थ्रेड में: उस थ्रेड को बाइंड करें। थ्रेड के बाहर: समर्थित होने पर चाइल्ड थ्रेड बनाएँ/बाइंड करें। |
    | `here` | मौजूदा सक्रिय थ्रेड आवश्यक है; उसमें न होने पर विफल हों।                                                  |
    | `off`  | कोई बाइंडिंग नहीं। सत्र बिना बाइंडिंग के शुरू होता है।                                                                 |

    टिप्पणियाँ:

    - गैर-थ्रेड बाइंडिंग सतहों पर, डिफ़ॉल्ट व्यवहार प्रभावी रूप से `off` होता है।
    - थ्रेड-बाउंड स्पॉन के लिए चैनल नीति समर्थन आवश्यक है:
      - Discord: `channels.discord.threadBindings.spawnSessions=true`
      - Telegram: `channels.telegram.threadBindings.spawnSessions=true`
    - चाइल्ड थ्रेड बनाए बिना मौजूदा वार्तालाप को पिन करने के लिए `--bind here` का उपयोग करें।

  </Tab>
</Tabs>

## डिलीवरी मॉडल

ACP सत्र इंटरैक्टिव कार्यस्थान या पैरेंट-स्वामित्व वाला पृष्ठभूमि कार्य,
दोनों में से कोई हो सकते हैं। डिलीवरी पथ इस संरचना पर निर्भर करता है।

<AccordionGroup>
  <Accordion title="इंटरैक्टिव ACP सत्र">
    इंटरैक्टिव सत्र किसी दृश्य चैट सतह पर वार्तालाप जारी रखने के लिए होते हैं:

    - `/acp spawn ... --bind here` मौजूदा वार्तालाप को ACP सत्र से बाइंड करता है।
    - `/acp spawn ... --thread ...` किसी चैनल थ्रेड/विषय को ACP सत्र से बाइंड करता है।
    - स्थायी रूप से कॉन्फ़िगर किए गए `bindings[].type="acp"` मेल खाने वाले वार्तालापों को उसी ACP सत्र पर रूट करते हैं।

    बाउंड वार्तालाप में आने वाले अगले संदेश सीधे ACP
    सत्र पर रूट होते हैं, और ACP आउटपुट उसी
    चैनल/थ्रेड/विषय पर वापस डिलीवर किया जाता है।

    OpenClaw हार्नेस को क्या भेजता है:

    - सामान्य संबद्ध फ़ॉलो-अप प्रॉम्प्ट टेक्स्ट के रूप में भेजे जाते हैं, और अटैचमेंट केवल तभी भेजे जाते हैं जब हार्नेस/बैकएंड उनका समर्थन करता है।
    - `/acp` प्रबंधन कमांड और स्थानीय Gateway कमांड ACP डिस्पैच से पहले इंटरसेप्ट किए जाते हैं।
    - रनटाइम द्वारा जनरेट किए गए पूर्णता इवेंट प्रत्येक लक्ष्य के लिए मूर्त रूप दिए जाते हैं। OpenClaw एजेंट को OpenClaw का आंतरिक रनटाइम-कॉन्टेक्स्ट एनवेलप मिलता है; बाहरी ACP हार्नेस को चाइल्ड परिणाम और निर्देश वाला एक साधारण प्रॉम्प्ट मिलता है। अपरिष्कृत `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>` एनवेलप कभी भी बाहरी हार्नेस को नहीं भेजा जाना चाहिए या ACP उपयोगकर्ता ट्रांसक्रिप्ट टेक्स्ट के रूप में संरक्षित नहीं किया जाना चाहिए।
    - ACP ट्रांसक्रिप्ट प्रविष्टियाँ उपयोगकर्ता को दिखाई देने वाला ट्रिगर टेक्स्ट या साधारण पूर्णता प्रॉम्प्ट उपयोग करती हैं। जहाँ संभव हो, आंतरिक इवेंट मेटाडेटा OpenClaw में संरचित रहता है और उसे उपयोगकर्ता द्वारा लिखी गई चैट सामग्री नहीं माना जाता।

  </Accordion>

## सैंडबॉक्स संगतता

ACP सेशन वर्तमान में होस्ट रनटाइम पर चलते हैं, OpenClaw
सैंडबॉक्स के अंदर **नहीं**।

<Warning>
**सुरक्षा सीमा:**

- बाहरी हार्नेस अपनी CLI अनुमतियों और चयनित `cwd` के अनुसार पढ़/लिख सकता है।
- OpenClaw की सैंडबॉक्स नीति ACP हार्नेस निष्पादन को आवृत **नहीं** करती।
- OpenClaw अब भी ACP फ़ीचर गेट, अनुमत एजेंट, सेशन स्वामित्व, चैनल बाइंडिंग और Gateway डिलीवरी नीति लागू करता है।
- सैंडबॉक्स द्वारा लागू OpenClaw-नेटिव कार्य के लिए `runtime: "subagent"` का उपयोग करें।

</Warning>

वर्तमान सीमाएँ:

- यदि अनुरोधकर्ता सेशन सैंडबॉक्स में है, तो `sessions_spawn({ runtime: "acp" })` और `/acp spawn` दोनों के लिए ACP स्पॉन अवरुद्ध होते हैं।
- `runtime: "acp"` के साथ `sessions_spawn`, `sandbox: "require"` का समर्थन नहीं करता।

## सेशन लक्ष्य समाधान

अधिकांश `/acp` क्रियाएँ वैकल्पिक सेशन लक्ष्य (`session-key`,
`session-id` या `session-label`) स्वीकार करती हैं।

**समाधान क्रम:**

1. स्पष्ट लक्ष्य आर्ग्युमेंट (या `/acp steer` के लिए `--session`)
   - पहले कुंजी आज़माता है
   - फिर UUID-आकार की सेशन आईडी
   - फिर लेबल
2. वर्तमान थ्रेड बाइंडिंग (यदि यह वार्तालाप/थ्रेड किसी ACP सेशन से बंधा है)।
3. वर्तमान अनुरोधकर्ता सेशन फ़ॉलबैक।

वर्तमान-वार्तालाप बाइंडिंग और थ्रेड बाइंडिंग दोनों चरण 2 में भाग लेती हैं।

यदि कोई लक्ष्य हल नहीं होता, तो OpenClaw स्पष्ट त्रुटि
(`Unable to resolve session target: ...`) लौटाता है।

## ACP नियंत्रण

| कमांड              | यह क्या करता है                                              | उदाहरण                                                       |
| -------------------- | --------------------------------------------------------- | ------------------------------------------------------------- |
| `/acp spawn`         | ACP सेशन बनाएँ; वैकल्पिक वर्तमान बाइंड या थ्रेड बाइंड। | `/acp spawn codex --bind here --cwd /repo`                    |
| `/acp cancel`        | लक्ष्य सेशन का चल रहा टर्न रद्द करें।                 | `/acp cancel agent:codex:acp:<uuid>`                          |
| `/acp steer`         | चल रहे सेशन को दिशा-निर्देश भेजें।                | `/acp steer --session support inbox prioritize failing tests` |
| `/acp close`         | सेशन बंद करें और थ्रेड लक्ष्यों की बाइंडिंग हटाएँ।                  | `/acp close`                                                  |
| `/acp status`        | बैकएंड, मोड, स्थिति, रनटाइम विकल्प और क्षमताएँ दिखाएँ। | `/acp status`                                                 |
| `/acp set-mode`      | लक्ष्य सेशन का रनटाइम मोड सेट करें।                      | `/acp set-mode plan`                                          |
| `/acp set`           | सामान्य रनटाइम कॉन्फ़िगरेशन विकल्प लिखें।                      | `/acp set model openai/gpt-5.4`                               |
| `/acp cwd`           | रनटाइम कार्यशील डायरेक्टरी ओवरराइड सेट करें।                   | `/acp cwd /Users/user/Projects/repo`                          |
| `/acp permissions`   | अनुमोदन नीति प्रोफ़ाइल सेट करें।                              | `/acp permissions strict`                                     |
| `/acp timeout`       | रनटाइम टाइमआउट (सेकंड) सेट करें।                            | `/acp timeout 120`                                            |
| `/acp model`         | रनटाइम मॉडल ओवरराइड सेट करें।                               | `/acp model anthropic/claude-opus-4-6`                        |
| `/acp reset-options` | सेशन रनटाइम विकल्प ओवरराइड हटाएँ।                  | `/acp reset-options`                                          |
| `/acp sessions`      | स्टोर से हाल के ACP सेशन सूचीबद्ध करें।                      | `/acp sessions`                                               |
| `/acp doctor`        | बैकएंड स्वास्थ्य, क्षमताएँ और कार्रवाई योग्य सुधार।           | `/acp doctor`                                                 |
| `/acp install`       | नियतात्मक इंस्टॉल और सक्षम करने के चरण प्रिंट करें।             | `/acp install`                                                |

रनटाइम नियंत्रण (`spawn`, `cancel`, `steer`, `close`, `status`, `set-mode`,
`set`, `cwd`, `permissions`, `timeout`, `model` और `reset-options`) के लिए
बाहरी चैनलों से स्वामी पहचान और आंतरिक Gateway क्लाइंट से
`operator.admin` आवश्यक है। अधिकृत गैर-स्वामी प्रेषक अब भी `sessions`,
`doctor`, `install` और `help` का उपयोग कर सकते हैं। गैर-स्वामी प्रेषकों के लिए, `/acp sessions`
केवल वर्तमान बाउंड या अनुरोधकर्ता सेशन सूचीबद्ध करता है; स्वामी पहचान और
`operator.admin` क्लाइंट सभी हाल के सेशन देखते हैं।

`/acp status` प्रभावी रनटाइम विकल्पों के साथ रनटाइम-स्तरीय और
बैकएंड-स्तरीय सेशन पहचानकर्ता दिखाता है। जब किसी बैकएंड में कोई क्षमता नहीं होती,
तो असमर्थित-नियंत्रण त्रुटियाँ स्पष्ट रूप से दिखाई देती हैं। लक्ष्य टोकन
(`session-key`, `session-id` या `session-label`) स्वीकार करने वाले कमांड, कस्टम प्रति-एजेंट
`session.store` रूट सहित, Gateway सेशन खोज के माध्यम से उन्हें हल करते हैं। `/acp sessions`
लक्ष्य टोकन स्वीकार नहीं करता।

### रनटाइम विकल्प मैपिंग

`/acp` में सुविधा कमांड और एक सामान्य सेटर है। समकक्ष क्रियाएँ:

| कमांड                      | इससे मैप होता है                              | टिप्पणियाँ                                                                                                                                                                                                      |
| ---------------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/acp model <id>`            | रनटाइम कॉन्फ़िगरेशन कुंजी `model`           | Codex ACP के लिए, OpenClaw `openai/<model>` को अडैप्टर मॉडल आईडी में सामान्यीकृत करता है और `openai/gpt-5.4/high` जैसे स्लैश रीजनिंग प्रत्ययों को `reasoning_effort` से मैप करता है।                                         |
| `/acp set thinking <level>`  | कैनोनिकल विकल्प `thinking`          | उपलब्ध होने पर OpenClaw बैकएंड द्वारा विज्ञापित समतुल्य भेजता है, जिसमें पहले `thinking`, फिर `effort`, `reasoning_effort`, या `thought_level` को प्राथमिकता दी जाती है। Codex ACP के लिए, अडैप्टर मानों को `reasoning_effort` से मैप करता है। |
| `/acp permissions <profile>` | कैनोनिकल विकल्प `permissionProfile` | उपलब्ध होने पर OpenClaw बैकएंड द्वारा विज्ञापित समतुल्य भेजता है, जैसे `approval_policy`, `permission_profile`, `permissions`, या `permission_mode`।                                                       |
| `/acp timeout <seconds>`     | कैनोनिकल विकल्प `timeoutSeconds`    | उपलब्ध होने पर OpenClaw बैकएंड द्वारा विज्ञापित समतुल्य भेजता है, जैसे `timeout` या `timeout_seconds`।                                                                                                     |
| `/acp cwd <path>`            | रनटाइम की वर्तमान कार्यशील डायरेक्टरी का ओवरराइड                 | प्रत्यक्ष अपडेट।                                                                                                                                                                                             |
| `/acp set <key> <value>`     | सामान्य                              | `key=cwd` वर्तमान कार्यशील डायरेक्टरी के ओवरराइड पथ का उपयोग करता है।                                                                                                                                                                      |
| `/acp reset-options`         | सभी रनटाइम ओवरराइड हटाता है         | -                                                                                                                                                                                                          |

## acpx हार्नेस, Plugin सेटअप और अनुमतियाँ

acpx हार्नेस कॉन्फ़िगरेशन (Claude Code / Codex / Gemini CLI उपनाम),
plugin-tools और OpenClaw-tools MCP ब्रिज तथा ACP अनुमति मोड के लिए,
