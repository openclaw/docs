Tool Search एक प्रयोगात्मक OpenClaw एजेंट रनटाइम सुविधा है। यह एजेंटों को बड़े टूल कैटलॉग खोजने और उनमें मौजूद टूल कॉल करने का एक
संक्षिप्त तरीका देता है। यह तब उपयोगी होता है जब रन में कई टूल उपलब्ध हों,
लेकिन मॉडल को संभवतः उनमें से केवल कुछ की आवश्यकता हो।

यह पृष्ठ OpenClaw Tool Search का दस्तावेज़ीकरण करता है। यह Codex-नेटिव टूल
सर्च या डायनेमिक-टूल्स सरफ़ेस नहीं है। Codex-नेटिव कोड मोड, टूल सर्च, स्थगित
डायनेमिक टूल और नेस्टेड टूल कॉल स्थिर Codex हार्नेस सरफ़ेस हैं और
`tools.toolSearch` पर निर्भर नहीं हैं।

Tool Search नियंत्रणों के बजाय QuickJS-WASI `exec`/`wait`
सरफ़ेस उपलब्ध कराने वाले सामान्य OpenClaw रनटाइम के लिए, [कोड मोड](/tools/code-mode) देखें।

OpenClaw रन के लिए सक्षम होने पर, मॉडल को डिफ़ॉल्ट रूप से एक `tool_search_code` टूल
और वे सभी डायरेक्ट-ओनली टूल मिलते हैं जिनके संरचित परिणाम संक्षिप्त
ब्रिज को पार नहीं कर सकते। कोड टूल एक अलग-थलग Node सबप्रोसेस में `openclaw.tools`
ब्रिज के साथ एक छोटा JavaScript बॉडी चलाता है:

```js
const hits = await openclaw.tools.search("GitHub इश्यू बनाएँ");
const tool = await openclaw.tools.describe(hits[0].id);
return await openclaw.tools.call(tool.id, {
  title: "स्टार्टअप पर क्रैश",
  body: "दोबारा उत्पन्न करने के चरण...",
});
```

कैटलॉग में कैटलॉग-योग्य OpenClaw टूल, Plugin टूल, MCP
टूल और क्लाइंट द्वारा दिए गए टूल शामिल हो सकते हैं। मॉडल प्रत्येक कैटलॉग किए गए स्कीमा को
पहले से नहीं देखता। इसके बजाय, यह संक्षिप्त डिस्क्रिप्टर खोजता है, सटीक स्कीमा की आवश्यकता होने पर किसी चुने गए
टूल का विवरण प्राप्त करता है और OpenClaw के माध्यम से उस टूल को कॉल करता है।
डायरेक्ट-ओनली टूल मॉडल को दिखाई देते रहते हैं और कैटलॉग में नहीं जोड़े जाते।

Codex हार्नेस रन को ये प्रयोगात्मक OpenClaw Tool Search
नियंत्रण नहीं मिलते। OpenClaw उत्पाद क्षमताओं को Codex तक डायनेमिक टूल के रूप में पहुँचाता है और
स्थिर नेटिव कोड मोड, नेटिव टूल सर्च, स्थगित डायनेमिक
टूल और नेस्टेड टूल कॉल का स्वामित्व Codex के पास होता है।

## टर्न कैसे चलता है

योजना बनाते समय OpenClaw का एम्बेडेड रनर रन के लिए प्रभावी कैटलॉग तैयार करता
है:

1. एजेंट, प्रोफ़ाइल, सैंडबॉक्स और सत्र के लिए सक्रिय टूल नीति का समाधान करें।
2. योग्य OpenClaw और Plugin टूल सूचीबद्ध करें।
3. सत्र MCP रनटाइम के माध्यम से योग्य MCP टूल सूचीबद्ध करें।
4. वर्तमान रन के लिए दिए गए योग्य क्लाइंट टूल जोड़ें।
5. डायरेक्ट-ओनली टूल को मॉडल के लिए दृश्यमान रखें और शेष कैटलॉग-योग्य टूल के
   संक्षिप्त डिस्क्रिप्टर इंडेक्स करें।
6. उन डायरेक्ट-ओनली टूल के साथ OpenClaw कोड ब्रिज, संरचित फ़ॉलबैक टूल या
   संक्षिप्त डायरेक्टरी सरफ़ेस उपलब्ध कराएँ।

निष्पादन के समय प्रत्येक वास्तविक टूल कॉल OpenClaw में लौटता है। अलग-थलग Node
रनटाइम में Plugin कार्यान्वयन, MCP क्लाइंट ऑब्जेक्ट या सीक्रेट नहीं होते।
`openclaw.tools.call(...)` ब्रिज पार करके Gateway में वापस जाता है, जहाँ
सामान्य नीति, अनुमोदन, हुक, लॉगिंग और परिणाम प्रबंधन अब भी लागू होते हैं।

## मोड

`tools.toolSearch` में मॉडल के लिए तीन मोड हैं:

- `code`: डायरेक्ट-ओनली टूल के साथ डिफ़ॉल्ट संक्षिप्त JavaScript ब्रिज
  `tool_search_code` उपलब्ध कराता है।
- `tools`: उन प्रोवाइडर के लिए, जिन्हें कोड नहीं मिलना चाहिए, डायरेक्ट-ओनली
  टूल के साथ `tool_search`, `tool_describe` और `tool_call` को सामान्य
  संरचित टूल के रूप में उपलब्ध कराता है।
- `directory`: उन प्रोवाइडर के लिए, जिन्हें प्रत्येक पूर्ण स्कीमा के बिना टूल के नाम
  दिखने चाहिए, उपलब्ध टूल नामों और विवरणों की सीमित प्रॉम्प्ट डायरेक्टरी के साथ
  `tool_search`, `tool_describe` और `tool_call` उपलब्ध कराता है। OpenClaw
  वर्तमान टर्न के लिए संभावित या आवश्यक टूल स्कीमा का एक छोटा सीमित समूह सीधे
  उपलब्ध भी करा सकता है। इस मोड में भी डायरेक्ट-ओनली टूल दिखाई देते हैं।

सभी मोड समान नीति-फ़िल्टर किए गए कैटलॉग और सामान्य OpenClaw निष्पादन
पथ का उपयोग करते हैं। `catalogMode: "direct-only"` चिह्नित टूल उस कैटलॉग से बाहर रहते हैं और
मॉडल को दिखाई देते रहते हैं। यदि वर्तमान रनटाइम अलग-थलग Node कोड-मोड चाइल्ड
प्रोसेस लॉन्च नहीं कर सकता, तो डिफ़ॉल्ट `code` मोड कैटलॉग
कॉम्पैक्शन से पहले `tools` पर फ़ॉलबैक करता है। `directory` मोड में, क्लाइंट द्वारा दिए गए टूल
वर्तमान रन के लिए सीधे दिखाई देते रहते हैं, जबकि OpenClaw टूल, Plugin टूल और MCP टूल को
डायरेक्टरी कैटलॉग के पीछे संक्षिप्त किया जा सकता है। डायरेक्टरी के किसी सटीक छिपे हुए
नाम की सीधी कॉल को निष्पादन से पहले उसी अधिकृत कैटलॉग से हाइड्रेट किया जाता है।

सभी मोड प्रयोगात्मक हैं। छोटे OpenClaw टूल कैटलॉग के लिए टूल को सीधे उपलब्ध कराना
पसंद करें और Codex हार्नेस रन के लिए स्थिर Codex-नेटिव सरफ़ेस को प्राथमिकता दें।

स्रोत चयन का कोई अलग कॉन्फ़िगरेशन नहीं है। Tool Search सक्षम होने पर,
कैटलॉग में सामान्य नीति फ़िल्टरिंग के बाद कैटलॉग-योग्य OpenClaw, MCP और क्लाइंट टूल
शामिल होते हैं; डायरेक्ट-ओनली टूल अलग रखे जाते हैं।

## यह क्यों मौजूद है

बड़े कैटलॉग उपयोगी हैं, लेकिन महँगे हैं। प्रत्येक टूल स्कीमा मॉडल को भेजने से
अनुरोध बड़ा होता है, योजना बनाने की गति धीमी होती है और गलती से टूल
चुने जाने की संभावना बढ़ती है।

Tool Search इस स्वरूप को बदलता है:

- डायरेक्ट टूल: मॉडल पहले टोकन से पहले प्रत्येक चुना गया स्कीमा देखता है
- Tool Search कोड मोड: मॉडल एक संक्षिप्त कोड टूल, एक छोटा API
  अनुबंध और सभी डायरेक्ट-ओनली टूल देखता है
- Tool Search टूल मोड: मॉडल तीन संक्षिप्त संरचित फ़ॉलबैक
  टूल और सभी डायरेक्ट-ओनली टूल देखता है
- Tool Search डायरेक्टरी मोड: मॉडल एक सीमित डायरेक्टरी के साथ
  खोज/विवरण/कॉल नियंत्रण, संभावित या आवश्यक स्कीमा का एक छोटा सीमित समूह
  और सभी डायरेक्ट-ओनली टूल देखता है
- टर्न के दौरान: मॉडल आवश्यकतानुसार शेष स्कीमा लोड कर सकता है

छोटे कैटलॉग के लिए टूल को सीधे उपलब्ध कराना अब भी सही डिफ़ॉल्ट है। Tool Search
तब सबसे अच्छा है जब एक रन कई टूल देख सकता हो, विशेष रूप से MCP सर्वर या
क्लाइंट द्वारा दिए गए ऐप टूल से।

## API

`openclaw.tools.search(query, options?)`

वर्तमान रन के प्रभावी कैटलॉग में खोज करता है। परिणाम संक्षिप्त होते हैं और उन्हें
प्रॉम्प्ट संदर्भ में वापस रखना सुरक्षित होता है। प्रत्येक परिणाम में एक सीमित TypeScript-शैली का
`input` सिग्नेचर शामिल होता है, जैसे `{ id: string; mode?: "drip" | "flood" }`, ताकि
वह सिग्नेचर पर्याप्त होने पर मॉडल `describe` को छोड़ सके। किसी विश्वसनीय
OpenClaw कोर या Plugin टूल में एक संक्षिप्त `output` संकेत भी शामिल हो सकता है, जैसे
`Array<{ id: string; paid: boolean }>`। MCP और क्लाइंट आउटपुट-स्कीमा दावों को
इस विश्वसनीय संकेत में प्रोमोट नहीं किया जाता। उनके अविश्वसनीय इनपुट स्कीमा को भी
`input: "unknown"` के रूप में स्थगित किया जाता है; उन्हें कॉल करने से पहले `describe` का उपयोग करें। खुले,
अत्यधिक बड़े या अन्यथा आंशिक आउटपुट स्कीमा संकेत को छोड़ देते हैं और इसके बजाय
`describe` के माध्यम से उपलब्ध रहते हैं।

```js
const hits = await openclaw.tools.search("कैलेंडर इवेंट", { limit: 5 });
```

`openclaw.tools.describe(id)`

किसी एक खोज परिणाम का पूर्ण मेटाडेटा लोड करता है, जिसमें सटीक इनपुट स्कीमा और
टूल द्वारा घोषित होने पर विश्वसनीय पूर्ण `outputSchema` शामिल है।

```js
const calendarCreate = await openclaw.tools.describe("mcp:calendar:create_event");
```

`openclaw.tools.call(id, args)`

OpenClaw के माध्यम से चुने गए टूल को कॉल करता है और अपरिष्कृत `{ tool, result }`
एनवेलप लौटाता है। JSON लौटाने वाले टूल सामान्यतः अपना मान
`result.details` में रखते हैं। यदि कोई विश्वसनीय टूल `outputSchema` घोषित करता है, तो OpenClaw निष्पादन से पहले
स्कीमा कंपाइल करता है और कैटलॉग कॉल लौटाने से पहले सामान्य टूल
हुक के बाद अंतिम `details` को सत्यापित करता है।

```js
await openclaw.tools.call(calendarCreate.id, {
  summary: "योजना",
  start: "2026-05-09T14:00:00Z",
});
```

टूल लेखक टूल की `outputSchema` प्रॉपर्टी पर आउटपुट अनुबंध घोषित करते हैं।
यह रेंडर किए गए कॉन्टेंट ब्लॉक नहीं, बल्कि `AgentToolResult.details` का वर्णन करता है।
सभी त्रुटि न फेंकने वाले वैरिएंट शामिल करें या अस्थिर परिणामों के लिए इसे छोड़ दें। [कोड मोड आउटपुट अनुबंध](/tools/code-mode#declared-output-contracts) और
[टूल Plugin](/hi/plugins/tool-plugins#output-contracts) देखें।

संरचित फ़ॉलबैक मोड टूल के रूप में वही ऑपरेशन उपलब्ध कराता है:

- `tool_search`
- `tool_describe`
- `tool_call`

डायरेक्टरी मोड उपलब्ध कराता है:

- `tool_search`
- `tool_describe`
- `tool_call`

यह क्लाइंट द्वारा दिए गए टूल और सभी डायरेक्ट-ओनली टूल को सीधे दृश्यमान भी रखता है
और वर्तमान टर्न के लिए संभावित या आवश्यक कैटलॉग टूल स्कीमा का एक छोटा सीमित समूह
सीधे उपलब्ध करा सकता है। यदि सीमित डायरेक्टरी में प्रविष्टियाँ छूट जाती हैं, तो उन्हें खोजने के लिए
`tool_search` का उपयोग करें। यदि मॉडल डायरेक्टरी के किसी सटीक छिपे हुए
टूल नाम का सीधे अनुरोध करता है, तो OpenClaw सामान्य निष्पादन से पहले उसे अधिकृत कैटलॉग से
हाइड्रेट करता है।
डायरेक्टरी-मोड क्लाइंट टूल नामों को OpenClaw, Plugin या MCP
टूल नामों से टकराना नहीं चाहिए, क्योंकि सटीक स्थगित डिस्पैच उन्हीं नामों का उपयोग करता है।

## रनटाइम सीमा

कोड ब्रिज अल्पकालिक Node सबप्रोसेस में चलता है। सबप्रोसेस
Node अनुमति मोड सक्षम, खाली एनवायरनमेंट, फ़ाइल सिस्टम या
नेटवर्क की कोई अनुमति नहीं और चाइल्ड-प्रोसेस या वर्कर की कोई अनुमति नहीं होने के साथ शुरू होता है। OpenClaw
पैरेंट-प्रोसेस वॉल-क्लॉक टाइमआउट लागू करता है और टाइमआउट पर सबप्रोसेस को बंद कर देता है,
जिसमें एसिंक निरंतरताओं के बाद का समय भी शामिल है।

रनटाइम केवल इन्हें उपलब्ध कराता है:

- `console.log`, `console.warn` और `console.error`
- `openclaw.tools.search`
- `openclaw.tools.describe`
- `openclaw.tools.call`

अंतिम कॉल पर सामान्य OpenClaw व्यवहार अब भी लागू होता है:

- टूल अनुमति और निषेध नीतियाँ
- प्रति-एजेंट और प्रति-सैंडबॉक्स टूल प्रतिबंध
- चैनल/रनटाइम टूल नीति
- अनुमोदन हुक
- Plugin `before_tool_call` हुक
- सत्र पहचान, लॉग और टेलीमेट्री

## कॉन्फ़िगरेशन

डिफ़ॉल्ट कोड ब्रिज के साथ OpenClaw रन के लिए Tool Search सक्षम करें:

```bash
openclaw config set tools.toolSearch true
```

समतुल्य JSON:

```json5
{
  tools: {
    toolSearch: true,
  },
}
```

इसके बजाय OpenClaw रन के लिए संरचित फ़ॉलबैक टूल का उपयोग करें:

```json5
{
  tools: {
    toolSearch: {
      mode: "tools",
    },
  },
}
```

इसके बजाय OpenClaw रन के लिए संक्षिप्त डायरेक्टरी सरफ़ेस का उपयोग करें:

```json5
{
  tools: {
    toolSearch: {
      mode: "directory",
    },
  },
}
```

कोड-मोड टाइमआउट और खोज परिणाम सीमाएँ समायोजित करें (दिखाए गए मान डिफ़ॉल्ट हैं):

```json5
{
  tools: {
    toolSearch: {
      mode: "code",
      codeTimeoutMs: 10000,
      searchDefaultLimit: 8,
      maxSearchLimit: 20,
    },
  },
}
```

रनटाइम `codeTimeoutMs` को 1000-60000, `maxSearchLimit` को 1-50 और
`searchDefaultLimit` को 1..`maxSearchLimit` तक सीमित करता है।

इसे अक्षम करें:

```json5
{
  tools: {
    toolSearch: false,
  },
}
```

## प्रॉम्प्ट और टेलीमेट्री

Tool Search सीधे टूल उपलब्ध कराने से तुलना करने के लिए पर्याप्त टेलीमेट्री रिकॉर्ड करता है:

- हार्नेस को भेजे गए क्रमबद्ध टूल और प्रॉम्प्ट बाइट की कुल संख्या
- कैटलॉग का आकार और स्रोत विभाजन
- खोज, विवरण और कॉल की संख्या
- OpenClaw के माध्यम से निष्पादित अंतिम टूल कॉल
- चुनी गई टूल आईडी और स्रोत

सत्र लॉग से इन प्रश्नों के उत्तर देना संभव होना चाहिए:

- मॉडल ने पहले से कितने टूल स्कीमा देखे
- उसने कितने खोज और विवरण ऑपरेशन किए
- कौन-सा अंतिम टूल कॉल किया गया
- परिणाम OpenClaw, MCP या क्लाइंट टूल में से किससे आया

## E2E सत्यापन

QA Lab Gateway परिदृश्य OpenClaw रनटाइम के साथ दोनों पथों को सिद्ध करता है:

```bash
pnpm openclaw qa suite --provider-mode mock-openai --scenario tool-search-gateway-e2e
```

यह एक बड़े टूल कैटलॉग वाला अस्थायी नकली Plugin बनाता है, नकली
OpenAI प्रोवाइडर शुरू करता है, Gateway को एक बार डायरेक्ट मोड में और एक बार Tool Search
सक्षम करके शुरू करता है, फिर प्रोवाइडर अनुरोध पेलोड और सत्र लॉग की तुलना करता है।

रिग्रेशन सिद्ध करता है:

1. डायरेक्ट मोड नकली Plugin टूल को कॉल कर सकता है।
2. टूल सर्च उसी नकली Plugin टूल को कॉल कर सकता है।
3. डायरेक्ट मोड नकली Plugin टूल स्कीमा को सीधे प्रोवाइडर के सामने प्रस्तुत करता है।
4. टूल सर्च केवल कॉम्पैक्ट ब्रिज और किसी भी केवल-डायरेक्ट टूल को प्रस्तुत करता है।
5. बड़े नकली कैटलॉग के लिए टूल सर्च अनुरोध पेलोड छोटा होता है।
6. सेशन लॉग अपेक्षित टूल-कॉल संख्याएँ और ब्रिज की गई कॉल की टेलीमेट्री दिखाते हैं।

## विफलता का व्यवहार

टूल सर्च को अवरुद्ध स्थिति में विफल होना चाहिए:

- यदि कोई टूल प्रभावी नीति में नहीं है, तो खोज को उसे नहीं लौटाना चाहिए
- यदि कोई चयनित टूल अनुपलब्ध हो जाता है, तो `tool_call` को विफल होना चाहिए
- यदि नीति या स्वीकृति निष्पादन को अवरुद्ध करती है, तो कॉल परिणाम को उसे दरकिनार करने के बजाय
  उस अवरोध की रिपोर्ट करनी चाहिए
- यदि कोड ब्रिज पृथक रनटाइम नहीं बना सकता, तो `mode: "tools"` का उपयोग करें या
  उस परिनियोजन के लिए टूल सर्च अक्षम करें
