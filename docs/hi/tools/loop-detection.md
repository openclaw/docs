OpenClaw में दो सहयोगी सुरक्षा-सीमाएँ हैं जो दोहराव वाले टूल-कॉल पैटर्न से बचाती हैं,
दोनों को `tools.loopDetection` के अंतर्गत कॉन्फ़िगर किया जाता है:

1. **लूप पहचान** (`enabled`) - डिफ़ॉल्ट रूप से अक्षम। दोहराए गए पैटर्न और अज्ञात-टूल पुनः प्रयासों के लिए हालिया
   टूल-कॉल इतिहास पर नज़र रखती है।
2. **पोस्ट-कॉम्पैक्शन सुरक्षा** (`postCompactionGuard`) - जब भी
   `enabled` को स्पष्ट रूप से `false` पर सेट नहीं किया जाता, सक्षम रहती है। प्रत्येक कॉम्पैक्शन-पुनः प्रयास के बाद सक्रिय होती है और
   यदि एजेंट विंडो के भीतर उसी `(tool, args, result)` त्रिक को
   दोहराता है, तो रन को निरस्त कर देती है।

दोनों सुरक्षा-सीमाओं को निष्क्रिय करने के लिए `tools.loopDetection.enabled: false` सेट करें।

## यह क्यों मौजूद है

- ऐसे दोहराव वाले अनुक्रमों का पता लगाना जो कोई प्रगति नहीं करते।
- उच्च-आवृत्ति वाले परिणाम-विहीन लूप का पता लगाना (समान टूल, समान इनपुट, बार-बार होने वाली
  त्रुटियाँ)।
- ज्ञात पोलिंग टूल के लिए विशिष्ट दोहराए गए-कॉल पैटर्न का पता लगाना।
- कॉन्टेक्स्ट-ओवरफ़्लो -> कॉम्पैक्शन -> समान-लूप चक्रों को अनिश्चित काल तक
  चलने देने के बजाय तोड़ना।

## कॉन्फ़िगरेशन ब्लॉक

वैश्विक डिफ़ॉल्ट, जिनमें प्रत्येक दस्तावेज़ीकृत फ़ील्ड दिखाई गई है:

```json5
{
  tools: {
    loopDetection: {
      enabled: false, // हालिया-इतिहास डिटेक्टरों का मुख्य स्विच
      historySize: 30,
      warningThreshold: 10,
      criticalThreshold: 20,
      unknownToolThreshold: 10,
      globalCircuitBreakerThreshold: 30,
      detectors: {
        genericRepeat: true,
        knownPollNoProgress: true,
        pingPong: true,
      },
      postCompactionGuard: {
        windowSize: 3, // कॉम्पैक्शन-पुनः प्रयास के बाद सक्रिय; तब तक चलती है जब तक enabled स्पष्ट रूप से false न हो
      },
    },
  },
}
```

प्रति-एजेंट ओवरराइड (वैकल्पिक, `agents.list[].tools.loopDetection` पर):

```json5
{
  agents: {
    list: [
      {
        id: "safe-runner",
        tools: {
          loopDetection: {
            enabled: true,
            warningThreshold: 8,
            criticalThreshold: 16,
          },
        },
      },
    ],
  },
}
```

प्रति-एजेंट सेटिंग वैश्विक ब्लॉक पर फ़ील्ड-दर-फ़ील्ड लागू होती हैं (नेस्टेड
`detectors` और `postCompactionGuard` सहित), इसलिए एजेंट को केवल वे
फ़ील्ड सेट करनी होती हैं जिन्हें वह बदलना चाहता है।

### फ़ील्ड का व्यवहार

| फ़ील्ड                            | डिफ़ॉल्ट | प्रभाव                                                                                                                                     |
| -------------------------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `enabled`                        | `false` | हालिया-इतिहास डिटेक्टरों का मुख्य स्विच। `false` पोस्ट-कॉम्पैक्शन सुरक्षा को भी अक्षम कर देता है।                                          |
| `historySize`                    | `30`    | विश्लेषण के लिए रखे जाने वाले हालिया टूल कॉल की संख्या।                                                                                             |
| `warningThreshold`               | `10`    | किसी पैटर्न को केवल-चेतावनी के रूप में वर्गीकृत करने से पहले दोहराव की संख्या।                                                                               |
| `criticalThreshold`              | `20`    | प्रगति-विहीन लूप पैटर्न को अवरुद्ध करने के लिए दोहराव की संख्या। गलत कॉन्फ़िगरेशन होने पर रनटाइम इसे `warningThreshold` से ऊपर सीमित करता है।                       |
| `unknownToolThreshold`           | `10`    | इतनी बार अनुपलब्ध मिलने के बाद समान अनुपलब्ध टूल के दोहराए गए कॉल को अवरुद्ध करता है। यह `detectors` द्वारा नियंत्रित नहीं होता।                                       |
| `globalCircuitBreakerThreshold`  | `30`    | सभी डिटेक्टरों के लिए वैश्विक प्रगति-विहीन सर्किट ब्रेकर। गलत कॉन्फ़िगरेशन होने पर रनटाइम इसे `criticalThreshold` से ऊपर सीमित करता है। यह `detectors` द्वारा नियंत्रित नहीं होता। |
| `detectors.genericRepeat`        | `true`  | समान-टूल + समान-आर्ग्युमेंट वाले दोहराए गए कॉल पर चेतावनी देता है; जब वे कॉल समान परिणाम भी लौटाते हैं, तो उन्हें अवरुद्ध करता है।                                     |
| `detectors.knownPollNoProgress`  | `true`  | ज्ञात प्रगति-विहीन पोलिंग पैटर्न का पता लगाता है (`action: "poll"`/`"log"` के साथ `process`, `command_status`)।                                    |
| `detectors.pingPong`             | `true`  | दो कॉल के बीच बारी-बारी से होने वाले प्रगति-विहीन पिंग-पोंग पैटर्न का पता लगाता है।                                                                      |
| `postCompactionGuard.windowSize` | `3`     | कॉम्पैक्शन के बाद सुरक्षा जितने प्रयासों तक सक्रिय रहती है, और समान त्रिकों की वह संख्या जो रन को निरस्त करती है।                                   |

`exec` के लिए, प्रगति-विहीन हैशिंग स्थिर कमांड परिणामों (स्थिति,
एग्ज़िट कोड, टाइम-आउट फ़्लैग, आउटपुट) की तुलना करती है और अवधि, PID, सेशन ID तथा कार्यशील डायरेक्टरी जैसे
परिवर्तनशील रनटाइम मेटाडेटा को अनदेखा करती है। आउटबाउंड संदेश-प्रेषण
परिणामों को प्रत्येक कॉल के परिवर्तनशील ID (संदेश ID, फ़ाइल ID, टाइमस्टैम्प)
हटाकर हैश किया जाता है, ताकि एक "भेजा गया" परिणाम किसी दूसरे "भेजा गया"
परिणाम के समान न दिखे। रन ID उपलब्ध होने पर इतिहास का मूल्यांकन केवल उसी रन के भीतर किया जाता है,
इसलिए निर्धारित Heartbeat चक्र और नए रन पिछले रन की पुरानी लूप गणनाएँ
प्राप्त नहीं करते।

## अनुशंसित सेटअप

- छोटे मॉडल के लिए, `enabled: true` सेट करें और थ्रेशोल्ड को उनके
  डिफ़ॉल्ट पर रहने दें। प्रमुख मॉडल को हालिया-इतिहास पहचान की शायद ही कभी आवश्यकता होती है और वे
  पोस्ट-कॉम्पैक्शन सुरक्षा का लाभ लेते हुए भी मुख्य स्विच को `false` रख सकते हैं।
- थ्रेशोल्ड को `warningThreshold < criticalThreshold <
globalCircuitBreakerThreshold` क्रम में रखें; यदि आप `criticalThreshold` और
  `globalCircuitBreakerThreshold` को उस थ्रेशोल्ड पर या उससे नीचे सेट करते हैं जिससे उन्हें अधिक होना चाहिए,
  तो रनटाइम उन्हें ऊपर समायोजित करता है।
- यदि गलत सकारात्मक पहचान होती है:
  - `warningThreshold` और/या `criticalThreshold` बढ़ाएँ।
  - वैकल्पिक रूप से `globalCircuitBreakerThreshold` बढ़ाएँ।
  - केवल समस्या उत्पन्न करने वाले विशिष्ट डिटेक्टर को अक्षम करें (`detectors.<name>: false`)।
  - छोटी ऐतिहासिक विंडो के लिए `historySize` घटाएँ।
- पोस्ट-कॉम्पैक्शन सुरक्षा सहित सब कुछ अक्षम करने के लिए,
  `tools.loopDetection.enabled: false` को स्पष्ट रूप से सेट करें।

## पोस्ट-कॉम्पैक्शन सुरक्षा

कॉन्टेक्स्ट-ओवरफ़्लो के बाद कॉम्पैक्शन-पुनः प्रयास होने पर, रनर अगले कुछ टूल कॉल के लिए
एक छोटी-विंडो वाली सुरक्षा सक्रिय करता है। यदि एजेंट उस विंडो के भीतर समान
`(toolName, argsHash, resultHash)` त्रिक `postCompactionGuard.windowSize`
बार उत्पन्न करता है, तो सुरक्षा निष्कर्ष निकालती है कि कॉम्पैक्शन ने
लूप को नहीं तोड़ा और `compaction_loop_persisted` त्रुटि के साथ रन को निरस्त कर देती है।

सुरक्षा मुख्य `tools.loopDetection.enabled` फ़्लैग द्वारा नियंत्रित होती है, लेकिन इसमें एक
विशेषता है: फ़्लैग सेट न होने या `true` होने पर यह **सक्षम रहती है**, और केवल तभी
बंद होती है जब फ़्लैग को स्पष्ट रूप से `false` पर सेट किया जाता है। यह जानबूझकर किया गया है - सुरक्षा
उन कॉम्पैक्शन लूप से बाहर निकलने के लिए मौजूद है जो अन्यथा असीमित टोकन खर्च करते,
इसलिए बिना कॉन्फ़िगरेशन वाले उपयोगकर्ता को भी सुरक्षा मिलती है।

```json5
{
  tools: {
    loopDetection: {
      // मुख्य स्विच; हालिया डिटेक्टरों के साथ सुरक्षा अक्षम करने के लिए false सेट करें
      enabled: true,
      postCompactionGuard: {
        windowSize: 3, // डिफ़ॉल्ट
      },
    },
  },
}
```

- कम `windowSize` अधिक सख्त है (निरस्त होने से पहले कम प्रयास)।
- अधिक `windowSize` एजेंट को पुनर्प्राप्ति के अधिक प्रयास देता है।
- परिणाम बदलते रहने पर सुरक्षा कभी निरस्त नहीं करती; केवल पूरी तरह समान
  परिणाम ही विंडो में इसे ट्रिगर करते हैं।
- यह केवल कॉम्पैक्शन-पुनः प्रयास के तुरंत बाद सक्रिय होती है, रन के अन्य
  बिंदुओं पर नहीं।

<Note>
  पोस्ट-कॉम्पैक्शन सुरक्षा तब भी चलती है जब मुख्य फ़्लैग स्पष्ट रूप से `false` न हो, भले ही आपने कभी `tools.loopDetection` ब्लॉक न लिखा हो। सत्यापित करने के लिए, कॉम्पैक्शन घटना के तुरंत बाद Gateway लॉग में `post-compaction guard armed for N attempts` खोजें।
</Note>

## लॉग और अपेक्षित व्यवहार

लूप का पता चलने पर OpenClaw एक लूप घटना लॉग करता है और गंभीरता के आधार पर
अगले टूल-चक्र पर चेतावनी देता है या उसे अवरुद्ध करता है, जिससे सामान्य टूल पहुँच बनाए रखते हुए अनियंत्रित टोकन
खर्च और लॉकअप से सुरक्षा होती है।

- चेतावनियाँ पहले आती हैं।
- पैटर्न के चेतावनी थ्रेशोल्ड से आगे बने रहने पर अवरोधन होता है।
- गंभीर थ्रेशोल्ड अगले टूल-चक्र को अवरुद्ध करते हैं और रन रिकॉर्ड में स्पष्ट
  लूप-पहचान कारण दिखाते हैं।
- पोस्ट-कॉम्पैक्शन सुरक्षा आपत्तिजनक टूल और समान-कॉल की संख्या बताने वाली
  `compaction_loop_persisted` त्रुटियाँ उत्पन्न करती है।
