# Chrome एक्सटेंशन

OpenClaw Chrome एक्सटेंशन किसी एजेंट को अलग प्रबंधित ब्राउज़र शुरू किए बिना आपके **साइन-इन किए हुए Chrome
टैब** नियंत्रित करने देता है, और वह भी Chrome के अवरोधक
"Allow remote debugging?" प्रॉम्प्ट के **बिना**।

जब आप फ़ोन (Telegram, WhatsApp आदि) से OpenClaw चलाते हैं, तब यह महत्वपूर्ण होता है:
[`user` प्रोफ़ाइल](/hi/tools/browser#profiles-openclaw-user-chrome), Chrome के रिमोट-डीबगिंग पोर्ट के माध्यम से जुड़ती है,
जिससे डेस्कटॉप पर सहमति संवाद खुलता है और आपके दूर होने पर कोई उसे क्लिक नहीं कर
सकता। इसके बजाय एक्सटेंशन `chrome.debugger` API का उपयोग करता है,
इसलिए पेज के भीतर एकमात्र संकेत Chrome का खारिज किया जा सकने वाला "OpenClaw started debugging
this browser" बैनर होता है।

Anthropic के Claude in Chrome और OpenAI के Codex
Chrome एक्सटेंशन भी इसी संरचना का उपयोग करते हैं।

## यह कैसे काम करता है

इसके तीन भाग हैं:

- **ब्राउज़र नियंत्रण सेवा** (Gateway या नोड होस्ट): वह API जिसे `browser`
  टूल कॉल करता है।
- **एक्सटेंशन रिले** (लूपबैक WebSocket): एक छोटा सर्वर, जिसे नियंत्रण सेवा
  `127.0.0.1` पर शुरू करती है। यह OpenClaw को Chrome DevTools Protocol एंडपॉइंट उपलब्ध कराता है
  और एक्सटेंशन के साथ संचार करता है। दोनों पक्ष होस्ट-स्थानीय टोकन से प्रमाणीकरण करते हैं
  (नीचे देखें)।
- **OpenClaw Chrome एक्सटेंशन** (MV3): `chrome.debugger` के साथ टैब से जुड़ता है,
  CDP ट्रैफ़िक अग्रेषित करता है और **OpenClaw टैब समूह** को प्रबंधित करता है।

OpenClaw केवल **OpenClaw टैब समूह** में मौजूद टैब देख और नियंत्रित कर सकता है।
यह समूह सहमति की सीमा है: किसी टैब को साझा करने के लिए उसे समूह में खींचें और पहुँच तुरंत
निरस्त करने के लिए उसे बाहर खींचें (या टूलबार बटन पर क्लिक करें)।

## इंस्टॉल और पेयर करें

1. अनपैक किए गए एक्सटेंशन का पथ प्रिंट करें:

   ```bash
   openclaw browser extension path
   ```

2. `chrome://extensions` खोलें, **Developer mode** सक्षम करें, **Load
   unpacked** पर क्लिक करें और प्रिंट की गई डायरेक्टरी चुनें।

3. पेयरिंग स्ट्रिंग प्रिंट करें:

   ```bash
   openclaw browser extension pair
   ```

4. OpenClaw टूलबार आइकन पर क्लिक करें और पॉपअप में पेयरिंग स्ट्रिंग पेस्ट करें।
   एक्सटेंशन के रिले से कनेक्ट होने पर बैज **ON** हो जाता है।

पेयरिंग टोकन एक **होस्ट-स्थानीय सीक्रेट** है, जो पहली बार उपयोग करने पर बनाया जाता है और
स्टेट डायरेक्टरी में `credentials/` के अंतर्गत (मोड `0600`) संग्रहीत होता है। ब्राउज़र
चलाने वाली प्रत्येक मशीन—Gateway होस्ट और प्रत्येक ब्राउज़र नोड होस्ट—का अपना
टोकन होता है, इसलिए किसी क्रेडेंशियल को मशीनों के बीच भेजने की आवश्यकता नहीं होती। इसे रोटेट करने के लिए
`browser-extension-relay.secret` फ़ाइल हटाएँ और फिर से पेयर करें।

## इसका उपयोग करें

`browser` टूल कॉल में अंतर्निहित `chrome` प्रोफ़ाइल चुनें या उसे
डिफ़ॉल्ट बनाएँ:

```bash
openclaw config set browser.defaultProfile chrome
```

```json5
{
  browser: {
    profiles: {
      chrome: { driver: "extension", color: "#FF4500" },
    },
  },
}
```

- टैब साझा करें: उस टैब पर OpenClaw टूलबार बटन क्लिक करें (वह
  OpenClaw टैब समूह में शामिल हो जाएगा), या किसी भी टैब को समूह में खींचें।
- एजेंट नए टैब भी खोल सकता है; वे स्वतः समूह में शामिल हो जाते हैं।
- पहुँच निरस्त करें: बटन पर फिर से क्लिक करें, टैब को समूह से बाहर खींचें या
  Chrome का डीबगिंग बैनर खारिज करें। एजेंट तुरंत उस टैब की पहुँच खो देता है।

### टैब कोपायलट साइड पैनल

एक्सटेंशन को पेयर करने के बाद उसके टूलबार पॉपअप में **Open tab copilot** पर क्लिक करें।
OpenClaw उसी सटीक Chrome टैब के लिए `sidepanel.html` कॉन्फ़िगर करता है; मैनिफ़ेस्ट में
कोई वैश्विक साइड-पैनल पथ नहीं है। इसलिए प्रत्येक टैब को अलग पैनल दस्तावेज़,
Gateway सत्र, संदेश सदस्यता और टाइपयुक्त ब्राउज़र-टूल बाइंडिंग मिलती है।

पैनल आपके संदेश में पेज URL, शीर्षक, DOM या दृश्यमान टेक्स्ट नहीं डालता।
यह केवल आपके द्वारा टाइप किया गया टेक्स्ट भेजता है। ब्राउज़र क्रियाओं के साथ एक अलग
Gateway-प्रमाणीकृत बाइंडिंग होती है, जिसमें Chrome टैब और CDP लक्ष्य शामिल होते हैं, और
ब्राउज़र टूल उस लक्ष्य को बदलने या पूरे ब्राउज़र पर लागू होने वाली
क्रियाओं का उपयोग करने के प्रयास अस्वीकार करता है। उत्तर पैनल (`deliver: false`) में ही रहते हैं; उन्हें
Telegram, Discord या किसी अन्य चैनल का मार्ग विरासत में नहीं मिलता।

कोपायलट एक समर्पित, पेयर किया हुआ Gateway डिवाइस है, जिसके पास `operator.read` और
`operator.write` स्कोप हैं। पहली बार उपयोग करते समय उसके अनुरोध की जाँच करके उसे अनुमोदित करें:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

एक्सटेंशन उस डिवाइस पहचान और Gateway द्वारा जारी डिवाइस टोकन को बनाए रखता है,
जिनका स्कोप उन्हें जारी करने वाले कैनॉनिकल Gateway एंडपॉइंट तक सीमित होता है। किसी अलग
Gateway को पेयर करने पर अलग पहचान, टोकन और सत्र अभिरक्षा बनती है; क्रेडेंशियल और
सत्रों का एंडपॉइंट के बीच कभी पुनः उपयोग नहीं होता। एक्सटेंशन
Gateway साझा सीक्रेट को स्थायी रूप से संग्रहीत नहीं करता। कोई पैनल केवल अपने टैब सत्रों की सदस्यता ले सकता है और
Gateway उन इवेंट को डिलीवरी से पहले फ़िल्टर करता है।

यदि किसी रन के दौरान Gateway कनेक्शन टूट जाता है, तो एक्सटेंशन उस रन ID की स्थायी
अभिरक्षा बनाए रखता है। दोबारा कनेक्ट होने पर, किसी भी पैनल को पुनः सक्षम करने से पहले वह
अनसुलझे रन को निरस्त करता है और फिर ट्रांसक्रिप्ट इतिहास पुनः लोड करता है। यह फ़ेल-क्लोज़्ड चरण
डिलीवरी अंतराल के दौरान ब्राउज़र क्रियाओं को बिना दिखाई दिए जारी रहने से रोकता है।

टैब बंद करने पर उसकी लाइव सदस्यता तुरंत हट जाती है, कोई भी दृश्यमान
रन निरस्त हो जाता है और उस टैब का सत्र आर्काइव किया हुआ चिह्नित हो जाता है। यदि Gateway अस्थायी रूप से
ऑफ़लाइन है, तो एक्सटेंशन लंबित आर्काइव को स्थायी रूप से रखता है और केवल उसी
Gateway एंडपॉइंट के दोबारा कनेक्ट होने पर पुनः प्रयास करता है; वह किसी अलग Gateway को
आर्काइव अनुरोध कभी नहीं भेजता। ब्राउज़र क्रैश के बाद, अगली बार शुरू होने पर पिछले
ब्राउज़र इंस्टेंस द्वारा छोड़े गए सत्र आर्काइव हो जाते हैं। आर्काइव किए गए सत्र नया कार्य अस्वीकार करते हैं, जबकि
उनके ट्रांसक्रिप्ट सत्र इतिहास में उपलब्ध रहते हैं। ब्राउज़र-कोपायलट कुंजियाँ
थ्रेड सत्र होती हैं, इसलिए आयु और प्रविष्टि-संख्या का सामान्य रखरखाव उन्हें संरक्षित रखता है।
प्रति-एजेंट सत्र डिस्क बजट अब भी लागू होता है (डिफ़ॉल्ट `2gb`) और दबाव की स्थिति में
सबसे पुराने सत्र हटा सकता है; [सत्र रखरखाव](/hi/reference/session-management-compaction#store-maintenance-and-disk-controls) देखें।

साइड पैनल को वर्तमान में या तो Gateway द्वारा होस्ट किया गया एक्सटेंशन रिले या
प्रत्यक्ष रिमोट Gateway रिले चाहिए। ब्राउज़र नोड पर मौजूद लूपबैक रिले अभी
टाइपयुक्त टैब बाइंडिंग के लिए आवश्यक नोड मार्ग उपलब्ध नहीं करा सकता, इसलिए पैनल
पूरे ब्राउज़र की रूटिंग पर फ़ॉलबैक करने के बजाय इस टोपोलॉजी को अस्वीकार करता है।

## किसी पेज को OpenClaw पर भेजें

पढ़ने योग्य पेज टेक्स्ट को अपने मुख्य OpenClaw सत्र के साथ साझा करने के लिए टूलबार पॉपअप में
**Send page to OpenClaw** का उपयोग करें। आप वैकल्पिक नोट जोड़ सकते हैं, पेज या
चयन के राइट-क्लिक मेन्यू का उपयोग कर सकते हैं या `Alt+Shift+S` दबा सकते हैं। चयन मौजूद होने पर OpenClaw आपके वर्तमान
चयन को प्राथमिकता देता है, साझा सामग्री को सिस्टम इवेंट के रूप में कतारबद्ध करता है और
मुख्य सत्र को तुरंत सक्रिय करता है।

टैब का OpenClaw टैब समूह में होना आवश्यक नहीं है। यह एक बार का,
स्पष्ट साझा करना है: पेज की कोई अन्य सामग्री उजागर नहीं होती और इससे कोई निरंतर
पहुँच नहीं मिलती। Google API सेटअप के बिना, आपके साइन-इन किए हुए ब्राउज़र
सत्र का उपयोग करके Google Docs को सादे टेक्स्ट के रूप में निर्यात किया जाता है। X और Twitter थ्रेड को
आसपास के इंटरफ़ेस क्रोम के बिना निकाला जाता है।

पेज टेक्स्ट को OpenClaw की बाहरी-सामग्री सुरक्षा सीमा में लपेटा जाता है। आपका
वैकल्पिक नोट आपके अपने निर्देश के रूप में उस सीमा से बाहर रहता है। पेज टेक्स्ट
और चयन लगभग 120,000 वर्णों तक सीमित होते हैं और छोटा किए जाने पर उनमें ट्रंकेशन
मार्कर शामिल होता है।

पेज साझा करना तब काम करता है, जब एक्सटेंशन रिले Gateway द्वारा होस्ट किया जाता है और
समान-होस्ट पेयरिंग या प्रत्यक्ष `wss://` Gateway पेयरिंग का उपयोग होता है। नोड द्वारा होस्ट किए गए रिले
फ़िलहाल स्पष्ट त्रुटि लौटाते हैं। कीबोर्ड शॉर्टकट को रीमैप करने के लिए
`chrome://extensions/shortcuts` खोलें।

## रिमोट / क्रॉस-मशीन

Chrome को Gateway होस्ट पर चलना आवश्यक नहीं है। तीन टोपोलॉजी काम करती हैं:

- **समान होस्ट** (एक मशीन पर Gateway + Chrome): उस मशीन पर
  `openclaw browser extension pair` से पेयर करें। रिले केवल लूपबैक के लिए होता है।
  यदि स्थानीय Gateway TLS का उपयोग करता है, तो उसके सर्टिफ़िकेट का होस्टनेम
  `--gateway-url wss://gateway-host.example` के साथ स्पष्ट रूप से दें; पेयरिंग कभी उसकी जगह लूपबैक IP नहीं लगाती।
- **सीधे रिमोट Gateway से** (आपके लैपटॉप पर Chrome, VPS पर Gateway और
  **लैपटॉप पर कुछ भी नहीं**): Gateway पर
  `openclaw browser extension pair --gateway-url wss://your-gateway.example.com` चलाएँ।
  यह एक `wss://…/browser/extension#<secret>` स्ट्रिंग प्रिंट करता है; लैपटॉप पर एक्सटेंशन
  लोड करके पेयर करें। एक्सटेंशन `wss://` के माध्यम से **सीधे Gateway से** कनेक्ट होता है—
  लैपटॉप पर OpenClaw इंस्टॉल, Node, CLI या खुले इनबाउंड पोर्ट की आवश्यकता नहीं है।
  यह प्रबंधित होस्टिंग का मार्ग है।
- **ब्राउज़र नोड होस्ट के माध्यम से** (पहले से OpenClaw
  नोड चला रही मशीन पर Chrome): नोड पर `pair` चलाएँ और स्थानीय रूप से पेयर करें; Gateway अपने
  मौजूदा प्रमाणीकृत नोड लिंक के माध्यम से ब्राउज़र क्रियाएँ नोड तक प्रॉक्सी करता है।

पेयरिंग सीक्रेट प्रत्येक होस्ट के लिए अलग होता है (प्रत्यक्ष स्थिति में Gateway का) और
Gateway के `/browser/extension` मार्ग द्वारा सत्यापित किया जाता है। प्रत्यक्ष मार्ग के लिए Gateway को
TLS (`wss://`) पर सर्व करें, ताकि पेयरिंग सीक्रेट और CDP ट्रैफ़िक एन्क्रिप्टेड रहें।
सीक्रेट पेयरिंग स्ट्रिंग के URL फ़्रैगमेंट में रहता है और
WebSocket हैंडशेक के दौरान सबप्रोटोकॉल क्रेडेंशियल के रूप में प्रस्तुत किया जाता है, इसलिए सामान्य प्रॉक्सी एक्सेस
लॉग को वह अनुरोध URL में प्राप्त नहीं होता। सुनिश्चित करें कि कोई भी रिवर्स प्रॉक्सी
मानक `Sec-WebSocket-Protocol` हेडर को सुरक्षित रखता है।

## निदान

```bash
openclaw browser status --browser-profile chrome
openclaw browser doctor --browser-profile chrome
```

जब तक एक्सटेंशन पॉपअप **Connected** नहीं दिखाता, तब तक `doctor`
**Chrome एक्सटेंशन रिले** जाँच को विफल बताता है।

## सुरक्षा मॉडल

- रिले केवल लूपबैक से बाइंड होता है; दोनों WebSocket पक्षों का प्रमाणीकरण
  व्युत्पन्न टोकन से होता है और एक्सटेंशन पक्ष का ओरिजिन `chrome-extension://` के विरुद्ध जाँचा जाता है।
- प्रत्यक्ष Gateway पेयरिंग अनुरोध URL में रिले टोकन स्वीकार नहीं करती;
  इसके बजाय बंडल किया हुआ एक्सटेंशन उसे WebSocket सबप्रोटोकॉल सूची में भेजता है।
- एजेंट केवल **OpenClaw टैब समूह** के टैब देख और चला सकता है। आपके
  अन्य टैब निजी रहते हैं।
- साइड-पैनल रन का स्कोप दो बार सीमित किया जाता है: Gateway डिलीवरी प्रति-सत्र
  अनुमतिसूची का उपयोग करती है और ब्राउज़र टूल प्रॉम्प्ट के बाहर भेजी गई Chrome टैब/लक्ष्य
  बाइंडिंग लागू करते हैं।
- `user` (Chrome MCP) प्रोफ़ाइल की तुलना में, जो रिमोट-डीबगिंग प्रॉम्प्ट
  अनुमोदित करते ही आपके पूरे साइन-इन किए हुए ब्राउज़र को उजागर कर देती है, एक्सटेंशन
  साझा सतह को ऐसे टैब समूह तक सीमित रखता है जिसे आप एक नज़र में नियंत्रित कर सकते हैं।

यह भी देखें: पूर्ण प्रोफ़ाइल मॉडल तथा प्रबंधित `openclaw` और Chrome MCP
`user` प्रोफ़ाइल के लिए [ब्राउज़र](/hi/tools/browser)।
