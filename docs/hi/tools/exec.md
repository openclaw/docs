वर्कस्पेस में शेल कमांड चलाएँ। `exec` एक परिवर्तनकारी शेल सतह है: चयनित होस्ट या सैंडबॉक्स फ़ाइल सिस्टम जहाँ भी अनुमति देता है, वहाँ कमांड फ़ाइलें बना, संपादित या हटा सकते हैं। `write`, `edit`, या `apply_patch` जैसे OpenClaw फ़ाइल सिस्टम टूल अक्षम करने से `exec` केवल-पढ़ने योग्य नहीं बनता।

`process` के माध्यम से अग्रभूमि और पृष्ठभूमि निष्पादन समर्थित है। यदि `process` की अनुमति नहीं है, तो `exec` समकालिक रूप से चलता है और `yieldMs`/`background` को अनदेखा करता है। पृष्ठभूमि सत्र प्रत्येक एजेंट के दायरे में होते हैं; `process` केवल उसी एजेंट के सत्र देखता है।

</ParamField>

टिप्पणियाँ:

- `host` केवल `auto`, `sandbox`, `gateway`, या `node` स्वीकार करता है। यह होस्टनाम चयनकर्ता नहीं है; होस्टनाम जैसे मान कमांड चलने से पहले अस्वीकार कर दिए जाते हैं।
- प्रति-कॉल `host=node` की अनुमति `auto` से है; प्रति-कॉल `host=gateway` की अनुमति केवल तभी है, जब कोई सैंडबॉक्स रनटाइम सक्रिय न हो।
- बिना किसी अतिरिक्त कॉन्फ़िगरेशन के भी `host=auto` "बस काम करता है": सैंडबॉक्स न होने पर इसका समाधान `gateway` में होता है; सक्रिय सैंडबॉक्स होने पर यह सैंडबॉक्स में ही रहता है।
- `elevated` सैंडबॉक्स से बाहर कॉन्फ़िगर किए गए होस्ट पथ पर जाता है: डिफ़ॉल्ट रूप से `gateway`, या `tools.exec.host=node` (अथवा सत्र डिफ़ॉल्ट `host=node`) होने पर `node`। यह केवल तभी उपलब्ध है, जब मौजूदा सत्र/प्रदाता के लिए उन्नत पहुँच सक्षम हो।
- `gateway`/`node` अनुमोदन होस्ट अनुमोदन फ़ाइल द्वारा नियंत्रित होते हैं।
- `node` के लिए युग्मित Node (सहायक ऐप या हेडलेस Node होस्ट) आवश्यक है। यदि अनेक Node उपलब्ध हों, तो किसी एक को चुनने के लिए `exec.node` या `tools.exec.node` सेट करें।
- `exec host=node` Node के लिए एकमात्र शेल-निष्पादन पथ है; पुराने `nodes.run` रैपर को हटा दिया गया है।
- गैर-Windows होस्ट पर, सेट होने पर exec `SHELL` का उपयोग करता है; यदि `SHELL`, `fish` है, तो fish-असंगत bash प्रयोगों से बचने के लिए यह `PATH` से `bash` (या `sh`) को प्राथमिकता देता है, फिर इनमें से कोई भी मौजूद न होने पर `SHELL` पर लौटता है।
- Windows होस्ट पर, exec PowerShell 7 (`pwsh`) खोज को प्राथमिकता देता है (Program Files, ProgramW6432, फिर PATH), और फिर Windows PowerShell 5.1 पर लौटता है।
- गैर-Windows gateway होस्ट पर, bash और zsh exec कमांड स्टार्टअप स्नैपशॉट का उपयोग करते हैं। OpenClaw शेल स्टार्टअप फ़ाइलों से सोर्स किए जा सकने वाले उपनाम/फ़ंक्शन और एक छोटा सुरक्षित परिवेश सेट `$OPENCLAW_STATE_DIR/cache/shell-snapshots/` में कैप्चर करता है, फिर प्रत्येक exec कमांड से पहले उस स्नैपशॉट को सोर्स करता है। गुप्त जानकारी जैसे दिखने वाले चर बाहर रखे जाते हैं; सैंडबॉक्स और Node exec इस स्नैपशॉट का उपयोग नहीं करते। इस स्नैपशॉट पथ को अक्षम करने के लिए Gateway प्रक्रिया परिवेश में `OPENCLAW_EXEC_SHELL_SNAPSHOT=0` सेट करें।
- होस्ट निष्पादन (`gateway`/`node`) बाइनरी हाइजैकिंग या अंतःक्षेपित कोड रोकने के लिए `env.PATH` और लोडर ओवरराइड (`LD_*`/`DYLD_*`) अस्वीकार करता है।
- OpenClaw आरंभ किए गए कमांड परिवेश में (PTY और सैंडबॉक्स निष्पादन सहित) `OPENCLAW_SHELL=exec` सेट करता है, ताकि शेल/प्रोफ़ाइल नियम exec-टूल संदर्भ पहचान सकें।
- चैनल से आरंभ हुए रन के लिए, चैनल द्वारा वे आईडी प्रदान किए जाने पर OpenClaw `OPENCLAW_CHANNEL_CONTEXT` में एक सीमित प्रेषक/चैट पहचान JSON पेलोड भी उपलब्ध कराता है।
- `exec`, `openclaw channels login` या `/approve` शेल कमांड नहीं चला सकता: `openclaw channels login` एक इंटरैक्टिव चैनल-प्रमाणीकरण प्रवाह है, और `/approve` को शेल के बजाय अनुमोदन कमांड हैंडलर से होकर जाना आवश्यक है। चैनल लॉगिन को gateway होस्ट के टर्मिनल में चलाएँ, या उपलब्ध होने पर चैनल-विशिष्ट लॉगिन एजेंट टूल का उपयोग करें (उदाहरण के लिए `whatsapp_login`)।
- महत्वपूर्ण: सैंडबॉक्सिंग **डिफ़ॉल्ट रूप से बंद** होती है। यदि सैंडबॉक्सिंग बंद है, तो अंतर्निहित `host=auto` का समाधान `gateway` में होता है। स्पष्ट `host=sandbox`, gateway होस्ट पर चुपचाप चलने के बजाय सुरक्षित रूप से विफल होता है। सैंडबॉक्सिंग सक्षम करें या अनुमोदनों के साथ `host=gateway` का उपयोग करें।
- स्क्रिप्ट पूर्व-जाँच (सामान्य Python/Node शेल-सिंटैक्स गलतियों के लिए) केवल प्रभावी `workdir` सीमा के भीतर की फ़ाइलों का निरीक्षण करती है। यदि किसी स्क्रिप्ट पथ का समाधान `workdir` के बाहर होता है, तो उस फ़ाइल के लिए पूर्व-जाँच छोड़ दी जाती है। `host=gateway` और `ask=off` के साथ प्रभावी नीति `security=full` होने पर पूर्व-जाँच पूरी तरह छोड़ दी जाती है।
- अभी शुरू होने वाले लंबे कार्य को एक बार शुरू करें और सक्षम होने पर, कमांड द्वारा आउटपुट देने या विफल होने पर स्वचालित पूर्णता-जागरण पर निर्भर रहें। लॉग, स्थिति, इनपुट या हस्तक्षेप के लिए `process` का उपयोग करें; sleep लूप, टाइमआउट लूप या बार-बार पोलिंग से शेड्यूलिंग का अनुकरण न करें।
- जो कार्य बाद में या किसी समय-सारणी के अनुसार होना चाहिए, उसके लिए `exec` sleep/विलंब पैटर्न के बजाय cron का उपयोग करें।

बिना अनुमोदन वाला होस्ट exec gateway और node (`security=full`, `ask=off`) के लिए डिफ़ॉल्ट है—यह होस्ट-नीति के डिफ़ॉल्ट से आता है, `host=auto` से नहीं। यदि आप अनुमोदन/अनुमति-सूची व्यवहार चाहते हैं, तो `tools.exec.*` और होस्ट अनुमोदन फ़ाइल, दोनों को कठोर बनाएँ; [Exec अनुमोदन](/hi/tools/exec-approvals#yolo-mode-no-approval) देखें। sandbox स्थिति की परवाह किए बिना gateway या node रूटिंग लागू करने के लिए, `tools.exec.host` सेट करें या `/exec host=...` का उपयोग करें।

उदाहरण:

```json5
{
  tools: {
    exec: {
      pathPrepend: ["~/bin", "/opt/oss/bin"],
    },
  },
}
```

### मोड

`tools.exec.mode` सामान्यीकृत नीति नियंत्रण है। इसे सेट करने से `security`/`ask` व्युत्पन्न होते हैं और इसे स्पष्ट `tools.exec.security`/`tools.exec.ask` के साथ संयोजित नहीं किया जा सकता।

| मोड        | सुरक्षा    | पूछें       | व्यवहार                                                                                                                       |
| ----------- | ----------- | --------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `deny`      | `deny`      | `off`     | Exec अस्वीकृत किया जाता है।                                                                                                                |
| `allowlist` | `allowlist` | `off`     | केवल अनुमति-सूचीबद्ध/सुरक्षित-बाइनरी कमांड चलते हैं; किसी अन्य के लिए नहीं पूछा जाता।                                                                 |
| `ask`       | `allowlist` | `on-miss` | अनुमति-सूची से मेल खाने वाले कमांड सीधे चलते हैं; अन्य सभी के लिए किसी मानव से पूछा जाता है।                                                                  |
| `auto`      | `allowlist` | `on-miss` | अनुमति-सूची/सुरक्षित-बाइनरी से मेल खाने वाले कमांड सीधे चलते हैं; अन्य सभी किसी मानव से पूछने से पहले OpenClaw के मूल स्वचालित समीक्षक के माध्यम से रूट होते हैं। |
| `full`      | `full`      | `off`     | कोई अनुमोदन नियंत्रण नहीं।                                                                                                              |

मोड की परवाह किए बिना `ask`/`ask=always` फिर भी हर बार किसी मानव से पूछता है।

स्वचालित-समीक्षा अनुमोदन केवल एक बार उपयोग होता है। gateway पर, OpenClaw समीक्षक को रिज़ॉल्व किया गया निष्पादन-योग्य पथ देता है और निष्पादन को उसी पथ पर पिन करता है। जिन कमांड को एक लागू करने योग्य निष्पादन योजना में सीमित नहीं किया जा सकता—जैसे heredoc, shell विस्तार या असमर्थित wrapper quoting—उनके लिए मॉडल द्वारा अन्यथा अनुमति दिए जाने पर भी मानव अनुमोदन का फ़ॉलबैक होता है।

Codex app-server कमांड अनुमोदन, जिनका निर्णय स्पष्ट रनटाइम या मूल नीति द्वारा पहले से नहीं हुआ है, मानव अनुमोदन मार्ग का उपयोग करते हैं। OpenClaw इन अनुरोधों के लिए अपने कॉन्फ़िगर किए गए exec समीक्षक को नहीं चलाता, क्योंकि Codex ऐसा लागू करने योग्य रिज़ॉल्व किया गया निष्पादन-योग्य पथ उपलब्ध नहीं कराता जो समीक्षा निर्णय को Codex द्वारा चलाए जाने वाले कमांड से बाँध सके।

### इनलाइन eval (`strictInlineEval`)

जब `tools.exec.strictInlineEval`, `true` होता है, तो इनलाइन interpreter-eval रूपों के लिए समीक्षक या स्पष्ट अनुमोदन आवश्यक होता है: `python -c`, `node -e`, `ruby -e`, `perl -e`, `php -r`, `lua -e`, `osascript -e`, और अन्य समर्थित interpreters तथा command carriers (`awk`, `find -exec`, `make`, `sed`, `xargs`, आदि) में समान रूप। `mode=auto` में, सामान्य exec अनुमोदन पथ मूल स्वचालित समीक्षक को स्पष्ट रूप से कम जोखिम वाले एकबारगी कमांड की अनुमति देने दे सकता है; सीधे node-host `system.run` कॉल के लिए फिर भी स्पष्ट अनुमोदन आवश्यक है, क्योंकि वे कमांड को मानव अनुमोदन मार्ग को नहीं सौंप सकते। यदि समीक्षक पूछने को कहता है, तो अनुरोध किसी मानव के पास जाता है। `allow-always` अब भी निरापद interpreter/script आह्वानों को स्थायी बना सकता है, लेकिन inline-eval रूप टिकाऊ अनुमति नियम नहीं बनते।

### PATH प्रबंधन

- `host=gateway`: आपके login-shell `PATH` को exec वातावरण में मर्ज करता है। होस्ट निष्पादन के लिए `env.PATH` ओवरराइड अस्वीकृत किए जाते हैं। daemon स्वयं फिर भी न्यूनतम `PATH` के साथ चलता है:
  - macOS: `/opt/homebrew/bin`, `/usr/local/bin`, `/usr/bin`, `/bin`
  - Linux: `/usr/local/bin`, `/usr/bin`, `/bin`
  - स्टार्टअप के दौरान उपयोगकर्ता shell कॉन्फ़िगरेशन (जैसे `~/.zshenv` या `/etc/zshenv`) को प्राथमिकता वाले पथ ओवरराइड करने से रोकने के लिए, निष्पादन से ठीक पहले shell कमांड के भीतर अंतिम `PATH` के आरंभ में `tools.exec.pathPrepend` प्रविष्टियाँ सुरक्षित रूप से जोड़ी जाती हैं।
- `host=sandbox`: कंटेनर के भीतर `sh -lc` (login shell) चलाता है, इसलिए `/etc/profile`, `PATH` को रीसेट कर सकता है। OpenClaw आंतरिक env var के माध्यम से प्रोफ़ाइल स्रोत किए जाने के बाद `env.PATH` को आरंभ में जोड़ता है (कोई shell interpolation नहीं); `tools.exec.pathPrepend` यहाँ भी लागू होता है।
- `host=node`: आपके द्वारा दिए गए केवल गैर-अवरुद्ध env ओवरराइड node को भेजे जाते हैं। होस्ट निष्पादन के लिए `env.PATH` ओवरराइड अस्वीकृत किए जाते हैं और node होस्ट उन्हें अनदेखा करते हैं। यदि आपको किसी node पर अतिरिक्त PATH प्रविष्टियों की आवश्यकता है, तो node होस्ट सेवा वातावरण (systemd/launchd) कॉन्फ़िगर करें या टूल को मानक स्थानों पर इंस्टॉल करें।

प्रति-एजेंट node बाइंडिंग (कॉन्फ़िगरेशन में एजेंट सूची इंडेक्स का उपयोग करें):

```bash
openclaw config get agents.list
openclaw config set 'agents.list[0].tools.exec.node' "node-id-or-name"
```

Control UI: **Devices** पृष्ठ में इन्हीं सेटिंग्स के लिए एक छोटा "Exec node binding" पैनल शामिल है।

## सत्र ओवरराइड (`/exec`)

`host`, `security`, `ask`, और `node` के लिए **प्रति-सत्र** डिफ़ॉल्ट सेट करने हेतु `/exec` का उपयोग करें। वर्तमान मान दिखाने के लिए बिना किसी तर्क के `/exec` भेजें।

उदाहरण:

```text
/exec host=auto security=allowlist ask=on-miss node=mac-1
```

`/exec` केवल **अधिकृत प्रेषकों** (चैनल अनुमति-सूचियाँ/पेयरिंग और `commands.useAccessGroups`) के लिए स्वीकार किया जाता है। यह केवल **सत्र स्थिति** अपडेट करता है और कॉन्फ़िगरेशन नहीं लिखता। अधिकृत बाहरी चैनल प्रेषक ये सत्र डिफ़ॉल्ट सेट कर सकते हैं। आंतरिक gateway/webchat क्लाइंट को इन्हें स्थायी रखने के लिए `operator.admin` की आवश्यकता होती है।

exec को पूरी तरह अक्षम करने के लिए, इसे टूल नीति (`tools.deny: ["exec"]` या प्रति-एजेंट) के माध्यम से अस्वीकार करें। होस्ट अनुमोदन तब तक लागू रहते हैं, जब तक आप स्पष्ट रूप से `security=full` और `ask=off` सेट नहीं करते।

## Exec अनुमोदन (सहायक ऐप / node होस्ट)

sandbox किए गए एजेंट gateway या node होस्ट पर `exec` चलने से पहले प्रति-अनुरोध अनुमोदन की आवश्यकता रख सकते हैं। नीति, अनुमति-सूची और UI प्रवाह के लिए [Exec अनुमोदन](/hi/tools/exec-approvals) देखें।

जब मानव अनुमोदन आवश्यक होता है, तो node-host और गैर-मूल gateway प्रवाह `status: "approval-pending"` तथा एक अनुमोदन id के साथ तुरंत लौटते हैं। इसके बजाय मूल चैट और Web UI gateway प्रवाह इनलाइन प्रतीक्षा कर सकते हैं और अनुमोदन के बाद अंतिम कमांड परिणाम लौटा सकते हैं। `approval-pending` परिणाम का अर्थ है कि कमांड शुरू नहीं हुआ है, इसलिए अग्रभूमि फ़ॉलबैक चेतावनियाँ केवल तभी दिखाई देती हैं जब अनुमोदित कमांड वास्तव में इनलाइन चलता है। अनुमोदित अतुल्यकालिक रन कमांड प्रगति और पूर्णता सिस्टम इवेंट (`Exec running` / `Exec finished`) भेजते हैं; अस्वीकृत या समय-सीमा समाप्त अनुमोदन अंतिम होते हैं और एजेंट सत्र को अस्वीकृति सिस्टम इवेंट के साथ सक्रिय नहीं करते।

नेटिव अनुमोदन कार्ड/बटन वाले चैनलों पर, एजेंट को पहले उस नेटिव UI पर निर्भर रहना चाहिए और मैन्युअल `/approve` कमांड केवल तभी शामिल करनी चाहिए जब टूल परिणाम स्पष्ट रूप से बताए कि चैट अनुमोदन उपलब्ध नहीं हैं या मैन्युअल अनुमोदन ही एकमात्र तरीका है।

## अनुमति-सूची + सुरक्षित बिन

मैन्युअल अनुमति-सूची प्रवर्तन समाधान किए गए बाइनरी पथ ग्लोब और केवल कमांड-नाम वाले ग्लोब से मिलान करता है। केवल नाम, सिर्फ़ PATH के माध्यम से चलाए गए कमांड से मेल खाते हैं, इसलिए जब कमांड `rg` हो, तब `rg`, `/opt/homebrew/bin/rg` से मेल खा सकता है, लेकिन `./rg` या `/tmp/rg` से नहीं।

जब `security=allowlist`, शेल कमांड स्वतः केवल तभी अनुमत होते हैं जब प्रत्येक पाइपलाइन खंड अनुमति-सूची में हो या सुरक्षित बिन हो। अनुमति-सूची मोड में चेनिंग (`;`, `&&`, `||`) और पुनर्निर्देशन तब तक अस्वीकार किए जाते हैं जब तक प्रत्येक शीर्ष-स्तरीय खंड अनुमति-सूची को संतुष्ट न करे (सुरक्षित बिन सहित)। पुनर्निर्देशन असमर्थित रहते हैं। स्थायी `allow-always` विश्वास इस नियम को दरकिनार नहीं करता: चेन किए गए कमांड में अब भी प्रत्येक शीर्ष-स्तरीय खंड का मेल खाना आवश्यक है।

`autoAllowSkills`, exec अनुमोदनों में एक अलग सुविधा-पथ है, यह मैन्युअल पथ अनुमति-सूची प्रविष्टियों के समान नहीं है। सख्त स्पष्ट विश्वास के लिए, `autoAllowSkills` को अक्षम रखें।

दोनों नियंत्रणों का उपयोग अलग-अलग कार्यों के लिए करें:

- `tools.exec.safeBins`: छोटे, केवल-stdin स्ट्रीम फ़िल्टर।
- `tools.exec.safeBinTrustedDirs`: सुरक्षित-बिन निष्पादन योग्य पथों के लिए स्पष्ट अतिरिक्त विश्वसनीय डायरेक्टरियाँ।
- `tools.exec.safeBinProfiles`: कस्टम सुरक्षित बिन के लिए स्पष्ट argv नीति।
- allowlist: निष्पादन योग्य पथों के लिए स्पष्ट विश्वास।

`safeBins` को सामान्य अनुमति-सूची न मानें और इसमें इंटरप्रेटर/रनटाइम बाइनरी (उदाहरण के लिए `python3`, `node`, `ruby`, `bash`) न जोड़ें। यदि आपको इनकी आवश्यकता है, तो स्पष्ट अनुमति-सूची प्रविष्टियों का उपयोग करें और अनुमोदन प्रॉम्प्ट सक्षम रखें।

`openclaw security audit` तब चेतावनी देता है जब इंटरप्रेटर/रनटाइम `safeBins` प्रविष्टियों में स्पष्ट प्रोफ़ाइल नहीं होतीं, और `openclaw doctor --fix` अनुपलब्ध कस्टम `safeBinProfiles` प्रविष्टियों का ढाँचा तैयार कर सकता है। जब आप `jq` जैसे व्यापक-व्यवहार वाले बिन को स्पष्ट रूप से वापस `safeBins` में जोड़ते हैं, तब `openclaw security audit` और `openclaw doctor` भी चेतावनी देते हैं (`jq` परिवेश डेटा पढ़ सकता है और मॉड्यूल या स्टार्टअप फ़ाइलों से jq कोड लोड कर सकता है, इसलिए इसके बजाय स्पष्ट अनुमति-सूची प्रविष्टियों या अनुमोदन-नियंत्रित रन को प्राथमिकता दें)। `jq` को सुरक्षित बिन के रूप में अस्वीकार किया जाता है, भले ही वह स्पष्ट रूप से सूचीबद्ध हो। यदि आप इंटरप्रेटर को स्पष्ट रूप से अनुमति-सूची में रखते हैं, तो `tools.exec.strictInlineEval` सक्षम करें ताकि इनलाइन कोड-मूल्यांकन रूपों के लिए फिर भी समीक्षक या स्पष्ट अनुमोदन आवश्यक हो।

पूरी नीति के विवरण और उदाहरणों के लिए, [Exec अनुमोदन](/hi/tools/exec-approvals-advanced#safe-bins-stdin-only) और [सुरक्षित बिन बनाम अनुमति-सूची](/hi/tools/exec-approvals-advanced#safe-bins-versus-allowlist) देखें।

## उदाहरण

अग्रभूमि:

```json
{ "tool": "exec", "command": "ls -la" }
```

पृष्ठभूमि + पोल:

```json
{"tool":"exec","command":"npm run build","yieldMs":1000}
{"tool":"process","action":"poll","sessionId":"<id>"}
```

पोलिंग माँग पर स्थिति जानने के लिए है, प्रतीक्षा लूप के लिए नहीं। यदि स्वचालित पूर्णता-वेक सक्षम है, तो आउटपुट उत्सर्जित करने या विफल होने पर कमांड सत्र को जगा सकता है।

कुंजियाँ भेजें (tmux-शैली):

```json
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["Enter"]}
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["C-c"]}
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["Up","Up","Enter"]}
```

सबमिट करें (केवल CR भेजें):

```json
{ "tool": "process", "action": "submit", "sessionId": "<id>" }
```

पेस्ट करें (डिफ़ॉल्ट रूप से ब्रैकेटेड):

```json
{ "tool": "process", "action": "paste", "sessionId": "<id>", "text": "line1\nline2\n" }
```

## apply_patch

`apply_patch`, संरचित बहु-फ़ाइल संपादनों के लिए `exec` का एक उप-टूल है। यह डिफ़ॉल्ट रूप से सक्षम है और किसी भी मॉडल प्रदाता के लिए उपलब्ध है; `allowModels` इसे प्रतिबंधित कर सकता है। कॉन्फ़िग का उपयोग केवल तब करें जब आप इसे अक्षम करना चाहते हों या इसे विशिष्ट मॉडलों तक सीमित करना चाहते हों:

```json5
{
  tools: {
    exec: {
      applyPatch: { workspaceOnly: true, allowModels: ["gpt-5.6-sol"] },
    },
  },
}
```

टिप्पणियाँ:

- टूल नीति अब भी लागू होती है; `allow: ["write"]`, `apply_patch` को अंतर्निहित रूप से अनुमति देता है।
- `deny: ["write"]`, `apply_patch` को अस्वीकार नहीं करता; `apply_patch` को स्पष्ट रूप से अस्वीकार करें या जब पैच लेखन भी अवरुद्ध होना चाहिए तब `deny: ["group:fs"]` का उपयोग करें।
- कॉन्फ़िग `tools.exec.applyPatch` के अंतर्गत रहता है।
- `tools.exec.applyPatch.enabled` का डिफ़ॉल्ट `true` है; टूल को अक्षम करने के लिए इसे `false` पर सेट करें।
- `tools.exec.applyPatch.workspaceOnly` का डिफ़ॉल्ट `true` (वर्कस्पेस के भीतर सीमित) है। इसे `false` पर केवल तभी सेट करें जब आप जानबूझकर चाहते हों कि `apply_patch` वर्कस्पेस डायरेक्टरी के बाहर लिखे/हटाए।
- `tools.exec.applyPatch.allowModels` मॉडल आईडी की वैकल्पिक अनुमति-सूची है (रॉ, जैसे `gpt-5.4`, या पूर्ण, जैसे `openai/gpt-5.4`)। सेट होने पर केवल मेल खाने वाले मॉडलों को टूल मिलता है; सेट न होने पर सभी मॉडलों को यह मिलता है।
