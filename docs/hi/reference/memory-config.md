जब तक अन्यथा उल्लेख न किया गया हो, सभी मेमोरी खोज सेटिंग्स `openclaw.json` में `agents.defaults.memorySearch` के अंतर्गत (या प्रति-एजेंट `agents.list[].memorySearch` ओवरराइड में) होती हैं।

<Note>
अनुशंसित व्यक्तिगत-एजेंट कार्यप्रवाह के लिए,
`memorySearch.rememberAcrossConversations` का उपयोग करें। उन्नत Active Memory लक्ष्यीकरण,
मॉडल, प्रॉम्प्ट और विलंबता नियंत्रण `plugins.entries.active-memory` के अंतर्गत होते हैं।

सक्रियण के दोनों मार्गों, ट्रांस्क्रिप्ट स्थायित्व और सुरक्षित रोलआउट मार्गदर्शन के लिए
[Active Memory](/hi/concepts/active-memory) देखें।
</Note>

## वार्तालापों के बीच याद रखना

| कुंजी                           | प्रकार      | डिफ़ॉल्ट                                                    | विवरण                                                                    |
| ----------------------------- | --------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `rememberAcrossConversations` | `boolean` | व्यक्तिगत इंस्टॉल के लिए चालू; कॉन्फ़िगर किए गए DM पृथक्करण के साथ बंद | इस एजेंट के अन्य पहचाने गए निजी वार्तालापों से प्रासंगिक संदर्भ का उपयोग करें। |

जब केवल किसी विश्वसनीय व्यक्तिगत एजेंट को वार्तालापों के बीच ट्रांस्क्रिप्ट
स्मरण का उपयोग करना चाहिए, तो इसे प्रति एजेंट कॉन्फ़िगर करें:

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        memorySearch: {
          rememberAcrossConversations: true,
        },
      },
    ],
  },
}
```

यह मान प्रति-एजेंट ओवरराइड के साथ सामान्य `agents.defaults.memorySearch` इनहेरिटेंस का
पालन करता है। सेट न होने पर, यह केवल तभी डिफ़ॉल्ट रूप से चालू होता है जब वैश्विक
`session.dmScope` सेट न हो या `"main"` हो और किसी बाइंडिंग में `session.dmScope`
ओवरराइड न हो। कॉन्फ़िगर किया गया कोई भी DM पृथक्करण इसे डिफ़ॉल्ट रूप से बंद कर देता है। स्पष्ट `true` या
`false` हमेशा प्रभावी होता है। इसे सक्षम करने से सत्र ट्रांस्क्रिप्ट इंडेक्सिंग निहित होती है और
एजेंट के हल किए गए मेमोरी स्रोतों में `sessions` जुड़ जाता है। QMD के साथ, यह
उस एजेंट के सत्र निर्यात को भी सक्षम करता है; इस मोड के लिए अलग
`memory.qmd.sessions.enabled` सेटिंग आवश्यक नहीं है।

OpenClaw का अंतर्निहित मेमोरी प्रदाता, अंतर्निहित और QMD दोनों बैकएंड के साथ इस
संरक्षित मार्ग का समर्थन करता है। वैकल्पिक मेमोरी प्रदाता अपने स्वयं के
स्मरण हुक और उन्नत Active Memory टूल का उपयोग जारी रख सकते हैं, लेकिन यह सेटिंग
तब तक छोड़ दी जाती है जब तक वर्तमान प्रदाता संरक्षित निजी ट्रांस्क्रिप्ट स्मरण का समर्थन न करे।
`openclaw doctor` किसी असमर्थित प्रदाता या ऐसी स्पष्ट Active Memory
`toolsAllow` सूची की रिपोर्ट करता है जिसमें `memory_search` शामिल नहीं है।

पुनर्प्राप्ति सीमा सामान्य सत्र खोज की तुलना में अधिक संकीर्ण है:

- केवल उसी एजेंट के पहचाने गए निजी वार्तालाप पात्र हैं
- जिस वार्तालाप का उत्तर दिया जा रहा है, उसे बाहर रखा जाता है
- समूहों और चैनलों को स्रोतों तथा गंतव्यों के रूप में बाहर रखा जाता है
- अज्ञात वार्तालाप प्रकार फ़ेल-क्लोज़ होते हैं
- सैंडबॉक्स किया गया स्मरण विशेष क्रॉस-कन्वर्सेशन प्राधिकरण का उपयोग नहीं कर सकता

यह सेटिंग `tools.sessions.visibility`, सत्र कुंजियों,
ट्रांस्क्रिप्ट भंडारण, डिलीवरी रूटिंग या `sessions_list`,
`sessions_history`, और `sessions_send` की अनुमतियों को नहीं बदलती। Active Memory एक सीमित
केवल-पठन पुनर्प्राप्ति पास करता है; अनुपलब्ध या समय-समाप्त पुनर्प्राप्ति
उत्तर को अवरुद्ध नहीं करती।

## प्रदाता चयन

| कुंजी        | प्रकार      | डिफ़ॉल्ट          | विवरण                                                                                                                                                                                                                                                                                 |
| ---------- | --------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`  | `boolean` | `true`           | मेमोरी खोज सक्षम या अक्षम करें                                                                                                                                                                                                                                                             |
| `provider` | `string`  | `"openai"`       | एम्बेडिंग अडैप्टर ID, जैसे `bedrock`, `deepinfra`, `gemini`, `github-copilot`, `local`, `mistral`, `ollama`, `openai`, `openai-compatible`, या `voyage`; यह कॉन्फ़िगर किया गया ऐसा `models.providers.<id>` भी हो सकता है जिसका `api` किसी मेमोरी एम्बेडिंग अडैप्टर या OpenAI-संगत मॉडल API की ओर संकेत करता हो |
| `model`    | `string`  | प्रदाता डिफ़ॉल्ट | एम्बेडिंग मॉडल का नाम                                                                                                                                                                                                                                                                        |
| `fallback` | `string`  | `"none"`         | प्राथमिक के विफल होने पर फ़ॉलबैक अडैप्टर ID                                                                                                                                                                                                                                                  |

जब `provider` सेट नहीं होता, तो OpenClaw OpenAI एम्बेडिंग का उपयोग करता है। Bedrock, DeepInfra, Gemini, GitHub Copilot, Mistral, Ollama,
Voyage, किसी स्थानीय GGUF मॉडल या OpenAI-संगत `/v1/embeddings` एंडपॉइंट का उपयोग करने के लिए `provider`
स्पष्ट रूप से सेट करें।
वे पुराने कॉन्फ़िगरेशन जिनमें अभी भी `provider: "auto"` है, `openai` में हल होते हैं।

<Warning>
एम्बेडिंग प्रदाता, मॉडल, प्रदाता सेटिंग्स, स्रोत, दायरा,
चंकिंग या टोकनाइज़र बदलने से मौजूदा SQLite वेक्टर इंडेक्स असंगत हो सकता है।
OpenClaw स्वचालित रूप से सब कुछ फिर से एम्बेड करने के बजाय वेक्टर खोज रोक देता है और
इंडेक्स पहचान चेतावनी की रिपोर्ट करता है। तैयार होने पर
`openclaw memory status --index --agent <id>` या
`openclaw memory index --force --agent <id>` से पुनर्निर्माण करें।
</Warning>

जब `provider` सेट न हो, पुराना `provider: "auto"` मौजूद हो, या
`provider: "none"` जानबूझकर केवल-FTS मोड चुने, तो एम्बेडिंग अनुपलब्ध होने पर भी
मेमोरी स्मरण लेक्सिकल FTS रैंकिंग का उपयोग कर सकता है।

स्पष्ट गैर-स्थानीय प्रदाता फ़ेल-क्लोज़ होते हैं। यदि आप `memorySearch.provider` को
Bedrock, DeepInfra, Gemini, GitHub
Copilot, LM Studio, Mistral, Ollama, OpenAI, Voyage या OpenAI-संगत
कस्टम प्रदाता जैसे किसी निश्चित रिमोट-समर्थित प्रदाता पर सेट करते हैं और वह प्रदाता रनटाइम पर अनुपलब्ध है, तो `memory_search`
चुपचाप केवल-FTS स्मरण का उपयोग करने के बजाय अनुपलब्ध परिणाम लौटाता है। प्रदाता/प्रमाणीकरण
कॉन्फ़िगरेशन ठीक करें, किसी उपलब्ध प्रदाता पर स्विच करें, या यदि आप जानबूझकर केवल-FTS स्मरण चाहते हैं तो
`provider: "none"` सेट करें।

### कस्टम प्रदाता ID

`memorySearch.provider` मेमोरी-विशिष्ट प्रदाता अडैप्टरों, जैसे `ollama`, या OpenAI-संगत मॉडल API, जैसे `openai-responses` / `openai-completions`, के लिए किसी कस्टम `models.providers.<id>` प्रविष्टि की ओर संकेत कर सकता है। OpenClaw कस्टम प्रदाता ID को एंडपॉइंट, प्रमाणीकरण और मॉडल-प्रीफ़िक्स प्रबंधन के लिए संरक्षित रखते हुए एम्बेडिंग अडैप्टर के लिए उस प्रदाता के `api` स्वामी को हल करता है। इससे मल्टी-GPU या मल्टी-होस्ट सेटअप किसी विशिष्ट स्थानीय एंडपॉइंट को मेमोरी एम्बेडिंग के लिए समर्पित कर सकते हैं:

```json5
{
  models: {
    providers: {
      "ollama-5080": {
        api: "ollama",
        baseUrl: "http://gpu-box.local:11435",
        apiKey: "ollama-local",
        models: [{ id: "qwen3-embedding:0.6b", name: "Qwen3 Embedding 0.6B" }],
      },
    },
  },
  agents: {
    defaults: {
      memorySearch: {
        provider: "ollama-5080",
        model: "qwen3-embedding:0.6b",
      },
    },
  },
}
```

### API कुंजी समाधान

रिमोट एम्बेडिंग के लिए API कुंजी आवश्यक है। इसके बजाय Bedrock AWS SDK की डिफ़ॉल्ट क्रेडेंशियल शृंखला का उपयोग करता है (इंस्टेंस भूमिकाएँ, SSO, एक्सेस कुंजियाँ या Bedrock API कुंजी)।

| प्रदाता       | परिवेश चर                                             | कॉन्फ़िगरेशन कुंजी                          |
| -------------- | --------------------------------------------------- | ----------------------------------- |
| Bedrock        | AWS क्रेडेंशियल शृंखला, या `AWS_BEARER_TOKEN_BEDROCK` | API कुंजी आवश्यक नहीं                   |
| DeepInfra      | `DEEPINFRA_API_KEY`                                 | `models.providers.deepinfra.apiKey` |
| Gemini         | `GEMINI_API_KEY`                                    | `models.providers.google.apiKey`    |
| GitHub Copilot | `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, `GITHUB_TOKEN`  | डिवाइस लॉगिन के माध्यम से प्रमाणीकरण प्रोफ़ाइल       |
| Mistral        | `MISTRAL_API_KEY`                                   | `models.providers.mistral.apiKey`   |
| Ollama         | `OLLAMA_API_KEY` (प्लेसहोल्डर)                      | --                                  |
| OpenAI         | `OPENAI_API_KEY`                                    | `models.providers.openai.apiKey`    |
| Voyage         | `VOYAGE_API_KEY`                                    | `models.providers.voyage.apiKey`    |

<Note>
Codex OAuth केवल चैट/कम्प्लीशन को कवर करता है और एम्बेडिंग अनुरोधों को पूरा नहीं करता।
</Note>

## रिमोट एंडपॉइंट कॉन्फ़िगरेशन

ऐसे सामान्य OpenAI-संगत `/v1/embeddings` सर्वर के लिए `provider: "openai-compatible"` का उपयोग करें
जिसे वैश्विक OpenAI चैट क्रेडेंशियल इनहेरिट नहीं करने चाहिए।

<ParamField path="remote.baseUrl" type="string">
  कस्टम API बेस URL।
</ParamField>
<ParamField path="remote.apiKey" type="string">
  API कुंजी ओवरराइड करें।
</ParamField>
<ParamField path="remote.headers" type="object">
  अतिरिक्त HTTP हेडर (प्रदाता डिफ़ॉल्ट के साथ मर्ज किए जाते हैं)।
</ParamField>

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai-compatible",
        model: "text-embedding-3-small",
        remote: {
          baseUrl: "https://api.example.com/v1/",
          apiKey: "YOUR_KEY",
        },
      },
    },
  },
}
```

    | ------------------- | -------- | ------- | -------------------------------------------------------- |
    | `inputType`         | `string` | सेट नहीं   | क्वेरी और दस्तावेज़ एम्बेडिंग के लिए साझा `input_type`   |
    | `queryInputType`    | `string` | सेट नहीं   | क्वेरी-समय का `input_type`; `inputType` को ओवरराइड करता है          |
    | `documentInputType` | `string` | सेट नहीं   | इंडेक्स/दस्तावेज़ `input_type`; `inputType` को ओवरराइड करता है      |

      agents: {
        defaults: {
          memorySearch: {
            provider: "openai-compatible",
            remote: {
              baseUrl: "https://embeddings.example/v1",
              apiKey: "${EMBEDDINGS_API_KEY}",
            },
            model: "asymmetric-embedder",
            queryInputType: "query",
            documentInputType: "passage",
          },
        },
      },

## अतिरिक्त मेमोरी पथ

| कुंजी          | प्रकार       | विवरण                              |
| ------------ | ---------- | ---------------------------------------- |
| `extraPaths` | `string[]` | इंडेक्स करने के लिए अतिरिक्त डायरेक्टरियाँ या फ़ाइलें |

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        extraPaths: ["../team-docs", "/srv/shared-notes"],
      },
    },
  },
}
```

पथ निरपेक्ष या वर्कस्पेस-सापेक्ष हो सकते हैं। डायरेक्टरियों में `.md` फ़ाइलों के लिए पुनरावर्ती रूप से स्कैन किया जाता है। सिमलिंक प्रबंधन सक्रिय बैकएंड पर निर्भर करता है: बिल्टइन इंजन सिमलिंक छोड़ देता है, जबकि QMD अंतर्निहित QMD स्कैनर के व्यवहार का अनुसरण करता है।

एजेंट-स्कोप वाले क्रॉस-एजेंट ट्रांसक्रिप्ट खोज के लिए, `memory.qmd.paths` के बजाय `agents.list[].memorySearch.qmd.extraCollections` का उपयोग करें। वे अतिरिक्त संग्रह उसी `{ path, name, pattern? }` संरचना का पालन करते हैं, लेकिन उन्हें प्रति एजेंट मर्ज किया जाता है और जब पथ वर्तमान वर्कस्पेस के बाहर इंगित करता है, तो वे स्पष्ट साझा नामों को बनाए रख सकते हैं। यदि वही रिज़ॉल्व किया गया पथ `memory.qmd.paths` और `memorySearch.qmd.extraCollections` दोनों में दिखाई देता है, तो QMD पहली प्रविष्टि रखता है और डुप्लिकेट को छोड़ देता है।

| `remote.batch.concurrency`    | `number`  | `2`     | समानांतर बैच जॉब        |

| `remote.batch.timeoutMinutes` | `number`  | `60`    | बैच टाइमआउट              |

`gemini`, `openai`, और `voyage` के लिए उपलब्ध है। बड़े बैकफ़िल के लिए OpenAI बैच आमतौर पर सबसे तेज़ और सबसे सस्ता होता है।

`remote.nonBatchConcurrency` स्थानीय/स्वयं-होस्ट किए गए प्रोवाइडर और होस्ट किए गए प्रोवाइडर द्वारा उपयोग की जाने वाली इनलाइन एम्बेडिंग कॉल को नियंत्रित करता है, जब प्रोवाइडर बैच API सक्रिय नहीं होते। छोटे स्थानीय होस्ट पर अत्यधिक भार पड़ने से बचाने के लिए, गैर-बैच इंडेक्सिंग हेतु Ollama का डिफ़ॉल्ट `1` है; बड़ी मशीनों पर अधिक मान सेट करें।

यह `sync.embeddingBatchTimeoutSeconds` से अलग है, जो इनलाइन एम्बेडिंग कॉल का टाइमआउट नियंत्रित करता है।

## सेशन मेमोरी खोज (प्रयोगात्मक)

सेशन ट्रांसक्रिप्ट को इंडेक्स करें और उन्हें `memory_search` के माध्यम से उपलब्ध कराएँ:

| कुंजी                           | प्रकार       | डिफ़ॉल्ट      | विवरण                             |
| ----------------------------- | ---------- | ------------ | --------------------------------------- |
| `experimental.sessionMemory`  | `boolean`  | `false`      | सेशन इंडेक्सिंग सक्षम करें                 |
| `sources`                     | `string[]` | `["memory"]` | ट्रांसक्रिप्ट शामिल करने के लिए `"sessions"` जोड़ें |
| `sync.sessions.deltaBytes`    | `number`   | `100000`     | रीइंडेक्स के लिए बाइट सीमा              |
| `sync.sessions.deltaMessages` | `number`   | `50`         | रीइंडेक्स के लिए संदेश सीमा           |

<Warning>
सेशन इंडेक्सिंग वैकल्पिक है और एसिंक्रोनस रूप से चलती है। परिणाम थोड़े पुराने हो सकते हैं। सेशन लॉग डिस्क पर रहते हैं, इसलिए फ़ाइल सिस्टम पहुँच को विश्वास सीमा मानें।
</Warning>

मॉडल द्वारा सामान्य रूप से शुरू की गई सेशन ट्रांसक्रिप्ट खोज
[`tools.sessions.visibility`](/hi/gateway/config-tools#toolssessions) का पालन करती है। डिफ़ॉल्ट
`tree` दृश्यता वर्तमान सेशन, उसके द्वारा शुरू किए गए सेशन और
परिवेशी समूह जागरूकता के माध्यम से देखे जाने वाले समान-एजेंट समूह सेशन को उपलब्ध कराती है। अन्य
असंबंधित सेशन के लिए `agent` दृश्यता आवश्यक है (या केवल तब `all`, जब क्रॉस-एजेंट
रिकॉल भी आवश्यक हो और एजेंट-से-एजेंट नीति इसकी अनुमति देती हो)।

`rememberAcrossConversations` उस सेटिंग का दायरा नहीं बढ़ाता। यह सीमित Active Memory पास के दौरान
समान-एजेंट निजी ट्रांसक्रिप्ट तक सीमित, अलग और केवल-रनटाइम प्राधिकरण प्रदान करता है।

नीचे दिए गए उदाहरण इन सेटिंग को `agents.defaults` के अंतर्गत रखते हैं। जब केवल एक
एजेंट को सेशन ट्रांसक्रिप्ट इंडेक्स और खोजने चाहिए, तब आप प्रति-एजेंट ओवरराइड में समकक्ष
`memorySearch` सेटिंग भी लागू कर सकते हैं।

समान-एजेंट Gateway-से-DM रिकॉल के लिए:

<Tabs>
  <Tab title="अंतर्निहित बैकएंड">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
  <Tab title="QMD बैकएंड">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      memory: {
        backend: "qmd",
        qmd: {
          sessions: { enabled: true },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
</Tabs>

QMD का उपयोग करते समय, `agents.defaults.memorySearch.experimental.sessionMemory` और
`sources: ["sessions"]` अपने आप ट्रांसक्रिप्ट को QMD में एक्सपोर्ट नहीं करते। साथ ही
`memory.qmd.sessions.enabled: true` भी सेट करें। उच्च-स्तरीय
`rememberAcrossConversations: true` सेटिंग इसका अपवाद है: यह उस एजेंट के लिए आवश्यक
QMD सेशन एक्सपोर्ट को अंतर्निहित करती है। अंतर्निहित एक्सपोर्ट निजी रहते हैं:
वे हमेशा डिफ़ॉल्ट आंतरिक एक्सपोर्ट स्थान का उपयोग करते हैं (कॉन्फ़िगर किया गया
`sessions.exportDir` केवल स्पष्ट एक्सपोर्ट पर लागू होता है), उन्हें केवल
उस एजेंट के क्रॉस-कन्वर्सेशन रिकॉल के दौरान खोजा जाता है, और सामान्य `memory_get`
उन्हें नहीं पढ़ सकता। स्पष्ट
`memory.qmd.sessions.enabled: true` अपने मौजूदा व्यवहार को बनाए रखता है और
एक्सपोर्ट किए गए ट्रांसक्रिप्ट को सामान्य मेमोरी कॉर्पस का हिस्सा बनाता है।

## QMD बैकएंड कॉन्फ़िगरेशन

सक्षम करने के लिए `memory.backend = "qmd"` सेट करें। सभी QMD सेटिंग `memory.qmd` के अंतर्गत होती हैं:

| कुंजी                      | प्रकार      | डिफ़ॉल्ट  | विवरण                                                                           |
| ------------------------ | --------- | -------- | ------------------------------------------------------------------------------------- |
| `command`                | `string`  | `qmd`    | QMD एक्ज़ीक्यूटेबल पथ; जब सेवा `PATH` आपके शेल से अलग हो, तब निरपेक्ष पथ सेट करें |
| `searchMode`             | `string`  | `search` | खोज कमांड: `search`, `vsearch`, `query`                                          |
| `rerank`                 | `boolean` | --       | QMD रीरैंकिंग छोड़ने के लिए `searchMode: "query"` और QMD 2.1+ के साथ `false` पर सेट करें          |
| `includeDefaultMemory`   | `boolean` | `true`   | `MEMORY.md` + `memory/**/*.md` को स्वतः इंडेक्स करें                                             |
| `paths[]`                | `array`   | --       | अतिरिक्त पथ: `{ name, path, pattern? }`                                               |
| `sessions.enabled`       | `boolean` | `false`  | सेशन ट्रांसक्रिप्ट को QMD में एक्सपोर्ट करें                                                   |
| `sessions.retentionDays` | `number`  | --       | ट्रांसक्रिप्ट प्रतिधारण                                                                  |
| `sessions.exportDir`     | `string`  | --       | एक्सपोर्ट डायरेक्टरी                                                                      |

`searchMode: "search"` केवल लेक्सिकल/BM25 है। OpenClaw इस मोड के लिए सिमेंटिक वेक्टर तत्परता जाँच या QMD एम्बेडिंग रखरखाव नहीं चलाता, जिसमें `memory status --deep` के दौरान भी शामिल है; `vsearch` और `query` के लिए QMD वेक्टर तत्परता और एम्बेडिंग आवश्यक बनी रहती हैं।

`rerank: false` केवल QMD `query` मोड बदलता है और इसके लिए QMD 2.1 या नया संस्करण आवश्यक है। प्रत्यक्ष CLI मोड में OpenClaw `--no-rerank` पास करता है; mcporter-समर्थित MCP मोड में यह QMD के एकीकृत क्वेरी टूल को `rerank: false` पास करता है। QMD के डिफ़ॉल्ट क्वेरी रीरैंकिंग व्यवहार का उपयोग करने के लिए इसे सेट न करें।

OpenClaw वर्तमान QMD संग्रह और MCP क्वेरी संरचनाओं को प्राथमिकता देता है, लेकिन आवश्यकता होने पर संगत संग्रह पैटर्न फ़्लैग और पुराने MCP टूल नाम आज़माकर पुराने QMD रिलीज़ को भी कार्यशील रखता है। जब QMD एकाधिक संग्रह फ़िल्टर के लिए समर्थन घोषित करता है, तब समान-स्रोत संग्रहों को एक QMD प्रोसेस से खोजा जाता है; पुराने QMD बिल्ड प्रति-संग्रह संगतता पथ बनाए रखते हैं। समान-स्रोत का अर्थ है कि टिकाऊ मेमोरी संग्रहों (डिफ़ॉल्ट मेमोरी फ़ाइलें और कस्टम पथ) को एक साथ समूहित किया जाता है, जबकि सेशन ट्रांसक्रिप्ट संग्रह अलग समूह रहते हैं, ताकि स्रोत विविधीकरण में दोनों इनपुट बने रहें।

<Note>
QMD मॉडल ओवरराइड QMD की ओर रहते हैं, OpenClaw कॉन्फ़िगरेशन में नहीं। यदि आपको QMD के मॉडल वैश्विक रूप से ओवरराइड करने हैं, तो Gateway रनटाइम परिवेश में `QMD_EMBED_MODEL`, `QMD_RERANK_MODEL`, और `QMD_GENERATE_MODEL` जैसे पर्यावरण चर सेट करें।
</Note>

### mcporter एकीकरण

सभी सेटिंग `memory.qmd.mcporter` के अंतर्गत हैं। प्रति क्वेरी `qmd` शुरू करने के बजाय QMD खोजों को लंबे समय तक चलने वाले `mcporter` MCP डेमन के माध्यम से रूट करता है, जिससे बड़े मॉडल के लिए कोल्ड-स्टार्ट ओवरहेड कम होता है।

| कुंजी           | प्रकार      | डिफ़ॉल्ट | विवरण                                                            |
| ------------- | --------- | ------- | ---------------------------------------------------------------------- |
| `enabled`     | `boolean` | `false` | प्रति अनुरोध `qmd` शुरू करने के बजाय QMD कॉल को mcporter के माध्यम से रूट करें |
| `serverName`  | `string`  | `qmd`   | `lifecycle: keep-alive` के साथ `qmd mcp` चलाने वाला mcporter सर्वर नाम  |
| `startDaemon` | `boolean` | `true`  | `enabled` के true होने पर mcporter डेमन स्वचालित रूप से शुरू करें         |

इसके लिए `mcporter` का इंस्टॉल और PATH पर उपलब्ध होना आवश्यक है, साथ ही ऐसा कॉन्फ़िगर किया गया mcporter सर्वर चाहिए जो `qmd mcp` चलाता हो। सरल स्थानीय सेटअप के लिए इसे अक्षम रखें, जहाँ प्रति-क्वेरी प्रोसेस शुरू करने की लागत स्वीकार्य है।

  </Accordion>

जब Gateway-स्टार्ट QMD आरंभीकरण सक्षम होता है, OpenClaw केवल पात्र एजेंटों के लिए QMD शुरू करता है। यदि `update.onBoot` true है और कोई अंतराल/एम्बेड रखरखाव कॉन्फ़िगर नहीं है, तो स्टार्टअप बूट रीफ़्रेश के लिए एकबारगी प्रबंधक का उपयोग करके उसे बंद कर देता है। यदि कोई अपडेट या एम्बेड अंतराल कॉन्फ़िगर है, तो स्टार्टअप लंबे समय तक सक्रिय QMD प्रबंधक खोलता है, ताकि वह वॉचर और अंतराल टाइमर का स्वामी हो सके; `update.onBoot: false` केवल तत्काल बूट रीफ़्रेश को छोड़ता है।

### पूर्ण QMD उदाहरण

```json5
{
  memory: {
    backend: "qmd",
    citations: "auto",
    qmd: {
      includeDefaultMemory: true,
      update: { interval: "5m", debounceMs: 15000 },
      limits: { maxResults: 4, timeoutMs: 4000 },
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

## Dreaming

Dreaming को `plugins.entries.memory-core.config.dreaming` के अंतर्गत कॉन्फ़िगर किया जाता है, `agents.defaults.memorySearch` के अंतर्गत नहीं।

Dreaming एक निर्धारित स्वीप के रूप में चलता है और कार्यान्वयन विवरण के रूप में आंतरिक हल्के/गहन/REM चरणों का उपयोग करता है।

अवधारणात्मक व्यवहार और स्लैश कमांड के लिए, [Dreaming](/hi/concepts/dreaming) देखें।

### उपयोगकर्ता सेटिंग्स

| कुंजी                                    | प्रकार      | डिफ़ॉल्ट       | विवरण                                                                                                                      |
| -------------------------------------- | --------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`                              | `boolean` | `false`       | Dreaming को पूरी तरह सक्षम या अक्षम करें                                                                                              |
| `frequency`                            | `string`  | `0 3 * * *`   | पूर्ण Dreaming स्वीप के लिए वैकल्पिक Cron आवृत्ति                                                                                |
| `model`                                | `string`  | डिफ़ॉल्ट मॉडल | वैकल्पिक Dream Diary सबएजेंट मॉडल ओवरराइड                                                                                     |
| `phases.deep.maxPromotedSnippetTokens` | `number`  | `160`         | `MEMORY.md` में पदोन्नत प्रत्येक अल्पकालिक रिकॉल स्निपेट से रखे जाने वाले अधिकतम अनुमानित टोकन; उद्गम मेटाडेटा दृश्यमान रहता है |

### उदाहरण

```json5
{
  plugins: {
    entries: {
      "memory-core": {
        subagent: {
          allowModelOverride: true,
          allowedModels: ["anthropic/claude-sonnet-4-6"],
        },
        config: {
          dreaming: {
            enabled: true,
            frequency: "0 3 * * *",
            model: "anthropic/claude-sonnet-4-6",
          },
        },
      },
    },
  },
}
```

<Note>
- Dreaming मशीन स्थिति को `memory/.dreams/` में लिखता है।
- Dreaming मानव-पठनीय वर्णनात्मक आउटपुट को `DREAMS.md` (या मौजूदा `dreams.md`) में लिखता है।
- `dreaming.model` मौजूदा Plugin सबएजेंट विश्वास गेट का उपयोग करता है; इसे सक्षम करने से पहले `plugins.entries.memory-core.subagent.allowModelOverride: true` सेट करें।
- कॉन्फ़िगर किया गया मॉडल अनुपलब्ध होने पर Dream Diary सत्र के डिफ़ॉल्ट मॉडल के साथ एक बार पुनः प्रयास करता है। विश्वास या अनुमत-सूची की विफलताएँ लॉग की जाती हैं और उनके लिए बिना बताए पुनः प्रयास नहीं किया जाता।
- हल्के/गहन/REM चरण की नीति और सीमाएँ आंतरिक व्यवहार हैं, उपयोगकर्ता-सामना करने वाला कॉन्फ़िगरेशन नहीं।

</Note>
