`Full Release Validation` रिलीज़ उत्पाद-सत्यापन का समग्र आवरण है। अधिकांश कार्य
चाइल्ड वर्कफ़्लो में होता है, ताकि विफल बॉक्स को पूरी रिलीज़ पुनः आरंभ किए बिना
दोबारा चलाया जा सके। Code SHA को फ़्रीज़ करने से पहले रिलीज़ तैयारी चलाएँ; यदि
बैकग्राउंड बॉट ने अभी तक Control UI लोकेल आउटपुट लैंड नहीं किया है, तो यह उसे
रीफ़्रेश करती है और फिर रिलीज़ CI द्वारा उपयोग की जाने वाली उसी सख्त शून्य-फ़ॉलबैक जाँच को लागू करती है।

चेंजलॉग-पूर्व उत्पाद-पूर्ण कमिट को **Code SHA** के रूप में फ़्रीज़ करें, फिर चलाएँ:

```bash
pnpm ci:full-release \
  --sha <code-sha> \
  --target-ref release/YYYY.M.PATCH
```

`provider` क्रॉस-OS ऑनबोर्डिंग और एंड-टू-एंड एजेंट टर्न के लिए `anthropic` या `minimax` भी स्वीकार करता है। हेल्पर अल्फ़ा/बीटा
पैकेज संस्करणों से `beta` प्रोफ़ाइल का अनुमान लगाता है और अन्यथा `stable` का।
वैकल्पिक वर्कफ़्लो इनपुट `-f key=value` के साथ पास करें; व्यापक एडवाइज़री स्वीप के लिए ही `-f release_profile=full` का उपयोग करें।

हेल्पर एक अस्थायी `release-ci/*` रेफ़ बनाता है, जो एक विश्वसनीय
`origin/main` वर्कफ़्लो SHA पर पिन किया होता है, लक्ष्य SHA को केवल उम्मीदवार `ref` के रूप में पास करता है,
और सत्यापन के बाद अस्थायी रेफ़ हटा देता है। डिस्पैच किए गए प्रत्येक चाइल्ड को
उसी वर्कफ़्लो SHA की रिपोर्ट करनी होगी। नया रन बाध्य करने के लिए
`-f reuse_evidence=false` पास करें या वर्तमान `origin/main` से अब भी
पहुँच योग्य किसी पुराने वर्कफ़्लो कमिट को चुनने के लिए
`--workflow-sha <trusted-main-sha>` पास करें। वर्कफ़्लो स्वयं कभी भी
रिपॉज़िटरी रेफ़ नहीं बनाता या अपडेट करता।

Code SHA के सफल होने पर केवल `CHANGELOG.md` जनरेट और कमिट करें। यह नया
कमिट **Release SHA** है। Release SHA के लिए वही हेल्पर चलाएँ। उत्पाद
साक्ष्य का पुनः उपयोग केवल तभी होता है, जब GitHub सिद्ध करता है कि Release SHA,
Code SHA का वंशज है और बदले गए पथों का पूरा सेट ठीक `CHANGELOG.md` है; npm
प्रीफ़्लाइट और पैकेज/इंस्टॉल स्वीकृति फिर भी Release SHA पर चलते हैं।

`release_profile=stable` और `release_profile=full` हमेशा विस्तृत
लाइव/Docker सोक चलाते हैं। `beta` प्रोफ़ाइल के साथ वही सोक लेन शामिल करने के लिए `run_release_soak=true` पास करें।
स्थिर प्रकाशन इस सोक और ब्लॉकिंग उत्पाद-प्रदर्शन साक्ष्य के बिना सत्यापन मैनिफ़ेस्ट को अस्वीकार करता है।

Package Acceptance सामान्यतः समाधान किए गए
`ref` से उम्मीदवार टारबॉल बनाता है, जिसमें `pnpm ci:full-release` के साथ डिस्पैच किए गए पूर्ण-SHA रन भी शामिल हैं। बीटा
प्रकाशन के बाद, रिलीज़ जाँचों, Package Acceptance, क्रॉस-OS,
रिलीज़-पथ Docker और पैकेज Telegram में प्रकाशित npm पैकेज का पुनः उपयोग करने के लिए `release_package_spec=openclaw@YYYY.M.PATCH-beta.N` पास करें। `package_acceptance_package_spec` का उपयोग
केवल तभी करें, जब Package Acceptance को जानबूझकर किसी अलग पैकेज को सिद्ध करना हो।
Codex Plugin की लाइव पैकेज लेन भी इसी स्थिति का अनुसरण करती है: प्रकाशित
`release_package_spec` मान `codex_plugin_spec=npm:@openclaw/codex@<version>` व्युत्पन्न करते हैं;
SHA/आर्टिफ़ैक्ट रन चुने गए रेफ़ से `extensions/codex` पैक करते हैं; और ऑपरेटर
`npm:`, `npm-pack:` या `git:` Plugin
स्रोतों के लिए सीधे `codex_plugin_spec` सेट कर सकते हैं। लेन उस Plugin के लिए आवश्यक स्पष्ट Codex CLI इंस्टॉल स्वीकृति देती है,
फिर Codex CLI प्रीफ़्लाइट और उसी सत्र में OpenAI एजेंट टर्न चलाती है।
इसका अंतिम शून्य-पुनःप्रयास, मध्यम-विचार वाला टर्न छोड़े गए
Codex `final` के साथ दृश्यमान प्रगति भेजता है, यादृच्छिक वर्कस्पेस इनपुट पढ़ता है, उनका सटीक आर्टिफ़ैक्ट लिखता है,
और स्पष्ट पूर्णता भेजता है। यह v2026.7.1 की उस रिग्रेशन को पकड़ता है, जिसमें
सामान्य प्रगति भेजने से टर्न समाप्त हो जाता था।

## शीर्ष-स्तरीय चरण

`rerun_group=all` के लिए पहले एक `Check for reusable validation evidence` जॉब चलता है।
यह समान रिलीज़ प्रोफ़ाइल, प्रभावी सोक सेटिंग और सत्यापन इनपुट वाले नवीनतम पूर्व सफल पूर्ण सत्यापन की खोज करता है। ठीक उसी लक्ष्य के पुनःरन
`exact-target-full-validation-v1` का उपयोग करते हैं। ऐसा वंशज, जिसका पूरा डेल्टा ठीक
`CHANGELOG.md` है, `changelog-only-release-v1` का उपयोग करता है; प्रत्येक उत्पाद लेन छोड़ दी जाती है
और सत्यापक स्वतंत्र रूप से GitHub कमिट तुलना, अपरिवर्तनीय
पैरेंट आर्टिफ़ैक्ट, चाइल्ड रन और डिस्पैच लॉग की पुनः जाँच करता है। किसी भी अन्य लक्ष्य परिवर्तन के लिए
नए Code SHA सत्यापन की आवश्यकता होती है। नया पूर्ण रन बाध्य करने के लिए `reuse_evidence=false` पास करें।
साक्ष्य का पुनः उपयोग केवल `main` या कैनोनिकल SHA-पिन किए गए
`release-ci/*` रेफ़ से चलता है, जिसका वर्कफ़्लो कमिट विश्वसनीय `main` वंशावली पर बना रहता है;
अन्य वर्कफ़्लो रेफ़ चुनी गई लेन नए सिरे से चलाते हैं।

नया पैकेज-सामना करने वाला सत्यापन Plugin Prerelease और OpenClaw Release Checks डिस्पैच करने से पहले एक अपरिवर्तनीय टारबॉल और एक Docker
इमेज आर्टिफ़ैक्ट तैयार करता है।
दोनों चाइल्ड उपयोग से पहले समान पैकेज SHA, आर्टिफ़ैक्ट ID, सेवा डाइजेस्ट,
निर्माता रन प्रयास और Docker आर्काइव डाइजेस्ट सत्यापित करते हैं। पैकेज-स्वतंत्र
बेसिक Docker परत सामग्री-संबोधित GHCR कैश का उपयोग करती है; उम्मीदवार-विशिष्ट इमेज
अपरिवर्तनीय GitHub आर्टिफ़ैक्ट बनी रहती हैं। स्पष्ट प्रकाशित
पैकेज विनिर्देश वाले केंद्रित रन इसके बजाय मौजूदा पैकेज पथ बनाए रखते हैं।

`rerun_group=all` के लिए भी, एक `Verify Docker runtime image assets` जॉब
`OPENCLAW_EXTENSIONS=diagnostics-otel,codex` के साथ
`runtime-assets` Docker लक्ष्य बनाता है।
यह अन्य चरणों के समानांतर चलता है और समग्र सत्यापक द्वारा लागू किया जाता है; लेन अब
डिस्पैच करने से पहले इसकी प्रतीक्षा नहीं करतीं। अधिक सीमित `rerun_group` इस प्रीफ़्लाइट को छोड़ देता है।

समग्र आवरण हमेशा केवल-आर्टिफ़ैक्ट मोड में उत्पाद प्रदर्शन डिस्पैच करता है।

मैन्युअल डिस्पैच के लिए रिपोर्ट प्रकाशन की अनुमति देता है। केवल-आर्टिफ़ैक्ट
गार्ड को सफलतापूर्वक पूरा होना होगा, जिससे सिद्ध हो कि प्रकाशक जॉब छोड़ा गया रहा।
नए और पुनः उपयोग किए गए साक्ष्य
`controls.performanceReportPublication=artifact-only` रिकॉर्ड करते हैं; सत्यापक और पुनः उपयोग
चयनकर्ता मिलान करने वाले सामान्यीकृत प्रदर्शन-चाइल्ड प्रमाण के बिना साक्ष्य अस्वीकार करते हैं।

सत्यापक कैनोनिकल मैनिफ़ेस्ट को
`full-release-validation-<run-id>-<run-attempt>` के रूप में अपलोड करता है। साक्ष्य टूलिंग
ठीक उसी आर्टिफ़ैक्ट ID को डाउनलोड करने से पहले उसकी आर्टिफ़ैक्ट ID, डाइजेस्ट, निर्माता रन और प्रयास को सत्यापित करती है।
यह डाउनलोड किए गए ZIP का आकार सीमित करती है, उसके बाइट्स को REST
`sha256:` डाइजेस्ट के विरुद्ध सत्यापित करती है और आर्काइव निकाले बिना
एकमात्र अनुमत सीमित मैनिफ़ेस्ट प्रविष्टि को स्ट्रीम करती है। पुराने
प्रकाशन उपभोक्ताओं के लिए स्थिर-नाम उपनाम अस्थायी रूप से बना रहता है। सत्यापक हमेशा प्रयास-योग्य आर्टिफ़ैक्ट को प्राथमिकता देता है;
संक्रमण के रूप में, यह स्थिर नाम केवल प्रयास-1 मैनिफ़ेस्ट v2
निर्माता के लिए स्वीकार करता है। यह बाद के प्रयासों और मैनिफ़ेस्ट v3 के लिए उस विरासती नाम को अस्वीकार करता है।

`ref=main` के लिए `rerun_group=all` के साथ, `release/*` रेफ़ के लिए, और Tideclaw
अल्फ़ा रेफ़ के लिए, समान रेफ़ और पुनः-रन समूह वाला नया अंब्रेला रन पुराने रन का स्थान ले लेता है।
जब पैरेंट रद्द किया जाता है, तो उसका मॉनिटर पहले से डिस्पैच किए गए किसी भी चाइल्ड
वर्कफ़्लो को रद्द कर देता है। टैग और पिन किए गए-SHA सत्यापन रन
एक-दूसरे को रद्द नहीं करते।

## रिलीज़ जाँच के चरण

`OpenClaw Release Checks` सबसे बड़ा चाइल्ड वर्कफ़्लो है। यह लक्ष्य को
एक बार रिज़ॉल्व करता है और उपलब्ध होने पर अंब्रेला के साझा पैकेज आर्टिफ़ैक्ट को सत्यापित करता है। कोई
प्रत्यक्ष या केंद्रित डिस्पैच अपना स्वयं का `release-package-under-test`
आर्टिफ़ैक्ट तैयार करता है, जब पैकेज या Docker-संबंधित चरणों को इसकी आवश्यकता होती है।

## Docker रिलीज़-पथ खंड

जब `live_suite_filter` रिक्त होता है, तो Docker रिलीज़-पथ चरण इन खंडों को
चलाता है:

| खंड                                                           | कवरेज                                                                                                                                     |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `core`                                                          | मुख्य Docker रिलीज़-पथ स्मोक लेन।                                                                                                        |
| `package-update-openai`                                         | OpenAI पैकेज इंस्टॉल/अपडेट व्यवहार, Codex का ऑन-डिमांड इंस्टॉल, Codex Plugin की लाइव प्रगति का पूर्ण अनुवर्तन, और Chat Completions टूल कॉल। |
| `package-update-anthropic`                                      | Anthropic पैकेज का इंस्टॉल और अपडेट व्यवहार।                                                                                               |
| `package-update-core`                                           | प्रदाता-निरपेक्ष पैकेज और अपडेट व्यवहार।                                                                                                |
| `plugins-runtime-plugins`                                       | Plugin के व्यवहार का परीक्षण करने वाली Plugin रनटाइम लेन।                                                                                          |
| `plugins-runtime-services`                                      | सेवा-समर्थित और लाइव Plugin रनटाइम लेन।                                                                                                |
| `plugins-runtime-install-a` से `plugins-runtime-install-h` | समानांतर रिलीज़ सत्यापन के लिए विभाजित Plugin इंस्टॉल/रनटाइम बैच।                                                                        |
| `openwebui`                                                     | अनुरोध किए जाने पर समर्पित बड़ी-डिस्क रनर पर पृथक OpenWebUI संगतता स्मोक।                                                      |

जब केवल एक Docker लेन विफल हुई हो, तब पुनः उपयोग योग्य लाइव/E2E वर्कफ़्लो पर लक्षित
`docker_lanes=<lane[,lane]>` का उपयोग करें। उपलब्ध होने पर रिलीज़ आर्टिफ़ैक्ट में पैकेज
आर्टिफ़ैक्ट और इमेज के पुनः उपयोग इनपुट सहित प्रत्येक लेन के पुनः चलाने के कमांड शामिल होते हैं।

## रिलीज़ प्रोफ़ाइल

`release_profile` मुख्यतः रिलीज़ जाँचों के भीतर लाइव/प्रदाता विस्तार को नियंत्रित करता है।
यह सामान्य पूर्ण CI, Plugin Prerelease, इंस्टॉल स्मोक, पैकेज
स्वीकृति या QA Lab को नहीं हटाता। स्थिर और पूर्ण प्रोफ़ाइल हमेशा विस्तृत रेपो/लाइव
E2E और Docker रिलीज़-पथ सोक कवरेज चलाती हैं। बीटा प्रोफ़ाइल
`run_release_soak=true` के साथ इसे चुन सकती है। Package Acceptance प्रत्येक पूर्ण उम्मीदवार के लिए प्रामाणिक पैकेज
Telegram E2E प्रदान करता है, इसलिए समग्र प्रवाह उस
लाइव पोलर को दोहराता नहीं है।

| प्रोफ़ाइल  | अभिप्रेत उपयोग                      | शामिल लाइव/प्रदाता कवरेज                                                                                                                                                                            |
| -------- | --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `beta`   | सबसे तेज़ रिलीज़-महत्वपूर्ण स्मोक।   | OpenAI/मुख्य लाइव पथ, OpenAI के लिए Docker लाइव मॉडल, नेटिव Gateway कोर, नेटिव OpenAI Gateway प्रोफ़ाइल, नेटिव OpenAI Plugin और Docker लाइव Gateway OpenAI।                                            |
| `stable` | डिफ़ॉल्ट रिलीज़ अनुमोदन प्रोफ़ाइल। | `beta` के साथ Anthropic स्मोक, Google, MiniMax, बैकएंड, नेटिव लाइव टेस्ट हार्नेस, Docker लाइव CLI बैकएंड, Docker ACP बाइंड, Docker Codex हार्नेस, Docker सबएजेंट-घोषणा और एक OpenCode Go स्मोक शार्ड। |
| `full`   | व्यापक परामर्शी जाँच।             | `stable` के साथ परामर्शी प्रदाता, Plugin लाइव शार्ड और मीडिया लाइव शार्ड।                                                                                                                               |

## केवल पूर्ण प्रोफ़ाइल में शामिल अतिरिक्त भाग

ये सुइट `stable` द्वारा छोड़े जाते हैं और `full` द्वारा शामिल किए जाते हैं:

| क्षेत्र                             | केवल पूर्ण प्रोफ़ाइल की कवरेज                                                                                                          |
| -------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| Docker लाइव मॉडल               | OpenCode Go, OpenRouter, xAI, Z.ai और Fireworks।                                                                          |
| Docker लाइव Gateway              | DeepSeek/Fireworks, OpenCode Go/OpenRouter और xAI/Z.ai शार्ड में विभाजित परामर्शी प्रदाता।                              |
| नेटिव Gateway प्रदाता प्रोफ़ाइल | पूर्ण Anthropic Opus और Sonnet/Haiku शार्ड, Fireworks, DeepSeek, पूर्ण OpenCode Go मॉडल शार्ड, OpenRouter, xAI और Z.ai। |
| नेटिव Plugin लाइव शार्ड        | Plugins A-K, L-N, O-Z के अन्य, Moonshot और xAI।                                                                             |
| नेटिव मीडिया लाइव शार्ड         | ऑडियो, Google संगीत, MiniMax संगीत और वीडियो समूह A-D।                                                                   |

`stable` में `native-live-src-gateway-profiles-anthropic-smoke` और
`native-live-src-gateway-profiles-opencode-go-smoke` शामिल हैं; इसके बजाय `full` अधिक व्यापक
Anthropic और OpenCode Go मॉडल शार्ड का उपयोग करता है। लक्षित पुनः संचालन में अब भी
समग्र `native-live-src-gateway-profiles-anthropic` या
`native-live-src-gateway-profiles-opencode-go` हैंडल का उपयोग किया जा सकता है।

## लक्षित पुनः संचालन

असंबंधित रिलीज़ बॉक्स दोहराने से बचने के लिए `rerun_group` का उपयोग करें:

| हैंडल              | दायरा                                                                                           |
| ------------------- | ----------------------------------------------------------------------------------------------- |
| `all`               | सभी पूर्ण रिलीज़ सत्यापन चरण।                                                             |
| `ci`                | केवल मैन्युअल पूर्ण CI चाइल्ड।                                                                      |
| `plugin-prerelease` | केवल Plugin Prerelease चाइल्ड।                                                                   |
| `release-checks`    | OpenClaw रिलीज़ जाँच के सभी चरण।                                                             |
| `install-smoke`     | इंस्टॉल स्मोक से रिलीज़ जाँच तक।                                                           |
| `cross-os`          | क्रॉस-OS रिलीज़ जाँच।                                                                        |
| `live-e2e`          | रेपो/लाइव E2E और Docker रिलीज़-पथ सत्यापन।                                               |
| `package`           | पैकेज स्वीकृति।                                                                             |
| `qa`                | QA समानता और QA लाइव लेन।                                                                   |
| `qa-parity`         | केवल QA समानता लेन और रिपोर्ट।                                                                |
| `qa-live`           | सक्षम होने पर QA लाइव Matrix/Telegram और गेटेड Discord, WhatsApp तथा Slack लेन।             |
| `npm-telegram`      | प्रकाशित-पैकेज Telegram E2E; इसके लिए `release_package_spec` या `npm_telegram_package_spec` आवश्यक है। |
| `performance`       | केवल उत्पाद प्रदर्शन प्रमाण।                                                              |

जब एक लाइव सुइट विफल हो, तब `rerun_group=live-e2e` के साथ `live_suite_filter` का उपयोग करें।
मान्य फ़िल्टर आईडी पुनः उपयोग योग्य लाइव/E2E वर्कफ़्लो में परिभाषित हैं, जिनमें
`docker-live-models`, `live-gateway-docker`,
`live-gateway-anthropic-docker`, `live-gateway-google-docker`,
`live-gateway-minimax-docker`, `live-gateway-advisory-docker`,
`live-cli-backend-docker`, `live-acp-bind-docker` और
`live-codex-harness-docker` शामिल हैं।

लक्षित QA ट्रांसपोर्ट पुनः संचालन के लिए, `rerun_group=qa-live` सेट करें और
प्रामाणिक चयनकर्ता `qa-live-matrix`, `qa-live-telegram`, `qa-live-discord`,
`qa-live-whatsapp` या `qa-live-slack` का उपयोग करें।

`live-gateway-advisory-docker` हैंडल अपने तीन प्रदाता शार्ड के लिए एक समग्र पुनः संचालन हैंडल है,
इसलिए यह अब भी सभी परामर्शी Docker Gateway जॉब में विस्तारित होता है।

जब एक क्रॉस-OS लेन विफल हो, तब `rerun_group=cross-os` के साथ `cross_os_suite_filter` का उपयोग करें।
फ़िल्टर किसी OS आईडी, किसी सुइट आईडी या किसी OS/सुइट युग्म को स्वीकार करता है, उदाहरण के लिए
`windows/packaged-upgrade`, `windows` या `packaged-fresh`। क्रॉस-OS
सारांश में पैकेज किए गए अपग्रेड लेन के प्रत्येक चरण का समय शामिल होता है और लंबे समय तक
चलने वाले कमांड Heartbeat पंक्तियाँ प्रिंट करते हैं, ताकि जॉब के
टाइमआउट से पहले अटका हुआ अपडेट दिखाई दे।

QA रिलीज़-जाँच विफलताएँ केवल चुनी गई Matrix, Telegram और QA रनटाइम
टूल कवरेज लेन के लिए सामान्य रिलीज़ सत्यापन को अवरुद्ध करती हैं। QA समानता, रनटाइम
समानता और गेटेड Discord, WhatsApp तथा Slack लाइव लेन परामर्शी हैं और
रिलीज़ सत्यापक को अवरुद्ध किए बिना स्थिति आर्टिफ़ैक्ट प्रकाशित करती हैं। Tideclaw
अल्फ़ा संचालन अब भी गैर-पैकेज-सुरक्षा रिलीज़-जाँच लेन को परामर्शी मान सकते हैं। `release_profile=beta` के साथ,
`Run repo/live E2E validation` लाइव-प्रदाता सुइट परामर्शी होते हैं: तृतीय-पक्ष मॉडल परिनियोजन
रिलीज़ के दौरान बदलते रहते हैं, इसलिए बीटा उनकी विफलताओं को चेतावनियों के रूप में दिखाता है,
जबकि स्थिर और पूर्ण प्रोफ़ाइल उन्हें अवरोधक बनाए रखती हैं। जब
`live_suite_filter` स्पष्ट रूप से Discord, WhatsApp या Slack जैसी गेटेड QA लाइव लेन का अनुरोध करता है,
तब संबंधित `OPENCLAW_RELEASE_QA_*_LIVE_CI_ENABLED` रेपो
वेरिएबल सक्षम होना आवश्यक है; अन्यथा लेन को चुपचाप छोड़ने के बजाय इनपुट कैप्चर विफल हो जाता है।
जब नए QA प्रमाण की आवश्यकता हो, तब `rerun_group=qa`, `qa-parity` या `qa-live`
को पुनः चलाएँ।

## सुरक्षित रखने योग्य प्रमाण

रिलीज़-स्तरीय सूचकांक के रूप में `Full Release Validation` सारांश सुरक्षित रखें। यह
चाइल्ड रन आईडी से लिंक करता है और इसमें सबसे धीमे जॉब की तालिकाएँ शामिल होती हैं। विफलताओं के लिए पहले चाइल्ड
वर्कफ़्लो जाँचें, फिर ऊपर दिए गए सबसे छोटे मेल खाते हैंडल को पुनः चलाएँ।

Code SHA और Release SHA, पुनः उपयोग नीति और परिवर्तित-पथ सेट,
हरा Code SHA पैरेंट रन तथा हल्का Release SHA पैरेंट रन—सभी रिकॉर्ड करें।

उपयोगी आर्टिफ़ैक्ट:

- `release-package-under-test`, `OpenClaw Release Checks` से
- Docker रिलीज़-पथ आर्टिफ़ैक्ट, `.artifacts/docker-tests/` के अंतर्गत
- Package Acceptance `package-under-test` और Docker स्वीकृति आर्टिफ़ैक्ट
- प्रत्येक OS और सुइट के लिए क्रॉस-OS रिलीज़-जाँच आर्टिफ़ैक्ट
- QA समानता, रनटाइम समानता और चयनित Matrix, Telegram, Discord, WhatsApp
  या Slack आर्टिफ़ैक्ट
