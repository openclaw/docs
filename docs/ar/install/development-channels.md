OpenClaw يوفر ثلاث قنوات تحديث:

- **stable**: وسم توزيعة npm `latest`. موصى بها لمعظم المستخدمين.
- **beta**: وسم توزيعة npm `beta` عندما يكون حاليًا؛ إذا كان beta مفقودًا أو أقدم من
  أحدث إصدار stable، يتراجع تدفق التحديث إلى `latest`.
- **dev**: رأس متحرك من `main` (git). وسم توزيعة npm: `dev` (عند نشره).
  فرع `main` مخصص للتجربة والتطوير النشط. قد يحتوي على
  ميزات غير مكتملة أو تغييرات كاسرة. لا تستخدمه لبوابات الإنتاج.

عادةً ننشر إصدارات stable إلى **beta** أولًا، ونختبرها هناك، ثم نشغّل
خطوة ترقية صريحة تنقل الإصدار المفحوص إلى `latest` دون
تغيير رقم الإصدار. يمكن للمشرفين أيضًا نشر إصدار stable
مباشرة إلى `latest` عند الحاجة. وسوم التوزيعة هي مصدر الحقيقة لتثبيتات npm.

## خطوط الدعم الشهرية المخطط لها

لا يوفر OpenClaw حتى الآن قناة LTS أو قناة دعم شهرية. نحن نعمل
باتجاه خطوط دعم شهرية متوافقة مع SemVer حتى يتمكن المستخدمون من البقاء على خط
أكثر هدوءًا بينما يواصل `latest` التحرك بسرعة.

شكل الإصدار المخطط له هو `YYYY.M.PATCH`:

- `YYYY` هو العام.
- `M` هو خط الإصدار الشهري، دون صفر بادئ.
- `PATCH` يزيد داخل ذلك الخط الشهري ويمكن أن يتجاوز 100 عند الحاجة.

أمثلة على وسوم مستقبلية:

- `v2026.6.0`، و`v2026.6.1`، و`v2026.6.2` لخط يونيو.
- `v2026.6.3-beta.1` لإصدار تمهيدي على مسار fast/latest.
- قد يشير وسم توزيعة لخط دعم مستقبلي مثل `stable-2026-6` أو `lts-2026-6` إلى
  خط شهري، لكن لا تتوفر قناة كهذه اليوم.

إلى أن تكتمل تلك الهجرة، تظل قنوات التحديث العامة هي `stable` و`beta`
و`dev`.

## تبديل القنوات

```bash
openclaw update --channel stable
openclaw update --channel beta
openclaw update --channel dev
```

يحفظ `--channel` اختيارك في الإعدادات (`update.channel`) ويوائم
طريقة التثبيت:

- **`stable`** (تثبيتات الحزمة): تُحدّث عبر وسم توزيعة npm `latest`.
- **`beta`** (تثبيتات الحزمة): تفضّل وسم توزيعة npm `beta`، لكنها تتراجع إلى
  `latest` عندما يكون `beta` مفقودًا أو أقدم من وسم stable الحالي.
- **`stable`** (تثبيتات git): تتحقق من أحدث وسم git لإصدار stable.
- **`beta`** (تثبيتات git): تفضّل أحدث وسم git لإصدار beta، لكنها تتراجع إلى
  أحدث وسم git لإصدار stable عندما يكون beta مفقودًا أو أقدم.
- **`dev`**: يضمن وجود نسخة git محلية (افتراضيًا `~/openclaw`، ويمكن تجاوزه باستخدام
  `OPENCLAW_GIT_DIR`)، وينتقل إلى `main`، ويعيد التأسيس على المصدر الأعلى، ويبني، ثم
  يثبت CLI العام من تلك النسخة المحلية.

<Tip>
إذا أردت تشغيل stable وdev بالتوازي، فاحتفظ بنسختين محليتين ووجّه Gateway لديك إلى نسخة stable.
</Tip>

## استهداف إصدار أو وسم لمرة واحدة

استخدم `--tag` لاستهداف وسم توزيعة أو إصدار أو مواصفة حزمة محددة لتحديث واحد
**دون** تغيير القناة المحفوظة لديك:

```bash
# Install a specific version
openclaw update --tag 2026.4.1-beta.1

# Install from the beta dist-tag (one-off, does not persist)
openclaw update --tag beta

# Install from GitHub main branch (npm tarball)
openclaw update --tag main

# Install a specific npm package spec
openclaw update --tag openclaw@2026.4.1-beta.1
```

ملاحظات:

- ينطبق `--tag` على **تثبيتات الحزمة (npm) فقط**. تتجاهله تثبيتات git.
- لا يتم حفظ الوسم. يستخدم أمر `openclaw update` التالي القناة المكوّنة لديك
  كالمعتاد.
- حماية الرجوع إلى إصدار أقدم: إذا كان الإصدار المستهدف أقدم من إصدارك الحالي،
  يطلب OpenClaw التأكيد (يمكن التخطي باستخدام `--yes`).
- يختلف `--channel beta` عن `--tag beta`: يمكن لتدفق القناة التراجع
  إلى stable/latest عندما يكون beta مفقودًا أو أقدم، بينما يستهدف `--tag beta`
  وسم التوزيعة الخام `beta` لتلك التشغيله الواحدة.

## تشغيل تجريبي

عاين ما سيفعله `openclaw update` دون إجراء تغييرات:

```bash
openclaw update --dry-run
openclaw update --channel beta --dry-run
openclaw update --tag 2026.4.1-beta.1 --dry-run
openclaw update --dry-run --json
```

يعرض التشغيل التجريبي القناة الفعلية، والإصدار المستهدف، والإجراءات المخطط لها، وما إذا
كان تأكيد الرجوع إلى إصدار أقدم سيكون مطلوبًا.

## Plugins والقنوات

عند تبديل القنوات باستخدام `openclaw update`، يزامن OpenClaw أيضًا
مصادر Plugin:

- يفضّل `dev` Plugins المضمّنة من نسخة git المحلية.
- يعيد `stable` و`beta` حزم Plugin المثبتة عبر npm.
- يتم تحديث Plugins المثبتة عبر npm بعد اكتمال تحديث النواة.

## التحقق من الحالة الحالية

```bash
openclaw update status
```

يعرض القناة النشطة، ونوع التثبيت (git أو حزمة)، والإصدار الحالي، و
المصدر (إعدادات، وسم git، فرع git، أو الافتراضي).

## أفضل ممارسات الوسم

- ضع وسومًا للإصدارات التي تريد لنسخ git المحلية أن تستقر عليها (`vYYYY.M.D` لإصدارات
  stable الحالية، و`vYYYY.M.D-beta.N` لإصدارات beta الحالية).
- يتم التعرف أيضًا على `vYYYY.M.D.beta.N` للتوافق، لكن يُفضّل `-beta.N`.
- لا تزال وسوم `vYYYY.M.D-<patch>` القديمة معروفة كإصدارات stable (غير beta)،
  لكن نموذج الدعم الشهري المخطط له سيستخدم أرقام تصحيح عادية
  (`vYYYY.M.PATCH`) بدلًا من لاحقة تصحيح بشرطة.
- أبقِ الوسوم غير قابلة للتغيير: لا تنقل وسمًا أو تعيد استخدامه أبدًا.
- تظل وسوم توزيعة npm مصدر الحقيقة لتثبيتات npm:
  - `latest` -> stable
  - `beta` -> إصدار مرشح أو إصدار stable منشور إلى beta أولًا
  - `dev` -> لقطة main (اختياري)

## توفر تطبيق macOS

قد **لا** تتضمن إصدارات beta وdev إصدار تطبيق macOS. هذا مقبول:

- ما زال من الممكن نشر وسم git ووسم توزيعة npm.
- اذكر "لا يوجد بناء macOS لهذا beta" في ملاحظات الإصدار أو سجل التغييرات.
