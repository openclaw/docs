يدعم OpenClaw ثلاثة مسارات للترحيل: الاستيراد من نظام وكيل آخر، ونقل تثبيت قائم إلى جهاز جديد، وترقية Plugin في موضعه.

## الاستيراد من نظام وكيل آخر

تجلب موفّرات الترحيل المضمّنة التعليمات، وخوادم MCP، وSkills، وإعدادات النماذج، ومفاتيح API (عند الاشتراك) إلى OpenClaw. تُعاين الخطط قبل إجراء أي تغيير، وتُحجب الأسرار في التقارير، ويستند التطبيق إلى نسخة احتياطية تم التحقق منها.

نقطة دخول CLI هي [`openclaw migrate`](/ar/cli/migrate). ويمكن لعملية الإعداد الأولي أيضًا عرض الترحيل عند اكتشاف مصدر معروف (`openclaw onboard --flow import`).

## نقل OpenClaw إلى جهاز جديد

انسخ **دليل الحالة** (`~/.openclaw/` افتراضيًا) و**مساحة العمل** للحفاظ على:

- **الإعدادات** — ملف `openclaw.json` وجميع إعدادات Gateway.
- **المصادقة** — ملف `auth-profiles.json` الخاص بكل وكيل (مفاتيح API بالإضافة إلى OAuth)، وأي حالة للقنوات أو الموفّرين ضمن `credentials/`.
- **الجلسات** — سجل المحادثات وحالة الوكيل.
- **حالة القنوات** — تسجيل دخول WhatsApp، وجلسة Telegram، وما شابه.
- **ملفات مساحة العمل** — `MEMORY.md`، و`USER.md`، وSkills، والموجّهات.

<Tip>
شغّل `openclaw status` على الجهاز القديم للتأكد من مسار دليل الحالة. تستخدم الملفات الشخصية المخصصة المسار `~/.openclaw-<profile>/` أو مسارًا يُعيَّن عبر `OPENCLAW_STATE_DIR`.
</Tip>

### خطوات الترحيل

<Steps>
  <Step title="إيقاف Gateway وإنشاء نسخة احتياطية">
    على الجهاز **القديم**، أوقف Gateway حتى لا تتغير الملفات أثناء النسخ، ثم أنشئ أرشيفًا:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    إذا كنت تستخدم ملفات شخصية متعددة (مثل `~/.openclaw-work`)، فأرشِف كلًا منها على حدة.

  </Step>

  <Step title="تثبيت OpenClaw على الجهاز الجديد">
    [ثبّت](/ar/install) CLI (وNode إذا لزم الأمر) على الجهاز الجديد. لا بأس إذا أنشأت عملية الإعداد الأولي دليلًا جديدًا باسم `~/.openclaw/` — فستستبدله في الخطوة التالية.
  </Step>

  <Step title="نسخ دليل الحالة ومساحة العمل">
    انقل الأرشيف عبر `scp` أو `rsync -a` أو محرك أقراص خارجي، ثم استخرجه:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    تأكد من تضمين الأدلة المخفية ومن تطابق ملكية الملفات مع المستخدم الذي سيشغّل Gateway.

  </Step>

  <Step title="تشغيل أداة الفحص والتحقق">
    على الجهاز الجديد، شغّل [أداة الفحص](/ar/gateway/doctor) لتطبيق عمليات ترحيل الإعدادات وإصلاح الخدمات:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

إذا كان Telegram أو Discord يستخدم القيمة الاحتياطية الافتراضية من متغيرات البيئة (`TELEGRAM_BOT_TOKEN` أو `DISCORD_BOT_TOKEN`)، فتحقق من أن ملف `.env` في دليل الحالة المنقول يحتوي على هذه المفاتيح دون طباعة القيم السرية:

```bash
awk -F= '/^(TELEGRAM_BOT_TOKEN|DISCORD_BOT_TOKEN)=/ { print $1 "=present" }' ~/.openclaw/.env
```

يحذّر `openclaw doctor` أيضًا عندما لا يملك حساب Telegram أو Discord الافتراضي المفعّل رمزًا مميزًا مُعدًّا، ولا يكون متغير البيئة المطابق متاحًا لعملية أداة الفحص.

### المشكلات الشائعة

<AccordionGroup>
  <Accordion title="عدم تطابق الملف الشخصي أو دليل الحالة">
    إذا كان Gateway القديم يستخدم `--profile` أو `OPENCLAW_STATE_DIR` بينما لا يستخدمهما الجديد، فستظهر القنوات كما لو تم تسجيل الخروج منها وستكون الجلسات فارغة. شغّل Gateway باستخدام الملف الشخصي أو دليل الحالة **نفسه** الذي نقلته، ثم أعد تشغيل `openclaw doctor`.
  </Accordion>

  <Accordion title="نسخ openclaw.json فقط">
    ملف الإعدادات وحده لا يكفي. توجد ملفات تعريف مصادقة النماذج ضمن `agents/<agentId>/agent/auth-profiles.json`، وتوجد حالة القنوات والموفّرين ضمن `credentials/`. انقل دائمًا دليل الحالة **بأكمله**.
  </Accordion>

  <Accordion title="الأذونات والملكية">
    إذا أجريت النسخ بصفتك المستخدم الجذر أو بدّلت المستخدمين، فقد يتعذر على Gateway قراءة بيانات الاعتماد. تأكد من أن دليل الحالة ومساحة العمل مملوكان للمستخدم الذي يشغّل Gateway.
  </Accordion>

  <Accordion title="الوضع البعيد">
    إذا كانت واجهة المستخدم لديك تشير إلى Gateway **بعيد**، فإن المضيف البعيد هو مالك الجلسات ومساحة العمل. انقل مضيف Gateway نفسه، وليس حاسوبك المحمول المحلي. راجع [الأسئلة الشائعة](/ar/help/faq#where-things-live-on-disk).
  </Accordion>

  <Accordion title="الأسرار في النسخ الاحتياطية">
    يحتوي دليل الحالة على ملفات تعريف المصادقة، وبيانات اعتماد القنوات، وحالات أخرى للموفّرين. خزّن النسخ الاحتياطية مشفّرة، وتجنب قنوات النقل غير الآمنة، ودوّر المفاتيح إذا اشتبهت في انكشافها.
  </Accordion>
</AccordionGroup>

### قائمة التحقق

على الجهاز الجديد، تأكد مما يلي:

- [ ] يُظهر `openclaw status` أن Gateway قيد التشغيل.
- [ ] لا تزال القنوات متصلة (من دون الحاجة إلى إعادة الاقتران).
- [ ] تُفتح لوحة المعلومات وتعرض الجلسات القائمة.
- [ ] ملفات مساحة العمل (الذاكرة والإعدادات) موجودة.

## ترقية Plugin في موضعه

تحافظ ترقيات Plugin في موضعه على معرّف Plugin ومفاتيح الإعدادات نفسيهما، لكنها قد تنقل الحالة المخزنة على القرص إلى التخطيط الحالي. توجد أدلة الترقية الخاصة بكل Plugin بجانب قنواته:

- [ترحيل Matrix](/ar/channels/matrix-migration): حدود استعادة الحالة المشفّرة، وسلوك اللقطات التلقائية، وأوامر الاستعادة اليدوية.
