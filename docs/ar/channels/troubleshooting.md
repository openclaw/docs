استخدم هذه الصفحة عندما تتصل قناة لكن يكون سلوكها غير صحيح.

## تسلسل الأوامر

شغّل هذه الأوامر بالترتيب أولًا:

```bash
openclaw status
openclaw gateway status
openclaw logs --follow
openclaw doctor
openclaw channels status --probe
```

خط الأساس السليم:

- `Runtime: running`
- `Connectivity probe: ok`
- `Capability: read-only` أو `write-capable` أو `admin-capable`
- يُظهر فحص القناة أن وسيلة النقل متصلة، ويُظهر `works` أو `audit ok` حيثما كان ذلك مدعومًا

## بعد التحديث

استخدم هذا عندما تختفي Telegram أو iMessage أو إعدادات حقبة BlueBubbles أو قناة Plugin أخرى بعد التحديث.

```bash
openclaw status --all
openclaw doctor --fix
openclaw gateway restart
openclaw status --all
```

ابحث عن `plugin load failed: dependency tree corrupted; run openclaw doctor --fix` في `openclaw status --all`. يعني ذلك أن القناة مُعدّة، لكن إعداد Plugin أو تحميله واجه شجرة تبعيات تالفة بدلًا من تسجيل القناة. يمسح `openclaw doctor --fix` الروابط الرمزية القديمة لتبعيات وقت تشغيل Plugin وظلال المصادقة القديمة، ثم يعيد `openclaw gateway restart` تحميل حالة سليمة.

## WhatsApp

### مؤشرات فشل WhatsApp

| العَرَض | أسرع فحص | الإصلاح |
| ----------------------------------- | --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| متصل لكن لا توجد ردود على الرسائل الخاصة | `openclaw pairing list whatsapp` | وافق على المُرسِل أو غيّر سياسة الرسائل الخاصة/قائمة السماح. |
| تُتجاهل رسائل المجموعة | تحقّق من `requireMention` وأنماط الإشارة في الإعدادات | أشر إلى البوت أو خفّف سياسة الإشارة لتلك المجموعة. |
| تنتهي مهلة تسجيل الدخول عبر رمز QR بالخطأ 408 | تحقّق من متغيري البيئة `HTTPS_PROXY` و`HTTP_PROXY` الخاصين بـ Gateway | عيّن وكيلًا يمكن الوصول إليه؛ استخدم `NO_PROXY` لحالات التجاوز فقط. |
| حلقات عشوائية لقطع الاتصال/إعادة تسجيل الدخول | `openclaw channels status --probe` والسجلات | تُعلَّم عمليات إعادة الاتصال الحديثة حتى عند وجود اتصال حالي؛ راقب السجلات، وأعد تشغيل Gateway، ثم أعد الربط إذا استمر عدم استقرار الاتصال. |
| حلقة `status=408 Request Time-out` | الفحص والسجلات وdoctor ثم حالة Gateway | أصلح اتصال المضيف/التوقيت أولًا؛ احتفظ بنسخة احتياطية من بيانات المصادقة وأعد ربط الحساب إذا استمرت الحلقة. |
| تصل الردود متأخرة ثوانٍ/دقائق | `openclaw doctor --fix` | يوقف doctor عملاء TUI المحليين المتقادمين الذين تأكد أنهم يضعفون حلقة أحداث Gateway. |

## Telegram

### مؤشرات فشل Telegram

| العَرَض | أسرع فحص | الإصلاح |
| ------------------------------------ | ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `/start` لكن لا يوجد مسار رد قابل للاستخدام | `openclaw pairing list telegram` | وافق على الاقتران أو غيّر سياسة الرسائل الخاصة. |
| البوت متصل لكن المجموعة تظل صامتة | تحقّق من شرط الإشارة ووضع خصوصية البوت | عطّل وضع الخصوصية لإتاحة رؤية رسائل المجموعة أو أشر إلى البوت. |
| فشل الإرسال مع أخطاء في الشبكة | افحص السجلات بحثًا عن حالات فشل استدعاء واجهة Telegram البرمجية | أصلح توجيه DNS أو IPv6 أو الوكيل إلى `api.telegram.org`. |
| يُبلغ بدء التشغيل بأن `getMe returned 401` | تحقّق من مصدر الرمز المميز المُعدّ | أعد نسخ رمز BotFather المميز أو أنشئه مجددًا وحدّث `botToken` أو `tokenFile` أو `TELEGRAM_BOT_TOKEN` للحساب الافتراضي. |
| يتوقف الاستقصاء أو يعيد الاتصال ببطء | استخدم `openclaw logs --follow` لتشخيصات الاستقصاء | رقِّ البرنامج؛ إذا كانت عمليات إعادة التشغيل نتائج إيجابية زائفة، فاضبط `pollingStallThresholdMs`. لا تزال حالات التوقف المستمرة تشير إلى الوكيل أو DNS أو IPv6. |
| يُرفض `setMyCommands` عند بدء التشغيل | افحص السجلات بحثًا عن `BOT_COMMANDS_TOO_MUCH` | قلّل أوامر Telegram الخاصة بـ Plugin أو Skills أو الأوامر المخصصة، أو عطّل القوائم الأصلية. |
| أجريت ترقية وأصبحت قائمة السماح تحظرك | `openclaw security audit` وقوائم السماح في الإعدادات | شغّل `openclaw doctor --fix` أو استبدل `@username` بمعرّفات المُرسِلين الرقمية. |
