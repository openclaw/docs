“الإقران” هو خطوة الموافقة الصريحة على الوصول في OpenClaw.
يُستخدم في موضعين:

1. **إقران DM** (من يُسمح له بالتحدث إلى البوت)
2. **إقران Node** (أي الأجهزة/العُقد يُسمح لها بالانضمام إلى شبكة Gateway)

سياق الأمان: [الأمان](/ar/gateway/security)

## 1) إقران DM (الوصول إلى المحادثات الواردة)

عند تهيئة قناة بسياسة DM هي `pairing`، يحصل المرسلون غير المعروفين على رمز قصير ولا تتم **معالجة** رسالتهم حتى توافق عليها.

سياسات DM الافتراضية موثقة في: [الأمان](/ar/gateway/security)

تكون `dmPolicy: "open"` عامة فقط عندما تتضمن قائمة سماح DM الفعلية `"*"`.
يتطلب الإعداد والتحقق وجود حرف البدل هذا للتهيئات العامة المفتوحة. إذا كانت الحالة الموجودة
تحتوي على `open` مع إدخالات `allowFrom` محددة، فإن وقت التشغيل لا يزال يقبل
هؤلاء المرسلين فقط، ولا توسع موافقات مخزن الإقران وصول `open`.

رموز الإقران:

- 8 أحرف، كبيرة، ولا تحتوي على أحرف ملتبسة (`0O1I`).
- **تنتهي صلاحيتها بعد ساعة واحدة**. لا يرسل البوت رسالة الإقران إلا عند إنشاء طلب جديد (تقريبًا مرة واحدة في الساعة لكل مرسل).
- تُحدد طلبات إقران DM المعلقة افتراضيًا عند **3 لكل قناة**؛ ويتم تجاهل الطلبات الإضافية حتى تنتهي صلاحية أحدها أو تتم الموافقة عليه.

### الموافقة على مرسل

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

إذا لم يتم تكوين مالك أوامر بعد، فإن الموافقة على رمز إقران DM تهيئ أيضًا
`commands.ownerAllowFrom` إلى المرسل الموافق عليه، مثل `telegram:123456789`.
يمنح ذلك عمليات الإعداد لأول مرة مالكًا صريحًا للأوامر ذات الامتياز ومطالبات الموافقة على التنفيذ.
بعد وجود مالك، لا تمنح موافقات الإقران اللاحقة إلا وصول DM؛ ولا تضيف مزيدًا من المالكين.

القنوات المدعومة: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### مكان وجود الحالة

تُخزن تحت `~/.openclaw/credentials/`:

- الطلبات المعلقة: `<channel>-pairing.json`
- مخزن قائمة السماح الموافق عليها:
  - الحساب الافتراضي: `<channel>-allowFrom.json`
  - حساب غير افتراضي: `<channel>-<accountId>-allowFrom.json`

سلوك نطاق الحساب:

- الحسابات غير الافتراضية تقرأ/تكتب ملف قائمة السماح الخاص بنطاقها فقط.
- يستخدم الحساب الافتراضي ملف قائمة السماح غير المحدد بنطاق حساب والخاص بالقناة.

تعامل معها كبيانات حساسة (فهي تتحكم في الوصول إلى مساعدك).

<Note>
مخزن قائمة سماح الإقران مخصص لوصول DM. تخويل المجموعات منفصل.
الموافقة على رمز إقران DM لا تسمح تلقائيًا لذلك المرسل بتشغيل أوامر المجموعة
أو التحكم في البوت داخل المجموعات. تهيئة المالك الأول هي حالة إعداد منفصلة
في `commands.ownerAllowFrom`، ولا يزال تسليم محادثات المجموعات يتبع قوائم السماح للمجموعات
الخاصة بالقناة (مثل `groupAllowFrom` أو `groups` أو التجاوزات لكل مجموعة
أو لكل موضوع حسب القناة).
</Note>

## 2) إقران أجهزة Node (iOS/Android/macOS/العُقد بلا واجهة)

تتصل Nodes بـ Gateway بوصفها **أجهزة** مع `role: node`. ينشئ Gateway
طلب إقران جهاز يجب الموافقة عليه.

### الإقران عبر Telegram (موصى به لـ iOS)

إذا كنت تستخدم Plugin `device-pair`، يمكنك إجراء إقران الجهاز لأول مرة بالكامل من Telegram:

1. في Telegram، أرسل رسالة إلى بوتك: `/pair`
2. يرد البوت برسالتين: رسالة تعليمات ورسالة **رمز إعداد** منفصلة (سهلة النسخ/اللصق في Telegram).
3. على هاتفك، افتح تطبيق OpenClaw على iOS ← Settings ← Gateway.
4. الصق رمز الإعداد واتصل.
5. عُد إلى Telegram: `/pair pending` (راجع معرفات الطلب والدور والنطاقات)، ثم وافق.

رمز الإعداد هو حمولة JSON مشفرة بـ base64 تحتوي على:

- `url`: عنوان URL الخاص بـ Gateway WebSocket (`ws://...` أو `wss://...`)
- `bootstrapToken`: رمز تمهيد قصير العمر لجهاز واحد يُستخدم لمصافحة الإقران الأولية

يحمل رمز التمهيد هذا ملف تعريف تمهيد الإقران المدمج:

- يبقى رمز `node` الأساسي المُسلَّم `scopes: []`
- يبقى أي رمز `operator` مُسلَّم محدودًا بقائمة سماح التمهيد:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- تكون فحوصات نطاق التمهيد مسبوقة بالدور، وليست مجموعة نطاقات مسطحة واحدة:
  تلبي إدخالات نطاق operator طلبات operator فقط، ويجب على الأدوار غير operator
  أن تطلب النطاقات ضمن بادئة دورها الخاصة
- يظل تدوير/إبطال الرموز لاحقًا محدودًا بكل من عقد الدور الموافق عليه للجهاز
  ونطاقات operator الخاصة بجلسة المستدعي

تعامل مع رمز الإعداد مثل كلمة مرور أثناء صلاحيته.

### الموافقة على جهاز Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

إذا أعاد الجهاز نفسه المحاولة بتفاصيل مصادقة مختلفة (مثل دور/نطاقات/مفتاح عام مختلف)،
يتم استبدال الطلب المعلق السابق وإنشاء `requestId` جديد.

<Note>
لا يحصل الجهاز المقترن مسبقًا على وصول أوسع بصمت. إذا أعاد الاتصال طالبًا نطاقات أكثر أو دورًا أوسع، يُبقي OpenClaw الموافقة الحالية كما هي وينشئ طلب ترقية معلقًا جديدًا. استخدم `openclaw devices list` لمقارنة الوصول الموافق عليه حاليًا بالوصول المطلوب حديثًا قبل أن توافق.
</Note>

### موافقة تلقائية اختيارية لأجهزة Node ضمن CIDR موثوق

يبقى إقران الأجهزة يدويًا افتراضيًا. بالنسبة لشبكات Node الخاضعة لتحكم صارم،
يمكنك الاشتراك في الموافقة التلقائية لأول مرة على Node باستخدام CIDR صريحة أو عناوين IP دقيقة:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ينطبق هذا فقط على طلبات إقران `role: node` الجديدة بلا نطاقات مطلوبة.
لا يزال عملاء Operator والمتصفح وControl UI وWebChat يتطلبون موافقة يدوية.
كما لا تزال تغييرات الدور والنطاق والبيانات الوصفية والمفتاح العام تتطلب موافقة يدوية.

### تخزين حالة إقران Node

تُخزن تحت `~/.openclaw/devices/`:

- `pending.json` (قصير العمر؛ تنتهي صلاحية الطلبات المعلقة)
- `paired.json` (الأجهزة المقترنة + الرموز)

### ملاحظات

- واجهة API القديمة `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) هي
  مخزن إقران منفصل مملوك لـ Gateway. لا تزال WS nodes تتطلب إقران الأجهزة.
- سجل الإقران هو مصدر الحقيقة الدائم للأدوار الموافق عليها. تبقى رموز الجهاز النشطة
  محدودة بمجموعة الأدوار الموافق عليها تلك؛ ولا ينشئ إدخال رمز شاذ
  خارج الأدوار الموافق عليها وصولًا جديدًا.
