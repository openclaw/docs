"الإقران" هو خطوة موافقة الوصول الصريحة في OpenClaw.
يُستخدم في موضعين:

1. **إقران الرسائل الخاصة** (من يُسمح له بالتحدث إلى البوت)
2. **إقران Node** (أي الأجهزة/العُقد يُسمح لها بالانضمام إلى شبكة Gateway)

سياق الأمان: [الأمان](/ar/gateway/security)

## 1) إقران الرسائل الخاصة (وصول الدردشة الواردة)

عند تكوين قناة بسياسة رسائل خاصة `pairing`، يحصل المرسلون غير المعروفين على رمز قصير ولا تتم **معالجة** رسالتهم حتى توافق عليها.

سياسات الرسائل الخاصة الافتراضية موثقة في: [الأمان](/ar/gateway/security)

`dmPolicy: "open"` تكون عامة فقط عندما تتضمن قائمة السماح الفعلية للرسائل الخاصة `"*"`.
يتطلب الإعداد والتحقق وجود حرف البدل هذا لتكوينات public-open. إذا كانت الحالة
الموجودة تحتوي على `open` مع إدخالات `allowFrom` محددة، فإن وقت التشغيل لا يزال يسمح
لهؤلاء المرسلين فقط، ولا توسع موافقات مخزن الإقران وصول `open`.

رموز الإقران:

- 8 أحرف، بأحرف كبيرة، ومن دون أحرف ملتبسة (`0O1I`).
- **تنتهي صلاحيتها بعد ساعة واحدة**. لا يرسل البوت رسالة الإقران إلا عند إنشاء طلب جديد (تقريبًا مرة واحدة في الساعة لكل مرسل).
- تقتصر طلبات إقران الرسائل الخاصة المعلقة على **3 لكل قناة** افتراضيًا؛ ويتم تجاهل الطلبات الإضافية حتى تنتهي صلاحية أحدها أو تتم الموافقة عليه.

### الموافقة على مرسل

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

إذا لم يتم تكوين مالك أوامر بعد، فإن الموافقة على رمز إقران رسائل خاصة تُمهّد أيضًا
`commands.ownerAllowFrom` للمرسل الموافق عليه، مثل `telegram:123456789`.
وهذا يمنح الإعدادات لأول مرة مالكًا صريحًا للأوامر ذات الامتياز ومطالبات موافقة
exec. بعد وجود مالك، لا تمنح موافقات الإقران اللاحقة إلا وصول الرسائل الخاصة؛
ولا تضيف مزيدًا من المالكين.

القنوات المدعومة: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### مجموعات مرسلين قابلة لإعادة الاستخدام

استخدم `accessGroups` في المستوى الأعلى عندما ينبغي تطبيق مجموعة المرسلين الموثوقين نفسها على
قنوات رسائل متعددة أو على قوائم السماح للرسائل الخاصة والمجموعات معًا.

تستخدم المجموعات الثابتة `type: "message.senders"` وتُشار إليها باستخدام
`accessGroup:<name>` من قوائم سماح القنوات:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

مجموعات الوصول موثقة بالتفصيل هنا: [مجموعات الوصول](/ar/channels/access-groups)

### أين توجد الحالة

مخزنة ضمن `~/.openclaw/credentials/`:

- الطلبات المعلقة: `<channel>-pairing.json`
- مخزن قائمة السماح المعتمدة:
  - الحساب الافتراضي: `<channel>-allowFrom.json`
  - الحساب غير الافتراضي: `<channel>-<accountId>-allowFrom.json`

سلوك تحديد نطاق الحساب:

- تقرأ/تكتب الحسابات غير الافتراضية ملف قائمة السماح محدد النطاق الخاص بها فقط.
- يستخدم الحساب الافتراضي ملف قائمة السماح غير محدد النطاق والخاص بالقناة.

تعامل معها على أنها حساسة (فهي تتحكم في الوصول إلى مساعدك).

<Note>
مخزن قائمة سماح الإقران مخصص لوصول الرسائل الخاصة. تفويض المجموعات منفصل.
لا تؤدي الموافقة على رمز إقران رسائل خاصة تلقائيًا إلى السماح لذلك المرسل بتشغيل
أوامر المجموعات أو التحكم في البوت داخل المجموعات. تمهيد المالك الأول هو حالة تكوين
منفصلة في `commands.ownerAllowFrom`، ولا يزال تسليم دردشة المجموعة يتبع قوائم سماح
المجموعات الخاصة بالقناة (على سبيل المثال `groupAllowFrom` أو `groups` أو التجاوزات
لكل مجموعة أو لكل موضوع حسب القناة).
</Note>

## 2) إقران جهاز Node (عُقد iOS/Android/macOS/بدون واجهة)

تتصل العُقد بـ Gateway بوصفها **أجهزة** مع `role: node`. ينشئ Gateway
طلب إقران جهاز يجب الموافقة عليه.

### الإقران عبر Telegram (موصى به لـ iOS)

إذا كنت تستخدم Plugin `device-pair`، يمكنك تنفيذ إقران الجهاز لأول مرة بالكامل من Telegram:

1. في Telegram، أرسل رسالة إلى البوت: `/pair`
2. يرد البوت برسالتين: رسالة تعليمات ورسالة **رمز إعداد** منفصلة (سهلة النسخ/اللصق في Telegram).
3. على هاتفك، افتح تطبيق OpenClaw لنظام iOS ← Settings ← Gateway.
4. امسح رمز QR أو الصق رمز الإعداد واتصل.
5. ارجع إلى Telegram: `/pair pending` (راجع معرفات الطلبات، والدور، والنطاقات)، ثم وافق.

رمز الإعداد هو حمولة JSON مشفرة بـ base64 تحتوي على:

- `url`: عنوان URL لـ Gateway WebSocket (`ws://...` أو `wss://...`)
- `bootstrapToken`: رمز تمهيد قصير العمر لجهاز واحد يُستخدم لمصافحة الإقران الأولية

يحمل رمز التمهيد هذا ملف تعريف تمهيد الإقران المدمج:

- يظل رمز `node` الأساسي المُسلَّم `scopes: []`
- يظل أي رمز `operator` مُسلَّم مقيدًا بقائمة سماح التمهيد:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- فحوصات نطاق التمهيد مسبوقة بالدور، وليست مجموعة نطاقات مسطحة واحدة:
  إدخالات نطاق operator لا تلبي إلا طلبات operator، ولا يزال يجب على الأدوار غير operator
  طلب النطاقات تحت بادئة دورها الخاصة
- يظل تدوير/إبطال الرموز لاحقًا مقيدًا بكل من عقد الدور الموافق عليه للجهاز
  ونطاقات operator لجلسة المستدعي

تعامل مع رمز الإعداد مثل كلمة مرور أثناء صلاحيته.

بالنسبة إلى Tailscale أو الإقران المحمول العام أو البعيد الآخر، استخدم Tailscale Serve/Funnel
أو عنوان URL آخر لـ Gateway يبدأ بـ `wss://`. تُقبل رموز إعداد `ws://` بنص عادي فقط
لـ local loopback، وعناوين LAN الخاصة، ومضيفي Bonjour `.local`، ومضيف محاكي Android.
لا تزال عناوين Tailnet CGNAT، وأسماء `.ts.net`، والمضيفون العامون تفشل بشكل مغلق
قبل إصدار رمز QR/رمز الإعداد.

### الموافقة على جهاز Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

عندما تُرفض موافقة صريحة لأن جلسة الجهاز المقترن التي توافق
فُتحت بنطاق خاص بالإقران فقط، يعيد CLI محاولة الطلب نفسه باستخدام
`operator.admin`. يتيح ذلك لجهاز مقترن حالي قادر على الإدارة استعادة إقران
Control UI/المتصفح جديد دون تحرير `devices/paired.json` يدويًا. لا يزال
Gateway يتحقق من الاتصال المعاد؛ وتبقى الرموز التي لا يمكنها المصادقة
باستخدام `operator.admin` محظورة.

إذا أعاد الجهاز نفسه المحاولة بتفاصيل مصادقة مختلفة (على سبيل المثال
دور/نطاقات/مفتاح عام مختلف)، يُستبدل الطلب المعلق السابق ويُنشأ
`requestId` جديد.

<Note>
لا يحصل الجهاز المقترن بالفعل على وصول أوسع بصمت. إذا أعاد الاتصال طالبًا نطاقات أكثر أو دورًا أوسع، يُبقي OpenClaw الموافقة الحالية كما هي وينشئ طلب ترقية معلقًا جديدًا. استخدم `openclaw devices list` لمقارنة الوصول الموافق عليه حاليًا بالوصول المطلوب حديثًا قبل الموافقة.
</Note>

### موافقة تلقائية اختيارية لعُقد CIDR الموثوقة

يبقى إقران الأجهزة يدويًا افتراضيًا. بالنسبة إلى شبكات العُقد المضبوطة بإحكام،
يمكنك الاشتراك في الموافقة التلقائية على Node لأول مرة باستخدام CIDR صريحة أو عناوين IP دقيقة:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ينطبق هذا فقط على طلبات إقران `role: node` الجديدة من دون نطاقات مطلوبة.
لا يزال عملاء Operator والمتصفح وControl UI وWebChat يتطلبون موافقة يدوية.
لا تزال تغييرات الدور والنطاق والبيانات الوصفية والمفتاح العام تتطلب موافقة
يدوية.

### تخزين حالة إقران Node

مخزنة ضمن `~/.openclaw/devices/`:

- `pending.json` (قصير العمر؛ تنتهي صلاحية الطلبات المعلقة)
- `paired.json` (الأجهزة المقترنة + الرموز)

### ملاحظات

- واجهة API القديمة `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) هي
  مخزن إقران منفصل مملوك لـ Gateway. لا تزال عُقد WS تتطلب إقران جهاز.
- سجل الإقران هو المصدر الدائم للحقيقة للأدوار الموافق عليها. تبقى
  رموز الأجهزة النشطة مقيدة بمجموعة الأدوار الموافق عليها تلك؛ ولا ينشئ إدخال رمز عارض
  خارج الأدوار الموافق عليها وصولًا جديدًا.
