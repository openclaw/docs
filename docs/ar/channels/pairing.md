"الاقتران" هو خطوة الموافقة الصريحة على الوصول في OpenClaw.
يُستخدم في موضعين:

1. **اقتران الرسائل الخاصة** (من يُسمح له بالتحدث إلى البوت)
2. **اقتران Node** (الأجهزة/العُقد التي يُسمح لها بالانضمام إلى شبكة Gateway)

سياق الأمان: [الأمان](/ar/gateway/security)

## 1) اقتران الرسائل الخاصة (وصول الدردشة الواردة)

عند ضبط قناة بسياسة رسائل خاصة `pairing`، يحصل المرسلون غير المعروفين على رمز قصير ولا تتم **معالجة** رسالتهم حتى توافق عليها.

سياسات الرسائل الخاصة الافتراضية موثقة في: [الأمان](/ar/gateway/security)

`dmPolicy: "open"` تكون عامة فقط عندما تتضمن قائمة السماح الفعالة للرسائل الخاصة `"*"`.
يتطلب الإعداد والتحقق وجود حرف البدل هذا في إعدادات public-open. إذا كانت الحالة
الحالية تحتوي على `open` مع إدخالات `allowFrom` محددة، فسيظل وقت التشغيل يسمح
لهؤلاء المرسلين فقط، ولا توسع موافقات مخزن الاقتران وصول `open`.

رموز الاقتران:

- 8 أحرف، بأحرف كبيرة، ومن دون أحرف ملتبسة (`0O1I`).
- **تنتهي صلاحيتها بعد ساعة واحدة**. يرسل البوت رسالة الاقتران فقط عند إنشاء طلب جديد (تقريبا مرة واحدة في الساعة لكل مرسل).
- طلبات اقتران الرسائل الخاصة المعلقة محدودة افتراضيا بـ **3 لكل قناة**؛ يتم تجاهل الطلبات الإضافية حتى تنتهي صلاحية أحدها أو تتم الموافقة عليه.

### الموافقة على مرسل

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

إذا لم يتم ضبط مالك أوامر بعد، فإن الموافقة على رمز اقتران للرسائل الخاصة تقوم أيضا بتهيئة
`commands.ownerAllowFrom` إلى المرسل الموافق عليه، مثل `telegram:123456789`.
يمنح ذلك الإعدادات لأول مرة مالكا صريحا للأوامر ذات الامتيازات ومطالبات موافقة
التنفيذ. بعد وجود مالك، تمنح موافقات الاقتران اللاحقة وصول الرسائل الخاصة فقط؛
ولا تضيف مزيدا من المالكين.

القنوات المدعومة: `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### مجموعات المرسلين القابلة لإعادة الاستخدام

استخدم `accessGroups` في المستوى الأعلى عندما ينبغي تطبيق مجموعة المرسلين الموثوقين نفسها على
قنوات رسائل متعددة أو على قوائم السماح للرسائل الخاصة والمجموعات معا.

تستخدم المجموعات الثابتة `type: "message.senders"` وتتم الإشارة إليها باستخدام
`accessGroup:<name>` من قوائم سماح القنوات:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

مجموعات الوصول موثقة بالتفصيل هنا: [مجموعات الوصول](/ar/channels/access-groups)

### مكان وجود الحالة

مخزنة تحت `~/.openclaw/credentials/`:

- الطلبات المعلقة: `<channel>-pairing.json`
- مخزن قائمة السماح الموافق عليها:
  - الحساب الافتراضي: `<channel>-allowFrom.json`
  - الحساب غير الافتراضي: `<channel>-<accountId>-allowFrom.json`

سلوك نطاق الحساب:

- تقرأ/تكتب الحسابات غير الافتراضية ملف قائمة السماح المحدد النطاق الخاص بها فقط.
- يستخدم الحساب الافتراضي ملف قائمة السماح غير محدد النطاق على مستوى القناة.

تعامل معها على أنها حساسة (فهي تتحكم في الوصول إلى مساعدك).

<Note>
مخزن قائمة سماح الاقتران مخصص لوصول الرسائل الخاصة. تخويل المجموعات منفصل.
الموافقة على رمز اقتران للرسائل الخاصة لا تسمح تلقائيا لذلك المرسل بتشغيل أوامر المجموعات
أو التحكم في البوت داخل المجموعات. تهيئة المالك الأول هي حالة ضبط منفصلة
في `commands.ownerAllowFrom`، ولا يزال تسليم دردشة المجموعات يتبع
قوائم سماح المجموعات الخاصة بالقناة (على سبيل المثال `groupAllowFrom` أو `groups` أو التجاوزات
لكل مجموعة أو لكل موضوع حسب القناة).
</Note>

## 2) اقتران جهاز Node (عُقد iOS/Android/macOS/headless)

تتصل العُقد بـ Gateway بصفتها **أجهزة** مع `role: node`. ينشئ Gateway
طلب اقتران جهاز يجب الموافقة عليه.

### الاقتران عبر Telegram (موصى به لـ iOS)

إذا كنت تستخدم Plugin `device-pair`، يمكنك إجراء اقتران الجهاز لأول مرة بالكامل من Telegram:

1. في Telegram، أرسل رسالة إلى البوت: `/pair`
2. يرد البوت برسالتين: رسالة تعليمات ورسالة **رمز إعداد** منفصلة (يسهل نسخها/لصقها في Telegram).
3. على هاتفك، افتح تطبيق OpenClaw على iOS ← الإعدادات ← Gateway.
4. امسح رمز QR أو الصق رمز الإعداد واتصل.
5. عد إلى Telegram: `/pair pending` (راجع معرفات الطلبات والدور والنطاقات)، ثم وافق.

رمز الإعداد هو حمولة JSON مشفرة بـ base64 تحتوي على:

- `url`: عنوان WebSocket لـ Gateway (`ws://...` أو `wss://...`)
- `bootstrapToken`: رمز تمهيد قصير الأجل لجهاز واحد يُستخدم لمصافحة الاقتران الأولية

يحمل رمز التمهيد هذا ملف تعريف تمهيد الاقتران المدمج:

- يسمح ملف تعريف الإعداد المدمج بخط الأساس الجديد لرمز QR/رمز الإعداد فقط:
  `node` إضافة إلى تسليم `operator` محدود
- يبقى رمز `node` الذي تم تسليمه `scopes: []`
- يقتصر رمز `operator` الذي تم تسليمه على `operator.approvals`,
  `operator.read`, `operator.talk.secrets`, و `operator.write`
- لا يتم منح `operator.admin` عبر تمهيد QR/رمز الإعداد؛ فهو يتطلب
  تدفق اقتران أو رمز `operator` منفصل تمت الموافقة عليه
- يظل تدوير/إلغاء الرموز لاحقا محدودا بكل من عقد دور الجهاز الموافق عليه
  ونطاقات `operator` لجلسة المستدعي

تعامل مع رمز الإعداد مثل كلمة مرور طالما أنه صالح.

بالنسبة إلى Tailscale أو الاقتران العام أو أي اقتران محمول عن بُعد آخر، استخدم Tailscale Serve/Funnel
أو عنوان Gateway آخر بصيغة `wss://`. لا تُقبل رموز إعداد النص الصريح `ws://` إلا
لـ loopback وعناوين LAN الخاصة ومضيفي Bonjour بصيغة `.local` ومضيف محاكي
Android. لا تزال عناوين Tailnet CGNAT وأسماء `.ts.net` والمضيفون العامون
تفشل بإغلاق آمن قبل إصدار QR/رمز الإعداد.

### الموافقة على جهاز Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

عند رفض موافقة صريحة لأن جلسة الجهاز المقترن التي توافق
فُتحت بنطاق الاقتران فقط، تعيد CLI محاولة الطلب نفسه باستخدام
`operator.admin`. يتيح ذلك لجهاز مقترن موجود قادر على الإدارة استرداد اقتران جديد
لـ Control UI/المتصفح من دون تحرير `devices/paired.json` يدويا. لا يزال
Gateway يتحقق من الاتصال المُعاد؛ وتبقى الرموز التي لا يمكنها المصادقة
باستخدام `operator.admin` محظورة.

إذا أعاد الجهاز نفسه المحاولة بتفاصيل مصادقة مختلفة (على سبيل المثال
دور/نطاقات/مفتاح عام مختلف)، يتم استبدال الطلب المعلق السابق وإنشاء
`requestId` جديد.

<Note>
لا يحصل الجهاز المقترن مسبقا على وصول أوسع بصمت. إذا أعاد الاتصال طالبا نطاقات أكثر أو دورا أوسع، يحتفظ OpenClaw بالموافقة الحالية كما هي وينشئ طلب ترقية معلقا جديدا. استخدم `openclaw devices list` لمقارنة الوصول الموافق عليه حاليا مع الوصول المطلوب حديثا قبل الموافقة.
</Note>

### موافقة تلقائية اختيارية على Node عبر CIDR موثوق

يبقى اقتران الأجهزة يدويا افتراضيا. بالنسبة إلى شبكات Node محكمة التحكم،
يمكنك الاشتراك في الموافقة التلقائية لأول مرة على Node باستخدام CIDR صريحة أو عناوين IP دقيقة:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

ينطبق هذا فقط على طلبات اقتران `role: node` الجديدة التي لا تطلب
أي نطاقات. لا يزال عملاء Operator والمتصفح وControl UI وWebChat يتطلبون موافقة
يدوية. ولا تزال تغييرات الدور والنطاق والبيانات الوصفية والمفتاح العام تتطلب موافقة
يدوية.

### تخزين حالة اقتران Node

مخزنة تحت `~/.openclaw/devices/`:

- `pending.json` (قصير الأجل؛ تنتهي صلاحية الطلبات المعلقة)
- `paired.json` (الأجهزة المقترنة + الرموز)

### ملاحظات

- واجهة API القديمة `node.pair.*` ‏(CLI: `openclaw nodes pending|approve|reject|remove|rename`) هي
  مخزن اقتران منفصل مملوك لـ Gateway. لا تزال عُقد WS تتطلب اقتران جهاز.
- سجل الاقتران هو مصدر الحقيقة الدائم للأدوار الموافق عليها. تبقى
  رموز الأجهزة النشطة محدودة بمجموعة الأدوار الموافق عليها تلك؛ ولا ينشئ إدخال رمز عارض
  خارج الأدوار الموافق عليها وصولا جديدا.
