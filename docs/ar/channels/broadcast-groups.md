## نظرة عامة

تُمكّن مجموعات البث عدة وكلاء من معالجة الرسالة نفسها والرد عليها في الوقت نفسه. يتيح لك ذلك إنشاء فرق وكلاء متخصصة تعمل معًا داخل مجموعة WhatsApp واحدة أو رسالة خاصة — وكل ذلك باستخدام رقم هاتف واحد.

النطاق الحالي: **WhatsApp فقط** (قناة الويب).

يتم تقييم مجموعات البث بعد قوائم السماح الخاصة بالقنوات وقواعد تفعيل المجموعات. في مجموعات WhatsApp، هذا يعني أن عمليات البث تحدث عندما يكون من المعتاد أن يرد OpenClaw (على سبيل المثال: عند الإشارة، بحسب إعدادات مجموعتك).

## حالات الاستخدام

### 1. فرق وكلاء متخصصة

انشر عدة وكلاء بمسؤوليات محددة ومركزة:

```
المجموعة: "فريق التطوير"
الوكلاء:
  - CodeReviewer (يراجع مقتطفات الشيفرة)
  - DocumentationBot (ينشئ الوثائق)
  - SecurityAuditor (يفحص الثغرات)
  - TestGenerator (يقترح حالات اختبار)
```

يعالج كل وكيل الرسالة نفسها ويقدّم منظوره المتخصص.

### 2. دعم متعدد اللغات

```
المجموعة: "الدعم الدولي"
الوكلاء:
  - Agent_EN (يرد باللغة الإنجليزية)
  - Agent_DE (يرد باللغة الألمانية)
  - Agent_ES (يرد باللغة الإسبانية)
```

### 3. سير عمل ضمان الجودة

```
المجموعة: "دعم العملاء"
الوكلاء:
  - SupportAgent (يقدّم الإجابة)
  - QAAgent (يراجع الجودة، ويرد فقط عند العثور على مشكلات)
```

### 4. أتمتة المهام

```
المجموعة: "إدارة المشاريع"
الوكلاء:
  - TaskTracker (يحدّث قاعدة بيانات المهام)
  - TimeLogger (يسجل الوقت المستغرق)
  - ReportGenerator (ينشئ الملخصات)
```

## التكوين

### الإعداد الأساسي

أضف قسم `broadcast` على المستوى الأعلى (بجانب `bindings`). المفاتيح هي معرّفات نظراء WhatsApp:

- محادثات المجموعات: معرّف المجموعة JID (مثل `120363403215116621@g.us`)
- الرسائل الخاصة: رقم هاتف بصيغة E.164 (مثل `+15551234567`)

```json
{
  "broadcast": {
    "120363403215116621@g.us": ["alfred", "baerbel", "assistant3"]
  }
}
```

**النتيجة:** عندما يكون من المفترض أن يرد OpenClaw في هذه الدردشة، سيشغّل الوكلاء الثلاثة جميعًا.

### استراتيجية المعالجة

تحكّم في كيفية معالجة الوكلاء للرسائل:

#### بالتوازي (الافتراضي)

يعالج جميع الوكلاء الرسائل في الوقت نفسه:

```json
{
  "broadcast": {
    "strategy": "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

#### بالتسلسل

يعالج الوكلاء الرسائل بالترتيب (ينتظر كل وكيل انتهاء السابق):

```json
{
  "broadcast": {
    "strategy": "sequential",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

## كيف يعمل

### تدفق الرسائل

1. **تصل رسالة واردة** إلى مجموعة WhatsApp
2. **فحص البث**: يتحقق النظام مما إذا كان معرّف النظير موجودًا في `broadcast`
3. **إذا كان ضمن قائمة البث**:
   - يعالج جميع الوكلاء المدرجين الرسالة
   - لكل وكيل مفتاح جلسة خاص به وسياق معزول
   - يعالج الوكلاء الرسائل بالتوازي (افتراضيًا) أو بالتسلسل
4. **إذا لم يكن ضمن قائمة البث**:
   - يُطبَّق التوجيه العادي (أول ربط مطابق)

ملاحظة: لا تتجاوز مجموعات البث قوائم السماح الخاصة بالقنوات أو قواعد تفعيل المجموعات (الإشارات/الأوامر/إلخ). إنها تغيّر فقط _أي الوكلاء يعملون_ عندما تكون الرسالة مؤهلة للمعالجة.

### عزل الجلسات

يحافظ كل وكيل في مجموعة البث على ما يلي بشكل منفصل تمامًا:

- **مفاتيح الجلسات** (`agent:alfred:whatsapp:group:120363...` مقابل `agent:baerbel:whatsapp:group:120363...`)
- **سجل المحادثة** (لا يرى الوكيل رسائل الوكلاء الآخرين)
- **مساحة العمل** (بيئات معزولة منفصلة إذا كانت مُكوّنة)
- **إمكانية الوصول إلى الأدوات** (قوائم سماح/منع مختلفة)
- **الذاكرة/السياق** (ملفات `IDENTITY.md` و`SOUL.md` وغيرها منفصلة)
- **مخزن سياق المجموعة** (الرسائل الأخيرة في المجموعة المستخدمة للسياق) يُشارك لكل نظير، لذلك يرى جميع وكلاء البث السياق نفسه عند التفعيل

وهذا يتيح لكل وكيل أن يمتلك:

- شخصيات مختلفة
- وصولًا مختلفًا إلى الأدوات (مثل: للقراءة فقط مقابل القراءة والكتابة)
- نماذج مختلفة (مثل: opus مقابل sonnet)
- Skills مختلفة مثبّتة

### مثال: جلسات معزولة

في المجموعة `120363403215116621@g.us` مع الوكلاء `["alfred", "baerbel"]`:

**سياق Alfred:**

```
الجلسة: agent:alfred:whatsapp:group:120363403215116621@g.us
السجل: [رسالة المستخدم، ردود alfred السابقة]
مساحة العمل: /Users/user/openclaw-alfred/
الأدوات: read, write, exec
```

**سياق Bärbel:**

```
الجلسة: agent:baerbel:whatsapp:group:120363403215116621@g.us
السجل: [رسالة المستخدم، ردود baerbel السابقة]
مساحة العمل: /Users/user/openclaw-baerbel/
الأدوات: read only
```

## أفضل الممارسات

### 1. اجعل الوكلاء ذوي تركيز واضح

صمّم كل وكيل بمسؤولية واحدة واضحة ومحددة:

```json
{
  "broadcast": {
    "DEV_GROUP": ["formatter", "linter", "tester"]
  }
}
```

✅ **جيد:** لكل وكيل مهمة واحدة  
❌ **سيئ:** وكيل عام واحد باسم "dev-helper"

### 2. استخدم أسماء وصفية

اجعل وظيفة كل وكيل واضحة:

```json
{
  "agents": {
    "security-scanner": { "name": "Security Scanner" },
    "code-formatter": { "name": "Code Formatter" },
    "test-generator": { "name": "Test Generator" }
  }
}
```

### 3. اضبط وصولًا مختلفًا إلى الأدوات

امنح الوكلاء فقط الأدوات التي يحتاجون إليها:

```json
{
  "agents": {
    "reviewer": {
      "tools": { "allow": ["read", "exec"] } // للقراءة فقط
    },
    "fixer": {
      "tools": { "allow": ["read", "write", "edit", "exec"] } // قراءة وكتابة
    }
  }
}
```

### 4. راقب الأداء

مع وجود عدد كبير من الوكلاء، ضع في الاعتبار ما يلي:

- استخدام `"strategy": "parallel"` (الافتراضي) للسرعة
- حصر مجموعات البث في 5 إلى 10 وكلاء
- استخدام نماذج أسرع للوكلاء الأبسط

### 5. تعامل مع الإخفاقات بسلاسة

يفشل الوكلاء بشكل مستقل. خطأ أحد الوكلاء لا يمنع الآخرين:

```
الرسالة → [الوكيل A ✓، الوكيل B ✗ خطأ، الوكيل C ✓]
النتيجة: يرد الوكيلان A وC، ويسجل الوكيل B الخطأ
```

## التوافق

### المزوّدون

تعمل مجموعات البث حاليًا مع:

- ✅ WhatsApp (مُنفّذ)
- 🚧 Telegram (مخطط له)
- 🚧 Discord (مخطط له)
- 🚧 Slack (مخطط له)

### التوجيه

تعمل مجموعات البث جنبًا إلى جنب مع التوجيه الحالي:

```json
{
  "bindings": [
    {
      "match": { "channel": "whatsapp", "peer": { "kind": "group", "id": "GROUP_A" } },
      "agentId": "alfred"
    }
  ],
  "broadcast": {
    "GROUP_B": ["agent1", "agent2"]
  }
}
```

- `GROUP_A`: يرد alfred فقط (توجيه عادي)
- `GROUP_B`: يرد كل من agent1 وagent2 (بث)

**الأولوية:** يأخذ `broadcast` أولوية أعلى من `bindings`.

## استكشاف الأخطاء وإصلاحها

### الوكلاء لا يردون

**تحقق من:**

1. أن معرّفات الوكلاء موجودة في `agents.list`
2. أن تنسيق معرّف النظير صحيح (مثل `120363403215116621@g.us`)
3. أن الوكلاء ليسوا ضمن قوائم المنع

**التصحيح:**

```bash
tail -f ~/.openclaw/logs/gateway.log | grep broadcast
```

### وكيل واحد فقط يرد

**السبب:** قد يكون معرّف النظير موجودًا في `bindings` لكنه غير موجود في `broadcast`.

**الحل:** أضفه إلى إعدادات البث أو أزله من `bindings`.

### مشكلات الأداء

**إذا كان الأداء بطيئًا مع عدد كبير من الوكلاء:**

- قلّل عدد الوكلاء في كل مجموعة
- استخدم نماذج أخف (sonnet بدلًا من opus)
- تحقّق من وقت بدء تشغيل البيئة المعزولة

## مرجع API

### مخطط التكوين

```typescript
interface OpenClawConfig {
  broadcast?: {
    strategy?: "parallel" | "sequential";
    [peerId: string]: string[];
  };
}
```

### الحقول

- `strategy` (اختياري): كيفية معالجة الوكلاء
  - `"parallel"` (الافتراضي): يعالج جميع الوكلاء في الوقت نفسه
  - `"sequential"`: يعالج الوكلاء حسب ترتيبهم في المصفوفة
- `[peerId]`: معرّف مجموعة WhatsApp JID، أو رقم E.164، أو أي معرّف نظير آخر
  - القيمة: مصفوفة من معرّفات الوكلاء الذين يجب أن يعالجوا الرسائل

## القيود

1. **الحد الأقصى للوكلاء:** لا يوجد حد صارم، لكن أكثر من 10 وكلاء قد يكون بطيئًا
2. **السياق المشترك:** لا يرى الوكلاء ردود بعضهم بعضًا (عن قصد)
3. **ترتيب الرسائل:** قد تصل الردود المتوازية بأي ترتيب
4. **حدود المعدل:** يُحتسب جميع الوكلاء ضمن حدود المعدل الخاصة بـ WhatsApp

## تحسينات مستقبلية

الميزات المخطط لها:

- [ ] وضع السياق المشترك (يرى الوكلاء ردود بعضهم بعضًا)
- [ ] تنسيق الوكلاء (يمكن للوكلاء إرسال إشارات إلى بعضهم بعضًا)
- [ ] اختيار ديناميكي للوكلاء (اختيار الوكلاء بناءً على محتوى الرسالة)
- [ ] أولويات الوكلاء (يرد بعض الوكلاء قبل غيرهم)
