`qa-channel` هو ناقل رسائل اصطناعي محلي للمستودع لاختبارات ضمان الجودة الآلية في OpenClaw (`extensions/qa-channel`، حزمة خاصة، مستثناة من عمليات التثبيت المجمّعة). وهو ليس قناة إنتاجية؛ بل وُجد لاختبار حد Plugin القناة نفسه الذي تستخدمه النواقل الحقيقية، مع إبقاء الحالة حتمية وقابلة للفحص بالكامل.

## ما الذي يفعله

- صيغة أهداف من فئة Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- تُعرض محادثات `channel:` و`group:` المشتركة للوكلاء كتفاعلات غرف جماعية/قنوات، بحيث تختبر سياسة توجيه الردود المرئية وأداة الرسائل نفسها المستخدمة في Discord وSlack وTelegram والنواقل المشابهة.
- ناقل اصطناعي مدعوم ببروتوكول HTTP لحقن الرسائل الواردة، والتقاط سجل الرسائل الصادرة، وإنشاء سلاسل المحادثات، والتفاعلات، والتعديلات، والحذف، وإجراءات البحث/القراءة.
- مشغّل تحقق ذاتي على جانب المضيف يكتب تقرير Markdown في `.artifacts/qa-e2e/`.

## الإعداد

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

مفاتيح الحساب:

- `enabled` - مفتاح التفعيل الرئيسي لهذا الحساب.
- `name` - تسمية عرض اختيارية.
- `baseUrl` - عنوان URL للناقل الاصطناعي. يُعد الحساب مهيأً بمجرد تعيينه.
- `botUserId` - معرّف مستخدم البوت الاصطناعي المستخدم في صيغة الأهداف (الافتراضي: `openclaw`).
- `botDisplayName` - اسم العرض للرسائل الصادرة (الافتراضي: `OpenClaw QA`).
- `pollTimeoutMs` - نافذة انتظار الاستقصاء الطويل. عدد صحيح بين 100 و30000 (الافتراضي: 1000).
- `allowFrom` - قائمة السماح بالمرسلين (معرّفات المستخدمين أو `"*"`؛ الافتراضي: `["*"]`). تستخدم الرسائل المباشرة دائمًا سياسة `open`؛ كما تستخدم سياسة المجموعات ذات قائمة السماح معرّفات المرسلين الاصطناعية هذه.
- `groupPolicy` - سياسة الغرف المشتركة: `"open"` (الافتراضي)، أو `"allowlist"`، أو `"disabled"`.
- `groupAllowFrom` - قائمة سماح اختيارية بمرسلي الغرف المشتركة. عند إغفالها مع `"allowlist"`، تعود قناة ضمان الجودة إلى `allowFrom`.
- `groups.<room>.requireMention` - اشتراط الإشارة إلى البوت قبل الرد في غرفة مجموعة/قناة محددة (الافتراضي: false). يحدد `groups."*"` القيمة الافتراضية؛ وتحدد `tools` / `toolsBySender` لكل غرفة تجاوزات سياسة الأدوات.
- `defaultTo` - الهدف الاحتياطي عند عدم توفير هدف.
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` - تقييد الأدوات لكل إجراء.

مفاتيح الحسابات المتعددة في المستوى الأعلى:

- `accounts` - سجل تجاوزات كل حساب مسمى، مفهرس بحسب معرّف الحساب.
- `defaultAccount` - معرّف الحساب المفضل عند تهيئة عدة حسابات.

## المشغّلات

التحقق الذاتي على جانب المضيف (يكتب تقرير Markdown ضمن `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

يوجّه هذا عبر `qa-lab`، ويبدأ ناقل ضمان الجودة الموجود داخل المستودع، ويشغّل شريحة وقت التشغيل الخاصة بـ`qa-channel`، وينفّذ تحققًا ذاتيًا حتميًا.

مجموعة السيناريوهات الكاملة المدعومة بالمستودع:

```bash
pnpm openclaw qa suite
```

تشغّل السيناريوهات بالتوازي على مسار Gateway الخاص بضمان الجودة. راجع [نظرة عامة على ضمان الجودة](/ar/concepts/qa-e2e-automation) للاطلاع على السيناريوهات وملفات التعريف وأوضاع المزوّدين.

موقع ضمان الجودة المدعوم بـDocker (Gateway + واجهة مستخدم مصحح QA Lab في حزمة تشغيل واحدة):

```bash
pnpm qa:lab:up
```

ينشئ موقع ضمان الجودة، ويبدأ حزمة Gateway + QA Lab المدعومة بـDocker، ويطبع عنوان URL الخاص بـQA Lab. ومن هناك يمكنك اختيار السيناريوهات، وتحديد مسار النموذج، وتشغيل عمليات فردية، ومشاهدة النتائج مباشرة. مصحح QA Lab منفصل عن حزمة واجهة مستخدم التحكم المشحونة.
