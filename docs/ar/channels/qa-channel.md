`qa-channel` هو ناقل رسائل اصطناعي مضمّن لأتمتة ضمان الجودة في OpenClaw.

ليس قناة إنتاج. وهو موجود لاختبار حد Plugin القناة نفسه
المستخدم بواسطة النواقل الحقيقية، مع إبقاء الحالة حتمية وقابلة للفحص بالكامل.

## ما الذي يفعله حاليًا

- قواعد هدف من فئة Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `thread:<room>/<thread>`
- ناقل اصطناعي مدعوم بـ HTTP من أجل:
  - حقن الرسائل الواردة
  - التقاط النصوص الصادرة
  - إنشاء سلاسل المحادثات
  - التفاعلات
  - التعديلات
  - عمليات الحذف
  - إجراءات البحث والقراءة
- مشغّل فحص ذاتي مضمّن على جهة المضيف يكتب تقرير Markdown

{

## المشغّل

الشريحة الرأسية الحالية:

```bash
pnpm qa:e2e
```

يمر هذا الآن عبر امتداد `qa-lab` المضمّن. فهو يبدأ
ناقل ضمان الجودة داخل المستودع، ويشغّل شريحة وقت التشغيل المضمّنة `qa-channel`، ويجري
فحصًا ذاتيًا حتميًا، ويكتب تقرير Markdown تحت `.artifacts/qa-e2e/`.

واجهة debugger خاصة:

```bash
pnpm qa:lab:up
```

يبني هذا الأمر الواحد موقع ضمان الجودة، ويشغّل مكدس Gateway + QA Lab
المدعوم بـ Docker، ويطبع عنوان URL الخاص بـ QA Lab. ومن خلال ذلك الموقع يمكنك اختيار
السيناريوهات، وتحديد مسار النموذج، وتشغيل عمليات فردية،
ومشاهدة النتائج مباشرة.

مجموعة ضمان الجودة الكاملة المدعومة بالمستودع:

```bash
pnpm openclaw qa suite
```

يؤدي ذلك إلى تشغيل debugger ضمان الجودة الخاص على عنوان URL محلي، منفصلًا عن
حزمة واجهة Control UI المضمنة.

## النطاق

النطاق الحالي ضيق عمدًا:

- الناقل + ناقل Plugin
- قواعد التوجيه الخاصة بسلاسل المحادثات
- إجراءات الرسائل المملوكة للقناة
- تقارير Markdown
- موقع ضمان جودة مدعوم بـ Docker مع عناصر تحكم للتشغيل

ستضيف الأعمال اللاحقة:

- تنفيذ مصفوفة provider/model
- اكتشافًا أغنى للسيناريوهات
- تنسيقًا أصليًا لـ OpenClaw لاحقًا
