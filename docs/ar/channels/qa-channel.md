`qa-channel` هو ناقل رسائل اصطناعي محلي للمستودع لضمان الجودة الآلي في OpenClaw ‏(`extensions/qa-channel`، حزمة خاصة، مستبعدة من عمليات التثبيت المعبأة). وهو ليس قناة إنتاجية، بل وُجد لاختبار حد Plugin القناة نفسه الذي تستخدمه النواقل الحقيقية، مع إبقاء الحالة حتمية وقابلة للفحص بالكامل.

## ما الذي يفعله

- صياغة أهداف من فئة Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- تُعرض محادثات `channel:` و`group:` المشتركة للوكلاء على هيئة أدوار في غرف المجموعات/القنوات، ولذلك تختبر سياسة توجيه الردود المرئية وأداة الرسائل نفسها المستخدمة في Discord وSlack وTelegram والنواقل المشابهة.
- ناقل اصطناعي مدعوم عبر HTTP لحقن الرسائل الواردة، والتقاط نصوص المحادثات الصادرة، وإنشاء سلاسل المحادثات، والتفاعلات، والتعديلات، والحذف، وإجراءات البحث/القراءة.
- مشغّل تحقق ذاتي على جانب المضيف يكتب تقرير Markdown إلى `.artifacts/qa-e2e/`.

## الإعداد

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

مفاتيح الحساب:

- `enabled` - مفتاح التفعيل الرئيسي لهذا الحساب.
- `name` - تسمية عرض اختيارية.
- `baseUrl` - عنوان URL للناقل الاصطناعي. يُعدّ الحساب مضبوطًا بمجرد تعيينه.
- `botUserId` - معرّف مستخدم البوت الاصطناعي المستخدم في صياغة الهدف (الافتراضي: `openclaw`).
- `botDisplayName` - اسم العرض للرسائل الصادرة (الافتراضي: `OpenClaw QA`).
- `pollTimeoutMs` - نافذة انتظار الاستطلاع الطويل. عدد صحيح بين 100 و30000 (الافتراضي: 1000).
- `allowFrom` - قائمة السماح للمرسلين (معرّفات المستخدمين أو `"*"`؛ الافتراضي: `["*"]`). تستخدم الرسائل المباشرة دائمًا سياسة `open`؛ كما تستخدم سياسة المجموعات ذات قائمة السماح معرّفات المرسلين الاصطناعية هذه.
- `groupPolicy` - سياسة الغرف المشتركة: `"open"` (الافتراضي)، أو `"allowlist"`، أو `"disabled"`.
- `groupAllowFrom` - قائمة سماح اختيارية لمرسلي الغرف المشتركة. عند إغفالها ضمن `"allowlist"`، تعود قناة ضمان الجودة إلى `allowFrom`.
- `groups.<room>.requireMention` - اشتراط الإشارة إلى البوت قبل الرد في غرفة مجموعة/قناة محددة (الافتراضي: false). يعيّن `groups."*"` الإعداد الافتراضي؛ وتعيّن `tools` / `toolsBySender` لكل غرفة تجاوزات سياسة الأدوات.
- `defaultTo` - الهدف الاحتياطي عند عدم تقديم أي هدف.
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` - تقييد الأدوات لكل إجراء.

مفاتيح الحسابات المتعددة في المستوى الأعلى:

- `accounts` - سجل تجاوزات مسمّاة لكل حساب، مفهرسة حسب معرّف الحساب.
- `defaultAccount` - معرّف الحساب المفضّل عند ضبط عدة حسابات.

## المشغّلات

التحقق الذاتي على جانب المضيف (يكتب تقرير Markdown ضمن `.artifacts/qa-e2e/`):

```bash
pnpm qa:e2e
```

يوجّه هذا عبر `qa-lab`، ويبدأ ناقل ضمان الجودة داخل المستودع، ويشغّل شريحة وقت تشغيل `qa-channel`، وينفّذ تحققًا ذاتيًا حتميًا.

حزمة السيناريوهات الكاملة المدعومة بالمستودع:

```bash
pnpm openclaw qa suite
```

تشغّل السيناريوهات بالتوازي مقابل مسار Gateway الخاص بضمان الجودة. راجع [نظرة عامة على ضمان الجودة](/ar/concepts/qa-e2e-automation) للاطلاع على السيناريوهات والملفات التعريفية وأوضاع المزوّدين.

موقع ضمان الجودة المدعوم بـ Docker ‏(Gateway + واجهة مصحح QA Lab في حزمة واحدة):

```bash
pnpm qa:lab:up
```

يبني موقع ضمان الجودة، ويبدأ حزمة Gateway + QA Lab المدعومة بـ Docker، ويطبع عنوان URL الخاص بـ QA Lab. ومن هناك يمكن اختيار السيناريوهات، واختيار مسار النموذج، وتشغيل عمليات فردية، ومشاهدة النتائج مباشرة. مصحح QA Lab منفصل عن حزمة واجهة التحكم الموزّعة.
