## لماذا

- تأكد من تشغيل مثيل Gateway واحد فقط لكل منفذ أساسي على المضيف نفسه؛ يجب أن تستخدم مثيلات Gateway الإضافية ملفات تعريف معزولة ومنافذ فريدة.
- الصمود بعد الأعطال/`SIGKILL` من دون ترك ملفات قفل قديمة.
- الفشل بسرعة مع خطأ واضح عندما يكون منفذ التحكم مشغولًا بالفعل.

## الآلية

- يحصل Gateway أولًا على ملف قفل لكل إعداد ضمن دليل أقفال الحالة ويفحص المنفذ المضبوط بحثًا عن مستمع موجود.
- إذا كان مالك القفل المسجل غير موجود، أو كان المنفذ حرًا، أو كان القفل قديمًا، تستعيد عملية البدء القفل وتتابع.
- بعد ذلك يربط Gateway مستمع HTTP/WebSocket (الافتراضي `ws://127.0.0.1:18789`) باستخدام مستمع TCP حصري.
- إذا فشل الربط بسبب `EADDRINUSE`، تطرح عملية البدء `GatewayLockError("another gateway instance is already listening on ws://127.0.0.1:<port>")`.
- عند الإيقاف، يغلق Gateway خادم HTTP/WebSocket ويزيل ملف القفل.

## واجهة الأخطاء

- إذا كانت عملية أخرى تحتفظ بالمنفذ، تطرح عملية البدء `GatewayLockError("another gateway instance is already listening on ws://127.0.0.1:<port>")`.
- تظهر حالات فشل الربط الأخرى بصيغة `GatewayLockError("failed to bind gateway socket on ws://127.0.0.1:<port>: …")`.

## ملاحظات تشغيلية

- إذا كان المنفذ مشغولًا بواسطة عملية _أخرى_، يكون الخطأ نفسه؛ حرر المنفذ أو اختر منفذًا آخر باستخدام `openclaw gateway --port <port>`.
- ضمن مشرف خدمة، تنتهي عملية Gateway جديدة ترى مستجيب `/healthz` سليمًا موجودًا بنجاح وتترك تلك العملية في موضع التحكم. إذا لم تصبح العملية الموجودة سليمة أبدًا، تكون المحاولات محدودة وتفشل عملية البدء مع خطأ قفل واضح بدلًا من الدوران إلى الأبد.
- لا يزال تطبيق macOS يحافظ على حارس PID خفيف خاص به قبل تشغيل Gateway؛ يُفرض قفل وقت التشغيل بواسطة ملف القفل إضافةً إلى ربط HTTP/WebSocket.
