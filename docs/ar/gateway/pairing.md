في الاقتران الذي يملكه Gateway، يكون **Gateway** هو مصدر الحقيقة للعُقد
المسموح لها بالانضمام. واجهات المستخدم (تطبيق macOS، والعملاء المستقبليون) هي مجرد واجهات أمامية
توافق على الطلبات المعلّقة أو ترفضها.

**مهم:** تستخدم عُقد WS **اقتران الجهاز** (الدور `node`) أثناء `connect`.
`node.pair.*` هو مخزن اقتران منفصل ولا يتحكم في مصافحة WS.
العملاء الذين يستدعون `node.pair.*` صراحة فقط يستخدمون هذا التدفق.

## المفاهيم

- **طلب معلّق**: طلبت عقدة الانضمام؛ ويتطلب ذلك الموافقة.
- **عقدة مقترنة**: عقدة تمت الموافقة عليها وصدر لها رمز مصادقة.
- **النقل**: تمرّر نقطة نهاية WS في Gateway الطلبات لكنها لا تقرر
  العضوية. (أزيل دعم جسر TCP القديم.)

## كيف يعمل الاقتران

1. تتصل عقدة بنقطة WS في Gateway وتطلب الاقتران.
2. يخزن Gateway **طلبًا معلّقًا** ويصدر `node.pair.requested`.
3. توافق على الطلب أو ترفضه (CLI أو واجهة المستخدم).
4. عند الموافقة، يصدر Gateway **رمزًا جديدًا** (تُدوَّر الرموز عند إعادة الاقتران).
5. تعيد العقدة الاتصال باستخدام الرمز وتصبح الآن "مقترنة".

تنتهي صلاحية الطلبات المعلّقة تلقائيًا بعد **5 دقائق**.

## سير عمل CLI (ملائم للتشغيل دون واجهة)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

يعرض `nodes status` العُقد المقترنة/المتصلة وقدراتها.

## سطح API (بروتوكول gateway)

الأحداث:

- `node.pair.requested` — يصدر عند إنشاء طلب معلّق جديد.
- `node.pair.resolved` — يصدر عند الموافقة على طلب أو رفضه أو انتهاء صلاحيته.

الطرق:

- `node.pair.request` — إنشاء طلب معلّق أو إعادة استخدامه.
- `node.pair.list` — عرض العُقد المعلّقة + المقترنة (`operator.pairing`).
- `node.pair.approve` — الموافقة على طلب معلّق (يصدر رمزًا).
- `node.pair.reject` — رفض طلب معلّق.
- `node.pair.remove` — إزالة إدخال عقدة مقترنة قديم.
- `node.pair.verify` — التحقق من `{ nodeId, token }`.

ملاحظات:

- `node.pair.request` غير متغير النتيجة لكل عقدة: تعيد الاستدعاءات المتكررة الطلب
  المعلّق نفسه.
- تحدّث الطلبات المتكررة للعقدة المعلّقة نفسها أيضًا بيانات العقدة الوصفية المخزنة
  وأحدث لقطة أوامر مصرّح بها ضمن قائمة السماح لرؤية المشغّل.
- تنشئ الموافقة **دائمًا** رمزًا جديدًا؛ ولا يُعاد أي رمز مطلقًا من
  `node.pair.request`.
- قد تتضمن الطلبات `silent: true` كتلميح لتدفقات الموافقة التلقائية.
- يستخدم `node.pair.approve` أوامر الطلب المعلّق المعلنة لفرض
  نطاقات موافقة إضافية:
  - طلب بلا أوامر: `operator.pairing`
  - طلب أوامر غير تنفيذية: `operator.pairing` + `operator.write`
  - طلب `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
اقتران Node هو تدفق ثقة وهوية إضافة إلى إصدار الرموز. وهو لا يثبّت سطح أوامر العقدة الحية لكل عقدة.

- تأتي أوامر العقدة الحية مما تعلنه العقدة عند الاتصال بعد تطبيق سياسة أوامر العقد العامة في gateway (`gateway.nodes.allowCommands` و`denyCommands`).
- توجد سياسة السماح والسؤال الخاصة بـ `system.run` لكل عقدة على العقدة في `exec.approvals.node.*`، وليس في سجل الاقتران.

</Warning>

## تقييد أوامر Node (2026.3.31+)

<Warning>
**تغيير كاسر:** بدءًا من `2026.3.31`، تُعطَّل أوامر العقدة حتى تتم الموافقة على اقتران العقدة. لم يعد اقتران الجهاز وحده كافيًا لكشف أوامر العقدة المعلنة.
</Warning>

عندما تتصل عقدة للمرة الأولى، يُطلب الاقتران تلقائيًا. إلى أن تتم الموافقة على طلب الاقتران، تُرشَّح جميع أوامر العقدة المعلّقة من تلك العقدة ولن تُنفَّذ. بعد ترسيخ الثقة عبر الموافقة على الاقتران، تصبح أوامر العقدة المعلنة متاحة وفقًا لسياسة الأوامر العادية.

يعني هذا:

- يجب على العُقد التي كانت تعتمد سابقًا على اقتران الجهاز وحده لكشف الأوامر أن تكمل الآن اقتران العقدة.
- تُسقط الأوامر الموضوعة في قائمة الانتظار قبل الموافقة على الاقتران، ولا تؤجَّل.

## حدود الثقة لأحداث Node (2026.3.31+)

<Warning>
**تغيير كاسر:** تبقى عمليات التشغيل الصادرة من العقدة الآن ضمن سطح موثوق مخفّض.
</Warning>

تُقيَّد الملخصات الصادرة من العقدة وأحداث الجلسة ذات الصلة بالسطح الموثوق المقصود. قد تحتاج التدفقات المدفوعة بالإشعارات أو التي تشغّلها العقدة والتي كانت تعتمد سابقًا على وصول أوسع إلى أدوات المضيف أو الجلسة إلى تعديل. يضمن هذا التعزيز أن أحداث العقدة لا يمكنها التصعيد إلى وصول أدوات على مستوى المضيف يتجاوز ما تسمح به حدود ثقة العقدة.

تتبع تحديثات حضور العقدة الدائمة حدود الهوية نفسها. يُقبل حدث `node.presence.alive`
فقط من جلسات أجهزة عقدة مصادَق عليها، ولا يحدّث بيانات الاقتران الوصفية إلا عندما تكون
هوية الجهاز/العقدة مقترنة بالفعل. لا تكفي قيم `client.id` المعلنة ذاتيًا لكتابة
حالة آخر ظهور.

## الموافقة التلقائية (تطبيق macOS)

يمكن لتطبيق macOS اختياريًا محاولة **موافقة صامتة** عندما:

- يكون الطلب موسومًا بـ `silent`، و
- يستطيع التطبيق التحقق من اتصال SSH بمضيف gateway باستخدام المستخدم نفسه.

إذا فشلت الموافقة الصامتة، يعود إلى مطالبة "موافقة/رفض" العادية.

## الموافقة التلقائية للأجهزة عبر CIDR الموثوق

يبقى اقتران أجهزة WS لـ `role: node` يدويًا افتراضيًا. بالنسبة إلى شبكات
العُقد الخاصة حيث يثق Gateway بالفعل بمسار الشبكة، يمكن للمشغّلين
الاشتراك باستخدام CIDR صريحة أو عناوين IP دقيقة:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

حدود الأمان:

- يكون معطّلًا عندما لا يكون `gateway.nodes.pairing.autoApproveCidrs` معيّنًا.
- لا يوجد وضع موافقة تلقائية شامل للشبكة المحلية LAN أو الشبكة الخاصة.
- فقط اقتران الجهاز الجديد لـ `role: node` من دون نطاقات مطلوبة يكون مؤهلًا.
- يبقى عملاء المشغّل، والمتصفح، وControl UI، وWebChat يدويين.
- تبقى ترقيات الدور والنطاق والبيانات الوصفية والمفتاح العام يدوية.
- لا تكون مسارات ترويسة الوكيل الموثوق عبر same-host loopback مؤهلة لأن ذلك
  المسار يمكن انتحاله من قِبل المستدعين المحليين.

## الموافقة التلقائية على ترقية البيانات الوصفية

عندما يعيد جهاز مقترن بالفعل الاتصال مع تغييرات غير حساسة في البيانات الوصفية فقط
(مثل اسم العرض أو تلميحات منصة العميل)، يعامل OpenClaw ذلك
كـ `metadata-upgrade`. الموافقة التلقائية الصامتة ضيقة: فهي تنطبق فقط
على عمليات إعادة الاتصال المحلية الموثوقة غير المتصفحية التي أثبتت مسبقًا حيازة بيانات اعتماد محلية
أو مشتركة، بما في ذلك عمليات إعادة اتصال تطبيقات same-host الأصلية بعد تغييرات بيانات وصفية لإصدار نظام التشغيل. لا يزال عملاء المتصفح/Control UI والعملاء البعيدون
يستخدمون تدفق إعادة الموافقة الصريح. ترقيات النطاق (من قراءة إلى كتابة/admin) وتغييرات
المفتاح العام **ليست** مؤهلة للموافقة التلقائية على ترقية البيانات الوصفية —
وتبقى طلبات إعادة موافقة صريحة.

## أدوات مساعدة لاقتران QR

يعرض `/pair qr` حمولة الاقتران كوسائط منظمة حتى يتمكن عملاء الهاتف المحمول
والمتصفح من مسحها مباشرة.

كما يؤدي حذف جهاز إلى مسح أي طلبات اقتران معلّقة قديمة لذلك
معرّف الجهاز، لذلك لا يعرض `nodes pending` صفوفًا يتيمة بعد الإلغاء.

## المحلية والترويسات المعاد توجيهها

يعامل اقتران Gateway اتصالًا ما على أنه loopback فقط عندما يتفق كل من المقبس الخام
وأي دليل وكيل علوي. إذا وصل طلب عبر loopback لكنه
يحمل ترويسات `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto`
تشير إلى أصل غير محلي، فإن دليل الترويسة المعاد توجيهها هذا يستبعد
ادعاء محلية loopback. حينها يتطلب مسار الاقتران موافقة صريحة
بدلًا من معاملة الطلب صامتًا كاتصال same-host. راجع
[مصادقة الوكيل الموثوق](/ar/gateway/trusted-proxy-auth) لمعرفة القاعدة المكافئة في
مصادقة المشغّل.

## التخزين (محلي، خاص)

تُخزَّن حالة الاقتران ضمن دليل حالة Gateway (الافتراضي `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

إذا تجاوزت `OPENCLAW_STATE_DIR`، ينتقل مجلد `nodes/` معه.

ملاحظات الأمان:

- الرموز أسرار؛ تعامل مع `paired.json` كملف حساس.
- يتطلب تدوير الرمز إعادة الموافقة (أو حذف إدخال العقدة).

## سلوك النقل

- النقل **عديم الحالة**؛ لا يخزن العضوية.
- إذا كان Gateway غير متصل أو كان الاقتران معطّلًا، فلن تتمكن العُقد من الاقتران.
- إذا كان Gateway في الوضع البعيد، فسيظل الاقتران يحدث مقابل مخزن Gateway البعيد.
