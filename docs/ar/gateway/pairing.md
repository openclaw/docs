في الاقتران المملوك من Gateway، يكون **Gateway** هو مصدر الحقيقة للعُقد المسموح لها بالانضمام. واجهات المستخدم (تطبيق macOS، والعملاء المستقبليون) ليست سوى واجهات أمامية توافق على الطلبات المعلّقة أو ترفضها.

**مهم:** تستخدم عُقد WS **اقتران الأجهزة** (الدور `node`) أثناء `connect`.
`node.pair.*` هو مخزن اقتران منفصل ولا يتحكم في مصافحة WS. العملاء الذين يستدعون `node.pair.*` صراحةً هم فقط من يستخدمون هذا التدفق.

## المفاهيم

- **طلب معلّق**: طلبت عقدة الانضمام؛ يتطلب موافقة.
- **Node مقترنة**: عقدة تمت الموافقة عليها مع رمز مصادقة صادر.
- **النقل**: تعيد نقطة نهاية WS في Gateway توجيه الطلبات لكنها لا تقرر العضوية. (أزيل دعم جسر TCP القديم.)

## كيفية عمل الاقتران

1. تتصل عقدة بـ Gateway WS وتطلب الاقتران.
2. يخزّن Gateway **طلبًا معلّقًا** ويصدر `node.pair.requested`.
3. توافق على الطلب أو ترفضه (CLI أو واجهة المستخدم).
4. عند الموافقة، يصدر Gateway **رمزًا جديدًا** (تُدار الرموز عند إعادة الاقتران).
5. تعيد العقدة الاتصال باستخدام الرمز وتصبح الآن "مقترنة".

تنتهي صلاحية الطلبات المعلّقة تلقائيًا بعد **5 دقائق**.

## سير عمل CLI (ملائم للبيئات دون واجهة)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

يعرض `nodes status` العُقد المقترنة/المتصلة وإمكاناتها.

## سطح API (بروتوكول gateway)

الأحداث:

- `node.pair.requested` — يصدر عند إنشاء طلب معلّق جديد.
- `node.pair.resolved` — يصدر عند الموافقة على طلب/رفضه/انتهاء صلاحيته.

الطرائق:

- `node.pair.request` — إنشاء طلب معلّق أو إعادة استخدامه.
- `node.pair.list` — سرد العُقد المعلّقة + المقترنة (`operator.pairing`).
- `node.pair.approve` — الموافقة على طلب معلّق (يصدر رمزًا).
- `node.pair.reject` — رفض طلب معلّق.
- `node.pair.remove` — إزالة إدخال Node مقترنة قديم.
- `node.pair.verify` — التحقق من `{ nodeId, token }`.

ملاحظات:

- `node.pair.request` عديم الأثر المتكرر لكل عقدة: تعيد الاستدعاءات المتكررة الطلب المعلّق نفسه.
- تحدّث الطلبات المتكررة للعقدة المعلّقة نفسها أيضًا بيانات تعريف العقدة المخزّنة وأحدث لقطة أوامر مصرّح بها ومدرجة في قائمة السماح لظهور المشغّل.
- تولّد الموافقة **دائمًا** رمزًا جديدًا؛ لا يُعاد أي رمز مطلقًا من `node.pair.request`.
- تُلخّص مستويات نطاق المشغّل وفحوصات وقت الموافقة في [نطاقات المشغّل](/ar/gateway/operator-scopes).
- قد تتضمن الطلبات `silent: true` كتلميح لتدفقات الموافقة التلقائية.
- يستخدم `node.pair.approve` الأوامر المصرّح بها في الطلب المعلّق لفرض نطاقات موافقة إضافية:
  - طلب بلا أوامر: `operator.pairing`
  - طلب أمر غير تنفيذي: `operator.pairing` + `operator.write`
  - طلب `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
اقتران Node هو تدفق ثقة وهوية بالإضافة إلى إصدار الرموز. إنه **لا** يثبّت سطح أوامر Node الحية لكل عقدة.

- تأتي أوامر Node الحية مما تصرّح به العقدة عند الاتصال بعد تطبيق سياسة أوامر Node العامة في gateway (`gateway.nodes.allowCommands` و`denyCommands`).
- توجد سياسة السماح والسؤال لكل Node لـ `system.run` على العقدة في `exec.approvals.node.*`، وليس في سجل الاقتران.

</Warning>

## ضبط أوامر Node (2026.3.31+)

<Warning>
**تغيير كاسر:** بدءًا من `2026.3.31`، تُعطّل أوامر Node إلى أن تتم الموافقة على اقتران Node. لم يعد اقتران الجهاز وحده كافيًا لكشف أوامر Node المصرّح بها.
</Warning>

عندما تتصل عقدة للمرة الأولى، يُطلب الاقتران تلقائيًا. إلى أن تتم الموافقة على طلب الاقتران، تُرشّح كل أوامر Node المعلّقة من تلك العقدة ولن تُنفّذ. بمجرد إنشاء الثقة عبر الموافقة على الاقتران، تصبح الأوامر التي صرّحت بها العقدة متاحة وفقًا لسياسة الأوامر العادية.

يعني ذلك:

- يجب على العُقد التي كانت تعتمد سابقًا على اقتران الجهاز وحده لكشف الأوامر أن تكمل الآن اقتران Node.
- تُسقط الأوامر المصطفة قبل الموافقة على الاقتران، ولا تُؤجّل.

## حدود الثقة لأحداث Node (2026.3.31+)

<Warning>
**تغيير كاسر:** تبقى التشغيلات الناشئة من Node الآن على سطح موثوق مخفّض.
</Warning>

تُقيّد الملخصات الناشئة من Node وأحداث الجلسات ذات الصلة بالسطح الموثوق المقصود. قد تحتاج التدفقات المدفوعة بالإشعارات أو التي تشغّلها Node، والتي كانت تعتمد سابقًا على وصول أوسع إلى أدوات المضيف أو الجلسة، إلى تعديل. يضمن هذا التقوية ألا تتمكن أحداث Node من التصعيد إلى وصول أدوات على مستوى المضيف بما يتجاوز ما تسمح به حدود ثقة Node.

تتبع تحديثات حضور Node الدائمة حد الهوية نفسه. يُقبل حدث `node.presence.alive` فقط من جلسات أجهزة Node المصادق عليها ويحدّث بيانات تعريف الاقتران فقط عندما تكون هوية الجهاز/العقدة مقترنة مسبقًا. لا تكفي قيم `client.id` المصرّح بها ذاتيًا لكتابة حالة آخر ظهور.

## الموافقة التلقائية (تطبيق macOS)

يمكن لتطبيق macOS اختياريًا محاولة **موافقة صامتة** عندما:

- يكون الطلب موسومًا بـ `silent`، و
- يستطيع التطبيق التحقق من اتصال SSH بمضيف gateway باستخدام المستخدم نفسه.

إذا فشلت الموافقة الصامتة، يعود إلى مطالبة "موافقة/رفض" العادية.

## الموافقة التلقائية للأجهزة عبر CIDR موثوق

يبقى اقتران أجهزة WS لـ `role: node` يدويًا افتراضيًا. بالنسبة إلى شبكات Node الخاصة حيث يثق Gateway مسبقًا بمسار الشبكة، يمكن للمشغّلين الاشتراك باستخدام CIDR صريحة أو عناوين IP دقيقة:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

حد الأمان:

- يكون معطّلًا عندما لا تكون `gateway.nodes.pairing.autoApproveCidrs` مضبوطة.
- لا يوجد وضع موافقة تلقائية شامل لشبكة LAN أو الشبكات الخاصة.
- لا يكون مؤهلًا سوى اقتران جهاز `role: node` جديد بلا نطاقات مطلوبة.
- يظل عملاء المشغّل والمتصفح وControl UI وWebChat يدويين.
- تظل ترقيات الدور والنطاق والبيانات التعريفية والمفتاح العام يدوية.
- لا تكون مسارات ترويسة الوكيل الموثوق عبر local loopback على المضيف نفسه مؤهلة لأن ذلك المسار يمكن انتحاله من قبل المستدعين المحليين.

## الموافقة التلقائية على ترقية البيانات التعريفية

عندما يعيد جهاز مقترن بالفعل الاتصال مع تغييرات بيانات تعريف غير حساسة فقط (على سبيل المثال، اسم العرض أو تلميحات منصة العميل)، يتعامل OpenClaw مع ذلك على أنه `metadata-upgrade`. الموافقة التلقائية الصامتة ضيقة: تنطبق فقط على عمليات إعادة الاتصال المحلية الموثوقة غير المتصفحية التي أثبتت بالفعل حيازة بيانات اعتماد محلية أو مشتركة، بما في ذلك عمليات إعادة اتصال التطبيق الأصلي على المضيف نفسه بعد تغييرات بيانات تعريف إصدار نظام التشغيل. لا يزال عملاء المتصفح/Control UI والعملاء البعيدون يستخدمون تدفق إعادة الموافقة الصريح. ترقيات النطاق (من قراءة إلى كتابة/مسؤول) وتغييرات المفتاح العام **ليست** مؤهلة للموافقة التلقائية على ترقية البيانات التعريفية — بل تبقى طلبات إعادة موافقة صريحة.

## مساعدات اقتران QR

يعرض `/pair qr` حمولة الاقتران كوسائط منظمة بحيث يمكن لعملاء الهاتف المحمول والمتصفح مسحها مباشرة.

يحذف حذف جهاز أيضًا أي طلبات اقتران معلّقة قديمة لمعرّف ذلك الجهاز، بحيث لا يعرض `nodes pending` صفوفًا يتيمة بعد الإلغاء.

## المحلية والترويسات المعاد توجيهها

يتعامل اقتران Gateway مع الاتصال على أنه loopback فقط عندما يتفق كل من المقبس الخام وأي دليل وكيل أعلى. إذا وصل طلب على loopback لكنه يحمل ترويسات `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto` تشير إلى أصل غير محلي، فإن دليل الترويسة المعاد توجيهها هذا يستبعد ادعاء محلية loopback. عندها يتطلب مسار الاقتران موافقة صريحة بدلًا من التعامل مع الطلب بصمت كاتصال من المضيف نفسه. راجع [مصادقة الوكيل الموثوق](/ar/gateway/trusted-proxy-auth) للقاعدة المكافئة في مصادقة المشغّل.

## التخزين (محلي، خاص)

تُخزّن حالة الاقتران ضمن دليل حالة Gateway (الافتراضي `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

إذا تجاوزت `OPENCLAW_STATE_DIR`، ينتقل مجلد `nodes/` معه.

ملاحظات أمان:

- الرموز أسرار؛ تعامل مع `paired.json` على أنه حساس.
- يتطلب تدوير رمز إعادة موافقة (أو حذف إدخال العقدة).

## سلوك النقل

- النقل **عديم الحالة**؛ لا يخزّن العضوية.
- إذا كان Gateway غير متصل أو كان الاقتران معطّلًا، فلا يمكن للعُقد الاقتران.
- إذا كان Gateway في الوضع البعيد، لا يزال الاقتران يحدث مقابل مخزن Gateway البعيد.
