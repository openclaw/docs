# الاكتشاف ووسائل النقل

لدى OpenClaw مشكلتان متميزتان تبدوان متشابهتين ظاهريًا:

1. **التحكم البعيد للمشغّل**: تطبيق شريط قوائم macOS الذي يتحكم في Gateway يعمل في مكان آخر.
2. **إقران Node**: عثور iOS/Android (والعقد المستقبلية) على Gateway والإقران بأمان.

هدف التصميم هو إبقاء كل اكتشاف الشبكة/الإعلان عنها داخل **Node Gateway** (`openclaw gateway`) وإبقاء العملاء (تطبيق Mac وiOS) كمستهلكين.

## المصطلحات

- **Gateway**: عملية Gateway واحدة طويلة التشغيل تمتلك الحالة (الجلسات، الإقران، سجل Node) وتشغّل القنوات. تستخدم معظم الإعدادات واحدًا لكل مضيف؛ كما أن إعدادات Gateway المتعددة المعزولة ممكنة.
- **Gateway WS (مستوى التحكم)**: نقطة نهاية WebSocket على `127.0.0.1:18789` افتراضيًا؛ يمكن ربطها بشبكة LAN/‏tailnet عبر `gateway.bind`.
- **نقل WS مباشر**: نقطة نهاية Gateway WS مواجهة لشبكة LAN/‏tailnet (من دون SSH).
- **نقل SSH (احتياطي)**: تحكم بعيد عبر تمرير `127.0.0.1:18789` عبر SSH.
- **جسر TCP القديم (أُزيل)**: نقل Node أقدم (راجع
  [بروتوكول الجسر](/ar/gateway/bridge-protocol))؛ لم يعد يُعلن عنه
  للاكتشاف ولم يعد جزءًا من الإصدارات الحالية.

تفاصيل البروتوكول:

- [بروتوكول Gateway](/ar/gateway/protocol)
- [بروتوكول الجسر (قديم)](/ar/gateway/bridge-protocol)

## لماذا نُبقي على كل من "المباشر" وSSH

- **WS المباشر** يوفّر أفضل تجربة استخدام على الشبكة نفسها وداخل tailnet:
  - الاكتشاف التلقائي على LAN عبر Bonjour
  - رموز الإقران + قوائم ACL يملكها Gateway
  - لا يلزم وصول إلى shell؛ يمكن أن يبقى سطح البروتوكول محدودًا وقابلًا للتدقيق
- يبقى **SSH** الاحتياطي الشامل:
  - يعمل في أي مكان يتوفر لديك فيه وصول SSH (حتى عبر شبكات غير مرتبطة)
  - يتجاوز مشكلات multicast/mDNS
  - لا يتطلب منافذ واردة جديدة غير SSH

## مدخلات الاكتشاف (كيف يعرف العملاء مكان Gateway)

### 1) اكتشاف Bonjour / DNS-SD

Bonjour متعدد البث يعمل بأفضل جهد ولا يعبر الشبكات. يمكن لـ OpenClaw أيضًا تصفح
إشارة Gateway نفسها عبر نطاق DNS-SD واسع النطاق مهيأ، وبذلك يمكن للاكتشاف أن يغطي:

- `local.` على LAN نفسها
- نطاق DNS-SD أحادي البث مهيأ للاكتشاف عبر الشبكات

الاتجاه المستهدف:

- يعلن **Gateway** عن نقطة نهاية WS الخاصة به عبر Bonjour عندما يكون Plugin
  `bonjour` المضمّن مفعّلًا. يبدأ Plugin تلقائيًا على مضيفات macOS ويكون
  اختياريًا في غير ذلك.
- يتصفح العملاء ويعرضون قائمة "اختر Gateway"، ثم يخزنون نقطة النهاية المختارة.

تفاصيل استكشاف الأخطاء وإشارات الإعلان: [Bonjour](/ar/gateway/bonjour).

#### تفاصيل إشارة الخدمة

- أنواع الخدمات:
  - `_openclaw-gw._tcp` (إشارة نقل Gateway)
- مفاتيح TXT (غير سرية):
  - `role=gateway`
  - `transport=gateway`
  - `displayName=<friendly name>` (اسم عرض يضبطه المشغّل)
  - `lanHost=<hostname>.local`
  - `gatewayPort=18789` (Gateway WS + HTTP)
  - `gatewayTls=1` (فقط عند تفعيل TLS)
  - `gatewayTlsSha256=<sha256>` (فقط عند تفعيل TLS وتوفر البصمة)
  - `canvasPort=<port>` (منفذ مضيف اللوحة؛ حاليًا هو نفسه `gatewayPort` عند تفعيل مضيف اللوحة)
  - `tailnetDns=<magicdns>` (تلميح اختياري؛ يُكتشف تلقائيًا عند توفر Tailscale)
  - `sshPort=<port>` (وضع mDNS الكامل فقط؛ قد يحذفه DNS-SD واسع النطاق، وفي هذه الحالة تبقى افتراضيات SSH على `22`)
  - `cliPath=<path>` (وضع mDNS الكامل فقط؛ ما زال DNS-SD واسع النطاق يكتبه كتلميح تثبيت بعيد)

ملاحظات الأمان:

- سجلات Bonjour/mDNS TXT **غير مصادَق عليها**. يجب أن يتعامل العملاء مع قيم TXT كتلميحات لتجربة الاستخدام فقط.
- يجب أن يفضّل التوجيه (المضيف/المنفذ) **نقطة نهاية الخدمة المحلولة** (SRV + A/AAAA) على `lanHost` أو `tailnetDns` أو `gatewayPort` المقدمة عبر TXT.
- يجب ألا يسمح تثبيت TLS أبدًا لقيمة `gatewayTlsSha256` المُعلن عنها بتجاوز تثبيت مخزن سابقًا.
- يجب أن تتطلب عقد iOS/Android تأكيدًا صريحًا "وثق بهذه البصمة" قبل تخزين تثبيت لأول مرة (تحقق خارج النطاق) كلما كان المسار المختار آمنًا/مستندًا إلى TLS.

التفعيل/التعطيل/التجاوز:

- `openclaw plugins enable bonjour` يفعّل إعلان LAN متعدد البث.
- `OPENCLAW_DISABLE_BONJOUR=1` يعطّل الإعلان.
- عندما يكون Plugin Bonjour مفعّلًا و`OPENCLAW_DISABLE_BONJOUR` غير مضبوط،
  يعلن Bonjour على المضيفات العادية ويتعطل تلقائيًا داخل الحاويات المكتشفة.
  تشغيل macOS Gateway بتكوين فارغ يفعّل Plugin تلقائيًا؛ أما عمليات نشر Linux
  وWindows والحاويات فتحتاج إلى تفعيل صريح.
  استخدم `0` فقط على المضيف أو macvlan أو شبكة أخرى تدعم mDNS؛ واستخدم `1`
  لفرض التعطيل.
- يتحكم `gateway.bind` في `~/.openclaw/openclaw.json` في وضع ربط Gateway.
- يتجاوز `OPENCLAW_SSH_PORT` منفذ SSH المُعلن عنه عند إصدار `sshPort`.
- ينشر `OPENCLAW_TAILNET_DNS` تلميح `tailnetDns` (MagicDNS).
- يتجاوز `OPENCLAW_CLI_PATH` مسار CLI المُعلن عنه.

### 2) Tailnet (عبر الشبكات)

لإعدادات بنمط لندن/فيينا، لن يفيد Bonjour. الهدف "المباشر" الموصى به هو:

- اسم Tailscale MagicDNS (مفضل) أو IP ثابت داخل tailnet.

إذا تمكن Gateway من اكتشاف أنه يعمل تحت Tailscale، فإنه ينشر `tailnetDns` كتلميح اختياري للعملاء (بما في ذلك الإشارات واسعة النطاق).

يفضل تطبيق macOS الآن أسماء MagicDNS على عناوين IP الخام الخاصة بـ Tailscale لاكتشاف Gateway. يحسّن ذلك الاعتمادية عندما تتغير عناوين IP الخاصة بـ tailnet (على سبيل المثال بعد إعادة تشغيل Node أو إعادة تعيين CGNAT)، لأن أسماء MagicDNS تُحل إلى عنوان IP الحالي تلقائيًا.

بالنسبة لإقران Node المحمولة، لا تخفف تلميحات الاكتشاف أمان النقل على مسارات tailnet/العامة:

- ما زالت iOS/Android تتطلب مسار اتصال أولي آمنًا على tailnet/عام (`wss://` أو Tailscale Serve/Funnel).
- عنوان IP الخام المكتشف داخل tailnet هو تلميح توجيه، وليس إذنًا لاستخدام `ws://` بعيد بنص صريح.
- يبقى الاتصال المباشر الخاص عبر LAN باستخدام `ws://` مدعومًا.
- إذا أردت أبسط مسار Tailscale لعقد المحمول، فاستخدم Tailscale Serve لكي يحل كل من الاكتشاف ورمز الإعداد إلى نقطة نهاية MagicDNS الآمنة نفسها.

### 3) هدف يدوي / SSH

عندما لا يكون هناك مسار مباشر (أو يكون المباشر معطلًا)، يمكن للعملاء دائمًا الاتصال عبر SSH بتمرير منفذ Gateway على loopback.

راجع [الوصول البعيد](/ar/gateway/remote).

## اختيار النقل (سياسة العميل)

سلوك العميل الموصى به:

1. إذا كانت نقطة نهاية مباشرة مقترنة مهيأة وقابلة للوصول، فاستخدمها.
2. وإلا، إذا وجد الاكتشاف Gateway على `local.` أو النطاق واسع النطاق المهيأ، فاعرض خيار "استخدم Gateway هذا" بنقرة واحدة واحفظه كنقطة النهاية المباشرة.
3. وإلا، إذا كان DNS/IP الخاص بـ tailnet مهيأ، فجرّب المباشر.
   بالنسبة لعقد المحمول على مسارات tailnet/العامة، يعني المباشر نقطة نهاية آمنة، وليس `ws://` بعيدًا بنص صريح.
4. وإلا، فارجع إلى SSH.

## الإقران + المصادقة (النقل المباشر)

Gateway هو مصدر الحقيقة لقبول Node/العميل.

- تُنشأ طلبات الإقران/تُوافق/تُرفض في Gateway (راجع [إقران Gateway](/ar/gateway/pairing)).
- يفرض Gateway:
  - المصادقة (رمز / زوج مفاتيح)
  - النطاقات/قوائم ACL (Gateway ليس وكيلًا خامًا لكل طريقة)
  - حدود المعدل

## المسؤوليات حسب المكوّن

- **Gateway**: يعلن إشارات الاكتشاف، ويمتلك قرارات الإقران، ويستضيف نقطة نهاية WS.
- **تطبيق macOS**: يساعدك في اختيار Gateway، ويعرض مطالبات الإقران، ويستخدم SSH كاحتياطي فقط.
- **عقد iOS/Android**: تتصفح Bonjour للتيسير وتتصل بـ Gateway WS المقترن.
