تحدد نطاقات المشغّل ما يجوز لعميل Gateway فعله بعد المصادقة.
وهي حاجز حماية في مستوى التحكم داخل نطاق مشغّل Gateway موثوق واحد،
وليست عزلاً معادياً متعدد المستأجرين. إذا كنت تحتاج إلى فصل قوي بين
الأشخاص أو الفرق أو الأجهزة، فشغّل Gateways منفصلة تحت مستخدمي نظام تشغيل أو
مضيفين منفصلين.

ذات صلة: [الأمان](/ar/gateway/security)، [بروتوكول Gateway](/ar/gateway/protocol)،
[إقران Gateway](/ar/gateway/pairing)، [CLI الأجهزة](/ar/cli/devices).

## الأدوار

يتصل عملاء WebSocket في Gateway بدور واحد:

- `operator`: عملاء مستوى التحكم مثل CLI، وواجهة التحكم، والأتمتة، وعمليات
  المساعدة الموثوقة.
- `node`: مضيفو القدرات مثل macOS أو iOS أو Android أو العقد بلا واجهة التي
  تعرض الأوامر عبر `node.invoke`.

تتطلب طرق RPC الخاصة بالمشغّل دور `operator`. وتتطلب الطرق الصادرة من العقدة
دور `node`.

## مستويات النطاق

| النطاق                  | المعنى                                                                                                                                                                                   |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `operator.read`         | حالة للقراءة فقط، وقوائم، وفهرس، وسجلات، وقراءات جلسات، واستدعاءات أخرى غير معدِّلة في مستوى التحكم.                                                                                   |
| `operator.write`        | إجراءات المشغّل المعدِّلة العادية مثل إرسال الرسائل، واستدعاء الأدوات، وتحديث إعدادات الحديث/الصوت، وتمرير أوامر العقد. كما يفي بـ `operator.read`.                                    |
| `operator.admin`        | وصول إداري إلى مستوى التحكم. يفي بكل نطاق `operator.*`. مطلوب لتعديل الإعدادات، والتحديثات، والخطافات الأصلية، ومساحات الأسماء المحجوزة الحساسة، والموافقات عالية المخاطر.              |
| `operator.pairing`      | إدارة إقران الأجهزة والعقد، بما في ذلك سرد سجلات الإقران أو رموز الأجهزة أو الموافقة عليها أو رفضها أو إزالتها أو تدويرها أو إبطالها.                                                  |
| `operator.approvals`    | واجهات API الخاصة بموافقات التنفيذ وPlugin.                                                                                                                                              |
| `operator.talk.secrets` | قراءة إعدادات الحديث مع تضمين الأسرار.                                                                                                                                                  |

تتطلب نطاقات `operator.*` المستقبلية غير المعروفة تطابقاً دقيقاً ما لم يكن لدى المستدعي
`operator.admin`.

## نطاق الطريقة هو البوابة الأولى فقط

لكل RPC في Gateway نطاق طريقة بأقل امتيازات. يحدد نطاق الطريقة ذلك
ما إذا كان الطلب يمكنه الوصول إلى المعالج. ثم تطبق بعض المعالجات فحوصاً أشد
وقت الموافقة بناءً على الشيء المحدد الذي تتم الموافقة عليه أو تعديله.

أمثلة:

- يمكن الوصول إلى `device.pair.approve` باستخدام `operator.pairing`، لكن الموافقة على
  جهاز مشغّل لا يمكنها إلا سك أو الحفاظ على النطاقات التي يمتلكها المستدعي بالفعل.
- يمكن الوصول إلى `node.pair.approve` باستخدام `operator.pairing`، ثم يشتق نطاقات
  موافقة إضافية من قائمة أوامر العقدة المعلّقة.
- تكون `chat.send` عادةً طريقة بنطاق كتابة، لكن `/config set` و`/config unset`
  الدائمين يتطلبان `operator.admin` على مستوى الأمر.

يسمح هذا للمشغّلين ذوي النطاق الأدنى بتنفيذ إجراءات إقران منخفضة المخاطر دون جعل
كل موافقات الإقران مقتصرة على المسؤول.

## موافقات إقران الأجهزة

سجلات إقران الأجهزة هي المصدر الدائم للأدوار والنطاقات المعتمدة.
لا تحصل الأجهزة المقترنة مسبقاً على وصول أوسع بصمت: عمليات إعادة الاتصال التي تطلب
دوراً أوسع أو نطاقات أوسع تنشئ طلب ترقية معلّقاً جديداً.

عند الموافقة على طلب جهاز:

- لا يحتاج الطلب الذي لا يحتوي على دور مشغّل إلى موافقة على نطاق رمز المشغّل.
- يتطلب طلب `operator.read` أو `operator.write` أو `operator.approvals`
  أو `operator.pairing` أو `operator.talk.secrets` أن يمتلك المستدعي
  تلك النطاقات، أو `operator.admin`.
- يتطلب طلب `operator.admin` وجود `operator.admin`.
- يمكن لطلب إصلاح بلا نطاقات صريحة أن يرث نطاقات رمز المشغّل الحالية.
  إذا كان ذلك الرمز الحالي بنطاق مسؤول، فستظل الموافقة تتطلب
  `operator.admin`.

بالنسبة إلى جلسات رموز الأجهزة المقترنة، تكون الإدارة ذاتية النطاق ما لم يكن لدى المستدعي
أيضاً `operator.admin`: يمكن للمستدعين غير المسؤولين تدوير أو إبطال أو إزالة
إدخال جهازهم فقط.

## موافقات إقران العقد

يستخدم `node.pair.*` القديم مخزن إقران عقد منفصلاً مملوكاً لـ Gateway. تستخدم عقد WS
إقران الأجهزة مع `role: node`، لكن مفردات مستوى الموافقة نفسها
تنطبق.

يستخدم `node.pair.approve` قائمة أوامر الطلب المعلّق لاشتقاق
النطاقات الإضافية المطلوبة:

- طلب بلا أوامر: `operator.pairing`
- أوامر عقد غير تنفيذية: `operator.pairing` + `operator.write`
- `system.run` أو `system.run.prepare` أو `system.which`:
  `operator.pairing` + `operator.admin`

ينشئ إقران العقد الهوية والثقة. ولا يحل محل سياسة موافقة التنفيذ
الخاصة بـ `system.run` لدى العقدة نفسها.

## مصادقة السر المشترك

تُعامل مصادقة رمز/كلمة مرور Gateway المشتركة كوسيلة وصول مشغّل موثوقة لذلك
Gateway. تستعيد أسطح HTTP المتوافقة مع OpenAI و`/tools/invoke`
مجموعة النطاقات الافتراضية الكاملة العادية للمشغّل لمصادقة الحامل بالسر المشترك، حتى إذا
أرسل المستدعي نطاقات معلنة أضيق.

يمكن لأنماط تحمل الهوية، مثل مصادقة الوكيل الموثوق أو `none` في الدخول الخاص،
أن تظل قادرة على احترام النطاقات الصريحة المعلنة. استخدم Gateways منفصلة للفصل الحقيقي
لحدود الثقة.
