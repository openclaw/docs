يمكن لـ OpenClaw إنشاء ملف zip تشخيصي محلي لتقارير الأخطاء. يجمع
حالة Gateway المنقّاة، والصحة، والسجلات، وشكل الإعدادات، وأحداث الاستقرار
الحديثة الخالية من الحمولات.

تعامل مع حزم التشخيص كالأسرار إلى أن تراجعها. صُممت لحذف الحمولات
وبيانات الاعتماد أو تنقيحها، لكنها لا تزال تلخص سجلات Gateway المحلية
وحالة وقت التشغيل على مستوى المضيف.

## بدء سريع

```bash
openclaw gateway diagnostics export
```

يطبع الأمر مسار ملف zip المكتوب. لاختيار مسار:

```bash
openclaw gateway diagnostics export --output openclaw-diagnostics.zip
```

للأتمتة:

```bash
openclaw gateway diagnostics export --json
```

## أمر المحادثة

يمكن للمالكين استخدام `/diagnostics [note]` في المحادثة لطلب تصدير Gateway محلي.
استخدم هذا عندما يحدث الخطأ في محادثة حقيقية وتريد تقريرًا واحدًا
قابلًا للنسخ واللصق للدعم:

1. أرسل `/diagnostics` في المحادثة التي لاحظت فيها المشكلة. أضف
   ملاحظة قصيرة إذا كان ذلك مفيدًا، مثل `/diagnostics bad tool choice`.
2. يرسل OpenClaw تمهيد التشخيص ويطلب موافقة تنفيذ صريحة واحدة.
   تشغّل الموافقة `openclaw gateway diagnostics export --json`.
   لا توافق على التشخيص عبر قاعدة سماح شاملة.
3. بعد الموافقة، يرد OpenClaw بتقرير قابل للصق يحتوي على مسار الحزمة المحلي،
   وملخص البيان، وملاحظات الخصوصية، ومعرفات الجلسات ذات الصلة.

في محادثات المجموعات، لا يزال بإمكان المالك تشغيل `/diagnostics`، لكن OpenClaw لا
ينشر تفاصيل التشخيص مرة أخرى في المحادثة المشتركة. يرسل التمهيد،
ومطالبات الموافقة، ونتيجة تصدير Gateway، وتفصيل جلسة/سلسلة Codex إلى
المالك عبر مسار الموافقة الخاص. لا تحصل المجموعة إلا على إشعار قصير
بأن تدفق التشخيص أُرسل بشكل خاص. إذا لم يتمكن OpenClaw من العثور على مسار
خاص للمالك، يفشل الأمر بأمان ويطلب من المالك تشغيله من رسالة مباشرة.

عندما تستخدم جلسة OpenClaw النشطة حاضنة OpenAI Codex الأصلية،
تغطي موافقة التنفيذ نفسها أيضًا رفع ملاحظات OpenAI لسلاسل وقت تشغيل Codex
التي يعرفها OpenClaw. يكون هذا الرفع منفصلًا عن ملف zip المحلي لـ Gateway
ولا يظهر إلا لجلسات حاضنة Codex. قبل الموافقة، يوضح المطلب أن الموافقة
على التشخيص سترسل أيضًا ملاحظات Codex، لكنه لا يسرد معرفات جلسات أو
سلاسل Codex. بعد الموافقة، يسرد رد المحادثة القنوات، ومعرفات جلسات OpenClaw،
ومعرفات سلاسل Codex، وأوامر الاستئناف المحلية للسلاسل التي أُرسلت إلى
خوادم OpenAI. إذا رفضت الموافقة أو تجاهلتها، فلن يشغّل OpenClaw التصدير،
ولن يرسل ملاحظات Codex، ولن يطبع معرفات Codex.

هذا يجعل حلقة تصحيح أخطاء Codex الشائعة قصيرة: لاحظ السلوك الخاطئ في
Telegram أو Discord أو قناة أخرى، شغّل `/diagnostics`، وافق مرة واحدة، شارك
التقرير مع الدعم، ثم شغّل أمر `codex resume <thread-id>` المطبوع محليًا إذا
أردت فحص سلسلة Codex الأصلية بنفسك. راجع
[حاضنة Codex](/ar/plugins/codex-harness#inspect-a-codex-thread-from-the-cli) لمعرفة
سير عمل الفحص هذا.

## ما يحتويه التصدير

يتضمن ملف zip:

- `summary.md`: نظرة عامة قابلة للقراءة من البشر للدعم.
- `diagnostics.json`: ملخص قابل للقراءة آليًا للإعدادات، والسجلات، والحالة، والصحة،
  وبيانات الاستقرار.
- `manifest.json`: بيانات تعريف التصدير وقائمة الملفات.
- شكل الإعدادات المنقّى وتفاصيل الإعدادات غير السرية.
- ملخصات السجلات المنقّاة وأسطر السجلات الحديثة المنقحة.
- لقطات حالة وصحة Gateway بأفضل جهد.
- `stability/latest.json`: أحدث حزمة استقرار محفوظة، عند توفرها.

يكون التصدير مفيدًا حتى عندما يكون Gateway غير سليم. إذا لم يتمكن Gateway من
الإجابة عن طلبات الحالة أو الصحة، فستظل السجلات المحلية، وشكل الإعدادات،
وأحدث حزمة استقرار تُجمع عند توفرها.

## نموذج الخصوصية

صُممت التشخيصات لتكون قابلة للمشاركة. يحتفظ التصدير ببيانات تشغيلية
تساعد في تصحيح الأخطاء، مثل:

- أسماء الأنظمة الفرعية، ومعرفات Plugin، ومعرفات المزوّدين، ومعرفات القنوات، والأوضاع المضبوطة
- رموز الحالة، والمدد، وعدّادات البايتات، وحالة الطابور، وقراءات الذاكرة
- بيانات تعريف السجلات المنقّاة والرسائل التشغيلية المنقحة
- شكل الإعدادات وإعدادات الميزات غير السرية

يحذف التصدير أو ينقّح:

- نصوص المحادثات، والمطالبات، والتعليمات، وأجسام Webhook، ومخرجات الأدوات
- بيانات الاعتماد، ومفاتيح API، والرموز، وملفات تعريف الارتباط، والقيم السرية
- أجسام الطلبات أو الاستجابات الخام
- معرفات الحسابات، ومعرفات الرسائل، ومعرفات الجلسات الخام، وأسماء المضيفين، وأسماء المستخدمين المحلية

عندما تبدو رسالة سجل كنص مستخدم أو محادثة أو مطالبة أو حمولة أداة، يحتفظ
التصدير فقط بأن رسالة حُذفت وبعدد البايتات.

## مسجّل الاستقرار

يسجل Gateway تدفق استقرار محدودًا وخاليًا من الحمولات افتراضيًا عندما
تكون التشخيصات مفعّلة. وهو مخصص للحقائق التشغيلية، لا للمحتوى.

يسجل Heartbeat التشخيصي نفسه تحذيرات الحيوية عندما يظل Gateway قيد التشغيل
لكن حلقة أحداث Node.js أو CPU تبدو مشبعة. تتضمن أحداث
`diagnostic.liveness.warning` هذه تأخير حلقة الأحداث، واستخدام حلقة الأحداث،
ونسبة أنوية CPU، وأعداد الجلسات النشطة/المنتظرة/المصطفة. وهي لا تعيد
تشغيل Gateway من تلقاء نفسها.

افحص المسجّل المباشر:

```bash
openclaw gateway stability
openclaw gateway stability --type payload.large
openclaw gateway stability --json
```

افحص أحدث حزمة استقرار محفوظة بعد خروج فادح، أو انتهاء مهلة إيقاف التشغيل،
أو فشل بدء التشغيل بعد إعادة التشغيل:

```bash
openclaw gateway stability --bundle latest
```

أنشئ ملف zip تشخيصيًا من أحدث حزمة محفوظة:

```bash
openclaw gateway stability --bundle latest --export
```

توجد الحزم المحفوظة ضمن `~/.openclaw/logs/stability/` عند وجود أحداث.

## خيارات مفيدة

```bash
openclaw gateway diagnostics export \
  --output openclaw-diagnostics.zip \
  --log-lines 5000 \
  --log-bytes 1000000
```

- `--output <path>`: اكتب إلى مسار zip محدد.
- `--log-lines <count>`: الحد الأقصى لأسطر السجلات المنقّاة المراد تضمينها.
- `--log-bytes <bytes>`: الحد الأقصى لبايتات السجلات المراد فحصها.
- `--url <url>`: عنوان URL لـ WebSocket في Gateway للقطات الحالة والصحة.
- `--token <token>`: رمز Gateway للقطات الحالة والصحة.
- `--password <password>`: كلمة مرور Gateway للقطات الحالة والصحة.
- `--timeout <ms>`: مهلة لقطات الحالة والصحة.
- `--no-stability-bundle`: تخطَّ البحث عن حزمة الاستقرار المحفوظة.
- `--json`: اطبع بيانات تعريف التصدير القابلة للقراءة آليًا.

## تعطيل التشخيصات

التشخيصات مفعّلة افتراضيًا. لتعطيل مسجّل الاستقرار وجمع أحداث التشخيص:

```json5
{
  diagnostics: {
    enabled: false,
  },
}
```

يقلل تعطيل التشخيصات تفاصيل تقارير الأخطاء. ولا يؤثر في تسجيل Gateway العادي.
