# التسجيل

للحصول على نظرة عامة موجهة للمستخدم (CLI + واجهة التحكم + الإعدادات)، راجع [/logging](/ar/logging).

لدى OpenClaw سطحا تسجيل:

- **إخراج وحدة التحكم** (ما تراه في الطرفية / واجهة التصحيح).
- **سجلات الملفات** (أسطر JSON) التي يكتبها مسجل Gateway.

عند بدء التشغيل، يسجل Gateway نموذج الوكيل الافتراضي المحلول مع
افتراضات النمط التي تؤثر في الجلسات الجديدة، على سبيل المثال:

```text
agent model: openai-codex/gpt-5.5 (thinking=medium, fast=on)
```

تأتي `thinking` من الوكيل الافتراضي، أو معاملات النموذج، أو الافتراضي العام للوكيل؛
وعندما لا تكون معينة، يعرض ملخص بدء التشغيل `medium`. تأتي `fast` من
الوكيل الافتراضي أو معاملات `fastMode` للنموذج.

## المسجل المعتمد على الملفات

- يوجد ملف السجل المتناوب الافتراضي ضمن `/tmp/openclaw/` (ملف واحد لكل يوم): `openclaw-YYYY-MM-DD.log`
  - يستخدم التاريخ المنطقة الزمنية المحلية لمضيف Gateway.
- تدور ملفات السجل النشطة عند `logging.maxFileBytes` (الافتراضي: 100 MB)، مع الاحتفاظ
  بما يصل إلى خمسة أرشيفات مرقمة ومواصلة الكتابة إلى ملف نشط جديد.
- يمكن تكوين مسار ملف السجل ومستواه عبر `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

تنسيق الملف هو كائن JSON واحد في كل سطر.

يتابع تبويب السجلات في واجهة التحكم هذا الملف عبر Gateway (`logs.tail`).
يمكن لـ CLI فعل الأمر نفسه:

```bash
openclaw logs --follow
```

**الإسهاب مقابل مستويات السجل**

- **سجلات الملفات** يتحكم بها `logging.level` حصريا.
- يؤثر `--verbose` فقط في **إسهاب وحدة التحكم** (ونمط سجل WS)؛ ولا يقوم **برفع**
  مستوى سجل الملف.
- لالتقاط التفاصيل الخاصة بالإسهاب فقط في سجلات الملفات، اضبط `logging.level` على `debug` أو
  `trace`.
- يتضمن تسجيل التتبع أيضا ملخصات توقيت تشخيصية لمسارات ساخنة محددة،
  مثل تحضير مصنع أدوات Plugin. راجع
  [/tools/plugin#slow-plugin-tool-setup](/ar/tools/plugin#slow-plugin-tool-setup).

## التقاط وحدة التحكم

يلتقط CLI استدعاءات `console.log/info/warn/error/debug/trace` ويكتبها إلى سجلات الملفات،
مع الاستمرار في طباعتها إلى stdout/stderr.

يمكنك ضبط إسهاب وحدة التحكم بشكل مستقل عبر:

- `logging.consoleLevel` (الافتراضي `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## التنقيح

يمكن لـ OpenClaw إخفاء الرموز الحساسة قبل أن يغادر إخراج السجل أو النص المنسوخ
العملية. تطبق سياسة تنقيح التسجيل هذه على مصارف نص وحدة التحكم، وسجل الملف، وسجل OTLP
والنص المنسوخ للجلسة، لذلك يتم إخفاء قيم الأسرار المطابقة
قبل كتابة أسطر JSONL أو الرسائل إلى القرص.

- `logging.redactSensitive`: `off` | `tools` (الافتراضي: `tools`)
- `logging.redactPatterns`: مصفوفة من سلاسل regex (تتجاوز الافتراضيات)
  - استخدم سلاسل regex خام (تلقائيا `gi`)، أو `/pattern/flags` إذا كنت تحتاج إلى أعلام مخصصة.
  - يتم إخفاء المطابقات مع إبقاء أول 6 + آخر 4 أحرف (الطول >= 18)، وإلا `***`.
  - تغطي الافتراضيات تعيينات المفاتيح الشائعة، وأعلام CLI، وحقول JSON، وترويسات bearer، وكتل PEM، وبادئات الرموز الشائعة، وأسماء حقول بيانات اعتماد الدفع مثل رقم البطاقة، وCVC/CVV، ورمز الدفع المشترك، وبيان اعتماد الدفع.

هناك حدود أمان تنقح دائما بغض النظر عن `logging.redactSensitive`.
يشمل ذلك أحداث استدعاء الأدوات في واجهة التحكم، ومخرجات أداة `sessions_history`،
وصادرات دعم التشخيصات، وملاحظات أخطاء المزود، وعرض أمر موافقة exec،
وسجلات بروتوكول WebSocket في Gateway. قد تظل هذه الأسطح تستخدم
`logging.redactPatterns` كنماذج إضافية، لكن `redactSensitive: "off"`
لا يجعلها تصدر أسرارا خاما.

## سجلات WebSocket في Gateway

يطبع Gateway سجلات بروتوكول WebSocket في نمطين:

- **النمط العادي (دون `--verbose`)**: تتم طباعة نتائج RPC "المهمة" فقط:
  - الأخطاء (`ok=false`)
  - الاستدعاءات البطيئة (العتبة الافتراضية: `>= 50ms`)
  - أخطاء التحليل
- **النمط المسهب (`--verbose`)**: يطبع كل حركة طلبات/استجابات WS.

### نمط سجل WS

يدعم `openclaw gateway` مفتاح نمط لكل Gateway:

- `--ws-log auto` (الافتراضي): النمط العادي محسن؛ يستخدم النمط المسهب إخراجا مضغوطا
- `--ws-log compact`: إخراج مضغوط (طلب/استجابة مقترنان) عند الإسهاب
- `--ws-log full`: إخراج كامل لكل إطار عند الإسهاب
- `--compact`: اسم بديل لـ `--ws-log compact`

أمثلة:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## تنسيق وحدة التحكم (تسجيل الأنظمة الفرعية)

منسق وحدة التحكم **واع بحالة TTY** ويطبع أسطرا متسقة ذات بادئات.
تحافظ مسجلات الأنظمة الفرعية على تجميع الإخراج وقابليته للمسح السريع.

السلوك:

- **بادئات الأنظمة الفرعية** في كل سطر (مثل `[gateway]`، `[canvas]`، `[tailscale]`)
- **ألوان الأنظمة الفرعية** (ثابتة لكل نظام فرعي) إضافة إلى تلوين المستوى
- **اللون عندما يكون الإخراج TTY أو تبدو البيئة كطرفية غنية** (`TERM`/`COLORTERM`/`TERM_PROGRAM`)، مع احترام `NO_COLOR`
- **بادئات أنظمة فرعية مختصرة**: تسقط `gateway/` + `channels/` البادئتين، وتبقي آخر مقطعين (مثل `whatsapp/outbound`)
- **مسجلات فرعية حسب النظام الفرعي** (بادئة تلقائية + حقل منظم `{ subsystem }`)
- **`logRaw()`** لإخراج QR/UX (بلا بادئة، بلا تنسيق)
- **أنماط وحدة التحكم** (مثل `pretty | compact | json`)
- **مستوى سجل وحدة التحكم** منفصل عن مستوى سجل الملف (يحتفظ الملف بالتفاصيل الكاملة عند ضبط `logging.level` على `debug`/`trace`)
- **نصوص رسائل WhatsApp** تسجل عند `debug` (استخدم `--verbose` لرؤيتها)

يحافظ هذا على استقرار سجلات الملفات الحالية مع جعل الإخراج التفاعلي قابلا للمسح السريع.
