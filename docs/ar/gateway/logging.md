# التسجيل

للحصول على نظرة عامة موجهة للمستخدم (CLI + Control UI + الإعداد)، راجع [/logging](/ar/logging).

يحتوي OpenClaw على “سطحين” للتسجيل:

- **مخرجات وحدة التحكم** (ما تراه في الطرفية / Debug UI).
- **سجلات الملفات** (أسطر JSON) التي يكتبها مسجل gateway.

## المسجل المعتمد على الملفات

- يوجد ملف السجل الدوّار الافتراضي تحت `/tmp/openclaw/` (ملف واحد لكل يوم): `openclaw-YYYY-MM-DD.log`
  - يستخدم التاريخ المنطقة الزمنية المحلية لمضيف gateway.
- يمكن إعداد مسار ملف السجل والمستوى عبر `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

تنسيق الملف هو كائن JSON واحد في كل سطر.

يقوم تبويب Logs في Control UI بمتابعة هذا الملف عبر gateway (`logs.tail`).
ويمكن لـ CLI فعل الشيء نفسه:

```bash
openclaw logs --follow
```

**Verbose مقابل مستويات السجل**

- يتم التحكم في **سجلات الملفات** حصريًا عبر `logging.level`.
- يؤثر `--verbose` فقط في **تفصيل وحدة التحكم** (ونمط سجل WS)؛ وهو **لا**
  يرفع مستوى سجل الملف.
- لالتقاط التفاصيل الخاصة بـ verbose فقط في سجلات الملفات، اضبط `logging.level` على `debug` أو
  `trace`.

## التقاط وحدة التحكم

يلتقط CLI الاستدعاءات `console.log/info/warn/error/debug/trace` ويكتبها إلى سجلات الملفات،
مع الاستمرار في طباعتها إلى stdout/stderr.

يمكنك ضبط مستوى تفصيل وحدة التحكم بشكل مستقل عبر:

- `logging.consoleLevel` (الافتراضي `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## حجب ملخصات الأدوات

يمكن لملخصات الأدوات المفصلة (مثل `🛠️ Exec: ...`) أن تحجب الرموز الحساسة قبل وصولها إلى
دفق وحدة التحكم. وهذا خاص **بالأدوات فقط** ولا يغيّر سجلات الملفات.

- `logging.redactSensitive`: `off` | `tools` (الافتراضي: `tools`)
- `logging.redactPatterns`: مصفوفة من سلاسل regex (تتجاوز القيم الافتراضية)
  - استخدم سلاسل regex خام (بشكل تلقائي `gi`) أو `/pattern/flags` إذا كنت تحتاج إلى أعلام مخصصة.
  - تُحجب المطابقات مع الإبقاء على أول 6 + آخر 4 محارف (إذا كان الطول >= 18)، وإلا تُستبدل بـ `***`.
  - تغطي القيم الافتراضية تعيينات المفاتيح الشائعة، وأعلام CLI، وحقول JSON، ورؤوس bearer، وكتل PEM، وبادئات الرموز الشائعة.

## سجلات Gateway WebSocket

تطبع gateway سجلات بروتوكول WebSocket في وضعين:

- **الوضع العادي (من دون `--verbose`)**: يتم طباعة نتائج RPC “المثيرة للاهتمام” فقط:
  - الأخطاء (`ok=false`)
  - الاستدعاءات البطيئة (الحد الافتراضي: `>= 50ms`)
  - أخطاء التحليل
- **الوضع المفصل (`--verbose`)**: يطبع كل حركة طلب/استجابة WS.

### نمط سجل WS

يدعم `openclaw gateway` مفتاح نمط لكل gateway:

- `--ws-log auto` (الافتراضي): يتم تحسين الوضع العادي؛ ويستخدم الوضع المفصل خرجًا مضغوطًا
- `--ws-log compact`: خرج مضغوط (طلب/استجابة مقترنان) عند استخدام verbose
- `--ws-log full`: خرج كامل لكل إطار عند استخدام verbose
- `--compact`: اسم بديل لـ `--ws-log compact`

أمثلة:

```bash
# محسّن (أخطاء/بطء فقط)
openclaw gateway

# عرض كل حركة WS (مقترنة)
openclaw gateway --verbose --ws-log compact

# عرض كل حركة WS (بيانات وصفية كاملة)
openclaw gateway --verbose --ws-log full
```

## تنسيق وحدة التحكم (تسجيل الأنظمة الفرعية)

منسّق وحدة التحكم **مدرك لـ TTY** ويطبع أسطرًا متسقة ذات بادئات.
تحافظ مسجلات الأنظمة الفرعية على تجميع الخرج وإمكانية مسحه بصريًا.

السلوك:

- **بادئات الأنظمة الفرعية** في كل سطر (مثل `[gateway]` و`[canvas]` و`[tailscale]`)
- **ألوان الأنظمة الفرعية** (ثابتة لكل نظام فرعي) بالإضافة إلى تلوين المستوى
- **لون عندما يكون الخرج TTY أو عندما تبدو البيئة طرفية غنية** (`TERM`/`COLORTERM`/`TERM_PROGRAM`)، مع احترام `NO_COLOR`
- **اختصار بادئات الأنظمة الفرعية**: إسقاط `gateway/` و`channels/` البادئتين، مع الاحتفاظ بآخر مقطعين (مثل `whatsapp/outbound`)
- **مسجلات فرعية حسب النظام الفرعي** (بادئة تلقائية + الحقل المنظم `{ subsystem }`)
- **`logRaw()`** لخرج QR/UX (من دون بادئة، ومن دون تنسيق)
- **أنماط وحدة التحكم** (مثل `pretty | compact | json`)
- **مستوى سجل وحدة التحكم** منفصل عن مستوى سجل الملف (يحافظ الملف على التفاصيل الكاملة عندما يكون `logging.level` مضبوطًا على `debug`/`trace`)
- **يتم تسجيل متون رسائل WhatsApp** عند المستوى `debug` (استخدم `--verbose` لرؤيتها)

يحافظ هذا على استقرار سجلات الملفات الحالية مع جعل الخرج التفاعلي أسهل في المسح.
