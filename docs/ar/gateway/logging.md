# التسجيل

للحصول على نظرة عامة موجهة للمستخدم (CLI + واجهة التحكم + الإعدادات)، راجع [/logging](/ar/logging).

لدى OpenClaw سطحا تسجيل:

- **مخرجات وحدة التحكم** (ما تراه في الطرفية / واجهة التصحيح).
- **سجلات الملفات** (أسطر JSON) التي يكتبها مسجّل Gateway.

عند بدء التشغيل، يسجّل Gateway نموذج الوكيل الافتراضي المحسوم مع
افتراضات الوضع التي تؤثر في الجلسات الجديدة، على سبيل المثال:

```text
agent model: openai-codex/gpt-5.5 (thinking=medium, fast=on)
```

تأتي `thinking` من الوكيل الافتراضي أو معاملات النموذج أو الافتراضي العام للوكيل؛
وعندما لا تكون مضبوطة، يعرض ملخص بدء التشغيل `medium`. تأتي `fast` من
الوكيل الافتراضي أو معاملات `fastMode` للنموذج.

## المسجّل المعتمد على الملفات

- يوجد ملف السجل الدوّار الافتراضي تحت `/tmp/openclaw/` (ملف واحد لكل يوم): `openclaw-YYYY-MM-DD.log`
  - يستخدم التاريخ المنطقة الزمنية المحلية لمضيف Gateway.
- تدور ملفات السجل النشطة عند `logging.maxFileBytes` (الافتراضي: 100 MB)، مع الاحتفاظ
  بما يصل إلى خمسة أرشيفات مرقمة والاستمرار في الكتابة إلى ملف نشط جديد.
- يمكن ضبط مسار ملف السجل ومستواه عبر `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

تنسيق الملف هو كائن JSON واحد في كل سطر.

يتابع تبويب السجلات في واجهة التحكم هذا الملف عبر Gateway (`logs.tail`).
يمكن لـ CLI فعل الشيء نفسه:

```bash
openclaw logs --follow
```

**الإسهاب مقابل مستويات السجل**

- **سجلات الملفات** يتحكم بها `logging.level` حصريا.
- يؤثر `--verbose` فقط في **إسهاب وحدة التحكم** (ونمط سجل WS)؛ ولا يقوم **برفع**
  مستوى سجل الملف.
- لالتقاط التفاصيل الخاصة بالإسهاب فقط في سجلات الملفات، اضبط `logging.level` إلى `debug` أو
  `trace`.
- يتضمن تسجيل Trace أيضا ملخصات توقيت تشخيصية لمسارات ساخنة مختارة،
  مثل تجهيز مصنع أدوات Plugin. راجع
  [/tools/plugin#slow-plugin-tool-setup](/ar/tools/plugin#slow-plugin-tool-setup).

## التقاط وحدة التحكم

يلتقط CLI ‏`console.log/info/warn/error/debug/trace` ويكتبها إلى سجلات الملفات،
مع الاستمرار في الطباعة إلى stdout/stderr.

يمكنك ضبط إسهاب وحدة التحكم بشكل مستقل عبر:

- `logging.consoleLevel` (الافتراضي `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## التنقيح

يمكن لـ OpenClaw إخفاء الرموز الحساسة قبل أن تخرج مخرجات السجل أو النصوص المكتوبة
من العملية. تُطبّق سياسة تنقيح التسجيل هذه على مصارف نصوص وحدة التحكم وسجل الملفات وسجلات OTLP
ونصوص محاضر الجلسات، بحيث تُخفى قيم الأسرار المطابقة
قبل كتابة أسطر JSONL أو الرسائل إلى القرص.

- `logging.redactSensitive`: `off` | `tools` (الافتراضي: `tools`)
- `logging.redactPatterns`: مصفوفة من سلاسل regex (تتجاوز الافتراضات)
  - استخدم سلاسل regex خاما (تلقائيا `gi`)، أو `/pattern/flags` إذا كنت تحتاج إلى أعلام مخصصة.
  - تُخفى المطابقات مع إبقاء أول 6 + آخر 4 أحرف (عند الطول >= 18)، وإلا `***`.
  - تغطي الافتراضات تعيينات المفاتيح الشائعة، وأعلام CLI، وحقول JSON، وترويسات bearer، وكتل PEM، وبادئات الرموز الشائعة، وأسماء حقول بيانات اعتماد الدفع مثل رقم البطاقة، وCVC/CVV، ورمز الدفع المشترك، وبيان اعتماد الدفع.

تقوم بعض حدود الأمان بالتنقيح دائما بغض النظر عن `logging.redactSensitive`.
يشمل ذلك أحداث استدعاء الأدوات في واجهة التحكم، ومخرجات أداة `sessions_history`،
وصادرات دعم التشخيص، وملاحظات أخطاء المزوّدين، وعرض أمر موافقة exec،
وسجلات بروتوكول WebSocket في Gateway. قد تظل هذه الأسطح تستخدم
`logging.redactPatterns` كنمط إضافي، لكن `redactSensitive: "off"`
لا يجعلها تصدر أسرارا خاما.

## سجلات WebSocket في Gateway

يطبع Gateway سجلات بروتوكول WebSocket في وضعين:

- **الوضع العادي (بدون `--verbose`)**: تُطبع فقط نتائج RPC “المهمة”:
  - أخطاء (`ok=false`)
  - استدعاءات بطيئة (الحد الافتراضي: `>= 50ms`)
  - أخطاء التحليل
- **وضع الإسهاب (`--verbose`)**: يطبع كل حركة طلبات/استجابات WS.

### نمط سجل WS

يدعم `openclaw gateway` مفتاح نمط لكل Gateway:

- `--ws-log auto` (الافتراضي): الوضع العادي محسّن؛ يستخدم وضع الإسهاب مخرجات مضغوطة
- `--ws-log compact`: مخرجات مضغوطة (طلب/استجابة مقترنان) عند الإسهاب
- `--ws-log full`: مخرجات كاملة لكل إطار عند الإسهاب
- `--compact`: اسم مستعار لـ `--ws-log compact`

أمثلة:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## تنسيق وحدة التحكم (تسجيل الأنظمة الفرعية)

منسّق وحدة التحكم **واعٍ بـ TTY** ويطبع أسطرا متسقة مسبوقة ببادئات.
تحافظ مسجلات الأنظمة الفرعية على المخرجات مجمعة وسهلة المسح.

السلوك:

- **بادئات الأنظمة الفرعية** على كل سطر (مثل `[gateway]`، و`[canvas]`، و`[tailscale]`)
- **ألوان الأنظمة الفرعية** (ثابتة لكل نظام فرعي) إضافة إلى تلوين المستوى
- **الألوان عندما تكون المخرجات TTY أو عندما تبدو البيئة كطرفية غنية** (`TERM`/`COLORTERM`/`TERM_PROGRAM`)، مع احترام `NO_COLOR`
- **بادئات أنظمة فرعية مختصرة**: تحذف `gateway/` + `channels/` البادئتين، وتحتفظ بآخر مقطعين (مثل `whatsapp/outbound`)
- **مسجلات فرعية حسب النظام الفرعي** (بادئة تلقائية + حقل منظم `{ subsystem }`)
- **`logRaw()`** لمخرجات QR/UX (بدون بادئة، وبدون تنسيق)
- **أنماط وحدة التحكم** (مثل `pretty | compact | json`)
- **مستوى سجل وحدة التحكم** منفصل عن مستوى سجل الملف (يحتفظ الملف بالتفاصيل الكاملة عندما يكون `logging.level` مضبوطا على `debug`/`trace`)
- **نصوص رسائل WhatsApp** تُسجّل عند `debug` (استخدم `--verbose` لرؤيتها)

يحافظ هذا على استقرار سجلات الملفات الحالية مع جعل المخرجات التفاعلية سهلة المسح.
