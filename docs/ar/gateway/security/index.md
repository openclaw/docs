## ابدأ بالنطاق: نموذج أمان المساعد الشخصي

يفترض إرشاد الأمان في OpenClaw نشرًا على نمط **المساعد الشخصي**: حد مشغّل موثوق واحد، مع احتمال وجود عدة وكلاء.

- وضعية الأمان المدعومة: مستخدم واحد/حد ثقة واحد لكل Gateway (ويُفضَّل مستخدم OS/مضيف/VPS واحد لكل حد).
- ليس حدًا أمنيًا مدعومًا: Gateway/وكيل مشترك واحد يستخدمه مستخدمون غير موثوقين أو خصوم بعضهم لبعض.
- إذا كانت هناك حاجة إلى عزل مستخدمين خصوم، فافصل حسب حد الثقة (Gateway + بيانات اعتماد منفصلة، ويفضل أيضًا مستخدمو OS/مضيفون منفصلون).
- إذا كان بإمكان عدة مستخدمين غير موثوقين مراسلة وكيل واحد مفعّل الأدوات، فاعتبرهم جميعًا يشاركون سلطة الأدوات المفوّضة نفسها لذلك الوكيل.

تشرح هذه الصفحة التحصين **ضمن هذا النموذج**. وهي لا تدّعي توفير عزل معادٍ متعدد المستأجرين على Gateway مشترك واحد.

## فحص سريع: `openclaw security audit`

راجع أيضًا: [التحقق الشكلي (نماذج الأمان)](/ar/security/formal-verification)

شغّل هذا بانتظام (خصوصًا بعد تغيير الإعدادات أو تعريض أسطح الشبكة):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

يبقى `security audit --fix` ضيق النطاق عن قصد: فهو يحوّل سياسات
المجموعات المفتوحة الشائعة إلى allowlists، ويستعيد `logging.redactSensitive: "tools"`، ويشدّد
أذونات state/config/include-file، ويستخدم إعادة تعيين Windows ACL بدل
`chmod` الخاص بـ POSIX عند التشغيل على Windows.

وهو يكتشف الأخطاء الشائعة الخطيرة (تعريض مصادقة Gateway، وتعريض التحكم في المتصفح، وallowlists المرفوعة، وأذونات نظام الملفات، وموافقات exec المتساهلة، وتعريض الأدوات عبر القنوات المفتوحة).

OpenClaw هو منتج وتجربة في آنٍ واحد: فأنت توصل سلوك models المتقدمة بأسطح مراسلة حقيقية وأدوات حقيقية. **لا توجد تهيئة “آمنة تمامًا”.** الهدف هو أن تكون متعمدًا بشأن:

- من يمكنه التحدث إلى bot الخاص بك
- أين يُسمح لـ bot بالتصرف
- ما الذي يمكن لـ bot لمسه

ابدأ بأقل وصول لا يزال يفي بالغرض، ثم وسّعه تدريجيًا مع اكتسابك الثقة.

### النشر وثقة المضيف

يفترض OpenClaw أن المضيف وحد الإعدادات موثوقان:

- إذا كان بإمكان شخص ما تعديل حالة/إعدادات مضيف Gateway ‏(`~/.openclaw`، بما في ذلك `openclaw.json`)، فاعتبره مشغّلًا موثوقًا.
- تشغيل Gateway واحد لعدة مشغّلين غير موثوقين/خصوم بعضهم لبعض **ليس إعدادًا موصى به**.
- بالنسبة إلى الفرق ذات الثقة المختلطة، افصل حدود الثقة عبر Gateways منفصلة (أو على الأقل مستخدمي OS/مضيفين منفصلين).
- الوضع الافتراضي الموصى به: مستخدم واحد لكل جهاز/مضيف (أو VPS)، وgateway واحد لذلك المستخدم، ووكيل واحد أو أكثر داخل ذلك gateway.
- داخل مثيل Gateway واحد، يكون وصول المشغّل الموثق دورًا موثوقًا في مستوى التحكم، وليس دور مستأجر لكل مستخدم.
- معرّفات الجلسة (`sessionKey`، ومعرّفات الجلسة، وlabels) هي محددات توجيه، وليست رموز تفويض.
- إذا كان بإمكان عدة أشخاص مراسلة وكيل واحد مفعّل الأدوات، فيمكن لكل واحد منهم توجيه مجموعة الأذونات نفسها. ويساعد عزل الجلسات/الذاكرة لكل مستخدم في الخصوصية، لكنه لا يحوّل الوكيل المشترك إلى تفويض مضيف منفصل لكل مستخدم.

### مساحة Slack مشتركة: الخطر الحقيقي

إذا كان "كل من في Slack يمكنه مراسلة bot"، فإن الخطر الأساسي هو سلطة الأدوات المفوّضة:

- يمكن لأي مرسل مسموح له أن يحفّز استدعاءات الأدوات (`exec`، والمتصفح، وأدوات الشبكة/الملفات) ضمن سياسة الوكيل؛
- يمكن أن يؤدي حقن prompt/المحتوى من مرسل واحد إلى أفعال تؤثر في الحالة المشتركة أو الأجهزة أو المخرجات؛
- إذا كان لدى وكيل مشترك واحد بيانات اعتماد/ملفات حساسة، فقد يتمكن أي مرسل مسموح له من دفع عملية استخراج بيانات عبر استخدام الأدوات.

استخدم وكلاء/Gateways منفصلة مع أقل قدر ممكن من الأدوات لسير عمل الفريق؛ واحتفظ بوكلاء البيانات الشخصية على نحو خاص.

### وكيل مشترك على مستوى الشركة: نمط مقبول

يكون هذا مقبولًا عندما يكون كل من يستخدم ذلك الوكيل ضمن حد الثقة نفسه (مثل فريق واحد في شركة) ويكون نطاق الوكيل مهنيًا بشكل صارم.

- شغّله على جهاز/VM/container مخصص؛
- استخدم مستخدم OS مخصصًا + متصفحًا/ملفًا شخصيًا/حسابات مخصصة لذلك runtime؛
- لا تسجل دخول هذا runtime إلى حسابات Apple/Google الشخصية أو إلى ملفات مدير كلمات المرور/المتصفح الشخصية.

إذا خلطت الهويات الشخصية وهويات الشركة على runtime نفسه، فأنت تُسقط هذا الفصل وتزيد خطر تعريض البيانات الشخصية.

## مفهوم الثقة في Gateway وNode

تعامل مع Gateway وNode بوصفهما نطاق ثقة واحدًا للمشغّل، مع أدوار مختلفة:

- **Gateway** هو مستوى التحكم وسطح السياسة (`gateway.auth`، وسياسة الأدوات، والتوجيه).
- **Node** هو سطح التنفيذ البعيد المقترن بذلك Gateway (الأوامر، وإجراءات الجهاز، والقدرات المحلية للمضيف).
- يكون المتصل الموثق لدى Gateway موثوقًا ضمن نطاق Gateway. وبعد الاقتران، تُعد إجراءات node إجراءات مشغّل موثوق على ذلك node.
- يمكن لعملاء الواجهة الخلفية المباشرين عبر loopback والمصادَق عليهم باستخدام
  token/password المشترك للـ gateway إجراء RPCs داخلية في مستوى التحكم دون تقديم
  هوية جهاز مستخدم. هذا ليس تجاوزًا للاقتران عن بُعد أو عبر المتصفح: فما تزال
  عملاء الشبكة، وعملاء node، وعملاء device-token، وهويات الأجهزة الصريحة
  تمر عبر الاقتران وفرض ترقية النطاق.
- `sessionKey` هو اختيار للتوجيه/السياق، وليس مصادقة لكل مستخدم.
- تمثل موافقات exec ‏(allowlist + ask) حواجز لإظهار نية المشغّل، وليست عزلًا معاديًا متعدد المستأجرين.
- الوضع الافتراضي في منتج OpenClaw لإعدادات المشغّل الموثوق الواحد هو السماح بتنفيذ المضيف على `gateway`/`node` دون مطالبات موافقة (`security="full"`، و`ask="off"` ما لم تقم بتشديده). هذا الافتراضي مقصود لتجربة الاستخدام، وليس ثغرة بحد ذاته.
- ترتبط موافقات exec بسياق الطلب الدقيق وبأفضل جهد للمعاملات المحلية المباشرة للملفات؛ وهي لا تمثل دلاليًا كل مسار تحميل في runtime/interpreter. استخدم sandboxing وعزل المضيف للحصول على حدود قوية.

إذا كنت تحتاج إلى عزل مستخدمين خصوم، فافصل حدود الثقة حسب مستخدم OS/المضيف وشغّل Gateways منفصلة.

## مصفوفة حدود الثقة

استخدم هذا كنموذج سريع عند فرز المخاطر:

| الحد أو عنصر التحكم                                      | ما الذي يعنيه                                   | سوء الفهم الشائع                                                              |
| -------------------------------------------------------- | ----------------------------------------------- | ----------------------------------------------------------------------------- |
| `gateway.auth` ‏(token/password/trusted-proxy/device auth) | يثبت هوية المتصلين بواجهات gateway APIs         | "لكي يكون آمنًا يجب أن توجد تواقيع لكل رسالة على كل frame"                   |
| `sessionKey`                                             | مفتاح توجيه لاختيار السياق/الجلسة               | "مفتاح الجلسة هو حد مصادقة للمستخدم"                                          |
| حواجز prompt/المحتوى                                     | تقلل خطر إساءة استخدام model                    | "حقن prompt وحده يثبت تجاوز المصادقة"                                        |
| `canvas.eval` / evaluate في المتصفح                      | قدرة مشغّل مقصودة عند تفعيلها                    | "أي primitive لـ JS eval هي تلقائيًا ثغرة في نموذج الثقة هذا"                |
| `!` shell في TUI المحلية                                 | تنفيذ محلي يُطلقه المشغّل صراحةً                | "أمر shell المحلي المريح هو حقن عن بُعد"                                     |
| اقتران node وأوامر node                                  | تنفيذ عن بُعد على مستوى المشغّل على الأجهزة المقترنة | "يجب معاملة التحكم في الجهاز البعيد كأنه وصول مستخدم غير موثوق افتراضيًا" |
| `gateway.nodes.pairing.autoApproveCidrs`                 | سياسة تسجيل opt-in للأجهزة ضمن شبكة موثوقة      | "allowlist معطلة افتراضيًا هي تلقائيًا ثغرة في الاقتران"                     |

## خط أساس مُحصَّن خلال 60 ثانية

استخدم هذا الخط الأساس أولًا، ثم أعد تفعيل الأدوات بشكل انتقائي لكل وكيل موثوق:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

يبقي هذا Gateway محلية فقط، ويعزل الرسائل الخاصة، ويعطل أدوات مستوى التحكم/وقت التشغيل افتراضيًا.

## قاعدة سريعة لصندوق الوارد المشترك

إذا كان بإمكان أكثر من شخص واحد إرسال رسالة خاصة إلى bot الخاص بك:

- اضبط `session.dmScope: "per-channel-peer"` (أو `"per-account-channel-peer"` للقنوات متعددة الحسابات).
- أبقِ `dmPolicy: "pairing"` أو allowlists صارمة.
- لا تجمع أبدًا بين الرسائل الخاصة المشتركة ووصول واسع إلى الأدوات.
- يعزز هذا أمان صناديق الوارد التعاونية/المشتركة، لكنه غير مصمم كعزل لخصوم يشاركون المستضيف/الإعدادات.

## نموذج ظهور السياق

يفصل OpenClaw بين مفهومين:

- **تفويض التشغيل**: من يمكنه تشغيل الوكيل (`dmPolicy`، و`groupPolicy`، وallowlists، وبوابات الإشارة).
- **ظهور السياق**: ما السياق الإضافي الذي يُحقن في مدخل model (نص الرد، والنص المقتبس، وسجل thread، وبيانات forwarded الوصفية).

تتحكم allowlists في التشغيل وتفويض الأوامر. أما إعداد `contextVisibility` فيتحكم في كيفية تصفية السياق الإضافي (الردود المقتبسة، وجذور thread، والسجل الذي جرى جلبه):

- القيمة `contextVisibility: "all"` (الافتراضية) تُبقي السياق الإضافي كما تم استلامه.
- القيمة `contextVisibility: "allowlist"` تُصفّي السياق الإضافي بحيث يقتصر على المرسلين المسموح بهم وفق فحوص allowlist النشطة.
- القيمة `contextVisibility: "allowlist_quote"` تعمل مثل `allowlist`، لكنها تُبقي مع ذلك ردًا مقتبسًا صريحًا واحدًا.

اضبط `contextVisibility` لكل قناة أو لكل غرفة/محادثة. راجع [دردشات المجموعات](/ar/channels/groups#context-visibility-and-allowlists) لمعرفة تفاصيل الإعداد.

إرشادات فرز استشارية:

- الادعاءات التي تُظهر فقط أن "model يمكنه رؤية نص مقتبس أو تاريخي من مرسلين غير موجودين في allowlist" تُعد نتائج تحصين يمكن معالجتها عبر `contextVisibility`، وليست بحد ذاتها تجاوزات لحدود المصادقة أو sandbox.
- لكي تكون التقارير ذات أثر أمني، لا يزال يجب أن تتضمن تجاوزًا مثبتًا لحدود الثقة (مصادقة، أو سياسة، أو sandbox، أو موافقة، أو حد موثَّق آخر).

## ما الذي يتحقق منه التدقيق (على مستوى عالٍ)

- **الوصول الوارد** (سياسات الرسائل الخاصة، وسياسات المجموعات، وallowlists): هل يمكن للغرباء تشغيل bot؟
- **نطاق انفجار الأدوات** (الأدوات المرتفعة + الغرف المفتوحة): هل يمكن أن يتحول حقن prompt إلى أفعال على shell/الملفات/الشبكة؟
- **انجراف موافقات Exec** ‏(`security=full`، و`autoAllowSkills`، وallowlists الخاصة بالمفسرات دون `strictInlineEval`): هل لا تزال حواجز host-exec تؤدي ما تعتقده؟
  - تمثل `security="full"` تحذيرًا عامًا حول الوضعية، وليست دليلًا على وجود خطأ. وهي الوضع الافتراضي المختار لإعدادات المساعد الشخصي الموثوق؛ شددها فقط عندما يتطلب نموذج التهديد لديك حواجز موافقة أو allowlist.
- **تعرض الشبكة** (ربط Gateway/mصادقتها، وTailscale Serve/Funnel، وtokens ضعيفة/قصيرة للمصادقة).
- **تعرض التحكم في المتصفح** (العُقد البعيدة، ومنافذ relay، ونقاط نهاية CDP البعيدة).
- **نظافة القرص المحلي** (الأذونات، وsymlinks، وinclude الخاصة بالإعدادات، ومسارات “المجلدات المتزامنة”).
- **Plugins** (تحميل Plugins من دون allowlist صريحة).
- **انجراف السياسة/سوء الإعداد** (إعدادات sandbox docker مضبوطة بينما وضع sandbox معطّل؛ أنماط `gateway.nodes.denyCommands` غير الفعالة لأن المطابقة تكون لاسم الأمر الدقيق فقط مثل `system.run` ولا تفحص نص shell؛ إدخالات `gateway.nodes.allowCommands` الخطيرة؛ تجاوز `tools.profile="minimal"` العالمي عبر ملفات تعريف لكل وكيل؛ إمكانية الوصول إلى أدوات مملوكة للـ Plugin تحت سياسة أدوات متساهلة).
- **انجراف توقعات وقت التشغيل** (مثل افتراض أن التنفيذ الضمني لا يزال يعني `sandbox` عندما تكون القيمة الافتراضية لـ `tools.exec.host` الآن هي `auto`، أو ضبط `tools.exec.host="sandbox"` صراحة بينما وضع sandbox معطّل).
- **نظافة Models** (تحذير عندما تبدو Models المهيأة قديمة؛ وليس حظرًا صارمًا).

إذا شغّلت `--deep`، فسيحاول OpenClaw أيضًا إجراء probe حي لـ Gateway بأفضل جهد.

## خريطة تخزين بيانات الاعتماد

استخدم هذا عند تدقيق الوصول أو تقرير ما الذي يجب نسخه احتياطيًا:

- **WhatsApp**: ‏`~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
- **Telegram bot token**: الإعدادات/env أو `channels.telegram.tokenFile` (ملف عادي فقط؛ يتم رفض symlinks)
- **Discord bot token**: الإعدادات/env أو SecretRef ‏(مزودات env/file/exec)
- **Slack tokens**: الإعدادات/env ‏(`channels.slack.*`)
- **allowlists الخاصة بالاقتران**:
  - `~/.openclaw/credentials/<channel>-allowFrom.json` (الحساب الافتراضي)
  - `~/.openclaw/credentials/<channel>-<accountId>-allowFrom.json` (الحسابات غير الافتراضية)
- **ملفات تعريف مصادقة Model**: ‏`~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- **حمولة الأسرار المدعومة بملف (اختياري)**: ‏`~/.openclaw/secrets.json`
- **استيراد OAuth القديم**: ‏`~/.openclaw/credentials/oauth.json`

## قائمة التحقق من التدقيق الأمني

عندما يطبع التدقيق نتائج، تعامل مع هذا وفق ترتيب الأولوية التالي:

1. **أي شيء “مفتوح” + أدوات مفعلة**: أمِّن الرسائل الخاصة/المجموعات أولًا (pairing/allowlists)، ثم شدد سياسة الأدوات/sandboxing.
2. **تعرض الشبكة العامة** (ربط LAN، وFunnel، وغياب المصادقة): أصلحه فورًا.
3. **تعرض التحكم البعيد في المتصفح**: عامله كأنه وصول مشغّل (tailnet فقط، واقرن nodes عمدًا، وتجنب التعرض العام).
4. **الأذونات**: تأكد من أن state/config/credentials/auth ليست قابلة للقراءة من المجموعة/العالم.
5. **Plugins**: حمّل فقط ما تثق به صراحة.
6. **اختيار Model**: فضّل Models حديثة ومحصنة بالتعليمات لأي bot لديه أدوات.

## مسرد التدقيق الأمني

يُفهرس كل اكتشاف تدقيق بواسطة `checkId` منظم (مثل
`gateway.bind_no_auth` أو `tools.exec.security_full_configured`). وتشمل فئات الشدة الحرجة الشائعة:

- `fs.*` — أذونات نظام الملفات لـ state، وconfig، وcredentials، وملفات تعريف auth.
- `gateway.*` — وضع الربط، والمصادقة، وTailscale، وControl UI، وإعداد trusted-proxy.
- `hooks.*`، و`browser.*`، و`sandbox.*`، و`tools.exec.*` — التحصين لكل سطح على حدة.
- `plugins.*`، و`skills.*` — سلسلة توريد plugin/skill ونتائج الفحص.
- `security.exposure.*` — فحوصات شاملة حيث تلتقي سياسة الوصول مع نطاق انفجار الأدوات.

راجع الفهرس الكامل مع مستويات الشدة، ومفاتيح الإصلاح، ودعم الإصلاح التلقائي في
[فحوصات التدقيق الأمني](/ar/gateway/security/audit-checks).

## Control UI عبر HTTP

تحتاج Control UI إلى **سياق آمن** (HTTPS أو localhost) لتوليد هوية الجهاز.
يُعد `gateway.controlUi.allowInsecureAuth` مفتاح توافق محلي:

- على localhost، يسمح بمصادقة Control UI دون هوية جهاز عندما
  يتم تحميل الصفحة عبر HTTP غير آمن.
- لا يتجاوز فحوصات الاقتران.
- لا يرخّي متطلبات هوية الجهاز عن بُعد (غير localhost).

فضّل HTTPS ‏(Tailscale Serve) أو افتح الواجهة على `127.0.0.1`.

لحالات الطوارئ فقط، يقوم `gateway.controlUi.dangerouslyDisableDeviceAuth`
بتعطيل فحوصات هوية الجهاز بالكامل. ويعد هذا خفضًا أمنيًا شديدًا؛
اتركه معطّلًا ما لم تكن تقوم بتصحيح أخطاء نشط ويمكنك التراجع سريعًا.

وبشكل منفصل عن هذه العلامات الخطيرة، يمكن لنجاح `gateway.auth.mode: "trusted-proxy"`
أن يسمح بجلسات **مشغّل** في Control UI من دون هوية جهاز. هذا
سلوك مقصود لوضع المصادقة، وليس اختصارًا من `allowInsecureAuth`، ومع ذلك
فهو لا يمتد إلى جلسات Control UI ذات دور node.

يحذّر `openclaw security audit` عند تفعيل هذا الإعداد.

## إعداد reverse proxy

إذا كنت تشغّل Gateway خلف reverse proxy ‏(nginx، أو Caddy، أو Traefik، وما إلى ذلك)، فاضبط
`gateway.trustedProxies` من أجل التعامل الصحيح مع IP العميل المُمرَّر.

عندما يكتشف Gateway ترويسات proxy من عنوان **غير** موجود في `trustedProxies`، فلن
يعامل الاتصالات على أنها عملاء محليون. وإذا كانت مصادقة gateway معطلة، فسيتم رفض تلك الاتصالات. وهذا يمنع تجاوز المصادقة حيث كان من الممكن أن تبدو الاتصالات المُمرَّرة على أنها صادرة من localhost وتحصل على ثقة تلقائية.

يغذي `gateway.trustedProxies` أيضًا الوضع `gateway.auth.mode: "trusted-proxy"`، لكن
هذا الوضع أكثر صرامة:

- تفشل مصادقة trusted-proxy **بشكل مغلق على proxies ذات المصدر loopback**
- لا يزال بإمكان reverse proxies المحلية على المضيف نفسه عبر loopback استخدام `gateway.trustedProxies` لاكتشاف العملاء المحليين ومعالجة عناوين IP المُمرَّرة
- بالنسبة إلى reverse proxies المحلية على المضيف نفسه عبر loopback، استخدم مصادقة token/password بدلًا من `gateway.auth.mode: "trusted-proxy"`

```yaml
gateway:
  trustedProxies:
    - "10.0.0.1" # عنوان IP الخاص بالـ reverse proxy
  # اختياري. الافتراضي false.
  # فعّله فقط إذا كان proxy الخاص بك لا يستطيع توفير X-Forwarded-For.
  allowRealIpFallback: false
  auth:
    mode: password
    password: ${OPENCLAW_GATEWAY_PASSWORD}
```

عند ضبط `trustedProxies`، يستخدم Gateway الترويسة `X-Forwarded-For` لتحديد IP العميل. ويتم تجاهل `X-Real-IP` افتراضيًا ما لم يتم ضبط `gateway.allowRealIpFallback: true` صراحةً.

لا تجعل ترويسات trusted proxy اقتران أجهزة node موثوقًا تلقائيًا.
إن `gateway.nodes.pairing.autoApproveCidrs` سياسة منفصلة للمشغّل، ومعطلة افتراضيًا.
وحتى عند تفعيلها، تُستبعد مسارات ترويسات trusted-proxy ذات المصدر loopback
من الموافقة التلقائية على node لأن المتصلين المحليين يمكنهم تزوير هذه
الترويسات.

سلوك reverse proxy الجيد (استبدال ترويسات التمرير الواردة):

```nginx
proxy_set_header X-Forwarded-For $remote_addr;
proxy_set_header X-Real-IP $remote_addr;
```

سلوك reverse proxy السيئ (إلحاق/الاحتفاظ بترويسات تمرير غير موثوقة):

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```

## ملاحظات HSTS وorigin

- يعمل Gateway في OpenClaw وفق مبدأ local/loopback أولًا. إذا أنهيت TLS عند reverse proxy، فاضبط HSTS على نطاق HTTPS المواجه للـ proxy هناك.
- إذا كان gateway نفسه ينهي HTTPS، فيمكنك ضبط `gateway.http.securityHeaders.strictTransportSecurity` لإرسال ترويسة HSTS من استجابات OpenClaw.
- توجد إرشادات النشر المفصلة في [Trusted Proxy Auth](/ar/gateway/trusted-proxy-auth#tls-termination-and-hsts).
- بالنسبة إلى عمليات نشر Control UI غير المعتمدة على loopback، يكون `gateway.controlUi.allowedOrigins` مطلوبًا افتراضيًا.
- تمثل `gateway.controlUi.allowedOrigins: ["*"]` سياسة صريحة للسماح بكل browser-origin، وليست افتراضيًا محصنًا. تجنبها خارج الاختبارات المحلية المحكمة.
- لا تزال إخفاقات مصادقة browser-origin على loopback خاضعة لتحديد المعدل حتى عندما
  يكون الإعفاء العام للـ loopback مفعّلًا، لكن مفتاح الحظر يُنطاق لكل
  قيمة `Origin` مطبّعة بدلًا من دلو localhost مشترك واحد.
- يؤدي `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` إلى تفعيل وضع fallback لـ origin المستند إلى ترويسة Host؛ تعامل معه كسياسة خطيرة يختارها المشغّل.
- تعامل مع DNS rebinding وسلوك ترويسة مضيف proxy باعتبارهما من مسائل تحصين النشر؛ واحرص على إبقاء `trustedProxies` محكمة وتجنب تعريض gateway مباشرةً إلى الإنترنت العام.

## سجلات الجلسات المحلية موجودة على القرص

يخزن OpenClaw نصوص الجلسات على القرص ضمن `~/.openclaw/agents/<agentId>/sessions/*.jsonl`.
وهذا مطلوب لاستمرارية الجلسة و(اختياريًا) لفهرسة ذاكرة الجلسة، لكنه يعني أيضًا أن
**أي عملية/مستخدم لديه وصول إلى نظام الملفات يمكنه قراءة تلك السجلات**. تعامل مع الوصول إلى القرص باعتباره
حد الثقة، وشدّد الأذونات على `~/.openclaw` (راجع قسم التدقيق أدناه). إذا كنت تحتاج إلى
عزل أقوى بين الوكلاء، فشغّلهم تحت مستخدمي OS منفصلين أو على مضيفين منفصلين.

## تنفيذ Node ‏(`system.run`)

إذا كان هناك macOS node مقترن، يمكن لـ Gateway استدعاء `system.run` على ذلك node. وهذا **تنفيذ تعليمات برمجية عن بُعد** على جهاز Mac:

- يتطلب اقتران node ‏(موافقة + token).
- لا يُعد اقتران Gateway مع node سطح موافقة لكل أمر. بل يثبت هوية/ثقة node ويصدر token.
- يطبق Gateway سياسة عامة خشنة لأوامر node عبر `gateway.nodes.allowCommands` / `denyCommands`.
- يتم التحكم فيها على جهاز Mac عبر **Settings → Exec approvals** ‏(security + ask + allowlist).
- إن سياسة `system.run` لكل node هي ملف موافقات exec الخاص بالـ node نفسه (`exec.approvals.node.*`)، والذي قد يكون أكثر أو أقل تشددًا من سياسة معرّف الأوامر العامة في gateway.
- إذا كان node يعمل بالقيمتين `security="full"` و`ask="off"`، فهو يتبع نموذج المشغّل الموثوق الافتراضي. تعامل مع هذا بوصفه سلوكًا متوقعًا ما لم يكن نشرُك يتطلب صراحةً وضعًا أكثر تشددًا للموافقة أو allowlist.
- يربط وضع الموافقة سياق الطلب الدقيق، وعند الإمكان، معامل script/file محليًا واحدًا وملموسًا. وإذا لم يستطع OpenClaw تحديد ملف محلي مباشر واحد تمامًا لأمر interpreter/runtime، فسيتم رفض التنفيذ المعتمد على الموافقة بدلًا من الادعاء بتغطية دلالية كاملة.
- بالنسبة إلى `host=node`، تخزّن التشغيلات المدعومة بالموافقة أيضًا
  `systemRunPlan` مُحضَّرًا وقياسيًا؛ وتعيد عمليات التمرير الموافق عليها لاحقًا
  استخدام تلك الخطة المخزنة، ويرفض تحقق gateway تعديلات المتصل على command/cwd/session context بعد إنشاء طلب الموافقة.
- إذا كنت لا تريد تنفيذًا عن بُعد، فاضبط security على **deny** وأزل اقتران node لذلك الجهاز Mac.

هذا التمييز مهم في الفرز:

- إن node مقترنًا يعيد الاتصال ويعلن قائمة أوامر مختلفة ليس، بحد ذاته، ثغرة إذا كانت السياسة العامة للـ Gateway وموافقات exec المحلية على node لا تزال تفرض حد التنفيذ الفعلي.
- التقارير التي تتعامل مع بيانات اقتران node الوصفية على أنها طبقة موافقة ثانية مخفية لكل أمر غالبًا ما تكون التباسًا في السياسة/تجربة الاستخدام، لا تجاوزًا لحد أمني.

## Skills الديناميكية (المراقِب / العُقد البعيدة)

يمكن لـ OpenClaw تحديث قائمة Skills أثناء الجلسة:

- **مراقب Skills**: يمكن أن تؤدي التغييرات على `SKILL.md` إلى تحديث لقطة Skills في دورة الوكيل التالية.
- **العُقد البعيدة**: قد يؤدي اتصال macOS node إلى جعل Skills الخاصة بـ macOS فقط مؤهلة (استنادًا إلى فحص bin).

تعامل مع مجلدات Skills على أنها **كود موثوق** وقيّد من يمكنه تعديلها.

## نموذج التهديد

يمكن لمساعدك AI أن:

- ينفذ أوامر shell اعتباطية
- يقرأ/يكتب الملفات
- يصل إلى خدمات الشبكة
- يرسل رسائل إلى أي شخص (إذا منحته وصولًا إلى WhatsApp)

يمكن للأشخاص الذين يرسلون إليك رسائل أن:

- يحاولوا خداع AI الخاص بك لفعل أشياء سيئة
- يمارسوا هندسة اجتماعية للوصول إلى بياناتك
- يستكشفوا تفاصيل البنية التحتية

## المفهوم الأساسي: التحكم في الوصول قبل الذكاء

معظم الإخفاقات هنا ليست استغلالات معقدة — بل هي ببساطة "شخص ما راسل bot وbot فعل ما طلبه."

موقف OpenClaw:

- **الهوية أولًا:** قرر من يمكنه التحدث إلى bot ‏(اقتران الرسائل الخاصة / allowlists / “open” صريح).
- **النطاق ثانيًا:** قرر أين يُسمح لـ bot بالتصرف (allowlists للمجموعات + بوابات الإشارة، والأدوات، وsandboxing، وأذونات الأجهزة).
- **Model أخيرًا:** افترض أن model يمكن التلاعب به؛ وصمّم النظام بحيث يكون نطاق الضرر محدودًا.

## نموذج تفويض الأوامر

لا يتم احترام أوامر الشرطة المائلة والتوجيهات إلا للـ **مرسلين المصرح لهم**. ويُشتق التفويض من
allowlists/الاقتران الخاصة بالقنوات بالإضافة إلى `commands.useAccessGroups` (راجع [الإعدادات](/ar/gateway/configuration)
و[أوامر الشرطة المائلة](/ar/tools/slash-commands)). وإذا كانت allowlist الخاصة بالقناة فارغة أو تتضمن `"*"`,
فإن الأوامر تكون مفتوحة فعليًا لتلك القناة.

يمثل `/exec` وسيلة راحة داخل الجلسة فقط للمشغلين المصرح لهم. وهو **لا** يكتب في الإعدادات ولا
يغيّر الجلسات الأخرى.

## خطر أدوات مستوى التحكم

تستطيع أداتان مدمجتان إجراء تغييرات دائمة في مستوى التحكم:

- يمكن لأداة `gateway` فحص الإعدادات عبر `config.schema.lookup` / `config.get`، ويمكنها إجراء تغييرات دائمة عبر `config.apply`، و`config.patch`، و`update.run`.
- يمكن لأداة `cron` إنشاء مهام مجدولة تستمر في العمل بعد انتهاء الدردشة/المهمة الأصلية.

لا تزال أداة runtime الخاصة بـ `gateway` والمقصورة على المالك ترفض إعادة كتابة
`tools.exec.ask` أو `tools.exec.security`؛ ويتم تطبيع الأسماء المستعارة القديمة `tools.bash.*`
إلى مسارات exec المحمية نفسها قبل الكتابة.
تُطبَّق تعديلات `gateway config.apply` و`gateway config.patch` المدفوعة من الوكيل
بوضع fail-closed افتراضيًا: فمجموعة ضيقة فقط من مسارات
prompt وmodel وبوابات الإشارة قابلة للتعديل من قبل الوكيل. ولذلك فإن أشجار الإعدادات الحساسة الجديدة تكون محمية
ما لم تتم إضافتها عمدًا إلى allowlist.

بالنسبة إلى أي وكيل/سطح يتعامل مع محتوى غير موثوق، ارفض هذه الأدوات افتراضيًا:

```json5
{
  tools: {
    deny: ["gateway", "cron", "sessions_spawn", "sessions_send"],
  },
}
```

لا تمنع `commands.restart=false` إلا إجراءات إعادة التشغيل. وهي لا تعطل إجراءات إعداد/تحديث `gateway`.

## Plugins

تعمل Plugins **داخل العملية نفسها** مع Gateway. تعامل معها على أنها كود موثوق:

- ثبّت فقط Plugins من مصادر تثق بها.
- فضّل allowlists صريحة عبر `plugins.allow`.
- راجع إعدادات Plugin قبل التفعيل.
- أعد تشغيل Gateway بعد تغييرات Plugin.
- إذا ثبّتّ أو حدّثت Plugins ‏(`openclaw plugins install <package>`، و`openclaw plugins update <id>`)، فتعامل مع ذلك كما لو كنت تشغّل كودًا غير موثوق:
  - يكون مسار التثبيت هو الدليل الخاص بكل Plugin تحت جذر تثبيت الـ Plugin النشط.
  - يشغّل OpenClaw فحصًا مدمجًا للكود الخطير قبل التثبيت/التحديث. وتمنع النتائج `critical` افتراضيًا.
  - يستخدم OpenClaw الأمر `npm pack`، ثم يشغّل أمرًا محليًا بالمشروع هو `npm install --omit=dev --ignore-scripts` داخل ذلك الدليل. ويتم تجاهل إعدادات تثبيت npm العامة الموروثة حتى تبقى التبعيات داخل مسار تثبيت الـ Plugin.
  - فضّل الإصدارات المثبتة الدقيقة (`@scope/pkg@1.2.3`) وافحص الكود المفكوك على القرص قبل التفعيل.
  - إن `--dangerously-force-unsafe-install` مخصص فقط لحالات الطوارئ في حال النتائج الإيجابية الكاذبة للفحص المدمج أثناء تدفقات تثبيت/تحديث Plugin. وهو لا يتجاوز حظر سياسة خطاف `before_install` في الـ Plugin ولا يتجاوز إخفاقات الفحص.
  - تتبع عمليات تثبيت تبعيات Skills المدعومة من Gateway الفصل نفسه بين dangerous/suspicious: تمنع النتائج `critical` المدمجة ما لم يضبط المتصل صراحةً `dangerouslyForceUnsafeInstall`، بينما تظل النتائج المشبوهة مجرد تحذيرات. ويبقى `openclaw skills install` هو تدفق تنزيل/تثبيت Skills المنفصل من ClawHub.

التفاصيل: [Plugins](/ar/tools/plugin)

## نموذج الوصول إلى الرسائل الخاصة: pairing، وallowlist، وopen، وdisabled

تدعم جميع القنوات الحالية القادرة على الرسائل الخاصة سياسة رسائل خاصة (`dmPolicy` أو `*.dm.policy`) تتحكم في الرسائل الخاصة الواردة **قبل** معالجة الرسالة:

- `pairing` (الافتراضي): يتلقى المرسلون غير المعروفين رمز اقتران قصيرًا ويتجاهل bot رسالتهم حتى تتم الموافقة. تنتهي صلاحية الرموز بعد ساعة واحدة؛ ولن تعيد الرسائل الخاصة المتكررة إرسال رمز حتى يتم إنشاء طلب جديد. ويُحدَّد الحد الأقصى للطلبات المعلقة عند **3 لكل قناة** افتراضيًا.
- `allowlist`: يُحظر المرسلون غير المعروفين (من دون مصافحة اقتران).
- `open`: اسمح لأي شخص بإرسال رسالة خاصة (عام). **يتطلب** أن تتضمن allowlist الخاصة بالقناة `"*"` (تفعيل صريح).
- `disabled`: تجاهل الرسائل الخاصة الواردة بالكامل.

للموافقة عبر CLI:

```bash
openclaw pairing list <channel>
openclaw pairing approve <channel> <code>
```

التفاصيل + الملفات على القرص: [الاقتران](/ar/channels/pairing)

## عزل جلسات الرسائل الخاصة (الوضع متعدد المستخدمين)

افتراضيًا، يوجّه OpenClaw **كل الرسائل الخاصة إلى الجلسة الرئيسية** حتى يحافظ مساعدك على الاستمرارية عبر الأجهزة والقنوات. إذا كان **عدة أشخاص** يستطيعون مراسلة bot على الخاص (رسائل خاصة مفتوحة أو allowlist متعددة الأشخاص)، ففكّر في عزل جلسات الرسائل الخاصة:

```json5
{
  session: { dmScope: "per-channel-peer" },
}
```

وهذا يمنع تسرب السياق بين المستخدمين مع إبقاء دردشات المجموعات معزولة.

هذا حدّ لسياق الرسائل، وليس حدًا لإدارة المضيف. وإذا كان المستخدمون خصومًا لبعضهم ويشاركون مضيف Gateway/الإعدادات نفسها، فشغّل Gateways منفصلة لكل حد ثقة بدلًا من ذلك.

### وضع الرسائل الخاصة الآمن (موصى به)

اعتبر المقتطف أعلاه **وضع الرسائل الخاصة الآمن**:

- الافتراضي: `session.dmScope: "main"` (كل الرسائل الخاصة تشترك في جلسة واحدة من أجل الاستمرارية).
- الإعداد الأوّلي المحلي عبر CLI افتراضيًا: يكتب `session.dmScope: "per-channel-peer"` عندما لا تكون مضبوطة (مع الإبقاء على القيم الصريحة الموجودة).
- وضع الرسائل الخاصة الآمن: `session.dmScope: "per-channel-peer"` (كل زوج قناة+مرسل يحصل على سياق رسائل خاصة معزول).
- العزل بين القنوات للمرسل الواحد: `session.dmScope: "per-peer"` (كل مرسل يحصل على جلسة واحدة عبر كل القنوات من النوع نفسه).

إذا كنت تشغّل عدة حسابات على القناة نفسها، فاستخدم `per-account-channel-peer` بدلًا من ذلك. وإذا اتصل بك الشخص نفسه على عدة قنوات، فاستخدم `session.identityLinks` لدمج جلسات الرسائل الخاصة تلك في هوية أساسية واحدة. راجع [إدارة الجلسات](/ar/concepts/session) و[الإعدادات](/ar/gateway/configuration).

## allowlists للرسائل الخاصة والمجموعات

يحتوي OpenClaw على طبقتين منفصلتين من "من يمكنه تشغيلي؟":

- **allowlist للرسائل الخاصة** (`allowFrom` / `channels.discord.allowFrom` / `channels.slack.allowFrom`؛ والقديم: `channels.discord.dm.allowFrom`، `channels.slack.dm.allowFrom`): من المسموح له بالتحدث إلى bot في الرسائل المباشرة.
  - عندما تكون `dmPolicy="pairing"`، تُكتب الموافقات إلى مخزن allowlist الخاص بالاقتران والمحدّد بنطاق الحساب ضمن `~/.openclaw/credentials/` ‏(` <channel>-allowFrom.json` للحساب الافتراضي، و`<channel>-<accountId>-allowFrom.json` للحسابات غير الافتراضية)، ثم تُدمج مع allowlists الموجودة في الإعدادات.
- **allowlist للمجموعات** (خاصة بكل قناة): ما هي المجموعات/القنوات/guilds التي سيقبل bot الرسائل منها أصلًا.
  - الأنماط الشائعة:
    - `channels.whatsapp.groups`، و`channels.telegram.groups`، و`channels.imessage.groups`: إعدادات افتراضية لكل مجموعة مثل `requireMention`؛ وعند ضبطها، تعمل أيضًا كـ allowlist للمجموعات (ضمّن `"*"` للحفاظ على سلوك السماح للجميع).
    - `groupPolicy="allowlist"` + `groupAllowFrom`: قيّد من يمكنه تشغيل bot _داخل_ جلسة مجموعة (WhatsApp/Telegram/Signal/iMessage/Microsoft Teams).
    - `channels.discord.guilds` / `channels.slack.channels`: allowlists لكل سطح + إعدادات افتراضية للإشارة.
  - تعمل فحوصات المجموعات بهذا الترتيب: `groupPolicy`/allowlists للمجموعات أولًا، ثم تفعيل الإشارة/الرد ثانيًا.
  - إن الرد على رسالة bot ‏(إشارة ضمنية) **لا** يتجاوز allowlists الخاصة بالمرسلين مثل `groupAllowFrom`.
  - **ملاحظة أمنية:** تعامل مع `dmPolicy="open"` و`groupPolicy="open"` على أنهما إعدادان أخيران عند الاضطرار. ينبغي أن نادرًا ما يُستخدما؛ فضّل pairing + allowlists ما لم تكن تثق بكل عضو في الغرفة ثقة كاملة.

التفاصيل: [الإعدادات](/ar/gateway/configuration) و[المجموعات](/ar/channels/groups)

## حقن Prompt ‏(ما هو، ولماذا يهم)

يحدث حقن prompt عندما يصوغ مهاجم رسالة تتلاعب بالـ model لدفعه إلى القيام بشيء غير آمن ("تجاهل تعليماتك"، أو "افرغ نظام الملفات لديك"، أو "اتبع هذا الرابط ونفّذ أوامر"، وما إلى ذلك).

حتى مع وجود system prompts قوية، **فإن حقن prompt ليس مشكلة محلولة**. فحواجز system prompt ليست سوى إرشادات رخوة؛ أما الإنفاذ الصارم فيأتي من سياسة الأدوات، وموافقات exec، وsandboxing، وallowlists الخاصة بالقنوات (ويمكن للمشغّلين تعطيل هذه الحواجز بحسب التصميم). ما الذي يساعد عمليًا:

- أبقِ الرسائل الخاصة الواردة مغلقة بإحكام (pairing/allowlists).
- فضّل بوابة الإشارة في المجموعات؛ وتجنب bots التي تعمل "دائمًا" في الغرف العامة.
- تعامل مع الروابط، والمرفقات، والتعليمات الملصقة على أنها معادية افتراضيًا.
- شغّل تنفيذ الأدوات الحساسة داخل sandbox؛ وأبقِ الأسرار خارج نظام الملفات الذي يمكن للوكيل الوصول إليه.
- ملاحظة: إن sandboxing يتم عبر opt-in. فإذا كان وضع sandbox معطّلًا، فإن القيمة الضمنية `host=auto` تُحل إلى مضيف gateway. أما القيمة الصريحة `host=sandbox` فستفشل بشكل مغلق لأنه لا يوجد runtime متاح لـ sandbox. اضبط `host=gateway` إذا كنت تريد أن يكون هذا السلوك صريحًا في الإعدادات.
- قيّد الأدوات عالية الخطورة (`exec`، و`browser`، و`web_fetch`، و`web_search`) على الوكلاء الموثوقين أو على allowlists صريحة.
- إذا أضفت المفسرات إلى allowlist ‏(`python`، و`node`، و`ruby`، و`perl`، و`php`، و`lua`، و`osascript`)، ففعّل `tools.exec.strictInlineEval` حتى تظل صيغ eval المضمنة تحتاج إلى موافقة صريحة.
- يرفض تحليل موافقة shell أيضًا صيغ توسعة معلمات POSIX ‏(`$VAR`، و`$?`، و`$$`، و`$1`، و`$@`، و`${…}`) داخل **heredocs غير المقتبسة**، حتى لا يتمكن متن heredoc موضوع في allowlist من تمرير توسعة shell متسللة على أنها نص عادي أثناء مراجعة allowlist. اقتبس محدد heredoc النهائي (مثل `<<'EOF'`) لتفعيل دلالات المتن الحرفية؛ أما heredocs غير المقتبسة التي كانت ستوسع المتغيرات فيتم رفضها.
- **اختيار Model مهم:** إن Models الأقدم/الأصغر/القديمة أقل متانة بشكل ملحوظ ضد حقن prompt وإساءة استخدام الأدوات. بالنسبة إلى الوكلاء المفعّلة لديهم الأدوات، استخدم أقوى model متاح من أحدث الأجيال والمحصّن بالتعليمات.

إشارات حمراء يجب التعامل معها على أنها غير موثوقة:

- "اقرأ هذا الملف/URL وافعل بالضبط ما يقوله."
- "تجاهل system prompt أو قواعد الأمان الخاصة بك."
- "اكشف تعليماتك المخفية أو مخرجات أدواتك."
- "الصق كامل محتويات ~/.openclaw أو سجلاتك."

## تنظيف الرموز الخاصة في المحتوى الخارجي

يقوم OpenClaw بإزالة literals الشائعة للرموز الخاصة بقوالب دردشة LLM المستضافة ذاتيًا من المحتوى الخارجي المغلف والبيانات الوصفية قبل أن تصل إلى model. وتشمل عائلات العلامات المغطاة رموز الأدوار/الأدوار-الدورات الخاصة بـ Qwen/ChatML وLlama وGemma وMistral وPhi وGPT-OSS.

السبب:

- الواجهات الخلفية المتوافقة مع OpenAI التي تضع نماذج مستضافة ذاتيًا في الواجهة الأمامية قد تحتفظ بالرموز الخاصة التي تظهر في نص المستخدم بدلًا من إخفائها. ويمكن لمهاجم يستطيع الكتابة في المحتوى الخارجي الوارد (صفحة جرى جلبها، أو متن بريد إلكتروني، أو مخرجات أداة قراءة ملف) أن يحقن حدًّا اصطناعيًا لدور `assistant` أو `system` ويتجاوز حواجز المحتوى المغلف.
- يحدث التنظيف في طبقة تغليف المحتوى الخارجي، لذلك يطبَّق بشكل موحد على أدوات fetch/read وعلى محتوى القنوات الواردة بدلًا من أن يكون خاصًا بكل provider.
- لدى مخرجات model الصادرة بالفعل أداة تنظيف منفصلة تزيل البنية المسرّبة مثل `<tool_call>` و`<function_calls>` وما شابهها من الردود الظاهرة للمستخدم. أما منظف المحتوى الخارجي فهو النظير الوارد لذلك.

هذا لا يحل محل التحصينات الأخرى في هذه الصفحة — فما تزال `dmPolicy`، وallowlists، وموافقات exec، وsandboxing، و`contextVisibility` تؤدي العمل الأساسي. لكنه يغلق مسار تجاوز محدد على مستوى tokenizer ضد البنى المستضافة ذاتيًا التي تمرر نص المستخدم مع الرموز الخاصة سليمة.

## علامات تجاوز المحتوى الخارجي غير الآمن

يتضمن OpenClaw علامات تجاوز صريحة تعطل تغليف الأمان للمحتوى الخارجي:

- `hooks.mappings[].allowUnsafeExternalContent`
- `hooks.gmail.allowUnsafeExternalContent`
- حقل الحمولة `allowUnsafeExternalContent` في Cron

إرشادات:

- اترك هذه القيم غير مضبوطة/false في الإنتاج.
- فعّلها مؤقتًا فقط لأغراض تصحيح ضيقة النطاق.
- إذا فُعّلت، فاعزل ذلك الوكيل (sandbox + حد أدنى من الأدوات + نطاق جلسات مخصص).

ملاحظة حول خطر الخطافات:

- حمولات الخطافات هي محتوى غير موثوق، حتى عندما يأتي التسليم من أنظمة تتحكم بها (فمحتوى البريد/المستندات/الويب قد يحمل حقن prompt).
- تزيد مستويات model الضعيفة هذا الخطر. بالنسبة إلى الأتمتة المعتمدة على الخطافات، فضّل مستويات Models قوية وحديثة، وأبقِ سياسة الأدوات مشددة (`tools.profile: "messaging"` أو أشد)، مع sandboxing كلما أمكن.

### حقن Prompt لا يتطلب رسائل خاصة عامة

حتى إذا كنت **أنت فقط** من يمكنه مراسلة bot، فلا يزال حقن prompt ممكنًا عبر
أي **محتوى غير موثوق** يقرؤه bot (نتائج web search/fetch، وصفحات المتصفح،
والرسائل الإلكترونية، والمستندات، والمرفقات، والسجلات/الكود الملصوق). بمعنى آخر: المرسل ليس
هو سطح التهديد الوحيد؛ بل إن **المحتوى نفسه** قد يحمل تعليمات عدائية.

عندما تكون الأدوات مفعلة، يكون الخطر المعتاد هو تسريب السياق أو تحفيز
استدعاءات الأدوات. قلّل نطاق الضرر عبر:

- استخدام **وكيل قارئ** للقراءة فقط أو معطّلة الأدوات لتلخيص المحتوى غير الموثوق،
  ثم تمرير الملخص إلى وكيلك الرئيسي.
- إبقاء `web_search` / `web_fetch` / `browser` معطلة للوكلاء المفعلة لديهم الأدوات ما لم تكن هناك حاجة إليها.
- بالنسبة إلى مدخلات URL في OpenResponses ‏(`input_file` / `input_image`)، اضبط
  `gateway.http.endpoints.responses.files.urlAllowlist` و
  `gateway.http.endpoints.responses.images.urlAllowlist` بإحكام، وأبقِ قيمة `maxUrlParts` منخفضة.
  وتُعامل allowlists الفارغة على أنها غير مضبوطة؛ استخدم `files.allowUrl: false` / `images.allowUrl: false`
  إذا كنت تريد تعطيل جلب URL بالكامل.
- بالنسبة إلى مدخلات الملفات في OpenResponses، لا يزال النص المفكوك من `input_file` يُحقن باعتباره
  **محتوى خارجيًا غير موثوق**. لا تعتمد على كون نص الملف موثوقًا لمجرد
  أن Gateway فك ترميزه محليًا. فالمقطع المحقون لا يزال يحمل
  علامات الحدود الصريحة `<<<EXTERNAL_UNTRUSTED_CONTENT ...>>>` بالإضافة إلى بيانات وصفية
  `Source: External`، رغم أن هذا المسار يحذف الشعار الأطول `SECURITY NOTICE:`.
- ويُطبَّق التغليف نفسه القائم على العلامات عندما يستخرج فهم الوسائط نصًا
  من المستندات المرفقة قبل إلحاق ذلك النص بـ prompt الوسائط.
- تفعيل sandboxing وallowlists صارمة للأدوات لأي وكيل يتعامل مع مدخلات غير موثوقة.
- إبقاء الأسرار خارج prompts؛ ومررها عبر env/config على مضيف gateway بدلًا من ذلك.

### الواجهات الخلفية لـ LLM المستضافة ذاتيًا

قد تختلف الواجهات الخلفية المتوافقة مع OpenAI والمستضافة ذاتيًا مثل vLLM وSGLang وTGI وLM Studio،
أو حِزم tokenizer المخصصة من Hugging Face، عن providers المستضافين في كيفية
التعامل مع الرموز الخاصة بقوالب الدردشة. فإذا قامت واجهة خلفية ما بترميز سلاسل حرفية
مثل `<|im_start|>` أو `<|start_header_id|>` أو `<start_of_turn>` على أنها
رموز بنيوية لقالب الدردشة داخل محتوى المستخدم، فيمكن للنص غير الموثوق أن يحاول
تزوير حدود الأدوار على مستوى tokenizer.

يقوم OpenClaw بإزالة literals الشائعة للرموز الخاصة بعائلات models من
المحتوى الخارجي المغلف قبل إرساله إلى model. أبقِ تغليف المحتوى الخارجي
مفعّلًا، وفضّل إعدادات الواجهة الخلفية التي تفصل أو تهرّب الرموز الخاصة
في المحتوى المقدَّم من المستخدم عندما يكون ذلك متاحًا. أما providers المستضافون مثل OpenAI
وAnthropic فيطبقون بالفعل عمليات تنظيف خاصة بهم على جانب الطلب.

### قوة Model ‏(ملاحظة أمنية)

إن مقاومة حقن prompt **ليست موحدة** عبر مستويات Models. فعادةً ما تكون Models الأصغر/الأرخص أكثر قابلية لإساءة استخدام الأدوات وخطف التعليمات، خصوصًا تحت prompts العدائية.

<Warning>
بالنسبة إلى الوكلاء المفعلة لديهم الأدوات أو الوكلاء الذين يقرأون محتوى غير موثوق، فإن خطر حقن prompt مع Models الأقدم/الأصغر يكون غالبًا مرتفعًا للغاية. لا تشغّل تلك الأحمال على مستويات Models الضعيفة.
</Warning>

التوصيات:

- **استخدم أحدث جيل وأفضل Model متاح** لأي bot يمكنه تشغيل أدوات أو الوصول إلى الملفات/الشبكات.
- **لا تستخدم مستويات أقدم/أضعف/أصغر** للوكلاء المفعلة لديهم الأدوات أو لصناديق الوارد غير الموثوقة؛ فخطر حقن prompt مرتفع جدًا.
- إذا اضطررت إلى استخدام Model أصغر، **فقلل نطاق الضرر** (أدوات للقراءة فقط، وsandboxing قوية، ووصول أدنى إلى نظام الملفات، وallowlists صارمة).
- عند تشغيل Models صغيرة، **فعّل sandboxing لكل الجلسات** و**عطّل web_search/web_fetch/browser** ما لم تكن المدخلات محكومة بإحكام.
- بالنسبة إلى المساعدين الشخصيين للدردشة فقط مع مدخلات موثوقة ومن دون أدوات، فعادةً ما تكون Models الأصغر مناسبة.

## الاستدلال والمخرجات المطولة في المجموعات

قد تكشف `/reasoning` و`/verbose` و`/trace` عن استدلال داخلي، أو مخرجات أدوات،
أو تشخيصات Plugin
لم يكن المقصود عرضها في قناة عامة. وفي إعدادات المجموعات، تعامل معها على أنها **للتصحيح فقط**
وأبقها معطلة ما لم تكن تحتاجها صراحةً.

إرشادات:

- أبقِ `/reasoning` و`/verbose` و`/trace` معطلة في الغرف العامة.
- إذا فعّلتها، فافعَل ذلك فقط في الرسائل الخاصة الموثوقة أو الغرف المحكمة بإحكام.
- تذكّر: قد تتضمن المخرجات المطولة ومخرجات التتبع معاملات الأدوات، وعناوين URL، وتشخيصات Plugin، وبيانات رآها model.

## أمثلة لتحصين الإعدادات

### أذونات الملفات

احتفظ بالإعدادات + الحالة على نحو خاص على مضيف gateway:

- `~/.openclaw/openclaw.json`: ‏`600` ‏(قراءة/كتابة للمستخدم فقط)
- `~/.openclaw`: ‏`700` ‏(للمستخدم فقط)

يمكن لـ `openclaw doctor` التحذير واقتراح تشديد هذه الأذونات.

### تعرض الشبكة (الربط، والمنفذ، والجدار الناري)

يقوم Gateway بعمل multiplexing لكل من **WebSocket + HTTP** على منفذ واحد:

- الافتراضي: `18789`
- الإعدادات/العلامات/env: ‏`gateway.port`، و`--port`، و`OPENCLAW_GATEWAY_PORT`

يتضمن هذا السطح HTTP كلًا من Control UI ومضيف canvas:

- Control UI ‏(أصول SPA) ‏(المسار الأساسي الافتراضي `/`)
- مضيف Canvas: ‏`/__openclaw__/canvas/` و`/__openclaw__/a2ui/` ‏(HTML/JS اعتباطية؛ تعامل معها على أنها محتوى غير موثوق)

إذا حمّلت محتوى canvas في متصفح عادي، فتعامل معه كما تتعامل مع أي صفحة ويب غير موثوقة أخرى:

- لا تعرّض مضيف canvas لشبكات/مستخدمين غير موثوقين.
- لا تجعل محتوى canvas يشارك الأصل نفسه مع أسطح الويب ذات الامتيازات ما لم تكن تفهم الآثار تمامًا.

يتحكم وضع الربط في المكان الذي يستمع فيه Gateway:

- `gateway.bind: "loopback"` ‏(الافتراضي): لا يمكن الاتصال إلا من العملاء المحليين.
- توسّع أوضاع الربط غير المعتمدة على loopback ‏(`"lan"`، و`"tailnet"`، و`"custom"`) سطح الهجوم. استخدمها فقط مع مصادقة gateway ‏(token/password مشتركان أو trusted proxy مهيأ بشكل صحيح وغير معتمد على loopback) ومع جدار ناري حقيقي.

قواعد عامة:

- فضّل Tailscale Serve على ربط LAN ‏(يبقي Serve Gateway على loopback، بينما يتولى Tailscale التحكم في الوصول).
- إذا اضطررت إلى الربط على LAN، فاحصر المنفذ في الجدار الناري على allowlist ضيقة من عناوين IP المصدر؛ ولا تقم بعمل port-forward له على نطاق واسع.
- لا تعرّض Gateway أبدًا من دون مصادقة على `0.0.0.0`.

### نشر منافذ Docker مع UFW

إذا كنت تشغّل OpenClaw باستخدام Docker على VPS، فتذكّر أن منافذ الحاويات المنشورة
‏(`-p HOST:CONTAINER` أو `ports:` في Compose) تُوجَّه عبر سلاسل forwarding الخاصة بـ Docker،
وليس فقط عبر قواعد `INPUT` الخاصة بالمضيف.

ولإبقاء حركة Docker متوافقة مع سياسة الجدار الناري، افرض القواعد في
`DOCKER-USER` ‏(يتم تقييم هذه السلسلة قبل قواعد القبول الخاصة بـ Docker).
وفي كثير من التوزيعات الحديثة، تستخدم `iptables`/`ip6tables` الواجهة الأمامية `iptables-nft`
ومع ذلك تطبّق هذه القواعد على الواجهة الخلفية nftables.

مثال minimal allowlist ‏(IPv4):

```bash
# /etc/ufw/after.rules (أضفه كقسم *filter مستقل)
*filter
:DOCKER-USER - [0:0]
-A DOCKER-USER -m conntrack --ctstate ESTABLISHED,RELATED -j RETURN
-A DOCKER-USER -s 127.0.0.0/8 -j RETURN
-A DOCKER-USER -s 10.0.0.0/8 -j RETURN
-A DOCKER-USER -s 172.16.0.0/12 -j RETURN
-A DOCKER-USER -s 192.168.0.0/16 -j RETURN
-A DOCKER-USER -s 100.64.0.0/10 -j RETURN
-A DOCKER-USER -p tcp --dport 80 -j RETURN
-A DOCKER-USER -p tcp --dport 443 -j RETURN
-A DOCKER-USER -m conntrack --ctstate NEW -j DROP
-A DOCKER-USER -j RETURN
COMMIT
```

لدى IPv6 جداول منفصلة. أضف سياسة مطابقة في `/etc/ufw/after6.rules` إذا
كان Docker IPv6 مفعّلًا.

تجنب تثبيت أسماء الواجهات مثل `eth0` في مقتطفات التوثيق. تختلف أسماء الواجهات
عبر صور VPS ‏(`ens3`، و`enp*`، وما إلى ذلك) وقد يؤدي عدم التطابق إلى
تجاوز قاعدة الرفض الخاصة بك دون قصد.

تحقق سريع بعد إعادة التحميل:

```bash
ufw reload
iptables -S DOCKER-USER
ip6tables -S DOCKER-USER
nmap -sT -p 1-65535 <public-ip> --open
```

يجب أن تكون المنافذ الخارجية المتوقعة فقط هي تلك التي تعرّضها عمدًا (وفي معظم
الإعدادات: SSH + منافذ reverse proxy الخاصة بك).

### اكتشاف mDNS/Bonjour

يبث Gateway وجوده عبر mDNS ‏(`_openclaw-gw._tcp` على المنفذ 5353) لاكتشاف الأجهزة المحلية. وفي الوضع الكامل، يتضمن هذا سجلات TXT قد تكشف تفاصيل تشغيلية:

- `cliPath`: المسار الكامل لنظام الملفات إلى CLI binary ‏(يكشف اسم المستخدم وموقع التثبيت)
- `sshPort`: يعلن عن توفر SSH على المضيف
- `displayName`، و`lanHost`: معلومات اسم المضيف

**اعتبار الأمان التشغيلي:** إن بث تفاصيل البنية التحتية يجعل الاستطلاع أسهل لأي شخص على الشبكة المحلية. وحتى المعلومات "غير الضارة" مثل مسارات نظام الملفات وتوفر SSH تساعد المهاجمين على رسم خريطة لبيئتك.

**التوصيات:**

1. **الوضع الأدنى** (الافتراضي، وموصى به للـ gateways المعرّضة): احذف الحقول الحساسة من بث mDNS:

   ```json5
   {
     discovery: {
       mdns: { mode: "minimal" },
     },
   }
   ```

2. **عطّله بالكامل** إذا لم تكن تحتاج إلى اكتشاف الأجهزة المحلية:

   ```json5
   {
     discovery: {
       mdns: { mode: "off" },
     },
   }
   ```

3. **الوضع الكامل** (عبر opt-in): ضمّن `cliPath` و`sshPort` في سجلات TXT:

   ```json5
   {
     discovery: {
       mdns: { mode: "full" },
     },
   }
   ```

4. **متغير البيئة** (بديل): اضبط `OPENCLAW_DISABLE_BONJOUR=1` لتعطيل mDNS من دون تغييرات في الإعدادات.

في الوضع الأدنى، لا يزال Gateway يبث ما يكفي لاكتشاف الأجهزة (`role`، و`gatewayPort`، و`transport`) لكنه يحذف `cliPath` و`sshPort`. ويمكن للتطبيقات التي تحتاج إلى معلومات مسار CLI جلبها عبر اتصال WebSocket الموثق بدلًا من ذلك.

### أقفل WebSocket الخاص بـ Gateway ‏(المصادقة المحلية)

تكون مصادقة Gateway **مطلوبة افتراضيًا**. وإذا لم يكن هناك مسار صالح لمصادقة gateway مهيأ،
يرفض Gateway اتصالات WebSocket ‏(fail‑closed).

يولّد الإعداد الأوّلي token افتراضيًا (حتى لـ loopback) بحيث
يجب على العملاء المحليين المصادقة.

اضبط token بحيث **يجب على كل** عملاء WS المصادقة:

```json5
{
  gateway: {
    auth: { mode: "token", token: "your-token" },
  },
}
```

يمكن لـ Doctor توليد واحد لك: `openclaw doctor --generate-gateway-token`.

ملاحظة: إن `gateway.remote.token` / `.password` هما مصدرَا بيانات اعتماد للعميل. وهما
**لا** يحميان وصول WS المحلي بمفردهما.
يمكن للمسارات المحلية استخدام `gateway.remote.*` كخيار fallback فقط عندما تكون `gateway.auth.*`
غير مضبوطة.
وإذا كانت `gateway.auth.token` / `gateway.auth.password` مضبوطة صراحةً عبر
SecretRef ولم تُحل، يفشل الحل بشكل مغلق (من دون fallback بعيد يخفي المشكلة).
اختياري: ثبّت TLS البعيد عبر `gateway.remote.tlsFingerprint` عند استخدام `wss://`.
تكون `ws://` النصية الصريحة مخصصة لـ loopback فقط افتراضيًا. أما لمسارات
الشبكات الخاصة الموثوقة، فاضبط `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` على عملية العميل
كإجراء طوارئ. وهذا مقصود ليكون في بيئة العملية فقط، وليس
مفتاح إعدادات في `openclaw.json`.
يكون الاقتران على الأجهزة المحمولة والمسارات اليدوية أو الممسوحة على Android إلى gateway أكثر صرامة:
يُقبل الاتصال النصي الصريح على loopback، لكن أسماء المضيف الخاصة بشبكات LAN الخاصة، وlink-local، و`.local`،
وأسماء المضيف الخالية من النقاط يجب أن تستخدم TLS ما لم تختر صراحةً
المسار النصي الصريح الموثوق للشبكات الخاصة.

الاقتران المحلي للأجهزة:

- تتم الموافقة تلقائيًا على اقتران الأجهزة للاتصالات المباشرة عبر loopback المحلي لإبقاء
  العملاء على المضيف نفسه سلسين.
- ويملك OpenClaw أيضًا مسار self-connect ضيقًا للخلفية/الحاويات على المضيف المحلي
  لتدفقات المساعدين الموثوقين ذات السر المشترك.
- أما الاتصالات عبر tailnet وLAN، بما في ذلك ربط tailnet على المضيف نفسه، فتُعامل على أنها
  بعيدة لأغراض الاقتران ولا تزال تحتاج إلى موافقة.
- تؤدي أدلة forwarded-header في طلب loopback إلى إسقاط
  صفة المحلية القائمة على loopback. وتكون الموافقة التلقائية على ترقية البيانات الوصفية ضيقة النطاق.
  راجع [اقتران Gateway](/ar/gateway/pairing) للاطلاع على القاعدتين.

أوضاع المصادقة:

- `gateway.auth.mode: "token"`: bearer token مشتركة (موصى بها لمعظم الإعدادات).
- `gateway.auth.mode: "password"`: مصادقة بكلمة مرور (ويُفضَّل ضبطها عبر env: `OPENCLAW_GATEWAY_PASSWORD`).
- `gateway.auth.mode: "trusted-proxy"`: الثقة في reverse proxy واعية بالهوية لمصادقة المستخدمين وتمرير الهوية عبر الترويسات (راجع [Trusted Proxy Auth](/ar/gateway/trusted-proxy-auth)).

قائمة التحقق من التدوير (token/password):

1. ولّد/اضبط سرًا جديدًا (`gateway.auth.token` أو `OPENCLAW_GATEWAY_PASSWORD`).
2. أعد تشغيل Gateway ‏(أو أعد تشغيل تطبيق macOS إذا كان يشرف على Gateway).
3. حدّث أي عملاء بعيدين (`gateway.remote.token` / `.password` على الأجهزة التي تستدعي Gateway).
4. تحقّق من أنك لم تعد تستطيع الاتصال باستخدام بيانات الاعتماد القديمة.

### ترويسات هوية Tailscale Serve

عندما تكون `gateway.auth.allowTailscale` بقيمة `true` (الافتراضي لـ Serve)، يقبل OpenClaw
ترويسات هوية Tailscale Serve ‏(`tailscale-user-login`) لمصادقة Control
UI/WebSocket. ويتحقق OpenClaw من الهوية عبر حل عنوان
`x-forwarded-for` من خلال daemon المحلي لـ Tailscale ‏(`tailscale whois`)
ومطابقته مع الترويسة. ولا يتم تفعيل هذا إلا للطلبات التي تضرب loopback
وتتضمن `x-forwarded-for` و`x-forwarded-proto` و`x-forwarded-host` كما
يحقنها Tailscale.
وبالنسبة إلى هذا المسار غير المتزامن للتحقق من الهوية، تُسلسَل المحاولات الفاشلة لنفس `{scope, ip}`
قبل أن يسجّل المحدِّد الفشل. ولذلك فإن عمليات إعادة المحاولة السيئة المتزامنة
من عميل Serve واحد قد تؤدي إلى حظر المحاولة الثانية فورًا
بدلًا من أن تمر كسلسلتين عاديتين من عدم التطابق.
أما نقاط نهاية HTTP API ‏(مثل `/v1/*`، و`/tools/invoke`، و`/api/channels/*`)
فهي **لا** تستخدم مصادقة ترويسات هوية Tailscale. بل تتبع
وضع HTTP auth المهيأ في gateway.

ملاحظة مهمة حول الحدود:

- مصادقة bearer الخاصة بـ Gateway HTTP هي عمليًا وصول شامل أو لا شيء على مستوى المشغّل.
- تعامل مع بيانات الاعتماد القادرة على استدعاء `/v1/chat/completions`، أو `/v1/responses`، أو `/api/channels/*` على أنها أسرار مشغّل كاملة الصلاحية لذلك gateway.
- على سطح HTTP المتوافق مع OpenAI، تعيد مصادقة bearer بالسر المشترك كامل نطاقات المشغّل الافتراضية (`operator.admin`، و`operator.approvals`، و`operator.pairing`، و`operator.read`، و`operator.talk.secrets`، و`operator.write`) بالإضافة إلى دلالات المالك لدورات الوكيل؛ ولا تُقلّل القيم الأضيق لـ `x-openclaw-scopes` من ذلك المسار المعتمد على السر المشترك.
- تنطبق دلالات النطاق لكل طلب على HTTP فقط عندما يأتي الطلب من وضع حامل للهوية مثل trusted proxy auth أو `gateway.auth.mode="none"` على ingress خاص.
- في هذه الأوضاع الحاملة للهوية، يؤدي حذف `x-openclaw-scopes` إلى الرجوع إلى مجموعة نطاقات المشغّل الافتراضية العادية؛ لذا أرسل الترويسة صراحةً عندما تريد مجموعة نطاقات أضيق.
- يتبع `/tools/invoke` قاعدة السر المشترك نفسها: فمصادقة bearer عبر token/password تُعامل هناك أيضًا على أنها وصول كامل للمشغّل، بينما لا تزال الأوضاع الحاملة للهوية تحترم النطاقات المعلنة.
- لا تشارك بيانات الاعتماد هذه مع جهات غير موثوقة؛ وفضّل Gateways منفصلة لكل حد ثقة.

**افتراض الثقة:** تفترض مصادقة Serve من دون token أن مضيف gateway موثوق.
ولا تتعامل مع هذا على أنه حماية ضد العمليات العدائية الموجودة على المضيف نفسه. وإذا كان
من المحتمل تشغيل كود محلي غير موثوق على مضيف gateway، فعطّل `gateway.auth.allowTailscale`
واشترط مصادقة صريحة عبر السر المشترك باستخدام `gateway.auth.mode: "token"` أو
`"password"`.

**قاعدة أمان:** لا تمرر هذه الترويسات من reverse proxy الخاصة بك. وإذا
أنهيت TLS أو مررت الطلبات أمام gateway، فعطّل
`gateway.auth.allowTailscale` واستخدم مصادقة السر المشترك (`gateway.auth.mode:
"token"` أو `"password"`) أو [Trusted Proxy Auth](/ar/gateway/trusted-proxy-auth)
بدلًا من ذلك.

Proxies موثوقة:

- إذا أنهيت TLS أمام Gateway، فاضبط `gateway.trustedProxies` على عناوين IP الخاصة بالـ proxy.
- سيثق OpenClaw في `x-forwarded-for` (أو `x-real-ip`) القادمة من تلك العناوين لتحديد IP العميل لأغراض فحوصات الاقتران المحلي وفحوصات HTTP auth/local.
- تأكد من أن proxy الخاصة بك **تستبدل** `x-forwarded-for` وتحظر الوصول المباشر إلى منفذ Gateway.

راجع [Tailscale](/ar/gateway/tailscale) و[نظرة عامة على الويب](/ar/web).

### التحكم في المتصفح عبر مضيف node ‏(موصى به)

إذا كان Gateway بعيدًا لكن المتصفح يعمل على جهاز آخر، فشغّل **مضيف node**
على جهاز المتصفح ودع Gateway يمرر إجراءات المتصفح بالوكالة (راجع [أداة Browser](/ar/tools/browser)).
تعامل مع اقتران node كما تتعامل مع وصول المشغّل الإداري.

النمط الموصى به:

- أبقِ Gateway ومضيف node على tailnet نفسها (Tailscale).
- اقترن بالـ node عن قصد؛ وعطّل توجيه proxy للمتصفح إذا لم تكن تحتاج إليه.

تجنب:

- تعريض منافذ relay/control عبر LAN أو الإنترنت العام.
- استخدام Tailscale Funnel مع نقاط نهاية التحكم في المتصفح (تعرض عام).

### الأسرار على القرص

افترض أن أي شيء تحت `~/.openclaw/` (أو `$OPENCLAW_STATE_DIR/`) قد يحتوي على أسرار أو بيانات خاصة:

- `openclaw.json`: قد تتضمن الإعدادات tokens ‏(gateway، وgateway البعيدة)، وإعدادات provider، وallowlists.
- `credentials/**`: بيانات اعتماد القنوات (مثل بيانات اعتماد WhatsApp)، وallowlists الخاصة بالاقتران، واستيرادات OAuth القديمة.
- `agents/<agentId>/agent/auth-profiles.json`: API keys، وملفات تعريف tokens، وOAuth tokens، و`keyRef`/`tokenRef` الاختيارية.
- `secrets.json` ‏(اختياري): حمولة أسرار مدعومة بملف وتُستخدم من قبل مزودات SecretRef من نوع `file` ‏(`secrets.providers`).
- `agents/<agentId>/agent/auth.json`: ملف توافق قديم. ويتم تنظيف إدخالات `api_key` الثابتة عند اكتشافها.
- `agents/<agentId>/sessions/**`: نصوص الجلسات (`*.jsonl`) + بيانات التوجيه الوصفية (`sessions.json`) التي قد تحتوي على رسائل خاصة ومخرجات أدوات.
- حزم Plugins المضمّنة: Plugins المثبتة (بالإضافة إلى `node_modules/` الخاصة بها).
- `sandboxes/**`: مساحات عمل sandbox للأدوات؛ وقد تتراكم فيها نسخ من الملفات التي تقرأها/تكتبها داخل sandbox.

نصائح للتحصين:

- أبقِ الأذونات مشددة (`700` على الأدلة، و`600` على الملفات).
- استخدم تشفيرًا كاملًا للقرص على مضيف gateway.
- فضّل حساب مستخدم OS مخصصًا لـ Gateway إذا كان المضيف مشتركًا.

### ملفات `.env` في مساحة العمل

يحمّل OpenClaw ملفات `.env` المحلية في مساحة العمل للوكلاء والأدوات، لكنه لا يسمح أبدًا لتلك الملفات بأن تتجاوز بصمت عناصر التحكم في runtime الخاصة بالـ gateway.

- يتم حظر أي مفتاح يبدأ بـ `OPENCLAW_*` من ملفات `.env` غير الموثوقة في مساحة العمل.
- كما يتم حظر إعدادات نقاط نهاية القنوات الخاصة بـ Matrix وMattermost وIRC وSynology Chat من التجاوز عبر `.env` في مساحة العمل، حتى لا تتمكن مساحات العمل المستنسخة من إعادة توجيه حركة الموصلات المضمّنة عبر إعدادات نقطة نهاية محلية. ويجب أن تأتي مفاتيح env الخاصة بنقاط النهاية (مثل `MATRIX_HOMESERVER`، و`MATTERMOST_URL`، و`IRC_HOST`، و`SYNOLOGY_CHAT_INCOMING_URL`) من بيئة عملية gateway أو من `env.shellEnv`، لا من `.env` المحمّلة من مساحة العمل.
- يعمل الحظر بوضع fail-closed: فلا يمكن لمتغير جديد للتحكم في runtime يُضاف في إصدار مستقبلي أن يُورّث من ملف `.env` مضمّن في المستودع أو مضاف من مهاجم؛ بل يتم تجاهل المفتاح ويحتفظ gateway بقيمته الخاصة.
- لا تزال متغيرات البيئة الموثوقة الخاصة بالعملية/نظام التشغيل (shell الخاصة بالـ gateway، أو وحدة launchd/systemd، أو حزمة التطبيق) تُطبّق — فهذا القيد يقتصر فقط على تحميل ملفات `.env`.

السبب: غالبًا ما تعيش ملفات `.env` في مساحة العمل بجوار كود الوكيل، ويجري تضمينها في commit بالخطأ، أو تكتبها الأدوات. إن حظر كامل البادئة `OPENCLAW_*` يعني أن إضافة علامة `OPENCLAW_*` جديدة لاحقًا لا يمكن أبدًا أن تتراجع إلى وراثة صامتة من حالة مساحة العمل.

### السجلات ونصوص الجلسات (التنقيح والاحتفاظ)

قد تؤدي السجلات ونصوص الجلسات إلى تسريب معلومات حساسة حتى عندما تكون عناصر التحكم في الوصول صحيحة:

- قد تتضمن سجلات Gateway ملخصات الأدوات، والأخطاء، وعناوين URL.
- قد تتضمن نصوص الجلسات أسرارًا ملصوقة، ومحتويات ملفات، ومخرجات أوامر، وروابط.

التوصيات:

- أبقِ تنقيح ملخص الأدوات مفعّلًا (`logging.redactSensitive: "tools"`؛ وهو الافتراضي).
- أضف أنماطًا مخصصة لبيئتك عبر `logging.redactPatterns` ‏(tokens، وأسماء المضيفين، وعناوين URL الداخلية).
- عند مشاركة التشخيصات، فضّل `openclaw status --all` ‏(قابل للصق، مع تنقيح الأسرار) على السجلات الخام.
- قلّم نصوص الجلسات القديمة وملفات السجل إذا لم تكن تحتاج إلى احتفاظ طويل.

التفاصيل: [Logging](/ar/gateway/logging)

### الرسائل الخاصة: الاقتران افتراضيًا

```json5
{
  channels: { whatsapp: { dmPolicy: "pairing" } },
}
```

### المجموعات: اشترط الإشارة في كل مكان

```json
{
  "channels": {
    "whatsapp": {
      "groups": {
        "*": { "requireMention": true }
      }
    }
  },
  "agents": {
    "list": [
      {
        "id": "main",
        "groupChat": { "mentionPatterns": ["@openclaw", "@mybot"] }
      }

```

في دردشات المجموعات، لا ترد إلا عند الإشارة الصريحة.

### أرقام منفصلة (WhatsApp، وSignal، وTelegram)

بالنسبة إلى القنوات المعتمدة على أرقام الهواتف، فكّر في تشغيل AI الخاص بك على رقم هاتف منفصل عن رقمك الشخصي:

- الرقم الشخصي: تبقى محادثاتك خاصة
- رقم bot: يتعامل معه AI، مع الحدود المناسبة

### وضع القراءة فقط (عبر sandbox والأدوات)

يمكنك بناء ملف تعريف للقراءة فقط عبر الجمع بين:

- `agents.defaults.sandbox.workspaceAccess: "ro"` (أو `"none"` لعدم السماح بأي وصول إلى مساحة العمل)
- قوائم السماح/الرفض للأدوات التي تحظر `write` و`edit` و`apply_patch` و`exec` و`process` وما إلى ذلك.

خيارات تحصين إضافية:

- `tools.exec.applyPatch.workspaceOnly: true` (الافتراضي): يضمن أن `apply_patch` لا يمكنه الكتابة/الحذف خارج دليل مساحة العمل حتى عندما يكون sandboxing معطلًا. اضبطه على `false` فقط إذا كنت تريد عمدًا أن يلمس `apply_patch` ملفات خارج مساحة العمل.
- `tools.fs.workspaceOnly: true` (اختياري): يقيّد مسارات `read`/`write`/`edit`/`apply_patch` ومسارات التحميل التلقائي الأصلية لصور prompt إلى دليل مساحة العمل (وهو مفيد إذا كنت تسمح اليوم بالمسارات المطلقة وتريد حاجزًا واحدًا شاملًا).
- أبقِ جذور نظام الملفات ضيقة: تجنب الجذور الواسعة مثل دليلك المنزلي لمساحات عمل الوكلاء/مساحات عمل sandbox. فقد تكشف الجذور الواسعة ملفات محلية حساسة (مثل state/config تحت `~/.openclaw`) لأدوات نظام الملفات.

### خط أساس آمن (نسخ/لصق)

إعداد "آمن افتراضيًا" واحد يُبقي Gateway خاصة، ويشترط pairing للرسائل الخاصة، ويتجنب bots للمجموعات تعمل دائمًا:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    port: 18789,
    auth: { mode: "token", token: "your-long-random-token" },
  },
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } },
    },
  },
}
```

إذا كنت تريد تنفيذ أدوات "أكثر أمانًا افتراضيًا" أيضًا، فأضف sandbox + ارفض الأدوات الخطرة لأي وكيل غير مملوك (راجع المثال أدناه تحت "ملفات تعريف الوصول لكل وكيل").

الخط الأساس المدمج لدورات الوكيل المدفوعة بالدردشة: لا يستطيع المرسلون غير المالكين استخدام أداتي `cron` أو `gateway`.

## Sandboxing ‏(موصى به)

توثيق مخصص: [Sandboxing](/ar/gateway/sandboxing)

نهجان متكاملان:

- **شغّل Gateway كاملة داخل Docker** (حد حاوية): [Docker](/ar/install/docker)
- **Tool sandbox** ‏(`agents.defaults.sandbox`، مع Gateway على المضيف وأدوات معزولة داخل sandbox؛ وDocker هو الواجهة الخلفية الافتراضية): [Sandboxing](/ar/gateway/sandboxing)

ملاحظة: لمنع الوصول بين الوكلاء، أبقِ `agents.defaults.sandbox.scope` على `"agent"` (الافتراضي)
أو `"session"` لعزل أشد لكل جلسة. أما `scope: "shared"` فتستخدم
حاوية/مساحة عمل واحدة مشتركة.

وفكّر أيضًا في وصول مساحة عمل الوكيل داخل sandbox:

- `agents.defaults.sandbox.workspaceAccess: "none"` (الافتراضي) يبقي مساحة عمل الوكيل خارج المتناول؛ وتعمل الأدوات على مساحة عمل sandbox ضمن `~/.openclaw/sandboxes`
- `agents.defaults.sandbox.workspaceAccess: "ro"` يربط مساحة عمل الوكيل للقراءة فقط عند `/agent` (ويعطل `write`/`edit`/`apply_patch`)
- `agents.defaults.sandbox.workspaceAccess: "rw"` يربط مساحة عمل الوكيل قراءة/كتابة عند `/workspace`
- يتم التحقق من عناصر `sandbox.docker.binds` الإضافية مقابل مسارات المصدر المطبعّة والمقننة. وتفشل حيل parent-symlink والأسماء المستعارة القياسية للمنزل بشكل مغلق إذا كانت تُحل إلى جذور محظورة مثل `/etc`، أو `/var/run`، أو أدلة بيانات الاعتماد تحت دليل المنزل الخاص بنظام التشغيل.

مهم: تمثل `tools.elevated` منفذ الهروب الأساسي العام الذي يشغّل exec خارج sandbox. ويكون المضيف الفعلي `gateway` افتراضيًا، أو `node` عندما يكون هدف exec مضبوطًا على `node`. أبقِ `tools.elevated.allowFrom` ضيقة ولا تفعّلها للغرباء. ويمكنك تقييد elevated أكثر لكل وكيل عبر `agents.list[].tools.elevated`. راجع [Elevated Mode](/ar/tools/elevated).

### حاجز تفويض الوكلاء الفرعيين

إذا سمحت بأدوات الجلسات، فتعامل مع التشغيلات المفوَّضة للوكلاء الفرعيين على أنها قرار حدودي آخر:

- ارفض `sessions_spawn` ما لم يكن الوكيل يحتاج التفويض فعلًا.
- أبقِ `agents.defaults.subagents.allowAgents` وأي تجاوزات على مستوى كل وكيل في `agents.list[].subagents.allowAgents` مقتصرة على الوكلاء المستهدفين المعروفين بأنهم آمنون.
- لأي سير عمل يجب أن يبقى داخل sandbox، استدعِ `sessions_spawn` مع `sandbox: "require"` (الافتراضي هو `inherit`).
- تفشل `sandbox: "require"` بسرعة عندما لا يكون runtime الفرعي المستهدف داخل sandbox.

## مخاطر التحكم في المتصفح

إن تفعيل التحكم في المتصفح يمنح model القدرة على قيادة متصفح حقيقي.
وإذا كان ملف تعريف ذلك المتصفح يحتوي بالفعل على جلسات مسجّل دخولها،
فيمكن للـ model الوصول إلى تلك الحسابات والبيانات. تعامل مع ملفات تعريف المتصفح على أنها **حالة حساسة**:

- فضّل ملف تعريف مخصصًا للوكيل (ملف تعريف `openclaw` الافتراضي).
- تجنب توجيه الوكيل إلى ملفك الشخصي اليومي.
- أبقِ التحكم في متصفح المضيف معطلًا للوكلاء الموجودين داخل sandbox ما لم تكن تثق بهم.
- لا تحترم API المستقلة للتحكم في المتصفح عبر loopback إلا المصادقة
  القائمة على السر المشترك (bearer token الخاصة بـ gateway أو كلمة مرور gateway). وهي لا تستهلك
  ترويسات الهوية trusted-proxy أو Tailscale Serve.
- تعامل مع تنزيلات المتصفح كمدخلات غير موثوقة؛ وفضّل دليل تنزيلات معزولًا.
- عطّل مزامنة المتصفح/مديري كلمات المرور في ملف تعريف الوكيل إن أمكن (لتقليل نطاق الضرر).
- بالنسبة إلى Gateways البعيدة، افترض أن "التحكم في المتصفح" يعادل "وصول المشغّل" إلى كل ما يمكن لذلك الملف الشخصي الوصول إليه.
- أبقِ Gateway ومضيفي node ضمن tailnet فقط؛ وتجنب تعريض منافذ التحكم في المتصفح إلى LAN أو الإنترنت العام.
- عطّل توجيه proxy للمتصفح عندما لا تحتاجه (`gateway.nodes.browser.mode="off"`).
- إن وضع الجلسة الموجودة مسبقًا في Chrome MCP **ليس** "أكثر أمانًا"؛ فهو يمكنه التصرف بصفتك في كل ما يمكن لملف تعريف Chrome ذاك الوصول إليه.

### سياسة SSRF الخاصة بالمتصفح (صارمة افتراضيًا)

إن سياسة التنقل في المتصفح لدى OpenClaw صارمة افتراضيًا: تظل الوجهات الخاصة/الداخلية محظورة ما لم تختر صراحةً خلاف ذلك.

- الافتراضي: لا تكون `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` مضبوطة، لذا يظل التنقل في المتصفح يحظر الوجهات الخاصة/الداخلية/ذات الاستخدام الخاص.
- الاسم المستعار القديم: لا تزال `browser.ssrfPolicy.allowPrivateNetwork` مقبولة من أجل التوافق.
- وضع opt-in: اضبط `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` للسماح بالوجهات الخاصة/الداخلية/ذات الاستخدام الخاص.
- في الوضع الصارم، استخدم `hostnameAllowlist` (أنماط مثل `*.example.com`) و`allowedHostnames` (استثناءات للمضيفات الدقيقة، بما في ذلك الأسماء المحظورة مثل `localhost`) للاستثناءات الصريحة.
- يتم التحقق من التنقل قبل الطلب، ويُعاد التحقق بأفضل جهد من عنوان URL النهائي `http(s)` بعد التنقل لتقليل الانعطافات المعتمدة على إعادة التوجيه.

مثال على سياسة صارمة:

```json5
{
  browser: {
    ssrfPolicy: {
      dangerouslyAllowPrivateNetwork: false,
      hostnameAllowlist: ["*.example.com", "example.com"],
      allowedHostnames: ["localhost"],
    },
  },
}
```

## ملفات تعريف الوصول لكل وكيل (متعدد الوكلاء)

مع التوجيه متعدد الوكلاء، يمكن أن يكون لكل وكيل سياسة sandbox + أدوات خاصة به:
استخدم هذا لمنح **وصول كامل**، أو **قراءة فقط**، أو **عدم وصول** لكل وكيل.
راجع [Sandbox & Tools متعددة الوكلاء](/ar/tools/multi-agent-sandbox-tools) للاطلاع على التفاصيل الكاملة
وقواعد الأولوية.

حالات الاستخدام الشائعة:

- وكيل شخصي: وصول كامل، بلا sandbox
- وكيل للعائلة/العمل: داخل sandbox + أدوات للقراءة فقط
- وكيل عام: داخل sandbox + بلا أدوات لنظام الملفات/الـ shell

### مثال: وصول كامل (بلا sandbox)

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        workspace: "~/.openclaw/workspace-personal",
        sandbox: { mode: "off" },
      },
    ],
  },
}
```

### مثال: أدوات للقراءة فقط + مساحة عمل للقراءة فقط

```json5
{
  agents: {
    list: [
      {
        id: "family",
        workspace: "~/.openclaw/workspace-family",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "ro",
        },
        tools: {
          allow: ["read"],
          deny: ["write", "edit", "apply_patch", "exec", "process", "browser"],
        },
      },
    ],
  },
}
```

### مثال: بدون وصول إلى نظام الملفات/الـ shell (مع السماح برسائل provider)

```json5
{
  agents: {
    list: [
      {
        id: "public",
        workspace: "~/.openclaw/workspace-public",
        sandbox: {
          mode: "all",
          scope: "agent",
          workspaceAccess: "none",
        },
        // يمكن لأدوات الجلسات كشف بيانات حساسة من النصوص. يقيّد OpenClaw هذه الأدوات افتراضيًا
        // على الجلسة الحالية + جلسات الوكلاء الفرعيين التي جرى إنشاؤها، لكن يمكنك التشديد أكثر عند الحاجة.
        // راجع `tools.sessions.visibility` في مرجع الإعدادات.
        tools: {
          sessions: { visibility: "tree" }, // self | tree | agent | all
          allow: [
            "sessions_list",
            "sessions_history",
            "sessions_send",
            "sessions_spawn",
            "session_status",
            "whatsapp",
            "telegram",
            "slack",
            "discord",
          ],
          deny: [
            "read",
            "write",
            "edit",
            "apply_patch",
            "exec",
            "process",
            "browser",
            "canvas",
            "nodes",
            "cron",
            "gateway",
            "image",
          ],
        },
      },
    ],
  },
}
```

## الاستجابة للحوادث

إذا فعل AI الخاص بك شيئًا سيئًا:

### الاحتواء

1. **أوقفه:** أوقف تطبيق macOS ‏(إذا كان يشرف على Gateway) أو أنهِ عملية `openclaw gateway`.
2. **أغلق التعرض:** اضبط `gateway.bind: "loopback"` ‏(أو عطّل Tailscale Funnel/Serve) إلى أن تفهم ما الذي حدث.
3. **جمّد الوصول:** حوّل الرسائل الخاصة/المجموعات الخطرة إلى `dmPolicy: "disabled"` / اشترط الإشارات، وأزل إدخالات `"*"` التي تسمح للجميع إذا كنت قد استخدمتها.

### التدوير (افترض وجود اختراق إذا تسرّبت الأسرار)

1. دوّر مصادقة Gateway ‏(`gateway.auth.token` / `OPENCLAW_GATEWAY_PASSWORD`) وأعد التشغيل.
2. دوّر أسرار العملاء البعيدين (`gateway.remote.token` / `.password`) على أي جهاز يمكنه استدعاء Gateway.
3. دوّر بيانات اعتماد provider/API ‏(بيانات اعتماد WhatsApp، وtokens الخاصة بـ Slack/Discord، وModels/API keys في `auth-profiles.json`، وقيم حمولة الأسرار المشفرة عند استخدامها).

### التدقيق

1. تحقّق من سجلات Gateway: ‏`/tmp/openclaw/openclaw-YYYY-MM-DD.log` ‏(أو `logging.file`).
2. راجع transcript(s) ذات الصلة: ‏`~/.openclaw/agents/<agentId>/sessions/*.jsonl`.
3. راجع تغييرات الإعدادات الحديثة (أي شيء قد يكون وسّع الوصول: `gateway.bind`، و`gateway.auth`، وسياسات الرسائل الخاصة/المجموعات، و`tools.elevated`، وتغييرات Plugin).
4. أعد تشغيل `openclaw security audit --deep` وتأكد من حل النتائج الحرجة.

### اجمع ما يلزم للتقرير

- الطابع الزمني، ونظام تشغيل مضيف gateway + إصدار OpenClaw
- transcript(s) الخاصة بالجلسة + ذيل سجل قصير (بعد التنقيح)
- ما الذي أرسله المهاجم + ما الذي فعله الوكيل
- ما إذا كان Gateway معرّضًا خارج loopback ‏(LAN/Tailscale Funnel/Serve)

## فحص الأسرار باستخدام detect-secrets

تشغّل CI خطاف `detect-secrets` الخاص بـ pre-commit في مهمة `secrets`.
وتشغّل عمليات الدفع إلى `main` دائمًا فحصًا لكل الملفات. أما طلبات السحب فتستخدم
مسارًا سريعًا يعتمد على الملفات المتغيرة عندما يكون commit أساسي متاحًا، وتعود إلى فحص كل الملفات
في غير ذلك. وإذا فشل الفحص، فهذا يعني وجود مرشحين جدد لم يدخلوا بعد في baseline.

### إذا فشلت CI

1. أعِد إنتاج المشكلة محليًا:

   ```bash
   pre-commit run --all-files detect-secrets
   ```

2. افهم الأدوات:
   - يشغّل `detect-secrets` داخل pre-commit الأمر `detect-secrets-hook` مع
     baseline والاستثناءات الخاصة بالمستودع.
   - يفتح `detect-secrets audit` مراجعة تفاعلية لوضع علامة على كل عنصر في baseline
     على أنه حقيقي أو نتيجة إيجابية كاذبة.
3. بالنسبة إلى الأسرار الحقيقية: دوّرها/أزلها، ثم أعد تشغيل الفحص لتحديث baseline.
4. بالنسبة إلى النتائج الإيجابية الكاذبة: شغّل التدقيق التفاعلي وضع علامة عليها على أنها كاذبة:

   ```bash
   detect-secrets audit .secrets.baseline
   ```

5. إذا كنت تحتاج إلى استثناءات جديدة، فأضفها إلى `.detect-secrets.cfg` وأعد توليد
   baseline مع أعلام `--exclude-files` / `--exclude-lines` المطابقة (ملف
   الإعدادات مرجعي فقط؛ لا يقرؤه detect-secrets تلقائيًا).

قم بعمل commit للملف `.secrets.baseline` المحدَّث بمجرد أن يعكس الحالة المقصودة.
