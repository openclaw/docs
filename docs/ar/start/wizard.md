```bash
openclaw onboard
```

يُعد الإعداد الأولي عبر CLI مسار الإعداد الطرفي الموصى به على macOS وLinux وWindows (أصليًا أو عبر WSL2). يكتشف افتراضيًا إمكانية الوصول إلى الذكاء الاصطناعي المتاحة بالفعل على الجهاز، ويتحقق منها عبر عملية إكمال فعلية، ثم يبدأ Crestodian لإعداد مساحة العمل وGateway والميزات الاختيارية. يشغّل `openclaw setup` التدفق نفسه (تغطي [الإعداد](/ar/cli/setup) صيغة `--baseline` التي تقتصر على الإعدادات). ويمكن لمستخدمي سطح مكتب Windows أيضًا البدء من [مركز Windows](/ar/platforms/windows).

يبدأ الإعداد الأولي الموجّه بتهيئة الاستدلال. إذ يكتشف إمكانية الوصول المتاحة إلى الذكاء الاصطناعي، ويتطلب عملية إكمال فعلية، وبعد ذلك فقط يبدأ [Crestodian](/ar/cli/crestodian) لإعداد بقية OpenClaw. لا يتوفر في التدفق الموجّه تشغيل Crestodian قبل الاستدلال أو مسار لتخطي الذكاء الاصطناعي.

يظل المعالج التقليدي متاحًا لتسجيل الدخول إلى المزوّد، وإعداد Gateway بعيد، وإقران القنوات، وعناصر التحكم في الخدمة الخفية، والمهارات، وعمليات الاستيراد. شغّله صراحةً باستخدام `openclaw onboard --classic`؛ فشاشة مرشحي الاستدلال الموجّه لا تفوّض إليه. بعد نجاح الاستدلال، يمكن لـCrestodian استخدام `open channel wizard for <channel>` لتسليم إعداد القناة الذي يتطلب أسرارًا إلى معالج طرفي يخفي المدخلات. لتغيير مزوّد النموذج أو مصادقته، اخرج من Crestodian وشغّل `openclaw onboard`؛ لا يفتح Crestodian تدفقات المزوّد الموجّهة أو التقليدية.

<Info>
لأسرع محادثة أولى: أكمل الإعداد الموجّه، وشغّل `openclaw dashboard`، ثم تحدث في المتصفح عبر واجهة التحكم. الوثائق: [لوحة المعلومات](/ar/web/dashboard).
</Info>

## اللغة والمنطقة

يترجم المعالج نصوص الإعداد الأولي الثابتة. ترتيب تحديد اللغة: `OPENCLAW_LOCALE`، ثم `LC_ALL`، ثم `LC_MESSAGES`، ثم `LANG`، ثم الإنجليزية. اللغات المدعومة: `en` و`zh-CN` و`zh-TW`.

```bash
OPENCLAW_LOCALE=zh-CN openclaw onboard
```

تظل أسماء المنتجات والأوامر ومفاتيح الإعداد وعناوين URL ومعرّفات المزوّدين ومعرّفات النماذج وتسميات الإضافات والقنوات باللغة الإنجليزية بغض النظر عن اللغة والمنطقة.

لإعادة إعداد الخيارات غير المتعلقة بالاستدلال لاحقًا:

```bash
openclaw configure
openclaw agents add <name>
```

<Note>
لا يعني `--json` تشغيل الوضع غير التفاعلي. بالنسبة إلى البرامج النصية، استخدم `--non-interactive` (راجع [أتمتة CLI](/ar/start/wizard-cli-automation)).
</Note>

<Tip>
يتضمن المعالج التقليدي خطوة للبحث على الويب يمكنك فيها اختيار مزوّد: Brave أو DuckDuckGo أو Exa أو Firecrawl أو Gemini أو Grok أو Kimi أو MiniMax Search أو Ollama Web Search أو Perplexity أو SearXNG أو Tavily. يتطلب بعضها مفتاح API، بينما لا يتطلبه البعض الآخر. اضبط ذلك لاحقًا باستخدام `openclaw configure --section web`. الوثائق: [أدوات الويب](/ar/tools/web).
</Tip>

## التدفق الموجّه الافتراضي

يتبع الأمر `openclaw onboard` دون خيارات هذا المسار:

1. اقبل إشعار الأمان.
2. اكتشف النماذج المضبوطة ومتغيرات البيئة الخاصة بمفاتيح API وواجهات CLI المحلية المدعومة للذكاء الاصطناعي.
3. اختبر أول مرشح مكتشف عبر عملية إكمال فعلية. عند الفشل، اعرض السبب وتابع إلى المرشح التالي القابل للاستخدام.
4. إذا استُنفدت نتائج الاكتشاف، فأعد محاولة مرشح مكتشف أو أدخل مفتاح API لمزوّد في مطالبة تخفي المدخلات. لا يتيح الإعداد الأولي الموجّه Crestodian أو مخرجًا لتخطي الذكاء الاصطناعي قبل نجاح الاستدلال.
5. احفظ فقط مسار النموذج المتحقق منه وأي حالة لبيانات الاعتماد أو الإضافات يتطلبها. تظل إعدادات مساحة العمل وGateway دون تغيير.
6. ابدأ Crestodian باستخدام النموذج المتحقق منه ليتمكن من إعداد مساحة العمل وGateway والقنوات والوكلاء والإضافات وبقية الإعداد الاختياري.

عند إعادة تشغيل الأمر على تثبيت مضبوط، يُختبر النموذج الافتراضي الحالي أولًا، ما يجعل التدفق الموجّه عملية تحقق وإصلاح. لا يستبدل الاختبار الفاشل النموذج المضبوط تلقائيًا أبدًا؛ بل يتوقف الإعداد الأولي ويسأل عن كيفية المتابعة. شغّل `openclaw channels add` أو `openclaw configure` للإضافات اللاحقة غير المتعلقة بالاستدلال؛ واستخدم `openclaw onboard` لتغيير مسارات المزوّد أو المصادقة.

## المعالج التقليدي: البدء السريع مقابل المتقدم

شغّل `openclaw onboard --classic` لفتح المعالج الكامل. يبدأ بخيار بين **البدء السريع** (الإعدادات الافتراضية) و**المتقدم** (التحكم الكامل). مرّر `--flow quickstart` أو `--flow advanced` (واسمه البديل `manual`) لاختيار التدفق التقليدي وتخطي تلك المطالبة.

<Tabs>
  <Tab title="QuickStart (defaults)">
    - Gateway محلي، وربط بعنوان loopback
    - مساحة العمل الافتراضية (أو مساحة العمل الحالية)
    - منفذ Gateway‏ **18789**
    - مصادقة Gateway باستخدام **رمز مميز** (يُنشأ تلقائيًا، حتى مع loopback)
    - سياسة الأدوات: `tools.profile: "coding"` لعمليات الإعداد الجديدة (يُحتفظ بأي ملف تعريف صريح حالي)
    - عزل الرسائل المباشرة: `session.dmScope: "per-channel-peer"` لعمليات الإعداد الجديدة. التفاصيل: [مرجع إعداد CLI](/ar/start/wizard-cli-reference#outputs-and-internals)
    - إتاحة Tailscale‏ **متوقفة**
    - تستخدم رسائل Telegram وWhatsApp المباشرة **قائمة السماح** افتراضيًا: يطلب Telegram معرّف مستخدم Telegram رقميًا، ويطلب WhatsApp رقم هاتف

  </Tab>
  <Tab title="Advanced (full control)">
    - يعرض كل خطوة: الوضع، ومساحة العمل، وGateway، والقنوات، والخدمة الخفية، والمهارات

  </Tab>
</Tabs>

يستخدم الوضع البعيد (`--mode remote`) التدفق المتقدم دائمًا؛ فهو لا يفعل سوى إعداد هذا الجهاز للاتصال بـGateway موجود في مكان آخر، ولا يثبت أو يغير أي شيء على المضيف البعيد.

## ما يضبطه الإعداد الأولي التقليدي

يمر الوضع المحلي (الافتراضي) بهذه الخطوات:

1. **النموذج/المصادقة** - اختر تدفق مصادقة للمزوّد (مفتاح API أو OAuth أو مصادقة يدوية خاصة بالمزوّد)، بما في ذلك مزوّد مخصص (متوافق مع OpenAI أو OpenAI Responses أو Anthropic، أو اكتشاف تلقائي غير معروف). اختر نموذجًا افتراضيًا.
   تستخدم عمليات إعداد مفتاح OpenAI API الجديدة النموذج `openai/gpt-5.6` افتراضيًا (يُحل معرّف API المباشر المجرّد إلى Sol)؛ وتستخدم عمليات إعداد ChatGPT/Codex الجديدة النموذج `openai/gpt-5.6-sol` افتراضيًا. تحتفظ إعادة تشغيل الإعداد بالنموذج الصريح الحالي، بما في ذلك `openai/gpt-5.5`. اختر `openai/gpt-5.5` صراحةً إذا كان الحساب لا يتيح GPT-5.6.
   ملاحظة أمنية: إذا كان هذا الوكيل سيشغّل أدوات أو يعالج محتوى Webhook أو الخطافات، ففضّل أقوى نموذج متاح من أحدث جيل، وأبقِ سياسة الأدوات صارمة؛ فالطبقات الأضعف أو الأقدم أكثر عرضة لحقن المطالبات.
   في عمليات التشغيل غير التفاعلية، يخزّن `--secret-input-mode ref` مراجع مدعومة بمتغيرات البيئة بدلًا من قيم مفاتيح API بنص صريح؛ ويجب أن يكون متغير البيئة المشار إليه مضبوطًا مسبقًا، وإلا يفشل الإعداد الأولي فورًا. يمكن لوضع المراجع السرية التفاعلي الإشارة إلى متغير بيئة أو مرجع مزوّد مضبوط (`file` أو `exec`)، مع فحص تمهيدي سريع قبل الحفظ. بعد إعداد النموذج والمصادقة، يعرض المعالج اختبار إكمال فعليًا اختياريًا؛ ويمكن عند الفشل الرجوع إلى إعداد النموذج والمصادقة مرة واحدة أو تجاهل الفشل دون منع بقية المعالج التقليدي. لا يؤدي تجاهله إلى إتاحة Crestodian؛ فلا يزال الإعداد الحواري يتطلب اجتياز فحص الاستدلال.
2. **مساحة العمل** - دليل ملفات الوكيل (الافتراضي `~/.openclaw/workspace`). ويُنشئ ملفات التمهيد الأولية.
3. **Gateway** - المنفذ، وعنوان الربط، ووضع المصادقة، وإتاحة Tailscale. في وضع الرمز المميز التفاعلي، اختر تخزين الرمز المميز بنص صريح (الافتراضي) أو اشترك في استخدام SecretRef. مسار SecretRef غير التفاعلي: `--gateway-token-ref-env <ENV_VAR>`.
4. **القنوات** - قنوات الدردشة المضمنة وقنوات الإضافات الرسمية، بما في ذلك Discord وFeishu وGoogle Chat وiMessage وMattermost وMicrosoft Teams وQQ Bot وSignal وSlack وTelegram وWhatsApp وغيرها.
5. **الخدمة الخفية** - يثبت LaunchAgent على macOS، أو وحدة مستخدم systemd على Linux/WSL2، أو مهمة Windows مجدولة أصلية مع مسار احتياطي في مجلد بدء التشغيل لكل مستخدم.
   إذا كانت مصادقة الرمز المميز مطلوبة وكانت `gateway.auth.token` مُدارة عبر SecretRef، فإن تثبيت الخدمة الخفية يتحقق منها لكنه لا يحفظ رمزًا مميزًا محلولًا في بيانات بيئة خدمة المشرف؛ ويمنع SecretRef غير القابل للحل التثبيت مع عرض إرشادات. إذا كان كل من `gateway.auth.token` و`gateway.auth.password` مضبوطًا بينما `gateway.auth.mode` غير مضبوط، فسيُمنع التثبيت حتى تضبط الوضع صراحةً.
6. **فحص السلامة** - يبدأ Gateway ويتحقق من إمكانية الوصول إليه.
7. **Skills** - يثبت المهارات الموصى بها واعتمادياتها الاختيارية.

<Note>
لا تؤدي إعادة تشغيل الإعداد الأولي إلى مسح أي شيء **إلا** إذا اخترت **إعادة الضبط** صراحةً (أو مرّرت `--reset`). يعيد خيار CLI‏ `--reset` ضبط الإعدادات وبيانات الاعتماد والجلسات افتراضيًا؛ واستخدم `--reset-scope full` لإزالة مساحة العمل أيضًا. إذا كانت الإعدادات غير صالحة أو تحتوي على مفاتيح قديمة، فسيطلب منك الإعداد الأولي تشغيل `openclaw doctor` أولًا.
</Note>

يشغّل `--flow import` تدفق ترحيل مكتشفًا (مثل Hermes) داخل المعالج التقليدي بدلًا من إعداد جديد؛ راجع [الترحيل](/ar/cli/migrate) وأدلة الترحيل ضمن [التثبيت](/ar/install/migrating-hermes). يُعد `openclaw onboard --modern` اسمًا بديلًا للتوافق مع [Crestodian](/ar/cli/crestodian). ويستخدم بوابة الاستدلال نفسها التي يستخدمها `openclaw crestodian`: يبدأ الاستدلال المتحقق منه المساعد، بينما يعيد الفشل التفاعلي المستخدم إلى إعداد الاستدلال الموجّه.

## إضافة وكيل آخر

استخدم `openclaw agents add <name>` لإنشاء وكيل منفصل له مساحة عمل وجلسات وملفات تعريف مصادقة خاصة به. يؤدي التشغيل دون `--workspace` إلى بدء تدفق تفاعلي للاسم ومساحة العمل والمصادقة والقنوات والروابط؛ وهو ليس معالج `openclaw onboard` الكامل.

ما يضبطه:

- `agents.list[].name`
- `agents.list[].workspace`
- `agents.list[].agentDir`

ملاحظات:

- مساحة العمل الافتراضية: `~/.openclaw/workspace-<agentId>` (أو ضمن `agents.defaults.workspace` إذا كانت مضبوطة).
- أضف `bindings` لتوجيه الرسائل الواردة إلى هذا الوكيل (يمكن للإعداد الأولي فعل ذلك نيابةً عنك).
- خيارات الوضع غير التفاعلي: `--model` و`--agent-dir` و`--bind` و`--non-interactive`.
