لدى OpenClaw طبقتا بث منفصلتان:

- **البث الكتلي (القنوات):** إصدار **كتل** مكتملة أثناء كتابة المساعد. هذه رسائل قنوات عادية (وليست فروق رموز).
- **بث المعاينة (Telegram/Discord/Slack):** تحديث **رسالة معاينة** مؤقتة أثناء التوليد.

لا يوجد **بث حقيقي لفروق الرموز** إلى رسائل القنوات اليوم. بث المعاينة قائم على الرسائل (إرسال + تعديلات/إلحاقات).

## البث الكتلي (رسائل القنوات)

يرسل البث الكتلي مخرجات المساعد في مقاطع كبيرة نسبياً فور توفرها.

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

المفتاح:

- `text_delta/events`: أحداث تدفق النموذج (قد تكون متباعدة للنماذج غير الباثة).
- `chunker`: يطبق `EmbeddedBlockChunker` حدود الحد الأدنى/الأقصى + تفضيل الفاصل.
- `channel send`: الرسائل الصادرة الفعلية (ردود الكتل).

**عناصر التحكم:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (الافتراضي إيقاف).
- تجاوزات القنوات: `*.blockStreaming` (ومتغيرات كل حساب) لفرض `"on"`/`"off"` لكل قناة.
- `agents.defaults.blockStreamingBreak`: `"text_end"` أو `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (دمج الكتل المبثوثة قبل الإرسال).
- الحد الأقصى الصارم للقناة: `*.textChunkLimit` (مثلاً، `channels.whatsapp.textChunkLimit`).
- وضع تقسيم القناة: `*.chunkMode` (`length` افتراضي، و`newline` يقسم عند الأسطر الفارغة (حدود الفقرات) قبل التقسيم حسب الطول).
- الحد المرن في Discord: `channels.discord.maxLinesPerMessage` (الافتراضي 17) يقسم الردود الطويلة لتجنب قص الواجهة.

**دلالات الحدود:**

- `text_end`: بث الكتل فور إصدارها من المقسم؛ التفريغ عند كل `text_end`.
- `message_end`: الانتظار حتى تنتهي رسالة المساعد، ثم تفريغ المخرجات المخزنة.

ما يزال `message_end` يستخدم المقسم إذا تجاوز النص المخزن `maxChars`، لذا يمكنه إصدار عدة مقاطع في النهاية.

### تسليم الوسائط مع البث الكتلي

توجيهات `MEDIA:` هي بيانات وصفية عادية للتسليم. عندما يرسل البث الكتلي كتلة وسائط مبكراً، يتذكر OpenClaw ذلك التسليم للدور. إذا كررت حمولة المساعد النهائية عنوان URL نفسه للوسائط، يزيل التسليم النهائي الوسائط المكررة بدلاً من إرسال المرفق مرة أخرى.

تُحجب الحمولات النهائية المطابقة تماماً. إذا أضافت الحمولة النهائية نصاً مميزاً حول وسائط بُثت مسبقاً، يظل OpenClaw يرسل النص الجديد مع إبقاء الوسائط بتسليم واحد. يمنع هذا تكرار الملاحظات الصوتية أو الملفات على قنوات مثل Telegram عندما يصدر وكيل `MEDIA:` أثناء البث ويضمنها المزود أيضاً في الرد المكتمل.

## خوارزمية التقسيم (حدود منخفضة/عالية)

ينفذ `EmbeddedBlockChunker` التقسيم الكتلي:

- **الحد المنخفض:** لا يصدر حتى يصبح المخزن >= `minChars` (ما لم يُفرض ذلك).
- **الحد العالي:** يفضل التقسيم قبل `maxChars`؛ إذا فُرض، يقسم عند `maxChars`.
- **تفضيل الفاصل:** `paragraph` → `newline` → `sentence` → `whitespace` → فاصل قسري.
- **أسوار الكود:** لا يقسم أبداً داخل الأسوار؛ عند الفرض عند `maxChars`، يغلق السور ثم يعيد فتحه للحفاظ على صلاحية Markdown.

يُقيَّد `maxChars` بحد `textChunkLimit` الخاص بالقناة، لذلك لا يمكنك تجاوز الحدود القصوى لكل قناة.

## الدمج (دمج الكتل المبثوثة)

عند تمكين البث الكتلي، يمكن لـ OpenClaw **دمج مقاطع كتل متتالية** قبل إرسالها. يقلل هذا من «إغراق الأسطر المفردة» مع استمرار توفير مخرجات تدريجية.

- ينتظر الدمج **فجوات خمول** (`idleMs`) قبل التفريغ.
- تُقيَّد المخازن بـ `maxChars` وستُفرَّغ إذا تجاوزته.
- يمنع `minChars` إرسال الشذرات الصغيرة حتى يتراكم نص كافٍ (يرسل التفريغ النهائي النص المتبقي دائماً).
- يُشتق الفاصل من `blockStreamingChunk.breakPreference` (`paragraph` → `\n\n`، و`newline` → `\n`، و`sentence` → مسافة).
- تتوفر تجاوزات القنوات عبر `*.blockStreamingCoalesce` (بما في ذلك إعدادات كل حساب).
- يُرفع `minChars` الافتراضي للدمج إلى 1500 لـ Signal/Slack/Discord ما لم يُتجاوز.

## إيقاع يشبه البشر بين الكتل

عند تمكين البث الكتلي، يمكنك إضافة **وقفة عشوائية** بين ردود الكتل (بعد الكتلة الأولى). يجعل هذا الردود متعددة الفقاعات تبدو طبيعية أكثر.

- الإعداد: `agents.defaults.humanDelay` (تجاوز لكل وكيل عبر `agents.list[].humanDelay`).
- الأوضاع: `off` (افتراضي)، و`natural` (800–2500ms)، و`custom` (`minMs`/`maxMs`).
- ينطبق فقط على **ردود الكتل**، وليس الردود النهائية أو ملخصات الأدوات.

## "بث المقاطع أو كل شيء"

يناظر هذا:

- **بث المقاطع:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (الإصدار أثناء التقدم). تحتاج القنوات غير Telegram أيضاً إلى `*.blockStreaming: true`.
- **بث كل شيء في النهاية:** `blockStreamingBreak: "message_end"` (تفريغ مرة واحدة، وربما عدة مقاطع إذا كان طويلاً جداً).
- **لا بث كتلي:** `blockStreamingDefault: "off"` (الرد النهائي فقط).

**ملاحظة القناة:** يكون البث الكتلي **متوقفاً ما لم** يُضبط `*.blockStreaming` صراحة على `true`. يمكن للقنوات بث معاينة مباشرة (`channels.<channel>.streaming`) دون ردود كتل.

تذكير بموقع الإعداد: تقع إعدادات `blockStreaming*` الافتراضية تحت `agents.defaults`، وليس في إعداد الجذر.

## أوضاع بث المعاينة

المفتاح القياسي: `channels.<channel>.streaming`

الأوضاع:

- `off`: تعطيل بث المعاينة.
- `partial`: معاينة واحدة تُستبدل بأحدث نص.
- `block`: تحديثات معاينة بخطوات مقسمة/ملحقة.
- `progress`: معاينة تقدم/حالة أثناء التوليد، والجواب النهائي عند الاكتمال.

### تعيين القنوات

| القناة    | `off` | `partial` | `block` | `progress`        |
| ---------- | ----- | --------- | ------- | ----------------- |
| Telegram   | ✅    | ✅        | ✅      | يناظر `partial` |
| Discord    | ✅    | ✅        | ✅      | يناظر `partial` |
| Slack      | ✅    | ✅        | ✅      | ✅                |
| Mattermost | ✅    | ✅        | ✅      | ✅                |

Slack فقط:

- يبدل `channels.slack.streaming.nativeTransport` استدعاءات واجهة Slack الأصلية للبث عندما يكون `channels.slack.streaming.mode="partial"` (الافتراضي: `true`).
- يتطلب بث Slack الأصلي وحالة سلسلة مساعد Slack هدف سلسلة ردود؛ لا تعرض الرسائل المباشرة عالية المستوى تلك المعاينة بأسلوب السلسلة.

ترحيل المفاتيح القديمة:

- Telegram: تُكتشف قيم `streamMode` القديمة وقيم `streaming` العددية/المنطقية وتُرحَّل عبر مسارات توافق الطبيب/الإعداد إلى `streaming.mode`.
- Discord: يُرحَّل `streamMode` + `streaming` المنطقي تلقائياً إلى تعداد `streaming`.
- Slack: يُرحَّل `streamMode` تلقائياً إلى `streaming.mode`؛ ويُرحَّل `streaming` المنطقي تلقائياً إلى `streaming.mode` بالإضافة إلى `streaming.nativeTransport`؛ ويُرحَّل `nativeStreaming` القديم تلقائياً إلى `streaming.nativeTransport`.

### سلوك وقت التشغيل

Telegram:

- يستخدم تحديثات معاينة `sendMessage` + `editMessageText` عبر الرسائل المباشرة والمجموعات/المواضيع.
- يرسل رسالة نهائية جديدة بدلاً من التعديل في المكان عندما تكون المعاينة ظاهرة لنحو دقيقة واحدة، ثم ينظف المعاينة حتى يعكس الطابع الزمني في Telegram اكتمال الرد.
- يُتخطى بث المعاينة عندما يكون البث الكتلي في Telegram ممكناً صراحة (لتجنب البث المزدوج).
- يمكن لـ `/reasoning stream` كتابة الاستدلال إلى المعاينة.

Discord:

- يستخدم رسائل معاينة بالإرسال + التعديل.
- يستخدم وضع `block` تقسيم المسودة (`draftChunk`).
- يُتخطى بث المعاينة عندما يكون البث الكتلي في Discord ممكناً صراحة.
- تلغي حمولات الوسائط النهائية والأخطاء والردود الصريحة المعاينات المعلقة دون تفريغ مسودة جديدة، ثم تستخدم التسليم العادي.

Slack:

- يمكن أن يستخدم `partial` بث Slack الأصلي (`chat.startStream`/`append`/`stop`) عند توفره.
- يستخدم `block` معاينات مسودة بأسلوب الإلحاق.
- يستخدم `progress` نص معاينة الحالة، ثم الجواب النهائي.
- يحجب بث المعاينة الأصلي وبث مسودة المعاينة ردود الكتل لذلك الدور، بحيث يُبث رد Slack عبر مسار تسليم واحد فقط.
- لا تنشئ حمولات الوسائط/الأخطاء النهائية والنهايات المرحلية رسائل مسودة مؤقتة؛ فقط نهايات النص/الكتل التي يمكنها تعديل المعاينة تفرغ نص المسودة المعلق.

Mattermost:

- يبث التفكير ونشاط الأدوات ونص الرد الجزئي في منشور معاينة مسودة واحد يُنهى في مكانه عندما يصبح الجواب النهائي آمناً للإرسال.
- يعود إلى إرسال منشور نهائي جديد إذا حُذف منشور المعاينة أو كان غير متاح لسبب آخر وقت الإنهاء.
- تلغي حمولات الوسائط/الأخطاء النهائية تحديثات المعاينة المعلقة قبل التسليم العادي بدلاً من تفريغ منشور معاينة مؤقت.

Matrix:

- تُنهى معاينات المسودة في مكانها عندما يستطيع النص النهائي إعادة استخدام حدث المعاينة.
- تلغي نهايات الوسائط فقط والأخطاء وعدم تطابق هدف الرد تحديثات المعاينة المعلقة قبل التسليم العادي؛ وتُنقح أي معاينة قديمة ظاهرة بالفعل.

### تحديثات معاينة تقدم الأدوات

يمكن أن يتضمن بث المعاينة أيضاً تحديثات **تقدم الأدوات** — أسطر حالة قصيرة مثل "البحث في الويب"، أو "قراءة ملف"، أو "استدعاء أداة" — تظهر في رسالة المعاينة نفسها أثناء تشغيل الأدوات، قبل الرد النهائي. يحافظ هذا على حيوية أدوار الأدوات متعددة الخطوات بصرياً بدلاً من الصمت بين معاينة التفكير الأولى والجواب النهائي.

الأسطح المدعومة:

- تبث **Discord** و**Slack** و**Telegram** و**Matrix** تقدم الأدوات إلى تعديل المعاينة المباشر افتراضياً عندما يكون بث المعاينة نشطاً.
- صدر Telegram مع تمكين تحديثات معاينة تقدم الأدوات منذ `v2026.4.22`؛ وإبقاؤها مفعلة يحافظ على ذلك السلوك الصادر.
- يدمج **Mattermost** نشاط الأدوات بالفعل في منشور معاينة المسودة الواحد لديه (انظر أعلاه).
- تتبع تعديلات تقدم الأدوات وضع بث المعاينة النشط؛ وتُتخطى عندما يكون بث المعاينة `off` أو عندما يتولى البث الكتلي الرسالة. في Telegram، يكون `streaming.mode: "off"` للنهائي فقط: تُحجب أيضاً محادثات التقدم العامة بدلاً من تسليمها كرسائل مستقلة "جارٍ العمل..."، بينما تستمر مطالبات الموافقة وحمولات الوسائط والأخطاء في المسار العادي.
- للإبقاء على بث المعاينة مع إخفاء أسطر تقدم الأدوات، اضبط `streaming.preview.toolProgress` على `false` لتلك القناة. لتعطيل تعديلات المعاينة بالكامل، اضبط `streaming.mode` على `off`.

مثال:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": false
        }
      }
    }
  }
}
```
