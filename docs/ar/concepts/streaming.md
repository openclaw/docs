لدى OpenClaw طبقتا بث منفصلتان:

- **بث الكتل (القنوات):** يرسل **كتلًا** مكتملة أثناء كتابة المساعد. هذه رسائل قناة عادية (وليست فروقات رموز).
- **بث المعاينة (Telegram/Discord/Slack):** يحدّث **رسالة معاينة** مؤقتة أثناء التوليد.

لا يوجد **بث حقيقي لفروقات الرموز** إلى رسائل القنوات اليوم. بث المعاينة قائم على الرسائل (إرسال + تعديلات/إلحاقات).

## بث الكتل (رسائل القنوات)

يرسل بث الكتل مخرجات المساعد في أجزاء كبيرة نسبيًا عندما تصبح متاحة.

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

وسيلة الإيضاح:

- `text_delta/events`: أحداث بث النموذج (قد تكون متفرقة في النماذج غير الداعمة للبث).
- `chunker`: يطبّق `EmbeddedBlockChunker` حدود الحد الأدنى/الأقصى + تفضيل الفاصل.
- `channel send`: الرسائل الصادرة الفعلية (ردود الكتل).

**عناصر التحكم:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (الافتراضي إيقاف).
- تجاوزات القنوات: `*.blockStreaming` (ومتغيرات لكل حساب) لفرض `"on"`/`"off"` لكل قناة.
- `agents.defaults.blockStreamingBreak`: `"text_end"` أو `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (دمج الكتل المبثوثة قبل الإرسال).
- الحد الصارم للقناة: `*.textChunkLimit` (مثل `channels.whatsapp.textChunkLimit`).
- وضع تجزئة القناة: `*.chunkMode` (`length` افتراضيًا، و`newline` يقسم عند الأسطر الفارغة (حدود الفقرات) قبل التجزئة حسب الطول).
- الحد المرن في Discord: `channels.discord.maxLinesPerMessage` (افتراضيًا 17) يقسم الردود الطويلة لتجنب قص الواجهة.

**دلالات الحدود:**

- `text_end`: بث الكتل فور أن يصدرها المجزّئ؛ التفريغ عند كل `text_end`.
- `message_end`: الانتظار حتى تنتهي رسالة المساعد، ثم تفريغ المخرجات المخزنة مؤقتًا.

لا يزال `message_end` يستخدم المجزّئ إذا تجاوز النص المخزن `maxChars`، لذلك يمكنه إصدار عدة أجزاء في النهاية.

### تسليم الوسائط مع بث الكتل

يجب أن تستخدم وسائط البث حقول حمولة منظمة مثل `mediaUrl` أو
`mediaUrls`؛ ولا يُحلَّل النص المبثوث كأمر مرفق. عندما يرسل بث
الكتل الوسائط مبكرًا، يتذكر OpenClaw ذلك التسليم للدورة. إذا
كررت حمولة المساعد النهائية عنوان URL الوسيطي نفسه، فإن التسليم النهائي
يزيل الوسائط المكررة بدلًا من إرسال المرفق مرة أخرى.

تُمنع الحمولات النهائية المتطابقة تمامًا. إذا أضافت الحمولة النهائية
نصًا مميزًا حول وسائط سبق بثها، لا يزال OpenClaw يرسل
النص الجديد مع إبقاء الوسائط بتسليم واحد. يمنع هذا تكرار الملاحظات
الصوتية أو الملفات على قنوات مثل Telegram.

## خوارزمية التجزئة (حدود دنيا/عليا)

ينفّذ `EmbeddedBlockChunker` تجزئة الكتل:

- **الحد الأدنى:** لا تصدر حتى يكون المخزن المؤقت >= `minChars` (ما لم يكن ذلك إجباريًا).
- **الحد الأعلى:** فضّل التقسيم قبل `maxChars`؛ وإذا كان إجباريًا، فاقسم عند `maxChars`.
- **تفضيل الفاصل:** `paragraph` ← `newline` ← `sentence` ← `whitespace` ← فاصل قسري.
- **أسوار الشيفرة:** لا تقسّم أبدًا داخل الأسوار؛ وعند التقسيم الإجباري عند `maxChars`، أغلق السور وأعد فتحه للحفاظ على صلاحية Markdown.

يُقيَّد `maxChars` بحد `textChunkLimit` الخاص بالقناة، لذلك لا يمكنك تجاوز حدود كل قناة.

## الدمج (دمج الكتل المبثوثة)

عند تمكين بث الكتل، يستطيع OpenClaw **دمج أجزاء الكتل المتتالية**
قبل إرسالها. يقلل هذا "إزعاج الأسطر المفردة" مع الاستمرار في توفير
مخرجات تدريجية.

- ينتظر الدمج **فجوات الخمول** (`idleMs`) قبل التفريغ.
- تُحدّ المخازن المؤقتة بواسطة `maxChars` وستُفرَّغ إذا تجاوزته.
- يمنع `minChars` إرسال الشظايا الصغيرة حتى يتراكم نص كافٍ
  (يرسل التفريغ النهائي النص المتبقي دائمًا).
- يُشتق الموصل من `blockStreamingChunk.breakPreference`
  (`paragraph` ← `\n\n`، و`newline` ← `\n`، و`sentence` ← مسافة).
- تتوفر تجاوزات القنوات عبر `*.blockStreamingCoalesce` (بما في ذلك إعدادات كل حساب).
- يُرفع `minChars` الافتراضي للدمج إلى 1500 في Signal/Slack/Discord ما لم يُتجاوز.

## إيقاع شبيه بالبشر بين الكتل

عند تمكين بث الكتل، يمكنك إضافة **توقف عشوائي** بين
ردود الكتل (بعد الكتلة الأولى). يجعل هذا الاستجابات متعددة الفقاعات تبدو
أكثر طبيعية.

- الإعداد: `agents.defaults.humanDelay` (تجاوزه لكل وكيل عبر `agents.list[].humanDelay`).
- الأوضاع: `off` (افتراضيًا)، و`natural` (800-2500ms)، و`custom` (`minMs`/`maxMs`).
- ينطبق فقط على **ردود الكتل**، وليس الردود النهائية أو ملخصات الأدوات.

## "بث الأجزاء أو كل شيء"

يرتبط هذا بما يلي:

- **بث الأجزاء:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (الإصدار أثناء التقدم). تحتاج القنوات غير Telegram أيضًا إلى `*.blockStreaming: true`.
- **بث كل شيء في النهاية:** `blockStreamingBreak: "message_end"` (تفريغ مرة واحدة، وربما عدة أجزاء إذا كان طويلًا جدًا).
- **لا بث للكتل:** `blockStreamingDefault: "off"` (الرد النهائي فقط).

**ملاحظة القناة:** يكون بث الكتل **متوقفًا ما لم**
يُضبط `*.blockStreaming` صراحةً على `true`. يمكن للقنوات بث معاينة حية
(`channels.<channel>.streaming`) دون ردود كتل.

تذكير بموقع الإعدادات: تعيش افتراضيات `blockStreaming*` تحت
`agents.defaults`، وليس في الإعداد الجذر.

## أوضاع بث المعاينة

المفتاح القانوني: `channels.<channel>.streaming`

الأوضاع:

- `off`: تعطيل بث المعاينة.
- `partial`: معاينة واحدة تُستبدل بأحدث نص.
- `block`: تحديثات معاينة بخطوات مجزأة/ملحقة.
- `progress`: معاينة تقدم/حالة أثناء التوليد، والإجابة النهائية عند الاكتمال.

`streaming.mode: "block"` هو وضع بث معاينة للقنوات القابلة للتحرير
مثل Discord وTelegram. لا يمكّن تسليم كتل القناة هناك.
استخدم `streaming.block.enabled` أو مفتاح القناة القديم `blockStreaming` عندما
تريد ردود كتل عادية. Microsoft Teams هو الاستثناء: لا يملك
نقل كتل معاينة مسودة، لذلك يُطابق `streaming.mode: "block"` تسليم كتل Teams
بدلًا من بث جزئي/تقدم أصلي.

### تعيين القنوات

| القناة | `off` | `partial` | `block` | `progress` |
| ---------- | ----- | --------- | ------- | ----------------------- |
| Telegram | ✅ | ✅ | ✅ | مسودة تقدم قابلة للتحرير |
| Discord | ✅ | ✅ | ✅ | مسودة تقدم قابلة للتحرير |
| Slack | ✅ | ✅ | ✅ | ✅ |
| Mattermost | ✅ | ✅ | ✅ | ✅ |
| MS Teams | ✅ | ✅ | ✅ | بث تقدم أصلي |

خاص بـ Slack فقط:

- يبدّل `channels.slack.streaming.nativeTransport` استدعاءات Slack API الأصلية للبث عندما تكون `channels.slack.streaming.mode="partial"` (افتراضيًا: `true`).
- يتطلب بث Slack الأصلي وحالة سلسلة مساعد Slack هدف سلسلة رد. لا تُظهر الرسائل الخاصة من المستوى الأعلى تلك المعاينة بنمط السلاسل، لكنها لا تزال قادرة على استخدام منشورات معاينة مسودة Slack وتعديلاتها.

ترحيل المفاتيح القديمة:

- Telegram: تُكتشف قيم `streamMode` القديمة وقيم `streaming` العددية/المنطقية وتُرحَّل بواسطة مسارات توافق doctor/الإعدادات إلى `streaming.mode`.
- Discord: تبقى `streamMode` + `streaming` المنطقية أسماء بديلة وقت التشغيل لتعداد `streaming`؛ شغّل `openclaw doctor --fix` لإعادة كتابة الإعدادات المحفوظة.
- Slack: يبقى `streamMode` اسمًا بديلًا وقت التشغيل لـ `streaming.mode`؛ وتبقى `streaming` المنطقية اسمًا بديلًا وقت التشغيل لـ `streaming.mode` بالإضافة إلى `streaming.nativeTransport`؛ ويبقى `nativeStreaming` القديم اسمًا بديلًا وقت التشغيل لـ `streaming.nativeTransport`. شغّل `openclaw doctor --fix` لإعادة كتابة الإعدادات المحفوظة.

### سلوك وقت التشغيل

Telegram:

- يستخدم تحديثات معاينة `sendMessage` + `editMessageText` عبر الرسائل الخاصة والمجموعات/المواضيع.
- لا تزال المعاينات الأولية القصيرة مؤجلة لتحسين تجربة إشعارات الدفع، لكن Telegram يحققها الآن بعد تأخير محدود حتى لا تبقى عمليات التشغيل النشطة صامتة بصريًا.
- يحرر النص النهائي المعاينة النشطة في مكانها؛ تعيد النهايات الطويلة استخدام تلك الرسالة للجزء الأول وترسل الأجزاء المتبقية فقط.
- يدوّر وضع `block` المعاينة إلى رسالة جديدة عند `streaming.preview.chunk.maxChars` (افتراضيًا 800، ومقيّد بحد تحرير Telegram البالغ 4096)؛ تنمّي الأوضاع الأخرى معاينة واحدة حتى 4096 حرفًا.
- يحافظ وضع `progress` على تقدم الأدوات في مسودة حالة قابلة للتحرير، ويحقق تسمية الحالة عندما يكون بث الإجابة نشطًا لكن لا يتوفر سطر أداة بعد، ويمسح تلك المسودة عند الاكتمال، ويرسل الإجابة النهائية عبر التسليم العادي.
- إذا فشل التحرير النهائي قبل تأكيد النص المكتمل، يستخدم OpenClaw التسليم النهائي العادي وينظف المعاينة القديمة.
- يُتخطى بث المعاينة عندما يكون بث كتل Telegram ممكّنًا صراحةً (لتجنب البث المزدوج).
- يمكن لـ `/reasoning stream` كتابة الاستدلال إلى معاينة عابرة تُحذف بعد التسليم النهائي.

Discord:

- يستخدم رسائل معاينة بالإرسال + التحرير.
- يستخدم وضع `block` تجزئة المسودات (`draftChunk`).
- يُتخطى بث المعاينة عندما يكون بث كتل Discord ممكّنًا صراحةً.
- تلغي حمولات الوسائط النهائية والأخطاء والردود الصريحة المعاينات المعلقة دون تفريغ مسودة جديدة، ثم تستخدم التسليم العادي.

Slack:

- يمكن أن يستخدم `partial` بث Slack الأصلي (`chat.startStream`/`append`/`stop`) عندما يكون متاحًا.
- يستخدم `block` معاينات مسودة بنمط الإلحاق.
- يستخدم `progress` نص معاينة الحالة، ثم الإجابة النهائية.
- تستخدم الرسائل الخاصة من المستوى الأعلى دون سلسلة رد منشورات معاينة المسودة وتعديلاتها بدلًا من بث Slack الأصلي.
- يمنع بث المعاينة الأصلي والمسودة ردود الكتل لتلك الدورة، لذلك يُبث رد Slack عبر مسار تسليم واحد فقط.
- لا تنشئ حمولات الوسائط/الأخطاء النهائية ونهايات التقدم رسائل مسودة عابرة؛ فقط نهايات النص/الكتل التي يمكنها تحرير المعاينة تفرّغ نص المسودة المعلق.

Mattermost:

- يبث التفكير ونشاط الأدوات ونص الرد الجزئي في منشور معاينة مسودة واحد يُنهى في مكانه عندما يصبح إرسال الإجابة النهائية آمنًا.
- يعود إلى إرسال منشور نهائي جديد إذا حُذف منشور المعاينة أو أصبح غير متاح وقت الإنهاء.
- تلغي حمولات الوسائط/الأخطاء النهائية تحديثات المعاينة المعلقة قبل التسليم العادي بدلًا من تفريغ منشور معاينة مؤقت.

Matrix:

- تُنهى معاينات المسودة في مكانها عندما يستطيع النص النهائي إعادة استخدام حدث المعاينة.
- تلغي النهايات التي تحتوي على وسائط فقط، والأخطاء، وحالات عدم تطابق هدف الرد تحديثات المعاينة المعلقة قبل التسليم العادي؛ وتُنقح المعاينة القديمة المرئية بالفعل.

### تحديثات معاينة تقدم الأدوات

يمكن أن يتضمن بث المعاينة أيضًا **تحديثات تقدم الأدوات** - أسطر حالة قصيرة مثل "البحث في الويب"، أو "قراءة ملف"، أو "استدعاء أداة" - تظهر في رسالة المعاينة نفسها أثناء تشغيل الأدوات، قبل الرد النهائي. في وضع خادم تطبيق Codex، تستخدم رسائل التمهيد/التعليق من Codex مسار المعاينة نفسه، لذلك يمكن لملاحظات تقدم قصيرة مثل "أتحقق..." أن تُبث إلى المسودة القابلة للتحرير دون أن تصبح جزءًا من الإجابة النهائية. يحافظ هذا على الحيوية البصرية لدورات الأدوات متعددة الخطوات بدلًا من الصمت بين معاينة التفكير الأولى والإجابة النهائية.

قد تصدر الأدوات طويلة التشغيل تقدمًا مصنفًا قبل أن تعود. على سبيل المثال،
يضبط `web_fetch` مؤقتًا مدته خمس ثوانٍ عند بدئه: إذا كان الجلب لا يزال
معلقًا، يمكن للمعاينة عرض `Fetching page content...`؛ وإذا انتهى الجلب
أو أُلغي قبل ذلك، فلا يصدر سطر تقدم. لا تزال نتيجة الأداة النهائية اللاحقة
تُسلّم إلى النموذج بشكل عادي.

الأسطح المدعومة:

- تبث **Discord** و**Slack** و**Telegram** و**Matrix** تقدّم الأدوات وتحديثات تمهيد Codex إلى تعديل المعاينة الحية افتراضيًا عندما يكون بث المعاينة نشطًا. يستخدم Microsoft Teams بث التقدّم الأصلي لديه في المحادثات الشخصية.
- شُحن Telegram مع تفعيل تحديثات معاينة تقدّم الأدوات منذ `v2026.4.22`؛ والإبقاء عليها مفعّلة يحافظ على ذلك السلوك المُصدر.
- يدمج **Mattermost** بالفعل نشاط الأدوات في منشور معاينة المسودة الفردي لديه (انظر أعلاه).
- تتبع تعديلات تقدّم الأدوات وضع بث المعاينة النشط؛ ويتم تخطيها عندما يكون بث المعاينة `off` أو عندما يتولى بث الكتل الرسالة. في Telegram، يكون `streaming.mode: "off"` للنتيجة النهائية فقط: يتم أيضًا كبت رسائل التقدّم العامة بدلًا من تسليمها كرسائل حالة مستقلة، بينما تستمر مطالبات الموافقة وحمولات الوسائط والأخطاء في التوجيه بشكل طبيعي.
- للاحتفاظ ببث المعاينة مع إخفاء أسطر تقدّم الأدوات، اضبط `streaming.preview.toolProgress` على `false` لتلك القناة. للإبقاء على أسطر تقدّم الأدوات مرئية مع إخفاء نص الأوامر/التنفيذ، اضبط `streaming.preview.commandText` على `"status"` أو `streaming.progress.commandText` على `"status"`؛ القيمة الافتراضية هي `"raw"` للحفاظ على السلوك المُصدر. تشترك قنوات المسودة/التقدّم التي تستخدم عارض التقدّم المضغوط في OpenClaw في هذه السياسة، بما في ذلك Discord وMatrix وMicrosoft Teams وMattermost ومعاينات مسودات Slack وTelegram. لتعطيل تعديلات المعاينة بالكامل، اضبط `streaming.mode` على `off`.
- تُعد ردود الاقتباس المحددة في Telegram استثناءً: عندما لا يكون `replyToMode` هو `"off"` وكان نص الاقتباس المحدد موجودًا، يتخطى OpenClaw بث معاينة الإجابة لتلك الدورة، بحيث لا يمكن عرض أسطر معاينة تقدّم الأدوات. لا تزال الردود على الرسالة الحالية بدون نص اقتباس محدد تحتفظ ببث المعاينة. راجع [وثائق قناة Telegram](/ar/channels/telegram) للتفاصيل.

أبقِ أسطر التقدّم مرئية مع إخفاء نص الأوامر/التنفيذ الخام:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```

استخدم البنية نفسها تحت مفتاح قناة تقدّم مضغوط آخر، مثل `channels.discord` أو `channels.matrix` أو `channels.msteams` أو `channels.mattermost` أو معاينات مسودات Slack. لوضع مسودة التقدّم، ضع السياسة نفسها تحت `streaming.progress`:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "progress",
        "progress": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```
