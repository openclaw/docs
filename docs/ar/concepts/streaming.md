لدى OpenClaw طبقتا بث منفصلتان:

- **بث الكتل (القنوات):** إصدار **كتل** مكتملة أثناء كتابة المساعد. هذه رسائل قنوات عادية (وليست فروق رموز).
- **بث المعاينة (Telegram/Discord/Slack):** تحديث **رسالة معاينة** مؤقتة أثناء التوليد.

لا يوجد **بث حقيقي لفروق الرموز** إلى رسائل القنوات اليوم. بث المعاينة قائم على الرسائل (إرسال + تعديلات/إلحاقات).

## بث الكتل (رسائل القنوات)

يرسل بث الكتل مخرجات المساعد في أجزاء كبيرة نسبيًا عندما تصبح متاحة.

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

وسيلة الإيضاح:

- `text_delta/events`: أحداث بث النموذج (قد تكون قليلة في النماذج غير الباثة).
- `chunker`: يطبق `EmbeddedBlockChunker` الحدود الدنيا/العليا + تفضيل الفاصل.
- `channel send`: الرسائل الصادرة الفعلية (ردود الكتل).

**عناصر التحكم:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (الافتراضي إيقاف).
- تجاوزات القنوات: `*.blockStreaming` (ومتغيرات لكل حساب) لفرض `"on"`/`"off"` لكل قناة.
- `agents.defaults.blockStreamingBreak`: `"text_end"` أو `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (دمج الكتل المبثوثة قبل الإرسال).
- الحد الأقصى الصارم للقناة: `*.textChunkLimit` (مثلًا، `channels.whatsapp.textChunkLimit`).
- وضع تقطيع القناة: `*.chunkMode` (الافتراضي `length`، ويقسم `newline` على الأسطر الفارغة (حدود الفقرات) قبل التقطيع حسب الطول).
- الحد المرن في Discord: `channels.discord.maxLinesPerMessage` (الافتراضي 17) يقسم الردود الطويلة لتجنب اقتطاع الواجهة.

**دلالات الحدود:**

- `text_end`: بث الكتل فور إصدارها من المقسم؛ التفريغ عند كل `text_end`.
- `message_end`: الانتظار حتى تنتهي رسالة المساعد، ثم تفريغ المخرجات المخزنة مؤقتًا.

ما زال `message_end` يستخدم المقسم إذا تجاوز النص المخزن مؤقتًا `maxChars`، لذلك يمكنه إصدار عدة أجزاء في النهاية.

### تسليم الوسائط مع بث الكتل

توجيهات `MEDIA:` هي بيانات وصفية عادية للتسليم. عندما يرسل بث الكتل كتلة وسائط مبكرًا، يتذكر OpenClaw ذلك التسليم للدورة. إذا كررت حمولة المساعد النهائية عنوان URL للوسائط نفسه، فإن التسليم النهائي يزيل الوسائط المكررة بدلًا من إرسال المرفق مرة أخرى.

تُمنع الحمولات النهائية المطابقة تمامًا من التكرار. إذا أضافت الحمولة النهائية نصًا مميزًا حول وسائط بُثت بالفعل، فسيظل OpenClaw يرسل النص الجديد مع إبقاء الوسائط ذات تسليم واحد. يمنع هذا تكرار الملاحظات الصوتية أو الملفات على قنوات مثل Telegram عندما يصدر الوكيل `MEDIA:` أثناء البث ويضمّنها الموفر أيضًا في الرد المكتمل.

## خوارزمية التقطيع (الحدود الدنيا/العليا)

ينفذ `EmbeddedBlockChunker` تقطيع الكتل:

- **الحد الأدنى:** لا تُصدر حتى يكون المخزن المؤقت >= `minChars` (ما لم يُفرض ذلك).
- **الحد الأعلى:** تفضيل التقسيم قبل `maxChars`؛ وإذا فُرض، فقسّم عند `maxChars`.
- **تفضيل الفاصل:** `paragraph` → `newline` → `sentence` → `whitespace` → فاصل صلب.
- **أسوار الكود:** لا تقسّم داخل الأسوار أبدًا؛ عند الفرض عند `maxChars`، أغلق السور وأعد فتحه للحفاظ على صلاحية Markdown.

يُحصر `maxChars` ضمن `textChunkLimit` للقناة، لذلك لا يمكنك تجاوز حدود كل قناة.

## الدمج (دمج الكتل المبثوثة)

عند تمكين بث الكتل، يستطيع OpenClaw **دمج أجزاء الكتل المتتالية**
قبل إرسالها. يقلل هذا من "إغراق الأسطر المفردة" مع الاستمرار في توفير
مخرجات تدريجية.

- ينتظر الدمج **فجوات الخمول** (`idleMs`) قبل التفريغ.
- تُقيَّد المخازن المؤقتة بـ `maxChars` وستُفرَّغ إذا تجاوزته.
- يمنع `minChars` إرسال الأجزاء الصغيرة حتى يتراكم نص كاف
  (يرسل التفريغ النهائي دائمًا النص المتبقي).
- يُشتق رابط الدمج من `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`، `newline` → `\n`، `sentence` → مسافة).
- تتوفر تجاوزات القنوات عبر `*.blockStreamingCoalesce` (بما في ذلك إعدادات كل حساب).
- تُرفع قيمة `minChars` الافتراضية للدمج إلى 1500 في Signal/Slack/Discord ما لم يتم تجاوزها.

## إيقاع شبيه بالبشر بين الكتل

عند تمكين بث الكتل، يمكنك إضافة **إيقاف مؤقت عشوائي** بين
ردود الكتل (بعد الكتلة الأولى). يجعل ذلك الردود متعددة الفقاعات تبدو
أكثر طبيعية.

- الإعداد: `agents.defaults.humanDelay` (تجاوز لكل وكيل عبر `agents.list[].humanDelay`).
- الأوضاع: `off` (الافتراضي)، `natural` (800–2500 مللي ثانية)، `custom` (`minMs`/`maxMs`).
- ينطبق فقط على **ردود الكتل**، وليس الردود النهائية أو ملخصات الأدوات.

## "بث الأجزاء أو كل شيء"

يتطابق هذا مع:

- **بث الأجزاء:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (الإصدار أثناء التقدم). تحتاج القنوات غير Telegram أيضًا إلى `*.blockStreaming: true`.
- **بث كل شيء في النهاية:** `blockStreamingBreak: "message_end"` (تفريغ مرة واحدة، وربما عدة أجزاء إذا كان طويلًا جدًا).
- **لا بث كتل:** `blockStreamingDefault: "off"` (الرد النهائي فقط).

**ملاحظة القناة:** يكون بث الكتل **متوقفًا ما لم**
يُضبط `*.blockStreaming` صراحة على `true`. يمكن للقنوات بث معاينة مباشرة
(`channels.<channel>.streaming`) دون ردود كتل.

تذكير بموقع الإعدادات: توجد افتراضيات `blockStreaming*` تحت
`agents.defaults`، وليس في إعداد الجذر.

## أوضاع بث المعاينة

المفتاح القياسي: `channels.<channel>.streaming`

الأوضاع:

- `off`: تعطيل بث المعاينة.
- `partial`: معاينة واحدة تُستبدل بأحدث نص.
- `block`: تحديثات المعاينة في خطوات مقطعة/ملحقة.
- `progress`: معاينة تقدم/حالة أثناء التوليد، والجواب النهائي عند الاكتمال.

`streaming.mode: "block"` هو وضع بث معاينة للقنوات القادرة على التعديل
مثل Discord وTelegram. لا يفعّل تسليم كتل القناة هناك.
استخدم `streaming.block.enabled` أو مفتاح القناة القديم `blockStreaming` عندما
تريد ردود كتل عادية. Microsoft Teams هو الاستثناء: ليست لديه وسيلة نقل
كتل لمسودات المعاينة، لذلك يتطابق `streaming.mode: "block"` مع تسليم كتل Teams
بدلًا من بث جزئي/تقدم أصلي.

### ربط القنوات

| القناة    | `off` | `partial` | `block` | `progress`              |
| ---------- | ----- | --------- | ------- | ----------------------- |
| Telegram   | ✅    | ✅        | ✅      | مسودة تقدم قابلة للتحرير |
| Discord    | ✅    | ✅        | ✅      | مسودة تقدم قابلة للتحرير |
| Slack      | ✅    | ✅        | ✅      | ✅                      |
| Mattermost | ✅    | ✅        | ✅      | ✅                      |
| MS Teams   | ✅    | ✅        | ✅      | بث تقدم أصلي  |

خاص بـ Slack فقط:

- يبدّل `channels.slack.streaming.nativeTransport` استدعاءات واجهة برمجة تطبيقات البث الأصلية في Slack عندما يكون `channels.slack.streaming.mode="partial"` (الافتراضي: `true`).
- يتطلب البث الأصلي في Slack وحالة سلسلة مساعد Slack هدف سلسلة رد. لا تعرض الرسائل المباشرة ذات المستوى الأعلى تلك المعاينة بنمط السلسلة، لكنها ما زالت تستطيع استخدام منشورات معاينة مسودة Slack وتعديلاتها.

ترحيل المفاتيح القديمة:

- Telegram: تُكتشف قيم `streamMode` القديمة وقيم `streaming` العددية/المنطقية وتُرحّل عبر مسارات توافق الطبيب/الإعدادات إلى `streaming.mode`.
- Discord: يُرحّل `streamMode` + `streaming` المنطقي تلقائيًا إلى تعداد `streaming`.
- Slack: يُرحّل `streamMode` تلقائيًا إلى `streaming.mode`؛ ويُرحّل `streaming` المنطقي تلقائيًا إلى `streaming.mode` بالإضافة إلى `streaming.nativeTransport`؛ ويُرحّل `nativeStreaming` القديم تلقائيًا إلى `streaming.nativeTransport`.

### سلوك وقت التشغيل

Telegram:

- يستخدم تحديثات المعاينة `sendMessage` + `editMessageText` عبر الرسائل المباشرة والمجموعات/المواضيع.
- يرسل رسالة نهائية جديدة بدلًا من التعديل في مكانه عندما تكون المعاينة مرئية لنحو دقيقة واحدة، ثم ينظف المعاينة بحيث يعكس الطابع الزمني في Telegram اكتمال الرد.
- يُتخطى بث المعاينة عندما يكون بث كتل Telegram مفعّلًا صراحة (لتجنب البث المزدوج).
- يستطيع `/reasoning stream` كتابة التفكير إلى معاينة عابرة تُحذف بعد التسليم النهائي.

Discord:

- يستخدم رسائل معاينة بالإرسال + التعديل.
- يستخدم وضع `block` تقطيع المسودة (`draftChunk`).
- يُتخطى بث المعاينة عندما يكون بث كتل Discord مفعّلًا صراحة.
- تلغي حمولات الوسائط النهائية والأخطاء والردود الصريحة المعاينات المعلقة دون تفريغ مسودة جديدة، ثم تستخدم التسليم العادي.

Slack:

- يمكن أن يستخدم `partial` بث Slack الأصلي (`chat.startStream`/`append`/`stop`) عندما يكون متاحًا.
- يستخدم `block` معاينات مسودة بنمط الإلحاق.
- يستخدم `progress` نص معاينة الحالة، ثم الجواب النهائي.
- تستخدم الرسائل المباشرة ذات المستوى الأعلى دون سلسلة رد منشورات معاينة مسودة وتعديلاتها بدلًا من بث Slack الأصلي.
- يمنع بث المعاينة الأصلي وبث مسودة المعاينة ردود الكتل لتلك الدورة، لذلك يُبث رد Slack عبر مسار تسليم واحد فقط.
- لا تنشئ حمولات الوسائط/الأخطاء النهائية ونهايات التقدم رسائل مسودة مؤقتة؛ فقط النهائيات النصية/الكتلية التي يمكنها تعديل المعاينة تفرغ نص المسودة المعلقة.

Mattermost:

- يبث التفكير ونشاط الأدوات ونص الرد الجزئي في منشور معاينة مسودة واحد يُنهى في مكانه عندما يكون إرسال الجواب النهائي آمنًا.
- يعود إلى إرسال منشور نهائي جديد إذا حُذف منشور المعاينة أو لم يكن متاحًا بطريقة أخرى وقت الإنهاء.
- تلغي حمولات الوسائط/الأخطاء النهائية تحديثات المعاينة المعلقة قبل التسليم العادي بدلًا من تفريغ منشور معاينة مؤقت.

Matrix:

- تُنهى معاينات المسودة في مكانها عندما يستطيع النص النهائي إعادة استخدام حدث المعاينة.
- تلغي النهائيات الخاصة بالوسائط فقط والأخطاء وعدم تطابق هدف الرد تحديثات المعاينة المعلقة قبل التسليم العادي؛ وتُنقح معاينة قديمة مرئية بالفعل.

### تحديثات معاينة تقدم الأدوات

يمكن أن يتضمن بث المعاينة أيضًا تحديثات **تقدم الأدوات** — أسطر حالة قصيرة مثل "البحث في الويب"، أو "قراءة ملف"، أو "استدعاء أداة" — تظهر في رسالة المعاينة نفسها أثناء تشغيل الأدوات، قبل الرد النهائي. يُبقي هذا دورات الأدوات متعددة الخطوات حية بصريًا بدلًا من الصمت بين أول معاينة تفكير والجواب النهائي.

الأسطح المدعومة:

- يبث **Discord** و**Slack** و**Telegram** و**Matrix** تقدم الأدوات في تعديل المعاينة المباشر افتراضيًا عندما يكون بث المعاينة نشطًا. يستخدم Microsoft Teams بث التقدم الأصلي في المحادثات الشخصية.
- شُحن Telegram مع تمكين تحديثات معاينة تقدم الأدوات منذ `v2026.4.22`؛ وإبقاؤها مفعلة يحافظ على ذلك السلوك المنشور.
- يدمج **Mattermost** بالفعل نشاط الأدوات في منشور معاينة المسودة الواحد لديه (انظر أعلاه).
- تتبع تعديلات تقدم الأدوات وضع بث المعاينة النشط؛ وتُتخطى عندما يكون بث المعاينة `off` أو عندما يتولى بث الكتل الرسالة. في Telegram، يكون `streaming.mode: "off"` نهائيًا فقط: يُمنع أيضًا كلام التقدم العام بدلًا من تسليمه كرسائل حالة مستقلة، بينما تظل مطالبات الموافقة وحمولات الوسائط والأخطاء تسلك مساراتها العادية.
- للإبقاء على بث المعاينة مع إخفاء أسطر تقدم الأدوات، اضبط `streaming.preview.toolProgress` على `false` لتلك القناة. للإبقاء على أسطر تقدم الأدوات مرئية مع إخفاء نص الأوامر/التنفيذ، اضبط `streaming.preview.commandText` على `"status"` أو `streaming.progress.commandText` على `"status"`؛ الافتراضي هو `"raw"` للحفاظ على السلوك المنشور. هذه السياسة مشتركة بين قنوات المسودات/التقدم التي تستخدم عارض التقدم المضغوط في OpenClaw، بما في ذلك Discord وMatrix وMicrosoft Teams وMattermost ومعاينات مسودة Slack وTelegram. لتعطيل تعديلات المعاينة بالكامل، اضبط `streaming.mode` على `off`.
- تُعد ردود الاقتباس المحددة في Telegram استثناءً: عندما لا يكون `replyToMode` هو `"off"` ويكون نص اقتباس محدد موجودًا، يتخطى OpenClaw بث معاينة الجواب لتلك الدورة بحيث لا يمكن عرض أسطر معاينة تقدم الأدوات. ما زالت ردود الرسالة الحالية دون نص اقتباس محدد تبقي بث المعاينة. راجع [وثائق قناة Telegram](/ar/channels/telegram) للتفاصيل.

أبقِ أسطر التقدم مرئية مع إخفاء نص الأمر/التنفيذ الخام:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```

استخدم البنية نفسها تحت مفتاح قناة تقدم موجزة آخر، مثل `channels.discord` أو `channels.matrix` أو `channels.msteams` أو `channels.mattermost` أو معاينات مسودات Slack. لوضع مسودات التقدم، ضع السياسة نفسها تحت `streaming.progress`:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "progress",
        "progress": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```
