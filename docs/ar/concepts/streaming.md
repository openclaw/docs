لدى OpenClaw طبقتا بث منفصلتان:

- **بث الكتل (القنوات):** إصدار **كتل** مكتملة أثناء كتابة المساعد. هذه رسائل قنوات عادية (وليست فروق رموز).
- **بث المعاينة (Telegram/Discord/Slack):** تحديث **رسالة معاينة** مؤقتة أثناء التوليد.

لا يوجد اليوم **بث فروق رموز حقيقي** إلى رسائل القنوات. بث المعاينة قائم على الرسائل (إرسال + تعديلات/إلحاقات).

## بث الكتل (رسائل القنوات)

يرسل بث الكتل خرج المساعد في أجزاء كبيرة عند توفرها.

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker emits blocks as buffer grows
       └─ (blockStreamingBreak=message_end)
            └─ chunker flushes at message_end
                   └─ channel send (block replies)
```

المفتاح:

- `text_delta/events`: أحداث بث النموذج (قد تكون متباعدة للنماذج غير الباثة).
- `chunker`: يطبق `EmbeddedBlockChunker` الحدود الدنيا/العليا + تفضيل الفاصل.
- `channel send`: الرسائل الصادرة الفعلية (ردود الكتل).

**عناصر التحكم:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (الافتراضي إيقاف).
- تجاوزات القنوات: `*.blockStreaming` (ومتغيرات لكل حساب) لفرض `"on"`/`"off"` لكل قناة.
- `agents.defaults.blockStreamingBreak`: `"text_end"` أو `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (دمج الكتل المبثوثة قبل الإرسال).
- الحد الأقصى الصارم للقناة: `*.textChunkLimit` (مثل `channels.whatsapp.textChunkLimit`).
- وضع تقطيع القناة: `*.chunkMode` (`length` هو الافتراضي، و`newline` يقسم على الأسطر الفارغة (حدود الفقرات) قبل التقطيع حسب الطول).
- الحد المرن في Discord: `channels.discord.maxLinesPerMessage` (الافتراضي 17) يقسم الردود الطويلة لتجنب قص الواجهة.

**دلالات الحدود:**

- `text_end`: بث الكتل بمجرد أن يصدرها المقسم؛ التفريغ عند كل `text_end`.
- `message_end`: الانتظار حتى تنتهي رسالة المساعد، ثم تفريغ الخرج المخزن.

ما زال `message_end` يستخدم المقسم إذا تجاوز النص المخزن `maxChars`، لذلك يمكنه إصدار عدة أجزاء في النهاية.

### تسليم الوسائط مع بث الكتل

توجيهات `MEDIA:` هي بيانات وصفية عادية للتسليم. عندما يرسل بث الكتل كتلة وسائط مبكرا، يتذكر OpenClaw ذلك التسليم للدورة. إذا كررت حمولة المساعد النهائية عنوان URL نفسه للوسائط، يزيل التسليم النهائي الوسائط المكررة بدلا من إرسال المرفق مرة أخرى.

يتم منع الحمولات النهائية المطابقة تماما. إذا أضافت الحمولة النهائية نصا مميزا حول وسائط تم بثها بالفعل، يظل OpenClaw يرسل النص الجديد مع إبقاء الوسائط بتسليم واحد. يمنع هذا تكرار الملاحظات الصوتية أو الملفات على قنوات مثل Telegram عندما يصدر الوكيل `MEDIA:` أثناء البث ويضمها المزود أيضا في الرد المكتمل.

## خوارزمية التقطيع (الحدود الدنيا/العليا)

يتم تنفيذ تقطيع الكتل بواسطة `EmbeddedBlockChunker`:

- **الحد الأدنى:** لا تصدر حتى يكون المخزن >= `minChars` (إلا إذا تم الإجبار).
- **الحد الأعلى:** تفضيل التقسيم قبل `maxChars`؛ إذا تم الإجبار، يتم التقسيم عند `maxChars`.
- **تفضيل الفاصل:** `paragraph` → `newline` → `sentence` → `whitespace` → فاصل صارم.
- **أسوار الكود:** لا تقسم أبدا داخل الأسوار؛ عند الإجبار عند `maxChars`، أغلق السور وأعد فتحه للحفاظ على صحة Markdown.

يتم تقييد `maxChars` إلى `textChunkLimit` الخاص بالقناة، لذلك لا يمكنك تجاوز الحدود القصوى لكل قناة.

## الدمج (دمج الكتل المبثوثة)

عند تفعيل بث الكتل، يمكن لـ OpenClaw **دمج أجزاء الكتل المتتالية** قبل إرسالها. يقلل هذا "إزعاج السطر الواحد" مع الاستمرار في توفير خرج تدريجي.

- ينتظر الدمج **فجوات الخمول** (`idleMs`) قبل التفريغ.
- يتم تقييد المخازن بواسطة `maxChars` وستفرغ إذا تجاوزته.
- يمنع `minChars` إرسال الشظايا الصغيرة حتى يتراكم نص كاف
  (يرسل التفريغ النهائي دائما النص المتبقي).
- يتم اشتقاق الواصل من `blockStreamingChunk.breakPreference`
  (`paragraph` → `\n\n`، و`newline` → `\n`، و`sentence` → مسافة).
- تتوفر تجاوزات القنوات عبر `*.blockStreamingCoalesce` (بما في ذلك إعدادات كل حساب).
- تتم زيادة `minChars` الافتراضي للدمج إلى 1500 في Signal/Slack/Discord ما لم يتم تجاوزه.

## إيقاع يشبه البشر بين الكتل

عند تفعيل بث الكتل، يمكنك إضافة **توقف عشوائي** بين ردود الكتل (بعد الكتلة الأولى). هذا يجعل الردود متعددة الفقاعات تبدو أكثر طبيعية.

- الإعداد: `agents.defaults.humanDelay` (يمكن تجاوزه لكل وكيل عبر `agents.list[].humanDelay`).
- الأوضاع: `off` (الافتراضي)، و`natural` (800-2500ms)، و`custom` (`minMs`/`maxMs`).
- ينطبق فقط على **ردود الكتل**، وليس الردود النهائية أو ملخصات الأدوات.

## "بث الأجزاء أو كل شيء"

هذا يقابل:

- **بث الأجزاء:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (الإصدار أثناء التقدم). تحتاج القنوات غير Telegram أيضا إلى `*.blockStreaming: true`.
- **بث كل شيء في النهاية:** `blockStreamingBreak: "message_end"` (تفريغ مرة واحدة، وربما عدة أجزاء إذا كان طويلا جدا).
- **بلا بث كتل:** `blockStreamingDefault: "off"` (الرد النهائي فقط).

**ملاحظة القناة:** يكون بث الكتل **متوقفا إلا إذا**
تم تعيين `*.blockStreaming` صراحة إلى `true`. يمكن للقنوات بث معاينة حية
(`channels.<channel>.streaming`) دون ردود كتل.

تذكير بموقع الإعداد: تعيش افتراضيات `blockStreaming*` تحت
`agents.defaults`، وليس إعداد الجذر.

## أوضاع بث المعاينة

المفتاح المعياري: `channels.<channel>.streaming`

الأوضاع:

- `off`: تعطيل بث المعاينة.
- `partial`: معاينة واحدة يتم استبدالها بأحدث نص.
- `block`: تحديثات المعاينة في خطوات مقطعة/ملحقة.
- `progress`: معاينة تقدم/حالة أثناء التوليد، والإجابة النهائية عند الاكتمال.

`streaming.mode: "block"` هو وضع بث معاينة للقنوات القادرة على التحرير
مثل Discord وTelegram. لا يفعّل تسليم كتل القناة هناك.
استخدم `streaming.block.enabled` أو مفتاح القناة القديم `blockStreaming` عندما
تريد ردود كتل عادية. Microsoft Teams هو الاستثناء: لا يملك
نقل كتل مسودة معاينة، لذلك يطابق `streaming.mode: "block"` تسليم كتل Teams
بدلا من البث الجزئي/التقدمي الأصلي.

### ربط القنوات

| القناة    | `off` | `partial` | `block` | `progress`              |
| ---------- | ----- | --------- | ------- | ----------------------- |
| Telegram   | ✅    | ✅        | ✅      | مسودة تقدم قابلة للتحرير |
| Discord    | ✅    | ✅        | ✅      | مسودة تقدم قابلة للتحرير |
| Slack      | ✅    | ✅        | ✅      | ✅                      |
| Mattermost | ✅    | ✅        | ✅      | ✅                      |
| MS Teams   | ✅    | ✅        | ✅      | بث تقدم أصلي  |

Slack فقط:

- يبدل `channels.slack.streaming.nativeTransport` استدعاءات Slack API للبث الأصلي عند `channels.slack.streaming.mode="partial"` (الافتراضي: `true`).
- يتطلب بث Slack الأصلي وحالة سلسلة مساعد Slack هدف سلسلة رد. لا تعرض الرسائل المباشرة ذات المستوى الأعلى تلك المعاينة بنمط السلسلة، لكنها لا تزال قادرة على استخدام منشورات معاينة مسودة Slack وتعديلاتها.

ترحيل المفاتيح القديمة:

- Telegram: يتم اكتشاف قيم `streamMode` القديمة وقيم `streaming` العددية/المنطقية وترحيلها بواسطة مسارات توافق doctor/config إلى `streaming.mode`.
- Discord: يتم ترحيل `streamMode` + `streaming` المنطقية تلقائيا إلى تعداد `streaming`.
- Slack: يتم ترحيل `streamMode` تلقائيا إلى `streaming.mode`؛ ويتم ترحيل `streaming` المنطقية تلقائيا إلى `streaming.mode` إضافة إلى `streaming.nativeTransport`؛ ويتم ترحيل `nativeStreaming` القديمة تلقائيا إلى `streaming.nativeTransport`.

### سلوك وقت التشغيل

Telegram:

- يستخدم تحديثات المعاينة `sendMessage` + `editMessageText` عبر الرسائل المباشرة والمجموعات/المواضيع.
- يعدل النص النهائي المعاينة النشطة في مكانها؛ تعيد النهايات الطويلة استخدام تلك الرسالة لأول جزء وترسل الأجزاء المتبقية فقط.
- يحتفظ وضع `progress` بتقدم الأداة في مسودة حالة قابلة للتحرير، ويمسح تلك المسودة عند الاكتمال، ويرسل الإجابة النهائية عبر التسليم العادي.
- إذا فشل التعديل النهائي قبل تأكيد النص المكتمل، يستخدم OpenClaw التسليم النهائي العادي وينظف المعاينة القديمة.
- يتم تخطي بث المعاينة عندما يكون بث كتل Telegram مفعلا صراحة (لتجنب البث المزدوج).
- يمكن لـ `/reasoning stream` كتابة الاستدلال إلى معاينة عابرة يتم حذفها بعد التسليم النهائي.

Discord:

- يستخدم رسائل معاينة بالإرسال + التعديل.
- يستخدم وضع `block` تقطيع المسودة (`draftChunk`).
- يتم تخطي بث المعاينة عندما يكون بث كتل Discord مفعلا صراحة.
- تلغي الوسائط النهائية، والأخطاء، وحمولات الرد الصريح المعاينات المعلقة دون تفريغ مسودة جديدة، ثم تستخدم التسليم العادي.

Slack:

- يمكن أن يستخدم `partial` بث Slack الأصلي (`chat.startStream`/`append`/`stop`) عند توفره.
- يستخدم `block` معاينات مسودة بنمط الإلحاق.
- يستخدم `progress` نص معاينة الحالة، ثم الإجابة النهائية.
- تستخدم الرسائل المباشرة ذات المستوى الأعلى دون سلسلة رد منشورات معاينة مسودة وتعديلاتها بدلا من بث Slack الأصلي.
- يمنع بث المعاينة الأصلي والمسودة ردود الكتل لتلك الدورة، لذلك يتم بث رد Slack عبر مسار تسليم واحد فقط.
- لا تنشئ حمولات الوسائط/الأخطاء النهائية ونهايات التقدم رسائل مسودة مؤقتة؛ وحدها النهايات النصية/الكتلية التي تستطيع تعديل المعاينة تفرغ نص المسودة المعلق.

Mattermost:

- يبث التفكير، ونشاط الأدوات، ونص الرد الجزئي في منشور معاينة مسودة واحد يتم إنهاؤه في مكانه عندما تصبح الإجابة النهائية آمنة للإرسال.
- يعود إلى إرسال منشور نهائي جديد إذا تم حذف منشور المعاينة أو لم يكن متاحا بطريقة أخرى وقت الإنهاء.
- تلغي حمولات الوسائط/الأخطاء النهائية تحديثات المعاينة المعلقة قبل التسليم العادي بدلا من تفريغ منشور معاينة مؤقت.

Matrix:

- يتم إنهاء معاينات المسودة في مكانها عندما يستطيع النص النهائي إعادة استخدام حدث المعاينة.
- تلغي النهايات الخاصة بالوسائط فقط، والأخطاء، وعدم تطابق هدف الرد تحديثات المعاينة المعلقة قبل التسليم العادي؛ ويتم تنقيح المعاينة القديمة المرئية بالفعل.

### تحديثات معاينة تقدم الأدوات

يمكن أن يتضمن بث المعاينة أيضا تحديثات **تقدم الأدوات** - أسطر حالة قصيرة مثل "البحث في الويب"، أو "قراءة ملف"، أو "استدعاء أداة" - تظهر في رسالة المعاينة نفسها أثناء تشغيل الأدوات، قبل الرد النهائي. يبقي هذا دورات الأدوات متعددة الخطوات حية بصريا بدلا من أن تكون صامتة بين معاينة التفكير الأولى والإجابة النهائية.

الأسطح المدعومة:

- يبث **Discord** و**Slack** و**Telegram** و**Matrix** تقدم الأدوات داخل تعديل المعاينة المباشرة افتراضيًا عندما يكون بث المعاينة نشطًا. يستخدم Microsoft Teams بث التقدم الأصلي الخاص به في المحادثات الشخصية.
- تم إصدار Telegram مع تمكين تحديثات معاينة تقدم الأدوات منذ `v2026.4.22`؛ وإبقاؤها مفعّلة يحافظ على ذلك السلوك المُصدَر.
- يدمج **Mattermost** بالفعل نشاط الأدوات في منشور معاينة المسودة المفرد الخاص به (انظر أعلاه).
- تتبع تعديلات تقدم الأدوات وضع بث المعاينة النشط؛ ويتم تخطيها عندما يكون بث المعاينة `off` أو عندما يتولى بث الكتل الرسالة. في Telegram، يكون `streaming.mode: "off"` نهائيًا فقط: يتم أيضًا كتم دردشة التقدم العامة بدلًا من تسليمها كرسائل حالة مستقلة، بينما تظل مطالبات الموافقة وحمولات الوسائط والأخطاء تُوجَّه بشكل طبيعي.
- للإبقاء على بث المعاينة مع إخفاء أسطر تقدم الأدوات، اضبط `streaming.preview.toolProgress` على `false` لذلك القناة. ولإبقاء أسطر تقدم الأدوات مرئية مع إخفاء نص الأمر/التنفيذ، اضبط `streaming.preview.commandText` على `"status"` أو `streaming.progress.commandText` على `"status"`؛ الافتراضي هو `"raw"` للحفاظ على السلوك المُصدَر. تشترك في هذه السياسة قنوات المسودات/التقدم التي تستخدم عارض التقدم المضغوط في OpenClaw، بما في ذلك Discord وMatrix وMicrosoft Teams وMattermost ومعاينات مسودات Slack وTelegram. لتعطيل تعديلات المعاينة بالكامل، اضبط `streaming.mode` على `off`.
- ردود الاقتباس المحددة في Telegram هي استثناء: عندما لا يكون `replyToMode` هو `"off"` ويكون نص الاقتباس المحدد موجودًا، يتخطى OpenClaw بث معاينة الإجابة لذلك الدور بحيث لا يمكن عرض أسطر معاينة تقدم الأدوات. تظل ردود الرسالة الحالية التي لا تحتوي على نص اقتباس محدد محتفظة ببث المعاينة. راجع [وثائق قناة Telegram](/ar/channels/telegram) للتفاصيل.

أبقِ أسطر التقدم مرئية مع إخفاء نص الأمر/التنفيذ الخام:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "partial",
        "preview": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```

استخدم الشكل نفسه تحت مفتاح قناة تقدم مضغوط آخر، على سبيل المثال `channels.discord` أو `channels.matrix` أو `channels.msteams` أو `channels.mattermost` أو معاينات مسودات Slack. بالنسبة إلى وضع مسودة التقدم، ضع السياسة نفسها تحت `streaming.progress`:

```json
{
  "channels": {
    "telegram": {
      "streaming": {
        "mode": "progress",
        "progress": {
          "toolProgress": true,
          "commandText": "status"
        }
      }
    }
  }
}
```
