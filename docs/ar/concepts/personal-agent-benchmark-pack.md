حزمة معايير الوكيل الشخصي هي حزمة صغيرة من سيناريوهات ضمان الجودة مدعومة بمستودع لسير عمل المساعد الشخصي المحلي. وهي ليست معيارًا عامًا للنماذج ولا تحتاج إلى مشغّل جديد؛ إذ تعيد استخدام منظومة ضمان الجودة الخاصة ([نظرة عامة على ضمان الجودة](/ar/concepts/qa-e2e-automation))، و[قناة ضمان الجودة](/ar/channels/qa-channel) الاصطناعية، وفهرس YAML الحالي في `qa/scenarios`.

## السيناريوهات

عشرة سيناريوهات، معرّفة في `qa/scenarios/personal/*.yaml`:

| معرّف السيناريو                             | عمليات التحقق                                                                                           |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------- |
| `personal-reminder-roundtrip`              | تذكيرات شخصية وهمية عبر التسليم المحلي باستخدام Cron                                                   |
| `personal-channel-thread-reply`            | توجيه رسائل مباشرة وهمية وردود سلاسل المحادثات عبر `qa-channel`                                        |
| `personal-memory-preference-recall`        | استرجاع تفضيلات وهمية من ملفات الذاكرة في مساحة عمل ضمان الجودة المؤقتة                                |
| `personal-redaction-no-secret-leak`        | عمليات تحقق وهمية من عدم إعادة عرض الأسرار                                                              |
| `personal-tool-safety-followthrough`       | متابعة آمنة باستخدام أداة مدعومة بالقراءة بعد تفاعل قصير بأسلوب طلب الموافقة                            |
| `personal-approval-denial-stop`            | سلوك التوقف عند رفض الموافقة على طلب حساس للقراءة المحلية                                               |
| `personal-task-followthrough-status`       | الإبلاغ عن حالة المهمة استنادًا إلى أدلة مع إبقاء الحالات المعلّقة والمحظورة والمكتملة منفصلة           |
| `personal-share-safe-diagnostics-artifact` | عناصر تشخيصية آمنة للمشاركة تحافظ على الحالة المفيدة مع حذف المحتوى الشخصي الخام                        |
| `personal-no-fake-progress`                | ادعاءات إكمال مستندة إلى أدلة تتجنب إظهار تقدم زائف قبل توفر أدلة محلية                                 |
| `personal-failure-recovery`                | التعافي من الفشل مع الإبلاغ عن الحالة الجزئية والحفاظ على وضوح حدود إعادة المحاولة                       |

توجد بيانات الحزمة الوصفية القابلة للقراءة آليًا (قائمة المعرّفات والعنوان والوصف) في `extensions/qa-lab/src/scenario-packs.ts` تحت الاسم `QA_PERSONAL_AGENT_SCENARIO_IDS`.
شغّل الحزمة باستخدام `--pack personal-agent`:

```bash
OPENCLAW_ENABLE_PRIVATE_QA_CLI=1 pnpm openclaw qa suite \
  --provider-mode mock-openai \
  --pack personal-agent \
  --concurrency 1
```

يعمل `--pack` بصورة تراكمية مع علامات `--scenario` المتكررة. تُشغَّل السيناريوهات المحددة صراحةً أولًا، ثم تُشغَّل سيناريوهات الحزمة وفق ترتيب `QA_PERSONAL_AGENT_SCENARIO_IDS` بعد إزالة التكرارات.

تستهدف الحزمة `qa-channel` باستخدام `mock-openai` أو مسار موفّر محلي آخر لضمان الجودة. لا توجّهها إلى خدمات محادثة مباشرة أو حسابات شخصية حقيقية.

## نموذج الخصوصية

لا تستخدم السيناريوهات إلا مستخدمين وهميين وتفضيلات وهمية وأسرارًا وهمية ومساحة عمل Gateway المؤقتة لضمان الجودة التي تنشئها الحزمة. ويجب ألا تقرأ ذاكرة مستخدم OpenClaw الحقيقية أو جلساته أو بيانات اعتماده أو وكلاء التشغيل أو الإعدادات العامة أو حالة Gateway المباشرة، وألا تكتب فيها.

تبقى العناصر الناتجة ضمن دليل عناصر حزمة ضمان الجودة الحالي وتُعامل كمخرجات اختبار. تستخدم عمليات التحقق من التنقيح علامات وهمية، لذا يكون فحص حالات الفشل وتوثيقها في المشكلات آمنًا.

## توسيع الحزمة

أضف حالات `.yaml` جديدة ضمن `qa/scenarios/personal/`، ثم أضف معرّف السيناريو إلى `QA_PERSONAL_AGENT_SCENARIO_IDS`. اجعل كل حالة صغيرة ومحلية وحتمية عند استخدام `mock-openai` ومركزة على سلوك واحد للمساعد الشخصي.

من المرشحين الجيدين للمتابعة: عمليات التحقق من تصدير مسارات منقّحة، وعمليات التحقق من سير عمل Plugin محلي فقط.

تجنب إضافة مشغّل جديد أو Plugin أو تبعية أو وسيلة نقل مباشرة أو نموذج تحكيم إلى أن يتضمن فهرس السيناريوهات عددًا كافيًا من الحالات المستقرة يبرر إضافة ذلك السطح.
