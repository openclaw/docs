تتيح مسارات التخصص المتوازية لـ Gateway واحد توجيه محادثات أو غرف مختلفة إلى
وكلاء مختلفين، مع إبقاء تجربة المستخدم سريعة. تكمن الحيلة في التعامل مع
التوازي بوصفه مشكلة تصميم لمورد نادر، وليس فقط باعتباره "مزيدا من الوكلاء".

## المبادئ الأولى

لا يحسن مسار التخصص معدل الإنتاج إلا عندما يقلل التنازع على
الاختناقات الحقيقية:

- **أقفال الجلسات**: يجب أن يعدل تشغيل واحد فقط جلسة معينة في كل مرة.
- **السعة العامة للنموذج**: لا تزال كل تشغيلات المحادثة المرئية تشترك في حدود المزوّد.
- **سعة الأدوات**: يمكن أن يكون عمل الصدفة والمتصفح والشبكة والمستودع أبطأ
  من دورة النموذج نفسها.
- **ميزانية السياق**: تجعل النصوص الطويلة كل دورة مستقبلية أبطأ وأقل
  تركيزا.
- **غموض الملكية**: الوكلاء المكررون الذين يؤدون المهمة نفسها يهدرون السعة.

يسلسل OpenClaw بالفعل التشغيلات لكل جلسة ويحد التوازي العام من خلال
[طابور الأوامر](/ar/concepts/queue). تضيف مسارات التخصص سياسة فوق ذلك:
أي وكيل يملك أي عمل، وما الذي يبقى في المحادثة، وما الذي يصبح عملا في الخلفية.

## الطرح الموصى به

### المرحلة 1: عقود المسارات + العمل الثقيل في الخلفية

امنح كل مسار عقدا مكتوبا في مساحة عمله وموجه النظام الخاص به:

- **الغرض**: العمل الذي يملكه هذا المسار.
- **غير الأهداف**: العمل الذي يجب أن يمرره بدلا من محاولة تنفيذه.
- **ميزانية المحادثة**: تبقى الإجابات السريعة في المحادثة؛ أما المهام الطويلة فيجب أن تقر
  بإيجاز، ثم تعمل في وكيل فرعي أو مهمة في الخلفية.
- **قاعدة التسليم**: عندما يملك مسار آخر العمل، قل إلى أين يجب أن يذهب
  وقدم ملخص تسليم مضغوطا.
- **قاعدة مخاطر الأدوات**: فضّل أصغر سطح أدوات قادر على إنجاز المهمة.

هذه هي المرحلة الأقل تكلفة وتصلح معظم حالات الانسداد: لم تعد مهمة ترميز واحدة
تحول مسار البحث إلى مسار شديد البطء، وتحافظ كل محادثة على نظافة سياقها الخاص.

### المرحلة 2: عناصر التحكم في الأولوية والتزامن

اضبط الطابور وسعة النموذج حول قيمة الأعمال لكل مسار:

```json5
{
  agents: {
    defaults: {
      maxConcurrent: 4,
      subagents: { maxConcurrent: 8, delegationMode: "prefer" },
    },
  },
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
    },
  },
}
```

استخدم المحادثات المباشرة/الشخصية ووكلاء عمليات الإنتاج للعمل عالي الأولوية. دع
البحث والصياغة والترميز على دفعات تنتقل إلى مهام الخلفية عندما يكون النظام
مشغولا.

### المرحلة 3: المنسق / متحكم المرور

أضف نمط منسق صغيرا بمجرد تنشيط مسارات متعددة:

- تتبع مهام المسارات النشطة ومالكيها.
- اكتشف الطلبات المكررة عبر المجموعات.
- وجّه ملخصات التسليم بين المسارات.
- اعرض فقط العوائق والنتائج المكتملة والقرارات التي يجب على الإنسان اتخاذها.

لا تبدأ من هنا. المنسق بلا عقود مسارات لا ينسق إلا الفوضى.

## قالب عقد مسار بسيط

```md
# Lane contract

## Owns

- <job this lane is responsible for>

## Does not own

- <work to hand off>

## Chat budget

- Answer quick questions directly.
- For multi-step, slow, or tool-heavy work: acknowledge briefly, spawn/background
  the work, then return the result when complete.

## Handoff

If another lane owns the request, reply with:

- target lane
- objective
- relevant context
- exact next action

## Tool posture

Use the smallest tool surface that can complete the task. Avoid broad shell or
network work unless this lane explicitly owns it.
```
