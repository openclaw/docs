**بيئة تشغيل الوكيل** هي المكوّن الذي يملك حلقة نموذج واحدة مُحضّرة: فهي
تتلقى الموجّه، وتقود مخرجات النموذج، وتتولى استدعاءات الأدوات الأصلية، وتعيد
الدورة المكتملة إلى OpenClaw.

يسهل الخلط بين بيئات التشغيل والمزوّدين لأن كليهما يظهر بالقرب من
إعدادات النموذج. إنهما طبقتان مختلفتان:

| الطبقة         | أمثلة                                 | ما تعنيه                                                            |
| ------------- | ------------------------------------- | ------------------------------------------------------------------- |
| المزوّد        | `openai`, `anthropic`, `openai-codex` | كيف يصادق OpenClaw ويكتشف النماذج ويسمّي مراجع النماذج.             |
| النموذج        | `gpt-5.5`, `claude-opus-4-6`          | النموذج المحدد لدورة الوكيل.                                        |
| بيئة تشغيل الوكيل | `pi`, `codex`, `claude-cli`           | الحلقة أو الخلفية منخفضة المستوى التي تنفّذ الدورة المُحضّرة.        |
| القناة         | Telegram, Discord, Slack, WhatsApp    | حيث تدخل الرسائل إلى OpenClaw وتخرج منه.                            |

سترى أيضًا كلمة **حاضنة** في الشيفرة. الحاضنة هي التنفيذ الذي يوفّر
بيئة تشغيل وكيل. على سبيل المثال، تنفّذ حاضنة Codex المضمّنة
بيئة التشغيل `codex`. يستخدم الإعداد العام `agentRuntime.id`؛ ويعيد `openclaw
doctor --fix` كتابة مفاتيح سياسة بيئة التشغيل الأقدم إلى هذا الشكل.

هناك عائلتان من بيئات التشغيل:

- **الحاضنات المضمّنة** تعمل داخل حلقة الوكيل المُحضّرة في OpenClaw. حاليًا هذه
  هي بيئة التشغيل المدمجة `pi` بالإضافة إلى حاضنات Plugin المسجلة مثل
  `codex`.
- **الخلفيات عبر CLI** تشغّل عملية CLI محلية مع إبقاء مرجع النموذج
  قانونيًا. على سبيل المثال، `anthropic/claude-opus-4-7` مع
  `agentRuntime.id: "claude-cli"` تعني "حدّد نموذج Anthropic، ونفّذ
  عبر Claude CLI." إن `claude-cli` ليس معرّف حاضنة مضمّنة ويجب ألا
  يُمرّر إلى اختيار AgentHarness.

## أسطح Codex

يأتي معظم الالتباس من عدة أسطح مختلفة تشترك في اسم Codex:

| السطح                                                | اسم/إعداد OpenClaw                         | ما يفعله                                                                                                 |
| ---------------------------------------------------- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------- |
| بيئة تشغيل خادم تطبيق Codex الأصلية                 | `openai/*` بالإضافة إلى `agentRuntime.id: "codex"` | تشغّل دورة الوكيل المضمّنة عبر خادم تطبيق Codex. هذا هو إعداد اشتراك ChatGPT/Codex المعتاد.              |
| مسار مزوّد OAuth في Codex                            | مراجع نماذج `openai-codex/*`                | يستخدم OAuth لاشتراك ChatGPT/Codex عبر مشغّل OpenClaw PI العادي.                                          |
| محوّل ACP في Codex                                   | `runtime: "acp"`, `agentId: "codex"`       | يشغّل Codex عبر مستوى التحكم الخارجي ACP/acpx. استخدمه فقط عندما يُطلب ACP/acpx صراحة.                    |
| مجموعة أوامر التحكم بالمحادثة الأصلية في Codex       | `/codex ...`                               | تربط وتستأنف وتوجّه وتوقف وتفحص سلاسل خادم تطبيق Codex من المحادثة.                                      |
| مسار OpenAI Platform API لنماذج GPT/Codex النمطية    | مراجع نماذج `openai/*`                      | يستخدم مصادقة مفتاح OpenAI API ما لم يشغّل تجاوز بيئة تشغيل، مثل `agentRuntime.id: "codex"`، الدورة.     |

هذه الأسطح مستقلة عمدًا. تمكين Plugin `codex` يجعل
ميزات خادم التطبيق الأصلية متاحة؛ ولا يعيد كتابة
`openai-codex/*` إلى `openai/*`، ولا يغيّر الجلسات الحالية، ولا
يجعل ACP افتراضي Codex. اختيار `openai-codex/*` يعني "استخدم مسار
مزوّد OAuth في Codex" ما لم تفرض بيئة تشغيل بشكل منفصل.

يستخدم إعداد اشتراك ChatGPT/Codex الشائع OAuth في Codex للمصادقة، لكنه يُبقي
مرجع النموذج بصيغة `openai/*` ويختار بيئة التشغيل `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

هذا يعني أن OpenClaw يحدد مرجع نموذج OpenAI، ثم يطلب من بيئة تشغيل خادم تطبيق
Codex تشغيل دورة الوكيل المضمّنة. ولا يعني "استخدم فوترة API"، ولا
يعني أن القناة أو كتالوج مزوّد النماذج أو مخزن جلسات OpenClaw
يصبح Codex.

عند تمكين Plugin `codex` المضمّن، يجب أن يستخدم التحكم الطبيعي باللغة في Codex
سطح أوامر `/codex` الأصلي (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`) بدلًا من ACP. استخدم ACP مع
Codex فقط عندما يطلب المستخدم صراحة ACP/acpx أو يختبر مسار محوّل ACP.
ما تزال Claude Code وGemini CLI وOpenCode وCursor والحاضنات الخارجية
المشابهة تستخدم ACP.

هذه هي شجرة القرار الموجهة للوكيل:

1. إذا طلب المستخدم **ربط/تحكم/سلسلة/استئناف/توجيه/إيقاف Codex**، فاستخدم
   سطح أوامر `/codex` الأصلي عندما يكون Plugin `codex` المضمّن مفعّلًا.
2. إذا طلب المستخدم **Codex كبيئة تشغيل مضمّنة** أو أراد تجربة وكيل Codex
   العادية المدعومة بالاشتراك، فاستخدم
   `openai/<model>` مع `agentRuntime.id: "codex"`.
3. إذا طلب المستخدم **مصادقة OAuth/الاشتراك في Codex على مشغّل OpenClaw
   العادي**، فاستخدم `openai-codex/<model>` واترك بيئة التشغيل كـ PI.
4. إذا قال المستخدم صراحة **ACP** أو **acpx** أو **محوّل Codex ACP**، فاستخدم
   ACP مع `runtime: "acp"` و`agentId: "codex"`.
5. إذا كان الطلب يخص **Claude Code أو Gemini CLI أو OpenCode أو Cursor أو Droid أو
   حاضنة خارجية أخرى**، فاستخدم ACP/acpx، وليس بيئة تشغيل الوكيل الفرعي الأصلية.

| ما تقصده...                            | استخدم...                                    |
| -------------------------------------- | -------------------------------------------- |
| تحكم محادثة/سلسلة خادم تطبيق Codex     | `/codex ...` من Plugin `codex` المضمّن       |
| بيئة تشغيل الوكيل المضمّنة لخادم تطبيق Codex | `agentRuntime.id: "codex"`                   |
| OAuth في OpenAI Codex على مشغّل PI     | مراجع نماذج `openai-codex/*`                 |
| Claude Code أو حاضنة خارجية أخرى       | ACP/acpx                                     |

للاطلاع على تقسيم بادئات عائلة OpenAI، راجع [OpenAI](/ar/providers/openai) و
[مزوّدي النماذج](/ar/concepts/model-providers). وبالنسبة إلى عقد دعم بيئة تشغيل
Codex، راجع [حاضنة Codex](/ar/plugins/codex-harness#v1-support-contract).

## ملكية بيئة التشغيل

تملك بيئات التشغيل المختلفة مقادير مختلفة من الحلقة.

| السطح                       | PI المضمّن في OpenClaw                  | خادم تطبيق Codex                                                            |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| مالك حلقة النموذج           | OpenClaw عبر مشغّل PI المضمّن           | خادم تطبيق Codex                                                            |
| حالة السلسلة القانونية      | نص OpenClaw                             | سلسلة Codex، بالإضافة إلى مرآة نص OpenClaw                                  |
| أدوات OpenClaw الديناميكية  | حلقة أدوات OpenClaw الأصلية             | تُجسر عبر محوّل Codex                                                        |
| أدوات الطرفية والملفات الأصلية | مسار PI/OpenClaw                        | أدوات Codex الأصلية، تُجسر عبر الخطافات الأصلية حيث يكون ذلك مدعومًا        |
| محرك السياق                 | تجميع سياق OpenClaw الأصلي              | يجمّع OpenClaw سياق المشاريع داخل دورة Codex                                 |
| Compaction                  | OpenClaw أو محرك السياق المحدد          | Compaction أصلية في Codex، مع إشعارات OpenClaw وصيانة المرآة                 |
| تسليم القناة                | OpenClaw                                | OpenClaw                                                                    |

هذا التقسيم في الملكية هو قاعدة التصميم الرئيسية:

- إذا كان OpenClaw يملك السطح، فيمكن لـ OpenClaw توفير سلوك خطافات Plugin العادي.
- إذا كانت بيئة التشغيل الأصلية تملك السطح، يحتاج OpenClaw إلى أحداث بيئة التشغيل أو خطافات أصلية.
- إذا كانت بيئة التشغيل الأصلية تملك حالة السلسلة القانونية، فيجب على OpenClaw أن يعكس السياق ويسقطه، لا أن يعيد كتابة داخليات غير مدعومة.

## اختيار بيئة التشغيل

يختار OpenClaw بيئة تشغيل مضمّنة بعد حلّ المزوّد والنموذج:

1. تفوز بيئة التشغيل المسجلة للجلسة. لا تبدّل تغييرات الإعداد نصًا
   حاليًا إلى نظام سلاسل أصلي مختلف بشكل فوري.
2. يفرض `OPENCLAW_AGENT_RUNTIME=<id>` بيئة التشغيل هذه للجلسات الجديدة أو المعاد ضبطها.
3. يمكن أن يضبط `agents.defaults.agentRuntime.id` أو `agents.list[].agentRuntime.id`
   `auto` أو `pi` أو معرّف حاضنة مضمّنة مسجلة مثل `codex` أو
   اسمًا مستعارًا لخلفية CLI مدعومة مثل `claude-cli`.
4. في وضع `auto`، يمكن لبيئات تشغيل Plugin المسجلة أن تطالب بأزواج
   المزوّد/النموذج المدعومة.
5. إذا لم تطالب أي بيئة تشغيل بدورة في وضع `auto` وكان `fallback: "pi"` مضبوطًا
   (وهو الافتراضي)، يستخدم OpenClaw PI كرجوع توافق. اضبط
   `fallback: "none"` لجعل اختيار وضع `auto` غير المطابق يفشل بدلًا من ذلك.

تفشل بيئات تشغيل Plugin الصريحة بشكل مغلق افتراضيًا. على سبيل المثال،
`agentRuntime.id: "codex"` يعني Codex أو خطأ اختيار واضح ما لم تضبط
`fallback: "pi"` في نطاق التجاوز نفسه. لا يرث تجاوز بيئة التشغيل
إعداد رجوع أوسع، لذلك لا يُوجّه `agentRuntime.id: "codex"` على مستوى الوكيل
بصمت إلى PI فقط لأن الإعدادات الافتراضية استخدمت `fallback: "pi"`.

تختلف الأسماء المستعارة لخلفيات CLI عن معرّفات الحاضنات المضمّنة. الصيغة
المفضلة لـ Claude CLI هي:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

تبقى المراجع القديمة مثل `claude-cli/claude-opus-4-7` مدعومة من أجل
التوافق، لكن يجب أن يُبقي الإعداد الجديد المزوّد/النموذج قانونيًا وأن يضع
خلفية التنفيذ في `agentRuntime.id`.

وضع `auto` محافظ عمدًا. يمكن لبيئات تشغيل Plugin أن تطالب
بأزواج المزوّد/النموذج التي تفهمها، لكن Plugin Codex لا يطالب بمزوّد
`openai-codex` في وضع `auto`. هذا يبقي
`openai-codex/*` كمسار OAuth الصريح لـ PI Codex ويتجنب نقل
إعدادات مصادقة الاشتراك بصمت إلى حاضنة خادم التطبيق الأصلية.

إذا حذّر `openclaw doctor` من أن Plugin `codex` مفعّل بينما ما يزال
`openai-codex/*` يُوجّه عبر PI، فتعامل مع ذلك كتشخيص، لا كترحيل.
اترك الإعداد بلا تغيير عندما يكون OAuth في PI Codex هو ما تريده.
انتقل إلى `openai/<model>` بالإضافة إلى `agentRuntime.id: "codex"` فقط عندما تريد
تنفيذ خادم تطبيق Codex الأصلي.

## عقد التوافق

عندما لا تكون بيئة التشغيل PI، يجب أن توثق أسطح OpenClaw التي تدعمها.
استخدم هذا الشكل لوثائق بيئة التشغيل:

| السؤال                               | سبب أهميته                                                                                    |
| -------------------------------------- | ------------------------------------------------------------------------------------------------- |
| من يملك حلقة النموذج؟               | يحدد أين تحدث عمليات إعادة المحاولة، واستمرار الأدوات، وقرارات الإجابة النهائية.                   |
| من يملك سجل المحادثة الأساسي؟     | يحدد ما إذا كان بإمكان OpenClaw تعديل السجل أم نسخه فقط.                                   |
| هل تعمل أدوات OpenClaw الديناميكية؟        | تعتمد المراسلة، والجلسات، وcron، والأدوات المملوكة لـ OpenClaw على ذلك.                                 |
| هل تعمل خطافات الأدوات الديناميكية؟            | تتوقع Plugins وجود `before_tool_call` و`after_tool_call` ووسيط حول الأدوات المملوكة لـ OpenClaw. |
| هل تعمل خطافات الأدوات الأصلية؟             | تحتاج أدوات Shell وpatch والأدوات المملوكة لوقت التشغيل إلى دعم خطافات أصلية للسياسة والمراقبة.        |
| هل تعمل دورة حياة محرك السياق؟ | تعتمد Plugins الذاكرة والسياق على دورة حياة التجميع، والاستيعاب، وما بعد الدور، وCompaction.      |
| ما بيانات Compaction التي تُكشف؟       | تحتاج بعض Plugins إلى إشعارات فقط، بينما تحتاج أخرى إلى بيانات وصفية لما أُبقي عليه وما أُسقط.                    |
| ما غير المدعوم عمدًا؟     | يجب ألا يفترض المستخدمون تكافؤ PI عندما يملك وقت التشغيل الأصلي حالة أكثر.                  |

عقد دعم وقت تشغيل Codex موثق في
[حاضنة Codex](/ar/plugins/codex-harness#v1-support-contract).

## تسميات الحالة

قد يعرض خرج الحالة تسميتي `Execution` و`Runtime` معًا. اقرأهما على أنهما
تشخيصات، لا أسماء مزودين.

- مرجع نموذج مثل `openai/gpt-5.5` يخبرك بالمزود/النموذج المحدد.
- معرّف وقت تشغيل مثل `codex` يخبرك بأي حلقة تنفذ الدور.
- تسمية قناة مثل Telegram أو Discord تخبرك بمكان حدوث المحادثة.

إذا كانت الجلسة لا تزال تعرض PI بعد تغيير إعدادات وقت التشغيل، فابدأ جلسة جديدة
باستخدام `/new` أو امسح الحالية باستخدام `/reset`. تحتفظ الجلسات الحالية بوقت
التشغيل المسجل لها حتى لا يُعاد تشغيل النص عبر نظامي جلسات أصليين غير متوافقين.
