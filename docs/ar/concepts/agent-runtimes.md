**زمن تشغيل الوكيل** هو المكوّن الذي يملك حلقة نموذج واحدة مُعدّة: فهو
يتلقى الموجّه، ويقود مخرجات النموذج، ويتعامل مع استدعاءات الأدوات الأصلية،
ويعيد الدور المكتمل إلى OpenClaw.

يسهل الخلط بين أزمنة التشغيل والمزوّدين لأن كليهما يظهر قرب إعدادات
النموذج. إنهما طبقتان مختلفتان:

| الطبقة        | أمثلة                                 | ما تعنيه                                                             |
| ------------- | ------------------------------------- | ------------------------------------------------------------------- |
| المزوّد       | `openai`, `anthropic`, `openai-codex` | كيف يصادق OpenClaw، ويكتشف النماذج، ويسمّي مراجع النماذج.           |
| النموذج       | `gpt-5.5`, `claude-opus-4-6`          | النموذج المحدد لدور الوكيل.                                         |
| زمن تشغيل الوكيل | `pi`, `codex`, `claude-cli`           | الحلقة أو الخلفية منخفضة المستوى التي تنفّذ الدور المُعدّ.          |
| القناة        | Telegram, Discord, Slack, WhatsApp    | حيث تدخل الرسائل إلى OpenClaw وتخرج منه.                            |

سترى أيضًا كلمة **harness** في الكود. الـ harness هو التنفيذ الذي يوفّر زمن
تشغيل وكيل. على سبيل المثال، يطبّق Codex harness المضمّن زمن تشغيل `codex`.
تستخدم الإعدادات العامة `agentRuntime.id`؛ ويعيد `openclaw
doctor --fix` كتابة مفاتيح سياسة زمن التشغيل القديمة إلى هذا الشكل.

توجد عائلتان من أزمنة التشغيل:

- **الحاضنات المضمّنة** تعمل داخل حلقة الوكيل المُعدّة في OpenClaw. اليوم
  يشمل ذلك زمن التشغيل المدمج `pi` إضافة إلى حاضنات Plugin المسجّلة مثل
  `codex`.
- **خلفيات CLI** تشغّل عملية CLI محلية مع إبقاء مرجع النموذج معيارياً.
  على سبيل المثال، `anthropic/claude-opus-4-7` مع
  `agentRuntime.id: "claude-cli"` يعني "حدّد نموذج Anthropic، ونفّذ عبر
  Claude CLI." لا يُعدّ `claude-cli` معرّف harness مضمّنًا، ويجب عدم تمريره
  إلى اختيار AgentHarness.

## أسطح Codex

ينشأ معظم الالتباس من عدة أسطح مختلفة تشترك في اسم Codex:

| السطح                                               | اسم/إعداد OpenClaw                         | ما يفعله                                                                                                  |
| --------------------------------------------------- | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| زمن تشغيل خادم تطبيق Codex الأصلي                  | `openai/*` إضافة إلى `agentRuntime.id: "codex"` | يشغّل دور الوكيل المضمّن عبر خادم تطبيق Codex. هذا هو إعداد اشتراك ChatGPT/Codex المعتاد.                 |
| مسار مزوّد Codex OAuth                              | مراجع نماذج `openai-codex/*`               | يستخدم اشتراك ChatGPT/Codex OAuth عبر مشغّل OpenClaw PI العادي.                                           |
| محوّل Codex ACP                                    | `runtime: "acp"`, `agentId: "codex"`       | يشغّل Codex عبر مستوى تحكم ACP/acpx الخارجي. استخدمه فقط عندما يُطلب ACP/acpx صراحة.                    |
| مجموعة أوامر تحكم دردشة Codex الأصلية              | `/codex ...`                               | تربط، وتستأنف، وتوجّه، وتوقف، وتفحص سلاسل خادم تطبيق Codex من الدردشة.                                  |
| مسار OpenAI Platform API لنماذج بأسلوب GPT/Codex   | مراجع نماذج `openai/*`                     | يستخدم مصادقة مفتاح OpenAI API ما لم يشغّل تجاوز زمن تشغيل، مثل `agentRuntime.id: "codex"`، الدور.       |

هذه الأسطح مستقلة عن قصد. يؤدي تفعيل Plugin `codex` إلى إتاحة ميزات خادم
التطبيق الأصلي؛ ولا يعيد كتابة `openai-codex/*` إلى `openai/*`، ولا يغيّر
الجلسات الحالية، ولا يجعل ACP الإعداد الافتراضي لـ Codex. يعني اختيار
`openai-codex/*` "استخدم مسار مزوّد Codex OAuth" ما لم تفرض زمن تشغيل
بشكل منفصل.

يستخدم إعداد اشتراك ChatGPT/Codex الشائع Codex OAuth للمصادقة، لكنه يبقي
مرجع النموذج بصيغة `openai/*` ويحدد زمن تشغيل `codex`:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

يعني ذلك أن OpenClaw يحدد مرجع نموذج OpenAI، ثم يطلب من زمن تشغيل خادم تطبيق
Codex تشغيل دور الوكيل المضمّن. لا يعني ذلك "استخدم فوترة API"، ولا يعني أن
القناة أو كتالوج مزوّدي النماذج أو مخزن جلسات OpenClaw يصبح Codex.

عند تفعيل Plugin `codex` المضمّن، يجب أن يستخدم التحكم في Codex باللغة
الطبيعية سطح أوامر `/codex` الأصلي (`/codex bind`, `/codex threads`,
`/codex resume`, `/codex steer`, `/codex stop`) بدلاً من ACP. استخدم ACP مع
Codex فقط عندما يطلب المستخدم ACP/acpx صراحة أو يختبر مسار محوّل ACP.
لا تزال Claude Code وGemini CLI وOpenCode وCursor والحاضنات الخارجية المشابهة
تستخدم ACP.

هذه شجرة القرار الموجهة للوكيل:

1. إذا طلب المستخدم **ربط/تحكم/سلسلة/استئناف/توجيه/إيقاف Codex**، فاستخدم
   سطح أوامر `/codex` الأصلي عندما يكون Plugin `codex` المضمّن مفعلاً.
2. إذا طلب المستخدم **Codex كزمن التشغيل المضمّن** أو أراد تجربة وكيل Codex
   العادية المدعومة بالاشتراك، فاستخدم `openai/<model>` مع
   `agentRuntime.id: "codex"`.
3. إذا طلب المستخدم **مصادقة Codex OAuth/الاشتراك على مشغّل OpenClaw
   العادي**، فاستخدم `openai-codex/<model>` واترك زمن التشغيل PI.
4. إذا قال المستخدم صراحة **ACP** أو **acpx** أو **محوّل Codex ACP**، فاستخدم
   ACP مع `runtime: "acp"` و`agentId: "codex"`.
5. إذا كان الطلب متعلقًا بـ **Claude Code أو Gemini CLI أو OpenCode أو Cursor
   أو Droid أو حاضنة خارجية أخرى**، فاستخدم ACP/acpx، وليس زمن تشغيل الوكيل
   الفرعي الأصلي.

| ما تقصده...                            | استخدم...                                    |
| -------------------------------------- | -------------------------------------------- |
| تحكم دردشة/سلسلة خادم تطبيق Codex      | `/codex ...` من Plugin `codex` المضمّن       |
| زمن تشغيل الوكيل المضمّن لخادم تطبيق Codex | `agentRuntime.id: "codex"`                   |
| OpenAI Codex OAuth على مشغّل PI        | مراجع نماذج `openai-codex/*`                |
| Claude Code أو حاضنة خارجية أخرى       | ACP/acpx                                     |

لتقسيم بادئات عائلة OpenAI، راجع [OpenAI](/ar/providers/openai) و
[مزوّدو النماذج](/ar/concepts/model-providers). لعقد دعم زمن تشغيل Codex، راجع
[Codex harness](/ar/plugins/codex-harness#v1-support-contract).

## ملكية زمن التشغيل

تمتلك أزمنة التشغيل المختلفة مقادير مختلفة من الحلقة.

| السطح                      | OpenClaw PI المضمّن                     | خادم تطبيق Codex                                                            |
| -------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| مالك حلقة النموذج          | OpenClaw عبر مشغّل PI المضمّن           | خادم تطبيق Codex                                                            |
| حالة السلسلة المعيارية     | نص OpenClaw                             | سلسلة Codex، إضافة إلى مرآة نص OpenClaw                                     |
| أدوات OpenClaw الديناميكية | حلقة أدوات OpenClaw الأصلية             | موصولة عبر محوّل Codex                                                       |
| أدوات الطرفية والملفات الأصلية | مسار PI/OpenClaw                        | أدوات Codex الأصلية، موصولة عبر الخطافات الأصلية حيث يكون ذلك مدعومًا       |
| محرك السياق                | تجميع سياق OpenClaw الأصلي              | يجمع OpenClaw سياق المشاريع داخل دور Codex                                  |
| Compaction                 | OpenClaw أو محرك السياق المحدد          | Compaction أصلي من Codex، مع إشعارات OpenClaw وصيانة المرآة                 |
| تسليم القنوات              | OpenClaw                                | OpenClaw                                                                    |

تقسيم الملكية هذا هو قاعدة التصميم الرئيسية:

- إذا كان OpenClaw يملك السطح، يستطيع OpenClaw توفير سلوك خطافات Plugin العادي.
- إذا كان زمن التشغيل الأصلي يملك السطح، يحتاج OpenClaw إلى أحداث زمن التشغيل أو خطافات أصلية.
- إذا كان زمن التشغيل الأصلي يملك حالة السلسلة المعيارية، فيجب على OpenClaw أن يعكس السياق ويسقطه، لا أن يعيد كتابة داخليات غير مدعومة.

## اختيار زمن التشغيل

يختار OpenClaw زمن تشغيل مضمّنًا بعد حلّ المزوّد والنموذج:

1. يفوز زمن التشغيل المسجل في الجلسة. لا تبدّل تغييرات الإعدادات نصًا موجودًا
   مباشرة إلى نظام سلاسل أصلي مختلف.
2. يفرض `OPENCLAW_AGENT_RUNTIME=<id>` زمن التشغيل ذلك للجلسات الجديدة أو المعاد ضبطها.
3. يمكن لـ `agents.defaults.agentRuntime.id` أو `agents.list[].agentRuntime.id`
   تعيين `auto` أو `pi` أو معرّف harness مضمّن مسجّل مثل `codex` أو اسمًا
   مستعارًا لخلفية CLI مدعومة مثل `claude-cli`.
4. في وضع `auto`، يمكن لأزمنة تشغيل Plugin المسجّلة المطالبة بأزواج المزوّد/النموذج المدعومة.
5. إذا لم يطالب أي زمن تشغيل بدور في وضع `auto`، يستخدم OpenClaw PI كزمن
   تشغيل توافق. استخدم معرّف زمن تشغيل صريحًا عندما يجب أن يكون التشغيل صارمًا.

تفشل أزمنة تشغيل Plugin الصريحة بإغلاق واضح. على سبيل المثال،
`agentRuntime.id: "codex"` يعني Codex أو خطأ اختيار/زمن تشغيل واضحًا؛ ولا
يُوجّه أبدًا بصمت إلى PI.

تختلف الأسماء المستعارة لخلفيات CLI عن معرّفات الحاضنات المضمّنة. الصيغة
المفضلة لـ Claude CLI هي:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

تبقى المراجع القديمة مثل `claude-cli/claude-opus-4-7` مدعومة للتوافق، لكن
يجب أن تبقي الإعدادات الجديدة المزوّد/النموذج معياريًا وأن تضع خلفية التنفيذ
في `agentRuntime.id`.

وضع `auto` محافظ عن قصد. يمكن لأزمنة تشغيل Plugin المطالبة بأزواج
المزوّد/النموذج التي تفهمها، لكن Plugin Codex لا يطالب بمزوّد
`openai-codex` في وضع `auto`. يبقي ذلك `openai-codex/*` بوصفه مسار Codex
OAuth الصريح عبر PI ويتجنب نقل إعدادات مصادقة الاشتراك بصمت إلى حاضنة خادم
التطبيق الأصلية.

إذا حذّر `openclaw doctor` من أن Plugin `codex` مفعّل بينما لا يزال
`openai-codex/*` يمر عبر PI، فتعامل مع ذلك كتشخيص، لا كهجرة. أبقِ الإعدادات
دون تغيير عندما يكون PI Codex OAuth هو ما تريده. انتقل إلى `openai/<model>`
مع `agentRuntime.id: "codex"` فقط عندما تريد تنفيذ خادم تطبيق Codex الأصلي.

## عقد التوافق

عندما لا يكون زمن التشغيل PI، يجب أن يوثّق أسطح OpenClaw التي يدعمها. استخدم
هذا الشكل لمستندات زمن التشغيل:

| السؤال                                | لماذا يهم                                                                                       |
| ------------------------------------- | ------------------------------------------------------------------------------------------------ |
| من يملك حلقة النموذج؟                 | يحدد أين تحدث إعادة المحاولة، ومتابعة الأدوات، وقرارات الإجابة النهائية.                         |
| من يملك سجل السلاسل المعياري؟         | يحدد ما إذا كان OpenClaw يستطيع تعديل السجل أم يعكسه فقط.                                       |
| هل تعمل أدوات OpenClaw الديناميكية؟   | تعتمد الرسائل والجلسات وcron والأدوات المملوكة لـ OpenClaw على ذلك.                             |
| هل تعمل خطافات الأدوات الديناميكية؟   | تتوقع Plugins وجود `before_tool_call` و`after_tool_call` والوسيط حول الأدوات المملوكة لـ OpenClaw. |
| هل تعمل خطافات الأدوات الأصلية؟       | تحتاج أدوات الطرفية والتصحيح والأدوات المملوكة لزمن التشغيل إلى دعم خطافات أصلية للسياسة والمراقبة. |
| هل تعمل دورة حياة محرك السياق؟        | تعتمد Plugins الذاكرة والسياق على دورة حياة التجميع، والاستيعاب، وما بعد الدور، وCompaction.     |
| ما بيانات Compaction المكشوفة؟        | تحتاج بعض Plugins إلى الإشعارات فقط، بينما تحتاج أخرى إلى بيانات وصفية لما احتُفظ به أو أُسقط.   |
| ما غير المدعوم عمدًا؟                 | يجب ألا يفترض المستخدمون التكافؤ مع PI حيث يمتلك زمن التشغيل الأصلي حالة أكثر.                   |

عقد دعم زمن تشغيل Codex موثّق في
[Codex harness](/ar/plugins/codex-harness#v1-support-contract).

## تسميات الحالة

قد يُظهر مخرج الحالة تسميتي `Execution` و`Runtime` معًا. اقرأهما
كتشخيصات، وليس كأسماء موفرين.

- مرجع نموذج مثل `openai/gpt-5.5` يخبرك بالموفر/النموذج المحدد.
- معرف وقت تشغيل مثل `codex` يخبرك بأي حلقة تنفذ الدور.
- تسمية قناة مثل Telegram أو Discord تخبرك بمكان حدوث المحادثة.

إذا ظلت الجلسة تُظهر PI بعد تغيير تهيئة وقت التشغيل، فابدأ جلسة جديدة
باستخدام `/new` أو امسح الجلسة الحالية باستخدام `/reset`. تحتفظ الجلسات الحالية
بوقت التشغيل المسجل لديها حتى لا يُعاد تشغيل النص عبر نظامي جلسات أصليين غير
متوافقين.
