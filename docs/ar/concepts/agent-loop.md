الحلقة الوكيلية هي التشغيل “الحقيقي” الكامل للوكيل: الإدخال → تجميع السياق → استدلال النموذج →
تنفيذ الأدوات → بث الردود → الاستمرارية. إنها المسار الرسمي الذي يحوّل الرسالة
إلى إجراءات ورد نهائي، مع الحفاظ على اتساق حالة الجلسة.

في OpenClaw، الحلقة هي تشغيل واحد متسلسل لكل جلسة يصدر أحداث دورة الحياة والبث
بينما يفكر النموذج، ويستدعي الأدوات، ويبث المخرجات. يشرح هذا المستند كيفية توصيل تلك الحلقة الأصلية
من البداية إلى النهاية.

## نقاط الدخول

- Gateway RPC: `agent` و`agent.wait`.
- CLI: أمر `agent`.

## كيف يعمل (نظرة عامة)

1. يتحقق `agent` RPC من المعلمات، ويحل الجلسة (sessionKey/sessionId)، ويحفظ بيانات تعريف الجلسة، ويعيد `{ runId, acceptedAt }` فورًا.
2. يشغّل `agentCommand` الوكيل:
   - يحل النموذج + الإعدادات الافتراضية للتفكير/الإسهاب/التتبع
   - يحمّل لقطة Skills
   - يستدعي `runEmbeddedPiAgent` (بيئة تشغيل pi-agent-core)
   - يصدر **نهاية/خطأ دورة الحياة** إذا لم تصدر الحلقة المضمنة واحدًا
3. `runEmbeddedPiAgent`:
   - يسلسل عمليات التشغيل عبر طوابير لكل جلسة + طوابير عامة
   - يحل النموذج + ملف تعريف المصادقة ويبني جلسة Pi
   - يشترك في أحداث Pi ويبث دلتا المساعد/الأداة
   - يفرض المهلة -> يلغي التشغيل إذا تم تجاوزها
   - بالنسبة لدورات خادم تطبيق Codex، يلغي دورة مقبولة تتوقف عن إنتاج تقدم خادم التطبيق قبل حدث نهائي
   - يعيد الحمولات + بيانات تعريف الاستخدام
4. يربط `subscribeEmbeddedPiSession` أحداث pi-agent-core ببث `agent` في OpenClaw:
   - أحداث الأدوات => `stream: "tool"`
   - دلتا المساعد => `stream: "assistant"`
   - أحداث دورة الحياة => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. يستخدم `agent.wait` الدالة `waitForAgentRun`:
   - ينتظر **نهاية/خطأ دورة الحياة** لـ `runId`
   - يعيد `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## الطوابير + التزامن

- تُسلسل عمليات التشغيل لكل مفتاح جلسة (مسار الجلسة) واختياريًا عبر مسار عام.
- يمنع ذلك سباقات الأدوات/الجلسات ويحافظ على اتساق سجل الجلسة.
- يمكن لقنوات المراسلة اختيار أوضاع الطوابير (collect/steer/followup) التي تغذي نظام المسارات هذا.
  راجع [طابور الأوامر](/ar/concepts/queue).
- تتم حماية كتابات النص الكامل أيضًا بواسطة قفل كتابة للجلسة على ملف الجلسة. القفل
  واعٍ بالعملية ومستند إلى الملفات، لذا يلتقط الكتّاب الذين يتجاوزون الطابور داخل العملية أو يأتون من
  عملية أخرى.
- أقفال كتابة الجلسة غير قابلة لإعادة الدخول افتراضيًا. إذا قام مساعد عمدًا بتداخل الحصول على
  القفل نفسه مع الحفاظ على كاتب منطقي واحد، فيجب أن يختار ذلك صراحةً باستخدام
  `allowReentrant: true`.

## إعداد الجلسة + مساحة العمل

- يتم حل مساحة العمل وإنشاؤها؛ وقد تعيد عمليات التشغيل المعزولة توجيهها إلى جذر مساحة عمل معزولة.
- يتم تحميل Skills (أو إعادة استخدامها من لقطة) وحقنها في البيئة والمطالبة.
- يتم حل ملفات التمهيد/السياق وحقنها في تقرير مطالبة النظام.
- يتم الحصول على قفل كتابة للجلسة؛ يتم فتح `SessionManager` وتجهيزه قبل البث. يجب على أي
  مسار لاحق لإعادة كتابة النص الكامل أو Compaction أو الاقتطاع أن يأخذ القفل نفسه قبل فتح
  ملف النص الكامل أو تعديله.

## تجميع المطالبة + مطالبة النظام

- تُبنى مطالبة النظام من المطالبة الأساسية لـ OpenClaw، ومطالبة Skills، وسياق التمهيد، والتجاوزات لكل تشغيل.
- تُفرض حدود النموذج المحددة ورموز احتياطي Compaction.
- راجع [مطالبة النظام](/ar/concepts/system-prompt) لمعرفة ما يراه النموذج.

## نقاط Hook (حيث يمكنك الاعتراض)

لدى OpenClaw نظاما Hook:

- **Hooks داخلية** (Gateway hooks): نصوص مدفوعة بالأحداث للأوامر وأحداث دورة الحياة.
- **Plugin hooks**: نقاط توسعة داخل دورة حياة الوكيل/الأداة ومسار Gateway.

### Hooks داخلية (Gateway hooks)

- **`agent:bootstrap`**: يعمل أثناء بناء ملفات التمهيد قبل الانتهاء من مطالبة النظام.
  استخدم هذا لإضافة/إزالة ملفات سياق التمهيد.
- **Command hooks**: `/new` و`/reset` و`/stop` وأحداث أوامر أخرى (راجع مستند Hooks).

راجع [Hooks](/ar/automation/hooks) للإعداد والأمثلة.

### Plugin hooks (دورة حياة الوكيل + Gateway)

تعمل هذه داخل حلقة الوكيل أو مسار Gateway:

- **`before_model_resolve`**: يعمل قبل الجلسة (بدون `messages`) لتجاوز المزوّد/النموذج بشكل حتمي قبل حل النموذج.
- **`before_prompt_build`**: يعمل بعد تحميل الجلسة (مع `messages`) لحقن `prependContext` أو `systemPrompt` أو `prependSystemContext` أو `appendSystemContext` قبل إرسال المطالبة. استخدم `prependContext` للنص الديناميكي لكل دورة، وحقول سياق النظام للإرشاد المستقر الذي يجب أن يوجد في مساحة مطالبة النظام.
- **`before_agent_start`**: Hook توافق قديم قد يعمل في أي من المرحلتين؛ فضّل Hooks الصريحة أعلاه.
- **`before_agent_reply`**: يعمل بعد الإجراءات المضمنة وقبل استدعاء LLM، مما يتيح لـ Plugin المطالبة بالدورة وإرجاع رد اصطناعي أو إسكات الدورة بالكامل.
- **`agent_end`**: افحص قائمة الرسائل النهائية وبيانات تعريف التشغيل بعد الاكتمال.
- **`before_compaction` / `after_compaction`**: راقب دورات Compaction أو علّق عليها.
- **`before_tool_call` / `after_tool_call`**: اعترض معلمات/نتائج الأدوات.
- **`before_install`**: افحص نتائج الفحص المضمنة ويمكنك اختياريًا حظر تثبيت Skills أو Plugin.
- **`tool_result_persist`**: حوّل نتائج الأدوات بشكل متزامن قبل كتابتها إلى نص جلسة تملكه OpenClaw.
- **`message_received` / `message_sending` / `message_sent`**: Hooks الرسائل الواردة + الصادرة.
- **`session_start` / `session_end`**: حدود دورة حياة الجلسة.
- **`gateway_start` / `gateway_stop`**: أحداث دورة حياة Gateway.

قواعد قرار Hooks لحراس الصادر/الأدوات:

- `before_tool_call`: `{ block: true }` نهائي ويوقف المعالجات ذات الأولوية الأدنى.
- `before_tool_call`: `{ block: false }` بلا تأثير ولا يمحو حظرًا سابقًا.
- `before_install`: `{ block: true }` نهائي ويوقف المعالجات ذات الأولوية الأدنى.
- `before_install`: `{ block: false }` بلا تأثير ولا يمحو حظرًا سابقًا.
- `message_sending`: `{ cancel: true }` نهائي ويوقف المعالجات ذات الأولوية الأدنى.
- `message_sending`: `{ cancel: false }` بلا تأثير ولا يمحو إلغاءً سابقًا.

راجع [Plugin hooks](/ar/plugins/hooks) للحصول على API الخاص بالـ Hook وتفاصيل التسجيل.

قد تكيف حزم الاختبار هذه Hooks بطرق مختلفة. تحافظ حزمة اختبار خادم تطبيق Codex على
OpenClaw plugin hooks كعقد توافق للأسطح المعكوسة الموثقة،
بينما تظل Hooks الأصلية في Codex آلية منفصلة ذات مستوى أدنى في Codex.

## البث + الردود الجزئية

- تُبث دلتا المساعد من pi-agent-core وتصدر كأحداث `assistant`.
- يمكن لبث الكتل إصدار ردود جزئية إما عند `text_end` أو `message_end`.
- يمكن إصدار بث الاستدلال كبث منفصل أو كردود كتل.
- راجع [البث](/ar/concepts/streaming) لسلوك التجزئة وردود الكتل.

## تنفيذ الأدوات + أدوات المراسلة

- تُصدر أحداث بدء/تحديث/انتهاء الأداة على بث `tool`.
- تُنظف نتائج الأدوات من حيث الحجم وحمولات الصور قبل التسجيل/الإصدار.
- يتم تتبع إرسال أدوات المراسلة لقمع تأكيدات المساعد المكررة.

## تشكيل الرد + القمع

- تُجمع الحمولات النهائية من:
  - نص المساعد (والاستدلال الاختياري)
  - ملخصات الأدوات المضمنة (عند السماح بالإسهاب + السماح)
  - نص خطأ المساعد عند حدوث خطأ في النموذج
- تتم تصفية الرمز الصامت الدقيق `NO_REPLY` / `no_reply` من الحمولات
  الصادرة.
- تتم إزالة تكرارات أدوات المراسلة من قائمة الحمولات النهائية.
- إذا لم تبق أي حمولات قابلة للعرض وحدث خطأ في أداة، يتم إصدار رد خطأ احتياطي للأداة
  (ما لم تكن أداة مراسلة قد أرسلت بالفعل ردًا مرئيًا للمستخدم).

## Compaction + إعادة المحاولات

- يصدر Compaction التلقائي أحداث بث `compaction` ويمكن أن يطلق إعادة محاولة.
- عند إعادة المحاولة، تتم إعادة تعيين المخازن المؤقتة في الذاكرة وملخصات الأدوات لتجنب تكرار المخرجات.
- راجع [Compaction](/ar/concepts/compaction) لمعرفة مسار Compaction.

## تدفقات الأحداث (اليوم)

- `lifecycle`: يصدره `subscribeEmbeddedPiSession` (وكاحتياطي بواسطة `agentCommand`)
- `assistant`: دلتا مبثوثة من pi-agent-core
- `tool`: أحداث أدوات مبثوثة من pi-agent-core

## معالجة قناة الدردشة

- يتم تخزين دلتا المساعد مؤقتًا في رسائل دردشة `delta`.
- يتم إصدار `final` للدردشة عند **نهاية/خطأ دورة الحياة**.

## المهل

- الافتراضي لـ `agent.wait`: 30 ثانية (الانتظار فقط). تتجاوز معلمة `timeoutMs` ذلك.
- وقت تشغيل الوكيل: الافتراضي لـ `agents.defaults.timeoutSeconds` هو 172800 ثانية (48 ساعة)؛ يُفرض في مؤقت إلغاء `runEmbeddedPiAgent`.
- وقت تشغيل Cron: يملك cron قيمة `timeoutSeconds` لدورة الوكيل المعزولة. يبدأ المجدول هذا المؤقت عند بدء التنفيذ، ويلغي التشغيل الأساسي عند الموعد النهائي المكوّن، ثم يجري تنظيفًا محدودًا قبل تسجيل المهلة حتى لا تتمكن جلسة فرعية قديمة من إبقاء المسار عالقًا.
- تشخيصات حيوية الجلسة: مع تمكين التشخيصات، يصنف `diagnostics.stuckSessionWarnMs` جلسات `processing` الطويلة التي لا تملك أي رد أو أداة أو حالة أو كتلة أو تقدم ACP مرصود. تُبلغ عمليات التشغيل المضمنة النشطة، واستدعاءات النموذج، واستدعاءات الأدوات كـ `session.long_running`؛ ويُبلغ العمل النشط دون تقدم حديث كـ `session.stalled`؛ ويُحجز `session.stuck` لمحاسبة الجلسات القديمة دون عمل نشط، وهذا المسار فقط يحرر مسار الجلسة المتأثر حتى يمكن تصريف عمل بدء التشغيل الموجود في الطابور. تتراجع تشخيصات `session.stuck` المتكررة بينما تظل الجلسة دون تغيير.
- مهلة خمول النموذج: يلغي OpenClaw طلب النموذج عندما لا تصل أي أجزاء استجابة قبل نافذة الخمول. يمدد `models.providers.<id>.timeoutSeconds` مراقب الخمول هذا للمزوّدين المحليين/المستضافين ذاتيًا البطيئين؛ وإلا يستخدم OpenClaw قيمة `agents.defaults.timeoutSeconds` عند تكوينها، وبحد أقصى 120 ثانية افتراضيًا. عمليات التشغيل المشغلة بواسطة Cron التي لا تملك مهلة نموذج أو وكيل صريحة تعطل مراقب الخمول وتعتمد على مهلة cron الخارجية.
- مهلة طلب HTTP للمزوّد: ينطبق `models.providers.<id>.timeoutSeconds` على جلب HTTP للنموذج الخاص بذلك المزوّد، بما في ذلك الاتصال، والترويسات، والجسم، ومهلة طلب SDK، ومعالجة إلغاء الجلب المحروس الإجمالية، ومراقب خمول بث النموذج. استخدم هذا للمزوّدين المحليين/المستضافين ذاتيًا البطيئين مثل Ollama قبل رفع مهلة وقت تشغيل الوكيل بالكامل.

## أين يمكن أن تنتهي الأمور مبكرًا

- مهلة الوكيل (إلغاء)
- AbortSignal (إلغاء)
- انقطاع Gateway أو مهلة RPC
- مهلة `agent.wait` (انتظار فقط، لا توقف الوكيل)
