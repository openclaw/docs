# حلقة الوكيل (OpenClaw)

تمثل الحلقة الوكيلة التشغيل الكامل «الحقيقي» للوكيل: الاستقبال → تجميع السياق → استدلال النموذج →
تنفيذ الأدوات → بث الردود → الاستمرارية. وهي المسار المرجعي الذي يحوّل الرسالة
إلى إجراءات ورد نهائي، مع الحفاظ على اتساق حالة الجلسة.

في OpenClaw، تكون الحلقة تشغيلًا واحدًا متسلسلًا لكل جلسة، ويُصدر أحداث دورة الحياة وأحداث البث
بينما يفكر النموذج، ويستدعي الأدوات، ويبث المخرجات. توضح هذه الوثيقة كيف تُوصَّل هذه الحلقة
الحقيقية من البداية إلى النهاية.

## نقاط الدخول

- Gateway RPC: ‏`agent` و`agent.wait`.
- CLI: أمر `agent`.

## كيف تعمل (نظرة عالية المستوى)

1. يتحقق RPC ‏`agent` من صحة المعلمات، ويحلّل الجلسة (sessionKey/sessionId)، ويحفظ بيانات الجلسة الوصفية، ويعيد `{ runId, acceptedAt }` فورًا.
2. يشغّل `agentCommand` الوكيل:
   - يحلّل القيم الافتراضية للنموذج + التفكير/verbose/trace
   - يحمّل لقطة Skills
   - يستدعي `runEmbeddedPiAgent` (بيئة تشغيل pi-agent-core)
   - يُصدر **نهاية/خطأ دورة الحياة** إذا لم تصدر الحلقة المضمنة أيًا منهما
3. يقوم `runEmbeddedPiAgent` بما يلي:
   - يسلسل التشغيلات عبر قوائم انتظار لكل جلسة + قائمة انتظار عامة
   - يحلّل النموذج + ملف تعريف المصادقة ويبني جلسة pi
   - يشترك في أحداث pi ويبث فروق assistant/tool
   - يفرض مهلة زمنية -> ويوقف التشغيل إذا تم تجاوزها
   - يعيد الحمولات + بيانات الاستخدام الوصفية
4. يقوم `subscribeEmbeddedPiSession` بجسر أحداث pi-agent-core إلى بث `agent` في OpenClaw:
   - أحداث الأدوات => `stream: "tool"`
   - فروق المساعد => `stream: "assistant"`
   - أحداث دورة الحياة => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. يستخدم `agent.wait` الدالة `waitForAgentRun`:
   - ينتظر **نهاية/خطأ دورة الحياة** لـ `runId`
   - يعيد `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## الاصطفاف + التوازي

- تُسلسل التشغيلات لكل مفتاح جلسة (مسار الجلسة) ويمكن أيضًا تمريرها اختياريًا عبر مسار عام.
- يمنع هذا سباقات الأدوات/الجلسات ويحافظ على اتساق سجل الجلسة.
- يمكن لقنوات المراسلة اختيار أوضاع قائمة الانتظار (collect/steer/followup) التي تغذي نظام المسارات هذا.
  راجع [قائمة انتظار الأوامر](/ar/concepts/queue).
- كما أن كتابات النصوص المفرغة محمية أيضًا بقفل كتابة للجلسة على ملف الجلسة. هذا القفل
  واعٍ بالعمليات ومعتمد على الملفات، لذا يلتقط الكتّاب الذين يتجاوزون قائمة الانتظار داخل العملية أو يأتون من
  عملية أخرى.
- تكون أقفال كتابة الجلسة غير قابلة لإعادة الدخول افتراضيًا. وإذا كان مساعد ما يتعمد تداخل الحصول على
  القفل نفسه مع الحفاظ على كاتب منطقي واحد، فيجب عليه الاشتراك صراحةً باستخدام
  `allowReentrant: true`.

## إعداد الجلسة + مساحة العمل

- يتم تحليل مساحة العمل وإنشاؤها؛ وقد تعيد التشغيلات المعزولة التوجيه إلى جذر مساحة عمل sandbox.
- يتم تحميل Skills (أو إعادة استخدامها من لقطة) وحقنها في البيئة والمطالبة.
- يتم تحليل ملفات bootstrap/context وحقنها في تقرير system prompt.
- يتم الحصول على قفل كتابة للجلسة؛ ويتم فتح `SessionManager` وتجهيزه قبل البث. وأي
  مسار لاحق لإعادة كتابة النص المفرغ أو Compaction أو الاقتطاع يجب أن يحصل على القفل نفسه قبل فتح
  ملف النص المفرغ أو تعديله.

## تجميع المطالبة + system prompt

- يتم بناء system prompt من المطالبة الأساسية لـ OpenClaw، ومطالبة Skills، وسياق bootstrap، وتجاوزات كل تشغيل.
- يتم فرض الحدود الخاصة بالنموذج ورموز الاحتياطي الخاصة بـ Compaction.
- راجع [System prompt](/ar/concepts/system-prompt) لمعرفة ما يراه النموذج.

## نقاط الخطافات (أين يمكنك الاعتراض)

يحتوي OpenClaw على نظامي hooks:

- **الخطافات الداخلية** (خطافات Gateway): نصوص برمجية مدفوعة بالأحداث للأوامر وأحداث دورة الحياة.
- **خطافات Plugin**: نقاط امتداد داخل دورة حياة الوكيل/الأداة وخط أنابيب gateway.

### الخطافات الداخلية (خطافات Gateway)

- **`agent:bootstrap`**: يعمل أثناء بناء ملفات bootstrap قبل إنهاء system prompt.
  استخدمه لإضافة/إزالة ملفات سياق bootstrap.
- **خطافات الأوامر**: ‏`/new` و`/reset` و`/stop` وأحداث الأوامر الأخرى (راجع وثيقة Hooks).

راجع [Hooks](/ar/automation/hooks) للإعداد والأمثلة.

### خطافات Plugin (دورة حياة الوكيل + gateway)

تعمل هذه داخل حلقة الوكيل أو خط أنابيب gateway:

- **`before_model_resolve`**: يعمل قبل الجلسة (من دون `messages`) لتجاوز provider/model بشكل حتمي قبل تحليل النموذج.
- **`before_prompt_build`**: يعمل بعد تحميل الجلسة (مع `messages`) لحقن `prependContext` أو `systemPrompt` أو `prependSystemContext` أو `appendSystemContext` قبل إرسال المطالبة. استخدم `prependContext` للنص الديناميكي لكل دور، واستخدم حقول سياق النظام للإرشاد الثابت الذي يجب أن يوجد ضمن مساحة system prompt.
- **`before_agent_start`**: خطاف توافق قديم قد يعمل في أي من المرحلتين؛ ويفضَّل الخطافات الصريحة أعلاه.
- **`before_agent_reply`**: يعمل بعد الإجراءات المضمنة وقبل استدعاء LLM، مما يتيح لـ Plugin المطالبة بالدور وإرجاع رد اصطناعي أو إسكات الدور بالكامل.
- **`agent_end`**: يفحص قائمة الرسائل النهائية وبيانات التشغيل الوصفية بعد الاكتمال.
- **`before_compaction` / `after_compaction`**: لمراقبة دورات Compaction أو إضافة تعليقات عليها.
- **`before_tool_call` / `after_tool_call`**: لاعتراض معلمات الأدوات/نتائجها.
- **`before_install`**: لفحص نتائج الفحص المضمنة واختيار حظر تثبيت Skill أو Plugin.
- **`tool_result_persist`**: لتحويل نتائج الأدوات تزامنيًا قبل كتابتها إلى نص جلسة مملوك لـ OpenClaw.
- **`message_received` / `message_sending` / `message_sent`**: خطافات الرسائل الواردة + الصادرة.
- **`session_start` / `session_end`**: حدود دورة حياة الجلسة.
- **`gateway_start` / `gateway_stop`**: أحداث دورة حياة gateway.

قواعد قرار الخطافات لحراس الإرسال/الأدوات الصادرة:

- `before_tool_call`: تكون `{ block: true }` نهائية وتوقف المعالجات ذات الأولوية الأدنى.
- `before_tool_call`: تكون `{ block: false }` بلا أثر ولا تمحو حظرًا سابقًا.
- `before_install`: تكون `{ block: true }` نهائية وتوقف المعالجات ذات الأولوية الأدنى.
- `before_install`: تكون `{ block: false }` بلا أثر ولا تمحو حظرًا سابقًا.
- `message_sending`: تكون `{ cancel: true }` نهائية وتوقف المعالجات ذات الأولوية الأدنى.
- `message_sending`: تكون `{ cancel: false }` بلا أثر ولا تمحو إلغاءً سابقًا.

راجع [خطافات Plugin](/ar/plugins/architecture-internals#provider-runtime-hooks) للحصول على API الخاص بالخطافات وتفاصيل التسجيل.

قد تكيّف harnesses هذه الخطافات بشكل مختلف. ويحافظ Codex app-server harness على
خطافات Plugin في OpenClaw بوصفها عقد التوافق للأسطح المعكوسة الموثقة،
بينما تبقى خطافات Codex الأصلية آلية Codex منفصلة منخفضة المستوى.

## البث + الردود الجزئية

- تُبث فروق المساعد من pi-agent-core وتُصدر كأحداث `assistant`.
- يمكن للبث على مستوى الكتلة إصدار ردود جزئية إما عند `text_end` أو `message_end`.
- يمكن إصدار بث الاستدلال كبث منفصل أو كردود على مستوى الكتلة.
- راجع [البث](/ar/concepts/streaming) لمعرفة سلوك التقطيع وردود الكتلة.

## تنفيذ الأدوات + أدوات المراسلة

- تُصدر أحداث بدء/تحديث/نهاية الأداة على بث `tool`.
- تُنقّى نتائج الأدوات من حيث الحجم وحمولات الصور قبل التسجيل/الإصدار.
- يتم تتبع عمليات إرسال أدوات المراسلة لكبت تأكيدات المساعد المكررة.

## تشكيل الرد + الكبت

- يتم تجميع الحمولات النهائية من:
  - نص المساعد (واختياريًا الاستدلال)
  - ملخصات الأدوات المضمنة (عند تفعيل verbose والسماح بها)
  - نص خطأ المساعد عند حدوث خطأ في النموذج
- تتم تصفية الرمز الصامت الدقيق `NO_REPLY` / `no_reply` من
  الحمولات الصادرة.
- تتم إزالة تكرارات أدوات المراسلة من قائمة الحمولات النهائية.
- إذا لم تتبقَّ أي حمولات قابلة للعرض وحدث خطأ في أداة ما، فسيتم إصدار
  رد احتياطي لخطأ الأداة (ما لم تكن أداة مراسلة قد أرسلت بالفعل ردًا مرئيًا للمستخدم).

## Compaction + إعادة المحاولة

- يُصدر Compaction التلقائي أحداث بث `compaction` ويمكنه تشغيل إعادة محاولة.
- عند إعادة المحاولة، تتم إعادة تعيين المخازن داخل الذاكرة وملخصات الأدوات لتجنب المخرجات المكررة.
- راجع [Compaction](/ar/concepts/compaction) لمعرفة خط أنابيب Compaction.

## تدفقات الأحداث (حاليًا)

- `lifecycle`: يصدره `subscribeEmbeddedPiSession` (وكاحتياط من `agentCommand`)
- `assistant`: فروق مبثوثة من pi-agent-core
- `tool`: أحداث أدوات مبثوثة من pi-agent-core

## معالجة قناة الدردشة

- تُخزن فروق المساعد مؤقتًا في رسائل دردشة `delta`.
- يتم إصدار `final` للدردشة عند **نهاية/خطأ دورة الحياة**.

## المهل الزمنية

- `agent.wait` الافتراضي: 30 ثانية (للانتظار فقط). وتؤدي المعلمة `timeoutMs` إلى تجاوزه.
- وقت تشغيل الوكيل: الافتراضي لـ `agents.defaults.timeoutSeconds` هو 172800 ثانية (48 ساعة)؛ ويتم فرضه في مؤقت الإيقاف داخل `runEmbeddedPiAgent`.
- مهلة خمول LLM: يقوم `agents.defaults.llm.idleTimeoutSeconds` بإيقاف طلب النموذج عندما لا تصل أي أجزاء استجابة قبل نافذة الخمول. اضبطها صراحةً للنماذج المحلية البطيئة أو مزوّدي الاستدلال/استدعاء الأدوات؛ واضبطها على 0 لتعطيلها. وإذا لم تُضبط، يستخدم OpenClaw القيمة `agents.defaults.timeoutSeconds` عند ضبطها، وإلا 120 ثانية. وتقوم التشغيلات المشغلة عبر Cron من دون مهلة صريحة لـ LLM أو الوكيل بتعطيل مراقب الخمول وتعتمد على المهلة الخارجية لـ Cron.

## أين يمكن أن تنتهي الأمور مبكرًا

- مهلة الوكيل (إيقاف)
- `AbortSignal` (إلغاء)
- انقطاع Gateway أو مهلة RPC
- مهلة `agent.wait` (للانتظار فقط، ولا توقف الوكيل)
