يتذكر OpenClaw الأشياء من خلال كتابة **ملفات Markdown عادية** في
مساحة عمل وكيلك. ولا «يتذكر» النموذج إلا ما يتم حفظه على القرص -- فلا توجد
أي حالة مخفية.

## كيف يعمل

يحتوي وكيلك على ثلاثة ملفات مرتبطة بالذاكرة:

- **`MEMORY.md`** -- الذاكرة طويلة الأمد. الحقائق والتفضيلات والقرارات
  الدائمة. ويتم تحميلها عند بداية كل جلسة رسالة مباشرة.
- **`memory/YYYY-MM-DD.md`** -- ملاحظات يومية. سياق وملاحظات جارية.
  ويتم تحميل ملاحظات اليوم والأمس تلقائيًا.
- **`DREAMS.md`** (اختياري) -- يوميات Dreaming وملخصات
  جولات Dreaming للمراجعة البشرية، بما في ذلك إدخالات التعبئة التاريخية المؤصلة.

توجد هذه الملفات في مساحة عمل الوكيل (الافتراضية `~/.openclaw/workspace`).

<Tip>
إذا كنت تريد أن يتذكر وكيلك شيئًا ما، فما عليك سوى أن تطلب منه ذلك: «تذكر أنني
أفضل TypeScript.» وسيكتب ذلك في الملف المناسب.
</Tip>

## أدوات الذاكرة

يمتلك الوكيل أداتين للعمل مع الذاكرة:

- **`memory_search`** -- يعثر على الملاحظات ذات الصلة باستخدام البحث الدلالي، حتى عندما
  تختلف الصياغة عن الأصل.
- **`memory_get`** -- يقرأ ملف ذاكرة محددًا أو نطاق أسطر.

توفَّر كلتا الأداتين بواسطة Plugin الذاكرة النشط (الافتراضي: `memory-core`).

## Plugin المرافق Memory Wiki

إذا كنت تريد أن تعمل الذاكرة الدائمة بشكل أقرب إلى قاعدة معرفة مُدارة جيدًا بدلًا
من مجرد ملاحظات خام، فاستخدم Plugin ‏`memory-wiki` المضمّن.

يقوم `memory-wiki` بتجميع المعرفة الدائمة في خزينة wiki تحتوي على:

- بنية صفحات حتمية
- ادعاءات وأدلة منظمة
- تتبع التناقضات والحداثة
- لوحات معلومات مُولَّدة
- ملخصات مُجمَّعة لمستهلكي الوكيل/وقت التشغيل
- أدوات أصلية لـ wiki مثل `wiki_search` و`wiki_get` و`wiki_apply` و`wiki_lint`

وهو لا يستبدل Plugin الذاكرة النشط. فما يزال Plugin الذاكرة النشط
هو المسؤول عن الاستدعاء والترقية وDreaming. ويضيف `memory-wiki` طبقة
معرفة غنية بالمصدر إلى جانبه.

راجع [Memory Wiki](/ar/plugins/memory-wiki).

## البحث في الذاكرة

عند إعداد مزوّد embeddings، يستخدم `memory_search` **بحثًا
هجينًا** -- يجمع بين التشابه المتجهي (المعنى الدلالي) ومطابقة الكلمات المفتاحية
(المصطلحات الدقيقة مثل المعرّفات ورموز الشيفرة). ويعمل هذا مباشرةً بمجرد أن تمتلك
مفتاح API لأي مزوّد مدعوم.

<Info>
يكتشف OpenClaw تلقائيًا مزوّد embeddings الخاص بك من مفاتيح API المتاحة. فإذا
كان لديك مفتاح OpenAI أو Gemini أو Voyage أو Mistral مضبوطًا،
فسيتم تفعيل البحث في الذاكرة تلقائيًا.
</Info>

للاطلاع على تفاصيل كيفية عمل البحث وخيارات الضبط وإعداد المزوّد، راجع
[البحث في الذاكرة](/ar/concepts/memory-search).

## تفريغ الذاكرة التلقائي

قبل أن يقوم [Compaction](/ar/concepts/compaction) بتلخيص محادثتك، يشغّل OpenClaw
دورًا صامتًا يذكّر الوكيل بحفظ السياق المهم في ملفات الذاكرة.
ويكون هذا مفعّلًا افتراضيًا -- ولا تحتاج إلى إعداد أي شيء.

<Tip>
يمنع تفريغ الذاكرة فقدان السياق أثناء Compaction. وإذا كانت لدى وكيلك
حقائق مهمة في المحادثة لم تُكتب بعد إلى ملف، فسيتم حفظها تلقائيًا
قبل حدوث التلخيص.
</Tip>

## Dreaming

Dreaming هو مسار دمج اختياري في الخلفية للذاكرة. فهو يجمع
الإشارات قصيرة الأمد، ويقيّم المرشحين، ويرقي فقط العناصر المؤهلة إلى
الذاكرة طويلة الأمد (`MEMORY.md`).

وقد صُمم للحفاظ على ارتفاع قيمة الإشارة في الذاكرة طويلة الأمد:

- **اختياري**: معطل افتراضيًا.
- **مجدول**: عند تفعيله، يدير `memory-core` تلقائيًا وظيفة Cron متكررة واحدة
  لجولة Dreaming كاملة.
- **محدود بعتبات**: يجب أن تجتاز الترقيات عتبات التقييم، وتكرار الاستدعاء، وتنوع
  الاستعلام.
- **قابل للمراجعة**: تُكتب ملخصات المراحل وإدخالات اليوميات إلى `DREAMS.md`
  للمراجعة البشرية.

للاطلاع على سلوك المراحل، وإشارات التقييم، وتفاصيل يوميات Dreaming، راجع
[Dreaming](/ar/concepts/dreaming).

## التعبئة المؤصلة والترقية المباشرة

يحتوي نظام Dreaming الآن على مساري مراجعة متقاربين:

- **Dreaming المباشر** يعمل من مخزن Dreaming قصير الأمد تحت
  `memory/.dreams/`، وهو ما تستخدمه المرحلة العميقة العادية عند تقرير ما
  يمكن أن يتخرج إلى `MEMORY.md`.
- **التعبئة المؤصلة** تقرأ ملاحظات `memory/YYYY-MM-DD.md` التاريخية بوصفها
  ملفات أيام مستقلة وتكتب مخرجات مراجعة منظمة إلى `DREAMS.md`.

تكون التعبئة المؤصلة مفيدة عندما تريد إعادة تشغيل الملاحظات الأقدم وفحص ما
يعتبره النظام دائمًا من دون تحرير `MEMORY.md` يدويًا.

عند استخدام:

```bash
openclaw memory rem-backfill --path ./memory --stage-short-term
```

لا تتم ترقية المرشحين الدائمين المؤصلين مباشرة. بل يتم تجهيزهم في
مخزن Dreaming قصير الأمد نفسه الذي تستخدمه المرحلة العميقة العادية بالفعل. وهذا
يعني ما يلي:

- يبقى `DREAMS.md` سطح المراجعة البشرية.
- يبقى المخزن قصير الأمد سطح الترتيب الموجّه للآلة.
- لا تتم كتابة `MEMORY.md` إلا عبر الترقية العميقة.

إذا قررت أن إعادة التشغيل لم تكن مفيدة، فيمكنك إزالة العناصر المجهزة
من دون المساس بإدخالات اليوميات العادية أو حالة الاستدعاء العادية:

```bash
openclaw memory rem-backfill --rollback
openclaw memory rem-backfill --rollback-short-term
```

## CLI

```bash
openclaw memory status          # تحقق من حالة الفهرس والمزوّد
openclaw memory search "query"  # ابحث من سطر الأوامر
openclaw memory index --force   # أعد بناء الفهرس
```
