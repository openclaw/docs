مساحة العمل هي منزل الوكيل. وهي دليل العمل الوحيد المستخدم لأدوات
الملفات ولسياق مساحة العمل. احتفظ بها خاصة وتعامل معها على أنها ذاكرة.

وهذا منفصل عن `~/.openclaw/`، الذي يخزن التكوين وبيانات الاعتماد
والجلسات.

**مهم:** مساحة العمل هي **cwd الافتراضي**، وليست صندوق حماية صارمًا. تقوم الأدوات
بتحليل المسارات النسبية انطلاقًا من مساحة العمل، لكن المسارات المطلقة يمكنها
مع ذلك الوصول إلى أماكن أخرى على المضيف ما لم يتم تفعيل sandboxing. إذا كنت
تحتاج إلى عزل، فاستخدم
[`agents.defaults.sandbox`](/ar/gateway/sandboxing) (و/أو تكوين sandbox لكل وكيل).
عند تفعيل sandboxing وعندما لا تكون `workspaceAccess` مساوية لـ `"rw"`، تعمل الأدوات
داخل مساحة عمل sandbox تحت `~/.openclaw/sandboxes`، وليس داخل مساحة عمل المضيف.

## الموقع الافتراضي

- الافتراضي: `~/.openclaw/workspace`
- إذا تم ضبط `OPENCLAW_PROFILE` ولم يكن `"default"`، يصبح الافتراضي
  `~/.openclaw/workspace-<profile>`.
- يمكن تجاوزه في `~/.openclaw/openclaw.json`:

```json5
{
  agent: {
    workspace: "~/.openclaw/workspace",
  },
}
```

سيقوم `openclaw onboard` أو `openclaw configure` أو `openclaw setup` بإنشاء
مساحة العمل وملء ملفات bootstrap إذا كانت مفقودة.
لا تقبل نسخ seed الخاصة بـ sandbox إلا الملفات العادية الموجودة داخل مساحة العمل؛ أما
الأسماء المستعارة من نوع symlink/hardlink التي تُحل إلى خارج مساحة العمل المصدر فيتم تجاهلها.

إذا كنت تدير ملفات مساحة العمل بنفسك بالفعل، يمكنك تعطيل إنشاء
ملفات bootstrap:

```json5
{ agent: { skipBootstrap: true } }
```

## مجلدات إضافية لمساحة العمل

قد تكون التثبيتات الأقدم قد أنشأت `~/openclaw`. ويمكن أن يؤدي الاحتفاظ
بعدة أدلة لمساحات العمل إلى انحراف مربك في المصادقة أو الحالة،
لأن مساحة عمل واحدة فقط تكون نشطة في كل مرة.

**التوصية:** احتفظ بمساحة عمل نشطة واحدة. وإذا لم تعد تستخدم
المجلدات الإضافية، فأرشفها أو انقلها إلى سلة المهملات (على سبيل المثال `trash ~/openclaw`).
وإذا كنت تحتفظ عمدًا بعدة مساحات عمل، فتأكد من أن
`agents.defaults.workspace` يشير إلى المساحة النشطة.

يحذر `openclaw doctor` عند اكتشافه لأدلة إضافية لمساحة العمل.

## خريطة ملفات مساحة العمل (ماذا يعني كل ملف)

هذه هي الملفات القياسية التي يتوقعها OpenClaw داخل مساحة العمل:

- `AGENTS.md`
  - تعليمات تشغيلية للوكيل وكيف يجب أن يستخدم الذاكرة.
  - يتم تحميله في بداية كل جلسة.
  - مكان جيد للقواعد والأولويات وتفاصيل "كيفية التصرف".

- `SOUL.md`
  - الشخصية، والنبرة، والحدود.
  - يتم تحميله في كل جلسة.
  - الدليل: [دليل شخصية SOUL.md](/ar/concepts/soul)

- `USER.md`
  - من هو المستخدم وكيفية مخاطبته.
  - يتم تحميله في كل جلسة.

- `IDENTITY.md`
  - اسم الوكيل، وطابعه، وemoji الخاص به.
  - يتم إنشاؤه/تحديثه أثناء طقس bootstrap.

- `TOOLS.md`
  - ملاحظات حول أدواتك المحلية واصطلاحاتك.
  - لا يتحكم في توفر الأدوات؛ إنه مجرد إرشاد.

- `HEARTBEAT.md`
  - قائمة تحقق صغيرة اختيارية لتشغيلات Heartbeat.
  - اجعلها قصيرة لتجنب استهلاك الرموز المميزة.

- `BOOT.md`
  - قائمة تحقق اختيارية عند بدء التشغيل تُشغَّل تلقائيًا عند إعادة تشغيل Gateway (عند تفعيل [الخطافات الداخلية](/ar/automation/hooks)).
  - اجعلها قصيرة؛ واستخدم أداة الرسائل للإرسال الصادر.

- `BOOTSTRAP.md`
  - طقس التشغيل الأول لمرة واحدة.
  - لا يتم إنشاؤه إلا لمساحة عمل جديدة تمامًا.
  - احذفه بعد اكتمال الطقس.

- `memory/YYYY-MM-DD.md`
  - سجل الذاكرة اليومي (ملف واحد لكل يوم).
  - يُنصح بقراءة اليوم + الأمس عند بدء الجلسة.

- `MEMORY.md` (اختياري)
  - ذاكرة طويلة الأمد منسقة.
  - حمّله فقط في الجلسة الرئيسية الخاصة (وليس في السياقات المشتركة/الجماعية).

راجع [الذاكرة](/ar/concepts/memory) لسير العمل والتفريغ التلقائي للذاكرة.

- `skills/` (اختياري)
  - Skills خاصة بمساحة العمل.
  - أعلى موقع أولوية لـ Skills في تلك المساحة.
  - تتجاوز Skills الوكلاء الخاصة بالمشروع، وSkills الوكلاء الشخصية، وSkills المُدارة، وSkills المضمّنة، و`skills.load.extraDirs` عند تعارض الأسماء.

- `canvas/` (اختياري)
  - ملفات Canvas UI لعروض Node (على سبيل المثال `canvas/index.html`).

إذا كان أي ملف bootstrap مفقودًا، يحقن OpenClaw علامة "ملف مفقود" داخل
الجلسة ويواصل العمل. ويتم اقتطاع ملفات bootstrap الكبيرة عند حقنها؛
اضبط الحدود باستخدام `agents.defaults.bootstrapMaxChars` (الافتراضي: 12000) و
`agents.defaults.bootstrapTotalMaxChars` (الافتراضي: 60000).
يمكن لـ `openclaw setup` إعادة إنشاء القيم الافتراضية المفقودة من دون الكتابة فوق
الملفات الموجودة.

## ما الذي لا يوجد في مساحة العمل

توجد هذه العناصر تحت `~/.openclaw/` ويجب **ألا** تُحفَظ في مستودع مساحة العمل:

- `~/.openclaw/openclaw.json` (التكوين)
- `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` (ملفات تعريف مصادقة النموذج: OAuth + مفاتيح API)
- `~/.openclaw/credentials/` (حالة القناة/المزوّد بالإضافة إلى بيانات استيراد OAuth القديمة)
- `~/.openclaw/agents/<agentId>/sessions/` (نصوص الجلسات + البيانات الوصفية)
- `~/.openclaw/skills/` (Skills المُدارة)

إذا كنت بحاجة إلى ترحيل الجلسات أو التكوين، فانسخها بشكل منفصل واحتفظ بها
خارج نظام التحكم في الإصدارات.

## النسخ الاحتياطي عبر Git (موصى به، خاص)

تعامل مع مساحة العمل كذاكرة خاصة. ضعها في مستودع Git **خاص** حتى تكون
منسوخة احتياطيًا وقابلة للاستعادة.

نفذ هذه الخطوات على الجهاز الذي يعمل عليه Gateway (فهناك
توجد مساحة العمل).

### 1) تهيئة المستودع

إذا كان git مثبتًا، تتم تهيئة مساحات العمل الجديدة تلقائيًا. وإذا لم تكن
مساحة العمل هذه مستودعًا بالفعل، فشغّل:

```bash
cd ~/.openclaw/workspace
git init
git add AGENTS.md SOUL.md TOOLS.md IDENTITY.md USER.md HEARTBEAT.md memory/
git commit -m "Add agent workspace"
```

### 2) إضافة remote خاص (خيارات سهلة للمبتدئين)

الخيار A: واجهة GitHub على الويب

1. أنشئ مستودعًا جديدًا **خاصًا** على GitHub.
2. لا تقم بتهيئته بملف README (لتجنب تعارضات الدمج).
3. انسخ عنوان HTTPS الخاص بـ remote.
4. أضف remote وادفع:

```bash
git branch -M main
git remote add origin <https-url>
git push -u origin main
```

الخيار B: GitHub CLI (`gh`)

```bash
gh auth login
gh repo create openclaw-workspace --private --source . --remote origin --push
```

الخيار C: واجهة GitLab على الويب

1. أنشئ مستودعًا جديدًا **خاصًا** على GitLab.
2. لا تقم بتهيئته بملف README (لتجنب تعارضات الدمج).
3. انسخ عنوان HTTPS الخاص بـ remote.
4. أضف remote وادفع:

```bash
git branch -M main
git remote add origin <https-url>
git push -u origin main
```

### 3) التحديثات المستمرة

```bash
git status
git add .
git commit -m "Update memory"
git push
```

## لا تحفظ الأسرار في المستودع

حتى في مستودع خاص، تجنب تخزين الأسرار داخل مساحة العمل:

- مفاتيح API، أو رموز OAuth المميزة، أو كلمات المرور، أو بيانات الاعتماد الخاصة.
- أي شيء تحت `~/.openclaw/`.
- تفريغات خام للمحادثات أو مرفقات حساسة.

إذا اضطررت إلى تخزين مراجع حساسة، فاستخدم عناصر نائبة واحتفظ بالسر
الحقيقي في مكان آخر (مدير كلمات المرور، أو متغيرات البيئة، أو `~/.openclaw/`).

بداية مقترحة لملف `.gitignore`:

```gitignore
.DS_Store
.env
**/*.key
**/*.pem
**/secrets*
```

## نقل مساحة العمل إلى جهاز جديد

1. انسخ المستودع إلى المسار المطلوب (الافتراضي `~/.openclaw/workspace`).
2. اضبط `agents.defaults.workspace` على ذلك المسار في `~/.openclaw/openclaw.json`.
3. شغّل `openclaw setup --workspace <path>` لملء أي ملفات مفقودة.
4. إذا كنت بحاجة إلى الجلسات، فانسخ `~/.openclaw/agents/<agentId>/sessions/` من
   الجهاز القديم بشكل منفصل.

## ملاحظات متقدمة

- يمكن للتوجيه متعدد الوكلاء استخدام مساحات عمل مختلفة لكل وكيل. راجع
  [توجيه القنوات](/ar/channels/channel-routing) لتكوين التوجيه.
- إذا كان `agents.defaults.sandbox` مفعّلًا، فيمكن للجلسات غير الرئيسية استخدام مساحات عمل sandbox
  لكل جلسة تحت `agents.defaults.sandbox.workspaceRoot`.
