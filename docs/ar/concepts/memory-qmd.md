[QMD](https://github.com/tobi/qmd) مرافق بحث محلي أولًا يعمل
إلى جانب OpenClaw. يجمع بين BM25 والبحث المتجهي وإعادة الترتيب في ملف
تنفيذي واحد، ويمكنه فهرسة محتوى يتجاوز ملفات ذاكرة مساحة عملك.

## ما يضيفه مقارنة بالمحرك المضمن

- **إعادة الترتيب وتوسيع الاستعلام** لتحسين الاسترجاع.
- **فهرسة أدلة إضافية** -- مستندات المشروع، ملاحظات الفريق، أي شيء على القرص.
- **فهرسة نصوص الجلسات** -- استدعاء المحادثات السابقة.
- **محلي بالكامل** -- يعمل مع Plugin موفر llama.cpp الرسمي وينزل نماذج GGUF
  تلقائيًا.
- **رجوع تلقائي** -- إذا لم يكن QMD متاحًا، يرجع OpenClaw بسلاسة إلى
  المحرك المضمن.

## البدء

### المتطلبات الأساسية

- ثبّت QMD: `npm install -g @tobilu/qmd` أو `bun install -g @tobilu/qmd`
- بنية SQLite تسمح بالامتدادات (`brew install sqlite` على macOS).
- يجب أن يكون QMD موجودًا في `PATH` الخاص بالـ Gateway.
- يعمل macOS وLinux مباشرة. أما Windows فدعمه الأفضل عبر WSL2.

### التفعيل

```json5
{
  memory: {
    backend: "qmd",
  },
}
```

ينشئ OpenClaw موطن QMD مستقلًا تحت
`~/.openclaw/agents/<agentId>/qmd/` ويدير دورة حياة المرافق الجانبي
تلقائيًا -- تُدار المجموعات والتحديثات وتشغيلات التضمين نيابة عنك.
يفضل أشكال مجموعة QMD واستعلام MCP الحالية، لكنه يظل يرجع إلى أعلام أنماط
المجموعات البديلة وأسماء أدوات MCP الأقدم عند الحاجة.
تعيد تسوية وقت الإقلاع أيضًا إنشاء المجموعات المُدارة القديمة إلى أنماطها
المعيارية عندما تكون مجموعة QMD أقدم بالاسم نفسه ما زالت موجودة.

## كيف يعمل المرافق الجانبي

- ينشئ OpenClaw مجموعات من ملفات ذاكرة مساحة عملك وأي
  `memory.qmd.paths` مهيأة، ثم يشغّل `qmd update` عند فتح مدير QMD
  وبشكل دوري بعد ذلك (افتراضيًا كل 5 دقائق). تعمل عمليات التحديث هذه عبر
  عمليات QMD فرعية، لا عبر زحف نظام ملفات داخل العملية. وتشغّل الأوضاع
  الدلالية أيضًا `qmd embed`.
- تتتبع مجموعة مساحة العمل الافتراضية `MEMORY.md` إضافة إلى شجرة `memory/`.
  لا تتم فهرسة `memory.md` بالأحرف الصغيرة كملف ذاكرة جذري.
- يتجاهل الماسح الخاص بـ QMD المسارات المخفية وأدلة الاعتماديات/البناء
  الشائعة مثل `.git` و`.cache` و`node_modules` و`vendor` و`dist` و
  `build`. لا يهيئ بدء Gateway QMD افتراضيًا، لذلك يتجنب الإقلاع البارد
  استيراد وقت تشغيل الذاكرة أو إنشاء المراقب طويل العمر قبل استخدام الذاكرة
  لأول مرة.
- إذا أردت تهيئة QMD عند بدء Gateway رغم ذلك، فاضبط
  `memory.qmd.update.startup` على `idle` أو `immediate`. مع
  `memory.qmd.update.onBoot: true`، يشغّل البدء التحديث الأولي. ومع
  `onBoot: false`، يتجاوز البدء ذلك التحديث الفوري لكنه يظل يفتح المدير
  طويل العمر عند تهيئة فواصل التحديث أو التضمين، حتى يستطيع QMD امتلاك
  مراقبه ومؤقتاته المنتظمة.
- تستخدم عمليات البحث `searchMode` المهيأ (افتراضيًا: `search`؛ ويدعم أيضًا
  `vsearch` و`query`). وضع `search` يعتمد على BM25 فقط، لذلك يتجاوز
  OpenClaw فحوص جاهزية المتجهات الدلالية وصيانة التضمين في ذلك الوضع. إذا
  فشل وضع ما، يعيد OpenClaw المحاولة باستخدام `qmd query`.
- عندما يكون `searchMode` هو `query`، اضبط `memory.qmd.rerank` على `false`
  لاستخدام مسار الاستعلام الهجين في QMD بدون معيد الترتيب. يمرر OpenClaw
  `--no-rerank` إلى مسار CLI المباشر لـ QMD و`rerank: false` إلى أداة
  استعلام MCP الخاصة بـ QMD. يتطلب هذا الخيار QMD 2.1 أو أحدث.
- مع إصدارات QMD التي تعلن عن مرشحات متعددة المجموعات، يجمع OpenClaw
  المجموعات ذات المصدر نفسه في استدعاء بحث QMD واحد. وتحافظ إصدارات QMD
  الأقدم على الرجوع المتوافق لكل مجموعة على حدة.
- إذا فشل QMD بالكامل، يرجع OpenClaw إلى محرك SQLite المضمن.
  تتراجع محاولات أدوار المحادثة المتكررة لفترة وجيزة بعد فشل الفتح حتى لا
  ينشئ ملف تنفيذي مفقود أو اعتماد مرافق جانبي معطوب عاصفة إعادة محاولة؛
  يظل `openclaw memory status` وفحوص CLI لمرة واحدة تعيد فحص QMD مباشرة.

<Info>
قد يكون البحث الأول بطيئًا -- ينزل QMD نماذج GGUF تلقائيًا (~2 GB) من أجل
إعادة الترتيب وتوسيع الاستعلام عند تشغيل `qmd query` لأول مرة.
</Info>

## أداء البحث والتوافق

يحافظ OpenClaw على توافق مسار بحث QMD مع تثبيتات QMD الحالية والأقدم.

عند البدء، يفحص OpenClaw نص مساعدة QMD المثبت مرة واحدة لكل مدير. إذا أعلن
الملف التنفيذي دعم مرشحات مجموعات متعددة، يبحث OpenClaw في كل المجموعات
ذات المصدر نفسه بأمر واحد:

```bash
qmd search "router notes" --json -n 10 -c memory-root-main -c memory-dir-main
```

يتجنب هذا بدء عملية QMD فرعية لكل مجموعة ذاكرة دائمة.
تبقى مجموعات نصوص الجلسات في مجموعة المصدر الخاصة بها، لذلك تظل عمليات البحث
المختلطة بين `memory` + `sessions` تمنح مُنوّع النتائج مدخلات من كلا
المصدرين.

تقبل بُنى QMD الأقدم مرشح مجموعة واحدًا فقط. عندما يكتشف OpenClaw إحدى تلك
البُنى، يحافظ على مسار التوافق ويبحث في كل مجموعة على حدة قبل دمج النتائج
وإزالة التكرارات.

لفحص العقد المثبت يدويًا، شغّل:

```bash
qmd --help | grep -i collection
```

تقول مساعدة QMD الحالية إن مرشحات المجموعات يمكن أن تستهدف مجموعة واحدة أو
أكثر. أما المساعدة الأقدم فعادة ما تصف مجموعة واحدة.

## تجاوزات النماذج

تمر متغيرات بيئة نماذج QMD كما هي من عملية Gateway، لذلك يمكنك ضبط QMD
عموميًا دون إضافة تهيئة OpenClaw جديدة:

```bash
export QMD_EMBED_MODEL="hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"
export QMD_RERANK_MODEL="/absolute/path/to/reranker.gguf"
export QMD_GENERATE_MODEL="/absolute/path/to/generator.gguf"
```

بعد تغيير نموذج التضمين، أعد تشغيل التضمينات حتى يطابق الفهرس فضاء المتجهات
الجديد.

## فهرسة مسارات إضافية

وجّه QMD إلى أدلة إضافية لجعلها قابلة للبحث:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

تظهر المقاطع من المسارات الإضافية بصيغة `qmd/<collection>/<relative-path>` في
نتائج البحث. يفهم `memory_get` هذه البادئة ويقرأ من جذر المجموعة الصحيح.

## فهرسة نصوص الجلسات

فعّل فهرسة الجلسات لاستدعاء المحادثات السابقة:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      sessions: { enabled: true },
    },
  },
}
```

تُصدَّر النصوص كأدوار User/Assistant منقحة إلى مجموعة QMD مخصصة تحت
`~/.openclaw/agents/<id>/qmd/sessions/`.

## نطاق البحث

افتراضيًا، تظهر نتائج بحث QMD في الجلسات المباشرة وجلسات القنوات
(وليس المجموعات). هيئ `memory.qmd.scope` لتغيير ذلك:

```json5
{
  memory: {
    qmd: {
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
    },
  },
}
```

عندما يرفض النطاق بحثًا، يسجل OpenClaw تحذيرًا بالقناة ونوع المحادثة
المستنتجين حتى يكون تصحيح النتائج الفارغة أسهل.

## الاستشهادات

عندما يكون `memory.citations` هو `auto` أو `on`، تتضمن مقاطع البحث تذييل
`Source: <path#line>`. اضبط `memory.citations = "off"` لحذف التذييل مع
الاستمرار في تمرير المسار إلى الوكيل داخليًا.

## متى تستخدمه

اختر QMD عندما تحتاج إلى:

- إعادة ترتيب للحصول على نتائج أعلى جودة.
- البحث في مستندات المشروع أو الملاحظات خارج مساحة العمل.
- استدعاء محادثات الجلسات السابقة.
- بحث محلي بالكامل بلا مفاتيح API.

للإعدادات الأبسط، يعمل [المحرك المضمن](/ar/concepts/memory-builtin) جيدًا
بدون اعتماديات إضافية.

## استكشاف الأخطاء وإصلاحها

**لم يتم العثور على QMD؟** تأكد من أن الملف التنفيذي موجود في `PATH` الخاص
بالـ Gateway. إذا كان OpenClaw يعمل كخدمة، فأنشئ رابطًا رمزيًا:
`sudo ln -s ~/.bun/bin/qmd /usr/local/bin/qmd`.

إذا كان `qmd --version` يعمل في صدفتك لكن OpenClaw ما زال يبلغ عن
`spawn qmd ENOENT`، فمن المرجح أن عملية Gateway لديها `PATH` مختلف عن
صدفتك التفاعلية. ثبت الملف التنفيذي صراحة:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      command: "/absolute/path/to/qmd",
    },
  },
}
```

استخدم `command -v qmd` في البيئة التي ثُبت فيها QMD، ثم أعد الفحص باستخدام
`openclaw memory status --deep`.

**البحث الأول بطيء جدًا؟** ينزل QMD نماذج GGUF عند أول استخدام. مهّد التشغيل
باستخدام `qmd query "test"` مع أدلة XDG نفسها التي يستخدمها OpenClaw.

**عمليات QMD فرعية كثيرة أثناء البحث؟** حدّث QMD إن أمكن. يستخدم OpenClaw
عملية واحدة لعمليات البحث متعددة المجموعات ذات المصدر نفسه فقط عندما يعلن
QMD المثبت دعم مرشحات `-c` متعددة؛ وإلا فيحافظ على الرجوع الأقدم لكل مجموعة
على حدة لضمان الصحة.

**QMD المعتمد على BM25 فقط ما زال يحاول بناء llama.cpp؟** اضبط
`memory.qmd.searchMode = "search"`. يتعامل OpenClaw مع ذلك الوضع كمعجمي فقط،
ولا يشغّل فحوص حالة متجهات QMD أو صيانة التضمين، ويترك فحوص الجاهزية
الدلالية لإعدادات `vsearch` أو `query`.

**تنتهي مهلة البحث؟** زد `memory.qmd.limits.timeoutMs` (الافتراضي: 4000ms).
اضبطها على `120000` للأجهزة الأبطأ.

**نتائج فارغة في محادثات المجموعات؟** تحقق من `memory.qmd.scope` -- الافتراضي
يسمح فقط بالجلسات المباشرة وجلسات القنوات.

**أصبح بحث الذاكرة الجذرية واسعًا فجأة؟** أعد تشغيل Gateway أو انتظر تسوية
البدء التالية. يعيد OpenClaw إنشاء المجموعات المُدارة القديمة إلى أنماط
`MEMORY.md` و`memory/` المعيارية عندما يكتشف تعارضًا بالاسم نفسه.

**هل تسبب مستودعات مؤقتة ظاهرة من مساحة العمل `ENAMETOOLONG` أو فهرسة
معطوبة؟**
يتبع اجتياز QMD حاليًا سلوك ماسح QMD الأساسي بدلًا من قواعد الروابط الرمزية
المضمنة في OpenClaw. أبقِ نسخ monorepo المؤقتة تحت أدلة مخفية مثل `.tmp/` أو
خارج جذور QMD المفهرسة إلى أن يكشف QMD عن اجتياز آمن من الدورات أو عناصر تحكم
استبعاد صريحة.

## التهيئة

للاطلاع على سطح التهيئة الكامل (`memory.qmd.*`) وأوضاع البحث وفواصل التحديث
وقواعد النطاق وكل عناصر الضبط الأخرى، راجع
[مرجع تهيئة الذاكرة](/ar/reference/memory-config).
