[QMD](https://github.com/tobi/qmd) هو خدمة جانبية للبحث محليًا أولًا تعمل
إلى جانب OpenClaw. وهو يجمع بين BM25، والبحث المتجهي، وإعادة الترتيب في
ملف تنفيذي واحد، ويمكنه فهرسة محتوى يتجاوز ملفات ذاكرة مساحة العمل الخاصة بك.

## ما الذي يضيفه مقارنةً بالمضمن

- **إعادة الترتيب وتوسيع الاستعلام** لتحسين الاسترجاع.
- **فهرسة أدلة إضافية** -- وثائق المشاريع، وملاحظات الفريق، وأي شيء على القرص.
- **فهرسة نصوص الجلسات** -- لاسترجاع المحادثات السابقة.
- **محلي بالكامل** -- يعمل عبر Bun + node-llama-cpp، ويحمّل نماذج GGUF تلقائيًا.
- **رجوع احتياطي تلقائي** -- إذا لم يكن QMD متاحًا، يعود OpenClaw إلى
  المحرك المضمن بسلاسة.

## البدء

### المتطلبات الأساسية

- ثبّت QMD: ‏`npm install -g @tobilu/qmd` أو `bun install -g @tobilu/qmd`
- إصدار SQLite يسمح بالامتدادات (`brew install sqlite` على macOS).
- يجب أن يكون QMD موجودًا على `PATH` الخاص بـ gateway.
- يعمل macOS وLinux مباشرةً. أما Windows فأفضل دعم له يكون عبر WSL2.

### التفعيل

```json5
{
  memory: {
    backend: "qmd",
  },
}
```

ينشئ OpenClaw موطن QMD مكتفيًا ذاتيًا تحت
`~/.openclaw/agents/<agentId>/qmd/` ويدير دورة حياة الخدمة الجانبية
تلقائيًا -- حيث تُدار المجموعات، والتحديثات، وتشغيلات التضمين نيابةً عنك.
وهو يفضّل أشكال مجموعات QMD الحالية وأشكال استعلامات MCP الحالية، لكنه لا يزال يعود إلى
علامات المجموعات القديمة `--mask` وأسماء أدوات MCP الأقدم عند الحاجة.
كما أن التسوية وقت الإقلاع تعيد أيضًا إنشاء المجموعات المُدارة القديمة إلى أنماطها
القياسية عندما تكون مجموعة QMD أقدم تحمل الاسم نفسه لا تزال موجودة.

## كيف تعمل الخدمة الجانبية

- ينشئ OpenClaw مجموعات من ملفات ذاكرة مساحة العمل وأي
  `memory.qmd.paths` مُعدّة، ثم يشغّل `qmd update` و`qmd embed` عند الإقلاع
  وبشكل دوري (الافتراضي كل 5 دقائق).
- تتتبع مجموعة مساحة العمل الافتراضية الملف `MEMORY.md` بالإضافة إلى شجرة
  `memory/`. ولا تتم فهرسة `memory.md` بالأحرف الصغيرة كملف ذاكرة جذري.
- يعمل تحديث الإقلاع في الخلفية حتى لا يتم حظر بدء الدردشة.
- تستخدم عمليات البحث `searchMode` المُعدّة (الافتراضي: `search`؛ ويدعم أيضًا
  `vsearch` و`query`). وإذا فشل وضع ما، يعيد OpenClaw المحاولة باستخدام `qmd query`.
- إذا فشل QMD بالكامل، يعود OpenClaw إلى محرك SQLite المضمن.

<Info>
قد يكون البحث الأول بطيئًا -- إذ يقوم QMD بتنزيل نماذج GGUF تلقائيًا (~2 GB) من أجل
إعادة الترتيب وتوسيع الاستعلام عند أول تشغيل لـ `qmd query`.
</Info>

## تجاوزات النماذج

تمر متغيرات بيئة نموذج QMD كما هي من عملية
gateway، لذلك يمكنك ضبط QMD عالميًا من دون إضافة إعدادات جديدة في OpenClaw:

```bash
export QMD_EMBED_MODEL="hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"
export QMD_RERANK_MODEL="/absolute/path/to/reranker.gguf"
export QMD_GENERATE_MODEL="/absolute/path/to/generator.gguf"
```

بعد تغيير نموذج التضمين، أعد تشغيل عمليات التضمين حتى يتطابق الفهرس مع
الفضاء المتجهي الجديد.

## فهرسة مسارات إضافية

وجّه QMD إلى أدلة إضافية لجعلها قابلة للبحث:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

تظهر المقتطفات من المسارات الإضافية على شكل `qmd/<collection>/<relative-path>` في
نتائج البحث. ويفهم `memory_get` هذه السابقة ويقرأ من جذر المجموعة الصحيح.

## فهرسة نصوص الجلسات

فعّل فهرسة الجلسات لاسترجاع المحادثات السابقة:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      sessions: { enabled: true },
    },
  },
}
```

تُصدَّر النصوص كأدوار مستخدم/مساعد منقّاة إلى مجموعة QMD مخصصة
تحت `~/.openclaw/agents/<id>/qmd/sessions/`.

## نطاق البحث

افتراضيًا، تظهر نتائج بحث QMD في الجلسات المباشرة وجلسات القنوات
(وليس في المجموعات). اضبط `memory.qmd.scope` لتغيير ذلك:

```json5
{
  memory: {
    qmd: {
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
    },
  },
}
```

عندما يمنع النطاق عملية بحث، يسجّل OpenClaw تحذيرًا مع القناة المشتقة و
نوع الدردشة بحيث يصبح تصحيح النتائج الفارغة أسهل.

## الاستشهادات

عندما تكون `memory.citations` مساوية لـ `auto` أو `on`، تتضمن مقتطفات البحث
تذييلًا بالشكل `Source: <path#line>`. اضبط `memory.citations = "off"` لحذف التذييل
مع الاستمرار في تمرير المسار إلى الوكيل داخليًا.

## متى تستخدمه

اختر QMD عندما تحتاج إلى:

- إعادة ترتيب للحصول على نتائج أعلى جودة.
- البحث في وثائق المشروع أو الملاحظات خارج مساحة العمل.
- استرجاع محادثات الجلسات السابقة.
- بحث محلي بالكامل من دون مفاتيح API.

بالنسبة إلى الإعدادات الأبسط، يعمل [المحرك المضمن](/ar/concepts/memory-builtin) جيدًا
من دون أي تبعيات إضافية.

## استكشاف الأخطاء وإصلاحها

**QMD غير موجود؟** تأكد من أن الملف التنفيذي موجود على `PATH` الخاص بـ gateway. إذا كان OpenClaw
يعمل كخدمة، فأنشئ رابطًا رمزيًا:
`sudo ln -s ~/.bun/bin/qmd /usr/local/bin/qmd`.

**هل البحث الأول بطيء جدًا؟** يقوم QMD بتنزيل نماذج GGUF عند أول استخدام. قم بالتهيئة المسبقة
باستخدام `qmd query "test"` مع أدلة XDG نفسها التي يستخدمها OpenClaw.

**هل انتهت مهلة البحث؟** زد قيمة `memory.qmd.limits.timeoutMs` (الافتراضي: 4000ms).
اضبطها على `120000` للأجهزة الأبطأ.

**نتائج فارغة في دردشات المجموعات؟** تحقق من `memory.qmd.scope` -- فالافتراضي
يسمح فقط بالجلسات المباشرة وجلسات القنوات.

**هل أصبح بحث الذاكرة الجذرية أوسع من اللازم فجأة؟** أعد تشغيل gateway أو انتظر
حتى تسوية بدء التشغيل التالية. يعيد OpenClaw إنشاء المجموعات المُدارة القديمة
إلى أنماط `MEMORY.md` و`memory/` القياسية عندما يكتشف تعارضًا
في الاسم نفسه.

**هل تتسبب المستودعات المؤقتة المرئية لمساحة العمل في `ENAMETOOLONG` أو فهرسة معطلة؟**
يتبع اجتياز QMD حاليًا سلوك ماسح QMD الأساسي بدلًا من
قواعد الروابط الرمزية المضمنة في OpenClaw. احتفظ بعمليات سحب monorepo المؤقتة تحت
أدلة مخفية مثل `.tmp/` أو خارج جذور QMD المفهرسة إلى أن يوفّر QMD
اجتيازًا آمنًا من الحلقات أو عناصر تحكم صريحة في الاستثناءات.

## الإعدادات

للحصول على سطح الإعدادات الكامل (`memory.qmd.*`) وأوضاع البحث وفترات
التحديث وقواعد النطاق وجميع العناصر الأخرى، راجع
[مرجع إعدادات الذاكرة](/ar/reference/memory-config).
