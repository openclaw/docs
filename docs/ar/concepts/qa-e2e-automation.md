تهدف مجموعة QA الخاصة إلى اختبار OpenClaw بطريقة أكثر واقعية وتشبه سلوك القنوات، أكثر مما يمكن أن يفعله اختبار وحدات واحد.

المكونات الحالية:

- `extensions/qa-channel`: قناة رسائل اصطناعية تحتوي على أسطح للرسائل الخاصة والقنوات والخيوط والتفاعلات والتعديل والحذف.
- `extensions/qa-lab`: واجهة مستخدم لتصحيح الأخطاء وناقل QA لمراقبة السجل النصي وحقن الرسائل الواردة وتصدير تقرير Markdown.
- `qa/`: أصول seed مدعومة من المستودع لمهمة الانطلاق وسيناريوهات QA الأساسية.

التدفق التشغيلي الحالي لـ QA هو موقع QA ذو لوحتين:

- اليسار: لوحة تحكم Gateway (Control UI) مع الوكيل.
- اليمين: QA Lab، وتعرض سجلًا شبيهًا بـ Slack وخطة السيناريو.

شغّله باستخدام:

```bash
pnpm qa:lab:up
```

يقوم هذا ببناء موقع QA، وبدء مسار Gateway المعتمد على Docker، وإظهار صفحة QA Lab حيث يمكن للمشغّل أو حلقة الأتمتة إعطاء الوكيل مهمة QA ومراقبة سلوك القناة الحقيقي وتسجيل ما نجح وما فشل وما بقي معطّلًا.

للحصول على دورة تطوير أسرع لواجهة QA Lab من دون إعادة بناء صورة Docker في كل مرة، ابدأ المجموعة باستخدام حزمة QA Lab مربوطة بالتحميل:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

يحافظ `qa:lab:up:fast` على خدمات Docker على صورة مبنية مسبقًا ويجري bind-mount لـ
`extensions/qa-lab/web/dist` داخل حاوية `qa-lab`. ويعيد `qa:lab:watch`
بناء تلك الحزمة عند التغيير، ويعيد المتصفح التحميل تلقائيًا عندما يتغير hash الخاص بأصل QA Lab.

لتشغيل smoke محلي لتتبّع OpenTelemetry، استخدم:

```bash
pnpm qa:otel:smoke
```

يبدأ هذا السكربت مستقبل traces محليًا عبر OTLP/HTTP، ويشغّل
سيناريو QA `otel-trace-smoke` مع تمكين Plugin `diagnostics-otel`، ثم
يفك ترميز spans المصدّرة بصيغة protobuf ويتحقق من البنية الحرجة للإصدار:
يجب أن تكون `openclaw.run` و`openclaw.harness.run` و`openclaw.model.call` و
`openclaw.context.assembled` و`openclaw.message.delivery` موجودة؛
ويجب ألا تصدّر استدعاءات النموذج `StreamAbandoned` في الدورات الناجحة؛ كما يجب أن تبقى
معرّفات التشخيص الخام وخصائص `openclaw.content.*` خارج trace. ويكتب
`otel-smoke-summary.json` بجوار artifacts الخاصة بحزمة QA.

لتشغيل مسار smoke واقعي للنقل عبر Matrix، استخدم:

```bash
pnpm openclaw qa matrix
```

يقوم هذا المسار بتهيئة homeserver مؤقت من Tuwunel داخل Docker، ويسجّل
مستخدمين مؤقتين للسائق وSUT والمراقب، وينشئ غرفة خاصة واحدة، ثم يشغّل
Plugin Matrix الحقيقي داخل Gateway فرعي لـ QA. ويحافظ مسار النقل الحي على
تهيئة الطفل محصورة بالنقل قيد الاختبار، لذلك يعمل Matrix من دون
`qa-channel` في تهيئة الطفل. ويكتب artifacts التقرير المنظّم
وسجلًا مدمجًا لـ stdout/stderr في دليل إخراج Matrix QA المحدد. ولالتقاط
مخرجات البناء/المشغّل الخارجية لـ `scripts/run-node.mjs` أيضًا، اضبط
`OPENCLAW_RUN_NODE_OUTPUT_LOG=<path>` على ملف سجل محلي في المستودع.
يتم طباعة تقدم Matrix افتراضيًا. ويقيّد `OPENCLAW_QA_MATRIX_TIMEOUT_MS`
التشغيل الكامل، بينما يقيّد `OPENCLAW_QA_MATRIX_CLEANUP_TIMEOUT_MS` التنظيف
بحيث يُبلّغ تفكيك Docker العالق عن أمر الاستعادة الدقيق بدلًا من التعليق.

لتشغيل مسار smoke واقعي للنقل عبر Telegram، استخدم:

```bash
pnpm openclaw qa telegram
```

يستهدف هذا المسار مجموعة Telegram خاصة حقيقية واحدة بدلًا من تهيئة
خادم مؤقت. ويتطلب `OPENCLAW_QA_TELEGRAM_GROUP_ID` و
`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN` و
`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`، بالإضافة إلى
روبوتين مختلفين في المجموعة الخاصة نفسها. ويجب أن يمتلك روبوت SUT
اسم مستخدم Telegram، وتعمل المراقبة بين الروبوتات بأفضل شكل عندما يكون
Bot-to-Bot Communication Mode مفعّلًا لكلا الروبوتين في `@BotFather`.
يخرج الأمر بقيمة غير صفرية عند فشل أي سيناريو. استخدم `--allow-failures` عندما
تريد artifacts من دون رمز خروج فاشل.
يتضمن تقرير Telegram وملخصه RTT لكل رد من طلب إرسال رسالة السائق
إلى رد SUT المرصود، بدءًا من canary.

قبل استخدام بيانات اعتماد live المجمّعة، شغّل:

```bash
pnpm openclaw qa credentials doctor
```

يتحقق doctor من متغيرات بيئة وسيط Convex، ويتحقق من إعدادات نقطة النهاية،
ويتحقق من قابلية وصول admin/list عند وجود السر الخاص بالمشرف.
ولا يُبلغ إلا عن حالة set/missing للأسرار.

لتشغيل مسار smoke واقعي للنقل عبر Discord، استخدم:

```bash
pnpm openclaw qa discord
```

يستهدف هذا المسار قناة خاصة حقيقية واحدة ضمن خادم Discord باستخدام روبوتين:
روبوت سائق يتحكم به harness وروبوت SUT يبدأه Gateway الفرعي
لـ OpenClaw عبر Plugin Discord المضمّن. ويتطلب
`OPENCLAW_QA_DISCORD_GUILD_ID` و`OPENCLAW_QA_DISCORD_CHANNEL_ID` و
`OPENCLAW_QA_DISCORD_DRIVER_BOT_TOKEN` و`OPENCLAW_QA_DISCORD_SUT_BOT_TOKEN` و
`OPENCLAW_QA_DISCORD_SUT_APPLICATION_ID` عند استخدام بيانات اعتماد env.
ويتحقق المسار من معالجة الإشارات في القناة ويتأكد من أن روبوت SUT قد
سجّل الأمر الأصلي `/help` لدى Discord.
يخرج الأمر بقيمة غير صفرية عند فشل أي سيناريو. استخدم `--allow-failures` عندما
تريد artifacts من دون رمز خروج فاشل.

تشترك مسارات النقل الحية الآن في عقد أصغر واحد بدلًا من أن يخترع كل منها
بنية قائمة السيناريوهات الخاصة به:

يبقى `qa-channel` حزمة سلوك المنتج الاصطناعية الواسعة وليس جزءًا من
مصفوفة تغطية النقل الحي.

| -------- | ------ | ------------- | ------------- | ---------------- | ------------------------- | ------------ | ---------- | --------------- | ------------ | ---------------------- |
| Matrix   | x      | x             | x             | x                | x                         | x            | x          | x               |              |                        |

يحافظ هذا على `qa-channel` باعتباره حزمة سلوك المنتج الواسعة بينما
تشترك Matrix وTelegram وعمليات النقل الحية المستقبلية في قائمة تحقق صريحة واحدة لعقد النقل.

لتشغيل مسار Linux VM مؤقت من دون إدخال Docker في مسار QA، استخدم:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

يقلع هذا ضيف Multipass جديدًا، ويثبت التبعيات، ويبني OpenClaw
داخل الضيف، ويشغّل `qa suite`، ثم ينسخ تقرير QA العادي
والملخص مرة أخرى إلى `.artifacts/qa-e2e/...` على المضيف.
ويعيد استخدام سلوك اختيار السيناريو نفسه الذي يستخدمه `qa suite` على المضيف.
تنفذ تشغيلات الحزمة على المضيف وMultipass عدة سيناريوهات محددة بالتوازي
مع عمال Gateway معزولين افتراضيًا. ويكون `qa-channel` افتراضيًا
بدرجة توازٍ 4، مقيّدة بعدد السيناريوهات المحددة. استخدم `--concurrency <count>` لضبط
عدد العمال، أو `--concurrency 1` للتنفيذ التسلسلي.
يخرج الأمر بقيمة غير صفرية عند فشل أي سيناريو. استخدم `--allow-failures` عندما
تريد artifacts من دون رمز خروج فاشل.
تُمرَّر في التشغيلات الحية مدخلات مصادقة QA المدعومة والعملية للضيف:
مفاتيح المزوّد المعتمدة على env، ومسار إعدادات مزوّد QA live،
و`CODEX_HOME` عند وجوده. أبقِ `--output-dir` تحت جذر المستودع حتى يتمكن
الضيف من الكتابة مرة أخرى عبر مساحة العمل المربوطة.

## Seeds مدعومة من المستودع

توجد أصول seed في `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/<theme>/*.md`

وهذه موجودة عمدًا في git حتى تكون خطة QA مرئية للبشر
وللوكيل معًا.

يجب أن يبقى `qa-lab` مشغّل Markdown عامًا. ويجب أن يكون كل ملف Markdown
للسيناريو هو مصدر الحقيقة لتشغيل اختبار واحد، وأن يعرّف:

- بيانات السيناريو الوصفية
- بيانات وصفية اختيارية للفئة والقدرة والمسار والمخاطر
- مراجع الوثائق والكود
- متطلبات Plugin اختيارية
- رقعة إعدادات Gateway اختيارية
- `qa-flow` القابل للتنفيذ

يسمح للسطح التنفيذي القابل لإعادة الاستخدام الذي يدعم `qa-flow` أن يبقى عامًا
وعابرًا للجوانب. فعلى سبيل المثال، يمكن لسيناريوهات Markdown أن تجمع بين
helpers جانب النقل وhelpers جانب المتصفح التي تقود Control UI المضمّنة عبر
واجهة Gateway `browser.request` من دون إضافة مشغّل بحالة خاصة.

يجب تجميع ملفات السيناريو حسب قدرة المنتج بدلًا من مجلد شجرة المصدر.
حافظ على ثبات معرّفات السيناريو عند نقل الملفات؛ واستخدم `docsRefs` و`codeRefs`
لتتبع التنفيذ.

يجب أن تبقى القائمة الأساسية واسعة بما يكفي لتغطية:

- دردشة الرسائل الخاصة والقنوات
- سلوك الخيوط
- دورة حياة إجراءات الرسائل
- استدعاءات Cron الراجعة
- استدعاء الذاكرة
- تبديل النموذج
- تسليم subagent
- قراءة المستودع وقراءة الوثائق
- مهمة بناء صغيرة مثل Lobster Invaders

## مسارات mock للمزوّد

يمتلك `qa suite` مسارين محليين لـ provider mock:

- `mock-openai` هو mock الخاص بـ OpenClaw والمدرك للسيناريو. وهو يظل
  مسار mock الحتمي الافتراضي لـ QA المدعوم من المستودع وParity gate.
- يبدأ `aimock` خادم مزوّد مدعومًا بـ AIMock لتغطية البروتوكول
  والتجهيزات وrecord/replay وchaos بشكل تجريبي. وهو إضافة
  ولا يستبدل موزّع السيناريو `mock-openai`.

يوجد تنفيذ مسارات المزوّد تحت `extensions/qa-lab/src/providers/`.
ويمتلك كل مزوّد إعداداته الافتراضية وبدء خادمه المحلي
وتهيئة نموذج Gateway واحتياجات تهيئة auth-profile
وعلامات القدرات الحية/الوهمية. ويجب أن يمر كود الحزمة وGateway المشترك
عبر سجل المزوّد بدلًا من التفرع حسب أسماء المزوّدين.

## محولات النقل

يمتلك `qa-lab` واجهة نقل عامة لسيناريوهات QA المبنية على Markdown.
ويُعد `qa-channel` أول محول على هذه الواجهة، لكن الهدف التصميمي أوسع:
يجب أن تتمكن القنوات المستقبلية الحقيقية أو الاصطناعية من الاندماج في مشغّل الحزمة نفسه
بدلًا من إضافة مشغّل QA خاص بالنقل.

على مستوى architecture، يكون التقسيم كالتالي:

- يمتلك `qa-lab` تنفيذ السيناريو العام، وتوازي العمال، وكتابة artifacts، وإعداد التقارير.
- يمتلك محول النقل إعدادات Gateway، والجاهزية، ومراقبة الوارد والصادر، وإجراءات النقل، وحالة النقل الموحّدة.
- تعرّف ملفات سيناريو Markdown تحت `qa/scenarios/` تشغيل الاختبار؛ ويقدّم `qa-lab` السطح التنفيذي القابل لإعادة الاستخدام الذي ينفذها.

توجد إرشادات التبني الموجهة للمشرفين لمحولات القنوات الجديدة في
[الاختبار](/ar/help/testing#adding-a-channel-to-qa).

## التقارير

يُصدّر `qa-lab` تقرير بروتوكول بصيغة Markdown من الخط الزمني المرصود للناقل.
ويجب أن يجيب التقرير عن:

- ما الذي نجح
- ما الذي فشل
- ما الذي بقي معطّلًا
- ما سيناريوهات المتابعة الجديرة بالإضافة

لفحوصات الطابع والأسلوب، شغّل السيناريو نفسه عبر عدة مراجع نماذج حية
واكتب تقرير Markdown محكومًا:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.5,thinking=medium,fast \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.5,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

يشغّل الأمر عمليات Gateway فرعية محلية لـ QA، وليس Docker. ويجب أن تضبط
سيناريوهات character eval الشخصية عبر `SOUL.md`، ثم تشغّل أدوار مستخدم عادية
مثل الدردشة، ومساعدة مساحة العمل، ومهام الملفات الصغيرة. ولا ينبغي إبلاغ
النموذج المرشّح بأنه قيد التقييم. ويحافظ الأمر على كل سجل نصي كامل، ويسجل
إحصاءات تشغيل أساسية، ثم يطلب من نماذج التحكيم في الوضع السريع ومع
استدلال `xhigh` حيثما كان مدعومًا أن ترتب التشغيلات حسب الطبيعية، والطابع، والفكاهة.
استخدم `--blind-judge-models` عند مقارنة المزوّدين: إذ يستمر موجه التحكيم في تلقي
كل سجل نصي وحالة التشغيل، لكن تُستبدل مراجع المرشحين بعلامات محايدة مثل
`candidate-01`؛ ويعيد التقرير بعد التحليل ربط الترتيب بالمراجع الحقيقية.
تعمل تشغيلات المرشحين افتراضيًا عند مستوى تفكير `high`، مع `medium` لـ GPT-5.5 و`xhigh`
لمراجع التقييم الأقدم من OpenAI التي تدعمه. وتجاوز مرشحًا معينًا inline باستخدام
`--model provider/model,thinking=<level>`. ولا يزال `--thinking <level>` يضبط
قيمة احتياطية عامة، كما يُحتفَظ بالصيغة الأقدم `--model-thinking <provider/model=level>` للتوافق.
تعمل مراجع مرشحي OpenAI افتراضيًا في الوضع السريع بحيث تُستخدم المعالجة ذات
الأولوية حيثما دعمها المزوّد. أضف `,fast` أو `,no-fast` أو `,fast=false` inline عندما
يحتاج مرشح واحد أو حكم واحد إلى تجاوز. ولا تمرر `--fast` إلا عندما تريد
فرض الوضع السريع على كل نموذج مرشح. ويتم تسجيل مدد المرشحين والحكام في
التقرير لتحليل القياس المرجعي، لكن موجهات التحكيم تنص صراحةً على عدم الترتيب حسب السرعة.
تكون تشغيلات النماذج المرشحة ونماذج التحكيم افتراضيًا عند درجة توازٍ 16. اخفض
`--concurrency` أو `--judge-concurrency` عندما تجعل حدود المزوّد أو ضغط Gateway المحلي
التشغيل صاخبًا أكثر من اللازم.
عند عدم تمرير أي `--model` للمرشح، فإن character eval يستخدم افتراضيًا
`openai/gpt-5.5` و`openai/gpt-5.2` و`openai/gpt-5` و`anthropic/claude-opus-4-6` و
`anthropic/claude-sonnet-4-6` و`zai/glm-5.1` و
`moonshot/kimi-k2.5` و
`google/gemini-3.1-pro-preview` عند عدم تمرير `--model`.
وعند عدم تمرير `--judge-model`، تكون النماذج الحاكمة افتراضيًا:
`openai/gpt-5.5,thinking=xhigh,fast` و
`anthropic/claude-opus-4-6,thinking=high`.
