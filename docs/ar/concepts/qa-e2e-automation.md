# أتمتة ضمان الجودة من البداية إلى النهاية

تهدف حزمة ضمان الجودة الخاصة إلى اختبار OpenClaw بطريقة أكثر واقعية وبهيئة قنوات، مقارنةً بما يمكن أن يحققه اختبار وحدات واحد.

المكوّنات الحالية:

- `extensions/qa-channel`: قناة رسائل اصطناعية تتضمن أسطح الرسائل الخاصة، والقنوات، والخيوط، والتفاعلات، والتعديل، والحذف.
- `extensions/qa-lab`: واجهة Debugger وممر ضمان جودة لمراقبة النص الحواري، وحقن الرسائل الواردة، وتصدير تقرير Markdown.
- `qa/`: أصول تهيئة مدعومة بالمستودع لمهمة البداية وسيناريوهات ضمان الجودة الأساسية.

تدفّق تشغيل ضمان الجودة الحالي عبارة عن موقع ضمان جودة ذي لوحتين:

- اليسار: لوحة تحكم Gateway ‏(Control UI) مع الوكيل.
- اليمين: QA Lab، ويعرض نصًا حواريًا شبيهًا بـ Slack وخطة السيناريو.

شغّله باستخدام:

```bash
pnpm qa:lab:up
```

يقوم ذلك ببناء موقع ضمان الجودة، وبدء مسار Gateway مدعومًا بـ Docker، وكشف صفحة QA Lab حيث يمكن لمشغّل أو لحلقة أتمتة إسناد مهمة ضمان جودة إلى الوكيل، ومراقبة سلوك القناة الفعلي، وتسجيل ما نجح وما فشل وما بقي معطّلًا.

للحصول على تكرار أسرع في واجهة QA Lab من دون إعادة بناء صورة Docker في كل مرة، ابدأ الحزمة باستخدام حزمة QA Lab مركّبة عبر bind mount:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

يبقي `qa:lab:up:fast` خدمات Docker على صورة مبنية مسبقًا، ويجري bind-mount للدليل `extensions/qa-lab/web/dist` داخل حاوية `qa-lab`. ويعيد `qa:lab:watch` بناء تلك الحزمة عند التغيير، ويُعاد تحميل المتصفح تلقائيًا عند تغيّر بصمة أصول QA Lab.

لتشغيل مسار فحص Matrix حقيقي على مستوى النقل، شغّل:

```bash
pnpm openclaw qa matrix
```

يقوم هذا المسار بتهيئة خادم منزلي Tuwunel مؤقت داخل Docker، وتسجيل مستخدمين مؤقتين للسائق وSUT والمراقب، وإنشاء غرفة خاصة واحدة، ثم تشغيل Plugin ‏Matrix الفعلي داخل عملية فرعية لـ QA gateway. ويحافظ مسار النقل الحي على إعدادات العملية الفرعية محصورة في النقل قيد الاختبار، بحيث يعمل Matrix من دون `qa-channel` في إعدادات العملية الفرعية. كما يكتب مخرجات التقرير المنظَّمة وسجلًا مدمجًا لـ stdout/stderr داخل دليل إخراج Matrix QA المحدد. ولالتقاط مخرجات البناء/المشغّل الخارجية من `scripts/run-node.mjs` أيضًا، اضبط `OPENCLAW_RUN_NODE_OUTPUT_LOG=<path>` إلى ملف سجل محلي داخل المستودع.

لتشغيل مسار فحص Telegram حقيقي على مستوى النقل، شغّل:

```bash
pnpm openclaw qa telegram
```

يستهدف هذا المسار مجموعة Telegram خاصة حقيقية واحدة بدلًا من تهيئة خادم مؤقت. ويتطلب `OPENCLAW_QA_TELEGRAM_GROUP_ID` و`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN` و`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`، بالإضافة إلى روبوتين مختلفين داخل المجموعة الخاصة نفسها. يجب أن يملك روبوت SUT اسم مستخدم Telegram، وتعمل المراقبة بين الروبوتات على أفضل نحو عندما يكون لدى كلا الروبوتين وضع Bot-to-Bot Communication Mode مفعّلًا في `@BotFather`.

تتشارك مسارات النقل الحية الآن عقدًا أصغر واحدًا بدلًا من أن يبتكر كل مسار منها شكل قائمة السيناريوهات الخاصة به.

يبقى `qa-channel` مجموعة الاختبارات الاصطناعية الواسعة لسلوك المنتج وليس جزءًا من مصفوفة تغطية النقل الحي.

| المسار | Canary | حصر الإشارات | حظر قائمة السماح | رد على المستوى الأعلى | الاستئناف بعد إعادة التشغيل | متابعة الخيط | عزل الخيط | رصد التفاعل | أمر المساعدة |
| ------ | ------ | ------------ | ---------------- | ---------------------- | ---------------------------- | ------------ | --------- | ------------ | ------------ |
| Matrix   | x      | x              | x               | x               | x              | x                | x                | x                    |              |
| Telegram | x      |                |                 |                 |                |                  |                  |                      | x            |

يحافظ هذا على `qa-channel` بوصفه مجموعة اختبارات سلوك المنتج الواسعة، بينما تتشارك Matrix وTelegram ووسائط النقل الحية المستقبلية قائمة تحقق صريحة واحدة لعقد النقل.

لتشغيل مسار Linux VM مؤقت من دون إدخال Docker في مسار ضمان الجودة، شغّل:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

يقوم هذا بتشغيل ضيف Multipass جديد، وتثبيت الاعتماديات، وبناء OpenClaw داخل الضيف، وتشغيل `qa suite`، ثم نسخ تقرير ضمان الجودة العادي والملخص إلى `.artifacts/qa-e2e/...` على المضيف.
ويعيد استخدام سلوك اختيار السيناريو نفسه الذي تستخدمه `qa suite` على المضيف.
وتنفّذ عمليات suite على المضيف وMultipass عدة سيناريوهات محددة بالتوازي مع عمّال Gateway معزولين افتراضيًا، حتى 64 عاملًا أو عدد السيناريوهات المحددة. استخدم `--concurrency <count>` لضبط عدد العمّال، أو `--concurrency 1` للتنفيذ التسلسلي.
وتُمرَّر في التشغيلات الحية مدخلات مصادقة ضمان الجودة المدعومة والعملية داخل الضيف: مفاتيح الموفر المعتمدة على البيئة، ومسار إعدادات موفر QA الحي، و`CODEX_HOME` عند وجوده. أبقِ `--output-dir` تحت جذر المستودع حتى يتمكن الضيف من الكتابة عكسيًا عبر مساحة العمل المركّبة.

## البذور المدعومة بالمستودع

توجد أصول التهيئة في `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/*.md`

وهذه موجودة عمدًا في git حتى تكون خطة ضمان الجودة مرئية لكلٍّ من البشر والوكيل.

يجب أن يظل `qa-lab` مشغّل Markdown عامًا. ويجب أن يكون كل ملف Markdown للسيناريو هو مصدر الحقيقة لتشغيل اختبار واحد، وأن يعرّف ما يلي:

- بيانات وصفية للسيناريو
- مراجع الوثائق والكود
- متطلبات Plugin اختيارية
- تصحيح اختياري لإعدادات Gateway
- `qa-flow` القابل للتنفيذ

يُسمح لسطح وقت التشغيل القابل لإعادة الاستخدام الذي يدعم `qa-flow` بأن يظل عامًا وعابرًا للجوانب. على سبيل المثال، يمكن لسيناريوهات Markdown أن تجمع بين أدوات مساعدة على جانب النقل وأخرى على جانب المتصفح تدفع Control UI المضمّنة عبر واجهة Gateway ‏`browser.request` من دون إضافة مشغّل مخصص لحالة خاصة.

يجب أن تظل القائمة الأساسية واسعة بما يكفي لتغطية:

- المحادثة في الرسائل الخاصة والقنوات
- سلوك الخيوط
- دورة حياة إجراءات الرسائل
- استدعاءات Cron
- استرجاع الذاكرة
- تبديل النماذج
- تسليم المهام إلى وكيل فرعي
- قراءة المستودع وقراءة الوثائق
- مهمة بناء صغيرة واحدة مثل Lobster Invaders

## مسارات محاكاة الموفّر

يتضمن `qa suite` مسارين محليين لمحاكاة الموفّر:

- `mock-openai` هو محاكي OpenClaw الواعي بالسيناريو. ويظل مسار المحاكاة الحتمي الافتراضي لضمان الجودة المدعوم بالمستودع ولبوابات التماثل.
- `aimock` يشغّل خادم موفّر مدعومًا بـ AIMock لتغطية البروتوكولات والتجهيزات والتسجيل/إعادة التشغيل وحالات الفوضى على نحو تجريبي. وهو إضافة مكملة ولا يستبدل موزّع السيناريو `mock-openai`.

يوجد تنفيذ مسار الموفّر تحت `extensions/qa-lab/src/providers/`.
ويمتلك كل موفّر إعداداته الافتراضية، وبدء تشغيل خادمه المحلي، وإعدادات نموذج Gateway، واحتياجات تجهيز ملف المصادقة، وأعلام القدرات الحية/الوهمية. ويجب أن يمرّ كود suite وGateway المشترك عبر سجل الموفّرين بدلًا من التفرع بناءً على أسماء الموفّرين.

## مُكيّفات النقل

يمتلك `qa-lab` واجهة نقل عامة لسيناريوهات ضمان الجودة المعتمدة على Markdown.
ويُعد `qa-channel` أول مُكيّف على هذه الواجهة، لكن الهدف التصميمي أوسع: يجب أن تتمكن القنوات المستقبلية، سواء كانت حقيقية أو اصطناعية، من الاندماج في مشغّل suite نفسه بدلًا من إضافة مشغّل ضمان جودة خاص بالنقل.

على مستوى المعمارية، يكون التقسيم كما يلي:

- يمتلك `qa-lab` تنفيذ السيناريوهات العامة، وتوازي العمّال، وكتابة المخرجات، وإعداد التقارير.
- يمتلك مُكيّف النقل إعدادات Gateway، والجاهزية، ومراقبة الوارد والصادر، وإجراءات النقل، وحالة النقل المطَبَّعة.
- تحدد ملفات سيناريو Markdown ضمن `qa/scenarios/` تشغيل الاختبار؛ ويوفّر `qa-lab` سطح وقت التشغيل القابل لإعادة الاستخدام الذي ينفذها.

توجد إرشادات الاعتماد الموجهة للمشرفين لمُكيّفات القنوات الجديدة في
[الاختبار](/ar/help/testing#adding-a-channel-to-qa).

## إعداد التقارير

يصدّر `qa-lab` تقرير بروتوكول بصيغة Markdown من المخطط الزمني للممر المرصود.
ويجب أن يجيب التقرير عن:

- ما الذي نجح
- ما الذي فشل
- ما الذي بقي معطّلًا
- ما سيناريوهات المتابعة الجديرة بالإضافة

لفحوصات الطابع والأسلوب، شغّل السيناريو نفسه عبر عدة مراجع لنماذج حية واكتب تقرير Markdown خاضعًا للتحكيم:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.4,thinking=xhigh \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.4,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

يشغّل هذا الأمر عمليات QA gateway فرعية محلية، وليس Docker. يجب أن تضبط سيناريوهات تقييم الطابع الشخصية عبر `SOUL.md`، ثم تُجري أدوار مستخدم عادية مثل المحادثة، ومساعدة مساحة العمل، والمهام الصغيرة على الملفات. ويجب ألا يُبلَّغ النموذج المرشح بأنه قيد التقييم. ويحافظ الأمر على كل نص حواري كامل، ويسجل إحصاءات التشغيل الأساسية، ثم يطلب من نماذج التحكيم في الوضع السريع مع استدلال `xhigh` ترتيب التشغيلات بحسب الطبيعية والطابع والفكاهة.
استخدم `--blind-judge-models` عند المقارنة بين الموفّرين: سيظل موجّه التحكيم يتلقى كل نص حواري وحالة تشغيل، لكن ستُستبدل مراجع المرشحين بتسميات محايدة مثل `candidate-01`؛ ويعيد التقرير ربط الترتيب بالمراجع الحقيقية بعد التحليل.
تستخدم تشغيلات المرشحين افتراضيًا مستوى تفكير `high`، مع `xhigh` لنماذج OpenAI التي تدعمه. ويمكنك تجاوز مرشح محدد ضمنيًا باستخدام
`--model provider/model,thinking=<level>`. وما زال `--thinking <level>` يضبط قيمة احتياطية عامة، كما يُحتفظ بالشكل الأقدم `--model-thinking <provider/model=level>` من أجل التوافق.
تستخدم مراجع المرشحين الخاصة بـ OpenAI الوضع السريع افتراضيًا بحيث تُستخدم المعالجة ذات الأولوية عندما يدعمها الموفّر. أضف `,fast` أو `,no-fast` أو `,fast=false` ضمنيًا عندما يحتاج مرشح واحد أو حكم واحد إلى تجاوز. مرّر `--fast` فقط عندما تريد فرض الوضع السريع على كل نموذج مرشح. وتُسجَّل مدد تشغيل المرشحين والحكام في التقرير لتحليل المقاييس، لكن توجيهات التحكيم تنص صراحةً على عدم الترتيب حسب السرعة.
تستخدم تشغيلات نماذج المرشحين والحكام افتراضيًا توازيًا مقداره 16. خفّض
`--concurrency` أو `--judge-concurrency` عندما تجعل حدود الموفّر أو ضغط Gateway المحلي التشغيل صاخبًا أكثر من اللازم.
عندما لا يُمرَّر أي `--model` للمرشح، يستخدم تقييم الطابع افتراضيًا
`openai/gpt-5.4` و`openai/gpt-5.2` و`openai/gpt-5` و`anthropic/claude-opus-4-6` و`anthropic/claude-sonnet-4-6` و`zai/glm-5.1` و
`moonshot/kimi-k2.5` و
`google/gemini-3.1-pro-preview`.
وعندما لا يُمرَّر أي `--judge-model`، تكون النماذج الحكَمية الافتراضية هي
`openai/gpt-5.4,thinking=xhigh,fast` و
`anthropic/claude-opus-4-6,thinking=high`.
