# أتمتة ضمان الجودة الشاملة من البداية إلى النهاية

تهدف حزمة ضمان الجودة الخاصة إلى اختبار OpenClaw بطريقة أكثر واقعية،
وبصيغة تشبه القنوات أكثر مما يمكن أن يقدمه اختبار وحدة واحد.

المكوّنات الحالية:

- `extensions/qa-channel`: قناة رسائل اصطناعية تتضمن أسطح الرسائل الخاصة DM، والقنوات، والخيوط،
  والتفاعلات، والتعديل، والحذف.
- `extensions/qa-lab`: واجهة مستخدم لتصحيح الأخطاء وناقل ضمان الجودة لمراقبة النص الحواري،
  وحقن الرسائل الواردة، وتصدير تقرير Markdown.
- `qa/`: أصول تهيئة مدعومة بالمستودع لمهمة الانطلاق وخط الأساس لسيناريوهات
  ضمان الجودة.

مسار عمل مشغّل ضمان الجودة الحالي هو موقع ضمان جودة مكوّن من لوحتين:

- اليسار: لوحة معلومات Gateway ‏(Control UI) مع الوكيل.
- اليمين: QA Lab، تعرض النص الحواري بأسلوب يشبه Slack وخطة السيناريو.

شغّله باستخدام:

```bash
pnpm qa:lab:up
```

يقوم ذلك ببناء موقع ضمان الجودة، وبدء مسار Gateway المدعوم بـ Docker، وإتاحة
صفحة QA Lab حيث يمكن لمشغّل أو حلقة أتمتة إعطاء الوكيل مهمة ضمان جودة،
ومراقبة سلوك القناة الحقيقي، وتسجيل ما نجح وما فشل وما بقي معطّلًا.

للحصول على تكرار أسرع في واجهة مستخدم QA Lab بدون إعادة بناء صورة Docker في كل مرة،
ابدأ الحزمة مع حزمة QA Lab مركّبة عبر bind mount:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

يبقي `qa:lab:up:fast` خدمات Docker على صورة مبنية مسبقًا ويركّب عبر bind mount
`extensions/qa-lab/web/dist` داخل حاوية `qa-lab`. ويعيد `qa:lab:watch`
بناء تلك الحزمة عند التغيير، ويُعاد تحميل المتصفح تلقائيًا عند تغيّر hash أصل QA Lab.

لتشغيل مسار Matrix تجريبي عبر نقل حقيقي، استخدم:

```bash
pnpm openclaw qa matrix
```

يقوم هذا المسار بتجهيز خادم Tuwunel منزلي مؤقت داخل Docker، ويسجل
مستخدمين مؤقتين للسائق وSUT والمراقب، وينشئ غرفة خاصة واحدة، ثم يشغّل
إضافة Matrix الحقيقية داخل عملية فرعية QA gateway. يحافظ مسار النقل الحي
على ضبط العملية الفرعية مقتصرًا على النقل قيد الاختبار، لذلك يعمل Matrix بدون
`qa-channel` في ضبط العملية الفرعية.

لتشغيل مسار Telegram تجريبي عبر نقل حقيقي، استخدم:

```bash
pnpm openclaw qa telegram
```

يستهدف هذا المسار مجموعة Telegram خاصة حقيقية واحدة بدلًا من تجهيز خادم
مؤقت. ويتطلب `OPENCLAW_QA_TELEGRAM_GROUP_ID`،
و`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN`، و
`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`، بالإضافة إلى
روبوتين مختلفين داخل المجموعة الخاصة نفسها. يجب أن يمتلك روبوت SUT اسم مستخدم
في Telegram، وتعمل المراقبة بين الروبوتات بأفضل صورة عندما يكون كلا الروبوتين قد فعّلا
وضع التواصل بين الروبوتات في `@BotFather`.

تشارك مسارات النقل الحية الآن عقدًا أصغر واحدًا بدلًا من أن يبتكر كل منها
شكل قائمة السيناريوهات الخاص به.

يبقى `qa-channel` مجموعة سلوك المنتج الاصطناعية الواسعة ولا يُعد جزءًا
من مصفوفة تغطية النقل الحي.

| المسار | Canary | تقييد الإشارات | حظر قائمة السماح | رد من المستوى الأعلى | الاستئناف بعد إعادة التشغيل | متابعة الخيط | عزل الخيط | مراقبة التفاعل | أمر المساعدة |
| ------- | ------ | -------------- | ----------------- | --------------------- | ---------------------------- | ------------- | ---------- | ---------------- | ------------ |
| Matrix  | x      | x              | x                 | x                     | x                            | x             | x          | x                |              |
| Telegram | x      |                |                   |                       |                              |               |            |                  | x            |

يحافظ هذا على `qa-channel` كحزمة سلوك المنتج الواسعة، بينما تشترك
Matrix وTelegram ووسائط النقل الحية المستقبلية في قائمة تحقق صريحة واحدة لعقد النقل.

لتشغيل مسار آلة Linux افتراضية مؤقتة بدون إدخال Docker في مسار ضمان الجودة، استخدم:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

يقوم هذا بتشغيل ضيف Multipass جديد، وتثبيت التبعيات، وبناء OpenClaw
داخل الضيف، وتشغيل `qa suite`، ثم نسخ تقرير وضمان الجودة والملخص المعتادين
مرة أخرى إلى `.artifacts/qa-e2e/...` على المضيف.
ويعيد استخدام سلوك اختيار السيناريو نفسه المستخدم في `qa suite` على المضيف.
تقوم عمليات التشغيل على المضيف وMultipass بتنفيذ عدة سيناريوهات محددة بالتوازي
مع عمّال Gateway معزولين افتراضيًا، حتى 64 عاملًا أو عدد السيناريوهات المحدد. استخدم
`--concurrency <count>` لضبط عدد العمّال، أو `--concurrency 1` للتنفيذ التسلسلي.
تقوم عمليات التشغيل الحية بتمرير مُدخلات مصادقة QA المدعومة والعملية
للضيف: مفاتيح المزوّد المعتمدة على env، ومسار ضبط مزوّد QA الحي، و
`CODEX_HOME` عند وجوده. أبقِ `--output-dir` تحت جذر المستودع حتى يتمكن الضيف
من الكتابة مرة أخرى عبر مساحة العمل المركبة.

## التهيئات المدعومة بالمستودع

توجد أصول التهيئة في `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/*.md`

توجد هذه عمدًا في git حتى تكون خطة ضمان الجودة مرئية للبشر
وللوكيل معًا. يجب أن تظل قائمة خط الأساس واسعة بما يكفي لتغطية:

- دردشة DM والقنوات
- سلوك الخيوط
- دورة حياة إجراءات الرسائل
- استدعاءات cron الراجعة
- استدعاء الذاكرة
- تبديل النموذج
- تسليم المهمة إلى وكيل فرعي
- قراءة المستودع وقراءة الوثائق
- مهمة بناء صغيرة مثل Lobster Invaders

## التقارير

يصدر `qa-lab` تقرير بروتوكول Markdown من المخطط الزمني المرصود للناقل.
يجب أن يجيب التقرير عن:

- ما الذي نجح
- ما الذي فشل
- ما الذي بقي معطّلًا
- ما سيناريوهات المتابعة الجديرة بالإضافة

لفحوصات الطابع والأسلوب، شغّل السيناريو نفسه عبر عدة
مراجع نماذج حية واكتب تقرير Markdown مُقيّمًا:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.4,thinking=xhigh \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.4,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

يشغّل هذا الأمر عمليات فرعية محلية لـ QA gateway، وليس Docker. ينبغي أن تضبط
سيناريوهات تقييم الطابع الشخصية عبر `SOUL.md`، ثم تشغّل أدوار مستخدم عادية
مثل الدردشة، ومساعدة مساحة العمل، ومهام الملفات الصغيرة. يجب ألا يُخبَر
النموذج المرشح بأنه قيد التقييم. يحتفظ الأمر بكل نص حواري كامل،
ويسجل إحصاءات تشغيل أساسية، ثم يطلب من نماذج التحكيم في الوضع السريع مع
استدلال `xhigh` ترتيب التشغيلات بحسب الطبيعية والأجواء والفكاهة.
استخدم `--blind-judge-models` عند مقارنة المزوّدين:
فما يزال طلب التحكيم يتلقى كل نص حواري وحالة تشغيل، لكن تُستبدل
مراجع المرشحين بتسميات محايدة مثل `candidate-01`؛ ويربط التقرير
الترتيبات مرة أخرى بالمراجع الحقيقية بعد التحليل.
تستخدم تشغيلات النماذج المرشحة افتراضيًا مستوى تفكير `high`،
مع `xhigh` لنماذج OpenAI التي تدعمه. يمكنك تجاوز مرشح معيّن ضمنيًا باستخدام
`--model provider/model,thinking=<level>`. وما يزال `--thinking <level>` يضبط
بديلًا عامًا، بينما يبقى الشكل الأقدم `--model-thinking <provider/model=level>`
لأغراض التوافق.
تستخدم مراجع المرشحين من OpenAI الوضع السريع افتراضيًا حتى تُستخدم المعالجة ذات الأولوية
حيث يدعمها المزوّد. أضف `,fast` أو `,no-fast` أو `,fast=false` ضمنيًا عندما يحتاج
مرشح واحد أو حكم واحد إلى تجاوز. مرّر `--fast` فقط عندما تريد
فرض الوضع السريع على كل نموذج مرشح. تُسجّل مدد المرشحين والحكام
في التقرير لتحليل المقاييس، لكن طلبات التحكيم تنص صراحة على
عدم الترتيب وفق السرعة.
يستخدم كل من تشغيلات النماذج المرشحة والحكام افتراضيًا درجة توازٍ 16. خفّض
`--concurrency` أو `--judge-concurrency` عندما تجعل حدود المزوّد أو ضغط Gateway المحلي
التشغيل شديد الضجيج.
عند عدم تمرير أي مرشح `--model`، يستخدم تقييم الطابع افتراضيًا
`openai/gpt-5.4` و`openai/gpt-5.2` و`openai/gpt-5` و`anthropic/claude-opus-4-6`،
و`anthropic/claude-sonnet-4-6` و`zai/glm-5.1`،
و`moonshot/kimi-k2.5`، و
`google/gemini-3.1-pro-preview` عندما لا يتم تمرير `--model`.
وعند عدم تمرير أي `--judge-model`، تكون النماذج الحاكمة افتراضيًا:
`openai/gpt-5.4,thinking=xhigh,fast` و
`anthropic/claude-opus-4-6,thinking=high`.
