Mantis هو نظام التحقق الشامل من OpenClaw للأخطاء التي تحتاج إلى بيئة تشغيل
حقيقية، ووسيط نقل حقيقي، ودليل مرئي. يشغّل سيناريو مقابل مرجع معروف بأنه
سيئ، ويلتقط الأدلة، ثم يشغّل السيناريو نفسه مقابل مرجع مرشح، وينشر المقارنة
كArtifacts يمكن للمشرف فحصها من PR أو من أمر محلي.

يبدأ Mantis بـ Discord لأن Discord يمنحنا مسارا أوليا عالي القيمة:
مصادقة بوت حقيقية، وقنوات guild حقيقية، وتفاعلات، وسلاسل محادثات، وأوامر أصلية، وواجهة
متصفح يمكن للبشر من خلالها تأكيد ما عرضه وسيط النقل بصريا.

## الأهداف

- إعادة إنتاج خطأ من issue أو PR في GitHub بنفس شكل وسيط النقل الذي يراه
  المستخدمون.
- التقاط artifact **قبل** على مرجع الأساس قبل تطبيق الإصلاح.
- التقاط artifact **بعد** على المرجع المرشح بعد تطبيق الإصلاح.
- استخدام محك تحقق حتمي كلما أمكن، مثل قراءة تفاعل Discord عبر REST أو
  فحص نسخة قناة.
- التقاط لقطات شاشة عندما يكون للخطأ سطح UI مرئي.
- التشغيل محليا من CLI يتحكم به agent وعن بعد من GitHub.
- الحفاظ على حالة كافية للآلة لإنقاذ VNC عندما يتعطل تسجيل الدخول أو أتمتة المتصفح أو
  مصادقة provider.
- نشر حالة موجزة إلى قناة Discord للمشغل عندما يكون التشغيل محظورا، أو
  يحتاج إلى مساعدة VNC يدوية، أو ينتهي.

## ما ليس من الأهداف

- Mantis ليس بديلا لاختبارات الوحدة. ينبغي أن يتحول تشغيل Mantis عادة
  إلى اختبار انحدار أصغر بعد فهم الإصلاح.
- Mantis ليس بوابة CI السريعة العادية. فهو أبطأ، ويستخدم بيانات اعتماد حية، و
  مخصص للأخطاء التي تكون فيها البيئة الحية مهمة.
- يجب ألا يتطلب Mantis تدخلا بشريا للتشغيل العادي. VNC اليدوي هو مسار إنقاذ
  وليس المسار السعيد.
- لا يخزن Mantis الأسرار الخام في artifacts أو السجلات أو لقطات الشاشة أو تقارير Markdown
  أو تعليقات PR.

## الملكية

يعيش Mantis في مكدس QA الخاص بـ OpenClaw.

- تملك OpenClaw بيئة تشغيل السيناريو، ومحوّلات وسيط النقل، ومخطط الأدلة، و
  CLI المحلي ضمن `pnpm openclaw qa mantis`.
- يملك QA Lab أجزاء حزمة وسيط النقل الحية، ومساعدات التقاط المتصفح، و
  كاتبات artifacts.
- يملك Crabbox آلات Linux الجاهزة عندما تكون هناك حاجة إلى VM بعيدة.
- تملك GitHub Actions نقطة دخول سير العمل البعيد والاحتفاظ بـ artifacts.
- يملك ClawSweeper توجيه تعليقات GitHub: تحليل أوامر المشرفين،
  وإرسال سير العمل، ونشر تعليق PR النهائي.
- تقود agents الخاصة بـ OpenClaw نظام Mantis عبر Codex عندما يحتاج السيناريو إلى إعداد agentic،
  أو تصحيح أخطاء، أو الإبلاغ عن حالة عالقة.

يبقي هذا الحد معرفة وسيط النقل في OpenClaw، وجدولة الآلات في
Crabbox، وغراء سير عمل المشرفين في ClawSweeper.

## شكل الأمر

يتحقق الأمر المحلي الأول من بوت Discord، وguild، والقناة، وإرسال الرسالة،
وإرسال التفاعل، ومسار artifact:

```bash
pnpm openclaw qa mantis discord-smoke \
  --output-dir .artifacts/qa-e2e/mantis/discord-smoke
```

يقبل مشغّل ما قبل وما بعد المحلي هذا الشكل:

```bash
pnpm openclaw qa mantis run \
  --transport discord \
  --scenario discord-status-reactions-tool-only \
  --baseline origin/main \
  --candidate HEAD \
  --output-dir .artifacts/qa-e2e/mantis/local-discord-status-reactions
```

ينشئ المشغّل worktrees منفصلة لمرجع الأساس والمرجع المرشح ضمن دليل الإخراج،
ويثبت الاعتماديات، ويبني كل ref، ويشغّل السيناريو مع
`--allow-failures`، ثم يكتب `baseline/` و`candidate/` و`comparison.json`
و`mantis-report.md`. بالنسبة إلى أول سيناريو Discord، يعني التحقق الناجح
أن حالة الأساس هي `fail` وأن حالة المرشح هي `pass`.

سير عمل smoke في GitHub هو `Mantis Discord Smoke`. أما سير عمل ما قبل وما بعد في GitHub
لأول سيناريو حقيقي فهو `Mantis Discord Status Reactions`. يقبل:

- `baseline_ref`: المرجع المتوقع أن يعيد إنتاج سلوك queued-only.
- `candidate_ref`: المرجع المتوقع أن يعرض `queued -> thinking -> done`.

يفحص مرجع حزمة سير العمل، ويبني worktrees منفصلة لمرجع الأساس والمرشح،
ويشغّل `discord-status-reactions-tool-only` مقابل كل worktree، ويرفع
`baseline/` و`candidate/` و`comparison.json` و`mantis-report.md` كـ
artifacts في Actions.

يمكنك أيضا تشغيل تنفيذ status-reactions مباشرة من تعليق PR:

```text
@Mantis discord status reactions
```

مشغّل التعليق ضيق عمدا. لا يعمل إلا على تعليقات pull request
من مستخدمين لديهم صلاحية write أو maintain أو admin، ولا يتعرف إلا على
طلبات status-reaction الخاصة بـ Discord. افتراضيا يستخدم مرجع الأساس السيئ المعروف
وSHA رأس PR الحالي كمرشح. يمكن للمشرفين تجاوز أي من
المرجعين:

```text
@Mantis discord status reactions baseline=origin/main candidate=HEAD
```

أمثلة أوامر ClawSweeper:

```text
@clawsweeper mantis discord discord-status-reactions-tool-only
@clawsweeper verify e2e discord
```

الأمر الأول صريح ومركّز على السيناريو. يمكن للثاني لاحقا ربط PR
أو issue بسيناريوهات Mantis الموصى بها من التسميات، والملفات المعدلة، و
نتائج مراجعة ClawSweeper.

## دورة التشغيل

1. الحصول على بيانات الاعتماد.
2. تخصيص VM أو إعادة استخدامها.
3. إعداد checkout نظيف لمرجع الأساس.
4. تثبيت الاعتماديات وبناء ما يحتاجه السيناريو فقط.
5. بدء OpenClaw Gateway فرعي بدليل حالة معزول.
6. تكوين وسيط النقل الحي، وprovider، والنموذج، وملف تعريف المتصفح.
7. تشغيل السيناريو والتقاط أدلة الأساس.
8. إيقاف Gateway والحفاظ على السجلات.
9. إعداد المرجع المرشح في VM نفسها.
10. تشغيل السيناريو نفسه والتقاط أدلة المرشح.
11. مقارنة نتائج محك التحقق والأدلة المرئية.
12. كتابة Markdown وJSON والسجلات ولقطات الشاشة وartifacts التتبع الاختيارية.
13. رفع artifacts إلى GitHub Actions.
14. نشر رسالة حالة موجزة إلى PR أو Discord.

ينبغي أن يكون السيناريو قادرا على الفشل بطريقتين مختلفتين:

- **أعيد إنتاج الخطأ**: فشل الأساس بالطريقة المتوقعة.
- **فشل الحزمة**: فشل إعداد البيئة أو بيانات الاعتماد أو Discord API أو المتصفح أو
  provider قبل أن يكون محك الخطأ ذا معنى.

يجب أن يفصل التقرير النهائي بين هاتين الحالتين حتى لا يخلط المشرفون بين بيئة
غير مستقرة وسلوك المنتج.

## الحد الأدنى القابل للتطبيق لـ Discord

ينبغي أن يستهدف السيناريو الأول تفاعلات حالة Discord في قنوات guild حيث
يكون وضع تسليم رد المصدر هو `message_tool_only`.

لماذا هو بذرة جيدة لـ Mantis:

- إنه مرئي في Discord كتفاعلات على الرسالة المشغّلة.
- لديه محك REST قوي من خلال حالة تفاعل رسالة Discord.
- يمرّن OpenClaw Gateway حقيقيا، ومصادقة بوت Discord، وإرسال الرسائل،
  ووضع تسليم رد المصدر، وحالة تفاعل الحالة، ودورة حياة دور النموذج.
- إنه ضيق بما يكفي لإبقاء التنفيذ الأول دقيقا.

شكل السيناريو المتوقع:

```yaml
id: discord-status-reactions-tool-only
transport: discord
baseline:
  expect:
    reproduced: true
candidate:
  expect:
    fixed: true
config:
  messages:
    ackReaction: "👀"
    ackReactionScope: "group-mentions"
    groupChat:
      visibleReplies: "message_tool"
    statusReactions:
      enabled: true
      timing:
        debounceMs: 0
discord:
  requireMention: true
  notifyChannel: operator-notify
evidence:
  rest:
    messageReactions: true
  browser:
    screenshotMessageRow: true
```

ينبغي أن تظهر أدلة الأساس تفاعل إقرار queued لكن بلا انتقال في دورة الحياة
في وضع tool-only. وينبغي أن تظهر أدلة المرشح تفاعلات حالة دورة الحياة وهي تعمل عندما يكون
`messages.statusReactions.enabled` مفعلا صراحة.

الشريحة التنفيذية الأولى هي سيناريو QA الحي الاختياري لـ Discord:

```bash
pnpm openclaw qa discord \
  --scenario discord-status-reactions-tool-only \
  --provider-mode live-frontier \
  --model openai/gpt-5.4 \
  --alt-model openai/gpt-5.4 \
  --fast \
  --output-dir .artifacts/qa-e2e/mantis/discord-status-reactions-candidate
```

يكون SUT مع معالجة guild دائمة التشغيل، و`visibleReplies:
"message_tool"`، و`ackReaction: "👀"`، وتفاعلات حالة صريحة. يستطلع محك التحقق
رسالة Discord المشغّلة الحقيقية ويتوقع التسلسل المرصود
`👀 -> 🤔 -> 👍`. تتضمن artifacts `discord-qa-reaction-timelines.json`،
و`discord-status-reactions-tool-only-timeline.html`، و
`discord-status-reactions-tool-only-timeline.png`.

## أجزاء QA الموجودة

ينبغي أن يبني Mantis على مكدس QA الخاص الموجود بدلا من البدء من
الصفر:

- `pnpm openclaw qa discord` يشغّل بالفعل مسار Discord حيا مع بوتات driver و
  SUT.
- يكتب مشغّل وسيط النقل الحي التقارير وartifacts الرسائل المرصودة بالفعل
  ضمن `.artifacts/qa-e2e/`.
- توفر حجوزات بيانات اعتماد Convex وصولا حصريا إلى بيانات اعتماد وسيط النقل الحية
  المشتركة.
- تدعم خدمة التحكم في المتصفح بالفعل لقطات الشاشة، واللقطات، و
  ملفات التعريف المدارة headless، وملفات تعريف CDP البعيدة.
- لدى QA Lab بالفعل واجهة مصحح أخطاء وbus للاختبار ذي شكل وسيط النقل.

يمكن أن يكون تنفيذ Mantis الأول مشغّل ما قبل/ما بعد رقيقا فوق هذه
الأجزاء، بالإضافة إلى طبقة دليل مرئي واحدة.

Markdown مخصص لتعليقات PR والمراجعة البشرية.

يجب أن يتضمن الملخص:

- refs وSHAs التي تم اختبارها
- وسيط النقل ومعرف السيناريو
- provider الآلة ومعرف الآلة أو معرف الحجز
- مصدر بيانات الاعتماد دون قيم الأسرار
- نتيجة الأساس
- نتيجة المرشح
- ما إذا أعيد إنتاج الخطأ على الأساس
- ما إذا أصلحه المرشح
- مسارات artifacts
- مشكلات إعداد أو تنظيف منقحة

لقطات الشاشة أدلة، وليست أسرارا. ومع ذلك لا تزال تحتاج إلى انضباط في التنقيح:
قد تظهر أسماء قنوات خاصة، أو أسماء مستخدمين، أو محتوى رسائل. بالنسبة إلى PRs العامة،
فضّل روابط artifacts في GitHub Actions على الصور المضمنة حتى تصبح قصة التنقيح
أقوى.

## المتصفح وVNC

لمسار المتصفح وضعان:

- **أتمتة headless**: الافتراضي لـ CI. يعمل Chrome مع تمكين CDP، و
  يلتقط Playwright أو تحكم متصفح OpenClaw لقطات الشاشة.
- **إنقاذ VNC**: ممكّن على VM نفسها عندما يحتاج تسجيل الدخول أو MFA أو منع الأتمتة في Discord
  أو التصحيح المرئي إلى تدخل بشري.

ينبغي أن يكون ملف تعريف متصفح مراقب Discord دائما بما يكفي لتجنب
تسجيل الدخول في كل تشغيل، لكنه معزول عن حالة المتصفح الشخصية. ينتمي ملف التعريف
إلى مجموعة آلات Mantis، لا إلى حاسوب مطور محمول.

عندما يتعطل Mantis، ينشر رسالة حالة Discord تتضمن:

- معرف التشغيل
- معرف السيناريو
- provider الآلة
- دليل artifacts
- تعليمات اتصال VNC أو noVNC إن توفرت
- نصا قصيرا للمانع

يمكن للنشر الخاص الأول أن يرسل هذه الرسائل إلى قناة المشغل الموجودة
وينتقل إلى قناة مخصصة لـ Mantis لاحقا.

## الآلات

ينبغي أن يفضل Mantis استخدام AWS عبر Crabbox للتنفيذ البعيد الأول.
يوفر لنا Crabbox آلات جاهزة، وتتبع حجوزات، وترطيبا، وسجلات، ونتائج، و
تنظيفا. إذا كانت سعة AWS بطيئة جدا أو غير متاحة، فأضف provider لـ Hetzner
خلف واجهة الآلة نفسها.

متطلبات VM الدنيا:

- Linux مع تثبيت Chrome أو Chromium قادر على سطح المكتب
- وصول CDP لأتمتة المتصفح
- VNC أو noVNC للإنقاذ
- Node 22 وpnpm
- checkout لـ OpenClaw وذاكرة تخزين مؤقت للاعتماديات
- ذاكرة تخزين مؤقت لمتصفح Playwright Chromium عند استخدام Playwright
- CPU وذاكرة كافيان لتشغيل OpenClaw Gateway واحد، ومتصفح واحد، وتشغيل نموذج واحد
- وصول خارجي إلى Discord وGitHub وproviders النماذج ووسيط بيانات الاعتماد

ينبغي ألا تحتفظ VM بأسرار خام طويلة العمر خارج مخازن بيانات الاعتماد أو
ملفات تعريف المتصفح المتوقعة.

## الأسرار

تعيش الأسرار في أسرار مؤسسة GitHub أو المستودع للتشغيلات البعيدة، وفي
ملف أسرار محلي يتحكم به المشغل للتشغيلات المحلية.

أسماء الأسرار الموصى بها:

- `OPENCLAW_QA_DISCORD_MANTIS_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_DRIVER_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_SUT_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_GUILD_ID`
- `OPENCLAW_QA_DISCORD_CHANNEL_ID`
- `OPENCLAW_QA_DISCORD_NOTIFY_CHANNEL_ID`
- `OPENCLAW_QA_REDACT_PUBLIC_METADATA=1` لعمليات رفع عناصر GitHub الأثرية العامة
- `OPENCLAW_QA_CONVEX_SITE_URL`
- `OPENCLAW_QA_CONVEX_SECRET_CI`

على المدى الطويل، يجب أن يبقى مجمّع بيانات اعتماد Convex هو المصدر العادي لبيانات اعتماد النقل الحية. تعمل أسرار GitHub على تمهيد الوسيط ومسارات الرجوع الاحتياطية.

يجب ألا يطبع مشغّل Mantis أبدًا:

- رموز بوتات Discord
- مفاتيح API للمزوّدين
- ملفات تعريف ارتباط المتصفح
- محتويات ملفات تعريف المصادقة
- كلمات مرور VNC
- حمولات بيانات الاعتماد الخام

يجب أيضًا أن تنقّح عمليات رفع العناصر الأثرية العامة بيانات تعريف هدف Discord مثل معرّفات البوت، والمجموعة، والقناة، والرسالة. يفعّل سير عمل فحص GitHub السريع
`OPENCLAW_QA_REDACT_PUBLIC_METADATA=1` لهذا السبب.

إذا لُصق رمز بالخطأ في issue أو PR أو دردشة أو سجل، فقم بتدويره بعد تخزين السر الجديد.

## عناصر GitHub الأثرية وتعليقات PR

يجب أن ترفع سير عمل Mantis حزمة الأدلة الكاملة كعنصر أثري قصير العمر في Actions. عند تشغيل سير العمل من أجل تقرير خطأ أو PR إصلاح، يجب عليه أيضًا نشر لقطات شاشة PNG المنقّحة إلى فرع `qa-artifacts` وإدراج أو تحديث تعليق على ذلك الخطأ أو PR الإصلاح مع لقطات شاشة قبل/بعد مضمنة. لا تنشر الدليل الأساسي فقط على PR عام لأتمتة QA. تبقى السجلات الخام، والرسائل المرصودة، والأدلة الكبيرة الأخرى في عنصر Actions الأثري.

يجب أن تنشر سير عمل الإنتاج هذه التعليقات باستخدام GitHub App الخاص بـ Mantis، وليس باستخدام `github-actions[bot]`. خزّن معرّف التطبيق والمفتاح الخاص كأسرار GitHub Actions باسم
`MANTIS_GITHUB_APP_ID` و`MANTIS_GITHUB_APP_PRIVATE_KEY`. يستخدم سير العمل علامة مخفية كمفتاح للإدراج أو التحديث، ويحدّث ذلك التعليق عندما يستطيع الرمز تحريره، وينشئ تعليقًا جديدًا مملوكًا لـ Mantis عندما يتعذر تحرير علامة أقدم مملوكة لبوت.

يجب أن يكون تعليق PR قصيرًا ومرئيًا:

```md
Mantis Discord Status Reactions QA

Summary: Mantis reran the reported Discord status-reaction bug against the known
bad baseline and the candidate fix. The baseline reproduced the bug, while the
candidate showed the expected queued -> thinking -> done sequence.

- Scenario: `discord-status-reactions-tool-only`
- Run: <workflow run link>
- Artifact: <artifact link>
- Baseline: `<status>` at `<sha>`
- Candidate: `<status>` at `<sha>`

| Baseline            | Candidate           |
| ------------------- | ------------------- |
| <inline screenshot> | <inline screenshot> |
```

عندما يفشل التشغيل لأن الحاضنة فشلت، يجب أن يقول التعليق ذلك بدلًا من الإيحاء بأن المرشح فشل.

## ملاحظات النشر الخاص

قد يكون لدى نشر خاص تطبيق Mantis Discord بالفعل. أعد استخدام ذلك التطبيق بدلًا من إنشاء تطبيق آخر عندما تكون لديه أذونات البوت الصحيحة ويمكن تدويره بأمان.

اضبط قناة إشعارات المشغّل الأولية عبر الأسرار أو إعدادات النشر. يمكن أن تشير أولًا إلى قناة مشرفين أو عمليات حالية، ثم تنتقل إلى قناة Mantis مخصصة عند توفر واحدة.

لا تضع معرّفات المجموعات، أو معرّفات القنوات، أو رموز البوتات، أو ملفات تعريف ارتباط المتصفح، أو كلمات مرور VNC في هذا المستند. خزّنها في أسرار GitHub، أو وسيط بيانات الاعتماد، أو مخزن الأسرار المحلي الخاص بالمشغّل.

## إضافة سيناريو

يجب أن يعلن سيناريو Mantis عن:

- المعرّف والعنوان
- النقل
- بيانات الاعتماد المطلوبة
- سياسة مرجع خط الأساس
- سياسة مرجع المرشح
- تصحيح إعدادات OpenClaw
- خطوات الإعداد
- المحفّز
- أوراكل خط الأساس المتوقع
- أوراكل المرشح المتوقع
- أهداف الالتقاط المرئي
- ميزانية المهلة
- خطوات التنظيف

يجب أن تفضّل السيناريوهات أوراكلات صغيرة ومكتوبة الأنواع:

- حالة تفاعل Discord لأخطاء التفاعلات
- مراجع رسائل Discord لأخطاء التسلسل
- الطابع الزمني لسلسلة Slack وحالة API للتفاعلات لأخطاء Slack
- معرّفات رسائل البريد الإلكتروني ورؤوسها لأخطاء البريد الإلكتروني
- لقطات شاشة المتصفح عندما تكون واجهة المستخدم هي المرصود الموثوق الوحيد

يجب أن تكون فحوص الرؤية إضافية. إذا كان بإمكان API منصة إثبات الخطأ، فاستخدم API كأوراكل نجاح/فشل واحتفظ بلقطات الشاشة لثقة البشر.

## توسيع المزوّدين

بعد Discord، يمكن للمشغّل نفسه إضافة:

- Slack: التفاعلات، السلاسل، إشارات التطبيق، النوافذ المشروطة، رفع الملفات.
- البريد الإلكتروني: مصادقة Gmail وتسلسل الرسائل باستخدام `gog` عندما لا تكفي الموصلات.
- WhatsApp: تسجيل الدخول عبر QR، إعادة التعريف، تسليم الرسائل، الوسائط، التفاعلات.
- Telegram: ضبط إشارات المجموعات، الأوامر، التفاعلات حيثما توفرت.
- Matrix: الغرف المشفّرة، علاقات السلاسل أو الردود، استئناف ما بعد إعادة التشغيل.

يجب أن يحتوي كل نقل على سيناريو فحص سريع رخيص واحد وسيناريو واحد أو أكثر لفئة أخطاء. يجب أن تبقى السيناريوهات المرئية المكلفة اختيارية.

## أسئلة مفتوحة

- أي بوت Discord يجب أن يكون المشغّل، وأيها يجب أن يكون SUT، عند إعادة استخدام بوت Mantis الحالي؟
- هل يجب أن يستخدم تسجيل دخول متصفح المراقب حساب Discord بشريًا، أو حساب اختبار، أو أدلة REST القابلة للقراءة بواسطة البوت فقط للمرحلة الأولى؟
- إلى متى يجب أن يحتفظ GitHub بعناصر Mantis الأثرية الخاصة بـ PRs؟
- متى يجب أن يوصي ClawSweeper تلقائيًا بـ Mantis بدلًا من انتظار أمر من مشرف؟
- هل يجب تنقيح لقطات الشاشة أو قصّها قبل رفعها إلى PRs العامة؟
