Mantis هو نظام التحقق الشامل في OpenClaw للأخطاء التي تحتاج إلى وقت تشغيل حقيقي،
ونقل حقيقي، ودليل مرئي. يشغّل سيناريو مقابل مرجع معروف بأنه سيئ،
ويلتقط الأدلة، ثم يشغّل السيناريو نفسه مقابل مرجع مرشح، وينشر
المقارنة كأدوات أثرية يمكن للمشرف فحصها من PR أو من أمر محلي.

يبدأ Mantis بـ Discord لأن Discord يمنحنا المسار الأول عالي القيمة:
مصادقة بوت حقيقية، وقنوات guild حقيقية، وتفاعلات، وسلاسل، وأوامر أصلية، وواجهة
متصفح يمكن للبشر من خلالها التأكد بصريًا مما عرضه النقل.

## الأهداف

- إعادة إنتاج خطأ من مشكلة GitHub أو PR بنفس شكل النقل الذي يراه
  المستخدمون.
- التقاط أداة أثرية **قبل** على مرجع الأساس قبل تطبيق الإصلاح.
- التقاط أداة أثرية **بعد** على المرجع المرشح بعد تطبيق الإصلاح.
- استخدام أوراكل حتمي كلما أمكن، مثل قراءة تفاعل Discord REST
  أو فحص نص قناة.
- التقاط لقطات شاشة عندما يكون للخطأ سطح واجهة مستخدم مرئي.
- التشغيل محليًا من CLI يتحكم به وكيل وعن بُعد من GitHub.
- حفظ قدر كافٍ من حالة الجهاز لإنقاذ VNC عندما يتعطل تسجيل الدخول أو أتمتة المتصفح أو
  مصادقة المزوّد.
- نشر حالة موجزة إلى قناة Discord خاصة بالمشغّل عندما يكون التشغيل محظورًا،
  أو يحتاج إلى مساعدة VNC يدوية، أو ينتهي.

## ليست أهدافًا

- Mantis ليس بديلًا عن اختبارات الوحدة. عادةً يجب أن يتحول تشغيل Mantis إلى
  اختبار تراجع أصغر بعد فهم الإصلاح.
- Mantis ليس بوابة CI السريعة العادية. فهو أبطأ، ويستخدم بيانات اعتماد حية، وهو
  مخصص للأخطاء التي تكون فيها البيئة الحية مهمة.
- لا ينبغي أن يتطلب Mantis تدخلًا بشريًا في التشغيل العادي. VNC اليدوي مسار
  إنقاذ، وليس المسار الاعتيادي.
- لا يخزن Mantis الأسرار الخام في الأدوات الأثرية أو السجلات أو لقطات الشاشة أو تقارير Markdown
  أو تعليقات PR.

## الملكية

يوجد Mantis ضمن حزمة QA في OpenClaw.

- تملك OpenClaw وقت تشغيل السيناريو، ومحولات النقل، ومخطط الأدلة، وCLI
  المحلي ضمن `pnpm openclaw qa mantis`.
- يملك QA Lab أجزاء أحزمة النقل الحي، ومساعدات التقاط المتصفح، وكتّاب
  الأدوات الأثرية.
- يملك Crabbox أجهزة Linux المهيأة مسبقًا عندما تكون هناك حاجة إلى VM بعيدة.
- يملك GitHub Actions نقطة دخول سير العمل البعيد واحتفاظ الأدوات الأثرية.
- يملك ClawSweeper توجيه تعليقات GitHub: تحليل أوامر المشرفين،
  وإرسال سير العمل، ونشر تعليق PR النهائي.
- تقود وكلاء OpenClaw تشغيل Mantis عبر Codex عندما يحتاج السيناريو إلى إعداد وكيل،
  أو تصحيح أخطاء، أو الإبلاغ عن حالة عالقة.

يحافظ هذا الحد على معرفة النقل داخل OpenClaw، وجدولة الأجهزة داخل
Crabbox، وغراء سير عمل المشرفين داخل ClawSweeper.

## شكل الأمر

يتحقق الأمر المحلي الأول من بوت Discord، وguild، والقناة، وإرسال الرسائل،
وإرسال التفاعلات، ومسار الأدوات الأثرية:

```bash
pnpm openclaw qa mantis discord-smoke \
  --output-dir .artifacts/qa-e2e/mantis/discord-smoke
```

يقبل مشغل ما قبل وما بعد المحلي هذا الشكل:

```bash
pnpm openclaw qa mantis run \
  --transport discord \
  --scenario discord-status-reactions-tool-only \
  --baseline origin/main \
  --candidate HEAD \
  --output-dir .artifacts/qa-e2e/mantis/local-discord-status-reactions
```

ينشئ المشغل أشجار عمل منفصلة للأساس والمرشح ضمن دليل الإخراج،
ويثبت الاعتماديات، ويبني كل مرجع، ويشغّل السيناريو مع
`--allow-failures`، ثم يكتب `baseline/`، و`candidate/`، و`comparison.json`،
و`mantis-report.md`. بالنسبة لأول سيناريو Discord، يعني التحقق الناجح
أن حالة الأساس هي `fail` وحالة المرشح هي `pass`.

أول بدائية VM/متصفح هي فحص سطح المكتب:

```bash
pnpm openclaw qa mantis desktop-browser-smoke \
  --output-dir .artifacts/qa-e2e/mantis/desktop-browser
```

يستأجر أو يعيد استخدام جهاز سطح مكتب Crabbox، ويبدأ متصفحًا مرئيًا داخل
جلسة VNC، ويلتقط سطح المكتب، ويسحب الأدوات الأثرية إلى دليل الإخراج
المحلي، ويكتب أمر إعادة الاتصال في التقرير. يستخدم الأمر افتراضيًا مزود
Hetzner لأنه أول مزود لديه تغطية سطح مكتب/VNC عاملة في مسار Mantis. تجاوزه
باستخدام `--provider` أو `--crabbox-bin` أو
`OPENCLAW_MANTIS_CRABBOX_PROVIDER` عند التشغيل مقابل أسطول Crabbox آخر.

أعلام مفيدة لفحص سطح المكتب:

- يعيد `--lease-id <cbx_...>` أو `OPENCLAW_MANTIS_CRABBOX_LEASE_ID` استخدام سطح مكتب مهيأ مسبقًا.
- يغيّر `--browser-url <url>` الصفحة المفتوحة في المتصفح المرئي.
- يعرض `--html-file <path>` أداة أثرية HTML محلية من المستودع في المتصفح المرئي. يستخدم Mantis هذا لالتقاط مخطط تفاعلات حالة Discord المولّد عبر سطح مكتب Crabbox حقيقي.
- يبقي `--keep-lease` أو `OPENCLAW_MANTIS_KEEP_VM=1` إيجارًا جديدًا ناجحًا مفتوحًا لفحص VNC. تبقي عمليات التشغيل الفاشلة الإيجار افتراضيًا عند إنشائه حتى يتمكن المشغّل من إعادة الاتصال.
- تضبط `--class`، و`--idle-timeout`، و`--ttl` حجم الجهاز وعمر الإيجار.

أول بدائية نقل سطح مكتب كاملة هي فحص سطح مكتب Slack:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --output-dir .artifacts/qa-e2e/mantis/slack-desktop \
  --gateway-setup \
  --scenario slack-canary \
  --keep-lease
```

يستأجر أو يعيد استخدام جهاز سطح مكتب Crabbox، ويزامن نسخة checkout الحالية إلى
VM، ويشغّل `pnpm openclaw qa slack` داخل تلك VM، ويفتح Slack Web في متصفح
VNC، ويلتقط سطح المكتب المرئي، وينسخ أدوات Slack QA الأثرية ولقطة شاشة VNC
إلى دليل الإخراج المحلي. هذا هو أول شكل Mantis يعيش فيه Gateway OpenClaw
للنظام قيد الاختبار والمتصفح كلاهما داخل VM سطح مكتب Linux نفسها.

مع `--gateway-setup`، يجهّز الأمر منزل OpenClaw مؤقتًا ومستمرًا عند
`$HOME/.openclaw-mantis/slack-openclaw`، ويرقّع إعدادات Slack Socket Mode
للقناة المحددة، ويبدأ `openclaw gateway run` على المنفذ
`38973`، ويبقي Chrome قيد التشغيل في جلسة VNC. هذا هو وضع "اترك لي سطح مكتب
Linux مع Slack ومخلب قيد التشغيل"؛ ويبقى مسار Slack QA من بوت إلى بوت هو
الافتراضي عند حذف `--gateway-setup`.

المدخلات المطلوبة لـ `--credential-source env`:

- `OPENCLAW_QA_SLACK_CHANNEL_ID`
- `OPENCLAW_QA_SLACK_DRIVER_BOT_TOKEN`
- `OPENCLAW_QA_SLACK_SUT_BOT_TOKEN`
- `OPENCLAW_QA_SLACK_SUT_APP_TOKEN`
- `OPENCLAW_LIVE_OPENAI_KEY` لمسار النموذج البعيد. إذا كان
  `OPENAI_API_KEY` مضبوطًا محليًا فقط، يطابقه Mantis إلى `OPENCLAW_LIVE_OPENAI_KEY`
  قبل استدعاء Crabbox بحيث يمكن لتمرير بيئة Crabbox `OPENCLAW_*` حمله
  إلى VM.

أعلام مفيدة لسطح مكتب Slack:

- يعيد `--lease-id <cbx_...>` التشغيل مقابل جهاز سجّل فيه مشغّل الدخول مسبقًا إلى Slack Web عبر VNC.
- يبدأ `--gateway-setup` Gateway OpenClaw Slack مستمرًا في VM بدلًا من تشغيل مسار QA من بوت إلى بوت فقط.
- يفتح `--slack-url <url>` عنوان Slack Web محددًا. بدونه، يشتق Mantis `https://app.slack.com/client/<team>/<channel>` من Slack `auth.test` عندما يتوفر رمز بوت النظام قيد الاختبار.
- يتحكم `--slack-channel-id <id>` في قائمة السماح لقنوات Slack المستخدمة بواسطة إعداد Gateway.
- يتحكم `OPENCLAW_MANTIS_SLACK_BROWSER_PROFILE_DIR` في ملف Chrome الشخصي المستمر داخل VM. الافتراضي هو `$HOME/.config/openclaw-mantis/slack-chrome-profile`، لذلك يبقى تسجيل الدخول اليدوي إلى Slack Web بعد إعادة التشغيل على الإيجار نفسه.
- يستخدم `--credential-source convex --credential-role ci` مجموعة بيانات الاعتماد المشتركة بدلًا من رموز Slack env المباشرة.
- تمر `--provider-mode`، و`--model`، و`--alt-model`، و`--fast` إلى مسار Slack الحي.

سير عمل فحص GitHub هو `Mantis Discord Smoke`. أما سير عمل GitHub لما قبل وما بعد
لأول سيناريو حقيقي فهو `Mantis Discord Status Reactions`. يقبل:

- `baseline_ref`: المرجع المتوقع أن يعيد إنتاج سلوك queued-only.
- `candidate_ref`: المرجع المتوقع أن يظهر `queued -> thinking -> done`.

يتحقق من مرجع حزمة سير العمل، ويبني أشجار عمل منفصلة للأساس والمرشح،
ويشغّل `discord-status-reactions-tool-only` مقابل كل شجرة عمل، ويرفع
`baseline/`، و`candidate/`، و`comparison.json`، و`mantis-report.md` كأدوات
أثرية في Actions. كما يعرض HTML مخطط كل مسار في متصفح سطح مكتب Crabbox
وينشر لقطات شاشة VNC هذه بجانب صور PNG الحتمية للمخطط في تعليق PR. يربط تعليق
PR نفسه بتسجيلات MP4 لسطح المكتب الملتقطة أثناء عرض متصفح VNC، بينما تبقى
لقطات الشاشة مضمّنة للمراجعة السريعة. يبني سير العمل Crabbox CLI من
`openclaw/crabbox` main حتى يتمكن من استخدام أعلام إيجار سطح المكتب/المتصفح الحالية
قبل إصدار ملف Crabbox الثنائي التالي.

يمكنك أيضًا تشغيل تشغيل status-reactions مباشرةً من تعليق PR:

```text
@Mantis discord status reactions
```

مشغّل التعليق ضيق عمدًا. يعمل فقط على تعليقات pull request من مستخدمين لديهم
صلاحية write أو maintain أو admin، ولا يتعرف إلا على طلبات تفاعلات حالة
Discord. افتراضيًا يستخدم مرجع الأساس السيئ المعروف وSHA رأس PR الحالي كمرشح.
يمكن للمشرفين تجاوز أي من المرجعين:

```text
@Mantis discord status reactions baseline=origin/main candidate=HEAD
```

أمثلة أوامر ClawSweeper:

```text
@clawsweeper mantis discord discord-status-reactions-tool-only
@clawsweeper verify e2e discord
```

الأمر الأول صريح ومركّز على السيناريو. ويمكن للأمر الثاني لاحقًا ربط PR
أو مشكلة بسيناريوهات Mantis الموصى بها من التسميات، والملفات المتغيرة،
ونتائج مراجعة ClawSweeper.

## دورة حياة التشغيل

1. الحصول على بيانات الاعتماد.
2. تخصيص VM أو إعادة استخدامها.
3. تجهيز ملف سطح المكتب/المتصفح الشخصي عندما يحتاج السيناريو إلى دليل واجهة مستخدم.
4. تجهيز نسخة checkout نظيفة لمرجع الأساس.
5. تثبيت الاعتماديات وبناء ما يحتاجه السيناريو فقط.
6. بدء Gateway OpenClaw فرعي مع دليل حالة معزول.
7. تكوين النقل الحي، والمزوّد، والنموذج، وملف المتصفح الشخصي.
8. تشغيل السيناريو والتقاط أدلة الأساس.
9. إيقاف Gateway وحفظ السجلات.
10. تجهيز المرجع المرشح في VM نفسها.
11. تشغيل السيناريو نفسه والتقاط أدلة المرشح.
12. مقارنة نتائج الأوراكل والأدلة المرئية.
13. كتابة Markdown، وJSON، والسجلات، ولقطات الشاشة، وأدوات التتبع الأثرية الاختيارية.
14. رفع أدوات GitHub Actions الأثرية.
15. نشر رسالة حالة موجزة في PR أو Discord.

يجب أن يكون السيناريو قادرًا على الفشل بطريقتين مختلفتين:

- **إعادة إنتاج الخطأ**: فشل الأساس بالطريقة المتوقعة.
- **فشل الحزمة**: فشل إعداد البيئة، أو بيانات الاعتماد، أو Discord API، أو المتصفح، أو
  المزوّد قبل أن تكون أوراكل الخطأ ذات معنى.

يجب أن يفصل التقرير النهائي بين هذه الحالات حتى لا يخلط المشرفون بين بيئة
متذبذبة وسلوك المنتج.

## الحد الأدنى القابل للتطبيق لـ Discord

يجب أن يستهدف السيناريو الأول تفاعلات حالة Discord في قنوات guild حيث يكون
وضع تسليم رد المصدر هو `message_tool_only`.

لماذا يُعد بذرة Mantis جيدة:

- إنه مرئي في Discord كتفاعلات على الرسالة المشغّلة.
- لديه أوراكل REST قوي عبر حالة تفاعلات رسالة Discord.
- يختبر Gateway OpenClaw حقيقيًا، ومصادقة بوت Discord، وإرسال الرسائل،
  ووضع تسليم رد المصدر، وحالة تفاعل الحالة، ودورة حياة دورة النموذج.
- إنه ضيق بما يكفي لإبقاء التنفيذ الأول صادقًا.

شكل السيناريو المتوقع:

```yaml
id: discord-status-reactions-tool-only
transport: discord
baseline:
  expect:
    reproduced: true
candidate:
  expect:
    fixed: true
config:
  messages:
    ackReaction: "👀"
    ackReactionScope: "group-mentions"
    groupChat:
      visibleReplies: "message_tool"
    statusReactions:
      enabled: true
      timing:
        debounceMs: 0
discord:
  requireMention: true
  notifyChannel: operator-notify
evidence:
  rest:
    messageReactions: true
  browser:
    screenshotMessageRow: true
```

يجب أن تُظهر أدلة الأساس تفاعل إقرار queued ولكن بدون انتقال دورة الحياة في
وضع tool-only. ويجب أن تُظهر أدلة المرشح تشغيل تفاعلات حالة دورة الحياة عندما يكون
`messages.statusReactions.enabled` مضبوطًا صراحةً على
true.

الشريحة التنفيذية الأولى هي سيناريو Discord live QA الاختياري:

```bash
pnpm openclaw qa discord \
  --scenario discord-status-reactions-tool-only \
  --provider-mode live-frontier \
  --model openai/gpt-5.4 \
  --alt-model openai/gpt-5.4 \
  --fast \
  --output-dir .artifacts/qa-e2e/mantis/discord-status-reactions-candidate
```

يضبط ذلك النظام قيد الاختبار مع معالجة النقابات المفعّلة دائماً، و`visibleReplies:
"message_tool"`، و`ackReaction: "👀"`، وتفاعلات حالة صريحة. يستطلع المتحقق
رسالة Discord الحقيقية المحفزة ويتوقع التسلسل المرصود
`👀 -> 🤔 -> 👍`. تشمل القطع الأثرية `discord-qa-reaction-timelines.json`،
و`discord-status-reactions-tool-only-timeline.html`، و
`discord-status-reactions-tool-only-timeline.png`.

## مكونات ضمان الجودة الحالية

ينبغي أن يبني Mantis على حزمة ضمان الجودة الخاصة الحالية بدلاً من البدء من
الصفر:

- يشغل `pnpm openclaw qa discord` بالفعل مسار Discord مباشراً مع بوتات المشغل و
  النظام قيد الاختبار.
- يكتب مشغل النقل المباشر بالفعل التقارير وقطع الرسائل المرصودة الأثرية
  ضمن `.artifacts/qa-e2e/`.
- توفر عقود اعتماد Convex بالفعل وصولاً حصرياً إلى بيانات اعتماد النقل المباشر
  المشتركة.
- تدعم خدمة التحكم في المتصفح بالفعل لقطات الشاشة، واللقطات، والملفات الشخصية
  المُدارة دون واجهة، وملفات CDP الشخصية البعيدة.
- لدى QA Lab بالفعل واجهة مصحح وحافلة لاختبارات على شكل النقل.

يمكن أن يكون تنفيذ Mantis الأول مشغلاً رقيقاً قبل/بعد فوق هذه المكونات،
إضافةً إلى طبقة دليل مرئي واحدة.

```text
  mantis-report.md
  mantis-summary.json
  baseline/
    summary.json
    discord-message.json
    screenshot-message-row.png
    gateway-debug/
  candidate/
    summary.json
    discord-message.json

    gateway-debug/
  comparison.json
  run.log
```

ينبغي أن يكون `mantis-summary.json` مصدر الحقيقة القابل للقراءة آلياً. أما
تقرير Markdown فهو لتعليقات PR والمراجعة البشرية.

- المراجع وSHAs التي اختُبرت
- النقل ومعرّف السيناريو
- مزود الجهاز ومعرّف الجهاز أو معرّف العقد
- مصدر بيانات الاعتماد دون قيم سرية
- نتيجة الخط الأساسي
- نتيجة المرشح
- ما إذا كان الخطأ قد تكرر على الخط الأساسي
- ما إذا كان المرشح قد أصلحه
- مسارات القطع الأثرية
- مشكلات الإعداد أو التنظيف المنقحة

لقطات الشاشة أدلة، وليست أسراراً. لكنها لا تزال تحتاج إلى انضباط التنقيح:
قد تظهر أسماء القنوات الخاصة أو أسماء المستخدمين أو محتوى الرسائل. بالنسبة إلى
PRs العامة، فضّل روابط قطع GitHub Actions الأثرية على الصور المضمنة إلى أن تصبح
قصة التنقيح أقوى.

## المتصفح وVNC

لمسار المتصفح وضعان:

- **أتمتة دون واجهة**: الوضع الافتراضي في CI. يعمل Chrome مع تمكين CDP، وتلتقط
  Playwright أو أداة التحكم في متصفح OpenClaw لقطات الشاشة.
- **إنقاذ VNC**: مُمكّن على الجهاز الافتراضي نفسه عندما يحتاج تسجيل الدخول أو
  MFA أو مكافحة الأتمتة في Discord أو التصحيح المرئي إلى تدخل بشري.

ينبغي أن يكون ملف تعريف متصفح مراقب Discord دائماً بما يكفي لتجنب تسجيل الدخول
في كل تشغيل، لكنه معزول عن حالة المتصفح الشخصية. ينتمي الملف الشخصي إلى مجموعة
أجهزة Mantis، لا إلى حاسوب مطور محمول.

عندما يتعثر Mantis، ينشر رسالة حالة Discord تحتوي على:

- معرّف التشغيل
- معرّف السيناريو
- مزود الجهاز
- دليل القطع الأثرية
- تعليمات اتصال VNC أو noVNC إن توفرت
- نص قصير يصف العائق

يمكن للنشر الخاص الأول أن يرسل هذه الرسائل إلى قناة المشغل الحالية ثم ينتقل
لاحقاً إلى قناة Mantis مخصصة.

## الأجهزة

ينبغي أن يفضل Mantis استخدام AWS عبر Crabbox في أول تنفيذ بعيد. يمنحنا Crabbox
أجهزة مُدفأة، وتتبع عقود، وتهيئة، وسجلات، ونتائج، وتنظيفاً. إذا كانت سعة AWS
بطيئة جداً أو غير متاحة، فأضف مزود Hetzner خلف واجهة الجهاز نفسها.

الحد الأدنى لمتطلبات الجهاز الافتراضي:

- Linux مع تثبيت Chrome أو Chromium قادر على سطح المكتب
- وصول CDP لأتمتة المتصفح
- VNC أو noVNC للإنقاذ
- Node 22 وpnpm
- نسخة OpenClaw ومخزن تبعيات مؤقت
- مخزن متصفح Playwright Chromium المؤقت عند استخدام Playwright
- ما يكفي من CPU والذاكرة لتشغيل OpenClaw Gateway واحد، ومتصفح واحد، وتشغيل نموذج واحد
- وصول صادر إلى Discord وGitHub ومزودي النماذج ووسيط بيانات الاعتماد

ينبغي ألا يحتفظ الجهاز الافتراضي بأسرار خام طويلة الأمد خارج مخازن بيانات
الاعتماد أو ملفات تعريف المتصفح المتوقعة.

## الأسرار

توجد الأسرار في أسرار مؤسسة GitHub أو المستودع لعمليات التشغيل البعيدة، وفي
ملف أسرار محلي يتحكم فيه المشغل لعمليات التشغيل المحلية.

أسماء الأسرار الموصى بها:

- `OPENCLAW_QA_DISCORD_MANTIS_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_DRIVER_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_SUT_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_GUILD_ID`
- `OPENCLAW_QA_DISCORD_CHANNEL_ID`
- `OPENCLAW_QA_DISCORD_NOTIFY_CHANNEL_ID`
- `OPENCLAW_QA_REDACT_PUBLIC_METADATA=1` لعمليات رفع قطع GitHub العامة
- `OPENCLAW_QA_CONVEX_SITE_URL`
- `OPENCLAW_QA_CONVEX_SECRET_CI`
- `OPENCLAW_QA_MANTIS_CRABBOX_COORDINATOR`
- `OPENCLAW_QA_MANTIS_CRABBOX_COORDINATOR_TOKEN`

على المدى الطويل، ينبغي أن يبقى تجمع بيانات اعتماد Convex المصدر الطبيعي
لبيانات اعتماد النقل المباشر. تؤسس أسرار GitHub الوسيط ومسارات الرجوع. يطابق
سير عمل تفاعلات حالة Discord أسرار Mantis Crabbox مرة أخرى مع متغيري البيئة
`CRABBOX_COORDINATOR` و`CRABBOX_COORDINATOR_TOKEN` اللذين يتوقعهما Crabbox CLI.
وتظل أسماء أسرار GitHub العادية `CRABBOX_*` مقبولة كخيار توافق احتياطي.

يجب ألا يطبع مشغل Mantis أبداً:

- رموز بوتات Discord
- مفاتيح API الخاصة بالمزودين
- ملفات تعريف ارتباط المتصفح
- محتويات ملفات تعريف المصادقة
- كلمات مرور VNC
- حمولات بيانات الاعتماد الخام

ينبغي أن تنقح عمليات رفع القطع الأثرية العامة أيضاً بيانات تعريف أهداف Discord
مثل معرّفات البوت والنقابة والقناة والرسالة. يفعّل سير عمل اختبار الدخان في
GitHub `OPENCLAW_QA_REDACT_PUBLIC_METADATA=1` لهذا السبب.

إذا لُصق رمز بطريق الخطأ في قضية أو PR أو محادثة أو سجل، فدوّره بعد تخزين
السر الجديد.

## قطع GitHub الأثرية وتعليقات PR

ينبغي أن ترفع مسارات عمل Mantis حزمة الأدلة الكاملة كقطعة Actions أثرية قصيرة
العمر. عندما يُشغّل سير العمل لتقرير خطأ أو PR إصلاح، ينبغي أيضاً أن ينشر لقطات
شاشة PNG المنقحة إلى فرع `qa-artifacts` وأن يحدّث أو ينشئ تعليقاً على ذلك الخطأ
أو PR الإصلاح مع لقطات شاشة مضمنة قبل/بعد. لا تنشر الدليل الأساسي فقط على PR
عام لأتمتة ضمان الجودة. تبقى السجلات الخام والرسائل المرصودة والأدلة الضخمة
الأخرى في قطعة Actions الأثرية.

ينبغي أن تنشر مسارات عمل الإنتاج تلك التعليقات باستخدام تطبيق Mantis GitHub،
وليس باستخدام `github-actions[bot]`. خزّن معرّف التطبيق والمفتاح الخاص كأسرار
GitHub Actions باسم `MANTIS_GITHUB_APP_ID` و`MANTIS_GITHUB_APP_PRIVATE_KEY`.
يستخدم سير العمل علامة مخفية كمفتاح تحديث أو إنشاء، ويحدّث ذلك التعليق عندما
يستطيع الرمز تحريره، وينشئ تعليقاً جديداً مملوكاً لـ Mantis عندما لا يمكن تحرير
علامة أقدم يملكها بوت.

ينبغي أن يكون تعليق PR قصيراً ومرئياً:

```md
Mantis Discord Status Reactions QA

Summary: Mantis reran the reported Discord status-reaction bug against the known
bad baseline and the candidate fix. The baseline reproduced the bug, while the
candidate showed the expected queued -> thinking -> done sequence.

- Scenario: `discord-status-reactions-tool-only`
- Run: <workflow run link>
- Artifact: <artifact link>
- Baseline: `<status>` at `<sha>`
- Candidate: `<status>` at `<sha>`

| Baseline            | Candidate           |
| ------------------- | ------------------- |
| <inline screenshot> | <inline screenshot> |
```

عندما يفشل التشغيل بسبب فشل عدة الاختبار، يجب أن يقول التعليق ذلك بدلاً من
الإيحاء بأن المرشح فشل.

## ملاحظات النشر الخاص

قد يكون لدى نشر خاص تطبيق Discord خاص بـ Mantis بالفعل. أعد استخدام ذلك التطبيق
بدلاً من إنشاء تطبيق آخر عندما تكون لديه أذونات البوت الصحيحة ويمكن تدويره
بأمان.

اضبط قناة إشعار المشغل الأولية عبر الأسرار أو إعدادات النشر. يمكن أن تشير أولاً
إلى قناة صيانة أو عمليات موجودة، ثم تنتقل إلى قناة Mantis مخصصة عند وجودها.

لا تضع معرّفات النقابات أو معرّفات القنوات أو رموز البوتات أو ملفات تعريف
ارتباط المتصفح أو كلمات مرور VNC في هذا المستند. خزّنها في أسرار GitHub أو
وسيط بيانات الاعتماد أو مخزن الأسرار المحلي الخاص بالمشغل.

## إضافة سيناريو

ينبغي أن يعلن سيناريو Mantis عن:

- المعرّف والعنوان
- النقل
- بيانات الاعتماد المطلوبة
- سياسة مرجع الخط الأساسي
- سياسة مرجع المرشح
- رقعة إعداد OpenClaw
- خطوات الإعداد
- المحفز
- معيار تحقق الخط الأساسي المتوقع
- معيار تحقق المرشح المتوقع
- أهداف الالتقاط المرئي
- ميزانية المهلة
- خطوات التنظيف

ينبغي أن تفضل السيناريوهات معايير تحقق صغيرة ومكتوبة الأنواع:

- حالة تفاعل Discord لأخطاء التفاعلات
- مراجع رسائل Discord لأخطاء التفرع
- قيمة ts لسلسلة Slack وحالة API للتفاعلات لأخطاء Slack
- معرّفات رسائل البريد الإلكتروني وترويساتها لأخطاء البريد الإلكتروني
- لقطات شاشة المتصفح عندما تكون UI هي الشيء الوحيد الموثوق ملاحظته

ينبغي أن تكون فحوصات الرؤية إضافية. إذا كان بإمكان API المنصة إثبات الخطأ،
فاستخدم API كمعيار نجاح/فشل واحتفظ بلقطات الشاشة لزيادة الثقة البشرية.

## توسيع المزودين

بعد Discord، يمكن للمشغل نفسه إضافة:

- Slack: التفاعلات، والسلاسل، وإشارات التطبيق، والنوافذ، ورفع الملفات.
- البريد الإلكتروني: مصادقة Gmail وتفريع الرسائل باستخدام `gog` عندما لا تكون الموصلات كافية.
- WhatsApp: تسجيل الدخول عبر QR، وإعادة التعريف، وتسليم الرسائل، والوسائط، والتفاعلات.
- Telegram: حصر إشارات المجموعة، والأوامر، والتفاعلات حيثما توفرت.
- Matrix: الغرف المشفرة، وعلاقات السلاسل أو الردود، واستئناف التشغيل بعد إعادة التشغيل.

ينبغي أن يكون لكل نقل سيناريو دخان رخيص وسيناريو واحد أو أكثر لفئات الأخطاء.
ينبغي أن تبقى السيناريوهات المرئية المكلفة اختيارية.

## أسئلة مفتوحة

- أي بوت Discord ينبغي أن يكون المشغل، وأيهما ينبغي أن يكون النظام قيد الاختبار، عند
  إعادة استخدام بوت Mantis الحالي؟
- هل ينبغي أن يستخدم تسجيل دخول متصفح المراقب حساب Discord بشرياً، أو حساب اختبار،
  أو أدلة REST التي يمكن للبوت فقط قراءتها في المرحلة الأولى؟
- كم من الوقت ينبغي أن يحتفظ GitHub بقطع Mantis الأثرية الخاصة بـ PRs؟
- متى ينبغي أن يوصي ClawSweeper تلقائياً بـ Mantis بدلاً من انتظار أمر من
  أحد المشرفين؟
- هل ينبغي تنقيح لقطات الشاشة أو قصها قبل رفعها إلى PRs العامة؟
