واجهة التحكم هي تطبيق صفحة واحدة صغير مبني باستخدام **Vite + Lit** ويقدّمه Gateway:

- الافتراضي: `http://<host>:18789/`
- بادئة اختيارية: اضبط `gateway.controlUi.basePath` (مثل `/openclaw`)

وهو يتحدث **مباشرةً مع WebSocket الخاص بـ Gateway** على المنفذ نفسه.

## فتح سريع (محلي)

إذا كان Gateway يعمل على الكمبيوتر نفسه، فافتح:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (أو [http://localhost:18789/](http://localhost:18789/))

إذا فشل تحميل الصفحة، فابدأ Gateway أولًا: `openclaw gateway`.

تُمرَّر المصادقة أثناء مصافحة WebSocket عبر:

- `connect.params.auth.token`
- `connect.params.auth.password`
- ترويسات هوية Tailscale Serve عندما يكون `gateway.auth.allowTailscale: true`
- ترويسات هوية الوكيل الموثوق عندما يكون `gateway.auth.mode: "trusted-proxy"`

تحتفظ لوحة إعدادات لوحة المعلومات برمز مميز للجلسة الحالية لعلامة تبويب المتصفح
وعنوان URL المحدد لـ gateway؛ ولا تُحفَظ كلمات المرور. عادةً ما ينشئ onboarding
رمز gateway للمصادقة بالمفتاح المشترك عند أول اتصال، لكن مصادقة كلمة المرور
تعمل أيضًا عندما يكون `gateway.auth.mode` هو `"password"`.

## إقران الجهاز (أول اتصال)

عند الاتصال بواجهة التحكم من متصفح أو جهاز جديد، يطلب Gateway
**موافقة إقران لمرة واحدة** — حتى إذا كنت على Tailnet نفسها
مع `gateway.auth.allowTailscale: true`. هذا إجراء أمني لمنع
الوصول غير المصرح به.

**ما ستراه:** "disconnected (1008): pairing required"

**للموافقة على الجهاز:**

```bash
# List pending requests
openclaw devices list

# Approve by request ID
openclaw devices approve <requestId>
```

إذا أعاد المتصفح محاولة الإقران مع تفاصيل مصادقة متغيرة (الدور/النطاقات/المفتاح
العام)، فسيتم استبدال الطلب المعلق السابق ويُنشأ `requestId` جديد.
أعِد تشغيل `openclaw devices list` قبل الموافقة.

إذا كان المتصفح مقترنًا بالفعل وغيّرته من وصول قراءة إلى
وصول كتابة/مشرف، فسيُعامل هذا على أنه ترقية موافقة، وليس إعادة
اتصال صامتة. يحتفظ OpenClaw بالموافقة القديمة نشطة، ويمنع إعادة الاتصال
الأوسع، ويطلب منك الموافقة على مجموعة النطاقات الجديدة صراحةً.

بمجرد الموافقة، يتم تذكر الجهاز ولن يحتاج إلى إعادة موافقة ما لم
تقم بإلغائه باستخدام `openclaw devices revoke --device <id> --role <role>`. راجع
[Devices CLI](/ar/cli/devices) لمعرفة تدوير الرموز المميزة والإلغاء.

**ملاحظات:**

- اتصالات المتصفح المحلية المباشرة عبر local loopback (`127.0.0.1` / `localhost`) تتم
  الموافقة عليها تلقائيًا.
- اتصالات المتصفح عبر Tailnet وLAN لا تزال تتطلب موافقة صريحة، حتى عندما
  تصدر من الجهاز نفسه.
- ينشئ كل ملف تعريف متصفح معرّف جهاز فريدًا، لذا فإن تبديل المتصفحات أو
  مسح بيانات المتصفح سيتطلب إعادة الإقران.

## الهوية الشخصية (محلية للمتصفح)

تدعم واجهة التحكم هوية شخصية لكل متصفح (اسم عرض وصورة رمزية)
تُلحق بالرسائل الصادرة لأغراض الإسناد في الجلسات المشتركة. وهي
تُخزَّن في مساحة تخزين المتصفح، ومحصورة بملف تعريف المتصفح الحالي، ولا
تتم مزامنتها إلى أجهزة أخرى ولا تُحفَظ على الخادم إلا ضمن بيانات
نسبة ملكية الرسائل العادية في النصوص التي ترسلها فعليًا. يؤدي مسح بيانات
الموقع أو تبديل المتصفحات إلى إعادة ضبطها لتصبح فارغة.

## نقطة نهاية إعداد وقت التشغيل

تجلب واجهة التحكم إعدادات وقت التشغيل الخاصة بها من
`/__openclaw/control-ui-config.json`. وتخضع نقطة النهاية هذه لنفس
مصادقة gateway المطبقة على بقية سطح HTTP: لا يمكن للمتصفحات غير
المصادَق عليها جلبها، ويتطلب الجلب الناجح إما رمز gateway/كلمة مرور
صالحة بالفعل، أو هوية Tailscale Serve، أو هوية وكيل موثوق.

## دعم اللغة

يمكن لواجهة التحكم تحديد لغتها عند أول تحميل بناءً على لغة متصفحك.
ولتجاوز ذلك لاحقًا، افتح **Overview -> Gateway Access -> Language**. يوجد
منتقي اللغة في بطاقة Gateway Access، وليس ضمن Appearance.

- اللغات المدعومة: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `tr`, `uk`, `id`, `pl`, `th`
- تُحمَّل الترجمات غير الإنجليزية بتحميل كسول داخل المتصفح.
- تُحفَظ اللغة المحددة في مساحة تخزين المتصفح ويُعاد استخدامها في الزيارات اللاحقة.
- تعود مفاتيح الترجمة المفقودة إلى الإنجليزية.

## ما الذي يمكنها فعله (حاليًا)

- الدردشة مع النموذج عبر Gateway WS (`chat.history`, `chat.send`, `chat.abort`, `chat.inject`)
- التحدث إلى OpenAI Realtime مباشرةً من المتصفح عبر WebRTC. يقوم Gateway
  بإنشاء سر عميل Realtime قصير العمر باستخدام `talk.realtime.session`؛ ويرسل
  المتصفح صوت الميكروفون مباشرةً إلى OpenAI ويعيد تمرير استدعاءات أداة
  `openclaw_agent_consult` عبر `chat.send` إلى نموذج OpenClaw الأكبر
  المُعدّ.
- بث استدعاءات الأدوات + بطاقات مخرجات الأدوات الحية في Chat (أحداث الوكيل)
- القنوات: حالة القنوات المضمنة إضافةً إلى قنوات plugins المضمنة/الخارجية، وتسجيل دخول QR، وإعداد كل قناة على حدة (`channels.status`, `web.login.*`, `config.patch`)
- المثيلات: قائمة الحضور + التحديث (`system-presence`)
- الجلسات: قائمة + تجاوزات لكل جلسة للنموذج/التفكير/السريع/المفصل/التتبع/الاستدلال (`sessions.list`, `sessions.patch`)
- Dreams: حالة Dreaming، ومفتاح التمكين/التعطيل، وقارئ Dream Diary (`doctor.memory.status`, `doctor.memory.dreamDiary`, `config.patch`)
- مهام Cron: عرض/إضافة/تعديل/تشغيل/تمكين/تعطيل + سجل التشغيل (`cron.*`)
- Skills: الحالة، والتمكين/التعطيل، والتثبيت، وتحديثات مفاتيح API (`skills.*`)
- Nodes: القائمة + الحدود القصوى (`node.list`)
- موافقات التنفيذ: تعديل قوائم السماح الخاصة بـ gateway أو node + سياسة ask لـ `exec host=gateway/node` (`exec.approvals.*`)
- الإعداد: عرض/تحرير `~/.openclaw/openclaw.json` (`config.get`, `config.set`)
- الإعداد: تطبيق + إعادة تشغيل مع التحقق (`config.apply`) وإيقاظ آخر جلسة نشطة
- تتضمن كتابة الإعداد حارس base-hash لمنع الكتابة فوق تعديلات متزامنة
- تقوم عمليات كتابة الإعداد (`config.set`/`config.apply`/`config.patch`) أيضًا بإجراء فحص تمهيدي لحل SecretRef النشط للمراجع الموجودة في حمولة الإعداد المرسلة؛ وتُرفض المراجع النشطة غير المحلولة في الحمولة المرسلة قبل الكتابة
- مخطط الإعداد + عرض النموذج (`config.schema` / `config.schema.lookup`،
  بما في ذلك `title` / `description` للحقل، وتلميحات واجهة المستخدم المطابقة،
  والملخصات المباشرة للعناصر الأبناء، وبيانات التوثيق الوصفية على عقد
  الكائنات المتداخلة/الأحرف العامة/المصفوفات/التركيب،
  بالإضافة إلى مخططات plugin + القناة عند توفرها)؛ ولا يكون محرر Raw JSON
  متاحًا إلا عندما تكون اللقطة قابلة للإرجاع الخام الآمن
- إذا تعذر على لقطة ما أن تدعم الإرجاع الآمن للنص الخام، فإن واجهة التحكم تفرض وضع Form وتعطّل وضع Raw لتلك اللقطة
- يحافظ خيار "Reset to saved" في محرر Raw JSON على البنية المؤلفة بصيغة raw (التنسيق، والتعليقات، وتخطيط `$include`) بدلًا من إعادة عرض لقطة مسطحة، بحيث تبقى التعديلات الخارجية بعد إعادة الضبط عندما تكون اللقطة قابلة للإرجاع الآمن
- تُعرض قيم كائنات SecretRef المهيكلة للقراءة فقط داخل مدخلات نص النموذج لمنع تلف الكائن إلى سلسلة نصية عن طريق الخطأ
- التصحيح: لقطات الحالة/الصحة/النماذج + سجل الأحداث + استدعاءات RPC اليدوية (`status`, `health`, `models.list`)
- السجلات: تتبع مباشر لسجلات ملفات gateway مع التصفية/التصدير (`logs.tail`)
- التحديث: تشغيل تحديث حزمة/git + إعادة تشغيل (`update.run`) مع تقرير إعادة تشغيل

ملاحظات لوحة مهام Cron:

- بالنسبة إلى المهام المعزولة، يكون التسليم افتراضيًا إلى إعلان ملخص. يمكنك التبديل إلى none إذا أردت تشغيلات داخلية فقط.
- تظهر حقول القناة/الهدف عند تحديد announce.
- يستخدم وضع Webhook القيمة `delivery.mode = "webhook"` مع ضبط `delivery.to` على عنوان URL صالح لـ Webhook عبر HTTP(S).
- بالنسبة إلى مهام الجلسة الرئيسية، يتوفر وضعا التسليم webhook وnone.
- تتضمن عناصر التحكم المتقدمة في التحرير delete-after-run، ومسح تجاوز الوكيل، وخيارات Cron exact/stagger،
  وتجاوزات agent model/thinking، ومفاتيح تسليم best-effort.
- يكون التحقق من صحة النموذج مضمّنًا مع أخطاء على مستوى الحقل؛ وتؤدي القيم غير الصالحة إلى تعطيل زر الحفظ حتى يتم إصلاحها.
- اضبط `cron.webhookToken` لإرسال bearer token مخصص؛ وإذا أُهمل، فسيُرسَل Webhook من دون ترويسة مصادقة.
- تراجع مهجور: لا تزال المهام القديمة المخزنة التي تحتوي على `notify: true` قادرة على استخدام `cron.webhook` إلى أن تتم هجرتها.

## سلوك Chat

- `chat.send` **غير حاجب**: فهو يرسل إقرارًا فورًا مع `{ runId, status: "started" }` ويتم بث الرد عبر أحداث `chat`.
- تؤدي إعادة الإرسال باستخدام `idempotencyKey` نفسه إلى إرجاع `{ status: "in_flight" }` أثناء التشغيل، و`{ status: "ok" }` بعد الاكتمال.
- تكون استجابات `chat.history` محدودة الحجم لأمان واجهة المستخدم. عندما تكون إدخالات النص كبيرة جدًا، قد يقوم Gateway باقتطاع حقول النص الطويلة، أو حذف كتل البيانات الوصفية الثقيلة، أو استبدال الرسائل كبيرة الحجم بعنصر نائب (`[chat.history omitted: message too large]`).
- تُحفَظ الصور المولدة بواسطة المساعد/النموذج كمراجع وسائط مُدارة وتُقدَّم مجددًا عبر عناوين URL لوسائط Gateway المصادَق عليها، بحيث لا تعتمد إعادة التحميل على بقاء حمولة صور base64 الخام داخل استجابة سجل الدردشة.
- يقوم `chat.history` أيضًا بإزالة وسوم التوجيه المضمنة المخصّصة للعرض فقط من نص المساعد المرئي (مثل `[[reply_to_*]]` و`[[audio_as_voice]]`)، وحمولات XML النصية العادية الخاصة باستدعاء الأدوات (بما في ذلك `<tool_call>...</tool_call>` و`<function_call>...</function_call>` و`<tool_calls>...</tool_calls>` و`<function_calls>...</function_calls>` وكتل استدعاء الأدوات المقتطعة)، ويزيل رموز التحكم الخاصة بالنموذج المسرّبة بصيغة ASCII/العرض الكامل، ويحذف إدخالات المساعد التي يكون نصها المرئي بالكامل هو الرمز الصامت المطابق تمامًا `NO_REPLY` / `no_reply`.
- يقوم `chat.inject` بإلحاق ملاحظة مساعد إلى نص الجلسة ويذيع حدث `chat` لتحديثات واجهة المستخدم فقط (من دون تشغيل وكيل، ومن دون تسليم إلى القناة).
- تقوم منتقيات النموذج والتفكير في ترويسة الدردشة بتحديث الجلسة النشطة فورًا عبر `sessions.patch`؛ وهي تجاوزات دائمة للجلسة، وليست خيارات إرسال لمرة واحدة.
- يستخدم وضع Talk مزود صوت Realtime مسجلًا يدعم جلسات WebRTC في المتصفح. اضبط OpenAI عبر `talk.provider: "openai"` مع
  `talk.providers.openai.apiKey`، أو أعد استخدام إعداد مزود Voice Call من نوع realtime. لا يتلقى المتصفح مفتاح API القياسي لـ OpenAI؛ بل يتلقى فقط سر عميل Realtime المؤقت. يدعم صوت Google Live realtime
  النداءات الخلفية لـ Voice Call وجسور Google Meet، لكنه لا يدعم هذا
  المسار الخاص بـ WebRTC في المتصفح حتى الآن. يقوم Gateway بتجميع prompt الخاص بجلسة Realtime؛ ولا يقبل `talk.realtime.session` تجاوزات تعليمات يقدّمها المستدعي.
- في مؤلف Chat، يكون عنصر التحكم Talk هو زر الأمواج الموجود بجوار
  زر الإملاء عبر الميكروفون. عند بدء Talk، يعرض صف حالة المؤلف
  `Connecting Talk...`، ثم `Talk live` أثناء اتصال الصوت، أو
  `Asking OpenClaw...` بينما تستشير أداة Realtime النموذج الأكبر
  المُعدّ عبر `chat.send`.
- الإيقاف:
  - انقر **Stop** (يستدعي `chat.abort`)
  - أثناء نشاط تشغيل ما، تُصفّ الرسائل اللاحقة العادية في الطابور. انقر **Steer** على رسالة في الطابور لحقن تلك المتابعة في الدور الجاري.
  - اكتب `/stop` (أو عبارات إيقاف مستقلة مثل `stop` أو `stop action` أو `stop run` أو `stop openclaw` أو `please stop`) للإلغاء خارج النطاق
  - يدعم `chat.abort` القيمة `{ sessionKey }` (من دون `runId`) لإلغاء جميع التشغيلات النشطة لتلك الجلسة
- الاحتفاظ الجزئي بعد الإلغاء:
  - عند إلغاء تشغيل، قد يظل النص الجزئي للمساعد ظاهرًا في واجهة المستخدم
  - يحفظ Gateway النص الجزئي الملغى للمساعد في سجل النص عندما يكون خرج المخزن المؤقت موجودًا
  - تتضمن الإدخالات المحفوظة بيانات وصفية للإلغاء حتى يتمكن مستهلكو النص من التمييز بين الأجزاء الملغاة وخرج الاكتمال العادي

## التضمينات المستضافة

يمكن لرسائل المساعد عرض محتوى ويب مستضاف بشكل مضمّن باستخدام الرمز المختصر `[embed ...]`.
وتتحكم السياسة الخاصة بـ iframe sandbox في ذلك عبر
`gateway.controlUi.embedSandbox`:

- `strict`: يعطّل تنفيذ السكربتات داخل التضمينات المستضافة
- `scripts`: يسمح بالتضمينات التفاعلية مع الحفاظ على عزل الأصل؛ وهذا هو
  الافتراضي وعادةً ما يكون كافيًا للألعاب/العناصر التفاعلية ذاتية الاكتفاء في المتصفح
- `trusted`: يضيف `allow-same-origin` فوق `allow-scripts` للمستندات من الموقع نفسه
  التي تحتاج عمدًا إلى صلاحيات أقوى

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

استخدم `trusted` فقط عندما يحتاج المستند المضمَّن فعلًا إلى
سلوك same-origin. بالنسبة إلى معظم الألعاب المُنشأة بواسطة الوكيل والعناصر التفاعلية، يكون `scripts` الخيار الأكثر أمانًا.

تظل عناوين URL المطلقة الخارجية للتضمين `http(s)` محظورة افتراضيًا. إذا
كنت تريد عمدًا أن يقوم `[embed url="https://..."]` بتحميل صفحات خارجية، فاضبط
`gateway.controlUi.allowExternalEmbedUrls: true`.

## الوصول عبر Tailnet (مستحسن)

### Tailscale Serve المدمج (المفضل)

أبقِ Gateway على loopback ودع Tailscale Serve يمرره عبر HTTPS:

```bash
openclaw gateway --tailscale serve
```

افتح:

- `https://<magicdns>/` (أو `gateway.controlUi.basePath` الذي قمت بإعداده)

افتراضيًا، يمكن لطلبات Serve الخاصة بواجهة التحكم/WebSocket المصادقة عبر ترويسات هوية Tailscale
(`tailscale-user-login`) عندما يكون `gateway.auth.allowTailscale` مضبوطًا على `true`. يتحقق OpenClaw
من الهوية عبر حل عنوان `x-forwarded-for` باستخدام
`tailscale whois` ومطابقته مع الترويسة، ولا يقبل هذه القيم إلا عندما
يصل الطلب إلى loopback مع ترويسات `x-forwarded-*` الخاصة بـ Tailscale. اضبط
`gateway.auth.allowTailscale: false` إذا كنت تريد فرض بيانات اعتماد صريحة بالمفتاح المشترك
حتى على حركة Serve. ثم استخدم `gateway.auth.mode: "token"` أو
`"password"`.
وبالنسبة إلى مسار هوية Serve غير المتزامن هذا، تتم مَسلسلة محاولات المصادقة الفاشلة لنفس عنوان IP
للعميل ونطاق المصادقة قبل عمليات الكتابة الخاصة بمحدد المعدل. لذلك قد تعرض
إعادات المحاولة السيئة المتزامنة من المتصفح نفسه الرسالة `retry later` في الطلب الثاني
بدلًا من حالتي عدم تطابق عاديتين تتسابقان بالتوازي.
تفترض مصادقة Serve من دون رمز أن مضيف gateway موثوق. إذا كان من الممكن
تشغيل كود محلي غير موثوق على ذلك المضيف، فاطلب مصادقة token/password.

### الربط بـ tailnet + token

```bash
openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
```

ثم افتح:

- `http://<tailscale-ip>:18789/` (أو `gateway.controlUi.basePath` الذي قمت بإعداده)

ألصق المفتاح السري المشترك المطابق في إعدادات واجهة المستخدم (يُرسَل كـ
`connect.params.auth.token` أو `connect.params.auth.password`).

## HTTP غير الآمن

إذا فتحت لوحة المعلومات عبر HTTP عادي (`http://<lan-ip>` أو `http://<tailscale-ip>`)،
فسيعمل المتصفح في **سياق غير آمن** وسيحظر WebCrypto. افتراضيًا،
يقوم OpenClaw **بحظر** اتصالات واجهة التحكم من دون هوية جهاز.

الاستثناءات الموثقة:

- توافق HTTP غير الآمن المحلي فقط على localhost مع `gateway.controlUi.allowInsecureAuth=true`
- مصادقة واجهة التحكم الناجحة للمشغل عبر `gateway.auth.mode: "trusted-proxy"`
- خيار الطوارئ `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**الإصلاح الموصى به:** استخدم HTTPS (Tailscale Serve) أو افتح الواجهة محليًا:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (على مضيف gateway)

**سلوك مفتاح تبديل المصادقة غير الآمنة:**

```json5
{
  gateway: {
    controlUi: { allowInsecureAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

`allowInsecureAuth` هو مفتاح توافق محلي فقط:

- يسمح لجلسات واجهة التحكم على localhost بالاستمرار من دون هوية جهاز في
  سياقات HTTP غير الآمنة.
- لا يتجاوز فحوصات الإقران.
- لا يخفف متطلبات هوية الجهاز عن بُعد (غير localhost).

**للطوارئ فقط:**

```json5
{
  gateway: {
    controlUi: { dangerouslyDisableDeviceAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

يقوم `dangerouslyDisableDeviceAuth` بتعطيل فحوصات هوية جهاز واجهة التحكم ويُعد
خفضًا أمنيًا شديدًا. أعده سريعًا بعد استخدامه في الطوارئ.

ملاحظة trusted-proxy:

- يمكن لمصادقة trusted-proxy الناجحة أن تسمح بجلسات واجهة تحكم **للمشغل**
  من دون هوية جهاز
- هذا **لا** يمتد إلى جلسات واجهة التحكم ذات دور node
- لا تزال الوكلاء العكسية loopback على المضيف نفسه لا تفي بمصادقة trusted-proxy؛ راجع
  [مصادقة Trusted Proxy](/ar/gateway/trusted-proxy-auth)

راجع [Tailscale](/ar/gateway/tailscale) للحصول على إرشادات إعداد HTTPS.

## سياسة أمان المحتوى

تأتي واجهة التحكم بسياسة `img-src` محكمة: لا يُسمح إلا بالأصول ذات **same-origin** وعناوين `data:` URL. ترفض المتصفحات عناوين الصور البعيدة `http(s)` والعناوين النسبية للبروتوكول ولا تنفذ طلبات شبكة لها.

ما يعنيه ذلك عمليًا:

- لا تزال الصور الرمزية والصور المقدمة تحت مسارات نسبية (مثل `/avatars/<id>`) تُعرض.
- لا تزال عناوين `data:image/...` URL المضمنة تُعرض (وهو مفيد للحمولات داخل البروتوكول).
- تُزال عناوين الصور الرمزية البعيدة الصادرة من بيانات القنوات الوصفية في مساعدات الصور الرمزية في واجهة التحكم ويُستعاض عنها بالشعار/الشارة المضمنة، بحيث لا تتمكن قناة مخترقة أو خبيثة من فرض جلب صور بعيدة عشوائية من متصفح المشغل.

لا تحتاج إلى تغيير أي شيء للحصول على هذا السلوك — فهو مفعّل دائمًا وغير قابل للإعداد.

## مصادقة مسار الصورة الرمزية

عند إعداد مصادقة gateway، تتطلب نقطة نهاية الصورة الرمزية في واجهة التحكم نفس رمز gateway المميز مثل بقية API:

- يعيد `GET /avatar/<agentId>` صورة الصورة الرمزية فقط إلى المستدعين المصادَق عليهم. ويعيد `GET /avatar/<agentId>?meta=1` البيانات الوصفية للصورة الرمزية وفق القاعدة نفسها.
- تُرفض الطلبات غير المصادَق عليها إلى أي من المسارين (بما يطابق مسار وسائط المساعد الشقيق). وهذا يمنع مسار الصورة الرمزية من كشف هوية الوكيل على المضيفات المحمية بخلاف ذلك.
- تقوم واجهة التحكم نفسها بتمرير رمز gateway كـ bearer header عند جلب الصور الرمزية، وتستخدم عناوين blob URL مصادَق عليها بحيث تستمر الصورة في الظهور في لوحات المعلومات.

إذا عطّلت مصادقة gateway (غير مستحسن على المضيفات المشتركة)، فسيصبح مسار الصورة الرمزية أيضًا غير مصادَق عليه، بما يتماشى مع بقية gateway.

## بناء الواجهة

يقدّم Gateway ملفات ثابتة من `dist/control-ui`. ابنِها باستخدام:

```bash
pnpm ui:build
```

قاعدة مطلقة اختيارية (عندما تريد عناوين URL ثابتة للأصول):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

للتطوير المحلي (خادم تطوير منفصل):

```bash
pnpm ui:dev
```

ثم وجّه الواجهة إلى عنوان URL الخاص بـ Gateway WS (مثل `ws://127.0.0.1:18789`).

## التصحيح/الاختبار: خادم تطوير + Gateway بعيد

واجهة التحكم عبارة عن ملفات ثابتة؛ وهدف WebSocket قابل للإعداد ويمكن أن
يختلف عن أصل HTTP. وهذا مفيد عندما تريد خادم تطوير Vite
محليًا بينما يعمل Gateway في مكان آخر.

1. ابدأ خادم تطوير الواجهة: `pnpm ui:dev`
2. افتح عنوان URL مثل:

```text
http://localhost:5173/?gatewayUrl=ws://<gateway-host>:18789
```

مصادقة اختيارية لمرة واحدة (إذا لزم الأمر):

```text
http://localhost:5173/?gatewayUrl=wss://<gateway-host>:18789#token=<gateway-token>
```

ملاحظات:

- يُخزَّن `gatewayUrl` في localStorage بعد التحميل ويُزال من عنوان URL.
- يجب تمرير `token` عبر جزء عنوان URL (`#token=...`) كلما أمكن. لا تُرسل الأجزاء إلى الخادم، ما يتجنب تسرب سجلات الطلبات وReferer. لا تزال معاملات الاستعلام القديمة `?token=` تُستورَد مرة واحدة لأغراض التوافق، ولكن كحل احتياطي فقط، وتُزال فورًا بعد التهيئة.
- يُحفَظ `password` في الذاكرة فقط.
- عند ضبط `gatewayUrl`، لا تعود الواجهة إلى بيانات الاعتماد الموجودة في config أو البيئة.
  قدّم `token` (أو `password`) صراحةً. ويُعد غياب بيانات الاعتماد الصريحة خطأ.
- استخدم `wss://` عندما يكون Gateway خلف TLS (Tailscale Serve، وكيل HTTPS، إلخ).
- لا يُقبل `gatewayUrl` إلا في نافذة من المستوى الأعلى (غير مضمّنة) لمنع clickjacking.
- يجب على عمليات نشر واجهة التحكم غير loopback أن تضبط `gateway.controlUi.allowedOrigins`
  صراحةً (أصول كاملة). وهذا يشمل إعدادات التطوير عن بُعد.
- لا تستخدم `gateway.controlUi.allowedOrigins: ["*"]` إلا لاختبارات محلية
  مضبوطة بإحكام. فهذا يعني السماح لأي أصل متصفح، وليس “مطابقة أي مضيف
  أستخدمه”.
- يفعّل `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true`
  وضع الرجوع إلى أصل Host header، لكنه وضع أمني خطير.

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

تفاصيل إعداد الوصول البعيد: [الوصول البعيد](/ar/gateway/remote).
