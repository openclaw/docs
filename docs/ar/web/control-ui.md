واجهة التحكم هي تطبيق صفحة واحدة صغير مبني باستخدام **Vite + Lit** ويُقدَّم عبر Gateway:

- الافتراضي: `http://<host>:18789/`
- بادئة اختيارية: عيّن `gateway.controlUi.basePath` (مثل `/openclaw`)

وهي تتحدث **مباشرةً إلى Gateway WebSocket** على المنفذ نفسه.

## فتح سريع (محلي)

إذا كان Gateway يعمل على الجهاز نفسه، فافتح:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (أو [http://localhost:18789/](http://localhost:18789/))

إذا فشل تحميل الصفحة، فابدأ Gateway أولًا: `openclaw gateway`.

تُمرَّر المصادقة أثناء مصافحة WebSocket عبر:

- `connect.params.auth.token`
- `connect.params.auth.password`
- رؤوس هوية Tailscale Serve عندما يكون `gateway.auth.allowTailscale: true`
- رؤوس هوية trusted-proxy عندما يكون `gateway.auth.mode: "trusted-proxy"`

تحتفظ لوحة إعدادات لوحة المعلومات برمز مميز لجلسة علامة التبويب الحالية في
المتصفح وعنوان URL المحدد لـ gateway؛ ولا تُحفَظ كلمات المرور. ويولّد
onboarding عادةً رمز gateway للمصادقة بالمفتاح السري المشترك عند أول اتصال،
لكن مصادقة كلمة المرور تعمل أيضًا عندما يكون `gateway.auth.mode` هو `"password"`.

## إقران الجهاز (الاتصال الأول)

عندما تتصل بواجهة التحكم من متصفح أو جهاز جديد، فإن Gateway
يتطلب **موافقة إقران لمرة واحدة** — حتى إذا كنت على Tailnet نفسها
مع `gateway.auth.allowTailscale: true`. وهذا إجراء أمني لمنع
الوصول غير المصرح به.

**ما الذي ستراه:** "disconnected (1008): pairing required"

**للموافقة على الجهاز:**

```bash
# سرد الطلبات المعلقة
openclaw devices list

# الموافقة باستخدام معرّف الطلب
openclaw devices approve <requestId>
```

إذا أعاد المتصفح محاولة الإقران مع تفاصيل مصادقة متغيّرة (الدور/النطاقات/المفتاح
العام)، فسيتم الاستعاضة عن الطلب المعلّق السابق ويُنشأ `requestId` جديد.
أعد تشغيل `openclaw devices list` قبل الموافقة.

إذا كان المتصفح مقترنًا بالفعل ثم غيّرته من وصول قراءة إلى
وصول كتابة/إدارة، فسيُعامل هذا على أنه ترقية موافقة، وليس إعادة
اتصال صامتة. ويحافظ OpenClaw على الموافقة القديمة نشطة، ويمنع إعادة الاتصال
الأوسع، ويطلب منك الموافقة على مجموعة النطاقات الجديدة صراحةً.

وبعد الموافقة، يُتذكر الجهاز ولن يتطلب إعادة الموافقة ما لم
تقم بإلغائه باستخدام `openclaw devices revoke --device <id> --role <role>`. راجع
[Devices CLI](/ar/cli/devices) لمعرفة تدوير الرموز المميزة والإلغاء.

**ملاحظات:**

- تُعتمد اتصالات المتصفح المحلية المباشرة عبر loopback (`127.0.0.1` / `localhost`)
  تلقائيًا.
- لا تزال اتصالات المتصفح عبر Tailnet وLAN تتطلب موافقة صريحة، حتى عندما
  تصدر من الجهاز نفسه.
- يُنشئ كل ملف تعريف متصفح معرّف جهاز فريدًا، لذا فإن تبديل المتصفحات أو
  مسح بيانات المتصفح سيتطلب إعادة الإقران.

## الهوية الشخصية (محلية للمتصفح)

تدعم واجهة التحكم هوية شخصية لكل متصفح (اسم عرض وصورة رمزية)
تُرفَق بالرسائل الصادرة لأغراض الإسناد في الجلسات المشتركة. وتوجد هذه الهوية
في تخزين المتصفح، وتكون محصورة بملف تعريف المتصفح الحالي، ولا
تتزامن مع الأجهزة الأخرى ولا تُحفَظ على جانب الخادم خارج بيانات
نسبة التأليف المعتادة في النصوص للرسائل التي ترسلها فعلًا. ويؤدي مسح
بيانات الموقع أو تبديل المتصفحات إلى إعادتها إلى حالة فارغة.

وينطبق النمط المحلي نفسه في المتصفح على تجاوز الصورة الرمزية للمساعد.
تُطبَّق الصور الرمزية المرفوعة للمساعد فوق الهوية التي يحلّها gateway على
المتصفح المحلي فقط، ولا تمر أبدًا عبر `config.patch`. ولا يزال حقل config
المشترك `ui.assistant.avatar` متاحًا للعملاء غير المعتمدين على الواجهة
الذين يكتبون الحقل مباشرةً (مثل بوابات gateway المؤتمتة أو لوحات التحكم المخصصة).

## نقطة نهاية config وقت التشغيل

تجلب واجهة التحكم إعدادات وقت التشغيل الخاصة بها من
`/__openclaw/control-ui-config.json`. وتخضع هذه النقطة النهائية لبوابة
مصادقة gateway نفسها مثل بقية سطح HTTP: فلا يمكن للمتصفحات غير الموثقة
جلبها، ويتطلب الجلب الناجح إما رمزًا مميزًا/كلمة مرور صالحة بالفعل لـ gateway،
أو هوية Tailscale Serve، أو هوية trusted-proxy.

## دعم اللغة

يمكن لواجهة التحكم أن تترجم نفسها عند التحميل الأول بناءً على لغة المتصفح.
ولتجاوز ذلك لاحقًا، افتح **Overview -> Gateway Access -> Language**. ويوجد
منتقي اللغة في بطاقة Gateway Access، وليس ضمن Appearance.

- اللغات المدعومة: `en` و`zh-CN` و`zh-TW` و`pt-BR` و`de` و`es` و`ja-JP` و`ko` و`fr` و`tr` و`uk` و`id` و`pl` و`th`
- تُحمَّل الترجمات غير الإنجليزية بكسل في المتصفح.
- تُحفَظ اللغة المحددة في تخزين المتصفح ويُعاد استخدامها في الزيارات المستقبلية.
- تعود مفاتيح الترجمة المفقودة إلى الإنجليزية.

## ما الذي يمكنها فعله (حاليًا)

- الدردشة مع النموذج عبر Gateway WS (`chat.history`, `chat.send`, `chat.abort`, `chat.inject`)
- التحدث إلى OpenAI Realtime مباشرةً من المتصفح عبر WebRTC. ويولّد Gateway
  سر عميل Realtime قصير العمر باستخدام `talk.realtime.session`; ويرسل
  المتصفح صوت الميكروفون مباشرةً إلى OpenAI ويعيد تمرير
  استدعاءات الأداة `openclaw_agent_consult` عبر `chat.send` إلى
  نموذج OpenClaw الأكبر المكوَّن.
- بث استدعاءات الأدوات + بطاقات مخرجات الأدوات الحية في Chat (أحداث الوكيل)
- القنوات: حالة القنوات المضمّنة وقنوات الإضافات المجمعة/الخارجية، وتسجيل دخول QR، وإعدادات لكل قناة (`channels.status`, `web.login.*`, `config.patch`)
- Instances: قائمة الحضور + التحديث (`system-presence`)
- الجلسات: سرد + تجاوزات لكل جلسة للنموذج/thinking/fast/verbose/trace/reasoning (`sessions.list`, `sessions.patch`)
- Dreams: حالة Dreaming، ومفتاح تفعيل/تعطيل، وقارئ Dream Diary (`doctor.memory.status`, `doctor.memory.dreamDiary`, `config.patch`)
- مهام Cron: السرد/الإضافة/التحرير/التشغيل/التفعيل/التعطيل + سجل التشغيل (`cron.*`)
- Skills: الحالة، والتفعيل/التعطيل، والتثبيت، وتحديثات مفاتيح API (`skills.*`)
- Nodes: السرد + الحدود (`node.list`)
- موافقات Exec: تحرير قوائم السماح الخاصة بـ gateway أو node + سياسة ask لـ `exec host=gateway/node` (`exec.approvals.*`)
- Config: عرض/تحرير `~/.openclaw/openclaw.json` (`config.get`, `config.set`)
- Config: تطبيق + إعادة تشغيل مع التحقق (`config.apply`) وإيقاظ آخر جلسة نشطة
- تتضمن كتابات config حارس base-hash لمنع الكتابة فوق تعديلات متزامنة
- تقوم كتابات config (`config.set`/`config.apply`/`config.patch`) أيضًا بفحص تمهيدي لحل SecretRef النشط للمراجع الموجودة في حمولة config المقدمة؛ وتُرفض المراجع النشطة غير القابلة للحل قبل الكتابة
- Config schema + عرض النماذج (`config.schema` / `config.schema.lookup`،
  بما في ذلك الحقلين `title` / `description`، وتلميحات UI المطابقة، وملخصات العناصر
  الفرعية المباشرة، وبيانات تعريف المستندات على عقد الكائنات/الأحرف العامة/المصفوفات/التركيب المتداخلة،
  بالإضافة إلى مخططات Plugin والقنوات عند توفرها)؛ ويكون محرر Raw JSON
  متاحًا فقط عندما تكون اللقطة تملك round-trip خامًا آمنًا
- إذا تعذر على لقطة ما إجراء round-trip للنص الخام بأمان، تفرض واجهة التحكم وضع Form وتعطّل Raw mode لتلك اللقطة
- يحافظ خيار "Reset to saved" في محرر Raw JSON على الشكل الخام الذي أُلّف به (التنسيق، والتعليقات، وتخطيط `$include`) بدلًا من إعادة عرض لقطة مسطحة، حتى تبقى التعديلات الخارجية بعد إعادة التعيين عندما تستطيع اللقطة إجراء round-trip بأمان
- تُعرَض قيم كائنات SecretRef المنظَّمة للقراءة فقط في مدخلات نص النموذج لمنع التلف العرضي الناتج عن تحويل الكائن إلى سلسلة
- التصحيح: لقطات الحالة/الصحة/النماذج + سجل الأحداث + استدعاءات RPC يدوية (`status`, `health`, `models.list`)
- السجلات: تتبع حي لسجلات ملفات gateway مع التصفية/التصدير (`logs.tail`)
- التحديث: تشغيل تحديث package/git + إعادة التشغيل (`update.run`) مع تقرير إعادة تشغيل

ملاحظات لوحة مهام Cron:

- بالنسبة إلى المهام المعزولة، يكون التسليم افتراضيًا إعلان ملخص. ويمكنك التبديل إلى none إذا كنت تريد عمليات تشغيل داخلية فقط.
- تظهر حقول القناة/الهدف عندما يتم اختيار announce.
- يستخدم وضع Webhook القيمة `delivery.mode = "webhook"` مع تعيين `delivery.to` إلى عنوان URL صالح لـ HTTP(S) Webhook.
- بالنسبة إلى مهام الجلسة الرئيسية، تتوفر وضعات التسليم webhook وnone.
- تتضمن عناصر التحكم المتقدمة في التحرير خيارات delete-after-run، ومسح override الخاص بالوكيل، وخيارات Cron exact/stagger،
  وتجاوزات agent model/thinking، ومفاتيح تسليم best-effort.
- يكون التحقق من النموذج مضمّنًا مع أخطاء على مستوى الحقول؛ وتُعطَّل زر الحفظ عند وجود قيم غير صالحة حتى يتم إصلاحها.
- عيّن `cron.webhookToken` لإرسال bearer token مخصص؛ وإذا لم يُعيَّن فسيُرسل Webhook من دون رأس مصادقة.
- fallback متقادم: لا تزال المهام القديمة المخزنة مع `notify: true` قادرة على استخدام `cron.webhook` حتى يتم ترحيلها.

## سلوك الدردشة

- تكون `chat.send` **غير حاجزة**: إذ ترسل إقرارًا فوريًا بصيغة `{ runId, status: "started" }` ويتدفق الرد عبر أحداث `chat`.
- تؤدي إعادة الإرسال باستخدام `idempotencyKey` نفسه إلى إرجاع `{ status: "in_flight" }` أثناء التشغيل، و`{ status: "ok" }` بعد الاكتمال.
- تكون استجابات `chat.history` مقيّدة بالحجم حفاظًا على أمان واجهة المستخدم. وعندما تكون إدخالات النص كبيرة جدًا، قد يقوم Gateway باقتطاع حقول النص الطويلة، وحذف كتل البيانات الوصفية الثقيلة، واستبدال الرسائل كبيرة الحجم بعنصر نائب (`[chat.history omitted: message too large]`).
- تُحفَظ الصور المُولدة من المساعد/المولدة كمرجعيات وسائط مُدارة وتُقدَّم مرة أخرى عبر عناوين URL لوسائط Gateway موثقة، لذلك لا تعتمد إعادة التحميل على بقاء حمولات الصور الخام base64 في استجابة سجل الدردشة.
- يزيل `chat.history` أيضًا وسوم التوجيه المضمنة المعروضة فقط من نص المساعد المرئي (مثل `[[reply_to_*]]` و`[[audio_as_voice]]`)، وحمولات XML الخاصة باستدعاء الأدوات بصيغة نص عادي (بما في ذلك `<tool_call>...</tool_call>` و`<function_call>...</function_call>` و`<tool_calls>...</tool_calls>` و`<function_calls>...</function_calls>` وكتل استدعاء الأدوات المقتطعة)، وكذلك رموز التحكم بالنموذج المتسربة بصيغة ASCII/العرض الكامل، ويحذف إدخالات المساعد التي يكون نصها المرئي بالكامل هو الرمز الصامت الدقيق `NO_REPLY` / `no_reply`.
- أثناء وجود إرسال نشط وأثناء التحديث النهائي للسجل، يحتفظ عرض الدردشة
  برسائل المستخدم/المساعد المحلية المتفائلة ظاهرة إذا أعاد `chat.history`
  مؤقتًا لقطة أقدم؛ ويستبدل النص القانوني تلك الرسائل المحلية بمجرد
  أن يلحق سجل Gateway.
- يضيف `chat.inject` ملاحظة مساعد إلى سجل الجلسة ويبث حدث `chat` لتحديثات واجهة المستخدم فقط (من دون تشغيل وكيل، ومن دون تسليم للقناة).
- تقوم أدوات اختيار النموذج وthinking في رأس الدردشة بتصحيح الجلسة النشطة فورًا عبر `sessions.patch`؛ وهي تجاوزات ثابتة على مستوى الجلسة، وليست خيارات إرسال لجولة واحدة فقط.
- عندما تُظهر تقارير استخدام الجلسة الحديثة من Gateway ضغطًا مرتفعًا على السياق، تعرض
  منطقة محرر الدردشة إشعارًا بالسياق، وعند مستويات Compaction الموصى بها،
  زر compact يشغّل مسار Compaction المعتاد للجلسة. وتُخفى
  لقطات الرموز القديمة حتى يبلّغ Gateway عن استخدام حديث مرة أخرى.
- يستخدم وضع Talk مزوّد صوت فوريًا مسجلًا يدعم جلسات WebRTC في المتصفح. كوّن OpenAI باستخدام `talk.provider: "openai"` بالإضافة إلى
  `talk.providers.openai.apiKey`، أو أعد استخدام إعداد مزوّد Realtime في Voice Call. ولا يستلم
  المتصفح مفتاح OpenAI API القياسي أبدًا؛ بل يستلم فقط
  سر عميل Realtime المؤقت. ويدعم الصوت الفوري Google Live
  في Voice Call الخلفي وجسور Google Meet، لكن ليس بعد في مسار
  WebRTC هذا الخاص بالمتصفح. ويُجمَّع prompt جلسة Realtime بواسطة Gateway؛
  ولا يقبل `talk.realtime.session` تجاوزات التعليمات المقدمة من المستدعي.
- في محرر Chat، يكون عنصر تحكم Talk هو زر الأمواج بجانب
  زر الإملاء الصوتي للميكروفون. وعندما يبدأ Talk، يُظهر صف حالة المحرر
  `Connecting Talk...`، ثم `Talk live` أثناء اتصال الصوت، أو
  `Asking OpenClaw...` بينما يستشير استدعاء أداة فوري النموذج الأكبر
  المكوَّن عبر `chat.send`.
- الإيقاف:
  - انقر **Stop** (يستدعي `chat.abort`)
  - أثناء وجود تشغيل نشط، تُوضَع المتابعات العادية في الطابور. انقر **Steer** على رسالة موضوعة في الطابور لحقن تلك المتابعة في الجولة الجارية.
  - اكتب `/stop` (أو عبارات إجهاض مستقلة مثل `stop` أو `stop action` أو `stop run` أو `stop openclaw` أو `please stop`) للإيقاف خارج النطاق
  - يدعم `chat.abort` الصيغة `{ sessionKey }` (من دون `runId`) لإيقاف كل العمليات النشطة لتلك الجلسة
- الاحتفاظ الجزئي عند الإجهاض:
  - عندما تُجهَض عملية تشغيل، قد يظل نص المساعد الجزئي ظاهرًا في واجهة المستخدم
  - يحفظ Gateway نص المساعد الجزئي المُجهَض في سجل النص عندما يكون هناك خرج مخزَّن مؤقتًا
  - تتضمن الإدخالات المحفوظة بيانات وصفية للإجهاض حتى يتمكن مستهلكو النص من تمييز الأجزاء المُجهَضة عن خرج الإكمال العادي

## تثبيت PWA وإشعارات Web Push

تأتي واجهة التحكم مع `manifest.webmanifest` وservice worker،
بحيث يمكن للمتصفحات الحديثة تثبيتها كتطبيق PWA مستقل. ويتيح Web Push لـ
Gateway إيقاظ تطبيق PWA المثبت عبر الإشعارات حتى عندما لا تكون علامة التبويب أو
نافذة المتصفح مفتوحة.

| السطح | ما الذي يفعله |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest` | بيان PWA. تعرض المتصفحات خيار "تثبيت التطبيق" بمجرد أن يصبح متاحًا. |
| `ui/public/sw.js` | Service worker يتعامل مع أحداث `push` ونقرات الإشعارات. |
| `push/vapid-keys.json` (ضمن دليل حالة OpenClaw) | زوج مفاتيح VAPID مولَّد تلقائيًا يُستخدم لتوقيع حمولات Web Push. |
| `push/web-push-subscriptions.json` | نقاط اشتراك المتصفح المحفوظة. |

يمكنك تجاوز زوج مفاتيح VAPID عبر متغيرات env على عملية Gateway عندما
تريد تثبيت المفاتيح (لنشرات متعددة المضيفين، أو تدوير الأسرار، أو
الاختبارات):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (الافتراضي `mailto:openclaw@localhost`)

تستخدم واجهة التحكم أساليب Gateway المقيدة بالنطاق التالية لتسجيل
واختبار اشتراكات المتصفح:

- `push.web.vapidPublicKey` — يجلب مفتاح VAPID العام النشط.
- `push.web.subscribe` — يسجل `endpoint` بالإضافة إلى `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` — يزيل endpoint مسجّلًا.
- `push.web.test` — يرسل إشعار اختبار إلى اشتراك المستدعي.

يستقل Web Push عن مسار ترحيل iOS APNS
(راجع [Configuration](/ar/gateway/configuration) للدفع المعتمد على relay)
وعن الأسلوب الحالي `push.test`، اللذين يستهدفان الاقتران الأصلي على الأجهزة المحمولة.

## التضمينات المستضافة

يمكن لرسائل المساعد عرض محتوى ويب مستضاف مضمّنًا باستخدام الرمز المختصر `[embed ...]`.
وتتحكم السياسة sandbox الخاصة بـ iframe عبر
`gateway.controlUi.embedSandbox`:

- `strict`: يعطل تنفيذ السكربت داخل التضمينات المستضافة
- `scripts`: يسمح بالتضمينات التفاعلية مع الحفاظ على عزل الأصل؛ وهذا هو
  الافتراضي ويكون كافيًا عادةً للألعاب/العناصر المصغرة المستقلة في المتصفح
- `trusted`: يضيف `allow-same-origin` فوق `allow-scripts` للمستندات من الموقع نفسه
  التي تحتاج عمدًا إلى صلاحيات أقوى

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

استخدم `trusted` فقط عندما يحتاج المستند المضمّن فعلًا إلى
سلوك same-origin. وبالنسبة إلى معظم الألعاب التفاعلية ولوحات الرسم التي يولدها الوكيل، يكون `scripts`
هو الخيار الأكثر أمانًا.

تظل عناوين URL المطلقة الخارجية لتضمين `http(s)` محظورة افتراضيًا. وإذا كنت
تريد عمدًا أن يُحمَّل `[embed url="https://..."]` صفحات طرف ثالث، فعيّن
`gateway.controlUi.allowExternalEmbedUrls: true`.

## الوصول عبر Tailnet (موصى به)

### Tailscale Serve المدمج (المفضل)

أبقِ Gateway على loopback ودع Tailscale Serve يمرر الطلبات إليه عبر HTTPS:

```bash
openclaw gateway --tailscale serve
```

افتح:

- `https://<magicdns>/` (أو `gateway.controlUi.basePath` الذي قمت بتكوينه)

افتراضيًا، يمكن لمصادقة طلبات Control UI/WebSocket Serve أن تتم عبر رؤوس هوية Tailscale
(`tailscale-user-login`) عندما تكون `gateway.auth.allowTailscale` تساوي `true`. ويتحقق OpenClaw
من الهوية عبر حل عنوان `x-forwarded-for` باستخدام
`tailscale whois` ومطابقته مع الرأس، ولا يقبل هذه الرؤوس إلا عندما
يصل الطلب إلى loopback مع رؤوس `x-forwarded-*` الخاصة بـ Tailscale. وعيّن
`gateway.auth.allowTailscale: false` إذا كنت تريد فرض بيانات اعتماد صريحة بمفتاح سري مشترك
حتى لحركة Serve. ثم استخدم `gateway.auth.mode: "token"` أو
`"password"`.
وفي مسار هوية Serve غير المتزامن هذا، تُسلسَل محاولات المصادقة الفاشلة للعميل IP نفسه
ونطاق المصادقة نفسه قبل كتابة حدود المعدل. ولذلك قد تعرض
المحاولات السيئة المتزامنة من المتصفح نفسه عبارة `retry later` في الطلب الثاني
بدلًا من سباق خطأين عاديين متوازيين.
تفترض مصادقة Serve من دون رمز مميز أن مضيف gateway موثوق. وإذا كان من الممكن تشغيل
شيفرة محلية غير موثوقة على ذلك المضيف، فافرض مصادقة token/password.

### الربط بـ tailnet + token

```bash
openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
```

ثم افتح:

- `http://<tailscale-ip>:18789/` (أو `gateway.controlUi.basePath` الذي قمت بتكوينه)

ألصق المفتاح السري المشترك المطابق في إعدادات واجهة المستخدم (ويُرسل
على هيئة `connect.params.auth.token` أو `connect.params.auth.password`).

## HTTP غير الآمن

إذا فتحت لوحة المعلومات عبر HTTP عادي (`http://<lan-ip>` أو `http://<tailscale-ip>`)،
فسيعمل المتصفح في **سياق غير آمن** ويمنع WebCrypto. وافتراضيًا،
يقوم OpenClaw **بحظر** اتصالات واجهة التحكم من دون هوية جهاز.

الاستثناءات الموثقة:

- توافق HTTP غير الآمن على localhost فقط مع `gateway.controlUi.allowInsecureAuth=true`
- نجاح مصادقة المشغّل في واجهة التحكم عبر `gateway.auth.mode: "trusted-proxy"`
- وضع كسر الزجاج `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**الإصلاح الموصى به:** استخدم HTTPS (Tailscale Serve) أو افتح الواجهة محليًا:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (على مضيف gateway)

**سلوك مفتاح التبديل الخاص بالمصادقة غير الآمنة:**

```json5
{
  gateway: {
    controlUi: { allowInsecureAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

يُعد `allowInsecureAuth` مفتاح توافق محلي فقط:

- يسمح لجلسات واجهة التحكم على localhost بالمتابعة من دون هوية جهاز في
  سياقات HTTP غير الآمنة.
- لا يتجاوز فحوصات الإقران.
- لا يخفف متطلبات هوية الجهاز البعيدة (غير localhost).

**للطوارئ فقط:**

```json5
{
  gateway: {
    controlUi: { dangerouslyDisableDeviceAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

يعطّل `dangerouslyDisableDeviceAuth` فحوصات هوية جهاز واجهة التحكم ويُعد
خفضًا أمنيًا شديدًا. أعده بسرعة بعد الاستخدام الطارئ.

ملاحظة trusted-proxy:

- يمكن لنجاح مصادقة trusted-proxy أن يسمح بجلسات **مشغّل** واجهة التحكم من دون
  هوية جهاز
- لكن هذا **لا** يمتد إلى جلسات واجهة التحكم ذات دور node
- كما أن reverse proxies على loopback للمضيف نفسه لا تفي بمصادقة trusted-proxy؛ راجع
  [مصادقة trusted-proxy](/ar/gateway/trusted-proxy-auth)

راجع [Tailscale](/ar/gateway/tailscale) للحصول على إرشادات إعداد HTTPS.

## سياسة أمان المحتوى

تأتي واجهة التحكم بسياسة `img-src` محكمة: حيث يُسمح فقط بالأصول **من الأصل نفسه**، وعناوين `data:`، وعناوين `blob:` التي تُنشأ محليًا. وتُرفض عناوين الصور البعيدة `http(s)` والعناوين النسبية للبروتوكول من قبل المتصفح ولا تُنشئ طلبات شبكة.

ما الذي يعنيه هذا عمليًا:

- لا تزال الصور الرمزية والصور المقدمة عبر مسارات نسبية (مثل `/avatars/<id>`) تُعرض، بما في ذلك مسارات الصور الرمزية الموثقة التي تجلبها واجهة المستخدم وتحولها إلى عناوين `blob:` محلية.
- لا تزال عناوين `data:image/...` المضمنة تُعرض (وهو مفيد للحمولات داخل البروتوكول).
- لا تزال عناوين `blob:` المحلية التي تنشئها واجهة التحكم تُعرض.
- تُزال عناوين الصور الرمزية البعيدة التي تصدرها بيانات القنوات الوصفية من خلال مساعدات الصور الرمزية في واجهة التحكم وتُستبدل بالشعار/الشارة المدمجين، بحيث لا تستطيع قناة مخترقة أو خبيثة فرض طلبات جلب صور بعيدة عشوائية من متصفح المشغّل.

لا تحتاج إلى تغيير أي شيء للحصول على هذا السلوك — فهو مفعّل دائمًا وغير قابل للتكوين.

## مصادقة مسار الصورة الرمزية

عندما تكون مصادقة gateway مكوّنة، تتطلب نقطة نهاية الصورة الرمزية في واجهة التحكم الرمز المميز نفسه الخاص بـ gateway المستخدم في بقية API:

- يعيد `GET /avatar/<agentId>` صورة الصورة الرمزية فقط للمستدعين الموثقين. ويعيد `GET /avatar/<agentId>?meta=1` البيانات الوصفية للصورة الرمزية وفق القاعدة نفسها.
- تُرفض الطلبات غير الموثقة إلى أي من المسارين (بما يتطابق مع مسار assistant-media الشقيق). وهذا يمنع مسار الصورة الرمزية من تسريب هوية الوكيل على المضيفين المحميين بخلاف ذلك.
- تقوم واجهة التحكم نفسها بتمرير رمز gateway المميز كرأس bearer عند جلب الصور الرمزية، وتستخدم عناوين blob موثقة حتى تستمر الصورة في الظهور في لوحات المعلومات.

إذا عطّلت مصادقة gateway (وهو غير موصى به على المضيفين المشتركين)، فسيصبح مسار الصورة الرمزية أيضًا غير موثّق، بما يتماشى مع بقية gateway.

## بناء الواجهة

يقدّم Gateway الملفات الثابتة من `dist/control-ui`. ابنِها باستخدام:

```bash
pnpm ui:build
```

قاعدة مطلقة اختيارية (عندما تريد عناوين URL ثابتة للأصول):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

للتطوير المحلي (خادم تطوير منفصل):

```bash
pnpm ui:dev
```

ثم وجّه الواجهة إلى عنوان WS الخاص بـ Gateway لديك (مثل `ws://127.0.0.1:18789`).

## التصحيح/الاختبار: خادم تطوير + Gateway بعيد

واجهة التحكم عبارة عن ملفات ثابتة؛ ويمكن تكوين هدف WebSocket بحيث يختلف
عن أصل HTTP. ويكون هذا مفيدًا عندما تريد خادم تطوير Vite محليًا
بينما يعمل Gateway في مكان آخر.

1. ابدأ خادم تطوير الواجهة: `pnpm ui:dev`
2. افتح عنوان URL مثل:

```text
http://localhost:5173/?gatewayUrl=ws://<gateway-host>:18789
```

مصادقة لمرة واحدة اختيارية (إذا لزم الأمر):

```text
http://localhost:5173/?gatewayUrl=wss://<gateway-host>:18789#token=<gateway-token>
```

ملاحظات:

- يُخزَّن `gatewayUrl` في localStorage بعد التحميل ويُزال من عنوان URL.
- يجب تمرير `token` عبر جزء عنوان URL (`#token=...`) كلما أمكن. فالأجزاء لا تُرسَل إلى الخادم، مما يتجنب تسرب سجلات الطلبات وReferer. ولا تزال معاملات الاستعلام القديمة `?token=` تُستورد مرة واحدة للتوافق، ولكن فقط كمسار احتياطي، وتُزال فورًا بعد bootstrap.
- يُحتفَظ بـ `password` في الذاكرة فقط.
- عند تعيين `gatewayUrl`، لا تعود الواجهة إلى بيانات اعتماد config أو البيئة.
  وفّر `token` (أو `password`) صراحةً. ويُعد غياب بيانات الاعتماد الصريحة خطأً.
- استخدم `wss://` عندما يكون Gateway خلف TLS (مثل Tailscale Serve أو HTTPS proxy، إلخ).
- لا يُقبل `gatewayUrl` إلا في نافذة من المستوى الأعلى (غير مضمّنة) لمنع clickjacking.
- يجب أن تعيّن عمليات نشر واجهة التحكم غير الخاصة بـ loopback القيمة `gateway.controlUi.allowedOrigins`
  صراحةً (أصولًا كاملة). ويشمل ذلك إعدادات التطوير البعيدة.
- لا تستخدم `gateway.controlUi.allowedOrigins: ["*"]` إلا في اختبارات محلية
  مضبوطة بإحكام. فهي تعني السماح لأي أصل متصفح، وليس “مطابقة أي مضيف أستخدمه”.
- يفعّل `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true`
  وضع fallback لأصل Host-header، لكنه وضع أمني خطير.

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

تفاصيل إعداد الوصول البعيد: [الوصول البعيد](/ar/gateway/remote).
