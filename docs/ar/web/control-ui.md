تُعد Control UI تطبيق صفحة واحدة صغيرًا مبنيًا باستخدام **Vite + Lit** ويقدّمه Gateway:

- الافتراضي: `http://<host>:18789/`
- بادئة اختيارية: اضبط `gateway.controlUi.basePath` (مثل `/openclaw`)

وهي تتحدث **مباشرة إلى WebSocket الخاصة بـ Gateway** على المنفذ نفسه.

## فتح سريع (محلي)

إذا كان Gateway يعمل على الكمبيوتر نفسه، فافتح:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (أو [http://localhost:18789/](http://localhost:18789/))

إذا فشل تحميل الصفحة، فابدأ Gateway أولًا: `openclaw gateway`.

تُوفَّر المصادقة أثناء مصافحة WebSocket عبر:

- `connect.params.auth.token`
- `connect.params.auth.password`
- رؤوس هوية Tailscale Serve عندما تكون `gateway.auth.allowTailscale: true`
- رؤوس هوية trusted-proxy عندما تكون `gateway.auth.mode: "trusted-proxy"`

تحتفظ لوحة إعدادات لوحة المعلومات برمز token لجلسة علامة تبويب المتصفح الحالية
وعنوان URL المحدد لـ gateway؛ ولا تُحفَظ كلمات المرور. وعادةً ما يولّد الإعداد الأولي
رمز gateway token لمصادقة السر المشترك عند أول اتصال، لكن مصادقة كلمة المرور
تعمل أيضًا عندما تكون `gateway.auth.mode` مساوية لـ `"password"`.

## إقران الجهاز (الاتصال الأول)

عندما تتصل بـ Control UI من متصفح أو جهاز جديد، يطلب Gateway
**موافقة إقران لمرة واحدة** — حتى إذا كنت على Tailnet نفسها
مع `gateway.auth.allowTailscale: true`. وهذا إجراء أمني لمنع
الوصول غير المصرح به.

**ما ستراه:** `"disconnected (1008): pairing required"`

**للموافقة على الجهاز:**

```bash
# List pending requests
openclaw devices list

# Approve by request ID
openclaw devices approve <requestId>
```

إذا أعاد المتصفح محاولة الإقران مع تغيّر تفاصيل المصادقة (الدور/النطاقات/المفتاح العام)،
فيُستبدل الطلب المعلق السابق ويُنشأ `requestId` جديد. أعد تشغيل
`openclaw devices list` قبل الموافقة.

إذا كان المتصفح مقترنًا بالفعل وغيّرته من وصول قراءة إلى
وصول كتابة/إدارة، فيُعامل هذا على أنه ترقية موافقة، لا إعادة
اتصال صامتة. ويحافظ OpenClaw على الموافقة القديمة نشطة، ويحظر
إعادة الاتصال الأوسع، ويطلب منك الموافقة على مجموعة النطاقات الجديدة صراحةً.

بعد الموافقة، يُتذكر الجهاز ولن يحتاج إلى إعادة موافقة إلا إذا
ألغيتَه باستخدام `openclaw devices revoke --device <id> --role <role>`. راجع
[Devices CLI](/ar/cli/devices) لمعرفة تدوير الرموز وإلغائها.

**ملاحظات:**

- تُوافق اتصالات المتصفح المحلية المباشرة عبر loopback ‏(`127.0.0.1` / `localhost`) تلقائيًا.
- لا تزال اتصالات المتصفح عبر Tailnet وLAN تتطلب موافقة صريحة، حتى عندما
  تنشأ من الجهاز نفسه.
- يولد كل ملف تعريف متصفح معرّف جهاز فريدًا، لذا فإن تبديل المتصفحات أو
  مسح بيانات المتصفح سيتطلب إعادة الإقران.

## الهوية الشخصية (محلية للمتصفح)

تدعم Control UI هوية شخصية لكل متصفح (اسم عرض وصورة رمزية)
تُرفق بالرسائل الصادرة لأغراض الإسناد في الجلسات المشتركة. وهي
توجد في تخزين المتصفح، ومحددة بملف تعريف المتصفح الحالي، ولا
تُزامن مع أجهزة أخرى ولا تُحفَظ من جهة الخادم باستثناء بيانات
تأليف النص العادية للرسائل التي ترسلها فعليًا. ويؤدي مسح بيانات
الموقع أو تبديل المتصفحات إلى إعادة ضبطها إلى حالة فارغة.

## نقطة نهاية تهيئة وقت التشغيل

تجلب Control UI إعدادات وقت التشغيل من
`/__openclaw/control-ui-config.json`. وتخضع نقطة النهاية هذه
لنفس مصادقة gateway المستخدمة لبقية سطح HTTP: فلا يمكن للمتصفحات
غير الموثقة جلبها، ويتطلب الجلب الناجح إما رمز gateway token/كلمة مرور
صالحة مسبقًا، أو هوية Tailscale Serve، أو هوية trusted-proxy.

## دعم اللغة

يمكن لـ Control UI توطين نفسها عند التحميل الأول بناءً على لغة متصفحك.
وللتجاوز لاحقًا، افتح **Overview -> Gateway Access -> Language**.
يوجد محدد اللغة في بطاقة Gateway Access، وليس ضمن Appearance.

- اللغات المدعومة: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `tr`, `uk`, `id`, `pl`, `th`
- تُحمَّل الترجمات غير الإنجليزية بكسل في المتصفح.
- تُحفَظ اللغة المحددة في تخزين المتصفح ويُعاد استخدامها في الزيارات اللاحقة.
- تعود مفاتيح الترجمة المفقودة إلى الإنجليزية.

## ما الذي يمكنها فعله (حاليًا)

- الدردشة مع النموذج عبر Gateway WS ‏(`chat.history`, `chat.send`, `chat.abort`, `chat.inject`)
- التحدث إلى OpenAI Realtime مباشرة من المتصفح عبر WebRTC. حيث يقوم Gateway
  بإصدار سر عميل Realtime قصير العمر باستخدام `talk.realtime.session`؛ ويقوم
  المتصفح بإرسال صوت الميكروفون مباشرة إلى OpenAI وتمرير استدعاءات أداة
  `openclaw_agent_consult` مرة أخرى عبر `chat.send` إلى نموذج OpenClaw الأكبر
  المهيأ.
- بث استدعاءات الأدوات + بطاقات خرج الأدوات الحية في الدردشة (أحداث الوكيل)
- القنوات: حالة القنوات المضمنة بالإضافة إلى قنوات Plugins المضمنة/الخارجية، وتسجيل دخول QR، والتهيئة لكل قناة (`channels.status`, `web.login.*`, `config.patch`)
- Instances: قائمة الحضور + التحديث (`system-presence`)
- الجلسات: الإدراج + تجاوزات لكل جلسة للنموذج/التفكير/fast/verbose/trace/reasoning ‏(`sessions.list`, `sessions.patch`)
- Dreams: حالة Dreaming، ومفتاح التمكين/التعطيل، وقارئ Dream Diary ‏(`doctor.memory.status`, `doctor.memory.dreamDiary`, `config.patch`)
- وظائف Cron: الإدراج/الإضافة/التحرير/التشغيل/التمكين/التعطيل + سجل التشغيل (`cron.*`)
- Skills: الحالة، والتمكين/التعطيل، والتثبيت، وتحديثات مفتاح API ‏(`skills.*`)
- Nodes: الإدراج + الإمكانيات (`node.list`)
- موافقات Exec: تحرير allowlists الخاصة بـ gateway أو node + سياسة ask لـ `exec host=gateway/node` ‏(`exec.approvals.*`)
- التهيئة: عرض/تحرير `~/.openclaw/openclaw.json` ‏(`config.get`, `config.set`)
- التهيئة: التطبيق + إعادة التشغيل مع التحقق (`config.apply`) وإيقاظ آخر جلسة نشطة
- تتضمن كتابات التهيئة حارس base-hash لمنع الكتابة فوق تعديلات متزامنة
- تقوم كتابات التهيئة (`config.set`/`config.apply`/`config.patch`) أيضًا بفحص تمهيدي لحل SecretRef النشط للمراجع الموجودة في حمولة التهيئة المقدمة؛ وتُرفض المراجع النشطة غير المحلولة قبل الكتابة
- مخطط التهيئة + عرض النماذج (`config.schema` / `config.schema.lookup`,
  بما في ذلك `title` / `description` للحقل، وتلميحات واجهة المستخدم المطابقة،
  وملخصات الأبناء المباشرين، وبيانات المستندات الوصفية على العقد المتداخلة
  object/wildcard/array/composition، بالإضافة إلى مخططات Plugins + القنوات عند التوفر)؛
  ويكون محرر Raw JSON متاحًا فقط عندما تملك اللقطة raw round-trip آمنًا
- إذا تعذر على لقطة ما إجراء raw round-trip آمن، تفرض Control UI وضع Form وتعطّل وضع Raw لتلك اللقطة
- يحافظ خيار "Reset to saved" في محرر Raw JSON على الشكل المؤلف خامًا (التنسيق، والتعليقات، وتخطيط `$include`) بدلًا من إعادة عرض لقطة مسطحة، بحيث تبقى التعديلات الخارجية بعد إعادة الضبط عندما يمكن للّقطة إجراء raw round-trip آمن
- تُعرض قيم كائنات Structured SecretRef للقراءة فقط في حقول النص بالنموذج لمنع التلف العرضي من كائن إلى سلسلة
- التصحيح: لقطات status/health/models + سجل الأحداث + استدعاءات RPC يدوية (`status`, `health`, `models.list`)
- السجلات: تتبع حي لسجلات ملفات gateway مع التصفية/التصدير (`logs.tail`)
- التحديث: تشغيل تحديث package/git + إعادة التشغيل (`update.run`) مع تقرير إعادة تشغيل

ملاحظات لوحة وظائف Cron:

- بالنسبة إلى الوظائف المعزولة، يكون التسليم افتراضيًا إعلان ملخص. ويمكنك التبديل إلى none إذا أردت تشغيلات داخلية فقط.
- تظهر حقول القناة/الهدف عند اختيار announce.
- يستخدم وضع Webhook القيمة `delivery.mode = "webhook"` مع تعيين `delivery.to` إلى URL ‏Webhook صالح من نوع HTTP(S).
- بالنسبة إلى وظائف الجلسة الرئيسية، تكون أوضاع التسليم webhook وnone متاحة.
- تتضمن عناصر التحكم في التحرير المتقدم delete-after-run، ومسح تجاوز الوكيل، وخيارات cron الدقيقة/الموزعة،
  وتجاوزات model/thinking الخاصة بالوكيل، وخيارات التسليم بأفضل جهد.
- يكون التحقق من صحة النموذج ضمنيًا مع أخطاء على مستوى الحقل؛ وتعطل القيم غير الصالحة زر الحفظ حتى يتم إصلاحها.
- اضبط `cron.webhookToken` لإرسال bearer token مخصص، وإذا حُذف فسيُرسل الـ webhook من دون رأس مصادقة.
- fallback قديم مهمل: لا تزال الوظائف القديمة المخزنة مع `notify: true` قادرة على استخدام `cron.webhook` حتى تتم الترحيل.

## سلوك الدردشة

- `chat.send` **غير حاجز**: فهو يقرّ فورًا بالصيغة `{ runId, status: "started" }` ويُبث الرد عبر أحداث `chat`.
- تعيد إعادة الإرسال باستخدام `idempotencyKey` نفسه `{ status: "in_flight" }` أثناء التشغيل، و`{ status: "ok" }` بعد الاكتمال.
- تكون استجابات `chat.history` محدودة الحجم من أجل أمان واجهة المستخدم. وعندما تكون إدخالات النص كبيرة جدًا، قد يقوم Gateway باقتطاع حقول النص الطويلة، وحذف كتل البيانات الوصفية الثقيلة، واستبدال الرسائل الضخمة بعنصر نائب (`[chat.history omitted: message too large]`).
- تُحفَظ الصور المولدة بواسطة المساعد/الصور المُنشأة بوصفها مراجع وسائط مُدارة وتُقدَّم مرة أخرى عبر عناوين Gateway للوسائط الموثقة، لذلك لا تعتمد عمليات إعادة التحميل على بقاء حمولة الصورة base64 الخام في استجابة سجل الدردشة.
- تقوم `chat.history` أيضًا بتجريد وسوم التوجيه المضمنة المخصصة للعرض فقط من نص المساعد المرئي (مثل `[[reply_to_*]]` و`[[audio_as_voice]]`)، وحمولات XML الخاصة باستدعاءات الأدوات النصية العادية (بما في ذلك `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>`, وكتل استدعاء الأدوات المقتطعة)، وتحذف رموز التحكم الخاصة بالنموذج المسربة بصيغة ASCII/العرض الكامل، وتحذف إدخالات المساعد التي يكون كامل نصها المرئي هو الرمز الصامت الدقيق `NO_REPLY` / `no_reply`.
- تقوم `chat.inject` بإلحاق ملاحظة مساعد إلى نص الجلسة وتبث حدث `chat` لتحديثات واجهة المستخدم فقط (من دون تشغيل وكيل أو تسليم عبر القناة).
- تقوم محددات النموذج والتفكير في رأس الدردشة بترقيع الجلسة النشطة فورًا عبر `sessions.patch`؛ فهي تجاوزات دائمة للجلسة، وليست خيارات إرسال لدور واحد فقط.
- يستخدم وضع Talk موفّر الصوت الفوري المسجل. هيئ OpenAI باستخدام
  `talk.provider: "openai"` مع `talk.providers.openai.apiKey`، أو أعد استخدام
  تهيئة موفّر Voice Call الفوري. ولا يتلقى المتصفح مفتاح API القياسي لـ OpenAI أبدًا؛
  بل يتلقى فقط سر عميل Realtime مؤقتًا. ويقوم Gateway بتجميع مطالبة جلسة
  Realtime؛ ولا يقبل `talk.realtime.session` تجاوزات للتعليمات يوفرها
  المستدعي.
- في محرر الدردشة، يكون عنصر تحكم Talk هو زر الموجات الموجود بجانب
  زر الإملاء الصوتي للميكروفون. وعندما يبدأ Talk، يعرض صف الحالة في المحرر
  `Connecting Talk...`، ثم `Talk live` أثناء اتصال الصوت، أو
  `Asking OpenClaw...` بينما يستشير استدعاء أداة فورية النموذج
  الأكبر المهيأ عبر `chat.send`.
- الإيقاف:
  - انقر **Stop** ‏(يستدعي `chat.abort`)
  - أثناء نشاط تشغيل ما، تُوضَع المتابعات العادية في الطابور. انقر **Steer** على رسالة موضوعة في الطابور لحقن تلك المتابعة في الدور الجاري.
  - اكتب `/stop` ‏(أو عبارات إلغاء مستقلة مثل `stop`, `stop action`, `stop run`, `stop openclaw`, `please stop`) للإلغاء خارج النطاق
  - يدعم `chat.abort` الصيغة `{ sessionKey }` ‏(من دون `runId`) لإلغاء جميع التشغيلات النشطة لتلك الجلسة
- الاحتفاظ الجزئي عند الإلغاء:
  - عند إلغاء تشغيل ما، قد يظل نص المساعد الجزئي ظاهرًا في واجهة المستخدم
  - يحفظ Gateway نص المساعد الجزئي المجهض داخل سجل النص عندما يكون هناك خرج مخزّن
  - تتضمن الإدخالات المحفوظة بيانات وصفية للإلغاء حتى يتمكن مستهلكو النص من تمييز الأجزاء الملغاة عن خرج الإكمال العادي

## التضمينات المستضافة

يمكن لرسائل المساعد عرض محتوى ويب مستضاف ضمنيًا باستخدام shortcode من نوع `[embed ...]`.
وتتحكم القيمة `gateway.controlUi.embedSandbox` في سياسة sandbox الخاصة بـ iframe:

- `strict`: يعطّل تنفيذ النصوص البرمجية داخل التضمينات المستضافة
- `scripts`: يسمح بالتضمينات التفاعلية مع الحفاظ على عزل الأصل؛ وهذا هو
  الافتراضي ويكفي عادةً للألعاب/الأدوات المصغرة ذاتية الاكتفاء داخل المتصفح
- `trusted`: يضيف `allow-same-origin` فوق `allow-scripts` للمستندات من
  الموقع نفسه التي تحتاج عمدًا إلى امتيازات أقوى

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

استخدم `trusted` فقط عندما يحتاج المستند المضمن فعلًا إلى
سلوك same-origin. وبالنسبة إلى معظم الألعاب التفاعلية ولوحات الرسم
التي يولدها الوكيل، تكون `scripts` الخيار الأكثر أمانًا.

تظل عناوين URL الخارجية المطلقة `http(s)` الخاصة بالتضمين محظورة افتراضيًا. وإذا
كنت تريد عمدًا أن يقوم `[embed url="https://..."]` بتحميل صفحات من جهات خارجية، فاضبط
`gateway.controlUi.allowExternalEmbedUrls: true`.

## الوصول عبر Tailnet (موصى به)

### Tailscale Serve المدمج (المفضل)

أبقِ Gateway على loopback ودَع Tailscale Serve يمرره عبر HTTPS:

```bash
openclaw gateway --tailscale serve
```

افتح:

- `https://<magicdns>/` ‏(أو `gateway.controlUi.basePath` الذي هيأته)

افتراضيًا، يمكن لمصادقة طلبات Control UI/WebSocket الخاصة بـ Serve أن تستخدم رؤوس هوية Tailscale
(`tailscale-user-login`) عندما تكون `gateway.auth.allowTailscale` مساوية لـ `true`. ويقوم OpenClaw
بالتحقق من الهوية عبر حل عنوان `x-forwarded-for` باستخدام
`tailscale whois` ثم مطابقته مع الرأس، ولا يقبل ذلك إلا عندما
يضرب الطلب loopback مع رؤوس `x-forwarded-*` الخاصة بـ Tailscale. اضبط
`gateway.auth.allowTailscale: false` إذا كنت تريد فرض بيانات اعتماد صريحة
مبنية على سر مشترك حتى بالنسبة إلى حركة Serve. ثم استخدم `gateway.auth.mode: "token"` أو
`"password"`.
وبالنسبة إلى مسار هوية Serve غير المتزامن هذا، تُسلسَل محاولات المصادقة الفاشلة
لنفس عنوان IP للعميل ونطاق المصادقة نفسه قبل عمليات كتابة rate-limit. ولذلك قد
تُظهر عمليات الإعادة المتزامنة غير الصحيحة من المتصفح نفسه `retry later` في
الطلب الثاني بدلًا من سباق حالتي عدم تطابق عاديتين بالتوازي.
تفترض مصادقة Serve من دون token أن مضيف gateway موثوق. وإذا كان من الممكن تشغيل
شيفرة محلية غير موثوقة على ذلك المضيف، فاشترط مصادقة token/كلمة مرور.

### الربط بـ tailnet + token

```bash
openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
```

ثم افتح:

- `http://<tailscale-ip>:18789/` ‏(أو `gateway.controlUi.basePath` الذي هيأته)

ألصق السر المشترك المطابق في إعدادات واجهة المستخدم (يُرسَل بصفته
`connect.params.auth.token` أو `connect.params.auth.password`).

## HTTP غير الآمن

إذا فتحت لوحة المعلومات عبر HTTP عادي (`http://<lan-ip>` أو `http://<tailscale-ip>`)،
فسيعمل المتصفح ضمن **سياق غير آمن** ويمنع WebCrypto. وافتراضيًا،
يقوم OpenClaw **بحظر** اتصالات Control UI التي لا تحتوي على هوية جهاز.

الاستثناءات الموثقة:

- توافق HTTP غير الآمن الخاص بـ localhost فقط مع `gateway.controlUi.allowInsecureAuth=true`
- مصادقة المشغّل الناجحة لـ Control UI عبر `gateway.auth.mode: "trusted-proxy"`
- خيار break-glass ‏`gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**الإصلاح الموصى به:** استخدم HTTPS ‏(Tailscale Serve) أو افتح واجهة المستخدم محليًا:

- `https://<magicdns>/` ‏(Serve)
- `http://127.0.0.1:18789/` ‏(على مضيف gateway)

**سلوك مفتاح insecure-auth:**

```json5
{
  gateway: {
    controlUi: { allowInsecureAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

إن `allowInsecureAuth` هو مفتاح توافق محلي فقط:

- يسمح لجلسات localhost الخاصة بـ Control UI بالمتابعة من دون هوية جهاز في
  سياقات HTTP غير الآمنة.
- لا يتجاوز فحوصات الإقران.
- لا يخفف متطلبات هوية الجهاز البعيدة (غير localhost).

**للاستخدام عند الكسر فقط:**

```json5
{
  gateway: {
    controlUi: { dangerouslyDisableDeviceAuth: true },
    bind: "tailnet",
    auth: { mode: "token", token: "replace-me" },
  },
}
```

يقوم `dangerouslyDisableDeviceAuth` بتعطيل فحوصات هوية جهاز Control UI ويُعد
خفضًا أمنيًا شديدًا. أعِد التراجع عنه سريعًا بعد الاستخدام الطارئ.

ملاحظة trusted-proxy:

- يمكن لمصادقة trusted-proxy الناجحة أن تسمح بجلسات **operator** الخاصة بـ Control UI
  من دون هوية جهاز
- وهذا **لا** يمتد إلى جلسات Control UI ذات دور node
- لا تزال proxies العكسية على loopback للمضيف نفسه لا تستوفي مصادقة trusted-proxy؛ راجع
  [Trusted Proxy Auth](/ar/gateway/trusted-proxy-auth)

راجع [Tailscale](/ar/gateway/tailscale) للاطلاع على إرشادات إعداد HTTPS.

## سياسة أمان المحتوى

تشحن Control UI بسياسة `img-src` ضيقة: يُسمح فقط بالأصول ذات **الأصل نفسه** وروابط `data:`. وتُرفض عناوين الصور البعيدة `http(s)` والعناوين النسبية للبروتوكول من قبل المتصفح ولا تصدر طلبات جلب عبر الشبكة.

ما الذي يعنيه هذا عمليًا:

- تظل الصور الرمزية والصور المقدمة ضمن مسارات نسبية (مثل `/avatars/<id>`) قابلة للعرض.
- تظل عناوين `data:image/...` المضمنة قابلة للعرض (وهذا مفيد للحمولات داخل البروتوكول).
- تُجرّد عناوين الصور الرمزية البعيدة الصادرة من بيانات القنوات الوصفية في مساعدات الصور الرمزية الخاصة بـ Control UI وتُستبدل بالشعار/الشارة المضمنين، بحيث لا تستطيع قناة مخترقة أو خبيثة فرض عمليات جلب صور بعيدة عشوائية من متصفح المشغّل.

لا تحتاج إلى تغيير أي شيء للحصول على هذا السلوك — فهو دائم التشغيل وغير قابل للتهيئة.

## مصادقة مسار الصورة الرمزية

عندما تكون مصادقة gateway مهيأة، تتطلب نقطة نهاية الصورة الرمزية في Control UI نفس token الخاص بـ gateway مثل بقية API:

- يعيد `GET /avatar/<agentId>` صورة الصورة الرمزية فقط للمتصلين الموثقين. ويعيد `GET /avatar/<agentId>?meta=1` البيانات الوصفية للصورة الرمزية وفق القاعدة نفسها.
- تُرفض الطلبات غير الموثقة إلى كلا المسارين (بما يطابق مسار الوسائط الخاص بالمساعد المجاور). وهذا يمنع مسار الصورة الرمزية من تسريب هوية الوكيل على المضيفات المحمية بخلاف ذلك.
- تقوم Control UI نفسها بتمرير token الخاص بـ gateway بصفته bearer header عند جلب الصور الرمزية، وتستخدم blob URLs موثقة بحيث تظل الصورة قابلة للعرض داخل لوحات المعلومات.

إذا عطلت مصادقة gateway ‏(وهو غير موصى به على المضيفات المشتركة)، فسيصبح مسار الصورة الرمزية أيضًا غير موثق، بما يتماشى مع بقية gateway.

## بناء واجهة المستخدم

يقدّم Gateway الملفات الثابتة من `dist/control-ui`. ابنِها باستخدام:

```bash
pnpm ui:build
```

قاعدة مطلقة اختيارية (عندما تريد URLs ثابتة للأصول):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

للتطوير المحلي (خادم تطوير منفصل):

```bash
pnpm ui:dev
```

ثم وجّه واجهة المستخدم إلى عنوان Gateway WS لديك (مثل `ws://127.0.0.1:18789`).

## التصحيح/الاختبار: خادم تطوير + Gateway بعيد

تتكون Control UI من ملفات ثابتة؛ كما أن هدف WebSocket قابل للتهيئة ويمكن أن يكون
مختلفًا عن أصل HTTP. ويعد هذا مفيدًا عندما تريد خادم Vite للتطوير محليًا
لكن Gateway يعمل في مكان آخر.

1. ابدأ خادم تطوير واجهة المستخدم: `pnpm ui:dev`
2. افتح عنوان URL مثل:

```text
http://localhost:5173/?gatewayUrl=ws://<gateway-host>:18789
```

مصادقة اختيارية لمرة واحدة (إذا لزم الأمر):

```text
http://localhost:5173/?gatewayUrl=wss://<gateway-host>:18789#token=<gateway-token>
```

ملاحظات:

- يُخزَّن `gatewayUrl` في localStorage بعد التحميل ويُزال من عنوان URL.
- ينبغي تمرير `token` عبر جزء الرابط (`#token=...`) كلما أمكن. إذ لا تُرسَل الأجزاء إلى الخادم، مما يتجنب تسرب سجلات الطلبات وReferer. ولا تزال معاملات الاستعلام القديمة `?token=` تُستورد مرة واحدة للتوافق، ولكن فقط كخيار احتياطي، وتُزال فورًا بعد bootstrap.
- تُحفَظ `password` في الذاكرة فقط.
- عندما يكون `gatewayUrl` مضبوطًا، لا تعود واجهة المستخدم إلى بيانات الاعتماد الموجودة في التهيئة أو البيئة.
  قدّم `token` (أو `password`) صراحةً. وغياب بيانات اعتماد صريحة يمثل خطأ.
- استخدم `wss://` عندما يكون Gateway خلف TLS ‏(مثل Tailscale Serve أو HTTPS proxy، إلخ).
- لا يُقبل `gatewayUrl` إلا في نافذة من المستوى الأعلى (وليس عندما يكون مضمّنًا) لمنع clickjacking.
- يجب على عمليات نشر Control UI غير المعتمدة على loopback أن تضبط `gateway.controlUi.allowedOrigins`
  صراحةً (أصولًا كاملة). ويشمل هذا إعدادات التطوير البعيدة.
- لا تستخدم `gateway.controlUi.allowedOrigins: ["*"]` إلا في اختبارات
  محلية خاضعة لتحكم محكم. فهي تعني السماح لأي أصل متصفح، وليس "مطابقة أي مضيف أستخدمه".
- يفعّل `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true`
  وضع fallback المعتمد على أصل Host-header، لكنه وضع أمني خطير.

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

تفاصيل إعداد الوصول البعيد: [الوصول البعيد](/ar/gateway/remote).
