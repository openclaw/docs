يعمل مسار CI عند كل عملية push إلى `main` وكل طلب سحب. ويستخدم تحديد نطاق ذكيًا لتخطي المهام المكلفة عندما تكون التغييرات محصورة في مناطق غير ذات صلة.

يمتلك QA Lab مسارات CI مخصصة خارج سير العمل الرئيسي ذي النطاق الذكي. يعمل سير العمل `Parity gate` عند تغييرات PR المطابقة وعند التشغيل اليدوي؛ إذ يبني بيئة تشغيل QA الخاصة ويقارن حزم الوكلاء الوهمية GPT-5.4 وOpus 4.6. ويعمل سير العمل `QA-Lab - All Lanes` ليلًا على `main` وعند التشغيل اليدوي؛ ويُوزِّع مهمة parity gate الوهمية، ومسار Matrix الحي، ومسار Telegram الحي كمهام متوازية. تستخدم المهام الحية بيئة `qa-live-shared`، ويستخدم مسار Telegram عقود Convex. كما يشغّل `OpenClaw Release Checks` مسارات QA Lab نفسها قبل الموافقة على الإصدار.

إن سير العمل `Duplicate PRs After Merge` هو سير عمل يدوي للمشرفين لتنظيف التكرارات بعد الدمج. ويستخدم الوضع التجريبي افتراضيًا، ولا يغلق إلا طلبات السحب المدرجة صراحةً عندما تكون القيمة `apply=true`. وقبل إجراء تغييرات على GitHub، يتحقق من أن PR الذي هبط قد تم دمجه وأن كل PR مكرر لديه إما issue مشتركة مُشار إليها أو تداخل في المقاطع المعدلة.

إن سير العمل `Docs Agent` هو مسار صيانة Codex قائم على الأحداث للحفاظ على توافق المستندات الحالية مع التغييرات التي هبطت مؤخرًا. ولا يملك جدولًا زمنيًا مستقلًا: يمكن لنجاح تشغيل CI لعملية push غير صادرة عن bot على `main` أن يفعّله، كما يمكن للتشغيل اليدوي أن يشغله مباشرةً. وتتخطى استدعاءات workflow-run التنفيذ عندما يكون `main` قد تقدم، أو عندما يكون قد تم إنشاء تشغيل `Docs Agent` آخر غير متخطى خلال الساعة الماضية. وعندما يعمل، فإنه يراجع نطاق الالتزامات من SHA المصدر الخاص بآخر تشغيل `Docs Agent` غير متخطى حتى `main` الحالي، بحيث يمكن لتشغيل واحد كل ساعة أن يغطي جميع تغييرات main المتراكمة منذ آخر تمريرة للمستندات.

إن سير العمل `Test Performance Agent` هو مسار صيانة Codex قائم على الأحداث للاختبارات البطيئة. ولا يملك جدولًا زمنيًا مستقلًا: يمكن لنجاح تشغيل CI لعملية push غير صادرة عن bot على `main` أن يفعّله، لكنه يتخطى التنفيذ إذا كان استدعاء workflow-run آخر قد تم تشغيله بالفعل أو لا يزال قيد التشغيل في ذلك اليوم وفق UTC. أما التشغيل اليدوي فيتجاوز بوابة النشاط اليومية تلك. يقوم هذا المسار ببناء تقرير أداء Vitest مجمّع لمجموعة الاختبارات الكاملة، ويسمح لـ Codex بإجراء إصلاحات صغيرة فقط لأداء الاختبارات مع الحفاظ على التغطية بدلًا من إعادة هيكلة واسعة، ثم يعيد تشغيل تقرير المجموعة الكاملة ويرفض التغييرات التي تقلل عدد الاختبارات الناجحة في خط الأساس. وإذا كان خط الأساس يحتوي على اختبارات فاشلة، فقد يصلح Codex الإخفاقات الواضحة فقط، ويجب أن ينجح تقرير المجموعة الكاملة بعد الوكيل قبل اعتماد أي شيء. وعندما يتقدم `main` قبل هبوط push الخاص بالروبوت، يعيد هذا المسار تأسيس الرقعة المتحقق منها فوق الأساس الجديد، ويعيد تشغيل `pnpm check:changed`، ثم يعيد محاولة الـ push؛ أما الرقع القديمة المتعارضة فيتم تخطيها. ويستخدم Ubuntu المستضاف من GitHub حتى يتمكن إجراء Codex من الحفاظ على وضع أمان drop-sudo نفسه كما في وكيل المستندات.

```bash
gh workflow run duplicate-after-merge.yml \
  -f landed_pr=70532 \
  -f duplicate_prs='70530,70592' \
  -f apply=true
```

## ترتيب الإخفاق السريع

إن `extension-fast` مخصص لطلبات السحب فقط لأن عمليات push تشغّل بالفعل شرائح bundled plugin الكاملة. وهذا يحافظ على تغذية راجعة سريعة للـ plugin المتغير أثناء المراجعات من دون حجز عامل Blacksmith إضافي على `main` لتغطية موجودة أصلًا ضمن `checks-node-extensions`.

قد يضع GitHub علامة `cancelled` على المهام التي تم الاستغناء عنها عندما يصل push أحدث إلى PR نفسه أو المرجع `main` نفسه. تعامل مع ذلك على أنه ضوضاء CI ما لم يكن أحدث تشغيل للمرجع نفسه يفشل أيضًا. وتستخدم فحوصات الشرائح المجمعة `!cancelled() && always()` حتى تستمر في الإبلاغ عن إخفاقات الشرائح العادية، لكنها لا تدخل الطابور بعد أن يكون سير العمل بالكامل قد تم الاستغناء عنه بالفعل.
ومفتاح التزامن في CI مُرقَّم بالإصدار (`CI-v7-*`) حتى لا يتمكن zombie على جانب GitHub في مجموعة طوابير قديمة من حجب تشغيلات main الأحدث إلى أجل غير مسمى.

| العامل | المهام |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ubuntu-24.04` | `preflight`، ومهام الأمان السريعة والمُجمِّعات (`security-scm-fast` و`security-dependency-audit` و`security-fast`)، وفحوصات protocol/contract/bundled السريعة، وفحوصات channel contract المجزأة، وشرائح `check` باستثناء lint، وشرائح ومُجمِّعات `check-additional`، ومُتحقِّقات تجميع اختبارات Node، وفحوصات المستندات، وPython Skills، وworkflow-sanity، وlabeler، وauto-response؛ كما يستخدم preflight الخاص بـ install-smoke أيضًا Ubuntu المستضاف من GitHub حتى تتمكن مصفوفة Blacksmith من الدخول إلى الطابور بشكل أبكر |

## المكافئات المحلية

```bash
pnpm changed:lanes   # افحص مصنّف المسارات المحلية المتغيرة لـ origin/main...HEAD
pnpm check:changed   # البوابة المحلية الذكية: typecheck/lint/tests المتغيرة حسب مسار الحدود
pnpm check          # البوابة المحلية السريعة: `tsgo` للإنتاج + lint مُجزأ + حواجز سريعة متوازية
pnpm check:test-types
pnpm check:timed    # البوابة نفسها مع توقيتات لكل مرحلة
pnpm build:strict-smoke
pnpm check:architecture
pnpm test:gateway:watch-regression
pnpm test           # اختبارات vitest
pnpm test:channels
pnpm test:contracts:channels
pnpm check:docs     # تنسيق المستندات + lint + الروابط المعطلة
pnpm build          # ابنِ dist عندما تكون مسارات عناصر CI المبنية/ build-smoke مهمة
node scripts/ci-run-timings.mjs <run-id>      # لخّص زمن التنفيذ الكلي، وزمن الانتظار في الطابور، وأبطأ المهام
node scripts/ci-run-timings.mjs --recent 10   # قارن تشغيلات main CI الناجحة الحديثة
pnpm test:perf:groups --full-suite --allow-failures --output .artifacts/test-perf/baseline-before.json
pnpm test:perf:groups:compare .artifacts/test-perf/baseline-before.json .artifacts/test-perf/after-agent.json
```
