عند عمليات الدفع، تضيف مصفوفة `checks` المسار الخاص بالدفع فقط `compat-node22`. وعند طلبات السحب، يُتخطى ذلك المسار وتبقى المصفوفة مركزة على مسارات الاختبار/القنوات العادية.

تُقسَّم أو تُوازَن أبطأ عائلات اختبارات Node بحيث تظل كل مهمة صغيرة دون المبالغة في حجز المشغلات: تعمل عقود القنوات كثلاثة أجزاء موزونة، وتوازن اختبارات bundled plugin عبر ستة عمال للإضافات، وتُزاوَج مسارات وحدات core الصغيرة، ويعمل auto-reply عبر ثلاثة عمال متوازنين بدلًا من ستة عمال صغار، وتُوزَّع إعدادات agentic gateway/plugin عبر مهام agentic Node الموجودة الخاصة بالمصدر فقط بدلًا من انتظار built artifacts. وتستخدم اختبارات browser وQA وmedia وPlugin المتنوعة الواسعة إعدادات Vitest المخصصة لها بدلًا من shared plugin catch-all. وتشغّل مهام أجزاء extension ما يصل إلى مجموعتي إعدادات Plugin في الوقت نفسه مع عامل Vitest واحد لكل مجموعة وكومة Node أكبر حتى لا تنشئ دفعات Plugin الثقيلة من حيث الاستيراد مهام CI إضافية. ويستخدم مسار agents الواسع مجدول Vitest المشترك المتوازي على مستوى الملفات لأنه تهيمن عليه الاستيرادات/الجدولة بدلًا من ملف اختبار بطيء واحد. ويعمل `runtime-config` مع جزء infra core-runtime حتى لا يمتلك جزء runtime المشترك ذيل التنفيذ. ويحافظ `check-additional` على تجميع أعمال package-boundary compile/canary معًا ويفصل architecture الخاصة بـ runtime topology عن تغطية gateway watch؛ ويشغّل جزء boundary guard حواجزه الصغيرة المستقلة بالتوازي داخل مهمة واحدة. وتعمل gateway watch واختبارات القنوات وجزء core support-boundary بالتوازي داخل `build-artifacts` بعد أن يكون قد تم بناء `dist/` و`dist-runtime/` بالفعل، مع الاحتفاظ بأسماء الفحوصات القديمة بوصفها مهام تحقق خفيفة مع تجنب عاملَي Blacksmith إضافيين وطابور ثانٍ لمستهلكي artifacts.
يشغّل Android CI كلًا من `testPlayDebugUnitTest` و`testThirdPartyDebugUnitTest`، ثم يبني Play debug APK. ولا تملك نكهة third-party مجموعة مصادر أو manifest منفصلين؛ ومع ذلك فإن مسار اختبارات الوحدات الخاص بها ما يزال يصرّف تلك النكهة مع أعلام BuildConfig الخاصة بـ SMS/call-log، مع تجنب مهمة تغليف مكررة لـ debug APK عند كل دفع ذي صلة بـ Android.
ويكون `extension-fast` خاصًا بطلبات السحب فقط لأن عمليات الدفع تشغّل بالفعل أجزاء bundled plugin الكاملة. وهذا يحافظ على تغذية راجعة لتغييرات Plugin أثناء المراجعة دون حجز عامل Blacksmith إضافي على `main` لتغطية موجودة أصلًا في `checks-node-extensions`.

قد يضع GitHub علامة `cancelled` على المهام التي تم الاستعاضة عنها عندما يصل دفع أحدث إلى طلب السحب نفسه أو إلى المرجع `main`. تعامل مع ذلك على أنه ضجيج CI ما لم يكن أحدث تشغيل للمرجع نفسه يفشل أيضًا. وتستخدم فحوصات تجميع الأجزاء `!cancelled() && always()` بحيث تستمر في الإبلاغ عن إخفاقات الأجزاء العادية لكنها لا تُدرج في الطابور بعد أن يكون سير العمل بأكمله قد تم الاستعاضة عنه بالفعل.
مفتاح التزامن في CI مُرقّم بالإصدار (`CI-v7-*`) حتى لا يتمكن zombie على جانب GitHub ضمن مجموعة طابور قديمة من حظر تشغيلات main الأحدث إلى أجل غير مسمى.

## المكافئات المحلية

```bash
pnpm changed:lanes   # فحص مصنف المسارات المتغيرة المحلي لـ origin/main...HEAD
pnpm check:changed   # البوابة المحلية الذكية: typecheck/lint/tests المتغيرة حسب مسار الحدود
pnpm check          # البوابة المحلية السريعة: production tsgo + lint مجزأ + حواجز سريعة متوازية
pnpm check:test-types
pnpm check:timed    # البوابة نفسها مع توقيتات لكل مرحلة
pnpm build:strict-smoke
pnpm check:architecture
pnpm test:gateway:watch-regression
pnpm test           # اختبارات vitest
pnpm test:channels
pnpm test:contracts:channels
pnpm check:docs     # تنسيق المستندات + lint + الروابط المعطلة
pnpm build          # بناء dist عندما تكون مسارات CI الخاصة بـ artifact/build-smoke مهمة
node scripts/ci-run-timings.mjs <run-id>      # تلخيص وقت التنفيذ الفعلي، ووقت الانتظار في الطابور، وأبطأ المهام
node scripts/ci-run-timings.mjs --recent 10   # مقارنة تشغيلات CI الحديثة الناجحة على main
pnpm test:perf:groups --full-suite --allow-failures --output .artifacts/test-perf/baseline-before.json
pnpm test:perf:groups:compare .artifacts/test-perf/baseline-before.json .artifacts/test-perf/after-agent.json
```
