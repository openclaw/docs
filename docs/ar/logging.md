لدى OpenClaw سطحان رئيسيان للسجلات:

- **سجلات الملفات** (أسطر JSON) يكتبها Gateway.
- **مخرجات وحدة التحكم** المعروضة في الطرفيات وواجهة تصحيح أخطاء Gateway.

تعرض علامة تبويب **السجلات** في واجهة Control UI ذيل سجل ملف Gateway. تشرح هذه الصفحة أين
توجد السجلات، وكيفية قراءتها، وكيفية إعداد مستويات السجلات وتنسيقاتها.

## أين توجد السجلات

افتراضيًا، يكتب Gateway ملف سجل دوريًا ضمن:

`/tmp/openclaw/openclaw-YYYY-MM-DD.log`

يستخدم التاريخ المنطقة الزمنية المحلية لمضيف Gateway.

يدور كل ملف عندما يصل إلى `logging.maxFileBytes` (الافتراضي: 100 ميغابايت).
يحتفظ OpenClaw بما يصل إلى خمسة أرشيفات مرقمة بجانب الملف النشط، مثل
`openclaw-YYYY-MM-DD.1.log`، ويستمر في الكتابة إلى سجل نشط جديد بدلًا من
منع التشخيصات.

يمكنك تجاوز ذلك في `~/.openclaw/openclaw.json`:

```json
{
  "logging": {
    "file": "/path/to/openclaw.log"
  }
}
```

## كيفية قراءة السجلات

### CLI: متابعة مباشرة للذيل (موصى بها)

استخدم CLI لمتابعة ذيل ملف سجل Gateway عبر RPC:

```bash
openclaw logs --follow
```

خيارات حالية مفيدة:

- `--local-time`: عرض الطوابع الزمنية بمنطقتك الزمنية المحلية
- `--url <url>` / `--token <token>` / `--timeout <ms>`: أعلام RPC القياسية لـ Gateway
- `--expect-final`: علم انتظار الاستجابة النهائية لـ RPC المدعوم بالوكيل (مقبول هنا عبر طبقة العميل المشتركة)

أوضاع الإخراج:

- **جلسات TTY**: أسطر سجل منظمة، منسقة وملونة.
- **جلسات غير TTY**: نص عادي.
- `--json`: JSON مفصول بأسطر (حدث سجل واحد لكل سطر).
- `--plain`: فرض النص العادي في جلسات TTY.
- `--no-color`: تعطيل ألوان ANSI.

عندما تمرر `--url` صريحًا، لا يطبق CLI تلقائيًا الإعدادات أو
بيانات اعتماد البيئة؛ أدرج `--token` بنفسك إذا كان Gateway الهدف
يتطلب المصادقة.

في وضع JSON، يصدر CLI كائنات موسومة بـ `type`:

- `meta`: بيانات تعريف التدفق (الملف، المؤشر، الحجم)
- `log`: إدخال سجل محلل
- `notice`: تلميحات اقتطاع / دوران
- `raw`: سطر سجل غير محلل

إذا طلب Gateway الضمني عبر local loopback الاقتران، أو أغلق أثناء الاتصال،
أو انتهت المهلة قبل أن يجيب `logs.tail`، يرجع `openclaw logs` تلقائيًا إلى
ملف سجل Gateway المعد. لا تستخدم أهداف `--url` الصريحة
هذا الرجوع الاحتياطي.

إذا تعذر الوصول إلى Gateway، يطبع CLI تلميحًا قصيرًا لتشغيل:

```bash
openclaw doctor
```

### Control UI (الويب)

تعرض علامة تبويب **السجلات** في Control UI ذيل الملف نفسه باستخدام `logs.tail`.
راجع [Control UI](/ar/web/control-ui) لمعرفة كيفية فتحها.

### سجلات القنوات فقط

لتصفية نشاط القنوات (WhatsApp/Telegram/إلخ)، استخدم:

```bash
openclaw channels logs --channel whatsapp
```

## تنسيقات السجلات

### سجلات الملفات (JSONL)

كل سطر في ملف السجل هو كائن JSON. يحلل CLI وControl UI هذه
الإدخالات لعرض إخراج منظم (الوقت، المستوى، النظام الفرعي، الرسالة).

تتضمن سجلات JSONL الخاصة بالملفات أيضًا حقولًا علوية قابلة للتصفية آليًا عندما
تكون متاحة:

- `hostname`: اسم مضيف Gateway.
- `message`: نص رسالة السجل المسطح للبحث النصي الكامل.
- `agent_id`: معرّف الوكيل النشط عندما يحمل نداء السجل سياق الوكيل.
- `session_id`: معرّف/مفتاح الجلسة النشطة عندما يحمل نداء السجل سياق الجلسة.
- `channel`: القناة النشطة عندما يحمل نداء السجل سياق القناة.

يحافظ OpenClaw على وسيطات السجل المنظمة الأصلية بجانب هذه الحقول
كي تستمر المحللات الحالية التي تقرأ مفاتيح وسيطات tslog المرقمة في العمل.

### مخرجات وحدة التحكم

سجلات وحدة التحكم **مدركة لـ TTY** ومنسقة لقابلية القراءة:

- بادئات الأنظمة الفرعية (مثل `gateway/channels/whatsapp`)
- تلوين المستويات (info/warn/error)
- وضع مضغوط أو JSON اختياري

يتحكم `logging.consoleStyle` في تنسيق وحدة التحكم.

### سجلات WebSocket الخاصة بـ Gateway

يتضمن `openclaw gateway` أيضًا تسجيل بروتوكول WebSocket لحركة RPC:

- الوضع العادي: النتائج المهمة فقط (الأخطاء، أخطاء التحليل، النداءات البطيئة)
- `--verbose`: كل حركة الطلب/الاستجابة
- `--ws-log auto|compact|full`: اختيار نمط العرض المطول
- `--compact`: اسم بديل لـ `--ws-log compact`

أمثلة:

```bash
openclaw gateway
openclaw gateway --verbose --ws-log compact
openclaw gateway --verbose --ws-log full
```

## إعداد التسجيل

توجد كل إعدادات التسجيل ضمن `logging` في `~/.openclaw/openclaw.json`.

```json
{
  "logging": {
    "level": "info",
    "file": "/tmp/openclaw/openclaw-YYYY-MM-DD.log",
    "consoleLevel": "info",
    "consoleStyle": "pretty",
    "redactSensitive": "tools",
    "redactPatterns": ["sk-.*"]
  }
}
```

### مستويات السجلات

- `logging.level`: مستوى **سجلات الملفات** (JSONL).
- `logging.consoleLevel`: مستوى إسهاب **وحدة التحكم**.

يمكنك تجاوز كليهما عبر متغير البيئة **`OPENCLAW_LOG_LEVEL`** (مثل `OPENCLAW_LOG_LEVEL=debug`). تكون لمتغير البيئة أولوية على ملف الإعدادات، لذا يمكنك رفع الإسهاب لتشغيل واحد دون تعديل `openclaw.json`. يمكنك أيضًا تمرير خيار CLI العام **`--log-level <level>`** (على سبيل المثال، `openclaw --log-level debug gateway run`)، والذي يتجاوز متغير البيئة لهذا الأمر.

لا يؤثر `--verbose` إلا في مخرجات وحدة التحكم وإسهاب سجل WS؛ ولا يغير
مستويات سجل الملفات.

### ربط التتبعات

سجلات الملفات هي JSONL. عندما يحمل نداء سجل سياق تتبع تشخيصيًا صالحًا،
يكتب OpenClaw حقول التتبع كمفاتيح JSON علوية (`traceId`, `spanId`,
`parentSpanId`, `traceFlags`) كي تتمكن معالجات السجلات الخارجية من ربط السطر
بامتدادات OTEL ونشر `traceparent` الخاص بالموفر.

تنشئ طلبات HTTP الخاصة بـ Gateway وإطارات WebSocket الخاصة بـ Gateway نطاق تتبع طلب
داخليًا. ترث السجلات والأحداث التشخيصية الصادرة داخل ذلك النطاق غير المتزامن
تتبع الطلب عندما لا تمرر سياق تتبع صريحًا. تصبح تتبعات تشغيل الوكيل
ونداءات النموذج أبناء لتتبع الطلب النشط، لذا يمكن ربط السجلات المحلية،
ولقطات التشخيص، وامتدادات OTEL، وترويسات `traceparent` الموثوقة الخاصة بالموفر
بواسطة `traceId` دون تسجيل محتوى الطلب الخام أو محتوى النموذج.

### حجم نداء النموذج وتوقيته

تسجل تشخيصات نداءات النموذج قياسات محدودة للطلب/الاستجابة دون
التقاط محتوى الموجه أو الاستجابة الخام:

- `requestPayloadBytes`: حجم حمولة طلب النموذج النهائي بالبايت UTF-8
- `responseStreamBytes`: حجم أحداث استجابة النموذج المتدفقة بالبايت UTF-8
- `timeToFirstByteMs`: الوقت المنقضي قبل أول حدث استجابة متدفق
- `durationMs`: مدة نداء النموذج الإجمالية

تتوفر هذه الحقول للقطات التشخيص، وخطافات Plugin لنداءات النموذج، وامتدادات/مقاييس
نداءات النموذج في OTEL عند تمكين تصدير التشخيصات.

### أنماط وحدة التحكم

`logging.consoleStyle`:

- `pretty`: مناسب للبشر، ملون، مع طوابع زمنية.
- `compact`: إخراج أضيق (الأفضل للجلسات الطويلة).
- `json`: JSON لكل سطر (لمعالجات السجلات).

### التنقيح

يمكن لـ OpenClaw تنقيح الرموز الحساسة قبل وصولها إلى مخرجات وحدة التحكم، أو سجلات الملفات،
أو سجلات OTLP، أو نصوص محاضر الجلسات المستمرة، أو حمولات أحداث أدوات
Control UI (وسيطات بدء الأداة، حمولات النتائج الجزئية/النهائية، مخرجات
exec المشتقة، وملخصات التصحيحات):

- `logging.redactSensitive`: `off` | `tools` (الافتراضي: `tools`)
- `logging.redactPatterns`: قائمة سلاسل regex لتجاوز المجموعة الافتراضية. تطبق الأنماط المخصصة فوق الافتراضات المدمجة لحمولات أدوات Control UI، لذا فإن إضافة نمط لا تضعف أبدًا تنقيح القيم التي تلتقطها الافتراضات بالفعل.

تبقى سجلات الملفات ومحاضر الجلسات بتنسيق JSONL، لكن القيم السرية المطابقة
تُحجب قبل كتابة السطر أو الرسالة إلى القرص. التنقيح قائم على أفضل جهد:
ينطبق على محتوى الرسائل النصية وسلاسل السجلات، وليس على كل
معرّف أو حقل حمولة ثنائية.

تغطي الافتراضات المدمجة بيانات اعتماد API الشائعة وأسماء حقول بيانات اعتماد الدفع
مثل رقم البطاقة، وCVC/CVV، ورمز الدفع المشترك، وبيانات اعتماد الدفع
عندما تظهر كحقول JSON، أو معاملات URL، أو أعلام CLI، أو إسنادات.

يعطل `logging.redactSensitive: "off"` سياسة السجلات/المحاضر العامة هذه فقط.
لا يزال OpenClaw ينقح حمولات حدود السلامة التي يمكن عرضها لعملاء UI،
أو حزم الدعم، أو مراقبي التشخيصات، أو مطالبات الموافقة، أو أدوات الوكيل.
تشمل الأمثلة أحداث نداءات الأدوات في Control UI، ومخرجات `sessions_history`،
وتصديرات دعم التشخيصات، وملاحظات أخطاء الموفر، وعرض أمر موافقة exec،
وسجلات بروتوكول WebSocket الخاصة بـ Gateway. لا تزال `logging.redactPatterns`
المخصصة قادرة على إضافة أنماط خاصة بالمشروع على تلك الأسطح.

## التشخيصات وOpenTelemetry

التشخيصات هي أحداث منظمة قابلة للقراءة آليًا لتشغيلات النماذج وقياس تدفق
الرسائل (webhooks، الانتظار في الطابور، حالة الجلسة). وهي **لا**
تستبدل السجلات — بل تغذي المقاييس والتتبعات والمصدرين. تصدر الأحداث
داخل العملية سواء صدرتها أم لا.

سطحان متجاوران:

- **تصدير OpenTelemetry** — إرسال المقاييس والتتبعات والسجلات عبر OTLP/HTTP إلى
  أي جامع أو واجهة خلفية متوافقة مع OpenTelemetry (Grafana، Datadog،
  Honeycomb، New Relic، Tempo، إلخ). توجد الإعدادات الكاملة، وفهرس الإشارات،
  وأسماء المقاييس/الامتدادات، ومتغيرات البيئة، ونموذج الخصوصية في صفحة مخصصة:
  [تصدير OpenTelemetry](/ar/gateway/opentelemetry).
- **أعلام التشخيصات** — أعلام سجلات تصحيح أخطاء موجهة توجه سجلات إضافية إلى
  `logging.file` دون رفع `logging.level`. الأعلام غير حساسة لحالة الأحرف
  وتدعم أحرف البدل (`telegram.*`, `*`). اضبطها ضمن `diagnostics.flags`
  أو عبر تجاوز متغير البيئة `OPENCLAW_DIAGNOSTICS=...`. الدليل الكامل:
  [أعلام التشخيصات](/ar/diagnostics/flags).

لتمكين أحداث التشخيصات لـ plugins أو المصارف المخصصة دون تصدير OTLP:

```json5
{
  diagnostics: { enabled: true },
}
```

لتصدير OTLP إلى جامع، راجع [تصدير OpenTelemetry](/ar/gateway/opentelemetry).

## نصائح استكشاف الأخطاء وإصلاحها

- **Gateway غير قابل للوصول؟** شغل `openclaw doctor` أولًا.
- **السجلات فارغة؟** تحقق من أن Gateway يعمل ويكتب إلى مسار الملف
  في `logging.file`.
- **تحتاج إلى تفاصيل أكثر؟** اضبط `logging.level` على `debug` أو `trace` وأعد المحاولة.
