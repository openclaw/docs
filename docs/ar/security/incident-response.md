# الاستجابة للحوادث

## 1. الاكتشاف والفرز

نراقب إشارات الأمان من:

- GitHub Security Advisories (GHSA) وتقارير الثغرات الخاصة.
- مشكلات/مناقشات GitHub العامة عندما لا تكون التقارير حساسة.
- الإشارات الآلية (مثل Dependabot وCodeQL وتنبيهات npm وفحص الأسرار).

الفرز الأولي:

1. تأكيد المكوّن المتأثر والإصدار وتأثير حد الثقة.
2. تصنيفها كمسألة أمنية أو تعزيز أمان/لا إجراء باستخدام نطاق `SECURITY.md` في المستودع وقواعد ما هو خارج النطاق.
3. يرد مالك الحادث وفقًا لذلك.

## 2. التقييم

دليل الشدة:

- **حرجة:** اختراق الحزمة/الإصدار/المستودع، أو استغلال نشط، أو تجاوز حد الثقة دون مصادقة مع تحكم عالي التأثير أو كشف للبيانات.
- **عالية:** تجاوز مؤكد لحد الثقة يتطلب شروطًا مسبقة محدودة (مثل إجراء عالي التأثير من مستخدم مصادق عليه لكنه غير مصرّح له)، أو كشف بيانات اعتماد حساسة مملوكة لـ OpenClaw.
- **متوسطة:** ضعف أمني مهم له تأثير عملي لكن قابلية استغلاله مقيدة أو يتطلب متطلبات مسبقة كبيرة.
- **منخفضة:** نتائج دفاع متعمق، أو حجب خدمة محدود النطاق، أو فجوات تعزيز أمان/تكافؤ دون تجاوز مثبت لحد الثقة.

## 3. الاستجابة

1. الإقرار باستلام التقرير للمبلّغ (بشكل خاص عندما يكون حساسًا).
2. إعادة الإنتاج على الإصدارات المدعومة وأحدث `main`، ثم تنفيذ تصحيح والتحقق منه مع تغطية ارتدادية.
3. بالنسبة للحوادث الحرجة/العالية، جهّز إصدارًا/إصدارات مصححة بأسرع ما يمكن عمليًا.
4. بالنسبة للحوادث المتوسطة/المنخفضة، طبّق التصحيح ضمن مسار الإصدار العادي ووثّق إرشادات التخفيف.

## 4. التواصل

نتواصل عبر:

- GitHub Security Advisories في المستودع المتأثر.
- ملاحظات الإصدار/إدخالات سجل التغييرات للإصدارات المصححة.
- متابعة مباشرة مع المبلّغ بشأن الحالة والحل.

سياسة الإفصاح:

- ينبغي أن تحصل الحوادث الحرجة/العالية على إفصاح منسّق، مع إصدار CVE عند الاقتضاء.
- يمكن توثيق نتائج تعزيز الأمان منخفضة المخاطر في ملاحظات الإصدار أو التنبيهات دون CVE، حسب التأثير وتعرض المستخدمين.

## 5. الاسترداد والمتابعة

بعد شحن الإصلاح:

1. تحقق من المعالجات في CI ومخرجات الإصدار.
2. أجرِ مراجعة قصيرة بعد الحادث (الجدول الزمني، السبب الجذري، فجوة الاكتشاف، خطة الوقاية).
3. أضف مهام متابعة لتعزيز الأمان/الاختبارات/الوثائق وتتبعها حتى الاكتمال.
