# `openclaw cron`

إدارة مهام Cron لمُجدول Gateway.

<Tip>
شغّل `openclaw cron --help` للاطلاع على كامل واجهة الأوامر. راجع [مهام Cron](/ar/automation/cron-jobs) للدليل المفاهيمي.
</Tip>

## الجلسات

يقبل `--session` القيم `main` أو `isolated` أو `current` أو `session:<id>`.

<AccordionGroup>
  <Accordion title="مفاتيح الجلسات">
    - تربط `main` بجلسة الوكيل الرئيسية.
    - تنشئ `isolated` نصًا حواريًا جديدًا ومعرّف جلسة جديدًا لكل تشغيل.
    - تربط `current` بالجلسة النشطة وقت الإنشاء.
    - تثبّت `session:<id>` على مفتاح جلسة مستمرة صريح.

  </Accordion>
  <Accordion title="دلالات الجلسة المعزولة">
    تعيد التشغيلات المعزولة ضبط سياق المحادثة المحيط. يُعاد ضبط توجيه القنوات والمجموعات، وسياسة الإرسال/الانتظار في الطابور، والرفع، والأصل، وربط وقت تشغيل ACP للتشغيل الجديد. يمكن أن تنتقل التفضيلات الآمنة والتجاوزات الصريحة للنموذج أو المصادقة التي اختارها المستخدم عبر التشغيلات.
  </Accordion>
</AccordionGroup>

## التسليم

يعرض `openclaw cron list` و`openclaw cron show <job-id>` معاينة لمسار التسليم المحلول. بالنسبة إلى `channel: "last"`، تعرض المعاينة ما إذا كان المسار قد حُل من الجلسة الرئيسية أو الحالية، أو سيفشل مغلقًا.

<Note>
تكون مهام `cron add` المعزولة افتراضيًا على تسليم `--announce`. استخدم `--no-deliver` لإبقاء المخرجات داخلية. يبقى `--deliver` كاسم بديل مهمل لـ `--announce`.
</Note>

### ملكية التسليم

يتشارك الوكيل والمشغّل تسليم دردشة Cron المعزولة:

- يمكن للوكيل الإرسال مباشرة باستخدام أداة `message` عندما يكون مسار دردشة متاحًا.
- يسلّم `announce` الرد النهائي احتياطيًا فقط عندما لا يرسل الوكيل مباشرة إلى الهدف المحلول.
- ينشر `webhook` الحمولة المنتهية إلى URL.
- يعطّل `none` التسليم الاحتياطي من المشغّل.

`--announce` هو التسليم الاحتياطي من المشغّل للرد النهائي. يعطّل `--no-deliver` ذلك التسليم الاحتياطي، لكنه لا يزيل أداة `message` الخاصة بالوكيل عندما يكون مسار دردشة متاحًا.

تحافظ التذكيرات المُنشأة من دردشة نشطة على هدف تسليم الدردشة الحي لتسليم الإعلان الاحتياطي. قد تكون مفاتيح الجلسات الداخلية بأحرف صغيرة؛ لا تستخدمها كمصدر حقيقة لمعرّفات مزوّدين حساسة لحالة الأحرف مثل معرّفات غرف Matrix.

### تسليم الفشل

تُحل إشعارات الفشل بهذا الترتيب:

1. `delivery.failureDestination` في المهمة.
2. `cron.failureDestination` العام.
3. هدف الإعلان الأساسي للمهمة (عند عدم تعيين وجهة فشل صريحة).

<Note>
لا يجوز لمهام الجلسة الرئيسية استخدام `delivery.failureDestination` إلا عندما يكون وضع التسليم الأساسي هو `webhook`. تقبله المهام المعزولة في جميع الأوضاع.
</Note>

ملاحظة: تتعامل تشغيلات Cron المعزولة مع حالات فشل الوكيل على مستوى التشغيل كأخطاء مهمة حتى عندما
لا تُنتَج حمولة رد، لذلك تظل حالات فشل النموذج/المزوّد تزيد عدادات الأخطاء
وتطلق إشعارات الفشل.

## الجدولة

### المهام لمرة واحدة

يجدول `--at <datetime>` تشغيلًا لمرة واحدة. تُعامل التواريخ والأوقات بلا إزاحة على أنها UTC ما لم تمرر أيضًا `--tz <iana>`، والذي يفسر وقت الساعة المحلية في المنطقة الزمنية المعطاة.

<Note>
تُحذف المهام لمرة واحدة بعد النجاح افتراضيًا. استخدم `--keep-after-run` للاحتفاظ بها.
</Note>

### المهام المتكررة

تستخدم المهام المتكررة تراجع إعادة محاولة أُسّي بعد الأخطاء المتتالية: 30s، 1m، 5m، 15m، 60m. يعود الجدول إلى الوضع الطبيعي بعد التشغيل الناجح التالي.

تُتبع التشغيلات المتخطاة بشكل منفصل عن أخطاء التنفيذ. لا تؤثر في تراجع إعادة المحاولة، لكن يمكن لـ `openclaw cron edit <job-id> --failure-alert-include-skipped` تضمين تنبيهات الفشل في إشعارات متكررة عن التشغيلات المتخطاة.

بالنسبة إلى المهام المعزولة التي تستهدف مزوّد نماذج محليًا مهيأ، يجري Cron فحصًا تمهيديًا خفيفًا للمزوّد قبل بدء دور الوكيل. تُفحص مزوّدات Loopback والشبكات الخاصة و`.local` ذات `api: "ollama"` عند `/api/tags`؛ وتُفحص مزوّدات OpenAI-compatible المحلية مثل vLLM وSGLang وLM Studio عند `/models`. إذا تعذر الوصول إلى نقطة النهاية، يُسجل التشغيل على أنه `skipped` ويُعاد المحاولة في جدول لاحق؛ وتُخزن نقاط النهاية المعطلة المطابقة مؤقتًا لمدة 5 دقائق لتجنب أن تضغط مهام كثيرة على الخادم المحلي نفسه.

ملاحظة: تعيش تعريفات مهام Cron في `jobs.json`، بينما تعيش حالة وقت التشغيل المعلقة في `jobs-state.json`. إذا حُرر `jobs.json` خارجيًا، يعيد Gateway تحميل الجداول المتغيرة ويمسح الخانات المعلقة القديمة؛ ولا تمسح إعادات الكتابة التي تغيّر التنسيق فقط الخانة المعلقة.

### التشغيلات اليدوية

يعيد `openclaw cron run` النتيجة بمجرد وضع التشغيل اليدوي في الطابور. تتضمن الاستجابات الناجحة `{ ok: true, enqueued: true, runId }`. استخدم `openclaw cron runs --id <job-id>` لمتابعة النتيجة النهائية.

<Note>
يفرض `openclaw cron run <job-id>` التشغيل افتراضيًا. استخدم `--due` للاحتفاظ بسلوك "التشغيل فقط إذا كان مستحقًا" الأقدم.
</Note>

## النماذج

يحدد `cron add|edit --model <ref>` نموذجًا مسموحًا للمهمة.

<Warning>
إذا لم يكن النموذج مسموحًا أو تعذر حله، يفشل Cron التشغيل بخطأ تحقق صريح بدلًا من الرجوع إلى وكيل المهمة أو اختيار النموذج الافتراضي.
</Warning>

`--model` في Cron هو **أساسي للمهمة**، وليس تجاوز `/model` لجلسة دردشة. وهذا يعني:

- تظل احتياطات النماذج المهيأة مطبقة عندما يفشل نموذج المهمة المحدد.
- تستبدل حمولة كل مهمة `fallbacks` قائمة الاحتياطات المهيأة عند وجودها.
- تجعل قائمة احتياطات فارغة لكل مهمة (`fallbacks: []` في حمولة/API المهمة) تشغيل Cron صارمًا.
- عندما تحتوي مهمة على `--model` لكن لا توجد قائمة احتياطات مهيأة، يمرر OpenClaw تجاوز احتياطات فارغًا صريحًا كي لا يُضاف النموذج الأساسي للوكيل كهدف إعادة محاولة مخفي.

### أسبقية نموذج Cron المعزول

يحل Cron المعزول النموذج النشط بهذا الترتيب:

1. تجاوز Gmail-hook.
2. `--model` لكل مهمة.
3. تجاوز نموذج جلسة Cron المخزن (عندما يختار المستخدم واحدًا).
4. اختيار نموذج الوكيل أو النموذج الافتراضي.

### الوضع السريع

يتبع الوضع السريع في Cron المعزول اختيار النموذج الحي المحلول. تُطبق تهيئة النموذج `params.fastMode` افتراضيًا، لكن تجاوز الجلسة المخزن `fastMode` يظل له الأسبقية على التهيئة.

### إعادة المحاولة عند تبديل النموذج الحي

إذا ألقى تشغيل معزول `LiveSessionModelSwitchError`، يحفظ Cron المزوّد والنموذج اللذين تم التبديل إليهما (وتجاوز ملف المصادقة الذي تم التبديل إليه عند وجوده) للتشغيل النشط قبل إعادة المحاولة. حلقة إعادة المحاولة الخارجية محددة بمحاولتي تبديل بعد المحاولة الأولية، ثم تُجهض بدلًا من الدوران إلى الأبد.

## مخرجات التشغيل والرفض

### كبت إقرارات الاستلام القديمة

تكبت أدوار Cron المعزولة الردود القديمة التي لا تحتوي إلا على إقرار. إذا كانت النتيجة الأولى مجرد تحديث حالة مؤقت ولا يوجد تشغيل وكيل فرعي تابع مسؤول عن الإجابة النهائية، يعيد Cron المطالبة مرة واحدة للحصول على النتيجة الحقيقية قبل التسليم.

### كبت الرمز الصامت

إذا أعاد تشغيل Cron معزول الرمز الصامت فقط (`NO_REPLY` أو `no_reply`)، يكبت Cron كلًا من التسليم الصادر المباشر ومسار الملخص الاحتياطي الموضوع في الطابور، فلا يُنشر أي شيء مرة أخرى إلى الدردشة.

### الرفض المنظم

تفضل تشغيلات Cron المعزولة بيانات وصفية منظمة لرفض التنفيذ من التشغيل المضمّن، ثم ترجع إلى علامات الرفض المعروفة في المخرجات النهائية، مثل `SYSTEM_RUN_DENIED` و`INVALID_REQUEST` وعبارات الرفض المرتبطة بالموافقة.

يعرض `cron list` وسجل التشغيل سبب الرفض بدلًا من الإبلاغ عن أمر محظور على أنه `ok`.

## الاحتفاظ

يُتحكم في الاحتفاظ والتنظيف في التهيئة:

- يزيل `cron.sessionRetention` (الافتراضي `24h`) جلسات التشغيل المعزولة المكتملة.
- يزيل `cron.runLog.maxBytes` و`cron.runLog.keepLines` من `~/.openclaw/cron/runs/<jobId>.jsonl`.

## ترحيل المهام الأقدم

<Note>
إذا كانت لديك مهام Cron من قبل تنسيق التسليم والتخزين الحالي، فشغّل `openclaw doctor --fix`. تقوم أداة `doctor` بتطبيع حقول Cron القديمة (`jobId` و`schedule.cron` وحقول التسليم ذات المستوى الأعلى بما في ذلك `threadId` القديم، وأسماء التسليم البديلة `provider` في الحمولة) وترحّل مهام Webhook الاحتياطية البسيطة ذات `notify: true` إلى تسليم Webhook صريح عندما يكون `cron.webhook` مهيأ.
</Note>

## تعديلات شائعة

حدّث إعدادات التسليم من دون تغيير الرسالة:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

عطّل التسليم لمهمة معزولة:

```bash
openclaw cron edit <job-id> --no-deliver
```

فعّل سياق تمهيد خفيف لمهمة معزولة:

```bash
openclaw cron edit <job-id> --light-context
```

أعلن إلى قناة محددة:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

أعلن إلى موضوع منتدى Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

أنشئ مهمة معزولة بسياق تمهيد خفيف:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

ينطبق `--light-context` على مهام أدوار الوكيل المعزولة فقط. بالنسبة إلى تشغيلات Cron، يُبقي الوضع الخفيف سياق التمهيد فارغًا بدلًا من حقن مجموعة تمهيد مساحة العمل الكاملة.

## أوامر الإدارة الشائعة

التشغيل اليدوي والفحص:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

تتضمن إدخالات `cron runs` تشخيصات التسليم مع هدف Cron المقصود، والهدف المحلول، وإرسالات أداة الرسائل، واستخدام الاحتياط، وحالة التسليم.

إعادة استهداف الوكيل والجلسة:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

يحذر `openclaw cron add` عند حذف `--agent` في مهام أدوار الوكيل ويرجع إلى الوكيل الافتراضي (`main`). مرر `--agent <id>` وقت الإنشاء لتثبيت وكيل محدد.

تعديلات التسليم:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
