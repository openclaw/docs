# `openclaw cron`

إدارة مهام Cron لجدولة Gateway.

<Tip>
شغّل `openclaw cron --help` لعرض واجهة الأوامر كاملة. راجع [مهام Cron](/ar/automation/cron-jobs) للدليل المفاهيمي.
</Tip>

## الجلسات

يقبل `--session` القيم `main` أو `isolated` أو `current` أو `session:<id>`.

<AccordionGroup>
  <Accordion title="Session keys">
    - يربط `main` بجلسة الوكيل الرئيسية.
    - ينشئ `isolated` نسخة محادثة ومعرّف جلسة جديدين لكل تشغيل.
    - يربط `current` بالجلسة النشطة وقت الإنشاء.
    - يثبّت `session:<id>` إلى مفتاح جلسة مستمرة صريح.

  </Accordion>
  <Accordion title="Isolated session semantics">
    تعيد عمليات التشغيل المعزولة ضبط سياق المحادثة المحيط. يُعاد ضبط توجيه القنوات والمجموعات، وسياسة الإرسال/الطابور، والتصعيد، والأصل، وربط وقت تشغيل ACP للتشغيل الجديد. يمكن أن تنتقل التفضيلات الآمنة وتجاوزات النموذج أو المصادقة التي اختارها المستخدم صراحةً بين عمليات التشغيل.
  </Accordion>
</AccordionGroup>

## التسليم

يعرض `openclaw cron list` و`openclaw cron show <job-id>` معاينة لمسار التسليم المحسوم. بالنسبة إلى `channel: "last"`، تعرض المعاينة ما إذا كان المسار قد حُسم من الجلسة الرئيسية أو الحالية، أو سيفشل بصورة مغلقة.

يمكن للأهداف ذات البادئة الخاصة بالمزوّد إزالة الالتباس عن قنوات الإعلان غير المحسومة. على سبيل المثال، يختار `to: "telegram:123"` Telegram عندما يكون `delivery.channel` محذوفًا أو `last`. البادئات التي يعلنها Plugin المحمّل فقط هي محددات مزوّدين. إذا كان `delivery.channel` صريحًا، يجب أن تطابق البادئة تلك القناة؛ يتم رفض `channel: "whatsapp"` مع `to: "telegram:123"`. تبقى بادئات الخدمات مثل `imessage:` و`sms:` صياغة أهداف مملوكة للقناة.

<Note>
تستخدم مهام `cron add` المعزولة تسليم `--announce` افتراضيًا. استخدم `--no-deliver` للإبقاء على الناتج داخليًا. يبقى `--deliver` اسمًا مستعارًا مهملاً لـ `--announce`.
</Note>

### ملكية التسليم

يتشارك الوكيل والمشغّل تسليم دردشة Cron المعزولة:

- يمكن للوكيل الإرسال مباشرةً باستخدام أداة `message` عندما يكون مسار دردشة متاحًا.
- يسلّم `announce` الرد النهائي كآلية احتياطية فقط عندما لا يرسل الوكيل مباشرةً إلى الهدف المحسوم.
- ينشر `webhook` الحمولة المكتملة إلى عنوان URL.
- يعطّل `none` التسليم الاحتياطي من المشغّل.

`--announce` هو التسليم الاحتياطي من المشغّل للرد النهائي. يعطّل `--no-deliver` ذلك الاحتياط، لكنه لا يزيل أداة `message` الخاصة بالوكيل عندما يكون مسار دردشة متاحًا.

تحتفظ التذكيرات المنشأة من دردشة نشطة بهدف تسليم الدردشة الحي لتسليم الإعلان الاحتياطي. قد تكون مفاتيح الجلسات الداخلية بأحرف صغيرة؛ لا تستخدمها مصدرًا للحقيقة لمعرّفات المزوّدين الحساسة لحالة الأحرف، مثل معرّفات غرف Matrix.

### تسليم الفشل

تُحسم إشعارات الفشل بهذا الترتيب:

1. `delivery.failureDestination` في المهمة.
2. `cron.failureDestination` العام.
3. هدف الإعلان الأساسي للمهمة (عند عدم تعيين وجهة فشل صريحة).

<Note>
قد تستخدم مهام الجلسة الرئيسية `delivery.failureDestination` فقط عندما يكون وضع التسليم الأساسي `webhook`. تقبله المهام المعزولة في كل الأوضاع.
</Note>

ملاحظة: تتعامل عمليات Cron المعزولة مع إخفاقات الوكيل على مستوى التشغيل كأخطاء مهمة حتى عندما
لا تُنتَج حمولة رد، لذلك لا تزال إخفاقات النموذج/المزوّد تزيد عدادات الأخطاء
وتشغّل إشعارات الفشل.

## الجدولة

### المهام أحادية التشغيل

يجدول `--at <datetime>` تشغيلًا أحاديًا. تُعامل التواريخ والأوقات بلا إزاحة على أنها UTC ما لم تمرر أيضًا `--tz <iana>`، الذي يفسّر وقت ساعة الحائط في المنطقة الزمنية المعطاة.

<Note>
تُحذف المهام أحادية التشغيل بعد النجاح افتراضيًا. استخدم `--keep-after-run` للاحتفاظ بها.
</Note>

### المهام المتكررة

تستخدم المهام المتكررة تراجع إعادة المحاولة الأسي بعد الأخطاء المتتالية: 30 ثانية، دقيقة واحدة، 5 دقائق، 15 دقيقة، 60 دقيقة. يعود الجدول إلى الوضع الطبيعي بعد التشغيل الناجح التالي.

تُتبع عمليات التشغيل المتخطاة بشكل منفصل عن أخطاء التنفيذ. لا تؤثر في تراجع إعادة المحاولة، لكن يمكن لـ `openclaw cron edit <job-id> --failure-alert-include-skipped` إدراج تنبيهات الفشل في إشعارات التشغيل المتخطى المتكررة.

بالنسبة إلى المهام المعزولة التي تستهدف مزوّد نموذج محليًا مُعدًا، يشغّل Cron فحصًا تمهيديًا خفيفًا للمزوّد قبل بدء دورة الوكيل. تُفحص مزوّدات `api: "ollama"` الخاصة بـ local loopback والشبكة الخاصة و`.local` عند `/api/tags`؛ وتُفحص المزوّدات المحلية المتوافقة مع OpenAI مثل vLLM وSGLang وLM Studio عند `/models`. إذا تعذر الوصول إلى نقطة النهاية، يُسجَّل التشغيل على أنه `skipped` وتُعاد المحاولة في جدول لاحق؛ تُخزَّن نقاط النهاية الميتة المطابقة مؤقتًا لمدة 5 دقائق لتجنب إغراق العديد من المهام للخادم المحلي نفسه.

ملاحظة: تعيش تعريفات مهام Cron في `jobs.json`، بينما تعيش حالة وقت التشغيل المعلّقة في `jobs-state.json`. إذا عُدّل `jobs.json` خارجيًا، يعيد Gateway تحميل الجداول المتغيرة ويمسح الخانات المعلّقة القديمة؛ ولا تمسح عمليات إعادة الكتابة الخاصة بالتنسيق فقط الخانة المعلّقة.

### عمليات التشغيل اليدوية

يرجع `openclaw cron run` بمجرد وضع التشغيل اليدوي في الطابور. تتضمن الاستجابات الناجحة `{ ok: true, enqueued: true, runId }`. استخدم `openclaw cron runs --id <job-id>` لمتابعة النتيجة النهائية.

<Note>
يشغّل `openclaw cron run <job-id>` تشغيلًا قسريًا افتراضيًا. استخدم `--due` للاحتفاظ بسلوك "شغّل فقط إذا حان الموعد" الأقدم.
</Note>

## النماذج

يختار `cron add|edit --model <ref>` نموذجًا مسموحًا للمهمة.

<Warning>
إذا لم يكن النموذج مسموحًا أو تعذر حسمه، يفشل Cron التشغيل بخطأ تحقق صريح بدلًا من الرجوع إلى وكيل المهمة أو اختيار النموذج الافتراضي.
</Warning>

`--model` في Cron هو **النموذج الأساسي للمهمة**، وليس تجاوز `/model` لجلسة دردشة. هذا يعني:

- لا تزال بدائل النماذج المكوّنة تنطبق عند فشل نموذج المهمة المختار.
- تستبدل `fallbacks` في حمولة كل مهمة قائمة البدائل المكوّنة عند وجودها.
- تجعل قائمة البدائل الفارغة لكل مهمة (`fallbacks: []` في حمولة/واجهة API الخاصة بالمهمة) تشغيل Cron صارمًا.
- عندما تحتوي مهمة على `--model` لكن لا توجد قائمة بدائل مكوّنة، يمرر OpenClaw تجاوز بدائل فارغًا صريحًا حتى لا يُلحق النموذج الأساسي للوكيل كهدف إعادة محاولة مخفي.

### أسبقية نموذج Cron المعزول

يحسم Cron المعزول النموذج النشط بهذا الترتيب:

1. تجاوز خطاف Gmail.
2. `--model` لكل مهمة.
3. تجاوز نموذج جلسة Cron المخزن (عندما يختار المستخدم واحدًا).
4. اختيار الوكيل أو النموذج الافتراضي.

### الوضع السريع

يتبع الوضع السريع في Cron المعزول اختيار النموذج الحي المحسوم. تنطبق إعدادات النموذج `params.fastMode` افتراضيًا، لكن تجاوز `fastMode` للجلسة المخزن يبقى له الأولوية على الإعدادات.

### إعادة محاولات تبديل النموذج الحي

إذا ألقى تشغيل معزول `LiveSessionModelSwitchError`، يحفظ Cron المزوّد والنموذج اللذين تم التبديل إليهما (وتجاوز ملف تعريف المصادقة الذي تم التبديل إليه عند وجوده) للتشغيل النشط قبل إعادة المحاولة. حلقة إعادة المحاولة الخارجية محدودة بمحاولتي تبديل بعد المحاولة الأولى، ثم تُجهض بدلًا من الدوران إلى الأبد.

## ناتج التشغيل والرفض

### منع الإقرارات القديمة

تمنع دورات Cron المعزولة الردود القديمة التي تقتصر على الإقرار. إذا كانت النتيجة الأولى مجرد تحديث حالة مؤقت ولم تكن أي عملية وكيل فرعي تابعة مسؤولة عن الإجابة النهائية، يعيد Cron المطالبة مرة واحدة للحصول على النتيجة الحقيقية قبل التسليم.

### منع الرمز الصامت

إذا أعاد تشغيل Cron معزول الرمز الصامت فقط (`NO_REPLY` أو `no_reply`)، يمنع Cron التسليم المباشر الصادر ومسار الملخص الاحتياطي الموضوع في الطابور، لذلك لا يُنشر أي شيء إلى الدردشة.

### الرفض المنظم

تفضّل عمليات Cron المعزولة بيانات وصفية منظمة لرفض التنفيذ من التشغيل المضمّن، ثم ترجع إلى علامات رفض معروفة في الناتج النهائي، مثل `SYSTEM_RUN_DENIED` و`INVALID_REQUEST` وعبارات رفض ربط الموافقة.

يعرض `cron list` وسجل التشغيل سبب الرفض بدلًا من الإبلاغ عن أمر محظور على أنه `ok`.

## الاحتفاظ

يُتحكم في الاحتفاظ والتقليم من الإعدادات:

- `cron.sessionRetention` (الافتراضي `24h`) يقلّم جلسات التشغيل المعزولة المكتملة.
- يقلّم `cron.runLog.maxBytes` و`cron.runLog.keepLines` ملف `~/.openclaw/cron/runs/<jobId>.jsonl`.

## ترحيل المهام الأقدم

<Note>
إذا كانت لديك مهام Cron من قبل تنسيق التسليم والتخزين الحالي، فشغّل `openclaw doctor --fix`. يطبع Doctor حقول Cron القديمة بالشكل القياسي (`jobId` و`schedule.cron` وحقول التسليم ذات المستوى الأعلى بما في ذلك `threadId` القديم وأسماء التسليم المستعارة `provider` في الحمولة) ويرحّل مهام احتياط Webhook البسيطة `notify: true` إلى تسليم Webhook صريح عندما يكون `cron.webhook` مكوّنًا.
</Note>

## تعديلات شائعة

تحديث إعدادات التسليم بدون تغيير الرسالة:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

تعطيل التسليم لمهمة معزولة:

```bash
openclaw cron edit <job-id> --no-deliver
```

تمكين سياق تمهيد خفيف لمهمة معزولة:

```bash
openclaw cron edit <job-id> --light-context
```

الإعلان إلى قناة محددة:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

الإعلان إلى موضوع منتدى Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

إنشاء مهمة معزولة بسياق تمهيد خفيف:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

ينطبق `--light-context` على مهام دورات الوكيل المعزولة فقط. بالنسبة إلى عمليات Cron، يبقي الوضع الخفيف سياق التمهيد فارغًا بدلًا من حقن مجموعة تمهيد مساحة العمل الكاملة.

## أوامر إدارية شائعة

التشغيل اليدوي والفحص:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

يعرض `openclaw cron list` كل المهام المطابقة افتراضيًا. مرر `--agent <id>` لعرض المهام التي يطابق معرّف وكيلها الفعلي والمطبّع فقط؛ تُحتسب المهام التي لا تملك معرّف وكيل مخزنًا على أنها تابعة للوكيل الافتراضي المكوّن.

يتضمن `cron list --json` و`cron show <job-id> --json` حقل `status` على المستوى الأعلى في كل مهمة، محسوبًا من `enabled` و`state.runningAtMs` و`state.lastRunStatus`. القيم: `disabled` أو `running` أو `ok` أو `error` أو `skipped` أو `idle`. يعكس هذا عمود الحالة القابل للقراءة بشريًا حتى تتمكن الأدوات الخارجية من قراءة حالة المهمة بدون إعادة اشتقاقها.

تتضمن مدخلات `cron runs` تشخيصات تسليم تشمل هدف Cron المقصود، والهدف المحسوم، وإرسالات أداة الرسائل، واستخدام الاحتياط، وحالة التسليم.

إعادة استهداف الوكيل والجلسة:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

يحذر `openclaw cron add` عندما يُحذف `--agent` في مهام دورات الوكيل ويرجع إلى الوكيل الافتراضي (`main`). مرر `--agent <id>` وقت الإنشاء لتثبيت وكيل محدد.

تعديلات التسليم:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
