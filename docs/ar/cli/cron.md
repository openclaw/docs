# `openclaw cron`

إدارة مهام Cron لجدولة Gateway.

ذو صلة:

- مهام Cron: [مهام Cron](/ar/automation/cron-jobs)

نصيحة: شغّل `openclaw cron --help` للحصول على سطح الأوامر الكامل.

ملاحظة: يعرض `openclaw cron list` و`openclaw cron show <job-id>` معاينة لمسار
التسليم المحلَّل. بالنسبة إلى `channel: "last"`، تعرض المعاينة ما إذا كان
المسار قد تم تحليله من الجلسة الرئيسية/الحالية أو ما إذا كان سيفشل بشكل مغلق.

ملاحظة: تستخدم مهام `cron add` المعزولة افتراضيًا التسليم عبر `--announce`. استخدم `--no-deliver` للإبقاء على
المخرجات داخلية. يظل `--deliver` متاحًا بوصفه اسمًا مستعارًا قديمًا لـ `--announce`.

ملاحظة: تسليم الدردشة في Cron المعزول مشترك. يمثّل `--announce` تسليمًا احتياطيًا من المشغّل
للرد النهائي؛ ويعطّل `--no-deliver` ذلك الاحتياطي، لكنه
لا يزيل أداة `message` الخاصة بالوكيل عندما يكون مسار دردشة متاحًا.

ملاحظة: تُحذف المهام ذات التشغيل لمرة واحدة (`--at`) بعد النجاح افتراضيًا. استخدم `--keep-after-run` للاحتفاظ بها.

ملاحظة: يدعم `--session` القيم `main` و`isolated` و`current` و`session:<id>`.
استخدم `current` للربط بالجلسة النشطة وقت الإنشاء، أو `session:<id>` من أجل
مفتاح جلسة دائم وصريح.

ملاحظة: بالنسبة إلى مهام CLI ذات التشغيل لمرة واحدة، يتم التعامل مع تواريخ `--at` بدون إزاحة زمنية على أنها UTC ما لم تمرر أيضًا
`--tz <iana>`، والذي يفسر ذلك الوقت المحلي حسب الساعة الجدارية ضمن المنطقة الزمنية المعطاة.

ملاحظة: تستخدم المهام المتكررة الآن تراجع إعادة محاولة أسي بعد الأخطاء المتتالية (30 ثانية ← 1 دقيقة ← 5 دقائق ← 15 دقيقة ← 60 دقيقة)، ثم تعود إلى الجدول الزمني العادي بعد التشغيل الناجح التالي.

ملاحظة: يعيد `openclaw cron run` الآن النتيجة بمجرد إدراج التشغيل اليدوي في قائمة التنفيذ. تتضمن الاستجابات الناجحة `{ ok: true, enqueued: true, runId }`؛ استخدم `openclaw cron runs --id <job-id>` لمتابعة النتيجة النهائية.

ملاحظة: يفرض `openclaw cron run <job-id>` التشغيل افتراضيًا. استخدم `--due` للإبقاء على
السلوك الأقدم "شغّل فقط إذا حان الوقت".

ملاحظة: تمنع تشغيلات Cron المعزولة الردود القديمة الخاصة بالإقرار فقط. إذا كانت
النتيجة الأولى مجرد تحديث حالة مؤقت ولم يكن هناك أي تشغيل لوكيل فرعي تابع
مسؤول عن الإجابة النهائية، فإن Cron يعيد المطالبة مرة واحدة للحصول على النتيجة الفعلية
قبل التسليم.

ملاحظة: إذا أعاد تشغيل Cron معزول الرمز الصامت فقط (`NO_REPLY` /
`no_reply`)، فإن Cron يمنع التسليم الصادر المباشر وكذلك مسار الملخص
الاحتياطي المدرج في قائمة الانتظار، لذلك لا يتم نشر أي شيء مرة أخرى إلى الدردشة.

ملاحظة: يستخدم `cron add|edit --model ...` ذلك النموذج المسموح المحدد للمهمة.
إذا لم يكن النموذج مسموحًا، فسيحذر Cron ويعود إلى اختيار النموذج
الافتراضي للمهمة/الوكيل بدلًا من ذلك. تظل سلاسل الاحتياط المكوّنة مطبقة، لكن
تجاوز نموذج عادي من دون قائمة احتياط صريحة لكل مهمة لم يعد يضيف
النموذج الأساسي للوكيل كهدف إعادة محاولة إضافي مخفي.

ملاحظة: أولوية النموذج في Cron المعزول هي تجاوز Gmail hook أولًا، ثم
`--model` لكل مهمة، ثم أي تجاوز نموذج مخزّن لجلسة Cron، ثم الاختيار
العادي للوكيل/الافتراضي.

ملاحظة: يتبع الوضع السريع في Cron المعزول اختيار النموذج الحي المحلَّل.
يُطبّق تكوين النموذج `params.fastMode` افتراضيًا، لكن تجاوز `fastMode`
المخزن في الجلسة يظل أعلى أولوية من التكوين.

ملاحظة: إذا طرح تشغيل معزول الخطأ `LiveSessionModelSwitchError`، فإن Cron يحتفظ
بالمزوّد/النموذج المُبدَّل (وكذلك تجاوز ملف تعريف المصادقة المُبدَّل عند وجوده) قبل
إعادة المحاولة. تكون حلقة إعادة المحاولة الخارجية مقيّدة بمحاولتي تبديل بعد
المحاولة الأولية، ثم تُجهض بدلًا من الدوران إلى ما لا نهاية.

ملاحظة: تستخدم إشعارات الفشل `delivery.failureDestination` أولًا، ثم
`cron.failureDestination` العام، وأخيرًا تعود إلى هدف
الإعلان الأساسي للمهمة عندما لا يكون هناك هدف فشل صريح مكوَّن.

ملاحظة: يتم التحكم في الاحتفاظ/التقليم في التكوين:

- `cron.sessionRetention` (الافتراضي `24h`) يزيل جلسات التشغيل المعزولة المكتملة.
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` يقلّمان `~/.openclaw/cron/runs/<jobId>.jsonl`.

ملاحظة الترقية: إذا كانت لديك مهام Cron أقدم من تنسيق التسليم/التخزين الحالي، فشغّل
`openclaw doctor --fix`. يقوم Doctor الآن بتطبيع حقول Cron القديمة (`jobId` و`schedule.cron`،
وحقول التسليم ذات المستوى الأعلى بما في ذلك `threadId` القديم، وأسماء التسليم المستعارة لـ `provider` في الحمولة) وينقل
مهام Webhook الاحتياطية البسيطة ذات `notify: true` إلى تسليم Webhook صريح عندما يكون `cron.webhook`
مكوّنًا.

## تعديلات شائعة

حدّث إعدادات التسليم من دون تغيير الرسالة:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

تعطيل التسليم لمهمة معزولة:

```bash
openclaw cron edit <job-id> --no-deliver
```

تمكين سياق bootstrap خفيف لمهمة معزولة:

```bash
openclaw cron edit <job-id> --light-context
```

الإعلان في قناة محددة:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

إنشاء مهمة معزولة بسياق bootstrap خفيف:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

ينطبق `--light-context` على مهام أدوار الوكيل المعزولة فقط. بالنسبة إلى تشغيلات Cron، يحافظ الوضع الخفيف على سياق bootstrap فارغًا بدلًا من حقن مجموعة bootstrap الكاملة لمساحة العمل.

ملاحظة ملكية التسليم:

- تسليم الدردشة في Cron المعزول مشترك. يمكن للوكيل الإرسال مباشرة باستخدام
  أداة `message` عندما يكون مسار دردشة متاحًا.
- يسلّم `announce` الرد النهائي احتياطيًا فقط عندما لا يكن الوكيل قد أرسل
  مباشرة إلى الهدف المحلَّل. ينشر `webhook` الحمولة المكتملة إلى عنوان URL.
  وتعطّل `none` التسليم الاحتياطي من المشغّل.

## أوامر إدارية شائعة

تشغيل يدوي:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

تتضمن إدخالات `cron runs` تشخيصات التسليم مع هدف Cron المقصود،
والهدف المحلَّل، وعمليات الإرسال عبر أداة الرسائل، واستخدام الاحتياطي، وحالة التسليم.

إعادة توجيه الوكيل/الجلسة:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

تعديلات التسليم:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```

ملاحظة تسليم الفشل:

- `delivery.failureDestination` مدعوم للمهام المعزولة.
- لا يمكن لمهام الجلسة الرئيسية استخدام `delivery.failureDestination` إلا عندما
  يكون وضع التسليم الأساسي هو `webhook`.
- إذا لم تضبط أي وجهة للفشل وكانت المهمة تعلن بالفعل إلى
  قناة، فإن إشعارات الفشل تعيد استخدام هدف الإعلان نفسه.
