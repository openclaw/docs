# `openclaw cron`

إدارة مهام Cron لمجدول Gateway.

<Tip>
شغّل `openclaw cron --help` للاطلاع على كامل سطح الأوامر. راجع [مهام Cron](/ar/automation/cron-jobs) للدليل المفاهيمي.
</Tip>

## إنشاء المهام بسرعة

`openclaw cron create` هو اسم بديل لـ `openclaw cron add`. للمهام الجديدة، ضع الجدول أولًا والمطالبة ثانيًا:

```bash
openclaw cron create "0 7 * * *" \
  "Summarize overnight updates." \
  --name "Morning brief" \
  --agent ops
```

استخدم `--webhook <url>` عندما يجب أن ترسل المهمة الحمولة النهائية عبر POST بدلًا من تسليمها إلى هدف محادثة:

```bash
openclaw cron create "0 18 * * 1-5" \
  "Summarize today's deploys as JSON." \
  --name "Deploy digest" \
  --webhook "https://example.invalid/openclaw/cron"
```

استخدم `--command` للمهام الحتمية بأسلوب الصدفة التي يجب أن تعمل داخل Cron في OpenClaw دون بدء تشغيل وكيل/نموذج معزول:

<Note>
مهام Cron الأمر هي أتمتة Gateway يكتبها المسؤول. يتطلب إنشاؤها أو تعديلها
أو إزالتها أو تشغيلها يدويًا `operator.admin`؛ وينفذ التشغيل المجدول
لاحقًا داخل عملية Gateway، وليس كاستدعاء أداة وكيل `tools.exec`.
ما زالت `tools.exec.*` وموافقات exec تحكم أدوات exec المرئية للنموذج.
</Note>

```bash
openclaw cron create "*/15 * * * *" \
  --name "Queue depth probe" \
  --command "scripts/check-queue.sh" \
  --command-cwd "/srv/app" \
  --announce \
  --channel telegram \
  --to "-1001234567890"
```

يخزن `--command <shell>` القيمة `argv: ["sh", "-lc", <shell>]`. استخدم `--command-argv '["node","scripts/report.mjs"]'` لتنفيذ argv الدقيق. تلتقط مهام الأمر stdout/stderr، وتسجل سجل Cron العادي، وتوجه المخرجات عبر أوضاع التسليم نفسها `announce` أو `webhook` أو `none` كما في المهام المعزولة. يتم كبت الأمر الذي يطبع `NO_REPLY` فقط.

## التسليم

يعرض `openclaw cron list` و`openclaw cron show <job-id>` معاينة لمسار التسليم المحلول. بالنسبة إلى `channel: "last"`، تظهر المعاينة ما إذا كان المسار قد حُل من الجلسة الرئيسية أو الحالية، أو سيفشل بإغلاق آمن.

يمكن للأهداف المسبوقة بمزوّد أن تزيل الالتباس عن قنوات الإعلان غير المحلولة. على سبيل المثال، يحدد `to: "telegram:123"` ‏Telegram عندما تكون `delivery.channel` محذوفة أو `last`. فقط البادئات التي يعلنها Plugin المحمل هي محددات مزوّدين. إذا كانت `delivery.channel` صريحة، فيجب أن تطابق البادئة تلك القناة؛ يتم رفض `channel: "whatsapp"` مع `to: "telegram:123"`. تظل بادئات الخدمة مثل `imessage:` و`sms:` صياغة أهداف تملكها القناة.

<Note>
تستخدم مهام `cron add` المعزولة تسليم `--announce` افتراضيًا. استخدم `--no-deliver` لإبقاء المخرجات داخلية. يظل `--deliver` اسمًا بديلًا مهملًا لـ `--announce`.
</Note>

### ملكية التسليم

تسليم محادثة Cron المعزول مشترك بين الوكيل والمشغّل:

- يمكن للوكيل الإرسال مباشرة باستخدام أداة `message` عندما يكون مسار محادثة متاحًا.
- يسلّم احتياطي `announce` الرد النهائي فقط عندما لا يرسل الوكيل مباشرة إلى الهدف المحلول.
- ينشر `webhook` الحمولة المنتهية إلى عنوان URL.
- يعطل `none` تسليم المشغّل الاحتياطي.

استخدم `cron add|create --webhook <url>` أو `cron edit <job-id> --webhook <url>` لتعيين تسليم Webhook. لا تجمع `--webhook` مع أعلام تسليم المحادثة مثل `--announce` أو `--no-deliver` أو `--channel` أو `--to` أو `--thread-id` أو `--account`.

يمكن لـ `cron edit <job-id>` إلغاء تعيين حقول توجيه التسليم الفردية باستخدام `--clear-channel` و`--clear-to` و`--clear-thread-id` و`--clear-account` (يُرفض كل منها عند دمجه مع علم التعيين المطابق له). بخلاف `--no-deliver`، الذي يعطل تسليم المشغّل الاحتياطي فقط، تزيل هذه الأعلام الحقل المخزن بحيث تحل المهمة ذلك الجزء من مسارها من القيم الافتراضية مرة أخرى.

`--announce` هو تسليم المشغّل الاحتياطي للرد النهائي. يعطل `--no-deliver` ذلك الاحتياطي لكنه لا يزيل أداة `message` الخاصة بالوكيل عندما يكون مسار محادثة متاحًا.

تحافظ التذكيرات المنشأة من محادثة نشطة على هدف تسليم المحادثة المباشر لتسليم الإعلان الاحتياطي. قد تكون مفاتيح الجلسة الداخلية بأحرف صغيرة؛ لا تستخدمها كمصدر حقيقة لمعرفات المزوّدين الحساسة لحالة الأحرف مثل معرفات غرف Matrix.

### تسليم الفشل

تُحل إشعارات الفشل بهذا الترتيب:

1. `delivery.failureDestination` في المهمة.
2. `cron.failureDestination` العام.
3. هدف الإعلان الأساسي للمهمة (عندما لا يكون هناك وجهة فشل صريحة معينة).

<Note>
لا يجوز لمهام الجلسة الرئيسية استخدام `delivery.failureDestination` إلا عندما يكون وضع التسليم الأساسي هو `webhook`. تقبله المهام المعزولة في كل الأوضاع.
</Note>

ملاحظة: تتعامل عمليات Cron المعزولة مع إخفاقات الوكيل على مستوى التشغيل كأخطاء مهمة حتى عندما
لا تُنتج حمولة رد، لذلك ما زالت إخفاقات النموذج/المزوّد تزيد عدادات الأخطاء
وتطلق إشعارات الفشل.

لا تبدأ مهام Cron الأمر دورة وكيل معزولة. يسجل رمز خروج صفري
`ok`؛ ويسجل الخروج غير الصفري، أو الإشارة، أو انتهاء المهلة، أو انتهاء مهلة عدم الإخراج `error` ويمكن
أن يطلق مسار إشعار الفشل نفسه.

إذا انتهت مهلة تشغيل معزول قبل أول طلب نموذج، فإن `openclaw cron show`
و`openclaw cron runs` يتضمنان خطأ خاصًا بالمرحلة مثل
`setup timed out before runner start` أو
`stalled before first model call (last phase: context-engine)`.
بالنسبة إلى المزوّدين المدعومين بـ CLI، يظل مراقب ما قبل النموذج نشطًا حتى تبدأ دورة
CLI الخارجية، لذلك تُبلّغ حالات تعطل البحث عن الجلسة والخطاف والمصادقة والمطالبة وإعداد CLI
كإخفاقات Cron قبل النموذج.

## الجدولة

### المهام لمرة واحدة

يجدول `--at <datetime>` تشغيلًا لمرة واحدة. تُعامل التواريخ والأوقات بلا إزاحة على أنها UTC ما لم تمرر أيضًا `--tz <iana>`، الذي يفسر وقت ساعة الحائط في المنطقة الزمنية المعطاة.

<Note>
تحذف المهام لمرة واحدة بعد النجاح افتراضيًا. استخدم `--keep-after-run` للاحتفاظ بها.
</Note>

### المهام المتكررة

تستخدم المهام المتكررة تراجع إعادة محاولة أسيًا بعد الأخطاء المتتالية: 30s، 1m، 5m، 15m، 60m. يعود الجدول إلى طبيعته بعد التشغيل الناجح التالي.

تُتبع عمليات التشغيل المتخطاة بشكل منفصل عن أخطاء التنفيذ. لا تؤثر في تراجع إعادة المحاولة، لكن يمكن لـ `openclaw cron edit <job-id> --failure-alert-include-skipped` إدخال تنبيهات الفشل في إشعارات التشغيل المتخطى المتكررة.

بالنسبة إلى المهام المعزولة التي تستهدف مزوّد نموذج محليًا مكوّنًا، يشغّل Cron فحصًا تمهيديًا خفيفًا للمزوّد قبل بدء دورة الوكيل. يتم فحص مزوّدي `api: "ollama"` الذين يستخدمون local loopback أو شبكة خاصة أو `.local` عند `/api/tags`؛ ويتم فحص المزوّدين المحليين المتوافقين مع OpenAI مثل vLLM وSGLang وLM Studio عند `/models`. إذا تعذر الوصول إلى نقطة النهاية، يُسجل التشغيل كـ `skipped` ويُعاد في جدول لاحق؛ وتُخزن نقاط النهاية الميتة المطابقة مؤقتًا لمدة 5 دقائق لتجنب قيام العديد من المهام بالضغط على الخادم المحلي نفسه.

ملاحظة: تعيش مهام Cron وحالة وقت التشغيل المعلقة وسجل التشغيل في قاعدة بيانات حالة SQLite المشتركة. تُستورد ملفات `jobs.json` و`jobs-state.json` و`runs/*.jsonl` القديمة مرة واحدة وتُعاد تسميتها بلاحقة `.migrated`. بعد الاستيراد، عدّل الجداول باستخدام `openclaw cron add|edit|remove` بدلًا من تعديل ملفات JSON.

### عمليات التشغيل اليدوية

يفرض `openclaw cron run <job-id>` التشغيل افتراضيًا ويعود بمجرد وضع التشغيل اليدوي في الطابور. تتضمن الاستجابات الناجحة `{ ok: true, enqueued: true, runId }`. استخدم `runId` المُعاد لفحص النتيجة اللاحقة:

```bash
openclaw cron run <job-id>
openclaw cron runs --id <job-id> --run-id <run-id>
```

أضف `--wait` عندما يجب أن يحظر السكربت حتى يسجل ذلك التشغيل الموضوع في الطابور بالضبط حالة نهائية:

```bash
openclaw cron run <job-id> --wait --wait-timeout 10m --poll-interval 2s
```

مع `--wait`، ما زال CLI يستدعي `cron.run` أولًا، ثم يستطلع `cron.runs` من أجل `runId` المُعاد. يخرج الأمر بـ `0` فقط عندما ينتهي التشغيل بالحالة `ok`. ويخرج بقيمة غير صفرية عندما ينتهي التشغيل بـ `error` أو `skipped`، أو عندما لا تتضمن استجابة Gateway قيمة `runId`، أو عندما تنتهي مهلة `--wait-timeout`. يجب أن يكون `--poll-interval` أكبر من الصفر.

<Note>
استخدم `--due` عندما تريد أن يعمل الأمر اليدوي فقط إذا كانت المهمة مستحقة حاليًا. إذا لم يضع `--due --wait` تشغيلًا في الطابور، يعيد الأمر استجابة عدم التشغيل العادية بدلًا من الاستطلاع.
</Note>

## النماذج

يحدد `cron add|edit --model <ref>` نموذجًا مسموحًا به للمهمة. يضبط `cron add|edit --fallbacks <list>` نماذج احتياطية لكل مهمة، مثلًا `--fallbacks openrouter/gpt-4.1-mini,openai/gpt-5`؛ مرر `--fallbacks ""` لتشغيل صارم بلا احتياطيات. يزيل `cron edit <job-id> --clear-fallbacks` تجاوز الاحتياطيات الخاص بالمهمة. يزيل `cron edit <job-id> --clear-model` تجاوز النموذج الخاص بالمهمة بحيث تتبع المهمة أسبقية اختيار نموذج Cron العادية (تجاوز جلسة Cron مخزن إن وجد، وإلا نموذج الوكيل/الافتراضي)؛ ولا يمكن دمجه مع `--model`.

<Warning>
إذا لم يكن النموذج مسموحًا أو تعذر حله، يفشل Cron التشغيل بخطأ تحقق صريح بدلًا من الرجوع إلى اختيار نموذج وكيل المهمة أو النموذج الافتراضي.
</Warning>

`--model` في Cron هو **أساسي للمهمة**، وليس تجاوز `/model` لجلسة محادثة. وهذا يعني:

- تظل احتياطيات النموذج المكوّنة مطبقة عندما يفشل نموذج المهمة المحدد.
- تستبدل حمولة `fallbacks` لكل مهمة قائمة الاحتياطيات المكوّنة عند وجودها.
- تجعل قائمة احتياطيات فارغة لكل مهمة (`--fallbacks ""` أو `fallbacks: []` في حمولة/واجهة API المهمة) تشغيل Cron صارمًا.
- عندما تحتوي مهمة على `--model` لكن لا تكون هناك قائمة احتياطيات مكوّنة، يمرر OpenClaw تجاوز احتياطيات فارغًا صريحًا كي لا يُلحق النموذج الأساسي للوكيل كهدف إعادة محاولة مخفي.
- تسير فحوصات ما قبل التشغيل للمزوّد المحلي عبر الاحتياطيات المكوّنة قبل وسم تشغيل Cron بأنه `skipped`.

يبلغ `openclaw doctor` عن المهام التي لديها `payload.model` معين بالفعل، بما في ذلك أعداد مساحات أسماء المزوّدين وعدم التطابق مع `agents.defaults.model`. استخدم ذلك الفحص عندما يبدو سلوك المصادقة أو المزوّد أو الفوترة مختلفًا بين المحادثة المباشرة والمهام المجدولة.

### أسبقية نموذج Cron المعزول

يحل Cron المعزول النموذج النشط بهذا الترتيب:

1. تجاوز خطاف Gmail.
2. `--model` لكل مهمة.
3. تجاوز نموذج جلسة Cron المخزن (عندما يختار المستخدم واحدًا).
4. اختيار نموذج الوكيل أو النموذج الافتراضي.

### الوضع السريع

يتبع الوضع السريع في Cron المعزول اختيار النموذج المباشر المحلول. ينطبق إعداد النموذج `params.fastMode` افتراضيًا، لكن يظل تجاوز الجلسة المخزن `fastMode` يغلب الإعداد. عندما يكون الوضع المحلول `auto`، يستخدم الحد الفاصل قيمة `params.fastAutoOnSeconds` للنموذج المحدد، مع افتراض 60 ثانية.

### إعادة محاولات تبديل النموذج المباشر

إذا طرح تشغيل معزول `LiveSessionModelSwitchError`، يستمر Cron في تخزين المزوّد والنموذج المحولين (وتجاوز ملف المصادقة المحول عند وجوده) للتشغيل النشط قبل إعادة المحاولة. تُحد حلقة إعادة المحاولة الخارجية بإعادة محاولتين للتبديل بعد المحاولة الأولية، ثم تُجهض بدلًا من الدوران إلى الأبد.

## مخرجات التشغيل والرفض

### كبت الإقرار القديم

تكبت دورات Cron المعزولة الردود القديمة التي تقتصر على الإقرار. إذا كانت النتيجة الأولى مجرد تحديث حالة مؤقت ولم تكن أي دورة وكيل فرعي منحدرة مسؤولة عن الإجابة النهائية، يعيد Cron المطالبة مرة واحدة للحصول على النتيجة الحقيقية قبل التسليم.

### كبت الرمز الصامت

إذا أعاد تشغيل Cron معزول الرمز الصامت فقط (`NO_REPLY` أو `no_reply`)، يكبت Cron كلًا من التسليم الصادر المباشر ومسار الملخص الاحتياطي الموضوع في الطابور، لذلك لا يُنشر شيء مرة أخرى إلى المحادثة.

### الرفض المنظم

تستخدم تشغيلات Cron المعزولة بيانات تعريف رفض التنفيذ المنظمة من التشغيل المضمّن باعتبارها إشارة الرفض الموثوقة. كما أنها تراعي أغلفة مضيف Node بقيمة `UNAVAILABLE` عندما تبدأ رسالة الخطأ المنظمة المتداخلة بـ `SYSTEM_RUN_DENIED` أو `INVALID_REQUEST`.

لا يصنّف Cron نثر المخرجات النهائية أو عبارات الرفض التي تبدو كالموافقة على أنها حالات رفض إلا إذا وفّر التشغيل المضمّن أيضًا بيانات تعريف رفض منظمة، لذلك لا يُعامل نص المساعد العادي كأمر محظور.

يعرض `cron list` وسجل التشغيلات سبب الرفض بدلًا من الإبلاغ عن أمر محظور على أنه `ok`.

## الاحتفاظ

يتم التحكم في الاحتفاظ والتقليم في الإعدادات:

- `cron.sessionRetention` (الافتراضي `24h`) يقلّم جلسات التشغيل المعزولة المكتملة.
- `cron.runLog.keepLines` يقلّم صفوف سجل التشغيل في SQLite المحتفظ بها لكل مهمة. يظل `cron.runLog.maxBytes` مقبولًا للتوافق مع سجلات التشغيل الأقدم المدعومة بالملفات.

## ترحيل المهام الأقدم

<Note>
إذا كانت لديك مهام cron من قبل تنسيق التسليم والتخزين الحالي، فشغّل `openclaw doctor --fix`. يطبّع Doctor حقول cron القديمة (`jobId` و`schedule.cron` وحقول التسليم ذات المستوى الأعلى بما في ذلك `threadId` القديم، وأسماء التسليم البديلة `provider` في الحمولة) ويرحّل مهام fallback للـ webhook ذات `notify: true` من `cron.webhook` إلى تسليم webhook صريح. تحتفظ المهام التي تعلن بالفعل إلى دردشة بذلك التسليم وتحصل على وجهة webhook للإكمال. عندما لا يكون `cron.webhook` معيّنًا، تتم إزالة علامة `notify` الخاملة ذات المستوى الأعلى للمهام التي لا تملك هدف ترحيل (ويُحافظ على التسليم الموجود دون تغيير)، لذلك لم يعد `doctor --fix` يواصل إعادة التحذير بشأنها.
</Note>

## تعديلات شائعة

حدّث إعدادات التسليم دون تغيير الرسالة:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

عطّل التسليم لمهمة معزولة:

```bash
openclaw cron edit <job-id> --no-deliver
```

فعّل سياق تمهيد خفيف لمهمة معزولة:

```bash
openclaw cron edit <job-id> --light-context
```

أعلن إلى قناة محددة:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

أعلن إلى موضوع منتدى Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

أنشئ مهمة معزولة بسياق تمهيد خفيف:

```bash
openclaw cron create "0 7 * * *" \
  "Summarize overnight updates." \
  --name "Lightweight morning brief" \
  --session isolated \
  --light-context \
  --no-deliver
```

ينطبق `--light-context` على مهام دور الوكيل المعزولة فقط. بالنسبة لتشغيلات cron، يُبقي الوضع الخفيف سياق التمهيد فارغًا بدلًا من حقن مجموعة تمهيد مساحة العمل الكاملة.

أنشئ مهمة أمر مع argv وcwd وenv وstdin وحدود مخرجات دقيقة:

```bash
openclaw cron create "*/30 * * * *" \
  --name "Position export" \
  --command-argv '["node","scripts/export-position.mjs"]' \
  --command-cwd "/srv/app" \
  --command-env "NODE_ENV=production" \
  --command-input '{"mode":"summary"}' \
  --timeout-seconds 120 \
  --no-output-timeout-seconds 30 \
  --output-max-bytes 65536 \
  --webhook "https://example.invalid/openclaw/cron"
```

## أوامر إدارية شائعة

التشغيل اليدوي والفحص:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron get <job-id>
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron run <job-id> --wait --wait-timeout 10m
openclaw cron run <job-id> --wait --wait-timeout 10m --poll-interval 2s
openclaw cron runs --id <job-id> --limit 50
openclaw cron runs --id <job-id> --run-id <run-id>
```

يعرض `openclaw cron list` جميع المهام المطابقة افتراضيًا. مرّر `--agent <id>` لعرض المهام التي يطابق معرّف وكيلها المطبع الفعّال فقط؛ وتُحسب المهام التي لا تحتوي على معرّف وكيل مخزّن كأنها تابعة للوكيل الافتراضي المضبوط.

يعيد `openclaw cron get <job-id>` ملف JSON المخزّن للمهمة مباشرة. استخدم `cron show <job-id>` عندما تريد عرضًا قابلاً للقراءة البشرية مع معاينة مسار التسليم.

يتضمن `cron list --json` و`cron show <job-id> --json` حقل `status` ذا مستوى أعلى في كل مهمة، محسوبًا من `enabled` و`state.runningAtMs` و`state.lastRunStatus`. القيم: `disabled` أو `running` أو `ok` أو `error` أو `skipped` أو `idle`. يعكس هذا عمود الحالة القابل للقراءة البشرية حتى تتمكن الأدوات الخارجية من قراءة حالة المهمة دون إعادة اشتقاقها.

تتضمن إدخالات `cron runs` تشخيصات التسليم مع هدف cron المقصود، والهدف الذي تم حله، وإرسالات أداة الرسائل، واستخدام fallback، وحالة التسليم.

إعادة استهداف الوكيل والجلسة:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

يحذّر `openclaw cron add` عند حذف `--agent` في مهام دور الوكيل، ويعود إلى الوكيل الافتراضي (`main`). مرّر `--agent <id>` وقت الإنشاء لتثبيت وكيل محدد.

تعديلات التسليم:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --webhook "https://example.invalid/openclaw/cron"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
