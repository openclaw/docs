# `openclaw cron`

إدارة مهام cron لجدولة Gateway.

<Tip>
شغّل `openclaw cron --help` للاطلاع على كامل واجهة الأوامر. راجع [مهام Cron](/ar/automation/cron-jobs) للدليل المفاهيمي.
</Tip>

## الجلسات

يقبل `--session` القيم `main` أو `isolated` أو `current` أو `session:<id>`.

<AccordionGroup>
  <Accordion title="مفاتيح الجلسة">
    - يربط `main` بجلسة الوكيل الرئيسية.
    - ينشئ `isolated` نصًا جديدًا ومعرّف جلسة جديدًا لكل تشغيل.
    - يربط `current` بالجلسة النشطة وقت الإنشاء.
    - يثبّت `session:<id>` على مفتاح جلسة دائم صريح.

  </Accordion>
  <Accordion title="دلالات الجلسة المعزولة">
    تعيد التشغيلات المعزولة ضبط سياق المحادثة المحيط. تتم إعادة ضبط توجيه القنوات والمجموعات، وسياسة الإرسال/وضع الانتظار، والتصعيد، والمنشأ، وربط وقت تشغيل ACP للتشغيل الجديد. يمكن أن تنتقل التفضيلات الآمنة والنموذج الذي اختاره المستخدم صراحة أو تجاوزات المصادقة عبر التشغيلات.
  </Accordion>
</AccordionGroup>

## التسليم

يعرض `openclaw cron list` و`openclaw cron show <job-id>` معاينة لمسار التسليم المحلول. بالنسبة إلى `channel: "last"`، تعرض المعاينة ما إذا كان المسار قد حُلّ من الجلسة الرئيسية أو الحالية، أو أنه سيفشل مغلقًا.

<Note>
تستخدم مهام `cron add` المعزولة تسليم `--announce` افتراضيًا. استخدم `--no-deliver` لإبقاء المخرجات داخلية. يظل `--deliver` اسمًا مستعارًا مهجورًا لـ `--announce`.
</Note>

### ملكية التسليم

تسليم محادثة cron المعزولة مشترك بين الوكيل والمشغّل:

- يمكن للوكيل الإرسال مباشرة باستخدام أداة `message` عندما يكون مسار محادثة متاحًا.
- يسلّم `announce` الرد النهائي احتياطيًا فقط عندما لا يرسل الوكيل مباشرة إلى الهدف المحلول.
- ينشر `webhook` الحمولة المكتملة إلى عنوان URL.
- يعطّل `none` التسليم الاحتياطي من المشغّل.

`--announce` هو تسليم احتياطي من المشغّل للرد النهائي. يعطّل `--no-deliver` ذلك التسليم الاحتياطي لكنه لا يزيل أداة `message` الخاصة بالوكيل عندما يكون مسار محادثة متاحًا.

تحافظ التذكيرات المنشأة من محادثة نشطة على هدف تسليم المحادثة المباشر لاستخدامه في تسليم الإعلان الاحتياطي. قد تكون مفاتيح الجلسات الداخلية بأحرف صغيرة؛ لا تستخدمها كمصدر حقيقة لمعرّفات المزوّدين الحساسة لحالة الأحرف مثل معرّفات غرف Matrix.

### تسليم الفشل

تُحل إشعارات الفشل بهذا الترتيب:

1. `delivery.failureDestination` على المهمة.
2. `cron.failureDestination` العام.
3. هدف الإعلان الأساسي للمهمة (عندما لا تكون هناك وجهة فشل صريحة مضبوطة).

<Note>
يمكن لمهام الجلسة الرئيسية استخدام `delivery.failureDestination` فقط عندما يكون وضع التسليم الأساسي هو `webhook`. تقبله المهام المعزولة في كل الأوضاع.
</Note>

ملاحظة: تتعامل تشغيلات cron المعزولة مع حالات فشل الوكيل على مستوى التشغيل كأخطاء مهمة حتى عندما
لا تُنتج حمولة رد، لذلك لا تزال حالات فشل النموذج/المزوّد تزيد عدادات الأخطاء
وتشغّل إشعارات الفشل.

## الجدولة

### مهام مرة واحدة

يجدول `--at <datetime>` تشغيلًا لمرة واحدة. تُعامل أوقات التاريخ بلا إزاحة على أنها UTC ما لم تمرر أيضًا `--tz <iana>`، والذي يفسر وقت الساعة المحلي في المنطقة الزمنية المحددة.

<Note>
تُحذف مهام المرة الواحدة بعد النجاح افتراضيًا. استخدم `--keep-after-run` للاحتفاظ بها.
</Note>

### المهام المتكررة

تستخدم المهام المتكررة تراجع إعادة محاولة أسيًا بعد الأخطاء المتتالية: 30s، 1m، 5m، 15m، 60m. يعود الجدول إلى الوضع الطبيعي بعد التشغيل الناجح التالي.

تُتبع التشغيلات المتخطاة بشكل منفصل عن أخطاء التنفيذ. لا تؤثر في تراجع إعادة المحاولة، لكن يمكن لـ `openclaw cron edit <job-id> --failure-alert-include-skipped` اختيار إدراج التشغيلات المتخطاة المتكررة في تنبيهات الفشل.

بالنسبة إلى المهام المعزولة التي تستهدف مزوّد نموذج محليًا مهيأ، يشغّل cron فحصًا تمهيديًا خفيفًا للمزوّد قبل بدء دورة الوكيل. تُفحص مزوّدات `api: "ollama"` ذات Loopback والشبكة الخاصة و`.local` عند `/api/tags`؛ وتُفحص المزوّدات المحلية المتوافقة مع OpenAI مثل vLLM وSGLang وLM Studio عند `/models`. إذا تعذر الوصول إلى نقطة النهاية، يُسجل التشغيل كـ `skipped` وتُعاد المحاولة في جدول لاحق؛ تُخزن نقاط النهاية الميتة المطابقة مؤقتًا لمدة 5 دقائق لتجنب طرق الكثير من المهام للخادم المحلي نفسه.

ملاحظة: تعيش تعريفات مهام cron في `jobs.json`، بينما تعيش حالة وقت التشغيل المعلقة في `jobs-state.json`. إذا عُدّل `jobs.json` خارجيًا، يعيد Gateway تحميل الجداول المتغيرة ويمسح الخانات المعلقة القديمة؛ لا تمسح عمليات إعادة الكتابة الخاصة بالتنسيق فقط الخانة المعلقة.

### التشغيلات اليدوية

يعيد `openclaw cron run` النتيجة بمجرد وضع التشغيل اليدوي في قائمة الانتظار. تتضمن الاستجابات الناجحة `{ ok: true, enqueued: true, runId }`. استخدم `openclaw cron runs --id <job-id>` لمتابعة النتيجة النهائية.

<Note>
ينفذ `openclaw cron run <job-id>` تشغيلًا قسريًا افتراضيًا. استخدم `--due` للاحتفاظ بالسلوك الأقدم: "شغّل فقط إذا كان مستحقًا".
</Note>

## النماذج

يحدد `cron add|edit --model <ref>` نموذجًا مسموحًا به للمهمة.

<Warning>
إذا لم يكن النموذج مسموحًا به أو تعذر حله، يفشل cron التشغيل بخطأ تحقق صريح بدلًا من الرجوع إلى اختيار نموذج وكيل المهمة أو النموذج الافتراضي.
</Warning>

`--model` في Cron هو **أساسي المهمة**، وليس تجاوز `/model` لجلسة محادثة. يعني ذلك:

- لا تزال احتياطيات النماذج المهيأة تُطبق عندما يفشل نموذج المهمة المحدد.
- تستبدل `fallbacks` في حمولة كل مهمة قائمة الاحتياطيات المهيأة عند وجودها.
- تجعل قائمة الاحتياطيات الفارغة لكل مهمة (`fallbacks: []` في حمولة/واجهة API المهمة) تشغيل cron صارمًا.
- عندما تحتوي مهمة على `--model` لكن لا تكون هناك قائمة احتياطيات مهيأة، يمرر OpenClaw تجاوز احتياطي فارغًا صريحًا حتى لا يُلحق نموذج الوكيل الأساسي كهدف إعادة محاولة مخفي.

### أسبقية نموذج cron المعزول

يحل cron المعزول النموذج النشط بهذا الترتيب:

1. تجاوز خطاف Gmail.
2. `--model` لكل مهمة.
3. تجاوز نموذج جلسة cron المخزن (عندما يختار المستخدم واحدًا).
4. اختيار نموذج الوكيل أو النموذج الافتراضي.

### الوضع السريع

يتبع الوضع السريع في cron المعزول اختيار النموذج المباشر المحلول. يطبق إعداد النموذج `params.fastMode` افتراضيًا، لكن يظل تجاوز `fastMode` المخزن في الجلسة مقدّمًا على الإعداد.

### إعادة محاولات تبديل النموذج المباشر

إذا طرح تشغيل معزول `LiveSessionModelSwitchError`، يحفظ cron المزوّد والنموذج اللذين تم التبديل إليهما (وتجاوز ملف المصادقة الذي تم التبديل إليه عند وجوده) للتشغيل النشط قبل إعادة المحاولة. حلقة إعادة المحاولة الخارجية محدودة بإعادتَي محاولة للتبديل بعد المحاولة الأولية، ثم تتوقف بدل الدوران إلى الأبد.

## مخرجات التشغيل والرفض

### كبت الإقرارات القديمة

تكبت دورات cron المعزولة الردود القديمة التي لا تحتوي إلا على إقرار. إذا كانت النتيجة الأولى مجرد تحديث حالة مؤقت ولا توجد عملية وكيل فرعي لاحقة مسؤولة عن الإجابة النهائية، يعيد cron المطالبة مرة واحدة للحصول على النتيجة الحقيقية قبل التسليم.

### كبت الرمز الصامت

إذا أعاد تشغيل cron معزول الرمز الصامت فقط (`NO_REPLY` أو `no_reply`)، يكبت cron كلًا من التسليم الصادر المباشر ومسار الملخص الاحتياطي الموضوع في قائمة الانتظار، لذلك لا يُنشر أي شيء مرة أخرى إلى المحادثة.

### الرفض المنظم

تفضّل تشغيلات cron المعزولة بيانات تعريف رفض التنفيذ المنظمة من التشغيل المضمن، ثم تعود إلى علامات الرفض المعروفة في المخرجات النهائية، مثل `SYSTEM_RUN_DENIED` و`INVALID_REQUEST` وعبارات رفض ربط الموافقة.

يعرض `cron list` وسجل التشغيلات سبب الرفض بدلًا من الإبلاغ عن الأمر المحظور كـ `ok`.

## الاحتفاظ

يُتحكم في الاحتفاظ والتقليم في الإعداد:

- يقلّم `cron.sessionRetention` (الافتراضي `24h`) جلسات التشغيل المعزولة المكتملة.
- يقلّم `cron.runLog.maxBytes` و`cron.runLog.keepLines` الملف `~/.openclaw/cron/runs/<jobId>.jsonl`.

## ترحيل المهام الأقدم

<Note>
إذا كانت لديك مهام cron من قبل تنسيق التسليم والتخزين الحالي، شغّل `openclaw doctor --fix`. يطبّع Doctor حقول cron القديمة (`jobId` و`schedule.cron` وحقول التسليم على المستوى الأعلى بما في ذلك `threadId` القديم وأسماء `provider` المستعارة للتسليم في الحمولة) ويرحّل مهام احتياطي Webhook البسيطة ذات `notify: true` إلى تسليم Webhook صريح عندما يكون `cron.webhook` مهيأً.
</Note>

## تعديلات شائعة

حدّث إعدادات التسليم من دون تغيير الرسالة:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

عطّل التسليم لمهمة معزولة:

```bash
openclaw cron edit <job-id> --no-deliver
```

فعّل سياق تمهيد خفيف لمهمة معزولة:

```bash
openclaw cron edit <job-id> --light-context
```

أعلن إلى قناة محددة:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

أعلن إلى موضوع منتدى Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

أنشئ مهمة معزولة مع سياق تمهيد خفيف:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

ينطبق `--light-context` على مهام دورات الوكيل المعزولة فقط. بالنسبة إلى تشغيلات cron، يبقي الوضع الخفيف سياق التمهيد فارغًا بدلًا من حقن مجموعة تمهيد مساحة العمل الكاملة.

## أوامر إدارة شائعة

التشغيل اليدوي والفحص:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

تتضمن إدخالات `cron runs` تشخيصات التسليم مع هدف cron المقصود، والهدف المحلول، وإرسالات أداة الرسائل، واستخدام الاحتياطي، وحالة التسليم.

إعادة استهداف الوكيل والجلسة:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

تعديلات التسليم:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
