# `openclaw devices`

إدارة طلبات إقران الأجهزة والرموز المقيّدة بنطاق الجهاز.

## الأوامر

### `openclaw devices list`

يعرض طلبات الإقران المعلّقة والأجهزة المقترنة.

```
openclaw devices list
openclaw devices list --json
```

يعرض إخراج الطلبات المعلّقة الوصول المطلوب بجانب الوصول المعتمد الحالي للجهاز عندما يكون الجهاز مقترناً مسبقاً. يجعل هذا ترقيات النطاق/الدور صريحة بدلاً من أن تبدو وكأن الإقران قد فُقد.

### `openclaw devices remove <deviceId>`

يزيل إدخال جهاز مقترن واحد.

عندما تكون مصادقاً باستخدام رمز جهاز مقترن، يمكن للمتصلين غير المسؤولين إزالة إدخال جهازهم **الخاص بهم** فقط. تتطلب إزالة جهاز آخر `operator.admin`.

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

يمسح الأجهزة المقترنة دفعة واحدة.

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

يعتمد طلب إقران جهاز معلّقاً باستخدام `requestId` الدقيق. إذا حُذف `requestId` أو تم تمرير `--latest`، يطبع OpenClaw الطلب المعلّق المحدد فقط ثم يخرج؛ أعد تشغيل الاعتماد باستخدام معرف الطلب الدقيق بعد التحقق من التفاصيل.

<Note>
إذا أعاد جهاز محاولة الإقران مع تفاصيل مصادقة متغيرة (الدور أو النطاقات أو المفتاح العام)، يستبدل OpenClaw الإدخال المعلّق السابق ويصدر `requestId` جديداً. شغّل `openclaw devices list` مباشرة قبل الاعتماد لاستخدام المعرف الحالي.
</Note>

إذا كان الجهاز مقترناً مسبقاً وطلب نطاقات أوسع أو دوراً أوسع، يُبقي OpenClaw الاعتماد الحالي سارياً وينشئ طلب ترقية معلّقاً جديداً. راجع عمودي `Requested` و`Approved` في `openclaw devices list` أو استخدم `openclaw devices approve --latest` لمعاينة الترقية الدقيقة قبل اعتمادها.

إذا تم تكوين Gateway صراحةً باستخدام `gateway.nodes.pairing.autoApproveCidrs`، فيمكن اعتماد طلبات `role: node` لأول مرة من عناوين IP عميلة مطابقة قبل ظهورها في هذه القائمة. تكون هذه السياسة معطلة افتراضياً ولا تنطبق أبداً على عملاء المشغّل/المتصفح أو طلبات الترقية.

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

يرفض طلب إقران جهاز معلّقاً.

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

يدوّر رمز جهاز لدور محدد (مع تحديث النطاقات اختيارياً).
يجب أن يكون الدور الهدف موجوداً مسبقاً في عقد الإقران المعتمد لذلك الجهاز؛ لا يمكن للتدوير إصدار دور جديد غير معتمد.
إذا حذفت `--scope`، فإن عمليات إعادة الاتصال اللاحقة باستخدام الرمز المدوّر المخزن تعيد استخدام النطاقات المعتمدة المخزنة مؤقتاً لذلك الرمز. إذا مررت قيماً صريحة لـ `--scope`، فإنها تصبح مجموعة النطاقات المخزنة لعمليات إعادة الاتصال المستقبلية باستخدام الرمز المخزن مؤقتاً.
يمكن للمتصلين غير المسؤولين من الأجهزة المقترنة تدوير رمز جهازهم **الخاص بهم** فقط.
يجب أن تبقى مجموعة نطاقات الرمز الهدف ضمن نطاقات المشغّل الخاصة بجلسة المتصل نفسها؛ لا يمكن للتدوير إصدار رمز مشغّل أوسع أو الحفاظ عليه مما يملكه المتصل مسبقاً.

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

يعيد بيانات وصفية للتدوير بصيغة JSON. إذا كان المتصل يدوّر رمزه الخاص أثناء المصادقة باستخدام رمز ذلك الجهاز، فإن الاستجابة تتضمن أيضاً الرمز البديل حتى يتمكن العميل من حفظه قبل إعادة الاتصال. لا تعرض تدويرات المشاركة/المسؤول رمز الحامل.

### `openclaw devices revoke --device <id> --role <role>`

يلغي رمز جهاز لدور محدد.

يمكن للمتصلين غير المسؤولين من الأجهزة المقترنة إلغاء رمز جهازهم **الخاص بهم** فقط.
يتطلب إلغاء رمز جهاز آخر `operator.admin`.
يجب أن تتناسب مجموعة نطاقات الرمز الهدف أيضاً ضمن نطاقات المشغّل الخاصة بجلسة المتصل نفسها؛ لا يمكن للمتصلين المخصصين للإقران فقط إلغاء رموز مشغّل المسؤول/الكتابة.

```
openclaw devices revoke --device <deviceId> --role node
```

يعيد نتيجة الإلغاء بصيغة JSON.

## الخيارات الشائعة

- `--url <url>`: عنوان URL لـ WebSocket الخاص بـ Gateway (يستخدم `gateway.remote.url` افتراضياً عند تكوينه).
- `--token <token>`: رمز Gateway (إذا كان مطلوباً).
- `--password <password>`: كلمة مرور Gateway (مصادقة كلمة المرور).
- `--timeout <ms>`: مهلة RPC.
- `--json`: إخراج JSON (موصى به للبرمجة النصية).

<Warning>
عند ضبط `--url`، لا يعود CLI إلى بيانات اعتماد التكوين أو البيئة. مرّر `--token` أو `--password` صراحةً. غياب بيانات الاعتماد الصريحة خطأ.
</Warning>

## ملاحظات

- يعيد تدوير الرمز رمزاً جديداً (حساساً). عامله كسِرّ.
- تتطلب هذه الأوامر نطاق `operator.pairing` (أو `operator.admin`).
- `gateway.nodes.pairing.autoApproveCidrs` هي سياسة Gateway اختيارية لإقران أجهزة العقد الجديدة فقط؛ ولا تغيّر صلاحية اعتماد CLI.
- يبقى تدوير الرموز وإلغاؤها داخل مجموعة أدوار الإقران المعتمدة وخط أساس النطاق المعتمد لذلك الجهاز. لا يمنح إدخال رمز مخزن مؤقتاً شاذ هدفاً لإدارة الرموز.
- بالنسبة إلى جلسات رموز الأجهزة المقترنة، تكون الإدارة عبر الأجهزة مخصصة للمسؤولين فقط: `remove` و`rotate` و`revoke` ذاتية فقط ما لم يكن لدى المتصل `operator.admin`.
- تغيير الرمز محصور أيضاً بنطاق المتصل: لا يمكن لجلسة مخصصة للإقران فقط تدوير أو إلغاء رمز يحمل حالياً `operator.admin` أو `operator.write`.
- `devices clear` محمي عمداً بواسطة `--yes`.
- إذا كان نطاق الإقران غير متاح على local loopback (ولم يتم تمرير `--url` صريح)، فيمكن لـ list/approve استخدام آلية احتياطية محلية للإقران.
- يتطلب `devices approve` معرف طلب صريحاً قبل إصدار الرموز؛ حذف `requestId` أو تمرير `--latest` يعاين أحدث طلب معلّق فقط.

## قائمة تحقق لاسترداد انحراف الرمز

استخدم هذا عندما تستمر واجهة التحكم أو العملاء الآخرون بالفشل مع `AUTH_TOKEN_MISMATCH` أو `AUTH_DEVICE_TOKEN_MISMATCH`.

1. أكّد مصدر رمز Gateway الحالي:

```bash
openclaw config get gateway.auth.token
```

2. اعرض الأجهزة المقترنة وحدد معرف الجهاز المتأثر:

```bash
openclaw devices list
```

3. دوّر رمز المشغّل للجهاز المتأثر:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. إذا لم يكن التدوير كافياً، فأزل الإقران القديم واعتمده مجدداً:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. أعد محاولة اتصال العميل باستخدام الرمز/كلمة المرور المشتركة الحالية.

ملاحظات:

- أسبقية مصادقة إعادة الاتصال العادية هي الرمز/كلمة المرور المشتركة الصريحة أولاً، ثم `deviceToken` الصريح، ثم رمز الجهاز المخزن، ثم رمز bootstrap.
- يمكن لاسترداد `AUTH_TOKEN_MISMATCH` الموثوق إرسال كل من الرمز المشترك ورمز الجهاز المخزن معاً مؤقتاً لمحاولة إعادة واحدة محدودة.
