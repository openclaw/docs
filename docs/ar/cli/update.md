# `openclaw update`

حدّث OpenClaw بأمان وبدّل بين قنوات stable/beta/dev.

إذا ثبّت عبر **npm/pnpm/bun** (تثبيت عام، بلا بيانات git الوصفية)،
تتم التحديثات عبر مسار مدير الحزم في [التحديث](/ar/install/updating).

## الاستخدام

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## الخيارات

- `--no-restart`: تخطَّ إعادة تشغيل خدمة Gateway بعد نجاح التحديث. تحديثات مدير الحزم التي تعيد تشغيل Gateway تتحقق من أن الخدمة المعاد تشغيلها تبلّغ عن الإصدار المحدّث المتوقع قبل نجاح الأمر.
- `--channel <stable|beta|dev>`: اضبط قناة التحديث (git + npm؛ تُحفظ في الإعدادات).
- `--tag <dist-tag|version|spec>`: تجاوز هدف الحزمة لهذا التحديث فقط. بالنسبة إلى تثبيتات الحزم، يُطابق `main` القيمة `github:openclaw/openclaw#main`.
- `--dry-run`: عاين إجراءات التحديث المخطط لها (تدفق channel/tag/target/restart) من دون كتابة الإعدادات، أو التثبيت، أو مزامنة plugins، أو إعادة التشغيل.
- `--json`: اطبع JSON قابلًا للقراءة آليًا من نوع `UpdateRunResult`، بما في ذلك
  `postUpdate.plugins.warnings` عندما تحتاج plugins مُدارة تالفة أو غير قابلة للتحميل
  إلى إصلاح بعد نجاح تحديث النواة، وتفاصيل رجوع plugins في قناة beta
  عندما لا يكون لدى plugin إصدار beta، و`postUpdate.plugins.integrityDrifts`
  عند اكتشاف انحراف في أثر حزمة npm الخاصة بـ plugin أثناء مزامنة plugins بعد التحديث.
- `--timeout <seconds>`: مهلة لكل خطوة (القيمة الافتراضية هي 1800s).
- `--yes`: تخطَّ مطالبات التأكيد (مثل تأكيد الرجوع إلى إصدار أقدم).

لا يحتوي `openclaw update` على علم `--verbose`. استخدم `--dry-run` لمعاينة
إجراءات channel/tag/install/restart المخطط لها، و`--json` للحصول على
نتائج قابلة للقراءة آليًا، و`openclaw update status --json` عندما تحتاج فقط إلى
تفاصيل القناة والتوفر. إذا كنت تصحح سجلات Gateway حول تحديث،
فإن إسهاب وحدة التحكم ومستوى سجل الملفات منفصلان: يؤثر `--verbose` الخاص بـ Gateway
في إخراج الطرفية/WebSocket، بينما تتطلب سجلات الملفات `logging.level: "debug"` أو
`"trace"` في الإعدادات. راجع [تسجيل Gateway](/ar/gateway/logging).

<Note>
في وضع Nix (`OPENCLAW_NIX_MODE=1`)، تُعطَّل عمليات `openclaw update` التي تُحدث تغييرات. حدّث مصدر Nix أو إدخال flake لهذا التثبيت بدلًا من ذلك؛ بالنسبة إلى nix-openclaw، استخدم [البدء السريع](https://github.com/openclaw/nix-openclaw#quick-start) القائم على الوكيل أولًا. يظل `openclaw update status` و`openclaw update --dry-run` للقراءة فقط.
</Note>

<Warning>
تتطلب عمليات الرجوع إلى إصدار أقدم تأكيدًا لأن الإصدارات الأقدم يمكن أن تكسر الإعدادات.
</Warning>

## `update status`

اعرض قناة التحديث النشطة + وسم/فرع/SHA الخاص بـ git (لعمليات checkout من المصدر)، إضافة إلى توفر التحديث.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

الخيارات:

- `--json`: اطبع JSON حالة قابلًا للقراءة آليًا.
- `--timeout <seconds>`: مهلة الفحوصات (القيمة الافتراضية هي 3s).

## `update wizard`

تدفق تفاعلي لاختيار قناة تحديث وتأكيد ما إذا كان يجب إعادة تشغيل Gateway
بعد التحديث (الافتراضي هو إعادة التشغيل). إذا اخترت `dev` من دون git checkout، فإنه
يعرض إنشاء واحد.

الخيارات:

- `--timeout <seconds>`: مهلة لكل خطوة تحديث (القيمة الافتراضية `1800`)

## ما الذي يفعله

عند تبديل القنوات صراحةً (`--channel ...`)، يُبقي OpenClaw أيضًا
طريقة التثبيت متوافقة:

- `dev` → يضمن وجود git checkout (الافتراضي: `~/openclaw`، ويمكن تجاوزه باستخدام `OPENCLAW_GIT_DIR`)،
  ويحدّثه، ويثبّت CLI العام من ذلك checkout.
- `stable` → يثبّت من npm باستخدام `latest`.
- `beta` → يفضّل dist-tag الخاص بـ npm وهو `beta`، لكنه يرجع إلى `latest` عندما يكون beta
  مفقودًا أو أقدم من الإصدار المستقر الحالي.

يشغّل محدّث النواة التلقائي لـ Gateway (عند تمكينه عبر الإعدادات) مسار تحديث CLI
خارج معالج طلب Gateway الحي. تجبر تحديثات مدير الحزم عبر `update.run` في مستوى التحكم
إعادة تشغيل تحديث غير مؤجلة وبلا فترة تهدئة بعد تبديل الحزمة،
لأن عملية Gateway القديمة قد لا تزال تحتوي في الذاكرة على أجزاء تشير إلى
ملفات أزالتها الحزمة الجديدة.

بالنسبة إلى تثبيتات مدير الحزم، يحل `openclaw update` إصدار الحزمة الهدف
قبل استدعاء مدير الحزم. تستخدم تثبيتات npm العامة تثبيتًا مرحليًا:
يثبّت OpenClaw الحزمة الجديدة في بادئة npm مؤقتة، ويتحقق من
جرد `dist` المعبّأ هناك، ثم يبدّل شجرة الحزمة النظيفة تلك إلى
البادئة العامة الحقيقية. إذا فشل التحقق، فلن تُشغَّل أعمال doctor بعد التحديث، أو مزامنة plugins، أو
إعادة التشغيل من الشجرة المشتبه بها. حتى عندما يطابق الإصدار المثبّت
الهدف بالفعل، ينعش الأمر تثبيت الحزمة العامة،
ثم يشغّل مزامنة plugins، وتحديث إكمال أمر النواة، وأعمال إعادة التشغيل. يحافظ هذا
على اتساق الملحقات الجانبية المعبّأة وسجلات plugins المملوكة للقناة مع
بنية OpenClaw المثبّتة، مع ترك عمليات إعادة بناء إكمال أوامر plugins الكاملة
لعمليات `openclaw completion --write-state` الصريحة.

عندما تكون خدمة Gateway محلية مُدارة مثبّتة وتكون إعادة التشغيل ممكّنة،
توقف تحديثات مدير الحزم الخدمة العاملة قبل استبدال شجرة الحزمة،
ثم تنعش بيانات الخدمة الوصفية من التثبيت المحدّث، وتعيد تشغيل
الخدمة، وتتحقق من أن Gateway المعاد تشغيله يبلّغ عن الإصدار المتوقع قبل
الإبلاغ عن النجاح. على macOS، يتحقق فحص ما بعد التحديث أيضًا من أن LaunchAgent
محمّل/قيد التشغيل للملف الشخصي النشط وأن منفذ loopback المكوّن
سليم. إذا كان plist مثبّتًا لكن launchd لا يشرف عليه، يعيد OpenClaw
تمهيد LaunchAgent تلقائيًا، ثم يعيد تشغيل
فحوصات جاهزية الصحة/الإصدار/القناة. يحمّل التمهيد الجديد مهمة RunAtLoad
مباشرةً، لذلك لا تنفّذ استعادة التحديث فورًا `kickstart -k` على Gateway
الذي أُنشئ للتو. إذا ظل Gateway غير سليم، يخرج الأمر
برمز غير صفري ويطبع مسار سجل إعادة التشغيل إضافة إلى تعليمات صريحة لإعادة التشغيل، وإعادة التثبيت، و
التراجع عن الحزمة. مع `--no-restart`،
يظل استبدال الحزمة قيد التشغيل لكن الخدمة المُدارة لا تُوقف أو
تُعاد تشغيلها، لذلك قد يستمر Gateway العامل في استخدام كود قديم حتى تعيد تشغيله
يدويًا.

## تدفق git checkout

### اختيار القناة

- `stable`: اسحب أحدث وسم غير beta، ثم ابنِ وشغّل doctor.
- `beta`: فضّل أحدث وسم `-beta`، لكن ارجع إلى أحدث وسم مستقر عندما يكون beta مفقودًا أو أقدم.
- `dev`: اسحب `main`، ثم نفّذ fetch وrebase.

### خطوات التحديث

<Steps>
  <Step title="تحقق من نظافة شجرة العمل">
    يتطلب عدم وجود تغييرات غير ملتزم بها.
  </Step>
  <Step title="بدّل القناة">
    يبدّل إلى القناة المحددة (وسم أو فرع).
  </Step>
  <Step title="اجلب upstream">
    خاص بـ Dev فقط.
  </Step>
  <Step title="بناء تمهيدي (خاص بـ dev فقط)">
    يشغّل بناء TypeScript في شجرة عمل مؤقتة. إذا فشلت القمة، يرجع حتى 10 التزامات للعثور على أحدث التزام قابل للبناء. اضبط `OPENCLAW_UPDATE_PREFLIGHT_LINT=1` لتشغيل lint أيضًا أثناء هذا التمهيد؛ يعمل lint في وضع تسلسلي مقيّد لأن مضيفي تحديث المستخدمين غالبًا ما يكونون أصغر من مشغّلات CI.
  </Step>
  <Step title="نفّذ Rebase">
    ينفّذ rebase على الالتزام المحدد (خاص بـ dev فقط).
  </Step>
  <Step title="ثبّت الاعتماديات">
    يستخدم مدير حزم المستودع. بالنسبة إلى pnpm checkouts، يمهّد المحدّث `pnpm` عند الطلب (عبر `corepack` أولًا، ثم رجوع مؤقت إلى `npm install pnpm@11`) بدلًا من تشغيل `npm run build` داخل مساحة عمل pnpm.
  </Step>
  <Step title="ابنِ Control UI">
    يبني gateway وControl UI.
  </Step>
  <Step title="شغّل doctor">
    يعمل `openclaw doctor` كفحص التحديث الآمن النهائي.
  </Step>
  <Step title="زامن plugins">
    يزامن plugins إلى القناة النشطة. يستخدم Dev plugins المضمّنة؛ ويستخدم stable وbeta npm. يحدّث تثبيتات plugins المتتبعة.
  </Step>
</Steps>

على قناة تحديث beta، تحاول تثبيتات plugins الخاصة بـ npm وClawHub المتتبعة التي تتبع
المسار default/latest إصدار plugin باسم `@beta` أولًا. إذا لم يكن لدى plugin
إصدار beta، يرجع OpenClaw إلى مواصفة default/latest المسجلة ويبلّغ
عن ذلك كتحذير. بالنسبة إلى plugins الخاصة بـ npm، يرجع OpenClaw أيضًا عندما تكون حزمة beta
موجودة لكنها تفشل في تحقق التثبيت. لا تجعل تحذيرات رجوع plugins هذه
تحديث النواة يفشل. لا تُعاد كتابة الإصدارات الدقيقة والوسوم الصريحة.

<Warning>
إذا حُل تحديث plugin الخاصة بـ npm والمثبتة بإصدار دقيق إلى أثر تختلف سلامته عن سجل التثبيت المخزن، فإن `openclaw update` يوقف تحديث أثر ذلك plugin بدلًا من تثبيته. أعد تثبيت plugin أو حدّثه صراحةً فقط بعد التحقق من أنك تثق بالأثر الجديد.
</Warning>

<Note>
تُبلَّغ إخفاقات مزامنة plugins بعد التحديث والمحصورة في plugin مُدار كتحذيرات بعد نجاح تحديث النواة. تُبقي نتيجة JSON حالة التحديث ذات المستوى الأعلى `status: "ok"` وتبلّغ عن `postUpdate.plugins.status: "warning"` مع إرشادات `openclaw doctor --fix` و`openclaw plugins inspect <id> --runtime --json`. ما تزال استثناءات المحدّث أو المزامنة غير المتوقعة تفشل نتيجة التحديث. أصلح تثبيت plugin أو خطأ التحديث، ثم أعد تشغيل `openclaw doctor --fix` أو `openclaw update`.

عند بدء Gateway المحدّث، يكون تحميل plugins للتحقق فقط: لا يشغّل بدء التشغيل مديري الحزم أو يغيّر أشجار الاعتماديات. تتجاوز عمليات إعادة تشغيل مدير الحزم عبر `update.run` التأجيل العادي عند الخمول وفترة تهدئة إعادة التشغيل بعد تبديل شجرة الحزمة، حتى لا تتمكن العملية القديمة من الاستمرار في التحميل الكسول لأجزاء أُزيلت.

إذا ظل تمهيد pnpm يفشل، يتوقف المحدّث مبكرًا مع خطأ خاص بمدير الحزم بدلًا من محاولة `npm run build` داخل checkout.
</Note>

## اختصار `--update`

يعيد `openclaw --update` الكتابة إلى `openclaw update` (مفيد للأصداف وسكربتات التشغيل).
