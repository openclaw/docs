# `openclaw update`

حدّث OpenClaw بأمان وبدّل بين قنوات stable/beta/dev.

إذا ثبّت عبر **npm/pnpm/bun** (تثبيت عام، بلا بيانات git وصفية)،
تحدث التحديثات عبر مسار مدير الحزم في [التحديث](/ar/install/updating).

## الاستخدام

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## الخيارات

- `--no-restart`: تجاوز إعادة تشغيل خدمة Gateway بعد نجاح التحديث. تحديثات مدير الحزم التي تعيد تشغيل Gateway تتحقق من أن الخدمة المعاد تشغيلها تبلغ عن الإصدار المحدّث المتوقع قبل نجاح الأمر.
- `--channel <stable|beta|dev>`: عيّن قناة التحديث (git + npm؛ تُحفظ في الإعدادات).
- `--tag <dist-tag|version|spec>`: تجاوز هدف الحزمة لهذا التحديث فقط. بالنسبة إلى تثبيتات الحزم، يشير `main` إلى `github:openclaw/openclaw#main`.
- `--dry-run`: اعرض معاينة لإجراءات التحديث المخططة (channel/tag/target/restart flow) دون كتابة الإعدادات أو التثبيت أو مزامنة Plugins أو إعادة التشغيل.
- `--json`: اطبع JSON قابلًا للقراءة آليًا من نوع `UpdateRunResult`، بما في ذلك
  `postUpdate.plugins.integrityDrifts` عند اكتشاف انحراف في أثر Plugin الخاص بـ npm
  أثناء مزامنة Plugin بعد التحديث.
- `--timeout <seconds>`: مهلة لكل خطوة (القيمة الافتراضية 1800 ثانية).
- `--yes`: تجاوز مطالبات التأكيد (مثل تأكيد الرجوع إلى إصدار أقدم).

<Warning>
تتطلب الرجوعات إلى إصدارات أقدم تأكيدًا لأن الإصدارات الأقدم قد تكسر الإعدادات.
</Warning>

## `update status`

اعرض قناة التحديث النشطة + وسم/فرع/SHA الخاص بـ git (لنسخ المصدر)، إضافة إلى توافر التحديث.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

الخيارات:

- `--json`: اطبع JSON حالة قابلًا للقراءة آليًا.
- `--timeout <seconds>`: مهلة الفحوصات (القيمة الافتراضية 3 ثوانٍ).

## `update wizard`

مسار تفاعلي لاختيار قناة تحديث وتأكيد ما إذا كان يجب إعادة تشغيل Gateway
بعد التحديث (القيمة الافتراضية هي إعادة التشغيل). إذا اخترت `dev` دون نسخة git، فإنه
يعرض إنشاء واحدة.

الخيارات:

- `--timeout <seconds>`: مهلة لكل خطوة تحديث (الافتراضي `1800`)

## ما يفعله

عند تبديل القنوات صراحةً (`--channel ...`)، يحافظ OpenClaw أيضًا على
اتساق طريقة التثبيت:

- `dev` → يضمن وجود نسخة git (الافتراضي: `~/openclaw`، ويمكن التجاوز باستخدام `OPENCLAW_GIT_DIR`)،
  ويحدّثها، ثم يثبّت CLI العام من تلك النسخة.
- `stable` → يثبّت من npm باستخدام `latest`.
- `beta` → يفضّل وسم توزيع npm‏ `beta`، لكنه يعود إلى `latest` عندما يكون beta
  مفقودًا أو أقدم من الإصدار المستقر الحالي.

يستخدم المحدّث التلقائي لنواة Gateway (عند تمكينه عبر الإعدادات) مسار التحديث نفسه.

بالنسبة إلى تثبيتات مدير الحزم، يحل `openclaw update` إصدار الحزمة الهدف
قبل استدعاء مدير الحزم. تستخدم تثبيتات npm العامة تثبيتًا مرحليًا:
يثبّت OpenClaw الحزمة الجديدة في بادئة npm مؤقتة، ويتحقق من مخزون `dist`
المعبأ هناك، ثم يستبدل شجرة الحزمة النظيفة تلك في البادئة العامة الحقيقية.
إذا فشل التحقق، لا تُشغّل أعمال الطبيب بعد التحديث، ومزامنة Plugin، وإعادة
التشغيل من الشجرة المشكوك فيها. حتى عندما يطابق الإصدار المثبّت الهدف بالفعل،
ينعش الأمر تثبيت الحزمة العامة، ثم يشغّل مزامنة Plugin، وتحديث إكمال أمر
النواة، وأعمال إعادة التشغيل. يحافظ هذا على اتساق المكونات الجانبية المعبأة
وسجلات Plugin المملوكة للقناة مع بناء OpenClaw المثبّت، مع ترك إعادة بناء
إكمال أوامر Plugin الكاملة لعمليات `openclaw completion --write-state`
الصريحة.

عند تثبيت خدمة Gateway محلية مُدارة وتمكين إعادة التشغيل، توقف تحديثات مدير
الحزم الخدمة الجارية قبل استبدال شجرة الحزمة، ثم تنعش بيانات الخدمة الوصفية
من التثبيت المحدّث، وتعيد تشغيل الخدمة، وتتحقق من أن Gateway المعاد تشغيله
يبلغ عن الإصدار المتوقع. مع `--no-restart`، يستمر استبدال الحزمة لكن الخدمة
المُدارة لا تُوقف ولا تُعاد تشغيلها، لذلك قد يواصل Gateway الجاري استخدام
الكود القديم إلى أن تعيد تشغيله يدويًا.

## مسار نسخة git

### اختيار القناة

- `stable`: انتقل إلى أحدث وسم غير تجريبي، ثم ابنِ وشغّل الطبيب.
- `beta`: فضّل أحدث وسم `-beta`، لكن عُد إلى أحدث وسم مستقر عندما يكون beta مفقودًا أو أقدم.
- `dev`: انتقل إلى `main`، ثم اجلب وأعد تأسيس التغييرات.

### خطوات التحديث

<Steps>
  <Step title="تحقق من نظافة شجرة العمل">
    يتطلب عدم وجود تغييرات غير ملتزم بها.
  </Step>
  <Step title="بدّل القناة">
    يبدّل إلى القناة المحددة (وسم أو فرع).
  </Step>
  <Step title="اجلب من upstream">
    لـ dev فقط.
  </Step>
  <Step title="بناء تمهيدي (لـ dev فقط)">
    يشغّل الفحص وبناء TypeScript في شجرة عمل مؤقتة. إذا فشل الطرف، يرجع حتى 10 التزامات للعثور على أحدث بناء نظيف.
  </Step>
  <Step title="أعد التأسيس">
    يعيد التأسيس على الالتزام المحدد (لـ dev فقط).
  </Step>
  <Step title="ثبّت الاعتماديات">
    يستخدم مدير حزم المستودع. بالنسبة إلى نسخ pnpm، يمهّد المحدّث `pnpm` عند الطلب (عبر `corepack` أولًا، ثم بديل مؤقت `npm install pnpm@10`) بدلًا من تشغيل `npm run build` داخل مساحة عمل pnpm.
  </Step>
  <Step title="ابنِ Control UI">
    يبني Gateway وControl UI.
  </Step>
  <Step title="شغّل الطبيب">
    يعمل `openclaw doctor` كفحص التحديث الآمن النهائي.
  </Step>
  <Step title="زامن Plugins">
    يزامن Plugins مع القناة النشطة. يستخدم dev الـ Plugins المضمنة؛ ويستخدم stable وbeta‏ npm. يحدّث Plugins المثبتة عبر npm.
  </Step>
</Steps>

<Warning>
إذا حل تحديث Plugin الخاص بـ npm والمثبت بدقة إلى أثر تختلف سلامته عن سجل التثبيت المخزّن، فإن `openclaw update` يلغي تحديث أثر Plugin هذا بدلًا من تثبيته. أعد تثبيت Plugin أو حدّثه صراحةً فقط بعد التحقق من أنك تثق بالأثر الجديد.
</Warning>

<Note>
فشل مزامنة Plugin بعد التحديث يفشل نتيجة التحديث ويوقف أعمال متابعة إعادة التشغيل. أصلح خطأ تثبيت Plugin أو تحديثه، ثم أعد تشغيل `openclaw update`.

عند بدء Gateway المحدّث، تُحضّر اعتماديات وقت تشغيل Plugin المضمنة والممكّنة قبل تفعيل Plugin. عمليات إعادة التشغيل الناتجة عن التحديث تستنزف أي تحضير نشط لاعتماديات وقت التشغيل قبل إغلاق Gateway، لذلك لا تقاطع عمليات إعادة التشغيل من مدير الخدمة تثبيت npm جارٍ.

إذا ظل تمهيد pnpm يفشل، يتوقف المحدّث مبكرًا مع خطأ خاص بمدير الحزم بدلًا من محاولة `npm run build` داخل نسخة المصدر.
</Note>

## اختصار `--update`

يعيد `openclaw --update` الكتابة إلى `openclaw update` (مفيد للأصداف وسكربتات المشغّل).
