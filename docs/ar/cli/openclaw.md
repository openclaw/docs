# `openclaw setup`

يأتي OpenClaw مزودًا بوكيل نظام مدمج — يتحدث باسم "OpenClaw" — للإعداد المحلي والإصلاح والتهيئة (كان يُسمى سابقًا Crestodian). ولا يبدأ إلا بعد أن يُكمل النموذج الافتراضي الفعلي دورة حقيقية.
تُنشئ عمليات التثبيت الجديدة الاستدلال أولًا؛ وتبقى التهيئة غير الصالحة ضمن
مسار doctor التقليدي.

## متى يبدأ

يُوجّه تشغيل `openclaw` من دون أمر فرعي وفقًا لحالة التهيئة:

- التهيئة مفقودة، أو موجودة من دون إعدادات أنشأها المستخدم (فارغة، أو لا تحتوي إلا على مفاتيح `$schema`/`meta`): يبدأ الإعداد الأولي الموجّه مع تحقق مباشر بالذكاء الاصطناعي.
- التهيئة موجودة لكنها تفشل في التحقق: يبدأ الإعداد الأولي التقليدي، الذي يبلّغ عن المشكلات ويوجّهك إلى `openclaw doctor`.
- التهيئة موجودة وصالحة: يفتح واجهة TUI العادية للوكيل. ينتقل Gateway
  المهيأ والقابل للوصول، والذي يملك وكيله الافتراضي نموذجًا، مباشرةً إلى تلك الواجهة
  من دون إعداد أولي أو OpenClaw. استخدم `/openclaw` داخل TUI، أو شغّل
  `openclaw setup` مباشرةً، للوصول إلى OpenClaw لاحقًا.

يختبر تشغيل `openclaw setup` أولًا النموذج الافتراضي المهيأ اختبارًا مباشرًا. وتؤدي الدورة الناجحة إلى بدء OpenClaw. أما الفشل التفاعلي فيفتح إعداد الاستدلال الموجّه ويسلّم التحكم إلى OpenClaw بعد نجاح أحد الخيارات المرشحة. تفشل طلبات المرة الواحدة وJSON وغيرها من الطلبات غير التفاعلية مع تعليمات لتشغيل `openclaw onboard` عندما لا يكون الاستدلال متاحًا. يحتفظ `openclaw --help` و`openclaw --version` بمساريهما السريعين المعتادين.

يخرج تشغيل `openclaw` المجرد وغير التفاعلي (من دون TTY) برسالة قصيرة بدلًا من طباعة مساعدة الجذر: إذ يشير إلى الإعداد الأولي غير التفاعلي في التثبيت الجديد أو غير الصالح، أو إلى `openclaw agent --local ...` عندما تكون التهيئة صالحة.

يبقى `openclaw onboard --modern` اسمًا مستعارًا متوافقًا لـ OpenClaw، لكنه يستخدم بوابة الاستدلال نفسها: يفتح الاستدلال العامل المحادثة، وتبدأ حالات الفشل التفاعلية إعداد الاستدلال الموجّه، بينما تخرج حالات الفشل غير التفاعلية مع إرشادات الإعداد الأولي. يفتح `openclaw onboard --classic` المعالج الكامل خطوة بخطوة.

## ما يعرضه OpenClaw

يفتح OpenClaw التفاعلي غلاف TUI نفسه الذي يفتحه `openclaw tui`، مع خلفية محادثة OpenClaw. وتغطي تحية بدء التشغيل ما يلي:

- صلاحية التهيئة والوكيل الافتراضي
- النموذج المتحقق منه الذي يستخدمه OpenClaw
- إمكانية الوصول إلى Gateway من أول فحص عند بدء التشغيل
- إجراء تصحيح الأخطاء التالي الموصى به

ولا يعرض الأسرار أو يحمّل أوامر CLI الخاصة بالـ plugins لمجرد بدء التشغيل.

استخدم `status` للحصول على الجرد التفصيلي: مسار التهيئة، ومسارات الوثائق/المصدر، وفحوصات CLI المحلية، ووجود المفاتيح/الرموز، والوكلاء، والنموذج، وتفاصيل Gateway.

يستخدم OpenClaw اكتشاف المراجع نفسه الذي تستخدمه الوكلاء العادية: ففي نسخة Git يشير إلى `docs/` المحلي وشجرة المصدر؛ وفي تثبيت npm يستخدم الوثائق المضمّنة ويربط بـ [https://github.com/openclaw/openclaw](https://github.com/openclaw/openclaw)، مع إرشادات للتحقق من المصدر عندما لا تكفي الوثائق.

## العمليات والموافقة

يستخدم OpenClaw عمليات محددة الأنواع بدلًا من تعديل التهيئة ارتجالًا.

تُشغّل عمليات القراءة فقط فورًا: عرض النظرة العامة، وسرد الوكلاء، وسرد plugins المثبتة، والبحث في plugins على ClawHub، وعرض حالة النموذج/الخلفية، وتشغيل فحوصات الحالة/الصحة، والتحقق من إمكانية الوصول إلى Gateway، وتشغيل doctor من دون إصلاحات تفاعلية، والتحقق من التهيئة، وعرض مسار سجل التدقيق.

يُشغّل بدء إعداد القناة الموجّه (`connect telegram`) فورًا أيضًا. يجمع معالجه إجابات صريحة ويتولى عمليات الكتابة الناتجة.

تتطلب العمليات الدائمة موافقة ضمن المحادثة (أو `--yes` لأمر مباشر): كتابة التهيئة، و`config set`، و`config set-ref`، وتهيئة الإعداد/الإعداد الأولي، وتغيير النموذج الافتراضي، وبدء/إيقاف/إعادة تشغيل Gateway، وإنشاء الوكلاء، وتثبيت plugins.

لا تتوفر إصلاحات doctor داخل OpenClaw لأنها قد تعيد كتابة مسار استدلال الموفّر أو المصادقة أو الوكيل الافتراضي الذي يشغّل الجلسة. اخرج من OpenClaw وشغّل `openclaw doctor --fix` في طرفية. يظل `doctor` المخصص للقراءة فقط متاحًا داخل OpenClaw.

ترث الوكلاء الجديدة مسار الاستدلال الافتراضي المتحقق منه مباشرةً. معرّفا الوكيل `openclaw` و`crestodian` محجوزان لوكيل النظام ولا يمكن إنشاء وكيلين عاديين بهما. ويظل المعرّف المتقاعد محظورًا كي لا تتمكن تهيئة قديمة من الاستحواذ عليه.

لا يمكن لـ `config set` و`config set-ref` تغيير حالة مسار الاستدلال،
بما في ذلك بيانات اعتماد موفّر الاستدلال، و`auth.*` ذي المستوى الأعلى، وكتالوجات النماذج،
وخلفيات CLI، ومسارات النماذج الافتراضية/الخاصة بكل وكيل، ومعلمات/أدوات الوكيل، أو
`tools.*` الجذري. وتُرفض أيضًا عمليات الكتابة الخام ضمن `env.*`، و`secrets.*`، و`plugins.*`، و`$include`
لأنها قد تستبدل آلية حل بيانات الاعتماد أو تفعيل الموفّر. تظل مصادقة Gateway والقنوات ضمن أسطح التهيئة العادية. استخدم مسارات عمل plugins/القنوات محددة الأنواع و
`set default model <provider/model>` لمسار مهيأ
بالفعل؛ إذ يختبر المسار مباشرةً قبل حفظه. لتهيئة الوصول إلى الموفّر/المصادقة أو
إصلاحه، اخرج من OpenClaw وشغّل `openclaw onboard`.

يُرفض إلغاء تثبيت Plugin داخل OpenClaw لأن إزالة Plugin خاص بموفّر
قد تعطّل مسار الاستدلال الذي يشغّل الجلسة. اخرج من OpenClaw
وشغّل `openclaw plugins uninstall <id>` من طرفية.

تُقدّم الموافقة بكلماتك الخاصة: تُحسم الردود غير الملتبسة ("نعم"، "بالتأكيد"، "تابع"، "ليس الآن") من قائمة حتمية مغلقة. عندما يدعم المسار المهيأ استدعاء إكمال منفصلًا، يمكن تصنيف الردود الأخرى بالاعتماد على رسالتك والاقتراح المعلّق فقط — وليس بواسطة نموذج المحادثة نفسه مطلقًا، إذ لا يمكنه الموافقة على نفسه. وتُبقي الردود غير المصنفة أو الملتبسة الاقتراح معلقًا وتطلب المحادثة الإجابة مرة أخرى.

تُسجّل عمليات الكتابة المطبّقة في `~/.openclaw/audit/system-agent.jsonl`. لا يخضع الاكتشاف للتدقيق؛ بل العمليات وعمليات الكتابة المطبّقة فقط.

يمكن تشغيل إعداد القناة كمحادثة مستضافة حتى يصل إلى سر. لا تقبل واجهة
TUI المحلية لـ OpenClaw إجابات المعالج الحساسة لأن إدخال المحادثة في الطرفية
مرئي. وتعرض `open channel wizard` فورًا، مع تمرير
القناة المحددة إلى معالج الطرفية ذي الإدخال المحجوب؛ ويمكن أيضًا تشغيل
`openclaw channels add --channel <channel>` لاحقًا.

### التبديل إلى إعداد القناة ذي الإدخال المحجوب

يمكن للمحادثة المحلية تسليم التحكم إلى معالج القناة ذي الإدخال المحجوب:

```text
فتح معالج القناة لـ slack
معلومات القناة slack
```

يفتح `open channel wizard for <channel>` إعداد القناة ذا الإدخال المحجوب بعد إغلاق
واجهة TUI للمحادثة. استخدم `channel info <channel>` أولًا للحصول على تسمية القناة، وحالة الإعداد،
وملخص المتطلبات الأساسية، ورابط الوثائق.

لا يغيّر OpenClaw مطلقًا الوصول إلى الموفّر/المصادقة من داخل جلسته نفسها:
فالجلسة تعتمد بالفعل على مسار الاستدلال هذا. ولإعداد موفّر النموذج أو
إصلاحه، يعيد `configure model provider` إرشادات الخروج/الإعداد الأولي من دون
بدء معالج أو كتابة التهيئة. اخرج من OpenClaw وشغّل `openclaw
onboard`؛ إذ يجهّز الإعداد الأولي بيانات الاعتماد ولا يحفظ إلا مسارًا
يُكمل دورة حقيقية مباشرةً. ابدأ OpenClaw مرة أخرى بعد نجاح الإعداد الأولي.

## تمهيد الإعداد

يهيئ `setup` ما تبقى من حالة مساحة العمل وGateway بعد أن يكون الإعداد الأولي الموجّه قد أنشأ الاستدلال بالفعل. ولا يكتب إلا عبر عمليات تهيئة محددة الأنواع ويطلب الموافقة أولًا.

```text
الإعداد
إعداد مساحة العمل ~/Projects/work
```

يحافظ `setup` على النموذج الفعلي المتحقق منه. ولا يهيئ
الاستدلال أو يستبدله.

إذا كان الاستدلال مفقودًا أو فشل فحصه المباشر، فاترك OpenClaw وشغّل `openclaw onboard`. يكتشف الإعداد الأولي الموجّه النماذج المهيأة، ومفاتيح API، وواجهات CLI المحلية المصادق عليها، ويطلب ردًا حقيقيًا من كل خيار مرشح، ولا يحفظ إلا مسارًا ناجحًا. يبدأ OpenClaw فور تجاوز تلك الحدود، ويمكنه بعد ذلك تهيئة مساحة العمل، وGateway، والقنوات، والوكلاء، وplugins، والميزات الاختيارية الأخرى.

يتجاوز تطبيق macOS هذا التسلسل بالكامل عندما يصل إلى Gateway مهيأ
يملك وكيله الافتراضي نموذجًا مهيأ بالفعل؛ ويفتح واجهة الوكيل
العادية.
أما في Gateway جديد أو غير مكتمل، فيدير التطبيق تسلسل الاستدلال عبر
طريقتي Gateway ‏`openclaw.setup.detect` و`openclaw.setup.activate`:
تسرد عملية الاكتشاف كل خلفية مرشحة تعثر عليها، ويختبر التفعيل خيارًا
مرشحًا واحدًا مباشرةً (عملية إكمال حقيقية بطلب "الرد بكلمة OK")، ولا يحفظ إلا النموذج
وبيانات الاعتماد وحالة الموفّر/بيئة التشغيل اللازمة لذلك المسار بعد نجاح الاختبار. تظل الإعدادات الافتراضية لمساحة العمل وGateway من اختصاص OpenClaw. لا يغيّر الخيار المرشح الفاشل
التهيئة مطلقًا؛ إذ ينتقل التطبيق تلقائيًا عبر التسلسل ويعرض في النهاية
خطوة يدوية للمفتاح/الرمز معبأة من plugins الخاصة بموفّري
استدلال النص النشطين في Gateway. يملك الموفّر المحدد نموذجه الابتدائي
وتهيئته، ويُتحقق من بيانات الاعتماد بالطريقة نفسها قبل حفظها.

تبقى مراقبة Codex وغيرها من ميزات plugins الاختيارية خارج
معاملة تفعيل الاستدلال هذه. لا تهيئها إلا بعد أن يعمل الاستدلال
ويبدأ OpenClaw؛ وتظل سياسة Plugin الحالية وعمليات إلغاء الاشتراك الصريحة
في المراقبة دون تغيير أثناء إعداد الاستدلال.

## محادثة الذكاء الاصطناعي

تعمل محادثة OpenClaw التفاعلية حرة الصياغة عبر حلقة الوكيل نفسها التي تستخدمها وكلاء OpenClaw العادية، مع تقييدها بأداة صلاحية OpenClaw واحدة من الحلقة صفر، وهي `openclaw`، التي تغلف العمليات محددة الأنواع. تعمل إجراءات القراءة بحرية، وتتطلب التغييرات موافقتك ضمن المحادثة على تلك العملية بعينها (راجع العمليات والموافقة)، وتخضع كل عملية كتابة مطبقة للتدقيق وإعادة التحقق. تستمر جلسة الوكيل، لذا يمتلك OpenClaw ذاكرة حقيقية متعددة الدورات. إذا توقف مسار الاستدلال المتحقق منه عن العمل لاحقًا، فارجع إلى `openclaw onboard` وأصلحه قبل المتابعة.

لا يحلل المضيف طلبات اللغة الطبيعية ويحوّلها إلى عمليات. فالرسائل حرة الصياغة
— بما في ذلك النصوص التي تبدو كأوامر والأسئلة مثل "لماذا توقف
Gateway لدي؟" — تُرسل إلى الذكاء الاصطناعي، الذي يمكنه ربط الطلب بعملية محددة الأنواع
من خلال أداة `openclaw`.

عندما يكون هناك تغيير معلّق، لا تُحسم من دون استدلال إلا عبارات الموافقة أو الرفض
غير الملتبسة الواردة في قائمة مغلقة. تُرسل الموافقة الملتبسة إلى
استدعاء إكمال منفصل مهيأ، وإلا تُرفض افتراضيًا. حقول
المعالج المنظّمة والتنقل الدقيق للمضيف عناصر تحكم في واجهة المستخدم، وليست تحليلًا
للعمليات بلغة طبيعية. وهناك استثناء مهم خصوصًا لنظافة الأسرار: لا يصل
`config set` المطابق تمامًا على مسار حساس (الرموز، والمفاتيح، وكلمات المرور)
إلى نموذج مطلقًا. ينشئ المضيف اقتراحًا منقحًا، وتُحجب القيمة في
السجل المرئي للذكاء الاصطناعي. يُفضّل استخدام `config set-ref <path> env <ENV_VAR>` للأسرار.

لا يستخدم وضع الإنقاذ عبر قنوات الرسائل المخطط المدعوم بالنموذج مطلقًا. يظل الإنقاذ عن بُعد حتميًا كي لا يُستخدم مسار الوكيل العادي المعطّل أو المخترق محررًا للتهيئة.

### نموذج الثقة لحزمة اختبار CLI

تفرض بيئات التشغيل المضمّنة وحزمة تشغيل app-server الخاصة بـ Codex تقييد الحلقة صفر
مباشرةً: يحمل التشغيل قائمة سماح بأدوات OpenClaw لا تتضمن سوى أداة
`openclaw`. وبالنسبة إلى Codex، يعطّل OpenClaw أيضًا البيئات والتنفيذ
الأصلي والوكلاء المتعددين والأهداف والتطبيقات/Plugins وSkills/MCP والبحث على الويب
وواجهات `request_user_input` لذلك التشغيل. يظل Codex يحقن أداة `update_plan`
الأصلية الخاملة؛ إذ يمكنها تحديث قائمة التحقق المؤقتة للنموذج، لكنها لا تستطيع كتابة الملفات
أو إعدادات OpenClaw. لا تستهلك حزم تشغيل CLI قائمة السماح الخاصة بـ OpenClaw،
لذلك لا يقبل OpenClaw إلا الخلفيات التي يستطيع عقد اختيار الأدوات الخاص بها إثبات
التقييد نفسه:

- تُشغَّل الخلفيات القابلة للاختيار، بما فيها Claude Code، مع اختيار فارغ للأدوات الأصلية
  وأداة MCP واحدة، هي `openclaw`. ويُطبَّق إعداد MCP المُنشأ لـ Claude باستخدام
  `--strict-mcp-config`، بحيث لا تُحمَّل أي خوادم MCP أخرى.
- تتلقى الخلفيات التي تعلن عدم امتلاكها أدوات أصلية خادم MCP المخصص نفسه
  من OpenClaw.
- تفشل الخلفيات ذات الأدوات الأصلية الدائمة التشغيل أو غير المعروفة بشكل مغلق قبل الاستدلال؛
  ولا يمكنها استضافة جلسة OpenClaw.

لا تحصل على خادم openclaw MCP إلا جلسات OpenClaw؛ أما عمليات تشغيل الوكيل العادية
فلا ترى هذه الأداة مطلقًا. ولذلك تفرض خلفيات CLI القابلة للاختيار/الخالية من الأدوات الأصلية ونماذج
مفتاح API حلقة الأداة الواحدة حرفيًا. وتفرض نماذج app-server الخاصة بـ Codex
أداة صلاحية واحدة لـ OpenClaw بالإضافة إلى أداة التخطيط الأصلية الخاملة. وفي الحالات
الثلاث جميعها، تظل عمليات كتابة الإعداد محصورة ضمن عقد الموافقة المدقَّق
الخاص بـ OpenClaw.

يظل Gemini CLI متاحًا للوكلاء العاديين، لكنه لا يستطيع فرض
الاختبار الخالي من الأدوات الذي تتطلبه بوابة الاستدلال، ولذلك لا يمكنه استضافة OpenClaw.

## التبديل إلى وكيل

استخدم محددًا بلغة طبيعية لمغادرة OpenClaw وفتح TUI العادية:

```text
تحدث إلى الوكيل
تحدث إلى وكيل العمل
بدّل إلى الوكيل الرئيسي
```

تفتح `openclaw tui` و`openclaw chat` و`openclaw terminal` واجهة TUI العادية للوكيل مباشرةً؛ ولا تبدأ OpenClaw. بعد التبديل إلى TUI العادية، يعيدك `/openclaw` إلى OpenClaw، مع طلب متابعة اختياري:

```text
/openclaw
/openclaw restart gateway
```

## وضع إنقاذ الرسائل

وضع إنقاذ الرسائل هو نقطة دخول قناة الرسائل إلى OpenClaw: استخدمه عندما يتعطل وكيلك العادي، لكن تظل قناة موثوقة (مثل WhatsApp) تتلقى الأوامر.

هذا معالج أوامر طوارئ حتمي، وليس وكيل OpenClaw
التحاوري. ولا يُمهّد إعدادًا جديدًا أو يخفف بوابة الاستدلال
لدردشة OpenClaw.

الأمر المدعوم: `/openclaw <request>`. لا يقبل الإنقاذ إلا صياغة الأمر المكتوب المطابقة تمامًا — وتُرفض اللغة الطبيعية مع تلميح، ولا تُخمَّن أبدًا لتحويلها إلى عملية، ولا يُستشار أي نموذج مطلقًا.

```text
أنت، في رسالة مباشرة موثوقة من المالك: /openclaw status
OpenClaw: وضع إنقاذ OpenClaw. يمكن الوصول إلى Gateway: لا. الإعداد صالح: لا.
أنت: /openclaw restart gateway
OpenClaw: الخطة: إعادة تشغيل Gateway. أرسل /openclaw yes للتطبيق.
أنت: /openclaw yes
OpenClaw: تم التطبيق. كُتب إدخال التدقيق.
```

يمكن أيضًا وضع إنشاء الوكيل في قائمة انتظار محليًا أو عبر الإنقاذ:

```text
create agent work workspace ~/Projects/work model openai/gpt-5.6-sol
/openclaw create agent work workspace ~/Projects/work
```

لا يجوز عند إنشاء الوكيل تسمية سوى النموذج الافتراضي الحالي المتحقق منه مباشرةً. احذف
النموذج لوراثة ذلك المسار.

الإنقاذ عن بُعد واجهة إدارية ويجب التعامل معه مثل إصلاح الإعداد عن بُعد، لا كمحادثة عادية.

عقد الأمان للإنقاذ عن بُعد:

- يُعطَّل عندما يكون العزل مفعّلًا للوكيل/الجلسة؛ يرفض OpenClaw الإنقاذ عن بُعد ويوجّه إلى الإصلاح المحلي عبر CLI.
- الحالة الفعلية الافتراضية هي `auto`: لا تسمح بالإنقاذ عن بُعد إلا في تشغيل YOLO موثوق، حيث تمتلك بيئة التشغيل بالفعل صلاحية محلية من دون عزل (تُحل `tools.exec.security` إلى `full` وتُحل `tools.exec.ask` إلى `off`، مع وضع العزل `off`).
- يتطلب هوية مالك صريحة؛ ولا يُسمح بقواعد مرسل ذات أحرف بدل، أو سياسة مجموعات مفتوحة، أو Webhooks غير مصادَق عليها، أو قنوات مجهولة.
- يقتصر افتراضيًا على الرسائل المباشرة للمالك؛ ويتطلب الإنقاذ عبر المجموعات/القنوات اشتراكًا صريحًا.
- البحث عن Plugins وعرض قائمتها للقراءة فقط. تثبيت Plugin محلي فقط دائمًا (ومحظور في الإنقاذ حتى عندما يكون مفعّلًا بخلاف ذلك) لأنه ينزّل شيفرة قابلة للتنفيذ. ويُرفض إلغاء تثبيت Plugin في كل من OpenClaw المحلي والإنقاذ؛ شغّل `openclaw plugins uninstall <id>` من طرفية.
- لا يستطيع الإنقاذ عن بُعد فتح TUI المحلية أو التبديل إلى جلسة وكيل تفاعلية؛ استخدم `openclaw` محليًا لتسليم المهمة إلى الوكيل.
- لا تزال عمليات الكتابة الدائمة تتطلب موافقة، حتى في وضع الإنقاذ.
- الموافقات المعلّقة للاستخدام مرة واحدة. يلغي أي أمر إنقاذ أحدث للحساب والقناة والمرسل أنفسهم الخطة الأقدم؛ كما يستهلك فشل التنفيذ الموافقة، لذا أعد إرسال الأمر للمحاولة مجددًا.
- تُدقَّق كل عملية إنقاذ مطبّقة. يسجّل إنقاذ قناة الرسائل بيانات القناة والحساب والمرسل وعنوان المصدر الوصفية؛ كما تسجّل العمليات التي تعدّل الإعداد تجزئات الإعداد قبل التعديل وبعده.
- لا تُعرض الأسرار مطلقًا. يفيد فحص SecretRef بالتوفر، لا بالقيم.
- إذا كان Gateway يعمل، يفضّل الإنقاذ عمليات Gateway المنمّطة؛ وإذا كان متوقفًا، فلا يستخدم الإنقاذ إلا الحد الأدنى من واجهة الإصلاح المحلية التي لا تعتمد على حلقة الوكيل العادية.

بنية الإعداد:

```jsonc
{
  "systemAgent": {
    "rescue": {
      "enabled": "auto",
      "ownerDmOnly": true,
      "pendingTtlMinutes": 15,
    },
  },
}
```

- `enabled`: لا تسمح `"auto"` (الافتراضية) بالإنقاذ إلا عندما تكون بيئة التشغيل الفعلية YOLO ويكون العزل متوقفًا؛ ولا تسمح `false` بإنقاذ قناة الرسائل مطلقًا؛ وتسمح `true` صراحةً بالإنقاذ عند اجتياز فحوص المالك/القناة (مع بقائه خاضعًا للرفض عند تفعيل العزل).
- `ownerDmOnly`: يقصر الإنقاذ على الرسائل المباشرة للمالك. القيمة الافتراضية `true`.
- `pendingTtlMinutes`: مدة بقاء عملية كتابة إنقاذ معلّقة مفتوحة بانتظار موافقة `/openclaw yes` قبل انتهاء صلاحيتها. القيمة الافتراضية `15`.

يرحّل `openclaw doctor --fix` كتلة الإعداد القديمة `crestodian` إلى
`systemAgent`. ولا تقرأ بيئة التشغيل إلا الكتلة القياسية.

يغطي مسار Docker الإنقاذ عن بُعد:

```bash
pnpm test:docker:system-agent-rescue
```

يفحص اختبار دخان اختياري لواجهة أوامر القناة المباشرة `/openclaw status` بالإضافة إلى دورة موافقة دائمة كاملة عبر معالج الإنقاذ:

```bash
pnpm test:live:system-agent-rescue-channel
```

يغطي ما يلي الإعداد الأحادي المعبّأ والخاضع لبوابة الاستدلال:

```bash
pnpm test:docker:system-agent-first-run
```

يبدأ مسار CLI المعبّأ هذا بدليل حالة فارغ ويثبت أن OpenClaw
يفشل بشكل مغلق من دون استدلال. ثم يختبر Claude وهميًا ويفعّله من خلال
وحدة التفعيل المعبّأة. بعد ذلك فقط يصل طلب تقريبي إلى
المخطط ويُحل إلى إعداد منمّط، متبوعًا بأوامر أحادية تنشئ
وكيلًا إضافيًا، وتضبط Discord من خلال تفعيل Plugin بالإضافة إلى رمز
SecretRef، وتتحقق من الإعداد، وتفحص سجل التدقيق. يمثّل هذا المسار أدلة داعمة
للبوابة/العملية؛ ولا يختبر الإعداد التفاعلي أو
محادثة وكيل/أداة/موافقة OpenClaw. يعيد سيناريو QA Lab أدناه التوجيه
إلى مسار Docker نفسه:

```bash
pnpm openclaw qa suite --scenario system-agent-ring-zero-setup
```
