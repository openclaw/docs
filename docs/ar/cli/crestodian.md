# `openclaw crestodian`

Crestodian الحواري هو وكيل OpenClaw المحلي للإعداد والإصلاح والتهيئة. ولا يبدأ إلا بعد أن يُكمل النموذج الافتراضي الفعلي دورة حقيقية. تُنشئ عمليات التثبيت الجديدة الاستدلال أولًا؛ بينما تظل التهيئة غير الصالحة ضمن مسار doctor التقليدي.

## متى يبدأ

يُوجّه تشغيل `openclaw` دون أمر فرعي وفقًا لحالة التهيئة:

- إذا كانت التهيئة مفقودة، أو كانت موجودة من دون إعدادات أنشأها المستخدم (فارغة، أو لا تحتوي إلا على مفتاحي `$schema`/`meta`): يبدأ الإعداد الأولي الموجّه مع تحقق مباشر بالذكاء الاصطناعي.
- إذا كانت التهيئة موجودة لكنها تفشل في التحقق: يبدأ الإعداد الأولي التقليدي، الذي يعرض المشكلات ويوجهك إلى `openclaw doctor`.
- إذا كانت التهيئة موجودة وصالحة: يفتح TUI العادي للوكيل. وإذا كان Gateway المهيأ قابلًا للوصول وكان للوكيل الافتراضي نموذج، فسينتقل مباشرةً إلى تلك الواجهة من دون إعداد أولي أو Crestodian. استخدم `/crestodian` داخل TUI، أو شغّل `openclaw crestodian` مباشرةً، للوصول إلى Crestodian لاحقًا.

يختبر تشغيل `openclaw crestodian` أولًا النموذج الافتراضي المهيأ اختبارًا مباشرًا. وتؤدي الدورة الناجحة إلى بدء Crestodian. أما الفشل التفاعلي فيفتح إعداد الاستدلال الموجّه، ثم يسلّم التحكم إلى Crestodian بعد نجاح أحد الخيارات المرشحة. وتفشل طلبات التنفيذ لمرة واحدة وطلبات JSON وغيرها من الطلبات غير التفاعلية، مع إرشادات لتشغيل `openclaw onboard` عندما لا يكون الاستدلال متاحًا. ويحتفظ `openclaw --help` و`openclaw --version` بمساريهما السريعين المعتادين.

ينهي تشغيل `openclaw` المجرد في الوضع غير التفاعلي (من دون TTY) العملية برسالة قصيرة بدلًا من طباعة مساعدة الجذر: إذ يشير إلى الإعداد الأولي غير التفاعلي في تثبيت جديد أو غير صالح، أو إلى `openclaw agent --local ...` عندما تكون التهيئة صالحة.

يظل `openclaw onboard --modern` اسمًا بديلًا للتوافق مع Crestodian، لكنه يستخدم بوابة الاستدلال نفسها: يفتح الاستدلال العامل المحادثة، وتبدأ حالات الفشل التفاعلية إعداد الاستدلال الموجّه، وتنتهي حالات الفشل غير التفاعلية مع إرشادات الإعداد الأولي. ويفتح `openclaw onboard --classic` المعالج الكامل خطوة بخطوة.

## ما يعرضه Crestodian

يفتح Crestodian التفاعلي غلاف TUI نفسه الذي يفتحه `openclaw tui`، مع واجهة خلفية لمحادثة Crestodian. وتغطي تحية بدء التشغيل ما يلي:

- صلاحية التهيئة والوكيل الافتراضي
- النموذج المتحقق منه الذي يستخدمه Crestodian
- إمكانية الوصول إلى Gateway وفق أول فحص عند بدء التشغيل
- إجراء التصحيح التالي الموصى به

لا يعرض الأسرار ولا يحمّل أوامر CLI الخاصة بالـ Plugin لمجرد بدء التشغيل.

استخدم `status` لعرض الجرد التفصيلي: مسار التهيئة، ومسارات الوثائق/المصدر، وفحوصات CLI المحلية، ووجود المفاتيح/الرموز المميزة، والوكلاء، والنموذج، وتفاصيل Gateway.

يستخدم Crestodian اكتشاف المراجع نفسه الذي تستخدمه الوكلاء العادية: ففي نسخة Git يشير إلى `docs/` المحلية وشجرة المصدر؛ وفي تثبيت npm يستخدم الوثائق المضمّنة ويربط بـ [https://github.com/openclaw/openclaw](https://github.com/openclaw/openclaw)، مع إرشادات لمراجعة المصدر عندما لا تكون الوثائق كافية.

## أمثلة

```bash
openclaw
openclaw crestodian
openclaw crestodian --json
openclaw crestodian --message "models"
openclaw crestodian --message "validate config"
openclaw crestodian --message "setup workspace ~/Projects/work" --yes
openclaw crestodian --message "set default model openai/gpt-5.6" --yes
openclaw onboard --modern
```

داخل TUI الخاص بـ Crestodian:

```text

channels

open channel wizard for slack
plugins list

plugin install clawhub:openclaw-codex-app-server
talk to work agent
talk to agent for ~/Projects/work
audit
quit
```

## العمليات والموافقة

يستخدم Crestodian عمليات ذات أنواع محددة بدلًا من تعديل التهيئة ارتجاليًا.

تُنفّذ عمليات القراءة فقط فورًا: عرض النظرة العامة، وسرد الوكلاء، وسرد الـ Plugins المثبّتة، والبحث عن Plugins في ClawHub، وعرض حالة النموذج/الواجهة الخلفية، وتشغيل فحوصات الحالة/السلامة، والتحقق من إمكانية الوصول إلى Gateway، وتشغيل doctor من دون إصلاحات تفاعلية، والتحقق من التهيئة، وعرض مسار سجل التدقيق.

كما يبدأ إعداد القناة الموجّه (`connect telegram`) فورًا. ويجمع معالجه إجابات صريحة ويتولى عمليات الكتابة الناتجة.

تتطلب العمليات الدائمة موافقة حوارية (أو `--yes` للأمر المباشر): كتابة التهيئة، و`config set`، و`config set-ref`، وتمهيد الإعداد/الإعداد الأولي، وتغيير النموذج الافتراضي، وبدء/إيقاف/إعادة تشغيل Gateway، وإنشاء الوكلاء، وتثبيت Plugins.

لا تتوفر إصلاحات doctor داخل Crestodian لأنها قد تعيد كتابة مسار استدلال المزوّد أو المصادقة أو الوكيل الافتراضي الذي يشغّل الجلسة. اخرج من Crestodian وشغّل `openclaw doctor --fix` في طرفية. ويظل `doctor` للقراءة فقط متاحًا داخل Crestodian.

ترث الوكلاء الجديدة مسار الاستدلال الافتراضي المتحقق منه مباشرةً. ومعرّف الوكيل `crestodian` محجوز للقيّم الافتراضي ذي الامتيازات، ولا يمكن إنشاؤه كوكيل عادي.

لا يمكن لـ `config set` و`config set-ref` تغيير حالة مسار الاستدلال، بما في ذلك بيانات اعتماد مزوّد الاستدلال، و`auth.*` في المستوى الأعلى، وكتالوجات النماذج، وواجهات CLI الخلفية، ومسارات النماذج الافتراضية/الخاصة بكل وكيل، ومعلمات/أدوات الوكيل، أو `tools.*` الجذرية. كما تُرفض عمليات الكتابة الخام ضمن `env.*` و`secrets.*` و`plugins.*` و`$include` لأنها قد تستبدل آلية حل بيانات الاعتماد أو تفعيل المزوّد. وتظل مصادقة Gateway والقنوات أسطح تهيئة عادية. استخدم مسارات عمل الـ Plugin/القناة ذات الأنواع المحددة و`set default model <provider/model>` لمسار مهيأ بالفعل؛ إذ يختبر المسار مباشرةً قبل حفظه. ولتهيئة وصول المزوّد/المصادقة أو إصلاحه، اخرج من Crestodian وشغّل `openclaw onboard`.

يُرفض إلغاء تثبيت Plugin داخل Crestodian لأن إزالة Plugin خاص بمزوّد قد تعطّل مسار الاستدلال الذي يشغّل الجلسة. اخرج من Crestodian وشغّل `openclaw plugins uninstall <id>` من طرفية.

تُمنح الموافقة بكلماتك أنت: فالردود غير الملتبسة ("نعم"، "بالتأكيد"، "تابع"، "ليس الآن") تُحسم من قائمة مغلقة وحتمية. وعندما يدعم المسار المهيأ استدعاء إكمال منفصلًا، يمكن تصنيف الردود الأخرى بالاعتماد فقط على رسالتك والمقترح المعلّق — وليس بواسطة نموذج المحادثة نفسه مطلقًا، إذ لا يمكنه الموافقة على نفسه. وتُبقي الردود غير المصنفة أو الملتبسة المقترح معلّقًا، وتطلب المحادثة الإجابة مرة أخرى.

تُسجّل عمليات الكتابة المطبقة في `~/.openclaw/audit/crestodian.jsonl`. ولا يُسجّل الاكتشاف في سجل التدقيق؛ بل تُسجّل فقط العمليات وعمليات الكتابة المطبقة.

يمكن تشغيل إعداد القناة كمحادثة مستضافة حتى يصل إلى سر. ولا يقبل TUI المحلي لـ Crestodian إجابات المعالج الحساسة لأن إدخال المحادثة في الطرفية ظاهر. وهو يعرض `open channel wizard` فورًا، مع تمرير القناة المحددة إلى معالج الطرفية ذي الإدخال المحجوب؛ كما يمكنك تشغيل `openclaw channels add --channel <channel>` لاحقًا.

### الانتقال إلى إعداد القناة ذي الإدخال المحجوب

يمكن للمحادثة المحلية تسليم التحكم إلى معالج القناة ذي الإدخال المحجوب:

```text
open channel wizard for slack
channel info slack
```

يفتح `open channel wizard for <channel>` إعداد القناة ذي الإدخال المحجوب بعد إغلاق TUI الخاص بالمحادثة. استخدم `channel info <channel>` أولًا للحصول على تسمية القناة، وحالة الإعداد، وملخص المتطلبات الأساسية، ورابط الوثائق.

لا يغيّر Crestodian مطلقًا وصول المزوّد/المصادقة من داخل جلسته، لأن الجلسة تعتمد بالفعل على مسار الاستدلال هذا. ولإعداد مزوّد النموذج أو إصلاحه، يعيد `configure model provider` إرشادات الخروج/الإعداد الأولي من دون بدء معالج أو كتابة التهيئة. اخرج من Crestodian وشغّل `openclaw onboard`؛ إذ يجهّز الإعداد الأولي بيانات الاعتماد ولا يحفظ إلا مسارًا يُكمل دورة مباشرة حقيقية. ابدأ Crestodian مجددًا بعد نجاح الإعداد الأولي.

## تمهيد الإعداد

يهيّئ `setup` مساحة العمل وحالة Gateway المتبقيتين بعد أن يكون الإعداد الأولي الموجّه قد أنشأ الاستدلال بالفعل. ولا يكتب إلا من خلال عمليات تهيئة ذات أنواع محددة، ويطلب الموافقة أولًا.

```text
setup
setup workspace ~/Projects/work
```

يحافظ `setup` على النموذج الفعلي المتحقق منه. ولا يهيّئ الاستدلال أو يستبدله.

إذا كان الاستدلال مفقودًا أو فشل فحصه المباشر، فاترك Crestodian وشغّل `openclaw onboard`. يكتشف الإعداد الأولي الموجّه النماذج المهيأة، ومفاتيح API، وواجهات CLI المحلية التي تمت مصادقتها، ويطلب من كل خيار مرشح ردًا حقيقيًا، ولا يحفظ إلا مسارًا ناجحًا. يبدأ Crestodian فور عبور هذا الحد، ويمكنه بعد ذلك تهيئة مساحة العمل، وGateway، والقنوات، والوكلاء، وPlugins، والميزات الاختيارية الأخرى.

يتجاوز تطبيق macOS هذا التسلسل بالكامل عندما يصل إلى Gateway مهيأ يكون لوكيله الافتراضي نموذج مهيأ بالفعل؛ إذ يفتح واجهة الوكيل العادية.
أما في حالة Gateway جديد أو غير مكتمل، فيدير التطبيق تسلسل الاستدلال عبر طريقتي Gateway التاليتين: `crestodian.setup.detect` و`crestodian.setup.activate`: تسرد detect كل واجهة خلفية مرشحة تعثر عليها، وتختبر activate خيارًا مرشحًا واحدًا مباشرةً (عملية إكمال حقيقية بطلب "الرد بـ OK")، ولا تحفظ إلا حالة النموذج وبيانات الاعتماد والمزوّد/بيئة التشغيل اللازمة لذلك المسار بعد نجاح الاختبار. وتظل الإعدادات الافتراضية لمساحة العمل وGateway من اختصاص Crestodian. ولا يغيّر الخيار المرشح الفاشل التهيئة مطلقًا؛ بل ينتقل التطبيق تلقائيًا إلى الخيار التالي في التسلسل، ثم يعرض في النهاية خطوة يدوية للمفتاح/الرمز المميز، تُملأ من Plugins الخاصة بمزوّدي استدلال النص النشطين في Gateway. ويتولى المزوّد المحدد نموذجه الابتدائي وتهيئته، وتُتحقق بيانات الاعتماد بالطريقة نفسها قبل حفظها.

تظل مراقبة Codex وميزات Plugins الاختيارية الأخرى خارج معاملة تفعيل الاستدلال هذه. لا تهيّئها إلا بعد عمل الاستدلال وبدء Crestodian؛ وتظل سياسة الـ Plugin الحالية وحالات إلغاء الاشتراك الصريحة في المراقبة من دون تغيير أثناء إعداد الاستدلال.

## محادثة الذكاء الاصطناعي

تعمل محادثة Crestodian التفاعلية الحرة عبر حلقة الوكيل نفسها التي تستخدمها وكلاء OpenClaw العادية، مع تقييدها بأداة صلاحية واحدة من الحلقة الصفرية في OpenClaw، وهي `crestodian`، التي تغلّف العمليات ذات الأنواع المحددة. تعمل إجراءات القراءة بحرية، وتتطلب التعديلات موافقتك الحوارية على تلك العملية بعينها (راجع العمليات والموافقة)، وتُسجّل كل عملية كتابة مطبقة في سجل التدقيق ويُعاد التحقق منها. وتستمر جلسة الوكيل، لذا يمتلك Crestodian ذاكرة فعلية متعددة الدورات. وإذا توقف مسار الاستدلال المتحقق منه عن العمل لاحقًا، فارجع إلى `openclaw onboard` وأصلحه قبل المتابعة.

لا يحلّل المضيف طلبات اللغة الطبيعية إلى عمليات. فالرسائل الحرة — بما في ذلك النصوص التي تبدو كأوامر والأسئلة مثل "لماذا توقف Gateway لدي؟" — تُرسل إلى الذكاء الاصطناعي، الذي يمكنه ربط الطلب بعملية ذات نوع محدد عبر أداة `crestodian`.

عندما يكون تعديل ما معلّقًا، لا تُحسم من دون استدلال إلا عبارات الموافقة أو الرفض غير الملتبسة الواردة في قائمة مغلقة. وتُرسل الموافقة الملتبسة إلى استدعاء إكمال مهيأ منفصل، وإلا فتُرفض احترازيًا. حقول المعالج المنظّمة والتنقل الدقيق للمضيف هي عناصر تحكم في واجهة المستخدم، وليست تحليلًا للغة الطبيعية إلى عمليات. وهناك استثناء مهم خصوصًا لنظافة الأسرار: فأمر `config set` المطابق تمامًا لمسار حساس (الرموز المميزة، والمفاتيح، وكلمات المرور) لا يصل إلى نموذج مطلقًا. ينشئ المضيف مقترحًا منقّحًا، وتُحجب القيمة في السجل المرئي للذكاء الاصطناعي. استخدم `config set-ref <path> env <ENV_VAR>` للأسرار.

لا يستخدم وضع الإنقاذ عبر قنوات الرسائل المخطط المدعوم بالنموذج مطلقًا. ويظل الإنقاذ عن بُعد حتميًا بحيث لا يمكن استخدام مسار وكيل عادي معطّل أو مخترق كمحرر للتهيئة.

### نموذج ثقة مُسخّر CLI

تفرض بيئات التشغيل المضمّنة ومُسخّر خادم تطبيق Codex قيد الحلقة الصفرية مباشرةً: إذ يحمل التشغيل قائمة سماح لأدوات OpenClaw لا تضم إلا أداة `crestodian`. وبالنسبة إلى Codex، يعطّل OpenClaw أيضًا البيئات، والتنفيذ الأصلي، وتعدد الوكلاء، والهدف، والتطبيق/الـ Plugin، وskill/MCP، والبحث على الويب، وأسطح `request_user_input` لذلك التشغيل. ويظل Codex يحقن أداة `update_plan` الأصلية الخاملة الخاصة به؛ ويمكنها تحديث قائمة التحقق المؤقتة للنموذج، لكنها لا تستطيع كتابة الملفات أو تهيئة OpenClaw. ولا تستهلك مُسخّرات CLI قائمة السماح الخاصة بـ OpenClaw، لذلك لا يقبل Crestodian إلا الواجهات الخلفية التي يمكن لعقد اختيار الأدوات الخاص بها إثبات القيد نفسه:

- تُشغَّل الواجهات الخلفية القابلة للتحديد، بما فيها Claude Code، بتحديد فارغ للأدوات الأصلية وأداة MCP واحدة هي `crestodian`. يُطبَّق إعداد MCP المُنشأ بواسطة Claude باستخدام `--strict-mcp-config`، لذلك لا تُحمَّل أي خوادم MCP أخرى.
- تتلقى الواجهات الخلفية التي تعلن عدم امتلاكها أدوات أصلية خادم MCP المخصص نفسه لـ Crestodian.
- تفشل الواجهات الخلفية ذات الأدوات الأصلية الدائمة التشغيل أو غير المعروفة بشكل مغلق قبل الاستدلال؛ ولا يمكنها استضافة جلسة Crestodian.

تحصل جلسات Crestodian وحدها على خادم MCP الخاص بـ crestodian؛ ولا ترى عمليات الوكيل العادية هذه الأداة مطلقًا. لذلك تفرض واجهات CLI الخلفية القابلة للتحديد/الخالية من الأدوات الأصلية والنماذج المعتمدة على مفتاح API حلقة الأداة الواحدة حرفيًا. وتفرض نماذج خادم تطبيق Codex أداة صلاحية واحدة لـ OpenClaw إلى جانب أداة التخطيط الأصلية الخاملة. وفي الحالات الثلاث جميعًا، تظل عمليات كتابة الإعداد محصورة ضمن عقد الموافقة الخاضع للتدقيق في Crestodian.

يظل Gemini CLI متاحًا للوكلاء العاديين، لكنه لا يستطيع فرض الفحص الخالي من الأدوات الذي تتطلبه بوابة الاستدلال، ولذلك لا يمكنه استضافة Crestodian.

## التبديل إلى وكيل

استخدم محددًا بلغة طبيعية لمغادرة Crestodian وفتح TUI العادية:

```text
تحدث إلى الوكيل
تحدث إلى وكيل العمل
بدّل إلى الوكيل الرئيسي
```

تفتح `openclaw tui` و`openclaw chat` و`openclaw terminal` واجهة TUI العادية للوكيل مباشرةً؛ ولا تبدأ Crestodian. بعد التبديل إلى TUI العادية، يعيدك `/crestodian` إلى Crestodian، مع إمكانية إضافة طلب متابعة:

```text
/crestodian
/crestodian restart gateway
```

## وضع إنقاذ الرسائل

وضع إنقاذ الرسائل هو نقطة دخول قنوات الرسائل إلى Crestodian: استخدمه عندما يتوقف وكيلك العادي عن العمل، مع استمرار قناة موثوقة (مثل WhatsApp) في تلقي الأوامر.

هذا معالج حتمي لأوامر الطوارئ، وليس وكيل Crestodian التحاوري. وهو لا يُمهّد إعدادًا جديدًا ولا يخفف بوابة الاستدلال لمحادثة Crestodian.

الأمر المدعوم: `/crestodian <request>`. لا يقبل الإنقاذ سوى صياغة الأوامر المكتوبة حرفيًا — إذ تُرفض اللغة الطبيعية مع تلميح، ولا تُخمَّن أبدًا لتحويلها إلى عملية، ولا يُستشار أي نموذج مطلقًا.

```text
أنت، في رسالة مباشرة موثوقة من المالك: /crestodian status
OpenClaw: وضع إنقاذ Crestodian. هل يمكن الوصول إلى Gateway: لا. هل الإعداد صالح: لا.
أنت: /crestodian restart gateway
OpenClaw: الخطة: إعادة تشغيل Gateway. أرسل /crestodian yes للتطبيق.
أنت: /crestodian yes
OpenClaw: تم التطبيق. كُتب إدخال التدقيق.
```

يمكن أيضًا وضع إنشاء الوكيل في قائمة انتظار محليًا أو عبر الإنقاذ:

```text
create agent work workspace ~/Projects/work model openai/gpt-5.6-sol
/crestodian create agent work workspace ~/Projects/work
```

لا يجوز أن يحدد إنشاء الوكيل سوى النموذج الافتراضي الحالي المتحقق منه مباشرةً. احذف النموذج لوراثة ذلك المسار.

الإنقاذ عن بُعد سطح إداري ويجب التعامل معه كإصلاح إعدادات عن بُعد، لا كمحادثة عادية.

عقد الأمان للإنقاذ عن بُعد:

- يكون معطّلًا عندما يكون العزل نشطًا للوكيل/الجلسة؛ يرفض Crestodian الإنقاذ عن بُعد ويوجّه إلى الإصلاح المحلي عبر CLI.
- الحالة الفعلية الافتراضية هي `auto`: لا يُسمح بالإنقاذ عن بُعد إلا في تشغيل YOLO موثوق، حيث يمتلك وقت التشغيل أصلًا صلاحية محلية غير معزولة (تُحسم `tools.exec.security` إلى `full` وتُحسم `tools.exec.ask` إلى `off`، مع كون وضع العزل `off`).
- يتطلب هوية مالك صريحة؛ ولا يُسمح بقواعد مرسل بأحرف بدل، أو سياسة مجموعات مفتوحة، أو Webhook غير موثّقة، أو قنوات مجهولة.
- يقتصر افتراضيًا على الرسائل المباشرة للمالك؛ ويتطلب الإنقاذ في مجموعة/قناة اشتراكًا صريحًا.
- البحث عن Plugin وسردها للقراءة فقط. ويكون تثبيت Plugin محليًا فقط دائمًا (محظورًا في الإنقاذ، حتى عندما يكون مفعّلًا في حالات أخرى) لأنه ينزّل تعليمات برمجية قابلة للتنفيذ. ويُرفض إلغاء تثبيت Plugin في كل من Crestodian المحلي والإنقاذ؛ شغّل `openclaw plugins uninstall <id>` من طرفية.
- لا يستطيع الإنقاذ عن بُعد فتح TUI المحلية أو التبديل إلى جلسة وكيل تفاعلية؛ استخدم `openclaw` محليًا لتسليم التحكم إلى الوكيل.
- لا تزال عمليات الكتابة الدائمة تتطلب موافقة، حتى في وضع الإنقاذ.
- تخضع كل عملية إنقاذ مطبَّقة للتدقيق. يسجّل إنقاذ قناة الرسائل بيانات القناة والحساب والمرسل وعنوان المصدر الوصفية؛ كما تسجّل العمليات التي تعدّل الإعدادات تجزئات الإعدادات قبل التعديل وبعده.
- لا تُعرض الأسرار مطلقًا. يُبلغ فحص SecretRef عن التوافر، لا عن القيم.
- إذا كان Gateway يعمل، يفضّل الإنقاذ عمليات Gateway المعرَّفة نوعيًا؛ وإذا كان متوقفًا، فلا يستخدم الإنقاذ سوى الحد الأدنى من سطح الإصلاح المحلي الذي لا يعتمد على حلقة الوكيل العادية.

بنية الإعدادات:

```jsonc
{
  "crestodian": {
    "rescue": {
      "enabled": "auto",
      "ownerDmOnly": true,
      "pendingTtlMinutes": 15,
    },
  },
}
```

- `enabled`: تسمح `"auto"` (الافتراضية) بالإنقاذ فقط عندما يكون وقت التشغيل الفعلي في وضع YOLO ويكون العزل متوقفًا؛ ولا تسمح `false` مطلقًا بإنقاذ قناة الرسائل؛ وتسمح `true` صراحةً بالإنقاذ عند اجتياز فحوصات المالك/القناة (مع بقائها خاضعة لرفض العزل).
- `ownerDmOnly`: يقصر الإنقاذ على الرسائل المباشرة للمالك. القيمة الافتراضية `true`.
- `pendingTtlMinutes`: المدة التي تظل فيها عملية كتابة إنقاذ معلّقة مفتوحة للحصول على موافقة `/crestodian yes` قبل انتهاء صلاحيتها. القيمة الافتراضية `15`.

تغطي مسار Docker الإنقاذ عن بُعد:

```bash
pnpm test:docker:crestodian-rescue
```

يتحقق فحص دخاني اختياري مباشر لسطح أوامر القناة من `/crestodian status`، إضافةً إلى دورة موافقة كاملة على عملية دائمة عبر معالج الإنقاذ:

```bash
pnpm test:live:crestodian-rescue-channel
```

يغطي ما يلي إعداد الحزمة أحادي التنفيذ المحكوم ببوابة الاستدلال:

```bash
pnpm test:docker:crestodian-first-run
```

يبدأ مسار CLI المضمّن هذا بدليل حالة فارغ ويثبت أن Crestodian يفشل بشكل مغلق من دون استدلال. ثم يختبر Claude وهميًا ويفعّله عبر وحدة التفعيل المضمّنة. بعد ذلك فقط يصل طلب تقريبي إلى المخطِّط ويُحسم إلى إعداد معرَّف نوعيًا، تليه أوامر أحادية التنفيذ تنشئ وكيلًا إضافيًا، وتهيئ Discord عبر تمكين Plugin مع رمز SecretRef، وتتحقق من صحة الإعدادات، وتفحص سجل التدقيق. يوفر هذا المسار أدلة داعمة للبوابة/العمليات؛ ولا يختبر الإعداد التفاعلي الأولي أو محادثة الوكيل/الأداة/الموافقة في Crestodian. يعيد سيناريو مختبر ضمان الجودة أدناه التوجيه إلى مسار Docker نفسه:

```bash
pnpm openclaw qa suite --scenario crestodian-ring-zero-setup
```
