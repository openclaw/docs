# استخدام الرموز والتكاليف

يتتبع OpenClaw **الرموز**، وليس الأحرف. وتختلف الرموز حسب النموذج، لكن معظم
النماذج على نمط OpenAI يبلغ متوسطها نحو 4 أحرف لكل رمز في النص الإنجليزي.

## كيف تُبنى مطالبة النظام

يقوم OpenClaw بتجميع مطالبة النظام الخاصة به في كل تشغيل. وهي تتضمن:

- قائمة الأدوات + أوصافًا قصيرة
- قائمة Skills ‏(بيانات وصفية فقط؛ ويتم تحميل التعليمات عند الطلب باستخدام `read`).
  وتُقيَّد كتلة Skills المدمجة بواسطة `skills.limits.maxSkillsPromptChars`،
  مع تجاوز اختياري لكل وكيل عند
  `agents.list[].skillsLimits.maxSkillsPromptChars`.
- تعليمات التحديث الذاتي
- مساحة العمل + ملفات bootstrap ‏(`AGENTS.md`، و`SOUL.md`، و`TOOLS.md`، و`IDENTITY.md`، و`USER.md`، و`HEARTBEAT.md`، و`BOOTSTRAP.md` عندما يكون جديدًا، بالإضافة إلى `MEMORY.md` عند وجوده). ولا يتم حقن `memory.md` الصغير في الجذر؛ بل هو مدخل إصلاح قديم لـ `openclaw doctor --fix` عند اقترانه بـ `MEMORY.md`. وتُقتطع الملفات الكبيرة بواسطة `agents.defaults.bootstrapMaxChars` ‏(الافتراضي: 12000)، كما يُقيَّد إجمالي حقن bootstrap بواسطة `agents.defaults.bootstrapTotalMaxChars` ‏(الافتراضي: 60000). ولا تكون ملفات `memory/*.md` اليومية جزءًا من مطالبة bootstrap العادية؛ إذ تظل متاحة عند الطلب عبر أدوات الذاكرة في الأدوار العادية، لكن يمكن للأمرين `/new` و`/reset` المجردين أن يسبقا أول دور فقط بكتلة startup-context لمرة واحدة تحتوي على الذاكرة اليومية الحديثة. ويتم التحكم في هذا التقديم الأولي عبر `agents.defaults.startupContext`.
- الوقت (UTC + المنطقة الزمنية للمستخدم)
- علامات الرد + سلوك Heartbeat
- بيانات وصفية وقت التشغيل (المضيف/نظام التشغيل/النموذج/التفكير)

راجع التفصيل الكامل في [مطالبة النظام](/ar/concepts/system-prompt).

## ما الذي يُحتسب ضمن نافذة السياق

كل ما يستقبله النموذج يُحتسب ضمن حد السياق:

- مطالبة النظام (كل الأقسام المذكورة أعلاه)
- سجل المحادثة (رسائل المستخدم + المساعد)
- استدعاءات الأدوات ونتائجها
- المرفقات/النصوص المنسوخة (الصور، والصوت، والملفات)
- ملخصات Compaction والآثار الناتجة عن التنقية
- أغلفة المزوّد أو ترويسات الأمان (غير مرئية، لكنها لا تزال تُحتسب)

تملك بعض الأسطح الثقيلة وقت التشغيل حدودًا صريحة خاصة بها:

- `agents.defaults.contextLimits.memoryGetMaxChars`
- `agents.defaults.contextLimits.memoryGetDefaultLines`
- `agents.defaults.contextLimits.toolResultMaxChars`
- `agents.defaults.contextLimits.postCompactionMaxChars`

توجد التجاوزات لكل وكيل ضمن `agents.list[].contextLimits`. وهذه المفاتيح
مخصصة للمقتطفات المقيّدة وقت التشغيل والكتل المحقونة المملوكة لوقت التشغيل. وهي
منفصلة عن حدود bootstrap، وحدود startup-context، وحدود مطالبات Skills.

بالنسبة إلى الصور، يقوم OpenClaw بتقليص أبعاد حمولة صور transcript/tool قبل استدعاءات المزوّد.
استخدم `agents.defaults.imageMaxDimensionPx` ‏(الافتراضي: `1200`) لضبط ذلك:

- تؤدي القيم الأقل عادةً إلى تقليل استخدام رموز الرؤية وحجم الحمولة.
- تحافظ القيم الأعلى على مزيد من التفاصيل البصرية في لقطات OCR/واجهة المستخدم الثقيلة.

للحصول على تفصيل عملي (لكل ملف محقون، والأدوات، وSkills، وحجم مطالبة النظام)، استخدم `/context list` أو `/context detail`. راجع [السياق](/ar/concepts/context).

## كيف ترى استخدام الرموز الحالي

استخدم ما يلي في الدردشة:

- `/status` → **بطاقة حالة غنية بالرموز التعبيرية** تتضمن نموذج الجلسة، واستخدام السياق،
  ورموز الإدخال/الإخراج لآخر استجابة، و**التكلفة التقديرية** (لمفاتيح API فقط).
- `/usage off|tokens|full` → يضيف **تذييل استخدام لكل استجابة** إلى كل رد.
  - يُحفَظ لكل جلسة (ويُخزَّن على أنه `responseUsage`).
  - تخفي مصادقة OAuth **التكلفة** (وتعرض الرموز فقط).
- `/usage cost` → يعرض ملخص تكلفة محليًا من سجلات جلسات OpenClaw.

أسطح أخرى:

- **TUI/Web TUI:** يدعمان `/status` و`/usage`.
- **CLI:** يعرض `openclaw status --usage` و`openclaw channels list`
  نوافذ حصص مزوّدين مطبّعة (`X% left`، وليس تكاليف لكل استجابة).
  مزوّدو نوافذ الاستخدام الحالية: Anthropic، وGitHub Copilot، وGemini CLI،
  وOpenAI Codex، وMiniMax، وXiaomi، وz.ai.

تقوم أسطح الاستخدام بتطبيع الأسماء البديلة الشائعة للحقول الأصلية لدى المزوّد قبل العرض.
وبالنسبة إلى حركة Responses من عائلة OpenAI، يشمل ذلك كلًا من `input_tokens` /
`output_tokens` و`prompt_tokens` / `completion_tokens`، بحيث لا تؤدي أسماء
الحقول الخاصة بالنقل إلى تغيير `/status` أو `/usage` أو ملخصات الجلسة.
كما يتم تطبيع استخدام JSON الخاص بـ Gemini CLI أيضًا: إذ يأتي نص الرد من `response`،
ويُربط `stats.cached` إلى `cacheRead` مع استخدام `stats.input_tokens - stats.cached`
عندما يحذف CLI الحقل الصريح `stats.input`.
وبالنسبة إلى حركة Responses الأصلية من عائلة OpenAI، يتم تطبيع الأسماء البديلة لاستخدام WebSocket/SSE بالطريقة نفسها، كما تعود الإجماليات إلى الإدخال + الإخراج المطبّعين عندما تكون
`total_tokens` مفقودة أو تساوي `0`.
وعندما تكون لقطة الجلسة الحالية متناثرة، يمكن أيضًا لـ `/status` و`session_status`
استعادة عدادات الرموز/الذاكرة المؤقتة وعلامة نموذج وقت التشغيل النشط من أحدث سجل استخدام transcript. وتظل القيم الحية الموجودة وغير الصفرية لها الأولوية على القيم الاحتياطية من transcript، كما يمكن أن تفوز إجماليات transcript الأكبر والموجهة نحو prompt عندما تكون الإجماليات المخزنة مفقودة أو أصغر.
وتأتي مصادقة الاستخدام الخاصة بنوافذ حصص المزوّد من hooks خاصة بالمزوّد عند
توفرها؛ وإلا يعود OpenClaw إلى مطابقة بيانات اعتماد OAuth/API-key
من ملفات التعريف، أو env، أو التكوين.
وتحفظ إدخالات transcript الخاصة بالمساعد الشكل المطبّع نفسه للاستخدام، بما في ذلك
`usage.cost` عندما يكون للنموذج النشط تسعير مهيأ ويعيد المزوّد بيانات وصفية للاستخدام. ويوفر هذا مصدرًا ثابتًا لـ `/usage cost` وحالة الجلسة المعتمدة على transcript حتى بعد زوال حالة وقت التشغيل المباشرة.

يبقي OpenClaw محاسبة الاستخدام الخاصة بالمزوّد منفصلة عن لقطة السياق الحالية.
فقد تتضمن قيمة `usage.total` لدى المزوّد إدخالًا مخزنًا مؤقتًا، وإخراجًا، وعدة استدعاءات لنموذج داخل حلقة أداة، لذا فهي مفيدة للتكلفة وtelemetry لكنها قد تبالغ في تقدير نافذة السياق المباشرة. وتستخدم عروض وتشخيصات السياق أحدث لقطة prompt ‏(`promptTokens`، أو آخر استدعاء للنموذج عند غياب لقطة prompt) من أجل `context.used`.

## تقدير التكلفة (عند عرضه)

تُقدَّر التكاليف من تكوين تسعير النموذج لديك:

```
models.providers.<provider>.models[].cost
```

وهي **بالدولار الأمريكي لكل 1M رمز** لقيم `input`، و`output`، و`cacheRead`، و
`cacheWrite`. وإذا كان التسعير مفقودًا، يعرض OpenClaw الرموز فقط. ولا تعرض رموز OAuth
تكلفة بالدولار مطلقًا.

## تأثير TTL الخاص بالذاكرة المؤقتة والتنقية

لا تنطبق الذاكرة المؤقتة لمطالبة المزوّد إلا داخل نافذة TTL الخاصة بالذاكرة المؤقتة. ويمكن لـ OpenClaw
اختياريًا تشغيل **تنقية cache-ttl**: إذ يقوم بتنقية الجلسة بعد انتهاء
TTL الخاص بالذاكرة المؤقتة، ثم يعيد تعيين نافذة الذاكرة المؤقتة بحيث يمكن للطلبات اللاحقة إعادة استخدام
السياق المخزن حديثًا بدلًا من إعادة تخزين السجل الكامل مؤقتًا. وهذا يُبقي تكاليف
الكتابة في الذاكرة المؤقتة أقل عندما تظل الجلسة خاملة بعد TTL.

قم بتكوينه في [تكوين Gateway](/ar/gateway/configuration) وراجع
تفاصيل السلوك في [تنقية الجلسة](/ar/concepts/session-pruning).

يمكن لـ Heartbeat إبقاء الذاكرة المؤقتة **دافئة** خلال فجوات الخمول. فإذا كان TTL الخاص
بذاكرة النموذج المؤقتة هو `1h`، فإن ضبط فاصل Heartbeat على أقل قليلًا من ذلك (مثل `55m`) يمكن أن يتجنب
إعادة تخزين المطالبة الكاملة مؤقتًا، ما يقلل تكاليف الكتابة في الذاكرة المؤقتة.

في الإعدادات متعددة الوكلاء، يمكنك الاحتفاظ بتكوين نموذج مشترك واحد وضبط سلوك الذاكرة المؤقتة
لكل وكيل عبر `agents.list[].params.cacheRetention`.

للحصول على دليل كامل لكل مفتاح على حدة، راجع [Prompt Caching](/ar/reference/prompt-caching).

بالنسبة إلى تسعير Anthropic API، تكون قراءات الذاكرة المؤقتة أقل تكلفة بكثير من
رموز الإدخال، بينما تُفوَّتر عمليات الكتابة في الذاكرة المؤقتة بمعامل أعلى. راجع تسعير
Prompt Caching لدى Anthropic لأحدث الأسعار ومعاملات TTL:
[https://docs.anthropic.com/docs/build-with-claude/prompt-caching](https://docs.anthropic.com/docs/build-with-claude/prompt-caching)

### مثال: إبقاء الذاكرة المؤقتة 1h دافئة باستخدام Heartbeat

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
    heartbeat:
      every: "55m"
```

### مثال: حركة مختلطة مع استراتيجية ذاكرة مؤقتة لكل وكيل

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long" # خط أساس افتراضي لمعظم الوكلاء
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m" # إبقاء الذاكرة المؤقتة الطويلة دافئة للجلسات العميقة
    - id: "alerts"
      params:
        cacheRetention: "none" # تجنب الكتابة في الذاكرة المؤقتة للإشعارات المتقطعة
```

يتم دمج `agents.list[].params` فوق `params` الخاصة بالنموذج المحدد، لذا يمكنك
تجاوز `cacheRetention` فقط مع إبقاء افتراضيات النموذج الأخرى كما هي.

### مثال: تمكين ترويسة Anthropic 1M context التجريبية

تكون نافذة السياق 1M في Anthropic حاليًا محمية بمرحلة beta. ويمكن لـ OpenClaw حقن
قيمة `anthropic-beta` المطلوبة عندما تفعّل `context1m` على نماذج Opus
أو Sonnet المدعومة.

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          context1m: true
```

يُربط هذا إلى ترويسة beta الخاصة بـ Anthropic ‏`context-1m-2025-08-07`.

لا ينطبق هذا إلا عندما تكون `context1m: true` مضبوطة على إدخال ذلك النموذج.

المتطلب: يجب أن تكون بيانات الاعتماد مؤهلة لاستخدام السياق الطويل. وإذا لم تكن كذلك،
فستعيد Anthropic خطأ حد معدل من جهة المزوّد لهذا الطلب.

إذا قمت بالمصادقة مع Anthropic باستخدام رموز OAuth/الاشتراك ‏(`sk-ant-oat-*`)،
فإن OpenClaw يتخطى ترويسة beta ‏`context-1m-*` لأن Anthropic ترفض حاليًا
هذه التركيبة مع HTTP 401.

## نصائح لتقليل ضغط الرموز

- استخدم `/compact` لتلخيص الجلسات الطويلة.
- قلّص مخرجات الأدوات الكبيرة في تدفقات عملك.
- اخفض `agents.defaults.imageMaxDimensionPx` للجلسات الثقيلة باللقطات.
- أبقِ أوصاف Skills قصيرة (إذ تُحقن قائمة Skills في المطالبة).
- فضّل النماذج الأصغر للأعمال الاستكشافية المطوّلة.

راجع [Skills](/ar/tools/skills) لمعرفة الصيغة الدقيقة لحمولة قائمة Skills.
