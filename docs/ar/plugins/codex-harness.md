يتيح Plugin `codex` المضمّن لـ OpenClaw تشغيل دورات وكيل OpenAI المضمّنة
عبر خادم تطبيق Codex بدلًا من حاضنة OpenClaw المدمجة.

استخدم حاضنة Codex عندما تريد أن يتولى Codex جلسة الوكيل منخفضة المستوى:
استئناف سلسلة المحادثة الأصلي، ومتابعة الأدوات الأصلية، وCompaction الأصلية، وتنفيذ
خادم التطبيق. يظل OpenClaw مسؤولًا عن قنوات الدردشة، وملفات الجلسات، واختيار النموذج،
وأدوات OpenClaw الديناميكية، والموافقات، وتسليم الوسائط، ونسخة السجل المرئية.

يستخدم الإعداد العادي مراجع نماذج OpenAI القياسية مثل `openai/gpt-5.5`.
لا تضبط مراجع Codex GPT القديمة. ضع ترتيب مصادقة وكيل OpenAI
ضمن `auth.order.openai`؛ أما معرّفات ملفات مصادقة Codex القديمة
وإدخالات ترتيب مصادقة Codex القديمة فهي حالة قديمة يصلحها
`openclaw doctor --fix`.

عندما لا يكون أي صندوق عزل في OpenClaw نشطًا، يبدأ OpenClaw سلاسل خادم تطبيق Codex
مع تمكين وضع الكود الأصلي في Codex، مع ترك وضع الكود فقط معطّلًا افتراضيًا.
يبقي ذلك مساحة عمل Codex الأصلية وإمكانات الكود متاحة، بينما
تستمر أدوات OpenClaw الديناميكية عبر جسر `item/tool/call` في خادم التطبيق.
تؤدي سياسات العزل النشطة في OpenClaw وسياسات الأدوات المقيّدة إلى تعطيل وضع الكود الأصلي
بالكامل ما لم تختر مسار خادم تنفيذ الصندوق التجريبي.

هذه الميزة الأصلية في Codex منفصلة عن
[وضع كود OpenClaw](/ar/reference/code-mode)، وهو وقت تشغيل QuickJS-WASI اختياري
لتشغيلات OpenClaw العامة مع شكل إدخال `exec` مختلف.

للاطلاع على التقسيم الأوسع بين النموذج/المزوّد/وقت التشغيل، ابدأ بـ
[أوقات تشغيل الوكيل](/ar/concepts/agent-runtimes). الخلاصة المختصرة هي:
`openai/gpt-5.5` هو مرجع النموذج، و`codex` هو وقت التشغيل، وتبقى Telegram
أو Discord أو Slack أو قناة أخرى سطح الاتصال.

## المتطلبات

- OpenClaw مع توفر Plugin `codex` المضمّن.
- إذا كان إعدادك يستخدم `plugins.allow`، فأدرج `codex`.
- خادم تطبيق Codex الإصدار `0.125.0` أو أحدث. يدير Plugin المضمّن ثنائي خادم تطبيق Codex
  متوافقًا افتراضيًا، لذلك لا تؤثر أوامر `codex` المحلية على `PATH`
  في بدء تشغيل الحاضنة العادي.
- توفر مصادقة Codex عبر `openclaw models auth login --provider openai`،
  أو حساب خادم تطبيق في مجلد Codex الرئيسي للوكيل، أو ملف مصادقة Codex صريح بمفتاح API.

لأسبقية المصادقة، وعزل البيئة، وأوامر خادم التطبيق المخصصة، واكتشاف النماذج،
وجميع حقول الإعداد، راجع
[مرجع حاضنة Codex](/ar/plugins/codex-harness-reference).

## البدء السريع

يريد معظم المستخدمين الذين يرغبون في Codex داخل OpenClaw هذا المسار: تسجيل الدخول باستخدام
اشتراك ChatGPT/Codex، وتمكين Plugin `codex` المضمّن، واستخدام
مرجع نموذج `openai/gpt-*` قياسي.

سجّل الدخول باستخدام Codex OAuth:

```bash
openclaw models auth login --provider openai
```

مكّن Plugin `codex` المضمّن وحدد نموذج وكيل OpenAI:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
    },
  },
}
```

إذا كان إعدادك يستخدم `plugins.allow`، فأضف `codex` هناك أيضًا:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

أعد تشغيل Gateway بعد تغيير إعداد Plugin. إذا كانت دردشة موجودة لديها جلسة بالفعل،
فاستخدم `/new` أو `/reset` قبل اختبار تغييرات وقت التشغيل حتى تحلّ الدورة التالية
الحاضنة من الإعداد الحالي.

## الإعداد

إعداد البدء السريع هو الحد الأدنى القابل للاستخدام لإعداد حاضنة Codex. اضبط خيارات
حاضنة Codex في إعداد OpenClaw، واستخدم CLI فقط لمصادقة Codex:

| الحاجة                                  | اضبط                                                                             | المكان                            |
| --------------------------------------- | -------------------------------------------------------------------------------- | --------------------------------- |
| تمكين الحاضنة                           | `plugins.entries.codex.enabled: true`                                            | إعداد OpenClaw                    |
| الاحتفاظ بتثبيت Plugin ضمن قائمة السماح | Include `codex` in `plugins.allow`                                               | إعداد OpenClaw                    |
| توجيه دورات وكيل OpenAI عبر Codex       | `agents.defaults.model` or `agents.list[].model` as `openai/gpt-*`               | إعداد وكيل OpenClaw               |
| تسجيل الدخول باستخدام ChatGPT/Codex OAuth | `openclaw models auth login --provider openai`                                   | ملف مصادقة CLI                    |
| إضافة احتياطي مفتاح API لتشغيلات Codex  | `openai:*` API-key profile listed after subscription auth in `auth.order.openai` | ملف مصادقة CLI + إعداد OpenClaw   |
| الإغلاق بفشل عند عدم توفر Codex         | Provider or model `agentRuntime.id: "codex"`                                     | إعداد نموذج/مزوّد OpenClaw        |
| استخدام حركة OpenAI API المباشرة        | Provider or model `agentRuntime.id: "openclaw"` with normal OpenAI auth          | إعداد نموذج/مزوّد OpenClaw        |
| ضبط سلوك خادم التطبيق                   | `plugins.entries.codex.config.appServer.*`                                       | إعداد Plugin Codex                |
| تمكين تطبيقات Plugin Codex الأصلية      | `plugins.entries.codex.config.codexPlugins.*`                                    | إعداد Plugin Codex                |
| تمكين استخدام حاسوب Codex               | `plugins.entries.codex.config.computerUse.*`                                     | إعداد Plugin Codex                |

استخدم مراجع نماذج `openai/gpt-*` لدورات وكيل OpenAI المدعومة بـ Codex. فضّل
`auth.order.openai` للترتيب الذي يبدأ بالاشتراك ثم يستخدم مفتاح API كاحتياطي. معرّفات
ملفات مصادقة Codex القديمة وترتيب مصادقة Codex القديم هي حالة قديمة خاصة بالطبيب فقط؛
لا تكتب مراجع Codex GPT قديمة جديدة.

لا تضبط `compaction.model` أو `compaction.provider` على الوكلاء المدعومين بـ Codex.
ينفذ Codex عملية Compaction عبر حالة سلسلة خادم التطبيق الأصلية، لذلك يتجاهل OpenClaw
تجاوزات الملخّص المحلي هذه في وقت التشغيل ويزيلها `openclaw doctor --fix`
عندما يستخدم الوكيل Codex.

يبقى Lossless مدعومًا كمحرك سياق للتجميع، والاستيعاب، والصيانة حول دورات Codex. اضبطه عبر
`plugins.slots.contextEngine: "lossless-claw"` و
`plugins.entries.lossless-claw.config.summaryModel`، وليس عبر
`agents.defaults.compaction.provider`. يرحّل `openclaw doctor --fix` الشكل القديم
`compaction.provider: "lossless-claw"` إلى خانة محرك السياق Lossless
عندما يكون Codex هو وقت التشغيل النشط، لكن Codex الأصلي يظل مسؤولًا عن Compaction.

تدعم حاضنة خادم تطبيق Codex الأصلية محركات السياق التي تتطلب
تجميعًا قبل الموجّه. لا توفر الواجهات الخلفية العامة لـ CLI، بما في ذلك `codex-cli`،
قدرة المضيف هذه.

بالنسبة إلى الوكلاء المدعومين بـ Codex، يبدأ `/compact` عملية Compaction أصلية في خادم تطبيق Codex
على السلسلة المرتبطة. لا ينتظر OpenClaw الإكمال، ولا يفرض مهلة OpenClaw،
ولا يعيد تشغيل خادم التطبيق المشترك، ولا يعود إلى محرك سياق أو
ملخّص OpenAI عام. إذا كان ربط سلسلة Codex الأصلية مفقودًا أو قديمًا،
يفشل الأمر بإغلاق حتى يرى المشغّل حد وقت التشغيل الحقيقي
بدلًا من تبديل خلفيات Compaction بصمت.

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

في هذا الشكل، لا يزال كلا الملفين يعملان عبر Codex لدورات وكيل `openai/gpt-*`.
مفتاح API هو مجرد بديل للمصادقة، وليس طلبًا للتبديل إلى OpenClaw أو
OpenAI Responses العادي.

يغطي باقي هذه الصفحة المتغيرات الشائعة التي يجب على المستخدمين الاختيار بينها:
شكل النشر، والتوجيه الذي يفشل بإغلاق، وسياسة موافقة الحارس، وPlugins Codex الأصلية،
واستخدام الحاسوب. للحصول على قوائم الخيارات الكاملة، والقيم الافتراضية، والتعدادات، والاكتشاف،
وعزل البيئة، والمهلات، وحقول نقل خادم التطبيق، راجع
[مرجع حاضنة Codex](/ar/plugins/codex-harness-reference).

## التحقق من وقت تشغيل Codex

استخدم `/status` في الدردشة التي تتوقع فيها Codex. تعرض دورة وكيل OpenAI المدعومة بـ Codex:

```text
Runtime: OpenAI Codex
```

ثم تحقق من حالة خادم تطبيق Codex:

```text
/codex status
/codex models
```

يعرض `/codex status` اتصال خادم التطبيق، والحساب، وحدود المعدل، وخوادم MCP،
وSkills. يسرد `/codex models` كتالوج خادم تطبيق Codex المباشر للحاضنة والحساب.
إذا كانت نتيجة `/status` مفاجئة، فراجع
[استكشاف الأخطاء وإصلاحها](#troubleshooting).

## التوجيه واختيار النموذج

أبقِ مراجع المزوّد وسياسة وقت التشغيل منفصلين:

- استخدم `openai/gpt-*` لدورات وكيل OpenAI عبر Codex.
- لا تستخدم مراجع Codex GPT القديمة في الإعداد. شغّل `openclaw doctor --fix` لإصلاح
  المراجع القديمة وتثبيتات مسار الجلسة الراكدة.
- `agentRuntime.id: "codex"` اختياري لوضع OpenAI التلقائي العادي، لكنه مفيد
  عندما يجب أن يفشل النشر بإغلاق إذا كان Codex غير متوفر.
- `agentRuntime.id: "openclaw"` يوجّه مزوّدًا أو نموذجًا إلى وقت تشغيل OpenClaw
  المضمّن عندما يكون ذلك مقصودًا.
- يتحكم `/codex ...` في محادثات خادم تطبيق Codex الأصلية من الدردشة.
- ACP/acpx هو مسار حاضنة خارجي منفصل. استخدمه فقط عندما يطلب المستخدم
  ACP/acpx أو محوّل حاضنة خارجيًا.

توجيه الأوامر الشائعة:

| نية المستخدم                                           | استخدم                                                                                                 |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------ |
| إرفاق الدردشة الحالية                                  | `/codex bind [--cwd <path>]`                                                                           |
| استئناف سلسلة Codex موجودة                             | `/codex resume <thread-id>`                                                                            |
| سرد سلاسل Codex أو تصفيتها                             | `/codex threads [filter]`                                                                              |
| سرد Plugins Codex الأصلية                              | `/codex plugins list`                                                                                  |
| تمكين Plugin Codex أصلي مضبوط أو تعطيله                | `/codex plugins enable <name>`, `/codex plugins disable <name>`                                        |
| إرفاق جلسة Codex CLI موجودة على عقدة مقترنة            | `/codex sessions --host <node> [filter]`, then `/codex resume <session-id> --host <node> --bind here` |
| إرسال ملاحظات Codex فقط                                | `/codex diagnostics [note]`                                                                            |
| بدء مهمة ACP/acpx                                      | أوامر جلسة ACP/acpx، وليس `/codex`                                                                     |

| حالة الاستخدام                                             | الإعداد                                                              | التحقق                                  | ملاحظات                                 |
| ---------------------------------------------------- | ---------------------------------------------------------------------- | --------------------------------------- | ------------------------------------- |
| اشتراك ChatGPT/Codex مع وقت تشغيل Codex الأصلي | `openai/gpt-*` مع تفعيل Plugin `codex`                             | يعرض `/status` القيمة `Runtime: OpenAI Codex` | المسار الموصى به                      |

يتبع `agents.defaults.imageModel` تقسيم البادئات نفسه. استخدم `openai/gpt-*`
للمسار العادي في OpenAI، و`codex/gpt-*` فقط عندما ينبغي أن يعمل فهم الصور
عبر دورة محدودة في خادم تطبيقات Codex. لا تستخدم
مراجع Codex GPT القديمة؛ إذ يعيد doctor كتابة تلك البادئة القديمة إلى `openai/gpt-*`.

## أنماط النشر

### نشر Codex الأساسي

استخدم إعداد البدء السريع عندما يجب أن تستخدم كل دورات وكيل OpenAI Codex
افتراضيًا.

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
    },
  },
}
```

### نشر مختلط للمزوّدين

يبقي هذا الشكل Claude كوكيل افتراضي ويضيف وكيل Codex مسمى:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-6",
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
      },
    ],
  },
}
```

باستخدام هذا الإعداد، يستخدم الوكيل `main` مسار المزوّد العادي الخاص به، ويستخدم
الوكيل `codex` خادم تطبيقات Codex.

### نشر Codex بفشل مغلق

بالنسبة إلى دورات وكيل OpenAI، يحل `openai/gpt-*` بالفعل إلى Codex عندما يكون
Plugin المضمّن متاحًا. أضف سياسة وقت تشغيل صريحة عندما تريد قاعدة مكتوبة
للفشل المغلق:

```json5
{
  models: {
    providers: {
      openai: {
        agentRuntime: {
          id: "codex",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

عند فرض Codex، يفشل OpenClaw مبكرًا إذا كان Plugin الخاص بـ Codex معطلًا، أو كان
خادم التطبيقات قديمًا جدًا، أو تعذر بدء خادم التطبيقات.

## سياسة خادم التطبيقات

افتراضيًا، يبدأ Plugin ثنائية Codex المُدارة من OpenClaw محليًا مع نقل stdio.
عيّن `appServer.command` فقط عندما تريد عمدًا تشغيل ملف تنفيذي مختلف.
استخدم نقل WebSocket فقط عندما يكون خادم تطبيقات قيد التشغيل بالفعل في مكان آخر:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
          },
        },
      },
    },
  },
}
```

تستخدم جلسات خادم تطبيقات stdio المحلية افتراضيًا وضعية المشغّل المحلي الموثوق:
`approvalPolicy: "never"` و`approvalsReviewer: "user"` و
`sandbox: "danger-full-access"`. إذا كانت متطلبات Codex المحلية لا تسمح بتلك
الوضعية الضمنية YOLO، يحدد OpenClaw أذونات الحارس المسموح بها بدلًا من ذلك.
عندما يكون صندوق رمل OpenClaw نشطًا للجلسة، يعطل OpenClaw وضع Code Mode الأصلي
في Codex وخوادم MCP الخاصة بالمستخدم وتنفيذ Plugin المدعوم بالتطبيقات لتلك
الدورة بدلًا من الاعتماد على عزل Codex من جهة المضيف. يُكشف وصول الصدفة
عبر أدوات OpenClaw الديناميكية المدعومة بصندوق الرمل مثل `sandbox_exec` و
`sandbox_process` عندما تكون أدوات exec/process العادية متاحة.

استخدم وضع تنفيذ OpenClaw الموحّد عندما تريد مراجعة Codex التلقائية الأصلية قبل
تجاوز صناديق الرمل أو الأذونات الإضافية:

```json5
{
  tools: {
    exec: {
      mode: "auto",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

بالنسبة إلى جلسات خادم تطبيقات Codex، يطابق OpenClaw `tools.exec.mode: "auto"` مع موافقات
Codex التي يراجعها Guardian، عادةً
`approvalPolicy: "on-request"` و`approvalsReviewer: "auto_review"` و
`sandbox: "workspace-write"` عندما تسمح المتطلبات المحلية بهذه القيم.
في `tools.exec.mode: "auto"`، لا يحتفظ OpenClaw بتجاوزات Codex القديمة غير الآمنة
`approvalPolicy: "never"` أو `sandbox: "danger-full-access"`؛ استخدم
`tools.exec.mode: "full"` لوضعية Codex مقصودة بلا موافقة. لا يزال إعداد
`plugins.entries.codex.config.appServer.mode: "guardian"` القديم يعمل، لكن
`tools.exec.mode: "auto"` هو سطح OpenClaw الموحّد.

للمقارنة على مستوى الوضع مع موافقات exec المضيفة وأذونات ACPX،
راجع [أوضاع الأذونات](/ar/tools/permission-modes).

لكل حقل في خادم التطبيقات، وترتيب المصادقة، وعزل البيئة، والاكتشاف، وسلوك
المهلة، راجع [مرجع عدة Codex](/ar/plugins/codex-harness-reference).

## الأوامر والتشخيصات

يسجل Plugin المضمّن `/codex` كأمر شرطة مائلة على أي قناة تدعم أوامر OpenClaw النصية.

يتطلب التنفيذ والتحكم الأصليان مالكًا أو عميل Gateway بصلاحية `operator.admin`.
يشمل ذلك ربط الخيوط أو استئنافها، وإرسال الدورات أو إيقافها،
وتغيير النموذج أو وضع السرعة أو حالة الأذونات، والضغط أو المراجعة، و
فصل الربط. يحتفظ المرسلون المخولون الآخرون بأوامر القراءة فقط للحالة والمساعدة
والحساب والنموذج والخيط وخادم MCP وskill وفحص الربط.

الأشكال الشائعة:

- يتحقق `/codex status` من اتصال خادم التطبيقات والنماذج والحساب وحدود المعدل
  وخوادم MCP وskills.
- يسرد `/codex models` نماذج خادم تطبيقات Codex الحية.
- يسرد `/codex threads [filter]` خيوط خادم تطبيقات Codex الأخيرة.
- يرفق `/codex resume <thread-id>` جلسة OpenClaw الحالية
  بخيط Codex موجود.
- يطلب `/codex compact` من خادم تطبيقات Codex ضغط الخيط المرفق.
- يبدأ `/codex review` مراجعة Codex الأصلية للخيط المرفق.
- يطلب `/codex diagnostics [note]` قبل إرسال ملاحظات Codex للخيط
  المرفق.
- يعرض `/codex account` حالة الحساب وحدود المعدل.
- يسرد `/codex mcp` حالة خادم MCP في خادم تطبيقات Codex.
- يسرد `/codex skills` مهارات خادم تطبيقات Codex.

بالنسبة إلى معظم تقارير الدعم، ابدأ بـ `/diagnostics [note]` في المحادثة
التي حدث فيها الخلل. ينشئ ذلك تقرير تشخيص Gateway واحدًا، وبالنسبة إلى جلسات
عدة Codex، يطلب الموافقة على إرسال حزمة ملاحظات Codex ذات الصلة.
راجع [تصدير التشخيصات](/ar/gateway/diagnostics) لنموذج الخصوصية وسلوك الدردشات
الجماعية.

استخدم `/codex diagnostics [note]` فقط عندما تريد تحديدًا رفع ملاحظات Codex
للخيط المرفق حاليًا دون حزمة تشخيصات Gateway الكاملة.

### فحص خيوط Codex محليًا

غالبًا ما تكون أسرع طريقة لفحص تشغيل Codex سيئ هي فتح خيط Codex الأصلي
مباشرةً:

```bash
codex resume <thread-id>
```

احصل على معرف الخيط من رد `/diagnostics` المكتمل أو `/codex binding` أو
`/codex threads [filter]`.

لآليات الرفع وحدود التشخيصات على مستوى وقت التشغيل، راجع
[وقت تشغيل عدة Codex](/ar/plugins/codex-harness-runtime#codex-feedback-upload).

تُختار المصادقة بهذا الترتيب:

1. ملفات مصادقة OpenAI المرتبة للوكيل، ويفضل أن تكون تحت
   `auth.order.openai`. شغّل `openclaw doctor --fix` لترحيل
   معرفات ملفات مصادقة Codex القديمة وترتيب مصادقة Codex القديم.
2. حساب خادم التطبيقات الموجود في موطن Codex لذلك الوكيل.
3. لعمليات تشغيل خادم تطبيقات stdio المحلية فقط، `CODEX_API_KEY`، ثم
   `OPENAI_API_KEY`، عندما لا يكون هناك حساب خادم تطبيقات موجود وتظل مصادقة OpenAI
   مطلوبة.

عندما يرى OpenClaw ملف مصادقة Codex بنمط اشتراك ChatGPT، فإنه يزيل
`CODEX_API_KEY` و`OPENAI_API_KEY` من عملية Codex الفرعية المولدة. يحافظ ذلك
على مفاتيح API على مستوى Gateway متاحة للتضمينات أو نماذج OpenAI المباشرة
من دون جعل دورات خادم تطبيقات Codex الأصلية تُحتسب عبر API عن طريق الخطأ.
تستخدم ملفات مفاتيح API الصريحة الخاصة بـ Codex والرجوع المحلي إلى مفتاح البيئة في stdio
تسجيل دخول خادم التطبيقات بدلًا من بيئة العملية الفرعية الموروثة. لا تتلقى
اتصالات خادم تطبيقات WebSocket رجوع مفاتيح API البيئية من Gateway؛ استخدم ملف
مصادقة صريحًا أو حساب خادم التطبيقات البعيد نفسه.
عند إعداد Plugins Codex الأصلية، يثبت OpenClaw تلك Plugins أو يحدّثها
عبر خادم التطبيقات المتصل قبل كشف التطبيقات المملوكة من Plugin لخيط Codex.
يبقى `app/list` مصدر الحقيقة لمعرفات التطبيقات وإمكانية الوصول والبيانات الوصفية،
لكن OpenClaw يملك قرار التمكين لكل خيط: إذا سمحت السياسة بتطبيق مدرج قابل للوصول،
يرسل OpenClaw
`thread/start.config.apps[appId].enabled = true` حتى عندما يبلغ `app/list` حاليًا
أن ذلك التطبيق معطل. لا يخترع هذا المسار تثبيت تطبيق للمعرفات غير المعروفة؛
يفعّل OpenClaw Plugins السوق فقط باستخدام `plugin/install`
ثم يحدّث المخزون.

إذا بلغ ملف اشتراك حد استخدام Codex، يسجل OpenClaw وقت إعادة التعيين عندما
يبلغ Codex عنه، ويحاول ملف المصادقة المرتب التالي لتشغيل Codex نفسه.
عندما يمر وقت إعادة التعيين، يصبح ملف الاشتراك مؤهلًا مرة أخرى
من دون تغيير نموذج `openai/gpt-*` المحدد أو وقت تشغيل Codex.

لعمليات تشغيل خادم تطبيقات stdio المحلية، يعيّن OpenClaw `CODEX_HOME` إلى دليل
خاص بكل وكيل حتى لا تقرأ إعدادات Codex وملفات المصادقة/الحساب وذاكرة التخزين المؤقت/بيانات Plugin وحالة
الخيوط الأصلية من `~/.codex` الشخصي للمشغّل أو تكتب إليه
افتراضيًا. يحافظ OpenClaw على `HOME` العادي للعملية؛ إذ لا تزال العمليات الفرعية التي يشغّلها Codex
قادرة على العثور على إعدادات وتوكنات موطن المستخدم، وقد يكتشف Codex إدخالات
`$HOME/.agents/skills` و`$HOME/.agents/plugins/marketplace.json` المشتركة.

إذا احتاج نشر ما إلى عزل بيئي إضافي، فأضف تلك المتغيرات إلى
`appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

يؤثر `appServer.clearEnv` فقط في عملية خادم تطبيقات Codex الفرعية المولدة.
يزيل OpenClaw `CODEX_HOME` و`HOME` من هذه القائمة أثناء تطبيع التشغيل المحلي:
يبقى `CODEX_HOME` خاصًا بكل وكيل، ويبقى `HOME` موروثًا حتى تتمكن
العمليات الفرعية من استخدام حالة موطن المستخدم العادية.

تُحمَّل أدوات Codex الديناميكية افتراضياً بوضع `searchable`. لا يعرّض OpenClaw
أدوات ديناميكية تكرر عمليات مساحة العمل الأصلية في Codex: `read` و`write`
و`edit` و`apply_patch` و`exec` و`process` و`update_plan`. تتوفر معظم أدوات تكامل
OpenClaw المتبقية مثل المراسلة، والوسائط، وCron، والمتصفح، والعقد،
وGateway، و`heartbeat_respond` عبر بحث أدوات Codex ضمن مساحة الأسماء
`openclaw`، مما يبقي سياق النموذج الأولي أصغر. يستخدم بحث الويب أداة
`web_search` المستضافة في Codex افتراضياً عند تمكين البحث وعدم تحديد موفر
مُدار. البحث المستضاف الأصلي وأداة OpenClaw الديناميكية المُدارة
`web_search` متنافيان، بحيث لا يمكن للبحث المُدار تجاوز قيود النطاق الأصلية.
يستخدم OpenClaw الأداة المُدارة عندما يكون البحث المستضاف غير متاح، أو معطلاً
صراحةً، أو مستبدلاً بموفر مُدار محدد. يُبقي OpenClaw امتداد Codex المستقل
`web.run` معطلاً لأن حركة مرور خادم التطبيق الإنتاجية ترفض مساحة الأسماء
`web` المعرفة من المستخدم. يعطل `tools.web.search.enabled: false` كلا
المسارين، وكذلك عمليات تشغيل LLM فقط التي تكون الأدوات فيها معطلة. يتعامل
Codex مع `"cached"` كتفضيل ويحوّله إلى وصول خارجي مباشر لأدوار خادم التطبيق
غير المقيدة. يفشل الرجوع التلقائي المُدار بإغلاق آمن عند تعيين
`allowedDomains` الأصلية، بحيث لا يمكن تجاوز قائمة السماح. تؤدي تغييرات سياسة
البحث الفعالة المستمرة إلى تدوير سلسلة Codex المرتبطة قبل الدور التالي. تستخدم
القيود المؤقتة لكل دور سلسلة مؤقتة مقيدة وتحافظ على الربط الحالي للاستئناف
لاحقاً. تبقى ردود المصدر الخاصة بـ`sessions_yield` وأدوات الرسائل فقط مباشرة
لأنها عقود تحكم بالدور. يبقى `sessions_spawn` قابلاً للبحث بحيث يظل
`spawn_agent` الأصلي في Codex هو سطح الوكلاء الفرعيين الأساسي في Codex، بينما
يبقى تفويض OpenClaw أو ACP الصريح متاحاً عبر مساحة أسماء أدوات `openclaw`
الديناميكية. تطلب تعليمات التعاون في Heartbeat من Codex البحث عن
`heartbeat_respond` قبل إنهاء دور Heartbeat عندما لا تكون الأداة محملة مسبقاً.

عيّن `codexDynamicToolsLoading: "direct"` فقط عند الاتصال بخادم تطبيق Codex
مخصص لا يمكنه البحث في الأدوات الديناميكية المؤجلة أو عند تصحيح حمولة الأدوات
الكاملة.

`appServer.networkProxy` صريح لأنه يغيّر عقد صندوق رمل Codex.
عند تفعيله، يضبط OpenClaw أيضا `features.network_proxy.enabled` و
`default_permissions` في إعدادات خيط Codex بحيث يستطيع ملف الأذونات
المُنشأ بدء شبكات Codex المُدارة. افتراضيا، ينشئ OpenClaw اسم ملف
`openclaw-network-<fingerprint>` مقاوم للتصادم من جسم الملف؛ استخدم
`profileName` فقط عندما يكون الاسم المحلي المستقر مطلوبا.

```js
export default {
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            sandbox: "workspace-write",
            networkProxy: {
              enabled: true,
              domains: {
                "api.openai.com": "allow",
                "blocked.example.com": "deny",
              },
              unixSockets: {
                "/tmp/proxy.sock": "allow",
                "/tmp/blocked.sock": "none",
              },
              allowUpstreamProxy: true,
              proxyUrl: "http://127.0.0.1:3128",
            },
          },
        },
      },
    },
  },
};
```

إذا كان وقت تشغيل app-server العادي سيكون `danger-full-access`، فإن تفعيل
`networkProxy` يستخدم وصولا إلى نظام الملفات بأسلوب مساحة العمل لملف
الأذونات المُنشأ. إن فرض الشبكة المُدار بواسطة Codex هو شبكات ضمن صندوق رمل،
لذلك لن يحمي ملف الوصول الكامل حركة البيانات الصادرة.
تستخدم إدخالات النطاق `allow` أو `deny`؛ وتستخدم إدخالات مقابس Unix قيم
Codex وهي `allow` أو `none`.

تُقيَّد استدعاءات الأدوات الديناميكية المملوكة لـ OpenClaw بشكل مستقل عن
`appServer.requestTimeoutMs`: تستخدم طلبات Codex من نوع `item/tool/call` مراقب OpenClaw
افتراضيًا لمدة 90 ثانية. تؤدي وسيطة `timeoutMs` الإيجابية لكل استدعاء إلى تمديد
أو تقصير ميزانية تلك الأداة المحددة. تستخدم أداة `image_generate`
`agents.defaults.imageGenerationModel.timeoutMs` عندما لا يوفر استدعاء الأداة
مهلته الخاصة، أو تستخدم افتراضيًا مهلة 120 ثانية لتوليد الصور بخلاف ذلك.
تستخدم أداة `image` لفهم الوسائط
`tools.media.image.timeoutSeconds` أو افتراضي الوسائط البالغ 60 ثانية. بالنسبة إلى
فهم الصور، تنطبق تلك المهلة على الطلب نفسه ولا تُخفَّض بسبب أعمال التحضير السابقة.
تُحدَّد ميزانيات الأدوات الديناميكية بسقف قدره 600000 ms. عند انتهاء المهلة، يُجهض OpenClaw
إشارة الأداة حيث يكون ذلك مدعومًا، ويُرجع استجابة أداة ديناميكية فاشلة إلى Codex حتى يمكن
للدورة أن تستمر بدلًا من ترك الجلسة في `processing`.
هذا المراقب هو ميزانية `item/tool/call` الديناميكية الخارجية؛ أما مهل الطلبات الخاصة
بالموفر فتعمل داخل ذلك الاستدعاء وتحافظ على دلالات المهلة الخاصة بها.

بعد أن يقبل Codex دورة، وبعد أن يستجيب OpenClaw لطلب خادم تطبيق مقيّد بنطاق الدورة،
يتوقع الحزام من Codex أن يحرز تقدمًا في الدورة الحالية وأن ينهي في النهاية الدورة الأصلية
باستخدام `turn/completed`. إذا ظل خادم التطبيق صامتًا لمدة `appServer.turnCompletionIdleTimeoutMs`،
يقاطع OpenClaw دورة Codex بأفضل جهد، ويسجل مهلة تشخيصية، ويحرر مسار جلسة
OpenClaw حتى لا تُعلَّق رسائل الدردشة اللاحقة خلف دورة أصلية قديمة. تقوم معظم الإشعارات
غير النهائية للدورة نفسها بإلغاء تفعيل ذلك المراقب القصير لأن Codex أثبت أن الدورة لا تزال حية.
تستخدم تسليمات الأدوات ميزانية خمول أطول بعد الأداة: بعد أن يُرجع OpenClaw استجابة
`item/tool/call`، وبعد اكتمال عناصر الأدوات الأصلية مثل `commandExecution`، وبعد اكتمالات
`custom_tool_call_output` الخام، وبعد تقدم المساعد الخام بعد الأداة، أو اكتمالات التفكير الخام،
أو تقدم التفكير. يستخدم الحارس `appServer.postToolRawAssistantCompletionIdleTimeoutMs` عند تهيئته
ويعود افتراضيًا إلى خمس دقائق بخلاف ذلك. تمد ميزانية ما بعد الأداة نفسها أيضًا مراقب التقدم
لنافذة التوليف الصامتة قبل أن يصدر Codex حدث الدورة الحالية التالي. لا تعيد إشعارات خادم التطبيق
العامة، مثل تحديثات حدود المعدل، ضبط تقدم خمول الدورة. يمكن أن تتبع اكتمالات التفكير،
واكتمالات `agentMessage` في التعليق، وتقدم التفكير الخام أو المساعد الخام قبل الأداة
استجابة نهائية تلقائية، لذلك تستخدم حارس الرد بعد التقدم بدلًا من تحرير مسار الجلسة فورًا.
وحدها عناصر `agentMessage` النهائية/غير التعليقية المكتملة واكتمالات المساعد الخام قبل الأداة
تفعّل تحرير مخرجات المساعد: إذا صمت Codex بعد ذلك من دون `turn/completed`، يقاطع OpenClaw
الدورة الأصلية بأفضل جهد ويحرر مسار الجلسة. تُعاد محاولة إخفاقات خادم تطبيق stdio الآمنة
لإعادة التشغيل، بما في ذلك مهل خمول اكتمال الدورة من دون دليل على مساعد أو أداة أو عنصر نشط
أو أثر جانبي، مرة واحدة في محاولة جديدة لخادم التطبيق. أما المهل غير الآمنة فما زالت تُحيل
عميل خادم التطبيق العالق إلى التقاعد وتحرر مسار جلسة OpenClaw. كما تمسح ربط الخيط الأصلي
القديم بدلًا من إعادة تشغيله تلقائيًا. تُظهر مهل مراقبة الاكتمال نص مهلة خاصًا بـ Codex:
تقول الحالات الآمنة لإعادة التشغيل إن الاستجابة قد تكون غير مكتملة، بينما تطلب الحالات غير الآمنة
من المستخدم التحقق من الحالة الحالية قبل إعادة المحاولة. تتضمن تشخيصات المهلة العامة حقولًا
بنيوية مثل آخر طريقة إشعار من خادم التطبيق، ومعرّف/نوع/دور عنصر استجابة المساعد الخام،
وأعداد الطلبات/العناصر النشطة، وحالة المراقبة المفعّلة. عندما يكون آخر إشعار عنصر استجابة
مساعد خام، فإنها تتضمن أيضًا معاينة محدودة لنص المساعد. ولا تتضمن محتوى المطالبة الخام
أو محتوى الأداة.

تظل تجاوزات البيئة متاحة للاختبار المحلي:

- `OPENCLAW_CODEX_APP_SERVER_BIN`
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

يتجاوز `OPENCLAW_CODEX_APP_SERVER_BIN` الثنائي المُدار عندما لا يكون
`appServer.command` مضبوطًا.

أُزيل `OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1`. استخدم
`plugins.entries.codex.config.appServer.mode: "guardian"` بدلًا منه، أو
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` للاختبار المحلي لمرة واحدة. تُفضَّل التهيئة
لعمليات النشر القابلة للتكرار لأنها تُبقي سلوك Plugin في الملف المراجع نفسه كبقية إعداد
حزام Codex.

## Plugins الأصلية لـ Codex

يستخدم دعم Plugins الأصلية لـ Codex إمكانات التطبيق وPlugin الخاصة بخادم تطبيق Codex
داخل خيط Codex نفسه مثل دورة حزام OpenClaw. لا يترجم OpenClaw Plugins الخاصة بـ Codex
إلى أدوات ديناميكية اصطناعية من نوع `codex_plugin_*` في OpenClaw.

يؤثر `codexPlugins` فقط في الجلسات التي تختار حزام Codex الأصلي. ولا تأثير له في تشغيلات
الحزام المضمنة، أو تشغيلات موفر OpenAI العادية، أو روابط محادثات ACP، أو الأحزمة الأخرى.

التهيئة الدنيا المرحّلة:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

تُحسب تهيئة تطبيق الخيط عندما ينشئ OpenClaw جلسة حزام Codex أو يستبدل ربط خيط Codex
قديمًا. ولا يُعاد حسابها في كل دورة. بعد تغيير `codexPlugins`، استخدم `/new` أو `/reset`
أو أعد تشغيل Gateway حتى تبدأ جلسات حزام Codex المستقبلية بمجموعة التطبيقات المحدثة.

لأهلية الترحيل، ومخزون التطبيقات، وسياسة الإجراءات التدميرية، وطلبات الاستيضاح، وتشخيصات
Plugin الأصلي، راجع
[Plugins الأصلية لـ Codex](/ar/plugins/codex-native-plugins).

يتحكم حساب Codex المسجّل الدخول في الوصول إلى التطبيقات وPlugins من جهة OpenAI،
وكذلك عناصر التحكم في تطبيقات مساحة العمل بالنسبة إلى مساحات عمل Business وEnterprise/Edu. راجع
[استخدام Codex مع خطة ChatGPT الخاصة بك](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan)
للحصول على نظرة عامة من OpenAI حول عناصر التحكم في الحساب ومساحة العمل.

## استخدام الكمبيوتر

استخدام الكمبيوتر مشروح في دليل الإعداد الخاص به:
[استخدام الكمبيوتر في Codex](/ar/plugins/codex-computer-use).

الخلاصة: لا يضمّن OpenClaw تطبيق التحكم بسطح المكتب ولا ينفذ إجراءات سطح المكتب بنفسه.
إنه يجهز خادم تطبيق Codex، ويتحقق من توفر خادم MCP الخاص بـ `computer-use`، ثم يترك
Codex يملك استدعاءات أدوات MCP الأصلية أثناء دورات وضع Codex.

## حدود وقت التشغيل

يغيّر حزام Codex منفذ الوكيل المضمن منخفض المستوى فقط.

- أدوات OpenClaw الديناميكية مدعومة. يطلب Codex من OpenClaw تنفيذ تلك الأدوات،
  لذلك يبقى OpenClaw ضمن مسار التنفيذ.
- أدوات الصدفة والتصحيح وMCP والتطبيقات الأصلية الخاصة بـ Codex مملوكة لـ Codex.
  يمكن لـ OpenClaw ملاحظة أحداث أصلية مختارة أو حظرها عبر المرحّل المدعوم،
  لكنه لا يعيد كتابة وسيطات الأدوات الأصلية.
- يملك Codex الضغط الأصلي. يحتفظ OpenClaw بمرآة للنص لأجل سجل القناة
  والبحث و`/new` و`/reset` والتبديل المستقبلي بين النموذج أو الحزام، لكنه
  لا يستبدل ضغط Codex بملخّص من OpenClaw أو محرك السياق.
- يستمر توليد الوسائط وفهم الوسائط وTTS والموافقات ومخرجات أدوات المراسلة
  عبر إعدادات موفر/نموذج OpenClaw المطابقة.
- ينطبق `tool_result_persist` على نتائج أدوات النص المملوكة لـ OpenClaw، وليس
  سجلات نتائج أدوات Codex الأصلية.

للاطلاع على طبقات الخطافات، والأسطح المدعومة في V1، ومعالجة الأذونات الأصلية، وتوجيه
قوائم الانتظار، وآليات رفع ملاحظات Codex، وتفاصيل الضغط، راجع
[وقت تشغيل حزام Codex](/ar/plugins/codex-harness-runtime).

## استكشاف الأخطاء وإصلاحها

**لا يظهر Codex كموفر `/model` عادي:** هذا متوقع في التهيئات الجديدة. اختر نموذج
`openai/gpt-*`، وفعّل `plugins.entries.codex.enabled`، وتحقق مما إذا كان `plugins.allow`
يستثني `codex`.

**يستخدم OpenClaw الحزام المضمن بدلًا من Codex:** تأكد من أن مرجع النموذج هو
`openai/gpt-*` على موفر OpenAI الرسمي وأن Plugin الخاص بـ Codex مثبت ومفعّل.
إذا كنت تحتاج إلى دليل صارم أثناء الاختبار، فاضبط `agentRuntime.id: "codex"` على مستوى الموفر
أو النموذج. يفشل وقت تشغيل Codex القسري بدلًا من الرجوع إلى OpenClaw.

**يرجع وقت تشغيل OpenAI Codex إلى مسار مفتاح API:** اجمع مقتطف Gateway منقحًا
يوضح النموذج ووقت التشغيل والموفر المختار والفشل. اطلب من المتعاونين المتأثرين تشغيل هذا الأمر
للقراءة فقط على مضيف OpenClaw لديهم:

```bash
(
  pattern='openai/gpt-5\.[45]|openai[-]codex|agentRuntime(\.id)?|harnessRuntime|Runtime: OpenAI Codex|legacy OpenAI Codex prefix|resolveSelectedOpenAIRuntimeProvider|candidateProvider[": ]+openai|status[": ]+401|Incorrect API key|No API key|api-key path|API-key path|OAuth'

  if ls /tmp/openclaw/openclaw-*.log >/dev/null 2>&1; then
    grep -E -i -n "$pattern" /tmp/openclaw/openclaw-*.log 2>/dev/null || true
  else
    journalctl --user -u openclaw-gateway --since today --no-pager 2>/dev/null \
      | grep -E -i "$pattern" || true
  fi
) | sed -E \
    -e 's/(Authorization: Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(api[_ -]?key[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/(OPENAI_API_KEY[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/sk-[A-Za-z0-9_-]{12,}/sk-[REDACTED]/g' \
    -e 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL-REDACTED]/g' \
  | tail -200
```

تتضمن المقتطفات المفيدة عادةً `openai/gpt-5.5` أو `openai/gpt-5.4`،
و`Runtime: OpenAI Codex`، و`agentRuntime.id` أو `harnessRuntime`،
و`candidateProvider: "openai"`، ونتيجة `401` أو `Incorrect API key` أو
`No API key`. يجب أن يُظهر التشغيل المصحح مسار OAuth الخاص بـ OpenAI
بدلًا من فشل مفتاح API عادي لـ OpenAI.

**لا تزال تهيئة مراجع نماذج Codex القديمة موجودة:** شغّل `openclaw doctor --fix`.
يعيد Doctor كتابة مراجع النماذج القديمة إلى `openai/*`، ويزيل تثبيتات وقت التشغيل القديمة
للجلسة والوكيل بالكامل، ويحافظ على تجاوزات ملفات تعريف المصادقة الحالية.

**رُفض خادم التطبيق:** استخدم خادم تطبيق Codex الإصدار `0.125.0` أو أحدث.
تُرفض إصدارات ما قبل الإصدار من النسخة نفسها أو الإصدارات ذات لاحقة البناء مثل
`0.125.0-alpha.2` أو `0.125.0+custom` لأن OpenClaw يختبر حد بروتوكول
`0.125.0` المستقر الأدنى.

**لا يمكن لـ `/codex status` الاتصال:** تحقق من أن Plugin المضمن `codex` مفعّل،
وأن `plugins.allow` يتضمنه عند تهيئة قائمة سماح، وأن أي `appServer.command` أو
`url` أو `authToken` أو رؤوس مخصصة صالحة.

**اكتشاف النماذج بطيء:** خفّض
`plugins.entries.codex.config.discovery.timeoutMs` أو عطّل الاكتشاف. راجع
[مرجع حزام Codex](/ar/plugins/codex-harness-reference#model-discovery).

**يفشل نقل WebSocket فورًا:** تحقق من `appServer.url` و`authToken`
والرؤوس، وأن خادم التطبيق البعيد يتحدث بإصدار بروتوكول خادم تطبيق Codex نفسه.

**أدوات الصدفة أو التصحيح الأصلية محظورة برسالة `Native hook relay unavailable`:**
لا يزال خيط Codex يحاول استخدام معرّف مرحّل خطاف أصلي لم يعد OpenClaw مسجلًا لديه.
هذه مشكلة نقل خطاف أصلية في Codex، وليست فشلًا في خلفية ACP أو الموفر أو GitHub
أو أمر الصدفة. ابدأ جلسة جديدة في الدردشة المتأثرة باستخدام `/new` أو `/reset`، ثم أعد
محاولة أمر غير ضار. إذا نجح ذلك مرة واحدة لكن فشل استدعاء الأداة الأصلية التالي مرة أخرى،
فتعامل مع `/new` كحل مؤقت فقط: انسخ المطالبة إلى جلسة جديدة بعد إعادة تشغيل خادم تطبيق Codex
أو Gateway الخاص بـ OpenClaw حتى تُسقط الخيوط القديمة وتُعاد إنشاء تسجيلات الخطافات الأصلية.

**يستخدم نموذج غير Codex الحزام المضمن:** هذا متوقع ما لم توجهه سياسة وقت تشغيل الموفر
أو النموذج إلى حزام آخر. تظل مراجع الموفر العادية غير التابعة لـ OpenAI على مسار موفرها
العادي في وضع `auto`.

**Computer Use مثبت لكن الأدوات لا تعمل:** تحقّق من
`/codex computer-use status` من جلسة جديدة. إذا أبلغت أداة عن
`Native hook relay unavailable`، فاستخدم استرداد مرحّل الخطاف الأصلي أعلاه. راجع
[Codex Computer Use](/ar/plugins/codex-computer-use#troubleshooting).
