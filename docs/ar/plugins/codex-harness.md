يحتيح Plugin `codex` المضمّن لـ OpenClaw تشغيل دورات الوكيل المضمّنة عبر
خادم تطبيق Codex بدلا من حاضنة PI المدمجة.

استخدم هذا عندما تريد أن يتولى Codex جلسة الوكيل منخفضة المستوى: اكتشاف
النماذج، واستئناف السلاسل الأصلي، وCompaction الأصلي، وتنفيذ خادم التطبيق.
لا يزال OpenClaw يتولى قنوات الدردشة، وملفات الجلسات، واختيار النموذج، والأدوات،
والموافقات، وتسليم الوسائط، ومرآة النص الظاهرة.

إذا كنت تحاول تكوين تصور عام، فابدأ بـ
[تشغيلات الوكيل](/ar/concepts/agent-runtimes). النسخة المختصرة هي:
`openai/gpt-5.5` هو مرجع النموذج، و`codex` هو التشغيل، وتبقى Telegram،
Discord، Slack، أو قناة أخرى هي سطح التواصل.

## إعداد سريع

لاستخدام حاضنة Codex لدورات وكيل GPT، أبق مرجع النموذج قياسيا بصيغة
`openai/gpt-*`، وفعّل Plugin `codex` المضمّن، واضبط
`agentRuntime.id: "codex"`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
        fallback: "none",
      },
    },
  },
}
```

إذا كان إعدادك يستخدم `plugins.allow`، فأدرج `codex` هناك أيضا:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

لا تستخدم `openai-codex/gpt-*` لهذا المسار. فهذا يختار Codex OAuth عبر
مشغّل PI العادي ما لم تفرض تشغيلا منفصلا. تنطبق تغييرات الإعداد على الجلسات
الجديدة أو المعاد ضبطها؛ أما الجلسات الحالية فتحتفظ بالتشغيل المسجل لديها.

## ما الذي يغيره هذا Plugin

يسهم Plugin `codex` المضمّن بعدة قدرات منفصلة:

| القدرة                            | كيف تستخدمها                                         | ما الذي تفعله                                                                  |
| --------------------------------- | --------------------------------------------------- | ----------------------------------------------------------------------------- |
| تشغيل مضمّن أصلي                 | `agentRuntime.id: "codex"`                          | يشغّل دورات وكيل OpenClaw المضمّنة عبر خادم تطبيق Codex.                     |
| أوامر تحكم دردشة أصلية           | `/codex bind`, `/codex resume`, `/codex steer`, ... | تربط وتتحكم في سلاسل خادم تطبيق Codex من محادثة مراسلة.                     |
| مزود/كتالوج خادم تطبيق Codex     | داخليات `codex`، معروضة عبر الحاضنة                 | يتيح للتشغيل اكتشاف نماذج خادم التطبيق والتحقق منها.                         |
| مسار فهم وسائط Codex             | مسارات توافق نموذج الصور `codex/*`                  | يشغّل دورات محدودة من خادم تطبيق Codex لنماذج فهم الصور المدعومة.           |
| ترحيل الخطافات الأصلي            | خطافات Plugin حول أحداث Codex الأصلية               | يتيح لـ OpenClaw مراقبة/حظر أحداث الأدوات/الإنهاء الأصلية المدعومة في Codex. |

تفعيل Plugin يجعل هذه القدرات متاحة. لكنه **لا**:

- يبدأ باستخدام Codex لكل نموذج OpenAI
- يحوّل مراجع نموذج `openai-codex/*` إلى التشغيل الأصلي
- يجعل ACP/acpx مسار Codex الافتراضي
- يبدّل الجلسات الحالية فوريا إذا كانت قد سجلت تشغيل PI
- يستبدل تسليم قنوات OpenClaw، أو ملفات الجلسات، أو تخزين ملفات تعريف المصادقة، أو
  توجيه الرسائل

يتولى Plugin نفسه أيضا سطح أوامر التحكم بالدردشة الأصلي `/codex`. إذا كان
Plugin مفعّلا وطلب المستخدم الربط، أو الاستئناف، أو التوجيه، أو الإيقاف، أو فحص
سلاسل Codex من الدردشة، فينبغي للوكلاء تفضيل `/codex ...` على ACP. يبقى ACP
الخيار الاحتياطي الصريح عندما يطلب المستخدم ACP/acpx أو يختبر محوّل Codex
الخاص بـ ACP.

تحافظ دورات Codex الأصلية على خطافات Plugin في OpenClaw كطبقة التوافق العامة.
هذه خطافات OpenClaw داخل العملية، وليست خطافات أوامر Codex `hooks.json`:

- `before_prompt_build`
- `before_compaction`, `after_compaction`
- `llm_input`, `llm_output`
- `before_tool_call`, `after_tool_call`
- `before_message_write` لسجلات النص المنسوخة
- `before_agent_finalize` عبر ترحيل Codex `Stop`
- `agent_end`

يمكن للـ Plugins أيضا تسجيل وسيط نتائج أدوات محايد للتشغيل لإعادة كتابة نتائج
أدوات OpenClaw الديناميكية بعد أن ينفذ OpenClaw الأداة وقبل إرجاع النتيجة إلى
Codex. هذا منفصل عن خطاف Plugin العام
`tool_result_persist`، الذي يحوّل كتابات نتائج الأدوات في النص الذي يملكه
OpenClaw.

للاطلاع على دلالات خطافات Plugin نفسها، راجع [خطافات Plugin](/ar/plugins/hooks)
و[سلوك حارس Plugin](/ar/tools/plugin).

الحاضنة معطّلة افتراضيا. يجب أن تبقي الإعدادات الجديدة مراجع نماذج OpenAI
قياسية بصيغة `openai/gpt-*` وأن تفرض صراحة
`agentRuntime.id: "codex"` أو `OPENCLAW_AGENT_RUNTIME=codex` عندما تريد
تنفيذا أصليا عبر خادم التطبيق. لا تزال مراجع نماذج `codex/*` القديمة تختار
الحاضنة تلقائيا للتوافق، لكن بادئات المزود القديمة المدعومة بالتشغيل لا تظهر
كخيارات نموذج/مزود عادية.

إذا كان Plugin `codex` مفعّلا لكن النموذج الأساسي لا يزال
`openai-codex/*`، فإن `openclaw doctor` يصدر تحذيرا بدلا من تغيير المسار. هذا
متعمد: يبقى `openai-codex/*` مسار PI الخاص بـ Codex OAuth/الاشتراك، ويظل
تنفيذ خادم التطبيق الأصلي خيار تشغيل صريحا.

## خريطة المسارات

استخدم هذا الجدول قبل تغيير الإعداد:

| السلوك المطلوب                              | مرجع النموذج               | إعداد التشغيل                         | متطلب Plugin               | تسمية الحالة المتوقعة          |
| ------------------------------------------- | -------------------------- | -------------------------------------- | --------------------------- | ------------------------------ |
| OpenAI API عبر مشغّل OpenClaw العادي       | `openai/gpt-*`             | محذوف أو `runtime: "pi"`               | مزود OpenAI                 | `Runtime: OpenClaw Pi Default` |
| Codex OAuth/الاشتراك عبر PI                | `openai-codex/gpt-*`       | محذوف أو `runtime: "pi"`               | مزود OpenAI Codex OAuth     | `Runtime: OpenClaw Pi Default` |
| دورات مضمّنة أصلية لخادم تطبيق Codex       | `openai/gpt-*`             | `agentRuntime.id: "codex"`             | Plugin `codex`              | `Runtime: OpenAI Codex`        |
| مزودون مختلطون مع وضع تلقائي محافظ         | مراجع خاصة بالمزود         | `agentRuntime.id: "auto"`              | تشغيلات Plugin اختيارية     | يعتمد على التشغيل المحدد       |
| جلسة محوّل Codex ACP صريحة                 | يعتمد على مطالبة/نموذج ACP | `sessions_spawn` مع `runtime: "acp"`   | خلفية `acpx` سليمة          | حالة مهمة/جلسة ACP             |

الفصل المهم هو بين المزود والتشغيل:

- يجيب `openai-codex/*` عن "أي مسار مزود/مصادقة يجب أن يستخدمه PI؟"
- يجيب `agentRuntime.id: "codex"` عن "أي حلقة يجب أن تنفذ هذه
  الدورة المضمّنة؟"
- يجيب `/codex ...` عن "أي محادثة Codex أصلية يجب أن ترتبط بها هذه الدردشة
  أو تتحكم فيها؟"
- يجيب ACP عن "أي عملية حاضنة خارجية يجب أن يطلقها acpx؟"

## اختر بادئة النموذج الصحيحة

مسارات عائلة OpenAI خاصة بالبادئة. استخدم `openai-codex/*` عندما تريد
Codex OAuth عبر PI؛ واستخدم `openai/*` عندما تريد وصولا مباشرا إلى OpenAI API أو
عندما تفرض حاضنة خادم تطبيق Codex الأصلية:

| مرجع النموذج                                  | مسار التشغيل                                 | استخدمه عندما                                                             |
| --------------------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------- |
| `openai/gpt-5.4`                              | مزود OpenAI عبر سباكة OpenClaw/PI            | تريد وصولا مباشرا حاليا إلى OpenAI Platform API باستخدام `OPENAI_API_KEY`. |
| `openai-codex/gpt-5.5`                        | OpenAI Codex OAuth عبر OpenClaw/PI           | تريد مصادقة اشتراك ChatGPT/Codex مع مشغّل PI الافتراضي.                  |
| `openai/gpt-5.5` + `agentRuntime.id: "codex"` | حاضنة خادم تطبيق Codex                       | تريد تنفيذ خادم تطبيق Codex الأصلي لدورة الوكيل المضمّنة.                |

GPT-5.5 متاح حاليا في OpenClaw عبر الاشتراك/OAuth فقط. استخدم
`openai-codex/gpt-5.5` لـ PI OAuth، أو `openai/gpt-5.5` مع حاضنة خادم تطبيق
Codex. يكون الوصول المباشر بمفتاح API إلى `openai/gpt-5.5` مدعوما بمجرد أن
تفعّل OpenAI نموذج GPT-5.5 على API العام.

تبقى مراجع `codex/gpt-*` القديمة مقبولة كأسماء توافقية بديلة. تعيد ترحيلة
التوافق في Doctor كتابة مراجع التشغيل الأساسية القديمة إلى مراجع نماذج قياسية
وتسجل سياسة التشغيل بشكل منفصل، بينما تترك مراجع الإرث المخصصة للاحتياطي فقط
دون تغيير لأن التشغيل يضبط لحاوية الوكيل بالكامل. يجب أن تستخدم إعدادات PI
Codex OAuth الجديدة `openai-codex/gpt-*`؛ ويجب أن تستخدم إعدادات حاضنة خادم
التطبيق الأصلية الجديدة `openai/gpt-*` مع
`agentRuntime.id: "codex"`.

يتبع `agents.defaults.imageModel` الفصل نفسه للبادئات. استخدم
`openai-codex/gpt-*` عندما يجب تشغيل فهم الصور عبر مسار مزود OpenAI
Codex OAuth. استخدم `codex/gpt-*` عندما يجب تشغيل فهم الصور عبر دورة محدودة من
خادم تطبيق Codex. يجب أن يعلن نموذج خادم تطبيق Codex دعم إدخال الصور؛ تفشل
نماذج Codex النصية فقط قبل بدء دورة الوسائط.

استخدم `/status` لتأكيد الحاضنة الفعلية للجلسة الحالية. إذا كان الاختيار
مفاجئا، ففعّل تسجيل التصحيح للنظام الفرعي `agents/harness` وافحص سجل Gateway
المنظم `agent harness selected`. يتضمن معرف الحاضنة المحدد، وسبب الاختيار،
وسياسة التشغيل/الاحتياطي، وفي وضع `auto`، نتيجة دعم كل مرشح Plugin.

### ما معنى تحذيرات doctor

يحذر `openclaw doctor` عندما تكون كل هذه الشروط صحيحة:

- Plugin `codex` المضمّن مفعّل أو مسموح
- النموذج الأساسي لوكيل هو `openai-codex/*`
- التشغيل الفعلي لذلك الوكيل ليس `codex`

يوجد هذا التحذير لأن المستخدمين يتوقعون غالبا أن "تفعيل Plugin Codex" يعني
"تشغيل خادم تطبيق Codex الأصلي." لا يقوم OpenClaw بهذه القفزة. يعني التحذير:

- **لا يلزم أي تغيير** إذا كنت تقصد ChatGPT/Codex OAuth عبر PI.
- غيّر النموذج إلى `openai/<model>` واضبط
  `agentRuntime.id: "codex"` إذا كنت تقصد تنفيذ خادم التطبيق
  الأصلي.
- لا تزال الجلسات الحالية تحتاج إلى `/new` أو `/reset` بعد تغيير التشغيل،
  لأن تثبيتات تشغيل الجلسات لاصقة.

اختيار الحاضنة ليس تحكما مباشرا في جلسة حية. عندما تعمل دورة مضمّنة، يسجل
OpenClaw معرف الحاضنة المحدد على تلك الجلسة ويواصل استخدامه للدورات اللاحقة
في معرف الجلسة نفسه. غيّر إعداد `agentRuntime` أو
`OPENCLAW_AGENT_RUNTIME` عندما تريد أن تستخدم الجلسات المستقبلية حاضنة أخرى؛
واستخدم `/new` أو `/reset` لبدء جلسة جديدة قبل نقل محادثة حالية بين PI وCodex.
يتجنب هذا إعادة تشغيل نص واحد عبر نظامي جلسات أصليين غير متوافقين.

تعامل الجلسات القديمة التي أنشئت قبل تثبيتات الحاضنة كجلسات مثبتة على PI بمجرد
أن يكون لديها سجل نص. استخدم `/new` أو `/reset` لإدخال تلك المحادثة إلى Codex
بعد تغيير الإعداد.

يعرض `/status` تشغيل النموذج الفعلي. تظهر حاضنة PI الافتراضية باسم
`Runtime: OpenClaw Pi Default`، وتظهر حاضنة خادم تطبيق Codex باسم
`Runtime: OpenAI Codex`.

## المتطلبات

- OpenClaw مع توفر Plugin `codex` المضمّن.
- خادم تطبيق Codex الإصدار `0.125.0` أو أحدث. يدير Plugin المضمّن ثنائيا
  متوافقا لخادم تطبيق Codex افتراضيا، لذلك لا تؤثر أوامر `codex` المحلية على
  `PATH` في بدء تشغيل الحاضنة العادي.
- مصادقة Codex متاحة لعملية خادم التطبيق أو لجسر مصادقة Codex في OpenClaw.
  تستخدم عمليات إطلاق خادم التطبيق المحلي عبر stdio منزلا مدارًا من OpenClaw
  لـ Codex لكل وكيل و`HOME` فرعيا معزولا، لذلك لا تقرأ افتراضيا حسابك الشخصي
  `~/.codex`، أو skills، أو plugins، أو الإعداد، أو حالة السلاسل، أو
  `$HOME/.agents/skills` الأصلية.

يحظر Plugin المصافحات الأقدم أو غير المحددة الإصدار مع خادم التطبيق. يحافظ ذلك
على بقاء OpenClaw ضمن سطح البروتوكول الذي اختبر عليه.

بالنسبة لاختبارات الدخان الحية وDocker، تأتي المصادقة عادة من حساب Codex CLI
أو ملف تعريف مصادقة OpenClaw `openai-codex`. يمكن أيضا لعمليات إطلاق خادم
التطبيق المحلي عبر stdio الرجوع إلى `CODEX_API_KEY` / `OPENAI_API_KEY` عند عدم
وجود حساب.

## أضف Codex إلى جانب نماذج أخرى

لا تضبط `agentRuntime.id: "codex"` عموميًا إذا كان ينبغي للوكيل نفسه التبديل بحرية
بين Codex ونماذج مزودي غير Codex. ينطبق وقت تشغيل مفروض على كل
دور مضمّن لذلك الوكيل أو تلك الجلسة. إذا اخترت نموذج Anthropic بينما
وقت التشغيل ذلك مفروض، يظل OpenClaw يحاول استخدام حزمة Codex ويفشل بإغلاق
بدلًا من توجيه ذلك الدور بصمت عبر PI.

استخدم أحد هذه الأشكال بدلًا من ذلك:

- ضع Codex على وكيل مخصص مع `agentRuntime.id: "codex"`.
- أبق الوكيل الافتراضي على `agentRuntime.id: "auto"` واحتياطي PI للاستخدام العادي المختلط
  للمزوّدين.
- استخدم مراجع `codex/*` القديمة للتوافق فقط. ينبغي للتكوينات الجديدة تفضيل
  `openai/*` مع سياسة وقت تشغيل Codex صريحة.

على سبيل المثال، يبقي هذا الوكيل الافتراضي على الاختيار التلقائي العادي
ويضيف وكيل Codex منفصلًا:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      agentRuntime: {
        id: "auto",
        fallback: "pi",
      },
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
        agentRuntime: {
          id: "codex",
        },
      },
    ],
  },
}
```

مع هذا الشكل:

- يستخدم الوكيل الافتراضي `main` مسار المزوّد العادي واحتياطي توافق PI.
- يستخدم وكيل `codex` حزمة خادم تطبيق Codex.
- إذا كان Codex مفقودًا أو غير مدعوم لوكيل `codex`، يفشل الدور
  بدلًا من استخدام PI بهدوء.

## توجيه أوامر الوكيل

ينبغي للوكلاء توجيه طلبات المستخدم حسب القصد، وليس حسب كلمة "Codex" وحدها:

| عندما يطلب المستخدم...                                  | ينبغي للوكيل استخدام...                         |
| -------------------------------------------------------- | ------------------------------------------------ |
| "اربط هذه المحادثة بـ Codex"                             | `/codex bind`                                    |
| "استأنف سلسلة Codex `<id>` هنا"                         | `/codex resume <id>`                             |
| "اعرض سلاسل Codex"                                      | `/codex threads`                                 |
| "قدّم تقرير دعم عن تشغيل Codex سيئ"                     | `/diagnostics [note]`                            |
| "أرسل ملاحظات Codex فقط لهذه السلسلة المرفقة"            | `/codex diagnostics [note]`                      |
| "استخدم Codex كوقت تشغيل لهذا الوكيل"                   | تغيير تكوين إلى `agentRuntime.id`                |
| "استخدم اشتراكي في ChatGPT/Codex مع OpenClaw العادي"    | مراجع نماذج `openai-codex/*`                     |
| "شغّل Codex عبر ACP/acpx"                               | ACP `sessions_spawn({ runtime: "acp", ... })`    |
| "ابدأ Claude Code/Gemini/OpenCode/Cursor في سلسلة"      | ACP/acpx، وليس `/codex` وليس وكلاء فرعيين أصليين |

لا يعلن OpenClaw إرشادات إنشاء ACP للوكلاء إلا عندما يكون ACP مفعّلًا،
وقابلًا للإرسال، ومدعومًا بخلفية وقت تشغيل محمّلة. إذا لم يكن ACP متاحًا،
فينبغي ألا يعلّم موجّه النظام وSkills الخاصة بالـ Plugin الوكيل عن توجيه
ACP.

## عمليات نشر Codex فقط

افرض حزمة Codex عندما تحتاج إلى إثبات أن كل دور وكيل مضمّن
يستخدم Codex. أوقات تشغيل Plugin الصريحة لا تستخدم احتياطي PI افتراضيًا، لذلك
`fallback: "none"` اختياري لكنه مفيد غالبًا كتوضيح:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
        fallback: "none",
      },
    },
  },
}
```

تجاوز البيئة:

```bash
OPENCLAW_AGENT_RUNTIME=codex openclaw gateway run
```

مع فرض Codex، يفشل OpenClaw مبكرًا إذا كان Plugin الخاص بـ Codex معطّلًا، أو كان
خادم التطبيق قديمًا جدًا، أو تعذّر بدء خادم التطبيق. اضبط
`OPENCLAW_AGENT_HARNESS_FALLBACK=pi` فقط إذا كنت تريد عمدًا أن يتولى PI
اختيار الحزمة المفقود.

## Codex لكل وكيل

يمكنك جعل وكيل واحد مخصصًا لـ Codex فقط بينما يحتفظ الوكيل الافتراضي
بالاختيار التلقائي العادي:

```json5
{
  agents: {
    defaults: {
      agentRuntime: {
        id: "auto",
        fallback: "pi",
      },
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
        agentRuntime: {
          id: "codex",
          fallback: "none",
        },
      },
    ],
  },
}
```

استخدم أوامر الجلسة العادية للتبديل بين الوكلاء والنماذج. ينشئ `/new` جلسة
OpenClaw جديدة، وتنشئ حزمة Codex سلسلة خادم التطبيق الجانبية الخاصة بها أو تستأنفها
حسب الحاجة. يمسح `/reset` ربط جلسة OpenClaw لتلك السلسلة
ويسمح للدور التالي بحل الحزمة من التكوين الحالي مرة أخرى.

## اكتشاف النماذج

افتراضيًا، يطلب Plugin الخاص بـ Codex من خادم التطبيق النماذج المتاحة. إذا
فشل الاكتشاف أو انتهت مهلته، فإنه يستخدم كتالوجًا احتياطيًا مضمنًا لـ:

- GPT-5.5
- GPT-5.4 mini
- GPT-5.2

يمكنك ضبط الاكتشاف ضمن `plugins.entries.codex.config.discovery`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          discovery: {
            enabled: true,
            timeoutMs: 2500,
          },
        },
      },
    },
  },
}
```

عطّل الاكتشاف عندما تريد أن يتجنب بدء التشغيل فحص Codex وأن يلتزم
بالكتالوج الاحتياطي:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          discovery: {
            enabled: false,
          },
        },
      },
    },
  },
}
```

## اتصال خادم التطبيق والسياسة

افتراضيًا، يبدأ Plugin ثنائية Codex المُدارة من OpenClaw محليًا باستخدام:

```bash
codex app-server --listen stdio://
```

تُعلَن الثنائية المُدارة كاعتمادية وقت تشغيل Plugin مضمنة وتُحضّر
مع بقية اعتماديات Plugin `codex`. هذا يبقي إصدار خادم التطبيق
مرتبطًا بالـ Plugin المضمن بدلًا من أي Codex CLI منفصل قد يكون
مثبتًا محليًا. اضبط `appServer.command` فقط عندما تريد عمدًا
تشغيل ملف تنفيذي مختلف.

افتراضيًا، يبدأ OpenClaw جلسات حزمة Codex المحلية في وضع YOLO:
`approvalPolicy: "never"`، و`approvalsReviewer: "user"`، و
`sandbox: "danger-full-access"`. هذا هو وضع المشغّل المحلي الموثوق المستخدم
للـ Heartbeat المستقلة: يستطيع Codex استخدام أدوات الصدفة والشبكة دون
التوقف عند مطالبات الموافقة الأصلية التي لا يوجد أحد حولها للإجابة عنها.

للاشتراك في موافقات Codex التي يراجعها الحارس، اضبط `appServer.mode:
"guardian"`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            mode: "guardian",
            serviceTier: "fast",
          },
        },
      },
    },
  },
}
```

يستخدم وضع الحارس مسار الموافقة الأصلي بالمراجعة التلقائية في Codex. عندما يطلب Codex
مغادرة الصندوق الرملي، أو الكتابة خارج مساحة العمل، أو إضافة أذونات مثل الوصول
إلى الشبكة، يوجّه Codex طلب الموافقة ذلك إلى المراجع الأصلي بدلًا من
مطالبة بشرية. يطبّق المراجع إطار مخاطر Codex ويوافق على الطلب
المحدد أو يرفضه. استخدم Guardian عندما تريد حواجز حماية أكثر من وضع YOLO
لكن ما زلت تحتاج إلى وكلاء غير مراقبين يحرزون تقدمًا.

يتوسع الإعداد المسبق `guardian` إلى `approvalPolicy: "on-request"`،
و`approvalsReviewer: "auto_review"`، و`sandbox: "workspace-write"`.
ما زالت حقول السياسة الفردية تتجاوز `mode`، لذلك يمكن لعمليات النشر المتقدمة مزج
الإعداد المسبق مع اختيارات صريحة. ما تزال قيمة المراجع الأقدم `guardian_subagent`
مقبولة كاسم مستعار للتوافق، لكن ينبغي للتكوينات الجديدة استخدام
`auto_review`.

لخادم تطبيق قيد التشغيل بالفعل، استخدم نقل WebSocket:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://127.0.0.1:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
            requestTimeoutMs: 60000,
          },
        },
      },
    },
  },
}
```

ترث عمليات إطلاق خادم التطبيق عبر Stdio بيئة عملية OpenClaw افتراضيًا،
لكن OpenClaw يملك جسر حساب خادم تطبيق Codex ويضبط كلًا من
`CODEX_HOME` و`HOME` إلى أدلة لكل وكيل ضمن حالة OpenClaw الخاصة بذلك الوكيل.
يقرأ محمّل Skills الخاص بـ Codex نفسه `$CODEX_HOME/skills` و
`$HOME/.agents/skills`، لذلك تُعزل كلتا القيمتين لعمليات إطلاق خادم التطبيق
المحلية. هذا يبقي Skills الأصلية في Codex، والـ plugins، والتكوين، والحسابات، وحالة السلاسل
محصورة ضمن وكيل OpenClaw بدلًا من تسربها من منزل Codex CLI الشخصي
للمشغّل.

ما زالت Plugins الخاصة بـ OpenClaw ولقطات Skills الخاصة بـ OpenClaw تتدفق عبر
سجل Plugins ومحمّل Skills الخاصين بـ OpenClaw. لا تتدفق أصول Codex CLI الشخصية.
إذا كانت لديك Skills أو plugins مفيدة في Codex CLI ينبغي أن تصبح جزءًا من وكيل OpenClaw،
فاحصرها صراحة:

```bash
openclaw migrate codex --dry-run
openclaw migrate apply codex --yes
```

ينسخ مزوّد ترحيل Codex الـ Skills إلى مساحة عمل وكيل OpenClaw الحالي.
يتم الإبلاغ عن plugins الأصلية في Codex والخطافات وملفات التكوين أو أرشفتها
للمراجعة اليدوية بدلًا من تفعيلها تلقائيًا، لأنها قد
تنفّذ أوامر، أو تعرض خوادم MCP، أو تحمل بيانات اعتماد.

يُختار التوثيق بهذا الترتيب:

1. ملف تعريف توثيق OpenClaw Codex صريح للوكيل.
2. حساب خادم التطبيق الموجود في منزل Codex لذلك الوكيل.
3. لعمليات إطلاق خادم التطبيق المحلية عبر stdio فقط، `CODEX_API_KEY`، ثم
   `OPENAI_API_KEY`، عندما لا يكون حساب خادم تطبيق موجودًا وما يزال توثيق OpenAI
   مطلوبًا.

عندما يرى OpenClaw ملف تعريف توثيق Codex بنمط اشتراك ChatGPT، فإنه يزيل
`CODEX_API_KEY` و`OPENAI_API_KEY` من عملية Codex الفرعية المنشأة. هذا
يبقي مفاتيح API على مستوى Gateway متاحة للتضمينات أو نماذج OpenAI المباشرة
دون أن يجعل أدوار خادم تطبيق Codex الأصلية تُحتسب عبر API عن طريق الخطأ.
تستخدم ملفات تعريف مفاتيح API الصريحة لـ Codex واحتياطي مفاتيح البيئة المحلية عبر stdio
تسجيل الدخول إلى خادم التطبيق بدلًا من بيئة العملية الفرعية الموروثة. لا تتلقى اتصالات خادم التطبيق عبر WebSocket
احتياطي مفاتيح API من بيئة Gateway؛ استخدم ملف تعريف توثيق صريحًا أو
حساب خادم التطبيق البعيد نفسه.

إذا كانت عملية نشر تحتاج إلى عزل بيئة إضافي، فأضف تلك المتغيرات إلى
`appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

لا يؤثر `appServer.clearEnv` إلا في عملية خادم تطبيق Codex الفرعية المنشأة.

حقول `appServer` المدعومة:

| الحقل               | الافتراضي                                  | المعنى                                                                                                                                                                                                                              |
| ------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `transport`         | `"stdio"`                                | يشغّل `"stdio"` Codex؛ ويتصل `"websocket"` بـ `url`.                                                                                                                                                                             |
| `command`           | ثنائي Codex المُدار                     | الملف التنفيذي لنقل stdio. اتركه غير معيّن لاستخدام الثنائي المُدار؛ ولا تعيّنه إلا لتجاوز صريح.                                                                                                                         |
| `args`              | `["app-server", "--listen", "stdio://"]` | الوسائط لنقل stdio.                                                                                                                                                                                                       |
| `url`               | غير معيّن                                    | عنوان URL لخادم تطبيق WebSocket.                                                                                                                                                                                                            |
| `authToken`         | غير معيّن                                    | رمز Bearer لنقل WebSocket.                                                                                                                                                                                                |
| `headers`           | `{}`                                     | ترويسات WebSocket إضافية.                                                                                                                                                                                                             |
| `clearEnv`          | `[]`                                     | أسماء متغيرات بيئة إضافية تُزال من عملية خادم تطبيق stdio المُشغّلة بعد أن يبني OpenClaw بيئته الموروثة. `CODEX_HOME` و `HOME` محجوزان لعزل Codex لكل وكيل في OpenClaw عند عمليات التشغيل المحلية. |
| `requestTimeoutMs`  | `60000`                                  | مهلة استدعاءات مستوى التحكم لخادم التطبيق.                                                                                                                                                                                          |
| `mode`              | `"yolo"`                                 | إعداد مسبق لتنفيذ YOLO أو تنفيذ مُراجع عبر الحارس.                                                                                                                                                                                      |
| `approvalPolicy`    | `"never"`                                | سياسة موافقات Codex الأصلية المُرسلة إلى بدء/استئناف/دورة المحادثة.                                                                                                                                                                       |
| `sandbox`           | `"danger-full-access"`                   | وضع صندوق عزل Codex الأصلي المُرسل إلى بدء/استئناف دورة المحادثة.                                                                                                                                                                               |
| `approvalsReviewer` | `"user"`                                 | استخدم `"auto_review"` للسماح لـ Codex بمراجعة مطالبات الموافقة الأصلية. يظل `guardian_subagent` اسمًا مستعارًا قديمًا.                                                                                                                         |
| `serviceTier`       | غير معيّن                                    | طبقة خدمة خادم تطبيق Codex اختيارية: `"fast"` أو `"flex"` أو `null`. تُتجاهل القيم القديمة غير الصالحة.                                                                                                                            |

استدعاءات الأدوات الديناميكية المملوكة لـ OpenClaw محدودة بشكل مستقل عن
`appServer.requestTimeoutMs`: يجب أن يتلقى كل طلب Codex من نوع `item/tool/call`
استجابة من OpenClaw خلال 30 ثانية. عند انتهاء المهلة، يجهض OpenClaw إشارة
الأداة حيثما كان ذلك مدعومًا ويعيد استجابة أداة ديناميكية فاشلة إلى Codex كي
تستمر دورة المحادثة بدلًا من ترك الجلسة في حالة `processing`.

بعد أن يستجيب OpenClaw لطلب خادم تطبيق خاص بنطاق دورة محادثة Codex، يتوقع
الحزام أيضًا أن ينهي Codex دورة المحادثة الأصلية بـ `turn/completed`. إذا أصبح
خادم التطبيق صامتًا لمدة 60 ثانية بعد تلك الاستجابة، يحاول OpenClaw بأفضل جهد
مقاطعة دورة محادثة Codex، ويسجل مهلة تشخيصية، ويحرر مسار جلسة OpenClaw بحيث
لا تُصف رسائل الدردشة اللاحقة خلف دورة محادثة أصلية قديمة.

تظل تجاوزات البيئة متاحة للاختبار المحلي:

- `OPENCLAW_CODEX_APP_SERVER_BIN`
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

يتجاوز `OPENCLAW_CODEX_APP_SERVER_BIN` الثنائي المُدار عندما يكون
`appServer.command` غير معيّن.

أُزيل `OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1`. استخدم
`plugins.entries.codex.config.appServer.mode: "guardian"` بدلًا منه، أو
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` للاختبار المحلي لمرة واحدة. يُفضَّل
التكوين لعمليات النشر القابلة للتكرار لأنه يُبقي سلوك Plugin في الملف نفسه
المُراجع كبقية إعداد حزام Codex.

## استخدام الحاسوب

تتم تغطية استخدام الحاسوب في دليل إعداد مستقل:
[استخدام الحاسوب في Codex](/ar/plugins/codex-computer-use).

الخلاصة: لا يضمّن OpenClaw تطبيق التحكم بسطح المكتب ولا ينفّذ إجراءات سطح
المكتب بنفسه. إنه يجهّز خادم تطبيق Codex، ويتحقق من توفر خادم MCP الخاص بـ
`computer-use`، ثم يترك Codex يتعامل مع استدعاءات أدوات MCP الأصلية أثناء دورات
وضع Codex.

للوصول المباشر إلى برنامج تشغيل TryCua خارج مسار سوق Codex، سجّل
`cua-driver mcp` باستخدام `openclaw mcp set cua-driver '{"command":"cua-driver","args":["mcp"]}'`.
راجع [استخدام الحاسوب في Codex](/ar/plugins/codex-computer-use) للاطلاع على الفرق
بين استخدام الحاسوب المملوك لـ Codex وتسجيل MCP المباشر.

الحد الأدنى للتكوين:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          computerUse: {
            autoInstall: true,
          },
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
        fallback: "none",
      },
    },
  },
}
```

يمكن فحص الإعداد أو تثبيته من واجهة الأوامر:

- `/codex computer-use status`
- `/codex computer-use install`
- `/codex computer-use install --source <marketplace-source>`
- `/codex computer-use install --marketplace-path <path>`

استخدام الحاسوب خاص بنظام macOS وقد يتطلب أذونات محلية من نظام التشغيل قبل أن
يتمكن خادم Codex MCP من التحكم في التطبيقات. إذا كان `computerUse.enabled` هو
true وكان خادم MCP غير متاح، تفشل دورات وضع Codex قبل بدء المحادثة بدلًا من
العمل بصمت دون أدوات استخدام الحاسوب الأصلية. راجع
[استخدام الحاسوب في Codex](/ar/plugins/codex-computer-use) للاطلاع على خيارات
السوق، وحدود الكتالوج البعيد، وأسباب الحالة، واستكشاف الأخطاء وإصلاحها.

عندما يكون `computerUse.autoInstall` هو true، يمكن لـ OpenClaw تسجيل سوق Codex
Desktop القياسي المضمّن من
`/Applications/Codex.app/Contents/Resources/plugins/openai-bundled` إذا لم يكن
Codex قد اكتشف سوقًا محليًا بعد. استخدم `/new` أو `/reset` بعد تغيير تكوين
وقت التشغيل أو استخدام الحاسوب حتى لا تحتفظ الجلسات الحالية بارتباط PI أو
محادثة Codex قديم.

## وصفات شائعة

Codex محلي مع نقل stdio الافتراضي:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

التحقق من حزام Codex فقط:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

موافقات Codex المُراجعة عبر الحارس:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            mode: "guardian",
            approvalPolicy: "on-request",
            approvalsReviewer: "auto_review",
            sandbox: "workspace-write",
          },
        },
      },
    },
  },
}
```

خادم تطبيق بعيد مع ترويسات صريحة:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            headers: {
              "X-OpenClaw-Agent": "main",
            },
          },
        },
      },
    },
  },
}
```

يبقى تبديل النماذج تحت تحكم OpenClaw. عندما تُرفق جلسة OpenClaw بمحادثة Codex
حالية، ترسل دورة المحادثة التالية نموذج OpenAI المحدد حاليًا، والمزوّد، وسياسة
الموافقة، وصندوق العزل، وطبقة الخدمة إلى خادم التطبيق مرة أخرى. التبديل من
`openai/gpt-5.5` إلى `openai/gpt-5.2` يُبقي ارتباط المحادثة لكنه يطلب من Codex
المتابعة بالنموذج المحدد حديثًا.

## أمر Codex

يسجل Plugin المضمّن `/codex` كأمر شرطة مائلة مصرّح به. إنه عام ويعمل على أي
قناة تدعم أوامر OpenClaw النصية.

الصيغ الشائعة:

- يعرض `/codex status` اتصال خادم التطبيق المباشر، والنماذج، والحساب، وحدود المعدل، وخوادم MCP، وSkills.
- يسرد `/codex models` نماذج خادم تطبيق Codex المباشرة.
- يسرد `/codex threads [filter]` محادثات Codex الحديثة.
- يرفق `/codex resume <thread-id>` جلسة OpenClaw الحالية بمحادثة Codex موجودة.
- يطلب `/codex compact` من خادم تطبيق Codex ضغط المحادثة المرفقة.
- يبدأ `/codex review` مراجعة Codex الأصلية للمحادثة المرفقة.
- يطلب `/codex diagnostics [note]` قبل إرسال ملاحظات تشخيص Codex للمحادثة المرفقة.
- يتحقق `/codex computer-use status` من Plugin استخدام الحاسوب وخادم MCP المكوّنين.
- يثبّت `/codex computer-use install` Plugin استخدام الحاسوب المكوّن ويعيد تحميل خوادم MCP.
- يعرض `/codex account` حالة الحساب وحدود المعدل.
- يسرد `/codex mcp` حالة خادم MCP لخادم تطبيق Codex.
- يسرد `/codex skills` Skills خادم تطبيق Codex.

### سير عمل شائع للتصحيح

عندما يفعل وكيل مدعوم بـ Codex شيئًا مفاجئًا في Telegram أو Discord أو Slack
أو قناة أخرى، ابدأ بالمحادثة التي حدثت فيها المشكلة:

1. شغّل `/diagnostics bad tool choice after image upload` أو ملاحظة قصيرة أخرى
   تصف ما رأيته.
2. وافق على طلب التشخيصات مرة واحدة. تنشئ الموافقة ملف تشخيصات Gateway المضغوط
   المحلي، وبما أن الجلسة تستخدم حزام Codex، فإنها ترسل أيضًا حزمة ملاحظات
   Codex ذات الصلة إلى خوادم OpenAI.
3. انسخ رد التشخيصات المكتمل إلى تقرير الخطأ أو محادثة الدعم.
   يتضمن مسار الحزمة المحلي، وملخص الخصوصية، ومعرفات جلسات OpenClaw،
   ومعرفات محادثات Codex، وسطر `Inspect locally` لكل محادثة Codex.
4. إذا أردت تصحيح التشغيل بنفسك، فشغّل أمر `Inspect locally` المطبوع
   في طرفية. يبدو مثل `codex resume <thread-id>` ويفتح محادثة Codex الأصلية
   حتى تتمكن من فحص المحادثة، أو متابعتها محليًا، أو سؤال Codex عن سبب اختياره
   لأداة أو خطة معينة.

استخدم `/codex diagnostics [note]` فقط عندما تريد تحديدًا رفع ملاحظات Codex للخيط المرفق حاليًا بدون حزمة تشخيصات OpenClaw Gateway الكاملة. في معظم تقارير الدعم، يكون `/diagnostics [note]` نقطة البداية الأفضل لأنه يربط حالة Gateway المحلية ومعرّفات خيوط Codex معًا في رد واحد. راجع [تصدير التشخيصات](/ar/gateway/diagnostics) للاطلاع على نموذج الخصوصية الكامل وسلوك الدردشة الجماعية.

يوفّر OpenClaw الأساسي أيضًا الأمر `/diagnostics [note]` المخصص للمالكين فقط بوصفه أمر تشخيصات Gateway العام. تعرض مطالبة الموافقة الخاصة به تمهيد البيانات الحساسة، وتربط إلى [تصدير التشخيصات](/ar/gateway/diagnostics)، وتطلب `openclaw gateway diagnostics export --json` عبر موافقة تنفيذ صريحة في كل مرة. لا توافق على التشخيصات بقاعدة سماح شاملة. بعد الموافقة، يرسل OpenClaw تقريرًا قابلًا للصق يتضمن مسار الحزمة المحلي وملخص البيان. عندما تستخدم جلسة OpenClaw النشطة حزام Codex، فإن الموافقة نفسها تخوّل أيضًا إرسال حزم ملاحظات Codex ذات الصلة إلى خوادم OpenAI. تنص مطالبة الموافقة على أنه سيتم إرسال ملاحظات Codex، لكنها لا تسرد معرّفات جلسات أو خيوط Codex قبل الموافقة.

إذا استدعى مالك `/diagnostics` في دردشة جماعية، يحافظ OpenClaw على نظافة القناة المشتركة: تتلقى المجموعة إشعارًا قصيرًا فقط، بينما تُرسل تمهيدات التشخيصات ومطالبات الموافقة ومعرّفات جلسة/خيط Codex إلى المالك عبر مسار الموافقة الخاص. إذا لم يكن هناك مسار خاص للمالك، يرفض OpenClaw طلب المجموعة ويطلب من المالك تشغيله من رسالة مباشرة.

يستدعي رفع Codex الموافق عليه نقطة `feedback/upload` في app-server الخاص بـ Codex، ويطلب من app-server تضمين السجلات لكل خيط مدرج وخيوط Codex الفرعية المنشأة عند توفرها. يمر الرفع عبر مسار الملاحظات العادي في Codex إلى خوادم OpenAI؛ إذا كانت ملاحظات Codex معطلة في ذلك app-server، يعيد الأمر خطأ app-server. يسرد رد التشخيصات المكتمل القنوات، ومعرّفات جلسات OpenClaw، ومعرّفات خيوط Codex، وأوامر `codex resume <thread-id>` المحلية للخيوط التي أُرسلت. إذا رفضت الموافقة أو تجاهلتها، لا يطبع OpenClaw معرّفات Codex تلك. لا يحل هذا الرفع محل تصدير تشخيصات Gateway المحلي.

يكتب `/codex resume` ملف ربط sidecar نفسه الذي يستخدمه الحزام للمنعطفات العادية. في الرسالة التالية، يستأنف OpenClaw خيط Codex ذلك، ويمرر نموذج OpenClaw المحدد حاليًا إلى app-server، ويحافظ على تفعيل السجل الموسع.

### فحص خيط Codex من CLI

غالبًا ما تكون أسرع طريقة لفهم تشغيل Codex سيئ هي فتح خيط Codex الأصلي مباشرة:

```sh
codex resume <thread-id>
```

استخدم هذا عندما تلاحظ خطأ في محادثة قناة وتريد فحص جلسة Codex الإشكالية، أو متابعتها محليًا، أو سؤال Codex عن سبب اختياره أداة أو مسار تفكير معين. عادة ما يكون أسهل مسار هو تشغيل `/diagnostics [note]` أولًا: بعد أن توافق عليه، يسرد التقرير المكتمل كل خيط Codex ويطبع أمر `Inspect locally`، مثل `codex resume <thread-id>`. يمكنك نسخ ذلك الأمر مباشرة إلى الطرفية.

يمكنك أيضًا الحصول على معرّف خيط من `/codex binding` للدردشة الحالية أو `/codex threads [filter]` لخيوط Codex app-server الأخيرة، ثم تشغيل أمر `codex resume` نفسه في الصدفة لديك.

يتطلب سطح الأوامر Codex app-server إصدار `0.125.0` أو أحدث. تُبلّغ طرق التحكم الفردية بأنها `unsupported by this Codex app-server` إذا كان app-server مستقبلي أو مخصص لا يوفّر طريقة JSON-RPC تلك.

## حدود الخطافات

يحتوي حزام Codex على ثلاث طبقات خطافات:

| الطبقة                                 | المالك                    | الغرض                                                             |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| خطافات Plugin في OpenClaw                 | OpenClaw                 | توافق المنتج/Plugin عبر أحزمة PI وCodex.         |
| وسيط ملحق Codex app-server | Plugins المضمنة في OpenClaw | سلوك المهايئ لكل منعطف حول أدوات OpenClaw الديناميكية.            |
| خطافات Codex الأصلية                    | Codex                    | دورة حياة Codex منخفضة المستوى وسياسة الأدوات الأصلية من إعدادات Codex. |

لا يستخدم OpenClaw ملفات `hooks.json` الخاصة بالمشروع أو العامة في Codex لتوجيه سلوك Plugin في OpenClaw. بالنسبة إلى جسر الأدوات الأصلية والأذونات المدعوم، يحقن OpenClaw إعدادات Codex لكل خيط من أجل `PreToolUse` و`PostToolUse` و`PermissionRequest` و`Stop`. تبقى خطافات Codex الأخرى مثل `SessionStart` و`UserPromptSubmit` عناصر تحكم على مستوى Codex؛ ولا تُعرض بوصفها خطافات Plugin في OpenClaw ضمن عقد v1.

بالنسبة إلى أدوات OpenClaw الديناميكية، ينفذ OpenClaw الأداة بعد أن يطلب Codex الاستدعاء، لذلك يطلق OpenClaw سلوك Plugin والوسيط الذي يملكه في مهايئ الحزام. بالنسبة إلى أدوات Codex الأصلية، يملك Codex سجل الأداة المعتمد. يستطيع OpenClaw عكس أحداث محددة، لكنه لا يستطيع إعادة كتابة خيط Codex الأصلي ما لم يوفّر Codex تلك العملية عبر app-server أو ردود نداء الخطافات الأصلية.

تأتي إسقاطات دورة حياة Compaction وLLM من إشعارات Codex app-server وحالة مهايئ OpenClaw، وليس من أوامر خطافات Codex الأصلية. أحداث `before_compaction` و`after_compaction` و`llm_input` و`llm_output` في OpenClaw هي ملاحظات على مستوى المهايئ، وليست لقطات مطابقة بايتًا ببايت لطلب Codex الداخلي أو حمولات Compaction.

تُسقط إشعارات Codex الأصلية `hook/started` و`hook/completed` في app-server بوصفها أحداث وكيل `codex_app_server.hook` للتتبع والتصحيح. وهي لا تستدعي خطافات Plugin في OpenClaw.

## الأدوات والوسائط وCompaction

يغير مسخّر Codex منفّذ الوكيل المضمّن منخفض المستوى فقط.

لا يزال OpenClaw يبني قائمة الأدوات ويتلقى نتائج الأدوات الديناميكية من
المسخّر. يستمر النص والصور والفيديو والموسيقى وTTS والموافقات ومخرجات أدوات
المراسلة عبر مسار التسليم المعتاد في OpenClaw.

مرحّل الخطاف الأصلي عام عن قصد، لكن عقد دعم v1
محدود بمسارات الأدوات والأذونات الأصلية في Codex التي يختبرها OpenClaw. في
وقت تشغيل Codex، يشمل ذلك حمولات الصدفة والتصحيح وMCP `PreToolUse`،
و`PostToolUse`، و`PermissionRequest`. لا تفترض أن كل حدث خطاف مستقبلي في
Codex هو سطح Plugin في OpenClaw حتى يسميه عقد وقت التشغيل.

بالنسبة إلى `PermissionRequest`، لا يعيد OpenClaw إلا قرارات السماح أو الرفض
الصريحة عندما تقرر السياسة. نتيجة عدم وجود قرار ليست سماحًا. يعاملها Codex
على أنها عدم وجود قرار خطاف ويمررها إلى مسار الحارس الخاص به أو مسار موافقة
المستخدم.

تُوجّه استدعاءات طلب الموافقة على أدوات MCP في Codex عبر تدفق موافقة
plugins في OpenClaw عندما يضع Codex علامة `_meta.codex_approval_kind` كـ
`"mcp_tool_call"`. تُرسل مطالبات Codex `request_user_input` مرة أخرى إلى
المحادثة الأصلية، وتجيب رسالة المتابعة التالية في قائمة الانتظار على طلب
الخادم الأصلي هذا بدلًا من توجيهها كسياق إضافي. لا تزال طلبات استدعاء MCP
الأخرى تفشل بإغلاق آمن.

يرتبط توجيه قائمة انتظار التشغيل النشط بـ `turn/steer` في خادم تطبيق Codex. مع
الإعداد الافتراضي `messages.queue.mode: "steer"`، يجمع OpenClaw رسائل الدردشة
الموضوعة في قائمة الانتظار خلال نافذة السكون المهيأة ويرسلها كطلب `turn/steer`
واحد بترتيب الوصول. يرسل وضع `queue` القديم طلبات `turn/steer` منفصلة. يمكن
لأدوار مراجعة Codex وCompaction اليدوي رفض التوجيه في الدور نفسه، وفي هذه
الحالة يستخدم OpenClaw قائمة انتظار المتابعة عندما يسمح الوضع المحدد بالرجوع. راجع
[قائمة توجيه الانتظار](/ar/concepts/queue-steering).

عندما يستخدم النموذج المحدد مسخّر Codex، يُفوّض Compaction للسلسلة الأصلية
إلى خادم تطبيق Codex. يحتفظ OpenClaw بمرآة للسجل النصي من أجل سجل القناة،
والبحث، و`/new`، و`/reset`، وتبديل النموذج أو المسخّر مستقبلًا. تتضمن المرآة
مطالبة المستخدم، والنص النهائي للمساعد، وسجلات خفيفة لتفكير Codex أو خطته
عندما يصدرها خادم التطبيق. حاليًا، لا يسجل OpenClaw إلا إشارات بدء Compaction
الأصلي واكتماله. ولا يعرض بعد ملخص Compaction قابلًا للقراءة البشرية أو قائمة
قابلة للتدقيق بالإدخالات التي أبقاها Codex بعد Compaction.

لأن Codex يملك السلسلة الأصلية المرجعية، فإن `tool_result_persist` لا يعيد
حاليًا كتابة سجلات نتائج الأدوات الأصلية في Codex. ينطبق فقط عندما يكتب
OpenClaw نتيجة أداة في سجل نصي لجلسة يملكها OpenClaw.

لا يتطلب إنشاء الوسائط PI. تستمر الصور والفيديو والموسيقى وPDF وTTS وفهم
الوسائط في استخدام إعدادات المزود/النموذج المطابقة مثل
`agents.defaults.imageGenerationModel`، و`videoGenerationModel`، و`pdfModel`، و
`messages.tts`.

## استكشاف الأخطاء وإصلاحها

**لا يظهر Codex كمزود `/model` عادي:** هذا متوقع للتكوينات
الجديدة. حدد نموذج `openai/gpt-*` مع
`agentRuntime.id: "codex"` (أو مرجع `codex/*` قديمًا)، ومكّن
`plugins.entries.codex.enabled`، وتحقق مما إذا كان `plugins.allow` يستبعد
`codex`.

**يستخدم OpenClaw PI بدلًا من Codex:** لا يزال بإمكان `agentRuntime.id: "auto"` استخدام PI كخلفية
توافق عندما لا يطالب أي مسخّر Codex بالتشغيل. اضبط
`agentRuntime.id: "codex"` لفرض اختيار Codex أثناء الاختبار. وقت تشغيل Codex
المفروض يفشل الآن بدلًا من الرجوع إلى PI ما لم تضبط صراحة
`agentRuntime.fallback: "pi"`. بمجرد تحديد خادم تطبيق Codex، تظهر إخفاقاته
مباشرة من دون تكوين رجوع إضافي.

**يُرفض خادم التطبيق:** حدّث Codex بحيث تفيد مصافحة خادم التطبيق
بالإصدار `0.125.0` أو أحدث. تُرفض إصدارات ما قبل الإصدار المطابقة في الإصدار
أو الإصدارات ذات لاحقة البناء مثل `0.125.0-alpha.2` أو `0.125.0+custom` لأن
حد بروتوكول `0.125.0` المستقر هو ما يختبره OpenClaw.

**اكتشاف النماذج بطيء:** خفّض `plugins.entries.codex.config.discovery.timeoutMs`
أو عطّل الاكتشاف.

**يفشل نقل WebSocket فورًا:** تحقق من `appServer.url`، و`authToken`،
ومن أن خادم التطبيق البعيد يتحدث إصدار بروتوكول خادم تطبيق Codex نفسه.

**يستخدم نموذج غير Codex PI:** هذا متوقع ما لم تفرض
`agentRuntime.id: "codex"` لذلك الوكيل أو تحدد مرجع
`codex/*` قديمًا. تبقى مراجع `openai/gpt-*` العادية ومراجع المزودين الآخرين على مسار
المزود العادي في وضع `auto`. إذا فرضت `agentRuntime.id: "codex"`، فيجب أن يكون كل دور
مضمّن لذلك الوكيل نموذج OpenAI مدعومًا من Codex.

**Computer Use مثبت لكن الأدوات لا تعمل:** تحقق من
`/codex computer-use status` من جلسة جديدة. إذا أبلغت أداة عن
`Native hook relay unavailable`، فاستخدم `/new` أو `/reset`؛ وإذا استمر ذلك، فأعد تشغيل
Gateway لمسح تسجيلات الخطافات الأصلية القديمة. إذا انتهت مهلة `computer-use.list_apps`،
فأعد تشغيل Codex Computer Use أو Codex Desktop ثم أعد المحاولة.
