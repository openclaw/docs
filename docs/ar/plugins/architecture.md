## نموذج القدرات العام

القدرات هي نموذج **Plugins الأصلية** العام داخل OpenClaw. يسجّل كل
Plugin أصلية في OpenClaw نفسها مقابل نوع قدرة واحد أو أكثر:

| القدرة                | طريقة التسجيل                                  | أمثلة على Plugins                  |
| --------------------- | ---------------------------------------------- | ---------------------------------- |
| الاستدلال النصي       | `api.registerProvider(...)`                    | `openai`، `anthropic`              |
| واجهة CLI خلفية للاستدلال | `api.registerCliBackend(...)`               | `openai`، `anthropic`              |
| الكلام                | `api.registerSpeechProvider(...)`              | `elevenlabs`، `microsoft`          |
| النسخ الفوري          | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                         |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`       | `openai`                           |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)`  | `openai`، `google`                 |
| توليد الصور           | `api.registerImageGenerationProvider(...)`     | `openai`، `google`، `fal`، `minimax` |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`     | `google`، `minimax`                |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`     | `qwen`                             |
| جلب الويب             | `api.registerWebFetchProvider(...)`            | `firecrawl`                        |
| البحث على الويب       | `api.registerWebSearchProvider(...)`           | `google`                           |
| القناة / المراسلة     | `api.registerChannel(...)`                     | `msteams`، `matrix`                |
| اكتشاف Gateway        | `api.registerGatewayDiscoveryService(...)`     | `bonjour`                          |

تُعد Plugin التي تسجل صفر قدرات ولكنها توفّر hooks أو tools أو discovery
services أو background services بمثابة Plugin **قديمة من نوع hook-only**. ولا يزال
هذا النمط مدعومًا بالكامل.

### موقف التوافق الخارجي

لقد استقر نموذج القدرات في core ويُستخدم من قبل Plugins الأصلية/المجمعة
اليوم، لكن توافق Plugins الخارجية لا يزال يحتاج إلى معيار أكثر صرامة من
"إنه مُصدَّر، إذًا فهو ثابت".

| حالة Plugin                                  | التوجيه                                                                                           |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Plugins الخارجية الموجودة                    | حافظ على عمل عمليات التكامل القائمة على hooks؛ فهذا هو خط الأساس للتوافق.                         |
| Plugins الأصلية/المجمعة الجديدة              | فضّل تسجيل القدرات الصريح بدلًا من reach-ins الخاصة بالمورّد أو تصميمات hook-only الجديدة.      |
| Plugins الخارجية التي تتبنى تسجيل القدرات    | مسموح، لكن تعامل مع الأسطح المساعدة الخاصة بالقدرات على أنها متطورة ما لم توثّقها المستندات على أنها مستقرة. |

يُعد تسجيل القدرات هو الاتجاه المقصود. وتظل hooks القديمة هي
المسار الأكثر أمانًا من دون كسر بالنسبة إلى Plugins الخارجية خلال هذا الانتقال.
وليست كل المسارات الفرعية المساعدة المصدّرة متساوية — ففضّل العقود الضيقة الموثقة على صادرات المساعدة العرضية.

### أشكال Plugins

يصنّف OpenClaw كل Plugin محمّلة إلى شكل يعتمد على سلوك
التسجيل الفعلي (وليس فقط البيانات الوصفية الثابتة):

- **plain-capability**: تسجّل نوع قدرة واحدًا بالضبط (على سبيل المثال
  Plugin خاصة بموفّر فقط مثل `mistral`).
- **hybrid-capability**: تسجّل أنواع قدرات متعددة (على سبيل المثال
  تمتلك `openai` الاستدلال النصي، والكلام، وفهم الوسائط، وتوليد الصور).
- **hook-only**: تسجّل hooks فقط (typed أو مخصصة)، من دون قدرات،
  أو tools، أو commands، أو services.
- **non-capability**: تسجّل tools، أو commands، أو services، أو routes، ولكن من دون
  قدرات.

استخدم `openclaw plugins inspect <id>` لمعرفة شكل Plugin وتفصيل
قدراتها. راجع [مرجع CLI](/ar/cli/plugins#inspect) للتفاصيل.

### Legacy hooks

لا يزال hook ‏`before_agent_start` مدعومًا كمسار توافق بالنسبة إلى
Plugins من نوع hook-only. ولا تزال Plugins القديمة الواقعية تعتمد عليه.

الاتجاه:

- حافظ على عمله
- وثّقه على أنه قديم
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/الموفّر
- فضّل `before_prompt_build` لأعمال تعديل الموجّه
- أزِله فقط بعد انخفاض الاستخدام الحقيقي وإثبات تغطية fixtures لأمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه العلامات:

| الإشارة                  | المعنى                                                      |
| ------------------------ | ----------------------------------------------------------- |
| **config valid**         | يتم تحليل الإعدادات بشكل جيد ويتم حل Plugins                |
| **compatibility advisory** | تستخدم Plugin نمطًا مدعومًا لكن أقدم (مثل `hook-only`)    |
| **legacy warning**       | تستخدم Plugin القيمة `before_agent_start`، وهي مهملة        |
| **hard error**           | الإعدادات غير صالحة أو فشل تحميل Plugin                     |

لن تؤدي `hook-only` ولا `before_agent_start` إلى كسر Plugin لديك اليوم:
فـ `hook-only` استشارية، و`before_agent_start` لا تؤدي إلا إلى تحذير. كما
تظهر هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يمتلك نظام Plugins في OpenClaw أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على Plugins المرشحة من المسارات المهيأة، وجذور مساحات العمل،
   وجذور Plugins العالمية، وPlugins المجمعة. ويقرأ الاكتشاف
   بيانات `openclaw.plugin.json` الأصلية بالإضافة إلى بيانات الحزم المدعومة أولًا.
2. **التمكين + التحقق**
   تحدد core ما إذا كانت Plugin المكتشفة ممكّنة، أو معطّلة، أو محظورة، أو
   محددة لفتحة حصرية مثل الذاكرة.
3. **التحميل وقت التشغيل**
   تُحمَّل Plugins الأصلية لـ OpenClaw داخل العملية عبر jiti وتسجل
   القدرات داخل سجل مركزي. وتُطبَّع الحزم المتوافقة إلى
   سجلات في السجل من دون استيراد كود وقت التشغيل.
4. **استهلاك الأسطح**
   يقرأ باقي OpenClaw السجل لكشف tools، والقنوات، وإعداد
   الموفّر، وhooks، وHTTP routes، وCLI commands، وservices.

أما بالنسبة إلى CLI الخاصة بـ Plugin تحديدًا، فينقسم اكتشاف الأوامر الجذرية إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- ويمكن أن تبقى وحدة CLI الحقيقية الخاصة بـ Plugin كسولة وتُسجَّل عند أول استدعاء

وهذا يبقي كود CLI المملوك لـ Plugin داخل Plugin نفسها مع تمكين OpenClaw
من حجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعدادات من خلال **البيانات الوصفية للبيان/المخطط**
  من دون تنفيذ كود Plugin
- ويأتي السلوك الأصلي في وقت التشغيل من مسار `register(api)` الخاص بوحدة Plugin

يتيح هذا الفصل لـ OpenClaw التحقق من الإعدادات، وشرح Plugins
المفقودة/المعطلة، وبناء تلميحات UI/المخطط قبل تفعيل وقت التشغيل الكامل.

### تخطيط التفعيل

يُعد تخطيط التفعيل جزءًا من control plane. ويمكن للمتصلين سؤال النظام عن Plugins
ذات الصلة بأمر أو موفّر أو قناة أو route أو agent harness أو
قدرة محددة قبل تحميل سجلات وقت تشغيل أوسع.

يحافظ المخطِّط على توافق سلوك البيان الحالي:

- حقول `activation.*` هي تلميحات صريحة للمخطِّط
- تظل `providers` و`channels` و`commandAliases` و`setup.providers` و
  `contracts.tools` وhooks ملكية احتياطية للبيان
- تظل واجهة API الخاصة بالمخطِّط التي تعتمد على المعرفات فقط متاحة للمتصلين الحاليين
- تبلغ واجهة plan API عن تسميات الأسباب بحيث يمكن للتشخيص التفريق بين التلميحات الصريحة والملكية الاحتياطية

لا تتعامل مع `activation` على أنها hook لدورة الحياة أو بديل عن
`register(...)`. إنها بيانات وصفية تُستخدم لتضييق التحميل. وفضّل حقول الملكية
عندما تصف العلاقة بالفعل؛ واستخدم `activation` فقط لتلميحات مخطِّط إضافية.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة منفصلة للإرسال/التحرير/التفاعل من أجل
إجراءات الدردشة العادية. إذ يحتفظ OpenClaw بأداة `message` مشتركة واحدة في core،
وتمتلك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- تمتلك core مضيف أداة `message` المشتركة، وأسلاك الموجّه، وإدارة
  الجلسة/الخيط، وتوجيه التنفيذ
- وتمتلك Plugins القنوات اكتشاف الإجراءات ذات النطاق، واكتشاف القدرات، وأي أجزاء
  مخطط خاصة بالقناة
- وتمتلك Plugins القنوات قواعد محادثة الجلسات الخاصة بالموفّر، مثل
  كيفية ترميز معرّفات المحادثات لمعرّفات الخيوط أو وراثتها من المحادثات الأصلية
- وتنفّذ Plugins القنوات الإجراء النهائي عبر action adapter الخاصة بها

بالنسبة إلى Plugins القنوات، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يسمح استدعاء الاكتشاف الموحد
هذا لـ Plugin بإرجاع إجراءاتها المرئية، وقدراتها، ومساهمات المخطط
معًا بحيث لا تنفصل هذه الأجزاء عن بعضها.

عندما تحمل message-tool الخاصة بالقناة معلمة خاصة بمصدر وسائط مثل
مسار محلي أو URL وسائط بعيد، ينبغي على Plugin أيضًا أن تعيد
`mediaSourceParams` من `describeMessageTool(...)`. وتستخدم core هذه
القائمة الصريحة لتطبيق تطبيع مسارات sandbox وتلميحات الوصول إلى الوسائط الصادرة
من دون ترميز صريح لأسماء المعلمات المملوكة لـ Plugin.
وفضّل الخرائط ذات النطاق الإجرائي هناك، وليس قائمة مسطحة واحدة على مستوى القناة،
بحيث لا يتم تطبيع معلمة وسائط خاصة بملف تعريف على إجراءات غير ذات صلة مثل
`send`.

تمرر core نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتتضمن الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم بالنسبة إلى Plugins الحساسة للسياق. إذ يمكن للقناة إخفاء أو كشف
إجراءات الرسائل استنادًا إلى الحساب النشط، أو الغرفة/الخيط/الرسالة الحالية، أو
هوية الطالب الموثوقة، من دون ترميز صريح لفروع خاصة بالقناة داخل أداة `message` الأساسية.

ولهذا السبب تظل تغييرات التوجيه في embedded-runner عملًا خاصًا بـ Plugin: إذ يكون المشغّل
مسؤولًا عن تمرير هوية الدردشة/الجلسة الحالية إلى حد الاكتشاف الخاص بـ Plugin بحيث
تكشف أداة `message` المشتركة السطح الصحيح المملوك للقناة في
الدور الحالي.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، فيجب على Plugins المجمعة إبقاء وقت تشغيل التنفيذ
داخل وحدات extension الخاصة بها. فلم تعد core تمتلك أوقات تشغيل إجراءات الرسائل الخاصة
بـ Discord أو Slack أو Telegram أو WhatsApp تحت `src/agents/tools`.
ولا ننشر مسارات فرعية منفصلة من نوع `plugin-sdk/*-action-runtime`، ويجب على Plugins المجمعة
استيراد كود وقت التشغيل المحلي الخاص بها مباشرةً من وحداتها
المملوكة للامتداد.

وينطبق الحد نفسه على أسطح SDK المسماة باسم الموفّر عمومًا: يجب ألا
تستورد core قضبان convenience خاصة بقنوات Slack أو Discord أو Signal أو
WhatsApp أو الامتدادات المشابهة. وإذا احتاجت core إلى سلوك ما، فإما أن
تستهلك القضيب `api.ts` / `runtime-api.ts` الخاص بـ Plugin المجمّعة، أو ترقي الحاجة
إلى قدرة عامة ضيقة في SDK المشتركة.

بالنسبة إلى الاستفتاءات تحديدًا، يوجد مسارا تنفيذ:

- يُعد `outbound.sendPoll` خط الأساس المشترك للقنوات التي تناسب
  نموذج الاستفتاء المشترك
- يُعد `actions.handleAction("poll")` المسار المفضل لدلالات
  الاستفتاء الخاصة بالقناة أو لمعاملات الاستفتاء الإضافية

تؤجل Core الآن تحليل الاستفتاء المشترك إلى ما بعد رفض إرسال
الاستفتاء الخاص بـ Plugin للإجراء، بحيث يمكن لمعالجات الاستفتاء المملوكة لـ Plugin قبول
حقول استفتاء خاصة بالقناة من دون أن يحظرها محلل الاستفتاء العام أولًا.

راجع [التركيبات الداخلية لبنية Plugin](/ar/plugins/architecture-internals) للحصول على تسلسل بدء التشغيل الكامل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع Plugin الأصلية بوصفها حد الملكية الخاص بـ **شركة** أو **ميزة**، وليس كحقيبة عشوائية من عمليات التكامل غير المرتبطة.

وهذا يعني:

- يجب أن تمتلك Plugin الخاصة بشركة معينة عادةً جميع
  الأسطح المواجهة لـ OpenClaw الخاصة بتلك الشركة
- يجب أن تمتلك Plugin الخاصة بميزة معيّنة عادةً السطح الكامل للميزة
  التي تقدمها
- يجب أن تستهلك القنوات قدرات core المشتركة بدلًا من إعادة تنفيذ
  سلوك الموفّر بشكل مخصص

<Accordion title="أمثلة على أنماط الملكية عبر Plugins المجمعة">
  - **موفّر متعدد القدرات**: تمتلك `openai` الاستدلال النصي، والكلام، والصوت
    الفوري، وفهم الوسائط، وتوليد الصور. وتمتلك `google` الاستدلال النصي
    بالإضافة إلى فهم الوسائط، وتوليد الصور، والبحث على الويب.
    وتمتلك `qwen` الاستدلال النصي بالإضافة إلى فهم الوسائط وتوليد الفيديو.
  - **موفّر أحادي القدرة**: تمتلك `elevenlabs` و`microsoft` الكلام؛
    وتمتلك `firecrawl` جلب الويب؛ وتمتلك `minimax` / `mistral` / `moonshot` / `zai`
    واجهات فهم الوسائط الخلفية.
  - **Plugin ميزة**: تمتلك `voice-call` نقل المكالمات، والأدوات، وCLI، وroutes،
    وجسر وسائط Twilio media-stream، لكنها تستهلك قدرات الكلام المشتركة،
    والنسخ الفوري، والصوت الفوري بدلًا من استيراد Plugins المورّدين مباشرةً.
</Accordion>

الحالة النهائية المقصودة هي:

- تعيش OpenAI في Plugin واحدة حتى لو امتدت عبر نماذج النص، والكلام، والصور،
  والفيديو مستقبلًا
- ويمكن لمورّد آخر أن يفعل الشيء نفسه بالنسبة إلى المساحة الخاصة به
- لا تهتم القنوات بأي Plugin مورّد تمتلك الموفّر؛ فهي تستهلك
  عقد القدرة المشتركة التي تكشفها core

هذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد core يمكن لعدة Plugins تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فلن يكون السؤال الأول
"أي موفّر يجب أن يرمّز التعامل مع الفيديو بشكل صريح؟" بل السؤال الأول هو "ما هو
عقد القدرة الأساسية للفيديو؟" وبمجرد وجود ذلك العقد، يمكن لـ Plugins المورّدين
التسجيل مقابله ويمكن لـ Plugins القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فعادةً ما تكون الخطوة الصحيحة هي:

1. تعريف القدرة المفقودة في core
2. كشفها عبر Plugin API/وقت التشغيل بطريقة typed
3. ربط القنوات/الميزات بهذه القدرة
4. ترك Plugins المورّدين تسجل تنفيذاتها

وهذا يبقي الملكية صريحة مع تجنب سلوك في core يعتمد على
مورّد واحد أو على مسار كود مخصص لمرة واحدة خاص بـ Plugin.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تقرير مكان الكود:

- **طبقة القدرة الأساسية**: التنسيق المشترك، والسياسة، والرجوع الاحتياطي، وقواعد
  دمج الإعدادات، ودلالات التسليم، والعقود typed
- **طبقة Plugin الخاصة بالمورّد**: واجهات API الخاصة بالمورّد، والمصادقة، وفهارس النماذج، و
  تركيب الكلام، وتوليد الصور، والواجهات الخلفية المستقبلية للفيديو، ونقاط نهاية الاستخدام
- **طبقة Plugin الخاصة بالقنوات/الميزات**: تكامل Slack/Discord/voice-call/... الذي
  يستهلك القدرات الأساسية ويقدمها على سطح

على سبيل المثال، يتبع TTS هذا الشكل:

- تمتلك core سياسة TTS وقت الرد، وترتيب الرجوع الاحتياطي، والتفضيلات، وتسليم القنوات
- وتمتلك `openai` و`elevenlabs` و`microsoft` تنفيذات التركيب
- وتستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالهاتف

ويجب تفضيل هذا النمط نفسه للقدرات المستقبلية.

### مثال على Plugin شركة متعددة القدرات

يجب أن تبدو Plugin الشركة متماسكة من الخارج. فإذا كان لدى OpenClaw
عقود مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط،
وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث على الويب،
فيمكن لمورّد أن يمتلك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // إعداد الكلام الخاص بالمورّد — نفّذ واجهة SpeechProviderPlugin مباشرةً
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // منطق بيانات الاعتماد + الجلب
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. بل الشكل هو المهم:

- Plugin واحدة تمتلك سطح المورّد
- لا تزال core تمتلك عقود القدرات
- تستهلك Plugins القنوات والميزات مساعدات `api.runtime.*`، وليس كود المورّد
- ويمكن لاختبارات العقد أن تؤكد أن Plugin سجلت القدرات التي
  تزعم امتلاكها

### مثال قدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو على أنه قدرة مشتركة
واحدة. وينطبق نموذج الملكية نفسه هنا:

1. تعرّف core عقد فهم الوسائط
2. تسجل Plugins المورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` بحسب الاقتضاء
3. تستهلك Plugins القنوات والميزات السلوك الأساسي المشترك بدلًا من
   الربط المباشر مع كود المورّد

وهذا يتجنب تضمين افتراضات الفيديو الخاصة بموفّر واحد داخل core. فـ Plugin تمتلك
سطح المورّد؛ وcore تمتلك عقد القدرة وسلوك الرجوع الاحتياطي.

ويستخدم توليد الفيديو بالفعل التسلسل نفسه: إذ تمتلك core
عقد القدرة typed ومساعد وقت التشغيل، وتسجل Plugins المورّدين
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة تحقق عملية ملموسة؟ راجع
[دليل القدرات العملي](/ar/plugins/architecture).

## العقود والفرض

إن سطح Plugin API مقصود أن يكون typed ومركزيًا داخل
`OpenClawPluginApi`. وهذا العقد يعرّف نقاط التسجيل المدعومة و
مساعدات وقت التشغيل التي يمكن لـ Plugin الاعتماد عليها.

لماذا يهم هذا:

- يحصل مؤلفو Plugins على معيار داخلي واحد مستقر
- تستطيع core رفض الملكية المكررة مثل تسجيل Pluginين لمعرّف
  موفّر واحد
- يمكن لبدء التشغيل إظهار تشخيصات قابلة للتنفيذ للتسجيلات المشوّهة
- يمكن لاختبارات العقد فرض ملكية Plugins المجمعة ومنع الانحراف الصامت

توجد طبقتان من الفرض:

1. **فرض التسجيل وقت التشغيل**
   يتحقق سجل Plugin من التسجيلات عند تحميل Plugins. أمثلة:
   تؤدي معرّفات الموفّر المكررة، ومعرّفات موفّر الكلام المكررة، والتسجيلات
   المشوّهة إلى تشخيصات Plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   يتم التقاط Plugins المجمعة في سجلات العقود أثناء تشغيل الاختبارات بحيث
   يمكن لـ OpenClaw تأكيد الملكية صراحةً. ويُستخدم هذا اليوم من أجل
   موفّري النماذج، وموفّري الكلام، وموفّري البحث على الويب، وملكية
   التسجيل المجمّع.

والأثر العملي هو أن OpenClaw تعرف، مسبقًا، أي Plugin تمتلك أي
سطح. وهذا يسمح لـ core والقنوات بالتركيب بسلاسة لأن الملكية
معلنة، وtyped، وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي ينتمي إلى عقد

عقود Plugin الجيدة هي:

- typed
- صغيرة
- خاصة بالقدرة
- مملوكة لـ core
- قابلة لإعادة الاستخدام من قبل عدة Plugins
- قابلة للاستهلاك من قبل القنوات/الميزات من دون معرفة بالمورّد

أما عقود Plugin السيئة فهي:

- سياسة خاصة بالمورّد مخفية داخل core
- مسارات هروب مخصصة لـ Plugin لمرة واحدة تتجاوز السجل
- كود القناة الذي يصل مباشرةً إلى تنفيذ خاص بمورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

وعند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم دع Plugins
تتصل بها.

## نموذج التنفيذ

تعمل Plugins الأصلية لـ OpenClaw **داخل العملية نفسها** مع Gateway. وهي ليست
معزولة داخل sandbox. وتمتلك Plugin أصلية محمّلة حد الثقة على مستوى العملية نفسه الذي تملكه core code.

الآثار:

- يمكن لـ Plugin أصلية تسجيل tools، ومعالجات الشبكة، وhooks، وservices
- يمكن أن يؤدي خطأ في Plugin أصلية إلى تعطل gateway أو زعزعتها
- تعادل Plugin أصلية خبيثة تنفيذ كود عشوائي داخل
  عملية OpenClaw

تكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw تتعامل معها حاليًا
كحزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني ذلك في الغالب
Skills المجمعة.

استخدم allowlists ومسارات تثبيت/تحميل صريحة بالنسبة إلى Plugins غير المجمعة. وتعامل
مع Plugins مساحة العمل على أنها كود خاص بوقت التطوير، وليس افتراضيات إنتاج.

وبالنسبة إلى أسماء حزم مساحة العمل المجمعة، أبقِ معرّف Plugin مثبتًا في اسم
npm: ‏`@openclaw/<id>` افتراضيًا، أو لاحقة typed معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دور Plugin أضيق.

ملاحظة ثقة مهمة:

- تثق `plugins.allow` في **معرّفات Plugin**، وليس في مصدر المنشأ.
- إن Plugin مساحة عمل تملك المعرّف نفسه مثل Plugin مجمعة تحجب
  النسخة المجمعة عمدًا عندما تكون ممكّنة/موجودة في allowlist.
- وهذا طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات السريعة.
- يتم حل الثقة في Plugin المجمعة من لقطة المصدر — أي من البيان
  والكود الموجودين على القرص وقت التحميل — وليس من بيانات التثبيت الوصفية. فلا يمكن
  لسجل تثبيت تالف أو مستبدل أن يوسّع سطح الثقة الخاص بـ Plugin المجمعة
  بصمت إلى ما هو أبعد مما يزعمه المصدر الفعلي.

## حد الصادرات

تُصدّر OpenClaw القدرات، وليس وسائل راحة التنفيذ.

أبقِ تسجيل القدرات عامًا. وقلّم صادرات المساعدة غير التعاقدية:

- المسارات الفرعية للمساعدة الخاصة بـ Plugin المجمعة
- المسارات الفرعية الخاصة بسباكة وقت التشغيل غير المقصودة كواجهة API عامة
- وسائل الراحة الخاصة بالمورّد
- مساعدات الإعداد/‏onboarding التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية المساعدة الخاصة بـ Plugins المجمعة موجودة في خريطة
صادرات SDK المولدة من أجل التوافق وصيانة Plugins المجمعة. ومن الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo` و
`plugin-sdk/zalo-setup` والعديد من أسطح `plugin-sdk/matrix*`. تعامل مع هذه
على أنها صادرات تفاصيل تنفيذ محجوزة، وليست النمط الموصى به في SDK
بالنسبة إلى Plugins الطرف الثالث الجديدة.
