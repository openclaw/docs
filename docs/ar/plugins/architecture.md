## نموذج القدرات العام

القدرات هي نموذج **Plugin الأصلي** العام داخل OpenClaw. يسجّل كل Plugin أصلي في OpenClaw نفسه مقابل نوع واحد أو أكثر من أنواع القدرات:

| القدرة | طريقة التسجيل | أمثلة Plugins |
| ---------------------- | ------------------------------------------------ | ------------------------------------ |
| استدلال النص | `api.registerProvider(...)` | `openai`, `anthropic` |
| خلفية استدلال CLI | `api.registerCliBackend(...)` | `openai`, `anthropic` |
| الكلام | `api.registerSpeechProvider(...)` | `elevenlabs`, `microsoft` |
| النسخ الفوري | `api.registerRealtimeTranscriptionProvider(...)` | `openai` |
| الصوت الفوري | `api.registerRealtimeVoiceProvider(...)` | `openai` |
| فهم الوسائط | `api.registerMediaUnderstandingProvider(...)` | `openai`, `google` |
| توليد الصور | `api.registerImageGenerationProvider(...)` | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى | `api.registerMusicGenerationProvider(...)` | `google`, `minimax` |
| توليد الفيديو | `api.registerVideoGenerationProvider(...)` | `qwen` |
| جلب الويب | `api.registerWebFetchProvider(...)` | `firecrawl` |
| بحث الويب | `api.registerWebSearchProvider(...)` | `google` |
| القناة / المراسلة | `api.registerChannel(...)` | `msteams`, `matrix` |
| اكتشاف Gateway | `api.registerGatewayDiscoveryService(...)` | `bonjour` |

<Note>
أي Plugin يسجّل صفرًا من القدرات لكنه يوفّر خطافات أو أدوات أو خدمات اكتشاف أو خدمات خلفية هو Plugin **قديم للخطافات فقط**. لا يزال هذا النمط مدعومًا بالكامل.
</Note>

### موقف التوافق الخارجي

نموذج القدرات مدمج في النواة وتستخدمه Plugins المضمّنة/الأصلية اليوم، لكن توافق Plugins الخارجية لا يزال يحتاج إلى معيار أشد من "بما أنه مُصدَّر، فهو مجمّد."

| حالة Plugin | الإرشاد |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Plugins الخارجية الحالية | أبقِ التكاملات المعتمدة على الخطافات عاملة؛ هذا هو خط الأساس للتوافق. |
| Plugins المضمّنة/الأصلية الجديدة | فضّل تسجيل القدرات الصريح على الوصولات الخاصة بالمورّد أو التصاميم الجديدة القائمة على الخطافات فقط. |
| Plugins الخارجية التي تعتمد تسجيل القدرات | مسموح، لكن تعامل مع أسطح المساعدة الخاصة بالقدرات على أنها متطورة ما لم تضعها الوثائق كواجهات مستقرة. |

تسجيل القدرات هو الاتجاه المقصود. تظل الخطافات القديمة أكثر مسار أمانًا من حيث عدم كسر التوافق لـ Plugins الخارجية أثناء الانتقال. ليست كل مسارات المساعدة الفرعية المصدّرة متساوية — فضّل العقود الضيقة الموثقة على صادرات المساعدة العرضية.

### أشكال Plugin

يصنف OpenClaw كل Plugin محمّلًا في شكل بناءً على سلوك التسجيل الفعلي له (وليس فقط البيانات الوصفية الثابتة):

<AccordionGroup>
  <Accordion title="plain-capability">
    يسجّل نوع قدرة واحدًا بالضبط (مثل Plugin خاص بمزوّد فقط مثل `mistral`).
  </Accordion>
  <Accordion title="hybrid-capability">
    يسجّل عدة أنواع من القدرات (مثلًا يمتلك `openai` استدلال النص والكلام وفهم الوسائط وتوليد الصور).
  </Accordion>
  <Accordion title="hook-only">
    يسجّل الخطافات فقط (المطبوعة أو المخصصة)، بلا قدرات أو أدوات أو أوامر أو خدمات.
  </Accordion>
  <Accordion title="non-capability">
    يسجّل أدوات أو أوامر أو خدمات أو مسارات لكن بلا قدرات.
  </Accordion>
</AccordionGroup>

استخدم `openclaw plugins inspect <id>` للاطلاع على شكل Plugin وتفصيل قدراته. راجع [مرجع CLI](/ar/cli/plugins#inspect) للتفاصيل.

### الخطافات القديمة

لا يزال الخطاف `before_agent_start` مدعومًا كمسار توافق لـ Plugins التي تعمل بالخطافات فقط. لا تزال Plugins قديمة مستخدمة فعليًا تعتمد عليه.

الاتجاه:

- إبقاؤه عاملًا
- توثيقه كقديم
- تفضيل `before_model_resolve` لأعمال تجاوز النموذج/المزوّد
- تفضيل `before_prompt_build` لأعمال تعديل المطالبة
- إزالته فقط بعد انخفاض الاستخدام الحقيقي وإثبات تغطية التجهيزات سلامة الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى إحدى هذه التسميات:

| الإشارة | المعنى |
| -------------------------- | ------------------------------------------------------------ |
| **الإعداد صالح** | يتم تحليل الإعداد بنجاح وتُحل Plugins |
| **استشارة توافق** | يستخدم Plugin نمطًا مدعومًا لكنه أقدم (مثل `hook-only`) |
| **تحذير قديم** | يستخدم Plugin `before_agent_start`، وهو مهمل |
| **خطأ صارم** | الإعداد غير صالح أو فشل تحميل Plugin |

لن يكسر أي من `hook-only` أو `before_agent_start` الـ Plugin الخاص بك اليوم: `hook-only` استشاري، و`before_agent_start` لا يؤدي إلا إلى تحذير. تظهر هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يتكون نظام Plugin في OpenClaw من أربع طبقات:

<Steps>
  <Step title="البيان + الاكتشاف">
    يجد OpenClaw Plugins المرشحة من المسارات المضبوطة، وجذور مساحات العمل، وجذور Plugins العامة، وPlugins المضمّنة. يقرأ الاكتشاف بيانات `openclaw.plugin.json` الأصلية إضافة إلى بيانات الحزم المدعومة أولًا.
  </Step>
  <Step title="التمكين + التحقق">
    تقرر النواة ما إذا كان Plugin المكتشف ممكّنًا أو معطلًا أو محظورًا أو مختارًا لخانة حصرية مثل الذاكرة.
  </Step>
  <Step title="التحميل وقت التشغيل">
    تُحمّل Plugins OpenClaw الأصلية داخل العملية عبر jiti وتسجّل القدرات في سجل مركزي. تُطبع الحزم المتوافقة إلى سجلات في السجل دون استيراد كود وقت التشغيل.
  </Step>
  <Step title="استهلاك الأسطح">
    يقرأ باقي OpenClaw السجل لكشف الأدوات والقنوات وإعداد المزوّد والخطافات ومسارات HTTP وأوامر CLI والخدمات.
  </Step>
</Steps>

بالنسبة إلى CLI الخاص بـ Plugin تحديدًا، ينقسم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- يمكن أن تبقى وحدة CLI الحقيقية الخاصة بـ Plugin كسولة وأن تسجّل عند أول استدعاء

هذا يبقي كود CLI المملوك لـ Plugin داخل Plugin مع السماح لـ OpenClaw بحجز أسماء الأوامر الجذرية قبل التحليل.

حد التصميم المهم:

- ينبغي أن يعمل التحقق من البيان/الإعداد من **بيانات البيان/المخطط الوصفية** دون تنفيذ كود Plugin
- قد يحمّل اكتشاف القدرات الأصلي كود إدخال Plugin موثوقًا لبناء لقطة سجل غير مفعّلة
- يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` في وحدة Plugin مع `api.registrationMode === "full"`

يسمح هذا الفصل لـ OpenClaw بالتحقق من الإعداد، وشرح Plugins المفقودة/المعطلة، وبناء تلميحات واجهة المستخدم/المخطط قبل أن يصبح وقت التشغيل الكامل نشطًا.

### لقطة البيانات الوصفية لـ Plugin وجدول البحث

يبني بدء Gateway لقطة `PluginMetadataSnapshot` واحدة للقطة الإعداد الحالية. اللقطة خاصة بالبيانات الوصفية فقط: تخزّن فهرس Plugins المثبتة، وسجل البيانات، وتشخيصات البيانات، وخرائط المالكين، ومطبّع معرفات Plugin، وسجلات البيانات. لا تحتفظ بوحدات Plugin المحمّلة أو SDKs المزوّدين أو محتويات الحزم أو صادرات وقت التشغيل.

يستهلك التحقق من الإعداد الواعي بـ Plugin، والتمكين التلقائي عند بدء التشغيل، وتمهيد Plugin في Gateway تلك اللقطة بدلًا من إعادة بناء بيانات البيان/الفهرس الوصفية بشكل مستقل. يُشتق `PluginLookUpTable` من اللقطة نفسها ويضيف خطة Plugins الخاصة ببدء التشغيل لإعداد وقت التشغيل الحالي.

بعد بدء التشغيل، يحتفظ Gateway بلقطة البيانات الوصفية الحالية كمنتج وقت تشغيل قابل للاستبدال. يمكن لاكتشاف مزوّد وقت التشغيل المتكرر استعارة تلك اللقطة بدلًا من إعادة بناء الفهرس المثبت وسجل البيانات لكل تمريرة في كتالوج المزوّدين. تُمسح اللقطة أو تُستبدل عند إيقاف Gateway، وتغييرات الإعداد/مخزون Plugins، وكتابات الفهرس المثبت؛ ويعود المستدعون إلى مسار البيان/الفهرس البارد عند عدم وجود لقطة حالية متوافقة. يجب أن تتضمن فحوصات التوافق جذور اكتشاف Plugin مثل `plugins.load.paths` ومساحة عمل الوكيل الافتراضية، لأن Plugins مساحة العمل جزء من نطاق البيانات الوصفية.

تُبقي اللقطة وجدول البحث قرارات بدء التشغيل المتكررة على المسار السريع:

- ملكية القناة
- بدء تشغيل القناة المؤجل
- معرفات Plugins عند بدء التشغيل
- ملكية المزوّد وخلفية CLI
- ملكية مزوّد الإعداد، والاسم البديل للأمر، ومزوّد كتالوج النماذج، وعقد البيان
- التحقق من مخطط إعداد Plugin ومخطط إعداد القناة
- قرارات التمكين التلقائي عند بدء التشغيل

حد الأمان هو استبدال اللقطة، لا تعديلها. أعد بناء اللقطة عند تغيّر الإعداد أو مخزون Plugins أو سجلات التثبيت أو سياسة الفهرس المستمرة. لا تعاملها كسجل عالمي واسع قابل للتعديل، ولا تحتفظ بلقطات تاريخية غير محدودة. يبقى تحميل Plugin وقت التشغيل منفصلًا عن لقطات البيانات الوصفية حتى لا تختبئ حالة وقت تشغيل قديمة خلف ذاكرة تخزين مؤقت للبيانات الوصفية.

قاعدة التخزين المؤقت موثقة في [الأجزاء الداخلية لبنية Plugin](/ar/plugins/architecture-internals#plugin-cache-boundary): تكون بيانات البيان والاكتشاف الوصفية حديثة ما لم يحتفظ المستدعي بلقطة صريحة أو جدول بحث أو سجل بيانات للتدفق الحالي. ليست ذاكرات التخزين المؤقت المخفية للبيانات الوصفية ولا مدد TTL المعتمدة على ساعة الحائط جزءًا من تحميل Plugin. لا يجوز أن تستمر إلا ذاكرات التخزين المؤقت لمحمّل وقت التشغيل والوحدات وآثار الاعتماديات بعد تحميل الكود أو الآثار المثبتة فعليًا.

لا يزال بعض مستدعي المسار البارد يعيدون بناء سجلات البيانات مباشرة من فهرس Plugin المثبت المستمر بدلًا من تلقي `PluginLookUpTable` من Gateway. يعيد هذا المسار الآن بناء السجل عند الطلب؛ فضّل تمرير جدول البحث الحالي أو سجل بيانات صريح عبر تدفقات وقت التشغيل عندما يكون لدى المستدعي واحد بالفعل.

### تخطيط التفعيل

تخطيط التفعيل جزء من مستوى التحكم. يمكن للمستدعين السؤال عن Plugins ذات الصلة بأمر أو مزوّد أو قناة أو مسار أو عُدّة وكيل أو قدرة محددة قبل تحميل سجلات وقت تشغيل أوسع.

يحافظ المخطط على توافق سلوك البيان الحالي:

- حقول `activation.*` هي تلميحات صريحة للمخطط
- تظل `providers`، و`channels`، و`commandAliases`، و`setup.providers`، و`contracts.tools`، والخطافات بديل ملكية البيان
- تظل API المخطط الخاصة بالمعرفات فقط متاحة للمستدعين الحاليين
- تبلّغ API الخطة عن تسميات الأسباب حتى تستطيع التشخيصات تمييز التلميحات الصريحة من بديل الملكية

<Warning>
لا تتعامل مع `activation` كخطاف دورة حياة أو كبديل لـ `register(...)`. إنها بيانات وصفية تُستخدم لتضييق نطاق التحميل. فضّل حقول الملكية عندما تصف العلاقة بالفعل؛ واستخدم `activation` فقط لتلميحات المخطِّط الإضافية.
</Warning>

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة إرسال/تحرير/تفاعل منفصلة لإجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في النواة، وتمتلك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- تمتلك النواة مضيف أداة `message` المشتركة، وتوصيل الموجّه، ومسك دفاتر الجلسات/المواضيع، وتوجيه التنفيذ
- تمتلك Plugins القنوات اكتشاف الإجراءات المحدد النطاق، واكتشاف القدرات، وأي أجزاء مخطط خاصة بالقناة
- تمتلك Plugins القنوات قواعد محادثة الجلسات الخاصة بالمزوّد، مثل كيفية ترميز معرّفات المحادثات لمعرّفات المواضيع أو وراثتها من المحادثات الأصلية
- تنفذ Plugins القنوات الإجراء النهائي عبر محوّل الإجراء الخاص بها

بالنسبة إلى Plugins القنوات، يكون سطح SDK هو `ChannelMessageActionAdapter.describeMessageTool(...)`. يتيح استدعاء الاكتشاف الموحّد هذا لـ Plugin إرجاع إجراءاته المرئية وقدراته ومساهمات المخطط معًا حتى لا تنفصل هذه الأجزاء عن بعضها.

عندما تحمل معلمة خاصة بأداة الرسائل لقناة معينة مصدر وسائط مثل مسار محلي أو URL وسائط بعيد، ينبغي أن يعيد Plugin أيضًا `mediaSourceParams` من `describeMessageTool(...)`. تستخدم النواة هذه القائمة الصريحة لتطبيق تطبيع مسارات sandbox وتلميحات الوصول إلى الوسائط الصادرة من دون ترميز أسماء المعلمات المملوكة لـ Plugin بشكل ثابت. فضّل الخرائط المحددة بنطاق الإجراء هناك، لا قائمة مسطحة واحدة على مستوى القناة، حتى لا يجري تطبيع معلمة وسائط خاصة بالملف الشخصي في إجراءات غير مرتبطة مثل `send`.

تمرّر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف تلك. تشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` وارد موثوق

هذا مهم لـ Plugins الحساسة للسياق. يمكن للقناة إخفاء إجراءات الرسائل أو إظهارها بناءً على الحساب النشط، أو الغرفة/الموضوع/الرسالة الحالية، أو هوية الطالب الموثوقة من دون ترميز فروع خاصة بالقناة بشكل ثابت في أداة `message` في النواة.

لهذا السبب تبقى تغييرات توجيه المشغّل المضمّن عملاً خاصًا بـ Plugin: فالمشغّل مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد اكتشاف Plugin حتى تكشف أداة `message` المشتركة السطح الصحيح المملوك للقناة للدورة الحالية.

بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، ينبغي أن تُبقي Plugins المضمنة وقت تشغيل التنفيذ داخل وحدات الامتداد الخاصة بها. لم تعد النواة تمتلك أوقات تشغيل إجراءات رسائل Discord أو Slack أو Telegram أو WhatsApp ضمن `src/agents/tools`. لا ننشر مسارات فرعية منفصلة `plugin-sdk/*-action-runtime`، وينبغي أن تستورد Plugins المضمنة كود وقت التشغيل المحلي الخاص بها مباشرة من وحداتها المملوكة للامتداد.

ينطبق الحد نفسه على وصلات SDK المسماة باسم المزوّد عمومًا: ينبغي ألا تستورد النواة براميل الراحة الخاصة بقنوات معينة لـ Slack أو Discord أو Signal أو WhatsApp أو امتدادات مشابهة. إذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك برميل `api.ts` / `runtime-api.ts` الخاص بـ Plugin المضمن نفسه، أو ترفع الحاجة إلى قدرة عامة ضيقة في SDK المشترك.

تتبع Plugins المضمنة القاعدة نفسها. ينبغي ألا يعيد `runtime-api.ts` الخاص بـ Plugin مضمن تصدير واجهة `openclaw/plugin-sdk/<plugin-id>` الموسومة الخاصة به. تبقى هذه الواجهات الموسومة طبقات توافق لـ Plugins الخارجية والمستهلكين الأقدم، لكن ينبغي أن تستخدم Plugins المضمنة الصادرات المحلية إضافة إلى مسارات SDK عامة ضيقة مثل `openclaw/plugin-sdk/channel-policy` أو `openclaw/plugin-sdk/runtime-store` أو `openclaw/plugin-sdk/webhook-ingress`. ينبغي ألا يضيف الكود الجديد واجهات SDK خاصة بمعرّف Plugin ما لم تتطلبها حدود التوافق لمنظومة خارجية قائمة.

بالنسبة إلى الاستطلاعات تحديدًا، يوجد مسارا تنفيذ:

- `outbound.sendPoll` هو الخط الأساس المشترك للقنوات التي تلائم نموذج الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضّل لدلالات الاستطلاع الخاصة بالقناة أو معلمات الاستطلاع الإضافية

تؤجل النواة الآن تحليل الاستطلاعات المشترك إلى ما بعد رفض إرسال استطلاع Plugin للإجراء، حتى تتمكن معالجات الاستطلاعات المملوكة لـ Plugin من قبول حقول استطلاع خاصة بالقناة من دون أن يحجبها محلل الاستطلاعات العام أولًا.

راجع [تفاصيل بنية Plugin الداخلية](/ar/plugins/architecture-internals) لتسلسل بدء التشغيل الكامل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع Plugin أصلي كحد ملكية لـ **شركة** أو **ميزة**، وليس كتجميع عشوائي لتكاملات غير مرتبطة.

هذا يعني:

- ينبغي عادةً أن يمتلك Plugin الشركة كل الأسطح المواجهة لـ OpenClaw الخاصة بتلك الشركة
- ينبغي عادةً أن يمتلك Plugin الميزة سطح الميزة الكامل الذي يقدمه
- ينبغي أن تستهلك القنوات القدرات المشتركة للنواة بدلًا من إعادة تنفيذ سلوك المزوّد بطريقة مخصصة

<AccordionGroup>
  <Accordion title="مزوّد متعدد القدرات">
    يمتلك `openai` الاستدلال النصي، والكلام، والصوت الفوري، وفهم الوسائط، وتوليد الصور. يمتلك `google` الاستدلال النصي إضافة إلى فهم الوسائط، وتوليد الصور، وبحث الويب. يمتلك `qwen` الاستدلال النصي إضافة إلى فهم الوسائط وتوليد الفيديو.
  </Accordion>
  <Accordion title="مزوّد أحادي القدرة">
    يمتلك `elevenlabs` و`microsoft` الكلام؛ ويمتلك `firecrawl` جلب الويب؛ وتمتلك `minimax` / `mistral` / `moonshot` / `zai` خلفيات فهم الوسائط.
  </Accordion>
  <Accordion title="Plugin ميزة">
    يمتلك `voice-call` نقل المكالمات، والأدوات، وCLI، والمسارات، وربط تدفق وسائط Twilio، لكنه يستهلك قدرات الكلام، والنسخ الفوري، والصوت الفوري المشتركة بدلًا من استيراد Plugins المزوّدين مباشرة.
  </Accordion>
</AccordionGroup>

الحالة النهائية المقصودة هي:

- يعيش OpenAI في Plugin واحد حتى إذا امتد عبر نماذج النص، والكلام، والصور، والفيديو المستقبلي
- يمكن لمزوّد آخر أن يفعل الشيء نفسه لمساحة سطحه الخاصة
- لا تهتم القنوات بأي Plugin مزوّد يمتلك المزوّد؛ فهي تستهلك عقد القدرة المشتركة الذي تكشفه النواة

هذا هو الفرق الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد النواة الذي يمكن لعدة Plugins تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw نطاقًا جديدًا مثل الفيديو، فالسؤال الأول ليس "أي مزوّد ينبغي أن يرمّز التعامل مع الفيديو بشكل ثابت؟" السؤال الأول هو "ما عقد قدرة الفيديو في النواة؟" بمجرد وجود ذلك العقد، يمكن لـ Plugins المزوّدين التسجيل عليه ويمكن لـ Plugins القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فعادةً ما تكون الخطوة الصحيحة هي:

<Steps>
  <Step title="تعريف القدرة">
    عرّف القدرة الناقصة في النواة.
  </Step>
  <Step title="كشفها عبر SDK">
    اكشفها عبر API/وقت تشغيل Plugin بطريقة مكتوبة الأنواع.
  </Step>
  <Step title="توصيل المستهلكين">
    وصّل القنوات/الميزات بتلك القدرة.
  </Step>
  <Step title="تنفيذات المزوّدين">
    دع Plugins المزوّدين تسجل التنفيذات.
  </Step>
</Steps>

يبقي هذا الملكية صريحة مع تجنب سلوك في النواة يعتمد على مزوّد واحد أو مسار كود خاص بـ Plugin لمرة واحدة.

### طبقات القدرات

استخدم هذا النموذج الذهني عند تحديد مكان انتماء الكود:

<Tabs>
  <Tab title="طبقة قدرة النواة">
    التنسيق المشترك، والسياسة، والرجوع الاحتياطي، وقواعد دمج التهيئة، ودلالات التسليم، والعقود مكتوبة الأنواع.
  </Tab>
  <Tab title="طبقة Plugin المزوّد">
    واجهات API الخاصة بالمزوّد، والمصادقة، وكتالوجات النماذج، وتوليف الكلام، وتوليد الصور، وخلفيات الفيديو المستقبلية، ونقاط نهاية الاستخدام.
  </Tab>
  <Tab title="طبقة Plugin القناة/الميزة">
    تكامل Slack/Discord/voice-call/إلخ. الذي يستهلك قدرات النواة ويقدمها على سطح ما.
  </Tab>
</Tabs>

على سبيل المثال، يتبع TTS هذا الشكل:

- تمتلك النواة سياسة TTS وقت الرد، وترتيب الرجوع الاحتياطي، والتفضيلات، وتسليم القناة
- تمتلك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- يستهلك `voice-call` مساعد وقت تشغيل TTS للهاتفية

ينبغي تفضيل النمط نفسه للقدرات المستقبلية.

### مثال Plugin شركة متعدد القدرات

ينبغي أن يبدو Plugin الشركة متماسكًا من الخارج. إذا كان لدى OpenClaw عقود مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط، وتوليد الصور، وتوليد الفيديو، وجلب الويب، وبحث الويب، فيمكن لمزوّد أن يمتلك كل أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. المهم هو الشكل:

- يمتلك Plugin واحد سطح المزوّد
- تبقى النواة مالكة لعقود القدرات
- تستهلك القنوات وPlugins الميزات مساعدات `api.runtime.*`، لا كود المزوّد
- يمكن لاختبارات العقود تأكيد أن Plugin سجل القدرات التي يدعي امتلاكها

### مثال قدرة: فهم الفيديو

يتعامل OpenClaw بالفعل مع فهم الصور/الصوت/الفيديو كقدرة مشتركة واحدة. ينطبق نموذج الملكية نفسه هناك:

<Steps>
  <Step title="النواة تعرّف العقد">
    تعرّف النواة عقد فهم الوسائط.
  </Step>
  <Step title="Plugins المزوّدين تسجل">
    تسجل Plugins المزوّدين `describeImage` و`transcribeAudio` و`describeVideo` حسب الاقتضاء.
  </Step>
  <Step title="المستهلكون يستخدمون السلوك المشترك">
    تستهلك القنوات وPlugins الميزات سلوك النواة المشترك بدلًا من التوصيل مباشرة إلى كود المزوّد.
  </Step>
</Steps>

يتجنب ذلك تضمين افتراضات الفيديو الخاصة بمزوّد واحد في النواة. يمتلك Plugin سطح المزوّد؛ وتمتلك النواة عقد القدرة وسلوك الرجوع الاحتياطي.

يستخدم توليد الفيديو بالفعل التسلسل نفسه: تمتلك النواة عقد القدرة مكتوب الأنواع ومساعد وقت التشغيل، وتسجل Plugins المزوّدين تنفيذات `api.registerVideoGenerationProvider(...)` عليه.

هل تحتاج إلى قائمة تحقق طرح ملموسة؟ راجع [كتاب وصفات القدرات](/ar/plugins/architecture).

## العقود والإنفاذ

سطح API الخاص بـ Plugin مكتوب الأنواع ومركزي عمدًا في `OpenClawPluginApi`. يعرّف ذلك العقد نقاط التسجيل المدعومة ومساعدات وقت التشغيل التي قد يعتمد عليها Plugin.

لماذا هذا مهم:

- يحصل مؤلفو Plugin على معيار داخلي مستقر واحد
- يمكن للنواة رفض الملكية المكررة مثل تسجيل Plugins اثنين لمعرّف المزوّد نفسه
- يمكن لبدء التشغيل عرض تشخيصات قابلة للتنفيذ للتسجيل غير الصحيح
- يمكن لاختبارات العقود إنفاذ ملكية Plugins المضمنة ومنع الانحراف الصامت

توجد طبقتان من الإنفاذ:

<AccordionGroup>
  <Accordion title="فرض تسجيل وقت التشغيل">
    يتحقق سجل plugins من التسجيلات أثناء تحميل plugins. أمثلة: معرفات مزودي مكررة، ومعرفات مزودي كلام مكررة، وتسجيلات مشوهة تنتج تشخيصات plugins بدلا من سلوك غير محدد.
  </Accordion>
  <Accordion title="اختبارات العقد">
    يتم التقاط plugins المضمنة في سجلات العقود أثناء تشغيل الاختبارات حتى يتمكن OpenClaw من تأكيد الملكية صراحة. يستخدم هذا اليوم لمزودي النماذج، ومزودي الكلام، ومزودي البحث على الويب، وملكية التسجيل المضمن.
  </Accordion>
</AccordionGroup>

الأثر العملي هو أن OpenClaw يعرف مسبقا أي plugin يملك أي سطح. يتيح ذلك للنواة والقنوات التركيب بسلاسة لأن الملكية مصرح بها، ومكتوبة الأنواع، وقابلة للاختبار بدلا من أن تكون ضمنية.

### ما الذي ينتمي إلى عقد

<Tabs>
  <Tab title="عقود جيدة">
    - مكتوبة الأنواع
    - صغيرة
    - خاصة بقدرة محددة
    - مملوكة للنواة
    - قابلة لإعادة الاستخدام بواسطة عدة plugins
    - قابلة للاستهلاك بواسطة القنوات/الميزات دون معرفة بالمورد

  </Tab>
  <Tab title="عقود سيئة">
    - سياسة خاصة بالمورد مخفية في النواة
    - مخارج plugin لمرة واحدة تتجاوز السجل
    - كود قناة يصل مباشرة إلى تنفيذ مورد
    - كائنات وقت تشغيل مرتجلة ليست جزءا من `OpenClawPluginApi` أو `api.runtime`

  </Tab>
</Tabs>

عند الشك، ارفع مستوى التجريد: عرف القدرة أولا، ثم دع plugins تتصل بها.

## نموذج التنفيذ

تعمل plugins الأصلية في OpenClaw **داخل العملية** مع Gateway. وهي غير معزولة في صندوق حماية. يمتلك plugin الأصلي المحمل حدود الثقة نفسها على مستوى العملية التي يمتلكها كود النواة.

<Warning>
آثار plugin الأصلي: يمكن أن يسجل plugin أدوات، ومعالجات شبكة، وخطافات، وخدمات؛ ويمكن لخلل في plugin أن يعطل Gateway أو يزعزع استقراره؛ ويعادل plugin أصلي خبيث تنفيذ كود عشوائي داخل عملية OpenClaw.
</Warning>

الحزم المتوافقة أكثر أمانا افتراضيا لأن OpenClaw يتعامل معها حاليا كحزم بيانات وصفية/محتوى. في الإصدارات الحالية، يعني ذلك غالبا Skills المضمنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة مع plugins غير المضمنة. تعامل مع plugins مساحة العمل ككود وقت تطوير، وليس كإعدادات إنتاج افتراضية.

بالنسبة إلى أسماء حزم مساحة العمل المضمنة، أبق معرف plugin مرتكزا في اسم npm: `@openclaw/<id>` افتراضيا، أو لاحقة مكتوبة الأنواع ومعتمدة مثل `-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما تعرض الحزمة عمدا دور plugin أضيق.

<Note>
**ملاحظة الثقة:** يثق `plugins.allow` في **معرفات plugins**، وليس في مصدر المنشأ. إن plugin مساحة العمل الذي يحمل المعرف نفسه مثل plugin مضمن يحجب النسخة المضمنة عمدا عندما يكون ذلك plugin الخاص بمساحة العمل مفعلا/مسموحا به. هذا طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات العاجلة. يتم حل ثقة plugin المضمن من لقطة المصدر — البيان والكود على القرص وقت التحميل — بدلا من بيانات التثبيت الوصفية. لا يمكن لسجل تثبيت تالف أو مستبدل أن يوسع بصمت سطح ثقة plugin مضمن إلى ما يتجاوز ما يدعيه المصدر الفعلي.
</Note>

## حد التصدير

يصدر OpenClaw القدرات، لا وسائل راحة التنفيذ.

أبق تسجيل القدرات عاما. قلص صادرات المساعدات غير العقدية:

- مسارات فرعية لمساعدات خاصة بـ plugin مضمن
- مسارات فرعية لتمديدات وقت التشغيل غير مخصصة كواجهة API عامة
- مساعدات راحة خاصة بالمورد
- مساعدات إعداد/تهيئة هي تفاصيل تنفيذ

تم إيقاف مسارات مساعدات plugin المضمن المحجوزة من خريطة تصدير SDK المولدة. أبق المساعدات الخاصة بالمالك داخل حزمة plugin المالكة؛ ورق فقط سلوك المضيف القابل لإعادة الاستخدام إلى عقود SDK عامة مثل `plugin-sdk/gateway-runtime` و`plugin-sdk/security-runtime` و`plugin-sdk/plugin-config-runtime`.
