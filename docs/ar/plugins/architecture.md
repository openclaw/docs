## نموذج الإمكانات العام

الإمكانات هي نموذج **الإضافات الأصلية** العام داخل OpenClaw. تسجّل كل
إضافة OpenClaw أصلية في نوع واحد أو أكثر من أنواع الإمكانات:

| الإمكانية             | طريقة التسجيل                                  | أمثلة على الإضافات                  |
| --------------------- | ---------------------------------------------- | ----------------------------------- |
| استدلال النص          | `api.registerProvider(...)`                    | `openai`, `anthropic`               |
| خلفية استدلال CLI     | `api.registerCliBackend(...)`                  | `openai`, `anthropic`               |
| الكلام                | `api.registerSpeechProvider(...)`              | `elevenlabs`, `microsoft`           |
| النسخ الفوري          | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                          |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`       | `openai`                            |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)`  | `openai`, `google`                  |
| توليد الصور           | `api.registerImageGenerationProvider(...)`     | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`     | `google`, `minimax`                 |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`     | `qwen`                              |
| جلب الويب             | `api.registerWebFetchProvider(...)`            | `firecrawl`                         |
| البحث في الويب        | `api.registerWebSearchProvider(...)`           | `google`                            |
| القناة / المراسلة     | `api.registerChannel(...)`                     | `msteams`, `matrix`                 |

الإضافة التي تسجّل صفرًا من الإمكانات لكنها توفّر خطافات أو أدوات أو
خدمات هي إضافة **قديمة تعتمد على الخطافات فقط**. لا يزال هذا النمط
مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج الإمكانات في النواة ويُستخدم اليوم بواسطة الإضافات
المضمّنة/الأصلية، لكن توافق الإضافات الخارجية لا يزال يحتاج إلى معيار
أكثر صرامة من "تم تصديره، إذًا هو ثابت."

الإرشادات الحالية:

- **الإضافات الخارجية الحالية:** حافظ على عمل التكاملات المعتمدة على
  الخطافات؛ واعتبر ذلك خط الأساس للتوافق
- **الإضافات المضمّنة/الأصلية الجديدة:** فضّل تسجيل الإمكانات الصريح بدلًا
  من الوصولات الخاصة بالمورّد أو التصاميم الجديدة المعتمدة على الخطافات فقط
- **الإضافات الخارجية التي تعتمد تسجيل الإمكانات:** هذا مسموح، لكن اعتبر
  أسطح المساعدة الخاصة بالإمكانات قيد التطور ما لم تذكر الوثائق صراحة أن
  العقد ثابت

القاعدة العملية:

- واجهات API لتسجيل الإمكانات هي الاتجاه المقصود
- تظل الخطافات القديمة المسار الأكثر أمانًا لعدم كسر الإضافات الخارجية خلال
  مرحلة الانتقال
- ليست كل المسارات الفرعية المصدّرة متساوية؛ فضّل العقد الضيق الموثّق، لا
  صادرات المساعدة العرضية

### أشكال الإضافات

يصنّف OpenClaw كل إضافة محمّلة إلى شكل اعتمادًا على سلوك تسجيلها الفعلي
(وليس فقط على البيانات الوصفية الثابتة):

- **plain-capability** -- تسجّل نوع إمكانية واحدًا فقط (على سبيل المثال
  إضافة مزوّد فقط مثل `mistral`)
- **hybrid-capability** -- تسجّل عدة أنواع من الإمكانات (على سبيل المثال
  `openai` يملك استدلال النص والكلام وفهم الوسائط وتوليد الصور)
- **hook-only** -- تسجّل الخطافات فقط (المكتوبة Typed أو المخصّصة)، بدون
  إمكانات أو أدوات أو أوامر أو خدمات
- **non-capability** -- تسجّل أدوات أو أوامر أو خدمات أو مسارات، لكن بدون
  إمكانات

استخدم `openclaw plugins inspect <id>` لرؤية شكل الإضافة وتفصيل إمكاناتها.
راجع [مرجع CLI](/cli/plugins#inspect) لمزيد من التفاصيل.

### الخطافات القديمة

لا يزال الخطاف `before_agent_start` مدعومًا كمسار توافق للإضافات المعتمدة
على الخطافات فقط. لا تزال إضافات قديمة مستخدمة فعليًا تعتمد عليه.

الاتجاه:

- أبقه يعمل
- وثّقه بوصفه قديمًا
- فضّل `before_model_resolve` لعمليات تجاوز النموذج/المزوّد
- فضّل `before_prompt_build` لعمليات تعديل المطالبة
- أزله فقط بعد انخفاض الاستخدام الفعلي وعندما يثبت تغطية الاختبارات أمان
  الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه التسميات:

| الإشارة                    | المعنى                                                       |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يجري تحليل الإعدادات بشكل صحيح وتُحل الإضافات               |
| **compatibility advisory** | تستخدم الإضافة نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)    |
| **legacy warning**         | تستخدم الإضافة `before_agent_start`، وهو مهمل               |
| **hard error**             | الإعدادات غير صالحة أو فشل تحميل الإضافة                     |

لن يؤدي أي من `hook-only` أو `before_agent_start` إلى كسر إضافتك اليوم --
فـ `hook-only` إشارة استشارية، و`before_agent_start` يطلق تحذيرًا فقط. تظهر
هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يتكوّن نظام الإضافات في OpenClaw من أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على الإضافات المرشحة من المسارات المهيأة، وجذور مساحات
   العمل، وجذور الامتدادات العامة، والامتدادات المضمّنة. يقرأ الاكتشاف
   بيانات `openclaw.plugin.json` الخاصة بالإضافات الأصلية بالإضافة إلى
   بيانات الحِزم المدعومة أولًا.
2. **التمكين + التحقق**
   تحدد النواة ما إذا كانت الإضافة المكتشفة ممكّنة أو معطّلة أو محظورة أو
   محددة لخانة حصرية مثل الذاكرة.
3. **التحميل في وقت التشغيل**
   تُحمَّل إضافات OpenClaw الأصلية داخل العملية عبر jiti وتُسجّل الإمكانات
   في سجل مركزي. وتُطبَّع الحِزم المتوافقة إلى سجلات في السجل بدون استيراد
   شيفرة وقت التشغيل.
4. **استهلاك الأسطح**
   يقرأ بقية OpenClaw السجل لكشف الأدوات، والقنوات، وإعداد المزوّدين،
   والخطافات، ومسارات HTTP، وأوامر CLI، والخدمات.

وبالنسبة إلى CLI الخاص بالإضافات تحديدًا، ينقسم اكتشاف أوامر الجذر إلى
مرحلتين:

- تأتي البيانات الوصفية في وقت التحليل من `registerCli(..., { descriptors: [...] })`
- يمكن أن تبقى وحدة CLI الحقيقية الخاصة بالإضافة كسولة وتُسجَّل عند أول استدعاء

يحافظ ذلك على شيفرة CLI المملوكة للإضافة داخل الإضافة نفسها، مع تمكين
OpenClaw من حجز أسماء أوامر الجذر قبل التحليل.

الحدّ التصميمي المهم:

- يجب أن يعمل الاكتشاف والتحقق من الإعدادات من **بيانات البيان/المخطط
  الوصفية** من دون تنفيذ شيفرة الإضافة
- يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` في وحدة الإضافة

يتيح هذا الفصل لـ OpenClaw التحقق من الإعدادات، وشرح الإضافات
المفقودة/المعطّلة، وبناء تلميحات واجهة المستخدم/المخطط قبل تنشيط وقت
التشغيل الكامل.

### إضافات القنوات وأداة الرسائل المشتركة

لا تحتاج إضافات القنوات إلى تسجيل أداة منفصلة للإرسال/التحرير/التفاعل
لإجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في
النواة، وتملك إضافات القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- تملك النواة مضيف أداة `message` المشتركة، وربط المطالبات، وإدارة
  الجلسات/المواضيع، وتوجيه التنفيذ
- تملك إضافات القنوات اكتشاف الإجراءات المقيّدة بالنطاق، واكتشاف
  الإمكانات، وأي أجزاء مخطط خاصة بالقناة
- تملك إضافات القنوات قواعد محادثة الجلسات الخاصة بالمزوّد، مثل كيفية
  ترميز معرّفات المحادثات لمعرّفات المواضيع أو وراثتها من المحادثات الأصلية
- تنفذ إضافات القنوات الإجراء النهائي عبر مكيّف الإجراءات الخاص بها

بالنسبة إلى إضافات القنوات، فإن سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يتيح استدعاء
الاكتشاف الموحد هذا للإضافة إعادة إجراءاتها المرئية وإمكاناتها ومساهماتها
في المخطط معًا بحيث لا تنجرف هذه الأجزاء عن بعضها.

تمرر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الموثوق الوارد

وهذا مهم للإضافات الحساسة للسياق. يمكن للقناة إخفاء إجراءات الرسائل أو
إظهارها بناءً على الحساب النشط أو الغرفة/الموضوع/الرسالة الحالية أو هوية
الطالب الموثوقة، من دون ترميز فروع خاصة بالقناة داخل أداة `message` في
النواة.

ولهذا السبب، تظل تغييرات التوجيه في العارض المضمّن عملًا خاصًا
بالإضافة: فالعارض مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد
اكتشاف الإضافة بحيث تكشف أداة `message` المشتركة السطح المملوك للقناة
الصحيح للدور الحالي.

وبالنسبة إلى مساعدات التنفيذ المملوكة للقناة، ينبغي أن تحتفظ الإضافات
المضمّنة بوقت تشغيل التنفيذ داخل وحدات الامتداد الخاصة بها. لم تعد النواة
تملك أزمنة تشغيل إجراءات الرسائل لـ Discord أو Slack أو Telegram أو
WhatsApp ضمن `src/agents/tools`. ولا ننشر مسارات فرعية منفصلة
`plugin-sdk/*-action-runtime`، وينبغي أن تستورد الإضافات المضمّنة شيفرة
وقت التشغيل المحلية الخاصة بها مباشرةً من وحداتها المملوكة للامتداد.

وينطبق الحد نفسه على فجوات SDK المسماة بأسماء المزوّدين عمومًا: لا ينبغي
للنواة أن تستورد حاويات ملائمة خاصة بالقنوات لـ Slack أو Discord أو Signal
أو WhatsApp أو الامتدادات المشابهة. وإذا احتاجت النواة إلى سلوك ما،
فإما أن تستهلك الحاوية `api.ts` / `runtime-api.ts` الخاصة بالإضافة
المضمّنة نفسها، أو ترقّي الحاجة إلى إمكانية عامة ضيقة في SDK المشترك.

وبالنسبة إلى الاستطلاعات تحديدًا، هناك مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب نموذج
  الاستطلاع الشائع
- `actions.handleAction("poll")` هو المسار المفضّل لدلالات الاستطلاع
  الخاصة بالقناة أو لمعاملات الاستطلاع الإضافية

تؤجل النواة الآن تحليل الاستطلاع المشترك إلى ما بعد رفض إرسال الاستطلاع
من الإضافة، بحيث يمكن لمعالجات الاستطلاع المملوكة للإضافة قبول حقول
استطلاع خاصة بالقناة من دون أن يمنعها محلل الاستطلاع العام أولًا.

راجع [مسار التحميل](#load-pipeline) للاطلاع على تسلسل بدء التشغيل الكامل.

## نموذج ملكية الإمكانات

يتعامل OpenClaw مع الإضافة الأصلية بوصفها حدّ الملكية لشركة **company** أو
**feature**، وليس كمجموعة عشوائية من التكاملات غير المرتبطة.

وهذا يعني:

- ينبغي عادةً أن تملك إضافة الشركة جميع أسطح OpenClaw الخاصة بهذه الشركة
- ينبغي عادةً أن تملك إضافة الميزة السطح الكامل للميزة التي تقدمها
- ينبغي للقنوات أن تستهلك الإمكانات المشتركة في النواة بدلًا من إعادة
  تنفيذ سلوك المزوّد بصورة مخصصة

أمثلة:

- تملك الإضافة المضمّنة `openai` سلوك مزوّد نماذج OpenAI، وسلوك OpenAI
  للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تملك الإضافة المضمّنة `elevenlabs` سلوك كلام ElevenLabs
- تملك الإضافة المضمّنة `microsoft` سلوك كلام Microsoft
- تملك الإضافة المضمّنة `google` سلوك مزوّد نماذج Google بالإضافة إلى
  سلوك Google لفهم الوسائط + توليد الصور + البحث في الويب
- تملك الإضافة المضمّنة `firecrawl` سلوك جلب الويب الخاص بـ Firecrawl
- تملك الإضافات المضمّنة `minimax` و`mistral` و`moonshot` و`zai`
  واجهاتها الخلفية لفهم الوسائط
- تملك الإضافة المضمّنة `qwen` سلوك مزوّد النص لـ Qwen بالإضافة إلى
  سلوك فهم الوسائط وتوليد الفيديو
- إضافة `voice-call` هي إضافة ميزة: فهي تملك نقل المكالمات، والأدوات،
  وCLI، والمسارات، وجسر تدفقات وسائط Twilio، لكنها تستهلك إمكانات الكلام
  المشتركة بالإضافة إلى النسخ الفوري والصوت الفوري بدلًا من استيراد
  إضافات المورّدين مباشرةً

الحالة النهائية المقصودة هي:

- تعيش OpenAI داخل إضافة واحدة حتى لو كانت تمتد عبر نماذج النص والكلام والصور
  والفيديو مستقبلًا
- يمكن لمورّد آخر أن يفعل الشيء نفسه لمساحة السطح الخاصة به
- لا تهتم القنوات بأي إضافة مورّد تملك المزوّد؛ فهي تستهلك عقد
  الإمكانات المشتركة الذي تكشفه النواة

وهذا هو التمييز الأساسي:

- **plugin** = حدّ الملكية
- **capability** = عقد النواة الذي يمكن لعدة إضافات تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فإن السؤال الأول ليس:
"أي مزوّد ينبغي أن يضمّن معالجة الفيديو في شيفرته؟" السؤال الأول هو: "ما
هو عقد إمكانات الفيديو في النواة؟" وبمجرد وجود هذا العقد، يمكن لإضافات
المورّدين التسجيل عليه، ويمكن لإضافات القنوات/الميزات استهلاكه.

وإذا لم تكن الإمكانية موجودة بعد، فعادةً ما تكون الخطوة الصحيحة هي:

1. تعريف الإمكانية المفقودة في النواة
2. كشفها عبر واجهة API/وقت التشغيل الخاصة بالإضافة بطريقة مكتوبة Typed
3. ربط القنوات/الميزات بهذه الإمكانية
4. السماح لإضافات المورّدين بتسجيل التنفيذات

وهذا يُبقي الملكية صريحة مع تجنب سلوك في النواة يعتمد على مورّد واحد أو
مسار شيفرة خاص بإضافة واحدة فقط.

### طبقات الإمكانات

استخدم هذا النموذج الذهني عند تقرير مكان انتماء الشيفرة:

- **طبقة الإمكانات في النواة**: التنسيق المشترك، والسياسات، وآليات
  التراجع، وقواعد دمج الإعدادات، ودلالات التسليم، والعقود المكتوبة Typed
- **طبقة إضافات المورّدين**: واجهات API الخاصة بالمورّد، والمصادقة،
  وكتالوجات النماذج، وتركيب الكلام، وتوليد الصور، وواجهات الفيديو الخلفية
  مستقبلًا، ونقاط نهاية الاستخدام
- **طبقة إضافات القنوات/الميزات**: تكامل Slack/Discord/voice-call/إلخ
  الذي يستهلك إمكانات النواة ويعرضها على سطح معيّن

على سبيل المثال، يتبع TTS هذا الشكل:

- تملك النواة سياسة TTS وقت الرد، وترتيب التراجع، والتفضيلات، وتسليم القناة
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات التركيب
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالهاتف

ويُفضّل اعتماد هذا النمط نفسه للإمكانات المستقبلية.

### مثال على إضافة شركة متعددة الإمكانات

ينبغي أن تبدو إضافة الشركة متماسكة من الخارج. فإذا كان لدى OpenClaw عقود
مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط،
وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث في الويب، فيمكن لمورّد
أن يملك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. المهم هو الشكل:

- إضافة واحدة تملك سطح المورّد
- لا تزال النواة تملك عقود الإمكانات
- تستهلك إضافات القنوات والميزات مساعدات `api.runtime.*`، لا شيفرة المورّد
- يمكن لاختبارات العقود أن تؤكد أن الإضافة سجّلت الإمكانات التي تدّعي
  امتلاكها

### مثال على الإمكانية: فهم الفيديو

يتعامل OpenClaw بالفعل مع فهم الصور/الصوت/الفيديو على أنه إمكانية مشتركة
واحدة. وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد فهم الوسائط
2. تسجّل إضافات المورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. تستهلك إضافات القنوات والميزات سلوك النواة المشترك بدلًا من الربط
   المباشر مع شيفرة المورّد

ويمنع هذا ترسيخ افتراضات الفيديو الخاصة بمزوّد واحد داخل النواة. فالإضافة
تملك سطح المورّد؛ بينما تملك النواة عقد الإمكانية وسلوك التراجع.

ويستخدم توليد الفيديو بالفعل هذا التسلسل نفسه: تملك النواة عقد الإمكانية
المكتوب Typed ومساعد وقت التشغيل، وتسجّل إضافات المورّدين تنفيذات
`api.registerVideoGenerationProvider(...)` عليه.

هل تحتاج إلى قائمة تحقق عملية واضحة لطرح ذلك؟ راجع
[دليل الإمكانات العملي](/ar/plugins/architecture).

## العقود والإنفاذ

سطح واجهة API للإضافات مكتوب Typed ومركزي عمدًا في
`OpenClawPluginApi`. يعرّف هذا العقد نقاط التسجيل المدعومة ومساعدات وقت
التشغيل التي يمكن للإضافة الاعتماد عليها.

لماذا يهم ذلك:

- يحصل مؤلفو الإضافات على معيار داخلي ثابت واحد
- يمكن للنواة رفض الملكية المكررة، مثل تسجيل إضافتين لمعرّف المزوّد نفسه
- يمكن لبدء التشغيل إظهار تشخيصات عملية للتسجيلات غير السليمة
- يمكن لاختبارات العقود فرض ملكية الإضافات المضمّنة ومنع الانجراف الصامت

هناك طبقتان من الإنفاذ:

1. **إنفاذ التسجيل في وقت التشغيل**
   يتحقق سجل الإضافات من التسجيلات أثناء تحميل الإضافات. من الأمثلة:
   معرّفات المزوّدين المكررة، ومعرّفات مزوّدي الكلام المكررة، والتسجيلات
   غير السليمة التي تنتج تشخيصات للإضافة بدلًا من سلوك غير معرّف.
2. **اختبارات العقود**
   تُلتقط الإضافات المضمّنة في سجلات عقود أثناء تشغيل الاختبارات حتى يتمكن
   OpenClaw من تأكيد الملكية صراحةً. ويُستخدم هذا اليوم لمزوّدي النماذج،
   ومزوّدي الكلام، ومزوّدي البحث في الويب، وملكية التسجيلات المضمّنة.

والأثر العملي لذلك هو أن OpenClaw يعرف مسبقًا أي إضافة تملك أي سطح. وهذا
يمكّن النواة والقنوات من التركيب بسلاسة لأن الملكية مُعلنة ومكتوبة وقابلة
للاختبار بدلًا من أن تكون ضمنية.

### ما الذي يجب أن ينتمي إلى عقد

عقود الإضافات الجيدة تكون:

- مكتوبة Typed
- صغيرة
- خاصة بالإمكانية
- مملوكة للنواة
- قابلة لإعادة الاستخدام من عدة إضافات
- قابلة للاستهلاك من القنوات/الميزات من دون معرفة بالمورّد

أما عقود الإضافات السيئة فهي:

- سياسات خاصة بالمورّد مخفية داخل النواة
- مخارج هروب خاصة بإضافة واحدة تتجاوز السجل
- شيفرة قناة تصل مباشرةً إلى تنفيذ خاص بمورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

وعند الشك، ارفع مستوى التجريد: عرّف الإمكانية أولًا، ثم دع الإضافات تتصل
بها.

## نموذج التنفيذ

تعمل إضافات OpenClaw الأصلية **داخل العملية** مع Gateway. وهي ليست
معزولة. فالإضافة الأصلية المحمّلة تمتلك حد الثقة نفسه على مستوى العملية
مثل شيفرة النواة.

الآثار المترتبة:

- يمكن للإضافة الأصلية تسجيل الأدوات، ومعالجات الشبكة، والخطافات، والخدمات
- يمكن لخطأ في إضافة أصلية أن يتسبب في تعطل gateway أو زعزعة استقراره
- تعادل الإضافة الأصلية الخبيثة تنفيذ شيفرة عشوائية داخل عملية OpenClaw

الحِزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يتعامل معها حاليًا على
أنها حِزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني ذلك في الغالب
المهارات المضمّنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة للإضافات غير المضمّنة.
وتعامل مع إضافات مساحة العمل على أنها شيفرة وقت تطوير، لا افتراضات إنتاج.

وبالنسبة إلى أسماء حِزم مساحة العمل المضمّنة، أبقِ معرّف الإضافة مثبتًا في
اسم npm: `@openclaw/<id>` افتراضيًا، أو لاحقة مكتوبة معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو
`-media-understanding` عندما تكشف الحزمة عمدًا دور إضافة أضيق.

ملاحظة ثقة مهمة:

- يثق `plugins.allow` في **معرّفات الإضافات**، وليس في مصدر المنشأ.
- تعمّد إضافة مساحة عمل لها المعرّف نفسه كإضافة مضمّنة إلى تظليل النسخة
  المضمّنة عندما تكون إضافة مساحة العمل تلك مفعّلة/ضمن قائمة السماح.
- هذا سلوك طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات
  العاجلة.

## حدّ التصدير

يصدر OpenClaw الإمكانات، لا وسائل الراحة الخاصة بالتنفيذ.

أبقِ تسجيل الإمكانات عامًا. وقلّص صادرات المساعدة غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدة الإضافات المضمّنة
- المسارات الفرعية الخاصة ببنية وقت التشغيل غير المقصود بها أن تكون API
  عامة
- وسائل الراحة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة التي تعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية المساعدة للإضافات المضمّنة موجودة في خريطة
صادرات SDK المولّدة لأسباب تتعلق بالتوافق وصيانة الإضافات المضمّنة. تشمل
الأمثلة الحالية `plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و
`plugin-sdk/zalo` و`plugin-sdk/zalo-setup` وعدة فجوات
`plugin-sdk/matrix*`. تعامل مع هذه على أنها صادرات محجوزة خاصة بتفاصيل
التنفيذ، لا النمط الموصى به لـ SDK لإضافات الطرف الثالث الجديدة.

## مسار التحميل

عند بدء التشغيل، يفعل OpenClaw تقريبًا ما يلي:

1. يكتشف جذور الإضافات المرشحة
2. يقرأ بيانات الإضافات الأصلية أو الحِزم المتوافقة وبيانات الحزمة الوصفية
3. يرفض المرشحين غير الآمنين
4. يطبّع إعدادات الإضافات (`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. يقرر التمكين لكل مرشح
6. يحمّل الوحدات الأصلية الممكّنة عبر jiti
7. يستدعي خطافات `register(api)` الأصلية (أو `activate(api)` — وهو اسم
   بديل قديم) ويجمع التسجيلات في سجل الإضافات
8. يكشف السجل لأسطح الأوامر/وقت التشغيل

<Note>
`activate` هو اسم بديل قديم لـ `register` — يقوم المحمّل بحل أيهما موجود
(`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع
الإضافات المضمّنة `register`؛ ويفضّل استخدام `register` للإضافات الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. ويُحظر المرشحون عندما يفلت
المدخل من جذر الإضافة، أو يكون المسار قابلاً للكتابة من الجميع، أو تبدو
ملكية المسار مشبوهة بالنسبة إلى الإضافات غير المضمّنة.

### سلوك البيان أولًا

البيان هو مصدر الحقيقة في مستوى التحكم. يستخدمه OpenClaw من أجل:

- تحديد الإضافة
- اكتشاف القنوات/المهارات/مخطط الإعدادات أو إمكانات الحزمة المعلنة
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة في Control UI
- عرض بيانات التثبيت/الكتالوج الوصفية
- الحفاظ على واصفات تفعيل وإعداد منخفضة الكلفة من دون تحميل وقت تشغيل
  الإضافة

بالنسبة إلى الإضافات الأصلية، تكون وحدة وقت التشغيل هي جزء مستوى البيانات.
فهي تسجّل السلوك الفعلي مثل الخطافات، والأدوات، والأوامر، أو تدفقات
المزوّد.

وتبقى الكتلتان الاختياريتان `activation` و`setup` في البيان ضمن مستوى
التحكم. فهما واصفات بيانات وصفية فقط لتخطيط التفعيل واكتشاف الإعداد؛ ولا
تحلان محل التسجيل في وقت التشغيل أو `register(...)` أو `setupEntry`.

### ما الذي يخزّنه المحمّل مؤقتًا

يحتفظ OpenClaw بذاكرات مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل البيان
- سجلات الإضافات المحمّلة

تقلل هذه الذواكر المؤقتة من اندفاعات بدء التشغيل ومن كلفة الأوامر
المتكررة. ومن الآمن التفكير فيها على أنها ذواكر أداء مؤقتة قصيرة العمر،
لا تخزينًا دائمًا.

ملاحظة حول الأداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذواكر المؤقتة.
- اضبط نوافذ الذواكر المؤقتة عبر `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تقوم الإضافات المحمّلة بتعديل متغيرات عامة عشوائية في النواة مباشرةً.
بل تسجّل في سجل إضافات مركزي.

يتتبع السجل ما يلي:

- سجلات الإضافات (الهوية، والمصدر، والمنشأ، والحالة، والتشخيصات)
- الأدوات
- الخطافات القديمة والخطافات المكتوبة Typed
- القنوات
- المزوّدون
- معالجات Gateway RPC
- مسارات HTTP
- مسجّلات CLI
- الخدمات الخلفية
- الأوامر المملوكة للإضافة

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث مباشرةً إلى وحدات
الإضافات. وهذا يبقي التحميل أحادي الاتجاه:

- وحدة الإضافة -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

هذا الفصل مهم من أجل القابلية للصيانة. فهو يعني أن معظم أسطح النواة تحتاج
فقط إلى نقطة تكامل واحدة: "اقرأ السجل"، وليس "ضع حالة خاصة لكل وحدة
إضافة".

## استدعاءات ربط المحادثة

يمكن للإضافات التي تربط محادثة أن تتفاعل عند حسم الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقّي استدعاء بعد الموافقة
على طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة الاستدعاء:

- `status`: `"approved"` أو `"denied"`
- `decision`: `"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط الذي تم حسمه للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسل، وبيانات
  المحادثة الوصفية

هذا الاستدعاء مخصّص للإشعارات فقط. فهو لا يغيّر من يُسمح له بربط محادثة،
ويعمل بعد انتهاء معالجة الموافقة في النواة.

## خطافات وقت تشغيل المزوّد

تمتلك إضافات المزوّد الآن طبقتين:

- بيانات وصفية في البيان: `providerAuthEnvVars` لبحث سريع عن مصادقة
  المزوّد من متغيرات البيئة قبل تحميل وقت التشغيل، و`providerAuthAliases`
  لمتغيرات المزوّد التي تتشارك المصادقة، و`channelEnvVars` لبحث سريع عن
  إعداد/مصادقة القناة من متغيرات البيئة قبل تحميل وقت التشغيل، بالإضافة إلى
  `providerAuthChoices` لتسميات إعداد المستخدم/اختيار المصادقة الرخيصة
  وبيانات CLI الوصفية الخاصة بالأعلام قبل تحميل وقت التشغيل
- خطافات وقت الإعدادات: `catalog` / `discovery` القديم بالإضافة إلى
  `applyConfigDefaults`
- خطافات وقت التشغيل: `normalizeModelId`, `normalizeTransport`,
  `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `resolveExternalAuthProfiles`,
  `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`, `normalizeResolvedModel`,
  `contributeResolvedModelCompat`, `capabilities`,
  `normalizeToolSchemas`, `inspectToolSchemas`,
  `resolveReasoningOutputMode`, `prepareExtraParams`, `createStreamFn`,
  `wrapStreamFn`, `resolveTransportTurnState`,
  `resolveWebSocketSessionPolicy`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`, `matchesContextOverflowError`,
  `classifyFailoverReason`, `isCacheTtlEligible`,
  `buildMissingAuthMessage`, `suppressBuiltInModel`, `augmentModelCatalog`,
  `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `isModernModelRef`, `prepareRuntimeAuth`,
  `resolveUsageAuth`, `fetchUsageSnapshot`, `createEmbeddingProvider`,
  `buildReplayPolicy`,
  `sanitizeReplayHistory`, `validateReplayTurns`, `onModelSelected`

لا يزال OpenClaw يملك حلقة الوكيل العامة، وآلية التبديل الاحتياطي، ومعالجة
النصوص، وسياسة الأدوات. وتمثل هذه الخطافات سطح التوسعة للسلوك الخاص
بالمزوّد من دون الحاجة إلى نقل استدلال مخصص كامل.

استخدم `providerAuthEnvVars` في البيان عندما يمتلك المزوّد بيانات اعتماد
قائمة على متغيرات البيئة ينبغي لمسارات المصادقة/الحالة/منتقي النموذج
العامة رؤيتها من دون تحميل وقت تشغيل الإضافة. واستخدم
`providerAuthAliases` في البيان عندما ينبغي لمعرّف مزوّد واحد أن يعيد
استخدام متغيرات البيئة وملفات تعريف المصادقة والمصادقة القائمة على
الإعدادات وخيار إعداد مفتاح API الخاص بمعرّف مزوّد آخر. واستخدم
`providerAuthChoices` في البيان عندما ينبغي لأسطح CLI الخاصة بإعداد
المستخدم/اختيار المصادقة أن تعرف معرّف اختيار المزوّد وتسميات المجموعة
والربط البسيط للمصادقة بعَلَم واحد من دون تحميل وقت تشغيل المزوّد. وأبقِ
`envVars` في وقت تشغيل المزوّد لتلميحات المشغّل مثل تسميات الإعداد أو
متغيرات إعداد OAuth الخاصة بـ client-id/client-secret.

استخدم `channelEnvVars` في البيان عندما تمتلك القناة مصادقة أو إعدادًا
مدفوعًا بمتغيرات البيئة ينبغي لبدائل shell-env العامة أو فحوصات
الإعدادات/الحالة أو مطالبات الإعداد رؤيتها من دون تحميل وقت تشغيل القناة.

### ترتيب الخطافات واستخدامها

بالنسبة إلى إضافات النماذج/المزوّدين، يستدعي OpenClaw الخطافات تقريبًا
بهذا الترتيب.
ويمثل عمود "متى يُستخدم" دليل القرار السريع.

يتحقق `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
إضافة المزوّد المطابقة، ثم ينتقل إلى إضافات المزوّد الأخرى القادرة على
الخطافات إلى أن يغيّر أحدها فعليًا معرّف النموذج أو النقل/الإعدادات. وهذا
يبقي طبقات alias/compat الخاصة بالمزوّد تعمل من دون الحاجة إلى أن يعرف
المستدعي أي إضافة مضمّنة تملك إعادة الكتابة. وإذا لم يُعِد أي خطاف مزوّد
كتابة إدخال إعدادات Google-family مدعوم، فسيظل مطبّع إعدادات Google
المضمّن يطبّق تنظيف التوافق هذا.

إذا كان المزوّد يحتاج إلى بروتوكول wire مخصص بالكامل أو منفذ طلبات مخصص
بالكامل، فهذه فئة مختلفة من الامتداد. هذه الخطافات مخصّصة لسلوك المزوّد
الذي لا يزال يعمل على حلقة الاستدلال العادية في OpenClaw.

### مثال على مزوّد

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مضمّنة

- تستخدم Anthropic الخطافات `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`،
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`isCacheTtlEligible`،
  و`resolveDefaultThinkingLevel` و`applyConfigDefaults` و`isModernModelRef`،
  و`wrapStreamFn` لأنها تملك التوافق الأمامي لـ Claude 4.6، وتلميحات
  عائلة المزوّد، وإرشادات إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية cache الخاصة بالمطالبات، وإعدادات افتراضية واعية بالمصادقة،
  وسياسة التفكير الافتراضي/التكيفية لـ Claude، وتشكيل البث الخاص بـ
  Anthropic من أجل رؤوس beta، و`/fast` / `serviceTier`، و`context1m`.
- تبقى مساعدات البث الخاصة بـ Claude في Anthropic داخل الفجوة العامة
  `api.ts` / `contract-api.ts` الخاصة بالإضافة المضمّنة حاليًا. ويصدر سطح
  تلك الحزمة `wrapAnthropicProviderStream` و`resolveAnthropicBetas` و
  `resolveAnthropicFastMode` و`resolveAnthropicServiceTier` وبُنّاة أغلفة
  Anthropic منخفضة المستوى بدلًا من توسيع SDK العام حول قواعد رؤوس beta
  الخاصة بمزوّد واحد.
- تستخدم OpenAI الخطافات `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و
  `suppressBuiltInModel` و`augmentModelCatalog` و`supportsXHighThinking` و
  `isModernModelRef` لأنها تملك التوافق الأمامي لـ GPT-5.4، وتطبيع
  OpenAI المباشر من `openai-completions` إلى `openai-responses`، وتلميحات
  المصادقة الواعية بـ Codex، وإخفاء Spark، وصفوف قوائم OpenAI التركيبية،
  وسياسة التفكير/النماذج الحية لـ GPT-5؛ بينما تملك عائلة البث
  `openai-responses-defaults` أغلفة OpenAI Responses الأصلية المشتركة
  لرؤوس الإسناد، و`/fast`/`serviceTier`، وإسهاب النص، و`web search` الأصلية
  لـ Codex، وتشكيل حمولة reasoning-compat، وإدارة سياق Responses.
- تستخدم OpenRouter `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن المزوّد تمريري وقد يكشف معرّفات نماذج جديدة قبل
  تحديث الكتالوج الثابت في OpenClaw؛ كما يستخدم `capabilities` و
  `wrapStreamFn` و`isCacheTtlEligible` لإبقاء رؤوس الطلب الخاصة بالمزوّد،
  والبيانات الوصفية للتوجيه، وترقيعات الاستدلال، وسياسة cache الخاصة
  بالمطالبات خارج النواة. وتأتي سياسة إعادة التشغيل الخاصة به من عائلة
  `passthrough-gemini`، بينما تملك عائلة البث `openrouter-thinking` حقن
  الاستدلال عبر proxy وتجاوزات النماذج غير المدعومة / `auto`.
- يستخدم GitHub Copilot `catalog` و`auth` و`resolveDynamicModel` و
  `capabilities` بالإضافة إلى `prepareRuntimeAuth` و`fetchUsageSnapshot`
  لأنه يحتاج إلى تسجيل دخول بالجهاز يملكه المزوّد، وسلوك تراجع للنموذج،

## مساعدات وقت التشغيل

يمكن للإضافات الوصول إلى مجموعة محددة من مساعدات النواة عبر `api.runtime`.
وبالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- تعيد `textToSpeech` حمولة مخرجات TTS العادية في النواة لأسطح
  الملفات/الملاحظات الصوتية.
- تستخدم إعدادات `messages.tts` في النواة واختيار المزوّد.
- تعيد مخزن PCM الصوتي المؤقت + معدل العيّنة. وعلى الإضافات إعادة
  أخذ العينات/الترميز للمزوّدين.
- `listVoices` اختيارية بحسب المزوّد. استخدمها لمنتقيات الأصوات أو تدفقات
  الإعداد المملوكة للمورّد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل locale وgender
  وعلامات الشخصية لمنتقيات واعية بالمزوّد.
- يدعم OpenAI وElevenLabs الهاتف حاليًا. أما Microsoft فلا يدعمه.

يمكن للإضافات أيضًا تسجيل مزوّدي الكلام عبر
`api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS وآليات التراجع وتسليم الردود في النواة.
- استخدم مزوّدي الكلام لسلوك التركيب المملوك للمورّد.
- يُطبَّع الإدخال القديم `edge` الخاص بـ Microsoft إلى معرّف المزوّد
  `microsoft`.
- نموذج الملكية المفضّل موجه نحو الشركة: يمكن لإضافة مورّد واحدة أن تملك
  النص، والكلام، والصورة، ومزوّدي الوسائط مستقبلًا مع إضافة OpenClaw
  لعقود تلك الإمكانات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجّل الإضافات مزوّد
media-understanding مكتوبًا واحدًا بدلًا من حقيبة مفتاح/قيمة عامة:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، وآليات التراجع، والإعدادات، وربط القنوات في النواة.
- أبقِ سلوك المورّد داخل إضافة المزوّد.
- يجب أن يبقى التوسع الإضافي مكتوبًا Typed: أساليب اختيارية جديدة، وحقول
  نتائج اختيارية جديدة، وإمكانات اختيارية جديدة.
- يتبع توليد الفيديو بالفعل النمط نفسه:
  - تملك النواة عقد الإمكانية ومساعد وقت التشغيل
  - تسجّل إضافات المورّدين `api.registerVideoGenerationProvider(...)`
  - تستهلك إضافات الميزات/القنوات `api.runtime.videoGeneration.*`

وبالنسبة إلى مساعدات وقت التشغيل الخاصة بـ media-understanding، يمكن
للإضافات استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن للإضافات استخدام وقت تشغيل
media-understanding أو الاسم البديل الأقدم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- `api.runtime.mediaUnderstanding.*` هو السطح المشترك المفضل لفهم
  الصور/الصوت/الفيديو.
- يستخدم إعدادات الصوت الخاصة بـ media-understanding في النواة
  (`tools.media.audio`) وترتيب التراجع الخاص بالمزوّد.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج مخرجات نسخ
  (على سبيل المثال، عند تخطي الإدخال أو عدم دعمه).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا بديلًا للتوافق.

يمكن للإضافات أيضًا إطلاق تشغيلات subagent في الخلفية عبر
`api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- `provider` و`model` هما تجاوزات اختيارية لكل تشغيل، وليسا تغييرات دائمة
  للجلسة.
- لا يحترم OpenClaw حقول التجاوز هذه إلا للجهات الموثوقة.
- بالنسبة إلى تشغيلات التراجع المملوكة للإضافة، يجب على المشغّلين تفعيل
  ذلك صراحةً عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد الإضافات
  الموثوقة بأهداف `provider/model` قانونية محددة، أو `"*"` للسماح صراحةً
  بأي هدف.
- تظل تشغيلات subagent الخاصة بالإضافات غير الموثوقة تعمل، لكن تُرفَض طلبات
  التجاوز بدلًا من التراجع الصامت.

وبالنسبة إلى البحث في الويب، يمكن للإضافات استهلاك مساعد وقت التشغيل
المشترك بدلًا من الوصول إلى ربط أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن للإضافات أيضًا تسجيل مزوّدي البحث في الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار المزوّد، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في
  النواة.
- استخدم مزوّدي البحث في الويب لوسائط النقل الخاصة بالمورّد.
- `api.runtime.webSearch.*` هو السطح المشترك المفضل لإضافات الميزات/القنوات
  التي تحتاج إلى سلوك البحث من دون الاعتماد على غلاف أداة الوكيل.

## مسارات Gateway HTTP

يمكن للإضافات كشف نقاط نهاية HTTP عبر `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار route ضمن خادم Gateway HTTP.
- `auth`: مطلوب. استخدم `"gateway"` لفرض مصادقة Gateway العادية، أو
  `"plugin"` للمصادقة/التحقق من webhook الذي تديره الإضافة.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح للإضافة نفسها باستبدال تسجيل المسار
  الحالي الخاص بها.
- `handler`: أعد `true` عندما يكون المسار قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وستسبب خطأ في تحميل الإضافة.
  استخدم `api.registerHttpRoute(...)` بدلًا منها.
- يجب أن تعلن مسارات الإضافات `auth` صراحةً.
- تُرفَض تعارضات `path + match` المطابقة إلا عند
  `replaceExisting: true`، ولا يمكن لإضافة أن تستبدل مسار إضافة أخرى.
- تُرفَض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل
  التمرير `exact`/`prefix` ضمن مستوى المصادقة نفسه فقط.
- لا تتلقى المسارات ذات `auth: "plugin"` نطاقات وقت تشغيل المشغّل
  تلقائيًا. فهي مخصصة لـ webhooks/التحقق من التوقيع الذي تديره الإضافة، لا
  لاستدعاءات مساعدات Gateway المميّزة.
- تعمل المسارات ذات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway،
  لكن هذا النطاق محافظ عمدًا:
  - تحافظ مصادقة bearer بالسر المشترك
    (`gateway.auth.mode = "token"` / `"password"`) على تثبيت نطاقات وقت
    التشغيل لمسارات الإضافات عند `operator.write`، حتى إذا أرسل المستدعي
    `x-openclaw-scopes`
  - أوضاع HTTP الموثوقة الحاملة للهوية (على سبيل المثال `trusted-proxy` أو
    `gateway.auth.mode = "none"` على مدخل خاص) تحترم `x-openclaw-scopes`
    فقط عندما يكون الرأس موجودًا صراحةً
  - إذا كان `x-openclaw-scopes` غائبًا في طلبات مسارات الإضافات الحاملة
    للهوية تلك، فإن نطاق وقت التشغيل يتراجع إلى `operator.write`
- القاعدة العملية: لا تفترض أن مسار إضافة بمصادقة gateway هو سطح إدارة
  ضمني. إذا كان مسارك يحتاج إلى سلوك خاص بالمسؤول فقط، فاطلب وضع مصادقة
  حاملًا للهوية ووثّق عقد الرأس `x-openclaw-scopes` الصريح.

## مسارات استيراد Plugin SDK

استخدم المسارات الفرعية لـ SDK بدلًا من الاستيراد الأحادي
`openclaw/plugin-sdk` عند تأليف الإضافات:

- `openclaw/plugin-sdk/plugin-entry` لأساسيات تسجيل الإضافات.
- `openclaw/plugin-sdk/core` للعقد المشترك العام الموجه للإضافات.
- `openclaw/plugin-sdk/config-schema` لتصدير مخطط Zod الجذري
  `openclaw.json` (`OpenClawSchema`).
- الأساسيات المستقرة للقنوات مثل `openclaw/plugin-sdk/channel-setup`،
  و`openclaw/plugin-sdk/setup-runtime`،
  و`openclaw/plugin-sdk/setup-adapter-runtime`،
  و`openclaw/plugin-sdk/setup-tools`،
  و`openclaw/plugin-sdk/channel-pairing`،
  و`openclaw/plugin-sdk/channel-contract`،
  و`openclaw/plugin-sdk/channel-feedback`،
  و`openclaw/plugin-sdk/channel-inbound`،
  و`openclaw/plugin-sdk/channel-lifecycle`،
  و`openclaw/plugin-sdk/channel-reply-pipeline`،
  و`openclaw/plugin-sdk/command-auth`،
  و`openclaw/plugin-sdk/secret-input`، و
  `openclaw/plugin-sdk/webhook-ingress` لربط الإعداد/المصادقة/الرد/الويب هوك
  المشترك. ويُعد `channel-inbound` الموطن المشترك لإلغاء الارتداد debounce،
  ومطابقة الإشارات mention، ومساعدات سياسة الإشارات الواردة، وتنسيق
  المغلفات، ومساعدات سياق المغلفات الواردة.
  ويمثل `channel-setup` فجوة الإعداد الضيقة للتثبيت الاختياري.
  أما `setup-runtime` فهو سطح الإعداد الآمن وقت التشغيل المستخدم بواسطة
  `setupEntry` / بدء التشغيل المؤجل، بما في ذلك مكيّفات ترقيع الإعداد
  الآمنة للاستيراد.
  ويمثل `setup-adapter-runtime` فجوة مكيّف إعداد الحساب الواعية بالبيئة.
  أما `setup-tools` فهو فجوة المساعدات الصغيرة الخاصة بـ CLI/الأرشيف/
  الوثائق (`formatCliCommand`,
  `detectBinary`, `extractArchive`, `resolveBrewExecutable`, `formatDocsLink`,
  `CONFIG_DIR`).
- المسارات الفرعية الخاصة بالمجالات مثل
  `openclaw/plugin-sdk/channel-config-helpers`،
  و`openclaw/plugin-sdk/allow-from`،
  و`openclaw/plugin-sdk/channel-config-schema`،
  و`openclaw/plugin-sdk/telegram-command-config`،
  و`openclaw/plugin-sdk/channel-policy`،
  و`openclaw/plugin-sdk/approval-gateway-runtime`،
  و`openclaw/plugin-sdk/approval-handler-adapter-runtime`،
  و`openclaw/plugin-sdk/approval-handler-runtime`،
  و`openclaw/plugin-sdk/approval-runtime`،
  و`openclaw/plugin-sdk/config-runtime`،
  و`openclaw/plugin-sdk/infra-runtime`،
  و`openclaw/plugin-sdk/agent-runtime`،
  و`openclaw/plugin-sdk/lazy-runtime`،
  و`openclaw/plugin-sdk/reply-history`،
  و`openclaw/plugin-sdk/routing`،
  و`openclaw/plugin-sdk/status-helpers`،
  و`openclaw/plugin-sdk/text-runtime`،
  و`openclaw/plugin-sdk/runtime-store`، و
  `openclaw/plugin-sdk/directory-runtime` لمساعدات وقت التشغيل/الإعدادات
  المشتركة. ويُعد `telegram-command-config` الفجوة العامة الضيقة لتطبيع/
  التحقق من الأوامر المخصصة لـ Telegram، ويظل متاحًا حتى إذا كان سطح عقد
  Telegram المضمّن غير متاح مؤقتًا.
  أما `text-runtime` فهو فجوة النص/Markdown/التسجيل المشتركة، بما في ذلك
  إزالة النص المرئي للمساعد، ومساعدات عرض/تجزئة Markdown، ومساعدات
  الإخفاء، ومساعدات directive-tag، وأدوات النص الآمن.
- يجب أن تفضّل الفجوات الخاصة بالموافقات في القنوات عقد
  `approvalCapability` واحدًا على الإضافة. ثم تقرأ النواة مصادقة الموافقة،
  والتسليم، والعرض، والتوجيه الأصلي، وسلوك المعالج الأصلي الكسول عبر هذه
  الإمكانية الواحدة بدلًا من خلط سلوك الموافقة في حقول إضافة غير مرتبطة.
- أصبح `openclaw/plugin-sdk/channel-runtime` مهملًا ويبقى فقط كغلاف توافق
  للإضافات الأقدم. ينبغي للشيفرة الجديدة استيراد الأساسيات العامة الأضيق
  بدلًا منه، ولا ينبغي لشيفرة المستودع إضافة واردات جديدة لهذا الغلاف.
- تظل الواجهات الداخلية للامتدادات المضمّنة خاصة. ويجب على الإضافات
  الخارجية استخدام المسارات الفرعية `openclaw/plugin-sdk/*` فقط. ويمكن
  لشيفرة النواة/الاختبارات في OpenClaw استخدام نقاط الدخول العامة في
  المستودع تحت جذر حزمة الإضافة مثل `index.js` و`api.js` و
  `runtime-api.js` و`setup-entry.js`، والملفات الضيقة النطاق مثل
  `login-qr-api.js`. ولا تستورد أبدًا `src/*` من حزمة إضافة من النواة أو
  من امتداد آخر.
- تقسيم نقطة دخول المستودع:
  `<plugin-package-root>/api.js` هو حاوية المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو حاوية وقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو مدخل الإضافة المضمّنة،
  و`<plugin-package-root>/setup-entry.js` هو مدخل إضافة الإعداد.
- أمثلة مزوّدين مضمّنين حاليًا:
  - تستخدم Anthropic `api.js` / `contract-api.js` لمساعدات بث Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات رؤوس beta، وتحليل
    `service_tier`.
  - تستخدم OpenAI `api.js` لبُنّاة المزوّدين، ومساعدات النماذج الافتراضية،
    وبُنّاة مزوّدي الوقت الفعلي.
  - تستخدم OpenRouter `api.js` لباني المزوّد الخاص بها بالإضافة إلى
    مساعدات الإعداد/التهيئة، بينما يمكن لـ `register.runtime.js` أن يعيد
    تصدير مساعدات `plugin-sdk/provider-stream` العامة للاستخدام المحلي داخل
    المستودع.
- تفضّل نقاط الدخول العامة المحمّلة عبر الواجهة snapshot إعدادات وقت
  التشغيل النشط عند وجوده، ثم تتراجع إلى ملف الإعدادات المحلول على القرص
  عندما لا يكون OpenClaw قد بدأ بعد في تقديم snapshot وقت تشغيل.
- تظل الأساسيات العامة المشتركة هي عقد SDK العام المفضل. ولا تزال توجد
  مجموعة توافق صغيرة محجوزة من فجوات المساعدة المسماة بأسماء قنوات مضمّنة.
  تعامل مع هذه على أنها فجوات صيانة/توافق للإضافات المضمّنة، لا أهداف
  استيراد جديدة لطرف ثالث؛ ويجب أن تستمر العقود الجديدة العابرة للقنوات في
  الوصول إلى المسارات الفرعية العامة `plugin-sdk/*` أو إلى الحاويات
  المحلية للإضافة `api.js` / `runtime-api.js`.

ملاحظة التوافق:

- تجنّب الحاوية الجذرية `openclaw/plugin-sdk` في الشيفرة الجديدة.
- فضّل أولًا الأساسيات المستقرة الضيقة. فالمسارات الفرعية الأحدث الخاصة بـ
  setup/pairing/reply/feedback/contract/inbound/threading/command/secret-input/webhook/infra/
  allowlist/status/message-tool هي العقد المقصود لأعمال الإضافات المضمّنة
  والخارجية الجديدة.
  وينتمي تحليل/مطابقة الأهداف إلى `openclaw/plugin-sdk/channel-targets`.
  كما تنتمي بوابات إجراءات الرسائل ومساعدات معرّف رسالة التفاعل إلى
  `openclaw/plugin-sdk/channel-actions`.
- لا تُعد حاويات المساعدة الخاصة بالامتدادات المضمّنة مستقرة افتراضيًا. إذا
  كانت هناك مساعدة يحتاج إليها امتداد مضمّن فقط، فأبقها خلف الفجوة المحلية
  `api.js` أو `runtime-api.js` الخاصة بالامتداد بدلًا من ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- ينبغي أن تكون فجوات المساعدة المشتركة الجديدة عامة، لا ممهورة باسم
  قناة. وينتمي تحليل الأهداف المشترك إلى
  `openclaw/plugin-sdk/channel-targets`؛ بينما تبقى الواجهات الداخلية
  الخاصة بالقناة خلف الفجوة المحلية `api.js` أو `runtime-api.js` الخاصة
  بالإضافة المالكة.
- توجد مسارات فرعية خاصة بالإمكانات مثل `image-generation` و
  `media-understanding` و`speech` لأن الإضافات المضمّنة/الأصلية تستخدمها
  اليوم. ووجودها لا يعني بحد ذاته أن كل مساعد مصدَّر هو عقد خارجي ثابت على
  المدى الطويل.

## مخططات أداة الرسائل

ينبغي أن تملك الإضافات مساهمات المخطط الخاصة بالقناة في
`describeMessageTool(...)`. أبقِ الحقول الخاصة بالمورّد داخل الإضافة، لا في
النواة المشتركة.

وبالنسبة إلى أجزاء المخطط المشتركة القابلة للنقل، أعد استخدام المساعدات
العامة المصدَّرة عبر `openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحمولات نمط شبكة الأزرار
- `createMessageToolCardSchema()` لحمولات البطاقات المنظمة

إذا كان شكل مخطط ما منطقيًا لمورّد واحد فقط، فعرّفه في المصدر الخاص بتلك
الإضافة بدلًا من ترقيته إلى SDK المشترك.

## حل أهداف القنوات

ينبغي أن تملك إضافات القنوات دلالات الأهداف الخاصة بالقناة. أبقِ مضيف
الإرسال الخارجي مشتركًا وعامًا، واستخدم سطح مكيّف المراسلة لقواعد المورّد:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان ينبغي التعامل مع
  هدف مطبّع على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يحدّد `messaging.targetResolver.looksLikeId(raw, normalized)` ما إذا كان
  ينبغي للنواة أن تتجاوز مباشرةً إلى الحل الشبيه بالمعرّف بدلًا من البحث
  في الدليل.
- يمثّل `messaging.targetResolver.resolveTarget(...)` مسار التراجع الخاص
  بالإضافة عندما تحتاج النواة إلى حل نهائي يملكه المورّد بعد التطبيع أو
  بعد الإخفاق في الدليل.
- يملك `messaging.resolveOutboundSessionRoute(...)` إنشاء مسار الجلسة
  الخارجة الخاص بالمورّد بعد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات التصنيف التي ينبغي أن تحدث قبل
  البحث في الأقران/المجموعات.
- استخدم `looksLikeId` لفحوص "تعامل مع هذا كمعرّف هدف صريح/أصلي".
- استخدم `resolveTarget` لتراجع التطبيع الخاص بالمورّد، لا للبحث الواسع في
  الدليل.
- أبقِ المعرّفات الأصلية الخاصة بالمورّد مثل chat ids وthread ids وJIDs
  وhandles وroom ids داخل قيم `target` أو داخل معاملات خاصة بالمورّد، لا
  في حقول SDK العامة.

## الأدلة المعتمدة على الإعدادات

ينبغي أن تُبقي الإضافات التي تشتق إدخالات الدليل من الإعدادات هذا المنطق
داخل الإضافة وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى أقران/مجموعات مدعومة بالإعدادات مثل:

- أقران الرسائل المباشرة المعتمدين على allowlist
- خرائط القنوات/المجموعات المهيأة
- بدائل الدليل الثابتة المقيّدة بالحساب

لا تتولى المساعدات المشتركة في `directory-runtime` إلا العمليات العامة:

- تصفية الاستعلامات
- تطبيق الحدود
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

وينبغي أن يظل فحص الحساب الخاص بالقناة وتطبيع المعرّفات داخل تنفيذ
الإضافة.

## كتالوجات المزوّدين

يمكن لإضافات المزوّدين تعريف كتالوجات نماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

يعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw في
`models.providers`:

- `{ provider }` لإدخال مزوّد واحد
- `{ providers }` لعدة إدخالات مزوّدين

استخدم `catalog` عندما تملك الإضافة معرّفات نماذج خاصة بالمزوّد، أو
إعدادات افتراضية لـ base URL، أو بيانات وصفية للنماذج محكومة بالمصادقة.

يتحكم `catalog.order` في توقيت دمج كتالوج الإضافة مقارنةً بمزوّدي OpenClaw
الضمنيين المضمّنين:

- `simple`: مزوّدون عاديون يعتمدون على API-key أو متغيرات البيئة
- `profile`: مزوّدون يظهرون عند وجود ملفات تعريف للمصادقة
- `paired`: مزوّدون يركّبون عدة إدخالات مزوّدين مترابطة
- `late`: المرور الأخير، بعد المزوّدين الضمنيين الآخرين

يربح المزوّد اللاحق عند تعارض المفاتيح، لذلك يمكن للإضافات أن تتعمد تجاوز
إدخال مزوّد مضمّن له معرّف المزوّد نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم بديل قديم
- إذا سُجّل كل من `catalog` و`discovery`، يستخدم OpenClaw `catalog`

## فحص القنوات للقراءة فقط

إذا كانت إضافتك تسجّل قناة، فافضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

لماذا:

- `resolveAccount(...)` هو مسار وقت التشغيل. ويُسمح له بافتراض أن بيانات
  الاعتماد قد جرى تجسيدها بالكامل، ويمكنه الإخفاق بسرعة عند غياب الأسرار
  المطلوبة.
- مسارات الأوامر للقراءة فقط مثل `openclaw status` و
  `openclaw status --all` و`openclaw channels status` و
  `openclaw channels resolve` وتدفقات إصلاح doctor/config يجب ألا تحتاج إلى
  تجسيد بيانات اعتماد وقت التشغيل لمجرد وصف الإعداد.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد حالة وصفية للحساب فقط.
- احتفظ بـ `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الصلة، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لا تحتاج إلى إعادة قيم الرموز الخام لمجرد الإبلاغ عن التوافر للقراءة فقط.
  فإعادة `tokenStatus: "available"` (وحقل المصدر المطابق) كافية لأوامر
  نمط الحالة.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مهيأة عبر
  SecretRef لكنها غير متاحة في مسار الأمر الحالي.

يتيح ذلك لأوامر القراءة فقط الإبلاغ عن "مهيأ لكنه غير متاح في مسار الأمر
هذا" بدلًا من التعطل أو الإبلاغ الخاطئ عن أن الحساب غير مهيأ.

## حِزم الحزم

يمكن أن يتضمن دليل الإضافة `package.json` يحتوي على
`openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال إضافة. وإذا سردت الحزمة عدة امتدادات، يصبح معرّف الإضافة
`name/<fileBase>`.

إذا كانت إضافتك تستورد تبعيات npm، فثبّتها في ذلك الدليل بحيث يكون
`node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل
الإضافة بعد حل الروابط الرمزية. وتُرفَض الإدخالات التي تفلت من دليل
الحزمة.

ملاحظة أمنية: يثبت `openclaw plugins install` تبعيات الإضافة باستخدام
`npm install --omit=dev --ignore-scripts` (من دون lifecycle scripts، ومن
دون تبعيات تطوير في وقت التشغيل). أبقِ أشجار تبعيات الإضافة "JavaScript/TypeScript
نقية" وتجنب الحزم التي تتطلب بناء `postinstall`.

اختياري: يمكن لـ `openclaw.setupEntry` الإشارة إلى وحدة خفيفة مخصصة
للإعداد فقط. وعندما يحتاج OpenClaw إلى أسطح إعداد لإضافة قناة معطّلة، أو
عندما تكون إضافة القناة مفعلة لكنها لا تزال غير مهيأة، فإنه يحمّل
`setupEntry` بدلًا من مدخل الإضافة الكامل. وهذا يجعل بدء التشغيل والإعداد
أخف عندما يكون مدخل الإضافة الرئيسي لديك يربط أيضًا أدوات أو خطافات أو
شيفرة أخرى خاصة بوقت التشغيل فقط.

اختياري: يمكن لـ
`openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
إدخال إضافة قناة في مسار `setupEntry` نفسه أثناء مرحلة بدء التشغيل السابقة
للاستماع في gateway، حتى عندما تكون القناة مهيأة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن
يوجد قبل أن يبدأ gateway الاستماع. وعمليًا، يعني ذلك أن مدخل الإعداد يجب أن
يسجل كل إمكانية مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن يبدأ gateway الاستماع
- أي أساليب gateway أو أدوات أو خدمات يجب أن توجد خلال النافذة نفسها

إذا كان مدخلك الكامل لا يزال يملك أي إمكانية مطلوبة عند بدء التشغيل، فلا
تفعّل هذا العلم. وأبقِ الإضافة على السلوك الافتراضي ودع OpenClaw يحمّل
المدخل الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمّنة أيضًا نشر مساعدات لسطح العقد مخصصة للإعداد فقط بحيث
تستطيع النواة الرجوع إليها قبل تحميل وقت تشغيل القناة الكامل. وسطح ترقية
الإعداد الحالي هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعداد قناة قديم ذي حساب
واحد إلى `channels.<id>.accounts.*` من دون تحميل مدخل الإضافة الكامل.
وتمثل Matrix المثال المضمّن الحالي: فهي تنقل مفاتيح المصادقة/التهيئة
الأولية فقط إلى حساب مسمّى مُرقّى عند وجود حسابات مسمّاة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قانوني مهيأ بدلًا من إنشاء
`accounts.default` دائمًا.

وتحافظ مكيّفات ترقيع الإعداد هذه على كسَل اكتشاف سطح العقد المضمّن.
فيبقى وقت الاستيراد خفيفًا؛ ويُحمَّل سطح الترقية عند أول استخدام فقط بدلًا
من إعادة الدخول إلى بدء تشغيل القناة المضمّنة عند استيراد الوحدة.

وعندما تتضمن أسطح بدء التشغيل هذه أساليب Gateway RPC، فأبقها على بادئة
خاصة بالإضافة. وتظل مساحات أسماء إدارة النواة (`config.*`،
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتُحل دائمًا إلى
`operator.admin`، حتى إذا طلبت إضافة نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### البيانات الوصفية لكتالوج القنوات

يمكن لإضافات القنوات الإعلان عن بيانات وصفية للإعداد/الاكتشاف عبر
`openclaw.channel`، وتلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي
بيانات الكتالوج الأساسية خالية من البيانات.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الكتالوج/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص الرابط الخاص بالوثائق
- `preferOver`: معرّفات الإضافات/القنوات ذات الأولوية الأدنى التي ينبغي أن
  تتقدم عليها هذه الخانة في الكتالوج
- `selectionDocsPrefix`, `selectionDocsOmitLabel`, `selectionExtras`: عناصر
  تحكم بالنسخ الخاصة بسطح الاختيار
- `markdownCapable`: يحدد القناة على أنها قادرة على Markdown لقرارات
  التنسيق الصادر
- `exposure.configured`: يُخفي القناة من أسطح سرد القنوات المهيأة عند
  ضبطه على `false`
- `exposure.setup`: يُخفي القناة من منتقيات الإعداد/التهيئة التفاعلية عند
  ضبطه على `false`
- `exposure.docs`: يضع علامة على القناة على أنها داخلية/خاصة لأسطح التنقل
  في الوثائق
- `showConfigured` / `showInSetup`: أسماء بديلة قديمة لا تزال مقبولة
  للتوافق؛ ويفضّل `exposure`
- `quickstartAllowFrom`: يدرج القناة في تدفق `allowFrom` القياسي الخاص
  بالبدء السريع
- `forceAccountBinding`: يفرض ربط حساب صريحًا حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: يفضّل البحث في الجلسة عند حل هدف
  الإعلان

يمكن لـ OpenClaw أيضًا دمج **كتالوجات قنوات خارجية** (مثل تصدير سجل MPM).
ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولًا بفواصل أو فواصل منقوطة أو وفق `PATH`). يجب
أن يحتوي كل ملف على
`{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`.
كما يقبل المحلّل `"packages"` أو `"plugins"` كأسماء بديلة قديمة للمفتاح
`"entries"`.

## إضافات محرك السياق

تملك إضافات محرك السياق تنسيق سياق الجلسة لعمليات الإدخال، والتجميع،
والضغط. سجّلها من إضافتك باستخدام
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط باستخدام
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج إضافتك إلى استبدال مسار السياق الافتراضي أو توسيعه
بدلًا من مجرد إضافة بحث في الذاكرة أو خطافات.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يملك خوارزمية الضغط، فأبقِ `compact()` منفذة وقم
بتفويضها صراحةً:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة إمكانية جديدة

عندما تحتاج إضافة ما إلى سلوك لا يلائم واجهة API الحالية، فلا تتجاوز نظام
الإضافات عبر وصول خاص داخلي. أضف الإمكانية الناقصة.

التسلسل الموصى به:

1. عرّف عقد النواة
   حدّد السلوك المشترك الذي ينبغي أن تملكه النواة: السياسات، وآليات
   التراجع، ودمج الإعدادات، ودورة الحياة، والدلالات الموجهة للقنوات،
   وشكل مساعد وقت التشغيل.
2. أضف أسطح تسجيل/وقت تشغيل مكتوبة Typed للإضافات
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر سطح إمكانيات مكتوب
   Typed مفيد.
3. اربط مستهلكي النواة + القنوات/الميزات
   ينبغي أن تستهلك إضافات القنوات والميزات الإمكانية الجديدة عبر النواة،
   لا عبر استيراد تنفيذ مورّد مباشرةً.
4. سجّل تنفيذات المورّدين
   بعدها تسجّل إضافات المورّدين واجهاتها الخلفية ضمن الإمكانية.
5. أضف تغطية للعقد
   أضف اختبارات بحيث تبقى الملكية وشكل التسجيل صريحين مع الزمن.

هكذا يبقى OpenClaw ذا توجه واضح من دون أن يصبح مضمّنًا لرؤية مورّد واحد
للعالم. راجع [دليل الإمكانات العملي](/ar/plugins/architecture) للحصول على
قائمة ملفات عملية ومثال مفصّل.

### قائمة تحقق الإمكانية

عند إضافة إمكانية جديدة، ينبغي عادةً أن يلمس التنفيذ هذه الأسطح معًا:

- أنواع عقد النواة في `src/<capability>/types.ts`
- مساعد العارض/وقت التشغيل في النواة في `src/<capability>/runtime.ts`
- سطح تسجيل واجهة API للإضافة في `src/plugins/types.ts`
- ربط سجل الإضافات في `src/plugins/registry.ts`
- كشف وقت تشغيل الإضافة في `src/plugins/runtime/*` عندما تحتاج إضافات
  الميزات/القنوات إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- وثائق المشغّلين/الإضافات في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فعادةً ما تكون هذه إشارة إلى أن الإمكانية
لم تُدمج بالكامل بعد.

### قالب الإمكانية

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يبقي القاعدة بسيطة:

- تملك النواة عقد الإمكانية + التنسيق
- تملك إضافات المورّدين تنفيذات المورّد
- تستهلك إضافات الميزات/القنوات مساعدات وقت التشغيل
- تُبقي اختبارات العقود الملكية صريحة
