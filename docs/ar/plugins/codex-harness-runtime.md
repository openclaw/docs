توثق هذه الصفحة عقد وقت التشغيل لدورات مشغّل Codex. للإعداد
والتوجيه، ابدأ بـ [مشغّل Codex](/ar/plugins/codex-harness). لحقول الإعدادات،
راجع [مرجع مشغّل Codex](/ar/plugins/codex-harness-reference).

## نظرة عامة

وضع Codex ليس OpenClaw مع استدعاء نموذج مختلف في الخلفية. يمتلك Codex جزءًا أكبر من
حلقة النموذج الأصلية، ويكيّف OpenClaw أسطح Plugin والأداة والجلسة
والتشخيص حول ذلك الحد.

لا يزال OpenClaw يمتلك توجيه القنوات، وملفات الجلسات، وتسليم الرسائل المرئية،
وأدوات OpenClaw الديناميكية، والموافقات، وتسليم الوسائط، ونسخة مطابقة من النص.
ويمتلك Codex الخيط الأصلي المرجعي، وحلقة النموذج الأصلية، واستمرار الأداة الأصلي،
وCompaction الأصلي.

يتبع توجيه المطالبات وقت التشغيل المحدد، وليس مجرد سلسلة المزوّد. تتلقى
دورة Codex الأصلية تعليمات مطوّر خادم تطبيق Codex، بينما يحتفظ
مسار توافق OpenClaw الصريح بمطالبة نظام OpenClaw العادية حتى
عندما يستخدم مصادقة OpenAI أو النقل بنكهة Codex.

يحافظ Codex الأصلي على تعليمات الأساس/النموذج وسلوك مستندات المشروع المملوكة لـ Codex
وفقًا لإعداد خيط Codex النشط. يبدأ OpenClaw خيوط
Codex الأصلية ويستأنفها مع تعطيل الشخصية المدمجة في Codex حتى تبقى
ملفات شخصية مساحة العمل وهوية وكيل OpenClaw هي المرجع. لا تزال
تشغيلات OpenClaw الخفيفة تحافظ على كبح مستندات المشروع الحالي لديها. تغطي
تعليمات مطوّر OpenClaw اهتمامات وقت تشغيل OpenClaw مثل تسليم
قناة المصدر، وأدوات OpenClaw الديناميكية، وتفويض ACP، وسياق المهايئ، وملفات
ملف تعريف مساحة عمل الوكيل النشط. تُسقَط كتالوجات Skills في OpenClaw ومؤشرات
`MEMORY.md` الموجّهة عبر الأدوات كتعليمات مطوّر تعاون محددة بالدورة
لـ Codex الأصلي. لا يزال محتوى `BOOTSTRAP.md` النشط وحقن الاحتياط الكامل لـ
`MEMORY.md` يستخدمان سياق مراجع إدخال الدورة.

## ربط الخيوط وتغييرات النموذج

عندما تُرفق جلسة OpenClaw بخيط Codex موجود، ترسل الدورة التالية
نموذج OpenAI المحدد حاليًا، وسياسة الموافقة، ووضع الحماية، وطبقة الخدمة
إلى خادم التطبيق مرة أخرى. يحافظ الانتقال من `openai/gpt-5.5` إلى
`openai/gpt-5.2` على ربط الخيط، لكنه يطلب من Codex المتابعة باستخدام
النموذج المحدد حديثًا.

## الردود المرئية وHeartbeats

عندما تعمل دورة محادثة مباشرة/مصدرية عبر مشغّل Codex، تكون الردود المرئية
افتراضيًا هي التسليم التلقائي النهائي للمساعد لأسطح WebChat الداخلية.
يبقي هذا Codex متسقًا مع عقد مطالبة مشغّل Pi: يرد الوكلاء
بشكل طبيعي، وينشر OpenClaw النص النهائي إلى محادثة المصدر. اضبط
`messages.visibleReplies: "message_tool"` عندما ينبغي لمحادثة مباشرة/مصدرية
أن تبقي نص المساعد النهائي خاصًا عمدًا ما لم يستدعِ الوكيل
`message(action="send")`.

تحصل دورات Heartbeat في Codex أيضًا على `heartbeat_respond` في كتالوج أدوات OpenClaw
القابل للبحث افتراضيًا، بحيث يستطيع الوكيل تسجيل ما إذا كان ينبغي للاستيقاظ أن يبقى
هادئًا أو يرسل إشعارًا من دون ترميز تدفق التحكم ذلك في النص النهائي.

تُرسل إرشادات المبادرة الخاصة بـ Heartbeat كتعليمة مطوّر لوضع تعاون Codex
في دورة Heartbeat نفسها. تستعيد دورات المحادثة العادية وضع Codex Default
بدلًا من حمل فلسفة Heartbeat في مطالبة وقت التشغيل العادية لديها. عند وجود
`HEARTBEAT.md` غير فارغ، توجّه تعليمات وضع تعاون Heartbeat
‏Codex إلى الملف بدلًا من تضمين محتوياته مباشرة.

## حدود الخطافات

يحتوي مشغّل Codex على ثلاث طبقات خطافات:

| الطبقة                                 | المالك                    | الغرض                                                             |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| خطافات Plugin في OpenClaw                 | OpenClaw                 | توافق المنتج/Plugin عبر مشغّلات OpenClaw وCodex.   |
| وسيط امتداد خادم تطبيق Codex | Plugins المضمّنة في OpenClaw | سلوك المهايئ لكل دورة حول أدوات OpenClaw الديناميكية.            |
| خطافات Codex الأصلية                    | Codex                    | دورة حياة Codex منخفضة المستوى وسياسة الأدوات الأصلية من إعداد Codex. |

لا يستخدم OpenClaw ملفات `hooks.json` الخاصة بالمشروع أو العامة في Codex لتوجيه
سلوك Plugin في OpenClaw. للجسر المدعوم للأدوات الأصلية والأذونات،
يحقن OpenClaw إعداد Codex لكل خيط من أجل `PreToolUse` و`PostToolUse`
و`PermissionRequest` و`Stop`.

عند تمكين موافقات خادم تطبيق Codex، أي عندما لا تكون `approvalPolicy`
هي `"never"`، يحذف إعداد الخطاف الأصلي المحقون افتراضيًا `PermissionRequest` حتى
يتولى مراجع خادم تطبيق Codex وجسر موافقات OpenClaw التصعيدات الحقيقية
بعد المراجعة. يستطيع المشغّلون إضافة `permission_request` صراحةً إلى
`nativeHookRelay.events` عندما يحتاجون إلى ترحيل التوافق.

تبقى خطافات Codex الأخرى مثل `SessionStart` و`UserPromptSubmit`
عناصر تحكم على مستوى Codex. ولا تُعرض كخطافات Plugin في OpenClaw في عقد v1.

بالنسبة إلى أدوات OpenClaw الديناميكية، ينفذ OpenClaw الأداة بعد أن يطلب Codex
الاستدعاء، لذلك يشغّل OpenClaw سلوك Plugin والوسيط الذي يملكه في
مهايئ المشغّل. بالنسبة إلى أدوات Codex الأصلية، يمتلك Codex سجل الأداة المرجعي.
يمكن لـ OpenClaw عكس أحداث محددة، لكنه لا يستطيع إعادة كتابة خيط Codex
الأصلي ما لم يعرّض Codex تلك العملية عبر خادم التطبيق أو استدعاءات الخطافات
الأصلية.

تؤجل أحداث `PreToolUse` في وضع التقرير لخادم تطبيق Codex طلبات موافقة Plugin
إلى موافقة خادم التطبيق المطابقة. إذا أعاد خطاف OpenClaw ‏`before_tool_call`
القيمة `requireApproval` بينما تضبط الحمولة الأصلية وضع موافقة التقرير
(`openclaw_approval_mode` هو `"report"`)، يسجل ترحيل الخطاف الأصلي
متطلب موافقة Plugin ولا يعيد أي قرار أصلي. عندما يرسل Codex
طلب موافقة خادم التطبيق لاستخدام الأداة نفسه، يفتح OpenClaw مطالبة موافقة
Plugin ويربط القرار مرة أخرى بـ Codex. تُعد أحداث Codex ‏`PermissionRequest`
مسار موافقة منفصلًا، ولا يزال بإمكانها التوجيه عبر موافقات OpenClaw
عندما يكون وقت التشغيل معدًا لذلك الجسر.

توفر إشعارات عناصر خادم تطبيق Codex أيضًا ملاحظات `after_tool_call` غير متزامنة
لاكتملات الأدوات الأصلية التي لا يغطيها بالفعل ترحيل `PostToolUse`
الأصلي. هذه الملاحظات مخصصة للقياسات وتوافق Plugin فقط؛ ولا يمكنها حظر
استدعاء الأداة الأصلي أو تأخيره أو تعديله.

تأتي إسقاطات دورة حياة Compaction وLLM من إشعارات خادم تطبيق Codex
وحالة مهايئ OpenClaw، وليس من أوامر خطافات Codex الأصلية.
أحداث OpenClaw ‏`before_compaction` و`after_compaction` و`llm_input` و
`llm_output` هي ملاحظات على مستوى المهايئ، وليست لقطات مطابقة بايتًا ببايت
لطلب Codex الداخلي أو حمولات Compaction.

تُسقَط إشعارات خادم تطبيق Codex الأصلية `hook/started` و`hook/completed`
كأحداث وكيل `codex_app_server.hook` للمسار والتصحيح.
وهي لا تستدعي خطافات Plugin في OpenClaw.

## الأذونات الأصلية وطلبات الاستيضاح في MCP

بالنسبة إلى `PermissionRequest`، لا يرجع OpenClaw إلا قرارات السماح أو الرفض الصريحة
عندما تقرر السياسة. نتيجة عدم وجود قرار ليست سماحًا. يعاملها Codex على أنها بلا
قرار خطاف ويمررها إلى مسار الحارس الخاص به أو موافقة المستخدم.

تحذف أوضاع موافقة خادم تطبيق Codex هذا الخطاف الأصلي افتراضيًا. ينطبق هذا السلوك
عندما تُضمَّن `permission_request` صراحة في
`nativeHookRelay.events` أو يثبّتها وقت تشغيل توافق.

عندما يختار مشغّل `allow-always` لطلب إذن أصلي في Codex،
يتذكر OpenClaw بصمة ذلك المزود/الجلسة/إدخال الأداة/مجلد العمل الحالي الدقيقة ضمن
نافذة جلسة محدودة. القرار المتذكَّر مطابق عمدًا فقط:
أي تغيير في الأمر أو الوسيطات أو حمولة الأداة أو مجلد العمل الحالي ينشئ
موافقة جديدة.

تُوجّه طلبات استيضاح موافقة أدوات Codex MCP عبر تدفق موافقة Plugin في OpenClaw
عندما يعلّم Codex `_meta.codex_approval_kind` بالقيمة
`"mcp_tool_call"`. تُرسل مطالبات `request_user_input` في Codex مرة أخرى إلى
الدردشة الأصلية، وتجيب رسالة المتابعة التالية في الطابور عن طلب الخادم
الأصلي ذلك بدلًا من توجيهها كسياق إضافي. تفشل طلبات استيضاح MCP الأخرى بصورة مغلقة.

للاطلاع على تدفق موافقة Plugin العام الذي يحمل هذه المطالبات، راجع
[طلبات أذونات Plugin](/ar/plugins/plugin-permission-requests).

## توجيه الطابور

يتطابق توجيه طابور التشغيل النشط مع `turn/steer` في خادم تطبيق Codex. مع
القيمة الافتراضية `messages.queue.mode: "steer"`، يجمع OpenClaw رسائل الدردشة
بوضع التوجيه خلال نافذة الهدوء المضبوطة ويرسلها كطلب `turn/steer` واحد
بترتيب الوصول.

يمكن أن ترفض مراجعة Codex وأدوار Compaction اليدوية التوجيه في الدور نفسه. في تلك
الحالة، ينتظر OpenClaw انتهاء التشغيل النشط قبل بدء الموجّه.
استخدم `/queue followup` أو `/queue collect` عندما يجب وضع الرسائل في قائمة الانتظار افتراضيًا
بدلًا من التوجيه. راجع [قائمة انتظار التوجيه](/ar/concepts/queue-steering).

## رفع ملاحظات Codex

عند الموافقة على `/diagnostics [note]` لجلسة تستخدم حاضنة Codex الأصلية،
يستدعي OpenClaw أيضًا `feedback/upload` في خادم تطبيق Codex للخيوط ذات الصلة
في Codex. يطلب الرفع من خادم التطبيق تضمين السجلات لكل خيط مدرج
وخيوط Codex الفرعية المنشأة عند توفرها.

يمر الرفع عبر مسار ملاحظات Codex العادي إلى خوادم OpenAI. إذا كانت ملاحظات Codex
معطلة في خادم التطبيق ذلك، يعيد الأمر خطأ خادم التطبيق.
يسرد رد التشخيصات المكتمل القنوات، ومعرّفات جلسات OpenClaw،
ومعرّفات خيوط Codex، وأوامر `codex resume <thread-id>` المحلية للخيوط
التي أُرسلت.

إذا رفضت الموافقة أو تجاهلتها، فلن يطبع OpenClaw معرّفات Codex تلك
ولن يرسل ملاحظات Codex. لا يستبدل الرفع تصدير تشخيصات Gateway
المحلي. راجع [تصدير التشخيصات](/ar/gateway/diagnostics) لمعرفة سلوك
الموافقة والخصوصية والحزمة المحلية ومحادثة المجموعة.

استخدم `/codex diagnostics [note]` فقط عندما تريد تحديدًا رفع ملاحظات Codex
للخيط المرفق حاليًا بدون حزمة تشخيصات Gateway الكاملة.

## Compaction ومرآة النص

عندما يستخدم النموذج المحدد حاضنة Codex، تكون Compaction الخيط الأصلية
من مسؤولية خادم تطبيق Codex. لا يشغّل OpenClaw Compaction تمهيدية لأدوار Codex،
ولا يستبدل Compaction الخاصة بـ Codex بـ Compaction من محرك السياق، ولا
يلجأ إلى تلخيص OpenClaw أو OpenAI العام عندما يتعذر بدء Compaction الأصلية
في Codex. يحتفظ OpenClaw بمرآة للنص لسجل القناة
والبحث و`/new` و`/reset` والتبديل المستقبلي للنموذج أو الحاضنة.

تبدأ طلبات Compaction الصريحة، مثل `/compact` أو عملية compact يدوية
يطلبها Plugin، تشغيل Compaction الأصلية في Codex باستخدام `thread/compact/start`.
يبقي OpenClaw الطلب واستئجار العميل المشترك مفتوحين حتى يصدر Codex
عنصر الإكمال `contextCompaction` المطابق، ثم يبلّغ عن اكتمال دور Compaction.
إذا تجاوز ذلك الدور النهائي مهلة Compaction المضبوطة،
يطلب OpenClaw مقاطعة دور أصلية. يظل الاستئجار وحاجز Compaction لكل خيط
محتجزين حتى يبلّغ Codex عن حالة نهائية أو يؤكد استدعاء RPC للمقاطعة.
إذا لم يؤكد Codex خلال فترة السماح للمقاطعة، يتقاعد OpenClaw
الاتصال قبل تحرير الحاجز. تفصل الاتصالات البعيدة أيضًا
ربط الخيط المطابق حتى لا يتداخل العمل اللاحق مع دور بعيد غير مؤكد.
تفشل الأدوار الأخرى على اتصال متقاعد ويمكنها إعادة المحاولة على عميل جديد.
يؤدي إغلاق العميل أو إلغاء الطلب أو فشل دور Compaction إلى إرجاع
عملية فاشلة.

عندما يطلب محرك سياق إسقاط تمهيد خيط Codex، يُسقط OpenClaw
أسماء ومعرّفات استدعاءات الأدوات، وأشكال الإدخال، ومحتوى نتائج الأدوات المنقح
في خيط Codex الجديد. ولا ينسخ قيم وسيطات استدعاء الأدوات الخام
إلى ذلك الإسقاط.

تتضمن المرآة موجّه المستخدم، والنص النهائي للمساعد، وسجلات الاستدلال أو الخطة الخفيفة في Codex
عندما يصدرها خادم التطبيق. يسجل OpenClaw
بداية Compaction الأصلية وحالتها النهائية، لكنه لا يعرض
ملخص Compaction قابلًا للقراءة البشرية أو قائمة قابلة للتدقيق بالإدخالات التي احتفظ بها Codex
بعد Compaction.

لأن Codex يملك الخيط الأصلي المعياري، لا يعيد `tool_result_persist`
حاليًا كتابة سجلات نتائج الأدوات الأصلية في Codex. فهو ينطبق فقط عندما
يكتب OpenClaw نتيجة أداة في نص جلسة يملكها OpenClaw.

## الوسائط والتسليم

يواصل OpenClaw امتلاك تسليم الوسائط واختيار موفّر الوسائط. تستخدم الصور
والفيديو والموسيقى وPDF وTTS وفهم الوسائط إعدادات الموفر/النموذج
المطابقة مثل `agents.defaults.imageGenerationModel` و`videoGenerationModel`
و`pdfModel` و`messages.tts`.

يستمر النص والصور والفيديو والموسيقى وTTS والموافقات ومخرجات أداة المراسلة
عبر مسار تسليم OpenClaw العادي. لا يتطلب توليد الوسائط وقت التشغيل القديم.
عندما يصدر Codex عنصر توليد صورة أصليًا يحتوي على `savedPath`، يمرر OpenClaw
ذلك الملف بالضبط عبر مسار وسائط الرد العادي حتى إذا لم يكن لدور Codex
أي نص مساعد.
