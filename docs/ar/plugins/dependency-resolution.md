تبقي OpenClaw عمل تبعيات Plugin في وقت التثبيت/التحديث. لا يشغّل التحميل وقت التشغيل
مديري الحزم، ولا يصلح أشجار التبعيات، ولا يغيّر دليل حزمة OpenClaw.

## تقسيم المسؤوليات

تمتلك حزم Plugin مخطط تبعياتها:

- تبعيات وقت التشغيل تكون في `dependencies` أو
  `optionalDependencies` في حزمة Plugin
- استيرادات SDK/النواة تكون نظيرة أو استيرادات OpenClaw موفّرة
- إضافات التطوير المحلية تجلب تبعياتها المثبّتة مسبقًا
- تثبّت إضافات npm و git في جذور حزم مملوكة لـ OpenClaw

تمتلك OpenClaw دورة حياة Plugin فقط:

- اكتشاف مصدر Plugin
- تثبيت الحزمة أو تحديثها عند الطلب الصريح
- تسجيل بيانات التثبيت الوصفية
- تحميل نقطة دخول Plugin
- الفشل برسالة خطأ قابلة للتنفيذ عندما تكون التبعيات مفقودة

## جذور التثبيت

تستخدم OpenClaw جذورًا ثابتة لكل مصدر:

- تثبّت حزم npm في مشاريع لكل Plugin تحت
  `~/.openclaw/npm/projects/<encoded-package>`
- تنسخ حزم git تحت `~/.openclaw/git`
- تثبيتات local/path/archive تُنسخ أو يُشار إليها دون إصلاح التبعيات

تعمل تثبيتات npm في جذر مشروع كل Plugin بهذا الشكل:

```bash
cd ~/.openclaw/npm/projects/<encoded-package>
npm install --omit=dev --omit=peer --legacy-peer-deps --ignore-scripts --no-audit --no-fund
```

يستخدم `openclaw plugins install npm-pack:<path.tgz>` جذر مشروع npm نفسه لكل Plugin
لأرشيف npm-pack محلي. تقرأ OpenClaw بيانات npm الوصفية من الأرشيف،
وتضيفه إلى المشروع المُدار كتبعية `file:` منسوخة، وتشغّل تثبيت npm المعتاد،
ثم تتحقق من بيانات ملف القفل الوصفية المثبّتة قبل الوثوق بـ Plugin.
هذا مخصص لإثبات قبول الحزمة ومرشح الإصدار عندما يجب أن يتصرف أثر حزمة
محلي مثل أثر السجل الذي يحاكيه.

قد يرفع npm التبعيات الانتقالية إلى `node_modules` الخاص بمشروع كل Plugin
بجانب حزمة Plugin. تفحص OpenClaw جذر المشروع المُدار قبل الوثوق بالتثبيت
وتزيل ذلك المشروع أثناء إلغاء التثبيت، لذلك تبقى تبعيات وقت التشغيل المرفوعة
داخل حدود تنظيف ذلك Plugin.

يمكن لحزم Plugin المنشورة على npm شحن `npm-shrinkwrap.json`. يستخدم npm
ملف القفل القابل للنشر هذا أثناء التثبيت، ويدعمه جذر مشروع npm المُدار من
OpenClaw عبر مسار تثبيت npm المعتاد. يجب أن تتضمن حزم Plugin القابلة للنشر
والمملوكة لـ OpenClaw ملف shrinkwrap محليًا للحزمة مُنشأ من مخطط التبعيات
المنشور لحزمة Plugin تلك:

```bash
pnpm deps:shrinkwrap:generate
pnpm deps:shrinkwrap:check
```

يزيل المولّد `devDependencies` الخاصة بـ Plugin، ويطبق سياسة تجاوزات مساحة العمل،
ويكتب `extensions/<id>/npm-shrinkwrap.json` لكل Plugin يحمل `publishToNpm`.
قد تشحن حزم Plugin التابعة لجهات خارجية shrinkwrap أيضًا؛ لا تشترطه OpenClaw
لحزم المجتمع، لكن npm سيحترمه عند وجوده.

يمكن لحزم Plugin على npm والمملوكة لـ OpenClaw النشر أيضًا مع
`bundledDependencies` صريحة. يضع مسار نشر npm قائمة أسماء تبعيات وقت التشغيل
فوق الحزمة، ويزيل بيانات مساحة العمل الوصفية الخاصة بالتطوير فقط من بيان الحزمة
المنشور، ويشغّل تثبيت npm بلا نصوص برمجية لتبعيات وقت التشغيل المحلية للحزمة،
ثم يحزم أو ينشر أرشيف Plugin مع تضمين ملفات تلك التبعيات. الحزم الثقيلة
بالثنائيات الأصلية، بما في ذلك أوقات تشغيل Codex و ACP، تختار عدم الاشتراك
باستخدام `openclaw.release.bundleRuntimeDependencies: false`؛ لا تزال تلك
الحزم تشحن shrinkwrap الخاص بها، لكن npm يحل تبعيات وقت التشغيل أثناء التثبيت
بدل تضمين كل ثنائي منصة في أرشيف Plugin. لا تحزم حزمة `openclaw` الجذرية
شجرة تبعياتها الكاملة.

تعلن الإضافات التي تستورد `openclaw/plugin-sdk/*` عن `openclaw` كتبعية نظيرة.
لا تسمح OpenClaw لـ npm بتثبيت نسخة سجل منفصلة من حزمة المضيف داخل مشروع مُدار،
لأن حزم المضيف القديمة قد تؤثر في حل npm للتبعيات النظيرة داخل ذلك Plugin.
تتخطى تثبيتات npm المُدارة حل/تجسيد التبعيات النظيرة في npm، وتعيد OpenClaw
تأكيد روابط `node_modules/openclaw` المحلية لـ Plugin للحزم المثبّتة التي تعلن
التبعية النظيرة للمضيف بعد التثبيت أو التحديث.

تنسخ تثبيتات git المستودع أو تحدّثه، ثم تشغّل:

```bash
npm install --omit=dev --ignore-scripts --no-audit --no-fund
```

ثم يُحمّل Plugin المثبّت من دليل تلك الحزمة، لذلك يعمل حل `node_modules`
المحلي للحزمة والأبوي بالطريقة نفسها التي يعمل بها مع حزمة Node عادية.

## الإضافات المحلية

تُعامل الإضافات المحلية كأدلة يتحكم بها المطور. لا تشغّل OpenClaw
`npm install` أو `pnpm install` أو إصلاح التبعيات لها. إذا كان لدى Plugin
محلي تبعيات، فثبّتها في ذلك Plugin قبل تحميله.

يمكن لإضافات TypeScript المحلية التابعة لجهات خارجية استخدام مسار Jiti
الطارئ. تُحمّل إضافات JavaScript المعبأة والإضافات الداخلية المضمّنة عبر
import/require الأصلي بدل Jiti.

## بدء التشغيل وإعادة التحميل

لا يثبّت بدء تشغيل Gateway ولا إعادة تحميل الإعدادات تبعيات Plugin. يقرآن
سجلات تثبيت Plugin، ويحسبان نقطة الدخول، ويحمّلانها.

إذا كانت تبعية مفقودة وقت التشغيل، يفشل تحميل Plugin ويجب أن يوجّه الخطأ
المشغّل إلى إصلاح صريح:

```bash
openclaw plugins update <id>
openclaw plugins install <source>
openclaw doctor --fix
```

يمكن لـ `doctor --fix` تنظيف حالة التبعيات القديمة التي أنشأتها OpenClaw
واسترداد الإضافات القابلة للتنزيل المفقودة من سجلات التثبيت المحلية عندما
تشير إليها الإعدادات. لا يصلح Doctor التبعيات لـ Plugin محلي مثبّت مسبقًا.

## الإضافات المضمّنة

تُشحن الإضافات المضمّنة الخفيفة والحيوية للنواة كجزء من OpenClaw.
يجب إما ألا يكون لديها شجرة تبعيات وقت تشغيل ثقيلة أو أن تُنقل إلى حزمة
قابلة للتنزيل على ClawHub/npm.

للقائمة الحالية المنشأة للإضافات التي تُشحن في حزمة النواة، أو تُثبّت خارجيًا،
أو تبقى مصدرية فقط، راجع [مخزون Plugin](/ar/plugins/plugin-inventory).

يجب ألا تطلب بيانات Plugin المضمّنة تجهيز التبعيات. يجب تغليف وظائف Plugin
الكبيرة أو الاختيارية كـ Plugin عادي وتثبيتها عبر مسار npm/git/ClawHub نفسه
المستخدم مع إضافات الجهات الخارجية.

في نسخ المصدر، تعامل OpenClaw المستودع كمستودع pnpm أحادي. بعد
`pnpm install`، تُحمّل الإضافات المضمّنة من `extensions/<id>` بحيث تكون
تبعيات مساحة العمل المحلية للحزمة متاحة وتُلتقط التعديلات مباشرة. تطوير
نسخة المصدر يدعم pnpm فقط؛ لا يُعد `npm install` العادي في جذر المستودع
طريقة مدعومة لتحضير تبعيات Plugin المضمّنة.

| شكل التثبيت                    | موقع Plugin المضمّن               | مالك التبعيات                                                     |
| -------------------------------- | ------------------------------------- | -------------------------------------------------------------------- |
| `npm install -g openclaw`        | شجرة وقت تشغيل مبنية داخل الحزمة | حزمة OpenClaw وتدفقات تثبيت/تحديث/doctor الصريحة لـ Plugin     |
| نسخة Git مع `pnpm install` | حزم مساحة العمل `extensions/<id>`  | مساحة عمل pnpm، بما في ذلك تبعيات كل حزمة Plugin الخاصة بها |
| `openclaw plugins install ...`   | جذر npm project/git/ClawHub مُدار  | تدفق تثبيت/تحديث Plugin                                       |

## تنظيف القديم

أنشأت إصدارات OpenClaw الأقدم جذور تبعيات Plugin المضمّنة عند بدء التشغيل أو
أثناء إصلاح doctor. يزيل تنظيف doctor الحالي تلك الأدلة والروابط الرمزية
القديمة عند استخدام `--fix`، بما في ذلك جذور `plugin-runtime-deps` القديمة،
وروابط حزم بادئة Node العامة التي تشير إلى أهداف `plugin-runtime-deps`
المشذّبة، وبيانات `.openclaw-runtime-deps*`، و`node_modules` المنشأة لـ Plugin،
وأدلة مرحلة التثبيت، ومخازن pnpm المحلية للحزمة. يزيل postinstall المعبأ أيضًا
تلك الروابط الرمزية العامة قبل تشذيب جذور الأهداف القديمة بحيث لا تترك
الترقيات استيرادات حزم ESM معلّقة.

استخدمت تثبيتات npm الأقدم أيضًا جذر `~/.openclaw/npm/node_modules` مشتركًا.
لا تزال تدفقات التثبيت والتحديث وإلغاء التثبيت وdoctor الحالية تتعرف على ذلك
الجذر المسطح القديم للاسترداد والتنظيف فقط. يجب أن تنشئ تثبيتات npm الجديدة
جذور مشاريع لكل Plugin بدلًا من ذلك.
