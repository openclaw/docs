<Info>
  هذا **دليل للمساهمين** لمطوري نواة OpenClaw. إذا كنت
  تبني plugin خارجيًا، فراجع [بناء Plugins](/ar/plugins/building-plugins)
  بدلًا من ذلك.
</Info>

استخدم هذا عندما يحتاج OpenClaw إلى نطاق جديد مثل توليد الصور أو
توليد الفيديو أو أي مجال ميزات مستقبلي مدعوم من مزوّد.

القاعدة:

- plugin = حدّ الملكية
- capability = عقد نواة مشترك

هذا يعني أنه يجب ألا تبدأ بتوصيل مزوّد مباشرةً إلى قناة أو
أداة. ابدأ بتعريف capability.

## متى يجب إنشاء capability

أنشئ capability جديدة عندما تكون جميع النقاط التالية صحيحة:

1. يمكن لأكثر من مزوّد واحد تنفيذها بشكل معقول
2. يجب أن تستهلكها القنوات أو الأدوات أو plugins الميزات من دون الاهتمام
   بالمزوّد
3. تحتاج النواة إلى امتلاك سلوك التراجع أو السياسة أو الإعداد أو التسليم

إذا كان العمل خاصًا بمزوّد فقط ولا يوجد بعد عقد مشترك، فتوقّف وحدّد
العقد أولًا.

## التسلسل القياسي

1. حدّد عقد النواة المكتوب Typed.
2. أضف تسجيل plugin لهذا العقد.
3. أضف مساعد وقت تشغيل مشتركًا.
4. وصّل plugin مزوّد حقيقيًا واحدًا كإثبات.
5. انقل مستهلكي الميزات/القنوات إلى مساعد وقت التشغيل.
6. أضف اختبارات العقد.
7. وثّق الإعداد الموجّه للمشغّل ونموذج الملكية.

## ما الذي يوضع في أي مكان

النواة:

- أنواع الطلب/الاستجابة
- سجل المزود + آلية الحل
- سلوك التراجع
- مخطط الإعداد بالإضافة إلى بيانات التوثيق الوصفية `title` / `description` المُمرَّرة على العقد المتداخلة والكائنات العامة والعناصر المصفوفية وعقد التركيب
- سطح مساعد وقت التشغيل

plugin الخاص بالمزوّد:

- استدعاءات API الخاصة بالمزوّد
- معالجة المصادقة الخاصة بالمزوّد
- تطبيع الطلبات الخاص بالمزوّد
- تسجيل تنفيذ capability

plugin الخاص بالميزة/القناة:

- يستدعي `api.runtime.*` أو مساعد `plugin-sdk/*-runtime` المطابق
- لا يستدعي تنفيذ مزوّد مباشرةً أبدًا

## حدود Provider وHarness

استخدم hooks الخاصة بالـ provider عندما يكون السلوك تابعًا لعقد مزوّد النموذج
بدلًا من حلقة الوكيل العامة. تشمل الأمثلة معاملات الطلب الخاصة بالمزوّد بعد
اختيار النقل، وتفضيل ملف تعريف المصادقة، وتراكبات prompt، وتوجيه التراجع
اللاحق بعد فشل النموذج/الملف التعريفي.

استخدم hooks الخاصة بـ harness للوكيل عندما يكون السلوك تابعًا لوقت التشغيل الذي
ينفّذ الدور. يمكن لـ harnesses تصنيف نتائج المحاولة الناجحة ولكن غير القابلة
للاستخدام، مثل الردود الفارغة أو ردود الاستدلال فقط أو التخطيط فقط، بحيث
تتمكن سياسة التراجع الخارجية للنموذج من اتخاذ قرار إعادة المحاولة.

أبقِ كلا الحدّين ضيقين:

- النواة تملك سياسة إعادة المحاولة/التراجع
- plugins الخاصة بالـ provider تملك تلميحات الطلب/المصادقة/التوجيه الخاصة بالمزوّد
- plugins الخاصة بالـ harness تملك تصنيف المحاولة الخاص بوقت التشغيل
- plugins الخارجية تعيد تلميحات، لا تعديلات مباشرة على حالة النواة

## قائمة الملفات

بالنسبة إلى capability جديدة، توقّع أن تلمس هذه المناطق:

- `src/<capability>/types.ts`
- `src/<capability>/...registry/runtime.ts`
- `src/plugins/types.ts`
- `src/plugins/registry.ts`
- `src/plugins/captured-registration.ts`
- `src/plugins/contracts/registry.ts`
- `src/plugins/runtime/types-core.ts`
- `src/plugins/runtime/index.ts`
- `src/plugin-sdk/<capability>.ts`
- `src/plugin-sdk/<capability>-runtime.ts`
- حزمة plugin مضمّنة واحدة أو أكثر
- الإعداد/التوثيق/الاختبارات

## مثال: توليد الصور

يتبع توليد الصور البنية القياسية:

1. تعرّف النواة `ImageGenerationProvider`
2. تكشف النواة `registerImageGenerationProvider(...)`
3. تكشف النواة `runtime.imageGeneration.generate(...)`
4. تسجّل plugins `openai` و`google` و`fal` و`minimax` تنفيذات مدعومة من مزوّد
5. يمكن للمزوّدين المستقبليين تسجيل العقد نفسه من دون تغيير القنوات/الأدوات

مفتاح الإعداد منفصل عن توجيه تحليل الرؤية:

- `agents.defaults.imageModel` = تحليل الصور
- `agents.defaults.imageGenerationModel` = توليد الصور

أبقِ هذين منفصلين حتى يظل التراجع والسياسة واضحين.

## قائمة المراجعة

قبل شحن capability جديدة، تحقّق من:

- عدم وجود قناة/أداة تستورد كود مزوّد مباشرةً
- أن مساعد وقت التشغيل هو المسار المشترك
- أن اختبار عقد واحدًا على الأقل يثبت الملكية المضمّنة
- أن توثيق الإعداد يسمّي النموذج الجديد/مفتاح الإعداد الجديد
- أن توثيق plugin يشرح حدّ الملكية

إذا تجاوز PR طبقة capability وثبّت سلوك مزوّد بشكل مباشر داخل
قناة/أداة، فأعده وحدّد العقد أولًا.
