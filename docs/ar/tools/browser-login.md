# تسجيل دخول المتصفح + النشر على X/Twitter

## تسجيل الدخول اليدوي (موصى به)

عندما يتطلب موقع ما تسجيل الدخول، **قم بتسجيل الدخول يدويًا** في ملف تعريف المتصفح على **المضيف** (متصفح openclaw).

لا تعطِ النموذج بيانات اعتمادك. فعمليات تسجيل الدخول الآلية غالبًا ما تفعّل دفاعات مكافحة الروبوتات وقد تؤدي إلى قفل الحساب.

للعودة إلى وثائق المتصفح الرئيسية: [المتصفح](/ar/tools/browser).

## أي ملف تعريف Chrome يُستخدم؟

يتحكم OpenClaw في **ملف تعريف Chrome مخصص** (اسمه `openclaw` مع واجهة مستخدم مائلة إلى البرتقالي). وهو منفصل عن ملف تعريف المتصفح اليومي الخاص بك.

بالنسبة إلى استدعاءات أدوات المتصفح من الوكيل:

- الخيار الافتراضي: يجب على الوكيل استخدام متصفح `openclaw` المعزول الخاص به.
- استخدم `profile="user"` فقط عندما تكون الجلسات المسجلة الدخول الموجودة مهمة ويكون المستخدم أمام الحاسوب للنقر/الموافقة على أي مطالبة إرفاق.
- إذا كان لديك عدة ملفات تعريف لمتصفح المستخدم، فحدّد الملف صراحةً بدلًا من التخمين.

طريقتان سهلتان للوصول إليه:

1. **اطلب من الوكيل فتح المتصفح** ثم سجّل الدخول بنفسك.
2. **افتحه عبر CLI**:

```bash
openclaw browser start
openclaw browser open https://x.com
```

إذا كان لديك عدة ملفات تعريف، فمرر `--browser-profile <name>` (الافتراضي هو `openclaw`).

## X/Twitter: التدفق الموصى به

- **القراءة/البحث/الخيوط:** استخدم متصفح **المضيف** (تسجيل دخول يدوي).
- **نشر التحديثات:** استخدم متصفح **المضيف** (تسجيل دخول يدوي).

## Sandbox والوصول إلى متصفح المضيف

تكون جلسات المتصفح داخل Sandbox **أكثر عرضة** لإطلاق اكتشاف الروبوتات. وبالنسبة إلى X/Twitter (والمواقع الصارمة الأخرى)، فضّل متصفح **المضيف**.

إذا كان الوكيل داخل sandbox، فإن أداة المتصفح تستخدم sandbox افتراضيًا. وللسماح بالتحكم من المضيف:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        browser: {
          allowHostControl: true,
        },
      },
    },
  },
}
```

ثم استهدف متصفح المضيف:

```bash
openclaw browser open https://x.com --browser-profile openclaw --target host
```

أو عطّل sandboxing للوكيل الذي ينشر التحديثات.
