يمكن لـ OpenClaw منع الوكلاء من التعثر في أنماط متكررة لاستدعاءات الأدوات.
يكون الحارس **معطلًا افتراضيًا**.

فعّله فقط حيث يلزم، لأنه قد يحظر استدعاءات متكررة مشروعة عند استخدام إعدادات صارمة.

## لماذا يوجد هذا

- اكتشاف التسلسلات المتكررة التي لا تحقق تقدمًا.
- اكتشاف حلقات عدم وجود نتائج عالية التكرار (الأداة نفسها، والمدخلات نفسها، والأخطاء المتكررة).
- اكتشاف أنماط محددة للاستدعاءات المتكررة لأدوات الاستطلاع المعروفة.

## كتلة التكوين

الإعدادات الافتراضية العامة:

```json5
{
  tools: {
    loopDetection: {
      enabled: false,
      historySize: 30,
      warningThreshold: 10,
      criticalThreshold: 20,
      globalCircuitBreakerThreshold: 30,
      detectors: {
        genericRepeat: true,
        knownPollNoProgress: true,
        pingPong: true,
      },
    },
  },
}
```

تجاوز لكل وكيل (اختياري):

```json5
{
  agents: {
    list: [
      {
        id: "safe-runner",
        tools: {
          loopDetection: {
            enabled: true,
            warningThreshold: 8,
            criticalThreshold: 16,
          },
        },
      },
    ],
  },
}
```

### سلوك الحقول

- `enabled`: المفتاح الرئيسي. تعني `false` عدم تنفيذ أي اكتشاف للحلقات.
- `historySize`: عدد استدعاءات الأدوات الحديثة المحتفظ بها للتحليل.
- `warningThreshold`: العتبة قبل تصنيف النمط على أنه تحذيري فقط.
- `criticalThreshold`: العتبة لحظر أنماط الحلقات المتكررة.
- `globalCircuitBreakerThreshold`: عتبة قاطع عدم التقدم العام.
- `detectors.genericRepeat`: يكتشف أنماط تكرار الأداة نفسها + المعاملات نفسها.
- `detectors.knownPollNoProgress`: يكتشف الأنماط المعروفة الشبيهة بالاستطلاع دون تغيّر في الحالة.
- `detectors.pingPong`: يكتشف أنماط التناوب ذهابًا وإيابًا.

بالنسبة إلى `exec`، تقارن فحوصات عدم التقدم نتائج الأوامر المستقرة وتتجاهل بيانات تعريف وقت التشغيل المتغيرة مثل المدة، وPID، ومعرّف الجلسة، ودليل العمل.
عندما يكون معرّف تشغيل متاحًا، لا يُقيّم سجل استدعاءات الأدوات الحديثة إلا داخل ذلك التشغيل، بحيث لا ترث دورات Heartbeat المجدولة والتشغيلات الجديدة أعداد حلقات قديمة من تشغيلات سابقة.

## الإعداد الموصى به

- بالنسبة إلى النماذج الأصغر، ابدأ بـ `enabled: true` واترك الإعدادات الافتراضية دون تغيير. نادرًا ما تحتاج النماذج الرائدة إلى اكتشاف الحلقات ويمكن أن تتركه معطلًا.
- أبقِ العتبات مرتبة على النحو `warningThreshold < criticalThreshold < globalCircuitBreakerThreshold`.
- إذا حدثت نتائج موجبة كاذبة:
  - ارفع `warningThreshold` و/أو `criticalThreshold`
  - ارفع `globalCircuitBreakerThreshold` (اختياريًا)
  - عطّل فقط الكاشف الذي يسبب المشكلات
  - قلّل `historySize` للحصول على سياق تاريخي أقل صرامة

## حارس ما بعد Compaction

عندما يُكمل المشغّل إعادة محاولة تلقائية بعد Compaction (بعد فيضان السياق)، فإنه يسلّح حارسًا لنافذة قصيرة يراقب استدعاءات الأدوات القليلة التالية. إذا أصدر الوكيل ثلاثية `(toolName, args, result)` _نفسها_ عدة مرات داخل تلك النافذة، يستنتج الحارس أن Compaction لم يكسر الحلقة ويُجهض التشغيل بخطأ `compaction_loop_persisted`.

هذا مسار شيفرة منفصل عن كواشف `tools.loopDetection` العامة. ويمكن تكوينه بشكل مستقل:

```json5
{
  tools: {
    loopDetection: {
      enabled: true, // existing master switch; set false to disable loop guards
      postCompactionGuard: {
        windowSize: 3, // default: 3
      },
    },
  },
}
```

- `windowSize`: عدد استدعاءات الأدوات بعد Compaction التي يبقى الحارس خلالها مسلحًا _وكذلك_ عدد ثلاثيات (الأداة، المعاملات، النتيجة) المتطابقة الذي يؤدي إلى الإجهاض.

لا يُجهض الحارس أبدًا عندما تكون النتائج تتغير، بل فقط عندما تكون النتائج متطابقة بايتًا عبر النافذة. وهو ضيق النطاق عمدًا: لا يعمل إلا في الأثر الفوري لإعادة محاولة بعد Compaction.

## السجلات والسلوك المتوقع

عند اكتشاف حلقة، يبلغ OpenClaw عن حدث حلقة ويحظر دورة الأداة التالية أو يخففها بحسب الشدة.
يحمي ذلك المستخدمين من استهلاك الرموز غير المنضبط والتوقفات، مع الحفاظ على الوصول العادي إلى الأدوات.

- فضّل التحذير والتثبيط المؤقت أولًا.
- لا تُصعّد إلا عندما تتراكم أدلة متكررة.

## ملاحظات

- يُدمج `tools.loopDetection` مع تجاوزات مستوى الوكيل.
- يتجاوز تكوين كل وكيل القيم العامة أو يوسعها بالكامل.
- إذا لم يوجد أي تكوين، تبقى الحواجز الوقائية متوقفة.
