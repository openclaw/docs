# `openclaw promos`

ClawHub で公開されているプロモーションモデルのオファーを見つけて取得します。プロモーションを取得すると、プロバイダー（必要に応じて認証と Plugin）が設定され、プロモーションのモデルが登録されます。オンボーディングを再実行する必要はなく、明示的に指定しない限りデフォルトモデルも変更されません。

関連項目:

- デフォルトモデルとフォールバック: [モデル](/ja-JP/cli/models)
- プロバイダー認証の設定: [はじめに](/ja-JP/start/getting-started)

## コマンド

```bash
openclaw promos list
openclaw promos claim <slug>
openclaw promos claim <slug> --api-key <key> --set-default
```

## `openclaw promos list`

現在有効なプロモーションを、そのモデル、推奨デフォルト、残り時間、正確な取得コマンドとともに一覧表示します。`--json` は未加工のペイロードを出力します。

## `openclaw promos claim <slug>`

有効なプロモーションを取得します:

1. ClawHub からプロモーションを取得し、有効期間内であることを確認します。
2. プロモーションのプロバイダー、認証方式、宣言された Plugin パッケージが、インストール済みの OpenClaw バージョンと互換性があるか検証します。不明な ID やパッケージの不一致は拒否されます。プロモーションによって、CLI が既に実行方法を把握していないものを実行することは決してありません。
3. 既存のプロバイダー認証情報がある場合は再利用します。ない場合は、プロバイダーの通常の認証フローを案内します（最初に、無料キーを取得するためのプロモーションの登録 URL を表示します）。`--api-key <key>` を指定すると、プロンプトなしで API キー認証を完了します。これは `openclaw onboard` の非対話型フラグと同じ動作です。キーをコマンドラインに含めないようにするには、代わりにプロバイダーの環境変数（例: `OPENROUTER_API_KEY`）をエクスポートします。既存の環境変数の認証情報は自動的に検出されるため、フラグは不要です。
4. プロモーションのモデルをエイリアスとともに登録します。既存のエイリアスが上書きされることはありません。
5. プロモーションの推奨モデルをデフォルトとして設定するか確認します。`--set-default` を指定すると確認を省略します。それ以外の場合、デフォルト設定は一切変更されません。

プロモーションの有効期間が終了すると、プロバイダーは無料モデルの提供を停止しますが、設定と認証情報は変更されません。`openclaw models set <model>` を使用すれば、いつでも元のモデルに切り替えられます。

## `models list` での受動的な検出

`openclaw models list` は、ClawHub に直接問い合わせなくてもプロモーションを表示します:

- まだ設定していないモデルを含む有効なオファーは、表の下にある「プロモーションで利用可能」グループに、それぞれの取得コマンドとともに表示されます。
- `promos claim` を使用して登録したモデルには `promo` タグが付き、オファーの有効期間が過ぎると `promo ended` に変わります。
- 新しいオファーが初めて検出されると、`openclaw promos list` を案内する通知が一度だけ表示されます。既に一覧表示または取得したオファーが再度通知されることはありません。

これは、ClawHub でホストされているプロモーションフィードのローカルキャッシュを読み取ります（通常は条件付きリクエストにより 1 日に 1 回更新されますが、キャッシュされたスナップショットの有効期限が切れた場合はそれより早く更新されます。更新に失敗した場合は通知せずにスキップされます）。古いキャッシュの更新は最大 2.5 秒しか待機せず、一覧表示を妨げることはありません。`--json` と `--plain` の出力は機械処理しやすい状態に保たれ、プロモーションのセクションや通知は含まれません。取得時には常に ClawHub の稼働中の API に対して再検証されるため、キャッシュにまだ表示されていても、予定より早く取り下げられたオファーは拒否されます。
