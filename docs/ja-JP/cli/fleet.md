# `openclaw fleet`

`openclaw fleet` は、**セル**と呼ばれる完全な OpenClaw インスタンスを管理します。各セルには、それぞれ独自の Gateway、状態、認証情報、チャネルアカウント、コンテナ、およびループバック専用ホストポートがあります。テナントの信頼境界ごとに 1 つのセルを使用してください。敵対的なマルチテナント境界として 1 つの共有 Gateway を使用しないでください。

Fleet は**実験的機能**です。コマンド名、フラグ、出力形式、およびコンテナプロファイルは、非推奨期間なしでリリース間に変更される可能性があります。

Fleet は Docker と Podman をサポートします。デフォルトのイメージは `ghcr.io/openclaw/openclaw:latest` です。

Fleet は Linux および macOS ホストでテストされています。Windows ホストは現在テストされていません。

## クイックスタート

```bash
openclaw fleet create acme
openclaw fleet status acme
openclaw fleet list
```

`fleet create` は、生成された Gateway トークンをセルの URL とともに一度だけ表示します。トークンをすぐに保存してから、各テナントのチャネルアカウントをそのテナントのセル内で設定してください。

## テナント ID

テナント ID は次の正規表現に一致する必要があります。

```text
^[a-z0-9](?:[a-z0-9-]{0,38}[a-z0-9])?$
```

使用できるのは、1～40 文字の小文字、数字、および内部のハイフンです。ID の先頭と末尾は文字または数字である必要があります。大文字、アンダースコア、スラッシュ、ドット、空白、および `../acme` のようなトラバーサル文字列は拒否されます。

ID はコンテナ名の一部になります: `openclaw-cell-<tenant>`。

## `fleet create`

セルを作成して起動します。

```bash
openclaw fleet create acme
```

固定ポートで Podman セルを作成し、起動はしません。

```bash
openclaw fleet create acme \
  --runtime podman \
  --port 19125 \
  --no-start
```

`--env` を繰り返して、テナント固有の環境変数を渡します。

```bash
openclaw fleet create acme \
  --env TZ=America/Los_Angeles \
  --env OPENCLAW_DISABLE_BONJOUR=1
```

環境変数のキーには文字、数字、アンダースコアを使用でき、数字で始めることはできません。Fleet は保護されたランタイム環境ファイルを通じて値を渡すため、値は単一行である必要があります。Fleet は、[ストレージとコンテナのレイアウト](#storage-and-container-layout)に記載されている、管理対象のコンテナパス変数および Gateway トークン変数を上書きしようとする操作を拒否します。

### 作成オプション

| オプション                    | デフォルト                               | 説明                                                                                    |
| ------------------------- | ------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `--image <ref>`           | `ghcr.io/openclaw/openclaw:latest`    | セルのコンテナイメージ。                                                                  |
| `--runtime <runtime>`     | `docker`                              | コンテナ CLI: `docker` または `podman`。                                                           |
| `--port <number>`         | `19100` 以降から自動的に割り当て  | ループバックホストポート。明示的に選択したポートは、別の登録済みセルに属していてはなりません。    |
| `--memory <value>`        | `2g`                                  | Docker/Podman 構文によるコンテナのメモリ上限。                                                |
| `--cpus <value>`          | `2`                                   | コンテナの CPU 上限。                                                                           |
| `--disk <size>`           | なし                                  | ストレージバックエンドがクォータをサポートする場合に、コンテナの書き込み可能レイヤーを制限します。                     |
| `--network <mode>`        | `bridge`                              | 送信ネットワークモード: `bridge` または `internal`。                                                 |
| `--pids-limit <number>`   | `512`                                 | コンテナ内の最大プロセス数。                                                  |
| `--env <KEY=VALUE>`       | なし                                  | 環境変数をセルに渡します。複数の値を指定する場合は繰り返します。                          |
| `--gateway-token <value>` | ランダムな 32 文字の 16 進数トークン | 生成する代わりに、指定された Gateway トークンを使用します。[トークンの取り扱い](#token-handling)を参照してください。 |
| `--no-start`              | セルを起動                           | コンテナを起動せずに作成します。                                                      |
| `--json`                  | 人間が読みやすい出力                 | 機械可読形式の出力を表示します。                                                                 |

自動割り当てでは、`19100` 以上でレジストリに未使用の最初のポートが選択されます。Fleet は、重複するテナント ID、および別のセルにすでに割り当てられている明示的なポートを拒否します。

イメージ参照は、単一のコンテナランタイム引数として渡されます。イメージが Docker または Podman のオプションとして解釈されることを防ぐため、空の参照、および `-` で始まる値は拒否されます。

選択した Docker または Podman エンドポイントはローカルである必要があります。Fleet は、ポートの予約やローカル状態の作成を行う前に、リモート Docker コンテキスト、`DOCKER_HOST` エンドポイント、およびリモート Podman サービスを拒否します。リモートのセルホストはサポートされていません。

Fleet が新しいセルを起動すると、create はその Gateway が `/healthz` に応答するまで約 1 分間待機します。セルが正常な状態にならない場合、Fleet は `fleet status`、`fleet logs`、または明示的な削除に備えて、そのコンテナとレジストリ行をそのまま残します。`--no-start` はこのヘルスゲートをスキップします。正常でない新しいセルの生成済み Gateway トークンは失われず、コンテナ環境（`docker|podman inspect`）に残ります。また、そのセルはまだトラフィックを処理していないため、`fleet rm --force` の後に新しく create を実行する方法も常に安全です。

### ダイジェストによる固定

create と upgrade は、`--image ghcr.io/openclaw/openclaw@sha256:<digest>` のようなダイジェストで固定されたイメージ参照を受け付けます。Fleet はイメージ参照をそのまま Docker または Podman に渡すため、オペレーターは変動するタグではなく不変のイメージバイトにセルを固定できます。

create の結果には、テナント ID、コンテナ名、ホストポート、Gateway トークン、およびローカル URL が含まれます。JSON 出力の場合でも、結果にはトークンが含まれるため、機密情報を含むものとして扱ってください。

### ディスク上限

`--disk` は、コンテナの書き込み可能レイヤーのみを制限します。バインドマウントされたテナントごとの状態ディレクトリと認証ディレクトリは、引き続きホストストレージを使用します。これらのディレクトリにも厳格な上限が必要な場合は、ホストファイルシステムのプロジェクトクォータを使用してください。

| ランタイム／ストレージバックエンド | `--disk` のサポート                                                             |
| ----------------------- | ---------------------------------------------------------------------------- |
| XFS 上の Docker overlay2  | XFS の `pquota` マウントオプションが必要です。                                      |
| Docker btrfs または zfs     | ストレージドライバーによってサポートされます。                                             |
| Podman overlay          | XFS バッキングストレージが必要です。                                                |
| その他のバックエンド          | デーモンエラーと Fleet のバックエンドガイダンスを伴って、コンテナの作成に失敗します。 |

### 送信ポリシー

| モード       | Docker                                                                                                | Podman                                                                              |
| ---------- | ----------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| `bridge`   | サポートされています。デフォルトでは送信通信は制限されません。                                                | サポートされています。デフォルトでは送信通信は制限されません。                              |
| `internal` | Docker は内部ネットワーク上で公開されたループバック Gateway ポートを維持しないため、拒否されます。 | サポートされています。送信通信がブロックされている間も、ループバック Gateway は公開されたままです。 |

Docker では、ブリッジモードを維持し、`DOCKER-USER` チェーンなどのホストファイアウォールルールを使用して送信ポリシーを適用してください。

## `fleet list`

テナント ID 順にセルを一覧表示します。

```bash
openclaw fleet list
openclaw fleet ls
openclaw fleet list --json
```

表には次の項目が含まれます。

| 列    | 意味                                                                                                                                                                                                                                                                               |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `tenant`  | テナント ID。                                                                                                                                                                                                                                                                            |
| `state`   | Docker または Podman の検査から取得した現在のコンテナ状態。`unknown` は、ランタイムが利用できなかったか、セル名と同じ名前のコンテナが存在するものの、その Fleet 所有権ラベルがレジストリレコードと一致しないことを意味します（衝突または改ざんの兆候です。操作する前に手動で調査してください）。 |
| `port`    | セルの Gateway にマッピングされたループバックホストポート。                                                                                                                                                                                                                                        |
| `image`   | 記録されているコンテナイメージ。                                                                                                                                                                                                                                                             |
| `created` | セルの作成時刻。                                                                                                                                                                                                                                                                   |

Docker または Podman が利用できない場合でも、レジストリ行は引き続き表示されます。現在の状態のみが `unknown` になります。

## `fleet status`

1 つのセルを調査します。

```bash
openclaw fleet status acme
openclaw fleet status acme --json
```

status は、Fleet レジストリ行、現在のコンテナ検査、および次のエンドポイントへの短いベストエフォートリクエストを組み合わせます。

```text
http://127.0.0.1:<host-port>/healthz
```

ヘルス結果は `ok`、`failed`、または `skipped` です。`/healthz` は Gateway が稼働していることを示しますが、設定済みのすべてのチャネルや Plugin が完全に準備できていることを示すものではありません。確認可能なローカルエンドポイントがない場合、プローブはスキップされます。

## `fleet logs`

セルのコンテナログをターミナルへ直接ストリーミングします。

```bash
openclaw fleet logs acme
openclaw fleet logs acme --follow
openclaw fleet logs acme --tail 200
openclaw fleet logs acme --since 10m
```

Fleet はログを読み取る前に登録済みコンテナの所有権ラベルを検証するため、想定されたセル名を使用している外部コンテナからの読み取りを拒否します。ストリームは検査済みのコンテナ ID に固定されるため、同時に置換が発生しても、より新しい世代へリダイレクトされることはありません。オペレーターによる停止をコマンド失敗として扱わずに `--follow` を終了するには、Ctrl-C を押します。ログ出力は秘匿化フィルターを通過し、ターミナルに到達する前にセルの現在の Gateway トークンが `<redacted>` に置き換えられます。

コンテナログは生の stdout/stderr ストリームであるため、`fleet logs` には `--json` モードがありません。スクリプトでは、`--tail` で出力を制限し、通常のシェルリダイレクトまたはパイプラインを使用してください。

## `fleet start`、`fleet stop`、`fleet restart`

記録されたランタイムを使用して既存のセルを制御します。

```bash
openclaw fleet start acme
openclaw fleet stop acme
openclaw fleet restart acme
```

これらのコマンドは、登録済みのコンテナ名を対象に動作します。テナントが不明な場合、または記録されたランタイムで操作を実行できない場合は失敗します。

## `fleet upgrade`

記録されたイメージを再プルし、セルコンテナを置き換えます。

```bash
openclaw fleet upgrade acme
```

セルを別のイメージに移行します。

```bash
openclaw fleet upgrade acme --image ghcr.io/openclaw/openclaw:<version>
```

アップグレードでは、対象イメージをプルし、既存のコンテナとセルごとのネットワークを調査し、コンテナを停止して削除した後、再作成して起動します。置き換え後も、同じホストポート、データディレクトリ、セルごとのブリッジネットワーク、ランタイムプロファイル、リソース制限、再起動ポリシー、Fleet 管理の環境、および `--env` で最初に指定された値が維持されます。マウントされた状態はコンテナの置き換え後も保持されますが、イメージのデフォルト環境は対象イメージに応じて変わる場合があります。

置き換えは、その Gateway がセルの loopback ポート上で `/healthz` に応答した後にのみ確定されます。これは公式の compose ファイルが使用するヘルスコントラクトと一致します。置き換え後のコンテナが終了する、クラッシュループする、または約 1 分以内に正常状態にならない場合、そのコンテナは削除され、以前のコンテナが復元されます。そのため、壊れたイメージによって動作中のセルが停止することはありません。

Gateway トークンは意図的に Fleet レジストリへ保存されません。古いコンテナを削除する前に、Fleet はその環境を読み取り、`OPENCLAW_GATEWAY_TOKEN` を置き換え後のコンテナへ引き継ぎます。管理下のほかの場所にトークンが存在しない場合は、アップグレード前に古いコンテナを手動で削除しないでください。

## `fleet backup` と `fleet restore`

停止中のセルを 1 つバックアップします。

```bash
openclaw fleet stop acme
openclaw fleet backup acme --out ./acme.tgz
```

そのアーカイブを登録済みのセルへ復元します。

```bash
openclaw fleet restore acme --from ./acme.tgz
```

これらはホストオペレーター権限を必要とするコマンドです。アーカイブにはテナントの状態と認証シークレットが含まれ、モード `0600` で作成されるため、認証情報と同様に保管する必要があります。SQLite の状態を一貫して取得するため、実行中のセルはバックアップできません。復元では、`--force` が指定されていない限り実行中のセルを拒否し、そのテナントの状態のみを置き換え、Gateway トークンをローテーションして、新しいトークンを一度だけ表示します。Fleet が一度にバックアップするのは 1 つのテナントです。全テナントのバックアップは別のオペレーター操作です。

復元には既存の停止中コンテナが必要です。調査されたそのランタイムプロファイルから、置き換え後の制限、ユーザーマッピング、環境の由来、イメージが取得されるためです。登録済みコンテナが外部操作で削除されている場合は、まず `--purge-data` を付けずに `fleet rm <tenant> --force` を実行し、目的のイメージと `--no-start` を使用してセルを再作成してから、復元を再試行してください。最初の削除では、テナントの両方のデータディレクトリがそのまま保持されます。

どちらのコマンドも、アーカイブまたは展開されるファイルデータを制限する `--max-bytes <bytes>` を受け付けます。また、メタデータのみのアーカイブ爆弾によるホスト inode の枯渇を防ぎ、受け入れられたすべてのバックアップを復元可能に保つため、どちらにもアーカイブパスセグメント数 100 万という同じ固定上限が適用されます。バックアップは `--out <path>` を受け付け、両方のコマンドが `--json` をサポートします。

アーカイブに含まれるのは通常ファイルとディレクトリのみです。バックアップではシンボリックリンク、ハードリンク、ソケット、デバイスノードをたどったり保存したりすることはなく、スキップした数が結果に報告されます。復元では、それ以外のエントリ種別を含むアーカイブを拒否します。ワークスペースの `node_modules` など、再作成可能なシンボリックリンクツリーは、復元後にセル内で再インストールする必要があります。

## `fleet doctor`

ランタイムやファイルシステムの状態を変更せずに、すべてのセルまたは 1 つのテナントを監査します。

```bash
openclaw fleet doctor
openclaw fleet doctor acme --json
```

Doctor は、ランタイムのローカル性、所有権ラベル、正常性、ハードニング、リソース制限、loopback ポートのバインド、トークンの存在、ネットワークの所有権とエグレスモード、非公開状態ディレクトリの権限を確認します。警告には、停止中のセルや所有権の相違が記載されます。いずれかの検査結果が失敗すると、プロセスの終了コードは 0 以外になります。

## `fleet rm`

テナントデータを保持したまま、停止中のセルをランタイムとレジストリから削除します。

```bash
openclaw fleet rm acme
```

実行中のコンテナには `--force` が必要です。

```bash
openclaw fleet rm acme --force
```

セルのデータも完全に削除します。

```bash
openclaw fleet rm acme --purge-data --force
```

Fleet はセル専用のブリッジネットワークを削除する前に、セルコンテナを削除します。`--purge-data` には `--force` が必要です。再帰的に削除する前に、Fleet は Fleet 所有の両方のルートと、テナントごとの両方のディレクトリを解決します。各対象は想定されたテナントの末端そのものであり、ルートの厳密な配下に存在し、シンボリックリンクではないことが必要です。これらの包含チェックにより、破損したレジストリパスやテナントをまたぐシンボリックリンクによって、削除先が別の場所へ変更されるのを防ぎます。

想定されたテナントディレクトリがすでに存在しない場合でも、パージは再試行できます。これにより、まだ存在するディレクトリに対するパスチェックを緩和することなく、ファイルシステムの部分的な障害後に、後続の実行でクリーンアップを完了できます。

## ストレージとコンテナのレイアウト

セルの状態と認証プロファイルの暗号化キーには、アクティブな OpenClaw 状態ディレクトリ配下にある、テナントごとの個別のホストパスを使用します。

```text
<state-dir>/fleet/cells/<tenant>/
<state-dir>/fleet/auth-profile-secrets/<tenant>/
```

最初のディレクトリは `/home/node/.openclaw` にマウントされます。2 番目は `/home/node/.config/openclaw` にマウントされ、公式 Docker セットアップの暗号化キーマウントと一致します。そのため、暗号化キーは通常の状態マウント配下に公開されず、セル状態ディレクトリのみをバックアップまたは共有する場合にも含まれません。どちらのディレクトリも通常の削除やアップグレード後に保持されます。`fleet rm --purge-data --force` は個別の包含チェック後に両方を削除します。

初回起動前に、Fleet は `gateway.mode=local`、トークン認証、LAN コンテナバインド、および割り当てられたホストポート用の Control UI オリジンを使用してセル設定を初期化します。トークン値はその設定には書き込まれず、コンテナ環境に保持されます。

Fleet は、次の環境値を使用して公式イメージのコンテナパスを固定します。

| 変数                 | コンテナの値                      |
| ------------------------ | ------------------------------------ |
| `HOME`                   | `/home/node`                         |
| `OPENCLAW_HOME`          | `/home/node`                         |
| `OPENCLAW_STATE_DIR`     | `/home/node/.openclaw`               |
| `OPENCLAW_CONFIG_PATH`   | `/home/node/.openclaw/openclaw.json` |
| `OPENCLAW_WORKSPACE_DIR` | `/home/node/.openclaw/workspace`     |
| `OPENCLAW_GATEWAY_TOKEN` | 生成または指定されたセルトークン     |

公式イメージのデフォルトは、UID 1000 の非 root ユーザー `node` です。Fleet は、非公開の `0700` バインドマウントを誰でもアクセス可能にすることなく、書き込み可能な状態に保ちます。rootful Docker は、実行元の非 root UID と GID でセルを実行します。rootless Docker はコンテナ UID 0 を使用し、これはデーモンのユーザー名前空間内で、実行元の権限を持たないホストユーザーにマッピングされます。Podman は、実行元の UID と GID とともに `keep-id` を使用します。Fleet 自体が rootful ランタイムに対して root として実行される場合は、イメージのユーザーを維持し、初期マウントファイルを UID/GID 1000 に割り当てます。

SELinux ホストでは、Docker と Podman のマウントに非公開の `:Z` 再ラベルが適用されます。セルデータを復元または移動する場合は、実効コンテナユーザーがバインドマウントされたパスへ書き込める状態を維持してください。このプロファイルは rootless に適していますが、Docker または Podman はホスト上ですでに rootless 動作用に設定されている必要があります。Fleet が rootful デーモンを rootless に変換することはありません。

## セキュリティプロファイル

Fleet はすべてのセルに次のプロファイルを適用します。

| 制御項目              | 適用されるプロファイル                                      | 理由                                                                                    |
| -------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Linux ケーパビリティ   | `--cap-drop=ALL`                                     | Gateway は Node.js プロセスであり、追加の Linux ケーパビリティは必要ありません。                |
| 権限昇格 | `--security-opt no-new-privileges`                   | setuid または setgid バイナリを介してプロセスが権限を取得するのを防ぎます。          |
| init プロセス         | `--init`                                             | 子孫プロセスを回収し、コンテナのライフサイクルシグナルを転送します。                   |
| プロセス制限        | デフォルトで `--pids-limit 512`                        | fork とプロセスの枯渇を制限します。                                                    |
| メモリ制限         | デフォルトで `--memory 2g`                             | セルのメモリ使用量を制限します。                                                                |
| CPU 制限            | デフォルトで `--cpus 2`                                | セルの CPU 使用量を制限します。                                                                   |
| 書き込み可能レイヤーのディスク  | 任意の `--disk`                                    | ランタイムのストレージバックエンドがクォータをサポートする場合に、コンテナレイヤーを制限します。           |
| 再起動ポリシー       | `--restart unless-stopped`                           | 意図的な停止を上書きすることなく、失敗したセルを再起動します。                         |
| ホスト公開      | `127.0.0.1:<host-port>:18789` のみ                   | Gateway をホストのワイルドカードインターフェースから切り離します。                                        |
| セルネットワーク         | セルごとに 1 つのブリッジまたは Podman 内部ネットワーク       | コンテナ IP トラフィックを分離し、必要に応じて Podman の外向きエグレスをブロックします。           |
| コンテナ ID   | ホストに一致するユーザーマッピング                            | 誰でもアクセス可能にすることなく、非公開のバインドマウントを書き込み可能に保ちます。                      |
| 永続状態     | セルごとのマウント。共有状態マウントなし               | テナントの設定、認証情報、セッション、ワークスペースを、そのテナントのデータツリー内に保持します。 |
| コンテナコマンド    | `node dist/index.js gateway --bind lan --port 18789` | コンテナネットワーク上で待ち受けることで、loopback のみに限定されたホストポートマッピングから到達可能にします。  |

Fleet は `/var/run/docker.sock` をマウントせず、`--privileged` やホストネットワークを使用せず、ケーパビリティも追加しません。セルごとのブリッジはセル間を分離する境界であり、外向きファイアウォールではありません。セルはプロバイダーとチャネルに必要なネットワークエグレスを保持します。デプロイ環境に合ったプロキシ、SSH トンネル、または tailnet 設定を loopback ポートの前段に配置してください。`http://127.0.0.1:<port>` に直接到達できるのは Fleet ホストからのみです。

このプロファイルはテナントコンテナを分離しますが、Fleet オペレーター、コンテナランタイム管理者、または侵害されたホストからテナントを保護するものではありません。完全な信頼モデルと、より強力な分離オプションについては、[マルチテナントホスティング](/ja-JP/gateway/multi-tenant-hosting)を参照してください。

## トークンの取り扱い

デフォルトでは、`fleet create` は暗号学的にランダムな 32 文字の 16 進 Gateway トークンを生成し、作成結果に一度だけ表示します。承認済みのシークレットマネージャーに保存し、作成時の出力をログへ記録しないでください。

`--gateway-token` はカスタムトークンをローカルプロセスの引数に配置するため、シェル履歴に残ったり、プロセス一覧に表示されたりする場合があります。既存のシークレット管理ワークフローで指定値が必要な場合を除き、生成されたトークンを使用してください。

トークンと、`--env` で渡されるすべての値はコンテナ環境に格納されます。Fleet はそれらを短時間だけ存在するモード `0600` の環境ファイルに書き込み、そのファイルのパスのみを Docker または Podman に渡し、ランタイムコマンドの終了後に削除します。`openclaw fleet create --gateway-token ...` または `--env KEY=VALUE` に明示的に入力した値は、外側の `openclaw` プロセスの引数やシェル履歴に表示される可能性があります。

コンテナ環境の値は、信頼されたホスト管理者から隠されません。Docker または Podman の管理者は、コンテナの検査によってそれらを読み取れます。Fleet の「一度だけ表示」という注記は通常の CLI 出力について説明するものであり、ホスト管理者によるアクセスへの耐性を意味するものではありません。
