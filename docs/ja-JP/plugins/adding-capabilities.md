OpenClaw に埋め込み、画像生成、動画生成、または将来のベンダー支援機能領域のような新しい共有ドメインが必要なときに使用します。

ルール:

- **Plugin** = 所有権境界
- **ケイパビリティ** = 共有コアコントラクト

ベンダーをチャネルやツールに直接接続することから始めないでください。まずケイパビリティを定義します。

## ケイパビリティを作成するタイミング

以下のすべてが当てはまる場合に、新しいケイパビリティを作成します。

1. 複数のベンダーが実装できる可能性がある。
2. チャネル、ツール、または機能 Plugin がベンダーを意識せずにそれを利用できる必要がある。
3. コアがフォールバック、ポリシー、設定、または配信動作を所有する必要がある。

作業がベンダー専用で、共有コントラクトがまだ存在しない場合は、いったん止めて先にコントラクトを定義します。

## 標準的な手順

1. 型付きのコアコントラクトを定義します。
2. そのコントラクト用の Plugin 登録を追加します。
3. 共有ランタイムヘルパーを追加します。
4. 証明として実際のベンダー Plugin を 1 つ接続します。
5. 機能/チャネルの利用側をランタイムヘルパーに移行します。
6. コントラクトテストを追加します。
7. オペレーター向けの設定と所有権モデルを文書化します。

## 何をどこに置くか

**コア:**

- リクエスト/レスポンス型。
- プロバイダーレジストリ + 解決。
- フォールバック動作。
- ネストされたオブジェクト、ワイルドカード、配列項目、合成ノードに伝播される `title` / `description` ドキュメントメタデータを含む設定スキーマ。
- ランタイムヘルパーサーフェス。

**ベンダー Plugin:**

- ベンダー API 呼び出し。
- ベンダー認証処理。
- ベンダー固有のリクエスト正規化。
- ケイパビリティ実装の登録。

**機能/チャネル Plugin:**

- `api.runtime.*` または対応する `plugin-sdk/*-runtime` ヘルパーを呼び出す。
- ベンダー実装を直接呼び出さない。

## プロバイダーとハーネスの境界

動作が汎用エージェントループではなくモデルプロバイダーコントラクトに属する場合は、**プロバイダーフック**を使用します。例として、トランスポート選択後のプロバイダー固有リクエストパラメーター、認証プロファイルの優先設定、プロンプトオーバーレイ、モデル/プロファイルのフェイルオーバー後の後続フォールバックルーティングなどがあります。

動作がターンを実行しているランタイムに属する場合は、**エージェントハーネスフック**を使用します。ハーネスは、空の出力、表示される出力を伴わない推論、最終回答のない構造化された計画などの明示的なプロトコル結果を分類できるため、外側のモデルフォールバックポリシーが再試行の判断を行えます。

どちらの境界も狭く保ちます。

- コアは再試行/フォールバックポリシーを所有します。
- プロバイダー Plugin はプロバイダー固有のリクエスト/認証/ルーティングヒントを所有します。
- ハーネス Plugin はランタイム固有の試行分類を所有します。
- サードパーティ Plugin はヒントを返し、コア状態を直接変更しません。

## ファイルチェックリスト

新しいケイパビリティでは、以下の領域を触ることを想定します。

- `src/<capability>/types.ts`
- `src/<capability>/...registry/runtime.ts`
- `src/plugins/types.ts`
- `src/plugins/registry.ts`
- `src/plugins/captured-registration.ts`
- `src/plugins/contracts/registry.ts`
- `src/plugins/runtime/types-core.ts`
- `src/plugins/runtime/index.ts`
- `src/plugin-sdk/<capability>.ts`
- `src/plugin-sdk/<capability>-runtime.ts`
- 1 つ以上の同梱 Plugin パッケージ。
- 設定、ドキュメント、テスト。

## 実例: 画像生成

画像生成は標準的な形に従います。

1. コアが `ImageGenerationProvider` を定義します。
2. コアが `registerImageGenerationProvider(...)` を公開します。
3. コアが `runtime.imageGeneration.generate(...)` を公開します。
4. `openai`、`google`、`fal`、`minimax` Plugin がベンダー支援の実装を登録します。
5. 将来のベンダーは、チャネル/ツールを変更せずに同じコントラクトを登録します。

設定キーは、意図的に視覚分析ルーティングとは分離されています。

- `agents.defaults.imageModel` は画像を分析します。
- `agents.defaults.imageGenerationModel` は画像を生成します。

フォールバックとポリシーが明示的なままになるよう、それらは分けておきます。

## 埋め込みプロバイダー

再利用可能なベクトル埋め込みプロバイダーには `embeddingProviders` を使用します。このコントラクトは
意図的にメモリより広い範囲を対象にしています。ツール、検索、取得、インポーター、または
将来の機能 Plugin は、メモリ
エンジンに依存せずに埋め込みを利用できます。

メモリ検索は汎用の `embeddingProviders` を利用できます。古い
`memoryEmbeddingProviders` コントラクトは、既存の
メモリ固有プロバイダーが移行する間の非推奨互換性です。新しい再利用可能な埋め込みプロバイダーは
`embeddingProviders` を使用する必要があります。

## レビューチェックリスト

新しいケイパビリティを出荷する前に、以下を確認します。

- チャネル/ツールがベンダーコードを直接インポートしていない。
- ランタイムヘルパーが共有パスである。
- 少なくとも 1 つのコントラクトテストが同梱所有権をアサートしている。
- 設定ドキュメントが新しいモデル/設定キーの名前を示している。
- Plugin ドキュメントが所有権境界を説明している。

PR がケイパビリティレイヤーを省略し、ベンダー動作をチャネル/ツールにハードコードしている場合は、差し戻して先にコントラクトを定義します。
