Gateway を再起動してもエージェントの状態は失われません。会話、トランスクリプト、
スケジュール済みジョブ、バックグラウンドタスクの記録、キューに入った送信メッセージはすべて
ディスク上に保存され、ターンの途中で中断された作業は Gateway の復帰後に
自動的に検出されて再開されます。手動での介入は
不要で、設定するものもありません。リカバリーは常に有効です。

このページでは、再起動後も保持されるもの、中断された作業の検出方法、
自動再開の動作について説明します。

## 再起動後も保持されるもの

| 状態                          | ストレージ                                  | 再起動時の動作                                                          |
| ----------------------------- | ------------------------------------------- | ----------------------------------------------------------------------- |
| 会話履歴                      | エージェントごとの SQLite データベース      | 変更されず、セッションは保存済みトランスクリプトから継続される          |
| 中断されたメインセッションのターン | エージェントごとの SQLite セッション行とトランスクリプト | 起動から数秒後に自動的に再開または整合される                            |
| サブエージェントの実行        | SQLite（共有状態データベース）              | 起動時にレジストリが復元され、中断された実行が再開される                |
| バックグラウンドタスク        | SQLite（共有状態データベース）              | 起動時に整合され、孤立した実行が復旧されるか消失としてマークされる      |
| キューに入った送信処理        | SQLite 配信キュー                           | 再起動後に処理され、未配信の返信が再試行される                          |
| スケジュール済み（cron）ジョブ | SQLite cron ストア                          | スケジュールは保持され、起動時にスケジューラが再設定される              |
| 再起動後の継続処理            | SQLite 再起動センチネル                     | 再起動を要求したセッションに対して一度限りの後続処理がディスパッチされる |

## 正常な再起動では最初に処理完了を待つ

要求された再起動（`openclaw gateway restart`、再起動を必要とする設定変更、
または Gateway の更新）では、実行中の作業が即座に終了されることはありません。
Gateway は新しい作業の受け付けを停止し、アクティブなエージェントターンと
バックグラウンドタスクが完了するまで、ドレイン時間枠（デフォルトでは5分）を上限に待機します。したがって、
ほとんどの再起動では何も中断されません。

ドレイン時間枠内に完了できない作業（または強制再起動やクラッシュによって
中断された実行）のみが中止されます。その前に、影響を受ける各
セッションにはリカバリー対象であることを示すマーカーが付けられます。

## 中断された作業の検出方法

ターンが完了しなかったセッションは、相互補完的な3つの仕組みによってマークされます。

- **ターン受け付け時：**既存のメインセッションにおける通常のテキストターンでは、
  モデルまたは `before_agent_reply` フックを実行する前に、Gateway が1つの SQLite トランザクション内で
  ユーザーメッセージを追加し、セッションを実行中としてマークし、
  そのリカバリー配信クレームを記録します。Control UI では
  `started` 確認応答を返す前にこれを行い、チャネルディスパッチでは準備済みターンが
  エージェント実行を引き継ぐ際に行います。
  コマンド、添付ファイル、ターンごとのオーバーライド、保留中の配信、以前の中止
  ヒント、Plugin が所有するセッション、実行フックを伴うターンでは、それぞれ専用の
  受け付けパスが維持されます。
  `before_agent_reply` フックがインストールされている場合は、受け付け時にそのフェーズも記録されます。
  リカバリーでは、呼び出しの途中で中断されたフックを再実行することはありません。未処理のフックが
  完了すると、そのチェックポイントに結果が記録されますが、そのフックが有効な間、
  リカバリーは引き続き安全側に失敗します。チェックポイントだけでは、再起動後に同じ
  Plugin コードと設定が読み込まれたことを証明できないためです。処理済みテキストと
  サイレント結果は、決定的に確定できるよう個別にチェックポイントへ記録されます。
  古いバージョンによって書き込まれた永続的なリカバリークレームにはソース所有権
  マーカーがないため、アップグレード時にも同じ安全側のフックチェックが適用されます。
- **シャットダウン時：**再起動時のドレイン中、アクティブな実行を持つすべてのセッションには、
  実行が中止される前にセッションストア内でリカバリーマーカーが付けられます。
- **起動時：**Gateway はセッションストアをスキャンし、実行中であることを引き続き
  主張しているものの、新しいプロセス内に稼働中の所有者がいないセッションを検出します。これにより、
  シャットダウンコードが実行されなかったハードクラッシュや強制終了も検出できます。同時に、古くなったトランスクリプトのロック
  ファイルもクリーンアップされます。

## 自動再開

起動から数秒後、Gateway はマークされた各セッションを、
前のターンが再起動によって中断されたことと、既存のトランスクリプトから
継続するようエージェントに伝える合成システムメッセージとともに再ディスパッチします。
最終返信がすでに生成されているものの未配信の場合は、そのテキストが含まれるため、
エージェントは作業をやり直す代わりにその返信を配信できます。リカバリーは指数バックオフを使用して
最大3回再試行されます。すべての再試行で1つの永続的なディスパッチ
識別子を再利用するため、結果が不明確な接続障害によって同じリカバリーが
二重に開始されることはありません。完了済みおよび再開不能な Control UI ターンにも、有界な永続的
冪等性トゥームストーンが保持されるため、再接続した送信ボックスはリクエストを
再実行せずにそれらを処理済みにできます。

メッセージツールのみの返信では、2つ目の永続的な相関情報を使用します。同じ会話に対する終端
送信がチャネルに到達する前に、Gateway は該当する正確なセッションとソースターンに
未解決の配信意図を記録します。プロバイダーから成功が確認されると、その情報は永続的な
配信済みレシートとして確定され、失敗が確認されるとクリアされます。リカバリーはツールを再実行せずに
配信済みレシートを完了できます。クラッシュによってプロバイダー側の結果が不明になった場合、
外部への作用を再実行する代わりに、リカバリーは安全側に失敗します。

配信済みの返信は、そのソースメッセージ ID とともにトランスクリプトにもミラーされます。
終端ミラーには固有のレシートキーが使用されるため、同じプロバイダー冪等性キーを持つ進捗送信によって
終端マーカーが隠されることはありません。以前のターンからの進捗
送信やレシートによって現在のターンが完了扱いになることはありません。
メッセージアクションの権限を復元できるのは、永続的なチャネル受信クレームだけです。再開された
実行は、要求者の ID や同一チャネル／スレッドの制限を含め、元のソース配信モードとソース相関情報を
保持するため、リカバリー中に再度の再起動が発生しても同じレシートが
引き続き信頼できる根拠となります。再構築可能なチャネル権限を持たない
メッセージツールのみのターンは安全側に失敗し、一度限りの再送通知を受け取ります。

再開前に、Gateway はトランスクリプト末尾から安全に
継続できるか確認します。安全でない場合（たとえば、古い保留中の
承認でターンが終了している場合）、セッションが無条件に再実行されることはありません。代わりに、
エージェントは最後のリクエストを再送するようユーザーに求める短い通知を投稿します。WebChat では、
その通知がセッション履歴へ直接書き込まれるため、再接続後も表示されます。

OpenClaw は、中断された読み取り専用の [Code Mode](/ja-JP/reference/code-mode)
作業も再構築できます。Code Mode はこれらの実行を再起動安全としてマークし、副作用のある
カタログツールや Plugin 名前空間を実行前に拒否します。`wait` 制御で
再起動が発生した場合、新しい Gateway はトランスクリプトからターンを再構築し、
モデルがそのフラグを省略またはクリアした場合でも、再構築された実行を強制的に再起動安全な状態に保ちます。
ホストは、再起動後に Code Mode が無効になっている場合も含め、再構築された
ターン全体を、監査済みの読み取り専用コアツールと、再実行が明示的に安全な Plugin ツールのみに
制限します。副作用のある作業には、重複書き込みのリスクを冒す代わりに、
引き続き再送通知による保護が適用されます。

### サブエージェント

サブエージェントの実行は共有 SQLite 状態データベースに永続化されるため、
サブエージェントレジストリはプロセスをまたいで保持されます。起動時にレジストリが復元され、
中断されたサブエージェントセッションは元のタスクコンテキストとともに再開されます。
次の2つの安全弁が適用されます。

- 2時間以上前に中断された実行は、再開されずに完了処理されるため、
  一晩停止していた Gateway が古い作業を復活させることはありません。
- 繰り返しリカバリーに失敗するセッションは、行き詰まり状態としてトゥームストーン化されるため、
  リカバリーが永遠にループすることはありません。

### バックグラウンドタスク

[バックグラウンドタスクレジストリ](/ja-JP/automation/tasks)は SQLite を基盤としており、
起動時および定期的な間隔で整合されます。完了した実行によって記録された永続的な結果は
復旧され、所有プロセスが消失した実行は、永遠に停止状態になるのではなく、
猶予期間後に消失としてマークされます。

### エージェントが要求した再起動

エージェント自体が再起動をトリガーした場合（設定変更の適用、
Gateway の更新、または明示的な再起動要求）、プロセスが終了する前に再起動センチネルが
SQLite に書き込まれます。起動後、Gateway は結果を元のチャットに投稿し、
一度限りの継続ターンをディスパッチするため、エージェントは同じチャネルとスレッドで、
中断した位置から正確に作業を再開できます。

## 安全弁と可観測性

- **クラッシュループブレーカー：**5分以内に正常終了でない起動が3回発生するとブレーカーが作動し、
  次回の起動時に自動起動される補助サービスを抑止するため、クラッシュする Gateway が
  自ら障害を増幅することを防ぎます。正常終了でない起動の時間枠が経過すると復旧します。
- **メトリクス：**リカバリーアクティビティは
  [Prometheus](/ja-JP/gateway/prometheus) を介して `openclaw_session_recovery_total` および
  `openclaw_session_recovery_age_seconds` としてエクスポートされます。
- **ログ：**リカバリーの判断は
  `main-session-restart-recovery` および `subagent-interrupted-resume`
  サブシステムに記録されます。

## 再開されないもの

- 別の所有者がすでに処理しているため、メインセッションのリカバリーから除外される
  セッション：サブエージェントセッション（サブエージェントのリカバリー）、
  cron セッション（スケジューラがスケジュールに従って再実行）、ACP 管理セッション
  （接続された IDE またはクライアントが再開を担当）。
- トランスクリプト末尾から安全に継続できないセッション。これらは
  暗黙的に再実行される代わりに、前述の再送通知を受け取ります。
- 受け付けられなかった作業：ドレイン時間枠中に到着したメッセージは、
  終了中のプロセスへ暗黙的にキューイングされるのではなく、明示的な再起動エラーで
  拒否されます。
