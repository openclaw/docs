## ステータス

提案、リビジョン3。未実装。方向性は2026-07に合意済み。リビジョン2では敵対的レビューの指摘事項（専用ワーカープロトコル、配置／環境ステートマシン、git対応の受信同期、一方向のv1ハンドオフ、制御された外向き通信に関するセキュリティ表現）を反映した。リビジョン3では、同期の所有権モデル（ワーカーがコミットを作成し、Gatewayが取り込んで公開する）を確定し、gitを使用しないプレーン同期モードを追加し、ワーカーでのexecをボックス内フルアクセスに修正し、インターネットポリシーをプロビジョニング時に移し、エージェントのディスパッチをマイルストーン3に戻した。

## 問題

OpenClawのエージェントセッションは、単一マシン上のGatewayプロセス内でループ、ツール、推論を実行する。計算能力はそのマシンによって制限され、長時間のタスクがマシンを占有し、並列作業は同じリソースを奪い合う。ホスト型製品（Cursor cloud agents、Claude Code on the web、Codex cloud）は、タスクごとに一時的なクラウドサンドボックスを用意することでこれを解決しているが、ベンダーのインフラストラクチャとベンダーへの信頼が必要になる。

すでに余剰マシンを所有している（または安価に借りられる）運用者には、「このセッションをあちらで実行し、ほかのセッションと同様にサイドバーへ表示し、完了後にマシンを破棄する」と指定する手段がない。

## 目標

- セッションがローカルセッションとまったく同じようにControl UIに表示され、ストリーミングされる状態で、完全なエージェントセッション（ループ＋ツール）を一時的なリモートマシン（「クラウドワーカー」）上で実行する。
- ワーカーに常設の認証情報（プロバイダー認証やforgeトークン）を置かず、直接のネットワーク外向き通信も許可しない。ボックスに必要なのは、到達可能なsshdだけとする。
- プロビジョニング、同期、実行、収集、破棄を完全に自動化し、プロバイダーを差し替え可能にする（最初のプロバイダー：Crabbox形式のリースCLI）。
- ターン境界で、トランスクリプト、セッションID、または（リクエストバイト列が同等のままである場合）プロバイダーのキャッシュ親和性を失うことなく、実行中の作業をGatewayからワーカーへディスパッチし、結果を安全に取り戻す。
- 人間（UI）とエージェント（ツール）の両方が、作業をクラウドワーカーへディスパッチできるようにする。
- 数日間にわたるセッションをサポートする。存続期間はポリシーであり、ハードコードされた上限ではない。

## 非目標（v1）

- ワーカー上で外部コーディングハーネス（Claude Code、Codex CLI）を使用しない。ワーカーセッションではOpenClawの組み込みランナーのみを実行する。ハーネスは独自の認証情報を使って独自に推論するため、ハーネス対応はv2のオプトインとする。
- best-of-N／並列試行のファンアウトは行わない。
- VPN／tailnetに依存しない。トランスポートはSSHのみとする。
- 新しいサンドボックスランタイムは導入しない。ワーカーマシンを分離境界とし、ボックス内のOSサンドボックスは後から重ねられる。
- v1では対称的なライブマイグレーションを行わない。ディスパッチはローカル→ワーカーのみとする。ワーカー→ローカルには、停止済みのセッションと完了済みのワークスペース調整が必要になる。双方向ライブハンドオフは、後に同じバリア機構を基盤として構築する。
- GatewayにJSONの補助状態を置かない。環境、配置、カーソル、許可の状態はSQLiteに保存する。

## 先行事例（踏襲するもの、反転させるもの）

- Cursor cloud agents：エージェントループは同社のクラウドで実行され、VMはツール実行対象となる。追記専用の会話ストアがすべてのクライアントへストリーミングされる。インストール後のスナップショットによるウォームスタートを行う。セルフホスト型ワーカーは外向き接続のみを行うワーカープロセスである。「会話の信頼できる唯一の情報源をオーケストレーターに置く」という考え方とストリーミングモデルを踏襲し、ループの配置は反転させる（以下の決定を参照）。
- Codex cloud：ネットワーク接続可能なセットアップフェーズと、シークレットを除去したオフラインのエージェントフェーズからなる2段階ランタイム。迅速な後続処理のためのコンテナ状態キャッシュ。外向き通信に対する方針としてこのフェーズ分割を踏襲し、v2のウォームイメージに向けてキャッシュの考え方を採用する。
- Claude Code on the web：セッションごとのVM。認証情報を分離するgitプロキシ（実際のトークンはサンドボックスに一切入らず、pushはセッションブランチに制限される）。セットアップ後のファイルシステムスナップショット。テレポートハンドオフ＝push済みブランチ＋再生された履歴。認証情報の分離とハンドオフの枠組みを踏襲するが、外向き同期はGatewayからrsyncで行うため、変更未コミットのワーキングツリーにも対応でき、forgeトークンがボックス付近のどこにも存在しない。
- Copilot coding agent：パッケージレジストリの許可リストを備えたデフォルト拒否の外向き通信。推論とウェブ検索はSSHトンネル経由で届くため、定常状態におけるデフォルトはさらに強力（直接の外向き通信を一切許可しない）だが、これが「外向き通信ゼロ」ではなく「制御された外向き通信」である理由については「セキュリティ」を参照。

## アーキテクチャ上の決定：ループはワーカー上、推論はGateway経由

3つの配置を検討した。

1. ループはGatewayに残し、ワーカーがツールを実行する（Cursorモデル）。最も安全な障害領域であり（トランスクリプト、推論、承認、再起動後の復旧がすべてローカルに残る）、レビュー担当者が最初のマイルストーンとして推奨した方式でもある。製品アーキテクチャとしては不採用。OpenClawのexec以外のツールはプロセス内ファイルシステム操作であるため、ファイルの読み取り／編集／grepのたびにネットワーク往復が発生するか、ツールサーフェスを粗粒度のワークスペースRPCへ大規模にリファクタリングする必要がある。ランタイム動作は通信量が多く、レイテンシの制約を受ける。すでに構築済みの部分（Nodeへのexecオフロード）ではこの考え方を再利用するが、ツールのリモート化レイヤーは構築しない。
2. ループと推論の両方をワーカー上で実行する。障害領域は最も単純だが、モデルの認証情報（OAuthプロファイルを含む）を使い捨てマシンへ送る必要があり、Gatewayはポリシー／ルーティング／監査の制御を失い、マイグレーションによってプロバイダー呼び出し元のIDが切り替わるため、プロバイダーキャッシュが無効になる。
3. ループ＋ツールはワーカー上で実行し、モデル呼び出しはGateway経由でプロキシする。これを採用する。ツール呼び出しごとではなくモデルターンごとに1往復する。ツールはコードの近くで実行される。Gatewayは認証プロファイル、プロバイダーのルーティング、ポリシーの唯一の所有者であり続ける。ワーカーはシークレットを一切保持しない。

選択肢3では、各モデルターン中に同期的にGatewayへ依存することになる。そのため、耐久性に関する規則は後付けではなく、この決定の一部である。

- ターン途中でGatewayが失われると、実行中のプロバイダー呼び出しは失敗する。そのターンは失敗として記録され、再接続後に新しいターンとして再試行される。実行中のプロバイダーストリームを透過的に再生することはない（二重課金／ツール呼び出しの二重実行のリスク）。
- ワーカー↔Gateway間のすべての操作は永続的なID（「ワーカープロトコル」を参照）を持つため、再接続時には処理を宙に浮かせるのではなく、再開するか、キャッシュ済みの最終結果を取得する。
- Gatewayは容量管理されるコンポーネントである。v1の範囲には、同時ワーカー数の制限、フロー制御、負荷制限が含まれる（「容量」を参照）。

Gatewayはトランスクリプトを保存し、すべてのプロバイダートラフィックの発信元でもあるため、セッションは場所に依存しない。ループをGatewayとワーカーの間で移動しても、プロバイダー側にもUIのデータパスにも何も変化しない。これにより、ディスパッチと取り戻しを低コストで実現できる。

## コンポーネント

### 1. 環境ステートマシン＋プロバイダー契約

Gatewayプロトコルの`environments.*`は、現在はステータスのみのプロジェクションである。永続的な中核はSQLiteが所有する環境レコードとステートマシンであり、RPCの形状より先に設計する。

`requested → provisioning → bootstrapping → ready → (attached|idle) → draining → destroying → destroyed | failed | orphaned`

- プロビジョニングはクラッシュセーフである。プロバイダー呼び出しの前に、決定的な操作IDとともに意図を示す行を永続化するため、Gatewayの再起動時に二重プロビジョニングや有料マシンの孤立を引き起こすことなく、処理中のリースを引き継げる。
- 再起動時の調整と孤立リソースのスイーパー（プロバイダーの`inspect`とローカルレコードの比較）は、堅牢化ではなくv1の要件である。

プロバイダー契約（Pluginによる実装。コアにプロバイダー名やポリシーを含めない）：

```ts
type WorkerProvider = {
  id: string;
  provision(profile: WorkerProfile, opId: string): Promise<WorkerLease>; // → ssh host/port/user/key material
  inspect(lease: { leaseId: string; profile: WorkerProfile }): Promise<LeaseStatus>; // adopt/health/orphan sweep
  renew?(leaseId: string): Promise<void>; // long-lived sessions vs provider TTLs
  destroy(lease: { leaseId: string; profile: WorkerProfile }): Promise<void>; // idempotent, returns only on proof of teardown
};
```

RPC：`environments.create`、`environments.destroy`、拡張された`environments.list/status`（プロバイダー、リースID、状態、経過時間、アイドル時間、アタッチ済みセッション）。最初のプロバイダーは、Crabbox形式のリースCLIラッパー（製品向け経路）と、開発専用と明記された静的SSHホストプロバイダーとする。共有ホスト上のワーカーは同じホスト上の無関係なデータを読み取れるため、静的ホストは機能開発用であり、デフォルトのセキュリティ方針ではない。

### 2. ワーカーのブートストラップ：ボックスへのOpenClawのインストール

専用のワーカーアーティファクトは使用せず、npmの可用性にも依存しない。

- すべてのモードでの標準インストール：Gatewayが生成するコンテンツハッシュ付きワーカーバンドル（Gateway自身のビルド出力をtarballとしてパッケージ化したもの）をSSH経由で転送し、ボックスにインストールする。これにより、開発ビルドと未リリースのコミットにも構造上対応できる。
- Gatewayがリリース済みバージョンを実行している場合、`npm i -g openclaw@<exact gateway version>`を最適化として使用する。`latest`は決して使用しない。
- ブートストラップは冪等である。一致するバンドルハッシュを持つウォームリースでは、インストールを省略する。未設定のマシンには、ネットワーク接続可能なツールチェーンフェーズ（Nodeランタイム）が必要な場合がある。これはセットアップフェーズの一部であり、完了後に閉じる。
- ハンドシェイクでは、ワーカーのビルドハッシュ、プロトコル機能セット、ランタイム互換性を検証する。既存のGatewayバージョン／プロトコルチェックでは不十分である（SSHトンネル経由のNodeは厳密なバージョン拒否の対象外）ため、ワーカーの受け入れ処理で独自に完全一致のビルドチェックを行う。

ワーカーモード（`openclaw worker`）はフォークではなくエントリーポイントである。接続処理と組み込みエージェントランナーを備え、セッションの永続化とモデル呼び出しをGateway RPCで処理する。Gatewayのサーフェスを起動してはならない。チャンネルは起動せず、セッションのツールセット以外のPluginも自動起動せず、使い捨ての状態ディレクトリを使用し、ローカル認証プロファイルを保持しない。

### 3. トランスポート：すべてSSH経由

接続性はGatewayが所有し、ワーカーに必要なのはsshdだけである。

- GatewayはワーカーへのSSH接続を開き（プロバイダーリースから得た認証情報を使用し、プロビジョニング出力から得たホストキーを固定する。`StrictHostKeyChecking=no`は使用しない）、ワーカーローカルのソケットをGatewayのWSエンドポイントへ転送するリバーストンネルを確立する。
- 制御／モデルのトラフィックとワークスペース転送では、同じ固定済みの信頼情報を使用する別々のSSH接続を用いる。これにより、rsyncがトークンストリームをHOLブロッキングしないようにする。
- トンネルのライフサイクル（キープアライブ、バックオフ付き再接続）は、Gateway上の環境ランタイムが所有する。トンネルの一時的な切断はセッションレベルでは見えない。以下に示す永続的なプロトコル状態により、ワーカーは再アタッチして再開できる。

### 4. ワーカープロトコル（専用。Nodeプロトコルではない）

現在のNodeの接続境界に対する敵対的レビューにより、単純な再利用は不採用となった。保留中のNode呼び出しは接続とともに消滅するプロセスローカルなPromiseであり、Nodeの冪等性キーは解析されるものの重複排除されない。そして決定的な問題として、接続済みのNodeは通常のNodeイベント（エージェント実行リクエストを含む）を送信できるため、「Node種別＋ケイパビリティ上限」は受信セキュリティ境界にならない。そのため、ワーカーには閉じたバージョン管理済みのRPC／イベント許可リストを持つ、認証済みの`worker`ロールを割り当てる。ワーカー接続から従来のNodeイベントハンドラーへ到達することはできない。

IDと認証情報：プロビジョニング時に、環境ID、ワーカーキー、バンドルハッシュ、許可された単一セッション、許可されたRPCセット、有効期限に紐づく短期ワーカー認証情報を発行する。SSHで検証されたペアリングも引き続き適用される（ボックスをプロビジョニングし、キーを保持している）が、認可は宣言されたNodeサーフェスではなく、発行された認証情報に基づく。

永続的な操作セマンティクス（既存のACPランタイムとそのイベント台帳から形状を借用：安定したハンドル、セッション単位の直列化、永続的な`(session, seq)`再生）：

- すべての操作は`(sessionId, lifecycleRevision, runId, ownerEpoch, streamKind, seq)`のスコープを持つ。
- 所有権エポックによって古いワーカーをフェンスする。代替ワーカーはエポックを進め、古いエポックから遅れて到着した結果は決定的に拒否される。
- SQLiteに永続化されたACKカーソルとキャッシュ済みの最終結果を用いる、少なくとも1回の配信。重複排除は決定的に行う。厳密に1回の保証は行わない。
- キャンセル、クローズ、再開、最終結果のための明示的なフレームと、ストリーム上のクレジット／ウィンドウベースのフロー制御。
- プロトコル機能のネゴシエーションは、一般的なNodeプロトコルのバージョンから独立している。

### 5. セッションバックエンドRPC

2つの異なる契約 — 現在のコードベースは、永続的なトランスクリプト変更（セッションマネージャーが所有し、親/リーフ状態を持つ JSONL ツリー）とプロセスローカルなライブイベント（ストリーミング差分、ツールのライフサイクル、承認）を分離しており、ワーカープロトコルはこの分離を維持する必要があります。

- 永続的なトランスクリプトのコミット: ワーカーは `runEpoch` とベースリーフの compare-and-swap を伴うセマンティックな追記バッチを送信し、Gateway セッションマネージャーがエントリ ID と親 ID を生成します。ワーカーは、信頼済みトランスクリプト行、エントリ ID、親 ID、または別セッションの ID を決して指定できません。
- リプレイ可能なライブイベント: ワーカーのシーケンス番号、Gateway の ACK、上限付き保持、遅延イベントのフェンシングを備えた型付きイベントユニオンです。既存のエージェントイベントのファンアウトへ供給されるため、チャットビュー、ツール行、未読/ステータスロジックはローカルセッションと同一に動作します。

推論プロキシ: 既存のランタイムプロキシストリームクライアント（`src/agents/runtime/proxy.ts`）のイベント語彙を再利用しますが、信頼境界を移動します。ワーカーが送信できるのは、セッション/実行 ID、承認済みモデル参照、コンテキスト、制約付き生成オプションのみです。Gateway は独自のカタログから、プロバイダー、エンドポイント、認証、ヘッダー、ルーティング、コストポリシーを解決します。ワーカーが指定したモデルオブジェクト（例: 攻撃者が制御する `baseUrl`）は拒否されます。リクエストサイズ制限、キャンセル、監査、終了結果のリプレイが適用されます。Gateway 常駐ツール（websearch）は Gateway 上で実行され、同じチャネルを通じて結果を返します。

### 6. ワークスペース同期

同期アンカーは、排他的な配置所有権を持つ Gateway ローカルのワークスペースです。git ワークスペースの場合は専用の管理対象 worktree（既存の管理対象 worktree メタデータ — ブランチ、ベース、スナップショット所有権 — を基盤とする）、非 git ワークスペースの場合は Gateway 所有のターゲットディレクトリです。ユーザーのライブチェックアウトは決して使用しません。セッションがリモートに配置されている間の排他的所有権により、受信同期は構造上競合しません。

所有権の分離 — コミットと公開:

- ワーカー側のエージェントは、自身のコピー内で通常どおりコミットを作成します（`git commit` はローカルかつ認証情報不要の操作であり、作者 ID は Gateway 設定から投影されます）。これらのコミットは、Gateway が採用するまで不活性なオブジェクトです。
- 信頼を必要とするすべての処理は Gateway が行います。受信コミットが記録済みのベース上に構築されていることの検証、ローカル worktree の fast-forward、push、PR 作成、任意の署名/再署名を、すべて Gateway ローカルの認証情報で実行します。ワーカーは git や forge の認証情報を保持せず、リモートにアクセスすることもありません。

ワークスペースが git リポジトリかどうかに応じて、2つの同期モードを選択します。

- Git モード。送信: トンネルの SSH ID を使用して worktree を rsync します（未コミットファイルと対象となる未追跡ファイルを含み、crabbox 形式の include/exclude を使用し、`.worktreeinclude` を尊重します）。その結果は、不変のベースマニフェスト（コンテンツハッシュ + ベースコミット）として記録されます。受信: 新しいコミットは、記録済みのベースに対する git bundle または一時 ref として返されます。未追跡アーティファクトは、サイズ/タイプ/シンボリックリンク封じ込めチェックを伴う明示的なマニフェスト経由で返されます。採用時にはベースの祖先関係を検証し、分岐があれば停止します。どちらの側も暗黙に上書きされません。削除、名前変更、サブモジュール、シンボリックリンクによる脱出は、rsync のヒューリスティックではなくマニフェスト規則で処理されます。
- Plain モード（git なし — 例: ボックス上でプロジェクトをゼロから構築する場合）。送信は同じ rsync + ベースマニフェストです。受信は、マニフェスト差分に基づき、削除を伝播しながら Gateway 所有のターゲットディレクトリへミラーリングします。Git モードと同じ理由で安全です。排他的所有権により、競合する同時ローカル編集が存在しないためです。それでもベースマニフェストは予期しないローカルドリフトを検出し、上書きせずに停止します。

チェックポイントにより、数日間にわたるセッションをリース喪失から保護します。定期的な受信チェックポイント（git モードではセッションブランチのコミット、Plain モードではマニフェストスナップショット）を作成し、頻度はプロファイルポリシーで決定します（デフォルトはターン単位）。

### 7. 配置ステートマシン、セッション、UI

ランタイム配置は、疎な2つの行フィールドではなく、セッションをキーとする SQLite 所有のステートマシンです。

`local → requested → provisioning → syncing → starting → active(worker) → draining → reconciling → local | reclaimed | failed`

環境 ID、遷移世代、アクティブ所有者エポック、ワークスペースのベースマニフェスト、ワーカーバンドルハッシュ、最後の ACK カーソルを永続化します。ターン受付は、どちらかのループがターンを開始する前に配置をアトミックに取得します。そのため、古いスナップショットに基づいて受け付けられたローカルメッセージがワーカーターンと競合することはなく、常に1つのループだけがセッションを所有します。

UI:

- ワーカーセッションは、配置メタデータが付加された通常のセッション行です。通常のストアに存在し、`sessions.list` で一覧表示され、既存のサブスクリプションでストリーミングされます。サイドバーとチャットに新しいデータパスは不要で、必要なのは表示だけです。具体的には、ワーカーバッジと配置/環境ステータス（`provisioning / syncing / running / idle / reconciling / reclaimed`）です。
- 作成 UX: セッションターゲットバー（セッションサイドバーの再設計）に、Gateway と Node に加えてクラウドワーカーの宛先を追加します。設定済みのプロバイダープロファイルが必要であり、設定されるまでは機能を表示しません。
- エージェントディスパッチ: セッションツールにより、人間と同じようにエージェントが作業をクラウドワーカーへ引き渡せます（ワーカーを基盤とするサブセッション、サブエージェント形式）。人間によるディスパッチと同じマイルストーンで提供し、同じオプトインのプロバイダー設定で制御します。再帰は構造的に制限されます（v1 ではワーカーセッション自身がワーカーをディスパッチすることはできません）。支出管理は、クォータ機構ではなく環境単位の会計/監査で行います。

## ディスパッチとハンドオフ

v1 は意図的に非対称です。

- ローカル → ワーカー（ディスパッチ）: 後述の移行バリアを通過し、ワーカーをプロビジョニングまたは再利用し、同期して配置を切り替えます。次のターンはリモートで実行されます。
- ワーカー → ローカル（プルバック）: セッションを停止し（同じバリアに従ってワーカーをドレイン）、受信調整を完了して、配置をローカルへ切り替えます。ライブマイグレーションではありません。
- 対称ライブハンドオフ（稼働中のセッションを停止せずに双方向へ移動すること）は、同じバリアと調整機構を再利用し、フォールトインジェクションテストでバリアが実証された後に提供します。

移行バリア（「ターン境界」だけでは不十分です。承認、バックグラウンドプロセス、ロック解放後のトランスクリプトマージが境界をまたぐ可能性があります）:

1. 新しいターンの受付を停止します（配置の取得）。
2. アクティブな実行をキャンセルまたはドレインします。
3. 保留中の exec 承認と実行許可を失効させます。
4. トランスクリプトのサイド書き込みとライブイベントの ACK をドレインします。
5. ワーカーの子プロセスを終了します。
6. 所有者エポックを進め、以前の所有者をフェンスします。
7. ワークスペースを調整します（受信、競合認識）。
8. 新しい所有者をアクティブ化します。

キャッシュアフィニティ: どちらの配置でもプロバイダーリクエストは Gateway から発信されるため、シリアライズされたプロバイダーリクエストが同等であればキャッシュアフィニティは維持されます。同じツール順序、システム指示、プロバイダーラッパー、キャッシュメタデータ（Gateway 側に保持）が条件です。これは仮定ではなくテスト可能な特性です。サポート対象の各プロバイダー転送について、ローカル/ワーカー配置間のバイト同等性テストを、ワーカーループを導入するマイルストーンに含めます。

## セキュリティモデル

正確に述べると、ワーカーには直接のネットワーク外向き通信も、常設のプロバイダー/forge 認証情報もありません。ただし「外向き通信がゼロ」ではありません。推論と Gateway 実行ツールは制御された外向きチャネルです（プロンプトインジェクションを受けたワーカーでも、ワークスペースのバイト列をモデルコンテキストや websearch クエリに含められます）。したがって、以下を適用します。

- 制御された外向き通信の会計: 推論プロキシと Gateway ツールに対し、環境単位の監査とオペレーターが確認可能な会計を提供します。レート/バイト制限は、支出クォータ機構としてではなく、プロトコルのフロー制御（容量）として設けます。
- ワーカーから Gateway への受信は、閉じたワーカープロトコルの許可リストに限定されます。トランスクリプト書き込みは構造的に制約されます（Gateway が生成する ID、単一のバインド済みセッション）。
- ワーカーの exec はボックス内で完全な権限を持ちます。ボックスは破棄可能で認証情報を持たないため、コマンドごとの承認は何も保護せず、摩擦だけを追加します。保護される境界は受信調整と監査です。exec が Gateway の Node 承認パスを通過することはありません。
- インターネットポリシーはプロビジョニング時のプロバイダー判断です。環境プロファイルがボックス作成時に決定します（ファイアウォール/セキュリティグループ/外向き通信なしのネットワーク）。任意でネットワーク接続されたセットアップフェーズを設け、エージェントフェーズの前にプロバイダーが閉じることもできます。コアはランタイムのネットワーク切り替えを実装しません。
- プロビジョニング時のボックス衛生: クラウドメタデータエンドポイントをブロックするか、不在を検証し、インスタンスプロファイル、継承された SSH エージェント、Docker ソケットを持たせず、クリーンな環境/home を使用します。SSH ホストキーはプロビジョニング出力に基づいて固定します。
- Gateway 側のすべての処理（push、PR、プロバイダー呼び出し）に対する承認とポリシーは、引き続き Gateway 上で実行されます。

侵害されたワーカーセッションの影響範囲: 同期されたワークスペースのコピーと、監査対象のプロキシチャネルが許可する範囲です。認証情報、直接ネットワークアクセス、許可リスト外の Gateway サーフェスはありません。

## 容量

Gateway は N 個のワーカーに対するすべてのプロンプトとトークンストリームを中継するため、v1 では本番環境で問題を発見するのではなく、容量モデルを明示します。Gateway 単位の同時ワーカー数制限、ストリーム単位のクレジットウィンドウ（現在のイベントストリームキューは無制限で、Node ソケットのバッファ上限は低速なコンシューマーを強制切断するため、どちらも未変更のままでは不適切です）、バースト用の上限付きディスクスプーリング、UI に表示されるバックプレッシャー状態を伴う負荷制限を設けます。ワークスペース転送は独自の SSH チャネル上で維持されます。

## ライフサイクル

- アイドル時の自動停止と TTL は、固定定数ではなくプロバイダープロファイルのポリシーです。デフォルトは明示的なキープアライブを伴う十分に余裕のある値です。数日間にわたる作業を第一級に扱います（リース方式のバックエンド向けにプロバイダーの `renew` が存在します）。進行中のターンまたは直近のアクティビティがあるセッションは決して回収されません。
- ワーカーの停止または回収時: 配置は `reclaimed` に移行し、セッション行は残ります。次のメッセージで新しいワーカーをプロビジョニングし、最後のチェックポイントから再同期します。会話は決して失われません（Gateway 側のストア）。最後のチェックポイント以降のワークスペース変更は失われ、その旨を UI に表示します。
- 初日からウォームリースを再利用します（サポートするプロバイダーの場合）。ブートストラップ後のイメージスナップショットは、v2 の高速起動パスです。

## 設定サーフェス

最小限かつオプトインです。プロバイダープロファイルブロック（プロバイダー ID、認証情報/CLI 参照、同期規則、有効期間ポリシー、予算、任意のセットアップフェーズ）と、セッション単位の配置選択のみです。新しい環境変数は追加しません。未設定のインストールには何も表示されません。

## マイルストーン

実装は、小規模で独立してマージ可能な PR として導入します。以下の各マイルストーンは1つの変更ではなく、PR シリーズです。

1. 基盤: 環境ステートマシン + プロバイダー契約 + crabbox 形式のプロバイダー（開発ハーネスとして static-SSH）、ワーカーバンドルのブートストラップ + 受付ハンドシェイク、SSH トンネル + ホストキー固定、管理対象 worktree スナップショット + 送信同期（Git + Plain モード）。孤立リソースのスイープ + 再起動時の採用。
2. ワーカープロトコル + ワーカーループ: 認証済みワーカーロール、永続的な操作/エポック/ACK カーソル、トランスクリプトコミット + ライブイベント契約、Gateway 解決済みモデルを使用する推論プロキシ、フロー制御。1つのプロバイダー、新規セッションの人間によるディスパッチのみ、ハンドオフなし。フォールトインジェクションテスト（トンネル分断、Gateway 再起動、ワーカー停止）を完了条件とします。
3. ディスパッチ + プルバック + エージェントディスパッチ: 移行バリア、UI ターゲットバーに接続された配置ステートマシン、受信調整 + チェックポイント、環境単位の監査、容量制限、エージェントディスパッチツール（ワーカーセッションは再帰不可）。プロンプトキャッシュのバイト同等性テスト。
4. マイルストーン3のフォールトインジェクションによる実証後に、対称ライブハンドオフ。

以降: 環境単位の認証情報ハイドレーションをオプトインとするワーカー上の ACP ハーネス、スナップショット/ウォームイメージによる高速起動、ファンアウト（N 個のリース、同じプロンプト）、ボックス内 OS サンドボックス、アーティファクトスキーマを介したより高度なアーティファクト取得。

## 未解決の問題

- ワーカーでのPlugin/Skillsの利用可否: リポジトリに含まれるSkillsはワークスペースと自動的に同期されます。Gatewayで構成されたエージェントのSkills/Pluginについては、同期するか除外するかを明示的に決定する必要があります（いずれの場合も、ツール/Pluginマニフェストは受け入れハンドシェイクの一部です）。
- チェックポイントのデフォルト頻度: 非常に発言量の多いセッションで、ターンベースにするか時間ベースにするか。
- 環境プロファイルとマルチエージェントルーティングの連携方法（エージェントごとのデフォルトプロファイルか、セッションごとの選択のみにするか）。
