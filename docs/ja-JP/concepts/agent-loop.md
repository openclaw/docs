エージェントループは、メッセージをアクションと返信に変換する、セッションごとに直列化された実行です: 取り込み、コンテキスト組み立て、モデル推論、ツール実行、ストリーミング、永続化。

## エントリーポイント

- Gateway RPC: `agent` と `agent.wait`。
- CLI: `openclaw agent`。

## 実行シーケンス

1. `agent` RPC はパラメータを検証し、セッション (`sessionKey`/`sessionId`) を解決し、セッションメタデータを永続化し、ただちに `{ runId, acceptedAt }` を返します。
2. `agentCommand` はターンを実行します: モデル + thinking/verbose/trace のデフォルトを解決し、Skills スナップショットを読み込み、`runEmbeddedAgent` を呼び出し、埋め込みループがまだ発行していない場合はフォールバックの **ライフサイクル end/error** を発行します。
3. `runEmbeddedAgent`: セッションごとおよびグローバルキューを介して実行を直列化し、モデル + 認証プロファイルを解決し、OpenClaw セッションを構築し、ランタイムイベントを購読し、assistant/tool デルタをストリーミングし、実行タイムアウトを適用し (期限切れ時に中止)、ペイロードと使用量メタデータを返します。Codex app-server ターンでは、ターミナルイベントの前に app-server 進捗の生成が止まった受理済みターンも中止します。
4. `subscribeEmbeddedAgentSession` はランタイムイベントを `agent` ストリームへ橋渡しします: ツールイベントは `stream: "tool"`、assistant デルタは `stream: "assistant"`、ライフサイクルイベントは `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)。
5. `agent.wait` (`waitForAgentRun`) は `runId` の **ライフサイクル end/error** を待機し、`{ status: ok|error|timeout, startedAt, endedAt, error? }` を返します。

## キューイングと並行処理

実行はセッションキーごと (セッションレーン)、および任意でグローバルレーンを通じて直列化され、ツール/セッションの競合を防ぎます。メッセージングチャンネルは、このレーンシステムに入力されるキューモード (steer/followup/collect/interrupt) を選択します。[コマンドキュー](/ja-JP/concepts/queue) を参照してください。

トランスクリプト書き込みは、セッションファイル上のセッション書き込みロックによって追加で保護されます。このロックはプロセスを認識するファイルベースのロックであるため、プロセス内キューを迂回する書き込みや別プロセスからの書き込みを捕捉します。書き込み側は、セッションがビジーであると報告する前に、最大 `session.writeLock.acquireTimeoutMs` (デフォルト `60000` ms、env オーバーライド `OPENCLAW_SESSION_WRITE_LOCK_ACQUIRE_TIMEOUT_MS`) まで待機します。

セッション書き込みロックはデフォルトで再入不可です。同じロックの取得を意図的にネストしつつ、1 つの論理的な書き込み側を維持するヘルパーは、`allowReentrant: true` でオプトインする必要があります。

## セッションとワークスペースの準備

- ワークスペースが解決され作成されます。サンドボックス化された実行では、サンドボックスワークスペースルートにリダイレクトされる場合があります。
- Skills が読み込まれ (またはスナップショットから再利用され)、env とプロンプトに注入されます。
- ブートストラップ/コンテキストファイルが解決され、システムプロンプトに注入されます。
- ストリーミング開始前にセッション書き込みロックが取得され、`SessionManager` が開かれて準備されます。以後のトランスクリプト書き換え、Compaction、または切り詰めパスは、トランスクリプトファイルを開く、または変更する前に、同じロックを取得する必要があります。

## プロンプト組み立て

システムプロンプトは、OpenClaw のベースプロンプト、Skills プロンプト、ブートストラップコンテキスト、および実行ごとのオーバーライドから構築されます。モデル固有の制限と Compaction 予約トークンが適用されます。モデルが見る内容については [システムプロンプト](/ja-JP/concepts/system-prompt) を参照してください。

## フック

OpenClaw には 2 つのフックシステムがあります:

- **内部フック** (Gateway フック): コマンドとライフサイクルイベント用のイベント駆動スクリプト。
- **Plugin フック**: エージェント/ツールライフサイクルと gateway パイプライン内の拡張ポイント。

### 内部フック (Gateway フック)

- **`agent:bootstrap`**: システムプロンプトが確定される前にブートストラップファイルを構築している間に実行されます。ブートストラップコンテキストファイルを追加または削除するために使用します。
- **コマンドフック**: `/new`、`/reset`、`/stop`、その他のコマンドイベント (Hooks ドキュメントを参照)。

セットアップと例については [フック](/ja-JP/automation/hooks) を参照してください。

### Plugin フック

これらはエージェントループまたは gateway パイプライン内で実行されます:

| フック                                                    | 実行内容                                                                                                                                                                                                                                                                                        |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `before_model_resolve`                                  | セッション前 (`messages` なし) に、解決前のプロバイダー/モデルを決定論的にオーバーライドします。                                                                                                                                                                                                |
| `before_prompt_build`                                   | セッション読み込み後 (`messages` あり) に、送信前に `prependContext`、`systemPrompt`、`prependSystemContext`、または `appendSystemContext` を注入します。ターンごとの動的テキストには `prependContext` を使用し、システムプロンプト空間に属する安定したガイダンスにはシステムコンテキストフィールドを使用します。 |
| `before_agent_start`                                    | どちらのフェーズでも実行される可能性があるレガシー互換フックです。上記の明示的なフックを推奨します。                                                                                                                                                                                                    |
| `before_agent_reply`                                    | インラインアクションの後、LLM 呼び出しの前。Plugin がターンを引き受けて合成返信を返すか、完全に沈黙させることができます。                                                                                                                                                                |
| `agent_end`                                             | 完了後、最終メッセージリストと実行メタデータを伴って実行されます。                                                                                                                                                                                                                             |
| `before_compaction` / `after_compaction`                | Compaction サイクルを観察または注釈付けします。                                                                                                                                                                                                                                                      |
| `before_tool_call` / `after_tool_call`                  | ツールパラメータ/結果をインターセプトします。                                                                                                                                                                                                                                                              |
| `before_install`                                        | オペレーターのインストールポリシーが実行された後、ステージングされた skill/plugin インストール素材に対して、Plugin フックが現在のプロセスで読み込まれている場合に実行されます。                                                                                                                                                           |
| `tool_result_persist`                                   | ツール結果が OpenClaw 所有のセッショントランスクリプトに書き込まれる前に、同期的に変換します。                                                                                                                                                                                      |
| `message_received` / `message_sending` / `message_sent` | 受信および送信メッセージフック。                                                                                                                                                                                                                                                         |
| `session_start` / `session_end`                         | セッションライフサイクル境界。                                                                                                                                                                                                                                                               |
| `gateway_start` / `gateway_stop`                        | Gateway ライフサイクルイベント。                                                                                                                                                                                                                                                                   |

アウトバウンド/ツールガードのフック判定ルール:

- `before_tool_call`: `{ block: true }` はターミナルであり、優先度の低いハンドラーを停止します。`{ block: false }` は no-op であり、以前のブロックを解除しません。
- `before_install`: 上記と同じターミナル/no-op セマンティクスです。CLI インストールと更新パスをカバーする必要がある、オペレーター所有のインストール許可/ブロック判定には、`before_install` ではなく `security.installPolicy` を使用します。
- `message_sending`: `{ cancel: true }` はターミナルであり、優先度の低いハンドラーを停止します。`{ cancel: false }` は no-op であり、以前のキャンセルを解除しません。

フック API と登録の詳細については [Plugin フック](/ja-JP/plugins/hooks) を参照してください。

ハーネスはこれらのフックに適応できます。Codex app-server ハーネスは、文書化されたミラー対象サーフェスの互換性契約として OpenClaw Plugin フックを維持します。Codex ネイティブフックは、別個の低レベル Codex メカニズムです。

## ストリーミング

- assistant デルタは、エージェントランタイムから `assistant` イベントとしてストリーミングされます。
- ブロックストリーミングは、`text_end` または `message_end` で部分返信を発行できます。
- Reasoning ストリーミングは、別のストリームまたはブロック返信になり得ます。
- チャンク化とブロック返信の動作については [ストリーミング](/ja-JP/concepts/streaming) を参照してください。

## ツール実行

- ツール開始/更新/終了イベントは `tool` ストリームで発行されます。
- ツール結果は、ログ記録/発行前にサイズと画像ペイロードについてサニタイズされます。
- メッセージングツール送信は、重複する assistant 確認を抑制するために追跡されます。

## 返信整形

最終ペイロードは、assistant テキスト (任意の Reasoning を含む)、インラインツール要約 (verbose かつ許可されている場合)、およびモデルエラー時の assistant エラーテキストから組み立てられます。

- 正確なサイレントトークン `NO_REPLY` は送信ペイロードからフィルタリングされます。
- メッセージングツールの重複は、最終ペイロードリストから削除されます。
- レンダリング可能なペイロードが残っておらず、ツールでエラーが発生した場合、メッセージングツールがすでにユーザーに見える返信を送信していない限り、フォールバックのツールエラー返信が発行されます。

## Compaction とリトライ

自動 Compaction は `compaction` ストリームイベントを発行し、リトライをトリガーできます。リトライ時には、重複出力を避けるためにインメモリバッファとツール要約がリセットされます。[Compaction](/ja-JP/concepts/compaction) を参照してください。

## イベントストリーム

- `lifecycle`: `subscribeEmbeddedAgentSession` によって発行されます (`agentCommand` によるフォールバックとしても発行)。
- `assistant`: エージェントランタイムからストリーミングされるデルタ。
- `tool`: エージェントランタイムからストリーミングされるツールイベント。

## チャットチャンネル処理

assistant デルタはチャット `delta` メッセージへバッファリングされます。**ライフサイクル end/error** でチャット `final` が発行されます。

## タイムアウト

| タイムアウト | デフォルト | 備考 |
| ------------------------------------------------ | ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `agent.wait` | 30s | 待機のみ。`timeoutMs` パラメーターで上書きされます。基盤の実行は停止しません。 |
| エージェントランタイム（`agents.defaults.timeoutSeconds`） | 172800s（48h） | `runEmbeddedAgent` の中止タイマーによって適用されます。 |
| Cron の分離エージェントターン | Cron が所有 | スケジューラーは実行開始時に独自のタイマーを開始し、設定された期限で実行を中止してから、タイムアウトを記録する前に制限付きのクリーンアップを実行します。これにより、古い子セッションがレーンを停止状態のままにできなくなります。 |
| モデルアイドルタイムアウト | `agents.defaults.timeoutSeconds`、デフォルトで最大 120s | アイドル期間内に応答チャンクが到着しない場合、OpenClaw はモデルリクエストを中止します。`models.providers.<id>.timeoutSeconds` は、低速なローカルまたはセルフホストのプロバイダー向けにこのアイドル監視を延長しますが、より低い `agents.defaults.timeoutSeconds` または実行固有のタイムアウトがある場合はそれによって制限されます。これらはエージェント実行全体を管理するためです。明示的なモデルまたはエージェントのタイムアウトがない Cron トリガーのクラウドモデル実行では、同じデフォルトが使用されます。明示的な Cron 実行タイムアウトがある場合、クラウドモデルストリームの停止は 60s に制限されるため、外側の Cron 期限前に設定済みのモデルフォールバックを実行できます。Cron トリガーのローカルまたはセルフホストのモデル実行では、明示的なタイムアウトが設定されていない限り、暗黙の監視は無効になります。低速なローカルプロバイダーには `models.providers.<id>.timeoutSeconds` を設定してください。 |
| プロバイダー HTTP リクエストタイムアウト | `models.providers.<id>.timeoutSeconds` | そのプロバイダーの接続、ヘッダー、本文、SDK リクエストタイムアウト、保護された fetch の中止処理、およびモデルストリームのアイドル監視を対象にします。エージェントランタイム全体のタイムアウトを引き上げる前に、低速なローカルまたはセルフホストのプロバイダー（例: Ollama）に使用してください。モデルリクエストをより長く実行する必要がある場合は、エージェントまたはランタイムのタイムアウトを少なくとも同じ長さに保ってください。 |

### 停止セッションの診断

診断が有効な場合、`diagnostics.stuckSessionWarnMs`（デフォルトは `120000` ms）は、観測された返信、ツール、ステータス、ブロック、または ACP の進行がない長時間の `processing` セッションを分類します。

- アクティブな埋め込み実行、モデル呼び出し、ツール呼び出しは `session.long_running` として報告されます。所有されている無音のモデル呼び出しは、低速または非ストリーミングのプロバイダーが早すぎる段階で停止として扱われないよう、`diagnostics.stuckSessionAbortMs` まで `session.long_running` のままです。
- 最近の進行がないアクティブな作業は `session.stalled` として報告されます。所有されているモデル呼び出しは、中止しきい値以降に `session.stalled` に切り替わります。所有者のない古いモデルまたはツールのアクティビティは、長時間実行中として隠されません。
- `session.stuck` は、所有者のない古いモデルまたはツールのアクティビティを伴うアイドル状態のキュー済みセッションなど、回復可能な古いセッション管理情報のために予約されています。

`diagnostics.stuckSessionAbortMs` のデフォルトは、少なくとも 5 分かつ警告しきい値の 3 倍です。古いセッション管理情報は、回復ゲートを通過した直後に影響を受けたセッションレーンを解放します。停止した埋め込み実行は中止しきい値の後にのみ中止ドレインされるため、単に遅いだけの実行を打ち切ることなく、キュー済みの作業が再開されます。回復は構造化された requested/completed 結果を発行します。診断状態は、同じ処理世代がまだ現在のものである場合にのみアイドルとしてマークされ、繰り返される `session.stuck` 診断はセッションが変わらない間バックオフします。

## 早期終了が発生し得る場所

- エージェントタイムアウト（中止）
- AbortSignal（キャンセル）
- Gateway の切断または RPC タイムアウト
- `agent.wait` タイムアウト（待機のみ、エージェントは停止しません）
