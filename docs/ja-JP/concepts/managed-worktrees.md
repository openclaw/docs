管理対象ワークツリーは、一時ディレクトリをソースリポジトリ内に配置することなく、エージェントタスク専用の git ブランチとチェックアウトを提供します。OpenClaw はそれらを状態ディレクトリ配下に作成し、共有状態データベースに記録し、削除前に追跡対象の内容と無視されていない未追跡の内容のスナップショットを作成します。

## レイアウトと名前

各ワークツリーは次の場所に配置されます。

```text
<openclaw-state-dir>/worktrees/<repo-fingerprint>/<name>
```

リポジトリフィンガープリントは、正規化された git 共通ディレクトリと origin URL を対象とする SHA-256 ハッシュの先頭16桁の16進文字です。指定する名前は `[a-z0-9][a-z0-9-]{0,63}` に一致する必要があります。名前を指定しない場合、OpenClaw は `wt-` にランダムな8桁の16進文字を続けた名前を生成します。

OpenClaw は、要求されたベース ref にブランチ `openclaw/<name>` を作成します。ベース ref を指定しない場合は `origin` をフェッチし、利用可能であればリモートのデフォルトブランチを使用し、リポジトリがオフラインまたは使用可能なリモートがない場合はローカルの `HEAD` にフォールバックします。

## 無視対象ファイルのプロビジョニング

選択した無視対象の未追跡ファイルを新しいワークツリーへコピーするには、ソースリポジトリのルートに `.worktreeinclude` を追加します。このファイルでは gitignore パターン構文を使用し、1行につき1パターンを記述し、`#` でコメントを記述します。

```gitignore
.env.local
fixtures/generated/**
```

git によって無視対象かつ未追跡と報告されたファイルだけが対象です。追跡対象ファイルはすでに git によって存在するため、この手順ではコピーされません。OpenClaw はコピー先のファイルを上書きせず、シンボリックリンクされたディレクトリをたどらず、コピーしたファイルのモードを保持します。

## リポジトリセットアップの実行

ソースリポジトリに `.openclaw/worktree-setup.sh` が存在し、実行可能である場合、OpenClaw は新しいワークツリーをカレントディレクトリとしてそれを実行します。スクリプトには次の値が渡されます。

```text
OPENCLAW_SOURCE_TREE_PATH=<source checkout>
OPENCLAW_WORKTREE_PATH=<managed worktree>
```

0以外の終了コードでは作成が中止され、新しいワークツリーとブランチが削除されます。これはリポジトリローカルの契約であり、対応する OpenClaw の設定キーはありません。

## セッションワークツリー

アクティブなエージェントの git ワークスペースから、ワークツリーを基盤とするセッションで分離されたチャットを開始できます。Control UI の New session ページで **Worktree** を有効にするか（ベースブランチの選択と任意のワークツリー名の指定も可能）、iOS の Chat actions メニュー、または Android の New Chat の横にあるオーバーフローアクションを使用します。このオプションは、クライアントがその機能に対応している git ベースのエージェントでのみ利用できます。事前チェックできないクライアントでは、代わりに Gateway エラーが表示されます。

コーディングエージェントは、現在のタスク外に確認済みの後続作業を見つけた場合、`spawn_task` を呼び出すこともできます。Control UI は何も開始せずに提案チップを表示し、Gateway を基盤とする TUI は同じアクションを備えた対話型プロンプトを表示します。**Start in worktree** を選択すると、提案されたプロジェクトから新しいセッション所有のワークツリーを作成し、自己完結したプロンプトを最初のターンとして送信します。提案を閉じた場合、リポジトリは変更されません。提案とその ID は一時的なものであり、Gateway の再起動後には保持されません。

OpenClaw は、操作可能な Gateway UI を備えたオペレーターセッションにのみ、これらのツールを公開します。チャンネルセッションとローカルまたは組み込み TUI セッションには、それらのサーフェスで移植可能な型付きタスクアクション契約が利用可能になるまで提供されません。

生成された管理対象ワークツリーはセッションが所有し、そのセッション内のすべてのエージェント実行でそのチェックアウトが使用されます。ワークスペースがリポジトリのサブディレクトリである場合、ワークツリーはリポジトリルートを基準とし、セッションはその内部の対応するサブディレクトリから実行されます。セッションワークツリーの作成にはメソッドの `operator.write` スコープが使用されますが、`.openclaw/worktree-setup.sh` の手順はリポジトリコードを実行するため、`operator.admin` の呼び出し元でのみ実行されます。`.worktreeinclude` によるプロビジョニングは、引き続きすべての呼び出し元に適用されます。セッションを削除すると、損失なく削除できる場合に限りワークツリーも削除されます。変更のあるワークツリーやプッシュされていないコミットを含むブランチは保持されます。1時間ごとのクリーンアップでは、最近のセッションアクティビティをワークツリーのアクティビティとして扱い、7日間アイドル状態のセッションワークツリーのスナップショットを作成します。削除されたワークツリーは、以下の説明に従ってスナップショットから復元できます。

`sessions.create` には、設定済みのエージェントワークスペース以外のプロジェクトをタスクの対象とする場合、絶対パスの `cwd` と `worktree: true` を含めることができます。この明示的なホストパスには `operator.admin` が必要です。通常のワークツリーチャット作成には引き続き `operator.write` が使用され、設定済みのワークスペースを基準とします。

`sessions.create` は、`worktree: true` とともに `worktreeBaseRef` および `worktreeName` も受け付け、ベース ref とワークツリー名を選択できます（ブランチは `openclaw/<name>` になります）。どちらも `operator.write` のままです。作成されたワークツリーは作成結果で返され、セッション行に `worktree: { id, branch, repoRoot }` として永続化されるため、セッション一覧にチェックアウトとブランチを表示できます。セッションを削除した際に変更のあるチェックアウトが保持された場合は、黙って残すのではなく `worktreePreserved` として報告されます。

## スナップショット、クリーンアップ、復元

削除時には、まず追跡対象ファイルと無視されていない未追跡ファイルを含む合成コミットを作成し、`refs/openclaw/snapshots/<id>` に固定します。gitignore の対象ファイルはリポジトリオブジェクトデータベースから除外されます。`.worktreeinclude` で選択されたファイルは、復元時に再度コピーされます。スナップショットの作成に失敗した場合、削除は停止します。明示的な強制削除では、スナップショットなしで続行できます。

OpenClaw は次のクリーンアップルールを適用します。

- 実行終了時、`git status --porcelain` が空であり、かつ `git log HEAD --not --remotes --oneline` でプッシュされていないコミットが見つからない場合に限り、ワークツリーを削除します。それ以外の場合は、アクティビティロックのみを解放します。
- 1時間ごとのクリーンアップでは、変更の有無にかかわらず、7日を超えてアイドル状態でロックされていない Workboard 所有およびセッション所有のワークツリーのスナップショットを作成して削除します。手動ワークツリーは自動削除されません。
- スナップショットレコードは30日間復元可能です。その後、クリーンアップによってスナップショット ref とレジストリ行が削除されます。
- 稼働中の OpenClaw プロセスロック、および外部または認識されていない git ワークツリーロックが、ワークツリーをガベージコレクションから保護します。

復元では、元のスナップショット作成前のコミットに `openclaw/<name>` を再作成し、スナップショットの差分をステージされていない変更と未追跡ファイルとして再構築します。これにより、合成スナップショットコミットがブランチ履歴に含まれません。スナップショット ref は出所情報として記録されたままになります。

## CLI

```bash
openclaw worktrees list [--json]
openclaw worktrees create <repo-root> [--name <name>] [--base-ref <ref>] [--json]
openclaw worktrees remove <id> [--force] [--json]
openclaw worktrees restore <id> [--json]
openclaw worktrees gc [--json]
```

Settings 配下にある Control UI の **Worktrees** ページでは、同じアクションに加えてベースブランチ選択による作成を行えます。また、各ワークツリーの所有者（手動、Workboard、またはそのチャットへのリンクを含む所有セッション）を表示し、削除時にスナップショット失敗が報告された場合は強制再試行を提供します。

## Gateway メソッド

| メソッド             | 目的                                                                    |
| -------------------- | ----------------------------------------------------------------------- |
| `worktrees.list`     | アクティブおよび復元可能なワークツリーレコードを一覧表示します。        |
| `worktrees.branches` | ベース ref 選択用にリポジトリのローカルおよびリモートブランチを一覧表示します。 |
| `worktrees.create`   | 名前付き管理対象ワークツリーを作成または再利用します。                  |
| `worktrees.remove`   | ワークツリーのスナップショットを作成して削除します。強制削除では `snapshotError` が報告されます。 |
| `worktrees.restore`  | 削除されたワークツリーをスナップショットから復元します。                |
| `worktrees.gc`       | アイドル、孤立、保持期間のクリーンアップを直ちに実行します。            |

`worktrees.list` には `operator.read` が必要であり、変更を行うメソッドには `operator.admin` が必要です。`worktrees.branches` は、設定済みのエージェントワークスペースでは `operator.write` が必要ですが、それ以外のホストパスでは `operator.admin` が必要です（`sessions.create` の cwd に対する基準と同じです）。既存の ref のみを読み取り、フェッチは行いません。リモートにのみ存在するブランチはリモート修飾付き（`origin/feature-a`）で返されるため、返されたすべての名前をベース ref として解決できます。

## Workboard ワークスペース

同梱の [Workboard Plugin](/ja-JP/plugins/workboard) は、カードのワークスペースを管理対象ワークツリーとして実体化できます。

```json
{
  "kind": "worktree",
  "path": "/absolute/path/to/source-checkout",
  "branch": "main"
}
```

`path` はソース git チェックアウトを識別します。`branch` は任意であり、ベース ref になります。ディスパッチによってカードのワーカーが開始されると、Workboard は `wb-<card-id>` を作成または再利用し、管理対象チェックアウトを作業ディレクトリとしてサブエージェントを実行し、解決されたパスとブランチをカードへ書き戻します。Gateway から実行される実体化には `operator.admin` が必要です。実行終了時、Workboard は損失がないことを明確に証明できる場合に限りチェックアウトを削除します。変更のある作業やプッシュされていないコミットは保持されます。

サンドボックス化された組み込みエージェントは現在、設定済みのエージェントワークスペース外にあるタスク作業ディレクトリを拒否します。サンドボックスランタイムが追加のチェックアウトマウントをサポートするまでは、Workboard の管理対象ワークツリーカードにはサンドボックス化されていない対象エージェントを使用してください。
