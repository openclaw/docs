管理対象ワークツリーは、一時ディレクトリをソースリポジトリ内に配置せずに、エージェントタスク専用の git ブランチとチェックアウトを提供します。OpenClaw はこれらを状態ディレクトリ配下に作成し、共有状態データベースに記録し、削除前に追跡対象および無視対象でない未追跡コンテンツのスナップショットを作成します。

## レイアウトと名前

各ワークツリーは次の場所にあります。

```text
<openclaw-state-dir>/worktrees/<repo-fingerprint>/<name>
```

リポジトリフィンガープリントは、正規化された git 共通ディレクトリと origin URL に対する SHA-256 ハッシュの先頭 16 桁の 16 進文字です。指定する名前は `[a-z0-9][a-z0-9-]{0,63}` に一致する必要があります。名前を指定しない場合、OpenClaw は `wt-` にランダムな 8 桁の 16 進文字を続けた名前を生成します。

OpenClaw は、要求されたベース ref にブランチ `openclaw/<name>` を作成します。ベース ref を指定しない場合は `origin` をフェッチし、利用可能であればリモートのデフォルトブランチを使用し、リポジトリがオフラインまたは使用可能なリモートがない場合はローカルの `HEAD` にフォールバックします。

## 無視対象ファイルのプロビジョニング

選択した無視対象の未追跡ファイルを新しいワークツリーへコピーするには、ソースリポジトリのルートに `.worktreeinclude` を追加します。このファイルでは gitignore パターン構文を使用し、1 行につき 1 パターンを記述でき、`#` コメントも使用できます。

```gitignore
.env.local
fixtures/generated/**
```

対象となるのは、git が無視対象かつ未追跡として報告するファイルだけです。追跡対象ファイルは git によってすでに存在するため、この手順ではコピーされません。OpenClaw はコピー先のファイルを上書きせず、シンボリックリンクされたディレクトリをたどらず、コピーしたファイルのモードを保持します。

## リポジトリセットアップの実行

ソースリポジトリに `.openclaw/worktree-setup.sh` が存在し、実行可能な場合、OpenClaw は新しいワークツリーをカレントディレクトリとしてこれを実行します。スクリプトには次の値が渡されます。

```text
OPENCLAW_SOURCE_TREE_PATH=<source checkout>
OPENCLAW_WORKTREE_PATH=<managed worktree>
```

0 以外の終了コードでは作成が中止され、新しいワークツリーとブランチが削除されます。これはリポジトリローカルの契約であり、対応する OpenClaw の設定キーはありません。

## セッションワークツリー

アクティブなエージェントの git ワークスペースから、ワークツリーを使用するセッションで分離されたチャットを開始できます。Control UI の New session ページで **Worktree** を有効にする（このページにはベースブランチの選択と、任意のワークツリー名の入力もあります）か、iOS では Chat actions メニュー、Android では New Chat の横にあるオーバーフローアクションを使用します。このオプションを利用できるのは、クライアントがその機能を備えている git ベースのエージェントだけです。事前確認できないクライアントでは、代わりに Gateway エラーが表示されます。

コーディングエージェントは、現在のタスクの範囲外にある確認済みのフォローアップ作業を発見した場合、`spawn_task` を呼び出すこともできます。Control UI は何も開始せずに提案チップを表示し、Gateway を使用する TUI は同じアクションを含む対話型プロンプトを表示します。**Start in worktree** を選択すると、提案されたプロジェクトからセッション所有の新しいワークツリーを作成し、自己完結したプロンプトを最初のターンとして送信します。提案を閉じた場合、リポジトリは変更されません。提案とその ID は一時的なもので、Gateway の再起動後には保持されません。

OpenClaw は、操作可能な Gateway UI を持つオペレーターセッションにのみ、これらのツールを公開します。チャネルセッションとローカル／組み込み TUI セッションには、これらのサーフェスが移植可能な型付きタスクアクション契約を備えるまで提供されません。

生成された管理対象ワークツリーはセッションによって所有され、そのセッション内のすべてのエージェント実行でこのチェックアウトが使用されます。ワークスペースがリポジトリのサブディレクトリである場合、ワークツリーはリポジトリルートを基準に作成され、セッションはその内部の対応するサブディレクトリから実行されます。セッションワークツリーの作成ではメソッドの `operator.write` スコープを使用しますが、`.openclaw/worktree-setup.sh` の手順はリポジトリコードを実行するため、`operator.admin` の呼び出し元に対してのみ実行されます。`.worktreeinclude` によるプロビジョニングは、すべての呼び出し元に引き続き適用されます。セッションを削除すると、損失なく削除できる場合にのみワークツリーも削除されます。変更のあるワークツリーや未プッシュのコミットを含むブランチは保持されます。1 時間ごとのクリーンアップでは、最近のセッションアクティビティをワークツリーのアクティビティとして扱い、7 日間アイドル状態のセッションワークツリーのスナップショットを作成します。削除されたワークツリーは、以下の説明に従ってスナップショットから復元できます。

タスクが設定済みのエージェントワークスペースとは異なるプロジェクトを対象とする場合、`sessions.create` には `worktree: true` とともに絶対パスの `cwd` を含めることができます。この明示的なホストパスには `operator.admin` が必要です。通常のワークツリーチャット作成は引き続き `operator.write` で、設定済みのワークスペースを基準とします。

`sessions.create` は、`worktree: true` とともに `worktreeBaseRef` と `worktreeName` も受け付け、ベース ref とワークツリー名を選択できます（ブランチは `openclaw/<name>` になります）。どちらも `operator.write` のままです。作成されたワークツリーは作成結果で返され、セッション行に `worktree: { id, branch, repoRoot }` として永続化されるため、セッション一覧にチェックアウトとブランチを表示できます。セッションを削除した際、変更のあるチェックアウトが保持された場合は、暗黙的に残すのではなく `worktreePreserved` として報告されます。

## スナップショット、クリーンアップ、復元

削除時には、最初に追跡対象ファイルと無視対象でない未追跡ファイルを含む合成コミットを作成し、`refs/openclaw/snapshots/<id>` に固定します。gitignore の対象ファイルはリポジトリオブジェクトデータベースから除外されます。`.worktreeinclude` で選択されたファイルは、復元時に再度コピーされます。スナップショットの作成に失敗した場合、削除は停止します。明示的な強制削除では、スナップショットなしで続行できます。

OpenClaw は次のクリーンアップ規則を適用します。

- 実行終了時、OpenClaw は `git status --porcelain` が空で、かつ `git log HEAD --not --remotes --oneline` で未プッシュのコミットが見つからない場合にのみワークツリーを削除します。それ以外の場合は、アクティビティロックのみを解放します。
- 1 時間ごとのクリーンアップでは、7 日を超えてアイドル状態の、ロックされていない Workboard 所有およびセッション所有のワークツリーについて、変更の有無にかかわらずスナップショットを作成して削除します。手動ワークツリーが自動的に削除されることはありません。
- スナップショットレコードは 30 日間復元可能な状態で保持されます。その後、クリーンアップによってスナップショット ref とレジストリ行が削除されます。
- 稼働中の OpenClaw プロセスのロック、および外部または認識されない git ワークツリーロックによって、ワークツリーはガベージコレクションから保護されます。

復元では、元のスナップショット前のコミットに `openclaw/<name>` を再作成し、スナップショットの差分をステージされていない変更と未追跡ファイルとして再構築します。これにより、合成スナップショットコミットはブランチ履歴に含まれません。スナップショット ref は来歴として記録されたままです。

## CLI

```bash
openclaw worktrees list [--json]
openclaw worktrees create <repo-root> [--name <name>] [--base-ref <ref>] [--json]
openclaw worktrees remove <id> [--force] [--json]
openclaw worktrees restore <id> [--json]
openclaw worktrees gc [--json]
```

Settings 配下の Control UI **Worktrees** ページでは、同じアクションに加えてベースブランチを選択して作成できます。また、各ワークツリーの所有者（手動、Workboard、またはそのチャットへのリンク付きの所有セッション）が表示され、削除時にスナップショットの失敗が報告された場合は強制再試行を実行できます。

## Gateway メソッド

| メソッド             | 目的                                                                    |
| -------------------- | ----------------------------------------------------------------------- |
| `worktrees.list`     | アクティブおよび復元可能なワークツリーレコードを一覧表示します。        |
| `worktrees.branches` | ベース ref 選択用に、リポジトリのローカルおよびリモートブランチを一覧表示します。 |
| `worktrees.create`   | 名前付き管理対象ワークツリーを作成または再利用します。                  |
| `worktrees.remove`   | ワークツリーのスナップショットを作成して削除します。強制削除では `snapshotError` が報告されます。 |
| `worktrees.restore`  | 削除されたワークツリーをスナップショットから復元します。                |
| `worktrees.gc`       | アイドル、孤立、および保持期間に関するクリーンアップを今すぐ実行します。 |

`worktrees.list` には `operator.read` が必要で、変更を加えるメソッドには `operator.admin` が必要です。`worktrees.branches` は、設定済みのエージェントワークスペースでは `operator.write` が必要で、それ以外のホストパスでは `operator.admin` が必要です（`sessions.create` の cwd に関する基準と同じです）。既存の ref のみを読み取り、フェッチは行いません。また、リモートにのみ存在するブランチはリモート修飾付き（`origin/feature-a`）で返されるため、返されるすべての名前をベース ref として解決できます。

## Workboard ワークスペース

バンドルされている [Workboard Plugin](/ja-JP/plugins/workboard) は、カードのワークスペースを管理対象ワークツリーとして実体化できます。

```json
{
  "kind": "worktree",
  "path": "/absolute/path/to/source-checkout",
  "branch": "main"
}
```

`path` はソース git チェックアウトを指定します。`branch` は任意で、ベース ref になります。ディスパッチによってカードのワーカーが開始されると、Workboard は `wb-<card-id>` を作成または再利用し、管理対象チェックアウトを作業ディレクトリとしてサブエージェントを実行し、解決済みのパスとブランチをカードへ書き戻します。Gateway からトリガーされる実体化には `operator.admin` が必要です。実行終了時、Workboard は損失なく削除できることが証明できる場合にのみチェックアウトを削除します。変更のある作業や未プッシュのコミットは保持されます。

サンドボックス化された組み込みエージェントは現在、設定済みのエージェントワークスペース外にあるタスク作業ディレクトリを拒否します。サンドボックスランタイムが追加のチェックアウトマウントをサポートするまでは、Workboard の管理対象ワークツリーカードにはサンドボックス化されていない対象エージェントを使用してください。
