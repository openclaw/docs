進行状況ドラフトは、エージェントが作業している間、1 件のチャンネルメッセージをライブのステータス行に変え、一時的な「まだ作業中」という返信が積み重ならないようにします。
`channels.<channel>.streaming.mode: "progress"` を設定すると、OpenClaw は実際の作業が始まった時点でメッセージを一度作成し、エージェントが読み取り、計画し、ツールを呼び出し、または承認を待つたびに編集し、最後に最終回答へ変えます。

```text
シェルを実行中...
📖 docs/concepts/progress-drafts.md から読み取り
🔎 Web 検索: 「discord edit message」を検索
🛠️ Bash: テストを実行
```

<Note>
  `channels.discord.streaming` が未設定の場合、Discord ではすでに
  `streaming.mode: "progress"` がデフォルトであるため、設定なしで
  進行状況ドラフトが表示されます。その他のすべてのチャンネルでは `partial`
  または `off` がデフォルトです。チャンネルごとのデフォルト一覧については、
  [ストリーミングとチャンク化](/ja-JP/concepts/streaming#channel-mapping)を参照してください。
</Note>

## クイックスタート

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
      },
    },
  },
}
```

ここからのデフォルトは、自動生成される 1 語のラベル、5 秒の開始遅延
（または 2 回目の作業イベントで即時開始）、有用な作業の実行中に表示される
簡潔な進行状況行、およびそのターンで従来の独立した進行状況メッセージを
抑制する動作です。

このページでは、進行状況ドラフトの使用感と設定項目について説明します。
ストリーミングモードの完全な対応表、チャンネルごとのランタイムに関する注意事項、
および旧キーの移行については、[ストリーミングとチャンク化](/ja-JP/concepts/streaming)を
参照してください。

## ユーザーに表示される内容

| 部分           | 目的                                                                           |
| -------------- | --------------------------------------------------------------------------------- |
| ラベル          | `Working` や `Shelling` などの短い開始／ステータス行。                        |
| 進捗行 | `/verbose` と同じツールアイコンと詳細フォーマッターを使用した簡潔な実行状況の更新。 |

エージェントが意味のある作業を開始し、初期遅延の間も処理中である場合、または2回目の作業イベントが直ちに発生した場合に、ラベルが表示されます。ラベルは更新され続ける進捗行リストの先頭に配置されるため、具体的な作業行が十分に表示されるとスクロールして見えなくなります。プレーンテキストのみの応答では進捗の下書きは表示されません。行が表示されるのは実際の作業更新がある場合のみです。たとえば、`🛠️ Bash: run tests`、`🔎 Web Search: for "discord edit message"`、`✍️ Write: to /tmp/file` などです。

チャンネルで安全に置き換えられる場合、最終回答はその場で下書きを置き換えます。そうでない場合、OpenClaw は通常の配信経路で最終回答を送信し、下書きを削除するか更新を停止します（[完了処理](#finalization)を参照）。

## モードを選択する

`channels.<channel>.streaming.mode` は、処理中に表示される動作を制御します。

| モード       | 最適な用途                         | チャットに表示される内容                              |
| ---------- | -------------------------------- | ------------------------------------------------- |
| `off`      | 静かなチャンネル                   | 最終回答のみ。                            |
| `partial`  | 回答テキストが表示されていく様子の確認      | 最新の回答テキストで編集される1つの下書き。     |
| `block`    | より大きな回答プレビューチャンク     | より大きなチャンク単位で更新または追記される1つのプレビュー。 |
| `progress` | ツール使用が多い、または長時間実行されるターン | 1つのステータス下書きと、その後の最終回答。          |

回答テキストがトークン単位でストリーミングされる様子を見ることよりも、「何が起きているか」をユーザーが重視する場合は `progress` を選択します。回答テキスト自体を進捗のシグナルとする場合は `partial`、より大きなプレビューチャンクには `block` を選択します。Discord と Telegram では、`streaming.mode: "block"` は引き続きプレビューストリーミングであり、通常のブロック応答配信ではありません。通常のブロック応答配信には `streaming.block.enabled` を使用してください。

## ラベルを設定する

進行状況ラベルは `channels.<channel>.streaming.progress` で設定します。
デフォルトの `label` は `"auto"` で、OpenClaw に組み込まれている単語ラベルの
プールから選択されます。

```text
作業中、シェル実行中、移動中、爪とぎ中、つまみ中、脱皮中、泡立ち中、潮流中、
岩礁探索中、殻割り中、選別中、塩漬け中、オウムガイ中、オキアミ中、フジツボ中、
ロブスター中、潮だまり探索中、真珠採り中、ハサミ鳴らし中、浮上中
```

固定ラベルを使用する場合:

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          label: "調査中",
        },
      },
    },
  },
}
```

独自のラベルプールを使用する場合（`label: "auto"` のときは、引き続きランダムまたはシードに基づいて選択されます）:

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          label: "auto",
          labels: ["確認中", "読み取り中", "テスト中", "仕上げ中"],
        },
      },
    },
  },
}
```

ラベルを非表示にして、進行状況行のみを表示する場合:

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          label: false,
        },
      },
    },
  },
}
```

## 進行状況行を制御する

進行状況行は、ツールの開始、項目の更新、タスク計画、承認、コマンド出力、
パッチの要約など、実際の実行イベントや同様のエージェントアクティビティから生成されます。
これはデフォルトで有効です（`progress.toolProgress`、デフォルトは `true`）。

また、1 回の呼び出しがまだ実行中でも、ツールは型付きの進行状況を送信できます。
これにより、時間のかかる取得や検索では、ツールが最終結果を返す前に表示中の下書きを
更新できます。進行状況の更新は、モデル向けコンテンツが空で、公開チャンネルの
メタデータが明示された部分的なツール結果です。

```json
{
  "content": [],
  "progress": {
    "text": "ページのコンテンツを取得中...",
    "visibility": "channel",
    "privacy": "public",
    "id": "web_fetch:fetching"
  }
}
```

OpenClaw はチャンネルの進行状況 UI に `progress.text` のみを表示します。通常の
ツール結果は後から `content`/`details` として到着し、モデルに返されるのは
この部分だけです。

ツールに進行状況を追加する場合は、短く汎用的なメッセージを送信し、処理の保留時間が
十分に長く、表示する意味が生じるまで遅延させます。`web_fetch` では、まさにこれを
5 秒の遅延で行います。

```typescript
const clearProgressTimer = scheduleToolProgress(
  onUpdate,
  { text: "ページのコンテンツを取得中...", id: "web_fetch:fetching" },
  5_000,
  { signal },
);

try {
  return await runToolWork();
} finally {
  clearProgressTimer();
}
```

高速な呼び出しでは進行状況の行は表示されません。長時間の呼び出しでは、保留中に進行状況の行が表示されます。
キャンセルされた呼び出しでは、古い進行状況が表示される前にタイマーがクリアされます。進行状況のテキストは
公開 UI のサイドチャネルであるため、シークレット、生の引数、
取得したコンテンツ、コマンド出力、ページのテキストを決して含めてはなりません。

### 詳細モード

OpenClaw は、進行状況の下書きと `/verbose` に同じフォーマッターを使用します。

```json5
{
  agents: {
    defaults: {
      toolProgressDetail: "explain", // explain | raw
    },
  },
}
```

`"explain"` はデフォルトであり、簡潔なラベルによって下書きを安定させます。
`"raw"` は、利用可能な場合に基盤となるコマンドを追加します。これは
デバッグ中には便利ですが、チャットではノイズが増えます。たとえば、`node --check /tmp/app.js` の呼び出しは
モードによって表示が異なります。

| モード      | 進行状況行                                                   |
| --------- | --------------------------------------------------------------- |
| `explain` | `🛠️ check js syntax for /tmp/app.js`                            |
| `raw`     | `🛠️ check js syntax for /tmp/app.js · node --check /tmp/app.js` |

### コマンド/exec テキスト

`streaming.progress.commandText`（デフォルトは `"raw"`）は、上記の詳細モードとは独立して、exec/bash の進行状況行の横に表示するコマンドの詳細度を制御します。コマンドテキストを完全に非表示にしつつ、ツールの進行状況行を表示したままにするには、`"status"` に設定します。

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          commandText: "status",
        },
      },
    },
  },
}
```

### コメンタリーレーン

`streaming.progress.commentary`（デフォルトは `false`）は、モデルによるツール実行前のコメンタリー／前置きの説明（💬、たとえば「確認してから……」）を、下書き内のツール行と交互に表示します。チャネル間で共通の設定形式については、[ストリーミングとチャンク分割](/ja-JP/concepts/streaming#commentary-progress-lane)を参照してください。

### 状況のナレーション

エージェントに対してユーティリティモデルが解決される場合（明示的な
[`utilityModel`](/ja-JP/gateway/config-agents#utilitymodel)、またはプライマリ
プロバイダーが宣言した小規模モデルのデフォルト（OpenAI → `gpt-5.6-luna`、
Anthropic → `claude-haiku-4-5`））、進捗の下書きでは、逐次追加される
ツール行が、エージェントの実行内容を説明する短い平易なナレーションに
置き換わります。このナレーションは、より低コストなモデルによって作成され、
作業の進行に合わせて更新されます。

```text
作業中

設定内のデフォルトモデルを更新してから、変更を反映するために Gateway を
再起動しています。エージェント一覧の取得呼び出しが 1 回失敗したため、再試行しています。
```

ナレーションはデフォルトで有効です（`streaming.progress.narration`、デフォルトは `true`）。
プライマリモデルへフォールバックすることはありません。明示的な
`utilityModel`、またはエージェントのプライマリプロバイダー向けに
プロバイダーが宣言したデフォルトがある場合にのみ実行されます。
ユーティリティルーティングを完全に無効にするには、`utilityModel: ""` を設定します。
ツール行はその下で蓄積され続け、ナレーションが停止すると再び表示されます。
また、下書きはナレーションのテキストが実際に変更された場合にのみ編集されるため、
利用の多いチャネルでの編集の頻発も抑えられます。未加工のツール行を維持するには、
ナレーションを無効にします。

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          narration: false,
        },
      },
    },
  },
}
```

ナレーション入力は範囲が限定され、秘匿化されています。ユーティリティモデルが受け取るのは、
受信リクエストのテキストと、下書きに表示されるものと同じ、簡潔で秘匿化されたツールの要約です。
生のコマンド出力やツール結果を受け取ることはありません。
`commandText: "status"` の場合、ナレーション入力では exec/bash コマンドのテキストも省略され、
下書きに表示される内容と一致します。

### 行数制限

表示されたままにする行数を制限します（デフォルトは 8）。

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          maxLines: 4,
        },
      },
    },
  },
}
```

下書きの編集中にチャットバブルの再レイアウトを抑えるため、進行状況の行は自動的に圧縮されます。
また、下書きを繰り返し編集しても更新のたびに異なる位置で折り返されないように、OpenClaw は長い行を切り詰めます。
デフォルトの 1 行あたりの上限は 120 文字です。文章は単語の境界で切り詰められますが、パスや
生のコマンドなどの長い詳細は、末尾が見えるように中央の省略記号で短縮されます。

1 行あたりの上限を調整します。

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          maxLineChars: 160,
        },
      },
    },
  },
}
```

### リッチレンダリング（Slack）

Slack では、進行状況の行をプレーンテキストではなく、構造化された Block Kit フィールドとして
レンダリングできます。

```json5
{
  channels: {
    slack: {
      streaming: {
        mode: "progress",
        progress: {
          render: "rich",
        },
      },
    },
  },
}
```

リッチレンダリングでは、Block Kit フィールドとともに、常に同じプレーンテキスト本文も送信されます。
そのため、リッチな形式をレンダリングできないクライアントでも、簡潔な進行状況テキストが表示されます。

### ツール／タスク行を非表示にする

単一の進行状況の下書きを維持しながら、ツールとタスクの行を非表示にします。

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          toolProgress: false,
        },
      },
    },
  },
}
```

`toolProgress: false` の場合でも、OpenClaw はそのターンにおける従来の個別の
ツール進行状況メッセージを抑制します。ラベルが設定されている場合はそのラベルを除き、
最終回答までチャンネルの表示は静かなままです。

## チャンネルの動作

| チャネル        | 進捗の転送方式                               | 注記                                                                                                                                                                                  |
| --------------- | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Discord         | 1件のメッセージを送信し、その後編集します。 | デフォルトは`progress`モードです。最終回答には`-#`アクティビティ受領情報が付き、回答の送信後にステータス下書きが削除されます。                                                       |
| Matrix          | 1件のイベントを送信し、その後編集します。   | アカウントレベルのストリーミング設定によって、アカウントレベルの下書きが制御されます。                                                                                              |
| Microsoft Teams | 個人チャットでのTeamsネイティブストリーム。 | 代わりに`streaming.mode: "block"`はTeamsのブロック配信にマッピングされます。                                                                                                        |
| Slack           | ネイティブストリームまたは編集可能な下書き投稿。 | 返信スレッドの対象が必要です。対象のないトップレベルのDMでも、下書きのプレビュー投稿と編集は行われます。                                                                         |
| Telegram        | 1件のメッセージを送信し、その後編集します。 | 進捗下書きと回答の間に別のメッセージが届いた場合、クライアントのスクロール位置を急に移動させる代わりに、下書きをそのメッセージの下へ再投稿します（新規投稿後に古い投稿を削除）。 |
| Mattermost      | 編集可能な下書き投稿。                       | `block`モードでは、完成済みテキストの投稿とツールアクティビティの投稿が切り替わります。その他のモードでは、ツールアクティビティが同じ下書き形式の投稿に組み込まれます。             |

安全な編集をサポートしないチャネルでは、入力中インジケーターまたは
最終回答のみの配信にフォールバックします。チャネルごとの完全なランタイム動作の
内訳については、[ストリーミングとチャンク分割](/ja-JP/concepts/streaming)を参照してください。

## 最終処理

最終回答の準備が整うと、OpenClawはチャットを整理された状態に保とうとします。

- Discordの`progress`モードでは、最終回答は新しいメッセージとして送信され、
  末尾に小さな`-#`アクティビティ受領情報（例:
  `-# 🧠 2 thoughts · 🛠️ 5 tool calls · ⏱️ 12s`）が追加されます。回答の配信後、
  ステータス下書きは削除されます。頻繁にメッセージが流れるチャネルでも、返信の上に
  孤立したツールログは残りません。エラーの最終回答では、失敗したターンの
  表示記録として下書きが保持されます。
- 下書きを安全に最終回答へ変換できる場合（`partial`/`block`モード）、
  OpenClawはその場で下書きを編集します。
- チャネルがネイティブの進捗ストリーミングを使用している場合、ネイティブ転送方式が
  最終テキストを受け入れた時点で、OpenClawはそのストリームを完了します。
- それ以外の場合（メディア、承認プロンプト、明示的な返信先、チャンクが多すぎる場合、
  または編集や送信の失敗）、OpenClawは下書きを上書きせず、通常のチャネル配信経路で
  最終回答を送信します。

このフォールバックは意図的なものです。新しい最終回答を送信する方が、テキストを失ったり、
返信を誤ったスレッドに送ったり、チャネルが安全に表現できないペイロードで下書きを
上書きしたりするよりも適切です。

## トラブルシューティング

**最終回答しか表示されません。**

メッセージを処理したアカウントまたはチャネルで、
`channels.<channel>.streaming.mode`が`progress`になっていることを確認してください。
一部のグループまたは引用返信の経路では、チャネルが適切なメッセージを安全に編集できない場合、
そのターンの下書きプレビューが無効になります。

**ラベルは表示されますが、ツール行が表示されません。**

`streaming.progress.toolProgress`を確認してください。`false`の場合、OpenClawは
単一下書きの動作を維持しますが、ツールとタスクの進捗行を非表示にします。

**編集された下書きではなく、新しい最終メッセージが表示されます。**

これは[最終処理](#finalization)で説明した安全フォールバックです。
メディアへの返信、長い回答、明示的な返信先、古いTelegramの下書き、
Slackのスレッド対象の欠如、削除されたプレビューメッセージ、または
ネイティブストリームの最終処理の失敗によって発生することがあります。

**単独の進捗メッセージがまだ表示されます。**

下書きが有効な間、進捗モードはデフォルトの単独ツール進捗メッセージを抑制します。
単独メッセージが引き続き表示される場合は、そのターンが実際に`progress`モードを
使用しており、`streaming.mode: "off"`や、そのメッセージの下書きを作成できない
チャネル経路を使用していないことを確認してください。

**Teamsの動作がDiscordやTelegramと異なります。**

Microsoft Teamsは、一般的な送信後編集型のプレビュー転送方式ではなく、
個人チャットでネイティブストリームを使用します。また、DiscordやTelegramのような
下書きプレビューのブロックモードがないため、`streaming.mode: "block"`を
Teamsのブロック配信にマッピングします。
