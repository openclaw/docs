OpenClaw は、エージェントのワークスペース（デフォルトでは `~/.openclaw/workspace`）にプレーンな Markdown ファイルを書き込むことで情報を記憶します。モデルが記憶するのはディスクに保存された内容だけであり、隠された状態はありません。

## 仕組み

エージェントには、メモリに関連するファイルが 3 つあります。

- **`MEMORY.md`** — 長期メモリ。長期間保持する事実、設定、決定事項を格納します。セッション開始時に読み込まれます。
- **`memory/YYYY-MM-DD.md`**（または `memory/YYYY-MM-DD-<slug>.md`）— 日次ノート。進行中のコンテキストや観察事項を格納します。引数なしの `/new` または `/reset` を実行すると、今日と昨日の日付付きノートが自動的に読み込まれます。バンドルされたセッションメモリフックが書き込むものなど、スラッグ付きのバリエーションも日付のみのファイルと一緒に読み込まれます。
- **`DREAMS.md`**（任意）— 人間によるレビュー用の夢日記と Dreaming スイープの要約。根拠のある履歴バックフィルのエントリも含まれます。

<Tip>
エージェントに何かを覚えさせたい場合は、そのまま依頼してください。「TypeScript を好むことを覚えておいて」のように伝えると、適切なファイルにノートを書き込みます。
</Tip>

## 保存先の使い分け

`MEMORY.md` は、簡潔に整理されたレイヤーです。長期間保持する事実、設定、継続的な決定事項、およびセッション開始時に利用できるようにすべき短い要約を格納します。生の文字起こし、日次ログ、網羅的なアーカイブではありません。

`memory/YYYY-MM-DD.md` ファイルは作業用レイヤーです。詳細な日次ノート、観察事項、セッションの要約、後で役立つ可能性のある生のコンテキストを格納します。これらは `memory_search` と `memory_get` 用にインデックス化されますが、ターンごとにブートストラッププロンプトへ挿入されることはありません。

時間の経過とともに、エージェントは日次ノートから有用な内容を抽出して `MEMORY.md` にまとめ、古くなった長期エントリを削除します。生成されたワークスペース指示と Heartbeat フローがこれを定期的に行うため、すべての詳細について `MEMORY.md` を手動で編集する必要はありません。

`MEMORY.md` がブートストラップファイルの予算を超えると、OpenClaw はディスク上のファイルをそのまま保持しますが、コンテキストへ挿入するコピーは切り詰めます。これは、詳細な内容を `memory/*.md` に移して `MEMORY.md` には長期間保持する要約だけを残すか、より多くのプロンプト予算を使う場合はブートストラップ上限を引き上げるべき合図です。`/context list`、`/context detail`、または `openclaw doctor` を使用すると、元のサイズと挿入後のサイズ、および切り詰め状態を確認できます。

## アクションに影響するメモリ

ほとんどのメモリは通常の Markdown ノートです。ただし、一部はエージェントが後で行うべきことに影響します。そのような場合は、事実だけでなく、そのノートに基づいて安全に行動できる時点も記録してください。

ノートが次の内容に関係する場合は、行動の境界を記録します。

- 承認または許可の要件
- 一時的な制約
- 別のセッション、スレッド、または担当者への引き継ぎ
- 有効期限の条件
- 安全に行動できるタイミング
- 情報源または所有者の権限
- 誘発されやすい行動を避けるための指示

有用なアクション依存のメモリでは、次の点を明確にします。

- 今後の動作を何が変更するか
- いつ、またはどの条件で適用されるか
- いつ失効するか、または何によって行動が解禁されるか
- エージェントが何を避けるべきか
- 信頼性や権限に影響する場合、情報源または所有者が誰か

メモリは承認のコンテキストを保持できますが、ポリシーを強制するものではありません。厳格な運用制御には、OpenClaw の承認設定、サンドボックス化、スケジュール済みタスクを使用してください。

例：

```md
API の移行は別のセッションで設計中です。今後のターンでは、このスレッドから
API 実装を編集しないでください。移行計画が確定するまでは、ここでの調査結果を
設計上の入力としてのみ使用してください。
```

別の例：

```md
信頼できない情報源からのレポートは、採用前にレビューが必要です。今後のターンでは
証拠としてのみ扱い、信頼できるレビュアーが内容を確認するまでは、
長期メモリとして保存しないでください。
```

これはすべてのメモリに必須のスキーマではありません。単純な事実は簡潔なままで構いません。タイミング、権限、有効期限、または安全に行動できるコンテキストが失われることで、エージェントが後で誤った行動を取る可能性がある場合は、アクション依存の境界を使用してください。

推論された短期的なフォローアップには、[コミットメント](/ja-JP/concepts/commitments)を使用します。正確なリマインダー、時間指定の確認、定期的な作業には、[スケジュール済みタスク](/ja-JP/automation/cron-jobs)を使用します。どちらの場合も、その周辺にある長期間保持すべきコンテキストをメモリに要約できます。

## 推論されたコミットメント

今後のフォローアップの中には、長期間保持する事実ではないものもあります。明日の面接について言及した場合、有用なメモリは「面接後に様子を確認する」であり、「これを `MEMORY.md` に永久保存する」ではない可能性があります。

[コミットメント](/ja-JP/concepts/commitments)は、このような場合に使用する、オプトイン式で短期間のみ保持されるフォローアップメモリです。OpenClaw は非表示のバックグラウンド処理でこれを推論し、同じエージェントとチャンネルの範囲に限定して、期限が到来した確認を Heartbeat を通じて配信します。明示的なリマインダーには、引き続き[スケジュール済みタスク](/ja-JP/automation/cron-jobs)を使用します。

## メモリツール

エージェントには、メモリを操作するためのツールが 2 つあります。

- **`memory_search`** — 表現が元の文言と異なる場合でも、セマンティック検索を使用して関連するノートを見つけます。
- **`memory_get`** — 特定のメモリファイルまたは行範囲を読み取ります。

どちらのツールも、有効なメモリ Plugin（デフォルト：`memory-core`）によって提供されます。

## メモリ検索

埋め込みプロバイダーが設定されている場合、`memory_search` はハイブリッド検索を使用します。ベクトル類似度（意味的な近さ）とキーワード一致（ID やコードシンボルなどの正確な用語）を組み合わせます。サポートされている任意のプロバイダーの API キーがあれば、そのまま利用できます。

<Info>
OpenClaw はデフォルトで OpenAI の埋め込みを使用します。Gemini、Voyage、Mistral、Bedrock、DeepInfra、ローカルの GGUF、Ollama、LM Studio、GitHub Copilot、または汎用の OpenAI 互換エンドポイントを使用するには、`agents.defaults.memorySearch.provider` を明示的に設定してください。
</Info>

検索の仕組み、調整オプション、プロバイダーの設定については、[メモリ検索](/ja-JP/concepts/memory-search)を参照してください。

## ナレッジ Wiki レイヤー

長期メモリを生のノートではなく、管理されたナレッジベースに近い形で機能させたい場合は、バンドルされた `memory-wiki` Plugin を使用します。これは、決定論的なページ構造、構造化された主張と証拠、矛盾と鮮度の追跡、生成されたダッシュボード、コンパイル済みダイジェスト、および Wiki ネイティブツール（`wiki_status`、`wiki_search`、`wiki_get`、`wiki_apply`、`wiki_lint`）を備えた Wiki 保管庫へ、長期的な知識をコンパイルします。

`memory-wiki` は有効なメモリ Plugin を置き換えるものではありません。有効なメモリ Plugin が引き続き想起、昇格、Dreaming を担います。`memory-wiki` はその隣に、出典情報が豊富な知識レイヤーを追加します。

## メモリの自動フラッシュ

[Compaction](/ja-JP/concepts/compaction)によって会話が要約される前に、OpenClaw は重要なコンテキストをメモリファイルへ保存するようエージェントに促す、非表示のターンを実行します。これはデフォルトで有効です。無効にするには `agents.defaults.compaction.memoryFlush.enabled: false` を設定します。

この整理用ターンをローカルモデルで実行するには、メモリフラッシュのターンにのみ適用される正確なオーバーライドを設定します（有効なセッションのモデルフォールバックチェーンは継承されません）。

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "memoryFlush": {
          "model": "ollama/qwen3:8b"
        }
      }
    }
  }
}
```

<Tip>
メモリフラッシュは、Compaction 中のコンテキスト消失を防ぎます。会話内にまだファイルへ書き込まれていない重要な事実がある場合は、要約が行われる前に自動的に保存されます。
</Tip>

## Dreaming

Dreaming は、メモリを統合する任意のバックグラウンド処理です。短期的な想起シグナルを収集し、候補をスコアリングして、条件を満たした項目だけを長期メモリ（`MEMORY.md`）へ昇格させます。

- **オプトイン**：デフォルトでは無効です。
- **スケジュール実行**：有効にすると、`memory-core` が完全な Dreaming スイープ用の定期 Cron ジョブを 1 つ自動管理します。
- **しきい値判定**：昇格するには、スコア、想起頻度、クエリ多様性の条件を通過する必要があります。
- **レビュー可能**：フェーズの要約と日記エントリが、人間によるレビュー用に `DREAMS.md` へ書き込まれます。

フェーズの動作、スコアリングシグナル、夢日記の詳細については、[Dreaming](/ja-JP/concepts/dreaming)を参照してください。

## 根拠のあるバックフィルとライブ昇格

Dreaming システムには、関連する 2 つのレビューレーンがあります。

- **ライブ Dreaming**は、`memory/.dreams/` 内の短期 Dreaming ストアを使用します。通常のディープフェーズは、このストアに基づいて `MEMORY.md` へ昇格する内容を決定します。
- **根拠のあるバックフィル**は、過去の `memory/YYYY-MM-DD.md` ノートを独立した日次ファイルとして読み取り、構造化されたレビュー出力を `DREAMS.md` に書き込みます。

根拠のあるバックフィルは、`MEMORY.md` を手動で編集せずに、古いノートを再処理し、システムが何を長期間保持すべきと判断するかを確認する際に役立ちます。

```bash
openclaw memory rem-backfill --path ./memory --stage-short-term
```

`--stage-short-term` フラグは、根拠に基づいて長期間保持すべきと判断された候補を、通常のディープフェーズがすでに使用している同じ短期 Dreaming ストアへステージングします。直接昇格させることはありません。したがって、次のようになります。

- `DREAMS.md` は人間向けのレビュー画面のままです。
- 短期ストアは機械向けのランキング画面のままです。
- `MEMORY.md` に書き込むのは引き続きディープ昇格だけです。

通常の日記エントリや通常の想起状態に触れずに再処理を取り消すには、次を実行します。

```bash
openclaw memory rem-backfill --rollback
openclaw memory rem-backfill --rollback-short-term
```

```bash
openclaw memory status          # インデックスの状態とプロバイダーを確認

openclaw memory index --force   # インデックスを再構築
```
