OpenClaw は、セッションをまたいで作業し、ステータスを確認し、サブエージェントをオーケストレーションするためのツールをエージェントに提供します。

## 利用可能なツール

| ツール             | 機能                                                                        |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | オプションのフィルター（種類、ラベル、エージェント、アーカイブ、プレビュー）を指定してセッションを一覧表示する |
| `sessions_history` | 特定のセッションのトランスクリプトを読み取る                                |
| `sessions_send`    | 別のセッションにメッセージを送信し、必要に応じて待機する                    |
| `sessions_spawn`   | バックグラウンド作業用に分離されたサブエージェントセッションを起動する      |
| `sessions_yield`   | 現在のターンを終了し、後続のサブエージェント結果を待つ                      |
| `subagents`        | このセッションで起動したサブエージェントのステータスを一覧表示する          |
| `session_status`   | `/status` 形式のカードを表示し、必要に応じてセッション単位のモデルオーバーライドを設定する |

これらのツールには、引き続き有効なツールプロファイルと許可/拒否ポリシーが適用されます。`tools.profile: "coding"` には、`sessions_spawn`、`sessions_yield`、`subagents` を含む完全なセッションオーケストレーションセットが含まれます。`tools.profile: "messaging"` には、セッション間メッセージングツール（`sessions_list`、`sessions_history`、`sessions_send`、`session_status`）が含まれますが、サブエージェントの起動は含まれません。メッセージングプロファイルを維持しながらネイティブ委任も許可するには、次を追加します。

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

グループ、プロバイダー、サンドボックス、エージェント単位のポリシーによって、プロファイル段階の後にこれらのツールが削除される場合があります。影響を受けるセッションから `/tools` を使用して、実際に有効なツール一覧を確認してください。

## セッションの一覧表示と読み取り

`sessions_list` は、キー、agentId、種類、チャンネル、モデル、トークン数、タイムスタンプを含むセッションを返します。`kinds`（配列。指定可能な値: `main`、`group`、`cron`、`hook`、`node`、`other`）、完全一致の `label`、完全一致の `agentId`、`search` テキスト、または最近のアクティビティ（`activeMinutes`）で絞り込めます。デフォルトではアクティブなセッションが返されます。アーカイブ済みセッションを確認するには、代わりに `archived: true` を渡します。各行には `pinned` と `archived` の状態が含まれます。メールボックス形式のトリアージが必要な場合は、`includeDerivedTitles`、`includeLastMessage`、または `messageLimit`（上限 20）を設定します。これにより、可視範囲に限定された派生タイトル、最終メッセージのプレビュースニペット、または各行の件数を制限した最近のメッセージを取得できます。派生タイトルとプレビューは、設定されたセッションツールの可視性ポリシーに基づいて呼び出し元がすでに参照できるセッションに対してのみ生成されるため、無関係なセッションは非表示のままです。可視性が制限されている場合、`sessions_list` は実効モードと、結果がスコープによって制限される可能性があるという警告を示す、オプションの `visibility` メタデータを返します。

`sessions_history` は、特定のセッションの会話トランスクリプトを取得します。デフォルトではツール結果は除外されます。表示するには `includeTools: true` を渡します。最新部分の件数を制限するには `limit` を使用します。ページネーションメタデータが必要な場合は `offset: 0` を渡し、その後、返された `nextOffset` の値を渡すことで、生のトランスクリプトファイルを読み取らずに、古い OpenClaw トランスクリプトのウィンドウを過去方向へページングできます。明示的なオフセットページでは、外部 CLI のフォールバックインポートは統合されません。統合された表示履歴が必要な場合は、デフォルトの最新末尾ビュー（`offset` なし）を使用してください。

返されるビューは、意図的に範囲が制限され、安全性フィルターが適用されています。

- アシスタントのテキストは再表示前に正規化されます。
  - 思考タグは削除されます
  - `<relevant-memories>` / `<relevant_memories>` のスキャフォールディングブロックは削除されます
  - `<tool_call>...</tool_call>`、`<function_call>...</function_call>`、`<tool_calls>...</tool_calls>`、`<function_calls>...</function_calls>` などのプレーンテキストのツール呼び出し XML ペイロードブロックは、正常に閉じられていない切り詰められたペイロードも含めて削除されます
  - `[Tool Call: ...]`、`[Tool Result ...]`、`[Historical context ...]` など、格下げされたツール呼び出し/結果のスキャフォールディングは削除されます
  - `<|assistant|>`、その他の ASCII `<|...|>` トークン、全角の `<｜...｜>` バリアントなど、漏洩したモデル制御トークンは削除されます
  - `<invoke ...>` / `</minimax:tool_call>` など、不正な MiniMax ツール呼び出し XML は削除されます
- 認証情報やトークンに類するテキストは、返される前に編集されます
- 長いテキストブロックは切り詰められます
- 非常に大きな履歴では、古い行が削除されるか、サイズが過大な行が `[sessions_history omitted: message too large]` に置き換えられる場合があります
- ツールは、`truncated`、`droppedMessages`、`contentTruncated`、`contentRedacted`、`bytes`、ページネーションメタデータなどの概要フラグを報告します

どちらのツールも、**セッションキー**（`"main"` など）または以前の一覧呼び出しで取得した**セッション ID**のいずれかを受け付けます。

正確な生のトランスクリプトが必要な場合は、`sessions_history` をフィルターされていないダンプとして扱うのではなく、スコープ内の SQLite トランスクリプト行を確認してください。

## セッション間メッセージの送信

`sessions_send` は別のセッションにメッセージを配信し、必要に応じて応答を待ちます。

- **送信後に待機しない:** `timeoutSeconds: 0` を設定すると、キューに追加して即座に戻ります。
- **応答を待つ:** タイムアウトを設定すると、応答をインラインで取得します。

`:thread:<id>` で終わるキーなど、スレッドスコープのチャットセッションは、有効な `sessions_send` ターゲットではありません。ツール経由のメッセージが人間向けのアクティブなスレッド内に表示されないよう、エージェント間の調整には親チャンネルのセッションキーを使用してください。

メッセージと A2A の後続応答は、受信側のプロンプト（`[Inter-session message ... isUser=false]`）およびトランスクリプトの来歴で、セッション間データとしてマークされます。受信側エージェントは、それらをエンドユーザーが直接作成した指示ではなく、ツール経由のデータとして扱う必要があります。

ターゲットが応答した後、OpenClaw はエージェントが交互にメッセージを送る**返信ループ**を実行できます（最大 `session.agentToAgent.maxPingPongTurns`、範囲 0-20、デフォルト 5）。ターゲットエージェントは `REPLY_SKIP` と応答して早期に停止できます。

## ステータスとオーケストレーションのヘルパー

`session_status` は、現在または別の可視セッション向けの軽量な `/status` 相当ツールです。使用量、時刻、モデル/ランタイムの状態、および存在する場合はリンクされたバックグラウンドタスクのコンテキストを報告します。`/status` と同様に、最新のトランスクリプト使用量エントリから不足しているトークン/キャッシュカウンターを補完でき、`model=default` はセッション単位のオーバーライドを解除します。呼び出し元の現在のセッションには `sessionKey="current"` を使用します。`openclaw-tui` など、クライアントに表示されるラベルはセッションキーではありません。

ルートメタデータが利用可能な場合、`session_status` には可視の `Route context` JSON ブロックと、対応する構造化された `details` フィールドも含まれます。これらのフィールドは、セッションキーと、現在ライブ実行を処理しているルートを区別します。

- `origin` はセッションが作成された場所です。保存されたオリジンメタデータが古い状態に存在しない場合は、配信可能なセッションキープレフィックスから推測されたプロバイダーです。
- `active` は現在のライブ実行ルートです。現在処理中のライブセッションまたは現行セッションについてのみ報告されます。
- `deliveryContext` はセッションに保存された永続的な配信ルートです。アクティブなサーフェスが異なる場合でも、OpenClaw は後続の配信にこれを再利用できます。

## セッション状態の変更

OpenClaw は、選択されたセッション状態変更についてベストエフォートのシグナルログを保持します。対象には、子セッションへの人間からの直接メッセージ、子の実行の完了または失敗、子の作成、目標の変更、Compaction が含まれます。キャンセルまたはタイムアウトした子の実行は失敗として記録され、具体的な結果（`cancelled`、`timeout`、`error`）がイベントペイロードに保持されます。ログにはメタデータと 1 行の概要のみが含まれ、メッセージ内容は含まれません。その `stateVersion` はセッションのシグナルログの先頭であり、トランザクション型の変更データキャプチャバージョンではありません。セッションストアの変更とシグナルの追加では別々のストレージを使用するため、追加に失敗した場合は、元のターンを失敗させずにログへ記録されます。

`sessions_list` は、変更が記録された行に `stateVersion` を含めます。`session_status` は、構造化された詳細で常に `stateVersion` を返します。`changesSince: <previousStateVersion>` を渡すと、そのバージョン以降に保持されているイベントを最大 200 件取得できます。この読み取りでは、親の通知カーソルは確認済みにならず、進行もしません。`historyGap: true` という結果は、要求したバージョンが保持されている履歴より前であることを意味します。そのため、応答を正確な差分として扱わず、セッション状態全体を更新してください。

別のアクターが監視対象の子に人間からの直接ターンを送信した場合、またはその目標を変更した場合、親は最後に確認したバージョンを指定して `session_status` を呼び出すよう指示するシステム通知を受け取ります。メインセッションの親は事前に起動されます。Heartbeat ルーティングはネストされたサブエージェントのキューを直接対象にできないため、その親は次のターンで通知を受け取ります。通常の子の実行完了の配信は、引き続き完了通知が担当します。

履歴は 30 日間かつ 50,000 行に制限されますが、セッション単位の先頭値はプルーニング後も単調増加を維持します。通知の配信では Gateway のインメモリシステムイベントキューを使用し、共有状態データベースの配信を 1 つの Gateway プロセスが担当することを前提としています。複数の Gateway は引き続き永続ログと `changesSince` の照合サーフェスを共有しますが、v1 ではプロセス間で通知をプッシュしません。親への通知には、エージェントで修飾された親セッションキーが必要です。`session.scope="global"` では共有の `global` キーが複数エージェント間で曖昧になるため、これらの親は永続ログと `changesSince` を利用できますが、v1 では事前通知を受け取りません。

`sessions_yield` は、待機している後続イベントを次のメッセージとして受け取れるよう、意図的に現在のターンを終了します。ポーリングループを構築せず、完了結果を次のメッセージとして受け取りたい場合は、サブエージェントの起動後に使用してください。

`subagents` は、すでに起動済みの OpenClaw サブエージェント向けの可視性ヘルパーです。アクティブまたは最近の実行を確認するための `action: "list"` をサポートします。

## サブエージェントの起動

`sessions_spawn` は、デフォルトでバックグラウンドタスク用の分離されたセッションを作成します。常にノンブロッキングで、`runId` と `childSessionKey` を返して即座に戻ります。ネイティブサブエージェントの実行では、委任されたタスクが子セッションの最初に表示される `[Subagent Task]` メッセージとして渡されます。一方、システムプロンプトにはサブエージェントのランタイムルールとルーティングコンテキストのみが含まれます。

主なオプション:

- 外部ハーネスエージェントには `runtime: "subagent"`（デフォルト）または `"acp"`。
- 子セッション用の `model` および `thinking` オーバーライド。
- 起動をチャットスレッド（Discord、Slack など）にバインドする `thread: true`。
- 子でサンドボックス化を強制する `sandbox: "require"`。
- 子が現在のリクエスターのトランスクリプトを必要とする場合、ネイティブサブエージェントには `context: "fork"`。クリーンな子には省略するか `context: "isolated"` を使用します。`context: "fork"` は `runtime: "subagent"` でのみ有効です。スレッドにバインドされたネイティブサブエージェントは、`threadBindings.defaultSpawnContext` で別途指定されていない限り、デフォルトで `context: "fork"` を使用します。

デフォルトのリーフサブエージェントにはセッションツールが与えられません。`maxSpawnDepth >= 2` の場合、深さ 1 のオーケストレーターサブエージェントには、独自の子を管理できるよう、追加で `sessions_spawn`、`subagents`、`sessions_list`、`sessions_history` が提供されます。リーフ実行には、引き続き再帰的なオーケストレーションツールは提供されません。

完了後、通知ステップによって結果がリクエスターのチャンネルに投稿されます。完了の配信では、利用可能な場合、バインドされたスレッド/トピックのルーティングが保持されます。また、完了元がチャンネルのみを識別している場合でも、OpenClaw はリクエスターセッションに保存されたルート（`lastChannel` / `lastTo`）を直接配信に再利用できます。

ACP 固有の動作については、[ACP エージェント](/ja-JP/tools/acp-agents)を参照してください。

## 可視性

セッションツールは、エージェントが参照できる範囲を制限するようスコープ設定されています。

| レベル  | スコープ                                 |
| ------- | ---------------------------------------- |
| `self`  | 現在のセッションのみ                     |
| `tree`  | 現在のセッション + 起動したサブエージェント |
| `agent` | このエージェントのすべてのセッション     |
| `all`   | すべてのセッション（設定されている場合はエージェント間も含む） |

デフォルトは `tree` です。サンドボックス化されたセッションは、設定にかかわらず `tree` に制限されます。
