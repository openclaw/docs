このページは、2026年5月の OpenClaw のパフォーマンス、パッケージサイズ、依存関係、shrinkwrap クリーンアップの根拠となる証拠をまとめたものです。公開ブログ記事の技術的な補足資料です。

ここでは2つの監査を組み合わせています。

- **リリースパフォーマンススイープ:** `v2026.5.28` から安定版 `v2026.4.23` までの GitHub Releases を対象に、`OpenClaw Performance` ワークフロー、`profile=smoke`、mock-provider レーンを使用。ほとんどのタグ行は1サンプルです。`v2026.5.27` と `v2026.5.28` の行は、最新の repeat-3 リリースブランチ成果物を使用しています。
- **4月前半の文脈:** 公開済みの `clawgrit-reports` mock-provider ベースライン（`v2026.4.1` から `v2026.5.2` まで）を使用。壊れていた4月後半のリリースを公開パフォーマンスベースラインとして扱わないためだけに使っています。
- **インストールフットプリントスイープ:** 一時パッケージに対する新規 `npm install --ignore-scripts` インストールを行い、サイズには `du -sk node_modules`、パッケージインスタンス数には `node_modules` の走査を使用。
- **npm パッケージサイズスイープ:** 公開済みリリースに対して `npm pack openclaw@<version> --dry-run --json` を実行し、圧縮 tarball サイズ、展開後サイズ、ファイル数を記録。

<Warning>
メインのパフォーマンススイープは、`v2026.5.27` と `v2026.5.28` の行を除き、タグごとに1つの smoke サンプルを使用しています。この2行は最新の repeat-3 リリースブランチ成果物を使用しています。4月前半の文脈では、`clawgrit-reports` で公開された repeat-3 の中央値を使用しています。数値はリリースゲート統計ではなく、傾向の証拠と回帰調査のシグナルとして扱ってください。
</Warning>

### Shrinkwrap 境界

`2026.5.20` は、ルート shrinkwrap も大きなネストされた OpenClaw
依存関係ツリーもない状態で出荷されました。`2026.5.22` でルート shrinkwrap が導入され、
ネストされた `openclaw/node_modules` 配下に 911.8MB がインストールされました。
`2026.5.28` は shrinkwrap を維持しており、ネストされた `openclaw/node_modules`
配下に 259.7MiB をまだインストールしますが、ローカルの新規インストール監査では
`@napi-rs/canvas` パッケージは一切インストールされなくなりました。

公開済み tarball の検査により、この境界が確認されています。

| `2026.5.21` | いいえ          | 該当なし                     | 安定版 npm リリースなし              |
| `2026.5.22` | はい            | はい                         | shrinkwrap が導入された              |
| `2026.5.23` | いいえ          | 該当なし                     | 安定版 npm リリースなし              |
| `2026.5.24` | いいえ          | 該当なし                     | 安定版 npm リリースなし              |

| `2026.5.28` | はい            | はい                         | ネストされた依存関係ツリーは大幅に縮小 |

重要な違い: **shrinkwrap 自体が問題なのではありません**。
`v2026.5.28` もルート shrinkwrap を出荷しています。問題だったのは、npm が
大きなネストされた OpenClaw 依存関係ツリーと全 12 個の

ファンアウトはローカル監査に現れなくなりました。

shrinkwrap とメンテナーレベルのパッケージチェックについての平易な説明は、
[npm shrinkwrap](/ja-JP/gateway/security/shrinkwrap) を参照してください。

## サプライチェーンの解釈

依存関係数は、インストールサイズの指標であるだけでなく、運用上のセキュリティ指標でもあります。
各パッケージは、運用者が信頼しなければならないメンテナー、tarball、推移的更新、
任意のネイティブバイナリ、インストール時の動作の集合を広げます。

クリーンアップの方向性は次のとおりです。

- 重く任意の機能はデフォルトのコアインストールの外に置く
- Plugin パッケージに各自のランタイム依存関係グラフを所有させる
- Gateway 起動時のランタイムパッケージマネージャー修復を避ける
- 全プラットフォームのネイティブパッケージ実体化を引き起こさずに、決定的なインストールを維持する
- パッケージ受け入れと計測パスではインストールスクリプトを無効のままにする
- 公開前に、ネストされた依存関係ツリーと任意のネイティブ依存関係の爆発的増加を検出する
