# Codex 監督

## 目的

Codex 監督により、OpenClaw オペレーターはネイティブ Codex セッションを検出し、安全な場合は通常の OpenClaw Chat サーフェスを通じてローカルブランチを作成できます。
Codex App Server は引き続きスレッドとモデルループを所有します。OpenClaw は、フリートカタログ、認証済みオペレーター UI、セッションバインディング、チャネル配信を提供します。

この機能は公式の `codex` Plugin に属します。独立した Supervisor Plugin や、2 つ目の Codex プロトコル実装はありません。

## 製品の境界

Codex Plugin がアクティブな場合、カタログは常に登録されます。エージェント向け監督ツールを有効にするには、次を設定します。

```text
plugins.entries.codex.config.supervision.enabled = true
```

現在の初期製品は、意図的に長期的なフリート計画よりも小さい範囲に限定されています。

- アーカイブされていない Codex スレッドのみを一覧表示します。
- ローカルおよびオプトイン済みのペアリング Node の行を、安定したホスト ID ごとにグループ化します。
- 保存済みまたはアイドル状態の Gateway ローカルスレッドから、通常のモデル固定 Chat ブランチを作成し、最初のターンで完全な Codex ハーネススレッドを開始するか、以前のブランチ用に作成された Chat を開きます。
- 保存済みまたはアイドル状態の Gateway ローカルスレッドは、他のランナーが存在しないことを明示的に確認した後にのみアーカイブします。
- アクティブなローカルソースについては、新規ブランチまたはアーカイブのコントロールを表示しませんが、既存の監督対象 Chat は引き続き開けます。
- ホストごとの最新行をメインサイドバーに表示し、セッションページには完全なカタログを保持し、ローカルおよびペアリング Node の行に対して上限付きのカーソルページネーションによるトランスクリプト読み取りを提供します。
- カタログの障害をホスト単位で分離します。

カタログは、アーカイブされていないコレクションです。その中の行は、アイドル、アクティブ、`notLoaded`、またはエラーのターン状態を持つことがあります。

エージェント向け監督は引き続きオプトインです。ガイド付きオンボーディングでは、ネイティブ Codex のインストール検出に成功し、選択した推論バックエンドがライブチェックに合格した後、ユーザーが選択したプライマリバックエンドに関係なく、監督機能のインストールと有効化を試みます。監督は、その状況依存の Plugin セットアップが成功した場合にのみ有効になります。明示的に無効化された Plugin、ポリシーによるブロック、または `supervision.enabled: false` は監督ツールに対して引き続き優先されますが、オペレーターセッションカタログは無効化しません。

## 所有権

`codex` Plugin は、Codex App Server のすべての動作を所有します。

- エンドポイントの検出と接続ライフサイクル
- プロトコルの初期化とバージョンチェック
- スレッドの一覧表示、読み取り、再開、アーカイブ、イベント処理
- 承認およびユーザー入力ブリッジ
- ネイティブスレッドと OpenClaw セッションのバインディング
- 継続後の Codex 専用モデルおよびハーネスの強制

Control UI と Gateway は、その Plugin 所有のサービスを利用します。Codex のロールアウトファイルを直接読み取ることも、別の App Server クライアントを実装することもありません。

デフォルトのローカルトポロジは次のとおりです。

```text
Codex Desktop -> プライベート stdio App Server -> ユーザーの Codex ホーム
                                                  ^
OpenClaw Codex Plugin -> 監督用 App Server 接続
  （デフォルトは管理対象ユーザーホームの stdio。明示的な appServer 設定が優先される）
  -> パッシブなソースカタログと読み取り
  -> スナップショット固定 -> 正規の appServer ソースブランチ
  -> 表示履歴の注入と、それ以降の監督対象 Chat のすべてのターン

通常の OpenClaw Codex セッション -> デフォルトでは管理対象エージェントホームの stdio
  -> 通常の完全なハーネススレッド -> OpenClaw Chat とチャネル配信
```

監督を有効にしても、通常の Codex ハーネスは変更されず、デフォルトでは引き続きエージェントスコープです。独立した監督接続のデフォルトは管理対象ユーザーホームの stdio であるため、そのカタログとスナップショット操作からネイティブの保存済みスレッドを参照できます。明示的な `appServer` 接続設定が優先されます。`homeScope` が未設定の場合、監督接続では stdio または Unix に対して `"user"`、WebSocket に対して `"agent"` に解決されます。通常のハーネスでもネイティブ Codex ホームを共有する必要がある場合にのみ、`appServer.homeScope: "user"` を明示的に設定してください。Codex サイドバーグループから引き継がれた Chat は例外です。その非公開の監督バインディングにより、ソースの読み取り、正規ブランチの作成、および以降のターンでは監督接続が維持されます。ライブ状態と所有権は引き続きプロセスローカルです。OpenClaw の監督プロセスが認識していないスレッドは、Codex Desktop がアクティブに実行している場合でも `notLoaded` です。

Codex には、独立したインストーラー管理のブートストラップ契約を持つ、実験的な正規ローカルデーモンがあります。この機能は、そのデーモンを暗黙にブートストラップしたり、所有権を主張したり、存在を前提としたりしてはなりません。

## カタログフロー

汎用 Gateway メソッド `sessions.catalog.list` は `codex`
カタログプロバイダーにディスパッチされ、このプロバイダーは常に `archived: false` と
対話型の `cli` および `vscode` ソース種別を要求します。次の結果を統合します。

1. 監視用 App Server から取得する Gateway ローカルの `thread/list` 結果。
   デフォルトでは、管理対象ユーザーホームの stdio を使用します。
2. 接続済みでオプトインしている各 Node から取得する `codex.appServer.threads.list.v1` 結果。

トランスクリプトの選択では、ローカルの場合は `itemsView: "full"` を指定した
`thread/turns/list` を使用し、選択した Node の場合はバージョン付きの
`codex.appServer.thread.turns.list.v1` コマンドを使用します。各レスポンスには、永続化された
ターンが最大 20 件と、不透明な前方／後方カーソルが含まれます。Control UI は新しい順のページを要求し、
各ページを時系列順にレンダリングして、古いページを先頭に追加します。無制限の
`thread/read` にフォールバックすることはありません。また OpenClaw は、シリアライズされたアイテムページが
Node または Gateway のトランスポートを通過する前に、20 MiB を超えていれば拒否します。

ネイティブ macOS ペアリング済み Node の実装でサポートされるのは、未設定／デフォルトまたは
明示的な `appServer.transport: "stdio"` と、未設定／デフォルトの監視スコープまたは
明示的な `appServer.homeScope: "user"` の組み合わせのみです。設定された `command`、`args`、
および正規化された `clearEnv` を子プロセスに引き渡します。`"unix"`、`"websocket"`、
または明示的な `homeScope: "agent"` の場合、カタログ機能もコマンドも公開せず、
直接呼び出してもフェイルクローズします。エージェントスコープの設定でユーザーの
Codex ホームを公開したり、明示的なエンドポイントをローカル stdio で代用したりしてはなりません。

カタログプロジェクションは、識別子、タイトル、cwd、ステータス、アクティブな待機
フラグ、タイムスタンプ、ソース、モデルプロバイダー、Codex バージョン、Git ブランチを正規化します。
トランスクリプトのプレビュー、ターン、ロールアウトパス、Codex ホームパス、
Git リモート、コミット SHA、生のエンドポイント、生の App Server エラーは返しません。トランスクリプトの
レスポンスには、明示的に要求された App Server アイテムページとその
不透明なカーソルのみが含まれます。

ホストの障害は、各ホストの結果内に留まります。Node がオフラインであるか、
ローカル App Server が利用できない場合でも、正常なホストはページから削除されません。接続性は
スレッドのステータスではなくホストのプロパティです。失敗したホスト結果には新しい
セッション行が含まれず、ネイティブスレッドに `offline` を投影することもありません。

カタログ検出は受動的です。メタデータの一覧表示や読み取りによって
`thread/resume` を呼び出したり、OpenClaw クライアントをライブスレッドリクエストにサブスクライブさせたり、
承認に応答したりしてはなりません。

検索対象はタイトルのみで、大文字と小文字を区別しません。返される各カタログページについて、
Gateway とペアリング済み Mac は、クエリを App Server に渡さず、ネイティブページを
有限のページ数だけ走査します。ネイティブ検索ではトランスクリプトの
プレビューにも一致する可能性があるためです。返されたネイティブカーソルにより、呼び出し元は走査を継続できます。

## オペレーター CLI の境界

Plugin は、Gateway を使用する 3 つのシェルコマンドを登録します。

```text
openclaw codex sessions [--search <text>] [--host <id>] [--limit <count>] [--cursor <cursor>] [--json] [gateway-options]
openclaw codex continue <thread-id> [--json] [gateway-options]
openclaw codex archive <thread-id> --confirm-no-other-runner [--json] [gateway-options]
```

`[gateway-options]` は `--url <url>`、`--token <token>`、`--timeout <ms>`、および
継承された `--expect-final` スイッチです。セッション一覧のデフォルトは 75,000 ms、
continue と archive のデフォルトは 30,000 ms です。
これらの単項 RPC では、`--expect-final` に追加の効果はありません。セッション検索の対象は
タイトルのみで、大文字と小文字を区別しません。各レスポンスは有限のネイティブページ
チェーンを走査し、`--cursor` でさらに古い結果を取得します。上限のデフォルトはホストごとに 50 で、
1 から 100 までを指定できます。カーソルを使用するには、安定した単一の `--host`
宛先が必要です。アーカイブ済みセッションを含めるオプションは、どのコマンドにもありません。
ペアリング済みホストを対象にできるのは `sessions` のみです。
`continue` と `archive` は常に `hostId: "gateway:local"` を送信し、archive には
明示的な確認フラグが必要です。

シェルの名前空間は、チャット内の `/codex` ランタイム名前空間とは異なります。特に、
`/codex sessions --host <node>` は 1 つの Node 上にある Codex CLI セッションファイルを一覧表示し、
`/codex threads` は現在の会話接続に対する App Server スレッドを一覧表示し、
`/codex resume` または `/codex bind` はその会話の
バインディングを変更します。これらのコマンドは `sessions.catalog.continue` の代替ではなく、
`/codex continue` または `/codex archive` というランタイムコマンドは存在しません。

## ローカルでの継続

保存済みまたはアイドル状態の Gateway ローカル行について、UI は
`catalogId: "codex"` とホスト ID およびスレッド ID を指定して
`sessions.catalog.continue` を呼び出します。Plugin は次を行います。

1. ソースに監視対象の既存 Chat がある場合は、それを再利用します。
2. それ以外の場合、ソースの最後の終端状態にある永続化済みターン
   （完了、中断、または失敗）までの、上限付きのユーザー履歴とアシスタント履歴を新しい
   OpenClaw Chat に投影し、保留中のハーネスブランチを記録します。
3. 具体的なモデルまたはプロバイダーの選択ではなく、保留中の Codex 専用モデルロックポリシーと、
   非公開の監視接続スコープを保存し、OpenClaw の `sessionKey` を返します。

履歴プロジェクションでは、表示可能なユーザーメッセージとアシスタントメッセージの最新部分を選択します。
上限は 200 メッセージ、UTF-8 テキストの合計 512 KiB、メッセージごとに
64 KiB です。画像およびローカル画像の入力は
`[Image attachment]` に置き換え、画像ペイロードやパスは一切コピーせず、推論、
ツール呼び出し、ツール結果は除外します。

UI は、そのセッションキーを使用して通常の Chat に遷移します。正規のハーネス
スレッドはまだ存在しません。最初の通常 Chat ターンで、ハーネスは実際の
Codex の承認、追加情報要求、イベント、配信の各ハンドラーをインストールし、次を行います。

1. 監視接続を使用して、モデルまたはプロバイダーを上書きせずにネイティブの `thread/fork` を
   呼び出し、永続化されたソーススナップショットを固定します。Codex の現在の
   `ConfigManager` 状態がモデルとプロバイダーを選択し、fork レスポンスが
   実際の組み合わせを報告します。モデルがソースに最後に記録されたモデルと異なる場合、
   Codex は通常のモデル差異警告を発します。
2. 同じ接続上で、`threadSource: "appServer"`、OpenClaw の cwd、ポリシー、設定、環境、
   OpenClaw ハーネスの完全なツールサーフェス、およびこの初回開始用として
   fork が返したモデルとプロバイダーを正確に指定し、正規の完全な Codex ハーネススレッドを開始します。
3. 上限付きの表示可能なユーザー履歴とアシスタント履歴をその接続経由で注入し、
   監視スコープを破棄せずに正規のバインディングをコミットし、ターンを実行して、
   一時的な fork をアーカイブします。

最初のターンの前は、Chat は表示可能な履歴ミラーを備えたロック済みの保留ブランチです。
その後は、すべてのモデルターンが監視接続上の正規の Codex
ハーネススレッドを通じて実行されます。このブランチは完全なネイティブ
ロールアウトの複製ではありません。ソースの推論、ツール呼び出し、ツール結果は意図的に
除外されます。スナップショットの固定または正規スレッドの作成に失敗した場合、
保留ブランチは再試行可能な状態のままです。バインディングの競合、監視の無効化、または利用不能
もしくは不一致の監視接続が発生した場合、通常のエージェントホームハーネスに
フォールバックすることなく、ターンの実行前にフェイルクローズします。

これにより保証されるのは、ソースの履歴上のモデルの維持ではなく、Codex による選択です。
fork が返した組み合わせが正規スレッドの
開始に使用され、Codex はそのスレッドのネイティブなモデルとプロバイダーを永続化します。その後の再開では、
OpenClaw のモデルとプロバイダーの上書きを省略するため、Codex が永続化された組み合わせを復元します。
別のネイティブ Codex コントロールが正規スレッドを変更した場合、OpenClaw は
そのネイティブに永続化された選択を受け入れます。外側の OpenClaw モデルおよびフォールバックチェーンが
その代わりに使用されることはありません。

監視対象でモデルがロックされた Chat では、モデル変更、セッション削除、セッションのリセット／新規作成操作はフェイルクローズします。
バインディングを置換または消去するため、`/codex model <model>`、`/codex
bind`、`/codex resume`（Node の `--bind here` を含む）の変更操作、および `/codex detach` または
`/codex unbind` もフェイルクローズします。`/codex model` の照会と `/codex fast`、`/codex permissions`、`/codex
threads` は引き続き使用できます。`codex_threads` エージェントツールは、新しい
フォークをアタッチしたり、バインド済みのネイティブスレッドをアーカイブしたりできません。一覧取得とメタデータのみの読み取りは引き続き
使用できます。トランスクリプトフィールドには `supervision.allowRawTranscripts` が必要であり、
名前変更、アーカイブ解除、デタッチされたフォーク、および無関係なスレッドのアーカイブには
`supervision.allowWriteControls` が必要です。どちらのオプションでも、ロックされたバインディングを置換することはできません。
OpenClaw エントリを削除またはリセットすると、ネイティブ
バインディングが破棄され、Codex セッションのように見えるセッションの背後に汎用スレッドが作成されるか、その作成が許可されてしまいます。
そのため、保持メンテナンスでは、通常の経過時間、件数、ディスク容量の上限を
超えている場合でも、モデルがロックされたエントリを保持します。所有元の Plugin を無効化またはアンインストールしても、
ロックと Plugin 所有権マーカーは保持されます。同じ Plugin が再び有効化されるまで、Chat は
利用不能なままフェイルクローズします。クリーンアップによって
通常のモデルセッションに変換されることはありません。

この操作によって、ソースが再開または変更されることはありません。一時フォークは
スナップショットを固定しますが、永続的な継続スレッドではありません。最初のターンで別個の
正規ハーネススレッドを開始することで、プロセスローカルの状態が Desktop 所有のターンを
検出できなかっただけで、OpenClaw が競合するソース書き込み元になることを防ぎます。
表示履歴のミラーと固定されたスナップショットには、アクティブなソースでまだ完了していない作業が
含まれない場合があります。元の CLI または VS Code
ソースは、ネイティブカタログと OpenClaw カタログの両方の対象として引き続き適格です。正規
ブランチは監視ストア内でネイティブ Codex スレッドのままですが、ネイティブクライアントは
その `appServer` ソース種別を除外する場合があるため、Codex Desktop での表示可否は
契約ではありません。

## アーカイブ動作

保存済みまたはアイドル状態の Gateway ローカル行に対して、`catalogId: "codex"` を指定した
`sessions.catalog.archive` では、
明示的な `confirmNoOtherRunner: true` が必要です。現在のプロセスローカル
状態を新たに読み取り、`idle` または `notLoaded` の場合にのみ処理を続行し、ネイティブの `thread/archive` を呼び出し、
Codex が操作を受け入れた後にのみ成功を返します。その後、その行は
未アーカイブのカタログから除外されます。

新たな読み取りでアクティブ状態またはエラー状態が返された場合、アーカイブは拒否されます。ソースからの
初期化中または保留中の監視対象ブランチでも同様に拒否されます。ソースをアーカイブする前に、最初の Chat ターンで
正規ブランチを実体化する必要があります。対象と完全に一致する既知のアクティブな
OpenClaw バインディング所有者、またはアーカイブされていない生成済み子孫が存在する場合も、アーカイブは拒否されます。
OpenClaw は Codex の実験的な
`thread/list ancestorThreadId` 関係をページネーションし、リクエストまたはレスポンスの
エラー、カーソルまたはスレッドの循環、安全上限の超過時にはフェイルクローズします。ネイティブアーカイブは
ロード済みの親および子孫の作業を停止できるため、アーカイブは割り込みの
代替手段ではありません。読み取り、子孫の列挙、アーカイブの各呼び出しはアトミックではありません。
独立したクライアントが、ローカルではアイドル状態または
`notLoaded` に見える行を引き続き所有しているか、その行で作業を開始する可能性があります。他のランナーが存在しないことの確認は、
Codex に条件付きアーカイブまたはプロセス間リースが実装されるまで、未知のクライアントと
この競合を対象とします。ペアリングされた Node のアーカイブは禁止されています。

Codex カタログにはアーカイブ済みビューはありません。別の所有者承認済み Codex サーフェスで
`thread/unarchive` によって復元されたスレッドは、再び未アーカイブ
カタログの対象になります。

## アクティブスレッドの安全性

Codex は、1 つの App Server のクライアント間でスレッドに対する変更を直列化しますが、
排他的なプロセス間ランナーや承認所有者リースは公開しません。
独立した stdio App Server は同じ rollout に追記できますが、それぞれが確認できるのは
自身のメモリ内ステータスのみです。承認リクエストは 1 つのサーバーのすべてのサブスクライバーにも
到達でき、最初の有効な応答によってリクエストが完了します。

したがって、次のようになります。

- パッシブなカタログクライアントは、承認をサブスクライブせず、自動拒否もしません
- 現在アクティブと報告されている行には、新しいブランチも Archive も表示されません
- マッピングされていないソースは、可視履歴のブランチになり、その正規のハーネス
  スレッドがソースを再開することはありません
- `notLoaded` はアクティビティ不明として表示され、他にランナーが存在しないことを
  十分な情報に基づいて確認した後にのみアーカイブできます
- ローカルでのアーカイブには、その確認に加えて最新の `idle` または `notLoaded`
  の読み取りが必要であり、読み取りからアーカイブまでのプロトコル上の競合を認識する必要があります

中断と複数クライアント間の引き継ぎは、今後の製品判断事項です。アクティブな行を
表示することによって、それらが暗黙に保証されるわけではありません。

## ペアリングされた Node の境界

Node invoke は現在、リクエスト/レスポンスのみに対応しています。サイズが制限された
カタログメタデータとトランスクリプトのターンページは安全に返せますが、Codex
ハーネスの実行に必要な長時間存続するイベントストリーム、承認リクエスト、
ツール呼び出し、キャンセル、アシスタントの差分は伝送できません。

したがって、Node のコントラクトは一覧とトランスクリプトのターンページをサポートします。リモートの
行は引き続き読み取れますが、アイドル状態にかかわらず **Continue** と **Archive** は使用できません。実際にリモートで続行するには、
ローカルハーネスと同じ承認およびバインディングの不変条件を維持する Node 側のランナーとストリーミングブリッジが必要です。

## 権限

各コンピューターはローカルでオプトインします。Gateway を有効にしても、別の
Node に Codex メタデータを読み取る権限は付与されません。Node の機能は、通常のペアリングと
コマンドポリシーの承認を通過する必要があります。

フリートの一覧表示とトランスクリプトの表示では、ペアリング済み Node を呼び出すため、
`operator.write` Gateway スコープを使用します。ローカルでの継続とアーカイブは、
認証済みオペレーターの操作であり、引き続きホストとステータスのチェック対象です。

自律エージェントおよびスタンドアロン MCP のアクセスは別です。提供されている
`codex_endpoint_probe`、`codex_sessions_list`、`codex_session_read`、
`codex_session_send`、`codex_session_interrupt` のツールコントラクトは、引き続き
`codex` Plugin が所有します。監督が有効な場合、生の `codex_threads` トランスクリプトの
読み取りと、トランスクリプトから派生した一覧フィールドにも
`supervision.allowRawTranscripts` が必要です。また、`codex_threads` のフォーク、名前変更、アーカイブ、
アーカイブ解除には、すべて `supervision.allowWriteControls` が必要です。どちらのポリシーも
デフォルトでは無効です。

## 互換性

`openclaw doctor --fix` は、提供済みの `plugins.entries.codex-supervisor`
設定を移行します。これには、エンドポイント、トランスクリプトと書き込みのポリシー、および Plugin の
許可／拒否参照が含まれ、移行先は
`plugins.entries.codex.config.supervision` です。競合時には、明示的な正規の移行先
値が優先されます。移行後、ランタイムコードは正規の `codex` Plugin
形式のみを使用します。

公式 Plugin は、Supervisor 互換ツールを正確に 5 つ保持します。
`codex_endpoint_probe`、`codex_sessions_list`、`codex_session_read`、
`codex_session_send`、`codex_session_interrupt` です。セッション一覧はデフォルトで
読み込み済みのみを対象とし、`loaded_only` パラメーターはありません。`include_stored: true` を指定すると、
アーカイブされていない状態データベースの行が追加され、エンドポイントごとに `max_stored_sessions`
（デフォルト 200、許容範囲 1～1,000）で上限が設定されます。読み込み済みの行には、この
設定による上限は適用されません。トランスクリプト由来のフィールドと読み取りには、引き続き
`allowRawTranscripts` による制限が適用されます。送信と割り込みには、引き続き `allowWriteControls` による制限が適用されます。

互換送信は、アイドル状態のスレッドを開始または再開しません。`mode: "start"` は
常に拒否されます。`"auto"` と `"steer"` は、読み取り可能なアクティブターンのみを誘導します。
同様に、割り込みにも読み取り可能なアクティブターンが必要です。アイドル状態からの継続は、
ネイティブ Codex カタログにルーティングされ、完全なハーネスが承認、ツール、バインディングを管理します。
スタンドアロンのレガシー MCP アダプターは、公式
Plugin から同じツールを解決し、保持されているレガシーポリシー環境
変数を尊重する唯一の経路です。

7 月のカタログ UI、Gateway メソッド、Node 機能、CLI 登録は、
旧 Plugin ID では提供されていませんでした。これらは第 2 のランタイムファサードを設けず、
直接 `codex` の所有に移ります。

## 受け入れテスト

- 監督を有効にすると、アーカイブされていないローカルセッションが一覧表示される。
- アーカイブ済みセッションは、カタログのレスポンスにも UI にも決して表示されない。
- 別のホストで障害が発生しても、正常なホストは引き続き表示される。利用できないホストは、
  オフラインセッションステータスを捏造するのではなく、新しい行を返さない。
- 保存済みまたはアイドル状態のローカル行によって、Codex 専用の
  モデル／ランタイムロックを持つ Chat ミラーが作成される。最初のターンで一時スナップショットが固定され、
  正規の完全なハーネススレッドが開始される。Continue を再度実行すると、既存の Chat が開く。
- 最初のターンでは、スナップショットのフォーク時にモデル／プロバイダーのオーバーライドを省略し、
  Codex が現在のモデルとソースに最後に記録されたモデルが異なると警告した場合でも、
  正規の開始処理を Codex が返した正確なペアに固定する。
- 保留中およびコミット済みの監督対象バインディングは、ソースアクセス、
  正規ブランチの作成、および以後のすべてのターンに監督接続を使用する。通常の
  Codex セッションは、引き続きエージェントスコープとなる。
- 以後の再開では、OpenClaw のモデル／プロバイダーのオーバーライドを省略し、Codex の
  正規に永続化された選択を維持し、そのスレッドに対する別個のネイティブ変更を受け入れ、
  外側の OpenClaw モデルやフォールバックチェーンに決して置き換えない。
- 監督を無効にした場合、またはバインディング／接続のライフサイクルが失われた場合は、Chat を
  通常のエージェントホームのハーネスに移すのではなく、フェイルクローズする。
- 監督対象でモデルロックされた Chat は、ネイティブ
  バインディングを保護している間は削除できない。
- Chat は、ユーザーとアシスタントのメッセージを最大 200 件、合計 512 KiB、
  メッセージごとに 64 KiB までミラーする。画像はプレースホルダーになる。ソースの推論、ツール呼び出し、
  ツール結果、画像ペイロード、ローカルパスは複製されない。
- ブランチフローはソーススレッドを決して再開しない。
- 元のソースは、両方のカタログで引き続き対象となる。正規のネイティブ
  ブランチは `appServer` ソース種別を使用し、Codex Desktop に表示される保証はない。
- アクティブなローカルソースは、ブランチを作成することもアーカイブすることもできない。ただし、既存の
  監督対象 Chat は引き続き開くことができる。
- アクティビティが不明な行は確認なしでブランチを作成できるが、アーカイブには
  他のランナーが存在しないことの明示的な確認が必要となる。
- 初期化中または保留中の監督対象ブランチを持つソースは、
  最初の Chat ターンで正規ブランチが実体化するまでアーカイブできない。
- 正確な対象の既知のアクティブなバインディング所有者、またはアーカイブされていない生成済みの
  子孫が存在する場合、アーカイブはブロックされる。子孫の列挙に失敗した場合はフェイルクローズし、
  明示的な確認が、引き続き不明なクライアントと
  ステータス確認からアーカイブまでの競合に対する責任を負う。
- 確認済みの保存済みまたはアイドル状態のローカルアーカイブは、ネイティブ処理の成功後に行を削除する。
- ペアリング済み Node の行は、Continue または Archive なしでも引き続き表示される。
- 受動的な一覧表示では、スレッド承認を購読したり応答したりしない。
- レガシー Supervisor 設定は、正規の Codex 設定形式に移行される。
- レガシー一覧はデフォルトで読み込み済みのみを対象とし、保存済み項目の列挙はエンドポイントごとの
  上限に従い、互換送信はアイドル状態のスレッドを開始または再開しない。
