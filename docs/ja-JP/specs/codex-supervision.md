# Codex の監督

## 目的

Codex の監督により、OpenClaw のオペレーターはネイティブ Codex セッションを検出し、安全な場合は通常の OpenClaw Chat 画面からローカルブランチを作成できます。
Codex App Server は引き続きスレッドとモデルループを所有します。OpenClaw はフリートカタログ、認証済みオペレーター UI、セッションバインディング、チャネル配信を提供します。

この機能は公式の `codex` Plugin に属します。独立した Supervisor Plugin や、2 つ目の Codex プロトコル実装はありません。

## 製品境界

ネイティブセッション検出が次の設定で明示的に無効化されていない限り、Codex Plugin が有効になるとカタログが登録されます。

```text
plugins.entries.codex.config.sessionCatalog.enabled = false
```

エージェント向け監督ツールを有効にするには、次を設定します。

```text
plugins.entries.codex.config.supervision.enabled = true
```

現在の初期製品は、意図的に長期的なフリート計画よりも小さい範囲に限定されています。

- アーカイブされていない Codex スレッドのみを一覧表示します。
- ローカルおよびオプトイン済みのペアリング Node の行を、安定したホスト ID ごとにグループ化します。
- 保存済みまたはアイドル状態の Gateway ローカルスレッドから、通常のモデルロック済み Chat ブランチを作成し、最初のターンで完全な Codex ハーネススレッドを開始するか、以前のブランチ用に作成された Chat を開きます。
- 保存済みまたはアイドル状態の Gateway ローカルスレッドは、他にランナーが存在しないことを明示的に確認した後にのみアーカイブします。
- アクティブなローカルソースについては、新規ブランチやアーカイブのコントロールを表示せず、既存の監督対象 Chat は引き続き開けるようにします。
- メインサイドバーにはホストごとの最新行を表示し、セッションページには完全なカタログを保持し、ローカルおよびペアリング Node の行に対して上限付きのカーソルページネーションによるトランスクリプト読み取りを提供します。
- カタログ障害をホスト単位で分離します。

カタログは、アーカイブされていない項目の集合です。その中の行は、ターン状態がアイドル、アクティブ、`notLoaded`、またはエラーである場合があります。

エージェント向け監督は引き続きオプトインです。ガイド付きオンボーディングでは、ネイティブ Codex のインストール検出に成功し、選択された推論バックエンドがライブチェックに合格すると、ユーザーが選択したプライマリバックエンドに関係なく、この機能のインストールと有効化を試みます。監督が有効になるのは、この便宜的な Plugin セットアップに成功した場合のみです。明示的に無効化された Plugin、ポリシーブロック、または `supervision.enabled: false` は、監督ツールに対して引き続き優先されますが、オペレーターのセッションカタログは無効化しません。`sessionCatalog.enabled: false` は、オペレーターによる検出とペアリング Node のカタログコマンドを無効化しますが、Codex プロバイダーとハーネスは引き続き有効です。

## 所有権

`codex` Plugin は、Codex App Server のすべての動作を所有します。

- エンドポイントの検出と接続ライフサイクル
- プロトコルの初期化とバージョンチェック
- スレッドの一覧表示、読み取り、再開、アーカイブ、イベント処理
- 承認およびユーザー入力のブリッジ
- ネイティブスレッドと OpenClaw セッションのバインディング
- 継続後の Codex 専用モデルおよびハーネスの強制

Control UI と Gateway は、その Plugin が所有するサービスを利用します。Codex のロールアウトファイルを直接読み取ることも、別の App Server クライアントを実装することもありません。

デフォルトのローカルトポロジは次のとおりです。

```text
Codex Desktop -> 非公開の stdio App Server -> ユーザーの Codex ホーム
                                             ^
OpenClaw Codex Plugin -> 監督用 App Server 接続
  （デフォルトは管理対象のユーザーホーム stdio。明示的な appServer 設定が優先される）
  -> パッシブなソースカタログと読み取り
  -> スナップショットの固定 -> 正規の appServer ソースブランチ
  -> 表示可能な履歴の注入と、それ以降の監督対象 Chat のすべてのターン

通常の OpenClaw Codex セッション -> デフォルトでは管理対象のエージェントホーム stdio
  -> 通常の完全なハーネススレッド -> OpenClaw Chat とチャネル配信
```

監督を有効化しても、通常の Codex ハーネスは変更されません。デフォルトでは引き続きエージェントスコープです。独立した監督用接続のデフォルトは管理対象のユーザーホーム stdio であるため、そのカタログとスナップショット操作からネイティブの保存済みスレッドを参照できます。明示的な `appServer` 接続設定が優先されます。`homeScope` が未設定の場合、監督用接続では stdio または Unix に対して `"user"`、WebSocket に対して `"agent"` に解決されます。通常のハーネスでもネイティブ Codex ホームを共有する必要がある場合にのみ、`appServer.homeScope: "user"` を明示的に設定してください。Codex サイドバーグループから採用された Chat は例外です。その非公開の監督バインディングにより、ソースの読み取り、正規ブランチの作成、およびそれ以降のターンでは監督用接続が維持されます。ライブ状態と所有権は引き続きプロセスローカルです。OpenClaw の監督プロセスが認識していないスレッドは、Codex Desktop でアクティブに実行されている場合でも `notLoaded` です。

Codex には、インストーラー管理の独立したブートストラップ契約を持つ、実験的な正規ローカルデーモンがあります。この機能は、そのデーモンを暗黙的にブートストラップ、取得、または前提としてはなりません。

## カタログフロー

汎用 Gateway メソッド `sessions.catalog.list` は `codex` カタログプロバイダーにディスパッチします。このプロバイダーは常に `archived: false` と、対話型の `cli` および `vscode` ソース種別を要求します。次の結果を結合します。

1. 監督用 App Server から得られる Gateway ローカルの `thread/list` 結果。デフォルトは管理対象のユーザーホーム stdio です。
2. 接続済みでオプトインされた各 Node から得られる `codex.appServer.threads.list.v1` 結果。

トランスクリプトの選択には、ローカルでは `itemsView: "full"` を指定した `thread/turns/list`、選択した Node ではバージョン付きの `codex.appServer.thread.turns.list.v1` コマンドを使用します。各レスポンスには、永続化されたターンが最大 20 件と、不透明な前方／後方カーソルが含まれます。Control UI は新しい順のページを要求し、各ページを時系列順にレンダリングし、古いページを先頭に追加します。上限のない `thread/read` にフォールバックすることはありません。OpenClaw は、シリアライズされた項目ページが 20 MiB を超える場合、Node または Gateway のトランスポートを通過する前に拒否します。

ネイティブ macOS ペアリング Node 実装は、未設定／デフォルトまたは明示的な `appServer.transport: "stdio"` と、未設定／デフォルトの監督スコープまたは明示的な `appServer.homeScope: "user"` のみをサポートします。設定された `command`、`args`、および正規化された `clearEnv` を子プロセスに引き渡します。`"unix"`、`"websocket"`、または明示的な `homeScope: "agent"` の場合、カタログ機能もコマンドも公開せず、直接呼び出した場合もフェイルクローズします。エージェントスコープの設定でユーザーの Codex ホームを公開したり、明示的なエンドポイントをローカル stdio で置き換えたりしてはなりません。

カタログプロジェクションは、識別子、タイトル、cwd、状態、アクティブな待機フラグ、タイムスタンプ、ソース、モデルプロバイダー、Codex バージョン、Git ブランチを正規化します。トランスクリプトのプレビュー、ターン、ロールアウトパス、Codex ホームパス、Git リモート、コミット SHA、生のエンドポイント、生の App Server エラーは返しません。トランスクリプトのレスポンスには、明示的に要求された App Server の項目ページと、その不透明なカーソルのみが含まれます。

ホストの障害は、各ホストの結果内に限定されます。オフラインの Node や利用できないローカル App Server によって、正常なホストがページから消去されることはありません。接続性はホストのプロパティであり、スレッド状態ではありません。失敗したホストの結果には新しいセッション行が含まれず、ネイティブスレッドに `offline` を投影することもありません。

カタログ検出はパッシブです。メタデータの一覧表示または読み取りで、`thread/resume` を呼び出したり、OpenClaw クライアントをライブスレッド要求にサブスクライブしたり、承認に応答したりしてはなりません。

検索はタイトルのみを対象とし、大文字と小文字を区別しません。返されるカタログページごとに、Gateway とペアリング済み Mac は、クエリを App Server に渡さず、上限付きの数のネイティブページを走査します。ネイティブ検索ではトランスクリプトのプレビューにも一致する可能性があるためです。返されたネイティブカーソルを使用すると、呼び出し元は走査を続行できます。

## オペレーター CLI の境界

Plugin は、Gateway をバックエンドとする 3 つのシェルコマンドを登録します。

```text
openclaw codex sessions [--search <text>] [--host <id>] [--limit <count>] [--cursor <cursor>] [--json] [gateway-options]
openclaw codex continue <thread-id> [--json] [gateway-options]
openclaw codex archive <thread-id> --confirm-no-other-runner [--json] [gateway-options]
```

`[gateway-options]` は `--url <url>`、`--token <token>`、`--timeout <ms>`、および継承された `--expect-final` スイッチです。セッション一覧のデフォルトは 75,000 ms、継続とアーカイブのデフォルトは 30,000 ms です。これらの単項 RPC では、`--expect-final` に追加の効果はありません。セッション検索はタイトルのみを対象とし、大文字と小文字を区別しません。各レスポンスは上限付きのネイティブページチェーンを走査し、`--cursor` で古い結果へ続行します。上限のデフォルトはホストごとに 50 で、1 から 100 まで指定できます。カーソルを使用するには、安定した単一の `--host` 宛先が必要です。アーカイブ済み項目またはアーカイブ済み項目を含めるオプションを受け付けるコマンドはありません。ペアリング済みホストを対象にできるのは `sessions` のみです。`continue` と `archive` は常に `hostId: "gateway:local"` を送信し、アーカイブには明示的な確認フラグが必要です。

シェル名前空間は、Chat 内の `/codex` ランタイム名前空間とは異なります。具体的には、`/codex sessions --host <node>` は 1 つの Node 上の Codex CLI セッションファイルを一覧表示し、`/codex threads` は現在の会話接続に対する App Server スレッドを一覧表示し、`/codex resume` または `/codex bind` はその会話のバインディングを変更します。これらのコマンドは `sessions.catalog.continue` を置き換えるものではなく、`/codex continue` または `/codex archive` というランタイムコマンドもありません。

## ローカルでの継続

保存済みまたはアイドル状態の Gateway ローカル行について、UI はホスト ID とスレッド ID に加えて `catalogId: "codex"` を指定し、`sessions.catalog.continue` を呼び出します。Plugin は次の処理を行います。

1. ソースに監督対象 Chat がすでに存在する場合は、それを再利用します。
2. それ以外の場合は、ソースの最後の永続化済み終端ターン（完了、中断、または失敗）までの上限付きユーザー／アシスタント履歴を新しい OpenClaw Chat に投影し、保留中のハーネスブランチを記録します。
3. 具体的なモデルやプロバイダーの選択ではなく、保留中の Codex 専用モデルロックポリシーと非公開の監督接続スコープを保存し、OpenClaw の `sessionKey` を返します。

履歴プロジェクションでは、表示可能なユーザーメッセージとアシスタントメッセージの最新末尾を選択します。上限は 200 メッセージ、UTF-8 テキストの合計 512 KiB、メッセージごとに 64 KiB です。画像およびローカル画像の入力を `[Image attachment]` に置き換え、画像のペイロードやパスをコピーすることはなく、推論、ツール呼び出し、ツール結果を省略します。

UI は、そのセッションキーを使用して通常の Chat に移動します。この時点では正規のハーネススレッドはまだ存在しません。通常の Chat の最初のターンで、ハーネスは実際の Codex の承認、引き出し、イベント、配信ハンドラーをインストールしてから、次の処理を行います。

1. 監督用接続を使用して、モデルやプロバイダーを上書きせずにネイティブの `thread/fork` を呼び出し、永続化されたソーススナップショットを固定します。Codex の現在の `ConfigManager` 状態によってモデルとプロバイダーが選択され、フォークのレスポンスで実際の組み合わせが報告されます。モデルがソースに最後に記録されたモデルと異なる場合、Codex は通常のモデル相違警告を出します。
2. 同じ接続上で、`threadSource: "appServer"`、OpenClaw の cwd、ポリシー、設定、環境、OpenClaw ハーネスの完全なツール群、およびこの初回起動用としてフォークから返されたモデルとプロバイダーを正確に使用し、正規の完全な Codex ハーネススレッドを開始します。
3. その接続を通じて上限付きの表示可能なユーザー／アシスタント履歴を注入し、監督スコープを削除せずに正規バインディングをコミットし、ターンを実行して、一時フォークをアーカイブします。

最初のターンより前では、Chat は履歴ミラーが表示されるロック済みの保留ブランチです。その後は、すべてのモデルターンが監督接続上の正規 Codex ハーネススレッドを経由して実行されます。このブランチは完全なネイティブロールアウトの複製ではありません。ソースの推論、ツール呼び出し、ツール結果は意図的に省略されます。スナップショットの固定または正規スレッドの作成に失敗した場合、保留ブランチは再試行可能なままです。バインディングの競合、監督の無効化、または監督接続が利用不能か不一致である場合、通常の agent-home ハーネスへフォールバックせず、ターンの実行前にフェイルクローズします。

これは、ソースの履歴上のモデルを保持するのではなく、Codex が所有する選択を保証します。フォークから返されたペアが正規スレッドの開始に使用され、Codex はそのスレッドのネイティブモデルとプロバイダーを永続化します。それ以降の再開では OpenClaw のモデルおよびプロバイダーのオーバーライドが省略されるため、Codex は永続化されたペアを復元します。別のネイティブ Codex コントロールが正規スレッドを変更した場合、OpenClaw はそのネイティブに永続化された選択を受け入れます。外側の OpenClaw モデルおよびフォールバックチェーンがその代わりに使用されることはありません。

監督対象のモデルロック済み Chat では、モデル変更、セッション削除、セッションのリセット／新規作成操作はフェイルクローズします。`/codex model <model>`、`/codex
bind`、`/codex resume`（Node `--bind here` を含む）、および `/codex detach` または
`/codex unbind` の変更も、バインディングを置換または消去するためフェイルクローズします。
`/codex model` クエリと `/codex fast`、`/codex permissions`、`/codex
threads` は引き続き利用できます。`codex_threads` エージェントツールは、新しいフォークを接続したり、バインドされたネイティブスレッドをアーカイブしたりできません。リストおよびメタデータのみの読み取りは引き続き利用できます。トランスクリプトフィールドには `supervision.allowRawTranscripts` が必要であり、名前変更、アーカイブ解除、切り離されたフォーク、および無関係なスレッドのアーカイブには
`supervision.allowWriteControls` が必要です。どちらのオプションもロック済みバインディングを置換できません。
OpenClaw エントリを削除またはリセットすると、本来はネイティブバインディングが破棄され、Codex セッションのように見える汎用スレッドが作成または許可されます。そのため、保持メンテナンスでは、通常の経過時間、件数、またはディスク予算の上限を超えていても、モデルロック済みエントリを保持します。所有 Plugin を無効化またはアンインストールしても、ロックと Plugin 所有権マーカーは保持されます。同じ Plugin が再び有効化されるまで Chat は利用不能のままフェイルクローズし、クリーンアップによって通常のモデルセッションへ変換されることはありません。

この操作によってソースが再開または変更されることはありません。一時フォークはスナップショットを固定しますが、永続的な継続スレッドではありません。最初のターンで別個の正規ハーネススレッドを開始することで、プロセスローカルの状態が Desktop 所有のターンを検出できなかったという理由だけで、OpenClaw が競合するソース書き込み元になることを防ぎます。表示履歴ミラーと固定スナップショットには、アクティブなソースでまだ完了していない作業が含まれない場合があります。元の CLI または VS Code ソースは、ネイティブカタログと OpenClaw カタログの両方で引き続き対象となります。正規ブランチは監督ストア内のネイティブ Codex スレッドのままですが、ネイティブクライアントはその `appServer` ソース種別を除外する場合があるため、Codex Desktop での可視性は契約ではありません。

## アーカイブの動作

保存済みまたはアイドル状態の Gateway ローカル行では、`catalogId: "codex"` を伴う
`sessions.catalog.archive` に明示的な `confirmNoOtherRunner: true` が必要です。現在のプロセスローカル状態を改めて読み取り、`idle` または `notLoaded` の場合にのみ処理を続行し、ネイティブの `thread/archive` を呼び出し、Codex が操作を受け入れた後にのみ成功を返します。その後、その行は非アーカイブ済みカタログから除外されます。

改めて読み取った状態がアクティブまたはエラーの場合、アーカイブは拒否されます。ソースからの初期化中または保留中の監督対象ブランチも同様です。ソースをアーカイブするには、Chat の最初のターンで正規ブランチを実体化しておく必要があります。対象と完全に一致する既知のアクティブな OpenClaw バインディング所有者、またはアーカイブされていない生成済み子孫が存在する場合も、アーカイブは拒否されます。OpenClaw は Codex の実験的な
`thread/list ancestorThreadId` 関係をページネーションし、リクエストまたはレスポンスのエラー、カーソルまたはスレッドの循環、および安全上限の枯渇時にはフェイルクローズします。ネイティブアーカイブは読み込まれた親および子孫の作業を停止できるため、アーカイブは割り込みの近道ではありません。読み取り、子孫の列挙、およびアーカイブ呼び出しはアトミックではありません。別の独立したクライアントが、ローカルではアイドルまたは
`notLoaded` に見える行の作業を引き続き所有または開始する可能性があります。他のランナーが存在しないことの確認は、Codex に条件付きアーカイブまたはプロセス間リースが実装されるまで、未知のクライアントとその競合を対象とします。ペアリング済み Node のアーカイブは禁止されています。

Codex カタログにはアーカイブ済みビューがありません。別の所有者認可済み Codex サーフェスで
`thread/unarchive` によって復元されたスレッドは、再び非アーカイブ済みカタログの対象になります。

## アクティブスレッドの安全性

Codex は、1 つの App Server のクライアント間でスレッドに対する変更を直列化しますが、排他的なプロセス間ランナーまたは承認所有者リースは公開しません。独立した stdio App Server は同じロールアウトに追記でき、それぞれが自身のメモリ内状態のみを参照します。承認リクエストも 1 つのサーバーのすべてのサブスクライバーに届く可能性があり、最初の有効なレスポンスによってリクエストが完了します。

したがって、次のようになります。

- 受動的なカタログクライアントは承認をサブスクライブせず、自動拒否もしません
- 現在アクティブと報告されている行では、新しいブランチもアーカイブも公開されません
- マッピングされていないソースは表示履歴ブランチとなり、その正規ハーネス
  スレッドがソースを再開することはありません
- `notLoaded` はアクティビティ不明として表示され、他のランナーが存在しないことを理解したうえで確認した後にのみ
  アーカイブできます
- ローカルアーカイブには、その確認に加えて `idle` または `notLoaded` の
  改めての読み取りが必要であり、読み取りとアーカイブ間のプロトコル競合も認識する必要があります

割り込みとマルチクライアント間の引き継ぎは、将来の製品判断事項です。アクティブな行を表示することによって、それらが暗黙に示されるわけではありません。

## ペアリング済み Node の境界

Node 呼び出しは現在、リクエスト／レスポンスのみに対応しています。上限付きのカタログメタデータとトランスクリプトのターンページは安全に返せますが、Codex ハーネスの実行に必要な長時間存続するイベントストリーム、承認リクエスト、ツール呼び出し、キャンセル、およびアシスタント差分は伝送できません。

そのため、Node 契約はリストおよびトランスクリプトのターンページをサポートします。リモート行は引き続き読み取れますが、アイドル状態かどうかにかかわらず、**続行**と**アーカイブ**は利用できません。実際にリモートで継続するには、ローカルハーネスと同じ承認およびバインディングの不変条件を維持する Node 側ランナーとストリーミングブリッジが必要です。

## 権限

各コンピューターはローカルでオプトインします。Gateway を有効化しても、別の Node に Codex メタデータの読み取りを許可することにはなりません。Node 機能は通常のペアリングおよびコマンドポリシーの承認を通過する必要があります。

フリートの一覧表示とトランスクリプトの表示では、ペアリング済み Node を呼び出すため、`operator.write` Gateway スコープを使用します。ローカルでの継続とアーカイブは認証済みオペレーターの操作であり、引き続きホストおよび状態のチェック対象です。

自律エージェントおよびスタンドアロン MCP のアクセスは別です。出荷済みの
`codex_endpoint_probe`、`codex_sessions_list`、`codex_session_read`、
`codex_session_send`、および `codex_session_interrupt` ツール契約は、引き続き
`codex` Plugin が所有します。監督が有効な場合、生の `codex_threads` トランスクリプト読み取りとトランスクリプト由来のリストフィールドにも
`supervision.allowRawTranscripts` が必要です。すべての `codex_threads` フォーク、名前変更、アーカイブ、
またはアーカイブ解除には `supervision.allowWriteControls` が必要です。どちらのポリシーもデフォルトでは無効です。

## 互換性

`openclaw doctor --fix` は、エンドポイントとトランスクリプト／書き込みポリシー、および Plugin の許可／拒否参照を含む、出荷済みの `plugins.entries.codex-supervisor`
設定を
`plugins.entries.codex.config.supervision` へ移行します。競合時には、明示的な正規の移行先値が優先されます。移行後、ランタイムコードは正規の `codex` Plugin
形式のみを使用します。

公式 Plugin は、Supervisor 互換ツールを正確に 5 つ保持します。
`codex_endpoint_probe`、`codex_sessions_list`、`codex_session_read`、
`codex_session_send`、および `codex_session_interrupt` です。セッションリストはデフォルトで読み込み済みのもののみであり、`loaded_only` パラメーターはありません。`include_stored: true` は、エンドポイントごとに `max_stored_sessions`
（デフォルト 200、許容範囲 1～1,000）で制限された非アーカイブ済み状態データベース行を追加します。読み込み済みの行は、この設定による上限の対象外です。トランスクリプト由来のフィールドと読み取りは引き続き
`allowRawTranscripts` によって制御され、送信と割り込みは引き続き `allowWriteControls` によって制御されます。

互換送信は、アイドル状態のスレッドを開始または再開しません。`mode: "start"` は常に拒否され、`"auto"` と `"steer"` は読み取り可能なアクティブターンのみを制御します。
割り込みにもアクティブで読み取り可能なターンが必要です。アイドル状態からの継続はネイティブ Codex カタログへルーティングされるため、完全なハーネスが承認、ツール、およびバインディングを所有します。スタンドアロンのレガシー MCP アダプターは、これらと同じツールを公式 Plugin から解決し、保持されたレガシーポリシー環境変数に従う唯一の経路です。

7 月のカタログ UI、Gateway メソッド、Node 機能、および CLI 登録は、旧 Plugin ID では出荷されていませんでした。これらは第 2 のランタイムファサードを介さず、直接 `codex` の所有へ移行します。

## 受け入れテスト

- 監視を有効にすると、アーカイブされていないローカルセッションが一覧表示されます。
- アーカイブされたセッションは、カタログのレスポンスにも UI にも表示されません。
- 別のホストに障害が発生しても正常なホストは表示されたままになり、利用できないホストは
  オフラインというセッションステータスを捏造するのではなく、新しい行を返しません。
- 保存済みまたはアイドル状態のローカル行から、Codex 専用の
  モデル／ランタイムロックを持つ Chat ミラーが作成されます。最初のターンで一時スナップショットが固定され、
  正規の完全なハーネススレッドが開始されます。Continue を繰り返すと、既存の Chat が開きます。
- 最初のターンでは、スナップショットのフォーク時にモデル／プロバイダーのオーバーライドを省略し、
  Codex が現在のモデルはソースに最後に記録されたモデルと異なると警告した場合でも、
  正規の開始時点を Codex が返した厳密な組み合わせに固定します。
- 保留中およびコミット済みの監視対象バインディングでは、ソースへのアクセス、
  正規ブランチの作成、およびその後のすべてのターンに監視接続を使用します。通常の
  Codex セッションは引き続きエージェントスコープです。
- 以降の再開では OpenClaw のモデル／プロバイダーのオーバーライドを省略し、Codex の
  正規の永続化済み選択を維持し、そのスレッドに対する個別のネイティブ変更を受け入れ、
  外側の OpenClaw モデルやフォールバックチェーンに置き換えることはありません。
- 監視を無効にした場合、またはバインディング／接続のライフサイクルが失われた場合は、Chat を
  通常のエージェントホームのハーネスへ移動せず、フェイルクローズします。
- 監視対象でモデルロックされた Chat は、ネイティブバインディングを保護している間は
  削除できません。
- Chat がミラーリングするユーザーおよびアシスタントのメッセージは、最大 200 件、
  合計 512 KiB、メッセージごとに 64 KiB です。画像はプレースホルダーに変換され、ソースの推論、ツール呼び出し、
  ツール結果、画像ペイロード、ローカルパスは複製されません。
- ブランチフローがソーススレッドを再開することはありません。
- 元のソースは両方のカタログで引き続き対象となります。正規のネイティブ
  ブランチは `appServer` ソース種別を使用し、Codex Desktop に表示される保証は
  ありません。
- アクティブなローカルソースでは、ブランチの作成もアーカイブもできませんが、既存の
  監視対象 Chat は引き続き開けます。
- アクティビティが不明な行は確認なしでブランチを作成できますが、アーカイブには
  他のランナーが存在しないことの明示的な確認が必要です。
- 初期化中または保留中の監視対象ブランチを持つソースは、最初の Chat ターンによって
  正規ブランチが実体化されるまでアーカイブできません。
- 厳密な対象またはアーカイブされていない派生子孫に既知のアクティブなバインディング所有者が
  存在する場合、アーカイブはブロックされます。子孫の列挙に失敗した場合はフェイルクローズし、
  不明なクライアントとステータス確認からアーカイブまでの競合については、
  引き続き明示的な確認が責任を負います。
- 確認済みの保存済みまたはアイドル状態のローカルアーカイブでは、ネイティブ側で成功した後に行を削除します。
- ペアリング済み Node の行は、Continue または Archive なしでも表示されたままになります。
- 受動的な一覧表示では、スレッドの承認をサブスクライブしたり応答したりしません。
- 従来の Supervisor 設定は、正規の Codex 設定形式に移行されます。
- 従来の一覧はデフォルトで読み込み専用です。保存済み項目の列挙はエンドポイントごとの
  上限に従い、互換送信によってアイドル状態のスレッドが開始または再開されることはありません。
