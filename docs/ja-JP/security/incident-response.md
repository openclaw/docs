# インシデント対応

## 1. 検知とトリアージ

セキュリティシグナルは以下から監視します。

- GitHub セキュリティアドバイザリ (GHSA) と非公開の脆弱性報告。
- 報告が機密ではない場合の公開 GitHub issue/ディスカッション。
- 自動化されたシグナル (例: Dependabot、CodeQL、npm アドバイザリ、シークレットスキャン)。

初期トリアージ:

1. 影響を受けるコンポーネント、バージョン、トラスト境界への影響を確認します。
2. リポジトリの `SECURITY.md` のスコープと対象外ルールを使って、セキュリティ問題か、ハードニング/対応不要かを分類します。
3. インシデントオーナーがそれに応じて対応します。

## 2. 評価

重大度ガイド:

- **Critical:** パッケージ/リリース/リポジトリの侵害、能動的な悪用、または高影響の制御やデータ露出を伴う未認証のトラスト境界バイパス。
- **High:** 限定的な前提条件を必要とする、検証済みのトラスト境界バイパス (例: 認証済みだが権限のない高影響アクション)、または OpenClaw が所有する機密認証情報の露出。
- **Medium:** 実用上の影響がある重大なセキュリティ上の弱点だが、悪用可能性が制限されている、または相当な前提条件があるもの。
- **Low:** 多層防御の指摘、狭い範囲のサービス拒否、または実証済みのトラスト境界バイパスを伴わないハードニング/同等性の不足。

## 3. 対応

1. 報告者に受領を確認します (機密の場合は非公開で)。
2. サポート対象リリースと最新の `main` で再現し、回帰カバレッジを含むパッチを実装して検証します。
3. Critical/High のインシデントでは、実務上可能な限り速やかにパッチ適用済みリリースを準備します。
4. Medium/Low のインシデントでは、通常のリリースフローでパッチを適用し、緩和策のガイダンスを文書化します。

## 4. コミュニケーション

以下を通じて連絡します。

- 影響を受けるリポジトリの GitHub セキュリティアドバイザリ。
- 修正済みバージョンのリリースノート/チェンジログエントリ。
- ステータスと解決に関する報告者への直接フォローアップ。

開示ポリシー:

- Critical/High のインシデントは、適切な場合に CVE 発行を伴う協調的開示を行うべきです。
- 低リスクのハードニング指摘は、影響とユーザー露出に応じて、CVE なしでリリースノートまたはアドバイザリに文書化される場合があります。

## 5. 復旧とフォローアップ

修正を出荷した後:

1. CI とリリースアーティファクトで修復を検証します。
2. 短いインシデント後レビュー (タイムライン、根本原因、検知ギャップ、予防計画) を実施します。
3. フォローアップのハードニング/テスト/ドキュメントタスクを追加し、完了まで追跡します。
