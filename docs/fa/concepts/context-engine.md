یک **موتور زمینه** کنترل می‌کند که OpenClaw چگونه زمینهٔ مدل را برای هر اجرا می‌سازد: کدام پیام‌ها گنجانده شوند، تاریخچهٔ قدیمی‌تر چگونه خلاصه شود و زمینه در مرزهای زیرعامل چگونه مدیریت شود.

OpenClaw با یک موتور داخلی `legacy` عرضه می‌شود و به‌طور پیش‌فرض از آن استفاده می‌کند. تنها زمانی یک موتور Plugin را نصب و انتخاب کنید که رفتار متفاوتی برای مونتاژ، Compaction یا یادآوری میان‌نشستی می‌خواهید.

## نحوهٔ کار

هر بار که OpenClaw یک پرامپت مدل را اجرا می‌کند، موتور زمینه در چهار نقطه از چرخهٔ حیات مشارکت دارد:

<AccordionGroup>
  <Accordion title="1. دریافت">
    هنگامی فراخوانی می‌شود که پیام جدیدی به نشست افزوده شود. موتور می‌تواند پیام را در مخزن دادهٔ خود ذخیره یا نمایه‌سازی کند.
  </Accordion>
  <Accordion title="2. مونتاژ">
    پیش از هر اجرای مدل فراخوانی می‌شود. موتور مجموعه‌ای مرتب از پیام‌ها (و یک `systemPromptAddition` اختیاری) را برمی‌گرداند که در بودجهٔ توکن جای می‌گیرند.
  </Accordion>
  <Accordion title="3. فشرده‌سازی">
    هنگامی فراخوانی می‌شود که پنجرهٔ زمینه پر باشد یا کاربر `/compact` را اجرا کند. موتور برای آزادکردن فضا، تاریخچهٔ قدیمی‌تر را خلاصه می‌کند.
  </Accordion>
  <Accordion title="4. پس از نوبت">
    پس از تکمیل اجرا فراخوانی می‌شود. موتور می‌تواند وضعیت را ماندگار کند، Compaction پس‌زمینه را فعال کند یا نمایه‌ها را به‌روزرسانی کند.
  </Accordion>
</AccordionGroup>

موتورها همچنین می‌توانند یک متد اختیاری `maintain()` را برای نگه‌داری رونوشت (بازنویسی‌های امن از طریق `runtimeContext.rewriteTranscriptEntries()`) پس از راه‌اندازی اولیه، یک نوبت موفق یا Compaction پیاده‌سازی کنند. برای اجرای آن به‌صورت کار معوق به‌جای مسدودکردن پاسخ، `info.turnMaintenanceMode: "background"` را تنظیم کنید.

برای چارچوب Codex غیر ACP همراه، OpenClaw همین چرخهٔ حیات را با نگاشت زمینهٔ مونتاژشده به دستورالعمل‌های توسعه‌دهندهٔ Codex و پرامپت نوبت جاری اعمال می‌کند. Codex همچنان تاریخچهٔ بومی رشته و فشرده‌ساز بومی خود را مدیریت می‌کند.

### چرخهٔ حیات زیرعامل (اختیاری)

OpenClaw دو هوک اختیاری چرخهٔ حیات زیرعامل را فراخوانی می‌کند:

<ParamField path="prepareSubagentSpawn" type="method">
  وضعیت زمینهٔ مشترک را پیش از آغاز اجرای فرزند آماده می‌کند. این هوک کلیدهای نشست والد/فرزند، `contextMode` (`isolated` یا `fork`)، شناسه‌ها/فایل‌های رونوشت موجود و TTL اختیاری را دریافت می‌کند. اگر یک دستگیرهٔ بازگردانی برگرداند، وقتی ایجاد پس از آماده‌سازی موفق شکست بخورد، OpenClaw آن را فراخوانی می‌کند. ایجادهای بومی زیرعامل که `lightContext` را درخواست می‌کنند و به `contextMode="isolated"` تفکیک می‌شوند، عمداً این هوک را نادیده می‌گیرند تا فرزند بدون وضعیت پیش از ایجادِ مدیریت‌شده توسط موتور زمینه، از زمینهٔ سبک راه‌اندازی اولیه آغاز کند.
</ParamField>
<ParamField path="onSubagentEnded" type="method">
  هنگامی که نشست یک زیرعامل تکمیل یا پاک‌سازی می‌شود، پاک‌سازی را انجام می‌دهد.
</ParamField>

### افزودهٔ پرامپت سیستم

متد `assemble` می‌تواند یک رشتهٔ `systemPromptAddition` برگرداند. OpenClaw آن را به ابتدای پرامپت سیستم اجرا اضافه می‌کند. این قابلیت به موتورها اجازه می‌دهد بدون نیاز به فایل‌های ایستای فضای کاری، راهنمای پویای یادآوری، دستورالعمل‌های بازیابی یا نکته‌های آگاه از زمینه را تزریق کنند.

## موتور قدیمی

موتور داخلی `legacy` رفتار اصلی OpenClaw را حفظ می‌کند:

- **دریافت**: بدون عملیات (مدیر نشست، ماندگارسازی پیام را مستقیماً مدیریت می‌کند).
- **مونتاژ**: عبوری (خط لولهٔ موجود پاک‌سازی ← اعتبارسنجی ← محدودسازی در زمان اجرا، مونتاژ زمینه را مدیریت می‌کند).
- **فشرده‌سازی**: به Compaction خلاصه‌سازی داخلی واگذار می‌کند که یک خلاصهٔ واحد از پیام‌های قدیمی‌تر می‌سازد و پیام‌های اخیر را دست‌نخورده نگه می‌دارد.
- **پس از نوبت**: بدون عملیات.

موتور قدیمی هیچ ابزاری ثبت نمی‌کند و `systemPromptAddition` ارائه نمی‌دهد.

وقتی هیچ `plugins.slots.contextEngine` تنظیم نشده باشد (یا روی `"legacy"` تنظیم شده باشد)، این موتور به‌طور خودکار استفاده می‌شود.

## موتورهای Plugin

یک Plugin می‌تواند با استفاده از API افزونه، یک موتور زمینه ثبت کند:

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";
import { resolveSessionAgentId } from "openclaw/plugin-sdk/memory-host-core";

export default function register(api) {
  api.registerContextEngine("my-engine", (ctx) => ({
    info: {
      id: "my-engine",
      name: "My Context Engine",
      ownsCompaction: true,
    },

    async ingest({ sessionId, message, isHeartbeat }) {
      // پیام را در مخزن دادهٔ خود ذخیره کنید
      return { ingested: true };
    },

    async assemble({
      sessionId,
      sessionKey,
      messages,
      tokenBudget,
      availableTools,
      citationsMode,
    }) {
      // پیام‌هایی را برگردانید که در بودجه جای می‌گیرند
      return {
        messages: buildContext(messages, tokenBudget),
        estimatedTokens: countTokens(messages),
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
          agentId: resolveSessionAgentId({ config: ctx.config, sessionKey }),
          agentSessionKey: sessionKey,
        }),
      };
    },

    async compact({ sessionId, force }) {
      // زمینهٔ قدیمی‌تر را خلاصه کنید
      return { ok: true, compacted: true };
    },
  }));
}
```

کارخانهٔ `ctx` شامل مقادیر اختیاری `config`، `agentDir` و `workspaceDir` است
تا Pluginها بتوانند پیش از اجرای نخستین هوک چرخهٔ حیات، وضعیت هر عامل یا هر فضای کاری را
مقداردهی اولیه کنند.

سپس آن را در پیکربندی فعال کنید:

```json5
{
  plugins: {
    slots: {
      contextEngine: "my-engine",
    },
    entries: {
      "my-engine": {
        enabled: true,
      },
    },
  },
}
```

### رابط ContextEngine

اعضای الزامی:

| عضو             | نوع     | هدف                                                  |
| ------------------ | -------- | -------------------------------------------------------- |
| `info`             | ویژگی | شناسه، نام و نسخهٔ موتور و اینکه آیا Compaction را مدیریت می‌کند |
| `ingest(params)`   | متد   | ذخیره‌کردن یک پیام                                   |
| `assemble(params)` | متد   | ساخت زمینه برای اجرای مدل (`AssembleResult` را برمی‌گرداند) |
| `compact(params)`  | متد   | خلاصه‌سازی/کاهش زمینه                                 |

`assemble` یک `AssembleResult` با موارد زیر برمی‌گرداند:

<ParamField path="messages" type="Message[]" required>
  پیام‌های مرتبی که باید به مدل ارسال شوند.
</ParamField>
<ParamField path="estimatedTokens" type="number" required>
  برآورد موتور از مجموع توکن‌های زمینهٔ مونتاژشده. OpenClaw از این مقدار برای تصمیم‌گیری دربارهٔ آستانهٔ Compaction و گزارش تشخیصی استفاده می‌کند.
</ParamField>
<ParamField path="systemPromptAddition" type="string">
  به ابتدای پرامپت سیستم افزوده می‌شود.
</ParamField>
<ParamField path="promptAuthority" type='"assembled" | "preassembly_may_overflow"'>
  تعیین می‌کند اجراکننده برای پیش‌بررسی‌های پیشگیرانهٔ سرریز از کدام برآورد توکن
  استفاده کند. مقدار پیش‌فرض `"assembled"` است؛ یعنی برای موتورهایی که
  مالک Compaction نیستند، فقط برآورد پرامپت مونتاژشده بررسی می‌شود.
  موتورهایی که `ownsCompaction: true` را تنظیم می‌کنند، پذیرش پرامپت خود را مدیریت می‌کنند؛
  بنابراین OpenClaw به‌طور پیش‌فرض پیش‌بررسی عمومی پیش از پرامپت را رد می‌کند. تنها زمانی
  `"preassembly_may_overflow"` را تنظیم کنید که نمای مونتاژشدهٔ شما بتواند خطر سرریز
  در رونوشت زیربنایی را پنهان کند؛ در این صورت اجراکننده پیش‌بررسی عمومی را
  فعال نگه می‌دارد و هنگام تصمیم‌گیری دربارهٔ Compaction پیشگیرانه، بیشینهٔ
  برآورد مونتاژشده و برآورد تاریخچهٔ نشست پیش از مونتاژ (بدون پنجره‌بندی) را
  در نظر می‌گیرد. در هر دو حالت، پیام‌هایی که برمی‌گردانید همچنان همان‌هایی هستند که
  مدل می‌بیند؛ `promptAuthority` فقط بر پیش‌بررسی اثر می‌گذارد.
</ParamField>
<ParamField path="contextProjection" type="ContextEngineProjection">
  چرخهٔ حیات نگاشت اختیاری برای میزبان‌هایی با رشته‌های پایدار بخش پشتی (برای مثال Codex app-server). `mode: "thread_bootstrap"` همراه با یک `epoch` پایدار از میزبان می‌خواهد زمینهٔ مونتاژشده را در هر دوره یک‌بار تزریق کند و تا زمانی که دوره تغییر نکرده است، به‌جای نگاشت مجدد در هر نوبت، از رشتهٔ بخش پشتی دوباره استفاده کند. برای نگاشت عادی در هر نوبت، این فیلد را حذف کنید.
</ParamField>

`compact` یک `CompactResult` برمی‌گرداند. هنگامی که Compaction هویت نشست فعال را
تغییر می‌دهد، `result.sessionTarget` (یک `ContextEngineSessionTarget` نوع‌دار که
هویت نشست و دامنهٔ مخزن را حمل می‌کند) نشست جانشینی را مشخص می‌کند که
تلاش مجدد یا نوبت بعدی باید از آن استفاده کند؛ `result.sessionId` شناسهٔ جانشین را بازتاب می‌دهد.

اعضای اختیاری:

| عضو                         | نوع   | هدف                                                                                                                                      |
| ------------------------------ | ------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `bootstrap(params)`            | متد | مقداردهی اولیهٔ وضعیت موتور برای یک نشست. یک‌بار، هنگامی که موتور برای نخستین بار یک نشست را می‌بیند، فراخوانی می‌شود (برای مثال، درون‌ریزی تاریخچه).                              |
| `maintain(params)`             | متد | نگه‌داری رونوشت پس از راه‌اندازی اولیه، یک نوبت موفق یا Compaction. برای بازنویسی‌های امن از `runtimeContext.rewriteTranscriptEntries()` استفاده کنید. |
| `ingestBatch(params)`          | متد | دریافت دسته‌ای یک نوبت تکمیل‌شده. پس از تکمیل اجرا، با همهٔ پیام‌های آن نوبت به‌صورت یکجا فراخوانی می‌شود.                                  |
| `afterTurn(params)`            | متد | کار چرخهٔ حیات پس از اجرا (ماندگارسازی وضعیت، فعال‌کردن Compaction پس‌زمینه).                                                                      |
| `prepareSubagentSpawn(params)` | متد | راه‌اندازی وضعیت مشترک برای نشست فرزند پیش از آغاز آن.                                                                                    |
| `onSubagentEnded(params)`      | متد | پاک‌سازی پس از پایان یک زیرعامل.                                                                                                              |
| `dispose()`                    | متد | آزادسازی منابع. هنگام خاموش‌شدن Gateway یا بارگذاری مجدد Plugin فراخوانی می‌شود، نه برای هر نشست.                                                        |

### تنظیمات زمان اجرا

هوک‌های چرخهٔ حیاتی که درون OpenClaw اجرا می‌شوند، یک شیء اختیاری
`runtimeSettings` دریافت می‌کنند. این یک سطح API داخلی، نسخه‌بندی‌شده و فقط‌خواندنی
برای تولیدکننده/مصرف‌کننده است: OpenClaw آن را برای موتور زمینهٔ انتخاب‌شده تولید
می‌کند و موتور زمینه آن را در هوک‌های چرخهٔ حیات مصرف می‌کند. این شیء
مستقیماً برای کاربران نمایش داده نمی‌شود و سطح گزارش‌دهی اختصاصی ایجاد نمی‌کند.

- `schemaVersion`: در حال حاضر `1`
- `runtime`: میزبان OpenClaw، حالت زمان اجرا (`normal`، `fallback`، یا
  `degraded`) و شناسه‌های اختیاری چارچوب آزمایشی/زمان اجرا
- `contextEngineSelection`: شناسهٔ موتور زمینهٔ انتخاب‌شده و منبع انتخاب
- `executionHost`: شناسه و برچسب میزبان برای سطحی که هوک را فراخوانی می‌کند
- `model`: مدل درخواست‌شده، مدل نهایی، ارائه‌دهنده و خانوادهٔ اختیاری مدل
- `limits`: بودجهٔ توکن پرامپت و حداکثر توکن‌های خروجی، در صورت مشخص‌بودن
- `diagnostics`: کدهای دلیل بازگشت بسته و تنزل عملکرد، در صورت مشخص‌بودن

فیلدهایی که ممکن است ناشناخته باشند با `null` نمایش داده می‌شوند؛ فیلدهای متمایزکننده‌ای مانند
حالت زمان اجرا و منبع انتخاب همچنان نمی‌توانند تهی باشند. موتورهای قدیمی‌تر همچنان
سازگارند: اگر یک موتور قدیمی سخت‌گیر `runtimeSettings` را به‌عنوان ویژگی ناشناخته
رد کند، OpenClaw به‌جای قرنطینه‌کردن موتور، فراخوانی چرخهٔ عمر را بدون آن
دوباره امتحان می‌کند.

### الزامات میزبان

موتورهای زمینه می‌توانند الزامات قابلیت میزبان را در `info.hostRequirements` اعلام کنند.
OpenClaw پیش از آغاز عملیات این الزامات را بررسی می‌کند و اگر زمان اجرای انتخاب‌شده
نتواند آن‌ها را برآورده کند، با یک خطای توصیفی به‌صورت بسته شکست می‌خورد.

برای اجرای عامل، زمانی `assemble-before-prompt` را اعلام کنید که موتور باید پرامپت
واقعی مدل را از طریق `assemble()` کنترل کند:

```ts
info: {
  id: "my-context-engine",
  name: "My Context Engine",
  hostRequirements: {
    "agent-run": {
      requiredCapabilities: ["assemble-before-prompt"],
      unsupportedMessage:
        "از Codex بومی یا زمان اجرای تعبیه‌شدهٔ OpenClaw استفاده کنید، یا موتور زمینهٔ قدیمی را انتخاب کنید.",
    },
  },
}
```

اجرای عامل در Codex بومی و OpenClaw تعبیه‌شده، `assemble-before-prompt` را برآورده می‌کند.
بک‌اندهای عمومی CLI آن را برآورده نمی‌کنند؛ بنابراین موتورهایی که به آن نیاز دارند، پیش از
آغاز فرایند CLI رد می‌شوند.

### جداسازی خرابی

OpenClaw موتور Plugin انتخاب‌شده را از مسیر اصلی پاسخ جدا می‌کند. اگر یک
موتور غیرقدیمی وجود نداشته باشد، اعتبارسنجی قرارداد را با شکست طی کند، هنگام ایجاد کارخانه
استثنا پرتاب کند یا یکی از متدهای چرخهٔ عمر آن استثنا پرتاب کند، OpenClaw آن موتور را
برای فرایند فعلی Gateway قرنطینه می‌کند و کارهای موتور زمینه را به موتور داخلی
`legacy` تنزل می‌دهد. خطا همراه با عملیات ناموفق ثبت می‌شود تا
اپراتور بتواند بدون ساکت‌شدن عامل، Plugin را تعمیر، به‌روزرسانی یا غیرفعال
کند.

خرابی الزامات میزبان متفاوت است: وقتی موتوری اعلام می‌کند که یک زمان اجرا
فاقد قابلیت الزامی است، OpenClaw پیش از آغاز اجرا به‌صورت بسته شکست می‌خورد. این کار
از موتورهایی محافظت می‌کند که اجرای آن‌ها در میزبانی پشتیبانی‌نشده موجب خرابی وضعیت می‌شود.

### ownsCompaction

`ownsCompaction` تعیین می‌کند که آیا Compaction خودکار داخلی حین تلاش در زمان اجرای OpenClaw برای آن اجرا فعال بماند:

<AccordionGroup>
  <Accordion title="ownsCompaction: true">
    موتور مالک رفتار Compaction است. OpenClaw، Compaction خودکار داخلی زمان اجرای OpenClaw و پیش‌بررسی عمومی سرریز پیش از پرامپت را برای آن اجرا غیرفعال می‌کند و پیاده‌سازی `compact()` موتور، مسئول `/compact`، Compaction بازیابی از سرریز ارائه‌دهنده و هر Compaction پیش‌دستانه‌ای است که می‌خواهد در `afterTurn()` انجام دهد. وقتی موتور از `assemble()` مقدار `promptAuthority: "preassembly_may_overflow"` را برگرداند، OpenClaw همچنان سازوکار محافظتی سرریز پیش از پرامپت را اجرا می‌کند.
  </Accordion>
  <Accordion title="ownsCompaction: false یا تنظیم‌نشده">
    Compaction خودکار داخلی زمان اجرای OpenClaw ممکن است همچنان هنگام اجرای پرامپت انجام شود، اما متد `compact()` موتور فعال همچنان برای `/compact` و بازیابی از سرریز فراخوانی می‌شود.
  </Accordion>
</AccordionGroup>

<Warning>
`ownsCompaction: false` به این معنا **نیست** که OpenClaw به‌طور خودکار به مسیر Compaction موتور قدیمی بازمی‌گردد.
</Warning>

یعنی دو الگوی معتبر برای Plugin وجود دارد:

<Tabs>
  <Tab title="حالت مالکیت">
    الگوریتم Compaction خود را پیاده‌سازی و `ownsCompaction: true` را تنظیم کنید.
  </Tab>
  <Tab title="حالت واگذاری">
    `ownsCompaction: false` را تنظیم کنید و برای استفاده از رفتار Compaction داخلی OpenClaw، کاری کنید که `compact()`، تابع `delegateCompactionToRuntime(...)` را از `openclaw/plugin-sdk/core` فراخوانی کند.
  </Tab>
</Tabs>

یک `compact()` بدون عملیات برای یک موتور فعالِ فاقد مالکیت ناامن است، زیرا مسیر عادی Compaction مربوط به `/compact` و بازیابی از سرریز را برای جایگاه آن موتور غیرفعال می‌کند.

## مرجع پیکربندی

```json5
{
  plugins: {
    slots: {
      // موتور زمینهٔ فعال را انتخاب کنید. پیش‌فرض: "legacy".
      // برای استفاده از موتور Plugin، آن را روی یک شناسهٔ Plugin تنظیم کنید.
      contextEngine: "legacy",
    },
  },
}
```

<Note>
این جایگاه هنگام اجرا انحصاری است — برای یک عملیات اجرای مشخص یا Compaction، تنها یک موتور زمینهٔ ثبت‌شده تعیین می‌شود. دیگر Pluginهای فعال `kind: "context-engine"` همچنان می‌توانند بارگذاری شوند و کد ثبت خود را اجرا کنند؛ `plugins.slots.contextEngine` فقط تعیین می‌کند OpenClaw هنگام نیاز به موتور زمینه، کدام شناسهٔ موتور ثبت‌شده را تعیین کند.
</Note>

<Note>
**حذف نصب Plugin:** وقتی Pluginای را که در حال حاضر به‌عنوان `plugins.slots.contextEngine` انتخاب شده است حذف نصب می‌کنید، OpenClaw جایگاه را به مقدار پیش‌فرض (`legacy`) بازنشانی می‌کند. همین رفتار بازنشانی برای `plugins.slots.memory` نیز اعمال می‌شود. نیازی به ویرایش دستی پیکربندی نیست.
</Note>

## ارتباط با Compaction و حافظه

<AccordionGroup>
  <Accordion title="Compaction">
    Compaction یکی از مسئولیت‌های موتور زمینه است. موتور قدیمی این کار را به خلاصه‌سازی داخلی OpenClaw واگذار می‌کند. موتورهای Plugin می‌توانند هر راهبرد Compaction را پیاده‌سازی کنند (خلاصه‌های DAG، بازیابی برداری و غیره).
  </Accordion>
  <Accordion title="Pluginهای حافظه">
    Pluginهای حافظه (`plugins.slots.memory`) از موتورهای زمینه جدا هستند. Pluginهای حافظه جست‌وجو/بازیابی را فراهم می‌کنند؛ موتورهای زمینه آنچه مدل می‌بیند کنترل می‌کنند. آن‌ها می‌توانند با یکدیگر کار کنند — ممکن است یک موتور زمینه هنگام مونتاژ از داده‌های Plugin حافظه استفاده کند. موتورهای Plugin که مسیر فعال پرامپت حافظه را می‌خواهند باید `buildMemorySystemPromptAddition(...)` از `openclaw/plugin-sdk/core` را ترجیح دهند؛ این تابع بخش‌های فعال پرامپت حافظه را به یک `systemPromptAddition` آماده برای افزودن در ابتدا تبدیل می‌کند. اگر موتوری به کنترل سطح پایین‌تری نیاز داشته باشد، همچنان می‌تواند خطوط خام را از `openclaw/plugin-sdk/memory-host-core` از طریق `buildActiveMemoryPromptSection(...)` دریافت کند.
  </Accordion>
  <Accordion title="هرس نشست">
    کوتاه‌کردن نتایج قدیمی ابزار در حافظه، مستقل از اینکه کدام موتور زمینه فعال است، همچنان انجام می‌شود.
  </Accordion>
</AccordionGroup>

## نکته‌ها

- برای اطمینان از بارگذاری صحیح موتور خود، از `openclaw doctor` استفاده کنید.
- هنگام تعویض موتور، نشست‌های موجود با تاریخچهٔ فعلی خود ادامه می‌یابند. موتور جدید اجرای دفعات بعدی را بر عهده می‌گیرد.
- خطاهای موتور ثبت می‌شوند و موتور Plugin انتخاب‌شده برای فرایند فعلی Gateway قرنطینه می‌شود. OpenClaw برای نوبت‌های کاربر به `legacy` بازمی‌گردد تا پاسخ‌ها بتوانند ادامه پیدا کنند، اما همچنان باید Plugin خراب را تعمیر، به‌روزرسانی، غیرفعال یا حذف نصب کنید.
- برای توسعه، از `openclaw plugins install -l ./my-engine` استفاده کنید تا یک پوشهٔ محلی Plugin را بدون کپی‌کردن پیوند دهید.
