ما اجرای پاسخ خودکار ورودی (همه کانال‌ها) را از طریق یک صف کوچک درون‌فرایندی سریال‌سازی می‌کنیم تا از برخورد چند اجرای agent جلوگیری شود، در حالی که همچنان موازی‌سازی ایمن بین sessionها ممکن بماند.

## چرا

- اجرای پاسخ خودکار می‌تواند پرهزینه باشد (فراخوانی‌های LLM) و وقتی چند پیام ورودی با فاصله کم می‌رسند می‌تواند با هم برخورد کند.
- سریال‌سازی از رقابت بر سر منابع مشترک (فایل‌های session، گزارش‌ها، stdin مربوط به CLI) جلوگیری می‌کند و احتمال رسیدن به محدودیت‌های نرخ بالادستی را کاهش می‌دهد.

## نحوه کار

- یک صف FIFO آگاه از lane، هر lane را با سقف هم‌روندی قابل پیکربندی تخلیه می‌کند (پیش‌فرض 1 برای laneهای پیکربندی‌نشده؛ main به‌طور پیش‌فرض 4 و subagent به‌طور پیش‌فرض 8 است).
- `runEmbeddedPiAgent` بر اساس **کلید session** (lane `session:<key>`) در صف قرار می‌دهد تا تضمین کند برای هر session فقط یک اجرای فعال وجود دارد.
- سپس هر اجرای session در یک **lane سراسری** (`main` به‌طور پیش‌فرض) صف می‌شود تا موازی‌سازی کلی با `agents.defaults.maxConcurrent` محدود شود.
- وقتی گزارش‌گیری مفصل فعال باشد، اجراهای صف‌شده اگر پیش از شروع بیش از حدود 2 ثانیه منتظر مانده باشند، یک اعلان کوتاه منتشر می‌کنند.
- نشانگرهای تایپ همچنان بلافاصله هنگام قرار گرفتن در صف فعال می‌شوند (وقتی کانال پشتیبانی کند)، بنابراین تجربه کاربر هنگام انتظار برای نوبت تغییر نمی‌کند.

## پیش‌فرض‌ها

وقتی تنظیم نشده باشد، همه سطوح کانال ورودی از این موارد استفاده می‌کنند:

- `mode: "steer"`
- `debounceMs: 500`
- `cap: 20`
- `drop: "summarize"`

`steer` پیش‌فرض است چون نوبت مدل فعال را پاسخ‌گو نگه می‌دارد بدون اینکه
اجرای session دومی را شروع کند. این حالت همه پیام‌های هدایت‌گری را که
پیش از مرز بعدی مدل رسیده‌اند تخلیه می‌کند. اگر اجرای فعلی نتواند هدایت‌گری را بپذیرد،
OpenClaw به یک ورودی صف followup بازمی‌گردد.

## حالت‌های صف

پیام‌های ورودی می‌توانند اجرای فعلی را هدایت کنند، برای نوبت followup منتظر بمانند، یا هر دو را انجام دهند:

- `steer`: پیام‌های هدایت‌گری را در runtime فعال صف می‌کند. Pi همه پیام‌های هدایت‌گری معلق را **پس از پایان اجرای فراخوانی‌های ابزار توسط نوبت assistant فعلی** و پیش از فراخوانی LLM بعدی تحویل می‌دهد؛ app-server مربوط به Codex یک `turn/steer` دسته‌ای دریافت می‌کند. اگر اجرا به‌طور فعال در حال streaming نباشد یا هدایت‌گری در دسترس نباشد، OpenClaw به یک ورودی صف followup بازمی‌گردد.
- `queue` (قدیمی): هدایت‌گری قدیمی یکی‌یکی. Pi در هر مرز مدل، یک پیام هدایت‌گری صف‌شده را تحویل می‌دهد؛ app-server مربوط به Codex درخواست‌های جداگانه `turn/steer` دریافت می‌کند. مگر اینکه به رفتار سریال‌سازی‌شده قبلی نیاز داشته باشید، `steer` را ترجیح دهید.
- `followup`: هر پیام را برای نوبت agent بعدی پس از پایان اجرای فعلی در صف قرار می‌دهد.
- `collect`: پیام‌های صف‌شده را پس از پنجره سکوت در یک نوبت followup **واحد** ادغام می‌کند. اگر پیام‌ها کانال‌ها/رشته‌های متفاوتی را هدف بگیرند، برای حفظ مسیریابی به‌صورت جداگانه تخلیه می‌شوند.
- `steer-backlog` (یا `steer+backlog`): اکنون هدایت می‌کند **و** همان پیام را برای یک نوبت followup نگه می‌دارد.
- `interrupt` (قدیمی): اجرای فعال آن session را لغو می‌کند، سپس جدیدترین پیام را اجرا می‌کند.

Steer-backlog یعنی ممکن است پس از اجرای هدایت‌شده یک پاسخ followup دریافت کنید، بنابراین
سطوح streaming ممکن است شبیه تکراری‌ها به نظر برسند. اگر برای هر پیام ورودی
یک پاسخ می‌خواهید، `collect`/`steer` را ترجیح دهید.

برای زمان‌بندی و رفتار وابستگی مختص runtime، ببینید
[صف هدایت‌گری](/fa/concepts/queue-steering).

به‌صورت سراسری یا برای هر کانال از طریق `messages.queue` پیکربندی کنید:

```json5
{
  messages: {
    queue: {
      mode: "steer",
      debounceMs: 500,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## گزینه‌های صف

گزینه‌ها روی `followup`، `collect` و `steer-backlog` اعمال می‌شوند (و وقتی هدایت‌گری به followup بازمی‌گردد، روی `steer` یا `queue` قدیمی نیز اعمال می‌شوند):

- `debounceMs`: پنجره سکوت پیش از تخلیه followupهای صف‌شده. عددهای بدون واحد میلی‌ثانیه هستند؛ واحدهای `ms`، `s`، `m`، `h` و `d` توسط گزینه‌های `/queue` پذیرفته می‌شوند.
- `cap`: حداکثر پیام‌های صف‌شده برای هر session. مقدارهای کمتر از `1` نادیده گرفته می‌شوند.
- `drop: "summarize"`: پیش‌فرض. قدیمی‌ترین ورودی‌های صف‌شده را در صورت نیاز حذف می‌کند، خلاصه‌های فشرده را نگه می‌دارد، و آن‌ها را به‌عنوان یک prompt مصنوعی followup تزریق می‌کند.
- `drop: "old"`: قدیمی‌ترین ورودی‌های صف‌شده را در صورت نیاز حذف می‌کند، بدون حفظ خلاصه‌ها.
- `drop: "new"`: وقتی صف از قبل پر باشد، جدیدترین پیام را رد می‌کند.

پیش‌فرض‌ها: `debounceMs: 500`، `cap: 20`، `drop: summarize`.

## تقدم

برای انتخاب حالت، OpenClaw به این ترتیب حل می‌کند:

1. بازنویسی `/queue` درون‌خطی یا ذخیره‌شده برای هر session.
2. `messages.queue.byChannel.<channel>`.
3. `messages.queue.mode`.
4. `steer` پیش‌فرض.

برای گزینه‌ها، گزینه‌های `/queue` درون‌خطی یا ذخیره‌شده بر config مقدم هستند. سپس
debounce مختص کانال (`messages.queue.debounceMsByChannel`)، پیش‌فرض‌های debounce مربوط به Plugin،
گزینه‌های سراسری `messages.queue` و پیش‌فرض‌های داخلی اعمال می‌شوند. `cap` و `drop` گزینه‌های سراسری/session هستند، نه کلیدهای config برای هر کانال.

## بازنویسی‌های هر session

- برای ذخیره حالت برای session فعلی، `/queue <mode>` را به‌عنوان یک فرمان مستقل بفرستید.
- گزینه‌ها می‌توانند ترکیب شوند: `/queue collect debounce:0.5s cap:25 drop:summarize`
- `/queue default` یا `/queue reset` بازنویسی session را پاک می‌کند.

## دامنه و تضمین‌ها

- روی اجرای agent پاسخ خودکار در همه کانال‌های ورودی که از pipeline پاسخ Gateway استفاده می‌کنند اعمال می‌شود (WhatsApp web، Telegram، Slack، Discord، Signal، iMessage، webchat، و غیره).
- lane پیش‌فرض (`main`) در سطح فرایند برای ورودی + Heartbeatهای اصلی مشترک است؛ برای اجازه دادن به چند session به‌صورت موازی، `agents.defaults.maxConcurrent` را تنظیم کنید.
- laneهای اضافی ممکن است وجود داشته باشند (مثل `cron`، `cron-nested`، `nested`، `subagent`) تا کارهای پس‌زمینه بتوانند بدون مسدود کردن پاسخ‌های ورودی به‌صورت موازی اجرا شوند. نوبت‌های agent مربوط به cron ایزوله، هنگام اجرای agent داخلی خود از `cron-nested` استفاده می‌کنند و یک جایگاه `cron` را نگه می‌دارند؛ هر دو از `cron.maxConcurrentRuns` استفاده می‌کنند. جریان‌های `nested` غیر cron مشترک رفتار lane خود را حفظ می‌کنند. این اجراهای جداشده به‌عنوان [کارهای پس‌زمینه](/fa/automation/tasks) رهگیری می‌شوند.
- laneهای هر session تضمین می‌کنند که در هر لحظه فقط یک اجرای agent به یک session مشخص دست بزند.
- بدون وابستگی خارجی یا نخ‌های worker پس‌زمینه؛ فقط TypeScript + promiseها.

## عیب‌یابی

- اگر فرمان‌ها گیر کرده به نظر می‌رسند، گزارش‌های مفصل را فعال کنید و برای تأیید تخلیه شدن صف، به‌دنبال خطوط «queued for …ms» بگردید.
- اگر به عمق صف نیاز دارید، گزارش‌های مفصل را فعال کنید و خطوط زمان‌بندی صف را دنبال کنید.
- وقتی diagnostics فعال باشد، sessionهایی که پس از `diagnostics.stuckSessionWarnMs` در `processing` باقی می‌مانند یک هشدار session گیرکرده ثبت می‌کنند. اجراهای embedded فعال، عملیات پاسخ فعال و کارهای lane فعال به‌طور پیش‌فرض فقط هشدار باقی می‌مانند؛ bookkeeping شروع قدیمی بدون کار session فعال می‌تواند lane session متاثر را آزاد کند تا کارهای صف‌شده تخلیه شوند.
