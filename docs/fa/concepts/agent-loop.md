حلقه عامل‌محور اجرای کامل و «واقعی» یک عامل است: دریافت → گردآوری زمینه → استنتاج مدل →
اجرای ابزار → پاسخ‌های جریانی → پایداری. این مسیر مرجع است که یک پیام را
به کنش‌ها و پاسخ نهایی تبدیل می‌کند، در حالی که وضعیت نشست را سازگار نگه می‌دارد.

در OpenClaw، حلقه یک اجرای واحد و سری‌شده برای هر نشست است که هنگام فکر کردن مدل،
فراخوانی ابزارها، و جریان دادن خروجی، رویدادهای چرخه حیات و جریان را منتشر می‌کند. این سند توضیح می‌دهد که آن حلقه اصیل
چگونه از ابتدا تا انتها سیم‌کشی شده است.

## نقاط ورود

- RPC در Gateway: `agent` و `agent.wait`.
- CLI: فرمان `agent`.

## نحوه کار (سطح بالا)

1. RPC `agent` پارامترها را اعتبارسنجی می‌کند، نشست را حل می‌کند (sessionKey/sessionId)، فراداده نشست را پایدار می‌کند، و بلافاصله `{ runId, acceptedAt }` را برمی‌گرداند.
2. `agentCommand` عامل را اجرا می‌کند:
   - مدل + پیش‌فرض‌های thinking/verbose/trace را حل می‌کند
   - عکس فوری Skills را بارگذاری می‌کند
   - `runEmbeddedPiAgent` را فراخوانی می‌کند (runtime مربوط به pi-agent-core)
   - اگر حلقه تعبیه‌شده یکی منتشر نکند، **پایان/خطای چرخه حیات** را منتشر می‌کند
3. `runEmbeddedPiAgent`:
   - اجراها را از طریق صف‌های هر نشست + صف‌های سراسری سری می‌کند
   - مدل + پروفایل احراز هویت را حل می‌کند و نشست pi را می‌سازد
   - در رویدادهای pi مشترک می‌شود و دلتاهای دستیار/ابزار را جریان می‌دهد
   - مهلت زمانی را اعمال می‌کند -> اگر از حد بگذرد اجرا را لغو می‌کند
   - برای نوبت‌های app-server در Codex، نوبت پذیرفته‌شده‌ای را که پیش از رویداد پایانی دیگر پیشرفت app-server تولید نمی‌کند لغو می‌کند
   - بارهای داده + فراداده مصرف را برمی‌گرداند
4. `subscribeEmbeddedPiSession` رویدادهای pi-agent-core را به جریان `agent` در OpenClaw پل می‌زند:
   - رویدادهای ابزار => `stream: "tool"`
   - دلتاهای دستیار => `stream: "assistant"`
   - رویدادهای چرخه حیات => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` از `waitForAgentRun` استفاده می‌کند:
   - برای **پایان/خطای چرخه حیات** مربوط به `runId` منتظر می‌ماند
   - `{ status: ok|error|timeout, startedAt, endedAt, error? }` را برمی‌گرداند

## صف‌بندی + هم‌زمانی

- اجراها برای هر کلید نشست (مسیر نشست) و به‌صورت اختیاری از طریق یک مسیر سراسری سری می‌شوند.
- این کار از رقابت‌های ابزار/نشست جلوگیری می‌کند و تاریخچه نشست را سازگار نگه می‌دارد.
- کانال‌های پیام‌رسانی می‌توانند حالت‌های صف (collect/steer/followup) را انتخاب کنند که این سامانه مسیر را تغذیه می‌کنند.
  [صف فرمان](/fa/concepts/queue) را ببینید.
- نوشتن‌های رونوشت نیز با قفل نوشتن نشست روی فایل نشست محافظت می‌شوند. قفل
  آگاه از فرایند و مبتنی بر فایل است، بنابراین نویسندگانی را که صف درون‌فرایندی را دور می‌زنند یا از
  فرایند دیگری می‌آیند شناسایی می‌کند. نویسندگان رونوشت نشست تا `session.writeLock.acquireTimeoutMs`
  منتظر می‌مانند و سپس نشست را مشغول گزارش می‌کنند؛ مقدار پیش‌فرض `60000` میلی‌ثانیه است.
- قفل‌های نوشتن نشست به‌طور پیش‌فرض غیر بازدرآیند هستند. اگر یک کمک‌کننده عمدا گرفتن
  همان قفل را در حالی تو در تو می‌کند که یک نویسنده منطقی واحد را حفظ می‌کند، باید صریحا با
  `allowReentrant: true` آن را فعال کند.

## آماده‌سازی نشست + فضای کاری

- فضای کاری حل و ایجاد می‌شود؛ اجراهای sandbox ممکن است به ریشه فضای کاری sandbox هدایت شوند.
- Skills بارگذاری می‌شود (یا از یک عکس فوری بازاستفاده می‌شود) و به env و prompt تزریق می‌شود.
- فایل‌های bootstrap/context حل می‌شوند و به گزارش system prompt تزریق می‌شوند.
- یک قفل نوشتن نشست گرفته می‌شود؛ `SessionManager` پیش از جریان‌دهی باز و آماده می‌شود. هر
  مسیر بازنویسی، Compaction، یا کوتاه‌سازی بعدی رونوشت باید پیش از باز کردن یا
  تغییر دادن فایل رونوشت، همان قفل را بگیرد.

## ساخت prompt + system prompt

- system prompt از prompt پایه OpenClaw، prompt مربوط به Skills، زمینه bootstrap، و overrideهای هر اجرا ساخته می‌شود.
- محدودیت‌های ویژه مدل و توکن‌های ذخیره Compaction اعمال می‌شوند.
- برای آنچه مدل می‌بیند، [system prompt](/fa/concepts/system-prompt) را ببینید.

## نقاط hook (جاهایی که می‌توانید مداخله کنید)

OpenClaw دو سامانه hook دارد:

- **hookهای داخلی** (hookهای Gateway): اسکریپت‌های رویدادمحور برای فرمان‌ها و رویدادهای چرخه حیات.
- **hookهای Plugin**: نقاط توسعه داخل چرخه حیات عامل/ابزار و خط لوله Gateway.

### hookهای داخلی (hookهای Gateway)

- **`agent:bootstrap`**: هنگام ساخت فایل‌های bootstrap، پیش از نهایی شدن system prompt اجرا می‌شود.
  از این برای افزودن/حذف فایل‌های زمینه bootstrap استفاده کنید.
- **hookهای فرمان**: `/new`، `/reset`، `/stop`، و سایر رویدادهای فرمان (سند Hooks را ببینید).

برای راه‌اندازی و نمونه‌ها، [Hooks](/fa/automation/hooks) را ببینید.

### hookهای Plugin (چرخه حیات عامل + gateway)

این‌ها داخل حلقه عامل یا خط لوله gateway اجرا می‌شوند:

- **`before_model_resolve`**: پیش از نشست اجرا می‌شود (بدون `messages`) تا پیش از حل مدل، provider/model را به‌صورت قطعی override کند.
- **`before_prompt_build`**: پس از بارگذاری نشست اجرا می‌شود (با `messages`) تا پیش از ارسال prompt، `prependContext`، `systemPrompt`، `prependSystemContext`، یا `appendSystemContext` را تزریق کند. از `prependContext` برای متن پویای هر نوبت و از فیلدهای system-context برای راهنمایی پایدار که باید در فضای system prompt قرار بگیرد استفاده کنید.
- **`before_agent_start`**: hook سازگاری قدیمی که ممکن است در هرکدام از دو فاز اجرا شود؛ hookهای صریح بالا را ترجیح دهید.
- **`before_agent_reply`**: پس از کنش‌های inline و پیش از فراخوانی LLM اجرا می‌شود، و به یک Plugin امکان می‌دهد نوبت را در اختیار بگیرد و یک پاسخ مصنوعی برگرداند یا نوبت را کاملا ساکت کند.
- **`agent_end`**: فهرست پیام نهایی و فراداده اجرا را پس از تکمیل بررسی کنید.
- **`before_compaction` / `after_compaction`**: چرخه‌های Compaction را مشاهده یا حاشیه‌نویسی کنید.
- **`before_tool_call` / `after_tool_call`**: پارامترها/نتایج ابزار را مداخله کنید.
- **`before_install`**: یافته‌های scan داخلی را بررسی کنید و در صورت نیاز نصب‌های Skills یا Plugin را مسدود کنید.
- **`tool_result_persist`**: نتایج ابزار را به‌صورت همگام، پیش از نوشته شدن در رونوشت نشست متعلق به OpenClaw، تبدیل کنید.
- **`message_received` / `message_sending` / `message_sent`**: hookهای پیام ورودی + خروجی.
- **`session_start` / `session_end`**: مرزهای چرخه حیات نشست.
- **`gateway_start` / `gateway_stop`**: رویدادهای چرخه حیات gateway.

قواعد تصمیم‌گیری hook برای محافظ‌های خروجی/ابزار:

- `before_tool_call`: `{ block: true }` پایانی است و handlerهای با اولویت پایین‌تر را متوقف می‌کند.
- `before_tool_call`: `{ block: false }` بی‌اثر است و block قبلی را پاک نمی‌کند.
- `before_install`: `{ block: true }` پایانی است و handlerهای با اولویت پایین‌تر را متوقف می‌کند.
- `before_install`: `{ block: false }` بی‌اثر است و block قبلی را پاک نمی‌کند.
- `message_sending`: `{ cancel: true }` پایانی است و handlerهای با اولویت پایین‌تر را متوقف می‌کند.
- `message_sending`: `{ cancel: false }` بی‌اثر است و cancel قبلی را پاک نمی‌کند.

برای API مربوط به hook و جزئیات ثبت، [hookهای Plugin](/fa/plugins/hooks) را ببینید.

هارنس‌ها ممکن است این hookها را به‌شکل متفاوتی تطبیق دهند. هارنس app-server در Codex،
hookهای Plugin در OpenClaw را به‌عنوان قرارداد سازگاری برای سطح‌های آینه‌شده مستند نگه می‌دارد،
در حالی که hookهای native در Codex همچنان یک سازوکار جداگانه و سطح پایین‌تر Codex هستند.

## جریان‌دهی + پاسخ‌های جزئی

- دلتاهای دستیار از pi-agent-core جریان می‌یابند و به‌عنوان رویدادهای `assistant` منتشر می‌شوند.
- جریان‌دهی بلوک می‌تواند پاسخ‌های جزئی را یا روی `text_end` یا `message_end` منتشر کند.
- جریان‌دهی استدلال می‌تواند به‌عنوان جریان جداگانه یا به‌عنوان پاسخ‌های بلوکی منتشر شود.
- برای رفتار chunking و پاسخ بلوکی، [جریان‌دهی](/fa/concepts/streaming) را ببینید.

## اجرای ابزار + ابزارهای پیام‌رسانی

- رویدادهای شروع/به‌روزرسانی/پایان ابزار روی جریان `tool` منتشر می‌شوند.
- نتایج ابزار پیش از ثبت/انتشار، از نظر اندازه و payloadهای تصویر پاک‌سازی می‌شوند.
- ارسال‌های ابزار پیام‌رسانی پیگیری می‌شوند تا تاییدیه‌های تکراری دستیار سرکوب شوند.

## شکل‌دهی پاسخ + سرکوب

- payloadهای نهایی از این‌ها مونتاژ می‌شوند:
  - متن دستیار (و استدلال اختیاری)
  - خلاصه‌های inline ابزار (وقتی verbose + مجاز باشد)
  - متن خطای دستیار وقتی مدل خطا می‌دهد
- توکن سکوت دقیق `NO_REPLY` / `no_reply` از payloadهای خروجی
  فیلتر می‌شود.
- تکراری‌های ابزار پیام‌رسانی از فهرست payload نهایی حذف می‌شوند.
- اگر هیچ payload قابل رندر باقی نماند و ابزاری خطا داده باشد، یک پاسخ خطای ابزار fallback منتشر می‌شود
  (مگر اینکه یک ابزار پیام‌رسانی پیش‌تر پاسخ قابل مشاهده برای کاربر فرستاده باشد).

## Compaction + تلاش‌های دوباره

- Auto-compaction رویدادهای جریان `compaction` را منتشر می‌کند و می‌تواند یک تلاش دوباره را فعال کند.
- در تلاش دوباره، bufferهای درون‌حافظه و خلاصه‌های ابزار reset می‌شوند تا از خروجی تکراری جلوگیری شود.
- برای خط لوله Compaction، [Compaction](/fa/concepts/compaction) را ببینید.

## جریان‌های رویداد (امروز)

- `lifecycle`: توسط `subscribeEmbeddedPiSession` منتشر می‌شود (و به‌عنوان fallback توسط `agentCommand`)
- `assistant`: دلتاهای جریانی از pi-agent-core
- `tool`: رویدادهای ابزار جریانی از pi-agent-core

## مدیریت کانال chat

- دلتاهای دستیار در پیام‌های chat با نوع `delta` بافر می‌شوند.
- یک chat با نوع `final` روی **پایان/خطای چرخه حیات** منتشر می‌شود.

## مهلت‌های زمانی

- پیش‌فرض `agent.wait`: ۳۰ ثانیه (فقط انتظار). پارامتر `timeoutMs` آن را override می‌کند.
- runtime عامل: پیش‌فرض `agents.defaults.timeoutSeconds` برابر 172800s (۴۸ ساعت) است؛ در تایمر لغو `runEmbeddedPiAgent` اعمال می‌شود.
- runtime مربوط به Cron: `timeoutSeconds` برای نوبت عامل ایزوله تحت مالکیت cron است. زمان‌بند آن تایمر را هنگام شروع اجرا آغاز می‌کند، اجرای زیرین را در deadline پیکربندی‌شده لغو می‌کند، سپس پیش از ثبت timeout پاک‌سازی محدود انجام می‌دهد تا یک نشست فرزند stale نتواند مسیر را گیر بیندازد.
- عیب‌یابی زنده بودن نشست: با فعال بودن diagnostics، `diagnostics.stuckSessionWarnMs` نشست‌های طولانی `processing` را که هیچ پیشرفت مشاهده‌شده‌ای در پاسخ، ابزار، وضعیت، بلوک، یا ACP ندارند طبقه‌بندی می‌کند. اجراهای تعبیه‌شده فعال، فراخوانی‌های مدل، و فراخوانی‌های ابزار به‌عنوان `session.long_running` گزارش می‌شوند؛ کار فعال بدون پیشرفت اخیر به‌عنوان `session.stalled` گزارش می‌شود؛ `session.stuck` برای bookkeeping نشست stale بدون کار فعال کنار گذاشته شده است. bookkeeping نشست stale مسیر نشست تحت تاثیر را فورا آزاد می‌کند؛ اجراهای تعبیه‌شده stalled فقط پس از `diagnostics.stuckSessionAbortMs` (پیش‌فرض: حداقل ۱۰ دقیقه و ۵ برابر آستانه هشدار) abort-drain می‌شوند تا کار صف‌شده بتواند بدون قطع کردن اجراهایی که صرفا کند هستند از سر گرفته شود. بازیابی outcomeهای ساخت‌یافته requested/completed منتشر می‌کند، و وضعیت diagnostic فقط اگر همان نسل processing همچنان current باشد idle علامت‌گذاری می‌شود. diagnostics تکراری `session.stuck` تا وقتی نشست بدون تغییر بماند back off می‌کند.
- مهلت زمانی بیکاری مدل: OpenClaw وقتی پیش از پنجره idle هیچ response chunk نرسد، درخواست مدل را لغو می‌کند. `models.providers.<id>.timeoutSeconds` این watchdog بیکاری را برای providerهای local/self-hosted کند گسترش می‌دهد؛ در غیر این صورت OpenClaw وقتی `agents.defaults.timeoutSeconds` پیکربندی شده باشد از آن استفاده می‌کند، با سقف پیش‌فرض ۱۲۰ ثانیه. اجراهای فعال‌شده با Cron که timeout صریح مدل یا عامل ندارند، watchdog بیکاری را غیرفعال می‌کنند و به timeout بیرونی cron تکیه می‌کنند.
- مهلت زمانی درخواست HTTP provider: `models.providers.<id>.timeoutSeconds` روی fetchهای HTTP مدل آن provider اعمال می‌شود، شامل connect، headers، body، timeout درخواست SDK، مدیریت لغو guarded-fetch کل، و watchdog بیکاری جریان مدل. پیش از افزایش timeout کل runtime عامل، از این برای providerهای local/self-hosted کند مانند Ollama استفاده کنید.
