یک حلقهٔ عاملیت‌محور، اجرای کامل و «واقعی» یک عامل است: دریافت → گردآوری زمینه → استنتاج مدل →
اجرای ابزار → پاسخ‌های جریانی → پایداری. این مسیر معتبر است که یک پیام را
به کنش‌ها و یک پاسخ نهایی تبدیل می‌کند، در حالی که وضعیت نشست را سازگار نگه می‌دارد.

در OpenClaw، یک حلقه یک اجرای واحد و سری‌سازی‌شده برای هر نشست است که رویدادهای چرخهٔ حیات و جریان را
هم‌زمان با فکر کردن مدل، فراخوانی ابزارها، و جریان‌دادن خروجی منتشر می‌کند. این سند توضیح می‌دهد این حلقهٔ اصیل
چگونه از ابتدا تا انتها سیم‌کشی شده است.

## نقاط ورود

- Gateway RPC: `agent` و `agent.wait`.
- CLI: فرمان `agent`.

## نحوهٔ کارکرد (سطح بالا)

1. RPCِ `agent` پارامترها را اعتبارسنجی می‌کند، نشست را حل می‌کند (sessionKey/sessionId)، فرادادهٔ نشست را پایدار می‌کند، و بلافاصله `{ runId, acceptedAt }` را برمی‌گرداند.
2. `agentCommand` عامل را اجرا می‌کند:
   - مدل + پیش‌فرض‌های تفکر/پرگویی/ردیابی را حل می‌کند
   - تصویر لحظه‌ای Skills را بارگذاری می‌کند
   - `runEmbeddedPiAgent` را فراخوانی می‌کند (زمان اجرای pi-agent-core)
   - اگر حلقهٔ توکار رویدادی منتشر نکند، **پایان/خطای چرخهٔ حیات** را منتشر می‌کند
3. `runEmbeddedPiAgent`:
   - اجراها را از طریق صف‌های هر نشست + صف‌های سراسری سری‌سازی می‌کند
   - مدل + نمایهٔ احراز هویت را حل می‌کند و نشست pi را می‌سازد
   - در رویدادهای pi مشترک می‌شود و دلتاهای دستیار/ابزار را جریان می‌دهد
   - مهلت زمانی را اعمال می‌کند -> اگر از حد بگذرد اجرا را لغو می‌کند
   - payloadها + فرادادهٔ مصرف را برمی‌گرداند
4. `subscribeEmbeddedPiSession` رویدادهای pi-agent-core را به جریان `agent` در OpenClaw پل می‌زند:
   - رویدادهای ابزار => `stream: "tool"`
   - دلتاهای دستیار => `stream: "assistant"`
   - رویدادهای چرخهٔ حیات => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. `agent.wait` از `waitForAgentRun` استفاده می‌کند:
   - برای **پایان/خطای چرخهٔ حیات** مربوط به `runId` منتظر می‌ماند
   - `{ status: ok|error|timeout, startedAt, endedAt, error? }` را برمی‌گرداند

## صف‌بندی + هم‌زمانی

- اجراها برای هر کلید نشست (مسیر نشست) و به‌صورت اختیاری از طریق یک مسیر سراسری سری‌سازی می‌شوند.
- این کار از رقابت‌های ابزار/نشست جلوگیری می‌کند و تاریخچهٔ نشست را سازگار نگه می‌دارد.
- کانال‌های پیام‌رسانی می‌توانند حالت‌های صف (جمع‌آوری/هدایت/پیگیری) را انتخاب کنند که این سامانهٔ مسیر را تغذیه می‌کنند.
  [صف فرمان](/fa/concepts/queue) را ببینید.
- نوشتن transcript نیز با یک قفل نوشتن نشست روی فایل نشست محافظت می‌شود. قفل
  آگاه از پردازه و مبتنی بر فایل است، بنابراین نویسندگانی را که صف درون‌پردازه‌ای را دور می‌زنند یا از
  پردازهٔ دیگری می‌آیند نیز می‌گیرد.
- قفل‌های نوشتن نشست به‌طور پیش‌فرض بازدرون‌شدنی نیستند. اگر یک helper عمداً گرفتن
  همان قفل را با حفظ یک نویسندهٔ منطقی واحد تودرتو کند، باید صریحاً با
  `allowReentrant: true` اعلام کند.

## آماده‌سازی نشست + workspace

- workspace حل و ایجاد می‌شود؛ اجراهای sandboxed ممکن است به ریشهٔ workspace در sandbox هدایت شوند.
- Skills بارگذاری می‌شوند (یا از یک snapshot دوباره استفاده می‌شوند) و در env و prompt تزریق می‌شوند.
- فایل‌های bootstrap/context حل و در گزارش system prompt تزریق می‌شوند.
- یک قفل نوشتن نشست گرفته می‌شود؛ `SessionManager` پیش از جریان‌دهی باز و آماده می‌شود. هر
  مسیر بازنویسی transcript، Compaction، یا کوتاه‌سازی بعدی باید پیش از باز کردن یا
  تغییر فایل transcript همان قفل را بگیرد.

## گردآوری prompt + system prompt

- system prompt از prompt پایهٔ OpenClaw، prompt مربوط به Skills، زمینهٔ bootstrap، و overrideهای هر اجرا ساخته می‌شود.
- محدودیت‌های ویژهٔ مدل و توکن‌های ذخیرهٔ Compaction اعمال می‌شوند.
- برای آنچه مدل می‌بیند، [System prompt](/fa/concepts/system-prompt) را ببینید.

## نقاط hook (جایی که می‌توانید مداخله کنید)

OpenClaw دو سامانهٔ hook دارد:

- **hookهای داخلی** (hookهای Gateway): اسکریپت‌های رویدادمحور برای فرمان‌ها و رویدادهای چرخهٔ حیات.
- **hookهای Plugin**: نقاط توسعه داخل چرخهٔ حیات عامل/ابزار و pipeline گیت‌وی.

### hookهای داخلی (hookهای Gateway)

- **`agent:bootstrap`**: هنگام ساخت فایل‌های bootstrap، پیش از نهایی شدن system prompt اجرا می‌شود.
  از این برای افزودن/حذف فایل‌های زمینهٔ bootstrap استفاده کنید.
- **hookهای فرمان**: `/new`، `/reset`، `/stop`، و سایر رویدادهای فرمان (سند Hooks را ببینید).

برای راه‌اندازی و نمونه‌ها، [Hooks](/fa/automation/hooks) را ببینید.

### hookهای Plugin (چرخهٔ حیات عامل + گیت‌وی)

این‌ها داخل حلقهٔ عامل یا pipeline گیت‌وی اجرا می‌شوند:

- **`before_model_resolve`**: پیش از نشست اجرا می‌شود (بدون `messages`) تا پیش از حل مدل، provider/model را به‌صورت قطعی override کند.
- **`before_prompt_build`**: پس از بارگذاری نشست اجرا می‌شود (با `messages`) تا پیش از ارسال prompt، `prependContext`، `systemPrompt`، `prependSystemContext`، یا `appendSystemContext` را تزریق کند. از `prependContext` برای متن پویای هر نوبت و از فیلدهای system-context برای راهنمایی پایدار استفاده کنید که باید در فضای system prompt قرار بگیرد.
- **`before_agent_start`**: hook سازگاری قدیمی که ممکن است در هر یک از دو فاز اجرا شود؛ hookهای صریح بالا را ترجیح دهید.
- **`before_agent_reply`**: پس از کنش‌های inline و پیش از فراخوانی LLM اجرا می‌شود و به یک Plugin اجازه می‌دهد نوبت را claim کند و یک پاسخ ساختگی برگرداند یا نوبت را کاملاً ساکت کند.
- **`agent_end`**: فهرست پیام نهایی و فرادادهٔ اجرا را پس از تکمیل بررسی می‌کند.
- **`before_compaction` / `after_compaction`**: چرخه‌های Compaction را مشاهده یا حاشیه‌نویسی می‌کند.
- **`before_tool_call` / `after_tool_call`**: پارامترها/نتایج ابزار را رهگیری می‌کند.
- **`before_install`**: یافته‌های scan داخلی را بررسی می‌کند و به‌صورت اختیاری نصب Skills یا Plugin را مسدود می‌کند.
- **`tool_result_persist`**: نتایج ابزار را پیش از نوشته‌شدن در transcript نشست متعلق به OpenClaw، به‌صورت هم‌زمان دگرگون می‌کند.
- **`message_received` / `message_sending` / `message_sent`**: hookهای پیام ورودی + خروجی.
- **`session_start` / `session_end`**: مرزهای چرخهٔ حیات نشست.
- **`gateway_start` / `gateway_stop`**: رویدادهای چرخهٔ حیات گیت‌وی.

قواعد تصمیم‌گیری hook برای محافظ‌های خروجی/ابزار:

- `before_tool_call`: مقدار `{ block: true }` نهایی است و handlerهای با اولویت پایین‌تر را متوقف می‌کند.
- `before_tool_call`: مقدار `{ block: false }` بی‌اثر است و یک block قبلی را پاک نمی‌کند.
- `before_install`: مقدار `{ block: true }` نهایی است و handlerهای با اولویت پایین‌تر را متوقف می‌کند.
- `before_install`: مقدار `{ block: false }` بی‌اثر است و یک block قبلی را پاک نمی‌کند.
- `message_sending`: مقدار `{ cancel: true }` نهایی است و handlerهای با اولویت پایین‌تر را متوقف می‌کند.
- `message_sending`: مقدار `{ cancel: false }` بی‌اثر است و یک cancel قبلی را پاک نمی‌کند.

برای API hook و جزئیات ثبت، [hookهای Plugin](/fa/plugins/hooks) را ببینید.

Harnessها ممکن است این hookها را به‌شکل متفاوتی تطبیق دهند. harness مربوط به Codex app-server
hookهای Plugin در OpenClaw را به‌عنوان قرارداد سازگاری برای سطح‌های mirror مستندشده نگه می‌دارد،
در حالی که hookهای بومی Codex همچنان یک سازوکار جداگانه و سطح پایین‌تر Codex هستند.

## جریان‌دهی + پاسخ‌های جزئی

- دلتاهای دستیار از pi-agent-core جریان داده می‌شوند و به‌عنوان رویدادهای `assistant` منتشر می‌شوند.
- جریان‌دهی بلوک می‌تواند پاسخ‌های جزئی را یا روی `text_end` یا `message_end` منتشر کند.
- جریان‌دهی استدلال می‌تواند به‌عنوان یک جریان جداگانه یا به‌عنوان پاسخ‌های بلوکی منتشر شود.
- برای رفتار قطعه‌بندی و پاسخ بلوکی، [جریان‌دهی](/fa/concepts/streaming) را ببینید.

## اجرای ابزار + ابزارهای پیام‌رسانی

- رویدادهای شروع/به‌روزرسانی/پایان ابزار روی جریان `tool` منتشر می‌شوند.
- نتایج ابزار پیش از ثبت/انتشار، از نظر اندازه و payloadهای تصویری پاک‌سازی می‌شوند.
- ارسال‌های ابزار پیام‌رسانی ردیابی می‌شوند تا تأییدهای تکراری دستیار سرکوب شوند.

## شکل‌دهی پاسخ + سرکوب

- payloadهای نهایی از این‌ها گردآوری می‌شوند:
  - متن دستیار (و استدلال اختیاری)
  - خلاصه‌های inline ابزار (وقتی verbose + مجاز باشد)
  - متن خطای دستیار وقتی مدل خطا می‌دهد
- توکن سکوت دقیق `NO_REPLY` / `no_reply` از payloadهای خروجی
  فیلتر می‌شود.
- تکرارهای ابزار پیام‌رسانی از فهرست payload نهایی حذف می‌شوند.
- اگر هیچ payload قابل‌رندری باقی نماند و یک ابزار خطا داده باشد، یک پاسخ خطای ابزار fallback منتشر می‌شود
  (مگر اینکه یک ابزار پیام‌رسانی از قبل پاسخی قابل مشاهده برای کاربر فرستاده باشد).

## Compaction + تلاش‌های دوباره

- Compaction خودکار رویدادهای جریان `compaction` را منتشر می‌کند و می‌تواند یک تلاش دوباره را تحریک کند.
- در تلاش دوباره، bufferهای درون‌حافظه‌ای و خلاصه‌های ابزار reset می‌شوند تا از خروجی تکراری جلوگیری شود.
- برای pipeline مربوط به Compaction، [Compaction](/fa/concepts/compaction) را ببینید.

## جریان‌های رویداد (امروز)

- `lifecycle`: توسط `subscribeEmbeddedPiSession` منتشر می‌شود (و به‌عنوان fallback توسط `agentCommand`)
- `assistant`: دلتاهای جریانی از pi-agent-core
- `tool`: رویدادهای ابزار جریانی از pi-agent-core

## مدیریت کانال chat

- دلتاهای دستیار در پیام‌های chat از نوع `delta` buffer می‌شوند.
- روی **پایان/خطای چرخهٔ حیات** یک chat `final` منتشر می‌شود.

## مهلت‌های زمانی

- پیش‌فرض `agent.wait`: 30s (فقط انتظار). پارامتر `timeoutMs` آن را override می‌کند.
- زمان اجرای عامل: پیش‌فرض `agents.defaults.timeoutSeconds` برابر 172800s (48 ساعت) است؛ در تایمر abort مربوط به `runEmbeddedPiAgent` اعمال می‌شود.
- زمان اجرای Cron: `timeoutSeconds` مربوط به نوبت عامل ایزوله متعلق به cron است. scheduler آن تایمر را هنگام آغاز اجرا شروع می‌کند، اجرای زیربنایی را در موعد پیکربندی‌شده abort می‌کند، سپس پیش از ثبت timeout پاک‌سازی محدود اجرا می‌کند تا یک نشست فرزند stale نتواند مسیر را گیر بیندازد.
- بازیابی نشست گیرکرده: با فعال بودن diagnostics، `diagnostics.stuckSessionWarnMs` نشست‌های طولانی در حالت `processing` را تشخیص می‌دهد. اجراهای توکار فعال، عملیات پاسخ فعال، و وظایف فعال مسیر نشست به‌طور پیش‌فرض فقط هشدار باقی می‌مانند؛ اگر diagnostics هیچ کار فعالی برای نشست نشان ندهد، watchdog مسیر نشست آسیب‌دیده را آزاد می‌کند تا کارهای startup صف‌شده تخلیه شوند.
- مهلت idle مدل: OpenClaw وقتی پیش از پنجرهٔ idle هیچ قطعهٔ پاسخی نرسد، درخواست مدل را abort می‌کند. `models.providers.<id>.timeoutSeconds` این watchdog idle را برای providerهای local/self-hosted کند گسترش می‌دهد؛ در غیر این صورت OpenClaw وقتی پیکربندی شده باشد از `agents.defaults.timeoutSeconds` استفاده می‌کند، که به‌طور پیش‌فرض تا 120s سقف دارد. اجراهای triggerشده با Cron که timeout صریح مدل یا عامل ندارند، watchdog idle را غیرفعال می‌کنند و به timeout بیرونی cron تکیه می‌کنند.
- مهلت درخواست HTTP provider: مقدار `models.providers.<id>.timeoutSeconds` روی fetchهای HTTP مدل همان provider اعمال می‌شود، از جمله connect، headers، body، مهلت درخواست SDK، مدیریت abort در guarded-fetch کلی، و watchdog idle جریان مدل. پیش از بالا بردن timeout کل زمان اجرای عامل، از این برای providerهای local/self-hosted کند مانند Ollama استفاده کنید.

## جاهایی که امور می‌توانند زود پایان یابند

- قطع اتصال Gateway یا timeout RPC
- timeout مربوط به `agent.wait` (فقط انتظار، عامل را متوقف نمی‌کند)
