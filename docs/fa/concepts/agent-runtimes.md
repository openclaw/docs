یک **زمان اجرای عامل** مؤلفه‌ای است که مالک یک حلقهٔ مدل آماده است: اعلان را دریافت می‌کند، خروجی مدل را پیش می‌برد، فراخوانی‌های ابزار بومی را مدیریت می‌کند و نوبت تکمیل‌شده را به OpenClaw برمی‌گرداند.

اشتباه گرفتن زمان‌های اجرا با ارائه‌دهندگان آسان است، چون هر دو نزدیک پیکربندی مدل دیده می‌شوند. این‌ها لایه‌های متفاوتی هستند:

| لایه          | نمونه‌ها                              | معنی آن                                                           |
| ------------- | ------------------------------------- | ----------------------------------------------------------------- |
| ارائه‌دهنده   | `openai`, `anthropic`, `openai-codex` | OpenClaw چگونه احراز هویت می‌کند، مدل‌ها را کشف می‌کند و ارجاع‌های مدل را نام‌گذاری می‌کند. |
| مدل           | `gpt-5.5`, `claude-opus-4-6`          | مدلی که برای نوبت عامل انتخاب شده است.                           |
| زمان اجرای عامل | `pi`, `codex`, `claude-cli`           | حلقه یا بک‌اند سطح پایینی که نوبت آماده‌شده را اجرا می‌کند.       |
| کانال         | Telegram, Discord, Slack, WhatsApp    | جایی که پیام‌ها وارد OpenClaw می‌شوند و از آن خارج می‌شوند.       |

در کد همچنین واژهٔ **هارنس** را خواهید دید. هارنس پیاده‌سازی‌ای است که یک زمان اجرای عامل را فراهم می‌کند. برای مثال، هارنس Codex همراه‌شده زمان اجرای `codex` را پیاده‌سازی می‌کند. پیکربندی عمومی از `agentRuntime.id` استفاده می‌کند؛ `openclaw doctor --fix` کلیدهای قدیمی‌تر سیاست زمان اجرا را به آن شکل بازنویسی می‌کند.

دو خانوادهٔ زمان اجرا وجود دارد:

- **هارنس‌های تعبیه‌شده** داخل حلقهٔ عامل آماده‌شدهٔ OpenClaw اجرا می‌شوند. امروز این شامل زمان اجرای داخلی `pi` به‌همراه هارنس‌های ثبت‌شدهٔ Plugin مانند `codex` است.
- **بک‌اندهای CLI** یک فرایند CLI محلی را اجرا می‌کنند، در حالی که ارجاع مدل را استاندارد نگه می‌دارند. برای مثال، `anthropic/claude-opus-4-7` با `agentRuntime.id: "claude-cli"` یعنی «مدل Anthropic را انتخاب کن، از طریق Claude CLI اجرا کن.» `claude-cli` شناسهٔ هارنس تعبیه‌شده نیست و نباید به انتخاب AgentHarness داده شود.

## سه چیز با نام Codex

بیشتر سردرگمی از سه سطح متفاوت می‌آید که نام Codex را مشترک دارند:

| سطح                                                 | نام/پیکربندی OpenClaw                 | کاری که انجام می‌دهد                                                                                 |
| ---------------------------------------------------- | ------------------------------------ | --------------------------------------------------------------------------------------------------- |
| مسیر ارائه‌دهندهٔ OAuth مربوط به Codex              | ارجاع‌های مدل `openai-codex/*`       | از OAuth اشتراک ChatGPT/Codex از طریق اجراکنندهٔ عادی OpenClaw PI استفاده می‌کند.                  |
| زمان اجرای بومی app-server مربوط به Codex           | `agentRuntime.id: "codex"`           | نوبت عامل تعبیه‌شده را از طریق هارنس app-server همراه‌شدهٔ Codex اجرا می‌کند.                      |
| آداپتور ACP مربوط به Codex                          | `runtime: "acp"`, `agentId: "codex"` | Codex را از طریق صفحهٔ کنترل خارجی ACP/acpx اجرا می‌کند. فقط وقتی استفاده کنید که ACP/acpx صراحتا خواسته شده باشد. |
| مجموعهٔ فرمان کنترل چت بومی Codex                   | `/codex ...`                         | نخ‌های app-server مربوط به Codex را از چت متصل، ازسرگرفته، هدایت، متوقف و بررسی می‌کند.            |
| مسیر OpenAI Platform API برای مدل‌های سبک GPT/Codex | ارجاع‌های مدل `openai/*`             | از احراز هویت با کلید API مربوط به OpenAI استفاده می‌کند، مگر اینکه بازنویسی زمان اجرا، مانند `runtime: "codex"`، نوبت را اجرا کند. |

این سطوح عمدا مستقل هستند. فعال کردن Plugin `codex` قابلیت‌های بومی app-server را در دسترس می‌گذارد؛ `openai-codex/*` را به `openai/*` بازنویسی نمی‌کند، نشست‌های موجود را تغییر نمی‌دهد و ACP را پیش‌فرض Codex نمی‌کند. انتخاب `openai-codex/*` یعنی «از مسیر ارائه‌دهندهٔ OAuth مربوط به Codex استفاده کن»، مگر اینکه جداگانه یک زمان اجرا را اجبار کنید.

راه‌اندازی رایج Codex از ارائه‌دهندهٔ `openai` با زمان اجرای `codex` استفاده می‌کند:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

این یعنی OpenClaw یک ارجاع مدل OpenAI را انتخاب می‌کند، سپس از زمان اجرای app-server مربوط به Codex می‌خواهد نوبت عامل تعبیه‌شده را اجرا کند. این به این معنا نیست که کانال، کاتالوگ ارائه‌دهندهٔ مدل، یا ذخیرهٔ نشست OpenClaw به Codex تبدیل می‌شود.

وقتی Plugin همراه‌شدهٔ `codex` فعال است، کنترل Codex با زبان طبیعی باید به‌جای ACP از سطح فرمان بومی `/codex` (`/codex bind`, `/codex threads`, `/codex resume`, `/codex steer`, `/codex stop`) استفاده کند. برای Codex فقط زمانی از ACP استفاده کنید که کاربر صراحتا ACP/acpx را بخواهد یا مسیر آداپتور ACP را آزمایش کند. Claude Code، Gemini CLI، OpenCode، Cursor و هارنس‌های خارجی مشابه همچنان از ACP استفاده می‌کنند.

این درخت تصمیم روبه‌روی عامل است:

1. اگر کاربر **اتصال/کنترل/نخ/ازسرگیری/هدایت/توقف Codex** را بخواهد، وقتی Plugin همراه‌شدهٔ `codex` فعال است از سطح فرمان بومی `/codex` استفاده کنید.
2. اگر کاربر **Codex به‌عنوان زمان اجرای تعبیه‌شده** را بخواهد، از `openai/<model>` با `agentRuntime.id: "codex"` استفاده کنید.
3. اگر کاربر **احراز هویت OAuth/اشتراک Codex روی اجراکنندهٔ عادی OpenClaw** را بخواهد، از `openai-codex/<model>` استفاده کنید و زمان اجرا را PI نگه دارید.
4. اگر کاربر صراحتا **ACP**، **acpx**، یا **آداپتور ACP مربوط به Codex** را گفت، از ACP با `runtime: "acp"` و `agentId: "codex"` استفاده کنید.
5. اگر درخواست برای **Claude Code، Gemini CLI، OpenCode، Cursor، Droid یا هارنس خارجی دیگری** است، از ACP/acpx استفاده کنید، نه زمان اجرای بومی زیرعامل.

| منظورتان این است...                    | استفاده کنید...                              |
| --------------------------------------- | -------------------------------------------- |
| کنترل چت/نخ app-server مربوط به Codex  | `/codex ...` از Plugin همراه‌شدهٔ `codex`    |
| زمان اجرای عامل تعبیه‌شدهٔ app-server مربوط به Codex | `agentRuntime.id: "codex"`                   |
| OAuth مربوط به OpenAI Codex روی اجراکنندهٔ PI | ارجاع‌های مدل `openai-codex/*`               |
| Claude Code یا هارنس خارجی دیگر        | ACP/acpx                                     |

برای جداسازی پیشوند خانوادهٔ OpenAI، [OpenAI](/fa/providers/openai) و [ارائه‌دهندگان مدل](/fa/concepts/model-providers) را ببینید. برای قرارداد پشتیبانی زمان اجرای Codex، [هارنس Codex](/fa/plugins/codex-harness#v1-support-contract) را ببینید.

## مالکیت زمان اجرا

زمان‌های اجرای مختلف، مقدار متفاوتی از حلقه را مالک هستند.

| سطح                         | OpenClaw PI تعبیه‌شده                  | app-server مربوط به Codex                                                     |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| مالک حلقهٔ مدل              | OpenClaw از طریق اجراکنندهٔ PI تعبیه‌شده | app-server مربوط به Codex                                                    |
| وضعیت استاندارد نخ          | رونوشت OpenClaw                         | نخ Codex، به‌همراه آینهٔ رونوشت OpenClaw                                    |
| ابزارهای پویای OpenClaw     | حلقهٔ ابزار بومی OpenClaw               | از طریق آداپتور Codex پل زده می‌شود                                          |
| ابزارهای بومی پوسته و فایل  | مسیر PI/OpenClaw                        | ابزارهای بومی Codex، در جاهایی که پشتیبانی می‌شود از طریق هوک‌های بومی پل زده می‌شوند |
| موتور زمینه                 | سرهم‌بندی زمینهٔ بومی OpenClaw          | OpenClaw زمینهٔ سرهم‌شده را به نوبت Codex می‌فرستد                           |
| Compaction                  | OpenClaw یا موتور زمینهٔ انتخاب‌شده     | Compaction بومی Codex، با اعلان‌های OpenClaw و نگهداری آینه                  |
| تحویل کانال                 | OpenClaw                                | OpenClaw                                                                    |

این جداسازی مالکیت، قاعدهٔ طراحی اصلی است:

- اگر OpenClaw مالک سطح باشد، OpenClaw می‌تواند رفتار عادی هوک Plugin را فراهم کند.
- اگر زمان اجرای بومی مالک سطح باشد، OpenClaw به رویدادهای زمان اجرا یا هوک‌های بومی نیاز دارد.
- اگر زمان اجرای بومی مالک وضعیت استاندارد نخ باشد، OpenClaw باید زمینه را آینه و تصویر کند، نه اینکه داخلی‌های پشتیبانی‌نشده را بازنویسی کند.

## انتخاب زمان اجرا

OpenClaw پس از حل ارائه‌دهنده و مدل، یک زمان اجرای تعبیه‌شده را انتخاب می‌کند:

1. زمان اجرای ثبت‌شدهٔ یک نشست برنده است. تغییرات پیکربندی یک رونوشت موجود را به‌صورت داغ به سامانهٔ نخ بومی دیگری سوییچ نمی‌کند.
2. `OPENCLAW_AGENT_RUNTIME=<id>` آن زمان اجرا را برای نشست‌های جدید یا بازنشانی‌شده اجبار می‌کند.
3. `agents.defaults.agentRuntime.id` یا `agents.list[].agentRuntime.id` می‌تواند `auto`، `pi`، یک شناسهٔ هارنس تعبیه‌شدهٔ ثبت‌شده مانند `codex`، یا یک نام مستعار بک‌اند CLI پشتیبانی‌شده مانند `claude-cli` را تنظیم کند.
4. در حالت `auto`، زمان‌های اجرای ثبت‌شدهٔ Plugin می‌توانند جفت‌های ارائه‌دهنده/مدلی را که پشتیبانی می‌کنند ادعا کنند.
5. اگر هیچ زمان اجرایی در حالت `auto` یک نوبت را ادعا نکند و `fallback: "pi"` تنظیم شده باشد (پیش‌فرض)، OpenClaw از PI به‌عنوان جایگزین سازگاری استفاده می‌کند. `fallback: "none"` را تنظیم کنید تا انتخاب‌های تطبیق‌نیافته در حالت `auto` به‌جای آن شکست بخورند.

زمان‌های اجرای صریح Plugin به‌طور پیش‌فرض بسته و ناموفق می‌شوند. برای مثال، `runtime: "codex"` یعنی Codex یا یک خطای انتخاب روشن، مگر اینکه در همان دامنهٔ بازنویسی `fallback: "pi"` را تنظیم کنید. بازنویسی زمان اجرا یک تنظیم جایگزین گسترده‌تر را به ارث نمی‌برد، پس `runtime: "codex"` در سطح عامل فقط چون پیش‌فرض‌ها از `fallback: "pi"` استفاده کرده‌اند، بی‌صدا به PI برگردانده نمی‌شود.

نام‌های مستعار بک‌اند CLI با شناسه‌های هارنس تعبیه‌شده متفاوت هستند. شکل ترجیحی Claude CLI این است:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

ارجاع‌های قدیمی مانند `claude-cli/claude-opus-4-7` برای سازگاری همچنان پشتیبانی می‌شوند، اما پیکربندی جدید باید ارائه‌دهنده/مدل را استاندارد نگه دارد و بک‌اند اجرا را در `agentRuntime.id` قرار دهد.

حالت `auto` عمدا محافظه‌کار است. زمان‌های اجرای Plugin می‌توانند جفت‌های ارائه‌دهنده/مدلی را که می‌شناسند ادعا کنند، اما Plugin مربوط به Codex ارائه‌دهندهٔ `openai-codex` را در حالت `auto` ادعا نمی‌کند. این کار `openai-codex/*` را به‌عنوان مسیر صریح OAuth مربوط به PI Codex نگه می‌دارد و از انتقال بی‌صدای پیکربندی‌های احراز هویت اشتراکی به هارنس بومی app-server جلوگیری می‌کند.

اگر `openclaw doctor` هشدار داد که Plugin `codex` فعال است در حالی که `openai-codex/*` هنوز از طریق PI مسیریابی می‌شود، آن را تشخیص در نظر بگیرید، نه مهاجرت. وقتی OAuth مربوط به PI Codex همان چیزی است که می‌خواهید، پیکربندی را بدون تغییر نگه دارید. فقط وقتی اجرای بومی app-server مربوط به Codex را می‌خواهید، به `openai/<model>` به‌همراه `agentRuntime.id: "codex"` سوییچ کنید.

## قرارداد سازگاری

وقتی یک زمان اجرا PI نیست، باید مستند کند از کدام سطوح OpenClaw پشتیبانی می‌کند. برای مستندات زمان اجرا از این شکل استفاده کنید:

| پرسش                                  | چرا مهم است                                                                                       |
| -------------------------------------- | ------------------------------------------------------------------------------------------------- |
| چه کسی مالک حلقهٔ مدل است؟            | تعیین می‌کند تلاش مجدد، ادامهٔ ابزار و تصمیم‌های پاسخ نهایی کجا رخ می‌دهند.                       |
| چه کسی مالک تاریخچهٔ استاندارد نخ است؟ | تعیین می‌کند آیا OpenClaw می‌تواند تاریخچه را ویرایش کند یا فقط آن را آینه کند.                    |
| آیا ابزارهای پویای OpenClaw کار می‌کنند؟ | پیام‌رسانی، نشست‌ها، cron و ابزارهای تحت مالکیت OpenClaw به این وابسته‌اند.                       |
| آیا هوک‌های ابزار پویا کار می‌کنند؟   | Plugin ها انتظار `before_tool_call`، `after_tool_call` و میان‌افزار پیرامون ابزارهای تحت مالکیت OpenClaw را دارند. |
| آیا هوک‌های ابزار بومی کار می‌کنند؟   | پوسته، patch و ابزارهای تحت مالکیت زمان اجرا برای سیاست و مشاهده به پشتیبانی هوک بومی نیاز دارند. |
| آیا چرخهٔ عمر موتور زمینه اجرا می‌شود؟ | Plugin های حافظه و زمینه به چرخهٔ عمر assemble، ingest، after-turn و compaction وابسته‌اند.       |
| چه داده‌های Compaction نمایان می‌شود؟ | برخی Plugin ها فقط به اعلان‌ها نیاز دارند، در حالی که برخی دیگر به فرادادهٔ نگه‌داشته‌شده/حذف‌شده نیاز دارند. |
| چه چیزی عمدا پشتیبانی نمی‌شود؟        | کاربران نباید در جایی که زمان اجرای بومی وضعیت بیشتری را مالک است، هم‌ارزی با PI را فرض کنند.     |

قرارداد پشتیبانی زمان اجرای Codex در [هارنس Codex](/fa/plugins/codex-harness#v1-support-contract) مستند شده است.

## برچسب‌های وضعیت

خروجی وضعیت ممکن است هر دو برچسب `Execution` و `Runtime` را نشان دهد. آن‌ها را به‌عنوان
عیب‌یابی بخوانید، نه به‌عنوان نام ارائه‌دهنده‌ها.

- یک ارجاع مدل مانند `openai/gpt-5.5` ارائه‌دهنده/مدل انتخاب‌شده را به شما نشان می‌دهد.
- یک شناسه زمان اجرا مانند `codex` به شما نشان می‌دهد کدام حلقه در حال اجرای نوبت است.
- یک برچسب کانال مانند Telegram یا Discord به شما نشان می‌دهد گفتگو کجا در جریان است.

اگر پس از تغییر پیکربندی زمان اجرا، یک نشست همچنان PI را نشان می‌دهد، یک نشست جدید
با `/new` شروع کنید یا نشست فعلی را با `/reset` پاک کنید. نشست‌های موجود زمان اجرای
ثبت‌شده خود را نگه می‌دارند تا یک رونوشت گفتگو از طریق دو سامانه بومی ناسازگار نشست
دوباره پخش نشود.
