یک **زمان اجرای عامل** مؤلفه‌ای است که مالک یک حلقهٔ مدل آماده است: فرمان را دریافت می‌کند، خروجی مدل را پیش می‌برد، فراخوانی‌های ابزار بومی را مدیریت می‌کند، و نوبت کامل‌شده را به OpenClaw برمی‌گرداند.

زمان‌های اجرا به‌راحتی با ارائه‌دهنده‌ها اشتباه گرفته می‌شوند، چون هر دو نزدیک پیکربندی مدل ظاهر می‌شوند. آن‌ها لایه‌های متفاوتی هستند:

| لایه | نمونه‌ها | معنی آن |
| ------------- | ------------------------------------- | ------------------------------------------------------------------- |
| ارائه‌دهنده | `openai`, `anthropic`, `openai-codex` | اینکه OpenClaw چگونه احراز هویت می‌کند، مدل‌ها را کشف می‌کند، و ارجاع‌های مدل را نام‌گذاری می‌کند. |
| مدل | `gpt-5.5`, `claude-opus-4-6` | مدلی که برای نوبت عامل انتخاب شده است. |
| زمان اجرای عامل | `pi`, `codex`, `claude-cli` | حلقه یا بک‌اند سطح‌پایینی که نوبت آماده‌شده را اجرا می‌کند. |
| کانال | Telegram, Discord, Slack, WhatsApp | جایی که پیام‌ها وارد OpenClaw می‌شوند و از آن خارج می‌شوند. |

در کد، واژهٔ **harness** را نیز خواهید دید. هارنس پیاده‌سازی‌ای است که یک زمان اجرای عامل را فراهم می‌کند. برای مثال، هارنس Codex همراه محصول، زمان اجرای `codex` را پیاده‌سازی می‌کند. پیکربندی عمومی از `agentRuntime.id` استفاده می‌کند؛ `openclaw doctor --fix` کلیدهای قدیمی‌تر سیاست زمان اجرا را به همین شکل بازنویسی می‌کند.

دو خانوادهٔ زمان اجرا وجود دارد:

- **هارنس‌های تعبیه‌شده** داخل حلقهٔ عامل آمادهٔ OpenClaw اجرا می‌شوند. امروز این شامل زمان اجرای داخلی `pi` به‌همراه هارنس‌های Plugin ثبت‌شده مانند `codex` است.
- **بک‌اندهای CLI** یک فرایند CLI محلی را اجرا می‌کنند و هم‌زمان ارجاع مدل را canonical نگه می‌دارند. برای مثال، `anthropic/claude-opus-4-7` همراه با `agentRuntime.id: "claude-cli"` یعنی «مدل Anthropic را انتخاب کن، از طریق Claude CLI اجرا کن.» `claude-cli` شناسهٔ هارنس تعبیه‌شده نیست و نباید به انتخاب AgentHarness پاس داده شود.

## سطوح Codex

بیشتر سردرگمی از چند سطح متفاوت می‌آید که نام Codex را مشترکاً استفاده می‌کنند:

| سطح | نام/پیکربندی OpenClaw | کارکرد |
| ---------------------------------------------------- | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| زمان اجرای بومی app-server در Codex | `openai/*` به‌همراه `agentRuntime.id: "codex"` | نوبت عامل تعبیه‌شده را از طریق app-server در Codex اجرا می‌کند. این همان راه‌اندازی معمول اشتراک ChatGPT/Codex است. |
| مسیر ارائه‌دهندهٔ OAuth برای Codex | ارجاع‌های مدل `openai-codex/*` | از OAuth اشتراک ChatGPT/Codex از طریق اجراکنندهٔ عادی OpenClaw PI استفاده می‌کند. |
| آداپتور Codex ACP | `runtime: "acp"`, `agentId: "codex"` | Codex را از طریق صفحهٔ کنترل خارجی ACP/acpx اجرا می‌کند. فقط وقتی استفاده کنید که ACP/acpx صراحتاً خواسته شده باشد. |
| مجموعه دستورهای بومی کنترل چت Codex | `/codex ...` | رشته‌های app-server در Codex را از چت متصل می‌کند، از سر می‌گیرد، هدایت می‌کند، متوقف می‌کند، و بررسی می‌کند. |
| مسیر OpenAI Platform API برای مدل‌های سبک GPT/Codex | ارجاع‌های مدل `openai/*` | از احراز هویت با کلید API مربوط به OpenAI استفاده می‌کند، مگر اینکه یک بازنویسی زمان اجرا، مانند `agentRuntime.id: "codex"`، نوبت را اجرا کند. |

این سطوح عمداً مستقل هستند. فعال کردن Plugin با نام `codex` ویژگی‌های بومی app-server را در دسترس قرار می‌دهد؛ `openai-codex/*` را به `openai/*` بازنویسی نمی‌کند، نشست‌های موجود را تغییر نمی‌دهد، و ACP را پیش‌فرض Codex نمی‌کند. انتخاب `openai-codex/*` یعنی «از مسیر ارائه‌دهندهٔ OAuth مربوط به Codex استفاده کن»، مگر اینکه جداگانه یک زمان اجرا را به‌اجبار تعیین کنید.

راه‌اندازی رایج اشتراک ChatGPT/Codex برای احراز هویت از OAuth مربوط به Codex استفاده می‌کند، اما ارجاع مدل را به‌شکل `openai/*` نگه می‌دارد و زمان اجرای `codex` را انتخاب می‌کند:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

یعنی OpenClaw یک ارجاع مدل OpenAI را انتخاب می‌کند، سپس از زمان اجرای app-server در Codex می‌خواهد نوبت عامل تعبیه‌شده را اجرا کند. این به معنی «استفاده از صورت‌حساب API» نیست، و به این معنی هم نیست که کانال، کاتالوگ ارائه‌دهندهٔ مدل، یا ذخیره‌گاه نشست OpenClaw به Codex تبدیل می‌شود.

وقتی Plugin همراه محصول با نام `codex` فعال است، کنترل Codex با زبان طبیعی باید به‌جای ACP از سطح دستور بومی `/codex` استفاده کند (`/codex bind`, `/codex threads`, `/codex resume`, `/codex steer`, `/codex stop`). برای Codex فقط وقتی از ACP استفاده کنید که کاربر صراحتاً ACP/acpx را بخواهد یا در حال آزمودن مسیر آداپتور ACP باشد. Claude Code، Gemini CLI، OpenCode، Cursor، و هارنس‌های خارجی مشابه همچنان از ACP استفاده می‌کنند.

این درخت تصمیم مخصوص عامل است:

1. اگر کاربر **اتصال/کنترل/رشته/ازسرگیری/هدایت/توقف Codex** را می‌خواهد، وقتی Plugin همراه محصول با نام `codex` فعال است از سطح دستور بومی `/codex` استفاده کنید.
2. اگر کاربر **Codex به‌عنوان زمان اجرای تعبیه‌شده** را می‌خواهد یا تجربهٔ عادی عامل Codex مبتنی بر اشتراک را می‌خواهد، از `openai/<model>` با `agentRuntime.id: "codex"` استفاده کنید.
3. اگر کاربر **احراز هویت OAuth/اشتراک Codex روی اجراکنندهٔ عادی OpenClaw** را می‌خواهد، از `openai-codex/<model>` استفاده کنید و زمان اجرا را PI نگه دارید.
4. اگر کاربر صراحتاً **ACP**، **acpx**، یا **آداپتور Codex ACP** را می‌گوید، از ACP با `runtime: "acp"` و `agentId: "codex"` استفاده کنید.
5. اگر درخواست برای **Claude Code، Gemini CLI، OpenCode، Cursor، Droid، یا هارنس خارجی دیگری** است، از ACP/acpx استفاده کنید، نه زمان اجرای بومی زیرعامل.

| منظورتان این است... | استفاده کنید از... |
| --------------------------------------- | -------------------------------------------- |
| کنترل چت/رشتهٔ app-server در Codex | `/codex ...` از Plugin همراه محصول با نام `codex` |
| زمان اجرای عامل تعبیه‌شدهٔ app-server در Codex | `agentRuntime.id: "codex"` |
| OAuth مربوط به OpenAI Codex روی اجراکنندهٔ PI | ارجاع‌های مدل `openai-codex/*` |
| Claude Code یا هارنس خارجی دیگر | ACP/acpx |

برای تفکیک پیشوند خانوادهٔ OpenAI، [OpenAI](/fa/providers/openai) و [ارائه‌دهنده‌های مدل](/fa/concepts/model-providers) را ببینید. برای قرارداد پشتیبانی زمان اجرای Codex، [هارنس Codex](/fa/plugins/codex-harness#v1-support-contract) را ببینید.

## مالکیت زمان اجرا

زمان‌های اجرای مختلف، بخش‌های متفاوتی از حلقه را مالکیت می‌کنند.

| سطح | تعبیه‌شدهٔ OpenClaw PI | app-server در Codex |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| مالک حلقهٔ مدل | OpenClaw از طریق اجراکنندهٔ تعبیه‌شدهٔ PI | app-server در Codex |
| وضعیت canonical رشته | رونوشت OpenClaw | رشتهٔ Codex، به‌همراه آینهٔ رونوشت OpenClaw |
| ابزارهای پویای OpenClaw | حلقهٔ ابزار بومی OpenClaw | از طریق آداپتور Codex پل زده می‌شود |
| ابزارهای بومی پوسته و فایل | مسیر PI/OpenClaw | ابزارهای بومی Codex، که در صورت پشتیبانی از طریق هوک‌های بومی پل زده می‌شوند |
| موتور زمینه | مونتاژ زمینهٔ بومی OpenClaw | OpenClaw زمینهٔ اسمبل‌شده را به نوبت Codex می‌تاباند |
| Compaction | OpenClaw یا موتور زمینهٔ انتخاب‌شده | Compaction بومی Codex، با اعلان‌های OpenClaw و نگهداری آینه |
| تحویل کانال | OpenClaw | OpenClaw |

این تفکیک مالکیت، قاعدهٔ اصلی طراحی است:

- اگر OpenClaw مالک سطح باشد، OpenClaw می‌تواند رفتار عادی هوک Plugin را فراهم کند.
- اگر زمان اجرای بومی مالک سطح باشد، OpenClaw به رویدادهای زمان اجرا یا هوک‌های بومی نیاز دارد.
- اگر زمان اجرای بومی مالک وضعیت canonical رشته باشد، OpenClaw باید زمینه را آینه و project کند، نه اینکه internals پشتیبانی‌نشده را بازنویسی کند.

## انتخاب زمان اجرا

OpenClaw پس از حل ارائه‌دهنده و مدل، یک زمان اجرای تعبیه‌شده را انتخاب می‌کند:

1. زمان اجرای ثبت‌شدهٔ یک نشست برنده است. تغییرات پیکربندی، یک رونوشت موجود را به‌صورت زنده به سیستم رشتهٔ بومی دیگری منتقل نمی‌کند.
2. `OPENCLAW_AGENT_RUNTIME=<id>` آن زمان اجرا را برای نشست‌های جدید یا بازنشانی‌شده اجباری می‌کند.
3. `agents.defaults.agentRuntime.id` یا `agents.list[].agentRuntime.id` می‌تواند `auto`، `pi`، یک شناسهٔ هارنس تعبیه‌شدهٔ ثبت‌شده مانند `codex`، یا یک نام مستعار بک‌اند CLI پشتیبانی‌شده مانند `claude-cli` را تنظیم کند.
4. در حالت `auto`، زمان‌های اجرای Plugin ثبت‌شده می‌توانند جفت‌های ارائه‌دهنده/مدل پشتیبانی‌شده را claim کنند.
5. اگر هیچ زمان اجرایی در حالت `auto` یک نوبت را claim نکند، OpenClaw از PI به‌عنوان زمان اجرای سازگاری استفاده می‌کند. وقتی اجرا باید سخت‌گیرانه باشد، از شناسهٔ زمان اجرای صریح استفاده کنید.

زمان‌های اجرای صریح Plugin fail closed می‌شوند. برای مثال، `agentRuntime.id: "codex"` یعنی Codex یا یک خطای روشن انتخاب/زمان اجرا؛ هرگز بی‌صدا دوباره به PI مسیردهی نمی‌شود.

نام‌های مستعار بک‌اند CLI با شناسه‌های هارنس تعبیه‌شده متفاوت هستند. شکل ترجیحی Claude CLI این است:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

ارجاع‌های قدیمی مانند `claude-cli/claude-opus-4-7` همچنان برای سازگاری پشتیبانی می‌شوند، اما پیکربندی جدید باید ارائه‌دهنده/مدل را canonical نگه دارد و بک‌اند اجرا را در `agentRuntime.id` قرار دهد.

حالت `auto` عمداً محافظه‌کارانه است. زمان‌های اجرای Plugin می‌توانند جفت‌های ارائه‌دهنده/مدلی را که می‌شناسند claim کنند، اما Plugin مربوط به Codex ارائه‌دهندهٔ `openai-codex` را در حالت `auto` claim نمی‌کند. این کار `openai-codex/*` را به‌عنوان مسیر صریح PI برای OAuth مربوط به Codex نگه می‌دارد و از جابه‌جایی بی‌صدای پیکربندی‌های احراز هویت اشتراکی روی هارنس بومی app-server جلوگیری می‌کند.

اگر `openclaw doctor` هشدار دهد که Plugin مربوط به `codex` فعال است در حالی که `openai-codex/*` همچنان از طریق PI مسیردهی می‌شود، آن را یک تشخیص در نظر بگیرید، نه مهاجرت. وقتی PI Codex OAuth همان چیزی است که می‌خواهید، پیکربندی را بدون تغییر نگه دارید. فقط وقتی به اجرای بومی app-server در Codex نیاز دارید، به `openai/<model>` به‌همراه `agentRuntime.id: "codex"` تغییر دهید.

## قرارداد سازگاری

وقتی یک زمان اجرا PI نیست، باید مستند کند از کدام سطوح OpenClaw پشتیبانی می‌کند. برای مستندات زمان اجرا از این شکل استفاده کنید:

| پرسش | چرا مهم است |
| -------------------------------------- | ------------------------------------------------------------------------------------------------- |
| چه کسی مالک حلقهٔ مدل است؟ | تعیین می‌کند تصمیم‌های مربوط به تلاش دوباره، ادامهٔ ابزار، و پاسخ نهایی کجا رخ می‌دهند. |
| چه کسی مالک تاریخچهٔ canonical رشته است؟ | تعیین می‌کند آیا OpenClaw می‌تواند تاریخچه را ویرایش کند یا فقط آن را آینه کند. |
| آیا ابزارهای پویای OpenClaw کار می‌کنند؟ | پیام‌رسانی، نشست‌ها، cron، و ابزارهای تحت مالکیت OpenClaw به این متکی هستند. |
| آیا هوک‌های ابزار پویا کار می‌کنند؟ | Pluginها انتظار `before_tool_call`، `after_tool_call`، و میان‌افزار پیرامون ابزارهای تحت مالکیت OpenClaw را دارند. |
| آیا هوک‌های ابزار بومی کار می‌کنند؟ | پوسته، patch، و ابزارهای تحت مالکیت زمان اجرا برای سیاست و مشاهده‌پذیری به پشتیبانی هوک بومی نیاز دارند. |
| آیا چرخهٔ حیات موتور زمینه اجرا می‌شود؟ | Pluginهای حافظه و زمینه به چرخهٔ حیات assemble، ingest، after-turn، و compaction وابسته‌اند. |
| چه داده‌های compaction آشکار می‌شود؟ | بعضی Pluginها فقط به اعلان‌ها نیاز دارند، در حالی که برخی دیگر به فرادادهٔ نگه‌داشته‌شده/حذف‌شده نیاز دارند. |
| چه چیزی عمداً پشتیبانی نمی‌شود؟ | کاربران نباید در جایی که زمان اجرای بومی وضعیت بیشتری را مالک است، هم‌ارزی با PI را فرض کنند. |

قرارداد پشتیبانی زمان اجرای Codex در [هارنس Codex](/fa/plugins/codex-harness#v1-support-contract) مستند شده است.

## برچسب‌های وضعیت

خروجی وضعیت ممکن است هر دو برچسب `Execution` و `Runtime` را نشان دهد. آن‌ها را به‌عنوان
عیب‌یابی بخوانید، نه به‌عنوان نام‌های ارائه‌دهنده.

- یک ارجاع مدل مانند `openai/gpt-5.5` ارائه‌دهنده/مدل انتخاب‌شده را به شما نشان می‌دهد.
- یک شناسه زمان اجرا مانند `codex` به شما نشان می‌دهد کدام حلقه در حال اجرای نوبت است.
- یک برچسب کانال مانند Telegram یا Discord به شما نشان می‌دهد گفتگو کجا در حال انجام است.

اگر یک نشست پس از تغییر پیکربندی زمان اجرا همچنان PI را نشان می‌دهد، با `/new` یک نشست تازه
شروع کنید یا نشست فعلی را با `/reset` پاک کنید. نشست‌های موجود زمان اجرای ثبت‌شده خود را نگه می‌دارند
تا رونوشت از میان دو سامانه نشست بومی ناسازگار دوباره پخش نشود.
