یک **زمان اجرای عامل** مؤلفه‌ای است که مالک یک حلقه مدل آماده‌شده است: پرامپت را
دریافت می‌کند، خروجی مدل را پیش می‌برد، فراخوانی‌های ابزار بومی را مدیریت می‌کند، و نوبت
کامل‌شده را به OpenClaw برمی‌گرداند.

اشتباه گرفتن زمان‌های اجرا با ارائه‌دهنده‌ها آسان است، چون هر دو نزدیک به پیکربندی مدل
نمایان می‌شوند. آن‌ها لایه‌های متفاوتی هستند:

| لایه          | نمونه‌ها                              | معنی آن                                                            |
| ------------- | ------------------------------------- | ------------------------------------------------------------------- |
| ارائه‌دهنده   | `openai`, `anthropic`, `openai-codex` | اینکه OpenClaw چگونه احراز هویت می‌کند، مدل‌ها را کشف می‌کند، و ارجاع‌های مدل را نام‌گذاری می‌کند. |
| مدل           | `gpt-5.5`, `claude-opus-4-6`          | مدلی که برای نوبت عامل انتخاب شده است.                              |
| زمان اجرای عامل | `pi`, `codex`, `claude-cli`           | حلقه یا بک‌اند سطح پایینی که نوبت آماده‌شده را اجرا می‌کند.      |
| کانال         | Telegram, Discord, Slack, WhatsApp    | جایی که پیام‌ها وارد OpenClaw و از آن خارج می‌شوند.                            |

در کد همچنین واژه **هارنس** را خواهید دید. هارنس پیاده‌سازی‌ای است که یک زمان اجرای
عامل را فراهم می‌کند. برای مثال، هارنس همراه Codex زمان اجرای `codex` را پیاده‌سازی
می‌کند. پیکربندی عمومی از `agentRuntime.id` استفاده می‌کند؛ `openclaw
doctor --fix` کلیدهای قدیمی‌تر سیاست زمان اجرا را به آن شکل بازنویسی می‌کند.

دو خانواده زمان اجرا وجود دارد:

- **هارنس‌های تعبیه‌شده** داخل حلقه عامل آماده‌شده OpenClaw اجرا می‌شوند. امروز این
  شامل زمان اجرای داخلی `pi` به‌علاوه هارنس‌های Plugin ثبت‌شده مانند
  `codex` است.
- **بک‌اندهای CLI** یک فرایند CLI محلی را اجرا می‌کنند و در عین حال ارجاع مدل را
  کانونی نگه می‌دارند. برای مثال، `anthropic/claude-opus-4-7` با
  `agentRuntime.id: "claude-cli"` یعنی «مدل Anthropic را انتخاب کن، از طریق
  Claude CLI اجرا کن.» `claude-cli` یک شناسه هارنس تعبیه‌شده نیست و نباید
  به انتخاب AgentHarness پاس داده شود.

## سطوح Codex

بیشتر سردرگمی از چند سطح متفاوت می‌آید که نام Codex را مشترکاً دارند:

| سطح                                                  | نام/پیکربندی OpenClaw                       | کارکرد آن                                                                                               |
| ---------------------------------------------------- | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| زمان اجرای بومی سرور برنامه Codex                    | `openai/*` به‌علاوه `agentRuntime.id: "codex"` | نوبت عامل تعبیه‌شده را از طریق سرور برنامه Codex اجرا می‌کند. این راه‌اندازی معمول اشتراک ChatGPT/Codex است. |
| مسیر ارائه‌دهنده OAuth در Codex                      | ارجاع‌های مدل `openai-codex/*`                | از OAuth اشتراک ChatGPT/Codex از طریق اجراکننده عادی PI در OpenClaw استفاده می‌کند.                               |
| آداپتور ACP در Codex                                 | `runtime: "acp"`, `agentId: "codex"`       | Codex را از طریق صفحه کنترل خارجی ACP/acpx اجرا می‌کند. فقط زمانی استفاده کنید که ACP/acpx صراحتاً درخواست شده باشد.        |
| مجموعه فرمان کنترل چت بومی Codex                    | `/codex ...`                               | ریسمان‌های سرور برنامه Codex را از چت متصل، ازسرگرفته، هدایت، متوقف، و بررسی می‌کند.                            |
| مسیر API پلتفرم OpenAI برای مدل‌های سبک GPT/Codex   | ارجاع‌های مدل `openai/*`                      | از احراز هویت کلید API OpenAI استفاده می‌کند مگر اینکه یک بازنویسی زمان اجرا، مانند `agentRuntime.id: "codex"`، نوبت را اجرا کند.     |

این سطوح عمداً مستقل هستند. فعال کردن Plugin `codex` قابلیت‌های بومی سرور برنامه را
در دسترس قرار می‌دهد؛ `openai-codex/*` را به `openai/*` بازنویسی نمی‌کند، نشست‌های
موجود را تغییر نمی‌دهد، و ACP را به پیش‌فرض Codex تبدیل نمی‌کند. انتخاب
`openai-codex/*` یعنی «از مسیر ارائه‌دهنده OAuth در Codex استفاده کن» مگر اینکه جداگانه
یک زمان اجرا را اجبار کنید.

راه‌اندازی رایج اشتراک ChatGPT/Codex از OAuth در Codex برای احراز هویت استفاده می‌کند،
اما ارجاع مدل را به شکل `openai/*` نگه می‌دارد و زمان اجرای `codex` را انتخاب می‌کند:

```json5
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
      agentRuntime: {
        id: "codex",
      },
    },
  },
}
```

این یعنی OpenClaw یک ارجاع مدل OpenAI را انتخاب می‌کند، سپس از زمان اجرای سرور برنامه
Codex می‌خواهد نوبت عامل تعبیه‌شده را اجرا کند. به معنی «استفاده از صورت‌حساب API» نیست،
و به این معنی نیست که کانال، کاتالوگ ارائه‌دهنده مدل، یا مخزن نشست OpenClaw به Codex
تبدیل می‌شود.

وقتی Plugin همراه `codex` فعال است، کنترل Codex با زبان طبیعی باید از سطح فرمان بومی
`/codex` (`/codex bind`, `/codex threads`, `/codex resume`, `/codex steer`,
`/codex stop`) به‌جای ACP استفاده کند. ACP را برای Codex فقط زمانی به کار ببرید که
کاربر صراحتاً ACP/acpx را درخواست می‌کند یا مسیر آداپتور ACP را آزمایش می‌کند.
Claude Code، Gemini CLI، OpenCode، Cursor، و هارنس‌های خارجی مشابه همچنان از ACP
استفاده می‌کنند.

این درخت تصمیم روبه‌روی عامل است:

1. اگر کاربر **اتصال/کنترل/ریسمان/ازسرگیری/هدایت/توقف Codex** را می‌خواهد، وقتی
   Plugin همراه `codex` فعال است از سطح فرمان بومی `/codex` استفاده کنید.
2. اگر کاربر **Codex را به‌عنوان زمان اجرای تعبیه‌شده** می‌خواهد یا تجربه عادی عامل
   Codex مبتنی بر اشتراک را می‌خواهد، از
   `openai/<model>` با `agentRuntime.id: "codex"` استفاده کنید.
3. اگر کاربر **احراز هویت OAuth/اشتراک Codex روی اجراکننده عادی OpenClaw** را می‌خواهد،
   از `openai-codex/<model>` استفاده کنید و زمان اجرا را PI بگذارید.
4. اگر کاربر صراحتاً **ACP**، **acpx**، یا **آداپتور ACP در Codex** را می‌گوید، از
   ACP با `runtime: "acp"` و `agentId: "codex"` استفاده کنید.
5. اگر درخواست برای **Claude Code، Gemini CLI، OpenCode، Cursor، Droid، یا
   هارنس خارجی دیگری** است، از ACP/acpx استفاده کنید، نه زمان اجرای بومی زیرعامل.

| منظورتان...                          | استفاده کنید...                              |
| --------------------------------------- | -------------------------------------------- |
| کنترل چت/ریسمان سرور برنامه Codex      | `/codex ...` از Plugin همراه `codex` |
| زمان اجرای عامل تعبیه‌شده سرور برنامه Codex | `agentRuntime.id: "codex"`                   |
| OAuth در OpenAI Codex روی اجراکننده PI  | ارجاع‌های مدل `openai-codex/*`                  |
| Claude Code یا هارنس خارجی دیگر        | ACP/acpx                                     |

برای تفکیک پیشوند خانواده OpenAI، [OpenAI](/fa/providers/openai) و
[ارائه‌دهندگان مدل](/fa/concepts/model-providers) را ببینید. برای قرارداد پشتیبانی زمان
اجرای Codex، [هارنس Codex](/fa/plugins/codex-harness#v1-support-contract) را ببینید.

## مالکیت زمان اجرا

زمان‌های اجرای مختلف مقدارهای متفاوتی از حلقه را مالک هستند.

| سطح                        | PI تعبیه‌شده OpenClaw                    | سرور برنامه Codex                                                            |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------------- |
| مالک حلقه مدل              | OpenClaw از طریق اجراکننده تعبیه‌شده PI | سرور برنامه Codex                                                            |
| وضعیت کانونی ریسمان        | رونوشت OpenClaw                         | ریسمان Codex، به‌علاوه آینه رونوشت OpenClaw                               |
| ابزارهای پویای OpenClaw    | حلقه ابزار بومی OpenClaw                | از طریق آداپتور Codex پل زده می‌شود                                           |
| ابزارهای بومی پوسته و فایل | مسیر PI/OpenClaw                        | ابزارهای بومی Codex، جایی که پشتیبانی شود از طریق هوک‌های بومی پل زده می‌شوند |
| موتور زمینه                | اسمبلی زمینه بومی OpenClaw              | OpenClaw زمینه را در نوبت Codex پروژه می‌کند                     |
| Compaction                  | OpenClaw یا موتور زمینه انتخاب‌شده      | Compaction بومی Codex، با اعلان‌های OpenClaw و نگهداری آینه |
| تحویل کانال                | OpenClaw                                | OpenClaw                                                                    |

این تفکیک مالکیت قانون اصلی طراحی است:

- اگر OpenClaw مالک سطح باشد، OpenClaw می‌تواند رفتار عادی هوک Plugin را فراهم کند.
- اگر زمان اجرای بومی مالک سطح باشد، OpenClaw به رویدادهای زمان اجرا یا هوک‌های بومی نیاز دارد.
- اگر زمان اجرای بومی مالک وضعیت کانونی ریسمان باشد، OpenClaw باید زمینه را آینه و پروژه کند، نه اینکه بخش‌های داخلی پشتیبانی‌نشده را بازنویسی کند.

## انتخاب زمان اجرا

OpenClaw پس از حل ارائه‌دهنده و مدل، یک زمان اجرای تعبیه‌شده انتخاب می‌کند:

1. زمان اجرای ثبت‌شده یک نشست برنده است. تغییرات پیکربندی یک رونوشت موجود را
   به‌صورت زنده به سیستم ریسمان بومی متفاوتی جابه‌جا نمی‌کند.
2. `OPENCLAW_AGENT_RUNTIME=<id>` آن زمان اجرا را برای نشست‌های جدید یا بازنشانی‌شده اجبار می‌کند.
3. `agents.defaults.agentRuntime.id` یا `agents.list[].agentRuntime.id` می‌تواند
   `auto`، `pi`، یک شناسه هارنس تعبیه‌شده ثبت‌شده مانند `codex`، یا یک
   نام مستعار بک‌اند CLI پشتیبانی‌شده مانند `claude-cli` را تنظیم کند.
4. در حالت `auto`، زمان‌های اجرای Plugin ثبت‌شده می‌توانند جفت‌های ارائه‌دهنده/مدلی را
   که پشتیبانی می‌کنند مطالبه کنند.
5. اگر هیچ زمان اجرایی در حالت `auto` یک نوبت را مطالبه نکند و `fallback: "pi"` تنظیم شده باشد
   (پیش‌فرض)، OpenClaw از PI به‌عنوان جایگزین سازگاری استفاده می‌کند. برای اینکه انتخاب
   حالت `auto` بدون تطابق شکست بخورد، `fallback: "none"` را تنظیم کنید.

زمان‌های اجرای Plugin صریح به‌صورت پیش‌فرض بسته شکست می‌خورند. برای مثال،
`agentRuntime.id: "codex"` یعنی Codex یا یک خطای انتخاب روشن، مگر اینکه
`fallback: "pi"` را در همان محدوده بازنویسی تنظیم کنید. یک بازنویسی زمان اجرا تنظیم
جایگزین گسترده‌تر را به ارث نمی‌برد، بنابراین `agentRuntime.id: "codex"` در سطح عامل
فقط به این دلیل که پیش‌فرض‌ها از `fallback: "pi"` استفاده کرده‌اند بی‌صدا به PI
برگردانده نمی‌شود.

نام‌های مستعار بک‌اند CLI با شناسه‌های هارنس تعبیه‌شده متفاوت هستند. شکل ترجیحی
Claude CLI این است:

```json5
{
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-7",
      agentRuntime: { id: "claude-cli" },
    },
  },
}
```

ارجاع‌های قدیمی مانند `claude-cli/claude-opus-4-7` برای سازگاری همچنان پشتیبانی می‌شوند،
اما پیکربندی جدید باید ارائه‌دهنده/مدل را کانونی نگه دارد و بک‌اند اجرا را در
`agentRuntime.id` قرار دهد.

حالت `auto` عمداً محافظه‌کار است. زمان‌های اجرای Plugin می‌توانند جفت‌های
ارائه‌دهنده/مدلی را که می‌فهمند مطالبه کنند، اما Plugin مربوط به Codex ارائه‌دهنده
`openai-codex` را در حالت `auto` مطالبه نمی‌کند. این کار `openai-codex/*` را به‌عنوان
مسیر صریح OAuth در PI برای Codex نگه می‌دارد و از انتقال بی‌صدای پیکربندی‌های احراز
هویت اشتراکی به هارنس بومی سرور برنامه جلوگیری می‌کند.

اگر `openclaw doctor` هشدار دهد که Plugin `codex` فعال است در حالی که
`openai-codex/*` همچنان از طریق PI مسیریابی می‌شود، با آن مثل یک تشخیص برخورد کنید،
نه یک مهاجرت. وقتی OAuth در PI برای Codex همان چیزی است که می‌خواهید، پیکربندی را
بدون تغییر نگه دارید. فقط زمانی به `openai/<model>` به‌علاوه `agentRuntime.id: "codex"`
تغییر دهید که اجرای بومی سرور برنامه Codex را می‌خواهید.

## قرارداد سازگاری

وقتی یک زمان اجرا PI نیست، باید مستند کند که از چه سطوح OpenClaw پشتیبانی می‌کند.
برای مستندات زمان اجرا از این شکل استفاده کنید:

| پرسش                                  | چرا مهم است                                                                                      |
| -------------------------------------- | ------------------------------------------------------------------------------------------------- |
| مالک حلقه مدل کیست؟                   | تعیین می‌کند تلاش‌های مجدد، ادامه ابزار، و تصمیم‌های پاسخ نهایی کجا رخ می‌دهند.                 |
| مالک تاریخچه مرجع رشته گفتگو کیست؟    | تعیین می‌کند آیا OpenClaw می‌تواند تاریخچه را ویرایش کند یا فقط آن را بازتاب دهد.              |
| آیا ابزارهای پویای OpenClaw کار می‌کنند؟ | پیام‌رسانی، نشست‌ها، cron، و ابزارهای متعلق به OpenClaw به این وابسته‌اند.                    |
| آیا hookهای ابزار پویا کار می‌کنند؟   | Pluginها انتظار `before_tool_call`، `after_tool_call`، و میان‌افزار پیرامون ابزارهای متعلق به OpenClaw را دارند. |
| آیا hookهای ابزار بومی کار می‌کنند؟   | ابزارهای shell، patch، و متعلق به زمان اجرا برای سیاست‌گذاری و مشاهده به پشتیبانی hook بومی نیاز دارند. |
| آیا چرخه عمر موتور زمینه اجرا می‌شود؟ | Pluginهای حافظه و زمینه به چرخه عمر assemble، ingest، after-turn، و Compaction وابسته‌اند.     |
| چه داده‌های Compaction آشکار می‌شود؟  | برخی Pluginها فقط به اعلان‌ها نیاز دارند، درحالی‌که برخی دیگر به فراداده نگه‌داشته‌شده/حذف‌شده نیاز دارند. |
| چه چیزی عمدا پشتیبانی نمی‌شود؟        | کاربران نباید در جایی که زمان اجرای بومی مالک وضعیت بیشتری است، هم‌ارزی PI را فرض کنند.        |

قرارداد پشتیبانی زمان اجرای Codex در
[هارنس Codex](/fa/plugins/codex-harness#v1-support-contract) مستند شده است.

## برچسب‌های وضعیت

خروجی وضعیت ممکن است هر دو برچسب `Execution` و `Runtime` را نشان دهد. آن‌ها را
تشخیص در نظر بگیرید، نه نام ارائه‌دهنده.

- ارجاع مدلی مانند `openai/gpt-5.5` ارائه‌دهنده/مدل انتخاب‌شده را نشان می‌دهد.
- شناسه زمان اجرایی مانند `codex` نشان می‌دهد کدام حلقه نوبت را اجرا می‌کند.
- برچسب کانالی مانند Telegram یا Discord نشان می‌دهد گفتگو کجا در حال انجام است.

اگر یک نشست پس از تغییر پیکربندی زمان اجرا همچنان PI را نشان می‌دهد، با
`/new` یک نشست جدید شروع کنید یا نشست فعلی را با `/reset` پاک کنید. نشست‌های موجود زمان
اجرای ثبت‌شده خود را نگه می‌دارند تا رونوشت گفتگو از دو سامانه نشست بومی ناسازگار بازپخش نشود.
