OpenClaw به عامل‌ها ابزارهایی می‌دهد تا در میان نشست‌ها کار کنند، وضعیت را بررسی کنند، و
زیرعامل‌ها را هماهنگ‌سازی کنند.

## ابزارهای موجود

| ابزار               | کارکرد                                                                |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | نشست‌ها را با فیلترهای اختیاری فهرست می‌کند (نوع، برچسب، عامل، تازگی، پیش‌نمایش)  |
| `sessions_history` | رونوشت یک نشست مشخص را می‌خواند                                   |
| `sessions_send`    | پیامی را به نشست دیگری می‌فرستد و به‌صورت اختیاری منتظر می‌ماند                       |
| `sessions_spawn`   | یک نشست زیرعامل ایزوله برای کار پس‌زمینه ایجاد می‌کند                     |
| `sessions_yield`   | نوبت فعلی را پایان می‌دهد و منتظر نتایج بعدی زیرعامل می‌ماند               |
| `subagents`        | زیرعامل‌های ایجادشده برای این نشست را فهرست، هدایت، یا متوقف می‌کند                    |
| `session_status`   | کارتی به سبک `/status` نشان می‌دهد و به‌صورت اختیاری یک override مدل برای هر نشست تنظیم می‌کند |

این ابزارها همچنان تابع پروفایل ابزار فعال و سیاست‌های allow/deny هستند.
`tools.profile: "coding"` مجموعه کامل هماهنگ‌سازی نشست را شامل می‌شود،
از جمله `sessions_spawn`، `sessions_yield`، و `subagents`.
`tools.profile: "messaging"` ابزارهای پیام‌رسانی میان‌نشستی
(`sessions_list`، `sessions_history`، `sessions_send`، `session_status`) را شامل می‌شود، اما
ایجاد زیرعامل را شامل نمی‌شود. برای حفظ پروفایل پیام‌رسانی و همچنان
اجازه دادن به واگذاری بومی، اضافه کنید:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

سیاست‌های گروه، ارائه‌دهنده، سندباکس، و هر عامل همچنان می‌توانند آن ابزارها را
پس از مرحله پروفایل حذف کنند. از `/tools` در نشست تحت تأثیر استفاده کنید تا
فهرست مؤثر ابزارها را بررسی کنید.

## فهرست کردن و خواندن نشست‌ها

`sessions_list` نشست‌ها را همراه با کلید، agentId، نوع، کانال، مدل،
شمارش توکن‌ها، و زمان‌مهرها برمی‌گرداند. بر اساس نوع (`main`، `group`، `cron`، `hook`،
`node`)، `label` دقیق، `agentId` دقیق، متن جست‌وجو، یا تازگی
(`activeMinutes`) فیلتر کنید. وقتی به تریاژ به سبک صندوق ورودی نیاز دارید، می‌تواند
برای هر ردیف یک عنوان مشتق‌شده با دامنه دید، یک قطعه پیش‌نمایش از آخرین پیام، یا
پیام‌های اخیر محدودشده نیز درخواست کند. عنوان‌ها و پیش‌نمایش‌های مشتق‌شده فقط برای
نشست‌هایی تولید می‌شوند که فراخواننده از قبل تحت سیاست پیکربندی‌شده دید ابزار نشست
می‌تواند ببیند، بنابراین نشست‌های نامرتبط پنهان می‌مانند.

`sessions_history` رونوشت گفت‌وگو را برای یک نشست مشخص دریافت می‌کند.
به‌طور پیش‌فرض، نتایج ابزارها حذف می‌شوند -- برای دیدن آن‌ها `includeTools: true` را ارسال کنید.
نمای برگشتی عمداً محدود و با فیلتر ایمنی است:

- متن دستیار پیش از یادآوری نرمال‌سازی می‌شود:
  - تگ‌های تفکر حذف می‌شوند
  - بلوک‌های داربست `<relevant-memories>` / `<relevant_memories>` حذف می‌شوند
  - بلوک‌های payload XML فراخوانی ابزار به‌صورت متن ساده، مانند `<tool_call>...</tool_call>`،
    `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، و
    `<function_calls>...</function_calls>` حذف می‌شوند، از جمله payloadهای ناقصی که
    هرگز تمیز بسته نمی‌شوند
  - داربست‌های تنزل‌یافته فراخوانی/نتیجه ابزار مانند `[Tool Call: ...]`،
    `[Tool Result ...]`، و `[Historical context ...]` حذف می‌شوند
  - توکن‌های کنترل مدل نشت‌کرده مانند `<|assistant|>`، دیگر توکن‌های ASCII
    `<|...|>`، و گونه‌های تمام‌عرض `<｜...｜>` حذف می‌شوند
  - XML بدشکل فراخوانی ابزار MiniMax مانند `<invoke ...>` /
    `</minimax:tool_call>` حذف می‌شود
- متن شبیه اعتبارنامه/توکن پیش از بازگردانده شدن پاک‌سازی می‌شود
- بلوک‌های متنی بلند کوتاه می‌شوند
- تاریخچه‌های بسیار بزرگ می‌توانند ردیف‌های قدیمی‌تر را حذف کنند یا یک ردیف بیش‌ازحد بزرگ را با
  `[sessions_history omitted: message too large]` جایگزین کنند
- ابزار پرچم‌های خلاصه‌ای مانند `truncated`، `droppedMessages`،
  `contentTruncated`، `contentRedacted`، و `bytes` را گزارش می‌دهد

هر دو ابزار یا یک **کلید نشست** (مانند `"main"`) یا یک **شناسه نشست**
از فراخوانی فهرست قبلی را می‌پذیرند.

اگر به رونوشت دقیق بایت‌به‌بایت نیاز دارید، به‌جای برخورد با `sessions_history` به‌عنوان یک dump خام،
فایل رونوشت روی دیسک را بررسی کنید.

## فرستادن پیام‌های میان‌نشستی

`sessions_send` پیامی را به نشست دیگری تحویل می‌دهد و به‌صورت اختیاری منتظر
پاسخ می‌ماند:

- **ارسال و فراموشی:** `timeoutSeconds: 0` را تنظیم کنید تا در صف قرار بگیرد و
  فوراً برگردد.
- **انتظار برای پاسخ:** یک timeout تنظیم کنید و پاسخ را inline دریافت کنید.

پیام‌ها و پاسخ‌های پیگیری A2A در prompt گیرنده
(`[Inter-session message ... isUser=false]`) و در منشأ رونوشت، به‌عنوان داده میان‌نشستی علامت‌گذاری می‌شوند.
عامل گیرنده باید با آن‌ها به‌عنوان داده مسیریابی‌شده از طریق ابزار رفتار کند، نه به‌عنوان یک
دستور مستقیم نوشته‌شده توسط کاربر نهایی.

پس از پاسخ دادن هدف، OpenClaw می‌تواند یک **حلقه پاسخ‌برگشتی** اجرا کند که در آن
عامل‌ها پیام‌ها را به‌نوبت ردوبدل می‌کنند (تا ۵ نوبت). عامل هدف می‌تواند با
`REPLY_SKIP` زودتر متوقف شود.

## وضعیت و helperهای هماهنگ‌سازی

`session_status` ابزار سبک‌وزن معادل `/status` برای نشست فعلی
یا یک نشست قابل‌مشاهده دیگر است. مصرف، زمان، وضعیت مدل/زمان اجرا، و
زمینه کار پس‌زمینه پیوندشده را در صورت وجود گزارش می‌دهد. مانند `/status`، می‌تواند
شمارنده‌های sparse توکن/کش را از آخرین ورودی مصرف رونوشت backfill کند، و
`model=default` یک override مدل برای هر نشست را پاک می‌کند. برای نشست فعلی فراخواننده از
`sessionKey="current"` استفاده کنید؛ برچسب‌های قابل‌مشاهده کلاینت مانند `openclaw-tui`
کلید نشست نیستند.

`sessions_yield` عمداً نوبت فعلی را پایان می‌دهد تا پیام بعدی بتواند
رویداد پیگیری‌ای باشد که منتظرش هستید. پس از ایجاد زیرعامل‌ها، وقتی
می‌خواهید نتایج تکمیل به‌جای ساختن حلقه‌های polling به‌عنوان پیام بعدی برسند،
از آن استفاده کنید.

`subagents` helper صفحه کنترل برای زیرعامل‌های OpenClaw است که
قبلاً ایجاد شده‌اند. پشتیبانی می‌کند از:

- `action: "list"` برای بررسی اجراهای فعال/اخیر
- `action: "steer"` برای فرستادن راهنمایی پیگیری به یک فرزند در حال اجرا
- `action: "kill"` برای متوقف کردن یک فرزند یا `all`

## ایجاد زیرعامل‌ها

`sessions_spawn` به‌طور پیش‌فرض یک نشست ایزوله برای یک کار پس‌زمینه ایجاد می‌کند.
همیشه non-blocking است -- فوراً با یک `runId` و
`childSessionKey` برمی‌گردد.

گزینه‌های کلیدی:

- `runtime: "subagent"` (پیش‌فرض) یا `"acp"` برای عامل‌های harness خارجی.
- overrideهای `model` و `thinking` برای نشست فرزند.
- `thread: true` برای پیوند دادن ایجاد به یک thread چت (Discord، Slack، و غیره).
- `sandbox: "require"` برای اعمال سندباکس روی فرزند.
- `context: "fork"` برای زیرعامل‌های بومی وقتی فرزند به رونوشت درخواست‌کننده فعلی
  نیاز دارد؛ آن را حذف کنید یا برای یک فرزند تمیز از `context: "isolated"` استفاده کنید.

زیرعامل‌های leaf پیش‌فرض ابزارهای نشست را دریافت نمی‌کنند. وقتی
`maxSpawnDepth >= 2` باشد، زیرعامل‌های هماهنگ‌کننده depth-1 علاوه بر آن
`sessions_spawn`، `subagents`، `sessions_list`، و `sessions_history` را دریافت می‌کنند تا بتوانند
فرزندان خود را مدیریت کنند. اجراهای leaf همچنان ابزارهای هماهنگ‌سازی بازگشتی
دریافت نمی‌کنند.

پس از تکمیل، یک مرحله اعلام نتیجه را در کانال درخواست‌کننده منتشر می‌کند.
تحویل تکمیل، در صورت وجود، مسیریابی thread/topic پیوندشده را حفظ می‌کند، و اگر
مبدأ تکمیل فقط یک کانال را مشخص کند، OpenClaw همچنان می‌تواند از مسیر ذخیره‌شده
نشست درخواست‌کننده (`lastChannel` / `lastTo`) برای تحویل مستقیم
دوباره استفاده کند.

برای رفتار اختصاصی ACP، [عامل‌های ACP](/fa/tools/acp-agents) را ببینید.

## دید

ابزارهای نشست دامنه‌بندی شده‌اند تا آنچه عامل می‌تواند ببیند محدود شود:

| سطح   | دامنه                                    |
| ------- | ---------------------------------------- |
| `self`  | فقط نشست فعلی                 |
| `tree`  | نشست فعلی + زیرعامل‌های ایجادشده     |
| `agent` | همه نشست‌های این عامل              |
| `all`   | همه نشست‌ها (در صورت پیکربندی، میان‌عاملی) |

پیش‌فرض `tree` است. نشست‌های سندباکس‌شده بدون توجه به پیکربندی به `tree` محدود می‌شوند.
