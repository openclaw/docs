OpenClaw به عامل‌ها ابزارهایی می‌دهد تا در میان نشست‌ها کار کنند، وضعیت را بررسی کنند و
زیرعامل‌ها را هماهنگ کنند.

## ابزارهای موجود

| ابزار              | کاری که انجام می‌دهد                                                         |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | فهرست کردن نشست‌ها با فیلترهای اختیاری (kind، label، agent، تازگی، پیش‌نمایش) |
| `sessions_history` | خواندن رونوشت یک نشست مشخص                                                  |
| `sessions_send`    | ارسال پیام به نشست دیگر و در صورت نیاز انتظار                               |
| `sessions_spawn`   | ایجاد یک نشست زیرعامل ایزوله برای کار پس‌زمینه                              |
| `sessions_yield`   | پایان دادن به نوبت فعلی و انتظار برای نتایج پیگیری زیرعامل                  |
| `subagents`        | فهرست کردن وضعیت زیرعامل‌های ایجادشده برای این نشست                         |
| `session_status`   | نمایش کارتی شبیه `/status` و در صورت نیاز تنظیم بازنویسی مدل برای هر نشست   |

این ابزارها همچنان تابع پروفایل ابزار فعال و سیاست مجاز/ممنوع هستند.
`tools.profile: "coding"` مجموعه کامل هماهنگ‌سازی نشست را شامل می‌شود، از جمله
`sessions_spawn`، `sessions_yield`، و `subagents`.
`tools.profile: "messaging"` ابزارهای پیام‌رسانی میان‌نشستی
(`sessions_list`، `sessions_history`، `sessions_send`، `session_status`) را شامل می‌شود، اما
ایجاد زیرعامل را شامل نمی‌شود. برای حفظ پروفایل پیام‌رسانی و همچنان
اجازه دادن به تفویض بومی، اضافه کنید:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

سیاست‌های گروه، ارائه‌دهنده، سندباکس، و هر عامل همچنان می‌توانند پس از مرحله
پروفایل آن ابزارها را حذف کنند. برای بررسی فهرست ابزارهای مؤثر، از نشست
متأثر `/tools` را استفاده کنید.

## فهرست کردن و خواندن نشست‌ها

`sessions_list` نشست‌ها را همراه با کلید، agentId، kind، کانال، مدل،
شمار توکن‌ها و مُهرهای زمانی آن‌ها برمی‌گرداند. بر اساس kind (`main`، `group`، `cron`، `hook`،
`node`)، `label` دقیق، `agentId` دقیق، متن جست‌وجو، یا تازگی
(`activeMinutes`) فیلتر کنید. وقتی به تریاژ سبک صندوق ورودی نیاز دارید، همچنین می‌تواند یک
عنوان مشتق‌شده در محدوده دیدپذیری، یک قطعه پیش‌نمایش از آخرین پیام، یا پیام‌های اخیر
محدودشده را برای هر ردیف درخواست کند. عنوان‌های مشتق‌شده و پیش‌نمایش‌ها فقط برای نشست‌هایی
تولید می‌شوند که فراخواننده از قبل طبق سیاست پیکربندی‌شده دیدپذیری ابزار نشست می‌تواند ببیند، بنابراین
نشست‌های نامرتبط پنهان می‌مانند. وقتی دیدپذیری محدود باشد، `sessions_list`
فراداده اختیاری `visibility` را برمی‌گرداند که حالت مؤثر و هشداری را نشان می‌دهد مبنی بر اینکه
نتایج ممکن است از نظر دامنه محدود باشند.

`sessions_history` رونوشت گفت‌وگو را برای یک نشست مشخص واکشی می‌کند.
به‌طور پیش‌فرض، نتایج ابزارها حذف می‌شوند؛ برای دیدن آن‌ها `includeTools: true` را ارسال کنید.
برای تازه‌ترین دنباله محدودشده از `limit` استفاده کنید. وقتی به فراداده صفحه‌بندی نیاز دارید
`offset: 0` را ارسال کنید، سپس مقدارهای برگشتی `nextOffset` را برای پیمایش رو به عقب
در پنجره‌های قدیمی‌تر رونوشت OpenClaw بدون خواندن فایل‌های خام رونوشت ارسال کنید.
صفحه‌های offset صریح، واردات fallback خارجی CLI را ادغام نمی‌کنند؛ وقتی به آن تاریخچه نمایش
ادغام‌شده نیاز دارید، از نمای پیش‌فرض تازه‌ترین دنباله استفاده کنید.
نمای برگشتی عمداً محدود و با فیلتر ایمنی است:

- متن دستیار پیش از یادآوری نرمال‌سازی می‌شود:
  - برچسب‌های تفکر حذف می‌شوند
  - بلوک‌های داربست `<relevant-memories>` / `<relevant_memories>` حذف می‌شوند
  - بلوک‌های payload XML فراخوانی ابزار به‌صورت متن ساده مانند `<tool_call>...</tool_call>`،
    `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، و
    `<function_calls>...</function_calls>` حذف می‌شوند، از جمله payloadهای ناقصی که
    هرگز تمیز بسته نمی‌شوند
  - داربست‌های تنزل‌یافته فراخوانی/نتیجه ابزار مانند `[Tool Call: ...]`،
    `[Tool Result ...]`، و `[Historical context ...]` حذف می‌شوند
  - توکن‌های کنترل مدلِ نشت‌کرده مانند `<|assistant|>`، توکن‌های ASCII دیگر
    `<|...|>`، و گونه‌های تمام‌عرض `<｜...｜>` حذف می‌شوند
  - XML فراخوانی ابزار MiniMax بدشکل مانند `<invoke ...>` /
    `</minimax:tool_call>` حذف می‌شود
- متن‌های شبیه اعتبارنامه/توکن پیش از بازگردانده شدن پوشانده می‌شوند
- بلوک‌های متنی بلند کوتاه می‌شوند
- تاریخچه‌های بسیار بزرگ می‌توانند ردیف‌های قدیمی‌تر را حذف کنند یا یک ردیف بسیار بزرگ را با
  `[sessions_history omitted: message too large]` جایگزین کنند
- ابزار پرچم‌های خلاصه‌ای مانند `truncated`، `droppedMessages`،
  `contentTruncated`، `contentRedacted`، `bytes`، و فراداده صفحه‌بندی را گزارش می‌کند

هر دو ابزار یا یک **کلید نشست** (مانند `"main"`) یا یک **شناسه نشست**
از یک فراخوانی فهرست قبلی را می‌پذیرند.

اگر به رونوشت دقیق بایت‌به‌بایت نیاز دارید، به‌جای تلقی کردن `sessions_history`
به‌عنوان یک تخلیه خام، فایل رونوشت روی دیسک را بررسی کنید.

## ارسال پیام‌های میان‌نشستی

`sessions_send` پیامی را به نشست دیگر تحویل می‌دهد و در صورت نیاز منتظر
پاسخ می‌ماند:

- **ارسال و فراموشی:** `timeoutSeconds: 0` را تنظیم کنید تا در صف قرار گیرد و
  بلافاصله برگردد.
- **انتظار برای پاسخ:** یک timeout تنظیم کنید و پاسخ را درون‌خطی دریافت کنید.

نشست‌های گفت‌وگوی محدود به thread، مانند کلیدهای Slack یا Discord که به
`:thread:<id>` ختم می‌شوند، هدف‌های معتبر `sessions_send` نیستند. برای هماهنگی
میان‌عاملی از کلید نشست کانال والد استفاده کنید تا پیام‌های مسیریابی‌شده با ابزار درون
یک thread فعال رو‌به‌روی انسان ظاهر نشوند.

پیام‌ها و پاسخ‌های پیگیری A2A به‌عنوان داده میان‌نشستی در پرامپت گیرنده
(`[Inter-session message ... isUser=false]`) و در منشأ رونوشت نشانه‌گذاری می‌شوند.
عامل گیرنده باید آن‌ها را داده مسیریابی‌شده با ابزار بداند، نه دستور مستقیمی که
کاربر نهایی نوشته است.

پس از پاسخ هدف، OpenClaw می‌تواند یک **حلقه پاسخ‌برگشتی** اجرا کند که در آن
عامل‌ها به‌نوبت پیام ردوبدل می‌کنند (تا `session.agentToAgent.maxPingPongTurns`، بازه
0-20، پیش‌فرض 5). عامل هدف می‌تواند برای توقف زودهنگام
`REPLY_SKIP` پاسخ دهد.

## کمک‌کننده‌های وضعیت و هماهنگ‌سازی

`session_status` ابزار سبک‌وزن معادل `/status` برای نشست فعلی
یا نشست قابل‌مشاهده دیگر است. مصرف، زمان، وضعیت مدل/زمان اجرا، و
زمینه کار پس‌زمینه پیوندشده را در صورت وجود گزارش می‌کند. مانند `/status`، می‌تواند
شمارنده‌های پراکنده توکن/کش را از آخرین ورودی مصرف رونوشت تکمیل کند، و
`model=default` یک بازنویسی هرنشستی را پاک می‌کند. برای نشست فعلی فراخواننده از
`sessionKey="current"` استفاده کنید؛ برچسب‌های قابل‌مشاهده مشتری مانند `openclaw-tui`
کلید نشست نیستند.

وقتی فراداده مسیر موجود باشد، `session_status` همچنین یک بلوک JSON قابل‌مشاهده
`Route context` و فیلدهای ساختاریافته همسان `details` را شامل می‌شود. این
فیلدها کلید نشست را از مسیری که هم‌اکنون اجرای زنده را مدیریت می‌کند متمایز می‌کنند:

- `origin` جایی است که نشست ایجاد شده، یا ارائه‌دهنده‌ای است که از یک
  پیشوند کلید نشست قابل‌تحویل استنتاج شده وقتی وضعیت قدیمی‌تر فراداده origin ذخیره‌شده ندارد.
- `active` مسیر اجرای زنده فعلی است. فقط برای نشست زنده یا
  نشست فعلی که اکنون مدیریت می‌شود گزارش می‌شود.
- `deliveryContext` مسیر تحویل ماندگارشده ذخیره‌شده روی نشست است،
  که OpenClaw می‌تواند حتی وقتی سطح فعال متفاوت است برای تحویل بعدی دوباره استفاده کند.

`sessions_yield` عمداً نوبت فعلی را پایان می‌دهد تا پیام بعدی بتواند
رویداد پیگیری‌ای باشد که منتظرش هستید. پس از ایجاد زیرعامل‌ها، وقتی
می‌خواهید نتایج تکمیل به‌جای ساختن حلقه‌های polling به‌عنوان پیام بعدی برسند، از آن استفاده کنید.

`subagents` کمک‌کننده دیدپذیری برای زیرعامل‌های OpenClaw است که از قبل
ایجاد شده‌اند. از `action: "list"` برای بررسی اجراهای فعال/اخیر پشتیبانی می‌کند.

## ایجاد زیرعامل‌ها

`sessions_spawn` به‌طور پیش‌فرض یک نشست ایزوله برای یک کار پس‌زمینه ایجاد می‌کند.
همیشه غیرمسدودکننده است؛ بلافاصله با یک `runId` و
`childSessionKey` برمی‌گردد. اجراهای زیرعامل بومی کار تفویض‌شده را در
نخستین پیام قابل‌مشاهده `[Subagent Task]` نشست فرزند دریافت می‌کنند، در حالی که پرامپت
سیستم فقط قوانین زمان اجرای زیرعامل و زمینه مسیریابی را حمل می‌کند.

گزینه‌های کلیدی:

- `runtime: "subagent"` (پیش‌فرض) یا `"acp"` برای عامل‌های harness خارجی.
- بازنویسی‌های `model` و `thinking` برای نشست فرزند.
- `thread: true` برای پیوند دادن ایجاد به یک thread گفت‌وگو (Discord، Slack، و غیره).
- `sandbox: "require"` برای اعمال سندباکس روی فرزند.
- `context: "fork"` برای زیرعامل‌های بومی وقتی فرزند به رونوشت درخواست‌کننده فعلی نیاز دارد؛
  آن را حذف کنید یا برای فرزند تمیز از `context: "isolated"` استفاده کنید.
  زیرعامل‌های بومی پیوندخورده به thread به‌طور پیش‌فرض از `context: "fork"` استفاده می‌کنند، مگر اینکه
  `threadBindings.defaultSpawnContext` خلاف آن را بگوید.

زیرعامل‌های برگ پیش‌فرض ابزارهای نشست را دریافت نمی‌کنند. وقتی
`maxSpawnDepth >= 2` باشد، زیرعامل‌های هماهنگ‌کننده عمق 1 افزون بر آن
`sessions_spawn`، `subagents`، `sessions_list`، و `sessions_history` را دریافت می‌کنند تا
بتوانند فرزندان خود را مدیریت کنند. اجراهای برگ همچنان ابزارهای هماهنگ‌سازی
بازگشتی را دریافت نمی‌کنند.

پس از تکمیل، یک مرحله اعلام نتیجه را به کانال درخواست‌کننده ارسال می‌کند.
تحویل تکمیل، مسیریابی thread/topic پیوندشده را در صورت وجود حفظ می‌کند، و اگر
مبدأ تکمیل فقط یک کانال را شناسایی کند، OpenClaw همچنان می‌تواند مسیر ذخیره‌شده
نشست درخواست‌کننده (`lastChannel` / `lastTo`) را برای تحویل مستقیم دوباره استفاده کند.

برای رفتارهای اختصاصی ACP، [عامل‌های ACP](/fa/tools/acp-agents) را ببینید.

## دیدپذیری

ابزارهای نشست برای محدود کردن آنچه عامل می‌تواند ببیند دامنه‌بندی شده‌اند:

| سطح    | دامنه                                    |
| ------ | ---------------------------------------- |
| `self` | فقط نشست فعلی                            |
| `tree` | نشست فعلی + زیرعامل‌های ایجادشده         |
| `agent` | همه نشست‌های این عامل                   |
| `all`  | همه نشست‌ها (در صورت پیکربندی، میان‌عاملی) |

پیش‌فرض `tree` است. نشست‌های سندباکس‌شده صرف‌نظر از پیکربندی به `tree` محدود می‌شوند.
