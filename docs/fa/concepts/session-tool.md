OpenClaw به عامل‌ها ابزارهایی می‌دهد تا در میان نشست‌ها کار کنند، وضعیت را بررسی کنند، و
زیرعامل‌ها را هماهنگ کنند.

## ابزارهای موجود

| ابزار               | کاری که انجام می‌دهد                                                                |
| ------------------ | --------------------------------------------------------------------------- |
| `sessions_list`    | فهرست‌کردن نشست‌ها با فیلترهای اختیاری (نوع، برچسب، عامل، تازگی، پیش‌نمایش)  |
| `sessions_history` | خواندن رونوشت یک نشست مشخص                                   |
| `sessions_send`    | ارسال پیام به نشست دیگر و در صورت نیاز انتظار                       |
| `sessions_spawn`   | ایجاد یک نشست زیرعامل جداافتاده برای کار پس‌زمینه                     |
| `sessions_yield`   | پایان‌دادن به نوبت فعلی و انتظار برای نتایج پیگیری زیرعامل               |
| `subagents`        | فهرست‌کردن وضعیت زیرعامل‌های ایجادشده برای این نشست                              |
| `session_status`   | نمایش یک کارت به سبک `/status` و در صورت نیاز تنظیم یک بازنویسی مدل برای هر نشست |

این ابزارها همچنان تابع پروفایل ابزار فعال و سیاست اجازه/رد هستند.
`tools.profile: "coding"` مجموعه کامل هماهنگ‌سازی نشست را شامل می‌شود،
از جمله `sessions_spawn`، `sessions_yield`، و `subagents`.
`tools.profile: "messaging"` ابزارهای پیام‌رسانی میان‌نشستی
(`sessions_list`، `sessions_history`، `sessions_send`، `session_status`) را شامل می‌شود اما
ایجاد زیرعامل را شامل نمی‌شود. برای حفظ پروفایل پیام‌رسانی و همچنان
اجازه‌دادن به واگذاری بومی، اضافه کنید:

```json5
{
  tools: {
    profile: "messaging",
    alsoAllow: ["sessions_spawn", "sessions_yield", "subagents"],
  },
}
```

سیاست‌های گروه، ارائه‌دهنده، sandbox، و هر عامل همچنان می‌توانند آن ابزارها را
پس از مرحله پروفایل حذف کنند. از `/tools` در نشست متاثر استفاده کنید تا
فهرست ابزار موثر را بررسی کنید.

## فهرست‌کردن و خواندن نشست‌ها

`sessions_list` نشست‌ها را همراه با کلید، agentId، نوع، کانال، مدل،
شمارش توکن‌ها، و زمان‌نماهای آن‌ها برمی‌گرداند. بر اساس نوع (`main`، `group`، `cron`، `hook`،
`node`)، `label` دقیق، `agentId` دقیق، متن جست‌وجو، یا تازگی
(`activeMinutes`) فیلتر کنید. وقتی به دسته‌بندی به سبک صندوق پیام نیاز دارید، می‌تواند یک
عنوان مشتق‌شده در محدوده دید، یک قطعه پیش‌نمایش از آخرین پیام، یا پیام‌های اخیر محدود
را هم برای هر ردیف درخواست کند. عنوان‌های مشتق‌شده و پیش‌نمایش‌ها فقط برای نشست‌هایی تولید می‌شوند
که فراخواننده از پیش طبق سیاست پیکربندی‌شده دید ابزار نشست می‌تواند ببیند، بنابراین
نشست‌های نامرتبط پنهان می‌مانند. وقتی دید محدود باشد، `sessions_list`
فراداده اختیاری `visibility` را برمی‌گرداند که حالت موثر و هشداری را نشان می‌دهد که
نتایج ممکن است به محدوده محدود شده باشند.

`sessions_history` رونوشت گفت‌وگو را برای یک نشست مشخص دریافت می‌کند.
به‌طور پیش‌فرض، نتایج ابزارها حذف می‌شوند -- برای دیدن آن‌ها `includeTools: true` را ارسال کنید.
نمای برگشتی عمدا محدود و با فیلترهای ایمنی پردازش شده است:

- متن دستیار پیش از یادآوری عادی‌سازی می‌شود:
  - برچسب‌های تفکر حذف می‌شوند
  - بلوک‌های داربستی `<relevant-memories>` / `<relevant_memories>` حذف می‌شوند
  - بلوک‌های payload فراخوانی ابزار در XML متن ساده مانند `<tool_call>...</tool_call>`،
    `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، و
    `<function_calls>...</function_calls>` حذف می‌شوند، از جمله payloadهای کوتاه‌شده‌ای
    که هرگز به‌درستی بسته نمی‌شوند
  - داربست‌های تنزل‌یافته فراخوانی/نتیجه ابزار مانند `[Tool Call: ...]`،
    `[Tool Result ...]`، و `[Historical context ...]` حذف می‌شوند
  - توکن‌های کنترل مدل نشت‌کرده مانند `<|assistant|>`، سایر توکن‌های ASCII
    `<|...|>`، و گونه‌های تمام‌عرض `<｜...｜>` حذف می‌شوند
  - XML بدشکل فراخوانی ابزار MiniMax مانند `<invoke ...>` /
    `</minimax:tool_call>` حذف می‌شود
- متن‌های شبیه اعتبارنامه/توکن پیش از بازگرداندن پوشانده می‌شوند
- بلوک‌های متنی بلند کوتاه می‌شوند
- تاریخچه‌های بسیار بزرگ می‌توانند ردیف‌های قدیمی‌تر را حذف کنند یا یک ردیف بیش‌ازحد بزرگ را با
  `[sessions_history omitted: message too large]` جایگزین کنند
- ابزار پرچم‌های خلاصه‌ای مانند `truncated`، `droppedMessages`،
  `contentTruncated`، `contentRedacted`، و `bytes` را گزارش می‌کند

هر دو ابزار یا یک **کلید نشست** (مانند `"main"`) یا یک **شناسه نشست**
از فراخوانی فهرست قبلی را می‌پذیرند.

اگر به رونوشت دقیق بایت‌به‌بایت نیاز دارید، به‌جای تلقی‌کردن `sessions_history` به‌عنوان dump خام،
فایل رونوشت روی دیسک را بررسی کنید.

## ارسال پیام‌های میان‌نشستی

`sessions_send` پیامی را به نشست دیگر تحویل می‌دهد و در صورت نیاز منتظر
پاسخ می‌ماند:

- **ارسال و فراموش‌کردن:** `timeoutSeconds: 0` را تنظیم کنید تا در صف قرار بگیرد و
  بلافاصله برگردد.
- **انتظار برای پاسخ:** یک timeout تنظیم کنید و پاسخ را درون‌خطی دریافت کنید.

نشست‌های گفت‌وگوی محدود به thread، مانند کلیدهای Slack یا Discord که به
`:thread:<id>` ختم می‌شوند، هدف‌های معتبر `sessions_send` نیستند. برای هماهنگی میان‌عاملی
از کلید نشست کانال والد استفاده کنید تا پیام‌های مسیردهی‌شده با ابزار داخل یک thread فعال
روبروی انسان ظاهر نشوند.

پیام‌ها و پاسخ‌های پیگیری A2A به‌عنوان داده میان‌نشستی در prompt دریافتی
(`[Inter-session message ... isUser=false]`) و در منشأ رونوشت علامت‌گذاری می‌شوند.
عامل دریافت‌کننده باید با آن‌ها به‌عنوان داده مسیردهی‌شده با ابزار رفتار کند، نه به‌عنوان
دستور مستقیم نوشته‌شده توسط کاربر نهایی.

پس از پاسخ‌دادن هدف، OpenClaw می‌تواند یک **حلقه پاسخ‌برگشتی** اجرا کند که در آن
عامل‌ها به‌نوبت پیام می‌دهند (تا `session.agentToAgent.maxPingPongTurns`، بازه
0-20، پیش‌فرض 5). عامل هدف می‌تواند برای توقف زودهنگام
`REPLY_SKIP` پاسخ دهد.

## کمک‌کننده‌های وضعیت و هماهنگ‌سازی

`session_status` ابزار سبک‌وزن معادل `/status` برای نشست فعلی
یا یک نشست قابل‌مشاهده دیگر است. مصرف، زمان، وضعیت مدل/runtime، و
زمینه کار پس‌زمینه پیوندشده را در صورت وجود گزارش می‌کند. مانند `/status`، می‌تواند
شمارنده‌های پراکنده توکن/cache را از تازه‌ترین ورودی مصرف رونوشت تکمیل کند، و
`model=default` یک بازنویسی هرنشستی را پاک می‌کند. برای نشست فعلی فراخواننده از
`sessionKey="current"` استفاده کنید؛ برچسب‌های قابل‌مشاهده کلاینت مانند `openclaw-tui`
کلید نشست نیستند.

وقتی فراداده مسیر موجود باشد، `session_status` همچنین یک بلوک JSON قابل‌مشاهده
`Route context` و فیلدهای ساخت‌یافته متناظر `details` را شامل می‌شود. این
فیلدها کلید نشست را از مسیری که در حال حاضر اجرای زنده را مدیریت می‌کند
متمایز می‌کنند:

- `origin` جایی است که نشست ایجاد شده، یا ارائه‌دهنده‌ای است که از پیشوند
  کلید نشست قابل‌تحویل استنباط شده، وقتی وضعیت قدیمی‌تر فراداده origin ذخیره‌شده ندارد.
- `active` مسیر اجرای زنده فعلی است. فقط برای نشست زنده یا
  فعلی که همین حالا مدیریت می‌شود گزارش می‌شود.
- `deliveryContext` مسیر تحویل ماندگار ذخیره‌شده روی نشست است،
  که OpenClaw می‌تواند بعدها حتی وقتی سطح فعال متفاوت است برای تحویل دوباره استفاده کند.

`sessions_yield` عمدا نوبت فعلی را پایان می‌دهد تا پیام بعدی بتواند
رویداد پیگیری‌ای باشد که منتظر آن هستید. پس از ایجاد زیرعامل‌ها از آن استفاده کنید وقتی
می‌خواهید نتایج تکمیل به‌عنوان پیام بعدی برسند نه اینکه حلقه‌های polling بسازید.

`subagents` کمک‌کننده دید برای زیرعامل‌های OpenClaw است که از قبل ایجاد شده‌اند.
از `action: "list"` برای بررسی اجراهای فعال/اخیر پشتیبانی می‌کند.

## ایجاد زیرعامل‌ها

`sessions_spawn` به‌طور پیش‌فرض یک نشست جداافتاده برای یک کار پس‌زمینه ایجاد می‌کند.
همیشه غیرمسدودکننده است -- بلافاصله با یک `runId` و
`childSessionKey` برمی‌گردد. اجراهای زیرعامل بومی کار واگذارشده را در
اولین پیام قابل‌مشاهده `[Subagent Task]` نشست فرزند دریافت می‌کنند، در حالی که prompt
سیستم فقط قوانین runtime زیرعامل و زمینه مسیریابی را حمل می‌کند.

گزینه‌های کلیدی:

- `runtime: "subagent"` (پیش‌فرض) یا `"acp"` برای عامل‌های harness خارجی.
- بازنویسی‌های `model` و `thinking` برای نشست فرزند.
- `thread: true` برای اتصال spawn به یک thread گفت‌وگو (Discord، Slack، و غیره).
- `sandbox: "require"` برای اجبار sandboxing روی فرزند.
- `context: "fork"` برای زیرعامل‌های بومی وقتی فرزند به رونوشت درخواست‌کننده فعلی
  نیاز دارد؛ آن را حذف کنید یا از `context: "isolated"` برای یک فرزند تمیز استفاده کنید.
  زیرعامل‌های بومی متصل به thread به‌طور پیش‌فرض `context: "fork"` هستند مگر اینکه
  `threadBindings.defaultSpawnContext` چیز دیگری بگوید.

زیرعامل‌های برگ پیش‌فرض ابزارهای نشست را دریافت نمی‌کنند. وقتی
`maxSpawnDepth >= 2` باشد، زیرعامل‌های هماهنگ‌کننده عمق 1 علاوه بر آن
`sessions_spawn`، `subagents`، `sessions_list`، و `sessions_history` را دریافت می‌کنند تا
بتوانند فرزندان خود را مدیریت کنند. اجراهای برگ همچنان ابزارهای هماهنگ‌سازی
بازگشتی را دریافت نمی‌کنند.

پس از تکمیل، یک مرحله اعلان نتیجه را به کانال درخواست‌کننده ارسال می‌کند.
تحویل تکمیل، مسیریابی thread/topic متصل را در صورت موجودبودن حفظ می‌کند، و اگر
origin تکمیل فقط یک کانال را شناسایی کند، OpenClaw همچنان می‌تواند از مسیر ذخیره‌شده
نشست درخواست‌کننده (`lastChannel` / `lastTo`) برای تحویل مستقیم دوباره استفاده کند.

برای رفتار ویژه ACP، [عامل‌های ACP](/fa/tools/acp-agents) را ببینید.

## دید

ابزارهای نشست محدوده‌بندی شده‌اند تا آنچه عامل می‌تواند ببیند محدود شود:

| سطح   | محدوده                                    |
| ------- | ---------------------------------------- |
| `self`  | فقط نشست فعلی                 |
| `tree`  | نشست فعلی + زیرعامل‌های ایجادشده     |
| `agent` | همه نشست‌های این عامل              |
| `all`   | همه نشست‌ها (در صورت پیکربندی، میان‌عاملی) |

پیش‌فرض `tree` است. نشست‌های sandbox شده، صرف‌نظر از پیکربندی، به `tree` محدود می‌شوند.
