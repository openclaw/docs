OpenClaw یک **زمان اجرای عامل تعبیه‌شدهٔ واحد** را اجرا می‌کند؛ یک فرایند عامل برای هر Gateway، با فضای کاری، فایل‌های راه‌انداز و مخزن نشست اختصاصی خودش. این صفحه قرارداد آن زمان اجرا را پوشش می‌دهد: فضای کاری باید شامل چه چیزهایی باشد، کدام فایل‌ها تزریق می‌شوند، و نشست‌ها چگونه بر پایهٔ آن راه‌اندازی می‌شوند.

## فضای کاری (الزامی)

OpenClaw از یک پوشهٔ فضای کاری عامل واحد (`agents.defaults.workspace`) به‌عنوان **تنها** پوشهٔ کاری عامل (`cwd`) برای ابزارها و زمینه استفاده می‌کند.

توصیه‌شده: از `openclaw setup` برای ساختن `~/.openclaw/openclaw.json` در صورت نبودن آن و مقداردهی اولیهٔ فایل‌های فضای کاری استفاده کنید.

چیدمان کامل فضای کاری + راهنمای پشتیبان‌گیری: [فضای کاری عامل](/fa/concepts/agent-workspace)

اگر `agents.defaults.sandbox` فعال باشد، نشست‌های غیر اصلی می‌توانند این را با فضاهای کاری مخصوص هر نشست زیر `agents.defaults.sandbox.workspaceRoot` بازنویسی کنند (ببینید
[پیکربندی Gateway](/fa/gateway/configuration)).

## فایل‌های راه‌انداز (تزریق‌شده)

درون `agents.defaults.workspace`، OpenClaw انتظار این فایل‌های قابل‌ویرایش توسط کاربر را دارد:

- `AGENTS.md` — دستورالعمل‌های عملیاتی + «حافظه»
- `SOUL.md` — شخصیت، مرزها، لحن
- `TOOLS.md` — یادداشت‌های ابزار که کاربر نگهداری می‌کند (مثلاً `imsg`، `sag`، قراردادها)
- `BOOTSTRAP.md` — آیین یک‌بارهٔ اجرای نخست (پس از تکمیل حذف می‌شود)
- `IDENTITY.md` — نام/حال‌وهوا/ایموجی عامل
- `USER.md` — نمایهٔ کاربر + شیوهٔ خطاب ترجیحی

در نخستین نوبت یک نشست جدید، OpenClaw محتوای این فایل‌ها را مستقیماً به زمینهٔ عامل تزریق می‌کند.

فایل‌های خالی نادیده گرفته می‌شوند. فایل‌های بزرگ کوتاه و با یک نشانگر بریده می‌شوند تا پرامپت‌ها سبک بمانند (برای محتوای کامل، فایل را بخوانید).

اگر فایلی وجود نداشته باشد، OpenClaw یک خط نشانگر «فایل مفقود» تزریق می‌کند (و `openclaw setup` یک قالب پیش‌فرض امن می‌سازد).

`BOOTSTRAP.md` فقط برای یک **فضای کاری کاملاً جدید** ساخته می‌شود (وقتی هیچ فایل راه‌انداز دیگری وجود ندارد). اگر آن را پس از تکمیل آیین حذف کنید، نباید در راه‌اندازی‌های بعدی دوباره ساخته شود.

برای غیرفعال کردن کامل ساخت فایل‌های راه‌انداز (برای فضاهای کاری ازپیش‌آماده‌شده)، تنظیم کنید:

```json5
{ agents: { defaults: { skipBootstrap: true } } }
```

## ابزارهای داخلی

ابزارهای هسته (خواندن/اجرا/ویرایش/نوشتن و ابزارهای سامانه‌ای مرتبط) همیشه در دسترس‌اند، مشروط به سیاست ابزار. `apply_patch` اختیاری است و با `tools.exec.applyPatch` کنترل می‌شود. `TOOLS.md` کنترل نمی‌کند کدام ابزارها وجود دارند؛ راهنمایی است برای اینکه _شما_ می‌خواهید آن‌ها چگونه استفاده شوند.

## Skills

OpenClaw Skills را از این مکان‌ها بارگذاری می‌کند (بالاترین اولویت ابتدا):

- فضای کاری: `<workspace>/skills`
- Skills عامل پروژه: `<workspace>/.agents/skills`
- Skills شخصی عامل: `~/.agents/skills`
- مدیریت‌شده/محلی: `~/.openclaw/skills`
- همراه بسته (ارسال‌شده با نصب)
- پوشه‌های Skills اضافی: `skills.load.extraDirs`

Skills می‌توانند با پیکربندی/محیط کنترل شوند (بخش `skills` را در [پیکربندی Gateway](/fa/gateway/configuration) ببینید).

## مرزهای زمان اجرا

زمان اجرای عامل تعبیه‌شده بر پایهٔ هستهٔ عامل Pi ساخته شده است (مدل‌ها، ابزارها و خط لولهٔ پرامپت). مدیریت نشست، کشف، اتصال ابزار و تحویل کانال، لایه‌هایی هستند که OpenClaw روی آن هسته مالکیتشان را دارد.

## نشست‌ها

رونوشت‌های نشست به‌صورت JSONL در این مسیر ذخیره می‌شوند:

- `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

شناسهٔ نشست پایدار است و توسط OpenClaw انتخاب می‌شود.
پوشه‌های نشست قدیمی از ابزارهای دیگر خوانده نمی‌شوند.

## هدایت هنگام استریم

وقتی حالت صف `steer` باشد، پیام‌های ورودی به اجرای جاری تزریق می‌شوند.
هدایت صف‌شده **پس از پایان نوبت فعلی دستیار در اجرای فراخوانی‌های ابزارش** و پیش از فراخوانی بعدی LLM تحویل داده می‌شود. هدایت دیگر فراخوانی‌های ابزار باقی‌مانده از پیام فعلی دستیار را رد نمی‌کند؛ در عوض پیام صف‌شده را در مرز مدل بعدی تزریق می‌کند.

وقتی حالت صف `followup` یا `collect` باشد، پیام‌های ورودی تا پایان نوبت فعلی نگه داشته می‌شوند، سپس یک نوبت عامل جدید با payloadهای صف‌شده آغاز می‌شود. برای رفتار حالت + debounce/cap، [صف](/fa/concepts/queue) را ببینید.

استریم بلوکی، بلوک‌های کامل‌شدهٔ دستیار را به‌محض پایانشان ارسال می‌کند؛ این قابلیت **به‌صورت پیش‌فرض خاموش است** (`agents.defaults.blockStreamingDefault: "off"`).
مرز را از طریق `agents.defaults.blockStreamingBreak` تنظیم کنید (`text_end` در برابر `message_end`؛ پیش‌فرض text_end است).
قطعه‌بندی نرم بلوک را با `agents.defaults.blockStreamingChunk` کنترل کنید (پیش‌فرض ۸۰۰ تا ۱۲۰۰ نویسه؛ شکست پاراگراف‌ها را ترجیح می‌دهد، سپس خطوط جدید؛ جمله‌ها در آخر).
قطعه‌های استریم‌شده را با `agents.defaults.blockStreamingCoalesce` ادغام کنید تا هرزپیام تک‌خطی کاهش یابد (ادغام مبتنی بر بیکاری پیش از ارسال). کانال‌های غیر Telegram برای فعال کردن پاسخ‌های بلوکی به `*.blockStreaming: true` صریح نیاز دارند.
خلاصه‌های مفصل ابزار هنگام شروع ابزار منتشر می‌شوند (بدون debounce)؛ رابط کاربری Control در صورت موجود بودن، خروجی ابزار را از طریق رویدادهای عامل استریم می‌کند.
جزئیات بیشتر: [استریم + قطعه‌بندی](/fa/concepts/streaming).

## ارجاع‌های مدل

ارجاع‌های مدل در پیکربندی (برای نمونه `agents.defaults.model` و `agents.defaults.models`) با جدا کردن روی **اولین** `/` تحلیل می‌شوند.

- هنگام پیکربندی مدل‌ها از `provider/model` استفاده کنید.
- اگر خود شناسهٔ مدل شامل `/` است (به سبک OpenRouter)، پیشوند ارائه‌دهنده را اضافه کنید (مثال: `openrouter/moonshotai/kimi-k2`).
- اگر ارائه‌دهنده را حذف کنید، OpenClaw ابتدا یک alias را امتحان می‌کند، سپس یک تطابق یکتای ارائه‌دهندهٔ پیکربندی‌شده برای همان شناسهٔ دقیق مدل، و فقط پس از آن به ارائه‌دهندهٔ پیش‌فرض پیکربندی‌شده برمی‌گردد. اگر آن ارائه‌دهنده دیگر مدل پیش‌فرض پیکربندی‌شده را عرضه نکند، OpenClaw به‌جای نمایش یک پیش‌فرض کهنهٔ ارائه‌دهندهٔ حذف‌شده، به اولین ارائه‌دهنده/مدل پیکربندی‌شده برمی‌گردد.

## پیکربندی (حداقلی)

در حداقل حالت، تنظیم کنید:

- `agents.defaults.workspace`
- `channels.whatsapp.allowFrom` (قویاً توصیه می‌شود)
