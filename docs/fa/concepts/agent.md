OpenClaw یک **زمان اجرای تعبیه‌شده واحد برای عامل** اجرا می‌کند: یک فرایند عامل برای هر
Gateway، با فضای کاری، فایل‌های راه‌اندازی اولیه، و مخزن نشست خودش. این صفحه
قرارداد آن زمان اجرا را پوشش می‌دهد: فضای کاری باید چه چیزهایی داشته باشد، کدام فایل‌ها
تزریق می‌شوند، و نشست‌ها چگونه بر اساس آن راه‌اندازی اولیه می‌شوند.

## فضای کاری (الزامی)

OpenClaw از یک پوشه فضای کاری واحد برای عامل (`agents.defaults.workspace`) به‌عنوان **تنها** پوشه کاری (`cwd`) عامل برای ابزارها و زمینه استفاده می‌کند.

توصیه‌شده: از `openclaw setup` استفاده کنید تا اگر `~/.openclaw/openclaw.json` وجود ندارد ساخته شود و فایل‌های فضای کاری مقداردهی اولیه شوند.

طرح کامل فضای کاری + راهنمای پشتیبان‌گیری: [فضای کاری عامل](/fa/concepts/agent-workspace)

اگر `agents.defaults.sandbox` فعال باشد، نشست‌های غیر اصلی می‌توانند این مورد را با
فضاهای کاری مخصوص هر نشست زیر `agents.defaults.sandbox.workspaceRoot` بازنویسی کنند (نگاه کنید به
[پیکربندی Gateway](/fa/gateway/configuration)).

## فایل‌های راه‌اندازی اولیه (تزریق‌شده)

داخل `agents.defaults.workspace`، OpenClaw انتظار دارد این فایل‌های قابل ویرایش توسط کاربر وجود داشته باشند:

- `AGENTS.md` — دستورالعمل‌های عملیاتی + «حافظه»
- `SOUL.md` — شخصیت، مرزها، لحن
- `TOOLS.md` — یادداشت‌های ابزار که کاربر نگهداری می‌کند (مثلاً `imsg`، `sag`، قراردادها)
- `BOOTSTRAP.md` — آیین یک‌باره اجرای نخست (پس از تکمیل حذف می‌شود)
- `IDENTITY.md` — نام/حال‌وهوا/ایموجی عامل
- `USER.md` — نمایه کاربر + شیوه خطاب ترجیحی

در نخستین نوبت یک نشست جدید، OpenClaw محتوای این فایل‌ها را مستقیماً در زمینه عامل تزریق می‌کند.

فایل‌های خالی نادیده گرفته می‌شوند. فایل‌های بزرگ کوتاه و با یک نشانگر قطع می‌شوند تا پرامپت‌ها سبک بمانند (برای محتوای کامل، فایل را بخوانید).

اگر فایلی وجود نداشته باشد، OpenClaw یک خط نشانگر «فایل گم‌شده» تزریق می‌کند (و `openclaw setup` یک الگوی پیش‌فرض امن می‌سازد).

`BOOTSTRAP.md` فقط برای یک **فضای کاری کاملاً جدید** ساخته می‌شود (وقتی هیچ فایل راه‌اندازی اولیه دیگری وجود ندارد). اگر آن را پس از تکمیل آیین حذف کنید، نباید در راه‌اندازی‌های بعدی دوباره ساخته شود.

برای غیرفعال کردن کامل ساخت فایل‌های راه‌اندازی اولیه (برای فضاهای کاری از پیش آماده‌شده)، تنظیم کنید:

```json5
{ agents: { defaults: { skipBootstrap: true } } }
```

## ابزارهای داخلی

ابزارهای هسته‌ای (read/exec/edit/write و ابزارهای سیستمی مرتبط) همیشه در دسترس هستند،
با رعایت سیاست ابزار. `apply_patch` اختیاری است و با
`tools.exec.applyPatch` کنترل می‌شود. `TOOLS.md` کنترل نمی‌کند کدام ابزارها وجود دارند؛ بلکه
راهنمایی است برای اینکه _شما_ می‌خواهید آن‌ها چگونه استفاده شوند.

## Skills

OpenClaw، Skills را از این مکان‌ها بارگذاری می‌کند (بالاترین اولویت ابتدا):

- فضای کاری: `<workspace>/skills`
- Skills عامل پروژه: `<workspace>/.agents/skills`
- Skills شخصی عامل: `~/.agents/skills`
- مدیریت‌شده/محلی: `~/.openclaw/skills`
- بسته‌بندی‌شده (همراه نصب ارائه شده)
- پوشه‌های Skills اضافی: `skills.load.extraDirs`

Skills می‌تواند با پیکربندی/env محدود شود (بخش `skills` را در [پیکربندی Gateway](/fa/gateway/configuration) ببینید).

## مرزهای زمان اجرا

زمان اجرای تعبیه‌شده عامل بر پایه هسته عامل Pi ساخته شده است (مدل‌ها، ابزارها، و
خط لوله پرامپت). مدیریت نشست، کشف، اتصال ابزارها، و تحویل کانال
لایه‌هایی متعلق به OpenClaw هستند که روی آن هسته قرار می‌گیرند.

## نشست‌ها

رونوشت‌های نشست به‌صورت JSONL در این مسیر ذخیره می‌شوند:

- `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

شناسه نشست پایدار است و توسط OpenClaw انتخاب می‌شود.
پوشه‌های نشست قدیمی از ابزارهای دیگر خوانده نمی‌شوند.

## هدایت هنگام پخش جریانی

وقتی حالت صف `steer` باشد، پیام‌های ورودی در اجرای جاری تزریق می‌شوند.
هدایت‌های صف‌شده **پس از پایان نوبت فعلی دستیار و اجرای فراخوانی‌های ابزار آن**،
پیش از فراخوانی بعدی LLM تحویل داده می‌شوند. Pi برای `steer` همه پیام‌های هدایت
در انتظار را با هم تخلیه می‌کند؛ `queue` قدیمی در هر مرز مدل یک پیام را تخلیه می‌کند.
هدایت دیگر فراخوانی‌های ابزار باقی‌مانده از پیام فعلی دستیار را رد نمی‌کند.

وقتی حالت صف `followup` یا `collect` باشد، پیام‌های ورودی تا پایان
نوبت فعلی نگه داشته می‌شوند، سپس یک نوبت عامل جدید با بارهای صف‌شده آغاز می‌شود. برای رفتار حالت‌ها
و مرزها، [صف](/fa/concepts/queue) و [صف هدایت](/fa/concepts/queue-steering) را ببینید.

پخش جریانی بلوک، بلوک‌های کامل‌شده دستیار را به‌محض پایان یافتنشان ارسال می‌کند؛ این گزینه
**به‌صورت پیش‌فرض خاموش است** (`agents.defaults.blockStreamingDefault: "off"`).
مرز را با `agents.defaults.blockStreamingBreak` تنظیم کنید (`text_end` در برابر `message_end`؛ پیش‌فرض text_end است).
قطعه‌بندی نرم بلوک را با `agents.defaults.blockStreamingChunk` کنترل کنید (پیش‌فرض
800 تا 1200 نویسه؛ شکست پاراگراف را ترجیح می‌دهد، سپس خطوط جدید؛ جمله‌ها در آخر).
قطعه‌های پخش‌شده را با `agents.defaults.blockStreamingCoalesce` ادغام کنید تا
هرزپیام تک‌خطی کاهش یابد (ادغام مبتنی بر بیکاری پیش از ارسال). کانال‌های غیر Telegram برای
فعال‌سازی پاسخ‌های بلوکی به `*.blockStreaming: true` صریح نیاز دارند.
خلاصه‌های پرجزئیات ابزار هنگام شروع ابزار منتشر می‌شوند (بدون debounce)؛ رابط کاربری کنترل
در صورت موجود بودن، خروجی ابزار را از طریق رویدادهای عامل پخش می‌کند.
جزئیات بیشتر: [پخش جریانی + قطعه‌بندی](/fa/concepts/streaming).

## ارجاعات مدل

ارجاعات مدل در پیکربندی (برای نمونه `agents.defaults.model` و `agents.defaults.models`) با تقسیم روی **نخستین** `/` تجزیه می‌شوند.

- هنگام پیکربندی مدل‌ها از `provider/model` استفاده کنید.
- اگر خود شناسه مدل شامل `/` است (به سبک OpenRouter)، پیشوند ارائه‌دهنده را وارد کنید (نمونه: `openrouter/moonshotai/kimi-k2`).
- اگر ارائه‌دهنده را حذف کنید، OpenClaw ابتدا یک نام مستعار را امتحان می‌کند، سپس یک
  تطابق ارائه‌دهنده پیکربندی‌شده یکتا برای همان شناسه دقیق مدل، و تنها پس از آن به
  ارائه‌دهنده پیش‌فرض پیکربندی‌شده برمی‌گردد. اگر آن ارائه‌دهنده دیگر مدل پیش‌فرض
  پیکربندی‌شده را ارائه نکند، OpenClaw به‌جای نمایش یک پیش‌فرض کهنه مربوط به ارائه‌دهنده حذف‌شده،
  به نخستین ارائه‌دهنده/مدل پیکربندی‌شده برمی‌گردد.

## پیکربندی (حداقلی)

حداقل، این موارد را تنظیم کنید:

- `agents.defaults.workspace`
- `channels.whatsapp.allowFrom` (به‌شدت توصیه می‌شود)
