OpenClaw از «احراز هویت اشتراکی» از طریق OAuth برای ارائه‌دهندگانی که آن را ارائه می‌کنند پشتیبانی می‌کند
(به‌ویژه **OpenAI Codex (ChatGPT OAuth)**). برای Anthropic، تقسیم‌بندی عملی
اکنون چنین است:

- **کلید API Anthropic**: صورت‌حساب عادی Anthropic API
- **Anthropic Claude CLI / احراز هویت اشتراکی درون OpenClaw**: کارکنان Anthropic
  به ما گفتند این نوع استفاده دوباره مجاز است

OpenAI Codex OAuth به‌صراحت برای استفاده در ابزارهای خارجی مانند
OpenClaw پشتیبانی می‌شود. این صفحه توضیح می‌دهد:

برای Anthropic در محیط تولید، احراز هویت با کلید API مسیر امن‌تر و توصیه‌شده است.

- **تبادل توکن** OAuth چگونه کار می‌کند (PKCE)
- توکن‌ها کجا **ذخیره** می‌شوند (و چرا)
- چگونه **چندین حساب** را مدیریت کنید (پروفایل‌ها + بازنویسی‌های هر نشست)

OpenClaw همچنین از **Pluginهای ارائه‌دهنده** پشتیبانی می‌کند که جریان‌های OAuth یا کلید API
خودشان را همراه دارند. آن‌ها را با این دستور اجرا کنید:

```bash
openclaw models auth login --provider <id>
```

## محل جذب توکن (چرا وجود دارد)

ارائه‌دهندگان OAuth معمولا هنگام جریان‌های ورود/نوسازی، یک **توکن نوسازی جدید** صادر می‌کنند. برخی ارائه‌دهندگان (یا کلاینت‌های OAuth) می‌توانند وقتی توکن جدیدی برای همان کاربر/برنامه صادر می‌شود، توکن‌های نوسازی قدیمی‌تر را نامعتبر کنند.

نشانه عملی:

- از طریق OpenClaw _و_ از طریق Claude Code / Codex CLI وارد می‌شوید → یکی از آن‌ها بعدا به‌صورت تصادفی «خارج‌شده» می‌شود

برای کاهش این مشکل، OpenClaw با `auth-profiles.json` مثل یک **محل جذب توکن** رفتار می‌کند:

- زمان اجرا اعتبارنامه‌ها را از **یک مکان** می‌خواند
- می‌توانیم چندین پروفایل را نگه داریم و آن‌ها را به‌صورت قطعی مسیریابی کنیم
- استفاده مجدد از CLI خارجی مختص ارائه‌دهنده است: Codex CLI می‌تواند یک پروفایل خالی
  `openai-codex:default` را راه‌اندازی اولیه کند، اما وقتی OpenClaw یک پروفایل OAuth محلی دارد،
  توکن نوسازی محلی مرجع است؛ یکپارچه‌سازی‌های دیگر می‌توانند همچنان
  به‌صورت خارجی مدیریت شوند و فروشگاه احراز هویت CLI خود را دوباره بخوانند
- مسیرهای وضعیت و راه‌اندازی که از قبل مجموعه ارائه‌دهندگان پیکربندی‌شده را می‌شناسند،
  کشف CLI خارجی را به همان مجموعه محدود می‌کنند، تا فروشگاه ورود CLI نامرتبط
  برای یک راه‌اندازی تک‌ارائه‌دهنده بررسی نشود

## ذخیره‌سازی (توکن‌ها کجا قرار دارند)

رازها در فروشگاه‌های احراز هویت عامل ذخیره می‌شوند:

- پروفایل‌های احراز هویت (OAuth + کلیدهای API + ارجاع‌های اختیاری در سطح مقدار): `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- فایل سازگاری قدیمی: `~/.openclaw/agents/<agentId>/agent/auth.json`
  (ورودی‌های ثابت `api_key` هنگام شناسایی پاک‌سازی می‌شوند)

فایل قدیمی فقط برای واردسازی (هنوز پشتیبانی می‌شود، اما فروشگاه اصلی نیست):

- `~/.openclaw/credentials/oauth.json` (در اولین استفاده به `auth-profiles.json` وارد می‌شود)

همه موارد بالا از `$OPENCLAW_STATE_DIR` نیز پیروی می‌کنند (بازنویسی دایرکتوری وضعیت). مرجع کامل: [/gateway/configuration](/fa/gateway/configuration-reference#auth-storage)

برای ارجاع‌های راز ثابت و رفتار فعال‌سازی snapshot زمان اجرا، [مدیریت رازها](/fa/gateway/secrets) را ببینید.

وقتی یک عامل ثانویه پروفایل احراز هویت محلی ندارد، OpenClaw از وراثت خوانشی
از فروشگاه عامل پیش‌فرض/اصلی استفاده می‌کند. هنگام خواندن، `auth-profiles.json` عامل اصلی را
کپی نمی‌کند. توکن‌های نوسازی OAuth به‌ویژه حساس‌اند: جریان‌های کپی عادی به‌طور پیش‌فرض
از آن‌ها صرف‌نظر می‌کنند، چون برخی ارائه‌دهندگان پس از استفاده، توکن‌های نوسازی را چرخش می‌دهند
یا نامعتبر می‌کنند. وقتی یک عامل به حساب مستقل نیاز دارد، برای آن
یک ورود OAuth جداگانه پیکربندی کنید.

## مهاجرت Anthropic Claude CLI

OpenClaw دوباره از استفاده مجدد از Anthropic Claude CLI پشتیبانی می‌کند. اگر از قبل روی میزبان
یک ورود محلی Claude دارید، onboarding/configure می‌تواند مستقیما از آن استفاده مجدد کند.

## تبادل OAuth (ورود چگونه کار می‌کند)

جریان‌های ورود تعاملی OpenClaw در `@mariozechner/pi-ai` پیاده‌سازی شده‌اند و به ویزاردها/دستورها متصل شده‌اند.

### setup-token مربوط به Anthropic

شکل جریان:

1. setup-token مربوط به Anthropic یا paste-token را از OpenClaw شروع کنید
2. OpenClaw اعتبارنامه Anthropic حاصل را در یک پروفایل احراز هویت ذخیره می‌کند
3. انتخاب مدل روی `anthropic/...` باقی می‌ماند
4. پروفایل‌های احراز هویت Anthropic موجود برای بازگشت/کنترل ترتیب همچنان در دسترس می‌مانند

### OpenAI Codex (ChatGPT OAuth)

OpenAI Codex OAuth به‌صراحت برای استفاده خارج از Codex CLI، از جمله جریان‌های کاری OpenClaw، پشتیبانی می‌شود.

شکل جریان (PKCE):

1. verifier/challenge مربوط به PKCE + `state` تصادفی را تولید کنید
2. `https://auth.openai.com/oauth/authorize?...` را باز کنید
3. تلاش کنید callback را روی `http://127.0.0.1:1455/auth/callback` دریافت کنید
4. اگر callback نتواند bind شود (یا شما remote/headless هستید)، URL/کد redirect را جای‌گذاری کنید
5. در `https://auth.openai.com/oauth/token` تبادل کنید
6. `accountId` را از توکن دسترسی استخراج کنید و `{ access, refresh, expires, accountId }` را ذخیره کنید

مسیر ویزارد `openclaw onboard` → انتخاب احراز هویت `openai-codex` است.

## نوسازی + انقضا

پروفایل‌ها یک timestamp به نام `expires` ذخیره می‌کنند.

در زمان اجرا:

- اگر `expires` در آینده باشد → از توکن دسترسی ذخیره‌شده استفاده کنید
- اگر منقضی شده باشد → نوسازی کنید (زیر file lock) و اعتبارنامه‌های ذخیره‌شده را بازنویسی کنید
- اگر یک عامل ثانویه یک پروفایل OAuth به‌ارث‌رسیده از عامل اصلی را بخواند، نوسازی
  به‌جای کپی کردن توکن نوسازی در فروشگاه عامل ثانویه،
  در فروشگاه عامل اصلی نوشته می‌شود
- استثنا: برخی اعتبارنامه‌های CLI خارجی به‌صورت خارجی مدیریت‌شده باقی می‌مانند؛ OpenClaw
  به‌جای مصرف توکن‌های نوسازی کپی‌شده، آن فروشگاه‌های احراز هویت CLI را دوباره می‌خواند.
  راه‌اندازی اولیه Codex CLI عمدا محدودتر است: یک پروفایل خالی
  `openai-codex:default` ایجاد می‌کند، سپس نوسازی‌های تحت مالکیت OpenClaw پروفایل محلی را
  مرجع نگه می‌دارند.

جریان نوسازی خودکار است؛ عموما لازم نیست توکن‌ها را دستی مدیریت کنید.

## چندین حساب (پروفایل‌ها) + مسیریابی

دو الگو:

### 1) ترجیحی: عامل‌های جداگانه

اگر می‌خواهید «شخصی» و «کاری» هرگز با هم تعامل نداشته باشند، از عامل‌های ایزوله استفاده کنید (نشست‌ها + اعتبارنامه‌ها + فضای کاری جداگانه):

```bash
openclaw agents add work
openclaw agents add personal
```

سپس احراز هویت را برای هر عامل (ویزارد) پیکربندی کنید و گفتگوها را به عامل درست مسیریابی کنید.

### 2) پیشرفته: چندین پروفایل در یک عامل

`auth-profiles.json` از چندین شناسه پروفایل برای همان ارائه‌دهنده پشتیبانی می‌کند.

انتخاب کنید کدام پروفایل استفاده شود:

- به‌صورت سراسری از طریق ترتیب پیکربندی (`auth.order`)
- برای هر نشست از طریق `/model ...@<profileId>`

مثال (بازنویسی نشست):

- `/model Opus@anthropic:work`

چگونه ببینید چه شناسه‌های پروفایلی وجود دارند:

- `openclaw channels list --json` (`auth[]` را نشان می‌دهد)
