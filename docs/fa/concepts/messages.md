OpenClaw پیام‌های ورودی را از طریق خط لوله‌ای شامل تعیین جلسه، صف‌گذاری، جریان‌دهی، اجرای ابزار و نمایش‌پذیری استدلال مدیریت می‌کند. این صفحه مسیر پیام ورودی تا پاسخ را ترسیم می‌کند.

## جریان پیام (سطح بالا)

```
Inbound message
  -> routing/bindings -> session key
  -> queue (if a run is active)
  -> agent run (streaming + tools)
  -> outbound replies (channel limits + chunking)
```

تنظیمات کلیدی در پیکربندی قرار دارند:

- `messages.*` برای پیشوندها، صف‌گذاری و رفتار گروه.
- `agents.defaults.*` برای پیش‌فرض‌های جریان‌دهی بلوکی و قطعه‌بندی.
- بازنویسی‌های کانال (`channels.whatsapp.*`، `channels.telegram.*`، و غیره) برای سقف‌ها و کلیدهای جریان‌دهی.

برای طرح‌واره کامل، [پیکربندی](/fa/gateway/configuration) را ببینید.

## حذف تکرار ورودی

کانال‌ها می‌توانند پس از اتصال دوباره، همان پیام را دوباره تحویل دهند. OpenClaw یک
کش کوتاه‌مدت با کلید کانال/حساب/همتا/جلسه/شناسه پیام نگه می‌دارد تا تحویل‌های
تکراری اجرای دوباره عامل را فعال نکنند.

## دیبانس ورودی

پیام‌های پیاپی و سریع از **همان فرستنده** می‌توانند از طریق `messages.inbound` در یک
نوبت واحد عامل دسته‌بندی شوند. دیبانس به ازای هر کانال + گفتگو محدود می‌شود
و از جدیدترین پیام برای رشته‌بندی/شناسه‌های پاسخ استفاده می‌کند.

پیکربندی (پیش‌فرض سراسری + بازنویسی‌های به‌ازای کانال):

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

نکات:

- دیبانس برای پیام‌های **فقط متنی** اعمال می‌شود؛ رسانه/پیوست‌ها بلافاصله تخلیه می‌شوند.
- دستورهای کنترلی دیبانس را دور می‌زنند تا مستقل باقی بمانند — **مگر** وقتی کانالی صراحتا ادغام پیام مستقیمِ هم‌فرستنده را فعال کند (مثلا [BlueBubbles `coalesceSameSenderDms`](/fa/channels/bluebubbles#coalescing-split-send-dms-command--url-in-one-composition))، که در آن دستورهای پیام مستقیم داخل پنجره دیبانس منتظر می‌مانند تا محتوای ارسالیِ تکه‌تکه بتواند به همان نوبت عامل بپیوندد.

## جلسه‌ها و دستگاه‌ها

جلسه‌ها متعلق به Gateway هستند، نه کلاینت‌ها.

- گفتگوهای مستقیم در کلید جلسه اصلی عامل ادغام می‌شوند.
- گروه‌ها/کانال‌ها کلیدهای جلسه خودشان را می‌گیرند.
- ذخیره‌گاه جلسه و رونوشت‌ها روی میزبان Gateway قرار دارند.

چند دستگاه/کانال می‌توانند به همان جلسه نگاشت شوند، اما تاریخچه به‌طور کامل
به همه کلاینت‌ها همگام‌سازی نمی‌شود. توصیه: برای گفتگوهای طولانی از یک دستگاه
اصلی استفاده کنید تا از واگرایی زمینه جلوگیری شود. رابط کاربری کنترل و TUI همیشه
رونوشت جلسه پشتیبانی‌شده توسط Gateway را نشان می‌دهند، پس منبع حقیقت هستند.

جزئیات: [مدیریت جلسه](/fa/concepts/session).

## فراداده نتیجه ابزار

`content` نتیجه ابزار، نتیجه قابل مشاهده برای مدل است. `details` نتیجه ابزار،
فراداده زمان اجرا برای رندر رابط کاربری، عیب‌یابی، تحویل رسانه و Pluginها است.

OpenClaw این مرز را صریح نگه می‌دارد:

- `toolResult.details` پیش از بازپخش ارائه‌دهنده و ورودی Compaction حذف می‌شود.
- رونوشت‌های جلسه پایدارشده فقط `details` محدودشده را نگه می‌دارند؛ فراداده بیش‌ازحد بزرگ
  با خلاصه‌ای فشرده جایگزین می‌شود که با `persistedDetailsTruncated: true` نشانه‌گذاری شده است.
- Pluginها و ابزارها باید متنی را که مدل باید بخواند در `content` بگذارند، نه فقط
  در `details`.

## بدنه‌های ورودی و زمینه تاریخچه

OpenClaw **بدنه پرامپت** را از **بدنه دستور** جدا می‌کند:

- `Body`: متن پرامپت ارسالی به عامل. این می‌تواند شامل پوش‌های کانال و
  پوشش‌های اختیاری تاریخچه باشد.
- `CommandBody`: متن خام کاربر برای تحلیل directive/دستور.
- `RawBody`: نام مستعار قدیمی برای `CommandBody` (برای سازگاری نگه داشته شده است).

وقتی کانالی تاریخچه ارائه می‌کند، از یک پوشش مشترک استفاده می‌کند:

- `[Chat messages since your last reply - for context]`
- `[Current message - respond to this]`

برای **گفتگوهای غیرمستقیم** (گروه‌ها/کانال‌ها/اتاق‌ها)، **بدنه پیام فعلی** با برچسب
فرستنده پیشوند می‌گیرد (همان سبکی که برای ورودی‌های تاریخچه استفاده می‌شود). این کار پیام‌های
بلادرنگ و صف‌شده/تاریخچه را در پرامپت عامل سازگار نگه می‌دارد.

بافرهای تاریخچه **فقط معلق** هستند: آن‌ها شامل پیام‌های گروهی می‌شوند که اجرای عامل را
فعال _نکرده‌اند_ (مثلا پیام‌های محدودشده به منشن) و پیام‌هایی را که
از قبل در رونوشت جلسه هستند **حذف** می‌کنند.

حذف directive فقط روی بخش **پیام فعلی** اعمال می‌شود تا تاریخچه
دست‌نخورده باقی بماند. کانال‌هایی که تاریخچه را می‌پوشانند باید `CommandBody` (یا
`RawBody`) را روی متن اصلی پیام تنظیم کنند و `Body` را به‌عنوان پرامپت ترکیبی نگه دارند.
بافرهای تاریخچه از طریق `messages.groupChat.historyLimit` (پیش‌فرض سراسری)
و بازنویسی‌های به‌ازای کانال مثل `channels.slack.historyLimit` یا
`channels.telegram.accounts.<id>.historyLimit` قابل پیکربندی هستند (برای غیرفعال‌سازی، `0` تنظیم کنید).

## صف‌گذاری و پیگیری‌ها

اگر اجرایی از قبل فعال باشد، پیام‌های ورودی می‌توانند صف‌گذاری شوند، به
اجرای فعلی هدایت شوند، یا برای یک نوبت پیگیری جمع‌آوری شوند.

- از طریق `messages.queue` (و `messages.queue.byChannel`) پیکربندی کنید.
- حالت پیش‌فرض `steer` است، با دیبانس پیگیری ۵۰۰ میلی‌ثانیه‌ای وقتی هدایت
  به تحویل پیگیری صف‌شده بازمی‌گردد.
- حالت‌ها: `steer`، `followup`، `collect`، `steer-backlog`، `interrupt`، و
  نام مستعار قدیمی `queue`.

جزئیات: [صف دستور](/fa/concepts/queue).

## مالکیت اجرای کانال

Pluginهای کانال می‌توانند ترتیب را حفظ کنند، ورودی را دیبانس کنند و پیش از ورود پیام
به صف جلسه، فشار برگشتی انتقال را اعمال کنند. آن‌ها نباید یک مهلت زمانی جداگانه
دور خود نوبت عامل تحمیل کنند. وقتی پیام به یک جلسه مسیریابی شد،
کار طولانی‌مدت توسط چرخه عمر جلسه، ابزار و زمان اجرا اداره می‌شود تا همه کانال‌ها
نوبت‌های کند را به‌طور سازگار گزارش کنند و بازیابی شوند.

## جریان‌دهی، قطعه‌بندی و دسته‌بندی

جریان‌دهی بلوکی، پاسخ‌های جزئی را همزمان با تولید بلوک‌های متنی توسط مدل ارسال می‌کند.
قطعه‌بندی محدودیت‌های متن کانال را رعایت می‌کند و از شکستن کدهای حصارگذاری‌شده جلوگیری می‌کند.

تنظیمات کلیدی:

- `agents.defaults.blockStreamingDefault` (`on|off`، پیش‌فرض خاموش)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (دسته‌بندی مبتنی بر بیکاری)
- `agents.defaults.humanDelay` (مکث شبیه انسان بین پاسخ‌های بلوکی)
- بازنویسی‌های کانال: `*.blockStreaming` و `*.blockStreamingCoalesce` (کانال‌های غیر Telegram به `*.blockStreaming: true` صریح نیاز دارند)

جزئیات: [جریان‌دهی + قطعه‌بندی](/fa/concepts/streaming).

## نمایش‌پذیری استدلال و توکن‌ها

OpenClaw می‌تواند استدلال مدل را آشکار یا پنهان کند:

- `/reasoning on|off|stream` نمایش‌پذیری را کنترل می‌کند.
- محتوای استدلال، وقتی توسط مدل تولید شود، همچنان در مصرف توکن حساب می‌شود.
- Telegram از جریان استدلال در حباب پیش‌نویس پشتیبانی می‌کند.

جزئیات: [directiveهای فکر کردن + استدلال](/fa/tools/thinking) و [مصرف توکن](/fa/reference/token-use).

## پیشوندها، رشته‌بندی و پاسخ‌ها

قالب‌بندی پیام خروجی در `messages` متمرکز شده است:

- `messages.responsePrefix`، `channels.<channel>.responsePrefix`، و `channels.<channel>.accounts.<id>.responsePrefix` (آبشار پیشوند خروجی)، به‌علاوه `channels.whatsapp.messagePrefix` (پیشوند ورودی WhatsApp)
- رشته‌بندی پاسخ از طریق `replyToMode` و پیش‌فرض‌های به‌ازای کانال

جزئیات: [پیکربندی](/fa/gateway/config-agents#messages) و مستندات کانال.

## پاسخ‌های بی‌صدا

توکن بی‌صدای دقیق `NO_REPLY` / `no_reply` یعنی «پاسخ قابل مشاهده برای کاربر تحویل نده».
وقتی یک نوبت رسانه ابزار معلق هم دارد، مثل صدای TTS تولیدشده، OpenClaw
متن بی‌صدا را حذف می‌کند اما همچنان پیوست رسانه را تحویل می‌دهد.
OpenClaw این رفتار را بر اساس نوع گفتگو حل می‌کند:

- گفتگوهای مستقیم به‌صورت پیش‌فرض سکوت را مجاز نمی‌دانند و یک پاسخ بی‌صدای تنها را
  به جایگزینی کوتاه و قابل مشاهده بازنویسی می‌کنند.
- گروه‌ها/کانال‌ها به‌صورت پیش‌فرض سکوت را مجاز می‌دانند.
- هماهنگ‌سازی داخلی به‌صورت پیش‌فرض سکوت را مجاز می‌داند.

OpenClaw همچنین از پاسخ‌های بی‌صدا برای شکست‌های runner داخلی استفاده می‌کند که
پیش از هر پاسخ دستیار در گفتگوهای غیرمستقیم رخ می‌دهند، تا گروه‌ها/کانال‌ها
متن کلیشه‌ای خطای Gateway را نبینند. گفتگوهای مستقیم به‌صورت پیش‌فرض متن خطای فشرده نشان می‌دهند؛
جزئیات خام runner فقط وقتی نشان داده می‌شوند که `/verbose` روی `on` یا `full` باشد.

پیش‌فرض‌ها زیر `agents.defaults.silentReply` و
`agents.defaults.silentReplyRewrite` قرار دارند؛ `surfaces.<id>.silentReply` و
`surfaces.<id>.silentReplyRewrite` می‌توانند آن‌ها را به‌ازای هر سطح بازنویسی کنند.

وقتی جلسه والد یک یا چند اجرای subagent ایجادشده معلق دارد، پاسخ‌های
بی‌صدای تنها به‌جای بازنویسی شدن، در همه سطح‌ها حذف می‌شوند تا
والد تا زمانی که رویداد تکمیل فرزند پاسخ واقعی را تحویل دهد، ساکت بماند.
