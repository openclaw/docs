پشتهٔ خصوصی QA، OpenClaw را به‌شکلی واقع‌گرایانه و مشابه کانال آزمایش می‌کند که
یک آزمون واحد قادر به انجام آن نیست.

اجزا:

- `extensions/qa-channel`: کانال پیام مصنوعی با سطوح پیام خصوصی، کانال، رشته،
  واکنش، ویرایش و حذف.
- `extensions/qa-lab`: رابط کاربری اشکال‌زدا، گذرگاه QA، پروفایل‌های سناریو و آداپتورهای زندهٔ
  انتقال برای مشاهدهٔ رونوشت، تزریق پیام‌های ورودی
  و صادرکردن گزارش Markdown.
- `qa/`: دارایی‌های اولیهٔ مبتنی بر مخزن برای وظیفهٔ آغازین و سناریوهای پایهٔ
  QA.
- [Mantis](/fa/concepts/mantis): راستی‌آزمایی زندهٔ قبل/بعد برای باگ‌هایی که
  به انتقال‌های واقعی، نماگرفت‌های مرورگر، وضعیت ماشین مجازی و شواهد PR نیاز دارند.

## سطح فرمان

هر جریان QA زیرمجموعهٔ `pnpm openclaw qa <subcommand>` اجرا می‌شود. بسیاری از آن‌ها نام‌های مستعار اسکریپتی
`pnpm qa:*` دارند؛ هر دو شکل کار می‌کنند.

### `qa run` مبتنی بر پروفایل

`qa run` مبتنی بر پروفایل، عضویت را از `taxonomy.yaml` می‌خواند و سپس سناریوهای
حل‌شده را از طریق `qa suite` ارسال می‌کند. `--surface` و `--category`
به‌جای تعریف مسیرهای جداگانه، پروفایل انتخاب‌شده را فیلتر می‌کنند. خروجی
`qa-evidence.json` شامل خلاصهٔ کارت امتیاز پروفایل با تعداد دسته‌های انتخاب‌شده
و شناسه‌های پوشش مفقود است؛ مدخل‌های شواهد منفرد همچنان
منبع حقیقت برای آزمون‌ها، نقش‌های پوشش و نتایج باقی می‌مانند. شناسه‌های پوشش ویژگی
رده‌بندی، اهداف دقیق اثبات هستند، نه نام مستعار: پوشش سناریوی اصلی
شناسه‌های منطبق را برآورده می‌کند و پوشش ثانویه جنبهٔ راهنما دارد. شناسه‌های پوشش از
شکل نقطه‌گذاری‌شدهٔ `namespace.behavior` با بخش‌های حروف کوچک الفبایی‌عددی/خط‌تیره استفاده می‌کنند؛
شناسه‌های پروفایل، سطح و دسته همچنان می‌توانند از شناسه‌های خط‌تیره‌دار یا نقطه‌گذاری‌شدهٔ
رده‌بندی موجود استفاده کنند.

شواهد کم‌حجم، `execution` هر مدخل را حذف و `evidenceMode: "slim"` را تنظیم می‌کنند؛
`smoke-ci` به‌طور پیش‌فرض کم‌حجم است و `--evidence-mode full` مدخل‌های کامل را بازیابی می‌کند:

```bash
pnpm openclaw qa run \
  --qa-profile smoke-ci \
  --category channel-framework.conversation-routing-and-delivery \
  --provider-mode mock-openai \
  --output-dir .artifacts/qa-e2e/smoke-ci-profile-dispatch
```

برای اثبات قطعی پروفایل با ارائه‌دهندگان مدل شبیه‌سازی‌شده و سرورهای ارائه‌دهندهٔ محلی
Crabline از `smoke-ci` استفاده کنید. برای اثبات Stable/LTS در برابر
کانال‌های زنده از `release` استفاده کنید. از `all` فقط برای اجرای صریح شواهد
رده‌بندی کامل استفاده کنید؛ این گزینه تمام دسته‌های بلوغ فعال را انتخاب می‌کند و می‌توان آن را از طریق گردش‌کار GitHub Actions به نام `QA
Profile Evidence` با `qa_profile=all` ارسال کرد. هنگامی که
یک فرمان به پروفایل ریشهٔ OpenClaw نیز نیاز دارد، پروفایل ریشه را پیش از
فرمان QA قرار دهید:

```bash
pnpm openclaw --profile work qa run --qa-profile smoke-ci
```

## جریان اپراتور

جریان کنونی اپراتور QA یک سایت QA دوپنجره‌ای است:

- چپ: داشبورد Gateway (رابط کاربری کنترل) همراه با عامل.
- راست: QA Lab، نمایش‌دهندهٔ رونوشت مشابه Slack و طرح سناریو.

آن را با فرمان زیر اجرا کنید:

```bash
pnpm qa:lab:up
```

این فرمان سایت QA را می‌سازد، مسیر Gateway مبتنی بر Docker را راه‌اندازی می‌کند و
صفحهٔ QA Lab را در دسترس قرار می‌دهد تا اپراتور یا حلقهٔ خودکارسازی بتواند یک مأموریت QA
به عامل بدهد، رفتار واقعی کانال را مشاهده کند و موارد موفق، ناموفق یا
مسدودمانده را ثبت کند.

برای تکرار سریع‌تر رابط کاربری QA Lab بدون بازسازی تصویر Docker در هر بار،
پشته را با بستهٔ QA Lab متصل‌شده از طریق bind mount راه‌اندازی کنید:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

`qa:lab:up:fast` سرویس‌های Docker را روی یک تصویر ازپیش‌ساخته نگه می‌دارد و
`extensions/qa-lab/web/dist` را با bind mount در کانتینر `qa-lab` متصل می‌کند.
`qa:lab:watch` آن بسته را هنگام تغییر بازسازی می‌کند و مرورگر زمانی که
هش دارایی QA Lab تغییر کند، به‌طور خودکار بازبارگذاری می‌شود.

### دودآزمایی‌های مشاهده‌پذیری

<Note>
QA مشاهده‌پذیری فقط در نسخهٔ استخراج‌شدهٔ کد منبع باقی می‌ماند. بستهٔ tar مربوط به npm عمداً
QA Lab (و `qa-channel`) را حذف می‌کند، بنابراین مسیرهای انتشار Docker بسته
فرمان‌های `qa` را اجرا نمی‌کنند. هنگام تغییر ابزارگذاری عیب‌یابی،
این فرمان‌ها را از یک نسخهٔ ساخته‌شدهٔ کد منبع اجرا کنید.
</Note>

| نام مستعار                                   | آنچه اجرا می‌کند                                                                                                                            |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| `pnpm qa:otel:smoke`                    | گیرنده محلی OpenTelemetry به‌همراه سناریوی `otel-trace-smoke` با `diagnostics-otel` فعال‌شده.                                      |
| `pnpm qa:otel:collector-smoke`          | همان مسیر پشت یک کانتینر Docker واقعی OpenTelemetry Collector. هنگام تغییر سیم‌کشی نقطه پایانی یا سازگاری collector/OTLP از آن استفاده کنید. |
| `pnpm qa:prometheus:smoke`              | سناریوی `docker-prometheus-smoke` با `diagnostics-prometheus` فعال‌شده.                                                           |
| `pnpm qa:observability:smoke`           | `qa:otel:smoke` و پس از آن `qa:prometheus:smoke`.                                                                                      |
| `pnpm qa:observability:collector-smoke` | `qa:otel:collector-smoke` و پس از آن `qa:prometheus:smoke`.                                                                            |

`qa:otel:smoke` یک گیرنده محلی OTLP/HTTP را راه‌اندازی می‌کند، یک نوبت حداقلی عامل
در کانال QA را اجرا می‌کند و سپس بررسی می‌کند که ردپاها، معیارها و گزارش‌ها صادر شده‌اند. این فرایند
بازه‌های ردپای protobuf صادرشده را رمزگشایی می‌کند و ساختار حیاتی برای انتشار را بررسی می‌کند:
`openclaw.run`، `openclaw.harness.run`، یک بازه فراخوانی مدل مطابق با جدیدترین قرارداد معنایی GenAI،
`openclaw.context.assembled` و `openclaw.message.delivery`
همگی باید موجود باشند. آزمون دود
`OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` را به‌اجبار اعمال می‌کند، بنابراین بازه فراخوانی
مدل باید از نام `{gen_ai.operation.name} {gen_ai.request.model}` استفاده کند؛ فراخوانی‌های مدل
در نوبت‌های موفق نباید `StreamAbandoned` را صادر کنند؛ شناسه‌های تشخیصی خام
و ویژگی‌های `openclaw.content.*` باید خارج از ردپا باقی بمانند. اعلان سناریو
از مدل می‌خواهد با یک نشانگر ثابت پاسخ دهد و یک رشته محرمانه ثابت را افشا نکند؛ محموله‌های خام
OTLP نباید شامل هیچ‌یک از آن‌ها یا کلید نشست QA مشتق‌شده از شناسه سناریو
باشند. این فرایند `otel-smoke-summary.json` را
در کنار مصنوعات مجموعه QA می‌نویسد.

`qa:prometheus:smoke` بررسی می‌کند که گردآوری‌های بدون احراز هویت رد شوند، سپس
بررسی می‌کند که گردآوری احرازشده شامل خانواده‌های معیار حیاتی برای انتشار باشد،
بدون محتوای اعلان، محتوای پاسخ، شناسه‌های تشخیصی خام، توکن‌های احراز هویت
یا مسیرهای محلی.

### مسیرهای آزمون دود Matrix

برای یک مسیر آزمون دود Matrix با انتقال واقعی که به اعتبارنامه‌های ارائه‌دهنده مدل
نیاز ندارد، نمایه انتشار را با ارائه‌دهنده آزمایشی قطعی OpenAI اجرا کنید:

```bash
pnpm openclaw qa matrix --provider-mode mock-openai --profile release
```

برای مسیر ارائه‌دهنده زنده پیشرو، اعتبارنامه‌های سازگار با OpenAI را
به‌صراحت ارائه کنید:

```bash
OPENCLAW_LIVE_OPENAI_KEY="${OPENAI_API_KEY}" \
  pnpm openclaw qa matrix --provider-mode live-frontier --profile release
```

اجرای ساده `pnpm openclaw qa matrix` کل نمایه `all` را اجرا می‌کند و پس از
شکست سناریوها ادامه می‌دهد. برای چرخه بازخورد کوتاه‌تر از `--fail-fast` استفاده کنید یا
`--scenario <id>` را برای انتخاب سناریوهای جداگانه تکرار کنید؛ شناسه‌های صریح سناریو بر
`--profile` اولویت دارند.

| نمایه      | سناریوها | هدف                                                                                                                                  |
| ------------ | --------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `all`        | 93        | کاتالوگ کامل (پیش‌فرض).                                                                                                              |
| `release`    | 2         | خط مبنای حیاتی کانال برای انتشار و بارگذاری مجدد زنده فهرست مجاز.                                                                             |
| `fast`       | 12        | پوشش متمرکز رشته‌ها، واکنش‌ها، تأییدها، خط‌مشی، دروازه‌بندی ربات و پاسخ رمزگذاری‌شده.                                               |
| `transport`  | 50        | رشته‌ها، مسیریابی پیام مستقیم/اتاق، پیوستن خودکار، تأییدها، واکنش‌ها، راه‌اندازی‌های مجدد، خط‌مشی اشاره/فهرست مجاز، ویرایش‌ها و ترتیب چندکنشگری.         |
| `media`      | 7         | پوشش تصویر، تصویر تولیدشده، صدا، پیوست، رسانه پشتیبانی‌نشده و رسانه رمزگذاری‌شده.                                              |
| `e2ee-smoke` | 8         | حداقل پوشش پاسخ رمزگذاری‌شده، رشته‌ها، راه‌اندازی اولیه، بازیابی، راه‌اندازی مجدد، حذف محتوا و شکست.                                       |
| `e2ee-deep`  | 18        | ازدست‌رفتن وضعیت، پشتیبان‌گیری، بازیابی کلید، بهداشت دستگاه و راستی‌آزمایی SAS/QR/پیام مستقیم.                                                            |
| `e2ee-cli`   | 9         | `openclaw matrix encryption setup`، کلید بازیابی، چندحسابی، رفت‌وبرگشت Gateway و فرمان‌های خودراستی‌آزمایی از طریق مهار آزمون. |

عضویت در نمایه و الزامات کانال همراه با سناریوهای اعلانی Matrix
در `qa/scenarios/channels/` قرار دارند. اجرای آزمون، راه‌انداز کانال را انتخاب می‌کند.
پیاده‌سازی‌های زنده آن‌ها در
`extensions/qa-lab/src/live-transports/matrix/scenarios/` قرار دارند.

آداپتور یک سرور خانگی یک‌بارمصرف Tuwunel را در Docker فراهم می‌کند (تصویر پیش‌فرض
`ghcr.io/matrix-construct/tuwunel:v1.5.1`، نام سرور `matrix-qa.test`،
درگاه `28008`)، کاربران موقت راه‌انداز، سامانه تحت آزمون و ناظر را ثبت می‌کند، اتاق‌های
لازم را مقداردهی اولیه می‌کند و مرز درخواست/پاسخ ویرایش‌شده را ثبت می‌کند. سپس
Plugin واقعی Matrix را درون یک Gateway فرزند QA محدود به همان انتقال
(بدون `qa-channel`) اجرا می‌کند و محیط را برمی‌چیند.

| پرچم                     | پیش‌فرض           | هدف                                                                              |
| ------------------------ | ----------------- | ------------------------------------------------------------------------------------ |
| `--profile <profile>`    | `all`             | یکی از نمایه‌های بالا را انتخاب می‌کند.                                                    |
| `--scenario <id>`        | -                 | یک سناریو را انتخاب می‌کند؛ قابل تکرار است.                                                     |
| `--fail-fast`            | خاموش               | پس از نخستین بررسی یا سناریوی ناموفق متوقف می‌شود.                                       |
| `--allow-failures`       | خاموش               | مصنوعات را می‌نویسد، بدون اینکه برای شکست سناریوها کد خروج ناموفق برگرداند.         |
| `--provider-mode <mode>` | `live-frontier`   | برای اعزام قطعی از `mock-openai` یا برای ارائه‌دهنده زنده از `live-frontier` استفاده می‌کند. |
| `--model <ref>`          | پیش‌فرض ارائه‌دهنده  | مرجع اصلی `provider/model` را تنظیم می‌کند.                                          |
| `--alt-model <ref>`      | پیش‌فرض ارائه‌دهنده  | مدل جایگزینی را تنظیم می‌کند که سناریوهای تعویض مدل از آن استفاده می‌کنند.                        |
| `--fast`                 | خاموش               | حالت سریع ارائه‌دهنده را در صورت پشتیبانی فعال می‌کند.                                           |
| `--output-dir <path>`    | تولیدشده         | پوشه گزارش را انتخاب می‌کند؛ مسیرهای نسبی نسبت به `--repo-root` رفع می‌شوند.           |
| `--repo-root <path>`     | پوشه جاری | از یک پوشه کاری خنثی اجرا می‌کند.                                                |
| `--sut-account <id>`     | `sut`             | شناسه حساب Matrix را در پیکربندی Gateway فرزند انتخاب می‌کند.                            |

QA مربوط به Matrix اعتبارنامه‌های مشترک Matrix را اجاره نمی‌کند: آداپتور کاربران
یک‌بارمصرف را به‌صورت محلی ایجاد می‌کند، بنابراین `--credential-source` یا
`--credential-role` را نمی‌پذیرد. تصویر سرور خانگی را با
`OPENCLAW_QA_MATRIX_TUWUNEL_IMAGE` بازنویسی کنید؛ بررسی‌های منفیِ نبود پاسخ را با
`OPENCLAW_QA_MATRIX_NO_REPLY_WINDOW_MS` تنظیم کنید (پیش‌فرض `8000`، محدودشده به مهلت زمانی
سناریوی فعال). فرمان تک‌اجرا معمولاً پس از تخلیه مصنوعات، خروج پاک را به‌اجبار انجام می‌دهد،
زیرا دسته‌های بومی رمزنگاری Matrix ممکن است پس از پاک‌سازی نیز زنده بمانند؛ `OPENCLAW_QA_MATRIX_DISABLE_FORCE_EXIT=1`
را فقط برای مهار آزمون مستقیمی تنظیم کنید که
نیاز دارد فرمان در عوض بازگردد.

هر اجرا مصنوعات معمول QA Lab را زیر پوشه خروجی انتخاب‌شده
می‌نویسد: `qa-suite-report.md`، `qa-suite-summary.json`، `qa-evidence.json`
و یک مانیفست ویرایش‌شده `matrix-harness-*/matrix-qa-harness.json`. اگر پاک‌سازی
شکست خورد، فرمان بازیابی `docker compose ... down --remove-orphans` چاپ‌شده را
اجرا کنید. در اجراگرهای کند، بازه نبود پاسخ را افزایش دهید؛ در CI سریع، یک بازه کوچک‌تر
می‌تواند بررسی‌های منفی را کوتاه‌تر کند.

سناریوها رفتار انتقالی را پوشش می‌دهند که آزمون‌های واحد نمی‌توانند به‌صورت سرتاسری
اثبات کنند: دروازه‌بندی اشاره، خط‌مشی‌های اجازه ربات، فهرست‌های مجاز، پاسخ‌های سطح بالا و رشته‌ای،
مسیریابی پیام مستقیم، مدیریت واکنش، جلوگیری از ویرایش ورودی، حذف تکرار بازپخش پس از راه‌اندازی مجدد،
بازیابی پس از وقفه سرور خانگی، تحویل فراداده تأیید،
مدیریت رسانه و جریان‌های راه‌اندازی اولیه/بازیابی/راستی‌آزمایی E2EE در Matrix. نمایه CLI
مربوط به E2EE همچنین `openclaw matrix encryption setup` و
فرمان‌های راستی‌آزمایی را از طریق همان سرور خانگی یک‌بارمصرف اجرا می‌کند و سپس
پاسخ‌های Gateway را بررسی می‌کند.

`matrix-room-block-streaming` و `subagent-thread-spawn` با
انتخاب صریح `--scenario` همچنان در دسترس‌اند، اما بیرون از نمایه پیش‌فرض `all` باقی می‌مانند.

CI از همان سطح فرمان در
`.github/workflows/qa-live-transports-convex.yml` استفاده می‌کند. اجراهای زمان‌بندی‌شده و انتشار
سناریوهای انتشار را اجرا می‌کنند. اعزام‌های دستی `matrix_profile=all`
نمایه‌های `transport`، `media`، `e2ee-smoke`، `e2ee-deep` و `e2ee-cli` را منشعب می‌کنند؛
اعزام‌های متمرکز، `fast`، `release` یا `transport` را در یک کار انتخاب می‌کنند.

### سناریوهای Mantis در Discord

Discord همچنین سناریوهای انتخابیِ مختص Mantis برای بازتولید اشکال دارد. از
`--scenario discord-status-reactions-tool-only` برای خط زمانی صریح واکنش وضعیت
یا از `--scenario discord-thread-reply-filepath-attachment`
برای ایجاد یک رشته واقعی Discord و راستی‌آزمایی اینکه `message.thread-reply`
یک پیوست `filePath` را حفظ می‌کند، استفاده کنید. این سناریوها بیرون از مسیر زنده پیش‌فرض
Discord باقی می‌مانند، زیرا به‌جای پوشش گسترده آزمون دود، کاوشگرهای بازتولید قبل/بعد هستند.
گردش‌کار Mantis برای پیوست رشته همچنین می‌تواند یک ویدیوی شاهد از Discord Web با کاربر واردشده
اضافه کند، هنگامی که
`MANTIS_DISCORD_VIEWER_CHROME_PROFILE_DIR` یا
`MANTIS_DISCORD_VIEWER_CHROME_PROFILE_TGZ_B64` در محیط QA
پیکربندی شده باشد. این نمایه مشاهده‌گر فقط برای ثبت بصری است؛ تصمیم قبولی/رد
همچنان از اوراکل REST مربوط به Discord می‌آید.

برای سایر مسیرهای آزمون دود با انتقال واقعی:

```bash
pnpm openclaw qa discord
pnpm openclaw qa slack
pnpm openclaw qa telegram
pnpm openclaw qa whatsapp
```

آن‌ها یک کانال واقعی ازپیش‌موجود را با دو ربات یا حساب (راه‌انداز +
سامانه تحت آزمون) هدف می‌گیرند. متغیرهای محیطی لازم، فهرست سناریوها، مصنوعات خروجی و مخزن
اعتبارنامه Convex برای آن چهار انتقال در
[مرجع QA برای Discord، Slack، Telegram و WhatsApp](#discord-slack-telegram-and-whatsapp-qa-reference)
در ادامه مستند شده‌اند.

### اجراگرهای دسکتاپ Slack و وظایف بصری Mantis

برای اجرای کامل ماشین مجازی دسکتاپ Slack با امکان نجات از طریق VNC، اجرا کنید:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --gateway-setup \
  --scenario slack-canary \
  --keep-lease
```

آن فرمان یک ماشین دسکتاپ/مرورگر Crabbox را اجاره می‌کند، مسیر زنده Slack را
درون VM اجرا می‌کند، Slack Web را در مرورگر VNC باز می‌کند، از دسکتاپ تصویربرداری می‌کند،
و `slack-qa/`، `slack-desktop-smoke.png` و
`slack-desktop-smoke.mp4` را (هنگامی که ضبط ویدئو در دسترس باشد) به
دایرکتوری مصنوعات Mantis بازمی‌گرداند. اجاره‌های دسکتاپ/مرورگر Crabbox ابزارهای تصویربرداری
و بسته‌های کمکی مرورگر/ساخت بومی را از ابتدا فراهم می‌کنند، بنابراین سناریو
فقط باید در اجاره‌های قدیمی‌تر گزینه‌های جایگزین را نصب کند. Mantis زمان‌بندی کل و
هر مرحله را در `mantis-slack-desktop-smoke-report.md` گزارش می‌کند تا اجراهای کند نشان دهند
که زمان صرف آماده‌سازی اجاره، دریافت اعتبارنامه‌ها، راه‌اندازی راه دور یا
کپی مصنوعات شده است. پس از ورود دستی به Slack Web
از طریق VNC، از `--lease-id <cbx_...>` دوباره استفاده کنید؛ اجاره‌های بازاستفاده‌شده همچنین کش فروشگاه pnpm در Crabbox را
گرم نگه می‌دارند. `--hydrate-mode source` پیش‌فرض از یک checkout منبع اعتبارسنجی می‌کند و
نصب/ساخت را درون VM اجرا می‌کند. از `--hydrate-mode prehydrated` فقط زمانی استفاده کنید که
فضای کاری راه دور بازاستفاده‌شده از قبل `node_modules` و یک `dist/` ساخته‌شده داشته باشد؛
آن حالت مرحله پرهزینه نصب/ساخت را رد می‌کند و وقتی
فضای کاری آماده نباشد، به‌صورت بسته شکست می‌خورد. با `--gateway-setup`،‏ Mantis یک
Gateway پایدار OpenClaw برای Slack را درون VM و روی درگاه `38973` در حال اجرا نگه می‌دارد؛ بدون آن،
فرمان مسیر معمول QA ربات‌به‌ربات Slack را اجرا می‌کند و پس از ثبت
مصنوعات خارج می‌شود.

برای اثبات رابط کاربری بومی تأیید Slack همراه با شواهد دسکتاپ، حالت
نقطه بازرسی تأیید Mantis را اجرا کنید:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --approval-checkpoints \
  --credential-source convex \
  --credential-role maintainer
```

این حالت با `--gateway-setup` ناسازگار است. سناریوهای
تأیید Slack را اجرا می‌کند، شناسه‌های سناریوی غیرتأییدی را رد می‌کند، در هر وضعیت تأیید
در انتظار و حل‌شده منتظر می‌ماند، پیام مشاهده‌شده Slack API را در
`approval-checkpoints/<scenario>-pending.png` و
`approval-checkpoints/<scenario>-resolved.png` رندر می‌کند، سپس اگر هر نقطه بازرسی،
شواهد پیام، اعلام وصول یا نماگرفت رندرشده مفقود یا
خالی باشد شکست می‌خورد. اجاره‌های سرد CI ممکن است همچنان ورود به Slack را در
`slack-desktop-smoke.png` نشان دهند؛ تصاویر نقطه بازرسی تأیید، اثبات بصری
این مسیر هستند.

اجرای پیش‌فرض نقطه بازرسی دو سناریوی استاندارد تأیید Slack را نگه می‌دارد.
برای ثبت هریک از مسیرهای تأیید اختیاری Codex، آن را صریحاً با
`--scenario slack-codex-approval-exec-native` یا
`--scenario slack-codex-approval-plugin-native` انتخاب کنید؛ Mantis هردو را می‌پذیرد و
همان جفت نماگرفت در انتظار/حل‌شده را تولید می‌کند. اجراکننده مهلت‌های نقطه بازرسی
و فرمان راه دور خود را برای هر مسیر Codex انتخاب‌شده افزایش می‌دهد تا دنباله کامل
تأیید، تکمیل عامل و به‌روزرسانی وضعیت حل‌شده بتواند پایان یابد.

چک‌لیست اپراتور، فرمان dispatch گردش‌کار GitHub، قرارداد نظرِ شواهد،
جدول تصمیم حالت hydrate، تفسیر زمان‌بندی و مراحل رسیدگی به شکست در
[راهنمای اجرای دسکتاپ Slack با Mantis](/fa/concepts/mantis-slack-desktop-runbook)
قرار دارند.

برای یک وظیفه دسکتاپ به سبک عامل/CV، اجرا کنید:

```bash
pnpm openclaw qa mantis visual-task \
  --browser-url https://example.net \
  --expect-text "Example Domain" \
  --vision-model openai/gpt-5.6-luna
```

`visual-task` یک ماشین دسکتاپ/مرورگر Crabbox را اجاره می‌کند یا دوباره به‌کار می‌گیرد،
`crabbox record --while` را راه‌اندازی می‌کند، مرورگر قابل‌مشاهده را از طریق یک
`visual-driver` تودرتو کنترل می‌کند، `visual-task.png` را ثبت می‌کند، وقتی `--vision-mode image-describe`
انتخاب شده باشد `openclaw infer image
describe` را روی نماگرفت اجرا می‌کند و `visual-task.mp4`،‏ `mantis-visual-task-summary.json`،
`mantis-visual-task-driver-result.json` و
`mantis-visual-task-report.md` را می‌نویسد. وقتی `--expect-text` تنظیم شده باشد، اعلان بینایی
یک نتیجه ساخت‌یافته JSON (`visible`،‏ `evidence`،‏ `reason`)
درخواست می‌کند و فقط هنگامی موفق می‌شود که مدل `visible: true` را همراه با شواهدی
گزارش کند که به متن مورد انتظار استناد می‌کنند؛ پاسخ `visible: false` که فقط
متن هدف را نقل کند همچنان در ارزیابی شکست می‌خورد. برای یک آزمون دود بدون مدل
که زیرساخت دسکتاپ، مرورگر، نماگرفت و ویدئو را بدون فراخوانی
یک ارائه‌دهنده درک تصویر اثبات می‌کند، از `--vision-mode metadata` استفاده کنید. ضبط برای `visual-task`
یک مصنوع الزامی است؛ اگر Crabbox هیچ `visual-task.mp4` غیرخالی ضبط نکند،
وظیفه حتی در صورت موفقیت راه‌انداز بصری شکست می‌خورد. در
صورت شکست، Mantis اجاره را برای VNC نگه می‌دارد، مگر اینکه وظیفه از قبل موفق شده
و `--keep-lease` تنظیم نشده باشد.

### بررسی سلامت مخزن اعتبارنامه‌ها

پیش از استفاده از اعتبارنامه‌های زنده اشتراکی، اجرا کنید:

```bash
pnpm openclaw qa credentials doctor
```

doctor متغیرهای محیطی واسط Convex (`OPENCLAW_QA_CONVEX_SITE_URL`،
`OPENCLAW_QA_CONVEX_ENDPOINT_PREFIX`) را بررسی می‌کند، تنظیمات نقطه پایانی را اعتبارسنجی می‌کند،
برای `OPENCLAW_QA_CONVEX_SECRET_CI` و
`OPENCLAW_QA_CONVEX_SECRET_MAINTAINER` فقط وضعیت تنظیم‌شده/مفقود را گزارش می‌کند و
هنگامی که راز نگه‌دارنده موجود باشد، دسترسی‌پذیری admin/list را تأیید می‌کند.

## پوشش سناریوی مرجع

`taxonomy.yaml` ریشه شناسه‌های پوشش معنایی را تعریف می‌کند. فایل‌های YAML سناریو
در `qa/scenarios/` هر سناریو را به آن شناسه‌ها نگاشت می‌کنند و مالک فراداده
اجرا هستند: `channel` تنها الزام کانال است و `profiles` عضویت
نام‌گذاری‌شده در اجرا را اعلام می‌کنند. راه‌انداز کانال یک انتخاب پیاده‌سازی قابل‌تعویض
در سطح اجرا است. اجراکننده‌های TypeScript
آن کاتالوگ را پرس‌وجو می‌کنند؛ آن‌ها فهرست‌های موازی سناریو یا پوشش
را نگه نمی‌دارند.

خروجی ایستای `qa coverage` نگاشت طبقه‌بندی به سناریو را گزارش می‌کند. اثبات
واقعی از `qa-evidence.json` به دست می‌آید که سناریوی اجراشده،
شناسه‌های پوشش، کانال، راه‌انداز واقعاً استفاده‌شده و نتیجه را ثبت می‌کند. کانال و راه‌انداز
ابعاد گزارش هستند، نه واژگان اضافی شناسه پوشش یا محورهای
واجد شرایط بودن سناریو.

برای یک مسیر VM یک‌بارمصرف Linux بدون واردکردن Docker به مسیر QA، اجرا کنید:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

این فرمان یک مهمان تازه Multipass را راه‌اندازی می‌کند، وابستگی‌ها را نصب می‌کند، OpenClaw را
درون مهمان می‌سازد، `qa suite` را اجرا می‌کند، سپس گزارش و
خلاصه معمول QA را به `.artifacts/qa-e2e/...` روی میزبان بازمی‌گرداند. این مسیر از همان
رفتار انتخاب سناریوی `qa suite` روی میزبان استفاده می‌کند.

اجراهای مجموعه روی میزبان و Multipass به‌طور پیش‌فرض چند سناریوی انتخاب‌شده را
به‌موازات و با workerهای Gateway ایزوله اجرا می‌کنند. `qa-channel` به‌طور پیش‌فرض
هم‌زمانی 4 دارد که به تعداد سناریوهای انتخاب‌شده محدود می‌شود. برای تنظیم تعداد workerها از `--concurrency
<count>` یا برای اجرای سریالی از `--concurrency 1` استفاده کنید.
برای اجرای بسته معیارسنجی دستیار شخصی (10
سناریو) از `--pack personal-agent` استفاده کنید. انتخابگر بسته با پرچم‌های تکراری `--scenario` جمع‌پذیر است:
ابتدا سناریوهای صریح اجرا می‌شوند، سپس سناریوهای بسته به ترتیب بسته و با
حذف موارد تکراری اجرا می‌شوند. هنگامی که یک اجراکننده سفارشی QA از قبل راه‌اندازی گردآورنده OpenTelemetry را فراهم می‌کند،
برای انتخاب هم‌زمان سناریوهای
`otel-trace-smoke` و `docker-prometheus-smoke` از `--pack observability` استفاده کنید.

اگر هر سناریویی شکست بخورد، فرمان با وضعیت غیرصفر خارج می‌شود. هنگامی که مصنوعات را بدون
کد خروج شکست‌خورده می‌خواهید، از `--allow-failures` استفاده کنید.

اجراهای زنده ورودی‌های احراز هویت QA پشتیبانی‌شده‌ای را که برای
مهمان عملی هستند منتقل می‌کنند: کلیدهای ارائه‌دهنده مبتنی بر محیط، مسیر پیکربندی ارائه‌دهنده زنده QA و
`CODEX_HOME` در صورت وجود. `--output-dir` را زیر ریشه مخزن نگه دارید تا
مهمان بتواند از طریق فضای کاری سوارشده در آن بنویسد.

## مرجع QA برای Discord،‏ Slack،‏ Telegram و WhatsApp

آداپتور Matrix از مسیر یک‌بارمصرف مبتنی بر Docker که در بالا مستند شده استفاده می‌کند.
Discord،‏ Slack،‏ Telegram و WhatsApp در برابر
انتقال‌های واقعی ازپیش‌موجود اجرا می‌شوند، بنابراین مرجع آن‌ها در اینجا قرار دارد.

### پرچم‌های مشترک CLI

این مسیرها از طریق
`extensions/qa-lab/src/live-transports/shared/live-transport-cli.ts` ثبت می‌شوند و
پرچم‌های یکسانی را می‌پذیرند:

| پرچم                                  | پیش‌فرض                                            | توضیحات                                                                                                                                     |
| ------------------------------------- | -------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `--scenario <id>`                     | -                                                  | فقط این سناریو را اجرا کنید. قابل تکرار است.                                                                                                             |
| `--output-dir <path>`                 | `<repo>/.artifacts/qa-e2e/<transport>-<timestamp>` | محل نوشتن گزارش‌ها، خلاصه‌ها، شواهد، مصنوعات خاص انتقال و گزارش خروجی. مسیرهای نسبی بر مبنای `--repo-root` حل می‌شوند. |
| `--repo-root <path>`                  | `process.cwd()`                                    | ریشه مخزن هنگام فراخوانی از یک cwd خنثی.                                                                                               |
| `--sut-account <id>`                  | `sut`                                              | شناسه حساب موقت درون پیکربندی Gateway مربوط به QA.                                                                                              |
| `--provider-mode <mode>`              | `live-frontier`                                    | `mock-openai`،‏ `aimock` یا `live-frontier`.                                                                                                    |
| `--model <ref>` / `--alt-model <ref>` | پیش‌فرض ارائه‌دهنده                                   | ارجاع‌های مدل اصلی/جایگزین.                                                                                                                   |
| `--fast`                              | خاموش                                                | حالت سریع ارائه‌دهنده در صورت پشتیبانی.                                                                                                             |
| `--credential-source <env\|convex>`   | `env`                                              | [مخزن اعتبارنامه Convex](#convex-credential-pool) را ببینید.                                                                                          |
| `--credential-role <maintainer\|ci>`  | `ci` در CI، در غیر این صورت `maintainer`                 | نقشی که هنگام `--credential-source convex` استفاده می‌شود.                                                                                                    |
| `--allow-failures`                    | خاموش                                                | هنگامی که سناریوها شکست می‌خورند، مصنوعات را بدون بازگرداندن کد خروج شکست‌خورده بنویسید.                                                                      |

هر مسیر در صورت شکست هر سناریویی با وضعیت غیرصفر خارج می‌شود. `--allow-failures`
مصنوعات را بدون تنظیم کد خروج شکست‌خورده می‌نویسد. Telegram همچنین
`--list-scenarios` را برای چاپ شناسه‌های سناریوی موجود و خروج می‌پذیرد؛ مسیرهای دیگر
این پرچم را ارائه نمی‌کنند.

### QA برای Telegram

```bash
pnpm openclaw qa telegram
```

یک گروه خصوصی واقعی Telegram با دو ربات متمایز (راه‌انداز +
سامانه تحت آزمون) را هدف قرار می‌دهد. ربات سامانه تحت آزمون باید نام کاربری Telegram داشته باشد؛ مشاهده ربات‌به‌ربات
هنگامی بهترین عملکرد را دارد که **Bot-to-Bot Communication Mode** برای هر دو ربات در
`@BotFather` فعال باشد.

متغیرهای محیطی لازم هنگامی که `--credential-source env`:

- `OPENCLAW_QA_TELEGRAM_GROUP_ID` - شناسه عددی گفت‌وگو (رشته).
- `OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN`
- `OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`

پروفایل `release` سناریوهای نگه‌داری‌شده YAML برای Telegram را انتخاب می‌کند؛ `all`
بررسی‌های اختیاری فشار برای نشست، میزان استفاده، زنجیره پاسخ و جریان را اضافه می‌کند. مقادیر صریح
`--scenario` پروفایل را بازنویسی می‌کنند.

- `channel-canary`
- `channel-mention-gating`
- `telegram-help-command`
- `telegram-commands-command`
- `telegram-tools-compact-command`
- `telegram-whoami-command`
- `telegram-status-command`
- `telegram-repeated-command-authorization`
- `telegram-other-bot-command-gating`
- `telegram-context-command`
- `telegram-current-session-status-tool`
- `telegram-tool-only-usage-footer`
- `telegram-reply-chain-exact-marker`
- `telegram-stream-final-single-message`
- `telegram-long-final-reuses-preview`
- `telegram-long-final-three-chunks`

نمایهٔ `release` همیشه canary، دروازه‌بندی اشاره، پاسخ‌های فرمان بومی،
آدرس‌دهی فرمان و پاسخ‌های گروهی بات‌به‌بات را پوشش می‌دهد. `mock-openai`
بررسی قطعی پیش‌نمایش نهایی طولانی را نیز شامل می‌شود.
`telegram-current-session-status-tool` و
`telegram-tool-only-usage-footer` همچنان اختیاری هستند: اولی فقط
هنگامی پایدار است که مستقیماً پس از canary در رشته قرار گیرد، و دومی اثباتی با Telegram واقعی
برای پاورقی `/usage` در پاسخ‌های فقط‌ابزار است. برای چاپ تفکیک فعلی
پیش‌فرض/اختیاری همراه با ارجاع‌های رگرسیون، از `pnpm openclaw qa telegram
--list-scenarios --provider-mode mock-openai` استفاده کنید. برای هر
سناریوی آداپتور زندهٔ Telegram از `--profile all` استفاده کنید.

مصنوعات خروجی:

- `qa-suite-report.md`
- `qa-suite-summary.json`
- `qa-evidence.json` - ورودی‌های شواهد برای بررسی‌های انتقال زنده،
  شامل فیلدهای نمایه، پوشش، ارائه‌دهنده، کانال، مصنوعات، نتیجه و RTT.

اجراهای بستهٔ Telegram از همان قرارداد اعتبارنامهٔ Telegram استفاده می‌کنند. اندازه‌گیری
تکراری RTT بخشی از مسیر عادی زندهٔ Telegram بسته است؛ توزیع RTT
برای بررسی RTT انتخاب‌شده، در `qa-evidence.json` زیر `result.timing`
ادغام می‌شود.

```bash
OPENCLAW_QA_CREDENTIAL_SOURCE=convex \
pnpm test:docker:npm-telegram-live
```

وقتی `OPENCLAW_QA_CREDENTIAL_SOURCE=convex` تنظیم شده باشد، پوشش‌دهندهٔ زندهٔ بسته
یک اعتبارنامهٔ `kind: "telegram"` را اجاره می‌کند، متغیرهای محیطی گروه/راه‌انداز/بات SUT
اجاره‌شده را به اجرای بستهٔ نصب‌شده صادر می‌کند، Heartbeat اجاره را برقرار نگه می‌دارد و هنگام
خاموش‌شدن آن را آزاد می‌کند. پوشش‌دهندهٔ بسته به‌طور پیش‌فرض 20 بررسی RTT از
`channel-canary`، مهلت RTT برابر با 30s و در صورت انتخاب Convex، نقش Convex
با مقدار `maintainer` را خارج از CI استفاده می‌کند. برای تنظیم اندازه‌گیری RTT
بدون ایجاد فرمان RTT جداگانه یا قالب خلاصهٔ ویژهٔ Telegram، مقادیر
`OPENCLAW_NPM_TELEGRAM_RTT_SAMPLES`، `OPENCLAW_NPM_TELEGRAM_RTT_TIMEOUT_MS`
یا `OPENCLAW_NPM_TELEGRAM_RTT_MAX_FAILURES` را بازنویسی کنید.

### QA در Discord

```bash
pnpm openclaw qa discord
```

یک کانال واقعی و خصوصی در یک سرور Discord را با دو بات هدف می‌گیرد: یک بات راه‌انداز
که توسط مهار آزمون کنترل می‌شود و یک بات SUT که Gateway فرزند OpenClaw
آن را از طریق Plugin همراه Discord راه‌اندازی می‌کند. مدیریت اشاره در کانال،
ثبت‌شدن فرمان بومی `/help` توسط بات SUT در Discord و
سناریوهای شواهد اختیاری Mantis را تأیید می‌کند.

محیط لازم هنگام `--credential-source env`:

- `OPENCLAW_QA_DISCORD_GUILD_ID`
- `OPENCLAW_QA_DISCORD_CHANNEL_ID`
- `OPENCLAW_QA_DISCORD_DRIVER_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_SUT_BOT_TOKEN`
- `OPENCLAW_QA_DISCORD_SUT_APPLICATION_ID` - باید با شناسهٔ کاربر بات SUT که
  Discord برمی‌گرداند مطابقت داشته باشد (در غیر این صورت مسیر فوراً ناموفق می‌شود).

اختیاری:

- `OPENCLAW_QA_DISCORD_VOICE_CHANNEL_ID` کانال صوتی/Stage را برای
  `discord-voice-autojoin` انتخاب می‌کند؛ بدون آن، سناریو نخستین کانال
  صوتی/Stage قابل‌مشاهده برای بات SUT را انتخاب می‌کند.

سناریوهای ماژول YAML در Discord ‏(`qa/scenarios/channels/discord-*.yaml`):

- `discord-canary`
- `discord-mention-gating`
- `discord-native-help-command-registration`
- `discord-voice-autojoin` - سناریوی صوتی اختیاری. به‌تنهایی اجرا می‌شود،
  `channels.discord.voice.autoJoin` را فعال می‌کند و تأیید می‌کند که وضعیت صوتی فعلی
  بات SUT در Discord همان کانال صوتی/Stage هدف است. اعتبارنامه‌های Discord در Convex
  ممکن است `voiceChannelId` اختیاری را شامل شوند؛ در غیر این صورت آداپتور اجراکننده
  نخستین کانال صوتی/Stage قابل‌مشاهده را در سرور پیدا می‌کند.
- `discord-status-reactions-tool-only` - سناریوی اختیاری Mantis. به‌تنهایی
  اجرا می‌شود، زیرا SUT را با `messages.statusReactions.enabled=true` به پاسخ‌های
  همیشه‌فعال و فقط‌ابزار در سرور تغییر می‌دهد و سپس یک خط زمانی واکنش REST
  به‌همراه مصنوعات بصری HTML/PNG ثبت می‌کند. گزارش‌های قبل/بعد Mantis
  مصنوعات MP4 ارائه‌شده توسط سناریو را نیز با نام‌های `baseline.mp4`
  و `candidate.mp4` حفظ می‌کنند.
- `discord-thread-reply-filepath-attachment` - سناریوی اختیاری Mantis؛
  [سناریوهای Mantis در Discord](#discord-mantis-scenarios) را ببینید.

سناریوی پیوستن خودکار به صدای Discord را صریحاً اجرا کنید:

```bash
pnpm openclaw qa discord \
  --scenario discord-voice-autojoin \
  --provider-mode mock-openai
```

سناریوی واکنش وضعیت Mantis را صریحاً اجرا کنید:

```bash
pnpm openclaw qa discord \
  --scenario discord-status-reactions-tool-only \
  --provider-mode live-frontier \
  --model openai/gpt-5.6-luna \
  --alt-model openai/gpt-5.6-luna \
  --fast
```

مصنوعات خروجی:

- `qa-suite-report.md`
- `qa-suite-summary.json`
- `qa-evidence.json` - ورودی‌های شواهد برای بررسی‌های انتقال زنده.
- `discord-qa-reaction-timelines.json` و
  `discord-status-reactions-tool-only-timeline.png` هنگام اجرای سناریوی واکنش وضعیت.

### QA در Slack

```bash
pnpm openclaw qa slack
```

یک کانال واقعی و خصوصی Slack را با دو بات متمایز هدف می‌گیرد: یک بات راه‌انداز
که توسط مهار آزمون کنترل می‌شود و یک بات SUT که Gateway فرزند OpenClaw
آن را از طریق Plugin همراه Slack راه‌اندازی می‌کند.

محیط لازم هنگام `--credential-source env`:

- `OPENCLAW_QA_SLACK_CHANNEL_ID`
- `OPENCLAW_QA_SLACK_DRIVER_BOT_TOKEN`
- `OPENCLAW_QA_SLACK_SUT_BOT_TOKEN`
- `OPENCLAW_QA_SLACK_SUT_APP_TOKEN`

اختیاری:

- `OPENCLAW_QA_SLACK_APPROVAL_CHECKPOINT_DIR` نقاط بازرسی تأیید بصری
  را برای Mantis فعال می‌کند. آداپتور `<scenario>.pending.json` و
  `<scenario>.resolved.json` را می‌نویسد، سپس منتظر فایل‌های `.ack.json`
  منطبق می‌ماند.
- `OPENCLAW_QA_SLACK_APPROVAL_CHECKPOINT_TIMEOUT_MS` مهلت تأیید
  نقطهٔ بازرسی را بازنویسی می‌کند. مقدار پیش‌فرض `120000` است.

سناریوهای مرجع YAML که از طریق آداپتور زندهٔ Slack ارائه می‌شوند:

- `thread-follow-up`
- `thread-isolation`

سناریوهای ماژول YAML در Slack ‏(`qa/scenarios/channels/slack-*.yaml`):

- `slack-canary`
- `slack-mention-gating`
- `slack-allowlist-block`
- `slack-channel-disabled-warning` - کاوش اختیاری Slack واقعی که تأیید می‌کند
  یک کانال غیرفعالِ پیکربندی‌شده بدون پاسخ‌دادن، هشداری ساخت‌یافته منتشر می‌کند.
- `slack-top-level-reply-shape`
- `slack-restart-resume`
- `slack-progress-commentary-true`، `slack-progress-commentary-false`،
  `slack-progress-commentary-omitted` و
  `slack-progress-commentary-verbose-dedupe` - کاوش‌های اختیاری Slack واقعی برای
  کنترل‌های مستقل توضیح/پیشرفت ابزار، پیش‌فرض قدیمیِ کلید حذف‌شده
  و رفتار تحویل یکتا هنگامی که پیشرفت تفصیلی پایدار روشن است.
- `slack-reaction-glyph-native` - سناریوی اختیاری واکنش ابزار پیام زنده.
  به عامل دستور می‌دهد نویسهٔ دقیق `✅` را ارسال کند و تأیید می‌کند Slack
  مقدار `white_check_mark` را برای بات SUT روی پیام هدف ذخیره کرده است.
- `slack-chart-presentation-native` - سناریوی اختیاری نمودار قابل‌انتقال که
  بلوک بومی `data_visualization` و متن دسترس‌پذیر دقیق را تأیید می‌کند.
- `slack-table-presentation-native` - سناریوی اختیاری جدول قابل‌انتقال که
  بلوک بومی `data_table`، ردیف‌های دقیق و متن دسترس‌پذیر را تأیید می‌کند.
- `slack-table-invalid-blocks-fallback` - سناریوی اختیاری انتقال مستقیم
  که یک جدول خام با ساختار خوانا و بیش‌ازحد مجاز، شامل 101 ردیف داده
  به‌علاوهٔ سرآیند آن، را از مسیر ارسال عملیاتی Slack می‌فرستد،
  ثابت می‌کند خود Slack مقدار `invalid_blocks` را برمی‌گرداند
  و تأیید می‌کند جایگزین ذخیره‌شده با قالب‌بندی غیرفعال کامل است و هیچ
  بلوک دادهٔ بومی ندارد. جزئیات سناریو فقط شواهد امنِ کد خطا، تعداد و
  بولی را نگه می‌دارند.
- `slack-approval-exec-native` - سناریوی اختیاری تأیید اجرای بومی Slack.
  یک تأیید اجرا را از طریق Gateway درخواست می‌کند، تأیید می‌کند پیام Slack
  دکمه‌های تأیید بومی دارد، آن را تعیین‌تکلیف می‌کند و به‌روزرسانی تعیین‌تکلیف‌شدهٔ Slack
  را تأیید می‌کند.
- `slack-approval-plugin-native` - سناریوی اختیاری تأیید بومی Plugin در Slack.
  هدایت تأیید اجرا و Plugin را هم‌زمان فعال می‌کند تا رویدادهای Plugin
  توسط مسیریابی تأیید اجرا سرکوب نشوند، سپس همان مسیر رابط کاربری بومی
  در انتظار/تعیین‌تکلیف‌شدهٔ Slack را تأیید می‌کند.
- `slack-codex-approval-exec-native` - سناریوی اختیاری تأیید فرمان Codex Guardian.
  Plugin مربوط به Codex را در حالت Guardian فعال می‌کند، یک نوبت عامل Gateway
  با مبدأ Slack را از طریق مهار app-server مربوط به Codex مسیریابی می‌کند،
  منتظر اعلان تأیید بومی Plugin در Slack برای
  `openclaw-codex-app-server` می‌ماند، آن را تعیین‌تکلیف می‌کند و تأیید می‌کند نوبت Codex
  با نشانگرهای موردانتظار خروجی فرمان و دستیار پایان می‌یابد.
- `slack-codex-approval-plugin-native` - سناریوی اختیاری تأیید فایل Codex Guardian.
  از یک دستورالعمل `apply_patch` خارج از فضای کاری استفاده می‌کند تا Codex
  مسیر تأیید تغییر فایل app-server را منتشر کند، سپس همان مسیر تأیید بومی
  در انتظار/تعیین‌تکلیف‌شدهٔ Slack، نشانگر نهایی دستیار و محتوای دقیق فایل
  را پیش از پاک‌سازی تأیید می‌کند.

سناریوهای تأیید Codex به یک `openai/*` یا `codex/*` `--model`،
اعتبارنامه‌های عادی مدل زنده و احراز هویت Codex یا احراز هویت کلید API پذیرفته‌شده توسط Plugin مربوط به Codex نیاز دارند.
جزئیات سناریو، متد app-server مربوط به Codex، کلید مدل Codex انتخاب‌شده،
وضعیت نهایی نوبت Codex و تأیید نشانگر عملیات را در کنار فرادادهٔ
تأیید Slack که اطلاعات حساس آن حذف شده است، شامل می‌شوند.

مصنوعات خروجی:

- `qa-suite-report.md`
- `qa-suite-summary.json`
- `qa-evidence.json` - ورودی‌های شواهد برای بررسی‌های انتقال زنده.
- `approval-checkpoints/` - فقط هنگامی که Mantis
  مقدار `OPENCLAW_QA_SLACK_APPROVAL_CHECKPOINT_DIR` را تنظیم کند؛ شامل JSON نقطهٔ بازرسی،
  JSON تأیید و نماگرفت‌های در انتظار/تعیین‌تکلیف‌شده است.

#### راه‌اندازی فضای کاری Slack

این مسیر به دو برنامهٔ متمایز Slack در یک فضای کاری، به‌علاوهٔ کانالی که هر دو
بات عضو آن باشند، نیاز دارد:

- `channelId` - شناسهٔ `Cxxxxxxxxxx` کانالی که هر دو بات
  به آن دعوت شده‌اند. از یک کانال اختصاصی استفاده کنید؛ مسیر در هر اجرا پیام ارسال می‌کند.
- `driverBotToken` - توکن بات (`xoxb-...`) برنامهٔ **راه‌انداز**.
- `sutBotToken` - توکن بات (`xoxb-...`) برنامهٔ **SUT** که باید
  برنامه‌ای جدا از راه‌انداز در Slack باشد تا شناسهٔ کاربر بات آن متمایز باشد.
- `sutAppToken` - توکن سطح برنامه (`xapp-...`) برنامهٔ SUT با
  `connections:write` که توسط Socket Mode استفاده می‌شود تا برنامهٔ SUT بتواند رویدادها را دریافت کند.

استفاده از یک فضای کاری Slack اختصاص‌یافته به QA را بر استفادهٔ مجدد از فضای کاری
عملیاتی ترجیح دهید.

مانیفست SUT زیر عمداً نصب عملیاتی Plugin همراه Slack
‏(`extensions/slack/src/setup-shared.ts:12`) را به مجوزها و رویدادهای
پوشش‌داده‌شده توسط مجموعهٔ QA زندهٔ Slack محدود می‌کند. برای راه‌اندازی کانال عملیاتی
آن‌گونه که کاربران می‌بینند، [راه‌اندازی سریع کانال Slack](/fa/channels/slack#quick-setup)
را ببینید؛ جفت QA راه‌انداز/SUT عمداً جدا است، زیرا مسیر به دو شناسهٔ کاربر بات
متمایز در یک فضای کاری نیاز دارد.

**1. ایجاد برنامهٔ Driver**

به [api.slack.com/apps](https://api.slack.com/apps) بروید → _Create New App_ →
_From a manifest_ → فضای کاری QA را انتخاب کنید، مانیفست زیر را جای‌گذاری کنید،
سپس _Install to Workspace_ را انتخاب کنید:

```json
{
  "display_information": {
    "name": "OpenClaw QA Driver",
    "description": "بات راه‌انداز آزمون برای مسیر زندهٔ Slack در QA مربوط به OpenClaw"
  },
  "features": {
    "bot_user": {
      "display_name": "OpenClaw QA Driver",
      "always_online": true
    }
  },
  "oauth_config": {
    "scopes": {
      "bot": ["chat:write", "channels:history", "groups:history", "users:read"]
    }
  },
  "settings": {
    "socket_mode_enabled": false
  }
}
```

_Bot User OAuth Token_ ‏(`xoxb-...`) را کپی کنید - این مقدار
به `driverBotToken` تبدیل می‌شود. راه‌انداز فقط باید پیام ارسال کند و
خود را شناسایی کند؛ بدون رویداد و بدون Socket Mode.

**2. ایجاد برنامهٔ SUT**

_Create New App → From a manifest_ را در همان فضای کاری تکرار کنید. این برنامهٔ QA
عمداً از نسخه‌ای محدودتر از مانیفست عملیاتی Plugin همراه Slack
‏(`extensions/slack/src/setup-shared.ts:12`) استفاده می‌کند: دامنه‌ها
و رویدادهای واکنش حذف شده‌اند، زیرا مجموعهٔ QA زندهٔ Slack هنوز مدیریت
واکنش را پوشش نمی‌دهد.

```json
{
  "display_information": {
    "name": "OpenClaw QA SUT",
    "description": "رابط OpenClaw QA SUT برای OpenClaw"
  },
  "features": {
    "bot_user": {

    "app_home": {
      "home_tab_enabled": true,
      "messages_tab_enabled": true,

    }
  },
  "oauth_config": {
    "scopes": {
      "bot": [
        "app_mentions:read",
        "assistant:write",
        "channels:history",
        "channels:read",
        "chat:write",
        "commands",
        "emoji:read",
        "files:read",
        "files:write",
        "groups:history",
        "groups:read",
        "im:history",
        "im:read",
        "im:write",
        "mpim:history",
        "mpim:read",
        "mpim:write",
        "pins:read",
        "pins:write",
        "usergroups:read",
        "users:read"
```

پس از اینکه Slack برنامه را ایجاد کرد، در صفحه تنظیمات آن دو کار انجام دهید:

- _Install to Workspace_ → مقدار _Bot User OAuth Token_ را کپی کنید → این مقدار به
  `sutBotToken` تبدیل می‌شود.
- _Basic Information → App-Level Tokens → Generate Token and Scopes_ → محدوده
  `connections:write` را اضافه کنید → ذخیره کنید → مقدار `xapp-...` را کپی کنید → این
  مقدار به `sutAppToken` تبدیل می‌شود.

با فراخوانی `auth.test` برای هر توکن، بررسی کنید که دو ربات شناسه‌های کاربری متمایزی دارند.
زمان اجرا، درایور و SUT را با شناسه کاربری از هم متمایز می‌کند؛ استفاده مجدد از یک برنامه
برای هر دو، بلافاصله در دروازه‌گذاری اشاره شکست می‌خورد.

**3. کانال را ایجاد کنید**

در فضای کاری QA، یک کانال ایجاد کنید (برای مثال `#openclaw-qa`) و هر دو
ربات را از داخل کانال دعوت کنید:

```text
/invite @OpenClaw QA Driver
/invite @OpenClaw QA SUT
```

شناسه `Cxxxxxxxxxx` را از _channel info → About → Channel ID_ کپی کنید؛ این
مقدار به `channelId` تبدیل می‌شود. کانال عمومی کار می‌کند؛ اگر از کانال خصوصی استفاده کنید،
هر دو برنامه از قبل `groups:history` را دارند، بنابراین خواندن تاریخچه توسط چارچوب
همچنان موفق خواهد بود.

**4. اطلاعات ورود را ثبت کنید**

دو گزینه وجود دارد. برای اشکال‌زدایی روی یک دستگاه از متغیرهای محیطی استفاده کنید (چهار
متغیر `OPENCLAW_QA_SLACK_*` را تنظیم و `--credential-source env` را ارسال کنید)، یا
مخزن مشترک Convex را مقداردهی اولیه کنید تا CI و دیگر نگه‌دارندگان بتوانند آن‌ها را اجاره کنند.

برای مخزن Convex، چهار فیلد را در یک فایل JSON بنویسید:

```json
{
  "channelId": "Cxxxxxxxxxx",
  "driverBotToken": "xoxb-...",
  "sutBotToken": "xoxb-...",
  "sutAppToken": "xapp-..."
}
```

درحالی‌که `OPENCLAW_QA_CONVEX_SITE_URL` و `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER`
در پوسته شما صادر شده‌اند، ثبت و بررسی کنید:

```bash
pnpm openclaw qa credentials add \
  --kind slack \
  --payload-file slack-creds.json \
  --note "مقداردهی اولیه مخزن QA برای Slack"

pnpm openclaw qa credentials list --kind slack --status all --json
```

انتظار می‌رود `count: 1` و `status: "active"` وجود داشته باشند و فیلد `lease` وجود نداشته باشد.

**5. بررسی سرتاسری**

مسیر را به‌صورت محلی اجرا کنید تا تأیید شود هر دو ربات می‌توانند از طریق
واسط با یکدیگر ارتباط برقرار کنند:

```bash
pnpm openclaw qa slack \
  --credential-source convex \
  --credential-role maintainer \
  --output-dir .artifacts/qa-e2e/slack-local
```

یک اجرای موفق در زمانی بسیار کمتر از 30 ثانیه تکمیل می‌شود و `qa-suite-report.md`
هر دو `slack-canary` و `slack-mention-gating` را با وضعیت `pass` نشان می‌دهد. اگر
مسیر حدود ~90 ثانیه متوقف بماند و با `Convex credential pool exhausted
for kind "slack"` خارج شود، یا مخزن خالی است یا همه ردیف‌ها اجاره شده‌اند؛ `qa
credentials list --kind slack --status all --json` مشخص می‌کند کدام مورد رخ داده است.

### مخزن اطلاعات ورود Convex

مسیرهای Discord، Slack، Telegram و WhatsApp می‌توانند به‌جای خواندن
متغیرهای محیطی بالا، اطلاعات ورود را از یک مخزن مشترک Convex اجاره کنند.
`--credential-source convex` را ارسال کنید (یا `OPENCLAW_QA_CREDENTIAL_SOURCE=convex` را تنظیم کنید)؛
آزمایشگاه QA یک اجاره انحصاری دریافت می‌کند، در طول اجرا Heartbeat آن را حفظ می‌کند
و هنگام خاموش‌شدن آن را آزاد می‌کند. انواع مخزن عبارت‌اند از `"discord"`، `"slack"`،
`"telegram"` و `"whatsapp"`.

شکل بارهایی که واسط هنگام `admin/add` اعتبارسنجی می‌کند:

- Discord ‏(`kind: "discord"`): `{ guildId: string, channelId: string,
driverBotToken: string, sutBotToken: string, sutApplicationId: string }`.
- Telegram ‏(`kind: "telegram"`): `{ groupId: string, driverToken: string,
sutToken: string }` - مقدار `groupId` باید یک رشته عددی شناسه گفت‌وگو باشد.
- کاربر واقعی Telegram ‏(`kind: "telegram-user"`): `{ groupId: string, sutToken:
string, testerUserId: string, testerUsername: string, telegramApiId:
string, telegramApiHash: string, tdlibDatabaseEncryptionKey: string,
tdlibArchiveBase64: string, tdlibArchiveSha256: string,
desktopTdataArchiveBase64: string, desktopTdataArchiveSha256: string }` -
  فقط برای اثبات Mantis در Telegram Desktop. مسیرهای عمومی آزمایشگاه QA نباید
  این نوع را دریافت کنند.
- WhatsApp ‏(`kind: "whatsapp"`): `{ driverPhoneE164: string, sutPhoneE164:
string, driverAuthArchiveBase64: string, sutAuthArchiveBase64: string,
groupJid?: string }` - شماره‌های تلفن باید رشته‌های متمایز E.164 باشند.

گردش‌کار اثبات Mantis در Telegram Desktop یک اجاره انحصاری Convex از نوع
`telegram-user` را برای هر دو مورد درایور CLI مبتنی بر TDLib و شاهد Telegram Desktop
نگه می‌دارد و سپس پس از انتشار اثبات آن را آزاد می‌کند.

وقتی یک PR به تفاوت بصری قطعی نیاز دارد، Mantis می‌تواند همان پاسخ
مدل شبیه‌سازی‌شده را روی `main` و روی سر PR استفاده کند، درحالی‌که قالب‌بند Telegram یا
لایه تحویل تغییر می‌کند. پیش‌فرض‌های ضبط برای دیدگاه‌های PR تنظیم شده‌اند: کلاس استاندارد
Crabbox، ضبط دسکتاپ با 24fps، فایل GIF حرکتی با 24fps و عرض پیش‌نمایش
1920px. دیدگاه‌های قبل/بعد باید یک بسته تمیز منتشر کنند که
فقط شامل فایل‌های GIF موردنظر باشد.

مسیرهای Slack نیز می‌توانند از مخزن استفاده کنند. بررسی شکل بار Slack در حال حاضر
به‌جای واسط، در اجراکننده QA برای Slack قرار دارد؛ از `{ channelId: string,
driverBotToken: string, sutBotToken: string, sutAppToken: string }` با یک
شناسه کانال Slack مانند `Cxxxxxxxxxx` استفاده کنید. برای فراهم‌سازی برنامه
و محدوده‌ها، به [راه‌اندازی فضای کاری Slack](#setting-up-the-slack-workspace) مراجعه کنید.

متغیرهای محیطی عملیاتی و قرارداد نقطه پایانی واسط Convex در
[آزمایش → اطلاعات ورود مشترک Telegram از طریق Convex](/fa/help/testing#shared-telegram-credentials-via-convex-v1)
قرار دارند (نام این بخش پیش از مخزن چندکاناله انتخاب شده است؛ معناشناسی اجاره
میان انواع مختلف مشترک است).

## بذرهای پشتیبانی‌شده توسط مخزن

دارایی‌های بذر در `qa/` قرار دارند:

- `qa/scenarios/index.yaml`
- `qa/scenarios/<theme>/*.yaml`

این موارد عمداً در git قرار دارند تا طرح QA هم برای انسان‌ها و هم برای
عامل قابل‌مشاهده باشد.

`qa-lab` یک اجراکننده عمومی سناریوی YAML باقی می‌ماند. هر فایل YAML سناریو
منبع حقیقت برای یک اجرای آزمایش است و باید موارد زیر را تعریف کند:

- `title` سطح بالا
- فراداده `scenario`
- فراداده اختیاری دسته، قابلیت، مسیر و ریسک در `scenario`
- مراجع مستندات و کد در `scenario`
- نیازمندی‌های اختیاری Plugin در `scenario`
- وصله اختیاری پیکربندی Gateway در `scenario`
- `flow` اجرایی سطح بالا برای سناریوهای جریان، یا
  `scenario.execution.kind` / `scenario.execution.path` برای سناریوهای Vitest و
  Playwright

سطح زمان اجرای قابل‌استفادهٔ مجدد که زیربنای `flow` است، عمومی و
سراسری باقی می‌ماند. برای مثال، سناریوهای YAML می‌توانند کمک‌ابزارهای سمت انتقال
را با کمک‌ابزارهای سمت مرورگر ترکیب کنند که رابط کنترل تعبیه‌شده را از طریق
درز `browser.request` در Gateway هدایت می‌کنند، بدون آنکه اجراکننده‌ای ویژه اضافه شود.

فایل‌های سناریو باید بر اساس قابلیت محصول گروه‌بندی شوند، نه پوشهٔ درخت
منبع. هنگام جابه‌جایی فایل‌ها، شناسه‌های سناریو را ثابت نگه دارید؛ برای
ردیابی‌پذیری پیاده‌سازی از `docsRefs` و `codeRefs` استفاده کنید.

فهرست پایه باید به‌اندازه‌ای گسترده بماند که موارد زیر را پوشش دهد:

- پیام خصوصی و گفت‌وگوی کانال
- رفتار رشته
- چرخهٔ عمر کنش پیام
- فراخوانی‌های برگشتی Cron
- بازیابی حافظه
- تعویض مدل
- تحویل به زیرعامل
- خواندن مخزن و مستندات
- یک وظیفهٔ ساخت کوچک مانند Lobster Invaders

## مسیرهای شبیه‌ساز ارائه‌دهنده

`qa suite` دو مسیر شبیه‌ساز محلی ارائه‌دهنده دارد:

- `mock-openai` شبیه‌ساز آگاه از سناریوی OpenClaw است. این مسیر همچنان
  مسیر شبیه‌ساز قطعی پیش‌فرض برای تضمین کیفیت مبتنی بر مخزن و دروازه‌های هم‌ارزی است.
- `aimock` یک سرور ارائه‌دهنده مبتنی بر AIMock را برای پوشش آزمایشی
  پروتکل، داده‌های ثابت، ضبط/بازپخش و آشوب راه‌اندازی می‌کند. این مسیر افزایشی است و
  جایگزین توزیع‌کنندهٔ سناریوی `mock-openai` نمی‌شود.

پیاده‌سازی مسیر ارائه‌دهنده در `extensions/qa-lab/src/providers/` قرار دارد.
هر ارائه‌دهنده مالک پیش‌فرض‌های خود، راه‌اندازی سرور محلی، پیکربندی مدل Gateway،
نیازهای آماده‌سازی نمایهٔ احراز هویت و پرچم‌های قابلیت زنده/شبیه‌ساز است. کد مشترک
مجموعه و Gateway به‌جای شاخه‌زنی بر اساس نام ارائه‌دهندگان، از رجیستری ارائه‌دهنده
مسیریابی می‌شود.

## سازگارکننده‌های انتقال

`qa-lab` مالک یک درز انتقال عمومی برای سناریوهای تضمین کیفیت YAML است. `qa-channel`
پیش‌فرض مصنوعی است. `crabline` سرورهای محلی هم‌شکل با ارائه‌دهنده را راه‌اندازی می‌کند و
Pluginهای عادی کانال OpenClaw را در برابر آن‌ها اجرا می‌کند. `live` برای
اعتبارنامه‌های واقعی ارائه‌دهنده و کانال‌های خارجی رزرو شده است.

در سطح معماری، این تفکیک به‌صورت زیر است:

- `qa-lab` مالک اجرای عمومی سناریو، هم‌زمانی کارگرها، نوشتن
  مصنوعات و گزارش‌دهی است.
- سازگارکنندهٔ انتقال مالک پیکربندی Gateway، آمادگی، مشاهدهٔ ورودی و خروجی،
  کنش‌های انتقال و وضعیت نرمال‌شدهٔ انتقال است.
- فایل‌های سناریوی YAML در `qa/scenarios/` اجرای آزمون را تعریف می‌کنند؛ `qa-lab`
  سطح زمان اجرای قابل‌استفادهٔ مجدد برای اجرای آن‌ها را فراهم می‌کند.

### افزودن یک کانال

افزودن یک کانال به سامانهٔ تضمین کیفیت YAML به پیاده‌سازی کانال
به‌همراه یک بستهٔ سناریو نیاز دارد که قرارداد کانال را تمرین کند. برای پوشش
CI دود، سرور محلی ارائه‌دهندهٔ متناظر Crabline را اضافه کنید و آن را
از طریق درایور `crabline` در دسترس قرار دهید.

هنگامی که میزبان مشترک `qa-lab` می‌تواند مالک جریان باشد، یک ریشهٔ فرمان
سطح‌بالای تضمین کیفیت جدید اضافه نکنید.

`qa-lab` مالک سازوکارهای میزبان مشترک است:

- ریشهٔ فرمان `openclaw qa`
- راه‌اندازی و خاتمهٔ مجموعه
- هم‌زمانی کارگرها
- نوشتن مصنوعات
- تولید گزارش
- اجرای سناریو
- نام‌های مستعار سازگاری برای سناریوهای قدیمی‌تر `qa-channel`

Pluginهای اجراکننده مالک قرارداد انتقال هستند:

- نحوهٔ سوارشدن `openclaw qa <runner>` زیر ریشهٔ مشترک `qa`
- نحوهٔ پیکربندی Gateway برای آن انتقال
- نحوهٔ بررسی آمادگی
- نحوهٔ تزریق رویدادهای ورودی
- نحوهٔ مشاهدهٔ پیام‌های خروجی
- نحوهٔ ارائهٔ رونوشت‌ها و وضعیت نرمال‌شدهٔ انتقال
- نحوهٔ اجرای کنش‌های متکی بر انتقال
- نحوهٔ رسیدگی به بازنشانی یا پاک‌سازی مختص انتقال

حداقل معیار پذیرش برای یک کانال جدید:

1. `qa-lab` را به‌عنوان مالک ریشهٔ مشترک `qa` نگه دارید.
2. اجراکنندهٔ انتقال را روی درز میزبان مشترک `qa-lab` پیاده‌سازی کنید.
3. سازوکارهای مختص انتقال را درون Plugin اجراکننده یا مهار
   کانال نگه دارید.
4. اجراکننده را به‌صورت `openclaw qa <runner>` سوار کنید، نه اینکه یک
   فرمان ریشهٔ رقیب ثبت کنید. Pluginهای اجراکننده باید `qaRunners` را در
   `openclaw.plugin.json` اعلام کنند و یک آرایهٔ متناظر `qaRunnerCliRegistrations`
   را از `runtime-api.ts` صادر کنند. `runtime-api.ts` را سبک نگه دارید؛ اجرای
   تنبل CLI و اجراکننده باید پشت نقاط ورودی جداگانه باقی بماند. یک
   `adapterFactory` اختیاری، انتقال را بدون تغییر
   فهرست سناریوهای موجود فرمان در اختیار سناریوهای مشترک قرار می‌دهد.
5. سناریوهای YAML را در پوشه‌های موضوعی `qa/scenarios/`
   بنویسید یا سازگار کنید.
6. برای سناریوهای جدید از کمک‌ابزارهای عمومی سناریو استفاده کنید.
7. نام‌های مستعار سازگاری موجود را فعال نگه دارید، مگر آنکه مخزن در حال انجام
   یک مهاجرت عمدی باشد.

قاعدهٔ تصمیم‌گیری سخت‌گیرانه است:

- اگر رفتاری را بتوان یک‌بار در `qa-lab` بیان کرد، آن را در `qa-lab` قرار دهید.
- اگر رفتار به انتقال یک کانال وابسته است، آن را در Plugin اجراکننده
  یا مهار همان Plugin نگه دارید.
- اگر سناریویی به قابلیت جدیدی نیاز دارد که بیش از یک کانال می‌تواند از آن استفاده کند،
  به‌جای یک شاخهٔ مختص کانال در `suite.ts` یک کمک‌ابزار عمومی اضافه کنید.
- اگر رفتاری فقط برای یک انتقال معنادار است، سناریو را
  مختص انتقال نگه دارید و این موضوع را در قرارداد سناریو صریح بیان کنید.

### نام کمک‌ابزارهای سناریو

کمک‌ابزارهای عمومی ترجیحی برای سناریوهای جدید:

- `waitForTransportReady`
- `waitForChannelReady`
- `injectInboundMessage`
- `injectOutboundMessage`
- `waitForTransportOutboundMessage`
- `waitForChannelOutboundMessage`
- `waitForNoTransportOutbound`
- `getTransportSnapshot`
- `readTransportMessage`
- `readTransportTranscript`
- `formatTransportTranscript`
- `resetTransport`

نام‌های مستعار سازگاری برای سناریوهای موجود همچنان در دسترس‌اند -
`waitForQaChannelReady`، `waitForOutboundMessage`، `waitForNoOutbound`،
`formatConversationTranscript`، `resetBus` - اما در نگارش سناریوهای جدید
باید از نام‌های عمومی استفاده شود. نام‌های مستعار برای جلوگیری از
مهاجرت یک‌باره وجود دارند، نه به‌عنوان الگوی آینده.

## گزارش‌دهی

`qa-lab` یک گزارش پروتکل Markdown را از خط زمانی گذرگاه مشاهده‌شده صادر می‌کند.
گزارش باید به این پرسش‌ها پاسخ دهد:

- چه چیزی کار کرد
- چه چیزی شکست خورد
- چه چیزی مسدود باقی ماند
- افزودن چه سناریوهای پیگیری ارزشمند است

برای فهرست سناریوهای موجود - که هنگام برآورد حجم کار پیگیری
یا سیم‌کشی یک انتقال جدید مفید است - `pnpm openclaw qa coverage` را اجرا کنید (برای خروجی
قابل‌خواندن توسط ماشین، `--json` را اضافه کنید). هنگام انتخاب اثبات متمرکز برای
یک رفتار یا مسیر فایل تغییریافته، `pnpm openclaw qa coverage --match <query>` را اجرا کنید. گزارش
تطبیق، فرادادهٔ سناریو، ارجاع‌های مستندات، ارجاع‌های کد، شناسه‌های پوشش،
Pluginها و نیازمندی‌های ارائه‌دهنده را جست‌وجو می‌کند و سپس اهداف منطبق
`qa suite
--scenario ...` را چاپ می‌کند.

هر اجرای `qa suite` مصنوعات سطح‌بالای `qa-evidence.json`،
`qa-suite-summary.json` و `qa-suite-report.md` را برای مجموعهٔ
سناریوی انتخاب‌شده می‌نویسد. سناریوهایی که `execution.kind: vitest` یا
`execution.kind: playwright` را اعلام می‌کنند، مسیر آزمون متناظر را اجرا می‌کنند و همچنین
گزارش‌های هر سناریو را می‌نویسند. سناریوهایی که `execution.kind: script` را اعلام می‌کنند،
تولیدکنندهٔ شواهد در `execution.path` را از طریق `node --import tsx` اجرا می‌کنند (با
گسترش `${outputDir}` و `${scenarioId}` در `execution.args`)؛
تولیدکننده `qa-evidence.json` خودش را می‌نویسد که ورودی‌هایش به
خروجی مجموعه وارد می‌شوند و مسیر مصنوعات آن نسبت به
`qa-evidence.json` همان تولیدکننده تفکیک می‌شود. هنگامی که از طریق `qa run
--qa-profile` به `qa suite` می‌رسید، همان `qa-evidence.json` خلاصهٔ
کارت امتیاز نمایه را نیز برای دسته‌های طبقه‌بندی انتخاب‌شده شامل می‌شود.

خروجی پوشش را یک ابزار کمکی کشف بدانید، نه جایگزین دروازه؛ سناریوی
انتخاب‌شده همچنان به حالت ارائه‌دهنده، انتقال زنده،
Multipass، Testbox یا مسیر انتشار مناسب برای رفتار تحت آزمون نیاز دارد. برای
زمینهٔ کارت امتیاز، [کارت امتیاز بلوغ](/fa/maturity/scorecard) را ببینید.

برای بررسی شخصیت و سبک، همان سناریو را روی چندین ارجاع مدل زنده
اجرا کنید و یک گزارش Markdown داوری‌شده بنویسید:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.6-luna,thinking=medium,fast \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-8,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.6-sol,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-8,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

این فرمان فرایندهای فرزند محلی Gateway تضمین کیفیت را اجرا می‌کند، نه Docker.
سناریوهای ارزیابی شخصیت باید شخصیت را از طریق `SOUL.md` تنظیم کنند، سپس نوبت‌های عادی
کاربر مانند گفت‌وگو، راهنمایی فضای کاری و وظایف کوچک فایل را اجرا کنند. نباید به مدل
نامزد گفته شود که در حال ارزیابی است. فرمان هر رونوشت کامل را حفظ می‌کند،
آمار پایهٔ اجرا را ثبت می‌کند، سپس از مدل‌های داور در حالت سریع با استدلال
`xhigh`، در صورت پشتیبانی، می‌خواهد اجراها را بر اساس
طبیعی‌بودن، حال‌وهوا و شوخ‌طبعی رتبه‌بندی کنند. هنگام مقایسهٔ
ارائه‌دهندگان از `--blind-judge-models` استفاده کنید: اعلان داور همچنان همهٔ رونوشت‌ها و وضعیت اجرا را دریافت می‌کند، اما
ارجاع‌های نامزد با برچسب‌های خنثی مانند `candidate-01` جایگزین می‌شوند؛
گزارش پس از تجزیه، رتبه‌بندی‌ها را دوباره به ارجاع‌های واقعی نگاشت می‌کند.

اجرای نامزدها به‌طور پیش‌فرض از تفکر `high` استفاده می‌کند، همراه با `medium` برای GPT-5.6 Luna و
`xhigh` برای ارجاع‌های ارزیابی قدیمی‌تر OpenAI که از آن پشتیبانی می‌کنند. یک
نامزد مشخص را به‌صورت درون‌خطی با `--model provider/model,thinking=<level>` بازنویسی کنید؛ گزینه‌های
درون‌خطی همچنین از `fast`، `no-fast` و `fast=<bool>` پشتیبانی می‌کنند. `--thinking
<level>` همچنان یک مقدار جایگزین سراسری تنظیم می‌کند و شکل قدیمی‌تر `--model-thinking
<provider/model=level>` برای سازگاری حفظ شده است. ارجاع‌های نامزد
OpenAI به‌طور پیش‌فرض در حالت سریع هستند تا هر جا ارائه‌دهنده پشتیبانی می‌کند،
پردازش اولویت‌دار استفاده شود. فقط زمانی `--fast` را ارسال کنید که می‌خواهید حالت سریع را برای
همهٔ مدل‌های نامزد اجباری کنید. مدت‌زمان اجرای نامزد و داور برای
تحلیل بنچمارک در گزارش ثبت می‌شود، اما اعلان‌های داور صراحتاً می‌گویند که
بر اساس سرعت رتبه‌بندی نکنند. اجرای مدل‌های نامزد و داور هر دو به‌طور پیش‌فرض
هم‌زمانی 16 دارند. هنگامی که محدودیت‌های ارائه‌دهنده یا فشار Gateway محلی
باعث نویز بیش‌ازحد اجرا می‌شود، `--concurrency` یا `--judge-concurrency` را کاهش دهید.

هنگامی که هیچ `--model` نامزدی ارسال نشود، ارزیابی شخصیت به‌طور پیش‌فرض از
`openai/gpt-5.6-luna`، `openai/gpt-5.2`، `openai/gpt-5`،
`anthropic/claude-opus-4-8`، `anthropic/claude-sonnet-4-6`، `zai/glm-5.1`،
`moonshot/kimi-k2.5` و `google/gemini-3.1-pro-preview` استفاده می‌کند. هنگامی که هیچ
`--judge-model` ارسال نشود، داورها به‌طور پیش‌فرض
`openai/gpt-5.6-sol,thinking=xhigh,fast` و
`anthropic/claude-opus-4-8,thinking=high` هستند.
