OpenClaw برای هر اجرای عامل، یک system prompt سفارشی می‌سازد. این prompt **متعلق به OpenClaw** است و از prompt پیش‌فرض pi-coding-agent استفاده نمی‌کند.

این prompt توسط OpenClaw مونتاژ و به هر اجرای عامل تزریق می‌شود.

Pluginهای ارائه‌دهنده می‌توانند راهنمای prompt آگاه از cache را بدون جایگزین‌کردن
کل prompt متعلق به OpenClaw اضافه کنند. runtime ارائه‌دهنده می‌تواند:

- مجموعه کوچکی از بخش‌های هسته نام‌گذاری‌شده را جایگزین کند (`interaction_style`،
  `tool_call_style`، `execution_bias`)
- یک **پیشوند پایدار** را بالای مرز prompt cache تزریق کند
- یک **پسوند پویا** را پایین مرز prompt cache تزریق کند

از مشارکت‌های متعلق به ارائه‌دهنده برای تنظیم‌های خاص خانواده مدل استفاده کنید. جهش prompt قدیمی
`before_prompt_build` را برای سازگاری یا تغییرات واقعاً سراسری prompt نگه دارید،
نه رفتار عادی ارائه‌دهنده.

لایه OpenAI GPT-5 family قانون اجرای هسته را کوچک نگه می‌دارد و
راهنمای خاص مدل را برای قفل‌شدن پرسونا، خروجی موجز، انضباط ابزار،
جست‌وجوی موازی، پوشش خروجی قابل‌تحویل، راستی‌آزمایی، زمینه مفقود، و
بهداشت ابزار ترمینال اضافه می‌کند.

## ساختار

prompt عمداً فشرده است و از بخش‌های ثابت استفاده می‌کند:

- **ابزارها**: یادآور منبع حقیقت ابزار ساخت‌یافته به‌علاوه راهنمای runtime برای استفاده از ابزار.
- **سوگیری اجرا**: راهنمای فشرده برای پیگیری کار: روی درخواست‌های قابل‌اقدام
  در همان نوبت عمل کنید، تا پایان یا مسدودشدن ادامه دهید، از نتایج ضعیف ابزار
  بازیابی کنید، وضعیت قابل‌تغییر را زنده بررسی کنید، و پیش از نهایی‌کردن راستی‌آزمایی کنید.
- **ایمنی**: یادآور کوتاه guardrail برای پرهیز از رفتار قدرت‌طلبانه یا دورزدن نظارت.
- **Skills** (در صورت موجودبودن): به مدل می‌گوید چگونه دستورالعمل‌های skill را در صورت نیاز بارگذاری کند.
- **خودبه‌روزرسانی OpenClaw**: نحوه بررسی ایمن پیکربندی با
  `config.schema.lookup`، وصله‌کردن پیکربندی با `config.patch`، جایگزینی کامل
  پیکربندی با `config.apply`، و اجرای `update.run` فقط با درخواست صریح کاربر.
  ابزار فقط-مالک `gateway` همچنین از بازنویسی
  `tools.exec.ask` / `tools.exec.security` خودداری می‌کند، از جمله aliasهای قدیمی
  `tools.bash.*` که به آن مسیرهای exec محافظت‌شده نرمال‌سازی می‌شوند.
- **فضای کاری**: دایرکتوری کاری (`agents.defaults.workspace`).
- **مستندات**: مسیر محلی به مستندات OpenClaw (repo یا بسته npm) و زمان خواندن آن‌ها.
- **فایل‌های فضای کاری (تزریق‌شده)**: نشان می‌دهد فایل‌های bootstrap در ادامه گنجانده شده‌اند.
- **Sandbox** (وقتی فعال باشد): runtime سندباکس‌شده، مسیرهای sandbox، و اینکه exec ارتقایافته در دسترس است یا نه را نشان می‌دهد.
- **تاریخ و زمان فعلی**: زمان محلی کاربر، منطقه زمانی، و قالب زمان.
- **برچسب‌های پاسخ**: نحو اختیاری برچسب پاسخ برای ارائه‌دهندگان پشتیبانی‌شده.
- **Heartbeatها**: prompt مربوط به Heartbeat و رفتار تأیید، وقتی Heartbeatها برای عامل پیش‌فرض فعال هستند.
- **Runtime**: میزبان، OS، node، مدل، ریشه repo (وقتی تشخیص داده شود)، سطح فکرکردن (یک خط).
- **استدلال**: سطح دید فعلی + راهنمای toggle با /reasoning.

OpenClaw محتوای پایدار بزرگ، از جمله **زمینه پروژه**، را بالای
مرز داخلی prompt cache نگه می‌دارد. بخش‌های ناپایدار کانال/نشست مانند
راهنمای embed رابط کنترل، **پیام‌رسانی**، **صدا**، **زمینه چت گروهی**،
**واکنش‌ها**، **Heartbeatها**، و **Runtime** پایین آن مرز افزوده می‌شوند
تا backendهای محلی دارای prefix cache بتوانند پیشوند پایدار فضای کاری را
در نوبت‌های کانال بازاستفاده کنند. توضیحات ابزار نیز به همین شکل باید از
جاسازی نام‌های فعلی کانال خودداری کنند، وقتی schema پذیرفته‌شده از قبل آن
جزئیات runtime را حمل می‌کند.

بخش ابزارها همچنین شامل راهنمای runtime برای کارهای طولانی‌اجرا است:

- برای پیگیری آینده (`check back later`، یادآوری‌ها، کارهای تکرارشونده) از cron استفاده کنید
  به‌جای حلقه‌های sleep با `exec`، ترفندهای تأخیر `yieldMs`، یا polling مکرر `process`
- از `exec` / `process` فقط برای دستورهایی استفاده کنید که اکنون شروع می‌شوند و
  در پس‌زمینه به اجرا ادامه می‌دهند
- وقتی wake تکمیل خودکار فعال است، دستور را یک‌بار شروع کنید و به مسیر wake
  مبتنی بر push هنگام تولید خروجی یا شکست تکیه کنید
- وقتی نیاز دارید یک دستور در حال اجرا را بررسی کنید، برای logها، وضعیت، ورودی،
  یا مداخله از `process` استفاده کنید
- اگر کار بزرگ‌تر است، `sessions_spawn` را ترجیح دهید؛ تکمیل sub-agent مبتنی بر
  push است و به‌صورت خودکار به درخواست‌کننده اعلام می‌شود
- برای انتظار تکمیل، `subagents list` / `sessions_list` را در یک حلقه poll نکنید

وقتی ابزار آزمایشی `update_plan` فعال باشد، بخش ابزارها همچنین به
مدل می‌گوید فقط برای کارهای چندمرحله‌ای غیرساده از آن استفاده کند، دقیقاً یک
گام `in_progress` نگه دارد، و پس از هر به‌روزرسانی کل طرح را تکرار نکند.

guardrailهای ایمنی در system prompt جنبه مشورتی دارند. آن‌ها رفتار مدل را هدایت می‌کنند اما policy را enforce نمی‌کنند. برای enforce سخت‌گیرانه از policy ابزار، تأییدهای exec، sandboxing، و allowlistهای کانال استفاده کنید؛ اپراتورها به‌صورت طراحی‌شده می‌توانند این‌ها را غیرفعال کنند.

در کانال‌هایی که کارت/دکمه تأیید بومی دارند، prompt runtime اکنون به
عامل می‌گوید ابتدا به همان رابط تأیید بومی تکیه کند. فقط وقتی نتیجه ابزار می‌گوید
تأییدهای چت در دسترس نیستند یا تأیید دستی تنها مسیر است، باید دستور دستی
`/approve` را شامل کند.

## حالت‌های prompt

OpenClaw می‌تواند system promptهای کوچک‌تری برای sub-agentها render کند. runtime برای
هر اجرا یک `promptMode` تنظیم می‌کند (نه یک پیکربندی کاربرنما):

- `full` (پیش‌فرض): همه بخش‌های بالا را شامل می‌شود.
- `minimal`: برای sub-agentها استفاده می‌شود؛ **Skills**، **Memory Recall**، **خودبه‌روزرسانی OpenClaw**،
  **نام‌های مستعار مدل**، **هویت کاربر**، **برچسب‌های پاسخ**،
  **پیام‌رسانی**، **پاسخ‌های بی‌صدا**، و **Heartbeatها** را حذف می‌کند. ابزارها، **ایمنی**،
  فضای کاری، Sandbox، تاریخ و زمان فعلی (وقتی معلوم باشد)، Runtime، و زمینه تزریق‌شده
  همچنان در دسترس می‌مانند.
- `none`: فقط خط هویت پایه را برمی‌گرداند.

وقتی `promptMode=minimal` باشد، promptهای تزریق‌شده اضافی به‌جای **زمینه چت گروهی**
با **زمینه Subagent** برچسب‌گذاری می‌شوند.

برای اجراهای auto-reply کانال، OpenClaw می‌تواند بخش عمومی **پاسخ‌های بی‌صدا**
را وقتی زمینه چت مستقیم/گروهی از قبل رفتار conversation-specific `NO_REPLY`
حل‌شده را شامل می‌شود حذف کند. این از تکرار سازوکارهای token هم در system prompt
سراسری و هم در زمینه کانال جلوگیری می‌کند.

## snapshotهای prompt

OpenClaw snapshotهای commit‌شده prompt را برای مسیر موفق runtime Codex در
`test/fixtures/agents/prompt-snapshots/codex-runtime-happy-path/` نگه می‌دارد. آن‌ها
پارامترهای thread/turn انتخاب‌شده app-server به‌علاوه یک پشته layer بازسازی‌شده prompt
متصل به مدل را برای نوبت‌های مستقیم Telegram، گروه Discord، و heartbeat render می‌کنند.
آن پشته شامل یک fixture prompt مدل Codex `gpt-5.5` پین‌شده است که از شکل
catalog/cache مدل Codex تولید شده، متن developer مجوز مسیر موفق Codex،
دستورالعمل‌های developer OpenClaw، دستورالعمل‌های turn-scoped حالت همکاری
وقتی OpenClaw آن‌ها را فراهم می‌کند، ورودی نوبت کاربر، و ارجاع‌هایی به specهای پویا ابزار را شامل می‌شود.

fixture پین‌شده prompt مدل Codex را با
`pnpm prompt:snapshots:sync-codex-model` تازه‌سازی کنید. به‌صورت پیش‌فرض، اسکریپت ابتدا
cache runtime Codex را در `$CODEX_HOME/models_cache.json` جست‌وجو می‌کند، سپس
`~/.codex/models_cache.json`، و فقط پس از آن به قرارداد checkout نگه‌دارنده Codex در
`~/code/codex/codex-rs/models-manager/models.json` fallback می‌کند. اگر
هیچ‌کدام از این منابع وجود نداشته باشد، دستور بدون تغییر fixture commit‌شده
خارج می‌شود. برای تازه‌سازی از یک فایل مشخص `models_cache.json` یا `models.json`،
`--catalog <path>` را پاس دهید.

این snapshotها هنوز capture خام byte-for-byte درخواست OpenAI نیستند. Codex
می‌تواند زمینه فضای کاری متعلق به runtime مانند `AGENTS.md`، context محیط،
memoryها، دستورالعمل‌های app/plugin، و دستورالعمل‌های built-in حالت همکاری Default
را داخل runtime Codex پس از ارسال پارامترهای thread و turn توسط OpenClaw اضافه کند.

آن‌ها را با `pnpm prompt:snapshots:gen` دوباره تولید کنید و drift را با
`pnpm prompt:snapshots:check` راستی‌آزمایی کنید. CI بررسی drift را در shard مرزی
اضافی اجرا می‌کند تا تغییرات prompt و به‌روزرسانی‌های snapshot به همان PR متصل بمانند.

## تزریق bootstrap فضای کاری

فایل‌های bootstrap کوتاه‌سازی و زیر **زمینه پروژه** افزوده می‌شوند تا مدل بدون نیاز به خواندن صریح، context هویت و profile را ببیند:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md` (فقط در workspaceهای کاملاً جدید)
- `MEMORY.md` وقتی موجود باشد

همه این فایل‌ها در هر نوبت **به پنجره context تزریق می‌شوند** مگر اینکه
gate خاص فایل اعمال شود. `HEARTBEAT.md` در اجراهای عادی وقتی
heartbeatها برای عامل پیش‌فرض غیرفعال باشند یا
`agents.defaults.heartbeat.includeSystemPromptSection` برابر false باشد حذف می‌شود. فایل‌های تزریق‌شده
را موجز نگه دارید — به‌ویژه `MEMORY.md`، که می‌تواند در طول زمان رشد کند و به
مصرف context غیرمنتظره بالا و Compactionهای مکررتر منجر شود.

وقتی یک نشست روی harness بومی Codex اجرا می‌شود، Codex فایل `AGENTS.md` را
از طریق کشف project-doc خودش بارگذاری می‌کند. OpenClaw همچنان فایل‌های
bootstrap باقی‌مانده را resolve می‌کند و آن‌ها را به‌عنوان دستورالعمل‌های config به Codex می‌فرستد، بنابراین `SOUL.md`،
`TOOLS.md`، `IDENTITY.md`، `USER.md`، `HEARTBEAT.md`، `BOOTSTRAP.md`، و
`MEMORY.md` همان نقش context فضای کاری را بدون duplicate کردن
`AGENTS.md` حفظ می‌کنند.

<Note>
فایل‌های روزانه `memory/*.md` بخشی از Project Context معمول bootstrap **نیستند**. در نوبت‌های عادی، آن‌ها در صورت نیاز از طریق ابزارهای `memory_search` و `memory_get` در دسترس قرار می‌گیرند، بنابراین تا وقتی مدل صریحاً آن‌ها را نخواند، در پنجره context حساب نمی‌شوند. نوبت‌های bare `/new` و `/reset` استثنا هستند: runtime می‌تواند memory روزانه اخیر را به‌عنوان یک بلوک startup-context یک‌باره برای همان نوبت اول prepend کند.
</Note>

فایل‌های بزرگ با یک marker کوتاه‌سازی می‌شوند. حداکثر اندازه هر فایل با
`agents.defaults.bootstrapMaxChars` کنترل می‌شود (پیش‌فرض: 12000). کل محتوای bootstrap تزریق‌شده
در میان فایل‌ها با `agents.defaults.bootstrapTotalMaxChars` محدود می‌شود
(پیش‌فرض: 60000). فایل‌های مفقود یک marker کوتاه missing-file تزریق می‌کنند. وقتی truncation
رخ می‌دهد، OpenClaw می‌تواند یک بلوک هشدار در Project Context تزریق کند؛ این را با
`agents.defaults.bootstrapPromptTruncationWarning` کنترل کنید (`off`، `once`، `always`؛
پیش‌فرض: `once`).

نشست‌های sub-agent فقط `AGENTS.md` و `TOOLS.md` را تزریق می‌کنند (دیگر فایل‌های bootstrap
برای کوچک نگه‌داشتن context sub-agent فیلتر می‌شوند).

hookهای داخلی می‌توانند این گام را از طریق `agent:bootstrap` intercept کنند تا
فایل‌های bootstrap تزریق‌شده را جهش دهند یا جایگزین کنند (برای مثال جایگزین‌کردن `SOUL.md` با یک persona جایگزین).

اگر می‌خواهید صدای عامل کمتر عمومی باشد، از
[راهنمای شخصیت SOUL.md](/fa/concepts/soul) شروع کنید.

برای بررسی اینکه هر فایل تزریق‌شده چه مقدار سهم دارد (خام در برابر تزریق‌شده، truncation، به‌علاوه overhead schema ابزار)، از `/context list` یا `/context detail` استفاده کنید. [Context](/fa/concepts/context) را ببینید.

## مدیریت زمان

system prompt وقتی منطقه زمانی کاربر معلوم باشد شامل یک بخش اختصاصی **تاریخ و زمان فعلی** است. برای پایدار نگه‌داشتن prompt cache، اکنون فقط
**time zone** را شامل می‌شود (بدون ساعت پویا یا قالب زمان).

وقتی عامل به زمان فعلی نیاز دارد از `session_status` استفاده کنید؛ کارت وضعیت
یک خط timestamp دارد. همان ابزار می‌تواند به‌صورت اختیاری یک override مدل per-session
تنظیم کند (`model=default` آن را پاک می‌کند).

با این‌ها پیکربندی کنید:

- `agents.defaults.userTimezone`
- `agents.defaults.timeFormat` (`auto` | `12` | `24`)

برای جزئیات کامل رفتار، [تاریخ و زمان](/fa/date-time) را ببینید.

## Skills

وقتی skillهای واجد شرایط وجود داشته باشند، OpenClaw یک **فهرست فشرده Skills موجود**
(`formatSkillsForPrompt`) تزریق می‌کند که **مسیر فایل** هر skill را شامل می‌شود. این
prompt به مدل دستور می‌دهد از `read` برای بارگذاری SKILL.md در مکان فهرست‌شده
(workspace، managed، یا bundled) استفاده کند. اگر هیچ skill واجد شرایطی نباشد، بخش
Skills حذف می‌شود.

واجد شرایط بودن شامل gateهای metadata مربوط به skill، بررسی‌های runtime environment/config،
و allowlist مؤثر skill عامل وقتی `agents.defaults.skills` یا
`agents.list[].skills` پیکربندی شده باشد است.

skillهای bundled در Plugin فقط وقتی واجد شرایط هستند که Plugin مالک آن‌ها فعال باشد.
این به Pluginهای ابزار اجازه می‌دهد راهنماهای عملیاتی عمیق‌تری را بدون embed کردن همه
آن راهنماها مستقیماً در هر توضیح ابزار ارائه کنند.

```
<available_skills>
  <skill>
    <name>...</name>
    <description>...</description>
    <location>...</location>
  </skill>
</available_skills>
```

این کار prompt پایه را کوچک نگه می‌دارد و همچنان استفاده هدفمند از skill را ممکن می‌کند.

بودجه فهرست Skills متعلق به زیرسیستم skills است:

- پیش‌فرض سراسری: `skills.limits.maxSkillsPromptChars`
- override مخصوص عامل: `agents.list[].skillsLimits.maxSkillsPromptChars`

برش‌های عمومی و محدودِ زمان اجرا از سطح متفاوتی استفاده می‌کنند:

- `agents.defaults.contextLimits.*`
- `agents.list[].contextLimits.*`

این تفکیک، اندازه‌گذاری Skills را از اندازه‌گذاری خواندن/تزریق در زمان اجرا، مانند `memory_get`، نتایج زنده ابزار، و تازه‌سازی‌های AGENTS.md پس از Compaction جدا نگه می‌دارد.

## مستندات

پرامپت سیستمی شامل بخش **مستندات** است. وقتی مستندات محلی در دسترس باشد، به دایرکتوری مستندات محلی OpenClaw اشاره می‌کند (`docs/` در یک checkout گیت یا مستندات بسته‌بندی‌شده در بسته npm). اگر مستندات محلی در دسترس نباشد، به [https://docs.openclaw.ai](https://docs.openclaw.ai) برمی‌گردد.

همان بخش همچنین مکان منبع OpenClaw را هم شامل می‌شود. checkoutهای گیت، ریشه منبع محلی را در اختیار عامل می‌گذارند تا بتواند کد را مستقیماً بررسی کند. نصب‌های بسته‌ای شامل URL منبع GitHub هستند و به عامل می‌گویند هر زمان مستندات ناقص یا کهنه بود، منبع را آنجا بررسی کند. پرامپت همچنین به آینه عمومی مستندات، Discord جامعه، و ClawHub ([https://clawhub.ai](https://clawhub.ai)) برای کشف Skills اشاره می‌کند. این پرامپت به مدل می‌گوید برای رفتار، دستورها، پیکربندی، یا معماری OpenClaw ابتدا به مستندات مراجعه کند و در صورت امکان خودش `openclaw status` را اجرا کند (فقط زمانی از کاربر بپرسد که دسترسی ندارد). به‌طور خاص برای پیکربندی، عامل‌ها را برای مستندات دقیق در سطح فیلد و محدودیت‌ها به کنش ابزار `gateway` یعنی `config.schema.lookup` و سپس برای راهنمایی گسترده‌تر به `docs/gateway/configuration.md` و `docs/gateway/configuration-reference.md` ارجاع می‌دهد.
