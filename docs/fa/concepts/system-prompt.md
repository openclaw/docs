OpenClaw برای هر اجرای عامل یک اعلان سیستمی سفارشی می‌سازد. این اعلان **متعلق به OpenClaw** است و از اعلان پیش‌فرض pi-coding-agent استفاده نمی‌کند.

اعلان توسط OpenClaw مونتاژ و به هر اجرای عامل تزریق می‌شود.

Pluginهای ارائه‌دهنده می‌توانند راهنمای اعلان آگاه از کش را بدون جایگزین کردن
کل اعلان متعلق به OpenClaw ارائه کنند. runtime ارائه‌دهنده می‌تواند:

- مجموعه کوچکی از بخش‌های هسته نام‌گذاری‌شده را جایگزین کند (`interaction_style`،
  `tool_call_style`، `execution_bias`)
- یک **پیشوند پایدار** را بالای مرز کش اعلان تزریق کند
- یک **پسوند پویا** را زیر مرز کش اعلان تزریق کند

از مشارکت‌های متعلق به ارائه‌دهنده برای تنظیم اختصاصی خانواده مدل استفاده کنید. جهش اعلان
`before_prompt_build` قدیمی را برای سازگاری یا تغییرات واقعاً سراسری اعلان نگه دارید،
نه رفتار معمول ارائه‌دهنده.

لایه پوششی خانواده OpenAI GPT-5 قانون اجرای هسته را کوچک نگه می‌دارد و
راهنمای اختصاصی مدل را برای قفل شدن روی شخصیت، خروجی مختصر، انضباط ابزار،
جست‌وجوی موازی، پوشش خروجی قابل تحویل، راستی‌آزمایی، نبود زمینه، و
بهداشت ابزار ترمینال اضافه می‌کند.

## ساختار

اعلان عمداً فشرده است و از بخش‌های ثابت استفاده می‌کند:

- **ابزاردهی**: یادآور منبع حقیقت ابزار ساخت‌یافته به‌همراه راهنمای runtime برای استفاده از ابزار.
- **سوگیری اجرا**: راهنمای فشرده پیگیری: در همان نوبت روی درخواست‌های
  قابل اقدام عمل کنید، تا تکمیل یا مسدود شدن ادامه دهید، از نتایج ضعیف ابزار
  بازیابی کنید، وضعیت تغییرپذیر را زنده بررسی کنید، و پیش از نهایی‌سازی راستی‌آزمایی کنید.
- **ایمنی**: یادآور کوتاه حفاظتی برای پرهیز از رفتار قدرت‌طلبانه یا دور زدن نظارت.
- **Skills** (وقتی موجود باشد): به مدل می‌گوید چگونه دستورالعمل‌های Skills را در صورت نیاز بارگذاری کند.
- **خودبه‌روزرسانی OpenClaw**: نحوه بررسی امن پیکربندی با
  `config.schema.lookup`، patch کردن پیکربندی با `config.patch`، جایگزینی کل
  پیکربندی با `config.apply`، و اجرای `update.run` فقط با درخواست صریح کاربر.
  ابزار فقط-مالک `gateway` نیز از بازنویسی
  `tools.exec.ask` / `tools.exec.security` خودداری می‌کند، از جمله نام‌های مستعار قدیمی
  `tools.bash.*` که به آن مسیرهای exec محافظت‌شده نرمال‌سازی می‌شوند.
- **فضای کاری**: دایرکتوری کاری (`agents.defaults.workspace`).
- **مستندات**: مسیر محلی به مستندات OpenClaw (repo یا بسته npm) و زمان خواندن آن‌ها.
- **فایل‌های فضای کاری (تزریق‌شده)**: نشان می‌دهد فایل‌های bootstrap در ادامه گنجانده شده‌اند.
- **sandbox** (وقتی فعال باشد): runtime در sandbox، مسیرهای sandbox، و اینکه exec با دسترسی ارتقایافته موجود است یا نه را نشان می‌دهد.
- **تاریخ و زمان فعلی**: زمان محلی کاربر، منطقه زمانی، و قالب زمان.
- **برچسب‌های پاسخ**: نحو اختیاری برچسب پاسخ برای ارائه‌دهنده‌های پشتیبانی‌شده.
- **Heartbeatها**: اعلان Heartbeat و رفتار تأیید، وقتی Heartbeatها برای عامل پیش‌فرض فعال باشند.
- **runtime**: میزبان، OS، node، مدل، ریشه repo (وقتی شناسایی شود)، سطح تفکر (یک خط).
- **استدلال**: سطح نمایانی فعلی + راهنمای تغییر وضعیت /reasoning.

OpenClaw محتوای پایدار بزرگ، از جمله **زمینه پروژه**، را بالای
مرز داخلی کش اعلان نگه می‌دارد. بخش‌های فرّار کانال/نشست مانند
راهنمای توکار Control UI، **پیام‌رسانی**، **صدا**، **زمینه گپ گروهی**،
**واکنش‌ها**، **Heartbeatها**، و **runtime** زیر آن مرز اضافه می‌شوند
تا backendهای محلی دارای کش پیشوند بتوانند پیشوند پایدار فضای کاری را
در نوبت‌های کانال دوباره استفاده کنند. توضیحات ابزار نیز باید از درج نام‌های فعلی
کانال خودداری کنند وقتی schema پذیرفته‌شده از قبل آن جزئیات runtime را حمل می‌کند.

بخش ابزاردهی همچنین راهنمای runtime برای کارهای طولانی‌مدت را شامل می‌شود:

- برای پیگیری آینده (`check back later`، یادآورها، کارهای تکرارشونده) از cron استفاده کنید
  نه حلقه‌های sleep با `exec`، ترفندهای تأخیر `yieldMs`، یا polling تکراری `process`
- از `exec` / `process` فقط برای فرمان‌هایی استفاده کنید که اکنون شروع می‌شوند و
  در پس‌زمینه ادامه می‌یابند
- وقتی بیدارسازی تکمیل خودکار فعال است، فرمان را یک‌بار شروع کنید و به
  مسیر بیدارسازی push-based تکیه کنید وقتی خروجی می‌دهد یا شکست می‌خورد
- وقتی لازم است یک فرمان در حال اجرا را بررسی کنید، برای لاگ‌ها، وضعیت، ورودی،
  یا مداخله از `process` استفاده کنید
- اگر کار بزرگ‌تر است، `sessions_spawn` را ترجیح دهید؛ تکمیل زیرعامل
  push-based است و خودکار به درخواست‌کننده اعلام می‌شود
- فقط برای انتظار تکمیل، `subagents list` / `sessions_list` را در حلقه poll نکنید

وقتی ابزار آزمایشی `update_plan` فعال باشد، ابزاردهی همچنین به
مدل می‌گوید آن را فقط برای کارهای چندمرحله‌ای غیرساده استفاده کند، دقیقاً یک
مرحله `in_progress` نگه دارد، و از تکرار کل برنامه پس از هر به‌روزرسانی پرهیز کند.

حفاظ‌های ایمنی در اعلان سیستمی مشورتی هستند. آن‌ها رفتار مدل را هدایت می‌کنند اما policy را اعمال نمی‌کنند. برای اجرای سخت‌گیرانه از policy ابزار، تأییدهای exec، sandboxing، و allowlistهای کانال استفاده کنید؛ اپراتورها می‌توانند این‌ها را طبق طراحی غیرفعال کنند.

در کانال‌هایی با کارت‌ها/دکمه‌های تأیید بومی، اعلان runtime اکنون به
عامل می‌گوید ابتدا به آن UI تأیید بومی تکیه کند. تنها زمانی باید یک فرمان دستی
`/approve` را شامل کند که نتیجه ابزار بگوید تأییدهای چت در دسترس نیستند یا
تأیید دستی تنها مسیر است.

## حالت‌های اعلان

OpenClaw می‌تواند اعلان‌های سیستمی کوچک‌تری را برای زیرعامل‌ها render کند. runtime برای هر اجرا یک
`promptMode` تنظیم می‌کند (نه یک پیکربندی روبه‌روی کاربر):

- `full` (پیش‌فرض): همه بخش‌های بالا را شامل می‌شود.
- `minimal`: برای زیرعامل‌ها استفاده می‌شود؛ **Skills**، **فراخوانی حافظه**، **خودبه‌روزرسانی OpenClaw**،
  **نام‌های مستعار مدل**، **هویت کاربر**، **برچسب‌های پاسخ**،
  **پیام‌رسانی**، **پاسخ‌های خاموش**، و **Heartbeatها** را حذف می‌کند. ابزاردهی، **ایمنی**،
  فضای کاری، sandbox، تاریخ و زمان فعلی (وقتی شناخته شده باشد)، runtime، و زمینه تزریق‌شده
  همچنان در دسترس می‌مانند.
- `none`: فقط خط هویت پایه را برمی‌گرداند.

وقتی `promptMode=minimal` باشد، اعلان‌های تزریق‌شده اضافی به‌جای **زمینه گپ گروهی**
با برچسب **زمینه زیرعامل** مشخص می‌شوند.

برای اجراهای پاسخ خودکار کانال، OpenClaw می‌تواند بخش عمومی **پاسخ‌های خاموش**
را حذف کند وقتی زمینه چت مستقیم/گروهی از قبل رفتار اختصاصی مکالمه
`NO_REPLY` را شامل می‌شود. این کار از تکرار سازوکارهای token
هم در اعلان سیستمی سراسری و هم در زمینه کانال جلوگیری می‌کند.

## snapshotهای اعلان

OpenClaw snapshotهای اعلان مسیر موفقیت commit‌شده را برای runtime ابزار پیام/Codex
در `test/fixtures/agents/prompt-snapshots/happy-path/` نگه می‌دارد. آن‌ها
دستورالعمل‌های توسعه‌دهنده app-server متعلق به OpenClaw برای Codex، پارامترهای انتخاب‌شده
شروع/ازسرگیری thread، ورودی کاربر نوبت، و مشخصات پویای ابزار را برای مستقیم Telegram،
گروه Discord، و نوبت‌های Heartbeat render می‌کنند. اعلان سیستمی پایه پنهان Codex و
دستورالعمل‌های حالت همکاری Codex محدود به نوبت، متعلق به runtime Codex هستند
و توسط OpenClaw render نمی‌شوند.

آن‌ها را با `pnpm prompt:snapshots:gen` بازتولید کنید و drift را با
`pnpm prompt:snapshots:check` بررسی کنید.

## تزریق bootstrap فضای کاری

فایل‌های bootstrap کوتاه می‌شوند و زیر **زمینه پروژه** اضافه می‌شوند تا مدل بدون نیاز به خواندن صریح، زمینه هویت و profile را ببیند:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md` (فقط در فضاهای کاری کاملاً جدید)
- `MEMORY.md` وقتی موجود باشد

همه این فایل‌ها در هر نوبت **به پنجره زمینه تزریق می‌شوند** مگر اینکه
یک gate اختصاصی فایل اعمال شود. `HEARTBEAT.md` در اجراهای عادی حذف می‌شود وقتی
Heartbeatها برای عامل پیش‌فرض غیرفعال باشند یا
`agents.defaults.heartbeat.includeSystemPromptSection` برابر false باشد. فایل‌های تزریق‌شده را
مختصر نگه دارید — به‌ویژه `MEMORY.md`، که می‌تواند با گذشت زمان رشد کند و به
مصرف غیرمنتظره بالای زمینه و Compactionهای مکررتر منجر شود.

<Note>
فایل‌های روزانه `memory/*.md` بخشی از زمینه پروژه bootstrap عادی **نیستند**. در نوبت‌های معمولی، آن‌ها در صورت نیاز از طریق ابزارهای `memory_search` و `memory_get` دسترسی داده می‌شوند، بنابراین مگر اینکه مدل آن‌ها را صراحتاً بخواند، در پنجره زمینه حساب نمی‌شوند. نوبت‌های خالی `/new` و `/reset` استثنا هستند: runtime می‌تواند حافظه روزانه اخیر را به‌عنوان یک بلوک یک‌باره زمینه شروع برای آن نوبت اول پیشوند کند.
</Note>

فایل‌های بزرگ با یک marker کوتاه می‌شوند. حداکثر اندازه هر فایل توسط
`agents.defaults.bootstrapMaxChars` کنترل می‌شود (پیش‌فرض: 12000). کل محتوای bootstrap تزریق‌شده
در همه فایل‌ها توسط `agents.defaults.bootstrapTotalMaxChars` محدود می‌شود
(پیش‌فرض: 60000). فایل‌های مفقود یک marker کوتاه فایل مفقود تزریق می‌کنند. وقتی کوتاه‌سازی
رخ می‌دهد، OpenClaw می‌تواند یک بلوک هشدار در زمینه پروژه تزریق کند؛ این را با
`agents.defaults.bootstrapPromptTruncationWarning` کنترل کنید (`off`، `once`، `always`؛
پیش‌فرض: `once`).

نشست‌های زیرعامل فقط `AGENTS.md` و `TOOLS.md` را تزریق می‌کنند (دیگر فایل‌های bootstrap
برای کوچک نگه داشتن زمینه زیرعامل فیلتر می‌شوند).

hookهای داخلی می‌توانند این مرحله را از طریق `agent:bootstrap` رهگیری کنند تا
فایل‌های bootstrap تزریق‌شده را تغییر دهند یا جایگزین کنند (برای مثال جایگزین کردن `SOUL.md` با یک شخصیت جایگزین).

اگر می‌خواهید عامل کمتر عمومی به نظر برسد، از
[راهنمای شخصیت SOUL.md](/fa/concepts/soul) شروع کنید.

برای بررسی اینکه هر فایل تزریق‌شده چقدر سهم دارد (خام در برابر تزریق‌شده، کوتاه‌سازی، به‌علاوه سربار schema ابزار)، از `/context list` یا `/context detail` استفاده کنید. [زمینه](/fa/concepts/context) را ببینید.

## مدیریت زمان

اعلان سیستمی وقتی منطقه زمانی کاربر شناخته شده باشد، یک بخش اختصاصی **تاریخ و زمان فعلی** شامل می‌کند. برای پایدار نگه داشتن کش اعلان، اکنون فقط
**منطقه زمانی** را شامل می‌شود (بدون ساعت پویا یا قالب زمان).

وقتی عامل به زمان فعلی نیاز دارد از `session_status` استفاده کنید؛ کارت وضعیت
یک خط timestamp دارد. همین ابزار می‌تواند به‌صورت اختیاری یک override مدل برای هر نشست تنظیم کند
(`model=default` آن را پاک می‌کند).

با این‌ها پیکربندی کنید:

- `agents.defaults.userTimezone`
- `agents.defaults.timeFormat` (`auto` | `12` | `24`)

برای جزئیات کامل رفتار، [تاریخ و زمان](/fa/date-time) را ببینید.

## Skills

وقتی Skills واجد شرایط وجود داشته باشند، OpenClaw یک **فهرست Skills موجود** فشرده
(`formatSkillsForPrompt`) تزریق می‌کند که **مسیر فایل** هر skill را شامل می‌شود. اعلان
به مدل دستور می‌دهد از `read` برای بارگذاری SKILL.md در مکان فهرست‌شده
(فضای کاری، مدیریت‌شده، یا bundled) استفاده کند. اگر هیچ skill واجد شرایط نباشد،
بخش Skills حذف می‌شود.

واجد شرایط بودن شامل gateهای metadata مهارت، بررسی‌های runtime environment/config،
و allowlist مؤثر skill عامل است وقتی `agents.defaults.skills` یا
`agents.list[].skills` پیکربندی شده باشد.

Skills bundled با Plugin فقط زمانی واجد شرایط هستند که Plugin مالک آن‌ها فعال باشد.
این به Pluginهای ابزار اجازه می‌دهد راهنماهای عملیاتی عمیق‌تری را بدون درج همه
آن راهنماها مستقیماً در هر توضیح ابزار ارائه کنند.

```
<available_skills>
  <skill>
    <name>...</name>
    <description>...</description>
    <location>...</location>
  </skill>
</available_skills>
```

این کار اعلان پایه را کوچک نگه می‌دارد و در عین حال استفاده هدفمند از skill را ممکن می‌کند.

بودجه فهرست Skills متعلق به زیرسیستم Skills است:

- پیش‌فرض سراسری: `skills.limits.maxSkillsPromptChars`
- override برای هر عامل: `agents.list[].skillsLimits.maxSkillsPromptChars`

گزیده‌های runtime محدودشده عمومی از سطح دیگری استفاده می‌کنند:

- `agents.defaults.contextLimits.*`
- `agents.list[].contextLimits.*`

این تفکیک اندازه‌گذاری Skills را از اندازه‌گذاری خواندن/تزریق runtime مانند
`memory_get`، نتایج ابزار زنده، و refreshهای AGENTS.md پس از Compaction جدا نگه می‌دارد.

## مستندات

اعلان سیستمی یک بخش **مستندات** شامل می‌کند. وقتی مستندات محلی در دسترس باشند، به
دایرکتوری محلی مستندات OpenClaw اشاره می‌کند (`docs/` در یک Git checkout یا مستندات bundled بسته npm).
اگر مستندات محلی در دسترس نباشند، به
[https://docs.openclaw.ai](https://docs.openclaw.ai) fallback می‌کند.

همین بخش همچنین مکان source OpenClaw را شامل می‌شود. Git checkoutها ریشه source محلی را ارائه می‌کنند
تا عامل بتواند code را مستقیماً بررسی کند. نصب‌های بسته شامل URL source در GitHub هستند
و به عامل می‌گویند هر زمان مستندات ناقص یا stale هستند source را آنجا بررسی کند.
اعلان همچنین به آینه مستندات عمومی، Discord جامعه، و ClawHub
([https://clawhub.ai](https://clawhub.ai)) برای کشف Skills اشاره می‌کند. به مدل می‌گوید
برای رفتار، فرمان‌ها، پیکربندی، یا معماری OpenClaw ابتدا مستندات را مشورت کند، و
هر وقت ممکن است خودش `openclaw status` را اجرا کند (فقط وقتی دسترسی ندارد از کاربر بپرسد).
به‌طور مشخص برای پیکربندی، عامل‌ها را به action ابزار `gateway`
یعنی `config.schema.lookup` برای مستندات و قیود دقیق در سطح field، سپس به
`docs/gateway/configuration.md` و `docs/gateway/configuration-reference.md`
برای راهنمای گسترده‌تر ارجاع می‌دهد.
