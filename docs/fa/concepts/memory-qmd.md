[QMD](https://github.com/tobi/qmd) یک سایدکار جست‌وجوی محلی‌اول است که در کنار OpenClaw اجرا می‌شود. این ابزار BM25، جست‌وجوی برداری، و بازرتبه‌بندی را در یک باینری واحد ترکیب می‌کند و می‌تواند محتوایی فراتر از فایل‌های حافظه فضای کاری شما را نمایه‌سازی کند.

## چیزی که نسبت به موتور داخلی اضافه می‌کند

- **بازرتبه‌بندی و گسترش پرس‌وجو** برای بازیابی بهتر.
- **نمایه‌سازی دایرکتوری‌های اضافی** -- مستندات پروژه، یادداشت‌های تیم، هر چیزی روی دیسک.
- **نمایه‌سازی رونوشت‌های نشست** -- فراخوانی گفت‌وگوهای قبلی.
- **کاملا محلی** -- با Plugin رسمی ارائه‌دهنده llama.cpp اجرا می‌شود و مدل‌های GGUF را خودکار دانلود می‌کند.
- **بازگشت خودکار** -- اگر QMD در دسترس نباشد، OpenClaw به‌صورت یکپارچه به موتور داخلی برمی‌گردد.

## شروع کار

### پیش‌نیازها

- QMD را نصب کنید: `npm install -g @tobilu/qmd` یا `bun install -g @tobilu/qmd`
- ساخت SQLite که افزونه‌ها را مجاز می‌کند (`brew install sqlite` در macOS).
- QMD باید در `PATH` متعلق به Gateway باشد.
- macOS و Linux بدون تنظیم اضافی کار می‌کنند. Windows بهترین پشتیبانی را از طریق WSL2 دارد.

### فعال‌سازی

```json5
{
  memory: {
    backend: "qmd",
  },
}
```

OpenClaw یک خانه QMD خودبسنده زیر
`~/.openclaw/agents/<agentId>/qmd/` می‌سازد و چرخه عمر سایدکار را به‌صورت خودکار مدیریت می‌کند -- کالکشن‌ها، به‌روزرسانی‌ها، و اجرای embedding برای شما انجام می‌شوند. شکل‌های فعلی کالکشن QMD و پرس‌وجوی MCP را ترجیح می‌دهد، اما همچنان در صورت نیاز به پرچم‌های الگوی کالکشن جایگزین و نام‌های قدیمی‌تر ابزار MCP برمی‌گردد. همسان‌سازی زمان راه‌اندازی همچنین کالکشن‌های مدیریت‌شده کهنه را دوباره به الگوهای canonical آن‌ها بازمی‌سازد، وقتی یک کالکشن قدیمی‌تر QMD با همان نام هنوز وجود داشته باشد.

## سایدکار چگونه کار می‌کند

- OpenClaw از فایل‌های حافظه فضای کاری شما و هر `memory.qmd.paths` پیکربندی‌شده کالکشن می‌سازد، سپس وقتی مدیر QMD باز می‌شود و پس از آن به‌صورت دوره‌ای (پیش‌فرض هر 5 دقیقه) `qmd update` را اجرا می‌کند. این بازآوری‌ها از طریق زیرفرایندهای QMD اجرا می‌شوند، نه یک پیمایش فایل‌سیستم درون‌فرایندی. حالت‌های معنایی همچنین `qmd embed` را اجرا می‌کنند.
- کالکشن پیش‌فرض فضای کاری، `MEMORY.md` به‌همراه درخت `memory/` را دنبال می‌کند. `memory.md` با حروف کوچک به‌عنوان فایل حافظه ریشه نمایه‌سازی نمی‌شود.
- اسکنر خود QMD مسیرهای مخفی و دایرکتوری‌های رایج وابستگی/ساخت مانند `.git`، `.cache`، `node_modules`، `vendor`، `dist`، و `build` را نادیده می‌گیرد. راه‌اندازی Gateway به‌صورت پیش‌فرض QMD را مقداردهی اولیه نمی‌کند، بنابراین بوت سرد از وارد کردن runtime حافظه یا ساخت watcher بلندمدت پیش از نخستین استفاده از حافظه جلوگیری می‌کند.
- اگر با این حال می‌خواهید QMD هنگام شروع Gateway مقداردهی اولیه شود، `memory.qmd.update.startup` را روی `idle` یا `immediate` تنظیم کنید. با `memory.qmd.update.onBoot: true`، راه‌اندازی بازآوری اولیه را اجرا می‌کند. با `onBoot: false`، راه‌اندازی آن بازآوری فوری را رد می‌کند اما وقتی بازه‌های update یا embed پیکربندی شده باشند همچنان مدیر بلندمدت را باز می‌کند، تا QMD بتواند watcher و تایمرهای منظم خود را مالک شود.
- جست‌وجوها از `searchMode` پیکربندی‌شده استفاده می‌کنند (پیش‌فرض: `search`؛ از `vsearch` و `query` هم پشتیبانی می‌کند). `search` فقط BM25 است، بنابراین OpenClaw در آن حالت probeهای آمادگی برداری معنایی و نگهداری embedding را رد می‌کند. اگر یک حالت شکست بخورد، OpenClaw با `qmd query` دوباره تلاش می‌کند.
- وقتی `searchMode` برابر `query` است، `memory.qmd.rerank` را روی `false` تنظیم کنید تا از مسیر پرس‌وجوی هیبریدی QMD بدون reranker استفاده شود. OpenClaw در مسیر مستقیم QMD CLI مقدار `--no-rerank` و در ابزار پرس‌وجوی MCP متعلق به QMD مقدار `rerank: false` را ارسال می‌کند. این گزینه به QMD 2.1 یا جدیدتر نیاز دارد.
- با نسخه‌های QMD که فیلترهای چندکالکشنی را اعلام می‌کنند، OpenClaw کالکشن‌های هم‌منبع را در یک فراخوانی جست‌وجوی QMD گروه‌بندی می‌کند. نسخه‌های قدیمی‌تر QMD fallback سازگارِ هر کالکشن را نگه می‌دارند.
- اگر QMD کاملا شکست بخورد، OpenClaw به موتور داخلی SQLite برمی‌گردد. تلاش‌های تکراری نوبت چت پس از شکست باز کردن، برای مدت کوتاهی عقب‌نشینی می‌کنند تا باینری گم‌شده یا وابستگی خراب سایدکار طوفان تلاش مجدد ایجاد نکند؛ `openclaw memory status` و probeهای یک‌باره CLI همچنان QMD را مستقیم دوباره بررسی می‌کنند.

<Info>
نخستین جست‌وجو ممکن است کند باشد -- QMD در نخستین اجرای `qmd query` مدل‌های GGUF (حدود 2 GB) را برای بازرتبه‌بندی و گسترش پرس‌وجو خودکار دانلود می‌کند.
</Info>

## کارایی جست‌وجو و سازگاری

OpenClaw مسیر جست‌وجوی QMD را با نصب‌های فعلی و قدیمی‌تر QMD سازگار نگه می‌دارد.

هنگام راه‌اندازی، OpenClaw متن راهنمای QMD نصب‌شده را یک بار برای هر مدیر بررسی می‌کند. اگر باینری پشتیبانی از چند فیلتر کالکشن را اعلام کند، OpenClaw همه کالکشن‌های هم‌منبع را با یک فرمان جست‌وجو می‌کند:

```bash
qmd search "router notes" --json -n 10 -c memory-root-main -c memory-dir-main
```

این کار از شروع یک زیرفرایند QMD برای هر کالکشن حافظه پایدار جلوگیری می‌کند. کالکشن‌های رونوشت نشست در گروه منبع خودشان می‌مانند، بنابراین جست‌وجوهای ترکیبی `memory` + `sessions` همچنان ورودی متنوع‌ساز نتیجه را از هر دو منبع می‌دهند.

ساخت‌های قدیمی‌تر QMD فقط یک فیلتر کالکشن را می‌پذیرند. وقتی OpenClaw یکی از آن ساخت‌ها را تشخیص دهد، مسیر سازگاری را نگه می‌دارد و هر کالکشن را جداگانه جست‌وجو می‌کند، سپس نتایج را ادغام و تکراری‌زدایی می‌کند.

برای بررسی دستی قرارداد نصب‌شده، اجرا کنید:

```bash
qmd --help | grep -i collection
```

راهنمای فعلی QMD می‌گوید فیلترهای کالکشن می‌توانند یک یا چند کالکشن را هدف بگیرند. راهنمای قدیمی‌تر معمولا یک کالکشن واحد را توصیف می‌کند.

## بازنویسی مدل‌ها

متغیرهای محیطی مدل QMD بدون تغییر از فرایند Gateway عبور می‌کنند، بنابراین می‌توانید QMD را به‌صورت سراسری تنظیم کنید بدون اینکه پیکربندی جدید OpenClaw اضافه کنید:

```bash
export QMD_EMBED_MODEL="hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"
export QMD_RERANK_MODEL="/absolute/path/to/reranker.gguf"
export QMD_GENERATE_MODEL="/absolute/path/to/generator.gguf"
```

پس از تغییر مدل embedding، embeddingها را دوباره اجرا کنید تا نمایه با فضای برداری جدید هماهنگ شود.

## نمایه‌سازی مسیرهای اضافی

QMD را به دایرکتوری‌های اضافی اشاره دهید تا قابل جست‌وجو شوند:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

قطعه‌های مسیرهای اضافی در نتایج جست‌وجو به شکل `qmd/<collection>/<relative-path>` ظاهر می‌شوند. `memory_get` این پیشوند را می‌فهمد و از ریشه کالکشن درست می‌خواند.

## نمایه‌سازی رونوشت‌های نشست

نمایه‌سازی نشست را فعال کنید تا گفت‌وگوهای قبلی فراخوانی شوند:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      sessions: { enabled: true },
    },
  },
}
```

رونوشت‌ها به‌صورت نوبت‌های پاک‌سازی‌شده کاربر/دستیار به یک کالکشن اختصاصی QMD زیر `~/.openclaw/agents/<id>/qmd/sessions/` صادر می‌شوند.

## دامنه جست‌وجو

به‌صورت پیش‌فرض، نتایج جست‌وجوی QMD در نشست‌های مستقیم و کانالی نمایش داده می‌شوند (نه گروه‌ها). برای تغییر این رفتار `memory.qmd.scope` را پیکربندی کنید:

```json5
{
  memory: {
    qmd: {
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
    },
  },
}
```

وقتی scope یک جست‌وجو را رد کند، OpenClaw هشداری با کانال و نوع چت مشتق‌شده ثبت می‌کند تا اشکال‌زدایی نتایج خالی آسان‌تر شود.

## ارجاع‌ها

وقتی `memory.citations` برابر `auto` یا `on` باشد، قطعه‌های جست‌وجو یک پانویس `Source: <path#line>` دارند. `memory.citations = "off"` را تنظیم کنید تا پانویس حذف شود، در حالی که مسیر همچنان به‌صورت داخلی به agent ارسال می‌شود.

## چه زمانی استفاده شود

QMD را زمانی انتخاب کنید که نیاز دارید:

- بازرتبه‌بندی برای نتایج باکیفیت‌تر.
- جست‌وجوی مستندات یا یادداشت‌های پروژه خارج از فضای کاری.
- فراخوانی گفت‌وگوهای نشست‌های گذشته.
- جست‌وجوی کاملا محلی بدون کلیدهای API.

برای راه‌اندازی‌های ساده‌تر، [موتور داخلی](/fa/concepts/memory-builtin) بدون وابستگی اضافی خوب کار می‌کند.

## عیب‌یابی

**QMD پیدا نمی‌شود؟** مطمئن شوید باینری در `PATH` متعلق به Gateway است. اگر OpenClaw به‌صورت سرویس اجرا می‌شود، یک symlink بسازید:
`sudo ln -s ~/.bun/bin/qmd /usr/local/bin/qmd`.

اگر `qmd --version` در shell شما کار می‌کند اما OpenClaw همچنان `spawn qmd ENOENT` گزارش می‌دهد، احتمالا فرایند Gateway نسبت به shell تعاملی شما `PATH` متفاوتی دارد. باینری را صریحا pin کنید:

```json5
{
  memory: {
    backend: "qmd",
    qmd: {
      command: "/absolute/path/to/qmd",
    },
  },
}
```

در محیطی که QMD نصب شده است از `command -v qmd` استفاده کنید، سپس با `openclaw memory status --deep` دوباره بررسی کنید.

**نخستین جست‌وجو خیلی کند است؟** QMD در نخستین استفاده مدل‌های GGUF را دانلود می‌کند. با `qmd query "test"` و با همان دایرکتوری‌های XDG که OpenClaw استفاده می‌کند، از قبل گرم کنید.

**زیرفرایندهای QMD زیادی هنگام جست‌وجو؟** در صورت امکان QMD را به‌روزرسانی کنید. OpenClaw فقط وقتی از یک فرایند برای جست‌وجوهای چندکالکشنی هم‌منبع استفاده می‌کند که QMD نصب‌شده پشتیبانی از چند فیلتر `-c` را اعلام کند؛ در غیر این صورت، برای درستی، fallback قدیمی‌تر هر کالکشن را نگه می‌دارد.

**QMD فقط BM25 هنوز تلاش می‌کند llama.cpp را بسازد؟** `memory.qmd.searchMode = "search"` را تنظیم کنید. OpenClaw آن حالت را فقط واژگانی در نظر می‌گیرد، probeهای وضعیت برداری QMD یا نگهداری embedding را اجرا نمی‌کند، و بررسی‌های آمادگی معنایی را به راه‌اندازی‌های `vsearch` یا `query` واگذار می‌کند.

**جست‌وجو timeout می‌شود؟** `memory.qmd.limits.timeoutMs` را افزایش دهید (پیش‌فرض: 4000ms). برای سخت‌افزار کندتر آن را روی `120000` تنظیم کنید.

**نتایج خالی در چت‌های گروهی؟** `memory.qmd.scope` را بررسی کنید -- پیش‌فرض فقط نشست‌های مستقیم و کانالی را مجاز می‌کند.

**جست‌وجوی حافظه ریشه ناگهان بیش از حد گسترده شد؟** Gateway را راه‌اندازی مجدد کنید یا تا همسان‌سازی راه‌اندازی بعدی صبر کنید. OpenClaw وقتی یک تداخل هم‌نام تشخیص دهد، کالکشن‌های مدیریت‌شده کهنه را دوباره به الگوهای canonical `MEMORY.md` و `memory/` بازمی‌سازد.

**مخزن‌های موقت قابل‌مشاهده در فضای کاری باعث `ENAMETOOLONG` یا نمایه‌سازی خراب می‌شوند؟**
پیمایش QMD در حال حاضر به‌جای قوانین symlink داخلی OpenClaw، رفتار اسکنر زیرین QMD را دنبال می‌کند. checkoutهای موقت monorepo را زیر دایرکتوری‌های مخفی مانند `.tmp/` یا خارج از ریشه‌های QMD نمایه‌شده نگه دارید تا زمانی که QMD پیمایش cycle-safe یا کنترل‌های حذف صریح را ارائه کند.
