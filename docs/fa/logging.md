OpenClaw دو سطح اصلی برای لاگ‌ها دارد:

- **لاگ‌های فایل** (خطوط JSON) که توسط Gateway نوشته می‌شوند.
- **خروجی کنسول** که در ترمینال‌ها و رابط اشکال‌زدایی Gateway نمایش داده می‌شود.

زبانه **لاگ‌ها** در Control UI لاگ فایل gateway را به‌صورت tail نمایش می‌دهد. این صفحه توضیح می‌دهد لاگ‌ها کجا قرار دارند، چگونه آن‌ها را بخوانید، و چگونه سطح‌ها و قالب‌های لاگ را پیکربندی کنید.

## محل قرارگیری لاگ‌ها

به‌طور پیش‌فرض، Gateway یک فایل لاگ چرخشی را در مسیر زیر می‌نویسد:

`/tmp/openclaw/openclaw-YYYY-MM-DD.log`

تاریخ از منطقه زمانی محلی میزبان gateway استفاده می‌کند.

هر فایل وقتی به `logging.maxFileBytes` برسد می‌چرخد (پیش‌فرض: ۱۰۰ مگابایت). OpenClaw تا پنج آرشیو شماره‌گذاری‌شده را کنار فایل فعال نگه می‌دارد، مانند `openclaw-YYYY-MM-DD.1.log`، و به‌جای سرکوب تشخیص‌ها، نوشتن را در یک لاگ فعال تازه ادامه می‌دهد.

می‌توانید این را در `~/.openclaw/openclaw.json` بازنویسی کنید:

```json
{
  "logging": {
    "file": "/path/to/openclaw.log"
  }
}
```

## نحوه خواندن لاگ‌ها

### CLI: دنبال‌کردن زنده tail (توصیه‌شده)

از CLI برای tail کردن فایل لاگ gateway از طریق RPC استفاده کنید:

```bash
openclaw logs --follow
```

گزینه‌های کاربردی فعلی:

- `--local-time`: زمان‌ها را در منطقه زمانی محلی شما نمایش می‌دهد
- `--url <url>` / `--token <token>` / `--timeout <ms>`: پرچم‌های استاندارد RPC برای Gateway
- `--expect-final`: پرچم انتظار برای پاسخ نهایی RPC مبتنی بر عامل (از طریق لایه مشترک کلاینت در اینجا پذیرفته می‌شود)

حالت‌های خروجی:

- **نشست‌های TTY**: خطوط لاگ ساخت‌یافته، رنگی و خوانا.
- **نشست‌های غیر TTY**: متن ساده.
- `--json`: JSON خط‌به‌خط (یک رویداد لاگ در هر خط).
- `--plain`: اجبار به متن ساده در نشست‌های TTY.
- `--no-color`: غیرفعال‌کردن رنگ‌های ANSI.

وقتی یک `--url` صریح می‌دهید، CLI به‌طور خودکار پیکربندی یا اعتبارنامه‌های محیطی را اعمال نمی‌کند؛ اگر Gateway هدف به احراز هویت نیاز دارد، خودتان `--token` را هم اضافه کنید.

در حالت JSON، CLI اشیای برچسب‌خورده با `type` تولید می‌کند:

- `meta`: فراداده جریان (فایل، مکان‌نما، اندازه)
- `log`: ورودی لاگ تحلیل‌شده
- `notice`: نکته‌های کوتاه‌سازی / چرخش
- `raw`: خط لاگ تحلیل‌نشده

اگر Gateway ضمنی local loopback درخواست pair کردن بدهد، هنگام اتصال بسته شود، یا پیش از پاسخ‌دادن `logs.tail` مهلتش تمام شود، `openclaw logs` به‌طور خودکار به فایل لاگ Gateway پیکربندی‌شده برمی‌گردد. اهداف صریح `--url` از این fallback استفاده نمی‌کنند.

اگر Gateway در دسترس نباشد، CLI یک راهنمای کوتاه برای اجرای دستور زیر چاپ می‌کند:

```bash
openclaw doctor
```

### Control UI (وب)

زبانه **لاگ‌ها** در Control UI همان فایل را با استفاده از `logs.tail` دنبال می‌کند. برای نحوه بازکردن آن، [Control UI](/fa/web/control-ui) را ببینید.

### لاگ‌های فقط کانال

برای فیلتر کردن فعالیت کانال‌ها (WhatsApp/Telegram/و غیره)، از این دستور استفاده کنید:

```bash
openclaw channels logs --channel whatsapp
```

## قالب‌های لاگ

### لاگ‌های فایل (JSONL)

هر خط در فایل لاگ یک شیء JSON است. CLI و Control UI این ورودی‌ها را تحلیل می‌کنند تا خروجی ساخت‌یافته (زمان، سطح، زیرسامانه، پیام) نمایش دهند.

رکوردهای JSONL مربوط به لاگ فایل، در صورت وجود، فیلدهای سطح‌بالای قابل فیلتر توسط ماشین را نیز شامل می‌شوند:

- `hostname`: نام میزبان gateway.
- `message`: متن پیام لاگ تخت‌شده برای جست‌وجوی تمام‌متنی.
- `agent_id`: شناسه عامل فعال وقتی فراخوانی لاگ دارای زمینه عامل باشد.
- `session_id`: شناسه/کلید نشست فعال وقتی فراخوانی لاگ دارای زمینه نشست باشد.
- `channel`: کانال فعال وقتی فراخوانی لاگ دارای زمینه کانال باشد.

OpenClaw آرگومان‌های لاگ ساخت‌یافته اصلی را در کنار این فیلدها حفظ می‌کند تا تحلیلگرهای موجودی که کلیدهای آرگومان شماره‌گذاری‌شده tslog را می‌خوانند همچنان کار کنند.

### خروجی کنسول

لاگ‌های کنسول **TTY-aware** هستند و برای خوانایی قالب‌بندی می‌شوند:

- پیشوندهای زیرسامانه (برای مثال `gateway/channels/whatsapp`)
- رنگ‌بندی سطح‌ها (info/warn/error)
- حالت فشرده یا JSON اختیاری

قالب‌بندی کنسول با `logging.consoleStyle` کنترل می‌شود.

### لاگ‌های WebSocket مربوط به Gateway

`openclaw gateway` همچنین برای ترافیک RPC لاگ‌برداری پروتکل WebSocket دارد:

- حالت عادی: فقط نتایج مهم (خطاها، خطاهای تحلیل، فراخوانی‌های کند)
- `--verbose`: همه ترافیک درخواست/پاسخ
- `--ws-log auto|compact|full`: انتخاب سبک نمایش verbose
- `--compact`: نام مستعار برای `--ws-log compact`

مثال‌ها:

```bash
openclaw gateway
openclaw gateway --verbose --ws-log compact
openclaw gateway --verbose --ws-log full
```

## پیکربندی لاگ‌برداری

همه پیکربندی لاگ‌برداری زیر `logging` در `~/.openclaw/openclaw.json` قرار دارد.

```json
{
  "logging": {
    "level": "info",
    "file": "/tmp/openclaw/openclaw-YYYY-MM-DD.log",
    "consoleLevel": "info",
    "consoleStyle": "pretty",
    "redactSensitive": "tools",
    "redactPatterns": ["sk-.*"]
  }
}
```

### سطح‌های لاگ

- `logging.level`: سطح **لاگ‌های فایل** (JSONL).
- `logging.consoleLevel`: سطح جزئیات **کنسول**.

می‌توانید هر دو را با متغیر محیطی **`OPENCLAW_LOG_LEVEL`** بازنویسی کنید (برای مثال `OPENCLAW_LOG_LEVEL=debug`). متغیر محیطی نسبت به فایل پیکربندی اولویت دارد، بنابراین می‌توانید بدون ویرایش `openclaw.json` سطح جزئیات را برای یک اجرای واحد افزایش دهید. همچنین می‌توانید گزینه سراسری CLI یعنی **`--log-level <level>`** را بدهید (برای مثال، `openclaw --log-level debug gateway run`) که متغیر محیطی را برای همان دستور بازنویسی می‌کند.

`--verbose` فقط خروجی کنسول و سطح جزئیات لاگ WS را تحت تأثیر قرار می‌دهد؛ سطح‌های لاگ فایل را تغییر نمی‌دهد.

### هم‌بستگی trace

لاگ‌های فایل JSONL هستند. وقتی یک فراخوانی لاگ دارای زمینه trace تشخیصی معتبر باشد، OpenClaw فیلدهای trace را به‌صورت کلیدهای JSON سطح‌بالا (`traceId`, `spanId`, `parentSpanId`, `traceFlags`) می‌نویسد تا پردازشگرهای خارجی لاگ بتوانند آن خط را با spanهای OTEL و انتشار `traceparent` ارائه‌دهنده هم‌بسته کنند.

درخواست‌های HTTP مربوط به Gateway و فریم‌های WebSocket مربوط به Gateway یک محدوده trace داخلی درخواست ایجاد می‌کنند. لاگ‌ها و رویدادهای تشخیصی منتشرشده داخل آن محدوده async، وقتی زمینه trace صریحی ارسال نکنند، trace درخواست را به ارث می‌برند. traceهای اجرای عامل و فراخوانی مدل به فرزندان trace درخواست فعال تبدیل می‌شوند، بنابراین لاگ‌های محلی، snapshotهای تشخیصی، spanهای OTEL، و headerهای `traceparent` قابل اعتماد ارائه‌دهنده می‌توانند بدون لاگ‌کردن محتوای خام درخواست یا مدل، با `traceId` به هم متصل شوند.

### اندازه و زمان‌بندی فراخوانی مدل

تشخیص‌های فراخوانی مدل، اندازه‌گیری‌های محدودشده درخواست/پاسخ را بدون ضبط محتوای خام prompt یا پاسخ ثبت می‌کنند:

- `requestPayloadBytes`: اندازه بایتی UTF-8 payload نهایی درخواست مدل
- `responseStreamBytes`: اندازه بایتی UTF-8 رویدادهای پاسخ مدل به‌صورت stream
- `timeToFirstByteMs`: زمان سپری‌شده پیش از نخستین رویداد پاسخ stream شده
- `durationMs`: مدت کل فراخوانی مدل

وقتی export تشخیص‌ها فعال باشد، این فیلدها برای snapshotهای تشخیصی، hookهای Plugin مربوط به فراخوانی مدل، و spanها/metricهای فراخوانی مدل OTEL در دسترس هستند.

### سبک‌های کنسول

`logging.consoleStyle`:

- `pretty`: مناسب انسان، رنگی، همراه با timestamp.
- `compact`: خروجی فشرده‌تر (بهترین گزینه برای نشست‌های طولانی).
- `json`: یک JSON در هر خط (برای پردازشگرهای لاگ).

### Redaction

OpenClaw می‌تواند tokenهای حساس را پیش از رسیدن به خروجی کنسول، لاگ‌های فایل، رکوردهای لاگ OTLP، متن transcript نشست ذخیره‌شده، یا payloadهای رویداد ابزار در Control UI، redaction کند (آرگومان‌های شروع ابزار، payloadهای نتیجه جزئی/نهایی، خروجی exec مشتق‌شده، و خلاصه‌های patch):

- `logging.redactSensitive`: `off` | `tools` (پیش‌فرض: `tools`)
- `logging.redactPatterns`: فهرست رشته‌های regex برای بازنویسی مجموعه پیش‌فرض. الگوهای سفارشی روی پیش‌فرض‌های داخلی برای payloadهای ابزار در Control UI اعمال می‌شوند، بنابراین افزودن یک الگو هرگز redaction مقدارهایی را که پیش‌فرض‌ها از قبل می‌گیرند ضعیف نمی‌کند.

لاگ‌های فایل و transcriptهای نشست همچنان JSONL می‌مانند، اما مقدارهای secret منطبق پیش از نوشته‌شدن خط یا پیام روی دیسک mask می‌شوند. Redaction به‌صورت best-effort است: روی محتوای پیام متنی و رشته‌های لاگ اعمال می‌شود، نه روی هر فیلد شناسه یا payload دودویی.

پیش‌فرض‌های داخلی اعتبارنامه‌های رایج API و نام فیلدهای اعتبارنامه پرداخت مانند شماره کارت، CVC/CVV، token پرداخت مشترک، و اعتبارنامه پرداخت را وقتی به‌صورت فیلدهای JSON، پارامترهای URL، پرچم‌های CLI، یا assignment ظاهر شوند پوشش می‌دهند.

`logging.redactSensitive: "off"` فقط این سیاست عمومی لاگ/transcript را غیرفعال می‌کند. OpenClaw همچنان payloadهای مرز ایمنی را که می‌توانند به کلاینت‌های UI، support bundleها، مشاهده‌گرهای تشخیصی، promptهای تأیید، یا ابزارهای عامل نشان داده شوند redaction می‌کند. نمونه‌ها شامل رویدادهای فراخوانی ابزار در Control UI، خروجی `sessions_history`، exportهای پشتیبانی تشخیصی، مشاهده‌های خطای ارائه‌دهنده، نمایش دستور تأیید exec، و لاگ‌های پروتکل WebSocket مربوط به Gateway هستند. `logging.redactPatterns` سفارشی همچنان می‌تواند الگوهای خاص پروژه را روی آن سطح‌ها اضافه کند.

## تشخیص‌ها و OpenTelemetry

تشخیص‌ها رویدادهای ساخت‌یافته و قابل خواندن توسط ماشین برای اجراهای مدل و telemetry جریان پیام هستند (webhookها، صف‌گذاری، وضعیت نشست). آن‌ها جایگزین لاگ‌ها **نمی‌شوند**؛ آن‌ها metricها، traceها، و exporterها را تغذیه می‌کنند. رویدادها چه آن‌ها را export کنید چه نکنید، درون فرایند منتشر می‌شوند.

دو سطح مجاور:

- **export OpenTelemetry** — ارسال metricها، traceها، و لاگ‌ها از طریق OTLP/HTTP به هر collector یا backend سازگار با OpenTelemetry (Grafana، Datadog، Honeycomb، New Relic، Tempo، و غیره). پیکربندی کامل، catalog سیگنال‌ها، نام‌های metric/span، متغیرهای محیطی، و مدل حریم خصوصی در یک صفحه اختصاصی قرار دارد: [export OpenTelemetry](/fa/gateway/opentelemetry).
- **پرچم‌های تشخیص** — پرچم‌های هدفمند debug-log که لاگ‌های اضافی را بدون افزایش `logging.level` به `logging.file` هدایت می‌کنند. پرچم‌ها به بزرگی و کوچکی حروف حساس نیستند و wildcardها (`telegram.*`, `*`) را پشتیبانی می‌کنند. زیر `diagnostics.flags` یا از طریق override محیطی `OPENCLAW_DIAGNOSTICS=...` پیکربندی کنید. راهنمای کامل: [پرچم‌های تشخیص](/fa/diagnostics/flags).

برای فعال‌کردن رویدادهای تشخیصی برای Pluginها یا sinkهای سفارشی بدون export OTLP:

```json5
{
  diagnostics: { enabled: true },
}
```

برای export OTLP به یک collector، [export OpenTelemetry](/fa/gateway/opentelemetry) را ببینید.

## نکته‌های عیب‌یابی

- **Gateway در دسترس نیست؟** ابتدا `openclaw doctor` را اجرا کنید.
- **لاگ‌ها خالی هستند؟** بررسی کنید Gateway در حال اجراست و در مسیر فایل داخل `logging.file` می‌نویسد.
- **به جزئیات بیشتری نیاز دارید؟** `logging.level` را روی `debug` یا `trace` بگذارید و دوباره تلاش کنید.
