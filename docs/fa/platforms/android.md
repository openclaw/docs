## نصب خارج از Google Play

انتشارهای نهایی و اصلاحی معمول GitHub شامل یک `OpenClaw-Android.apk` همگانی و `OpenClaw-Android-SHA256SUMS.txt` هستند. APK از برچسب انتشار ساخته می‌شود، با کلید انتشار Android متعلق به OpenClaw امضا می‌شود و گواهی منشأ GitHub Actions را همراه دارد.

یک [انتشار](https://github.com/openclaw/openclaw/releases) را انتخاب کنید که هر دو دارایی را فهرست کرده باشد، سپس پیش از نصب جانبی، همان برچسب دقیق را دانلود و تأیید کنید:

```bash
release_tag=vYYYY.M.PATCH
gh release download "$release_tag" \
  --repo openclaw/openclaw \
  --pattern OpenClaw-Android.apk \
  --pattern OpenClaw-Android-SHA256SUMS.txt
sha256sum --check OpenClaw-Android-SHA256SUMS.txt
gh attestation verify OpenClaw-Android.apk \
  --repo openclaw/openclaw \
  --signer-workflow openclaw/openclaw/.github/workflows/android-release.yml \
  --source-ref "refs/tags/${release_tag}" \
  --deny-self-hosted-runners
```

<Warning>
نصب‌های Google Play و APK مستقل از کانال‌های به‌روزرسانی متفاوتی استفاده می‌کنند و ممکن است هویت‌های امضای متفاوتی داشته باشند. برای جابه‌جایی بین کانال‌ها، ممکن است Android حذف برنامه موجود را الزامی کند که داده‌های محلی برنامه را پاک می‌کند. برای به‌روزرسانی‌های معمول در یک کانال بمانید.
</Warning>

## بازتاب و کنترل Android از یک Mac راه دور

[scrcpy](https://github.com/Genymobile/scrcpy) صفحه Android را در یک پنجره macOS بازتاب می‌دهد و
ورودی صفحه‌کلید و اشاره‌گر را از طریق Android Debug Bridge (ADB) منتقل می‌کند. این یک گردش‌کار
سمت اپراتور و جدا از اتصال Node در OpenClaw است. هنگامی مفید است که دستگاه Android و
Mac در مکان‌های متفاوتی باشند اما یک شبکه خصوصی Tailscale مشترک داشته باشند.

### پیش از شروع

- Tailscale را روی دستگاه Android و Mac نصب کنید و هر دو را به یک tailnet متصل کنید.
- در Android، **Developer options** و **USB debugging** را فعال کنید. Android 16 گزینه **Wireless
  debugging** را در مسیر **Settings > System > Developer options** قرار می‌دهد. [گزینه‌های توسعه‌دهنده
  Android](https://developer.android.com/studio/debug/dev-options) را ببینید.
- scrcpy و ADB را روی Mac نصب کنید:

  ```bash
  brew install scrcpy
  brew install --cask android-platform-tools
  ```

- دستگاه Android را برای نخستین اتصال در دسترس نگه دارید. پیش از آنکه هر Mac بتواند دستگاه را کنترل کند،
  Android باید کلید ADB آن Mac را تأیید کند.

### فعال‌سازی ADB روی TCP

برای راه‌اندازی اولیه، دستگاه Android را با USB به یک رایانه مورداعتماد متصل کنید و درخواست
اشکال‌زدایی آن را تأیید کنید. سپس اجرا کنید:

```bash
adb devices
adb tcpip 5555
```

اکنون می‌توانید USB را جدا کنید. اگر پورت 5555 پس از راه‌اندازی مجدد دستگاه یا بازنشانی اشکال‌زدایی
دیگر در حال شنود نبود، این مرحله راه‌اندازی محلی را تکرار کنید. Android 11 و نسخه‌های جدیدتر همچنین می‌توانند اعتماد اولیه را با
**Wireless debugging > Pair device with pairing code** و `adb pair` برقرار کنند.

### اجازه دسترسی فقط به Mac کنترل‌کننده

tailnetهایی با مجوزهای محدودکننده باید صراحتاً به Mac کنترل‌کننده اجازه دهند به پورت TCP شماره 5555
روی دستگاه Android دسترسی پیدا کند. یک قانون محدود به خط‌مشی tailnet اضافه کنید و نشانی‌های نمونه
را با IPهای پایدار Tailscale دو دستگاه جایگزین کنید:

```json5
{
  grants: [
    {
      src: ["<remote-mac-tailnet-ip>"],
      dst: ["<android-tailnet-ip>"],
      ip: ["tcp:5555"],
    },
  ],
}
```

برای نام‌های مستعار میزبان و سایر انتخاب‌گرها، [مجوزهای Tailscale](https://tailscale.com/docs/reference/syntax/grants) را ببینید.
این پورت را در اختیار اینترنت عمومی قرار ندهید یا با Funnel در معرض دسترسی نگذارید: یک کلاینت مجاز ADB
کنترل گسترده‌ای بر دستگاه دارد.

### اتصال و آغاز بازتاب

روی Mac راه دور:

```bash
adb connect <android-tailnet-ip>:5555
adb devices
scrcpy --serial <android-tailnet-ip>:5555
```

نخستین `adb connect` از این Mac یک کادر مجوز در Android نمایش می‌دهد. قفل دستگاه را باز کنید،
اثر انگشت کلید را تأیید کنید و فقط در صورتی که Mac مورداعتماد است، **Always allow from this computer** را انتخاب کنید.
یک ورودی موفق `adb devices` با `device` پایان می‌یابد؛ `unauthorized` یعنی درخواست روی دستگاه
تأیید نشده است.

پس از باز شدن پنجره scrcpy، مستقیماً از آن استفاده کنید یا آن را با یک ابزار خودکارسازی صفحه macOS مانند
[Peekaboo](https://peekaboo.sh/) هدف بگیرید. scrcpy نمایشگر و ورودی را منتقل می‌کند؛ Tailscale فقط
مسیر شبکه خصوصی را فراهم می‌کند.

### عیب‌یابی

- `Connection timed out`: مجوز tailnet برای TCP 5555 را بررسی کنید. یک `tailscale ping` موفق،
  دسترس‌پذیری همتا را اثبات می‌کند، نه اینکه خط‌مشی این پورت TCP را مجاز می‌داند. از Mac با
  `nc -vz <android-tailnet-ip> 5555` آزمایش کنید.
- `unauthorized`: قفل Android را باز و کلید ADB مربوط به Mac راه دور را تأیید کنید، یا ایستگاه کاری قدیمی را
  در **Wireless debugging > Paired devices** حذف و دوباره جفت کنید.
- `Connection refused`: دوباره به‌صورت محلی متصل شوید و `adb tcpip 5555` را مجدداً اجرا کنید.
- بیش از یک دستگاه فهرست شده است: آرگومان صریح `--serial <android-tailnet-ip>:5555` را حفظ کنید.

پس از پایان کار، scrcpy را ببندید و اتصال ADB را قطع کنید:

```bash
adb disconnect <android-tailnet-ip>:5555
```

## راهنمای عملیاتی اتصال

برنامه Node در Android ⇄ (mDNS/NSD + WebSocket) ⇄ **Gateway**

Android مستقیماً به WebSocket متعلق به Gateway متصل می‌شود و از جفت‌سازی دستگاه (`role: node`) استفاده می‌کند.

برای میزبان‌های Tailscale یا عمومی، Android به یک نقطه پایانی امن نیاز دارد:

- ترجیحی: Tailscale Serve / Funnel با `https://<magicdns>` / `wss://<magicdns>`
- همچنین پشتیبانی می‌شود: هر URL دیگر `wss://` برای Gateway با یک نقطه پایانی TLS واقعی
- اتصال بدون رمزنگاری `ws://` همچنان در نشانی‌های LAN خصوصی / میزبان‌های `.local`، به‌علاوه `localhost`، `127.0.0.1` و پل شبیه‌ساز Android (`10.0.2.2`) پشتیبانی می‌شود؛ راه‌اندازی غیر-loopback به‌طور خودکار از دسترسی محدود اپراتور استفاده می‌کند

### پیش‌نیازها

- Gateway روی دستگاه دیگری در حال اجرا باشد (یا از طریق SSH قابل‌دسترسی باشد).
- دستگاه/شبیه‌ساز Android بتواند به WebSocket متعلق به Gateway دسترسی پیدا کند:
  - همان LAN با mDNS/NSD، **یا**
  - همان tailnet متعلق به Tailscale با استفاده از Wide-Area Bonjour / DNS-SD تک‌پخشی (پایین را ببینید)، **یا**
  - میزبان/پورت دستی Gateway (راهکار جایگزین)
- جفت‌سازی موبایل در tailnet/عمومی از نقاط پایانی خام IP مربوط به tailnet یعنی `ws://` استفاده **نمی‌کند**. به‌جای آن از Tailscale Serve یا یک URL دیگر `wss://` استفاده کنید.
- CLI با نام `openclaw` روی دستگاه Gateway (یا از طریق SSH) برای تأیید درخواست‌های جفت‌سازی در دسترس باشد.

### 1. راه‌اندازی Gateway

```bash
openclaw gateway --port 18789 --verbose
```

در گزارش‌ها تأیید کنید که چیزی شبیه این می‌بینید:

- `listening on ws://0.0.0.0:18789`

برای دسترسی راه دور Android از طریق Tailscale، به‌جای اتصال خام tailnet، Serve/Funnel را ترجیح دهید:

```bash
openclaw gateway --tailscale serve
```

این کار یک نقطه پایانی امن `wss://` / `https://` در اختیار Android قرار می‌دهد. یک راه‌اندازی ساده `gateway.bind: "tailnet"` برای نخستین جفت‌سازی راه دور Android کافی نیست، مگر اینکه TLS را نیز جداگانه خاتمه دهید.

### 2. تأیید کشف (اختیاری)

از دستگاه Gateway:

```bash
dns-sd -B _openclaw-gw._tcp local.
```

نکات بیشتر برای اشکال‌زدایی: [Bonjour](/fa/gateway/bonjour).

اگر یک دامنه کشف گسترده نیز پیکربندی کرده‌اید، با این مورد مقایسه کنید:

```bash
openclaw gateway discover --json
```

این دستور `local.` و دامنه گسترده پیکربندی‌شده را در یک مرحله نمایش می‌دهد و به‌جای راهنمایی‌های صرفاً TXT، از نقطه پایانی سرویس حل‌شده استفاده می‌کند.

#### کشف بین‌شبکه‌ای از طریق DNS-SD تک‌پخشی

کشف NSD/mDNS در Android از شبکه‌ها عبور نمی‌کند. اگر Node در Android و Gateway روی شبکه‌های متفاوتی هستند اما از طریق Tailscale متصل‌اند، به‌جای آن از Wide-Area Bonjour / DNS-SD تک‌پخشی استفاده کنید. کشف به‌تنهایی برای جفت‌سازی Android در tailnet/عمومی کافی نیست — مسیر کشف‌شده همچنان به یک نقطه پایانی امن نیاز دارد (`wss://` یا Tailscale Serve):

1. یک ناحیه DNS-SD (برای نمونه `openclaw.internal.`) روی میزبان Gateway راه‌اندازی و رکوردهای `_openclaw-gw._tcp` را منتشر کنید.
2. DNS تفکیکی Tailscale را برای دامنه انتخابی خود پیکربندی کنید تا به آن سرور DNS اشاره کند.

جزئیات و نمونه پیکربندی CoreDNS: [Bonjour](/fa/gateway/bonjour).

### 3. اتصال از Android

در برنامه Android:

- برنامه اتصال خود به Gateway را از طریق یک **foreground service** (اعلان پایدار) زنده نگه می‌دارد.
- زبانه **Connect** را باز کنید.
- از حالت **Setup Code** یا **Manual** استفاده کنید.
- اگر کشف مسدود است، از میزبان/پورت دستی در **Advanced controls** استفاده کنید. برای میزبان‌های LAN خصوصی، `ws://` همچنان کار می‌کند. برای میزبان‌های Tailscale/عمومی، TLS را روشن کنید و از نقطه پایانی `wss://` / Tailscale Serve استفاده کنید.

پس از نخستین جفت‌سازی موفق، Android هنگام اجرا به‌طور خودکار به Gateway جفت‌شده فعال وصل می‌شود (برای Gatewayهای کشف‌شده به‌صورت بهترین تلاش؛ آن‌ها باید در شبکه قابل‌مشاهده باشند).

کدهای راه‌اندازی رسمی، Android را به‌عنوان یک Node متصل می‌کنند و به‌طور پیش‌فرض
از طریق `wss://` دسترسی کامل اپراتور Gateway را اعطا می‌کنند. راه‌اندازی متنی ساده
غیر-loopback با `ws://` برای ایمنی توکن حامل، به‌طور خودکار از دسترسی محدود استفاده می‌کند. بخش **Settings → Gateway**
دسترسی **Full** یا **Limited** را نشان می‌دهد. برای یک اتصال محدود، `wss://`
یا Tailscale Serve را پیکربندی کنید، یک کد جدید با دسترسی کامل در رابط کنترل یا
با `openclaw qr` ایجاد کنید، سپس آن را در آن صفحه اسکن یا جای‌گذاری و دوباره متصل شوید. اپراتورهایی
که نمایه محدودتر را می‌خواهند، می‌توانند **Limited access** را در رابط کنترل انتخاب کنند یا
`openclaw qr --limited` را اجرا کنند.

### چند Gateway

برنامه یک رجیستری از تمام Gatewayهایی که با آن‌ها جفت شده است نگه می‌دارد، بنابراین می‌توانید بدون جفت‌سازی مجدد بین آن‌ها جابه‌جا شوید:

- **Settings -> Gateways**، Gatewayهای جفت‌شده را فهرست می‌کند و مورد فعال را علامت می‌زند. برای جابه‌جایی روی یک ورودی بزنید؛ برنامه نشست‌های فعلی را خاتمه می‌دهد و دوباره به Gateway انتخاب‌شده متصل می‌شود.
- هنگامی که بیش از یک Gateway جفت شده باشد، زبانه **Connect** یک جابه‌جاکننده سریع نمایش می‌دهد.
- اعتبارنامه‌ها، توکن‌های دستگاه، اعتماد TLS، تاریخچه گفت‌وگو و پیام‌های آفلاین در صف، به‌ازای هر Gateway ذخیره می‌شوند. جابه‌جایی هرگز وضعیت Gatewayها را با هم ترکیب نمی‌کند و پیام‌هایی که هنگام آفلاین بودن در صف قرار گرفته‌اند فقط به Gateway مقصد خود تحویل داده می‌شوند.
- **Forget** ورودی رجیستری یک Gateway را همراه با اعتبارنامه‌ها، توکن‌های دستگاه، پین TLS و گفت‌وگوهای ذخیره‌شده در حافظه نهان حذف می‌کند.

### بیکن‌های زنده‌بودن حضور

پس از اتصال نشست احراز هویت‌شده Node، و هنگامی که برنامه در حالی به پس‌زمینه می‌رود که سرویس پیش‌زمینه هنوز متصل است، Android تابع `node.event` را با `event: "node.presence.alive"` فراخوانی می‌کند. Gateway تنها پس از مشخص شدن هویت دستگاه احراز هویت‌شده Node، این مورد را به‌شکل `lastSeenAtMs`/`lastSeenReason` در فراداده Node/دستگاه جفت‌شده ثبت می‌کند.

برنامه تنها زمانی بیکن را با موفقیت ثبت‌شده محسوب می‌کند که پاسخ Gateway شامل `handled: true` باشد. Gatewayهای قدیمی‌تر ممکن است `node.event` را با `{ "ok": true }` تأیید کنند؛ این پاسخ سازگار است، اما به‌عنوان به‌روزرسانی پایدار آخرین مشاهده محسوب نمی‌شود.

### 4. تأیید جفت‌سازی (CLI)

روی دستگاه Gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

جزئیات جفت‌سازی: [جفت‌سازی](/fa/channels/pairing).

اختیاری: اگر Node اندروید همیشه از یک زیرشبکهٔ کاملاً کنترل‌شده متصل می‌شود، می‌توانید با CIDRهای صریح یا IPهای دقیق، تأیید خودکار نخستین جفت‌سازی Node را فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این قابلیت به‌طور پیش‌فرض غیرفعال است. فقط برای جفت‌سازی جدید `role: node` بدون دامنه‌های درخواستی اعمال می‌شود. جفت‌سازی اپراتور/مرورگر و هرگونه تغییر نقش، دامنه، فراداده یا کلید عمومی همچنان به تأیید دستی نیاز دارد.

### 5. بررسی اتصال Node

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

### 6. گفت‌وگو + تاریخچه

زبانهٔ گفت‌وگوی اندروید از انتخاب نشست پشتیبانی می‌کند (نشست پیش‌فرض `main`، به‌علاوهٔ سایر نشست‌های موجود):

- تاریخچه: `chat.history` (برای نمایش عادی‌سازی می‌شود — برچسب‌های دستور درون‌خطی، محتوای XML متن سادهٔ فراخوانی ابزار (`<tool_call>`، `<function_call>`، `<tool_calls>`، `<function_calls>` و گونه‌های کوتاه‌شده) و توکن‌های کنترلی ASCII/تمام‌عرض مدل که نشت کرده‌اند حذف می‌شوند؛ ردیف‌های دستیار دارای توکن خاموش، مانند مقادیر دقیق `NO_REPLY` / `no_reply`، کنار گذاشته می‌شوند؛ ردیف‌های بیش‌ازحد بزرگ ممکن است با جای‌نگهدار جایگزین شوند)
- ارسال: `chat.send`
- ارسال پایدار: هر ارسال (متن، تصاویر انتخاب‌شده و یادداشت‌های صوتی) پیش از هر تلاش شبکه‌ای در صندوق خروجی روی‌دستگاه و مختص هر Gateway ثبت می‌شود؛ بنابراین خاتمهٔ برنامه نمی‌تواند ورودی ارسال‌شده را از بین ببرد. ارسال‌هایی که هنگام آفلاین‌بودن در صف قرار گرفته‌اند، پس از اتصال مجدد با کلیدهای پایدار هم‌توانی به‌ترتیب تحویل می‌شوند و یک ارسال فقط زمانی از صف خارج می‌شود که نوبت در `chat.history` مرجع قابل‌مشاهده باشد — صرفاً دریافت تأیید به‌عنوان اثبات تحویل در نظر گرفته نمی‌شود. پیامدهای مبهم (از‌دست‌رفتن تأیید، بسته‌شدن برنامه در میانهٔ ارسال یا راه‌اندازی مجدد Gateway پیش از نوشتن رونوشت) به‌جای ارسال مجدد خودکار، به‌صورت ردیف‌های قابل‌مشاهده با گزینه‌های صریح **تلاش مجدد**/**حذف** نمایش داده می‌شوند. فرمان‌های اسلش‌دار هرگز پس از اتصال مجدد به‌طور خودکار بازپخش نمی‌شوند؛ آن‌ها برای تلاش مجدد صریح متوقف می‌مانند. صف محدود است (50 پیام و 48 MB دادهٔ پیوست برای هر Gateway) و ردیف‌های ارسال‌نشده پس از 48 ساعت منقضی می‌شوند. پیش‌نویس‌های کادر نوشتن که هرگز ارسال نشده‌اند، در برابر خاتمهٔ فرایند پایدار نیستند.
- به‌روزرسانی‌های ارسالی (در حد تلاش): `chat.subscribe` -> `event:"chat"`
- شنیدن: یک پیام دستیار را طولانی فشار دهید و **شنیدن** را انتخاب کنید؛ صدا از طریق `tts.speak` در Gateway و با زنجیرهٔ پیکربندی‌شدهٔ ارائه‌دهندهٔ TTS تولید می‌شود و وقتی Gateway نتواند صدا را تولید کند، از TTS سیستمی روی دستگاه استفاده می‌شود. پخش با تعویض نشست، گفت‌وگوی جدید، رفتن برنامه به پس‌زمینه یا بستن گفت‌وگو متوقف می‌شود.

### 7. بوم + دوربین

#### میزبان بوم Gateway (توصیه‌شده برای محتوای وب)

برای اینکه Node محتوای واقعی HTML/CSS/JS را نمایش دهد که عامل بتواند آن را روی دیسک ویرایش کند، Node را به میزبان بوم Gateway هدایت کنید.

<Note>
Nodeها بوم را از سرور HTTP در Gateway بارگیری می‌کنند (همان درگاه `gateway.port`، مقدار پیش‌فرض `18789`).
</Note>

1. در میزبان Gateway، `~/.openclaw/workspace/canvas/index.html` را ایجاد کنید.
2. Node را به آن هدایت کنید (LAN):

```bash
openclaw nodes invoke --node "<Android Node>" --command canvas.navigate --params '{"url":"http://<gateway-hostname>.local:18789/__openclaw__/canvas/"}'
```

Tailnet (اختیاری): اگر هر دو دستگاه روی Tailscale هستند، به‌جای `.local` از نام MagicDNS یا IP در tailnet استفاده کنید؛ برای مثال `http://<gateway-magicdns>:18789/__openclaw__/canvas/`.

این سرور یک کارخواه بازبارگذاری زنده را در HTML تزریق می‌کند و با تغییر فایل‌ها صفحه را بازبارگذاری می‌کند. Gateway همچنین `/__openclaw__/a2ui/` را ارائه می‌کند، اما برنامهٔ اندروید صفحات دوردست A2UI را فقط برای رندر در نظر می‌گیرد. فرمان‌های A2UI دارای قابلیت اقدام از صفحهٔ A2UI همراه و متعلق به برنامه استفاده می‌کنند.

فرمان‌های بوم (فقط در پیش‌زمینه):

- `canvas.eval`، `canvas.snapshot`، `canvas.navigate` (برای بازگشت به داربست پیش‌فرض از `{"url":""}` یا `{"url":"/"}` استفاده کنید). `canvas.snapshot` مقدار `{ format, base64 }` را برمی‌گرداند (مقدار پیش‌فرض `format="jpeg"`).
- A2UI: `canvas.a2ui.push`، `canvas.a2ui.reset` (نام مستعار قدیمی `canvas.a2ui.pushJSONL`). این فرمان‌ها برای رندر دارای قابلیت اقدام از صفحهٔ A2UI همراه و متعلق به برنامه استفاده می‌کنند.

فرمان‌های دوربین (فقط در پیش‌زمینه؛ مشروط به مجوز): `camera.snap` (jpg)، `camera.clip` (mp4). برای پارامترها و ابزارهای کمکی CLI به [Node دوربین](/fa/nodes/camera) مراجعه کنید.

### 8. صدا + سطح گستردهٔ فرمان‌های اندروید

- زبانهٔ صدا: اندروید دو حالت صریح ضبط دارد. **میکروفون** یک نشست دستی در زبانهٔ صدا است که هر مکث را به‌صورت یک نوبت گفت‌وگو ارسال می‌کند و وقتی برنامه از پیش‌زمینه خارج شود یا کاربر زبانهٔ صدا را ترک کند، متوقف می‌شود. **گفت‌وگو** حالت گفت‌وگوی پیوسته است و تا زمان خاموش‌شدن یا قطع اتصال Node به شنیدن ادامه می‌دهد.
- حالت گفت‌وگو، سرویس پیش‌زمینهٔ موجود را پیش از آغاز ضبط از `connectedDevice` به `connectedDevice|microphone` ارتقا می‌دهد و پس از توقف حالت گفت‌وگو آن را تنزل می‌دهد. سرویس Node، `FOREGROUND_SERVICE_CONNECTED_DEVICE` را با `CHANGE_NETWORK_STATE` اعلام می‌کند؛ اندروید 14+ همچنین به اعلان `FOREGROUND_SERVICE_MICROPHONE`، اعطای مجوز زمان اجرای `RECORD_AUDIO` و نوع سرویس میکروفون در زمان اجرا نیاز دارد.
- به‌طور پیش‌فرض، گفت‌وگوی اندروید از تشخیص گفتار بومی، گفت‌وگوی Gateway و `talk.speak` از طریق ارائه‌دهندهٔ پیکربندی‌شدهٔ گفت‌وگو در Gateway استفاده می‌کند. فقط هنگامی که `talk.speak` در دسترس نباشد، از TTS سیستمی محلی استفاده می‌شود.
- گفت‌وگوی اندروید تنها زمانی از رلهٔ بی‌درنگ Gateway استفاده می‌کند که `talk.realtime.mode` برابر با `realtime` و `talk.realtime.transport` برابر با `gateway-relay` باشد.
- اندروید قابلیت `voiceWake` را اعلام نمی‌کند. برای ورودی صوتی از **میکروفون** یا **گفت‌وگو** استفاده کنید.
- خانواده‌های فرمان اضافی اندروید (دسترس‌پذیری به دستگاه، مجوزها و تنظیمات کاربر بستگی دارد):
  - `device.status`، `device.info`، `device.permissions`، `device.health`
  - `device.apps` فقط زمانی که **Settings > Phone Capabilities > Installed Apps** فعال باشد؛ به‌طور پیش‌فرض برنامه‌های قابل‌مشاهده در راه‌انداز را فهرست می‌کند (برای فهرست کامل، `includeNonLaunchable` را ارسال کنید).
  - `notifications.list`، `notifications.actions` (بخش [هدایت اعلان‌ها](#notification-forwarding) را در ادامه ببینید)
  - `photos.latest`
  - `contacts.search`، `contacts.add`
  - `calendar.events`، `calendar.add`
  - `callLog.search`
  - `sms.search`
  - `motion.activity`، `motion.pedometer`

### 9. فایل‌های فضای کاری (فقط خواندنی)

نمای کلی خانه شامل کارت **فایل‌ها** است که از طریق RPCهای فقط‌خواندنی `agents.workspace.list` / `agents.workspace.get` در Gateway، فضای کاری عامل فعال را مرور می‌کند: پیمایش سلسله‌مراتبی پوشه‌ها، پیش‌نمایش متن و تصویر و صادرکردن از طریق صفحهٔ اشتراک‌گذاری اندروید. هیچ عملیات نوشتنی وجود ندارد و اندازهٔ پیش‌نمایش‌ها توسط Gateway محدود می‌شود.

## بررسی تأیید فرمان‌ها

یک اتصال اپراتور با `operator.admin`، یا یک اتصال جفت‌شدهٔ
`operator.approvals` که Gateway صریحاً آن را هدف قرار داده است، می‌تواند
درخواست‌های اجرای در انتظار را در **Settings -> Approvals** بررسی کند. برنامه پیش از فعال‌کردن
دکمه‌های خود، رکورد پاک‌سازی‌شدهٔ تأیید را از Gateway بارگیری می‌کند، هرگونه
هشدار امنیتی و تصمیم‌های دقیقی را که آن درخواست ارائه می‌دهد نمایش می‌دهد و
شناسهٔ تأیید و نوع مالک را به Gateway ارسال می‌کند.

وضعیت تأیید با رابط کاربری کنترل و سطوح گفت‌وگوی پشتیبانی‌شده مشترک است. نخستین
پاسخ ثبت‌شده برنده است؛ اندروید نتیجهٔ مرجع را حتی هنگامی نمایش می‌دهد که
سطح دیگری زودتر پاسخ داده باشد. اگر پاسخ حل‌وفصل از دست برود یا اتصال Gateway
قطع شود، برنامه اقدام را قفل نگه می‌دارد و پیش از ارائهٔ تصمیمی دیگر،
تأیید را دوباره می‌خواند.

Gatewayهایی که پیش از روش‌های یکپارچهٔ تأیید ارائه شده‌اند، به روش‌های ارسال‌شدهٔ
مختص اجرا بازمی‌گردند. بررسی موارد در انتظار همچنان کار می‌کند، اما وضعیت نگه‌داری‌شدهٔ پایانه
و نتیجهٔ غنی‌تر میان‌سطحی به Gateway به‌روزشده نیاز دارند.

## نقاط ورود دستیار

اندروید از راه‌اندازی OpenClaw از طریق محرک دستیار سیستم (Google Assistant) پشتیبانی می‌کند. نگه‌داشتن دکمهٔ خانه (یا محرک دیگری از نوع `ACTION_ASSIST`) برنامه را باز می‌کند؛ گفتن «Hey Google, ask OpenClaw `<prompt>`» با الگوی جست‌وجوی App Actions اعلام‌شدهٔ برنامه مطابقت پیدا می‌کند و درخواست را بدون ارسال خودکار به کادر نوشتن گفت‌وگو منتقل می‌کند.

این قابلیت از **App Actions** اندروید (قابلیت `shortcuts.xml`) استفاده می‌کند که در مانیفست برنامه اعلام شده است. هیچ پیکربندی سمت Gateway لازم نیست — intent دستیار کاملاً توسط برنامهٔ اندروید مدیریت می‌شود.

<Note>
دسترس‌پذیری App Actions به دستگاه، نسخهٔ Google Play Services و اینکه آیا کاربر OpenClaw را به‌عنوان برنامهٔ دستیار پیش‌فرض تنظیم کرده است یا نه بستگی دارد.
</Note>

## هدایت اعلان‌ها

اندروید می‌تواند اعلان‌های دستگاه را به‌صورت موارد `node.event` به Gateway هدایت کند. این قابلیت **روی دستگاه** و در صفحهٔ تنظیمات برنامه پیکربندی می‌شود — نه در پیکربندی gateway/`openclaw.json`.

| تنظیم                     | توضیحات                                                                                                                                                                                            |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Forward Notification Events | کلید اصلی. به‌طور پیش‌فرض خاموش است؛ ابتدا باید Notification Listener Access اعطا شود.                                                                                                              |
| Package Filter              | **Allowlist** (فقط شناسه‌های بستهٔ فهرست‌شده هدایت می‌شوند) یا **Blocklist** (پیش‌فرض: همهٔ بسته‌ها به‌جز شناسه‌های فهرست‌شده). بستهٔ خود OpenClaw همیشه در حالت Blocklist مستثنا می‌شود تا از حلقه‌های هدایت جلوگیری شود. |
| Quiet Hours                 | بازهٔ محلی شروع/پایان با قالب HH:mm که هدایت را متوقف می‌کند. به‌طور پیش‌فرض غیرفعال است؛ پس از فعال‌سازی، مقدار پیش‌فرض `22:00`-`07:00` است.                                                                                |
| Max Events / Minute         | محدودیت نرخ اعلان‌های هدایت‌شده برای هر دستگاه. مقدار پیش‌فرض 20 است.                                                                                                                                          |
| Route Session Key           | اختیاری. رویدادهای اعلان هدایت‌شده را به‌جای مسیر پیش‌فرض اعلان دستگاه، به یک نشست مشخص متصل می‌کند.                                                                               |

<Note>
هدایت اعلان‌ها به مجوز Notification Listener اندروید نیاز دارد. برنامه هنگام راه‌اندازی این مجوز را درخواست می‌کند.
</Note>

اعلان‌های WhatsApp، WhatsApp Business، Telegram، Telegram X، Discord و Signal همیشه مستثنا هستند. پیام‌های آن‌ها از قبل در مالکیت نشست‌های بومی کانال OpenClaw قرار دارند؛ هدایت اعلان اندروید به‌عنوان یک رویداد جداگانهٔ Node ممکن است پاسخ را از طریق گفت‌وگوی نادرست مسیریابی کند.
