OpenClaw می‌تواند **PeekabooBridge** را به‌عنوان یک واسط خودکارسازی رابط کاربری محلی و آگاه از مجوزها میزبانی کند. این امکان به CLIِ `peekaboo` اجازه می‌دهد خودکارسازی رابط کاربری را اجرا کند، در حالی که از مجوزهای TCC برنامه macOS دوباره استفاده می‌کند.

## این چیست (و چه چیزی نیست)

- **میزبان**: OpenClaw.app می‌تواند به‌عنوان میزبان PeekabooBridge عمل کند.
- **کلاینت**: از CLIِ `peekaboo` استفاده کنید (هیچ سطح جداگانه `openclaw ui ...` وجود ندارد).
- **رابط کاربری**: هم‌پوشانی‌های بصری در Peekaboo.app باقی می‌مانند؛ OpenClaw فقط یک میزبان واسط سبک است.

## رابطه با استفاده از رایانه

OpenClaw سه مسیر کنترل دسکتاپ دارد و آن‌ها عمداً جدا از هم باقی می‌مانند:

- **میزبان PeekabooBridge**: OpenClaw.app می‌تواند سوکت محلی PeekabooBridge را میزبانی کند.
  CLIِ `peekaboo` همچنان کلاینت می‌ماند و برای ابتداییات خودکارسازی Peekaboo مانند اسکرین‌شات‌ها، کلیک‌ها، منوها، دیالوگ‌ها، اقدامات Dock و مدیریت پنجره، از مجوزهای macOS برنامه OpenClaw.app استفاده می‌کند.
- **استفاده از رایانه Codex**: Plugin بسته‌بندی‌شده `codex`، سرور برنامه Codex را آماده می‌کند، بررسی می‌کند که سرور MCPِ `computer-use` مربوط به Codex در دسترس باشد، و سپس اجازه می‌دهد Codex در نوبت‌های حالت Codex مالک فراخوانی‌های ابزار کنترل دسکتاپ بومی باشد. OpenClaw این اقدامات را از طریق PeekabooBridge پروکسی نمی‌کند.
- **MCP مستقیم `cua-driver`**: OpenClaw می‌تواند سرور بالادستی `cua-driver mcp` مربوط به TryCua را به‌عنوان یک سرور MCP عادی ثبت کند. این کار طرحواره‌های خود CUA driver و گردش کار pid/پنجره/شاخص-عنصر را بدون مسیریابی از طریق بازار Codex یا سوکت PeekabooBridge در اختیار عامل‌ها قرار می‌دهد.

وقتی سطح گسترده خودکارسازی macOS و میزبان پل آگاه از مجوزِ OpenClaw.app را می‌خواهید، از Peekaboo استفاده کنید. وقتی یک عامل حالت Codex باید به Plugin بومی استفاده از رایانه Codex متکی باشد، از استفاده از رایانه Codex استفاده کنید. وقتی می‌خواهید CUA driver به‌عنوان یک سرور MCP عادی در معرض هر زمان اجرای مدیریت‌شده توسط OpenClaw قرار گیرد، از `cua-driver mcp` مستقیم استفاده کنید.

## فعال‌سازی پل

در برنامه macOS:

- Settings → **فعال‌سازی Peekaboo Bridge**

وقتی فعال باشد، OpenClaw یک سرور سوکت محلی UNIX راه‌اندازی می‌کند. اگر غیرفعال شود، میزبان متوقف می‌شود و `peekaboo` به میزبان‌های در دسترس دیگر بازمی‌گردد.

## ترتیب کشف کلاینت

کلاینت‌های Peekaboo معمولاً میزبان‌ها را به این ترتیب امتحان می‌کنند:

1. Peekaboo.app (تجربه کاربری کامل)
2. Claude.app (اگر نصب شده باشد)
3. OpenClaw.app (واسط سبک)

برای دیدن اینکه کدام میزبان فعال است و کدام مسیر سوکت در حال استفاده است، از `peekaboo bridge status --verbose` استفاده کنید. می‌توانید با این مورد بازنویسی کنید:

```bash
export PEEKABOO_BRIDGE_SOCKET=/path/to/bridge.sock
```

## امنیت و مجوزها

- پل **امضاهای کد فراخواننده** را اعتبارسنجی می‌کند؛ یک فهرست مجاز از TeamIDها اعمال می‌شود (TeamID میزبان Peekaboo + TeamID برنامه OpenClaw).
- درخواست‌ها پس از حدود ۱۰ ثانیه منقضی می‌شوند.
- اگر مجوزهای لازم موجود نباشند، پل به‌جای باز کردن System Settings، یک پیام خطای روشن برمی‌گرداند.

## رفتار Snapshot (خودکارسازی)

Snapshotها در حافظه ذخیره می‌شوند و پس از یک بازه کوتاه به‌طور خودکار منقضی می‌شوند.
اگر به نگهداری طولانی‌تر نیاز دارید، دوباره از کلاینت ثبت کنید.

## عیب‌یابی

- اگر `peekaboo` گزارش داد «کلاینت پل مجاز نیست»، مطمئن شوید کلاینت به‌درستی امضا شده است یا میزبان را فقط در حالت **اشکال‌زدایی** با `PEEKABOO_ALLOW_UNSIGNED_SOCKET_CLIENTS=1` اجرا کنید.
- اگر هیچ میزبانی پیدا نشد، یکی از برنامه‌های میزبان (Peekaboo.app یا OpenClaw.app) را باز کنید و تأیید کنید که مجوزها اعطا شده‌اند.
