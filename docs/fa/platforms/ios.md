دسترس‌پذیری: بیلدهای اپ iPhone، در صورت فعال‌بودن برای یک انتشار، از طریق کانال‌های Apple توزیع می‌شوند. بیلدهای توسعه محلی نیز می‌توانند از کد منبع اجرا شوند.

## کارکرد آن

- از طریق WebSocket (LAN یا tailnet) به یک Gateway متصل می‌شود.
- قابلیت‌های Node را ارائه می‌کند: Canvas، عکس فوری از صفحه، ثبت تصویر دوربین، موقعیت مکانی، حالت مکالمه، بیدارباش صوتی و خلاصه‌های اختیاری سلامت.
- فرمان‌های `node.invoke` را دریافت و رویدادهای وضعیت Node را گزارش می‌کند.
- فضای کاری عامل انتخاب‌شده را به‌صورت فقط‌خواندنی از بخش عامل‌ها (فایل‌ها) مرور می‌کند: پیمایش تو‌در‌توی پوشه‌ها، پیش‌نمایش متن با برجسته‌سازی نحو، پیش‌نمایش تصویر و برون‌بری از طریق برگه اشتراک‌گذاری. هیچ عملیات نوشتنی انجام نمی‌شود؛ اندازه پیش‌نمایش‌ها توسط Gateway محدود می‌شود.
- برای هر Gateway جفت‌شده، یک حافظه نهان آفلاین کوچک و فقط‌خواندنی از نشست‌ها و رونوشت‌های اخیر چت نگه می‌دارد: هنگام بازشدن سرد، آخرین رونوشت شناخته‌شده فوراً نمایش داده می‌شود و پس از پاسخ Gateway تازه‌سازی می‌شود؛ چت‌های اخیر هنگام قطع اتصال همچنان قابل مرور می‌مانند؛ و بازنشانی/فراموش‌کردن، حافظه نهان محلی محافظت‌شده را پاک می‌کند.
- پیام‌های متنی ارسال‌شده هنگام قطع اتصال را در یک صندوق خروجی پایدار برای هر Gateway (تا 50 پیام) در صف قرار می‌دهد: حباب‌های در صف در رونوشت نمایش داده می‌شوند، پس از اتصال مجدد به‌ترتیب و با تلاش‌های مجدد هم‌توان تخلیه می‌شوند، تا زمانی که تاریخچه مرجع ارسال را تأیید کند پایدار می‌مانند، پیش از نمایش اقدام تلاش مجدد/حذف با تأخیر افزایشی دوباره تلاش می‌شوند و پس از 48 ساعت آفلاین‌بودن به‌جای ارسال منقضی می‌شوند؛ بازنشانی/فراموش‌کردن، صف را همراه حافظه نهان پاک می‌کند.
- پیام‌های دستیار را در صورت درخواست می‌خواند: در چت، یک پیام را لمس طولانی کنید و **گوش‌دادن** را انتخاب کنید. اپ کلیپ‌های پشتیبانی‌شده `tts.speak` از Gateway را با ارائه‌دهنده پیکربندی‌شده TTS پخش می‌کند و اگر صدای Gateway در دسترس یا قابل پخش نباشد، به گفتار روی دستگاه برمی‌گردد. پخش هنگام تغییر نشست یا رفتن اپ به پس‌زمینه متوقف می‌شود.

## الزامات

- Gateway در حال اجرا روی دستگاهی دیگر (macOS، Linux یا Windows از طریق WSL2).
- مسیر شبکه:
  - همان LAN از طریق Bonjour، **یا**
  - Tailnet از طریق DNS-SD تک‌پخشی (دامنه نمونه: `openclaw.internal.`)، **یا**
  - میزبان/درگاه دستی (مسیر جایگزین).

## شروع سریع (جفت‌سازی + اتصال)

در نخستین اجرا، اپ یک توضیح کوتاه درباره جفت‌سازی و یک صفحه
مجوزها (اعلان‌ها، دوربین، میکروفون، عکس‌ها، مخاطبان،
تقویم، یادآورها، موقعیت مکانی) را نمایش می‌دهد. اعطای همه مجوزها اختیاری است و می‌توان آن‌ها را
بعداً در **Settings** -> **Permissions** یا در اپ تنظیمات iOS تغییر داد.

1. یک Gateway احرازشده را با مسیری که تلفن بتواند به آن دسترسی پیدا کند راه‌اندازی کنید. Tailscale
   Serve مسیر راه‌دور پیشنهادی است:

```bash
openclaw gateway --port 18789 --tailscale serve
```

برای راه‌اندازی مورد اعتماد روی همان LAN، به‌جای آن از یک `gateway.bind: "lan"` احرازشده
استفاده کنید. اتصال پیش‌فرض loopback از تلفن قابل دسترسی نیست. اگر
Gateway هنوز پیکربندی نشده است، ابتدا `openclaw onboard` را اجرا کنید تا ساخت کد راه‌اندازی
دارای مسیر احراز هویت با توکن یا گذرواژه باشد.

2. [رابط کنترل](/fa/web/control-ui) را باز کنید، **Nodes** را انتخاب کنید و در صفحه **Devices**
   روی **Pair mobile device** کلیک کنید. دسترسی کامل پیشنهاد می‌شود
   و به‌طور پیش‌فرض انتخاب شده است؛ فقط زمانی Limited access را انتخاب کنید که می‌خواهید
   کنترل‌های مدیریتی Gateway حذف شوند، سپس روی **Create setup code** کلیک کنید.

3. در اپ iOS، **Settings** -> **Gateway** را باز کنید، کد QR را اسکن کنید (یا
   کد راه‌اندازی را جای‌گذاری کنید) و متصل شوید.

   اگر کد راه‌اندازی شامل هر دو مسیر LAN و Tailscale Serve باشد، اپ
   آن‌ها را به‌ترتیب بررسی می‌کند و نخستین نقطه پایانی قابل دسترسی را ذخیره می‌کند.

4. اپ رسمی به‌طور خودکار متصل می‌شود. اگر **Pending approval** درخواستی را
   نشان داد، پیش از تأیید، نقش و دامنه‌های آن را بررسی کنید.

   **Settings → Gateway** نشان می‌دهد اتصال اپراتور ذخیره‌شده
   دسترسی **Full** دارد یا **Limited**. راه‌اندازی متن ساده LAN با `ws://` برای ایمنی
   توکن حامل، به‌طور خودکار محدود می‌شود. اگر محدود است، `wss://` یا
   Tailscale Serve را پیکربندی کنید، یک کد جدید با دسترسی کامل را از رابط کنترل یا `openclaw qr`
   اسکن کنید، سپس برای فعال‌سازی تنظیمات و ارتقاها دوباره متصل شوید.

دکمه رابط کنترل به نشستی از پیش جفت‌شده با `operator.admin` نیاز دارد.
به‌عنوان راهکار جایگزین ترمینال، یک Gateway کشف‌شده را در اپ iOS انتخاب کنید (یا
Manual Host را فعال و میزبان/درگاه را وارد کنید)، سپس درخواست را روی میزبان Gateway تأیید کنید:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

اگر اپ با جزئیات احراز هویت تغییرکرده (نقش/دامنه‌ها/کلید عمومی) دوباره برای جفت‌سازی تلاش کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، `openclaw devices list` را دوباره اجرا کنید.

اختیاری: اگر Node مربوط به iOS همیشه از یک زیرشبکه کاملاً کنترل‌شده متصل می‌شود، می‌توانید با CIDRها یا IPهای دقیق، تأیید خودکار نخستین جفت‌سازی Node را فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این قابلیت به‌طور پیش‌فرض غیرفعال است. فقط برای جفت‌سازی تازه `role: node` بدون دامنه درخواستی اعمال می‌شود. جفت‌سازی اپراتور/مرورگر و هرگونه تغییر نقش، دامنه، فراداده یا کلید عمومی همچنان به تأیید دستی نیاز دارد.

5. اتصال را بررسی کنید:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## خلاصه‌های سلامت

Node مربوط به iOS می‌تواند یک تجمیع اختیاری و فقط‌خواندنی HealthKit را برای روز جاری
تقویمی برگرداند. رضایت در iPhone و مجوز صریح فرمان Gateway،
دو مانع مستقل هستند. برای راه‌اندازی، فراخوانی، فیلدهای بار داده، رفتار حریم خصوصی و عیب‌یابی،
[خلاصه‌های HealthKit](/platforms/ios-healthkit) را ببینید.

به‌طور پیش‌فرض، همراه Apple Watch همچنان از رله موجود iPhone استفاده می‌کند و
به جفت‌سازی جداگانه با Gateway نیاز ندارد. ساعت را در اپ Watch متعلق به
Apple با iPhone جفت کنید، OpenClaw را از **Watch app -> My Watch -> Available
Apps** نصب کنید، سپس OpenClaw را یک‌بار روی هر دو دستگاه باز کنید.

## بررسی تأیید فرمان‌ها

یک اتصال اپراتور با `operator.admin` یا یک اتصال جفت‌شده
`operator.approvals` که صراحتاً توسط Gateway هدف‌گذاری شده است، می‌تواند
درخواست‌های اجرای در انتظار را روی iPhone بررسی کند. کارت تأیید، پیش‌نمایش پاک‌سازی‌شده
فرمان Gateway، هشدار، زمینه میزبان، زمان انقضا و فقط
تصمیم‌های ارائه‌شده توسط همان درخواست را نشان می‌دهد. Apple Watch جفت‌شده همان
درخواست ایمن برای بازبین را از طریق رله موجود iPhone دریافت می‌کند و زیرمجموعه فشرده
تصمیم‌های اجازه یک‌باره/رد را ارائه می‌دهد. حالت مستقیم Watch Gateway
درخواست‌های تأیید را منتقل نمی‌کند.

وضعیت تأیید با رابط کنترل و سطوح پشتیبانی‌شده چت مشترک است.
نخستین پاسخ ثبت‌شده برنده است. iPhone و Watch پس از آنکه سطح دیگری درخواست را حل کند،
پس از اعلان حل‌شدن از راه دور و هر زمان که احتمال گم‌شدن
تأیید دریافت حل وجود داشته باشد، رکورد پایانی مرجع Gateway را دریافت می‌کنند.
اقدامات تا زمانی که این بازخوانی تأیید نکند درخواست همچنان در انتظار است، در دسترس نمی‌مانند.

مالکیت تأیید به Gateway انتخاب‌شده وابسته است. تغییر Gateway نمی‌تواند
یک درخواست قدیمی را روی اتصال جایگزین اعمال کند. Gatewayهایی که پیش از
روش‌های یکپارچه تأیید ساخته شده‌اند، به روش‌های عرضه‌شده مخصوص اجرا برمی‌گردند؛
حفظ وضعیت پایانی و نتایج غنی‌تر میان‌سطحی به Gateway به‌روزشده نیاز دارد.

## Node مستقیم اختیاری برای Apple Watch

حالت مستقیم به ساعت هویت امضاشده Node و اتصال Gateway مستقل می‌دهد.
فرمان‌های پشتیبانی‌شده Node، تا وقتی OpenClaw فعال است، از طریق Wi-Fi یا شبکه همراه
ساعت همچنان کار می‌کنند، حتی اگر iPhone جفت‌شده در دسترس نباشد.

الزامات:

- iPhone با دامنه `operator.admin` به Gateway متصل است.
- کد راه‌اندازی یک نقطه پایانی Gateway با `wss://` و گواهی مورد اعتماد
  watchOS را اعلام می‌کند؛ ساعت مبدأ متناظر `https://` را نظرسنجی می‌کند. HTTP ساده و
  اعتماد صرفاً مبتنی بر خودامضایی یا اثر انگشت پشتیبانی نمی‌شوند. برای پیکربندی نقطه پایانی، [جفت‌سازی
  تحت مالکیت Gateway](/fa/gateway/pairing) را ببینید. مسیرهای loopback، مختص iPhone
  و صرفاً tailnet به‌طور مستقل توسط ساعت قابل دسترسی نیستند.
- استفاده از شبکه همراه به Apple Watch دارای قابلیت اتصال همراه با سرویس فعال نیاز دارد.
- OpenClaw روی ساعت فعال است. Apple به اپ‌های عادی watchOS اجازه نمی‌دهد
  اتصال‌های عمومی WebSocket/TCP را حفظ کنند، بنابراین Node مستقیم از نظرسنجی‌های کوتاه HTTPS
  استفاده می‌کند و وقتی اپ به پیش‌زمینه بازمی‌گردد دوباره متصل می‌شود. [راهنمای شبکه‌سازی سطح پایین watchOS](https://developer.apple.com/documentation/technotes/tn3135-low-level-networking-on-watchOS)
  متعلق به Apple را ببینید.

راه‌اندازی:

1. در iPhone، **Settings -> Apple Watch** را باز کنید.
2. روی **Enable Direct Gateway Connection** بزنید.
3. پیش از انقضای کد راه‌اندازی کوتاه‌عمر، OpenClaw را روی ساعت باز کنید.
4. ردیف جداگانه Apple Watch را با `openclaw nodes status` بررسی کنید.

کد راه‌اندازی شامل یک اعتبارنامه راه‌انداز کوتاه‌عمر و مختص Node است؛ تا زمان انقضا با آن
مانند گذرواژه رفتار کنید. این کد هرگز شامل گذرواژه یا توکن ذخیره‌شده Gateway در iPhone
نیست. پس از جفت‌سازی، ساعت توکن دستگاه خودش را ذخیره و
اعتبارنامه راه‌انداز را حذف می‌کند. حالت مستقیم فقط فرمان‌های زیر را پوشش می‌دهد.
چت، مکالمه، تأییدها و جریان اعلان موجود `watch.*` همچنان
ویژگی‌های رله iPhone هستند و هنوز به iPhone جفت‌شده نیاز دارند.

فرمان‌های مستقیم Node در watchOS:

| سطح           | فرمان‌ها                        | یادداشت‌ها                                               |
| ------------- | ------------------------------ | ------------------------------------------------------- |
| دستگاه        | `device.info`, `device.status` | هویت ساعت، باتری، وضعیت حرارتی، فضای ذخیره‌سازی و شبکه. |
| اعلان‌ها      | `system.notify`                | هنگام فعال‌بودن اپ؛ به مجوز ساعت نیاز دارد.             |

watchOS، WebKit را در اختیار اپ‌های شخص ثالث قرار نمی‌دهد، بنابراین Node مستقیم ساعت
فرمان‌های Canvas را اعلام نمی‌کند.

## پوش رله‌محور برای بیلدهای رسمی

بیلدهای رسمی توزیع‌شده iOS به‌جای انتشار توکن خام APNs برای Gateway، از یک رله پوش خارجی استفاده می‌کنند. بیلدهای رسمی App Store از مسیر انتشار عمومی، از رله میزبانی‌شده در `https://ios-push-relay.openclaw.ai` استفاده می‌کنند؛ این URL پایه برای توزیع App Store در کد ثابت شده است و هیچ مقدار بازنویسی را نمی‌خواند.

استقرارهای رله سفارشی به مسیر بیلد/استقرار عمداً جداگانه iOS نیاز دارند که URL رله آن با URL رله Gateway مطابقت داشته باشد. مسیر انتشار App Store هرگز URL رله سفارشی را نمی‌پذیرد. اگر از بیلد رله سفارشی استفاده می‌کنید، URL رله متناظر Gateway را تنظیم کنید:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

نحوه کار جریان:

- اپ iOS با استفاده از App Attest و JWS تراکنش اپ StoreKit، در رله ثبت‌نام می‌کند.
- رله یک شناسه مبهم رله به‌همراه مجوز ارسال محدود به ثبت‌نام برمی‌گرداند.
- اپ iOS هویت Gateway جفت‌شده (`gateway.identity.get`) را دریافت و آن را در ثبت‌نام رله درج می‌کند؛ بنابراین ثبت‌نام رله‌محور به همان Gateway مشخص واگذار می‌شود.
- اپ، ثبت‌نام رله‌محور را با `push.apns.register` به Gateway جفت‌شده ارسال می‌کند.
- Gateway از شناسه ذخیره‌شده رله برای `push.test`، بیدارسازی‌های پس‌زمینه و تلنگرهای بیدارسازی استفاده می‌کند.
- اگر اپ بعداً به Gateway دیگری یا بیلدی با URL پایه رله متفاوت متصل شود، به‌جای استفاده مجدد از اتصال قبلی، ثبت‌نام رله را تازه‌سازی می‌کند.

آنچه Gateway برای این مسیر **نیاز ندارد**: نه توکن رله سراسری استقرار و نه کلید مستقیم APNs برای ارسال‌های رسمی رله‌محور App Store.

جریان مورد انتظار اپراتور:

1. اپ رسمی iOS را نصب کنید.
2. اختیاری: فقط هنگام استفاده از یک بیلد رله سفارشی و عمداً جداگانه، `gateway.push.apns.relay.baseUrl` را روی Gateway تنظیم کنید.
3. اپ را با Gateway جفت کنید و اجازه دهید اتصال را کامل کند.
4. اپ پس از داشتن توکن APNs، اتصال نشست اپراتور و موفقیت ثبت‌نام رله، `push.apns.register` را منتشر می‌کند.
5. پس از آن، `push.test`، بیدارسازی‌های اتصال مجدد و تلنگرهای بیدارسازی می‌توانند از ثبت‌نام رله‌محور ذخیره‌شده استفاده کنند.

## نشانگرهای زنده‌بودن در پس‌زمینه

وقتی iOS برنامه را برای یک اعلان بی‌صدای push، تازه‌سازی پس‌زمینه یا رویداد تغییر چشمگیر مکان بیدار می‌کند، برنامه تلاش می‌کند اتصال کوتاهی به Node برقرار کند و سپس `node.event` را با `event: "node.presence.alive"` فراخوانی می‌کند. Gateway این مورد را تنها پس از مشخص‌شدن هویت احراز‌شده دستگاه Node، به‌صورت `lastSeenAtMs`/`lastSeenReason` در فراداده Node/دستگاه جفت‌شده ثبت می‌کند.

برنامه بیدارشدن در پس‌زمینه را فقط زمانی با موفقیت ثبت‌شده در نظر می‌گیرد که پاسخ Gateway شامل `handled: true` باشد. Gatewayهای قدیمی‌تر ممکن است `node.event` را با `{ "ok": true }` تأیید کنند؛ این پاسخ سازگار است، اما به‌عنوان به‌روزرسانی پایدار آخرین مشاهده محسوب نمی‌شود.

نکته سازگاری:

- `OPENCLAW_APNS_RELAY_BASE_URL` همچنان به‌عنوان بازنویسی موقت محیطی برای Gateway کار می‌کند (`gateway.push.apns.relay.baseUrl` مسیر مبتنی بر پیکربندی است).
- حالت push در بیلد انتشار App Store، میزبان رله میزبانی‌شده را به‌صورت ثابت در کد دارد و هرگز بازنویسی URL رله را نمی‌خواند — متغیر محیطی زمان بیلد `OPENCLAW_PUSH_RELAY_BASE_URL` فقط بر حالت‌های بیلد محلی/sandbox در iOS اثر می‌گذارد.

## جریان احراز هویت و اعتماد

رله برای اعمال دو محدودیتی وجود دارد که APNs مستقیم روی Gateway نمی‌تواند برای بیلدهای رسمی iOS فراهم کند:

- فقط بیلدهای اصیل OpenClaw برای iOS که از طریق Apple توزیع شده‌اند می‌توانند از رله میزبانی‌شده استفاده کنند.
- یک Gateway فقط برای دستگاه‌های iOS که با همان Gateway مشخص جفت شده‌اند می‌تواند اعلان‌های push مبتنی بر رله ارسال کند.

گام‌به‌گام:

1. `iOS app -> gateway`: برنامه از طریق جریان عادی احراز هویت Gateway با Gateway جفت می‌شود و یک نشست احراز‌شده Node به‌همراه یک نشست احراز‌شده اپراتور دریافت می‌کند. نشست اپراتور `gateway.identity.get` را فراخوانی می‌کند.
2. `iOS app -> relay`: برنامه نقاط پایانی ثبت‌نام رله را از طریق HTTPS با مدرک App Attest و یک JWS تراکنش برنامه StoreKit فراخوانی می‌کند. رله شناسه بسته، مدرک App Attest و مدرک توزیع Apple را اعتبارسنجی می‌کند و مسیر توزیع رسمی/تولیدی را الزامی می‌داند — همین موضوع مانع استفاده بیلدهای محلی Xcode/توسعه از رله میزبانی‌شده می‌شود، زیرا یک بیلد محلی نمی‌تواند مدرک توزیع رسمی Apple را ارائه کند.
3. `gateway identity delegation`: پیش از ثبت‌نام رله، برنامه هویت Gateway جفت‌شده را از `gateway.identity.get` دریافت می‌کند و آن را در محموله ثبت‌نام رله می‌گنجاند. رله یک شناسه رله و یک مجوز ارسال محدود به ثبت‌نام را که به آن هویت Gateway تفویض شده است برمی‌گرداند.
4. `gateway -> relay`: Gateway شناسه رله و مجوز ارسال دریافتی از `push.apns.register` را ذخیره می‌کند. هنگام `push.test`، بیدارسازی‌های اتصال مجدد و تلنگرهای بیدارسازی، Gateway درخواست ارسال را با هویت دستگاه خودش امضا می‌کند؛ رله هم مجوز ارسال ذخیره‌شده و هم امضای Gateway را در برابر هویت Gateway تفویض‌شده هنگام ثبت‌نام تأیید می‌کند. Gateway دیگری نمی‌تواند از آن ثبت‌نام ذخیره‌شده دوباره استفاده کند، حتی اگر به‌نحوی شناسه را به‌دست آورد.
5. `relay -> APNs`: رله مالک اطلاعات احراز هویت APNs تولیدی و توکن خام APNs برای بیلد رسمی است. Gateway هرگز توکن خام APNs را برای بیلدهای رسمی مبتنی بر رله ذخیره نمی‌کند؛ رله اعلان push نهایی را از طرف Gateway جفت‌شده به APNs ارسال می‌کند.

دلیل ایجاد این طراحی: دور نگه‌داشتن اطلاعات احراز هویت APNs تولیدی از Gatewayهای کاربران، جلوگیری از ذخیره توکن‌های خام APNs بیلد رسمی در Gateway، محدودکردن استفاده از رله میزبانی‌شده به بیلدهای رسمی OpenClaw برای iOS و جلوگیری از ارسال اعلان‌های push بیدارسازی توسط یک Gateway به دستگاه‌های iOS متعلق به Gateway دیگر.

بیلدهای محلی/دستی همچنان از APNs مستقیم استفاده می‌کنند. اگر این بیلدها را بدون رله آزمایش می‌کنید، Gateway همچنان به اطلاعات احراز هویت مستقیم APNs نیاز دارد:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

این‌ها متغیرهای محیطی زمان اجرای میزبان Gateway هستند، نه تنظیمات Fastlane. `apps/ios/fastlane/.env` فقط اطلاعات احراز هویت App Store Connect مانند `APP_STORE_CONNECT_KEY_ID` و `APP_STORE_CONNECT_ISSUER_ID` را ذخیره می‌کند؛ این مورد تحویل مستقیم APNs را برای بیلدهای محلی iOS پیکربندی نمی‌کند.

روش پیشنهادی ذخیره‌سازی در میزبان Gateway، هماهنگ با سایر اطلاعات احراز هویت ارائه‌دهندگان در `~/.openclaw/credentials/`:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

فایل `.p8` را commit نکنید و آن را در checkout مخزن قرار ندهید.

## مسیرهای کشف

### Bonjour (شبکه محلی)

برنامه iOS، `_openclaw-gw._tcp` را در `local.` مرور می‌کند و در صورت پیکربندی، همین کار را در دامنه کشف گسترده DNS-SD نیز انجام می‌دهد. Gatewayهای موجود در همان شبکه محلی به‌صورت خودکار از `local.` ظاهر می‌شوند؛ کشف بین‌شبکه‌ای می‌تواند بدون تغییر نوع نشانگر از دامنه گسترده پیکربندی‌شده استفاده کند.

### Tailnet (بین‌شبکه‌ای)

اگر mDNS مسدود است، از یک ناحیه تک‌پخشی DNS-SD (یک دامنه انتخاب کنید؛ نمونه: `openclaw.internal.`) و DNS تفکیکی Tailscale استفاده کنید. برای نمونه CoreDNS به [Bonjour](/fa/gateway/bonjour) مراجعه کنید.

### میزبان/درگاه دستی

در Settings، **Manual Host** را فعال کنید و میزبان + درگاه Gateway را وارد کنید (پیش‌فرض `18789`).

## چند Gateway

برنامه فهرستی از همه Gatewayهایی که با آن‌ها جفت شده است نگه می‌دارد تا بتوانید بدون جفت‌سازی دوباره بین آن‌ها جابه‌جا شوید:

- **Settings -> Gateway** فهرست **Paired Gateways** را نمایش می‌دهد که Gateway فعال در آن علامت‌گذاری شده است. برای جابه‌جایی روی یک مورد بزنید؛ برنامه نشست‌های فعلی را قطع می‌کند و دوباره به Gateway انتخاب‌شده متصل می‌شود. وقتی بیش از یک Gateway جفت شده باشد، منوی جابه‌جایی سریع کنار ردیف اتصال ظاهر می‌شود.
- اطلاعات احراز هویت، تصمیم‌های اعتماد TLS، ترجیحات مختص هر Gateway و تاریخچه ذخیره‌شده گفت‌وگو برای هر Gateway جداگانه ذخیره می‌شوند. جابه‌جایی هرگز وضعیت Gatewayها را با یکدیگر ترکیب نمی‌کند و ثبت‌نام push از Gateway فعال پیروی می‌کند.
- یک Gateway جفت‌شده را بکشید (یا از منوی زمینه آن استفاده کنید) تا آن را **Forget** کنید؛ با این کار اطلاعات احراز هویت، توکن‌های دستگاه، پین TLS و گفت‌وگوهای ذخیره‌شده آن حذف می‌شوند.
- برای جابه‌جایی به Gatewayهای کشف‌شده، آن‌ها باید در شبکه قابل مشاهده باشند؛ Gatewayهای دستی با میزبان و درگاه ذخیره‌شده دوباره متصل می‌شوند.

## Canvas + A2UI

Node مربوط به iOS یک Canvas مبتنی بر WKWebView را رندر می‌کند. برای کنترل آن از `node.invoke` استفاده کنید:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

نکات:

- میزبان Canvas در Gateway، `/__openclaw__/canvas/` و `/__openclaw__/a2ui/` را از سرور HTTP Gateway ارائه می‌کند (همان درگاه `gateway.port`، پیش‌فرض `18789`).
- Node مربوط به iOS، چارچوب داخلی را به‌عنوان نمای پیش‌فرض متصل نگه می‌دارد. `canvas.a2ui.push` و `canvas.a2ui.reset` از صفحه A2UI همراه و متعلق به برنامه استفاده می‌کنند.
- صفحه‌های A2UI مربوط به Gateway راه‌دور در iOS فقط قابل رندر هستند؛ کنش‌های بومی دکمه A2UI فقط از صفحه‌های همراه و متعلق به برنامه پذیرفته می‌شوند.
- با `canvas.navigate` و `{"url":""}` به چارچوب داخلی بازگردید.

## ارتباط با Computer Use

برنامه iOS یک سطح Node موبایل است، نه یک backend برای Codex Computer Use. ‏Codex Computer Use و `cua-driver mcp` یک دسکتاپ محلی macOS را از طریق ابزارهای MCP کنترل می‌کنند؛ برنامه iOS قابلیت‌های iPhone را از طریق فرمان‌های Node در OpenClaw مانند `canvas.*`، `camera.*`، `screen.*`، `location.*` و `talk.*` ارائه می‌کند.

عامل‌ها همچنان می‌توانند با فراخوانی فرمان‌های Node، برنامه iOS را از طریق OpenClaw کنترل کنند، اما این فراخوانی‌ها از پروتکل Node در Gateway عبور می‌کنند و از محدودیت‌های پیش‌زمینه/پس‌زمینه iOS پیروی می‌کنند. برای کنترل دسکتاپ محلی از [Codex Computer Use](/fa/plugins/codex-computer-use) و برای قابلیت‌های Node در iOS از این صفحه استفاده کنید.

openclaw nodes invoke --node "iOS Node" --command canvas.eval --params '{"javaScript":"(() => { const {ctx} = window.__openclaw; ctx.clearRect(0,0,innerWidth,innerHeight); ctx.lineWidth=6; ctx.strokeStyle=\"#ff2d55\"; ctx.beginPath(); ctx.moveTo(40,40); ctx.lineTo(innerWidth-40, innerHeight-40); ctx.stroke(); return \"ok\"; })()"}'

## بیدارسازی صوتی + حالت مکالمه

- بیدارسازی صوتی و حالت مکالمه در Settings در دسترس هستند.
- مکالمه بلادرنگ OpenAI هنگامی که `talk.realtime.transport` برابر `webrtc` باشد، از WebRTC متعلق به کلاینت استفاده می‌کند؛ پیکربندی صریح `gateway-relay` همچنان تحت مالکیت Gateway باقی می‌ماند. به [حالت مکالمه](/fa/nodes/talk) مراجعه کنید.
- Nodeهای iOS با قابلیت مکالمه، قابلیت `talk` را اعلام می‌کنند و می‌توانند `talk.ptt.start`، `talk.ptt.stop`، `talk.ptt.cancel` و `talk.ptt.once` را تعریف کنند؛ Gateway به‌صورت پیش‌فرض آن فرمان‌های فشردن‌برای‌صحبت را برای Nodeهای معتمد دارای قابلیت مکالمه مجاز می‌داند.
- ممکن است iOS صدای پس‌زمینه را تعلیق کند؛ وقتی برنامه فعال نیست، قابلیت‌های صوتی را در حد بهترین تلاش در نظر بگیرید.

## خطاهای رایج

- `NODE_BACKGROUND_UNAVAILABLE`: برنامه iOS را به پیش‌زمینه بیاورید (فرمان‌های Canvas/دوربین/صفحه‌نمایش به آن نیاز دارند).
- `A2UI_HOST_UNAVAILABLE`: صفحه همراه A2UI در WebView برنامه قابل دسترسی نبود؛ برنامه را در برگه Screen در پیش‌زمینه نگه دارید و دوباره تلاش کنید.
- درخواست جفت‌سازی هرگز ظاهر نمی‌شود: `openclaw devices list` را اجرا و به‌صورت دستی تأیید کنید.
- Watch هیچ وضعیتی از iPhone نشان نمی‌دهد: تأیید کنید که iPhone، `watchPaired: true`
  و `watchAppInstalled: true` را در `watch.status` گزارش می‌کند. اگر جفت‌سازی false است،
  Watch را در برنامه Watch متعلق به Apple جفت کنید. اگر نصب false است، برنامه همراه
  را از **My Watch -> Available Apps** نصب کنید. پس از هر یک از این تغییرات، OpenClaw را
  یک‌بار روی Watch باز کنید؛ دسترسی فوری همچنان مستلزم اجرای هر دو برنامه است،
  درحالی‌که به‌روزرسانی‌های در صف می‌توانند بعداً در پس‌زمینه برسند.
- اتصال مجدد پس از نصب دوباره ناموفق است: توکن جفت‌سازی Keychain پاک شده است؛ Node را دوباره جفت کنید.
