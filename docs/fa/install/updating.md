OpenClaw را به‌روز نگه دارید.

برای جایگزینی ایمیج‌های Docker، Podman و Kubernetes، به
[ارتقای ایمیج‌های کانتینر](/fa/install/docker#upgrading-container-images) مراجعه کنید. Gateway
پیش از آماده‌شدن، عملیات ارتقای ایمن برای راه‌اندازی را اجرا می‌کند و اگر وضعیت
متصل‌شده به تعمیر دستی نیاز داشته باشد، خارج می‌شود.

## توصیه‌شده: `openclaw update`

نوع نصب را (npm، pnpm، Bun یا git) تشخیص می‌دهد، آخرین نسخه را دریافت می‌کند، `openclaw doctor` را اجرا می‌کند و Gateway را دوباره راه‌اندازی می‌کند.

```bash
openclaw update
```

کانال‌ها را تغییر دهید یا نسخه مشخصی را هدف بگیرید:

```bash
openclaw update --channel beta
openclaw update --channel extended-stable
openclaw update --channel dev
openclaw update --dry-run   # پیش‌نمایش بدون اعمال
```

`openclaw update` پرچم `--verbose` ندارد (نصب‌کننده دارد). برای عیب‌یابی، از
`--dry-run` برای پیش‌نمایش اقدامات برنامه‌ریزی‌شده، از `--json` برای نتایج ساختاریافته، یا از
`openclaw update status --json` برای بررسی وضعیت کانال و دسترس‌پذیری استفاده کنید.

`--channel beta` تگ توزیع beta در npm را ترجیح می‌دهد، اما وقتی تگ beta وجود نداشته باشد
یا نسخه آن از آخرین انتشار پایدار قدیمی‌تر باشد، به stable/latest
بازمی‌گردد. برای به‌روزرسانی یک‌باره بسته که مستقیماً به تگ توزیع beta خام npm
سنجاق شده است، به‌جای آن از `--tag beta` استفاده کنید.

`--channel extended-stable` فقط مخصوص بسته است و نصب همچنان
فقط در پیش‌زمینه انجام می‌شود. OpenClaw انتخابگر عمومی npm یعنی `extended-stable` را می‌خواند،
بسته دقیق انتخاب‌شده را تأیید می‌کند و همان نسخه دقیق را نصب می‌کند. نبودن
یا ناسازگاری داده‌های رجیستری موجب توقف ایمن می‌شود؛ هرگز به `latest` بازنمی‌گردد.
اگر نسخه انتخاب‌شده از نسخه نصب‌شده قدیمی‌تر باشد، تأیید معمول
تنزل نسخه همچنان اعمال می‌شود. CLI پس از
به‌روزرسانی موفق هسته، کانال را ذخیره می‌کند؛ اجرای مستقیم `npm install -g openclaw@extended-stable`
مقدار `update.channel` را به‌روزرسانی نمی‌کند.
پس از تعویض هسته، Pluginهای رسمی و واجد شرایط npm با قصد bare/default یا
`latest` به همان نسخه دقیق هسته همگرا می‌شوند. سنجاق‌های دقیق و تگ‌های صریح
غیرِ `latest`، Pluginهای شخص ثالث و منابع غیر npm بدون تغییر می‌مانند.
نصب‌های کاتالوگی ایجادشده با نسخه‌های کنونی OpenClaw، همان قصد پیش‌فرض را
حفظ می‌کنند. رکوردهای قدیمی‌تر که فقط یک نسخه دقیق دارند، همچنان سنجاق‌شده
می‌مانند، زیرا OpenClaw نمی‌تواند با اطمینان یک سنجاق خودکار قدیمی را از سنجاق کاربر
تشخیص دهد؛ برای بازگرداندن آن Plugin به رهگیری دقیق نسخه هسته، یک‌بار
`openclaw plugins update @openclaw/name` را در کانال extended-stable اجرا کنید.

`--channel dev` یک checkout متحرک و دائمی GitHub از `main` فراهم می‌کند. برای به‌روزرسانی
یک‌باره بسته، `--tag main` به مشخصه بسته `github:openclaw/openclaw#main`
نگاشت می‌شود و آن را مستقیماً از طریق مدیر بسته مقصد (npm/pnpm/bun) نصب می‌کند.

برای Pluginهای مدیریت‌شده، نبودن انتشار beta هشدار است، نه خطا:
به‌روزرسانی هسته همچنان می‌تواند موفق شود، درحالی‌که Plugin به انتشار
default/latest ثبت‌شده خود بازمی‌گردد.

برای معنای کانال‌ها، به [کانال‌های انتشار](/fa/install/development-channels) مراجعه کنید.

## جابه‌جایی میان نصب‌های npm و git

برای تغییر نوع نصب از کانال‌ها استفاده کنید. به‌روزرسان وضعیت، پیکربندی،
اعتبارنامه‌ها و فضای کاری را در `~/.openclaw` نگه می‌دارد؛ فقط مشخص می‌کند CLI و Gateway از کدام
نصب کد OpenClaw استفاده کنند.

```bash
# نصب بسته npm -> checkout قابل‌ویرایش git
openclaw update --channel dev

# checkout گیت -> نصب بسته npm
openclaw update --channel stable
```

ابتدا تغییر حالت نصب را پیش‌نمایش کنید:

```bash
openclaw update --channel dev --dry-run
openclaw update --channel stable --dry-run
```

`dev` وجود یک checkout گیت را تضمین می‌کند، آن را می‌سازد و CLI سراسری را از همان
checkout نصب می‌کند. کانال‌های `stable`، `extended-stable` و `beta` از نصب‌های
بسته‌ای استفاده می‌کنند. extended-stable در یک checkout گیت، بدون تغییر یا
تبدیل آن رد می‌شود. اگر Gateway از قبل نصب شده باشد، `openclaw update`
فراداده سرویس را تازه‌سازی و آن را دوباره راه‌اندازی می‌کند، مگر اینکه `--no-restart` را ارسال کنید.

برای نصب‌های بسته‌ای دارای سرویس مدیریت‌شده Gateway، `openclaw update`
ریشه بسته‌ای را هدف می‌گیرد که آن سرویس استفاده می‌کند. اگر فرمان `openclaw` پوسته
از نصب دیگری آمده باشد، به‌روزرسان هر دو ریشه و مسیر Node سرویس
مدیریت‌شده را نمایش می‌دهد و پیش از جایگزینی بسته، نسخه Node را با نیازمندی
`engines.node` انتشار مقصد بررسی می‌کند.

## جایگزین: اجرای دوباره نصب‌کننده

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
```

برای ردکردن راه‌اندازی اولیه، `--no-onboard` را اضافه کنید. برای اجبار یک نوع نصب مشخص،
`--install-method git --no-onboard` یا `--install-method npm --no-onboard` را ارسال کنید.

اگر `openclaw update` پس از مرحله نصب بسته npm ناموفق شد، به‌جای آن
نصب‌کننده را دوباره اجرا کنید. نصب‌کننده به‌روزرسان را فراخوانی نمی‌کند؛ نصب سراسری بسته را
مستقیماً اجرا می‌کند و می‌تواند یک نصب npm نیمه‌به‌روزشده را بازیابی کند.

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method npm
```

بازیابی را با `--version` به یک نسخه یا تگ توزیع مشخص سنجاق کنید:

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method npm --version <version-or-dist-tag>
```

## جایگزین: npm، pnpm یا bun دستی

```bash
npm i -g openclaw@latest
```

برای نصب‌های تحت نظارت، `openclaw update` را ترجیح دهید: این گزینه می‌تواند تعویض بسته را
با سرویس درحال‌اجرای Gateway هماهنگ کند. اگر نصب تحت نظارت را دستی
به‌روزرسانی می‌کنید، ابتدا Gateway مدیریت‌شده را متوقف کنید. مدیرهای بسته، فایل‌ها را
درجا جایگزین می‌کنند و در غیر این صورت یک Gateway درحال‌اجرا ممکن است هنگام تعویض
برای بارگذاری فایل‌های هسته یا Plugin تلاش کند. پس از پایان کار مدیر بسته، Gateway را
دوباره راه‌اندازی کنید تا نصب جدید را دریافت کند.

برای یک نصب سراسری سیستمی در Linux که مالک آن root است، اگر `openclaw update` با
`EACCES` ناموفق شد، درحالی‌که Gateway برای جایگزینی دستی متوقف می‌ماند، با npm سیستم
آن را بازیابی کنید. از همان پرچم‌های پروفایل/محیطی استفاده کنید که معمولاً برای
آن Gateway به‌کار می‌برید. `/usr/bin/npm` را با npm سیستمی جایگزین کنید که مالک
پیشوند سراسری متعلق به root در میزبان شماست:

```bash
openclaw gateway stop
sudo /usr/bin/npm i -g openclaw@latest
openclaw gateway install --force
openclaw gateway restart
```

سپس تأیید کنید:

```bash
openclaw --version
curl -fsS http://127.0.0.1:18789/readyz
openclaw plugins list --json
openclaw gateway status --deep --json
openclaw doctor --lint --json
```

وقتی `openclaw update` یک نصب سراسری npm را مدیریت می‌کند، ابتدا مقصد را
در یک پیشوند موقت npm نصب می‌کند. بسته نامزد در طول `preinstall` نسخه Node
میزبان را اعتبارسنجی می‌کند؛ تنها پس از آن OpenClaw فهرست بسته‌بندی‌شده
`dist` را تأیید می‌کند و درخت پاک بسته را با پیشوند سراسری واقعی تعویض می‌کند. یک
محافظ تکمیل بسته‌بندی‌شده از فهرست مورد انتظار حذف می‌شود و فقط
پس از موفقیت `preinstall` برداشته می‌شود، بنابراین ردشدن اسکریپت‌های چرخه عمر نیز پیش از
تعویض موجب شکست می‌شود. در npm 12 و جدیدتر، به‌روزرسان فقط چرخه عمر OpenClaw
نامزد را تأیید می‌کند؛ اسکریپت‌های وابستگی‌های انتقالی همچنان مسدود می‌مانند. این کار مانع می‌شود npm
یک بسته جدید را روی فایل‌های کهنه بسته قبلی هم‌پوشانی کند. اگر فرمان نصب
ناموفق شود، OpenClaw یک‌بار با `--omit=optional` دوباره تلاش می‌کند؛ این کار به میزبان‌هایی کمک می‌کند
که وابستگی‌های اختیاری بومی در آن‌ها قابل کامپایل نیستند.

فرمان‌های به‌روزرسانی npm و به‌روزرسانی Plugin که توسط OpenClaw مدیریت می‌شوند، قرنطینه
زنجیره تأمین `min-release-age` در npm (یا کلید پیکربندی قدیمی‌تر `before`) را نیز
برای فرایند فرزند npm پاک می‌کنند. این سیاست برای حفاظت عمومی وجود دارد، اما
به‌روزرسانی صریح OpenClaw یعنی «انتشار انتخاب‌شده را همین حالا نصب کن.»

```bash
pnpm add -g openclaw@latest
```

اگر pnpm 11 نسخه OpenClaw 2026.7.1 را نصب کرده است، آن فرمان دستی را یک‌بار اجرا کنید. آن
انتشار مربوط به پیش از چیدمان ایزوله بسته‌های سراسری pnpm 11 است، بنابراین به‌روزرسان آن ممکن است
نصب npm دیگری را با CLI درحال‌اجرا اشتباه بگیرد. انتشارهای بعدی مالکیت
pnpm را حفظ می‌کنند و هنگام به‌روزرسانی‌ها ریشه بسته جایگزین را دنبال می‌کنند. آن‌ها
همچنین از دایرکتوری bin سراسری گزارش‌شده توسط مدیر مالک استفاده می‌کنند و پیش از
تغییر متوقف می‌شوند، اگر فرمان pnpm موجود ریشه یا نسخه اصلی سراسری دیگری را گزارش کند،
یا اگر بسته فراخوان یتیم باشد یا تنها نصب فعال OpenClaw
در آنجا نباشد.

اگر OpenClaw یک گروه نصب سراسری pnpm 11 را با بسته دیگری به‌اشتراک بگذارد،
به‌روزرسان خودکار پیش از تغییر گروه متوقف می‌شود. گروه اصلی
جداشده با ویرگول را دستی به‌روزرسانی کنید تا بسته‌های هم‌گروه و سیاست ساخت آن
دست‌نخورده بمانند.

```bash
bun add -g openclaw@latest
```

## به‌روزرسان خودکار

به‌طور پیش‌فرض غیرفعال است. آن را در `~/.openclaw/openclaw.json` فعال کنید:

```json5
{
  update: {
    channel: "stable",
    auto: {
      enabled: true,
      stableDelayHours: 6,
      stableJitterHours: 12,
      betaCheckIntervalHours: 1,
    },
  },
}
```

| کانال           | رفتار                                                                                                                                     |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `stable`          | به‌مدت `stableDelayHours` (پیش‌فرض: 6) منتظر می‌ماند، سپس با نوسان قطعی در بازه `stableJitterHours` (پیش‌فرض: 12) برای انتشار تدریجی اعمال می‌کند. |
| `extended-stable` | هنگام راه‌اندازی و هر 24 ساعت، وقتی `checkOnStart` فعال است، یک راهنمای به‌روزرسانی فقط‌خواندنی را بررسی می‌کند. هرگز به‌طور خودکار اعمال نمی‌کند.                |
| `beta`            | هر `betaCheckIntervalHours` (پیش‌فرض: 1) بررسی می‌کند و بلافاصله اعمال می‌کند.                                                                  |
| `dev`             | اعمال خودکار ندارد. از `openclaw update` به‌صورت دستی استفاده کنید.                                                                                          |

Gateway هنگام راه‌اندازی یک راهنمای به‌روزرسانی را نیز ثبت می‌کند (با
`update.checkOnStart: false` غیرفعال کنید). انتخاب‌های ذخیره‌شده extended-stable از همین
مسیر راهنمای فقط‌خواندنی و فاصله زمانی 24 ساعته موجود استفاده می‌کنند، اما هرگز
نصب خودکار، تحویل، راه‌اندازی مجدد، تأخیر/نوسان stable یا نظرسنجی beta را فراخوانی نمی‌کنند.
برای تنزل نسخه یا بازیابی پس از رخداد، `OPENCLAW_NO_AUTO_UPDATE=1` را در محیط Gateway تنظیم کنید تا اعمال خودکار حتی در صورت پیکربندی `update.auto.enabled` مسدود شود. راهنماهای به‌روزرسانی هنگام راه‌اندازی همچنان می‌توانند اجرا شوند، مگر اینکه `update.checkOnStart` نیز غیرفعال شده باشد.

به‌روزرسانی‌های مدیر بسته که از طریق صفحه کنترل زنده Gateway درخواست می‌شوند
(`update.run`) درخت بسته را داخل فرایند درحال‌اجرای Gateway
جایگزین نمی‌کنند. در نصب‌های سرویس مدیریت‌شده، Gateway یک تحویل جداشده را آغاز می‌کند،
خارج می‌شود و اجازه می‌دهد مسیر عادی CLI یعنی `openclaw update --yes --json`
سرویس را متوقف کند، بسته را جایگزین کند، فراداده سرویس را تازه‌سازی کند، دوباره راه‌اندازی کند، نسخه
و دسترس‌پذیری Gateway را تأیید کند و در صورت امکان LaunchAgent نصب‌شده اما بارگذاری‌نشده macOS را
بازیابی کند. اگر Gateway نتواند آن تحویل را با اطمینان انجام دهد،
`update.run` به‌جای اجرای مدیر بسته درون فرایند، یک فرمان امن پوسته
گزارش می‌کند.

کارت به‌روزرسانی نوار کناری رابط کنترل، زمانی که قرار است این فرایند
`update.run` را مستقیماً آغاز کند، **به‌روزرسانی Gateway** را نشان می‌دهد. این حالت رابط کنترل میزبانی‌شده در مرورگر، Gatewayهای
راه دور و Gatewayهای محلی با مدیریت دستی را پوشش می‌دهد.

در برنامه امضاشده macOS، برای یک Gateway محلی تحت مالکیت برنامه، متن آن کارت به
**به‌روزرسانی برنامه Mac + Gateway** تغییر می‌کند. Sparkle ابتدا برنامه را به‌روزرسانی می‌کند؛ پس از اجرای مجدد،
برنامه `openclaw update --tag <app-version> --json` را اجرا می‌کند، Gateway خود را مجدداً راه‌اندازی می‌کند
و سلامت آن را در پنجره پیشرفتی مشابه راه‌اندازی اولیه بررسی می‌کند. این پنجره فقط
زمانی ظاهر می‌شود که آن Gateway مدیریت‌شده به به‌روزرسانی، تعمیر یا نصب نیاز داشته باشد؛ به‌روزرسانی‌های صرفاً مربوط به برنامه
پس از اجرای مجدد مستقیماً وارد برنامه می‌شوند. جزئیات خرابی همراه با Retry، [راهنمای به‌روزرسانی](/fa/install/updating) و
کنش‌های [Discord](https://discord.gg/clawd) قابل مشاهده باقی می‌مانند. برنامه هرگز برای یک Gateway
راه دور یا دارای مدیریت خارجی از این مسیر هماهنگ‌شده استفاده نمی‌کند، هرگز Gateway
جدیدتر را به نسخه پایین‌تر برنمی‌گرداند و هرگز پین کانال `extended-stable` را نادیده نمی‌گیرد.

وقتی به‌روزرسانی موفق باشد، برنامه یک رویداد خوشامدگویی یک‌باره را برای جدیدترین
نشست مستقیم سطح‌بالا که دارای تعامل واقعی کاربر/کانال است در صف قرار می‌دهد. اجراهای
Cron، Heartbeatها و به‌روزرسانی‌های نشستِ صرفاً پس‌زمینه آن انتخاب را تغییر نمی‌دهند. در
حالت راه دور، برنامه فقط زمان‌اجرای Node محلی Mac خود را به‌روزرسانی می‌کند و رویداد را
فقط زمانی می‌فرستد که Gateway راه دور متصل، دست‌کم به‌اندازه برنامه جدید باشد.

## بازگردانی

بازگردانی دو لایه دارد:

1. کد قدیمی‌تر OpenClaw را با حفظ وضعیت فعلی دوباره نصب کنید.
2. وضعیت پیش از به‌روزرسانی را فقط زمانی بازیابی کنید که کد قدیمی‌تر نتواند از پیکربندی
   یا پایگاه داده مهاجرت‌یافته استفاده کند.

با بازگردانی صرفاً کد آغاز کنید. بازیابی وضعیت، تغییرات ایجادشده پس از
پشتیبان‌گیری را دور می‌ریزد.

### پیش از به‌روزرسانی: ایجاد یک پشتیبان راستی‌آزمایی‌شده

`openclaw update` یک کپی خودکار از پیکربندی پیش از به‌روزرسانی نگه می‌دارد، اما
یک نقطه بازیابی کامل از وضعیت ایجاد نمی‌کند. پیش از یک به‌روزرسانی مهم، آن را
به‌طور صریح ایجاد کنید:

```bash
mkdir -p ~/Backups/openclaw
openclaw backup create --output ~/Backups/openclaw --verify
```

مانیفست بایگانی، نسخه OpenClaw و مسیرهای مبدأ گنجانده‌شده
در پشتیبان را ثبت می‌کند. بایگانی ممکن است شامل اعتبارنامه‌ها، پروفایل‌های احراز هویت و وضعیت
کانال باشد؛ بنابراین آن را با مجوزهای مختص مالک و همان سطح حفاظتی
دایرکتوری وضعیت زنده نگه‌داری کنید. برای فایل‌های گنجانده‌شده و فایل‌هایی که عمداً
حذف شده‌اند، [پشتیبان‌گیری](/fa/cli/backup) را ببینید.

برای یک نقطه بازیابی بایت‌به‌بایت که شامل مصنوعات ناپایدارِ حذف‌شده از
بایگانی قابل‌انتقال نیز باشد، Gateway را متوقف کنید و از اسنپ‌شات سامانه فایل، ولوم یا ماشین مجازی
ارائه‌شده توسط پلتفرم خود استفاده کنید.

### بازگردانی نصب بسته

نسخه‌های منتشرشده را فهرست کنید، سپس نسخه سالم شناخته‌شده را پیش‌نمایش و نصب کنید:

```bash
npm view openclaw versions --json
openclaw update --tag <known-good-version> --dry-run
openclaw update --tag <known-good-version>
```

`openclaw update --tag` بر نصب مستقیم با مدیر بسته ترجیح داده می‌شود. این روش
کاهش نسخه را تشخیص می‌دهد، تأیید می‌گیرد، همگرایی Pluginهای مدیریت‌شده
و بررسی‌های سازگاری با مقصد نصب‌شده را اجرا می‌کند، فراداده سرویس را
تازه‌سازی می‌کند، Gateway را مجدداً راه‌اندازی می‌کند و نسخه در حال اجرا را راستی‌آزمایی می‌کند. اگر کانال ذخیره‌شده
`extended-stable` است، از
`--channel stable --tag <known-good-version>` استفاده کنید، زیرا برچسب‌های دقیق یک‌باره را نمی‌توان
با گزینشگر `extended-stable` ترکیب کرد.

به‌روزرسانی‌های بسته، گزینه نامزد را پیش از فعال‌سازی آماده و راستی‌آزمایی می‌کنند. اگر
جابه‌جایی سامانه فایل یا جایگزینی واسط فرمان شکست بخورد، OpenClaw بسته قدیمی را
به‌طور خودکار بازیابی می‌کند. پس از یک جابه‌جایی موفق، اگر بررسی سلامت Gateway در ادامه
شکست بخورد، به‌جای جایگزینی خودکار دوباره بسته، نسخه قبلی
و دستورالعمل‌های بازگردانی دستی گزارش می‌شوند.

اگر مسیر به‌روزرسانی CLI در دسترس نیست، از همان مدیر بسته و محدوده نصبی
استفاده کنید که مالک Gateway فعلی هستند:

```bash
openclaw gateway stop
npm i -g openclaw@<known-good-version>
openclaw gateway install --force
openclaw gateway restart
```

وقتی آن مدیر مالک نصب است، `npm` را با `pnpm` یا `bun` جایگزین کنید. هنگام
بازیابی از رخداد، با تنظیم `OPENCLAW_NO_AUTO_UPDATE=1` در محیط Gateway مانع شوید که یک به‌روزرسان خودکار فعال، فوراً
نسخه جدیدتری را اعمال کند.

### بازگردانی یک نسخه دریافت‌شده از منبع

از یک نسخه دریافت‌شده تمیز استفاده کنید و یک برچسب یا کامیت سالم شناخته‌شده را انتخاب کنید:

```bash
git fetch --all --tags
git checkout --detach <known-good-tag-or-commit>
pnpm install && pnpm build
openclaw gateway restart
```

برای بازگشت به جدیدترین نسخه: `git checkout main && git pull`.

اگر پس از آغاز یک به‌روزرسانی git، نصب وابستگی، ساخت، ساخت رابط کاربری یا doctor
شکست بخورد، به‌روزرسان به‌طور خودکار نسخه دریافت‌شده git را به شاخه و
SHA قبلی آن بازمی‌گرداند. وقتی عمداً یک کامیت قدیمی‌تر را انتخاب می‌کنید،
همچنان باید نسخه را به‌صورت دستی دریافت کنید.

### کاهش نسخه در مهاجرت SQLite نشست

پیش از راه‌اندازی یک نسخه قدیمی‌تر OpenClaw مبتنی بر فایل، از CLI فعلی برای
بازیابی مصنوعات بایگانی‌شده رونوشت قدیمی استفاده کنید:

```bash
openclaw gateway stop
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

این کار داده‌های SQLite را حذف نمی‌کند. نشست‌هایی که پس از مهاجرت SQLite ایجاد شده‌اند
فقط در SQLite وجود دارند و در زمان‌اجرای قدیمی‌تر ظاهر نمی‌شوند. به
[کاهش نسخه پس از مهاجرت SQLite نشست](/fa/cli/doctor#downgrading-after-session-sqlite-migration) مراجعه کنید.

### بازیابی وضعیت فقط در صورت ضرورت

اگر کد قدیمی‌تر نمی‌تواند پیکربندی جدیدتر یا طرح‌واره پایگاه داده را بخواند،
Gateway را متوقف و اسنپ‌شات راستی‌آزمایی‌شده سامانه فایل، ولوم یا ماشین مجازیِ پیش از به‌روزرسانی را بازیابی کنید.
پیش از بازیابی، وضعیت فعلی را جداگانه نگه دارید، زیرا این کار
تغییرات ایجادشده پس از اسنپ‌شات را حذف می‌کند.

بایگانی‌های گسترده `openclaw backup create` از ایجاد و راستی‌آزمایی پشتیبانی می‌کنند، اما
از فعال‌سازی درجا و کامل بایگانی پشتیبانی نمی‌کنند. یک بایگانی گسترده را در یک دایرکتوری
آماده‌سازی استخراج کنید و برای بازیابی آفلاین از نگاشت مبدأ به بایگانی
`manifest.json` آن استفاده کنید. `openclaw backup sqlite restore` نیز یک پایگاه داده
راستی‌آزمایی‌شده را در مقصدی تازه می‌نویسد؛ فعال‌سازی آن مقصد همچنان یک مرحله صریح و آفلاین
برای متصدی است.

## اگر گیر کرده‌اید

- دوباره `openclaw doctor` را اجرا کنید و خروجی را با دقت بخوانید.
- برای `openclaw update --channel dev` در نسخه‌های دریافت‌شده از منبع، به‌روزرسان در صورت نیاز `pnpm` را به‌طور خودکار راه‌اندازی می‌کند. اگر خطای راه‌اندازی اولیه pnpm/corepack را مشاهده کردید، `pnpm` را به‌صورت دستی نصب کنید (یا `corepack` را دوباره فعال کنید) و به‌روزرسانی را مجدداً اجرا کنید.

- در Discord بپرسید: [https://discord.gg/clawd](https://discord.gg/clawd)
