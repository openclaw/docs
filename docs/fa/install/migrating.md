OpenClaw از سه مسیر مهاجرت پشتیبانی می‌کند: وارد کردن از یک سامانه عامل دیگر، انتقال نصب موجود به دستگاهی جدید، و ارتقای درجا یک Plugin.

## وارد کردن از یک سامانه عامل دیگر

ارائه‌دهندگان مهاجرت همراه، دستورالعمل‌ها، سرورهای MCP، Skills، پیکربندی مدل و کلیدهای API (با انتخاب کاربر) را به OpenClaw وارد می‌کنند. طرح‌ها پیش از هر تغییری پیش‌نمایش می‌شوند، مقادیر محرمانه در گزارش‌ها پوشانده می‌شوند و اعمال تغییرات با یک نسخه پشتیبان تأییدشده پشتیبانی می‌شود.

نقطه ورود CLI، فرمان [`openclaw migrate`](/fa/cli/migrate) است. فرایند راه‌اندازی اولیه نیز هنگام شناسایی یک منبع شناخته‌شده می‌تواند مهاجرت را پیشنهاد دهد (`openclaw onboard --flow import`).

## انتقال OpenClaw به دستگاهی جدید

برای حفظ موارد زیر، **دایرکتوری وضعیت** (به‌طور پیش‌فرض `~/.openclaw/`) و **فضای کاری** خود را کپی کنید:

- **پیکربندی** — `openclaw.json` و همه تنظیمات Gateway.
- **احراز هویت** — فایل `auth-profiles.json` هر عامل (کلیدهای API به‌همراه OAuth) و هرگونه وضعیت کانال یا ارائه‌دهنده در `credentials/`.
- **نشست‌ها** — تاریخچه مکالمه و وضعیت عامل.
- **وضعیت کانال** — ورود WhatsApp، نشست Telegram و موارد مشابه.
- **فایل‌های فضای کاری** — `MEMORY.md`، `USER.md`، Skills و پرامپت‌ها.

<Tip>
برای تأیید مسیر دایرکتوری وضعیت خود، `openclaw status` را روی دستگاه قدیمی اجرا کنید. پروفایل‌های سفارشی از `~/.openclaw-<profile>/` یا مسیری که از طریق `OPENCLAW_STATE_DIR` تنظیم شده است استفاده می‌کنند.
</Tip>

### مراحل مهاجرت

<Steps>
  <Step title="توقف Gateway و تهیه نسخه پشتیبان">
    در دستگاه **قدیمی**، Gateway را متوقف کنید تا فایل‌ها هنگام کپی تغییر نکنند، سپس بایگانی تهیه کنید:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    اگر از چند پروفایل استفاده می‌کنید (برای مثال `~/.openclaw-work`)، هرکدام را جداگانه بایگانی کنید.

  </Step>

  <Step title="نصب OpenClaw روی دستگاه جدید">
    CLI (و در صورت نیاز Node) را روی دستگاه جدید [نصب کنید](/fa/install). اگر راه‌اندازی اولیه یک `~/.openclaw/` تازه ایجاد کند مشکلی نیست؛ در مرحله بعد آن را بازنویسی می‌کنید.
  </Step>

  <Step title="کپی دایرکتوری وضعیت و فضای کاری">
    بایگانی را از طریق `scp`،‏ `rsync -a` یا یک درایو خارجی منتقل و سپس استخراج کنید:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    تأیید کنید که دایرکتوری‌های مخفی نیز کپی شده‌اند و مالکیت فایل‌ها با کاربری که Gateway را اجرا خواهد کرد مطابقت دارد.

  </Step>

  <Step title="اجرای Doctor و تأیید">
    برای اعمال مهاجرت‌های پیکربندی و تعمیر سرویس‌ها، روی دستگاه جدید [Doctor](/fa/gateway/doctor) را اجرا کنید:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

اگر Telegram یا Discord از بازگشت پیش‌فرض به متغیر محیطی (`TELEGRAM_BOT_TOKEN` یا `DISCORD_BOT_TOKEN`) استفاده می‌کند، بدون چاپ مقادیر محرمانه تأیید کنید که فایل `.env` در دایرکتوری وضعیت مهاجرت‌داده‌شده شامل این کلیدها است:

```bash
awk -F= '/^(TELEGRAM_BOT_TOKEN|DISCORD_BOT_TOKEN)=/ { print $1 "=present" }' ~/.openclaw/.env
```

`openclaw doctor` همچنین زمانی هشدار می‌دهد که حساب پیش‌فرض فعال Telegram یا Discord هیچ توکن پیکربندی‌شده‌ای نداشته باشد و متغیر محیطی متناظر در دسترس فرایند Doctor نباشد.

### مشکلات رایج

<AccordionGroup>
  <Accordion title="عدم تطابق پروفایل یا دایرکتوری وضعیت">
    اگر Gateway قدیمی از `--profile` یا `OPENCLAW_STATE_DIR` استفاده می‌کرد اما Gateway جدید استفاده نمی‌کند، کانال‌ها خارج‌شده از حساب به نظر می‌رسند و نشست‌ها خالی خواهند بود. Gateway را با **همان** پروفایل یا دایرکتوری وضعیت مهاجرت‌داده‌شده اجرا کنید، سپس `openclaw doctor` را دوباره اجرا کنید.
  </Accordion>

  <Accordion title="کپی کردن فقط openclaw.json">
    فایل پیکربندی به‌تنهایی کافی نیست. پروفایل‌های احراز هویت مدل در `agents/<agentId>/agent/auth-profiles.json` و وضعیت کانال و ارائه‌دهنده در `credentials/` قرار دارند. همیشه **کل** دایرکتوری وضعیت را مهاجرت دهید.
  </Accordion>

  <Accordion title="مجوزها و مالکیت">
    اگر فایل‌ها را با کاربر ریشه کپی کرده‌اید یا کاربر را تغییر داده‌اید، ممکن است Gateway نتواند اطلاعات احراز هویت را بخواند. مطمئن شوید دایرکتوری وضعیت و فضای کاری متعلق به کاربری هستند که Gateway را اجرا می‌کند.
  </Accordion>

  <Accordion title="حالت راه دور">
    اگر رابط کاربری شما به یک Gateway **راه دور** متصل است، نشست‌ها و فضای کاری متعلق به میزبان راه دور هستند. خود میزبان Gateway را مهاجرت دهید، نه لپ‌تاپ محلی خود را. به [پرسش‌های متداول](/fa/help/faq#where-things-live-on-disk) مراجعه کنید.
  </Accordion>

  <Accordion title="مقادیر محرمانه در نسخه‌های پشتیبان">
    دایرکتوری وضعیت شامل پروفایل‌های احراز هویت، اطلاعات احراز هویت کانال و سایر وضعیت‌های ارائه‌دهنده است. نسخه‌های پشتیبان را به‌صورت رمزگذاری‌شده نگهداری کنید، از کانال‌های انتقال ناامن پرهیز کنید و اگر به افشای اطلاعات مشکوک هستید، کلیدها را تعویض کنید.
  </Accordion>
</AccordionGroup>

### فهرست وارسی تأیید

روی دستگاه جدید، موارد زیر را تأیید کنید:

- [ ] `openclaw status` نشان می‌دهد Gateway در حال اجرا است.
- [ ] کانال‌ها همچنان متصل هستند (نیازی به جفت‌سازی مجدد نیست).
- [ ] داشبورد باز می‌شود و نشست‌های موجود را نمایش می‌دهد.
- [ ] فایل‌های فضای کاری (حافظه و پیکربندی‌ها) موجود هستند.

## ارتقای درجا یک Plugin

ارتقاهای درجای Plugin همان شناسه Plugin و کلیدهای پیکربندی را حفظ می‌کنند، اما ممکن است وضعیت ذخیره‌شده روی دیسک را به چیدمان فعلی منتقل کنند. راهنماهای ارتقای ویژه هر Plugin در کنار کانال مربوط به آن قرار دارند:

- [مهاجرت Matrix](/fa/channels/matrix-migration): محدودیت‌های بازیابی وضعیت رمزگذاری‌شده، رفتار عکس فوری خودکار و فرمان‌های بازیابی دستی.
