OpenClaw می‌تواند برای گزارش‌های باگ، یک فایل zip تشخیصی محلی ایجاد کند. این فایل
وضعیت، سلامت، لاگ‌ها، شکل پیکربندی، و رویدادهای پایداری اخیر بدون payload و
پاک‌سازی‌شدهٔ Gateway را با هم ترکیب می‌کند.

تا وقتی بسته‌های تشخیصی را بررسی نکرده‌اید، با آن‌ها مانند اسرار رفتار کنید. این بسته‌ها
طوری طراحی شده‌اند که payloadها و اعتبارنامه‌ها را حذف یا ویرایش کنند، اما همچنان
لاگ‌های محلی Gateway و وضعیت runtime در سطح میزبان را خلاصه می‌کنند.

## شروع سریع

```bash
openclaw gateway diagnostics export
```

این فرمان مسیر zip نوشته‌شده را چاپ می‌کند. برای انتخاب مسیر:

```bash
openclaw gateway diagnostics export --output openclaw-diagnostics.zip
```

برای خودکارسازی:

```bash
openclaw gateway diagnostics export --json
```

## فرمان چت

مالک‌ها می‌توانند در چت از `/diagnostics [note]` برای درخواست export محلی Gateway استفاده کنند.
وقتی باگ در یک گفت‌وگوی واقعی رخ داده و یک گزارش قابل کپی‌پیست برای پشتیبانی می‌خواهید،
از این استفاده کنید:

1. در گفت‌وگویی که مشکل را در آن دیدید، `/diagnostics` را ارسال کنید. اگر کمک می‌کند،
   یک یادداشت کوتاه اضافه کنید، برای نمونه `/diagnostics bad tool choice`.
2. OpenClaw مقدمهٔ تشخیص را ارسال می‌کند و یک تأیید exec صریح درخواست می‌کند.
   این تأیید `openclaw gateway diagnostics export --json` را اجرا می‌کند.
   تشخیص‌ها را از طریق قاعدهٔ allow-all تأیید نکنید.
3. پس از تأیید، OpenClaw با گزارشی قابل چسباندن پاسخ می‌دهد که شامل مسیر بستهٔ محلی،
   خلاصهٔ manifest، یادداشت‌های حریم خصوصی، و شناسه‌های session مرتبط است.

در چت‌های گروهی، مالک همچنان می‌تواند `/diagnostics` را اجرا کند، اما OpenClaw جزئیات
تشخیصی را دوباره در چت مشترک منتشر نمی‌کند. مقدمه، درخواست‌های تأیید، نتیجهٔ export
Gateway، و تفکیک session/thread مربوط به Codex را از مسیر تأیید خصوصی برای مالک
می‌فرستد. گروه فقط یک اعلان کوتاه دریافت می‌کند که جریان تشخیص به‌صورت خصوصی ارسال شد.
اگر OpenClaw نتواند یک مسیر خصوصی برای مالک پیدا کند، فرمان به‌صورت بسته شکست می‌خورد
و از مالک می‌خواهد آن را از یک DM اجرا کند.

وقتی session فعال OpenClaw از harness بومی OpenAI Codex استفاده می‌کند،
همان تأیید exec یک بارگذاری بازخورد OpenAI را نیز برای threadهای runtime مربوط به Codex
که OpenClaw می‌شناسد پوشش می‌دهد. این بارگذاری از zip محلی Gateway جدا است و فقط برای
sessionهای harness مربوط به Codex ظاهر می‌شود. پیش از تأیید، prompt توضیح می‌دهد که
تأیید تشخیص‌ها بازخورد Codex را نیز ارسال خواهد کرد، اما شناسه‌های session یا thread مربوط
به Codex را فهرست نمی‌کند. پس از تأیید، پاسخ چت channelها، شناسه‌های session مربوط به
OpenClaw، شناسه‌های thread مربوط به Codex، و فرمان‌های resume محلی را برای threadهایی
که به سرورهای OpenAI ارسال شدند فهرست می‌کند. اگر تأیید را رد یا نادیده بگیرید،
OpenClaw export را اجرا نمی‌کند، بازخورد Codex را ارسال نمی‌کند، و شناسه‌های Codex را
چاپ نمی‌کند.

این کار حلقهٔ رایج اشکال‌زدایی Codex را کوتاه می‌کند: رفتار نادرست را در Telegram،
Discord، یا channel دیگری ببینید، `/diagnostics` را اجرا کنید، یک بار تأیید کنید، گزارش
را با پشتیبانی به اشتراک بگذارید، سپس اگر می‌خواهید thread بومی Codex را خودتان بررسی
کنید، فرمان چاپ‌شدهٔ `codex resume <thread-id>` را به‌صورت محلی اجرا کنید. برای این
جریان بررسی، [harness مربوط به Codex](/fa/plugins/codex-harness#inspect-a-codex-thread-from-the-cli)
را ببینید.

## محتوای export

فایل zip شامل موارد زیر است:

- `summary.md`: نمای کلی خوانا برای انسان، برای پشتیبانی.
- `diagnostics.json`: خلاصهٔ خوانا برای ماشین از پیکربندی، لاگ‌ها، وضعیت، سلامت،
  و داده‌های پایداری.
- `manifest.json`: فرادادهٔ export و فهرست فایل‌ها.
- شکل پیکربندی پاک‌سازی‌شده و جزئیات پیکربندی غیرمحرمانه.
- خلاصه‌های لاگ پاک‌سازی‌شده و خطوط لاگ اخیر با ویرایش اطلاعات حساس.
- snapshotهای بهترین‌تلاش از وضعیت و سلامت Gateway.
- `stability/latest.json`: تازه‌ترین بستهٔ پایداری ذخیره‌شده، در صورت وجود.

export حتی وقتی Gateway ناسالم است هم مفید است. اگر Gateway نتواند به درخواست‌های وضعیت
یا سلامت پاسخ دهد، لاگ‌های محلی، شکل پیکربندی، و تازه‌ترین بستهٔ پایداری همچنان در صورت
وجود جمع‌آوری می‌شوند.

## مدل حریم خصوصی

تشخیص‌ها طوری طراحی شده‌اند که قابل اشتراک‌گذاری باشند. export داده‌های عملیاتی‌ای را
نگه می‌دارد که به اشکال‌زدایی کمک می‌کنند، مانند:

- نام‌های زیرسیستم، شناسه‌های Plugin، شناسه‌های provider، شناسه‌های channel، و modeهای پیکربندی‌شده
- کدهای وضعیت، مدت‌ها، شمارش بایت‌ها، وضعیت صف، و خوانش‌های حافظه
- فرادادهٔ لاگ پاک‌سازی‌شده و پیام‌های عملیاتی با اطلاعات حساس ویرایش‌شده
- شکل پیکربندی و تنظیمات قابلیت‌های غیرمحرمانه

export موارد زیر را حذف یا ویرایش می‌کند:

- متن چت، promptها، instructionها، بدنه‌های Webhook، و خروجی‌های ابزار
- اعتبارنامه‌ها، کلیدهای API، tokenها، cookieها، و مقدارهای محرمانه
- بدنه‌های خام درخواست یا پاسخ
- شناسه‌های حساب، شناسه‌های پیام، شناسه‌های خام session، نام‌های میزبان، و نام‌های کاربری محلی

وقتی یک پیام لاگ شبیه متن payload کاربر، چت، prompt، یا ابزار باشد،
export فقط این را نگه می‌دارد که پیامی حذف شده است، همراه با شمارش بایت.

## ضبط‌کنندهٔ پایداری

وقتی تشخیص‌ها فعال باشند، Gateway به‌صورت پیش‌فرض یک جریان پایداری محدود و بدون payload
را ضبط می‌کند. این جریان برای واقعیت‌های عملیاتی است، نه محتوا.

همان Heartbeat تشخیصی وقتی Gateway همچنان اجرا می‌شود اما event loop مربوط به Node.js یا
CPU اشباع به نظر می‌رسد، هشدارهای زنده‌بودن را ثبت می‌کند. این رویدادهای
`diagnostic.liveness.warning` شامل تأخیر event-loop، بهره‌برداری event-loop،
نسبت هستهٔ CPU، و شمارش sessionهای فعال/در انتظار/در صف هستند. این رویدادها
به‌خودی‌خود Gateway را restart نمی‌کنند.

ضبط‌کنندهٔ زنده را بررسی کنید:

```bash
openclaw gateway stability
openclaw gateway stability --type payload.large
openclaw gateway stability --json
```

تازه‌ترین بستهٔ پایداری ذخیره‌شده را پس از خروج fatal، timeout خاموشی،
یا شکست startup هنگام restart بررسی کنید:

```bash
openclaw gateway stability --bundle latest
```

از تازه‌ترین بستهٔ ذخیره‌شده یک zip تشخیصی بسازید:

```bash
openclaw gateway stability --bundle latest --export
```

بسته‌های ذخیره‌شده وقتی رویدادها وجود داشته باشند، زیر `~/.openclaw/logs/stability/` قرار دارند.

## گزینه‌های مفید

```bash
openclaw gateway diagnostics export \
  --output openclaw-diagnostics.zip \
  --log-lines 5000 \
  --log-bytes 1000000
```

- `--output <path>`: نوشتن در یک مسیر zip مشخص.
- `--log-lines <count>`: حداکثر خطوط لاگ پاک‌سازی‌شده برای گنجاندن.
- `--log-bytes <bytes>`: حداکثر بایت‌های لاگ برای بررسی.
- `--url <url>`: URL مربوط به WebSocket در Gateway برای snapshotهای وضعیت و سلامت.
- `--token <token>`: token مربوط به Gateway برای snapshotهای وضعیت و سلامت.
- `--password <password>`: گذرواژهٔ Gateway برای snapshotهای وضعیت و سلامت.
- `--timeout <ms>`: timeout مربوط به snapshot وضعیت و سلامت.
- `--no-stability-bundle`: رد شدن از جست‌وجوی بستهٔ پایداری ذخیره‌شده.
- `--json`: چاپ فرادادهٔ export به‌صورت خوانا برای ماشین.

## غیرفعال کردن تشخیص‌ها

تشخیص‌ها به‌صورت پیش‌فرض فعال هستند. برای غیرفعال کردن ضبط‌کنندهٔ پایداری و
جمع‌آوری رویدادهای تشخیصی:

```json5
{
  diagnostics: {
    enabled: false,
  },
}
```

غیرفعال کردن تشخیص‌ها جزئیات گزارش باگ را کاهش می‌دهد. این کار روی لاگ‌گیری عادی
Gateway تأثیری ندارد.
