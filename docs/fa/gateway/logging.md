# ثبت گزارش

برای نمای کلیِ رو به کاربر (CLI + رابط کاربری Control + پیکربندی)، [/ثبت-گزارش](/fa/logging) را ببینید.

OpenClaw دو «سطح» گزارش دارد:

- **خروجی کنسول** (آنچه در ترمینال / رابط کاربری Debug می‌بینید).
- **گزارش‌های فایل** (خطوط JSON) که توسط ثبت‌کنندهٔ گزارش Gateway نوشته می‌شوند.

## ثبت‌کنندهٔ گزارش مبتنی بر فایل

- فایل گزارش چرخشی پیش‌فرض زیر `/tmp/openclaw/` قرار دارد (یک فایل برای هر روز): `openclaw-YYYY-MM-DD.log`
  - تاریخ از منطقهٔ زمانی محلی میزبان Gateway استفاده می‌کند.
- فایل‌های گزارش فعال در `logging.maxFileBytes` می‌چرخند (پیش‌فرض: 100 MB)، تا
  پنج بایگانی شماره‌گذاری‌شده را نگه می‌دارند و نوشتن در یک فایل فعال تازه را ادامه می‌دهند.
- مسیر و سطح فایل گزارش را می‌توان از طریق `~/.openclaw/openclaw.json` پیکربندی کرد:
  - `logging.file`
  - `logging.level`

قالب فایل، یک شیء JSON در هر خط است.

زبانهٔ گزارش‌های رابط کاربری Control این فایل را از طریق Gateway دنبال می‌کند (`logs.tail`).
CLI هم می‌تواند همین کار را انجام دهد:

```bash
openclaw logs --follow
```

**پرگویی در برابر سطح‌های گزارش**

- **گزارش‌های فایل** فقط و فقط با `logging.level` کنترل می‌شوند.
- `--verbose` فقط روی **میزان پرگویی کنسول** (و سبک گزارش WS) اثر دارد؛ سطح گزارش فایل را
  بالا نمی‌برد.
- برای ثبت جزئیات فقط-پرگو در گزارش‌های فایل، `logging.level` را روی `debug` یا
  `trace` تنظیم کنید.

## ثبت کنسول

CLI فراخوانی‌های `console.log/info/warn/error/debug/trace` را ثبت می‌کند و آن‌ها را در گزارش‌های فایل می‌نویسد،
در حالی که همچنان در stdout/stderr هم چاپ می‌کند.

می‌توانید میزان پرگویی کنسول را به‌صورت مستقل با این گزینه‌ها تنظیم کنید:

- `logging.consoleLevel` (پیش‌فرض `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## پوشاندن داده‌های حساس

OpenClaw می‌تواند توکن‌های حساس را پیش از آنکه خروجی گزارش یا رونوشت از
فرایند خارج شود، ماسک کند. این سیاست پوشاندن در گزارش‌گیری روی مقصدهای متن کنسول، گزارش فایل، رکورد گزارش OTLP
و رونوشت نشست اعمال می‌شود، بنابراین مقدارهای محرمانهٔ منطبق پیش از نوشته‌شدن خطوط JSONL یا پیام‌ها روی دیسک
ماسک می‌شوند.

- `logging.redactSensitive`: `off` | `tools` (پیش‌فرض: `tools`)
- `logging.redactPatterns`: آرایه‌ای از رشته‌های regex (پیش‌فرض‌ها را جایگزین می‌کند)
  - از رشته‌های خام regex (به‌صورت خودکار `gi`) استفاده کنید، یا اگر به پرچم‌های سفارشی نیاز دارید از `/pattern/flags` استفاده کنید.
  - تطابق‌ها با نگه‌داشتن 6 نویسهٔ اول + 4 نویسهٔ آخر ماسک می‌شوند (طول >= 18)، در غیر این صورت `***`.
  - پیش‌فرض‌ها انتساب‌های رایج کلید، پرچم‌های CLI، فیلدهای JSON، سرآیندهای bearer، بلوک‌های PEM، پیشوندهای محبوب توکن، و نام فیلدهای اطلاعات پرداخت مانند شمارهٔ کارت، CVC/CVV، توکن پرداخت مشترک، و اطلاعات پرداخت را پوشش می‌دهند.

برخی مرزهای ایمنی همیشه، صرف‌نظر از `logging.redactSensitive`، پوشاندن را اعمال می‌کنند.
این شامل رویدادهای فراخوانی ابزار در رابط کاربری Control، خروجی ابزار `sessions_history`،
خروجی‌های پشتیبانی عیب‌یابی، مشاهده‌های خطای ارائه‌دهنده، نمایش فرمان تأیید exec،
و گزارش‌های پروتکل WebSocket مربوط به Gateway است. این سطح‌ها همچنان ممکن است از
`logging.redactPatterns` به‌عنوان الگوهای اضافی استفاده کنند، اما `redactSensitive: "off"`
باعث نمی‌شود اسرار خام را منتشر کنند.

## گزارش‌های WebSocket مربوط به Gateway

Gateway گزارش‌های پروتکل WebSocket را در دو حالت چاپ می‌کند:

- **حالت عادی (بدون `--verbose`)**: فقط نتیجه‌های RPC «جالب» چاپ می‌شوند:
  - خطاها (`ok=false`)
  - فراخوانی‌های کند (آستانهٔ پیش‌فرض: `>= 50ms`)
  - خطاهای تجزیه
- **حالت پرگو (`--verbose`)**: تمام ترافیک درخواست/پاسخ WS را چاپ می‌کند.

### سبک گزارش WS

`openclaw gateway` از یک سوییچ سبک برای هر Gateway پشتیبانی می‌کند:

- `--ws-log auto` (پیش‌فرض): حالت عادی بهینه‌سازی شده است؛ حالت پرگو از خروجی فشرده استفاده می‌کند
- `--ws-log compact`: خروجی فشرده (درخواست/پاسخ جفت‌شده) هنگام پرگویی
- `--ws-log full`: خروجی کامل برای هر فریم هنگام پرگویی
- `--compact`: نام مستعار برای `--ws-log compact`

نمونه‌ها:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## قالب‌بندی کنسول (گزارش‌گیری زیرسیستم)

قالب‌بند کنسول **آگاه از TTY** است و خط‌هایی سازگار با پیشوند چاپ می‌کند.
ثبت‌کننده‌های گزارش زیرسیستم، خروجی را گروه‌بندی‌شده و قابل مرور نگه می‌دارند.

رفتار:

- **پیشوندهای زیرسیستم** در هر خط (مثلاً `[gateway]`، `[canvas]`، `[tailscale]`)
- **رنگ‌های زیرسیستم** (پایدار برای هر زیرسیستم) به‌همراه رنگ‌آمیزی سطح
- **رنگ وقتی خروجی یک TTY است یا محیط شبیه ترمینال غنی به نظر می‌رسد** (`TERM`/`COLORTERM`/`TERM_PROGRAM`)، با رعایت `NO_COLOR`
- **پیشوندهای کوتاه‌شدهٔ زیرسیستم**: `gateway/` + `channels/` آغازین را حذف می‌کند، 2 بخش آخر را نگه می‌دارد (مثلاً `whatsapp/outbound`)
- **ثبت‌کننده‌های فرعی بر پایهٔ زیرسیستم** (پیشوند خودکار + فیلد ساخت‌یافتهٔ `{ subsystem }`)
- **`logRaw()`** برای خروجی QR/UX (بدون پیشوند، بدون قالب‌بندی)
- **سبک‌های کنسول** (مثلاً `pretty | compact | json`)
- **سطح گزارش کنسول** جدا از سطح گزارش فایل است (وقتی `logging.level` روی `debug`/`trace` تنظیم شده باشد، فایل جزئیات کامل را نگه می‌دارد)
- **بدنهٔ پیام‌های WhatsApp** در سطح `debug` ثبت می‌شوند (برای دیدن آن‌ها از `--verbose` استفاده کنید)

این کار گزارش‌های فایل موجود را پایدار نگه می‌دارد و در عین حال خروجی تعاملی را قابل مرور می‌کند.
