# ثبت لاگ

برای یک نمای کلی رو به کاربر (CLI + رابط کاربری کنترل + پیکربندی)، به [/logging](/fa/logging) مراجعه کنید.

OpenClaw دو «سطح» ثبت لاگ دارد:

- **خروجی کنسول** (آنچه در ترمینال / رابط کاربری اشکال‌زدایی می‌بینید).
- **لاگ‌های فایل** (خطوط JSON) که توسط لاگر Gateway نوشته می‌شوند.

هنگام راه‌اندازی، Gateway مدل پیش‌فرض عاملِ حل‌شده را همراه با
پیش‌فرض‌های حالتی که بر نشست‌های جدید اثر می‌گذارند لاگ می‌کند، برای مثال:

```text
agent model: openai-codex/gpt-5.5 (thinking=medium, fast=on)
```

`thinking` از عامل پیش‌فرض، پارامترهای مدل، یا پیش‌فرض سراسری عامل می‌آید؛
وقتی تنظیم نشده باشد، خلاصهٔ راه‌اندازی `medium` را نشان می‌دهد. `fast` از عامل
پیش‌فرض یا پارامترهای `fastMode` مدل می‌آید.

## لاگر مبتنی بر فایل

- فایل لاگ چرخشی پیش‌فرض زیر `/tmp/openclaw/` قرار دارد (یک فایل برای هر روز): `openclaw-YYYY-MM-DD.log`
  - تاریخ از منطقهٔ زمانی محلی میزبان Gateway استفاده می‌کند.
- فایل‌های لاگ فعال در `logging.maxFileBytes` می‌چرخند (پیش‌فرض: 100 MB)، تا
  پنج بایگانی شماره‌گذاری‌شده را نگه می‌دارند و نوشتن را در یک فایل فعال تازه ادامه می‌دهند.
- مسیر و سطح فایل لاگ را می‌توان از طریق `~/.openclaw/openclaw.json` پیکربندی کرد:
  - `logging.file`
  - `logging.level`

قالب فایل یک شیء JSON در هر خط است.

زبانهٔ لاگ‌ها در رابط کاربری کنترل این فایل را از طریق Gateway دنبال می‌کند (`logs.tail`).
CLI نیز می‌تواند همین کار را انجام دهد:

```bash
openclaw logs --follow
```

**پرحرفی در برابر سطح‌های لاگ**

- **لاگ‌های فایل** منحصراً توسط `logging.level` کنترل می‌شوند.
- `--verbose` فقط بر **پرحرفی کنسول** (و سبک لاگ WS) اثر می‌گذارد؛ سطح لاگ فایل را **بالا نمی‌برد**.
- برای ثبت جزئیات مخصوص حالت پرحرف در لاگ‌های فایل، `logging.level` را روی `debug` یا
  `trace` تنظیم کنید.
- ثبت لاگ Trace همچنین شامل خلاصه‌های زمان‌بندی تشخیصی برای مسیرهای داغ منتخب است،
  مانند آماده‌سازی کارخانهٔ ابزار Plugin. ببینید
  [/tools/plugin#slow-plugin-tool-setup](/fa/tools/plugin#slow-plugin-tool-setup).

## ثبت کنسول

CLI فراخوانی‌های `console.log/info/warn/error/debug/trace` را ثبت می‌کند و آن‌ها را در لاگ‌های فایل می‌نویسد،
در حالی که همچنان در stdout/stderr چاپ می‌کند.

می‌توانید پرحرفی کنسول را مستقل تنظیم کنید با:

- `logging.consoleLevel` (پیش‌فرض `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## پوشاندن

OpenClaw می‌تواند پیش از خروج خروجی لاگ یا رونوشت از فرایند، توکن‌های حساس را ماسک کند.
این سیاست پوشاندنِ لاگ در سینک‌های متنی کنسول، لاگ فایل، رکورد لاگ OTLP،
و رونوشت نشست اعمال می‌شود، بنابراین مقدارهای محرمانهٔ مطابق پیش از نوشته شدن
خطوط JSONL یا پیام‌ها روی دیسک ماسک می‌شوند.

- `logging.redactSensitive`: `off` | `tools` (پیش‌فرض: `tools`)
- `logging.redactPatterns`: آرایه‌ای از رشته‌های regex (پیش‌فرض‌ها را بازنویسی می‌کند)
  - از رشته‌های خام regex (به‌صورت خودکار `gi`) استفاده کنید، یا اگر به پرچم‌های سفارشی نیاز دارید از `/pattern/flags` استفاده کنید.
  - تطابق‌ها با نگه داشتن 6 نویسهٔ نخست + 4 نویسهٔ آخر ماسک می‌شوند (طول >= 18)، در غیر این صورت `***`.
  - پیش‌فرض‌ها انتساب‌های رایج کلید، پرچم‌های CLI، فیلدهای JSON، هدرهای bearer، بلوک‌های PEM، پیشوندهای محبوب توکن، و نام فیلدهای اعتبار پرداخت مانند شمارهٔ کارت، CVC/CVV، توکن پرداخت مشترک، و اعتبار پرداخت را پوشش می‌دهند.

برخی مرزهای ایمنی همیشه صرف‌نظر از `logging.redactSensitive` پوشانده می‌شوند.
این شامل رویدادهای فراخوانی ابزار در رابط کاربری کنترل، خروجی ابزار `sessions_history`،
صادرات پشتیبانی تشخیصی، مشاهده‌های خطای ارائه‌دهنده، نمایش فرمان تأیید exec،
و لاگ‌های پروتکل WebSocket در Gateway می‌شود. این سطح‌ها ممکن است همچنان از
`logging.redactPatterns` به‌عنوان الگوهای اضافی استفاده کنند، اما `redactSensitive: "off"`
باعث نمی‌شود محرمانه‌های خام را منتشر کنند.

## لاگ‌های WebSocket در Gateway

Gateway لاگ‌های پروتکل WebSocket را در دو حالت چاپ می‌کند:

- **حالت عادی (بدون `--verbose`)**: فقط نتایج RPC «جالب» چاپ می‌شوند:
  - خطاها (`ok=false`)
  - فراخوانی‌های کند (آستانهٔ پیش‌فرض: `>= 50ms`)
  - خطاهای parse
- **حالت پرحرف (`--verbose`)**: همهٔ ترافیک درخواست/پاسخ WS را چاپ می‌کند.

### سبک لاگ WS

`openclaw gateway` از یک تغییر سبک برای هر Gateway پشتیبانی می‌کند:

- `--ws-log auto` (پیش‌فرض): حالت عادی بهینه‌سازی شده است؛ حالت پرحرف از خروجی فشرده استفاده می‌کند
- `--ws-log compact`: خروجی فشرده (درخواست/پاسخ جفت‌شده) هنگام پرحرفی
- `--ws-log full`: خروجی کامل برای هر فریم هنگام پرحرفی
- `--compact`: نام مستعار برای `--ws-log compact`

نمونه‌ها:

```bash
# optimized (only errors/slow)
openclaw gateway

# show all WS traffic (paired)
openclaw gateway --verbose --ws-log compact

# show all WS traffic (full meta)
openclaw gateway --verbose --ws-log full
```

## قالب‌بندی کنسول (ثبت لاگ زیرسامانه)

قالب‌ساز کنسول **از TTY آگاه است** و خط‌های پیشونددار و سازگار چاپ می‌کند.
لاگرهای زیرسامانه خروجی را گروه‌بندی‌شده و قابل پویش نگه می‌دارند.

رفتار:

- **پیشوندهای زیرسامانه** در هر خط (مثلاً `[gateway]`، `[canvas]`، `[tailscale]`)
- **رنگ‌های زیرسامانه** (پایدار برای هر زیرسامانه) به‌همراه رنگ‌آمیزی سطح
- **رنگ وقتی خروجی TTY است یا محیط شبیه یک ترمینال غنی به نظر می‌رسد** (`TERM`/`COLORTERM`/`TERM_PROGRAM`)، با رعایت `NO_COLOR`
- **پیشوندهای کوتاه‌شدهٔ زیرسامانه**: `gateway/` + `channels/` ابتدایی را حذف می‌کند، 2 بخش آخر را نگه می‌دارد (مثلاً `whatsapp/outbound`)
- **زیرلاگرها بر اساس زیرسامانه** (پیشوند خودکار + فیلد ساخت‌یافتهٔ `{ subsystem }`)
- **`logRaw()`** برای خروجی QR/UX (بدون پیشوند، بدون قالب‌بندی)
- **سبک‌های کنسول** (مثلاً `pretty | compact | json`)
- **سطح لاگ کنسول** جدا از سطح لاگ فایل (وقتی `logging.level` روی `debug`/`trace` تنظیم شده باشد، فایل جزئیات کامل را نگه می‌دارد)
- **بدنه‌های پیام WhatsApp** در سطح `debug` لاگ می‌شوند (برای دیدن آن‌ها از `--verbose` استفاده کنید)

این کار لاگ‌های فایل موجود را پایدار نگه می‌دارد و در عین حال خروجی تعاملی را قابل پویش می‌کند.
