## ابتدا دامنه: مدل امنیتی دستیار شخصی

راهنمای امنیتی OpenClaw یک استقرار **دستیار شخصی** را فرض می‌کند: یک مرز اپراتور مورد اعتماد، احتمالاً با عامل‌های متعدد.

- وضعیت امنیتی پشتیبانی‌شده: یک کاربر/مرز اعتماد برای هر gateway (ترجیحاً یک کاربر سیستم‌عامل/میزبان/VPS برای هر مرز).
- مرز امنیتی پشتیبانی‌نشده: یک gateway/عامل مشترک که توسط کاربران متقابلاً نامطمئن یا متخاصم استفاده می‌شود.
- اگر جداسازی کاربران متخاصم لازم است، بر اساس مرز اعتماد جدا کنید (gateway + اعتبارنامه‌های جداگانه، و در حالت ایده‌آل کاربران/میزبان‌های سیستم‌عامل جداگانه).
- اگر چند کاربر نامطمئن بتوانند به یک عامل دارای ابزار پیام بدهند، آن‌ها را به‌عنوان افرادی در نظر بگیرید که همان اختیار ابزار تفویض‌شده برای آن عامل را به اشتراک دارند.

این صفحه سخت‌سازی **در همین مدل** را توضیح می‌دهد. این راهنما ادعای جداسازی چندمستاجره خصمانه روی یک gateway مشترک را ندارد.

## بررسی سریع: `openclaw security audit`

همچنین ببینید: [راستی‌آزمایی رسمی (مدل‌های امنیتی)](/fa/security/formal-verification)

این را به‌طور منظم اجرا کنید (به‌ویژه پس از تغییر پیکربندی یا در معرض قرار دادن سطوح شبکه):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

`security audit --fix` عمداً محدود می‌ماند: سیاست‌های گروهی باز رایج را به
فهرست‌های مجاز تغییر می‌دهد، `logging.redactSensitive: "tools"` را بازیابی
می‌کند، مجوزهای state/config/include-file را سخت‌گیرانه‌تر می‌کند، و هنگام اجرا
روی Windows به‌جای POSIX `chmod` از بازنشانی‌های ACL ویندوز استفاده می‌کند.

این فرمان خطاهای رایج را علامت‌گذاری می‌کند (در معرض بودن احراز هویت Gateway، در معرض بودن کنترل مرورگر، فهرست‌های مجاز ارتقایافته، مجوزهای فایل‌سیستم، تأییدهای اجرای سهل‌گیرانه، و در معرض بودن ابزارهای کانال باز).

OpenClaw هم یک محصول است و هم یک آزمایش: شما رفتار مدل‌های پیشرو را به سطوح پیام‌رسانی واقعی و ابزارهای واقعی وصل می‌کنید. **هیچ راه‌اندازی «کاملاً امنی» وجود ندارد.** هدف این است که آگاهانه تصمیم بگیرید:

- چه کسی می‌تواند با ربات شما صحبت کند
- ربات در کجا مجاز است عمل کند
- ربات به چه چیزهایی می‌تواند دست بزند

با کوچک‌ترین سطح دسترسی که هنوز کار می‌کند شروع کنید، سپس با افزایش اطمینان آن را گسترده‌تر کنید.

### استقرار و اعتماد به میزبان

OpenClaw فرض می‌کند مرز میزبان و پیکربندی مورد اعتماد است:

- اگر کسی بتواند وضعیت/پیکربندی میزبان Gateway (`~/.openclaw`، شامل `openclaw.json`) را تغییر دهد، او را اپراتور مورد اعتماد در نظر بگیرید.
- اجرای یک Gateway برای چند اپراتور متقابلاً نامطمئن/متخاصم **راه‌اندازی توصیه‌شده‌ای نیست**.
- برای تیم‌های با اعتماد ترکیبی، مرزهای اعتماد را با gatewayهای جداگانه جدا کنید (یا حداقل کاربران/میزبان‌های سیستم‌عامل جداگانه).
- پیش‌فرض توصیه‌شده: یک کاربر برای هر ماشین/میزبان (یا VPS)، یک gateway برای آن کاربر، و یک یا چند عامل در آن gateway.
- درون یک نمونه Gateway، دسترسی اپراتور احراز هویت‌شده یک نقش کنترل‌پلین مورد اعتماد است، نه نقش مستاجر به‌ازای هر کاربر.
- شناسه‌های نشست (`sessionKey`، شناسه‌های نشست، برچسب‌ها) انتخابگرهای مسیریابی هستند، نه توکن‌های مجوزدهی.
- اگر چند نفر بتوانند به یک عامل دارای ابزار پیام بدهند، هرکدام می‌توانند همان مجموعه مجوز را هدایت کنند. جداسازی نشست/حافظه به‌ازای هر کاربر به حریم خصوصی کمک می‌کند، اما یک عامل مشترک را به مجوزدهی میزبان به‌ازای هر کاربر تبدیل نمی‌کند.

### فضای کاری Slack مشترک: ریسک واقعی

اگر «همه افراد در Slack می‌توانند به ربات پیام بدهند»، ریسک اصلی اختیار ابزار تفویض‌شده است:

- هر فرستنده مجاز می‌تواند فراخوانی ابزارها (`exec`، مرورگر، ابزارهای شبکه/فایل) را در چارچوب سیاست عامل القا کند؛
- تزریق prompt/محتوا از یک فرستنده می‌تواند باعث اقداماتی شود که بر وضعیت، دستگاه‌ها یا خروجی‌های مشترک اثر می‌گذارد؛
- اگر یک عامل مشترک اعتبارنامه‌ها/فایل‌های حساس داشته باشد، هر فرستنده مجاز به‌طور بالقوه می‌تواند از طریق استفاده از ابزارها خروج داده را هدایت کند.

برای گردش‌کارهای تیمی از عامل‌ها/gatewayهای جداگانه با ابزارهای حداقلی استفاده کنید؛ عامل‌های دارای داده‌های شخصی را خصوصی نگه دارید.

### عامل مشترک شرکتی: الگوی قابل قبول

این زمانی قابل قبول است که همه استفاده‌کنندگان از آن عامل در یک مرز اعتماد یکسان باشند (برای مثال یک تیم شرکتی) و عامل کاملاً در دامنه کسب‌وکار محدود شده باشد.

- آن را روی یک ماشین/VM/container اختصاصی اجرا کنید؛
- برای آن runtime از یک کاربر سیستم‌عامل اختصاصی + مرورگر/پروفایل/حساب‌های اختصاصی استفاده کنید؛
- آن runtime را وارد حساب‌های شخصی Apple/Google یا پروفایل‌های شخصی مدیریت رمز عبور/مرورگر نکنید.

اگر هویت‌های شخصی و شرکتی را روی همان runtime ترکیب کنید، جداسازی را از بین می‌برید و ریسک در معرض قرار گرفتن داده‌های شخصی را افزایش می‌دهید.

## مفهوم اعتماد Gateway و Node

Gateway و Node را یک دامنه اعتماد اپراتور با نقش‌های متفاوت در نظر بگیرید:

- **Gateway** کنترل‌پلین و سطح سیاست است (`gateway.auth`، سیاست ابزار، مسیریابی).
- **Node** سطح اجرای راه‌دور جفت‌شده با آن Gateway است (فرمان‌ها، اقدامات دستگاه، قابلیت‌های محلی میزبان).
- فراخوانی‌کننده‌ای که در Gateway احراز هویت شده باشد، در دامنه Gateway مورد اعتماد است. پس از جفت‌سازی، اقدامات Node اقدامات اپراتور مورد اعتماد روی آن Node هستند.
- کلاینت‌های بک‌اند loopback مستقیم که با توکن/رمز مشترک gateway احراز هویت شده‌اند
  می‌توانند بدون ارائه هویت دستگاه کاربر، RPCهای داخلی کنترل‌پلین را انجام دهند.
  این دور زدن جفت‌سازی راه‌دور یا مرورگر نیست: کلاینت‌های شبکه، کلاینت‌های Node،
  کلاینت‌های دارای توکن دستگاه، و هویت‌های صریح دستگاه همچنان از مسیر جفت‌سازی
  و اعمال ارتقای دامنه عبور می‌کنند.
- `sessionKey` انتخاب مسیریابی/زمینه است، نه احراز هویت به‌ازای هر کاربر.
- تأییدهای Exec (فهرست مجاز + پرسش) حفاظ‌هایی برای نیت اپراتور هستند، نه جداسازی چندمستاجره خصمانه.
- پیش‌فرض محصول OpenClaw برای راه‌اندازی‌های تک‌اپراتور مورد اعتماد این است که اجرای میزبان روی `gateway`/`node` بدون درخواست تأیید مجاز باشد (`security="full"`، `ask="off"` مگر اینکه آن را سخت‌گیرانه‌تر کنید). این پیش‌فرض یک تجربه کاربری عمدی است، نه به‌خودی‌خود یک آسیب‌پذیری.
- تأییدهای Exec زمینه دقیق درخواست و عملوندهای فایل محلی مستقیم را به‌صورت بهترین تلاش متصل می‌کنند؛ آن‌ها از نظر معنایی هر مسیر بارگذار runtime/مفسر را مدل نمی‌کنند. برای مرزهای قوی از sandboxing و جداسازی میزبان استفاده کنید.

اگر به جداسازی کاربران خصمانه نیاز دارید، مرزهای اعتماد را بر اساس کاربر/میزبان سیستم‌عامل جدا کنید و gatewayهای جداگانه اجرا کنید.

## ماتریس مرز اعتماد

هنگام تریاژ ریسک از این به‌عنوان مدل سریع استفاده کنید:

| مرز یا کنترل                                             | معنای آن                                           | برداشت نادرست رایج                                                            |
| --------------------------------------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------- |
| `gateway.auth` (token/password/trusted-proxy/device auth) | فراخوانی‌کنندگان را برای APIهای gateway احراز هویت می‌کند | «برای امن بودن به امضاهای به‌ازای هر پیام روی هر فریم نیاز دارد»              |
| `sessionKey`                                              | کلید مسیریابی برای انتخاب زمینه/نشست              | «کلید نشست یک مرز احراز هویت کاربر است»                                       |
| حفاظ‌های prompt/محتوا                                    | ریسک سوءاستفاده از مدل را کاهش می‌دهند            | «تزریق prompt به‌تنهایی دور زدن احراز هویت را ثابت می‌کند»                    |
| `canvas.eval` / browser evaluate                          | قابلیت عمدی اپراتور هنگام فعال بودن               | «هر primitive ارزیابی JS به‌طور خودکار در این مدل اعتماد آسیب‌پذیری است»     |
| پوسته `!` در TUI محلی                                    | اجرای محلی صریحاً فعال‌شده توسط اپراتور           | «فرمان کاربردی پوسته محلی تزریق راه‌دور است»                                  |
| جفت‌سازی Node و فرمان‌های Node                            | اجرای راه‌دور در سطح اپراتور روی دستگاه‌های جفت‌شده | «کنترل دستگاه راه‌دور باید به‌طور پیش‌فرض دسترسی کاربر نامطمئن تلقی شود»      |
| `gateway.nodes.pairing.autoApproveCidrs`                  | سیاست ثبت‌نام Node در شبکه مورد اعتماد به‌صورت opt-in | «یک فهرست مجاز غیرفعال به‌صورت پیش‌فرض، آسیب‌پذیری جفت‌سازی خودکار است»       |

## خط پایه سخت‌شده در ۶۰ ثانیه

ابتدا از این خط پایه استفاده کنید، سپس ابزارها را به‌صورت انتخابی برای هر عامل مورد اعتماد دوباره فعال کنید:

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

این کار Gateway را فقط محلی نگه می‌دارد، DMها را جدا می‌کند، و ابزارهای کنترل‌پلین/runtime را به‌طور پیش‌فرض غیرفعال می‌کند.

## قاعده سریع صندوق ورودی مشترک

اگر بیش از یک نفر می‌تواند به ربات شما DM بدهد:

- `session.dmScope: "per-channel-peer"` را تنظیم کنید (یا برای کانال‌های چندحسابی `"per-account-channel-peer"`).
- `dmPolicy: "pairing"` یا فهرست‌های مجاز سخت‌گیرانه را حفظ کنید.
- هرگز DMهای مشترک را با دسترسی گسترده به ابزارها ترکیب نکنید.
- این کار صندوق‌های ورودی مشارکتی/مشترک را سخت‌سازی می‌کند، اما وقتی کاربران دسترسی نوشتن به میزبان/پیکربندی را به اشتراک دارند، برای جداسازی هم‌مستاجر خصمانه طراحی نشده است.

## مدل دیدپذیری زمینه

OpenClaw دو مفهوم را جدا می‌کند:

- **مجوز تحریک**: چه کسی می‌تواند عامل را تحریک کند (`dmPolicy`، `groupPolicy`، فهرست‌های مجاز، دروازه‌های mention).
- **دیدپذیری زمینه**: چه زمینه تکمیلی‌ای به ورودی مدل تزریق می‌شود (بدنه پاسخ، متن نقل‌قول‌شده، تاریخچه thread، متادیتای forward‌شده).

فهرست‌های مجاز، تحریک‌ها و مجوزدهی فرمان را کنترل می‌کنند. تنظیم `contextVisibility` کنترل می‌کند زمینه تکمیلی (پاسخ‌های نقل‌قول‌شده، ریشه‌های thread، تاریخچه واکشی‌شده) چگونه فیلتر شود:

- `contextVisibility: "all"` (پیش‌فرض) زمینه تکمیلی را همان‌طور که دریافت شده نگه می‌دارد.
- `contextVisibility: "allowlist"` زمینه تکمیلی را به فرستندگانی فیلتر می‌کند که توسط بررسی‌های فعال فهرست مجاز اجازه دارند.
- `contextVisibility: "allowlist_quote"` مانند `allowlist` رفتار می‌کند، اما همچنان یک پاسخ نقل‌قول‌شده صریح را نگه می‌دارد.

`contextVisibility` را به‌ازای هر کانال یا هر اتاق/گفت‌وگو تنظیم کنید. برای جزئیات راه‌اندازی، [گفت‌وگوهای گروهی](/fa/channels/groups#context-visibility-and-allowlists) را ببینید.

راهنمای تریاژ مشورتی:

- ادعاهایی که فقط نشان می‌دهند «مدل می‌تواند متن نقل‌شده یا تاریخی را از فرستندگانی که در فهرست مجاز نیستند ببیند» یافته‌های مقاوم‌سازی هستند که با `contextVisibility` قابل رسیدگی‌اند، نه این‌که به‌خودی‌خود دورزدن مرز auth یا sandbox باشند.
- برای داشتن اثر امنیتی، گزارش‌ها همچنان به یک دورزدن نشان‌داده‌شده‌ی مرز اعتماد نیاز دارند (auth، سیاست، sandbox، approval، یا مرز مستند دیگری).

## آنچه ممیزی بررسی می‌کند (در سطح بالا)

- **دسترسی ورودی** (سیاست‌های DM، سیاست‌های گروه، فهرست‌های مجاز): آیا افراد ناشناس می‌توانند ربات را فعال کنند؟
- **دامنه اثر ابزار** (ابزارهای دارای سطح دسترسی بالا + اتاق‌های باز): آیا prompt injection می‌تواند به عملیات shell/file/network تبدیل شود؟
- **انحراف تأیید exec** (`security=full`، `autoAllowSkills`، فهرست‌های مجاز مفسر بدون `strictInlineEval`): آیا محافظ‌های host-exec هنوز همان کاری را می‌کنند که فکر می‌کنید؟
  - `security="full"` یک هشدار وضعیت کلی است، نه اثبات یک باگ. این پیش‌فرض انتخاب‌شده برای راه‌اندازی‌های دستیار شخصی قابل‌اعتماد است؛ فقط زمانی آن را سخت‌گیرانه‌تر کنید که مدل تهدید شما به محافظ‌های approval یا فهرست مجاز نیاز دارد.
- **در معرض بودن شبکه** (bind/auth مربوط به Gateway، Tailscale Serve/Funnel، توکن‌های auth ضعیف/کوتاه).
- **در معرض بودن کنترل مرورگر** (نودهای راه‌دور، پورت‌های relay، endpointهای CDP راه‌دور).
- **بهداشت دیسک محلی** (مجوزها، symlinkها، includeهای config، مسیرهای «synced folder»).
- **Pluginها** (Pluginها بدون فهرست مجاز صریح بارگذاری می‌شوند).
- **انحراف/پیکربندی نادرست سیاست** (تنظیمات sandbox docker پیکربندی شده اما حالت sandbox خاموش است؛ الگوهای بی‌اثر `gateway.nodes.denyCommands` چون تطبیق فقط بر اساس نام دقیق دستور انجام می‌شود (برای مثال `system.run`) و متن shell را بررسی نمی‌کند؛ ورودی‌های خطرناک `gateway.nodes.allowCommands`؛ مقدار سراسری `tools.profile="minimal"` که با profileهای هر agent override شده است؛ ابزارهای متعلق به Plugin که زیر سیاست ابزار آسان‌گیرانه قابل دسترسی‌اند).
- **انحراف انتظار runtime** (برای مثال فرض این‌که exec ضمنی هنوز به معنی `sandbox` است در حالی که `tools.exec.host` اکنون به‌طور پیش‌فرض `auto` است، یا تنظیم صریح `tools.exec.host="sandbox"` در حالی که حالت sandbox خاموش است).
- **بهداشت مدل** (وقتی مدل‌های پیکربندی‌شده قدیمی به نظر می‌رسند هشدار می‌دهد؛ مانع سخت نیست).

اگر `--deep` را اجرا کنید، OpenClaw همچنین یک probe زنده‌ی Gateway را در حد بهترین تلاش انجام می‌دهد.

## نقشه ذخیره‌سازی اعتبارنامه‌ها

هنگام ممیزی دسترسی یا تصمیم‌گیری درباره پشتیبان‌گیری از این استفاده کنید:

- **WhatsApp**: `~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
- **توکن ربات Telegram**: config/env یا `channels.telegram.tokenFile` (فقط فایل معمولی؛ symlinkها رد می‌شوند)
- **توکن ربات Discord**: config/env یا SecretRef (providerهای env/file/exec)
- **توکن‌های Slack**: config/env (`channels.slack.*`)
- **فهرست‌های مجاز pairing**:
  - `~/.openclaw/credentials/<channel>-allowFrom.json` (حساب پیش‌فرض)
  - `~/.openclaw/credentials/<channel>-<accountId>-allowFrom.json` (حساب‌های غیرپیش‌فرض)
- **profileهای auth مدل**: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`

- **واردسازی OAuth قدیمی**: `~/.openclaw/credentials/oauth.json`

## چک‌لیست ممیزی امنیتی

وقتی ممیزی یافته‌ها را چاپ می‌کند، این را به‌عنوان ترتیب اولویت در نظر بگیرید:

1. **هر چیزی که “open” است + ابزارها فعال‌اند**: ابتدا DMها/گروه‌ها را قفل کنید (pairing/فهرست‌های مجاز)، سپس سیاست ابزار/sandboxing را سخت‌گیرانه‌تر کنید.
2. **در معرض بودن شبکه عمومی** (LAN bind، Funnel، auth گم‌شده): فوراً رفع کنید.
3. **در معرض بودن کنترل مرورگر از راه‌دور**: با آن مانند دسترسی operator برخورد کنید (فقط tailnet، نودها را عامدانه pair کنید، از در معرض بودن عمومی اجتناب کنید).
4. **مجوزها**: مطمئن شوید state/config/credentials/auth برای group/world قابل خواندن نیستند.
5. **Pluginها**: فقط آنچه را صریحاً به آن اعتماد دارید بارگذاری کنید.
6. **انتخاب مدل**: برای هر رباتی که ابزار دارد، مدل‌های مدرن و مقاوم‌شده در برابر دستورالعمل را ترجیح دهید.

## واژه‌نامه ممیزی امنیتی

هر یافته ممیزی با یک `checkId` ساخت‌یافته کلیدگذاری می‌شود (برای مثال
`gateway.bind_no_auth` یا `tools.exec.security_full_configured`). کلاس‌های شدت
بحرانی رایج:

- `fs.*` — مجوزهای فایل‌سیستم برای state، config، credentials، profileهای auth.
- `gateway.*` — حالت bind، auth، Tailscale، Control UI، راه‌اندازی trusted-proxy.
- `hooks.*`، `browser.*`، `sandbox.*`، `tools.exec.*` — مقاوم‌سازی برای هر سطح.
- `plugins.*`، `skills.*` — زنجیره تأمین plugin/skill و یافته‌های scan.
- `security.exposure.*` — بررسی‌های میان‌بُر که در آن‌ها سیاست دسترسی با دامنه اثر ابزار تلاقی می‌کند.

کاتالوگ کامل را با سطوح شدت، کلیدهای رفع، و پشتیبانی auto-fix در
[بررسی‌های ممیزی امنیتی](/fa/gateway/security/audit-checks) ببینید.

## Control UI روی HTTP

Control UI برای تولید هویت دستگاه به یک **زمینه امن** (HTTPS یا localhost) نیاز دارد. `gateway.controlUi.allowInsecureAuth` یک toggle سازگاری محلی است:

- روی localhost، وقتی صفحه از طریق HTTP غیرامن بارگذاری شده باشد، auth مربوط به Control UI را بدون هویت دستگاه مجاز می‌کند.
- بررسی‌های pairing را دور نمی‌زند.
- الزامات هویت دستگاه راه‌دور (غیر-localhost) را سست نمی‌کند.

HTTPS (Tailscale Serve) را ترجیح دهید یا UI را روی `127.0.0.1` باز کنید.

فقط برای سناریوهای break-glass، `gateway.controlUi.dangerouslyDisableDeviceAuth`
بررسی‌های هویت دستگاه را کاملاً غیرفعال می‌کند. این یک کاهش شدید امنیتی است؛
مگر زمانی که فعالانه در حال اشکال‌زدایی هستید و می‌توانید سریعاً آن را برگردانید، آن را خاموش نگه دارید.

جدا از آن پرچم‌های خطرناک، موفقیت در `gateway.auth.mode: "trusted-proxy"`
می‌تواند sessionهای Control UI مربوط به **operator** را بدون هویت دستگاه بپذیرد. این یک رفتار عمدی auth-mode است، نه میان‌بری برای `allowInsecureAuth`، و همچنان
به sessionهای Control UI با نقش node گسترش پیدا نمی‌کند.

`openclaw security audit` وقتی این تنظیم فعال باشد هشدار می‌دهد.

## پیکربندی reverse proxy

اگر Gateway را پشت یک reverse proxy (nginx، Caddy، Traefik، و غیره) اجرا می‌کنید، برای رسیدگی درست به IP کلاینت forwarded، `gateway.trustedProxies` را پیکربندی کنید.

وقتی Gateway headerهای proxy را از آدرسی تشخیص دهد که در `trustedProxies` **نیست**، اتصال‌ها را به‌عنوان کلاینت محلی در نظر نمی‌گیرد. اگر auth مربوط به gateway غیرفعال باشد، آن اتصال‌ها رد می‌شوند. این از دورزدن authentication جلوگیری می‌کند که در آن اتصال‌های proxied در غیر این صورت طوری به نظر می‌رسند که از localhost آمده‌اند و اعتماد خودکار دریافت می‌کنند.

`gateway.trustedProxies` همچنین به `gateway.auth.mode: "trusted-proxy"` خوراک می‌دهد، اما آن حالت auth سخت‌گیرانه‌تر است:

- auth نوع trusted-proxy به‌طور پیش‌فرض **روی proxyهای با منبع loopback به‌صورت fail closed عمل می‌کند**
- reverse proxyهای loopback روی همان host می‌توانند از `gateway.trustedProxies` برای تشخیص local-client و رسیدگی به IP forwarded استفاده کنند
- reverse proxyهای loopback روی همان host فقط زمانی می‌توانند `gateway.auth.mode: "trusted-proxy"` را برآورده کنند که `gateway.auth.trustedProxy.allowLoopback = true` باشد؛ در غیر این صورت از auth توکن/گذرواژه استفاده کنید

```yaml
gateway:
  trustedProxies:
    - "10.0.0.1" # reverse proxy IP
  # Optional. Default false.
  # Only enable if your proxy cannot provide X-Forwarded-For.
  allowRealIpFallback: false
  auth:
    mode: password
    password: ${OPENCLAW_GATEWAY_PASSWORD}
```

وقتی `trustedProxies` پیکربندی شده باشد، Gateway برای تعیین IP کلاینت از `X-Forwarded-For` استفاده می‌کند. `X-Real-IP` به‌طور پیش‌فرض نادیده گرفته می‌شود مگر این‌که `gateway.allowRealIpFallback: true` صریحاً تنظیم شده باشد.

headerهای trusted proxy باعث نمی‌شوند pairing دستگاه node به‌طور خودکار trusted شود.
`gateway.nodes.pairing.autoApproveCidrs` یک سیاست operator جداگانه است که به‌طور پیش‌فرض غیرفعال است. حتی وقتی فعال باشد، مسیرهای header مربوط به trusted-proxy با منبع loopback
از auto-approval نود حذف می‌شوند چون فراخواننده‌های محلی می‌توانند آن
headerها را جعل کنند، از جمله زمانی که auth نوع loopback trusted-proxy صریحاً فعال شده باشد.

رفتار خوب reverse proxy (بازنویسی headerهای forwarding ورودی):

```nginx
proxy_set_header X-Forwarded-For $remote_addr;
proxy_set_header X-Real-IP $remote_addr;
```

رفتار بد reverse proxy (افزودن/حفظ headerهای forwarding غیرقابل‌اعتماد):

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
```

## یادداشت‌های HSTS و origin

- Gateway مربوط به OpenClaw ابتدا محلی/loopback است. اگر TLS را در یک reverse proxy خاتمه می‌دهید، HSTS را همان‌جا روی دامنه HTTPS روبه‌روی proxy تنظیم کنید.
- اگر خود gateway HTTPS را خاتمه می‌دهد، می‌توانید `gateway.http.securityHeaders.strictTransportSecurity` را تنظیم کنید تا header HSTS از پاسخ‌های OpenClaw منتشر شود.
- راهنمای دقیق deployment در [Trusted Proxy Auth](/fa/gateway/trusted-proxy-auth#tls-termination-and-hsts) است.
- برای deploymentهای غیر-loopback مربوط به Control UI، `gateway.controlUi.allowedOrigins` به‌طور پیش‌فرض لازم است.
- `gateway.controlUi.allowedOrigins: ["*"]` یک سیاست صریح allow-all برای browser-origin است، نه یک پیش‌فرض مقاوم‌سازی‌شده. خارج از تست محلی کاملاً کنترل‌شده از آن اجتناب کنید.
- شکست‌های auth مربوط به browser-origin روی loopback حتی وقتی معافیت عمومی loopback فعال باشد همچنان rate-limit می‌شوند، اما کلید lockout به‌جای یک bucket مشترک localhost، برای هر مقدار normalized `Origin` محدوده‌بندی می‌شود.
- `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت fallback مربوط به Host-header origin را فعال می‌کند؛ با آن به‌عنوان یک سیاست خطرناک انتخاب‌شده توسط operator برخورد کنید.
- با DNS rebinding و رفتار header مربوط به proxy-host به‌عنوان دغدغه‌های مقاوم‌سازی deployment برخورد کنید؛ `trustedProxies` را محدود نگه دارید و از در معرض قرار دادن مستقیم gateway در اینترنت عمومی اجتناب کنید.

## لاگ‌های session محلی روی دیسک قرار دارند

OpenClaw رونوشت‌های session را روی دیسک زیر `~/.openclaw/agents/<agentId>/sessions/*.jsonl` ذخیره می‌کند.
این برای تداوم session و (اختیاراً) indexing حافظه session لازم است، اما همچنین یعنی
**هر process/user با دسترسی فایل‌سیستم می‌تواند آن لاگ‌ها را بخواند**. دسترسی دیسک را به‌عنوان مرز اعتماد در نظر بگیرید و مجوزهای `~/.openclaw` را محدود کنید (بخش ممیزی زیر را ببینید). اگر به جداسازی قوی‌تر بین agentها نیاز دارید، آن‌ها را زیر کاربران OS جداگانه یا hostهای جداگانه اجرا کنید.

## اجرای Node (`system.run`)

اگر یک نود macOS pair شده باشد، Gateway می‌تواند `system.run` را روی آن نود فراخوانی کند. این **اجرای کد از راه‌دور** روی Mac است:

- به جفت‌سازی Node نیاز دارد (تأیید + توکن).
- جفت‌سازی Node در Gateway یک سطح تأیید برای هر فرمان نیست. این کار هویت/اعتماد Node و صدور توکن را برقرار می‌کند.
- Gateway یک سیاست کلی سراسری برای فرمان‌های Node را از طریق `gateway.nodes.allowCommands` / `denyCommands` اعمال می‌کند.
- روی Mac از طریق **Settings → Exec approvals** کنترل می‌شود (امنیت + پرسش + فهرست مجاز).
- سیاست `system.run` برای هر Node همان فایل تأییدهای اجرای خود Node است (`exec.approvals.node.*`) که می‌تواند از سیاست سراسری شناسه فرمان Gateway سخت‌گیرانه‌تر یا آسان‌گیرانه‌تر باشد.
- Nodeای که با `security="full"` و `ask="off"` اجرا می‌شود، از مدل پیش‌فرض اپراتور مورد اعتماد پیروی می‌کند. این را رفتار مورد انتظار بدانید، مگر اینکه استقرار شما صراحتاً موضع تأیید یا فهرست مجاز سخت‌گیرانه‌تری بخواهد.
- حالت تأیید، زمینه دقیق درخواست و، در صورت امکان، یک عملوند مشخص اسکریپت/فایل محلی را مقید می‌کند. اگر OpenClaw نتواند دقیقاً یک فایل محلی مستقیم را برای یک فرمان مفسر/زمان اجرا شناسایی کند، اجرای متکی به تأیید رد می‌شود، نه اینکه پوشش معنایی کامل وعده داده شود.
- برای `host=node`، اجراهای متکی به تأیید همچنین یک
  `systemRunPlan` آماده‌شده متعارف را ذخیره می‌کنند؛ ارسال‌های تأییدشده بعدی همان طرح ذخیره‌شده را دوباره استفاده می‌کنند، و اعتبارسنجی Gateway ویرایش‌های فراخواننده روی زمینه فرمان/cwd/session را پس از ایجاد درخواست تأیید رد می‌کند.
- اگر اجرای راه دور را نمی‌خواهید، امنیت را روی **deny** بگذارید و جفت‌سازی Node را برای آن Mac حذف کنید.

این تمایز برای تریاژ مهم است:

- یک Node جفت‌شده که دوباره وصل می‌شود و فهرست فرمان متفاوتی اعلام می‌کند، به‌خودی‌خود آسیب‌پذیری نیست، اگر سیاست سراسری Gateway و تأییدهای اجرای محلی Node همچنان مرز اجرای واقعی را اعمال کنند.
- گزارش‌هایی که فراداده جفت‌سازی Node را به‌عنوان یک لایه تأیید پنهان دوم برای هر فرمان تلقی می‌کنند، معمولاً سردرگمی سیاست/تجربه کاربری هستند، نه دور زدن مرز امنیتی.

## Skills پویا (ناظر / Nodeهای راه دور)

OpenClaw می‌تواند فهرست Skills را در میانه نشست تازه‌سازی کند:

- **ناظر Skills**: تغییرات در `SKILL.md` می‌توانند اسنپ‌شات Skills را در نوبت بعدی عامل به‌روزرسانی کنند.
- **Nodeهای راه دور**: وصل شدن یک Node در macOS می‌تواند Skills مخصوص macOS را واجد شرایط کند (بر پایه کاوش bin).

پوشه‌های Skills را به‌عنوان **کد مورد اعتماد** تلقی کنید و محدود کنید چه کسانی می‌توانند آن‌ها را تغییر دهند.

## مدل تهدید

دستیار هوش مصنوعی شما می‌تواند:

- فرمان‌های دلخواه shell را اجرا کند
- فایل‌ها را بخواند/بنویسد
- به سرویس‌های شبکه دسترسی داشته باشد
- برای هر کسی پیام بفرستد (اگر به آن دسترسی WhatsApp بدهید)

افرادی که به شما پیام می‌دهند می‌توانند:

- تلاش کنند هوش مصنوعی شما را فریب دهند تا کارهای بد انجام دهد
- برای دسترسی به داده‌های شما مهندسی اجتماعی انجام دهند
- برای جزئیات زیرساختی کاوش کنند

## مفهوم اصلی: کنترل دسترسی پیش از هوشمندی

بیشتر شکست‌ها در اینجا سوءاستفاده‌های پیچیده نیستند؛ «کسی به ربات پیام داد و ربات همان کاری را کرد که خواسته بودند.»

موضع OpenClaw:

- **اول هویت:** تصمیم بگیرید چه کسی می‌تواند با ربات صحبت کند (جفت‌سازی پیام مستقیم / فهرست‌های مجاز / «باز» صریح).
- **بعد دامنه:** تصمیم بگیرید ربات کجا مجاز به اقدام است (فهرست‌های مجاز گروه + دروازه‌گذاری با منشن، ابزارها، sandboxing، مجوزهای دستگاه).
- **در پایان مدل:** فرض کنید مدل می‌تواند دستکاری شود؛ طوری طراحی کنید که دستکاری شعاع اثر محدودی داشته باشد.

## مدل مجوزدهی فرمان

فرمان‌های اسلش و دستورالعمل‌ها فقط برای **فرستندگان مجاز** پذیرفته می‌شوند. مجوزدهی از
فهرست‌های مجاز/جفت‌سازی کانال به‌علاوه `commands.useAccessGroups` به دست می‌آید (ببینید [پیکربندی](/fa/gateway/configuration)
و [فرمان‌های اسلش](/fa/tools/slash-commands)). اگر فهرست مجاز یک کانال خالی باشد یا شامل `"*"` باشد،
فرمان‌ها عملاً برای آن کانال باز هستند.

`/exec` فقط یک میان‌بر نشست‌محور برای اپراتورهای مجاز است. این فرمان پیکربندی را نمی‌نویسد و
نشست‌های دیگر را تغییر نمی‌دهد.

## ریسک ابزارهای صفحه کنترل

دو ابزار داخلی می‌توانند تغییرات ماندگار در صفحه کنترل ایجاد کنند:

- `gateway` می‌تواند پیکربندی را با `config.schema.lookup` / `config.get` بررسی کند، و با `config.apply`،‏ `config.patch` و `update.run` تغییرات ماندگار ایجاد کند.
- `cron` می‌تواند کارهای زمان‌بندی‌شده‌ای بسازد که پس از پایان چت/وظیفه اصلی هم به اجرا ادامه دهند.

ابزار زمان اجرای فقط-مالک `gateway` همچنان از بازنویسی
`tools.exec.ask` یا `tools.exec.security` خودداری می‌کند؛ نام‌های مستعار قدیمی `tools.bash.*` نیز
پیش از نوشتن، به همان مسیرهای اجرای محافظت‌شده نرمال‌سازی می‌شوند.
ویرایش‌های عامل‌محور `gateway config.apply` و `gateway config.patch` به‌صورت پیش‌فرض
بسته-در-خطا هستند: فقط مجموعه باریکی از مسیرهای prompt، مدل و دروازه‌گذاری منشن
برای تنظیم توسط عامل مجازند. بنابراین درخت‌های پیکربندی حساس جدید محافظت می‌شوند،
مگر اینکه عمداً به فهرست مجاز افزوده شوند.

برای هر عامل/سطحی که محتوای غیرقابل اعتماد را پردازش می‌کند، این‌ها را به‌صورت پیش‌فرض رد کنید:

```json5
{
  tools: {
    deny: ["gateway", "cron", "sessions_spawn", "sessions_send"],
  },
}
```

`commands.restart=false` فقط اقدام‌های راه‌اندازی مجدد را مسدود می‌کند. این گزینه اقدام‌های پیکربندی/به‌روزرسانی `gateway` را غیرفعال نمی‌کند.

## Plugins

Plugins **درون همان فرایند** با Gateway اجرا می‌شوند. آن‌ها را کد مورد اعتماد تلقی کنید:

- فقط Plugins را از منابعی نصب کنید که به آن‌ها اعتماد دارید.
- فهرست‌های مجاز صریح `plugins.allow` را ترجیح دهید.
- پیش از فعال‌سازی، پیکربندی Plugin را بازبینی کنید.
- پس از تغییرات Plugin، Gateway را راه‌اندازی مجدد کنید.
- اگر Plugins را نصب یا به‌روزرسانی می‌کنید (`openclaw plugins install <package>`،‏ `openclaw plugins update <id>`)، با آن مثل اجرای کد غیرقابل اعتماد برخورد کنید:
  - مسیر نصب، پوشه هر Plugin زیر ریشه نصب Plugin فعال است.
  - OpenClaw پیش از نصب/به‌روزرسانی یک اسکن داخلی کد خطرناک اجرا می‌کند. یافته‌های `critical` به‌صورت پیش‌فرض مسدود می‌کنند.
  - OpenClaw از `npm pack` استفاده می‌کند، سپس در همان پوشه یک `npm install --omit=dev --ignore-scripts` محلی پروژه اجرا می‌کند. تنظیمات نصب npm سراسری به‌ارث‌رسیده نادیده گرفته می‌شوند تا وابستگی‌ها زیر مسیر نصب Plugin بمانند.
  - نسخه‌های سنجاق‌شده و دقیق (`@scope/pkg@1.2.3`) را ترجیح دهید و پیش از فعال‌سازی، کد بازشده روی دیسک را بررسی کنید.
  - `--dangerously-force-unsafe-install` فقط برای موارد اضطراریِ مثبت کاذب اسکن داخلی در جریان‌های نصب/به‌روزرسانی Plugin است. این گزینه بلوک‌های سیاست hook‏ `before_install` در Plugin را دور نمی‌زند و شکست‌های اسکن را دور نمی‌زند.
  - نصب‌های وابستگی Skill با پشتیبانی Gateway از همان تفکیک خطرناک/مشکوک پیروی می‌کنند: یافته‌های داخلی `critical` مسدود می‌شوند مگر اینکه فراخواننده صراحتاً `dangerouslyForceUnsafeInstall` را تنظیم کند، در حالی که یافته‌های مشکوک همچنان فقط هشدار می‌دهند. `openclaw skills install` همچنان جریان جداگانه دانلود/نصب Skill از ClawHub باقی می‌ماند.

جزئیات: [Plugins](/fa/tools/plugin)

## مدل دسترسی پیام مستقیم: جفت‌سازی، فهرست مجاز، باز، غیرفعال

همه کانال‌های فعلیِ دارای قابلیت پیام مستقیم از یک سیاست پیام مستقیم (`dmPolicy` یا `*.dm.policy`) پشتیبانی می‌کنند که پیام‌های مستقیم ورودی را **پیش از** پردازش پیام دروازه‌گذاری می‌کند:

- `pairing` (پیش‌فرض): فرستندگان ناشناخته یک کد کوتاه جفت‌سازی دریافت می‌کنند و ربات پیامشان را تا زمان تأیید نادیده می‌گیرد. کدها پس از 1 ساعت منقضی می‌شوند؛ پیام‌های مستقیم تکراری تا زمانی که درخواست جدیدی ایجاد نشود کد را دوباره نمی‌فرستند. درخواست‌های در انتظار به‌صورت پیش‌فرض به **3 برای هر کانال** محدود می‌شوند.
- `allowlist`: فرستندگان ناشناخته مسدود می‌شوند (بدون دست‌دهی جفت‌سازی).
- `open`: به هر کسی اجازه پیام مستقیم بدهید (عمومی). **نیاز دارد** فهرست مجاز کانال شامل `"*"` باشد (انتخاب آگاهانه صریح).
- `disabled`: پیام‌های مستقیم ورودی را کاملاً نادیده بگیرید.

تأیید از طریق CLI:

```bash
openclaw pairing list <channel>
openclaw pairing approve <channel> <code>
```

جزئیات + فایل‌های روی دیسک: [جفت‌سازی](/fa/channels/pairing)

## ایزوله‌سازی نشست پیام مستقیم (حالت چندکاربره)

به‌صورت پیش‌فرض، OpenClaw **همه پیام‌های مستقیم را به نشست اصلی هدایت می‌کند** تا دستیار شما در دستگاه‌ها و کانال‌ها تداوم داشته باشد. اگر **چند نفر** می‌توانند به ربات پیام مستقیم بدهند (پیام‌های مستقیم باز یا فهرست مجاز چندنفره)، ایزوله‌سازی نشست‌های پیام مستقیم را در نظر بگیرید:

```json5
{
  session: { dmScope: "per-channel-peer" },
}
```

این کار از نشت زمینه بین کاربران جلوگیری می‌کند و هم‌زمان چت‌های گروهی را ایزوله نگه می‌دارد.

این یک مرز زمینه پیام‌رسانی است، نه مرز مدیر میزبان. اگر کاربران نسبت به هم متخاصم‌اند و یک میزبان/پیکربندی Gateway مشترک دارند، به‌جای آن برای هر مرز اعتماد Gatewayهای جداگانه اجرا کنید.

### حالت امن پیام مستقیم (توصیه‌شده)

قطعه بالا را **حالت امن پیام مستقیم** بدانید:

- پیش‌فرض: `session.dmScope: "main"` (همه پیام‌های مستقیم برای تداوم، یک نشست را به اشتراک می‌گذارند).
- پیش‌فرض onboarding در CLI محلی: وقتی تنظیم نشده باشد `session.dmScope: "per-channel-peer"` را می‌نویسد (مقادیر صریح موجود را حفظ می‌کند).
- حالت امن پیام مستقیم: `session.dmScope: "per-channel-peer"` (هر جفت کانال+فرستنده یک زمینه پیام مستقیم ایزوله می‌گیرد).
- ایزوله‌سازی همتای بین‌کانالی: `session.dmScope: "per-peer"` (هر فرستنده در همه کانال‌های هم‌نوع یک نشست می‌گیرد).

اگر چند حساب را روی همان کانال اجرا می‌کنید، به‌جای آن از `per-account-channel-peer` استفاده کنید. اگر همان شخص از چند کانال با شما تماس می‌گیرد، از `session.identityLinks` برای ادغام آن نشست‌های پیام مستقیم در یک هویت متعارف استفاده کنید. ببینید [مدیریت نشست](/fa/concepts/session) و [پیکربندی](/fa/gateway/configuration).

## فهرست‌های مجاز برای پیام‌های مستقیم و گروه‌ها

OpenClaw دو لایه جداگانه «چه کسی می‌تواند مرا فعال کند؟» دارد:

- **فهرست مجاز پیام مستقیم** (`allowFrom` / `channels.discord.allowFrom` / `channels.slack.allowFrom`؛ قدیمی: `channels.discord.dm.allowFrom`،‏ `channels.slack.dm.allowFrom`): چه کسی اجازه دارد در پیام‌های مستقیم با ربات صحبت کند.
  - وقتی `dmPolicy="pairing"` باشد، تأییدها در مخزن فهرست مجاز جفت‌سازی با دامنه حساب زیر `~/.openclaw/credentials/` نوشته می‌شوند (`<channel>-allowFrom.json` برای حساب پیش‌فرض، `<channel>-<accountId>-allowFrom.json` برای حساب‌های غیردیفالت)، و با فهرست‌های مجاز پیکربندی ادغام می‌شوند.
- **فهرست مجاز گروه** (مختص کانال): ربات اصولاً از کدام گروه‌ها/کانال‌ها/سرورها پیام می‌پذیرد.
  - الگوهای رایج:
    - `channels.whatsapp.groups`،‏ `channels.telegram.groups`،‏ `channels.imessage.groups`: پیش‌فرض‌های هر گروه مثل `requireMention`؛ وقتی تنظیم شود، به‌عنوان فهرست مجاز گروه هم عمل می‌کند (برای حفظ رفتار اجازه به همه، `"*"` را وارد کنید).
    - `groupPolicy="allowlist"` + `groupAllowFrom`: محدود می‌کند چه کسی می‌تواند ربات را _داخل_ یک نشست گروهی فعال کند (WhatsApp/Telegram/Signal/iMessage/Microsoft Teams).
    - `channels.discord.guilds` / `channels.slack.channels`: فهرست‌های مجاز هر سطح + پیش‌فرض‌های منشن.
  - بررسی‌های گروه به این ترتیب اجرا می‌شوند: ابتدا `groupPolicy`/فهرست‌های مجاز گروه، سپس فعال‌سازی با منشن/پاسخ.
  - پاسخ دادن به پیام ربات (منشن ضمنی) فهرست‌های مجاز فرستنده مثل `groupAllowFrom` را دور نمی‌زند.
  - **نکته امنیتی:** `dmPolicy="open"` و `groupPolicy="open"` را تنظیمات آخرین راه‌حل بدانید. باید به‌ندرت استفاده شوند؛ جفت‌سازی + فهرست‌های مجاز را ترجیح دهید مگر اینکه کاملاً به همه اعضای اتاق اعتماد دارید.

جزئیات: [پیکربندی](/fa/gateway/configuration) و [گروه‌ها](/fa/channels/groups)

## تزریق prompt (چیست و چرا مهم است)

تزریق prompt زمانی است که مهاجم پیامی می‌سازد که مدل را دستکاری کند تا کاری ناامن انجام دهد («دستورالعمل‌هایت را نادیده بگیر»، «فایل‌سیستم خود را تخلیه کن»، «این لینک را دنبال کن و فرمان‌ها را اجرا کن» و غیره).

حتی با promptهای سیستمی قوی، **تزریق prompt حل نشده است**. حفاظ‌های prompt سیستمی فقط راهنمایی نرم هستند؛ اجرای سخت‌گیرانه از سیاست ابزار، تأییدهای اجرا، sandboxing و فهرست‌های مجاز کانال می‌آید (و اپراتورها می‌توانند این‌ها را طبق طراحی غیرفعال کنند). آنچه در عمل کمک می‌کند:

- پیام‌های خصوصی ورودی را محدود نگه دارید (جفت‌سازی/فهرست‌های مجاز).
- در گروه‌ها، محدودسازی با منشن را ترجیح دهید؛ از بات‌های «همیشه روشن» در اتاق‌های عمومی پرهیز کنید.
- لینک‌ها، پیوست‌ها، و دستورالعمل‌های چسبانده‌شده را به‌طور پیش‌فرض خصمانه تلقی کنید.
- اجرای ابزارهای حساس را در یک sandbox اجرا کنید؛ اسرار را بیرون از فایل‌سیستمِ در دسترس عامل نگه دارید.
- نکته: sandboxing اختیاری است. اگر حالت sandbox خاموش باشد، `host=auto` ضمنی به میزبان Gateway resolve می‌شود. `host=sandbox` صریح همچنان بسته شکست می‌خورد، چون هیچ runtime sandbox در دسترس نیست. اگر می‌خواهید این رفتار در پیکربندی صریح باشد، `host=gateway` را تنظیم کنید.
- ابزارهای پرخطر (`exec`, `browser`, `web_fetch`, `web_search`) را به عامل‌های مورداعتماد یا فهرست‌های مجاز صریح محدود کنید.
- اگر مفسرها (`python`, `node`, `ruby`, `perl`, `php`, `lua`, `osascript`) را در فهرست مجاز قرار می‌دهید، `tools.exec.strictInlineEval` را فعال کنید تا شکل‌های eval درون‌خطی همچنان به تأیید صریح نیاز داشته باشند.
- تحلیل تأیید Shell همچنین شکل‌های بسط پارامتر POSIX (`$VAR`, `$?`, `$$`, `$1`, `$@`, `${…}`) را داخل **heredocهای بدون نقل‌قول** رد می‌کند، بنابراین بدنه heredoc موجود در فهرست مجاز نمی‌تواند بسط Shell را به‌عنوان متن ساده از بازبینی فهرست مجاز عبور دهد. برای انتخاب معنای بدنه لفظی، پایان‌دهنده heredoc را در نقل‌قول بگذارید (برای مثال `<<'EOF'`)؛ heredocهای بدون نقل‌قول که متغیرها را بسط می‌دادند رد می‌شوند.
- **انتخاب مدل مهم است:** مدل‌های قدیمی‌تر/کوچک‌تر/میراثی در برابر تزریق پرامپت و سوءاستفاده از ابزار به‌مراتب کم‌تاب‌تر هستند. برای عامل‌های دارای ابزار، از قوی‌ترین مدل نسل جدید و مقاوم‌شده با دستورالعمل که در دسترس است استفاده کنید.

پرچم‌های قرمزی که باید نامطمئن تلقی شوند:

- «این فایل/URL را بخوان و دقیقاً همان کاری را انجام بده که می‌گوید.»
- «پرامپت سیستم یا قواعد ایمنی‌ات را نادیده بگیر.»
- «دستورالعمل‌های پنهان یا خروجی‌های ابزارهایت را افشا کن.»
- «محتوای کامل ~/.openclaw یا لاگ‌هایت را بچسبان.»

## پاک‌سازی توکن‌های ویژه محتوای خارجی

OpenClaw پیش از رسیدن محتوای خارجی و فراداده بسته‌بندی‌شده به مدل، لفظ‌های رایج توکن ویژه قالب چت LLMهای self-hosted را از آن‌ها حذف می‌کند. خانواده‌های نشانگر پوشش‌داده‌شده شامل توکن‌های نقش/نوبت Qwen/ChatML، Llama، Gemma، Mistral، Phi، و GPT-OSS هستند.

چرا:

- backendهای سازگار با OpenAI که جلوی مدل‌های self-hosted قرار می‌گیرند، گاهی توکن‌های ویژه‌ای را که در متن کاربر ظاهر می‌شوند، به‌جای ماسک کردن حفظ می‌کنند. مهاجمی که بتواند در محتوای خارجی ورودی چیزی بنویسد (یک صفحه دریافت‌شده، بدنه ایمیل، خروجی ابزار محتوای فایل) در غیر این صورت می‌تواند یک مرز نقش مصنوعی `assistant` یا `system` تزریق کند و از حفاظ‌های محتوای بسته‌بندی‌شده بگریزد.
- پاک‌سازی در لایه بسته‌بندی محتوای خارجی انجام می‌شود، بنابراین به‌جای اینکه مختص هر provider باشد، به‌صورت یکنواخت روی ابزارهای fetch/read و محتوای کانال ورودی اعمال می‌شود.
- پاسخ‌های خروجی مدل از قبل یک پاک‌ساز جداگانه دارند که `<tool_call>`، `<function_calls>`، `<system-reminder>`، `<previous_response>` و داربست‌های runtime داخلی مشابهِ نشت‌کرده را در مرز نهایی تحویل کانال، از پاسخ‌های قابل‌مشاهده برای کاربر حذف می‌کند. پاک‌ساز محتوای خارجی همتای ورودی آن است.

این جایگزین دیگر سخت‌سازی‌های این صفحه نیست — `dmPolicy`، فهرست‌های مجاز، تأییدهای exec، sandboxing، و `contextVisibility` همچنان کار اصلی را انجام می‌دهند. این کار یک دورزدن مشخص در لایه tokenizer را علیه stackهای self-hosted می‌بندد که متن کاربر را با توکن‌های ویژه دست‌نخورده forward می‌کنند.

## پرچم‌های دورزدن ناامن محتوای خارجی

OpenClaw شامل پرچم‌های دورزدن صریحی است که بسته‌بندی ایمنی محتوای خارجی را غیرفعال می‌کنند:

- `hooks.mappings[].allowUnsafeExternalContent`
- `hooks.gmail.allowUnsafeExternalContent`
- فیلد payload در Cron با نام `allowUnsafeExternalContent`

راهنما:

- این‌ها را در production تنظیم‌نشده/false نگه دارید.
- فقط به‌طور موقت برای debugging با دامنه بسیار محدود فعال کنید.
- اگر فعال شد، آن عامل را ایزوله کنید (sandbox + حداقل ابزارها + namespace اختصاصی session).

نکته ریسک hooks:

- محتوای بارهای دادهٔ Hook نامطمئن است، حتی وقتی تحویل از سیستم‌هایی می‌آید که شما کنترلشان می‌کنید (محتوای ایمیل/اسناد/وب می‌تواند حامل تزریق پرامپت باشد).
- رده‌های ضعیف مدل این ریسک را افزایش می‌دهند. برای اتوماسیون مبتنی بر Hook، رده‌های قوی و مدرن مدل را ترجیح دهید و سیاست ابزار را محدود نگه دارید (`tools.profile: "messaging"` یا سخت‌گیرانه‌تر)، همراه با سندباکس‌سازی در صورت امکان.

### تزریق پرامپت به پیام‌های خصوصی عمومی نیاز ندارد

حتی اگر **فقط شما** بتوانید به بات پیام بدهید، تزریق پرامپت همچنان می‌تواند از طریق
هر **محتوای نامطمئنی** که بات می‌خواند رخ دهد (نتایج جست‌وجو/دریافت وب، صفحات مرورگر،
ایمیل‌ها، اسناد، پیوست‌ها، لاگ‌ها/کدهای چسبانده‌شده). به بیان دیگر: فرستنده
تنها سطح تهدید نیست؛ **خود محتوا** می‌تواند حامل دستورالعمل‌های خصمانه باشد.

وقتی ابزارها فعال هستند، ریسک معمول، برون‌بری زمینه یا فعال‌سازی
فراخوانی‌های ابزار است. شعاع اثر را با این کارها کاهش دهید:

- استفاده از یک **عامل خواننده** فقط‌خواندنی یا بدون ابزار برای خلاصه‌سازی محتوای نامطمئن،
  سپس ارسال خلاصه به عامل اصلی‌تان.
- غیرفعال نگه داشتن `web_search` / `web_fetch` / `browser` برای عامل‌های دارای ابزار، مگر در صورت نیاز.
- برای ورودی‌های URL در OpenResponses (`input_file` / `input_image`)،
  `gateway.http.endpoints.responses.files.urlAllowlist` و
  `gateway.http.endpoints.responses.images.urlAllowlist` را محدود تنظیم کنید، و `maxUrlParts` را پایین نگه دارید.
  فهرست‌های مجاز خالی تنظیم‌نشده تلقی می‌شوند؛ اگر می‌خواهید دریافت URL را کاملاً غیرفعال کنید، از `files.allowUrl: false` / `images.allowUrl: false`
  استفاده کنید.
- برای ورودی‌های فایل در OpenResponses، متن رمزگشایی‌شدهٔ `input_file` همچنان به‌عنوان
  **محتوای خارجی نامطمئن** تزریق می‌شود. فقط به این دلیل که
  Gateway آن را به‌صورت محلی رمزگشایی کرده است، به مورداعتماد بودن متن فایل تکیه نکنید. بلوک تزریق‌شده همچنان نشانگرهای مرزی صریح
  `<<<EXTERNAL_UNTRUSTED_CONTENT ...>>>` به‌همراه فرادادهٔ `Source: External`
  را دارد، حتی اگر این مسیر بنر طولانی‌تر `SECURITY NOTICE:` را حذف کند.
- همان بسته‌بندی مبتنی بر نشانگر زمانی اعمال می‌شود که درک رسانه متن را
  از اسناد پیوست‌شده استخراج می‌کند و سپس آن متن را به پرامپت رسانه می‌افزاید.
- فعال‌سازی سندباکس‌سازی و فهرست‌های مجاز سخت‌گیرانهٔ ابزار برای هر عاملی که با ورودی نامطمئن سروکار دارد.
- خارج نگه داشتن رازها از پرامپت‌ها؛ در عوض آن‌ها را از طریق env/config روی میزبان Gateway عبور دهید.

بک‌اندهای خودمیزبان سازگار با OpenAI مانند vLLM، SGLang، TGI، LM Studio،

مدیریت توکن‌های ویژهٔ قالب چت با ارائه‌دهندگان میزبانی‌شده متفاوت باشند. اگر یک بک‌اند رشته‌های لفظی
مانند `<|im_start|

OpenClaw پیش از ارسال محتوای خارجیِ بسته‌بندی‌شده به مدل، لیترال‌های رایج توکن‌های ویژهٔ خانواده‌های مدل را از آن حذف می‌کند. بسته‌بندی محتوای خارجی را فعال نگه دارید، و در صورت وجود، تنظیمات بک‌اندی را ترجیح دهید که توکن‌های ویژه را در محتوای ارائه‌شده توسط کاربر جدا یا escape می‌کنند. ارائه‌دهندگان میزبانی‌شده مانند OpenAI و Anthropic از قبل پاک‌سازی سمت درخواست خودشان را اعمال می‌کنند.

### قدرت مدل (نکتهٔ امنیتی)

مقاومت در برابر تزریق پرامپت در رده‌های مختلف مدل **یکسان** نیست. مدل‌های کوچک‌تر/ارزان‌تر معمولاً نسبت به سوءاستفاده از ابزار و ربودن دستورالعمل آسیب‌پذیرتر هستند، به‌ویژه تحت پرامپت‌های خصمانه.

<Warning>
برای عامل‌های دارای ابزار یا عامل‌هایی که محتوای نامطمئن را می‌خوانند، خطر تزریق پرامپت با مدل‌های قدیمی‌تر/کوچک‌تر اغلب بیش از حد بالاست. این بارهای کاری را روی رده‌های ضعیف مدل اجرا نکنید.
</Warning>

توصیه‌ها:

- برای هر باتی که می‌تواند ابزارها را اجرا کند یا به فایل‌ها/شبکه‌ها دسترسی داشته باشد، از **جدیدترین نسل و بهترین ردهٔ مدل** استفاده کنید.
- برای عامل‌های دارای ابزار یا صندوق‌های ورودی نامطمئن، از **رده‌های قدیمی‌تر/ضعیف‌تر/کوچک‌تر** استفاده نکنید؛ خطر تزریق پرامپت بیش از حد بالاست.
- اگر ناچارید از مدل کوچک‌تر استفاده کنید، **دامنهٔ اثر را کاهش دهید** (ابزارهای فقط‌خواندنی، sandboxing قوی، دسترسی حداقلی به فایل‌سیستم، فهرست‌های مجاز سخت‌گیرانه).
- هنگام اجرای مدل‌های کوچک، **sandboxing را برای همهٔ نشست‌ها فعال کنید** و **web_search/web_fetch/browser را غیرفعال کنید** مگر اینکه ورودی‌ها به‌دقت کنترل شده باشند.
- برای دستیارهای شخصی فقط‌گفت‌وگو با ورودی قابل‌اعتماد و بدون ابزار، مدل‌های کوچک‌تر معمولاً مناسب هستند.

## استدلال و خروجی پرجزئیات در گروه‌ها

`/reasoning`، `/verbose` و `/trace` می‌توانند استدلال داخلی، خروجی ابزار، یا عیب‌یابی‌های Plugin را که برای کانال عمومی در نظر گرفته نشده‌اند افشا کنند. در محیط‌های گروهی، با آن‌ها فقط به‌عنوان **اشکال‌زدایی** برخورد کنید و مگر اینکه صراحتاً به آن‌ها نیاز داشته باشید، آن‌ها را خاموش نگه دارید.

راهنما:

- `/reasoning`، `/verbose` و `/trace` را در اتاق‌های عمومی غیرفعال نگه دارید.
- اگر آن‌ها را فعال می‌کنید، این کار را فقط در پیام‌های مستقیمِ مورد اعتماد یا اتاق‌هایی با کنترل دقیق انجام دهید.
- به یاد داشته باشید: خروجی verbose و trace می‌تواند شامل آرگومان‌های ابزار، URLها، داده‌های عیب‌یابی Plugin و داده‌هایی باشد که مدل دیده است.

## نمونه‌های سخت‌سازی پیکربندی

### مجوزهای فایل

پیکربندی + وضعیت را روی میزبان Gateway خصوصی نگه دارید:

- `~/.openclaw/openclaw.json`: `600` (فقط خواندن/نوشتن برای کاربر)
- `~/.openclaw`: `700` (فقط کاربر)

`openclaw doctor` می‌تواند هشدار دهد و پیشنهاد کند این مجوزها را محدودتر کند.

### در معرض شبکه قرار گرفتن (bind، پورت، دیوار آتش)

Gateway، **WebSocket + HTTP** را روی یک پورت واحد چندبخشی می‌کند:

- پیش‌فرض: `18789`
- پیکربندی/پرچم‌ها/env: `gateway.port`، `--port`، `OPENCLAW_GATEWAY_PORT`

این سطح HTTP شامل Control UI و میزبان canvas است:

- Control UI (دارایی‌های SPA) (مسیر پایه پیش‌فرض `/`)
- میزبان Canvas: `/__openclaw__/canvas/` و `/__openclaw__/a2ui/` (HTML/JS دلخواه؛ آن را محتوای غیرقابل اعتماد در نظر بگیرید)

اگر محتوای canvas را در یک مرورگر معمولی بارگذاری می‌کنید، با آن مانند هر صفحه وب غیرقابل اعتماد دیگری رفتار کنید:

- میزبان canvas را در معرض شبکه‌ها/کاربران غیرقابل اعتماد قرار ندهید.
- محتوای canvas را با سطوح وب دارای امتیاز در یک origin مشترک قرار ندهید، مگر اینکه پیامدهای آن را کاملاً درک کرده باشید.

حالت bind کنترل می‌کند Gateway کجا گوش بدهد:

- `gateway.bind: "loopback"` (پیش‌فرض): فقط کلاینت‌های محلی می‌توانند متصل شوند.
- bindهای غیر loopback (`"lan"`، `"tailnet"`، `"custom"`) سطح حمله را گسترش می‌دهند. فقط همراه با احراز هویت Gateway (توکن/رمز عبور مشترک یا پراکسی مورد اعتمادِ درست پیکربندی‌شده) و یک دیوار آتش واقعی از آن‌ها استفاده کنید.

قواعد سرانگشتی:

- Tailscale Serve را به bindهای LAN ترجیح دهید (Serve، Gateway را روی loopback نگه می‌دارد و Tailscale دسترسی را مدیریت می‌کند).
- اگر مجبورید به LAN bind کنید، پورت را در دیوار آتش به یک فهرست مجاز محدود از IPهای مبدأ محدود کنید؛ آن را به‌طور گسترده port-forward نکنید.
- هرگز Gateway را بدون احراز هویت روی `0.0.0.0` در معرض دسترس قرار ندهید.

### انتشار پورت Docker با UFW

اگر OpenClaw را با Docker روی یک VPS اجرا می‌کنید، به یاد داشته باشید که پورت‌های منتشرشده کانتینر
(`-p HOST:CONTAINER` یا `ports:` در Compose) از طریق زنجیره‌های forwarding در Docker مسیریابی می‌شوند،
نه فقط قواعد `INPUT` میزبان.

برای هم‌راستا نگه داشتن ترافیک Docker با سیاست دیوار آتش خود، قواعد را در
`DOCKER-USER` اعمال کنید (این زنجیره پیش از قواعد accept خود Docker ارزیابی می‌شود).
در بسیاری از توزیع‌های مدرن، `iptables`/`ip6tables` از frontend به نام `iptables-nft` استفاده می‌کنند
و همچنان این قواعد را روی backend مربوط به nftables اعمال می‌کنند.

نمونه حداقلی فهرست مجاز (IPv4):
__OC_I18N_900008__
IPv6 جدول‌های جداگانه دارد. اگر
Docker IPv6 فعال است، یک سیاست متناظر در `/etc/ufw/after6.rules` اضافه کنید.

از hardcode کردن نام interfaceهایی مانند `eth0` در قطعه‌های مستندات پرهیز کنید. نام interfaceها
در imageهای مختلف VPS متفاوت است (`ens3`، `enp*` و غیره) و ناهماهنگی‌ها می‌توانند به‌طور تصادفی
باعث شوند قاعده deny شما اعمال نشود.

اعتبارسنجی سریع پس از reload:
__OC_I18N_900009__
پورت‌های خارجی مورد انتظار باید فقط همان‌هایی باشند که عمداً در معرض دسترس قرار می‌دهید (برای بیشتر
راه‌اندازی‌ها: SSH + پورت‌های reverse proxy شما).

### کشف mDNS/Bonjour

Gateway حضور خود را برای کشف دستگاه‌های محلی از طریق mDNS (`_openclaw-gw._tcp` روی پورت 5353) اعلام می‌کند. در حالت کامل، این شامل رکوردهای TXT می‌شود که ممکن است جزئیات عملیاتی را افشا کنند:

- `cliPath`: مسیر کامل سامانهٔ فایل به فایل باینری CLI (نام کاربری و محل نصب را آشکار می‌کند)
- `sshPort`: در دسترس بودن SSH روی میزبان را اعلام می‌کند
- `displayName`, `lanHost`: اطلاعات نام میزبان

**ملاحظهٔ امنیت عملیاتی:** پخش جزئیات زیرساخت، شناسایی را برای هر کسی در شبکهٔ محلی آسان‌تر می‌کند. حتی اطلاعات «بی‌ضرر» مانند مسیرهای سامانهٔ فایل و در دسترس بودن SSH به مهاجمان کمک می‌کند محیط شما را نقشه‌برداری کنند.

**توصیه‌ها:**

1. **حالت حداقلی** (پیش‌فرض، توصیه‌شده برای Gatewayهای در معرض دسترسی): فیلدهای حساس را از پخش‌های mDNS حذف کنید:
__OC_I18N_900010__
2. اگر به کشف دستگاه‌های محلی نیاز ندارید، **کاملاً غیرفعال کنید**:
__OC_I18N_900011__
3. **حالت کامل** (با انتخاب صریح): `cliPath` + `sshPort` را در رکوردهای TXT وارد کنید:
__OC_I18N_900012__
4. **متغیر محیطی** (جایگزین): برای غیرفعال کردن mDNS بدون تغییرات پیکربندی، `OPENCLAW_DISABLE_BONJOUR=1` را تنظیم کنید.

در حالت حداقلی، Gateway همچنان به اندازهٔ لازم برای کشف دستگاه (`role`, `gatewayPort`, `transport`) پخش می‌کند، اما `cliPath` و `sshPort` را حذف می‌کند. برنامه‌هایی که به اطلاعات مسیر CLI نیاز دارند می‌توانند به‌جای آن، از طریق اتصال WebSocket احراز هویت‌شده آن را دریافت کنند.

### قفل‌کردن WebSocket Gateway (احراز هویت محلی)

احراز هویت Gateway به‌صورت **پیش‌فرض الزامی است**. اگر هیچ مسیر معتبر احراز هویت gateway پیکربندی نشده باشد،
Gateway اتصال‌های WebSocket را رد می‌کند (fail‑closed).

فرایند راه‌اندازی به‌صورت پیش‌فرض یک توکن تولید می‌کند (حتی برای loopback)، بنابراین
کلاینت‌های محلی باید احراز هویت شوند.

یک توکن تنظیم کنید تا **همهٔ** کلاینت‌های WS ملزم به احراز هویت باشند:
__OC_I18N_900013__
Doctor می‌تواند یکی برای شما تولید کند: `openclaw doctor --generate-gateway-token`.

<Note>
`gateway.remote.token` و `gateway.remote.password` منابع اعتبارنامهٔ کلاینت هستند. آن‌ها به‌تنهایی دسترسی WS محلی را محافظت نمی‌کنند. مسیرهای فراخوانی محلی فقط وقتی `gateway.auth.*` تنظیم نشده باشد می‌توانند از `gateway.remote.*` به‌عنوان fallback استفاده کنند. اگر `gateway.auth.token` یا `gateway.auth.password` به‌طور صریح از طریق SecretRef پیکربندی شده و حل‌نشده باشد، حل‌کردن fail closed می‌شود (بدون اینکه fallback راه دور آن را بپوشاند).
</Note>
اختیاری: هنگام استفاده از `wss://`، TLS راه دور را با `gateway.remote.tlsFingerprint` پین کنید.
متن سادهٔ `ws://` به‌صورت پیش‌فرض فقط برای loopback است. برای مسیرهای مورد اعتماد شبکهٔ خصوصی،
`OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` را روی فرایند کلاینت به‌عنوان
break-glass تنظیم کنید. این عمداً فقط محیط فرایند است، نه یک کلید پیکربندی
`openclaw.json`.
جفت‌سازی موبایل و مسیرهای Gateway دستی یا اسکن‌شدهٔ Android سخت‌گیرانه‌تر هستند:
cleartext برای loopback پذیرفته می‌شود، اما private-LAN، link-local، `.local` و
نام‌های میزبان بدون نقطه باید از TLS استفاده کنند، مگر اینکه صریحاً مسیر cleartext
شبکهٔ خصوصی مورد اعتماد را انتخاب کنید.

جفت‌سازی دستگاه محلی:

- جفت‌سازی دستگاه برای اتصال‌های مستقیم local loopback به‌صورت خودکار تأیید می‌شود تا
  کلاینت‌های همان میزبان روان کار کنند.
- OpenClaw همچنین یک مسیر باریک self-connect محلی backend/container برای
  جریان‌های کمکی مورد اعتماد با راز مشترک دارد.
- اتصال‌های tailnet و LAN، از جمله bindهای tailnet همان میزبان، برای جفت‌سازی
  راه دور محسوب می‌شوند و همچنان به تأیید نیاز دارند.
- شواهد forwarded-header روی یک درخواست loopback، محلی بودن loopback را
  نامعتبر می‌کند. تأیید خودکار ارتقای فراداده دامنه‌ای باریک دارد. برای هر دو قاعده، به
  [جفت‌سازی Gateway](/gateway/pairing) مراجعه کنید.

حالت‌های احراز هویت:

- `gateway.auth.mode: "token"`: توکن bearer مشترک (برای بیشتر راه‌اندازی‌ها توصیه می‌شود).
- `gateway.auth.mode: "password"`: احراز هویت با گذرواژه (ترجیحاً از طریق env تنظیم شود: `OPENCLAW_GATEWAY_PASSWORD`).
- `gateway.auth.mode: "trusted-proxy"`: اعتماد به یک پراکسی معکوس آگاه از هویت برای احراز هویت کاربران و ارسال هویت از طریق headerها (به [احراز هویت پراکسی مورد اعتماد](/gateway/trusted-proxy-auth) مراجعه کنید).

چک‌لیست چرخش (توکن/گذرواژه):

1. یک secret جدید تولید/تنظیم کنید (`gateway.auth.token` یا `OPENCLAW_GATEWAY_PASSWORD`).
2. Gateway را restart کنید (یا اگر برنامهٔ macOS آن را نظارت می‌کند، برنامهٔ macOS را restart کنید).
3. هر کلاینت راه دور را به‌روزرسانی کنید (`gateway.remote.token` / `.password` روی ماشین‌هایی که با Gateway تماس می‌گیرند).
4. تأیید کنید که دیگر نمی‌توانید با اعتبارنامه‌های قدیمی وصل شوید.

### headerهای هویت Tailscale Serve

وقتی `gateway.auth.allowTailscale` برابر `true` باشد (پیش‌فرض برای Serve)، OpenClaw
headerهای هویت Tailscale Serve (`tailscale-user-login`) را برای احراز هویت Control
UI/WebSocket می‌پذیرد. OpenClaw هویت را با resolve کردن نشانی
`x-forwarded-for` از طریق daemon محلی Tailscale (`tailscale whois`)
و تطبیق آن با header تأیید می‌کند. این فقط برای درخواست‌هایی فعال می‌شود که به loopback برسند
و شامل `x-forwarded-for`، `x-forwarded-proto` و `x-forwarded-host` باشند، همان‌طور که
Tailscale تزریق می‌کند.
برای این مسیر بررسی هویت async، تلاش‌های ناموفق برای همان `{scope, ip}`
قبل از اینکه limiter شکست را ثبت کند، به‌صورت سریالی انجام می‌شوند. بنابراین retryهای بد هم‌زمان
از یک کلاینت Serve می‌توانند تلاش دوم را فوراً قفل کنند
به‌جای اینکه مثل دو عدم‌تطابق ساده با هم race کنند.
endpointهای HTTP API (برای مثال `/v1/*`، `/tools/invoke` و `/api/channels/*`)
از احراز هویت identity-header Tailscale استفاده نمی‌کنند. آن‌ها همچنان از حالت
احراز هویت HTTP پیکربندی‌شدهٔ gateway پیروی می‌کنند.

نکتهٔ مهم دربارهٔ مرز:

- احراز هویت bearer در HTTP Gateway عملاً دسترسی اپراتوری همه‌یا‌هیچ است.
- اعتبارنامه‌هایی را که می‌توانند `/v1/chat/completions`، `/v1/responses` یا `/api/channels/*` را فراخوانی کنند، به‌عنوان secretهای اپراتوری با دسترسی کامل برای آن gateway در نظر بگیرید.
- روی سطح HTTP سازگار با OpenAI، احراز هویت bearer با secret مشترک، دامنه‌های کامل پیش‌فرض اپراتور (`operator.admin`, `operator.approvals`, `operator.pairing`, `operator.read`, `operator.talk.secrets`, `operator.write`) و معنای مالک برای نوبت‌های agent را بازمی‌گرداند؛ مقدارهای محدودتر `x-openclaw-scopes` آن مسیر secret مشترک را کاهش نمی‌دهند.
- معنای scope در هر درخواست روی HTTP فقط وقتی اعمال می‌شود که درخواست از یک حالت دارای هویت مثل احراز هویت پراکسی مورد اعتماد یا `gateway.auth.mode="none"` روی یک ingress خصوصی بیاید.
- در این حالت‌های دارای هویت، حذف `x-openclaw-scopes` به مجموعهٔ scope پیش‌فرض عادی اپراتور fallback می‌کند؛ وقتی مجموعهٔ scope محدودتری می‌خواهید، header را صریحاً ارسال کنید.
- `/tools/invoke` از همان قاعدهٔ secret مشترک پیروی می‌کند: احراز هویت bearer با token/password در آنجا هم دسترسی کامل اپراتور محسوب می‌شود، در حالی‌که حالت‌های دارای هویت همچنان scopeهای اعلام‌شده را رعایت می‌کنند.
- این اعتبارنامه‌ها را با فراخوان‌های غیرقابل‌اعتماد به اشتراک نگذارید؛ برای هر مرز اعتماد، Gateway جداگانه را ترجیح دهید.

**فرض اعتماد:** احراز هویت Serve بدون توکن فرض می‌کند میزبان gateway مورد اعتماد است.
این را محافظت در برابر فرایندهای خصمانهٔ همان میزبان تلقی نکنید. اگر ممکن است کد محلی
غیرقابل‌اعتماد روی میزبان gateway اجرا شود، `gateway.auth.allowTailscale`
را غیرفعال کنید و احراز هویت صریح با secret مشترک را با `gateway.auth.mode: "token"` یا
`"password"` الزامی کنید.

**قاعدهٔ امنیتی:** این headerها را از پراکسی معکوس خودتان forward نکنید. اگر
TLS را terminate می‌کنید یا جلوی gateway پراکسی می‌گذارید،
`gateway.auth.allowTailscale` را غیرفعال کنید و به‌جای آن از احراز هویت با secret مشترک (`gateway.auth.mode:
"token"` یا `"password"`) یا [احراز هویت پراکسی مورد اعتماد](/gateway/trusted-proxy-auth)
استفاده کنید.

پراکسی‌های مورد اعتماد:

- اگر TLS را جلوی Gateway terminate می‌کنید، `gateway.trustedProxies` را روی IPهای پراکسی خود تنظیم کنید.
- OpenClaw به `x-forwarded-for` (یا `x-real-ip`) از آن IPها اعتماد می‌کند تا IP کلاینت را برای بررسی‌های جفت‌سازی محلی و احراز هویت HTTP/بررسی‌های محلی تعیین کند.
- مطمئن شوید پراکسی شما `x-forwarded-for` را **بازنویسی** می‌کند و دسترسی مستقیم به پورت Gateway را مسدود می‌کند.

به [Tailscale](/gateway/tailscale) و [نمای کلی وب](/web) مراجعه کنید.

### کنترل مرورگر از طریق میزبان node (توصیه‌شده)

اگر Gateway شما راه دور است اما مرورگر روی ماشین دیگری اجرا می‌شود، یک **میزبان node**
روی ماشین مرورگر اجرا کنید و اجازه دهید Gateway عملیات مرورگر را پراکسی کند (به [ابزار مرورگر](/tools/browser) مراجعه کنید).
جفت‌سازی node را مانند دسترسی ادمین در نظر بگیرید.

الگوی پیشنهادی:

- Gateway و میزبان node را روی یک tailnet یکسان نگه دارید (Tailscale).
- node را آگاهانه جفت کنید؛ اگر به مسیریابی پراکسی مرورگر نیاز ندارید، آن را غیرفعال کنید.

پرهیز کنید از:

- در معرض قرار دادن پورت‌های relay/control روی LAN یا اینترنت عمومی.
- Tailscale Funnel برای endpointهای کنترل مرورگر (در معرض‌گذاری عمومی).

### secretهای روی دیسک

فرض کنید هر چیزی زیر `~/.openclaw/` (یا `$OPENCLAW_STATE_DIR/`) ممکن است شامل secret یا دادهٔ خصوصی باشد:

- `openclaw.json`: پیکربندی ممکن است شامل tokenها (gateway، gateway راه دور)، تنظیمات provider و allowlistها باشد.
- `credentials/**`: اعتبارنامه‌های channel (مثال: اعتبارنامه‌های WhatsApp)، allowlistهای جفت‌سازی، importهای OAuth قدیمی.
- `agents/<agentId>/agent/auth-profiles.json`: کلیدهای API، پروفایل‌های token، tokenهای OAuth و `keyRef`/`tokenRef` اختیاری.
- `agents/<agentId>/agent/codex-home/**`: حساب app-server مخصوص هر agent در Codex، پیکربندی، Skills، plugins، وضعیت thread بومی و عیب‌یابی‌ها.
- `secrets.json` (اختیاری): payload secret مبتنی بر فایل که providerهای SecretRef از نوع `file` استفاده می‌کنند (`secrets.providers`).
- `agents/<agentId>/agent/auth.json`: فایل سازگاری legacy. ورودی‌های static `api_key` هنگام کشف پاک‌سازی می‌شوند.
- `agents/<agentId>/sessions/**`: transcriptهای session (`*.jsonl`) + فرادادهٔ مسیریابی (`sessions.json`) که می‌توانند پیام‌های خصوصی و خروجی ابزار را شامل شوند.
- بسته‌های Plugin bundled: pluginهای نصب‌شده (به‌علاوهٔ `node_modules/` آن‌ها).
- `sandboxes/**`: workspaceهای sandbox ابزار؛ می‌توانند کپی‌هایی از فایل‌هایی را که داخل sandbox می‌خوانید/می‌نویسید انباشته کنند.

نکته‌های سخت‌سازی:

- مجوزها را محدود نگه دارید (`700` روی dirها، `600` روی فایل‌ها).
- روی میزبان gateway از رمزگذاری کامل دیسک استفاده کنید.
- اگر میزبان مشترک است، یک حساب کاربری اختصاصی OS برای Gateway را ترجیح دهید.

### فایل‌های `.env` در workspace

OpenClaw فایل‌های `.env` محلی workspace را برای agentها و ابزارها بارگذاری می‌کند، اما هرگز اجازه نمی‌دهد این فایل‌ها کنترل‌های runtime مربوط به gateway را بی‌صدا override کنند.

- هر کلیدی که با `OPENCLAW_*` شروع شود از فایل‌های `.env` غیرقابل‌اعتماد workspace مسدود می‌شود.
- تنظیمات endpoint مربوط به channel برای Matrix، Mattermost، IRC و Synology Chat نیز از override شدن توسط `.env` در workspace مسدود می‌شوند، بنابراین workspaceهای clone‌شده نمی‌توانند ترافیک connector bundled را از طریق پیکربندی endpoint محلی redirect کنند. کلیدهای env مربوط به endpoint (مانند `MATRIX_HOMESERVER`، `MATTERMOST_URL`، `IRC_HOST`، `SYNOLOGY_CHAT_INCOMING_URL`) باید از محیط فرایند gateway یا `env.shellEnv` بیایند، نه از `.env` بارگذاری‌شده از workspace.
- این مسدودسازی fail-closed است: یک متغیر کنترل runtime جدید که در نسخهٔ آینده اضافه شود نمی‌تواند از یک `.env` commit‌شده یا تأمین‌شده توسط مهاجم به ارث برسد؛ کلید نادیده گرفته می‌شود و gateway مقدار خودش را نگه می‌دارد.
- متغیرهای محیطی مورد اعتماد فرایند/OS (shell خود gateway، واحد launchd/systemd، app bundle) همچنان اعمال می‌شوند — این فقط بارگذاری فایل `.env` را محدود می‌کند.

چرا: فایل‌های `.env` در workspace اغلب کنار کد agent قرار می‌گیرند، تصادفاً commit می‌شوند، یا توسط ابزارها نوشته می‌شوند. مسدود کردن کل پیشوند `OPENCLAW_*` یعنی اضافه‌کردن یک پرچم جدید `OPENCLAW_*` در آینده هرگز نمی‌تواند به ارث‌بری بی‌صدای وضعیت workspace تبدیل شود.

### logها و transcriptها (redaction و نگهداری)

logها و transcriptها حتی وقتی کنترل‌های دسترسی درست هستند می‌توانند اطلاعات حساس را نشت دهند:

- logهای Gateway ممکن است شامل خلاصه‌های ابزار، خطاها و URLها باشند.
- transcriptهای session می‌توانند شامل secretهای pasted، محتوای فایل‌ها، خروجی command و linkها باشند.

توصیه‌ها:

- redaction مربوط به log و transcript را روشن نگه دارید (`logging.redactSensitive: "tools"`؛ پیش‌فرض).
- الگوهای سفارشی را برای محیط خود از طریق `logging.redactPatterns` اضافه کنید (tokenها، hostnameها، URLهای داخلی).
- هنگام اشتراک‌گذاری عیب‌یابی، به‌جای log خام، `openclaw status --all` را ترجیح دهید (قابل paste، secretها redacted).
- اگر به نگهداری بلندمدت نیاز ندارید، transcriptهای session و فایل‌های log قدیمی را پاک کنید.

### حالت فقط‌خواندنی (از طریق sandbox و ابزارها)

می‌توانید با ترکیب موارد زیر یک پروفایل فقط‌خواندنی بسازید:

- `agents.defaults.sandbox.workspaceAccess: "ro"` (یا `"none"` برای نداشتن دسترسی به workspace)
- فهرست‌های مجاز/ممنوع ابزارها که `write`، `edit`، `apply_patch`، `exec`، `process` و غیره را مسدود می‌کنند.

گزینه‌های سخت‌سازی بیشتر:

- `tools.exec.applyPatch.workspaceOnly: true` (پیش‌فرض): تضمین می‌کند `apply_patch` حتی وقتی sandboxing خاموش است نتواند بیرون از دایرکتوری workspace بنویسد/حذف کند. فقط وقتی آن را روی `false` بگذارید که عمدا می‌خواهید `apply_patch` فایل‌های بیرون از workspace را لمس کند.
- `tools.fs.workspaceOnly: true` (اختیاری): مسیرهای `read`/`write`/`edit`/`apply_patch` و مسیرهای بارگذاری خودکار تصویر پرامپت بومی را به دایرکتوری workspace محدود می‌کند (اگر امروز مسیرهای مطلق را مجاز می‌دانید و یک guardrail واحد می‌خواهید، مفید است).
- ریشه‌های filesystem را محدود نگه دارید: برای workspaceهای عامل/sandbox workspaceها از ریشه‌های گسترده مثل دایرکتوری خانه خودتان پرهیز کنید. ریشه‌های گسترده می‌توانند فایل‌های محلی حساس (برای مثال state/config زیر `~/.openclaw`) را در معرض ابزارهای filesystem قرار دهند.

</Note>

همچنین دسترسی عامل به workspace داخل sandbox را در نظر بگیرید:

- `agents.defaults.sandbox.workspaceAccess: "none"` (پیش‌فرض) workspace عامل را خارج از دسترس نگه می‌دارد؛ ابزارها روی یک sandbox workspace زیر `~/.openclaw/sandboxes` اجرا می‌شوند
- `agents.defaults.sandbox.workspaceAccess: "ro"` workspace عامل را به‌صورت فقط‌خواندنی در `/agent` mount می‌کند (`write`/`edit`/`apply_patch` را غیرفعال می‌کند)
- `agents.defaults.sandbox.workspaceAccess: "rw"` workspace عامل را به‌صورت خواندن/نوشتن در `/workspace` mount می‌کند
- `sandbox.docker.binds` اضافی بر اساس مسیرهای منبع normalized و canonicalized اعتبارسنجی می‌شوند. ترفندهای symlink والد و نام‌های مستعار canonical خانه همچنان به‌صورت fail closed عمل می‌کنند اگر به ریشه‌های مسدودشده‌ای مثل `/etc`، `/var/run`، یا دایرکتوری‌های credential زیر خانه OS resolve شوند.

<Warning>
`tools.elevated` راه فرار baseline سراسری است که exec را بیرون از sandbox اجرا می‌کند. میزبان موثر به‌صورت پیش‌فرض `gateway` است، یا وقتی target exec روی `node` پیکربندی شده باشد `node` است. `tools.elevated.allowFrom` را محدود نگه دارید و آن را برای غریبه‌ها فعال نکنید. می‌توانید elevated را برای هر عامل از طریق `agents.list[].tools.elevated` بیشتر محدود کنید. [حالت Elevated](/tools/elevated) را ببینید.
</Warning>

### guardrail واگذاری به زیرعامل

اگر ابزارهای session را مجاز می‌کنید، اجرای زیرعامل‌های واگذارشده را به‌عنوان یک تصمیم مرزی دیگر در نظر بگیرید:

- `sessions_spawn` را ممنوع کنید مگر اینکه عامل واقعا به واگذاری نیاز داشته باشد.
- `agents.defaults.subagents.allowAgents` و هر override مربوط به هر عامل در `agents.list[].subagents.allowAgents` را محدود به عامل‌های هدف شناخته‌شده و امن نگه دارید.
- برای هر workflow که باید sandboxed باقی بماند، `sessions_spawn` را با `sandbox: "require"` فراخوانی کنید (پیش‌فرض `inherit` است).
- `sandbox: "require"` وقتی runtime فرزند هدف sandboxed نباشد سریع شکست می‌خورد.

## خطرهای کنترل مرورگر

فعال‌کردن کنترل مرورگر به مدل امکان می‌دهد یک مرورگر واقعی را هدایت کند.
اگر آن پروفایل مرورگر از قبل sessionهای واردشده داشته باشد، مدل می‌تواند
به آن حساب‌ها و داده‌ها دسترسی پیدا کند. پروفایل‌های مرورگر را **وضعیت حساس** در نظر بگیرید:

- یک پروفایل اختصاصی برای عامل را ترجیح دهید (پروفایل پیش‌فرض `openclaw`).
- از اشاره‌دادن عامل به پروفایل شخصی روزمره خودتان پرهیز کنید.
- کنترل مرورگر میزبان را برای عامل‌های sandboxed غیرفعال نگه دارید مگر اینکه به آن‌ها اعتماد داشته باشید.
- API کنترل مرورگر loopback مستقل فقط auth با shared-secret را رعایت می‌کند
  (gateway token bearer auth یا gateway password). این API
  trusted-proxy یا headerهای هویت Tailscale Serve را مصرف نمی‌کند.
- دانلودهای مرورگر را ورودی غیرقابل اعتماد در نظر بگیرید؛ یک دایرکتوری دانلود ایزوله را ترجیح دهید.
- در صورت امکان، browser sync/password managers را در پروفایل عامل غیرفعال کنید (blast radius را کاهش می‌دهد).
- برای gatewayهای remote، فرض کنید «کنترل مرورگر» معادل «دسترسی اپراتور» به هر چیزی است که آن پروفایل می‌تواند به آن برسد.
- Gateway و میزبان‌های Node را فقط tailnet نگه دارید؛ از در معرض قراردادن پورت‌های کنترل مرورگر در LAN یا اینترنت عمومی پرهیز کنید.
- وقتی به مسیریابی proxy مرورگر نیاز ندارید، آن را غیرفعال کنید (`gateway.nodes.browser.mode="off"`).
- حالت existing-session در Chrome MCP **امن‌تر** نیست؛ می‌تواند در هر چیزی که پروفایل Chrome آن میزبان به آن دسترسی دارد، مثل شما عمل کند.

### سیاست SSRF مرورگر (به‌صورت پیش‌فرض سخت‌گیرانه)

سیاست ناوبری مرورگر OpenClaw به‌صورت پیش‌فرض سخت‌گیرانه است: مقصدهای خصوصی/داخلی مسدود می‌مانند مگر اینکه صراحتا opt in کنید.

- پیش‌فرض: `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` unset است، بنابراین ناوبری مرورگر مقصدهای خصوصی/داخلی/special-use را مسدود نگه می‌دارد.
- alias قدیمی: `browser.ssrfPolicy.allowPrivateNetwork` همچنان برای سازگاری پذیرفته می‌شود.
- حالت opt-in: برای مجازکردن مقصدهای خصوصی/داخلی/special-use، `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true` را تنظیم کنید.
- در حالت سخت‌گیرانه، برای استثناهای صریح از `hostnameAllowlist` (patternهایی مثل `*.example.com`) و `allowedHostnames` (استثناهای دقیق host، شامل نام‌های مسدودشده مثل `localhost`) استفاده کنید.
- ناوبری پیش از request بررسی می‌شود و پس از ناوبری، برای کاهش pivotهای مبتنی بر redirect، روی URL نهایی `http(s)` هم به‌صورت best-effort دوباره بررسی می‌شود.

## پاسخ به رخداد

اگر AI شما کار بدی انجام داد:

### مهار

1. **متوقفش کنید:** اپ macOS را متوقف کنید (اگر Gateway را supervise می‌کند) یا process مربوط به `openclaw gateway` خود را terminate کنید.
2. **در معرض‌بودن را ببندید:** تا زمانی که بفهمید چه اتفاقی افتاده است، `gateway.bind: "loopback"` را تنظیم کنید (یا Tailscale Funnel/Serve را غیرفعال کنید).
3. **دسترسی را freeze کنید:** DMها/گروه‌های پرخطر را به `dmPolicy: "disabled"` تغییر دهید / mentionها را الزامی کنید، و اگر ورودی‌های allow-all با `"*"` داشتید آن‌ها را حذف کنید.

### چرخش (اگر secretها leak شده‌اند، compromise را فرض کنید)

1. auth مربوط به Gateway (`gateway.auth.token` / `OPENCLAW_GATEWAY_PASSWORD`) را rotate کنید و restart کنید.
2. secretهای client remote (`gateway.remote.token` / `.password`) را روی هر ماشینی که می‌تواند Gateway را فراخوانی کند rotate کنید.
3. credentialهای provider/API را rotate کنید (credentialهای WhatsApp، tokenهای Slack/Discord، کلیدهای model/API در `auth-profiles.json`، و مقادیر payload مربوط به secretهای encrypted هنگام استفاده).

### Audit

1. لاگ‌های Gateway را بررسی کنید: `/tmp/openclaw/openclaw-YYYY-MM-DD.log` (یا `logging.file`).
2. transcript(های) مرتبط را مرور کنید: `~/.openclaw/agents/<agentId>/sessions/*.jsonl`.
3. تغییرات اخیر config را مرور کنید (هر چیزی که می‌توانسته دسترسی را گسترش داده باشد: `gateway.bind`، `gateway.auth`، سیاست‌های dm/group، `tools.elevated`، تغییرات Plugin).
4. دوباره `openclaw security audit --deep` را اجرا کنید و تایید کنید یافته‌های بحرانی حل شده‌اند.

### گردآوری برای گزارش

- timestamp، OS میزبان gateway + نسخه OpenClaw
- transcript(های) session + یک tail کوتاه از لاگ (پس از redact کردن)
- مهاجم چه چیزی فرستاد + عامل چه کاری انجام داد
- آیا Gateway فراتر از loopback در معرض بوده است یا نه (LAN/Tailscale Funnel/Serve)

## Secret scanning با detect-secrets

CI، hook مربوط به `detect-secrets` در pre-commit را در job مربوط به `secrets` اجرا می‌کند.
Pushها به `main` همیشه یک scan روی همه فایل‌ها اجرا می‌کنند. Pull requestها وقتی base commit در دسترس باشد از fast path فایل‌های تغییریافته استفاده می‌کنند و در غیر این صورت به scan همه فایل‌ها fallback می‌کنند. اگر شکست بخورد، candidateهای جدیدی وجود دارند که هنوز در baseline نیستند.
