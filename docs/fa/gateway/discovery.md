# کشف و انتقال‌ها

OpenClaw دو مسئلهٔ متمایز دارد که در ظاهر مشابه به نظر می‌رسند:

1. **کنترل از راه دور اپراتور**: برنامهٔ نوار منوی macOS که Gateway در حال اجرا در جای دیگری را کنترل می‌کند.
2. **جفت‌سازی Node**: iOS/Android و Nodeهای آینده که یک Gateway را پیدا می‌کنند و به‌صورت امن جفت می‌شوند.

هدف طراحی این است که همهٔ کشف/تبلیغ شبکه در **Node Gateway** (`openclaw gateway`) نگه داشته شود و کلاینت‌ها (برنامهٔ Mac، iOS) مصرف‌کننده باشند.

## اصطلاحات

- **Gateway**: یک فرایند Gateway تکی و بلندمدت که مالک وضعیت است (نشست‌ها، جفت‌سازی، رجیستری Node) و کانال‌ها را اجرا می‌کند. بیشتر راه‌اندازی‌ها از یک نمونه برای هر میزبان استفاده می‌کنند؛ راه‌اندازی‌های چند-Gateway ایزوله نیز ممکن‌اند.
- **Gateway WS (صفحهٔ کنترل)**: نقطهٔ پایانی WebSocket روی `127.0.0.1:18789` به‌صورت پیش‌فرض؛ می‌تواند از طریق `gateway.bind` به LAN/tailnet متصل شود.
- **انتقال WS مستقیم**: یک نقطهٔ پایانی Gateway WS رو به LAN/tailnet (بدون SSH).
- **انتقال SSH (جایگزین)**: کنترل از راه دور با فوروارد کردن `127.0.0.1:18789` از طریق SSH.
- **پل TCP قدیمی (حذف‌شده)**: انتقال قدیمی‌تر Node (ببینید
  [پروتکل پل](/fa/gateway/bridge-protocol))؛ دیگر برای
  کشف تبلیغ نمی‌شود و دیگر بخشی از بیلدهای فعلی نیست.

جزئیات پروتکل:

- [پروتکل Gateway](/fa/gateway/protocol)
- [پروتکل پل (قدیمی)](/fa/gateway/bridge-protocol)

## چرا هم «مستقیم» و هم SSH را نگه می‌داریم

- **WS مستقیم** بهترین تجربهٔ کاربری در همان شبکه و داخل یک tailnet است:
  - کشف خودکار روی LAN از طریق Bonjour
  - توکن‌های جفت‌سازی + ACLهایی که مالک آن‌ها Gateway است
  - دسترسی shell لازم نیست؛ سطح پروتکل می‌تواند محدود و قابل ممیزی بماند
- **SSH** همچنان جایگزین همگانی است:
  - هر جا دسترسی SSH داشته باشید کار می‌کند (حتی در شبکه‌های نامرتبط)
  - از مشکلات multicast/mDNS عبور می‌کند
  - به پورت‌های ورودی جدیدی غیر از SSH نیاز ندارد

## ورودی‌های کشف (کلاینت‌ها چگونه می‌فهمند Gateway کجاست)

### 1) کشف Bonjour / DNS-SD

Bonjour چندپخشی بهترین تلاش است و از شبکه‌ها عبور نمی‌کند. OpenClaw همچنین می‌تواند همان نشان Gateway را از طریق یک دامنهٔ DNS-SD گستردهٔ پیکربندی‌شده مرور کند، بنابراین کشف می‌تواند پوشش دهد:

- `local.` روی همان LAN
- یک دامنهٔ DNS-SD تک‌پخشی پیکربندی‌شده برای کشف بین‌شبکه‌ای

جهت هدف:

- **Gateway** نقطهٔ پایانی WS خود را از طریق Bonjour تبلیغ می‌کند.
- کلاینت‌ها مرور می‌کنند و فهرست «انتخاب یک Gateway» را نشان می‌دهند، سپس نقطهٔ پایانی انتخاب‌شده را ذخیره می‌کنند.

عیب‌یابی و جزئیات نشان: [Bonjour](/fa/gateway/bonjour).

#### جزئیات نشان سرویس

- نوع‌های سرویس:
  - `_openclaw-gw._tcp` (نشان انتقال Gateway)
- کلیدهای TXT (غیرمحرمانه):
  - `role=gateway`
  - `transport=gateway`
  - `displayName=<friendly name>` (نام نمایشی پیکربندی‌شده توسط اپراتور)
  - `lanHost=<hostname>.local`
  - `gatewayPort=18789` (Gateway WS + HTTP)
  - `gatewayTls=1` (فقط وقتی TLS فعال باشد)
  - `gatewayTlsSha256=<sha256>` (فقط وقتی TLS فعال باشد و اثر انگشت در دسترس باشد)
  - `canvasPort=<port>` (پورت میزبان canvas؛ در حال حاضر وقتی میزبان canvas فعال است همان `gatewayPort` است)
  - `tailnetDns=<magicdns>` (راهنمای اختیاری؛ وقتی Tailscale در دسترس باشد به‌صورت خودکار تشخیص داده می‌شود)
  - `sshPort=<port>` (فقط حالت کامل mDNS؛ DNS-SD گسترده ممکن است آن را حذف کند، که در این صورت پیش‌فرض‌های SSH روی `22` می‌مانند)
  - `cliPath=<path>` (فقط حالت کامل mDNS؛ DNS-SD گسترده همچنان آن را به‌عنوان راهنمای نصب از راه دور می‌نویسد)

نکات امنیتی:

- رکوردهای TXT در Bonjour/mDNS **احراز اصالت نشده‌اند**. کلاینت‌ها باید مقدارهای TXT را فقط به‌عنوان راهنماهای تجربهٔ کاربری در نظر بگیرند.
- مسیریابی (میزبان/پورت) باید **نقطهٔ پایانی سرویس حل‌شده** (SRV + A/AAAA) را به `lanHost`، `tailnetDns` یا `gatewayPort` ارائه‌شده توسط TXT ترجیح دهد.
- پین‌کردن TLS هرگز نباید اجازه دهد `gatewayTlsSha256` تبلیغ‌شده یک پین ذخیره‌شدهٔ قبلی را بازنویسی کند.
- Nodeهای iOS/Android باید پیش از ذخیرهٔ یک پین برای نخستین بار، هر زمان مسیر انتخاب‌شده امن/مبتنی بر TLS است، تأیید صریح «به این اثر انگشت اعتماد شود» را الزامی کنند (راستی‌آزمایی خارج از باند).

غیرفعال‌سازی/بازنویسی:

- `OPENCLAW_DISABLE_BONJOUR=1` تبلیغ را غیرفعال می‌کند.
- وقتی `OPENCLAW_DISABLE_BONJOUR` تنظیم نشده باشد، Bonjour روی میزبان‌های عادی تبلیغ می‌کند
  و داخل کانتینرهای تشخیص‌داده‌شده به‌صورت خودکار غیرفعال می‌شود. از `0` فقط روی میزبان، macvlan،
  یا شبکهٔ دیگری با قابلیت mDNS استفاده کنید؛ از `1` برای غیرفعال‌سازی اجباری استفاده کنید.
- `gateway.bind` در `~/.openclaw/openclaw.json` حالت اتصال Gateway را کنترل می‌کند.
- `OPENCLAW_SSH_PORT` پورت SSH تبلیغ‌شده را وقتی `sshPort` منتشر می‌شود بازنویسی می‌کند.
- `OPENCLAW_TAILNET_DNS` یک راهنمای `tailnetDns` منتشر می‌کند (MagicDNS).
- `OPENCLAW_CLI_PATH` مسیر CLI تبلیغ‌شده را بازنویسی می‌کند.

### 2) Tailnet (بین‌شبکه‌ای)

برای راه‌اندازی‌هایی به سبک London/Vienna، Bonjour کمکی نمی‌کند. هدف «مستقیم» پیشنهادی این است:

- نام Tailscale MagicDNS (ترجیحی) یا یک IP پایدار tailnet.

اگر Gateway بتواند تشخیص دهد که تحت Tailscale اجرا می‌شود، `tailnetDns` را به‌عنوان راهنمای اختیاری برای کلاینت‌ها منتشر می‌کند (از جمله نشان‌های گسترده).

برنامهٔ macOS اکنون برای کشف Gateway نام‌های MagicDNS را به IPهای خام Tailscale ترجیح می‌دهد. این کار قابلیت اطمینان را وقتی IPهای tailnet تغییر می‌کنند بهتر می‌کند (برای مثال پس از راه‌اندازی مجدد Node یا تخصیص دوبارهٔ CGNAT)، چون نام‌های MagicDNS به‌صورت خودکار به IP فعلی resolve می‌شوند.

برای جفت‌سازی Node موبایل، راهنماهای کشف امنیت انتقال را روی مسیرهای tailnet/عمومی سست نمی‌کنند:

- iOS/Android همچنان برای نخستین اتصال tailnet/عمومی به یک مسیر امن نیاز دارند (`wss://` یا Tailscale Serve/Funnel).
- یک IP خام tailnet کشف‌شده یک راهنمای مسیریابی است، نه مجوزی برای استفاده از `ws://` راه دور و متن ساده.
- اتصال مستقیم `ws://` در LAN خصوصی همچنان پشتیبانی می‌شود.
- اگر ساده‌ترین مسیر Tailscale را برای Nodeهای موبایل می‌خواهید، از Tailscale Serve استفاده کنید تا هم کشف و هم کد راه‌اندازی به همان نقطهٔ پایانی امن MagicDNS resolve شوند.

### 3) هدف دستی / SSH

وقتی مسیر مستقیمی وجود ندارد (یا مستقیم غیرفعال است)، کلاینت‌ها همیشه می‌توانند با فوروارد کردن پورت Gateway روی loopback از طریق SSH متصل شوند.

ببینید [دسترسی از راه دور](/fa/gateway/remote).

## انتخاب انتقال (سیاست کلاینت)

رفتار پیشنهادی کلاینت:

1. اگر یک نقطهٔ پایانی مستقیم جفت‌شده پیکربندی شده و در دسترس است، از آن استفاده کنید.
2. وگرنه، اگر کشف یک Gateway روی `local.` یا دامنهٔ گستردهٔ پیکربندی‌شده پیدا کند، انتخاب تک‌ضربه‌ای «استفاده از این Gateway» را ارائه دهید و آن را به‌عنوان نقطهٔ پایانی مستقیم ذخیره کنید.
3. وگرنه، اگر DNS/IP مربوط به tailnet پیکربندی شده است، اتصال مستقیم را امتحان کنید.
   برای Nodeهای موبایل روی مسیرهای tailnet/عمومی، مستقیم یعنی یک نقطهٔ پایانی امن، نه `ws://` راه دور و متن ساده.
4. وگرنه، به SSH برگردید.

## جفت‌سازی + احراز هویت (انتقال مستقیم)

Gateway منبع حقیقت برای پذیرش Node/کلاینت است.

- درخواست‌های جفت‌سازی در Gateway ایجاد/تأیید/رد می‌شوند (ببینید [جفت‌سازی Gateway](/fa/gateway/pairing)).
- Gateway این موارد را اعمال می‌کند:
  - احراز هویت (توکن / جفت‌کلید)
  - دامنه‌ها/ACLها (Gateway یک پراکسی خام برای هر متد نیست)
  - محدودیت‌های نرخ

## مسئولیت‌ها بر اساس مؤلفه

- **Gateway**: نشان‌های کشف را تبلیغ می‌کند، مالک تصمیم‌های جفت‌سازی است، و نقطهٔ پایانی WS را میزبانی می‌کند.
- **برنامهٔ macOS**: به شما کمک می‌کند یک Gateway انتخاب کنید، اعلان‌های جفت‌سازی را نشان می‌دهد، و فقط به‌عنوان جایگزین از SSH استفاده می‌کند.
- **Nodeهای iOS/Android**: Bonjour را به‌عنوان یک امکان کمکی مرور می‌کنند و به Gateway WS جفت‌شده متصل می‌شوند.
