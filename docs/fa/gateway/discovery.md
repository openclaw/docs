# کشف و ترابری‌ها

OpenClaw دو مسئله متمایز دارد که در ظاهر شبیه به هم به نظر می‌رسند:

1. **کنترل راه‌دور اپراتور**: برنامه نوار منوی macOS که یک Gateway در حال اجرا در جای دیگر را کنترل می‌کند.
2. **جفت‌سازی Node**: iOS/Android (و Nodeهای آینده) که یک Gateway را پیدا می‌کنند و به‌صورت امن جفت می‌شوند.

هدف طراحی این است که همه کشف/اعلان شبکه در **Node Gateway** (`openclaw gateway`) نگه داشته شود و کلاینت‌ها (برنامه Mac، iOS) مصرف‌کننده باشند.

## اصطلاحات

- **Gateway**: یک فرایند Gateway واحد و طولانی‌مدت که مالک وضعیت است (نشست‌ها، جفت‌سازی، رجیستری Node) و کانال‌ها را اجرا می‌کند. بیشتر راه‌اندازی‌ها از یکی برای هر میزبان استفاده می‌کنند؛ راه‌اندازی‌های چند-Gateway ایزوله هم ممکن است.
- **Gateway WS (صفحه کنترل)**: endpoint وب‌سوکت روی `127.0.0.1:18789` به‌صورت پیش‌فرض؛ می‌تواند از طریق `gateway.bind` به LAN/tailnet متصل شود.
- **ترابری Direct WS**: یک endpoint Gateway WS رو به LAN/tailnet (بدون SSH).
- **ترابری SSH (جایگزین)**: کنترل راه‌دور با فوروارد کردن `127.0.0.1:18789` روی SSH.
- **پل TCP قدیمی (حذف‌شده)**: ترابری قدیمی‌تر Node (ببینید
  [پروتکل پل](/fa/gateway/bridge-protocol))؛ دیگر برای
  کشف اعلان نمی‌شود و دیگر بخشی از buildهای فعلی نیست.

جزئیات پروتکل:

- [پروتکل Gateway](/fa/gateway/protocol)
- [پروتکل پل (قدیمی)](/fa/gateway/bridge-protocol)

## چرا هم «مستقیم» و هم SSH را نگه می‌داریم

- **Direct WS** بهترین UX روی همان شبکه و درون یک tailnet است:
  - کشف خودکار روی LAN از طریق Bonjour
  - توکن‌های جفت‌سازی + ACLها که مالکشان Gateway است
  - بدون نیاز به دسترسی shell؛ سطح پروتکل می‌تواند محدود و قابل ممیزی بماند
- **SSH** جایگزین عمومی باقی می‌ماند:
  - هر جا که دسترسی SSH داشته باشید کار می‌کند (حتی بین شبکه‌های نامرتبط)
  - از مشکلات multicast/mDNS جان سالم به در می‌برد
  - به پورت ورودی جدیدی جز SSH نیاز ندارد

## ورودی‌های کشف (کلاینت‌ها چگونه محل Gateway را یاد می‌گیرند)

### 1) کشف Bonjour / DNS-SD

Bonjour چندپخشی best-effort است و از شبکه‌ها عبور نمی‌کند. OpenClaw همچنین می‌تواند همان beacon Gateway را از طریق یک دامنه DNS-SD پهن‌ناحیه پیکربندی‌شده browse کند، بنابراین کشف می‌تواند این موارد را پوشش دهد:

- `local.` روی همان LAN
- یک دامنه DNS-SD unicast پیکربندی‌شده برای کشف میان‌شبکه‌ای

جهت هدف:

- **Gateway** وقتی Plugin همراه
  `bonjour` فعال باشد endpoint WS خود را از طریق Bonjour اعلان می‌کند. این Plugin روی میزبان‌های macOS به‌طور خودکار شروع می‌شود و در جاهای دیگر opt-in است.
- کلاینت‌ها browse می‌کنند و فهرست «یک Gateway انتخاب کنید» را نشان می‌دهند، سپس endpoint انتخاب‌شده را ذخیره می‌کنند.

جزئیات عیب‌یابی و beacon: [Bonjour](/fa/gateway/bonjour).

#### جزئیات beacon سرویس

- انواع سرویس:
  - `_openclaw-gw._tcp` (beacon ترابری Gateway)
- کلیدهای TXT (غیرمحرمانه):
  - `role=gateway`
  - `transport=gateway`
  - `displayName=<friendly name>` (نام نمایشی پیکربندی‌شده توسط اپراتور)
  - `lanHost=<hostname>.local`
  - `gatewayPort=18789` (Gateway WS + HTTP)
  - `gatewayTls=1` (فقط وقتی TLS فعال است)
  - `gatewayTlsSha256=<sha256>` (فقط وقتی TLS فعال است و fingerprint در دسترس است)
  - `canvasPort=<port>` (پورت میزبان canvas؛ در حال حاضر وقتی میزبان canvas فعال است، همان `gatewayPort` است)
  - `tailnetDns=<magicdns>` (راهنمای اختیاری؛ وقتی Tailscale در دسترس باشد خودکار شناسایی می‌شود)
  - `sshPort=<port>` (فقط حالت کامل mDNS؛ DNS-SD پهن‌ناحیه ممکن است آن را حذف کند، که در این صورت پیش‌فرض‌های SSH روی `22` می‌مانند)
  - `cliPath=<path>` (فقط حالت کامل mDNS؛ DNS-SD پهن‌ناحیه همچنان آن را به‌عنوان راهنمای نصب راه‌دور می‌نویسد)

نکات امنیتی:

- رکوردهای TXT در Bonjour/mDNS **احرازاصالت‌نشده** هستند. کلاینت‌ها باید مقادیر TXT را فقط راهنمای UX بدانند.
- مسیریابی (host/port) باید **endpoint سرویس resolve‌شده** (SRV + A/AAAA) را به `lanHost`، `tailnetDns` یا `gatewayPort` ارائه‌شده در TXT ترجیح دهد.
- pinning در TLS هرگز نباید اجازه دهد یک `gatewayTlsSha256` اعلان‌شده، pin ذخیره‌شده قبلی را override کند.
- Nodeهای iOS/Android باید پیش از ذخیره pin برای نخستین بار، هر زمان مسیر انتخاب‌شده امن/مبتنی بر TLS است، تأیید صریح «به این fingerprint اعتماد کن» را الزامی کنند (راستی‌آزمایی خارج از باند).

فعال‌سازی/غیرفعال‌سازی/override:

- `openclaw plugins enable bonjour` اعلان چندپخشی LAN را فعال می‌کند.
- `OPENCLAW_DISABLE_BONJOUR=1` اعلان را غیرفعال می‌کند.
- وقتی Plugin Bonjour فعال باشد و `OPENCLAW_DISABLE_BONJOUR` تنظیم نشده باشد،
  Bonjour روی میزبان‌های عادی اعلان می‌کند و داخل کانتینرهای شناسایی‌شده به‌طور خودکار غیرفعال می‌شود.
  راه‌اندازی Gateway در macOS با پیکربندی خالی، Plugin را به‌طور خودکار فعال می‌کند؛ استقرارهای Linux،
  Windows و کانتینری به فعال‌سازی صریح نیاز دارند.
  فقط روی host، macvlan یا شبکه دیگری که mDNS را پشتیبانی می‌کند از `0` استفاده کنید؛ برای
  غیرفعال‌سازی اجباری از `1` استفاده کنید.
- `gateway.bind` در `~/.openclaw/openclaw.json` حالت bind Gateway را کنترل می‌کند.
- `OPENCLAW_SSH_PORT` پورت SSH اعلان‌شده را وقتی `sshPort` منتشر می‌شود override می‌کند.
- `OPENCLAW_TAILNET_DNS` یک راهنمای `tailnetDns` منتشر می‌کند (MagicDNS).
- `OPENCLAW_CLI_PATH` مسیر CLI اعلان‌شده را override می‌کند.

### 2) Tailnet (میان‌شبکه‌ای)

برای راه‌اندازی‌هایی به سبک لندن/وین، Bonjour کمکی نمی‌کند. هدف «مستقیم» پیشنهادی این است:

- نام Tailscale MagicDNS (ترجیحی) یا یک IP پایدار tailnet.

اگر Gateway بتواند تشخیص دهد که زیر Tailscale اجرا می‌شود، `tailnetDns` را به‌عنوان راهنمای اختیاری برای کلاینت‌ها منتشر می‌کند (از جمله beaconهای پهن‌ناحیه).

برنامه macOS اکنون برای کشف Gateway نام‌های MagicDNS را به IPهای خام Tailscale ترجیح می‌دهد. این کار قابلیت اطمینان را وقتی IPهای tailnet تغییر می‌کنند بهبود می‌دهد (برای مثال پس از restart شدن Node یا تخصیص دوباره CGNAT)، چون نام‌های MagicDNS به‌طور خودکار به IP فعلی resolve می‌شوند.

برای جفت‌سازی Node موبایل، راهنماهای کشف امنیت ترابری را در مسیرهای tailnet/عمومی آسان‌تر نمی‌کنند:

- iOS/Android همچنان برای مسیر اتصال نخستین tailnet/عمومی به یک مسیر امن نیاز دارند (`wss://` یا Tailscale Serve/Funnel).
- یک IP خام tailnet کشف‌شده یک راهنمای مسیریابی است، نه مجوزی برای استفاده از `ws://` راه‌دور plaintext.
- اتصال مستقیم خصوصی LAN با `ws://` همچنان پشتیبانی می‌شود.
- اگر ساده‌ترین مسیر Tailscale را برای Nodeهای موبایل می‌خواهید، از Tailscale Serve استفاده کنید تا هم کشف و هم کد راه‌اندازی به همان endpoint امن MagicDNS resolve شوند.

### 3) هدف دستی / SSH

وقتی هیچ مسیر مستقیمی وجود ندارد (یا مسیر مستقیم غیرفعال است)، کلاینت‌ها همیشه می‌توانند با فوروارد کردن پورت Gateway روی loopback از طریق SSH متصل شوند.

ببینید [دسترسی راه‌دور](/fa/gateway/remote).

## انتخاب ترابری (سیاست کلاینت)

رفتار پیشنهادی کلاینت:

1. اگر یک endpoint مستقیم جفت‌شده پیکربندی شده و در دسترس است، از آن استفاده کن.
2. در غیر این صورت، اگر کشف یک Gateway را روی `local.` یا دامنه پهن‌ناحیه پیکربندی‌شده پیدا کرد، گزینه تک‌ضرب «از این Gateway استفاده کن» را ارائه کن و آن را به‌عنوان endpoint مستقیم ذخیره کن.
3. در غیر این صورت، اگر یک DNS/IP مربوط به tailnet پیکربندی شده است، مستقیم را امتحان کن.
   برای Nodeهای موبایل روی مسیرهای tailnet/عمومی، مستقیم یعنی یک endpoint امن، نه `ws://` راه‌دور plaintext.
4. در غیر این صورت، به SSH برگرد.

## جفت‌سازی + auth (ترابری مستقیم)

Gateway منبع حقیقت برای پذیرش Node/کلاینت است.

- درخواست‌های جفت‌سازی در Gateway ایجاد/تأیید/رد می‌شوند (ببینید [جفت‌سازی Gateway](/fa/gateway/pairing)).
- Gateway این موارد را اعمال می‌کند:
  - auth (توکن / keypair)
  - scopes/ACLها (Gateway یک پروکسی خام به همه methodها نیست)
  - rate limitها

## مسئولیت‌ها بر اساس مؤلفه

- **Gateway**: beaconهای کشف را اعلان می‌کند، مالک تصمیم‌های جفت‌سازی است، و endpoint WS را میزبانی می‌کند.
- **برنامه macOS**: کمک می‌کند یک Gateway انتخاب کنید، promptهای جفت‌سازی را نشان می‌دهد، و فقط به‌عنوان fallback از SSH استفاده می‌کند.
- **Nodeهای iOS/Android**: برای راحتی Bonjour را browse می‌کنند و به Gateway WS جفت‌شده متصل می‌شوند.
