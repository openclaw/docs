# ابزار Exec پس‌زمینه + ابزار Process

OpenClaw فرمان‌های شل را از طریق ابزار `exec` اجرا می‌کند و کارهای بلندمدت را در حافظه نگه می‌دارد. ابزار `process` این نشست‌های پس‌زمینه را مدیریت می‌کند.

## ابزار exec

پارامترهای کلیدی:

- `command` (ضروری)
- `yieldMs` (پیش‌فرض 10000): پس از این تأخیر به‌صورت خودکار به پس‌زمینه می‌رود
- `background` (بولی): فوراً در پس‌زمینه اجرا می‌شود
- `timeout` (ثانیه، پیش‌فرض `tools.exec.timeoutSec`): پس از این زمان فرایند را می‌کشد؛ فقط برای غیرفعال کردن مهلت زمانی فرایند exec برای همان فراخوانی، `timeout: 0` را تنظیم کنید
- `elevated` (بولی): اگر حالت elevated فعال/مجاز باشد، خارج از sandbox اجرا می‌شود (`gateway` به‌صورت پیش‌فرض، یا وقتی هدف exec برابر `node` باشد، `node`)
- به TTY واقعی نیاز دارید؟ `pty: true` را تنظیم کنید.
- `workdir`, `env`

رفتار:

- اجراهای پیش‌زمینه خروجی را مستقیماً برمی‌گردانند.
- وقتی به پس‌زمینه برود (صریح یا به‌دلیل timeout)، ابزار `status: "running"` + `sessionId` و یک دنباله کوتاه را برمی‌گرداند.
- اجراهای پس‌زمینه و `yieldMs` از `tools.exec.timeoutSec` ارث‌بری می‌کنند، مگر اینکه فراخوانی یک `timeout` صریح ارائه کند.
- خروجی تا زمانی که نشست poll یا پاک‌سازی شود در حافظه نگه داشته می‌شود.
- اگر ابزار `process` مجاز نباشد، `exec` به‌صورت همگام اجرا می‌شود و `yieldMs`/`background` را نادیده می‌گیرد.
- فرمان‌های exec ایجادشده برای قواعد شل/پروفایل آگاه از زمینه، `OPENCLAW_SHELL=exec` را دریافت می‌کنند.
- برای کار بلندمدتی که اکنون شروع می‌شود، آن را یک‌بار شروع کنید و وقتی فعال است و فرمان خروجی تولید می‌کند یا شکست می‌خورد، به بیدارسازی خودکار تکمیل تکیه کنید.
- اگر بیدارسازی خودکار تکمیل در دسترس نیست، یا برای فرمانی که بدون خروجی با موفقیت خارج شده به تأیید موفقیت بی‌صدا نیاز دارید، از `process` برای تأیید تکمیل استفاده کنید.
- یادآورها یا پیگیری‌های تأخیری را با حلقه‌های `sleep` یا polling تکراری شبیه‌سازی نکنید؛ برای کارهای آینده از cron استفاده کنید.

## پل‌زنی فرایند فرزند

هنگام ایجاد فرایندهای فرزند بلندمدت خارج از ابزارهای exec/process (برای مثال، بازاجرای CLI یا helperهای Gateway)، helper پل فرایند فرزند را وصل کنید تا سیگنال‌های خاتمه ارسال شوند و listenerها هنگام خروج/خطا جدا شوند. این کار از فرایندهای یتیم روی systemd جلوگیری می‌کند و رفتار خاموش‌سازی را در پلتفرم‌ها سازگار نگه می‌دارد.

بازنویسی‌های محیطی:

- `PI_BASH_YIELD_MS`: yield پیش‌فرض (ms)
- `PI_BASH_MAX_OUTPUT_CHARS`: سقف خروجی درون‌حافظه‌ای (نویسه)
- `OPENCLAW_BASH_PENDING_MAX_OUTPUT_CHARS`: سقف stdout/stderr معلق برای هر جریان (نویسه)
- `PI_BASH_JOB_TTL_MS`: TTL برای نشست‌های پایان‌یافته (ms، محدود به 1m–3h)

پیکربندی (ترجیحی):

- `tools.exec.backgroundMs` (پیش‌فرض 10000)
- `tools.exec.timeoutSec` (پیش‌فرض 1800)
- `tools.exec.cleanupMs` (پیش‌فرض 1800000)
- `tools.exec.notifyOnExit` (پیش‌فرض true): وقتی exec پس‌زمینه خارج می‌شود، یک رویداد سیستم را در صف می‌گذارد + درخواست Heartbeat می‌کند.
- `tools.exec.notifyOnExitEmptySuccess` (پیش‌فرض false): وقتی true باشد، برای اجراهای پس‌زمینه موفقی که هیچ خروجی تولید نکرده‌اند نیز رویدادهای تکمیل را در صف می‌گذارد.

## ابزار process

اقدام‌ها:

- `list`: نشست‌های در حال اجرا + پایان‌یافته
- `poll`: خروجی جدید یک نشست را تخلیه می‌کند (و وضعیت خروج را نیز گزارش می‌دهد)
- `log`: خروجی تجمیع‌شده را می‌خواند (از `offset` + `limit` پشتیبانی می‌کند)
- `write`: stdin را ارسال می‌کند (`data`، `eof` اختیاری)
- `send-keys`: توکن‌های کلید صریح یا بایت‌ها را به یک نشست پشتیبانی‌شده با PTY ارسال می‌کند
- `submit`: Enter / carriage return را به یک نشست پشتیبانی‌شده با PTY ارسال می‌کند
- `paste`: متن literal را ارسال می‌کند، به‌صورت اختیاری با bracketed paste mode
- `kill`: یک نشست پس‌زمینه را خاتمه می‌دهد
- `clear`: یک نشست پایان‌یافته را از حافظه حذف می‌کند
- `remove`: اگر در حال اجرا باشد آن را می‌کشد، در غیر این صورت اگر پایان یافته باشد پاک‌سازی می‌کند

نکته‌ها:

- فقط نشست‌های پس‌زمینه فهرست/در حافظه پایدار می‌شوند.
- نشست‌ها با راه‌اندازی مجدد فرایند از دست می‌روند (پایداری دیسکی وجود ندارد).
- لاگ‌های نشست فقط در صورتی در تاریخچه چت ذخیره می‌شوند که `process poll/log` را اجرا کنید و نتیجه ابزار ثبت شود.
- `process` برای هر agent محدوده‌بندی شده است؛ فقط نشست‌هایی را می‌بیند که همان agent شروع کرده است.
- برای وضعیت، لاگ‌ها، تأیید موفقیت بی‌صدا، یا تأیید تکمیل وقتی بیدارسازی خودکار تکمیل در دسترس نیست، از `poll` / `log` استفاده کنید.
- وقتی به ورودی یا مداخله نیاز دارید از `write` / `send-keys` / `submit` / `paste` / `kill` استفاده کنید.
- `process list` برای مرور سریع شامل یک `name` مشتق‌شده (فعل فرمان + هدف) است.
- `process log` از `offset`/`limit` مبتنی بر خط استفاده می‌کند.
- وقتی هر دو `offset` و `limit` حذف شوند، 200 خط آخر را برمی‌گرداند و یک راهنمای صفحه‌بندی شامل می‌کند.
- وقتی `offset` ارائه شود و `limit` حذف شود، خروجی را از `offset` تا انتها برمی‌گرداند (به 200 محدود نمی‌شود).
- Polling برای وضعیت درخواستی است، نه زمان‌بندی حلقه انتظار. اگر کار باید بعداً انجام شود، به‌جای آن از cron استفاده کنید.

## نمونه‌ها

اجرای یک کار بلندمدت و poll کردن آن در ادامه:

```json
{ "tool": "exec", "command": "sleep 5 && echo done", "yieldMs": 1000 }
```

```json
{ "tool": "process", "action": "poll", "sessionId": "<id>" }
```

شروع فوری در پس‌زمینه:

```json
{ "tool": "exec", "command": "npm run build", "background": true }
```

ارسال stdin:

```json
{ "tool": "process", "action": "write", "sessionId": "<id>", "data": "y\n" }
```

ارسال کلیدهای PTY:

```json
{ "tool": "process", "action": "send-keys", "sessionId": "<id>", "keys": ["C-c"] }
```

ارسال خط فعلی:

```json
{ "tool": "process", "action": "submit", "sessionId": "<id>" }
```

چسباندن متن literal:

```json
{ "tool": "process", "action": "paste", "sessionId": "<id>", "text": "line1\nline2\n" }
```
