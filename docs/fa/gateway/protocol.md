پروتکل WS در Gateway، صفحهٔ کنترل واحد و مسیر انتقال Node برای
OpenClaw است. کلاینت‌های اپراتور و Node (CLI، رابط کاربری وب، برنامهٔ macOS، Nodeهای iOS/Android و
Nodeهای بدون رابط گرافیکی) از طریق WebSocket متصل می‌شوند و هنگام
دست‌دهی یک **نقش** و **دامنه** اعلام می‌کنند.

## انتقال و قاب‌بندی

- WebSocket، قاب‌های متنی، بارهای JSON.
- قاب نخست **باید** یک درخواست `connect` باشد.
- قاب‌های پیش از اتصال به 64 KiB محدود می‌شوند (`MAX_PREAUTH_PAYLOAD_BYTES`). پس از
  دست‌دهی، از `hello-ok.policy.maxPayload` و
  `hello-ok.policy.maxBufferedBytes` پیروی کنید. با فعال‌بودن عیب‌یابی، قاب‌های ورودی
  بیش‌ازحد بزرگ و بافرهای خروجی کُند، پیش از آنکه Gateway قاب را ببندد یا
  کنار بگذارد، رویدادهای `payload.large` منتشر می‌کنند. این رویدادها شامل `surface`، اندازه‌های
  بایتی، محدودیت‌ها و یک کد دلیل امن هستند و هرگز شامل بدنهٔ پیام‌ها، محتوای
  پیوست‌ها، بایت‌های خام قاب، توکن‌ها، کوکی‌ها یا اسرار نیستند.

شکل قاب‌ها:

- درخواست: `{type:"req", id, method, params}`
- پاسخ: `{type:"res", id, ok, payload|error}`
- رویداد: `{type:"event", event, payload, seq?, stateVersion?}`

متدهای دارای اثر جانبی به کلیدهای هم‌توانی نیاز دارند (به طرح‌واره مراجعه کنید).

## دست‌دهی

Gateway یک چالش پیش از اتصال ارسال می‌کند:

```json
{
  "type": "event",
  "event": "connect.challenge",
  "payload": { "nonce": "…", "ts": 1737264000000 }
}
```

کلاینت با `connect` پاسخ می‌دهد:

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "cli",
      "version": "1.2.3",
      "platform": "macos",
      "mode": "operator"
    },
    "role": "operator",
    "scopes": ["operator.read", "operator.write"],
    "caps": [],
    "commands": [],
    "permissions": {},
    "auth": { "token": "…" },
    "locale": "en-US",
    "userAgent": "openclaw-cli/1.2.3",
    "device": {
      "id": "device_fingerprint",
      "publicKey": "…",
      "signature": "…",
      "signedAt": 1737264000000,
      "nonce": "…"
    }
  }
}
```

Gateway با `hello-ok` پاسخ می‌دهد:

```json
{
  "type": "res",
  "id": "…",
  "ok": true,
  "payload": {
    "type": "hello-ok",
    "protocol": 4,
    "server": { "version": "…", "connId": "…" },
    "features": { "methods": ["…"], "events": ["…"] },
    "snapshot": { "…": "…" },
    "auth": {
      "role": "operator",
      "scopes": ["operator.read", "operator.write"]
    },
    "policy": {
      "maxPayload": 26214400,
      "maxBufferedBytes": 52428800,
      "tickIntervalMs": 15000
    }
  }
}
```

`server`، `features`، `snapshot`، `policy` و `auth` همگی توسط
`HelloOkSchema` (`packages/gateway-protocol/src/schema/frames.ts`) الزامی هستند. `auth`
نقش/دامنه‌های توافق‌شده را حتی زمانی که هیچ توکن دستگاهی صادر نشده است گزارش می‌کند (شکل
بالا). `pluginSurfaceUrls` اختیاری است و نام‌های سطح Plugin (برای مثال
`canvas`) را به URLهای میزبانی‌شدهٔ دارای دامنه نگاشت می‌کند؛ ممکن است منقضی شود، بنابراین Nodeها
برای دریافت ورودی تازه، `node.pluginSurface.refresh` را با `{ "surface": "canvas" }` فراخوانی می‌کنند.
مسیر منسوخ‌شدهٔ `canvasHostUrl` / `canvasCapability` / `node.canvas.capability.refresh`
پشتیبانی نمی‌شود؛ از سطوح Plugin استفاده کنید.
`appliedConfigHash` اختیاری در اسنپ‌شات، بازبینی پیکربندی مبدأ حل‌شده‌ای است که
زمان اجرای فعال Gateway پذیرفته است. کلاینت‌ها می‌توانند آن را با
`config.get.configRevisionHash` مقایسه کنند تا مشخص شود آیا پیکربندی ذخیره‌شدهٔ جدیدتر همچنان
به راه‌اندازی مجدد نیاز دارد یا نه. `config.get.hash` همچنان بازبینی خام فایل ریشه است که
محافظ‌های تعارض نوشتن پیکربندی از آن استفاده می‌کنند.

تا زمانی که Gateway همچنان در حال تکمیل سرویس‌های جانبی راه‌اندازی است، `connect` می‌تواند یک
خطای قابل‌تلاش مجدد `UNAVAILABLE` را با `details.reason: "startup-sidecars"` و
`retryAfterMs` برگرداند. به‌جای درنظرگرفتن آن به‌عنوان شکست نهایی دست‌دهی،
در محدودهٔ بودجهٔ اتصال دوباره تلاش کنید.

هنگامی که توکن دستگاه صادر شود، `hello-ok.auth` آن را اضافه می‌کند:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "operator",
    "scopes": ["operator.read", "operator.write"]
  }
}
```

راه‌اندازی اولیهٔ داخلی با QR/کد راه‌اندازی، مسیر واگذاری به موبایل است. اتصال موفق
پایه با کد راه‌اندازی، یک توکن اصلی Node به‌همراه یک توکن محدودشدهٔ
اپراتور برمی‌گرداند:

```json
{
  "auth": {
    "deviceToken": "…",
    "role": "node",
    "scopes": [],
    "deviceTokens": [
      {
        "deviceToken": "…",
        "role": "operator",
        "scopes": ["operator.approvals", "operator.read", "operator.talk.secrets", "operator.write"]
      }
    ]
  }
}
```

این واگذاری اپراتور عمداً محدود شده است: برای آغاز چرخهٔ اپراتور
موبایل و راه‌اندازی بومی، از جمله `operator.talk.secrets` برای خواندن پیکربندی Talk،
کافی است، اما دامنه‌های تغییر جفت‌سازی و `operator.admin` را ندارد. دسترسی گسترده‌تر
جفت‌سازی/مدیریتی به یک جریان جداگانه و تأییدشدهٔ جفت‌سازی یا توکن نیاز دارد. تنها زمانی
`hello-ok.auth.deviceTokens` را ماندگار کنید که احراز هویت راه‌اندازی اولیه از طریق یک انتقال
مورداعتماد (`wss://` یا جفت‌سازی حلقهٔ بازگشتی/محلی) اجرا شده باشد.

کلاینت‌های بک‌اند مورداعتماد در همان فرایند (`client.id: "gateway-client"`،
`client.mode: "backend"`) هنگام احراز هویت با توکن/گذرواژهٔ مشترک Gateway، می‌توانند در
اتصال‌های مستقیم حلقهٔ بازگشتی `device` را حذف کنند. این مسیر برای
RPCهای داخلی صفحهٔ کنترل (برای مثال به‌روزرسانی نشست زیربرنامه) رزرو شده است و
از مسدودشدن کار محلی بک‌اند توسط خط‌مبناهای قدیمی جفت‌سازی CLI/دستگاه جلوگیری می‌کند. کلاینت‌های
راه‌دور، دارای مبدأ مرورگر، Node و کلاینت‌های صریح توکن دستگاه/هویت دستگاه همچنان
بررسی‌های معمول جفت‌سازی و ارتقای دامنه را طی می‌کنند.

### نقش Worker و پروتکل بسته

Workerهای ابری از یک ورودی اختصاصی حلقهٔ بازگشتی از طریق تونل SSH متعلق به Gateway و
سنجاق‌شده به کلید میزبان استفاده می‌کنند. این ورودی فقط هویت Worker را می‌پذیرد و هرگز
احراز هویت عمومی، رویدادهای Node، RPCهای اپراتور یا متدهای Plugin را توزیع نمی‌کند. یک `connect`
سخت‌گیرانه، اعتبارنامه‌ای کوتاه‌عمر و به‌صورت هش در حالت ذخیره را تأیید می‌کند که به محیط، هش
بسته، دورهٔ مالک، نسخهٔ مجموعهٔ RPC، انقضا و یک نشست قابل‌تهی مقید است؛ همچنین
نسخه و مجموعه قابلیت‌های فعلی را جداگانه بررسی می‌کند. موفقیت حداقل
`worker-hello-ok` را برمی‌گرداند؛ مذاکرهٔ قابلیت مستقل از نسخهٔ عمومی پروتکل
است. قاب‌ها زیر 64 KiB باقی می‌مانند، به‌جز قاب توافق‌شدهٔ `worker.inference.start`
که می‌تواند تا 25 MiB باشد. فهرست مجاز بسته شامل `worker.heartbeat`،
`worker.transcript.commit`، `worker.live-event`، `worker.inference.start` و
`worker.inference.cancel` است.

ثبت‌های رونوشت از حصارگذاری دورهٔ مالک، اتصال نشست متعلق به Gateway،
مقایسه‌و‌تعویض برگ پایه و بازپخش پایدار توالی استفاده می‌کنند؛ Gateway شناسه‌های
ورودی و والد رونوشت را از طریق نویسندهٔ معمول نشست تولید می‌کند. مالکیت و
انقضا در هر RPC دوباره بررسی می‌شوند.

### قابلیت‌های کلاینت

کلاینت‌های اپراتور می‌توانند قابلیت‌های اختیاری را در `connect.params.caps` اعلام کنند:

- `tool-events`: رویدادهای ساختاریافتهٔ چرخهٔ عمر ابزار را می‌پذیرد.
- `inline-widgets`: می‌تواند نتایج ابزار ویجت درون‌خطی میزبانی‌شده را رندر کند.

قابلیت‌های کلاینت، کلاینت متصل را توصیف می‌کنند، نه مجوزدهی را. ابزارهای عامل ممکن است قابلیت‌های لازم را اعلام کنند؛ Gateway این ابزارها را حذف می‌کند، مگر اینکه همهٔ الزامات در `caps` کلاینت مبدأ وجود داشته باشند. اجراهای دارای مبدأ کانال هیچ قابلیت کلاینت Gateway ندارند، بنابراین ابزارهای مقید به قابلیت حتی زمانی که سیاست ابزار صراحتاً آن‌ها را مجاز می‌کند، در دسترس نیستند.

### نمونهٔ اتصال Node

```json
{
  "type": "req",
  "id": "…",
  "method": "connect",
  "params": {
    "minProtocol": 4,
    "maxProtocol": 4,
    "client": {
      "id": "ios-node",
      "version": "1.2.3",
      "platform": "ios",
      "mode": "node"
    },
    "role": "node",
    "scopes": [],
    "caps": ["camera", "canvas", "screen", "location", "voice"],
    "commands": ["camera.snap", "canvas.navigate", "screen.record", "location.get"],
    "permissions": { "camera.capture": true, "screen.record": false },
    "auth": { "token": "…" },

    "userAgent": "openclaw-ios/1.2.3",
    "device": {
      "id": "device_fingerprint",

```

Nodeها ادعاهای قابلیت را هنگام اتصال اعلام می‌کنند:

- `caps`: دسته‌های سطح‌بالا مانند `camera`، `canvas`، `screen`،
  `location`، `voice`، `talk`.
- `commands`: فهرست مجاز فرمان‌ها برای فراخوانی.
- `permissions`: کلیدهای جزئی (برای مثال `screen.record`، `camera.capture`).

Gateway این موارد را به‌عنوان ادعا در نظر می‌گیرد و فهرست‌های مجاز سمت سرور را اعمال می‌کند.

## نقش‌ها و دامنه‌ها

برای مدل کامل دامنهٔ اپراتور، بررسی‌های زمان تأیید و معناشناسی راز
مشترک، به [دامنه‌های اپراتور](/fa/gateway/operator-scopes) مراجعه کنید.

نقش‌ها:

- `operator`: کلاینت صفحهٔ کنترل (CLI/رابط کاربری/اتوماسیون).
- `node`: میزبان قابلیت (دوربین/صفحه‌نمایش/بوم/system.run).
- `worker`: میزبان اجرای ابری روی پروتکل اختصاصی و بستهٔ Worker.

دامنه‌های اپراتور (`src/gateway/operator-scopes.ts`)، مجموعهٔ بستهٔ کامل:

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`
- `operator.pairing`
- `operator.talk.secrets`

`talk.config` همراه با `includeSecrets: true` به `operator.talk.secrets` (یا
`operator.admin`) نیاز دارد. هنگامی که اسرار گنجانده شده‌اند، اعتبارنامهٔ ارائه‌دهندهٔ فعال Talk را
از `talk.resolved.config.apiKey` بخوانید؛ `talk.providers.<id>.apiKey`
شکل مبدأ را حفظ می‌کند و ممکن است یک شیء SecretRef یا رشتهٔ پوشانده‌شده باشد.

متدهای RPC در Gateway که توسط Plugin ثبت شده‌اند، ممکن است دامنهٔ اپراتور مخصوص خود را
درخواست کنند، اما این پیشوندهای رزروشدهٔ هسته همیشه به `operator.admin`
(`src/shared/gateway-method-policy.ts`) حل می‌شوند: `config.*`، `exec.approvals.*`،
`wizard.*`، `update.*`.

دامنهٔ متد فقط نخستین دروازه است. برخی فرمان‌های اسلش که از طریق
`chat.send` قابل‌دسترسی هستند، بررسی‌های سخت‌گیرانه‌تری در سطح فرمان اعمال می‌کنند: نوشتن پایدار `/config set` و
`/config unset` به `operator.admin` نیاز دارد، حتی برای کلاینت‌های Gateway که
از پیش یک دامنهٔ اپراتور پایین‌تر دارند.

`node.pair.approve` افزون بر دامنهٔ پایهٔ متد
(`operator.pairing`)، یک بررسی دامنهٔ اضافی هنگام تأیید دارد که بر اساس
`commands` اعلام‌شدهٔ درخواست در انتظار (`src/infra/node-pairing-authz.ts`) انجام می‌شود:

| فرمان‌های اعلام‌شده                                                                                                           | دامنه‌های لازم                         |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| هیچ‌کدام                                                                                                                       | `operator.pairing`                    |
| فرمان‌های عادی                                                                                                                | `operator.pairing` + `operator.write` |
| شامل `system.run`، `system.run.prepare`، `system.which`، `browser.proxy`، `fs.listDir` یا `system.execApprovals.get/set` | `operator.pairing` + `operator.admin` |

### قابلیت‌ها/فرمان‌ها/مجوزها (Node)

Nodeها ادعاهای قابلیت را هنگام اتصال اعلام می‌کنند:

- `caps`: دسته‌های قابلیت سطح‌بالا مانند `camera`، `canvas`، `screen`،
  `location`، `voice` و `talk`.
- `commands`: فهرست مجاز فرمان‌ها برای فراخوانی.
- `permissions`: کلیدهای جزئی (برای مثال `screen.record`، `camera.capture`).

Gateway این موارد را به‌عنوان **ادعاها** در نظر می‌گیرد و فهرست‌های مجاز را در سمت سرور اعمال می‌کند.
Nodeهای متصل می‌توانند پس از اتصال یا اتصال مجدد موفق، توصیف‌گرهای اختیاری Plugin یا ابزار MCP قابل‌مشاهده برای عامل را با `node.pluginTools.update` منتشر کنند.
میزبان‌های Node بدون رابط برای اعمال تغییرات اعلانی موجودی MCP راه‌اندازی مجدد می‌شوند.
این روش به‌روزرسانی تنها مسیر انتشار است؛ توصیف‌گرهای ابزار Plugin در پارامترهای
`connect` پذیرفته نمی‌شوند. هر توصیف‌گر باید از یک `name` ابزار ایمن برای ارائه‌دهنده استفاده کند و
یک `command` موجود در فهرست مجاز فرمان فعلی Node را نام ببرد. Gateway به فراداده توصیف‌گر
دریافتی از Node جفت‌شده اعتماد می‌کند، توصیف‌گرهای خارج از سطح فرمان تأییدشده را
فیلتر می‌کند، هنگام قطع اتصال Node آن‌ها را حذف می‌کند و تلاش‌های اپراتور برای
تغییر کاتالوگ Node دیگر را رد می‌کند. برای نادیده‌گرفتن توصیف‌گرهای منتشرشده توسط Node، `gateway.nodes.pluginTools.enabled: false`
را تنظیم کنید.

میزبان‌های Node متصل، کاتالوگ کامل جایگزین Skills خود را با
`node.skills.update` منتشر می‌کنند. این روش نقش Node تنها مسیر انتشار Skills توسط Node
است؛ Skills در پارامترهای `connect` پذیرفته نمی‌شوند. هر توصیف‌گر شامل یک
نام ایمن، توضیح و محتوای محدودشده `SKILL.md` است. Gateway این
محتوا را با بارگذار معمول Skills تجزیه می‌کند، تا زمانی که Node متصل است آن را در تصویرهای لحظه‌ای Skills عامل
می‌گنجاند و هنگام قطع اتصال حذف می‌کند. برای نادیده‌گرفتن Skills منتشرشده توسط Node،
`gateway.nodes.skills.enabled: false` را تنظیم کنید.

## حضور

- `system-presence` ورودی‌هایی را برمی‌گرداند که بر اساس هویت دستگاه کلیدگذاری شده‌اند و شامل
  `deviceId`، `roles` و `scopes` هستند؛ بنابراین رابط‌های کاربری می‌توانند برای هر دستگاه یک ردیف نمایش دهند، حتی
  هنگامی که دستگاه هم به‌عنوان اپراتور و هم به‌عنوان Node متصل می‌شود.
- `node.list` شامل `lastSeenAtMs` و `lastSeenReason` اختیاری است. Nodeهای متصل
  زمان اتصال فعلی را با دلیل `connect` گزارش می‌کنند؛ Nodeهای جفت‌شده همچنین می‌توانند
  حضور پایدار در پس‌زمینه را از طریق یک رویداد قابل‌اعتماد Node گزارش کنند.

Nodeهای بومی macOS همچنین می‌توانند رویدادهای احرازهویت‌شده `node.presence.activity` را
با زمان بیکاری ورودی محدودشده ارسال کنند. Gateway مُهرهای زمانی فعالیت را بر اساس
ساعت خودش استخراج می‌کند، تازه‌ترین Mac متصل را از طریق `node.list` و
`node.describe` در دسترس قرار می‌دهد و به‌روزرسانی‌های `node.presence` را برای کلاینت‌های دارای دامنه خواندن پخش می‌کند.
برای رفتار انتخاب، حریم خصوصی، زمینه مدل و مسیریابی اعلان‌ها، به [حضور رایانه فعال](/nodes/presence)
مراجعه کنید.

### رویداد زنده‌بودن Node در پس‌زمینه

Nodeها `node.event` را با `event: "node.presence.alive"` فراخوانی می‌کنند تا ثبت شود که یک
Node جفت‌شده هنگام بیدارسازی پس‌زمینه زنده بوده است، بدون آنکه متصل علامت‌گذاری شود:

```json
{
  "event": "node.presence.alive",
  "payloadJSON": "{\"trigger\":\"silent_push\",\"sentAtMs\":1737264000000,\"displayName\":\"Peter's iPhone\",\"version\":\"2026.4.28\",\"platform\":\"iOS 18.4.0\",\"deviceFamily\":\"iPhone\",\"modelIdentifier\":\"iPhone17,1\",\"pushTransport\":\"relay\"}"
}
```

`trigger` یک شمارش بسته است: `background`، `silent_push`، `bg_app_refresh`،
`significant_location`، `manual`، `connect`. مقادیر ناشناخته به
`background` (`src/shared/node-presence.ts`) نرمال‌سازی می‌شوند. رویداد فقط برای
نشست‌های احرازهویت‌شده دستگاه Node پایدار می‌ماند؛ نشست‌های بدون دستگاه یا جفت‌نشده
`handled: false` را برمی‌گردانند.

Gatewayهای موفق یک نتیجه ساخت‌یافته برمی‌گردانند:

```json
{
  "ok": true,
  "event": "node.presence.alive",
  "handled": true,
  "reason": "persisted"
}
```

Gatewayهای قدیمی‌تر ممکن است برای `node.event` فقط `{ "ok": true }` را برگردانند؛ آن را
یک RPC تأییدشده در نظر بگیرید، نه ماندگارسازی پایدار حضور.

## دامنه‌بندی رویدادهای پخشی

رویدادهای پخشی ارسال‌شده از سرور بر اساس دامنه محدود می‌شوند تا نشست‌های
محدود به جفت‌سازی یا فقط Node، محتوای نشست را به‌صورت منفعل دریافت نکنند
(`src/gateway/server-broadcast.ts`):

- قاب‌های گفت‌وگو، عامل و نتیجه ابزار (رویدادهای جریانی `agent` و رویدادهای
  نتیجه ابزار) دست‌کم به `operator.read` نیاز دارند. نشست‌های فاقد آن، این
  قاب‌ها را کاملاً نادیده می‌گیرند.
- پخش‌های `plugin.*` تعریف‌شده توسط Plugin به‌طور پیش‌فرض به `operator.write` یا
  `operator.admin` محدود می‌شوند؛ ورودی‌های صریحی مانند
  `plugin.approval.requested` / `plugin.approval.resolved` به‌جای آن از
  `operator.approvals` استفاده می‌کنند.
- رویدادهای وضعیت/انتقال (`heartbeat`، `presence`، `tick` و چرخه عمر
  اتصال/قطع اتصال) بدون محدودیت باقی می‌مانند تا سلامت انتقال برای هر
  نشست احرازهویت‌شده قابل‌مشاهده باشد.
- خانواده‌های ناشناخته رویداد پخشی به‌طور پیش‌فرض بر اساس دامنه محدود می‌شوند (بسته در برابر خطا)،
  مگر آنکه یک کنترل‌گر ثبت‌شده صراحتاً محدودیت آن‌ها را کاهش دهد.

هر اتصال کلاینت شماره توالی مختص همان کلاینت را نگه می‌دارد؛ بنابراین پخش‌ها
در آن سوکت به‌ترتیب یکنواخت باقی می‌مانند، حتی زمانی که کلاینت‌های مختلف
زیرمجموعه‌های متفاوتی از جریان رویداد را پس از فیلتر دامنه می‌بینند.

## خانواده‌های روش RPC

`hello-ok.features.methods` یک فهرست اکتشافی محافظه‌کارانه است که از
`src/gateway/server-methods-list.ts` به‌همراه خروجی‌های روش Plugin/کانال بارگذاری‌شده
ساخته می‌شود؛ این فهرست، تخلیه تولیدشده همه روش‌ها نیست و برخی روش‌ها (برای
مثال `push.test`، `web.login.start`، `web.login.wait`، `sessions.usage`)
عمداً از اکتشاف کنار گذاشته شده‌اند، هرچند روش‌هایی واقعی و قابل‌فراخوانی
هستند. این را برای کشف قابلیت‌ها در نظر بگیرید، نه فهرست کامل
`src/gateway/server-methods/*.ts`.

    - `talk.session.submitToolResult` فراخوانی ابزار ارائه‌دهنده را که یک نشست رله بلادرنگ تحت مالکیت Gateway منتشر کرده است، تکمیل می‌کند. درخواست منتظر هرگونه سیگنال تکمیل ناهمگام عرضه‌شده توسط پل ارائه‌دهنده می‌ماند؛ ارسال‌های ناموفق، اجرای پیوندخورده را فعال نگه می‌دارند و رویداد نتیجه موفق ابزار منتشر نمی‌کنند. برای خروجی موقت ابزار، `options: { willContinue: true }` را ارسال کنید؛ یا هنگامی که پل ارائه‌دهنده پشتیبانی از سرکوب را اعلام می‌کند و نتیجه نباید پاسخ دیگری را آغاز کند، `options: { suppressResponse: true }` را ارسال کنید.
    - `talk.session.steer` کنترل صوتی اجرای فعال را به یک نشست گفت‌وگوی متکی بر عامل و تحت مالکیت Gateway ارسال می‌کند: `{ sessionId, text, mode? }`، که در آن `mode` یکی از `status`، `steer`، `cancel` یا `followup` است؛ در صورت حذف حالت، آن از روی متن گفته‌شده طبقه‌بندی می‌شود.
    - `talk.session.close` یک نشست رله، رونویسی یا اتاق مدیریت‌شده تحت مالکیت Gateway را می‌بندد و رویدادهای پایانی گفت‌وگو را منتشر می‌کند.
    - `talk.mode` وضعیت فعلی حالت گفت‌وگو را برای کلاینت‌های WebChat/Control UI تنظیم/پخش می‌کند.
    - `talk.client.create` با استفاده از `webrtc` یا `provider-websocket` یک نشست ارائه‌دهنده بلادرنگ تحت مالکیت کلاینت ایجاد می‌کند، درحالی‌که Gateway مالک پیکربندی، اعتبارنامه‌ها، دستورالعمل‌ها و خط‌مشی ابزار است.
    - `talk.client.toolCall` به انتقال‌های بلادرنگ تحت مالکیت کلاینت امکان می‌دهد فراخوانی‌های ابزار ارائه‌دهنده را به خط‌مشی Gateway هدایت کنند. نخستین ابزار پشتیبانی‌شده `openclaw_agent_consult` است؛ کلاینت‌ها یک شناسه اجرا دریافت می‌کنند و پیش از ارسال نتیجه ابزار مختص ارائه‌دهنده، منتظر رویدادهای عادی چرخه عمر چت می‌مانند.
    - `talk.client.steer` کنترل صوتی اجرای فعال را برای انتقال‌های بلادرنگ تحت مالکیت کلاینت ارسال می‌کند. Gateway اجرای تعبیه‌شده فعال را از `sessionKey` حل می‌کند و به‌جای کنار گذاشتن بی‌سروصدای هدایت، یک نتیجه ساخت‌یافته پذیرفته‌شده/ردشده برمی‌گرداند.
    - `talk.event` کانال واحد رویداد گفت‌وگو برای آداپتورهای بلادرنگ، رونویسی، STT/TTS، اتاق مدیریت‌شده، تلفن و جلسه است.
    - `talk.speak` از طریق ارائه‌دهنده فعال گفتار گفت‌وگو، گفتار را ترکیب می‌کند.
    - `tts.status` وضعیت فعال‌بودن TTS، ارائه‌دهنده فعال، ارائه‌دهندگان جایگزین و وضعیت پیکربندی ارائه‌دهنده را برمی‌گرداند.
    - `tts.providers` فهرست قابل‌مشاهده ارائه‌دهندگان TTS را برمی‌گرداند.
    - `tts.enable` و `tts.disable` وضعیت ترجیحات TTS را تغییر می‌دهند.
    - `tts.setProvider` ارائه‌دهنده ترجیحی TTS را به‌روزرسانی می‌کند.
    - `tts.convert` یک تبدیل یک‌باره متن به گفتار را اجرا می‌کند.
    - `tts.speak` (`operator.write`) مقدار غیرخالی `text` را با زنجیره پیکربندی‌شده ارائه‌دهندگان عمومی TTS رندر می‌کند و یک کلیپ کامل را به‌صورت درون‌خطی در قالب `audioBase64`، به‌همراه `provider` و فراداده اختیاری `outputFormat`، `mimeType` و `fileExtension` برمی‌گرداند. برخلاف `tts.convert`، مسیر محلی Gateway را برنمی‌گرداند؛ برخلاف `talk.speak`، به ارائه‌دهنده گفت‌وگو نیاز ندارد. متن بالاتر از `messages.tts.maxTextLength` مقدار `INVALID_REQUEST` را برمی‌گرداند؛ شکست‌های ترکیب گفتار، `UNAVAILABLE` را برمی‌گردانند.

  </Accordion>

  <Accordion title="اسرار، پیکربندی، به‌روزرسانی و راه‌انداز">
    - `secrets.reload` ارجاع‌های فعال SecretRef را دوباره حل می‌کند و فقط در صورت موفقیت کامل، وضعیت اسرار زمان اجرا را جایگزین می‌کند.
    - `secrets.resolve` انتساب‌های اسرار مقصد فرمان را برای یک مجموعه مشخص فرمان/مقصد حل می‌کند.
    - `config.get` تصویر لحظه‌ای پیکربندی فعلی روی دیسک، `hash` خام فایل ریشه، `configRevisionHash` حل‌شده و `appliedConfigHash` اختیاری را برای بازبینی حل‌شده پذیرفته‌شده توسط زمان اجرای فعال Gateway برمی‌گرداند.
    - `config.set` یک محموله پیکربندی اعتبارسنجی‌شده را می‌نویسد.
    - `config.patch` یک به‌روزرسانی جزئی پیکربندی را ادغام می‌کند. جایگزینی مخرب آرایه مستلزم قرارگرفتن مسیر متأثر در `replacePaths` است؛ آرایه‌های تودرتو زیر ورودی‌های آرایه از مسیرهای `[]` مانند `agents.list[].skills` استفاده می‌کنند.
    - `config.apply` کل محموله پیکربندی را اعتبارسنجی و جایگزین می‌کند.
    - `config.schema` محموله زنده شِمای پیکربندی مورد استفاده ابزارهای Control UI و CLI را برمی‌گرداند: شِما، `uiHints`، نسخه، فراداده تولید و در صورت بارگذاری‌پذیری، فراداده شِمای Plugin و کانال. این محموله، فراداده `title` / `description` را از همان برچسب‌ها/متن راهنمای رابط کاربری شامل می‌شود؛ ازجمله شاخه‌های ترکیبی شیء تودرتو، نویسه عام، مورد آرایه و `anyOf` / `oneOf` / `allOf`، درصورتی‌که مستندات منطبق فیلد وجود داشته باشد.
    - `config.schema.lookup` برای یک مسیر پیکربندی، محموله جست‌وجوی محدود به مسیر را برمی‌گرداند: مسیر نرمال‌شده، یک گره کم‌عمق شِما، راهنمای منطبق به‌همراه `hintPath`، `reloadKind` اختیاری و خلاصه‌های فرزندان بلافصل برای پیمایش جزئی‌تر UI/CLI. `reloadKind` یکی از `restart`، `hot` یا `none` (`src/config/schema.ts`) است و برنامه‌ریز بارگذاری مجدد پیکربندی Gateway را برای مسیر درخواستی بازتاب می‌دهد. گره‌های شِمای جست‌وجو، مستندات کاربرمحور و فیلدهای رایج اعتبارسنجی (`title`، `description`، `type`، `enum`، `const`، `format`، `pattern`، کران‌های عدد/رشته/آرایه/شیء، `additionalProperties`، `deprecated`، `readOnly`، `writeOnly`) را نگه می‌دارند. خلاصه‌های فرزند، `key`، `path` نرمال‌شده، `type`، `required`، `hasChildren`، `reloadKind` اختیاری و نیز `hint` / `hintPath` منطبق را عرضه می‌کنند.
    - `update.run` جریان به‌روزرسانی Gateway را اجرا می‌کند و فقط در صورت موفقیت به‌روزرسانی، راه‌اندازی مجدد را زمان‌بندی می‌کند؛ فراخوانندگانی که نشست دارند می‌توانند `continuationMessage` را اضافه کنند تا هنگام آغاز به کار، یک نوبت پیگیری عامل از طریق صف ادامه پس از راه‌اندازی مجدد از سر گرفته شود. به‌روزرسانی‌های مدیر بسته و به‌روزرسانی‌های تحت نظارت نسخه کاری git از صفحه کنترل، به‌جای جایگزینی درخت بسته یا تغییر نسخه کاری/خروجی ساخت درون Gateway فعال، از یک تحویل سرویس مدیریت‌شده جداشده استفاده می‌کنند. تحویل آغازشده، `ok: true` را با `result.reason: "managed-service-handoff-started"` و `handoff.status: "started"` برمی‌گرداند؛ تحویل‌های دردسترس‌نبوده یا ناموفق، `ok: false` را با `managed-service-handoff-unavailable` یا `managed-service-handoff-failed` و در صورت نیاز به به‌روزرسانی دستی پوسته، به‌همراه `handoff.command` برمی‌گردانند. دردسترس‌نبودن یعنی OpenClaw فاقد یک مرز امن ناظر یا هویت پایدار سرویس است، مانند `OPENCLAW_SYSTEMD_UNIT` برای systemd. طی یک تحویل آغازشده، نشانگر راه‌اندازی مجدد ممکن است برای مدت کوتاهی `stats.reason: "restart-health-pending"` را گزارش کند؛ ادامه تا زمانی به‌تعویق می‌افتد که CLI، Gateway راه‌اندازی‌شده مجدد را تأیید کند و نشانگر نهایی `ok` را بنویسد.
    - `update.status` جدیدترین نشانگر راه‌اندازی مجدد پس از به‌روزرسانی، ازجمله نسخه در حال اجرای پس از راه‌اندازی مجدد در صورت موجودبودن، را تازه‌سازی و برمی‌گرداند.
    - `wizard.start`، `wizard.next`، `wizard.status` و `wizard.cancel` راه‌انداز ورود اولیه را از طریق WS RPC عرضه می‌کنند.

  <Accordion title="راهنماهای عامل و فضای کاری">
    - `agents.list` ورودی‌های پیکربندی‌شدهٔ عامل، از جمله مدل مؤثر و فراداده‌های محیط اجرا را برمی‌گرداند.
    - `agents.create`، `agents.update` و `agents.delete` رکوردهای عامل و اتصال فضای کاری را مدیریت می‌کنند.
    - `agents.files.list`، `agents.files.get` و `agents.files.set` فایل‌های فضای کاری راه‌اندازی اولیه را که برای یک عامل در دسترس‌اند مدیریت می‌کنند.
    - `audit.activity.list` دفتر ثبت فعالیتِ نسخه‌بندی‌شده و صرفاً شامل فراداده را برمی‌گرداند؛ `audit.list` همچنان RPC اجرای/ابزارِ ایمن از نظر سازگاری باقی می‌ماند.

    - `tasks.list`، `tasks.get` و `tasks.cancel` دفتر ثبت وظایف Gateway را در اختیار کلاینت‌های SDK و اپراتور قرار می‌دهند. بخش [RPCهای دفتر ثبت وظایف](#task-ledger-rpcs) را در ادامه ببینید.
    - `artifacts.list`، `artifacts.get` و `artifacts.download` خلاصه‌ها و بارگیری‌های مصنوعات استخراج‌شده از رونوشت را برای یک دامنهٔ صریح `sessionKey`، `runId` یا `taskId` فراهم می‌کنند. پرس‌وجوهای اجرا و وظیفه، نشست مالک را در سمت سرور تشخیص می‌دهند و فقط رسانه‌های رونوشت با منشأ منطبق را برمی‌گردانند؛ منابع URL ناامن یا محلی، به‌جای واکشی در سمت سرور، بارگیری‌های پشتیبانی‌نشده برمی‌گردانند.
    - `environments.list` و `environments.status` کشف محیط محلی Gateway و Node را حفظ می‌کنند. کارگرهای ابری پیکربندی‌شده و رکوردهای ماندگاری که نمایه‌های پیشین باقی گذاشته‌اند، فرادادهٔ `worker` را با `providerId`، مقدار اختیاری `leaseId`، `state`، `ageMs`، مقدار اختیاری `idleMs` و `attachedSessionIds` اضافه می‌کنند. وضعیت‌های چرخهٔ عمر کارگر عبارت‌اند از `requested`، `provisioning`، `bootstrapping`، `ready`، `attached`، `idle`، `draining`، `destroying`، `destroyed`، `failed` و `orphaned`.
    - `environments.create` ‏(`{ profileId, idempotencyKey }`) یک کارگر را از نمایهٔ ارائه‌دهندهٔ Plugin پیکربندی‌شده تأمین می‌کند؛ تلاش‌های مجدد با همان کلید، عملیات ماندگار را دوباره استفاده می‌کنند. `environments.destroy` ‏(`{ environmentId }`) برچیدن ایدمپوتنت یک محیط کارگر ماندگار را درخواست می‌کند. هر دو به `operator.admin` نیاز دارند، نوشتن‌های صفحهٔ کنترل هستند و همان ساختار خلاصهٔ محیطی را برمی‌گردانند که پاسخ‌های وضعیت استفاده می‌کنند.
    - `agent.identity.get` هویت مؤثر دستیار را برای یک عامل یا نشست برمی‌گرداند.
    - `agent.wait` تا پایان یک اجرا منتظر می‌ماند و در صورت موجود بودن، تصویر لحظه‌ای نهایی را برمی‌گرداند.

  <Accordion title="کنترل نشست">
    - `sessions.list` نمایهٔ فعلی نشست را، از جمله فرادادهٔ `agentRuntime` برای هر ردیف در صورت پیکربندی بودن پشتیبان محیط اجرای عامل، برمی‌گرداند. وقتی جانمایی کارگر ابری فعال باشد یا وضعیت بازیابی ماندگاری وجود داشته باشد، ردیف‌های نشست همچنین یک وضعیت بستهٔ `placement` ‏(`local`، `requested`، `provisioning`، `syncing`، `starting`، `active`، `draining`، `reconciling`، `reclaimed` یا `failed`) به‌همراه فیلدهای مختص وضعیت برای محیط، دورهٔ مالک، فضای کاری، بسته، مکان‌نمای ACK یا بازیابی دارند.
    - `sessions.subscribe` و `sessions.unsubscribe` اشتراک رویدادهای تغییر نشست را برای کلاینت WS فعلی فعال یا غیرفعال می‌کنند.
    - `sessions.messages.subscribe` و `sessions.messages.unsubscribe` اشتراک رویدادهای رونوشت/پیام را برای یک نشست فعال یا غیرفعال می‌کنند. برای دریافت رویدادهای پاک‌سازی‌شدهٔ چرخهٔ عمر `session.approval` مربوط به تأییدهایی که مخاطبان ماندگارشان دقیقاً شامل همان نشست است و اتصال بازبین آن‌ها کلاینت مشترک‌شونده را مجاز می‌کند، `includeApprovals: true` را نیز ارسال کنید. پاسخ اشتراک سپس یک `approvalReplay` معلق و کران‌دار را شامل می‌شود؛ وقتی `truncated` نادرست باشد، این مقدار مرجع معتبر است. این انتخاب برای هر فراخوانی اشتراک جداگانه است و ماندگار نیست: اشتراک مجدد در همان نشست بدون `includeApprovals: true`، اشتراک تأیید موجود را حذف می‌کند. این انتخاب، افزون بر اختیار عادی خواندن نشست، به `operator.admin` یا در دستگاه جفت‌شده به `operator.approvals` نیاز دارد.
    - `sessions.preview` پیش‌نمایش‌های کران‌دار رونوشت را برای کلیدهای نشست مشخص برمی‌گرداند.
    - `sessions.describe` یک ردیف نشست Gateway را برای یک کلید نشست دقیق برمی‌گرداند.
    - `sessions.resolve` یک مقصد نشست را تشخیص می‌دهد یا به شکل متعارف درمی‌آورد.
    - `sessions.create` یک ورودی نشست جدید ایجاد می‌کند. مقادیر اختیاری `model` و `thinkingLevel`، جایگزین‌های اولیهٔ مدل و استدلال را به‌صورت اتمی ماندگار می‌کنند. `worktree: true` یک درخت‌کاری مدیریت‌شده تأمین می‌کند؛ `worktreeBaseRef`/`worktreeName` اختیاری، مرجع پایه و نام شاخه را انتخاب می‌کنند و `execNode` ‏(`operator.admin`) اجرای نشست را به یک میزبان Node متصل می‌کند. درخت‌کاری ایجادشده در نتیجه بازگردانده می‌شود و در ردیف نشست (`worktree: { id, branch, repoRoot }`) ماندگار می‌ماند. وقتی ورودی ایجاد شود اما `chat.send` اولیهٔ تو‌در‌توی آن رد شود، نتیجهٔ موفق شامل `runStarted: false` و `runError` است؛ کلاینت‌ها می‌توانند اعلان را حفظ کنند و با کلید نشست بازگردانده‌شده دوباره تلاش کنند.
    - `sessions.dispatch` ‏(`operator.admin`) یک نشست محلی موجود OpenClaw را که دارای درخت‌کاری مدیریت‌شده و متعلق به نشست است، به یک نمایهٔ کارگر ابری پیکربندی‌شده منتقل می‌کند. `{ key, profileId, agentId? }` را ارسال کنید. وقتی هیچ نمایهٔ کارگری پیکربندی نشده باشد، این روش وجود ندارد؛ پیش از تخلیهٔ کار فعال، پذیرش نوبت محلی را می‌بندد و فقط پس از رسیدن جانمایی به مالکیت کارگر `active` بازمی‌گردد. اعزام یک‌طرفه است؛ بازکشیدن از کارگر به محیط محلی بخشی از این RPC نیست.
    - `sessions.groups.list`، `sessions.groups.put`، `sessions.groups.rename` و `sessions.groups.delete` فهرست گروه‌های نشست سفارشیِ متعلق به Gateway را مدیریت می‌کنند (نام‌ها + ترتیب نمایش). عضویت در فیلد `category` هر نشست باقی می‌ماند؛ تغییر نام و حذف، نشست‌های عضو را در سمت سرور به‌روزرسانی می‌کنند.
    - `sessions.send` پیامی را به یک نشست موجود ارسال می‌کند.
    - `sessions.steer` گونهٔ وقفه‌دادن و هدایت‌کردن برای یک نشست فعال است.
    - `sessions.abort` کار فعال یک نشست را لغو می‌کند. `key` را به‌همراه `runId` اختیاری ارسال کنید، یا برای اجراهای فعالی که Gateway می‌تواند نشست آن‌ها را تشخیص دهد، فقط `runId` را ارسال کنید.
    - `sessions.patch` فراداده/جایگزین‌های نشست را به‌روزرسانی می‌کند و مدل متعارف تشخیص‌داده‌شده را به‌همراه `agentRuntime` مؤثر گزارش می‌دهد.
    - `sessions.reset`، `sessions.delete` و `sessions.compact` نگهداری نشست را انجام می‌دهند.
    - `sessions.get` ردیف کامل ذخیره‌شدهٔ نشست را برمی‌گرداند.
    - اجرای گفت‌وگو همچنان از `chat.history`، `chat.send`، `chat.abort` و `chat.inject` استفاده می‌کند. `chat.history` برای کلاینت‌های رابط کاربری از نظر نمایش عادی‌سازی می‌شود: برچسب‌های دستور درون‌خطی از متن قابل‌مشاهده حذف می‌شوند، محموله‌های XML فراخوانی ابزار با متن ساده (`<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>` و بلوک‌های کوتاه‌شدهٔ فراخوانی ابزار) و توکن‌های کنترل مدل ASCII/تمام‌عرضِ نشت‌کرده حذف می‌شوند، ردیف‌های دستیار که صرفاً حاوی توکن سکوت هستند (دقیقاً `NO_REPLY` / `no_reply`) کنار گذاشته می‌شوند و ردیف‌های بیش‌ازحد بزرگ ممکن است با جای‌نگهدار جایگزین شوند.
    - `chat.message.get` خوانشگر افزایشی، کران‌دار و تمام‌پیام برای یک ورودی قابل‌مشاهدهٔ رونوشت است. `sessionKey`، در صورت عامل‌محور بودن انتخاب نشست مقدار اختیاری `agentId`، و یک `messageId` رونوشت را که پیش‌تر از طریق `chat.history` ارائه شده است ارسال کنید؛ اگر ورودی ذخیره‌شده همچنان موجود و بیش‌ازحد بزرگ نباشد، Gateway همان تصویر عادی‌سازی‌شده برای نمایش را بدون سقف کوتاه‌سازی تاریخچهٔ سبک‌وزن برمی‌گرداند.
    - `chat.toolTitles` عنوان‌های کوتاه هدف را برای فراخوانی‌های ابزاری که در رابط کاربری کنترل نمایش داده می‌شوند برمی‌گرداند (دسته‌ای، حداکثر 24 مورد با ورودی‌های کران‌دار). این قابلیت از طریق `gateway.controlUi.toolTitles` اختیاری است (به‌طور پیش‌فرض خاموش)؛ Gatewayهای غیرفعال بدون فراخوانی مدل با `{ titles: {}, disabled: true }` پاسخ می‌دهند تا کلاینت‌ها دیگر درخواست نکنند. وقتی فعال باشد، عنوان‌ها از مسیریابی استاندارد مدل کمکی استفاده می‌کنند: یا `utilityModel` صریحاً پیکربندی‌شده (تصمیمی از سوی اپراتور که مانند همهٔ وظایف کمکی ممکن است محتوای کران‌دار وظیفه را به ارائه‌دهندهٔ انتخاب‌شده ارسال کند)، یا پیش‌فرض مدل کوچک اعلام‌شدهٔ ارائه‌دهندهٔ نشست تا هیچ مقصد خروجی جدیدی به‌طور ضمنی پدید نیاید؛ `utilityModel` خالی آن‌ها را کاملاً غیرفعال می‌کند. عنوان‌ها هرگز به مدل اصلی بازنمی‌گردند. نتایج با کلید نام ابزار + ورودی در پایگاه دادهٔ وضعیت هر عامل ذخیرهٔ موقت می‌شوند، بنابراین مشاهده‌های تکراری هرگز دوباره هزینهٔ همان فراخوانی‌ها را ایجاد نمی‌کنند.
    - `chat.send` مقدار تک‌نوبتی `fastMode: "auto"` را می‌پذیرد تا برای فراخوانی‌های مدلی که پیش از برش خودکار آغاز می‌شوند از حالت سریع استفاده کند و سپس فراخوانی‌های تلاش مجدد، بازگشت، نتیجهٔ ابزار یا ادامه را که بعدتر آغاز می‌شوند بدون حالت سریع اجرا کند. مقدار پیش‌فرض برش 60 ثانیه است (`DEFAULT_FAST_MODE_AUTO_ON_SECONDS`) و می‌توان آن را برای هر مدل با `agents.defaults.models["<provider>/<model>"].params.fastAutoOnSeconds` پیکربندی کرد. فراخوانندهٔ `chat.send` می‌تواند مقدار تک‌نوبتی `fastAutoOnSeconds` را برای بازنویسی برش در همان درخواست ارسال کند. برای بازنویسی حالت صف ذخیره‌شده فقط برای این درخواست، `queueMode` ‏(`steer`، `followup`، `collect` یا `interrupt`) را ارسال کنید؛ کنش‌های صریح هدایت در رابط کاربری کنترل از `queueMode: "steer"` استفاده می‌کنند.

### خانواده‌های متداول رویداد

- `chat`: به‌روزرسانی‌های گفت‌وگوی رابط کاربری مانند `chat.inject` و دیگر رویدادهای گفت‌وگو که فقط در رونوشت هستند. در پروتکل v4، محموله‌های تغییرات شامل `deltaText` هستند؛ `message` همچنان
  نمای لحظه‌ای تجمعی دستیار است. جایگزینی‌هایی که پیشوند نیستند،
  `replace=true` را تنظیم می‌کنند و از `deltaText` به‌عنوان متن جایگزین استفاده می‌کنند.
- `session.message`، `session.operation`، `session.tool`: به‌روزرسانی‌های رونوشت، عملیات در حال اجرای
  نشست و جریان رویداد برای یک نشست مشترک‌شده.
- `session.approval`: حقیقت پاک‌سازی‌شده تأییدهای در انتظار و نهایی برای یک
  مشترکِ نشست دقیق که صریحاً پذیرفته است. تأییدهای فرزند از مخاطبان نیای
  ذخیره‌شده استفاده می‌کنند؛ رویدادها هرگز رونوشت‌ها را تغییر نمی‌دهند یا عامل‌ها را بیدار نمی‌کنند.
- `sessions.changed`: نمایه یا فراداده نشست تغییر کرده است.
- `presence`: به‌روزرسانی‌های نمای لحظه‌ای حضور سیستم.
- `tick`: رویداد دوره‌ای زنده‌نگه‌داشتن/فعال‌بودن.
- `health`: به‌روزرسانی نمای لحظه‌ای سلامت Gateway.
- `heartbeat`: به‌روزرسانی جریان رویداد Heartbeat.
- `cron`: رویداد تغییر اجرای Cron/کار.
- `shutdown`: اعلان خاموش‌شدن Gateway.
- `node.pair.requested` / `node.pair.resolved`: چرخه حیات جفت‌سازی Node.
- `node.invoke.request`: پخش درخواست فراخوانی Node.
- `device.pair.requested` / `device.pair.resolved`: چرخه حیات دستگاه جفت‌شده.
- `voicewake.changed`: پیکربندی محرک واژه بیدارباش تغییر کرده است.
- `exec.approval.requested` / `exec.approval.resolved`: چرخه حیات
  تأیید اجرا.
- `plugin.approval.requested` / `plugin.approval.resolved`: چرخه حیات
  تأیید Plugin.

## RPC دفتر کل ممیزی

`audit.activity.list` نمایی پایدار و از جدیدترین به قدیمی‌ترین از فراداده چرخه حیات
اجرای عامل، کنش ابزار و پیام‌های انتخابی به کلاینت‌های اپراتور می‌دهد. این روش به
`operator.read` نیاز دارد. پرس‌وجوها رکوردهای قدیمی‌تر از 30 روز را حذف می‌کنند و دفتر کل مشترک
SQLite به 100,000 رکورد محدود است. ردیف‌های منقضی‌شده هنگام
راه‌اندازی Gateway، نگه‌داری ساعتی و نوشتن‌های بعدی حذف می‌شوند. برای مدل داده و معناشناسی حریم خصوصی،
[تاریخچه ممیزی](/gateway/audit) را ببینید.

- پارامترها: `agentId`، `sessionKey` یا `runId` دقیق و اختیاری؛ `kind` اختیاری
  (`"agent_run"`، `"tool_action"` یا `"message"`)؛ `status` اختیاری
  (`"started"`، `"succeeded"`، `"failed"`، `"cancelled"`، `"timed_out"`،
  `"blocked"` یا `"unknown"`)؛ پیام اختیاریِ `direction` (`"inbound"` یا
  `"outbound"`) و `channel` دقیق؛ کران‌های فراگیر و اختیاریِ `after` / `before`
  برحسب میلی‌ثانیه یونیکس؛ `limit` اختیاری از `1` تا `500`؛ و رشته اختیاریِ
  `cursor` از صفحه پیشین.
- نتیجه: `{ "events": AuditActivityEventV1[], "nextCursor"?: string }`.

اجتماع نتیجه نام‌گذاری‌شده V1 برای اجرای عامل، کنش ابزار، پیام ورودی
و پیام خروجی طرح‌واره‌های جداگانه دارد. تفکیک‌گر `eventType` به‌ترتیب
`agent_run`، `tool_action`، `inbound_message` یا `outbound_message` است؛ `kind` و
`direction` پیام همچنان برای پالایش و نمایش در دسترس هستند. هر رویداد دارای
`schemaVersion: 1` صحیح است. ارجاع‌های هویت پیام از قالب دقیق
`hmac-sha256:v1:<32 hex key id>:<64 hex digest>` استفاده می‌کنند؛ شناسه کنشگر فرستنده کانال نیز
از همین قالب استفاده می‌کند.

همه گونه‌ها به `eventType`، `schemaVersion`، `eventId`، `sequence`،
`sourceSequence`، `occurredAt`، `kind`، `action`، `status`، `actor` و
`redaction` نیاز دارند. فیلدهای گونه‌ها عبارت‌اند از:

| `eventType`        | فیلدهای الزامی                                                   | فیلدهای اختیاری                                                                                                                 |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `agent_run`        | `agentId`، `runId`؛ `kind: "agent_run"`                           | `sessionKey`، `sessionId`، `errorCode`                                                                                          |
| `tool_action`      | `agentId`، `runId`؛ `kind: "tool_action"`                         | `sessionKey`، `sessionId`، `toolCallId`، `toolName`، `errorCode`                                                                |
| `inbound_message`  | `direction: "inbound"`، `channel`، `conversationKind`، `outcome`  | `agentId`، `runId`، `durationMs`، `resultCount`، ارجاع‌های هویت، `reasonCode`، `errorCode`                                 |
| `outbound_message` | `direction: "outbound"`، `channel`، `conversationKind`، `outcome` | `agentId`، `runId`، `durationMs`، `resultCount`، ارجاع‌های هویت، `reasonCode`، `deliveryKind`، `failureStage`، `errorCode` |

شمارش‌های بسته پیام عبارت‌اند از:

- `conversationKind`: `direct`، `group`، `channel` یا `unknown`.
- `outcome` ورودی: `completed`، `skipped` یا `failed`؛
  `reasonCode` اختیاری: `duplicate`، `reply_operation_active`،
  `reply_operation_aborted`، `fast_abort`، `plugin_bound_handled`،
  `plugin_bound_unavailable`، `plugin_bound_declined`، `plugin_bound_error`،
  `before_dispatch_handled`، `acp_dispatch_completed`، `acp_dispatch_failed`،
  `acp_dispatch_empty` یا `acp_dispatch_aborted`.
- `outcome` خروجی: `sent`، `suppressed`، `failed` یا `unknown`؛
  `reasonCode` اختیاری: `cancelled_by_message_sending_hook`،
  `cancelled_by_reply_payload_sending_hook`،
  `empty_after_message_sending_hook`، `empty_after_reply_payload_sending_hook`
  یا `no_visible_payload`. آداپتوری که هیچ هویت پلتفرمی برنمی‌گرداند
  `unknown` است، زیرا اثر جانبی خارجی را نمی‌توان رد کرد.
- `deliveryKind`: `text`، `media` یا `other`؛ `failureStage`:
  `platform_send`، `queue` یا `unknown`.

فیلدهای نهایی با یکدیگر هم‌بسته‌اند و به‌طور مستقل اختیاری نیستند:

| گونه          | نگاشت نهایی                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| اجرای عامل        | `started` هیچ `errorCode` ندارد؛ هر وضعیت پایان‌یافته ناموفق به کد متناظر `run_*` خود نیاز دارد.                                                                 |
| کنش ابزار      | `started` و موفق‌شده هیچ `errorCode` ندارند؛ هر وضعیت پایان‌یافته دیگر به کد متناظر `tool_*` خود نیاز دارد.                                                       |
| پیام ورودی  | موفق‌شده = `completed`؛ مسدودشده = `skipped`؛ شکست‌خورده = `failed` به‌همراه `message_processing_failed`. در صورت وجود، `reasonCode` باید متعلق به همان خانواده نهایی باشد. |
| پیام خروجی | موفق‌شده = `sent`؛ مسدودشده = `suppressed` به‌همراه `reasonCode`؛ شکست‌خورده = `failed` به‌همراه `errorCode` و `failureStage`؛ ناشناخته = `unknown` به‌همراه `failureStage`.      |

هر رویداد فعالیت شامل یک شناسهٔ پایدار رویداد، توالی یکنواخت دفترکل،
توالی رویداد مبدأ، برچسب زمانی، کنشگر، عمل، وضعیت، مقدار صحیح
`schemaVersion: 1` و `redaction: "metadata_only"` است. رکوردهای اجرا و ابزار
به منشأ عامل و اجرا نیاز دارند و ممکن است شامل منشأ نشست باشند. رکوردهای
پیام ممکن است شامل شناسه‌های عامل و اجرا باشند، اما عمداً هرگز شامل
`sessionKey` یا `sessionId` نیستند؛ بنابراین فیلتر پرس‌وجوی `sessionKey` فقط بر
ردیف‌های اجرا و ابزار اعمال می‌شود. رویدادهای ابزار ممکن است شامل شناسهٔ فراخوانی ابزار و نام ابزار باشند.

رکوردهای پیام از `message.inbound.processed` یا
`message.outbound.finished` استفاده می‌کنند و جهت، کانال، نوع مکالمه،
نتیجهٔ نرمال‌شده و در صورت وجود، نوع تحویل، مرحلهٔ شکست، مدت‌زمان،
تعداد نتایج، کد دلیل و نام‌های مستعار کلیددار محلیِ نصب برای
حساب/مکالمه/پیام/مقصد را می‌افزایند. این نام‌های مستعار به
هم‌بستگی کمک می‌کنند، اما ناشناس‌سازی نیستند: پایگاه دادهٔ وضعیت کلید آن‌ها را در خود دارد،
درحالی‌که خروجی‌های RPC و CLI آن را ندارند. دفترکل اعلان‌ها، بدنهٔ پیام‌ها،
آرگومان‌های ابزار، نتایج ابزار، خروجی فرمان یا متن خام خطا را ذخیره نمی‌کند.
مقادیر `sessionKey` اجرا/ابزار به‌صورت فرادادهٔ خام هم‌بستگی باقی می‌مانند و می‌توانند
شناسه‌های حساب پلتفرم یا همتا را در خود جای دهند؛ رکوردهای پیام کلیدهای نشست را حذف می‌کنند.

برای ردیف‌های ورودی، `durationMs` ارسال هسته تا پایان آن را اندازه‌گیری می‌کند و
`resultCount` تعداد محموله‌های نهایی‌شدهٔ ابزار، بلوک و پاسخِ صف‌شده را می‌شمارد. برای
ردیف‌های خروجی، `durationMs` بازهٔ مالکیت تحویل تا تأیید دریافت،
نامهٔ مرده یا تطبیق را پوشش می‌دهد (از جمله زمان انتظار در صف)، و `resultCount`
تعداد ارسال‌های فیزیکی شناسایی‌شده در پلتفرم را می‌شمارد. `deliveryKind`، در صورت وجود،
محمولهٔ مؤثر پس از قلاب‌ها و رندر را توصیف می‌کند؛ ردیف‌های سرکوب‌شده یا
دارای ابهام ناشی از خرابی، آن را حذف می‌کنند.

پوشش فعلی پیام شامل پیام‌های ورودی پذیرفته‌شده‌ای است که به ارسال هسته
می‌رسند، از جمله نتایج تکراری/نهایی هسته. پوشش خروجی برای هر محمولهٔ منطقی
اصلی پاسخ که به تحویل پایدار مشترک می‌رسد، یک ردیف نهایی می‌نویسد؛ قطعه‌بندی
و انشعاب آداپتور در `resultCount` تجمیع می‌شوند. ارسال‌های صف‌شدهٔ
قابل‌تلاش مجدد یا مبهم فقط پس از تأیید دریافت، نامهٔ مرده یا تطبیق ثبت
می‌شوند. مسیرهای محلی Plugin و ارسال مستقیم که این مرزهای مشترک را دور
می‌زنند، هنوز پوشش داده نمی‌شوند. صف محدود کارگر به‌صورت بهترین‌تلاش عمل
می‌کند و ممکن است هنگام شکست یا اشباع، رکوردها را کنار بگذارد؛ بنابراین این
سطح یک بایگانی انطباق بدون اتلاف نیست.

ثبت به‌طور پیش‌فرض فعال است و با
[`audit.enabled`](/fa/gateway/configuration-reference#audit) کنترل می‌شود. ثبت پیام
به‌صورت جداگانه با `audit.messages` کنترل می‌شود و مقدار پیش‌فرض آن `"off"` است. وقتی
ثبت غیرفعال باشد، `audit.activity.list` تا زمان انقضای رکوردهای قبلاً نوشته‌شده
به ارائهٔ آن‌ها ادامه می‌دهد.

شِماهای عرضه‌شدهٔ درخواست، نتیجه و `AuditEvent` مربوط به `audit.list`
بدون تغییر باقی مانده‌اند و فقط رکوردهای اجرای عامل و عمل ابزار را برمی‌گردانند. کلاینت‌های
جدید اپراتور باید هنگامی که Gateway آن را اعلام می‌کند، `audit.activity.list` را فراخوانی کنند. Gatewayهای
قدیمی‌تر ممکن است به یک درخواست با دامنهٔ خواندن، یا `unknown method: audit.activity.list` یا، چون
در نسخه‌های عرضه‌شده مجوزدهی پیش از جست‌وجوی متد انجام می‌شد، `missing scope:
operator.admin` را گزارش کنند. مورد دوم را تنها زمانی نبود متد
در نظر بگیرید که متد اعلام نشده باشد. سپس کلاینت فقط زمانی می‌تواند `audit.list`
را دوباره امتحان کند که فیلترهایش به پشتیبانی از نوع پیام، جهت یا کانال
نیاز نداشته باشند.

برای پرس‌وجوهای متنی و خروجی‌های محدود JSON از [`openclaw audit`](/fa/cli/audit) استفاده کنید.

## RPCهای دفترکل وظایف

کلاینت‌های اپراتور از طریق RPCهای دفترکل وظایف (`packages/gateway-protocol/src/schema/tasks.ts`)
رکوردهای وظایف پس‌زمینهٔ Gateway را بررسی و لغو می‌کنند. این RPCها
خلاصه‌های پاک‌سازی‌شدهٔ وظایف را برمی‌گردانند، نه وضعیت خام زمان اجرا را.

- `tasks.list` به `operator.read` نیاز دارد.
  - پارامترها: `status` اختیاری (`"queued"`، `"running"`، `"completed"`،
    `"failed"`، `"cancelled"` یا `"timed_out"`) یا آرایه‌ای از این وضعیت‌ها،
    `agentId` اختیاری، `sessionKey` اختیاری، `limit` اختیاری از `1` تا
    `500` و رشتهٔ اختیاری `cursor`.
  - نتیجه: `{ "tasks": TaskSummary[], "nextCursor"?: string }`.
- `tasks.get` به `operator.read` نیاز دارد.
  - پارامترها: `{ "taskId": string }`.
  - نتیجه: `{ "task": TaskSummary }`.
  - شناسه‌های وظیفهٔ ناموجود، قالب خطای یافت‌نشدن Gateway را برمی‌گردانند.
- `tasks.cancel` به `operator.write` نیاز دارد.
  - پارامترها: `{ "taskId": string, "reason"?: string }`.
  - نتیجه: `{ "found": boolean, "cancelled": boolean, "reason"?: string, "task"?: TaskSummary }`.
  - `found` گزارش می‌کند که آیا دفترکل وظیفهٔ منطبقی داشته است. `cancelled`
    گزارش می‌کند که آیا زمان اجرا لغو را پذیرفته یا ثبت کرده است.

`TaskSummary` شامل `id`، `status` و فرادادهٔ اختیاری است: `kind`،
`runtime`، `title`، `agentId`، `sessionKey`، `childSessionKey`، `ownerKey`،
`runId`، `taskId`، `flowId`، `parentTaskId`، `sourceId`، برچسب‌های زمانی، پیشرفت،
خلاصهٔ نهایی و متن پاک‌سازی‌شدهٔ خطا. `agentId` عاملی را شناسایی می‌کند که
وظیفه را اجرا می‌کند؛ `sessionKey` و `ownerKey` زمینهٔ درخواست‌کننده و کنترل را
حفظ می‌کنند.

## متدهای کمکی اپراتور

- `commands.list` (`operator.read`) فهرست فرمان‌های زمان اجرا را برای
  یک عامل دریافت می‌کند.
  - `agentId` اختیاری است؛ برای خواندن فضای کاری عامل پیش‌فرض، آن را حذف کنید.
  - `scope` کنترل می‌کند که `name` اصلی کدام سطح را هدف بگیرد: `text`
    توکن اصلی فرمان متنی را بدون `/` ابتدایی برمی‌گرداند؛ `native` و
    مسیر پیش‌فرض `both` در صورت وجود، نام‌های بومیِ آگاه از ارائه‌دهنده را برمی‌گردانند.
  - `textAliases` نام‌های مستعار دقیق اسلش مانند `/model` و `/m` را حمل می‌کند.
  - `nativeName` در صورت وجود، نام بومی فرمانِ آگاه از ارائه‌دهنده را حمل
    می‌کند.
  - `provider` اختیاری است و فقط بر نام‌گذاری بومی و دسترس‌پذیری فرمان بومی
    Plugin اثر می‌گذارد.
  - `includeArgs=false` فرادادهٔ سریال‌شدهٔ آرگومان را از پاسخ حذف می‌کند.
- `tools.catalog` (`operator.read`) کاتالوگ ابزار زمان اجرا را برای یک
  عامل دریافت می‌کند. پاسخ شامل ابزارهای گروه‌بندی‌شده و فرادادهٔ منشأ است:
  - `source`: `core` یا `plugin`
  - `pluginId`: مالک Plugin در صورت `source="plugin"`
  - `optional`: اینکه آیا ابزار Plugin اختیاری است
- `tools.effective` (`operator.read`) فهرست مؤثر ابزارهای زمان اجرا را
  برای یک نشست دریافت می‌کند.
  - `sessionKey` الزامی است.
  - Gateway به‌جای پذیرش زمینهٔ احراز هویت یا تحویلِ ارائه‌شده از سوی فراخواننده،
    زمینهٔ مورد اعتماد زمان اجرا را در سمت سرور از نشست استخراج می‌کند.
  - پاسخ، نمایشی مشتق‌شده از سرور و محدود به نشست از فهرست فعال
    است که ابزارهای هسته، Plugin، کانال و سرور MCP ازپیش‌کشف‌شده را شامل می‌شود.
  - `tools.effective` برای MCP فقط‌خواندنی است: ممکن است کاتالوگ MCP یک نشست گرم را
    از طریق سیاست نهایی ابزار نمایش دهد، اما زمان‌های اجرای MCP را ایجاد نمی‌کند،
    ترابردها را متصل نمی‌کند و `tools/list` صادر نمی‌کند. اگر هیچ کاتالوگ گرم منطبقی
    وجود نداشته باشد، پاسخ ممکن است شامل اعلانی مانند `mcp-not-yet-connected`،
    `mcp-not-yet-listed` یا `mcp-stale-catalog` باشد.
  - ورودی‌های مؤثر ابزار از `source="core"`، `source="plugin"`،
    `source="channel"` یا `source="mcp"` استفاده می‌کنند.
- `tools.invoke` (`operator.write`) یک ابزار در دسترس را از طریق همان
  مسیر سیاست Gateway که `/tools/invoke` استفاده می‌کند، فراخوانی می‌کند.
  - `name` الزامی است. `args`، `sessionKey`، `agentId`، `confirm` و
    `idempotencyKey` اختیاری هستند.
  - اگر هر دو `sessionKey` و `agentId` وجود داشته باشند، عامل نشست حل‌شده
    باید با `agentId` منطبق باشد.
  - پوشش‌دهنده‌های هستهٔ مختص مالک مانند `cron`، `gateway` و `nodes` به
    هویت مالک/مدیر (`operator.admin`) نیاز دارند، با اینکه خود `tools.invoke`
    برابر `operator.write` است.
  - پاسخ یک پوشش رو به SDK با `ok`، `toolName`، `output` اختیاری
    و فیلدهای نوع‌دار `error` است. رد شدن به‌علت تأیید یا سیاست،
    `ok:false` را در محموله برمی‌گرداند و خط‌لولهٔ سیاست ابزار Gateway را
    دور نمی‌زند.
- `skills.status` (`operator.read`) فهرست قابل‌مشاهدهٔ Skills را برای یک
  عامل دریافت می‌کند.
  - `agentId` اختیاری است؛ برای خواندن فضای کاری عامل پیش‌فرض، آن را حذف کنید.
  - پاسخ شامل واجدشرایط‌بودن، نیازمندی‌های مفقود، بررسی‌های پیکربندی
    و گزینه‌های پاک‌سازی‌شدهٔ نصب است، بدون آنکه مقادیر خام اسرار را افشا کند.
- `skills.search` و `skills.detail` (`operator.read`) فرادادهٔ کشف ClawHub را
  برمی‌گردانند.
- `skills.upload.begin`، `skills.upload.chunk` و `skills.upload.commit`
  (`operator.admin`) یک بایگانی خصوصی مهارت را پیش از نصب مرحله‌بندی می‌کنند. این
  یک مسیر بارگذاری مدیریتی جداگانه برای کلاینت‌های مورد اعتماد است، نه جریان عادی
  نصب مهارت ClawHub، و به‌طور پیش‌فرض غیرفعال است مگر اینکه
  `skills.install.allowUploadedArchives` فعال باشد.
  - `skills.upload.begin({ kind: "skill-archive", slug, sizeBytes, sha256?, force?, idempotencyKey? })`
    بارگذاری‌ای مقید به آن نامک و مقدار اجبار ایجاد می‌کند.
  - `skills.upload.chunk({ uploadId, offset, dataBase64 })` بایت‌ها را در
    آفست دقیق رمزگشایی‌شده اضافه می‌کند.
  - `skills.upload.commit({ uploadId, sha256? })` اندازهٔ نهایی و
    SHA-256 را تأیید می‌کند. ثبت فقط بارگذاری را نهایی می‌کند؛ مهارت را نصب نمی‌کند.
  - بایگانی‌های بارگذاری‌شدهٔ مهارت، بایگانی‌های zip حاوی ریشهٔ `SKILL.md` هستند. نام
    پوشهٔ داخلی بایگانی هرگز مقصد نصب را انتخاب نمی‌کند.
- `skills.install` (`operator.admin`) سه حالت دارد:
  - حالت ClawHub: `{ source: "clawhub", slug, version?, force? }` یک
    پوشهٔ مهارت را در پوشهٔ `skills/` فضای کاری عامل پیش‌فرض نصب می‌کند.
  - حالت بارگذاری: `{ source: "upload", uploadId, slug, force?, sha256?, timeoutMs? }`
    یک بارگذاری ثبت‌شده را در پوشهٔ `skills/<slug>`
    فضای کاری عامل پیش‌فرض نصب می‌کند. نامک و مقدار اجبار باید با
    درخواست اصلی `skills.upload.begin` منطبق باشند. مگر اینکه
    `skills.install.allowUploadedArchives` فعال باشد، رد می‌شود؛ این تنظیم بر نصب‌های ClawHub
    اثر نمی‌گذارد.
  - حالت نصب‌کنندهٔ Gateway: `{ name, installId, timeoutMs? }` یک عمل اعلام‌شدهٔ
    `metadata.openclaw.install` را روی میزبان Gateway اجرا می‌کند. کلاینت‌های قدیمی‌تر ممکن است
    همچنان `dangerouslyForceUnsafeInstall` را بفرستند؛ این فیلد منسوخ شده،
    فقط برای سازگاری پروتکل پذیرفته می‌شود و نادیده گرفته می‌شود. برای
    تصمیم‌های نصب تحت مالکیت اپراتور از `security.installPolicy` استفاده کنید.
- `skills.update` (`operator.admin`) دو حالت دارد:
  - حالت ClawHub یک نامک رهگیری‌شده یا همهٔ نصب‌های رهگیری‌شدهٔ ClawHub را در
    فضای کاری عامل پیش‌فرض به‌روزرسانی می‌کند.
  - حالت پیکربندی مقادیر `skills.entries.<skillKey>` مانند `enabled`،
    `apiKey` و `env` را وصله می‌کند.

### نماهای `models.list`

`models.list` یک پارامتر اختیاری `view` را می‌پذیرد
(`src/agents/model-catalog-visibility.ts`):

- حذف‌شده یا `"default"`: اگر `agents.defaults.models` پیکربندی شده باشد،
  پاسخ، کاتالوگ مجاز است که مدل‌های کشف‌شده به‌صورت پویا
  برای ورودی‌های `provider/*` را نیز شامل می‌شود. در غیر این صورت، پاسخ کاتالوگ کامل Gateway
  است.
- `"configured"`: رفتار متناسب با انتخابگر. اگر `agents.defaults.models`
  پیکربندی شده باشد، همچنان اولویت دارد، از جمله کشف محدود به ارائه‌دهنده برای
  ورودی‌های `provider/*`. بدون فهرست مجاز، پاسخ از ورودی‌های صریح
  `models.providers.<provider>.models` استفاده می‌کند و تنها زمانی به کاتالوگ کامل
  بازمی‌گردد که هیچ ردیف مدل پیکربندی‌شده‌ای وجود نداشته باشد.
- `"provider-config"`: موجودی `models.providers.*.models` تعریف‌شده در منبع،
  مستقل از فهرست‌های مجاز انتخابگر. ردیف‌ها قابلیت‌های عمومی مدل و
  دسترس‌پذیری آگاه از مسیر را شامل می‌شوند، اما نقاط پایانی ارائه‌دهنده، اطلاعات احراز هویت و
  پیکربندی درخواست زمان اجرا را حذف می‌کنند.
- `"all"`: کاتالوگ کامل Gateway، با دور زدن `agents.defaults.models`. برای
  رابط‌های کاربری عیب‌یابی/کشف استفاده کنید، نه انتخابگرهای عادی مدل.

## تأییدهای اجرا

- وقتی یک درخواست اجرا به تأیید نیاز دارد، Gateway
  `exec.approval.requested` را پخش می‌کند.
- کلاینت‌های اپراتور با فراخوانی `exec.approval.resolve` آن را تعیین‌تکلیف می‌کنند (نیازمند
  `operator.approvals`).
- برای `host=node`، `exec.approval.request` باید شامل `systemRunPlan`
  باشد (فراداده متعارف `argv`/`cwd`/`rawCommand`/نشست). درخواست‌هایی که
  `systemRunPlan` را نداشته باشند رد می‌شوند.
- پس از تأیید، فراخوانی‌های هدایت‌شده `node.invoke system.run` همان
  `systemRunPlan` متعارف را به‌عنوان بافت معتبر فرمان/cwd/نشست دوباره استفاده می‌کنند.
- اگر فراخواننده بین آماده‌سازی و هدایت نهایی و تأییدشده `system.run`،
  `command`، `rawCommand`، `cwd`، `agentId` یا
  `sessionKey` را تغییر دهد، Gateway به‌جای اعتماد به بار دادهٔ تغییریافته، اجرا را رد می‌کند.

## بازگشت جایگزین تحویل عامل

- درخواست‌های `agent` می‌توانند برای درخواست تحویل خروجی شامل `deliver=true` باشند.
- `bestEffortDeliver=false` (مقدار پیش‌فرض) رفتار سخت‌گیرانه را حفظ می‌کند: مقصدهای تحویل
  حل‌نشده یا صرفاً داخلی، `INVALID_REQUEST` را برمی‌گردانند.
- `bestEffortDeliver=true` هنگامی که هیچ مسیر خارجی قابل تحویلی
  قابل حل نباشد، بازگشت به اجرای صرفاً درون‌نشستی را مجاز می‌کند (برای مثال نشست‌های داخلی/وب‌چت
  یا پیکربندی‌های چندکانالهٔ مبهم).
- نتایج نهایی `agent` ممکن است هنگام درخواست تحویل شامل `result.deliveryStatus` باشند
  و از همان وضعیت‌های `sent`، `suppressed`، `partial_failed` و
  `failed` مستندشده برای
  [`openclaw agent --json --deliver`](/fa/cli/agent#json-delivery-status) استفاده کنند.

## نسخه‌بندی

- `PROTOCOL_VERSION`، `MIN_CLIENT_PROTOCOL_VERSION`،
  `MIN_NODE_PROTOCOL_VERSION` و `MIN_PROBE_PROTOCOL_VERSION` در
  `packages/gateway-protocol/src/version.ts` قرار دارند.
- کلاینت‌ها `minProtocol` + `maxProtocol` را ارسال می‌کنند. کلاینت‌های اپراتور و رابط کاربری باید
  پروتکل فعلی را در آن بازه بگنجانند؛ کلاینت‌ها و سرورهای فعلی از
  پروتکل v4 استفاده می‌کنند.
- کلاینت‌های احراز هویت‌شده‌ای که هم `role: "node"` و هم `client.mode: "node"`
  را دارند، می‌توانند از پروتکل Node نسخهٔ N-1 (در حال حاضر v3) استفاده کنند. کاوشگرهای سبک راه‌اندازی مجدد نیز از
  همان بازهٔ N-1 استفاده می‌کنند. احراز هویت دستگاه، جفت‌سازی، دامنه‌ها، خط‌مشی فرمان و تأییدهای
  اجرا تحت تأثیر این بازهٔ سازگاری قرار نمی‌گیرند. قابلیت‌ها و فرمان‌های Node
  متعلق به Plugin تا زمانی که Node به پروتکل فعلی ارتقا یابد ارائه نمی‌شوند،
  زیرا سطوح میزبانی‌شدهٔ آن‌ها بخشی از قرارداد N-1 نیستند.
- طرحواره‌ها و مدل‌ها از تعریف‌های TypeBox تولید می‌شوند:
  - `pnpm protocol:gen`
  - `pnpm protocol:gen:swift`
  - `pnpm protocol:check`

پیاده‌سازی مرجع کلاینت در `packages/gateway-client/src/` قرار دارد
(OpenClaw آن را از طریق نمای نازک `src/gateway/client.ts` پوشش می‌دهد). این
مقادیر پیش‌فرض در سراسر پروتکل v4 پایدارند و خط مبنای مورد انتظار برای
کلاینت‌های شخص ثالث هستند.

| ثابت                                      | پیش‌فرض                                               | منبع                                                                                                                      |
| ----------------------------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `PROTOCOL_VERSION`                        | `4`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| `MIN_CLIENT_PROTOCOL_VERSION`             | `4`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| `MIN_NODE_PROTOCOL_VERSION`               | `3`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| `MIN_PROBE_PROTOCOL_VERSION`              | `3`                                                   | `packages/gateway-protocol/src/version.ts`                                                                                |
| مهلت درخواست (برای هر RPC)                | `30_000` میلی‌ثانیه                                    | `packages/gateway-client/src/client.ts` (`requestTimeoutMs`)                                                              |
| مهلت پیش‌احراز هویت / چالش اتصال          | `15_000` میلی‌ثانیه                                    | `packages/gateway-client/src/timeouts.ts` (متغیر محیطی `OPENCLAW_HANDSHAKE_TIMEOUT_MS` می‌تواند بودجهٔ سرور/کلاینت جفت‌شده را افزایش دهد) |
| تأخیر اولیهٔ اتصال مجدد                   | `1_000` میلی‌ثانیه                                     | `packages/gateway-client/src/client.ts` (`GATEWAY_RECONNECT_POLICY`)                                                      |
| حداکثر تأخیر اتصال مجدد                   | `30_000` میلی‌ثانیه                                    | `packages/gateway-client/src/client.ts` (`GATEWAY_RECONNECT_POLICY`)                                                      |
| محدودیت تلاش مجدد سریع پس از بسته‌شدن توکن دستگاه | `250` میلی‌ثانیه                            | `packages/gateway-client/src/client.ts`                                                                                   |
| مهلت توقف اجباری پیش از `terminate()`     | `250` میلی‌ثانیه                                     | `FORCE_STOP_TERMINATE_GRACE_MS`                                                                                           |
| مهلت پیش‌فرض `stopAndWait()`           | `1_000` میلی‌ثانیه                                     | `STOP_AND_WAIT_TIMEOUT_MS`                                                                                                |
| فاصلهٔ پیش‌فرض تیک (پیش از `hello-ok`)    | `30_000` میلی‌ثانیه                                    | `packages/gateway-client/src/client.ts`                                                                                   |
| بسته‌شدن بر اثر پایان مهلت تیک            | کد `4000` وقتی سکوت از `tickIntervalMs * 2` فراتر رود | `packages/gateway-client/src/client.ts`                                                                                   |
| `MAX_PAYLOAD_BYTES`                       | `25 * 1024 * 1024` (25 MB)                            | `src/gateway/server-constants.ts`                                                                                         |

سرور مقادیر مؤثر `policy.tickIntervalMs`،
`policy.maxPayload` و `policy.maxBufferedBytes` را در `hello-ok` اعلام می‌کند؛ کلاینت‌ها
باید به‌جای مقادیر پیش‌فرض پیش از دست‌دهی، از این مقادیر پیروی کنند.

کلاینت مرجع هنگامی که هر درخواست در انتظار مهلت مشخصی دارد، اجازه می‌دهد
درخواست‌های محدود مالک ضرب‌الاجل پیکربندی‌شدهٔ خود باشند. یک درخواست `expectFinal` بدون
`timeoutMs` محدود، هر درخواستی با `timeoutMs: null`، یا ترکیبی از درخواست‌های محدود و
نامحدود، ناظر تیک را فعال نگه می‌دارد. اگر رویدادها و
پاسخ‌های ورودی تا پس از آستانهٔ پایان مهلت تیک همچنان ساکت بمانند، کلاینت
سوکت را با کد `4000` می‌بندد، همهٔ درخواست‌های در انتظار را رد می‌کند و دوباره متصل می‌شود. پس از
اتصال مجدد، درخواست‌های ردشده را دوباره اجرا نمی‌کند.

## احراز هویت

- احراز هویت Gateway با راز مشترک، بسته به
  `gateway.auth.mode` پیکربندی‌شده (`"none" | "token" | "password" | "trusted-proxy"`)، از `connect.params.auth.token` یا
  `connect.params.auth.password` استفاده می‌کند.
- حالت‌های دارای هویت مانند Tailscale Serve (`gateway.auth.allowTailscale: true`)
  یا `gateway.auth.mode: "trusted-proxy"` غیرِ loopback، بررسی احراز هویت اتصال را به‌جای
  `connect.params.auth.*` از طریق سرآیندهای درخواست برآورده می‌کنند.
- `gateway.auth.mode: "none"` با ورودی خصوصی، احراز هویت اتصال با راز مشترک را
  کاملاً نادیده می‌گیرد؛ این حالت را روی ورودی عمومی/غیرقابل‌اعتماد در معرض دسترس قرار ندهید.
- پس از جفت‌سازی، Gateway یک توکن دستگاه با دامنه محدود به نقش اتصال
  و دامنه‌ها صادر می‌کند که در `hello-ok.auth.deviceToken` بازگردانده می‌شود. کلاینت‌ها باید
  آن را پس از هر اتصال موفق ذخیره کنند.
- اتصال مجدد با آن توکن دستگاه ذخیره‌شده باید مجموعه دامنه تأییدشده
  ذخیره‌شده برای همان توکن را نیز دوباره استفاده کند. این کار دسترسی خواندن/کاوش/وضعیت
  را که قبلاً اعطا شده حفظ می‌کند و مانع از آن می‌شود که اتصال‌های مجدد بی‌سروصدا
  به دامنه ضمنی محدودتر و مخصوص مدیر تقلیل یابند.
- ساخت احراز هویت اتصال در سمت کلاینت (`selectConnectAuth` در
  `packages/gateway-client/src/client.ts`):
  - `auth.password` مستقل است و در صورت تنظیم، همیشه ارسال می‌شود.
  - `auth.token` به‌ترتیب اولویت مقداردهی می‌شود: ابتدا توکن مشترک صریح،
    سپس یک `deviceToken` صریح، و پس از آن توکن ذخیره‌شده مختص هر دستگاه (با کلید
    `deviceId` + `role`).
  - `auth.bootstrapToken` تنها زمانی ارسال می‌شود که هیچ‌یک از موارد بالا
    `auth.token` را تعیین نکرده باشند. توکن مشترک یا هر توکن دستگاه تعیین‌شده‌ای
    آن را سرکوب می‌کند.
  - ارتقای خودکار یک توکن دستگاه ذخیره‌شده در تلاش مجدد یک‌باره
    `AUTH_TOKEN_MISMATCH` فقط برای نقاط پایانی قابل‌اعتماد مجاز است: loopback،
    یا `wss://` همراه با `tlsFingerprint` سنجاق‌شده. `wss://` عمومی بدون سنجاق‌کردن
    واجد شرایط نیست.
- راه‌اندازی اولیه داخلی با کد راه‌اندازی، `hello-ok.auth.deviceToken` مربوط به Node اصلی
  را به‌همراه یک توکن اپراتور محدودشده در
  `hello-ok.auth.deviceTokens` برای تحویل امن به دستگاه همراه بازمی‌گرداند. توکن اپراتور
  شامل `operator.talk.secrets` برای خواندن پیکربندی بومی Talk است، اما
  دامنه‌های تغییر جفت‌سازی و `operator.admin` را شامل نمی‌شود.
- هنگامی که راه‌اندازی اولیه با کد راه‌اندازی غیربنیادی منتظر تأیید است،
  جزئیات `PAIRING_REQUIRED` شامل `recommendedNextStep: "wait_then_retry"`،
  `retryable: true` و `pauseReconnect: false` است. اتصال مجدد با همان
  توکن راه‌اندازی اولیه را تا زمان تأیید درخواست یا نامعتبر شدن توکن
  ادامه دهید.
- `hello-ok.auth.deviceTokens` را تنها زمانی ذخیره کنید که اتصال از احراز هویت راه‌اندازی اولیه
  روی یک انتقال قابل‌اعتماد مانند `wss://` یا جفت‌سازی loopback/محلی استفاده کرده باشد.
- اگر کلاینت یک `deviceToken` صریح یا `scopes` صریح ارائه کند، همان
  مجموعه دامنه درخواستی فراخواننده مرجع نهایی باقی می‌ماند؛ دامنه‌های حافظه نهان تنها
  زمانی دوباره استفاده می‌شوند که کلاینت توکن ذخیره‌شده مختص هر دستگاه را دوباره به‌کار می‌گیرد.
- توکن‌های دستگاه را می‌توان از طریق `device.token.rotate` و
  `device.token.revoke` چرخاند/لغو کرد (نیازمند `operator.pairing`). چرخاندن یا لغو توکن
  یک Node یا نقش غیر‌اپراتوری دیگر، به `operator.admin` نیز نیاز دارد.
- `device.token.rotate` فراداده چرخش را بازمی‌گرداند. این متد، توکن حامل جایگزین را
  تنها برای فراخوانی‌های همان دستگاه که از قبل با همان توکن دستگاه احراز هویت شده‌اند
  بازتاب می‌دهد تا کلاینت‌های صرفاً متکی به توکن بتوانند پیش از اتصال مجدد، جایگزین خود را ذخیره کنند.
  چرخش‌های مشترک/مدیریتی، توکن حامل را بازتاب نمی‌دهند.
- صدور، چرخش و لغو توکن به مجموعه نقش تأییدشده ثبت‌شده در ورودی جفت‌سازی
  همان دستگاه محدود می‌ماند؛ تغییر توکن نمی‌تواند نقش دستگاهی را گسترش دهد یا هدف بگیرد
  که تأیید جفت‌سازی هرگز آن را اعطا نکرده است.
- برای نشست‌های توکن دستگاه جفت‌شده، مدیریت دستگاه به خود دستگاه محدود است، مگر اینکه
  فراخواننده `operator.admin` را نیز داشته باشد: فراخوانندگان غیرمدیر فقط می‌توانند
  توکن اپراتور مربوط به ورودی دستگاه خود را مدیریت کنند. مدیریت توکن Node و سایر نقش‌های
  غیر‌اپراتوری، حتی برای دستگاه خود فراخواننده، فقط در اختیار مدیر است.
- `device.token.rotate` و `device.token.revoke` مجموعه دامنه توکن اپراتور
  هدف را نیز در برابر دامنه‌های نشست کنونی فراخواننده بررسی می‌کنند.
  فراخوانندگان غیرمدیر نمی‌توانند توکن اپراتوری با دامنه گسترده‌تر از آنچه خود
  در اختیار دارند بچرخانند یا لغو کنند.
- خطاهای احراز هویت شامل `error.details.code` به‌همراه راهنمای بازیابی هستند:
  - `error.details.canRetryWithDeviceToken` (بولی)
  - `error.details.recommendedNextStep`: یکی از `retry_with_device_token`،
    `update_auth_configuration`، `update_auth_credentials`،
    `wait_then_retry`، `review_auth_configuration`
    (`packages/gateway-protocol/src/connect-error-details.ts`).
- رفتار کلاینت برای `AUTH_TOKEN_MISMATCH`:
  - کلاینت‌های قابل‌اعتماد می‌توانند یک تلاش مجدد محدود با توکن
    مختص هر دستگاه از حافظه نهان انجام دهند.
  - اگر آن تلاش مجدد ناموفق بود، حلقه‌های اتصال مجدد خودکار را متوقف کنید و
    راهنمای اقدام اپراتور را نمایش دهید.
- `AUTH_SCOPE_MISMATCH` یعنی توکن دستگاه شناسایی شده، اما
  نقش/دامنه‌های درخواستی را پوشش نمی‌دهد. این وضعیت را به‌عنوان توکن نامعتبر نمایش ندهید؛
  از اپراتور بخواهید دوباره جفت‌سازی کند یا قرارداد دامنه محدودتر/گسترده‌تر را تأیید کند.

## هویت دستگاه و جفت‌سازی

- Nodeها باید یک هویت پایدار دستگاه (`device.id`) برگرفته از
  اثر انگشت جفت‌کلید ارائه کنند.
- Gatewayها برای هر دستگاه + نقش توکن صادر می‌کنند.
- برای شناسه‌های دستگاه جدید، تأیید جفت‌سازی الزامی است، مگر اینکه
  تأیید خودکار محلی فعال باشد.
- تأیید خودکار جفت‌سازی بر اتصال‌های مستقیم loopback محلی متمرکز است.
- OpenClaw همچنین برای جریان‌های کمکی قابل‌اعتماد با راز مشترک، یک مسیر محدود
  اتصال به خود در backend/container محلی دارد.
- اتصال‌های tailnet یا LAN روی همان میزبان همچنان برای جفت‌سازی
  راه‌دور محسوب می‌شوند و به تأیید نیاز دارند.
- کلاینت‌های WS معمولاً هنگام `connect` هویت `device` را ارائه می‌کنند (اپراتور +
  Node). تنها استثناهای اپراتور بدون دستگاه، مسیرهای اعتماد صریح زیر هستند:
  - `gateway.controlUi.allowInsecureAuth=true` برای سازگاری HTTP ناامن
    و محدود به localhost.
  - احراز هویت موفق اپراتور در Control UI با `gateway.auth.mode: "trusted-proxy"`.
  - `gateway.controlUi.dangerouslyDisableDeviceAuth=true` (راهکار اضطراری، کاهش شدید
    امنیت).
  - RPCهای backend در `gateway-client` با loopback مستقیم، روی مسیر داخلی
    رزروشده ابزار کمکی.
- حذف هویت دستگاه بر دامنه‌ها پیامد دارد. وقتی اتصال اپراتور بدون دستگاه
  از طریق یک مسیر اعتماد صریح مجاز شود، OpenClaw همچنان دامنه‌های خوداظهاری را
  به مجموعه‌ای خالی پاک می‌کند، مگر اینکه آن مسیر استثنای نام‌گذاری‌شده‌ای برای
  حفظ دامنه داشته باشد. سپس متدهای محدودشده بر اساس دامنه با
  `missing scope` ناموفق می‌شوند.
- `gateway.controlUi.dangerouslyDisableDeviceAuth=true` یک مسیر اضطراری Control UI
  برای حفظ دامنه است. این مسیر به کلاینت‌های WebSocket سفارشی دلخواه با شکل backend یا CLI
  دامنه اعطا نمی‌کند.
- مسیر رزروشده ابزار کمکی backend با loopback مستقیم در `gateway-client`،
  دامنه‌ها را فقط برای RPCهای داخلی و محلی صفحه کنترل حفظ می‌کند؛ شناسه‌های backend سفارشی
  از این استثنا برخوردار نمی‌شوند.
- همه اتصال‌ها باید nonce ارائه‌شده توسط سرور در `connect.challenge` را امضا کنند.

### عیب‌یابی مهاجرت احراز هویت دستگاه

برای کلاینت‌های قدیمی که هنوز از رفتار امضای پیش از challenge استفاده می‌کنند، `connect`
کدهای جزئیات `DEVICE_AUTH_*` را زیر `error.details.code` با یک
`error.details.reason` پایدار بازمی‌گرداند.

خطاهای رایج مهاجرت:

| پیام                     | details.code                     | details.reason           | مفهوم                                            |
| --------------------------- | -------------------------------- | ------------------------ | -------------------------------------------------- |
| `device nonce required`     | `DEVICE_AUTH_NONCE_REQUIRED`     | `device-nonce-missing`   | کلاینت `device.nonce` را حذف کرده است (یا خالی فرستاده است).     |
| `device nonce mismatch`     | `DEVICE_AUTH_NONCE_MISMATCH`     | `device-nonce-mismatch`  | کلاینت با nonce منقضی/نادرست امضا کرده است.            |
| `device signature invalid`  | `DEVICE_AUTH_SIGNATURE_INVALID`  | `device-signature`       | بار امضا با بار v2 مطابقت ندارد.       |
| `device signature expired`  | `DEVICE_AUTH_SIGNATURE_EXPIRED`  | `device-signature-stale` | مهر زمانی امضاشده خارج از انحراف مجاز است.          |
| `device identity mismatch`  | `DEVICE_AUTH_DEVICE_ID_MISMATCH` | `device-id-mismatch`     | `device.id` با اثر انگشت کلید عمومی مطابقت ندارد. |
| `device public key invalid` | `DEVICE_AUTH_PUBLIC_KEY_INVALID` | `device-public-key`      | قالب/متعارف‌سازی کلید عمومی ناموفق بود.         |

هدف مهاجرت:

- همیشه منتظر `connect.challenge` بمانید.
- بار v2 شامل nonce سرور را امضا کنید.
- همان nonce را در `connect.params.device.nonce` ارسال کنید.
- بار امضای ترجیحی `v3` است
  (`buildDeviceAuthPayloadV3` در `packages/gateway-client/src/device-auth.ts`) که علاوه بر
  فیلدهای دستگاه/کلاینت/نقش/دامنه‌ها/توکن/nonce، `platform` و `deviceFamily` را
  نیز مقید می‌کند.
- امضاهای قدیمی `v2` برای سازگاری همچنان پذیرفته می‌شوند، اما سنجاق‌کردن
  فراداده دستگاه جفت‌شده همچنان سیاست فرمان را هنگام اتصال مجدد کنترل می‌کند.

## TLS و سنجاق‌کردن

- TLS برای اتصال‌های WS پشتیبانی می‌شود (پیکربندی `gateway.tls`).
- کلاینت‌ها می‌توانند به‌صورت اختیاری اثر انگشت گواهی Gateway را از طریق
  `gateway.remote.tlsFingerprint` یا `--tls-fingerprint` در CLI سنجاق کنند.
