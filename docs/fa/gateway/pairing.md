در جفت‌سازی تحت مالکیت Gateway، **Gateway** مرجع حقیقت برای این است که کدام گره‌ها اجازه پیوستن دارند. رابط‌های کاربری (برنامه macOS، کلاینت‌های آینده) فقط فرانت‌اندهایی هستند که درخواست‌های در انتظار را تأیید یا رد می‌کنند.

**مهم:** گره‌های WS هنگام `connect` از **جفت‌سازی دستگاه** (نقش `node`) استفاده می‌کنند. `node.pair.*` یک مخزن جفت‌سازی جداگانه است و دست‌دهی WS را **کنترل نمی‌کند**. فقط کلاینت‌هایی که به‌طور صریح `node.pair.*` را فراخوانی می‌کنند از این جریان استفاده می‌کنند.

## مفاهیم

- **درخواست در انتظار**: یک گره درخواست پیوستن داده است؛ به تأیید نیاز دارد.
- **گره جفت‌شده**: گره تأییدشده با توکن احراز هویت صادرشده.
- **انتقال**: نقطه پایانی WS در Gateway درخواست‌ها را بازارسال می‌کند، اما درباره عضویت تصمیم نمی‌گیرد. (پشتیبانی از پل TCP قدیمی حذف شده است.)

## جفت‌سازی چگونه کار می‌کند

1. یک گره به WS در Gateway متصل می‌شود و درخواست جفت‌سازی می‌دهد.
2. Gateway یک **درخواست در انتظار** ذخیره می‌کند و `node.pair.requested` را منتشر می‌کند.
3. شما درخواست را تأیید یا رد می‌کنید (CLI یا رابط کاربری).
4. پس از تأیید، Gateway یک **توکن جدید** صادر می‌کند (توکن‌ها هنگام جفت‌سازی دوباره چرخش می‌کنند).
5. گره با استفاده از توکن دوباره متصل می‌شود و اکنون «جفت‌شده» است.

درخواست‌های در انتظار پس از **۵ دقیقه** به‌طور خودکار منقضی می‌شوند.

## گردش کار CLI (مناسب برای حالت بدون رابط گرافیکی)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` گره‌های جفت‌شده/متصل و قابلیت‌های آن‌ها را نشان می‌دهد.

## سطح API (پروتکل Gateway)

رویدادها:

- `node.pair.requested` — زمانی منتشر می‌شود که یک درخواست در انتظار جدید ایجاد شود.
- `node.pair.resolved` — زمانی منتشر می‌شود که یک درخواست تأیید/رد/منقضی شود.

متدها:

- `node.pair.request` — ایجاد یا استفاده دوباره از یک درخواست در انتظار.
- `node.pair.list` — فهرست کردن گره‌های در انتظار + جفت‌شده (`operator.pairing`).
- `node.pair.approve` — تأیید یک درخواست در انتظار (توکن صادر می‌کند).
- `node.pair.reject` — رد یک درخواست در انتظار.
- `node.pair.remove` — حذف یک ورودی گره جفت‌شده قدیمی.
- `node.pair.verify` — راستی‌آزمایی `{ nodeId, token }`.

نکته‌ها:

- `node.pair.request` برای هر گره idempotent است: فراخوانی‌های تکراری همان درخواست در انتظار را برمی‌گردانند.
- درخواست‌های تکراری برای همان گره در انتظار، فراداده ذخیره‌شده گره و تازه‌ترین snapshot فرمان‌های اعلام‌شده و مجازشده را نیز برای دیدپذیری اپراتور تازه‌سازی می‌کنند.
- تأیید **همیشه** یک توکن تازه تولید می‌کند؛ هیچ توکنی هرگز از `node.pair.request` برگردانده نمی‌شود.
- سطح‌های دامنه اپراتور و بررسی‌های زمان تأیید در [دامنه‌های اپراتور](/fa/gateway/operator-scopes) خلاصه شده‌اند.
- درخواست‌ها می‌توانند `silent: true` را به‌عنوان راهنمایی برای جریان‌های تأیید خودکار شامل شوند.
- `node.pair.approve` از فرمان‌های اعلام‌شده درخواست در انتظار برای اعمال دامنه‌های تأیید اضافی استفاده می‌کند:
  - درخواست بدون فرمان: `operator.pairing`
  - درخواست فرمان غیر اجرایی: `operator.pairing` + `operator.write`
  - درخواست `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
جفت‌سازی گره یک جریان اعتماد و هویت به‌همراه صدور توکن است. این جریان سطح فرمان زنده گره را برای هر گره پین نمی‌کند.

- فرمان‌های زنده گره از چیزی می‌آیند که گره پس از اعمال سیاست فرمان گره سراسری Gateway (`gateway.nodes.allowCommands` و `denyCommands`) هنگام اتصال اعلام می‌کند.
- سیاست مجازسازی و پرسش برای `system.run` به‌ازای هر گره، روی خود گره در `exec.approvals.node.*` قرار دارد، نه در رکورد جفت‌سازی.

</Warning>

## کنترل فرمان گره (2026.3.31+)

<Warning>
**تغییر ناسازگار:** از `2026.3.31`، فرمان‌های گره تا زمانی که جفت‌سازی گره تأیید نشود غیرفعال هستند. جفت‌سازی دستگاه به‌تنهایی دیگر برای آشکار کردن فرمان‌های اعلام‌شده گره کافی نیست.
</Warning>

وقتی یک گره برای اولین بار متصل می‌شود، جفت‌سازی به‌طور خودکار درخواست می‌شود. تا زمانی که درخواست جفت‌سازی تأیید نشود، همه فرمان‌های در انتظار گره از آن گره فیلتر می‌شوند و اجرا نخواهند شد. پس از اینکه اعتماد از طریق تأیید جفت‌سازی برقرار شد، فرمان‌های اعلام‌شده گره با رعایت سیاست فرمان عادی در دسترس قرار می‌گیرند.

این یعنی:

- گره‌هایی که پیش‌تر فقط به جفت‌سازی دستگاه برای آشکار کردن فرمان‌ها متکی بودند، اکنون باید جفت‌سازی گره را کامل کنند.
- فرمان‌هایی که پیش از تأیید جفت‌سازی در صف قرار گرفته‌اند حذف می‌شوند، نه اینکه به تعویق بیفتند.

## مرزهای اعتماد رویداد گره (2026.3.31+)

<Warning>
**تغییر ناسازگار:** اجراهای مبدأگرفته از گره اکنون روی سطح مورد اعتماد کاهش‌یافته باقی می‌مانند.
</Warning>

خلاصه‌های مبدأگرفته از گره و رویدادهای نشست مرتبط، به سطح مورد اعتماد مورد نظر محدود می‌شوند. جریان‌های مبتنی بر اعلان یا تحریک‌شده از سوی گره که پیش‌تر به دسترسی گسترده‌تر به ابزار میزبان یا نشست متکی بودند ممکن است به تنظیم نیاز داشته باشند. این سخت‌سازی تضمین می‌کند که رویدادهای گره نتوانند فراتر از آنچه مرز اعتماد گره اجازه می‌دهد به دسترسی ابزار در سطح میزبان ارتقا پیدا کنند.

به‌روزرسانی‌های پایدار حضور گره از همان مرز هویت پیروی می‌کنند. رویداد `node.presence.alive` فقط از نشست‌های دستگاه گره احراز هویت‌شده پذیرفته می‌شود و فقط زمانی فراداده جفت‌سازی را به‌روزرسانی می‌کند که هویت دستگاه/گره از قبل جفت شده باشد. مقدارهای خوداعلام‌شده `client.id` برای نوشتن وضعیت آخرین مشاهده کافی نیستند.

## تأیید خودکار (برنامه macOS)

برنامه macOS می‌تواند به‌صورت اختیاری زمانی برای **تأیید بی‌صدا** تلاش کند که:

- درخواست با `silent` علامت‌گذاری شده باشد، و
- برنامه بتواند اتصال SSH به میزبان Gateway را با همان کاربر راستی‌آزمایی کند.

اگر تأیید بی‌صدا ناموفق باشد، به اعلان عادی «تأیید/رد» بازمی‌گردد.

## تأیید خودکار دستگاه با CIDR مورد اعتماد

جفت‌سازی دستگاه WS برای `role: node` به‌طور پیش‌فرض دستی باقی می‌ماند. برای شبکه‌های خصوصی گره که Gateway از قبل مسیر شبکه را مورد اعتماد می‌داند، اپراتورها می‌توانند با CIDRهای صریح یا IPهای دقیق فعال‌سازی کنند:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

مرز امنیتی:

- وقتی `gateway.nodes.pairing.autoApproveCidrs` تنظیم نشده باشد غیرفعال است.
- هیچ حالت تأیید خودکار کلی برای LAN یا شبکه خصوصی وجود ندارد.
- فقط جفت‌سازی تازه دستگاه با `role: node` و بدون دامنه‌های درخواستی واجد شرایط است.
- اپراتور، مرورگر، Control UI و کلاینت‌های WebChat دستی باقی می‌مانند.
- ارتقاهای نقش، دامنه، فراداده و کلید عمومی دستی باقی می‌مانند.
- مسیرهای هدر پراکسی مورد اعتماد loopback هم‌میزبان واجد شرایط نیستند، چون این مسیر می‌تواند توسط فراخواننده‌های محلی جعل شود.

## تأیید خودکار ارتقای فراداده

وقتی دستگاهی که از قبل جفت شده است دوباره فقط با تغییرات فراداده غیرحساس متصل می‌شود (برای مثال، نام نمایشی یا راهنمایی‌های پلتفرم کلاینت)، OpenClaw آن را به‌عنوان `metadata-upgrade` در نظر می‌گیرد. تأیید خودکار بی‌صدا محدود است: فقط برای اتصال‌های دوباره محلی غیرمرورگری و مورد اعتماد اعمال می‌شود که از قبل مالکیت اعتبارنامه‌های محلی یا مشترک را اثبات کرده‌اند، از جمله اتصال‌های دوباره برنامه بومی هم‌میزبان پس از تغییرات فراداده نسخه سیستم‌عامل. کلاینت‌های مرورگر/Control UI و کلاینت‌های راه‌دور همچنان از جریان تأیید دوباره صریح استفاده می‌کنند. ارتقاهای دامنه (خواندن به نوشتن/مدیر) و تغییرات کلید عمومی واجد شرایط تأیید خودکار ارتقای فراداده **نیستند** — آن‌ها به‌صورت درخواست‌های تأیید دوباره صریح باقی می‌مانند.

## کمک‌کننده‌های جفت‌سازی QR

`/pair qr` payload جفت‌سازی را به‌صورت رسانه ساختاریافته رندر می‌کند تا کلاینت‌های موبایل و مرورگر بتوانند آن را مستقیم اسکن کنند.

حذف یک دستگاه همچنین هر درخواست جفت‌سازی در انتظار قدیمی برای آن شناسه دستگاه را پاک می‌کند، بنابراین `nodes pending` پس از لغو، ردیف‌های بی‌صاحب نشان نمی‌دهد.

## محلی‌بودن و هدرهای بازارسال‌شده

جفت‌سازی Gateway فقط زمانی یک اتصال را loopback در نظر می‌گیرد که هم سوکت خام و هم هر شاهد پراکسی بالادستی توافق داشته باشند. اگر درخواستی روی loopback برسد اما هدرهای `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto` را حمل کند که به یک مبدأ غیرمحلی اشاره می‌کنند، آن شاهد هدر بازارسال‌شده ادعای محلی‌بودن loopback را رد صلاحیت می‌کند. سپس مسیر جفت‌سازی به‌جای اینکه درخواست را بی‌صدا به‌عنوان اتصال هم‌میزبان در نظر بگیرد، به تأیید صریح نیاز دارد. برای قاعده معادل در احراز هویت اپراتور، [احراز هویت پراکسی مورد اعتماد](/fa/gateway/trusted-proxy-auth) را ببینید.

## ذخیره‌سازی (محلی، خصوصی)

وضعیت جفت‌سازی زیر دایرکتوری وضعیت Gateway ذخیره می‌شود (پیش‌فرض `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`
- `~/.openclaw/nodes/pending.json`

اگر `OPENCLAW_STATE_DIR` را override کنید، پوشه `nodes/` نیز همراه آن جابه‌جا می‌شود.

نکات امنیتی:

- توکن‌ها محرمانه‌اند؛ با `paired.json` به‌عنوان داده حساس رفتار کنید.
- چرخش یک توکن به تأیید دوباره نیاز دارد (یا حذف ورودی گره).

## رفتار انتقال

- انتقال **بدون وضعیت** است؛ عضویت را ذخیره نمی‌کند.
- اگر Gateway آفلاین باشد یا جفت‌سازی غیرفعال باشد، گره‌ها نمی‌توانند جفت شوند.
- اگر Gateway در حالت راه‌دور باشد، جفت‌سازی همچنان در برابر مخزن Gateway راه‌دور انجام می‌شود.
