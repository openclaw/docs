**هارنس عامل** اجراکنندهٔ سطح‌پایین یک نوبت آماده‌شدهٔ عامل OpenClaw است.
این نه ارائه‌دهندهٔ مدل است، نه کانال و نه رجیستری ابزار. برای مدل ذهنی کاربرمحور، به
[زمان‌اجراهای عامل](/fa/concepts/agent-runtimes) مراجعه کنید.

از این سطح فقط برای Pluginهای باندل‌شده یا بومیِ مورد اعتماد استفاده کنید. این قرارداد
هنوز آزمایشی است، زیرا نوع پارامترها عمداً بازتاب‌دهندهٔ اجراکنندهٔ تعبیه‌شدهٔ
فعلی هستند.

## چه زمانی از هارنس استفاده کنید

هنگامی یک هارنس عامل ثبت کنید که یک خانوادهٔ مدل، زمان‌اجرای بومیِ نشست
خود را دارد و انتقال معمول ارائه‌دهندهٔ OpenClaw انتزاع مناسبی نیست:

- یک سرور بومی عامل کدنویسی که مالک رشته‌ها و Compaction است
- یک CLI یا دیمن محلی که باید رویدادهای بومی برنامه‌ریزی/استدلال/ابزار را به‌صورت جریانی ارسال کند
- یک زمان‌اجرای مدل که علاوه بر رونوشت نشست OpenClaw، به شناسهٔ ازسرگیری
  مختص خود نیاز دارد

فقط برای افزودن یک API جدید LLM، هارنس ثبت **نکنید**. برای APIهای معمول مدل مبتنی بر HTTP یا
WebSocket، یک [Plugin ارائه‌دهنده](/fa/plugins/sdk-provider-plugins) بسازید.

## مواردی که همچنان تحت مالکیت هسته هستند

پیش از انتخاب هارنس، OpenClaw موارد زیر را از قبل تعیین کرده است:

- ارائه‌دهنده و مدل
- وضعیت احراز هویت زمان‌اجرا، مگر اینکه هارنس اعلام کند مالک راه‌اندازی اولیهٔ احراز هویت است
- سطح تفکر و بودجهٔ زمینه
- فایل رونوشت/نشست OpenClaw
- فضای کاری، جعبهٔ شنی و سیاست ابزار
- فراخوان‌های بازگشتی پاسخ کانال و فراخوان‌های بازگشتی جریان
- سیاست جایگزینی مدل و تعویض زندهٔ مدل

هارنس یک تلاش آماده‌شده را اجرا می‌کند؛ ارائه‌دهندگان را انتخاب نمی‌کند، تحویل
کانال را جایگزین نمی‌کند و مدل‌ها را بی‌سروصدا تغییر نمی‌دهد.

### راه‌اندازی اولیهٔ احراز هویت تحت مالکیت هارنس

به‌طور پیش‌فرض، هسته پیش از فراخوانی هارنس، اطلاعات احراز هویت ارائه‌دهنده را تعیین می‌کند. یک
هارنس مورد اعتماد که می‌تواند از طریق زمان‌اجرای بومی خود احراز هویت کند، ممکن است
`authBootstrap: "harness"` را در ثبت ایستای `AgentHarness` خود تنظیم کند. سپس هسته
راه‌اندازی اولیهٔ عمومی اطلاعات احراز هویت ارائه‌دهنده و خطای نبود اطلاعات احراز هویت را
برای هر تلاشی که آن هارنس مالکیتش را می‌پذیرد، رد می‌کند.

هسته همچنان در صورت وجود، یک پروفایل احراز هویت OpenClaw سازگار و صراحتاً انتخاب‌شده یا مرتب‌شده
و مخزن دارای محدودهٔ آن را ارسال می‌کند. هارنس باید پیش از صدور درخواست‌های مدل،
آن پروفایل یا اطلاعات احراز هویت بومی خود را تعیین کند، اسرار را به محدودهٔ
تلاش محدود نگه دارد و خطاهای احراز هویت قابل اقدام ارائه دهد. این قابلیت را روی
هارنسی که فقط گاهی مالک احراز هویت است تنظیم نکنید.

### مصنوعات تأییدشدهٔ زمان‌اجرای راه‌اندازی

هارنس محلی‌ای که می‌تواند برای راه‌اندازی اجرای نخست استنتاج فراهم کند، باید
پیاده‌سازی‌ای را که کاوش را تکمیل کرده است تصدیق کند. هنگامی که
`params.captureRuntimeArtifact` برابر با true است، یک
`result.runtimeArtifact` مات با شناسهٔ پایدار و اثر انگشت محتوا برگردانید. یک
قابلیت `runtimeArtifact.validate(...)` منطبق ثبت کنید که بدون
بارگذاری هارنسی دیگر یا اسکن Pluginهای نامرتبط، آن اتصال را دوباره بررسی کند.

ادامه‌های تأییدشدهٔ OpenClaw همچنین `params.expectedRuntimeArtifact` را ارسال می‌کنند.
هارنس باید آن را با پردازش بومی دقیقی که به دست آورده مقایسه کند و اگر متفاوت بودند،
پیش از آغاز یا ازسرگیری یک رشتهٔ بومی با خطا متوقف شود. نوبت‌های معمول عامل
هر دو فیلد را حذف می‌کنند، بنابراین هش‌کردن محتوا از مسیر داغ درخواست عادی
خارج می‌ماند. هارنس‌های راه‌دور/WebSocket پیش از مشارکت به
قرارداد تصدیق سرور نیاز دارند؛ رشتهٔ نسخه به‌تنهایی هویت مصنوع نیست.

تلاش آماده‌شده همچنین شامل `params.runtimePlan` است، یک بستهٔ سیاستی تحت مالکیت
OpenClaw برای تصمیم‌های زمان‌اجرا که باید میان OpenClaw و
هارنس‌های بومی مشترک بمانند:

- `runtimePlan.tools.normalize(...)` و `runtimePlan.tools.logDiagnostics(...)`
  برای سیاست طرح‌وارهٔ ابزار آگاه از ارائه‌دهنده
- `runtimePlan.transcript.resolvePolicy(...)` برای پاک‌سازی رونوشت و
  سیاست ترمیم فراخوانی ابزار
- `runtimePlan.delivery.isSilentPayload(...)` برای `NO_REPLY` مشترک و جلوگیری
  از تحویل رسانه
- `runtimePlan.outcome.classifyRunResult(...)` برای طبقه‌بندی
  جایگزینی مدل
- `runtimePlan.observability` برای فرادادهٔ تعیین‌شدهٔ ارائه‌دهنده/مدل/هارنس

هارنس‌ها می‌توانند برای تصمیم‌هایی که باید با رفتار OpenClaw مطابقت داشته باشند از برنامه استفاده کنند،
اما با آن مانند وضعیت تلاش تحت مالکیت میزبان رفتار کنند: آن را تغییر ندهند و برای تعویض
ارائه‌دهنده‌ها/مدل‌ها درون یک نوبت استفاده نکنند.

### قرارداد انتقال درخواست

`supports(ctx)` انتقال تعیین‌شدهٔ مدل را در `ctx.modelProvider` دریافت می‌کند.
دو واقعیت بدون راز و تحت مالکیت ارائه‌دهنده، مسیر انتخاب‌شده را توصیف می‌کنند:

- `runtimePolicy.compatibleIds` شناسه‌های زمان‌اجرایی را فهرست می‌کند که ارائه‌دهنده
  با آن مسیر مشخص سازگار اعلام می‌کند. نبود سیاست یعنی ارائه‌دهنده
  سازگاری در سطح مسیر را اعلام نکرده است؛ این اجازه‌ای برای فرض پشتیبانی نیست.
- `requestTransportOverrides: "none"` یعنی هیچ بازنویسی تألیفی درخواست
  ارائه‌دهنده/مدل نباید بازتولید شود. `"present"` یعنی سرآیندهای تألیفی، انتقال
  احراز هویت، پراکسی، TLS، رفتار سرویس محلی، شبکهٔ خصوصی یا پارامترهای
  درخواست وجود دارند. این واقعیت آن مقادیر را افشا نمی‌کند.

هنگامی که هارنس نمی‌تواند انتقال آماده‌شده را بازتولید کند، `{ supported: false, reason }` را برگردانید.
پس از انتخاب، با خواندن پیکربندی خام پشتیبانی را استنباط نکنید.
هنگامی که آماده‌سازی احراز هویت چندین مسیر تلاش مجدد ایجاد می‌کند، یک هارنس باید
پیش از ارسال از همهٔ آن‌ها پشتیبانی کند. اگر هیچ Plugin نتواند مالک کل مجموعه باشد،
انتخاب ضمنی از OpenClaw استفاده می‌کند؛ انتخاب صریح یا ماندگارشدهٔ Plugin به‌صورت بسته با خطا متوقف می‌شود.

## ثبت هارنس

**درون‌ریزی:** `openclaw/plugin-sdk/agent-harness`

```typescript
import type { AgentHarness } from "openclaw/plugin-sdk/agent-harness";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const myHarness: AgentHarness = {
  id: "my-harness",
  label: "هارنس بومی عامل من",

  supports(ctx) {
    const routeSupportsHarness =
      ctx.modelProvider?.runtimePolicy?.compatibleIds.includes("my-harness") === true;
    const canReproduceRequest = ctx.modelProvider?.requestTransportOverrides !== "present";
    return ctx.provider === "my-provider" && routeSupportsHarness && canReproduceRequest
      ? { supported: true, priority: 100 }
      : { supported: false, reason: "مسیر مؤثر با هارنس سازگار نیست" };
  },

  async runAttempt(params) {
    // رشتهٔ بومی خود را آغاز یا از سر بگیرید.
    // از params.prompt، params.tools، params.images، params.onPartialReply،
    // params.onAgentEvent و دیگر فیلدهای تلاش آماده‌شده استفاده کنید.
    return await runMyNativeTurn(params);
  },
};

export default definePluginEntry({
  id: "my-native-agent",
  name: "عامل بومی من",
  description: "مدل‌های انتخاب‌شده را از طریق یک دیمن بومی عامل اجرا می‌کند.",
  register(api) {
    api.registerAgentHarness(myHarness);
  },
});
```

`authBootstrap` عمداً در این مثال عمومی وجود ندارد. فقط هنگامی
`authBootstrap: "harness"` را اضافه کنید که هارنس قرارداد بالا را برآورده کند.

### اجرای واگذارشده

مالک هارنس می‌تواند `delegatedExecutionPluginIds` را روی شناسه‌های Pluginهای
مورد اعتمادی تنظیم کند که باید یک نشست موجود و قفل‌شده به مدل را اجرا کنند، مانند یک انتقال
صوتی که مکالمه‌ای مبتنی بر Codex را ادامه می‌دهد. این رضایت ایستای مالک است،
نه فهرست مجاز هسته. آن را محدود نگه دارید.

واگذارشده‌ها فقط پذیرش کار و اجرای تعبیه‌شده را دریافت می‌کنند. OpenClaw
کلید دقیق نشست ذخیره‌شده، مسیر مخزن و شناسهٔ نشست؛ `modelSelectionLocked:
true`؛ و مقادیر منطبق `agentHarnessId` و `agentHarnessRuntimeOverride` را الزامی می‌کند.
سپس اجرا از طریق مالک هارنس محدود می‌شود. ایجاد، وصله‌کردن،
بازنشانی، حذف، بایگانی نشست و تغییر Gateway فقط در اختیار مالک باقی می‌مانند.

## سیاست انتخاب

OpenClaw پس از تعیین ارائه‌دهنده/مدل، یک هارنس انتخاب می‌کند:

1. سیاست زمان‌اجرای محدود به مدل اولویت دارد.
2. سیاست زمان‌اجرای محدود به ارائه‌دهنده در رتبهٔ بعدی قرار می‌گیرد.
3. `auto` از هارنس‌های ثبت‌شده می‌پرسد آیا از مسیر مؤثر تعیین‌شده
   پشتیبانی می‌کنند یا نه. پیشوندهای ارائه‌دهنده/مدل به‌تنهایی هرگز هارنسی را انتخاب نمی‌کنند.
4. اگر هیچ هارنس ثبت‌شده‌ای منطبق نباشد، OpenClaw از زمان‌اجرای تعبیه‌شدهٔ خود استفاده می‌کند.

شکست‌های هارنس Plugin به‌عنوان شکست اجرا ارائه می‌شوند. در حالت `auto`،
جایگزینی تعبیه‌شده فقط زمانی اعمال می‌شود که هیچ هارنس Plugin ثبت‌شده‌ای از
ارائه‌دهنده/مدل تعیین‌شده پشتیبانی نکند. پس از اینکه یک هارنس Plugin مالکیت اجرایی را پذیرفت،
OpenClaw همان نوبت را از طریق زمان‌اجرای دیگری بازپخش نمی‌کند، زیرا این کار می‌تواند
معنای احراز هویت/زمان‌اجرا را تغییر دهد یا عوارض جانبی را تکرار کند.

سیاست زمان‌اجرای پیکربندی‌شده دربارهٔ زمان‌اجرای مطلوب همچنان مرجع نهایی است. یک
نشست ماندگارشدهٔ `agentHarnessId` درحالی‌که آماده‌سازی مسیر/احراز هویت هنوز در انتظار است،
مالکیت رونوشت بومی خود را حفظ می‌کند. هیچ‌یک مسیری ناسازگار را
سازگار نمی‌کنند: پس از وجود واقعیت‌های آماده‌شده، هارنس انتخاب‌شده یا سنجاق‌شده
باید از آن‌ها پشتیبانی کند، وگرنه اجرا به‌صورت بسته با خطا متوقف می‌شود. `/status` زمان‌اجرای مؤثر
انتخاب‌شده از سیاست، مالکیت ماندگارشده و پشتیبانی مسیر را نشان می‌دهد.
وضعیت آماده‌سازی صریح است: `runtimePolicy` مفقود، اعلام‌نشده باقی می‌ماند
و از فیلدهای انتقالی که اتفاقاً وجود دارند استنباط نمی‌شود.
هنگامی که احراز هویت تحت مالکیت هارنس چندین مسیر فیزیکی را تعیین‌نشده باقی می‌گذارد،
واقعیت پشتیبانی آماده‌شده اشتراک شناسه‌های زمان‌اجرای سازگار آن‌هاست و
اگر هر نامزدی بازنویسی درخواست داشته باشد، آن را گزارش می‌کند. بنابراین یک نامزد اعلام‌نشده
سازگاری بومی را تهی می‌کند؛ `preparedAuth.source: "harness"`
مالک احراز هویت است، نه اجازه‌ای برای استنباط پشتیبانی مسیر.

اگر هارنس انتخاب‌شده غیرمنتظره است، ثبت اشکال‌زدایی `agents/harness` را فعال کنید
و رکورد ساختاریافتهٔ `agent harness selected` در Gateway را بررسی کنید:
این رکورد شامل شناسهٔ هارنس انتخاب‌شده، دلیل انتخاب، سیاست زمان‌اجرا/جایگزینی
و در حالت `auto`، نتیجهٔ پشتیبانی هر نامزد Plugin است.

Plugin باندل‌شدهٔ Codex، `codex` را به‌عنوان شناسهٔ هارنس خود ثبت می‌کند. هسته با آن
مانند یک شناسهٔ عادی هارنس Plugin رفتار می‌کند؛ نام‌های مستعار ویژهٔ Codex متعلق به Plugin
یا پیکربندی اپراتور هستند، نه انتخابگر مشترک زمان‌اجرا.

## جفت‌سازی ارائه‌دهنده و هارنس

بیشتر هارنس‌ها باید یک ارائه‌دهنده نیز ثبت کنند. ارائه‌دهنده ارجاع‌های مدل،
وضعیت احراز هویت، فرادادهٔ مدل و انتخاب `/model` را برای باقی بخش‌های
OpenClaw قابل مشاهده می‌کند. سپس هارنس در `supports(...)` مالکیت آن ارائه‌دهنده را می‌پذیرد.

Plugin باندل‌شدهٔ Codex از این الگو پیروی می‌کند:

- ارجاع‌های ترجیحی مدل کاربر: `openai/gpt-5.6-sol`
- ارجاع‌های سازگاری: ارجاع‌های قدیمی `codex/gpt-*` همچنان پذیرفته می‌شوند، اما پیکربندی‌های
  جدید نباید از آن‌ها به‌عنوان ارجاع‌های معمول ارائه‌دهنده/مدل استفاده کنند
- شناسهٔ هارنس: `codex`
- احراز هویت: دسترس‌پذیری مصنوعی ارائه‌دهنده، زیرا هارنس Codex مالک
  ورود/نشست بومی Codex است
- درخواست app-server: OpenClaw شناسهٔ خام مدل را به Codex می‌فرستد و اجازه می‌دهد
  هارنس با پروتکل بومی app-server ارتباط برقرار کند

Plugin مربوط به Codex افزایشی است. هنگامی که سیاست زمان‌اجرا تنظیم نشده یا `auto` باشد،
OpenAI فقط زمانی می‌تواند Codex را انتخاب کند که قرارداد مسیر تحت مالکیت ارائه‌دهندهٔ آن،
`codex` را سازگار اعلام کند: یک مسیر دقیق و رسمی HTTPS از نوع Platform Responses یا ChatGPT Responses
بدون بازنویسی تألیفی درخواست. پیشوند `openai/*` به‌تنهایی هرگز
Codex را انتخاب نمی‌کند. نقاط پایانی سفارشی، آداپتورهای Completions و رفتار تألیفی
درخواست روی OpenClaw باقی می‌مانند. نقاط پایانی رسمی HTTP با متن ساده رد می‌شوند. ارجاع‌های قدیمی‌تر `codex/gpt-*`
به‌عنوان ورودی‌های سازگاری باقی می‌مانند. به
[زمان‌اجرای ضمنی عامل OpenAI](/fa/providers/openai#implicit-agent-runtime) مراجعه کنید.

برای راه‌اندازی اپراتور، نمونه‌های پیشوند مدل و پیکربندی‌های مختص Codex، به
[هارنس Codex](/fa/plugins/codex-harness) مراجعه کنید.

Plugin مربوط به Codex حداقل نسخهٔ app-server مستندشده در
[هارنس Codex](/fa/plugins/codex-harness) را اعمال می‌کند. این Plugin دست‌دهی initialize را بررسی می‌کند و
سرورهای قدیمی‌تر یا بدون نسخه را مسدود می‌کند تا OpenClaw فقط در برابر سطح
پروتکلی اجرا شود که آزمایش کرده است.

### میان‌افزار نتیجهٔ ابزار

Pluginهای باندل‌شده و Pluginهای نصب‌شده‌ای که صراحتاً فعال شده‌اند و قراردادهای
مانیفست منطبق دارند، می‌توانند از طریق `api.registerAgentToolResultMiddleware(...)` میان‌افزار نتیجهٔ ابزار مستقل از زمان‌اجرا
را متصل کنند، مشروط بر اینکه مانیفست آن‌ها شناسه‌های
زمان‌اجرای هدف را در `contracts.agentToolResultMiddleware` اعلام کند. این
درز مورد اعتماد برای تبدیل‌های ناهمگام نتیجهٔ ابزار است که باید پیش از آن اجرا شوند که OpenClaw یا
Codex خروجی ابزار را دوباره به مدل بدهد.

Pluginهای همراه قدیمی همچنان می‌توانند از
`api.registerCodexAppServerExtensionFactory(...)` برای میان‌افزار مختص app-server در Codex
استفاده کنند، اما تبدیل‌های جدید نتیجه باید از API مستقل از runtime استفاده کنند. هوک
`api.registerEmbeddedExtensionFactory(...)` که فقط مختص embedded runner بود،
حذف شده است؛ تبدیل‌های نتیجه ابزار تعبیه‌شده باید از میان‌افزار مستقل از runtime استفاده کنند.

### طبقه‌بندی نتیجه نهایی

هارنس‌های بومی که نگاشت پروتکل خود را مدیریت می‌کنند، وقتی یک نوبت تکمیل‌شده هیچ
متن قابل‌مشاهده‌ای از دستیار تولید نکرده است، می‌توانند از
`classifyAgentHarnessTerminalOutcome(...)` در
`openclaw/plugin-sdk/agent-harness-runtime` استفاده کنند. این تابع کمکی `empty`، `reasoning-only` یا
`planning-only` را برمی‌گرداند تا سیاست بازگشت OpenClaw تصمیم بگیرد که آیا باید با
مدلی دیگر دوباره تلاش شود. `planning-only` به فیلد صریح `planText`
هارنس نیاز دارد؛ OpenClaw آن را از نثر دستیار استنباط نمی‌کند. این تابع کمکی
عمداً خطاهای پرامپت، نوبت‌های در حال اجرا و پاسخ‌های سکوت عمدی
مانند `NO_REPLY` را طبقه‌بندی‌نشده باقی می‌گذارد.

### اثرات جانبی پایان عامل

هارنس‌های بومی باید پس از نهایی‌کردن یک تلاش، `runAgentEndSideEffects(...)` را از
`openclaw/plugin-sdk/agent-harness-runtime` فراخوانی کنند. این تابع،
هوک قابل‌حمل `agent_end` و ثبت پژوهش OpenClaw را بدون ایجاد تأخیر در
پاسخ‌های تعاملی اجرا می‌کند. برای اجراهای محلی و غیرتعاملی که در آن‌ها تلاش نباید تا
پایان این اثرات جانبی خاتمه یابد، از `awaitAgentEndSideEffects(...)` استفاده کنید. هر دو تابع کمکی
همان payload مربوط به `{ event, ctx }` را مانند
`runAgentHarnessAgentEndHook(...)` می‌پذیرند؛ خطاهای آن‌ها نتیجه تلاش تکمیل‌شده
را تغییر نمی‌دهد.

### سطوح ورودی کاربر و ابزار

هارنس‌های بومی که درخواست ورودی کاربر در سطح runtime ارائه می‌دهند باید از
توابع کمکی ورودی کاربر در `openclaw/plugin-sdk/agent-harness-runtime` برای قالب‌بندی
پرامپت، تحویل آن از مسیر پاسخ مسدودکننده OpenClaw و عادی‌سازی پاسخ‌های
انتخابی یا آزاد به شکل پاسخ بومی runtime استفاده کنند. این
تابع کمکی ارائه در کانال/TUI را یکدست نگه می‌دارد، درحالی‌که هر هارنس
تجزیه پروتکل و چرخه عمر درخواست‌های معلق خود را مدیریت می‌کند.

هارنس‌های بومی که به مسیریابی فشرده ابزار مشابه PI نیاز دارند باید از
`createAgentHarnessToolSurfaceRuntime(...)` در
`openclaw/plugin-sdk/agent-harness-tool-runtime` استفاده کنند. این تابع
انتخاب کنترل جست‌وجوی ابزار/حالت کد، پیش‌فرض‌های کم‌حجم مدل محلی،
فیلترکردن شِمای سازگار با runtime، اجرای کاتالوگ پنهان، آماده‌سازی
دایرکتوری و پاک‌سازی کاتالوگ را مدیریت می‌کند. هارنس‌ها همچنان مسئول تبدیل ابزار
مختص SDK خود و callback اجرای بومی هستند.

### حالت هارنس بومی Codex

هارنس همراه `codex`، حالت بومی Codex برای نوبت‌های عامل تعبیه‌شده OpenClaw
است. ابتدا Plugin همراه `codex` را فعال کنید و اگر پیکربندی شما از allowlist محدودکننده استفاده می‌کند، `codex` را در
`plugins.allow` بگنجانید. پیکربندی‌های بومی app-server باید از `openai/gpt-*` استفاده کنند؛ نوبت‌های عامل OpenAI تنها زمانی هارنس Codex را
انتخاب می‌کنند که مسیر مؤثر، سازگاری با Codex را اعلام کند. ارجاع‌های قدیمی مدل Codex
باید با `openclaw doctor --fix` اصلاح شوند و ارجاع‌های قدیمی مدل `codex/*`
همچنان نام‌های مستعار سازگاری برای هارنس بومی باقی می‌مانند.

هنگام اجرای این حالت، Codex شناسه رشته بومی، رفتار ازسرگیری،
Compaction و اجرای app-server را مدیریت می‌کند. OpenClaw همچنان کانال گفت‌وگو،
آینه رونوشت قابل‌مشاهده، سیاست ابزار، تأییدها، تحویل رسانه و انتخاب نشست
را مدیریت می‌کند. وقتی لازم است ثابت کنید که فقط مسیر app-server مربوط به Codex می‌تواند
اجرا را در اختیار بگیرد، از ارائه‌دهنده/مدل `agentRuntime.id: "codex"` استفاده کنید.
runtimeهای صریح Plugin به‌صورت بسته شکست می‌خورند؛ شکست‌های انتخاب app-server در Codex و شکست‌های runtime
از طریق runtime دیگری دوباره امتحان نمی‌شوند.

## سخت‌گیری runtime

OpenClaw به‌طور پیش‌فرض از سیاست runtime ارائه‌دهنده/مدل `auto` استفاده می‌کند: هارنس‌های
Plugin ثبت‌شده می‌توانند مسیرهای مؤثر سازگار را در اختیار بگیرند و وقتی هیچ‌کدام مطابقت نداشته باشند، runtime
تعبیه‌شده نوبت را مدیریت می‌کند. پیشوند ارائه‌دهنده/مدل به‌تنهایی هرگز
هارنسی را انتخاب نمی‌کند. وقتی نبود انتخاب هارنس باید به‌جای
مسیریابی از طریق runtime تعبیه‌شده منجر به شکست شود، از یک runtime صریح Plugin برای ارائه‌دهنده/مدل مانند
`agentRuntime.id: "codex"` استفاده کنید. انتخاب صریح، یک
مسیر ناسازگار را سازگار نمی‌کند. شکست هارنس‌های Plugin انتخاب‌شده همیشه
قطعی است. این کار یک `agentRuntime.id: "openclaw"` صریح
برای ارائه‌دهنده/مدل را مسدود نمی‌کند.

برای اجراهای تعبیه‌شده مختص Codex:

```json
{
  "models": {
    "providers": {
      "openai": {
        "agentRuntime": {
          "id": "codex"
        }
      }
    }
  },
  "agents": {
    "defaults": {
      "model": "openai/gpt-5.6-sol"
    }
  }
}
```

اگر برای یک مدل مرجع به backend مبتنی بر CLI نیاز دارید، runtime را در ورودی
همان مدل قرار دهید:

```json
{
  "agents": {
    "defaults": {
      "model": "anthropic/claude-opus-4-8",
      "models": {
        "anthropic/claude-opus-4-8": {
          "agentRuntime": {
            "id": "claude-cli"
          }
        }
      }
    }
  }
}
```

بازنویسی‌های مختص هر عامل از همان ساختار محدود به مدل استفاده می‌کنند:

```json
{
  "agents": {
    "list": [
      {
        "id": "codex-only",
        "model": "openai/gpt-5.6-sol",
        "models": {
          "openai/gpt-5.6-sol": {
            "agentRuntime": { "id": "codex" }
          }
        }
      }
    ]
  }
}
```

نمونه‌های قدیمی runtime برای کل عامل، مانند مورد زیر، نادیده گرفته می‌شوند:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "codex"
      }
    }
  }
}
```

با یک runtime صریح Plugin، وقتی هارنس درخواستی
ثبت نشده باشد، از ارائه‌دهنده/مدل resolveشده پشتیبانی نکند یا
پیش از ایجاد اثرات جانبی نوبت شکست بخورد، نشست زودهنگام شکست می‌خورد. این رفتار برای استقرارهای مختص Codex
و آزمون‌های زنده‌ای که باید ثابت کنند مسیر app-server مربوط به Codex واقعاً
در حال استفاده است، عمدی است.

این تنظیم فقط هارنس عامل تعبیه‌شده را کنترل می‌کند. مسیریابی
مختص ارائه‌دهنده برای مدل‌های تصویر، ویدئو، موسیقی، TTS، PDF یا موارد دیگر را غیرفعال نمی‌کند.

## نشست‌های بومی و آینه رونوشت

یک هارنس ممکن است شناسه نشست بومی، شناسه رشته یا توکن ازسرگیری
در سمت daemon را نگه دارد. این اتصال را به‌طور صریح با نشست OpenClaw مرتبط نگه دارید و
خروجی قابل‌مشاهده دستیار/ابزار را همچنان در رونوشت OpenClaw
آینه کنید.

رونوشت OpenClaw همچنان لایه سازگاری موارد زیر است:

- تاریخچه نشست قابل‌مشاهده در کانال
- جست‌وجو و نمایه‌سازی رونوشت
- بازگشت به هارنس داخلی OpenClaw در نوبتی بعدی
- رفتار عمومی `/new`، `/reset` و حذف نشست

اگر هارنس شما اتصال sidecar را ذخیره می‌کند، `reset(...)` را پیاده‌سازی کنید تا OpenClaw
بتواند هنگام بازنشانی نشست مالک OpenClaw، آن را پاک کند.

## نتایج ابزار و رسانه

هسته فهرست ابزار OpenClaw را می‌سازد و آن را به تلاش
آماده‌شده می‌فرستد. وقتی یک هارنس فراخوانی پویای ابزار را اجرا می‌کند، نتیجه ابزار را
به‌جای ارسال مستقیم رسانه کانال، از طریق ساختار نتیجه هارنس
بازگردانید.

این کار خروجی‌های متن، تصویر، ویدئو، موسیقی، TTS، تأیید و ابزار پیام‌رسانی
را در همان مسیر تحویل اجراهای پشتیبانی‌شده توسط OpenClaw نگه می‌دارد.

### نتایج نهایی ابزار

`AgentHarnessAttemptParams.observeToolTerminal` انباشتگر تحت مالکیت میزبان برای
نتایج نهایی است. هارنسی که ابزارهای پویای OpenClaw یا ابزارهای بومی را اجرا می‌کند،
باید وقتی هر ابزار به یک نتیجه نهایی می‌رسد و پیش از
نهایی‌شدن نتیجه تلاش، آن را فراخوانی کند. هارنس‌هایی که ابزار اجرا نمی‌کنند نیازی به
فراخوانی آن ندارند.

واقعیت‌ها را از مرز اجرا گزارش کنید:

- در صورت وجود، شناسه فراخوانی پروتکل، نام مرجع ابزار و
  آرگومان‌هایی را که پس از آماده‌سازی یا بازنویسی هوک واقعاً به ابزار رسیده‌اند، ارسال کنید.
- وقتی اعتبارسنجی، تأیید یا محافظ دیگری
  پیش از آغاز پیاده‌سازی ابزار فراخوانی را متوقف کرده است، `executionStarted: false` را تنظیم کنید. وقتی ممکن است dispatch
  انجام شده باشد، `true` را با احتیاط گزارش کنید.
- `outcome: "success"` یا `outcome: "failure"` را گزارش کنید. به‌جای استنباط شکست از
  متن نمایشی، فیلدهای ساختاریافته شکست موجود در runtime را بگنجانید.
- از `nativeMutation` فقط برای ابزارهای بومی‌ای استفاده کنید که از تعریف ابزار OpenClaw
  استفاده نمی‌کنند. واقعیت‌های تغییر و بازپخش متعلق به پروتکل را در آن ارائه دهید؛ طبقه‌بندی‌کننده
  تغییر OpenClaw را در هارنس کپی نکنید.

callback تفکیک مرجع آن فراخوانی را برمی‌گرداند. مقدار
`lastToolError` آن را به `AgentHarnessAttemptResult` منتقل کنید و به‌جای استخراج
حالت موازی، از واقعیت‌های اجرا، آرگومان‌ها و اثرات جانبی آن در نگاشت هارنس استفاده کنید.
میزبان یک شکست تغییر‌دهنده حل‌نشده را در میان ابزارهای موفق نامرتبط نگه می‌دارد
و آن را فقط پس از موفقیت کنش متناظر پاک می‌کند.

این callback برای سازگاری کد منبع با هارنس‌های آزمایشی قدیمی
اختیاری باقی می‌ماند. اختیاری‌بودن برای هارنسی که ابزار اجرا می‌کند به‌معنای قابل‌چشم‌پوشی‌بودن نیست:
بدون گزارش‌های نهایی، OpenClaw نمی‌تواند واقعیت شکست ابزار تغییر‌دهنده را
در فراخوانی‌های بعدی ابزار، از جمله تکمیل بی‌صدای Heartbeat، حفظ کند.

## محدودیت‌های کنونی

- مسیر import عمومی فراگیر است، اما برخی نام‌های مستعار نوع تلاش/نتیجه
  هنوز برای سازگاری نام‌های قدیمی را حفظ کرده‌اند.
- نصب هارنس شخص ثالث آزمایشی است. تا زمانی که به runtime نشست بومی
  نیاز ندارید، Pluginهای ارائه‌دهنده را ترجیح دهید.
- تغییر هارنس بین نوبت‌ها پشتیبانی می‌شود. پس از آغاز
  ابزارهای بومی، تأییدها، متن دستیار یا ارسال پیام،
  هارنس را در میانه نوبت تغییر ندهید.
