یک **چارچوب اجرای عامل** اجراکننده‌ی سطح پایین برای یک نوبت آماده‌شده‌ی عامل OpenClaw است. این نه ارائه‌دهنده‌ی مدل است، نه کانال، و نه رجیستری ابزار. برای مدل ذهنی کاربرمحور، [زمان‌های اجرای عامل](/fa/concepts/agent-runtimes) را ببینید.

از این سطح فقط برای Pluginهای باندل‌شده یا بومی مورد اعتماد استفاده کنید. قرارداد هنوز آزمایشی است، چون نوع‌های پارامترها عمدا بازتاب‌دهنده‌ی اجراکننده‌ی تعبیه‌شده‌ی فعلی هستند.

## چه زمانی از چارچوب اجرا استفاده کنیم

زمانی یک چارچوب اجرای عامل ثبت کنید که یک خانواده‌ی مدل، زمان اجرای نشست بومی خودش را دارد و انتقال‌دهنده‌ی معمول ارائه‌دهنده‌ی OpenClaw انتزاع درستی نیست.

نمونه‌ها:

- یک سرور عامل کدنویسی بومی که مالک رشته‌ها و Compaction است
- یک CLI یا دیمون محلی که باید رویدادهای بومی برنامه‌ریزی/استدلال/ابزار را استریم کند
- یک زمان اجرای مدل که افزون بر رونوشت نشست OpenClaw به شناسه‌ی resume خودش نیاز دارد

فقط برای افزودن یک API جدید LLM، چارچوب اجرا ثبت نکنید. برای APIهای معمول مدل مبتنی بر HTTP یا WebSocket، یک [Plugin ارائه‌دهنده](/fa/plugins/sdk-provider-plugins) بسازید.

## هسته همچنان مالک چیست

پیش از انتخاب چارچوب اجرا، OpenClaw این موارد را از قبل حل کرده است:

- ارائه‌دهنده و مدل
- وضعیت احراز هویت زمان اجرا
- سطح تفکر و بودجه‌ی زمینه
- فایل رونوشت/نشست OpenClaw
- فضای کاری، sandbox، و سیاست ابزار
- callbackهای پاسخ کانال و callbackهای استریم
- سیاست fallback مدل و تعویض زنده‌ی مدل

این تفکیک عمدی است. چارچوب اجرا یک تلاش آماده‌شده را اجرا می‌کند؛ ارائه‌دهنده‌ها را انتخاب نمی‌کند، تحویل کانال را جایگزین نمی‌کند، یا مدل‌ها را بی‌صدا عوض نمی‌کند.

تلاش آماده‌شده همچنین شامل `params.runtimePlan` است؛ یک بسته‌ی سیاست متعلق به OpenClaw برای تصمیم‌های زمان اجرا که باید بین PI و چارچوب‌های اجرای بومی مشترک بماند:

- `runtimePlan.tools.normalize(...)` و
  `runtimePlan.tools.logDiagnostics(...)` برای سیاست schema ابزار آگاه از ارائه‌دهنده
- `runtimePlan.transcript.resolvePolicy(...)` برای پاک‌سازی رونوشت و
  سیاست ترمیم فراخوانی ابزار
- `runtimePlan.delivery.isSilentPayload(...)` برای سرکوب مشترک تحویل `NO_REPLY` و رسانه
- `runtimePlan.outcome.classifyRunResult(...)` برای طبقه‌بندی fallback مدل
- `runtimePlan.observability` برای فراداده‌ی حل‌شده‌ی ارائه‌دهنده/مدل/چارچوب اجرا

چارچوب‌های اجرا می‌توانند برای تصمیم‌هایی که باید با رفتار PI همخوان باشند از این برنامه استفاده کنند، اما همچنان باید با آن به‌عنوان وضعیت تلاش متعلق به میزبان برخورد کنند. آن را تغییر ندهید یا از آن برای تعویض ارائه‌دهنده‌ها/مدل‌ها داخل یک نوبت استفاده نکنید.

## ثبت یک چارچوب اجرا

**Import:** `openclaw/plugin-sdk/agent-harness`

```typescript
import type { AgentHarness } from "openclaw/plugin-sdk/agent-harness";
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

const myHarness: AgentHarness = {
  id: "my-harness",
  label: "My native agent harness",

  supports(ctx) {
    return ctx.provider === "my-provider"
      ? { supported: true, priority: 100 }
      : { supported: false };
  },

  async runAttempt(params) {
    // Start or resume your native thread.
    // Use params.prompt, params.tools, params.images, params.onPartialReply,
    // params.onAgentEvent, and the other prepared attempt fields.
    return await runMyNativeTurn(params);
  },
};

export default definePluginEntry({
  id: "my-native-agent",
  name: "My Native Agent",
  description: "Runs selected models through a native agent daemon.",
  register(api) {
    api.registerAgentHarness(myHarness);
  },
});
```

## سیاست انتخاب

OpenClaw پس از حل ارائه‌دهنده/مدل، یک چارچوب اجرا را انتخاب می‌کند:

1. شناسه‌ی چارچوب اجرای ثبت‌شده‌ی یک نشست موجود برنده است، بنابراین تغییرات config/env آن رونوشت را به‌صورت داغ به زمان اجرای دیگری منتقل نمی‌کنند.
2. `OPENCLAW_AGENT_RUNTIME=<id>` برای نشست‌هایی که هنوز pin نشده‌اند، یک چارچوب اجرای ثبت‌شده با همان شناسه را اجبار می‌کند.
3. `OPENCLAW_AGENT_RUNTIME=pi` چارچوب اجرای داخلی PI را اجبار می‌کند.
4. `OPENCLAW_AGENT_RUNTIME=auto` از چارچوب‌های اجرای ثبت‌شده می‌پرسد آیا از ارائه‌دهنده/مدل حل‌شده پشتیبانی می‌کنند یا نه.
5. اگر هیچ چارچوب اجرای ثبت‌شده‌ای مطابق نباشد، OpenClaw از PI استفاده می‌کند، مگر اینکه fallback به PI غیرفعال شده باشد.

خرابی‌های چارچوب اجرای Plugin به‌صورت خرابی اجرا نمایش داده می‌شوند. در حالت `auto`، fallback به PI فقط زمانی استفاده می‌شود که هیچ چارچوب اجرای Plugin ثبت‌شده‌ای از ارائه‌دهنده/مدل حل‌شده پشتیبانی نکند. وقتی یک چارچوب اجرای Plugin یک اجرا را claimed کرد، OpenClaw همان نوبت را دوباره از طریق PI پخش نمی‌کند، چون این کار می‌تواند معنای احراز هویت/زمان اجرا را تغییر دهد یا اثرهای جانبی را تکرار کند.

شناسه‌ی چارچوب اجرای انتخاب‌شده پس از یک اجرای تعبیه‌شده همراه با شناسه‌ی نشست پایدار می‌شود. نشست‌های قدیمی که پیش از pinهای چارچوب اجرا ساخته شده‌اند، وقتی سابقه‌ی رونوشت داشته باشند، PI-pinned محسوب می‌شوند. هنگام تغییر بین PI و یک چارچوب اجرای Plugin بومی، از یک نشست جدید/بازنشانی‌شده استفاده کنید. `/status` شناسه‌های چارچوب اجرای غیرفرضی مانند `codex` را کنار `Fast` نشان می‌دهد؛ PI پنهان می‌ماند، چون مسیر سازگاری پیش‌فرض است. اگر چارچوب اجرای انتخاب‌شده غیرمنتظره است، logging اشکال‌زدایی `agents/harness` را فعال کنید و رکورد ساختاریافته‌ی `agent harness selected` در Gateway را بررسی کنید. این رکورد شامل شناسه‌ی چارچوب اجرای انتخاب‌شده، دلیل انتخاب، سیاست runtime/fallback، و در حالت `auto`، نتیجه‌ی پشتیبانی هر نامزد Plugin است.

Plugin باندل‌شده‌ی Codex، `codex` را به‌عنوان شناسه‌ی چارچوب اجرای خود ثبت می‌کند. هسته با آن به‌عنوان یک شناسه‌ی معمولی چارچوب اجرای Plugin برخورد می‌کند؛ aliasهای ویژه‌ی Codex متعلق به Plugin یا config اپراتور هستند، نه انتخابگر زمان اجرای مشترک.

## جفت‌سازی ارائه‌دهنده و چارچوب اجرا

بیشتر چارچوب‌های اجرا باید یک ارائه‌دهنده نیز ثبت کنند. ارائه‌دهنده ارجاع‌های مدل، وضعیت احراز هویت، فراداده‌ی مدل، و انتخاب `/model` را برای بقیه‌ی OpenClaw قابل مشاهده می‌کند. سپس چارچوب اجرا همان ارائه‌دهنده را در `supports(...)` claimed می‌کند.

Plugin باندل‌شده‌ی Codex از همین الگو پیروی می‌کند:

- ارجاع‌های مدل ترجیحی کاربر: `openai/gpt-5.5` به‌علاوه‌ی
  `agentRuntime.id: "codex"`
- ارجاع‌های سازگاری: ارجاع‌های قدیمی `codex/gpt-*` همچنان پذیرفته می‌شوند، اما configهای جدید نباید از آن‌ها به‌عنوان ارجاع‌های معمولی provider/model استفاده کنند
- شناسه‌ی چارچوب اجرا: `codex`
- احراز هویت: دسترسی‌پذیری ارائه‌دهنده‌ی مصنوعی، چون چارچوب اجرای Codex مالک login/session بومی Codex است
- درخواست app-server: OpenClaw شناسه‌ی خام مدل را به Codex می‌فرستد و اجازه می‌دهد چارچوب اجرا با پروتکل app-server بومی صحبت کند

Plugin Codex افزایشی است. ارجاع‌های ساده‌ی `openai/gpt-*` همچنان از مسیر معمول ارائه‌دهنده‌ی OpenClaw استفاده می‌کنند، مگر اینکه چارچوب اجرای Codex را با `agentRuntime.id: "codex"` اجبار کنید. ارجاع‌های قدیمی‌تر `codex/gpt-*` همچنان برای سازگاری، ارائه‌دهنده و چارچوب اجرای Codex را انتخاب می‌کنند.

برای راه‌اندازی اپراتور، نمونه‌های پیشوند مدل، و configهای فقط Codex، [چارچوب اجرای Codex](/fa/plugins/codex-harness) را ببینید.

OpenClaw به app-server Codex نسخه‌ی `0.125.0` یا جدیدتر نیاز دارد. Plugin Codex handshake آغازین app-server را بررسی می‌کند و سرورهای قدیمی‌تر یا بدون نسخه را مسدود می‌کند تا OpenClaw فقط روی سطح پروتکلی اجرا شود که با آن آزمایش شده است. کف `0.125.0` شامل پشتیبانی payload hook بومی MCP است که در Codex `0.124.0` اضافه شد، درحالی‌که OpenClaw را به خط پایدار آزمایش‌شده‌ی جدیدتر pin می‌کند.

### میان‌افزار نتیجه‌ی ابزار

Pluginهای باندل‌شده می‌توانند از طریق `api.registerAgentToolResultMiddleware(...)` میان‌افزار نتیجه‌ی ابزار مستقل از زمان اجرا متصل کنند، زمانی که manifest آن‌ها شناسه‌های زمان اجرای هدف را در `contracts.agentToolResultMiddleware` اعلام می‌کند. این seam مورد اعتماد برای تبدیل‌های ناهمگام نتیجه‌ی ابزار است که باید پیش از اینکه PI یا Codex خروجی ابزار را دوباره وارد مدل کند، اجرا شوند.

Pluginهای باندل‌شده‌ی قدیمی همچنان می‌توانند برای میان‌افزار فقط مختص app-server Codex از `api.registerCodexAppServerExtensionFactory(...)` استفاده کنند، اما تبدیل‌های نتیجه‌ی جدید باید از API مستقل از زمان اجرا استفاده کنند. hook فقط Pi با نام `api.registerEmbeddedExtensionFactory(...)` حذف شده است؛ تبدیل‌های نتیجه‌ی ابزار Pi باید از میان‌افزار مستقل از زمان اجرا استفاده کنند.

### طبقه‌بندی نتیجه‌ی پایانی

چارچوب‌های اجرای بومی که مالک projection پروتکل خودشان هستند، زمانی که یک نوبت تکمیل‌شده هیچ متن دستیار قابل مشاهده‌ای تولید نکرده است، می‌توانند از `classifyAgentHarnessTerminalOutcome(...)` از `openclaw/plugin-sdk/agent-harness-runtime` استفاده کنند. این helper مقدار `empty`، `reasoning-only`، یا `planning-only` را برمی‌گرداند تا سیاست fallback OpenClaw بتواند تصمیم بگیرد آیا روی مدل دیگری دوباره تلاش کند یا نه. این helper عمدا خطاهای prompt، نوبت‌های در حال اجرا، و پاسخ‌های عمدا بی‌صدا مانند `NO_REPLY` را طبقه‌بندی‌نشده باقی می‌گذارد.

### حالت چارچوب اجرای بومی Codex

چارچوب اجرای باندل‌شده‌ی `codex` حالت بومی Codex برای نوبت‌های عامل تعبیه‌شده‌ی OpenClaw است. ابتدا Plugin باندل‌شده‌ی `codex` را فعال کنید، و اگر config شما از allowlist محدودکننده استفاده می‌کند، `codex` را در `plugins.allow` قرار دهید. configهای app-server بومی باید از `openai/gpt-*` همراه با `agentRuntime.id: "codex"` استفاده کنند. برای OAuth Codex از طریق PI از `openai-codex/*` استفاده کنید. ارجاع‌های مدل قدیمی `codex/*` همچنان aliasهای سازگاری برای چارچوب اجرای بومی هستند.

وقتی این حالت اجرا می‌شود، Codex مالک شناسه‌ی رشته‌ی بومی، رفتار resume، compaction، و اجرای app-server است. OpenClaw همچنان مالک کانال چت، mirror رونوشت قابل مشاهده، سیاست ابزار، تاییدیه‌ها، تحویل رسانه، و انتخاب نشست است. وقتی نیاز دارید ثابت کنید که فقط مسیر app-server Codex می‌تواند اجرا را claimed کند، از `agentRuntime.id: "codex"` بدون override برای `fallback` استفاده کنید. زمان‌های اجرای Plugin صریح از قبل به‌صورت پیش‌فرض fail closed می‌شوند. فقط زمانی `fallback: "pi"` را تنظیم کنید که عمدا می‌خواهید PI انتخاب نشدن چارچوب اجرا را مدیریت کند. خرابی‌های app-server Codex از قبل مستقیما fail می‌شوند، به‌جای اینکه از طریق PI دوباره تلاش شوند.

## غیرفعال کردن fallback به PI

به‌صورت پیش‌فرض، OpenClaw عامل‌های تعبیه‌شده را با `agents.defaults.agentRuntime` تنظیم‌شده روی `{ id: "auto", fallback: "pi" }` اجرا می‌کند. در حالت `auto`، چارچوب‌های اجرای Plugin ثبت‌شده می‌توانند یک جفت ارائه‌دهنده/مدل را claimed کنند. اگر هیچ‌کدام مطابق نباشند، OpenClaw به PI fallback می‌کند.

در حالت `auto`، وقتی نیاز دارید انتخاب نشدن چارچوب اجرای Plugin به‌جای استفاده از PI باعث fail شود، `fallback: "none"` را تنظیم کنید. زمان‌های اجرای Plugin صریح مانند `runtime: "codex"` از قبل به‌صورت پیش‌فرض fail closed می‌شوند، مگر اینکه `fallback: "pi"` در همان محدوده‌ی config یا override محیطی تنظیم شده باشد. خرابی‌های چارچوب اجرای Plugin انتخاب‌شده همیشه hard fail می‌شوند. این تنظیم یک `runtime: "pi"` صریح یا `OPENCLAW_AGENT_RUNTIME=pi` را مسدود نمی‌کند.

برای اجراهای تعبیه‌شده‌ی فقط Codex:

```json
{
  "agents": {
    "defaults": {
      "model": "openai/gpt-5.5",
      "agentRuntime": {
        "id": "codex"
      }
    }
  }
}
```

اگر می‌خواهید هر چارچوب اجرای Plugin ثبت‌شده مدل‌های مطابق را claimed کند، اما هرگز نمی‌خواهید OpenClaw بی‌صدا به PI fallback کند، `runtime: "auto"` را نگه دارید و fallback را غیرفعال کنید:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto",
        "fallback": "none"
      }
    }
  }
}
```

overrideهای هر عامل از همان شکل استفاده می‌کنند:

```json
{
  "agents": {
    "defaults": {
      "agentRuntime": {
        "id": "auto",
        "fallback": "pi"
      }
    },
    "list": [
      {
        "id": "codex-only",
        "model": "openai/gpt-5.5",
        "agentRuntime": {
          "id": "codex",
          "fallback": "none"
        }
      }
    ]
  }
}
```

`OPENCLAW_AGENT_RUNTIME` همچنان زمان اجرای پیکربندی‌شده را override می‌کند. برای غیرفعال کردن fallback به PI از محیط، از `OPENCLAW_AGENT_HARNESS_FALLBACK=none` استفاده کنید.

```bash
OPENCLAW_AGENT_RUNTIME=codex \
OPENCLAW_AGENT_HARNESS_FALLBACK=none \
openclaw gateway run
```

با غیرفعال بودن fallback، وقتی چارچوب اجرای درخواست‌شده ثبت نشده باشد، از ارائه‌دهنده/مدل حل‌شده پشتیبانی نکند، یا پیش از تولید اثرهای جانبی نوبت fail شود، نشست زود fail می‌شود. این برای استقرارهای فقط Codex و برای آزمون‌های زنده‌ای که باید ثابت کنند مسیر app-server Codex واقعا در حال استفاده است، عمدی است.

این تنظیم فقط چارچوب اجرای عامل تعبیه‌شده را کنترل می‌کند. این تنظیم مسیریابی مدل ویژه‌ی ارائه‌دهنده برای تصویر، ویدئو، موسیقی، TTS، PDF، یا موارد دیگر را غیرفعال نمی‌کند.

## نشست‌های بومی و mirror رونوشت

یک چارچوب اجرا ممکن است یک شناسه‌ی نشست بومی، شناسه‌ی رشته، یا token resume سمت دیمون نگه دارد. آن binding را صریحا مرتبط با نشست OpenClaw نگه دارید، و خروجی دستیار/ابزار قابل مشاهده برای کاربر را همچنان در رونوشت OpenClaw mirror کنید.

رونوشت OpenClaw لایه‌ی سازگاری برای این موارد باقی می‌ماند:

- سابقه‌ی نشست قابل مشاهده در کانال
- جست‌وجو و نمایه‌سازی رونوشت
- برگشت به چارچوب اجرای داخلی PI در یک نوبت بعدی
- رفتار عمومی `/new`، `/reset`، و حذف نشست

اگر چارچوب اجرای شما یک binding جانبی ذخیره می‌کند، `reset(...)` را پیاده‌سازی کنید تا OpenClaw بتواند هنگام بازنشانی نشست OpenClaw مالک، آن را پاک کند.

## نتیجه‌های ابزار و رسانه

هسته فهرست ابزارهای OpenClaw را می‌سازد و آن را به تلاش آماده‌شده پاس می‌دهد.
وقتی یک مهار یک فراخوانی ابزار پویا را اجرا می‌کند، نتیجه ابزار را از طریق
شکل نتیجه مهار برگردانید، نه اینکه خودتان رسانه کانال را ارسال کنید.

این کار خروجی‌های متن، تصویر، ویدیو، موسیقی، TTS، تأیید و ابزار پیام‌رسانی را
در همان مسیر تحویل اجراهای مبتنی بر PI نگه می‌دارد.

## محدودیت‌های فعلی

- مسیر import عمومی کلی است، اما برخی aliasهای نوع تلاش/نتیجه همچنان برای
  سازگاری نام‌های `Pi` را دارند.
- نصب مهار شخص ثالث آزمایشی است. تا زمانی که به زمان اجرای نشست بومی نیاز ندارید،
  Pluginهای ارائه‌دهنده را ترجیح دهید.
- جابه‌جایی مهار بین نوبت‌ها پشتیبانی می‌شود. پس از شروع ابزارهای بومی، تأییدها،
  متن دستیار یا ارسال پیام، مهارها را در میانه یک نوبت عوض نکنید.
