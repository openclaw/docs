پشتیبانی بومی از Pluginهای Codex به عامل OpenClaw در حالت Codex اجازه می‌دهد از قابلیت‌های برنامه و Plugin خودِ سرور برنامه Codex در همان رشته Codex استفاده کند که نوبت OpenClaw را مدیریت می‌کند.

OpenClaw، Pluginهای Codex را به ابزارهای پویا و ساختگی `codex_plugin_*` در OpenClaw ترجمه نمی‌کند. فراخوانی‌های Plugin در رونوشت بومی Codex باقی می‌مانند و سرور برنامه Codex اجرای MCP مبتنی بر برنامه را مدیریت می‌کند.

پس از راه‌اندازی هارنس پایه [Codex harness](/fa/plugins/codex-harness)، از این صفحه استفاده کنید.

## الزامات

- زمان اجرای عامل انتخاب‌شده OpenClaw باید هارنس بومی Codex باشد.
- `plugins.entries.codex.enabled` باید true باشد.
- `plugins.entries.codex.config.codexPlugins.enabled` باید true باشد.
- نسخه V1 فقط از Pluginهای `openai-curated` پشتیبانی می‌کند که مهاجرت آن‌ها را به‌عنوان نصب‌شده از مبدأ در خانه Codex مبدأ مشاهده کرده باشد.
- سرور برنامه Codex مقصد باید بتواند بازار، Plugin، و موجودی برنامه مورد انتظار را ببیند.

`codexPlugins` روی اجراهای PI، اجراهای عادی ارائه‌دهنده OpenAI، اتصال‌های گفت‌وگوی ACP، یا هارنس‌های دیگر اثری ندارد، زیرا این مسیرها رشته‌های سرور برنامه Codex را با پیکربندی بومی `apps` ایجاد نمی‌کنند.

## شروع سریع

پیش‌نمایش مهاجرت از خانه Codex مبدأ:

```bash
openclaw migrate codex --dry-run
```

وقتی طرح درست به نظر می‌رسد، مهاجرت را اعمال کنید:

```bash
openclaw migrate apply codex --yes
```

مهاجرت، ورودی‌های صریح `codexPlugins` را برای Pluginهای واجد شرایط می‌نویسد و برای Pluginهای انتخاب‌شده، `plugin/install` سرور برنامه Codex را فراخوانی می‌کند. یک پیکربندی مهاجرت‌یافته معمولی این‌گونه است:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

پس از تغییر `codexPlugins`، از `/new`، `/reset`، یا راه‌اندازی دوباره Gateway استفاده کنید تا نشست‌های آینده هارنس Codex با مجموعه برنامه به‌روزشده شروع شوند.

## راه‌اندازی بومی Plugin چگونه کار می‌کند

این یکپارچه‌سازی سه وضعیت جداگانه دارد:

- نصب‌شده: Codex بسته محلی Plugin را در زمان اجرای سرور برنامه مقصد دارد.
- فعال‌شده: پیکربندی OpenClaw آماده است Plugin را برای نوبت‌های هارنس Codex در دسترس قرار دهد.
- قابل دسترس: سرور برنامه Codex تأیید می‌کند که ورودی‌های برنامه Plugin برای حساب فعال در دسترس‌اند و می‌توانند به هویت Plugin مهاجرت‌یافته نگاشت شوند.

مهاجرت، گام پایدار نصب/احراز صلاحیت است. موجودی برنامه در زمان اجرا، بررسی دسترسی است. سپس راه‌اندازی نشست هارنس Codex، یک پیکربندی محدودکننده برنامه رشته را برای برنامه‌های Plugin فعال و قابل دسترس محاسبه می‌کند.

پیکربندی برنامه رشته زمانی محاسبه می‌شود که OpenClaw یک نشست هارنس Codex برقرار می‌کند یا اتصال رشته Codex منسوخ را جایگزین می‌کند. این پیکربندی در هر نوبت دوباره محاسبه نمی‌شود.

## مرز پشتیبانی V1

نسخه V1 عمداً محدود است:

- فقط Pluginهای `openai-curated` که از قبل در موجودی سرور برنامه Codex مبدأ نصب شده بودند، واجد شرایط مهاجرت هستند.
- مهاجرت هویت‌های صریح Plugin را با `marketplaceName` و `pluginName` می‌نویسد؛ مسیرهای کش محلی `marketplacePath` را نمی‌نویسد.
- `codexPlugins.enabled` کلید فعال‌سازی سراسری است.
- هیچ wildcard به شکل `plugins["*"]` و هیچ کلید پیکربندی‌ای که اختیار نصب دلخواه بدهد وجود ندارد.
- بازارهای پشتیبانی‌نشده، بسته‌های کش‌شده Plugin، هوک‌ها، و فایل‌های پیکربندی Codex برای بازبینی دستی در گزارش مهاجرت حفظ می‌شوند.

## موجودی برنامه و مالکیت

OpenClaw موجودی برنامه Codex را از طریق `app/list` سرور برنامه می‌خواند، آن را برای یک ساعت کش می‌کند، و ورودی‌های منسوخ یا گمشده را به‌صورت ناهمگام تازه‌سازی می‌کند.

یک برنامه Plugin فقط زمانی آشکار می‌شود که OpenClaw بتواند آن را از طریق مالکیت پایدار به Plugin مهاجرت‌یافته برگرداند:

- شناسه دقیق برنامه از جزئیات Plugin
- نام سرور MCP شناخته‌شده
- فراداده پایدار یکتا

مالکیتی که فقط بر اساس نام نمایشی باشد یا مبهم باشد، تا زمانی که تازه‌سازی بعدی موجودی مالکیت را اثبات کند، حذف می‌شود.

## پیکربندی برنامه رشته

OpenClaw یک وصله محدودکننده `config.apps` را برای رشته Codex تزریق می‌کند: `_default` غیرفعال می‌شود و فقط برنامه‌هایی که متعلق به Pluginهای مهاجرت‌یافته فعال هستند، فعال می‌شوند.

OpenClaw مقدار `destructive_enabled` در سطح برنامه را از سیاست مؤثر سراسری یا مخصوص هر Plugin برای `allow_destructive_actions` تنظیم می‌کند و اجازه می‌دهد Codex فراداده ابزارهای مخرب را از حاشیه‌نویسی‌های ابزار برنامه بومی خودش اعمال کند. پیکربندی برنامه `_default` با `open_world_enabled: false` غیرفعال می‌شود. برنامه‌های Plugin فعال با `open_world_enabled: true` منتشر می‌شوند؛ OpenClaw یک کنترل سیاست جهان‌باز جداگانه برای Plugin ارائه نمی‌کند و فهرست‌های رد نام ابزار مخرب مخصوص هر Plugin را نگه نمی‌دارد.

حالت تأیید ابزار به‌طور پیش‌فرض برای برنامه‌های Plugin خودکار است تا ابزارهای خواندنی غیرمخرب بتوانند بدون UI تأیید در همان رشته اجرا شوند. ابزارهای مخرب همچنان تحت کنترل سیاست `destructive_enabled` هر برنامه باقی می‌مانند.

## سیاست کنش مخرب

درخواست‌های مخرب Plugin برای Pluginهای Codex مهاجرت‌یافته به‌طور پیش‌فرض مجازند، در حالی که schemaهای ناامن و مالکیت مبهم همچنان به‌صورت بسته شکست می‌خورند:

- مقدار پیش‌فرض `allow_destructive_actions` سراسری برابر `true` است.
- مقدار `allow_destructive_actions` مخصوص هر Plugin، سیاست سراسری را برای آن Plugin بازنویسی می‌کند.
- وقتی سیاست `false` باشد، OpenClaw یک رد قطعی برمی‌گرداند.
- وقتی سیاست `true` باشد، OpenClaw فقط schemaهای امنی را که بتواند به پاسخ تأیید نگاشت کند به‌صورت خودکار می‌پذیرد، مانند یک فیلد تأیید بولی.
- نبود هویت Plugin، مالکیت مبهم، شناسه نوبت گمشده، شناسه نوبت نادرست، یا schema ناامن درخواست، به‌جای نمایش درخواست، رد می‌شود.

## عیب‌یابی

**`auth_required`:** مهاجرت Plugin را نصب کرده است، اما یکی از برنامه‌های آن هنوز به احراز هویت نیاز دارد. ورودی صریح Plugin تا زمانی که دوباره مجوزدهی و فعالش کنید، غیرفعال نوشته می‌شود.

**`marketplace_missing` یا `plugin_missing`:** سرور برنامه Codex مقصد نمی‌تواند بازار یا Plugin مورد انتظار `openai-curated` را ببیند. مهاجرت را دوباره روی زمان اجرای مقصد اجرا کنید یا وضعیت Plugin سرور برنامه Codex را بررسی کنید.

**`app_inventory_missing` یا `app_inventory_stale`:** آمادگی برنامه از یک کش خالی یا منسوخ آمده است. OpenClaw یک تازه‌سازی ناهمگام زمان‌بندی می‌کند و تا زمانی که مالکیت و آمادگی مشخص شوند، برنامه‌های Plugin را حذف می‌کند.

**`app_ownership_ambiguous`:** موجودی برنامه فقط بر اساس نام نمایشی تطبیق یافته است، بنابراین برنامه در معرض رشته Codex قرار نمی‌گیرد.

**پیکربندی تغییر کرده اما عامل نمی‌تواند Plugin را ببیند:** از `/new`، `/reset`، یا راه‌اندازی دوباره Gateway استفاده کنید. اتصال‌های رشته Codex موجود، پیکربندی برنامه‌ای را که با آن شروع کرده‌اند نگه می‌دارند تا زمانی که OpenClaw یک نشست هارنس جدید برقرار کند یا یک اتصال منسوخ را جایگزین کند.

**کنش مخرب رد می‌شود:** مقدارهای سراسری و مخصوص هر Plugin برای `allow_destructive_actions` را بررسی کنید. حتی وقتی سیاست true باشد، schemaهای درخواست ناامن و هویت مبهم Plugin همچنان به‌صورت بسته شکست می‌خورند.
