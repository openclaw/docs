پشتیبانی بومی Pluginهای Codex به عامل OpenClaw در حالت Codex اجازه می‌دهد قابلیت‌های app و Plugin خودِ app-server متعلق به Codex را در همان رشته Codex که نوبت OpenClaw را مدیریت می‌کند، به کار بگیرد. فراخوانی‌های Plugin در رونوشت بومی Codex باقی می‌مانند؛ app-server متعلق به Codex اجرای MCP مبتنی بر app را بر عهده دارد. OpenClaw، Pluginهای Codex را به ابزارهای پویای مصنوعی `codex_plugin_*` در OpenClaw تبدیل نمی‌کند.

پس از راه‌اندازی [مهار Codex](/fa/plugins/codex-harness)، از این صفحه استفاده کنید.

## الزامات

- زمان‌اجرای عامل باید مهار بومی Codex باشد.
- `plugins.entries.codex.enabled` باید `true` باشد.
- `plugins.entries.codex.config.codexPlugins.enabled` باید `true` باشد.
- app-server مقصد Codex باید به بازارگاه، Pluginها و موجودی app مورد انتظار دسترسی داشته باشد.
- مهاجرت فقط از Pluginهای `openai-curated` پشتیبانی می‌کند که نصب آن‌ها از منبع را در خانه مبدأ Codex مشاهده کرده باشد.
- Pluginهای `workspace-directory` که به‌صورت دستی پیکربندی شده‌اند، به app-server متعلق به Codex نیاز دارند که `plugin/list` آن `marketplaceKinds` را بپذیرد و خلاصه‌های فضای کاری بدون مسیر آن شامل `remotePluginId` باشند. Plugin باید از قبل نصب و فعال شده باشد و appهای تحت مالکیت آن باید در `app/list` قابل دسترسی باشند.

`codexPlugins` بر اجراهای ارائه‌دهنده OpenClaw، اتصال‌های مکالمه ACP یا سایر مهارها اثری ندارد، زیرا این مسیرها هرگز رشته‌های app-server متعلق به Codex را با پیکربندی بومی `apps` ایجاد نمی‌کنند.

حساب Codex در سمت OpenAI، دسترس‌پذیری app و کنترل‌های app/Plugin فضای کاری از حساب واردشده Codex تأمین می‌شوند. برای مدل حساب و مدیریت OpenAI، به [استفاده از Codex با طرح ChatGPT خود](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan) مراجعه کنید.

## شروع سریع

پیش‌نمایش مهاجرت از خانه مبدأ Codex:

```bash
openclaw migrate codex --dry-run
```

برای اینکه مهاجرت `app/list` مبدأ را فراخوانی کند و پیش از برنامه‌ریزی فعال‌سازی بومی، وجود، فعال‌بودن و دسترس‌پذیری همه appهای تحت مالکیت را الزامی کند، `--verify-plugin-apps` را اضافه کنید:

```bash
openclaw migrate codex --dry-run --verify-plugin-apps
```

وقتی برنامه درست به نظر رسید، مهاجرت را اعمال کنید:

```bash
openclaw migrate apply codex --yes
```

مهاجرت برای Pluginهای واجد شرایط، ورودی‌های صریح `codexPlugins` را می‌نویسد و برای Pluginهای انتخاب‌شده، `plugin/install` در app-server متعلق به Codex را فراخوانی می‌کند. یک پیکربندی مهاجرت‌یافته به این شکل است:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

مهاجرت همچنان به `openai-curated` محدود است. برای استفاده از یک Plugin موجود `workspace-directory`، آن را به‌صورت دستی با `summary.id` دقیق و واجد نام بازارگاه که `plugin/list` برمی‌گرداند، اضافه کنید. برای مثال، اگر Codex مقدار `example-plugin@workspace-directory` را برگرداند، به‌جای نام نمایشی آن، همین مقدار کامل را پیکربندی کنید:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            plugins: {
              "example-plugin": {
                enabled: true,
                marketplaceName: "workspace-directory",
                pluginName: "example-plugin@workspace-directory",
              },
            },
          },
        },
      },
    },
  },
}
```

OpenClaw برای یک Plugin از نوع `workspace-directory`، `plugin/install` را فراخوانی نمی‌کند و احراز هویت را آغاز نمی‌کند. پیش از افزودن یا فعال‌کردن خط‌مشی OpenClaw، آن را در Codex نصب، فعال و احراز هویت کنید. وقتی پاسخ، شواهد دقیق بازارگاه، شناسه Plugin، شناسه جزئیات یا آمادگی app را حذف کند، OpenClaw appها را پنهان نگه می‌دارد. اگر Codex درخواست صریح `plugin/list` فضای کاری را رد کند، OpenClaw برای هر Plugin فعال فضای کاری، `marketplace_missing` را گزارش می‌کند و Pluginهای گزینش‌شده‌ای را که مستقل کشف شده‌اند، در دسترس نگه می‌دارد.

پس از تغییر `codexPlugins`، مکالمه‌های جدید Codex مجموعه به‌روزشده appها را به‌طور خودکار دریافت می‌کنند. برای نوسازی مکالمه فعلی، `/new` یا `/reset` را اجرا کنید. برای تغییرات فعال/غیرفعال‌سازی Plugin، راه‌اندازی مجدد Gateway لازم نیست.

## مدیریت Pluginها از چت

`/codex plugins`، Pluginهای بومی پیکربندی‌شده Codex را از همان چتی که مهار Codex را در آن به کار می‌گیرید، بررسی یا تغییر می‌دهد:

```text
/codex plugins
/codex plugins list
/codex plugins disable google-calendar
/codex plugins enable google-calendar
```

`/codex plugins` نام مستعار `/codex plugins list` است. فهرست، کلید، وضعیت روشن/خاموش، نام Plugin در Codex و بازارگاه هر Plugin پیکربندی‌شده را از `plugins.entries.codex.config.codexPlugins.plugins` نشان می‌دهد.

`enable`/`disable` فقط در `~/.openclaw/openclaw.json` می‌نویسند؛ آن‌ها هرگز `~/.codex/config.toml` را ویرایش یا Pluginهای جدید Codex را نصب نمی‌کنند. فقط مالک یا یک کارخواه Gateway با دامنه `operator.admin` می‌تواند آن‌ها را اجرا کند.

فعال‌کردن یک Plugin پیکربندی‌شده، کلید سراسری `codexPlugins.enabled` را نیز روشن می‌کند. اگر یک Plugin گزینش‌شده به‌دلیل بازگرداندن `auth_required` توسط مهاجرت، به‌صورت غیرفعال نوشته شده است، پیش از فعال‌کردن آن در OpenClaw، مجوز app را در Codex دوباره صادر کنید. برای یک ورودی `workspace-directory`، فعال‌کردن آن در اینجا فقط خط‌مشی OpenClaw را تغییر می‌دهد؛ Plugin و app باید از قبل در Codex فعال باشند.

## نحوه کار راه‌اندازی بومی Plugin

این یکپارچه‌سازی سه وضعیت را پیگیری می‌کند:

| وضعیت       | معنا                                                                                                                                 |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| نصب‌شده     | Codex بسته Plugin را در زمان‌اجرای app-server مقصد دارد.                                                                              |
| فعال        | Codex فعال‌بودن Plugin را گزارش می‌کند و پیکربندی OpenClaw استفاده از آن را برای نوبت‌های مهار Codex مجاز می‌داند.                     |
| قابل دسترسی | app-server متعلق به Codex تأیید می‌کند که ورودی‌های app متعلق به Plugin برای حساب فعال در دسترس‌اند و به هویت پیکربندی‌شده Plugin نگاشت می‌شوند. |

برای Pluginهای `openai-curated`، مهاجرت مرحله پایدار نصب/احراز صلاحیت است:

- هنگام برنامه‌ریزی، OpenClaw جزئیات `plugin/read` در Codex مبدأ را می‌خواند و بررسی می‌کند که حساب app-server مبدأ Codex یک حساب اشتراک ChatGPT باشد. پاسخ مربوط به حساب غیر ChatGPT یا نبود پاسخ حساب باعث می‌شود Pluginهای مبتنی بر app با `codex_subscription_required` نادیده گرفته شوند.
- مهاجرت به‌طور پیش‌فرض فراخوانی `app/list` مبدأ را نادیده می‌گیرد: Pluginهای مبدأ مبتنی بر app که از دروازه حساب عبور کنند، بدون تأیید دسترس‌پذیری app مبدأ برنامه‌ریزی می‌شوند و خطاهای انتقال در جست‌وجوی حساب با `codex_account_unavailable` نادیده گرفته می‌شوند.
- با `--verify-plugin-apps`، مهاجرت یک تصویر لحظه‌ای تازه از `app/list` مبدأ می‌گیرد و پیش از برنامه‌ریزی فعال‌سازی بومی، وجود، فعال‌بودن و دسترس‌پذیری همه appهای تحت مالکیت را الزامی می‌کند. در این حالت، خطاهای انتقال در جست‌وجوی حساب به‌جای نادیده‌گرفته‌شدن کامل، به دروازه موجودی app مبدأ منتقل می‌شوند.

برای Pluginهای `workspace-directory`، راه‌اندازی خارج از OpenClaw انجام می‌شود. OpenClaw فقط زمانی آن بازارگاه را پرس‌وجو می‌کند که دست‌کم یک ورودی فعال فضای کاری پیکربندی شده باشد، هر Plugin را با `summary.id` دقیق آن حل می‌کند و بررسی‌های موجود مالکیت `plugin/read` و آمادگی `app/list` را دوباره به کار می‌گیرد. یک Plugin نصب‌نشده، غیرفعال، غیرقابل دسترسی یا احراز هویت‌نشده هیچ appی را ارائه نمی‌کند؛ OpenClaw برای نصب یا احراز هویت تلاش نمی‌کند.

موجودی app در زمان اجرا، بررسی دسترس‌پذیری نشست مقصد برای Pluginهای گزینش‌شده مهاجرت‌یافته و Pluginهای فضای کاری پیکربندی‌شده به‌صورت دستی است. راه‌اندازی نشست مهار Codex، یک پیکربندی محدودکننده app رشته را از appهای فعال و قابل دسترسی Plugin محاسبه می‌کند؛ این پیکربندی در هر نوبت دوباره محاسبه نمی‌شود، بنابراین `/codex plugins enable`/`disable` فقط بر مکالمه‌های جدید Codex اثر می‌گذارد. برای اعمال تغییر در مکالمه فعلی، از `/new` یا `/reset` استفاده کنید.

## مرز پشتیبانی V1

- فقط Pluginهای `openai-curated` که از قبل در موجودی app-server مبدأ Codex نصب شده‌اند، واجد شرایط مهاجرت هستند.
- زمان اجرا همچنین از ورودی‌های صریح `workspace-directory` در ساخت‌های app-server پشتیبانی می‌کند که `plugin/list` آن‌ها `marketplaceKinds` را پیاده‌سازی می‌کند و برای خلاصه‌های فضای کاری بدون مسیر، `remotePluginId` را برمی‌گرداند. این ورودی‌ها باید از `summary.id` دقیق و واجد نام بازارگاه خود استفاده کنند و از قبل نصب، فعال و برای app قابل دسترسی باشند. ردشدن درخواست فهرست فضای کاری، تشخیص موجود `marketplace_missing` را به‌ازای هر Plugin ایجاد می‌کند؛ نبود شواهد بازارگاه، Plugin، جزئیات یا app باعث می‌شود هیچ app فضای کاری ارائه نشود. موجودی گزینش‌شده حاصل از درخواست فهرست پیش‌فرض همچنان قابل استفاده است.
- Pluginهای مبدأ مبتنی بر app باید از دروازه اشتراک هنگام مهاجرت عبور کنند. `--verify-plugin-apps` دروازه موجودی app مبدأ را اضافه می‌کند. حساب‌های محدودشده توسط اشتراک و در حالت تأیید، appهای مبدأ غیرقابل دسترسی/غیرفعال/مفقود یا خطاهای نوسازی موجودی app، به‌جای ورودی‌های پیکربندی فعال، به‌عنوان موارد دستی نادیده‌گرفته‌شده گزارش می‌شوند. جزئیات ناخوانای Plugin پیش از دروازه موجودی app نادیده گرفته می‌شوند.
- مهاجرت هویت‌های صریح Plugin (`marketplaceName` و `pluginName`) را می‌نویسد؛ مسیرهای محلی نهانگاه `marketplacePath` را نمی‌نویسد.
- `codexPlugins.enabled` تنها کلید فعال‌سازی سراسری است؛ هیچ نویسه عام `plugins["*"]` یا کلید پیکربندی‌ای وجود ندارد که اختیار نصب دلخواه را اعطا کند.
- بازارگاه‌های غیرگزینش‌شده، بسته‌های نهان‌شده Plugin، قلاب‌ها و فایل‌های پیکربندی Codex برای بازبینی دستی در گزارش مهاجرت حفظ می‌شوند و به‌طور خودکار فعال نمی‌شوند. زمان اجرا ورودی‌های `workspace-directory` پیکربندی‌شده به‌صورت دستی را می‌پذیرد؛ سایر بازارگاه‌ها همچنان پشتیبانی نمی‌شوند.

## موجودی app و مالکیت

OpenClaw موجودی app در Codex را از طریق `app/list` متعلق به app-server می‌خواند، آن را به‌مدت یک ساعت در حافظه نهان می‌کند و ورودی‌های کهنه یا مفقود را به‌صورت ناهمگام نوسازی می‌کند. نهانگاه مختص فرایند است؛ راه‌اندازی مجدد CLI یا Gateway آن را حذف می‌کند و OpenClaw آن را از خواندن بعدی `app/list` بازسازی می‌کند.

مهاجرت و زمان اجرا از کلیدهای نهانگاه جداگانه استفاده می‌کنند:

- تأیید مهاجرت مبدأ از خانه Codex مبدأ و گزینه‌های آغاز استفاده می‌کند. این فرایند فقط با `--verify-plugin-apps` اجرا می‌شود و برای آن اجرای برنامه‌ریزی، پیمایش تازه `app/list` مبدأ را اجباری می‌کند.
- راه‌اندازی زمان اجرای مقصد هنگام ساخت پیکربندی app رشته، از هویت app-server متعلق به Codex عامل مقصد استفاده می‌کند. فعال‌سازی Plugin گزینش‌شده، آن کلید نهانگاه مقصد را نامعتبر می‌کند و سپس پس از `plugin/install` آن را به‌اجبار نوسازی می‌کند. راه‌اندازی `workspace-directory` هرگز این مسیر فعال‌سازی را اجرا نمی‌کند.

یک app متعلق به Plugin فقط زمانی ارائه می‌شود که OpenClaw بتواند آن را از طریق مالکیت پایدار به Plugin پیکربندی‌شده بازنگاشت کند: شناسه دقیق app از جزئیات Plugin، نام شناخته‌شده کارساز MCP یا فراداده پایدار و یکتا. مالکیت صرفاً مبتنی بر نام نمایشی یا مالکیت مبهم کنار گذاشته می‌شود تا نوسازی بعدی موجودی، مالکیت را اثبات کند.

## appهای حساب متصل

عامل‌هایی که مالک اداره می‌کند می‌توانند بدون نیاز به بسته Plugin منطبق، استفاده از همه appهای از قبل متصل به حساب Codex خود را انتخاب کنند:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_all_plugins: true,
            allow_destructive_actions: "auto",
          },
        },
      },
    },
  },
}
```

`allow_all_plugins: true` هنگام ایجاد یک رشته بومی جدید Codex، یک تصویر لحظه‌ای کامل از `app/list` می‌گیرد و فقط appهایی را می‌پذیرد که برای آن حساب قابل دسترسی علامت‌گذاری شده‌اند. این گزینه appها را به‌صورت سراسری نصب، احراز هویت یا فعال نمی‌کند. رشته‌های موجود مجموعه پایدار app خود را حفظ می‌کنند؛ برای اعمال appهای تازه متصل‌شده یا لغوشده، از `/new` یا `/reset` استفاده کنید یا Gateway را دوباره راه‌اندازی کنید.

appهای حساب، مقدار سراسری `codexPlugins.allow_destructive_actions` را به ارث می‌برند که `true`، `false`، `"auto"` یا `"ask"` را می‌پذیرد. خط‌مشی صریح هر Plugin برای شناسه‌های app هم‌پوشان، خط‌مشی سراسری را لغو می‌کند. خطاهای موجودی به‌صورت بسته شکست می‌خورند و به مقدار پیش‌فرض نامحدود بازنمی‌گردند.

## پیکربندی app رشته

OpenClaw برای رشتهٔ Codex یک وصلهٔ محدودکنندهٔ `config.apps` تزریق می‌کند:
`_default` غیرفعال است و فقط برنامه‌هایی فعال می‌شوند که متعلق به Pluginهای پیکربندی‌شده و فعال باشند، یا برنامه‌های حساب دسترس‌پذیری که از طریق `allow_all_plugins` پذیرفته شده‌اند.

مقدار `destructive_enabled` در هر برنامه از خط‌مشی مؤثر سراسری یا مختص هر Plugin در `allow_destructive_actions` گرفته می‌شود؛ `true`، `"auto"` و `"ask"` همگی `destructive_enabled: true` را تنظیم می‌کنند و `false` آن را روی `false` قرار می‌دهد. Codex همچنان فرادادهٔ ابزارهای مخرب را از حاشیه‌نویسی‌های بومی ابزار برنامهٔ خود اعمال می‌کند.
`_default` با `open_world_enabled: false` غیرفعال می‌شود؛ برنامه‌های Plugin فعال، `open_world_enabled: true` دریافت می‌کنند. OpenClaw یک گزینهٔ خط‌مشی جداگانه در سطح Plugin برای دسترسی جهان‌باز ارائه نمی‌کند و فهرست‌های منع نام ابزارهای مخرب را به‌ازای هر Plugin نگهداری نمی‌کند.

حالت تأیید ابزار برای برنامه‌های پذیرفته‌شده به‌طور پیش‌فرض خودکار است، بنابراین ابزارهای خواندن غیرمخرب بدون درخواست تأیید در همان رشته اجرا می‌شوند. ابزارهای مخرب همچنان تحت کنترل خط‌مشی `destructive_enabled` هر برنامه باقی می‌مانند.

## خط‌مشی کنش‌های مخرب

درخواست‌های تعاملی مخرب Plugin به‌طور پیش‌فرض برای Pluginهای پیکربندی‌شدهٔ Codex مجاز هستند، اما طرح‌واره‌های ناامن و مالکیت مبهم به‌صورت بسته رد می‌شوند:

- مقدار پیش‌فرض `allow_destructive_actions` سراسری، `true` است.
- مقدار `allow_destructive_actions` مختص هر Plugin، خط‌مشی سراسری را برای همان Plugin بازنویسی می‌کند.
- `false`:‏ OpenClaw یک پاسخ رد قطعی برمی‌گرداند.
- `true`:‏ OpenClaw فقط طرح‌واره‌های امنی را به‌طور خودکار می‌پذیرد که بتواند آن‌ها را به یک پاسخ تأیید نگاشت کند، مانند یک فیلد بولی تأیید.
- `"auto"`:‏ OpenClaw کنش‌های مخرب Plugin را در اختیار Codex قرار می‌دهد، سپس درخواست‌های تأیید MCP با مالکیت اثبات‌شده را پیش از بازگرداندن پاسخ تأیید Codex، به تأییدهای Plugin در OpenClaw تبدیل می‌کند.
- `"ask"`:‏ OpenClaw از همان محدودسازی نوشتن/مخرب Codex در `"auto"` استفاده می‌کند، بازنویسی‌های ماندگار تأیید به‌ازای هر ابزار Codex را پیش از شروع رشته برای برنامه پاک می‌کند و فقط تأیید یا رد یک‌باره ارائه می‌دهد تا تأییدهای ماندگار نتوانند درخواست‌های بعدی کنش نوشتن را سرکوب کنند. برای هر برنامهٔ پذیرفته‌شده‌ای که از `"ask"` استفاده می‌کند، OpenClaw بازبین تأییدهای انسانی Codex را برای آن برنامه انتخاب می‌کند تا Codex درخواست‌های تأیید خود را به OpenClaw بفرستد؛ سایر برنامه‌ها و تأییدهای رشته‌ای غیرمرتبط با برنامه، بازبین و خط‌مشی پیکربندی‌شدهٔ خود را حفظ می‌کنند.
- نبود هویت Plugin، مالکیت مبهم، نبود یا عدم تطابق شناسهٔ نوبت، یا طرح‌وارهٔ ناامن درخواست تعاملی، به‌جای نمایش درخواست تأیید، موجب رد می‌شود.

| `codex_subscription_required`                     | حساب کارساز برنامهٔ Codex مبدأ، حساب اشتراک ChatGPT نبود.                                                                             | با احراز هویت اشتراک وارد برنامهٔ Codex شوید، سپس مهاجرت را دوباره اجرا کنید.                                                   |
| `codex_account_unavailable`                       | حساب کارساز برنامهٔ Codex مبدأ قابل خواندن نبود.                                                                                       | احراز هویت کارساز برنامهٔ Codex مبدأ را اصلاح کنید، یا با `--verify-plugin-apps` دوباره اجرا کنید تا فهرست برنامه‌های مبدأ واجد شرایط بودن را تعیین کند. |
| `marketplace_missing`, `plugin_missing`           | بازارچه یا Plugin دقیق در دسترس نیست؛ ممکن است درخواست صریح کاتالوگ فضای کاری رد شده باشد؛ برنامه‌های فضای کاری به‌صورت بسته رد می‌شوند. | قرارداد سازگار کارساز برنامه و شناسهٔ دقیق توضیح‌داده‌شده در ادامه را بررسی کنید.                                               |
| `plugin_detail_unavailable`                       | OpenClaw نتوانست جزئیات مالکیت Plugin را بخواند.                                                                                        | پاسخ‌های `plugin/list` و `plugin/read` کارساز برنامهٔ مقصد را بررسی کنید.                                                        |
| `plugin_disabled`                                 | Codex گزارش می‌کند که Plugin نصب شده اما غیرفعال است.                                                                                  | فعال‌سازی گزینش‌شده ممکن است آن را اصلاح کند؛ پیش از تلاش مجدد، یک Plugin فضای کاری را در Codex فعال کنید.                    |
| `plugin_activation_failed`                        | فعال‌سازی Plugin تکمیل نشد.                                                                                                            | از اطلاعات تشخیصی پیوست‌شده برای تفکیک خطاهای بازارچه، احراز هویت، تازه‌سازی یا آمادگی فضای کاری استفاده کنید.                 |
| `app_inventory_missing`, `app_inventory_stale`    | آمادگی برنامه از یک حافظهٔ نهان خالی یا منقضی به‌دست آمد.                                                                               | OpenClaw به‌طور خودکار یک تازه‌سازی ناهمگام زمان‌بندی می‌کند؛ برنامه‌های Plugin تا مشخص‌شدن مالکیت و آمادگی کنار گذاشته می‌شوند. |
| `app_ownership_ambiguous`                         | فهرست برنامه‌ها فقط بر اساس نام نمایشی تطابق یافت.                                                                                      | برنامه تا زمانی که یک تازه‌سازی بعدی مالکیت را اثبات کند، از رشتهٔ Codex پنهان می‌ماند.                                        |

**Plugin فضای کاری نصب شده اما قابل مشاهده نیست:** تأیید کنید نتیجهٔ `plugin/list` فضای کاری، شناسهٔ دقیق پیکربندی‌شده را به‌صورت نصب‌شده و فعال گزارش می‌کند، سپس تأیید کنید `app/list` همهٔ برنامه‌های تحت مالکیت را برای همان حساب Codex دسترس‌پذیر گزارش می‌کند. OpenClaw می‌تواند یک برنامهٔ دسترس‌پذیر را برای رشته فعال کند، حتی اگر فهرست حساب در حال حاضر آن برنامه را غیرفعال گزارش کند. اگر پس از ذخیره‌شدن فهرست برنامه‌ها در حافظهٔ نهان Gateway این وضعیت را تغییر داده‌اید، منتظر تازه‌سازی یک‌ساعتهٔ حافظهٔ نهان بمانید یا Gateway را راه‌اندازی مجدد کنید، سپس از `/new` یا `/reset` استفاده کنید. OpenClaw‏ Pluginهای فضای کاری را اصلاح یا احراز هویت نمی‌کند.
اگر درخواست صریح فهرست فضای کاری رد شود، هر ورودی فعال فضای کاری `marketplace_missing` را گزارش می‌کند؛ ورودی‌های گزینش‌شدهٔ نامرتبط همچنان بر اساس پاسخ فهرست پیش‌فرض ادامه می‌یابند.

برای `plugin_detail_unavailable`، خلاصهٔ فضای کاری بدون مسیر باید شامل `remotePluginId` باشد؛ هنگامی که آن انتخابگر یا نتیجهٔ بعدی `plugin/read` در دسترس نباشد، OpenClaw برنامه‌های تحت مالکیت را پنهان نگه می‌دارد. برای `plugin_activation_failed`،‏ Pluginهای گزینش‌شده ممکن است خطای بازارچه، احراز هویت یا تازه‌سازی پس از نصب را گزارش کنند. یک Plugin فضای کاری زمانی این کد را گزارش می‌کند که از قبل فعال نباشد؛ آن را خارج از OpenClaw نصب، فعال و احراز هویت کنید.

**پیکربندی تغییر کرده اما عامل نمی‌تواند Plugin را ببیند:** برای تأیید وضعیت پیکربندی‌شده، `/codex plugins list` را اجرا کنید، سپس `/new` یا `/reset` را اجرا کنید. اتصال‌های موجود رشتهٔ Codex تا زمانی که OpenClaw یک نشست جدید چارچوب اجرایی ایجاد کند یا اتصال منقضی را جایگزین کند، پیکربندی برنامه‌ای را که با آن آغاز شده‌اند حفظ می‌کنند.

**کنش مخرب رد می‌شود:** مقادیر سراسری و مختص هر Plugin در `allow_destructive_actions` را بررسی کنید. حتی با `true`، `"auto"` یا `"ask"`، طرح‌واره‌های ناامن درخواست تعاملی و هویت مبهم Plugin همچنان به‌صورت بسته رد می‌شوند.
