Plugin وُرک‌بورد یک تابلوی اختیاری به سبک کانبان را به
[رابط کاربری کنترل](/fa/web/control-ui) اضافه می‌کند: کارت‌های کاری در اندازه مناسب عامل، واگذاری به عامل‌ها،
و پیوندی برای بازگشت به وظیفه، اجرا و نشست داشبورد مرتبط با کارت.

وُرک‌بورد عمداً کوچک طراحی شده است: کارهای عملیاتی محلی یک
OpenClaw Gateway را پیگیری می‌کند. این ابزار جایگزینی برای GitHub Issues، Linear، Jira یا
دیگر سامانه‌های مدیریت پروژه تیمی نیست.

## فعال‌سازی

وُرک‌بورد همراه محصول ارائه می‌شود، اما به‌طور پیش‌فرض غیرفعال است:

1. در رابط کاربری کنترل، **Plugins** را باز کنید یا نسبت به مسیر پایه پیکربندی‌شده
   رابط کاربری کنترل، از `/settings/plugins` استفاده کنید. برای مثال، با مسیر پایه `/openclaw`
   باید از `/openclaw/settings/plugins` استفاده شود.
2. **Workboard** را پیدا کرده و **Enable** را انتخاب کنید. چون وُرک‌بورد همراه
   OpenClaw ارائه می‌شود، به اقدام **Install** نیاز ندارد.
3. اگر رابط کاربری اعلام کرد که راه‌اندازی مجدد لازم است، Gateway را راه‌اندازی مجدد کنید.

پس از بارگیری زمان اجرای Plugin، زبانه وُرک‌بورد در پیمایش داشبورد ظاهر می‌شود.
تا زمانی که غیرفعال باشد، این زبانه در پیمایش پنهان می‌ماند. باز کردن مستقیم مسیر
`/workboard` هنگامی که Plugin غیرفعال است یا توسط
`plugins.allow`/`plugins.deny` مسدود شده است، به‌جای داده‌های کارت
وضعیت در‌دسترس‌نبودن Plugin را نشان می‌دهد.

گردش‌کار معادل در CLI چنین است:

```bash
openclaw plugins enable workboard
openclaw gateway restart
openclaw dashboard
```

## پیکربندی

وُرک‌بورد پیکربندی مختص Plugin ندارد. آن را با ورودی استاندارد
Plugin فعال یا غیرفعال کنید:

```json5
{
  plugins: {
    entries: {
      workboard: {
        enabled: true,
        config: {},
      },
    },
  },
}
```

```bash
openclaw plugins disable workboard
openclaw gateway restart
```

## فیلدهای کارت

| فیلد        | مقادیر                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------- |
| `status`    | `triage`، `backlog`، `todo`، `scheduled`، `ready`، `running`، `review`، `blocked`، `done`                     |
| `priority`  | `low`، `normal`، `high`، `urgent`                                                                             |
| `labels`    | رشته‌های آزاد                                                                                                 |
| `agentId`   | عامل واگذارشده اختیاری                                                                                        |
| ارجاع‌های پیوندشده | وظیفه، اجرا، نشست یا نشانی وب منبع به‌صورت اختیاری                                                     |
| `execution` | فراداده اختیاری برای اجرای Codex/Claude که از کارت آغاز شده است (موتور، حالت، مدل، نشست، شناسه اجرا، وضعیت) |

کارت‌ها همچنین فراداده‌ای فشرده برای تلاش‌ها، دیدگاه‌ها، پیوندها، مدرک‌ها،
مصنوعات، تنظیمات خودکارسازی، پیوست‌ها، گزارش‌های کارکن، وضعیت پروتکل
کارکن، ادعاها، اطلاعات تشخیصی، اعلان‌ها، شناسه الگو، وضعیت بایگانی و
تشخیص نشست منقضی دارند؛ همچنین دارای فهرستی از رویدادهای اخیر هستند (`created`، `edited`،
`moved`، `linked`، `specified`، `decomposed`، `claimed`، `heartbeat`،
`execution_updated`، `attempt_started`، `attempt_updated`، `comment_added`،
`link_added`، `proof_added`، `artifact_added`، `attachment_added`،
`diagnostic`، `notification`، `dispatch`، `orchestration`،
`protocol_violation`، `archived`، `unarchived`، `stale`). این فراداده به
اپراتور امکان می‌دهد بدون باز کردن نشست پیوندشده، نحوه جابه‌جایی کارت در تابلو را
مشاهده کند؛ این اطلاعات زمینه عملیاتی محلی است و جایگزینی برای رونوشت‌های نشست
یا تاریخچه GitHub Issues نیست.

کارت‌ها در وضعیت اختصاصی Gateway متعلق به Plugin ذخیره می‌شوند و همراه با سایر
وضعیت‌های OpenClaw آن Gateway جابه‌جا می‌شوند (به [ذخیره‌سازی](#storage) مراجعه کنید).

## آغاز کار از یک کارت

کارت‌های بدون پیوند می‌توانند کار را مستقیماً آغاز کنند:

- **اجرای Codex** / **اجرای Claude** یک اجرای عاملِ پیگیری‌شده با وظیفه را با
  موتور صریح آغاز می‌کند، درخواست کارت را می‌فرستد و کارت را در وضعیت `running` قرار می‌دهد. اجراهای Codex
  از `openai/gpt-5.6-sol` و اجراهای Claude از `anthropic/claude-sonnet-4-6` استفاده می‌کنند.
- **باز کردن Codex** / **باز کردن Claude** بدون ارسال درخواست کارت یا
  جابه‌جایی آن، یک نشست داشبورد پیوندشده برای کار دستی ایجاد می‌کند که
  به تابلو متصل باقی می‌ماند.

شروع‌های خودکار از مسیر اجرای عاملِ پیگیری‌شده با وظیفه در Gateway استفاده می‌کنند (عامل
و مدل پیش‌فرض، مگر اینکه Codex/Claude صراحتاً انتخاب شده باشد)؛ سپس وُرک‌بورد
وظیفه حاصل، شناسه اجرا و کلید نشست را به کارت پیوند می‌دهد. هر اجرای
پیوندشده همچنین خلاصه‌ای از تلاش را ثبت می‌کند (موتور، حالت، مدل، شناسه اجرا،
مُهرهای زمانی، وضعیت و شمارنده جاری شکست‌ها) تا شکست‌های تکراری قابل مشاهده بمانند.

داشبورد وضعیت وظیفه را از دفترکل وظایف Gateway تازه‌سازی می‌کند و
وظایف را بر اساس شناسه وظیفه، شناسه اجرا یا کلید نشست پیوندشده با کارت‌ها تطبیق می‌دهد. یک وظیفه
در صف یا در حال اجرا، چرخه عمر کارت را فعال نگه می‌دارد؛ وظیفه پایان‌یافته، ناموفق، مهلت‌گذشته یا
لغوشده با استفاده از همان قاعده همگام‌سازی نشست‌های پیوندشده، کارت را به‌سمت
`review` یا `blocked` می‌برد (به [همگام‌سازی چرخه عمر نشست](#session-lifecycle-sync) مراجعه کنید).

`workboard_promote`/`workboard_reassign`/`workboard_reclaim` انجام می‌شود که به
توکن نیاز ندارند.

## توزیع

توزیع در سطح محلی Gateway انجام می‌شود: فرایندهای دلخواه سیستم‌عامل را ایجاد نمی‌کند. جلسه‌های
عوامل فرعی معمول OpenClaw همچنان مالک اجرا هستند. یک نوبت توزیع:

1. کارت‌هایی را که وابستگی‌هایشان آماده است ارتقا می‌دهد.
2. فراداده توزیع را روی کارت‌های آماده ثبت می‌کند.
3. ادعاهای منقضی یا اجراهای پایان‌یافته بر اثر مهلت زمانی را مسدود می‌کند.
4. کارت‌های تریاژ پیکربندی‌شده در تابلو را به‌عنوان نامزدهای هماهنگ‌سازی علامت‌گذاری می‌کند.
5. دسته کوچکی از کارت‌های آماده را ادعا می‌کند و اجرای عامل‌ها را از طریق
   محیط اجرای عامل فرعی Gateway آغاز می‌کند.

عامل‌ها زمینه محدود کارت را به‌همراه توکن ادعای لازم برای ارسال Heartbeat،
تکمیل یا مسدودکردن کارت از طریق ابزارهای Workboard دریافت می‌کنند.

### انتخاب عامل

هر نوبت به‌طور پیش‌فرض **حداکثر ۳ عامل را آغاز می‌کند**. کارت‌های آماده ابتدا بر اساس
اولویت، سپس موقعیت و بعد زمان ایجاد مرتب می‌شوند. هر نوبت برای هر مالک/عامل فقط یک کارت
را آغاز می‌کند و از مالکانی که از قبل روی تابلو کار در حال اجرا یا بازبینی دارند عبور می‌کند.
کارت‌های بایگانی‌شده، کارت‌های دارای ادعای فعال و کارت‌هایی که در وضعیت `ready` نیستند
هرگز برای آغاز عامل انتخاب نمی‌شوند (بااین‌حال بخش داده‌ای توزیع همچنان می‌تواند بر آن‌ها
اثر بگذارد: پاک‌سازی ادعای راکد، ارتقای وابستگی و پاک‌سازی مهلت زمانی).

کلیدهای جلسه برای هر تابلو/کارت قطعی هستند؛ بنابراین توزیع‌های تکراری به‌جای ایجاد
جلسه‌های نامرتبط، دوباره به همان مسیر عامل هدایت می‌شوند:

- کارت‌های تخصیص‌یافته: `agent:<agentId>:subagent:workboard-<boardId>-<cardId>`
- کارت‌های تخصیص‌نیافته: `subagent:workboard-<boardId>-<cardId>` (Gateway عامل
  پیش‌فرض پیکربندی‌شده را تعیین می‌کند)

اگر پس از ادعای یک کارت نتوان عامل را آغاز کرد، Workboard کارت را مسدود می‌کند،
ادعا را پاک می‌کند، شکست آغاز اجرا را ثبت می‌کند و یک خط گزارش عامل می‌افزاید که
در داشبورد، JSON ابزار CLI، ابزارهای عامل و تشخیص‌های کارت قابل‌مشاهده است.

### نقاط ورود

- کنش توزیع در داشبورد
- `openclaw workboard dispatch`
- `/workboard dispatch` در یک کانال دارای قابلیت فرمان

هر سه، در صورت دردسترس‌بودن Gateway، از محیط اجرای عامل فرعی Gateway استفاده می‌کنند.
CLI یک راهکار جایگزین برای گرداننده دارد: اگر فراخوانی Gateway با خطای
اتصال/دردسترس‌نبودن (یا خطای `unknown method` برای Gatewayهای قدیمی‌تر) شکست بخورد،
و هیچ مقصد صریح `--url`/`--token` و هیچ Gateway راه دور پیکربندی‌شده‌ای
(`OPENCLAW_GATEWAY_URL` یا `gateway.mode: remote`) اعمال نشود، CLI توزیع صرفاً
داده‌ای را روی وضعیت محلی SQLite اجرا می‌کند؛ می‌تواند وابستگی‌ها را ارتقا دهد،
ادعاهای راکد را پاک کند و اجراهای پایان‌یافته بر اثر مهلت زمانی را مسدود کند، اما نمی‌تواند
عامل‌ها را آغاز کند. شکست‌های احراز هویت، مجوز و اعتبارسنجی از یک Gateway دردسترس،
دردسترس‌نبودن تلقی نمی‌شوند؛ آن‌ها به‌صورت خطاهای فرمان نمایش داده می‌شوند. همچنین هر
شکست Gateway در صورت ارائه مقصد صریح `--url`/`--token` نیز به‌صورت خطای فرمان
نمایش داده می‌شود.

فراداده تابلو می‌تواند `autoDecompose`، `autoDecomposePerDispatch`،
`defaultAssignee` و `orchestratorProfile` را تنظیم کند. OpenClaw این قصد را
ثبت می‌کند و در زمینه عامل ارائه می‌دهد؛ مشخص‌سازی/تقسیم واقعی همچنان از طریق
ابزارهای معمول Workboard اجرا می‌شود.

## CLI و فرمان اسلش

```bash
openclaw workboard list [--board <id>] [--status <status>] [--include-archived] [--json]
openclaw workboard create "Fix stale card lifecycle" --priority high --labels bug,workboard
openclaw workboard show <card-id> [--json]
openclaw workboard dispatch [--board <id>] [--json]
```

خروجی متنی `list` به‌طور پیش‌فرض کارت‌های بایگانی‌شده را پنهان می‌کند
(`--include-archived` این رفتار را لغو می‌کند)؛ `--json` همیشه کارت‌های
بایگانی‌شده را نیز شامل می‌شود و با قرارداد کامل کارت که اسکریپت‌های موجود استفاده می‌کنند،
مطابقت دارد. `show` یک پیشوند شناسه بدون ابهام را می‌پذیرد.
`list`، `create` و `show` همیشه وضعیت محلی Plugin را مستقیماً می‌خوانند/می‌نویسند.
فقط `dispatch`، Gateway در حال اجرا را فراخوانی می‌کند و از راهکار جایگزین توضیح‌داده‌شده
در بالا بهره می‌برد.

برای پرچم‌های کامل، خروجی JSON، رفتار راهکار جایگزین Gateway، مدیریت پیشوند شناسه،
قواعد انتخاب توزیع و عیب‌یابی، به [CLI در Workboard](/fa/cli/workboard) مراجعه کنید.

`/workboard list`، `/workboard show <card-id>`، `/workboard create <title>`
و `/workboard dispatch` معادل CLI هستند. فهرست‌کردن و نمایش، عملیات خواندن
برای هر فرستنده فرمان مجاز هستند. ایجاد و توزیع در سطوح چت به وضعیت مالک،
یا به یک کارخواه Gateway با `operator.write`/`operator.admin` نیاز دارند.

## همگام‌سازی چرخه‌عمر جلسه

کارت‌ها می‌توانند به یک نشست موجود در داشبورد یا نشستی که هنگام شروع کار از کارت ایجاد می‌شود پیوند داشته باشند. کارت‌های پیوندشده چرخهٔ عمر نشست را به‌صورت درون‌خطی نشان می‌دهند:
در حال اجرا، راکد، پیوندشده و بی‌کار، انجام‌شده، ناموفق یا مفقود. همچنین می‌توانید از زبانهٔ Sessions با **Add to Workboard** یک نشست موجود را ثبت کنید؛ کارت به آن نشست پیوند می‌خورد، از برچسب نشست یا درخواست اخیر کاربر به‌عنوان عنوان استفاده می‌کند و در صورت وجود، یادداشت‌ها را با درخواست اخیر کاربر به‌همراه جدیدترین پاسخ دستیار مقداردهی اولیه می‌کند.

اگر نشست پیوندشده مفقود شود، کارت برای حفظ زمینه همچنان پیوندشده باقی می‌ماند و کنترل‌های شروع را برای راه‌اندازی مجدد در یک نشست تازه ارائه می‌دهد. اگر یک نشست پیوندشدهٔ فعال دیگر فعالیت اخیر را گزارش نکند، Workboard کارت را `stale` علامت‌گذاری می‌کند و تا زمانی که چرخهٔ عمر آن را پاک کند، این وضعیت را به‌عنوان فراداده ذخیره می‌کند.

هنگامی که کارت در وضعیت کار فعال قرار دارد، Workboard نشست پیوندشده را دنبال می‌کند:

| وضعیت نشست پیوندشده                     | وضعیت کارت |
| --------------------------------------- | ---------- |
| فعال                                    | `running`  |
| تکمیل‌شده                               | `review`   |
| ناموفق، خاتمه‌یافته، منقضی یا لغوشده    | `blocked`  |

**وضعیت‌های بازبینی دستی اولویت دارند.** انتقال کارت به `review`، `blocked` یا `done`
همگام‌سازی خودکار آن کارت را متوقف می‌کند تا زمانی که دوباره آن را به `todo` یا `running` منتقل کنید.

شروع یک کارت از نشست‌های معمول Gateway استفاده می‌کند؛ Workboard فقط فراداده و پیوندهای کارت را ذخیره می‌کند. رونوشت مکالمه، انتخاب مدل و چرخهٔ عمر اجرا همچنان در اختیار سامانهٔ معمول نشست باقی می‌مانند. برای لغو اجرای فعال، روی کارت پیوندشدهٔ زنده از **Stop** استفاده کنید؛ Workboard آن کارت را `blocked` علامت‌گذاری می‌کند تا برای پیگیری قابل مشاهده بماند.

کارت‌های جدید می‌توانند از الگوهای Workboard (`bugfix`، `docs`، `release`،
`pr_review`، `plugin`) شروع شوند. الگوها عنوان، یادداشت‌ها، برچسب‌ها و اولویت را از پیش پر می‌کنند؛ شناسهٔ الگو به‌عنوان فرادادهٔ کارت ذخیره می‌شود.

## گردش کار داشبورد

1. زبانهٔ Workboard را در Control UI باز کنید.
2. کارتی با عنوان، یادداشت‌ها، اولویت، برچسب‌ها، عامل اختیاری و نشست پیوندشدهٔ اختیاری ایجاد کنید؛ یا Sessions را باز کنید و برای یک نشست موجود **Add to Workboard** را انتخاب کنید.
3. کارت را میان ستون‌ها بکشید، یا کنترل فشردهٔ وضعیت آن را متمرکز کنید و از منو یا ArrowLeft/ArrowRight استفاده کنید.
4. کار را از کارت شروع کنید تا یک نشست داشبورد ایجاد یا دوباره استفاده شود.
5. در حین کار عامل، نشست پیوندشده را از کارت باز کنید.
6. اجازه دهید همگام‌سازی چرخهٔ عمر، کار در حال اجرا را به `review`/`blocked` منتقل کند؛ سپس پس از پذیرش، کارت را به‌صورت دستی به `done` منتقل کنید.

## عیب‌یابی

عیب‌یابی‌ها از فرادادهٔ محلی کارت محاسبه می‌شوند. بررسی‌های داخلی موارد زیر را علامت‌گذاری می‌کنند:

| نوع                         | شرط                                                                             |
| --------------------------- | ------------------------------------------------------------------------------- |
| `stranded_ready`            | کارت تخصیص‌یافتهٔ `todo`/`backlog`/`ready` که بیش از ۱ ساعت به‌روزرسانی نشده است. |
| `running_without_heartbeat` | کارت `running` بدون Heartbeat ادعا یا به‌روزرسانی اجرا در بیش از ۲۰ دقیقه.       |
| `blocked_too_long`          | کارت `blocked` که بیش از ۲۴ ساعت به‌روزرسانی نشده است.                           |
| `repeated_failures`         | تعداد شکست‌های پیگیری‌شدهٔ کارت به ۲ یا بیشتر رسیده است.                         |
| `missing_proof`             | کارت `done` بدون مدرک، مصنوعات یا پیوست‌ها.                                      |
| `orphaned_session`          | کارت `running` دارای `sessionKey` اما بدون فرادادهٔ `execution`.                  |

## مجوزها

روش‌های RPC مربوط به Gateway زیر فضای نام `workboard.*` قرار دارند:

| دامنه            | روش‌ها                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `operator.read`  | `cards.list`، `cards.export`، `cards.diagnostics`، فهرست/دریافت پیوست، خواندن رویدادهای اعلان، `boards.list`، `cards.stats`، `cards.runs`                                                                                                                                                                                                                                       |
| `operator.write` | `cards.diagnostics.refresh`، ایجاد/به‌روزرسانی/انتقال/حذف/نظر/پیوند/linkDependency/مدرک/مصنوع، افزودن/حذف پیوست، گزارش کارگر، تخلف پروتکل، ادعا/Heartbeat/آزادسازی/ارتقا/تخصیص مجدد/بازپس‌گیری/تکمیل/مسدودسازی/رفع مسدودی، `cards.dispatch`، `cards.bulk`، بایگانی، `boards.upsert`/`archive`/`delete`، `cards.specify`/`decompose`، اشتراک/حذف/پیش‌برد اعلان |

هیچ روش RPC به `operator.admin` نیاز ندارد. مرورگرهایی که با دسترسی فقط‌خواندنی اپراتور متصل هستند می‌توانند برد را بررسی کنند، اما نمی‌توانند کارت‌ها را تغییر دهند.

## ذخیره‌سازی

Workboard داده‌های پایدار را در یک پایگاه‌دادهٔ رابطه‌ای SQLite متعلق به Plugin و زیر شاخهٔ وضعیت OpenClaw ذخیره می‌کند: بردها، کارت‌ها، برچسب‌ها، رویدادهای چرخهٔ عمر، تلاش‌های اجرا، نظرها، پیوندهای وابستگی، مدارک، ارجاع‌های مصنوعات، فراداده و داده‌های دودویی پیوست‌ها، عیب‌یابی‌ها، اعلان‌ها، گزارش‌های کارگر، وضعیت پروتکل و اشتراک‌ها همگی در جدول‌های Workboard قرار دارند (نه در ورودی‌های کلید-مقدار Plugin). خروجی کارت، روایت برد را بدون درج درون‌خطی محتوای دودویی پیوست‌ها حفظ می‌کند.

نصب‌هایی که در انتشار `.28` از Workboard استفاده کرده‌اند، می‌توانند
`openclaw doctor --fix` را اجرا کنند تا فضاهای نام قدیمی و منتشرشدهٔ وضعیت Plugin
(`workboard.cards`، `workboard.boards`، `workboard.notify` و در صورت وجود،
`workboard.attachments`) به پایگاه‌دادهٔ رابطه‌ای مهاجرت کنند.

## رفع اشکال

**زبانه اعلام می‌کند Workboard در دسترس نیست**

```bash
openclaw plugins inspect workboard --runtime --json
```

اگر `plugins.allow` پیکربندی شده است، `workboard` را به آن اضافه کنید. اگر `plugins.deny`
شامل `workboard` است، پیش از فعال‌کردن Plugin آن را حذف کنید.

**کارت‌ها ذخیره نمی‌شوند**

تأیید کنید اتصال مرورگر دسترسی `operator.write` دارد. نشست‌های فقط‌خواندنی اپراتور می‌توانند کارت‌ها را فهرست کنند، اما نمی‌توانند آن‌ها را ایجاد، ویرایش، انتقال یا حذف کنند.

**شروع یک کارت، نشست مورد انتظار را باز نمی‌کند**

شناسهٔ عامل و نشست پیوندشدهٔ کارت را بررسی کنید، سپس Sessions یا Chat را باز کنید تا وضعیت واقعی اجرا را بررسی کنید.

**توزیع، کارگری را شروع نمی‌کند**

تأیید کنید دست‌کم یک کارت `ready` بدون ادعای فعال وجود دارد:

```bash
openclaw workboard list --status ready
```

اگر CLI توزیع فقط‌داده‌ای را گزارش می‌کند، Gateway را شروع یا دوباره راه‌اندازی کنید و مجدداً تلاش کنید؛ توزیع فقط‌داده‌ای وضعیت محلی برد را به‌روزرسانی می‌کند، اما نمی‌تواند اجرای کارگرِ عامل فرعی را شروع کند. همچنین ممکن است کارت‌ها زمانی نادیده گرفته شوند که کارت دیگری برای همان مالک یا عامل از قبل در حال اجرا یا در انتظار بازبینی باشد؛ پیش از توزیع کارهای بیشتر برای همان مالک، آن کار فعال را تکمیل، مسدود یا آزاد کنید.
