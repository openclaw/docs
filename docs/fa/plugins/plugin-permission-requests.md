درخواست‌های مجوز Plugin به کد Plugin امکان می‌دهند اجرای یک فراخوانی ابزار یا عملیات تحت مالکیت Plugin را تا زمانی که کاربر آن را تأیید یا رد کند، متوقف کند. آن‌ها از جریان Gateway
`plugin.approval.*` و همان سطوح رابط کاربری تأیید استفاده می‌کنند که دکمه‌های تأیید در گفت‌وگو و فرمان‌های `/approve` را مدیریت می‌کنند.

از درخواست‌های مجوز Plugin برای مجوزهای Plugin/برنامه استفاده کنید. این درخواست‌ها جایگزین تأییدهای اجرای میزبان، فهرست‌های مجاز اختیاری ابزارها یا بازبینی مجوز بومی Codex نمی‌شوند.

## انتخاب دروازه مناسب

دروازه‌ای را انتخاب کنید که با نقطه تصمیم‌گیری موردنیاز مطابقت دارد:

| دروازه                             | زمانی از آن استفاده کنید که                                                              | آنچه کنترل می‌کند                                                                                                  |
| -------------------------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------- |
| ابزارهای اختیاری                   | ابزار نباید تا پیش از اعلام رضایت کاربر برای مدل قابل‌مشاهده باشد.        | نمایش ابزار از طریق `tools.allow`.                                                                              |
| درخواست‌های مجوز Plugin       | یک هوک Plugin یا عملیات تحت مالکیت Plugin باید پیش از اجرای یک اقدام اجازه بگیرد. | تأیید زمان اجرا از طریق `plugin.approval.*`.                                                                     |
| تأییدهای اجرا                   | یک فرمان میزبان یا ابزار شبیه پوسته به تأیید اپراتور نیاز دارد.               | سیاست اجرای میزبان و فهرست‌های مجاز پایدار اجرا.                                                                     |
| درخواست‌های مجوز بومی Codex | Codex پیش از اقدامات بومی پوسته، فایل، MCP یا سرور برنامه اجازه می‌گیرد.        | مدیریت تأیید سرور برنامه یا هوک بومی Codex که وقتی OpenClaw مالک اعلان است، از طریق تأییدهای Plugin مسیریابی می‌شود. |
| درخواست‌های تأیید MCP        | یک سرور MCP متعلق به Codex برای فراخوانی ابزار درخواست تأیید می‌کند.                    | پاسخ‌های تأیید MCP که از طریق تأییدهای Plugin در OpenClaw منتقل می‌شوند.                                                 |

ابزارهای اختیاری، دروازه‌ای در زمان کشف هستند. درخواست‌های مجوز Plugin، دروازه‌ای برای هر فراخوانی هستند. هنگامی که یک ابزار حساس باید پیش از مشاهده‌شدن توسط مدل نیازمند اعلام رضایت صریح باشد و پیش از اجرای اقدام نیز تأیید شود، از هر دو استفاده کنید.

## درخواست تأیید پیش از فراخوانی ابزار

بیشتر اعلان‌های ایجادشده توسط Plugin باید در یک هوک `before_tool_call` آغاز شوند. این هوک پس از انتخاب ابزار توسط مدل و پیش از اجرای آن توسط OpenClaw اجرا می‌شود:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

export default definePluginEntry({
  id: "deploy-policy",
  name: "Deploy Policy",
  register(api) {
    api.on("before_tool_call", async (event) => {
      if (event.toolName !== "deploy_service") {
        return;
      }

      const environment =
        typeof event.params.environment === "string" ? event.params.environment : "unknown";

      return {
        requireApproval: {
          title: "Deploy service",
          description: `Deploy service to ${environment}.`,
          severity: environment === "production" ? "critical" : "warning",
          allowedDecisions:
            environment === "production"
              ? ["allow-once", "deny"]
              : ["allow-once", "allow-always", "deny"],
          timeoutMs: 120_000,
          onResolution(decision) {
            console.log(`deploy approval resolved: ${decision}`);
          },
        },
      };
    });
  },
});
```

متن اعلان را برای شخصی بنویسید که اقدام را تأیید خواهد کرد:

- `title` را کوتاه و متمرکز بر اقدام نگه دارید؛ Gateway آن را به 80 نویسه محدود می‌کند.
- `description` را مشخص و محدود نگه دارید؛ Gateway آن را به 512
  نویسه محدود می‌کند.
- اقدام، هدف و ریسک را درج کنید. اسرار، توکن‌ها یا
  بارهای داده خصوصی را که نباید در سطوح تأیید گفت‌وگو ظاهر شوند، درج نکنید.
- اگر `severity` حذف شود، مقدار پیش‌فرض آن `"warning"` است. از `"critical"` فقط برای
  اقداماتی استفاده کنید که تصمیم اشتباه درباره آن‌ها می‌تواند موجب آسیب به محیط عملیاتی یا ازدست‌رفتن داده شود.
- اگر `allowedDecisions` حذف شود، مقدار پیش‌فرض آن `["allow-once", "allow-always", "deny"]` است.
  هنگامی که اعتماد پایدار برای آن اقدام ناامن است، `["allow-once", "deny"]` را ارسال کنید.
- مقدار پیش‌فرض `timeoutMs` برابر با 120000 (2 دقیقه) است و صرف‌نظر از مقدار درخواستی، حداکثر به 600000 (10
  دقیقه) محدود می‌شود.

## رفتار تصمیم‌گیری

OpenClaw یک تأیید در انتظار با شناسه `plugin:` ایجاد می‌کند، آن را به سطوح تأیید موجود می‌فرستد و منتظر تصمیم می‌ماند.

| تصمیم          | نتیجه                                                                    |
| ----------------- | ------------------------------------------------------------------------- |
| `allow-once`      | فراخوانی جاری ادامه می‌یابد.                                               |
| `allow-always`    | فراخوانی جاری ادامه می‌یابد و تصمیم به Plugin ارسال می‌شود.      |
| `deny`            | فراخوانی با نتیجه ابزارِ ردشده مسدود می‌شود.                            |
| پایان مهلت           | فراخوانی مسدود می‌شود.                                                      |
| لغو      | هنگامی که اجرا متوقف شود، فراخوانی مسدود می‌شود.                              |
| نبود مسیر تأیید | فراخوانی مسدود می‌شود، زیرا هیچ سطح تأیید متصلی نمی‌تواند آن را تعیین تکلیف کند. |

فقط تصمیم‌های دقیق `allow-once` و `allow-always` که درخواست مجاز دانسته است، اجازه اجرا می‌دهند. تصمیم‌های ناشناخته، بدشکل، نامطابق، مفقود و منقضی‌شده با حالت بسته شکست می‌خورند. فیلد قدیمی `timeoutBehavior` همچنان برای سازگاری Plugin پذیرفته می‌شود، اما منسوخ و نادیده گرفته شده است؛ آن را در هوک‌های جدید تنظیم نکنید.

`allow-always` فقط زمانی پایدار است که Plugin یا زمان اجرای درخواست‌کننده آن ماندگاری را پیاده‌سازی کند. برای هوک‌های معمولی `before_tool_call.requireApproval`،
OpenClaw مقادیر `allow-once` و `allow-always` را به‌عنوان تصمیم‌های تأیید برای فراخوانی جاری در نظر می‌گیرد و مقدار تعیین‌شده را به `onResolution` ارسال می‌کند. اگر Plugin شما `allow-always` را ارائه می‌دهد، دقیقاً مستند و پیاده‌سازی کنید که به کدام فراخوانی‌های آینده اعتماد می‌کند.

اگر هوک همچنین `params` را برگرداند، OpenClaw آن تغییرات پارامتر را فقط پس از موفقیت تأیید اعمال می‌کند. یک هوک با اولویت پایین‌تر همچنان می‌تواند پس از درخواست تأیید توسط یک هوک با اولویت بالاتر، اجرا را مسدود کند.

`allowedDecisions` دکمه‌ها و فرمان‌های نمایش‌داده‌شده به کاربر را محدود می‌کند. Gateway تلاش برای تعیین تکلیف با هر تصمیمی را که درخواست ارائه نکرده باشد، رد می‌کند.

## مسیریابی اعلان‌های تأیید

اعلان‌های تأیید می‌توانند در سطوح رابط کاربری محلی یا کانال‌های گفت‌وگویی که از مدیریت تأیید پشتیبانی می‌کنند، تعیین تکلیف شوند. برای ارسال اعلان‌های تأیید Plugin به اهداف صریح گفت‌وگو، `approvals.plugin` را پیکربندی کنید:

```json5
{
  approvals: {
    plugin: {
      enabled: true,
      mode: "targets",
      agentFilter: ["main"],
      targets: [{ channel: "slack", to: "U12345678" }],
    },
  },
}
```

`approvals.plugin` از `approvals.exec` مستقل است. فعال‌سازی ارسال تأیید اجرا، اعلان‌های تأیید Plugin را مسیریابی نمی‌کند و فعال‌سازی ارسال تأیید Plugin، سیاست اجرای میزبان را تغییر نمی‌دهد.

هنگامی که اعلان شامل متن تأیید دستی است، آن را با یکی از تصمیم‌های ارائه‌شده تعیین تکلیف کنید:

```text
/approve <id> allow-once
/approve <id> allow-always
/approve <id> deny
```

برای مدل کامل ارسال، رفتار تأیید در همان گفت‌وگو، تحویل بومی کانال و قواعد تأییدکننده مختص هر کانال، به [تأییدهای پیشرفته اجرا](/fa/tools/exec-approvals-advanced#plugin-approval-forwarding) مراجعه کنید.

## مجوزهای بومی Codex

اعلان‌های مجوز بومی Codex نیز می‌توانند از طریق تأییدهای Plugin منتقل شوند، اما مالکیت آن‌ها با هوک‌های ایجادشده توسط Plugin متفاوت است.

- درخواست‌های تأیید سرور برنامه Codex پس از بازبینی Codex از طریق OpenClaw مسیریابی می‌شوند.
- رله هوک بومی `permission_request` می‌تواند هنگام فعال‌بودن آن رله، از طریق
  `plugin.approval.request` درخواست کند.
- هنگامی که Codex مقدار `_meta.codex_approval_kind` را به‌صورت `"mcp_tool_call"` علامت‌گذاری کند، درخواست‌های تأیید ابزار MCP از طریق تأییدهای Plugin مسیریابی می‌شوند.

برای رفتار مختص Codex و قواعد بازگشت، به [زمان اجرای مهار Codex](/fa/plugins/codex-harness-runtime#native-permissions-and-mcp-elicitations) مراجعه کنید.

## عیب‌یابی

**ابزار اعلام می‌کند تأییدهای Plugin در دسترس نیستند.** هیچ رابط کاربری تأیید یا مسیر تأیید پیکربندی‌شده‌ای درخواست را نپذیرفت. یک کلاینت دارای قابلیت تأیید متصل کنید، از کانالی استفاده کنید که از `/approve` در همان گفت‌وگو پشتیبانی می‌کند، یا `approvals.plugin` را پیکربندی کنید.

**`allow-always` ظاهر می‌شود، اما فراخوانی بعدی دوباره اعلان می‌دهد.** جریان عمومی تأیید Plugin، اعتماد را برای هوک‌های دلخواه به‌طور خودکار پایدار نمی‌کند. پس از `onResolution("allow-always")`، اعتماد تحت مالکیت Plugin را در Plugin خود ماندگار کنید، یا فقط `allow-once` و `deny` را ارائه دهید.

**`/approve` تصمیم را رد می‌کند.** درخواست، `allowedDecisions` را محدود کرده است. از یکی از تصمیم‌های چاپ‌شده در اعلان استفاده کنید.

**یک اعلان Discord، Matrix، Slack یا Telegram به‌شکلی متفاوت از تأییدهای اجرا مسیریابی می‌شود.** تأییدهای Plugin و تأییدهای اجرا از پیکربندی‌های جداگانه استفاده می‌کنند و ممکن است بررسی‌های مجوزدهی متفاوتی داشته باشند. به‌جای بررسی صرفِ `approvals.exec`، `approvals.plugin` و پشتیبانی کانال از تأیید Plugin را بررسی کنید.
