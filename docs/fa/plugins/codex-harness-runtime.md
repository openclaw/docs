این صفحه قرارداد زمان اجرای نوبت‌های Codex harness را مستند می‌کند. برای راه‌اندازی و
مسیریابی، با [Codex harness](/fa/plugins/codex-harness) شروع کنید. برای فیلدهای پیکربندی،
[مرجع Codex harness](/fa/plugins/codex-harness-reference) را ببینید.

## نمای کلی

حالت Codex، PI با یک فراخوانی مدل متفاوت در زیرساخت نیست. Codex بخش بیشتری از
حلقه مدل بومی را در اختیار دارد و OpenClaw سطح‌های Plugin، ابزار، نشست و
عیب‌یابی خود را پیرامون آن مرز تطبیق می‌دهد.

OpenClaw همچنان مالک مسیریابی کانال، فایل‌های نشست، تحویل پیام قابل مشاهده،
ابزارهای پویای OpenClaw، تأییدها، تحویل رسانه و یک آینه رونوشت است.
Codex مالک رشته بومی مرجع، حلقه مدل بومی، ادامه ابزار بومی و Compaction بومی است.

## اتصال‌های رشته و تغییرات مدل

وقتی یک نشست OpenClaw به یک رشته Codex موجود متصل می‌شود، نوبت بعدی
مدل OpenAI انتخاب‌شده فعلی، سیاست تأیید، sandbox و رده سرویس را دوباره به app-server
می‌فرستد. جابه‌جایی از `openai/gpt-5.5` به
`openai/gpt-5.2` اتصال رشته را حفظ می‌کند، اما از Codex می‌خواهد با مدل
تازه انتخاب‌شده ادامه دهد.

## پاسخ‌های قابل مشاهده و Heartbeatها

وقتی یک نوبت گفت‌وگوی منبع از طریق Codex harness اجرا می‌شود، اگر استقرار
`messages.visibleReplies` را صریحاً پیکربندی نکرده باشد، پاسخ‌های قابل مشاهده به‌طور پیش‌فرض
از ابزار `message` در OpenClaw استفاده می‌کنند. عامل همچنان می‌تواند نوبت Codex خود را
به‌صورت خصوصی تمام کند؛ فقط زمانی در کانال پست می‌کند که `message(action="send")` را فراخوانی کند. برای نگه داشتن پاسخ‌های نهایی گفت‌وگوی مستقیم در
مسیر تحویل خودکار قدیمی، `messages.visibleReplies: "automatic"` را تنظیم کنید.

نوبت‌های Heartbeat در Codex نیز به‌طور پیش‌فرض `heartbeat_respond` را در کاتالوگ ابزار قابل جست‌وجوی OpenClaw دریافت می‌کنند، تا عامل بتواند ثبت کند که آیا بیدارسازی باید
بی‌صدا بماند یا بدون کدگذاری آن جریان کنترل در متن نهایی، اعلان بدهد.

راهنمای ابتکار ویژه Heartbeat به‌صورت دستور توسعه‌دهنده حالت همکاری Codex
در خود نوبت Heartbeat ارسال می‌شود. نوبت‌های گفت‌وگوی عادی به‌جای حمل فلسفه Heartbeat در پرامپت زمان اجرای معمول خود، حالت پیش‌فرض Codex را بازیابی می‌کنند.

## مرزهای hook

Codex harness سه لایه hook دارد:

| لایه                                  | مالک                     | هدف                                                                |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| hookهای Plugin در OpenClaw            | OpenClaw                 | سازگاری محصول/Plugin در سراسر PI و Codex harnessها.                |
| میان‌افزار افزونه app-server در Codex | Pluginهای بسته‌بندی‌شده OpenClaw | رفتار تطبیق‌دهنده هر نوبت پیرامون ابزارهای پویای OpenClaw.        |
| hookهای بومی Codex                    | Codex                    | چرخه عمر سطح پایین Codex و سیاست ابزار بومی از پیکربندی Codex.    |

OpenClaw از فایل‌های پروژه‌ای یا سراسری `hooks.json` در Codex برای مسیریابی
رفتار Pluginهای OpenClaw استفاده نمی‌کند. برای ابزار بومی و پل مجوز پشتیبانی‌شده،
OpenClaw پیکربندی Codex را برای هر رشته برای `PreToolUse`، `PostToolUse`،
`PermissionRequest` و `Stop` تزریق می‌کند.

وقتی تأییدهای app-server در Codex فعال باشند، یعنی `approvalPolicy` برابر
`"never"` نباشد، پیکربندی hook بومی تزریق‌شده پیش‌فرض `PermissionRequest` را حذف می‌کند تا
بازبین app-server در Codex و پل تأیید OpenClaw، ارتقاهای واقعی را پس از بازبینی مدیریت کنند.
اپراتورها می‌توانند زمانی که به رله سازگاری نیاز دارند، `permission_request` را
به‌صراحت به `nativeHookRelay.events` اضافه کنند.

hookهای دیگر Codex مانند `SessionStart` و `UserPromptSubmit` همچنان
کنترل‌های سطح Codex باقی می‌مانند. آن‌ها در قرارداد v1 به‌عنوان hookهای Plugin در OpenClaw
در معرض قرار نمی‌گیرند.

برای ابزارهای پویای OpenClaw، OpenClaw ابزار را پس از درخواست فراخوانی از سوی Codex
اجرا می‌کند، بنابراین OpenClaw رفتار Plugin و میان‌افزاری را که مالک آن است در
تطبیق‌دهنده harness اجرا می‌کند. برای ابزارهای بومی Codex، Codex مالک رکورد ابزار مرجع است.
OpenClaw می‌تواند رویدادهای انتخاب‌شده را آینه کند، اما نمی‌تواند رشته بومی Codex را بازنویسی کند مگر اینکه Codex آن عملیات را از طریق app-server یا callbackهای hook بومی
در معرض قرار دهد.

پروجکشن‌های چرخه عمر Compaction و LLM از اعلان‌های app-server در Codex
و وضعیت تطبیق‌دهنده OpenClaw می‌آیند، نه از فرمان‌های hook بومی Codex.
رویدادهای `before_compaction`، `after_compaction`، `llm_input` و
`llm_output` در OpenClaw مشاهده‌های سطح تطبیق‌دهنده هستند، نه ثبت بایت‌به‌بایت
درخواست داخلی یا payloadهای Compaction در Codex.

اعلان‌های app-server بومی `hook/started` و `hook/completed` در Codex
به‌عنوان رویدادهای عامل `codex_app_server.hook` برای مسیر رخداد و عیب‌یابی
پروژه می‌شوند. آن‌ها hookهای Plugin در OpenClaw را فراخوانی نمی‌کنند.

## مجوزهای بومی و elicitationهای MCP

برای `PermissionRequest`، OpenClaw فقط وقتی سیاست تصمیم بگیرد، تصمیم‌های صریح اجازه یا رد
را برمی‌گرداند. نتیجه بدون تصمیم، اجازه نیست. Codex آن را به‌عنوان نبود تصمیم hook
در نظر می‌گیرد و به مسیر guardian یا تأیید کاربر خودش ادامه می‌دهد.

حالت‌های تأیید app-server در Codex این hook بومی را به‌طور پیش‌فرض حذف می‌کنند. این رفتار
وقتی اعمال می‌شود که `permission_request` به‌صراحت در
`nativeHookRelay.events` گنجانده شده باشد یا یک زمان اجرای سازگاری آن را نصب کند.

وقتی یک اپراتور برای یک درخواست مجوز بومی Codex گزینهٔ `allow-always` را انتخاب می‌کند،
OpenClaw اثرانگشت دقیق ورودی provider/session/tool/cwd را برای یک
پنجرهٔ نشست محدود به خاطر می‌سپارد. تصمیم به‌خاطر سپرده‌شده عمداً فقط
بر پایهٔ تطابق دقیق است: تغییر فرمان، آرگومان‌ها، payload ابزار، یا cwd یک
تأیید تازه ایجاد می‌کند.

درخواست‌های تأیید ابزار Codex MCP از طریق جریان تأیید Plugin در OpenClaw
مسیریابی می‌شوند وقتی Codex مقدار `_meta.codex_approval_kind` را به‌صورت
`"mcp_tool_call"` علامت‌گذاری کند. اعلان‌های Codex `request_user_input` به
چت مبدأ بازگردانده می‌شوند، و پیام پیگیری بعدی در صف به آن درخواست
سرور بومی پاسخ می‌دهد، به‌جای اینکه به‌عنوان زمینهٔ اضافی هدایت شود.
درخواست‌های دیگر MCP elicitation به‌صورت بسته شکست می‌خورند.

## هدایت صف

هدایت صف اجرای فعال روی `turn/steer` سرور برنامهٔ Codex نگاشت می‌شود. با
پیش‌فرض `messages.queue.mode: "steer"`، OpenClaw پیام‌های چت صف‌شده را
برای پنجرهٔ سکوت پیکربندی‌شده دسته‌بندی می‌کند و آن‌ها را به‌ترتیب ورود
به‌صورت یک درخواست `turn/steer` ارسال می‌کند. حالت قدیمی `queue`
درخواست‌های جداگانهٔ `turn/steer` ارسال می‌کند.

نوبت‌های بازبینی Codex و Compaction دستی می‌توانند هدایت همان نوبت را رد
کنند. در آن حالت، OpenClaw وقتی حالت انتخاب‌شده اجازهٔ fallback بدهد از
صف پیگیری استفاده می‌کند. [صف هدایت](/fa/concepts/queue-steering) را ببینید.

## بارگذاری بازخورد Codex

وقتی `/diagnostics [note]` برای نشستی که از harness بومی Codex استفاده
می‌کند تأیید شود، OpenClaw همچنین برای threadهای مرتبط Codex، فراخوانی
`feedback/upload` سرور برنامهٔ Codex را انجام می‌دهد. این بارگذاری از
سرور برنامه می‌خواهد برای هر thread فهرست‌شده و زیرthreadهای ایجادشدهٔ
Codex، در صورت موجود بودن، لاگ‌ها را شامل کند.

این بارگذاری از مسیر عادی بازخورد Codex به سرورهای OpenAI عبور می‌کند.
اگر بازخورد Codex در آن سرور برنامه غیرفعال باشد، فرمان خطای سرور برنامه
را برمی‌گرداند. پاسخ کامل‌شدهٔ diagnostics کانال‌ها، شناسه‌های نشست
OpenClaw، شناسه‌های thread Codex، و فرمان‌های محلی `codex resume <thread-id>`
را برای threadهایی که ارسال شده‌اند فهرست می‌کند.

اگر تأیید را رد یا نادیده بگیرید، OpenClaw آن شناسه‌های Codex را چاپ
نمی‌کند و بازخورد Codex را ارسال نمی‌کند. این بارگذاری جایگزین export
محلی diagnostics در Gateway نمی‌شود. برای رفتار تأیید، حریم خصوصی، بستهٔ
محلی، و چت گروهی، [export diagnostics](/fa/gateway/diagnostics) را ببینید.

از `/codex diagnostics [note]` فقط زمانی استفاده کنید که مشخصاً بارگذاری
بازخورد Codex را برای thread فعلاً متصل‌شده، بدون بستهٔ کامل diagnostics
در Gateway، می‌خواهید.

## Compaction و آینهٔ رونوشت

وقتی مدل انتخاب‌شده از harness Codex استفاده می‌کند، Compaction بومی thread
به سرور برنامهٔ Codex واگذار می‌شود. OpenClaw برای تاریخچهٔ کانال، جست‌وجو،
`/new`، `/reset`، و تغییر آیندهٔ مدل یا harness، یک آینهٔ رونوشت نگه می‌دارد.

این آینه شامل prompt کاربر، متن نهایی assistant، و رکوردهای سبک استدلال یا
طرح Codex است وقتی سرور برنامه آن‌ها را منتشر کند. امروزه، OpenClaw فقط
سیگنال‌های شروع و تکمیل Compaction بومی را ثبت می‌کند. هنوز خلاصهٔ
خوانای انسانی Compaction یا فهرست قابل ممیزی از اینکه Codex پس از
Compaction کدام ورودی‌ها را نگه داشته است ارائه نمی‌کند.

چون Codex مالک thread بومی canonical است، `tool_result_persist` در حال حاضر
رکوردهای نتیجهٔ ابزار بومی Codex را بازنویسی نمی‌کند. این فقط زمانی اعمال
می‌شود که OpenClaw در حال نوشتن نتیجهٔ ابزار رونوشت نشستی باشد که مالکیت
آن با OpenClaw است.

## رسانه و تحویل

OpenClaw همچنان مالک تحویل رسانه و انتخاب provider رسانه است. تصویر،
ویدئو، موسیقی، PDF، TTS، و درک رسانه از تنظیمات provider/model متناظر
مانند `agents.defaults.imageGenerationModel`، `videoGenerationModel`،
`pdfModel`، و `messages.tts` استفاده می‌کنند.

متن، تصاویر، ویدئو، موسیقی، TTS، تأییدها، و خروجی ابزار پیام‌رسانی همچنان
از مسیر تحویل عادی OpenClaw عبور می‌کنند. تولید رسانه به PI نیاز ندارد.
وقتی Codex یک مورد تولید تصویر بومی با `savedPath` منتشر کند، OpenClaw
همان فایل دقیق را از طریق مسیر عادی reply-media ارسال می‌کند حتی اگر
نوبت Codex متن assistant نداشته باشد.
