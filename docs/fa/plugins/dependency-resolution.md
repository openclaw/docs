OpenClaw کار مربوط به وابستگی‌های Plugin را به زمان نصب/به‌روزرسانی محدود می‌کند. بارگذاری زمان اجرا
مدیرهای بسته را اجرا نمی‌کند، درخت‌های وابستگی را ترمیم نمی‌کند، یا دایرکتوری بسته‌ی OpenClaw را تغییر نمی‌دهد.

## تفکیک مسئولیت

بسته‌های Plugin مالک گراف وابستگی خود هستند:

- وابستگی‌های زمان اجرا در `dependencies` یا
  `optionalDependencies` بسته‌ی Plugin قرار می‌گیرند
- ایمپورت‌های SDK/هسته peer یا ایمپورت‌های فراهم‌شده توسط OpenClaw هستند
- Pluginهای توسعه‌ی محلی وابستگی‌های ازقبل‌نصب‌شده‌ی خود را همراه دارند
- Pluginهای npm و git در ریشه‌های بسته‌ی متعلق به OpenClaw نصب می‌شوند

OpenClaw فقط مالک چرخه‌ی عمر Plugin است:

- کشف منبع Plugin
- نصب یا به‌روزرسانی بسته وقتی صراحتا درخواست شود
- ثبت فراداده‌ی نصب
- بارگذاری نقطه‌ی ورود Plugin
- شکست با خطایی قابل اقدام وقتی وابستگی‌ها وجود ندارند

## ریشه‌های نصب

OpenClaw از ریشه‌های پایدار برای هر منبع استفاده می‌کند:

- بسته‌های npm زیر `~/.openclaw/npm` نصب می‌شوند
- بسته‌های git زیر `~/.openclaw/git` کلون می‌شوند
- نصب‌های محلی/مسیر/آرشیو بدون ترمیم وابستگی کپی یا ارجاع داده می‌شوند

نصب‌های npm در ریشه‌ی npm با این دستور اجرا می‌شوند:

```bash
npm install --prefix ~/.openclaw/npm <spec> --omit=dev --ignore-scripts --no-audit --no-fund
```

`openclaw plugins install npm-pack:<path.tgz>` از همان ریشه‌ی npm مدیریت‌شده
برای یک tarball محلی npm-pack استفاده می‌کند. OpenClaw فراداده‌ی npm موجود در tarball را می‌خواند، آن را
به‌عنوان یک وابستگی `file:` کپی‌شده به ریشه‌ی مدیریت‌شده اضافه می‌کند، نصب معمول npm را اجرا می‌کند،
و سپس پیش از اعتماد به Plugin، فراداده‌ی lockfile نصب‌شده را راستی‌آزمایی می‌کند.
این برای اثبات پذیرش بسته و نامزد انتشار در نظر گرفته شده است، جایی که یک
مصنوع بسته‌بندی محلی باید مانند مصنوع رجیستری‌ای رفتار کند که شبیه‌سازی می‌کند.

npm ممکن است وابستگی‌های گذرای وابسته را کنار بسته‌ی Plugin به
`~/.openclaw/npm/node_modules` hoist کند. OpenClaw پیش از اعتماد به نصب،
ریشه‌ی npm مدیریت‌شده را اسکن می‌کند و هنگام حذف نصب، از npm برای حذف بسته‌های مدیریت‌شده توسط npm استفاده می‌کند،
بنابراین وابستگی‌های زمان اجرای hoistشده داخل مرز پاک‌سازی مدیریت‌شده باقی می‌مانند.

Pluginهایی که `openclaw/plugin-sdk/*` را ایمپورت می‌کنند، `openclaw` را به‌عنوان وابستگی peer
اعلام می‌کنند. OpenClaw اجازه نمی‌دهد npm یک کپی جداگانه از بسته‌ی میزبان را از رجیستری
داخل ریشه‌ی مدیریت‌شده نصب کند، چون بسته‌های میزبان قدیمی می‌توانند هنگام نصب‌های بعدی Plugin
بر حل‌وفصل peer توسط npm اثر بگذارند. در عوض، پس از آنکه npm در زمان نصب، به‌روزرسانی، یا حذف نصب
ریشه‌ی مشترک را تغییر داد، OpenClaw لینک‌های Plugin-محلی `node_modules/openclaw` را برای بسته‌های نصب‌شده‌ای که
peer میزبان را اعلام می‌کنند دوباره برقرار می‌کند.

نصب‌های git مخزن را کلون یا تازه‌سازی می‌کنند، سپس اجرا می‌کنند:

```bash
npm install --omit=dev --ignore-scripts --no-audit --no-fund
```

سپس Plugin نصب‌شده از همان دایرکتوری بسته بارگذاری می‌شود، بنابراین حل‌وفصل `node_modules`
محلی بسته و والد همان‌طور کار می‌کند که برای یک بسته‌ی معمولی
Node کار می‌کند.

## Pluginهای محلی

Pluginهای محلی به‌عنوان دایرکتوری‌های تحت کنترل توسعه‌دهنده در نظر گرفته می‌شوند. OpenClaw
برای آن‌ها `npm install`، `pnpm install`، یا ترمیم وابستگی را اجرا نمی‌کند. اگر یک Plugin محلی
وابستگی‌هایی دارد، پیش از بارگذاری آن، آن‌ها را در همان Plugin نصب کنید.

Pluginهای محلی TypeScript شخص ثالث می‌توانند از مسیر اضطراری Jiti استفاده کنند. Pluginهای
JavaScript بسته‌بندی‌شده و Pluginهای داخلی bundled به‌جای Jiti از طریق
import/require بومی بارگذاری می‌شوند.

## راه‌اندازی و بارگذاری دوباره

راه‌اندازی Gateway و بارگذاری دوباره‌ی پیکربندی هرگز وابستگی‌های Plugin را نصب نمی‌کنند. آن‌ها
رکوردهای نصب Plugin را می‌خوانند، نقطه‌ی ورود را محاسبه می‌کنند، و آن را بارگذاری می‌کنند.

اگر در زمان اجرا وابستگی‌ای وجود نداشته باشد، Plugin بارگذاری نمی‌شود و خطا
باید اپراتور را به یک اصلاح صریح هدایت کند:

```bash
openclaw plugins update <id>
openclaw plugins install <source>
openclaw doctor --fix
```

`doctor --fix` می‌تواند وضعیت وابستگی legacy تولیدشده توسط OpenClaw را پاک‌سازی کند و
Pluginهای قابل دانلودی را که وقتی پیکربندی به آن‌ها ارجاع می‌دهد از رکوردهای نصب محلی جا افتاده‌اند بازیابی کند.
Doctor وابستگی‌ها را برای یک Plugin محلی که از قبل نصب شده است ترمیم نمی‌کند.

## Pluginهای bundled

Pluginهای bundled سبک و حیاتی برای هسته به‌عنوان بخشی از OpenClaw ارسال می‌شوند.
آن‌ها یا نباید درخت وابستگی زمان اجرای سنگینی داشته باشند یا باید به یک
بسته‌ی قابل دانلود روی ClawHub/npm منتقل شوند.

برای فهرست تولیدشده‌ی فعلی Pluginهایی که در بسته‌ی هسته ارسال می‌شوند، بیرونی نصب می‌شوند،
یا فقط به‌صورت source-only باقی می‌مانند، [موجودی Plugin](/fa/plugins/plugin-inventory) را ببینید.

manifestهای Pluginهای bundled نباید staging وابستگی را درخواست کنند. قابلیت‌های بزرگ یا اختیاری
Plugin باید به‌عنوان یک Plugin معمولی بسته‌بندی شوند و از همان مسیر npm/git/ClawHub مانند
Pluginهای شخص ثالث نصب شوند.

در checkoutهای سورس، OpenClaw مخزن را به‌عنوان یک monorepo مبتنی بر pnpm در نظر می‌گیرد. پس از
`pnpm install`، Pluginهای bundled از `extensions/<id>` بارگذاری می‌شوند تا وابستگی‌های workspace
محلی بسته در دسترس باشند و ویرایش‌ها مستقیما اعمال شوند. توسعه در checkout سورس فقط با pnpm پشتیبانی می‌شود؛
اجرای ساده‌ی `npm install` در ریشه‌ی مخزن راه پشتیبانی‌شده‌ای برای آماده‌سازی وابستگی‌های Pluginهای bundled نیست.

| شکل نصب                         | محل Plugin bundled                  | مالک وابستگی                                                         |
| -------------------------------- | ----------------------------------- | -------------------------------------------------------------------- |
| `npm install -g openclaw`        | درخت زمان اجرای ساخته‌شده داخل بسته | بسته‌ی OpenClaw و جریان‌های صریح نصب/به‌روزرسانی/doctor برای Plugin |
| Git checkout plus `pnpm install` | بسته‌های workspace در `extensions/<id>` | workspace مبتنی بر pnpm، شامل وابستگی‌های خود هر بسته‌ی Plugin       |
| `openclaw plugins install ...`   | ریشه‌ی مدیریت‌شده‌ی Plugin در npm/git/ClawHub | جریان نصب/به‌روزرسانی Plugin                                         |

## پاک‌سازی legacy

نسخه‌های قدیمی‌تر OpenClaw در زمان راه‌اندازی یا هنگام ترمیم doctor،
ریشه‌های وابستگی Pluginهای bundled را تولید می‌کردند. پاک‌سازی فعلی doctor وقتی از `--fix` استفاده شود
آن دایرکتوری‌ها و symlinkهای قدیمی را حذف می‌کند، از جمله ریشه‌های قدیمی `plugin-runtime-deps`، symlinkهای بسته‌ی
Node-prefix سراسری که به هدف‌های هرس‌شده‌ی `plugin-runtime-deps` اشاره می‌کنند،
manifestهای `.openclaw-runtime-deps*`، `node_modules` تولیدشده‌ی Plugin، دایرکتوری‌های
مرحله‌ی نصب، و storeهای pnpm محلی بسته. postinstall بسته‌بندی‌شده نیز
پیش از هرس ریشه‌های هدف legacy، آن symlinkهای سراسری را حذف می‌کند تا ارتقاها
ایمپورت‌های بسته‌ی ESM آویزان باقی نگذارند.

این مسیرها فقط بقایای legacy هستند. نصب‌های جدید نباید آن‌ها را ایجاد کنند.
