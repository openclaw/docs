# حل وابستگی‌های Plugin

OpenClaw کار وابستگی‌های Plugin را در زمان نصب/به‌روزرسانی نگه می‌دارد. بارگذاری در زمان اجرا
مدیر بسته اجرا نمی‌کند، درخت‌های وابستگی را ترمیم نمی‌کند، یا دایرکتوری بسته OpenClaw
را تغییر نمی‌دهد.

## تفکیک مسئولیت

بسته‌های Plugin مالک گراف وابستگی خود هستند:

- وابستگی‌های زمان اجرا در `dependencies` یا
  `optionalDependencies` بسته Plugin قرار می‌گیرند
- ایمپورت‌های SDK/هسته، peer یا ایمپورت‌های تامین‌شده توسط OpenClaw هستند
- Pluginهای توسعه محلی وابستگی‌های از پیش نصب‌شده خودشان را می‌آورند
- Pluginهای npm و git در ریشه‌های بسته تحت مالکیت OpenClaw نصب می‌شوند

OpenClaw فقط مالک چرخه عمر Plugin است:

- کشف منبع Plugin
- نصب یا به‌روزرسانی بسته وقتی صراحتا درخواست شود
- ثبت فراداده نصب
- بارگذاری نقطه ورود Plugin
- شکست با خطایی قابل اقدام وقتی وابستگی‌ها غایب باشند

## ریشه‌های نصب

OpenClaw از ریشه‌های پایدار برای هر منبع استفاده می‌کند:

- بسته‌های npm زیر `~/.openclaw/npm` نصب می‌شوند
- بسته‌های git زیر `~/.openclaw/git` کلون می‌شوند
- نصب‌های محلی/مسیر/آرشیو بدون ترمیم وابستگی کپی یا ارجاع داده می‌شوند

نصب‌های npm در ریشه npm با این دستور اجرا می‌شوند:

```bash
npm install --prefix ~/.openclaw/npm <spec> --omit=dev --ignore-scripts --no-audit --no-fund
```

`openclaw plugins install npm-pack:<path.tgz>` برای یک تاربال محلی npm-pack از همان ریشه npm مدیریت‌شده
استفاده می‌کند. OpenClaw فراداده npm تاربال را می‌خواند، آن را
به‌عنوان یک وابستگی کپی‌شده `file:` به ریشه مدیریت‌شده اضافه می‌کند، نصب عادی npm را اجرا می‌کند،
و سپس پیش از اعتماد به Plugin، فراداده lockfile نصب‌شده را تایید می‌کند.
این برای اثبات پذیرش بسته و نامزد انتشار در نظر گرفته شده است، جایی که یک
مصنوع بسته‌بندی محلی باید مانند مصنوع رجیستری که شبیه‌سازی می‌کند رفتار کند.

npm ممکن است وابستگی‌های انتقالی را کنار
بسته Plugin به `~/.openclaw/npm/node_modules` منتقل کند. OpenClaw پیش از اعتماد به
نصب، ریشه npm مدیریت‌شده را اسکن می‌کند و هنگام حذف نصب، از npm برای حذف بسته‌های مدیریت‌شده توسط npm استفاده می‌کند، بنابراین وابستگی‌های زمان اجرای منتقل‌شده
داخل مرز پاک‌سازی مدیریت‌شده باقی می‌مانند.

Pluginهایی که `openclaw/plugin-sdk/*` را ایمپورت می‌کنند، `openclaw` را به‌عنوان وابستگی peer
اعلام می‌کنند. OpenClaw اجازه نمی‌دهد npm یک کپی جداگانه از بسته میزبان را از رجیستری
در ریشه مدیریت‌شده نصب کند، چون بسته‌های میزبان قدیمی می‌توانند بر
حل peer توسط npm در نصب‌های بعدی Plugin اثر بگذارند. در عوض، پس از آنکه npm
ریشه مشترک را هنگام نصب، به‌روزرسانی، یا حذف نصب تغییر داد، OpenClaw دوباره
لینک‌های Plugin-محلی `node_modules/openclaw` را برای بسته‌های نصب‌شده‌ای که
peer میزبان را اعلام کرده‌اند برقرار می‌کند.

نصب‌های git مخزن را کلون یا تازه‌سازی می‌کنند، سپس اجرا می‌کنند:

```bash
npm install --omit=dev --ignore-scripts --no-audit --no-fund
```

سپس Plugin نصب‌شده از همان دایرکتوری بسته بارگذاری می‌شود، بنابراین حل
وابستگی‌های package-local و والد `node_modules` همان‌طور کار می‌کند که برای یک بسته عادی
Node کار می‌کند.

## Pluginهای محلی

Pluginهای محلی به‌عنوان دایرکتوری‌های تحت کنترل توسعه‌دهنده تلقی می‌شوند. OpenClaw برای آن‌ها
`npm install`، `pnpm install`، یا ترمیم وابستگی اجرا نمی‌کند. اگر یک Plugin
محلی وابستگی دارد، پیش از بارگذاری آن، وابستگی‌ها را در همان Plugin نصب کنید.

Pluginهای محلی TypeScript شخص ثالث می‌توانند از مسیر اضطراری Jiti استفاده کنند. Pluginهای
بسته‌بندی‌شده JavaScript و Pluginهای داخلی همراه‌شده به‌جای Jiti از طریق
import/require بومی بارگذاری می‌شوند.

## راه‌اندازی و بارگذاری دوباره

راه‌اندازی Gateway و بارگذاری دوباره پیکربندی هرگز وابستگی‌های Plugin را نصب نمی‌کنند. آن‌ها
رکوردهای نصب Plugin را می‌خوانند، نقطه ورود را محاسبه می‌کنند، و آن را بارگذاری می‌کنند.

اگر وابستگی‌ای در زمان اجرا غایب باشد، Plugin بارگذاری نمی‌شود و خطا
باید اپراتور را به یک اصلاح صریح هدایت کند:

```bash
openclaw plugins update <id>
openclaw plugins install <source>
openclaw doctor --fix
```

`doctor --fix` می‌تواند وضعیت وابستگی قدیمی تولیدشده توسط OpenClaw را پاک‌سازی کند و Pluginهای
قابل دانلودی را که هنگام ارجاع پیکربندی به آن‌ها از رکوردهای نصب محلی
غایب هستند بازیابی کند. Doctor وابستگی‌های یک Plugin محلی از پیش نصب‌شده
را ترمیم نمی‌کند.

## Pluginهای همراه‌شده

Pluginهای سبک و حیاتی برای هسته به‌عنوان بخشی از OpenClaw ارسال می‌شوند.
آن‌ها یا نباید درخت وابستگی زمان اجرای سنگین داشته باشند، یا باید به یک
بسته قابل دانلود در ClawHub/npm منتقل شوند.

برای فهرست تولیدشده فعلی Pluginهایی که در بسته هسته ارسال می‌شوند، بیرونی نصب
می‌شوند، یا فقط-منبع باقی می‌مانند، [فهرست Plugin](/fa/plugins/plugin-inventory) را ببینید.

مانیفست‌های Plugin همراه‌شده نباید staging وابستگی درخواست کنند. قابلیت‌های بزرگ یا اختیاری
Plugin باید به‌عنوان یک Plugin عادی بسته‌بندی شوند و از طریق
همان مسیر npm/git/ClawHub مانند Pluginهای شخص ثالث نصب شوند.

در checkoutهای منبع، OpenClaw مخزن را به‌عنوان یک مونورپوی pnpm در نظر می‌گیرد. پس از
`pnpm install`، Pluginهای همراه‌شده از `extensions/<id>` بارگذاری می‌شوند تا وابستگی‌های
workspace package-local در دسترس باشند و ویرایش‌ها مستقیما اعمال شوند. توسعه در checkout منبع فقط با pnpm پشتیبانی می‌شود؛ اجرای ساده `npm install` در ریشه مخزن
روش پشتیبانی‌شده‌ای برای آماده‌سازی وابستگی‌های Plugin همراه‌شده نیست.

| شکل نصب                         | مکان Plugin همراه‌شده                 | مالک وابستگی                                                         |
| -------------------------------- | ------------------------------------- | -------------------------------------------------------------------- |
| `npm install -g openclaw`        | درخت زمان اجرای ساخته‌شده داخل بسته  | بسته OpenClaw و جریان‌های صریح نصب/به‌روزرسانی/doctor برای Plugin   |
| Git checkout plus `pnpm install` | بسته‌های workspace در `extensions/<id>` | workspace مربوط به pnpm، شامل وابستگی‌های خود هر بسته Plugin        |
| `openclaw plugins install ...`   | ریشه Plugin مدیریت‌شده npm/git/ClawHub | جریان نصب/به‌روزرسانی Plugin                                        |

## پاک‌سازی قدیمی

نسخه‌های قدیمی‌تر OpenClaw ریشه‌های وابستگی Plugin همراه‌شده را هنگام راه‌اندازی یا
در طول ترمیم doctor تولید می‌کردند. پاک‌سازی doctor فعلی هنگام استفاده از `--fix`
آن دایرکتوری‌ها و symlinkهای کهنه را حذف می‌کند، از جمله ریشه‌های قدیمی
`plugin-runtime-deps`، symlinkهای بسته global Node-prefix که به مقصدهای
`plugin-runtime-deps` هرس‌شده اشاره می‌کنند، مانیفست‌های `.openclaw-runtime-deps*`، Plugin تولیدشده
`node_modules`، دایرکتوری‌های install stage، و storeهای package-local مربوط به pnpm. postinstall بسته‌بندی‌شده نیز
پیش از هرس ریشه‌های هدف قدیمی، آن symlinkهای global را حذف می‌کند تا ارتقاها
ایمپورت‌های بسته ESM آویزان باقی نگذارند.

این مسیرها فقط بقایای قدیمی هستند. نصب‌های جدید نباید آن‌ها را ایجاد کنند.
