# `openclaw devices`

درخواست‌های جفت‌سازی دستگاه و توکن‌های محدود به دستگاه را مدیریت کنید.

## فرمان‌ها

### `openclaw devices list`

درخواست‌های جفت‌سازی در انتظار و دستگاه‌های جفت‌شده را فهرست کنید.

```
openclaw devices list
openclaw devices list --json
```

خروجی درخواست در انتظار، دسترسی درخواست‌شده را کنار دسترسی تأییدشده فعلی دستگاه نشان می‌دهد، زمانی که دستگاه از قبل جفت شده باشد. این کار ارتقاهای محدوده/نقش را صریح می‌کند، به‌جای اینکه شبیه از دست رفتن جفت‌سازی به نظر برسد.

### `openclaw devices remove <deviceId>`

یک ورودی دستگاه جفت‌شده را حذف کنید.

وقتی با توکن دستگاه جفت‌شده احراز هویت شده‌اید، فراخواننده‌های غیرمدیر فقط می‌توانند ورودی دستگاه **خودشان** را حذف کنند. حذف دستگاهی دیگر به `operator.admin` نیاز دارد.

```
openclaw devices remove <deviceId>
openclaw devices remove <deviceId> --json
```

### `openclaw devices clear --yes [--pending]`

دستگاه‌های جفت‌شده را به‌صورت گروهی پاک کنید.

```
openclaw devices clear --yes
openclaw devices clear --yes --pending
openclaw devices clear --yes --pending --json
```

### `openclaw devices approve [requestId] [--latest]`

یک درخواست جفت‌سازی دستگاه در انتظار را با `requestId` دقیق تأیید کنید. اگر `requestId` حذف شود یا `--latest` ارسال شود، OpenClaw فقط درخواست در انتظار انتخاب‌شده را چاپ می‌کند و خارج می‌شود؛ پس از بررسی جزئیات، تأیید را با شناسه دقیق درخواست دوباره اجرا کنید.

<Note>
اگر دستگاه با جزئیات احراز هویت تغییرکرده دوباره برای جفت‌سازی تلاش کند (نقش، محدوده‌ها، یا کلید عمومی)، OpenClaw ورودی در انتظار قبلی را جایگزین می‌کند و یک `requestId` جدید صادر می‌کند. درست پیش از تأیید، `openclaw devices list` را اجرا کنید تا از شناسه فعلی استفاده کنید.
</Note>

اگر دستگاه از قبل جفت شده باشد و محدوده‌های گسترده‌تر یا نقش گسترده‌تری درخواست کند، OpenClaw تأیید موجود را در جای خود نگه می‌دارد و یک درخواست ارتقای در انتظار جدید ایجاد می‌کند. ستون‌های `Requested` و `Approved` را در `openclaw devices list` بررسی کنید، یا از `openclaw devices approve --latest` استفاده کنید تا ارتقای دقیق را پیش از تأیید پیش‌نمایش کنید.

اگر Gateway به‌صراحت با `gateway.nodes.pairing.autoApproveCidrs` پیکربندی شده باشد، درخواست‌های بار اول `role: node` از IPهای مشتری مطابق می‌توانند پیش از ظاهر شدن در این فهرست تأیید شوند. این سیاست به‌صورت پیش‌فرض غیرفعال است و هرگز برای مشتری‌های اپراتور/مرورگر یا درخواست‌های ارتقا اعمال نمی‌شود.

```
openclaw devices approve
openclaw devices approve <requestId>
openclaw devices approve --latest
```

### `openclaw devices reject <requestId>`

یک درخواست جفت‌سازی دستگاه در انتظار را رد کنید.

```
openclaw devices reject <requestId>
```

### `openclaw devices rotate --device <id> --role <role> [--scope <scope...>]`

یک توکن دستگاه را برای یک نقش مشخص بچرخانید (با امکان به‌روزرسانی محدوده‌ها).
نقش هدف باید از قبل در قرارداد جفت‌سازی تأییدشده آن دستگاه وجود داشته باشد؛ چرخش نمی‌تواند یک نقش تأییدنشده جدید صادر کند.
اگر `--scope` را حذف کنید، اتصال‌های مجدد بعدی با توکن چرخانده‌شده ذخیره‌شده، از محدوده‌های تأییدشده کش‌شده همان توکن دوباره استفاده می‌کنند. اگر مقدارهای صریح `--scope` را ارسال کنید، همان‌ها به مجموعه محدوده ذخیره‌شده برای اتصال‌های مجدد آینده با توکن کش‌شده تبدیل می‌شوند.
فراخواننده‌های غیرمدیرِ دستگاه جفت‌شده فقط می‌توانند توکن دستگاه **خودشان** را بچرخانند.
مجموعه محدوده توکن هدف باید در محدوده‌های اپراتورِ خود نشست فراخواننده باقی بماند؛ چرخش نمی‌تواند توکن اپراتور گسترده‌تری از آنچه فراخواننده از قبل دارد صادر یا حفظ کند.

```
openclaw devices rotate --device <deviceId> --role operator --scope operator.read --scope operator.write
```

فراداده چرخش را به‌صورت JSON برمی‌گرداند. اگر فراخواننده هنگام احراز هویت با همان توکن دستگاه، توکن خودش را بچرخاند، پاسخ شامل توکن جایگزین نیز می‌شود تا مشتری بتواند پیش از اتصال دوباره آن را پایدار کند. چرخش‌های مشترک/مدیر، توکن حامل را بازتاب نمی‌دهند.

### `openclaw devices revoke --device <id> --role <role>`

یک توکن دستگاه را برای یک نقش مشخص باطل کنید.

فراخواننده‌های غیرمدیرِ دستگاه جفت‌شده فقط می‌توانند توکن دستگاه **خودشان** را باطل کنند. باطل کردن توکن دستگاهی دیگر به `operator.admin` نیاز دارد.
مجموعه محدوده توکن هدف نیز باید در محدوده‌های اپراتورِ خود نشست فراخواننده جا بگیرد؛ فراخواننده‌هایی که فقط جفت‌سازی دارند نمی‌توانند توکن‌های اپراتور مدیر/نوشتن را باطل کنند.

```
openclaw devices revoke --device <deviceId> --role node
```

نتیجه ابطال را به‌صورت JSON برمی‌گرداند.

## گزینه‌های رایج

- `--url <url>`: URL وب‌سوکت Gateway (وقتی پیکربندی شده باشد، به‌صورت پیش‌فرض از `gateway.remote.url` استفاده می‌کند).
- `--token <token>`: توکن Gateway (در صورت نیاز).
- `--password <password>`: گذرواژه Gateway (احراز هویت با گذرواژه).
- `--timeout <ms>`: مهلت زمانی RPC.
- `--json`: خروجی JSON (برای اسکریپت‌نویسی توصیه می‌شود).

<Warning>
وقتی `--url` را تنظیم می‌کنید، CLI به اعتبارنامه‌های پیکربندی یا محیط برنمی‌گردد. `--token` یا `--password` را به‌صراحت ارسال کنید. نبود اعتبارنامه‌های صریح یک خطا است.
</Warning>

## یادداشت‌ها

- چرخش توکن یک توکن جدید برمی‌گرداند (حساس). با آن مانند یک راز رفتار کنید.
- این فرمان‌ها به محدوده `operator.pairing` (یا `operator.admin`) نیاز دارند.
- `gateway.nodes.pairing.autoApproveCidrs` یک سیاست اختیاری Gateway فقط برای جفت‌سازی تازه دستگاه node است؛ اختیار تأیید CLI را تغییر نمی‌دهد.
- چرخش و ابطال توکن درون مجموعه نقش جفت‌سازی تأییدشده و خط مبنای محدوده تأییدشده برای همان دستگاه باقی می‌مانند. یک ورودی توکن کش‌شده سرگردان، هدف مدیریت توکن اعطا نمی‌کند.
- برای نشست‌های توکن دستگاه جفت‌شده، مدیریت بین‌دستگاهی فقط برای مدیر است: `remove`، `rotate`، و `revoke` فقط برای خود دستگاه مجازند، مگر اینکه فراخواننده `operator.admin` داشته باشد.
- تغییر توکن نیز به محدوده فراخواننده محدود است: نشستی که فقط جفت‌سازی دارد نمی‌تواند توکنی را بچرخاند یا باطل کند که در حال حاضر `operator.admin` یا `operator.write` دارد.
- `devices clear` عمداً با `--yes` محافظت می‌شود.
- اگر محدوده جفت‌سازی روی local loopback در دسترس نباشد (و هیچ `--url` صریحی ارسال نشده باشد)، فهرست/تأیید می‌تواند از جایگزین جفت‌سازی محلی استفاده کند.
- `devices approve` پیش از صدور توکن‌ها به شناسه درخواست صریح نیاز دارد؛ حذف `requestId` یا ارسال `--latest` فقط جدیدترین درخواست در انتظار را پیش‌نمایش می‌کند.

## فهرست بررسی بازیابی انحراف توکن

از این مورد زمانی استفاده کنید که Control UI یا مشتری‌های دیگر همچنان با `AUTH_TOKEN_MISMATCH` یا `AUTH_DEVICE_TOKEN_MISMATCH` شکست می‌خورند.

1. منبع فعلی توکن gateway را تأیید کنید:

```bash
openclaw config get gateway.auth.token
```

2. دستگاه‌های جفت‌شده را فهرست کنید و شناسه دستگاه آسیب‌دیده را مشخص کنید:

```bash
openclaw devices list
```

3. توکن اپراتور را برای دستگاه آسیب‌دیده بچرخانید:

```bash
openclaw devices rotate --device <deviceId> --role operator
```

4. اگر چرخش کافی نیست، جفت‌سازی کهنه را حذف کنید و دوباره تأیید کنید:

```bash
openclaw devices remove <deviceId>
openclaw devices list
openclaw devices approve <requestId>
```

5. اتصال مشتری را با توکن/گذرواژه مشترک فعلی دوباره امتحان کنید.

یادداشت‌ها:

- تقدم معمول احراز هویت در اتصال مجدد، ابتدا توکن/گذرواژه مشترک صریح، سپس `deviceToken` صریح، سپس توکن دستگاه ذخیره‌شده، و سپس توکن راه‌اندازی اولیه است.
- بازیابی مورداعتماد `AUTH_TOKEN_MISMATCH` می‌تواند موقتاً هم توکن مشترک و هم توکن دستگاه ذخیره‌شده را برای یک تلاش مجدد محدود با هم ارسال کند.
