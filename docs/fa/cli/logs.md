# `openclaw logs`

گزارش‌های فایل Gateway را از طریق RPC به‌صورت زنده دنبال می‌کند. در حالت راه‌دور کار می‌کند.

## گزینه‌ها

- `--limit <n>`: حداکثر تعداد خطوط گزارش برای بازگرداندن (پیش‌فرض `200`)
- `--max-bytes <n>`: حداکثر تعداد بایت برای خواندن از فایل گزارش (پیش‌فرض `250000`)
- `--follow`: جریان گزارش را به‌صورت زنده دنبال می‌کند
- `--interval <ms>`: فاصلهٔ زمانی نظرسنجی هنگام دنبال‌کردن (پیش‌فرض `1000`)
- `--json`: رویدادهای JSON جداشده با خط جدید را خروجی می‌دهد
- `--plain`: خروجی متن ساده بدون قالب‌بندی سبک‌دار
- `--no-color`: رنگ‌های ANSI را غیرفعال می‌کند
- `--local-time`: مُهرهای زمانی را در منطقهٔ زمانی محلی شما نمایش می‌دهد (پیش‌فرض)
- `--utc`: مُهرهای زمانی را به وقت UTC نمایش می‌دهد

## گزینه‌های مشترک RPC در Gateway

- `--url <url>`: نشانی WebSocket مربوط به Gateway
- `--token <token>`: توکن Gateway
- `--timeout <ms>`: مهلت زمانی بر حسب میلی‌ثانیه (پیش‌فرض `30000`)
- `--expect-final`: وقتی فراخوانی Gateway توسط عامل پشتیبانی می‌شود، منتظر پاسخ نهایی می‌ماند

ارسال `--url` باعث می‌شود اعتبارنامه‌های پیکربندی به‌طور خودکار اعمال نشوند؛ اگر Gateway مقصد به احراز هویت نیاز دارد، `--token` را صریحاً وارد کنید.

## مثال‌ها

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## رفتار بازگشت جایگزین و بازیابی

- اگر Gateway ضمنی local loopback درخواست جفت‌سازی کند، هنگام اتصال بسته شود، یا پیش از پاسخ‌دادن `logs.tail` مهلت آن تمام شود، `openclaw logs` به‌طور خودکار به گزارش فایل پیکربندی‌شدهٔ Gateway بازمی‌گردد. مقصدهای صریح `--url` هرگز از این بازگشت جایگزین استفاده نمی‌کنند.
- پس از خرابی RPC در Gateway محلی ضمنی، `--follow` به فایل پیکربندی‌شده بازنمی‌گردد؛ زیرا یک فایل قدیمی در کنار آن می‌تواند دنبال‌کردن زنده را گمراه‌کننده کند. در Linux، در صورت دسترس‌بودن، به‌جای آن از دفتر ثبت Gateway در user-systemd فعال بر اساس PID استفاده می‌کند (منبع انتخاب‌شده را چاپ می‌کند)؛ در غیر این صورت، تلاش برای اتصال به Gateway زنده را ادامه می‌دهد.
- هنگام `--follow`، قطع‌های موقت (بسته‌شدن WebSocket، پایان مهلت زمانی، قطع اتصال) باعث اتصال مجدد خودکار با تأخیر تصاعدی می‌شوند: حداکثر ۸ تلاش مجدد، با سقف ۳۰ ثانیه فاصله میان تلاش‌ها. در هر تلاش مجدد، یک هشدار در stderr چاپ می‌شود و پس از موفقیت یک نظرسنجی، اعلان `[logs] gateway reconnected` یک‌بار چاپ می‌شود. در حالت `--json`، هر دو به‌صورت رکوردهای `{"type":"notice"}` در stderr خروجی داده می‌شوند. خطاهای بازیابی‌ناپذیر (شکست احراز هویت، پیکربندی نامعتبر) همچنان بلافاصله باعث خروج می‌شوند.
- در حالت `--follow --json`، تغییرات منبع گزارش به‌صورت رکوردهای `{"type":"meta"}` خروجی داده می‌شوند. مکان‌نماها را برای هر `sourceKind` جداگانه ردیابی کنید: یک جریان می‌تواند از خروجی فایل Gateway (`sourceKind: "file"`) به بازگشت جایگزین دفتر ثبت محلی (`sourceKind: "journal"`،‏ `localFallback: true`، همراه با `service.pid`/`service.unit`) منتقل شود و پس از بازیابی دوباره به خروجی فایل Gateway بازگردد. یک منبع یا مکان‌نمای ثابت را برای کل نشست فرض نکنید و هنگام بازپخش مکان‌نمای فایل Gateway در فرایند بازیابی، خطوط هم‌پوشان را بپذیرید.
