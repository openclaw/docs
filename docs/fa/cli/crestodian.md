# `openclaw crestodian`

Crestodian مکالمه‌ای، عامل محلی OpenClaw برای راه‌اندازی، تعمیر و پیکربندی است. این عامل فقط پس از آن شروع می‌شود که مدل پیش‌فرض مؤثر یک نوبت واقعی را تکمیل کند. در نصب‌های تازه، ابتدا استنتاج برقرار می‌شود؛ پیکربندی نادرست در مسیر کلاسیک doctor باقی می‌ماند.

## چه زمانی شروع می‌شود

اجرای `openclaw` بدون زیرفرمان، بر اساس وضعیت پیکربندی مسیریابی می‌شود:

- اگر پیکربندی وجود نداشته باشد، یا وجود داشته باشد اما فاقد تنظیمات ثبت‌شده باشد (خالی، یا فقط دارای کلیدهای `$schema`/`meta`): فرایند هدایت‌شده آغازین را با راستی‌آزمایی زنده هوش مصنوعی شروع می‌کند.
- اگر پیکربندی وجود داشته باشد اما اعتبارسنجی آن ناموفق باشد: فرایند کلاسیک آغازین را شروع می‌کند که مشکلات را گزارش می‌دهد و شما را به `openclaw doctor` هدایت می‌کند.
- اگر پیکربندی وجود داشته و معتبر باشد: TUI عادی عامل را باز می‌کند. یک Gateway پیکربندی‌شده و در دسترس که عامل پیش‌فرض آن مدل دارد، بدون فرایند آغازین یا Crestodian مستقیماً به آن رابط کاربری می‌رود. برای دسترسی بعدی به Crestodian، درون TUI از `/crestodian` استفاده کنید یا مستقیماً `openclaw crestodian` را اجرا کنید.

اجرای `openclaw crestodian` ابتدا مدل پیش‌فرض پیکربندی‌شده را به‌صورت زنده آزمایش می‌کند. یک نوبت موفق، Crestodian را شروع می‌کند. شکست تعاملی، راه‌اندازی هدایت‌شده استنتاج را باز می‌کند و پس از موفقیت یک گزینه، کنترل را به Crestodian می‌سپارد. درخواست‌های تک‌مرحله‌ای، JSON و سایر درخواست‌های غیرتعاملی، هنگامی که استنتاج در دسترس نباشد با دستورالعمل اجرای `openclaw onboard` شکست می‌خورند. `openclaw --help` و `openclaw --version` مسیرهای سریع عادی خود را حفظ می‌کنند.

اجرای غیرتعاملی `openclaw` بدون زیرفرمان (بدون TTY)، به‌جای چاپ راهنمای ریشه با پیامی کوتاه خارج می‌شود: در نصب تازه یا نامعتبر، به فرایند غیرتعاملی آغازین اشاره می‌کند و هنگامی که پیکربندی معتبر است، به `openclaw agent --local ...` هدایت می‌کند.

`openclaw onboard --modern` همچنان نام مستعار سازگاری برای Crestodian است، اما از همان دروازه استنتاج استفاده می‌کند: استنتاج فعال، گفتگو را باز می‌کند؛ شکست‌های تعاملی، راه‌اندازی هدایت‌شده استنتاج را شروع می‌کنند؛ و شکست‌های غیرتعاملی با راهنمای فرایند آغازین خارج می‌شوند. `openclaw onboard --classic` جادوگر کامل گام‌به‌گام را باز می‌کند.

## آنچه Crestodian نمایش می‌دهد

Crestodian تعاملی، همان پوسته TUI مربوط به `openclaw tui` را با یک بخش پشتی گفتگوی Crestodian باز می‌کند. پیام خوشامد آغازین شامل موارد زیر است:

- اعتبار پیکربندی و عامل پیش‌فرض
- مدل راستی‌آزمایی‌شده‌ای که Crestodian استفاده می‌کند
- دسترس‌پذیری Gateway بر اساس نخستین کاوش هنگام شروع
- اقدام پیشنهادی بعدی برای اشکال‌زدایی

این فرایند اسرار را نمایش نمی‌دهد و صرفاً برای شروع، فرمان‌های CLI مربوط به Plugin را بارگیری نمی‌کند.

برای موجودی تفصیلی از `status` استفاده کنید: مسیر پیکربندی، مسیرهای مستندات/منبع، کاوش‌های CLI محلی، وجود کلید/توکن، عامل‌ها، مدل و جزئیات Gateway.

Crestodian از همان کشف مرجع عامل‌های عادی استفاده می‌کند: در یک دریافت Git، به `docs/` محلی و درخت منبع اشاره می‌کند؛ در نصب npm، از مستندات همراه استفاده می‌کند و به [https://github.com/openclaw/openclaw](https://github.com/openclaw/openclaw) پیوند می‌دهد، با این راهنمایی که وقتی مستندات کافی نیستند، منبع را بررسی کنید.

## نمونه‌ها

```bash
openclaw
openclaw crestodian
openclaw crestodian --json
openclaw crestodian --message "models"
openclaw crestodian --message "validate config"
openclaw crestodian --message "setup workspace ~/Projects/work" --yes
openclaw crestodian --message "set default model openai/gpt-5.6" --yes
openclaw onboard --modern
```

درون TUI مربوط به Crestodian:

```text
status
health
doctor
validate config
setup
setup workspace ~/Projects/work
config set gateway.port 19001
config set-ref gateway.auth.token env OPENCLAW_GATEWAY_TOKEN
gateway status
restart gateway
agents
create agent work workspace ~/Projects/work
models
configure model provider
set default model openai/gpt-5.6
channels
channel info slack
connect slack
open channel wizard for slack
plugins list
plugins search slack
plugin install clawhub:openclaw-codex-app-server
talk to work agent
talk to agent for ~/Projects/work
audit
quit
```

## عملیات و تأیید

Crestodian به‌جای ویرایش موردی پیکربندی، از عملیات نوع‌دار استفاده می‌کند.

عملیات فقط‌خواندنی بلافاصله اجرا می‌شوند: نمایش نمای کلی، فهرست عامل‌ها، فهرست Pluginهای نصب‌شده، جست‌وجوی Pluginهای ClawHub، نمایش وضعیت مدل/بخش پشتی، اجرای بررسی‌های وضعیت/سلامت، بررسی دسترس‌پذیری Gateway، اجرای doctor بدون اصلاحات تعاملی، اعتبارسنجی پیکربندی و نمایش مسیر گزارش ممیزی.

شروع راه‌اندازی هدایت‌شده کانال (`connect telegram`) نیز بلافاصله اجرا می‌شود. جادوگر آن پاسخ‌های صریح را جمع‌آوری می‌کند و مالک نوشتن تغییرات حاصل است.

عملیات پایدار به تأیید مکالمه‌ای نیاز دارند (یا برای فرمان مستقیم به `--yes`): نوشتن پیکربندی، `config set`، `config set-ref`، راه‌اندازی اولیه setup/فرایند آغازین، تغییر مدل پیش‌فرض، شروع/توقف/راه‌اندازی مجدد Gateway، ایجاد عامل‌ها و نصب Pluginها.

تعمیرات doctor درون Crestodian در دسترس نیستند، زیرا ممکن است ارائه‌دهنده، احراز هویت یا مسیر استنتاج عامل پیش‌فرضی را که نشست بر آن متکی است بازنویسی کنند. از Crestodian خارج شوید و `openclaw doctor --fix` را در پایانه اجرا کنید. `doctor` فقط‌خواندنی همچنان درون Crestodian در دسترس است.

عامل‌های جدید، مسیر استنتاج پیش‌فرض راستی‌آزمایی‌شده زنده را به ارث می‌برند. شناسه عامل `crestodian` برای متولی مجازی دارای امتیاز رزرو شده است و نمی‌توان آن را به‌عنوان عامل عادی ایجاد کرد.

`config set` و `config set-ref` نمی‌توانند وضعیت مسیر استنتاج را تغییر دهند؛ از جمله اعتبارنامه‌های ارائه‌دهنده استنتاج، `auth.*` سطح بالا، فهرست‌های مدل، بخش‌های پشتی CLI، مسیرهای مدل پیش‌فرض/مختص عامل، پارامترها/ابزارهای عامل یا `tools.*` ریشه. نوشتن خام در `env.*`، `secrets.*`، `plugins.*` و `$include` نیز رد می‌شود، زیرا می‌تواند تفکیک اعتبارنامه یا فعال‌سازی ارائه‌دهنده را جایگزین کند. احراز هویت Gateway و کانال همچنان سطوح عادی پیکربندی هستند. برای مسیری که از پیش پیکربندی شده است، از گردش‌کارهای نوع‌دار Plugin/کانال و `set default model <provider/model>` استفاده کنید؛ این فرمان پیش از ذخیره، مسیر را به‌صورت زنده آزمایش می‌کند. برای پیکربندی یا تعمیر دسترسی ارائه‌دهنده/احراز هویت، از Crestodian خارج شوید و `openclaw onboard` را اجرا کنید.

حذف نصب Plugin درون Crestodian رد می‌شود، زیرا حذف Plugin ارائه‌دهنده ممکن است مسیر استنتاجی را که نشست بر آن متکی است غیرفعال کند. از Crestodian خارج شوید و `openclaw plugins uninstall <id>` را در پایانه اجرا کنید.

تأیید با کلمات خودتان انجام می‌شود: پاسخ‌های بدون ابهام ("بله"، "حتماً"، "انجامش بده"، "الان نه") بر اساس یک فهرست بسته و قطعی تفسیر می‌شوند. هنگامی که مسیر پیکربندی‌شده از فراخوانی تکمیل جداگانه پشتیبانی می‌کند، پاسخ‌های دیگر فقط بر اساس پیام شما و پیشنهاد معلق دسته‌بندی می‌شوند — هرگز توسط خود مدل مکالمه، زیرا مدل نمی‌تواند خودش را تأیید کند. پاسخ‌های دسته‌بندی‌نشده یا مبهم، پیشنهاد را معلق نگه می‌دارند و مکالمه دوباره درخواست می‌کند.

نوشتن‌های اعمال‌شده در `~/.openclaw/audit/crestodian.jsonl` ثبت می‌شوند. کشف ممیزی نمی‌شود؛ فقط عملیات اعمال‌شده و نوشتن‌ها ثبت می‌شوند.

راه‌اندازی کانال می‌تواند تا رسیدن به یک راز، به‌صورت مکالمه میزبانی‌شده اجرا شود. TUI محلی Crestodian پاسخ‌های حساس جادوگر را نمی‌پذیرد، زیرا ورودی گفتگوی پایانه قابل مشاهده است. این رابط بلافاصله `open channel wizard` را پیشنهاد می‌دهد و کانال انتخاب‌شده را به جادوگر پایانه‌ای با ورودی پوشیده منتقل می‌کند؛ همچنین می‌توانید بعداً `openclaw channels add --channel <channel>` را اجرا کنید.

### تغییر به راه‌اندازی کانال با ورودی پوشیده

گفتگوی محلی می‌تواند کنترل را به جادوگر کانال با ورودی پوشیده واگذار کند:

```text
open channel wizard for slack
channel info slack
```

`open channel wizard for <channel>` پس از بسته‌شدن TUI گفتگو، راه‌اندازی کانال با ورودی پوشیده را باز می‌کند. ابتدا از `channel info <channel>` برای مشاهده برچسب کانال، وضعیت راه‌اندازی، خلاصه پیش‌نیازها و پیوند مستندات استفاده کنید.

Crestodian هرگز دسترسی ارائه‌دهنده/احراز هویت را از درون نشست خودش تغییر نمی‌دهد: نشست از قبل به آن مسیر استنتاج وابسته است. برای راه‌اندازی یا تعمیر ارائه‌دهنده مدل، `configure model provider` بدون شروع جادوگر یا نوشتن پیکربندی، راهنمای خروج/فرایند آغازین را بازمی‌گرداند. از Crestodian خارج شوید و `openclaw onboard` را اجرا کنید؛ فرایند آغازین، اعتبارنامه‌ها را مرحله‌بندی می‌کند و فقط مسیری را ذخیره می‌کند که یک نوبت واقعی زنده را تکمیل کند. پس از موفقیت فرایند آغازین، Crestodian را دوباره شروع کنید.

## راه‌اندازی اولیه setup

`setup` پس از آنکه فرایند هدایت‌شده آغازین استنتاج را برقرار کرد، وضعیت باقی‌مانده فضای کاری و Gateway را پیکربندی می‌کند. این فرمان فقط از طریق عملیات نوع‌دار پیکربندی می‌نویسد و ابتدا تأیید می‌گیرد.

```text
setup
setup workspace ~/Projects/work
```

`setup` مدل مؤثر راستی‌آزمایی‌شده را حفظ می‌کند. این فرمان استنتاج را پیکربندی یا جایگزین نمی‌کند.

اگر استنتاج وجود ندارد یا بررسی زنده آن ناموفق است، Crestodian را ترک کنید و `openclaw onboard` را اجرا کنید. فرایند هدایت‌شده آغازین، مدل‌های پیکربندی‌شده، کلیدهای API و CLIهای محلی احرازشده را شناسایی می‌کند، از هر گزینه یک پاسخ واقعی درخواست می‌کند و فقط مسیر موفق را پایدار می‌سازد. Crestodian بلافاصله پس از این مرز شروع می‌شود و سپس می‌تواند فضای کاری، Gateway، کانال‌ها، عامل‌ها، Pluginها و سایر قابلیت‌های اختیاری را پیکربندی کند.

برنامه macOS هنگامی که به یک Gateway پیکربندی‌شده می‌رسد که عامل پیش‌فرض آن از قبل مدل پیکربندی‌شده‌ای دارد، این نردبان را کاملاً نادیده می‌گیرد و رابط کاربری عادی عامل را باز می‌کند.
برای یک Gateway تازه یا ناقص، برنامه نردبان استنتاج را از طریق متدهای Gateway یعنی `crestodian.setup.detect` و `crestodian.setup.activate` هدایت می‌کند: detect هر بخش پشتی گزینه‌ای را که پیدا می‌کند فهرست می‌کند؛ activate یک گزینه را به‌صورت زنده آزمایش می‌کند (یک تکمیل واقعی «با OK پاسخ بده») و تنها پس از موفقیت آزمایش، مدل، اعتبارنامه و وضعیت ارائه‌دهنده/زمان‌اجرای لازم برای آن مسیر را پایدار می‌سازد. پیش‌فرض‌های فضای کاری و Gateway برای Crestodian باقی می‌مانند. گزینه ناموفق هرگز پیکربندی را تغییر نمی‌دهد؛ برنامه به‌طور خودکار در نردبان پایین می‌رود و در نهایت مرحله دستی کلید/توکن را ارائه می‌کند که از Pluginهای فعال ارائه‌دهنده استنتاج متنی Gateway پر شده است. ارائه‌دهنده انتخاب‌شده مالک مدل آغازین و پیکربندی خود است و اعتبارنامه نیز پیش از ذخیره به همان روش راستی‌آزمایی می‌شود.

نظارت Codex و سایر قابلیت‌های اختیاری Plugin خارج از این تراکنش فعال‌سازی استنتاج باقی می‌مانند. آن‌ها را فقط پس از فعال‌شدن استنتاج و شروع Crestodian پیکربندی کنید؛ سیاست موجود Plugin و انصراف‌های صریح از نظارت در طول راه‌اندازی استنتاج دست‌نخورده باقی می‌مانند.

## مکالمه هوش مصنوعی

مکالمه آزاد Crestodian تعاملی از همان حلقه عامل‌های عادی OpenClaw عبور می‌کند و به یک ابزار اختیاردار سطح صفر OpenClaw با نام `crestodian` محدود است که عملیات نوع‌دار را دربر می‌گیرد. عملیات خواندن آزادانه اجرا می‌شوند؛ جهش‌ها برای همان عملیات دقیق به تأیید مکالمه‌ای شما نیاز دارند (به بخش عملیات و تأیید مراجعه کنید)؛ و هر نوشتن اعمال‌شده ممیزی و دوباره اعتبارسنجی می‌شود. نشست عامل پایدار می‌ماند، بنابراین Crestodian حافظه چندنوبتی واقعی دارد. اگر مسیر استنتاج راستی‌آزمایی‌شده بعداً از کار افتاد، به `openclaw onboard` بازگردید و پیش از ادامه آن را تعمیر کنید.

میزبان درخواست‌های زبان طبیعی را به عملیات تجزیه نمی‌کند. پیام‌های آزاد — از جمله متن‌های شبیه فرمان و پرسش‌هایی مانند «چرا Gateway من متوقف شد؟» — به هوش مصنوعی می‌روند که می‌تواند درخواست را از طریق ابزار `crestodian` به یک عملیات نوع‌دار نگاشت کند.

هنگامی که جهشی معلق است، فقط عبارت‌های بدون ابهام تأیید یا رد از یک فهرست بسته، بدون استنتاج تفسیر می‌شوند. رضایت مبهم به یک فراخوانی تکمیل پیکربندی‌شده جداگانه می‌رود و در غیر این صورت به‌صورت بسته شکست می‌خورد. فیلدهای ساختاریافته جادوگر و پیمایش دقیق میزبان، کنترل‌های رابط کاربری هستند، نه تجزیه عملیات از زبان طبیعی. یک استثنای بهداشت اسرار اهمیت ویژه‌ای دارد: یک `config set` دقیق روی مسیر حساس (توکن‌ها، کلیدها، گذرواژه‌ها) هرگز به مدل نمی‌رسد. میزبان پیشنهادی ویرایش‌شده ایجاد می‌کند و مقدار در تاریخچه قابل مشاهده برای هوش مصنوعی پوشانده می‌شود. برای اسرار، `config set-ref <path> env <ENV_VAR>` را ترجیح دهید.

حالت نجات کانال پیام هرگز از برنامه‌ریز کمک‌گرفته از مدل استفاده نمی‌کند. نجات از راه دور قطعی باقی می‌ماند تا نتوان از مسیر عامل عادی خراب یا به‌خطر‌افتاده به‌عنوان ویرایشگر پیکربندی استفاده کرد.

### مدل اعتماد پوسته CLI

زمان‌های اجرای تعبیه‌شده و پوسته برنامه‌سرور Codex محدودیت سطح صفر را مستقیماً اعمال می‌کنند: اجرا دارای فهرست مجاز ابزار OpenClaw است که فقط ابزار `crestodian` را شامل می‌شود. برای Codex، OpenClaw همچنین محیط‌ها، اجرای بومی، چندعاملی، هدف، برنامه/Plugin، skill/MCP، جست‌وجوی وب و سطوح `request_user_input` را برای آن اجرا غیرفعال می‌کند. Codex همچنان ابزار بومی غیرفعال `update_plan` را تزریق می‌کند؛ این ابزار می‌تواند فهرست بررسی موقت مدل را به‌روزرسانی کند، اما نمی‌تواند فایل‌ها یا پیکربندی OpenClaw را بنویسد. پوسته‌های CLI فهرست مجاز OpenClaw را مصرف نمی‌کنند، بنابراین Crestodian فقط بخش‌های پشتی‌ای را می‌پذیرد که قرارداد انتخاب ابزار خودشان بتواند همان محدودیت را اثبات کند:

- بک‌اندهای قابل‌انتخاب، از جمله Claude Code، با انتخاب خالی ابزارهای بومی
  و یک ابزار MCP به نام `crestodian` اجرا می‌شوند. پیکربندی MCP تولیدشده توسط Claude
  با `--strict-mcp-config` اعمال می‌شود؛ بنابراین هیچ سرور MCP دیگری بارگذاری نمی‌شود.
- بک‌اندهایی که هیچ ابزار بومی‌ای اعلام نمی‌کنند، همان سرور اختصاصی MCP مربوط به Crestodian
  را دریافت می‌کنند.
- بک‌اندهای دارای ابزارهای بومی همواره‌فعال یا ناشناخته، پیش از استنتاج به‌صورت بسته شکست می‌خورند؛
  آن‌ها نمی‌توانند میزبان نشست Crestodian باشند.

فقط نشست‌های Crestodian سرور MCP مربوط به crestodian را دریافت می‌کنند؛ اجراهای عادی عامل
هرگز این ابزار را نمی‌بینند. بنابراین، بک‌اندهای CLI قابل‌انتخابِ فاقد ابزار بومی و مدل‌های
دارای کلید API، حلقهٔ تک‌ابزاریِ صریح را اعمال می‌کنند. مدل‌های app-server در Codex
یک ابزار مرجع OpenClaw را به‌همراه ابزار بومی و غیرفعال برنامه‌ریزی اعمال می‌کنند. در هر
سه حالت، نوشتن‌های راه‌اندازی همچنان به قرارداد تأیید ممیزی‌شدهٔ Crestodian محدود می‌مانند.

Gemini CLI همچنان برای عامل‌های عادی در دسترس است، اما نمی‌تواند وارسیِ بدون ابزارِ
موردنیاز دروازهٔ استنتاج را اعمال کند؛ بنابراین نمی‌تواند میزبان Crestodian باشد.

## جابه‌جایی به یک عامل

برای خروج از Crestodian و باز کردن TUI عادی، از یک انتخاب‌گر زبان طبیعی استفاده کنید:

```text
talk to agent
talk to work agent
switch to main agent
```

`openclaw tui`،‏ `openclaw chat` و `openclaw terminal` مستقیماً TUI عادی عامل را باز می‌کنند؛ آن‌ها Crestodian را آغاز نمی‌کنند. پس از جابه‌جایی به TUI عادی، `/crestodian` شما را به Crestodian بازمی‌گرداند و می‌تواند به‌صورت اختیاری شامل یک درخواست تکمیلی باشد:

```text
/crestodian
/crestodian restart gateway
```

## حالت نجات پیام

حالت نجات پیام، نقطهٔ ورود کانال پیام برای Crestodian است: زمانی از آن استفاده کنید که عامل عادی شما از کار افتاده، اما یک کانال مورداعتماد (برای مثال WhatsApp) همچنان فرمان‌ها را دریافت می‌کند.

این یک کنترل‌کنندهٔ قطعی فرمان‌های اضطراری است، نه عامل مکالمه‌ای
Crestodian. این حالت راه‌اندازی تازه‌ای را بوت‌استرپ نمی‌کند و دروازهٔ استنتاج
را برای گفت‌وگوی Crestodian آسان‌گیرانه‌تر نمی‌کند.

فرمان پشتیبانی‌شده: `/crestodian <request>`. نجات فقط دستور زبان دقیق فرمان تایپ‌شده را می‌پذیرد — زبان طبیعی با یک راهنما رد می‌شود، هرگز به‌صورت حدسی به یک عملیات تبدیل نمی‌شود و هیچ مدلی نیز هرگز مورد مشورت قرار نمی‌گیرد.

```text
You, in a trusted owner DM: /crestodian status
OpenClaw: Crestodian rescue mode. Gateway reachable: no. Config valid: no.
You: /crestodian restart gateway
OpenClaw: Plan: restart the Gateway. Reply /crestodian yes to apply.
You: /crestodian yes
OpenClaw: Applied. Audit entry written.
```

ایجاد عامل را نیز می‌توان به‌صورت محلی یا از طریق نجات در صف قرار داد:

```text
create agent work workspace ~/Projects/work model openai/gpt-5.6-sol
/crestodian create agent work workspace ~/Projects/work
```

در ایجاد عامل فقط می‌توان مدل پیش‌فرض فعلیِ تأییدشده به‌صورت زنده را نام برد. برای به‌ارث‌بردن
آن مسیر، مدل را حذف کنید.

نجات راه‌دور یک سطح مدیریتی است و باید مانند تعمیر راه‌دور پیکربندی با آن رفتار شود، نه گفت‌وگوی عادی.

قرارداد امنیتی نجات راه‌دور:

- هنگامی که سندباکس‌سازی برای عامل/نشست فعال است، غیرفعال می‌شود؛ Crestodian نجات راه‌دور را رد می‌کند و کاربر را به تعمیر محلی از طریق CLI هدایت می‌کند.
- حالت مؤثر پیش‌فرض `auto` است: نجات راه‌دور را فقط در عملیات مورداعتماد YOLO مجاز می‌کند؛ جایی که زمان‌اجرای برنامه از قبل اختیار محلیِ بدون سندباکس دارد (`tools.exec.security` به `full` و `tools.exec.ask` به `off` تفکیک می‌شود و حالت سندباکس `off` است).
- به هویت صریح مالک نیاز دارد؛ قوانین فرستندهٔ عام، خط‌مشی گروه باز، Webhookهای احرازنشده یا کانال‌های ناشناس مجاز نیستند.
- به‌طور پیش‌فرض فقط پیام‌های مستقیم مالک؛ نجات در گروه/کانال نیازمند فعال‌سازی صریح است.
- جست‌وجو و فهرست Plugin فقط خواندنی هستند. نصب Plugin همیشه فقط محلی است (در نجات مسدود می‌شود، حتی اگر در حالت‌های دیگر فعال باشد)، زیرا کد اجرایی را دانلود می‌کند. حذف Plugin هم در Crestodian محلی و هم در نجات رد می‌شود؛ `openclaw plugins uninstall <id>` را از یک ترمینال اجرا کنید.
- نجات راه‌دور نمی‌تواند TUI محلی را باز کند یا به یک نشست تعاملی عامل جابه‌جا شود؛ برای تحویل کار به عامل از `openclaw` محلی استفاده کنید.
- نوشتن‌های ماندگار حتی در حالت نجات نیز همچنان به تأیید نیاز دارند.
- هر عملیات نجات اعمال‌شده ممیزی می‌شود. نجات از طریق کانال پیام، فرادادهٔ کانال، حساب، فرستنده و آدرس مبدأ را ثبت می‌کند؛ عملیات تغییردهندهٔ پیکربندی نیز هش‌های پیکربندی را پیش و پس از تغییر ثبت می‌کنند.
- اسرار هرگز بازتاب داده نمی‌شوند. بررسی SecretRef فقط دسترس‌پذیری را گزارش می‌کند، نه مقادیر را.
- اگر Gateway فعال باشد، نجات عملیات نوع‌دار Gateway را ترجیح می‌دهد؛ اگر از کار افتاده باشد، نجات فقط از حداقل سطح تعمیر محلی استفاده می‌کند که به حلقهٔ عادی عامل وابسته نیست.

ساختار پیکربندی:

```jsonc
{
  "crestodian": {
    "rescue": {
      "enabled": "auto",
      "ownerDmOnly": true,
      "pendingTtlMinutes": 15,
    },
  },
}
```

- `enabled`: مقدار `"auto"` (پیش‌فرض) نجات را فقط زمانی مجاز می‌کند که زمان‌اجرای مؤثر در حالت YOLO و سندباکس‌سازی خاموش باشد؛ `false` هرگز نجات از طریق کانال پیام را مجاز نمی‌کند؛ `true` هنگامی که بررسی‌های مالک/کانال با موفقیت انجام شوند، نجات را صریحاً مجاز می‌کند (همچنان مشمول رد شدن به‌دلیل سندباکس‌سازی است).
- `ownerDmOnly`: نجات را به پیام‌های مستقیم مالک محدود می‌کند. مقدار پیش‌فرض `true` است.
- `pendingTtlMinutes`: مدت‌زمانی که یک نوشتن نجاتِ در انتظار، پیش از انقضا برای تأیید با `/crestodian yes` باز می‌ماند. مقدار پیش‌فرض `15` است.

نجات راه‌دور توسط مسیر Docker پوشش داده می‌شود:

```bash
pnpm test:docker:crestodian-rescue
```

یک آزمون دودِ اختیاری برای سطح فرمان کانال زنده، `/crestodian status` را به‌همراه یک رفت‌وبرگشت تأیید ماندگار از طریق کنترل‌کنندهٔ نجات بررسی می‌کند:

```bash
pnpm test:live:crestodian-rescue-channel
```

راه‌اندازی یک‌مرحله‌ای بسته‌بندی‌شده و محافظت‌شده با استنتاج توسط دستور زیر پوشش داده می‌شود:

```bash
pnpm test:docker:crestodian-first-run
```

این مسیر CLI بسته‌بندی‌شده با یک پوشهٔ وضعیت خالی آغاز می‌شود و ثابت می‌کند که Crestodian
بدون استنتاج به‌صورت بسته شکست می‌خورد. سپس Claude جعلی را از طریق ماژول فعال‌سازی
بسته‌بندی‌شده آزمایش و فعال می‌کند. فقط پس از آن، یک درخواست تقریبی به برنامه‌ریز
می‌رسد و به راه‌اندازی نوع‌دار تفکیک می‌شود؛ سپس فرمان‌های یک‌مرحله‌ای، یک عامل
اضافی ایجاد می‌کنند، Discord را از طریق فعال‌سازی Plugin به‌همراه SecretRef توکن
پیکربندی می‌کنند، پیکربندی را اعتبارسنجی می‌کنند و گزارش ممیزی را بررسی می‌کنند. این مسیر،
شواهد پشتیبان دروازه/عملیات را فراهم می‌کند؛ اما ورود تعاملی یا مکالمهٔ
عامل/ابزار/تأیید Crestodian را اجرا نمی‌کند. سناریوی QA Lab زیر به همان مسیر Docker
هدایت می‌شود:

```bash
pnpm openclaw qa suite --scenario crestodian-ring-zero-setup
```
