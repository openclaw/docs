# `openclaw cron`

کارهای Cron را برای زمان‌بند Gateway مدیریت کنید.

<Tip>
برای دیدن کل سطح فرمان، `openclaw cron --help` را اجرا کنید. برای راهنمای مفهومی، [کارهای Cron](/fa/automation/cron-jobs) را ببینید.
</Tip>

## نشست‌ها

`--session` مقدارهای `main`، `isolated`، `current` یا `session:<id>` را می‌پذیرد.

<AccordionGroup>
  <Accordion title="کلیدهای نشست">
    - `main` به نشست اصلی عامل متصل می‌شود.
    - `isolated` برای هر اجرا یک رونوشت تازه و شناسه نشست تازه می‌سازد.
    - `current` در زمان ایجاد به نشست فعال متصل می‌شود.
    - `session:<id>` به یک کلید نشست پایدار و صریح سنجاق می‌شود.

  </Accordion>
  <Accordion title="معنای نشست ایزوله">
    اجراهای ایزوله زمینه گفت‌وگوی محیطی را بازنشانی می‌کنند. مسیریابی کانال و گروه، سیاست ارسال/صف، ارتقا، مبدأ، و اتصال زمان اجرای ACP برای اجرای جدید بازنشانی می‌شوند. ترجیح‌های ایمن و بازنویسی‌های صریح مدل یا احراز هویت که کاربر انتخاب کرده است می‌توانند بین اجراها منتقل شوند.
  </Accordion>
</AccordionGroup>

## تحویل

`openclaw cron list` و `openclaw cron show <job-id>` مسیر تحویل حل‌شده را پیش‌نمایش می‌کنند. برای `channel: "last"`، پیش‌نمایش نشان می‌دهد مسیر از نشست اصلی یا جاری حل شده است، یا به‌صورت بسته شکست می‌خورد.

هدف‌های دارای پیشوند ارائه‌دهنده می‌توانند کانال‌های اعلان حل‌نشده را رفع ابهام کنند. برای مثال، وقتی `delivery.channel` حذف شده یا `last` باشد، `to: "telegram:123"`، Telegram را انتخاب می‌کند. فقط پیشوندهایی که Plugin بارگذاری‌شده اعلام کرده است انتخابگر ارائه‌دهنده هستند. اگر `delivery.channel` صریح باشد، پیشوند باید با همان کانال هم‌خوان باشد؛ `channel: "whatsapp"` همراه با `to: "telegram:123"` رد می‌شود. پیشوندهای سرویس مانند `imessage:` و `sms:` همچنان نحو هدف تحت مالکیت کانال باقی می‌مانند.

<Note>
کارهای ایزوله `cron add` به‌صورت پیش‌فرض از تحویل `--announce` استفاده می‌کنند. برای داخلی نگه داشتن خروجی از `--no-deliver` استفاده کنید. `--deliver` همچنان به‌عنوان نام مستعار منسوخ برای `--announce` باقی می‌ماند.
</Note>

### مالکیت تحویل

تحویل گفت‌وگوی Cron ایزوله بین عامل و اجراکننده مشترک است:

- عامل وقتی مسیر گفت‌وگو در دسترس باشد، می‌تواند با استفاده از ابزار `message` مستقیماً ارسال کند.
- `announce` فقط وقتی عامل مستقیماً به هدف حل‌شده ارسال نکرده باشد، پاسخ نهایی را با تحویل جایگزین ارسال می‌کند.
- `webhook` بار نهایی‌شده را به یک URL ارسال می‌کند.
- `none` تحویل جایگزین اجراکننده را غیرفعال می‌کند.

`--announce` تحویل جایگزین اجراکننده برای پاسخ نهایی است. `--no-deliver` آن جایگزین را غیرفعال می‌کند، اما وقتی مسیر گفت‌وگو در دسترس باشد ابزار `message` عامل را حذف نمی‌کند.

یادآورها که از یک گفت‌وگوی فعال ساخته می‌شوند، هدف تحویل گفت‌وگوی زنده را برای تحویل اعلان جایگزین حفظ می‌کنند. کلیدهای نشست داخلی ممکن است حروف کوچک باشند؛ از آن‌ها به‌عنوان منبع حقیقت برای شناسه‌های ارائه‌دهنده حساس به بزرگی و کوچکی حروف، مانند شناسه‌های اتاق Matrix، استفاده نکنید.

### تحویل شکست

اعلان‌های شکست به این ترتیب حل می‌شوند:

1. `delivery.failureDestination` روی کار.
2. `cron.failureDestination` سراسری.
3. هدف اعلان اصلی کار، وقتی مقصد شکست صریحی تنظیم نشده باشد.

<Note>
کارهای نشست اصلی فقط زمانی می‌توانند از `delivery.failureDestination` استفاده کنند که حالت تحویل اصلی `webhook` باشد. کارهای ایزوله آن را در همه حالت‌ها می‌پذیرند.
</Note>

نکته: اجراهای Cron ایزوله، شکست‌های عامل در سطح اجرا را حتی وقتی
هیچ بار پاسخ تولید نشود، به‌عنوان خطای کار در نظر می‌گیرند؛ بنابراین شکست‌های مدل/ارائه‌دهنده همچنان شمارنده‌های خطا را افزایش می‌دهند
و اعلان‌های شکست را فعال می‌کنند.

اگر یک اجرای ایزوله پیش از نخستین درخواست مدل به پایان زمان مجاز برسد، `openclaw cron show`
و `openclaw cron runs` یک خطای ویژه مرحله مانند
`setup timed out before runner start` یا
`stalled before first model call (last phase: context-engine)` را شامل می‌شوند.
برای ارائه‌دهنده‌های متکی به CLI، نگهبان پیش از مدل تا زمان شروع نوبت CLI خارجی فعال می‌ماند؛ بنابراین توقف‌های جست‌وجوی نشست، hook، احراز هویت، اعلان، و راه‌اندازی CLI
به‌عنوان شکست‌های Cron پیش از مدل گزارش می‌شوند.

## زمان‌بندی

### کارهای یک‌باره

`--at <datetime>` یک اجرای یک‌باره را زمان‌بندی می‌کند. تاریخ‌زمان‌های بدون offset به‌عنوان UTC در نظر گرفته می‌شوند، مگر اینکه `--tz <iana>` را نیز پاس بدهید؛ در این صورت زمان ساعت دیواری در منطقه زمانی داده‌شده تفسیر می‌شود.

<Note>
کارهای یک‌باره به‌صورت پیش‌فرض پس از موفقیت حذف می‌شوند. برای حفظ آن‌ها از `--keep-after-run` استفاده کنید.
</Note>

### کارهای تکرارشونده

کارهای تکرارشونده پس از خطاهای پیاپی از عقب‌نشینی نمایی برای تلاش دوباره استفاده می‌کنند: 30s، 1m، 5m، 15m، 60m. زمان‌بندی پس از اجرای موفق بعدی به حالت عادی بازمی‌گردد.

اجراهای ردشده جدا از خطاهای اجرا پیگیری می‌شوند. آن‌ها بر عقب‌نشینی تلاش دوباره اثر نمی‌گذارند، اما `openclaw cron edit <job-id> --failure-alert-include-skipped` می‌تواند هشدارهای شکست را شامل اعلان‌های تکرارشونده اجراهای ردشده کند.

برای کارهای ایزوله‌ای که یک ارائه‌دهنده مدل محلی پیکربندی‌شده را هدف می‌گیرند، Cron پیش از شروع نوبت عامل یک پیش‌پرواز سبک ارائه‌دهنده اجرا می‌کند. ارائه‌دهنده‌های local loopback، شبکه خصوصی، و `.local` با `api: "ollama"` در `/api/tags` بررسی می‌شوند؛ ارائه‌دهنده‌های محلی سازگار با OpenAI مانند vLLM، SGLang، و LM Studio در `/models` بررسی می‌شوند. اگر endpoint در دسترس نباشد، اجرا به‌عنوان `skipped` ثبت می‌شود و در زمان‌بندی بعدی دوباره تلاش می‌شود؛ endpointهای ازکارافتاده هم‌خوان به‌مدت ۵ دقیقه cache می‌شوند تا از فشار آوردن تعداد زیادی کار به همان سرور محلی جلوگیری شود.

نکته: تعریف‌های کارهای Cron در `jobs.json` قرار دارند، در حالی که وضعیت زمان اجرای در انتظار در `jobs-state.json` قرار دارد. اگر `jobs.json` بیرونی ویرایش شود، Gateway زمان‌بندی‌های تغییرکرده را دوباره بارگذاری می‌کند و جایگاه‌های در انتظار کهنه را پاک می‌کند؛ بازنویسی‌های صرفاً قالب‌بندی جایگاه در انتظار را پاک نمی‌کنند.

### اجراهای دستی

`openclaw cron run` به‌محض اینکه اجرای دستی در صف قرار بگیرد بازمی‌گردد. پاسخ‌های موفق شامل `{ ok: true, enqueued: true, runId }` هستند. برای دنبال کردن نتیجه نهایی از `openclaw cron runs --id <job-id>` استفاده کنید.

<Note>
`openclaw cron run <job-id>` به‌صورت پیش‌فرض اجرای اجباری انجام می‌دهد. برای حفظ رفتار قدیمی‌تر «فقط اگر موعدش رسیده اجرا شود» از `--due` استفاده کنید.
</Note>

## مدل‌ها

`cron add|edit --model <ref>` یک مدل مجاز را برای کار انتخاب می‌کند.

<Warning>
اگر مدل مجاز نباشد یا قابل حل نباشد، Cron اجرا را با یک خطای اعتبارسنجی صریح ناموفق می‌کند، به‌جای اینکه به عامل کار یا انتخاب مدل پیش‌فرض برگردد.
</Warning>

`--model` در Cron یک **اصلی کار** است، نه بازنویسی `/model` نشست گفت‌وگو. یعنی:

- fallbackهای مدل پیکربندی‌شده همچنان وقتی مدل انتخاب‌شده کار شکست بخورد اعمال می‌شوند.
- وقتی `fallbacks` در بار هر کار وجود داشته باشد، فهرست fallback پیکربندی‌شده را جایگزین می‌کند.
- یک فهرست fallback خالی برای هر کار (`fallbacks: []` در بار کار/API) اجرای Cron را سخت‌گیرانه می‌کند.
- وقتی کاری `--model` داشته باشد اما فهرست fallback پیکربندی نشده باشد، OpenClaw یک بازنویسی fallback خالی صریح پاس می‌دهد تا مدل اصلی عامل به‌عنوان هدف تلاش دوباره پنهان افزوده نشود.

### تقدم مدل Cron ایزوله

Cron ایزوله مدل فعال را به این ترتیب حل می‌کند:

1. بازنویسی Gmail-hook.
2. `--model` هر کار.
3. بازنویسی مدل ذخیره‌شده نشست Cron، وقتی کاربر یکی را انتخاب کرده باشد.
4. انتخاب مدل عامل یا پیش‌فرض.

### حالت سریع

حالت سریع Cron ایزوله از انتخاب مدل زنده حل‌شده پیروی می‌کند. پیکربندی مدل `params.fastMode` به‌صورت پیش‌فرض اعمال می‌شود، اما بازنویسی `fastMode` نشست ذخیره‌شده همچنان بر پیکربندی مقدم است.

### تلاش‌های دوباره برای تغییر مدل زنده

اگر یک اجرای ایزوله `LiveSessionModelSwitchError` بیندازد، Cron پیش از تلاش دوباره، ارائه‌دهنده و مدل تغییرکرده را، و در صورت وجود بازنویسی پروفایل احراز هویت تغییرکرده، برای اجرای فعال پایدار می‌کند. حلقه تلاش دوباره بیرونی پس از تلاش اولیه به دو تلاش دوباره تغییر محدود است، سپس به‌جای تکرار بی‌پایان متوقف می‌شود.

## خروجی اجرا و ردها

### سرکوب تأیید کهنه

نوبت‌های Cron ایزوله پاسخ‌های فقط-تأیید کهنه را سرکوب می‌کنند. اگر نتیجه نخست فقط یک به‌روزرسانی وضعیت موقت باشد و هیچ اجرای زیرعامل فرزندی مسئول پاسخ نهایی نباشد، Cron پیش از تحویل یک‌بار دیگر برای نتیجه واقعی اعلان می‌دهد.

### سرکوب توکن خاموش

اگر یک اجرای Cron ایزوله فقط توکن خاموش (`NO_REPLY` یا `no_reply`) را برگرداند، Cron هم تحویل خروجی مستقیم و هم مسیر خلاصه صف‌شده جایگزین را سرکوب می‌کند، بنابراین چیزی به گفت‌وگو ارسال نمی‌شود.

### ردهای ساختاریافته

اجراهای Cron ایزوله ابتدا فراداده رد اجرای ساختاریافته را از اجرای تعبیه‌شده ترجیح می‌دهند، سپس به نشانگرهای شناخته‌شده رد در خروجی نهایی برمی‌گردند، مانند `SYSTEM_RUN_DENIED`، `INVALID_REQUEST`، و عبارت‌های امتناع اتصال تأیید.

`cron list` و تاریخچه اجرا دلیل رد را نشان می‌دهند، به‌جای اینکه یک فرمان مسدودشده را `ok` گزارش کنند.

## نگه‌داری

نگه‌داری و هرس در پیکربندی کنترل می‌شوند:

- `cron.sessionRetention` (پیش‌فرض `24h`) نشست‌های اجرای ایزوله تکمیل‌شده را هرس می‌کند.
- `cron.runLog.maxBytes` و `cron.runLog.keepLines` فایل `~/.openclaw/cron/runs/<jobId>.jsonl` را هرس می‌کنند.

## مهاجرت کارهای قدیمی‌تر

<Note>
اگر کارهای Cron مربوط به پیش از قالب تحویل و ذخیره‌سازی فعلی دارید، `openclaw doctor --fix` را اجرا کنید. Doctor فیلدهای قدیمی Cron (`jobId`، `schedule.cron`، فیلدهای تحویل سطح بالا شامل `threadId` قدیمی، نام‌های مستعار تحویل `provider` در بار) را عادی‌سازی می‌کند و وقتی `cron.webhook` پیکربندی شده باشد، کارهای fallback وبهوک ساده با `notify: true` را به تحویل وبهوک صریح مهاجرت می‌دهد.
</Note>

## ویرایش‌های رایج

تنظیمات تحویل را بدون تغییر پیام به‌روزرسانی کنید:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

تحویل را برای یک کار ایزوله غیرفعال کنید:

```bash
openclaw cron edit <job-id> --no-deliver
```

زمینه bootstrap سبک را برای یک کار ایزوله فعال کنید:

```bash
openclaw cron edit <job-id> --light-context
```

به یک کانال مشخص اعلان کنید:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

به یک موضوع انجمن Telegram اعلان کنید:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

یک کار ایزوله با زمینه bootstrap سبک بسازید:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` فقط برای کارهای نوبت عامل ایزوله اعمال می‌شود. برای اجراهای Cron، حالت سبک به‌جای تزریق مجموعه کامل bootstrap فضای کاری، زمینه bootstrap را خالی نگه می‌دارد.

## فرمان‌های رایج مدیریتی

اجرای دستی و بازرسی:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

`openclaw cron list` به‌صورت پیش‌فرض همه کارهای هم‌خوان را نشان می‌دهد. برای نمایش فقط کارهایی که شناسه عامل عادی‌شده مؤثرشان هم‌خوان است، `--agent <id>` را پاس بدهید؛ کارهای بدون شناسه عامل ذخیره‌شده به‌عنوان عامل پیش‌فرض پیکربندی‌شده شمرده می‌شوند.

`cron list --json` و `cron show <job-id> --json` روی هر کار یک فیلد سطح بالای `status` شامل می‌شوند که از `enabled`، `state.runningAtMs`، و `state.lastRunStatus` محاسبه می‌شود. مقدارها: `disabled`، `running`، `ok`، `error`، `skipped`، یا `idle`. این ستون وضعیت خوانای انسانی را بازتاب می‌دهد تا ابزارهای خارجی بتوانند وضعیت کار را بدون محاسبه دوباره آن بخوانند.

مدخل‌های `cron runs` شامل تشخیص‌های تحویل با هدف Cron موردنظر، هدف حل‌شده، ارسال‌های ابزار message، استفاده از fallback، و وضعیت تحویل‌شده هستند.

هدف‌گذاری دوباره عامل و نشست:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` وقتی `--agent` در کارهای نوبت عامل حذف شود هشدار می‌دهد و به عامل پیش‌فرض (`main`) برمی‌گردد. برای سنجاق کردن یک عامل مشخص در زمان ایجاد، `--agent <id>` را پاس بدهید.

تنظیمات جزئی تحویل:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
