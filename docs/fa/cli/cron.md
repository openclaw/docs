# `openclaw cron`

کارهای Cron را برای زمان‌بند Gateway مدیریت کنید.

<Tip>
برای مشاهده سطح کامل فرمان، `openclaw cron --help` را اجرا کنید. برای راهنمای مفهومی، [کارهای Cron](/fa/automation/cron-jobs) را ببینید.
</Tip>

## نشست‌ها

`--session` مقادیر `main`، `isolated`، `current` یا `session:<id>` را می‌پذیرد.

<AccordionGroup>
  <Accordion title="کلیدهای نشست">
    - `main` به نشست اصلی عامل متصل می‌شود.
    - `isolated` برای هر اجرا یک رونوشت و شناسه نشست تازه ایجاد می‌کند.
    - `current` در زمان ایجاد، به نشست فعال متصل می‌شود.
    - `session:<id>` به یک کلید نشست پایدار صریح سنجاق می‌شود.

  </Accordion>
  <Accordion title="معنای نشست ایزوله">
    اجراهای ایزوله زمینه گفت‌وگوی محیطی را بازنشانی می‌کنند. مسیریابی کانال و گروه، سیاست ارسال/صف، ارتقا، مبدا، و اتصال زمان اجرای ACP برای اجرای جدید بازنشانی می‌شوند. ترجیحات امن و مدل یا بازنویسی‌های احراز هویت صریحا انتخاب‌شده توسط کاربر می‌توانند بین اجراها منتقل شوند.
  </Accordion>
</AccordionGroup>

## تحویل

`openclaw cron list` و `openclaw cron show <job-id>` مسیر تحویل حل‌شده را پیش‌نمایش می‌کنند. برای `channel: "last"`، پیش‌نمایش نشان می‌دهد که مسیر از نشست اصلی یا فعلی حل شده است، یا به‌صورت بسته شکست خواهد خورد.

<Note>
کارهای ایزوله `cron add` به‌طور پیش‌فرض از تحویل `--announce` استفاده می‌کنند. برای داخلی نگه داشتن خروجی از `--no-deliver` استفاده کنید. `--deliver` همچنان به‌عنوان نام مستعار منسوخ‌شده برای `--announce` باقی می‌ماند.
</Note>

### مالکیت تحویل

تحویل گفت‌وگوی Cron ایزوله بین عامل و اجراکننده مشترک است:

- عامل می‌تواند وقتی مسیر گفت‌وگو در دسترس است، مستقیما با ابزار `message` ارسال کند.
- `announce` فقط وقتی عامل مستقیما به مقصد حل‌شده ارسال نکرده باشد، پاسخ نهایی را به‌صورت جایگزین تحویل می‌دهد.
- `webhook` بار نهایی‌شده را به یک URL ارسال می‌کند.
- `none` تحویل جایگزین اجراکننده را غیرفعال می‌کند.

`--announce` تحویل جایگزین اجراکننده برای پاسخ نهایی است. `--no-deliver` آن جایگزین را غیرفعال می‌کند، اما وقتی مسیر گفت‌وگو در دسترس است ابزار `message` عامل را حذف نمی‌کند.

یادآورها که از یک گفت‌وگوی فعال ایجاد شده‌اند، مقصد تحویل گفت‌وگوی زنده را برای تحویل اعلام جایگزین حفظ می‌کنند. کلیدهای نشست داخلی ممکن است با حروف کوچک باشند؛ از آن‌ها به‌عنوان منبع حقیقت برای شناسه‌های ارائه‌دهنده حساس به بزرگی و کوچکی حروف، مانند شناسه‌های اتاق Matrix، استفاده نکنید.

### تحویل شکست

اعلان‌های شکست به این ترتیب حل می‌شوند:

1. `delivery.failureDestination` روی کار.
2. `cron.failureDestination` سراسری.
3. مقصد اعلام اصلی کار، وقتی مقصد شکست صریحی تنظیم نشده باشد.

<Note>
کارهای نشست اصلی فقط وقتی حالت تحویل اصلی `webhook` باشد می‌توانند از `delivery.failureDestination` استفاده کنند. کارهای ایزوله آن را در همه حالت‌ها می‌پذیرند.
</Note>

نکته: اجراهای Cron ایزوله شکست‌های عامل در سطح اجرا را حتی وقتی هیچ بار پاسخ تولید نشود، به‌عنوان خطای کار در نظر می‌گیرند؛ بنابراین شکست‌های مدل/ارائه‌دهنده همچنان شمارنده‌های خطا را افزایش می‌دهند و اعلان‌های شکست را فعال می‌کنند.

## زمان‌بندی

### کارهای تک‌اجرا

`--at <datetime>` یک اجرای تک‌مرحله‌ای را زمان‌بندی می‌کند. تاریخ‌زمان‌های بدون offset به‌عنوان UTC در نظر گرفته می‌شوند، مگر اینکه `--tz <iana>` را نیز پاس بدهید؛ در این حالت زمان ساعت دیواری در منطقه زمانی داده‌شده تفسیر می‌شود.

<Note>
کارهای تک‌اجرا به‌طور پیش‌فرض پس از موفقیت حذف می‌شوند. برای حفظ آن‌ها از `--keep-after-run` استفاده کنید.
</Note>

### کارهای تکرارشونده

کارهای تکرارشونده پس از خطاهای متوالی از عقب‌گرد نمایی برای تلاش دوباره استفاده می‌کنند: 30s، 1m، 5m، 15m، 60m. زمان‌بندی پس از اجرای موفق بعدی به حالت عادی برمی‌گردد.

اجراهای ردشده جدا از خطاهای اجرا ردیابی می‌شوند. آن‌ها بر عقب‌گرد تلاش دوباره اثر نمی‌گذارند، اما `openclaw cron edit <job-id> --failure-alert-include-skipped` می‌تواند هشدارهای شکست را برای اعلان‌های تکراری اجرای ردشده فعال کند.

برای کارهای ایزوله‌ای که یک ارائه‌دهنده مدل پیکربندی‌شده محلی را هدف می‌گیرند، Cron پیش از شروع نوبت عامل، یک پیش‌پرواز سبک برای ارائه‌دهنده اجرا می‌کند. ارائه‌دهنده‌های Loopback، شبکه خصوصی و `.local` با `api: "ollama"` در `/api/tags` بررسی می‌شوند؛ ارائه‌دهنده‌های محلی سازگار با OpenAI مانند vLLM، SGLang و LM Studio در `/models` بررسی می‌شوند. اگر endpoint در دسترس نباشد، اجرا به‌عنوان `skipped` ثبت می‌شود و در زمان‌بندی بعدی دوباره تلاش می‌شود؛ endpointهای مرده همسان به‌مدت 5 دقیقه cache می‌شوند تا از کوبیدن کارهای زیاد به همان سرور محلی جلوگیری شود.

نکته: تعریف‌های کار Cron در `jobs.json` قرار دارند، در حالی که وضعیت زمان اجرای در انتظار در `jobs-state.json` قرار دارد. اگر `jobs.json` بیرونی ویرایش شود، Gateway زمان‌بندی‌های تغییریافته را دوباره بارگذاری می‌کند و شکاف‌های در انتظار کهنه را پاک می‌کند؛ بازنویسی‌هایی که فقط قالب‌بندی را تغییر می‌دهند شکاف در انتظار را پاک نمی‌کنند.

### اجراهای دستی

`openclaw cron run` به‌محض اینکه اجرای دستی در صف قرار بگیرد برمی‌گردد. پاسخ‌های موفق شامل `{ ok: true, enqueued: true, runId }` هستند. برای دنبال کردن نتیجه نهایی از `openclaw cron runs --id <job-id>` استفاده کنید.

<Note>
`openclaw cron run <job-id>` به‌طور پیش‌فرض اجرای اجباری انجام می‌دهد. برای حفظ رفتار قدیمی‌تر «فقط اگر موعدش رسیده اجرا شود» از `--due` استفاده کنید.
</Note>

## مدل‌ها

`cron add|edit --model <ref>` یک مدل مجاز را برای کار انتخاب می‌کند.

<Warning>
اگر مدل مجاز نباشد یا قابل حل نباشد، Cron به‌جای بازگشت به عامل کار یا انتخاب مدل پیش‌فرض، اجرا را با یک خطای اعتبارسنجی صریح شکست می‌دهد.
</Warning>

Cron `--model` یک **اصلیِ کار** است، نه بازنویسی `/model` نشست گفت‌وگو. یعنی:

- وقتی مدل انتخاب‌شده کار شکست بخورد، fallbackهای مدل پیکربندی‌شده همچنان اعمال می‌شوند.
- وقتی وجود داشته باشد، `fallbacks` بار هر کار فهرست fallback پیکربندی‌شده را جایگزین می‌کند.
- فهرست خالی fallback برای هر کار (`fallbacks: []` در بار/API کار) اجرای Cron را سخت‌گیرانه می‌کند.
- وقتی کاری `--model` دارد اما هیچ فهرست fallback پیکربندی نشده است، OpenClaw یک بازنویسی fallback خالی صریح پاس می‌دهد تا مدل اصلی عامل به‌عنوان مقصد تلاش دوباره پنهان اضافه نشود.

### اولویت مدل Cron ایزوله

Cron ایزوله مدل فعال را به این ترتیب حل می‌کند:

1. بازنویسی hook جیمیل.
2. `--model` هر کار.
3. بازنویسی مدل نشست Cron ذخیره‌شده، وقتی کاربر یکی را انتخاب کرده باشد.
4. انتخاب مدل عامل یا پیش‌فرض.

### حالت سریع

حالت سریع Cron ایزوله از انتخاب مدل زنده حل‌شده پیروی می‌کند. پیکربندی مدل `params.fastMode` به‌طور پیش‌فرض اعمال می‌شود، اما بازنویسی `fastMode` نشست ذخیره‌شده همچنان بر پیکربندی مقدم است.

### تلاش‌های دوباره تعویض مدل زنده

اگر یک اجرای ایزوله `LiveSessionModelSwitchError` پرتاب کند، Cron پیش از تلاش دوباره، ارائه‌دهنده و مدل تعویض‌شده را، و در صورت وجود بازنویسی نمایه احراز هویت تعویض‌شده را، برای اجرای فعال ماندگار می‌کند. حلقه تلاش دوباره بیرونی به دو تلاش دوباره تعویض پس از تلاش اولیه محدود است، سپس به‌جای حلقه بی‌پایان متوقف می‌شود.

## خروجی اجرا و ردها

### سرکوب تاییدهای کهنه

نوبت‌های Cron ایزوله پاسخ‌های فقط-تایید کهنه را سرکوب می‌کنند. اگر نتیجه اول فقط یک به‌روزرسانی وضعیت موقت باشد و هیچ اجرای زیرعامل فرزندی مسئول پاسخ نهایی نباشد، Cron پیش از تحویل، یک بار دیگر برای نتیجه واقعی درخواست می‌کند.

### سرکوب توکن خاموش

اگر یک اجرای Cron ایزوله فقط توکن خاموش (`NO_REPLY` یا `no_reply`) را برگرداند، Cron هم تحویل مستقیم خروجی و هم مسیر خلاصه صف‌شده جایگزین را سرکوب می‌کند، بنابراین چیزی به گفت‌وگو ارسال نمی‌شود.

### ردهای ساختاریافته

اجراهای Cron ایزوله ابتدا فراداده رد اجرای ساختاریافته را از اجرای توکار ترجیح می‌دهند، سپس به نشانگرهای رد شناخته‌شده در خروجی نهایی برمی‌گردند، مانند `SYSTEM_RUN_DENIED`، `INVALID_REQUEST` و عبارت‌های امتناع اتصال تایید.

`cron list` و تاریخچه اجرا دلیل رد را نشان می‌دهند، به‌جای اینکه یک فرمان مسدودشده را `ok` گزارش کنند.

## نگهداشت

نگهداشت و هرس در پیکربندی کنترل می‌شوند:

- `cron.sessionRetention` (پیش‌فرض `24h`) نشست‌های اجرای ایزوله تکمیل‌شده را هرس می‌کند.
- `cron.runLog.maxBytes` و `cron.runLog.keepLines` فایل `~/.openclaw/cron/runs/<jobId>.jsonl` را هرس می‌کنند.

## مهاجرت کارهای قدیمی‌تر

<Note>
اگر کارهای Cron مربوط به پیش از قالب تحویل و ذخیره فعلی دارید، `openclaw doctor --fix` را اجرا کنید. Doctor فیلدهای Cron قدیمی (`jobId`، `schedule.cron`، فیلدهای تحویل سطح بالا شامل `threadId` قدیمی، نام‌های مستعار تحویل `provider` در payload) را نرمال می‌کند و وقتی `cron.webhook` پیکربندی شده باشد، کارهای fallback Webhook ساده با `notify: true` را به تحویل Webhook صریح مهاجرت می‌دهد.
</Note>

## ویرایش‌های رایج

تنظیمات تحویل را بدون تغییر پیام به‌روزرسانی کنید:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

تحویل را برای یک کار ایزوله غیرفعال کنید:

```bash
openclaw cron edit <job-id> --no-deliver
```

زمینه bootstrap سبک را برای یک کار ایزوله فعال کنید:

```bash
openclaw cron edit <job-id> --light-context
```

به یک کانال مشخص اعلام کنید:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

به یک موضوع انجمن Telegram اعلام کنید:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

یک کار ایزوله با زمینه bootstrap سبک ایجاد کنید:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` فقط برای کارهای نوبت-عامل ایزوله اعمال می‌شود. برای اجراهای Cron، حالت سبک به‌جای تزریق مجموعه کامل bootstrap فضای کاری، زمینه bootstrap را خالی نگه می‌دارد.

## فرمان‌های رایج مدیریت

اجرای دستی و بازرسی:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

ورودی‌های `cron runs` شامل عیب‌یابی‌های تحویل با مقصد Cron مورد نظر، مقصد حل‌شده، ارسال‌های ابزار پیام، استفاده از fallback و وضعیت تحویل‌شده هستند.

هدف‌گیری دوباره عامل و نشست:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

تنظیمات تحویل:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
