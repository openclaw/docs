# `openclaw cron`

کارهای cron را برای زمان‌بند Gateway مدیریت کنید.

<Tip>
برای دیدن سطح کامل دستور، `openclaw cron --help` را اجرا کنید. برای راهنمای مفهومی، [کارهای Cron](/fa/automation/cron-jobs) را ببینید.
</Tip>

## نشست‌ها

`--session` مقدارهای `main`، `isolated`، `current`، یا `session:<id>` را می‌پذیرد.

<AccordionGroup>
  <Accordion title="کلیدهای نشست">
    - `main` به نشست اصلی عامل متصل می‌شود.
    - `isolated` برای هر اجرا یک رونوشت تازه و شناسه نشست تازه ایجاد می‌کند.
    - `current` در زمان ایجاد به نشست فعال متصل می‌شود.
    - `session:<id>` به یک کلید نشست پایدار صریح پین می‌شود.

  </Accordion>
  <Accordion title="معناشناسی نشست ایزوله">
    اجراهای ایزوله زمینه گفت‌وگوی محیطی را بازنشانی می‌کنند. مسیریابی کانال و گروه، خط‌مشی ارسال/صف، ارتقا، مبدأ، و اتصال زمان اجرای ACP برای اجرای تازه بازنشانی می‌شوند. ترجیحات امن و بازنویسی‌های صریح مدل یا احراز هویت انتخاب‌شده توسط کاربر می‌توانند بین اجراها منتقل شوند.
  </Accordion>
</AccordionGroup>

## تحویل

`openclaw cron list` و `openclaw cron show <job-id>` مسیر تحویل نهایی‌شده را پیش‌نمایش می‌کنند. برای `channel: "last"`، پیش‌نمایش نشان می‌دهد که آیا مسیر از نشست اصلی یا فعلی حل شده است، یا با حالت بسته شکست می‌خورد.

<Note>
کارهای ایزوله `cron add` به‌طور پیش‌فرض از تحویل `--announce` استفاده می‌کنند. برای داخلی نگه داشتن خروجی از `--no-deliver` استفاده کنید. `--deliver` همچنان به‌عنوان نام مستعار منسوخ برای `--announce` باقی می‌ماند.
</Note>

### مالکیت تحویل

تحویل گفت‌وگوی cron ایزوله بین عامل و اجراکننده مشترک است:

- عامل می‌تواند زمانی که مسیر گفت‌وگو در دسترس است، مستقیماً با ابزار `message` ارسال کند.
- `announce` فقط زمانی پاسخ نهایی را به‌صورت جایگزین تحویل می‌دهد که عامل مستقیماً به مقصد حل‌شده ارسال نکرده باشد.
- `webhook` محتوای نهایی‌شده را به یک URL ارسال می‌کند.
- `none` تحویل جایگزین اجراکننده را غیرفعال می‌کند.

`--announce` تحویل جایگزین اجراکننده برای پاسخ نهایی است. `--no-deliver` آن جایگزین را غیرفعال می‌کند، اما وقتی مسیر گفت‌وگو در دسترس باشد ابزار `message` عامل را حذف نمی‌کند.

یادآورها که از یک گفت‌وگوی فعال ایجاد می‌شوند، مقصد تحویل گفت‌وگوی زنده را برای تحویل اعلان جایگزین حفظ می‌کنند. کلیدهای نشست داخلی ممکن است با حروف کوچک باشند؛ از آن‌ها به‌عنوان منبع حقیقت برای شناسه‌های ارائه‌دهنده حساس به بزرگی/کوچکی حروف، مانند شناسه‌های اتاق Matrix، استفاده نکنید.

### تحویل شکست

اعلان‌های شکست به این ترتیب حل می‌شوند:

1. `delivery.failureDestination` روی کار.
2. `cron.failureDestination` سراسری.
3. مقصد اعلان اصلی کار (وقتی مقصد شکست صریحی تنظیم نشده باشد).

<Note>
کارهای نشست اصلی فقط زمانی می‌توانند از `delivery.failureDestination` استفاده کنند که حالت تحویل اصلی `webhook` باشد. کارهای ایزوله آن را در همه حالت‌ها می‌پذیرند.
</Note>

نکته: اجراهای cron ایزوله، شکست‌های عامل در سطح اجرا را حتی وقتی
هیچ محتوای پاسخی تولید نشده باشد به‌عنوان خطای کار در نظر می‌گیرند، بنابراین شکست‌های مدل/ارائه‌دهنده همچنان شمارنده‌های خطا را افزایش می‌دهند
و اعلان‌های شکست را فعال می‌کنند.

## زمان‌بندی

### کارهای یک‌باره

`--at <datetime>` یک اجرای یک‌باره را زمان‌بندی می‌کند. تاریخ‌وزمان‌های بدون offset به‌عنوان UTC در نظر گرفته می‌شوند، مگر اینکه `--tz <iana>` را نیز ارسال کنید، که زمان ساعت دیواری را در منطقه زمانی داده‌شده تفسیر می‌کند.

<Note>
کارهای یک‌باره به‌طور پیش‌فرض پس از موفقیت حذف می‌شوند. برای حفظ آن‌ها از `--keep-after-run` استفاده کنید.
</Note>

### کارهای تکرارشونده

کارهای تکرارشونده پس از خطاهای متوالی از backoff نمایی برای تلاش مجدد استفاده می‌کنند: 30s، 1m، 5m، 15m، 60m. زمان‌بندی پس از اجرای موفق بعدی به حالت عادی برمی‌گردد.

اجراهای ردشده جدا از خطاهای اجرا ردیابی می‌شوند. آن‌ها بر backoff تلاش مجدد اثر نمی‌گذارند، اما `openclaw cron edit <job-id> --failure-alert-include-skipped` می‌تواند هشدارهای شکست را شامل اعلان‌های تکراری اجرای ردشده کند.

برای کارهای ایزوله‌ای که یک ارائه‌دهنده مدل محلی پیکربندی‌شده را هدف می‌گیرند، cron پیش از شروع نوبت عامل، یک پیش‌پرواز سبک ارائه‌دهنده اجرا می‌کند. ارائه‌دهنده‌های local loopback، شبکه خصوصی، و `.local` با `api: "ollama"` در `/api/tags` بررسی می‌شوند؛ ارائه‌دهنده‌های محلی سازگار با OpenAI مانند vLLM، SGLang، و LM Studio در `/models` بررسی می‌شوند. اگر endpoint در دسترس نباشد، اجرا به‌عنوان `skipped` ثبت می‌شود و در یک زمان‌بندی بعدی دوباره تلاش می‌شود؛ endpointهای مرده منطبق برای 5 دقیقه cache می‌شوند تا از هجوم تعداد زیادی کار به همان سرور محلی جلوگیری شود.

نکته: تعریف‌های کار cron در `jobs.json` قرار دارند، درحالی‌که وضعیت زمان اجرای معلق در `jobs-state.json` قرار دارد. اگر `jobs.json` از بیرون ویرایش شود، Gateway زمان‌بندی‌های تغییرکرده را دوباره بارگذاری می‌کند و slotهای معلق کهنه را پاک می‌کند؛ بازنویسی‌های صرفاً قالب‌بندی slot معلق را پاک نمی‌کنند.

### اجراهای دستی

`openclaw cron run` به‌محض اینکه اجرای دستی در صف قرار گرفت برمی‌گردد. پاسخ‌های موفق شامل `{ ok: true, enqueued: true, runId }` هستند. برای دنبال کردن نتیجه نهایی از `openclaw cron runs --id <job-id>` استفاده کنید.

<Note>
`openclaw cron run <job-id>` به‌طور پیش‌فرض اجرا را اجباری می‌کند. برای حفظ رفتار قدیمی «فقط اگر موعدش رسیده اجرا کن» از `--due` استفاده کنید.
</Note>

## مدل‌ها

`cron add|edit --model <ref>` یک مدل مجاز را برای کار انتخاب می‌کند.

<Warning>
اگر مدل مجاز نباشد یا قابل حل نباشد، cron به‌جای بازگشت به مدل عامل کار یا انتخاب مدل پیش‌فرض، اجرا را با خطای اعتبارسنجی صریح ناموفق می‌کند.
</Warning>

`--model` در Cron یک **اصلیِ کار** است، نه بازنویسی `/model` نشست گفت‌وگو. یعنی:

- fallbackهای مدل پیکربندی‌شده همچنان زمانی اعمال می‌شوند که مدل انتخاب‌شده کار شکست بخورد.
- محتوای هر کار `fallbacks`، وقتی وجود داشته باشد، فهرست fallback پیکربندی‌شده را جایگزین می‌کند.
- یک فهرست fallback خالی برای هر کار (`fallbacks: []` در محتوای کار/API) اجرای cron را سخت‌گیرانه می‌کند.
- وقتی کاری `--model` دارد اما هیچ فهرست fallback پیکربندی نشده است، OpenClaw یک بازنویسی fallback خالی صریح ارسال می‌کند تا مدل اصلی عامل به‌عنوان مقصد تلاش مجدد پنهان افزوده نشود.

### اولویت مدل cron ایزوله

cron ایزوله مدل فعال را به این ترتیب حل می‌کند:

1. بازنویسی Gmail-hook.
2. `--model` هر کار.
3. بازنویسی مدل ذخیره‌شده نشست cron (وقتی کاربر یکی را انتخاب کرده باشد).
4. انتخاب مدل عامل یا پیش‌فرض.

### حالت سریع

حالت سریع cron ایزوله از انتخاب مدل زنده حل‌شده پیروی می‌کند. پیکربندی مدل `params.fastMode` به‌طور پیش‌فرض اعمال می‌شود، اما بازنویسی `fastMode` نشست ذخیره‌شده همچنان بر پیکربندی مقدم است.

### تلاش‌های مجدد تغییر مدل زنده

اگر یک اجرای ایزوله `LiveSessionModelSwitchError` پرتاب کند، cron پیش از تلاش مجدد، ارائه‌دهنده و مدل تغییرکرده (و بازنویسی پروفایل احراز هویت تغییرکرده، وقتی وجود داشته باشد) را برای اجرای فعال پایدار می‌کند. حلقه تلاش مجدد بیرونی پس از تلاش اولیه به دو تلاش مجدد تغییر محدود است، سپس به‌جای حلقه بی‌پایان متوقف می‌شود.

## خروجی اجرا و ردها

### سرکوب تأیید کهنه

نوبت‌های cron ایزوله پاسخ‌های فقط-تأیید کهنه را سرکوب می‌کنند. اگر نتیجه نخست فقط یک به‌روزرسانی وضعیت موقت باشد و هیچ اجرای subagent فرزندی مسئول پاسخ نهایی نباشد، cron یک‌بار دیگر برای نتیجه واقعی پیش از تحویل prompt می‌دهد.

### سرکوب توکن خاموش

اگر یک اجرای cron ایزوله فقط توکن خاموش (`NO_REPLY` یا `no_reply`) را برگرداند، cron هم تحویل مستقیم خروجی و هم مسیر خلاصه صف‌شده جایگزین را سرکوب می‌کند، بنابراین چیزی به گفت‌وگو ارسال نمی‌شود.

### ردهای ساختاریافته

اجراهای cron ایزوله متادیتای ساختاریافته رد اجرا را از اجرای تعبیه‌شده ترجیح می‌دهند، سپس به نشانگرهای رد شناخته‌شده در خروجی نهایی، مانند `SYSTEM_RUN_DENIED`، `INVALID_REQUEST`، و عبارت‌های امتناع مربوط به اتصال تأیید، fallback می‌کنند.

`cron list` و تاریخچه اجرا دلیل رد را به‌جای گزارش یک دستور مسدودشده به‌عنوان `ok` نمایش می‌دهند.

## نگهداری

نگهداری و pruning در پیکربندی کنترل می‌شوند:

- `cron.sessionRetention` (پیش‌فرض `24h`) نشست‌های اجرای ایزوله تکمیل‌شده را prune می‌کند.
- `cron.runLog.maxBytes` و `cron.runLog.keepLines` فایل `~/.openclaw/cron/runs/<jobId>.jsonl` را prune می‌کنند.

## مهاجرت کارهای قدیمی‌تر

<Note>
اگر کارهای cron مربوط به پیش از قالب تحویل و ذخیره فعلی دارید، `openclaw doctor --fix` را اجرا کنید. Doctor فیلدهای cron قدیمی (`jobId`، `schedule.cron`، فیلدهای تحویل سطح بالا شامل `threadId` قدیمی، نام‌های مستعار تحویل `provider` در payload) را نرمال‌سازی می‌کند و کارهای fallback ساده `notify: true` webhook را وقتی `cron.webhook` پیکربندی شده باشد به تحویل webhook صریح مهاجرت می‌دهد.
</Note>

## ویرایش‌های رایج

به‌روزرسانی تنظیمات تحویل بدون تغییر پیام:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

غیرفعال کردن تحویل برای یک کار ایزوله:

```bash
openclaw cron edit <job-id> --no-deliver
```

فعال کردن زمینه bootstrap سبک برای یک کار ایزوله:

```bash
openclaw cron edit <job-id> --light-context
```

اعلان به یک کانال مشخص:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

اعلان به یک موضوع انجمن Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

ایجاد یک کار ایزوله با زمینه bootstrap سبک:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` فقط برای کارهای نوبت-عامل ایزوله اعمال می‌شود. برای اجراهای cron، حالت سبک به‌جای تزریق مجموعه کامل bootstrap workspace، زمینه bootstrap را خالی نگه می‌دارد.

## دستورهای رایج مدیریتی

اجرای دستی و بازرسی:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

ورودی‌های `cron runs` شامل عیب‌یابی تحویل با مقصد موردنظر cron، مقصد حل‌شده، ارسال‌های ابزار پیام، استفاده از fallback، و وضعیت تحویل‌شده هستند.

هدف‌گیری مجدد عامل و نشست:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` وقتی `--agent` در کارهای نوبت-عامل حذف شده باشد هشدار می‌دهد و به عامل پیش‌فرض (`main`) fallback می‌کند. برای پین کردن یک عامل مشخص، هنگام ایجاد `--agent <id>` را ارسال کنید.

تنظیمات جزئی تحویل:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
