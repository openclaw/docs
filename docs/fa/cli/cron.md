# `openclaw cron`

jobهای Cron را برای زمان‌بند Gateway مدیریت کنید.

<Tip>
برای دیدن سطح کامل فرمان، `openclaw cron --help` را اجرا کنید. برای راهنمای مفهومی، [jobهای Cron](/fa/automation/cron-jobs) را ببینید.
</Tip>

## نشست‌ها

`--session` مقدارهای `main`، `isolated`، `current`، یا `session:<id>` را می‌پذیرد.

<AccordionGroup>
  <Accordion title="کلیدهای نشست">
    - `main` به نشست اصلی عامل متصل می‌شود.
    - `isolated` برای هر اجرا یک رونوشت و شناسه نشست تازه ایجاد می‌کند.
    - `current` در زمان ایجاد به نشست فعال متصل می‌شود.
    - `session:<id>` به یک کلید نشست پایدار صریح سنجاق می‌شود.

  </Accordion>
  <Accordion title="معناشناسی نشست ایزوله">
    اجراهای ایزوله زمینه گفت‌وگوی محیطی را بازنشانی می‌کنند. مسیریابی کانال و گروه، خط‌مشی ارسال/صف، ارتقا، مبدأ، و اتصال زمان اجرای ACP برای اجرای جدید بازنشانی می‌شوند. ترجیحات ایمن و بازنویسی‌های صریح مدل یا احراز هویت انتخاب‌شده توسط کاربر می‌توانند بین اجراها منتقل شوند.
  </Accordion>
</AccordionGroup>

## تحویل

`openclaw cron list` و `openclaw cron show <job-id>` مسیر تحویل حل‌شده را پیش‌نمایش می‌کنند. برای `channel: "last"`، پیش‌نمایش نشان می‌دهد که مسیر از نشست اصلی یا جاری حل شده است، یا به‌صورت بسته شکست خواهد خورد.

هدف‌های دارای پیشوند ارائه‌دهنده می‌توانند کانال‌های اعلان حل‌نشده را ابهام‌زدایی کنند. برای مثال، وقتی `delivery.channel` حذف شده یا `last` باشد، `to: "telegram:123"`، Telegram را انتخاب می‌کند. فقط پیشوندهایی که Plugin بارگذاری‌شده اعلام کرده است، انتخاب‌گر ارائه‌دهنده هستند. اگر `delivery.channel` صریح باشد، پیشوند باید با آن کانال مطابقت داشته باشد؛ `channel: "whatsapp"` همراه با `to: "telegram:123"` رد می‌شود. پیشوندهای سرویس مانند `imessage:` و `sms:` همچنان نحو هدف متعلق به کانال باقی می‌مانند.

<Note>
jobهای ایزوله `cron add` به‌طور پیش‌فرض از تحویل `--announce` استفاده می‌کنند. برای داخلی نگه داشتن خروجی از `--no-deliver` استفاده کنید. `--deliver` همچنان به‌عنوان نام مستعار منسوخ‌شده برای `--announce` باقی می‌ماند.
</Note>

### مالکیت تحویل

تحویل گفت‌وگوی Cron ایزوله بین عامل و اجراکننده مشترک است:

- عامل وقتی مسیر گفت‌وگو موجود باشد می‌تواند مستقیماً با ابزار `message` ارسال کند.
- `announce` فقط وقتی پاسخ نهایی را به‌صورت fallback تحویل می‌دهد که عامل مستقیماً به هدف حل‌شده ارسال نکرده باشد.
- `webhook` بار نهایی‌شده را به یک URL ارسال می‌کند.
- `none` تحویل fallback اجراکننده را غیرفعال می‌کند.

`--announce` تحویل fallback اجراکننده برای پاسخ نهایی است. `--no-deliver` آن fallback را غیرفعال می‌کند، اما وقتی مسیر گفت‌وگو موجود باشد ابزار `message` عامل را حذف نمی‌کند.

یادآورها که از یک گفت‌وگوی فعال ایجاد شده‌اند، هدف تحویل گفت‌وگوی زنده را برای تحویل اعلان fallback حفظ می‌کنند. کلیدهای نشست داخلی ممکن است با حروف کوچک باشند؛ از آن‌ها به‌عنوان منبع حقیقت برای شناسه‌های ارائه‌دهنده حساس به بزرگی/کوچکی حروف، مانند شناسه‌های اتاق Matrix، استفاده نکنید.

### تحویل شکست

اعلان‌های شکست به این ترتیب حل می‌شوند:

1. `delivery.failureDestination` روی job.
2. `cron.failureDestination` سراسری.
3. هدف اعلان اصلی job، وقتی مقصد شکست صریحی تنظیم نشده باشد.

<Note>
jobهای نشست اصلی فقط وقتی می‌توانند از `delivery.failureDestination` استفاده کنند که حالت تحویل اصلی `webhook` باشد. jobهای ایزوله آن را در همه حالت‌ها می‌پذیرند.
</Note>

نکته: اجراهای Cron ایزوله شکست‌های عامل در سطح اجرا را حتی وقتی
هیچ بار پاسخی تولید نشود، به‌عنوان خطای job در نظر می‌گیرند؛ بنابراین شکست‌های مدل/ارائه‌دهنده همچنان شمارنده‌های خطا را افزایش می‌دهند
و اعلان‌های شکست را فعال می‌کنند.

## زمان‌بندی

### jobهای یک‌باره

`--at <datetime>` یک اجرای یک‌باره را زمان‌بندی می‌کند. تاریخ‌زمان‌های بدون offset به‌عنوان UTC در نظر گرفته می‌شوند، مگر این‌که `--tz <iana>` را هم پاس دهید که زمان ساعت دیواری را در منطقه زمانی داده‌شده تفسیر می‌کند.

<Note>
jobهای یک‌باره به‌طور پیش‌فرض پس از موفقیت حذف می‌شوند. برای حفظ آن‌ها از `--keep-after-run` استفاده کنید.
</Note>

### jobهای تکرارشونده

jobهای تکرارشونده پس از خطاهای پیاپی از backoff نمایی برای تلاش مجدد استفاده می‌کنند: 30s، 1m، 5m، 15m، 60m. زمان‌بندی پس از اجرای موفق بعدی به حالت عادی برمی‌گردد.

اجراهای ردشده جدا از خطاهای اجرا ردیابی می‌شوند. آن‌ها روی backoff تلاش مجدد اثر نمی‌گذارند، اما `openclaw cron edit <job-id> --failure-alert-include-skipped` می‌تواند هشدارهای شکست را به اعلان‌های تکراری اجرای ردشده وارد کند.

برای jobهای ایزوله‌ای که یک ارائه‌دهنده مدل محلی پیکربندی‌شده را هدف می‌گیرند، Cron پیش از شروع نوبت عامل، یک preflight سبک ارائه‌دهنده اجرا می‌کند. ارائه‌دهنده‌های local loopback، شبکه خصوصی و `.local` با `api: "ollama"` در `/api/tags` بررسی می‌شوند؛ ارائه‌دهنده‌های محلی سازگار با OpenAI مانند vLLM، SGLang و LM Studio در `/models` بررسی می‌شوند. اگر endpoint در دسترس نباشد، اجرا به‌عنوان `skipped` ثبت می‌شود و در زمان‌بندی بعدی دوباره تلاش می‌شود؛ endpointهای مرده مطابق برای ۵ دقیقه cache می‌شوند تا از کوبیدن jobهای زیاد به همان سرور محلی جلوگیری شود.

نکته: تعریف‌های jobهای Cron در `jobs.json` قرار دارند، درحالی‌که وضعیت زمان اجرای در انتظار در `jobs-state.json` قرار دارد. اگر `jobs.json` از بیرون ویرایش شود، Gateway زمان‌بندی‌های تغییرکرده را دوباره بارگذاری می‌کند و slotهای در انتظار کهنه را پاک می‌کند؛ بازنویسی‌های فقط قالب‌بندی slot در انتظار را پاک نمی‌کنند.

### اجراهای دستی

`openclaw cron run` به‌محض صف‌شدن اجرای دستی برمی‌گردد. پاسخ‌های موفق شامل `{ ok: true, enqueued: true, runId }` هستند. برای دنبال‌کردن نتیجه نهایی از `openclaw cron runs --id <job-id>` استفاده کنید.

<Note>
`openclaw cron run <job-id>` به‌طور پیش‌فرض اجرای اجباری انجام می‌دهد. برای نگه داشتن رفتار قدیمی‌تر «فقط اگر موعدش رسیده باشد اجرا کن» از `--due` استفاده کنید.
</Note>

## مدل‌ها

`cron add|edit --model <ref>` یک مدل مجاز را برای job انتخاب می‌کند.

<Warning>
اگر مدل مجاز نباشد یا حل نشود، Cron به‌جای fallback به عامل job یا انتخاب مدل پیش‌فرض، اجرا را با یک خطای اعتبارسنجی صریح شکست می‌دهد.
</Warning>

`--model` در Cron یک **اصلیِ job** است، نه یک override نشست گفت‌وگوی `/model`. یعنی:

- وقتی مدل انتخاب‌شده job شکست بخورد، fallbackهای پیکربندی‌شده مدل همچنان اعمال می‌شوند.
- payload هر job، یعنی `fallbacks`، وقتی حاضر باشد فهرست fallback پیکربندی‌شده را جایگزین می‌کند.
- یک فهرست fallback خالی برای هر job (`fallbacks: []` در payload/API job) اجرای Cron را سخت‌گیر می‌کند.
- وقتی job دارای `--model` باشد اما هیچ فهرست fallback پیکربندی نشده باشد، OpenClaw یک override fallback خالی صریح پاس می‌دهد تا مدل اصلی عامل به‌عنوان هدف تلاش مجدد پنهان اضافه نشود.

### تقدم مدل Cron ایزوله

Cron ایزوله مدل فعال را به این ترتیب حل می‌کند:

1. override قلاب Gmail.
2. `--model` هر job.
3. override مدل نشست Cron ذخیره‌شده، وقتی کاربر یکی را انتخاب کرده باشد.
4. انتخاب مدل عامل یا پیش‌فرض.

### حالت سریع

حالت سریع Cron ایزوله از انتخاب مدل زنده حل‌شده پیروی می‌کند. پیکربندی مدل `params.fastMode` به‌طور پیش‌فرض اعمال می‌شود، اما override ذخیره‌شده نشست یعنی `fastMode` همچنان بر پیکربندی مقدم است.

### تلاش‌های مجدد تغییر مدل زنده

اگر یک اجرای ایزوله `LiveSessionModelSwitchError` پرتاب کند، Cron پیش از تلاش مجدد، ارائه‌دهنده و مدل تغییرکرده، و در صورت وجود override پروفایل احراز هویت تغییرکرده، را برای اجرای فعال ذخیره می‌کند. حلقه بیرونی تلاش مجدد پس از تلاش اولیه به دو تلاش مجدد تغییر محدود است و سپس به‌جای حلقه بی‌پایان متوقف می‌شود.

## خروجی اجرا و ردها

### سرکوب تأیید کهنه

نوبت‌های Cron ایزوله پاسخ‌های فقط-تأیید کهنه را سرکوب می‌کنند. اگر نتیجه نخست فقط یک به‌روزرسانی وضعیت موقت باشد و هیچ اجرای زیرعامل نواده‌ای مسئول پاسخ نهایی نباشد، Cron یک‌بار دیگر برای نتیجه واقعی prompt می‌کند و سپس تحویل می‌دهد.

### سرکوب توکن خاموش

اگر یک اجرای Cron ایزوله فقط توکن خاموش (`NO_REPLY` یا `no_reply`) را برگرداند، Cron هم تحویل خروجی مستقیم و هم مسیر خلاصه صف‌شده fallback را سرکوب می‌کند؛ بنابراین چیزی به گفت‌وگو ارسال نمی‌شود.

### ردهای ساختاریافته

اجراهای Cron ایزوله ابتدا فراداده رد اجرای ساختاریافته را از اجرای embedded ترجیح می‌دهند، سپس به نشانگرهای رد شناخته‌شده در خروجی نهایی fallback می‌کنند، مانند `SYSTEM_RUN_DENIED`، `INVALID_REQUEST` و عبارت‌های امتناع اتصال approval.

`cron list` و تاریخچه اجرا دلیل رد را به‌جای گزارش یک فرمان مسدودشده به‌صورت `ok` نمایش می‌دهند.

## نگهداری

نگهداری و هرس در پیکربندی کنترل می‌شوند:

- `cron.sessionRetention`، با پیش‌فرض `24h`، نشست‌های اجرای ایزوله تکمیل‌شده را هرس می‌کند.
- `cron.runLog.maxBytes` و `cron.runLog.keepLines`، فایل `~/.openclaw/cron/runs/<jobId>.jsonl` را هرس می‌کنند.

## مهاجرت jobهای قدیمی‌تر

<Note>
اگر jobهای Cron مربوط به پیش از قالب تحویل و ذخیره فعلی دارید، `openclaw doctor --fix` را اجرا کنید. Doctor فیلدهای legacy Cron را نرمال می‌کند (`jobId`، `schedule.cron`، فیلدهای سطح‌بالای تحویل از جمله `threadId` legacy، نام‌های مستعار تحویل `provider` در payload) و وقتی `cron.webhook` پیکربندی شده باشد، jobهای webhook fallback ساده با `notify: true` را به تحویل webhook صریح مهاجرت می‌دهد.
</Note>

## ویرایش‌های رایج

به‌روزرسانی تنظیمات تحویل بدون تغییر پیام:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

غیرفعال‌کردن تحویل برای یک job ایزوله:

```bash
openclaw cron edit <job-id> --no-deliver
```

فعال‌کردن زمینه bootstrap سبک برای یک job ایزوله:

```bash
openclaw cron edit <job-id> --light-context
```

اعلان به یک کانال مشخص:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

اعلان به یک موضوع انجمن Telegram:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

ایجاد یک job ایزوله با زمینه bootstrap سبک:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` فقط برای jobهای نوبت عامل ایزوله اعمال می‌شود. برای اجراهای Cron، حالت سبک به‌جای تزریق مجموعه کامل bootstrap فضای کاری، زمینه bootstrap را خالی نگه می‌دارد.

## فرمان‌های رایج مدیر

اجرای دستی و بازرسی:

```bash
openclaw cron list
openclaw cron list --agent ops
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

`openclaw cron list` به‌طور پیش‌فرض همه jobهای مطابق را نشان می‌دهد. برای نمایش فقط jobهایی که شناسه عامل نرمال‌شده مؤثر آن‌ها مطابقت دارد، `--agent <id>` را پاس دهید؛ jobهای بدون شناسه عامل ذخیره‌شده، به‌عنوان عامل پیش‌فرض پیکربندی‌شده شمرده می‌شوند.

`cron list --json` و `cron show <job-id> --json` روی هر job یک فیلد سطح‌بالای `status` دارند که از `enabled`، `state.runningAtMs` و `state.lastRunStatus` محاسبه می‌شود. مقدارها: `disabled`، `running`، `ok`، `error`، `skipped`، یا `idle`. این وضعیت ستون خوانای انسانی را بازتاب می‌دهد تا ابزارهای بیرونی بتوانند وضعیت job را بدون محاسبه دوباره آن بخوانند.

ورودی‌های `cron runs` شامل عیب‌یابی‌های تحویل با هدف Cron موردنظر، هدف حل‌شده، ارسال‌های ابزار پیام، استفاده از fallback و وضعیت تحویل‌شده هستند.

هدف‌گیری دوباره عامل و نشست:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` وقتی `--agent` در jobهای نوبت عامل حذف شده باشد هشدار می‌دهد و به عامل پیش‌فرض (`main`) fallback می‌کند. برای سنجاق‌کردن یک عامل مشخص، هنگام ایجاد `--agent <id>` را پاس دهید.

تنظیم‌های تحویل:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
