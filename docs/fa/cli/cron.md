# `openclaw cron`

کارهای Cron را برای زمان‌بند Gateway مدیریت کنید.

<Tip>
برای دیدن سطح کامل فرمان، `openclaw cron --help` را اجرا کنید. برای راهنمای مفهومی، [کارهای Cron](/fa/automation/cron-jobs) را ببینید.
</Tip>

## نشست‌ها

`--session` مقدارهای `main`، `isolated`، `current` یا `session:<id>` را می‌پذیرد.

<AccordionGroup>
  <Accordion title="Session keys">
    - `main` به نشست اصلی عامل متصل می‌شود.
    - `isolated` برای هر اجرا یک رونوشت و شناسه نشست تازه ایجاد می‌کند.
    - `current` در زمان ایجاد به نشست فعال متصل می‌شود.
    - `session:<id>` به یک کلید نشست پایدار و صریح پین می‌شود.

  </Accordion>
  <Accordion title="Isolated session semantics">
    اجراهای ایزوله، زمینه گفت‌وگوی محیطی را بازنشانی می‌کنند. مسیریابی کانال و گروه، سیاست ارسال/صف، ارتقا، مبدا، و اتصال زمان اجرای ACP برای اجرای جدید بازنشانی می‌شوند. ترجیح‌های ایمن و بازنویسی‌های صریح مدل یا احراز هویت انتخاب‌شده توسط کاربر می‌توانند بین اجراها منتقل شوند.
  </Accordion>
</AccordionGroup>

## تحویل

`openclaw cron list` و `openclaw cron show <job-id>` مسیر تحویل حل‌شده را پیش‌نمایش می‌کنند. برای `channel: "last"`، پیش‌نمایش نشان می‌دهد که مسیر از نشست اصلی یا فعلی حل شده است، یا به‌صورت بسته شکست خواهد خورد.

هدف‌های دارای پیشوند ارائه‌دهنده می‌توانند کانال‌های اعلام حل‌نشده را رفع ابهام کنند. برای نمونه، وقتی `delivery.channel` حذف شده یا `last` باشد، `to: "telegram:123"` گزینه Telegram را انتخاب می‌کند. فقط پیشوندهایی که Plugin بارگذاری‌شده اعلام کرده است انتخابگر ارائه‌دهنده هستند. اگر `delivery.channel` صریح باشد، پیشوند باید با همان کانال مطابقت داشته باشد؛ `channel: "whatsapp"` همراه با `to: "telegram:123"` رد می‌شود. پیشوندهای سرویس مانند `imessage:` و `sms:` همچنان نحو هدف تحت مالکیت کانال باقی می‌مانند.

<Note>
کارهای ایزوله `cron add` به‌طور پیش‌فرض از تحویل `--announce` استفاده می‌کنند. برای داخلی نگه داشتن خروجی از `--no-deliver` استفاده کنید. `--deliver` همچنان به‌عنوان نام مستعار منسوخ‌شده برای `--announce` باقی مانده است.
</Note>

### مالکیت تحویل

تحویل گفت‌وگوی Cron ایزوله بین عامل و اجراکننده مشترک است:

- عامل می‌تواند وقتی یک مسیر گفت‌وگو در دسترس باشد، مستقیما با ابزار `message` ارسال کند.
- `announce` فقط زمانی پاسخ نهایی را به‌صورت جایگزین تحویل می‌دهد که عامل مستقیما به هدف حل‌شده ارسال نکرده باشد.
- `webhook` محموله تکمیل‌شده را به یک URL ارسال می‌کند.
- `none` تحویل جایگزین اجراکننده را غیرفعال می‌کند.

`--announce` تحویل جایگزین اجراکننده برای پاسخ نهایی است. `--no-deliver` آن جایگزین را غیرفعال می‌کند اما وقتی یک مسیر گفت‌وگو در دسترس باشد، ابزار `message` عامل را حذف نمی‌کند.

یادآورها که از یک گفت‌وگوی فعال ساخته می‌شوند، هدف تحویل گفت‌وگوی زنده را برای تحویل اعلام جایگزین حفظ می‌کنند. کلیدهای نشست داخلی ممکن است با حروف کوچک باشند؛ از آن‌ها به‌عنوان منبع حقیقت برای شناسه‌های ارائه‌دهنده حساس به بزرگی و کوچکی حروف، مانند شناسه اتاق‌های Matrix، استفاده نکنید.

### تحویل شکست

اعلان‌های شکست به این ترتیب حل می‌شوند:

1. `delivery.failureDestination` روی کار.
2. `cron.failureDestination` سراسری.
3. هدف اعلام اصلی کار، وقتی مقصد شکست صریحی تنظیم نشده باشد.

<Note>
کارهای نشست اصلی فقط وقتی می‌توانند از `delivery.failureDestination` استفاده کنند که حالت تحویل اصلی `webhook` باشد. کارهای ایزوله آن را در همه حالت‌ها می‌پذیرند.
</Note>

نکته: اجراهای Cron ایزوله، شکست‌های عامل در سطح اجرا را حتی وقتی
هیچ محموله پاسخی تولید نشود، خطای کار در نظر می‌گیرند؛ بنابراین شکست‌های مدل/ارائه‌دهنده همچنان شمارنده‌های خطا را افزایش می‌دهند
و اعلان‌های شکست را فعال می‌کنند.

## زمان‌بندی

### کارهای تک‌اجرا

`--at <datetime>` یک اجرای تک‌باره زمان‌بندی می‌کند. زمان‌های بدون offset به‌عنوان UTC در نظر گرفته می‌شوند، مگر اینکه `--tz <iana>` را هم بدهید؛ در این حالت زمان ساعت دیواری در منطقه زمانی داده‌شده تفسیر می‌شود.

<Note>
کارهای تک‌اجرا به‌طور پیش‌فرض پس از موفقیت حذف می‌شوند. برای حفظ آن‌ها از `--keep-after-run` استفاده کنید.
</Note>

### کارهای تکرارشونده

کارهای تکرارشونده پس از خطاهای پیاپی از پس‌نشینی نمایی تلاش مجدد استفاده می‌کنند: 30s، 1m، 5m، 15m، 60m. زمان‌بندی پس از اجرای موفق بعدی به حالت عادی بازمی‌گردد.

اجراهای ردشده جدا از خطاهای اجرا ردیابی می‌شوند. آن‌ها بر پس‌نشینی تلاش مجدد اثر نمی‌گذارند، اما `openclaw cron edit <job-id> --failure-alert-include-skipped` می‌تواند اعلان‌های شکست را برای اعلان‌های تکراری اجرای ردشده فعال کند.

برای کارهای ایزوله‌ای که یک ارائه‌دهنده مدل محلی پیکربندی‌شده را هدف می‌گیرند، Cron پیش از شروع نوبت عامل یک پیش‌پرواز سبک ارائه‌دهنده اجرا می‌کند. ارائه‌دهندگان local loopback، شبکه خصوصی، و `.local` با `api: "ollama"` در `/api/tags` بررسی می‌شوند؛ ارائه‌دهندگان محلی سازگار با OpenAI مانند vLLM، SGLang و LM Studio در `/models` بررسی می‌شوند. اگر endpoint در دسترس نباشد، اجرا با وضعیت `skipped` ثبت می‌شود و در زمان‌بندی بعدی دوباره تلاش می‌شود؛ endpointهای مرده مطابق برای 5 دقیقه کش می‌شوند تا از کوبیدن کارهای زیاد به همان سرور محلی جلوگیری شود.

نکته: تعریف‌های کار Cron در `jobs.json` قرار دارند، در حالی که وضعیت runtime معلق در `jobs-state.json` قرار دارد. اگر `jobs.json` به‌صورت خارجی ویرایش شود، Gateway زمان‌بندی‌های تغییریافته را دوباره بارگذاری می‌کند و جایگاه‌های معلق قدیمی را پاک می‌کند؛ بازنویسی‌های صرفا قالب‌بندی، جایگاه معلق را پاک نمی‌کنند.

### اجراهای دستی

`openclaw cron run` به‌محض اینکه اجرای دستی در صف قرار بگیرد برمی‌گردد. پاسخ‌های موفق شامل `{ ok: true, enqueued: true, runId }` هستند. برای دنبال کردن نتیجه نهایی از `openclaw cron runs --id <job-id>` استفاده کنید.

<Note>
`openclaw cron run <job-id>` به‌طور پیش‌فرض اجرا را اجباری می‌کند. برای حفظ رفتار قدیمی «فقط اگر موعد رسیده باشد اجرا کن» از `--due` استفاده کنید.
</Note>

## مدل‌ها

`cron add|edit --model <ref>` یک مدل مجاز را برای کار انتخاب می‌کند.

<Warning>
اگر مدل مجاز نباشد یا قابل حل نباشد، Cron به‌جای بازگشت به انتخاب مدل عامل کار یا مدل پیش‌فرض، اجرا را با یک خطای اعتبارسنجی صریح ناموفق می‌کند.
</Warning>

`--model` در Cron یک **گزینه اصلی کار** است، نه بازنویسی `/model` در نشست گفت‌وگو. یعنی:

- fallbackهای پیکربندی‌شده مدل همچنان وقتی مدل انتخاب‌شده کار شکست بخورد اعمال می‌شوند.
- `fallbacks` در محموله هر کار، وقتی وجود داشته باشد، فهرست fallback پیکربندی‌شده را جایگزین می‌کند.
- فهرست خالی fallback برای هر کار (`fallbacks: []` در محموله/API کار) اجرای Cron را سخت‌گیرانه می‌کند.
- وقتی یک کار `--model` دارد اما هیچ فهرست fallback پیکربندی نشده است، OpenClaw یک بازنویسی fallback خالی و صریح می‌فرستد تا گزینه اصلی عامل به‌عنوان هدف تلاش مجدد پنهان اضافه نشود.

### تقدم مدل Cron ایزوله

Cron ایزوله مدل فعال را به این ترتیب حل می‌کند:

1. بازنویسی قلاب Gmail.
2. `--model` هر کار.
3. بازنویسی مدل نشست Cron ذخیره‌شده، وقتی کاربر یکی را انتخاب کرده باشد.
4. انتخاب مدل عامل یا پیش‌فرض.

### حالت سریع

حالت سریع Cron ایزوله از انتخاب مدل زنده حل‌شده پیروی می‌کند. پیکربندی مدل `params.fastMode` به‌طور پیش‌فرض اعمال می‌شود، اما بازنویسی ذخیره‌شده نشست برای `fastMode` همچنان بر پیکربندی مقدم است.

### تلاش‌های مجدد تعویض مدل زنده

اگر یک اجرای ایزوله `LiveSessionModelSwitchError` پرتاب کند، Cron ارائه‌دهنده و مدل تعویض‌شده، و بازنویسی پروفایل احراز هویت تعویض‌شده را وقتی وجود داشته باشد، پیش از تلاش مجدد برای اجرای فعال پایدار می‌کند. حلقه تلاش مجدد بیرونی به دو تلاش مجدد تعویض پس از تلاش اولیه محدود است، سپس به‌جای حلقه بی‌پایان متوقف می‌شود.

## خروجی اجرا و ردها

### سرکوب تایید کهنه

نوبت‌های Cron ایزوله پاسخ‌های صرفا تاییدی کهنه را سرکوب می‌کنند. اگر نتیجه نخست فقط یک به‌روزرسانی وضعیت موقت باشد و هیچ اجرای subagent فرزندی مسئول پاسخ نهایی نباشد، Cron یک بار دیگر برای نتیجه واقعی پیش از تحویل درخواست می‌کند.

### سرکوب توکن خاموش

اگر یک اجرای Cron ایزوله فقط توکن خاموش (`NO_REPLY` یا `no_reply`) را برگرداند، Cron هم تحویل خروجی مستقیم و هم مسیر خلاصه صف‌شده جایگزین را سرکوب می‌کند؛ بنابراین چیزی به گفت‌وگو ارسال نمی‌شود.

### ردهای ساختاریافته

اجراهای Cron ایزوله metadata رد اجرای ساختاریافته را از اجرای تعبیه‌شده ترجیح می‌دهند، سپس به نشانگرهای رد شناخته‌شده در خروجی نهایی، مانند `SYSTEM_RUN_DENIED`، `INVALID_REQUEST`، و عبارت‌های امتناع مربوط به اتصال approval، بازمی‌گردند.

`cron list` و تاریخچه اجرا دلیل رد را به‌جای گزارش یک فرمان مسدودشده به‌عنوان `ok` نمایش می‌دهند.

## نگهداری

نگهداری و هرس در پیکربندی کنترل می‌شوند:

- `cron.sessionRetention` (پیش‌فرض `24h`) نشست‌های اجرای ایزوله تکمیل‌شده را هرس می‌کند.
- `cron.runLog.maxBytes` و `cron.runLog.keepLines` فایل `~/.openclaw/cron/runs/<jobId>.jsonl` را هرس می‌کنند.

## مهاجرت کارهای قدیمی‌تر

<Note>
اگر از پیش از قالب فعلی تحویل و ذخیره‌سازی، کارهای Cron دارید، `openclaw doctor --fix` را اجرا کنید. Doctor فیلدهای قدیمی Cron (`jobId`، `schedule.cron`، فیلدهای تحویل سطح بالا از جمله `threadId` قدیمی، نام‌های مستعار تحویل `provider` در محموله) را عادی‌سازی می‌کند و وقتی `cron.webhook` پیکربندی شده باشد، کارهای ساده fallback وبهوک با `notify: true` را به تحویل وبهوک صریح مهاجرت می‌دهد.
</Note>

## ویرایش‌های رایج

تنظیمات تحویل را بدون تغییر پیام به‌روزرسانی کنید:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "123456789"
```

تحویل را برای یک کار ایزوله غیرفعال کنید:

```bash
openclaw cron edit <job-id> --no-deliver
```

زمینه راه‌اندازی سبک را برای یک کار ایزوله فعال کنید:

```bash
openclaw cron edit <job-id> --light-context
```

به یک کانال مشخص اعلام کنید:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
```

به یک موضوع انجمن Telegram اعلام کنید:

```bash
openclaw cron edit <job-id> --announce --channel telegram --to "-1001234567890" --thread-id 42
```

یک کار ایزوله با زمینه راه‌اندازی سبک ایجاد کنید:

```bash
openclaw cron add \
  --name "Lightweight morning brief" \
  --cron "0 7 * * *" \
  --session isolated \
  --message "Summarize overnight updates." \
  --light-context \
  --no-deliver
```

`--light-context` فقط برای کارهای نوبت عامل ایزوله اعمال می‌شود. برای اجراهای Cron، حالت سبک زمینه راه‌اندازی را به‌جای تزریق مجموعه کامل راه‌اندازی workspace، خالی نگه می‌دارد.

## فرمان‌های مدیریتی رایج

اجرای دستی و بازرسی:

```bash
openclaw cron list
openclaw cron show <job-id>
openclaw cron run <job-id>
openclaw cron run <job-id> --due
openclaw cron runs --id <job-id> --limit 50
```

ورودی‌های `cron runs` شامل عیب‌یابی تحویل با هدف Cron موردنظر، هدف حل‌شده، ارسال‌های ابزار پیام، استفاده از fallback، و وضعیت تحویل‌شده هستند.

تغییر هدف عامل و نشست:

```bash
openclaw cron edit <job-id> --agent ops
openclaw cron edit <job-id> --clear-agent
openclaw cron edit <job-id> --session current
openclaw cron edit <job-id> --session "session:daily-brief"
```

`openclaw cron add` وقتی `--agent` روی کارهای نوبت عامل حذف شده باشد هشدار می‌دهد و به عامل پیش‌فرض (`main`) بازمی‌گردد. هنگام ایجاد، `--agent <id>` را بدهید تا یک عامل مشخص پین شود.

تنظیمات کوچک تحویل:

```bash
openclaw cron edit <job-id> --announce --channel slack --to "channel:C1234567890"
openclaw cron edit <job-id> --best-effort-deliver
openclaw cron edit <job-id> --no-best-effort-deliver
openclaw cron edit <job-id> --no-deliver
```
