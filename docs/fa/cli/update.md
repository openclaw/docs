# `openclaw update`

OpenClaw را با ایمنی به‌روزرسانی کنید و بین کانال‌های پایدار/beta/dev جابه‌جا شوید.

اگر از طریق **npm/pnpm/bun** نصب کرده‌اید (نصب سراسری، بدون فرادادهٔ git)،
به‌روزرسانی‌ها از طریق جریان مدیر بسته در [به‌روزرسانی](/fa/install/updating) انجام می‌شوند.

## کاربرد

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## گزینه‌ها

- `--no-restart`: پس از به‌روزرسانی موفق، از راه‌اندازی مجدد سرویس Gateway صرف‌نظر می‌کند. به‌روزرسانی‌های مدیر بسته‌ای که Gateway را راه‌اندازی مجدد می‌کنند، پیش از موفق‌شدن فرمان بررسی می‌کنند که سرویسِ دوباره راه‌اندازی‌شده نسخهٔ به‌روزشدهٔ مورد انتظار را گزارش کند.
- `--channel <stable|beta|dev>`: کانال به‌روزرسانی را تنظیم می‌کند (git + npm؛ در پیکربندی ماندگار می‌شود).
- `--tag <dist-tag|version|spec>`: فقط برای همین به‌روزرسانی، هدف بسته را بازنویسی می‌کند. برای نصب‌های بسته‌ای، `main` به `github:openclaw/openclaw#main` نگاشت می‌شود.
- `--dry-run`: اقدامات برنامه‌ریزی‌شدهٔ به‌روزرسانی (جریان کانال/برچسب/هدف/راه‌اندازی مجدد) را بدون نوشتن پیکربندی، نصب، همگام‌سازی Pluginها، یا راه‌اندازی مجدد پیش‌نمایش می‌کند.
- `--json`: JSON قابل‌خواندن برای ماشینِ `UpdateRunResult` را چاپ می‌کند، از جمله
  `postUpdate.plugins.warnings` وقتی Pluginهای مدیریت‌شدهٔ خراب یا غیرقابل‌بارگذاری پس از موفقیت به‌روزرسانی هسته به
  تعمیر نیاز دارند، و `postUpdate.plugins.integrityDrifts`
  وقتی هنگام همگام‌سازی Plugin پس از به‌روزرسانی، انحراف آرتیفکت Plugin npm شناسایی شود.
- `--timeout <seconds>`: مهلت زمانی برای هر مرحله (پیش‌فرض 1800 ثانیه است).
- `--yes`: اعلان‌های تأیید را رد می‌کند (برای مثال تأیید کاهش نسخه).

`openclaw update` پرچم `--verbose` ندارد. از `--dry-run` برای پیش‌نمایش
اقدامات برنامه‌ریزی‌شدهٔ کانال/برچسب/نصب/راه‌اندازی مجدد، از `--json` برای نتایج
قابل‌خواندن برای ماشین، و از `openclaw update status --json` وقتی فقط به جزئیات
کانال و دسترس‌پذیری نیاز دارید استفاده کنید. اگر در حال اشکال‌زدایی لاگ‌های Gateway پیرامون یک به‌روزرسانی هستید،
پُرگویی کنسول و سطح لاگ فایل جدا هستند: `--verbose` در Gateway بر
خروجی ترمینال/WebSocket اثر می‌گذارد، در حالی که لاگ‌های فایل به `logging.level: "debug"` یا
`"trace"` در پیکربندی نیاز دارند. [لاگ‌برداری Gateway](/fa/gateway/logging) را ببینید.

<Note>
در حالت Nix (`OPENCLAW_NIX_MODE=1`)، اجرای تغییردهندهٔ `openclaw update` غیرفعال است. به‌جای آن، منبع Nix یا ورودی flake را برای این نصب به‌روزرسانی کنید؛ برای nix-openclaw، از [شروع سریع](https://github.com/openclaw/nix-openclaw#quick-start) مبتنی بر عامل استفاده کنید. `openclaw update status` و `openclaw update --dry-run` همچنان فقط‌خواندنی می‌مانند.
</Note>

<Warning>
کاهش نسخه به تأیید نیاز دارد، زیرا نسخه‌های قدیمی‌تر می‌توانند پیکربندی را خراب کنند.
</Warning>

## `update status`

کانال فعال به‌روزرسانی + برچسب/شاخه/SHA در git (برای checkoutهای منبع)، به‌همراه دسترس‌پذیری به‌روزرسانی را نشان می‌دهد.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

گزینه‌ها:

- `--json`: JSON وضعیت قابل‌خواندن برای ماشین را چاپ می‌کند.
- `--timeout <seconds>`: مهلت زمانی برای بررسی‌ها (پیش‌فرض 3 ثانیه است).

## `update wizard`

جریانی تعاملی برای انتخاب کانال به‌روزرسانی و تأیید اینکه آیا Gateway
پس از به‌روزرسانی راه‌اندازی مجدد شود یا نه (پیش‌فرض راه‌اندازی مجدد است). اگر `dev` را بدون checkout در git انتخاب کنید،
پیشنهاد می‌دهد یکی ایجاد کند.

گزینه‌ها:

- `--timeout <seconds>`: مهلت زمانی برای هر مرحلهٔ به‌روزرسانی (پیش‌فرض `1800`)

## چه کار می‌کند

وقتی کانال‌ها را به‌صورت صریح جابه‌جا می‌کنید (`--channel ...`)، OpenClaw همچنین
روش نصب را هم‌تراز نگه می‌دارد:

- `dev` → وجود یک checkout در git را تضمین می‌کند (پیش‌فرض: `~/openclaw`، قابل‌بازنویسی با `OPENCLAW_GIT_DIR`)،
  آن را به‌روزرسانی می‌کند و CLI سراسری را از همان checkout نصب می‌کند.
- `stable` → از npm با استفاده از `latest` نصب می‌کند.
- `beta` → dist-tag‏ npm با نام `beta` را ترجیح می‌دهد، اما وقتی beta
  موجود نباشد یا از انتشار پایدار فعلی قدیمی‌تر باشد، به `latest` برمی‌گردد.

OpenClaw هنوز کانال پشتیبانی LTS یا ماهانه ندارد. ما در حال حرکت
به‌سمت خطوط پشتیبانی ماهانه هستیم، اما `--channel` در حال حاضر فقط
`stable`، `beta`، و `dev` را می‌پذیرد. وقتی به یک آرتیفکت بستهٔ مشخص نیاز دارید، از `--tag <version-or-dist-tag>` برای یک
هدف تک‌باره استفاده کنید.

به‌روزرسان خودکار هستهٔ Gateway (وقتی از طریق پیکربندی فعال باشد) مسیر به‌روزرسانی CLI را
خارج از هندلر زندهٔ درخواست Gateway اجرا می‌کند. به‌روزرسانی‌های مدیر بستهٔ control-plane با `update.run`
پس از جابه‌جایی بسته، یک راه‌اندازی مجدد به‌روزرسانیِ غیربه‌تعویق‌افتاده و بدون دورهٔ انتظار را اجبار می‌کنند،
زیرا فرایند قدیمی Gateway ممکن است همچنان قطعه‌هایی در حافظه داشته باشد که به
فایل‌های حذف‌شده توسط بستهٔ جدید اشاره می‌کنند.

برای نصب‌های مدیر بسته، `openclaw update` نسخهٔ بستهٔ هدف را پیش از
فراخوانی مدیر بسته حل می‌کند. نصب‌های سراسری npm از نصب مرحله‌ای استفاده می‌کنند:
OpenClaw بستهٔ جدید را در یک پیشوند موقت npm نصب می‌کند، موجودی `dist`
بسته‌بندی‌شده را در آنجا بررسی می‌کند، سپس آن درخت بستهٔ پاک را با
پیشوند سراسری واقعی جابه‌جا می‌کند. اگر بررسی ناموفق شود، doctor پس از به‌روزرسانی، همگام‌سازی Plugin، و
کار راه‌اندازی مجدد از درخت مشکوک اجرا نمی‌شوند. حتی وقتی نسخهٔ نصب‌شده
از قبل با هدف مطابقت داشته باشد، فرمان نصب بستهٔ سراسری را تازه‌سازی می‌کند،
سپس همگام‌سازی Plugin، تازه‌سازی تکمیل فرمان هسته، و کار راه‌اندازی مجدد را اجرا می‌کند. این
کار، sidecarهای بسته‌بندی‌شده و رکوردهای Plugin تحت مالکیت کانال را با
build نصب‌شدهٔ OpenClaw هم‌تراز نگه می‌دارد، در حالی که بازسازی‌های کامل تکمیل فرمان Plugin را به
اجرای صریح `openclaw completion --write-state` واگذار می‌کند.

وقتی یک سرویس Gateway مدیریت‌شدهٔ محلی نصب شده و راه‌اندازی مجدد فعال باشد،
به‌روزرسانی‌های مدیر بسته پیش از جایگزین‌کردن درخت بسته، سرویس در حال اجرا را متوقف می‌کنند،
سپس فرادادهٔ سرویس را از نصب به‌روزشده تازه‌سازی می‌کنند، سرویس را دوباره راه‌اندازی می‌کنند،
و پیش از گزارش موفقیت، بررسی می‌کنند که Gateway دوباره راه‌اندازی‌شده نسخهٔ مورد انتظار را گزارش کند.
در macOS، بررسی پس از به‌روزرسانی همچنین تأیید می‌کند که LaunchAgent
برای پروفایل فعال بارگذاری/در حال اجرا است و پورت loopback پیکربندی‌شده
سالم است. اگر plist نصب شده باشد اما launchd بر آن نظارت نکند، OpenClaw
به‌طور خودکار LaunchAgent را دوباره bootstrap می‌کند، سپس بررسی‌های
آمادگی سلامت/نسخه/کانال را دوباره اجرا می‌کند. یک bootstrap تازه، کار RunAtLoad
را مستقیماً بارگذاری می‌کند، بنابراین بازیابی به‌روزرسانی بلافاصله روی Gateway
تازه ایجادشده `kickstart -k` اجرا نمی‌کند. اگر Gateway همچنان سالم نشود، فرمان با
کد غیرصفر خارج می‌شود و مسیر لاگ راه‌اندازی مجدد به‌علاوه دستورالعمل‌های صریح راه‌اندازی مجدد، نصب دوباره، و
بازگردانی بسته را چاپ می‌کند. با `--no-restart`،
جایگزینی بسته همچنان اجرا می‌شود، اما سرویس مدیریت‌شده متوقف یا
دوباره راه‌اندازی نمی‌شود؛ بنابراین Gateway در حال اجرا ممکن است تا زمانی که آن را
دستی راه‌اندازی مجدد کنید، کد قدیمی را نگه دارد.

## جریان checkout در git

### انتخاب کانال

- `stable`: آخرین برچسب غیر beta را checkout می‌کند، سپس build و doctor را اجرا می‌کند.
- `beta`: آخرین برچسب `-beta` را ترجیح می‌دهد، اما وقتی beta موجود نباشد یا قدیمی‌تر باشد، به آخرین برچسب پایدار برمی‌گردد.
- `dev`: `main` را checkout می‌کند، سپس fetch و rebase می‌کند.

### مراحل به‌روزرسانی

<Steps>
  <Step title="بررسی پاک‌بودن worktree">
    مستلزم نبود تغییرات commitنشده است.
  </Step>
  <Step title="جابه‌جایی کانال">
    به کانال انتخاب‌شده (برچسب یا شاخه) جابه‌جا می‌شود.
  </Step>
  <Step title="واکشی upstream">
    فقط dev.
  </Step>
  <Step title="Build پیش‌پرواز (فقط dev)">
    build TypeScript را در یک worktree موقت اجرا می‌کند. اگر نوک شاخه ناموفق شود، تا 10 commit به عقب می‌رود تا جدیدترین commit قابل build را پیدا کند. `OPENCLAW_UPDATE_PREFLIGHT_LINT=1` را تنظیم کنید تا lint نیز در طول این پیش‌پرواز اجرا شود؛ lint در حالت سریال محدود اجرا می‌شود، زیرا میزبان‌های به‌روزرسانی کاربر اغلب از runnerهای CI کوچک‌تر هستند.
  </Step>
  <Step title="Rebase">
    روی commit انتخاب‌شده rebase می‌کند (فقط dev).
  </Step>
  <Step title="نصب وابستگی‌ها">
    از مدیر بستهٔ repo استفاده می‌کند. برای checkoutهای pnpm، به‌روزرسان در صورت نیاز `pnpm` را bootstrap می‌کند (ابتدا از طریق `corepack`، سپس fallback موقت `npm install pnpm@10`) به‌جای اینکه `npm run build` را داخل یک workspace‏ pnpm اجرا کند.
  </Step>
  <Step title="Build کردن Control UI">
    gateway و Control UI را build می‌کند.
  </Step>
  <Step title="اجرای doctor">
    `openclaw doctor` به‌عنوان بررسی نهایی به‌روزرسانی ایمن اجرا می‌شود.
  </Step>
  <Step title="همگام‌سازی Pluginها">
    Pluginها را با کانال فعال همگام می‌کند. dev از Pluginهای bundled استفاده می‌کند؛ stable و beta از npm استفاده می‌کنند. نصب‌های Plugin رهگیری‌شده را به‌روزرسانی می‌کند.
  </Step>
</Steps>

در کانال به‌روزرسانی beta، نصب‌های Plugin رهگیری‌شدهٔ npm و ClawHub که خط
پیش‌فرض/latest را دنبال می‌کنند، ابتدا یک انتشار `@beta` برای Plugin را امتحان می‌کنند. اگر Plugin هیچ
انتشار beta نداشته باشد، OpenClaw به spec ثبت‌شدهٔ پیش‌فرض/latest برمی‌گردد. برای Pluginهای
npm، OpenClaw همچنین وقتی بستهٔ beta وجود داشته باشد اما در اعتبارسنجی نصب
ناموفق شود، به عقب برمی‌گردد. نسخه‌های دقیق و برچسب‌های صریح بازنویسی نمی‌شوند.

<Warning>
اگر یک به‌روزرسانی Plugin npm با پین دقیق به آرتیفکتی حل شود که یکپارچگی آن با رکورد نصب ذخیره‌شده متفاوت است، `openclaw update` به‌جای نصب آن، به‌روزرسانی آرتیفکت Plugin را لغو می‌کند. فقط پس از بررسی اینکه به آرتیفکت جدید اعتماد دارید، Plugin را صراحتاً دوباره نصب یا به‌روزرسانی کنید.
</Warning>

<Note>
شکست‌های همگام‌سازی Plugin پس از به‌روزرسانی که به یک Plugin مدیریت‌شده محدود هستند، پس از موفقیت به‌روزرسانی هسته به‌صورت هشدار گزارش می‌شوند. نتیجهٔ JSON وضعیت سطح بالای به‌روزرسانی را به‌شکل `status: "ok"` نگه می‌دارد و `postUpdate.plugins.status: "warning"` را همراه با راهنمایی `openclaw doctor --fix` و `openclaw plugins inspect <id> --runtime --json` گزارش می‌کند. استثناهای غیرمنتظرهٔ به‌روزرسان یا همگام‌سازی همچنان نتیجهٔ به‌روزرسانی را ناموفق می‌کنند. خطای نصب یا به‌روزرسانی Plugin را رفع کنید، سپس `openclaw doctor --fix` یا `openclaw update` را دوباره اجرا کنید.

وقتی Gateway به‌روزشده شروع به کار می‌کند، بارگذاری Plugin فقط-بررسی است: شروع به کار، مدیر بسته‌ها را اجرا نمی‌کند یا درخت‌های وابستگی را تغییر نمی‌دهد. راه‌اندازی‌های مجدد `update.run` مدیر بسته پس از جابه‌جایی درخت بسته، تعویق عادی حالت بیکار و دورهٔ انتظار راه‌اندازی مجدد را دور می‌زنند، بنابراین فرایند قدیمی نمی‌تواند به بارگذاری تنبل قطعه‌های حذف‌شده ادامه دهد.

اگر bootstrap کردن pnpm همچنان ناموفق شود، به‌روزرسان به‌جای تلاش برای `npm run build` داخل checkout، زودتر با یک خطای ویژهٔ مدیر بسته متوقف می‌شود.
</Note>

## اختصار `--update`

`openclaw --update` به `openclaw update` بازنویسی می‌شود (برای shellها و اسکریپت‌های launcher مفید است).
