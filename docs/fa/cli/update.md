# `openclaw update`

OpenClaw را به‌صورت ایمن به‌روزرسانی کنید و بین کانال‌های پایدار/بتا/توسعه جابه‌جا شوید.

اگر از طریق **npm/pnpm/bun** نصب کرده‌اید (نصب سراسری، بدون فراداده‌ی git)،
به‌روزرسانی‌ها از طریق جریان مدیر بسته در [به‌روزرسانی](/fa/install/updating) انجام می‌شوند.

## نحوه استفاده

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## گزینه‌ها

- `--no-restart`: پس از به‌روزرسانی موفق، راه‌اندازی مجدد سرویس Gateway را رد می‌کند. به‌روزرسانی‌های مدیر بسته‌ای که Gateway را دوباره راه‌اندازی می‌کنند، پیش از موفق شدن فرمان بررسی می‌کنند که سرویس دوباره راه‌اندازی‌شده نسخه‌ی به‌روزشده‌ی مورد انتظار را گزارش دهد.
- `--channel <stable|beta|dev>`: کانال به‌روزرسانی را تنظیم می‌کند (git + npm؛ در پیکربندی ماندگار می‌شود).
- `--tag <dist-tag|version|spec>`: هدف بسته را فقط برای همین به‌روزرسانی بازنویسی می‌کند. برای نصب‌های بسته‌ای، `main` به `github:openclaw/openclaw#main` نگاشت می‌شود.
- `--dry-run`: کنش‌های برنامه‌ریزی‌شده‌ی به‌روزرسانی (جریان کانال/برچسب/هدف/راه‌اندازی مجدد) را بدون نوشتن پیکربندی، نصب، همگام‌سازی plugins، یا راه‌اندازی مجدد پیش‌نمایش می‌دهد.
- `--json`: JSON قابل خواندن برای ماشینِ `UpdateRunResult` را چاپ می‌کند، از جمله
  `postUpdate.plugins.warnings` وقتی plugins مدیریت‌شده‌ی خراب یا بارنشدنی پس از
  موفقیت به‌روزرسانی هسته به ترمیم نیاز دارند، و `postUpdate.plugins.integrityDrifts`
  وقتی در همگام‌سازی Plugin پس از به‌روزرسانی، انحراف مصنوع بسته‌ی npm Plugin شناسایی می‌شود.
- `--timeout <seconds>`: مهلت زمانی برای هر گام (پیش‌فرض 1800s است).
- `--yes`: درخواست‌های تأیید را رد می‌کند (برای مثال تأیید تنزل نسخه).

`openclaw update` پرچم `--verbose` ندارد. برای پیش‌نمایش
کنش‌های برنامه‌ریزی‌شده‌ی کانال/برچسب/نصب/راه‌اندازی مجدد از `--dry-run`،
برای نتایج قابل خواندن برای ماشین از `--json`، و وقتی فقط به جزئیات کانال و
دسترس‌پذیری نیاز دارید از `openclaw update status --json` استفاده کنید.
اگر در حال اشکال‌زدایی لاگ‌های Gateway پیرامون یک به‌روزرسانی هستید،
پرحرفی کنسول و سطح لاگ فایل جدا هستند: `--verbose` در Gateway بر خروجی
ترمینال/WebSocket اثر می‌گذارد، در حالی که لاگ‌های فایل به `logging.level: "debug"` یا
`"trace"` در پیکربندی نیاز دارند. [لاگ‌گیری Gateway](/fa/gateway/logging) را ببینید.

<Note>
در حالت Nix (`OPENCLAW_NIX_MODE=1`)، اجراهای تغییردهنده‌ی `openclaw update` غیرفعال هستند. به‌جای آن، منبع Nix یا ورودی flake را برای این نصب به‌روزرسانی کنید؛ برای nix-openclaw، از [شروع سریع](https://github.com/openclaw/nix-openclaw#quick-start) عامل‌محور استفاده کنید. `openclaw update status` و `openclaw update --dry-run` همچنان فقط‌خواندنی می‌مانند.
</Note>

<Warning>
تنزل نسخه به تأیید نیاز دارد، چون نسخه‌های قدیمی‌تر می‌توانند پیکربندی را خراب کنند.
</Warning>

## `update status`

کانال به‌روزرسانی فعال + برچسب/شاخه/SHA در git (برای checkoutهای منبع)، به‌همراه دسترس‌پذیری به‌روزرسانی را نشان می‌دهد.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

گزینه‌ها:

- `--json`: JSON وضعیت قابل خواندن برای ماشین را چاپ می‌کند.
- `--timeout <seconds>`: مهلت زمانی برای بررسی‌ها (پیش‌فرض 3s است).

## `update wizard`

جریان تعاملی برای انتخاب یک کانال به‌روزرسانی و تأیید اینکه پس از به‌روزرسانی Gateway
دوباره راه‌اندازی شود یا نه (پیش‌فرض راه‌اندازی مجدد است). اگر بدون checkout از git گزینه‌ی `dev` را انتخاب کنید،
پیشنهاد می‌کند یکی ایجاد کند.

گزینه‌ها:

- `--timeout <seconds>`: مهلت زمانی برای هر گام به‌روزرسانی (پیش‌فرض `1800`)

## کاری که انجام می‌دهد

وقتی کانال‌ها را به‌صورت صریح عوض می‌کنید (`--channel ...`)، OpenClaw روش
نصب را نیز هم‌راستا نگه می‌دارد:

- `dev` → وجود یک checkout از git را تضمین می‌کند (پیش‌فرض: `~/openclaw`، قابل بازنویسی با `OPENCLAW_GIT_DIR`)،
  آن را به‌روزرسانی می‌کند، و CLI سراسری را از همان checkout نصب می‌کند.
- `stable` → با استفاده از `latest` از npm نصب می‌کند.
- `beta` → dist-tag مربوط به npm یعنی `beta` را ترجیح می‌دهد، اما وقتی beta
  وجود ندارد یا از انتشار پایدار فعلی قدیمی‌تر است، به `latest` برمی‌گردد.

به‌روزرسان خودکار هسته‌ی Gateway (وقتی از طریق پیکربندی فعال باشد) مسیر به‌روزرسانی CLI را
بیرون از handler درخواست زنده‌ی Gateway اجرا می‌کند. به‌روزرسانی‌های مدیر بسته‌ی `update.run`
در control plane پس از تعویض بسته، یک راه‌اندازی مجدد به‌روزرسانی غیربه‌تعویق‌افتاده و بدون دوره‌ی انتظار را اجباری می‌کنند،
چون فرایند Gateway قدیمی ممکن است همچنان chunkهای درون حافظه داشته باشد که به
فایل‌های حذف‌شده توسط بسته‌ی جدید اشاره می‌کنند.

برای نصب‌های مدیر بسته، `openclaw update` پیش از فراخوانی مدیر بسته، نسخه‌ی
بسته‌ی هدف را resolve می‌کند. نصب‌های سراسری npm از نصب مرحله‌بندی‌شده
استفاده می‌کنند: OpenClaw بسته‌ی جدید را در یک prefix موقت npm نصب می‌کند،
inventory بسته‌بندی‌شده‌ی `dist` را آنجا بررسی می‌کند، سپس آن درخت بسته‌ی پاک را در
prefix سراسری واقعی جایگزین می‌کند. اگر راستی‌آزمایی شکست بخورد، doctor پس از به‌روزرسانی، همگام‌سازی Plugin، و
کار راه‌اندازی مجدد از درخت مشکوک اجرا نمی‌شوند. حتی وقتی نسخه‌ی نصب‌شده
از پیش با هدف هم‌خوان است، فرمان نصب بسته‌ی سراسری را تازه‌سازی می‌کند،
سپس همگام‌سازی Plugin، تازه‌سازی تکمیل فرمان هسته، و کار راه‌اندازی مجدد را اجرا می‌کند. این
sidecarهای بسته‌بندی‌شده و رکوردهای Plugin متعلق به کانال را با ساخت
نصب‌شده‌ی OpenClaw هم‌راستا نگه می‌دارد، در حالی که بازسازی کامل تکمیل فرمان‌های Plugin را به
اجراهای صریح `openclaw completion --write-state` واگذار می‌کند.

وقتی یک سرویس Gateway مدیریت‌شده‌ی محلی نصب شده باشد و راه‌اندازی مجدد فعال باشد،
به‌روزرسانی‌های مدیر بسته، سرویس در حال اجرا را پیش از جایگزین کردن درخت بسته
متوقف می‌کنند، سپس فراداده‌ی سرویس را از نصب به‌روزشده تازه‌سازی می‌کنند، سرویس را دوباره
راه‌اندازی می‌کنند، و پیش از گزارش موفقیت بررسی می‌کنند که Gateway دوباره‌راه‌اندازی‌شده
نسخه‌ی مورد انتظار را گزارش دهد. در macOS، بررسی پس از به‌روزرسانی همچنین بررسی می‌کند که LaunchAgent
برای پروفایل فعال بارگذاری/در حال اجرا باشد و پورت loopback پیکربندی‌شده
سالم باشد. اگر plist نصب شده باشد اما launchd آن را نظارت نکند، OpenClaw
LaunchAgent را به‌صورت خودکار دوباره bootstrap می‌کند، سپس بررسی‌های
آمادگی سلامت/نسخه/کانال را دوباره اجرا می‌کند. یک bootstrap تازه کار RunAtLoad
را مستقیماً بارگذاری می‌کند، بنابراین بازیابی به‌روزرسانی بلافاصله روی Gateway
تازه ایجادشده `kickstart -k` اجرا نمی‌کند. اگر Gateway همچنان سالم نشود، فرمان با
کد غیرصفر خارج می‌شود و مسیر لاگ راه‌اندازی مجدد به‌همراه دستورالعمل‌های صریح راه‌اندازی مجدد، نصب مجدد، و
برگرداندن بسته را چاپ می‌کند. با `--no-restart`،
جایگزینی بسته همچنان اجرا می‌شود، اما سرویس مدیریت‌شده متوقف یا
دوباره راه‌اندازی نمی‌شود، بنابراین Gateway در حال اجرا ممکن است تا زمانی که آن را
دستی دوباره راه‌اندازی کنید، کد قدیمی را نگه دارد.

## جریان checkout از git

### انتخاب کانال

- `stable`: تازه‌ترین برچسب غیر بتا را checkout می‌کند، سپس build و doctor را اجرا می‌کند.
- `beta`: تازه‌ترین برچسب `-beta` را ترجیح می‌دهد، اما وقتی beta وجود ندارد یا قدیمی‌تر است، به تازه‌ترین برچسب پایدار برمی‌گردد.
- `dev`: `main` را checkout می‌کند، سپس fetch و rebase انجام می‌دهد.

### گام‌های به‌روزرسانی

<Steps>
  <Step title="Verify clean worktree">
    نیاز دارد هیچ تغییر commitنشده‌ای وجود نداشته باشد.
  </Step>
  <Step title="Switch channel">
    به کانال انتخاب‌شده (برچسب یا شاخه) جابه‌جا می‌شود.
  </Step>
  <Step title="Fetch upstream">
    فقط برای توسعه.
  </Step>
  <Step title="Preflight build (dev only)">
    build TypeScript را در یک worktree موقت اجرا می‌کند. اگر tip شکست بخورد، تا 10 commit به عقب می‌رود تا جدیدترین commit قابل build را پیدا کند. `OPENCLAW_UPDATE_PREFLIGHT_LINT=1` را تنظیم کنید تا lint نیز در این پیش‌بررسی اجرا شود؛ lint در حالت سری محدودشده اجرا می‌شود چون میزبان‌های به‌روزرسانی کاربران اغلب از runnerهای CI کوچک‌تر هستند.
  </Step>
  <Step title="Rebase">
    روی commit انتخاب‌شده rebase می‌کند (فقط توسعه).
  </Step>
  <Step title="Install dependencies">
    از مدیر بسته‌ی repo استفاده می‌کند. برای checkoutهای pnpm، به‌روزرسان `pnpm` را در صورت نیاز bootstrap می‌کند (ابتدا از طریق `corepack`، سپس fallback موقت `npm install pnpm@10`) به‌جای اینکه داخل یک workspace مربوط به pnpm فرمان `npm run build` را اجرا کند.
  </Step>
  <Step title="Build Control UI">
    Gateway و Control UI را build می‌کند.
  </Step>
  <Step title="Run doctor">
    `openclaw doctor` به‌عنوان آخرین بررسی به‌روزرسانی ایمن اجرا می‌شود.
  </Step>
  <Step title="Sync plugins">
    plugins را با کانال فعال همگام می‌کند. توسعه از plugins همراه‌شده استفاده می‌کند؛ stable و beta از npm استفاده می‌کنند. نصب‌های Plugin ردیابی‌شده را به‌روزرسانی می‌کند.
  </Step>
</Steps>

در کانال به‌روزرسانی beta، نصب‌های npm و ClawHub Plugin ردیابی‌شده که خط
پیش‌فرض/latest را دنبال می‌کنند ابتدا یک انتشار `@beta` از Plugin را امتحان می‌کنند. اگر Plugin انتشار
beta نداشته باشد، OpenClaw به spec ثبت‌شده‌ی پیش‌فرض/latest برمی‌گردد. برای npm
plugins، OpenClaw همچنین وقتی بسته‌ی beta وجود دارد اما راستی‌آزمایی نصب شکست می‌خورد
برمی‌گردد. نسخه‌های دقیق و برچسب‌های صریح بازنویسی نمی‌شوند.

<Warning>
اگر یک به‌روزرسانی npm Plugin دقیقاً pinشده به مصنوعی resolve شود که یکپارچگی آن با رکورد نصب ذخیره‌شده فرق دارد، `openclaw update` به‌جای نصب آن، به‌روزرسانی مصنوع Plugin را متوقف می‌کند. فقط پس از بررسی اینکه به مصنوع جدید اعتماد دارید، Plugin را صریحاً دوباره نصب یا به‌روزرسانی کنید.
</Warning>

<Note>
شکست‌های همگام‌سازی Plugin پس از به‌روزرسانی که به یک Plugin مدیریت‌شده محدود هستند، پس از موفقیت به‌روزرسانی هسته به‌صورت هشدار گزارش می‌شوند. نتیجه‌ی JSON وضعیت به‌روزرسانی سطح بالا را به‌شکل `status: "ok"` نگه می‌دارد و `postUpdate.plugins.status: "warning"` را همراه با راهنمایی‌های `openclaw doctor --fix` و `openclaw plugins inspect <id> --runtime --json` گزارش می‌کند. استثناهای غیرمنتظره‌ی به‌روزرسان یا همگام‌سازی همچنان نتیجه‌ی به‌روزرسانی را ناموفق می‌کنند. خطای نصب یا به‌روزرسانی Plugin را رفع کنید، سپس `openclaw doctor --fix` یا `openclaw update` را دوباره اجرا کنید.

وقتی Gateway به‌روزشده شروع به کار می‌کند، بارگذاری Plugin فقط برای راستی‌آزمایی است: startup مدیران بسته را اجرا نمی‌کند یا درخت‌های وابستگی را تغییر نمی‌دهد. راه‌اندازی‌های مجدد مدیر بسته‌ی `update.run` پس از جایگزین شدن درخت بسته، تعویق idle معمول و دوره‌ی انتظار راه‌اندازی مجدد را دور می‌زنند، بنابراین فرایند قدیمی نمی‌تواند chunkهای حذف‌شده را lazy-load نگه دارد.

اگر bootstrap مربوط به pnpm همچنان شکست بخورد، به‌روزرسان به‌جای تلاش برای اجرای `npm run build` داخل checkout، زودتر با یک خطای ویژه‌ی مدیر بسته متوقف می‌شود.
</Note>

## خلاصه‌نویسی `--update`

`openclaw --update` به `openclaw update` بازنویسی می‌شود (برای shellها و اسکریپت‌های launcher مفید است).
