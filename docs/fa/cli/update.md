# `openclaw update`

OpenClaw را با اطمینان به‌روزرسانی کنید و بین کانال‌های پایدار/بتا/توسعه جابه‌جا شوید.

اگر از طریق **npm/pnpm/bun** نصب کرده‌اید (نصب سراسری، بدون فراداده git)،
به‌روزرسانی‌ها از طریق جریان مدیر بسته در [به‌روزرسانی](/fa/install/updating) انجام می‌شوند.

## استفاده

```bash
openclaw update
openclaw update status
openclaw update wizard
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --json
openclaw --update
```

## گزینه‌ها

- `--no-restart`: پس از به‌روزرسانی موفق، راه‌اندازی دوباره سرویس Gateway را رد می‌کند. به‌روزرسانی‌های مدیر بسته که Gateway را دوباره راه‌اندازی می‌کنند، پیش از موفق شدن فرمان بررسی می‌کنند که سرویس راه‌اندازی‌شده دوباره نسخه به‌روزشده مورد انتظار را گزارش می‌دهد.
- `--channel <stable|beta|dev>`: کانال به‌روزرسانی را تنظیم می‌کند (git + npm؛ در پیکربندی ماندگار می‌شود).
- `--tag <dist-tag|version|spec>`: هدف بسته را فقط برای همین به‌روزرسانی بازنویسی می‌کند. برای نصب‌های بسته‌ای، `main` به `github:openclaw/openclaw#main` نگاشت می‌شود.
- `--dry-run`: کنش‌های برنامه‌ریزی‌شده به‌روزرسانی (جریان کانال/برچسب/هدف/راه‌اندازی دوباره) را بدون نوشتن پیکربندی، نصب، همگام‌سازی Pluginها، یا راه‌اندازی دوباره پیش‌نمایش می‌کند.
- `--json`: JSON قابل‌خواندن برای ماشینِ `UpdateRunResult` را چاپ می‌کند، شامل
  `postUpdate.plugins.integrityDrifts` وقتی رانش آرتیفکت Plugin مربوط به npm
  هنگام همگام‌سازی Plugin پس از به‌روزرسانی شناسایی شود.
- `--timeout <seconds>`: مهلت زمانی برای هر گام (پیش‌فرض 1800 ثانیه است).
- `--yes`: اعلان‌های تأیید را رد می‌کند (برای مثال تأیید بازگشت به نسخه قدیمی‌تر).

<Warning>
بازگشت به نسخه قدیمی‌تر نیازمند تأیید است، چون نسخه‌های قدیمی‌تر می‌توانند پیکربندی را خراب کنند.
</Warning>

## `update status`

کانال به‌روزرسانی فعال + برچسب/شاخه/SHA مربوط به git (برای checkoutهای منبع)، به‌همراه در دسترس بودن به‌روزرسانی را نمایش می‌دهد.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

گزینه‌ها:

- `--json`: JSON وضعیت قابل‌خواندن برای ماشین را چاپ می‌کند.
- `--timeout <seconds>`: مهلت زمانی برای بررسی‌ها (پیش‌فرض 3 ثانیه است).

## `update wizard`

جریانی تعاملی برای انتخاب کانال به‌روزرسانی و تأیید اینکه آیا Gateway
پس از به‌روزرسانی دوباره راه‌اندازی شود یا نه (پیش‌فرض، راه‌اندازی دوباره است). اگر `dev` را بدون checkout مربوط به git انتخاب کنید،
پیشنهاد می‌دهد یکی ایجاد کند.

گزینه‌ها:

- `--timeout <seconds>`: مهلت زمانی برای هر گام به‌روزرسانی (پیش‌فرض `1800`)

## چه کاری انجام می‌دهد

وقتی کانال‌ها را به‌صورت صریح عوض می‌کنید (`--channel ...`)، OpenClaw همچنین
روش نصب را همسو نگه می‌دارد:

- `dev` → وجود checkout مربوط به git را تضمین می‌کند (پیش‌فرض: `~/openclaw`، قابل بازنویسی با `OPENCLAW_GIT_DIR`)،
  آن را به‌روزرسانی می‌کند و CLI سراسری را از همان checkout نصب می‌کند.
- `stable` → از npm با استفاده از `latest` نصب می‌کند.
- `beta` → dist-tag مربوط به npm با نام `beta` را ترجیح می‌دهد، اما وقتی بتا
  وجود نداشته باشد یا از انتشار پایدار فعلی قدیمی‌تر باشد، به `latest` بازمی‌گردد.

به‌روزرسان خودکار هسته Gateway (وقتی از طریق پیکربندی فعال باشد) مسیر به‌روزرسانی CLI را
بیرون از رسیدگی‌کننده درخواست زنده Gateway اجرا می‌کند. به‌روزرسانی‌های مدیر بسته در صفحه کنترل با `update.run`
پس از جایگزینی بسته، راه‌اندازی دوباره به‌روزرسانی را به‌صورت غیربه‌تعویق‌افتاده و بدون دوره انتظار اجباری می‌کنند،
چون فرایند Gateway قدیمی ممکن است همچنان بخش‌هایی را در حافظه داشته باشد که به
فایل‌های حذف‌شده توسط بسته جدید اشاره می‌کنند.

برای نصب‌های مدیر بسته، `openclaw update` پیش از فراخوانی مدیر بسته،
نسخه بسته هدف را حل می‌کند. نصب‌های سراسری npm از نصب مرحله‌ای
استفاده می‌کنند: OpenClaw بسته جدید را در یک پیشوند موقت npm نصب می‌کند، موجودی
`dist` بسته‌بندی‌شده را در آنجا بررسی می‌کند، سپس آن درخت بسته پاک را به
پیشوند سراسری واقعی جابه‌جا می‌کند. اگر بررسی شکست بخورد، doctor پس از به‌روزرسانی، همگام‌سازی Plugin، و
کار راه‌اندازی دوباره از درخت مشکوک اجرا نمی‌شوند. حتی وقتی نسخه نصب‌شده
از قبل با هدف یکی باشد، فرمان نصب بسته سراسری را تازه‌سازی می‌کند،
سپس همگام‌سازی Plugin، تازه‌سازی تکمیل فرمان‌های هسته، و کار راه‌اندازی دوباره را اجرا می‌کند. این
کار sidecarهای بسته‌بندی‌شده و رکوردهای Plugin متعلق به کانال را با
ساخت نصب‌شده OpenClaw همسو نگه می‌دارد، در حالی که بازسازی‌های کامل تکمیل فرمان‌های Plugin را به
اجرای صریح `openclaw completion --write-state` واگذار می‌کند.

وقتی یک سرویس Gateway مدیریت‌شده محلی نصب باشد و راه‌اندازی دوباره فعال باشد،
به‌روزرسانی‌های مدیر بسته پیش از جایگزینی درخت بسته، سرویس در حال اجرا را متوقف می‌کنند،
سپس فراداده سرویس را از نصب به‌روزشده تازه‌سازی می‌کنند، سرویس را دوباره راه‌اندازی می‌کنند،
و بررسی می‌کنند که Gateway راه‌اندازی‌شده دوباره نسخه مورد انتظار را گزارش می‌دهد. با
`--no-restart`، جایگزینی بسته همچنان انجام می‌شود اما سرویس مدیریت‌شده
متوقف یا دوباره راه‌اندازی نمی‌شود، بنابراین Gateway در حال اجرا ممکن است تا وقتی آن را
دستی دوباره راه‌اندازی کنید، کد قدیمی را نگه دارد.

## جریان checkout مربوط به git

### انتخاب کانال

- `stable`: تازه‌ترین برچسب غیر بتا را checkout می‌کند، سپس build و doctor را اجرا می‌کند.
- `beta`: تازه‌ترین برچسب `-beta` را ترجیح می‌دهد، اما وقتی بتا وجود نداشته باشد یا قدیمی‌تر باشد، به تازه‌ترین برچسب پایدار بازمی‌گردد.
- `dev`: `main` را checkout می‌کند، سپس fetch و rebase را انجام می‌دهد.

### گام‌های به‌روزرسانی

<Steps>
  <Step title="Verify clean worktree">
    نیازمند نبود تغییرات commitنشده است.
  </Step>
  <Step title="Switch channel">
    به کانال انتخاب‌شده (برچسب یا شاخه) جابه‌جا می‌شود.
  </Step>
  <Step title="Fetch upstream">
    فقط برای Dev.
  </Step>
  <Step title="Preflight build (dev only)">
    lint و build مربوط به TypeScript را در یک worktree موقت اجرا می‌کند. اگر tip شکست بخورد، تا 10 commit به عقب می‌رود تا جدیدترین build پاک را پیدا کند.
  </Step>
  <Step title="Rebase">
    روی commit انتخاب‌شده rebase می‌کند (فقط dev).
  </Step>
  <Step title="Install dependencies">
    از مدیر بسته مخزن استفاده می‌کند. برای checkoutهای pnpm، به‌روزرسان `pnpm` را هنگام نیاز راه‌اندازی اولیه می‌کند (ابتدا از طریق `corepack`، سپس با جایگزین موقت `npm install pnpm@10`) به‌جای اینکه `npm run build` را داخل workspace مربوط به pnpm اجرا کند.
  </Step>
  <Step title="Build Control UI">
    gateway و Control UI را build می‌کند.
  </Step>
  <Step title="Run doctor">
    `openclaw doctor` به‌عنوان بررسی نهایی به‌روزرسانی ایمن اجرا می‌شود.
  </Step>
  <Step title="Sync plugins">
    Pluginها را با کانال فعال همگام‌سازی می‌کند. Dev از Pluginهای bundled استفاده می‌کند؛ stable و beta از npm استفاده می‌کنند. Pluginهای نصب‌شده با npm را به‌روزرسانی می‌کند.
  </Step>
</Steps>

<Warning>
اگر به‌روزرسانی دقیق و سنجاق‌شده یک Plugin مربوط به npm به آرتیفکتی حل شود که یکپارچگی آن با رکورد نصب ذخیره‌شده تفاوت دارد، `openclaw update` به‌جای نصب آن، به‌روزرسانی آرتیفکت Plugin را متوقف می‌کند. فقط پس از بررسی اینکه به آرتیفکت جدید اعتماد دارید، Plugin را به‌صورت صریح دوباره نصب یا به‌روزرسانی کنید.
</Warning>

<Note>
شکست‌های همگام‌سازی Plugin پس از به‌روزرسانی، نتیجه به‌روزرسانی را ناموفق می‌کنند و کارهای بعدی راه‌اندازی دوباره را متوقف می‌کنند. خطای نصب یا به‌روزرسانی Plugin را رفع کنید، سپس `openclaw update` را دوباره اجرا کنید.

وقتی Gateway به‌روزشده شروع به کار می‌کند، وابستگی‌های runtime مربوط به Pluginهای bundled فعال، پیش از فعال‌سازی Plugin مرحله‌بندی می‌شوند. راه‌اندازی‌های دوباره `update.run` مربوط به مدیر بسته، پس از جابه‌جایی درخت بسته، تعویق عادی هنگام بیکاری و دوره انتظار راه‌اندازی دوباره را دور می‌زنند، بنابراین فرایند قدیمی نمی‌تواند به بارگذاری تنبل بخش‌های حذف‌شده ادامه دهد. راه‌اندازی‌های دوباره مدیر سرویس همچنان پیش از بستن Gateway، مرحله‌بندی وابستگی runtime را تخلیه می‌کنند.

اگر راه‌اندازی اولیه pnpm همچنان شکست بخورد، به‌روزرسان به‌جای تلاش برای اجرای `npm run build` داخل checkout، زودتر با خطای مختص مدیر بسته متوقف می‌شود.
</Note>

## میان‌بر `--update`

`openclaw --update` به `openclaw update` بازنویسی می‌شود (برای shellها و اسکریپت‌های launcher مفید است).
