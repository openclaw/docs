# `openclaw status`

عیب‌یابی برای کانال‌ها + نشست‌ها.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

نکات:

- `--deep` پروب‌های زنده را اجرا می‌کند (WhatsApp Web + Telegram + Discord + Slack + Signal).
- `openclaw status` ساده در مسیر سریعِ فقط‌خواندنی می‌ماند و وقتی بازرسی حافظه را رد می‌کند، حافظه را به‌جای ناموجود بودن با `not checked` علامت‌گذاری می‌کند. ممیزی امنیتی سنگین، سازگاری Plugin، و پروب‌های بردار حافظه به `openclaw status --all`، `openclaw status --deep`، `openclaw security audit`، و `openclaw memory status --deep` واگذار می‌شوند.
- `status --json --all` جزئیات حافظه را از زمان‌اجرای Plugin حافظه فعال که توسط `plugins.slots.memory` انتخاب شده گزارش می‌کند. Pluginهای حافظه سفارشی می‌توانند `agents.defaults.memorySearch.enabled` داخلی را غیرفعال نگه دارند و همچنان فایل‌ها، تکه‌ها، بردار، و وضعیت FTS خودشان را گزارش کنند.
- `--usage` پنجره‌های مصرف نرمال‌سازی‌شده ارائه‌دهنده را به‌صورت `X% left` چاپ می‌کند.
- خروجی وضعیت نشست، `Execution:` را از `Runtime:` جدا می‌کند. `Execution` مسیر sandbox است (`direct`، `docker/*`)، در حالی که `Runtime` به شما می‌گوید آیا نشست از `OpenClaw Pi Default`، `OpenAI Codex`، یک پشتانه CLI، یا یک پشتانه ACP مانند `codex (acp/acpx)` استفاده می‌کند. برای تمایز ارائه‌دهنده/مدل/زمان‌اجرا، [زمان‌های اجرای عامل](/fa/concepts/agent-runtimes) را ببینید.
- فیلدهای خام `usage_percent` / `usagePercent` در MiniMax سهمیه باقی‌مانده هستند، بنابراین OpenClaw پیش از نمایش آن‌ها را معکوس می‌کند؛ فیلدهای مبتنی بر شمارش، وقتی وجود داشته باشند، اولویت دارند. پاسخ‌های `model_remains` ورودی مدل چت را ترجیح می‌دهند، در صورت نیاز برچسب پنجره را از زمان‌مهرها استخراج می‌کنند، و نام مدل را در برچسب طرح می‌گنجانند.
- وقتی اسنپ‌شات نشست فعلی پراکنده باشد، `/status` می‌تواند شمارنده‌های توکن و cache را از جدیدترین لاگ مصرف رونوشت تکمیل کند. مقادیر زنده غیرصفر موجود همچنان بر مقادیر جایگزین رونوشت اولویت دارند.
- جایگزین رونوشت همچنین می‌تواند برچسب مدل زمان‌اجرای فعال را وقتی ورودی نشست زنده آن را ندارد بازیابی کند. اگر آن مدل رونوشت با مدل انتخاب‌شده متفاوت باشد، status پنجره زمینه را به‌جای مدل انتخاب‌شده، بر اساس مدل زمان‌اجرای بازیابی‌شده حل می‌کند.
- برای محاسبه اندازه پرامپت، وقتی فراداده نشست وجود ندارد یا کوچک‌تر است، جایگزین رونوشت مجموع بزرگ‌ترِ متمرکز بر پرامپت را ترجیح می‌دهد تا نشست‌های ارائه‌دهنده سفارشی به نمایش توکن `0` فرو نریزند.
- وقتی چندین عامل پیکربندی شده باشند، خروجی شامل انبارهای نشست به‌ازای هر عامل است.
- نمای کلی، وقتی در دسترس باشد، وضعیت نصب/زمان‌اجرای سرویس میزبان Gateway + Node را شامل می‌شود.
- نمای کلی، کانال به‌روزرسانی + SHA گیت را شامل می‌شود (برای checkoutهای منبع).
- اطلاعات به‌روزرسانی در نمای کلی نمایش داده می‌شود؛ اگر به‌روزرسانی‌ای موجود باشد، status راهنمایی برای اجرای `openclaw update` چاپ می‌کند ([به‌روزرسانی](/fa/install/updating) را ببینید).
- سطوح وضعیت فقط‌خواندنی (`status`، `status --json`، `status --all`) در صورت امکان SecretRefهای پشتیبانی‌شده را برای مسیرهای پیکربندی هدفشان حل می‌کنند.
- اگر یک SecretRef کانال پشتیبانی‌شده پیکربندی شده باشد اما در مسیر فرمان فعلی در دسترس نباشد، status فقط‌خواندنی می‌ماند و به‌جای کرش کردن، خروجی تنزل‌یافته گزارش می‌کند. خروجی انسانی هشدارهایی مانند «توکن پیکربندی‌شده در این مسیر فرمان در دسترس نیست» نشان می‌دهد، و خروجی JSON شامل `secretDiagnostics` است.
- وقتی حل SecretRef محلیِ فرمان موفق شود، status اسنپ‌شات حل‌شده را ترجیح می‌دهد و نشانگرهای گذرای «راز در دسترس نیست» کانال را از خروجی نهایی پاک می‌کند.
- `status --all` یک ردیف نمای کلی Secrets و یک بخش تشخیص شامل می‌شود که عیب‌یابی رازها را (برای خوانایی کوتاه‌شده) بدون توقف تولید گزارش خلاصه می‌کند.
