# `openclaw status`

عیب‌یابی برای کانال‌ها + جلسه‌ها.

```bash
openclaw status
openclaw status --all
openclaw status --deep
openclaw status --usage
```

نکات:

- `--deep` کاوش‌های زنده را اجرا می‌کند (WhatsApp Web + Telegram + Discord + Slack + Signal).
- دستور سادهٔ `openclaw status` در مسیر سریع و فقط‌خواندنی باقی می‌ماند و وقتی بازرسی حافظه را رد می‌کند، به‌جای ناموجود نشان دادن حافظه، آن را با `not checked` علامت‌گذاری می‌کند. ممیزی امنیتی سنگین، سازگاری Plugin، و کاوش‌های بردار حافظه به `openclaw status --all`، `openclaw status --deep`، `openclaw security audit`، و `openclaw memory status --deep` واگذار می‌شوند.
- `status --json --all` جزئیات حافظه را از زمان اجرای Plugin حافظهٔ فعالی گزارش می‌کند که توسط `plugins.slots.memory` انتخاب شده است. Pluginهای حافظهٔ سفارشی می‌توانند `agents.defaults.memorySearch.enabled` داخلی را غیرفعال بگذارند و همچنان فایل‌ها، بخش‌ها، بردار، و وضعیت FTS خودشان را گزارش کنند.
- `--usage` پنجره‌های مصرف عادی‌سازی‌شدهٔ ارائه‌دهنده را به‌صورت `X% left` چاپ می‌کند.
- خروجی وضعیت جلسه، `Execution:` را از `Runtime:` جدا می‌کند. `Execution` مسیر sandbox است (`direct`، `docker/*`)، درحالی‌که `Runtime` به شما می‌گوید جلسه از `OpenClaw Pi Default`، `OpenAI Codex`، یک backend مبتنی بر CLI، یا یک backend مبتنی بر ACP مانند `codex (acp/acpx)` استفاده می‌کند. برای تمایز ارائه‌دهنده/مدل/زمان اجرا، [زمان‌های اجرای عامل](/fa/concepts/agent-runtimes) را ببینید.
- فیلدهای خام `usage_percent` / `usagePercent` در MiniMax سهمیهٔ باقی‌مانده هستند، بنابراین OpenClaw پیش از نمایش آن‌ها را معکوس می‌کند؛ فیلدهای مبتنی بر شمارش، وقتی وجود داشته باشند، اولویت دارند. پاسخ‌های `model_remains` ورودی مدل چت را ترجیح می‌دهند، در صورت نیاز برچسب پنجره را از زمان‌مهرها استخراج می‌کنند، و نام مدل را در برچسب طرح وارد می‌کنند.
- وقتی snapshot جلسهٔ فعلی پراکنده باشد، `/status` می‌تواند شمارنده‌های token و cache را از جدیدترین گزارش مصرف transcript تکمیل کند. مقادیر زندهٔ غیرصفر موجود همچنان بر مقادیر fallback مربوط به transcript اولویت دارند.
- `/status` مدت‌زمان اجرای فشردهٔ فرایند Gateway و مدت‌زمان روشن بودن سیستم میزبان را شامل می‌شود.
- fallback مربوط به transcript همچنین می‌تواند برچسب مدل runtime فعال را وقتی ورودی جلسهٔ زنده آن را ندارد بازیابی کند. اگر آن مدل transcript با مدل انتخاب‌شده متفاوت باشد، status پنجرهٔ context را به‌جای مدل انتخاب‌شده، بر اساس مدل runtime بازیابی‌شده حل می‌کند.
- برای محاسبهٔ اندازهٔ prompt، fallback مربوط به transcript وقتی فرادادهٔ جلسه وجود ندارد یا کوچک‌تر است، مجموع بزرگ‌تر و promptمحور را ترجیح می‌دهد، تا جلسه‌های ارائه‌دهندهٔ سفارشی به نمایش `0` token فرو نریزند.
- وقتی چند عامل پیکربندی شده باشند، خروجی شامل storeهای جلسهٔ جداگانه برای هر عامل است.
- در صورت موجود بودن، نمای کلی وضعیت نصب/زمان اجرای سرویس میزبان Gateway + node را شامل می‌شود.
- نمای کلی کانال به‌روزرسانی + git SHA را شامل می‌شود (برای checkoutهای منبع).
- اطلاعات به‌روزرسانی در Overview نمایش داده می‌شود؛ اگر به‌روزرسانی در دسترس باشد، status راهنمایی برای اجرای `openclaw update` چاپ می‌کند ([به‌روزرسانی](/fa/install/updating) را ببینید).
- سطوح فقط‌خواندنی status (`status`، `status --json`، `status --all`) در صورت امکان SecretRefهای پشتیبانی‌شده را برای مسیرهای config هدف خود resolve می‌کنند.
- اگر یک SecretRef کانال پشتیبانی‌شده پیکربندی شده باشد اما در مسیر فرمان فعلی در دسترس نباشد، status فقط‌خواندنی باقی می‌ماند و به‌جای crash کردن، خروجی تنزل‌یافته گزارش می‌کند. خروجی انسانی هشدارهایی مانند «token پیکربندی‌شده در این مسیر فرمان در دسترس نیست» نشان می‌دهد، و خروجی JSON شامل `secretDiagnostics` است.
- وقتی resolve کردن SecretRef محلیِ فرمان موفق شود، status snapshot resolveشده را ترجیح می‌دهد و نشانگرهای موقت «secret در دسترس نیست» مربوط به کانال را از خروجی نهایی پاک می‌کند.
- `status --all` یک ردیف نمای کلی Secrets و یک بخش تشخیص را شامل می‌شود که بدون توقف تولید گزارش، تشخیص‌های secret را خلاصه می‌کند (برای خوانایی کوتاه‌سازی شده است).
