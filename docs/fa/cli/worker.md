# `openclaw worker`

`openclaw worker` نقطهٔ ورود محدودشدهٔ زمان اجرا است که یک هماهنگ‌کنندهٔ کارگر ابری آن را درون محیط آماده‌شدهٔ کارگر راه‌اندازی می‌کند. این فرمانی عمومی برای ثبت دستی کارگر نیست.

Gateway بستهٔ منطبق OpenClaw را نصب و تونل SSH معکوسِ سنجاق‌شده به کلید میزبان را باز می‌کند. راه‌انداز کارگر این فرمان را با یک تخصیص آماده اجرا می‌کند. فرمان از طریق سوکت محلیِ هدایت‌شده به‌وسیلهٔ تونل متصل می‌شود و با نقش اختصاصی `worker` پذیرش می‌شود.

## قرارداد راه‌اندازی

فرمان دقیقاً یک پاکت JSON راه‌اندازی با اندازهٔ محدود را از ورودی استاندارد می‌خواند. این پاکت شامل مکان سوکت محلی، اعتبارنامهٔ صادرشدهٔ کارگر، هویت بسته و پروتکل، دورهٔ مالک، و تنها نشست و نوبت تخصیص‌یافته است. اعتبارنامه هرگز از طریق آرگومان‌های خط فرمان پذیرفته نمی‌شود و این صفحه عمداً هیچ نمونه‌ای از اعتبارنامه یا پاکت دست‌نویس ارائه نمی‌دهد.

اگر پاکت نامعتبر باشد، اعتبارنامه رد شود، قابلیت‌های بسته یا پروتکل مطابقت نداشته باشند، یا نشست و دورهٔ مالک دیگر جاری نباشند، پذیرش به‌صورت بسته و ایمن شکست می‌خورد. اپراتورها باید کارگرها را از طریق هماهنگ‌کنندهٔ کارگر ابری راه‌اندازی کنند، نه اینکه این نقطهٔ ورود را مستقیماً فراخوانی کنند.

## مرز زمان اجرا

فرایند، حلقهٔ عادی عاملِ تعبیه‌شده را با یک بک‌اند محدودشده اجرا می‌کند:

- ابزارهای کدنویسی `read`، `write`، `edit`، `apply_patch`، `exec` و `process`
  به‌صورت محلی در فضای کاری کارگر اجرا می‌شوند.
- فراخوانی‌های مدل از پراکسی استنتاج Gateway استفاده می‌کنند. هیچ نمایهٔ احراز هویت محلی مدل
  بارگذاری نمی‌شود.
- نوشتن رونوشت‌ها از RPC ثبت رونوشت Gateway استفاده می‌کند.
- به‌روزرسانی‌های جریان‌سازی و چرخهٔ عمر ابزار از RPC رویداد زندهٔ Gateway استفاده می‌کنند.
- فقط نشست و نوبت تخصیص‌یافته پذیرفته می‌شوند.

حالت کارگر، کانال‌ها، سطوح HTTP مربوط به Gateway یا شروع خودکار Plugin را فراتر از مجموعه‌ابزار نشست تخصیص‌یافته راه‌اندازی نمی‌کند. این حالت از یک پوشهٔ وضعیت یک‌بارمصرف استفاده می‌کند و هیچ اعتبارنامهٔ دائمی برای ارائه‌دهنده یا forge ندارد.

ارسال نشست از یک کارگر به کارگر دیگر در این حالت ارائه نمی‌شود. جانمایی و ارسال همچنان در مالکیت Gateway باقی می‌مانند: اپراتور می‌تواند یک نشست محلی موجود با درخت‌کاری مدیریت‌شده را از طریق Gateway ارسال کند، اما فرایند کارگر نمی‌تواند خودش یا کارگر دیگری را ارسال کند.

تخصیص آماده‌شده شامل بافت رونوشت، برگ پایهٔ پذیرفته‌شده، توالی ثبت و مکان‌نمای رویداد زنده است. هنگام اتصال مجدد تونل، فرایند با همان اعتبارنامه و دورهٔ مالک دوباره پذیرش می‌شود، پایهٔ پذیرفته‌شدهٔ رونوشت را حفظ می‌کند، دنبالهٔ تأییدنشدهٔ رویدادهای زندهٔ خود را بازپخش می‌کند و یک نوبت استنتاج در حال اجرا را با همان هویت دوباره متصل می‌کند. اگر دلتاهای جریانی از دست رفته باشند، پیام پایانی استنتاج مرجع قطعی است. دورهٔ مالک جایگزین، فرایند را محصور می‌کند و باعث خروج پاک آن می‌شود.

رد رونوشت `stale-base-leaf` اجرای جاری را فوراً متوقف می‌کند. حالت کارگر، توالی ردشده را روی برگی متفاوت دوباره امتحان نمی‌کند؛ بنابراین هیچ ثبت تکراری تولید نمی‌شود و هر دنبالهٔ درون‌حافظه‌ای که هنوز ثبت نشده باشد، از آن اجرا از دست می‌رود. راه‌اندازی مجدد بر عهدهٔ مالک جانمایی milestone-3 است که باید تخصیصی تازه را از رونوشت مرجع و دفتر ثبت Gateway ایجاد کند. به همین ترتیب، راه‌اندازی مجدد فرایند Gateway یک نوبت استنتاج معلق را با خطای ارائه‌دهنده خاتمه می‌دهد؛ فقط اتصال مجدد تونل یا WebSocket کارگر می‌تواند دوباره به جریان استنتاج فعال همان فرایند متصل شود.

برای سطح بستهٔ RPC کارگر به [پروتکل Gateway](/fa/gateway/protocol#worker-role-and-closed-protocol) و برای معماری و مدل امنیتی به [طرح کارگرهای ابری](/fa/plan/cloud-workers) مراجعه کنید.
