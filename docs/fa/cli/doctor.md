# `openclaw doctor`

بررسی‌های سلامت + اصلاح‌های سریع برای Gateway و کانال‌ها.

مرتبط:

- عیب‌یابی: [عیب‌یابی](/fa/gateway/troubleshooting)
- ممیزی امنیتی: [امنیت](/fa/gateway/security)

## نمونه‌ها

```bash
openclaw doctor
openclaw doctor --repair
openclaw doctor --deep
openclaw doctor --repair --non-interactive
openclaw doctor --generate-gateway-token
```

## گزینه‌ها

- `--no-workspace-suggestions`: پیشنهادهای حافظه/جست‌وجوی فضای کاری را غیرفعال کن
- `--yes`: پیش‌فرض‌ها را بدون درخواست تأیید بپذیر
- `--repair`: اصلاح‌های پیشنهادی غیرسرویسی را بدون درخواست تأیید اعمال کن؛ نصب‌ها و بازنویسی‌های سرویس Gateway همچنان به تأیید تعاملی یا فرمان‌های صریح Gateway نیاز دارند
- `--fix`: نام مستعار برای `--repair`
- `--force`: اصلاح‌های تهاجمی را اعمال کن، از جمله بازنویسی پیکربندی سفارشی سرویس در صورت نیاز
- `--non-interactive`: بدون اعلان اجرا کن؛ فقط مهاجرت‌های امن و اصلاح‌های غیرسرویسی
- `--generate-gateway-token`: یک توکن Gateway تولید و پیکربندی کن
- `--deep`: سرویس‌های سیستم را برای نصب‌های اضافی Gateway پویش کن و واگذاری‌های راه‌اندازی مجدد اخیر سرپرست Gateway را گزارش بده

نکته‌ها:

- اعلان‌های تعاملی (مانند اصلاح‌های keychain/OAuth) فقط زمانی اجرا می‌شوند که stdin یک TTY باشد و `--non-interactive` تنظیم **نشده** باشد. اجراهای بدون ترمینال (cron، Telegram، بدون ترمینال) اعلان‌ها را رد می‌کنند.
- کارایی: اجراهای غیرتعاملی `doctor` از بارگذاری مشتاقانه پلاگین‌ها عبور می‌کنند تا بررسی‌های سلامت بدون ترمینال سریع بمانند. نشست‌های تعاملی همچنان وقتی یک بررسی به مشارکت پلاگین‌ها نیاز داشته باشد، آن‌ها را کامل بارگذاری می‌کنند.
- `--fix` (نام مستعار برای `--repair`) یک نسخه پشتیبان در `~/.openclaw/openclaw.json.bak` می‌نویسد و کلیدهای پیکربندی ناشناخته را حذف می‌کند و هر حذف را فهرست می‌کند.
- `doctor --fix --non-interactive` تعریف‌های مفقود یا کهنه سرویس Gateway را گزارش می‌دهد اما خارج از حالت اصلاح به‌روزرسانی، آن‌ها را نصب یا بازنویسی نمی‌کند. برای سرویس مفقود، `openclaw gateway install` را اجرا کنید، یا زمانی که عمداً می‌خواهید راه‌انداز را جایگزین کنید از `openclaw gateway install --force` استفاده کنید.
- بررسی‌های یکپارچگی وضعیت اکنون فایل‌های رونویس یتیم را در دایرکتوری نشست‌ها تشخیص می‌دهند. بایگانی کردن آن‌ها به‌صورت `.deleted.<timestamp>` به تأیید تعاملی نیاز دارد؛ `--fix`، `--yes`، و اجراهای بدون ترمینال آن‌ها را سر جای خود باقی می‌گذارند.
- Doctor همچنین `~/.openclaw/cron/jobs.json` (یا `cron.store`) را برای شکل‌های قدیمی کارهای cron پویش می‌کند و می‌تواند پیش از آنکه زمان‌بند مجبور شود در زمان اجرا آن‌ها را خودکار نرمال‌سازی کند، آن‌ها را درجا بازنویسی کند.
- در Linux، doctor زمانی هشدار می‌دهد که crontab کاربر هنوز `~/.openclaw/bin/ensure-whatsapp.sh` قدیمی را اجرا می‌کند؛ آن اسکریپت دیگر نگهداری نمی‌شود و وقتی cron محیط systemd user-bus را ندارد می‌تواند قطعی‌های کاذب Gateway برای WhatsApp ثبت کند.
- وقتی WhatsApp فعال باشد، doctor وجود یک حلقه رویداد تنزل‌یافته Gateway را در حالی که کلاینت‌های محلی `openclaw-tui` هنوز در حال اجرا هستند بررسی می‌کند. `doctor --fix` فقط کلاینت‌های محلی TUI تأییدشده را متوقف می‌کند تا پاسخ‌های WhatsApp پشت حلقه‌های تازه‌سازی کهنه TUI صف نشوند.
- Doctor وضعیت staging وابستگی پلاگین قدیمی را که نسخه‌های قدیمی‌تر OpenClaw ساخته‌اند پاک می‌کند. همچنین پلاگین‌های قابل‌دانلود مفقودی را که پیکربندی به آن‌ها ارجاع داده، مانند `plugins.entries`، کانال‌های پیکربندی‌شده، تنظیمات پیکربندی‌شده provider/search، یا runtimeهای عامل پیکربندی‌شده، تعمیر می‌کند. در طول به‌روزرسانی‌های بسته، doctor اصلاح پلاگین توسط package-manager را تا پایان تعویض بسته رد می‌کند؛ اگر یک پلاگین پیکربندی‌شده همچنان به بازیابی نیاز دارد، پس از آن دوباره `openclaw doctor --fix` را اجرا کنید. اگر دانلود شکست بخورد، doctor خطای نصب را گزارش می‌دهد و ورودی پلاگین پیکربندی‌شده را برای تلاش اصلاح بعدی حفظ می‌کند.
- Doctor پیکربندی کهنه پلاگین را با حذف شناسه‌های پلاگین مفقود از `plugins.allow`/`plugins.entries`، به‌علاوه پیکربندی کانال آویزان متناظر، هدف‌های Heartbeat، و بازنویسی‌های مدل کانال، زمانی که کشف پلاگین سالم باشد اصلاح می‌کند.
- Doctor پیکربندی نامعتبر پلاگین را با غیرفعال کردن ورودی آسیب‌دیده `plugins.entries.<id>` و حذف payload نامعتبر `config` آن قرنطینه می‌کند. راه‌اندازی Gateway از قبل فقط همان پلاگین خراب را رد می‌کند تا پلاگین‌ها و کانال‌های دیگر بتوانند به اجرا ادامه دهند.
- وقتی سرپرست دیگری چرخه عمر Gateway را مالک است، `OPENCLAW_SERVICE_REPAIR_POLICY=external` را تنظیم کنید. Doctor همچنان سلامت Gateway/سرویس را گزارش می‌دهد و اصلاح‌های غیرسرویسی را اعمال می‌کند، اما نصب/شروع/راه‌اندازی مجدد/bootstrap سرویس و پاک‌سازی سرویس قدیمی را رد می‌کند.
- در Linux، doctor واحدهای systemd اضافی شبیه Gateway را که غیرفعال هستند نادیده می‌گیرد و در طول اصلاح، metadata فرمان/entrypoint را برای یک سرویس در حال اجرای systemd Gateway بازنویسی نمی‌کند. ابتدا سرویس را متوقف کنید یا زمانی که عمداً می‌خواهید راه‌انداز فعال را جایگزین کنید از `openclaw gateway install --force` استفاده کنید.
- Doctor پیکربندی تخت قدیمی Talk (`talk.voiceId`، `talk.modelId` و موارد مشابه) را به `talk.provider` + `talk.providers.<provider>` خودکار مهاجرت می‌دهد.
- اجراهای تکراری `doctor --fix` دیگر وقتی تنها تفاوت ترتیب کلیدهای آبجکت باشد، نرمال‌سازی Talk را گزارش/اعمال نمی‌کنند.
- Doctor یک بررسی آمادگی جست‌وجوی حافظه را شامل می‌شود و وقتی اعتبارنامه‌های embedding مفقود باشند می‌تواند `openclaw configure --section model` را پیشنهاد کند.
- Doctor زمانی هشدار می‌دهد که هیچ مالک فرمانی پیکربندی نشده باشد. مالک فرمان حساب اپراتور انسانی است که اجازه دارد فرمان‌های فقط‌مالک را اجرا کند و اقدام‌های خطرناک را تأیید کند. جفت‌سازی DM فقط اجازه می‌دهد کسی با ربات صحبت کند؛ اگر فرستنده‌ای را پیش از وجود bootstrap مالک اول تأیید کرده‌اید، `commands.ownerAllowFrom` را صریح تنظیم کنید.
- Doctor زمانی هشدار می‌دهد که عامل‌های حالت Codex پیکربندی شده‌اند و دارایی‌های شخصی Codex CLI در خانه Codex اپراتور وجود دارند. راه‌اندازی‌های محلی app-server Codex از خانه‌های جداگانه برای هر عامل استفاده می‌کنند، پس برای موجودی‌گیری دارایی‌هایی که باید آگاهانه ارتقا داده شوند از `openclaw migrate codex --dry-run` استفاده کنید.
- Doctor زمانی هشدار می‌دهد که skills مجاز برای عامل پیش‌فرض در محیط runtime فعلی در دسترس نیستند چون binها، env varها، پیکربندی، یا الزامات OS مفقود هستند. `doctor --fix` می‌تواند آن skills ناموجود را با `skills.entries.<skill>.enabled=false` غیرفعال کند؛ وقتی می‌خواهید skill فعال بماند، به‌جای آن الزام مفقود را نصب/پیکربندی کنید.
- اگر حالت sandbox فعال باشد اما Docker در دسترس نباشد، doctor یک هشدار پرسیگنال همراه با راهکار اصلاح گزارش می‌دهد (`install Docker` یا `openclaw config set agents.defaults.sandbox.mode off`).
- اگر فایل‌های registry قدیمی sandbox (`~/.openclaw/sandbox/containers.json` یا `~/.openclaw/sandbox/browsers.json`) وجود داشته باشند، doctor آن‌ها را گزارش می‌دهد؛ `openclaw doctor --fix` ورودی‌های معتبر را به دایرکتوری‌های registry شاردشده مهاجرت می‌دهد و فایل‌های قدیمی نامعتبر را قرنطینه می‌کند.
- اگر `gateway.auth.token`/`gateway.auth.password` با SecretRef مدیریت شوند و در مسیر فرمان فعلی در دسترس نباشند، doctor یک هشدار فقط‌خواندنی گزارش می‌دهد و credentialهای fallback متن ساده نمی‌نویسد.
- اگر بازرسی SecretRef کانال در مسیر اصلاح شکست بخورد، doctor ادامه می‌دهد و به‌جای خروج زودهنگام یک هشدار گزارش می‌دهد.
- پس از مهاجرت‌های دایرکتوری وضعیت، doctor زمانی هشدار می‌دهد که حساب‌های پیش‌فرض فعال Telegram یا Discord به fallback محیطی وابسته باشند و `TELEGRAM_BOT_TOKEN` یا `DISCORD_BOT_TOKEN` برای فرایند doctor در دسترس نباشد.
- رفع خودکار نام کاربری `allowFrom` در Telegram (`doctor --fix`) به یک توکن Telegram قابل‌رفع در مسیر فرمان فعلی نیاز دارد. اگر بازرسی توکن در دسترس نباشد، doctor یک هشدار گزارش می‌دهد و رفع خودکار را برای آن گذر رد می‌کند.

## macOS: بازنویسی‌های env در `launchctl`

اگر قبلاً `launchctl setenv OPENCLAW_GATEWAY_TOKEN ...` (یا `...PASSWORD`) را اجرا کرده باشید، آن مقدار فایل پیکربندی شما را بازنویسی می‌کند و می‌تواند باعث خطاهای پایدار «غیرمجاز» شود.

```bash
launchctl getenv OPENCLAW_GATEWAY_TOKEN
launchctl getenv OPENCLAW_GATEWAY_PASSWORD

launchctl unsetenv OPENCLAW_GATEWAY_TOKEN
launchctl unsetenv OPENCLAW_GATEWAY_PASSWORD
```
