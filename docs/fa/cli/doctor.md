# `openclaw doctor`

بررسی‌های سلامت و رفع سریع مشکلات Gateway، کانال‌ها، Pluginها، Skills، مسیریابی مدل، وضعیت محلی و مهاجرت‌های پیکربندی. هرگاه چیزی مطابق انتظار عمل نمی‌کند و می‌خواهید یک فرمان علت مشکل را توضیح دهد، از آن استفاده کنید.

مرتبط:

- عیب‌یابی: [عیب‌یابی](/fa/gateway/troubleshooting)
- ممیزی امنیتی: [امنیت](/fa/gateway/security)

## حالت‌ها

Doctor پنج حالت دارد:

| حالت                      | فرمان                                     | رفتار                                                                                         |
| ------------------------- | ----------------------------------------- | --------------------------------------------------------------------------------------------- |
| بازرسی                    | `openclaw doctor`                         | بررسی‌های مناسب برای انسان و اعلان‌های راهنما.                                                |
| تعمیر                     | `openclaw doctor --fix`                   | تعمیرات پشتیبانی‌شده را اعمال می‌کند و جز در صورت ایمن‌بودن تعمیر غیرتعاملی، از اعلان‌ها استفاده می‌کند. |
| Lint                      | `openclaw doctor --lint`                  | یافته‌های ساختاریافته و فقط‌خواندنی برای CI، پیش‌بررسی و دروازه‌های بازبینی.                    |
| نگه‌داری SQLite اشتراکی   | `openclaw doctor --state-sqlite compact`  | پایگاه دادهٔ متعارف وضعیت اشتراکی را به‌طور صریح checkpoint، فشرده و تأیید می‌کند.              |
| مهاجرت SQLite نشست        | `openclaw doctor --session-sqlite <mode>` | وضعیت نشست را بازرسی، وارد، اعتبارسنجی، فشرده، بازیابی یا بازگردانی می‌کند.                     |

وقتی خودکارسازی به نتیجه‌ای پایدار نیاز دارد، `--lint` را ترجیح دهید. وقتی اپراتور انسانی می‌خواهد Doctor پیکربندی یا وضعیت را ویرایش کند، `--fix` را ترجیح دهید.

## نمونه‌ها

```bash
openclaw doctor
openclaw doctor --lint
openclaw doctor --lint --json
openclaw doctor --lint --severity-min warning
openclaw doctor --lint --all
openclaw doctor --lint --allow-exec
openclaw doctor --deep
openclaw doctor --fix
openclaw doctor --fix --non-interactive
openclaw doctor --generate-gateway-token
openclaw doctor --post-upgrade
openclaw doctor --post-upgrade --json
openclaw doctor --state-sqlite compact
openclaw doctor --state-sqlite compact --json
openclaw doctor --session-sqlite inspect --session-sqlite-all-agents
openclaw doctor --session-sqlite dry-run --session-sqlite-agent main --json
openclaw doctor --session-sqlite import --session-sqlite-all-agents
openclaw doctor --session-sqlite validate --session-sqlite-all-agents --json
openclaw doctor --session-sqlite compact --session-sqlite-all-agents
openclaw doctor --session-sqlite recover --github-issue
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

برای مجوزهای مختص هر کانال، به‌جای `doctor` از کاوشگرهای کانال استفاده کنید:

```bash
openclaw channels capabilities --channel discord --target channel:<channel-id>
openclaw channels status --probe
```

`channels capabilities` مجوزهای مؤثر ربات را برای یک مقصد کانال مشخص گزارش می‌کند. `channels status --probe` همهٔ کانال‌های پیکربندی‌شده و مقصدهای پیوستن خودکار صوتی را ممیزی می‌کند.

## گزینه‌ها

| گزینه                           | اثر                                                                                                                                                                                     |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--no-workspace-suggestions`    | پیشنهادهای حافظه/جست‌وجوی فضای کاری را غیرفعال می‌کند.                                                                                                                                   |
| `--yes`                         | مقادیر پیش‌فرض را بدون اعلان می‌پذیرد.                                                                                                                                                   |
| `--repair` / `--fix`            | تعمیرات توصیه‌شدهٔ غیرسرویسی را بدون اعلان اعمال می‌کند (`--fix` نام مستعار است). نصب/بازنویسی سرویس Gateway همچنان به تأیید تعاملی یا فرمان‌های صریح `gateway` نیاز دارد. |
| `--force`                       | تعمیرات تهاجمی، از جمله بازنویسی پیکربندی سفارشی سرویس را اعمال می‌کند.                                                                                                                   |
| `--non-interactive`             | بدون اعلان اجرا می‌شود؛ فقط مهاجرت‌های ایمن و تعمیرات غیرسرویسی.                                                                                                                         |
| `--generate-gateway-token`      | یک توکن Gateway تولید و پیکربندی می‌کند.                                                                                                                                                |
| `--allow-exec`                  | به Doctor اجازه می‌دهد هنگام تأیید اسرار، SecretRefهای پیکربندی‌شدهٔ `exec` را اجرا کند.                                                                                     |
| `--deep`                        | سرویس‌های سیستم را برای نصب‌های اضافی Gateway پویش می‌کند؛ تحویل‌های اخیر راه‌اندازی مجدد ناظر Gateway را گزارش می‌دهد.                                                                  |
| `--lint`                        | بررسی‌های سلامت نوسازی‌شده را در حالت فقط‌خواندنی اجرا و یافته‌های تشخیصی را منتشر می‌کند.                                                                                                |
| `--post-upgrade`                | کاوش‌های سازگاری Plugin پس از ارتقا را اجرا می‌کند؛ یافته‌ها به stdout می‌روند؛ اگر هر یافته‌ای در سطح خطا وجود داشته باشد، کد خروج 1 است.                                                |
| `--state-sqlite <mode>`         | نگه‌داری صریح SQLite وضعیت اشتراکی را اجرا می‌کند. تنها حالت، `compact` است.                                                                                                    |
| `--session-sqlite <mode>`       | حالت هدفمند مهاجرت SQLite نشست را اجرا می‌کند: `inspect`، `dry-run`، `import`، `validate`، `compact`، `recover` یا `restore`. |
| `--session-sqlite-store <path>` | همراه با `--session-sqlite`: یک مسیر ذخیره‌گاه قدیمی `sessions.json` را انتخاب می‌کند.                                                                                               |
| `--session-sqlite-agent <id>`   | همراه با `--session-sqlite`: یک عامل پیکربندی‌شده را انتخاب می‌کند.                                                                                                                      |
| `--session-sqlite-all-agents`   | همراه با `--session-sqlite`: ذخیره‌گاه‌های عامل پیکربندی‌شده و کشف‌شده را انتخاب می‌کند.                                                                                                 |
| `--github-issue`                | همراه با `--session-sqlite recover`: یک گزارش پاک‌سازی‌شده برای مسئلهٔ openclaw/openclaw آماده می‌کند؛ Doctor پس از `--yes` یا تأیید تعاملی، آن را با `gh` ایجاد می‌کند. |
| `--json`                        | همراه با `--lint`: یافته‌های JSON. همراه با `--post-upgrade`: `{ probesRun, findings }`. همراه با `--state-sqlite` یا `--session-sqlite`: گزارش نگه‌داری به‌شکل JSON.               |
| `--severity-min <level>`        | همراه با `--lint`: یافته‌های پایین‌تر از `info`، `warning` یا `error` را حذف می‌کند.                                                           |
| `--all`                         | همراه با `--lint`: همهٔ بررسی‌های ثبت‌شده، از جمله بررسی‌های نیازمند انتخاب صریح که از مجموعهٔ پیش‌فرض کنار گذاشته شده‌اند، اجرا می‌شوند.                                      |
| `--skip <id>`                   | همراه با `--lint`: یک شناسهٔ بررسی را نادیده می‌گیرد. قابل تکرار است.                                                                                                          |
| `--only <id>`                   | همراه با `--lint`: فقط شناسه یا شناسه‌های بررسی داده‌شده را اجرا می‌کند. قابل تکرار است.                                                                                       |

`--severity-min`، `--all`، `--only` و `--skip` فقط همراه با `--lint` پذیرفته می‌شوند؛ `--json` همراه با `--lint`، `--post-upgrade`، `--state-sqlite` و `--session-sqlite` پذیرفته می‌شود.

## حالت Lint

`openclaw doctor --lint` فقط‌خواندنی است: بدون اعلان، بدون تعمیر و بدون بازنویسی پیکربندی/وضعیت.

```bash
openclaw doctor --lint
openclaw doctor --lint --severity-min warning
openclaw doctor --lint --json
openclaw doctor --lint --all
openclaw doctor --lint --allow-exec
openclaw doctor --lint --only core/doctor/gateway-config --json
openclaw doctor --lint --only core/doctor/local-audio-acceleration --severity-min info
```

خروجی انسانی فشرده است:

```text
doctor --lint: 6 بررسی اجرا شد، 1 یافته
  [warning] core/doctor/gateway-config gateway.mode - gateway.mode تنظیم نشده است؛ شروع gateway مسدود خواهد شد.
    رفع: `openclaw configure` را اجرا و حالت Gateway را (local/remote) تنظیم کنید، یا `openclaw config set gateway.mode local` را اجرا کنید.
```

خروجی JSON رابط اسکریپت‌نویسی است:

```json
{
  "ok": false,
  "checksRun": 5,
  "checksSkipped": 0,
  "findings": [
    {
      "checkId": "core/doctor/gateway-config",
      "severity": "warning",
      "message": "gateway.mode تنظیم نشده است؛ شروع gateway مسدود خواهد شد.",
      "path": "gateway.mode",
      "fixHint": "`openclaw configure` را اجرا و حالت Gateway را (local/remote) تنظیم کنید، یا `openclaw config set gateway.mode local` را اجرا کنید."
    }
  ]
}
```

کدهای خروج:

| کد | معنی                                                               |
| --- | ------------------------------------------------------------------ |
| `0`  | هیچ یافته‌ای در آستانهٔ شدت انتخاب‌شده یا بالاتر از آن وجود ندارد. |
| `1`  | دست‌کم یک یافته با آستانهٔ انتخاب‌شده مطابقت دارد.                  |
| `2`  | شکست فرمان/زمان اجرا پیش از آنکه یافته‌های Lint تولید شوند.         |

`--severity-min` هم یافته‌هایی را که نمایش داده می‌شوند و هم آستانهٔ خروج را کنترل می‌کند: حتی وقتی یافته‌های کم‌شدت‌تر `info`/`warning` وجود دارند، `openclaw doctor --lint --severity-min error` می‌تواند چیزی نمایش ندهد و با `0` خارج شود.

`--all` تعیین می‌کند کدام بررسی‌ها پیش از پالایش شدت انتخاب شوند. اجرای پیش‌فرض Lint بررسی‌های عمیق، تاریخی یا بررسی‌هایی را که احتمال بیشتری دارد بقایای قدیمی قابل تعمیر را آشکار کنند کنار می‌گذارد؛ برای فهرست کامل از `--all` استفاده کنید. `--only <id>` دقیق‌ترین انتخاب‌گر است و می‌تواند هر بررسی ثبت‌شده‌ای را بر اساس شناسه اجرا کند.

`core/doctor/local-audio-acceleration` فرمان محلی STT انتخاب‌شده به‌صورت خودکار، شواهد جداگانهٔ backend توانمند/درخواست‌شده/مشاهده‌شده و ترتیب fallback را بدون بارگیری مدل گفتار گزارش می‌کند. این گزینه یک یافتهٔ اطلاعاتی منتشر می‌کند؛ بنابراین برای نمایش آن `--severity-min info` را وارد کنید.

## بررسی‌های سلامت ساختاریافته

بررسی‌های مدرن Doctor از یک قرارداد تفکیک‌شدهٔ کوچک استفاده می‌کنند:

```ts
detect(ctx, scope?) -> HealthFinding[]
repair?(ctx, findings) -> HealthRepairResult
```

`detect()` نیروی محرک `doctor --lint` است. `repair()` اختیاری است و فقط تحت `doctor --fix` / `doctor --repair` اجرا می‌شود. بررسی‌هایی که هنوز به این ساختار مهاجرت نکرده‌اند، همچنان از جریان قدیمی مشارکت Doctor استفاده می‌کنند.

زمینه‌های تعمیر می‌توانند درخواست‌های `dryRun`/`diff` را حمل کنند؛ نتایج تعمیر می‌توانند `diffs` ساختاریافته (ویرایش‌های پیکربندی/فایل) و `effects` (سرویس، فرایند، بسته، وضعیت یا دیگر عوارض جانبی) را برگردانند؛ بنابراین بررسی‌های تبدیل‌شده می‌توانند بدون انتقال برنامه‌ریزی تغییر به `detect()` به‌سمت `doctor --fix --dry-run` گسترش یابند.

`repair()`، `status: "repaired" | "skipped" | "failed"` را گزارش می‌کند (حذف وضعیت به‌معنای `repaired` است). وقتی تعمیر مقدار `skipped` یا `failed` را برمی‌گرداند، doctor دلیل را گزارش می‌کند و اعتبارسنجی آن بررسی را نادیده می‌گیرد. پس از تعمیر موفق، doctor دوباره `detect()` را با دامنه محدود به یافته‌های تعمیرشده اجرا می‌کند؛ اگر یافته همچنان وجود داشته باشد، doctor به‌جای کامل تلقی‌کردن تغییر، هشدار تعمیر گزارش می‌کند.

هر یافته شامل موارد زیر است:

| فیلد             | هدف                                                |
| ----------------- | ------------------------------------------------------ |
| `checkId`         | شناسه پایدار برای فیلترهای ردکردن/فقط و فهرست‌های مجاز CI.     |
| `severity`        | `info`، `warning` یا `error`.                         |
| `message`         | شرح قابل‌فهم مشکل برای انسان.                      |
| `path`            | مسیر پیکربندی، فایل یا مسیر منطقی، در صورت وجود.          |
| `line` / `column` | محل منبع، در صورت وجود.                        |
| `ocPath`          | نشانی دقیق `oc://`، وقتی بررسی بتواند به یکی اشاره کند. |
| `fixHint`         | اقدام پیشنهادی اپراتور یا خلاصه تعمیر.           |

بررسی‌های نوسازی‌شده doctor در هسته، به مشارکت مرتب‌شده doctor که مالک رفتار انسانی `doctor` / `doctor --fix` آن‌ها است متصل می‌مانند. رجیستری مشترک سلامت ساخت‌یافته نقطه توسعه است: بررسی‌های همراه و مبتنی بر Plugin، پس از بررسی‌های doctor هسته اجرا می‌شوند؛ مشروط بر اینکه بسته مالک آن‌ها، بررسی‌ها را در مسیر فعال فرمان ثبت کند. `openclaw/plugin-sdk/health` همین قرارداد را برای نویسندگان Plugin ارائه می‌کند.

## انتخاب بررسی‌ها

```bash
openclaw doctor --lint --only core/doctor/gateway-config --json
openclaw doctor --lint --skip core/doctor/skills-readiness
openclaw doctor --lint --all --skip core/doctor/session-locks
```

`--only` و `--skip` شناسه‌های کامل بررسی را می‌پذیرند و می‌توان آن‌ها را تکرار کرد. اگر شناسه `--only` ثبت نشده باشد، هیچ بررسی‌ای برای آن شناسه اجرا نمی‌شود؛ برای تأیید اینکه یک دروازه متمرکز بررسی‌های مورد انتظار را انتخاب می‌کند، از `checksRun`/`checksSkipped` در خروجی استفاده کنید.

## حالت پس از ارتقا

`openclaw doctor --post-upgrade` کاوش‌های سازگاری Plugin را برای اجرای زنجیره‌ای پس از ساخت یا ارتقا اجرا می‌کند. یافته‌ها به stdout می‌روند؛ اگر هر یافته‌ای دارای `level: "error"` باشد، کد خروج 1 است. برای دریافت یک پوشش ماشین‌خوان (`{ probesRun, findings }`) که برای CI، Skill انجمن `fork-upgrade` و دیگر ابزارهای آزمون دود پس از ارتقا مناسب است، `--json` را اضافه کنید. اگر نمایه Plugin نصب‌شده وجود نداشته باشد یا بدشکل باشد، حالت JSON همچنان پوشش را با یک یافته خطای `plugin.index_unavailable` منتشر می‌کند.

راه‌اندازی ایمیج کانتینر، استثنای روند معمول «اجرای doctor پس از
به‌روزرسانی» است. وقتی `openclaw gateway run` روی نسخه جدیدی از OpenClaw آغاز می‌شود،
پیش از اعلام آمادگی، تعمیرات امن وضعیت و Plugin را اجرا می‌کند. اگر تعمیر نتواند
با ایمنی کامل شود، راه‌اندازی خاتمه می‌یابد و به شما می‌گوید همان ایمیج را یک‌بار با
`openclaw doctor --fix` در برابر همان وضعیت/پیکربندی متصل‌شده اجرا کنید و سپس
کانتینر را به‌صورت عادی دوباره راه‌اندازی کنید.

## Compaction پایگاه داده SQLite وضعیت مشترک

`openclaw doctor --state-sqlite compact` نگه‌داری آفلاین صریح برای
پایگاه داده متعارف وضعیت مشترک در
`<state-dir>/state/openclaw.sqlite` است. این فرمان مسیر دلخواه پایگاه داده را
نمی‌پذیرد، هرگز از سوی عملیات عادی Gateway فراخوانی نمی‌شود و بخشی از
`openclaw doctor --fix` نیست. فرمان همان قفل مالکیت وضعیتِ راه‌اندازی Gateway را
به‌دست می‌آورد و آن را در سراسر اعتبارسنجی، checkpoint، اجرای `VACUUM` و
بررسی‌های نهایی یکپارچگی نگه می‌دارد. تا زمانی که یک Gateway یا فرمان نگه‌داری
SQLite دیگری مالک آن قفل باشد، از اجرا خودداری می‌کند. وقتی
`OPENCLAW_ALLOW_MULTI_GATEWAY=1` نمونه یکتای Gateway برای هر پیکربندی را نادیده می‌گیرد،
قفل وضعیت همچنان فعال می‌ماند؛ بنابراین برای اینکه نگه‌داری بتواند آن را تشخیص دهد،
پوسته اپراتور نیازی به به‌ارث‌بردن محیط سرویس Gateway ندارد.

ابتدا Gateway را متوقف و یک پشتیبان تأییدشده ایجاد کنید:

```bash
openclaw gateway stop
openclaw backup create --verify
openclaw doctor --state-sqlite compact --json
openclaw gateway start
```

این فرمان:

1. به یک فایل عادی در مسیر متعارف وضعیت مشترک نیاز دارد. نبود
   پایگاه داده به‌صورت `skipped` گزارش می‌شود و با موفقیت خاتمه می‌یابد.
2. نسخه طرح‌واره پشتیبانی‌شده فعلی و
   `schema_meta.role = "global"` را پیش از checkpoint یا تغییر فایل اعتبارسنجی می‌کند.
3. به یک `wal_checkpoint(TRUNCATE)` غیرفعال نیاز دارد. اگر checkpoint مشغول است، همه فرایندهای باقی‌مانده OpenClaw را
   متوقف و دوباره تلاش کنید.
4. مقدار `auto_vacuum` را روی `INCREMENTAL` تنظیم می‌کند، یک `VACUUM` کامل را اجرا می‌کند و دوباره
   checkpoint می‌گیرد.
5. `quick_check`، `integrity_check` و `foreign_key_check` را اجرا می‌کند، سپس
   مجوزهای فقط‌مالک را دوباره روی پایگاه داده و فایل‌های جانبی SQLite اعمال می‌کند.

خروجی JSON اندازه‌های پایگاه داده و WAL، صفحه‌های freelist، اندازه صفحه و
مقدار `auto_vacuum` را پیش و پس از Compaction، همراه با بایت‌های بازیابی‌شده و
نتایج `quick_check` و `integrity_check` گزارش می‌کند. `foreign_key_check`
به‌صورت fail-closed اعمال می‌شود و فیلد موفقیت جداگانه‌ای ندارد. SQLite مقدار `auto_vacuum` را
برای none به‌شکل `0`، برای full به‌شکل `1` و برای incremental به‌شکل `2` گزارش می‌کند.

وقتی طرح‌واره قدیمی‌تر یا جدیدتر از نسخه در حال اجرای OpenClaw باشد، یا به
پایگاه داده یک عامل تعلق داشته باشد، Compaction بدون تغییر دادن داده‌ها شکست می‌خورد. برای طرح‌واره قدیمی‌تر وضعیت مشترک، ابتدا
`openclaw doctor --fix` را اجرا کنید. برای طرح‌واره جدیدتر، یک پشتیبان
سازگار را بازیابی کنید یا OpenClaw را ارتقا دهید.

## مهاجرت SQLite نشست

OpenClaw هنگام راه‌اندازی Gateway و نیز هنگام
`openclaw doctor --fix`، ردیف‌های نشست قدیمی و تاریخچه رونوشت را به‌طور خودکار به پایگاه داده
SQLite هر عامل وارد می‌کند. `openclaw doctor --session-sqlite <mode>`
ابزار هدفمند بازرسی و اعتبارسنجی آن مهاجرت است. ردیف‌های فعلی نشست زمان اجرا در
`~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite` قرار دارند. فایل‌های قدیمی
`sessions.json` منابع مهاجرت هستند. فایل‌های فعال JSONL رونوشت
پس از واردسازی موفق، وارد و از پوشه نشست‌های فعال بایگانی می‌شوند؛ فایل‌های JSONL
لایه بایگانی، مصنوعات پشتیبانی باقی می‌مانند، نه مسیرهای جایگزین زمان اجرا.

حالت‌ها:

| حالت       | رفتار                                                                                                               |
| ---------- | ---------------------------------------------------------------------------------------------------------------------- |
| `inspect`  | شمار ردیف‌های قدیمی و SQLite، به‌همراه فایل‌های JSONL بدون ارجاع را بدون واردسازی می‌خواند.                                       |
| `dry-run`  | ورودی‌های قدیمی و فایل‌های JSONL رونوشت را تجزیه می‌کند، ردیف‌های قابل‌واردسازی را می‌شمارد و مشکلات را بدون نوشتن ردیف‌های SQLite گزارش می‌کند. |
| `import`   | ورودی‌های قدیمی و رویدادهای رونوشت را برای هدف‌های انتخاب‌شده به SQLite وارد می‌کند.                                      |
| `validate` | منابع قدیمی انتخاب‌شده را با ردیف‌های SQLite و شمار رویدادهای رونوشت مقایسه می‌کند.                                   |
| `compact`  | از پایگاه‌های داده SQLite عامل انتخاب‌شده checkpoint می‌گیرد و آن‌ها را VACUUM می‌کند تا پس از حذف‌های بزرگ یا پاک‌سازی بایگانی، صفحه‌های آزاد بازیابی شوند.    |
| `recover`  | آخرین اجرای ناموفق مهاجرت را بازیابی می‌کند، هدف‌های آن را اعتبارسنجی می‌کند و یک گزارش پاک‌سازی‌شده برای مشکل GitHub آماده می‌کند.            |
| `restore`  | مصنوعات بایگانی‌شده رونوشت را از مانیفست‌های ثبت‌شده مهاجرت، بدون حذف داده‌های SQLite بازیابی می‌کند.                  |

انتخاب‌گرها:

- پیش‌فرض: مخزن عامل پیش‌فرض پیکربندی‌شده، هنگامی که فایل قدیمی آن مخزن وجود دارد.
- `--session-sqlite-agent <id>`: یک عامل پیکربندی‌شده.
- `--session-sqlite-all-agents`: مخزن‌های عامل پیکربندی‌شده به‌همراه مخزن‌های عامل کشف‌شده.
- `--session-sqlite-store <path>`: یک مسیر صریح قدیمی `sessions.json`.

توالی بازرسی دستی:

```bash
openclaw doctor --session-sqlite inspect --session-sqlite-all-agents
openclaw doctor --session-sqlite dry-run --session-sqlite-all-agents --json
openclaw doctor --session-sqlite import --session-sqlite-all-agents
openclaw doctor --session-sqlite validate --session-sqlite-all-agents --json
openclaw doctor --session-sqlite compact --session-sqlite-all-agents
openclaw doctor --session-sqlite recover --github-issue
```

پیش از اجرای `import` روی نصبی با تاریخچه مهم، از پوشه وضعیت OpenClaw
پشتیبان بگیرید. وقتی یک ورودی قدیمی انتخاب‌شده در SQLite وجود نداشته باشد،
شناسه نشست متفاوت باشد یا شمار رویدادهای رونوشت تفاوت داشته باشد، `validate` با کد غیرصفر خاتمه می‌یابد.
هنگام استفاده از `--session-sqlite-store <path>`، بررسی کنید گزارش شامل
شمار هدف مورد انتظار باشد؛ مسیر صریح مخزن که وجود ندارد هیچ هدفی را انتخاب نمی‌کند.

حذف‌های SQLite ابتدا صفحه‌های داخل پایگاه داده را بازیابی می‌کنند؛ این حذف‌ها لزوماً
فایل پایگاه داده را بلافاصله کوچک نمی‌کنند. پس از حذف یا بایگانی رونوشت‌های بزرگ،
`openclaw doctor --session-sqlite compact --session-sqlite-all-agents` را اجرا کنید تا فایل‌های WAL
checkpoint شوند، `VACUUM` اجرا شود و اندازه‌های پایگاه داده و WAL پیش و پس از عملیات
گزارش شوند. Compaction به یک فایل عادی با طرح‌واره فعلی عامل،
فراداده مالک پایدار عامل انتخاب‌شده و نداشتن دسته فایل باز در فرایند doctor
نیاز دارد. حالت‌های مخرب `import`، `compact`، `recover` و `restore`
در سراسر عملیات خود همان قفل مالکیت وضعیتِ راه‌اندازی Gateway را نگه می‌دارند؛
`inspect`، `dry-run` و `validate` فقط‌خواندنی باقی می‌مانند و آن را به‌دست نمی‌آورند. ابتدا
Gateway را متوقف کنید. حالت‌های مخرب به‌جای رقابت با نوشتن‌های زنده یا
فرمان نگه‌داری دیگری، شکست می‌خورند. هدف مخرب `--session-sqlite-store`
باید داخل پوشه وضعیت فعال باشد؛ پیش از نگه‌داری نصب دیگری، `OPENCLAW_STATE_DIR` را روی
پوشه وضعیت مالک آن مخزن تنظیم کنید.
هدف‌های hard-linked موجود رد می‌شوند، زیرا مسیری دیگر می‌تواند همان inode پایگاه داده را
بیرون از پوشه وضعیت قفل‌شده به‌اشتراک بگذارد. همان بررسی‌های مالکیت،
فایل‌های جانبی WAL، حافظه مشترک و rollback-journal مربوط به SQLite را نیز پوشش می‌دهند.

هر واردسازی پیش از انتقال مصنوعات رونوشت
به بایگانی، یک مانیفست در `~/.openclaw/session-sqlite-migration-runs/`
می‌نویسد. اگر راه‌اندازی پس از انتقال مصنوعات، مهاجرت ناموفق SQLite نشست را گزارش کرد،
بازیابی را اجرا کنید:

```bash
openclaw doctor --session-sqlite recover --github-issue
```

بازیابی آخرین مانیفست ناموفق مهاجرت را انتخاب می‌کند، فقط مصنوعات بایگانی‌شده
همان مانیفست را بازیابی می‌کند، هدف‌های متأثر را اعتبارسنجی می‌کند، گزارش‌های
پاک‌سازی‌شده `.failure.md` و `.failure.json` را تازه می‌کند و بدنه یک مشکل GitHub را
آماده می‌کند که از محتوای رونوشت، محیط خام، اسرار و پیکربندی
نامحدود اجتناب می‌کند. وقتی مانیفست ناموفق مهاجرت وجود ندارد اما پایگاه داده SQLite
عامل انتخاب‌شده خراب است، پایگاه داده نیست، یا فایل‌های جانبی journal بدون پایگاه داده اصلی
دارد، بازیابی مجموعه کامل فایل‌ها را به یک پوشه موقت بازرسی
کپی می‌کند. SQLite می‌تواند پیش از اجرای `quick_check`، `integrity_check` و `foreign_key_check`،
یک hot journal معتبر را در آن کپی یک‌بارمصرف rollback کند، درحالی‌که
فایل‌های اصلی جرم‌شناسی دست‌نخورده باقی می‌مانند. بررسی‌های ناموفق یکپارچگی یا فایل‌های جانبی
یتیم، فایل‌های DB، WAL، SHM و rollback-journal را با تغییر نام
کل مجموعه کشف‌شده با یک پسوند `.corrupt-<timestamp>` حفظ می‌کنند. اگر تغییر نام
با خطا مواجه شود، فایل‌های ازپیش‌منتقل‌شده پیش از گزارش شکست بازگردانده می‌شوند تا
یک مجموعه فایل قابل‌بازیابی، بی‌سروصدا تکه‌تکه نشود. پیش از بازیابی Gateway را متوقف کنید؛
کپی یا تغییر نام مجموعه فایل SQLite که فعالانه در حال تغییر است ناامن است و
در سیستم‌عامل‌های مختلف رفتار متفاوتی دارد. با `--github-issue --yes`، doctor
از GitHub CLI برای ایجاد مشکل در `openclaw/openclaw` استفاده می‌کند؛ بدون تأیید،
گزارش پشتیبانی محلی را می‌نویسد و URL ازپیش‌پرشده مشکل را چاپ می‌کند.

`restore` عملیات بازگردانی سطح پایین‌تر باقی می‌ماند. این عملیات از رکوردهای
`sourcePath -> archivePath` مانیفست استفاده می‌کند، مصنوعات بایگانی‌شده را فقط وقتی به مسیر قبلی
بازمی‌گرداند که مسیر اصلی وجود نداشته باشد، وقتی هر دو مسیر وجود دارند تعارض‌ها را
گزارش می‌کند و پایگاه داده SQLite را در جای خود باقی می‌گذارد.

### تنزل نسخه پس از مهاجرت SQLite نشست

پیش از آغاز یک نسخه قدیمی‌تر و فایل‌محور OpenClaw، مصنوعات بایگانی‌شده
رونوشت قدیمی را بازیابی کنید:

```bash
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

نسخه‌های قدیمی‌تر مدخل‌های `sessions.json` و مسیرهای `sessionFile` ثبت‌شده
در آن مدخل‌ها را می‌خوانند. پس از مهاجرت به SQLite، واردسازی‌های موفق رونوشت‌های داغ JSONL
را به `session-sqlite-import-archive/` منتقل می‌کنند؛ بنابراین زمان اجرای قدیمی‌تر تا زمانی که بازیابی، مصنوعات ثبت‌شده در مانیفست را به
مسیرهای اصلی‌شان بازنگرداند، نمی‌تواند آن تاریخچه را ببیند.

بازیابی داده‌های SQLite را حذف نمی‌کند. نشست‌هایی که پس از گذار به SQLite ایجاد شده‌اند
فقط در SQLite وجود دارند و در زمان اجرای قدیمی‌تر ظاهر نمی‌شوند. اگر بعداً
دوباره ارتقا دادید، توالی عادی اعتبارسنجی مهاجرتِ بالا را اجرا کنید تا OpenClaw بتواند
پیش از واردسازی، مصنوعات قدیمیِ بازیابی‌شده را با ردیف‌های SQLite مقایسه کند.

## نکات

- در حالت Nix ‏(`OPENCLAW_NIX_MODE=1`)، بررسی‌های فقط‌خواندنی doctor همچنان کار می‌کنند، اما `doctor --fix`، `doctor --repair`، `doctor --yes` و `doctor --generate-gateway-token` غیرفعال‌اند، زیرا `openclaw.json` تغییرناپذیر است. در عوض، منبع Nix این نصب را ویرایش کنید؛ برای nix-openclaw، از [شروع سریع](https://github.com/openclaw/nix-openclaw#quick-start) مبتنی بر عامل استفاده کنید.
- درخواست‌های تعاملی (اصلاحات keychain/OAuth و غیره) فقط زمانی اجرا می‌شوند که stdin یک TTY باشد و `--non-interactive` تنظیم **نشده** باشد. اجراهای بدون رابط (cron، Telegram، بدون ترمینال) درخواست‌ها را رد می‌کنند.
- اجراهای غیرتعاملی `doctor` بارگذاری زودهنگام Plugin را رد می‌کنند تا بررسی‌های سلامت بدون رابط سریع بمانند. نشست‌های تعاملی همچنان سطوح Plugin موردنیاز جریان قدیمی سلامت/ترمیم را بارگذاری می‌کنند.
- `--lint` از `--non-interactive` سخت‌گیرانه‌تر است: همیشه فقط‌خواندنی است، هرگز درخواست تعاملی نمایش نمی‌دهد و هرگز مهاجرت‌های ایمن را اعمال نمی‌کند. وقتی می‌خواهید doctor تغییراتی ایجاد کند، از `doctor --fix` یا `doctor --repair` استفاده کنید.
- doctor هنگام بررسی اسرار، به‌طور پیش‌فرض SecretRefهای `exec` را اجرا نمی‌کند. فقط زمانی که عمداً می‌خواهید doctor آن حل‌کننده‌های اسرار پیکربندی‌شده را اجرا کند، از `--allow-exec` (با یا بدون `--lint`) استفاده کنید.
- هر نوشتن پیکربندی (از جمله ترمیم `--fix`) نسخهٔ پشتیبان را به `~/.openclaw/openclaw.json.bak` می‌چرخاند (با حلقهٔ شماره‌گذاری‌شدهٔ `.bak.1`..`.bak.4`). ‏`--fix` همچنین کلیدهای ناشناختهٔ پیکربندی را که اعتبارسنجی شِما گزارش کرده حذف می‌کند و هر حذف را فهرست می‌کند؛ هنگام اجرای به‌روزرسانی، این کار را انجام نمی‌دهد تا وضعیت ارتقای نیمه‌نوشته پیش از پایان مهاجرتش حذف نشود.
- وقتی سرپرست دیگری مالک چرخهٔ عمر Gateway است، `OPENCLAW_SERVICE_REPAIR_POLICY=external` را تنظیم کنید. doctor همچنان سلامت Gateway/سرویس را گزارش می‌کند و ترمیم‌های غیروسرویسی را اعمال می‌کند، اما نصب/شروع/راه‌اندازی مجدد/راه‌اندازی اولیهٔ سرویس و پاک‌سازی سرویس قدیمی را رد می‌کند.
- در Linux، doctor واحدهای اضافی و غیرفعالِ شبیه Gateway در systemd را نادیده می‌گیرد و هنگام ترمیم، فرادادهٔ فرمان/نقطهٔ ورود سرویس Gateway در حال اجرای systemd را بازنویسی نمی‌کند. ابتدا سرویس را متوقف کنید، یا برای جایگزینی راه‌انداز فعال از `openclaw gateway install --force` استفاده کنید.
- `doctor --fix --non-interactive` تعریف‌های مفقود یا کهنهٔ سرویس Gateway را گزارش می‌کند، اما خارج از حالت ترمیم به‌روزرسانی آن‌ها را نصب یا بازنویسی نمی‌کند. برای سرویس مفقود، `openclaw gateway install` را اجرا کنید، یا برای جایگزینی راه‌انداز از `openclaw gateway install --force` استفاده کنید.
- بررسی‌های یکپارچگی وضعیت، فایل‌های رونوشت یتیم را در پوشهٔ نشست‌ها تشخیص می‌دهند. بایگانی آن‌ها به‌صورت `.deleted.<timestamp>` به تأیید تعاملی نیاز دارد؛ `--fix`، `--yes` و اجراهای بدون رابط آن‌ها را در جای خود باقی می‌گذارند.
- doctor برای یافتن شکل‌های قدیمی کارهای cron، ‏`~/.openclaw/cron/jobs.json` (یا `cron.store`) را پویش و پیش از واردکردن ردیف‌های متعارف به SQLite آن‌ها را بازنویسی می‌کند.
- doctor کارهای cron دارای بازنویسی صریح `payload.model` را، شامل شمارش فضای نام ارائه‌دهنده و عدم تطابق با `agents.defaults.model`، گزارش می‌کند تا کارهای زمان‌بندی‌شده‌ای که مدل پیش‌فرض را به ارث نمی‌برند هنگام بررسی‌های احراز هویت یا صورت‌حساب قابل مشاهده باشند.
- doctor کارهای cron را که همچنان در حال اجرا علامت‌گذاری شده‌اند (`state.runningAtMs`) گزارش می‌کند؛ این وضعیت می‌تواند باعث شود `openclaw cron list` آن‌ها را به‌صورت `running` نشان دهد. این بررسی فقط‌خواندنی است: اگر در حال حاضر هیچ Gateway کار علامت‌گذاری‌شده‌ای را اجرا نمی‌کند، راه‌اندازی بعدی سرویس cron اجرای قطع‌شده را ثبت و نشانگر را پاک می‌کند.
- در Linux، وقتی crontab کاربر همچنان `~/.openclaw/bin/ensure-whatsapp.sh` قدیمی و نگهداری‌نشده را اجرا می‌کند، doctor هشدار می‌دهد؛ این مورد زمانی که cron فاقد محیط گذرگاه کاربر systemd باشد ممکن است `Gateway inactive` را اشتباه گزارش کند.
- وقتی WhatsApp فعال است، doctor وجود حلقهٔ رویداد تنزل‌یافتهٔ Gateway را در حالی بررسی می‌کند که کلاینت‌های محلی `openclaw-tui` هنوز در حال اجرا هستند. `doctor --fix` فقط کلاینت‌های محلی TUI تأییدشده را متوقف می‌کند تا پاسخ‌های WhatsApp پشت حلقه‌های تازه‌سازی کهنهٔ TUI در صف نمانند.
- doctor ارجاع‌های مدل قدیمی `codex/*` و `openai-codex/*` را در مدل‌های اصلی، جایگزین‌ها، فهرست‌های مجاز مدل، مدل‌های تولید تصویر/ویدئو، بازنویسی‌های Heartbeat/زیرعامل/Compaction، قلاب‌ها، بازنویسی‌های مدل کانال، بارهای cron و پین‌های کهنهٔ مسیر نشست/رونوشت، به ارجاع‌های متعارف `openai/*` بازنویسی می‌کند. `--fix` همچنین در صورت ایمن‌بودن، پیکربندی قدیمی `models.providers.codex` و `models.providers.openai-codex` را ادغام می‌کند، پروفایل‌های احراز هویت قدیمی `openai-codex:*` و مدخل‌های `auth.order.openai-codex` را به `openai:*` مهاجرت می‌دهد، قصد Codex را به مدخل‌های `agentRuntime.id: "codex"` محدود به ارائه‌دهنده/مدل منتقل می‌کند، پین‌های کهنهٔ زمان اجرای کل عامل/نشست را حذف می‌کند و ارجاع‌های ترمیم‌شدهٔ عامل OpenAI را به‌جای احراز هویت مستقیم کلید API در OpenAI، روی مسیریابی احراز هویت Codex نگه می‌دارد.
- doctor فهرست‌های غیرخالی `auth.order.<provider>` را گزارش می‌کند که همهٔ پروفایل‌های ارجاع‌شدهٔ آن‌ها از بین رفته‌اند، اما اعتبارنامه‌های سازگارِ ذخیره‌شده وجود دارند. `doctor --fix` فقط همان بازنویسی‌های کهنه را حذف می‌کند و انتخاب خودکار اعتبارنامه برای هر عامل را بازمی‌گرداند؛ ترتیب‌های صریحاً خالی، فهرست‌های بخشی‌فعال و ترتیب‌های بدون اعتبارنامهٔ ذخیره‌شدهٔ سازگار بدون تغییر می‌مانند. اگر ذخیره‌گاه فعال احراز هویت SQLite خواندنی نباشد یا بدشکل باشد، doctor توضیح می‌دهد چرا این ترمیم را رد کرده است. اگر حالت بارگذاری مجدد پیکربندیِ Gateway در حال اجرا، نوشتن را خودکار اعمال نمی‌کند، پیش از بررسی دوبارهٔ وضعیت احراز هویت آن را راه‌اندازی مجدد کنید.
- doctor وضعیت قدیمی مرحله‌بندی وابستگی Plugin را از نسخه‌های قدیمی‌تر OpenClaw پاک می‌کند و بستهٔ میزبان `openclaw` را برای Pluginهای npm مدیریت‌شده‌ای که آن را به‌عنوان وابستگی همتا اعلام می‌کنند، دوباره پیوند می‌دهد. همچنین Pluginهای قابل‌دانلودِ مفقودی را که پیکربندی به آن‌ها ارجاع می‌دهد (`plugins.entries`، کانال‌های پیکربندی‌شده، تنظیمات ارائه‌دهنده/جست‌وجوی پیکربندی‌شده، زمان‌های اجرای عامل پیکربندی‌شده) ترمیم می‌کند. هنگام به‌روزرسانی بسته‌ها، doctor تا پایان جابه‌جایی بسته، ترمیم Plugin توسط مدیر بسته را رد می‌کند؛ اگر پس از آن یک Plugin پیکربندی‌شده همچنان به بازیابی نیاز داشت، `openclaw doctor --fix` را دوباره اجرا کنید. اگر دانلود ناموفق باشد، doctor خطای نصب را گزارش می‌کند و مدخل Plugin پیکربندی‌شده را برای تلاش ترمیم بعدی حفظ می‌کند.
- doctor با حذف شناسه‌های Plugin مفقود از `plugins.allow`/`plugins.deny`/`plugins.entries`، همراه با پیکربندی معلق متناظر کانال، اهداف Heartbeat و بازنویسی‌های مدل کانال، پیکربندی کهنهٔ Plugin را ترمیم می‌کند؛ مشروط بر اینکه کشف Plugin سالم باشد.
- doctor پیکربندی نامعتبر Plugin را با غیرفعال‌کردن مدخل `plugins.entries.<id>` تحت‌تأثیر و حذف بار نامعتبر `config` آن قرنطینه می‌کند. راه‌اندازی Gateway از قبل فقط همان Plugin معیوب را رد می‌کند تا سایر Pluginها و کانال‌ها همچنان اجرا شوند.
- doctor ‏`plugins.entries.codex.config.codexDynamicToolsProfile` بازنشسته‌شده را حذف می‌کند؛ app-server مربوط به Codex همیشه ابزارهای فضای کاری بومی Codex را بومی نگه می‌دارد.
- doctor پیکربندی تخت و قدیمی Talk ‏(`talk.voiceId`، `talk.modelId` و موارد مشابه) را به‌طور خودکار به `talk.provider` + `talk.providers.<provider>` مهاجرت می‌دهد. اجراهای تکراری `doctor --fix` دیگر زمانی که تنها تفاوت، ترتیب کلیدهای شیء است، عادی‌سازی Talk را گزارش/اعمال نمی‌کنند.
- doctor یک بررسی آمادگی جست‌وجوی حافظه را شامل می‌شود و در صورت مفقودبودن اعتبارنامه‌های تعبیه‌سازی می‌تواند `openclaw configure --section model` را توصیه کند.
- doctor وقتی هیچ مالک فرمانی پیکربندی نشده باشد هشدار می‌دهد. مالک فرمان، حساب اپراتور انسانی است که اجازه دارد فرمان‌های مختص مالک را اجرا و اقدامات خطرناک را تأیید کند. جفت‌سازی DM فقط به شخص اجازه می‌دهد با ربات صحبت کند؛ اگر فرستنده‌ای را پیش از وجود راه‌اندازی اولیهٔ نخستین مالک تأیید کرده‌اید، `commands.ownerAllowFrom` را صریحاً تنظیم کنید.
- doctor زمانی یک یادداشت اطلاعاتی گزارش می‌کند که عامل‌های حالت Codex پیکربندی شده باشند و دارایی‌های شخصی Codex CLI در خانهٔ Codex اپراتور وجود داشته باشند. اجراهای محلی app-server مربوط به Codex از خانه‌های مجزای هر عامل استفاده می‌کنند؛ در صورت نیاز ابتدا Plugin مربوط به Codex را نصب کنید، سپس برای فهرست‌برداری از دارایی‌هایی که باید آگاهانه ارتقا یابند، از `openclaw migrate plan codex` استفاده کنید.
- doctor وقتی Skills مجاز برای عامل پیش‌فرض در محیط زمان اجرای فعلی در دسترس نباشند (فقدان فایل‌های اجرایی، متغیرهای محیطی، پیکربندی یا الزامات سیستم‌عامل)، هشدار می‌دهد. `doctor --fix` می‌تواند آن Skills در دسترس‌نبودنی را با `skills.entries.<skill>.enabled=false` غیرفعال کند؛ اگر می‌خواهید Skill فعال بماند، در عوض الزام مفقود را نصب/پیکربندی کنید.
- اگر حالت sandbox فعال باشد اما Docker در دسترس نباشد، doctor هشداری برجسته همراه با راه‌حل (`install Docker` یا `openclaw config set agents.defaults.sandbox.mode off`) گزارش می‌کند.
- اگر فایل‌های قدیمی رجیستری sandbox یا پوشه‌های shard وجود داشته باشند (`~/.openclaw/sandbox/containers.json`، `~/.openclaw/sandbox/browsers.json`، `~/.openclaw/sandbox/containers/` یا `~/.openclaw/sandbox/browsers/`) doctor آن‌ها را گزارش می‌کند؛ `--fix` مدخل‌های معتبر را به SQLite مهاجرت می‌دهد و فایل‌های قدیمی نامعتبر را قرنطینه می‌کند.
- اگر `gateway.auth.token`/`gateway.auth.password` تحت مدیریت SecretRef باشند و در مسیر فرمان فعلی در دسترس نباشند، doctor هشداری فقط‌خواندنی گزارش می‌کند و اعتبارنامه‌های جایگزینِ متن ساده نمی‌نویسد. برای SecretRefهای مبتنی بر exec، doctor اجرا را رد می‌کند، مگر اینکه `--allow-exec` وجود داشته باشد.
- اگر بازرسی SecretRef کانال در مسیر اصلاح ناموفق باشد، doctor به‌جای خروج زودهنگام ادامه می‌دهد و هشدار گزارش می‌کند.
- پس از مهاجرت پوشهٔ وضعیت، doctor زمانی هشدار می‌دهد که حساب‌های پیش‌فرض فعال Telegram یا Discord به جایگزین محیطی وابسته باشند و `TELEGRAM_BOT_TOKEN` یا `DISCORD_BOT_TOKEN` برای فرایند doctor در دسترس نباشد.
- تفکیک خودکار نام کاربری `allowFrom` در Telegram ‏(`doctor --fix`) به توکن قابل‌تفکیک Telegram در مسیر فرمان فعلی نیاز دارد. اگر بازرسی توکن در دسترس نباشد، doctor هشدار گزارش می‌کند و تفکیک خودکار را برای آن نوبت رد می‌کند.

## macOS: بازنویسی‌های محیطی `launchctl`

اگر قبلاً `launchctl setenv OPENCLAW_GATEWAY_TOKEN ...` (یا `...PASSWORD`) را اجرا کرده‌اید، آن مقدار فایل پیکربندی شما را بازنویسی می‌کند و ممکن است خطاهای مداوم «غیرمجاز» ایجاد کند.

```bash
launchctl getenv OPENCLAW_GATEWAY_TOKEN
launchctl getenv OPENCLAW_GATEWAY_PASSWORD

launchctl unsetenv OPENCLAW_GATEWAY_TOKEN
launchctl unsetenv OPENCLAW_GATEWAY_PASSWORD
```
