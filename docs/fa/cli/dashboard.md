# `openclaw dashboard`

رابط کاربری کنترل را با استفاده از احراز هویت فعلی خود باز کنید.

```bash
openclaw dashboard
openclaw dashboard --no-open
openclaw dashboard --yes
```

- `--no-open`: نشانی URL را چاپ می‌کند، اما مرورگر را اجرا نمی‌کند.
- `--yes`: در صورت نیاز، Gateway را بدون درخواست تأیید راه‌اندازی/نصب می‌کند.

نکات:

- در صورت امکان، SecretRefهای پیکربندی‌شدهٔ `gateway.auth.token` را رفع می‌کند.
- از `gateway.tls.enabled` پیروی می‌کند: Gatewayهایی که TLS در آن‌ها فعال است، نشانی‌های URL رابط کاربری کنترل با `https://` را چاپ/باز می‌کنند و از طریق `wss://` متصل می‌شوند.
- برای اتصال `lan` یا اتصال `custom` با نویسهٔ عام، اجراها روی همان میزبان همیشه از loopback استفاده می‌کنند، زیرا نویسهٔ عام مقصد معتبری برای مرورگر نیست. اتصال‌های متن سادهٔ `tailnet` و `custom` نیز از `127.0.0.1` استفاده می‌کنند تا مرورگر زمینه‌ای امن داشته باشد؛ میزبان‌های مشخصی که TLS در آن‌ها فعال است، نشانی پیکربندی‌شده را حفظ می‌کنند تا نام‌های گواهی مطابقت داشته باشند.
- پیش از ارائهٔ یک نشانی URL احرازشدهٔ loopback برای اتصال به رابطی مشخص، فرمان رابط پیکربندی‌شده را بررسی می‌کند و تأیید می‌کند که آن رابط و `127.0.0.1` متعلق به یک فرایند Gateway هستند. اگر مالکیت شنونده مبهم باشد، عملیات به‌صورت بسته و همراه با راهنمای وضعیت ناموفق می‌شود.
- برای توکن‌های مدیریت‌شده با SecretRef، چه رفع‌شده و چه رفع‌نشده، نشانی URL چاپ‌شده/رونوشت‌شده/بازشده هرگز حاوی توکن نیست؛ بنابراین اسرار خارجی به خروجی پایانه، تاریخچهٔ کلیپ‌بورد یا آرگومان‌های اجرای مرورگر نشت نمی‌کنند.
- اگر `gateway.auth.token` با SecretRef مدیریت شود اما رفع‌نشده باشد، فرمان به‌جای جای‌نگهدار نامعتبر توکن، یک نشانی URL بدون توکن و راهنمای رفع مشکل را چاپ می‌کند.
- اگر ارائه از طریق کلیپ‌بورد/مرورگر برای یک نشانی URL احرازشده با توکن ناموفق باشد، فرمان بدون چاپ مقدار توکن، یک راهنمای امن برای احراز هویت دستی ثبت می‌کند که `OPENCLAW_GATEWAY_TOKEN`، `gateway.auth.token` و کلید قطعهٔ نشانی URL یعنی `token` را نام می‌برد.
