# پشتیبانی از تصویر و رسانه (2025-12-05)

کانال WhatsApp از طریق **Baileys Web** اجرا می‌شود. این سند قواعد فعلی مدیریت رسانه را برای ارسال، Gateway، و پاسخ‌های عامل ثبت می‌کند.

## اهداف

- ارسال رسانه با زیرنویس اختیاری از طریق `openclaw message send --media`.
- اجازه دادن به پاسخ‌های خودکار از صندوق ورودی وب برای گنجاندن رسانه در کنار متن.
- نگه داشتن محدودیت‌های هر نوع به شکلی معقول و قابل پیش‌بینی.

## سطح CLI

- `openclaw message send --media <path-or-url> [--message <caption>]`
  - `--media` اختیاری است؛ زیرنویس برای ارسال‌های فقط رسانه‌ای می‌تواند خالی باشد.
  - `--dry-run` بار نهایی‌شده را چاپ می‌کند؛ `--json` خروجی `{ channel, to, messageId, mediaUrl, caption }` تولید می‌کند.

## رفتار کانال WhatsApp Web

- ورودی: مسیر فایل محلی **یا** URL از نوع HTTP(S).
- جریان: بارگذاری در یک بافر، تشخیص نوع رسانه، و ساخت بار درست:
  - **تصاویر:** تغییر اندازه و فشرده‌سازی دوباره به JPEG (حداکثر ضلع 2048px) با هدف‌گذاری `channels.whatsapp.mediaMaxMb` (پیش‌فرض: 50 MB).
  - **صدا/صوت/ویدیو:** عبور مستقیم تا 16 MB؛ صدا به‌عنوان یادداشت صوتی (`ptt: true`) ارسال می‌شود.
  - **اسناد:** هر چیز دیگر، تا 100 MB، با حفظ نام فایل در صورت موجود بودن.
- پخش به سبک GIF در WhatsApp: یک MP4 با `gifPlayback: true` ارسال کنید (CLI: `--gif-playback`) تا کلاینت‌های موبایل آن را به‌صورت درون‌خطی تکرار کنند.
- تشخیص MIME ابتدا بایت‌های جادویی، سپس سرآیندها، و سپس پسوند فایل را ترجیح می‌دهد.
- زیرنویس از `--message` یا `reply.text` می‌آید؛ زیرنویس خالی مجاز است.
- گزارش‌گیری: حالت غیرپرمطلب `↩️`/`✅` را نشان می‌دهد؛ حالت پرمطلب اندازه و مسیر/URL منبع را نیز شامل می‌شود.

## خط لوله پاسخ خودکار

- `getReplyFromConfig` مقدار `{ text?, mediaUrl?, mediaUrls? }` را برمی‌گرداند.
- وقتی رسانه وجود دارد، فرستنده وب مسیرهای محلی یا URLها را با همان خط لوله `openclaw message send` نهایی می‌کند.
- اگر چند ورودی رسانه ارائه شود، به‌ترتیب ارسال می‌شوند.

## رسانه ورودی به فرمان‌ها (Pi)

- وقتی پیام‌های وب ورودی شامل رسانه باشند، OpenClaw آن را در یک فایل موقت دانلود می‌کند و متغیرهای قالب‌سازی را در دسترس قرار می‌دهد:
  - `{{MediaUrl}}` شبه-URL برای رسانه ورودی.
  - `{{MediaPath}}` مسیر موقت محلی که پیش از اجرای فرمان نوشته شده است.
- وقتی سندباکس Docker برای هر نشست فعال باشد، رسانه ورودی به فضای کاری سندباکس کپی می‌شود و `MediaPath`/`MediaUrl` به یک مسیر نسبی مانند `media/inbound/<filename>` بازنویسی می‌شوند.
- فهم رسانه (اگر از طریق `tools.media.*` یا `tools.media.models` مشترک پیکربندی شده باشد) پیش از قالب‌سازی اجرا می‌شود و می‌تواند بلوک‌های `[Image]`، `[Audio]`، و `[Video]` را در `Body` درج کند.
  - صدا `{{Transcript}}` را تنظیم می‌کند و از رونوشت برای تجزیه فرمان استفاده می‌کند تا فرمان‌های اسلش همچنان کار کنند.
  - توصیف‌های ویدیو و تصویر هر متن زیرنویس را برای تجزیه فرمان حفظ می‌کنند.
  - اگر مدل تصویر اصلی فعال از قبل به‌صورت بومی از بینایی پشتیبانی کند، OpenClaw بلوک خلاصه `[Image]` را رد می‌کند و به‌جای آن تصویر اصلی را به مدل می‌فرستد.
- به‌صورت پیش‌فرض فقط نخستین پیوست تصویر/صدا/ویدیوی منطبق پردازش می‌شود؛ برای پردازش چند پیوست، `tools.media.<cap>.attachments` را تنظیم کنید.

## محدودیت‌ها و خطاها

**سقف‌های ارسال خروجی (ارسال وب WhatsApp)**

- تصاویر: تا `channels.whatsapp.mediaMaxMb` (پیش‌فرض: 50 MB) پس از فشرده‌سازی دوباره.
- صدا/صوت/ویدیو: سقف 16 MB؛ اسناد: سقف 100 MB.
- رسانه بیش‌ازحد بزرگ یا ناخوانا → خطای روشن در گزارش‌ها و پاسخ رد می‌شود.

**سقف‌های فهم رسانه (رونویسی/توصیف)**

- پیش‌فرض تصویر: 10 MB (`tools.media.image.maxBytes`).
- پیش‌فرض صدا: 20 MB (`tools.media.audio.maxBytes`).
- پیش‌فرض ویدیو: 50 MB (`tools.media.video.maxBytes`).
- رسانه بیش‌ازحد بزرگ از فهم رد می‌شود، اما پاسخ‌ها همچنان با بدنه اصلی ادامه پیدا می‌کنند.

## نکات برای آزمون‌ها

- جریان‌های ارسال + پاسخ را برای حالت‌های تصویر/صدا/سند پوشش دهید.
- فشرده‌سازی دوباره تصاویر (کران اندازه) و پرچم یادداشت صوتی برای صدا را اعتبارسنجی کنید.
- مطمئن شوید پاسخ‌های چندرسانه‌ای به ارسال‌های ترتیبی منشعب می‌شوند.
