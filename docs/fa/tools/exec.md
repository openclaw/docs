دستورهای پوسته را در فضای کاری اجرا کنید. `exec` یک سطح پوسته تغییردهنده است: دستورها می‌توانند در هر جایی که میزبان انتخاب‌شده یا فایل‌سیستم سندباکس اجازه می‌دهد، فایل ایجاد، ویرایش یا حذف کنند. غیرفعال‌کردن ابزارهای فایل‌سیستم OpenClaw مانند `write`، `edit` یا `apply_patch`، `exec` را فقط‌خواندنی نمی‌کند.

از اجرای پیش‌زمینه و پس‌زمینه از طریق `process` پشتیبانی می‌کند. اگر `process` مجاز نباشد، `exec` به‌صورت همگام اجرا می‌شود و `yieldMs`/`background` را نادیده می‌گیرد. نشست‌های پس‌زمینه به هر عامل محدود می‌شوند؛ `process` فقط نشست‌های همان عامل را می‌بیند.

نکات:

- `host` فقط `auto`، `sandbox`، `gateway` یا `node` را می‌پذیرد. این گزینه انتخابگر نام میزبان نیست؛ مقادیر شبیه نام میزبان پیش از اجرای دستور رد می‌شوند.
- `host=node` برای هر فراخوانی از `auto` مجاز است؛ `host=gateway` برای هر فراخوانی فقط زمانی مجاز است که هیچ زمان‌اجرای سندباکسی فعال نباشد.
- بدون پیکربندی اضافی، `host=auto` همچنان «به‌سادگی کار می‌کند»: نبود سندباکس یعنی به `gateway` تبدیل می‌شود؛ وجود سندباکس فعال یعنی در سندباکس باقی می‌ماند.
- `elevated` از سندباکس به مسیر پیکربندی‌شده میزبان خارج می‌شود: به‌طور پیش‌فرض `gateway`، یا وقتی `tools.exec.host=node` (یا مقدار پیش‌فرض نشست `host=node`) باشد، `node`. این گزینه فقط زمانی در دسترس است که دسترسی ارتقایافته برای نشست/ارائه‌دهنده فعلی فعال باشد.
- تأییدهای `gateway`/`node` توسط فایل تأییدهای میزبان کنترل می‌شوند.
- `node` به یک Node جفت‌شده (برنامه همراه یا میزبان Node بدون رابط) نیاز دارد. اگر چند Node در دسترس است، برای انتخاب یکی از آن‌ها `exec.node` یا `tools.exec.node` را تنظیم کنید.
- `exec host=node` تنها مسیر اجرای پوسته برای Nodeها است؛ پوشش قدیمی `nodes.run` حذف شده است.
- در میزبان‌های غیر ویندوزی، exec در صورت تنظیم‌بودن از `SHELL` استفاده می‌کند؛ اگر `SHELL` برابر `fish` باشد، برای جلوگیری از bashismهای ناسازگار با fish، `bash` (یا `sh`) را از `PATH` ترجیح می‌دهد و اگر هیچ‌کدام وجود نداشته باشند، به `SHELL` برمی‌گردد.
- در میزبان‌های ویندوزی، exec شناسایی PowerShell 7 (`pwsh`) را ترجیح می‌دهد (Program Files، سپس ProgramW6432 و بعد PATH) و سپس به Windows PowerShell 5.1 برمی‌گردد.
- در میزبان‌های غیر ویندوزی Gateway، دستورهای exec در bash و zsh از یک تصویر لحظه‌ای راه‌اندازی استفاده می‌کنند. OpenClaw نام‌های مستعار/توابع قابل source و مجموعه‌ای کوچک و امن از متغیرهای محیطی را از فایل‌های راه‌اندازی پوسته در `$OPENCLAW_STATE_DIR/cache/shell-snapshots/` ثبت می‌کند و سپس پیش از هر دستور exec آن تصویر لحظه‌ای را source می‌کند. متغیرهایی که شبیه اطلاعات محرمانه‌اند کنار گذاشته می‌شوند؛ اجرای exec در سندباکس و Node از این تصویر لحظه‌ای استفاده نمی‌کند. برای غیرفعال‌کردن این مسیر تصویر لحظه‌ای، `OPENCLAW_EXEC_SHELL_SNAPSHOT=0` را در محیط فرایند Gateway تنظیم کنید.
- اجرای میزبان (`gateway`/`node`) برای جلوگیری از ربایش فایل اجرایی یا تزریق کد، `env.PATH` و بازنویسی‌های بارگذار (`LD_*`/`DYLD_*`) را رد می‌کند.
- OpenClaw متغیر `OPENCLAW_SHELL=exec` را در محیط دستور ایجادشده (از جمله اجرای PTY و سندباکس) تنظیم می‌کند تا قواعد پوسته/پروفایل بتوانند زمینه ابزار exec را تشخیص دهند.
- برای اجراهای با منشأ کانال، OpenClaw همچنین در صورتی که کانال آن شناسه‌ها را ارائه کرده باشد، یک بار JSON محدود از هویت فرستنده/گفت‌وگو را در `OPENCLAW_CHANNEL_CONTEXT` در دسترس قرار می‌دهد.
- `exec` نمی‌تواند دستورهای پوسته `openclaw channels login` یا `/approve` را اجرا کند: `openclaw channels login` یک جریان تعاملی احراز هویت کانال است و `/approve` باید از گرداننده دستور تأیید عبور کند، نه از پوسته. ورود به کانال را در یک پایانه روی میزبان Gateway اجرا کنید، یا اگر ابزار عامل ورود ویژه کانال وجود دارد از آن استفاده کنید (برای نمونه `whatsapp_login`).
- مهم: سندباکس‌سازی به‌طور پیش‌فرض **خاموش است**. اگر سندباکس‌سازی خاموش باشد، `host=auto` ضمنی به `gateway` تبدیل می‌شود. `host=sandbox` صریح همچنان به‌صورت بسته شکست می‌خورد و بی‌سروصدا روی میزبان Gateway اجرا نمی‌شود. سندباکس‌سازی را فعال کنید یا از `host=gateway` همراه با تأییدها استفاده کنید.
- بررسی‌های پیش از اجرای اسکریپت (برای خطاهای رایج نحو پوسته در Python/Node) فقط فایل‌های داخل محدوده مؤثر `workdir` را بررسی می‌کنند. اگر مسیر اسکریپت به خارج از `workdir` ختم شود، بررسی پیش از اجرا برای آن فایل انجام نمی‌شود. همچنین وقتی `host=gateway` و سیاست مؤثر `security=full` همراه با `ask=off` باشد، بررسی پیش از اجرا به‌طور کامل انجام نمی‌شود.
- برای کار طولانی‌مدتی که اکنون آغاز می‌شود، آن را یک‌بار شروع کنید و در صورت فعال‌بودن بیدارسازی خودکار تکمیل و خروجی‌دادن یا شکست دستور، به آن تکیه کنید. برای گزارش‌ها، وضعیت، ورودی یا مداخله از `process` استفاده کنید؛ زمان‌بندی را با حلقه‌های خواب، حلقه‌های مهلت زمانی یا نظرسنجی مکرر شبیه‌سازی نکنید.
- برای کاری که باید بعداً یا طبق برنامه انجام شود، به‌جای الگوهای خواب/تأخیر `exec` از cron استفاده کنید.

اجرای میزبان بدون تأیید، حالت پیش‌فرض Gateway و Node است (`security=full`، `ask=off`) — این رفتار از پیش‌فرض‌های سیاست میزبان می‌آید، نه از `host=auto`. اگر رفتار تأیید/فهرست مجاز را می‌خواهید، هم `tools.exec.*` و هم فایل تأییدهای میزبان را محدودتر کنید؛ بخش [تأییدهای اجرا](/fa/tools/exec-approvals#yolo-mode-no-approval) را ببینید. برای اجبار مسیریابی از طریق Gateway یا Node، صرف‌نظر از وضعیت سندباکس، `tools.exec.host` را تنظیم کنید یا از `/exec host=...` استفاده کنید.

### حالت‌ها

`tools.exec.mode` گزینه سیاست عادی‌سازی‌شده است. تنظیم آن مقادیر `security`/`ask` را تعیین می‌کند و نمی‌توان آن را با `tools.exec.security`/`tools.exec.ask` صریح ترکیب کرد.

| حالت        | امنیت    | پرسش       | رفتار                                                                                                                       |
| ----------- | ----------- | --------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `deny`      | `deny`      | `off`     | اجرا رد می‌شود.                                                                                                                |
| `allowlist` | `allowlist` | `off`     | فقط فرمان‌های موجود در فهرست مجاز/فایل‌های اجرایی امن اجرا می‌شوند؛ برای هیچ مورد دیگری پرسشی مطرح نمی‌شود.                                                                 |
| `ask`       | `allowlist` | `on-miss` | موارد منطبق با فهرست مجاز مستقیماً اجرا می‌شوند؛ همه موارد دیگر از انسان تأیید می‌خواهند.                                                                  |
| `auto`      | `allowlist` | `on-miss` | موارد منطبق با فهرست مجاز/فایل اجرایی امن مستقیماً اجرا می‌شوند؛ همه موارد دیگر پیش از درخواست تأیید از انسان، از بازبین خودکار بومی OpenClaw عبور می‌کنند. |
| `full`      | `full`      | `off`     | هیچ دروازه تأییدی وجود ندارد.                                                                                                              |

`ask`/`ask=always` صرف‌نظر از حالت، همچنان هر بار از انسان تأیید می‌خواهد.

تأیید بازبینی خودکار فقط یک‌بار قابل استفاده است. در Gateway، ‏OpenClaw مسیر حل‌شده فایل اجرایی را در اختیار بازبین قرار می‌دهد و اجرا را به همان مسیر مقید می‌کند. فرمان‌هایی که نمی‌توان آن‌ها را به یک طرح اجرای قابل‌اعمال فروکاست — مانند heredocها، بسط‌های پوسته یا نقل‌قول‌بندی پشتیبانی‌نشده wrapper — حتی اگر مدل در حالت عادی آن‌ها را مجاز بداند، به تأیید انسانی بازمی‌گردند.

تأیید فرمان‌های app-server متعلق به Codex که پیش‌تر از طریق سیاست صریح زمان اجرا یا سیاست بومی تعیین تکلیف نشده‌اند، از مسیر تأیید انسانی استفاده می‌کند. OpenClaw بازبین اجرای پیکربندی‌شده خود را برای این درخواست‌ها اجرا نمی‌کند، زیرا Codex فایل اجرایی حل‌شده و قابل‌اعمالی ارائه نمی‌دهد که بتوان تصمیم بازبینی را به فرمانی که Codex اجرا می‌کند مقید کرد.

### ارزیابی درون‌خطی (`strictInlineEval`)

وقتی `tools.exec.strictInlineEval` برابر با `true` باشد، شکل‌های ارزیابی درون‌خطی مفسر به بازبین یا تأیید صریح نیاز دارند: `python -c`، `node -e`، `ruby -e`، `perl -e`، `php -r`، `lua -e`، `osascript -e` و شکل‌های مشابه در سایر مفسرها و حامل‌های فرمان پشتیبانی‌شده (`awk`، `find -exec`، `make`، `sed`، `xargs` و موارد دیگر). در `mode=auto`، مسیر عادی تأیید اجرا ممکن است به بازبین خودکار بومی اجازه دهد یک فرمان یک‌باره و به‌وضوح کم‌خطر را مجاز کند؛ فراخوانی‌های مستقیم `system.run` روی میزبان Node همچنان به تأیید صریح نیاز دارند، زیرا نمی‌توانند فرمان را به مسیر تأیید انسانی تحویل دهند. اگر بازبین درخواست تأیید کند، درخواست به انسان ارجاع می‌شود. `allow-always` همچنان می‌تواند فراخوانی‌های بی‌خطر مفسر/اسکریپت را ماندگار کند، اما شکل‌های ارزیابی درون‌خطی به قوانین مجاز ماندگار تبدیل نمی‌شوند.

### مدیریت PATH

- `host=gateway`: مقدار `PATH` پوسته ورود شما را با محیط اجرا ادغام می‌کند. بازنویسی‌های `env.PATH` برای اجرای میزبان رد می‌شوند. خود daemon همچنان با یک `PATH` حداقلی اجرا می‌شود:
  - macOS: `/opt/homebrew/bin`، `/usr/local/bin`، `/usr/bin`، `/bin`
  - Linux: `/usr/local/bin`، `/usr/bin`، `/bin`
  - برای جلوگیری از بازنویسی مسیرهای اولویت‌دار هنگام راه‌اندازی توسط پیکربندی پوسته کاربر (مانند `~/.zshenv` یا `/etc/zshenv`) مدخل‌های `tools.exec.pathPrepend` به‌صورت امن، درست پیش از اجرا درون فرمان پوسته، به ابتدای `PATH` نهایی افزوده می‌شوند.
- `host=sandbox`: فرمان `sh -lc` (پوسته ورود) را داخل کانتینر اجرا می‌کند؛ بنابراین `/etc/profile` ممکن است `PATH` را بازنشانی کند. OpenClaw پس از بارگذاری پروفایل، `env.PATH` را از طریق یک متغیر محیطی داخلی (بدون درون‌یابی پوسته) به ابتدا اضافه می‌کند؛ `tools.exec.pathPrepend` نیز در اینجا اعمال می‌شود.
- `host=node`: فقط بازنویسی‌های محیطی مسدودنشده‌ای که ارسال می‌کنید به Node فرستاده می‌شوند. بازنویسی‌های `env.PATH` برای اجرای میزبان رد و توسط میزبان‌های Node نادیده گرفته می‌شوند. اگر به مدخل‌های PATH بیشتری روی یک Node نیاز دارید، محیط سرویس میزبان Node ‏(systemd/launchd) را پیکربندی کنید یا ابزارها را در مکان‌های استاندارد نصب کنید.

اتصال Node به تفکیک عامل (از نمایه فهرست عامل در پیکربندی استفاده کنید):

```bash
openclaw config get agents.list
openclaw config set 'agents.list[0].tools.exec.node' "node-id-or-name"
```

رابط کاربری کنترل: صفحه **دستگاه‌ها** یک پنل کوچک «اتصال Node اجرا» برای همین تنظیمات دارد.

## بازنویسی‌های نشست (`/exec`)

از `/exec` برای تنظیم پیش‌فرض‌های **مختص هر نشست** برای `host`، `security`، `ask` و `node` استفاده کنید. برای نمایش مقادیر فعلی، `/exec` را بدون آرگومان ارسال کنید.

مثال:

```text
/exec host=auto security=allowlist ask=on-miss node=mac-1
```

`/exec` فقط برای **فرستندگان مجاز** معتبر است (فهرست‌های مجاز کانال/جفت‌سازی به‌اضافه `commands.useAccessGroups`). این فرمان فقط **وضعیت نشست** را به‌روزرسانی می‌کند و پیکربندی را نمی‌نویسد. فرستندگان مجاز کانال خارجی می‌توانند این پیش‌فرض‌های نشست را تنظیم کنند. کلاینت‌های داخلی Gateway/webchat برای ماندگارکردن آن‌ها به `operator.admin` نیاز دارند.

برای غیرفعال‌کردن قطعی اجرا، آن را از طریق سیاست ابزار رد کنید (`tools.deny: ["exec"]` یا تنظیم مختص هر عامل). تأییدهای میزبان همچنان اعمال می‌شوند، مگر اینکه `security=full` و `ask=off` را صریحاً تنظیم کنید.

## تأییدهای اجرا (برنامه همراه / میزبان Node)

عامل‌های سندباکس‌شده می‌توانند پیش از اجرای `exec` روی Gateway یا میزبان Node، برای هر درخواست به تأیید نیاز داشته باشند. برای سیاست، فهرست مجاز و روند رابط کاربری، بخش [تأییدهای اجرا](/fa/tools/exec-approvals) را ببینید.

وقتی تأیید انسانی لازم باشد، جریان‌های میزبان Node و Gateway غیربومی بلافاصله با `status: "approval-pending"` و یک شناسه تأیید بازمی‌گردند. در عوض، جریان‌های گفت‌وگوی بومی و Gateway رابط وب می‌توانند به‌صورت درون‌خطی منتظر بمانند و پس از تأیید، نتیجه نهایی فرمان را بازگردانند. نتیجه `approval-pending` به این معناست که فرمان آغاز نشده است؛ بنابراین هشدارهای بازگشت به اجرای پیش‌زمینه فقط زمانی ظاهر می‌شوند که فرمان تأییدشده واقعاً به‌صورت درون‌خطی اجرا شود. اجراهای ناهمگام تأییدشده رویدادهای سیستمی پیشرفت و تکمیل فرمان را منتشر می‌کنند (`Exec running` / `Exec finished`)؛ تأییدهای ردشده یا منقضی‌شده نهایی هستند و نشست عامل را با رویداد سیستمی ردشدن بیدار نمی‌کنند.

در کانال‌هایی با کارت‌ها/دکمه‌های بومی تأیید، عامل باید ابتدا به همان رابط کاربری بومی تکیه کند و تنها زمانی فرمان دستی `/approve` را درج کند که نتیجهٔ ابزار صراحتاً اعلام کند تأییدهای چت در دسترس نیستند یا تأیید دستی تنها مسیر است.

## فهرست مجاز + باینری‌های امن

اعمال دستی فهرست مجاز با الگوهای glob مسیر حل‌شدهٔ باینری و الگوهای glob نام سادهٔ فرمان مطابقت می‌دهد. نام‌های ساده فقط با فرمان‌هایی مطابقت دارند که از طریق PATH فراخوانی شده‌اند؛ بنابراین `rg` می‌تواند با `/opt/homebrew/bin/rg` مطابقت داشته باشد، وقتی فرمان `rg` است، اما با `./rg` یا `/tmp/rg` مطابقت ندارد.

وقتی `security=allowlist`، فرمان‌های پوسته فقط زمانی به‌طور خودکار مجاز می‌شوند که هر بخش خط لوله در فهرست مجاز باشد یا یک باینری امن باشد. زنجیره‌سازی (`;`، `&&`، `||`) و تغییرمسیرها در حالت فهرست مجاز رد می‌شوند، مگر اینکه هر بخش سطح‌بالا شرایط فهرست مجاز را برآورده کند (از جمله باینری‌های امن). تغییرمسیرها همچنان پشتیبانی نمی‌شوند. اعتماد پایدار `allow-always` این قاعده را دور نمی‌زند: یک فرمان زنجیره‌ای همچنان مستلزم تطابق هر بخش سطح‌بالا است.

`autoAllowSkills` یک مسیر تسهیل‌کنندهٔ جداگانه در تأییدهای exec است و با ورودی‌های دستی فهرست مجاز مسیر یکسان نیست. برای اعتماد صریح و سخت‌گیرانه، `autoAllowSkills` را غیرفعال نگه دارید.

از این دو کنترل برای کارهای متفاوت استفاده کنید:

- `tools.exec.safeBins`: فیلترهای کوچک جریان، فقط از طریق stdin.
- `tools.exec.safeBinTrustedDirs`: دایرکتوری‌های مورد اعتماد اضافی و صریح برای مسیرهای اجرایی باینری امن.
- `tools.exec.safeBinProfiles`: سیاست صریح argv برای باینری‌های امن سفارشی.
- allowlist: اعتماد صریح برای مسیرهای اجرایی.

`safeBins` را یک فهرست مجاز عمومی تلقی نکنید و باینری‌های مفسر/محیط اجرا (برای مثال `python3`، `node`، `ruby`، `bash`) را اضافه نکنید. اگر به آن‌ها نیاز دارید، از ورودی‌های صریح فهرست مجاز استفاده کنید و اعلان‌های تأیید را فعال نگه دارید.

`openclaw security audit` هنگامی هشدار می‌دهد که ورودی‌های مفسر/محیط اجرای `safeBins` فاقد پروفایل صریح باشند، و `openclaw doctor --fix` می‌تواند ورودی‌های سفارشی و مفقود `safeBinProfiles` را داربست‌بندی کند. `openclaw security audit` و `openclaw doctor` همچنین هنگامی هشدار می‌دهند که باینری‌های دارای رفتار گسترده مانند `jq` را صراحتاً دوباره به `safeBins` اضافه کنید (`jq` می‌تواند داده‌های محیط را بخواند و کد jq را از ماژول‌ها یا فایل‌های راه‌اندازی بارگذاری کند؛ بنابراین به‌جای آن ورودی‌های صریح فهرست مجاز یا اجراهای مشروط به تأیید را ترجیح دهید). `jq` حتی در صورت درج صریح نیز به‌عنوان باینری امن رد می‌شود. اگر مفسرها را صراحتاً در فهرست مجاز قرار می‌دهید، `tools.exec.strictInlineEval` را فعال کنید تا شکل‌های ارزیابی درون‌خطی کد همچنان به تأیید بازبین یا تأیید صریح نیاز داشته باشند.

برای جزئیات کامل سیاست و مثال‌ها، به [تأییدهای Exec](/fa/tools/exec-approvals-advanced#safe-bins-stdin-only) و [باینری‌های امن در برابر فهرست مجاز](/fa/tools/exec-approvals-advanced#safe-bins-versus-allowlist) مراجعه کنید.

## مثال‌ها

پیش‌زمینه:

```json
{ "tool": "exec", "command": "ls -la" }
```

پس‌زمینه + نظرسنجی:

```json
{"tool":"exec","command":"npm run build","yieldMs":1000}
{"tool":"process","action":"poll","sessionId":"<id>"}
```

نظرسنجی برای دریافت وضعیت در صورت نیاز است، نه حلقه‌های انتظار. اگر بیدارسازی خودکار هنگام تکمیل فعال باشد، فرمان می‌تواند هنگام تولید خروجی یا شکست، نشست را بیدار کند.

ارسال کلیدها (به‌سبک tmux):

```json
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["Enter"]}
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["C-c"]}
{"tool":"process","action":"send-keys","sessionId":"<id>","keys":["Up","Up","Enter"]}
```

ثبت (فقط ارسال CR):

```json
{ "tool": "process", "action": "submit", "sessionId": "<id>" }
```

چسباندن (به‌طور پیش‌فرض کروشه‌گذاری‌شده):

```json
{ "tool": "process", "action": "paste", "sessionId": "<id>", "text": "line1\nline2\n" }
```

## apply_patch

`apply_patch` یک زیرابزار از `exec` برای ویرایش‌های ساخت‌یافتهٔ چندفایلی است. به‌طور پیش‌فرض فعال است و برای هر ارائه‌دهندهٔ مدل در دسترس قرار دارد؛ `allowModels` می‌تواند آن را محدود کند. فقط زمانی از پیکربندی استفاده کنید که می‌خواهید آن را غیرفعال کنید یا به مدل‌های مشخصی محدود سازید:

```json5
{
  tools: {
    exec: {
      applyPatch: { workspaceOnly: true, allowModels: ["gpt-5.6-sol"] },
    },
  },
}
```

نکات:

- سیاست ابزار همچنان اعمال می‌شود؛ `allow: ["write"]` به‌طور ضمنی `apply_patch` را مجاز می‌کند.
- `deny: ["write"]`، `apply_patch` را رد نمی‌کند؛ `apply_patch` را صراحتاً رد کنید یا زمانی که نوشتن وصله نیز باید مسدود شود از `deny: ["group:fs"]` استفاده کنید.
- پیکربندی زیر `tools.exec.applyPatch` قرار دارد.
- مقدار پیش‌فرض `tools.exec.applyPatch.enabled` برابر `true` است؛ برای غیرفعال‌کردن ابزار، آن را روی `false` تنظیم کنید.
- مقدار پیش‌فرض `tools.exec.applyPatch.workspaceOnly` برابر `true` (محدود به فضای کاری) است. فقط زمانی آن را روی `false` تنظیم کنید که عمداً می‌خواهید `apply_patch` خارج از دایرکتوری فضای کاری بنویسد/حذف کند.
- `tools.exec.applyPatch.allowModels` یک فهرست مجاز اختیاری از شناسه‌های مدل است (خام، مانند `gpt-5.4`، یا کامل، مانند `openai/gpt-5.4`). وقتی تنظیم شود، فقط مدل‌های منطبق ابزار را دریافت می‌کنند؛ وقتی تنظیم نشده باشد، همهٔ مدل‌ها آن را دریافت می‌کنند.
