عامل‌های فرعی اجراهای پس‌زمینه‌ی عامل هستند که از یک اجرای عامل موجود ایجاد می‌شوند.
آن‌ها در نشست خودشان (`agent:<agentId>:subagent:<uuid>`) اجرا می‌شوند و،
پس از پایان، نتیجه‌ی خود را به کانال گفت‌وگوی درخواست‌کننده **اعلام** می‌کنند.
هر اجرای عامل فرعی به‌عنوان یک
[وظیفه‌ی پس‌زمینه](/fa/automation/tasks) رهگیری می‌شود.

اهداف اصلی:

- موازی‌سازی کارهای «پژوهش / وظیفه‌ی طولانی / ابزار کند» بدون مسدود کردن اجرای اصلی.
- جدا نگه داشتن عامل‌های فرعی به‌صورت پیش‌فرض (جداسازی نشست + سندباکس اختیاری).
- سخت کردن سوءاستفاده از سطح ابزار: عامل‌های فرعی به‌صورت پیش‌فرض ابزارهای نشست را دریافت نمی‌کنند.
- پشتیبانی از عمق تودرتویی قابل پیکربندی برای الگوهای هماهنگ‌کننده.

<Note>
**نکته‌ی هزینه:** هر عامل فرعی به‌صورت پیش‌فرض زمینه و مصرف توکن خودش را دارد.
برای وظایف سنگین یا تکراری، یک مدل ارزان‌تر برای عامل‌های فرعی تنظیم کنید
و عامل اصلی خود را روی یک مدل باکیفیت‌تر نگه دارید. از طریق
`agents.defaults.subagents.model` یا بازنویسی‌های مخصوص هر عامل پیکربندی کنید.
وقتی یک فرزند واقعاً به متن گفت‌وگوی فعلی درخواست‌کننده نیاز دارد، عامل می‌تواند برای همان یک ایجاد، `context: "fork"` را درخواست کند.
نشست‌های عامل فرعی وابسته به رشته به‌صورت پیش‌فرض `context: "fork"` دارند،
زیرا گفت‌وگوی فعلی را به یک رشته‌ی پیگیری منشعب می‌کنند.
</Note>

## فرمان اسلش

از `/subagents` برای بررسی یا کنترل اجرای عامل‌های فرعی برای **نشست فعلی**
استفاده کنید:

```text
/subagents list
/subagents kill <id|#|all>
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
/subagents send <id|#> <message>
/subagents steer <id|#> <message>
/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]
```

از [`/steer <message>`](/fa/tools/steer) سطح بالا برای هدایت اجرای فعال نشست درخواست‌کننده‌ی فعلی استفاده کنید. وقتی هدف یک اجرای فرزند است، از `/subagents steer <id|#> <message>` استفاده کنید.

`/subagents info` فراداده‌ی اجرا را نشان می‌دهد (وضعیت، زمان‌مهرها، شناسه‌ی نشست،
مسیر رونوشت، پاک‌سازی). برای نمای یادآوری محدود و پالایش‌شده از نظر ایمنی از `sessions_history` استفاده کنید؛ وقتی به رونوشت کامل خام نیاز دارید، مسیر رونوشت روی دیسک را بررسی کنید.

### کنترل‌های اتصال رشته

این فرمان‌ها روی کانال‌هایی کار می‌کنند که از اتصال‌های رشته‌ی پایدار پشتیبانی می‌کنند.
در پایین، [کانال‌های پشتیبان رشته](#thread-supporting-channels) را ببینید.

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### رفتار ایجاد

`/subagents spawn` یک عامل فرعی پس‌زمینه را به‌عنوان فرمان کاربر (نه یک رله‌ی داخلی)
شروع می‌کند و وقتی اجرا پایان یابد، یک به‌روزرسانی نهایی تکمیل را به گفت‌وگوی
درخواست‌کننده می‌فرستد.

## حالت‌های زمینه

عامل‌های فرعی بومی جداشده شروع می‌شوند، مگر این‌که فراخواننده صریحاً درخواست کند
رونوشت فعلی fork شود.

| حالت       | زمان استفاده                                                                                                                         | رفتار                                                                          |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `isolated` | پژوهش تازه، پیاده‌سازی مستقل، کار ابزار کند، یا هر چیزی که بتوان آن را در متن وظیفه خلاصه کرد                           | یک رونوشت فرزند پاک ایجاد می‌کند. این حالت پیش‌فرض است و مصرف توکن را پایین‌تر نگه می‌دارد.  |
| `fork`     | کاری که به گفت‌وگوی فعلی، نتایج ابزار قبلی، یا دستورهای ظریفی وابسته است که از قبل در رونوشت درخواست‌کننده وجود دارند | رونوشت درخواست‌کننده را پیش از شروع فرزند، به نشست فرزند منشعب می‌کند. |

از `fork` با احتیاط استفاده کنید. این برای واگذاری حساس به زمینه است، نه
جایگزینی برای نوشتن یک درخواست وظیفه‌ی روشن.

## ابزار: `sessions_spawn`

یک اجرای عامل فرعی را با `deliver: false` روی مسیر سراسری `subagent` شروع می‌کند،
سپس یک گام اعلام را اجرا می‌کند و پاسخ اعلام را در کانال گفت‌وگوی درخواست‌کننده
ارسال می‌کند.

در دسترس بودن به خط‌مشی ابزار مؤثر فراخواننده بستگی دارد. پروفایل‌های `coding` و
`full` به‌صورت پیش‌فرض `sessions_spawn` را در معرض استفاده قرار می‌دهند. پروفایل `messaging`
این کار را نمی‌کند؛ برای عامل‌هایی که باید کار واگذار کنند، `tools.alsoAllow: ["sessions_spawn", "sessions_yield",
"subagents"]` را اضافه کنید یا از `tools.profile: "coding"` استفاده کنید.
خط‌مشی‌های اجازه/منع مربوط به کانال/گروه، ارائه‌دهنده، سندباکس، و هر عامل همچنان می‌توانند
پس از مرحله‌ی پروفایل ابزار را حذف کنند. از همان نشست از `/tools` استفاده کنید تا فهرست ابزار مؤثر را تأیید کنید.

**پیش‌فرض‌ها:**

- **مدل:** از فراخواننده به ارث می‌برد، مگر این‌که `agents.defaults.subagents.model` (یا `agents.list[].subagents.model` مخصوص هر عامل) را تنظیم کنید؛ `sessions_spawn.model` صریح همچنان اولویت دارد.
- **تفکر:** از فراخواننده به ارث می‌برد، مگر این‌که `agents.defaults.subagents.thinking` (یا `agents.list[].subagents.thinking` مخصوص هر عامل) را تنظیم کنید؛ `sessions_spawn.thinking` صریح همچنان اولویت دارد.
- **مهلت اجرای اجرا:** اگر `sessions_spawn.runTimeoutSeconds` حذف شود، OpenClaw وقتی `agents.defaults.subagents.runTimeoutSeconds` تنظیم شده باشد از آن استفاده می‌کند؛ در غیر این صورت به `0` (بدون مهلت زمانی) برمی‌گردد.

### حالت درخواست واگذاری

`agents.defaults.subagents.delegationMode` فقط راهنمایی درخواست را کنترل می‌کند؛ خط‌مشی ابزار را تغییر نمی‌دهد و واگذاری را enforce نمی‌کند.

- `suggest` (پیش‌فرض): یادآوری استاندارد درخواست برای استفاده از عامل‌های فرعی در کارهای بزرگ‌تر یا کندتر را نگه می‌دارد.
- `prefer`: به عامل اصلی می‌گوید پاسخ‌گو بماند و هر چیزی پیچیده‌تر از یک پاسخ مستقیم را از طریق `sessions_spawn` واگذار کند.

بازنویسی‌های مخصوص هر عامل از `agents.list[].subagents.delegationMode` استفاده می‌کنند.

```json5
{
  agents: {
    defaults: {
      subagents: {
        delegationMode: "prefer",
        maxConcurrent: 4,
      },
    },
    list: [
      {
        id: "coordinator",
        subagents: { delegationMode: "prefer" },
      },
    ],
  },
}
```

</ParamField>
<ParamField path="context" type='"isolated" | "fork"' default="isolated">
  `fork` رونوشت فعلی درخواست‌کننده را به نشست فرزند منشعب می‌کند. فقط زیرعامل‌های بومی. ایجادهای متصل به رشته به‌طور پیش‌فرض `fork` هستند؛ ایجادهای بدون رشته به‌طور پیش‌فرض `isolated` هستند.
</ParamField>

<Warning>
`sessions_spawn` پارامترهای تحویل کانال (`target`,
`channel`, `to`, `threadId`, `replyTo`, `transport`) را نمی‌پذیرد. برای تحویل، از
`message`/`sessions_send` از اجرای ایجادشده استفاده کنید.
</Warning>

### نام‌های وظیفه و هدف‌گیری

`taskName` یک شناسهٔ رو به مدل برای هماهنگ‌سازی است، نه کلید نشست.
وقتی یک هماهنگ‌کننده ممکن است بعداً نیاز داشته باشد آن فرزند را هدایت
یا متوقف کند، از آن برای نام‌های پایدار فرزند مانند `review_subagents`,
`linux_validation`, یا `docs_update` استفاده کنید.

تفکیک هدف، تطابق‌های دقیق `taskName` و پیشوندهای بدون ابهام را می‌پذیرد.
تطابق به همان پنجرهٔ هدف فعال/اخیر محدود است که توسط هدف‌های شماره‌دار
`/subagents` استفاده می‌شود، بنابراین یک فرزند کامل‌شدهٔ کهنه، استفادهٔ
دوباره از یک شناسه را مبهم نمی‌کند. اگر دو فرزند فعال یا اخیر همان
`taskName` را داشته باشند، هدف مبهم است؛ به‌جای آن از نمایهٔ فهرست،
کلید نشست، یا شناسهٔ اجرا استفاده کنید.

هدف‌های رزروشدهٔ `last` و `all` مقدارهای معتبر `taskName` نیستند
زیرا از قبل معنای کنترلی دارند.

## ابزار: `sessions_yield`

نوبت فعلی مدل را پایان می‌دهد و منتظر می‌ماند رویدادهای زمان اجرا،
عمدتاً رویدادهای تکمیل زیرعامل، به‌عنوان پیام بعدی برسند. وقتی پس از
ایجاد کار فرزندِ ضروری، درخواست‌کننده تا رسیدن آن تکمیل‌ها نمی‌تواند
پاسخ نهایی تولید کند، از آن استفاده کنید.

`sessions_yield` سازوکار انتظار است. آن را فقط برای تشخیص تکمیل فرزند
با حلقه‌های polling روی `subagents`, `sessions_list`, `sessions_history`,
`sleep` پوسته، یا polling فرایند جایگزین نکنید.

فقط وقتی از `sessions_yield` استفاده کنید که فهرست ابزار مؤثر نشست شامل
آن باشد. برخی پروفایل‌های حداقلی یا سفارشی ابزار ممکن است `sessions_spawn`
و `subagents` را بدون ارائهٔ `sessions_yield` ارائه کنند؛ در این حالت،
فقط برای انتظار تکمیل، حلقهٔ polling ابداع نکنید.

وقتی فرزندان فعال وجود دارند، OpenClaw یک بلوک prompt فشردهٔ تولیدشده
توسط زمان اجرا با نام `Active Subagents` را در نوبت‌های عادی تزریق
می‌کند تا درخواست‌کننده بتواند نشست‌های فرزند فعلی، شناسه‌های اجرا،
وضعیت‌ها، برچسب‌ها، وظیفه‌ها، و نام‌های مستعار `taskName` را بدون
polling ببیند. فیلدهای وظیفه و برچسب در آن بلوک به‌عنوان داده نقل‌قول
می‌شوند، نه دستورالعمل، زیرا می‌توانند از آرگومان‌های ایجاد فراهم‌شده
توسط کاربر/مدل منشأ بگیرند.

## ابزار: `subagents`

اجراهای زیرعامل ایجادشده‌ای را که مالکشان نشست درخواست‌کننده است فهرست،
هدایت، یا متوقف می‌کند. به درخواست‌کنندهٔ فعلی محدود است؛ یک فرزند فقط
می‌تواند فرزندان تحت کنترل خودش را ببیند/کنترل کند.

از `subagents` برای وضعیت درخواستی، اشکال‌زدایی، هدایت، یا توقف استفاده
کنید. برای انتظار رویدادهای تکمیل از `sessions_yield` استفاده کنید.

## نشست‌های متصل به رشته

وقتی اتصال‌های رشته برای یک کانال فعال باشند، یک زیرعامل می‌تواند به
یک رشته متصل بماند تا پیام‌های پیگیری کاربر در آن رشته همچنان به همان
نشست زیرعامل مسیر‌دهی شوند.

### کانال‌های پشتیبان رشته

**Discord** در حال حاضر تنها کانال پشتیبانی‌شده است. از نشست‌های زیرعامل
متصل به رشتهٔ پایدار (`sessions_spawn` با `thread: true`)، کنترل‌های
دستی رشته (`/focus`, `/unfocus`, `/agents`, `/session idle`,
`/session max-age`)، و کلیدهای آداپتر
`channels.discord.threadBindings.enabled`,
`channels.discord.threadBindings.idleHours`,
`channels.discord.threadBindings.maxAgeHours`, و
`channels.discord.threadBindings.spawnSessions` پشتیبانی می‌کند.

### جریان سریع

<Steps>
  <Step title="ایجاد">
    `sessions_spawn` با `thread: true` (و به‌صورت اختیاری `mode: "session"`).
  </Step>
  <Step title="اتصال">
    OpenClaw یک رشته را به هدف آن نشست در کانال فعال ایجاد یا متصل می‌کند.
  </Step>
  <Step title="مسیر‌دهی پیگیری‌ها">
    پاسخ‌ها و پیام‌های پیگیری در آن رشته به نشست متصل مسیر‌دهی می‌شوند.
  </Step>
  <Step title="بررسی مهلت‌ها">
    از `/session idle` برای بررسی/به‌روزرسانی خروج خودکار از تمرکز در اثر عدم فعالیت و
    از `/session max-age` برای کنترل سقف سخت استفاده کنید.
  </Step>
  <Step title="جداسازی">
    از `/unfocus` برای جداسازی دستی استفاده کنید.
  </Step>
</Steps>

### کنترل‌های دستی

| فرمان              | اثر                                                                  |
| ------------------ | --------------------------------------------------------------------- |
| `/focus <target>`  | رشتهٔ فعلی را به یک هدف زیرعامل/نشست متصل می‌کند (یا یکی ایجاد می‌کند) |
| `/unfocus`         | اتصال رشتهٔ متصل فعلی را حذف می‌کند                                  |
| `/agents`          | اجراهای فعال و وضعیت اتصال را فهرست می‌کند (`thread:<id>` یا `unbound`) |

| `/session max-age` | سقف سخت را بررسی/به‌روزرسانی می‌کند (فقط رشته‌های متصل متمرکز)       |

### کلیدهای پیکربندی

- **پیش‌فرض سراسری:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **کلیدهای بازنویسی کانال و اتصال خودکار هنگام ایجاد** مختص آداپتر هستند. به [کانال‌های پشتیبان رشته](#thread-supporting-channels) در بالا مراجعه کنید.

برای جزئیات فعلی آداپتر، [مرجع پیکربندی](/fa/gateway/configuration-reference) و
[فرمان‌های اسلش](/fa/tools/slash-commands) را ببینید.

### فهرست مجاز

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  فهرست شناسه‌های عاملی که می‌توانند از طریق `agentId` صریح هدف قرار گیرند (`["*"]` هر موردی را مجاز می‌کند). پیش‌فرض: فقط عامل درخواست‌کننده. اگر فهرستی تنظیم می‌کنید و همچنان می‌خواهید درخواست‌کننده خودش را با `agentId` ایجاد کند، شناسهٔ درخواست‌کننده را در فهرست قرار دهید.
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  فهرست مجاز عامل هدف پیش‌فرض که وقتی عامل درخواست‌کننده `subagents.allowAgents` خودش را تنظیم نکرده باشد استفاده می‌شود.
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  فراخوانی‌های `sessions_spawn` را که `agentId` را حذف می‌کنند مسدود می‌کند (انتخاب صریح پروفایل را اجباری می‌کند). بازنویسی برای هر عامل: `agents.list[].subagents.requireAgentId`.
</ParamField>

اگر نشست درخواست‌کننده sandbox شده باشد، `sessions_spawn` هدف‌هایی را
که بدون sandbox اجرا می‌شوند رد می‌کند.

### کشف

از `agents_list` استفاده کنید تا ببینید کدام شناسه‌های عامل در حال حاضر
برای `sessions_spawn` مجاز هستند. پاسخ، مدل مؤثر هر عامل فهرست‌شده و
فرادادهٔ زمان اجرای تعبیه‌شده را شامل می‌شود تا فراخواننده‌ها بتوانند
PI، سرور برنامهٔ Codex، و سایر زمان‌های اجرای بومی پیکربندی‌شده را از
هم تشخیص دهند.

### بایگانی خودکار

- نشست‌های زیرعامل پس از `agents.defaults.subagents.archiveAfterMinutes` (پیش‌فرض `60`) به‌طور خودکار بایگانی می‌شوند.
- بایگانی از `sessions.delete` استفاده می‌کند و رونوشت را به `*.deleted.<timestamp>` تغییر نام می‌دهد (همان پوشه).
- `cleanup: "delete"` بلافاصله پس از اعلام، بایگانی می‌کند (همچنان رونوشت را از طریق تغییر نام نگه می‌دارد).
- بایگانی خودکار به‌صورت بهترین تلاش است؛ اگر Gateway بازراه‌اندازی شود، تایمرهای معلق از دست می‌روند.
- `runTimeoutSeconds` بایگانی خودکار انجام نمی‌دهد؛ فقط اجرا را متوقف می‌کند. نشست تا زمان بایگانی خودکار باقی می‌ماند.
- بایگانی خودکار به‌طور یکسان برای نشست‌های عمق ۱ و عمق ۲ اعمال می‌شود.
- پاک‌سازی مرورگر جدا از پاک‌سازی بایگانی است: زبانه‌ها/فرایندهای ردیابی‌شدهٔ مرورگر وقتی اجرا تمام می‌شود به‌صورت بهترین تلاش بسته می‌شوند، حتی اگر رونوشت/رکورد نشست نگه داشته شود.

## زیرعامل‌های تودرتو

به‌طور پیش‌فرض، زیرعامل‌ها نمی‌توانند زیرعامل‌های خودشان را ایجاد کنند
(`maxSpawnDepth: 1`). برای فعال‌سازی یک سطح تودرتویی، `maxSpawnDepth: 2`
را تنظیم کنید — **الگوی هماهنگ‌کننده**: اصلی → زیرعامل هماهنگ‌کننده →
زیر-زیرعامل‌های کاری.

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // allow sub-agents to spawn children (default: 1)
        maxChildrenPerAgent: 5, // max active children per agent session (default: 5)
        maxConcurrent: 8, // global concurrency lane cap (default: 8)
        runTimeoutSeconds: 900, // default timeout for sessions_spawn when omitted (0 = no timeout)
      },
    },
  },
}
```

### سطح‌های عمق

| عمق | شکل کلید نشست                                | نقش                                          | می‌تواند ایجاد کند؟          |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | عامل اصلی                                    | همیشه                        |
| 1     | `agent:<id>:subagent:<uuid>`                 | زیرعامل (هماهنگ‌کننده وقتی عمق ۲ مجاز باشد) | فقط اگر `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | زیر-زیرعامل (کارگر برگ)                      | هرگز                         |

### زنجیرهٔ اعلام

نتایج در زنجیره به بالا برمی‌گردند:

1. کارگر عمق ۲ تمام می‌شود → به والدش اعلام می‌کند (هماهنگ‌کنندهٔ عمق ۱).
2. هماهنگ‌کنندهٔ عمق ۱ اعلام را دریافت می‌کند، نتایج را ترکیب می‌کند، تمام می‌شود → به اصلی اعلام می‌کند.
3. عامل اصلی اعلام را دریافت می‌کند و به کاربر تحویل می‌دهد.

هر سطح فقط اعلام‌های فرزندان مستقیم خودش را می‌بیند.

<Note>
**راهنمایی عملیاتی:** کار فرزند را یک‌بار شروع کنید و به‌جای ساختن
حلقه‌های polling پیرامون `sessions_list`, `sessions_history`,
`/subagents list`, یا فرمان‌های `exec` sleep، منتظر رویدادهای تکمیل
بمانید. `sessions_list` و `/subagents list` رابطه‌های نشست فرزند را روی
کار زنده متمرکز نگه می‌دارند — فرزندان زنده متصل می‌مانند، فرزندان
پایان‌یافته برای یک پنجرهٔ کوتاه اخیر قابل مشاهده می‌مانند، و پیوندهای
کهنهٔ فرزند که فقط در store هستند پس از پنجرهٔ تازگی‌شان نادیده گرفته
می‌شوند. این کار مانع می‌شود فرادادهٔ قدیمی `spawnedBy` /
`parentSessionKey` پس از راه‌اندازی دوباره، فرزندان شبح‌وار را دوباره
زنده کند. اگر پس از اینکه پاسخ نهایی را فرستاده‌اید یک رویداد تکمیل
فرزند برسد، پیگیری درست همان توکن بی‌صدای دقیق `NO_REPLY` / `no_reply`
است.
</Note>

### سیاست ابزار بر اساس عمق

- نقش و دامنه کنترل هنگام spawn شدن در فراداده نشست نوشته می‌شوند. این کار مانع می‌شود کلیدهای نشست تخت یا بازیابی‌شده به‌طور تصادفی دوباره امتیازهای orchestrator را به دست آورند.
- **عمق ۱ (orchestrator، وقتی `maxSpawnDepth >= 2`):** `sessions_spawn`، `subagents`، `sessions_list`، `sessions_history` را دریافت می‌کند تا بتواند فرزندان خود را مدیریت کند. سایر ابزارهای نشست/سیستم همچنان رد می‌شوند.
- **عمق ۱ (برگ، وقتی `maxSpawnDepth == 1`):** هیچ ابزار نشستی ندارد (رفتار پیش‌فرض فعلی).
- **عمق ۲ (worker برگ):** هیچ ابزار نشستی ندارد — `sessions_spawn` در عمق ۲ همیشه رد می‌شود. نمی‌تواند فرزندان بیشتری spawn کند.

### محدودیت spawn برای هر agent

هر نشست agent (در هر عمقی) می‌تواند هم‌زمان حداکثر `maxChildrenPerAgent`
(پیش‌فرض `5`) فرزند فعال داشته باشد. این از fan-out بی‌مهار
از یک orchestrator واحد جلوگیری می‌کند.

### توقف آبشاری

توقف یک orchestrator عمق ۱ به‌طور خودکار همه فرزندان عمق ۲
آن را متوقف می‌کند:

- `/stop` در گفت‌وگوی اصلی همه agentهای عمق ۱ را متوقف می‌کند و به فرزندان عمق ۲ آن‌ها آبشار می‌شود.
- `/subagents kill <id>` یک sub-agent مشخص را متوقف می‌کند و به فرزندان آن آبشار می‌شود.
- `/subagents kill all` همه sub-agentهای درخواست‌کننده را متوقف می‌کند و آبشار می‌شود.

## احراز هویت

احراز هویت sub-agent بر اساس **شناسه agent** حل می‌شود، نه بر اساس نوع نشست:

- کلید نشست sub-agent برابر است با `agent:<agentId>:subagent:<uuid>`.
- مخزن احراز هویت از `agentDir` همان agent بارگذاری می‌شود.
- پروفایل‌های احراز هویت agent اصلی به‌عنوان **fallback** ادغام می‌شوند؛ پروفایل‌های agent در تعارض‌ها پروفایل‌های اصلی را override می‌کنند.

این ادغام افزایشی است، بنابراین پروفایل‌های اصلی همیشه به‌عنوان
fallback در دسترس هستند. احراز هویت کاملا ایزوله برای هر agent هنوز پشتیبانی نمی‌شود.

## اعلام

Sub-agentها از طریق یک گام اعلام گزارش می‌دهند:

- گام اعلام داخل نشست sub-agent اجرا می‌شود (نه نشست درخواست‌کننده).
- اگر sub-agent دقیقا `ANNOUNCE_SKIP` پاسخ دهد، چیزی ارسال نمی‌شود.
- اگر آخرین متن assistant دقیقا توکن خاموش `NO_REPLY` / `no_reply` باشد، خروجی اعلام سرکوب می‌شود حتی اگر پیشرفت قابل مشاهده قبلی وجود داشته باشد.

تحویل به عمق درخواست‌کننده بستگی دارد:

- نشست‌های درخواست‌کننده سطح بالا از یک فراخوانی follow-up به `agent` با تحویل خارجی (`deliver=true`) استفاده می‌کنند.
- نشست‌های subagent درخواست‌کننده تودرتو یک تزریق follow-up داخلی (`deliver=false`) دریافت می‌کنند تا orchestrator بتواند نتایج فرزند را درون نشست ترکیب کند.
- اگر نشست subagent درخواست‌کننده تودرتو از بین رفته باشد، OpenClaw در صورت موجود بودن به درخواست‌کننده آن نشست fallback می‌کند.

برای نشست‌های درخواست‌کننده سطح بالا، تحویل مستقیم در حالت completion ابتدا
هر مسیر مکالمه/رشته bind‌شده و override hook را حل می‌کند، سپس
فیلدهای channel-target جاافتاده را از مسیر ذخیره‌شده نشست درخواست‌کننده پر می‌کند.
این باعث می‌شود completionها در گفت‌وگو/موضوع درست بمانند، حتی وقتی مبدا completion
فقط channel را شناسایی می‌کند.

تجمیع completion فرزند هنگام ساخت یافته‌های completion تودرتو به اجرای فعلی درخواست‌کننده
محدود می‌شود و از نشت خروجی‌های فرزندِ اجرای قبلیِ کهنه
به اعلام فعلی جلوگیری می‌کند. پاسخ‌های اعلام، وقتی روی channel adapterها در دسترس باشند،
مسیریابی رشته/موضوع را حفظ می‌کنند.

### زمینه اعلام

زمینه اعلام به یک بلوک رویداد داخلی پایدار نرمال‌سازی می‌شود:

| فیلد          | منبع                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| منبع         | `subagent` یا `cron`                                                                                          |
| شناسه‌های نشست    | کلید/شناسه نشست فرزند                                                                                          |
| نوع           | نوع اعلام + برچسب task                                                                                    |
| وضعیت         | مشتق‌شده از نتیجه runtime (`success`، `error`، `timeout` یا `unknown`) — **نه** استنباط‌شده از متن مدل |
| محتوای نتیجه | آخرین متن قابل مشاهده assistant، در غیر این صورت آخرین متن tool/toolResult پاک‌سازی‌شده                                |
| Follow-up      | دستورالعملی که توضیح می‌دهد چه زمانی پاسخ داده شود و چه زمانی خاموش بماند                                                           |

اجراهای نهاییِ شکست‌خورده وضعیت شکست را بدون بازپخش متن پاسخ
ثبت‌شده گزارش می‌کنند. در timeout، اگر فرزند فقط تا فراخوانی‌های tool پیش رفته باشد، اعلام
می‌تواند آن تاریخچه را به‌جای بازپخش خروجی خام tool
به یک خلاصه کوتاه از پیشرفت جزئی فشرده کند.

### خط آمار

payloadهای اعلام در انتها شامل یک خط آمار هستند (حتی وقتی wrap شده باشند):

- Runtime (مثلا `runtime 5m12s`).
- مصرف توکن (ورودی/خروجی/کل).
- هزینه برآوردی وقتی قیمت‌گذاری مدل پیکربندی شده باشد (`models.providers.*.models[].cost`).
- `sessionKey`، `sessionId` و مسیر transcript تا agent اصلی بتواند تاریخچه را از طریق `sessions_history` واکشی کند یا فایل روی دیسک را بررسی کند.

فراداده داخلی فقط برای orchestration است؛ پاسخ‌های کاربرمحور
باید با صدای عادی assistant بازنویسی شوند.

### چرا `sessions_history` ترجیح داده می‌شود

`sessions_history` مسیر امن‌تر orchestration است:

- یادآوری assistant ابتدا نرمال‌سازی می‌شود: برچسب‌های thinking حذف می‌شوند؛ scaffoldingهای `<relevant-memories>` / `<relevant_memories>` حذف می‌شوند؛ بلوک‌های payload متنی ساده XML مربوط به فراخوانی tool (`<tool_call>`، `<function_call>`، `<tool_calls>`، `<function_calls>`) حذف می‌شوند، شامل payloadهای کوتاه‌شده‌ای که هرگز درست بسته نمی‌شوند؛ scaffoldingهای tool-call/result تنزل‌یافته و نشانگرهای historical-context حذف می‌شوند؛ توکن‌های کنترل مدل نشت‌کرده (`<|assistant|>`، سایر `<|...|>`های ASCII، `<｜...｜>` تمام‌عرض) حذف می‌شوند؛ XML فراخوانی tool بدشکل MiniMax حذف می‌شود.
- متن‌های شبیه credential/token ردکت می‌شوند.
- بلوک‌های طولانی می‌توانند کوتاه شوند.
- تاریخچه‌های بسیار بزرگ می‌توانند ردیف‌های قدیمی‌تر را حذف کنند یا یک ردیف بیش‌ازحد بزرگ را با `[sessions_history omitted: message too large]` جایگزین کنند.
- بررسی transcript خام روی دیسک fallback است وقتی به transcript کامل بایت‌به‌بایت نیاز دارید.

## سیاست tool

Sub-agentها ابتدا از همان پروفایل و pipeline سیاست tool والد یا
agent هدف استفاده می‌کنند. پس از آن، OpenClaw لایه محدودیت
sub-agent را اعمال می‌کند.

بدون `tools.profile` محدودکننده، sub-agentها **همه toolها به‌جز
toolهای نشست** و toolهای سیستم را دریافت می‌کنند:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

`sessions_history` اینجا هم یک نمای یادآوری محدود و پاک‌سازی‌شده باقی می‌ماند —
dump خام transcript نیست.

وقتی `maxSpawnDepth >= 2` باشد، sub-agentهای orchestrator عمق ۱ علاوه بر آن
`sessions_spawn`، `subagents`، `sessions_list` و
`sessions_history` را دریافت می‌کنند تا بتوانند فرزندان خود را مدیریت کنند.

### Override از طریق config

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny wins
        deny: ["gateway", "cron"],
        // if allow is set, it becomes allow-only (deny still wins)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` یک فیلتر نهایی allow-only است. می‌تواند
مجموعه tool از پیش حل‌شده را محدودتر کند، اما نمی‌تواند tool حذف‌شده
توسط `tools.profile` را **دوباره اضافه کند**. برای مثال، `tools.profile: "coding"` شامل
`web_search`/`web_fetch` است اما tool `browser` را شامل نمی‌شود. برای اینکه
sub-agentهای coding-profile بتوانند از automation مرورگر استفاده کنند، browser را در
مرحله profile اضافه کنید:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

وقتی فقط یک agent باید automation مرورگر را دریافت کند، از
`agents.list[].tools.alsoAllow: ["browser"]` برای همان agent استفاده کنید.

## همزمانی

Sub-agentها از یک lane صف اختصاصی درون‌پردازشی استفاده می‌کنند:

- **نام lane:** `subagent`
- **همزمانی:** `agents.defaults.subagents.maxConcurrent` (پیش‌فرض `8`)

## زنده‌بودن و بازیابی

OpenClaw نبود `endedAt` را اثبات دائمی زنده بودن
یک sub-agent در نظر نمی‌گیرد. اجراهای پایان‌نیافته قدیمی‌تر از پنجره stale-run
دیگر در `/subagents list`، خلاصه‌های وضعیت،
گیت‌های completion فرزندان و بررسی‌های همزمانی هر نشست به‌عنوان active/pending شمرده نمی‌شوند.

پس از restart شدن gateway، اجراهای بازیابی‌شده پایان‌نیافته stale حذف می‌شوند مگر اینکه
نشست فرزند آن‌ها با `abortedLastRun: true` علامت‌گذاری شده باشد. آن
نشست‌های فرزند abort‌شده در restart همچنان از طریق جریان بازیابی orphan sub-agent
قابل بازیابی می‌مانند؛ این جریان پیش از پاک کردن نشانگر aborted،
یک پیام resume مصنوعی ارسال می‌کند.

بازیابی خودکار restart برای هر نشست فرزند محدود است. اگر همان فرزند
sub-agent درون پنجره rapid re-wedge بارها برای بازیابی orphan پذیرفته شود،
OpenClaw یک recovery tombstone را روی آن نشست پایدار می‌کند و در restartهای بعدی
دیگر آن را auto-resume نمی‌کند. برای هم‌سوسازی رکورد task،
`openclaw tasks maintenance --apply` را اجرا کنید، یا برای پاک کردن flagهای stale aborted recovery روی
نشست‌های tombstoned، `openclaw doctor --fix` را اجرا کنید.

<Note>
اگر spawn کردن sub-agent با Gateway `PAIRING_REQUIRED` /
`scope-upgrade` شکست خورد، پیش از ویرایش وضعیت pairing، RPC caller را بررسی کنید.
هماهنگی داخلی `sessions_spawn` باید با
`client.id: "gateway-client"` و `client.mode: "backend"` از طریق احراز هویت direct
loopback با shared-token/password وصل شود؛ این مسیر به baseline scope دستگاه paired شده
CLI وابسته نیست. callerهای ریموت، `deviceIdentity` صریح، مسیرهای device-token صریح،
و clientهای browser/node همچنان برای scope upgradeها به تایید عادی دستگاه نیاز دارند.
</Note>

## توقف

- ارسال `/stop` در گفت‌وگوی درخواست‌کننده، نشست درخواست‌کننده را abort می‌کند و هر اجرای active sub-agent که از آن spawn شده باشد را متوقف می‌کند و به فرزندان تودرتو آبشار می‌شود.
- `/subagents kill <id>` یک sub-agent مشخص را متوقف می‌کند و به فرزندان آن آبشار می‌شود.

## محدودیت‌ها

- اعلام sub-agent **best-effort** است. اگر gateway restart شود، کارهای pending «announce back» از دست می‌روند.
- Sub-agentها همچنان منابع همان فرایند gateway را به اشتراک می‌گذارند؛ `maxConcurrent` را به‌عنوان سوپاپ ایمنی در نظر بگیرید.
- `sessions_spawn` همیشه non-blocking است: بلافاصله `{ status: "accepted", runId, childSessionKey }` را برمی‌گرداند.
- زمینه sub-agent فقط `AGENTS.md`، `TOOLS.md`، `SOUL.md`، `IDENTITY.md` و `USER.md` را inject می‌کند (نه `MEMORY.md`، `HEARTBEAT.md` یا `BOOTSTRAP.md`).
- حداکثر عمق تودرتوسازی ۵ است (بازه `maxSpawnDepth`: ۱ تا ۵). عمق ۲ برای بیشتر موارد استفاده توصیه می‌شود.
- `maxChildrenPerAgent` فرزندان active هر نشست را محدود می‌کند (پیش‌فرض `5`، بازه `1–20`).
