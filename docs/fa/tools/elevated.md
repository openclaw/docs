وقتی یک عامل داخل یک سندباکس اجرا می‌شود، فرمان‌های `exec` آن به محیط سندباکس محدود می‌شوند. **حالت ارتقایافته** به عامل اجازه می‌دهد به‌جای آن از سندباکس خارج شود و فرمان‌ها را خارج از سندباکس اجرا کند، همراه با دروازه‌های تأیید قابل پیکربندی.

<Info>
  حالت ارتقایافته فقط زمانی رفتار را تغییر می‌دهد که عامل **سندباکس‌شده** باشد. برای
  عامل‌های بدون سندباکس، exec از قبل روی میزبان اجرا می‌شود.
</Info>

## دستورالعمل‌ها

حالت ارتقایافته را برای هر نشست با فرمان‌های اسلش کنترل کنید:

| دستورالعمل        | کاری که انجام می‌دهد                                                           |
| ---------------- | ---------------------------------------------------------------------- |
| `/elevated on`   | خارج از سندباکس روی مسیر میزبان پیکربندی‌شده اجرا می‌کند و تأییدها را نگه می‌دارد    |
| `/elevated ask`  | همانند `on` (نام مستعار)                                                   |
| `/elevated full` | خارج از سندباکس روی مسیر میزبان پیکربندی‌شده اجرا می‌کند و تأییدها را رد می‌کند |
| `/elevated off`  | به اجرای محدودشده به سندباکس برمی‌گردد                                   |

همچنین به‌صورت `/elev on|off|ask|full` در دسترس است.

برای دیدن سطح فعلی، `/elevated` را بدون آرگومان بفرستید.

## نحوه کار

<Steps>
  <Step title="بررسی دسترس‌پذیری">
    حالت ارتقایافته باید در پیکربندی فعال باشد و فرستنده باید در فهرست مجاز باشد:

    ```json5
    {
      tools: {
        elevated: {
          enabled: true,
          allowFrom: {
            discord: ["user-id-123"],
            whatsapp: ["+15555550123"],
          },
        },
      },
    }
    ```

  </Step>

  <Step title="تنظیم سطح">
    برای تنظیم پیش‌فرض نشست، یک پیام فقط شامل دستورالعمل بفرستید:

    ```
    /elevated full
    ```

    یا آن را به‌صورت درون‌خطی استفاده کنید (فقط برای همان پیام اعمال می‌شود):

    ```
    /elevated on run the deployment script
    ```

  </Step>

  <Step title="فرمان‌ها خارج از سندباکس اجرا می‌شوند">
    وقتی حالت ارتقایافته فعال باشد، فراخوانی‌های `exec` از سندباکس خارج می‌شوند. میزبان مؤثر به‌طور پیش‌فرض
    `gateway` است، یا وقتی هدف exec پیکربندی‌شده/نشست
    `node` باشد، `node` است. در حالت `full`، تأییدهای exec رد می‌شوند. در حالت `on`/`ask`،
    قوانین تأیید پیکربندی‌شده همچنان اعمال می‌شوند.
  </Step>
</Steps>

## ترتیب حل‌وفصل

1. **دستورالعمل درون‌خطی** در پیام (فقط برای همان پیام اعمال می‌شود)
2. **بازنویسی نشست** (با ارسال یک پیام فقط شامل دستورالعمل تنظیم می‌شود)
3. **پیش‌فرض سراسری** (`agents.defaults.elevatedDefault` در پیکربندی)

## دسترس‌پذیری و فهرست‌های مجاز

- **دروازه سراسری**: `tools.elevated.enabled` (باید `true` باشد)
- **فهرست مجاز فرستنده**: `tools.elevated.allowFrom` با فهرست‌های جداگانه برای هر کانال
- **دروازه هر عامل**: `agents.list[].tools.elevated.enabled` (فقط می‌تواند محدودتر کند)
- **فهرست مجاز هر عامل**: `agents.list[].tools.elevated.allowFrom` (فرستنده باید هم با سراسری و هم با هر عامل مطابقت داشته باشد)
- **جایگزین Discord**: اگر `tools.elevated.allowFrom.discord` حذف شده باشد، `channels.discord.allowFrom` به‌عنوان جایگزین استفاده می‌شود
- **همه دروازه‌ها باید عبور کنند**؛ در غیر این صورت حالت ارتقایافته ناموجود در نظر گرفته می‌شود

قالب‌های ورودی فهرست مجاز:

| پیشوند                  | مطابقت می‌دهد با                         |
| ----------------------- | ------------------------------- |
| (هیچ‌کدام)                  | شناسه فرستنده، E.164، یا فیلد From |
| `name:`                 | نام نمایشی فرستنده             |
| `username:`             | نام کاربری فرستنده                 |
| `tag:`                  | برچسب فرستنده                      |
| `id:`, `from:`, `e164:` | هدف‌گیری هویت صریح     |

## حالت ارتقایافته چه چیزهایی را کنترل نمی‌کند

- **سیاست ابزار**: اگر `exec` توسط سیاست ابزار رد شود، حالت ارتقایافته نمی‌تواند آن را نادیده بگیرد
- **سیاست انتخاب میزبان**: حالت ارتقایافته `auto` را به یک بازنویسی آزاد بین‌میزبانی تبدیل نمی‌کند. این حالت از قوانین هدف exec پیکربندی‌شده/نشست استفاده می‌کند و فقط وقتی هدف از قبل `node` باشد، `node` را انتخاب می‌کند.
- **جدا از `/exec`**: دستورالعمل `/exec` پیش‌فرض‌های exec هر نشست را برای فرستنده‌های مجاز تنظیم می‌کند و به حالت ارتقایافته نیاز ندارد
