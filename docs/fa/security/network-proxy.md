# پراکسی شبکه

OpenClaw می‌تواند ترافیک HTTP و WebSocket زمان اجرا را از طریق یک پراکسی پیش‌روی مدیریت‌شده توسط اپراتور مسیریابی کند. این یک لایه دفاعی اختیاری برای استقرارهایی است که کنترل متمرکز خروجی، محافظت قوی‌تر در برابر SSRF و قابلیت ممیزی بهتر شبکه می‌خواهند.

OpenClaw هیچ پراکسی‌ای را همراه خود ارائه، دانلود، شروع، پیکربندی یا تأیید نمی‌کند. شما فناوری پراکسی متناسب با محیط خود را اجرا می‌کنید و OpenClaw کلاینت‌های عادی HTTP و WebSocket محلیِ فرایند را از طریق آن مسیریابی می‌کند.

## چرا از پراکسی استفاده کنیم؟

پراکسی به اپراتورها یک نقطه کنترل شبکه برای ترافیک خروجی HTTP و WebSocket می‌دهد. این موضوع حتی خارج از سخت‌سازی SSRF هم می‌تواند مفید باشد:

- سیاست متمرکز: به‌جای تکیه بر اینکه هر محل فراخوانی HTTP در برنامه قوانین شبکه را درست اعمال کند، یک سیاست خروجی واحد را نگهداری کنید.
- بررسی‌های زمان اتصال: مقصد را پس از تفکیک DNS و بلافاصله پیش از اینکه پراکسی اتصال بالادستی را باز کند ارزیابی کنید.
- دفاع در برابر بازاتصال DNS: فاصله بین بررسی DNS در سطح برنامه و اتصال خروجی واقعی را کاهش دهید.
- پوشش گسترده‌تر JavaScript: کلاینت‌های معمولی `fetch`، `node:http`، `node:https`، WebSocket، axios، got، node-fetch و موارد مشابه را از یک مسیر مشترک عبور دهید.
- قابلیت ممیزی: مقصدهای مجاز و ردشده را در مرز خروجی ثبت کنید.
- کنترل عملیاتی: قوانین مقصد، بخش‌بندی شبکه، محدودیت نرخ یا فهرست‌های مجاز خروجی را بدون بازسازی OpenClaw اعمال کنید.

مسیریابی پراکسی یک حفاظ در سطح فرایند برای خروجی عادی HTTP و WebSocket است. این مسیرِ fail-closed را برای مسیریابی کلاینت‌های پشتیبانی‌شده HTTP در JavaScript از طریق پراکسی فیلترکننده خود اپراتورها فراهم می‌کند، اما sandbox شبکه در سطح سیستم‌عامل نیست و باعث نمی‌شود OpenClaw سیاست مقصد پراکسی را تأیید کند.

## OpenClaw چگونه ترافیک را مسیریابی می‌کند

وقتی `proxy.enabled=true` باشد و یک URL پراکسی پیکربندی شده باشد، فرایندهای محافظت‌شده زمان اجرا مانند `openclaw gateway run`، `openclaw node run` و `openclaw agent --local` خروجی عادی HTTP و WebSocket را از طریق پراکسی پیکربندی‌شده مسیریابی می‌کنند:

```text
OpenClaw process
  fetch                  -> operator-managed filtering proxy -> public internet
  node:http and https    -> operator-managed filtering proxy -> public internet
  WebSocket clients      -> operator-managed filtering proxy -> public internet
```

قرارداد عمومی، رفتار مسیریابی است، نه hookهای داخلی Node که برای پیاده‌سازی آن استفاده می‌شوند. کلاینت‌های WebSocket صفحه کنترل OpenClaw Gateway برای ترافیک RPC مربوط به local loopback Gateway، وقتی URL مربوط به Gateway از `localhost` یا یک IP لوپ‌بک صریح مانند `127.0.0.1` یا `[::1]` استفاده می‌کند، از یک مسیر مستقیم محدود استفاده می‌کنند. این مسیر صفحه کنترل باید حتی زمانی که پراکسی اپراتور مقصدهای لوپ‌بک را مسدود می‌کند بتواند به Gatewayهای لوپ‌بک دسترسی داشته باشد. درخواست‌های عادی HTTP و WebSocket زمان اجرا همچنان از پراکسی پیکربندی‌شده استفاده می‌کنند.

در داخل، OpenClaw برای این قابلیت از دو hook مسیریابی در سطح فرایند استفاده می‌کند:

- مسیریابی dispatcher مربوط به Undici، `fetch`، کلاینت‌های مبتنی بر undici و transportهایی را پوشش می‌دهد که dispatcher اختصاصی undici خودشان را ارائه می‌کنند.
- مسیریابی `global-agent` فراخوان‌های هسته Node یعنی `node:http` و `node:https` را پوشش می‌دهد، از جمله بسیاری از کتابخانه‌هایی که روی `http.request`، `https.request`، `http.get` و `https.get` ساخته شده‌اند. حالت پراکسی مدیریت‌شده آن agent سراسری را اجباری می‌کند تا agentهای صریح Node HTTP به‌طور تصادفی پراکسی اپراتور را دور نزنند.

برخی Pluginها مالک transportهای سفارشی هستند که حتی با وجود مسیریابی در سطح فرایند، به اتصال صریح پراکسی نیاز دارند. برای مثال، transport مربوط به Bot API در Telegram از dispatcher اختصاصی HTTP/1 undici خودش استفاده می‌کند و بنابراین env پراکسی فرایند را به‌همراه fallback مدیریت‌شده `OPENCLAW_PROXY_URL` در آن مسیر transport ویژه مالک رعایت می‌کند.

خود URL پراکسی باید از `http://` استفاده کند. مقصدهای HTTPS همچنان از طریق پراکسی با HTTP `CONNECT` پشتیبانی می‌شوند؛ این فقط به این معناست که OpenClaw انتظار یک listener ساده پراکسی پیش‌روی HTTP مانند `http://127.0.0.1:3128` را دارد.

هنگام فعال بودن پراکسی، OpenClaw مقادیر `no_proxy`، `NO_PROXY` و `GLOBAL_AGENT_NO_PROXY` را پاک می‌کند. این فهرست‌های دورزدن مبتنی بر مقصد هستند، بنابراین باقی ماندن `localhost` یا `127.0.0.1` در آن‌ها باعث می‌شود مقصدهای پرخطر SSRF پراکسی فیلترکننده را دور بزنند.

در زمان خاموش‌سازی، OpenClaw محیط پراکسی قبلی را بازمی‌گرداند و وضعیت مسیریابی کش‌شده فرایند را بازنشانی می‌کند.

## اصطلاحات مرتبط با پراکسی

- `proxy.enabled` / `proxy.proxyUrl`: مسیریابی پراکسی پیش‌روی خروجی برای خروجی زمان اجرای OpenClaw. این صفحه این قابلیت را مستند می‌کند.
- `gateway.auth.mode: "trusted-proxy"`: احراز هویت ورودی reverse-proxy آگاه از هویت برای دسترسی به Gateway. [احراز هویت پراکسی مورد اعتماد](/fa/gateway/trusted-proxy-auth) را ببینید.
- `openclaw proxy`: پراکسی اشکال‌زدایی محلی و بازرس capture برای توسعه و پشتیبانی. [openclaw proxy](/fa/cli/proxy) را ببینید.
- `tools.web.fetch.useTrustedEnvProxy`: فعال‌سازی اختیاری برای `web_fetch` تا اجازه دهد پراکسی env مربوط به HTTP(S) که توسط اپراتور کنترل می‌شود DNS را resolve کند، در حالی که pinning سخت‌گیرانه پیش‌فرض DNS و سیاست hostname حفظ می‌شود. [واکشی وب](/fa/tools/web-fetch#trusted-env-proxy) را ببینید.
- تنظیمات پراکسی ویژه channel یا provider: overrideهای ویژه مالک برای یک transport خاص. وقتی هدف کنترل متمرکز خروجی در کل زمان اجراست، پراکسی شبکه مدیریت‌شده را ترجیح دهید.

## پیکربندی

```yaml
proxy:
  enabled: true
  proxyUrl: http://127.0.0.1:3128
```

همچنین می‌توانید URL را از طریق محیط ارائه کنید، در حالی که `proxy.enabled=true` را در پیکربندی نگه می‌دارید:

```bash
OPENCLAW_PROXY_URL=http://127.0.0.1:3128 openclaw gateway run
```

`proxy.proxyUrl` بر `OPENCLAW_PROXY_URL` اولویت دارد.

### حالت لوپ‌بک Gateway

کلاینت‌های محلی صفحه کنترل Gateway معمولاً به یک WebSocket لوپ‌بک مانند `ws://127.0.0.1:18789` وصل می‌شوند. از `proxy.loopbackMode` استفاده کنید تا انتخاب کنید آن ترافیک هنگام فعال بودن پراکسی مدیریت‌شده چگونه رفتار کند:

```yaml
proxy:
  enabled: true
  proxyUrl: http://127.0.0.1:3128
  loopbackMode: gateway-only # gateway-only, proxy, or block
```

- `gateway-only` (پیش‌فرض): OpenClaw مرجع لوپ‌بک Gateway را در کنترلر فعال `NO_PROXY` مربوط به `global-agent` ثبت می‌کند تا ترافیک WebSocket محلی Gateway بتواند مستقیم متصل شود. پورت‌های سفارشی لوپ‌بک Gateway کار می‌کنند چون میزبان و پورت URL فعال Gateway ثبت می‌شوند.
- `proxy`: OpenClaw مرجع لوپ‌بک Gateway مربوط به `NO_PROXY` را ثبت نمی‌کند، بنابراین ترافیک محلی Gateway از طریق پراکسی مدیریت‌شده ارسال می‌شود. اگر پراکسی دوردست باشد، باید مسیریابی ویژه‌ای برای سرویس لوپ‌بک میزبان OpenClaw فراهم کند، مانند نگاشت آن به یک hostname، IP یا تونل قابل‌دسترسی برای پراکسی. پراکسی‌های دوردست استاندارد `127.0.0.1` و `localhost` را از میزبان پراکسی resolve می‌کنند، نه از میزبان OpenClaw.
- `block`: OpenClaw اتصال‌های صفحه کنترل لوپ‌بک Gateway را پیش از باز کردن socket رد می‌کند.

اگر `enabled=true` باشد اما هیچ URL معتبر پراکسی پیکربندی نشده باشد، فرمان‌های محافظت‌شده به‌جای fallback به دسترسی مستقیم شبکه، در شروع اجرا شکست می‌خورند.

برای سرویس‌های Gateway مدیریت‌شده که با `openclaw gateway start` شروع می‌شوند، ترجیح دهید URL را در پیکربندی ذخیره کنید:

```bash
openclaw config set proxy.enabled true
openclaw config set proxy.proxyUrl http://127.0.0.1:3128
openclaw gateway install --force
openclaw gateway start
```

fallback محیط برای اجراهای foreground مناسب‌تر است. اگر آن را با یک سرویس نصب‌شده استفاده می‌کنید، `OPENCLAW_PROXY_URL` را در محیط پایدار سرویس، مانند `$OPENCLAW_STATE_DIR/.env` یا `~/.openclaw/.env` قرار دهید، سپس سرویس را دوباره نصب کنید تا launchd، systemd یا Scheduled Tasks، Gateway را با آن مقدار شروع کند.

برای فرمان‌های `openclaw --container ...`، وقتی `OPENCLAW_PROXY_URL` تنظیم شده باشد، OpenClaw آن را به CLI فرزندِ هدف‌گذاری‌شده برای container منتقل می‌کند. URL باید از داخل container قابل‌دسترسی باشد؛ `127.0.0.1` به خود container اشاره می‌کند، نه میزبان. OpenClaw URLهای پراکسی لوپ‌بک را برای فرمان‌های هدف‌گذاری‌شده به container رد می‌کند، مگر اینکه آن بررسی ایمنی را صراحتاً override کنید.

## الزامات پراکسی

سیاست پراکسی مرز امنیتی است. OpenClaw نمی‌تواند تأیید کند که پراکسی مقصدهای درست را مسدود می‌کند.

پراکسی را طوری پیکربندی کنید که:

- فقط به لوپ‌بک یا یک interface خصوصی مورد اعتماد bind شود.
- دسترسی را محدود کند تا فقط فرایند، میزبان، container یا حساب سرویس OpenClaw بتواند از آن استفاده کند.
- مقصدها را خودش resolve کند و IPهای مقصد را پس از تفکیک DNS مسدود کند.
- سیاست را در زمان اتصال، هم برای درخواست‌های ساده HTTP و هم برای تونل‌های HTTPS `CONNECT` اعمال کند.
- دورزدن‌های مبتنی بر مقصد را برای بازه‌های لوپ‌بک، خصوصی، link-local، metadata، multicast، reserved یا documentation رد کند.
- از فهرست‌های مجاز hostname خودداری کند، مگر اینکه به مسیر تفکیک DNS کاملاً اعتماد داشته باشید.
- مقصد، تصمیم، وضعیت و دلیل را بدون ثبت بدنه درخواست‌ها، headerهای authorization، cookieها یا سایر اسرار ثبت کند.
- سیاست پراکسی را زیر کنترل نسخه نگه دارد و تغییرات را مانند پیکربندی حساس امنیتی بازبینی کند.

## مقصدهای پیشنهادی برای مسدودسازی

از این denylist به‌عنوان نقطه شروع برای هر پراکسی پیش‌رو، firewall یا سیاست خروجی استفاده کنید.

منطق classifier در سطح برنامه OpenClaw در `src/infra/net/ssrf.ts` و `src/shared/net/ip.ts` قرار دارد. hookهای parity مرتبط عبارت‌اند از `BLOCKED_HOSTNAMES`، `BLOCKED_IPV4_SPECIAL_USE_RANGES`، `BLOCKED_IPV6_SPECIAL_USE_RANGES`، `RFC2544_BENCHMARK_PREFIX` و مدیریت sentinel توکار IPv4 برای NAT64، 6to4، Teredo، ISATAP و فرم‌های IPv4-mapped. این فایل‌ها هنگام نگهداری سیاست پراکسی خارجی مرجع‌های مفیدی هستند، اما OpenClaw این قوانین را به‌طور خودکار به پراکسی شما export یا در آن enforce نمی‌کند.

| ------------------------------------------------------------------------------------ | ---------------------------------------------------- |

| `::1/128`                                                                            | لوپ‌بک IPv6                                         |

## اعتبارسنجی

پراکسی را از همان میزبان، container یا حساب سرویسی اعتبارسنجی کنید که OpenClaw را اجرا می‌کند:

```bash
openclaw proxy validate --proxy-url http://127.0.0.1:3128
```

به‌طور پیش‌فرض، وقتی مقصدهای سفارشی ارائه نشده باشند، فرمان بررسی می‌کند که `https://example.com/` موفق شود و یک کاناری موقت حلقه‌بازگشت راه‌اندازی می‌کند که پراکسی نباید به آن دسترسی پیدا کند. بررسی ردشدۀ پیش‌فرض زمانی موفق است که پراکسی یک پاسخ رد غیر 2xx برگرداند یا کاناری را با خطای انتقال مسدود کند؛ اگر یک پاسخ موفق به کاناری برسد، شکست می‌خورد. اگر هیچ پراکسی‌ای فعال و پیکربندی نشده باشد، اعتبارسنجی یک مشکل پیکربندی گزارش می‌کند؛ برای یک پیش‌بررسی یک‌باره پیش از تغییر پیکربندی از `--proxy-url` استفاده کنید. برای آزمودن انتظارهای مخصوص استقرار از `--allowed-url` و `--denied-url` استفاده کنید. `--apns-reachable` را اضافه کنید تا همچنین بررسی شود که تحویل مستقیم HTTP/2 مربوط به APNs می‌تواند از طریق پراکسی یک تونل CONNECT باز کند و یک پاسخ APNs sandbox دریافت کند؛ این کاوش از یک توکن ارائه‌دهندۀ عمداً نامعتبر استفاده می‌کند، بنابراین `403 InvalidProviderToken` انتظار می‌رود و به‌عنوان قابل‌دسترسی شمرده می‌شود. مقصدهای ردشدۀ سفارشی fail-closed هستند: هر پاسخ HTTP یعنی مقصد از طریق پراکسی قابل‌دسترسی بوده است، و هر خطای انتقال به‌عنوان نامشخص گزارش می‌شود، چون OpenClaw نمی‌تواند ثابت کند پراکسی یک مبداً قابل‌دسترسی را مسدود کرده است. در صورت شکست اعتبارسنجی، فرمان با کد 1 خارج می‌شود.

برای خودکارسازی از `--json` استفاده کنید. خروجی JSON شامل نتیجۀ کلی، منبع پیکربندی مؤثر پراکسی، هرگونه خطای پیکربندی، و هر بررسی مقصد است. اعتبارنامه‌های URL پراکسی در خروجی متنی و JSON پوشانده می‌شوند:

```json
{
  "ok": true,
  "config": {
    "enabled": true,
    "proxyUrl": "http://127.0.0.1:3128/",
    "source": "override",
    "errors": []
  },
  "checks": [
    {
      "kind": "allowed",
      "url": "https://example.com/",
      "ok": true,
      "status": 200
    },
    {
      "kind": "apns",
      "url": "https://api.sandbox.push.apple.com",
      "ok": true,
      "status": 403
    }
  ]
}
```

همچنین می‌توانید با `curl` به‌صورت دستی اعتبارسنجی کنید:

```bash
curl -x http://127.0.0.1:3128 https://example.com/
curl -x http://127.0.0.1:3128 http://127.0.0.1/
curl -x http://127.0.0.1:3128 http://169.254.169.254/
```

درخواست عمومی باید موفق شود. درخواست‌های حلقه‌بازگشت و metadata باید توسط پراکسی مسدود شوند. برای `openclaw proxy validate`، کاناری حلقه‌بازگشت داخلی می‌تواند رد پراکسی را از یک مبداً قابل‌دسترسی تشخیص دهد. بررسی‌های سفارشی `--denied-url` آن کاناری را ندارند، بنابراین هم پاسخ‌های HTTP و هم شکست‌های مبهم انتقال را شکست اعتبارسنجی تلقی کنید، مگر اینکه پراکسی شما یک سیگنال رد مخصوص استقرار ارائه کند که بتوانید جداگانه آن را تأیید کنید.

سپس مسیریابی پراکسی OpenClaw را فعال کنید:

```bash
openclaw config set proxy.enabled true
openclaw config set proxy.proxyUrl http://127.0.0.1:3128
openclaw gateway run
```

یا تنظیم کنید:

```yaml
proxy:
  enabled: true
  proxyUrl: http://127.0.0.1:3128
```

## محدودیت‌ها

- پراکسی پوشش را برای کلاینت‌های HTTP و WebSocket جاوااسکریپتِ محلیِ فرایند بهبود می‌دهد، اما sandbox شبکۀ سطح سیستم‌عامل نیست.
- ترافیک صفحه کنترل حلقه‌بازگشت Gateway به‌طور پیش‌فرض از طریق `proxy.loopbackMode: "gateway-only"` مستقیماً به مسیر local bypass می‌رود. OpenClaw این bypass را با ثبت authority حلقه‌بازگشت Gateway فعال در کنترل‌کنندۀ مدیریت‌شدۀ `global-agent` `NO_PROXY` پیاده‌سازی می‌کند. اپراتورها می‌توانند `proxy.loopbackMode: "proxy"` را تنظیم کنند تا ترافیک حلقه‌بازگشت Gateway از طریق پراکسی مدیریت‌شده ارسال شود، یا `proxy.loopbackMode: "block"` را تنظیم کنند تا اتصال‌های حلقه‌بازگشت Gateway رد شوند. برای هشدار پراکسی راه‌دور، [حالت حلقه‌بازگشت Gateway](#gateway-loopback-mode) را ببینید.
- سوکت‌های خام `net`، `tls` و `http2`، افزونه‌های native، و فرایندهای فرزند غیر OpenClaw ممکن است مسیریابی پراکسی در سطح Node را دور بزنند، مگر اینکه متغیرهای محیطی پراکسی را به ارث ببرند و رعایت کنند. CLIهای فرزند fork شدۀ OpenClaw، URL پراکسی مدیریت‌شده و وضعیت `proxy.loopbackMode` را به ارث می‌برند.
- IRC یک کانال خام TCP/TLS بیرون از مسیریابی پراکسی forward مدیریت‌شده توسط اپراتور است. در استقرارهایی که نیاز دارند همۀ خروجی‌ها از آن پراکسی forward عبور کنند، مگر اینکه خروجی مستقیم IRC صراحتاً تأیید شده باشد، `channels.irc.enabled=false` را تنظیم کنید.
- پراکسی اشکال‌زدایی local ابزار تشخیصی است و forward مستقیم بالادستی آن برای درخواست‌های پراکسی و تونل‌های CONNECT، تا وقتی حالت پراکسی مدیریت‌شده فعال است، به‌طور پیش‌فرض غیرفعال است؛ forward مستقیم را فقط برای تشخیص‌های local تأییدشده فعال کنید.
- WebUIهای local کاربر و سرورهای مدل local باید در صورت نیاز در سیاست پراکسی اپراتور allowlist شوند؛ OpenClaw برای آن‌ها bypass عمومی شبکۀ local ارائه نمی‌کند.
- bypass پراکسی صفحه کنترل Gateway عمداً به `localhost` و URLهای IP حلقه‌بازگشت literal محدود شده است. برای اتصال‌های مستقیم local صفحه کنترل Gateway از `ws://127.0.0.1:18789`، `ws://[::1]:18789` یا `ws://localhost:18789` استفاده کنید؛ نام میزبان‌های دیگر مثل ترافیک معمول مبتنی بر نام میزبان مسیریابی می‌شوند.
- OpenClaw سیاست پراکسی شما را بازرسی، آزمون یا تأیید نمی‌کند.
- تغییرات سیاست پراکسی را تغییرات عملیاتی حساس از نظر امنیتی تلقی کنید.
