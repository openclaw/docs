# کانال‌ها و مسیریابی

OpenClaw پاسخ‌ها را **به همان کانالی برمی‌گرداند که پیام از آن آمده است**. مدل کانال را انتخاب نمی‌کند؛ مسیریابی قطعی است و توسط پیکربندی میزبان کنترل می‌شود.

## اصطلاحات کلیدی

- **کانال**: `telegram`, `whatsapp`, `discord`, `irc`, `googlechat`, `slack`, `signal`, `imessage`, `line`، به‌علاوه کانال‌های Plugin. `webchat` کانال داخلی رابط کاربری WebChat است و یک کانال خروجی قابل پیکربندی نیست.
- **AccountId**: نمونه حساب به‌ازای هر کانال (وقتی پشتیبانی شود).
- حساب پیش‌فرض اختیاری کانال: `channels.<channel>.defaultAccount` انتخاب می‌کند وقتی مسیر خروجی `accountId` را مشخص نکرده است، کدام حساب استفاده شود.
  - در راه‌اندازی‌های چندحسابی، وقتی دو یا چند حساب پیکربندی شده‌اند، یک پیش‌فرض صریح (`defaultAccount` یا `accounts.default`) تنظیم کنید. بدون آن، مسیریابی جایگزین ممکن است نخستین شناسه حساب نرمال‌شده را انتخاب کند.
- **AgentId**: یک فضای کاری ایزوله + انبار نشست («مغز»).
- **SessionKey**: کلید سطلی که برای ذخیره زمینه و کنترل هم‌زمانی استفاده می‌شود.

## پیشوندهای مقصد خروجی

مقصدهای خروجی صریح ممکن است شامل یک پیشوند ارائه‌دهنده باشند، مانند `telegram:123` یا `tg:123`. هسته این پیشوند را فقط وقتی کانال انتخاب‌شده `last` باشد یا در غیر این صورت حل‌نشده باشد، و فقط وقتی Plugin بارگذاری‌شده آن پیشوند را اعلام کرده باشد، به‌عنوان راهنمای انتخاب کانال در نظر می‌گیرد. اگر فراخواننده از قبل یک کانال صریح انتخاب کرده باشد، پیشوند ارائه‌دهنده باید با همان کانال مطابق باشد؛ ترکیب‌های میان‌کانالی مانند تحویل WhatsApp به `telegram:123` پیش از نرمال‌سازی مقصد مخصوص Plugin شکست می‌خورند.

پیشوندهای نوع مقصد و سرویس مانند `channel:<id>`، `user:<id>`، `room:<id>`، `thread:<id>`، `imessage:<handle>`، و `sms:<number>` در دستور زبان کانال انتخاب‌شده باقی می‌مانند. آن‌ها به‌تنهایی ارائه‌دهنده را انتخاب نمی‌کنند.

## شکل‌های کلید نشست (نمونه‌ها)

پیام‌های مستقیم به‌طور پیش‌فرض در نشست **اصلی** عامل ادغام می‌شوند:

- `agent:<agentId>:<mainKey>` (پیش‌فرض: `agent:main:main`)

حتی وقتی تاریخچه گفت‌وگوی پیام مستقیم با نشست اصلی مشترک است، سیاست sandbox و ابزار برای DMهای خارجی از یک کلید زمان‌اجرای گفت‌وگوی مستقیم مشتق‌شده به‌ازای هر حساب استفاده می‌کند تا پیام‌های آمده از کانال مانند اجراهای نشست اصلی محلی تلقی نشوند.

گروه‌ها و کانال‌ها به‌ازای هر کانال ایزوله باقی می‌مانند:

- گروه‌ها: `agent:<agentId>:<channel>:group:<id>`
- کانال‌ها/اتاق‌ها: `agent:<agentId>:<channel>:channel:<id>`

رشته‌ها:

- رشته‌های Slack/Discord مقدار `:thread:<threadId>` را به کلید پایه اضافه می‌کنند.
- موضوع‌های انجمن Telegram مقدار `:topic:<topicId>` را در کلید گروه جاسازی می‌کنند.

نمونه‌ها:

- `agent:main:telegram:group:-1001234567890:topic:42`
- `agent:main:discord:channel:123456:thread:987654`

## سنجاق‌کردن مسیر DM اصلی

وقتی `session.dmScope` برابر `main` باشد، پیام‌های مستقیم ممکن است یک نشست اصلی مشترک داشته باشند. برای جلوگیری از بازنویسی `lastRoute` نشست توسط DMهای غیرمالک، OpenClaw وقتی همه موارد زیر درست باشند، مالک سنجاق‌شده را از `allowFrom` استنباط می‌کند:

- `allowFrom` دقیقاً یک ورودی غیر wildcard دارد.
- ورودی می‌تواند برای آن کانال به یک شناسه فرستنده مشخص نرمال شود.
- فرستنده DM ورودی با آن مالک سنجاق‌شده مطابقت ندارد.

در این حالت عدم تطابق، OpenClaw همچنان فراداده نشست ورودی را ثبت می‌کند، اما از به‌روزرسانی `lastRoute` نشست اصلی صرف‌نظر می‌کند.

## ثبت ورودی محافظت‌شده

Pluginهای کانال می‌توانند وقتی یک مسیر محافظت‌شده نباید نشست OpenClaw جدیدی ایجاد کند، رکورد نشست ورودی را با `createIfMissing: false` علامت‌گذاری کنند. در این حالت، OpenClaw ممکن است فراداده و `lastRoute` را برای یک نشست موجود به‌روزرسانی کند، اما فقط به این دلیل که پیامی مشاهده شده است، یک ورودی نشست فقط-مسیر ایجاد نمی‌کند.

## قواعد مسیریابی (نحوه انتخاب عامل)

مسیریابی برای هر پیام ورودی **یک عامل** انتخاب می‌کند:

1. **تطابق دقیق همتا** (`bindings` با `peer.kind` + `peer.id`).
2. **تطابق همتای والد** (وراثت رشته).
3. **تطابق guild + نقش‌ها** (Discord) از طریق `guildId` + `roles`.
4. **تطابق guild** (Discord) از طریق `guildId`.
5. **تطابق تیم** (Slack) از طریق `teamId`.
6. **تطابق حساب** (`accountId` روی کانال).
7. **تطابق کانال** (هر حساب روی آن کانال، `accountId: "*"`).
8. **عامل پیش‌فرض** (`agents.list[].default`، در غیر این صورت نخستین ورودی فهرست، جایگزین به `main`).

وقتی یک binding چند فیلد تطابق (`peer`، `guildId`، `teamId`، `roles`) دارد، **همه فیلدهای ارائه‌شده باید مطابق باشند** تا آن binding اعمال شود.

عامل تطبیق‌یافته تعیین می‌کند کدام فضای کاری و انبار نشست استفاده شود.

## گروه‌های پخش همگانی (اجرای چند عامل)

گروه‌های پخش همگانی به شما اجازه می‌دهند برای همان همتا **چند عامل** را اجرا کنید **وقتی OpenClaw به‌طور معمول پاسخ می‌دهد** (برای مثال: در گروه‌های WhatsApp، پس از گذر از دروازه نام‌بردن/فعال‌سازی).

پیکربندی:

```json5
{
  broadcast: {
    strategy: "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"],
    "+15555550123": ["support", "logger"],
  },
}
```

ببینید: [گروه‌های پخش همگانی](/fa/channels/broadcast-groups).

## نمای کلی پیکربندی

- `agents.list`: تعریف‌های عامل نام‌گذاری‌شده (فضای کاری، مدل، و غیره).
- `bindings`: کانال‌ها/حساب‌ها/همتاهای ورودی را به عامل‌ها نگاشت می‌کند.

نمونه:

```json5
{
  agents: {
    list: [{ id: "support", name: "Support", workspace: "~/.openclaw/workspace-support" }],
  },
  bindings: [
    { match: { channel: "slack", teamId: "T123" }, agentId: "support" },
    { match: { channel: "telegram", peer: { kind: "group", id: "-100123" } }, agentId: "support" },
  ],
}
```

## ذخیره‌سازی نشست

انبارهای نشست زیر پوشه وضعیت قرار دارند (پیش‌فرض `~/.openclaw`):

- `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- رونوشت‌های JSONL کنار انبار قرار دارند

می‌توانید مسیر انبار را از طریق `session.store` و قالب‌بندی `{agentId}` بازنویسی کنید.

کشف نشست Gateway و ACP همچنین انبارهای عامل مبتنی بر دیسک را زیر ریشه پیش‌فرض `agents/` و زیر ریشه‌های قالب‌بندی‌شده `session.store` پویش می‌کند. انبارهای کشف‌شده باید داخل همان ریشه عامل حل‌شده باقی بمانند و از یک فایل معمولی `sessions.json` استفاده کنند. symlinkها و مسیرهای خارج از ریشه نادیده گرفته می‌شوند.

## رفتار WebChat

WebChat به **عامل انتخاب‌شده** متصل می‌شود و به‌طور پیش‌فرض از نشست اصلی عامل استفاده می‌کند. به همین دلیل، WebChat به شما اجازه می‌دهد زمینه میان‌کانالی آن عامل را در یک مکان ببینید.

## زمینه پاسخ

پاسخ‌های ورودی شامل این موارد هستند:

- `ReplyToId`، `ReplyToBody`، و `ReplyToSender` وقتی در دسترس باشند.
- زمینه نقل‌قول‌شده به‌صورت یک بلوک `[Replying to ...]` به `Body` افزوده می‌شود.

این رفتار در همه کانال‌ها سازگار است.
