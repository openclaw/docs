# کانال‌ها و مسیریابی

OpenClaw پاسخ‌ها را **به همان کانالی برمی‌گرداند که پیام از آن آمده است**. مدل کانال را انتخاب نمی‌کند؛ مسیریابی قطعی است و توسط پیکربندی میزبان کنترل می‌شود.

## اصطلاحات کلیدی

- **کانال**: `telegram`, `whatsapp`, `discord`, `irc`, `googlechat`, `slack`, `signal`, `imessage`, `line`، به‌علاوه کانال‌های Plugin. `webchat` کانال داخلی رابط کاربری WebChat است و یک کانال خروجی قابل پیکربندی نیست.
- **AccountId**: نمونه حساب برای هر کانال (وقتی پشتیبانی شود).
- حساب پیش‌فرض اختیاری کانال: `channels.<channel>.defaultAccount` انتخاب می‌کند وقتی مسیر خروجی `accountId` را مشخص نمی‌کند، از کدام حساب استفاده شود.
  - در راه‌اندازی‌های چندحسابی، وقتی دو یا چند حساب پیکربندی شده‌اند، یک پیش‌فرض صریح (`defaultAccount` یا `accounts.default`) تنظیم کنید. بدون آن، مسیریابی fallback ممکن است نخستین شناسه حساب نرمال‌سازی‌شده را انتخاب کند.
- **AgentId**: یک فضای کاری ایزوله + ذخیره‌گاه نشست («مغز»).
- **SessionKey**: کلید سطلی که برای ذخیره زمینه و کنترل هم‌زمانی استفاده می‌شود.

## شکل‌های کلید نشست (نمونه‌ها)

پیام‌های مستقیم به‌طور پیش‌فرض به نشست **اصلی** عامل جمع می‌شوند:

- `agent:<agentId>:<mainKey>` (پیش‌فرض: `agent:main:main`)

حتی وقتی تاریخچه گفت‌وگوی پیام مستقیم با نشست اصلی مشترک است، sandbox و سیاست ابزار برای DMهای خارجی از یک کلید زمان‌اجرای گفت‌وگوی مستقیم مشتق‌شده برای هر حساب استفاده می‌کنند تا پیام‌های منشأگرفته از کانال مانند اجراهای محلی نشست اصلی تلقی نشوند.

گروه‌ها و کانال‌ها برای هر کانال ایزوله می‌مانند:

- گروه‌ها: `agent:<agentId>:<channel>:group:<id>`
- کانال‌ها/اتاق‌ها: `agent:<agentId>:<channel>:channel:<id>`

رشته‌ها:

- رشته‌های Slack/Discord مقدار `:thread:<threadId>` را به کلید پایه اضافه می‌کنند.
- موضوعات انجمن Telegram مقدار `:topic:<topicId>` را در کلید گروه جا می‌دهند.

نمونه‌ها:

- `agent:main:telegram:group:-1001234567890:topic:42`
- `agent:main:discord:channel:123456:thread:987654`

## سنجاق‌کردن مسیر DM اصلی

وقتی `session.dmScope` برابر `main` است، پیام‌های مستقیم ممکن است یک نشست اصلی مشترک داشته باشند. برای جلوگیری از بازنویسی `lastRoute` نشست توسط DMهای غیرمالک، OpenClaw وقتی همه موارد زیر درست باشند، مالک سنجاق‌شده را از `allowFrom` استنباط می‌کند:

- `allowFrom` دقیقاً یک ورودی غیر wildcard دارد.
- ورودی می‌تواند برای آن کانال به یک شناسه فرستنده مشخص نرمال‌سازی شود.
- فرستنده DM ورودی با آن مالک سنجاق‌شده مطابقت ندارد.

در آن حالت عدم تطابق، OpenClaw همچنان فراداده نشست ورودی را ثبت می‌کند، اما از به‌روزرسانی `lastRoute` نشست اصلی صرف‌نظر می‌کند.

## ثبت ورودی محافظت‌شده

Pluginهای کانال می‌توانند وقتی یک مسیر محافظت‌شده نباید نشست جدید OpenClaw ایجاد کند، رکورد نشست ورودی را با `createIfMissing: false` علامت‌گذاری کنند. در آن حالت، OpenClaw ممکن است فراداده و `lastRoute` را برای یک نشست موجود به‌روزرسانی کند، اما صرفاً به‌دلیل مشاهده شدن یک پیام، یک ورودی نشست فقط-مسیر ایجاد نمی‌کند.

## قواعد مسیریابی (نحوه انتخاب عامل)

مسیریابی برای هر پیام ورودی **یک عامل** را انتخاب می‌کند:

1. **تطابق دقیق همتا** (`bindings` با `peer.kind` + `peer.id`).
2. **تطابق همتای والد** (ارث‌بری رشته).
3. **تطابق انجمن + نقش‌ها** (Discord) از طریق `guildId` + `roles`.
4. **تطابق انجمن** (Discord) از طریق `guildId`.
5. **تطابق تیم** (Slack) از طریق `teamId`.
6. **تطابق حساب** (`accountId` روی کانال).
7. **تطابق کانال** (هر حسابی روی آن کانال، `accountId: "*"`).
8. **عامل پیش‌فرض** (`agents.list[].default`، در غیر این صورت نخستین ورودی فهرست، fallback به `main`).

وقتی یک binding چند فیلد تطابق (`peer`, `guildId`, `teamId`, `roles`) دارد، برای اعمال آن binding، **همه فیلدهای ارائه‌شده باید مطابقت داشته باشند**.

عامل تطبیق‌یافته تعیین می‌کند از کدام فضای کاری و ذخیره‌گاه نشست استفاده شود.

## گروه‌های پخش همگانی (اجرای چند عامل)

گروه‌های پخش همگانی اجازه می‌دهند برای یک همتا **چند عامل** را اجرا کنید، **وقتی OpenClaw معمولاً پاسخ می‌دهد** (برای مثال: در گروه‌های WhatsApp، پس از gating مبتنی بر اشاره/فعال‌سازی).

پیکربندی:

```json5
{
  broadcast: {
    strategy: "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"],
    "+15555550123": ["support", "logger"],
  },
}
```

ببینید: [گروه‌های پخش همگانی](/fa/channels/broadcast-groups).

## نمای کلی پیکربندی

- `agents.list`: تعریف‌های عامل نام‌گذاری‌شده (فضای کاری، مدل و غیره).
- `bindings`: کانال‌ها/حساب‌ها/همتاهای ورودی را به عامل‌ها نگاشت می‌کند.

نمونه:

```json5
{
  agents: {
    list: [{ id: "support", name: "Support", workspace: "~/.openclaw/workspace-support" }],
  },
  bindings: [
    { match: { channel: "slack", teamId: "T123" }, agentId: "support" },
    { match: { channel: "telegram", peer: { kind: "group", id: "-100123" } }, agentId: "support" },
  ],
}
```

## ذخیره‌سازی نشست

ذخیره‌گاه‌های نشست زیر دایرکتوری وضعیت قرار دارند (پیش‌فرض `~/.openclaw`):

- `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- رونوشت‌های JSONL کنار ذخیره‌گاه قرار دارند

می‌توانید مسیر ذخیره‌گاه را از طریق `session.store` و قالب‌بندی `{agentId}` بازنویسی کنید.

کشف نشست Gateway و ACP همچنین ذخیره‌گاه‌های عامل متکی بر دیسک را زیر ریشه پیش‌فرض `agents/` و زیر ریشه‌های قالب‌بندی‌شده `session.store` اسکن می‌کند. ذخیره‌گاه‌های کشف‌شده باید داخل همان ریشه عامل resolveشده بمانند و از یک فایل معمولی `sessions.json` استفاده کنند. symlinkها و مسیرهای بیرون از ریشه نادیده گرفته می‌شوند.

## رفتار WebChat

WebChat به **عامل انتخاب‌شده** متصل می‌شود و به‌طور پیش‌فرض به نشست اصلی عامل می‌رود. به همین دلیل، WebChat اجازه می‌دهد زمینه بین‌کانالی آن عامل را در یکجا ببینید.

## زمینه پاسخ

پاسخ‌های ورودی شامل موارد زیر هستند:

- `ReplyToId`، `ReplyToBody` و `ReplyToSender` در صورت موجود بودن.
- زمینه نقل‌قول‌شده به‌صورت یک بلوک `[Replying to ...]` به `Body` اضافه می‌شود.

این رفتار در همه کانال‌ها یکسان است.
