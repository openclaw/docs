ClickClack، OpenClaw را از طریق توکن‌های اختصاصی ربات ClickClack به یک فضای کاری ClickClack با میزبانی شخصی متصل می‌کند.

از این روش زمانی استفاده کنید که می‌خواهید یک عامل OpenClaw به‌عنوان کاربر ربات ClickClack ظاهر شود. ClickClack از ربات‌های سرویس مستقل و ربات‌های متعلق به کاربران پشتیبانی می‌کند؛ ربات‌های متعلق به کاربران یک `owner_user_id` را حفظ می‌کنند و فقط محدوده‌های دسترسی توکنی را دریافت می‌کنند که اعطا می‌کنید.

## راه‌اندازی سریع

در ClickClack، **Workspace settings → Integrations → OpenClaw** را باز کنید، یک
ربات بسازید و توکن آن را کپی کنید. سپس کانال را پیکربندی کنید:

```bash
openclaw channels add clickclack --base-url https://clickclack.example.com --token ccb_... --workspace default
```

`workspace` شناسه فضای کاری (`wsp_...`)، نامک یا نام نمایشی را می‌پذیرد.
`channels add` پس از ذخیره، سرور، توکن و فضای کاری را تأیید می‌کند و سپس
گزارش می‌دهد که آیا Gateway در حال اجرا حساب جدید را دریافت کرده است یا نه. اگر OpenClaw
از قبل در حال اجرا باشد، ClickClack به‌طور خودکار متصل می‌شود و نیازی به فرمان
دوم نیست. در غیر این صورت، آن را با فرمان زیر راه‌اندازی کنید:

```bash
openclaw gateway
```

برای راه‌اندازی هدایت‌شده، فرمان زیر را اجرا کنید:

```bash
openclaw onboard
```

ClickClack را انتخاب کنید و سپس هنگام درخواست، نشانی URL سرور، توکن ربات و فضای کاری را
وارد کنید. راه‌اندازی هدایت‌شده پس از ذخیره، سرور، توکن و فضای کاری را بررسی می‌کند؛
ناموفق‌بودن بررسی باعث حذف پیکربندی نمی‌شود.

### روش جایگزین: توکن مبتنی بر متغیر محیطی

حساب پیش‌فرض می‌تواند به‌جای ذخیره توکن
در پیکربندی، `CLICKCLACK_BOT_TOKEN` را بخواند:

```bash
export CLICKCLACK_BOT_TOKEN="ccb_..."
openclaw channels add clickclack --base-url https://clickclack.example.com --workspace default --use-env
openclaw gateway
```

حساب‌های نام‌گذاری‌شده باید از توکن پیکربندی‌شده یا فایل توکن استفاده کنند؛ متغیر محیطی
مشترک عمداً فقط به حساب پیش‌فرض محدود شده است.

### مرجع JSON5

ساختار پیکربندی معادل چنین است:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      defaultTo: "channel:general",
    },
  },
}
```

یک حساب فقط زمانی پیکربندی‌شده محسوب می‌شود که `baseUrl`، یک منبع توکن و
`workspace` همگی تنظیم شده باشند. منبع توکن برای حساب پیش‌فرض می‌تواند `token`، `tokenFile` یا
`CLICKCLACK_BOT_TOKEN` باشد. `workspace` شناسه فضای کاری
(`wsp_...`)، نامک یا نام را می‌پذیرد؛ Gateway هنگام راه‌اندازی آن را به شناسه تبدیل می‌کند.

### کلیدهای پیکربندی حساب

| کلید                     | پیش‌فرض             | توضیحات                                                                                   |
| ----------------------- | ------------------- | --------------------------------------------------------------------------------------- |
| `baseUrl`               | هیچ‌کدام (الزامی)     | نشانی URL سرور ClickClack.                                                                  |
| `token`                 | هیچ‌کدام                | توکن ربات به‌صورت رشته ساده یا ارجاع محرمانه (`source: "env" \| "file" \| "exec"`).        |
| `tokenFile`             | هیچ‌کدام                | مسیر فایل توکن ربات؛ بر `token` اولویت دارد.                                |
| `workspace`             | هیچ‌کدام (الزامی)     | شناسه، نامک یا نام فضای کاری.                                                            |
| `replyMode`             | `"agent"`           | `"agent"` خط لوله کامل عامل را اجرا می‌کند؛ `"model"` تکمیل‌های کوتاه و مستقیم مدل را ارسال می‌کند. |
| `defaultTo`             | `"channel:general"` | مقصدی که وقتی مسیر خروجی مقصدی ارائه نمی‌کند استفاده می‌شود.                                      |
| `allowFrom`             | `["*"]`             | فهرست مجاز شناسه‌های کاربر برای پیام‌های مستقیم و پیام‌های کانال ورودی.                                 |
| `botUserId`             | شناسایی خودکار       | هنگام راه‌اندازی از هویت توکن ربات تعیین می‌شود.                                        |
| `agentId`               | پیش‌فرض مسیر       | پیام‌های ورودی این حساب را به یک عامل مشخص مقید می‌کند.                                       |
| `toolsAllow`            | هیچ‌کدام                | فهرست مجاز ابزارها برای پاسخ‌های عامل از این حساب.                                     |
| `model`، `systemPrompt` | هیچ‌کدام                | توسط تکمیل‌های `replyMode: "model"` استفاده می‌شوند.                                               |
| `commandMenu`           | `true`              | فرمان‌های بومی را در تکمیل خودکار کادر نوشتن ClickClack منتشر می‌کند.                            |
| `reconnectMs`           | `1500`              | تأخیر اتصال مجدد بلادرنگ (100 تا 60000).                                                |

اگر `plugins.allow` فهرستی محدودکننده و غیرخالی باشد، انتخاب صریح
ClickClack در راه‌اندازی کانال یا اجرای `openclaw plugins enable clickclack`
مقدار `clickclack` را به آن فهرست می‌افزاید. نصب هنگام ورود اولیه نیز از همین
رفتار انتخاب صریح استفاده می‌کند. این مسیرها `plugins.deny` یا تنظیم
سراسری `plugins.enabled: false` را نادیده نمی‌گیرند.
اجرای مستقیم `openclaw plugins install @openclaw/clickclack` از خط‌مشی عادی
نصب Plugin پیروی می‌کند و ClickClack را نیز در فهرست مجاز موجود ثبت می‌کند.

## چند ربات

هر حساب اتصال بلادرنگ ClickClack مخصوص خود را باز می‌کند و از توکن ربات خودش استفاده می‌کند.

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      baseUrl: "https://clickclack.example.com",
      defaultAccount: "service",
      accounts: {
        service: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SERVICE_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "channel:general",
          agentId: "service-bot",
        },
        support: {
          token: { source: "env", provider: "default", id: "CLICKCLACK_SUPPORT_BOT_TOKEN" },
          workspace: "default",
          defaultTo: "dm:usr_...",
          agentId: "support-bot",
        },
      },
    },
  },
}
```

## حالت‌های پاسخ

- `replyMode: "agent"` (پیش‌فرض) پیام‌های ورودی را از طریق خط لوله عادی عامل، شامل ثبت نشست و خط‌مشی ابزار، هدایت می‌کند.
- `replyMode: "model"` خط لوله عامل را نادیده می‌گیرد و برای پاسخ‌های مستقیم ربات از `llm.complete` زمان اجرای Plugin استفاده می‌کند که می‌توان شکل آن را با `model` و `systemPrompt` تنظیم کرد. ارائه‌دهنده و مدل انتخاب‌شده بودجه تکمیل را در اختیار دارند.

حالت مدل تکمیل‌ها را با شناسه عامل ربات تعیین‌شده اجرا می‌کند که به بیت اعتماد
صریح `plugins.entries.clickclack.llm.allowAgentIdOverride: true`
نیاز دارد:

```json5
{
  plugins: {
    entries: {
      clickclack: {
        llm: {
          allowAgentIdOverride: true,
        },
      },
    },
  },
}
```

اگر فقط از حالت پاسخ پیش‌فرض `agent` استفاده می‌کنید، بیت اعتماد را غیرفعال نگه دارید؛
در آن حالت نیازی به آن نیست.

## منوی فرمان

هنگام راه‌اندازی Gateway، هر حساب پیکربندی‌شده فرمان‌های بومی OpenClaw را
در ClickClack منتشر می‌کند. این فرمان‌ها در تکمیل خودکار کادر نوشتن با برچسب شناسه
ربات ظاهر می‌شوند. مجموعه منتشرشده در هر راه‌اندازی به‌طور کامل جایگزین می‌شود؛
از جمله زمانی که کاتالوگ فرمان‌های بومی خالی است، منوی قدیمی پاک می‌شود.

همگام‌سازی منوی فرمان به‌طور پیش‌فرض فعال است. برای انصراف، `commandMenu: false` را روی یک حساب
تنظیم کنید:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      commandMenu: false,
    },
  },
}
```

توکن به `commands:write` نیاز دارد. بسته‌های کنونی `bot:write` و
`bot:admin` در ClickClack این محدوده دسترسی را شامل می‌شوند و می‌توان آن را
به‌صورت جداگانه نیز اعطا کرد. ممکن است لازم باشد این محدوده به توکن‌هایی که پیش از معرفی منوهای
فرمان ساخته شده‌اند اضافه شود یا توکن جایگزینی ایجاد شود.

همگام‌سازی به‌صورت نهایت تلاش انجام می‌شود و در هر راه‌اندازی Gateway یک‌بار اجرا می‌شود. نبود محدوده دسترسی یا خرابی
شبکه یک هشدار ثبت می‌کند؛ یک سرور قدیمی‌تر ClickClack که نقطه پایانی را ندارد، رویدادی در
سطح اشکال‌زدایی ثبت می‌کند. هیچ‌یک از این خرابی‌ها مانع راه‌اندازی بلادرنگ نمی‌شوند. منوها در زمان
آفلاین‌بودن عامل همچنان در دسترس می‌مانند و با خروج ربات از
فضای کاری حذف می‌شوند.

این نسخه فقط مشخصات فرمان‌های بومی را منتشر می‌کند. نام‌های مستعار و
کاتالوگ‌های فرمان مربوط به Skills،‏ Plugin یا فرمان‌های سفارشی به منو اضافه نمی‌شوند. اگر یک
نام به‌عنوان فرمان اسلش HTTP نیز ثبت شده باشد، ClickClack ابتدا همان
ثبت را هدایت می‌کند؛ سایر فرمان‌های منو از مسیر عادی تحویل پیام
ادامه می‌یابند.

برای شواهد هم‌بستگی میان سرویس‌ها از حالت `agent` استفاده کنید. برای یک
شناسه معتبر پیام ClickClack با ساختار متعارف `msg_<ulid>`، کانال
شناسه اجرای قطعی OpenClaw یعنی `clickclack:<message-id>` را استخراج می‌کند. سپس هر فراخوانی مدل
در عیب‌یابی به‌صورت `clickclack:<message-id>:model:<n>` قابل مشاهده است؛ هنگامی که آن
نوبت از ClawRouter استفاده می‌کند، همان شناسه فراخوانی مدل به‌صورت `X-Request-ID` ارسال می‌شود.
حالت `model` عیب‌یابی عادی اجرای عامل و نشست را دور می‌زند و بنابراین
برای این مسیر شواهد مناسب نیست.

هنگامی که یک رویداد بلادرنگ حاوی `payload.correlation_id` اعتبارسنجی‌شده باشد،
کانال آن را در واکشی معتبر پیام و
درخواست‌های پاسخ حاصل به ClickClack به‌صورت `X-Correlation-ID` منتقل می‌کند. مقادیر از مجموعه امن
128 نویسه‌ای ClickClack (`A-Z`، `a-z`، `0-9`، `.`، `_`، `:` و `-`) استفاده می‌کنند؛ مقادیر نامعتبر
حذف می‌شوند. این اتصال‌ها فقط شامل شناسه‌ها هستند و هرگز بدنه پیام،
پرامپت‌ها، تکمیل‌ها، اطلاعات احراز هویت یا خروجی ابزار را شامل نمی‌شوند.

## تحویل پایدار رسانه

پاسخ‌های عامل که حاوی رسانه هستند از تحویل پایدار الزامی استفاده می‌کنند. OpenClaw پیش از
نخستین نوشتن در ClickClack، nonceهای پایدار پیام و بارگذاری را برای هر بخش اختصاص می‌دهد؛ بنابراین
تلاش مجدد به‌جای مصرف سهمیه ذخیره‌سازی
یا انتشار موارد تکراری، همان بارگذاری و پیام را دوباره استفاده می‌کند. اگر بارگذاری پس از راه‌اندازی مجدد از قبل وجود داشته باشد،
OpenClaw مسیر محلی اصلی یا نشانی URL رسانه راه دور را دوباره نمی‌خواند.

این قرارداد بازیابی به سرور ClickClack با پشتیبانی از موارد زیر نیاز دارد:

- `GET /api/uploads/by-nonce` همراه با
  `X-ClickClack-Upload-Nonce: supported` در نتایج یافت‌شده و یافت‌نشده.
- `GET /api/messages/by-nonce` همراه با
  `X-ClickClack-Message-Nonce: supported` در نتایج یافت‌شده و یافت‌نشده.
- ایجاد پیام ایدمپوتنت و مرتبط‌سازی پیوست برای همان
  nonce و بارگذاری محدود به مالک.

خطای عمومی 404 یک سرور قدیمی‌تر به‌عنوان مدرکی برای نبود ارسال در نظر گرفته نمی‌شود.
OpenClaw به‌جای به‌خطرانداختن ایجاد مورد تکراری، تحویل را حل‌نشده باقی می‌گذارد؛ پیش از فعال‌کردن
پاسخ‌های عامل تولیدکننده رسانه، ClickClack را به‌روزرسانی کنید.

## ردیف‌های فعالیت عامل

به‌طور پیش‌فرض، کانال ClickClack هنگام اجرای نوبت عامل چیزی نشان نمی‌دهد؛ فقط پاسخ نهایی ثبت می‌شود. برای انتشار ردیف‌های پیام پایدار `agent_commentary` و `agent_tool` در حین اجرای نوبت، `agentActivity: true` را روی یک حساب تنظیم کنید:

```json5
{
  channels: {
    clickclack: {
      enabled: true,
      token: { source: "env", provider: "default", id: "CLICKCLACK_BOT_TOKEN" },
      workspace: "default",
      agentActivity: true,
    },
  },
}
```

الزامات و رفتار:

- **به‌طور پیش‌فرض غیرفعال است.** راه‌اندازی‌های استاندارد و سرورهای قدیمی‌تر ClickClack بدون تغییر باقی می‌مانند.
- **به محدوده دسترسی توکن `agent_activity:write` نیاز دارد.** این محدوده از `bot:write` جداست و از آن به ارث نمی‌رسد؛ پیش از فعال‌کردن این گزینه، توکن ربات را با `--scopes bot:write,agent_activity:write` بسازید (یا این محدوده را به توکن موجود اعطا کنید).
- **کاهش قابلیت با نهایت تلاش.** اگر توکن فاقد `agent_activity:write` باشد یا سرور نوشتن فعالیت‌ها را رد کند، خرابی‌ها ثبت می‌شوند و پاسخ نهایی همچنان به‌طور عادی تحویل داده می‌شود؛ هیچ ردیف فعالیتی ظاهر نمی‌شود.
- ردیف‌ها بر اساس هر نوبت (`turn_id`) گروه‌بندی و ادغام می‌شوند تا هر گام منطقی یک ردیف باشد؛ ردیف‌های ابزار نیز از همان قالب‌بندی پیشرفت Discord/Slack/Telegram استفاده می‌کنند (نام ابزار به‌همراه جزئیات فرمان).
- **فراداده انتساب.** پست‌های نوشته‌شده توسط عامل (ردیف‌های فعالیت و پاسخ نهایی) شامل فیلدهای `author_model` و `author_thinking` هستند که از مدل واقعی استفاده‌شده برای نوبت تعیین می‌شوند (از جمله پس از بازگشت به گزینه جایگزین). سرورهایی که این ستون‌ها را تعریف نمی‌کنند، فیلدهای ناشناخته JSON را نادیده می‌گیرند؛ سرورهایی که آن‌ها را ذخیره می‌کنند می‌توانند برای هر پیام پاسخ دهند «کدام مدل این خط را در چه سطحی از تفکر گفته است».

## مقصدها

- `channel:<name-or-id>` به یک کانال فضای کاری ارسال می‌کند. مقصدهای بدون پیشوند به‌طور پیش‌فرض `channel:` هستند.
- `dm:<user_id>` یک گفت‌وگوی مستقیم با آن کاربر ایجاد می‌کند یا از گفت‌وگوی موجود استفاده می‌کند.
- `thread:<message_id>` در رشته‌ای که از آن پیام آغاز شده است پاسخ می‌دهد.

مقصدهای خروجی صریح می‌توانند پیشوند ارائه‌دهندهٔ `clickclack:` یا `cc:` را نیز داشته باشند.

رسانهٔ خروجی از API بارگذاری ClickClack استفاده می‌کند و سپس بارگذاری ماندگار را
به پیام کانال، پاسخ رشته، یا پیام مستقیم ایجادشده پیوست می‌کند. فایل‌های محلی و
نشانی‌های URL پشتیبانی‌شدهٔ رسانه‌های راه‌دور از خط‌مشی معمول دسترسی به رسانه در OpenClaw پیروی می‌کنند و
برای هر فایل محدودیت 64 MiB دارند. ارسال‌های ماندگارِ صف‌شده برای هر
بارگذاری و بخش پیام از nonceهای جداگانه با دامنهٔ مالک استفاده می‌کنند و سپس برقراری ارتباط پیوست را با همان
اشیا دوباره امتحان می‌کنند. برای قرارداد سرور و رفتار بازیابی به
[تحویل ماندگار رسانه](#durable-media-delivery) مراجعه کنید.

مثال‌ها:

```bash
openclaw message send --channel clickclack --target channel:general --message "hello"
openclaw message send --channel clickclack --target dm:usr_123 --message "hello"
openclaw message send --channel clickclack --target thread:msg_123 --message "following up"
```

## مجوزها

دامنه‌های دسترسی توکن ClickClack توسط API این سرویس اعمال می‌شوند.

- `bot:read`: خواندن داده‌های فضای کاری/کانال/پیام/رشته/پیام مستقیم/بلادرنگ/نمایه.
- `bot:write`: `bot:read` به‌علاوهٔ پیام‌های کانال، پاسخ‌های رشته، پیام‌های مستقیم، بارگذاری‌ها و انتشار منوی فرمان.
- `bot:admin`: `bot:write` به‌علاوهٔ ایجاد کانال.
- `commands:write`: منوی فرمان ربات را منتشر می‌کند. در بسته‌های فعلی `bot:write` و `bot:admin` گنجانده شده و به‌صورت جداگانه نیز قابل اعطا است.
- `agent_activity:write`: ردیف‌های ماندگار فعالیت عامل (`agent_commentary` / `agent_tool`). از `bot:write` یا `bot:admin` به ارث نمی‌رسد؛ فقط زمانی لازم است که `agentActivity: true` تنظیم شده باشد.

OpenClaw برای گفت‌وگوی معمول عامل و همگام‌سازی منوی فرمان فقط به `bot:write` فعلی نیاز دارد. هنگام فعال‌کردن [ردیف‌های فعالیت عامل](#agent-activity-rows)، `agent_activity:write` را اضافه کنید.

## عیب‌یابی

- `ClickClack is not configured for account "<id>"`: برای آن حساب، `baseUrl`، `token` (برای مثال از طریق `CLICKCLACK_BOT_TOKEN`) و `workspace` را تنظیم کنید.
- `ClickClack workspace not found: <value>`: مقدار `workspace` را روی شناسه، نامک، یا نام فضای کاری برگردانده‌شده توسط ClickClack تنظیم کنید.
- پاسخ ورودی دریافت نمی‌شود: تأیید کنید توکن دسترسی خواندن بلادرنگ دارد و توجه داشته باشید که ربات پیام‌های خودش و پیام‌های ربات‌های دیگر را نادیده می‌گیرد.
- ارسال به کانال ناموفق است: بررسی کنید ربات عضو فضای کاری است و `bot:write` را دارد.
- منوی فرمان وجود ندارد: تأیید کنید `commandMenu` برابر با `false` نیست، سرور ClickClack از `PUT /api/bots/self/commands` پشتیبانی می‌کند و توکن `commands:write` را دارد.
