QQ Bot از طریق API رسمی QQ Bot (WebSocket gateway) به OpenClaw متصل می‌شود. این
Plugin از چت خصوصی C2C، @پیام‌های گروهی، و پیام‌های کانال guild همراه با
رسانه‌های غنی (تصاویر، صدا، ویدیو، فایل‌ها) پشتیبانی می‌کند.

وضعیت: Plugin بسته‌بندی‌شده. پیام‌های مستقیم، چت‌های گروهی، کانال‌های guild و
رسانه پشتیبانی می‌شوند. واکنش‌ها و رشته‌ها پشتیبانی نمی‌شوند.

## Plugin بسته‌بندی‌شده

انتشارهای فعلی OpenClaw شامل QQ Bot هستند، بنابراین ساخت‌های بسته‌بندی‌شده معمول
به مرحله جداگانه `openclaw plugins install` نیاز ندارند.

## راه‌اندازی

1. به [QQ Open Platform](https://q.qq.com/) بروید و کد QR را با QQ گوشی خود اسکن کنید تا
   ثبت‌نام / ورود انجام شود.
2. برای ساخت یک بات QQ جدید، روی **Create Bot** کلیک کنید.
3. **AppID** و **AppSecret** را در صفحه تنظیمات بات پیدا و کپی کنید.

> AppSecret به‌صورت متن ساده ذخیره نمی‌شود — اگر بدون ذخیره‌کردن آن از صفحه خارج شوید،
> باید یک مورد جدید دوباره تولید کنید.

4. کانال را اضافه کنید:

```bash
openclaw channels add --channel qqbot --token "AppID:AppSecret"
```

5. Gateway را بازراه‌اندازی کنید.

مسیرهای راه‌اندازی تعاملی:

```bash
openclaw channels add
openclaw configure --section channels
```

## پیکربندی

حداقل پیکربندی:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "YOUR_APP_ID",
      clientSecret: "YOUR_APP_SECRET",
    },
  },
}
```

متغیرهای محیطی حساب پیش‌فرض:

- `QQBOT_APP_ID`
- `QQBOT_CLIENT_SECRET`

AppSecret مبتنی بر فایل:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "YOUR_APP_ID",
      clientSecretFile: "/path/to/qqbot-secret.txt",
    },
  },
}
```

نکات:

- بازگشت به متغیر محیطی فقط برای حساب پیش‌فرض QQ Bot اعمال می‌شود.
- `openclaw channels add --channel qqbot --token-file ...` فقط
  AppSecret را ارائه می‌کند؛ AppID باید از قبل در پیکربندی یا `QQBOT_APP_ID` تنظیم شده باشد.
- `clientSecret` علاوه بر رشته متن ساده، ورودی SecretRef را نیز می‌پذیرد.

### راه‌اندازی چندحسابی

چند QQ bot را زیر یک نمونه OpenClaw اجرا کنید:

```json5
{
  channels: {
    qqbot: {
      enabled: true,
      appId: "111111111",
      clientSecret: "secret-of-bot-1",
      accounts: {
        bot2: {
          enabled: true,
          appId: "222222222",
          clientSecret: "secret-of-bot-2",
        },
      },
    },
  },
}
```

هر حساب اتصال WebSocket خودش را راه‌اندازی می‌کند و حافظه نهان توکن مستقلی
(ایزوله‌شده بر اساس `appId`) نگه می‌دارد.

یک بات دوم را از طریق CLI اضافه کنید:

```bash
openclaw channels add --channel qqbot --account bot2 --token "222222222:secret-of-bot-2"
```

### چت‌های گروهی

پشتیبانی QQ Bot از چت گروهی از OpenIDهای گروه QQ استفاده می‌کند، نه نام‌های نمایشی. بات را
به یک گروه اضافه کنید، سپس آن را منشن کنید یا گروه را طوری پیکربندی کنید که بدون منشن اجرا شود.

```json5
{
  channels: {
    qqbot: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["member_openid"],
      groups: {
        "*": {
          requireMention: true,
          historyLimit: 50,
          toolPolicy: "restricted",
        },
        GROUP_OPENID: {
          name: "Release room",
          requireMention: false,
          ignoreOtherMentions: true,
          historyLimit: 20,
          prompt: "Keep replies short and operational.",
        },
      },
    },
  },
}
```

`groups["*"]` پیش‌فرض‌ها را برای هر گروه تنظیم می‌کند، و یک ورودی مشخص
`groups.GROUP_OPENID` آن پیش‌فرض‌ها را برای یک گروه بازنویسی می‌کند. تنظیمات گروه
شامل موارد زیر است:

- `requireMention`: پیش از پاسخ بات، @mention را الزامی می‌کند. پیش‌فرض: `true`.
- `ignoreOtherMentions`: پیام‌هایی را که فرد دیگری را منشن می‌کنند اما بات را منشن نمی‌کنند حذف می‌کند.
- `historyLimit`: پیام‌های اخیر گروهی بدون منشن را به‌عنوان زمینه برای نوبت منشن‌شده بعدی نگه می‌دارد. برای غیرفعال‌سازی، `0` تنظیم کنید.
- `toolPolicy`: `full`، `restricted`، یا `none` برای ابزارهای محدود به گروه.
- `name`: برچسب دوستانه‌ای که در لاگ‌ها و زمینه گروه استفاده می‌شود.
- `prompt`: اعلان رفتار مخصوص هر گروه که به زمینه عامل افزوده می‌شود.

حالت‌های فعال‌سازی `mention` و `always` هستند. `requireMention: true` به
`mention` نگاشت می‌شود؛ `requireMention: false` به `always` نگاشت می‌شود. بازنویسی فعال‌سازی
در سطح نشست، در صورت وجود، بر پیکربندی مقدم است.

صف ورودی برای هر همتا جداست. همتاهای گروهی سقف صف بزرگ‌تری می‌گیرند، هنگام پر بودن
پیام‌های انسانی را جلوتر از گفت‌وگوی تولیدشده توسط بات نگه می‌دارند، و فوران پیام‌های عادی
گروهی را در یک نوبت منتسب ادغام می‌کنند. فرمان‌های اسلش همچنان یکی‌یکی اجرا می‌شوند.

### صدا (STT / TTS)

پشتیبانی STT و TTS پیکربندی دوسطحی با بازگشت اولویت‌دار دارد:

| تنظیم | مخصوص Plugin                                          | بازگشت چارچوب            |
| ------- | -------------------------------------------------------- | ----------------------------- |
| STT     | `channels.qqbot.stt`                                     | `tools.media.audio.models[0]` |
| TTS     | `channels.qqbot.tts`, `channels.qqbot.accounts.<id>.tts` | `messages.tts`                |

```json5
{
  channels: {
    qqbot: {
      stt: {
        provider: "your-provider",
        model: "your-stt-model",
      },
      tts: {
        provider: "your-provider",
        model: "your-tts-model",
        voice: "your-voice",
      },
      accounts: {
        qq-main: {
          tts: {
            providers: {
              openai: { voice: "shimmer" },
            },
          },
        },
      },
    },
  },
}
```

برای غیرفعال‌سازی هرکدام، `enabled: false` را روی آن تنظیم کنید.
بازنویسی‌های TTS در سطح حساب همان شکل `messages.tts` را دارند و به‌صورت عمیق
روی پیکربندی TTS کانال/سراسری ادغام می‌شوند.

پیوست‌های صوتی ورودی QQ به‌عنوان فراداده رسانه صوتی در اختیار عامل‌ها قرار می‌گیرند، در حالی که
فایل‌های صوتی خام از `MediaPaths` عمومی بیرون نگه داشته می‌شوند. پاسخ‌های متن ساده
`[[audio_as_voice]]` وقتی TTS پیکربندی شده باشد، TTS را سنتز می‌کنند و یک پیام صوتی بومی QQ می‌فرستند.

رفتار بارگذاری/ترنسکد صوت خروجی را نیز می‌توان با
`channels.qqbot.audioFormatPolicy` تنظیم کرد:

- `sttDirectFormats`
- `uploadDirectFormats`
- `transcodeEnabled`

## قالب‌های هدف

| قالب                     | توضیح        |
| -------------------------- | ------------------ |
| `qqbot:c2c:OPENID`         | چت خصوصی (C2C) |
| `qqbot:group:GROUP_OPENID` | چت گروهی         |
| `qqbot:channel:CHANNEL_ID` | کانال guild      |

> هر بات مجموعه OpenIDهای کاربر خودش را دارد. OpenID دریافت‌شده توسط Bot A **نمی‌تواند**
> برای ارسال پیام از طریق Bot B استفاده شود.

## فرمان‌های اسلش

فرمان‌های داخلی که پیش از صف AI رهگیری می‌شوند:

| فرمان        | توضیح                                                                                              |
| -------------- | -------------------------------------------------------------------------------------------------------- |
| `/bot-ping`    | آزمون تأخیر                                                                                             |
| `/bot-version` | نمایش نسخه چارچوب OpenClaw                                                                      |
| `/bot-help`    | فهرست همه فرمان‌ها                                                                                        |
| `/bot-me`      | نمایش شناسه کاربر QQ فرستنده (openid) برای راه‌اندازی `allowFrom`/`groupAllowFrom`                             |
| `/bot-upgrade` | نمایش لینک راهنمای ارتقای QQBot                                                                        |
| `/bot-logs`    | خروجی‌گرفتن از لاگ‌های اخیر Gateway به‌صورت فایل                                                                     |
| `/bot-approve` | تأیید یک اقدام در انتظار QQ Bot (برای مثال، تأیید بارگذاری C2C یا گروه) از طریق جریان بومی. |

برای راهنمای استفاده، به هر فرمان `?` اضافه کنید (برای مثال `/bot-upgrade ?`).

فرمان‌های مدیر (`/bot-me`, `/bot-upgrade`, `/bot-logs`, `/bot-clear-storage`, `/bot-streaming`, `/bot-approve`) فقط در پیام مستقیم قابل استفاده‌اند و نیاز دارند openid فرستنده در یک فهرست `allowFrom` صریح و غیر wildcard باشد. یک wildcard مثل `allowFrom: ["*"]` چت را مجاز می‌کند اما دسترسی به فرمان مدیر نمی‌دهد. پیام‌های گروهی ابتدا با `groupAllowFrom` تطبیق داده می‌شوند و سپس به `allowFrom` برمی‌گردند. اجرای یک فرمان مدیر در گروه به‌جای حذف بی‌صدا، یک راهنما برمی‌گرداند.

## معماری موتور

QQ Bot به‌صورت یک موتور خودبسنده داخل Plugin عرضه می‌شود:

- هر حساب یک پشته منبع ایزوله (اتصال WebSocket، کلاینت API، حافظه نهان توکن، ریشه ذخیره‌سازی رسانه) دارد که با `appId` کلیدگذاری شده است. حساب‌ها هرگز وضعیت ورودی/خروجی را به اشتراک نمی‌گذارند.
- لاگر چندحسابی خط‌های لاگ را با حساب مالک برچسب‌گذاری می‌کند تا وقتی چند بات را زیر یک gateway اجرا می‌کنید، تشخیص‌ها قابل تفکیک بمانند.
- مسیرهای ورودی، خروجی، و پل gateway یک ریشه محموله رسانه مشترک زیر `~/.openclaw/media` دارند، بنابراین بارگذاری‌ها، دانلودها، و حافظه‌های نهان ترنسکد به‌جای یک درخت برای هر زیرسیستم، زیر یک دایرکتوری محافظت‌شده قرار می‌گیرند.
- تحویل رسانه غنی از یک مسیر `sendMedia` برای هدف‌های C2C و گروه عبور می‌کند. فایل‌ها و بافرهای محلی بزرگ‌تر از آستانه فایل بزرگ از نقاط پایانی بارگذاری تکه‌ای QQ استفاده می‌کنند، در حالی که محموله‌های کوچک‌تر از API رسانه یک‌باره استفاده می‌کنند.
- اعتبارنامه‌ها می‌توانند به‌عنوان بخشی از snapshotهای استاندارد اعتبارنامه OpenClaw پشتیبان‌گیری و بازیابی شوند؛ موتور هنگام بازیابی، پشته منبع هر حساب را بدون نیاز به جفت‌سازی تازه با کد QR دوباره متصل می‌کند.

## راه‌اندازی با کد QR

به‌عنوان جایگزینی برای چسباندن دستی `AppID:AppSecret`، موتور از جریان راه‌اندازی با کد QR برای اتصال یک QQ Bot به OpenClaw پشتیبانی می‌کند:

1. مسیر راه‌اندازی QQ Bot را اجرا کنید (برای مثال `openclaw channels add --channel qqbot`) و هنگام درخواست، جریان کد QR را انتخاب کنید.
2. کد QR تولیدشده را با اپلیکیشن گوشی متصل به QQ Bot هدف اسکن کنید.
3. جفت‌سازی را روی گوشی تأیید کنید. OpenClaw اعتبارنامه‌های برگشتی را در `credentials/` زیر محدوده حساب درست پایدار می‌کند.

درخواست‌های تأیید تولیدشده توسط خود بات (برای مثال، جریان‌های "allow this action?" که از طریق API QQ Bot ارائه می‌شوند) به‌صورت درخواست‌های بومی OpenClaw ظاهر می‌شوند که می‌توانید به‌جای پاسخ‌دادن از طریق کلاینت خام QQ، با `/bot-approve` آن‌ها را بپذیرید.

## عیب‌یابی

- **بات پاسخ می‌دهد "gone to Mars":** اعتبارنامه‌ها پیکربندی نشده‌اند یا Gateway شروع نشده است.
- **پیام ورودی وجود ندارد:** بررسی کنید `appId` و `clientSecret` درست هستند، و
  بات در QQ Open Platform فعال شده است.
- **پاسخ‌های تکراری به خودش:** OpenClaw شاخص‌های ارجاع خروجی QQ را به‌عنوان
  تولیدشده توسط بات ثبت می‌کند و رویدادهای ورودی‌ای را که `msgIdx` فعلی‌شان با همان
  حساب بات یکسان است نادیده می‌گیرد. این کار از حلقه‌های پژواک پلتفرم جلوگیری می‌کند، در حالی که همچنان به کاربران اجازه می‌دهد
  پیام‌های قبلی بات را نقل‌قول کنند یا به آن‌ها پاسخ دهند.
- **راه‌اندازی با `--token-file` همچنان پیکربندی‌نشده نشان داده می‌شود:** `--token-file` فقط
  AppSecret را تنظیم می‌کند. همچنان به `appId` در پیکربندی یا `QQBOT_APP_ID` نیاز دارید.
- **پیام‌های کنش‌گرایانه نمی‌رسند:** QQ ممکن است پیام‌های آغازشده توسط بات را رهگیری کند اگر
  کاربر اخیراً تعامل نداشته باشد.
- **صدا رونویسی نمی‌شود:** مطمئن شوید STT پیکربندی شده و ارائه‌دهنده در دسترس است.
