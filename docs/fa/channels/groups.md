OpenClaw چت‌های گروهی را در سطح‌های مختلف به‌صورت یکسان مدیریت می‌کند: Discord، iMessage، Matrix، Microsoft Teams، QQBot، Signal، Slack، Telegram، WhatsApp، Zalo.

برای اتاق‌های همیشه‌فعال که باید زمینه‌ای آرام فراهم کنند مگر اینکه عامل صراحتاً پیام قابل‌مشاهده‌ای بفرستد، [رویدادهای اتاق محیطی](/fa/channels/ambient-room-events) را ببینید.

## معرفی مبتدی (۲ دقیقه)

OpenClaw روی حساب‌های پیام‌رسان خودتان «زندگی می‌کند». کاربر ربات جداگانه‌ای برای WhatsApp وجود ندارد. اگر **شما** در یک گروه باشید، OpenClaw می‌تواند آن گروه را ببیند و همان‌جا پاسخ بدهد.

رفتار پیش‌فرض:

- گروه‌ها محدود هستند (`groupPolicy: "allowlist"`).
- پاسخ‌ها به منشن نیاز دارند، مگر اینکه صراحتاً دروازه‌گذاری منشن را غیرفعال کنید.
- پاسخ‌های قابل‌مشاهده در گروه‌ها/کانال‌ها به‌طور پیش‌فرض از ابزار `message` استفاده می‌کنند.

ترجمه: فرستنده‌های موجود در allowlist می‌توانند با منشن کردن OpenClaw آن را فعال کنند.

<Note>
**خلاصه**

- **دسترسی DM** با `*.allowFrom` کنترل می‌شود.
- **دسترسی گروهی** با `*.groupPolicy` + allowlistها (`*.groups`، `*.groupAllowFrom`) کنترل می‌شود.
- **فعال‌سازی پاسخ** با دروازه‌گذاری منشن (`requireMention`، `/activation`) کنترل می‌شود.

</Note>

جریان سریع (برای یک پیام گروهی چه اتفاقی می‌افتد):

```
groupPolicy? disabled -> drop
groupPolicy? allowlist -> group allowed? no -> drop
requireMention? yes -> mentioned? no -> store for context only
mention/reply/command/DM -> user request
always-on group chatter -> user request, or room event when configured
```

## پاسخ‌های قابل‌مشاهده

برای درخواست‌های عادی گروه/کانال، OpenClaw به‌طور پیش‌فرض از `messages.groupChat.visibleReplies: "automatic"` استفاده می‌کند. متن نهایی دستیار از مسیر قدیمی پاسخ قابل‌مشاهده ارسال می‌شود، مگر اینکه اتاق را برای خروجی فقط از طریق ابزار پیام تنظیم کنید.

وقتی یک اتاق مشترک باید به عامل اجازه بدهد خودش با فراخوانی `message(action=send)` تصمیم بگیرد چه زمانی صحبت کند، از `messages.groupChat.visibleReplies: "message_tool"` استفاده کنید. این حالت برای اتاق‌های گروهی که با مدل‌های نسل جدید و قابل‌اعتماد در استفاده از ابزار، مانند GPT 5.5، پشتیبانی می‌شوند بهترین نتیجه را دارد. اگر مدل آن ابزار را از دست بدهد و متن نهایی معناداری برگرداند، OpenClaw آن متن نهایی را به‌جای ارسال به اتاق، خصوصی نگه می‌دارد.

برای مدل‌ها یا runtimeهای ضعیف‌تر که تحویل فقط از طریق ابزار را قابل‌اعتماد نمی‌فهمند، از `"automatic"` استفاده کنید. در حالت خودکار، متن نهایی دستیار مسیر پاسخ قابل‌مشاهده منبع است، بنابراین مدلی که نمی‌تواند به‌طور پایدار `message(action=send)` را فراخوانی کند همچنان می‌تواند معمولی پاسخ بدهد.

در حالت خودکار، پاسخ‌های نهایی متنی معمولی مستقیماً در اتاق ارسال می‌شوند. اگر پاسخ قابل‌مشاهده به فایل، تصویر یا پیوست‌های دیگر نیاز داشته باشد، عامل همچنان ممکن است برای آن پیوست به‌جای تلاش برای عبور دادن آن از پاسخ متن نهایی، از `message(action=send)` استفاده کند.

اگر ابزار پیام تحت سیاست ابزار فعال در دسترس نباشد، OpenClaw به‌جای سرکوب بی‌صدای پاسخ، به پاسخ‌های قابل‌مشاهده خودکار برمی‌گردد. `openclaw doctor` درباره این ناهماهنگی هشدار می‌دهد.

برای چت‌های مستقیم و هر رویداد منبع دیگر، از `messages.visibleReplies: "message_tool"` استفاده کنید تا همان رفتار پاسخ قابل‌مشاهده فقط از طریق ابزار را به‌صورت سراسری اعمال کنید. نوبت‌های مستقیم WebChat داخلی به‌طور پیش‌فرض از تحویل پاسخ نهایی خودکار استفاده می‌کنند تا Pi و Codex همان قرارداد پاسخ قابل‌مشاهده را دریافت کنند. برای اینکه عمداً خروجی قابل‌مشاهده به `message(action=send)` نیاز داشته باشد، `messages.visibleReplies: "message_tool"` را تنظیم کنید. `messages.groupChat.visibleReplies` همچنان override اختصاصی‌تر برای اتاق‌های گروهی/کانالی باقی می‌ماند.

این جایگزین الگوی قدیمی وادار کردن مدل به پاسخ دادن با `NO_REPLY` برای بیشتر نوبت‌های حالت کمین می‌شود. در حالت فقط ابزار، prompt قرارداد `NO_REPLY` تعریف نمی‌کند. انجام ندادن هیچ کار قابل‌مشاهده‌ای صرفاً یعنی ابزار پیام فراخوانی نشود.

پیوندهای مکالمه‌ای متعلق به Plugin استثنا هستند. وقتی یک Plugin رشته‌ای را bind کند و مالک نوبت ورودی شود، پاسخ برگشتی Plugin همان پاسخ قابل‌مشاهده bind است؛ به `message(action=send)` نیاز ندارد. آن پاسخ خروجی runtime مربوط به Plugin است، نه متن نهایی خصوصی مدل.

نشانگرهای تایپ همچنان برای درخواست‌های مستقیم گروهی ارسال می‌شوند. رویدادهای اتاق همیشه‌فعال محیطی، وقتی فعال باشند، سخت‌گیرانه و آرام باقی می‌مانند مگر اینکه عامل ابزار پیام را فراخوانی کند.

نشست‌ها به‌طور پیش‌فرض خلاصه‌های پرجزئیات ابزار/پیشرفت را سرکوب می‌کنند. هنگام اشکال‌زدایی از `/verbose on` استفاده کنید تا آن خلاصه‌ها برای نشست فعلی نشان داده شوند، و برای بازگشت به رفتار فقط پاسخ نهایی از `/verbose off` استفاده کنید. همان وضعیت verbose در چت‌های مستقیم، گروه‌ها، کانال‌ها و موضوعات انجمن اعمال می‌شود.

برای ثبت گفت‌وگوی گروهی همیشه‌فعال بدون منشن به‌عنوان زمینه آرام اتاق به‌جای درخواست کاربر، از [رویدادهای اتاق محیطی](/fa/channels/ambient-room-events) استفاده کنید:

```json5
{
  messages: {
    groupChat: {
      unmentionedInbound: "room_event",
    },
  },
}
```

پیش‌فرض `unmentionedInbound: "user_request"` است.

پیام‌های منشن‌شده، فرمان‌ها، درخواست‌های لغو، و DMها همچنان درخواست کاربر باقی می‌مانند.

برای اینکه خروجی قابل‌مشاهده در درخواست‌های گروه/کانال الزاماً از ابزار پیام عبور کند:

```json5
{
  messages: {
    groupChat: {
      visibleReplies: "message_tool",
    },
  },
}
```

Gateway پس از ذخیره شدن فایل، پیکربندی `messages` را hot-reload می‌کند. فقط زمانی restart کنید که پایش فایل یا بارگذاری مجدد پیکربندی در deployment غیرفعال باشد.

برای اینکه خروجی قابل‌مشاهده در هر چت منبع الزاماً از ابزار پیام عبور کند:

```json5
{
  messages: {
    visibleReplies: "message_tool",
  },
}
```

فرمان‌های slash بومی (Discord، Telegram، و سطح‌های دیگر با پشتیبانی فرمان بومی) از `visibleReplies: "message_tool"` عبور می‌کنند و همیشه به‌صورت قابل‌مشاهده پاسخ می‌دهند تا UI فرمان بومی کانال پاسخی را که انتظار دارد دریافت کند. این فقط برای نوبت‌های فرمان بومی اعتبارسنجی‌شده اعمال می‌شود؛ فرمان‌های `/...` تایپ‌شده به‌صورت متن و نوبت‌های عادی چت همچنان از پیش‌فرض گروه پیکربندی‌شده پیروی می‌کنند.

## قابلیت مشاهده زمینه و allowlistها

دو کنترل متفاوت در ایمنی گروه دخیل هستند:

- **مجوز فعال‌سازی**: چه کسی می‌تواند عامل را فعال کند (`groupPolicy`، `groups`، `groupAllowFrom`، allowlistهای مخصوص کانال).
- **قابلیت مشاهده زمینه**: چه زمینه تکمیلی به مدل تزریق می‌شود (متن پاسخ، نقل‌قول‌ها، تاریخچه رشته، فراداده forward شده).

به‌طور پیش‌فرض، OpenClaw رفتار معمول چت را در اولویت می‌گذارد و زمینه را عمدتاً همان‌طور که دریافت شده نگه می‌دارد. یعنی allowlistها عمدتاً تعیین می‌کنند چه کسی می‌تواند اقدام‌ها را فعال کند، نه اینکه برای هر قطعه نقل‌قول‌شده یا تاریخی یک مرز حذف اطلاعات سراسری باشند.

## کلیدهای نشست

- نشست‌های گروهی از کلیدهای نشست `agent:<agentId>:<channel>:group:<id>` استفاده می‌کنند (اتاق‌ها/کانال‌ها از `agent:<agentId>:<channel>:channel:<id>` استفاده می‌کنند).
- موضوعات انجمن Telegram به شناسه گروه `:topic:<threadId>` اضافه می‌کنند تا هر موضوع نشست خودش را داشته باشد.
- چت‌های مستقیم از نشست اصلی استفاده می‌کنند (یا در صورت پیکربندی، به‌ازای هر فرستنده).
- Heartbeatها برای نشست‌های گروهی رد می‌شوند.

## الگو: DMهای شخصی + گروه‌های عمومی (یک عامل)

بله، اگر ترافیک «شخصی» شما **DMها** و ترافیک «عمومی» شما **گروه‌ها** باشد، این روش خوب کار می‌کند.

دلیل: در حالت تک‌عاملی، DMها معمولاً وارد کلید نشست **اصلی** (`agent:main:main`) می‌شوند، در حالی که گروه‌ها همیشه از کلیدهای نشست **غیراصلی** (`agent:main:<channel>:group:<id>`) استفاده می‌کنند. اگر sandboxing را با `mode: "non-main"` فعال کنید، آن نشست‌های گروهی در backend پیکربندی‌شده sandbox اجرا می‌شوند، در حالی که نشست DM اصلی شما روی میزبان می‌ماند. اگر backend انتخاب نکنید، Docker backend پیش‌فرض است.

این به شما یک «مغز» عامل می‌دهد (workspace + حافظه مشترک)، اما دو وضعیت اجرایی:

- **DMها**: ابزارهای کامل (میزبان)
- **گروه‌ها**: sandbox + ابزارهای محدودشده

<Note>
اگر به workspaceها/پرسوناهای واقعاً جدا نیاز دارید («شخصی» و «عمومی» هرگز نباید مخلوط شوند)، از عامل دوم + bindingها استفاده کنید. [Multi-Agent Routing](/fa/concepts/multi-agent) را ببینید.
</Note>

## برچسب‌های نمایش

- برچسب‌های UI وقتی `displayName` در دسترس باشد از آن استفاده می‌کنند، با قالب `<channel>:<token>`.
- `#room` برای اتاق‌ها/کانال‌ها رزرو شده است؛ چت‌های گروهی از `g-<slug>` استفاده می‌کنند (حروف کوچک، فاصله‌ها -> `-`، حفظ `#@+._-`).

## سیاست گروه

کنترل کنید پیام‌های گروه/اتاق در هر کانال چگونه مدیریت شوند:

```json5
{
  channels: {
    whatsapp: {
      groupPolicy: "disabled", // "open" | "disabled" | "allowlist"
      groupAllowFrom: ["+15551234567"],
    },
    telegram: {
      groupPolicy: "disabled",
      groupAllowFrom: ["123456789"], // numeric Telegram user id (wizard can resolve @username)
    },
    signal: {
      groupPolicy: "disabled",
      groupAllowFrom: ["+15551234567"],
    },
    imessage: {
      groupPolicy: "disabled",
      groupAllowFrom: ["chat_id:123"],
    },
    msteams: {
      groupPolicy: "disabled",
      groupAllowFrom: ["user@org.com"],
    },
    discord: {
      groupPolicy: "allowlist",
      guilds: {
        GUILD_ID: { channels: { help: { allow: true } } },
      },
    },
    slack: {
      groupPolicy: "allowlist",
      channels: { "#general": { allow: true } },
    },
    matrix: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["@owner:example.org"],
      groups: {
        "!roomId:example.org": { enabled: true },
        "#alias:example.org": { enabled: true },
      },
    },
  },
}
```

| سیاست        | رفتار                                                     |
| ------------- | ------------------------------------------------------------ |
| `"open"`      | گروه‌ها فهرست‌های مجاز را دور می‌زنند؛ دروازه‌بانی منشن همچنان اعمال می‌شود.      |
| `"disabled"`  | همه پیام‌های گروهی را کاملا مسدود می‌کند.                           |
| `"allowlist"` | فقط گروه‌ها/اتاق‌هایی را مجاز می‌کند که با فهرست مجاز پیکربندی‌شده مطابقت داشته باشند. |

    - WhatsApp/Telegram/Signal/iMessage/Microsoft Teams/Zalo: از `groupAllowFrom` استفاده کنید (جایگزین: `allowFrom` صریح).
    - Signal: `groupAllowFrom` می‌تواند با شناسه گروه Signal ورودی یا شماره تلفن/UUID فرستنده مطابقت داشته باشد.
    - تاییدهای جفت‌سازی پیام مستقیم (ورودی‌های ذخیره‌سازی `*-allowFrom`) فقط برای دسترسی پیام مستقیم اعمال می‌شوند؛ مجوزدهی فرستنده گروه همچنان صریحا در فهرست‌های مجاز گروه باقی می‌ماند.
    - Discord: فهرست مجاز از `channels.discord.guilds.<id>.channels` استفاده می‌کند.
    - Slack: فهرست مجاز از `channels.slack.channels` استفاده می‌کند.
    - Matrix: فهرست مجاز از `channels.matrix.groups` استفاده می‌کند. شناسه‌های اتاق یا نام‌های مستعار را ترجیح دهید؛ جست‌وجوی نام اتاق‌های پیوسته به‌صورت بهترین تلاش انجام می‌شود، و نام‌های حل‌نشده در زمان اجرا نادیده گرفته می‌شوند. برای محدود کردن فرستنده‌ها از `channels.matrix.groupAllowFrom` استفاده کنید؛ فهرست‌های مجاز `users` برای هر اتاق نیز پشتیبانی می‌شوند.
    - پیام‌های مستقیم گروهی جداگانه کنترل می‌شوند (`channels.discord.dm.*`، `channels.slack.dm.*`).
    - فهرست مجاز Telegram می‌تواند با شناسه‌های کاربر (`"123456789"`، `"telegram:123456789"`، `"tg:123456789"`) یا نام‌های کاربری (`"@alice"` یا `"alice"`) مطابقت داشته باشد؛ پیشوندها به بزرگی/کوچکی حروف حساس نیستند.
    - مقدار پیش‌فرض `groupPolicy: "allowlist"` است؛ اگر فهرست مجاز گروه شما خالی باشد، پیام‌های گروهی مسدود می‌شوند.
    - ایمنی زمان اجرا: وقتی بلوک یک ارائه‌دهنده کاملا وجود ندارد (`channels.<provider>` غایب است)، سیاست گروه به‌جای ارث‌بری از `channels.defaults.groupPolicy` به حالت بسته در برابر خطا برمی‌گردد (معمولا `allowlist`).

  </Accordion>
</AccordionGroup>

مدل ذهنی سریع (ترتیب ارزیابی برای پیام‌های گروهی):

## دروازه‌بانی منشن (پیش‌فرض)

پیام‌های گروهی به منشن نیاز دارند، مگر اینکه برای هر گروه بازنویسی شده باشد. پیش‌فرض‌ها برای هر زیرسیستم زیر `*.groups."*"` قرار دارند.

پاسخ دادن به پیام ربات، وقتی کانال از فراداده پاسخ پشتیبانی کند، به‌عنوان یک منشن ضمنی شمرده می‌شود. نقل‌قول کردن پیام ربات نیز در کانال‌هایی که فراداده نقل‌قول را ارائه می‌کنند می‌تواند به‌عنوان منشن ضمنی شمرده شود. موارد داخلی فعلی شامل Telegram، WhatsApp، Slack، Discord، Microsoft Teams و ZaloUser هستند.

```json5
{
  channels: {
    whatsapp: {
      groups: {
        "*": { requireMention: true },
        "123@g.us": { requireMention: false },
      },
    },
    telegram: {
      groups: {
        "*": { requireMention: true },
        "123456789": { requireMention: false },
      },
    },
    imessage: {
      groups: {
        "*": { requireMention: true },
        "123": { requireMention: false },
      },
    },
  },
  agents: {
    list: [
      {
        id: "main",
        groupChat: {
          mentionPatterns: ["@openclaw", "openclaw", "\\+15555550123"],
          historyLimit: 50,
        },
      },
    ],
  },
}
```

## محدود کردن دامنه الگوهای منشن پیکربندی‌شده

`mentionPatterns` پیکربندی‌شده محرک‌های جایگزین رگکس هستند. وقتی پلتفرم منشن بومی ربات را ارائه نمی‌کند، یا وقتی می‌خواهید متن ساده‌ای مانند `openclaw:` به‌عنوان منشن شمرده شود، از آن‌ها استفاده کنید. منشن‌های بومی پلتفرم جدا هستند: وقتی Discord، Slack، Telegram، Matrix یا کانال دیگری بتواند ثابت کند پیام صراحتا ربات را منشن کرده است، همان منشن بومی همچنان فعال می‌شود، حتی اگر الگوهای رگکس پیکربندی‌شده رد شده باشند.

به‌صورت پیش‌فرض، الگوهای منشن پیکربندی‌شده هرجا که آن کانال داده‌های ارائه‌دهنده و مکالمه را به تشخیص منشن پاس بدهد اعمال می‌شوند. برای جلوگیری از اینکه الگوهای گسترده عامل را در هر گروهی بیدار کنند، آن‌ها را برای هر کانال با `channels.<channel>.mentionPatterns` محدود کنید.

وقتی الگوهای منشن رگکس باید به‌صورت پیش‌فرض برای یک کانال خاموش باشند، از `mode: "deny"` استفاده کنید، سپس با `allowIn` اتاق‌های مشخصی را فعال کنید:

```json5
{
  messages: {
    groupChat: {
      mentionPatterns: ["\\bopenclaw\\b", "\\bops bot\\b"],
    },
  },
  channels: {
    slack: {
      mentionPatterns: {
        mode: "deny",
        allowIn: ["C0123OPS"],
      },
    },
  },
}
```

وقتی الگوهای منشن رگکس باید به‌صورت گسترده اعمال شوند، از `mode: "allow"` پیش‌فرض استفاده کنید (یا `mode` را حذف کنید)، سپس آن‌ها را در اتاق‌های شلوغ با `denyIn` خاموش کنید:

```json5
{
  messages: {
    groupChat: {
      mentionPatterns: ["\\bopenclaw\\b"],
    },
  },
  channels: {
    telegram: {
      mentionPatterns: {
        denyIn: ["-1001234567890", "-1001234567890:topic:42"],
      },
    },
  },
}
```

حل‌وفصل سیاست:

| فیلد           | اثر                                                                                                                |
| --------------- | --------------------------------------------------------------------------------------------------------------------- |
| `mode: "allow"` | الگوهای منشن رگکس فعال هستند، مگر اینکه شناسه مکالمه در `denyIn` باشد. این مقدار پیش‌فرض است.                    |
| `mode: "deny"`  | الگوهای منشن رگکس غیرفعال هستند، مگر اینکه شناسه مکالمه در `allowIn` باشد.                                       |
| `allowIn`       | شناسه‌های مکالمه‌ای که در حالت رد، الگوهای منشن رگکس در آن‌ها فعال هستند.                                               |
| `denyIn`        | شناسه‌های مکالمه‌ای که الگوهای منشن رگکس در آن‌ها غیرفعال هستند. اگر هر دو شامل یک شناسه باشند، `denyIn` بر `allowIn` مقدم است. |

سیاست رگکس محدودشده‌ای که امروز پشتیبانی می‌شود:

| کانال  | شناسه‌های استفاده‌شده در `allowIn` / `denyIn`                             |
| -------- | ------------------------------------------------------------ |
| Discord  | شناسه‌های کانال Discord.                                         |
| Matrix   | شناسه‌های اتاق Matrix.                                             |
| Slack    | شناسه‌های کانال Slack.                                           |
| Telegram | شناسه‌های گفت‌وگوی گروهی، یا `chatId:topic:threadId` برای موضوعات انجمن. |
| WhatsApp | شناسه‌های مکالمه WhatsApp مانند `123@g.us`.                |

پیکربندی‌های کانال در سطح حساب، وقتی آن کانال از چند حساب پشتیبانی می‌کند، می‌توانند همان سیاست را زیر `channels.<channel>.accounts.<accountId>.mentionPatterns` تنظیم کنند. سیاست حساب برای آن حساب بر سیاست سطح بالای کانال مقدم است.

<AccordionGroup>
  <Accordion title="Mention gating notes">
    - `mentionPatterns` الگوهای رگکس ایمن و بدون حساسیت به بزرگی/کوچکی حروف هستند؛ الگوهای نامعتبر و شکل‌های تکرار تودرتوی ناامن نادیده گرفته می‌شوند.
    - سطوحی که منشن‌های صریح ارائه می‌کنند همچنان عبور می‌کنند؛ الگوهای رگکس پیکربندی‌شده یک جایگزین هستند.
    - `channels.<channel>.mentionPatterns.mode: "deny"` الگوهای منشن پیکربندی‌شده را به‌صورت پیش‌فرض برای آن کانال غیرفعال می‌کند؛ مکالمه‌های انتخاب‌شده را با `allowIn` دوباره فعال کنید.
    - `channels.<channel>.mentionPatterns.denyIn` الگوهای منشن پیکربندی‌شده را برای شناسه‌های مکالمه مشخص غیرفعال می‌کند، درحالی‌که @mentions بومی پلتفرم همچنان عبور می‌کنند.
    - بازنویسی برای هر عامل: `agents.list[].groupChat.mentionPatterns` (وقتی چند عامل یک گروه را به اشتراک می‌گذارند مفید است).
    - دروازه‌بانی منشن فقط وقتی اعمال می‌شود که تشخیص منشن ممکن باشد (منشن‌های بومی یا `mentionPatterns` پیکربندی شده باشند).
    - قرار دادن یک گروه یا فرستنده در فهرست مجاز، دروازه‌بانی منشن را غیرفعال نمی‌کند؛ وقتی همه پیام‌ها باید فعال کنند، `requireMention` آن گروه را روی `false` تنظیم کنید.
    - زمینه اعلان گفت‌وگوی گروهی خودکار در هر نوبت دستور پاسخ بی‌صدای حل‌شده را حمل می‌کند؛ فایل‌های فضای کاری نباید سازوکارهای `NO_REPLY` را تکرار کنند.
    - گروه‌هایی که پاسخ‌های بی‌صدای خودکار در آن‌ها مجاز است، نوبت‌های مدل تمیز خالی یا فقط استدلالی را بی‌صدا تلقی می‌کنند، معادل `NO_REPLY`. گفت‌وگوهای مستقیم هرگز راهنمایی `NO_REPLY` دریافت نمی‌کنند، و پاسخ‌های گروهی فقط با ابزار پیام با فراخوانی نکردن `message(action=send)` ساکت می‌مانند.
    - گفت‌وگوی گروهی محیطی و همیشه‌روشن به‌صورت پیش‌فرض از معناشناسی درخواست کاربر استفاده می‌کند. برای ارسال آن به‌عنوان زمینه ساکت، `messages.groupChat.unmentionedInbound: "room_event"` را تنظیم کنید. برای نمونه‌های راه‌اندازی، [رویدادهای محیطی اتاق](/fa/channels/ambient-room-events) را ببینید.
    - رویدادهای اتاق به‌عنوان درخواست‌های جعلی کاربر ذخیره نمی‌شوند، و متن خصوصی دستیار از رویدادهای اتاق بدون ابزار پیام به‌عنوان تاریخچه گفت‌وگو بازپخش نمی‌شود.
    - پیش‌فرض‌های Discord در `channels.discord.guilds."*"` قرار دارند (قابل بازنویسی برای هر سرور/کانال).
    - زمینه تاریخچه گروه در همه کانال‌ها به‌صورت یکنواخت بسته‌بندی می‌شود. گروه‌های دارای دروازه‌بانی منشن پیام‌های ردشده در انتظار را نگه می‌دارند؛ گروه‌های همیشه‌روشن نیز ممکن است وقتی کانال پشتیبانی کند، پیام‌های اخیر پردازش‌شده اتاق را نگه دارند. برای پیش‌فرض سراسری از `messages.groupChat.historyLimit` و برای بازنویسی‌ها از `channels.<channel>.historyLimit` (یا `channels.<channel>.accounts.*.historyLimit`) استفاده کنید. برای غیرفعال کردن، `0` را تنظیم کنید.

  </Accordion>
</AccordionGroup>

## محدودیت‌های ابزار گروه/کانال (اختیاری)

برخی پیکربندی‌های کانال از محدود کردن ابزارهایی که **داخل یک گروه/اتاق/کانال مشخص** در دسترس هستند پشتیبانی می‌کنند.

- `tools`: ابزارها را برای کل گروه مجاز/رد می‌کند.
- `toolsBySender`: بازنویسی‌های هر فرستنده درون گروه. از پیشوندهای کلید صریح استفاده کنید: `channel:<channelId>:<senderId>`، `id:<senderId>`، `e164:<phone>`، `username:<handle>`، `name:<displayName>` و wildcard `"*"`. شناسه‌های کانال از شناسه‌های کانال متعارف OpenClaw استفاده می‌کنند؛ نام‌های مستعاری مانند `teams` به `msteams` نرمال‌سازی می‌شوند. کلیدهای قدیمی بدون پیشوند همچنان پذیرفته می‌شوند و فقط به‌عنوان `id:` مطابقت داده می‌شوند.

ترتیب حل‌وفصل (خاص‌ترین مورد برنده است):

<Steps>
  <Step title="Group toolsBySender">
    مطابقت `toolsBySender` گروه/کانال.
  </Step>
  <Step title="Group tools">
    `tools` گروه/کانال.
  </Step>
  <Step title="Default toolsBySender">
    مطابقت `toolsBySender` پیش‌فرض (`"*"`).
  </Step>
  <Step title="Default tools">
    `tools` پیش‌فرض (`"*"`).
  </Step>
</Steps>

نمونه (Telegram):

```json5
{
  channels: {
    telegram: {
      groups: {
        "*": { tools: { deny: ["exec"] } },
        "-1001234567890": {
          tools: { deny: ["exec", "read", "write"] },
          toolsBySender: {
            "id:123456789": { alsoAllow: ["exec"] },
          },
        },
      },
    },
  },
}
```

<Note>
محدودیت‌های ابزار گروه/کانال علاوه بر سیاست ابزار سراسری/عامل اعمال می‌شوند (رد همچنان برنده است). برخی کانال‌ها از تودرتویی متفاوتی برای اتاق‌ها/کانال‌ها استفاده می‌کنند (برای مثال، Discord `guilds.*.channels.*`، Slack `channels.*`، Microsoft Teams `teams.*.channels.*`).
</Note>

## فهرست‌های مجاز گروه

وقتی `channels.whatsapp.groups`، `channels.telegram.groups` یا `channels.imessage.groups` پیکربندی شده باشد، کلیدها به‌عنوان فهرست مجاز گروه عمل می‌کنند. برای مجاز کردن همه گروه‌ها درحالی‌که همچنان رفتار پیش‌فرض منشن را تنظیم می‌کنید، از `"*"` استفاده کنید.

<Warning>
ابهام رایج: تأیید جفت‌سازی DM همان مجوزدهی گروه نیست. برای کانال‌هایی که از جفت‌سازی DM پشتیبانی می‌کنند، ذخیره‌گاه جفت‌سازی فقط DMها را باز می‌کند. فرمان‌های گروه همچنان به مجوزدهی صریح فرستنده گروه از فهرست‌های مجاز پیکربندی، مانند `groupAllowFrom`، یا جایگزین پیکربندی مستندشده برای آن کانال نیاز دارند.
</Warning>

## فعال‌سازی (فقط مالک)

مالکان گروه می‌توانند فعال‌سازی هر گروه را تغییر دهند:

- `/activation mention`
- `/activation always`

مالک با `channels.whatsapp.allowFrom` تعیین می‌شود (یا اگر تنظیم نشده باشد، E.164 خود بات). فرمان را به‌صورت یک پیام مستقل ارسال کنید. سطوح دیگر در حال حاضر `/activation` را نادیده می‌گیرند.

## فیلدهای زمینه

بارهای ورودی گروه این موارد را تنظیم می‌کنند:

- `ChatType=group`
- `GroupSubject` (اگر شناخته شده باشد)
- `GroupMembers` (اگر شناخته شده باشد)
- `WasMentioned` (نتیجه دروازه‌گذاری منشن)
- موضوع‌های انجمن Telegram همچنین شامل `MessageThreadId` و `IsForum` هستند.

پرامپت سیستمی عامل در نخستین نوبت یک نشست گروهی جدید، یک مقدمه گروهی را شامل می‌شود. این مقدمه به مدل یادآوری می‌کند مانند انسان پاسخ دهد، خطوط خالی را به حداقل برساند و از فاصله‌گذاری معمول چت پیروی کند، و از تایپ توالی‌های لفظی `\n` خودداری کند. گروه‌های غیر Telegram همچنین جدول‌های Markdown را توصیه نمی‌کنند؛ راهنمایی متن غنی Telegram از پرامپت کانال Telegram می‌آید. نام‌های گروه و برچسب‌های شرکت‌کنندگان که از کانال آمده‌اند، به‌صورت فراداده نامطمئن حصارگذاری‌شده رندر می‌شوند، نه دستورهای سیستمی درون‌خطی.

## نکات ویژه iMessage

- هنگام مسیریابی یا قرار دادن در فهرست مجاز، `chat_id:<id>` را ترجیح دهید.
- فهرست چت‌ها: `imsg chats --limit 20`.
- پاسخ‌های گروه همیشه به همان `chat_id` برمی‌گردند.

## پرامپت‌های سیستمی WhatsApp

برای قواعد مرجع پرامپت سیستمی WhatsApp، از جمله تفکیک پرامپت گروهی و مستقیم، رفتار wildcard، و معنای بازنویسی حساب، [WhatsApp](/fa/channels/whatsapp#system-prompts) را ببینید.
