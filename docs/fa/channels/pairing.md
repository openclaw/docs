«جفت‌سازی» مرحلهٔ تأیید دسترسی صریح OpenClaw است.
در دو جا استفاده می‌شود:

1. **جفت‌سازی DM** (چه کسی اجازه دارد با ربات صحبت کند)
2. **جفت‌سازی Node** (کدام دستگاه‌ها/نودها اجازه دارند به شبکهٔ Gateway بپیوندند)

زمینهٔ امنیتی: [امنیت](/fa/gateway/security)

## 1) جفت‌سازی DM (دسترسی چت ورودی)

وقتی یک کانال با سیاست DM به‌صورت `pairing` پیکربندی شده باشد، فرستنده‌های ناشناس یک کد کوتاه دریافت می‌کنند و پیام آن‌ها تا زمانی که تأیید کنید **پردازش نمی‌شود**.

سیاست‌های پیش‌فرض DM در اینجا مستند شده‌اند: [امنیت](/fa/gateway/security)

`dmPolicy: "open"` فقط زمانی عمومی است که فهرست مجاز مؤثر DM شامل `"*"` باشد.
راه‌اندازی و اعتبارسنجی برای پیکربندی‌های عمومی-باز به این wildcard نیاز دارند. اگر وضعیت موجود
شامل `open` با ورودی‌های مشخص `allowFrom` باشد، زمان اجرا همچنان فقط همان
فرستنده‌ها را می‌پذیرد، و تأییدهای مخزن جفت‌سازی دسترسی `open` را گسترش نمی‌دهند.

کدهای جفت‌سازی:

- ۸ نویسه، حروف بزرگ، بدون نویسه‌های مبهم (`0O1I`).
- **پس از ۱ ساعت منقضی می‌شوند**. ربات فقط زمانی پیام جفت‌سازی را می‌فرستد که درخواست جدیدی ایجاد شود (تقریباً هر ساعت یک بار برای هر فرستنده).
- درخواست‌های در انتظار جفت‌سازی DM به‌صورت پیش‌فرض در **۳ درخواست برای هر کانال** محدود می‌شوند؛ درخواست‌های اضافی نادیده گرفته می‌شوند تا یکی منقضی یا تأیید شود.

### تأیید یک فرستنده

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

اگر هنوز مالک فرمانی پیکربندی نشده باشد، تأیید یک کد جفت‌سازی DM همچنین
`commands.ownerAllowFrom` را با فرستندهٔ تأییدشده مقداردهی اولیه می‌کند، مانند `telegram:123456789`.
این کار برای راه‌اندازی‌های اولیه یک مالک صریح برای فرمان‌های دارای امتیاز و اعلان‌های تأیید exec
فراهم می‌کند. پس از وجود مالک، تأییدهای بعدی جفت‌سازی فقط دسترسی DM
می‌دهند؛ مالک‌های بیشتری اضافه نمی‌کنند.

کانال‌های پشتیبانی‌شده: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### گروه‌های فرستندهٔ قابل استفادهٔ مجدد

وقتی همان مجموعهٔ فرستندگان مورد اعتماد باید برای چند
کانال پیام یا هم برای فهرست‌های مجاز DM و گروه اعمال شود، از `accessGroups` سطح بالا استفاده کنید.

گروه‌های ایستا از `type: "message.senders"` استفاده می‌کنند و از فهرست‌های مجاز کانال با
`accessGroup:<name>` ارجاع داده می‌شوند:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

گروه‌های دسترسی اینجا با جزئیات مستند شده‌اند: [گروه‌های دسترسی](/fa/channels/access-groups)

### وضعیت کجا نگهداری می‌شود

زیر `~/.openclaw/credentials/` ذخیره می‌شود:

- درخواست‌های در انتظار: `<channel>-pairing.json`
- مخزن فهرست مجاز تأییدشده:
  - حساب پیش‌فرض: `<channel>-allowFrom.json`
  - حساب غیرپیش‌فرض: `<channel>-<accountId>-allowFrom.json`

رفتار دامنه‌بندی حساب:

- حساب‌های غیرپیش‌فرض فقط فایل فهرست مجاز دامنه‌بندی‌شدهٔ خود را می‌خوانند/می‌نویسند.
- حساب پیش‌فرض از فایل فهرست مجاز بدون دامنهٔ کانال‌محور استفاده می‌کند.

با این‌ها به‌عنوان داده‌های حساس رفتار کنید (آن‌ها دسترسی به دستیار شما را کنترل می‌کنند).

<Note>
مخزن فهرست مجاز جفت‌سازی برای دسترسی DM است. مجوزدهی گروه جداست.
تأیید یک کد جفت‌سازی DM به‌طور خودکار به آن فرستنده اجازه نمی‌دهد فرمان‌های گروهی را اجرا کند
یا ربات را در گروه‌ها کنترل کند. مقداردهی اولیهٔ مالک اول وضعیت پیکربندی جداگانه‌ای
در `commands.ownerAllowFrom` است، و تحویل چت گروهی همچنان از فهرست‌های مجاز گروهی
کانال پیروی می‌کند (برای مثال `groupAllowFrom`، `groups`، یا بازنویسی‌های هر گروه
یا هر موضوع، بسته به کانال).
</Note>

## 2) جفت‌سازی دستگاه Node (Nodeهای iOS/Android/macOS/headless)

Nodeها به‌عنوان **دستگاه**هایی با `role: node` به Gateway وصل می‌شوند. Gateway
یک درخواست جفت‌سازی دستگاه ایجاد می‌کند که باید تأیید شود.

### جفت‌سازی از طریق Telegram (توصیه‌شده برای iOS)

اگر از Plugin `device-pair` استفاده می‌کنید، می‌توانید جفت‌سازی اولیهٔ دستگاه را کاملاً از Telegram انجام دهید:

1. در Telegram، به ربات خود پیام بدهید: `/pair`
2. ربات با دو پیام پاسخ می‌دهد: یک پیام دستورالعمل و یک پیام جداگانهٔ **کد راه‌اندازی** (برای کپی/پیست در Telegram آسان است).
3. روی تلفن خود، برنامهٔ iOS OpenClaw را باز کنید → Settings → Gateway.
4. کد راه‌اندازی را جای‌گذاری کنید و وصل شوید.
5. دوباره در Telegram: `/pair pending` (شناسه‌های درخواست، نقش، و دامنه‌ها را مرور کنید)، سپس تأیید کنید.

کد راه‌اندازی یک payload JSON کدگذاری‌شده با base64 است که شامل موارد زیر است:

- `url`: نشانی WebSocket مربوط به Gateway (`ws://...` یا `wss://...`)
- `bootstrapToken`: یک token راه‌انداز کوتاه‌عمر و تک‌دستگاهی که برای دست‌دهی اولیهٔ جفت‌سازی استفاده می‌شود

آن token راه‌انداز پروفایل داخلی راه‌انداز جفت‌سازی را حمل می‌کند:

- token اصلی واگذار‌شدهٔ `node` روی `scopes: []` باقی می‌ماند
- هر token واگذارشدهٔ `operator` در محدودهٔ فهرست مجاز راه‌انداز باقی می‌ماند:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- بررسی‌های دامنهٔ راه‌انداز با پیشوند نقش انجام می‌شوند، نه یک مخزن دامنهٔ تخت:
  ورودی‌های دامنهٔ operator فقط درخواست‌های operator را برآورده می‌کنند، و نقش‌های غیر-operator
  همچنان باید دامنه‌ها را زیر پیشوند نقش خود درخواست کنند
- چرخش/ابطال بعدی token همچنان هم به قرارداد نقش تأییدشدهٔ دستگاه
  و هم به دامنه‌های operator نشست فراخواننده محدود می‌ماند

تا وقتی کد راه‌اندازی معتبر است، با آن مانند گذرواژه رفتار کنید.

### تأیید یک دستگاه Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

وقتی یک تأیید صریح رد می‌شود چون نشست دستگاه جفت‌شدهٔ تأییدکننده
با دامنهٔ فقط-جفت‌سازی باز شده بود، CLI همان درخواست را با
`operator.admin` دوباره امتحان می‌کند. این کار به یک دستگاه جفت‌شدهٔ دارای قابلیت ادمین اجازه می‌دهد یک جفت‌سازی جدید
Control UI/مرورگر را بدون ویرایش دستی `devices/paired.json` بازیابی کند.
Gateway همچنان اتصال دوباره‌امتحان‌شده را اعتبارسنجی می‌کند؛ tokenهایی که نمی‌توانند
با `operator.admin` احراز هویت کنند مسدود می‌مانند.

اگر همان دستگاه با جزئیات احراز هویت متفاوت دوباره تلاش کند (برای مثال
نقش/دامنه‌ها/کلید عمومی متفاوت)، درخواست در انتظار قبلی جایگزین می‌شود و یک
`requestId` جدید ایجاد می‌شود.

<Note>
دستگاهی که از قبل جفت شده است، به‌طور پنهانی دسترسی گسترده‌تر نمی‌گیرد. اگر دوباره وصل شود و دامنه‌های بیشتر یا نقش گسترده‌تری بخواهد، OpenClaw تأیید موجود را همان‌طور نگه می‌دارد و یک درخواست ارتقای تازهٔ در انتظار ایجاد می‌کند. پیش از تأیید، از `openclaw devices list` استفاده کنید تا دسترسی فعلاً تأییدشده را با دسترسی تازه درخواست‌شده مقایسه کنید.
</Note>

### تأیید خودکار اختیاری Node با CIDR مورد اعتماد

جفت‌سازی دستگاه به‌صورت پیش‌فرض دستی می‌ماند. برای شبکه‌های Node با کنترل سخت‌گیرانه،
می‌توانید با CIDRهای صریح یا IPهای دقیق، تأیید خودکار اولیهٔ Node را فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این فقط برای درخواست‌های جفت‌سازی تازهٔ `role: node` بدون دامنه‌های
درخواستی اعمال می‌شود. کلاینت‌های Operator، مرورگر، Control UI، و WebChat همچنان به تأیید دستی
نیاز دارند. تغییرات نقش، دامنه، فراداده، و کلید عمومی همچنان به تأیید دستی
نیاز دارند.

### ذخیره‌سازی وضعیت جفت‌سازی Node

زیر `~/.openclaw/devices/` ذخیره می‌شود:

- `pending.json` (کوتاه‌عمر؛ درخواست‌های در انتظار منقضی می‌شوند)
- `paired.json` (دستگاه‌های جفت‌شده + tokenها)

### نکته‌ها

- API قدیمی `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) یک
  مخزن جفت‌سازی جداگانه و متعلق به Gateway است. Nodeهای WS همچنان به جفت‌سازی دستگاه نیاز دارند.
- رکورد جفت‌سازی منبع حقیقت پایدار برای نقش‌های تأییدشده است. tokenهای فعال
  دستگاه در محدودهٔ همان مجموعه نقش تأییدشده باقی می‌مانند؛ یک ورودی token سرگردان
  خارج از نقش‌های تأییدشده دسترسی جدید ایجاد نمی‌کند.
