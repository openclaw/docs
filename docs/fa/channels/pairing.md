«جفت‌سازی» مرحلهٔ تأیید صریح دسترسی در OpenClaw است.
در دو جا استفاده می‌شود:

1. **جفت‌سازی پیام مستقیم** (چه کسی مجاز است با ربات صحبت کند)
2. **جفت‌سازی Node** (کدام دستگاه‌ها/نودها مجازند به شبکهٔ Gateway بپیوندند)

زمینهٔ امنیتی: [امنیت](/fa/gateway/security)

## 1) جفت‌سازی پیام مستقیم (دسترسی گفت‌وگوی ورودی)

وقتی یک کانال با سیاست پیام مستقیم `pairing` پیکربندی شده باشد، فرستنده‌های ناشناس یک کد کوتاه دریافت می‌کنند و پیامشان تا زمانی که تأیید نکنید **پردازش نمی‌شود**.

سیاست‌های پیش‌فرض پیام مستقیم در اینجا مستند شده‌اند: [امنیت](/fa/gateway/security)

`dmPolicy: "open"` فقط وقتی عمومی است که فهرست مجاز مؤثر پیام مستقیم شامل `"*"` باشد.
راه‌اندازی و اعتبارسنجی برای پیکربندی‌های عمومی-باز به این وایلدکارت نیاز دارند. اگر وضعیت موجود
شامل `open` همراه با ورودی‌های مشخص `allowFrom` باشد، زمان اجرا همچنان فقط همان
فرستنده‌ها را می‌پذیرد، و تأییدهای مخزن جفت‌سازی دسترسی `open` را گسترده‌تر نمی‌کنند.

کدهای جفت‌سازی:

- ۸ نویسه، حروف بزرگ، بدون نویسه‌های مبهم (`0O1I`).
- **پس از ۱ ساعت منقضی می‌شوند**. ربات فقط وقتی درخواست جدیدی ایجاد شود پیام جفت‌سازی را می‌فرستد (تقریباً هر ساعت یک بار برای هر فرستنده).
- درخواست‌های جفت‌سازی پیام مستقیم در انتظار به‌طور پیش‌فرض به **۳ درخواست برای هر کانال** محدود شده‌اند؛ درخواست‌های اضافی تا زمانی که یکی منقضی یا تأیید شود نادیده گرفته می‌شوند.

### تأیید یک فرستنده

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

اگر هنوز مالک دستوری پیکربندی نشده باشد، تأیید یک کد جفت‌سازی پیام مستقیم همچنین
`commands.ownerAllowFrom` را با فرستندهٔ تأییدشده، مانند `telegram:123456789`، مقداردهی اولیه می‌کند.
این کار برای راه‌اندازی‌های نخستین یک مالک صریح برای فرمان‌های ممتاز و اعلان‌های تأیید اجرا
فراهم می‌کند. پس از اینکه مالک وجود داشته باشد، تأییدهای بعدی جفت‌سازی فقط دسترسی پیام مستقیم
می‌دهند؛ مالک‌های بیشتری اضافه نمی‌کنند.

کانال‌های پشتیبانی‌شده: `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### گروه‌های فرستندهٔ قابل استفادهٔ مجدد

وقتی یک مجموعهٔ فرستندهٔ مورد اعتماد باید برای چند کانال پیام‌رسانی
یا هم برای فهرست‌های مجاز پیام مستقیم و گروهی اعمال شود، از `accessGroups` سطح بالا استفاده کنید.

گروه‌های ایستا از `type: "message.senders"` استفاده می‌کنند و از فهرست‌های مجاز کانال
با `accessGroup:<name>` ارجاع داده می‌شوند:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

گروه‌های دسترسی با جزئیات در اینجا مستند شده‌اند: [گروه‌های دسترسی](/fa/channels/access-groups)

### وضعیت کجا نگهداری می‌شود

زیر `~/.openclaw/credentials/` ذخیره می‌شود:

- درخواست‌های در انتظار: `<channel>-pairing.json`
- مخزن فهرست مجاز تأییدشده:
  - حساب پیش‌فرض: `<channel>-allowFrom.json`
  - حساب غیرپیش‌فرض: `<channel>-<accountId>-allowFrom.json`

رفتار محدوده‌بندی حساب:

- حساب‌های غیرپیش‌فرض فقط فایل فهرست مجاز محدوده‌بندی‌شدهٔ خود را می‌خوانند/می‌نویسند.
- حساب پیش‌فرض از فایل فهرست مجاز بدون محدودهٔ کانال استفاده می‌کند.

با این‌ها مثل دادهٔ حساس رفتار کنید (دسترسی به دستیار شما را کنترل می‌کنند).

<Note>
مخزن فهرست مجاز جفت‌سازی برای دسترسی پیام مستقیم است. مجوزدهی گروه جداست.
تأیید یک کد جفت‌سازی پیام مستقیم به‌طور خودکار آن فرستنده را مجاز نمی‌کند که فرمان‌های گروهی
را اجرا کند یا ربات را در گروه‌ها کنترل کند. مقداردهی اولیهٔ مالک نخستین وضعیت پیکربندی جداگانه‌ای
در `commands.ownerAllowFrom` است، و تحویل گفت‌وگوی گروهی همچنان از فهرست‌های مجاز گروهی
کانال پیروی می‌کند (برای مثال `groupAllowFrom`، `groups`، یا بازنویسی‌های هر گروه
یا هر موضوع بسته به کانال).
</Note>

## 2) جفت‌سازی دستگاه Node (نودهای iOS/Android/macOS/بدون‌سر)

نودها به‌عنوان **دستگاه** با `role: node` به Gateway متصل می‌شوند. Gateway
یک درخواست جفت‌سازی دستگاه ایجاد می‌کند که باید تأیید شود.

### جفت‌سازی از طریق Telegram (توصیه‌شده برای iOS)

اگر از Plugin `device-pair` استفاده می‌کنید، می‌توانید جفت‌سازی اولیهٔ دستگاه را کاملاً از Telegram انجام دهید:

1. در Telegram، به ربات خود پیام بدهید: `/pair`
2. ربات با دو پیام پاسخ می‌دهد: یک پیام راهنما و یک پیام جداگانهٔ **کد راه‌اندازی** (برای کپی/چسباندن در Telegram آسان است).
3. روی تلفن خود، برنامهٔ OpenClaw iOS را باز کنید ← Settings ← Gateway.
4. کد QR را اسکن کنید یا کد راه‌اندازی را بچسبانید و وصل شوید.
5. دوباره در Telegram: `/pair pending` (شناسه‌های درخواست، نقش، و محدوده‌ها را بررسی کنید)، سپس تأیید کنید.

کد راه‌اندازی یک payload JSON کدگذاری‌شده با base64 است که شامل این موارد است:

- `url`: نشانی WebSocket مربوط به Gateway (`ws://...` یا `wss://...`)
- `bootstrapToken`: یک توکن راه‌انداز کوتاه‌مدتِ تک‌دستگاهی که برای دست‌دهی اولیهٔ جفت‌سازی استفاده می‌شود

آن توکن راه‌انداز پروفایل راه‌انداز جفت‌سازی داخلی را حمل می‌کند:

- پروفایل راه‌اندازی داخلی فقط خط پایهٔ تازهٔ QR/کد راه‌اندازی را اجازه می‌دهد:
  `node` به‌علاوهٔ یک واگذاری محدود `operator`
- توکن `node` واگذارشده روی `scopes: []` باقی می‌ماند
- توکن `operator` واگذارشده به `operator.approvals`,
  `operator.read`, `operator.talk.secrets`, و `operator.write` محدود است
- `operator.admin` با راه‌انداز QR/کد راه‌اندازی اعطا نمی‌شود؛ به یک
  جفت‌سازی operator تأییدشدهٔ جداگانه یا جریان توکن جداگانه نیاز دارد
- چرخش/ابطال بعدی توکن همچنان هم به قرارداد نقش تأییدشدهٔ دستگاه
  و هم به محدوده‌های operator نشست فراخواننده محدود می‌ماند

تا زمانی که کد راه‌اندازی معتبر است، با آن مثل رمز عبور رفتار کنید.

برای Tailscale، جفت‌سازی عمومی، یا دیگر جفت‌سازی‌های موبایل از راه دور، از Tailscale Serve/Funnel
یا یک نشانی Gateway دیگر با `wss://` استفاده کنید. کدهای راه‌اندازی متنی سادهٔ `ws://` فقط
برای loopback، نشانی‌های LAN خصوصی، میزبان‌های Bonjour با `.local`، و میزبان شبیه‌ساز Android
پذیرفته می‌شوند. نشانی‌های Tailnet CGNAT، نام‌های `.ts.net`، و میزبان‌های عمومی همچنان
پیش از صدور QR/کد راه‌اندازی fail closed می‌شوند.

### تأیید یک دستگاه Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

وقتی یک تأیید صریح رد می‌شود چون نشست دستگاه جفت‌شدهٔ تأییدکننده
با محدودهٔ فقط-جفت‌سازی باز شده بود، CLI همان درخواست را با
`operator.admin` دوباره امتحان می‌کند. این به یک دستگاه جفت‌شدهٔ موجود که توانایی ادمین دارد اجازه می‌دهد یک
جفت‌سازی تازهٔ Control UI/مرورگر را بدون ویرایش دستی `devices/paired.json` بازیابی کند.
Gateway همچنان اتصال تکرارشده را اعتبارسنجی می‌کند؛ توکن‌هایی که نتوانند
با `operator.admin` احراز هویت کنند مسدود می‌مانند.

اگر همان دستگاه با جزئیات احراز هویت متفاوت دوباره تلاش کند (برای مثال
نقش/محدوده‌ها/کلید عمومی متفاوت)، درخواست در انتظار قبلی جایگزین می‌شود و یک
`requestId` جدید ایجاد می‌شود.

<Note>
دستگاهی که از قبل جفت شده است، بی‌صدا دسترسی گسترده‌تری دریافت نمی‌کند. اگر دوباره متصل شود و محدوده‌های بیشتر یا نقش گسترده‌تری بخواهد، OpenClaw تأیید موجود را همان‌طور که هست نگه می‌دارد و یک درخواست ارتقای تازه در انتظار ایجاد می‌کند. پیش از تأیید، از `openclaw devices list` برای مقایسهٔ دسترسی فعلاً تأییدشده با دسترسی تازهٔ درخواست‌شده استفاده کنید.
</Note>

### تأیید خودکار اختیاری Node با CIDR مورد اعتماد

جفت‌سازی دستگاه به‌طور پیش‌فرض دستی می‌ماند. برای شبکه‌های Node به‌شدت کنترل‌شده،
می‌توانید با CIDRهای صریح یا IPهای دقیق، تأیید خودکار نخستین Node را فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این فقط برای درخواست‌های تازهٔ جفت‌سازی `role: node` بدون محدوده‌های درخواست‌شده
اعمال می‌شود. کلاینت‌های operator، مرورگر، Control UI، و WebChat همچنان به تأیید دستی
نیاز دارند. تغییرات نقش، محدوده، فراداده، و کلید عمومی همچنان به تأیید دستی
نیاز دارند.

### ذخیره‌سازی وضعیت جفت‌سازی Node

زیر `~/.openclaw/devices/` ذخیره می‌شود:

- `pending.json` (کوتاه‌مدت؛ درخواست‌های در انتظار منقضی می‌شوند)
- `paired.json` (دستگاه‌های جفت‌شده + توکن‌ها)

### یادداشت‌ها

- API قدیمی `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) یک
  مخزن جفت‌سازی جداگانه تحت مالکیت Gateway است. نودهای WS همچنان به جفت‌سازی دستگاه نیاز دارند.
- رکورد جفت‌سازی منبع حقیقت پایدار برای نقش‌های تأییدشده است. توکن‌های فعال
  دستگاه به همان مجموعه نقش تأییدشده محدود می‌مانند؛ یک ورودی توکن سرگردان
  خارج از نقش‌های تأییدشده دسترسی تازه‌ای ایجاد نمی‌کند.
