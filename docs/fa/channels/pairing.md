«جفت‌سازی» مرحلهٔ صریح تأیید دسترسی در OpenClaw است.
این مرحله در دو بخش استفاده می‌شود:

1. **جفت‌سازی پیام خصوصی** (چه کسی مجاز است با ربات گفتگو کند)
2. **جفت‌سازی Node** (کدام دستگاه‌ها/Nodeها مجازند به شبکهٔ Gateway بپیوندند)

زمینهٔ امنیتی: [امنیت](/fa/gateway/security)

## 1) جفت‌سازی پیام خصوصی (دسترسی به گفت‌وگوی ورودی)

وقتی کانالی با خط‌مشی پیام خصوصی `pairing` پیکربندی شده باشد، فرستندگان ناشناس یک کد کوتاه دریافت می‌کنند و پیامشان تا زمان تأیید شما **پردازش نمی‌شود**.

خط‌مشی‌های پیش‌فرض پیام خصوصی در اینجا مستند شده‌اند: [امنیت](/fa/gateway/security)

`dmPolicy: "open"` تنها زمانی عمومی است که فهرست مجاز مؤثر پیام خصوصی شامل `"*"` باشد.
راه‌اندازی و اعتبارسنجی برای پیکربندی‌های عمومی و باز به این نویسهٔ عام نیاز دارند. اگر وضعیت موجود
شامل `open` با ورودی‌های مشخص `allowFrom` باشد، زمان اجرا همچنان فقط
همان فرستندگان را می‌پذیرد و تأییدهای مخزن جفت‌سازی دسترسی `open` را گسترش نمی‌دهند.

کدهای جفت‌سازی:

- ۸ نویسه، با حروف بزرگ و بدون نویسه‌های مبهم (`0O1I`).
- **پس از ۱ ساعت منقضی می‌شوند**. ربات فقط هنگام ایجاد درخواست جدید، پیام جفت‌سازی را ارسال می‌کند (تقریباً ساعتی یک‌بار برای هر فرستنده).
- درخواست‌های در انتظار جفت‌سازی پیام خصوصی به **۳ مورد برای هر حساب کانال** محدود می‌شوند؛ درخواست‌های اضافی تا زمان انقضا یا تأیید یکی از آن‌ها نادیده گرفته می‌شوند.

### تأیید یک فرستنده

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

برای اطلاع‌دادن به درخواست‌کننده در همان کانال، `--notify` را به فرمان تأیید اضافه کنید. کانال‌های چندحسابی `--account <id>` را می‌پذیرند.

اگر هنوز مالک فرمانی پیکربندی نشده باشد، تأیید کد جفت‌سازی پیام خصوصی همچنین
`commands.ownerAllowFrom` را با فرستندهٔ تأییدشده، مانند `telegram:123456789`، مقداردهی اولیه می‌کند.
این کار در راه‌اندازی‌های نخستین، مالکی صریح برای فرمان‌های دارای امتیاز و اعلان‌های
تأیید اجرا فراهم می‌کند. پس از ایجاد مالک، تأییدهای جفت‌سازی بعدی فقط دسترسی پیام خصوصی
می‌دهند و مالک دیگری اضافه نمی‌کنند.

کانال‌های پشتیبانی‌شده (هر Plugin کانال نصب‌شده‌ای که جفت‌سازی را اعلام کند؛ Pluginهای خارجی مانند `openclaw-weixin` می‌توانند موارد بیشتری اضافه کنند): `discord`، `feishu`، `googlechat`، `imessage`، `irc`، `line`، `matrix`، `mattermost`، `msteams`، `nextcloud-talk`، `nostr`، `signal`، `slack`، `sms`، `synology-chat`، `telegram`، `twitch`، `whatsapp`، `zalo`، `zalouser`.

### گروه‌های فرستندهٔ قابل‌استفادهٔ مجدد

وقتی قرار است مجموعهٔ یکسانی از فرستندگان مورداعتماد برای چندین کانال پیام یا
هم برای فهرست‌های مجاز پیام خصوصی و هم گروه اعمال شود، از `accessGroups` سطح‌بالا استفاده کنید.

گروه‌های ایستا از `type: "message.senders"` استفاده می‌کنند و با
`accessGroup:<name>` در فهرست‌های مجاز کانال ارجاع داده می‌شوند:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

گروه‌های دسترسی با جزئیات در اینجا مستند شده‌اند: [گروه‌های دسترسی](/fa/channels/access-groups)

### محل نگهداری وضعیت

در پایگاه دادهٔ وضعیت مشترک SQLite در
`~/.openclaw/state/openclaw.sqlite` ذخیره می‌شود:

- درخواست‌های در انتظار در `channel_pairing_requests`
- فرستندگان تأییدشده در `channel_pairing_allow_entries`

رفتار محدوده‌بندی حساب:

- هر درخواست و فرستندهٔ تأییدشده بر اساس کانال و حساب کلیدگذاری می‌شود
- زمان اجرا فقط ردیف‌های متعارف SQLite را می‌خواند و فایل‌های قدیمی را ادغام نمی‌کند

Gatewayهای قدیمی‌تر `<channel>-pairing.json` و
`<channel>-<accountId>-allowFrom.json` را زیر `~/.openclaw/credentials/` می‌نوشتند.
مهاجرت هنگام راه‌اندازی و `openclaw doctor --fix` این فایل‌ها را به SQLite وارد می‌کنند و
هر منبع را پس از واردکردن موفق حذف می‌کنند. پایگاه دادهٔ SQLite را
حساس تلقی کنید، زیرا این ردیف‌ها دسترسی به دستیار شما را کنترل می‌کنند.

<Note>
مخزن فهرست مجاز جفت‌سازی برای دسترسی پیام خصوصی است. مجوزدهی گروه جداگانه است.
تأیید یک کد جفت‌سازی پیام خصوصی، آن فرستنده را به‌طور خودکار مجاز نمی‌کند که فرمان‌های گروهی
را اجرا کند یا ربات را در گروه‌ها کنترل کند. مقداردهی اولیهٔ مالک نخست، وضعیت پیکربندی
جداگانه‌ای در `commands.ownerAllowFrom` است و تحویل گفت‌وگوی گروهی همچنان از
فهرست‌های مجاز گروهی کانال پیروی می‌کند (برای مثال `groupAllowFrom`، `groups`، یا بازنویسی‌های
مختص هر گروه یا موضوع، بسته به کانال).
</Note>

## 2) جفت‌سازی دستگاه Node (Nodeهای iOS/Android/macOS/بدون رابط)

Nodeها به‌عنوان **دستگاه** با `role: node` به Gateway متصل می‌شوند. Gateway
یک درخواست جفت‌سازی دستگاه ایجاد می‌کند که باید تأیید شود.

### جفت‌سازی از رابط کاربری کنترل (توصیه‌شده)

از یک نشست ازپیش‌متصل رابط کاربری کنترل با دسترسی `operator.admin` استفاده کنید:

1. رابط کاربری کنترل را باز کنید و به **Settings → Devices** بروید.
2. در صفحهٔ **Devices**، روی **Pair mobile device** کلیک کنید.
3. گزینهٔ **Full access (recommended)** را نگه دارید یا برای حذف
   کنترل‌های مدیریتی Gateway، **Limited access** را انتخاب کنید.
4. روی **Create setup code** کلیک کنید.
5. در تلفن خود، برنامهٔ OpenClaw را باز کنید ← **Settings** ← **Gateway**.
6. کد QR را اسکن یا کد راه‌اندازی را جای‌گذاری کنید، سپس متصل شوید.

برنامه‌های رسمی OpenClaw برای iOS و Android وقتی فرادادهٔ کد راه‌اندازی آن‌ها
مطابقت داشته باشد، به‌طور خودکار تأیید می‌شوند. اگر **Pending approval** درخواستی را نشان می‌دهد
(برای مثال، برای کارخواهی غیررسمی یا فرادادهٔ نامطابق)، پیش از تأیید، نقش و
دامنه‌های آن را بررسی کنید.

وقتی نشست فعلی رابط کاربری کنترل دسترسی مدیر نداشته باشد، دکمه غیرفعال است.
در این حالت از جریان تأیید CLI زیر در میزبان Gateway استفاده کنید.

### جفت‌سازی از طریق Telegram

اگر از Plugin `device-pair` استفاده می‌کنید، می‌توانید نخستین جفت‌سازی دستگاه را کاملاً از طریق Telegram انجام دهید:

1. در Telegram، این پیام را برای ربات خود بفرستید: `/pair`
2. ربات با دو پیام پاسخ می‌دهد: یک پیام راهنما و یک پیام جداگانهٔ **کد راه‌اندازی** (برای کپی/جای‌گذاری آسان در Telegram).
3. در تلفن خود، برنامهٔ OpenClaw برای iOS را باز کنید ← Settings ← Gateway.
4. کد QR (`/pair qr`) را اسکن کنید یا کد راه‌اندازی را جای‌گذاری و متصل شوید.
5. برنامهٔ رسمی موبایل به‌طور خودکار متصل می‌شود. اگر `/pair pending` درخواستی را نشان می‌دهد،
   پیش از تأیید، نقش و دامنه‌های آن را بررسی کنید.

کد راه‌اندازی یک بار دادهٔ JSON با کدگذاری base64 است که شامل موارد زیر می‌شود:

- `url`: نشانی WebSocket مربوط به Gateway (`ws://...` یا `wss://...`)
- `urls`: در صورت موجودبودن، مسیرهای مرتب‌شدهٔ LAN/Tailnet که برنامهٔ موبایل می‌تواند امتحان کند
- `bootstrapToken`: یک توکن راه‌اندازی اولیهٔ یک‌بارمصرف برای دست‌دهی جفت‌سازی آغازین؛ Gateway آن را پس از 10 دقیقه منقضی می‌کند

پس از پایان جفت‌سازی، برای بی‌اعتبارکردن کدهای راه‌اندازی استفاده‌نشده، `/pair cleanup` را اجرا کنید.

آن توکن راه‌اندازی اولیه، نمایهٔ داخلی راه‌اندازی اولیهٔ جفت‌سازی را حمل می‌کند:

- یک راه‌اندازی امن `wss://` (یا loopback همان میزبان) به‌طور پیش‌فرض `node` به‌علاوهٔ دسترسی کامل
  `operator` موبایل بومی را فراهم می‌کند
- توکن واگذارشدهٔ `node` همچنان `scopes: []` باقی می‌ماند
- توکن واگذارشدهٔ پیش‌فرض `operator` شامل `operator.admin`،
  `operator.approvals`، `operator.read`، `operator.talk.secrets` و
  `operator.write` است
- گزینهٔ **Limited access** رابط کاربری کنترل و `openclaw qr --limited`
  `operator.admin` را حذف می‌کنند، درحالی‌که سایر دامنه‌های اپراتور را نگه می‌دارند
- راه‌اندازی متنی سادهٔ LAN با `ws://` به‌طور خودکار از همان نمایهٔ محدود استفاده می‌کند؛
  برای دسترسی کامل، `wss://` یا Tailscale Serve را پیکربندی و کدی جدید ایجاد کنید
- چرخش/ابطال بعدی توکن همچنان هم به قرارداد نقش تأییدشدهٔ دستگاه
  و هم به دامنه‌های اپراتور نشست فراخواننده محدود می‌ماند

تا زمانی که کد راه‌اندازی معتبر است، با آن مانند گذرواژه رفتار کنید.

صفحه‌های **Settings → Gateway** در iOS و Android دسترسی **Full** یا **Limited**
را نشان می‌دهند. برای ارتقای تلفنی با دسترسی محدود، ابتدا یک مسیر امن `wss://` یا
Tailscale Serve را پیکربندی کنید، سپس یک کد راه‌اندازی جدید با دسترسی کامل ایجاد کنید، آن را
در همان صفحهٔ تنظیمات اسکن یا جای‌گذاری کنید و دوباره متصل شوید.

برای جفت‌سازی موبایل از طریق Tailscale، عمومی یا سایر مسیرهای دوردست، از Tailscale Serve/Funnel
یا نشانی Gateway دیگری با `wss://` استفاده کنید. کدهای راه‌اندازی متنی سادهٔ `ws://` فقط
برای loopback، نشانی‌های LAN خصوصی، میزبان‌های Bonjour با `.local` و میزبان
شبیه‌ساز Android پذیرفته می‌شوند. مسیرهای متنی سادهٔ غیر-loopback دسترسی محدود دریافت می‌کنند. نشانی‌های
CGNAT در Tailnet، نام‌های `.ts.net` و میزبان‌های عمومی همچنان پیش از
صدور کد QR/راه‌اندازی به‌صورت امن رد می‌شوند.

برای نشانی‌های راه‌اندازی `gateway.bind=lan`، OpenClaw ریشه‌های HTTPS پایدار Tailscale Serve
را که درگاه loopback مربوط به Gateway فعال را پراکسی می‌کنند، شناسایی کرده و آن‌ها را
در کنار مسیر LAN اعلام می‌کند. فرمان راه‌اندازی این مسیر جایگزین را فقط
برای `lan` اضافه می‌کند؛ `custom` و `tailnet` مسیرهای صریحاً اعلام‌شدهٔ خود را حفظ می‌کنند. برنامهٔ
iOS مسیرهای اعلام‌شده را به‌ترتیب بررسی و نخستین نقطهٔ پایانی در دسترس را ذخیره می‌کند.

### تأیید یک دستگاه Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

وقتی تأیید صریح به این دلیل رد شود که نشست دستگاه جفت‌شدهٔ تأییدکننده
فقط با دامنهٔ جفت‌سازی باز شده است، CLI همان درخواست را با
`operator.admin` دوباره امتحان می‌کند. این کار به دستگاه جفت‌شدهٔ موجود با قابلیت مدیریت اجازه می‌دهد جفت‌سازی جدید
رابط کاربری کنترل/مرورگر را بدون ویرایش دستی مخزن جفت‌سازی بازیابی کند. Gateway
همچنان اتصال تکرارشده را اعتبارسنجی می‌کند؛ توکن‌هایی که نمی‌توانند با
`operator.admin` احراز هویت کنند، مسدود می‌مانند.

اگر همان دستگاه با جزئیات احراز هویت متفاوت دوباره تلاش کند (برای مثال نقش/دامنه‌ها/کلید عمومی
متفاوت)، درخواست در انتظار قبلی جایگزین و یک
`requestId` جدید ایجاد می‌شود.

<Note>
دستگاهی که از قبل جفت شده است، بی‌سروصدا دسترسی گسترده‌تری دریافت نمی‌کند. اگر هنگام اتصال مجدد دامنه‌های بیشتر یا نقش گسترده‌تری درخواست کند، OpenClaw تأیید موجود را بدون تغییر نگه می‌دارد و یک درخواست ارتقای در انتظار جدید ایجاد می‌کند. پیش از تأیید، از `openclaw devices list` برای مقایسهٔ دسترسی تأییدشدهٔ فعلی با دسترسی تازه‌درخواست‌شده استفاده کنید.
</Note>

### تأیید خودکار اختیاری Node با CIDR مورداعتماد

جفت‌سازی دستگاه به‌طور پیش‌فرض دستی باقی می‌ماند. برای شبکه‌های Node کاملاً کنترل‌شده،
می‌توانید تأیید خودکار نخستین Node را با CIDRها یا IPهای دقیق و صریح فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این فقط برای درخواست‌های جفت‌سازی جدید `role: node` بدون دامنه‌های
درخواست‌شده اعمال می‌شود. کارخواه‌های اپراتور، مرورگر، رابط کاربری کنترل و WebChat همچنان به تأیید دستی
نیاز دارند. تغییرات نقش، دامنه، فراداده و کلید عمومی نیز همچنان به تأیید دستی
نیاز دارند.

### ذخیره‌سازی وضعیت جفت‌سازی Node

در پایگاه دادهٔ وضعیت مشترک SQLite در `~/.openclaw/state/openclaw.sqlite` ذخیره می‌شود:

- درخواست‌های در انتظار جفت‌سازی دستگاه (کوتاه‌عمر؛ پس از 5 دقیقه منقضی می‌شوند)
- دستگاه‌های جفت‌شده + توکن‌ها

Gatewayهای قدیمی‌تر این وضعیت را در `~/.openclaw/devices/*.json` نگه می‌داشتند؛ آن فایل‌ها
هنگام راه‌اندازی Gateway به SQLite وارد و با پسوند `.migrated` بایگانی می‌شوند.

### نکات

- API مربوط به `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`)
  تأییدهای قابلیت Node ذخیره‌شده در همان رکوردهای دستگاه جفت‌شده را مدیریت می‌کند. Nodeهای WS
  همچنان به جفت‌سازی دستگاه نیاز دارند؛ [جفت‌سازی Node](/fa/gateway/pairing) را ببینید.
- رکورد جفت‌سازی، منبع حقیقت پایدار برای نقش‌های تأییدشده است. توکن‌های فعال
  دستگاه به همان مجموعهٔ نقش‌های تأییدشده محدود می‌مانند؛ یک ورودی توکن سرگردان
  خارج از نقش‌های تأییدشده، دسترسی جدیدی ایجاد نمی‌کند.
