«جفت‌سازی» مرحلهٔ صریح تأیید دسترسی در OpenClaw است.
این مرحله در دو جا استفاده می‌شود:

1. **جفت‌سازی DM** (چه کسی اجازه دارد با ربات صحبت کند)
2. **جفت‌سازی Node** (کدام دستگاه‌ها/Nodeها اجازه دارند به شبکهٔ Gateway بپیوندند)

زمینهٔ امنیتی: [امنیت](/fa/gateway/security)

## 1) جفت‌سازی DM (دسترسی چت ورودی)

وقتی یک کانال با سیاست DM به صورت `pairing` پیکربندی شده باشد، فرستنده‌های ناشناس یک کد کوتاه دریافت می‌کنند و پیام آن‌ها تا زمانی که تأیید نکنید **پردازش نمی‌شود**.

سیاست‌های پیش‌فرض DM در اینجا مستند شده‌اند: [امنیت](/fa/gateway/security)

`dmPolicy: "open"` فقط زمانی عمومی است که لیست مجاز مؤثر DM شامل `"*"` باشد.
راه‌اندازی و اعتبارسنجی برای پیکربندی‌های عمومی-باز به آن wildcard نیاز دارند. اگر وضعیت موجود شامل `open` همراه با ورودی‌های مشخص `allowFrom` باشد، runtime همچنان فقط همان فرستنده‌ها را می‌پذیرد، و تأییدهای pairing-store دسترسی `open` را گسترده‌تر نمی‌کنند.

کدهای جفت‌سازی:

- 8 نویسه، حروف بزرگ، بدون نویسه‌های مبهم (`0O1I`).
- **پس از 1 ساعت منقضی می‌شوند**. ربات فقط وقتی پیام جفت‌سازی را می‌فرستد که یک درخواست جدید ایجاد شود (تقریباً هر ساعت یک‌بار برای هر فرستنده).
- درخواست‌های در انتظار جفت‌سازی DM به‌طور پیش‌فرض به **3 مورد برای هر کانال** محدود می‌شوند؛ درخواست‌های اضافی تا زمانی که یکی منقضی یا تأیید شود نادیده گرفته می‌شوند.

### تأیید یک فرستنده

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

اگر هنوز مالک فرمانی پیکربندی نشده باشد، تأیید یک کد جفت‌سازی DM همچنین `commands.ownerAllowFrom` را با فرستندهٔ تأییدشده راه‌اندازی اولیه می‌کند، مانند `telegram:123456789`.
این کار به راه‌اندازی‌های اولین‌بار یک مالک صریح برای فرمان‌های دارای امتیاز و درخواست‌های تأیید exec می‌دهد. پس از اینکه یک مالک وجود داشته باشد، تأییدهای جفت‌سازی بعدی فقط دسترسی DM می‌دهند؛ آن‌ها مالک‌های بیشتری اضافه نمی‌کنند.

کانال‌های پشتیبانی‌شده: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### گروه‌های فرستندهٔ قابل استفادهٔ مجدد

وقتی همان مجموعهٔ فرستندگان مورد اعتماد باید برای چند کانال پیام یا برای هر دو لیست مجاز DM و گروه اعمال شود، از `accessGroups` سطح بالا استفاده کنید.

گروه‌های ایستا از `type: "message.senders"` استفاده می‌کنند و با `accessGroup:<name>` از لیست‌های مجاز کانال ارجاع داده می‌شوند:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

گروه‌های دسترسی به‌صورت مفصل اینجا مستند شده‌اند: [گروه‌های دسترسی](/fa/channels/access-groups)

### وضعیت کجا نگهداری می‌شود

زیر `~/.openclaw/credentials/` ذخیره می‌شود:

- درخواست‌های در انتظار: `<channel>-pairing.json`
- مخزن لیست مجاز تأییدشده:
  - حساب پیش‌فرض: `<channel>-allowFrom.json`
  - حساب غیراپیش‌فرض: `<channel>-<accountId>-allowFrom.json`

رفتار محدوده‌بندی حساب:

- حساب‌های غیراپیش‌فرض فقط فایل لیست مجاز محدوده‌بندی‌شدهٔ خودشان را می‌خوانند/می‌نویسند.
- حساب پیش‌فرض از فایل لیست مجاز بدون محدوده‌بندیِ مختص کانال استفاده می‌کند.

با این موارد به‌عنوان دادهٔ حساس رفتار کنید (آن‌ها دسترسی به دستیار شما را کنترل می‌کنند).

<Note>
مخزن لیست مجاز جفت‌سازی برای دسترسی DM است. مجوزدهی گروه جداگانه است.
تأیید یک کد جفت‌سازی DM به‌طور خودکار به آن فرستنده اجازه نمی‌دهد فرمان‌های گروهی اجرا کند یا ربات را در گروه‌ها کنترل کند. راه‌اندازی اولیهٔ مالک اول وضعیت پیکربندی جداگانه‌ای در `commands.ownerAllowFrom` است، و تحویل چت گروهی همچنان از لیست‌های مجاز گروهِ کانال پیروی می‌کند (برای مثال `groupAllowFrom`، `groups`، یا بازنویسی‌های هر گروه یا هر موضوع بسته به کانال).
</Note>

## 2) جفت‌سازی دستگاه Node (Nodeهای iOS/Android/macOS/بدون رابط)

Nodeها به‌عنوان **دستگاه** با `role: node` به Gateway وصل می‌شوند. Gateway یک درخواست جفت‌سازی دستگاه ایجاد می‌کند که باید تأیید شود.

### جفت‌سازی از طریق Telegram (توصیه‌شده برای iOS)

اگر از Plugin `device-pair` استفاده می‌کنید، می‌توانید جفت‌سازی اولیهٔ دستگاه را کاملاً از Telegram انجام دهید:

1. در Telegram، به ربات خود پیام دهید: `/pair`
2. ربات با دو پیام پاسخ می‌دهد: یک پیام راهنما و یک پیام جداگانهٔ **کد راه‌اندازی** (برای کپی/چسباندن در Telegram آسان است).
3. در تلفن خود، برنامهٔ OpenClaw iOS را باز کنید ← Settings ← Gateway.
4. کد QR را اسکن کنید یا کد راه‌اندازی را جای‌گذاری کنید و وصل شوید.
5. دوباره در Telegram: `/pair pending` (شناسه‌های درخواست، نقش، و محدوده‌ها را بازبینی کنید)، سپس تأیید کنید.

کد راه‌اندازی یک payload JSON کدگذاری‌شده با base64 است که شامل این موارد است:

- `url`: نشانی WebSocket مربوط به Gateway (`ws://...` یا `wss://...`)
- `bootstrapToken`: یک توکن راه‌اندازی اولیهٔ تک‌دستگاهی با عمر کوتاه که برای handshake اولیهٔ جفت‌سازی استفاده می‌شود

آن توکن راه‌اندازی اولیه پروفایل داخلی راه‌اندازی اولیهٔ جفت‌سازی را حمل می‌کند:

- توکن اصلی واگذارشدهٔ `node` همچنان `scopes: []` باقی می‌ماند
- هر توکن واگذارشدهٔ `operator` همچنان به لیست مجاز راه‌اندازی اولیه محدود می‌ماند:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- بررسی‌های محدودهٔ راه‌اندازی اولیه با پیشوند نقش انجام می‌شوند، نه یک مخزن تخت از محدوده‌ها:
  ورودی‌های محدودهٔ operator فقط درخواست‌های operator را برآورده می‌کنند، و نقش‌های غیر operator همچنان باید محدوده‌ها را زیر پیشوند نقش خودشان درخواست کنند
- چرخش/ابطال بعدی توکن همچنان هم به قرارداد نقش تأییدشدهٔ دستگاه و هم به محدوده‌های operator نشست فراخواننده محدود می‌ماند

تا زمانی که کد راه‌اندازی معتبر است، با آن مثل رمز عبور رفتار کنید.

برای جفت‌سازی موبایل از راه دور از طریق Tailscale، عمومی، یا روش‌های دیگر، از Tailscale Serve/Funnel یا یک نشانی Gateway دیگر با `wss://` استفاده کنید. کدهای راه‌اندازی متن سادهٔ `ws://` فقط برای local loopback، نشانی‌های LAN خصوصی، میزبان‌های Bonjour با `.local`، و میزبان شبیه‌ساز Android پذیرفته می‌شوند. نشانی‌های Tailnet CGNAT، نام‌های `.ts.net`، و میزبان‌های عمومی همچنان پیش از صدور QR/کد راه‌اندازی به‌صورت بسته شکست می‌خورند.

### تأیید یک دستگاه Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

وقتی یک تأیید صریح رد می‌شود چون نشست دستگاه جفت‌شدهٔ تأییدکننده با محدودهٔ فقط جفت‌سازی باز شده بود، CLI همان درخواست را با `operator.admin` دوباره امتحان می‌کند. این کار به یک دستگاه جفت‌شدهٔ موجود با قابلیت admin اجازه می‌دهد بدون ویرایش دستی `devices/paired.json` یک جفت‌سازی جدید Control UI/مرورگر را بازیابی کند. Gateway همچنان اتصال دوباره‌امتحان‌شده را اعتبارسنجی می‌کند؛ توکن‌هایی که نمی‌توانند با `operator.admin` احراز هویت شوند همچنان مسدود می‌مانند.

اگر همان دستگاه با جزئیات احراز هویت متفاوت دوباره تلاش کند (برای مثال نقش/محدوده‌ها/کلید عمومی متفاوت)، درخواست در انتظار قبلی کنار گذاشته می‌شود و یک `requestId` جدید ایجاد می‌شود.

<Note>
یک دستگاهی که از قبل جفت شده است بی‌صدا دسترسی گسترده‌تری نمی‌گیرد. اگر دوباره وصل شود و محدوده‌های بیشتر یا نقش گسترده‌تری بخواهد، OpenClaw تأیید موجود را همان‌طور نگه می‌دارد و یک درخواست ارتقای تازه در حالت انتظار ایجاد می‌کند. پیش از تأیید، از `openclaw devices list` برای مقایسهٔ دسترسی تأییدشدهٔ فعلی با دسترسی تازه درخواست‌شده استفاده کنید.
</Note>

### تأیید خودکار اختیاری Node با CIDR مورد اعتماد

جفت‌سازی دستگاه به‌طور پیش‌فرض دستی باقی می‌ماند. برای شبکه‌های Node کاملاً کنترل‌شده، می‌توانید با CIDRهای صریح یا IPهای دقیق، تأیید خودکار اولین‌بار Node را فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این فقط برای درخواست‌های تازهٔ جفت‌سازی `role: node` بدون محدودهٔ درخواستی اعمال می‌شود. کلاینت‌های Operator، مرورگر، Control UI، و WebChat همچنان به تأیید دستی نیاز دارند. تغییرات نقش، محدوده، فراداده، و کلید عمومی همچنان به تأیید دستی نیاز دارند.

### ذخیره‌سازی وضعیت جفت‌سازی Node

زیر `~/.openclaw/devices/` ذخیره می‌شود:

- `pending.json` (کوتاه‌عمر؛ درخواست‌های در انتظار منقضی می‌شوند)
- `paired.json` (دستگاه‌های جفت‌شده + توکن‌ها)

### یادداشت‌ها

- API قدیمی `node.pair.*` ‏(CLI: `openclaw nodes pending|approve|reject|remove|rename`) یک مخزن جفت‌سازی جداگانه و متعلق به Gateway است. Nodeهای WS همچنان به جفت‌سازی دستگاه نیاز دارند.
- رکورد جفت‌سازی منبع پایدار حقیقت برای نقش‌های تأییدشده است. توکن‌های فعال دستگاه به همان مجموعه نقش تأییدشده محدود می‌مانند؛ یک ورودی توکن پراکنده خارج از نقش‌های تأییدشده دسترسی جدید ایجاد نمی‌کند.
