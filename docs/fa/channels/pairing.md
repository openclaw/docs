«جفت‌سازی» مرحله‌ی تأیید صریح دسترسی در OpenClaw است.
در دو جا استفاده می‌شود:

1. **جفت‌سازی DM** (چه کسی مجاز است با بات صحبت کند)
2. **جفت‌سازی Node** (کدام دستگاه‌ها/Nodeها مجازند به شبکه‌ی Gateway بپیوندند)

زمینه‌ی امنیتی: [امنیت](/fa/gateway/security)

## 1) جفت‌سازی DM (دسترسی چت ورودی)

وقتی یک کانال با سیاست DM به‌صورت `pairing` پیکربندی شده باشد، فرستنده‌های ناشناس یک کد کوتاه دریافت می‌کنند و پیام آن‌ها تا زمانی که شما تأیید نکنید **پردازش نمی‌شود**.

سیاست‌های پیش‌فرض DM در اینجا مستند شده‌اند: [امنیت](/fa/gateway/security)

`dmPolicy: "open"` فقط وقتی عمومی است که allowlist مؤثر DM شامل `"*"` باشد.
راه‌اندازی و اعتبارسنجی برای پیکربندی‌های عمومی-باز به این wildcard نیاز دارند. اگر وضعیت موجود شامل `open` همراه با ورودی‌های مشخص `allowFrom` باشد، زمان اجرا همچنان فقط همان فرستنده‌ها را می‌پذیرد، و تأییدهای pairing-store دسترسی `open` را گسترده‌تر نمی‌کنند.

کدهای جفت‌سازی:

- ۸ نویسه، حروف بزرگ، بدون نویسه‌های مبهم (`0O1I`).
- **پس از ۱ ساعت منقضی می‌شوند**. بات فقط وقتی پیام جفت‌سازی را می‌فرستد که یک درخواست جدید ایجاد شود (تقریباً هر ساعت یک‌بار برای هر فرستنده).
- درخواست‌های جفت‌سازی DM در انتظار، به‌صورت پیش‌فرض به **۳ درخواست برای هر کانال** محدود می‌شوند؛ درخواست‌های بیشتر تا وقتی یکی منقضی یا تأیید شود نادیده گرفته می‌شوند.

### تأیید یک فرستنده

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

اگر هنوز هیچ مالک فرمانی پیکربندی نشده باشد، تأیید یک کد جفت‌سازی DM همچنین
`commands.ownerAllowFrom` را با فرستنده‌ی تأییدشده راه‌اندازی اولیه می‌کند، مثل `telegram:123456789`.
این کار برای راه‌اندازی‌های نخستین، یک مالک صریح برای فرمان‌های ممتاز و درخواست‌های تأیید اجرا فراهم می‌کند. پس از وجود یک مالک، تأییدهای بعدی جفت‌سازی فقط دسترسی DM می‌دهند؛ مالک‌های بیشتری اضافه نمی‌کنند.

کانال‌های پشتیبانی‌شده: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### وضعیت کجا ذخیره می‌شود

زیر `~/.openclaw/credentials/` ذخیره می‌شود:

- درخواست‌های در انتظار: `<channel>-pairing.json`
- محل ذخیره‌ی allowlist تأییدشده:
  - حساب پیش‌فرض: `<channel>-allowFrom.json`
  - حساب غیردیفالت: `<channel>-<accountId>-allowFrom.json`

رفتار محدوده‌بندی حساب:

- حساب‌های غیردیفالت فقط فایل allowlist محدوده‌بندی‌شده‌ی خودشان را می‌خوانند/می‌نویسند.
- حساب پیش‌فرض از فایل allowlist بدون محدوده‌بندیِ مخصوص کانال استفاده می‌کند.

با این موارد مانند داده‌های حساس رفتار کنید (آن‌ها دسترسی به دستیار شما را کنترل می‌کنند).

<Note>
محل ذخیره‌ی allowlist جفت‌سازی برای دسترسی DM است. مجوزدهی گروه جداست.
تأیید یک کد جفت‌سازی DM به‌طور خودکار به آن فرستنده اجازه نمی‌دهد فرمان‌های گروهی را اجرا کند یا بات را در گروه‌ها کنترل کند. راه‌اندازی اولیه‌ی اولین مالک، وضعیت پیکربندی جداگانه‌ای در `commands.ownerAllowFrom` است، و تحویل چت گروهی همچنان از allowlistهای گروهی کانال پیروی می‌کند (برای مثال `groupAllowFrom`، `groups`، یا overrideهای هر گروه یا هر topic بسته به کانال).
</Note>

## 2) جفت‌سازی دستگاه Node ‏(iOS/Android/macOS/Nodeهای بدون واسط)

Nodeها به‌عنوان **دستگاه** با `role: node` به Gateway وصل می‌شوند. Gateway یک درخواست جفت‌سازی دستگاه ایجاد می‌کند که باید تأیید شود.

### جفت‌سازی از طریق Telegram (توصیه‌شده برای iOS)

اگر از Plugin ‏`device-pair` استفاده می‌کنید، می‌توانید جفت‌سازی اولیه‌ی دستگاه را کاملاً از داخل Telegram انجام دهید:

1. در Telegram، به بات خود پیام بدهید: `/pair`
2. بات با دو پیام پاسخ می‌دهد: یک پیام راهنما و یک پیام جداگانه‌ی **کد راه‌اندازی** (برای کپی/پیست در Telegram آسان است).
3. روی تلفن خود، برنامه‌ی OpenClaw iOS را باز کنید ← Settings ← Gateway.
4. کد راه‌اندازی را بچسبانید و وصل شوید.
5. دوباره در Telegram: `/pair pending` (شناسه‌های درخواست، نقش، و scopeها را بررسی کنید)، سپس تأیید کنید.

کد راه‌اندازی یک payload JSON رمزگذاری‌شده با base64 است که شامل این موارد است:

- `url`: نشانی WebSocket برای Gateway (`ws://...` یا `wss://...`)
- `bootstrapToken`: یک توکن bootstrap کوتاه‌عمر و تک‌دستگاهی که برای handshake اولیه‌ی جفت‌سازی استفاده می‌شود

آن توکن bootstrap پروفایل داخلی bootstrap جفت‌سازی را حمل می‌کند:

- توکن واگذارشده‌ی اصلی `node` روی `scopes: []` باقی می‌ماند
- هر توکن واگذارشده‌ی `operator` همچنان به allowlist bootstrap محدود می‌ماند:
  `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- بررسی‌های scope در bootstrap با پیشوند نقش انجام می‌شوند، نه یک مخزن تخت از scopeها:
  ورودی‌های scope اپراتور فقط درخواست‌های اپراتور را برآورده می‌کنند، و نقش‌های غیر اپراتور همچنان باید scopeها را زیر پیشوند نقش خودشان درخواست کنند
- چرخش/ابطال بعدی توکن همچنان هم به قرارداد نقش تأییدشده‌ی دستگاه و هم به scopeهای اپراتورِ نشست فراخواننده محدود می‌ماند

تا زمانی که کد راه‌اندازی معتبر است، با آن مانند یک گذرواژه رفتار کنید.

### تأیید یک دستگاه Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

اگر همان دستگاه با جزئیات احراز هویت متفاوت دوباره تلاش کند (برای مثال نقش/scopeها/کلید عمومی متفاوت)، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ایجاد می‌شود.

<Note>
یک دستگاهی که قبلاً جفت شده است، بی‌سروصدا دسترسی گسترده‌تر دریافت نمی‌کند. اگر دوباره وصل شود و scopeهای بیشتر یا نقش گسترده‌تری بخواهد، OpenClaw تأیید موجود را همان‌طور که هست نگه می‌دارد و یک درخواست ارتقای تازه در حالت انتظار ایجاد می‌کند. پیش از تأیید، از `openclaw devices list` برای مقایسه‌ی دسترسی تأییدشده‌ی فعلی با دسترسی تازه‌درخواست‌شده استفاده کنید.
</Note>

### تأیید خودکار اختیاری Node با CIDR مورد اعتماد

جفت‌سازی دستگاه به‌صورت پیش‌فرض دستی می‌ماند. برای شبکه‌های Node کاملاً کنترل‌شده،
می‌توانید با CIDRهای صریح یا IPهای دقیق، تأیید خودکار Node در اولین استفاده را فعال کنید:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

این فقط برای درخواست‌های جفت‌سازی تازه‌ی `role: node` بدون scope درخواستی اعمال می‌شود. کلاینت‌های اپراتور، مرورگر، Control UI، و WebChat همچنان به تأیید دستی نیاز دارند. تغییرات نقش، scope، فراداده، و کلید عمومی همچنان به تأیید دستی نیاز دارند.

### ذخیره‌سازی وضعیت جفت‌سازی Node

زیر `~/.openclaw/devices/` ذخیره می‌شود:

- `pending.json` (کوتاه‌عمر؛ درخواست‌های در انتظار منقضی می‌شوند)
- `paired.json` (دستگاه‌های جفت‌شده + توکن‌ها)

### نکته‌ها

- API قدیمی `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) یک
  محل ذخیره‌ی جفت‌سازی جداگانه و متعلق به gateway است. Nodeهای WS همچنان به جفت‌سازی دستگاه نیاز دارند.
- رکورد جفت‌سازی منبع حقیقت پایدار برای نقش‌های تأییدشده است. توکن‌های فعال دستگاه به همان مجموعه نقش تأییدشده محدود می‌مانند؛ یک ورودی توکن پراکنده خارج از نقش‌های تأییدشده، دسترسی جدید ایجاد نمی‌کند.
