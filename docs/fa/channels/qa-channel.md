`qa-channel` یک انتقال پیام مصنوعی بسته‌بندی‌شده برای QA خودکار OpenClaw است. این یک کانال تولیدی نیست — وجود دارد تا همان مرز Plugin کانال را که انتقال‌دهنده‌های واقعی استفاده می‌کنند تمرین دهد، در حالی که وضعیت را قطعی و کاملا قابل بازرسی نگه می‌دارد.

## چه کاری انجام می‌دهد

- دستور زبان هدف در سطح Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `thread:<room>/<thread>`
- گذرگاه مصنوعی مبتنی بر HTTP برای تزریق پیام ورودی، ثبت رونوشت خروجی، ایجاد رشته، واکنش‌ها، ویرایش‌ها، حذف‌ها، و کنش‌های جست‌وجو/خواندن.
- اجراکننده خودآزمایی سمت میزبان که یک گزارش Markdown را در `.artifacts/qa-e2e/` می‌نویسد.

## پیکربندی

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

کلیدهای حساب:

- `enabled` — کلید اصلی فعال‌سازی برای این حساب.
- `name` — برچسب نمایشی اختیاری.
- `baseUrl` — URL گذرگاه مصنوعی.
- `botUserId` — شناسه کاربر بات به سبک Matrix که در دستور زبان هدف استفاده می‌شود.
- `botDisplayName` — نام نمایشی برای پیام‌های خروجی.
- `pollTimeoutMs` — پنجره انتظار long-poll. عدد صحیح بین 100 و 30000.
- `allowFrom` — فهرست مجاز فرستنده‌ها (شناسه‌های کاربر یا `"*"`).
- `defaultTo` — هدف جایگزین وقتی هیچ هدفی ارائه نشده باشد.
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` — کنترل دسترسی ابزار برای هر کنش.

کلیدهای چندحسابی در سطح بالا:

- `accounts` — رکوردی از بازنویسی‌های نام‌گذاری‌شده برای هر حساب، با کلید شناسه حساب.
- `defaultAccount` — شناسه حساب ترجیحی وقتی چند حساب پیکربندی شده‌اند.

## اجراکننده‌ها

خودآزمایی سمت میزبان (یک گزارش Markdown زیر `.artifacts/qa-e2e/` می‌نویسد):

```bash
pnpm qa:e2e
```

این مسیر از طریق `qa-lab` عبور می‌کند، گذرگاه QA داخل مخزن را شروع می‌کند، برش runtime بسته‌بندی‌شده `qa-channel` را راه‌اندازی می‌کند، و یک خودآزمایی قطعی را اجرا می‌کند.

مجموعه سناریوی کامل متکی به مخزن:

```bash
pnpm openclaw qa suite
```

سناریوها را به‌صورت موازی در برابر مسیر QA Gateway اجرا می‌کند. برای سناریوها، پروفایل‌ها، و حالت‌های ارائه‌دهنده، [نمای کلی QA](/fa/concepts/qa-e2e-automation) را ببینید.

سایت QA مبتنی بر Docker (Gateway + رابط کاربری اشکال‌زدای QA Lab در یک پشته):

```bash
pnpm qa:lab:up
```

سایت QA را می‌سازد، پشته Gateway + QA Lab مبتنی بر Docker را شروع می‌کند، و URL مربوط به QA Lab را چاپ می‌کند. از آنجا می‌توانید سناریوها را انتخاب کنید، مسیر مدل را برگزینید، اجراهای جداگانه را راه‌اندازی کنید، و نتایج را زنده ببینید. اشکال‌زدای QA Lab از بسته Control UI منتشرشده جدا است.
