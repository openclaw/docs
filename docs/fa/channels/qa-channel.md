`qa-channel` یک انتقال پیام مصنوعی و محلیِ مخزن برای QA خودکار OpenClaw است (`extensions/qa-channel`، بستهٔ خصوصی، مستثنا از نصب‌های بسته‌بندی‌شده). این یک کانال عملیاتی نیست؛ برای آزمودن همان مرز Plugin کانال که انتقال‌های واقعی استفاده می‌کنند وجود دارد، درحالی‌که وضعیت را قطعی و کاملاً قابل‌بازرسی نگه می‌دارد.

## چه کاری انجام می‌دهد

- دستور زبان مقصد در سطح Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- گفت‌وگوهای اشتراکی `channel:` و `group:` به‌صورت نوبت‌های اتاق گروه/کانال به عامل‌ها ارائه می‌شوند؛ بنابراین همان سیاست مسیریابی پاسخ قابل‌مشاهده و ابزار پیام را می‌آزمایند که Discord، Slack، Telegram و انتقال‌های مشابه استفاده می‌کنند.
- گذرگاه مصنوعی مبتنی بر HTTP برای تزریق پیام ورودی، ثبت رونوشت خروجی، ایجاد رشته، واکنش‌ها، ویرایش‌ها، حذف‌ها و عملیات جست‌وجو/خواندن.
- اجراکنندهٔ خودآزمایی سمت میزبان که یک گزارش Markdown در `.artifacts/qa-e2e/` می‌نویسد.

## پیکربندی

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

کلیدهای حساب:

- `enabled` - کلید اصلی فعال‌سازی این حساب.
- `name` - برچسب نمایشی اختیاری.
- `baseUrl` - نشانی URL گذرگاه مصنوعی. پس از تنظیم این مقدار، حساب پیکربندی‌شده محسوب می‌شود.
- `botUserId` - شناسهٔ کاربر ربات مصنوعی که در دستور زبان مقصد استفاده می‌شود (پیش‌فرض: `openclaw`).
- `botDisplayName` - نام نمایشی پیام‌های خروجی (پیش‌فرض: `OpenClaw QA`).
- `pollTimeoutMs` - بازهٔ انتظار نظرسنجی طولانی. عدد صحیحی بین 100 و 30000 (پیش‌فرض: 1000).
- `allowFrom` - فهرست مجاز فرستندگان (شناسه‌های کاربر یا `"*"`؛ پیش‌فرض: `["*"]`). پیام‌های خصوصی
  همیشه از سیاست `open` پیروی می‌کنند؛ سیاست گروه مجازشده نیز از همین شناسه‌های
  مصنوعی فرستنده استفاده می‌کند.
- `groupPolicy` - سیاست اتاق اشتراکی: `"open"` (پیش‌فرض)، `"allowlist"` یا
  `"disabled"`.
- `groupAllowFrom` - فهرست مجاز اختیاری فرستندگان اتاق اشتراکی. اگر در حالت
  `"allowlist"` حذف شود، QA Channel به `allowFrom` بازمی‌گردد.
- `groups.<room>.requireMention` - الزام اشاره به ربات پیش از پاسخ‌دادن در یک
  اتاق گروه/کانال مشخص (پیش‌فرض: false). `groups."*"` مقدار پیش‌فرض را تعیین می‌کند؛
  مقادیر مختص هر اتاقِ `tools` / `toolsBySender` بازنویسی‌های سیاست ابزار را تعیین می‌کنند.
- `defaultTo` - مقصد جایگزین در صورت ارائه‌نشدن مقصد.
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` - کنترل دسترسی ابزار به تفکیک عملیات.

کلیدهای چندحسابی در سطح بالا:

- `accounts` - نگاشتی از بازنویسی‌های نام‌گذاری‌شدهٔ هر حساب که با شناسهٔ حساب کلیدگذاری شده‌اند.
- `defaultAccount` - شناسهٔ حساب ترجیحی هنگامی که چند حساب پیکربندی شده‌اند.

## اجراکننده‌ها

خودآزمایی سمت میزبان (یک گزارش Markdown زیر `.artifacts/qa-e2e/` می‌نویسد):

```bash
pnpm qa:e2e
```

این فرمان از مسیر `qa-lab` عبور می‌کند، گذرگاه QA درون‌مخزن را راه‌اندازی می‌کند، برش زمان‌اجرای `qa-channel` را بالا می‌آورد و یک خودآزمایی قطعی اجرا می‌کند.

مجموعهٔ کامل سناریوهای متکی به مخزن:

```bash
pnpm openclaw qa suite
```

سناریوها را به‌صورت موازی در مسیر Gateway مربوط به QA اجرا می‌کند. برای سناریوها، پروفایل‌ها و حالت‌های ارائه‌دهنده، به [نمای کلی QA](/fa/concepts/qa-e2e-automation) مراجعه کنید.

سایت QA مبتنی بر Docker ‏(Gateway و رابط کاربری اشکال‌زدای QA Lab در یک پشته):

```bash
pnpm qa:lab:up
```

سایت QA را می‌سازد، پشتهٔ مبتنی بر Docker شامل Gateway و QA Lab را راه‌اندازی می‌کند و نشانی URL ‏QA Lab را نمایش می‌دهد. از آنجا می‌توان سناریوها را انتخاب کرد، مسیر مدل را برگزید، اجراهای منفرد را آغاز کرد و نتایج را به‌صورت زنده مشاهده کرد. اشکال‌زدای QA Lab از بستهٔ منتشرشدهٔ رابط کاربری Control UI جدا است.
