`qa-channel` یک انتقال پیام مصنوعی و محلیِ مخزن برای تضمین کیفیت خودکار OpenClaw است (`extensions/qa-channel`، بستهٔ خصوصی، مستثنا از نصب‌های بسته‌بندی‌شده). این یک کانال عملیاتی نیست؛ برای به‌کارگیری همان مرز Plugin کانال که انتقال‌های واقعی استفاده می‌کنند وجود دارد، درحالی‌که وضعیت را قطعی و کاملاً قابل بازرسی نگه می‌دارد.

## چه کاری انجام می‌دهد

- دستور زبان مقصد در سطح Slack:
  - `dm:<user>`
  - `channel:<room>`
  - `group:<room>`
  - `thread:<room>/<thread>`
- مکالمات اشتراکی `channel:` و `group:` به‌صورت نوبت‌های اتاق گروه/کانال به عامل‌ها ارائه می‌شوند؛ بنابراین همان سیاست مسیریابی پاسخ قابل‌مشاهده و ابزار پیام را به‌کار می‌گیرند که Discord، Slack، Telegram و انتقال‌های مشابه استفاده می‌کنند.
- گذرگاه مصنوعی مبتنی بر HTTP برای تزریق پیام ورودی، ثبت رونوشت خروجی، ایجاد رشته‌گفت‌وگو، واکنش‌ها، ویرایش‌ها، حذف‌ها و عملیات جست‌وجو/خواندن.
- اجراکنندهٔ خودآزمایی سمت میزبان که یک گزارش Markdown در `.artifacts/qa-e2e/` می‌نویسد.

## پیکربندی

```json
{
  "channels": {
    "qa-channel": {
      "baseUrl": "http://127.0.0.1:43123",
      "botUserId": "openclaw",
      "botDisplayName": "OpenClaw QA",
      "allowFrom": ["*"],
      "pollTimeoutMs": 1000
    }
  }
}
```

کلیدهای حساب:

- `enabled` - کلید اصلی فعال‌سازی این حساب.
- `name` - برچسب نمایشی اختیاری.
- `baseUrl` - نشانی گذرگاه مصنوعی. پس از تنظیم این مقدار، حساب پیکربندی‌شده محسوب می‌شود.
- `botUserId` - شناسهٔ کاربر ربات مصنوعی که در دستور زبان مقصد استفاده می‌شود (پیش‌فرض: `openclaw`).
- `botDisplayName` - نام نمایشی پیام‌های خروجی (پیش‌فرض: `OpenClaw QA`).
- `pollTimeoutMs` - بازهٔ انتظار نظرسنجی طولانی. عدد صحیحی بین 100 و 30000 (پیش‌فرض: 1000).
- `allowFrom` - فهرست مجاز فرستندگان (شناسه‌های کاربر یا `"*"`؛ پیش‌فرض: `["*"]`). سیاست پیام‌های مستقیم
  همیشه `open` است؛ سیاست گروه مبتنی بر فهرست مجاز نیز از همین شناسه‌های
  مصنوعی فرستندگان استفاده می‌کند.
- `groupPolicy` - سیاست اتاق اشتراکی: `"open"` (پیش‌فرض)، `"allowlist"` یا
  `"disabled"`.
- `groupAllowFrom` - فهرست مجاز اختیاری فرستندگان اتاق اشتراکی. اگر در حالت
  `"allowlist"` مشخص نشود، کانال تضمین کیفیت از `allowFrom` استفاده می‌کند.
- `groups.<room>.requireMention` - پیش از پاسخ‌گویی در یک اتاق گروه/کانال
  مشخص، اشاره به ربات را الزامی می‌کند (پیش‌فرض: false). `groups."*"` مقدار پیش‌فرض را تنظیم می‌کند؛
  `tools` / `toolsBySender` مخصوص هر اتاق، بازنویسی‌های سیاست ابزار را تنظیم می‌کنند.
- `defaultTo` - مقصد جایگزین هنگامی که هیچ مقصدی ارائه نشده باشد.
- `actions.messages` / `actions.reactions` / `actions.search` / `actions.threads` - کنترل دسترسی ابزار به‌ازای هر عملیات.

کلیدهای چندحسابی در سطح بالا:

- `accounts` - نگاشتی از بازنویسی‌های نام‌گذاری‌شدهٔ هر حساب که با شناسهٔ حساب کلیدگذاری شده‌اند.
- `defaultAccount` - شناسهٔ حساب ترجیحی هنگامی که چند حساب پیکربندی شده‌اند.

## اجراکننده‌ها

خودآزمایی سمت میزبان (یک گزارش Markdown در `.artifacts/qa-e2e/` می‌نویسد):

```bash
pnpm qa:e2e
```

این فرمان از طریق `qa-lab` مسیریابی می‌شود، گذرگاه تضمین کیفیت درون مخزن را راه‌اندازی می‌کند، بخش زمان اجرای `qa-channel` را بالا می‌آورد و یک خودآزمایی قطعی اجرا می‌کند.

مجموعهٔ کامل سناریوهای مبتنی بر مخزن:

```bash
pnpm openclaw qa suite
```

سناریوها را به‌صورت موازی در مسیر Gateway تضمین کیفیت اجرا می‌کند. برای سناریوها، نمایه‌ها و حالت‌های ارائه‌دهنده، [نمای کلی تضمین کیفیت](/fa/concepts/qa-e2e-automation) را ببینید.

سایت تضمین کیفیت مبتنی بر Docker ‏(Gateway و رابط کاربری اشکال‌زدای آزمایشگاه تضمین کیفیت در یک پشته):

```bash
pnpm qa:lab:up
```

سایت تضمین کیفیت را می‌سازد، پشتهٔ مبتنی بر Docker شامل Gateway و آزمایشگاه تضمین کیفیت را راه‌اندازی می‌کند و نشانی آزمایشگاه تضمین کیفیت را چاپ می‌کند. از آنجا می‌توانید سناریوها را انتخاب کنید، مسیر مدل را برگزینید، اجراهای منفرد را آغاز کنید و نتایج را به‌صورت زنده مشاهده کنید. اشکال‌زدای آزمایشگاه تضمین کیفیت از بستهٔ رابط کاربری کنترلِ منتشرشده جدا است.
