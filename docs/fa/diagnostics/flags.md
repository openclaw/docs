پرچم‌های عیب‌یابی به شما امکان می‌دهند لاگ‌های اشکال‌زدایی هدفمند را بدون فعال‌کردن لاگ‌گیری پرجزئیات در همه‌جا فعال کنید. پرچم‌ها انتخابی هستند و مگر اینکه یک زیرسامانه آن‌ها را بررسی کند، اثری ندارند.

## نحوه کارکرد

- پرچم‌ها رشته هستند (به بزرگی و کوچکی حروف حساس نیستند).
- می‌توانید پرچم‌ها را در پیکربندی یا از طریق بازنویسی env فعال کنید.
- وایلدکارت‌ها پشتیبانی می‌شوند:
  - `telegram.*` با `telegram.http` مطابقت دارد
  - `*` همه پرچم‌ها را فعال می‌کند

## فعال‌سازی از طریق پیکربندی

```json
{
  "diagnostics": {
    "flags": ["telegram.http"]
  }
}
```

چند پرچم:

```json
{
  "diagnostics": {
    "flags": ["telegram.http", "gateway.*"]
  }
}
```

پس از تغییر پرچم‌ها، Gateway را راه‌اندازی مجدد کنید.

## بازنویسی env (یک‌باره)

```bash
OPENCLAW_DIAGNOSTICS=telegram.http,telegram.payload
```

غیرفعال‌کردن همه پرچم‌ها:

```bash
OPENCLAW_DIAGNOSTICS=0
```

## مصنوعات خط زمانی

پرچم `timeline` رویدادهای زمان‌بندی ساختاریافته راه‌اندازی و زمان اجرا را برای
هارنس‌های QA خارجی می‌نویسد:

```bash
OPENCLAW_DIAGNOSTICS=timeline \
OPENCLAW_DIAGNOSTICS_TIMELINE_PATH=/tmp/openclaw-timeline.jsonl \
openclaw gateway run
```

همچنین می‌توانید آن را در پیکربندی فعال کنید:

```json
{
  "diagnostics": {
    "flags": ["timeline"]
  }
}
```

مسیر فایل خط زمانی همچنان از
`OPENCLAW_DIAGNOSTICS_TIMELINE_PATH` می‌آید. وقتی `timeline` فقط از
پیکربندی فعال شده باشد، نخستین بازه‌های بارگذاری پیکربندی منتشر نمی‌شوند، چون OpenClaw هنوز
پیکربندی را نخوانده است؛ بازه‌های بعدی راه‌اندازی از پرچم پیکربندی استفاده می‌کنند.

`OPENCLAW_DIAGNOSTICS=1`، `OPENCLAW_DIAGNOSTICS=all` و
`OPENCLAW_DIAGNOSTICS=*` نیز خط زمانی را فعال می‌کنند، چون همه
پرچم‌های عیب‌یابی را فعال می‌کنند. وقتی فقط مصنوع زمان‌بندی JSONL را
می‌خواهید، `timeline` را ترجیح دهید.

رکوردهای خط زمانی از پوشش `openclaw.diagnostics.v1` استفاده می‌کنند. رویدادها می‌توانند شامل
شناسه‌های پردازش، نام‌های فاز، نام‌های بازه، مدت‌زمان‌ها، شناسه‌های Plugin، شمار وابستگی‌ها،
نمونه‌های تاخیر حلقه رویداد، نام‌های عملیات ارائه‌دهنده، وضعیت خروج فرایند فرزند،
و نام‌ها/پیام‌های خطای راه‌اندازی باشند. فایل‌های خط زمانی را به‌عنوان مصنوعات عیب‌یابی محلی
در نظر بگیرید؛ پیش از اشتراک‌گذاری بیرون از ماشین خود، آن‌ها را بازبینی کنید.

## لاگ‌ها کجا می‌روند

پرچم‌ها لاگ‌ها را در فایل لاگ عیب‌یابی استاندارد منتشر می‌کنند. به‌صورت پیش‌فرض:

```
/tmp/openclaw/openclaw-YYYY-MM-DD.log
```

اگر `logging.file` را تنظیم کرده‌اید، به‌جای آن از همان مسیر استفاده کنید. لاگ‌ها JSONL هستند (هر خط یک شیء JSON). بازنویسی محرمانه‌ها همچنان بر اساس `logging.redactSensitive` اعمال می‌شود.

## استخراج لاگ‌ها

جدیدترین فایل لاگ را انتخاب کنید:

```bash
ls -t /tmp/openclaw/openclaw-*.log | head -n 1
```

فیلتر برای عیب‌یابی HTTP Telegram:

```bash
rg "telegram http error" /tmp/openclaw/openclaw-*.log
```

یا هنگام بازتولید، آن را دنبال کنید:

```bash
tail -f /tmp/openclaw/openclaw-$(date +%F).log | rg "telegram http error"
```

برای Gatewayهای راه‌دور، می‌توانید از `openclaw logs --follow` نیز استفاده کنید (نگاه کنید به [/cli/logs](/fa/cli/logs)).

## یادداشت‌ها

- اگر `logging.level` بالاتر از `warn` تنظیم شده باشد، این لاگ‌ها ممکن است سرکوب شوند. مقدار پیش‌فرض `info` مناسب است.
- فعال‌ماندن پرچم‌ها امن است؛ آن‌ها فقط حجم لاگ را برای زیرسامانه مشخص تحت تاثیر قرار می‌دهند.
- برای تغییر مقصدها، سطح‌ها و بازنویسی محرمانه‌های لاگ، از [/logging](/fa/logging) استفاده کنید.
