جریان وظیفه زیرلایه هماهنگ‌سازی جریان است که بالای [وظایف پس‌زمینه](/fa/automation/tasks) قرار می‌گیرد. این قابلیت جریان‌های چندمرحله‌ای پایدار را با وضعیت، ردیابی بازنگری، و معناشناسی همگام‌سازی خودشان مدیریت می‌کند، در حالی که وظایف منفرد همچنان واحد کار جداشده باقی می‌مانند.

## چه زمانی از جریان وظیفه استفاده کنیم

وقتی کار شامل چند مرحله ترتیبی یا شاخه‌دار است و به ردیابی پایدار پیشرفت در میان راه‌اندازی‌های دوباره Gateway نیاز دارید، از جریان وظیفه استفاده کنید. برای عملیات تکی در پس‌زمینه، یک [وظیفه](/fa/automation/tasks) ساده کافی است.

| سناریو                               | استفاده              |
| ------------------------------------ | -------------------- |
| کار پس‌زمینه تکی                     | وظیفه ساده           |
| خط لوله چندمرحله‌ای (A سپس B سپس C) | جریان وظیفه (مدیریت‌شده) |
| مشاهده وظایف ایجادشده بیرونی        | جریان وظیفه (بازتابی) |
| یادآور یک‌باره                       | کار Cron             |

## الگوی قابل‌اعتماد برای گردش‌کار زمان‌بندی‌شده

برای گردش‌کارهای تکرارشونده مانند گزارش‌های اطلاعات بازار، زمان‌بندی، هماهنگ‌سازی، و بررسی‌های قابلیت اطمینان را به‌عنوان لایه‌های جداگانه در نظر بگیرید:

1. از [وظایف زمان‌بندی‌شده](/fa/automation/cron-jobs) برای زمان‌بندی استفاده کنید.
2. زمینه قبلی را در فایل‌ها، پایگاه داده، یا وضعیت ابزار خود گردش‌کار ذخیره کنید.
3. از [Lobster](/fa/tools/lobster) برای مراحل قطعی، دروازه‌های تأیید، و توکن‌های ادامه استفاده کنید.
4. از جریان وظیفه برای ردیابی اجرای چندمرحله‌ای در میان وظایف فرزند، انتظارها، تلاش‌های دوباره، و راه‌اندازی‌های دوباره Gateway استفاده کنید.

شکل نمونه Cron:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

وقتی کار باید یک گفت‌وگو/نشست شناخته‌شده را برای زمینه تحویل یا مقداردهی اولیه ایمن ترجیحات هدف بگیرد، از `session:<id>` استفاده کنید. Cron همچنان هر اجرا را در یک نشست جداشده اجرا می‌کند، پس خلاصه‌های اجرای قبلی و وضعیت ثابت گردش‌کار را در ذخیره‌سازی صریحی قرار دهید که کار بتواند آن را بخواند.

درون گردش‌کار، بررسی‌های قابلیت اطمینان را پیش از مرحله خلاصه‌سازی LLM قرار دهید:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

بررسی‌های پیشنهادی پیش از اجرا:

- در دسترس بودن مرورگر و انتخاب پروفایل، برای مثال `openclaw` برای وضعیت مدیریت‌شده یا `user` وقتی نشست Chrome واردشده لازم است. [مرورگر](/fa/tools/browser) را ببینید.
- اعتبارنامه‌های API و سهمیه برای هر منبع.
- دسترسی‌پذیری شبکه برای نقاط پایانی لازم.
- ابزارهای لازم فعال‌شده برای عامل، مانند `lobster`، `browser`، و `llm-task`.
- مقصد شکست پیکربندی‌شده برای Cron تا شکست‌های پیش از اجرا قابل مشاهده باشند. [وظایف زمان‌بندی‌شده](/fa/automation/cron-jobs#delivery-and-output) را ببینید.

فیلدهای پیشنهادی منشأ داده برای هر مورد گردآوری‌شده:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

گردش‌کار را طوری تنظیم کنید که موارد کهنه را پیش از خلاصه‌سازی رد یا علامت‌گذاری کند. مرحله LLM باید فقط JSON ساختاریافته دریافت کند و باید از آن خواسته شود `sourceUrl`، `retrievedAt`، و `asOf` را در خروجی خود حفظ کند. وقتی به یک مرحله مدل با اعتبارسنجی schema درون گردش‌کار نیاز دارید، از [LLM Task](/fa/tools/llm-task) استفاده کنید.

برای گردش‌کارهای قابل استفاده مجدد تیمی یا اجتماعی، CLI، فایل‌های `.lobster`، و هر یادداشت راه‌اندازی را به‌عنوان یک Skill یا Plugin بسته‌بندی کنید و آن را از طریق [ClawHub](/clawhub) منتشر کنید. محافظ‌های ویژه گردش‌کار را در همان بسته نگه دارید، مگر اینکه API مربوط به Plugin یک قابلیت عمومی لازم را نداشته باشد.

## حالت‌های همگام‌سازی

### حالت مدیریت‌شده

جریان وظیفه مالک چرخه عمر از ابتدا تا انتهاست. وظایف را به‌عنوان مراحل جریان ایجاد می‌کند، آن‌ها را تا تکمیل پیش می‌برد، و وضعیت جریان را به‌صورت خودکار جلو می‌برد.

نمونه: جریان گزارش هفتگی که (1) داده‌ها را گردآوری می‌کند، (2) گزارش را تولید می‌کند، و (3) آن را تحویل می‌دهد. جریان وظیفه هر مرحله را به‌عنوان یک وظیفه پس‌زمینه ایجاد می‌کند، منتظر تکمیل می‌ماند، سپس به مرحله بعد می‌رود.

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### حالت بازتابی

جریان وظیفه وظایف ایجادشده بیرونی را مشاهده می‌کند و بدون مالکیت ایجاد وظیفه، وضعیت جریان را همگام نگه می‌دارد. این حالت زمانی مفید است که وظایف از کارهای Cron، فرمان‌های CLI، یا منابع دیگر منشأ می‌گیرند و می‌خواهید نمایی یکپارچه از پیشرفت آن‌ها به‌عنوان یک جریان داشته باشید.

نمونه: سه کار Cron مستقل که با هم یک روال «عملیات صبحگاهی» را تشکیل می‌دهند. یک جریان بازتابی پیشرفت جمعی آن‌ها را بدون کنترل زمان یا شیوه اجرای آن‌ها ردیابی می‌کند.

## وضعیت پایدار و ردیابی بازنگری

هر جریان وضعیت خودش را پایدار می‌کند و بازنگری‌ها را ردیابی می‌کند تا پیشرفت پس از راه‌اندازی‌های دوباره Gateway حفظ شود. ردیابی بازنگری، تشخیص تعارض را زمانی ممکن می‌کند که چند منبع تلاش کنند همان جریان را هم‌زمان جلو ببرند.
رجیستری جریان از SQLite با نگهداری محدود write-ahead-log استفاده می‌کند، از جمله
checkpointهای دوره‌ای و هنگام خاموشی، تا Gatewayهای طولانی‌مدت فایل‌های جانبی
نامحدود `registry.sqlite-wal` را نگه ندارند.

## رفتار لغو

`openclaw tasks flow cancel` یک قصد لغو چسبنده روی جریان تنظیم می‌کند. وظایف فعال درون جریان لغو می‌شوند و هیچ مرحله جدیدی آغاز نمی‌شود. قصد لغو در میان راه‌اندازی‌های دوباره پایدار می‌ماند، بنابراین یک جریان لغوشده حتی اگر Gateway پیش از پایان همه وظایف فرزند دوباره راه‌اندازی شود، لغوشده باقی می‌ماند.

## فرمان‌های CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| فرمان                            | توضیح                                         |
| -------------------------------- | --------------------------------------------- |
| `openclaw tasks flow list`       | جریان‌های ردیابی‌شده را با وضعیت و حالت همگام‌سازی نشان می‌دهد |
| `openclaw tasks flow show <id>`  | یک جریان را با شناسه جریان یا کلید جست‌وجو بررسی می‌کند |
| `openclaw tasks flow cancel <id>` | یک جریان در حال اجرا و وظایف فعال آن را لغو می‌کند |

## ارتباط جریان‌ها با وظایف

جریان‌ها وظایف را هماهنگ می‌کنند، جایگزین آن‌ها نمی‌شوند. یک جریان واحد ممکن است در طول عمر خود چندین وظیفه پس‌زمینه را پیش ببرد. از `openclaw tasks` برای بررسی رکوردهای وظیفه منفرد و از `openclaw tasks flow` برای بررسی جریان هماهنگ‌کننده استفاده کنید.
