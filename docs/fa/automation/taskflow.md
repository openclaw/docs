TaskFlow زیرلایهٔ هماهنگ‌سازی جریان است که بالای [وظایف پس‌زمینه](/fa/automation/tasks) قرار می‌گیرد. این قابلیت جریان‌های چندمرحله‌ای پایدار را با وضعیت، رهگیری بازبینی، و معناشناسی همگام‌سازی مخصوص خودشان مدیریت می‌کند، در حالی که وظایف منفرد همچنان واحد کار جداشده باقی می‌مانند.

## چه زمانی از TaskFlow استفاده کنید

وقتی کار شامل چند مرحلهٔ ترتیبی یا شاخه‌دار است و به رهگیری پایدار پیشرفت در طول راه‌اندازی‌های دوبارهٔ Gateway نیاز دارید، از TaskFlow استفاده کنید. برای عملیات پس‌زمینهٔ منفرد، یک [وظیفه](/fa/automation/tasks) ساده کافی است.

| سناریو                               | استفاده             |
| ------------------------------------- | -------------------- |
| کار پس‌زمینهٔ منفرد                  | وظیفهٔ ساده          |
| خط لولهٔ چندمرحله‌ای (A سپس B سپس C) | TaskFlow (مدیریت‌شده) |
| مشاهدهٔ وظایف ایجادشدهٔ بیرونی       | TaskFlow (آینه‌شده)  |
| یادآور یک‌باره                       | کار Cron             |

## الگوی جریان کاری زمان‌بندی‌شدهٔ قابل‌اعتماد

برای جریان‌های کاری تکرارشونده مانند گزارش‌های اطلاعات بازار، زمان‌بندی، هماهنگ‌سازی، و بررسی‌های قابلیت اطمینان را به‌عنوان لایه‌های جداگانه در نظر بگیرید:

1. برای زمان‌بندی از [وظایف زمان‌بندی‌شده](/fa/automation/cron-jobs) استفاده کنید.
2. وقتی جریان کاری باید بر زمینهٔ قبلی بنا شود، از یک نشست cron پایدار استفاده کنید.
3. برای گام‌های قطعی، دروازه‌های تأیید، و توکن‌های ازسرگیری از [Lobster](/fa/tools/lobster) استفاده کنید.
4. برای رهگیری اجرای چندمرحله‌ای در میان وظایف فرزند، انتظارها، تلاش‌های دوباره، و راه‌اندازی‌های دوبارهٔ Gateway از TaskFlow استفاده کنید.

نمونهٔ شکل cron:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

وقتی جریان کاری تکرارشونده به تاریخچهٔ عمدی، خلاصه‌های اجرای قبلی، یا زمینهٔ ثابت نیاز دارد، به‌جای `isolated` از `session:<id>` استفاده کنید. وقتی هر اجرا باید از نو شروع شود و همهٔ وضعیت موردنیاز به‌صورت صریح در جریان کاری آمده است، از `isolated` استفاده کنید.

درون جریان کاری، بررسی‌های قابلیت اطمینان را پیش از گام خلاصه‌سازی LLM قرار دهید:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

بررسی‌های پیش‌پرواز پیشنهادی:

- در دسترس بودن مرورگر و انتخاب نمایه، برای مثال `openclaw` برای وضعیت مدیریت‌شده یا `user` وقتی نشست Chrome واردشده لازم است. [مرورگر](/fa/tools/browser) را ببینید.
- اعتبارنامه‌های API و سهمیه برای هر منبع.
- دسترسی‌پذیری شبکه برای نقاط پایانی موردنیاز.
- ابزارهای موردنیازِ فعال‌شده برای عامل، مانند `lobster`، `browser`، و `llm-task`.
- مقصد شکست برای cron پیکربندی شده باشد تا شکست‌های پیش‌پرواز قابل مشاهده باشند. [وظایف زمان‌بندی‌شده](/fa/automation/cron-jobs#delivery-and-output) را ببینید.

فیلدهای پیشنهادی منشأ داده برای هر مورد گردآوری‌شده:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

جریان کاری باید پیش از خلاصه‌سازی، موارد کهنه را رد کند یا علامت‌گذاری کند. گام LLM باید فقط JSON ساختاریافته دریافت کند و باید از آن خواسته شود `sourceUrl`، `retrievedAt`، و `asOf` را در خروجی خود حفظ کند. وقتی درون جریان کاری به یک گام مدل با اعتبارسنجی schema نیاز دارید، از [وظیفهٔ LLM](/fa/tools/llm-task) استفاده کنید.

برای جریان‌های کاری قابل‌استفادهٔ دوباره برای تیم یا جامعه، CLI، فایل‌های `.lobster`، و هر یادداشت راه‌اندازی را به‌عنوان یک skill یا plugin بسته‌بندی کنید و آن را از طریق [ClawHub](/fa/clawhub) منتشر کنید. محافظ‌های مخصوص جریان کاری را در همان بسته نگه دارید، مگر اینکه API plugin فاقد قابلیت عمومی موردنیاز باشد.

## حالت‌های همگام‌سازی

### حالت مدیریت‌شده

TaskFlow چرخهٔ عمر را از ابتدا تا انتها در اختیار دارد. این قابلیت وظایف را به‌عنوان گام‌های جریان ایجاد می‌کند، آن‌ها را تا تکمیل پیش می‌برد، و وضعیت جریان را به‌طور خودکار جلو می‌برد.

نمونه: جریان گزارش هفتگی که (1) داده‌ها را گردآوری می‌کند، (2) گزارش را تولید می‌کند، و (3) آن را تحویل می‌دهد. TaskFlow هر گام را به‌عنوان یک وظیفهٔ پس‌زمینه ایجاد می‌کند، منتظر تکمیل می‌ماند، سپس به گام بعدی می‌رود.

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### حالت آینه‌شده

TaskFlow وظایف ایجادشدهٔ بیرونی را مشاهده می‌کند و وضعیت جریان را بدون در اختیار گرفتن مالکیت ایجاد وظیفه همگام نگه می‌دارد. این حالت زمانی مفید است که وظایف از کارهای cron، فرمان‌های CLI، یا منابع دیگر منشأ می‌گیرند و شما نمایی یکپارچه از پیشرفت آن‌ها به‌عنوان یک جریان می‌خواهید.

نمونه: سه کار cron مستقل که با هم یک روال «عملیات صبحگاهی» را تشکیل می‌دهند. یک جریان آینه‌شده پیشرفت جمعی آن‌ها را بدون کنترل زمان یا شیوهٔ اجرای آن‌ها رهگیری می‌کند.

## وضعیت پایدار و رهگیری بازبینی

هر جریان وضعیت خودش را پایدار نگه می‌دارد و بازبینی‌ها را رهگیری می‌کند تا پیشرفت از راه‌اندازی‌های دوبارهٔ Gateway جان سالم به در ببرد. رهگیری بازبینی امکان تشخیص تعارض را وقتی چند منبع هم‌زمان تلاش می‌کنند همان جریان را جلو ببرند فراهم می‌کند.
رجیستری جریان از SQLite با نگه‌داری محدود write-ahead-log استفاده می‌کند، از جمله
checkpointهای دوره‌ای و هنگام خاموشی، تا Gatewayهای طولانی‌اجرا فایل‌های جانبی
`registry.sqlite-wal` نامحدود نگه ندارند.

## رفتار لغو

`openclaw tasks flow cancel` یک قصد لغو چسبنده روی جریان تنظیم می‌کند. وظایف فعال درون جریان لغو می‌شوند و هیچ گام تازه‌ای شروع نمی‌شود. قصد لغو در طول راه‌اندازی‌های دوباره پایدار می‌ماند، بنابراین یک جریان لغوشده حتی اگر Gateway پیش از پایان یافتن همهٔ وظایف فرزند دوباره راه‌اندازی شود، لغوشده باقی می‌ماند.

## فرمان‌های CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| فرمان                            | توضیح                                      |
| --------------------------------- | ------------------------------------------ |
| `openclaw tasks flow list`        | جریان‌های رهگیری‌شده را با وضعیت و حالت همگام‌سازی نشان می‌دهد |
| `openclaw tasks flow show <id>`   | یک جریان را بر اساس شناسهٔ جریان یا کلید lookup بررسی می‌کند |
| `openclaw tasks flow cancel <id>` | یک جریان در حال اجرا و وظایف فعال آن را لغو می‌کند |

## جریان‌ها چه ارتباطی با وظایف دارند

جریان‌ها وظایف را هماهنگ می‌کنند، نه اینکه جایگزین آن‌ها شوند. یک جریان واحد ممکن است در طول عمر خود چندین وظیفهٔ پس‌زمینه را پیش ببرد. برای بررسی رکوردهای وظیفهٔ منفرد از `openclaw tasks` و برای بررسی جریان هماهنگ‌کننده از `openclaw tasks flow` استفاده کنید.
