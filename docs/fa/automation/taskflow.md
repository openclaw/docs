Task Flow زیرساخت ارکستراسیون جریان است که بالای [وظایف پس‌زمینه](/fa/automation/tasks) قرار می‌گیرد. این بخش جریان‌های چندمرحله‌ای پایدار را با وضعیت، رهگیری بازبینی، و معناشناسی همگام‌سازی اختصاصی خود مدیریت می‌کند، در حالی که وظایف منفرد همچنان واحد کار جداشده باقی می‌مانند.

## زمان استفاده از Task Flow

وقتی کار چندین مرحله ترتیبی یا شاخه‌ای را در بر می‌گیرد و به رهگیری پایدار پیشرفت در طول راه‌اندازی‌های دوباره Gateway نیاز دارید، از Task Flow استفاده کنید. برای عملیات پس‌زمینه تکی، یک [وظیفه](/fa/automation/tasks) ساده کافی است.

| سناریو                              | استفاده                  |
| ------------------------------------- | -------------------- |
| کار پس‌زمینه تکی                 | وظیفه ساده           |
| خط لوله چندمرحله‌ای (A سپس B سپس C) | Task Flow (مدیریت‌شده)  |
| مشاهده وظایف ایجادشده از بیرون      | Task Flow (آینه‌شده) |
| یادآور یک‌باره                     | کار Cron             |

## الگوی گردش‌کار زمان‌بندی‌شده قابل‌اعتماد

برای گردش‌کارهای تکرارشونده مانند گزارش‌های اطلاعات بازار، زمان‌بندی، ارکستراسیون، و بررسی‌های قابلیت اعتماد را به‌عنوان لایه‌های جداگانه در نظر بگیرید:

1. از [وظایف زمان‌بندی‌شده](/fa/automation/cron-jobs) برای زمان‌بندی استفاده کنید.
2. وقتی گردش‌کار باید بر زمینه قبلی بنا شود، از یک نشست Cron پایدار استفاده کنید.
3. برای مراحل قطعی، درگاه‌های تأیید، و توکن‌های ازسرگیری از [Lobster](/fa/tools/lobster) استفاده کنید.
4. از Task Flow برای رهگیری اجرای چندمرحله‌ای در میان وظایف فرزند، انتظارها، تلاش‌های دوباره، و راه‌اندازی‌های دوباره Gateway استفاده کنید.

نمونه شکل Cron:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

وقتی گردش‌کار تکرارشونده به تاریخچه عمدی، خلاصه‌های اجرای قبلی، یا زمینه ثابت نیاز دارد، به‌جای `isolated` از `session:<id>` استفاده کنید. وقتی هر اجرا باید از نو شروع شود و همه وضعیت لازم به‌صورت صریح در گردش‌کار آمده است، از `isolated` استفاده کنید.

درون گردش‌کار، بررسی‌های قابلیت اعتماد را پیش از مرحله خلاصه‌سازی LLM قرار دهید:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

بررسی‌های پیش‌پرواز پیشنهادی:

- در دسترس بودن مرورگر و انتخاب پروفایل، برای مثال `openclaw` برای وضعیت مدیریت‌شده یا `user` وقتی نشست Chrome واردشده لازم است. [مرورگر](/fa/tools/browser) را ببینید.
- اعتبارنامه‌های API و سهمیه برای هر منبع.
- دسترسی‌پذیری شبکه برای نقاط پایانی لازم.
- ابزارهای لازم فعال‌شده برای عامل، مانند `lobster`، `browser`، و `llm-task`.
- مقصد شکست برای Cron پیکربندی شده باشد تا شکست‌های پیش‌پرواز قابل مشاهده باشند. [وظایف زمان‌بندی‌شده](/fa/automation/cron-jobs#delivery-and-output) را ببینید.

فیلدهای پیشنهادی منشأ داده برای هر مورد گردآوری‌شده:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

کاری کنید گردش‌کار، موردهای کهنه را پیش از خلاصه‌سازی رد کند یا علامت بزند. مرحله LLM باید فقط JSON ساخت‌یافته دریافت کند و باید از آن خواسته شود `sourceUrl`، `retrievedAt`، و `asOf` را در خروجی خود حفظ کند. وقتی درون گردش‌کار به مرحله مدل با اعتبارسنجی schema نیاز دارید، از [LLM Task](/fa/tools/llm-task) استفاده کنید.

برای گردش‌کارهای قابل‌استفاده دوباره تیمی یا اجتماعی، CLI، فایل‌های `.lobster`، و هر یادداشت راه‌اندازی را به‌صورت یک مهارت یا Plugin بسته‌بندی کنید و آن را از طریق [ClawHub](/clawhub) منتشر کنید. حفاظ‌های اختصاصی گردش‌کار را در همان بسته نگه دارید، مگر اینکه API مربوط به Plugin فاقد قابلیت عمومی مورد نیاز باشد.

## حالت‌های همگام‌سازی

### حالت مدیریت‌شده

Task Flow مالک چرخه عمر از ابتدا تا انتها است. این بخش وظایف را به‌عنوان مراحل جریان ایجاد می‌کند، آن‌ها را تا تکمیل پیش می‌برد، و وضعیت جریان را به‌صورت خودکار جلو می‌برد.

مثال: جریان گزارش هفتگی که (1) داده‌ها را گردآوری می‌کند، (2) گزارش را تولید می‌کند، و (3) آن را تحویل می‌دهد. Task Flow هر مرحله را به‌عنوان یک وظیفه پس‌زمینه ایجاد می‌کند، تا تکمیل آن منتظر می‌ماند، سپس به مرحله بعدی می‌رود.

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### حالت آینه‌شده

Task Flow وظایف ایجادشده از بیرون را مشاهده می‌کند و وضعیت جریان را بدون مالکیت ایجاد وظیفه همگام نگه می‌دارد. این حالت زمانی مفید است که وظایف از کارهای Cron، فرمان‌های CLI، یا منابع دیگر سرچشمه می‌گیرند و می‌خواهید نمایی یکپارچه از پیشرفت آن‌ها به‌عنوان یک جریان داشته باشید.

مثال: سه کار Cron مستقل که با هم یک روال «عملیات صبحگاهی» را تشکیل می‌دهند. یک جریان آینه‌شده پیشرفت جمعی آن‌ها را بدون کنترل اینکه چه زمانی یا چگونه اجرا می‌شوند، رهگیری می‌کند.

## وضعیت پایدار و رهگیری بازبینی

هر جریان وضعیت خود را پایدار نگه می‌دارد و بازبینی‌ها را رهگیری می‌کند تا پیشرفت پس از راه‌اندازی‌های دوباره Gateway باقی بماند. رهگیری بازبینی زمانی که چندین منبع تلاش می‌کنند همان جریان را هم‌زمان جلو ببرند، تشخیص تعارض را ممکن می‌کند.
رجیستری جریان از SQLite با نگهداشت محدود write-ahead-log استفاده می‌کند، از جمله
checkpointهای دوره‌ای و هنگام خاموشی، تا Gatewayهای طولانی‌اجرا فایل‌های جانبی
`registry.sqlite-wal` نامحدود نگه ندارند.

## رفتار لغو

`openclaw tasks flow cancel` یک قصد لغو چسبنده روی جریان تنظیم می‌کند. وظایف فعال درون جریان لغو می‌شوند و هیچ مرحله جدیدی شروع نمی‌شود. قصد لغو در طول راه‌اندازی‌های دوباره باقی می‌ماند، بنابراین یک جریان لغوشده حتی اگر Gateway پیش از پایان یافتن همه وظایف فرزند دوباره راه‌اندازی شود، لغوشده می‌ماند.

## فرمان‌های CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| فرمان                           | توضیح                                   |
| --------------------------------- | --------------------------------------------- |
| `openclaw tasks flow list`        | جریان‌های رهگیری‌شده را با وضعیت و حالت همگام‌سازی نشان می‌دهد |
| `openclaw tasks flow show <id>`   | یک جریان را بر اساس شناسه جریان یا کلید جست‌وجو بررسی کنید     |
| `openclaw tasks flow cancel <id>` | یک جریان در حال اجرا و وظایف فعال آن را لغو کنید    |

## ارتباط جریان‌ها با وظایف

جریان‌ها وظایف را هماهنگ می‌کنند، جایگزین آن‌ها نمی‌شوند. یک جریان واحد ممکن است در طول عمر خود چندین وظیفه پس‌زمینه را پیش ببرد. برای بررسی رکوردهای وظیفه منفرد از `openclaw tasks` و برای بررسی جریان ارکستراسیون‌کننده از `openclaw tasks flow` استفاده کنید.
