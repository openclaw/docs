OpenClaw پیش از اجرای یک نوبت (ساخت زمینه مدل)، **اصلاحات اختصاصی ارائه‌دهنده** را روی رونوشت‌ها اعمال می‌کند. بیشتر این موارد تنظیمات **درون‌حافظه‌ای** هستند که برای برآورده‌کردن الزامات سخت‌گیرانه ارائه‌دهنده استفاده می‌شوند. یک گذر جداگانه برای ترمیم فایل نشست نیز ممکن است پیش از بارگذاری نشست، JSONL ذخیره‌شده را بازنویسی کند؛ این کار یا با حذف خط‌های JSONL بدشکل انجام می‌شود یا با ترمیم نوبت‌های ماندگارشده‌ای که از نظر نحوی معتبرند اما شناخته شده‌اند که هنگام بازپخش توسط یک
ارائه‌دهنده رد می‌شوند. وقتی ترمیمی انجام شود، فایل اصلی در کنار
فایل نشست پشتیبان‌گیری می‌شود.

دامنه شامل این موارد است:

- بیرون‌ماندن زمینه اعلانِ فقط زمان اجرا از نوبت‌های رونوشتِ قابل مشاهده برای کاربر
- پاک‌سازی شناسه فراخوانی ابزار
- اعتبارسنجی ورودی فراخوانی ابزار
- ترمیم جفت‌سازی نتیجه ابزار
- اعتبارسنجی / مرتب‌سازی نوبت
- پاک‌سازی امضای فکر
- پاک‌سازی امضای Thinking
- پاک‌سازی payload تصویر
- پاک‌سازی بلوک متن خالی پیش از بازپخش ارائه‌دهنده
- برچسب‌گذاری خاستگاه ورودی کاربر (برای اعلان‌های مسیریابی‌شده بین نشست‌ها)
- ترمیم نوبت خطای خالی assistant برای بازپخش Bedrock Converse

اگر به جزئیات ذخیره‌سازی رونوشت نیاز دارید، ببینید:

- [بررسی عمیق مدیریت نشست](/fa/reference/session-management-compaction)

## قاعده سراسری: زمینه زمان اجرا رونوشت کاربر نیست

زمینه زمان اجرا/سیستم می‌تواند برای یک نوبت به اعلان مدل افزوده شود، اما
محتوای نوشته‌شده توسط کاربر نهایی نیست. OpenClaw یک بدنه اعلانِ جداگانه
روبه‌رونوشت برای پاسخ‌های Gateway، پیگیری‌های در صف، ACP، CLI، و اجراهای
Pi تعبیه‌شده نگه می‌دارد. نوبت‌های قابل مشاهده کاربر که ذخیره می‌شوند، به‌جای
اعلان غنی‌شده با زمان اجرا، از همان بدنه رونوشت استفاده می‌کنند.

برای نشست‌های قدیمی که قبلا wrapperهای زمان اجرا را ماندگار کرده‌اند، سطح‌های تاریخچه Gateway
پیش از برگرداندن پیام‌ها به کلاینت‌های WebChat،
TUI، REST، یا SSE یک نمایش‌نگاشت اعمال می‌کنند.

## قاعده سراسری: پاک‌سازی تصویر

payloadهای تصویر همیشه پاک‌سازی می‌شوند تا از ردشدن در سمت ارائه‌دهنده به دلیل محدودیت‌های
اندازه جلوگیری شود (کوچک‌سازی/فشرده‌سازی دوباره تصاویر base64 بیش‌ازحد بزرگ).

این کار همچنین به کنترل فشار توکن ناشی از تصویر برای مدل‌های دارای قابلیت بینایی کمک می‌کند.
ابعاد حداکثر پایین‌تر معمولا مصرف توکن را کاهش می‌دهند؛ ابعاد بالاتر جزئیات را حفظ می‌کنند.

پیاده‌سازی:

- `sanitizeSessionMessagesImages` در `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` در `src/agents/tool-images.ts`
- حداکثر ضلع تصویر از طریق `agents.defaults.imageMaxDimensionPx` قابل پیکربندی است (پیش‌فرض: `1200`).
- بلوک‌های متن خالی هنگام پیمایش محتوای بازپخش در این گذر حذف می‌شوند. نوبت‌های assistant
  که خالی می‌شوند از نسخه بازپخش حذف می‌شوند؛ نوبت‌های user و نتیجه ابزار
  که خالی می‌شوند یک placeholder غیرخالی برای محتوای حذف‌شده دریافت می‌کنند.

## قاعده سراسری: فراخوانی‌های ابزار بدشکل

بلوک‌های فراخوانی ابزار assistant که هم `input` و هم `arguments` را ندارند، پیش از ساخته‌شدن زمینه مدل حذف می‌شوند. این کار از ردشدن توسط ارائه‌دهنده به دلیل فراخوانی‌های ابزار نیمه‌ماندگار
جلوگیری می‌کند (برای مثال، پس از شکست ناشی از محدودیت نرخ).

پیاده‌سازی:

- `sanitizeToolCallInputs` در `src/agents/session-transcript-repair.ts`
- اعمال‌شده در `sanitizeSessionHistory` در `src/agents/pi-embedded-runner/replay-history.ts`

## قاعده سراسری: خاستگاه ورودی بین‌نشستی

وقتی یک agent از طریق `sessions_send` اعلانی را به نشستی دیگر می‌فرستد (شامل
گام‌های پاسخ/اعلان agent به agent)، OpenClaw نوبت user ایجادشده را با این مقدار ماندگار می‌کند:

- `message.provenance.kind = "inter_session"`

OpenClaw همچنین پیش از متن اعلان مسیریابی‌شده، در همان نوبت یک نشانگر `[Inter-session message ... isUser=false]`
اضافه می‌کند تا فراخوانی مدل فعال بتواند
خروجی نشست خارجی را از دستورهای کاربر نهایی بیرونی تشخیص دهد. این نشانگر در صورت وجود
نشست مبدا، کانال، و ابزار را شامل می‌شود. رونوشت همچنان برای سازگاری با ارائه‌دهنده از
`role: "user"` استفاده می‌کند، اما متن قابل مشاهده و فراداده خاستگاه
هر دو نوبت را به‌عنوان داده بین‌نشستی علامت‌گذاری می‌کنند.

هنگام بازسازی زمینه، OpenClaw همین نشانگر را روی نوبت‌های user بین‌نشستی
قدیمی‌تری که فقط فراداده خاستگاه دارند اعمال می‌کند.

## ماتریس ارائه‌دهنده (رفتار فعلی)

**OpenAI / OpenAI Codex**

- فقط پاک‌سازی تصویر.
- حذف امضاهای reasoning یتیم (آیتم‌های reasoning مستقل بدون بلوک محتوای بعدی) برای رونوشت‌های OpenAI Responses/Codex، و حذف reasoning قابل بازپخش OpenAI پس از تغییر مسیر مدل.
- حفظ payloadهای آیتم reasoning قابل بازپخش OpenAI Responses، شامل آیتم‌های خلاصه خالیِ رمزگذاری‌شده، تا بازپخش دستی/WebSocket حالت `rs_*` مورد نیاز را همراه با آیتم‌های خروجی assistant نگه دارد.
- بدون پاک‌سازی شناسه فراخوانی ابزار.
- ترمیم جفت‌سازی نتیجه ابزار ممکن است خروجی‌های واقعی منطبق را جابه‌جا کند و برای فراخوانی‌های ابزار گمشده خروجی‌های `aborted` به سبک Codex بسازد.
- بدون اعتبارسنجی یا مرتب‌سازی دوباره نوبت.
- خروجی‌های ابزار گمشده خانواده OpenAI Responses به‌صورت `aborted` ساخته می‌شوند تا با نرمال‌سازی بازپخش Codex همخوان باشند.
- بدون حذف امضای thought.

**Gemma 4 سازگار با OpenAI**

- بلوک‌های thinking/reasoning تاریخی assistant پیش از بازپخش حذف می‌شوند تا سرورهای
  Gemma 4 محلی سازگار با OpenAI محتوای reasoning نوبت‌های قبلی را دریافت نکنند.
- ادامه‌های فراخوانی ابزار در همان نوبت فعلی، بلوک reasoning assistant را
  تا زمانی که نتیجه ابزار بازپخش شود متصل به فراخوانی ابزار نگه می‌دارند.

**Google (Generative AI / Gemini CLI / Antigravity)**

- پاک‌سازی شناسه فراخوانی ابزار: فقط حروف و اعداد سخت‌گیرانه.
- ترمیم جفت‌سازی نتیجه ابزار و نتایج ابزار مصنوعی.
- اعتبارسنجی نوبت (تناوب نوبت به سبک Gemini).
- اصلاح مرتب‌سازی نوبت Google (اگر تاریخچه با assistant شروع شود، یک bootstrap کوچک user در ابتدا اضافه می‌شود).
- Antigravity Claude: نرمال‌سازی امضاهای thinking؛ حذف بلوک‌های thinking بدون امضا.

**Anthropic / Minimax (سازگار با Anthropic)**

- ترمیم جفت‌سازی نتیجه ابزار و نتایج ابزار مصنوعی.
- اعتبارسنجی نوبت (ادغام نوبت‌های user پیاپی برای برآوردن تناوب سخت‌گیرانه).
- نوبت‌های پیش‌پر assistant انتهایی از payloadهای خروجی Anthropic Messages
  هنگام فعال‌بودن thinking حذف می‌شوند، شامل مسیرهای Cloudflare AI Gateway.
- بلوک‌های Thinking با امضاهای بازپخش گمشده، خالی، یا فقط شامل فاصله
  پیش از تبدیل ارائه‌دهنده حذف می‌شوند. اگر این کار یک نوبت assistant را خالی کند، OpenClaw
  شکل نوبت را با متن omitted-reasoning غیرخالی حفظ می‌کند.
- نوبت‌های قدیمی‌ترِ فقط thinking از assistant که باید حذف شوند، با
  متن omitted-reasoning غیرخالی جایگزین می‌شوند تا adapterهای ارائه‌دهنده نوبت
  بازپخش را حذف نکنند.

**Amazon Bedrock (Converse API)**

- نوبت‌های خطای stream خالی assistant پیش از بازپخش به یک بلوک متن fallback غیرخالی
  ترمیم می‌شوند. Bedrock Converse پیام‌های assistant با `content: []` را رد می‌کند، بنابراین
  نوبت‌های assistant ماندگارشده با `stopReason: "error"` و محتوای خالی نیز
  پیش از بارگذاری روی دیسک ترمیم می‌شوند.
- نوبت‌های خطای stream از assistant که فقط بلوک‌های متن خالی دارند، به‌جای بازپخش یک بلوک خالی نامعتبر،
  از نسخه بازپخش درون‌حافظه‌ای حذف می‌شوند.
- بلوک‌های thinking مربوط به Claude با امضاهای بازپخش گمشده، خالی، یا فقط شامل فاصله
  پیش از بازپخش Converse حذف می‌شوند. اگر این کار یک نوبت assistant را خالی کند، OpenClaw
  شکل نوبت را با متن omitted-reasoning غیرخالی حفظ می‌کند.
- نوبت‌های قدیمی‌ترِ فقط thinking از assistant که باید حذف شوند، با
  متن omitted-reasoning غیرخالی جایگزین می‌شوند تا بازپخش Converse شکل سخت‌گیرانه نوبت را حفظ کند.
- بازپخش، نوبت‌های assistant مربوط به آینه تحویل OpenClaw و تزریق‌شده توسط gateway را فیلتر می‌کند.
- پاک‌سازی تصویر از طریق قاعده سراسری اعمال می‌شود.

**Mistral (شامل تشخیص بر پایه شناسه مدل)**

- پاک‌سازی شناسه فراخوانی ابزار: strict9 (حروف و اعداد با طول 9).

**OpenRouter Gemini**

- پاک‌سازی امضای thought: مقدارهای `thought_signature` غیر-base64 حذف می‌شوند (base64 نگه داشته می‌شود).

**سایر موارد**

- فقط پاک‌سازی تصویر.

## رفتار تاریخی (پیش از 2026.1.22)

پیش از انتشار 2026.1.22، OpenClaw چندین لایه بهداشت رونوشت اعمال می‌کرد:

- یک **افزونه پاک‌سازی رونوشت** در هر ساخت زمینه اجرا می‌شد و می‌توانست:
  - جفت‌سازی استفاده از ابزار/نتیجه را ترمیم کند.
  - شناسه‌های فراخوانی ابزار را پاک‌سازی کند (شامل یک حالت غیرسخت‌گیرانه که `_`/`-` را حفظ می‌کرد).
- runner همچنین پاک‌سازی اختصاصی ارائه‌دهنده را انجام می‌داد که باعث دوباره‌کاری می‌شد.
- تغییرات اضافی بیرون از سیاست ارائه‌دهنده رخ می‌داد، از جمله:
  - حذف تگ‌های `<final>` از متن assistant پیش از ماندگارسازی.
  - حذف نوبت‌های خطای خالی assistant.
  - کوتاه‌کردن محتوای assistant پس از فراخوانی‌های ابزار.

این پیچیدگی باعث رگرسیون‌های میان‌ارائه‌دهنده‌ای شد (به‌ویژه جفت‌سازی `call_id|fc_id` در
`openai-responses`). پاک‌سازی 2026.1.22 افزونه را حذف کرد، منطق را در runner متمرکز کرد، و OpenAI را به‌جز پاک‌سازی تصویر، **بدون دست‌کاری** کرد.
