## منبع حقیقت: Gateway

OpenClaw حول یک **فرایند Gateway** واحد طراحی شده است که مالک وضعیت جلسه است.

- رابط‌های کاربری (برنامه macOS، Control UI وب، TUI) باید فهرست جلسات و شمارش توکن‌ها را از Gateway پرس‌وجو کنند.
- در حالت راه‌دور، فایل‌های جلسه روی میزبان راه‌دور هستند؛ «بررسی فایل‌های محلی Mac شما» آنچه Gateway استفاده می‌کند را منعکس نخواهد کرد.

## دو لایه پایداری

OpenClaw جلسات را در دو لایه پایدار می‌کند:

1. **ذخیره‌گاه جلسه (`sessions.json`)**
   - نگاشت کلید/مقدار: `sessionKey -> SessionEntry`
   - کوچک، تغییرپذیر، و ایمن برای ویرایش (یا حذف ورودی‌ها)
   - فراداده جلسه را ردیابی می‌کند (شناسه جلسه فعلی، آخرین فعالیت، تغییرگرها، شمارنده‌های توکن و غیره)

2. **رونوشت (`<sessionId>.jsonl`)**
   - رونوشت فقط-افزودنی با ساختار درختی (ورودی‌ها `id` + `parentId` دارند)
   - گفت‌وگوی واقعی + فراخوانی‌های ابزار + خلاصه‌های Compaction را ذخیره می‌کند
   - برای بازسازی زمینه مدل در نوبت‌های آینده استفاده می‌شود
   - نقاط بازرسی بزرگ اشکال‌زدایی پیش از Compaction وقتی رونوشت فعال از سقف اندازه نقطه بازرسی فراتر برود، نادیده گرفته می‌شوند تا از ایجاد یک کپی غول‌آسای دوم `.checkpoint.*.jsonl` جلوگیری شود.

خواننده‌های تاریخچه Gateway باید از مادی‌سازی کل رونوشت پرهیز کنند، مگر اینکه سطح موردنظر صراحتا به دسترسی دلخواه به تاریخچه نیاز داشته باشد. تاریخچه صفحه اول، تاریخچه چت جاسازی‌شده، بازیابی پس از راه‌اندازی دوباره، و بررسی‌های توکن/مصرف از خواندن‌های محدودِ دُم استفاده می‌کنند. پویش‌های کامل رونوشت از طریق نمایه ناهمگام رونوشت انجام می‌شوند که بر اساس مسیر فایل به‌علاوه `mtimeMs`/`size` کش می‌شود و میان خواننده‌های همزمان مشترک است.

## مکان‌های روی دیسک

به ازای هر عامل، روی میزبان Gateway:

- ذخیره‌گاه: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- رونوشت‌ها: `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
  - جلسات موضوعی Telegram: `.../<sessionId>-topic-<threadId>.jsonl`

OpenClaw این‌ها را از طریق `src/config/sessions.ts` resolve می‌کند.

## نگهداری ذخیره‌گاه و کنترل‌های دیسک

پایداری جلسه برای `sessions.json`، آرتیفکت‌های رونوشت، و sidecarهای trajectory کنترل‌های نگهداری خودکار (`session.maintenance`) دارد:

- `mode`: `warn` (پیش‌فرض) یا `enforce`
- `pruneAfter`: آستانه سن ورودی کهنه (پیش‌فرض `30d`)
- `maxEntries`: سقف ورودی‌ها در `sessions.json` (پیش‌فرض `500`)
- `resetArchiveRetention`: نگهداری برای آرشیوهای رونوشت `*.reset.<timestamp>` (پیش‌فرض: همان `pruneAfter`؛ `false` پاک‌سازی را غیرفعال می‌کند)
- `maxDiskBytes`: بودجه اختیاری پوشه جلسات
- `highWaterBytes`: هدف اختیاری پس از پاک‌سازی (پیش‌فرض `80%` از `maxDiskBytes`)

نوشتن‌های عادی Gateway از طریق نویسنده جلسه مخصوص هر ذخیره‌گاه عبور می‌کنند که جهش‌های درون‌فرایندی را بدون گرفتن قفل فایل زمان اجرا سریالی می‌کند. کمک‌کننده‌های patch مسیر داغ، تا وقتی آن اسلات نویسنده را نگه داشته‌اند، کش تغییرپذیر اعتبارسنجی‌شده را قرض می‌گیرند؛ بنابراین فایل‌های بزرگ `sessions.json` برای هر به‌روزرسانی فراداده clone یا دوباره‌خوانی نمی‌شوند. کد زمان اجرا باید `updateSessionStore(...)` یا `updateSessionStoreEntry(...)` را ترجیح دهد؛ ذخیره‌های مستقیم کل ذخیره‌گاه ابزارهای سازگاری و نگهداری آفلاین هستند. وقتی Gateway در دسترس است، `openclaw sessions cleanup` و `openclaw agents delete` غیر dry-run جهش‌های ذخیره‌گاه را به Gateway واگذار می‌کنند تا پاک‌سازی به همان صف نویسنده بپیوندد؛ `--store <path>` مسیر صریح تعمیر آفلاین برای نگهداری مستقیم فایل است. پاک‌سازی `maxEntries` همچنان برای سقف‌های اندازه تولیدی دسته‌بندی می‌شود، بنابراین یک ذخیره‌گاه ممکن است پیش از آنکه پاک‌سازی high-water بعدی آن را دوباره کوچک کند، برای مدت کوتاهی از سقف پیکربندی‌شده فراتر برود. خواندن‌های ذخیره‌گاه جلسه هنگام راه‌اندازی Gateway ورودی‌ها را هرس یا محدود نمی‌کنند؛ برای پاک‌سازی از نوشتن‌ها یا `openclaw sessions cleanup --enforce` استفاده کنید. `openclaw sessions cleanup --enforce` همچنان سقف پیکربندی‌شده را بی‌درنگ اعمال می‌کند.

نگهداری اشاره‌گرهای بادوام گفت‌وگوی خارجی مانند جلسات گروه و جلسات چت scoped به thread را حفظ می‌کند، اما ورودی‌های مصنوعی زمان اجرا برای Cron، hookها، Heartbeat، ACP، و زیردستیارها همچنان وقتی از سن، تعداد، یا بودجه دیسک پیکربندی‌شده فراتر بروند، می‌توانند حذف شوند.

OpenClaw دیگر هنگام نوشتن‌های Gateway پشتیبان‌های چرخشی خودکار `sessions.json.bak.*` ایجاد نمی‌کند. کلید قدیمی `session.maintenance.rotateBytes` نادیده گرفته می‌شود و `openclaw doctor --fix` آن را از پیکربندی‌های قدیمی‌تر حذف می‌کند.

جهش‌های رونوشت از قفل نوشتن جلسه روی فایل رونوشت استفاده می‌کنند. کسب قفل تا `session.writeLock.acquireTimeoutMs` منتظر می‌ماند و سپس خطای جلسه مشغول را نشان می‌دهد؛ مقدار پیش‌فرض `60000` میلی‌ثانیه است. این مقدار را فقط زمانی افزایش دهید که کارهای مشروع آماده‌سازی، پاک‌سازی، Compaction، یا آینه‌سازی رونوشت روی ماشین‌های کند مدت بیشتری رقابت می‌کنند. تشخیص قفل کهنه و هشدارهای حداکثر زمان نگه‌داری همچنان سیاست‌های جداگانه باقی می‌مانند.

ترتیب اعمال پاک‌سازی بودجه دیسک (`mode: "enforce"`):

1. ابتدا قدیمی‌ترین آرتیفکت‌های آرشیوشده، رونوشت orphan، یا trajectory orphan را حذف کنید.
2. اگر هنوز بالاتر از هدف بود، قدیمی‌ترین ورودی‌های جلسه و فایل‌های رونوشت/trajectory آن‌ها را بیرون کنید.
3. ادامه دهید تا مصرف در `highWaterBytes` یا پایین‌تر از آن باشد.

در `mode: "warn"`، OpenClaw خروج‌های احتمالی را گزارش می‌کند اما ذخیره‌گاه/فایل‌ها را جهش نمی‌دهد.

نگهداری را در صورت نیاز اجرا کنید:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

## جلسات Cron و گزارش‌های اجرا

اجرای Cron ایزوله نیز ورودی‌ها/رونوشت‌های جلسه ایجاد می‌کند و کنترل‌های نگهداری اختصاصی دارند:

- `cron.sessionRetention` (پیش‌فرض `24h`) جلسات اجرای Cron ایزوله قدیمی را از ذخیره‌گاه جلسه هرس می‌کند (`false` غیرفعال می‌کند).
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` فایل‌های `~/.openclaw/cron/runs/<jobId>.jsonl` را هرس می‌کنند (پیش‌فرض‌ها: `2_000_000` بایت و `2000` خط).

وقتی Cron با اجبار یک جلسه اجرای ایزوله جدید ایجاد می‌کند، پیش از نوشتن ردیف جدید، ورودی جلسه قبلی `cron:<jobId>` را پاک‌سازی می‌کند. ترجیحات ایمن مانند تنظیمات thinking/fast/verbose، برچسب‌ها، و overrideهای صریح انتخاب‌شده توسط کاربر برای model/auth را حمل می‌کند. زمینه گفت‌وگوی محیطی مانند مسیریابی channel/group، سیاست ارسال یا صف، elevation، origin، و اتصال زمان اجرای ACP را حذف می‌کند تا یک اجرای ایزوله تازه نتواند تحویل کهنه یا اختیار زمان اجرا را از یک اجرای قدیمی‌تر به ارث ببرد.

## کلیدهای جلسه (`sessionKey`)

یک `sessionKey` مشخص می‌کند _در کدام bucket گفت‌وگو_ هستید (مسیریابی + ایزوله‌سازی).

الگوهای رایج:

- چت اصلی/مستقیم (به ازای هر عامل): `agent:<agentId>:<mainKey>` (پیش‌فرض `main`)
- گروه: `agent:<agentId>:<channel>:group:<id>`
- اتاق/channel (Discord/Slack): `agent:<agentId>:<channel>:channel:<id>` یا `...:room:<id>`
- Cron: `cron:<job.id>`
- Webhook: `hook:<uuid>` (مگر اینکه override شود)

قواعد canonical در [/concepts/session](/fa/concepts/session) مستند شده‌اند.

## شناسه‌های جلسه (`sessionId`)

هر `sessionKey` به یک `sessionId` فعلی اشاره می‌کند (فایل رونوشت که گفت‌وگو را ادامه می‌دهد).

قواعد کلی:

- **بازنشانی** (`/new`، `/reset`) برای آن `sessionKey` یک `sessionId` جدید ایجاد می‌کند.
- **بازنشانی روزانه** (پیش‌فرض ساعت 4:00 صبح به وقت محلی روی میزبان Gateway) در پیام بعدی پس از مرز بازنشانی، یک `sessionId` جدید ایجاد می‌کند.
- **انقضای بیکاری** (`session.reset.idleMinutes` یا `session.idleMinutes` قدیمی) وقتی پیامی پس از پنجره بیکاری برسد، یک `sessionId` جدید ایجاد می‌کند. وقتی روزانه + بیکاری هر دو پیکربندی شده باشند، هرکدام زودتر منقضی شود برنده است.
- **رویدادهای سیستمی** (Heartbeat، بیدارباش‌های Cron، اعلان‌های exec، bookkeeping Gateway) ممکن است ردیف جلسه را جهش دهند اما تازگی بازنشانی روزانه/بیکاری را تمدید نمی‌کنند. rollover بازنشانی، پیش از ساخته‌شدن prompt تازه، اعلان‌های رویداد سیستمی صف‌شده برای جلسه قبلی را دور می‌اندازد.
- **سیاست fork والد** هنگام ایجاد thread یا fork زیردستیار از شاخه فعال PI استفاده می‌کند. اگر آن شاخه خیلی بزرگ باشد، OpenClaw فرزند را به‌جای شکست‌خوردن یا به‌ارث‌بردن تاریخچه غیرقابل استفاده، با زمینه ایزوله شروع می‌کند. سیاست اندازه‌گذاری خودکار است؛ پیکربندی قدیمی `session.parentForkMaxTokens` توسط `openclaw doctor --fix` حذف می‌شود.

جزئیات پیاده‌سازی: تصمیم در `initSessionState()` در `src/auto-reply/reply/session.ts` انجام می‌شود.

## schema ذخیره‌گاه جلسه (`sessions.json`)

نوع مقدار ذخیره‌گاه `SessionEntry` در `src/config/sessions.ts` است.

فیلدهای کلیدی (کامل نیست):

- `sessionId`: شناسه رونوشت فعلی (نام فایل از این مشتق می‌شود مگر اینکه `sessionFile` تنظیم شده باشد)
- `sessionStartedAt`: timestamp شروع برای `sessionId` فعلی؛ تازگی بازنشانی روزانه از این استفاده می‌کند. ردیف‌های قدیمی ممکن است آن را از header جلسه JSONL مشتق کنند.
- `lastInteractionAt`: timestamp آخرین تعامل واقعی کاربر/channel؛ تازگی بازنشانی بیکاری از این استفاده می‌کند تا Heartbeat، Cron، و رویدادهای exec جلسات را زنده نگه ندارند. ردیف‌های قدیمی بدون این فیلد برای تازگی بیکاری به زمان شروع جلسه بازیابی‌شده برمی‌گردند.
- `updatedAt`: timestamp آخرین جهش ردیف ذخیره‌گاه، که برای فهرست‌کردن، هرس، و bookkeeping استفاده می‌شود. این مرجع تازگی بازنشانی روزانه/بیکاری نیست.
- `sessionFile`: override اختیاری مسیر صریح رونوشت
- `chatType`: `direct | group | room` (به رابط‌های کاربری و سیاست ارسال کمک می‌کند)
- `provider`، `subject`، `room`، `space`، `displayName`: فراداده برای برچسب‌گذاری گروه/channel
- تغییرگرها:
  - `thinkingLevel`، `verboseLevel`، `reasoningLevel`، `elevatedLevel`
  - `sendPolicy` (override به ازای هر جلسه)
- انتخاب مدل:
  - `providerOverride`، `modelOverride`، `authProfileOverride`
- شمارنده‌های توکن (best-effort / وابسته به ارائه‌دهنده):
  - `inputTokens`، `outputTokens`، `totalTokens`، `contextTokens`
- `compactionCount`: اینکه auto-compaction چند بار برای این کلید جلسه کامل شده است
- `memoryFlushAt`: timestamp آخرین flush حافظه پیش از Compaction
- `memoryFlushCompactionCount`: شمارش Compaction هنگام اجرای آخرین flush

ذخیره‌گاه برای ویرایش ایمن است، اما Gateway مرجع است: ممکن است هنگام اجرای جلسات، ورودی‌ها را بازنویسی یا بازآب‌رسانی کند.

## ساختار رونوشت (`*.jsonl`)

رونوشت‌ها توسط `SessionManager` متعلق به `@mariozechner/pi-coding-agent` مدیریت می‌شوند.

فایل JSONL است:

- خط اول: header جلسه (`type: "session"`، شامل `id`، `cwd`، `timestamp`، و `parentSession` اختیاری)
- سپس: ورودی‌های جلسه با `id` + `parentId` (درخت)

انواع ورودی قابل توجه:

- `message`: پیام‌های user/assistant/toolResult
- `custom_message`: پیام‌های تزریق‌شده توسط extension که _وارد_ زمینه مدل می‌شوند (می‌توانند از UI پنهان باشند)
- `custom`: وضعیت extension که وارد زمینه مدل _نمی‌شود_
- `compaction`: خلاصه Compaction پایدارشده با `firstKeptEntryId` و `tokensBefore`
- `branch_summary`: خلاصه پایدارشده هنگام پیمایش یک شاخه درخت

OpenClaw عمدا رونوشت‌ها را «اصلاح» نمی‌کند؛ Gateway برای خواندن/نوشتن آن‌ها از `SessionManager` استفاده می‌کند.

## پنجره‌های زمینه در برابر توکن‌های ردیابی‌شده

دو مفهوم متفاوت اهمیت دارد:

1. **پنجره زمینه مدل**: سقف سخت به ازای هر مدل (توکن‌های قابل مشاهده برای مدل)
2. **شمارنده‌های ذخیره‌گاه جلسه**: آمارهای rolling که در `sessions.json` نوشته می‌شوند (برای /status و داشبوردها استفاده می‌شود)

اگر محدودیت‌ها را تنظیم می‌کنید:

- پنجره زمینه از کاتالوگ مدل می‌آید (و می‌تواند از طریق پیکربندی override شود).
- `contextTokens` در ذخیره‌گاه یک مقدار تخمینی/گزارشی زمان اجرا است؛ آن را تضمین سخت ندانید.

برای اطلاعات بیشتر، [/token-use](/fa/reference/token-use) را ببینید.

## Compaction: چیست

Compaction گفت‌وگوی قدیمی‌تر را در یک ورودی `compaction` پایدارشده در رونوشت خلاصه می‌کند و پیام‌های اخیر را دست‌نخورده نگه می‌دارد.

پس از Compaction، نوبت‌های آینده این‌ها را می‌بینند:

- خلاصه Compaction
- پیام‌های پس از `firstKeptEntryId`

Compaction **پایدار** است (برخلاف هرس جلسه). [/concepts/session-pruning](/fa/concepts/session-pruning) را ببینید.

## مرزهای قطعه‌های Compaction و جفت‌سازی ابزار

وقتی OpenClaw یک رونوشت بلند را به قطعه‌های Compaction تقسیم می‌کند، فراخوانی‌های ابزار دستیار را با ورودی‌های متناظر `toolResult` جفت نگه می‌دارد.

- اگر تقسیم سهم توکن بین یک فراخوانی ابزار و نتیجه آن قرار بگیرد، OpenClaw به‌جای جدا کردن جفت، مرز را به پیام فراخوانی ابزار دستیار منتقل می‌کند.
- اگر یک بلوک tool-result انتهایی در حالت عادی قطعه را از هدف فراتر ببرد، OpenClaw آن بلوک ابزار معلق را حفظ می‌کند و دنباله خلاصه‌نشده را دست‌نخورده نگه می‌دارد.
- بلوک‌های فراخوانی ابزار لغوشده/خطادار یک تقسیم معلق را باز نگه نمی‌دارند.

## زمان رخ دادن auto-compaction (زمان اجرای Pi)

در عامل Pi توکار، auto-compaction در دو حالت فعال می‌شود:

1. **بازیابی سرریز**: مدل یک خطای سرریز زمینه برمی‌گرداند
   (`request_too_large`, `context length exceeded`, `input exceeds the maximum
number of tokens`, `input token count exceeds the maximum number of input
tokens`, `input is too long for the model`, `ollama error: context length
exceeded`، و گونه‌های مشابه با شکل ارائه‌دهنده) → فشرده‌سازی → تلاش دوباره.
2. **نگهداشت آستانه**: پس از یک نوبت موفق، وقتی:

`contextTokens > contextWindow - reserveTokens`

که در آن:

- `contextWindow` پنجره زمینه مدل است
- `reserveTokens` فضای اضافی رزروشده برای پرامپت‌ها + خروجی بعدی مدل است

این‌ها معناشناسی زمان اجرای Pi هستند (OpenClaw رویدادها را مصرف می‌کند، اما Pi تصمیم می‌گیرد چه زمانی فشرده‌سازی کند).

OpenClaw همچنین می‌تواند پیش از باز کردن اجرای بعدی، وقتی `agents.defaults.compaction.maxActiveTranscriptBytes` تنظیم شده و فایل رونوشت فعال به آن اندازه برسد، یک Compaction محلی پیش‌پرواز را فعال کند. این یک محافظ اندازه فایل برای هزینه بازگشایی محلی است، نه آرشیوسازی خام: OpenClaw همچنان Compaction معنایی عادی را اجرا می‌کند، و به `truncateAfterCompaction` نیاز دارد تا خلاصه فشرده‌شده بتواند به رونوشت جانشین جدید تبدیل شود.

برای اجراهای Pi توکار، `agents.defaults.compaction.midTurnPrecheck.enabled: true`
یک محافظ اختیاری حلقه ابزار اضافه می‌کند. پس از افزوده شدن نتیجه ابزار و پیش از فراخوانی بعدی مدل، OpenClaw فشار پرامپت را با همان منطق بودجه پیش‌پرواز که در شروع نوبت استفاده می‌شود برآورد می‌کند. اگر زمینه دیگر جا نشود، محافظ داخل hook مربوط به `transformContext` در Pi فشرده‌سازی نمی‌کند. یک سیگنال ساختاریافته پیش‌بررسی میانه نوبت ایجاد می‌کند، ارسال پرامپت فعلی را متوقف می‌کند، و می‌گذارد حلقه اجرای بیرونی از مسیر بازیابی موجود استفاده کند: وقتی کافی باشد نتیجه‌های ابزار بیش‌ازحد بزرگ را کوتاه کند، یا حالت Compaction پیکربندی‌شده را فعال کند و دوباره تلاش کند. این گزینه به‌صورت پیش‌فرض غیرفعال است و با هر دو حالت Compaction یعنی `default` و `safeguard` کار می‌کند، از جمله Compaction safeguard با پشتوانه ارائه‌دهنده.
این مستقل از `maxActiveTranscriptBytes` است: محافظ اندازه بایت پیش از باز شدن یک نوبت اجرا می‌شود، در حالی که پیش‌بررسی میانه نوبت بعدتر در حلقه ابزار Pi توکار، پس از افزوده شدن نتایج ابزار جدید اجرا می‌شود.

## تنظیمات Compaction (`reserveTokens`, `keepRecentTokens`)

تنظیمات Compaction مربوط به Pi در تنظیمات Pi قرار دارند:

```json5
{
  compaction: {
    enabled: true,
    reserveTokens: 16384,
    keepRecentTokens: 20000,
  },
}
```

OpenClaw همچنین برای اجراهای توکار یک کف ایمنی اعمال می‌کند:

- اگر `compaction.reserveTokens < reserveTokensFloor` باشد، OpenClaw آن را افزایش می‌دهد.
- کف پیش‌فرض `20000` توکن است.
- برای غیرفعال کردن کف، `agents.defaults.compaction.reserveTokensFloor: 0` را تنظیم کنید.
- اگر از قبل بالاتر باشد، OpenClaw آن را دست‌نخورده می‌گذارد.
- `/compact` دستی به یک `agents.defaults.compaction.keepRecentTokens` صریح احترام می‌گذارد و نقطه برش دنباله اخیر Pi را نگه می‌دارد. بدون بودجه نگهداری صریح، Compaction دستی همچنان یک checkpoint سخت است و زمینه بازسازی‌شده از خلاصه جدید شروع می‌شود.
- برای اجرای پیش‌بررسی اختیاری حلقه ابزار پس از نتایج ابزار جدید و پیش از فراخوانی بعدی مدل، `agents.defaults.compaction.midTurnPrecheck.enabled: true` را تنظیم کنید. این فقط یک trigger است؛ تولید خلاصه همچنان از مسیر Compaction پیکربندی‌شده استفاده می‌کند. این مستقل از `maxActiveTranscriptBytes` است که یک محافظ اندازه بایت رونوشت فعال در شروع نوبت است.
- `agents.defaults.compaction.maxActiveTranscriptBytes` را روی یک مقدار بایت یا رشته‌ای مانند `"20mb"` تنظیم کنید تا وقتی رونوشت فعال بزرگ می‌شود، پیش از یک نوبت Compaction محلی اجرا شود. این محافظ فقط وقتی فعال است که `truncateAfterCompaction` نیز فعال باشد. برای غیرفعال کردن، آن را تنظیم‌نشده بگذارید یا روی `0` تنظیم کنید.
- وقتی `agents.defaults.compaction.truncateAfterCompaction` فعال باشد، OpenClaw پس از Compaction رونوشت فعال را به یک JSONL جانشین فشرده‌شده می‌چرخاند. رونوشت کامل قدیمی به‌جای بازنویسی درجا، آرشیو می‌ماند و از checkpoint مربوط به Compaction لینک می‌شود.

دلیل: باقی گذاشتن فضای کافی برای «خانه‌داری» چندنوبتی (مانند نوشتن‌های حافظه) پیش از آنکه Compaction اجتناب‌ناپذیر شود.

پیاده‌سازی: `ensurePiCompactionReserveTokens()` در `src/agents/pi-settings.ts`
(فراخوانی‌شده از `src/agents/pi-embedded-runner.ts`).

## ارائه‌دهنده‌های قابل اتصال Compaction

Pluginها می‌توانند از طریق `registerCompactionProvider()` در API مربوط به Plugin یک ارائه‌دهنده Compaction ثبت کنند. وقتی `agents.defaults.compaction.provider` روی شناسه یک ارائه‌دهنده ثبت‌شده تنظیم شود، افزونه safeguard خلاصه‌سازی را به‌جای پایپ‌لاین داخلی `summarizeInStages` به آن ارائه‌دهنده واگذار می‌کند.

- `provider`: شناسه یک Plugin ارائه‌دهنده Compaction ثبت‌شده. برای خلاصه‌سازی LLM پیش‌فرض، تنظیم‌نشده بگذارید.
- تنظیم یک `provider` باعث تحمیل `mode: "safeguard"` می‌شود.
- ارائه‌دهنده‌ها همان دستورالعمل‌های Compaction و سیاست حفظ شناسه‌ها را دریافت می‌کنند که مسیر داخلی دریافت می‌کند.
- safeguard همچنان زمینه پسوند نوبت‌های اخیر و نوبت‌های تقسیم‌شده را پس از خروجی ارائه‌دهنده حفظ می‌کند.
- خلاصه‌سازی safeguard داخلی، خلاصه‌های قبلی را همراه با پیام‌های جدید دوباره تقطیر می‌کند، به‌جای اینکه خلاصه کامل قبلی را عیناً حفظ کند.
- حالت safeguard ممیزی‌های کیفیت خلاصه را به‌صورت پیش‌فرض فعال می‌کند؛ برای رد کردن رفتار تلاش دوباره هنگام خروجی بدشکل، `qualityGuard.enabled: false` را تنظیم کنید.
- اگر ارائه‌دهنده شکست بخورد یا نتیجه‌ای خالی برگرداند، OpenClaw به‌صورت خودکار به خلاصه‌سازی LLM داخلی بازمی‌گردد.
- سیگنال‌های لغو/مهلت زمانی دوباره پرتاب می‌شوند (بلعیده نمی‌شوند) تا لغو فراخواننده رعایت شود.

منبع: `src/plugins/compaction-provider.ts`, `src/agents/pi-hooks/compaction-safeguard.ts`.

## خانه‌داری بی‌صدا (`NO_REPLY`)

OpenClaw از نوبت‌های «بی‌صدا» برای کارهای پس‌زمینه که کاربر نباید خروجی میانی آن‌ها را ببیند پشتیبانی می‌کند.

قرارداد:

- دستیار خروجی خود را با توکن بی‌صدای دقیق `NO_REPLY` /
  `no_reply` شروع می‌کند تا نشان دهد «پاسخی به کاربر تحویل داده نشود».
- OpenClaw این را در لایه تحویل حذف/سرکوب می‌کند.
- سرکوب توکن بی‌صدای دقیق به بزرگی/کوچکی حروف حساس نیست، بنابراین وقتی کل payload فقط همان توکن بی‌صدا باشد، `NO_REPLY` و
  `no_reply` هر دو حساب می‌شوند.
- این فقط برای نوبت‌های واقعاً پس‌زمینه/بدون تحویل است؛ میان‌بری برای درخواست‌های معمول و اقدام‌پذیر کاربر نیست.

از `2026.1.10`، OpenClaw همچنین وقتی یک قطعه جزئی با `NO_REPLY` شروع می‌شود، **استریم‌کردن پیش‌نویس/تایپ** را سرکوب می‌کند، بنابراین عملیات بی‌صدا خروجی جزئی را در میانه نوبت نشت نمی‌دهند.

## «تخلیه حافظه» پیش از Compaction (پیاده‌سازی‌شده)

هدف: پیش از رخ دادن auto-compaction، یک نوبت agentic بی‌صدا اجرا شود که وضعیت پایدار را روی دیسک بنویسد (برای نمونه `memory/YYYY-MM-DD.md` در workspace عامل) تا Compaction نتواند زمینه حیاتی را پاک کند.

OpenClaw از رویکرد **pre-threshold flush** استفاده می‌کند:

1. مصرف زمینه نشست را پایش کنید.
2. وقتی از یک «آستانه نرم» عبور می‌کند (پایین‌تر از آستانه Compaction مربوط به Pi)، یک دستور بی‌صدای «اکنون حافظه را بنویس» برای عامل اجرا کنید.
3. از توکن بی‌صدای دقیق `NO_REPLY` / `no_reply` استفاده کنید تا کاربر چیزی نبیند.

پیکربندی (`agents.defaults.compaction.memoryFlush`):

- `enabled` (پیش‌فرض: `true`)
- `model` (override اختیاری و دقیق ارائه‌دهنده/مدل برای نوبت flush، برای مثال `ollama/qwen3:8b`)
- `softThresholdTokens` (پیش‌فرض: `4000`)
- `prompt` (پیام کاربر برای نوبت flush)
- `systemPrompt` (پرامپت سیستم اضافی که برای نوبت flush افزوده می‌شود)

یادداشت‌ها:

- پرامپت/پرامپت سیستم پیش‌فرض شامل یک اشاره `NO_REPLY` برای سرکوب تحویل است.
- وقتی `model` تنظیم شده باشد، نوبت flush از همان مدل استفاده می‌کند بدون اینکه زنجیره fallback نشست فعال را به ارث ببرد، بنابراین خانه‌داری فقط محلی به‌صورت بی‌صدا به یک مدل گفت‌وگوی پولی fallback نمی‌کند.
- flush یک بار در هر چرخه Compaction اجرا می‌شود (در `sessions.json` ردیابی می‌شود).
- flush فقط برای نشست‌های Pi توکار اجرا می‌شود (backendهای CLI آن را رد می‌کنند).
- وقتی workspace نشست فقط‌خواندنی باشد (`workspaceAccess: "ro"` یا `"none"`)، flush رد می‌شود.
- برای چیدمان فایل‌های workspace و الگوهای نوشتن، [حافظه](/fa/concepts/memory) را ببینید.

Pi همچنین یک hook به نام `session_before_compact` را در API افزونه در معرض می‌گذارد، اما منطق flush مربوط به OpenClaw امروز در سمت Gateway قرار دارد.

## چک‌لیست عیب‌یابی

- کلید نشست اشتباه است؟ از [/concepts/session](/fa/concepts/session) شروع کنید و `sessionKey` را در `/status` تأیید کنید.
- ناهماهنگی store و رونوشت؟ میزبان Gateway و مسیر store را از `openclaw status` تأیید کنید.
- هرزنامه Compaction؟ بررسی کنید:
  - پنجره زمینه مدل (بیش‌ازحد کوچک)
  - تنظیمات Compaction (`reserveTokens` بیش‌ازحد بالا برای پنجره مدل می‌تواند باعث Compaction زودتر شود)
  - تورم نتیجه ابزار: pruning نشست را فعال/تنظیم کنید
- نشت نوبت‌های بی‌صدا؟ تأیید کنید پاسخ با `NO_REPLY` شروع می‌شود (توکن دقیق، غیرحساس به بزرگی/کوچکی حروف) و روی buildای هستید که اصلاح سرکوب استریم را شامل می‌شود.
