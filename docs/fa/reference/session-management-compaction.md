OpenClaw نشست‌ها را به‌صورت سرتاسری در این حوزه‌ها مدیریت می‌کند:

- **مسیریابی نشست** (اینکه پیام‌های ورودی چگونه به یک `sessionKey` نگاشت می‌شوند)
- **انبار نشست** (`sessions.json`) و چیزهایی که پیگیری می‌کند
- **پایداری رونوشت** (`*.jsonl`) و ساختار آن
- **بهداشت رونوشت** (اصلاحات ویژه ارائه‌دهنده پیش از اجراها)

## منبع حقیقت: Gateway

OpenClaw پیرامون یک **فرایند Gateway** واحد طراحی شده است که مالک وضعیت نشست است.

- رابط‌های کاربری (برنامه macOS، رابط کنترل وب، TUI) باید فهرست نشست‌ها و شمار توکن‌ها را از Gateway پرس‌وجو کنند.
- در حالت راه‌دور، فایل‌های نشست روی میزبان راه‌دور هستند؛ «بررسی فایل‌های محلی Mac شما» چیزی را که Gateway استفاده می‌کند بازتاب نمی‌دهد.

## دو لایه پایداری

OpenClaw نشست‌ها را در دو لایه پایدار می‌کند:

1. **انبار نشست (`sessions.json`)**
   - نگاشت کلید/مقدار: `sessionKey -> SessionEntry`
   - کوچک، تغییرپذیر، و امن برای ویرایش (یا حذف ورودی‌ها)
   - فراداده نشست را پیگیری می‌کند (شناسه نشست فعلی، آخرین فعالیت، کلیدهای تغییر وضعیت، شمارنده‌های توکن، و غیره)

2. **رونوشت (`<sessionId>.jsonl`)**
   - رونوشت فقط-افزودنی با ساختار درختی (ورودی‌ها `id` + `parentId` دارند)
   - مکالمه واقعی + فراخوانی‌های ابزار + خلاصه‌های Compaction را ذخیره می‌کند
   - برای بازسازی زمینه مدل در نوبت‌های آینده استفاده می‌شود
   - نقطه‌های وارسی اشکال‌زدایی بزرگِ پیش از Compaction، پس از آنکه رونوشت فعال
     از سقف اندازه نقطه وارسی فراتر برود، رد می‌شوند تا از ایجاد یک کپی غول‌آسای دوم
     `.checkpoint.*.jsonl` جلوگیری شود.

خواننده‌های تاریخچه Gateway باید از مادی‌سازی کل رونوشت خودداری کنند، مگر اینکه
سطح موردنظر صراحتا به دسترسی دلخواه به تاریخچه نیاز داشته باشد. تاریخچه صفحه اول،
تاریخچه گفت‌وگوی جاسازی‌شده، بازیابی پس از راه‌اندازی دوباره، و بررسی‌های توکن/مصرف از خواندن‌های محدودِ انتهای فایل
استفاده می‌کنند. پیمایش‌های کامل رونوشت از نمایه ناهمگام رونوشت می‌گذرند که
بر اساس مسیر فایل به‌همراه `mtimeMs`/`size` کش می‌شود و میان خواننده‌های هم‌زمان مشترک است.

## مکان‌های روی دیسک

برای هر عامل، روی میزبان Gateway:

- انبار: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- رونوشت‌ها: `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
  - نشست‌های موضوع Telegram: `.../<sessionId>-topic-<threadId>.jsonl`

OpenClaw این‌ها را از طریق `src/config/sessions.ts` حل می‌کند.

## نگهداری انبار و کنترل‌های دیسک

پایداری نشست کنترل‌های نگهداری خودکار (`session.maintenance`) برای `sessions.json`، مصنوعات رونوشت، و فایل‌های جانبی مسیر اجرا دارد:

- `mode`: `warn` (پیش‌فرض) یا `enforce`
- `pruneAfter`: آستانه سن ورودی‌های کهنه (پیش‌فرض `30d`)
- `maxEntries`: سقف ورودی‌ها در `sessions.json` (پیش‌فرض `500`)
- `resetArchiveRetention`: نگهداری آرشیوهای رونوشت `*.reset.<timestamp>` (پیش‌فرض: همان `pruneAfter`؛ `false` پاک‌سازی را غیرفعال می‌کند)
- `maxDiskBytes`: بودجه اختیاری دایرکتوری نشست‌ها
- `highWaterBytes`: هدف اختیاری پس از پاک‌سازی (پیش‌فرض `80%` از `maxDiskBytes`)

نوشتن‌های عادی Gateway از یک نویسنده نشست برای هر انبار عبور می‌کنند که جهش‌های درون‌فرایندی را بدون گرفتن قفل فایل زمان اجرا سریالی می‌کند. کمک‌گرهای وصله مسیر داغ، هنگامی که آن جایگاه نویسنده را در اختیار دارند، کش تغییرپذیر اعتبارسنجی‌شده را قرض می‌گیرند؛ بنابراین فایل‌های بزرگ `sessions.json` برای هر به‌روزرسانی فراداده کپی یا دوباره خوانده نمی‌شوند. کد زمان اجرا باید `updateSessionStore(...)` یا `updateSessionStoreEntry(...)` را ترجیح دهد؛ ذخیره‌های مستقیم کل انبار ابزارهای سازگاری و نگهداری آفلاین هستند. هنگامی که یک Gateway قابل‌دسترسی باشد، `openclaw sessions cleanup` و `openclaw agents delete` بدون `dry-run` جهش‌های انبار را به Gateway واگذار می‌کنند تا پاک‌سازی به همان صف نویسنده بپیوندد؛ `--store <path>` مسیر صریح تعمیر آفلاین برای نگهداری مستقیم فایل است. پاک‌سازی `maxEntries` همچنان برای سقف‌های اندازه تولید به‌صورت دسته‌ای انجام می‌شود، بنابراین ممکن است یک انبار پیش از آنکه پاک‌سازی بعدی سطح بالا آن را دوباره کوچک کند، برای مدت کوتاهی از سقف پیکربندی‌شده فراتر برود. خواندن‌های انبار نشست هنگام راه‌اندازی Gateway ورودی‌ها را هرس یا محدود نمی‌کنند؛ برای پاک‌سازی از نوشتن‌ها یا `openclaw sessions cleanup --enforce` استفاده کنید. `openclaw sessions cleanup --enforce` همچنان سقف پیکربندی‌شده را فورا اعمال می‌کند و مصنوعات رونوشت، نقطه وارسی، و مسیر اجرای قدیمی و بدون ارجاع را حتی وقتی هیچ بودجه دیسکی پیکربندی نشده باشد هرس می‌کند.

نگهداری اشاره‌گرهای بادوام گفت‌وگوی خارجی مانند نشست‌های گروهی
و نشست‌های گفت‌وگوی محدود به رشته را نگه می‌دارد، اما ورودی‌های مصنوعی زمان اجرا برای Cron، hookها،
Heartbeat، ACP، و زیرعامل‌ها همچنان می‌توانند وقتی از
سن، شمار، یا بودجه دیسکی پیکربندی‌شده فراتر بروند حذف شوند.

OpenClaw دیگر هنگام نوشتن‌های Gateway پشتیبان‌های چرخشی خودکار `sessions.json.bak.*` ایجاد نمی‌کند. کلید قدیمی `session.maintenance.rotateBytes` نادیده گرفته می‌شود و `openclaw doctor --fix` آن را از پیکربندی‌های قدیمی‌تر حذف می‌کند.

جهش‌های رونوشت از یک قفل نوشتن نشست روی فایل رونوشت استفاده می‌کنند. گرفتن قفل تا
`session.writeLock.acquireTimeoutMs` منتظر می‌ماند و سپس خطای نشست مشغول را آشکار می‌کند؛ مقدار پیش‌فرض `60000`
میلی‌ثانیه است. این مقدار را فقط زمانی افزایش دهید که آماده‌سازی، پاک‌سازی، Compaction، یا کار آینه‌سازی رونوشتِ مشروع
روی ماشین‌های کند، مدت بیشتری رقابت ایجاد کند. تشخیص قفل کهنه و هشدارهای بیشینه مدت نگه‌داری قفل سیاست‌های جداگانه باقی می‌مانند.

ترتیب اعمال برای پاک‌سازی بودجه دیسک (`mode: "enforce"`):

1. ابتدا قدیمی‌ترین مصنوعات آرشیوشده، رونوشت یتیم، یا مسیر اجرای یتیم را حذف کنید.
2. اگر همچنان بالاتر از هدف بود، قدیمی‌ترین ورودی‌های نشست و فایل‌های رونوشت/مسیر اجرای آن‌ها را بیرون بیندازید.
3. ادامه دهید تا مصرف در `highWaterBytes` یا پایین‌تر از آن باشد.

در `mode: "warn"`، OpenClaw بیرون‌اندازی‌های احتمالی را گزارش می‌کند اما انبار/فایل‌ها را تغییر نمی‌دهد.

نگهداری را بنا به درخواست اجرا کنید:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

## نشست‌های Cron و گزارش‌های اجرا

اجرای‌های ایزوله Cron نیز ورودی/رونوشت نشست ایجاد می‌کنند و کنترل‌های نگهداری اختصاصی دارند:

- `cron.sessionRetention` (پیش‌فرض `24h`) نشست‌های اجرای Cron ایزوله قدیمی را از انبار نشست هرس می‌کند (`false` غیرفعال می‌کند).
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` فایل‌های `~/.openclaw/cron/runs/<jobId>.jsonl` را هرس می‌کنند (پیش‌فرض‌ها: `2_000_000` بایت و `2000` خط).

وقتی Cron با اجبار یک نشست اجرای ایزوله جدید ایجاد می‌کند، ورودی نشست قبلی
`cron:<jobId>` را پیش از نوشتن ردیف جدید پاک‌سازی می‌کند. ترجیح‌های امن
مانند تنظیمات فکر کردن/سریع/پرحرف، برچسب‌ها، و بازنویسی‌های صریح
مدل/احراز هویت انتخاب‌شده توسط کاربر را حمل می‌کند. زمینه مکالمه محیطی مانند
مسیریابی کانال/گروه، سیاست ارسال یا صف، ارتقا، مبدا، و اتصال زمان اجرای ACP
را حذف می‌کند تا یک اجرای ایزوله تازه نتواند تحویل کهنه یا
اختیار زمان اجرا را از اجرای قدیمی‌تر به ارث ببرد.

## کلیدهای نشست (`sessionKey`)

یک `sessionKey` مشخص می‌کند که در _کدام سطل مکالمه_ هستید (مسیریابی + ایزولاسیون).

الگوهای رایج:

- گفت‌وگوی اصلی/مستقیم (برای هر عامل): `agent:<agentId>:<mainKey>` (پیش‌فرض `main`)
- گروه: `agent:<agentId>:<channel>:group:<id>`
- اتاق/کانال (Discord/Slack): `agent:<agentId>:<channel>:channel:<id>` یا `...:room:<id>`
- Cron: `cron:<job.id>`
- Webhook: `hook:<uuid>` (مگر اینکه بازنویسی شده باشد)

قواعد کانونی در [/concepts/session](/fa/concepts/session) مستند شده‌اند.

## شناسه‌های نشست (`sessionId`)

هر `sessionKey` به یک `sessionId` فعلی اشاره می‌کند (فایل رونوشت که مکالمه را ادامه می‌دهد).

قواعد سرانگشتی:

- **بازنشانی** (`/new`, `/reset`) یک `sessionId` جدید برای آن `sessionKey` ایجاد می‌کند.
- **بازنشانی روزانه** (پیش‌فرض ۴:۰۰ صبح به وقت محلی روی میزبان gateway) در پیام بعدی پس از مرز بازنشانی، یک `sessionId` جدید ایجاد می‌کند.
- **انقضای بی‌کاری** (`session.reset.idleMinutes` یا `session.idleMinutes` قدیمی) وقتی پیامی پس از پنجره بی‌کاری برسد، یک `sessionId` جدید ایجاد می‌کند. وقتی روزانه + بی‌کاری هر دو پیکربندی شده باشند، هرکدام زودتر منقضی شود برنده است.
- **رویدادهای سیستمی** (Heartbeat، بیدارسازی‌های Cron، اعلان‌های exec، حسابداری gateway) ممکن است ردیف نشست را تغییر دهند اما تازگی بازنشانی روزانه/بی‌کاری را تمدید نمی‌کنند. چرخش بازنشانی، پیش از ساخته شدن درخواست تازه، اعلان‌های رویداد سیستمی صف‌شده برای نشست قبلی را دور می‌اندازد.
- **سیاست شاخه‌گیری والد** هنگام ایجاد یک رشته یا شاخه زیرعامل، از شاخه فعال Pi استفاده می‌کند. اگر آن شاخه بیش از حد بزرگ باشد، OpenClaw فرزند را به‌جای شکست خوردن یا به ارث بردن تاریخچه غیرقابل‌استفاده، با زمینه ایزوله شروع می‌کند. سیاست اندازه‌سنجی خودکار است؛ پیکربندی قدیمی `session.parentForkMaxTokens` با `openclaw doctor --fix` حذف می‌شود.

جزئیات پیاده‌سازی: تصمیم در `initSessionState()` در `src/auto-reply/reply/session.ts` گرفته می‌شود.

## شِمای انبار نشست (`sessions.json`)

نوع مقدار انبار `SessionEntry` در `src/config/sessions.ts` است.

فیلدهای کلیدی (غیرجامع):

- `sessionId`: شناسه رونوشت فعلی (نام فایل از این مشتق می‌شود مگر اینکه `sessionFile` تنظیم شده باشد)
- `sessionStartedAt`: مُهر زمانی شروع برای `sessionId` فعلی؛ تازگی بازنشانی روزانه
  از این استفاده می‌کند. ردیف‌های قدیمی ممکن است آن را از سربرگ نشست JSONL مشتق کنند.
- `lastInteractionAt`: مُهر زمانی آخرین تعامل واقعی کاربر/کانال؛ تازگی بازنشانی بی‌کاری
  از این استفاده می‌کند تا Heartbeat، Cron، و رویدادهای exec نشست‌ها را
  زنده نگه ندارند. ردیف‌های قدیمی بدون این فیلد برای تازگی بی‌کاری به زمان شروع نشست بازیابی‌شده
  برمی‌گردند.
- `updatedAt`: مُهر زمانی آخرین جهش ردیف انبار، که برای فهرست‌کردن، هرس، و
  حسابداری استفاده می‌شود. این مرجع معتبر تازگی بازنشانی روزانه/بی‌کاری نیست.
- `sessionFile`: بازنویسی اختیاری مسیر صریح رونوشت
- `chatType`: `direct | group | room` (به رابط‌های کاربری و سیاست ارسال کمک می‌کند)
- `provider`، `subject`، `room`، `space`، `displayName`: فراداده برای برچسب‌گذاری گروه/کانال
- کلیدهای تغییر وضعیت:
  - `thinkingLevel`، `verboseLevel`، `reasoningLevel`، `elevatedLevel`
  - `sendPolicy` (بازنویسی برای هر نشست)
- انتخاب مدل:
  - `providerOverride`، `modelOverride`، `authProfileOverride`
- شمارنده‌های توکن (بهترین تلاش / وابسته به ارائه‌دهنده):
  - `inputTokens`، `outputTokens`، `totalTokens`، `contextTokens`
- `compactionCount`: اینکه Compaction خودکار چند بار برای این کلید نشست کامل شده است
- `memoryFlushAt`: مُهر زمانی آخرین تخلیه حافظه پیش از Compaction
- `memoryFlushCompactionCount`: شمار Compaction هنگام اجرای آخرین تخلیه

ویرایش انبار امن است، اما Gateway مرجع معتبر است: ممکن است هنگام اجرای نشست‌ها ورودی‌ها را دوباره بنویسد یا بازآب‌رسانی کند.

## ساختار رونوشت (`*.jsonl`)

رونوشت‌ها توسط `SessionManager` متعلق به `@mariozechner/pi-coding-agent` مدیریت می‌شوند.

فایل JSONL است:

- خط اول: سربرگ نشست (`type: "session"`، شامل `id`، `cwd`، `timestamp`، `parentSession` اختیاری)
- سپس: ورودی‌های نشست با `id` + `parentId` (درخت)

انواع ورودی قابل‌توجه:

- `message`: پیام‌های کاربر/دستیار/toolResult
- `custom_message`: پیام‌های تزریق‌شده توسط افزونه که _وارد_ زمینه مدل می‌شوند (می‌توانند از UI پنهان شوند)
- `custom`: وضعیت افزونه که وارد زمینه مدل _نمی‌شود_
- `compaction`: خلاصه Compaction پایدارشده با `firstKeptEntryId` و `tokensBefore`
- `branch_summary`: خلاصه پایدارشده هنگام پیمایش یک شاخه درخت

OpenClaw عمدا رونوشت‌ها را «اصلاح» نمی‌کند؛ Gateway برای خواندن/نوشتن آن‌ها از `SessionManager` استفاده می‌کند.

## پنجره‌های زمینه در برابر توکن‌های پیگیری‌شده

دو مفهوم متفاوت مهم هستند:

1. **پنجره زمینه مدل**: سقف سخت برای هر مدل (توکن‌های قابل‌مشاهده برای مدل)
2. **شمارنده‌های انبار نشست**: آمار چرخشی نوشته‌شده در `sessions.json` (برای /status و داشبوردها استفاده می‌شود)

اگر محدودیت‌ها را تنظیم می‌کنید:

- پنجره زمینه از کاتالوگ مدل می‌آید (و می‌تواند از طریق پیکربندی بازنویسی شود).
- `contextTokens` در انبار یک مقدار برآورد/گزارش‌دهی زمان اجرا است؛ آن را تضمین سخت‌گیرانه تلقی نکنید.

برای اطلاعات بیشتر، [/token-use](/fa/reference/token-use) را ببینید.

## Compaction: چیست

Compaction مکالمه قدیمی‌تر را در یک ورودی `compaction` پایدارشده در رونوشت خلاصه می‌کند و پیام‌های اخیر را دست‌نخورده نگه می‌دارد.

پس از Compaction، نوبت‌های آینده این‌ها را می‌بینند:

- خلاصه Compaction
- پیام‌های پس از `firstKeptEntryId`

Compaction **پایدار** است (برخلاف هرس نشست). ببینید [/concepts/session-pruning](/fa/concepts/session-pruning).

## مرزهای قطعه‌های Compaction و جفت‌سازی ابزار

وقتی OpenClaw یک رونوشت طولانی را به قطعه‌های Compaction تقسیم می‌کند، فراخوانی‌های ابزارِ دستیار را همراه با ورودی‌های مطابق `toolResult` نگه می‌دارد.

- اگر تقسیم بر اساس سهم توکن بین یک فراخوانی ابزار و نتیجه‌اش قرار بگیرد، OpenClaw مرز را به پیام فراخوانی ابزارِ دستیار منتقل می‌کند، به‌جای اینکه این جفت را جدا کند.
- اگر یک بلوک انتهایی نتیجه ابزار در غیر این صورت قطعه را از هدف بزرگ‌تر کند، OpenClaw آن بلوک ابزارِ در انتظار را حفظ می‌کند و دنباله خلاصه‌نشده را دست‌نخورده نگه می‌دارد.
- بلوک‌های فراخوانی ابزارِ لغوشده/خطادار یک تقسیم در انتظار را باز نگه نمی‌دارند.

## زمان وقوع Compaction خودکار (زمان‌اجرای Pi)

در عامل Pi تعبیه‌شده، Compaction خودکار در دو حالت فعال می‌شود:

1. **بازیابی سرریز**: مدل یک خطای سرریز زمینه برمی‌گرداند
   (`request_too_large`, `context length exceeded`, `input exceeds the maximum
number of tokens`, `input token count exceeds the maximum number of input
tokens`, `input is too long for the model`, `ollama error: context length
exceeded` و گونه‌های مشابه با شکل‌دهی ارائه‌دهنده) → Compaction → تلاش دوباره.
2. **نگهداشت آستانه**: پس از یک نوبت موفق، وقتی:

`contextTokens > contextWindow - reserveTokens`

که در آن:

- `contextWindow` پنجره زمینه مدل است
- `reserveTokens` فضای اضافی رزروشده برای اعلان‌ها + خروجی بعدی مدل است

این‌ها معناشناسی زمان‌اجرای Pi هستند (OpenClaw رویدادها را مصرف می‌کند، اما Pi تصمیم می‌گیرد چه زمانی Compaction انجام شود).

OpenClaw همچنین می‌تواند پیش از باز کردن اجرای بعدی، زمانی که `agents.defaults.compaction.maxActiveTranscriptBytes` تنظیم شده و فایل رونوشت فعال به آن اندازه می‌رسد، یک Compaction محلی پیش‌پرواز را فعال کند. این یک محافظ اندازه فایل برای هزینه بازگشایی محلی است، نه بایگانی خام: OpenClaw همچنان Compaction معنایی عادی را اجرا می‌کند، و به `truncateAfterCompaction` نیاز دارد تا خلاصه فشرده‌شده بتواند به یک رونوشت جانشین جدید تبدیل شود.

برای اجراهای Pi تعبیه‌شده، `agents.defaults.compaction.midTurnPrecheck.enabled: true` یک محافظ اختیاری حلقه ابزار اضافه می‌کند. پس از افزوده شدن نتیجه ابزار و پیش از فراخوانی بعدی مدل، OpenClaw فشار اعلان را با همان منطق بودجه پیش‌پرواز که در شروع نوبت استفاده می‌شود برآورد می‌کند. اگر زمینه دیگر جا نشود، محافظ داخل قلاب `transformContext` در Pi فشرده‌سازی انجام نمی‌دهد. یک سیگنال ساخت‌یافته پیش‌بررسی میان‌نوبت ایجاد می‌کند، ارسال اعلان فعلی را متوقف می‌کند، و اجازه می‌دهد حلقه اجرای بیرونی از مسیر بازیابی موجود استفاده کند: وقتی کافی است، نتایج ابزار بیش‌ازحد بزرگ را کوتاه کند، یا حالت Compaction پیکربندی‌شده را فعال کند و دوباره تلاش کند. این گزینه به‌صورت پیش‌فرض غیرفعال است و با هر دو حالت Compaction یعنی `default` و `safeguard` کار می‌کند، از جمله Compaction محافظت‌شده با پشتیبانی ارائه‌دهنده.
این مستقل از `maxActiveTranscriptBytes` است: محافظ اندازه بایت پیش از باز شدن یک نوبت اجرا می‌شود، در حالی که پیش‌بررسی میان‌نوبت بعدتر در حلقه ابزار Pi تعبیه‌شده و پس از افزوده شدن نتایج ابزار جدید اجرا می‌شود.

## تنظیمات Compaction (`reserveTokens`, `keepRecentTokens`)

تنظیمات Compaction مربوط به Pi در تنظیمات Pi قرار دارند:

```json5
{
  compaction: {
    enabled: true,
    reserveTokens: 16384,
    keepRecentTokens: 20000,
  },
}
```

OpenClaw همچنین برای اجراهای تعبیه‌شده یک کف ایمنی اعمال می‌کند:

- اگر `compaction.reserveTokens < reserveTokensFloor` باشد، OpenClaw آن را بالا می‌برد.
- کف پیش‌فرض `20000` توکن است.
- برای غیرفعال کردن کف، `agents.defaults.compaction.reserveTokensFloor: 0` را تنظیم کنید.
- اگر از قبل بالاتر باشد، OpenClaw آن را دست‌نخورده می‌گذارد.
- دستور دستی `/compact` مقدار صریح `agents.defaults.compaction.keepRecentTokens` را رعایت می‌کند و نقطه برش دنباله اخیر Pi را حفظ می‌کند. بدون بودجه نگهداری صریح، Compaction دستی همچنان یک نقطه بازرسی سخت باقی می‌ماند و زمینه بازسازی‌شده از خلاصه جدید شروع می‌شود.
- برای اجرای پیش‌بررسی اختیاری حلقه ابزار پس از نتایج ابزار جدید و پیش از فراخوانی بعدی مدل، `agents.defaults.compaction.midTurnPrecheck.enabled: true` را تنظیم کنید. این فقط یک محرک است؛ تولید خلاصه همچنان از مسیر Compaction پیکربندی‌شده استفاده می‌کند. این مستقل از `maxActiveTranscriptBytes` است، که یک محافظ اندازه بایت رونوشت فعال در شروع نوبت است.
- `agents.defaults.compaction.maxActiveTranscriptBytes` را به یک مقدار بایتی یا رشته‌ای مانند `"20mb"` تنظیم کنید تا وقتی رونوشت فعال بزرگ می‌شود، پیش از یک نوبت Compaction محلی اجرا شود. این محافظ فقط زمانی فعال است که `truncateAfterCompaction` نیز فعال باشد. برای غیرفعال کردن، آن را تنظیم‌نشده بگذارید یا روی `0` تنظیم کنید.
- وقتی `agents.defaults.compaction.truncateAfterCompaction` فعال باشد، OpenClaw پس از Compaction رونوشت فعال را به یک JSONL جانشین فشرده‌شده می‌چرخاند. رونوشت کامل قدیمی به‌جای بازنویسی در محل، بایگانی می‌ماند و از نقطه بازرسی Compaction پیوند داده می‌شود.

چرا: برای «نگهداشت» چندنوبتی (مانند نوشتن حافظه) پیش از اجتناب‌ناپذیر شدن Compaction، فضای کافی باقی بماند.

پیاده‌سازی: `ensurePiCompactionReserveTokens()` در `src/agents/pi-settings.ts`
(فراخوانی‌شده از `src/agents/pi-embedded-runner.ts`).

## ارائه‌دهنده‌های قابل اتصال Compaction

Pluginها می‌توانند از طریق `registerCompactionProvider()` در API Plugin یک ارائه‌دهنده Compaction ثبت کنند. وقتی `agents.defaults.compaction.provider` روی شناسه یک ارائه‌دهنده ثبت‌شده تنظیم شود، افزونه محافظت خلاصه‌سازی را به‌جای خط لوله داخلی `summarizeInStages` به آن ارائه‌دهنده واگذار می‌کند.

- `provider`: شناسه یک Plugin ارائه‌دهنده Compaction ثبت‌شده. برای خلاصه‌سازی پیش‌فرض LLM تنظیم‌نشده بگذارید.
- تنظیم یک `provider` حالت `mode: "safeguard"` را اجباری می‌کند.
- ارائه‌دهنده‌ها همان دستورهای Compaction و خط‌مشی حفظ شناسه را مانند مسیر داخلی دریافت می‌کنند.
- محافظت پس از خروجی ارائه‌دهنده همچنان زمینه پسوند نوبت‌های اخیر و نوبت تقسیم‌شده را حفظ می‌کند.
- خلاصه‌سازی محافظت داخلی به‌جای حفظ لفظ‌به‌لفظ کل خلاصه قبلی، خلاصه‌های پیشین را همراه با پیام‌های جدید دوباره تقطیر می‌کند.
- حالت محافظت ممیزی کیفیت خلاصه را به‌صورت پیش‌فرض فعال می‌کند؛ برای رد کردن رفتار تلاش دوباره هنگام خروجی بدشکل، `qualityGuard.enabled: false` را تنظیم کنید.
- اگر ارائه‌دهنده شکست بخورد یا نتیجه خالی برگرداند، OpenClaw به‌طور خودکار به خلاصه‌سازی LLM داخلی بازمی‌گردد.
- سیگنال‌های لغو/مهلت دوباره پرتاب می‌شوند (بلعیده نمی‌شوند) تا لغو از سوی فراخواننده رعایت شود.

منبع: `src/plugins/compaction-provider.ts`، `src/agents/pi-hooks/compaction-safeguard.ts`.

## نگهداشت بی‌صدا (`NO_REPLY`)

OpenClaw از نوبت‌های «بی‌صدا» برای کارهای پس‌زمینه پشتیبانی می‌کند؛ جایی که کاربر نباید خروجی میانی را ببیند.

قرارداد:

- دستیار خروجی خود را با توکن بی‌صدای دقیق `NO_REPLY` /
  `no_reply` شروع می‌کند تا نشان دهد «پاسخی به کاربر تحویل نده».
- OpenClaw این را در لایه تحویل حذف/سرکوب می‌کند.
- سرکوب توکن بی‌صدای دقیق به بزرگی و کوچکی حروف حساس نیست، بنابراین وقتی کل محموله فقط توکن بی‌صدا باشد، `NO_REPLY` و
  `no_reply` هر دو محسوب می‌شوند.
- این فقط برای نوبت‌های پس‌زمینه واقعی/بدون تحویل است؛ میان‌بری برای درخواست‌های عادی و اقدام‌پذیر کاربر نیست.

از نسخه `2026.1.10`، OpenClaw همچنین وقتی یک قطعه جزئی با `NO_REPLY` شروع می‌شود، **استریم پیش‌نویس/در حال تایپ** را سرکوب می‌کند، بنابراین عملیات بی‌صدا خروجی جزئی را در میانه نوبت نشت نمی‌دهند.

## «تخلیه حافظه» پیش از Compaction (پیاده‌سازی‌شده)

هدف: پیش از وقوع Compaction خودکار، یک نوبت عاملانه بی‌صدا اجرا شود که وضعیت بادوام را روی دیسک بنویسد (مثلاً `memory/YYYY-MM-DD.md` در فضای کاری عامل) تا Compaction نتواند زمینه حیاتی را پاک کند.

OpenClaw از رویکرد **تخلیه پیش‌آستانه** استفاده می‌کند:

1. پایش مصرف زمینه نشست.
2. وقتی از یک «آستانه نرم» عبور می‌کند (پایین‌تر از آستانه Compaction مربوط به Pi)، یک دستور بی‌صدای «اکنون حافظه را بنویس» برای عامل اجرا شود.
3. از توکن بی‌صدای دقیق `NO_REPLY` / `no_reply` استفاده شود تا کاربر چیزی نبیند.

پیکربندی (`agents.defaults.compaction.memoryFlush`):

- `enabled` (پیش‌فرض: `true`)
- `model` (بازنویسی اختیاری و دقیق ارائه‌دهنده/مدل برای نوبت تخلیه، برای مثال `ollama/qwen3:8b`)
- `softThresholdTokens` (پیش‌فرض: `4000`)
- `prompt` (پیام کاربر برای نوبت تخلیه)
- `systemPrompt` (اعلان سیستمی اضافی که برای نوبت تخلیه افزوده می‌شود)

نکات:

- اعلان/اعلان سیستمی پیش‌فرض شامل یک راهنمایی `NO_REPLY` برای سرکوب تحویل است.
- وقتی `model` تنظیم شده باشد، نوبت تخلیه از همان مدل استفاده می‌کند بدون اینکه زنجیره بازگشت نشست فعال را به ارث ببرد، بنابراین نگهداشت فقط محلی بی‌صدا به یک مدل گفت‌وگوی پولی بازنمی‌گردد.
- تخلیه در هر چرخه Compaction یک‌بار اجرا می‌شود (در `sessions.json` ردیابی می‌شود).
- تخلیه فقط برای نشست‌های Pi تعبیه‌شده اجرا می‌شود (پشتانه‌های CLI از آن عبور می‌کنند).
- وقتی فضای کاری نشست فقط‌خواندنی باشد (`workspaceAccess: "ro"` یا `"none"`)، تخلیه رد می‌شود.
- برای چیدمان فایل فضای کاری و الگوهای نوشتن، [حافظه](/fa/concepts/memory) را ببینید.

Pi همچنین یک قلاب `session_before_compact` در API افزونه در معرض استفاده قرار می‌دهد، اما منطق تخلیه OpenClaw امروز در سمت Gateway قرار دارد.

## فهرست عیب‌یابی

- کلید نشست اشتباه است؟ با [/concepts/session](/fa/concepts/session) شروع کنید و `sessionKey` را در `/status` تأیید کنید.
- ناسازگاری بین ذخیره‌گاه و رونوشت؟ میزبان Gateway و مسیر ذخیره‌گاه را از `openclaw status` تأیید کنید.
- هرزنامه Compaction؟ بررسی کنید:
  - پنجره زمینه مدل (بیش از حد کوچک)
  - تنظیمات Compaction (`reserveTokens` اگر برای پنجره مدل بیش از حد بالا باشد می‌تواند باعث Compaction زودتر شود)
  - تورم نتیجه ابزار: هرس نشست را فعال/تنظیم کنید
- نوبت‌های بی‌صدا نشت می‌کنند؟ تأیید کنید پاسخ با `NO_REPLY` شروع می‌شود (توکن دقیق بدون حساسیت به بزرگی و کوچکی حروف) و روی ساختی هستید که شامل اصلاح سرکوب استریم است.
