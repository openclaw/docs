یک **فرایند Gateway** به‌صورت سرتاسری مالک وضعیت نشست است. رابط‌های کاربری (برنامه macOS، رابط وب Control UI و TUI) فهرست نشست‌ها و تعداد توکن‌ها را از Gateway استعلام می‌کنند. در حالت راه‌دور، فایل‌های نشست روی میزبان راه‌دور قرار دارند؛ بنابراین بررسی فایل‌های Mac محلی شما نشان‌دهنده مواردی نیست که Gateway استفاده می‌کند.

ابتدا مستندات کلی را ببینید: [مدیریت نشست](/fa/concepts/session)، [Compaction](/fa/concepts/compaction)، [نمای کلی حافظه](/fa/concepts/memory)، [جست‌وجوی حافظه](/fa/concepts/memory-search)، [هرس نشست](/fa/concepts/session-pruning)، [بهداشت رونوشت](/fa/reference/transcript-hygiene) و مرجع کامل پیکربندی در [پیکربندی عامل](/fa/gateway/config-agents).

## دو لایه ماندگاری

1. **ردیف‌های نشست (SQLite به‌ازای هر عامل)** - نگاشت کلید/مقدار `sessionKey -> SessionEntry`. وضعیت زمان اجرای تغییرپذیری که مالک آن Gateway است. فراداده‌هایی مانند شناسه نشست جاری، آخرین فعالیت، کلیدهای تغییر وضعیت و شمارنده‌های توکن را ردیابی می‌کند.
2. **رویدادهای رونوشت (SQLite به‌ازای هر عامل)** - فقط‌افزودنی و دارای ساختار درختی (ورودی‌ها دارای `id` + `parentId` هستند). مکالمه، فراخوانی ابزارها و خلاصه‌های Compaction را ذخیره می‌کند و زمینه مدل را برای نوبت‌های آینده بازسازی می‌کند. نقاط وارسی Compaction، فراداده‌ای روی رونوشت جانشین فشرده‌شده هستند؛ یک Compaction جدید نسخه دوم `.checkpoint.*.jsonl` را نمی‌نویسد.

نصب‌های قدیمی‌تر ممکن است همچنان فایل‌های `sessions.json` را زیر پوشه
`sessions/` عامل داشته باشند. این فایل‌ها را ورودی‌های قدیمی مهاجرت ردیف نشست یا اهداف صریح
نگه‌داری آفلاین در نظر بگیرید. راه‌اندازی Gateway و `openclaw doctor --fix`
ردیف‌های فعال قدیمی و تاریخچه رونوشت را به‌طور خودکار به مخزن SQLite هر عامل
وارد می‌کنند. هرگاه به شواهد صریح بازرسی یا اعتبارسنجی نیاز دارید،
`openclaw doctor --session-sqlite inspect
--session-sqlite-all-agents` را اجرا کنید و سپس [توالی مهاجرت Doctor
](/fa/cli/doctor#session-sqlite-migration) را دنبال کنید. اگر مهاجرت پس از بایگانی‌شدن
مصنوعات رونوشت قدیمی شکست خورد، از حالت بازیابی Doctor در همان توالی استفاده کنید.
بازیابی از مانیفست‌های مهاجرت استفاده می‌کند، فقط مصنوعات پشتیبانی بایگانی‌شده
متأثر را بازمی‌گرداند، در صورت درخواست یک گزارش پاک‌سازی‌شده برای مشکل GitHub آماده می‌کند
و باعث نمی‌شود زمان اجرای فعال دوباره فایل‌های JSONL را بخواند.

خوانشگرهای تاریخچه Gateway از بارگذاری کل رونوشت در حافظه خودداری می‌کنند، مگر آنکه سطح موردنظر به دسترسی دلخواه به تاریخچه نیاز داشته باشد. تاریخچه صفحه نخست، تاریخچه گفت‌وگوی تعبیه‌شده، بازیابی پس از راه‌اندازی مجدد و بررسی توکن/مصرف، از خواندن محدود انتهای SQLite استفاده می‌کنند. پیمایش کامل رونوشت‌ها از نمایه ناهمگام رونوشت عبور می‌کند و میان خوانشگرهای هم‌زمان به اشتراک گذاشته می‌شود.

## مکان‌های روی دیسک

برای هر عامل، روی میزبان Gateway (حل‌شده از طریق `src/config/sessions.ts`):

- مخزن ردیف نشست زمان اجرا: `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- ردیف‌های رونوشت زمان اجرا: `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- مصنوعات قدیمی/بایگانی رونوشت: `~/.openclaw/agents/<agentId>/sessions/`
- ورودی قدیمی مهاجرت ردیف: `~/.openclaw/agents/<agentId>/sessions/sessions.json`

## نگه‌داری مخزن و کنترل‌های دیسک

`session.maintenance` نگه‌داری خودکار ردیف‌های نشست SQLite، ردیف‌های رونوشت SQLite، مصنوعات بایگانی و فایل‌های جانبی مسیر حرکت را کنترل می‌کند:

| کلید                     | پیش‌فرض               | توضیحات                                                                                       |
| ----------------------- | --------------------- | ------------------------------------------------------------------------------------------- |
| `mode`                  | `"enforce"`           | یا `"warn"` (فقط گزارش، بدون تغییر)                                                      |
| `pruneAfter`            | `"30d"`               | آستانه سن ورودی‌های کهنه                                                                      |
| `maxEntries`            | `500`                 | سقف تعداد ورودی‌های نشست                                                                      |
| `resetArchiveRetention` | نگه‌داری (بدون آستانه سن)  | آستانه سن بایگانی‌های رونوشت `*.reset.*`/`*.deleted.*`؛ تعیین مدت‌زمان، حذف را فعال می‌کند |
| `maxDiskBytes`          | `2gb`                 | سهمیه دیسک نشست‌ها به‌ازای هر عامل؛ `false` آن را غیرفعال می‌کند                                            |
| `highWaterBytes`        | 80% از `maxDiskBytes` | هدف پس از پاک‌سازی سهمیه                                                                 |

رونوشت‌های بایگانی‌شده به‌طور پیش‌فرض نگه‌داری می‌شوند و هنگامی که زمان اجرا از آن پشتیبانی کند، با zstd (`*.jsonl.<reason>.<timestamp>.zst`) فشرده می‌شوند؛ بنابراین حذف یا بازنشانی یک نشست هرگز تاریخچه مکالمه را بی‌سروصدا دور نمی‌ریزد. سهمیه دیسک پیش از دست‌زدن به نشست‌های فعال، ابتدا قدیمی‌ترین بایگانی‌ها را بیرون می‌اندازد.

اعمال فعال `maxDiskBytes` در SQLite، مجموع بایت‌های JSON ردیف نشست و JSON رویدادهای رونوشت را برای هر نشست اندازه می‌گیرد؛ اعمال نگه‌داری آفلاین قدیمی، فایل‌های موجود در پوشه نشست‌های انتخاب‌شده را اندازه می‌گیرد.

نشست‌های وارسی اجرای مدل Gateway (کلیدهای منطبق با `agent:*:explicit:model-run-<uuid>`) نگه‌داری ثابت و جداگانه `24h` دارند. این هرس وابسته به فشار است: فقط هنگامی اجرا می‌شود که فشار نگه‌داری/سقف ورودی‌های نشست ایجاد شده باشد و تنها پیش از مرحله پاک‌سازی/اعمال سقف سراسری ورودی‌های کهنه اجرا می‌شود. دیگر نشست‌های صریح از این دوره نگه‌داری استفاده نمی‌کنند.

ترتیب اعمال پاک‌سازی سهمیه دیسک (`mode: "enforce"`):

1. ابتدا قدیمی‌ترین مصنوعات بایگانی رونوشت، مصنوعات قدیمی یتیم یا مصنوعات مسیر حرکت یتیم را حذف کنید.
2. اگر همچنان بالاتر از هدف بود، قدیمی‌ترین ورودی‌های نشست و ردیف‌های رونوشت یا مصنوعات مسیر حرکت آن‌ها را بیرون بیندازید.
3. تا زمانی که مصرف به `highWaterBytes` یا کمتر برسد، تکرار کنید.

`mode: "warn"` بدون تغییر مخزن یا فایل‌ها، حذف‌های احتمالی را گزارش می‌کند.

نگه‌داری را عندالمطالبه اجرا کنید:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

نگه‌داری، اشاره‌گرهای پایدار مکالمات خارجی مانند نشست‌های گروهی و نشست‌های گفت‌وگوی محدود به رشته را حفظ می‌کند؛ اما ورودی‌های مصنوعی زمان اجرا (cron، قلاب‌ها، Heartbeat، ACP و عامل‌های فرعی) پس از عبور از سن، تعداد یا سهمیه دیسک پیکربندی‌شده همچنان قابل حذف هستند. اجراهای مجزای cron از کنترل جداگانه `cron.sessionRetention` استفاده می‌کنند که مستقل از نگه‌داری وارسی اجرای مدل است.

نوشتن‌های عادی Gateway از دسترسی‌دهنده نشست عبور می‌کنند؛ این دسترسی‌دهنده تغییرات SQLite هر عامل را از طریق مسیر نویسنده زمان اجرا سریال‌سازی می‌کند. کد زمان اجرا باید کمک‌کننده‌های دسترسی‌دهنده در `src/config/sessions/session-accessor.ts` را ترجیح دهد؛ کمک‌کننده‌های قدیمی `sessions.json` ابزارهای مهاجرت و نگه‌داری آفلاین هستند. هنگامی که Gateway در دسترس باشد، `openclaw sessions cleanup` و `openclaw agents delete` در حالت غیر dry-run تغییرات مخزن را به Gateway واگذار می‌کنند تا پاک‌سازی به همان صف نویسنده ملحق شود؛ `--store <path>` مسیر صریح تعمیر آفلاین برای یک مخزن قدیمی انتخاب‌شده است و همیشه محلی باقی می‌ماند (همانند `--dry-run`). پاک‌سازی `maxEntries` برای مخازن در مقیاس تولید به‌صورت دسته‌ای انجام می‌شود؛ بنابراین ممکن است یک مخزن برای مدت کوتاهی از سقف پیکربندی‌شده عبور کند تا پاک‌سازی بعدی نقطه اوج آن را دوباره به زیر سقف بازنویسی کند. خواندن‌ها هنگام راه‌اندازی Gateway هرگز ورودی‌ها را هرس یا محدود نمی‌کنند؛ فقط نوشتن‌ها یا `openclaw sessions cleanup --enforce` این کار را انجام می‌دهند. مورد دوم همچنین سقف را بی‌درنگ اعمال می‌کند و حتی بدون پیکربندی سهمیه دیسک، مصنوعات قدیمی و بدون ارجاع رونوشت، نقطه وارسی و مسیر حرکت را هرس می‌کند.

OpenClaw دیگر هنگام نوشتن‌های Gateway پشتیبان‌های چرخشی خودکار `sessions.json.bak.*` ایجاد نمی‌کند. طرح‌واره فعلی کلید قدیمی `session.maintenance.rotateBytes` را رد می‌کند و `openclaw doctor --fix` آن را از پیکربندی‌های قدیمی‌تر حذف می‌کند.

تغییرات رونوشت برای هدف رونوشت SQLite از صف نوشتن نشست استفاده می‌کنند:

| تنظیم                              | پیش‌فرض   | بازنویسی محیطی                                     |
| ------------------------------------ | --------- | ------------------------------------------------ |
| `session.writeLock.acquireTimeoutMs` | `60000`   | `OPENCLAW_SESSION_WRITE_LOCK_ACQUIRE_TIMEOUT_MS` |
| `session.writeLock.staleMs`          | `1800000` | `OPENCLAW_SESSION_WRITE_LOCK_STALE_MS`           |
| `session.writeLock.maxHoldMs`        | `300000`  | `OPENCLAW_SESSION_WRITE_LOCK_MAX_HOLD_MS`        |

`acquireTimeoutMs` مدت‌زمانی است که انتظار برای قفل، پیش از انصراف خطای مشغول‌بودن نشست را آشکار می‌کند؛ آن را فقط هنگامی افزایش دهید که آماده‌سازی، پاک‌سازی، Compaction یا کار آینه‌سازی رونوشتِ مشروع در ماشین‌های کند مدت بیشتری رقابت ایجاد کند. `staleMs` زمانی است که یک قفل موجود را می‌توان به‌عنوان قفل کهنه بازپس گرفت. `maxHoldMs` آستانه آزادسازی نگهبان درون‌فرایندی است.

### بازگشت به نسخه قدیمی پس از گذار به SQLite

پیش از اجرای نسخه قدیمی‌تر و فایل‌محور OpenClaw، مصنوعات بایگانی‌شده رونوشت قدیمی
را بازیابی کنید:

```bash
openclaw doctor --session-sqlite restore --session-sqlite-all-agents
```

مهاجرت فایل‌های قدیمی `sessions.json` را برای پشتیبانی و
بازگشت در جای خود باقی می‌گذارد، اما فایل‌های فعال JSONL رونوشت که به SQLite وارد شده‌اند
به `session-sqlite-import-archive/` تغییر نام می‌یابند. زمان‌های اجرای فایل‌محور قدیمی‌تر
مسیرهای `sessionFile` در `sessions.json` را دنبال می‌کنند؛ بنابراین پیش از راه‌اندازی
به بازیابی آن مصنوعات نیاز دارند. بازیابی از مانیفست‌های مهاجرت استفاده می‌کند، فقط مصنوعات بایگانی‌شده ثبت‌شده‌ای را
که مسیر اصلی آن‌ها موجود نیست جابه‌جا می‌کند و پایگاه‌داده SQLite را برای
بازیابی روبه‌جلو در جای خود باقی می‌گذارد.

نشست‌هایی که پس از گذار به SQLite ایجاد شده‌اند، فقط در SQLite هستند و برای
زمان اجرای فایل‌محور قدیمی‌تر ظاهر نمی‌شوند. اگر پس از بازگشت به نسخه قدیمی دوباره ارتقا می‌دهید، توالی
بازرسی و اعتبارسنجی Doctor را دوباره اجرا کنید تا OpenClaw بتواند مصنوعات قدیمی
بازیابی‌شده را پیش از واردکردن تأیید کند.

## نشست‌های Cron و گزارش‌های اجرا

اجراهای مجزای cron ورودی‌ها/رونوشت‌های نشست خود را با نگه‌داری اختصاصی ایجاد می‌کنند:

- `cron.sessionRetention` (پیش‌فرض `"24h"`) نشست‌های قدیمی اجرای مجزای cron را از مخزن هرس می‌کند؛ `false` آن را غیرفعال می‌کند.
- تاریخچه اجرا، جدیدترین 2000 ردیف پایانی را برای هر کار cron نگه می‌دارد. ردیف‌های ازدست‌رفته بازه پاک‌سازی 24 ساعته خود را حفظ می‌کنند.

هنگامی که cron به‌اجبار یک نشست اجرای مجزای جدید ایجاد می‌کند، پیش از نوشتن ردیف جدید، ورودی نشست قبلی `cron:<jobId>` را پاک‌سازی می‌کند: ترجیحات امن (تنظیمات تفکر/سریع/پرحرفی/استدلال، برچسب‌ها و نام نمایشی) و بازنویسی‌های صریح مدل/احراز هویت انتخاب‌شده توسط کاربر را منتقل می‌کند، اما زمینه محیطی مکالمه (مسیریابی کانال/گروه، سیاست ارسال/صف، ارتقای دسترسی، مبدأ و اتصال زمان اجرای ACP) را کنار می‌گذارد تا اجرای مجزای تازه نتواند اختیار قدیمی تحویل یا زمان اجرا را از اجرای قبلی به ارث ببرد.

## کلیدهای نشست (`sessionKey`)

یک `sessionKey` مشخص می‌کند در کدام سطل مکالمه هستید (مسیریابی + جداسازی). قواعد معیار: [/concepts/session](/fa/concepts/session).

| الگو                      | نمونه                                                     |
| ---------------------------- | ----------------------------------------------------------- |
| گفت‌وگوی اصلی/مستقیم (به‌ازای هر عامل) | `agent:<agentId>:<mainKey>` (پیش‌فرض `main`)                |
| گروه                        | `agent:<agentId>:<channel>:group:<id>`                      |
| اتاق/کانال (Discord/Slack) | `agent:<agentId>:<channel>:channel:<id>` یا `...:room:<id>` |
| Cron                         | `cron:<job.id>`                                             |
| Webhook                      | `hook:<uuid>` (مگر آنکه بازنویسی شده باشد)                           |

## شناسه‌های نشست (`sessionId`)

هر `sessionKey` به یک `sessionId` جاری اشاره می‌کند (هویت رونوشت SQLite که مکالمه را ادامه می‌دهد). منطق تصمیم‌گیری در `initSessionState()` در `src/auto-reply/reply/session.ts` قرار دارد.

- **بازنشانی** (`/new`، `/reset`) یک `sessionId` جدید برای آن `sessionKey` ایجاد می‌کند.
- **بازنشانی روزانه** (به‌طور پیش‌فرض ساعت 4:00 AM به وقت محلی میزبان Gateway) در نخستین پیام پس از مرز بازنشانی، یک `sessionId` جدید ایجاد می‌کند.
- **انقضای بی‌کاری** (`session.reset.idleMinutes`، یا `session.idleMinutes` قدیمی) هنگامی که پیامی پس از بازه بی‌کاری برسد، یک `sessionId` جدید ایجاد می‌کند. اگر هر دو مورد روزانه و بی‌کاری پیکربندی شده باشند، هرکدام زودتر منقضی شود اولویت دارد.
- **ازسرگیری پس از اتصال مجدد رابط کاربری کنترل**، وقتی Gateway مقدار منطبق `sessionId` را از یک کلاینت رابط کاربری اپراتور دریافت می‌کند، نشستِ درحال‌نمایش را برای یک ارسال پس از اتصال مجدد حفظ می‌کند. این سیگنال یک‌بارمصرف است؛ ارسال‌های منقضی عادی همچنان یک `sessionId` جدید ایجاد می‌کنند.
- **رویدادهای سیستمی** (Heartbeat، بیدارسازی‌های Cron، اعلان‌های exec، امور ثبت و نگهداری Gateway) ممکن است ردیف نشست را تغییر دهند، اما هرگز تازگی بازنشانی روزانه/بی‌کاری را تمدید نمی‌کنند. جابه‌جایی بازنشانی، پیش از ساخت اعلان تازه، اعلان‌های صف‌شده رویداد سیستمی برای نشست قبلی را دور می‌اندازد.
- **سیاست انشعاب والد** هنگام ایجاد انشعاب یک رشته یا زیرعامل، از شاخه فعال OpenClaw استفاده می‌کند. اگر آن شاخه بیش‌ازحد بزرگ باشد (بیش از یک سقف داخلی ثابت که درحال‌حاضر 100K توکن است)، OpenClaw به‌جای شکست یا به‌ارث‌بردن تاریخچه‌ای غیرقابل‌استفاده، فرزند را با زمینه‌ای ایزوله آغاز می‌کند. اندازه‌سنجی خودکار و غیرقابل‌پیکربندی است؛ پیکربندی قدیمی `session.parentForkMaxTokens` توسط `openclaw doctor --fix` حذف می‌شود.
- **انشعاب‌های اپراتور**: `sessions.create { parentSessionKey, fork: true }` نشست جدیدی ایجاد می‌کند که رونوشت آن از وضعیت کنونی والد منشعب می‌شود (همان سازوکار انشعابِ ایجاد زیرعامل، شامل سقف اندازه بالا). تا زمانی که والد اجرای فعالی دارد، انشعاب رد می‌شود؛ انتخاب مدل والد را به ارث می‌برد مگر اینکه مدلی صریحاً ارائه شده باشد؛ و فرزند را با `forkedFromParent` و شمارنده‌های توکن تازه علامت‌گذاری می‌کند.

## طرح‌واره ذخیره‌گاه نشست

ذخیره‌گاه زمان اجرا، مقادیر `SessionEntry` را در SQLite مختص هر عامل نگه می‌دارد. نوع مقدار، `SessionEntry` در `src/config/sessions.ts` است. فیلدهای کلیدی (فهرست جامع نیست):

- `sessionId`: شناسه رونوشت کنونی که برای آدرس‌دهی ردیف‌های رونوشت SQLite استفاده می‌شود
- `sessionStartedAt`: برچسب زمانی آغاز `sessionId` کنونی؛ تازگی بازنشانی روزانه از این مقدار استفاده می‌کند. ردیف‌های قدیمی ممکن است آن را از سربرگ نشست JSONL استخراج کنند.
- `lastInteractionAt`: برچسب زمانی آخرین تعامل واقعی کاربر/کانال؛ تازگی بازنشانی بی‌کاری از این مقدار استفاده می‌کند تا رویدادهای Heartbeat، Cron و exec نشست‌ها را زنده نگه ندارند. ردیف‌های قدیمی فاقد این فیلد، به زمان آغاز بازیابی‌شده نشست رجوع می‌کنند.
- `updatedAt`: برچسب زمانی آخرین تغییر ردیف ذخیره‌گاه که برای فهرست‌سازی/هرس/ثبت و نگهداری استفاده می‌شود، نه به‌عنوان مرجع تازگی روزانه/بی‌کاری.
- `archivedAt`: برچسب زمانی اختیاری بایگانی. نشست‌های بایگانی‌شده با رونوشت دست‌نخورده خود در ذخیره‌گاه باقی می‌مانند و از فهرست‌های فعال عادی حذف می‌شوند.
- `pinnedAt`: برچسب زمانی اختیاری سنجاق‌کردن. نشست‌های فعالِ سنجاق‌شده پیش از نشست‌های سنجاق‌نشده مرتب می‌شوند؛ بایگانی‌کردن نشست، سنجاق آن را پاک می‌کند.
- تعامل‌پذیری رشته Codex: هر دو فیلد از ساختار مدیریت رشته Codex پیروی می‌کنند؛ مقادیر بولی `archived`/`pinned` روی ارتباط همواره از برچسب زمانی استخراج و سمت سرور ثبت می‌شوند، مطابق معنای `threads.archived_at` در Codex و سریال‌سازی camelCase. برچسب‌های زمانی OpenClaw برحسب میلی‌ثانیه یونیکس هستند، درحالی‌که Codex از ثانیه یونیکس استفاده می‌کند؛ بنابراین پل‌ها تبدیل را در مرز Pluginِ `codex` انجام می‌دهند. Codex هنوز API سنجاق‌کردن ندارد (فقط `thread/archive`/`thread/unarchive`)؛ تا زمان ارائه آن، وضعیت سنجاق‌شده در سمت OpenClaw باقی می‌ماند و پس از ارائه، ساختار منطبق به نشست‌های مقید اجازه می‌دهد وضعیت سنجاق را به‌طور مکانیکی رفت‌وبرگشت دهند.
- نظارت Codex فقط رشته‌های بومیِ بایگانی‌نشده را فهرست می‌کند. یک رشته محلی Gateway با فعالیت نامشخصِ `idle` یا `notLoaded` تنها زمانی می‌تواند از طریق `thread/archive` بومی بایگانی شود که اپراتور صریحاً تأیید کند هیچ فرایند Codex دیگری مالک آن نیست؛ Plugin ابتدا وضعیت محلی فرایند را از نو می‌خواند و سپس رشته از کاتالوگ ناپدید می‌شود. این خواندن نمی‌تواند ثابت کند که فرایند App Server دیگری از رشته استفاده نمی‌کند. OpenClaw از بایگانی ردیف‌های فعال و خطادار خودداری می‌کند و بایگانی Node جفت‌شده تا زمانی که پل Node بتواند مالک چرخه عمر کامل رشته جاری باشد، در دسترس نیست. خارج‌کردن از بایگانی در یک کلاینت بومی Codex، رشته را دوباره واجد شرایط نمایش می‌کند.
- `lastReadAt` / `markedUnreadAt`: برچسب‌های زمانی وضعیت خواندن که سمت سرور توسط `sessions.patch { unread }` ثبت می‌شوند؛ `unread: false` یک خواندن را ثبت می‌کند (`lastReadAt` را تنظیم و `markedUnreadAt` را پاک می‌کند)؛ `unread: true` نشست را تا خواندن بعدی خوانده‌نشده علامت می‌زند. ردیف‌های نشست یک مقدار بولی استخراج‌شده `unread` ارائه می‌کنند: یا صریحاً خوانده‌نشده علامت خورده‌اند، یا پیش از آخرین فعالیت خوانده شده‌اند. نشست‌هایی که هرگز خوانده‌شده علامت نخورده‌اند، `unread: false` باقی می‌مانند تا نصب‌های موجود پس از ارتقا روشن نشوند.
- `lastActivityAt`: برچسب زمانی آخرین اجرای تکمیل‌شده عامل که فعالیتی شایسته خوانده‌نشده‌بودن محسوب می‌شود (اجراهای کاربر، کانال و Cron). نوبت‌های Heartbeat و رویداد داخلی و نیز وصله‌های فراداده، آن را به‌روزرسانی نمی‌کنند؛ `updatedAt` سیگنال فعالیت نیست.
- `sessionFile`: نشانگر قدیمی که برای سازگاری مهاجرت/بایگانی حفظ شده است؛ زمان اجرای فعال از هویت SQLite استفاده می‌کند
- `chatType`: `direct | group | room`
- `provider`، `subject`، `room`، `space`، `displayName`: فراداده برچسب‌گذاری گروه/کانال
- کلیدهای تغییر وضعیت: `thinkingLevel`، `verboseLevel`، `reasoningLevel`، `elevatedLevel`، `sendPolicy` (بازنویسی مختص هر نشست)
- انتخاب مدل: `providerOverride`، `modelOverride`، `authProfileOverride`
- شمارنده‌های توکن (با بیشترین تلاش/وابسته به ارائه‌دهنده): `inputTokens`، `outputTokens`، `totalTokens`، `contextTokens`
- `compactionCount`: تعداد دفعات تکمیل فشرده‌سازی خودکار برای این کلید نشست
- `memoryFlushAt` / `memoryFlushCompactionCount`: برچسب زمانی و تعداد فشرده‌سازی آخرین تخلیه حافظه پیش از فشرده‌سازی

Gateway مرجع نهایی است: ممکن است هم‌زمان با اجرای نشست‌ها، ورودی‌ها را بازنویسی یا دوباره بارگذاری کند. برای نصب‌های قدیمی مبتنی بر فایل، به‌جای
ویرایش `sessions.json` و انتظار از زمان اجرا برای ادامه خواندن آن فایل، با
`openclaw doctor --session-sqlite import --session-sqlite-all-agents` مهاجرت کنید.

## ساختار رویداد رونوشت

رونوشت‌ها توسط دسترسی‌دهنده نشست OpenClaw مدیریت می‌شوند و از طریق یاریگرهای مبتنی بر هویت در اختیار کد زمان اجرا قرار می‌گیرند. جریان رویداد فقط الحاقی است:

- اولین ورودی: سربرگ نشست ــ `type: "session"`، `id`، `cwd`، `timestamp`، و `parentSession` اختیاری.
- سپس: ورودی‌هایی با `id` + `parentId` (ساختار درختی).

انواع ورودی قابل‌توجه:

- `message`: پیام‌های user/assistant/toolResult
- `custom_message`: پیام تزریق‌شده توسط افزونه که _وارد_ زمینه مدل می‌شود (وقتی `display: true` باشد در TUI رندر می‌شود و وقتی `display: false` باشد کاملاً پنهان است)
- `custom`: وضعیت افزونه که _وارد_ زمینه مدل نمی‌شود (برای ماندگارکردن وضعیت افزونه میان بارگذاری‌های مجدد)
- `compaction`: خلاصه فشرده‌سازی ماندگارشده با `firstKeptEntryId` و `tokensBefore`
- `branch_summary`: خلاصه ماندگارشده هنگام پیمایش یک شاخه درخت

OpenClaw عمداً رونوشت‌ها را «اصلاح» نمی‌کند؛ Gateway برای خواندن/نوشتن آن‌ها از `SessionManager` استفاده می‌کند.

## پنجره‌های زمینه در برابر توکن‌های رهگیری‌شده

دو مفهوم متفاوت:

1. **پنجره زمینه مدل**: سقف سخت برای هر مدل (توکن‌های قابل‌مشاهده برای مدل). از کاتالوگ مدل می‌آید و از طریق پیکربندی قابل بازنویسی است.
2. **شمارنده‌های ذخیره‌گاه نشست**: آمارهای چرخانی که در ردیف نشست نوشته می‌شوند (برای `/status` و داشبوردها استفاده می‌شوند). `contextTokens` یک مقدار تخمینی/گزارشی زمان اجرا است؛ آن را تضمینی قطعی تلقی نکنید.

اطلاعات بیشتر درباره محدودیت‌ها: [/reference/token-use](/fa/reference/token-use).

## Compaction: چیست

Compaction مکالمه قدیمی‌تر را در یک ورودی ماندگار `compaction` در رونوشت خلاصه می‌کند و پیام‌های اخیر را دست‌نخورده نگه می‌دارد. پس از Compaction، نوبت‌های آینده خلاصه Compaction را به‌همراه پیام‌های پس از `firstKeptEntryId` می‌بینند. Compaction برخلاف هرس نشست، **ماندگار** است؛ به [/concepts/session-pruning](/fa/concepts/session-pruning) مراجعه کنید.

تزریق مجدد بخش AGENTS.md پس از Compaction از طریق `agents.defaults.compaction.postCompactionSections` اختیاری است؛ وقتی تنظیم نشده یا `[]` باشد، OpenClaw گزیده‌های AGENTS.md را روی خلاصه Compaction الحاق نمی‌کند.

### مرزهای قطعه و جفت‌سازی ابزار

هنگام تقسیم یک رونوشت طولانی به قطعه‌های Compaction، OpenClaw فراخوانی‌های ابزار دستیار را با ورودی‌های منطبق `toolResult` جفت نگه می‌دارد:

- اگر تقسیم سهم توکن میان فراخوانی ابزار و نتیجه آن قرار گیرد، OpenClaw به‌جای جداکردن جفت، مرز را به پیام فراخوانی ابزار دستیار منتقل می‌کند.
- اگر یک بلوک نتیجه ابزار انتهایی در حالت عادی قطعه را از هدف فراتر ببرد، OpenClaw آن بلوک ابزار در انتظار را حفظ می‌کند و دنباله خلاصه‌نشده را دست‌نخورده نگه می‌دارد.
- بلوک‌های فراخوانی ابزار لغوشده/خطادار، تقسیم در انتظار را باز نگه نمی‌دارند.

## زمان وقوع Compaction خودکار

دو محرک در عامل تعبیه‌شده OpenClaw:

1. **بازیابی سرریز**: مدل یک خطای سرریز زمینه بازمی‌گرداند (`request_too_large`، `context length exceeded`، `input exceeds the maximum number of tokens`، `input token count exceeds the maximum number of input tokens`، `input is too long for the model`، `ollama error: context length exceeded` و گونه‌های دیگر متناسب با ارائه‌دهنده)؛ فشرده‌سازی کنید و سپس دوباره تلاش کنید. وقتی ارائه‌دهنده تعداد توکن تلاش‌شده را گزارش می‌کند، OpenClaw آن تعداد مشاهده‌شده را به Compaction بازیابی سرریز منتقل می‌کند؛ اگر ارائه‌دهنده سرریز را تأیید کند اما هیچ تعداد قابل‌تجزیه‌ای ارائه ندهد، OpenClaw یک تعداد مصنوعی با حداقل تجاوز از بودجه به موتورهای Compaction و عیب‌یابی منتقل می‌کند. اگر بازیابی سرریز همچنان ناموفق باشد، OpenClaw راهنمایی صریح نمایش می‌دهد و به‌جای چرخش بی‌سروصدا به یک شناسه نشست تازه، نگاشت نشست کنونی را حفظ می‌کند؛ پیام را دوباره امتحان کنید، `/compact` را اجرا کنید، یا `/new` را اجرا کنید.
2. **نگهداری آستانه**: پس از یک نوبت موفق، وقتی `contextTokens > contextWindow - reserveTokens`؛ که در آن `contextWindow` پنجره زمینه مدل و `reserveTokens` فضای ذخیره‌شده برای اعلان‌ها به‌علاوه خروجی بعدی مدل است.

دو محافظ اضافی خارج از این دو محرک اجرا می‌شوند:

- **Compaction محلی پیش‌پرواز**: `agents.defaults.compaction.maxActiveTranscriptBytes` را (به‌شکل بایت یا رشته‌ای مانند `"20mb"`) تنظیم کنید تا وقتی رونوشت فعال به آن اندازه رسید، پیش از بازکردن اجرای بعدی Compaction محلی فعال شود. این یک محافظ اندازه برای هزینه بازگشایی محلی است، نه بایگانی خام؛ Compaction معنایی عادی همچنان اجرا می‌شود و به `truncateAfterCompaction` نیاز دارد تا خلاصه فشرده‌شده به یک رونوشت جانشین جدید تبدیل شود.
- **پیش‌بررسی میان‌نوبت**: `agents.defaults.compaction.midTurnPrecheck.enabled: true` را (پیش‌فرض `false`) تنظیم کنید تا یک محافظ حلقه ابزار افزوده شود. پس از الحاق نتیجه ابزار و پیش از فراخوانی بعدی مدل، OpenClaw فشار اعلان را با همان منطق بودجه پیش‌پروازِ آغاز نوبت تخمین می‌زند. اگر زمینه دیگر جا نشود، محافظ Compaction را به‌صورت درون‌خطی انجام نمی‌دهد؛ یک سیگنال ساختاریافته پیش‌بررسی میان‌نوبت ایجاد می‌کند، ارسال اعلان کنونی را متوقف می‌سازد و به حلقه اجرای بیرونی اجازه می‌دهد از مسیر بازیابی موجود استفاده کند (اگر کافی باشد، نتایج بیش‌ازحد بزرگ ابزار را کوتاه کند، یا حالت Compaction پیکربندی‌شده را فعال و دوباره تلاش کند). با هر دو حالت Compaction یعنی `default` و `safeguard`، ازجمله Compaction محافظتی مبتنی بر ارائه‌دهنده، کار می‌کند. مستقل از `maxActiveTranscriptBytes` است: محافظ اندازه بایتی پیش از بازشدن نوبت اجرا می‌شود و پیش‌بررسی میان‌نوبت بعدتر، پس از الحاق نتایج جدید ابزار، اجرا می‌شود.

## تنظیمات Compaction

```json5
{
  agents: {
    defaults: {
      compaction: {
        enabled: true,
        reserveTokens: 16384,
        keepRecentTokens: 20000,
      },
    },
  },
}
```

OpenClaw همچنین یک حداقل ایمنی برای اجراهای تعبیه‌شده اعمال می‌کند: اگر `compaction.reserveTokens` کمتر از `reserveTokensFloor` (مقدار پیش‌فرض `20000`) باشد، OpenClaw آن را افزایش می‌دهد. برای غیرفعال‌کردن این حداقل، `agents.defaults.compaction.reserveTokensFloor: 0` را تنظیم کنید. وقتی پنجرهٔ زمینهٔ مدل فعال مشخص باشد، هم حداقل و هم ذخیرهٔ مؤثر نهایی سقف‌گذاری می‌شوند تا ذخیره نتواند کل بودجهٔ پرامپت را مصرف کند. این کار مانع می‌شود مدل‌های دارای زمینهٔ کوچک (برای مثال، یک مدل محلی با 16K توکن) از همان نخستین توکن وارد Compaction شوند؛ بدون پنجرهٔ زمینهٔ مشخص، بودجه‌های ذخیرهٔ پیکربندی‌شده و فعلی بدون سقف باقی می‌مانند. دلیل وجود این حداقل: باقی‌گذاشتن فضای کافی برای «امور نگه‌داری» چندمرحله‌ای (مانند تخلیهٔ حافظه در ادامه) پیش از اجتناب‌ناپذیرشدن Compaction. پیاده‌سازی: `applyAgentCompactionSettingsFromConfig()` در `src/agents/agent-settings.ts`، که از مسیرهای نوبت اجراکنندهٔ تعبیه‌شده و راه‌اندازی Compaction فراخوانی می‌شود.

`/compact` دستی، `agents.defaults.compaction.keepRecentTokens` صریح را رعایت می‌کند و نقطهٔ برش دنبالهٔ اخیر زمان اجرا را نگه می‌دارد. بدون بودجهٔ نگه‌داری صریح، Compaction دستی یک نقطهٔ بازرسی قطعی است و زمینهٔ بازسازی‌شده از خلاصهٔ جدید آغاز می‌شود.

وقتی `truncateAfterCompaction` فعال باشد، OpenClaw پس از Compaction، رونوشت فعال را به جانشین فشرده‌شدهٔ آن منتقل می‌کند. کنش‌های نقطهٔ بازرسی شاخه/بازیابی از آن جانشین فشرده‌شده استفاده می‌کنند؛ فایل‌های قدیمی نقطهٔ بازرسی پیش از Compaction تا زمانی که به آن‌ها ارجاع داده شود، خواندنی باقی می‌مانند.

## ارائه‌دهندگان قابل‌اتصال Compaction

Pluginها از طریق `registerCompactionProvider()` در API مربوط به Plugin، یک ارائه‌دهندهٔ Compaction ثبت می‌کنند. وقتی `agents.defaults.compaction.provider` روی شناسهٔ یک ارائه‌دهندهٔ ثبت‌شده تنظیم شود، افزونهٔ محافظ، خلاصه‌سازی را به‌جای خط لولهٔ داخلی `summarizeInStages` به آن ارائه‌دهنده واگذار می‌کند.

- `provider`: شناسهٔ یک Plugin ثبت‌شدهٔ ارائه‌دهندهٔ Compaction. برای خلاصه‌سازی پیش‌فرض LLM تنظیم‌نشده باقی بماند. تنظیم یک `provider`، استفاده از `mode: "safeguard"` را اجباری می‌کند.
- ارائه‌دهندگان همان دستورالعمل‌های Compaction و سیاست حفظ شناسه را که مسیر داخلی دریافت می‌کند، دریافت می‌کنند و محافظ همچنان زمینهٔ پسوند نوبت‌های اخیر و نوبت‌های تقسیم‌شده را پس از خروجی ارائه‌دهنده حفظ می‌کند.
- خلاصه‌سازی داخلی محافظ، به‌جای حفظ واژه‌به‌واژهٔ کل خلاصهٔ قبلی، خلاصه‌های پیشین را همراه با پیام‌های جدید دوباره چکیده می‌کند.
- حالت محافظ، ممیزی کیفیت خلاصه را به‌طور پیش‌فرض فعال می‌کند؛ برای ردکردن رفتار تلاش مجدد در صورت خروجی بدشکل، `qualityGuard.enabled: false` را تنظیم کنید.
- اگر ارائه‌دهنده شکست بخورد یا نتیجه‌ای خالی برگرداند، OpenClaw به‌طور خودکار به خلاصه‌سازی داخلی LLM بازمی‌گردد. سیگنال‌های لغو/مهلت زمانی که فراخواننده صراحتاً فعال کرده است، دوباره پرتاب می‌شوند و نادیده گرفته نمی‌شوند؛ بنابراین لغو همیشه رعایت می‌شود.

منبع: `src/plugins/compaction-provider.ts`، `src/agents/agent-hooks/compaction-safeguard.ts`.

## سطوح قابل‌مشاهده برای کاربر

- `/status` در هر نشست گفت‌وگو
- `openclaw status` (CLI)
- `openclaw sessions` / `openclaw sessions --json`
- گزارش‌های Gateway ‏(`pnpm gateway:watch` یا `openclaw logs --follow`):‏ `embedded run auto-compaction start` + `complete`
- حالت مشروح: `🧹 Auto-compaction complete` به‌همراه شمار Compaction

## امور نگه‌داری بی‌صدا (`NO_REPLY`)

OpenClaw برای وظایف پس‌زمینه‌ای که کاربر نباید خروجی میانی آن‌ها را ببیند، از نوبت‌های «بی‌صدا» پشتیبانی می‌کند.

- دستیار خروجی خود را با توکن بی‌صدای دقیق `NO_REPLY` / `no_reply` آغاز می‌کند تا به‌معنای «پاسخی به کاربر تحویل نده» باشد. OpenClaw آن را در لایهٔ تحویل حذف/سرکوب می‌کند.
- سرکوب توکن بی‌صدای دقیق به بزرگی و کوچکی حروف حساس نیست: وقتی کل بار داده فقط همان توکن بی‌صدا باشد، هم `NO_REPLY` و هم `no_reply` معتبرند.
- از `2026.1.10`، اگر یک قطعهٔ جزئی با `NO_REPLY` آغاز شود، OpenClaw پخش جریانی پیش‌نویس/نشانگر تایپ را نیز سرکوب می‌کند تا عملیات بی‌صدا در میانهٔ نوبت خروجی جزئی افشا نکنند.
- این فقط برای نوبت‌های واقعاً پس‌زمینه‌ای/بدون تحویل است و میان‌بری برای درخواست‌های عادی و اقدام‌پذیر کاربر نیست.

## تخلیهٔ حافظه پیش از Compaction

پیش از وقوع Compaction خودکار، OpenClaw می‌تواند یک نوبت عامل‌محور بی‌صدا اجرا کند که وضعیت ماندگار را روی دیسک می‌نویسد (برای مثال، `memory/YYYY-MM-DD.md` در فضای کاری عامل) تا Compaction نتواند زمینهٔ حیاتی را پاک کند. این قابلیت میزان استفاده از زمینهٔ نشست را پایش می‌کند و پس از عبور آن از آستانه‌ای نرم و پایین‌تر از آستانهٔ Compaction، با استفاده از توکن بی‌صدای دقیق `NO_REPLY` / `no_reply` دستور بی‌صدای «اکنون حافظه را بنویس» را ارسال می‌کند تا کاربر چیزی نبیند.

پیکربندی (`agents.defaults.compaction.memoryFlush`)، مرجع کامل در [/gateway/config-agents](/fa/gateway/config-agents#agentsdefaultscompaction):

| کلید                         | پیش‌فرض          | نکات                                                                                                                                  |
| --------------------------- | ---------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`                   | `true`           |                                                                                                                                        |
| `model`                     | تنظیم‌نشده            | بازنویسی دقیق ارائه‌دهنده/مدل فقط برای نوبت تخلیه، برای مثال `ollama/qwen3:8b`                                                   |
| `softThresholdTokens`       | `4000`           | فاصلهٔ زیر آستانهٔ Compaction که تخلیه را فعال می‌کند                                                                               |
| `forceFlushTranscriptBytes` | تنظیم‌نشده (غیرفعال) | پس از رسیدن فایل رونوشت به این اندازه برحسب بایت (یا رشته‌ای مانند `"2mb"`) تخلیه را اجباری می‌کند، حتی اگر شمارنده‌های توکن قدیمی باشند؛ `0` غیرفعال می‌کند |
| `prompt`                    | داخلی         | پیام کاربر برای نوبت تخلیه                                                                                                        |
| `systemPrompt`              | داخلی         | پرامپت سیستمی اضافی که به نوبت تخلیه افزوده می‌شود                                                                                        |

نکات:

- پرامپت/پرامپت سیستمی پیش‌فرض شامل راهنمای `NO_REPLY` برای سرکوب تحویل است.
- وقتی `model` تنظیم شده باشد، نوبت تخلیه از همان مدل استفاده می‌کند، بدون آنکه زنجیرهٔ جایگزین نشست فعال را به ارث ببرد؛ بنابراین امور نگه‌داری صرفاً محلی در صورت شکست، بی‌سروصدا به یک مدل گفت‌وگوی پولی بازنمی‌گردد.
- تخلیه در هر چرخهٔ Compaction یک‌بار اجرا می‌شود (در ردیف نشست ردیابی می‌شود).
- تخلیه فقط برای نشست‌های تعبیه‌شدهٔ OpenClaw اجرا می‌شود؛ پشتیبان‌های CLI و نوبت‌های Heartbeat از آن صرف‌نظر می‌کنند.
- وقتی فضای کاری نشست فقط‌خواندنی باشد (`workspaceAccess: "ro"` یا `"none"`) از تخلیه صرف‌نظر می‌شود.
- برای چیدمان فایل‌های فضای کاری و الگوهای نوشتن، به [حافظه](/fa/concepts/memory) مراجعه کنید.

OpenClaw یک قلاب `session_before_compact` در API افزونه ارائه می‌کند، اما منطق تخلیهٔ بالا در سمت Gateway ‏(`src/auto-reply/reply/memory-flush.ts`، `src/auto-reply/reply/agent-runner-memory.ts`) قرار دارد، نه در آن قلاب.

## فهرست بررسی عیب‌یابی

- **کلید نشست اشتباه است؟** از [/concepts/session](/fa/concepts/session) آغاز کنید و `sessionKey` را در `/status` تأیید کنید.
- **مخزن و رونوشت ناسازگارند؟** میزبان Gateway و مسیر مخزن را از `openclaw status` تأیید کنید.
- **Compaction بیش‌ازحد تکرار می‌شود؟** پنجرهٔ زمینهٔ مدل (کوچک‌بودن بیش‌ازحد، Compaction مکرر را تحمیل می‌کند)، `reserveTokens` (زیادبودن آن برای پنجرهٔ مدل باعث Compaction زودتر می‌شود) و حجیم‌شدن نتایج ابزارها (هرس نشست را تنظیم کنید) بررسی کنید.
- **به نظر می‌رسد هر پرامپت در یک مدل محلی کوچک سرریز می‌کند؟** تأیید کنید که ارائه‌دهنده پنجرهٔ زمینهٔ صحیح مدل را گزارش می‌کند. OpenClaw فقط وقتی این پنجره مشخص باشد می‌تواند ذخیرهٔ مؤثر را سقف‌گذاری کند.
- **نوبت‌های بی‌صدا نشت می‌کنند؟** تأیید کنید پاسخ با توکن بی‌صدای دقیق `NO_REPLY` (بدون حساسیت به بزرگی و کوچکی حروف) آغاز می‌شود و از ساختی استفاده می‌کنید که اصلاح سرکوب پخش جریانی (`2026.1.10`+) را دربر دارد.
