# انتشار

انتشار در ClawHub در محدودهٔ مالک انجام می‌شود: هر انتشار یک ناشر را هدف می‌گیرد، و
سرور تصمیم می‌گیرد که آیا کاربر واردشده مجاز است در آنجا منتشر کند یا نه.

## مالکان

مالک یک شناسهٔ ناشر در ClawHub است، مانند `@alice` یا `@openclaw`.
مالکان شخصی برای کاربران ایجاد می‌شوند. مالکان سازمانی می‌توانند چند عضو داشته باشند.

هنگام انتشار، یا از مالک شخصی خود استفاده می‌کنید یا یک مالک سازمانی را انتخاب می‌کنید
که در آن دسترسی ناشر دارید.

## Skills

Skills از یک پوشهٔ مهارت منتشر می‌شوند. صفحهٔ عمومی این است:

```text
https://clawhub.ai/<owner>/<slug>
```

مثال:

```text
https://clawhub.ai/alice/review-helper
```

درخواست انتشار شامل مالک انتخاب‌شده، slug، نسخه، تغییرات نسخه و
فایل‌ها است. سرور پیش از ایجاد انتشار، بررسی می‌کند که عامل بتواند به‌عنوان آن مالک منتشر کند.

برای انتقال یک مهارت موجود به مالک دیگر هنگام انتشار نسخهٔ جدید، مالک
جدید را انتخاب کنید و انتقال مالکیت را صریحاً تأیید کنید. در CLI/API، مالک
هدف را همراه با گزینهٔ موافقت با مهاجرت ارسال کنید:

```sh
clawhub skill publish ./review-helper --owner openclaw --migrate-owner --version 1.2.0
```

مهاجرت مالک مهارت به دسترسی مدیر یا مالک در هر دو مالک فعلی
و مالک مقصد نیاز دارد. این کار مهارت، تاریخچهٔ نسخه‌ها، آمار،
نظرها، forkها، aliasها و ردپای ممیزی را حفظ می‌کند؛ URLهای مالک قبلی از طریق
مسیر alias/redirect همچنان ادامه می‌یابند.

## Pluginها

Pluginها از نام‌های بسته به سبک npm استفاده می‌کنند. نام‌های بستهٔ scoped مالک را در
بخش اول نام شامل می‌شوند:

```text
@owner/package-name
```

scope باید با مالک انتشار انتخاب‌شده مطابقت داشته باشد. اگر نام بستهٔ شما
`@openclaw/dronzer` باشد، فقط می‌تواند به‌عنوان `@openclaw` منتشر شود. اگر به‌عنوان
`@vintageayu` منتشر می‌کنید، نام بسته را به `@vintageayu/dronzer` تغییر دهید.

این کار مانع می‌شود یک بسته namespace سازمانی‌ای را ادعا کند که ناشر
کنترلی روی آن ندارد.

## جریان انتشار

1. UI، CLI یا گردش‌کار GitHub فرادادهٔ بسته و فایل‌ها را گردآوری می‌کند.
2. درخواست انتشار با مالک انتخاب‌شده به ClawHub ارسال می‌شود.
3. سرور مجوزهای مالک، scope بسته، نام بسته، نسخه،
   محدودیت‌های فایل و فرادادهٔ منبع را اعتبارسنجی می‌کند.
4. ClawHub انتشار را ذخیره می‌کند و بررسی‌های امنیتی خودکار را آغاز می‌کند.
5. انتشارهای جدید تا زمانی که بازبینی
   و راستی‌آزمایی تمام شود، از سطوح نصب/دانلود عادی پنهان می‌مانند.

اگر اعتبارسنجی ناموفق باشد، انتشار ایجاد نمی‌شود.

## پرسش‌های متداول

### scope بسته باید با مالک انتخاب‌شده مطابقت داشته باشد

اگر scope بسته و مالک انتخاب‌شده مطابقت نداشته باشند، ClawHub
انتشار را رد می‌کند:

```text
Package scope "@openclaw" must match selected owner "@vintageayu".
Publish as "@openclaw" or rename this package to "@vintageayu/dronzer".
```

برای رفع آن، یا مالکی را انتخاب کنید که توسط scope بسته نام‌گذاری شده است، یا نام
بسته را تغییر دهید تا scope با مالکی که می‌توانید به‌عنوان آن منتشر کنید مطابقت داشته باشد.

اگر نام بسته از قبل scope درست را دارد اما بسته متعلق به ناشر
اشتباه است، به‌جای آن مالکیت را منتقل کنید:

```sh
clawhub package transfer @opik/opik-openclaw --to opik
```

فقط زمانی از انتقال بسته استفاده کنید که به هر دو مالک فعلی بسته
و ناشر مقصد دسترسی مدیر دارید. این امکان را نمی‌دهد که در scopeای منتشر کنید که
نمی‌توانید مدیریت کنید.

این از namespaceهای سازمانی محافظت می‌کند. بسته‌ای با نام `@openclaw/dronzer`
namespace‏ `@openclaw` را ادعا می‌کند، بنابراین فقط ناشرانی که به مالک `@openclaw`
دسترسی دارند می‌توانند آن را منتشر کنند.
