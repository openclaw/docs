# ادعاهای سازمان و فضای نام

ClawHub از هندل‌های مالک، هندل‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته‌ی Plugin و
دامنه‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر یک فضای نام به نظر برسد متعلق به یک
پروژه، برند، اکوسیستم بسته یا سازمان واقعی است اما در ClawHub از قبل
ادعا شده، رزرو شده، گمراه‌کننده یا مورد اختلاف است، از کارکنان بخواهید آن را
با
[فرم مسئله‌ی ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی مالکیت عمومی و غیرحساس استفاده کنید. برای ادعاهای فضای نام
از گزارش‌های درون محصول یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به‌دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، با نام مستعار همراه
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد هستند:

- هندل سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه‌ی شما مطابقت دارد
- دامنه‌ی بسته‌ای مانند `@example-org/*` که فقط باید تحت مالک
  متناظر ClawHub منتشر شود
- اسلاگ مهارت یا نام بسته‌ی Plugin که به نظر می‌رسد از یک پروژه جعل هویت می‌کند
- اختلاف بر سر برند، علامت تجاری، تغییرنام پروژه یا تاریخچه‌ی بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که فضای نام مالک
  برحق را مسدود می‌کند

اگر فهرست، فراتر از اختلاف مالکیت، ناامن، مخرب یا گمراه‌کننده است،
راهنمای مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دامنه‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی یا حذف منبع تحت تأثیر اصلاح کنید. وقتی ادعا ثبت کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچه‌ی سازمان GitHub، مخزن، انتشار یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- مدرک دامنه یا دامنه‌ی ایمیل رسمی
- کنترل دامنه در npm، PyPI، crates.io یا دیگر رجیستری‌های بسته
- شواهد مالکیت علامت تجاری، برند یا پروژه که بحث درباره‌ی آن
  به‌صورت عمومی امن است
- تاریخچه‌ی مخزن منبع، تاریخچه‌ی بسته یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته یا مسئله‌ی مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئله‌ی عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویت شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه‌ی مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مواد حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، مدرک بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبق منتقل شود.
کارکنان شواهد عمومی، استفاده‌ی موجود، ریسک امنیتی و تأثیر بر کاربر را می‌سنجند.
