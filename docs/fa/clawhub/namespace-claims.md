# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، نامک‌های مهارت، نام بسته‌های Plugin و
دامنه‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر برسد یک فضای نام به یک
پروژه، برند، زیست‌بوم بسته، یا سازمان واقعی تعلق دارد، اما در ClawHub از قبل
ادعا شده، رزرو شده، گمراه‌کننده است، یا مورد اختلاف قرار دارد، از تیم بخواهید آن را
با
[فرم issue ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های داخل محصول
یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی ادعا باز کنید

وقتی باور دارید تیم ClawHub باید بررسی کند که آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، مستعار،
یا به‌شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- دامنه بسته‌ای مانند `@example-org/*` که فقط باید تحت
  مالک ClawHub مطابق منتشر شود
- نامک مهارت یا نام بسته Plugin که به نظر می‌رسد یک پروژه را جعل هویت می‌کند
- اختلاف بر سر برند، علامت تجاری، تغییرنام پروژه، یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال، یا دسترس‌ناپذیری که مالک برحق فضای نام را
  مسدود می‌کند

اگر مورد فهرست‌شده فراتر از اختلاف مالکیت ناامن، مخرب، یا گمراه‌کننده است،
راهنمای مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دامنه‌داری مانند `@example-org/example-plugin` باید
به‌عنوان مالک مطابق `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا
استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا وقتی تیم باید یک
اختلاف را حل کند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. اثبات‌های مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که نام فضای نام را ذکر می‌کنند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل دامنه در npm، PyPI، crates.io، یا سایر رجیستری‌های بسته
- شواهد علامت تجاری، برند، یا مالکیت پروژه که بحث عمومی درباره آن امن است
- تاریخچه مخزن منبع، تاریخچه بسته، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا issue مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را اثبات می‌کند. تیم باید بتواند
رابطه را بدون نیاز به اطلاعات ورود خصوصی یا اسرار درک کند.

## چه چیزهایی را نباید اضافه کنید

اسرار یا اثبات خصوصی را در یک issue عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اطلاعات ورود
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی تیم نیاز دارد یا نه.
به‌جای انتشار عمومی مواد حساس، از همان گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، تیم ClawHub ممکن است یک فضای نام را رزرو کند،
مالکیت را منتقل کند، نام یک منبع را تغییر دهد، یک مورد فهرست‌شده موجود را پنهان یا قرنطینه کند،
یک نام مستعار یا تغییرمسیر اضافه کند، شواهد بیشتری بخواهد، یا درخواست را رد کند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابق منتقل شود.
تیم شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربر را ارزیابی می‌کند.
