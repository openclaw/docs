# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin، و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته، یا سازمان واقعی تعلق دارد، اما در ClawHub از قبل
ادعا شده، رزرو شده، گمراه‌کننده است، یا مورد اختلاف قرار دارد، از کارکنان بخواهید آن را با
[فرم issue ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml) بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های
داخل محصول یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

زمانی ادعای فضای نام باز کنید که باور دارید کارکنان ClawHub باید بررسی کنند آیا یک
فضای نام باید به دلیل مالکیت واقعی رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر داده شود.

نمونه‌ها شامل این موارد هستند:

- شناسه سازمانی که با سازمان، پروژه، شرکت، یا جامعه GitHub شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک ClawHub
  متناظر منتشر شود
- اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد وانمود می‌کند یک پروژه است
- اختلاف درباره برند، علامت تجاری، تغییرنام پروژه، یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال، یا دسترس‌ناپذیری که مانع مالک برحق فضای نام می‌شود

اگر فهرست، فراتر از اختلاف مالکیت، ناامن، مخرب، یا گمراه‌کننده است،
راهنمای مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع آسیب‌دیده اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا وقتی کارکنان باید اختلافی را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد هستند:

- تاریخچه سازمان، مخزن، انتشار، یا نگه‌دارنده GitHub
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا رجیستری‌های بسته دیگر
- شواهد مالکیت علامت تجاری، برند، یا پروژه که طرح عمومی آن امن است
- تاریخچه مخزن منبع، تاریخچه بسته، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا issue مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را اثبات می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌های خصوصی یا اسرار درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک issue عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویت شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتیجه‌های ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، منبعی را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، مدارک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و تأثیر بر کاربر را می‌سنجند.
