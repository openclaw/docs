# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های package مربوط به Plugin، و
scopeهای package به‌عنوان فضای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه واقعی، برند، اکوسیستم package، یا سازمان تعلق دارد اما در ClawHub قبلا
ادعا، رزرو، گمراه‌کننده، یا مورد اختلاف واقع شده است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بازبینی کنند.

از این مسیر برای بازبینی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام
از گزارش‌های درون محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، مستعارسازی،
یا به هر شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل موارد زیر است:

- یک شناسه سازمان که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- یک scope package مانند `@example-org/*` که فقط باید تحت مالک
  ClawHub مطابق منتشر شود
- یک اسلاگ مهارت یا نام package مربوط به Plugin که به نظر می‌رسد از یک پروژه جعل هویت می‌کند
- اختلاف درباره برند، نشان تجاری، تغییرنام پروژه، یا تاریخچه package
- مالکی حذف‌شده، غیرفعال، یا دسترس‌ناپذیر که مالک برحق فضای نام را مسدود می‌کند

اگر فهرست ناامن، مخرب، یا فراتر از اختلاف مالکیت گمراه‌کننده است،
راهنمایی‌های مرتبط با تعدیل محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بازبینی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای packageهای Plugin، نام‌های scoped مانند `@example-org/example-plugin` باید
به‌عنوان مالک مطابق `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیما با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا
استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا زمانی که کارکنان باید
یک اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. شواهد مفید شامل موارد زیر است:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- مدرک دامنه یا دامنه ایمیل رسمی
- کنترل scope در npm، PyPI، crates.io، یا سایر رجیستری‌های package
- شواهد مالکیت نشان تجاری، برند، یا پروژه که بحث عمومی درباره آن امن است
- تاریخچه مخزن منبع، تاریخچه package، یا اعلان‌های عمومی تغییرنام
- پیوندهایی به مالک، مهارت، Plugin، package، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند بدون نیاز
به اعتبارنامه‌های خصوصی یا اسرار، رابطه را درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. موارد زیر را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## پیامدهای ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک alias یا تغییرمسیر اضافه کنند، شواهد بیشتری بخواهند، یا درخواست را رد کنند.

بازبینی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربر را می‌سنجند.
