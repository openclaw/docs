# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، نامک‌های مهارت، نام‌های بسته Plugin و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه واقعی، برند، زیست‌بوم بسته، یا سازمان تعلق دارد اما در ClawHub از قبل
ادعا، رزرو، گمراه‌کننده، یا محل اختلاف شده است، از کارکنان بخواهید آن را با
[فرم issue ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی مالکیت عمومی و غیرحساس استفاده کنید. برای ادعاهای فضای نام از گزارش‌های درون محصول
یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک فضای نام به دلیل مالکیت واقعی
باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر کند، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک ClawHub
  مطابق منتشر شود
- نامک مهارت یا نام بسته Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییرنام پروژه، یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال، یا دسترس‌ناپذیری که فضای نام مالک
  برحق را مسدود کرده است

اگر مدخل، فراتر از اختلاف مالکیت، ناامن، مخرب، یا گمراه‌کننده است،
راهنمایی مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای فوری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دارای محدوده مانند `@example-org/example-plugin` باید
با مالک مطابق `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا وقتی کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید درج کنید

از شواهد عمومی و غیرحساس استفاده کنید. اثبات‌های مفید شامل این موارد است:

- تاریخچه سازمان، مخزن، انتشار، یا نگه‌دارنده GitHub
- مستندات رسمی پروژه که فضای نام را نام می‌برد
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا رجیستری بسته دیگر
- شواهد مالکیت علامت تجاری، برند، یا پروژه که بحث درباره آن به‌صورت عمومی
  امن است
- تاریخچه مخزن منبع، تاریخچه بسته، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا issue مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را درج نکنید

اسرار یا اثبات خصوصی را در یک issue عمومی GitHub قرار ندهید. این موارد را درج نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## پیامدهای ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک مدخل موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا هدایت اضافه کنند، شواهد بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابق منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربران را می‌سنجند.
