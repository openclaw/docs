# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin و
scopeهای بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر برسد یک فضای نام به یک
پروژه، برند، زیست‌بوم بسته یا سازمان واقعی تعلق دارد اما از قبل در ClawHub
ادعا، رزرو، گمراه‌کننده یا مورد اختلاف شده است، از کارکنان بخواهید آن را با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های
درون‌محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنیم

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک فضای نام باید به دلیل مالکیت واقعی
رزرو، منتقل، تغییرنام داده، پنهان، قرنطینه، مستعارسازی یا به‌نحوی دیگر تغییر داده شود،
یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- scope بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک ClawHub
  متناظر منتشر شود
- اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف درباره برند، نشان تجاری، تغییرنام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که مالک برحق فضای نام را مسدود می‌کند

اگر فهرست ناامن، مخرب یا فراتر از اختلاف مالکیت گمراه‌کننده است،
راهنمای مربوط به مدیریت محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دارای scope مانند `@example-org/example-plugin` باید به‌عنوان
مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی یا حذف منبع تحت‌تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. اثبات‌های مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل scope در npm، PyPI، crates.io یا دیگر رجیستری‌های بسته
- شواهد مالکیت نشان تجاری، برند یا پروژه که بحث عمومی درباره آن ایمن است
- تاریخچه مخزن منبع، تاریخچه بسته یا اطلاعیه‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را اثبات می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا اثبات خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، یک منبع را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، اثبات بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی و تأثیر بر کاربر را می‌سنجند.
