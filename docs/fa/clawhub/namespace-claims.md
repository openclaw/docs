# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، نامک‌های skill، نام بسته‌های plugin و
دامنه‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به
یک پروژه، برند، زیست‌بوم بسته یا سازمان واقعی تعلق دارد، اما در ClawHub از قبل
ادعا یا رزرو شده، گمراه‌کننده است یا بر سر آن اختلاف وجود دارد، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

برای بررسی عمومی و غیرحساس مالکیت از این مسیر استفاده کنید. برای ادعاهای فضای نام از
گزارش‌های درون‌محصولی یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی ادعایی ثبت شود

زمانی ادعای فضای نام ثبت کنید که معتقدید کارکنان ClawHub باید بررسی کنند آیا
یک فضای نام به‌دلیل مالکیت در دنیای واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار
یا به‌شکل دیگری تغییر داده شود.

نمونه‌ها عبارت‌اند از:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- دامنه بسته‌ای مانند `@example-org/*` که باید فقط تحت مالک متناظر
  ClawHub منتشر شود
- نامک skill یا نام بسته plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییرنام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که مانع مالک برحق فضای نام
  می‌شود

اگر فهرست، فراتر از اختلاف مالکیت، ناامن، مخرب یا گمراه‌کننده است،
راهنمای مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های plugin، نام‌های دامنه‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌کردن یا حذف منبع تحت‌تأثیر اصلاح کنید. زمانی از ادعا
استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید
اختلافی را حل‌وفصل کنند.

## شواهدی که باید ارائه شوند

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید عبارت‌اند از:

- تاریخچه سازمان، مخزن، انتشار یا نگه‌دارندگان در GitHub
- مستندات رسمی پروژه که فضای نام را ذکر می‌کنند
- مدرک دامنه یا دامنه رسمی ایمیل
- کنترل دامنه در npm، PyPI، crates.io یا دیگر سامانه‌های ثبت بسته
- شواهد مالکیت علامت تجاری، برند یا پروژه که گفت‌وگوی عمومی درباره آن
  ایمن است
- تاریخچه مخزن مبدأ، تاریخچه بسته یا اطلاعیه‌های عمومی تغییرنام
- پیوندها به مالک، skill، plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را اثبات می‌کند. کارکنان باید بتوانند
بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی، رابطه را درک کنند.

## چه مواردی نباید ارائه شوند

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. موارد زیر را وارد نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتریان

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارند.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را انتقال دهند، نام یک منبع را تغییر دهند، فهرست موجودی را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، شواهد بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبق منتقل شود.
کارکنان شواهد عمومی، استفاده فعلی، ریسک امنیتی و تأثیر بر کاربر را می‌سنجند.
