# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، slugهای مهارت، نام‌های بستهٔ Plugin، و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته، یا سازمان واقعی تعلق دارد اما در ClawHub از قبل
ادعا، رزرو، گمراه‌کننده، یا محل اختلاف شده است، از کارکنان بخواهید آن را
با
[فرم مسئلهٔ ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های
داخل محصول یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، alias،
یا به‌شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- یک شناسهٔ سازمان که با سازمان GitHub، پروژه، شرکت، یا جامعهٔ شما مطابقت دارد
- یک محدودهٔ بسته مانند `@example-org/*` که فقط باید تحت مالک
  متناظر ClawHub منتشر شود
- یک slug مهارت یا نام بستهٔ Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف بر سر برند، نشان تجاری، تغییرنام پروژه، یا تاریخچهٔ بسته
- یک مالک حذف‌شده، غیرفعال، یا دسترس‌ناپذیر که مانع مالک شایستهٔ فضای نام
  می‌شود

اگر فهرست فراتر از اختلاف مالکیت ناامن، مخرب، یا گمراه‌کننده است،
راهنمای مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید به‌عنوان
مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت‌تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچهٔ سازمان GitHub، مخزن، انتشار، یا نگهدارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- مدرک دامنه یا دامنهٔ ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا رجیستری بستهٔ دیگر
- شواهد مالکیت نشان تجاری، برند، یا پروژه که بحث عمومی دربارهٔ آن امن است
- تاریخچهٔ مخزن منبع، تاریخچهٔ بسته، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا مسئلهٔ مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئلهٔ عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویت شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانهٔ مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی محتوای حساس، از همان گزینه استفاده کنید.

## نتیجه‌های ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک alias یا تغییرمسیر اضافه کنند، مدرک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبق منتقل شود.
کارکنان شواهد عمومی، استفادهٔ موجود، ریسک امنیتی، و اثر بر کاربران را می‌سنجند.
