# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin و
scopeهای بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته یا سازمان واقعی تعلق دارد اما در ClawHub
قبلاً ادعا شده، رزرو شده، گمراه‌کننده است یا محل اختلاف است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی مالکیت عمومی و غیرحساس استفاده کنید. برای ادعاهای فضای نام از گزارش‌های درون‌محصول
یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، alias
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- scope بسته‌ای مانند `@example-org/*` که باید فقط زیر مالک
  متناظر ClawHub منتشر شود
- اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد جعل هویت یک پروژه باشد
- اختلاف درباره برند، علامت تجاری، تغییرنام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا غیرقابل‌دسترسی که جلوی مالک برحق فضای نام را
  می‌گیرد

اگر فهرست فراتر از اختلاف مالکیت ناامن، مخرب یا گمراه‌کننده است،
همچنین راهنمایی مربوط به نظارت یا امنیت را دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های scoped مانند `@example-org/example-plugin` باید به‌عنوان
مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی یا حذف منبع تحت تأثیر اصلاح کنید. وقتی نمی‌توانید
مالک فعلی را مدیریت کنید یا وقتی کارکنان باید اختلافی را حل کنند، از ادعا استفاده کنید.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- مدرک دامنه یا دامنه ایمیل رسمی
- کنترل scope در npm، PyPI، crates.io یا رجیستری‌های بسته دیگر
- شواهد مالکیت علامت تجاری، برند یا پروژه که برای بحث عمومی امن است
- تاریخچه مخزن منبع، تاریخچه بسته یا اطلاعیه‌های عمومی تغییرنام
- لینک به مالک، مهارت، Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر لینک چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی محتوای حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک alias یا redirect اضافه کنند، مدرک بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی و تأثیر بر کاربر را می‌سنجند.
