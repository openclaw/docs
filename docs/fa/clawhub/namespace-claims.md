# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin و
دامنه‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه واقعی، برند، زیست‌بوم بسته یا سازمان تعلق دارد، اما در ClawHub از قبل
ادعا شده، رزرو شده، گمراه‌کننده است یا درباره آن اختلاف وجود دارد، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از
گزارش‌های داخل محصول یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

زمانی یک ادعای فضای نام باز کنید که باور دارید کارکنان ClawHub باید بررسی کنند آیا یک
فضای نام باید به دلیل مالکیت واقعی رزرو، منتقل، تغییرنام، پنهان، قرنطینه، مستعار
یا به شکل دیگری تغییر داده شود.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- دامنه بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک منطبق
  ClawHub منتشر شود
- اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییر نام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که مالک برحق فضای نام را مسدود می‌کند

اگر فهرست فراتر از اختلاف مالکیت، ناامن، مخرب یا گمراه‌کننده است،
راهنمای مرتبط با تعدیل محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دامنه‌دار مانند `@example-org/example-plugin` باید
با مالک منطبق `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییر نام، انتقال، پنهان کردن یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا
استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. شواهد مفید شامل این موارد است:

- سازمان GitHub، مخزن، انتشار یا تاریخچه نگهدارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل دامنه در npm، PyPI، crates.io یا دیگر رجیستری‌های بسته
- شواهد مالکیت علامت تجاری، برند یا پروژه که گفت‌وگوی عمومی درباره آن امن است
- تاریخچه مخزن منبع، تاریخچه بسته یا اعلان‌های عمومی تغییر نام
- پیوندها به مالک، مهارت، Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند رابطه را
بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا شواهد خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویت شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، یک منبع را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، شواهد بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبق منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی و تأثیر بر کاربر را می‌سنجند.
