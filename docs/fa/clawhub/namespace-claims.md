# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، نامک‌های مهارت، نام‌های بسته Plugin، و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، زیست‌بوم بسته، یا سازمان واقعی تعلق دارد اما پیشاپیش در
ClawHub ادعا، رزرو، گمراه‌کننده، یا محل اختلاف شده است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی مالکیت عمومی و غیرحساس استفاده کنید. برای ادعاهای فضای نام از گزارش‌های درون‌محصولی
یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنیم

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به‌دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، مستعار،
یا به شکل دیگری تغییر کند، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که باید فقط زیر مالک
  متناظر ClawHub منتشر شود
- نامک مهارت یا نام بسته Plugin که به نظر می‌رسد هویت یک پروژه را جعل می‌کند
- اختلاف مربوط به برند، علامت تجاری، تغییرنام پروژه، یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال، یا دسترس‌ناپذیری که مالک برحق فضای نام را مسدود می‌کند

اگر فهرست فراتر از اختلاف مالکیت، ناامن، مخرب، یا گمراه‌کننده است،
راهنمای مرتبط با نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید وارد کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که نام فضای نام را ذکر می‌کنند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا دیگر رجیستری‌های بسته
- شواهد مالکیت علامت تجاری، برند، یا پروژه که برای بحث عمومی امن است
- تاریخچه مخزن منبع، تاریخچه بسته، یا اطلاعیه‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌های خصوصی یا اسرار درک کنند.

## چه چیزهایی را وارد نکنید

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را وارد نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویت شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، یک منبع را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، مدارک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربران را می‌سنجند.
