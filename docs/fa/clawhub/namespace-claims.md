# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های Skills، نام‌های بسته Plugin و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته یا سازمان واقعی تعلق دارد اما در ClawHub از قبل
ادعا، رزرو، گمراه‌کننده یا محل اختلاف شده است، از کارکنان بخواهید آن را با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های
داخل محصول یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی ادعا ثبت کنید

زمانی ادعای فضای نام ثبت کنید که باور دارید کارکنان ClawHub باید بررسی کنند آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر داده شود.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که فقط باید تحت مالک
  متناظر ClawHub منتشر شود
- اسلاگ Skills یا نام بسته Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف بر سر برند، نشان تجاری، تغییرنام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا غیرقابل‌دسترسی که مانع مالک برحق فضای نام می‌شود

اگر فهرست، فراتر از اختلاف مالکیت، ناامن، مخرب یا گمراه‌کننده است،
راهنمای مربوط به مدیریت محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی یا حذف منبع متأثر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برد
- مدرک دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io یا رجیستری‌های بسته دیگر
- شواهد مالکیت نشان تجاری، برند یا پروژه که بحث عمومی درباره آن امن است
- تاریخچه مخزن منبع، تاریخچه بسته یا اعلان‌های عمومی تغییرنام
- پیوندهایی به مالک، Skills، Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را اثبات می‌کند. کارکنان باید بتوانند این
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویت شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتریان

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از همان گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، مدارک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی و اثر بر کاربران را می‌سنجند.
