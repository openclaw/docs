# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بستهٔ Plugin، و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته، یا سازمان واقعی تعلق دارد اما در ClawHub از قبل
ادعا، رزرو، گمراه‌کننده، یا مورد اختلاف است، از کارکنان بخواهید آن را با
[فرم مسئلهٔ ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های
درون‌محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی فکر می‌کنید کارکنان ClawHub باید بررسی کنند که آیا یک فضای نام به دلیل مالکیت واقعی
باید رزرو، منتقل، تغییر نام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر کند، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- یک شناسهٔ سازمان که با سازمان GitHub، پروژه، شرکت، یا جامعهٔ شما مطابقت دارد
- محدودهٔ بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک
  متناظر ClawHub منتشر شود
- اسلاگ مهارت یا نام بستهٔ Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف دربارهٔ برند، علامت تجاری، تغییر نام پروژه، یا تاریخچهٔ بسته
- مالک حذف‌شده، غیرفعال، یا غیرقابل دسترسی که مانع مالک برحق فضای نام می‌شود

اگر فهرست ناامن، مخرب، یا فراتر از اختلاف مالکیت گمراه‌کننده است،
راهنمای تعدیل یا امنیت مرتبط را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دارای محدوده مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییر نام، انتقال، پنهان‌سازی، یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید درج شود

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچهٔ سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- مدرک دامنه یا دامنهٔ ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا رجیستری‌های بستهٔ دیگر
- شواهد مالکیت علامت تجاری، برند، یا پروژه که بحث عمومی دربارهٔ آن امن است
- تاریخچهٔ مخزن منبع، تاریخچهٔ بسته، یا اعلان‌های عمومی تغییر نام
- پیوندها به مالک، مهارت، Plugin، بسته، یا مسئلهٔ مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌های خصوصی یا اسرار درک کنند.

## چه چیزهایی را نباید درج کرد

اسرار یا مدارک خصوصی را در یک مسئلهٔ عمومی GitHub قرار ندهید. این موارد را درج نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- اسناد هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانهٔ مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مواد حساس، از همان گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، مدارک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفادهٔ موجود، ریسک امنیتی، و اثر بر کاربران را می‌سنجند.
