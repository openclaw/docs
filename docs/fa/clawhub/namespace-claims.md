# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های package مربوط به Plugin، و
محدوده‌های package به‌عنوان فضای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام متعلق به یک
پروژه، برند، زیست‌بوم package، یا سازمان واقعی است اما در ClawHub از قبل
ادعا شده، رزرو شده، گمراه‌کننده، یا مورد اختلاف است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام،
از گزارش‌های داخل محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک فضای نام باید به دلیل
مالکیت واقعی رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد هستند:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- محدوده package مانند `@example-org/*` که باید فقط زیر مالک ClawHub
  متناظر منتشر شود
- اسلاگ مهارت یا نام package مربوط به Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف درباره برند، نشان تجاری، تغییرنام پروژه، یا تاریخچه package
- مالک حذف‌شده، غیرفعال، یا غیرقابل‌دسترسی که مانع مالک برحق فضای نام می‌شود

اگر فهرست ناامن، مخرب، یا فراتر از اختلاف مالکیت گمراه‌کننده است،
راهنمای مرتبط با تعدیل محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای packageهای Plugin، نام‌های دارای محدوده مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت‌تأثیر اصلاح کنید. زمانی از ادعا
استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. شواهد مفید شامل این موارد هستند:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا دیگر رجیستری‌های package
- شواهد مالکیت نشان تجاری، برند، یا پروژه که بحث عمومی درباره آن امن است
- تاریخچه مخزن منبع، تاریخچه package، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، package، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند رابطه را
بدون نیاز به اعتبارنامه‌های خصوصی یا اسرار درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا شواهد خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، منبعی را تغییرنام دهند، فهرست موجودی را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، شواهد بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبق منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و تأثیر بر کاربر را می‌سنجند.
