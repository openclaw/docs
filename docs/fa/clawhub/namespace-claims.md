# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، slugهای Skills، نام‌های بسته Plugin و
scopeهای بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه واقعی، برند، اکوسیستم بسته، یا سازمان تعلق دارد اما در ClawHub از قبل
ادعا شده، رزرو شده، گمراه‌کننده است، یا بر سر آن اختلاف وجود دارد، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از
گزارش‌های درون‌محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، بازنام‌گذاری، پنهان، قرنطینه، دارای alias،
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- یک شناسه سازمان که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- یک scope بسته مانند `@example-org/*` که فقط باید زیر مالک
  متناظر ClawHub منتشر شود
- یک slug مهارت یا نام بسته Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییر نام پروژه، یا تاریخچه بسته
- مالکی حذف‌شده، غیرفعال، یا دسترس‌ناپذیر که مالک برحق فضای نام را
  مسدود می‌کند

اگر فهرست فراتر از اختلاف مالکیت ناامن، مخرب، یا گمراه‌کننده است،
راهنمای مرتبط تعدیل محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های scoped مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
بازنام‌گذاری، انتقال، پنهان‌سازی، یا حذف منبع متأثر اصلاح کنید. وقتی
نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید اختلافی را حل کنند، از ادعا استفاده کنید.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برد
- مدرک دامنه یا دامنه ایمیل رسمی
- کنترل scope در npm، PyPI، crates.io، یا دیگر registryهای بسته
- شواهد مالکیت علامت تجاری، برند، یا پروژه که بحث عمومی درباره آن امن است
- تاریخچه مخزن منبع، تاریخچه بسته، یا اعلان‌های عمومی تغییر نام
- پیوندها به مالک، مهارت، Plugin، بسته، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌های خصوصی یا اسرار درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- اسناد هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مواد حساس، از آن گزینه استفاده کنید.

## پیامدهای ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، یک منبع را بازنام‌گذاری کنند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک alias یا تغییرمسیر اضافه کنند، شواهد بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربر را می‌سنجند.
