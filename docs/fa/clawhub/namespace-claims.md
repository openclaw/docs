# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin و
دامنه‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر برسد یک فضای نام به یک
پروژه واقعی، برند، زیست‌بوم بسته، یا سازمان تعلق دارد، اما در ClawHub قبلا
ادعا شده، رزرو شده، گمراه‌کننده است، یا محل اختلاف است، از کارکنان بخواهید آن را با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از
گزارش‌های درون‌محصول یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک فضای نام به دلیل مالکیت واقعی
باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- دامنه بسته‌ای مانند `@example-org/*` که باید فقط زیر مالک
  متناظر ClawHub منتشر شود
- اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف درباره برند، نشان تجاری، تغییرنام پروژه، یا سابقه بسته
- مالکی حذف‌شده، غیرفعال، یا دسترس‌ناپذیر که جلوی مالک برحق فضای نام را
  می‌گیرد

اگر فهرست، فراتر از اختلاف مالکیت، ناامن، مخرب، یا گمراه‌کننده است،
راهنمایی مرتبط با مدیریت محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دامنه‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیما با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع متاثر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- سابقه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برد
- مدرک دامنه یا دامنه ایمیل رسمی
- کنترل دامنه npm، PyPI، crates.io، یا دیگر رجیستری‌های بسته
- شواهد مالکیت نشان تجاری، برند، یا پروژه که طرح عمومی آن امن است
- سابقه مخزن منبع، سابقه بسته، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از همان گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، منبعی را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، مدرک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربر را می‌سنجند.
