# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمانی، شناسه‌های کوتاه مهارت، نام‌های بسته‌های Plugin و
دامنه‌های بسته به‌عنوان فضای نام‌های عمومی استفاده می‌کند. اگر به نظر برسد یک فضای نام به یک
پروژه، برند، زیست‌بوم بسته، یا سازمان واقعی تعلق دارد اما پیش‌تر در
ClawHub ادعا شده، رزرو شده، گمراه‌کننده است، یا محل اختلاف است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام از گزارش‌های
داخل محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام باید به دلیل مالکیت واقعی رزرو، منتقل، تغییرنام، پنهان، قرنطینه، مستعارسازی،
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- یک شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- یک دامنه بسته مانند `@example-org/*` که فقط باید زیر مالک
  متناظر ClawHub منتشر شود
- یک شناسه کوتاه مهارت یا نام بسته Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف بر سر برند، علامت تجاری، تغییرنام پروژه، یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال، یا دسترس‌ناپذیری که مالک شایسته فضای نام را مسدود می‌کند

اگر فهرست فراتر از اختلاف مالکیت ناامن، مخرب، یا گمراه‌کننده است،
همچنین راهنمایی‌های مرتبط با مدیریت محتوا یا امنیت را دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دامنه‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت‌تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید
که نمی‌توانید مالک فعلی را مدیریت کنید یا زمانی که کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید ارائه شود

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برد
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل دامنه در npm، PyPI، crates.io، یا دیگر رجیستری‌های بسته
- شواهد مالکیت علامت تجاری، برند، یا پروژه که برای بحث عمومی امن است
- تاریخچه مخزن منبع، تاریخچه بسته، یا اطلاعیه‌های عمومی تغییرنام
- پیوندهایی به مالک، مهارت، Plugin، بسته، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
بدون نیاز به اعتبارنامه‌ها یا رازهای خصوصی، رابطه را درک کنند.

## چه چیزهایی را نباید ارائه کنید

رازها یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را وارد نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مواد حساس، از آن گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، مدارک بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربران را می‌سنجند.
