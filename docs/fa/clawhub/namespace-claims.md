# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالکان، شناسه‌های سازمان‌ها، نامک‌های Skills، نام بسته‌های Plugin و محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام متعلق به پروژه، برند، زیست‌بوم بسته یا سازمانی واقعی است، اما در ClawHub قبلاً ادعا یا رزرو شده، گمراه‌کننده است یا درباره آن اختلاف وجود دارد، از کارکنان بخواهید با استفاده از
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
آن را بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام، از گزارش‌های درون‌محصولی یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی ادعایی ثبت کنید

زمانی ادعای فضای نام ثبت کنید که معتقدید کارکنان ClawHub باید بررسی کنند آیا به‌دلیل مالکیت واقعی، فضای نامی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار یا به‌شکل دیگری تغییر داده شود.

نمونه‌ها عبارت‌اند از:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که باید فقط تحت مالک متناظر `example-org` در ClawHub منتشر شود
- نامک یک Skill یا نام بسته Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییر نام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که مانع مالک قانونی فضای نام می‌شود

اگر مورد فهرست‌شده علاوه بر اختلاف مالکیت، ناامن، مخرب یا گمراه‌کننده است، راهنمای مربوط به نظارت یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که انتشار را با مالکی انجام می‌دهید که با فضای نام مطابقت دارد. برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید با مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار، تغییر نام، انتقال، پنهان‌کردن یا حذف منبع تحت‌تأثیر اصلاح کنید. زمانی از ادعا استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا کارکنان باید اختلافی را حل‌وفصل کنند.

## شواهدی که باید ارائه شوند

از شواهد عمومی و غیرحساس استفاده کنید. مدارک مفید عبارت‌اند از:

- تاریخچه سازمان، مخزن، انتشار یا نگه‌دارندگان در GitHub
- مستندات رسمی پروژه که فضای نام را ذکر می‌کنند
- مدرک دامنه یا دامنه رسمی ایمیل
- کنترل محدوده در npm،‏ PyPI،‏ crates.io یا دیگر سامانه‌های ثبت بسته
- شواهد مالکیت علامت تجاری، برند یا پروژه که طرح عمومی آن‌ها بی‌خطر است
- تاریخچه مخزن منبع، تاریخچه بسته یا اطلاعیه‌های عمومی تغییر نام
- پیوندهای مالک، Skill،‏ Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را اثبات می‌کند. کارکنان باید بتوانند بدون نیاز به اطلاعات احراز هویت خصوصی یا اسرار، رابطه را درک کنند.

## مواردی که نباید ارائه شوند

اسرار یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. موارد زیر را ارائه نکنید:

- توکن‌های API، کلیدهای امضا یا اطلاعات احراز هویت
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتریان

فرم ادعا می‌پرسد آیا شواهد حساس به کانال خصوصی کارکنان نیاز دارند یا خیر. به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و خطر، کارکنان ClawHub ممکن است فضای نامی را رزرو کنند، مالکیت را انتقال دهند، نام منبعی را تغییر دهند، مورد فهرست‌شده موجودی را پنهان یا قرنطینه کنند، نام مستعار یا تغییرمسیر بیفزایند، مدارک بیشتری درخواست کنند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام منطبقی منتقل شود. کارکنان شواهد عمومی، استفاده موجود، خطر امنیتی و تأثیر بر کاربران را می‌سنجند.
