# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin و
scopeهای بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته یا سازمان واقعی تعلق دارد اما قبلا در
ClawHub ادعا، رزرو، گمراه‌کننده یا محل اختلاف شده است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام
از گزارش‌های درون‌محصول یا فرم فرجام‌خواهی حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام به دلیل مالکیت واقعی باید رزرو، منتقل، تغییرنام، پنهان، قرنطینه، دارای نام مستعار،
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- یک شناسه سازمان که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- یک scope بسته مانند `@example-org/*` که فقط باید زیر مالک
  متناظر ClawHub منتشر شود
- یک اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییرنام پروژه یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که فضای نام مالک برحق را مسدود می‌کند

اگر فهرست، فراتر از اختلاف مالکیت، ناامن، مخرب یا گمراه‌کننده است،
همچنین راهنمای مرتبط با تعدیل یا امنیت را دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تایید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های دارای scope مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیما با انتشار،
تغییرنام، انتقال، پنهان‌سازی یا حذف منبع متاثر اصلاح کنید. وقتی نمی‌توانید
مالک فعلی را مدیریت کنید یا کارکنان باید یک اختلاف را حل کنند، از ادعا استفاده کنید.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. اثبات‌های مفید شامل این موارد است:

- تاریخچه سازمان، مخزن، انتشار یا نگه‌دارنده GitHub
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل scope در npm، PyPI، crates.io یا رجیستری‌های بسته دیگر
- شواهد مالکیت علامت تجاری، برند یا پروژه که برای بحث عمومی امن است
- تاریخچه مخزن منبع، تاریخچه بسته یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
بدون نیاز به اعتبارنامه‌ها یا اسرار خصوصی، رابطه را درک کنند.

## چه چیزهایی را نباید اضافه کنید

اسرار یا اثبات خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- فایل‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مواد حساس، از آن گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، منبعی را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
یک نام مستعار یا تغییرمسیر اضافه کنند، اثبات بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابقی منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی و تاثیر بر کاربر را می‌سنجند.
