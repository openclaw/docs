# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های مهارت، نام‌های بسته Plugin و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، زیست‌بوم بسته، یا سازمان واقعی تعلق دارد اما از قبل در
ClawHub ادعا، رزرو، گمراه‌کننده، یا محل اختلاف شده است، از کارکنان بخواهید آن را
با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی مالکیت عمومی و غیرحساس استفاده کنید. برای ادعاهای فضای نام از گزارش‌های داخل محصول
یا فرم اعتراض حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنید

وقتی باور دارید کارکنان ClawHub باید بررسی کنند که آیا یک
فضای نام باید به دلیل مالکیت واقعی رزرو، منتقل، تغییرنام، پنهان، قرنطینه، مستعار
یا به شکل دیگری تغییر داده شود، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت، یا جامعه شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک
  متناظر ClawHub منتشر شود
- اسلاگ مهارت یا نام بسته Plugin که به نظر می‌رسد خود را به‌جای یک پروژه جا می‌زند
- اختلاف بر سر برند، نشان تجاری، تغییرنام پروژه، یا تاریخچه بسته
- مالک حذف‌شده، غیرفعال، یا غیرقابل‌دسترسی که فضای نام را برای مالک
  برحق آن مسدود کرده است

اگر فهرست علاوه بر اختلاف مالکیت، ناامن، مخرب، یا گمراه‌کننده است،
راهنمای تعدیل یا امنیتی مرتبط را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید
به‌عنوان مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیماً با انتشار،
تغییرنام، انتقال، پنهان‌سازی، یا حذف منبع تحت تأثیر اصلاح کنید. وقتی از عهده مدیریت مالک فعلی برنمی‌آیید یا کارکنان باید
اختلافی را حل کنند، از ادعا استفاده کنید.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. شواهد مفید شامل این موارد است:

- تاریخچه سازمان GitHub، مخزن، انتشار، یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- اثبات دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io، یا رجیستری‌های بسته دیگر
- شواهد مالکیت نشان تجاری، برند، یا پروژه که بتوان آن را به‌صورت عمومی
  مطرح کرد
- تاریخچه مخزن منبع، تاریخچه بسته، یا اعلان‌های عمومی تغییرنام
- پیوندها به مالک، مهارت، Plugin، بسته، یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند
رابطه را بدون نیاز به اعتبارنامه‌های خصوصی یا اسرار بفهمند.

## چه چیزهایی را اضافه نکنید

اسرار یا شواهد خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا، یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی، یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج احتمالی

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، منبعی را تغییرنام دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، شواهد بیشتری بخواهند، یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابق منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی، و اثر بر کاربر را می‌سنجند.
