# ادعاهای سازمان و فضای نام

ClawHub از شناسه‌های مالک، شناسه‌های سازمان، اسلاگ‌های Skills، نام‌های بسته Plugin و
محدوده‌های بسته به‌عنوان فضاهای نام عمومی استفاده می‌کند. اگر به نظر می‌رسد یک فضای نام به یک
پروژه، برند، اکوسیستم بسته یا سازمان واقعی تعلق دارد اما در ClawHub قبلا
ادعا، رزرو، گمراه‌کننده یا مورد اختلاف شده است، از کارکنان بخواهید آن را با
[فرم مسئله ادعای سازمان / فضای نام](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
بررسی کنند.

از این مسیر برای بررسی عمومی و غیرحساس مالکیت استفاده کنید. برای ادعاهای فضای نام
از گزارش‌های درون‌محصول یا فرم درخواست تجدیدنظر حساب استفاده نکنید.

## چه زمانی یک ادعا باز کنیم

وقتی معتقدید کارکنان ClawHub باید بررسی کنند که آیا یک فضای نام باید به دلیل
مالکیت واقعی رزرو، منتقل، تغییر نام داده، پنهان، قرنطینه، دارای نام مستعار
یا به شکل دیگری تغییر کند، یک ادعای فضای نام باز کنید.

نمونه‌ها شامل این موارد است:

- شناسه سازمانی که با سازمان GitHub، پروژه، شرکت یا جامعه شما مطابقت دارد
- محدوده بسته‌ای مانند `@example-org/*` که فقط باید زیر مالک متناظر
  ClawHub منتشر شود
- اسلاگ Skills یا نام بسته Plugin که به نظر می‌رسد خود را جای یک پروژه جا می‌زند
- اختلاف درباره برند، علامت تجاری، تغییر نام پروژه یا سابقه بسته
- مالک حذف‌شده، غیرفعال یا دسترس‌ناپذیری که مالک برحق فضای نام را مسدود می‌کند

اگر فهرست ناامن، مخرب یا فراتر از اختلاف مالکیت گمراه‌کننده است،
راهنمای مربوط به مدیریت محتوا یا امنیت را نیز دنبال کنید. فرم ادعای فضای نام
برای بررسی مالکیت است، نه افشای اضطراری آسیب‌پذیری.

## پیش از ثبت

ابتدا تأیید کنید که با مالکی منتشر می‌کنید که با فضای نام مطابقت دارد.
برای بسته‌های Plugin، نام‌های محدوده‌دار مانند `@example-org/example-plugin` باید
با مالک متناظر `example-org` منتشر شوند.

اگر می‌توانید مالک فعلی را مدیریت کنید، فضای نام را مستقیما با انتشار،
تغییر نام، انتقال، پنهان‌سازی یا حذف منبع تحت تأثیر اصلاح کنید. زمانی از ادعا
استفاده کنید که نمی‌توانید مالک فعلی را مدیریت کنید یا وقتی کارکنان باید یک
اختلاف را حل کنند.

## شواهدی که باید اضافه کنید

از شواهد عمومی و غیرحساس استفاده کنید. شواهد مفید شامل این موارد است:

- سابقه سازمان GitHub، مخزن، انتشار یا نگه‌دارنده
- مستندات رسمی پروژه که فضای نام را نام می‌برند
- مدرک دامنه یا دامنه ایمیل رسمی
- کنترل محدوده در npm، PyPI، crates.io یا رجیستری‌های بسته دیگر
- شواهد مالکیت علامت تجاری، برند یا پروژه که بحث عمومی درباره آن امن است
- سابقه مخزن منبع، سابقه بسته یا اعلان‌های عمومی تغییر نام
- پیوندها به مالک، Skills، Plugin، بسته یا مسئله مورد اختلاف در ClawHub

توضیح دهید هر پیوند چه چیزی را ثابت می‌کند. کارکنان باید بتوانند رابطه را
بدون نیاز به اعتبارنامه‌ها یا رازهای خصوصی درک کنند.

## چه چیزهایی را اضافه نکنید

رازها یا مدارک خصوصی را در یک مسئله عمومی GitHub قرار ندهید. این موارد را اضافه نکنید:

- توکن‌های API، کلیدهای امضا یا اعتبارنامه‌ها
- توکن‌های چالش DNS
- پرونده‌ها یا قراردادهای حقوقی خصوصی
- مدارک هویتی شخصی
- ایمیل‌های خصوصی، گزارش‌های امنیتی خصوصی یا داده‌های محرمانه مشتری

فرم ادعا می‌پرسد آیا شواهد حساس به یک کانال خصوصی کارکنان نیاز دارد یا نه.
به‌جای انتشار عمومی مطالب حساس، از آن گزینه استفاده کنید.

## نتایج ممکن

بسته به شواهد و ریسک، کارکنان ClawHub ممکن است یک فضای نام را رزرو کنند،
مالکیت را منتقل کنند، نام یک منبع را تغییر دهند، یک فهرست موجود را پنهان یا قرنطینه کنند،
نام مستعار یا تغییرمسیر اضافه کنند، مدارک بیشتری بخواهند یا درخواست را رد کنند.

بررسی فضای نام تضمین نمی‌کند که هر نام مطابق منتقل شود.
کارکنان شواهد عمومی، استفاده موجود، ریسک امنیتی و اثر بر کاربر را می‌سنجند.
