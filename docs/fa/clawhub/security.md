# امنیت + نظارت بر محتوا

ClawHub برای انتشار باز است، اما فهرست‌های عمومی همچنان از کنترل‌های اعتماد،
اسکن، گزارش‌دهی و نظارت عبور می‌کنند. هدف عملی است: کمک به کاربران برای
بررسی آنچه نصب می‌کنند، دادن مسیر بازیابی به ناشران برای موارد مثبت کاذب،
و دور نگه داشتن بسته‌های سوءاستفاده‌گر از کشف عمومی.

همچنین [استفاده قابل قبول](/fa/clawhub/acceptable-usage) را ببینید.

## آنچه کاربران می‌توانند بررسی کنند

پیش از نصب یک skill یا plugin، فهرست ClawHub آن را برای موارد زیر بررسی کنید:

- مالک و انتساب منبع
- آخرین نسخه و changelog
- متغیرهای محیطی یا مجوزهای موردنیاز
- فراداده سازگاری برای plugins
- وضعیت اسکن یا نظارت
- گزارش‌ها، دیدگاه‌ها، ستاره‌ها، دانلودها و نشانه‌های نصب، در صورت نمایش

فقط محتوایی را نصب کنید که آن را می‌فهمید و به آن اعتماد دارید.

## وضعیت‌های اسکن

ClawHub ممکن است نتایج اسکن یا نظارت را در صفحه‌های عمومی و عیب‌یابی‌های
قابل مشاهده برای مالک نشان دهد.

نتایج رایج شامل موارد زیر است:

- `clean`: هیچ مشکل مسدودکننده‌ای پیدا نشد.
- `suspicious`: این انتشار به احتیاط یا بازبینی نیاز دارد.
- `malicious`: این انتشار ناامن تلقی می‌شود.
- `pending`: بررسی‌ها هنوز تمام نشده‌اند.
- `held`، `quarantined`، `revoked`، یا `hidden`: این انتشار روی سطح‌های نصب
  عمومی به‌طور کامل در دسترس نیست.

عبارت دقیق ممکن است بسته به سطح متفاوت باشد، اما معنای عملی یکسان است: اگر
انتشاری نگه داشته یا مسدود شده باشد، کاربران نباید آن را نصب کنند تا زمانی
که مالک مشکل را حل کند یا نظارت آن را بازگرداند.

## Skills

اسکن‌های Skill بسته skill منتشرشده، فراداده، الزامات اعلام‌شده و دستورالعمل‌های
مشکوک را بررسی می‌کنند.

ClawHub به ناهماهنگی بین آنچه یک skill اعلام می‌کند و کاری که به نظر می‌رسد
انجام می‌دهد توجه ویژه دارد. برای مثال، skillای که به یک کلید API موردنیاز
اشاره می‌کند، باید آن الزام را در `SKILL.md` اعلام کند تا کاربران پیش از نصب
بتوانند آن را ببینند.

یافته‌های اسکن مبتنی بر artifact هستند. رفتار مورد انتظار provider، مانند
اعتبارنامه‌های API اعلام‌شده، callbackهای OAuth روی localhost، پاک‌سازی
حذف نصب با دامنه محدود، کدگذاری Basic Auth، یا بارگذاری فایل‌های انتخاب‌شده
توسط کاربر به provider اعلام‌شده، متفاوت از ارسال پنهانی اعتبارنامه‌ها،
دسترسی گسترده به فایل‌های خصوصی، مقصدهای شبکه‌ای نامرتبط، یا سوءاستفاده
پنهانی از مرورگر برخورد می‌شود.

[قالب Skill](/fa/clawhub/skill-format) را ببینید.

## Plugins

انتشارهای Plugin شامل فراداده بسته، انتساب منبع، فیلدهای سازگاری و اطلاعات
یکپارچگی artifact هستند.

OpenClaw پیش از نصب plugins میزبانی‌شده در ClawHub، سازگاری را بررسی می‌کند.
رکوردهای بسته همچنین ممکن است فراداده digest را نمایش دهند تا OpenClaw
بتواند artifactهای دانلودشده را تأیید کند. ClawScan هنگام بازبینی انتشارهای
plugin، فراداده env/config اعلام‌شده در `openclaw.environment` بسته را لحاظ
می‌کند تا الزامات runtime اعلام‌شده با رفتار مشاهده‌شده مقایسه شوند.

## گزارش‌ها

کاربران واردشده می‌توانند skills، بسته‌ها و دیدگاه‌ها را گزارش کنند.

گزارش‌ها باید مشخص و قابل اقدام باشند. سوءاستفاده از گزارش‌دهی می‌تواند خود
به اقدام علیه حساب منجر شود.

نمونه‌های گزارش:

- فراداده گمراه‌کننده
- الزامات اعلام‌نشده اعتبارنامه یا مجوز
- دستورالعمل‌های نصب مشکوک
- دیدگاه‌های کلاهبردارانه یا جعل هویت
- ثبت‌نام‌های با نیت بد یا سوءاستفاده از trademark
- محتوایی که [استفاده قابل قبول](/fa/clawhub/acceptable-usage) را نقض می‌کند

## گزارش‌های با نیت بد یا trademark

ClawHub برای ثبت‌نام‌های با نیت بد، جعل هویت و اختلاف‌های مرتبط با trademark
از همان مسیر گزارش‌دهی و نظارت کارکنان استفاده می‌کند. این گزارش‌ها به زمینه
کافی نیاز دارند تا کارکنان بتوانند مدعی، فهرست مورد اختلاف و اقدام درخواستی
را شناسایی کنند.

شامل موارد زیر باشد:

- URL رسمی skill یا بسته در ClawHub و handle مالک
- trademark، پروژه، شرکت، یا نام محصول مورد اختلاف
- شواهد عمومی مالکیت یا اختیار مدعی
- اینکه چرا مالک فعلی مجاز به انتشار با آن نام نیست
- اقدام درخواستی، مانند پنهان‌سازی تا زمان بازبینی، انتقال مالکیت، تغییر نام،
  یا حذف

اسرار خصوصی یا اسناد حقوقی حساس را در گزارش‌های عمومی قرار ندهید. یک issue در
GitHub با شواهد غیرحساس باز کنید و در صورت نیاز از نگه‌دارندگان مسیر تحویل
خصوصی بخواهید.

## درخواست تجدیدنظر و اسکن دوباره

مالکان وقتی باور دارند یک skill یا بسته به‌اشتباه نگه داشته یا علامت‌گذاری
شده است، می‌توانند درخواست اسکن دوباره بدهند. ناظران و مدیران پلتفرم می‌توانند
هنگام رسیدگی به گزارش‌ها یا درخواست‌های پشتیبانی، برای هر skill یا بسته
درخواست اسکن دوباره بدهند:

```bash
clawhub skill rescan <slug>
clawhub package rescan <name>
```

برای محتوای نظارت‌شده، مالکان ممکن است بتوانند از سطح‌های ClawHub قابل مشاهده
برای مالک، درخواست تجدیدنظر ثبت کنند. درخواست‌های تجدیدنظر باید توضیح دهند
چه چیزی تغییر کرده یا چرا علامت‌گذاری نادرست است.

## تعلیق‌های نظارتی

وقتی اسکنر static یک skill بارگذاری‌شده را malicious تشخیص دهد، ناشر به‌طور
خودکار تحت تعلیق نظارتی قرار می‌گیرد (`requiresModerationAt` روی کاربر تنظیم
می‌شود). این کار همه skills ناشر را پنهان می‌کند، باعث می‌شود انتشارهای آینده
به‌صورت پنهان شروع شوند، و یک ورودی audit log با نام `user.moderation.auto`
ایجاد می‌کند.

یافته‌های static suspicious به‌عنوان شواهد فایل/خط برای ناظران نگه داشته
می‌شوند، اما به‌تنهایی محتوا را پنهان نمی‌کنند یا verdict اسکن عمومی را تعیین
نمی‌کنند. بارگذاری‌های جدید تا نهایی شدن بازبینی‌های VirusTotal و LLM در
وضعیت بازبینی/در انتظار باقی می‌مانند؛ اسکن static فقط برای امضاهای malicious
بلافاصله مسدود می‌کند. بازبینی‌های LLM در ClawScan یادداشت‌های هم‌راستا با
هدف را به‌عنوان راهنما نگه می‌دارند؛ آن‌ها فقط زمانی verdict
Review/suspicious برمی‌گردانند که بازبینی ساخت‌یافته شامل یک نگرانی مهم باشد.

مدیران می‌توانند یک تعلیق مثبت کاذب را بردارند:

```bash
npx convex run users:liftModerationHold '{"userId": "<user-id>", "reason": "False positive from security tool scanning"}'
```

این کار `requiresModerationAt` و `requiresModerationReason` را پاک می‌کند،
skills پنهان‌شده توسط تعلیق سطح کاربر را بازمی‌گرداند، و یک ورودی audit log
با نام `user.moderation.lift` می‌نویسد. Skillsای که به دلایل دیگر پنهان
شده‌اند، یا اسکن static خودشان همچنان malicious است، پنهان می‌مانند.

## ممنوعیت‌ها و وضعیت حساب

حساب‌هایی که سیاست ClawHub را نقض کنند ممکن است دسترسی انتشار را از دست بدهند.
سوءاستفاده شدید می‌تواند به ممنوعیت حساب، لغو token، محتوای پنهان، یا
فهرست‌های حذف‌شده منجر شود.

حساب‌های حذف‌شده، ممنوع‌شده یا غیرفعال نمی‌توانند از tokenهای API در ClawHub
استفاده کنند. اگر پس از اقدام روی حساب، احراز هویت CLI شروع به شکست کرد، برای
بررسی وضعیت حساب وارد UI وب شوید یا از طریق کانال پشتیبانی مورد انتظار پروژه
با نگه‌دارندگان تماس بگیرید.

## راهنمایی ناشر

برای کاهش مثبت‌های کاذب و بهبود اعتماد کاربر:

- نام‌ها، خلاصه‌ها، tagها و changelogها را دقیق نگه دارید
- متغیرهای محیطی و مجوزهای موردنیاز را اعلام کنید
- از فرمان‌های نصب مبهم‌سازی‌شده پرهیز کنید
- در صورت امکان به منبع لینک دهید
- پیش از انتشار plugins از dry run استفاده کنید
- اگر کاربران یا ناظران درباره رفتار بسته پرسیدند، شفاف پاسخ دهید
