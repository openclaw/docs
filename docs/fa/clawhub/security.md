# امنیت + نظارت

ClawHub برای انتشار باز است، اما فهرست‌های عمومی همچنان از کنترل‌های اعتماد،
اسکن، گزارش‌دهی و نظارت عبور می‌کنند. هدف عملی است: کمک به کاربران برای
بررسی آنچه نصب می‌کنند، دادن مسیر بازیابی به ناشران برای مثبت‌های کاذب،
و دور نگه داشتن بسته‌های سوءاستفاده‌گر از کشف عمومی.

همچنین [استفاده قابل قبول](/fa/clawhub/acceptable-usage) را ببینید.

## کاربران چه چیزهایی را می‌توانند بررسی کنند

پیش از نصب یک مهارت یا Plugin، فهرست ClawHub آن را برای موارد زیر بررسی کنید:

- مالک و انتساب منبع
- آخرین نسخه و تغییرات
- متغیرهای محیطی یا مجوزهای موردنیاز
- فراداده سازگاری برای Pluginها
- وضعیت اسکن یا نظارت
- گزارش‌ها، نظرها، ستاره‌ها، دانلودها و سیگنال‌های نصب در صورت نمایش

فقط محتوایی را نصب کنید که آن را می‌فهمید و به آن اعتماد دارید.

## وضعیت‌های اسکن

ClawHub ممکن است نتایج اسکن یا نظارت را در صفحه‌های عمومی و عیب‌یابی‌های
قابل مشاهده برای مالک نشان دهد.

نتایج رایج شامل این موارد است:

- `clean`: هیچ مشکل مسدودکننده‌ای پیدا نشد.
- `suspicious`: انتشار به احتیاط یا بازبینی نیاز دارد.
- `malicious`: انتشار ناامن در نظر گرفته می‌شود.
- `pending`: بررسی‌ها هنوز تمام نشده‌اند.
- `held`، `quarantined`، `revoked` یا `hidden`: انتشار روی سطوح نصب عمومی
  به طور کامل در دسترس نیست.

عبارت دقیق ممکن است بسته به سطح متفاوت باشد، اما معنای عملی یکی است: اگر
انتشاری نگه داشته یا مسدود شده باشد، کاربران نباید آن را نصب کنند تا مالک
مشکل را حل کند یا نظارت آن را بازگرداند.

## Skills

اسکن‌های مهارت، بسته منتشرشده مهارت، فراداده، نیازمندی‌های اعلام‌شده
و دستورالعمل‌های مشکوک را بررسی می‌کنند.

ClawHub به ناسازگاری میان آنچه یک مهارت اعلام می‌کند و کاری که ظاهراً انجام
می‌دهد توجه ویژه دارد. برای نمونه، مهارتی که به یک کلید API موردنیاز اشاره
می‌کند باید آن نیازمندی را در `SKILL.md` اعلام کند تا کاربران پیش از نصب
آن را ببینند.

یافته‌های اسکن مبتنی بر مصنوع هستند. رفتار موردانتظار ارائه‌دهنده، مانند
اعتبارنامه‌های API اعلام‌شده، callbackهای OAuth روی localhost، پاک‌سازی
حذف نصب دامنه‌دار، کدگذاری Basic Auth، یا بارگذاری فایل‌های انتخاب‌شده توسط
کاربر به ارائه‌دهنده اعلام‌شده، با ارسال پنهانی اعتبارنامه‌ها، دسترسی گسترده
به فایل‌های خصوصی، مقصدهای شبکه نامرتبط، یا سوءاستفاده پنهانی از مرورگر
متفاوت برخورد می‌شود.

[قالب مهارت](/fa/clawhub/skill-format) را ببینید.

## Pluginها

انتشارهای Plugin شامل فراداده بسته، انتساب منبع، فیلدهای سازگاری
و اطلاعات یکپارچگی مصنوع هستند.

OpenClaw پیش از نصب Pluginهای میزبانی‌شده در ClawHub، سازگاری را بررسی
می‌کند. رکوردهای بسته همچنین ممکن است فراداده digest را ارائه کنند تا
OpenClaw بتواند مصنوع‌های دانلودشده را راستی‌آزمایی کند. ClawScan هنگام
بازبینی انتشارهای Plugin، فراداده env/config اعلام‌شده بسته
`openclaw.environment` را هم لحاظ می‌کند تا نیازمندی‌های runtime اعلام‌شده
با رفتار مشاهده‌شده مقایسه شوند.

## گزارش‌ها

کاربران واردشده می‌توانند مهارت‌ها، بسته‌ها و نظرها را گزارش کنند.

گزارش‌ها باید مشخص و قابل اقدام باشند. سوءاستفاده از گزارش‌دهی خود می‌تواند
به اقدام علیه حساب منجر شود.

نمونه‌های گزارش:

- فراداده گمراه‌کننده
- نیازمندی‌های اعلام‌نشده برای اعتبارنامه یا مجوز
- دستورالعمل‌های نصب مشکوک
- نظرهای کلاهبردارانه یا جعل هویت
- ثبت‌نام‌های بدخواهانه یا سوءاستفاده از نشان تجاری
- محتوایی که [استفاده قابل قبول](/fa/clawhub/acceptable-usage) را نقض می‌کند

## گزارش‌های بدخواهانه یا نشان تجاری

ClawHub برای ثبت‌نام‌های بدخواهانه، جعل هویت و اختلاف‌های مرتبط با نشان
تجاری از همان مسیر گزارش‌دهی و نظارت کارکنان استفاده می‌کند. این گزارش‌ها
به زمینه کافی نیاز دارند تا کارکنان بتوانند مدعی، فهرست مورد اختلاف
و اقدام درخواستی را شناسایی کنند.

شامل کنید:

- URL رسمی مهارت یا بسته ClawHub و شناسه مالک
- نشان تجاری، پروژه، شرکت یا نام محصول مورد بحث
- شواهد عمومی مالکیت یا اختیار مدعی
- اینکه چرا مالک فعلی مجاز نیست تحت آن نام منتشر کند
- اقدام درخواستی، مانند پنهان کردن تا زمان بازبینی، انتقال مالکیت، تغییر نام،
  یا حذف

اسرار خصوصی یا اسناد حقوقی حساس را در گزارش‌های عمومی قرار ندهید. در صورت
نیاز، یک issue در GitHub با شواهد غیرحساس باز کنید و از نگه‌دارندگان مسیر
تحویل خصوصی بخواهید.

## درخواست تجدیدنظر و اسکن دوباره

مالکان وقتی معتقدند یک مهارت یا بسته به‌اشتباه نگه داشته یا علامت‌گذاری شده
است می‌توانند درخواست اسکن دوباره بدهند. ناظران و مدیران پلتفرم می‌توانند
هنگام رسیدگی به گزارش‌ها یا درخواست‌های پشتیبانی برای هر مهارت یا بسته
درخواست اسکن دوباره بدهند:

```bash
clawhub skill rescan <slug>
clawhub package rescan <name>
```

برای محتوای نظارت‌شده، مالکان ممکن است بتوانند از سطوح ClawHub قابل مشاهده
برای مالک درخواست تجدیدنظر ارسال کنند. درخواست‌های تجدیدنظر باید توضیح دهند
چه چیزی تغییر کرده یا چرا علامت‌گذاری نادرست است.

## نگه‌داشت‌های نظارتی

وقتی اسکنر ایستا یک مهارت بارگذاری‌شده را مخرب علامت‌گذاری می‌کند، ناشر
به طور خودکار تحت نگه‌داشت نظارتی قرار می‌گیرد (`requiresModerationAt` روی
کاربر تنظیم می‌شود). این کار همه مهارت‌های ناشر را پنهان می‌کند، باعث می‌شود
انتشارهای آینده به صورت پنهان شروع شوند، و یک ورودی audit log
`user.moderation.auto` ایجاد می‌کند.

یافته‌های مشکوک ایستا به عنوان شواهد فایل/خط برای ناظران نگه داشته می‌شوند،
اما به‌تنهایی محتوا را پنهان نمی‌کنند یا رأی اسکن عمومی را تعیین نمی‌کنند.
بارگذاری‌های جدید تا زمان نهایی شدن بازبینی LLM در وضعیت بازبینی/در انتظار
می‌مانند. اسکن ایستا فقط برای امضاهای مخرب فوراً مسدود می‌کند. Trefferهای
موتور VirusTotal همچنان به عنوان شواهد امنیتی قابل مشاهده می‌مانند، اما
رأی‌های VirusTotal Code Insight/Palm مشورتی هستند و به‌تنهایی مهارت‌ها را
پنهان نمی‌کنند. بازبینی‌های LLM در ClawScan یادداشت‌های همسو با هدف را
به عنوان راهنما نگه می‌دارند. یافته‌های بازبینی متوسط روی مصنوع قابل مشاهده
می‌مانند، در حالی که فیلتر مشکوک برای نگرانی‌های LLM با اثر بالا، یافته‌های
مخرب، یا تشخیص‌های تأییدشده موتور AV رزرو شده است.

مدیران می‌توانند یک نگه‌داشت مثبت کاذب را بردارند:

```bash
npx convex run users:liftModerationHold '{"userId": "<user-id>", "reason": "False positive from security tool scanning"}'
```

این کار `requiresModerationAt` و `requiresModerationReason` را پاک می‌کند،
مهارت‌های پنهان‌شده به دلیل نگه‌داشت سطح کاربر را بازمی‌گرداند، و یک ورودی
audit log با نام `user.moderation.lift` می‌نویسد. مهارت‌هایی که به دلایل
دیگر پنهان شده‌اند، یا اسکن ایستای خودشان همچنان مخرب است، پنهان می‌مانند.

## ممنوعیت‌ها و وضعیت حساب

حساب‌هایی که سیاست ClawHub را نقض کنند ممکن است دسترسی انتشار را از دست
بدهند. سوءاستفاده شدید می‌تواند به ممنوعیت حساب، ابطال token، محتوای پنهان،
یا حذف فهرست‌ها منجر شود.

حساب‌های حذف‌شده، ممنوع‌شده یا غیرفعال نمی‌توانند از tokenهای API در ClawHub
استفاده کنند. اگر احراز هویت CLI پس از اقدام علیه حساب شروع به شکست کرد،
برای بررسی وضعیت حساب وارد UI وب شوید یا از طریق کانال پشتیبانی موردانتظار
پروژه با نگه‌دارندگان تماس بگیرید.

## راهنمای ناشر

برای کاهش مثبت‌های کاذب و بهبود اعتماد کاربران:

- نام‌ها، خلاصه‌ها، برچسب‌ها و تغییرات را دقیق نگه دارید
- متغیرهای محیطی و مجوزهای موردنیاز را اعلام کنید
- از دستورهای نصب مبهم‌سازی‌شده پرهیز کنید
- در صورت امکان به منبع لینک دهید
- پیش از انتشار Pluginها از اجرای آزمایشی استفاده کنید
- اگر کاربران یا ناظران درباره رفتار بسته پرسیدند، شفاف پاسخ دهید
