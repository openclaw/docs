Control UI یک برنامهٔ تک‌صفحه‌ای کوچک با **Vite + Lit** است که توسط Gateway سرو می‌شود:

- پیش‌فرض: `http://<host>:18789/`
- پیشوند اختیاری: `gateway.controlUi.basePath` را تنظیم کنید (مثلاً `/openclaw`)

این رابط **مستقیماً با WebSocket Gateway** روی همان پورت صحبت می‌کند.

## باز کردن سریع (محلی)

اگر Gateway روی همان رایانه در حال اجرا است، باز کنید:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (یا [http://localhost:18789/](http://localhost:18789/))

اگر صفحه بارگذاری نشد، ابتدا Gateway را شروع کنید: `openclaw gateway`.

احراز هویت هنگام دست‌دهی WebSocket از طریق موارد زیر ارائه می‌شود:

- `connect.params.auth.token`
- `connect.params.auth.password`
- سرآیندهای هویت Tailscale Serve وقتی `gateway.auth.allowTailscale: true` باشد
- سرآیندهای هویت پراکسی مورد اعتماد وقتی `gateway.auth.mode: "trusted-proxy"` باشد

پنل تنظیمات داشبورد یک توکن را برای نشست تب فعلی مرورگر و URL انتخاب‌شدهٔ gateway نگه می‌دارد؛ گذرواژه‌ها ماندگار نمی‌شوند. راه‌اندازی اولیه معمولاً در اولین اتصال، یک توکن gateway برای احراز هویت با راز مشترک تولید می‌کند، اما احراز هویت با گذرواژه نیز وقتی `gateway.auth.mode` برابر `"password"` باشد کار می‌کند.

## جفت‌سازی دستگاه (اولین اتصال)

وقتی از یک مرورگر یا دستگاه جدید به Control UI وصل می‌شوید، Gateway معمولاً به **تأیید جفت‌سازی یک‌باره** نیاز دارد. این یک اقدام امنیتی برای جلوگیری از دسترسی غیرمجاز است.

**آنچه خواهید دید:** "disconnected (1008): pairing required"

<Steps>
  <Step title="فهرست کردن درخواست‌های در انتظار">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="تأیید با شناسهٔ درخواست">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

اگر مرورگر جفت‌سازی را با جزئیات احراز هویت تغییرکرده (نقش/دامنه‌ها/کلید عمومی) دوباره امتحان کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، دوباره `openclaw devices list` را اجرا کنید.

اگر مرورگر از قبل جفت شده باشد و آن را از دسترسی خواندن به دسترسی نوشتن/مدیر تغییر دهید، این به‌عنوان ارتقای تأیید در نظر گرفته می‌شود، نه اتصال مجدد بی‌صدا. OpenClaw تأیید قدیمی را فعال نگه می‌دارد، اتصال مجدد گسترده‌تر را مسدود می‌کند و از شما می‌خواهد مجموعهٔ دامنهٔ جدید را صریحاً تأیید کنید.

پس از تأیید، دستگاه به خاطر سپرده می‌شود و دیگر به تأیید دوباره نیاز ندارد، مگر اینکه آن را با `openclaw devices revoke --device <id> --role <role>` لغو کنید. برای چرخش و لغو توکن به [CLI دستگاه‌ها](/fa/cli/devices) مراجعه کنید.

<Note>
- اتصال‌های مستقیم مرورگر از طریق local loopback (`127.0.0.1` / `localhost`) به‌طور خودکار تأیید می‌شوند.
- Tailscale Serve می‌تواند وقتی `gateway.auth.allowTailscale: true` است، هویت Tailscale تأیید می‌شود، و مرورگر هویت دستگاه خود را ارائه می‌کند، رفت‌وبرگشت جفت‌سازی را برای نشست‌های اپراتور Control UI رد کند.
- اتصال‌های مستقیم Tailnet، اتصال‌های مرورگر از LAN، و پروفایل‌های مرورگر بدون هویت دستگاه همچنان به تأیید صریح نیاز دارند.
- هر پروفایل مرورگر یک شناسهٔ دستگاه یکتا تولید می‌کند، بنابراین تغییر مرورگر یا پاک کردن داده‌های مرورگر به جفت‌سازی دوباره نیاز خواهد داشت.

</Note>

## هویت شخصی (محلیِ مرورگر)

Control UI از یک هویت شخصی برای هر مرورگر پشتیبانی می‌کند (نام نمایشی و آواتار) که برای انتساب در نشست‌های مشترک به پیام‌های خروجی پیوست می‌شود. این هویت در فضای ذخیره‌سازی مرورگر زندگی می‌کند، به پروفایل فعلی مرورگر محدود است، و به دستگاه‌های دیگر همگام‌سازی نمی‌شود یا در سمت سرور فراتر از فرادادهٔ معمول نویسندگی رونوشت روی پیام‌هایی که واقعاً ارسال می‌کنید، ماندگار نمی‌شود. پاک کردن داده‌های سایت یا تغییر مرورگر آن را به حالت خالی بازنشانی می‌کند.

همین الگوی محلیِ مرورگر برای بازنویسی آواتار دستیار نیز اعمال می‌شود. آواتارهای بارگذاری‌شدهٔ دستیار فقط در مرورگر محلی روی هویت حل‌شده توسط gateway قرار می‌گیرند و هرگز از طریق `config.patch` رفت‌وبرگشت نمی‌کنند. فیلد پیکربندی مشترک `ui.assistant.avatar` همچنان برای کلاینت‌های غیر UI که مستقیماً این فیلد را می‌نویسند در دسترس است (مانند gatewayهای اسکریپتی یا داشبوردهای سفارشی).

## نقطهٔ پایانی پیکربندی زمان اجرا

Control UI تنظیمات زمان اجرای خود را از `/__openclaw/control-ui-config.json` دریافت می‌کند. این نقطهٔ پایانی با همان احراز هویت gateway که برای بقیهٔ سطح HTTP استفاده می‌شود محافظت می‌شود: مرورگرهای احراز هویت‌نشده نمی‌توانند آن را دریافت کنند، و دریافت موفق به یک توکن/گذرواژهٔ gateway از قبل معتبر، هویت Tailscale Serve، یا هویت پراکسی مورد اعتماد نیاز دارد.

## پشتیبانی از زبان

Control UI می‌تواند هنگام اولین بارگذاری، خود را بر اساس locale مرورگر شما بومی‌سازی کند. برای بازنویسی آن بعداً، **Overview -> Gateway Access -> Language** را باز کنید. انتخابگر locale در کارت Gateway Access قرار دارد، نه زیر Appearance.

- localeهای پشتیبانی‌شده: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- ترجمه‌های غیرانگلیسی به‌صورت lazy در مرورگر بارگذاری می‌شوند.
- locale انتخاب‌شده در فضای ذخیره‌سازی مرورگر ذخیره می‌شود و در بازدیدهای بعدی دوباره استفاده می‌شود.
- کلیدهای ترجمهٔ مفقود به انگلیسی بازمی‌گردند.

ترجمه‌های مستندات برای همان مجموعهٔ locale غیرانگلیسی تولید می‌شوند، اما انتخابگر زبان داخلی سایت مستندات Mintlify به کدهای localeای محدود است که Mintlify می‌پذیرد. مستندات تایلندی (`th`) و فارسی (`fa`) همچنان در مخزن انتشار تولید می‌شوند؛ ممکن است تا وقتی Mintlify از آن کدها پشتیبانی کند در آن انتخابگر ظاهر نشوند.

## تم‌های ظاهری

پنل Appearance تم‌های داخلی Claw، Knot، و Dash را به‌همراه یک جایگاه واردسازی tweakcn محلیِ مرورگر نگه می‌دارد. برای وارد کردن یک تم، [تم‌های tweakcn](https://tweakcn.com/themes) را باز کنید، یک تم را انتخاب یا ایجاد کنید، روی **Share** کلیک کنید، و پیوند تم کپی‌شده را در Appearance جای‌گذاری کنید. واردکننده همچنین URLهای رجیستری `https://tweakcn.com/r/themes/<id>`، URLهای ویرایشگر مانند `https://tweakcn.com/editor/theme?theme=amethyst-haze`، مسیرهای نسبی `/themes/<id>`، شناسه‌های خام تم، و نام‌های پیش‌فرض تم مانند `amethyst-haze` را می‌پذیرد.

تم‌های واردشده فقط در پروفایل فعلی مرورگر ذخیره می‌شوند. آن‌ها در پیکربندی gateway نوشته نمی‌شوند و میان دستگاه‌ها همگام‌سازی نمی‌شوند. جایگزین کردن تم واردشده همان یک جایگاه محلی را به‌روزرسانی می‌کند؛ پاک کردن آن، اگر تم واردشده انتخاب شده باشد، تم فعال را به Claw برمی‌گرداند.

## رفتار چت

<AccordionGroup>
  <Accordion title="رفتار ارسال و تاریخچه">
    - `chat.send` **غیرمسدودکننده** است: بلافاصله با `{ runId, status: "started" }` تأیید می‌کند و پاسخ از طریق رویدادهای `chat` به‌صورت جریانی ارسال می‌شود.
    - آپلودهای گفت‌وگو تصاویر و فایل‌های غیر ویدیویی را می‌پذیرند. تصاویر مسیر تصویر بومی را نگه می‌دارند؛ فایل‌های دیگر به‌عنوان رسانه مدیریت‌شده ذخیره می‌شوند و در تاریخچه به‌صورت لینک‌های پیوست نمایش داده می‌شوند.
    - ارسال دوباره با همان `idempotencyKey` هنگام اجرا `{ status: "in_flight" }` و پس از تکمیل `{ status: "ok" }` برمی‌گرداند.
    - پاسخ‌های `chat.history` برای ایمنی UI محدودیت اندازه دارند. وقتی مدخل‌های رونوشت بیش از حد بزرگ باشند، Gateway ممکن است فیلدهای متنی طولانی را کوتاه کند، بلوک‌های فراداده سنگین را حذف کند، و پیام‌های بیش‌ازحد بزرگ را با یک placeholder جایگزین کند (`[chat.history omitted: message too large]`).
    - تصاویر دستیار/تولیدشده به‌عنوان ارجاع‌های رسانه مدیریت‌شده پایدار می‌شوند و از طریق URLهای رسانه احراز هویت‌شده Gateway دوباره ارائه می‌شوند، بنابراین بارگذاری دوباره به باقی ماندن payloadهای خام تصویر base64 در پاسخ تاریخچه گفت‌وگو وابسته نیست.
    - `chat.history` همچنین تگ‌های دستور درون‌خطیِ صرفاً نمایشی را از متن قابل مشاهده دستیار حذف می‌کند (برای مثال `[[reply_to_*]]` و `[[audio_as_voice]]`)، payloadهای XML فراخوانی ابزار به‌صورت متن ساده (از جمله `<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>`، و بلوک‌های کوتاه‌شده فراخوانی ابزار)، و توکن‌های کنترلی مدل از نوع ASCII/تمام‌عرض که نشت کرده‌اند را حذف می‌کند، و مدخل‌های دستیار را که کل متن قابل مشاهده‌شان فقط توکن سکوت دقیق `NO_REPLY` / `no_reply` است کنار می‌گذارد.
    - هنگام یک ارسال فعال و تازه‌سازی نهایی تاریخچه، نمای گفت‌وگو پیام‌های خوش‌بینانه محلی کاربر/دستیار را در صورتی قابل مشاهده نگه می‌دارد که `chat.history` برای مدت کوتاهی یک snapshot قدیمی‌تر برگرداند؛ وقتی تاریخچه Gateway به‌روز شد، رونوشت مرجع جایگزین آن پیام‌های محلی می‌شود.
    - `chat.inject` یک یادداشت دستیار را به رونوشت نشست اضافه می‌کند و یک رویداد `chat` را برای به‌روزرسانی‌های فقط UI پخش می‌کند (بدون اجرای عامل، بدون تحویل کانال).
    - گزیننده‌های مدل و تفکر در سربرگ گفت‌وگو، نشست فعال را بلافاصله از طریق `sessions.patch` وصله می‌کنند؛ آن‌ها overrideهای پایدار نشست هستند، نه گزینه‌های ارسال فقط برای یک نوبت.
    - تایپ کردن `/new` در رابط کاربری کنترل، همان نشست تازه داشبورد را مثل گفت‌وگوی جدید می‌سازد و به آن جابه‌جا می‌شود. تایپ کردن `/reset` بازنشانی صریح درجا توسط Gateway را برای نشست جاری نگه می‌دارد.
    - گزیننده مدل گفت‌وگو، نمای مدل پیکربندی‌شده Gateway را درخواست می‌کند. اگر `agents.defaults.models` موجود باشد، همان فهرست مجاز گزیننده را هدایت می‌کند. در غیر این صورت گزیننده مدخل‌های صریح `models.providers.*.models` به‌علاوه ارائه‌دهنده‌هایی با احراز هویت قابل استفاده را نشان می‌دهد. کاتالوگ کامل از طریق RPC دیباگ `models.list` با `view: "all"` همچنان در دسترس می‌ماند.
    - وقتی گزارش‌های تازه مصرف نشست Gateway فشار بالای context را نشان می‌دهند، ناحیه نوشتن گفت‌وگو یک اعلان context و، در سطوح توصیه‌شده Compaction، یک دکمه فشرده نشان می‌دهد که مسیر عادی Compaction نشست را اجرا می‌کند. snapshotهای قدیمی توکن تا وقتی Gateway دوباره مصرف تازه را گزارش کند پنهان می‌مانند.

  </Accordion>
  <Accordion title="حالت گفتار (بی‌درنگ در مرورگر)">
    حالت گفتار از یک ارائه‌دهنده صدای بی‌درنگ ثبت‌شده استفاده می‌کند. OpenAI را با `talk.provider: "openai"` به‌علاوه `talk.providers.openai.apiKey` پیکربندی کنید، یا Google را با `talk.provider: "google"` به‌علاوه `talk.providers.google.apiKey` پیکربندی کنید؛ پیکربندی ارائه‌دهنده بی‌درنگ Voice Call همچنان می‌تواند به‌عنوان fallback دوباره استفاده شود. مرورگر هرگز یک کلید API استاندارد ارائه‌دهنده را دریافت نمی‌کند. OpenAI یک رمز موقت کلاینت Realtime برای WebRTC دریافت می‌کند. Google Live یک توکن احراز هویت محدودشده و یک‌بارمصرف Live API برای نشست WebSocket مرورگر دریافت می‌کند، که دستورالعمل‌ها و تعریف‌های ابزار توسط Gateway در توکن قفل شده‌اند. ارائه‌دهنده‌هایی که فقط یک پل بی‌درنگ backend ارائه می‌کنند از طریق انتقال رله Gateway اجرا می‌شوند، بنابراین اعتبارنامه‌ها و socketهای vendor سمت سرور می‌مانند، در حالی که صدای مرورگر از طریق RPCهای احراز هویت‌شده Gateway جابه‌جا می‌شود. پرامپت نشست Realtime توسط Gateway ساخته می‌شود؛ `talk.realtime.session` overrideهای دستورالعمل ارائه‌شده از سوی فراخواننده را نمی‌پذیرد.

    در ناحیه نوشتن گفت‌وگو، کنترل گفتار همان دکمه موج‌ها کنار دکمه دیکته میکروفون است. وقتی گفتار شروع می‌شود، ردیف وضعیت ناحیه نوشتن ابتدا `Connecting Talk...` را نشان می‌دهد، سپس تا وقتی صدا وصل است `Talk live` را، یا وقتی یک فراخوانی ابزار بی‌درنگ از مدل بزرگ‌تر پیکربندی‌شده از طریق `chat.send` کمک می‌گیرد `Asking OpenClaw...` را نشان می‌دهد.

    تست Smoke زنده نگه‌دارنده: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` تبادل SDP مرورگر WebRTC برای OpenAI، راه‌اندازی WebSocket مرورگر با توکن محدودشده Google Live، و adapter مرورگر رله Gateway با رسانه میکروفون جعلی را بررسی می‌کند. این فرمان فقط وضعیت ارائه‌دهنده را چاپ می‌کند و secretها را لاگ نمی‌کند.

  </Accordion>
  <Accordion title="توقف و لغو">
    - روی **توقف** کلیک کنید (`chat.abort` را فراخوانی می‌کند).
    - وقتی یک اجرا فعال است، پیگیری‌های عادی در صف قرار می‌گیرند. روی **هدایت** در یک پیام صف‌شده کلیک کنید تا آن پیگیری به نوبت در حال اجرا تزریق شود.
    - برای لغو خارج از باند، `/stop` را تایپ کنید (یا عبارت‌های مستقل لغو مانند `stop`، `stop action`، `stop run`، `stop openclaw`، `please stop`).
    - `chat.abort` از `{ sessionKey }` (بدون `runId`) برای لغو همه اجراهای فعال آن نشست پشتیبانی می‌کند.

  </Accordion>
  <Accordion title="نگه‌داری خروجی جزئی پس از لغو">
    - وقتی یک اجرا لغو می‌شود، متن جزئی دستیار همچنان می‌تواند در UI نشان داده شود.
    - Gateway وقتی خروجی بافرشده موجود باشد، متن جزئی لغوشده دستیار را در تاریخچه رونوشت پایدار می‌کند.
    - مدخل‌های پایدارشده فراداده لغو را شامل می‌شوند تا مصرف‌کنندگان رونوشت بتوانند خروجی‌های جزئی لغو را از خروجی تکمیل عادی تشخیص دهند.

  </Accordion>
</AccordionGroup>

## نصب PWA و Web Push

رابط کاربری کنترل همراه با یک `manifest.webmanifest` و یک service worker عرضه می‌شود، بنابراین مرورگرهای مدرن می‌توانند آن را به‌عنوان یک PWA مستقل نصب کنند. Web Push به Gateway امکان می‌دهد PWA نصب‌شده را حتی وقتی برگه یا پنجره مرورگر باز نیست با اعلان‌ها بیدار کند.

| سطح                                                  | کارکرد                                                             |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | manifest مربوط به PWA. مرورگرها پس از در دسترس شدن آن، «نصب برنامه» را پیشنهاد می‌کنند. |
| `ui/public/sw.js`                                     | service worker که رویدادهای `push` و کلیک‌های اعلان را مدیریت می‌کند. |
| `push/vapid-keys.json` (زیر دایرکتوری وضعیت OpenClaw) | جفت‌کلید VAPID خودکار تولیدشده که برای امضای payloadهای Web Push استفاده می‌شود. |
| `push/web-push-subscriptions.json`                    | endpointهای subscription مرورگر که پایدار شده‌اند.                |

وقتی می‌خواهید کلیدها را ثابت نگه دارید (برای استقرارهای چندمیزبانه، چرخش secretها، یا تست‌ها)، جفت‌کلید VAPID را از طریق متغیرهای محیطی روی فرایند Gateway override کنید:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (پیش‌فرض `mailto:openclaw@localhost` است)

رابط کاربری کنترل از این متدهای Gateway محدودشده با دامنه دسترسی برای ثبت و تست subscriptionهای مرورگر استفاده می‌کند:

- `push.web.vapidPublicKey` — کلید عمومی VAPID فعال را می‌گیرد.
- `push.web.subscribe` — یک `endpoint` به‌علاوه `keys.p256dh`/`keys.auth` را ثبت می‌کند.
- `push.web.unsubscribe` — یک endpoint ثبت‌شده را حذف می‌کند.
- `push.web.test` — یک اعلان تست به subscription فراخواننده می‌فرستد.

<Note>
Web Push مستقل از مسیر رله APNS در iOS است (برای push متکی به رله، [پیکربندی](/fa/gateway/configuration) را ببینید) و همچنین مستقل از متد موجود `push.test` است؛ این‌ها جفت‌سازی موبایل بومی را هدف می‌گیرند.
</Note>

## جاسازی‌های میزبانی‌شده

پیام‌های دستیار می‌توانند محتوای وب میزبانی‌شده را با shortcode `[embed ...]` به‌صورت درون‌خطی رندر کنند. سیاست sandbox مربوط به iframe توسط `gateway.controlUi.embedSandbox` کنترل می‌شود:

<Tabs>
  <Tab title="strict">
    اجرای script را داخل جاسازی‌های میزبانی‌شده غیرفعال می‌کند.
  </Tab>
  <Tab title="scripts (پیش‌فرض)">
    جاسازی‌های تعاملی را مجاز می‌کند و هم‌زمان جداسازی مبدأ را نگه می‌دارد؛ این حالت پیش‌فرض است و معمولاً برای بازی‌ها/ویجت‌های مرورگری خودبسنده کافی است.
  </Tab>
  <Tab title="trusted">
    برای سندهای هم‌سایتی که عمداً به امتیازهای قوی‌تری نیاز دارند، `allow-same-origin` را روی `allow-scripts` اضافه می‌کند.
  </Tab>
</Tabs>

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
از `trusted` فقط وقتی استفاده کنید که سند جاسازی‌شده واقعاً به رفتار هم‌مبدأ نیاز دارد. برای بیشتر بازی‌های تولیدشده توسط عامل و canvasهای تعاملی، `scripts` انتخاب امن‌تری است.
</Warning>

URLهای جاسازی خارجی مطلق `http(s)` به‌طور پیش‌فرض مسدود می‌مانند. اگر عمداً می‌خواهید `[embed url="https://..."]` صفحه‌های شخص ثالث را بارگذاری کند، `gateway.controlUi.allowExternalEmbedUrls: true` را تنظیم کنید.

## پهنای پیام گفت‌وگو

پیام‌های گروه‌بندی‌شده گفت‌وگو از یک max-width پیش‌فرض خوانا استفاده می‌کنند. استقرارهای مانیتور عریض می‌توانند بدون وصله کردن CSS بسته‌بندی‌شده، آن را با تنظیم `gateway.controlUi.chatMessageMaxWidth` override کنند:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

مقدار پیش از رسیدن به مرورگر اعتبارسنجی می‌شود. مقدارهای پشتیبانی‌شده شامل طول‌ها و درصدهای ساده مانند `960px` یا `82%`، به‌علاوه عبارت‌های پهنای محدودشده `min(...)`، `max(...)`، `clamp(...)`، `calc(...)`، و `fit-content(...)` هستند.

## دسترسی به tailnet (توصیه‌شده)

<Tabs>
  <Tab title="Tailscale Serve یکپارچه (ترجیحی)">
    Gateway را روی loopback نگه دارید و اجازه دهید Tailscale Serve آن را با HTTPS پروکسی کند:

    ```bash
    openclaw gateway --tailscale serve
    ```

    باز کنید:

    - `https://<magicdns>/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    به‌طور پیش‌فرض، درخواست‌های Serve مربوط به رابط کاربری کنترل/WebSocket وقتی `gateway.auth.allowTailscale` برابر `true` باشد می‌توانند از طریق headerهای هویت Tailscale (`tailscale-user-login`) احراز هویت شوند. OpenClaw هویت را با resolve کردن نشانی `x-forwarded-for` با `tailscale whois` و تطبیق آن با header بررسی می‌کند، و این‌ها را فقط وقتی می‌پذیرد که درخواست با headerهای `x-forwarded-*` متعلق به Tailscale به loopback برسد. برای نشست‌های اپراتور رابط کاربری کنترل با هویت دستگاه مرورگر، این مسیر Serve تأییدشده رفت‌وبرگشت جفت‌سازی دستگاه را هم رد می‌کند؛ مرورگرهای بدون دستگاه و اتصال‌های با نقش node همچنان بررسی‌های عادی دستگاه را دنبال می‌کنند. اگر می‌خواهید حتی برای ترافیک Serve هم اعتبارنامه‌های shared-secret صریح لازم باشد، `gateway.auth.allowTailscale: false` را تنظیم کنید. سپس از `gateway.auth.mode: "token"` یا `"password"` استفاده کنید.

    برای آن مسیر هویت ناهمگام Serve، تلاش‌های احراز هویت ناموفق برای همان IP کلاینت و دامنه احراز هویت، پیش از نوشتن محدودیت نرخ سریالی می‌شوند. بنابراین تلاش‌های دوباره نامعتبر هم‌زمان از همان مرورگر ممکن است روی درخواست دوم به‌جای دو عدم‌تطابق ساده که موازی با هم رقابت می‌کنند، `retry later` را نشان دهند.

    <Warning>
    احراز هویت Serve بدون توکن فرض می‌کند میزبان Gateway قابل اعتماد است. اگر ممکن است کد محلی غیرقابل اعتماد روی آن میزبان اجرا شود، احراز هویت با توکن/گذرواژه را الزامی کنید.
    </Warning>

  </Tab>
  <Tab title="اتصال به tailnet + توکن">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    سپس باز کنید:

    - `http://<tailscale-ip>:18789/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    shared secret مطابق را در تنظیمات UI وارد کنید (به‌صورت `connect.params.auth.token` یا `connect.params.auth.password` ارسال می‌شود).

  </Tab>
</Tabs>

## HTTP ناامن

اگر داشبورد را روی HTTP ساده (`http://<lan-ip>` یا `http://<tailscale-ip>`) باز کنید، مرورگر در یک **زمینه غیرامن** اجرا می‌شود و WebCrypto را مسدود می‌کند. به‌طور پیش‌فرض، OpenClaw اتصال‌های رابط کاربری کنترل بدون هویت دستگاه را **مسدود** می‌کند.

استثناهای مستند:

- سازگاری HTTP ناامنِ فقط localhost با `gateway.controlUi.allowInsecureAuth=true`
- احراز هویت موفق اپراتور رابط کاربری کنترل از طریق `gateway.auth.mode: "trusted-proxy"`
- استثنای اضطراری `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**راهکار پیشنهادی:** از HTTPS (Tailscale Serve) استفاده کنید یا UI را به‌صورت محلی باز کنید:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (روی میزبان gateway)

<AccordionGroup>
  <Accordion title="رفتار تغییر وضعیت احراز هویت ناامن">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` فقط یک تغییر وضعیت سازگاری محلی است:

    - به نشست‌های رابط کاربری کنترل روی localhost اجازه می‌دهد در زمینه‌های HTTP غیرامن، بدون هویت دستگاه ادامه پیدا کنند.
    - بررسی‌های جفت‌سازی را دور نمی‌زند.
    - الزامات هویت دستگاه راه‌دور (غیر localhost) را کاهش نمی‌دهد.

  </Accordion>
  <Accordion title="فقط برای شرایط اضطراری">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` بررسی‌های هویت دستگاه رابط کاربری کنترل را غیرفعال می‌کند و یک کاهش امنیتی شدید است. پس از استفاده اضطراری، سریع آن را برگردانید.
    </Warning>

  </Accordion>
  <Accordion title="نکته پراکسی مورد اعتماد">
    - احراز هویت موفق پراکسی مورد اعتماد می‌تواند نشست‌های **اپراتور** رابط کاربری کنترل را بدون هویت دستگاه بپذیرد.
    - این موضوع به نشست‌های رابط کاربری کنترل با نقش گره گسترش نمی‌یابد.
    - پراکسی‌های معکوس loopback روی همان میزبان همچنان احراز هویت پراکسی مورد اعتماد را برآورده نمی‌کنند؛ [احراز هویت پراکسی مورد اعتماد](/fa/gateway/trusted-proxy-auth) را ببینید.

  </Accordion>
</AccordionGroup>

برای راهنمایی راه‌اندازی HTTPS، [Tailscale](/fa/gateway/tailscale) را ببینید.

## سیاست امنیت محتوا

رابط کاربری کنترل با یک سیاست سخت‌گیرانه `img-src` ارائه می‌شود: فقط دارایی‌های **هم‌مبدأ**، نشانی‌های `data:`، و نشانی‌های `blob:` تولیدشده به‌صورت محلی مجاز هستند. نشانی‌های تصویر راه‌دور `http(s)` و نشانی‌های تصویر نسبی به پروتکل توسط مرورگر رد می‌شوند و واکشی شبکه‌ای انجام نمی‌دهند.

معنای عملی این موضوع:

- آواتارها و تصویرهایی که در مسیرهای نسبی ارائه می‌شوند (برای مثال `/avatars/<id>`) همچنان نمایش داده می‌شوند، از جمله مسیرهای آواتار احرازشده که رابط کاربری آن‌ها را واکشی و به نشانی‌های محلی `blob:` تبدیل می‌کند.
- نشانی‌های درون‌خطی `data:image/...` همچنان نمایش داده می‌شوند (برای بارهای ارسالی درون پروتکل مفید است).
- نشانی‌های محلی `blob:` که رابط کاربری کنترل می‌سازد همچنان نمایش داده می‌شوند.
- نشانی‌های آواتار راه‌دور که از فراداده کانال منتشر می‌شوند، در کمک‌کننده‌های آواتار رابط کاربری کنترل حذف و با نشان‌واره/نشان داخلی جایگزین می‌شوند؛ بنابراین یک کانال آلوده یا مخرب نمی‌تواند مرورگر اپراتور را مجبور به واکشی تصویر راه‌دور دلخواه کند.

برای دریافت این رفتار نیازی نیست چیزی را تغییر دهید؛ این رفتار همیشه فعال است و قابل پیکربندی نیست.

## احراز هویت مسیر آواتار

وقتی احراز هویت Gateway پیکربندی شده باشد، نقطه پایانی آواتار رابط کاربری کنترل همان توکن Gateway را مثل بقیه API لازم دارد:

- `GET /avatar/<agentId>` تصویر آواتار را فقط به فراخوان‌های احرازشده برمی‌گرداند. `GET /avatar/<agentId>?meta=1` فراداده آواتار را با همین قاعده برمی‌گرداند.
- درخواست‌های احرازنشده به هر دو مسیر رد می‌شوند (مطابق با مسیر هم‌رده رسانه دستیار). این کار مانع نشت هویت عامل از مسیر آواتار روی میزبان‌هایی می‌شود که در غیر این صورت محافظت شده‌اند.
- خود رابط کاربری کنترل هنگام واکشی آواتارها، توکن Gateway را به‌عنوان سربرگ bearer ارسال می‌کند و از نشانی‌های blob احرازشده استفاده می‌کند تا تصویر همچنان در داشبوردها نمایش داده شود.

اگر احراز هویت Gateway را غیرفعال کنید (روی میزبان‌های اشتراکی توصیه نمی‌شود)، مسیر آواتار نیز مطابق با بقیه Gateway بدون احراز هویت می‌شود.

## ساخت رابط کاربری

Gateway فایل‌های ایستا را از `dist/control-ui` ارائه می‌کند. آن‌ها را با این دستور بسازید:

```bash
pnpm ui:build
```

پایه مطلق اختیاری (وقتی نشانی‌های ثابت دارایی می‌خواهید):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

برای توسعه محلی (سرور توسعه جداگانه):

```bash
pnpm ui:dev
```

سپس رابط کاربری را به نشانی WS مربوط به Gateway خود اشاره دهید (مثلاً `ws://127.0.0.1:18789`).

## اشکال‌زدایی/آزمایش: سرور توسعه + Gateway راه‌دور

رابط کاربری کنترل از فایل‌های ایستا تشکیل شده است؛ مقصد WebSocket قابل پیکربندی است و می‌تواند با مبدأ HTTP متفاوت باشد. این وقتی مفید است که سرور توسعه Vite را به‌صورت محلی می‌خواهید اما Gateway جای دیگری اجرا می‌شود.

<Steps>
  <Step title="سرور توسعه رابط کاربری را شروع کنید">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="با gatewayUrl باز کنید">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    احراز هویت یک‌باره اختیاری (در صورت نیاز):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="نکته‌ها">
    - `gatewayUrl` پس از بارگذاری در localStorage ذخیره و از URL حذف می‌شود.
    - اگر یک نقطه پایانی کامل `ws://` یا `wss://` را از طریق `gatewayUrl` می‌فرستید، مقدار `gatewayUrl` را URL-encode کنید تا مرورگر رشته پرس‌وجو را درست تجزیه کند.
    - هر زمان ممکن است، `token` باید از طریق قطعه URL (`#token=...`) فرستاده شود. قطعه‌ها به سرور ارسال نمی‌شوند و این کار از نشت در لاگ درخواست و Referer جلوگیری می‌کند. پارامترهای پرس‌وجوی قدیمی `?token=` همچنان برای سازگاری یک بار وارد می‌شوند، اما فقط به‌عنوان fallback، و بلافاصله پس از bootstrap حذف می‌شوند.
    - `password` فقط در حافظه نگه داشته می‌شود.
    - وقتی `gatewayUrl` تنظیم شده باشد، رابط کاربری به اعتبارنامه‌های پیکربندی یا محیط fallback نمی‌کند. `token` (یا `password`) را صریح ارائه کنید. نبود اعتبارنامه‌های صریح خطا است.
    - وقتی Gateway پشت TLS است از `wss://` استفاده کنید (Tailscale Serve، پراکسی HTTPS و غیره).
    - `gatewayUrl` فقط در پنجره سطح بالا پذیرفته می‌شود (نه embedded) تا از clickjacking جلوگیری شود.
    - استقرارهای غیر loopback رابط کاربری کنترل باید `gateway.controlUi.allowedOrigins` را به‌صورت صریح تنظیم کنند (originهای کامل). این شامل راه‌اندازی‌های توسعه راه‌دور نیز می‌شود.
    - راه‌اندازی Gateway ممکن است originهای محلی مانند `http://localhost:<port>` و `http://127.0.0.1:<port>` را از bind و پورت مؤثر زمان اجرا seed کند، اما originهای مرورگر راه‌دور همچنان به ورودی‌های صریح نیاز دارند.
    - جز برای آزمایش محلی کاملاً کنترل‌شده، از `gateway.controlUi.allowedOrigins: ["*"]` استفاده نکنید. معنایش اجازه دادن به هر origin مرورگر است، نه «مطابق‌سازی با هر میزبانی که استفاده می‌کنم».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت fallback مبدأ مبتنی بر سربرگ Host را فعال می‌کند، اما این یک حالت امنیتی خطرناک است.

  </Accordion>
</AccordionGroup>

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

جزئیات راه‌اندازی دسترسی راه‌دور: [دسترسی راه‌دور](/fa/gateway/remote).
