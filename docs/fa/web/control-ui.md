واسط کاربری Control UI یک برنامه تک‌صفحه‌ای کوچک **Vite + Lit** است که توسط Gateway سرو می‌شود:

- پیش‌فرض: `http://<host>:18789/`
- پیشوند اختیاری: `gateway.controlUi.basePath` را تنظیم کنید (مثلاً `/openclaw`)

این برنامه **مستقیماً با وب‌سوکت Gateway** روی همان پورت صحبت می‌کند.

## باز کردن سریع (محلی)

اگر Gateway روی همان رایانه در حال اجرا است، باز کنید:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (یا [http://localhost:18789/](http://localhost:18789/))

اگر صفحه بارگذاری نشد، ابتدا Gateway را راه‌اندازی کنید: `openclaw gateway`.

<Note>
در اتصال‌های LAN بومی Windows، Windows Firewall یا Group Policy مدیریت‌شده توسط سازمان همچنان می‌تواند URL اعلام‌شده LAN را مسدود کند، حتی وقتی `127.0.0.1` روی میزبان Gateway کار می‌کند. روی میزبان Windows فرمان `openclaw gateway status --deep` را اجرا کنید؛ این فرمان پورت‌های احتمالاً مسدود، ناسازگاری‌های پروفایل، و قوانین فایروال محلی را که policy ممکن است نادیده بگیرد گزارش می‌کند.
</Note>

احراز هویت هنگام دست‌دهی وب‌سوکت از طریق موارد زیر تأمین می‌شود:

- `connect.params.auth.token`
- `connect.params.auth.password`
- هدرهای هویت Tailscale Serve وقتی `gateway.auth.allowTailscale: true` باشد
- هدرهای هویت پروکسی مورد اعتماد وقتی `gateway.auth.mode: "trusted-proxy"` باشد

پنل تنظیمات داشبورد یک توکن را برای نشست زبانه فعلی مرورگر و URL Gateway انتخاب‌شده نگه می‌دارد؛ گذرواژه‌ها پایدارسازی نمی‌شوند. راه‌اندازی اولیه معمولاً در نخستین اتصال، یک توکن Gateway برای احراز هویت با shared-secret تولید می‌کند، اما وقتی `gateway.auth.mode` برابر `"password"` باشد احراز هویت با گذرواژه هم کار می‌کند.

## جفت‌سازی دستگاه (اولین اتصال)

وقتی از یک مرورگر یا دستگاه جدید به Control UI وصل می‌شوید، Gateway معمولاً به **تأیید یک‌باره جفت‌سازی** نیاز دارد. این یک اقدام امنیتی برای جلوگیری از دسترسی غیرمجاز است.

**آنچه خواهید دید:** "disconnected (1008): pairing required"

<Steps>
  <Step title="فهرست کردن درخواست‌های در انتظار">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="تأیید با شناسه درخواست">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

اگر مرورگر با جزئیات احراز هویت تغییرکرده (نقش/دامنه‌ها/کلید عمومی) دوباره برای جفت‌سازی تلاش کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، دوباره `openclaw devices list` را اجرا کنید.

اگر مرورگر از قبل جفت شده باشد و آن را از دسترسی خواندن به دسترسی نوشتن/مدیر تغییر دهید، این مورد به‌عنوان ارتقای تأیید در نظر گرفته می‌شود، نه اتصال مجدد بی‌صدا. OpenClaw تأیید قدیمی را فعال نگه می‌دارد، اتصال مجدد گسترده‌تر را مسدود می‌کند، و از شما می‌خواهد مجموعه دامنه جدید را صریحاً تأیید کنید.

پس از تأیید، دستگاه به خاطر سپرده می‌شود و دیگر به تأیید مجدد نیاز ندارد، مگر اینکه آن را با `openclaw devices revoke --device <id> --role <role>` لغو کنید. برای چرخش و لغو توکن، [CLI دستگاه‌ها](/fa/cli/devices) را ببینید.

عامل‌های Paperclip که از طریق آداپتر `openclaw_gateway` وصل می‌شوند از همان جریان تأیید اجرای اول استفاده می‌کنند. پس از تلاش اتصال اولیه، `openclaw devices approve --latest` را اجرا کنید تا درخواست در انتظار را پیش‌نمایش کنید، سپس فرمان چاپ‌شده `openclaw devices approve <requestId>` را دوباره اجرا کنید تا آن را تأیید کنید. برای Gateway راه‌دور، مقادیر صریح `--url` و `--token` را پاس دهید. برای پایدار نگه داشتن تأییدها در میان راه‌اندازی‌های مجدد، در Paperclip یک `adapterConfig.devicePrivateKeyPem` پایدار پیکربندی کنید، به‌جای اینکه اجازه دهید در هر اجرا یک هویت دستگاه گذرای جدید تولید کند.

<Note>
- اتصال‌های مرورگر local loopback مستقیم (`127.0.0.1` / `localhost`) به‌صورت خودکار تأیید می‌شوند.
- Tailscale Serve می‌تواند رفت‌وبرگشت جفت‌سازی را برای نشست‌های اپراتور Control UI رد کند، وقتی `gateway.auth.allowTailscale: true` باشد، هویت Tailscale تأیید شود، و مرورگر هویت دستگاه خود را ارائه کند.
- اتصال‌های مستقیم Tailnet، اتصال‌های مرورگر LAN، و پروفایل‌های مرورگر بدون هویت دستگاه همچنان به تأیید صریح نیاز دارند.
- هر پروفایل مرورگر یک شناسه دستگاه یکتا تولید می‌کند، بنابراین تغییر مرورگرها یا پاک کردن داده‌های مرورگر به جفت‌سازی دوباره نیاز خواهد داشت.

</Note>

## جفت‌سازی یک دستگاه موبایل

یک مدیر از پیش جفت‌شده می‌تواند QR اتصال iOS/Android را بدون
باز کردن ترمینال بسازد:

<Steps>
  <Step title="باز کردن جفت‌سازی موبایل">
    **گره‌ها** را انتخاب کنید، سپس در کارت **دستگاه‌ها** روی **جفت‌سازی دستگاه موبایل** کلیک کنید.
  </Step>
  <Step title="اتصال تلفن">
    در برنامه موبایل OpenClaw، **تنظیمات** → **Gateway** را باز کنید و کد QR را
    اسکن کنید. همچنین می‌توانید کد راه‌اندازی را کپی و جای‌گذاری کنید.
  </Step>
  <Step title="تأیید اتصال">
    برنامه رسمی iOS/Android به‌صورت خودکار وصل می‌شود. اگر **دستگاه‌ها** یک
    درخواست در انتظار نشان می‌دهد، پیش از تأیید، نقش و دامنه‌های آن را بررسی کنید.
  </Step>
</Steps>

ساخت یک کد راه‌اندازی به `operator.admin` نیاز دارد؛ این دکمه برای
نشست‌هایی که آن را ندارند غیرفعال است. یک کد راه‌اندازی شامل اعتبارنامه bootstrap کوتاه‌عمر است،
پس تا زمانی که معتبر هستند با QR و کد کپی‌شده مانند گذرواژه رفتار کنید. برای جفت‌سازی راه‌دور،
Gateway باید به `wss://` resolve شود (برای مثال، از طریق Tailscale
Serve/Funnel)؛ `ws://` ساده به آدرس‌های loopback و LAN خصوصی محدود است.
برای جزئیات کامل امنیت و fallback، [جفت‌سازی](/fa/channels/pairing#pair-from-the-control-ui-recommended) را ببینید.

## هویت شخصی (محلی مرورگر)

Control UI از یک هویت شخصی برای هر مرورگر (نام نمایشی و آواتار) پشتیبانی می‌کند که برای انتساب در نشست‌های مشترک به پیام‌های خروجی پیوست می‌شود. این هویت در فضای ذخیره‌سازی مرورگر زندگی می‌کند، به پروفایل فعلی مرورگر محدود است، و با دستگاه‌های دیگر همگام‌سازی نمی‌شود یا در سمت سرور فراتر از فراداده معمول نویسندگی transcript روی پیام‌هایی که واقعاً ارسال می‌کنید پایدار نمی‌ماند. پاک کردن داده‌های سایت یا تغییر مرورگرها آن را به حالت خالی بازنشانی می‌کند.

همین الگوی محلی مرورگر برای override آواتار دستیار نیز اعمال می‌شود. آواتارهای دستیار بارگذاری‌شده فقط در مرورگر محلی روی هویت resolve‌شده توسط Gateway قرار می‌گیرند و هرگز از طریق `config.patch` رفت‌وبرگشت نمی‌کنند. فیلد پیکربندی مشترک `ui.assistant.avatar` همچنان برای کلاینت‌های غیر UI که فیلد را مستقیماً می‌نویسند در دسترس است (مانند Gatewayهای اسکریپت‌شده یا داشبوردهای سفارشی).

## نقطه پایانی پیکربندی زمان اجرا

Control UI تنظیمات زمان اجرای خود را از `/control-ui-config.json` دریافت می‌کند، که نسبت به مسیر پایه Control UI مربوط به Gateway resolve می‌شود (برای مثال `/__openclaw__/control-ui-config.json` وقتی UI زیر `/__openclaw__/` سرو می‌شود). آن نقطه پایانی با همان احراز هویت Gateway مانند بقیه سطح HTTP محافظت می‌شود: مرورگرهای احراز هویت‌نشده نمی‌توانند آن را دریافت کنند، و دریافت موفق به یک توکن/گذرواژه Gateway از قبل معتبر، هویت Tailscale Serve، یا هویت پروکسی مورد اعتماد نیاز دارد.

## پشتیبانی زبان

Control UI می‌تواند در اولین بارگذاری بر اساس locale مرورگر شما خودش را محلی‌سازی کند. برای override کردن آن در آینده، **نمای کلی -> دسترسی Gateway -> زبان** را باز کنید. انتخابگر locale در کارت دسترسی Gateway قرار دارد، نه زیر ظاهر.

- localeهای پشتیبانی‌شده: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- ترجمه‌های غیرانگلیسی در مرورگر به‌صورت lazy-loaded بارگذاری می‌شوند.
- locale انتخاب‌شده در فضای ذخیره‌سازی مرورگر ذخیره می‌شود و در بازدیدهای آینده دوباره استفاده می‌شود.
- کلیدهای ترجمه موجودنبودن به انگلیسی fallback می‌کنند.

ترجمه‌های مستندات برای همان مجموعه locale غیرانگلیسی تولید می‌شوند، اما انتخابگر زبان داخلی سایت مستندات Mintlify به کدهای localeای محدود است که Mintlify می‌پذیرد. مستندات تایلندی (`th`) و فارسی (`fa`) همچنان در مخزن انتشار تولید می‌شوند؛ ممکن است تا زمانی که Mintlify از آن کدها پشتیبانی کند در آن انتخابگر ظاهر نشوند.

## تم‌های ظاهر

پنل ظاهر تم‌های داخلی Claw، Knot، و Dash را نگه می‌دارد، به‌علاوه یک جایگاه واردسازی tweakcn محلی مرورگر. برای وارد کردن یک تم، [ویرایشگر tweakcn](https://tweakcn.com/editor/theme) را باز کنید، یک تم انتخاب یا ایجاد کنید، روی **اشتراک‌گذاری** کلیک کنید، و لینک تم کپی‌شده را در ظاهر جای‌گذاری کنید. واردکننده همچنین URLهای رجیستری `https://tweakcn.com/r/themes/<id>`، URLهای ویرایشگر مانند `https://tweakcn.com/editor/theme?theme=amethyst-haze`، مسیرهای نسبی `/themes/<id>`، شناسه‌های خام تم، و نام‌های تم پیش‌فرض مانند `amethyst-haze` را می‌پذیرد.

ظاهر همچنین شامل یک تنظیم اندازه متن محلی مرورگر است. این تنظیم همراه با بقیه ترجیحات Control UI ذخیره می‌شود، روی متن چت، متن composer، کارت‌های ابزار، و نوارهای کناری چت اعمال می‌شود، و ورودی‌های متن را دست‌کم 16px نگه می‌دارد تا Safari موبایل هنگام فوکوس به‌صورت خودکار زوم نکند.

تم‌های واردشده فقط در پروفایل فعلی مرورگر ذخیره می‌شوند. آن‌ها در پیکربندی Gateway نوشته نمی‌شوند و میان دستگاه‌ها همگام‌سازی نمی‌شوند. جایگزین کردن تم واردشده همان یک جایگاه محلی را به‌روزرسانی می‌کند؛ پاک کردن آن اگر تم واردشده انتخاب شده باشد، تم فعال را به Claw برمی‌گرداند.

  </Accordion>

    - نوشتن‌ها شامل یک محافظ هش پایه هستند تا از بازنویسی و خراب کردن ویرایش‌های هم‌زمان جلوگیری شود.
    - نوشتن‌ها (`config.set`/`config.apply`/`config.patch`) پیش از اجرا، رفع SecretRefهای فعال را برای ارجاع‌های موجود در بار پیکربندی ارسال‌شده بررسی می‌کنند؛ ارجاع‌های فعال ارسال‌شده که رفع نشوند، پیش از نوشتن رد می‌شوند.
    - ذخیره‌های فرم، جای‌نگهدارهای ویرایش‌شده قدیمی را که از پیکربندی ذخیره‌شده قابل بازیابی نیستند دور می‌ریزند، در حالی که مقدارهای ویرایش‌شده‌ای را که هنوز به رازهای ذخیره‌شده نگاشت می‌شوند حفظ می‌کنند.
    - طرح‌واره + رندر فرم (`config.schema` / `config.schema.lookup`، شامل فیلدهای `title` / `description`، راهنمایی‌های UI مطابق، خلاصه‌های فرزند بلافاصله، فراداده مستندات روی گره‌های شیء تودرتو/وایلدکارت/آرایه/ترکیب، به‌علاوه طرح‌واره‌های Plugin + کانال در صورت موجود بودن)؛ ویرایشگر JSON خام فقط زمانی در دسترس است که نماگرفت یک رفت‌وبرگشت خام امن داشته باشد.
    - اگر یک نماگرفت نتواند متن خام را با ایمنی رفت‌وبرگشت کند، رابط کاربری کنترل حالت فرم را اجباری می‌کند و حالت خام را برای آن نماگرفت غیرفعال می‌کند.
    - گزینه «بازنشانی به ذخیره‌شده» در ویرایشگر JSON خام، به‌جای رندر دوباره یک نماگرفت تخت‌شده، شکل خام نوشته‌شده را حفظ می‌کند (قالب‌بندی، نظرها، چیدمان `$include`)؛ بنابراین وقتی نماگرفت بتواند با ایمنی رفت‌وبرگشت کند، ویرایش‌های بیرونی پس از بازنشانی نیز باقی می‌مانند.
    - مقدارهای شیء ساخت‌یافته SecretRef در ورودی‌های متنی فرم به‌صورت فقط‌خواندنی رندر می‌شوند تا از خراب شدن تصادفی شیء به رشته جلوگیری شود.

  </Accordion>
  <Accordion title="اشکال‌زدایی، گزارش‌ها، به‌روزرسانی">
    - اشکال‌زدایی: نماگرفت‌های وضعیت/سلامت/مدل‌ها + گزارش رویداد + فراخوانی‌های دستی RPC (`status`، `health`، `models.list`).
    - گزارش رویداد شامل زمان‌بندی‌های تازه‌سازی/RPC رابط کاربری کنترل، زمان‌بندی‌های کند رندر چت/پیکربندی، و ورودی‌های پاسخ‌گویی مرورگر برای فریم‌های انیمیشن طولانی یا وظایف طولانی است، زمانی که مرورگر آن نوع‌های ورودی PerformanceObserver را ارائه کند.
    - گزارش‌ها: دنبال‌کردن زنده گزارش‌های فایل gateway با فیلتر/خروجی‌گیری (`logs.tail`).
    - به‌روزرسانی: اجرای به‌روزرسانی بسته/git + راه‌اندازی دوباره (`update.run`) همراه با گزارش راه‌اندازی دوباره، سپس نظرسنجی `update.status` پس از اتصال دوباره برای تأیید نسخه Gateway در حال اجرا.

  </Accordion>
  <Accordion title="نکته‌های پنل کارهای Cron">
    - برای کارهای ایزوله، تحویل به‌طور پیش‌فرض اعلام خلاصه است. اگر اجراهای فقط داخلی می‌خواهید، می‌توانید آن را به هیچ تغییر دهید.
    - فیلدهای کانال/هدف وقتی اعلام انتخاب شده باشد ظاهر می‌شوند.
    - حالت Webhook از `delivery.mode = "webhook"` استفاده می‌کند و `delivery.to` روی یک URL معتبر webhook از نوع HTTP(S) تنظیم می‌شود.
    - برای کارهای نشست اصلی، حالت‌های تحویل webhook و هیچ در دسترس هستند.
    - کنترل‌های ویرایش پیشرفته شامل حذف پس از اجرا، پاک‌کردن بازنویسی عامل، گزینه‌های دقیق/پلکانی cron، بازنویسی‌های مدل/تفکر عامل، و سوییچ‌های تحویل با بهترین تلاش هستند.
    - اعتبارسنجی فرم درون‌خطی و همراه با خطاهای سطح فیلد است؛ مقدارهای نامعتبر دکمه ذخیره را تا زمان اصلاح غیرفعال می‌کنند.
    - برای ارسال یک توکن bearer اختصاصی، `cron.webhookToken` را تنظیم کنید؛ اگر حذف شود، webhook بدون سرآیند احراز هویت ارسال می‌شود.
    - جایگزین منسوخ: برای مهاجرت کارهای قدیمی ذخیره‌شده با `notify: true` از `cron.webhook` به webhook صریح برای هر کار یا تحویل تکمیل، `openclaw doctor --fix` را اجرا کنید.

## صفحه MCP

صفحه اختصاصی MCP یک نمای اپراتور برای سرورهای MCP مدیریت‌شده توسط OpenClaw زیر `mcp.servers` است. این صفحه خودش ترابردهای MCP را شروع نمی‌کند؛ از آن برای بررسی و ویرایش پیکربندی ذخیره‌شده استفاده کنید، سپس وقتی به اثبات زنده سرور نیاز دارید از `openclaw mcp doctor --probe` استفاده کنید.

گردش کار معمول:

1. **MCP** را از نوار کناری باز کنید.
2. کارت‌های خلاصه را برای شمار کل سرورها، سرورهای فعال، OAuth، و تعداد سرورهای فیلترشده بررسی کنید.
3. هر ردیف سرور را از نظر ترابرد، فعال‌سازی، احراز هویت، فیلترها، زمان‌پایان‌ها، و راهنمای فرمان‌ها مرور کنید.
4. وقتی سروری باید پیکربندی‌شده باقی بماند اما از کشف زمان اجرا بیرون بماند، فعال‌سازی را تغییر دهید.
5. بخش پیکربندی `mcp` با دامنه محدود را برای تعریف‌های سرور، سرآیندها، مسیرهای TLS/mTLS، فراداده OAuth، فیلترهای ابزار، و فراداده تصویرسازی Codex ویرایش کنید.
6. برای نوشتن پیکربندی از **ذخیره** استفاده کنید، یا وقتی Gateway در حال اجرا باید پیکربندی تغییرکرده را اعمال کند از **ذخیره و انتشار** استفاده کنید.
7. وقتی فرایند ویرایش‌شده به عیب‌یابی ایستا، اثبات زنده، یا دور انداختن زمان‌اجرای کش‌شده نیاز دارد، `openclaw mcp status --verbose`، `openclaw mcp doctor --probe`، یا `openclaw mcp reload` را از پایانه اجرا کنید.

این صفحه مقدارهای URLمانند دارای اعتبارنامه را پیش از رندر ویرایش می‌کند و نام سرورها را در قطعه‌های فرمان نقل‌قول‌گذاری می‌کند تا فرمان‌های کپی‌شده همچنان با فاصله‌ها یا نویسه‌های ویژه shell کار کنند. مرجع کامل CLI و پیکربندی در [MCP](/fa/cli/mcp) قرار دارد.

## زبانه فعالیت

زبانه فعالیت یک ناظر موقتی و محلیِ مرورگر برای فعالیت زنده ابزار است. این زبانه از همان جریان رویداد `session.tool` / ابزار Gateway مشتق می‌شود که کارت‌های ابزار چت را تغذیه می‌کند؛ هیچ خانواده رویداد Gateway، نقطه پایانی، ذخیره پایدار فعالیت، خوراک معیارها، یا جریان ناظر بیرونی دیگری اضافه نمی‌کند.

ورودی‌های فعالیت فقط خلاصه‌های پاک‌سازی‌شده و پیش‌نمایش‌های خروجی ویرایش‌شده و کوتاه‌شده را نگه می‌دارند. مقدارهای آرگومان ابزار در وضعیت فعالیت ذخیره نمی‌شوند؛ UI نشان می‌دهد که آرگومان‌ها پنهان هستند و فقط تعداد فیلدهای آرگومان را ثبت می‌کند. فهرست درون‌حافظه‌ای از زبانه فعلی مرورگر پیروی می‌کند، در پیمایش داخل رابط کاربری کنترل باقی می‌ماند، و با بارگذاری دوباره صفحه، تغییر نشست، یا **پاک‌کردن** بازنشانی می‌شود.

  <Accordion title="معناشناسی ارسال و تاریخچه">
    - `chat.send` **غیرمسدودکننده** است: بلافاصله با `{ runId, status: "started" }` تأیید می‌کند و پاسخ از طریق رویدادهای `chat` جریان می‌یابد. کلاینت‌های مورد اعتماد رابط کاربری کنترل ممکن است فراداده اختیاری زمان‌بندی ACK را نیز برای عیب‌یابی محلی دریافت کنند.
    - بارگذاری‌های چت، تصویرها و فایل‌های غیر ویدئویی را می‌پذیرند. تصویرها مسیر تصویر بومی را حفظ می‌کنند؛ فایل‌های دیگر به‌عنوان رسانه مدیریت‌شده ذخیره می‌شوند و در تاریخچه به‌صورت پیوندهای پیوست نمایش داده می‌شوند.
    - ارسال دوباره با همان `idempotencyKey` در زمان اجرا `{ status: "in_flight" }` و پس از تکمیل `{ status: "ok" }` برمی‌گرداند.
    - پاسخ‌های `chat.history` برای ایمنی UI از نظر اندازه محدود هستند. وقتی ورودی‌های رونوشت بیش از حد بزرگ باشند، Gateway ممکن است فیلدهای متنی بلند را کوتاه کند، بلوک‌های فراداده سنگین را حذف کند، و پیام‌های بیش‌ازحد بزرگ را با یک جای‌نگهدار جایگزین کند (`[chat.history omitted: message too large]`).
    - وقتی یک پیام قابل مشاهده دستیار در `chat.history` کوتاه شده باشد، خواننده کناری می‌تواند ورودی کامل رونوشت نرمال‌شده برای نمایش را در صورت نیاز از طریق `chat.message.get` با `sessionKey`، `agentId` فعال در صورت نیاز، و `messageId` رونوشت دریافت کند. اگر Gateway همچنان نتواند مقدار بیشتری برگرداند، خواننده به‌جای تکرار بی‌صدای پیش‌نمایش کوتاه‌شده، یک وضعیت صریحِ در دسترس نبودن نشان می‌دهد.
    - تصویرهای دستیار/تولیدشده به‌صورت ارجاع‌های رسانه مدیریت‌شده پایدار می‌شوند و از طریق URLهای رسانه احراز هویت‌شده Gateway بازگردانده می‌شوند؛ بنابراین بارگذاری‌های دوباره به باقی ماندن بارهای تصویر base64 خام در پاسخ تاریخچه چت وابسته نیستند.
    - هنگام رندر `chat.history`، رابط کاربری کنترل برچسب‌های دستور درون‌خطیِ فقط‌نمایشی را از متن قابل مشاهده دستیار حذف می‌کند (برای نمونه `[[reply_to_*]]` و `[[audio_as_voice]]`)، بارهای XML فراخوانی ابزار به‌صورت متن ساده (شامل `<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>`، و بلوک‌های فراخوانی ابزار کوتاه‌شده)، و توکن‌های کنترل مدل ASCII/تمام‌عرض نشت‌کرده را نیز حذف می‌کند، و ورودی‌های دستیار را که کل متن قابل مشاهده آن‌ها فقط توکن بی‌صدای دقیق `NO_REPLY` / `no_reply` یا توکن تأیید Heartbeat یعنی `HEARTBEAT_OK` است کنار می‌گذارد.
    - هنگام یک ارسال فعال و تازه‌سازی نهایی تاریخچه، نمای چت پیام‌های کاربر/دستیار خوش‌بینانه محلی را قابل مشاهده نگه می‌دارد اگر `chat.history` برای مدت کوتاهی یک نماگرفت قدیمی‌تر برگرداند؛ وقتی تاریخچه Gateway به‌روز شد، رونوشت رسمی آن پیام‌های محلی را جایگزین می‌کند.
    - رویدادهای زنده `chat` وضعیت تحویل هستند، در حالی که `chat.history` از رونوشت پایدار نشست بازسازی می‌شود. پس از رویدادهای نهایی ابزار، رابط کاربری کنترل تاریخچه را دوباره بارگذاری می‌کند و فقط یک دنباله کوچک خوش‌بینانه را ادغام می‌کند؛ مرز رونوشت در [WebChat](/fa/web/webchat) مستند شده است.
    - `chat.inject` یک یادداشت دستیار را به رونوشت نشست اضافه می‌کند و یک رویداد `chat` را برای به‌روزرسانی‌های فقط UI پخش می‌کند (بدون اجرای عامل، بدون تحویل کانال).
    - نوار کناری نشست‌های اخیر را با اقدام نشست جدید، پیوند همه نشست‌ها، و دکمه جست‌وجوی نشست فهرست می‌کند که انتخابگر کامل نشست را باز می‌کند (محدود به عامل انتخاب‌شده، همراه با جست‌وجو و صفحه‌بندی). تغییر عامل‌ها فقط نشست‌های مرتبط با آن عامل را نشان می‌دهد و وقتی هنوز هیچ نشست داشبورد ذخیره‌شده‌ای ندارد، به نشست اصلی همان عامل برمی‌گردد.
    - در عرض‌های دسکتاپ، کنترل‌های چت روی یک ردیف فشرده می‌مانند و هنگام پیمایش به پایین رونوشت جمع می‌شوند؛ پیمایش به بالا، بازگشت به ابتدای صفحه، یا رسیدن به انتها کنترل‌ها را بازمی‌گرداند.
    - پیام‌های متنیِ تکراریِ پشت‌سرهم به‌صورت یک حباب با نشان تعداد رندر می‌شوند. پیام‌هایی که تصویر، پیوست، خروجی ابزار، یا پیش‌نمایش canvas دارند جمع نمی‌شوند.
    - انتخابگرهای مدل و تفکر در سربرگ چت، نشست فعال را بلافاصله از طریق `sessions.patch` وصله می‌کنند؛ آن‌ها بازنویسی‌های پایدار نشست هستند، نه گزینه‌های ارسال فقط برای یک نوبت.
    - اگر در حالی پیامی ارسال کنید که تغییر انتخابگر مدل برای همان نشست هنوز در حال ذخیره است، composer پیش از فراخوانی `chat.send` منتظر آن وصله نشست می‌ماند تا ارسال از مدل انتخاب‌شده استفاده کند.
    - تایپ کردن `/new` در رابط کاربری کنترل، همان نشست تازه داشبورد را مانند چت جدید می‌سازد و به آن جابه‌جا می‌شود، مگر وقتی `session.dmScope: "main"` پیکربندی شده باشد و والد فعلی نشست اصلی عامل باشد؛ در آن حالت، نشست اصلی را درجا بازنشانی می‌کند. تایپ کردن `/reset` بازنشانی درجا و صریح Gateway را برای نشست فعلی نگه می‌دارد.
    - انتخابگر مدل چت نمای مدل پیکربندی‌شده Gateway را درخواست می‌کند. اگر `agents.defaults.models` موجود باشد، آن فهرست مجاز انتخابگر را هدایت می‌کند، شامل ورودی‌های `provider/*` که کاتالوگ‌های محدود به ارائه‌دهنده را پویا نگه می‌دارند. در غیر این صورت انتخابگر ورودی‌های صریح `models.providers.*.models` به‌علاوه ارائه‌دهنده‌های دارای احراز هویت قابل استفاده را نشان می‌دهد. کاتالوگ کامل از طریق RPC اشکال‌زدایی `models.list` با `view: "all"` در دسترس می‌ماند.
    - وقتی گزارش‌های تازه استفاده نشست Gateway شامل توکن‌های زمینه فعلی باشند، نوار ابزار composer چت یک حلقه کوچک استفاده از زمینه را با درصد استفاده‌شده نشان می‌دهد؛ جزئیات کامل توکن در tooltip آن قرار دارد. حلقه در فشار بالای زمینه به سبک هشدار تغییر می‌کند و در سطوح توصیه‌شده Compaction، یک دکمه فشرده نشان می‌دهد که مسیر عادی Compaction نشست را اجرا می‌کند. نماگرفت‌های قدیمی توکن تا وقتی Gateway دوباره استفاده تازه را گزارش کند پنهان می‌مانند.

  </Accordion>
  <Accordion title="حالت گفت‌وگو (بلادرنگ مرورگر)">
    حالت گفت‌وگو از یک ارائه‌دهنده صدای بلادرنگ ثبت‌شده استفاده می‌کند. OpenAI را با `talk.realtime.provider: "openai"` به‌علاوه یک پروفایل احراز هویت کلید API از نوع `openai`، `talk.realtime.providers.openai.apiKey`، یا `OPENAI_API_KEY` پیکربندی کنید؛ پروفایل‌های OAuth در OpenAI صدای Realtime را پیکربندی نمی‌کنند. Google را با `talk.realtime.provider: "google"` به‌علاوه `talk.realtime.providers.google.apiKey` پیکربندی کنید. مرورگر هرگز یک کلید API استاندارد ارائه‌دهنده را دریافت نمی‌کند. OpenAI یک راز موقتی کلاینت Realtime برای WebRTC دریافت می‌کند. Google Live یک توکن احراز هویت محدود و یک‌بارمصرف Live API برای نشست WebSocket مرورگر دریافت می‌کند، همراه با دستورالعمل‌ها و اعلان‌های ابزار که توسط Gateway در توکن قفل شده‌اند. ارائه‌دهنده‌هایی که فقط یک پل بلادرنگ پشتیبان ارائه می‌کنند از طریق ترابرد relay در Gateway اجرا می‌شوند؛ بنابراین اعتبارنامه‌ها و سوکت‌های فروشنده سمت سرور می‌مانند، در حالی که صدای مرورگر از طریق RPCهای احراز هویت‌شده Gateway جابه‌جا می‌شود. اعلان نشست Realtime توسط Gateway مونتاژ می‌شود؛ `talk.client.create` بازنویسی دستورالعمل ارائه‌شده از سوی فراخوان را نمی‌پذیرد.

    نگارندهٔ چت شامل دکمهٔ گزینه‌های گفت‌وگو در کنار دکمهٔ شروع/توقف گفت‌وگو است. این گزینه‌ها روی نشست گفت‌وگوی بعدی اعمال می‌شوند و می‌توانند ارائه‌دهنده، انتقال، مدل، صدا، میزان تلاش استدلال، آستانهٔ VAD، مدت سکوت، و padding پیشوند را بازنویسی کنند. وقتی گزینه‌ای خالی باشد، Gateway در صورت وجود از پیش‌فرض‌های پیکربندی‌شده یا پیش‌فرض ارائه‌دهنده استفاده می‌کند. انتخاب رلهٔ Gateway مسیر رلهٔ backend را اجبار می‌کند؛ انتخاب WebRTC نشست را در مالکیت client نگه می‌دارد و اگر ارائه‌دهنده نتواند نشست مرورگر ایجاد کند، به‌جای fallback بی‌صدا به رله، با خطا متوقف می‌شود.

    در نگارندهٔ چت، کنترل گفت‌وگو همان دکمهٔ موج‌ها در کنار دکمهٔ دیکتهٔ میکروفون است. وقتی گفت‌وگو شروع می‌شود، ردیف وضعیت نگارنده ابتدا `Connecting Talk...` را نشان می‌دهد، سپس هنگام اتصال صدا `Talk live` را، یا وقتی یک فراخوانی ابزار realtime از طریق `talk.client.toolCall` با مدل بزرگ‌تر پیکربندی‌شده مشورت می‌کند، `Asking OpenClaw...` را نشان می‌دهد.

    smoke زندهٔ نگه‌دارنده: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` پل WebSocket backend مربوط به OpenAI، تبادل SDP مرورگر WebRTC مربوط به OpenAI، راه‌اندازی WebSocket مرورگر Google Live با token محدود، و adapter مرورگر رلهٔ Gateway با رسانهٔ میکروفون جعلی را راستی‌آزمایی می‌کند. این فرمان فقط وضعیت ارائه‌دهنده را چاپ می‌کند و secretها را log نمی‌کند.

  </Accordion>
  <Accordion title="توقف و لغو">
    - روی **توقف** کلیک کنید (`chat.abort` را فراخوانی می‌کند).
    - وقتی یک اجرا فعال است، follow-upهای معمولی در صف قرار می‌گیرند. روی **هدایت** در یک پیام صف‌شده کلیک کنید تا آن follow-up را به نوبت در حال اجرا تزریق کنید.
    - برای لغو خارج از باند، `/stop` را تایپ کنید (یا عبارت‌های لغو مستقل مانند `stop`، `stop action`، `stop run`، `stop openclaw`، `please stop`).
    - `chat.abort` از `{ sessionKey }` (بدون `runId`) برای لغو همهٔ اجراهای فعال آن نشست پشتیبانی می‌کند.

  </Accordion>
  <Accordion title="نگه‌داری partial پس از لغو">
    - وقتی یک اجرا لغو می‌شود، متن partial دستیار همچنان می‌تواند در UI نمایش داده شود.
    - Gateway وقتی خروجی بافرشده وجود داشته باشد، متن partial دستیارِ لغوشده را در تاریخچهٔ transcript پایدار می‌کند.
    - ورودی‌های پایدارشده شامل metadata لغو هستند تا مصرف‌کنندگان transcript بتوانند partialهای لغو را از خروجی تکمیل عادی تشخیص دهند.

  </Accordion>
</AccordionGroup>

## نصب PWA و Web Push

رابط کاربری کنترل همراه با یک `manifest.webmanifest` و یک service worker ارائه می‌شود، بنابراین مرورگرهای مدرن می‌توانند آن را به‌عنوان یک PWA مستقل نصب کنند. Web Push به Gateway اجازه می‌دهد PWA نصب‌شده را حتی وقتی tab یا پنجرهٔ مرورگر باز نیست، با notificationها بیدار کند.

اگر صفحه درست پس از به‌روزرسانی OpenClaw پیام **ناسازگاری protocol** را نشان داد، ابتدا dashboard را با `openclaw dashboard` دوباره باز کنید و صفحه را hard-refresh کنید. اگر همچنان شکست خورد، site data مربوط به origin داشبورد را پاک کنید یا در پنجرهٔ مرورگر خصوصی آزمایش کنید؛ یک tab قدیمی یا cache سرویس‌ورکر مرورگر می‌تواند همچنان bundle رابط کاربری کنترلِ پیش از به‌روزرسانی را در برابر Gateway جدیدتر اجرا کند.

| سطح                                                   | کاری که انجام می‌دهد                                                |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | manifest PWA. مرورگرها پس از دسترس‌پذیر شدن آن، گزینهٔ "Install app" را پیشنهاد می‌کنند. |
| `ui/public/sw.js`                                     | service worker که رویدادهای `push` و کلیک‌های notification را مدیریت می‌کند. |
| `push/vapid-keys.json` (زیر dir وضعیت OpenClaw) | keypair خودکار VAPID که برای امضای payloadهای Web Push استفاده می‌شود. |
| `push/web-push-subscriptions.json`                    | endpointهای subscription مرورگر که پایدار شده‌اند.                 |

وقتی می‌خواهید کلیدها را pin کنید (برای استقرارهای چند-host، چرخش secretها، یا تست‌ها)، keypair VAPID را از طریق env varها روی process مربوط به Gateway بازنویسی کنید:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (پیش‌فرض: `https://openclaw.ai`)

رابط کاربری کنترل از این methodهای Gateway با scope محدود برای ثبت و آزمایش subscriptionهای مرورگر استفاده می‌کند:

- `push.web.vapidPublicKey` — کلید عمومی VAPID فعال را دریافت می‌کند.
- `push.web.subscribe` — یک `endpoint` به‌همراه `keys.p256dh`/`keys.auth` ثبت می‌کند.
- `push.web.unsubscribe` — یک endpoint ثبت‌شده را حذف می‌کند.
- `push.web.test` — یک notification آزمایشی به subscription فراخواننده می‌فرستد.

<Note>
Web Push مستقل از مسیر رلهٔ APNS در iOS است (برای push مبتنی بر رله، [پیکربندی](/fa/gateway/configuration) را ببینید) و همچنین مستقل از method موجود `push.test` است که هدف آن pairing موبایل native است.
</Note>

## embedهای میزبانی‌شده

پیام‌های دستیار می‌توانند محتوای وب میزبانی‌شده را به‌صورت inline با shortcode `[embed ...]` رندر کنند. سیاست sandbox iframe با `gateway.controlUi.embedSandbox` کنترل می‌شود:

<Tabs>
  <Tab title="strict">
    اجرای script را داخل embedهای میزبانی‌شده غیرفعال می‌کند.
  </Tab>
  <Tab title="scripts (default)">
    embedهای تعاملی را مجاز می‌کند و در عین حال isolation مربوط به origin را حفظ می‌کند؛ این مقدار پیش‌فرض است و معمولاً برای بازی‌ها/ویجت‌های مرورگری self-contained کافی است.
  </Tab>
  <Tab title="trusted">
    برای سندهای same-site که عمداً به privilegeهای قوی‌تر نیاز دارند، `allow-same-origin` را روی `allow-scripts` اضافه می‌کند.
  </Tab>
</Tabs>

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
از `trusted` فقط زمانی استفاده کنید که سند embedشده واقعاً به رفتار same-origin نیاز دارد. برای بیشتر بازی‌های agent-generated و canvasهای تعاملی، `scripts` انتخاب امن‌تری است.
</Warning>

URLهای embed خارجی مطلق `http(s)` به‌صورت پیش‌فرض مسدود می‌مانند. اگر عمداً می‌خواهید `[embed url="https://..."]` صفحه‌های third-party را load کند، `gateway.controlUi.allowExternalEmbedUrls: true` را تنظیم کنید.

## عرض پیام چت

پیام‌های گروه‌بندی‌شدهٔ چت از max-width پیش‌فرض خوانا استفاده می‌کنند. استقرارهای wide-monitor می‌توانند بدون patch کردن CSS bundled، آن را با تنظیم `gateway.controlUi.chatMessageMaxWidth` بازنویسی کنند:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

مقدار پیش از رسیدن به مرورگر validate می‌شود. مقدارهای پشتیبانی‌شده شامل طول‌ها و درصدهای ساده مانند `960px` یا `82%`، به‌علاوهٔ expressionهای عرض محدودشدهٔ `min(...)`، `max(...)`، `clamp(...)`، `calc(...)` و `fit-content(...)` هستند.

## دسترسی Tailnet (پیشنهادی)

<Tabs>
  <Tab title="Tailscale Serve یکپارچه (ترجیحی)">
    Gateway را روی loopback نگه دارید و بگذارید Tailscale Serve آن را با HTTPS proxy کند:

    ```bash
    openclaw gateway --tailscale serve
    ```

    باز کنید:

    - `https://<magicdns>/` (یا `gateway.controlUi.basePath` پیکربندی‌شدهٔ شما)

    به‌صورت پیش‌فرض، درخواست‌های Serve مربوط به رابط کاربری کنترل/WebSocket می‌توانند از طریق headerهای هویت Tailscale (`tailscale-user-login`) احراز هویت شوند، وقتی `gateway.auth.allowTailscale` برابر `true` باشد. OpenClaw هویت را با resolve کردن آدرس `x-forwarded-for` از طریق `tailscale whois` و match کردن آن با header راستی‌آزمایی می‌کند، و فقط زمانی آن‌ها را می‌پذیرد که درخواست با headerهای `x-forwarded-*` مربوط به Tailscale به loopback برسد. برای نشست‌های operator رابط کاربری کنترل با هویت دستگاه مرورگر، این مسیر Serve راستی‌آزمایی‌شده همچنین round trip مربوط به device-pairing را رد می‌کند؛ مرورگرهای بدون device و اتصال‌های node-role همچنان checkهای device عادی را دنبال می‌کنند. اگر می‌خواهید حتی برای ترافیک Serve هم credentialهای shared-secret صریح لازم باشد، `gateway.auth.allowTailscale: false` را تنظیم کنید. سپس از `gateway.auth.mode: "token"` یا `"password"` استفاده کنید.

    برای آن مسیر async هویت Serve، تلاش‌های auth ناموفق برای همان IP client و scope احراز هویت، پیش از writeهای rate-limit به‌صورت serialized انجام می‌شوند. بنابراین retryهای بد هم‌زمان از همان مرورگر می‌توانند در درخواست دوم به‌جای دو mismatch ساده که موازی race می‌کنند، `retry later` را نشان دهند.

    <Warning>
    احراز هویت Serve بدون token فرض می‌کند host مربوط به gateway قابل اعتماد است. اگر ممکن است کد local غیرقابل‌اعتماد روی آن host اجرا شود، احراز هویت token/password را الزامی کنید.
    </Warning>

  </Tab>
  <Tab title="Bind به tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    سپس باز کنید:

    - `http://<tailscale-ip>:18789/` (یا `gateway.controlUi.basePath` پیکربندی‌شدهٔ شما)

    shared secret متناظر را در تنظیمات UI paste کنید (به‌صورت `connect.params.auth.token` یا `connect.params.auth.password` ارسال می‌شود).

  </Tab>
</Tabs>

## HTTP ناامن

اگر داشبورد را روی HTTP ساده باز کنید (`http://<lan-ip>` یا `http://<tailscale-ip>`)، مرورگر در یک **context غیرامن** اجرا می‌شود و WebCrypto را مسدود می‌کند. به‌صورت پیش‌فرض، OpenClaw اتصال‌های رابط کاربری کنترل بدون هویت دستگاه را **مسدود** می‌کند.

استثناهای مستندشده:

- سازگاری HTTP ناامن فقط برای localhost با `gateway.controlUi.allowInsecureAuth=true`
- احراز هویت موفق operator رابط کاربری کنترل از طریق `gateway.auth.mode: "trusted-proxy"`
- حالت اضطراری `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**رفع پیشنهادی:** از HTTPS (Tailscale Serve) استفاده کنید یا UI را به‌صورت local باز کنید:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (روی host مربوط به gateway)

<AccordionGroup>
  <Accordion title="رفتار toggle احراز هویت ناامن">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` فقط یک toggle سازگاری local است:

    - به نشست‌های رابط کاربری کنترل روی localhost اجازه می‌دهد در contextهای HTTP غیرامن بدون هویت دستگاه ادامه پیدا کنند.
    - checkهای pairing را bypass نمی‌کند.
    - الزام‌های هویت دستگاه remote (غیر-localhost) را نرم‌تر نمی‌کند.

  </Accordion>
  <Accordion title="فقط حالت اضطراری">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` checkهای هویت دستگاه رابط کاربری کنترل را غیرفعال می‌کند و یک کاهش امنیتی شدید است. پس از استفادهٔ اضطراری، سریعاً آن را برگردانید.
    </Warning>

  </Accordion>
  <Accordion title="یادداشت trusted-proxy">
    - احراز هویت trusted-proxy موفق می‌تواند نشست‌های رابط کاربری کنترل **operator** را بدون هویت دستگاه بپذیرد.
    - این مورد به نشست‌های رابط کاربری کنترل node-role تعمیم پیدا نمی‌کند.
    - reverse proxyهای loopback روی همان host همچنان احراز هویت trusted-proxy را برآورده نمی‌کنند؛ [احراز هویت trusted proxy](/fa/gateway/trusted-proxy-auth) را ببینید.

  </Accordion>
</AccordionGroup>

برای راهنمایی راه‌اندازی HTTPS، [Tailscale](/fa/gateway/tailscale) را ببینید.

## سیاست امنیت محتوا

رابط کاربری کنترل با سیاست سخت‌گیرانهٔ `img-src` ارائه می‌شود: فقط assetهای **same-origin**، URLهای `data:`، و URLهای `blob:` تولیدشده به‌صورت local مجاز هستند. URLهای تصویر remote `http(s)` و protocol-relative توسط مرورگر رد می‌شوند و fetch شبکه‌ای صادر نمی‌کنند.

معنای عملی این موضوع:

- Avatarها و تصویرهایی که زیر مسیرهای relative ارائه می‌شوند (برای مثال `/avatars/<id>`) همچنان رندر می‌شوند، از جمله routeهای avatar احراز هویت‌شده که UI آن‌ها را fetch می‌کند و به URLهای `blob:` local تبدیل می‌کند.
- URLهای inline `data:image/...` همچنان رندر می‌شوند (برای payloadهای in-protocol مفید است).
- URLهای `blob:` local که توسط رابط کاربری کنترل ایجاد شده‌اند همچنان رندر می‌شوند.
- URLهای remote avatar که توسط metadata کانال emit می‌شوند، در helperهای avatar رابط کاربری کنترل strip می‌شوند و با logo/badge داخلی جایگزین می‌شوند؛ بنابراین یک کانال compromised یا مخرب نمی‌تواند fetch تصویر remote دلخواه را از مرورگر operator اجبار کند.

برای دریافت این رفتار لازم نیست چیزی را تغییر دهید — همیشه فعال است و قابل پیکربندی نیست.

## احراز هویت route آواتار

وقتی احراز هویت gateway پیکربندی شده باشد، endpoint آواتار رابط کاربری کنترل همان token gateway را مثل بقیهٔ API لازم دارد:

- `GET /avatar/<agentId>` تصویر آواتار را فقط به فراخوانندگان احراز هویت‌شده برمی‌گرداند. `GET /avatar/<agentId>?meta=1` metadata آواتار را تحت همان قاعده برمی‌گرداند.
- درخواست‌های احراز هویت‌نشده به هرکدام از routeها رد می‌شوند (همسان با route sibling مربوط به assistant-media). این کار مانع نشت هویت agent از route آواتار روی hostهایی می‌شود که در غیر این صورت محافظت‌شده هستند.
- خود رابط کاربری کنترل هنگام fetch کردن avatarها، token gateway را به‌عنوان bearer header forward می‌کند و از URLهای blob احراز هویت‌شده استفاده می‌کند تا تصویر همچنان در dashboardها رندر شود.

اگر احراز هویت gateway را غیرفعال کنید (روی میزبان‌های اشتراکی توصیه نمی‌شود)، مسیر آواتار نیز بدون احراز هویت می‌شود، همسو با بقیه gateway.

## احراز هویت مسیر رسانه دستیار

وقتی احراز هویت gateway پیکربندی شده باشد، پیش‌نمایش‌های رسانه محلی دستیار از یک مسیر دومرحله‌ای استفاده می‌کنند:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` به احراز هویت معمول اپراتور Control UI نیاز دارد. مرورگر هنگام بررسی دسترس‌پذیری، توکن gateway را به‌عنوان سرآیند bearer ارسال می‌کند.
- پاسخ‌های موفق فراداده شامل یک `mediaTicket` کوتاه‌عمر هستند که فقط به همان مسیر منبع دقیق محدود شده است.
- نشانی‌های تصویر، صدا، ویدئو و سند که در مرورگر رندر می‌شوند، به‌جای توکن یا گذرواژه فعال gateway از `mediaTicket=<ticket>` استفاده می‌کنند. این بلیت سریع منقضی می‌شود و نمی‌تواند منبع متفاوتی را مجاز کند.

این کار رندر عادی رسانه را با عناصر رسانه بومی مرورگر سازگار نگه می‌دارد، بدون اینکه اعتبارنامه‌های قابل‌استفاده مجدد gateway در نشانی‌های رسانه قابل مشاهده قرار بگیرند.

## ساخت UI

Gateway فایل‌های ایستا را از `dist/control-ui` ارائه می‌کند. آن‌ها را با این دستور بسازید:

```bash
pnpm ui:build
```

پایه مطلق اختیاری (وقتی نشانی‌های ثابت دارایی‌ها را می‌خواهید):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

برای توسعه محلی (سرور توسعه جداگانه):

```bash
pnpm ui:dev
```

سپس UI را به نشانی Gateway WS خود متصل کنید (برای مثال `ws://127.0.0.1:18789`).

## صفحه خالی Control UI

اگر مرورگر یک داشبورد خالی بارگذاری کند و DevTools خطای مفیدی نشان ندهد، ممکن است یک افزونه یا اسکریپت محتوای اولیه مانع ارزیابی برنامه ماژول JavaScript شده باشد. صفحه ایستا شامل یک پنل بازیابی HTML ساده است که وقتی `<openclaw-app>` پس از راه‌اندازی ثبت نشده باشد ظاهر می‌شود.

پس از تغییر محیط مرورگر، از کنش **دوباره تلاش کن** پنل استفاده کنید، یا پس از این بررسی‌ها به‌صورت دستی بازبارگذاری کنید:

- افزونه‌هایی را که به همه صفحه‌ها تزریق می‌شوند غیرفعال کنید، به‌ویژه افزونه‌هایی با اسکریپت‌های محتوای `<all_urls>`.
- یک پنجره خصوصی، یک نمایه مرورگر تمیز، یا مرورگر دیگری را امتحان کنید.
- Gateway را در حال اجرا نگه دارید و همان نشانی داشبورد را پس از تغییر مرورگر بررسی کنید.

## اشکال‌زدایی/آزمایش: سرور توسعه + Gateway راه دور

Control UI از فایل‌های ایستا تشکیل شده است؛ مقصد WebSocket قابل پیکربندی است و می‌تواند با مبدا HTTP متفاوت باشد. این وقتی مفید است که سرور توسعه Vite را به‌صورت محلی می‌خواهید اما Gateway جای دیگری اجرا می‌شود.

<Steps>
  <Step title="راه‌اندازی سرور توسعه UI">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="باز کردن با gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    احراز هویت یک‌باره اختیاری (در صورت نیاز):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="نکته‌ها">
    - `gatewayUrl` پس از بارگذاری در localStorage ذخیره و از URL حذف می‌شود.
    - اگر یک نقطه پایانی کامل `ws://` یا `wss://` را از طریق `gatewayUrl` ارسال می‌کنید، مقدار `gatewayUrl` را URL-encode کنید تا مرورگر رشته query را درست تجزیه کند.
    - `token` باید هر زمان ممکن است از طریق قطعه URL (`#token=...`) ارسال شود. قطعه‌ها به سرور ارسال نمی‌شوند، که از نشت در گزارش‌های درخواست و Referer جلوگیری می‌کند. پارامترهای query قدیمی `?token=` هنوز برای سازگاری یک‌بار وارد می‌شوند، اما فقط به‌عنوان fallback، و بلافاصله پس از bootstrap حذف می‌شوند.
    - `password` فقط در حافظه نگه داشته می‌شود.
    - وقتی `gatewayUrl` تنظیم شده باشد، UI به اعتبارنامه‌های config یا محیط fallback نمی‌کند. `token` (یا `password`) را صریح ارائه کنید. نبود اعتبارنامه‌های صریح یک خطا است.
    - وقتی Gateway پشت TLS است (Tailscale Serve، پراکسی HTTPS، و غیره) از `wss://` استفاده کنید.
    - `gatewayUrl` فقط در یک پنجره سطح بالا پذیرفته می‌شود (نه به‌صورت جاسازی‌شده) تا از clickjacking جلوگیری شود.
    - استقرارهای عمومی و غیر local loopback برای Control UI باید `gateway.controlUi.allowedOrigins` را صریح تنظیم کنند (مبداهای کامل). بارگذاری‌های خصوصی هم‌مبدا در LAN/Tailnet از local loopback، RFC1918/link-local، `.local`، `.ts.net`، یا میزبان‌های CGNAT مربوط به Tailscale بدون فعال‌سازی fallback سرآیند Host پذیرفته می‌شوند.
    - راه‌اندازی Gateway ممکن است مبداهای محلی مانند `http://localhost:<port>` و `http://127.0.0.1:<port>` را از bind و پورت موثر زمان اجرا seed کند، اما مبداهای مرورگر راه دور همچنان به ورودی‌های صریح نیاز دارند.
    - از `gateway.controlUi.allowedOrigins: ["*"]` استفاده نکنید مگر برای آزمایش محلی کاملا کنترل‌شده. معنی آن اجازه دادن به هر مبدا مرورگر است، نه «مطابقت با هر میزبانی که استفاده می‌کنم».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت fallback مبدا سرآیند Host را فعال می‌کند، اما این یک حالت امنیتی خطرناک است.

  </Accordion>
</AccordionGroup>

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
