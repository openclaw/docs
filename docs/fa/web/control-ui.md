The Control UI یک برنامهٔ تک‌صفحه‌ای کوچک **Vite + Lit** است که توسط Gateway سرو می‌شود:

- پیش‌فرض: `http://<host>:18789/`
- پیشوند اختیاری: `gateway.controlUi.basePath` را تنظیم کنید (مثلاً `/openclaw`)

این برنامه **مستقیماً با وب‌سوکت Gateway** روی همان پورت صحبت می‌کند.

## باز کردن سریع (محلی)

اگر Gateway روی همان رایانه اجرا می‌شود، این را باز کنید:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (یا [http://localhost:18789/](http://localhost:18789/))

اگر صفحه بارگذاری نشد، ابتدا Gateway را اجرا کنید: `openclaw gateway`.

احراز هویت هنگام دست‌دهی وب‌سوکت از این راه‌ها ارائه می‌شود:

- `connect.params.auth.token`
- `connect.params.auth.password`
- سرآیندهای هویت Tailscale Serve وقتی `gateway.auth.allowTailscale: true` باشد
- سرآیندهای هویت پروکسی مورداعتماد وقتی `gateway.auth.mode: "trusted-proxy"` باشد

پنل تنظیمات داشبورد یک توکن را برای نشست زبانهٔ فعلی مرورگر و URL انتخاب‌شدهٔ Gateway نگه می‌دارد؛ گذرواژه‌ها پایدار نمی‌شوند. راه‌اندازی اولیه معمولاً در اولین اتصال، برای احراز هویت با راز مشترک یک توکن Gateway تولید می‌کند، اما وقتی `gateway.auth.mode` برابر `"password"` باشد احراز هویت با گذرواژه هم کار می‌کند.

## جفت‌سازی دستگاه (اولین اتصال)

وقتی از یک مرورگر یا دستگاه جدید به Control UI وصل می‌شوید، Gateway معمولاً به **تأیید یک‌بارهٔ جفت‌سازی** نیاز دارد. این یک اقدام امنیتی برای جلوگیری از دسترسی غیرمجاز است.

**چیزی که خواهید دید:** "disconnected (1008): pairing required"

<Steps>
  <Step title="List pending requests">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="Approve by request ID">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

اگر مرورگر جفت‌سازی را با جزئیات احراز هویت تغییرکرده (نقش/دامنه‌ها/کلید عمومی) دوباره تلاش کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، `openclaw devices list` را دوباره اجرا کنید.

اگر مرورگر از قبل جفت شده باشد و شما آن را از دسترسی خواندن به دسترسی نوشتن/مدیریت تغییر دهید، این به‌عنوان ارتقای تأیید در نظر گرفته می‌شود، نه اتصال مجدد بی‌صدا. OpenClaw تأیید قبلی را فعال نگه می‌دارد، اتصال مجدد گسترده‌تر را مسدود می‌کند، و از شما می‌خواهد مجموعهٔ دامنه‌های جدید را صریحاً تأیید کنید.

پس از تأیید، دستگاه به خاطر سپرده می‌شود و دیگر نیازی به تأیید دوباره ندارد، مگر اینکه آن را با `openclaw devices revoke --device <id> --role <role>` لغو کنید. برای چرخش و لغو توکن، [CLI دستگاه‌ها](/fa/cli/devices) را ببینید.

<Note>
- اتصال‌های مستقیم مرورگر از طریق local loopback (`127.0.0.1` / `localhost`) به‌طور خودکار تأیید می‌شوند.
- Tailscale Serve می‌تواند رفت‌وبرگشت جفت‌سازی را برای نشست‌های اپراتور Control UI نادیده بگیرد، وقتی `gateway.auth.allowTailscale: true` باشد، هویت Tailscale تأیید شود، و مرورگر هویت دستگاه خود را ارائه کند.
- اتصال‌های مستقیم Tailnet، اتصال‌های مرورگر از LAN، و پروفایل‌های مرورگر بدون هویت دستگاه همچنان به تأیید صریح نیاز دارند.
- هر پروفایل مرورگر یک شناسهٔ دستگاه یکتا تولید می‌کند، بنابراین تعویض مرورگرها یا پاک کردن داده‌های مرورگر به جفت‌سازی دوباره نیاز خواهد داشت.

</Note>

## هویت شخصی (محلیِ مرورگر)

Control UI از یک هویت شخصی برای هر مرورگر (نام نمایشی و آواتار) پشتیبانی می‌کند که برای انتساب در نشست‌های مشترک به پیام‌های خروجی پیوست می‌شود. این هویت در ذخیره‌سازی مرورگر قرار دارد، به پروفایل فعلی مرورگر محدود است، و به دستگاه‌های دیگر همگام‌سازی نمی‌شود یا در سمت سرور، فراتر از فرادادهٔ معمول نویسندگی رونوشت روی پیام‌هایی که واقعاً ارسال می‌کنید، پایدار نمی‌ماند. پاک کردن داده‌های سایت یا تعویض مرورگر آن را به حالت خالی بازنشانی می‌کند.

همین الگوی محلیِ مرورگر برای بازنویسی آواتار دستیار هم اعمال می‌شود. آواتارهای بارگذاری‌شدهٔ دستیار، هویت حل‌شده توسط Gateway را فقط در مرورگر محلی پوشش می‌دهند و هرگز از مسیر `config.patch` رفت‌وبرگشت نمی‌کنند. فیلد پیکربندی مشترک `ui.assistant.avatar` همچنان برای کلاینت‌های غیر UI که فیلد را مستقیماً می‌نویسند در دسترس است (مانند Gatewayهای اسکریپتی یا داشبوردهای سفارشی).

## نقطهٔ پایانی پیکربندی زمان اجرا

Control UI تنظیمات زمان اجرای خود را از `/__openclaw/control-ui-config.json` دریافت می‌کند. این نقطهٔ پایانی با همان احراز هویت Gateway که برای بقیهٔ سطح HTTP استفاده می‌شود محافظت می‌شود: مرورگرهای احراز هویت‌نشده نمی‌توانند آن را دریافت کنند، و دریافت موفق نیازمند یک توکن/گذرواژهٔ Gateway معتبر از قبل، هویت Tailscale Serve، یا هویت پروکسی مورداعتماد است.

## پشتیبانی زبان

Control UI می‌تواند در اولین بارگذاری، بر اساس زبان مرورگر شما خودش را بومی‌سازی کند. برای بازنویسی بعدی، **نمای کلی -> دسترسی Gateway -> زبان** را باز کنید. انتخابگر زبان در کارت دسترسی Gateway قرار دارد، نه زیر ظاهر.

- زبان‌های پشتیبانی‌شده: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- ترجمه‌های غیرانگلیسی به‌صورت تنبل در مرورگر بارگذاری می‌شوند.
- زبان انتخاب‌شده در ذخیره‌سازی مرورگر ذخیره می‌شود و در بازدیدهای آینده دوباره استفاده می‌شود.
- کلیدهای ترجمهٔ موجودنبودن به انگلیسی برمی‌گردند.

## پوسته‌های ظاهری

پنل ظاهر، پوسته‌های داخلی Claw، Knot، و Dash، به‌علاوهٔ یک جایگاه واردسازی tweakcn محلیِ مرورگر را نگه می‌دارد. برای وارد کردن پوسته، [پوسته‌های tweakcn](https://tweakcn.com/themes) را باز کنید، یک پوسته انتخاب یا ایجاد کنید، روی **اشتراک‌گذاری** کلیک کنید، و پیوند پوستهٔ کپی‌شده را در ظاهر بچسبانید. واردکننده همچنین URLهای رجیستری `https://tweakcn.com/r/themes/<id>`، URLهای ویرایشگر مانند `https://tweakcn.com/editor/theme?theme=amethyst-haze`، مسیرهای نسبی `/themes/<id>`، شناسه‌های خام پوسته، و نام‌های پیش‌فرض پوسته مانند `amethyst-haze` را می‌پذیرد.

پوسته‌های واردشده فقط در پروفایل فعلی مرورگر ذخیره می‌شوند. آن‌ها در پیکربندی Gateway نوشته نمی‌شوند و بین دستگاه‌ها همگام‌سازی نمی‌شوند. جایگزینی پوستهٔ واردشده همان یک جایگاه محلی را به‌روزرسانی می‌کند؛ پاک کردن آن، اگر پوستهٔ واردشده انتخاب شده باشد، پوستهٔ فعال را به Claw برمی‌گرداند.

## رفتار گفتگو

<AccordionGroup>
  <Accordion title="معناشناسی ارسال و تاریخچه">
    - `chat.send` **غیرمسدودکننده** است: بلافاصله با `{ runId, status: "started" }` تأیید می‌کند و پاسخ از طریق رویدادهای `chat` به‌صورت جریانی ارسال می‌شود.
    - بارگذاری‌های چت تصویرها به‌همراه فایل‌های غیر ویدیویی را می‌پذیرد. تصویرها مسیر تصویر بومی را حفظ می‌کنند؛ فایل‌های دیگر به‌عنوان رسانه مدیریت‌شده ذخیره می‌شوند و در تاریخچه به‌صورت پیوندهای پیوست نشان داده می‌شوند.
    - ارسال دوباره با همان `idempotencyKey` هنگام اجرا `{ status: "in_flight" }` و پس از تکمیل `{ status: "ok" }` را برمی‌گرداند.
    - پاسخ‌های `chat.history` برای ایمنی UI از نظر اندازه محدود می‌شوند. وقتی ورودی‌های رونوشت بیش از حد بزرگ باشند، Gateway ممکن است فیلدهای متنی طولانی را کوتاه کند، بلوک‌های فراداده سنگین را حذف کند، و پیام‌های بیش‌ازحد بزرگ را با یک جای‌نگهدار (`[chat.history omitted: message too large]`) جایگزین کند.
    - تصویرهای دستیار/تولیدشده به‌عنوان ارجاع‌های رسانه مدیریت‌شده پایدار می‌شوند و از طریق URLهای رسانه احرازهویت‌شده Gateway دوباره ارائه می‌شوند، بنابراین بارگذاری‌های دوباره به باقی ماندن payloadهای خام تصویر base64 در پاسخ تاریخچه چت وابسته نیستند.
    - `chat.history` همچنین برچسب‌های دستوری درون‌خطی صرفاً نمایشی را از متن قابل مشاهده دستیار حذف می‌کند (برای مثال `[[reply_to_*]]` و `[[audio_as_voice]]`)، payloadهای XML فراخوانی ابزار به‌صورت متن ساده (از جمله `<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>`، و بلوک‌های فراخوانی ابزار کوتاه‌شده)، و توکن‌های کنترل مدل ASCII/تمام‌عرض نشت‌کرده را حذف می‌کند، و ورودی‌های دستیار را که کل متن قابل مشاهده آن‌ها فقط توکن سکوت دقیق `NO_REPLY` / `no_reply` است حذف می‌کند.
    - هنگام یک ارسال فعال و تازه‌سازی نهایی تاریخچه، اگر `chat.history` برای لحظه‌ای یک snapshot قدیمی‌تر برگرداند، نمای چت پیام‌های خوش‌بینانه محلی کاربر/دستیار را قابل مشاهده نگه می‌دارد؛ وقتی تاریخچه Gateway به‌روز شد، رونوشت canonical آن پیام‌های محلی را جایگزین می‌کند.
    - `chat.inject` یک یادداشت دستیار را به رونوشت نشست اضافه می‌کند و یک رویداد `chat` را برای به‌روزرسانی‌های فقط UI پخش می‌کند (بدون اجرای عامل، بدون تحویل کانال).
    - انتخابگرهای مدل و thinking در سربرگ چت، نشست فعال را فوراً از طریق `sessions.patch` وصله می‌کنند؛ آن‌ها overrideهای پایدار نشست هستند، نه گزینه‌های ارسال فقط برای یک نوبت.
    - انتخابگر مدل چت نمای مدل پیکربندی‌شده Gateway را درخواست می‌کند. اگر `agents.defaults.models` موجود باشد، همان allowlist انتخابگر را هدایت می‌کند. در غیر این صورت، انتخابگر ورودی‌های صریح `models.providers.*.models` به‌علاوه ارائه‌دهندگان دارای auth قابل استفاده را نشان می‌دهد. کاتالوگ کامل از طریق RPC اشکال‌زدایی `models.list` با `view: "all"` در دسترس می‌ماند.
    - وقتی گزارش‌های تازه استفاده نشست Gateway فشار بالای context را نشان دهند، ناحیه composer چت یک اعلان context و، در سطوح Compaction پیشنهادی، یک دکمه فشرده نشان می‌دهد که مسیر عادی Compaction نشست را اجرا می‌کند. snapshotهای توکن کهنه تا زمانی که Gateway دوباره استفاده تازه را گزارش کند پنهان می‌مانند.

  </Accordion>
  <Accordion title="حالت گفت‌وگو (بلادرنگ مرورگر)">
    حالت گفت‌وگو از یک ارائه‌دهنده صدای بلادرنگ ثبت‌شده استفاده می‌کند. OpenAI را با `talk.provider: "openai"` به‌همراه `talk.providers.openai.apiKey` پیکربندی کنید، یا Google را با `talk.provider: "google"` به‌همراه `talk.providers.google.apiKey` پیکربندی کنید؛ پیکربندی ارائه‌دهنده بلادرنگ Voice Call هنوز می‌تواند به‌عنوان fallback دوباره استفاده شود. مرورگر هرگز یک کلید API استاندارد ارائه‌دهنده دریافت نمی‌کند. OpenAI یک secret موقت Realtime client برای WebRTC دریافت می‌کند. Google Live یک توکن auth محدودشده Live API تک‌بارمصرف برای یک نشست WebSocket مرورگر دریافت می‌کند، که دستورالعمل‌ها و اعلان‌های ابزار توسط Gateway در توکن قفل شده‌اند. ارائه‌دهندگانی که فقط یک پل بلادرنگ backend عرضه می‌کنند از طریق transport رله Gateway اجرا می‌شوند، بنابراین credentials و socketهای vendor سمت سرور باقی می‌مانند و صدای مرورگر از طریق RPCهای احرازهویت‌شده Gateway منتقل می‌شود. prompt نشست Realtime توسط Gateway ساخته می‌شود؛ `talk.realtime.session` overrideهای دستورالعمل ارائه‌شده توسط caller را نمی‌پذیرد.

    در composer چت، کنترل گفت‌وگو دکمه موج‌ها کنار دکمه dictation میکروفون است. وقتی گفت‌وگو شروع می‌شود، ردیف وضعیت composer ابتدا `Connecting Talk...`، سپس هنگام اتصال صدا `Talk live`، یا هنگام مشورت یک فراخوانی ابزار بلادرنگ با مدل بزرگ‌تر پیکربندی‌شده از طریق `chat.send` عبارت `Asking OpenClaw...` را نشان می‌دهد.

    smoke زنده maintainer: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` تبادل SDP مرورگر WebRTC برای OpenAI، راه‌اندازی WebSocket مرورگر با توکن محدودشده Google Live، و adapter مرورگر رله Gateway را با رسانه میکروفون ساختگی تأیید می‌کند. این دستور فقط وضعیت ارائه‌دهنده را چاپ می‌کند و secrets را log نمی‌کند.

  </Accordion>
  <Accordion title="توقف و لغو">
    - روی **توقف** کلیک کنید (`chat.abort` را فراخوانی می‌کند).
    - وقتی یک اجرا فعال است، پیگیری‌های عادی در صف قرار می‌گیرند. روی **هدایت** در یک پیام صف‌شده کلیک کنید تا آن پیگیری به نوبت در حال اجرا تزریق شود.
    - برای لغو خارج از باند، `/stop` را تایپ کنید (یا عبارت‌های مستقل لغو مانند `stop`، `stop action`، `stop run`، `stop openclaw`، `please stop`).
    - `chat.abort` از `{ sessionKey }` (بدون `runId`) برای لغو همه اجراهای فعال آن نشست پشتیبانی می‌کند.

  </Accordion>
  <Accordion title="نگهداری بخش جزئی پس از لغو">
    - وقتی یک اجرا لغو می‌شود، متن جزئی دستیار هنوز می‌تواند در UI نشان داده شود.
    - Gateway وقتی خروجی بافرشده وجود داشته باشد، متن جزئی دستیارِ لغوشده را در تاریخچه رونوشت پایدار می‌کند.
    - ورودی‌های پایدارشده شامل فراداده لغو هستند تا مصرف‌کنندگان رونوشت بتوانند بخش‌های جزئی لغوشده را از خروجی تکمیل عادی تشخیص دهند.

  </Accordion>
</AccordionGroup>

## نصب PWA و web push

Control UI یک `manifest.webmanifest` و یک service worker ارائه می‌کند، بنابراین مرورگرهای مدرن می‌توانند آن را به‌عنوان یک PWA مستقل نصب کنند. Web Push به Gateway اجازه می‌دهد PWA نصب‌شده را با اعلان‌ها بیدار کند، حتی وقتی تب یا پنجره مرورگر باز نیست.

| سطح                                                  | کاری که انجام می‌دهد                                               |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | manifest مربوط به PWA. وقتی قابل دسترسی باشد، مرورگرها "Install app" را پیشنهاد می‌کنند. |
| `ui/public/sw.js`                                     | service worker که رویدادهای `push` و کلیک‌های اعلان را مدیریت می‌کند. |
| `push/vapid-keys.json` (زیر دایرکتوری state مربوط به OpenClaw) | جفت‌کلید VAPID تولیدشده خودکار که برای امضای payloadهای Web Push استفاده می‌شود. |
| `push/web-push-subscriptions.json`                    | endpointهای subscription مرورگر که پایدار شده‌اند.                 |

وقتی می‌خواهید کلیدها را ثابت نگه دارید (برای استقرارهای چندمیزبانه، چرخش secrets، یا تست‌ها)، جفت‌کلید VAPID را از طریق env vars در فرایند Gateway override کنید:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (پیش‌فرض: `mailto:openclaw@localhost`)

Control UI از این متدهای Gateway محدودشده به scope برای ثبت و تست subscriptionهای مرورگر استفاده می‌کند:

- `push.web.vapidPublicKey` — کلید عمومی VAPID فعال را دریافت می‌کند.
- `push.web.subscribe` — یک `endpoint` به‌همراه `keys.p256dh`/`keys.auth` را ثبت می‌کند.
- `push.web.unsubscribe` — یک endpoint ثبت‌شده را حذف می‌کند.
- `push.web.test` — یک اعلان تستی به subscription مربوط به caller ارسال می‌کند.

<Note>
Web Push مستقل از مسیر رله APNS در iOS است (برای push مبتنی بر رله، [پیکربندی](/fa/gateway/configuration) را ببینید) و همچنین مستقل از متد موجود `push.test` است که pairing موبایل native را هدف می‌گیرند.
</Note>

## embedهای میزبانی‌شده

پیام‌های دستیار می‌توانند محتوای وب میزبانی‌شده را با shortcode `[embed ...]` به‌صورت درون‌خطی render کنند. سیاست sandbox iframe توسط `gateway.controlUi.embedSandbox` کنترل می‌شود:

<Tabs>
  <Tab title="strict">
    اجرای script را داخل embedهای میزبانی‌شده غیرفعال می‌کند.
  </Tab>
  <Tab title="scripts (default)">
    embedهای تعاملی را مجاز می‌کند و هم‌زمان جداسازی origin را حفظ می‌کند؛ این حالت پیش‌فرض است و معمولاً برای بازی‌ها/widgetهای مرورگر خودبسنده کافی است.
  </Tab>
  <Tab title="trusted">
    برای سندهای همان سایت که عمداً به امتیازهای قوی‌تر نیاز دارند، `allow-same-origin` را روی `allow-scripts` اضافه می‌کند.
  </Tab>
</Tabs>

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
از `trusted` فقط وقتی استفاده کنید که سند embedشده واقعاً به رفتار same-origin نیاز دارد. برای بیشتر بازی‌ها و canvasهای تعاملی تولیدشده توسط agent، `scripts` گزینه امن‌تری است.
</Warning>

URLهای embed خارجی مطلق `http(s)` به‌صورت پیش‌فرض مسدود می‌مانند. اگر عمداً می‌خواهید `[embed url="https://..."]` صفحه‌های شخص ثالث را بارگذاری کند، `gateway.controlUi.allowExternalEmbedUrls: true` را تنظیم کنید.

## دسترسی Tailnet (توصیه‌شده)

<Tabs>
  <Tab title="Tailscale Serve یکپارچه (ترجیحی)">
    Gateway را روی loopback نگه دارید و اجازه دهید Tailscale Serve با HTTPS آن را proxy کند:

    ```bash
    openclaw gateway --tailscale serve
    ```

    باز کنید:

    - `https://<magicdns>/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    به‌صورت پیش‌فرض، درخواست‌های Control UI/WebSocket Serve می‌توانند از طریق headerهای هویت Tailscale (`tailscale-user-login`) احراز هویت شوند وقتی `gateway.auth.allowTailscale` برابر `true` باشد. OpenClaw هویت را با resolve کردن آدرس `x-forwarded-for` با `tailscale whois` و تطبیق آن با header تأیید می‌کند، و این موارد را فقط وقتی می‌پذیرد که درخواست با headerهای `x-forwarded-*` مربوط به Tailscale به loopback برسد. برای نشست‌های operator در Control UI با هویت دستگاه مرورگر، این مسیر Serve تأییدشده همچنین رفت‌وبرگشت device-pairing را رد می‌کند؛ مرورگرهای بدون دستگاه و اتصال‌های با نقش node همچنان بررسی‌های عادی دستگاه را دنبال می‌کنند. اگر می‌خواهید حتی برای traffic مربوط به Serve هم credentialهای shared-secret صریح لازم باشد، `gateway.auth.allowTailscale: false` را تنظیم کنید. سپس از `gateway.auth.mode: "token"` یا `"password"` استفاده کنید.

    برای آن مسیر async هویت Serve، تلاش‌های auth ناموفق برای همان IP کلاینت و scope احراز هویت، پیش از نوشتن rate-limit به‌صورت ترتیبی انجام می‌شوند. بنابراین retryهای بد هم‌زمان از همان مرورگر می‌توانند در درخواست دوم به‌جای دو mismatch ساده که موازی رقابت می‌کنند، `retry later` را نشان دهند.

    <Warning>
    auth بدون token در Serve فرض می‌کند میزبان gateway قابل اعتماد است. اگر ممکن است کد محلی غیرقابل اعتماد روی آن میزبان اجرا شود، auth با token/password را الزامی کنید.
    </Warning>

  </Tab>
  <Tab title="bind به tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    سپس باز کنید:

    - `http://<tailscale-ip>:18789/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    shared secret مطابق را در تنظیمات UI بچسبانید (به‌صورت `connect.params.auth.token` یا `connect.params.auth.password` ارسال می‌شود).

  </Tab>
</Tabs>

## HTTP ناامن

اگر dashboard را از طریق HTTP ساده باز کنید (`http://<lan-ip>` یا `http://<tailscale-ip>`)، مرورگر در یک **context غیرامن** اجرا می‌شود و WebCrypto را مسدود می‌کند. به‌صورت پیش‌فرض، OpenClaw اتصال‌های Control UI بدون هویت دستگاه را **مسدود** می‌کند.

استثناهای مستندشده:

- سازگاری HTTP ناامن فقط برای localhost با `gateway.controlUi.allowInsecureAuth=true`
- auth موفق operator در Control UI از طریق `gateway.auth.mode: "trusted-proxy"`
- گزینه اضطراری `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**راه‌حل پیشنهادی:** از HTTPS (Tailscale Serve) استفاده کنید یا UI را به‌صورت محلی باز کنید:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (روی میزبان gateway)

<AccordionGroup>
  <Accordion title="رفتار toggle مربوط به insecure-auth">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` فقط یک toggle سازگاری محلی است:

    - به نشست‌های Control UI روی localhost اجازه می‌دهد در contextهای HTTP غیرامن بدون هویت دستگاه ادامه پیدا کنند.
    - بررسی‌های pairing را دور نمی‌زند.
    - الزامات هویت دستگاه remote (غیر localhost) را آسان‌تر نمی‌کند.

  </Accordion>
  <Accordion title="فقط اضطراری">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` بررسی‌های هویت دستگاه در رابط کاربری کنترل را غیرفعال می‌کند و افت شدید امنیتی محسوب می‌شود. پس از استفاده اضطراری، آن را سریعاً برگردانید.
    </Warning>

  </Accordion>
  <Accordion title="یادداشت trusted-proxy">
    - احراز هویت موفق trusted-proxy می‌تواند نشست‌های رابط کاربری کنترل **operator** را بدون هویت دستگاه بپذیرد.
    - این مورد به نشست‌های رابط کاربری کنترل با نقش node-role گسترش نمی‌یابد.
    - reverse proxyهای loopback روی همان میزبان همچنان احراز هویت trusted-proxy را برآورده نمی‌کنند؛ [احراز هویت پراکسی مورد اعتماد](/fa/gateway/trusted-proxy-auth) را ببینید.

  </Accordion>
</AccordionGroup>

برای راهنمایی تنظیم HTTPS، [Tailscale](/fa/gateway/tailscale) را ببینید.

## سیاست امنیت محتوا

رابط کاربری کنترل با سیاست سخت‌گیرانه `img-src` ارائه می‌شود: فقط دارایی‌های **same-origin**، URLهای `data:`، و URLهای `blob:` تولیدشده به‌صورت محلی مجاز هستند. URLهای تصویر راه‌دور `http(s)` و وابسته به پروتکل توسط مرورگر رد می‌شوند و درخواست شبکه صادر نمی‌کنند.

معنای عملی این موضوع:

- آواتارها و تصویرهایی که از مسیرهای نسبی ارائه می‌شوند (برای مثال `/avatars/<id>`) همچنان رندر می‌شوند، از جمله مسیرهای آواتار احراز هویت‌شده‌ای که رابط کاربری واکشی می‌کند و به URLهای محلی `blob:` تبدیل می‌کند.
- URLهای درون‌خطی `data:image/...` همچنان رندر می‌شوند (برای payloadهای درون‌پروتکلی مفید است).
- URLهای محلی `blob:` که توسط رابط کاربری کنترل ساخته می‌شوند همچنان رندر می‌شوند.
- URLهای آواتار راه‌دور که از متادیتای کانال صادر می‌شوند در helperهای آواتار رابط کاربری کنترل حذف و با logo/badge داخلی جایگزین می‌شوند، بنابراین یک کانال آلوده یا مخرب نمی‌تواند مرورگر operator را وادار به واکشی تصویر راه‌دور دلخواه کند.

برای دریافت این رفتار لازم نیست چیزی را تغییر دهید — این قابلیت همیشه فعال است و قابل پیکربندی نیست.

## احراز هویت مسیر آواتار

وقتی احراز هویت Gateway پیکربندی شده باشد، endpoint آواتار رابط کاربری کنترل به همان توکن Gateway نیاز دارد که بقیه API نیاز دارند:

- `GET /avatar/<agentId>` تصویر آواتار را فقط به فراخواننده‌های احراز هویت‌شده برمی‌گرداند. `GET /avatar/<agentId>?meta=1` متادیتای آواتار را تحت همان قاعده برمی‌گرداند.
- درخواست‌های احراز هویت‌نشده به هر دو مسیر رد می‌شوند (همسو با مسیر sibling assistant-media). این کار از نشت هویت agent از مسیر آواتار در میزبان‌هایی که در غیر این صورت محافظت شده‌اند جلوگیری می‌کند.
- خود رابط کاربری کنترل هنگام واکشی آواتارها توکن Gateway را به‌عنوان bearer header ارسال می‌کند، و از URLهای blob احراز هویت‌شده استفاده می‌کند تا تصویر همچنان در داشبوردها رندر شود.

اگر احراز هویت Gateway را غیرفعال کنید (روی میزبان‌های مشترک توصیه نمی‌شود)، مسیر آواتار نیز همسو با بقیه Gateway بدون احراز هویت می‌شود.

## ساخت رابط کاربری

Gateway فایل‌های static را از `dist/control-ui` ارائه می‌کند. آن‌ها را با این دستور بسازید:

```bash
pnpm ui:build
```

base مطلق اختیاری (وقتی URLهای ثابت دارایی می‌خواهید):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

برای توسعه محلی (dev server جداگانه):

```bash
pnpm ui:dev
```

سپس رابط کاربری را به URL مربوط به Gateway WS خود اشاره دهید (مثلاً `ws://127.0.0.1:18789`).

## اشکال‌زدایی/آزمایش: dev server + Gateway راه‌دور

رابط کاربری کنترل مجموعه‌ای از فایل‌های static است؛ هدف WebSocket قابل پیکربندی است و می‌تواند با origin مربوط به HTTP متفاوت باشد. این زمانی مفید است که dev server مربوط به Vite را به‌صورت محلی می‌خواهید اما Gateway جای دیگری اجرا می‌شود.

<Steps>
  <Step title="dev server رابط کاربری را شروع کنید">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="با gatewayUrl باز کنید">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    احراز هویت یک‌باره اختیاری (در صورت نیاز):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="یادداشت‌ها">
    - `gatewayUrl` پس از بارگذاری در localStorage ذخیره و از URL حذف می‌شود.
    - اگر یک endpoint کامل `ws://` یا `wss://` را از طریق `gatewayUrl` ارسال می‌کنید، مقدار `gatewayUrl` را URL-encode کنید تا مرورگر query string را درست parse کند.
    - `token` باید هر زمان ممکن است از طریق fragment در URL (`#token=...`) ارسال شود. Fragmentها به سرور ارسال نمی‌شوند، که از نشت در request-log و Referer جلوگیری می‌کند. پارامترهای query قدیمی `?token=` همچنان برای سازگاری یک‌بار import می‌شوند، اما فقط به‌عنوان fallback، و بلافاصله پس از bootstrap حذف می‌شوند.
    - `password` فقط در حافظه نگه داشته می‌شود.
    - وقتی `gatewayUrl` تنظیم شده باشد، رابط کاربری به config یا credentials محیطی fallback نمی‌کند. `token` (یا `password`) را صریحاً ارائه کنید. نبود credentials صریح یک خطاست.
    - وقتی Gateway پشت TLS است از `wss://` استفاده کنید (Tailscale Serve، پراکسی HTTPS، و غیره).
    - `gatewayUrl` فقط در پنجره top-level پذیرفته می‌شود (نه حالت embedded) تا از clickjacking جلوگیری شود.
    - deploymentهای رابط کاربری کنترل غیر از loopback باید `gateway.controlUi.allowedOrigins` را صریحاً تنظیم کنند (originهای کامل). این شامل تنظیمات توسعه راه‌دور نیز می‌شود.
    - راه‌اندازی Gateway ممکن است originهای محلی مانند `http://localhost:<port>` و `http://127.0.0.1:<port>` را از bind و port مؤثر زمان اجرا seed کند، اما originهای مرورگر راه‌دور همچنان به entryهای صریح نیاز دارند.
    - از `gateway.controlUi.allowedOrigins: ["*"]` استفاده نکنید مگر برای آزمایش محلی کاملاً کنترل‌شده. معنای آن اجازه دادن به هر origin مرورگر است، نه «هر میزبانی را که استفاده می‌کنم match کن».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت fallback مربوط به Host-header origin را فعال می‌کند، اما این یک حالت امنیتی خطرناک است.

  </Accordion>
</AccordionGroup>

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
