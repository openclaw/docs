رابط کاربری کنترل یک برنامه تک‌صفحه‌ای کوچک **Vite + Lit** است که توسط Gateway ارائه می‌شود:

- پیش‌فرض: `http://<host>:18789/`
- پیشوند اختیاری: `gateway.controlUi.basePath` را تنظیم کنید (مثلاً `/openclaw`)

این برنامه **مستقیماً با وب‌سوکت Gateway** روی همان پورت صحبت می‌کند.

## باز کردن سریع (محلی)

اگر Gateway روی همان رایانه در حال اجراست، باز کنید:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (یا [http://localhost:18789/](http://localhost:18789/))

اگر صفحه بارگذاری نشد، ابتدا Gateway را شروع کنید: `openclaw gateway`.

<Note>
در اتصال‌های LAN بومی Windows، Windows Firewall یا Group Policy مدیریت‌شده توسط سازمان همچنان می‌تواند URL اعلام‌شده LAN را مسدود کند، حتی وقتی `127.0.0.1` روی میزبان Gateway کار می‌کند. روی میزبان Windows دستور `openclaw gateway status --deep` را اجرا کنید؛ این دستور پورت‌های احتمالاً مسدود، ناهماهنگی‌های پروفایل، و قوانین فایروال محلی را که سیاست ممکن است نادیده بگیرد گزارش می‌کند.
</Note>

احراز هویت هنگام دست‌دهی وب‌سوکت از طریق موارد زیر فراهم می‌شود:

- `connect.params.auth.token`
- `connect.params.auth.password`
- هدرهای هویت Tailscale Serve وقتی `gateway.auth.allowTailscale: true` باشد
- هدرهای هویت پراکسی مورد اعتماد وقتی `gateway.auth.mode: "trusted-proxy"` باشد

پنل تنظیمات داشبورد یک توکن را برای نشست برگه فعلی مرورگر و URL انتخاب‌شده Gateway نگه می‌دارد؛ گذرواژه‌ها پایدار ذخیره نمی‌شوند. راه‌اندازی اولیه معمولاً در اولین اتصال برای احراز هویت با راز مشترک یک توکن gateway تولید می‌کند، اما وقتی `gateway.auth.mode` برابر `"password"` باشد احراز هویت با گذرواژه هم کار می‌کند.

## جفت‌سازی دستگاه (اولین اتصال)

وقتی از یک مرورگر یا دستگاه جدید به رابط کاربری کنترل وصل می‌شوید، Gateway معمولاً به یک **تأیید جفت‌سازی یک‌باره** نیاز دارد. این یک اقدام امنیتی برای جلوگیری از دسترسی غیرمجاز است.

**چیزی که خواهید دید:** «disconnected (1008): pairing required»

<Steps>
  <Step title="فهرست کردن درخواست‌های در انتظار">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="تأیید با شناسه درخواست">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

اگر مرورگر جفت‌سازی را با جزئیات احراز هویت تغییر‌یافته (نقش/دامنه‌ها/کلید عمومی) دوباره امتحان کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، دوباره `openclaw devices list` را اجرا کنید.

اگر مرورگر از قبل جفت شده باشد و آن را از دسترسی خواندن به دسترسی نوشتن/مدیر تغییر دهید، این به‌عنوان ارتقای تأیید در نظر گرفته می‌شود، نه اتصال مجدد بی‌صدا. OpenClaw تأیید قبلی را فعال نگه می‌دارد، اتصال مجدد گسترده‌تر را مسدود می‌کند، و از شما می‌خواهد مجموعه دامنه جدید را صریحاً تأیید کنید.

پس از تأیید، دستگاه به خاطر سپرده می‌شود و دیگر به تأیید مجدد نیاز ندارد، مگر اینکه آن را با `openclaw devices revoke --device <id> --role <role>` لغو کنید. برای چرخش و لغو توکن، [CLI دستگاه‌ها](/fa/cli/devices) را ببینید.

عامل‌های Paperclip که از طریق آداپتور `openclaw_gateway` وصل می‌شوند از همان جریان تأیید اجرای اول استفاده می‌کنند. پس از تلاش اتصال اولیه، `openclaw devices approve --latest` را اجرا کنید تا درخواست در انتظار را پیش‌نمایش کنید، سپس فرمان چاپ‌شده `openclaw devices approve <requestId>` را دوباره اجرا کنید تا آن را تأیید کنید. برای یک gateway راه دور، مقدارهای صریح `--url` و `--token` را پاس دهید. برای پایدار نگه داشتن تأییدها در راه‌اندازی‌های مجدد، به‌جای اینکه Paperclip در هر اجرا یک هویت دستگاه موقت جدید تولید کند، یک `adapterConfig.devicePrivateKeyPem` پایدار در Paperclip پیکربندی کنید.

<Note>
- اتصال‌های مستقیم مرورگر از طریق local loopback (`127.0.0.1` / `localhost`) به‌طور خودکار تأیید می‌شوند.
- وقتی `gateway.auth.allowTailscale: true` باشد، هویت Tailscale تأیید شود، و مرورگر هویت دستگاه خود را ارائه کند، Tailscale Serve می‌تواند رفت‌وبرگشت جفت‌سازی را برای نشست‌های اپراتور رابط کاربری کنترل رد کند.
- اتصال‌های مستقیم Tailnet، اتصال‌های مرورگر از LAN، و پروفایل‌های مرورگر بدون هویت دستگاه همچنان به تأیید صریح نیاز دارند.
- هر پروفایل مرورگر یک شناسه دستگاه یکتا تولید می‌کند، بنابراین تغییر مرورگر یا پاک کردن داده‌های مرورگر به جفت‌سازی مجدد نیاز خواهد داشت.

</Note>

  ## هویت شخصی (محلیِ مرورگر)

  رابط کنترل از یک هویت شخصی برای هر مرورگر (نام نمایشی و آواتار) پشتیبانی می‌کند که برای انتساب در نشست‌های مشترک به پیام‌های خروجی پیوست می‌شود. این هویت در فضای ذخیره‌سازی مرورگر قرار دارد، به پروفایل مرورگر فعلی محدود است، و فراتر از فرادادهٔ معمول نویسندگی رونوشت برای پیام‌هایی که واقعاً ارسال می‌کنید، با دستگاه‌های دیگر همگام‌سازی یا در سمت سرور پایدار نمی‌شود. پاک‌کردن داده‌های سایت یا تغییر مرورگر آن را به حالت خالی بازمی‌گرداند.

  همین الگوی محلیِ مرورگر برای بازنویسی آواتار دستیار نیز اعمال می‌شود. آواتارهای بارگذاری‌شدهٔ دستیار فقط در مرورگر محلی هویت حل‌شده توسط Gateway را پوشش می‌دهند و هرگز از مسیر `config.patch` رفت‌وبرگشت نمی‌کنند. فیلد پیکربندی مشترک `ui.assistant.avatar` همچنان برای کلاینت‌های غیر UI که فیلد را مستقیم می‌نویسند (مانند Gatewayهای اسکریپت‌شده یا داشبوردهای سفارشی) در دسترس است.

  ## نقطهٔ پایانی پیکربندی زمان اجرا

  رابط کنترل تنظیمات زمان اجرای خود را از `/control-ui-config.json` دریافت می‌کند که نسبت به مسیر پایهٔ رابط کنترل Gateway حل می‌شود (برای نمونه، وقتی UI زیر `/__openclaw__/` ارائه می‌شود، مسیر `/__openclaw__/control-ui-config.json` است). این نقطهٔ پایانی با همان احراز هویت Gateway که برای بقیهٔ سطح HTTP استفاده می‌شود محافظت می‌شود: مرورگرهای احرازهویت‌نشده نمی‌توانند آن را دریافت کنند، و دریافت موفق نیازمند یک توکن/گذرواژهٔ معتبر Gateway از پیش موجود، هویت Tailscale Serve، یا هویت پروکسیِ مورد اعتماد است.

  ## پشتیبانی زبان

  رابط کنترل می‌تواند هنگام نخستین بارگذاری، خود را بر اساس زبان مرورگر شما بومی‌سازی کند. برای بازنویسی آن در زمان بعد، **نمای کلی -> دسترسی Gateway -> زبان** را باز کنید. گزینشگر زبان در کارت دسترسی Gateway قرار دارد، نه زیر Appearance.

  - زبان‌های پشتیبانی‌شده: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
  - ترجمه‌های غیرانگلیسی به‌صورت تنبل در مرورگر بارگذاری می‌شوند.
  - زبان انتخاب‌شده در فضای ذخیره‌سازی مرورگر ذخیره می‌شود و در بازدیدهای آینده دوباره استفاده می‌شود.
  - کلیدهای ترجمهٔ مفقود به انگلیسی بازمی‌گردند.

  ترجمه‌های مستندات برای همان مجموعهٔ زبان‌های غیرانگلیسی تولید می‌شوند، اما گزینشگر زبان داخلی سایت مستندات در Mintlify به کدهای زبانی محدود است که Mintlify می‌پذیرد. مستندات تایلندی (`th`) و فارسی (`fa`) همچنان در مخزن انتشار تولید می‌شوند؛ ممکن است تا زمانی که Mintlify از این کدها پشتیبانی کند در آن گزینشگر ظاهر نشوند.

  ## تم‌های ظاهری

  پنل Appearance تم‌های داخلی Claw، Knot، و Dash را نگه می‌دارد، به‌علاوهٔ یک جایگاه واردسازی tweakcn محلیِ مرورگر. برای واردکردن یک تم، [ویرایشگر tweakcn](https://tweakcn.com/editor/theme) را باز کنید، یک تم انتخاب یا ایجاد کنید، روی **Share** کلیک کنید، و پیوند تم کپی‌شده را در Appearance بچسبانید. واردکننده همچنین URLهای رجیستری `https://tweakcn.com/r/themes/<id>`، URLهای ویرایشگر مانند `https://tweakcn.com/editor/theme?theme=amethyst-haze`، مسیرهای نسبی `/themes/<id>`، شناسه‌های خام تم، و نام‌های پیش‌فرض تم مانند `amethyst-haze` را می‌پذیرد.

  Appearance همچنین شامل تنظیم اندازهٔ متن محلیِ مرورگر است. این تنظیم همراه با سایر ترجیحات رابط کنترل ذخیره می‌شود، بر متن چت، متن سازندهٔ پیام، کارت‌های ابزار، و نوارهای کناری چت اعمال می‌شود، و ورودی‌های متنی را دست‌کم 16px نگه می‌دارد تا Safari موبایل هنگام فوکوس به‌صورت خودکار بزرگ‌نمایی نکند.

  تم‌های واردشده فقط در پروفایل مرورگر فعلی ذخیره می‌شوند. آن‌ها در پیکربندی Gateway نوشته نمی‌شوند و بین دستگاه‌ها همگام‌سازی نمی‌شوند. جایگزین‌کردن تم واردشده همان یک جایگاه محلی را به‌روزرسانی می‌کند؛ پاک‌کردن آن، اگر تم واردشده انتخاب شده باشد، تم فعال را به Claw بازمی‌گرداند.

    - مشاهده/ویرایش `~/.openclaw/openclaw.json` (`config.get`, `config.set`).
    - MCP یک صفحهٔ تنظیمات اختصاصی برای سرورهای پیکربندی‌شده، فعال‌سازی، خلاصه‌های OAuth/فیلتر/موازی، فرمان‌های رایج اپراتور، و ویرایشگر پیکربندی محدوده‌دار `mcp` دارد.

    - نوشتن‌ها شامل محافظ هش پایه هستند تا از بازنویسی ناخواستهٔ ویرایش‌های هم‌زمان جلوگیری شود.

    - ذخیره‌های فرم، جای‌نگهدارهای ویرایش‌شدهٔ کهنه‌ای را که نمی‌توان از پیکربندی ذخیره‌شده بازیابی کرد دور می‌ریزند، در حالی که مقدارهای ویرایش‌شده‌ای را که همچنان به رازهای ذخیره‌شده نگاشت می‌شوند حفظ می‌کنند.
    - رندر schema + فرم (`config.schema` / `config.schema.lookup`، شامل فیلد `title` / `description`، راهنمایی‌های UI منطبق، خلاصه‌های فرزند بلافصل، فرادادهٔ مستندات روی گره‌های شیء/وایلدکارت/آرایه/ترکیب تودرتو، به‌علاوهٔ schemaهای Plugin + کانال در صورت دسترس‌بودن)؛ ویرایشگر Raw JSON فقط وقتی در دسترس است که نمای لحظه‌ای یک رفت‌وبرگشت خام امن داشته باشد.
    - اگر یک نمای لحظه‌ای نتواند متن خام را به‌صورت امن رفت‌وبرگشت کند، رابط کنترل حالت Form را اجباری می‌کند و حالت Raw را برای آن نمای لحظه‌ای غیرفعال می‌کند.
    - گزینهٔ «بازنشانی به ذخیره‌شده» در ویرایشگر Raw JSON شکل خامِ نوشته‌شده (قالب‌بندی، توضیحات، چیدمان `$include`) را به‌جای رندر دوبارهٔ یک نمای لحظه‌ای تخت حفظ می‌کند، بنابراین وقتی نمای لحظه‌ای بتواند به‌صورت امن رفت‌وبرگشت کند، ویرایش‌های خارجی پس از بازنشانی باقی می‌مانند.
    - مقدارهای شیء ساختاریافتهٔ SecretRef در ورودی‌های متنی فرم فقط-خواندنی رندر می‌شوند تا از خراب‌شدن تصادفی شیء به رشته جلوگیری شود.

## صفحه MCP

صفحه اختصاصی MCP یک نمای اپراتوری برای سرورهای MCP مدیریت‌شده توسط OpenClaw زیر `mcp.servers` است. این صفحه به‌تنهایی transportهای MCP را شروع نمی‌کند؛ از آن برای بررسی و ویرایش پیکربندی ذخیره‌شده استفاده کنید، سپس وقتی به اثبات زنده سرور نیاز دارید از `openclaw mcp doctor --probe` استفاده کنید.

گردش کار معمول:

1. از نوار کناری **MCP** را باز کنید.
2. کارت‌های خلاصه را برای تعداد کل، فعال، OAuth و سرورهای فیلترشده بررسی کنید.
3. هر ردیف سرور را از نظر transport، فعال‌بودن، احراز هویت، فیلترها، زمان‌انتظارها و راهنمای فرمان‌ها بازبینی کنید.
4. وقتی یک سرور باید پیکربندی‌شده بماند اما وارد کشف زمان اجرا نشود، فعال‌بودن را تغییر دهید.
5. بخش پیکربندی scoped `mcp` را برای تعریف‌های سرور، هدرها، مسیرهای TLS/mTLS، فراداده OAuth، فیلترهای ابزار و فراداده projection مربوط به Codex ویرایش کنید.
6. برای نوشتن پیکربندی از **ذخیره** استفاده کنید، یا وقتی Gateway در حال اجرا باید پیکربندی تغییرکرده را اعمال کند از **ذخیره و انتشار** استفاده کنید.
7. وقتی فرایند ویرایش‌شده به عیب‌یابی ایستا، اثبات زنده، یا دورریزی runtime کش‌شده نیاز دارد، `openclaw mcp status --verbose`، `openclaw mcp doctor --probe`، یا `openclaw mcp reload` را از ترمینال اجرا کنید.

این صفحه پیش از نمایش، مقادیر شبیه URL دارای اطلاعات اعتبارنامه را redaction می‌کند و نام سرورها را در قطعه‌فرمان‌ها داخل نقل‌قول می‌گذارد تا فرمان‌های کپی‌شده همچنان با فاصله‌ها یا metacharacterهای shell کار کنند. مرجع کامل CLI و پیکربندی در [MCP](/fa/cli/mcp) قرار دارد.

## زبانه فعالیت

زبانه فعالیت یک ناظر موقت و محلیِ مرورگر برای فعالیت زنده ابزارها است. این زبانه از همان جریان رویداد Gateway `session.tool` / ابزار مشتق می‌شود که کارت‌های ابزار Chat را تغذیه می‌کند؛ خانواده رویداد Gateway، endpoint، ذخیره فعالیت پایدار، خوراک metrics، یا جریان ناظر خارجی دیگری اضافه نمی‌کند.

ورودی‌های فعالیت فقط خلاصه‌های پاک‌سازی‌شده و پیش‌نمایش‌های خروجی redaction‌شده و کوتاه‌شده را نگه می‌دارند. مقادیر آرگومان ابزار در وضعیت فعالیت ذخیره نمی‌شوند؛ رابط کاربری نشان می‌دهد که آرگومان‌ها پنهان هستند و فقط تعداد فیلدهای آرگومان را ثبت می‌کند. فهرست درون‌حافظه‌ای زبانه مرورگر فعلی را دنبال می‌کند، در ناوبری داخل Control UI باقی می‌ماند، و با بارگذاری مجدد صفحه، تغییر نشست، یا **پاک‌کردن** بازنشانی می‌شود.

    - `chat.send` **غیرمسدودکننده** است: بلافاصله با `{ runId, status: "started" }` ack می‌کند و پاسخ از طریق رویدادهای `chat` stream می‌شود. کلاینت‌های مورد اعتماد Control UI ممکن است فراداده اختیاری زمان‌بندی ACK را نیز برای عیب‌یابی محلی دریافت کنند.
    - آپلودهای Chat تصویرها به‌علاوه فایل‌های غیر ویدیویی را می‌پذیرند. تصویرها مسیر تصویر بومی را حفظ می‌کنند؛ فایل‌های دیگر به‌عنوان رسانه مدیریت‌شده ذخیره می‌شوند و در تاریخچه به‌صورت لینک‌های پیوست نشان داده می‌شوند.
    - ارسال دوباره با همان `idempotencyKey` هنگام اجرا `{ status: "in_flight" }` و پس از تکمیل `{ status: "ok" }` برمی‌گرداند.
    - پاسخ‌های `chat.history` برای ایمنی رابط کاربری محدود به اندازه هستند. وقتی ورودی‌های transcript بیش از حد بزرگ باشند، Gateway ممکن است فیلدهای متنی بلند را کوتاه کند، بلوک‌های فراداده سنگین را حذف کند، و پیام‌های بیش‌ازحد بزرگ را با یک placeholder (`[chat.history omitted: message too large]`) جایگزین کند.
    - وقتی یک پیام قابل‌مشاهده assistant در `chat.history` کوتاه شده باشد، خواننده کناری می‌تواند ورودی transcript کامل و display-normalized را در صورت نیاز از طریق `chat.message.get` با `sessionKey`، در صورت نیاز `agentId` فعال، و `messageId` transcript واکشی کند. اگر Gateway همچنان نتواند مقدار بیشتری برگرداند، خواننده به‌جای تکرار بی‌صدا پیش‌نمایش کوتاه‌شده، وضعیت صریحاً ناموجود را نشان می‌دهد.
    - تصویرهای assistant/تولیدشده به‌صورت ارجاع‌های رسانه مدیریت‌شده پایدار می‌شوند و از طریق URLهای رسانه احرازهویت‌شده Gateway بازگردانده می‌شوند، بنابراین بارگذاری‌های مجدد وابسته به باقی‌ماندن payloadهای تصویر base64 خام در پاسخ تاریخچه Chat نیستند.
    - هنگام render کردن `chat.history`، Control UI تگ‌های directive درون‌خطی فقط-نمایشی را از متن قابل‌مشاهده assistant حذف می‌کند (برای مثال `[[reply_to_*]]` و `[[audio_as_voice]]`)، payloadهای XML فراخوانی ابزار به‌صورت متن ساده (از جمله `<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>`، و بلوک‌های فراخوانی ابزار کوتاه‌شده)، و توکن‌های کنترل مدل ASCII/تمام‌عرض نشت‌کرده را حذف می‌کند، و ورودی‌های assistant را که کل متن قابل‌مشاهده آن‌ها فقط توکن سکوت دقیق `NO_REPLY` / `no_reply` یا توکن تأیید Heartbeat یعنی `HEARTBEAT_OK` است حذف می‌کند.
    - در طول یک ارسال فعال و refresh نهایی تاریخچه، اگر `chat.history` برای لحظه‌ای snapshot قدیمی‌تری برگرداند، نمای Chat پیام‌های خوش‌بینانه محلی user/assistant را قابل‌مشاهده نگه می‌دارد؛ وقتی تاریخچه Gateway به‌روز شد، transcript مرجع آن پیام‌های محلی را جایگزین می‌کند.
    - رویدادهای زنده `chat` وضعیت تحویل هستند، در حالی که `chat.history` از transcript پایدار نشست بازسازی می‌شود. پس از رویدادهای نهایی ابزار، Control UI تاریخچه را دوباره بارگذاری می‌کند و فقط یک دنباله خوش‌بینانه کوچک را ادغام می‌کند؛ مرز transcript در [WebChat](/fa/web/webchat) مستند شده است.
    - `chat.inject` یک یادداشت assistant را به transcript نشست اضافه می‌کند و برای به‌روزرسانی‌های فقط UI یک رویداد `chat` broadcast می‌کند (بدون اجرای agent، بدون تحویل کانال).
    - نوار کناری نشست‌های اخیر را با یک اقدام نشست جدید، یک لینک همه نشست‌ها، و یک دکمه جست‌وجوی نشست فهرست می‌کند که انتخابگر کامل نشست را باز می‌کند (scoped بر اساس agent انتخاب‌شده، همراه با جست‌وجو و صفحه‌بندی). تغییر agentها فقط نشست‌های متصل به آن agent را نشان می‌دهد و وقتی هنوز نشست‌های داشبورد ذخیره‌شده ندارد، به نشست اصلی همان agent fallback می‌کند.
    - در عرض‌های دسکتاپ، کنترل‌های Chat روی یک ردیف فشرده می‌مانند و هنگام اسکرول به پایین transcript جمع می‌شوند؛ اسکرول به بالا، بازگشت به بالا، یا رسیدن به پایین کنترل‌ها را بازمی‌گرداند.
    - پیام‌های فقط متنی تکراریِ پیاپی به‌صورت یک حباب با نشان شمارش render می‌شوند. پیام‌هایی که تصویر، پیوست، خروجی ابزار، یا پیش‌نمایش canvas دارند جمع نمی‌شوند.
    - انتخابگرهای مدل و تفکر در سربرگ Chat، نشست فعال را بلافاصله از طریق `sessions.patch` patch می‌کنند؛ آن‌ها بازنویسی‌های پایدار نشست هستند، نه گزینه‌های ارسال فقط برای یک نوبت.
    - اگر در حالی پیامی ارسال کنید که تغییر انتخابگر مدل برای همان نشست هنوز در حال ذخیره است، composer پیش از فراخوانی `chat.send` منتظر patch آن نشست می‌ماند تا ارسال از مدل انتخاب‌شده استفاده کند.
    - تایپ `/new` در Control UI همان نشست داشبورد تازه‌ای را ایجاد می‌کند و به آن سوییچ می‌کند که New Chat ایجاد می‌کند، مگر وقتی `session.dmScope: "main"` پیکربندی شده و والد فعلی نشست اصلی agent باشد؛ در این حالت نشست اصلی را درجا بازنشانی می‌کند. تایپ `/reset` بازنشانی صریح درجا در Gateway را برای نشست فعلی نگه می‌دارد.
    - انتخابگر مدل Chat نمای مدل پیکربندی‌شده Gateway را درخواست می‌کند. اگر `agents.defaults.models` حاضر باشد، آن allowlist انتخابگر را هدایت می‌کند، از جمله ورودی‌های `provider/*` که کاتالوگ‌های scoped به provider را پویا نگه می‌دارند. در غیر این صورت انتخابگر ورودی‌های صریح `models.providers.*.models` به‌علاوه providerهای دارای احراز هویت قابل‌استفاده را نشان می‌دهد. کاتالوگ کامل از طریق RPC عیب‌یابی `models.list` با `view: "all"` در دسترس می‌ماند.
    - وقتی گزارش‌های تازه استفاده نشست Gateway شامل توکن‌های context فعلی باشند، نوار ابزار composer در Chat یک حلقه کوچک استفاده context را با درصد استفاده‌شده نشان می‌دهد؛ جزئیات کامل توکن در tooltip آن قرار دارد. این حلقه در فشار context بالا به سبک هشدار تغییر می‌کند و در سطح‌های پیشنهادی Compaction، یک دکمه فشرده نشان می‌دهد که مسیر عادی Compaction نشست را اجرا می‌کند. snapshotهای کهنه توکن پنهان می‌شوند تا Gateway دوباره استفاده تازه را گزارش کند.

  <Accordion title="حالت مکالمه (realtime مرورگر)">
    حالت مکالمه از یک provider صدای realtime ثبت‌شده استفاده می‌کند. OpenAI را با `talk.realtime.provider: "openai"` به‌علاوه یک پروفایل احراز هویت API-key برای `openai`، `talk.realtime.providers.openai.apiKey`، یا `OPENAI_API_KEY` پیکربندی کنید؛ پروفایل‌های OAuth مربوط به OpenAI صدای Realtime را پیکربندی نمی‌کنند. Google را با `talk.realtime.provider: "google"` به‌علاوه `talk.realtime.providers.google.apiKey` پیکربندی کنید. مرورگر هرگز یک کلید API استاندارد provider دریافت نمی‌کند. OpenAI یک راز کلاینت Realtime موقت برای WebRTC دریافت می‌کند. Google Live یک توکن احراز هویت Live API محدود و یک‌بارمصرف برای نشست WebSocket مرورگر دریافت می‌کند، همراه با دستورالعمل‌ها و declarationهای ابزار که توسط Gateway در توکن قفل شده‌اند. providerهایی که فقط یک پل realtime سمت backend ارائه می‌کنند از طریق transport relay در Gateway اجرا می‌شوند، بنابراین credentialها و socketهای فروشنده سمت سرور می‌مانند در حالی که صوت مرورگر از طریق RPCهای احرازهویت‌شده Gateway منتقل می‌شود. prompt نشست Realtime توسط Gateway assembled می‌شود؛ `talk.client.create` بازنویسی‌های دستورالعمل ارائه‌شده توسط caller را نمی‌پذیرد.

    Composer مربوط به Chat یک دکمه گزینه‌های Talk کنار دکمه شروع/توقف Talk دارد. گزینه‌ها روی نشست Talk بعدی اعمال می‌شوند و می‌توانند provider، transport، مدل، صدا، تلاش reasoning، آستانه VAD، مدت سکوت و padding پیشوند را بازنویسی کنند. وقتی یک گزینه خالی باشد، Gateway در صورت وجود از پیش‌فرض‌های پیکربندی‌شده یا پیش‌فرض provider استفاده می‌کند. انتخاب relay در Gateway مسیر relay سمت backend را اجباری می‌کند؛ انتخاب WebRTC نشست را در مالکیت کلاینت نگه می‌دارد و اگر provider نتواند نشست مرورگر ایجاد کند، به‌جای fallback بی‌صدا به relay شکست می‌خورد.

    در Composer مربوط به Chat، کنترل Talk دکمه موج‌ها کنار دکمه دیکته میکروفون است. وقتی Talk شروع می‌شود، ردیف وضعیت composer ابتدا `Connecting Talk...`، سپس هنگام اتصال صدا `Talk live`، یا زمانی که یک فراخوانی ابزار realtime در حال مشورت با مدل بزرگ‌تر پیکربندی‌شده از طریق `talk.client.toolCall` است `Asking OpenClaw...` را نشان می‌دهد.

GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` پل WebSocket backend مربوط به OpenAI، تبادل SDP در WebRTC مرورگر OpenAI، راه‌اندازی WebSocket مرورگر با توکن محدود Google Live، و adapter مرورگر relay در Gateway را با رسانه میکروفون جعلی بررسی می‌کند. فرمان فقط وضعیت provider را چاپ می‌کند و secretها را log نمی‌کند.

  </Accordion>
  <Accordion title="توقف و abort">
    - روی **توقف** کلیک کنید (`chat.abort` را فراخوانی می‌کند).
    - تا زمانی که یک اجرا فعال است، follow-upهای عادی در صف قرار می‌گیرند. روی **هدایت** در یک پیام صف‌شده کلیک کنید تا آن follow-up را به نوبت در حال اجرا inject کند.
    - برای abort خارج از باند، `/stop` را تایپ کنید (یا عبارت‌های مستقل abort مانند `stop`، `stop action`، `stop run`، `stop openclaw`، `please stop`).
    - `chat.abort` از `{ sessionKey }` (بدون `runId`) برای abort همه اجراهای فعال آن نشست پشتیبانی می‌کند.

  </Accordion>
  <Accordion title="نگهداشت جزئی پس از abort">
    - وقتی یک اجرا abort می‌شود، متن جزئی assistant همچنان می‌تواند در رابط کاربری نشان داده شود.

    - ورودی‌های پایدارشده شامل فراداده abort هستند تا مصرف‌کنندگان transcript بتوانند جزئیات abort را از خروجی تکمیل عادی تشخیص دهند.

  </Accordion>

## نصب PWA و Web Push

Control UI همراه با یک `manifest.webmanifest` و یک service worker عرضه می‌شود، بنابراین مرورگرهای مدرن می‌توانند آن را به‌عنوان یک PWA مستقل نصب کنند. Web Push به Gateway امکان می‌دهد حتی وقتی زبانه یا پنجره مرورگر باز نیست، PWA نصب‌شده را با اعلان‌ها بیدار کند.

اگر صفحه بلافاصله پس از به‌روزرسانی OpenClaw پیام **ناسازگاری پروتکل** را نشان می‌دهد، ابتدا داشبورد را با `openclaw dashboard` دوباره باز کنید و صفحه را hard-refresh کنید. اگر همچنان خطا داشت، داده‌های سایت را برای origin داشبورد پاک کنید یا در یک پنجره مرورگر خصوصی آزمایش کنید؛ یک تب قدیمی یا کش service-worker مرورگر می‌تواند همچنان بسته رابط کاربری کنترل مربوط به پیش از به‌روزرسانی را در برابر Gateway جدیدتر اجرا کند.

| سطح                                                  | کارکرد                                                            |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | مانیفست PWA. مرورگرها پس از دردسترس شدن آن، گزینه «نصب برنامه» را پیشنهاد می‌کنند. |
| `ui/public/sw.js`                                     | service worker که رویدادهای `push` و کلیک‌های اعلان را مدیریت می‌کند. |
| `push/vapid-keys.json` (زیر دایرکتوری state مربوط به OpenClaw) | جفت‌کلید VAPID که به‌طور خودکار تولید می‌شود و برای امضای محموله‌های Web Push استفاده می‌شود. |
| `push/web-push-subscriptions.json`                    | endpointهای اشتراک مرورگر که ماندگار شده‌اند.                    |

وقتی می‌خواهید کلیدها را ثابت نگه دارید (برای استقرارهای چندمیزبانه، چرخش رازها، یا تست‌ها)، جفت‌کلید VAPID را از طریق متغیرهای محیطی روی فرایند Gateway بازنویسی کنید:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (پیش‌فرض: `https://openclaw.ai`)

رابط کاربری کنترل از این متدهای Gateway با محدوده محدودشده برای ثبت و تست اشتراک‌های مرورگر استفاده می‌کند:

- `push.web.vapidPublicKey` — کلید عمومی VAPID فعال را دریافت می‌کند.
- `push.web.subscribe` — یک `endpoint` به‌همراه `keys.p256dh`/`keys.auth` ثبت می‌کند.
- `push.web.unsubscribe` — یک endpoint ثبت‌شده را حذف می‌کند.
- `push.web.test` — یک اعلان آزمایشی به اشتراک فراخواننده می‌فرستد.

<Note>
Web Push مستقل از مسیر relay مربوط به iOS APNS است (برای push پشتیبانی‌شده با relay، [پیکربندی](/fa/gateway/configuration) را ببینید) و از متد موجود `push.test` نیز جداست؛ این‌ها جفت‌سازی موبایل بومی را هدف می‌گیرند.
</Note>

## جای‌گذاری‌های میزبانی‌شده

پیام‌های دستیار می‌توانند محتوای وب میزبانی‌شده را به‌صورت درون‌خطی با shortcode‏ `[embed ...]` رندر کنند. سیاست sandbox iframe با `gateway.controlUi.embedSandbox` کنترل می‌شود:

<Tabs>
  <Tab title="سخت‌گیرانه">
    اجرای اسکریپت را داخل جای‌گذاری‌های میزبانی‌شده غیرفعال می‌کند.
  </Tab>
  <Tab title="اسکریپت‌ها (پیش‌فرض)">
    جای‌گذاری‌های تعاملی را مجاز می‌کند و در عین حال جداسازی origin را حفظ می‌کند؛ این حالت پیش‌فرض است و معمولا برای بازی‌ها/ویجت‌های مرورگری خودکفا کافی است.
  </Tab>
  <Tab title="مورداعتماد">
    برای سندهای هم‌سایتی که عمدا به امتیازهای قوی‌تری نیاز دارند، `allow-same-origin` را روی `allow-scripts` اضافه می‌کند.
  </Tab>
</Tabs>

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
فقط زمانی از `trusted` استفاده کنید که سند embedded واقعا به رفتار same-origin نیاز دارد. برای بیشتر بازی‌ها و canvasهای تعاملی تولیدشده توسط agent، `scripts` انتخاب امن‌تری است.
</Warning>

URLهای embed مطلق خارجی `http(s)` به‌طور پیش‌فرض مسدود می‌مانند. اگر عمدا می‌خواهید `[embed url="https://..."]` صفحه‌های شخص ثالث را بارگیری کند، `gateway.controlUi.allowExternalEmbedUrls: true` را تنظیم کنید.

## عرض پیام چت

پیام‌های چت گروه‌بندی‌شده از یک max-width پیش‌فرض خوانا استفاده می‌کنند. استقرارهای مانیتور عریض می‌توانند بدون patch کردن CSS بسته‌بندی‌شده، با تنظیم `gateway.controlUi.chatMessageMaxWidth` آن را بازنویسی کنند:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

مقدار پیش از رسیدن به مرورگر اعتبارسنجی می‌شود. مقدارهای پشتیبانی‌شده شامل طول‌ها و درصدهای ساده مانند `960px` یا `82%`، به‌علاوه عبارت‌های عرض محدودشده `min(...)`، `max(...)`، `clamp(...)`، `calc(...)`، و `fit-content(...)` هستند.

## دسترسی tailnet (توصیه‌شده)

<Tabs>
  <Tab title="Tailscale Serve یکپارچه (ترجیحی)">
    Gateway را روی loopback نگه دارید و اجازه دهید Tailscale Serve آن را با HTTPS پروکسی کند:

    ```bash
    openclaw gateway --tailscale serve
    ```

    باز کنید:

    - `https://<magicdns>/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    به‌طور پیش‌فرض، درخواست‌های Serve مربوط به رابط کاربری کنترل/WebSocket می‌توانند وقتی `gateway.auth.allowTailscale` برابر `true` است از طریق headerهای هویت Tailscale (`tailscale-user-login`) احراز هویت شوند. OpenClaw هویت را با resolve کردن آدرس `x-forwarded-for` توسط `tailscale whois` و تطبیق آن با header بررسی می‌کند، و فقط وقتی این‌ها را می‌پذیرد که درخواست با headerهای `x-forwarded-*` مربوط به Tailscale به loopback برسد. برای نشست‌های operator رابط کاربری کنترل با هویت دستگاه مرورگر، این مسیر Serve تأییدشده همچنین رفت‌وبرگشت جفت‌سازی دستگاه را رد می‌کند؛ مرورگرهای بدون دستگاه و اتصال‌های node-role همچنان بررسی‌های معمول دستگاه را دنبال می‌کنند. اگر می‌خواهید حتی برای ترافیک Serve نیز credentialهای shared-secret صریح لازم باشد، `gateway.auth.allowTailscale: false` را تنظیم کنید. سپس از `gateway.auth.mode: "token"` یا `"password"` استفاده کنید.

    برای آن مسیر async هویت Serve، تلاش‌های ناموفق احراز هویت برای همان IP کلاینت و همان محدوده auth پیش از نوشتن rate-limit سریالی می‌شوند. بنابراین تلاش‌های بد همزمان از همان مرورگر می‌توانند روی درخواست دوم به‌جای دو عدم‌تطابق ساده که موازی رقابت می‌کنند، `retry later` نشان دهند.

    <Warning>
    احراز هویت Serve بدون token فرض می‌کند میزبان gateway مورداعتماد است. اگر کد محلی نامطمئن ممکن است روی آن میزبان اجرا شود، احراز هویت token/password را الزامی کنید.
    </Warning>

  </Tab>
  <Tab title="Bind به tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    سپس باز کنید:

    - `http://<tailscale-ip>:18789/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    راز مشترک مطابق را در تنظیمات UI جای‌گذاری کنید (به‌صورت `connect.params.auth.token` یا `connect.params.auth.password` ارسال می‌شود).

  </Tab>
</Tabs>

## HTTP ناامن

اگر داشبورد را از طریق HTTP ساده (`http://<lan-ip>` یا `http://<tailscale-ip>`) باز کنید، مرورگر در یک **زمینه غیرامن** اجرا می‌شود و WebCrypto را مسدود می‌کند. به‌طور پیش‌فرض، OpenClaw اتصال‌های رابط کاربری کنترل را بدون هویت دستگاه **مسدود** می‌کند.

استثناهای مستندشده:

- سازگاری HTTP ناامن فقط برای localhost با `gateway.controlUi.allowInsecureAuth=true`
- احراز هویت موفق operator رابط کاربری کنترل از طریق `gateway.auth.mode: "trusted-proxy"`
- حالت اضطراری `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**راه‌حل توصیه‌شده:** از HTTPS (Tailscale Serve) استفاده کنید یا UI را به‌صورت محلی باز کنید:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (روی میزبان gateway)

<AccordionGroup>
  <Accordion title="رفتار toggle احراز هویت ناامن">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` فقط یک toggle سازگاری محلی است:

    - به نشست‌های رابط کاربری کنترل روی localhost اجازه می‌دهد در زمینه‌های HTTP غیرامن بدون هویت دستگاه ادامه پیدا کنند.
    - بررسی‌های pairing را دور نمی‌زند.
    - الزامات هویت دستگاه remote (غیر-localhost) را سست نمی‌کند.

  </Accordion>
  <Accordion title="فقط حالت اضطراری">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` بررسی‌های هویت دستگاه رابط کاربری کنترل را غیرفعال می‌کند و یک کاهش امنیتی شدید است. پس از استفاده اضطراری، سریع آن را برگردانید.
    </Warning>

  </Accordion>
  <Accordion title="یادداشت trusted-proxy">
    - احراز هویت trusted-proxy موفق می‌تواند نشست‌های رابط کاربری کنترل مربوط به **operator** را بدون هویت دستگاه بپذیرد.
    - این مورد به نشست‌های رابط کاربری کنترل با نقش node-role گسترش پیدا نمی‌کند.
    - reverse proxyهای loopback روی همان میزبان همچنان احراز هویت trusted-proxy را برآورده نمی‌کنند؛ [احراز هویت trusted proxy](/fa/gateway/trusted-proxy-auth) را ببینید.

  </Accordion>
</AccordionGroup>

برای راهنمای تنظیم HTTPS، [Tailscale](/fa/gateway/tailscale) را ببینید.

## سیاست امنیت محتوا

رابط کاربری کنترل با سیاست سخت‌گیرانه `img-src` ارائه می‌شود: فقط assetهای **same-origin**، URLهای `data:`، و URLهای `blob:` تولیدشده محلی مجاز هستند. URLهای تصویر remote با `http(s)` و URLهای تصویر protocol-relative توسط مرورگر رد می‌شوند و هیچ واکشی شبکه‌ای صادر نمی‌کنند.

معنای عملی این رفتار:

- آواتارها و تصویرهایی که زیر مسیرهای نسبی (برای مثال `/avatars/<id>`) سرو می‌شوند همچنان رندر می‌شوند، از جمله routeهای آواتار احرازشده که UI آن‌ها را fetch و به URLهای `blob:` محلی تبدیل می‌کند.
- URLهای درون‌خطی `data:image/...` همچنان رندر می‌شوند (برای payloadهای درون‌پروتکلی مفید است).
- URLهای `blob:` محلی ساخته‌شده توسط رابط کاربری کنترل همچنان رندر می‌شوند.
- URLهای آواتار remote که توسط metadata کانال منتشر می‌شوند، در helperهای آواتار رابط کاربری کنترل حذف می‌شوند و با logo/badge داخلی جایگزین می‌شوند؛ بنابراین یک کانال compromised یا مخرب نمی‌تواند مرورگر operator را مجبور به fetch کردن تصویر remote دلخواه کند.

برای دریافت این رفتار نیازی نیست چیزی را تغییر دهید — همیشه فعال است و قابل پیکربندی نیست.

## احراز هویت route آواتار

وقتی auth مربوط به gateway پیکربندی شده باشد، endpoint آواتار رابط کاربری کنترل همان token مربوط به gateway را مانند بقیه API لازم دارد:

- `GET /avatar/<agentId>` تصویر آواتار را فقط به فراخواننده‌های احرازشده برمی‌گرداند. `GET /avatar/<agentId>?meta=1` metadata آواتار را تحت همان قاعده برمی‌گرداند.
- درخواست‌های احرازنشده به هرکدام از routeها رد می‌شوند (همسان با route sibling assistant-media). این کار جلوی نشت هویت agent از route آواتار را روی میزبان‌هایی می‌گیرد که در غیر این صورت محافظت‌شده‌اند.
- خود رابط کاربری کنترل هنگام fetch کردن آواتارها، token مربوط به gateway را به‌صورت bearer header ارسال می‌کند و از URLهای blob احرازشده استفاده می‌کند تا تصویر همچنان در داشبوردها رندر شود.

اگر auth مربوط به gateway را غیرفعال کنید (روی میزبان‌های اشتراکی توصیه نمی‌شود)، route آواتار نیز همسو با بقیه gateway بدون احراز هویت می‌شود.

## احراز هویت route رسانه دستیار

وقتی auth مربوط به gateway پیکربندی شده باشد، پیش‌نمایش‌های local-media دستیار از یک route دومرحله‌ای استفاده می‌کنند:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` به auth معمول operator رابط کاربری کنترل نیاز دارد. مرورگر هنگام بررسی دردسترس‌بودن، token مربوط به gateway را به‌صورت bearer header ارسال می‌کند.
- پاسخ‌های metadata موفق شامل یک `mediaTicket` کوتاه‌عمر هستند که به همان مسیر source دقیق محدود شده است.
- URLهای تصویر، صدا، ویدیو، و سند که در مرورگر رندر می‌شوند، به‌جای token یا password فعال gateway از `mediaTicket=<ticket>` استفاده می‌کنند. ticket به‌سرعت منقضی می‌شود و نمی‌تواند source متفاوتی را مجاز کند.

این کار رندر معمول رسانه را با elementهای media بومی مرورگر سازگار نگه می‌دارد، بدون اینکه credentialهای قابل‌استفاده‌مجدد gateway را در URLهای رسانه قابل‌مشاهده قرار دهد.

## ساخت UI

Gateway فایل‌های static را از `dist/control-ui` سرو می‌کند. آن‌ها را با این دستور بسازید:

```bash
pnpm ui:build
```

base مطلق اختیاری (وقتی URLهای ثابت asset می‌خواهید):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

برای توسعه محلی (dev server جداگانه):

```bash
pnpm ui:dev
```

سپس UI را به URL مربوط به Gateway WS خود اشاره دهید (مثلا `ws://127.0.0.1:18789`).

## صفحه خالی رابط کاربری کنترل

اگر مرورگر یک داشبورد خالی بارگیری می‌کند و DevTools خطای مفیدی نشان نمی‌دهد، ممکن است یک extension یا content script اولیه مانع evaluate شدن برنامه ماژول JavaScript شده باشد. صفحه static شامل یک پنل بازیابی HTML ساده است که وقتی `<openclaw-app>` پس از startup ثبت نشده باشد ظاهر می‌شود.

پس از تغییر محیط مرورگر، از action **تلاش دوباره** پنل استفاده کنید، یا پس از این بررسی‌ها دستی reload کنید:

- extensionهایی را که به همه صفحه‌ها inject می‌شوند غیرفعال کنید، به‌ویژه extensionهایی با content scriptهای `<all_urls>`.
- یک پنجره خصوصی، یک profile تمیز مرورگر، یا مرورگر دیگری را امتحان کنید.
- Gateway را در حال اجرا نگه دارید و همان URL داشبورد را پس از تغییر مرورگر بررسی کنید.

## اشکال‌زدایی/تست: dev server + Gateway remote

رابط کاربری کنترل فایل‌های static است؛ target مربوط به WebSocket قابل پیکربندی است و می‌تواند با HTTP origin متفاوت باشد. این زمانی مفید است که dev server مربوط به Vite را محلی می‌خواهید اما Gateway جای دیگری اجرا می‌شود.

<Steps>
  <Step title="راه‌اندازی dev server UI">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="باز کردن با gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    auth یک‌باره اختیاری (در صورت نیاز):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="یادداشت‌ها">
    - `gatewayUrl` پس از بارگذاری در localStorage ذخیره و از URL حذف می‌شود.
    - اگر یک نقطهٔ پایانی کامل `ws://` یا `wss://` را از طریق `gatewayUrl` می‌فرستید، مقدار `gatewayUrl` را URL-encode کنید تا مرورگر رشتهٔ پرس‌وجو را درست تجزیه کند.
    - هر زمان ممکن است، `token` باید از طریق قطعهٔ URL (`#token=...`) ارسال شود. قطعه‌ها به سرور فرستاده نمی‌شوند و این از نشت در لاگ‌های درخواست و Referer جلوگیری می‌کند. پارامترهای پرس‌وجوی قدیمی `?token=` همچنان برای سازگاری یک‌بار وارد می‌شوند، اما فقط به‌عنوان fallback، و بلافاصله پس از bootstrap حذف می‌شوند.
    - `password` فقط در حافظه نگه داشته می‌شود.
    - وقتی `gatewayUrl` تنظیم شده باشد، رابط کاربری به credentials پیکربندی یا محیط fallback نمی‌کند. `token` (یا `password`) را صریحاً ارائه کنید. نبود credentials صریح یک خطا است.
    - وقتی Gateway پشت TLS است (Tailscale Serve، پروکسی HTTPS، و غیره)، از `wss://` استفاده کنید.
    - `gatewayUrl` فقط در یک پنجرهٔ سطح بالا پذیرفته می‌شود (نه embedded) تا از clickjacking جلوگیری شود.
    - استقرارهای عمومی و غیر-loopback رابط کاربری کنترل باید `gateway.controlUi.allowedOrigins` را صریحاً تنظیم کنند (originهای کامل). بارگذاری‌های خصوصی LAN/Tailnet با همان origin از loopback، RFC1918/link-local، `.local`، `.ts.net`، یا میزبان‌های CGNAT در Tailscale بدون فعال‌سازی fallback سرآیند Host پذیرفته می‌شوند.
    - راه‌اندازی Gateway ممکن است originهای محلی مانند `http://localhost:<port>` و `http://127.0.0.1:<port>` را از bind و پورت مؤثر زمان اجرا seed کند، اما originهای مرورگر راه دور همچنان به ورودی‌های صریح نیاز دارند.
    - از `gateway.controlUi.allowedOrigins: ["*"]` استفاده نکنید مگر برای آزمون محلی کاملاً کنترل‌شده. این یعنی اجازه به هر origin مرورگر، نه «مطابقت با هر میزبانی که استفاده می‌کنم».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت fallback origin بر اساس سرآیند Host را فعال می‌کند، اما این یک حالت امنیتی خطرناک است.

  </Accordion>
</AccordionGroup>

نمونه:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
