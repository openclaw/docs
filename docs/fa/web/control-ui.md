رابط کنترل یک اپ تک‌صفحه‌ای کوچک **Vite + Lit** است که توسط Gateway ارائه می‌شود:

- پیش‌فرض: `http://<host>:18789/`
- پیشوند اختیاری: `gateway.controlUi.basePath` را تنظیم کنید (مثلاً `/openclaw`)

این رابط **مستقیماً با وب‌سوکت Gateway** روی همان پورت ارتباط برقرار می‌کند.

## باز کردن سریع (محلی)

اگر Gateway روی همان رایانه در حال اجراست، باز کنید:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (یا [http://localhost:18789/](http://localhost:18789/))

اگر صفحه بارگذاری نشد، ابتدا Gateway را شروع کنید: `openclaw gateway`.

احراز هویت هنگام دست‌دهی وب‌سوکت از این طریق ارائه می‌شود:

- `connect.params.auth.token`
- `connect.params.auth.password`
- سرآیندهای هویت Tailscale Serve وقتی `gateway.auth.allowTailscale: true` باشد
- سرآیندهای هویت پراکسی مورد اعتماد وقتی `gateway.auth.mode: "trusted-proxy"` باشد

پنل تنظیمات داشبورد یک توکن را برای نشست تب فعلی مرورگر و URL انتخاب‌شده Gateway نگه می‌دارد؛ گذرواژه‌ها پایدار ذخیره نمی‌شوند. فرایند راه‌اندازی اولیه معمولاً در اولین اتصال، برای احراز هویت shared-secret یک توکن gateway تولید می‌کند، اما وقتی `gateway.auth.mode` برابر `"password"` باشد، احراز هویت با گذرواژه هم کار می‌کند.

## جفت‌سازی دستگاه (اولین اتصال)

وقتی از یک مرورگر یا دستگاه جدید به رابط کنترل وصل می‌شوید، Gateway معمولاً به **تأیید جفت‌سازی یک‌باره** نیاز دارد. این یک اقدام امنیتی برای جلوگیری از دسترسی غیرمجاز است.

**چیزی که خواهید دید:** "disconnected (1008): pairing required"

<Steps>
  <Step title="فهرست کردن درخواست‌های در انتظار">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="تأیید با شناسه درخواست">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

اگر مرورگر با جزئیات احراز هویت تغییرکرده (نقش/دامنه‌ها/کلید عمومی) دوباره برای جفت‌سازی تلاش کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، دوباره `openclaw devices list` را اجرا کنید.

اگر مرورگر از قبل جفت شده باشد و آن را از دسترسی خواندن به دسترسی نوشتن/مدیر تغییر دهید، این به‌عنوان ارتقای تأیید در نظر گرفته می‌شود، نه اتصال مجدد بی‌صدا. OpenClaw تأیید قدیمی را فعال نگه می‌دارد، اتصال مجدد گسترده‌تر را مسدود می‌کند و از شما می‌خواهد مجموعه دامنه جدید را صریحاً تأیید کنید.

پس از تأیید، دستگاه به خاطر سپرده می‌شود و دیگر به تأیید دوباره نیاز ندارد، مگر اینکه آن را با `openclaw devices revoke --device <id> --role <role>` لغو کنید. برای چرخش و لغو توکن، [CLI دستگاه‌ها](/fa/cli/devices) را ببینید.

عامل‌های Paperclip که از طریق آداپتور `openclaw_gateway` وصل می‌شوند، از همان جریان تأیید اجرای اول استفاده می‌کنند. پس از تلاش اولیه اتصال، `openclaw devices approve --latest` را اجرا کنید تا درخواست در انتظار را پیش‌نمایش کنید، سپس فرمان چاپ‌شده `openclaw devices approve <requestId>` را دوباره اجرا کنید تا آن را تأیید کنید. برای یک gateway راه‌دور، مقدارهای صریح `--url` و `--token` را بدهید. برای پایدار نگه داشتن تأییدها در راه‌اندازی‌های مجدد، به‌جای اینکه Paperclip در هر اجرا یک هویت دستگاه موقت جدید تولید کند، یک `adapterConfig.devicePrivateKeyPem` پایدار در Paperclip پیکربندی کنید.

<Note>
- اتصال‌های مستقیم مرورگر از طریق local loopback (`127.0.0.1` / `localhost`) به‌صورت خودکار تأیید می‌شوند.
- وقتی `gateway.auth.allowTailscale: true` باشد، هویت Tailscale تأیید شود، و مرورگر هویت دستگاه خود را ارائه کند، Tailscale Serve می‌تواند رفت‌وبرگشت جفت‌سازی را برای نشست‌های اپراتور رابط کنترل رد کند.
- اتصال‌های مستقیم Tailnet، اتصال‌های مرورگر LAN، و پروفایل‌های مرورگر بدون هویت دستگاه همچنان به تأیید صریح نیاز دارند.
- هر پروفایل مرورگر یک شناسه دستگاه یکتا تولید می‌کند، بنابراین تغییر مرورگر یا پاک کردن داده‌های مرورگر نیازمند جفت‌سازی دوباره خواهد بود.

</Note>

## هویت شخصی (محلیِ مرورگر)

رابط کنترل از یک هویت شخصی برای هر مرورگر (نام نمایشی و آواتار) پشتیبانی می‌کند که برای انتساب در نشست‌های مشترک به پیام‌های خروجی متصل می‌شود. این هویت در ذخیره‌سازی مرورگر قرار دارد، به پروفایل مرورگر فعلی محدود است، و به دستگاه‌های دیگر همگام‌سازی نمی‌شود یا در سمت سرور، فراتر از فراداده معمول نویسندگی رونوشت برای پیام‌هایی که واقعاً ارسال می‌کنید، پایدار ذخیره نمی‌شود. پاک کردن داده‌های سایت یا تغییر مرورگر آن را به حالت خالی بازنشانی می‌کند.

همین الگوی محلیِ مرورگر برای بازنویسی آواتار دستیار نیز اعمال می‌شود. آواتارهای بارگذاری‌شده دستیار فقط در مرورگر محلی روی هویت حل‌شده توسط gateway قرار می‌گیرند و هرگز از طریق `config.patch` رفت‌وبرگشت نمی‌کنند. فیلد پیکربندی مشترک `ui.assistant.avatar` همچنان برای کلاینت‌های غیر UI که فیلد را مستقیماً می‌نویسند (مانند gatewayهای اسکریپت‌شده یا داشبوردهای سفارشی) در دسترس است.

## نقطه پایانی پیکربندی زمان اجرا

رابط کنترل تنظیمات زمان اجرای خود را از `/control-ui-config.json` دریافت می‌کند که نسبت به مسیر پایه رابط کنترل gateway حل می‌شود (برای مثال وقتی UI زیر `/__openclaw__/` ارائه شود، `/__openclaw__/control-ui-config.json`). این نقطه پایانی با همان احراز هویت gateway مانند بقیه سطح HTTP محافظت می‌شود: مرورگرهای احراز هویت‌نشده نمی‌توانند آن را دریافت کنند، و دریافت موفق نیازمند یک توکن/گذرواژه معتبر از قبل برای gateway، هویت Tailscale Serve، یا هویت پراکسی مورد اعتماد است.

## پشتیبانی زبان

رابط کنترل می‌تواند در اولین بارگذاری، خود را بر اساس زبان مرورگر شما بومی‌سازی کند. برای بازنویسی آن در آینده، **Overview -> Gateway Access -> Language** را باز کنید. انتخابگر زبان در کارت Gateway Access قرار دارد، نه زیر Appearance.

- زبان‌های پشتیبانی‌شده: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- ترجمه‌های غیرانگلیسی در مرورگر به‌صورت lazy-loaded بارگذاری می‌شوند.
- زبان انتخاب‌شده در ذخیره‌سازی مرورگر ذخیره می‌شود و در بازدیدهای آینده دوباره استفاده می‌شود.
- کلیدهای ترجمه ازدست‌رفته به انگلیسی برمی‌گردند.

ترجمه‌های مستندات برای همان مجموعه زبان‌های غیرانگلیسی تولید می‌شوند، اما انتخابگر زبان داخلی سایت مستندات در Mintlify به کدهای زبانی محدود است که Mintlify می‌پذیرد. مستندات تایلندی (`th`) و فارسی (`fa`) همچنان در مخزن انتشار تولید می‌شوند؛ ممکن است تا زمانی که Mintlify از این کدها پشتیبانی نکند، در آن انتخابگر ظاهر نشوند.

## تم‌های ظاهر

پنل Appearance تم‌های داخلی Claw، Knot، و Dash را به‌همراه یک جایگاه واردسازی tweakcn محلیِ مرورگر نگه می‌دارد. برای وارد کردن یک تم، [ویرایشگر tweakcn](https://tweakcn.com/editor/theme) را باز کنید، یک تم انتخاب یا ایجاد کنید، روی **Share** کلیک کنید، و لینک تم کپی‌شده را در Appearance جای‌گذاری کنید. واردکننده همچنین URLهای رجیستری `https://tweakcn.com/r/themes/<id>`، URLهای ویرایشگر مانند `https://tweakcn.com/editor/theme?theme=amethyst-haze`، مسیرهای نسبی `/themes/<id>`، شناسه‌های خام تم، و نام‌های پیش‌فرض تم مانند `amethyst-haze` را می‌پذیرد.

Appearance همچنین شامل یک تنظیم اندازه متن محلیِ مرورگر است. این تنظیم همراه با بقیه ترجیحات رابط کنترل ذخیره می‌شود، روی متن چت، متن composer، کارت‌های ابزار، و نوارهای کناری چت اعمال می‌شود، و ورودی‌های متن را دست‌کم 16px نگه می‌دارد تا Safari موبایل هنگام فوکوس به‌صورت خودکار بزرگ‌نمایی نکند.

تم‌های واردشده فقط در پروفایل مرورگر فعلی ذخیره می‌شوند. آن‌ها در پیکربندی gateway نوشته نمی‌شوند و بین دستگاه‌ها همگام‌سازی نمی‌شوند. جایگزین کردن تم واردشده همان یک جایگاه محلی را به‌روزرسانی می‌کند؛ پاک کردن آن، اگر تم واردشده انتخاب شده باشد، تم فعال را به Claw برمی‌گرداند.

    - به‌روزرسانی: اجرای یک به‌روزرسانی package/git + راه‌اندازی مجدد (`update.run`) همراه با گزارش راه‌اندازی مجدد، سپس polling `update.status` پس از اتصال دوباره برای تأیید نسخه gateway در حال اجرا.

  </Accordion>
  <Accordion title="یادداشت‌های پنل کارهای Cron">
    - برای کارهای ایزوله، پیش‌فرض تحویل روی اعلام خلاصه است. اگر اجرای فقط داخلی می‌خواهید، می‌توانید آن را به هیچ تغییر دهید.
    - وقتی اعلام انتخاب شده باشد، فیلدهای کانال/هدف ظاهر می‌شوند.
    - حالت Webhook از `delivery.mode = "webhook"` استفاده می‌کند و `delivery.to` روی یک URL معتبر Webhook از نوع HTTP(S) تنظیم می‌شود.
    - برای کارهای نشست اصلی، حالت‌های تحویل Webhook و هیچ در دسترس هستند.
    - کنترل‌های ویرایش پیشرفته شامل حذف پس از اجرا، پاک‌کردن بازنویسی عامل، گزینه‌های دقیق/پراکنده Cron، بازنویسی‌های مدل/تفکر عامل، و کلیدهای تحویل با بهترین تلاش هستند.
    - اعتبارسنجی فرم به‌صورت درون‌خطی و با خطاهای سطح فیلد انجام می‌شود؛ مقدارهای نامعتبر تا زمان اصلاح، دکمه ذخیره را غیرفعال می‌کنند.
    - برای ارسال یک توکن حامل اختصاصی، `cron.webhookToken` را تنظیم کنید؛ اگر حذف شود، Webhook بدون سرآیند احراز هویت ارسال می‌شود.
    - جایگزین منسوخ: برای مهاجرت کارهای قدیمی ذخیره‌شده با `notify: true` از `cron.webhook` به Webhook صریح برای هر کار یا تحویل تکمیل، `openclaw doctor --fix` را اجرا کنید.

## صفحه MCP

صفحه اختصاصی MCP یک نمای اپراتوری برای سرورهای MCP مدیریت‌شده توسط OpenClaw در زیر `mcp.servers` است. این صفحه خودش انتقال‌های MCP را شروع نمی‌کند؛ از آن برای بررسی و ویرایش پیکربندی ذخیره‌شده استفاده کنید، سپس وقتی به اثبات زنده سرور نیاز دارید از `openclaw mcp doctor --probe` استفاده کنید.

گردش کار معمول:

1. از نوار کناری **MCP** را باز کنید.
2. کارت‌های خلاصه را برای شمار کل، فعال، OAuth، و سرورهای فیلترشده بررسی کنید.
3. هر ردیف سرور را از نظر انتقال، فعال‌بودن، احراز هویت، فیلترها، مهلت‌های زمانی، و راهنمای فرمان بررسی کنید.
4. وقتی سروری باید پیکربندی‌شده باقی بماند اما از کشف زمان اجرا خارج شود، فعال‌بودن را تغییر دهید.
5. بخش پیکربندی محدوده‌دار `mcp` را برای تعریف‌های سرور، سرآیندها، مسیرهای TLS/mTLS، فراداده OAuth، فیلترهای ابزار، و فراداده نمایش Codex ویرایش کنید.
6. برای نوشتن پیکربندی از **ذخیره** استفاده کنید، یا وقتی Gateway در حال اجرا باید پیکربندی تغییریافته را اعمال کند از **ذخیره و انتشار** استفاده کنید.
7. وقتی فرایند ویرایش‌شده به عیب‌یابی ایستا، اثبات زنده، یا دفع زمان اجرای کش‌شده نیاز دارد، از ترمینال `openclaw mcp status --verbose`، `openclaw mcp doctor --probe`، یا `openclaw mcp reload` را اجرا کنید.

این صفحه مقدارهای شبیه URL را که حامل اعتبارنامه هستند پیش از نمایش پنهان می‌کند و نام سرورها را در قطعه‌های فرمان داخل کوتیشن می‌گذارد تا فرمان‌های کپی‌شده همچنان با فاصله‌ها یا نویسه‌های ویژه پوسته کار کنند. مرجع کامل CLI و پیکربندی در [MCP](/fa/cli/mcp) قرار دارد.

## زبانه فعالیت

زبانه فعالیت یک ناظر موقت و محلیِ مرورگر برای فعالیت زنده ابزارها است. این زبانه از همان جریان رویداد Gateway `session.tool` / ابزار مشتق می‌شود که کارت‌های ابزار Chat را تغذیه می‌کند؛ خانواده رویداد Gateway، نقطه پایانی، ذخیره فعالیت ماندگار، خوراک سنجه‌ها، یا جریان ناظر خارجی دیگری اضافه نمی‌کند.

ورودی‌های فعالیت فقط خلاصه‌های پاک‌سازی‌شده و پیش‌نمایش‌های خروجی پنهان‌سازی‌شده و کوتاه‌شده را نگه می‌دارند. مقدارهای آرگومان ابزار در وضعیت Activity ذخیره نمی‌شوند؛ رابط کاربری نشان می‌دهد که آرگومان‌ها پنهان هستند و فقط شمار فیلدهای آرگومان را ثبت می‌کند. فهرست درون‌حافظه‌ای از زبانه مرورگر فعلی پیروی می‌کند، در پیمایش داخل Control UI باقی می‌ماند، و با بارگذاری دوباره صفحه، تغییر نشست، یا **پاک‌کردن** بازنشانی می‌شود.

## رفتار Chat

<AccordionGroup>
  <Accordion title="معناشناسی ارسال و تاریخچه">
    - `chat.send` **غیرمسدودکننده** است: بلافاصله با `{ runId, status: "started" }` تأیید می‌کند و پاسخ از طریق رویدادهای `chat` پخش می‌شود. کلاینت‌های مورد اعتماد Control UI ممکن است فراداده اختیاری زمان‌بندی ACK را نیز برای عیب‌یابی محلی دریافت کنند.
    - بارگذاری‌های Chat تصویرها به‌علاوه فایل‌های غیر ویدئویی را می‌پذیرند. تصویرها مسیر تصویر بومی را حفظ می‌کنند؛ فایل‌های دیگر به‌عنوان رسانه مدیریت‌شده ذخیره می‌شوند و در تاریخچه به‌صورت پیوندهای پیوست نمایش داده می‌شوند.
    - ارسال دوباره با همان `idempotencyKey` هنگام اجرا `{ status: "in_flight" }` و پس از تکمیل `{ status: "ok" }` برمی‌گرداند.
    - پاسخ‌های `chat.history` برای ایمنی رابط کاربری محدود به اندازه هستند. وقتی ورودی‌های رونوشت بیش از حد بزرگ باشند، Gateway ممکن است فیلدهای متنی طولانی را کوتاه کند، بلوک‌های فراداده سنگین را حذف کند، و پیام‌های بیش‌ازحد بزرگ را با یک جای‌نگهدار (`[chat.history omitted: message too large]`) جایگزین کند.
    - وقتی یک پیام قابل مشاهده دستیار در `chat.history` کوتاه شده باشد، خواننده جانبی می‌تواند ورودی کامل رونوشتِ نمایش-نرمال‌سازی‌شده را در صورت نیاز از طریق `chat.message.get` با `sessionKey`، `agentId` فعال در صورت نیاز، و `messageId` رونوشت واکشی کند. اگر Gateway همچنان نتواند مقدار بیشتری برگرداند، خواننده به‌جای تکرار بی‌صدای پیش‌نمایش کوتاه‌شده، وضعیت صریحاً در دسترس نبودن را نشان می‌دهد.
    - تصویرهای دستیار/تولیدشده به‌عنوان ارجاع‌های رسانه مدیریت‌شده پایدار می‌شوند و از طریق URLهای رسانه Gateway احرازشده دوباره ارائه می‌شوند، بنابراین بارگذاری‌های دوباره به باقی‌ماندن بارهای تصویر base64 خام در پاسخ تاریخچه Chat وابسته نیستند.
    - هنگام رندر کردن `chat.history`، Control UI برچسب‌های دستورالعمل درون‌خطیِ فقط نمایشی را از متن قابل مشاهده دستیار حذف می‌کند (برای مثال `[[reply_to_*]]` و `[[audio_as_voice]]`)، محتوای XML فراخوانی ابزارِ متن ساده (شامل `<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>`، و بلوک‌های فراخوانی ابزار کوتاه‌شده)، و توکن‌های کنترل مدل ASCII/تمام‌عرض نشت‌کرده را حذف می‌کند، و ورودی‌های دستیار را که کل متن قابل مشاهده‌شان فقط توکن خاموش دقیق `NO_REPLY` / `no_reply` یا توکن تأیید Heartbeat یعنی `HEARTBEAT_OK` است کنار می‌گذارد.
    - هنگام یک ارسال فعال و تازه‌سازی نهایی تاریخچه، اگر `chat.history` برای مدت کوتاهی یک نماگرفت قدیمی‌تر برگرداند، نمای Chat پیام‌های خوش‌بینانه محلی کاربر/دستیار را قابل مشاهده نگه می‌دارد؛ وقتی تاریخچه Gateway به‌روز شد، رونوشت رسمی جای آن پیام‌های محلی را می‌گیرد.
    - رویدادهای زنده `chat` وضعیت تحویل هستند، در حالی که `chat.history` از رونوشت ماندگار نشست بازسازی می‌شود. پس از رویدادهای نهایی ابزار، Control UI تاریخچه را دوباره بارگذاری می‌کند و فقط یک دنباله خوش‌بینانه کوچک را ادغام می‌کند؛ مرز رونوشت در [WebChat](/fa/web/webchat) مستند شده است.
    - `chat.inject` یک یادداشت دستیار را به رونوشت نشست اضافه می‌کند و یک رویداد `chat` را برای به‌روزرسانی‌های فقط رابط کاربری پخش می‌کند (بدون اجرای عامل، بدون تحویل کانال).
    - سرآیند Chat فیلتر عامل را پیش از انتخابگر نشست نشان می‌دهد، و انتخابگر نشست به عامل انتخاب‌شده محدود است. تغییر عامل‌ها فقط نشست‌های وابسته به آن عامل را نشان می‌دهد و وقتی هنوز نشست‌های داشبورد ذخیره‌شده ندارد، به نشست اصلی همان عامل بازمی‌گردد.
    - در عرض‌های دسکتاپ، کنترل‌های Chat روی یک ردیف فشرده می‌مانند و هنگام پیمایش به پایین رونوشت جمع می‌شوند؛ پیمایش به بالا، بازگشت به ابتدا، یا رسیدن به انتها کنترل‌ها را بازیابی می‌کند.
    - پیام‌های متوالی تکراری که فقط متن دارند، به‌صورت یک حباب با نشان شمارش رندر می‌شوند. پیام‌هایی که تصویر، پیوست، خروجی ابزار، یا پیش‌نمایش canvas دارند جمع نمی‌شوند.
    - انتخابگرهای مدل و تفکر در سرآیند Chat نشست فعال را بلافاصله از طریق `sessions.patch` وصله می‌کنند؛ آن‌ها بازنویسی‌های ماندگار نشست هستند، نه گزینه‌های ارسال فقط برای یک نوبت.
    - اگر هنگامی پیام بفرستید که تغییر انتخابگر مدل برای همان نشست هنوز در حال ذخیره شدن باشد، نگارنده پیش از فراخوانی `chat.send` منتظر وصله آن نشست می‌ماند تا ارسال از مدل انتخاب‌شده استفاده کند.
    - تایپ `/new` در Control UI همان نشست تازه داشبورد را مثل New Chat ایجاد می‌کند و به آن تغییر می‌دهد، مگر وقتی `session.dmScope: "main"` پیکربندی شده باشد و والد فعلی نشست اصلی عامل باشد؛ در آن حالت نشست اصلی را درجا بازنشانی می‌کند. تایپ `/reset` بازنشانی درجا و صریح Gateway را برای نشست فعلی نگه می‌دارد.
    - انتخابگر مدل Chat نمای مدل پیکربندی‌شده Gateway را درخواست می‌کند. اگر `agents.defaults.models` وجود داشته باشد، آن فهرست مجاز انتخابگر را هدایت می‌کند، از جمله ورودی‌های `provider/*` که کاتالوگ‌های محدوده‌دار به ارائه‌دهنده را پویا نگه می‌دارند. در غیر این صورت انتخابگر ورودی‌های صریح `models.providers.*.models` به‌علاوه ارائه‌دهندگانی با احراز هویت قابل استفاده را نشان می‌دهد. کاتالوگ کامل از طریق RPC عیب‌یابی `models.list` با `view: "all"` در دسترس می‌ماند.
    - وقتی گزارش‌های تازه مصرف نشست Gateway شامل توکن‌های زمینه فعلی باشند، ناحیه نگارنده Chat یک نشانگر فشرده مصرف زمینه نشان می‌دهد. در فشار بالای زمینه به سبک هشدار تغییر می‌کند و، در سطوح توصیه‌شده Compaction، یک دکمه فشرده نشان می‌دهد که مسیر عادی Compaction نشست را اجرا می‌کند. نماگرفت‌های توکن کهنه تا زمانی که Gateway دوباره مصرف تازه گزارش کند پنهان می‌شوند.

  </Accordion>
  <Accordion title="حالت گفت‌وگو (بی‌درنگ مرورگر)">
    حالت گفت‌وگو از یک ارائه‌دهنده صدای بی‌درنگ ثبت‌شده استفاده می‌کند. OpenAI را با `talk.realtime.provider: "openai"` به‌علاوه یک نمایه احراز هویت کلید API از نوع `openai`، `talk.realtime.providers.openai.apiKey`، یا `OPENAI_API_KEY` پیکربندی کنید؛ نمایه‌های OAuth مربوط به OpenAI صدای بی‌درنگ را پیکربندی نمی‌کنند. Google را با `talk.realtime.provider: "google"` به‌علاوه `talk.realtime.providers.google.apiKey` پیکربندی کنید. مرورگر هرگز کلید API استاندارد ارائه‌دهنده را دریافت نمی‌کند. OpenAI یک راز موقت کلاینت Realtime برای WebRTC دریافت می‌کند. Google Live یک توکن احراز هویت Live محدود و یک‌بارمصرف برای نشست WebSocket مرورگر دریافت می‌کند، همراه با دستورالعمل‌ها و اعلان‌های ابزار که توسط Gateway در توکن قفل شده‌اند. ارائه‌دهندگانی که فقط یک پل بی‌درنگ پشتیبان را ارائه می‌کنند، از طریق انتقال رله Gateway اجرا می‌شوند، بنابراین اعتبارنامه‌ها و سوکت‌های فروشنده در سمت سرور می‌مانند در حالی که صدای مرورگر از طریق RPCهای احرازشده Gateway جابه‌جا می‌شود. اعلان نشست Realtime توسط Gateway مونتاژ می‌شود؛ `talk.client.create` بازنویسی دستورالعمل ارائه‌شده توسط فراخواننده را نمی‌پذیرد.

    نگارنده Chat یک دکمه گزینه‌های گفت‌وگو کنار دکمه شروع/توقف گفت‌وگو دارد. گزینه‌ها روی نشست گفت‌وگوی بعدی اعمال می‌شوند و می‌توانند ارائه‌دهنده، انتقال، مدل، صدا، تلاش استدلال، آستانه VAD، مدت سکوت، و فاصله‌گذاری پیشوند را بازنویسی کنند. وقتی گزینه‌ای خالی باشد، Gateway در صورت وجود از پیش‌فرض‌های پیکربندی‌شده یا پیش‌فرض ارائه‌دهنده استفاده می‌کند. انتخاب رله Gateway مسیر رله پشتیبان را اجباری می‌کند؛ انتخاب WebRTC نشست را تحت مالکیت کلاینت نگه می‌دارد و اگر ارائه‌دهنده نتواند نشست مرورگر ایجاد کند، به‌جای بازگشت بی‌صدا به رله، شکست می‌خورد.

    در نگارنده Chat، کنترل گفت‌وگو دکمه موج‌ها کنار دکمه دیکته میکروفون است. وقتی گفت‌وگو شروع می‌شود، ردیف وضعیت نگارنده `Connecting Talk...` را نشان می‌دهد، سپس هنگام اتصال صدا `Talk live`، یا وقتی یک فراخوانی ابزار بی‌درنگ در حال مشورت با مدل بزرگ‌تر پیکربندی‌شده از طریق `talk.client.toolCall` است `Asking OpenClaw...` را نشان می‌دهد.

    آزمون زنده نگه‌دارنده: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` پل WebSocket پشتیبان OpenAI، تبادل SDP مرورگر WebRTC OpenAI، راه‌اندازی WebSocket مرورگر با توکن محدود Google Live، و آداپتر مرورگر رله Gateway با رسانه میکروفون ساختگی را راستی‌آزمایی می‌کند. این فرمان فقط وضعیت ارائه‌دهنده را چاپ می‌کند و رازها را ثبت نمی‌کند.

  </Accordion>
  <Accordion title="توقف و لغو">
    - روی **توقف** کلیک کنید (`chat.abort` را فراخوانی می‌کند).
    - تا وقتی یک اجرا فعال است، پیگیری‌های عادی در صف قرار می‌گیرند. روی **هدایت** در یک پیام صف‌شده کلیک کنید تا آن پیگیری را به نوبت در حال اجرا تزریق کنید.
    - برای لغو خارج از باند، `/stop` را تایپ کنید (یا عبارت‌های لغو مستقل مانند `stop`، `stop action`، `stop run`، `stop openclaw`، `please stop`).
    - `chat.abort` از `{ sessionKey }` (بدون `runId`) برای لغو همه اجراهای فعال آن نشست پشتیبانی می‌کند.

  </Accordion>

    - وقتی خروجی بافرشده وجود داشته باشد، Gateway متن بخشیِ لغوشده دستیار را در تاریخچه رونوشت پایدار می‌کند.
    - ورودی‌های پایدارشده شامل فراداده لغو هستند تا مصرف‌کنندگان رونوشت بتوانند بخش‌های لغوشده را از خروجی تکمیل عادی تشخیص دهند.

  </Accordion>
</AccordionGroup>

## نصب PWA و Web Push

Control UI یک `manifest.webmanifest` و یک سرویس‌ورکر عرضه می‌کند، بنابراین مرورگرهای مدرن می‌توانند آن را به‌عنوان یک PWA مستقل نصب کنند. Web Push به Gateway اجازه می‌دهد PWA نصب‌شده را حتی وقتی زبانه یا پنجره مرورگر باز نیست، با اعلان‌ها بیدار کند.

اگر صفحه درست پس از به‌روزرسانی OpenClaw پیام **ناسازگاری پروتکل** را نشان می‌دهد، ابتدا داشبورد را با `openclaw dashboard` دوباره باز کنید و صفحه را سخت‌نوسازی کنید. اگر همچنان شکست خورد، داده‌های سایت را برای مبدأ داشبورد پاک کنید یا در یک پنجره مرورگر خصوصی آزمایش کنید؛ یک زبانه قدیمی یا کش سرویس‌ورکر مرورگر می‌تواند همچنان بسته Control UI پیش از به‌روزرسانی را در برابر Gateway جدیدتر اجرا کند.

| سطح                                                   | کاری که انجام می‌دهد                                             |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | مانیفست PWA. مرورگرها پس از در دسترس شدن آن، گزینه «نصب برنامه» را ارائه می‌کنند. |
| `ui/public/sw.js`                                     | سرویس‌ورکری که رویدادهای `push` و کلیک‌های اعلان را مدیریت می‌کند. |
| `push/vapid-keys.json` (زیر دایرکتوری وضعیت OpenClaw) | جفت‌کلید VAPID که به‌صورت خودکار تولید می‌شود و برای امضای محموله‌های Web Push استفاده می‌شود. |
| `push/web-push-subscriptions.json`                    | نقاط پایانی اشتراک مرورگر که پایدارسازی شده‌اند.                  |

وقتی می‌خواهید کلیدها را ثابت نگه دارید (برای استقرارهای چندمیزبانه، چرخش محرمانه‌ها، یا آزمون‌ها)، جفت‌کلید VAPID را از طریق متغیرهای محیطی روی فرایند Gateway بازنویسی کنید:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (پیش‌فرض `https://openclaw.ai` است)

Control UI از این متدهای Gateway با محدوده کنترل‌شده برای ثبت و آزمودن اشتراک‌های مرورگر استفاده می‌کند:

- `push.web.vapidPublicKey` — کلید عمومی VAPID فعال را دریافت می‌کند.
- `push.web.subscribe` — یک `endpoint` به‌همراه `keys.p256dh`/`keys.auth` ثبت می‌کند.
- `push.web.unsubscribe` — یک نقطه پایانی ثبت‌شده را حذف می‌کند.
- `push.web.test` — یک اعلان آزمایشی به اشتراک فراخواننده می‌فرستد.

<Note>
Web Push از مسیر رله APNS در iOS مستقل است (برای push متکی به رله، [پیکربندی](/fa/gateway/configuration) را ببینید) و همچنین از متد موجود `push.test` که جفت‌سازی موبایل بومی را هدف می‌گیرد جداست.
</Note>

## جاسازی‌های میزبانی‌شده

پیام‌های دستیار می‌توانند محتوای وب میزبانی‌شده را به‌صورت درون‌خطی با کد کوتاه `[embed ...]` رندر کنند. سیاست sandbox iframe با `gateway.controlUi.embedSandbox` کنترل می‌شود:

<Tabs>
  <Tab title="strict">
    اجرای اسکریپت را داخل جاسازی‌های میزبانی‌شده غیرفعال می‌کند.
  </Tab>
  <Tab title="scripts (default)">
    جاسازی‌های تعاملی را مجاز می‌کند، درحالی‌که جداسازی مبدا حفظ می‌شود؛ این حالت پیش‌فرض است و معمولا برای بازی‌ها/ویجت‌های مرورگری خودبسنده کافی است.
  </Tab>
  <Tab title="trusted">
    افزون بر `allow-scripts`، برای سندهای هم‌سایتی که عمدا به امتیازهای قوی‌تر نیاز دارند، `allow-same-origin` را اضافه می‌کند.
  </Tab>
</Tabs>

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
فقط زمانی از `trusted` استفاده کنید که سند جاسازی‌شده واقعا به رفتار هم‌مبدا نیاز دارد. برای بیشتر بازی‌های تولیدشده توسط عامل و canvasهای تعاملی، `scripts` گزینه امن‌تری است.
</Warning>

URLهای جاسازی خارجی مطلق `http(s)` به‌صورت پیش‌فرض مسدود می‌مانند. اگر عمدا می‌خواهید `[embed url="https://..."]` صفحه‌های شخص ثالث را بارگذاری کند، `gateway.controlUi.allowExternalEmbedUrls: true` را تنظیم کنید.

## عرض پیام چت

پیام‌های چت گروه‌بندی‌شده از یک max-width خوانا به‌عنوان پیش‌فرض استفاده می‌کنند. استقرارهای مانیتور عریض می‌توانند بدون وصله‌کردن CSS بسته‌بندی‌شده، با تنظیم `gateway.controlUi.chatMessageMaxWidth` آن را بازنویسی کنند:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

مقدار پیش از رسیدن به مرورگر اعتبارسنجی می‌شود. مقادیر پشتیبانی‌شده شامل طول‌ها و درصدهای ساده مانند `960px` یا `82%`، به‌علاوه عبارت‌های عرض محدودشده `min(...)`، `max(...)`، `clamp(...)`، `calc(...)` و `fit-content(...)` هستند.

## دسترسی Tailnet (توصیه‌شده)

<Tabs>
  <Tab title="Tailscale Serve یکپارچه (ترجیحی)">
    Gateway را روی loopback نگه دارید و اجازه دهید Tailscale Serve آن را با HTTPS پروکسی کند:

    ```bash
    openclaw gateway --tailscale serve
    ```

    باز کنید:

    - `https://<magicdns>/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    به‌صورت پیش‌فرض، درخواست‌های Control UI/WebSocket Serve می‌توانند از طریق سرآیندهای هویت Tailscale (`tailscale-user-login`) احراز هویت شوند، وقتی `gateway.auth.allowTailscale` برابر `true` باشد. OpenClaw هویت را با resolve کردن نشانی `x-forwarded-for` از طریق `tailscale whois` و تطبیق آن با سرآیند بررسی می‌کند، و این موارد را فقط وقتی می‌پذیرد که درخواست با سرآیندهای `x-forwarded-*` مربوط به Tailscale به loopback برسد. برای نشست‌های اپراتور Control UI با هویت دستگاه مرورگر، این مسیر Serve تاییدشده همچنین رفت‌وبرگشت جفت‌سازی دستگاه را رد می‌کند؛ مرورگرهای بدون دستگاه و اتصال‌های نقش node همچنان بررسی‌های عادی دستگاه را دنبال می‌کنند. اگر می‌خواهید حتی برای ترافیک Serve هم اعتبارنامه‌های صریح secret مشترک لازم باشد، `gateway.auth.allowTailscale: false` را تنظیم کنید. سپس از `gateway.auth.mode: "token"` یا `"password"` استفاده کنید.

    برای آن مسیر ناهمگام هویت Serve، تلاش‌های ناموفق احراز هویت برای همان IP مشتری و محدوده احراز هویت، پیش از نوشتن‌های محدودیت نرخ، سریالی می‌شوند. بنابراین تلاش‌های مجدد بد همزمان از همان مرورگر می‌توانند در درخواست دوم به‌جای دو عدم‌تطابق ساده که موازی با هم رقابت می‌کنند، `retry later` نشان دهند.

    <Warning>
    احراز هویت Serve بدون توکن فرض می‌کند میزبان gateway مورد اعتماد است. اگر کد محلی نامطمئن ممکن است روی آن میزبان اجرا شود، احراز هویت توکن/رمزعبور را الزامی کنید.
    </Warning>

  </Tab>
  <Tab title="اتصال به tailnet + توکن">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    سپس باز کنید:

    - `http://<tailscale-ip>:18789/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    secret مشترک مطابق را در تنظیمات UI جای‌گذاری کنید (به‌صورت `connect.params.auth.token` یا `connect.params.auth.password` ارسال می‌شود).

  </Tab>
</Tabs>

## HTTP ناامن

اگر داشبورد را از طریق HTTP ساده (`http://<lan-ip>` یا `http://<tailscale-ip>`) باز کنید، مرورگر در یک **زمینه غیرامن** اجرا می‌شود و WebCrypto را مسدود می‌کند. به‌صورت پیش‌فرض، OpenClaw اتصال‌های Control UI بدون هویت دستگاه را **مسدود** می‌کند.

استثناهای مستندشده:

- سازگاری HTTP ناامن فقط برای localhost با `gateway.controlUi.allowInsecureAuth=true`
- احراز هویت موفق اپراتور Control UI از طریق `gateway.auth.mode: "trusted-proxy"`
- حالت اضطراری `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**راه‌حل پیشنهادی:** از HTTPS (Tailscale Serve) استفاده کنید یا UI را به‌صورت محلی باز کنید:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (روی میزبان gateway)

<AccordionGroup>
  <Accordion title="رفتار کلید احراز هویت ناامن">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` فقط یک کلید سازگاری محلی است:

    - اجازه می‌دهد نشست‌های Control UI روی localhost در زمینه‌های HTTP غیرامن، بدون هویت دستگاه ادامه پیدا کنند.
    - بررسی‌های جفت‌سازی را دور نمی‌زند.
    - الزامات هویت دستگاه راه دور (غیر از localhost) را سست نمی‌کند.

  </Accordion>
  <Accordion title="فقط حالت اضطراری">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` بررسی‌های هویت دستگاه Control UI را غیرفعال می‌کند و یک کاهش امنیتی شدید است. پس از استفاده اضطراری، سریع آن را برگردانید.
    </Warning>

  </Accordion>
  <Accordion title="یادداشت پروکسی مورد اعتماد">
    - احراز هویت موفق trusted-proxy می‌تواند نشست‌های Control UI **اپراتور** را بدون هویت دستگاه بپذیرد.
    - این موضوع به نشست‌های Control UI با نقش node تعمیم پیدا نمی‌کند.
    - پروکسی‌های معکوس loopback روی همان میزبان همچنان احراز هویت trusted-proxy را برآورده نمی‌کنند؛ [احراز هویت پروکسی مورد اعتماد](/fa/gateway/trusted-proxy-auth) را ببینید.

  </Accordion>
</AccordionGroup>

برای راهنمایی راه‌اندازی HTTPS، [Tailscale](/fa/gateway/tailscale) را ببینید.

## سیاست امنیت محتوا

Control UI با سیاست سخت‌گیرانه `img-src` عرضه می‌شود: فقط دارایی‌های **هم‌مبدا**، URLهای `data:` و URLهای `blob:` تولیدشده به‌صورت محلی مجاز هستند. URLهای تصویر راه دور `http(s)` و وابسته به پروتکل توسط مرورگر رد می‌شوند و fetch شبکه صادر نمی‌کنند.

معنای عملی این موضوع:

- آواتارها و تصویرهایی که زیر مسیرهای نسبی ارائه می‌شوند (برای مثال `/avatars/<id>`) همچنان رندر می‌شوند، از جمله مسیرهای آواتار احراز هویت‌شده که UI آن‌ها را fetch می‌کند و به URLهای محلی `blob:` تبدیل می‌کند.
- URLهای درون‌خطی `data:image/...` همچنان رندر می‌شوند (برای محموله‌های درون‌پروتکلی مفید است).
- URLهای محلی `blob:` ساخته‌شده توسط Control UI همچنان رندر می‌شوند.
- URLهای آواتار راه دور که توسط فراداده کانال منتشر می‌شوند، در helperهای آواتار Control UI حذف و با لوگو/نشان داخلی جایگزین می‌شوند، بنابراین یک کانال آلوده یا مخرب نمی‌تواند مرورگر اپراتور را وادار به fetch کردن تصویر راه دور دلخواه کند.

برای به‌دست‌آوردن این رفتار نیازی نیست چیزی را تغییر دهید — همیشه فعال است و قابل پیکربندی نیست.

## احراز هویت مسیر آواتار

وقتی احراز هویت gateway پیکربندی شده باشد، endpoint آواتار Control UI همان توکن gateway را که بقیه API استفاده می‌کند، لازم دارد:

- `GET /avatar/<agentId>` تصویر آواتار را فقط به فراخواننده‌های احراز هویت‌شده برمی‌گرداند. `GET /avatar/<agentId>?meta=1` فراداده آواتار را با همان قاعده برمی‌گرداند.
- درخواست‌های احراز هویت‌نشده به هر یک از دو مسیر رد می‌شوند (مطابق مسیر خواهر assistant-media). این کار مانع نشت هویت عامل از مسیر آواتار روی میزبان‌هایی می‌شود که در غیر این صورت محافظت شده‌اند.
- خود Control UI هنگام fetch کردن آواتارها، توکن gateway را به‌صورت سرآیند bearer ارسال می‌کند و از URLهای blob احراز هویت‌شده استفاده می‌کند تا تصویر همچنان در داشبوردها رندر شود.

اگر احراز هویت gateway را غیرفعال کنید (روی میزبان‌های مشترک توصیه نمی‌شود)، مسیر آواتار نیز مطابق بقیه gateway بدون احراز هویت می‌شود.

## احراز هویت مسیر رسانه دستیار

وقتی احراز هویت gateway پیکربندی شده باشد، پیش‌نمایش‌های رسانه محلی دستیار از یک مسیر دو مرحله‌ای استفاده می‌کنند:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` احراز هویت عادی اپراتور Control UI را لازم دارد. مرورگر هنگام بررسی دسترس‌پذیری، توکن gateway را به‌صورت سرآیند bearer ارسال می‌کند.
- پاسخ‌های موفق فراداده شامل یک `mediaTicket` کوتاه‌عمر هستند که به همان مسیر منبع دقیق محدود شده است.
- URLهای تصویر، صوت، ویدئو و سند که در مرورگر رندر می‌شوند، به‌جای توکن یا رمزعبور فعال gateway از `mediaTicket=<ticket>` استفاده می‌کنند. ticket به‌سرعت منقضی می‌شود و نمی‌تواند منبع متفاوتی را مجاز کند.

این کار رندر عادی رسانه را با عناصر رسانه بومی مرورگر سازگار نگه می‌دارد، بدون اینکه اعتبارنامه‌های gateway قابل استفاده مجدد را در URLهای رسانه قابل مشاهده قرار دهد.

## ساخت UI

Gateway فایل‌های ایستا را از `dist/control-ui` ارائه می‌کند. آن‌ها را با این دستور بسازید:

```bash
pnpm ui:build
```

base مطلق اختیاری (وقتی URLهای دارایی ثابت می‌خواهید):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

برای توسعه محلی (سرور توسعه جداگانه):

```bash
pnpm ui:dev
```

سپس UI را به URL WS مربوط به Gateway خود اشاره دهید (مثلا `ws://127.0.0.1:18789`).

## صفحه خالی Control UI

اگر مرورگر یک داشبورد خالی بارگذاری کند و DevTools خطای مفیدی نشان ندهد، ممکن است یک افزونه یا اسکریپت محتوای زودهنگام مانع ارزیابی برنامه ماژول JavaScript شده باشد. صفحه ایستا شامل یک پنل بازیابی HTML ساده است که وقتی `<openclaw-app>` پس از راه‌اندازی ثبت نشده باشد، ظاهر می‌شود.

پس از تغییر محیط مرورگر، از اقدام **تلاش دوباره** در پنل استفاده کنید، یا پس از این بررسی‌ها به‌صورت دستی بارگذاری مجدد کنید:

- افزونه‌هایی را که به همه صفحه‌ها تزریق می‌کنند غیرفعال کنید، به‌ویژه افزونه‌هایی با اسکریپت‌های محتوای `<all_urls>`.
- یک پنجره خصوصی، یک پروفایل مرورگر پاک، یا مرورگری دیگر را امتحان کنید.
- Gateway را در حال اجرا نگه دارید و پس از تغییر مرورگر، همان URL داشبورد را بررسی کنید.

## اشکال‌زدایی/آزمون: سرور توسعه + Gateway راه دور

Control UI فایل‌های ایستا است؛ هدف WebSocket قابل پیکربندی است و می‌تواند با مبدا HTTP متفاوت باشد. این زمانی مفید است که سرور توسعه Vite را به‌صورت محلی می‌خواهید اما Gateway جای دیگری اجرا می‌شود.

<Steps>
  <Step title="راه‌اندازی سرور توسعه UI">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="باز کردن با gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    احراز هویت یک‌باره اختیاری (در صورت نیاز):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="نکات">
    - `gatewayUrl` پس از بارگذاری در localStorage ذخیره و از URL حذف می‌شود.
    - اگر یک نقطه پایانی کامل `ws://` یا `wss://` را از طریق `gatewayUrl` ارسال می‌کنید، مقدار `gatewayUrl` را URL-encode کنید تا مرورگر query string را به‌درستی تجزیه کند.
    - هر زمان ممکن است، `token` باید از طریق fragment در URL (`#token=...`) ارسال شود. fragmentها به سرور فرستاده نمی‌شوند و این از نشت در request-log و Referer جلوگیری می‌کند. پارامترهای query قدیمی `?token=` همچنان برای سازگاری یک‌بار import می‌شوند، اما فقط به‌عنوان fallback، و بلافاصله پس از bootstrap حذف می‌شوند.
    - `password` فقط در حافظه نگه داشته می‌شود.
    - وقتی `gatewayUrl` تنظیم شده باشد، UI به اعتبارنامه‌های config یا environment fallback نمی‌کند. `token` (یا `password`) را صریحا ارائه کنید. نبود اعتبارنامه‌های صریح یک خطا است.
    - وقتی Gateway پشت TLS است (Tailscale Serve، پروکسی HTTPS و غیره)، از `wss://` استفاده کنید.
    - `gatewayUrl` فقط در یک پنجره سطح بالا پذیرفته می‌شود (نه embedded) تا از clickjacking جلوگیری شود.
    - استقرارهای عمومی و غیر loopback برای Control UI باید `gateway.controlUi.allowedOrigins` را صریحا تنظیم کنند (originهای کامل). بارگذاری‌های خصوصی same-origin در LAN/Tailnet از loopback، RFC1918/link-local، `.local`، `.ts.net`، یا میزبان‌های Tailscale CGNAT بدون فعال‌سازی Host-header fallback پذیرفته می‌شوند.
    - راه‌اندازی Gateway ممکن است originهای محلی مانند `http://localhost:<port>` و `http://127.0.0.1:<port>` را از bind و port مؤثر runtime مقداردهی اولیه کند، اما originهای مرورگر راه‌دور همچنان به ورودی‌های صریح نیاز دارند.
    - از `gateway.controlUi.allowedOrigins: ["*"]` استفاده نکنید، مگر برای آزمایش محلی کاملا کنترل‌شده. معنای آن اجازه دادن به هر origin مرورگر است، نه «مطابقت با هر میزبانی که استفاده می‌کنم».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت Host-header origin fallback را فعال می‌کند، اما این یک حالت امنیتی خطرناک است.

  </Accordion>
</AccordionGroup>

مثال:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
