رابط کاربری کنترل یک برنامه تک‌صفحه‌ای کوچک **Vite + Lit** است که توسط Gateway ارائه می‌شود:

- پیش‌فرض: `http://<host>:18789/`
- پیشوند اختیاری: `gateway.controlUi.basePath` را تنظیم کنید (مثلاً `/openclaw`)

این برنامه **مستقیماً با Gateway WebSocket** روی همان پورت ارتباط برقرار می‌کند.

## باز کردن سریع (محلی)

اگر Gateway روی همان رایانه در حال اجرا است، باز کنید:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (یا [http://localhost:18789/](http://localhost:18789/))

اگر صفحه بارگذاری نشد، ابتدا Gateway را اجرا کنید: `openclaw gateway`.

<Note>
در اتصال‌های LAN بومی Windows، Windows Firewall یا Group Policy مدیریت‌شده توسط سازمان همچنان می‌تواند URL اعلام‌شده LAN را مسدود کند، حتی وقتی `127.0.0.1` روی میزبان Gateway کار می‌کند. روی میزبان Windows دستور `openclaw gateway status --deep` را اجرا کنید؛ این دستور پورت‌های احتمالاً مسدود، ناسازگاری‌های پروفایل، و قواعد فایروال محلی را که ممکن است policy نادیده بگیرد گزارش می‌کند.
</Note>

احراز هویت هنگام دست‌دهی WebSocket از طریق موارد زیر ارائه می‌شود:

- `connect.params.auth.token`
- `connect.params.auth.password`
- سرآیندهای هویت Tailscale Serve وقتی `gateway.auth.allowTailscale: true` باشد
- سرآیندهای هویت پراکسی مورداعتماد وقتی `gateway.auth.mode: "trusted-proxy"` باشد

پنل تنظیمات داشبورد یک توکن را برای نشست تب فعلی مرورگر و URL انتخاب‌شده Gateway نگه می‌دارد؛ گذرواژه‌ها پایدار ذخیره نمی‌شوند. Onboarding معمولاً در اولین اتصال، برای احراز هویت راز مشترک یک توکن gateway تولید می‌کند، اما وقتی `gateway.auth.mode` برابر `"password"` باشد احراز هویت با گذرواژه هم کار می‌کند.

## جفت‌سازی دستگاه (اولین اتصال)

وقتی از یک مرورگر یا دستگاه جدید به رابط کاربری کنترل وصل می‌شوید، Gateway معمولاً به **تأیید جفت‌سازی یک‌باره** نیاز دارد. این یک اقدام امنیتی برای جلوگیری از دسترسی غیرمجاز است.

**آنچه می‌بینید:** "disconnected (1008): pairing required"

<Steps>
  <Step title="فهرست کردن درخواست‌های در انتظار">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="تأیید با شناسه درخواست">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

اگر مرورگر جفت‌سازی را با جزئیات احراز هویت تغییرکرده (نقش/دامنه‌ها/کلید عمومی) دوباره امتحان کند، درخواست در انتظار قبلی جایگزین می‌شود و یک `requestId` جدید ساخته می‌شود. پیش از تأیید، دوباره `openclaw devices list` را اجرا کنید.

اگر مرورگر از قبل جفت شده باشد و آن را از دسترسی خواندن به دسترسی نوشتن/admin تغییر دهید، این به‌عنوان ارتقای تأیید در نظر گرفته می‌شود، نه اتصال مجدد بی‌صدا. OpenClaw تأیید قبلی را فعال نگه می‌دارد، اتصال مجدد گسترده‌تر را مسدود می‌کند، و از شما می‌خواهد مجموعه دامنه جدید را صریحاً تأیید کنید.

پس از تأیید، دستگاه به خاطر سپرده می‌شود و تا زمانی که آن را با `openclaw devices revoke --device <id> --role <role>` لغو نکنید، به تأیید دوباره نیاز ندارد. برای چرخش و لغو توکن، [CLI دستگاه‌ها](/fa/cli/devices) را ببینید.

عامل‌های Paperclip که از طریق آداپتر `openclaw_gateway` وصل می‌شوند از همان جریان تأیید اجرای اول استفاده می‌کنند. پس از تلاش اتصال اولیه، `openclaw devices approve --latest` را اجرا کنید تا درخواست در انتظار را پیش‌نمایش کنید، سپس فرمان چاپ‌شده `openclaw devices approve <requestId>` را دوباره اجرا کنید تا آن را تأیید کنید. برای gateway راه‌دور، مقادیر صریح `--url` و `--token` را ارسال کنید. برای پایدار نگه داشتن تأییدها بین راه‌اندازی‌های مجدد، به‌جای اینکه Paperclip در هر اجرا یک هویت دستگاه موقت جدید تولید کند، یک `adapterConfig.devicePrivateKeyPem` پایدار در Paperclip پیکربندی کنید.

<Note>
- اتصال‌های مستقیم مرورگر از طریق local loopback (`127.0.0.1` / `localhost`) به‌صورت خودکار تأیید می‌شوند.
- Tailscale Serve می‌تواند رفت‌وبرگشت جفت‌سازی را برای نشست‌های اپراتور رابط کاربری کنترل رد کند، وقتی `gateway.auth.allowTailscale: true` باشد، هویت Tailscale تأیید شود، و مرورگر هویت دستگاه خود را ارائه کند.
- اتصال‌های مستقیم Tailnet، اتصال‌های مرورگر LAN، و پروفایل‌های مرورگر بدون هویت دستگاه همچنان به تأیید صریح نیاز دارند.
- هر پروفایل مرورگر یک شناسه دستگاه یکتا تولید می‌کند، بنابراین تغییر مرورگر یا پاک کردن داده‌های مرورگر نیازمند جفت‌سازی دوباره خواهد بود.

</Note>

## هویت شخصی (محلی مرورگر)

رابط کاربری کنترل از یک هویت شخصی برای هر مرورگر (نام نمایشی و آواتار) پشتیبانی می‌کند که برای انتساب در نشست‌های مشترک به پیام‌های خروجی پیوست می‌شود. این هویت در فضای ذخیره‌سازی مرورگر قرار دارد، به پروفایل مرورگر فعلی محدود است، و به دستگاه‌های دیگر همگام‌سازی نمی‌شود یا فراتر از فراداده معمول نویسندگی رونوشت برای پیام‌هایی که واقعاً ارسال می‌کنید، در سمت سرور پایدار ذخیره نمی‌شود. پاک کردن داده‌های سایت یا تغییر مرورگر آن را به حالت خالی بازنشانی می‌کند.

همین الگوی محلی مرورگر برای override آواتار دستیار نیز اعمال می‌شود. آواتارهای بارگذاری‌شده دستیار فقط در مرورگر محلی، هویت حل‌شده توسط gateway را پوشش می‌دهند و هرگز از طریق `config.patch` رفت‌وبرگشت نمی‌شوند. فیلد پیکربندی مشترک `ui.assistant.avatar` همچنان برای کلاینت‌های غیر UI که این فیلد را مستقیماً می‌نویسند (مانند gatewayهای اسکریپتی یا داشبوردهای سفارشی) در دسترس است.

## نقطه پایانی پیکربندی زمان اجرا

رابط کاربری کنترل تنظیمات زمان اجرای خود را از `/control-ui-config.json` دریافت می‌کند که نسبت به مسیر پایه رابط کاربری کنترل gateway حل می‌شود (برای مثال `/__openclaw__/control-ui-config.json` وقتی UI زیر `/__openclaw__/` ارائه می‌شود). این نقطه پایانی با همان احراز هویت gateway مانند بقیه سطح HTTP محافظت می‌شود: مرورگرهای احرازهویت‌نشده نمی‌توانند آن را دریافت کنند، و دریافت موفق به یکی از این موارد نیاز دارد: توکن/گذرواژه gateway که از قبل معتبر باشد، هویت Tailscale Serve، یا هویت پراکسی مورداعتماد.

## پشتیبانی زبان

رابط کاربری کنترل می‌تواند در اولین بارگذاری، خود را بر اساس locale مرورگر شما بومی‌سازی کند. برای override کردن آن در آینده، **نمای کلی -> دسترسی Gateway -> زبان** را باز کنید. انتخاب‌گر locale در کارت دسترسی Gateway قرار دارد، نه زیر ظاهر.

- localeهای پشتیبانی‌شده: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- ترجمه‌های غیرانگلیسی در مرورگر به‌صورت lazy-loaded بارگذاری می‌شوند.
- locale انتخاب‌شده در فضای ذخیره‌سازی مرورگر ذخیره می‌شود و در بازدیدهای آینده دوباره استفاده می‌شود.
- کلیدهای ترجمه موجودنبودن به انگلیسی بازمی‌گردند.

ترجمه‌های مستندات برای همان مجموعه localeهای غیرانگلیسی تولید می‌شوند، اما انتخاب‌گر زبان داخلی سایت مستندات Mintlify به کدهای localeای محدود است که Mintlify می‌پذیرد. مستندات تایلندی (`th`) و فارسی (`fa`) همچنان در مخزن انتشار تولید می‌شوند؛ ممکن است تا زمانی که Mintlify از این کدها پشتیبانی نکند، در آن انتخاب‌گر ظاهر نشوند.

## تم‌های ظاهر

پنل ظاهر تم‌های داخلی Claw، Knot و Dash را به‌همراه یک جایگاه import محلی مرورگر برای tweakcn نگه می‌دارد. برای import کردن یک تم، [ویرایشگر tweakcn](https://tweakcn.com/editor/theme) را باز کنید، یک تم را انتخاب یا ایجاد کنید، روی **اشتراک‌گذاری** کلیک کنید، و پیوند تم کپی‌شده را در ظاهر جای‌گذاری کنید. واردکننده همچنین URLهای رجیستری `https://tweakcn.com/r/themes/<id>`، URLهای ویرایشگر مانند `https://tweakcn.com/editor/theme?theme=amethyst-haze`، مسیرهای نسبی `/themes/<id>`، شناسه‌های خام تم، و نام‌های تم پیش‌فرض مانند `amethyst-haze` را می‌پذیرد.

ظاهر همچنین شامل یک تنظیم اندازه متن محلی مرورگر است. این تنظیم همراه با بقیه ترجیحات رابط کاربری کنترل ذخیره می‌شود، روی متن گفت‌وگو، متن composer، کارت‌های ابزار، و نوارهای کناری گفت‌وگو اعمال می‌شود، و ورودی‌های متن را دست‌کم 16px نگه می‌دارد تا Safari موبایل هنگام focus به‌صورت خودکار zoom نکند.

تم‌های import شده فقط در پروفایل مرورگر فعلی ذخیره می‌شوند. آن‌ها در پیکربندی gateway نوشته نمی‌شوند و بین دستگاه‌ها همگام‌سازی نمی‌شوند. جایگزین کردن تم import شده همان یک جایگاه محلی را به‌روزرسانی می‌کند؛ پاک کردن آن اگر تم import شده انتخاب شده باشد، تم فعال را به Claw برمی‌گرداند.

    - به‌روزرسانی: اجرای یک به‌روزرسانی package/git + راه‌اندازی مجدد (`update.run`) با گزارش راه‌اندازی مجدد، سپس polling `update.status` پس از اتصال مجدد برای تأیید نسخه gateway در حال اجرا.

  </Accordion>
  <Accordion title="یادداشت‌های پنل کارهای Cron">
    - برای کارهای ایزوله، تحویل به‌طور پیش‌فرض روی اعلام خلاصه است. اگر اجراهای فقط داخلی می‌خواهید، می‌توانید آن را به هیچ‌کدام تغییر دهید.
    - وقتی اعلام انتخاب شده باشد، فیلدهای کانال/هدف ظاهر می‌شوند.
    - حالت Webhook از `delivery.mode = "webhook"` استفاده می‌کند و `delivery.to` روی یک URL معتبر Webhook مبتنی بر HTTP(S) تنظیم می‌شود.
    - برای کارهای نشست اصلی، حالت‌های تحویل Webhook و هیچ‌کدام در دسترس هستند.
    - کنترل‌های ویرایش پیشرفته شامل حذف پس از اجرا، پاک‌کردن بازنویسی عامل، گزینه‌های دقیق/پلکانی Cron، بازنویسی‌های مدل/تفکر عامل، و تغییر‌دهنده‌های تحویل با بهترین تلاش هستند.
    - اعتبارسنجی فرم به‌صورت درون‌خطی با خطاهای سطح فیلد انجام می‌شود؛ مقادیر نامعتبر تا زمان اصلاح، دکمه ذخیره را غیرفعال می‌کنند.
    - `cron.webhookToken` را تنظیم کنید تا یک توکن bearer اختصاصی ارسال شود؛ اگر حذف شود، Webhook بدون سرآیند احراز هویت ارسال می‌شود.
    - جایگزین منسوخ: `openclaw doctor --fix` را اجرا کنید تا کارهای قدیمی ذخیره‌شده با `notify: true` از `cron.webhook` به Webhook صریح برای هر کار یا تحویل تکمیل مهاجرت داده شوند.

## صفحه MCP

صفحه اختصاصی MCP یک نمای اپراتوری برای سرورهای MCP مدیریت‌شده توسط OpenClaw زیر `mcp.servers` است. این صفحه خودش انتقال‌های MCP را شروع نمی‌کند؛ از آن برای بررسی و ویرایش پیکربندی ذخیره‌شده استفاده کنید، سپس وقتی به اثبات زنده سرور نیاز دارید از `openclaw mcp doctor --probe` استفاده کنید.

روند کاری معمول:

1. **MCP** را از نوار کناری باز کنید.
2. کارت‌های خلاصه را برای تعداد کل، فعال، OAuth، و سرورهای فیلترشده بررسی کنید.
3. هر ردیف سرور را از نظر انتقال، فعال‌سازی، احراز هویت، فیلترها، زمان‌سنج‌ها، و راهنمایی‌های فرمان بازبینی کنید.
4. وقتی سروری باید پیکربندی‌شده بماند اما از کشف زمان اجرا کنار گذاشته شود، فعال‌سازی را تغییر دهید.
5. بخش پیکربندی محدوده‌دار `mcp` را برای تعریف‌های سرور، سرآیندها، مسیرهای TLS/mTLS، فراداده OAuth، فیلترهای ابزار، و فراداده تصویرسازی Codex ویرایش کنید.
6. برای نوشتن پیکربندی از **ذخیره** استفاده کنید، یا وقتی Gateway در حال اجرا باید پیکربندی تغییرکرده را اعمال کند از **ذخیره و انتشار** استفاده کنید.
7. وقتی فرایند ویرایش‌شده به تشخیص ایستای عیب‌ها، اثبات زنده، یا دورریزی زمان اجرای کش‌شده نیاز دارد، `openclaw mcp status --verbose`، `openclaw mcp doctor --probe`، یا `openclaw mcp reload` را از ترمینال اجرا کنید.

این صفحه پیش از نمایش، مقدارهای شبیه URL دارای اعتبارنامه را سانسور می‌کند و نام سرورها را در قطعه‌فرمان‌ها داخل نقل‌قول می‌گذارد تا فرمان‌های کپی‌شده همچنان با فاصله‌ها یا نویسه‌های ویژه پوسته کار کنند. مرجع کامل CLI و پیکربندی در [MCP](/fa/cli/mcp) قرار دارد.

## زبانه فعالیت

زبانه فعالیت یک ناظر موقت و محلیِ مرورگر برای فعالیت زنده ابزارها است. این زبانه از همان جریان رویداد Gateway `session.tool` / ابزار مشتق می‌شود که کارت‌های ابزار گفت‌وگو را تغذیه می‌کند؛ خانواده رویداد Gateway، نقطه پایانی، ذخیره‌گاه پایدار فعالیت، خوراک سنجه‌ها، یا جریان ناظر بیرونی دیگری اضافه نمی‌کند.

ورودی‌های فعالیت فقط خلاصه‌های پاک‌سازی‌شده و پیش‌نمایش‌های خروجی سانسورشده و کوتاه‌شده را نگه می‌دارند. مقدارهای آرگومان ابزار در وضعیت فعالیت ذخیره نمی‌شوند؛ رابط کاربری نشان می‌دهد که آرگومان‌ها پنهان هستند و فقط تعداد فیلدهای آرگومان را ثبت می‌کند. فهرست درون‌حافظه‌ای از زبانه فعلی مرورگر پیروی می‌کند، هنگام ناوبری درون Control UI باقی می‌ماند، و با بارگذاری دوباره صفحه، تغییر نشست، یا **پاک‌کردن** بازنشانی می‌شود.

## رفتار گفت‌وگو

<AccordionGroup>
  <Accordion title="معناشناسی ارسال و تاریخچه">
    - `chat.send` **غیرمسدودکننده** است: بلافاصله با `{ runId, status: "started" }` تأیید می‌کند و پاسخ از طریق رویدادهای `chat` جریان می‌یابد. کلاینت‌های مورد اعتماد Control UI ممکن است فراداده اختیاری زمان‌بندی تأیید را نیز برای تشخیص‌های محلی دریافت کنند.
    - بارگذاری‌های گفت‌وگو، تصویرها به‌علاوه فایل‌های غیر ویدیویی را می‌پذیرند. تصویرها مسیر تصویر بومی را حفظ می‌کنند؛ فایل‌های دیگر به‌عنوان رسانه مدیریت‌شده ذخیره می‌شوند و در تاریخچه به‌صورت پیوندهای پیوست نشان داده می‌شوند.
    - ارسال دوباره با همان `idempotencyKey` هنگام اجرا `{ status: "in_flight" }` و پس از تکمیل `{ status: "ok" }` را برمی‌گرداند.
    - پاسخ‌های `chat.history` برای ایمنی رابط کاربری از نظر اندازه محدود هستند. وقتی ورودی‌های رونوشت بیش از حد بزرگ باشند، Gateway ممکن است فیلدهای متنی طولانی را کوتاه کند، بلوک‌های سنگین فراداده را حذف کند، و پیام‌های بیش‌ازحد بزرگ را با یک جای‌نگهدار (`[chat.history omitted: message too large]`) جایگزین کند.
    - وقتی یک پیام قابل‌مشاهده دستیار در `chat.history` کوتاه شده باشد، خواننده کناری می‌تواند ورودی کامل رونوشتِ نرمال‌شده برای نمایش را در صورت نیاز از طریق `chat.message.get` با `sessionKey`، `agentId` فعال در صورت نیاز، و `messageId` رونوشت دریافت کند. اگر Gateway همچنان نتواند مقدار بیشتری برگرداند، خواننده به‌جای تکرار بی‌صدای پیش‌نمایش کوتاه‌شده، وضعیت صریحاً ناموجود را نشان می‌دهد.
    - تصویرهای تولیدشده توسط دستیار به‌عنوان ارجاع‌های رسانه مدیریت‌شده پایدار می‌شوند و از طریق URLهای رسانه احراز هویت‌شده Gateway بازگردانده می‌شوند، بنابراین بارگذاری دوباره به باقی‌ماندن بارهای تصویر base64 خام در پاسخ تاریخچه گفت‌وگو وابسته نیست.
    - هنگام نمایش `chat.history`، Control UI برچسب‌های دستور درون‌خطی فقط نمایشی را از متن قابل‌مشاهده دستیار حذف می‌کند (برای مثال `[[reply_to_*]]` و `[[audio_as_voice]]`)، بارهای XML فراخوانی ابزار در متن ساده (از جمله `<tool_call>...</tool_call>`، `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`، `<function_calls>...</function_calls>`، و بلوک‌های کوتاه‌شده فراخوانی ابزار)، و توکن‌های کنترلی مدل ASCII/تمام‌عرض لو‌رفته را حذف می‌کند، و ورودی‌های دستیار را که کل متن قابل‌مشاهده آن‌ها فقط توکن ساکت دقیق `NO_REPLY` / `no_reply` یا توکن تأیید Heartbeat یعنی `HEARTBEAT_OK` است، کنار می‌گذارد.
    - هنگام یک ارسال فعال و تازه‌سازی نهایی تاریخچه، اگر `chat.history` برای مدت کوتاهی یک نماگرفت قدیمی‌تر برگرداند، نمای گفت‌وگو پیام‌های خوش‌بینانه محلی کاربر/دستیار را قابل‌مشاهده نگه می‌دارد؛ وقتی تاریخچه Gateway به‌روز شد، رونوشت مرجع آن پیام‌های محلی را جایگزین می‌کند.
    - رویدادهای زنده `chat` وضعیت تحویل هستند، در حالی که `chat.history` از رونوشت پایدار نشست بازسازی می‌شود. پس از رویدادهای نهایی ابزار، Control UI تاریخچه را دوباره بارگذاری می‌کند و فقط یک دنباله خوش‌بینانه کوچک را ادغام می‌کند؛ مرز رونوشت در [WebChat](/fa/web/webchat) مستند شده است.
    - `chat.inject` یک یادداشت دستیار را به رونوشت نشست اضافه می‌کند و یک رویداد `chat` را برای به‌روزرسانی‌های فقط رابط کاربری پخش می‌کند (بدون اجرای عامل، بدون تحویل کانال).
    - سرآیند گفت‌وگو فیلتر عامل را پیش از انتخابگر نشست نشان می‌دهد، و انتخابگر نشست با عامل انتخاب‌شده محدود می‌شود. تغییر عامل‌ها فقط نشست‌های مرتبط با همان عامل را نشان می‌دهد و وقتی هنوز نشست‌های داشبورد ذخیره‌شده ندارد، به نشست اصلی همان عامل بازمی‌گردد.
    - در عرض‌های دسکتاپ، کنترل‌های گفت‌وگو در یک ردیف فشرده می‌مانند و هنگام پیمایش رو به پایین در رونوشت جمع می‌شوند؛ پیمایش رو به بالا، بازگشت به بالا، یا رسیدن به پایین کنترل‌ها را بازمی‌گرداند.
    - پیام‌های فقط متنیِ تکراری و پشت‌سرهم به‌صورت یک حباب با نشان تعداد نمایش داده می‌شوند. پیام‌هایی که تصویر، پیوست، خروجی ابزار، یا پیش‌نمایش بوم دارند، جمع نمی‌شوند.
    - انتخابگرهای مدل و تفکر در سرآیند گفت‌وگو نشست فعال را بلافاصله از طریق `sessions.patch` وصله می‌کنند؛ آن‌ها بازنویسی‌های پایدار نشست هستند، نه گزینه‌های ارسال فقط برای یک نوبت.
    - اگر وقتی تغییر انتخابگر مدل برای همان نشست هنوز در حال ذخیره است پیامی بفرستید، ویرایشگر پیش از فراخوانی `chat.send` منتظر آن وصله نشست می‌ماند تا ارسال از مدل انتخاب‌شده استفاده کند.
    - تایپ `‎/new` در Control UI همان نشست تازه داشبورد را که گفت‌وگوی جدید می‌سازد، ایجاد می‌کند و به آن تغییر می‌دهد، مگر وقتی `session.dmScope: "main"` پیکربندی شده باشد و والد فعلی نشست اصلی عامل باشد؛ در آن حالت نشست اصلی را درجا بازنشانی می‌کند. تایپ `‎/reset` بازنشانی صریح درجا توسط Gateway را برای نشست فعلی حفظ می‌کند.
    - انتخابگر مدل گفت‌وگو نمای مدل پیکربندی‌شده Gateway را درخواست می‌کند. اگر `agents.defaults.models` وجود داشته باشد، همان فهرست مجاز انتخابگر را هدایت می‌کند، از جمله ورودی‌های `provider/*` که کاتالوگ‌های محدوده‌دار ارائه‌دهنده را پویا نگه می‌دارند. در غیر این صورت انتخابگر ورودی‌های صریح `models.providers.*.models` به‌علاوه ارائه‌دهندگان دارای احراز هویت قابل‌استفاده را نشان می‌دهد. کاتالوگ کامل از طریق RPC اشکال‌زدایی `models.list` با `view: "all"` در دسترس می‌ماند.
    - وقتی گزارش‌های تازه مصرف نشست Gateway شامل توکن‌های زمینه فعلی باشند، ناحیه ویرایشگر گفت‌وگو یک نشانگر فشرده مصرف زمینه را نشان می‌دهد. در فشار بالای زمینه به سبک هشدار تغییر می‌کند و در سطح‌های پیشنهادی Compaction، دکمه‌ای فشرده نشان می‌دهد که مسیر عادی Compaction نشست را اجرا می‌کند. نماگرفت‌های کهنه توکن تا وقتی Gateway دوباره مصرف تازه را گزارش کند، پنهان می‌مانند.

  </Accordion>
  <Accordion title="حالت مکالمه (بلادرنگ مرورگر)">
    حالت مکالمه از یک ارائه‌دهنده صدای بلادرنگ ثبت‌شده استفاده می‌کند. OpenAI را با `talk.realtime.provider: "openai"` به‌همراه یک نمایه احراز هویت کلید API برای `openai`، `talk.realtime.providers.openai.apiKey`، یا `OPENAI_API_KEY` پیکربندی کنید؛ نمایه‌های OAuth مربوط به OpenAI صدای بلادرنگ را پیکربندی نمی‌کنند. Google را با `talk.realtime.provider: "google"` به‌همراه `talk.realtime.providers.google.apiKey` پیکربندی کنید. مرورگر هرگز یک کلید API استاندارد ارائه‌دهنده دریافت نمی‌کند. OpenAI یک راز موقت کلاینت بلادرنگ برای WebRTC دریافت می‌کند. Google Live یک توکن احراز هویت محدود و یک‌بارمصرف Live API برای نشست WebSocket مرورگر دریافت می‌کند، در حالی که دستورالعمل‌ها و اعلان‌های ابزار توسط Gateway در توکن قفل شده‌اند. ارائه‌دهندگانی که فقط یک پل بلادرنگ بک‌اند در اختیار می‌گذارند از طریق انتقال رله Gateway اجرا می‌شوند، بنابراین اعتبارنامه‌ها و سوکت‌های فروشنده سمت سرور می‌مانند، در حالی که صدای مرورگر از طریق RPCهای احراز هویت‌شده Gateway جابه‌جا می‌شود. اعلان نشست بلادرنگ توسط Gateway مونتاژ می‌شود؛ `talk.client.create` بازنویسی دستورالعمل ارائه‌شده توسط فراخواننده را نمی‌پذیرد.

    ویرایشگر گفت‌وگو در کنار دکمه شروع/توقف مکالمه، یک دکمه گزینه‌های مکالمه دارد. این گزینه‌ها روی نشست مکالمه بعدی اعمال می‌شوند و می‌توانند ارائه‌دهنده، انتقال، مدل، صدا، تلاش استدلال، آستانه VAD، مدت سکوت، و فاصله‌گذاری پیشوند را بازنویسی کنند. وقتی گزینه‌ای خالی باشد، Gateway در صورت وجود از پیش‌فرض‌های پیکربندی‌شده یا از پیش‌فرض ارائه‌دهنده استفاده می‌کند. انتخاب رله Gateway مسیر رله بک‌اند را اجباری می‌کند؛ انتخاب WebRTC نشست را تحت مالکیت کلاینت نگه می‌دارد و اگر ارائه‌دهنده نتواند نشست مرورگر بسازد، به‌جای بازگشت بی‌صدای به رله، شکست می‌خورد.

    در ویرایشگر گفت‌وگو، کنترل مکالمه دکمه موج‌ها در کنار دکمه دیکته میکروفون است. وقتی مکالمه شروع می‌شود، ردیف وضعیت ویرایشگر `Connecting Talk...` را نشان می‌دهد، سپس وقتی صدا وصل است `Talk live` را نشان می‌دهد، یا وقتی یک فراخوانی ابزار بلادرنگ از طریق `talk.client.toolCall` با مدل بزرگ‌تر پیکربندی‌شده مشورت می‌کند، `Asking OpenClaw...` را نشان می‌دهد.

    دودآزمایی زنده نگه‌دارنده: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` پل WebSocket بک‌اند OpenAI، تبادل SDP مرورگر WebRTC برای OpenAI، راه‌اندازی WebSocket مرورگر با توکن محدود Google Live، و آداپتر مرورگر رله Gateway با رسانه میکروفون ساختگی را بررسی می‌کند. این فرمان فقط وضعیت ارائه‌دهنده را چاپ می‌کند و رازها را ثبت نمی‌کند.

  </Accordion>

    - وقتی یک اجرا فعال است، پیگیری‌های عادی در صف قرار می‌گیرند. روی **هدایت** در یک پیام صف‌شده کلیک کنید تا آن پیگیری به نوبت در حال اجرا تزریق شود.
    - برای لغو خارج از باند، `‎/stop` را تایپ کنید (یا عبارت‌های لغو مستقل مانند `stop`، `stop action`، `stop run`، `stop openclaw`، `please stop`).
    - `chat.abort` از `{ sessionKey }` (بدون `runId`) برای لغو همه اجراهای فعال آن نشست پشتیبانی می‌کند.

  </Accordion>
  <Accordion title="نگه‌داری بخشی پس از لغو">
    - وقتی یک اجرا لغو می‌شود، متن بخشی دستیار همچنان می‌تواند در رابط کاربری نشان داده شود.
    - وقتی خروجی بافرشده وجود داشته باشد، Gateway متن بخشی دستیارِ لغوشده را در تاریخچه رونوشت پایدار می‌کند.
    - ورودی‌های پایدارشده شامل فراداده لغو هستند تا مصرف‌کنندگان رونوشت بتوانند بخش‌های لغوشده را از خروجی تکمیل عادی تشخیص دهند.

  </Accordion>
</AccordionGroup>

## نصب PWA و پوش وب

Control UI همراه با یک `manifest.webmanifest` و یک سرویس‌ورکر عرضه می‌شود، بنابراین مرورگرهای مدرن می‌توانند آن را به‌عنوان یک PWA مستقل نصب کنند. پوش وب به Gateway اجازه می‌دهد PWA نصب‌شده را حتی وقتی زبانه یا پنجره مرورگر باز نیست، با اعلان‌ها بیدار کند.

اگر صفحه درست پس از یک به‌روزرسانی OpenClaw پیام **ناسازگاری پروتکل** را نشان داد، ابتدا داشبورد را با `openclaw dashboard` دوباره باز کنید و صفحه را سخت‌تازه‌سازی کنید. اگر همچنان شکست خورد، داده‌های سایت را برای مبدأ داشبورد پاک کنید یا در یک پنجره خصوصی مرورگر آزمایش کنید؛ یک زبانه قدیمی یا کش سرویس‌ورکر مرورگر می‌تواند همچنان یک بسته Control UI پیش از به‌روزرسانی را در برابر Gateway جدیدتر اجرا کند.

| سطح                                                  | کاری که انجام می‌دهد                                               |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | مانیفست PWA. مرورگرها پس از در دسترس شدن آن، گزینه «نصب برنامه» را پیشنهاد می‌کنند. |
| `ui/public/sw.js`                                     | سرویس‌ورکری که رویدادهای `push` و کلیک‌های اعلان را مدیریت می‌کند. |
| `push/vapid-keys.json` (در دایرکتوری وضعیت OpenClaw) | جفت‌کلید VAPID که به‌صورت خودکار تولید می‌شود و برای امضای payloadهای پوش وب استفاده می‌شود. |
| `push/web-push-subscriptions.json`                    | endpointهای اشتراک مرورگر که پایدارسازی شده‌اند.                  |

وقتی می‌خواهید کلیدها را ثابت نگه دارید (برای استقرارهای چندمیزبانه، چرخش secrets، یا تست‌ها)، جفت‌کلید VAPID را از طریق متغیرهای محیطی روی فرایند Gateway بازنویسی کنید:

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (پیش‌فرض `https://openclaw.ai` است)

رابط کاربری کنترل از این متدهای Gateway با محدوده کنترل‌شده برای ثبت و تست اشتراک‌های مرورگر استفاده می‌کند:

- `push.web.vapidPublicKey` — کلید عمومی فعال VAPID را دریافت می‌کند.
- `push.web.subscribe` — یک `endpoint` به‌همراه `keys.p256dh`/`keys.auth` ثبت می‌کند.
- `push.web.unsubscribe` — یک endpoint ثبت‌شده را حذف می‌کند.
- `push.web.test` — یک اعلان آزمایشی به اشتراک فراخواننده ارسال می‌کند.

<Note>
پوش وب مستقل از مسیر relay مربوط به APNS در iOS است (برای پوش پشتیبانی‌شده با relay، [پیکربندی](/fa/gateway/configuration) را ببینید) و همچنین از متد موجود `push.test` جداست؛ آن متدها جفت‌سازی بومی موبایل را هدف می‌گیرند.
</Note>

## embedهای میزبانی‌شده

پیام‌های دستیار می‌توانند محتوای وب میزبانی‌شده را به‌صورت inline با shortcode `[embed ...]` رندر کنند. سیاست sandbox iframe توسط `gateway.controlUi.embedSandbox` کنترل می‌شود:

<Tabs>
  <Tab title="strict">
    اجرای اسکریپت داخل embedهای میزبانی‌شده را غیرفعال می‌کند.
  </Tab>
  <Tab title="scripts (default)">
    embedهای تعاملی را مجاز می‌کند و در عین حال جداسازی origin را حفظ می‌کند؛ این مقدار پیش‌فرض است و معمولاً برای بازی‌ها/ویجت‌های مرورگری خودبسنده کافی است.
  </Tab>
  <Tab title="trusted">
    برای سندهای same-site که عمداً به امتیازهای قوی‌تری نیاز دارند، `allow-same-origin` را روی `allow-scripts` اضافه می‌کند.
  </Tab>
</Tabs>

مثال:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
از `trusted` فقط زمانی استفاده کنید که سند embed‌شده واقعاً به رفتار same-origin نیاز داشته باشد. برای بیشتر بازی‌ها و canvasهای تعاملی تولیدشده توسط agent، `scripts` گزینه امن‌تری است.
</Warning>

URLهای embed مطلق خارجی `http(s)` به‌صورت پیش‌فرض مسدود می‌مانند. اگر عمداً می‌خواهید `[embed url="https://..."]` صفحه‌های شخص ثالث را بارگذاری کند، `gateway.controlUi.allowExternalEmbedUrls: true` را تنظیم کنید.

## عرض پیام چت

پیام‌های چت گروه‌بندی‌شده از max-width پیش‌فرض خوانا استفاده می‌کنند. استقرارهای نمایشگر عریض می‌توانند بدون patch کردن CSS بسته‌بندی‌شده، با تنظیم `gateway.controlUi.chatMessageMaxWidth` آن را بازنویسی کنند:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

این مقدار پیش از رسیدن به مرورگر اعتبارسنجی می‌شود. مقادیر پشتیبانی‌شده شامل طول‌ها و درصدهای ساده مانند `960px` یا `82%`، به‌علاوه عبارت‌های عرض محدودشده `min(...)`، `max(...)`، `clamp(...)`، `calc(...)`، و `fit-content(...)` هستند.

## دسترسی Tailnet (توصیه‌شده)

<Tabs>
  <Tab title="Integrated Tailscale Serve (preferred)">
    Gateway را روی loopback نگه دارید و اجازه دهید Tailscale Serve آن را با HTTPS پروکسی کند:

    ```bash
    openclaw gateway --tailscale serve
    ```

    باز کنید:

    - `https://<magicdns>/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    به‌صورت پیش‌فرض، درخواست‌های Serve مربوط به رابط کاربری کنترل/WebSocket می‌توانند وقتی `gateway.auth.allowTailscale` برابر `true` است از طریق هدرهای هویت Tailscale (`tailscale-user-login`) احراز هویت شوند. OpenClaw هویت را با resolve کردن آدرس `x-forwarded-for` از طریق `tailscale whois` و تطبیق آن با هدر تأیید می‌کند، و فقط زمانی این موارد را می‌پذیرد که درخواست از طریق loopback همراه با هدرهای `x-forwarded-*` متعلق به Tailscale برسد. برای sessionهای اپراتور رابط کاربری کنترل با هویت دستگاه مرورگر، این مسیر Serve تأییدشده همچنین round trip جفت‌سازی دستگاه را رد می‌کند؛ مرورگرهای بدون دستگاه و اتصال‌های نقش node همچنان بررسی‌های عادی دستگاه را دنبال می‌کنند. اگر می‌خواهید حتی برای ترافیک Serve هم credentialهای shared-secret صریح لازم باشد، `gateway.auth.allowTailscale: false` را تنظیم کنید. سپس از `gateway.auth.mode: "token"` یا `"password"` استفاده کنید.

    برای آن مسیر ناهمگام هویت Serve، تلاش‌های ناموفق احراز هویت برای همان IP کلاینت و محدوده auth، پیش از نوشتن rate-limit به‌صورت ترتیبی انجام می‌شوند. بنابراین retryهای بد هم‌زمان از همان مرورگر می‌توانند روی درخواست دوم به‌جای دو mismatch ساده که موازی رقابت می‌کنند، `retry later` نشان دهند.

    <Warning>
    auth بدون token برای Serve فرض می‌کند میزبان gateway قابل اعتماد است. اگر ممکن است کد محلی غیرقابل اعتماد روی آن میزبان اجرا شود، auth با token/password را الزامی کنید.
    </Warning>

  </Tab>
  <Tab title="Bind to tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    سپس باز کنید:

    - `http://<tailscale-ip>:18789/` (یا `gateway.controlUi.basePath` پیکربندی‌شده شما)

    shared secret منطبق را در تنظیمات رابط کاربری وارد کنید (به‌صورت `connect.params.auth.token` یا `connect.params.auth.password` ارسال می‌شود).

  </Tab>
</Tabs>

## HTTP ناامن

اگر داشبورد را از طریق HTTP ساده باز کنید (`http://<lan-ip>` یا `http://<tailscale-ip>`)، مرورگر در یک **زمینه غیرامن** اجرا می‌شود و WebCrypto را مسدود می‌کند. به‌صورت پیش‌فرض، OpenClaw اتصال‌های رابط کاربری کنترل بدون هویت دستگاه را **مسدود** می‌کند.

استثناهای مستندشده:

- سازگاری HTTP ناامن فقط برای localhost با `gateway.controlUi.allowInsecureAuth=true`
- auth موفق اپراتور رابط کاربری کنترل از طریق `gateway.auth.mode: "trusted-proxy"`
- گزینه اضطراری `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**راه‌حل توصیه‌شده:** از HTTPS (Tailscale Serve) استفاده کنید یا رابط کاربری را به‌صورت محلی باز کنید:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (روی میزبان gateway)

<AccordionGroup>
  <Accordion title="Insecure-auth toggle behavior">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` فقط یک toggle سازگاری محلی است:

    - اجازه می‌دهد sessionهای رابط کاربری کنترل روی localhost بدون هویت دستگاه در زمینه‌های HTTP غیرامن ادامه پیدا کنند.
    - بررسی‌های جفت‌سازی را دور نمی‌زند.
    - الزامات هویت دستگاه remote (غیر-localhost) را سهل‌گیرانه‌تر نمی‌کند.

  </Accordion>
  <Accordion title="Break-glass only">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` بررسی‌های هویت دستگاه رابط کاربری کنترل را غیرفعال می‌کند و یک کاهش امنیتی شدید است. پس از استفاده اضطراری، سریع آن را برگردانید.
    </Warning>

  </Accordion>
  <Accordion title="Trusted-proxy note">
    - auth موفق trusted-proxy می‌تواند sessionهای رابط کاربری کنترل مربوط به **اپراتور** را بدون هویت دستگاه بپذیرد.
    - این مورد به sessionهای رابط کاربری کنترل با نقش node گسترش پیدا نمی‌کند.
    - reverse proxyهای loopback روی همان میزبان همچنان auth trusted-proxy را برآورده نمی‌کنند؛ [auth پروکسی قابل اعتماد](/fa/gateway/trusted-proxy-auth) را ببینید.

  </Accordion>
</AccordionGroup>

برای راهنمایی راه‌اندازی HTTPS، [Tailscale](/fa/gateway/tailscale) را ببینید.

## سیاست امنیت محتوا

رابط کاربری کنترل با سیاست سخت‌گیرانه `img-src` عرضه می‌شود: فقط assetهای **same-origin**، URLهای `data:`، و URLهای `blob:` تولیدشده به‌صورت محلی مجاز هستند. URLهای تصویر remote با `http(s)` و URLهای تصویر protocol-relative توسط مرورگر رد می‌شوند و fetch شبکه‌ای صادر نمی‌کنند.

معنای عملی این موضوع:

- آواتارها و تصویرهایی که زیر مسیرهای نسبی سرو می‌شوند (برای مثال `/avatars/<id>`) همچنان رندر می‌شوند، از جمله routeهای آواتار احراز هویت‌شده که رابط کاربری آن‌ها را fetch کرده و به URLهای `blob:` محلی تبدیل می‌کند.
- URLهای inline از نوع `data:image/...` همچنان رندر می‌شوند (برای payloadهای درون‌پروتکلی مفید است).
- URLهای `blob:` محلی ساخته‌شده توسط رابط کاربری کنترل همچنان رندر می‌شوند.
- URLهای آواتار remote که توسط metadata کانال منتشر می‌شوند در helperهای آواتار رابط کاربری کنترل حذف می‌شوند و با لوگو/badge داخلی جایگزین می‌شوند، بنابراین یک کانال compromised یا مخرب نمی‌تواند مرورگر اپراتور را مجبور به fetch تصویر remote دلخواه کند.

برای دریافت این رفتار لازم نیست چیزی را تغییر دهید — این رفتار همیشه فعال است و قابل پیکربندی نیست.

## auth مسیر آواتار

وقتی auth مربوط به gateway پیکربندی شده باشد، endpoint آواتار رابط کاربری کنترل همان token gateway را مانند بقیه API لازم دارد:

- `GET /avatar/<agentId>` تصویر آواتار را فقط به فراخواننده‌های احراز هویت‌شده برمی‌گرداند. `GET /avatar/<agentId>?meta=1` metadata آواتار را تحت همان قاعده برمی‌گرداند.
- درخواست‌های احراز هویت‌نشده به هر دو route رد می‌شوند (همسو با route هم‌سطح assistant-media). این کار از نشت هویت agent از مسیر آواتار روی میزبان‌هایی که در غیر این صورت محافظت‌شده‌اند جلوگیری می‌کند.
- خود رابط کاربری کنترل هنگام fetch کردن آواتارها، token gateway را به‌عنوان bearer header ارسال می‌کند و از URLهای blob احراز هویت‌شده استفاده می‌کند تا تصویر همچنان در داشبوردها رندر شود.

اگر auth gateway را غیرفعال کنید (روی میزبان‌های مشترک توصیه نمی‌شود)، route آواتار نیز همسو با بقیه gateway بدون احراز هویت می‌شود.

## auth مسیر رسانه دستیار

وقتی auth مربوط به gateway پیکربندی شده باشد، پیش‌نمایش‌های رسانه محلی دستیار از یک route دو مرحله‌ای استفاده می‌کنند:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` به auth عادی اپراتور رابط کاربری کنترل نیاز دارد. مرورگر هنگام بررسی موجود بودن، token gateway را به‌عنوان bearer header ارسال می‌کند.
- پاسخ‌های metadata موفق شامل یک `mediaTicket` کوتاه‌عمر هستند که به همان مسیر source دقیق محدود شده است.
- URLهای تصویر، صوت، ویدئو، و سند که در مرورگر رندر می‌شوند، به‌جای token یا password فعال gateway از `mediaTicket=<ticket>` استفاده می‌کنند. ticket سریع منقضی می‌شود و نمی‌تواند source متفاوتی را مجاز کند.

این کار رندر عادی رسانه را با عناصر رسانه بومی مرورگر سازگار نگه می‌دارد، بدون اینکه credentialهای قابل‌استفاده مجدد gateway را در URLهای رسانه قابل مشاهده قرار دهد.

## ساخت رابط کاربری

Gateway فایل‌های static را از `dist/control-ui` سرو می‌کند. آن‌ها را با دستور زیر بسازید:

```bash
pnpm ui:build
```

base مطلق اختیاری (وقتی URLهای asset ثابت می‌خواهید):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

برای توسعه محلی (dev server جداگانه):

```bash
pnpm ui:dev
```

سپس رابط کاربری را به URL مربوط به Gateway WS خود وصل کنید (مثلاً `ws://127.0.0.1:18789`).

## صفحه خالی رابط کاربری کنترل

اگر مرورگر یک داشبورد خالی بارگذاری کند و DevTools خطای مفیدی نشان ندهد، ممکن است یک extension یا content script اولیه از ارزیابی برنامه ماژول JavaScript جلوگیری کرده باشد. صفحه static شامل یک پنل بازیابی HTML ساده است که وقتی `<openclaw-app>` پس از startup ثبت نشده باشد نمایش داده می‌شود.

پس از تغییر محیط مرورگر از اکشن **دوباره امتحان کن** در پنل استفاده کنید، یا پس از این بررسی‌ها به‌صورت دستی reload کنید:

- extensionهایی را که به همه صفحه‌ها inject می‌کنند غیرفعال کنید، به‌ویژه extensionهایی با content scriptهای `<all_urls>`.
- یک پنجره private، یک پروفایل مرورگر تمیز، یا مرورگر دیگری را امتحان کنید.
- Gateway را در حال اجرا نگه دارید و پس از تغییر مرورگر همان URL داشبورد را بررسی کنید.

## اشکال‌زدایی/تست: dev server + Gateway remote

رابط کاربری کنترل فایل‌های static است؛ هدف WebSocket قابل پیکربندی است و می‌تواند با origin مربوط به HTTP متفاوت باشد. این برای زمانی مفید است که dev server مربوط به Vite را به‌صورت محلی می‌خواهید اما Gateway جای دیگری اجرا می‌شود.

<Steps>
  <Step title="Start the UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="Open with gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    auth یک‌باره اختیاری (در صورت نیاز):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="یادداشت‌ها">
    - `gatewayUrl` پس از بارگذاری در localStorage ذخیره می‌شود و از URL حذف می‌شود.
    - اگر یک نقطهٔ پایانی کامل `ws://` یا `wss://` را از طریق `gatewayUrl` ارسال می‌کنید، مقدار `gatewayUrl` را URL-encode کنید تا مرورگر رشتهٔ query را درست تجزیه کند.
    - `token` تا حد امکان باید از طریق fragment در URL (`#token=...`) ارسال شود. fragmentها به سرور فرستاده نمی‌شوند و این از نشت در لاگ‌های درخواست و Referer جلوگیری می‌کند. پارامترهای query قدیمی `?token=` همچنان برای سازگاری یک‌بار import می‌شوند، اما فقط به‌عنوان fallback، و بلافاصله پس از bootstrap حذف می‌شوند.
    - `password` فقط در حافظه نگه داشته می‌شود.
    - وقتی `gatewayUrl` تنظیم شده باشد، UI به اعتبارنامه‌های config یا محیط fallback نمی‌کند. `token` (یا `password`) را صراحتاً ارائه کنید. نبود اعتبارنامه‌های صریح خطا است.
    - وقتی Gateway پشت TLS است (Tailscale Serve، پراکسی HTTPS و غیره)، از `wss://` استفاده کنید.
    - `gatewayUrl` فقط در یک پنجرهٔ سطح بالا پذیرفته می‌شود (نه embedded) تا از clickjacking جلوگیری شود.
    - استقرارهای عمومی غیر-loopback برای Control UI باید `gateway.controlUi.allowedOrigins` را صراحتاً تنظیم کنند (originهای کامل). بارگذاری‌های خصوصی same-origin در LAN/Tailnet از loopback، RFC1918/link-local، `.local`، `.ts.net`، یا میزبان‌های Tailscale CGNAT بدون فعال کردن fallback بر پایهٔ Host-header پذیرفته می‌شوند.
    - راه‌اندازی Gateway ممکن است originهای محلی مانند `http://localhost:<port>` و `http://127.0.0.1:<port>` را از bind و پورت مؤثر runtime مقداردهی اولیه کند، اما originهای مرورگر راه‌دور همچنان به ورودی‌های صریح نیاز دارند.
    - از `gateway.controlUi.allowedOrigins: ["*"]` استفاده نکنید مگر برای تست محلی کاملاً کنترل‌شده. این یعنی اجازه دادن به هر origin مرورگر، نه «مطابقت با هر میزبانی که استفاده می‌کنم».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` حالت fallback مبدأ بر پایهٔ Host-header را فعال می‌کند، اما این یک حالت امنیتی خطرناک است.

  </Accordion>
</AccordionGroup>

```json5
{

    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
