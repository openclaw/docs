وضعیت: رابط کاربری چت SwiftUI در macOS/iOS مستقیما با وب‌سوکت Gateway صحبت می‌کند.

## چیستی آن

- یک رابط کاربری بومی چت برای Gateway (بدون مرورگر جاسازی‌شده و بدون سرور ایستای محلی).
- از همان نشست‌ها و قواعد مسیریابیِ کانال‌های دیگر استفاده می‌کند.
- مسیریابی قطعی: پاسخ‌ها همیشه به WebChat برمی‌گردند.

## شروع سریع

1. Gateway را شروع کنید.
2. رابط کاربری WebChat (اپ macOS/iOS) یا زبانه چت Control UI را باز کنید.
3. مطمئن شوید یک مسیر احراز هویت معتبر Gateway پیکربندی شده است (به‌صورت پیش‌فرض shared-secret،
   حتی روی loopback).

## نحوه کارکرد (رفتار)

- رابط کاربری به وب‌سوکت Gateway وصل می‌شود و از `chat.history`، `chat.send` و `chat.inject` استفاده می‌کند.
- `chat.history` برای پایداری محدود است: Gateway ممکن است فیلدهای متنی طولانی را کوتاه کند، فراداده‌های سنگین را حذف کند، و ورودی‌های بیش‌ازحد بزرگ را با `[chat.history omitted: message too large]` جایگزین کند.
- `chat.history` برای فایل‌های نشست مدرنِ فقط-افزودنی، شاخه رونوشت فعال را دنبال می‌کند؛ بنابراین شاخه‌های بازنویسی رهاشده و نسخه‌های درخواست جایگزین‌شده در WebChat نمایش داده نمی‌شوند.
- ورودی‌های Compaction به‌صورت یک جداکننده صریح تاریخچه فشرده‌شده نمایش داده می‌شوند. این جداکننده توضیح می‌دهد که نوبت‌های قبلی در یک نقطه وارسی حفظ شده‌اند و به کنترل‌های نقطه وارسی نشست‌ها پیوند می‌دهد؛ جایی که اپراتورها، اگر مجوزهایشان اجازه دهد، می‌توانند از نمای پیش از Compaction شاخه بسازند یا آن را بازیابی کنند.
- Control UI شناسه Gateway پشتیبان یعنی `sessionId` را که `chat.history` برمی‌گرداند به خاطر می‌سپارد و آن را در فراخوانی‌های بعدی `chat.send` شامل می‌کند؛ بنابراین اتصال‌های دوباره و تازه‌سازی صفحه همان گفت‌وگوی ذخیره‌شده را ادامه می‌دهند، مگر اینکه کاربر نشستی را شروع یا بازنشانی کند.
- Control UI ارسال‌های تکراری در حال انجام را برای همان نشست، پیام و پیوست‌ها، پیش از ساخت شناسه اجرای جدید `chat.send` ادغام می‌کند؛ Gateway همچنان درخواست‌های تکراری‌ای را که از همان کلید idempotency استفاده می‌کنند حذف تکراری می‌کند.
- `chat.history` برای نمایش نیز نرمال‌سازی می‌شود: زمینه OpenClaw که فقط مربوط به زمان اجراست،
  پوشش‌های envelope ورودی، برچسب‌های دستوری تحویل درون‌خطی
  مانند `[[reply_to_*]]` و `[[audio_as_voice]]`، payloadهای XML فراخوانی ابزار به‌صورت متن ساده
  (از جمله `<tool_call>...</tool_call>`،
  `<function_call>...</function_call>`، `<tool_calls>...</tool_calls>`،
  `<function_calls>...</function_calls>`، و بلوک‌های کوتاه‌شده فراخوانی ابزار)، و
  توکن‌های کنترلی مدل به‌صورت ASCII/تمام‌عرض که نشت کرده‌اند از متن قابل مشاهده حذف می‌شوند،
  و ورودی‌های دستیار که کل متن قابل مشاهده‌شان فقط توکن سکوت دقیق
  `NO_REPLY` / `no_reply` باشد حذف می‌شوند.
- payloadهای پاسخ دارای پرچم reasoning (`isReasoning: true`) از محتوای دستیار WebChat، متن بازپخش رونوشت و بلوک‌های محتوای صوتی کنار گذاشته می‌شوند؛ بنابراین payloadهای فقط-تفکر به‌صورت پیام‌های قابل مشاهده دستیار یا صوت قابل پخش ظاهر نمی‌شوند.
- `chat.inject` یک یادداشت دستیار را مستقیما به رونوشت اضافه می‌کند و آن را به رابط کاربری پخش می‌کند (بدون اجرای عامل).
- اجراهای لغوشده می‌توانند خروجی جزئی دستیار را در رابط کاربری قابل مشاهده نگه دارند.
- Gateway وقتی خروجی بافرشده وجود داشته باشد متن جزئی لغوشده دستیار را در تاریخچه رونوشت پایدار می‌کند و آن ورودی‌ها را با فراداده لغو علامت‌گذاری می‌کند.
- تاریخچه همیشه از Gateway دریافت می‌شود (بدون پایش فایل محلی).
- اگر Gateway در دسترس نباشد، WebChat فقط خواندنی است.

## پنل ابزارهای عامل‌ها در Control UI

- پنل ابزارهای `/agents` در Control UI دو نمای جدا دارد:
  - **همین حالا در دسترس** از `tools.effective(sessionKey=...)` استفاده می‌کند و نشان می‌دهد نشست فعلی
    در زمان اجرا واقعا از چه چیزهایی می‌تواند استفاده کند، از جمله ابزارهای متعلق به هسته، Plugin و کانال.
  - **پیکربندی ابزار** از `tools.catalog` استفاده می‌کند و روی پروفایل‌ها، بازنویسی‌ها و
    معنای کاتالوگ متمرکز می‌ماند.
- دسترس‌پذیری زمان اجرا وابسته به نشست است. تغییر نشست‌ها روی همان عامل می‌تواند فهرست
  **همین حالا در دسترس** را تغییر دهد.
- ویرایشگر پیکربندی به‌معنای دسترس‌پذیری در زمان اجرا نیست؛ دسترسی مؤثر همچنان از تقدم سیاست
  (`allow`/`deny`، بازنویسی‌های مختص عامل و provider/channel) پیروی می‌کند.

## استفاده از راه دور

- حالت راه دور وب‌سوکت Gateway را از طریق SSH/Tailscale تونل می‌کند.
- نیازی نیست یک سرور WebChat جدا اجرا کنید.

## مرجع پیکربندی (WebChat)

پیکربندی کامل: [پیکربندی](/fa/gateway/configuration)

گزینه‌های WebChat:

- `gateway.webchat.chatHistoryMaxChars`: حداکثر تعداد نویسه برای فیلدهای متنی در پاسخ‌های `chat.history`. وقتی یک ورودی رونوشت از این محدودیت بیشتر شود، Gateway فیلدهای متنی طولانی را کوتاه می‌کند و ممکن است پیام‌های بیش‌ازحد بزرگ را با یک placeholder جایگزین کند. مقدار `maxChars` در هر درخواست نیز می‌تواند توسط کلاینت فرستاده شود تا این مقدار پیش‌فرض را برای یک فراخوانی واحد `chat.history` بازنویسی کند.

گزینه‌های سراسری مرتبط:

- `gateway.port`، `gateway.bind`: میزبان/درگاه وب‌سوکت.
- `gateway.auth.mode`، `gateway.auth.token`، `gateway.auth.password`:
  احراز هویت وب‌سوکت با shared-secret.
- `gateway.auth.allowTailscale`: زبانه چت مرورگری Control UI وقتی فعال باشد می‌تواند از هدرهای هویت Tailscale
  Serve استفاده کند.
- `gateway.auth.mode: "trusted-proxy"`: احراز هویت reverse-proxy برای کلاینت‌های مرورگر پشت یک منبع پراکسی **غیر-loopback** آگاه از هویت (ببینید [احراز هویت Trusted Proxy](/fa/gateway/trusted-proxy-auth)).
- `gateway.remote.url`، `gateway.remote.token`، `gateway.remote.password`: هدف Gateway راه دور.
- `session.*`: ذخیره‌سازی نشست و مقادیر پیش‌فرض کلید اصلی.
