本頁記錄 2026 年 5 月 OpenClaw 效能、套件大小、相依性與 shrinkwrap 清理工作的佐證資料。這是公開部落格文章的技術補充文件。

此處合併了兩項稽核：

- **版本效能掃描：**從 `v2026.5.28` 回溯至穩定版 `v2026.4.23` 的 GitHub Releases，使用 `OpenClaw Performance` 工作流程、`profile=smoke` 與模擬供應商執行軌。多數標籤資料列採用單次樣本；`v2026.5.27` 與 `v2026.5.28` 資料列則採用最新的重複 3 次版本分支成品。
- **較早的 4 月背景資料：**從 `v2026.4.1` 到 `v2026.5.2` 已發布的 `clawgrit-reports` 模擬供應商基準，僅用於避免將 4 月下旬有問題的版本視為公開效能基準。
- **安裝占用空間掃描：**在臨時套件中執行全新的 `npm install --ignore-scripts` 安裝，使用 `du -sk node_modules` 測量大小，並走訪 `node_modules` 以統計套件實例數。
- **npm 套件大小掃描：**針對已發布版本執行 `npm pack openclaw@<version> --dry-run --json`，記錄壓縮 tarball 大小、解壓後大小與檔案數。

<Warning>
主要效能掃描對每個標籤使用一次 smoke 樣本，但 `v2026.5.27` 與 `v2026.5.28` 資料列使用最新的重複 3 次版本分支成品。較早的 4 月背景資料使用 `clawgrit-reports` 已發布的重複 3 次中位數。這些數據應視為趨勢佐證與尋找效能衰退的訊號，而非版本閘門統計資料。
</Warning>

### Shrinkwrap 分界點

`2026.5.20` 發布時沒有根層級 shrinkwrap，也沒有龐大的巢狀 OpenClaw
相依套件樹。`2026.5.22` 引入根層級 shrinkwrap，並在巢狀
`openclaw/node_modules` 下安裝了 911.8MB。`2026.5.28` 保留 shrinkwrap，且仍在
巢狀 `openclaw/node_modules` 下安裝 259.7MiB，但在本機全新安裝稽核中已不再安裝
任何 `@napi-rs/canvas` 套件。

已發布 tarball 的檢查驗證了此分界點：

| 版本        | 是否已發布穩定版？ | 根層級 `npm-shrinkwrap.json` | 備註                             |
| ----------- | ------------------ | ---------------------------- | -------------------------------- |
| `2026.5.20` | 是                 | 否                           | 引入 shrinkwrap 前的最後穩定版本 |
| `2026.5.21` | 否                 | 不適用                       | 沒有穩定版 npm 發布              |
| `2026.5.22` | 是                 | 是                           | 引入 shrinkwrap                  |
| `2026.5.23` | 否                 | 不適用                       | 沒有穩定版 npm 發布              |
| `2026.5.24` | 否                 | 不適用                       | 沒有穩定版 npm 發布              |
| `2026.5.25` | 否                 | 不適用                       | 沒有穩定版 npm 發布              |
| `2026.5.26` | 是                 | 是                           | 巢狀相依套件樹仍然存在           |
| `2026.5.27` | 是                 | 是                           | 巢狀相依套件樹仍然存在           |
| `2026.5.28` | 是                 | 是                           | 巢狀相依套件樹大幅縮小           |

重要的區別是：**問題不在 shrinkwrap 本身**。
`v2026.5.28` 仍隨附根層級 shrinkwrap。問題出在套件結構，使 npm 實體化龐大的巢狀
OpenClaw 相依套件樹，以及全部 12 個 `@napi-rs/canvas` 平台套件。`v2026.5.28`
中的巢狀套件樹較小，而 Canvas 平台套件的全面展開也不再出現在本機稽核結果中。

如需以淺白文字了解 shrinkwrap，以及維護者層級的套件檢查，請參閱
[npm shrinkwrap](/zh-TW/gateway/security/shrinkwrap)。

## 供應鏈解讀

相依套件數量是一項營運安全指標，而不只是安裝大小指標。每個套件都會擴大營運者
必須信任的維護者、tarball、遞移更新、選用原生二進位檔，以及安裝期間行為的集合。

清理方向如下：

- 將重量級與選用功能移出預設核心安裝
- 讓外掛套件自行管理其執行階段相依關係圖
- 避免在閘道啟動期間由執行階段套件管理器進行修復
- 在不導致所有平台的原生套件實體化下，維持確定性的安裝結果
- 在套件驗收與測量路徑中維持停用安裝指令碼
- 在發布前偵測巢狀相依套件樹及原生選用相依套件數量暴增
