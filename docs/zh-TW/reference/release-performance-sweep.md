本頁記錄 2026 年 5 月 OpenClaw 效能、套件大小、依賴項與 shrinkwrap 清理背後的證據。它是公開部落格文章的技術配套說明。

這裡結合了兩項稽核：

- **發行效能掃描：** GitHub Releases 從 `v2026.5.28` 回溯到穩定版 `v2026.4.23`，使用 `OpenClaw Performance` 工作流程、`profile=smoke`、mock-provider 通道。大多數標籤列都是一個樣本；`v2026.5.27` 與 `v2026.5.28` 列使用最新的 repeat-3 發行分支成品。
- **較早的 4 月脈絡：** 已發布的 `clawgrit-reports` mock-provider 基準，範圍從 `v2026.4.1` 到 `v2026.5.2`，僅用於避免將 4 月下旬損壞的發行版視為公開效能基準。
- **安裝佔用空間掃描：** 在暫存套件中執行全新的 `npm install --ignore-scripts` 安裝，以 `du -sk node_modules` 取得大小，並走訪 `node_modules` 取得套件執行個體數。
- **npm 套件大小掃描：** 針對已發布的發行版執行 `npm pack openclaw@<version> --dry-run --json`，記錄壓縮 tarball 大小、解壓後大小與檔案數。

<Warning>
主要效能掃描對每個標籤使用一個 smoke 樣本，但 `v2026.5.27` 與 `v2026.5.28` 列例外，使用最新的 repeat-3 發行分支成品。較早的 4 月脈絡使用 `clawgrit-reports` 已發布的 repeat-3 中位數。請將這些數字視為趨勢證據與迴歸狩獵訊號，而不是發行門檻統計。
</Warning>

## 主要數字

不要把 4 月下旬損壞的列用作公開效能基準。`v2026.4.23` 與 `v2026.4.29` 是有用的迴歸證據，但大型 `14x` 這類差異多半是在描述從不良發行線恢復的情況。

### Shrinkwrap 邊界

`2026.5.20` 發佈時沒有根 shrinkwrap，也沒有大型的巢狀 OpenClaw
相依套件樹。`2026.5.22` 引入根 shrinkwrap，並在巢狀
`openclaw/node_modules` 下安裝了 911.8MB。`2026.5.28` 保留 shrinkwrap，且仍在巢狀
`openclaw/node_modules` 下安裝 259.7MiB，但在本機全新安裝稽核中，不再安裝
任何 `@napi-rs/canvas` 套件。

已發佈 tarball 檢查確認了這個邊界：

| 版本        | 已發佈穩定版？ | 根 `npm-shrinkwrap.json` | 備註                                  |
| ----------- | ----------------- | -------------------------- | ------------------------------------- |
| `2026.5.20` | yes               | no                         | shrinkwrap 前的最後一個穩定版        |
| `2026.5.21` | no                | n/a                        | 沒有穩定版 npm 發佈                  |
| `2026.5.22` | yes               | yes                        | 引入 shrinkwrap                       |
| `2026.5.23` | no                | n/a                        | 沒有穩定版 npm 發佈                  |
| `2026.5.24` | no                | n/a                        | 沒有穩定版 npm 發佈                  |
| `2026.5.25` | no                | n/a                        | 沒有穩定版 npm 發佈                  |
| `2026.5.26` | yes               | yes                        | 巢狀相依套件樹仍然存在               |
| `2026.5.27` | yes               | yes                        | 巢狀相依套件樹仍然存在               |
| `2026.5.28` | yes               | yes                        | 巢狀相依套件樹大幅縮小               |

重要區別：**shrinkwrap 本身不是問題**。
`v2026.5.28` 仍然隨附根 shrinkwrap。問題在於套件形態
使 npm 具體生成大型巢狀 OpenClaw 相依套件樹，以及全部 12 個
`@napi-rs/canvas` 平台套件。`v2026.5.28` 中的巢狀樹變小了，
而 canvas 平台扇出不再落入本機稽核。

如需 shrinkwrap 的白話說明與維護者層級的套件
檢查，請參閱 [npm shrinkwrap](/zh-TW/gateway/security/shrinkwrap)。

## 供應鏈解讀

相依套件數量是一項營運安全指標，不只是安裝大小
指標。每個套件都會擴大維運人員必須信任的維護者、tarball、傳遞
更新、選用原生二進位檔，以及安裝時行為的集合。

清理方向是：

- 將大型和選用功能保留在預設核心安裝之外
- 讓外掛套件擁有自己的執行階段相依套件圖
- 避免在閘道啟動期間進行執行階段套件管理器修復
- 保留確定性安裝，同時避免具體生成所有平台的原生套件
- 在套件驗收與測量路徑中保持停用安裝指令碼
- 在發佈前捕捉巢狀相依套件樹與原生選用相依套件爆炸
