## 狀態

提案，第 3 次修訂。尚未實作。方向已於 2026-07 達成共識；第 2 次修訂納入了對抗性審查的發現（專用工作節點協定、放置／環境狀態機、具 Git 感知能力的入站同步、v1 單向交接、受控對外連線的安全措辭）。第 3 次修訂確立同步的所有權模型（工作節點撰寫提交，閘道接納並發布）、新增不使用 Git 的純同步模式、修正工作節點在機器內擁有完整執行權限，將網際網路政策移至佈建階段，並將代理程式分派恢復至里程碑 3。

## 問題

OpenClaw 代理程式工作階段會在單一機器上的閘道程序內執行其迴圈、工具與推論。運算能力受限於該機器，長時間任務會持續占用它，而平行工作則會爭用其資源。託管產品（Cursor cloud agents、Claude Code on the web、Codex cloud）以每個任務各自使用的臨時雲端沙箱解決此問題，但它們需要供應商基礎架構，且必須信任供應商。

已經擁有閒置機器（或能以低廉成本租用機器）的營運者，無法指定：在那台機器上執行此工作階段、像其他工作階段一樣顯示在我的側邊欄中，並在完成後丟棄該機器。

## 目標

- 在臨時遠端機器（「雲端工作節點」）上執行完整的代理程式工作階段（迴圈 + 工具），同時讓該工作階段在 Control UI 中的顯示與串流行為完全如同本機工作階段。
- 工作節點上不存放常駐認證資訊（不含供應商驗證資訊、不含程式碼託管平台權杖），且不可直接對外連線；該機器只需有可連線的 sshd。
- 自動完成佈建、同步、執行、收集、銷毀，並可抽換供應商（第一個供應商：Crabbox 類型的租用命令列介面）。
- 在回合邊界將執行中的工作從閘道分派至工作節點，不遺失逐字稿、工作階段身分，且在請求位元組維持等價時不失去供應商快取親和性；並能安全地拉回結果。
- 人類（UI）與代理程式（工具）皆可將工作分派至雲端工作節點。
- 支援持續數天的工作階段；存續時間由政策決定，而非寫死的上限。

## 非目標（v1）

- 工作節點上不使用外部程式設計工具框架（Claude Code、Codex CLI）。工作節點工作階段只執行 OpenClaw 的內嵌執行器。工具框架支援是 v2 的選用功能，因為工具框架會使用自己的認證資訊自行進行推論。
- 不支援 N 選一最佳結果／平行嘗試展開。
- 不依賴 VPN／tailnet。傳輸僅使用 SSH。
- 不新增沙箱執行環境。工作節點機器本身就是隔離邊界；之後可再疊加機器內的作業系統沙箱。
- v1 不支援對稱的即時移轉：分派方向為本機 → 工作節點；工作節點 → 本機則要求工作階段已停止，且工作區協調已完成。未來的雙向即時交接會建構於相同的屏障機制之上。
- 閘道上不使用 JSON 輔助狀態；環境、放置、游標與授權狀態均儲存在 SQLite 中。

## 先例（我們仿效什麼、反轉什麼）

- Cursor cloud agents：代理程式迴圈在其雲端執行；VM 是工具執行目標；僅附加的對話儲存區會串流至所有用戶端；安裝後建立快照以供暖啟動；自行託管的工作節點是僅對外連線的工作程序。我們仿效「對話的單一真實來源留在協調器上」與串流模型；但反轉迴圈的放置位置（請參閱下方決策）。
- Codex cloud：兩階段執行環境——可連網的設定階段，接著是移除機密資訊的離線代理程式階段。我們仿效此階段切分作為對外連線策略，並將快取概念用於 v2 暖映像檔。
- Claude Code on the web：每個工作階段各有一部 VM；隔離認證資訊的 Git Proxy（真實權杖永遠不會進入沙箱，推送僅限工作階段分支）；設定完成後建立檔案系統快照；傳送交接 = 已推送分支 + 重播的歷史記錄。我們仿效認證資訊隔離與交接架構，但對外同步由閘道透過 rsync 執行，因此未提交變更的工作樹也能運作，且任何靠近該機器的位置都不存在程式碼託管平台權杖。
- Copilot coding agent：預設拒絕對外連線，僅允許套件登錄檔清單。我們的穩定執行階段預設更嚴格（完全不可直接對外連線），因為推論與網路搜尋會透過 SSH 通道送達——但請參閱「安全性」，瞭解為何這是「受控對外連線」，而非「零對外連線」。

## 架構決策：迴圈位於工作節點，推論經由閘道

曾考慮三種放置方式：

1. 迴圈留在閘道，工作節點執行工具（Cursor 模型）。這是最安全的故障域（逐字稿、推論、核准、重新啟動復原全都留在本機），也是審查者偏好的第一個里程碑。未採用作為產品架構：OpenClaw 的非執行類工具是程序內檔案系統操作，因此每次檔案讀取／編輯／grep 都會變成一次網路往返，或必須大幅重構工具介面，改為粗粒度的工作區 RPC；執行環境互動頻繁且受延遲限制。我們會在現有已建置處沿用其精神（將執行卸載至節點），但不建置工具遠端呼叫層。
2. 迴圈與推論都位於工作節點。故障域最簡單，但模型認證資訊（包括 OAuth 設定檔）必須傳送至拋棄式機器，閘道會失去政策／路由／稽核控制，而且移轉會切換呼叫供應商的身分，導致供應商快取失效。
3. 迴圈 + 工具位於工作節點，模型呼叫則透過閘道代理。採用此方案。每個模型回合只需一次往返，而非每次工具呼叫都往返；工具在程式碼旁執行；閘道仍是驗證設定檔、供應商路由與政策的唯一所有者；工作節點不持有任何機密資訊。

選項 3 的代價是每個模型回合期間都同步依賴閘道，因此其耐久性規則是決策的一部分，而非事後補救：

- 在回合進行期間失去閘道會導致作用中的供應商呼叫失敗。該回合會標記為失敗，並於重新連線後作為新回合重試；不會透明重播進行中的供應商串流（存在重複計費／重複工具呼叫的風險）。
- 每項工作節點↔閘道操作都攜帶持久身分（請參閱「工作節點協定」），讓重新連線時可繼續執行或擷取已快取的終止結果，而不會懸置。
- 閘道是受容量管理的元件：並行工作節點限制、流量控制與負載卸除都屬於 v1 範圍（請參閱「容量」）。

由於閘道同時儲存逐字稿並發起所有供應商流量，工作階段與位置無關：在閘道與工作節點之間移動迴圈，不會改變供應商端的任何內容，也不會改變 UI 資料路徑。這正是分派與拉回成本低廉的原因。

## 元件

### 1. 環境狀態機 + 供應商合約

閘道協定中的 `environments.*` 目前只是狀態投影。持久核心是由 SQLite 擁有的環境記錄與狀態機，且先於 RPC 形狀完成設計：

`requested → provisioning → bootstrapping → ready → (attached|idle) → draining → destroying → destroyed | failed | orphaned`

- 佈建具備當機安全性：在呼叫供應商前，會先以確定性的操作 ID 保存意圖資料列，因此閘道重新啟動後可以接管進行中的租用，而不會重複佈建或遺留仍在付費的孤立機器。
- 重新啟動協調與孤立資源清理器（供應商 `inspect` 對照本機記錄）是 v1 要求，而非強化措施。

供應商合約（由外掛實作；核心中不包含供應商名稱或政策）：

```ts
type WorkerProvider = {
  id: string;
  provision(profile: WorkerProfile, opId: string): Promise<WorkerLease>; // → ssh host/port/user/key material
  inspect(lease: { leaseId: string; profile: WorkerProfile }): Promise<LeaseStatus>; // adopt/health/orphan sweep
  renew?(leaseId: string): Promise<void>; // long-lived sessions vs provider TTLs
  destroy(lease: { leaseId: string; profile: WorkerProfile }): Promise<void>; // idempotent, returns only on proof of teardown
};
```

RPC：`environments.create`、`environments.destroy`，以及擴充的 `environments.list/status`（供應商、租用 ID、狀態、存在時間、閒置時間、附加的工作階段）。第一批供應商：Crabbox 形態的租用命令列介面包裝函式（產品路徑），以及標記為僅供開發使用的靜態 SSH 主機供應商——共用主機上的工作節點可能讀取不相關的主機資料，因此靜態主機僅用於功能開發，而非預設安全策略。

### 2. 工作節點啟動程序：在機器上安裝 OpenClaw

不使用特製的工作節點成品，也不依賴 npm 可用性：

- 所有模式的標準安裝方式：由閘道產生、具有內容雜湊的工作節點套件組（將閘道自身的建置輸出封裝為 tarball），透過 SSH 推送並安裝至機器。依其設計，這也涵蓋開發版本與尚未發布的提交。
- 當閘道執行已發布版本時，`npm i -g openclaw@<exact gateway version>` 可作為最佳化方式；絕不使用 `latest`。
- 啟動程序具等冪性；具有相符套件組雜湊的暖租用會跳過安裝。全新機器可能需要可連網的工具鏈階段（Node 執行環境）——這屬於設定階段，之後會關閉網路。
- 交握會驗證工作節點建置雜湊、協定功能集與執行環境相容性。現有閘道版本／協定檢查不足以處理此需求（透過 SSH 通道連線的節點不受精確版本拒絕機制限制），因此工作節點准入會執行自己的精確建置檢查。

工作節點模式（`openclaw worker`）是一個進入點，而非分支版本：包含連線處理與內嵌代理程式執行器，並由閘道 RPC 支援工作階段持久化與模型呼叫。它不得啟動閘道介面：不啟動頻道、不自動啟動工作階段工具集以外的外掛、使用拋棄式狀態目錄，且不包含本機驗證設定檔。

### 3. 傳輸：一切皆透過 SSH

閘道擁有連線能力；工作節點只需要 sshd：

- 閘道會開啟至工作節點的 SSH 連線（認證資訊來自供應商租用，並以佈建輸出固定主機金鑰——不使用 `StrictHostKeyChecking=no`），並建立反向通道，將工作節點本機通訊端轉送至閘道的 WS 端點。
- 控制／模型流量與工作區傳輸會使用不同的 SSH 連線，但共用相同的固定信任資料，因此 rsync 不會因隊頭阻塞而阻礙權杖串流。
- 通道生命週期（保持連線、以退避策略重新連線）由閘道上的環境執行環境擁有。通道短暫中斷在工作階段層級不可見：下述持久協定狀態可讓工作節點重新附加並繼續執行。

### 4. 工作節點協定（專用；不是節點協定）

針對目前節點接縫進行的對抗性審查排除了直接重用：待處理的節點呼叫是程序內 Promise，會隨連線中斷而消失；節點等冪鍵雖會解析，卻不會去重；而且最關鍵的是，已連線節點可以發出一般節點事件（包括代理程式執行請求），因此「節點種類 + 能力上限」並非入站安全邊界。因此，工作節點會取得經驗證的 `worker` 角色，並使用封閉、具版本控制的 RPC／事件允許清單；工作節點連線無法觸及任何舊版節點事件處理常式。

身分與認證資訊：佈建會鑄造一組短效工作節點認證資訊，綁定環境 ID、工作節點金鑰、套件組雜湊、唯一允許的工作階段、允許的 RPC 集合與到期時間。經 SSH 驗證的配對仍然適用（機器由我們佈建且金鑰由我們持有），但授權來自鑄造的認證資訊，而非宣告的節點介面。

持久操作語意（形狀借鑑現有 ACP 執行環境及其事件分類帳——穩定控制代碼、依工作階段序列化、持久的 `(session, seq)` 重播）：

- 每項操作的範圍皆為 `(sessionId, lifecycleRevision, runId, ownerEpoch, streamKind, seq)`。
- 所有權世代會隔離過時的工作節點：替代工作節點會推進世代；舊世代的延遲結果會以確定性方式遭到拒絕。
- 採用至少一次傳遞，並在 SQLite 中保存 ACK 游標與快取的終止結果；去重具確定性。不承諾恰好一次。
- 為取消、關閉、繼續與終止結果提供明確框架；串流採用以信用額度／視窗為基礎的流量控制。
- 協定功能協商獨立於一般節點協定版本。

### 5. 工作階段後端 RPC

兩個不同的契約——目前的程式碼庫將持久性對話記錄變更（由工作階段管理器擁有，採用含父節點／葉節點狀態的 JSONL 樹狀結構）與程序本機即時事件（串流差異、工具生命週期、核准）分開，而工作程式協定必須維持這項區隔：

- 持久性對話記錄提交：工作程式使用 `runEpoch` + 基礎葉節點比較並交換來提交語意附加批次；閘道工作階段管理器產生項目 ID 和父項目 ID。工作程式絕不能提供可信任的對話記錄資料列、項目 ID、父項目 ID 或外部工作階段 ID。
- 可重播的即時事件：使用工作程序號、閘道 ACK、有限保留與延遲事件隔離的型別化事件聯集，輸入現有的代理程式事件分送機制，讓聊天檢視、工具資料列及未讀／狀態邏輯的行為與本機工作階段完全相同。

推論 Proxy：重用現有執行階段 Proxy 串流用戶端（`src/agents/runtime/proxy.ts`）的事件詞彙，但移動信任邊界。工作程式只傳送工作階段／執行身分、已核准的模型參照、脈絡及受限的生成選項；閘道從自己的目錄解析供應商、端點、驗證、標頭、路由及成本政策。拒絕工作程式提供的模型物件（例如由攻擊者控制的 `baseUrl`）。套用請求大小限制、取消、稽核及終止結果重播。位於閘道的工具（websearch）在閘道上執行，並透過相同通道傳回結果。

### 6. 工作區同步

同步錨點是由閘道本機專屬佈署擁有的工作區：對於 git 工作區，使用專用的受管理 worktree（現有的受管理 worktree 中繼資料——分支、基礎版本、快照擁有權——是其基礎）；對於非 git 工作區，則使用閘道擁有的目標目錄。絕不使用使用者的即時 checkout。工作階段遠端佈署期間的專屬擁有權，使輸入同步在結構設計上不會發生衝突。

擁有權區分——提交與發布：

- 工作程式端的代理程式在自己的副本中正常建立提交（`git commit` 是本機且不需要認證資訊的操作；作者身分由閘道設定投射）。在閘道採用這些提交之前，它們都只是非作用中的物件。
- 閘道執行所有需要信任的操作：驗證輸入提交建構於記錄的基礎版本之上、將本機 worktree 快轉、push、建立 PR，以及選用的簽署／重新簽署——全部使用閘道本機的認證資訊。工作程式絕不持有 git 或程式碼託管平台的認證資訊，也絕不接觸遠端。

兩種同步模式，依工作區是否為 git 儲存庫選擇：

- Git 模式。輸出：透過通道的 SSH 身分使用 rsync 同步 worktree（包含未提交及符合條件的未追蹤檔案；採用 crabbox 風格的包含／排除規則，並遵循 `.worktreeinclude`），記錄為不可變的基礎資訊清單（內容雜湊 + 基礎提交）。輸入：新提交以針對記錄基礎版本的 git bundle 或暫時 ref 傳回；未追蹤成品透過明確的資訊清單傳回，並接受大小／類型／符號連結限制範圍檢查。採用時會驗證基礎版本祖先關係，並在發生分歧時停止——不會以無提示方式覆寫任一端。刪除、重新命名、子模組及符號連結逸出由資訊清單規則處理，而非依賴 rsync 啟發式規則。
- 純文字模式（無 git——例如在機器上從頭建立專案）。輸出使用相同的 rsync + 基礎資訊清單。輸入則透過資訊清單差異比對，鏡像回閘道擁有的目標目錄，並傳播刪除操作。其安全原因與 git 模式相同：專屬擁有權表示不存在會造成衝突的並行本機編輯；基礎資訊清單仍會偵測非預期的本機漂移，並停止而非覆寫。

檢查點可保護持續數日的工作階段免受租用遺失影響：定期建立輸入檢查點（git 模式中的工作階段分支提交、純文字模式中的資訊清單快照）；頻率由設定檔政策決定（預設依回合）。

### 7. 佈署狀態機、工作階段與 UI

執行階段佈署是由 SQLite 擁有、以工作階段為索引鍵的狀態機，而非一對鬆散的資料列欄位：

`local → requested → provisioning → syncing → starting → active(worker) → draining → reconciling → local | reclaimed | failed`

它會保存環境 ID、轉換世代、作用中擁有者 epoch、工作區基礎資訊清單、工作程式 bundle 雜湊及最後的 ACK 游標。回合准入會在任一迴圈開始回合前，以不可分割方式宣告佈署擁有權，因此根據過時快照准入的本機訊息絕不會與工作程式回合發生競爭——任何時刻都恰好只有一個迴圈擁有該工作階段。

UI：

- 工作程式工作階段是一般工作階段資料列加上佈署中繼資料。它位於一般儲存區中、透過 `sessions.list` 列出，並透過現有訂閱進行串流——側邊欄和聊天不需要新的資料路徑，只需新增呈現方式：工作程式徽章及佈署／環境狀態（`provisioning / syncing / running / idle / reconciling / reclaimed`）。
- 建立 UX：工作階段目標列（工作階段側邊欄重新設計）會在閘道與節點之外，新增雲端工作程式目的地。需要已設定的供應商設定檔；設定完成前，此功能不可見。
- 代理程式分派：工作階段工具讓代理程式可如同人類一樣將工作交給雲端工作程式（由工作程式支援的子工作階段，採用子代理程式風格）。與人工作業分派在同一里程碑推出，並由相同的選用供應商設定控管。遞迴在結構上受到限制（工作程式工作階段本身在 v1 中無法再分派工作程式）；支出控制採用各環境的計費／稽核，而非配額機制。

## 分派與交接

v1 刻意採用非對稱設計：

- 本機 → 工作程式（分派）：通過下方的移轉屏障、佈建或重用工作程式、同步、切換佈署，下一個回合在遠端執行。
- 工作程式 → 本機（拉回）：停止工作階段（依相同屏障排空工作程式）、完成輸入協調、將佈署切換至本機。這不是即時移轉。
- 對稱即時交接（在不停止的情況下雙向移動正在作用中的工作階段）重用相同的屏障與協調機制，並在錯誤注入測試證明該屏障後推出。

移轉屏障（僅有「回合邊界」並不足夠——核准、背景程序及釋放鎖定後的對話記錄合併都可能跨越該邊界）：

1. 停止新的回合准入（佈署宣告）。
2. 取消或排空作用中的執行。
3. 撤銷待處理的 exec 核准與執行授權。
4. 排空對話記錄的旁路寫入與即時事件 ACK。
5. 終止工作程式子程序。
6. 推進擁有者 epoch，以隔離舊擁有者。
7. 協調工作區（輸入、可感知衝突）。
8. 啟用新擁有者。

快取親和性：由於兩種佈署中的供應商請求都從閘道發出，因此只要序列化後的供應商請求維持等價，便能保留快取親和性——相同的工具順序、系統指示、供應商包裝函式，以及快取中繼資料（後者留在閘道端）。這是可測試的屬性，而非假設：在各支援的供應商傳輸中，跨本機／工作程式佈署的位元組等價測試，會納入引入工作程式迴圈的里程碑。

## 安全模型

精確而言：工作程式沒有直接網路輸出，也沒有常駐的供應商／程式碼託管平台認證資訊。這並非「零輸出」——推論及閘道執行的工具是受控輸出通道（遭提示注入的工作程式仍可將工作區位元組放入模型脈絡或 websearch 查詢）。因此：

- 受控輸出計量：對推論 Proxy 及閘道工具執行各環境稽核，並提供操作人員可見的計量。速率／位元組限制以協定流量控制（容量）形式存在，而非支出配額機制。
- 工作程式對閘道的輸入僅限封閉的工作程式協定允許清單；對話記錄寫入在結構上受到限制（由閘道產生 ID，且僅限單一綁定工作階段）。
- 工作程式 exec 在機器內具有完整權限。該機器可拋棄且不含認證資訊，因此逐命令核准只會增加阻力，無法提供任何保護；受防護的邊界是輸入協調與稽核。Exec 絕不經過閘道的節點核准路徑。
- 網際網路政策是佈建時的供應商決策：環境設定檔在建立機器時決定（防火牆／安全性群組／無輸出網路），也可選擇允許連網的設定階段，並由供應商在代理程式階段開始前關閉。核心不實作執行階段網路切換。
- 佈建時的機器衛生：封鎖或確認不存在雲端中繼資料端點、無執行個體設定檔、無繼承的 SSH agent、無 Docker socket，並使用乾淨的環境／家目錄。SSH 主機金鑰從佈建輸出釘選。
- 所有閘道端操作（push、PR、供應商呼叫）的核准與政策仍在閘道上執行。

工作程式工作階段遭入侵時的影響範圍：已同步的工作區副本，加上經稽核的 Proxy 通道所允許的內容——無認證資訊、無直接網路、除允許清單外無其他閘道介面。

## 容量

閘道會轉送 N 個工作程式的每個提示與權杖串流，因此 v1 會明確定義容量模型，而非等到正式環境才發現問題：各閘道的並行工作程式限制、各串流的額度視窗（目前的事件串流佇列無上限，而節點 socket 緩衝區上限會強制關閉速度較慢的消費者——兩者都不適合原封不動地使用）、用於突發流量的有限磁碟暫存，以及在 UI 中顯示可見反壓狀態的負載卸除。工作區傳輸保留在自己的 SSH 通道上。

## 生命週期

- 閒置自動停止與 TTL 是供應商設定檔政策，而非固定常數。預設值寬裕且提供明確的保持連線；持續數日的工作是一級使用情境（針對採用租用機制的後端，供應商提供 `renew`）；具有執行中回合或最近有活動的工作階段絕不會被回收。
- 工作程式死亡或遭回收時：佈署移至 `reclaimed`，工作階段資料列保留；下一則訊息會佈建新的工作程式，並從最後一個檢查點重新同步。對話絕不會遺失（位於閘道端儲存區）；最後一個檢查點後的工作區變更會遺失，且 UI 會明確告知。
- 從第一天起就支援暖租用重用（限支援的供應商）；啟動程序後建立映像快照是 v2 的快速啟動路徑。

## 設定介面

最小化且採選用方式：供應商設定檔區塊（供應商 ID、認證資訊／命令列介面參照、同步規則、生命週期政策、預算、選用設定階段），加上各工作階段的佈署選擇。不新增環境變數。未設定的安裝不會看到任何相關功能。

## 里程碑

實作會以小型、可獨立合併的 PR 形式落地；下列每個里程碑都是一系列 PR，而非單一變更。

1. 基礎：環境狀態機 + 供應商契約 + crabbox 形式的供應商（以 static-SSH 作為開發測試框架）、工作程式 bundle 啟動程序 + 准入交握、SSH 通道 + 主機金鑰釘選、受管理 worktree 快照 + 輸出同步（git + 純文字模式）。孤立項目清理 + 重新啟動後採用。
2. 工作程式協定 + 工作程式迴圈：經驗證的工作程式角色、持久性操作／epoch／ACK 游標、對話記錄提交 + 即時事件契約、使用閘道解析模型的推論 Proxy、流量控制。一個供應商、僅支援人工作業分派新工作階段、不支援交接。以錯誤注入測試（通道分割、閘道重新啟動、工作程式死亡）作為完成關卡。
3. 分派 + 拉回 + 代理程式分派：移轉屏障、連接至 UI 目標列的佈署狀態機、輸入協調 + 檢查點、各環境稽核、容量限制、代理程式分派工具（工作程式工作階段無法遞迴）。提示快取位元組等價測試。
4. 對稱即時交接，在里程碑 3 的錯誤注入證明完成後推出。

後續：依各環境認證資訊注入的選用設定，在工作程式上執行 ACP 測試框架；快照／暖映像快速啟動；扇出（N 個租用、相同提示）；機器內作業系統沙箱；透過成品結構描述擷取更豐富的成品。

## 待釐清問題

- 工作節點上的外掛／Skill 可用性：儲存庫隨附的 Skill 會隨工作區自動同步，無須額外處理；透過閘道設定的代理程式 Skill／外掛，則需要明確決定要同步或排除（無論採取哪種方式，工具／外掛資訊清單都屬於准入交握的一部分）。
- 檢查點頻率預設值：對話非常頻繁的工作階段應採用以輪次為準或以時間為準。
- 環境設定檔如何與多代理程式路由互動（每個代理程式的預設設定檔，或僅限每個工作階段選取）。
