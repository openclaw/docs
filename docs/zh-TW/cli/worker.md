# `openclaw worker`

`openclaw worker` 是供雲端工作節點協調器在已準備好的工作節點環境中啟動的受限執行階段進入點。它不是用於手動註冊工作節點的通用命令。

閘道會安裝相符的 OpenClaw 套件組合，並開啟已固定主機金鑰的反向 SSH 通道。工作節點啟動器會使用已準備好的指派來啟動此命令。此命令會透過通道轉送的本機通訊端連線，並以專用的 `worker` 角色完成准入。

## 啟動合約

此命令會從標準輸入讀取恰好一個有界限的 JSON 啟動封套。封套包含本機通訊端位置、簽發給工作節點的認證資訊、套件組合與通訊協定識別資訊、擁有者週期，以及唯一獲指派的工作階段與回合。認證資訊絕不透過命令列引數接受，而且本頁刻意不提供認證資訊或手動撰寫的封套範例。

如果封套無效、認證資訊遭拒、套件組合或通訊協定功能不相符，或工作階段與擁有者週期已不再是目前版本，准入就會以封閉方式失敗。操作人員應透過雲端工作節點協調器啟動工作節點，而不是直接叫用此進入點。

## 執行階段邊界

此程序會使用受限後端執行一般的嵌入式代理程式迴圈：

- `read`、`write`、`edit`、`apply_patch`、`exec` 與 `process` 程式設計工具
  會在工作節點工作區中於本機執行。
- 模型呼叫會使用閘道推論代理。不會載入本機模型驗證設定檔。
- 逐字稿寫入會使用閘道逐字稿提交 RPC。
- 串流與工具生命週期更新會使用閘道即時事件 RPC。
- 只接受獲指派的工作階段與回合。

工作節點模式不會啟動頻道、閘道 HTTP 介面，或獲指派工作階段工具組以外的外掛自動啟動項目。它使用可拋棄的狀態目錄，且不持有常駐的供應商或程式碼託管服務認證資訊。

此模式不提供工作節點對工作節點的工作階段分派。配置與分派仍由閘道掌控：操作人員可以透過閘道分派現有的本機受管理工作樹工作階段，但工作節點程序無法分派自身或其他工作節點。

已準備好的指派包含逐字稿脈絡、已接受的基準葉節點、提交序號與即時事件游標。通道重新連線時，程序會使用相同的認證資訊與擁有者週期重新准入、保留已接受的逐字稿基準、重播尚未確認的即時事件尾端，並以相同識別資訊重新附加至進行中的推論回合。如果遺漏了串流差異內容，終止推論訊息具有最終權威。取代現有狀態的擁有者週期會隔離該程序，並使其正常結束。

`stale-base-leaf` 逐字稿拒絕會使目前執行立即停止。工作節點模式不會針對不同葉節點重試遭拒的序號，因此不會產生重複提交；該次執行中仍未提交且僅存在於記憶體內的任何尾端資料都會遺失。重新啟動屬於里程碑 3 配置擁有者的職責，其必須根據閘道的權威逐字稿與提交帳本建立全新的指派。同樣地，閘道程序重新啟動會以供應商錯誤終止待處理的推論回合；只有通道或工作節點 WebSocket 重新連線，才能重新附加至同一程序中仍在作用的推論串流。

如需封閉式工作節點 RPC 介面的資訊，請參閱[閘道通訊協定](/zh-TW/gateway/protocol#worker-role-and-closed-protocol)；如需架構與安全模型的資訊，請參閱[雲端工作節點計畫](/zh-TW/plan/cloud-workers)。
