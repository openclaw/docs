節點配對有兩個層級，兩者都儲存在閘道 SQLite 狀態資料庫的已配對裝置記錄中：

- **裝置配對**（角色 `node`）控管 `connect` 交握。請參閱下方的
  [受信任 CIDR 裝置自動核准](#trusted-cidr-device-auto-approval)
  以及[頻道配對](/zh-TW/channels/pairing)。
- **節點能力核准**（`node.pair.*`）控管已連線節點可公開哪些宣告的
  能力／命令。閘道是唯一真實來源；使用者介面（macOS 應用程式、Control UI）是用來核准或
  拒絕待處理要求的前端。

先前獨立的節點配對儲存區（`nodes/paired.json`，含每個節點各自的
權杖，已於 2026 年 1 月從連線路徑中淘汰）已移除：閘道會在啟動時一次性將
所有剩餘資料列併入裝置記錄，並以 `.migrated` 後綴封存
舊版檔案。舊版 TCP 橋接支援已移除。

## 能力核准的運作方式

1. 節點連線至閘道 WS（此步驟受裝置配對控管）。
2. 閘道會將宣告的能力／命令介面與已核准的介面進行比較；新的或擴大的介面會在
   裝置記錄上儲存一筆**待處理要求**，並發出 `node.pair.requested`。
3. 你核准或拒絕該要求（透過命令列介面或使用者介面）。
4. 核准之前，節點命令會持續遭到篩選；核准後會公開宣告的
   介面，但仍受一般命令原則約束。

待處理要求會在**節點最後一次重試後 5 分鐘**自動過期——持續主動重新連線的節點會讓其唯一的待處理要求保持有效，
而不是每次嘗試都產生新的要求（與核准提示）。

## 命令列介面工作流程（適合無頭環境）

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` 會顯示已配對／已連線的節點及其能力。

## API 介面（閘道通訊協定）

事件：

- `node.pair.requested` - 建立新的待處理要求時發出。
- `node.pair.resolved` - 要求獲得核准、遭到拒絕或
  過期時發出。

方法：

- `node.pair.list` - 列出待處理與已配對的節點（`operator.pairing`）。
- `node.pair.approve` - 核准待處理要求。
- `node.pair.reject` - 拒絕待處理要求。
- `node.pair.remove` - 移除已配對的節點。這會撤銷已配對裝置儲存區中該裝置的 `node`
  角色，同時移除已核准的節點介面，並使該裝置的節點角色工作階段失效／中斷連線。**混合角色**
  裝置（例如同時具有 `operator` 的裝置）會保留其資料列，且只會
  失去 `node` 角色；只有節點角色的裝置資料列則會遭到刪除。授權：
  `operator.pairing` 可移除非操作員節點資料列；使用裝置權杖的呼叫者若要在混合角色裝置上
  撤銷其**自身的**節點角色，還需要
  `operator.admin`。
- `node.rename` - 重新命名已配對節點面向操作員的顯示名稱。

於 2026.7 移除：`node.pair.request` 和 `node.pair.verify`。待處理
要求會在節點連線期間由閘道自行建立，而它們所服務的
獨立每節點權杖已不存在；節點驗證使用裝置配對權杖。

注意事項：

- 使用未變更介面重新連線時會重複使用待處理要求；重複
  要求會重新整理儲存的節點中繼資料，以及最新列入允許清單的
  宣告命令快照，供操作員查看。
- 操作員範圍層級與核准時檢查摘要於
  [操作員範圍](/zh-TW/gateway/operator-scopes)。
- `node.pair.approve` 使用待處理要求所宣告的命令強制執行
  額外的核准範圍：
  - 無命令要求：`operator.pairing`
  - 一般命令要求：`operator.pairing` + `operator.write`
  - 包含 `system.run`、`system.run.prepare`、
    `system.which`、`browser.proxy`、`fs.listDir` 或
    `system.execApprovals.get/set` 的管理員敏感要求：`operator.pairing` + `operator.admin`

<Warning>
節點配對核准會記錄受信任的能力介面。它**不會**針對各節點固定即時節點命令介面。

- 即時節點命令來自節點連線時所宣告的內容，並由
  閘道的全域節點命令原則（`gateway.nodes.allowCommands` 和
  `denyCommands`）進行篩選。
- 每個節點的 `system.run` 允許與詢問原則儲存在
  `exec.approvals.node.*` 的節點上，而非配對記錄中。

</Warning>

## 節點命令控管（2026.3.31+）

<Warning>
**重大變更：**自 `2026.3.31` 起，在節點配對獲得核准之前，節點命令均為停用狀態。僅完成裝置配對已不足以公開所宣告的節點命令。
</Warning>

節點首次連線時，系統會自動要求配對。
在該要求獲得核准前，來自該節點的所有待處理節點命令都會遭到
篩選，且不會執行。配對獲得核准後，節點所宣告的
命令便可使用，但仍受一般命令原則約束。

這表示：

- 先前僅依賴裝置配對來公開命令的節點，現在
  也必須完成節點配對。
- 在配對核准前排入佇列的命令會遭到捨棄，而非延後執行。

## 節點事件信任邊界（2026.3.31+）

<Warning>
**重大變更：**節點發起的執行現在會維持在縮減的受信任介面內。
</Warning>

節點發起的摘要和相關工作階段事件僅限於
預期的受信任介面。先前依賴更廣泛主機或工作階段工具存取權的通知驅動或節點觸發流程
可能需要調整。
這項強化措施可防止節點事件升級為超出
節點信任邊界所允許的主機層級工具存取權。

持久節點上線狀態更新遵循相同的身分邊界：
僅接受來自已驗證節點裝置工作階段的
`node.presence.alive` 事件，且只有在裝置／節點身分
已配對時才會更新配對中繼資料。僅自行宣告 `client.id` 值不足以寫入
最後出現狀態。

## 經 SSH 驗證的裝置自動核准（預設）

當閘道可透過 SSH **證明機器所有權**時，來自私人／CGNAT 位址的首次
`role: node` 裝置配對會自動獲得核准：它會
反向連線至配對主機（`BatchMode`、`StrictHostKeyChecking=yes`），
在該處執行 `openclaw node identity --json`，且僅在遠端
裝置 ID 與公開金鑰和待處理要求完全相符時才核准。金鑰相符
正是此機制安全的原因：僅能連線絕不會觸發核准，因此 NAT 共用租戶、
共用主機上的其他使用者以及區域網路欺騙都會回退至一般
提示。

預設啟用。觸發條件：

- 閘道處理程序使用者（或 `sshVerify.user`）可以非互動方式透過 SSH 連線至節點主機
  （金鑰／代理程式；也支援 Tailscale SSH），且主機金鑰
  已受信任。
- `openclaw` 可在遠端 `PATH` 上解析，以供非互動式 `sh -lc` 使用。
- 連線 IP 是直接（非經代理、非回送）的私人、ULA、
  連結本機或 CGNAT 位址，或在設定 `sshVerify.cidrs` 時與其相符。
- 與受信任 CIDR 核准採用相同的資格門檻：僅限全新且無範圍的節點
  配對；升級、瀏覽器、Control UI 和 WebChat 一律顯示提示。

探查執行期間，節點用戶端會收到持續重試
（`wait_then_retry`）的指示，而不是暫停並等待手動核准；若探查
失敗，下一次嘗試會回退至一般提示流程。失敗的目標
會進入短暫冷卻期（金鑰不相符後 5 分鐘）。

獲核准的裝置會記錄 `approvedVia: "ssh-verified"`，且其首次宣告的
能力介面會在同一步驟獲得核准——金鑰相符已能證明
節點是在操作員所擁有機器上的操作員帳號下執行，這與
手動能力核准所確認的聲明相同。後續介面升級仍會顯示
提示。

強化或停用：

```json5
{
  gateway: {
    nodes: {
      pairing: {
        // 完全停用：
        sshVerify: false,
        // ……或限制範圍／調整探查：
        // sshVerify: { user: "me", identity: "~/.ssh/probe", timeoutMs: 7000, cidrs: ["10.0.0.0/8"] },
      },
    },
  },
}
```

## 自動核准（macOS 應用程式）

在下列情況下，macOS 應用程式可嘗試**靜默核准**節點能力要求：

- 要求標記為 `silent`（當裝置配對以非互動方式獲得核准時，閘道會將第一個能力
  介面標記為靜默），且
- 應用程式可使用相同
  使用者驗證與閘道主機的 SSH 連線。

若靜默核准失敗，則會回退至一般 Approve/Reject 提示。

## 受信任 CIDR 裝置自動核准

`role: node` 的 WS 裝置配對預設維持手動。對於閘道已信任網路路徑的私人節點
網路，操作員可使用明確的 CIDR 或確切 IP 選擇啟用：

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

安全邊界：

- 未設定 `gateway.nodes.pairing.autoApproveCidrs` 時停用。
- 不存在全面自動核准區域網路或私人網路的模式；經 SSH 驗證的
  自動核准（如上）必須有密碼學裝置金鑰相符，絕不會
  僅依賴網路位置。
- 僅限未要求任何範圍的全新 `role: node` 裝置配對要求
  符合資格。
- 操作員、瀏覽器、Control UI 和 WebChat 用戶端維持手動。
- 角色、範圍、中繼資料和公開金鑰升級維持手動。
- 同一主機的回送受信任代理標頭路徑不符合資格，因為本機呼叫者
  可偽造該路徑。

## 靜默配對取代清理

非互動式核准會在已配對裝置資料列上記錄其來源：
同一主機的本機原則核准記為 `silent`，受信任 CIDR 節點核准記為
`trusted-cidr`，經 SSH 驗證的節點核准記為 `ssh-verified`。狀態目錄為暫時性的用戶端（暫存家目錄、
容器、每次執行各自的沙箱）會在每次執行時產生新的裝置金鑰組，而每次
執行都會以全新裝置靜默重新配對——若不進行清理，已配對清單
每次執行都會增加一筆過時資料列。

當閘道靜默核准**本機**裝置配對時，會淘汰
屬於相同用戶端叢集（`clientId`、`clientMode` 和顯示名稱皆相符）且目前未
連線的較舊 `silent` 核准記錄。本機用戶端是在閘道主機本身執行，因此叢集金鑰
不可能與其他機器相符。已淘汰資料列的權杖會立即失效；
任何相符的舊版節點配對項目都會遭到清除，並廣播 `node.pair.resolved`
移除事件。

邊界：

- 只有最新核准方式為同主機本機核准（`silent`）的記錄，才有資格作為觸發方與目標。受信任 CIDR 及經 SSH 驗證的配對會跨越不同主機，而其顯示中繼資料並非機器身分，因此絕不會自動移除——請使用控制介面清理功能或 `openclaw nodes remove` 來處理。
- 擁有者核准及 QR／設定碼（啟動）配對絕不會自動移除。在來源資訊存在之前核准的記錄會持續受到保護，即使之後以無提示方式重新核准相同的裝置 ID 亦然。
- 目前已連線的裝置會被略過，因此使用不同狀態目錄的並行本機工作階段，會在連線期間保留其權杖。最近一分鐘內核准的記錄也會被略過，因此同時進行的配對交握不會在彼此的連線完成註冊前將對方撤銷。
- 受影響的用戶端在設計上皆為本機用戶端，因此會在下次連線時以無提示方式重新配對。

## 中繼資料升級自動核准

當已配對的裝置重新連線，且只有非敏感中繼資料發生變更（例如顯示名稱或用戶端平台提示）時，OpenClaw 會將其視為 `metadata-upgrade`。無提示自動核准的適用範圍有限：僅適用於受信任、非瀏覽器的本機重新連線，且這些連線已證明持有本機或共用認證資訊；其中包括作業系統版本中繼資料變更後，同主機原生應用程式的重新連線。瀏覽器／控制介面用戶端及遠端用戶端仍須使用明確的重新核准流程。範圍升級（從讀取升級為寫入／管理員）及公開金鑰變更**不**符合中繼資料升級自動核准的資格；這些仍會保留為明確的重新核准要求。

## QR 配對輔助工具

`/pair qr` 會將配對承載資料呈現為結構化媒體，讓行動裝置及瀏覽器用戶端可直接掃描。

刪除裝置時，也會清除該裝置 ID 所有過時且待處理的配對要求，因此撤銷後，`nodes pending` 不會顯示無所屬的資料列。

## 本機性與轉送標頭

只有在原始通訊端與所有上游 Proxy 證據一致時，閘道配對才會將連線視為迴路位址。如果要求透過迴路位址送達，但帶有 `Forwarded`、任何 `X-Forwarded-*` 或 `X-Real-IP` 標頭證據，該轉送標頭證據便會使迴路位址的本機性宣告失效；配對路徑將要求明確核准，而不會以無提示方式將要求視為同主機連線。關於操作者驗證的等效規則，請參閱[受信任 Proxy 驗證](/zh-TW/gateway/trusted-proxy-auth)。

## 儲存空間（本機、私密）

配對狀態位於共用 SQLite 狀態資料庫中的已配對裝置記錄內；該資料庫位於閘道狀態目錄下（預設為 `~/.openclaw`）：

- `~/.openclaw/state/openclaw.sqlite`（包含裝置驗證資訊、已核准的節點介面、待處理的介面要求、待處理的裝置配對要求及啟動權杖之已配對裝置）

如果覆寫 `OPENCLAW_STATE_DIR`，資料庫也會隨之移動。從使用 JSON 儲存區的版本升級之閘道，會在啟動時匯入這些資料，並留下 `devices/*.json.migrated` 與 `nodes/*.json.migrated` 封存檔。

安全性注意事項：

- 裝置權杖是機密資訊；請將狀態資料庫視為敏感資料。
- 輪替裝置權杖會使用 `openclaw devices rotate`／`device.token.rotate`。

## 傳輸行為

- 傳輸是**無狀態的**；不會儲存成員資格。
- 如果閘道離線或配對功能已停用，節點便無法配對。
- 在遠端模式下，配對會針對遠端閘道的儲存區進行。
