節點配對分為兩層，兩者都儲存在閘道的 SQLite 狀態資料庫中的已配對裝置記錄上：

- **裝置配對**（角色 `node`）會管控 `connect` 交握。請參閱下方的
  [受信任 CIDR 裝置自動核准](#trusted-cidr-device-auto-approval)
  以及[頻道配對](/zh-TW/channels/pairing)。
- **節點能力核准**（`node.pair.*`）會管控已連線節點可公開哪些宣告的
  能力／命令。閘道是唯一真實來源；UI（macOS app、Control UI）則是核准或
  拒絕待處理請求的前端。

先前獨立的節點配對儲存區（`nodes/paired.json`，含每個節點各自的
權杖，已於 2026 年 1 月退出連線路徑）現已移除：閘道會在啟動時一次性將
任何剩餘資料列整併至裝置記錄，並以 `.migrated` 後綴封存
舊版檔案。舊版 TCP 橋接支援已移除。

## 能力核准的運作方式

1. 節點連線至閘道 WS（裝置配對會管控此步驟）。
2. 閘道會比較宣告的能力／命令介面與已核准的介面；新增或擴大的介面會在
   裝置記錄中儲存一筆**待處理請求**，並發出 `node.pair.requested`。
3. 你可以核准或拒絕該請求（命令列介面或 UI）。
4. 核准前，節點命令會維持被篩除；核准後會公開宣告的
   介面，但仍受一般命令政策約束。

待處理請求會在**節點最後一次重試後 5 分鐘**自動到期——持續主動重新連線的節點會讓其唯一的待處理請求保持有效，而不是每次嘗試都產生新的請求（和核准提示）。

## 命令列介面工作流程（適合無頭環境）

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` 會顯示已配對／已連線的節點及其能力。

## API 介面（閘道通訊協定）

事件：

- `node.pair.requested` - 建立新的待處理請求時發出。
- `node.pair.resolved` - 請求獲核准、遭拒絕或
  到期時發出。

方法：

- `node.pair.list` - 列出待處理和已配對的節點（`operator.pairing`）。
- `node.pair.approve` - 核准待處理請求。
- `node.pair.reject` - 拒絕待處理請求。
- `node.pair.remove` - 移除已配對的節點。這會在已配對裝置儲存區中撤銷該裝置的 `node`
  角色，同時移除已核准的節點介面，並
  使該裝置的節點角色工作階段失效／中斷連線。**混合角色**
  裝置（例如同時持有 `operator` 的裝置）會保留其資料列，且只會
  失去 `node` 角色；僅具節點角色的裝置資料列則會被刪除。授權：
  `operator.pairing` 可移除非操作者節點資料列；由裝置權杖呼叫者
  撤銷混合角色裝置上**自己的**節點角色時，還需要
  `operator.admin`。
- `node.rename` - 重新命名已配對節點面向操作者的顯示名稱。

已於 2026.7 移除：`node.pair.request` 和 `node.pair.verify`。待處理
請求由閘道本身在節點連線期間建立，而它們所服務的
獨立每節點權杖已不復存在；節點驗證使用
裝置配對權杖。

注意事項：

- 介面未變更的重新連線會重複使用待處理請求；重複
請求會重新整理儲存的節點中繼資料，以及最新列入允許清單的
宣告命令快照，供操作者查看。
- 操作者範圍層級與核准時檢查摘要請參閱
  [操作者範圍](/zh-TW/gateway/operator-scopes)。
- `node.pair.approve` 會使用待處理請求中宣告的命令，強制執行
  額外核准範圍：
  - 無命令請求：`operator.pairing`
  - 一般命令請求：`operator.pairing` + `operator.write`
  - 包含 `system.run`、`system.run.prepare`、
    `system.which`、`browser.proxy`、`fs.listDir` 或
    `system.execApprovals.get/set` 的管理員敏感請求：`operator.pairing` + `operator.admin`

<Warning>
節點配對核准會記錄受信任的能力介面。它**不會**針對各節點固定即時節點命令介面。

- 即時節點命令來自節點連線時的宣告，並由
  閘道的全域節點命令政策（`gateway.nodes.commands.allow` 和
  `gateway.nodes.commands.deny`）篩選。
- 每節點的 `system.run` 允許與詢問政策位於
  `exec.approvals.node.*` 中的節點上，而非配對記錄中。

</Warning>

## 節點命令管控（2026.3.31+）

<Warning>
**重大變更：**自 `2026.3.31` 起，節點命令在節點配對獲核准前會停用。僅完成裝置配對已不足以公開宣告的節點命令。
</Warning>

節點首次連線時，系統會自動要求配對。
在該請求獲核准前，來自該節點的所有待處理節點命令都會
被篩除且不會執行。配對獲核准後，節點宣告的
命令即可使用，但仍受一般命令政策約束。

這表示：

- 先前僅依賴裝置配對來公開命令的節點，現在
  也必須完成節點配對。
- 在配對核准前排入佇列的命令會被捨棄，而非延後執行。

## 節點事件信任邊界（2026.3.31+）

<Warning>
**重大變更：**由節點發起的執行現在會限制在縮減後的受信任介面內。
</Warning>

由節點發起的摘要及相關工作階段事件會受限於
預期的受信任介面。先前依賴較廣泛主機或工作階段工具存取權的
通知驅動或節點觸發流程可能需要調整。
此強化措施可防止節點事件提升至
超出節點信任邊界所允許的主機層級工具存取權。

持久性節點上線狀態更新遵循相同的身分邊界：
只有來自已驗證節點裝置工作階段的
`node.presence.alive` 事件才會被接受，且只有裝置／節點身分
已配對時才會更新配對中繼資料。自行宣告的 `client.id` 值不足以寫入
最後上線狀態。

## SSH 驗證的裝置自動核准（預設）

當閘道可透過 SSH **證明機器擁有權**時，來自私人／CGNAT 位址的首次
`role: node` 裝置配對會自動獲核准：閘道會
反向連線至配對主機（`BatchMode`、`StrictHostKeyChecking=yes`），
在該處執行 `openclaw node identity --json`，且只有遠端
裝置 ID 與公開金鑰完全符合待處理請求時才會核准。金鑰比對
是此機制安全的關鍵：僅能連線永遠不會觸發核准，因此 NAT 共用租戶、
共用主機上的其他使用者，以及區域網路偽造都會回退至一般
提示流程。

預設啟用。觸發條件：

- 閘道處理程序使用者（或 `sshVerify.user`）可透過 SSH 以
  非互動方式連線至節點主機（金鑰／代理程式；Tailscale SSH 亦可），且主機金鑰
  已受信任。
- `openclaw` 可在遠端 `PATH` 上針對非互動式 `sh -lc` 正確解析。
- 連線 IP 是直接（非代理、非回送）的私人、ULA、
  鏈路本機或 CGNAT 位址，或在設定 `sshVerify.cidrs` 時與其相符。
- 適用資格下限與受信任 CIDR 核准相同：僅限全新且無範圍的節點
  配對；升級、瀏覽器、Control UI 和 WebChat 一律顯示提示。

探測進行期間，節點用戶端會收到持續重試
（`wait_then_retry`）的指示，而非暫停等待手動核准；若探測
失敗，下一次嘗試會回退至一般提示流程。失敗目標
會進入短暫冷卻期（金鑰不符後 5 分鐘）。

獲核准的裝置會記錄 `approvedVia: "ssh-verified"`，且其首次宣告的
能力介面會在同一步驟中獲核准——金鑰比對已證明
節點是在操作者所擁有機器上，以操作者的帳戶執行，這與
手動核准能力所確認的主張相同。後續介面升級仍會
顯示提示。

強化或停用：

```json5
{
  gateway: {
    nodes: {
      pairing: {
        // 完全停用：
        sshVerify: false,
        // ...或限定／調整探測：
        // sshVerify: { user: "me", identity: "~/.ssh/probe", timeoutMs: 7000, cidrs: ["10.0.0.0/8"] },
      },
    },
  },
}
```

## 自動核准（macOS app）

在下列情況下，macOS app 可嘗試對節點能力請求進行**靜默核准**：

- 請求標記為 `silent`（以非互動方式核准裝置配對時，閘道會將第一個能力
  介面標記為靜默），且
- app 可使用同一
  使用者驗證與閘道主機的 SSH 連線。

若靜默核准失敗，則會回退至一般 Approve/Reject 提示。

## 受信任 CIDR 裝置自動核准

`role: node` 的 WS 裝置配對預設維持手動。對於閘道已信任其網路路徑的私人節點
網路，操作者可使用明確的 CIDR 或確切 IP 選擇啟用：

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

安全邊界：

- 未設定 `gateway.nodes.pairing.autoApproveCidrs` 時停用。
- 不存在全面性的區域網路或私人網路自動核准模式；上述 SSH 驗證的
  自動核准要求密碼學裝置金鑰比對，絕不
  僅依賴網路位置。
- 只有未要求任何範圍的全新 `role: node` 裝置配對請求
  符合資格。
- 操作者、瀏覽器、Control UI 和 WebChat 用戶端仍須手動核准。
- 角色、範圍、中繼資料與公開金鑰升級仍須手動核准。
- 同一主機的回送受信任代理標頭路徑不符合資格，因為
  本機呼叫者可偽造該路徑。

## 靜默配對取代清理

非互動式核准會在已配對裝置資料列中記錄其來源：
同一主機本機政策核准記為 `silent`，受信任 CIDR 節點核准記為
`trusted-cidr`，SSH 驗證的節點核准記為 `ssh-verified`。狀態目錄為暫時性的用戶端（暫存家目錄、
容器、每次執行各自的沙箱）會在每次執行時產生新的裝置金鑰組，且每次
執行都會以全新裝置身分靜默重新配對——若不清理，已配對清單
每次執行都會增加一筆過時資料列。

當閘道靜默核准**本機**裝置配對時，會淘汰
屬於相同用戶端叢集（`clientId`、`clientMode` 和顯示名稱均相符）、
目前未連線且較舊的 `silent` 核准記錄。本機用戶端是在閘道主機本身執行，
因此叢集金鑰不可能與其他機器相符。遭淘汰的資料列會立即失去其權杖；
任何相符的舊版節點配對項目都會被清除，並廣播 `node.pair.resolved`
移除事件。

邊界：

- 只有最新核准為同主機本機（`silent`）的記錄，才有資格作為觸發來源與目標。受信任 CIDR 與 SSH 驗證的配對會跨越主機，而顯示中繼資料並非機器身分，因此絕不會自動移除——請針對這些記錄使用控制介面清理功能或 `openclaw nodes remove`。
- 擁有者核准及 QR／設定碼（啟動）配對絕不會自動移除。在來源資訊機制存在之前核准的記錄會持續受到保護，即使之後對相同裝置 ID 進行無提示重新核准也是如此。
- 目前已連線的裝置會略過，因此使用不同狀態目錄的並行本機工作階段，在連線期間會保留其權杖。最近一分鐘內核准的記錄也會略過，因此同時進行的配對交握不會在連線完成登錄前互相汰除。
- 受影響的用戶端在設計上都是本機用戶端，因此會在下次連線時無提示地重新配對。

## 中繼資料升級自動核准

當已配對的裝置重新連線，且僅有非敏感中繼資料變更（例如顯示名稱或用戶端平台提示）時，OpenClaw 會將其視為 `metadata-upgrade`。無提示自動核准的適用範圍很窄：只適用於受信任、非瀏覽器的本機重新連線，且該連線先前已證明持有本機或共用認證資訊，包括作業系統版本中繼資料變更後，同主機原生應用程式的重新連線。瀏覽器／控制介面用戶端與遠端用戶端仍使用明確的重新核准流程。範圍升級（從讀取升級為寫入／管理員）和公開金鑰變更**不**符合中繼資料升級自動核准的資格；它們仍會作為明確的重新核准要求處理。

## QR 配對輔助工具

`/pair qr` 會將配對酬載呈現為結構化媒體，讓行動裝置與瀏覽器用戶端可直接掃描。

刪除裝置時，也會清除該裝置 ID 的所有過時待處理配對要求，因此撤銷後，`nodes pending` 不會顯示孤立的資料列。

## 本機性與轉送標頭

只有在原始通訊端與任何上游 Proxy 證據皆一致時，閘道配對才會將連線視為回送連線。如果要求透過回送介面抵達，但帶有 `Forwarded`、任何 `X-Forwarded-*` 或 `X-Real-IP` 標頭證據，該轉送標頭證據便會使回送本機性宣告失效，而且配對路徑會要求明確核准，而不會無提示地將該要求視為同主機連線。關於操作者驗證的對應規則，請參閱
[受信任 Proxy 驗證](/zh-TW/gateway/trusted-proxy-auth)。

## 儲存空間（本機、私人）

配對狀態位於已配對裝置記錄中，儲存在閘道狀態目錄下的共用 SQLite 狀態資料庫（預設為 `~/.openclaw`）：

- `~/.openclaw/state/openclaw.sqlite`（包含裝置驗證的已配對裝置、已核准的節點介面、待處理介面要求、待處理裝置配對要求及啟動權杖）

如果覆寫 `OPENCLAW_STATE_DIR`，資料庫也會隨之移動。從使用 JSON 儲存區的版本升級而來的閘道，會在啟動時匯入這些資料，並保留 `devices/*.json.migrated` 與 `nodes/*.json.migrated` 封存檔。

安全性注意事項：

- 裝置權杖是機密資料；請將狀態資料庫視為敏感資料。
- 輪替裝置權杖會使用 `openclaw devices rotate`／
  `device.token.rotate`。

## 傳輸行為

- 傳輸層是**無狀態的**；它不會儲存成員資格。
- 如果閘道離線或已停用配對，節點就無法配對。
- 在遠端模式下，配對會針對遠端閘道的儲存區進行。
