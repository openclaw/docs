節點配對分為兩層，兩者都儲存在閘道 SQLite 狀態資料庫中的已配對裝置記錄：

- **裝置配對**（角色 `node`）會管控 `connect` 交握。請參閱下方的
  [受信任 CIDR 裝置自動核准](#trusted-cidr-device-auto-approval)
  以及[頻道配對](/zh-TW/channels/pairing)。
- **節點能力核准**（`node.pair.*`）會管控已連線節點可公開哪些宣告的
  能力／命令。閘道是事實來源；UI（macOS 應用程式、Control UI）則是
  核准或拒絕待處理要求的前端。

先前獨立的節點配對儲存區（`nodes/paired.json`，包含各節點的權杖，
已於 2026 年 1 月從連線路徑淘汰）現已移除：閘道會在啟動時一次性地將
任何剩餘資料列併入裝置記錄，並以 `.migrated` 後綴封存舊版檔案。
舊版 TCP 橋接支援已移除。

## 能力核准的運作方式

1. 節點連線至閘道 WS（此步驟由裝置配對管控）。
2. 閘道會比較宣告的能力／命令介面與已核准的介面；新的或擴大的介面會在
   裝置記錄中儲存一筆**待處理要求**，並發出 `node.pair.requested`。
3. 你核准或拒絕該要求（透過命令列介面或 UI）。
4. 核准前，節點命令會持續被篩除；核准後會公開宣告的介面，但仍受一般命令
   政策約束。

待處理要求會在**節點上次重試後 5 分鐘**自動到期——持續主動重新連線的
節點會讓其唯一的待處理要求保持有效，而不會在每次嘗試時產生新的要求
（及核准提示）。

## 命令列介面工作流程（適合無頭環境）

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` 會顯示已配對／已連線的節點及其能力。

## API 介面（閘道協定）

事件：

- `node.pair.requested` - 建立新的待處理要求時發出。
- `node.pair.resolved` - 要求獲核准、遭拒絕或到期時發出。

方法：

- `node.pair.list` - 列出待處理及已配對的節點（`operator.pairing`）。
- `node.pair.approve` - 核准待處理要求。
- `node.pair.reject` - 拒絕待處理要求。
- `node.pair.remove` - 移除已配對的節點。這會撤銷已配對裝置儲存區中該裝置的
  `node` 角色、連同移除已核准的節點介面，並使該裝置的節點角色工作階段
  失效／中斷連線。**混合角色**裝置（例如也持有 `operator` 的裝置）會保留
  其資料列，且只失去 `node` 角色；只有節點角色的裝置資料列則會被刪除。
  授權：`operator.pairing` 可移除非操作員節點資料列；使用裝置權杖的呼叫端
  若要撤銷混合角色裝置上其**自身**的節點角色，還需要 `operator.admin`。
- `node.rename` - 重新命名已配對節點面向操作員的顯示名稱。

已於 2026.7 移除：`node.pair.request` 和 `node.pair.verify`。待處理要求現在
由閘道在節點連線期間自行建立，而這些方法所服務的獨立各節點權杖已不復存在；
節點驗證現在使用裝置配對權杖。

注意事項：

- 以未變更的介面重新連線時，會重複使用待處理要求；重複要求會重新整理已儲存的
  節點中繼資料，以及供操作員查看的最新允許清單內宣告命令快照。
- 操作員範圍層級與核准時檢查彙整於
  [操作員範圍](/zh-TW/gateway/operator-scopes)。
- `node.pair.approve` 會使用待處理要求中宣告的命令，強制要求額外的核准範圍：
  - 無命令要求：`operator.pairing`
  - 非 exec 命令要求：`operator.pairing` + `operator.write`
  - `system.run` / `system.run.prepare` / `system.which` 要求：
    `operator.pairing` + `operator.admin`

<Warning>
節點配對核准會記錄受信任的能力介面。它**不會**針對各節點固定即時節點命令介面。

- 即時節點命令來自節點在連線時宣告的內容，並由閘道的全域節點命令政策
  （`gateway.nodes.allowCommands` 和 `denyCommands`）篩選。
- 各節點的 `system.run` 允許及詢問政策位於節點上的
  `exec.approvals.node.*`，而非配對記錄中。

</Warning>

## 節點命令管控（2026.3.31+）

<Warning>
**破壞性變更：**自 `2026.3.31` 起，節點命令會保持停用，直到節點配對獲得核准。僅有裝置配對已不足以公開宣告的節點命令。
</Warning>

節點首次連線時，系統會自動要求配對。
在該要求獲核准前，來自該節點的所有待處理節點命令都會被篩除，且不會執行。
配對獲核准後，節點宣告的命令即可使用，但仍受一般命令政策約束。

這表示：

- 先前僅依賴裝置配對來公開命令的節點，現在也必須完成節點配對。
- 在配對核准前排入佇列的命令會被捨棄，而非延後執行。

## 節點事件信任邊界（2026.3.31+）

<Warning>
**破壞性變更：**節點發起的執行現在會限制在縮減後的受信任介面內。
</Warning>

節點發起的摘要及相關工作階段事件，僅限於預期的受信任介面。先前依賴較廣泛主機或
工作階段工具存取權的通知驅動或節點觸發流程，可能需要調整。這項強化可避免節點事件
升級取得超出節點信任邊界所允許的主機層級工具存取權。

持久的節點存在狀態更新遵循相同的身分邊界：僅接受來自已驗證節點裝置工作階段的
`node.presence.alive` 事件，且只有裝置／節點身分已配對時才會更新配對中繼資料。
自行宣告的 `client.id` 值不足以寫入上次出現狀態。

## SSH 驗證的裝置自動核准（預設）

當首次從私有／CGNAT 位址進行 `role: node` 裝置配對時，如果閘道能透過 SSH
**證明機器擁有權**，便會自動核准：閘道會反向連線至提出配對的主機
（`BatchMode`、`StrictHostKeyChecking=yes`），在該處執行
`openclaw node identity --json`，且僅在遠端裝置 ID 與公開金鑰完全符合
待處理要求時才核准。金鑰比對是此機制安全的關鍵：單憑可連線性絕不會核准，
因此 NAT 共用租戶、共用主機上的其他使用者，以及區域網路偽裝都會轉入一般提示流程。

預設啟用。觸發條件：

- 閘道處理程序使用者（或 `sshVerify.user`）能以非互動方式 SSH 至節點主機
  （金鑰／代理程式；Tailscale SSH 也可），且主機金鑰已受信任。
- 在非互動式 `sh -lc` 中，遠端 `PATH` 可解析 `openclaw`。
- 連線 IP 是直接（非代理、非迴路）的私有、ULA、鏈路本機或 CGNAT 位址，
  或在設定時符合 `sshVerify.cidrs`。
- 適用資格下限與受信任 CIDR 核准相同：僅限全新且無範圍的節點配對；
  升級、瀏覽器、Control UI 和 WebChat 一律顯示提示。

探測執行期間，節點用戶端會收到持續重試（`wait_then_retry`）的指示，而不是暫停以等待
手動核准；若探測失敗，下次嘗試會轉入一般提示流程。失敗的目標會進入短暫冷卻期
（金鑰不符後 5 分鐘）。

已核准的裝置會記錄 `approvedVia: "ssh-verified"`，且其首次宣告的能力介面會在同一
步驟獲得核准——金鑰比對已證明節點是在操作員所擁有機器上的操作員帳號下執行，
這與手動能力核准所確認的主張相同。後續的介面升級仍會顯示提示。

強化或停用：

```json5
{
  gateway: {
    nodes: {
      pairing: {
        // 完全停用：
        sshVerify: false,
        // ...或限制／調整探測：
        // sshVerify: { user: "me", identity: "~/.ssh/probe", timeoutMs: 7000, cidrs: ["10.0.0.0/8"] },
      },
    },
  },
}
```

## 自動核准（macOS 應用程式）

符合以下條件時，macOS 應用程式可嘗試**靜默核准**節點能力要求：

- 要求標記為 `silent`（當裝置配對以非互動方式獲核准時，閘道會將首次能力
  介面標記為靜默），且
- 應用程式可使用相同使用者驗證與閘道主機的 SSH 連線。

若靜默核准失敗，系統會轉回一般 Approve/Reject 提示。

## 受信任 CIDR 裝置自動核准

`role: node` 的 WS 裝置配對預設仍需手動進行。對於閘道已信任網路路徑的
私有節點網路，操作員可透過明確的 CIDR 或確切 IP 選擇啟用：

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

安全邊界：

- 未設定 `gateway.nodes.pairing.autoApproveCidrs` 時停用。
- 不存在一律核准區域網路或私有網路的模式；SSH 驗證的自動核准（見上方）
  需要密碼學裝置金鑰比對，絕不僅依賴網路位置。
- 僅全新的 `role: node` 裝置配對要求，且未要求任何範圍時符合資格。
- 操作員、瀏覽器、Control UI 和 WebChat 用戶端仍需手動處理。
- 角色、範圍、中繼資料和公開金鑰升級仍需手動處理。
- 同主機迴路的受信任代理標頭路徑不符合資格，因為本機呼叫端可能偽造該路徑。

## 靜默配對取代清理

非互動式核准會在已配對裝置資料列上記錄其來源：同主機本機政策核准記為
`silent`，受信任 CIDR 節點核准記為 `trusted-cidr`，SSH 驗證的節點核准記為
`ssh-verified`。狀態目錄為暫時性（暫存主目錄、容器、各次執行沙箱）的用戶端，
每次執行都會產生新的裝置金鑰組，並且每次都會以全新裝置的身分靜默重新配對——
若不清理，已配對清單會在每次執行後增加一筆過時資料列。

當閘道靜默核准**本機**裝置配對時，它會淘汰屬於相同用戶端叢集
（`clientId`、`clientMode` 和顯示名稱皆相符）且目前未連線的較舊
`silent` 核准記錄。本機用戶端在閘道主機本身執行，因此叢集鍵不會與其他機器相符。
被淘汰的資料列會立即失去其權杖；任何相符的舊版節點配對項目會被清除，並廣播一個
`node.pair.resolved` 移除事件。

邊界：

- 僅最新核准為同主機本機核准（`silent`）的記錄符合資格，無論作為觸發者或目標。
  受信任 CIDR 和 SSH 驗證的配對會跨越主機，而顯示中繼資料並非機器身分，因此
  絕不會自動移除——請使用 Control UI 清理功能或
  `openclaw nodes remove` 來移除。
- 擁有者核准及 QR／設定碼（啟動程序）配對絕不會自動移除。在來源記錄功能存在前
  核准的記錄仍受保護，即使之後同一裝置 ID 再次獲得靜默核准亦然。
- 目前已連線的裝置會被略過，因此使用不同狀態目錄的並行本機工作階段在連線期間
  仍會保有其權杖。最近一分鐘內核准的記錄也會被略過，因此同時進行的配對交握
  不會在彼此的連線完成登記前互相淘汰。
- 受影響的用戶端依設計必定是本機用戶端，因此會在下次連線時靜默重新配對。

## 中繼資料升級自動核准

當已配對裝置重新連線，且只有非敏感中繼資料變更（例如顯示名稱或用戶端平台提示）時，
OpenClaw 會將其視為 `metadata-upgrade`。靜默自動核准的適用範圍很窄：僅適用於
已證明持有本機或共用認證資訊的受信任非瀏覽器本機重新連線，包括 OS 版本中繼資料
變更後的同主機原生應用程式重新連線。瀏覽器／Control UI 用戶端和遠端用戶端仍使用
明確的重新核准流程。範圍升級（從讀取提升至寫入／管理員）和公開金鑰變更**不**
符合中繼資料升級自動核准資格；它們仍屬於明確的重新核准要求。

## QR 配對輔助工具

`/pair qr` 會將配對承載資料轉譯為結構化媒體，讓行動裝置和
瀏覽器用戶端可直接掃描。

刪除裝置時，也會清除該裝置 ID 任何過期的待處理配對請求，因此撤銷後，
`nodes pending` 不會顯示孤立的資料列。

## 區域性與轉送標頭

只有當原始通訊端與任何上游 Proxy 證據一致時，閘道配對才會將連線視為
迴路連線。如果請求從迴路介面抵達，但帶有 `Forwarded`、任何
`X-Forwarded-*` 或 `X-Real-IP` 標頭證據，該轉送標頭證據會使迴路區域性
宣告失效，配對路徑將要求明確核准，而不會無提示地將請求視為同一主機的
連線。操作員驗證的對等規則請參閱
[受信任的 Proxy 驗證](/zh-TW/gateway/trusted-proxy-auth)。

## 儲存空間（本機、私密）

配對狀態儲存在共用 SQLite 狀態資料庫的已配對裝置記錄中；該資料庫位於
閘道狀態目錄下（預設為 `~/.openclaw`）：

- `~/.openclaw/state/openclaw.sqlite`（已配對的裝置及其裝置驗證、已核准的
  節點介面、待處理的介面請求、待處理的裝置配對請求，以及啟動權杖）

如果覆寫 `OPENCLAW_STATE_DIR`，資料庫也會隨之移動。從使用 JSON 儲存區的
版本升級而來的閘道，會在啟動時匯入這些資料，並留下
`devices/*.json.migrated` 和 `nodes/*.json.migrated` 封存檔。

安全注意事項：

- 裝置權杖是機密資訊；請將狀態資料庫視為敏感資料。
- 輪替裝置權杖時，請使用 `openclaw devices rotate` /
  `device.token.rotate`。

## 傳輸行為

- 傳輸層是**無狀態的**；不會儲存成員資格。
- 如果閘道離線或已停用配對，節點便無法配對。
- 在遠端模式下，配對會使用遠端閘道的儲存區。
