重新啟動閘道不會遺失代理程式狀態。對話、逐字記錄、排程工作、背景任務記錄與排隊中的外送訊息全都儲存在磁碟上，而在回合進行途中遭中斷的工作，會在閘道重新啟動後自動被偵測並繼續執行。不需要人工介入，也沒有任何項目需要設定：復原功能永遠開啟。

本頁說明哪些內容會在重新啟動後保留、如何偵測中斷的工作，以及自動繼續執行的運作方式。

## 重新啟動後會保留的內容

| 狀態                          | 儲存位置                                    | 重新啟動時的行為                                                        |
| ----------------------------- | ------------------------------------------- | ----------------------------------------------------------------------- |
| 對話歷史記錄                  | 每個代理程式各自的 SQLite 資料庫            | 不受影響；工作階段會從儲存的逐字記錄繼續                               |
| 中斷的主要工作階段回合        | 每個代理程式各自的 SQLite 工作階段資料列與逐字記錄 | 啟動數秒後自動繼續執行或協調                                           |
| 子代理程式執行                | SQLite（共用狀態資料庫）                    | 開機時還原登錄檔；繼續執行中斷的作業                                   |
| 背景任務                      | SQLite（共用狀態資料庫）                    | 開機時協調；復原孤立的作業或將其標記為遺失                             |
| 排隊中的外送項目              | SQLite 傳送佇列                             | 重新啟動後清空佇列；重試尚未傳送的回覆                                 |
| 已排程的（排程）工作          | SQLite 排程儲存區                           | 排程會保留；排程器在開機時重新啟用                                     |
| 重新啟動後接續執行            | SQLite 重新啟動哨兵                         | 將一次性後續作業分派至要求重新啟動的工作階段                           |

## 正常重新啟動會先等待作業結束

要求的重新啟動（`openclaw gateway restart`、需要重新啟動的設定變更，或閘道更新）不會立即終止執行中的工作。閘道會停止接受新工作，接著等待進行中的代理程式回合與背景任務完成，最長等待至排空時間上限（預設為 5 分鐘）。因此，大多數重新啟動完全不會中斷任何工作。

只有無法在排空時間上限內完成的工作（或任何因強制重新啟動或當機而中斷的作業）才會中止，而在此之前，每個受影響的工作階段都會標記為待復原。

## 如何偵測中斷的工作

有三種互補機制會標記回合尚未完成的工作階段：

- **接受回合時：**對現有主要工作階段中的一般文字回合，閘道會附加使用者訊息、將工作階段標記為執行中，並在模型或 `before_agent_reply` 掛鉤執行前，於同一個 SQLite 交易中記錄其復原傳送宣告。Control UI 會在傳回 `started` 確認前執行此操作；通道分派則會在備妥的回合採用代理程式作業時執行。
  命令、附件、單回合覆寫、待處理傳送、先前的中止提示、由外掛擁有的工作階段，以及具有執行掛鉤的回合，會保留其專用的接受路徑。
  如果已安裝 `before_agent_reply` 掛鉤，接受程序也會記錄其階段。復原絕不會重播在呼叫途中中斷的掛鉤。未處理的掛鉤完成後，其檢查點會記錄該結果，但只要該掛鉤仍處於作用中，復原仍會採取失敗關閉：檢查點無法證明重新啟動後載入的是相同的外掛程式碼與設定。已處理文字與靜默結果會分別建立檢查點，以確保確定性結算。
  舊版本寫入的持久復原宣告沒有來源擁有權標記，因此升級期間會接受相同的失敗關閉掛鉤檢查。
- **關機時：**在重新啟動的排空期間，每個具有進行中作業的工作階段都會先在工作階段儲存區中加上復原標記，之後才中止作業。
- **啟動時：**閘道會掃描工作階段儲存區，尋找仍宣告為執行中、但在新處理程序中沒有存活擁有者的工作階段。這可涵蓋未曾執行任何關機程式碼的嚴重當機與強制終止。同時也會清除過期的逐字記錄鎖定檔。

## 自動繼續執行

啟動數秒後，閘道會向每個已標記的工作階段重新分派一則合成系統訊息，告知代理程式上一個回合因重新啟動而中斷，並要求從現有逐字記錄繼續執行。如果最終回覆已產生但尚未傳送，系統會納入其文字，讓代理程式直接傳送，而不是重新執行工作。復原最多會重試 3 次，並採用指數退避。每次重試都會重複使用同一個持久分派識別碼，因此語意不明的連線失敗不會重複啟動相同的復原。已完成及無法繼續執行的 Control UI 回合也會保留有界限的持久等冪墓碑，讓重新連線的寄件匣無須重新執行要求即可將其移除。

僅使用訊息工具的回覆會採用第二組持久關聯。在相同對話的終止傳送到達通道前，閘道會在確切的工作階段與來源回合上，記錄尚未解析的傳送意圖。確認提供者成功後，會將其解析為持久的已傳送收據；確認失敗則會將其清除。復原可完成已傳送收據，而無須重新執行工具。如果當機導致提供者的結果不明，復原會採取失敗關閉，而不會重播外部效果。

已傳送的回覆也會連同其來源訊息 ID 鏡像寫入逐字記錄。終止鏡像會使用不同的收據金鑰，因此使用相同提供者等冪金鑰的進度傳送不會遮蔽終止標記。來自舊回合的進度傳送與收據無法完成目前回合。只有持久的通道輸入宣告能還原訊息動作權限。繼續執行的作業會保留原始來源傳送模式與來源關聯，包括要求者身分及任何相同通道／討論串限制，因此即使復原期間再次重新啟動，同一張收據仍具權威性。沒有可重建通道權限且僅使用訊息工具的回合會採取失敗關閉，並收到一次性的重新傳送通知。

繼續執行前，閘道會檢查逐字記錄末端是否可安全接續。如果不安全（例如回合結束於過期的待處理核准），就不會盲目重新執行工作階段；代理程式會改為發出簡短通知，要求使用者重新傳送上一個要求。若為 WebChat，該通知會直接寫入工作階段歷史記錄，因此重新連線後仍然可見。

OpenClaw 也可以重建中斷的唯讀 [Code Mode](/zh-TW/reference/code-mode) 工作。Code Mode 會將這些作業標記為可安全重新啟動，並在具有副作用的目錄工具或外掛命名空間執行前加以拒絕。如果重新啟動發生在 `wait` 控制項上，新閘道會根據逐字記錄重建回合，並強制重建的執行維持可安全重新啟動，即使模型省略或清除該旗標亦然。即使重新啟動後停用 Code Mode，主機仍會將整個重建回合限制為經過稽核的唯讀核心工具與明確可安全重播的外掛工具。具有副作用的工作仍會由重新傳送通知保護，以免產生重複寫入的風險。

### 子代理程式

子代理程式作業會持久儲存在共用 SQLite 狀態資料庫中，因此子代理程式登錄檔可跨處理程序保留。開機時會還原登錄檔，並使用原始任務內容繼續執行中斷的子代理程式工作階段。以下有兩項安全閥：

- 超過 2 小時前中斷的作業會直接結束，而不會繼續執行，因此關閉一整晚的閘道不會復活過期工作。
- 重複復原失敗的工作階段會被標記為卡死墓碑，避免復原無限循環。

### 背景任務

[背景任務登錄檔](/zh-TW/automation/tasks) 以 SQLite 為後端，會在開機時與定期區間內進行協調：已完成作業所記錄的持久結果會被復原，而擁有它們的處理程序已消失的作業，會在寬限期後標記為遺失，而不是永久停滯。

### 代理程式要求的重新啟動

當代理程式本身觸發重新啟動（套用設定變更、更新閘道，或明確要求重新啟動）時，處理程序會先將重新啟動哨兵寫入 SQLite，然後才結束。開機後，閘道會將結果回傳至原始聊天，並分派一次性的接續回合，讓代理程式在相同通道與討論串中，從先前停止的位置精確接續。

## 安全閥與可觀測性

- **當機循環斷路器：**在 5 分鐘內發生 3 次未正常關閉的開機，會觸發斷路器，在下一次開機時抑制自動啟動的輔助服務，避免持續當機的閘道自我放大。未正常關閉的開機時間窗排空後，便會恢復正常。
- **指標：**復原活動會透過 [Prometheus](/zh-TW/gateway/prometheus) 匯出為 `openclaw_session_recovery_total` 與 `openclaw_session_recovery_age_seconds`。
- **記錄：**復原決策會記錄在 `main-session-restart-recovery` 與 `subagent-interrupted-resume` 子系統下。

## 不會繼續執行的內容

- 因已有其他擁有者負責處理，而排除於主要工作階段復原之外的工作階段：子代理程式工作階段（子代理程式復原）、排程工作階段（排程器會依排程重新執行），以及由 ACP 管理的工作階段（由已連線的 IDE 或用戶端負責繼續執行）。
- 逐字記錄末端無法安全接續的工作階段；這些工作階段會收到上述重新傳送通知，而不是在沒有提示的情況下重新執行。
- 從未被接受的工作：排空時間窗期間到達的訊息會遭拒絕並收到明確的重新啟動錯誤，而不是被悄悄排入即將終止的處理程序。
