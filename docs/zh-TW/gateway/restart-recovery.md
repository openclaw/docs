重新啟動閘道不會遺失代理程式狀態。對話、逐字記錄、
排程工作、背景任務記錄和已排入佇列的外送訊息全都儲存在
磁碟上，而在回合途中遭中斷的工作會在閘道恢復運作後
自動偵測並繼續。復原功能一律啟用，而且通常不需要手動介入。
持續復原失敗時會有次數限制，並可能隔離一個工作階段，直到你檢查或取代它。

本頁說明重新啟動後會保留哪些內容、如何偵測遭中斷的工作，
以及自動繼續的運作方式。

## 重新啟動後會保留的內容

| 狀態                          | 儲存位置                                    | 重新啟動時的行為                                                        |
| ----------------------------- | ------------------------------------------- | ----------------------------------------------------------------------- |
| 對話記錄                      | 每個代理程式的 SQLite 資料庫                | 保持不變；工作階段從已儲存的逐字記錄繼續                               |
| 遭中斷的主要工作階段回合      | 每個代理程式的 SQLite 工作階段資料列和逐字記錄 | 啟動數秒後自動繼續或協調                                                 |
| 子代理程式執行                | SQLite（共用狀態資料庫）                    | 啟動時還原登錄資料；繼續遭中斷的執行                                   |
| 背景任務                      | SQLite（共用狀態資料庫）                    | 啟動時協調；復原無主執行或將其標示為遺失                               |
| 已排入佇列的外送項目          | SQLite 傳送佇列                             | 重新啟動後清空佇列；重試未傳送的回覆                                   |
| 已排程（排程）工作            | SQLite 排程儲存區                           | 排程持續保留；排程器在啟動時重新設置                                   |
| 重新啟動後續處理              | SQLite 重新啟動哨兵                         | 對要求重新啟動的工作階段分派一次性後續處理                             |

## 正常重新啟動會先排空工作

要求重新啟動（`openclaw gateway restart`、需要重新啟動的設定變更，
或閘道更新）時，不會立即終止進行中的工作。閘道會停止接受新工作，
然後等待進行中的代理程式回合和背景任務完成，最長至排空時間額度
（預設為 5 分鐘）。因此，大多數重新啟動完全不會中斷任何工作。

只有無法在排空時間額度內完成的工作（或因強制重新啟動或當機而
中斷的任何執行）才會中止，而在此之前，每個受影響的工作階段
都會標記為待復原。

## 如何偵測遭中斷的工作

三種互補機制會標記回合未完成的工作階段：

- **接受回合時：**對現有主要工作階段中的一般文字回合，
  閘道會附加使用者訊息、將工作階段標記為執行中，並在模型或
  `before_agent_reply` 掛鉤執行前，於同一個 SQLite 交易中記錄其
  復原傳送宣告。Control UI 會在傳回 `started` 確認前完成此操作；
  頻道分派則會在準備好的回合採用代理程式執行時完成此操作。
  命令、附件、每回合覆寫、待處理傳送、先前的中止提示、
  外掛擁有的工作階段，以及具有執行掛鉤的回合，會保留其
  專用接受路徑。
  若已安裝 `before_agent_reply` 掛鉤，接受時也會記錄其階段。
  復原絕不會重播在呼叫途中遭中斷的掛鉤。未處理的掛鉤完成後，
  其檢查點會記錄該結果，但只要該掛鉤仍處於作用中，復原仍會
  採取失敗關閉：檢查點無法證明重新啟動後載入的是相同的
  外掛程式碼和設定。已處理的文字與靜默結果會分別建立檢查點，
  以確保確定性的結算。
  舊版寫入的持久復原宣告沒有來源擁有權標記，因此升級期間
  也會接受相同的失敗關閉掛鉤檢查。
- **關閉時：**在重新啟動的排空期間，每個有進行中執行的
  工作階段都會在執行中止前，於工作階段儲存區中加上復原標記。
- **啟動時：**閘道會掃描工作階段儲存區，尋找仍宣告為執行中、
  但在新程序中沒有作用中擁有者的工作階段。這能捕捉未執行任何
  關閉程式碼的嚴重當機和程序終止。同時也會清除過期的逐字記錄
  鎖定檔案。

## 自動繼續

啟動數秒後，閘道會重新分派每個已標記的工作階段，並附上一則
合成系統訊息，告知代理程式先前的回合因重新啟動而中斷，且應從
現有逐字記錄繼續。若最終回覆已產生但尚未傳送，訊息中會包含其
文字，讓代理程式可以直接傳送，而不必重做工作。

啟動協調會以指數退避方式重試暫時性失敗，最多三次。另外，每個
遭中斷的主要工作階段週期具有三次計入額度的自動分派嘗試持久
額度，且此額度會跨閘道重新啟動保留。OpenClaw 會在分派前計入
一次嘗試；若閘道在接受前明確拒絕要求，則退還該次嘗試；若分派後
結果不確定，則保留計次，以避免重播工作。已擁有工作階段的前景
工作會阻止自動復原進入，直到該工作完成。

持久額度用盡後，系統會為工作階段建立墓碑，而不是無限循環。
請檢查失敗的工作階段，並使用 `/new` 或 `/reset` 啟動
替代工作階段。`openclaw doctor --fix` 可以修復與墓碑衝突的過期中止旗標，
但不會重新啟用該復原週期。

每次重試都會重複使用同一個持久分派識別碼，因此不明確的連線
失敗無法重複啟動相同的復原。已完成及無法繼續的 Control UI 回合
也會保留有界的持久冪等性墓碑，讓重新連線的寄件匣可以淘汰它們，
而不會重新執行要求。

僅使用訊息工具的回覆會使用第二組持久關聯。在同一對話中的終止
傳送抵達頻道前，閘道會在確切的工作階段和來源回合上記錄未解決的
傳送意圖。確認提供者成功後，會將其解析為持久的已傳送收據；
確認失敗則會將其清除。復原會完成已有已傳送收據的工作，而不會
重新執行工具。若當機導致提供者結果未知，復原會採取失敗關閉，
而不會重播外部效果。

已傳送的回覆也會連同其來源訊息 ID 鏡像寫入逐字記錄。終止鏡像
使用不同的收據索引鍵，因此使用相同提供者冪等性索引鍵的進度
傳送無法遮蔽終止標記。先前回合的進度傳送和收據無法完成目前
回合。只有持久頻道進站宣告能還原訊息動作授權。繼續的執行會
保留原始來源傳送模式和來源關聯，包括要求者身分，以及任何相同
頻道／討論串限制，因此即使復原期間再次重新啟動，相同收據仍
具有權威性。若僅使用訊息工具的回合無法重建頻道授權，系統會
採取失敗關閉，並收到一次性的重新傳送通知。

繼續前，閘道會檢查逐字記錄尾端是否能安全接續。若不安全
（例如回合結束於過期的待處理核准），系統不會盲目重新執行
工作階段；代理程式會改為發布簡短通知，請使用者重新傳送最後
一個要求。對於 WebChat，該通知會直接寫入工作階段記錄，使其在
重新連線後仍然可見。

OpenClaw 也能重建遭中斷的唯讀 [Code Mode](/tools/code-mode)
工作。Code Mode 會將這些執行標記為重新啟動安全，並在具有副作用的
目錄工具或外掛命名空間執行前拒絕它們。若重新啟動發生在
`wait` 控制項上，新閘道會根據其逐字記錄重建回合，並強制
重建後的執行保持重新啟動安全，即使模型省略或清除該旗標亦然。
主機會將整個重建回合篩選為經稽核的唯讀核心工具和明確可安全
重播的外掛工具，包括重新啟動後停用 Code Mode 的情況。具有
副作用的工作仍由重新傳送通知保護，避免冒險產生重複寫入。

### 子代理程式

子代理程式執行會持久儲存在共用 SQLite 狀態資料庫中，因此
子代理程式登錄資料能跨程序保留。啟動時會還原登錄資料，並使用
原始任務內容繼續遭中斷的子代理程式工作階段。以下兩項安全機制
適用：

- 超過 2 小時前遭中斷的執行會完成結案，而不會繼續，因此
  停機一整夜的閘道不會讓過期工作復活。
- 持續復原失敗的工作階段會建立墓碑並標示為卡死，防止
  復原無限循環。

### 背景任務

[背景任務登錄資料](/zh-TW/automation/tasks)以 SQLite 為後端，並會在啟動時
及定期進行協調：系統會復原已完成執行所記錄的持久結果，而擁有
程序已消失的執行則會在寬限期後標示為遺失，而不是永遠停滯。

### 代理程式要求的重新啟動

當代理程式本身觸發重新啟動（套用設定變更、更新閘道，或明確要求
重新啟動）時，程序結束前會將重新啟動哨兵寫入 SQLite。啟動後，
閘道會將結果發布回原始聊天，並分派一次性的後續回合，讓代理程式
在相同頻道和討論串上，準確地從中斷處接續。

哨兵的具型別 SQLite 欄位是重新啟動處理的權威來源；
其 `payload_json` 值僅供重播／偵錯使用。執行階段會直接讀取、
寫入及清除 SQLite 狀態，不使用檔案備援。在儲存系統切換期間，
系統會在啟動時及透過 Doctor 執行有界的狀態移轉，以保留更新後
舊程序留下並已驗證的 `restart-sentinel.json`。
移轉程序會驗證具型別資料列並移除來源檔案，之後才繼續正常的
重新啟動處理。

## 安全機制與可觀測性

- **當機循環斷路器：**在 5 分鐘內發生 3 次非正常啟動時會觸發
  斷路器，在下一次啟動時抑制自動啟動的輔助服務，避免當機中的
  閘道擴大問題。非正常啟動時間窗結束後即會恢復。
- **主要工作階段嘗試額度：**每個遭中斷週期有三次計入額度的
  自動分派嘗試；額度耗盡後會為該工作階段建立墓碑，直到它被
  檢查並取代。
- **指標：**復原活動會透過
  [Prometheus](/zh-TW/gateway/prometheus) 匯出為 `openclaw_session_recovery_total` 和
  `openclaw_session_recovery_age_seconds`。
- **記錄：**復原決策會記錄在
  `main-session-restart-recovery` 和 `subagent-interrupted-resume`
  子系統下。

## 不會繼續的內容

- 由於已有其他擁有者處理，因此排除在主要工作階段復原之外的
  工作階段：子代理程式工作階段（子代理程式復原）、排程工作階段
  （排程器會依排程重新執行），以及由 ACP 管理的工作階段
  （由已連線的 IDE 或用戶端負責繼續）。
- 逐字記錄尾端無法安全接續的工作階段；這些工作階段會收到
  上述重新傳送通知，而不會在無提示的情況下重新執行。
- 從未獲准接受的工作：在排空時間窗內抵達的訊息會遭到拒絕，
  並傳回明確的重新啟動錯誤，而不是在無提示的情況下排入即將
  結束之程序的佇列。
- 獨立的嵌入式回合無法接管有待處理重新啟動復原的主要
  工作階段，因為兩者不共用閘道的生命週期擁有者。請透過閘道
  執行該回合，或在閘道中使用 `/new` 或 `/reset` 重設它。
