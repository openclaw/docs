`show_widget` 是一項核心工具，可在使用者目前的介面上顯示獨立完整的 HTML 小工具。OpenClaw 會在控制介面、iOS、Android 與 macOS 的聊天記錄中直接內嵌呈現；Linux 則使用瀏覽器控制介面。在已啟用 [Activities](/zh-TW/channels/discord-activities) 的 Discord 工作階段中，Discord 外掛會發布一個 **Open widget** 按鈕，以 Activity 形式啟動小工具。

## 小工具的運作方式

代理程式呼叫 `show_widget` 時，OpenClaw 核心會將 `widget_code` 包裝在最精簡的 HTML 文件中、儲存為 Canvas 文件，並傳回預覽控制代碼。控制介面會在工具呼叫正下方，以沙箱化 iframe 呈現該控制代碼；原生應用程式則使用隔離的網頁檢視。兩者都會在重新載入歷程記錄後還原小工具。

在控制介面工作階段中，也可將 Canvas 小工具釘選至工作階段儀表板。請在工具呼叫中設定 `pin: true`，或對現有的聊天記錄小工具使用 **釘選至儀表板**。釘選會重複使用完全相同的託管文件，不會透過瀏覽器擷取小工具 HTML。

針對瀏覽器內嵌，包裝文件會在小工具程式碼周圍注入四個小型主機橋接器：

- 大小回報器會將呈現內容的高度傳送至內嵌該小工具的聊天介面，由聊天介面限制其範圍並調整 iframe（160 至 1200 像素）。
- 提示橋接器會定義全域 `sendPrompt(text)` 函式，供小工具指令碼呼叫，以將後續訊息提交至聊天。在任何小工具程式碼執行前，橋接器會建立私密訊息通道，並向聊天介面提供一個端點；聊天介面只會採用第一次提供的端點。請參閱[互動式小工具](#interactive-widgets)。
- 主題橋接器會監聽控制介面目前的設計權杖，並在載入時及每次主題變更時將其套用為 CSS 變數。
- 快照橋接器會在內嵌該小工具的聊天介面要求匯出時，將目前的小工具文件呈現為 PNG。

其餘所有內容都會留在框架內：文件會在不透明來源中執行，並採用嚴格的內容安全政策，因此小工具指令碼無法存取控制介面、閘道或網路。

只有在發起要求的閘道用戶端宣告 `inline-widgets` 能力時，才能使用核心實作。控制介面與支援的原生應用程式會自動宣告此能力。只有已設定 Activities 的 Discord 工作階段才能使用 Discord 實作。其他頻道的執行程序不會收到 `show_widget`。

能力傳輸涵蓋內嵌式、Codex app-server 與命令列介面後端的模型後端。經授權認證的 MCP 呼叫端與直接 HTTP 工具叫用端仍會採取失敗時關閉，因為它們不會宣告用戶端能力。

## 設計系統

每個 Canvas 小工具都包含一份無類別基礎樣式表與一組精簡的權杖：

| 權杖                                                                                  | 用途                                  |
| ------------------------------------------------------------------------------------- | ------------------------------------- |
| `--surface`                                                                    | 頁面層級的介面色彩                    |
| `--card`                                                                    | 卡片、按鈕與程式碼背景                |
| `--elevated`                                                                    | 浮升式表單控制項背景                  |
| `--text`                                                                    | 預設的本文與控制項文字                |
| `--text-strong`                                                                    | 標題與醒目數值                        |
| `--muted`                                                                    | 次要文字與細微邊框                    |
| `--border`                                                                    | 標準分隔線與卡片邊框                  |
| `--border-strong`                                                                    | 醒目的控制項邊框                      |
| `--accent`                                                                    | 連結與焦點環                          |
| `--accent-fill`                                                                    | 主要動作填色                          |
| `--accent-fg`                                                                    | 主要動作上的文字                      |
| `--ok`                                                                    | 成功狀態                              |
| `--warn`                                                                    | 警告狀態                              |
| `--danger`                                                                    | 錯誤或破壞性狀態                      |
| `--info`                                                                    | 資訊狀態                              |
| `--radius`                                                                    | 控制項與卡片共用的圓角半徑            |
| `--font-body`                                                                    | 主機內文的字型堆疊                    |
| `--font-mono`                                                                    | 主機等寬字型堆疊                      |
| `--accent-subtle`、`--ok-subtle`、`--warn-subtle`、`--danger-subtle`、`--info-subtle` | 衍生的半透明狀態背景                  |

未套用類別的標題、段落、連結、按鈕、輸入欄位、選取欄位、文字區域、表格與程式碼區塊都會獲得基礎樣式。輔助類別提供常用模式：

- `.card` 用於有邊框的內容介面
- `.badge` 搭配 `.ok`、`.warn`、`.danger` 或 `.info`，用於精簡狀態標籤
- `.metric` 用於醒目的數值
- `.muted` 用於次要文字
- `.row` 用於可換行的水平版面配置
- `button.primary` 用於主要動作

控制介面會在小工具載入及每次主題變更時，傳送包含作用中主題值的 `openclaw:widget-theme` 訊息。因此，小工具無須重新載入，即可追蹤每個主題系列，包括 Claw、Knot、Dash 與自訂主題。在控制介面以外的環境（包括原生應用程式與直接開啟）中，小工具會使用由 `prefers-color-scheme` 選定的內建淺色或深色調色盤。

製作小工具時請遵守三項規則：

1. 所有色彩與背景都使用設計變數。請勿寫死色彩值。
2. 保持頁面背景透明，讓小工具融入其主機介面。
3. `--accent-fill` 最多只能保留給一個主要動作。

**匯出：** 在網頁聊天中，開啟小工具卡片選單，即可將呈現後的小工具複製到剪貼簿，或下載為 PNG。沒有快照橋接器的舊版小工具文件會改以下載 HTML 檔案作為備援。

## 使用工具

兩種實作使用相同的必填欄位：

<ParamField path="title" type="string" required>
  顯示於內嵌預覽及託管文件標題中的簡短標題。
</ParamField>

<ParamField path="widget_code" type="string" required>
  獨立完整的 HTML 或 SVG。對於內嵌小工具用戶端，修剪後以 `<svg` 開頭的輸入會以 SVG 模式呈現；長度上限為 262,144 個字元。Discord 接受最大 48 KiB 的完整 HTML 文件或 body 片段。
</ParamField>

Discord 也接受選用的 `button_label` 文字，作為 Activity 啟動按鈕。Canvas 結構描述刻意省略這個僅限 Discord 的欄位。

核心 Canvas 工具接受下列選用的儀表板放置欄位：

- `pin`：同時將小工具放置在工作階段儀表板上。
- `name`：穩定的小工具名稱；預設為 `title` 的 slug。
- `tab`：目的地分頁 slug。
- `size`：`sm`、`md`、`lg`、`xl` 或 `full` 其中之一。
- `after`：要將小工具放置於其後的同層小工具名稱。

核心結果包含 Canvas 預覽控制代碼，因此控制介面與支援的原生應用程式會直接從工具呼叫呈現小工具，並在重新載入歷程記錄後還原。釘選結果也會保留看板小工具名稱，因此控制介面不會在重新載入聊天記錄後再次提供重複的釘選選項。Discord 會傳回已儲存的小工具與已發布訊息的識別碼。

`discord_widget` 會以已棄用別名繼續註冊一個版本。新的代理程式呼叫應使用 `show_widget`。

## 互動式小工具

在控制介面中，小工具指令碼可以推進對話。包裝文件會定義全域 `sendPrompt(text)` 函式；呼叫該函式會將 `text` 提交至聊天，就像使用者自行輸入並傳送訊息一樣。可將它連接至按鈕或其他控制項，以建置選擇器、測驗或下鑽式儀表板等互動流程。原生應用程式會呈現互動式小工具程式碼，但不會公開此聊天提示橋接器。

```html
<button onclick="sendPrompt('詳細顯示失敗的測試')">失敗的測試</button>
```

每個提示都會在框架邊界的兩側進行驗證：

- `sendPrompt` 要求小工具內具有[暫時性使用者啟用狀態](https://developer.mozilla.org/en-US/docs/Web/Security/User_activation)：它只能在使用者於小工具中按一下或按鍵後的幾秒內運作，因此請將它連接至按鈕及其他點擊目標——在載入時自動呼叫不會有任何作用。橋接器會將傳送端點保持為自身私有，並在未公開使用者啟用狀態的瀏覽器中採取失敗時關閉，因此小工具程式碼無法略過此檢查。
- 提示權限只屬於原始小工具文件。可信任的橋接器會在小工具程式碼能夠執行或導覽框架前，向聊天介面提供其通道端點；聊天介面只會採用第一次提供的端點，且導覽時該通道會隨文件失效。外部允許的內嵌 URL 絕不會被採用。
- 小工具框架必須顯示於聊天記錄中並取得焦點——這是由主機觀察到的額外訊號，證明使用者確實正在與此小工具互動。
- 文字經修剪後不得為空，且最多為 4,000 個字元。
- 以 `/` 開頭的提示會遭到拒絕，因此小工具程式碼無法觸發 `/approve` 或 `/stop` 等聊天命令。
- 每份小工具文件在每個滾動分鐘內最多可傳送 10 個提示；超出的提示會被靜默捨棄。

接受的提示會以一般使用者訊息顯示在聊天記錄中，並在擁有該小工具的工作階段內開始正常的代理程式回合。小工具沒有任何回饋通道：遭捨棄的提示會靜默失敗，且小工具無法讀取代理程式的回覆。

## 安全性與儲存

小工具文件使用嚴格限制的內容安全政策。允許內嵌樣式與指令碼，但會封鎖外部擷取與資源載入。請將所有標記、樣式、指令碼與影像資料都放在 `widget_code` 內。

Control UI iframe 一律省略 `allow-same-origin`，即使全域嵌入模式為 `trusted` 亦然，因此小工具指令碼無法讀取父應用程式的來源。原生用戶端使用隔離且非持久性的 Web View，並阻止導覽離開託管的小工具。核心文件主機也會以 `Content-Security-Policy: sandbox allow-scripts` 回應標頭提供小工具，因此直接轉譯仍會讓小工具在不透明來源中執行，而非在應用程式來源中執行。只轉譯你願意在該隔離框架中執行的小工具程式碼。

iframe 也遵循 [`gateway.controlUi.embedSandbox`](/zh-TW/web/control-ui#hosted-embeds)。預設的 `scripts` 層級支援互動式小工具，同時維持來源隔離。

Canvas 每個工作階段最多保留 32 個小工具（沒有可用工作階段時則以每個代理程式為單位）。建立其他小工具時，會移除該範圍內最舊的文件。
