# 事件應變

## 1. 偵測與分類處理

我們監控來自以下來源的安全訊號：

- GitHub Security Advisories (GHSA) 與私密弱點回報。
- 回報不具敏感性時的公開 GitHub issue／討論。
- 自動化訊號（例如 Dependabot、CodeQL、npm advisory 與秘密掃描）。

初始分類處理：

1. 確認受影響的元件、版本與信任邊界影響。
2. 使用儲存庫 `SECURITY.md` 的範圍與範圍外規則，分類為安全性問題或強化／無需處置。
3. 事件負責人據此回應。

## 2. 評估

嚴重性指南：

- **嚴重：** 套件／發行版本／儲存庫遭入侵、遭主動利用，或未經驗證的信任邊界繞過，且造成高影響控制權或資料暴露。
- **高：** 已驗證的信任邊界繞過，且只需要有限前提條件（例如已驗證但未授權的高影響動作），或 OpenClaw 擁有的敏感憑證暴露。
- **中：** 具有實際影響的重大安全弱點，但可利用性受限或需要大量前置條件。
- **低：** 縱深防禦發現、範圍狹窄的阻斷服務，或未證實信任邊界繞過的強化／對等差距。

## 3. 回應

1. 向回報者確認收到回報（敏感情況下私下進行）。
2. 在支援的發行版本與最新 `main` 上重現，然後實作並驗證修補程式，並加入回歸涵蓋。
3. 對於嚴重／高嚴重性事件，盡快準備已修補的發行版本。
4. 對於中／低嚴重性事件，在一般發行流程中修補，並記錄緩解指引。

## 4. 溝通

我們透過以下方式溝通：

- 受影響儲存庫中的 GitHub Security Advisories。
- 已修復版本的發行說明／變更記錄項目。
- 直接向回報者追蹤狀態與解決結果。

揭露政策：

- 嚴重／高嚴重性事件應進行協調揭露，並在適當時發布 CVE。
- 低風險強化發現可依影響與使用者暴露程度，記錄於發行說明或 advisory 中而不發布 CVE。

## 5. 復原與後續追蹤

發布修正後：

1. 在 CI 與發行成品中驗證修復。
2. 執行簡短的事件後檢討（時間軸、根本原因、偵測缺口、預防計畫）。
3. 新增後續強化／測試／文件任務，並追蹤至完成。
