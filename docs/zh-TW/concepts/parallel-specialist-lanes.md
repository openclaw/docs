平行專責工作線可讓單一閘道將不同聊天或聊天室路由至不同代理程式，同時維持快速的使用者體驗。請將平行處理視為稀缺資源的設計問題，而不只是「更多代理程式」。

## 基本原則

專責工作線只有在能減少真正瓶頸的資源爭用時，才能提升吞吐量：

- **工作階段鎖定**：同一時間只能有一個執行作業變更特定工作階段。
- **全域模型容量**：所有使用者可見的聊天執行作業仍共用供應商限制。
- **工具容量**：殼層、瀏覽器、網路與儲存庫作業可能比模型本身的單輪處理更慢。
- **上下文額度**：過長的對話記錄會讓後續每輪處理變得更慢且較不聚焦。
- **權責不明**：多個代理程式重複執行相同工作會浪費容量。

OpenClaw 已按工作階段將執行作業序列化，並透過[命令佇列](/zh-TW/concepts/queue)限制全域平行處理量。專責工作線在此基礎上加入政策：由哪個代理程式負責哪些工作、哪些工作留在聊天中，以及哪些工作轉為背景作業。

## 建議的推行方式

### 階段 1：工作線契約與背景繁重作業

在每條工作線的工作區與系統提示詞中，為其提供書面契約：

- **用途**：此工作線負責的工作。
- **非目標**：此工作線應轉交而非自行嘗試的工作。
- **聊天額度**：快速回答留在聊天中；長時間工作先簡短確認，再交由背景子代理程式或任務執行。
- **轉交規則**：當工作由另一條工作線負責時，說明應轉至何處，並提供精簡的轉交摘要。
- **工具風險規則**：優先使用足以完成工作的最小工具範圍。

這是成本最低的階段，並能解決大部分阻塞問題：單一程式設計工作不再拖慢研究工作線，而每個聊天也能維持整潔的專屬上下文。

### 階段 2：優先順序與並行控制

依據每條工作線的業務價值調整佇列與模型容量：

```json5
{
  agents: {
    defaults: {
      maxConcurrent: 4,
      subagents: { maxConcurrent: 8, delegationMode: "prefer" },
    },
  },
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
    },
  },
}
```

將直接／私人聊天與正式環境維運代理程式用於高優先順序工作。系統繁忙時，讓研究、草擬與批次程式設計轉至背景任務。

### 階段 3：協調器／流量控制器

當多條工作線開始運作後，加入一個小型協調器模式：

- 追蹤進行中的工作線任務與負責者。
- 偵測不同群組間的重複要求。
- 在工作線之間路由轉交摘要。
- 只呈現阻礙、已完成的結果，以及必須由人員作出的決策。

請勿從此階段開始。沒有工作線契約的協調器只會協調混亂。

## 最小工作線契約範本

```md
# 工作線契約

## 負責

- <此工作線負責的工作>

## 不負責

- <要轉交的工作>

## 聊天額度

- 直接回答快速問題。
- 對於多步驟、耗時或大量使用工具的工作：先簡短確認，產生子代理程式／在背景執行工作，然後在完成時傳回結果。

## 轉交

如果要求由另一條工作線負責，請回覆：

- 目標工作線
- 目標
- 相關上下文
- 確切的下一步動作

## 工具使用方針

使用足以完成任務的最小工具範圍。除非此工作線明確負責，否則避免大範圍使用殼層或網路。
```
