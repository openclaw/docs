# Codex 監督

## 目標

Codex 監督可讓 OpenClaw 操作者探索原生 Codex 工作階段，並在安全的情況下，透過一般的 OpenClaw 聊天介面建立本機分支。
Codex App Server 仍是執行緒與模型迴圈的擁有者。OpenClaw 提供叢集目錄、已驗證身分的操作者使用者介面、工作階段繫結，以及頻道傳遞。

此功能屬於官方 `codex` 外掛。沒有獨立的 Supervisor 外掛，也沒有第二套 Codex 通訊協定實作。

## 產品邊界

只要 Codex 外掛處於啟用狀態，目錄就會註冊，除非使用下列設定明確停用原生工作階段探索：

```text
plugins.entries.codex.config.sessionCatalog.enabled = false
```

使用下列設定啟用供代理程式使用的監督工具：

```text
plugins.entries.codex.config.supervision.enabled = true
```

目前啟用的初始產品範圍刻意小於長期叢集計畫：

- 僅列出未封存的 Codex 執行緒。
- 依穩定的主機身分，將本機及已選擇加入的配對節點資料列分組。
- 從已儲存或閒置的閘道本機執行緒建立一般且鎖定模型的聊天分支、在首次互動時啟動其完整 Codex 控制框架執行緒，或開啟先前分支所建立的聊天。
- 僅在明確確認沒有其他執行器後，封存已儲存或閒置的閘道本機執行緒。
- 顯示作用中的本機來源，但不提供建立新分支或封存控制項，同時仍允許開啟現有的受監督聊天。
- 在主要側邊欄中顯示各主機最新的資料列、在工作階段頁面保留完整目錄，並為本機及配對節點資料列提供有界且使用游標分頁的對話記錄讀取功能。
- 依主機隔離目錄失敗。

目錄是未封存項目的集合。其中的資料列仍可具有閒置、作用中、`notLoaded` 或錯誤的互動狀態。

供代理程式使用的監督功能仍須選擇加入。引導式初始設定會在成功偵測原生 Codex 安裝，且所選推論後端通過即時檢查後，嘗試安裝並啟用此功能，無論使用者選擇哪個主要後端皆然。只有在這項機會式外掛設定成功時，監督功能才會啟用。明確停用的外掛、原則封鎖或 `supervision.enabled: false` 對監督工具仍具有最終決定權，但不會停用操作者工作階段目錄。`sessionCatalog.enabled: false` 會停用操作者探索及配對節點目錄命令；Codex 提供者與控制框架仍會保持啟用。

## 擁有權

`codex` 外掛擁有所有 Codex App Server 行為：

- 端點探索與連線生命週期
- 通訊協定初始化與版本檢查
- 執行緒列出、讀取、繼續、封存及事件處理
- 核准與使用者輸入橋接
- 原生執行緒與 OpenClaw 工作階段的繫結
- 繼續後強制使用僅限 Codex 的模型與控制框架

控制介面和閘道會使用該外掛擁有的服務。它們不會直接讀取 Codex rollout 檔案，也不會實作另一個 App Server 用戶端。

預設的本機拓撲如下：

```text
Codex Desktop -> 私有 stdio App Server -> 使用者 Codex 主目錄
                                             ^
OpenClaw Codex 外掛 -> 監督 App Server 連線
  （預設為受管理的使用者主目錄 stdio；明確的 appServer 設定會受到遵循）
  -> 被動來源目錄與讀取
  -> 快照固定 -> 標準 appServer 來源分支
  -> 可見歷程記錄注入及之後的每次受監督聊天互動

一般 OpenClaw Codex 工作階段 -> 預設為受管理的代理程式主目錄 stdio
  -> 一般完整控制框架執行緒 -> OpenClaw 聊天與頻道傳遞
```

啟用監督功能不會變更一般 Codex 控制框架：預設仍以代理程式為範圍。獨立的監督連線預設使用受管理的使用者主目錄 stdio，因此其目錄與快照作業可查看原生已儲存執行緒。明確的 `appServer` 連線設定會受到遵循。未設定 `homeScope` 時，監督連線會針對 stdio 或 Unix 將其解析為 `"user"`，針對 WebSocket 則解析為 `"agent"`。只有在一般控制框架也應共用原生 Codex 主目錄時，才明確設定 `appServer.homeScope: "user"`。從 Codex 側邊欄群組採用的聊天是例外：其私有監督繫結會讓來源讀取、標準分支建立及後續互動持續使用監督連線。即時狀態與擁有權仍僅限處理程序本機；OpenClaw 監督處理程序不知道的執行緒會是 `notLoaded`，即使 Codex Desktop 正在主動執行該執行緒亦然。

Codex 有一個實驗性的標準本機常駐程式，採用獨立、由安裝程式管理的啟動契約。此功能不得隱含啟動、宣稱擁有或假設該常駐程式。

## 目錄流程

通用閘道方法 `sessions.catalog.list` 會分派至 `codex` 目錄提供者；該提供者一律要求 `archived: false`，並讓 App Server 套用其互動式來源預設值：`cli`、`vscode`、Atlas 和 ChatGPT。它會合併：

1. 來自監督 App Server 的閘道本機 `thread/list` 結果；該伺服器預設使用受管理的使用者主目錄 stdio。
2. 來自每個已連線且已選擇加入之節點的 `codex.appServer.threads.list.v1` 結果。

對話記錄選取會在本機搭配 `itemsView: "full"` 使用 `thread/turns/list`，或在所選節點上使用具版本的 `codex.appServer.thread.turns.list.v1` 命令。每個回應最多包含 20 個已保存的互動，以及不透明的向前／向後游標。控制介面會要求由新到舊的頁面、依時間順序呈現每一頁，並將較舊頁面加到前面。它絕不會退回使用無界的 `thread/read`。OpenClaw 也會拒絕任何超過 20 MiB 的序列化項目頁面，避免其通過節點或閘道傳輸層。

原生 macOS 配對節點實作僅支援未設定／預設值或明確的 `appServer.transport: "stdio"`，並搭配未設定／預設的監督範圍或明確的 `appServer.homeScope: "user"`。它會將設定的 `command`、`args` 及正規化的 `clearEnv` 傳入子處理程序。使用 `"unix"`、`"websocket"` 或明確的 `homeScope: "agent"` 時，它不會公告目錄功能或命令；直接叫用也會以封閉方式失敗。對於代理程式範圍的設定，它絕不得公開使用者 Codex 主目錄，也不得使用本機 stdio 取代明確的端點。

目錄投影會正規化識別碼、標題、cwd、狀態、作用中的等待旗標、時間戳記、來源、模型提供者、Codex 版本及 Git 分支。它不會傳回對話記錄預覽、互動、rollout 路徑、Codex 主目錄路徑、Git 遠端、提交 SHA、原始端點或原始 App Server 錯誤。對話記錄回應僅包含明確要求的 App Server 項目頁面及其不透明游標。

主機失敗僅限於各自的主機結果。離線節點或無法使用的本機 App Server 不會從頁面移除正常主機。連線能力是主機屬性，而非執行緒狀態：失敗的主機結果不包含新的工作階段資料列，也不會將 `offline` 投影到原生執行緒上。

控制介面會要求漸進式目錄更新。每個本機或配對主機都會在其自身的 App Server 列表查詢完成後顯示；彙總回應仍作為相容性與復原快照。連線狀態變更後、取得焦點時，以及最多每 30 秒一次，可見頁面都會進行協調；變更後則會更快執行一次。因此，在其他用戶端建立的原生 Codex 工作階段最終會被探索到，而不會匯入 OpenClaw 儲存空間。

目錄探索是被動的。列出或讀取中繼資料時，不得呼叫 `thread/resume`、讓 OpenClaw 用戶端訂閱即時執行緒要求，或回應核准要求。

搜尋僅限標題且不區分大小寫。對於每個傳回的目錄頁面，閘道與配對 Mac 會掃描數量受限的原生頁面，而不將查詢傳給 App Server，因為原生搜尋也可能比對對話記錄預覽。傳回的原生游標可讓呼叫端繼續掃描。

## 操作者命令列介面邊界

此外掛會註冊三個由閘道支援的殼層命令：

```text
openclaw codex sessions [--search <text>] [--host <id>] [--limit <count>] [--cursor <cursor>] [--json] [gateway-options]
openclaw codex continue <thread-id> [--json] [gateway-options]
openclaw codex archive <thread-id> --confirm-no-other-runner [--json] [gateway-options]
```

`[gateway-options]` 是 `--url <url>`、`--token <token>`、`--timeout <ms>`，以及繼承的 `--expect-final` 開關。工作階段列表預設為 75,000 ms；繼續與封存預設為 30,000 ms；`--expect-final` 對這些一元 RPC 沒有額外作用。工作階段搜尋僅限標題且不區分大小寫；每個回應會掃描數量受限的原生頁面鏈，而 `--cursor` 會繼續取得較舊的結果。限制預設為每台主機 50 筆，並接受 1 至 100；使用游標時必須指定一個穩定的 `--host` 目的地。任何命令都不接受 archived/include-archived 選項。只有 `sessions` 可將配對主機設為目標；`continue` 和 `archive` 一律傳送 `hostId: "gateway:local"`，且封存需要明確的確認旗標。

殼層命名空間並非聊天內的 `/codex` 執行階段命名空間。具體而言，`/codex sessions --host <node>` 會列出單一節點上的 Codex 命令列介面工作階段檔案，`/codex threads` 會列出目前對話連線的 App Server 執行緒，而 `/codex resume` 或 `/codex bind` 會修改該對話的繫結。這些命令不會取代 `sessions.catalog.continue`，而且沒有 `/codex continue` 或 `/codex archive` 執行階段命令。

## 本機繼續

對於已儲存或閒置的閘道本機資料列，使用者介面會使用 `catalogId: "codex"` 加上主機與執行緒識別碼，呼叫 `sessions.catalog.continue`。此外掛會：

1. 若來源已有受監督聊天，則重複使用該聊天。
2. 否則，將直到來源最後一個終止狀態已保存互動（已完成、已中斷或失敗）為止的有界使用者與助理歷程記錄，投影至新的 OpenClaw 聊天，並記錄待處理的控制框架分支。
3. 儲存待處理的僅限 Codex 模型鎖定原則，而非具體的模型或提供者選項，再加上私有監督連線範圍，並傳回 OpenClaw `sessionKey`。

歷程記錄投影會選取最新一段可見的使用者與助理訊息，硬性限制為 200 則訊息、UTF-8 文字總計 512 KiB，以及每則訊息 64 KiB。它會將影像與本機影像輸入替換為 `[Image attachment]`，絕不複製影像承載資料或路徑，並略過推理、工具呼叫及工具結果。

使用者介面會使用該工作階段金鑰導向一般聊天。此時尚不存在標準控制框架執行緒。在首次一般聊天互動時，控制框架會安裝真正的 Codex 核准、引導提問、事件及傳遞處理常式，然後：

1. 使用監督連線呼叫原生 `thread/fork`，不覆寫模型或提供者，並固定已保存的來源快照。Codex 目前的 `ConfigManager` 狀態會選取模型與提供者，而分支回應會回報實際的配對。如果模型與來源中最後記錄的模型不同，Codex 會發出其一般的模型差異警告。
2. 在同一連線上，使用 `threadSource: "appServer"`、OpenClaw 的 cwd、原則、設定、環境、完整的 OpenClaw 控制框架工具介面，以及該分支為此次初始啟動所傳回的確切模型與提供者，啟動標準的完整 Codex 控制框架執行緒。
3. 透過該連線注入有界的可見使用者與助理歷程記錄、提交標準繫結且不捨棄其監督範圍、執行該次互動，並封存暫時分支。

在第一次輪次之前，Chat 是一個已鎖定的待處理分支，並具有可見的
歷史記錄鏡像；之後，每個模型輪次都會透過監督連線上的標準 Codex
框架執行。該分支並不是完整的原生
執行副本：來源推理、工具呼叫和工具結果都會刻意
省略。如果快照釘選或標準執行緒建立失敗，待處理
分支仍可重試。繫結競爭、停用監督，或監督連線無法使用
或不相符時，會在輪次執行前採取失敗關閉，
而不會退回一般的代理程式主目錄框架。

這可保證由 Codex 擁有選擇權，而不是保留來源
歷史使用的模型。分支所傳回的配對會用於啟動標準執行緒，
而 Codex 會持久保存該執行緒的原生模型和提供者。後續恢復時
會省略 OpenClaw 的模型和提供者覆寫，因此 Codex 會還原持久保存的配對。
如果獨立的原生 Codex 控制變更了標準執行緒，OpenClaw 會接受
該原生持久選擇。外層的 OpenClaw 模型和備援鏈
絕不會取代它。

對受監督且模型已鎖定的 Chat 進行模型變更、刪除工作階段，以及重設／新增工作階段操作時，皆會採取失敗關閉。
修改 `/codex model <model>`、`/codex
bind`、`/codex resume`（包括節點 `--bind here`），以及 `/codex detach` 或
`/codex unbind` 時，也會採取失敗關閉，因為這些操作會取代或清除繫結。
`/codex model` 查詢以及 `/codex fast`、`/codex permissions` 和 `/codex
threads` 仍可使用。`codex_threads` 代理程式工具無法附加新的
分支，也無法封存已繫結的原生執行緒。清單與僅中繼資料的讀取仍可
使用；逐字稿欄位需要 `supervision.allowRawTranscripts`，而
重新命名、取消封存、分離式分支，以及封存不相關的執行緒則需要
`supervision.allowWriteControls`。這兩個選項都無法取代已鎖定的繫結。
否則，刪除或重設 OpenClaw 項目會捨棄原生
繫結，並在看似 Codex 的工作階段背後建立或允許一般執行緒。
因此，即使模型鎖定項目超過一般的存留時間、數量或磁碟預算限制，
保留維護作業仍會保留這些項目。停用或解除安裝擁有該項目的
外掛，也會保留鎖定和外掛擁有權標記。Chat 會維持
無法使用並採取失敗關閉，直到重新啟用同一個外掛；清理作業絕不會
將其轉換成一般模型工作階段。

此操作絕不會恢復或修改來源。暫時分支會釘選
快照；它不是持久的延續執行緒。在第一次輪次啟動不同的
標準框架執行緒，可避免 OpenClaw 僅因程序本機狀態未能偵測到
Desktop 所擁有的輪次，而成為與來源競爭的寫入者。可見歷史記錄鏡像和釘選快照可能會省略
作用中來源裡尚未完成的工作。原始命令列介面、VS Code、
Atlas 或 ChatGPT 來源仍符合原生與 OpenClaw 目錄的資格。
標準分支在監督儲存區中仍是原生 Codex 執行緒，
但原生用戶端可能會篩除其 `appServer` 來源種類，因此無法保證會顯示在 Codex Desktop
中。

## 封存行為

對於已儲存或閒置的閘道本機資料列，搭配
`catalogId: "codex"` 的 `sessions.catalog.archive` 需要
明確的 `confirmNoOtherRunner: true`，會重新讀取目前的程序本機
狀態，且僅在狀態為 `idle` 或 `notLoaded` 時繼續，接著呼叫原生 `thread/archive`，
並且只有在 Codex 接受操作後才傳回成功。該資料列隨後會離開
未封存目錄。

重新讀取後若狀態為作用中或錯誤，會拒絕封存。來源中的
初始化中或待處理受監督分支也會被拒絕：第一次 Chat 輪次
必須先具現化其標準分支，才能封存來源。若確切目標存在
已知的作用中 OpenClaw 繫結擁有者，或存在任何未封存的
衍生後代，也會拒絕封存。OpenClaw 會對 Codex 的實驗性
`thread/list ancestorThreadId` 關係進行分頁，並在要求或回應
發生錯誤、游標或執行緒形成循環，以及耗盡安全限制時採取失敗關閉。原生封存可能會
關閉已載入的父項與後代工作，因此封存不是中斷
捷徑。讀取、列舉後代和封存呼叫並非不可分割。
獨立用戶端仍可能擁有看似閒置或在本機為
`notLoaded` 的資料列，或在其中啟動工作。在 Codex 提供條件式封存或跨程序租約之前，
「沒有其他執行器」確認涵蓋未知用戶端和此競爭情況。
禁止封存配對節點。

Codex 目錄中沒有已封存檢視。在另一個經擁有者授權的 Codex 介面中，
使用 `thread/unarchive` 還原的執行緒會再次符合
未封存目錄的資格。

## 作用中執行緒安全性

Codex 會在單一 App Server 的用戶端之間序列化執行緒修改，
但不會公開獨佔的跨程序執行器或核准擁有者租約。
各自獨立的 stdio App Server 可以附加至同一個執行記錄，而每一個伺服器
只能看到自己的記憶體內狀態。核准要求也可送達單一伺服器的每位訂閱者，
並由第一個有效回應完成該要求。

因此：

- 被動目錄用戶端不會訂閱或自動拒絕核准要求
- 目前回報為作用中的資料列不會提供新分支或 Archive
- 未對應的來源會成為可見歷史記錄分支，其標準框架
  執行緒絕不會恢復來源
- `notLoaded` 會顯示為活動狀態不明，且只有在知情確認沒有其他執行器後
  才能封存
- 本機封存需要該確認，並重新讀取 `idle` 或 `notLoaded`，
  同時確認協定在讀取與封存之間存在競爭情況

中斷和多用戶端交接是未來的產品決策。顯示作用中資料列
並不代表支援這些功能。

## 配對節點邊界

節點叫用目前僅支援要求／回應。它可以安全地傳回有界的
目錄中繼資料和逐字稿輪次頁面，但無法承載 Codex
框架執行所需的長時間事件串流、核准要求、
工具呼叫、取消和助理增量內容。

因此，節點合約支援清單和逐字稿輪次頁面。遠端
資料列仍可讀取，但無論是否為閒置狀態，**Continue** 和 **Archive** 都無法使用。真正的
遠端延續需要節點端執行器和串流橋接，以
維持與本機框架相同的核准和繫結不變條件。

## 權限

每台電腦都必須在本機選擇加入。啟用閘道並不會授權另一個
節點讀取其 Codex 中繼資料。節點能力必須通過一般配對
和命令原則核准。

機群清單和逐字稿檢視會使用 `operator.write` 閘道範圍，
因為它們會叫用配對節點。本機延續和封存是
經過驗證的操作者操作，且仍受主機和狀態檢查限制。

自主代理程式和獨立 MCP 存取是另一回事。已發布的
`codex_endpoint_probe`、`codex_sessions_list`、`codex_session_read`、
`codex_session_send` 和 `codex_session_interrupt` 工具合約仍由
`codex` 外掛擁有。啟用監督時，原始 `codex_threads` 逐字稿
讀取和從逐字稿衍生的清單欄位也需要
`supervision.allowRawTranscripts`；每次 `codex_threads` 分支、重新命名、封存
或取消封存都需要 `supervision.allowWriteControls`。這兩個原則預設皆為
停用。

## 相容性

`openclaw doctor --fix` 會遷移已發布的 `plugins.entries.codex-supervisor`
設定，包括端點和逐字稿／寫入原則，以及外掛
允許／拒絕參照，並移至
`plugins.entries.codex.config.supervision`。發生衝突時，以明確的標準目的地
值為準。遷移後，執行階段程式碼僅使用標準 `codex` 外掛
形狀。

官方外掛恰好保留五個 Supervisor 相容性工具：
`codex_endpoint_probe`、`codex_sessions_list`、`codex_session_read`、
`codex_session_send` 和 `codex_session_interrupt`。工作階段清單預設僅包含已載入項目；
沒有 `loaded_only` 參數。`include_stored: true` 會新增
未封存的狀態資料庫資料列，每個端點受 `max_stored_sessions`
限制（預設為 200，可接受範圍為 1 至 1,000）；該設定不會限制已載入的資料列。
從逐字稿衍生的欄位和讀取仍受
`allowRawTranscripts` 限制；傳送和中斷仍受 `allowWriteControls` 限制。

相容性傳送絕不會啟動或恢復閒置執行緒。`mode: "start"`
一律會被拒絕；`"auto"` 和 `"steer"` 僅會引導可讀取的作用中輪次。
中斷同樣需要可讀取的作用中輪次。閒置延續會路由至
原生 Codex 目錄，讓完整框架擁有核准、工具和繫結。
獨立的舊版 MCP 介接器會從官方
外掛解析這些相同工具，並且是唯一遵循所保留舊版原則環境
變數的路徑。

7 月的目錄使用者介面、閘道方法、節點能力和命令列介面註冊
尚未以舊外掛 ID 發布。它們會直接移交給 `codex` 擁有，
而不會加入第二個執行階段外觀介面。

- 在執行器擁有權租約或同等防護機制存在後支援遠端封存

## 驗收測試

- 啟用監督功能會列出未封存的本機工作階段。
- 已封存的工作階段絕不會出現在目錄回應或 UI 中。
- 當另一部主機發生故障時，健康的主機仍會保持可見；無法使用的主機
  不會虛構離線工作階段狀態，而是不傳回任何最新資料列。
- 已儲存或閒置的本機資料列會建立一個具有僅限 Codex
  模型／執行階段鎖定的 Chat 鏡像；第一個對話回合會固定一份暫時快照並啟動
  標準的完整工作框架執行緒，而再次執行 Continue 則會開啟現有的 Chat。
- 第一個對話回合會在快照分支上省略模型／供應商覆寫，並將
  標準啟動固定至 Codex 傳回的確切配對，即使 Codex 警告其目前模型
  與來源最後記錄的模型不同。
- 待處理及已提交的受監督繫結會使用監督連線來
  存取來源、建立標準分支，以及執行之後的每個對話回合；一般
  Codex 工作階段仍限定於代理程式範圍。
- 之後的恢復作業會省略 OpenClaw 模型／供應商覆寫、保留 Codex
  標準的持久化選擇、接受對該執行緒的個別原生變更，
  且絕不以外層 OpenClaw 模型或備援鏈取代。
- 停用監督功能，或失去繫結／連線生命週期時，會以關閉方式失敗，
  而不會將 Chat 移至一般代理程式主目錄的工作框架。
- 受監督且模型鎖定的 Chat 在保護原生
  繫結期間無法刪除。
- Chat 最多鏡像 200 則使用者與助理訊息、總計 512 KiB，且
  每則訊息最多 64 KiB。影像會變成預留位置；來源推理、工具呼叫、
  工具結果、影像承載資料及本機路徑不會被複製。
- 分支流程絕不會恢復來源執行緒。
- 原始來源仍符合出現在兩個目錄中的資格。標準原生
  分支使用 `appServer` 來源種類，且不保證會出現在
  Codex Desktop 中。
- 作用中的本機來源無法建立分支或封存；現有的
  受監督 Chat 仍可開啟。
- 活動狀態未知的資料列無須確認即可建立分支；封存則需要
  明確確認沒有其他執行者。
- 具有初始化中或待處理受監督分支的來源無法封存，
  直到第一個 Chat 對話回合具現化標準分支為止。
- 若確切目標或任何未封存的衍生
  後代具有已知的作用中繫結擁有者，便會阻止封存；後代列舉失敗時會以關閉方式失敗，而
  明確確認仍須負責處理未知用戶端及
  從狀態判定到封存之間的競爭條件。
- 經確認已儲存或閒置的本機封存作業，會在原生作業成功後移除資料列。
- 配對節點的資料列會保持可見，但不提供 Continue 或 Archive。
- 被動列出作業絕不會訂閱或回應執行緒核准要求。
- 舊版 Supervisor 設定會遷移至標準 Codex 設定格式。
- 舊版清單預設僅會載入，已儲存項目的列舉會遵循其各端點
  上限，而相容性傳送絕不會啟動或恢復閒置的執行緒。
