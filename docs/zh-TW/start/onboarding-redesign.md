# 新手引導重新設計實作計畫

> **持續更新的文件。** 本頁在實作層級追蹤系統管理員新手引導的重新設計，
> 並隨各階段完成而更新。最後一個階段合併後，
> 本頁將改寫為面向使用者的新手引導指南，並加入
> 文件導覽。在此之前，刻意不將其放入 `docs.json`。

## 核心願景

非技術使用者輸入 `openclaw onboard`（或開啟應用程式）後，
會由單一對話式角色迎接——OpenClaw，也就是系統管理員（「custodian」僅為
內部名稱；使用者始終看到「OpenClaw」）——它會找到使用者的 AI，
以事先告知的預設值取代提問來完成所有設定，透過可見的身分誕生時刻孵化使用者的
代理程式，並從此永遠保持可聯絡，擔任系統的照管者。預設即有魔法般的體驗、
只有一道同意界線，且絕不走入死路。

設計原則（已決定，請勿隨意重啟爭論）：

- **事先告知且可輕鬆復原的預設值**取代會阻斷流程的問題。唯一的
  硬性要求是推論功能可正常運作；其他一切都只是選項。
- **第零題就是同意界線**：「完整存取權」（建議）代表
  探索會靜默且自動執行；「先詢問」則會將所有探索——AI
  掃描、應用程式掃描及記憶來源掃描——統一置於一次
  明確同意之後，並提供一條完全手動、絕不掃描的路徑。
- **以對話作為介面，並逐步啟用智慧功能**：系統管理員介面
  在任何 AI 可運作前就已存在（使用腳本式對話），於路由驗證成功的
  當下改由模型支援，並清楚告知。它絕不假裝具備智慧：
  在路由驗證前輸入自由文字時，會得體地回應「先讓我把大腦運作起來」。
- **孵化是一場儀式**：使用同一個對話串、更換頭像，代理程式會自行命名
  並選擇自己的面孔。系統管理員只會教一次層級關係：「可以向我詢問
  系統相關問題，也可以直接問你的代理程式——它會轉達。」
- **信任依來源分級**：官方目錄項目可以預先選取；
  第三方 ClawHub Skills 無論模型排名如何都不會預先選取，
  且其標籤會說明將安裝發布者的程式碼。
- **已設定的安裝不可侵犯**：重新執行新手引導時只進行驗證。
  絕不重新套用設定，也絕不重新啟動閘道服務。
- **終端機是備援方案，而不是一道問題**：當閘道可連線時，優先使用瀏覽器
  儀表板；絕不詢問「終端機還是瀏覽器？」。
- **較弱的模型會使用精簡介面**（自動 `localModelLean`），並以
  白話說明——絕不提及工具、程式碼模式或上下文視窗。

## 目前已發布的流程（階段 1–3 完成後）

全新 macOS 安裝中的 `openclaw onboard` 順利流程——總共按四次 Enter：

1. 安全性注意事項 → 按一次 Enter 確認（持久保存；不再詢問）。
2. **第零題**：「我該如何進行設定？」——完整存取權（建議）
   或先詢問。選擇會以 `wizard.accessMode` 持久保存；重新執行時預設採用已儲存的
   選擇。受限模式 +「手動設定」可直接進入供應商選擇器，
   不執行任何掃描，也會略過記憶來源掃描。
3. **探索過場**：偵測程式設計命令列介面、環境變數金鑰及本機執行環境；
   找到程式設計代理程式時會給出一句俏皮回應；依序即時測試候選項目，並
   將失敗項目靜默彙整為一行摘要（詳細資料位於「See other
   options」之後）。第一條可正常運作的路由會作為預設值事先告知，並提供
   按一次按鍵即可前往完整選擇器的路徑；探索其他選項或略過時仍會保留
   可正常運作的路由。
4. 記憶匯入選項（Claude Code / Codex / Hermes），若拒絕探索則略過。
5. 僅限全新安裝：自動套用標準設定計畫
   （工作區、閘道服務、工作階段——與對話式「yes」所執行的計畫相同）。
   已設定的安裝會顯示「already set up」，且絕不變更服務。
6. **應用程式建議**：由已驗證模型比對已安裝應用程式與
   官方目錄 + ClawHub；官方頻道外掛會預先勾選，
   第三方 Skills 則需主動選取並附有警告標籤。可略過；
   停用開關為 `wizard.appRecommendations`。
7. **孵化**：當閘道可連線時，瀏覽器交接會開啟（GUI），或
   顯示（無頭模式/SSH）儀表板 URL，並等待控制介面
   連線——「儀表板已連線——將在瀏覽器中繼續。」若無法連線，或
   使用 `--tui`，則會開啟終端介面，並預先填入啟動孵化
   訊息，接著由代理程式自我介紹。

遠端閘道新手引導會保留舊有的對話式交接流程
（`handoffMode: "chat"`）；設定必須套用於遠端閘道。

## 各階段實作備註

### 階段 1——應用程式建議（PR #109668）

- 掃描器：`src/infra/installed-apps.ts`（無需 TCC 的 macOS 列舉；會追蹤
  符號連結的 `.app` 套件）。
- 候選項目：官方目錄 + ClawHub 搜尋，整體時間預算為 20 秒，
  離線時可平順降級為僅使用目錄候選項目。目錄項目是頂層沒有
  `id` 的套件資訊清單——候選項目以解析後的外掛 ID 作為鍵值
  （已使用實際內建目錄進行迴歸測試；曾經改以
  `entry.id` 作為鍵值，導致整個目錄合併成單一項目，並遺漏所有官方
  建議）。
- AI 配對器：在已驗證路由上執行一次補全
  （`src/system-agent/setup-app-recommendations.ts`）；沒有精選套件 ID 對照表——
  模型會排除名稱偶然重疊的項目。輸出受解析後模型本身的
  `maxTokens` 預算限制（未傳入明確上限時，由串流層套用）。
- **供應鏈防護**：ClawHub 清單文字由發布者控制，
  且會進入配對器提示，因此清單可以將自身宣傳為
  「recommended」。只有官方目錄項目可以預先選取；ClawHub
  Skills 一律需要明確勾選，並標示為「第三方 ClawHub
  Skill；會安裝其發布者的程式碼」。
- 節點命令 `device.apps`（TS 節點主機、與 Android 封裝格式一致），預設
  不分享；閘道停用開關為 `wizard.appRecommendations`。
- 交付位於傳統精靈與引導式系統管理員流程中
  （`src/wizard/setup.app-recommendations.ts`）；重新定位至啟動流程尾端仍屬
  階段 5（服務已接受可注入的清單來源）。單次語意（只提供到接受為止，
  並儲存掃描結果）也會隨階段 5 的儲存區一併完成；目前重新執行仍會再次提供。
- 另已修正：自訂 `completeSetupInference` 提示不再繼承
  32 個權杖的驗證探查輸出上限（`SETUP_INFERENCE_TEST_MAX_TOKENS`
  僅套用於「reply OK」探查）。

### 階段 2——命令列介面系統管理員骨幹（PR #109841）

- 在 `src/commands/onboard-guided.ts` 中重製流程；遠端閘道新手引導
  透過 `handoffMode: "chat"` 保留舊有的聊天交接。
- 第零題會持久保存 `wizard.accessMode`（"full" | "guarded"）；重新執行時
  預設採用已儲存的選擇（接受預設值絕不能在未告知的情況下，
  將受限模式降級為完整模式）。受限模式 + 手動設定會使用
  `listManualSetupInferenceOptions`（僅使用設定/資訊清單，不進行探查），並
  略過記憶來源掃描。
- 探索：靜默收集失敗項目（單一摘要行；詳細資料位於
  「See other options」之後）、程式設計代理程式俏皮回應，以及事先告知的路由預設值。
  在低成本的工作階段計數介面出現之前，俏皮回應中的工作階段
  數量會延後實作（目前僅提供定性描述）。
- 全新安裝：`applySystemAgentSetup`（確定性的對話式
  「yes」），接著透過 `launchTuiCli` 進行孵化，並預先填入啟動訊息。
  已設定的安裝（預先存在模型或閘道設定——精靈
  時間戳記無法證明任何事情，因為 configure/doctor 也會共用）：
  僅進行驗證——不套用、不重新啟動閘道服務。套用失敗時
  會退回對話式聊天。

### 階段 3——瀏覽器優先交接（PR #110054，已合併）

- `src/commands/onboard-browser-handoff.ts` 負責純圖形工作階段
  偵測（`SSH_CONNECTION`/`SSH_TTY`；Linux 上為 `DISPLAY`/`WAYLAND_DISPLAY`）
  以及 60 秒 GUI／300 秒 SSH 等待。目前引導式初始設定
  僅在 macOS 啟用交接；`--tui` 和其他平台保留
  終端退路。Linux/Windows 的啟用將於後續處理。
- Dashboard 連結使用與傳統完成流程相同的 `resolveAdvertisedControlUiLinks`、
  `resolveLocalControlUiProbeLinks` 和 `buildOnboardingControlUiUrl` 輔助函式。
  瀏覽器啟動使用共用的 `openUrl` 輔助函式。
- 就緒狀態會輪詢現有的 `system-presence` RPC，並作為**呈現已設定共用密鑰的命令列介面模式
  回送用戶端**運作——這是每個
  `openclaw` 命令使用的受信任路徑。在 SecretRef 閘道上，使用原始共用驗證的 Control UI 用戶端會遭到拒絕，
  並顯示 "device identity required"。連線能力
  預檢會解析與等待迴圈相同的目標（及密鑰），因此
  關卡與等待絕不會對驗證結果產生分歧。僅當已連線的
  `openclaw-control-ui`/`webchat` 在線狀態資料列相較於啟動前基準是新的時，
  交接才會完成（已開啟的 Dashboard
  無法完成交接）。
- `gateway.controlUi.enabled: false` 會在顯示任何 URL 前提前結束。
- 已針對使用相同隔離設定的閘道完成端對端驗證：列印 URL → 真實
  瀏覽器連線 → "Dashboard connected — continuing in your browser" → 無
  終端退路。先前因 "token mismatch" 而暫停是測試工具
  造成的假象——請參閱下方測試手冊。

### 階段 4 — Web 管理員介面（已合併：#110141、#110242）

- 透過 `openclaw.chat` 提供的 `/custodian` 頁面，搭配選項卡片元件
  （2-4 張卡片、最多一個建議選項、永遠可略過）；透過
  `?onboarding=1` 提供初始設定框架；首次完成模型設定後會交接至此。
- 結構化問題是 `SystemAgentChatResult` 上具型別且可附加的 `question` 欄位
  （每個選項有 `reply` 文字；macOS 應用程式／終端介面的文字敘述一律可獨立呈現）。產生來源：兩種初始設定歡迎流程，
  以及託管精靈中具有 2-4 個封閉選項的選取／確認步驟——實際頻道
  精靈會呈現為卡片。PR1 的字串標記權宜措施已刪除。
- 工作階段擁有權的範圍涵蓋閘道 URL + 所有提供的認證資訊
  （權杖、密碼、啟動權杖、已儲存的裝置權杖——在短暫的 hello 中斷期間仍持續有效）；
  失敗的使用者回合絕不可重新播放；敏感輸入會逐字傳送，
  並在逐字稿中遮蔽。

### 階段 5 — 孵化與啟動（已合併：#110173、#110331）

- 管理員建立無名代理程式（工具呼叫）；代理程式的啟動流程
  以自行命名開始。PR1 提供的儀式上限為三個節拍（名稱 → 靈魂
  描述 → Skills 問題），並將自行繪製頭像／圖像生成階梯
  （模型產生候選項目 → 預設標記 → 保留標誌）延至後續處理。在相同
  對話串中切換頭像；爪印仍保留給管理員使用。議定的
  身分會儲存兩次：寫入 `IDENTITY.md`/`SOUL.md`（代理程式
  讀取的內容），以及透過 `openclaw agents set-identity`（頻道和 UI
  顯示的內容）。
- 建議項目（階段 1 服務，具一次性語意的已儲存掃描）會成為
  啟動檔案移除前的最後一個啟動步驟："minimal set
  or maximum convenience?" 啟動流程透過
  `openclaw onboard recommendations --json` 讀取已儲存的提議（僅限不透明的安裝 ID），並在
  處理選擇後確認，使其絕不會再次詢問。頻道
  連線按鈕包含各頻道的設定手冊；代理程式以對話方式收集
  認證資訊，並將設定寫入轉交給管理員
  （"asking OpenClaw…" 是標準慣用語）。
- 自行學習應以詢問方式提出，而非直接宣布，並同時作為 Skills 工作坊的
  同意；描述 ClawHub 的發布信任、掃描、驗證和完整性
  檢查，以及發布者程式碼警告——絕不可暗示每個版本都有簽章。
- 自動孵化已推出：全新安裝的設定套用會宣布孵化並
  進行交接（終端介面／閘道用戶端的 `open-agent`）；網頁會
  進入代理程式聊天，並預先填入 "Wake up, my friend!" 草稿。
  僅在寫入後驗證完全成功時才會觸發交接。刪除後
  代理程式為零時提供建立選項（而非自動建立），仍屬後續潤飾。

### 階段 6 — 管理員在線狀態（PR1 已合併：#110269；評論／召喚屬於 PR2）

- PR1 已推出：預設釘選的 "OpenClaw" 側邊欄項目（全新設定檔；
  現有使用者保留已儲存的釘選項目，並可透過 customize/More 存取）、
  作為第一個 Settings 項目的 "Ask
  OpenClaw"，以及採用一般框架的 `/custodian` 瀏覽，
  其會要求管理員問候（不使用初始設定歡迎變體），且
  Exit setup 僅在初始設定模式中呈現。停駐式內嵌 Settings 窗格
  需要抽取共用對話檢視（後續處理）。
- 事件回應式評論，並設有防 Clippy 護欄：僅針對重大或
  失敗的變更，除非有人詢問，否則每次造訪設定最多一次。相同的
  事件介面日後也會讓管理員為降級的驗證或故障的
  頻道發聲。
- 頻道：日常使用時不可見（由代理程式轉交）；可透過明確
  召喚或同一對話串中的代理程式離線事件存取，並在平台允許時使用其自己的名稱和
  爪形頭像。
- 設定時偵測到弱模型：自動設定 `localModelLean`，且管理員
  會以白話說明，並提供升級選項。
- 管理員知道自己的內部暱稱（"some folks call me the
  custodian — OpenClaw's fine"），並一律以名稱稱呼代理程式。

### 階段 7 — 韌性（建置前需要擁有者決策）

原始構想——"the custodian must be reachable no matter how broken
the config is"——與儲存庫的安全性原則衝突：根指南
指出，當設定在結構上無效時，閘道會**拒絕啟動**，
且只有 SecretRef 擁有者失敗會降級為已設定但無法使用的
功能。從無效設定提供任何介面都是原則變更，
而非實作細節。有兩種範圍，請擇一：

- **選項 A（建議，符合原則）：命令列介面端自動 doctor。** 當
  閘道或命令列介面因已知形態的無效設定而啟動失敗時，命令列介面會提議
  執行（或在取得同意後執行）`openclaw doctor --fix`，接著重試一次並
  清楚回報。不變更閘道行為；管理員仍可透過
  現有的 SecretRef 降級路徑和終端存取。
- **選項 B（需要擁有者明確核准 + 安全性審查）：閘道
  最小介面模式。** 當設定在結構上無效時，啟動鎖定的
  介面，僅提供管理員對話和 doctor 動作。這會
  改寫故障時封閉的啟動合約，且必須在撰寫任何程式碼前定義其自身的輸入
  保護方案。

階段 4-6 的剩餘後續項目（已追蹤，尚未排程）：孵化流程的頭像／圖像生成
階梯；macOS 應用程式對具型別 `question` 欄位的呈現；管理員的
停駐式內嵌 Settings 窗格（需要抽取共用對話檢視）；事件回應式評論和頻道召喚／代理程式離線復原
（階段 6 PR2）；弱模型的自動 `localModelLean`；現有
使用者已儲存的側邊欄釘選項目是否應採用 OpenClaw 項目。

## 測試與合併手冊（得來不易；進行階段 4-6 前請先閱讀）

- **`OPENCLAW_STATE_DIR` 不會隔離閘道服務。**
  LaunchAgent 標籤（`ai.openclaw.gateway`）是整台機器共用的：使用隔離狀態目錄進行全新安裝
  初始設定測試，會改寫並重新啟動真實
  機器上的服務（包裝函式指令碼會放在隔離目錄中；清除該目錄後，
  下次服務啟動便會中斷）。進行任何全新安裝
  測試後，請從真實環境使用 `openclaw gateway install --force && openclaw gateway
restart` 還原，並驗證 plist。產品後續項目：
  狀態目錄範圍的服務標籤，或讓初始設定偵測外部服務。
- **安全的端對端測試工具**：預先在隔離設定中加入 `gateway`
  區段（使初始設定採用已設定安裝路徑，且永不觸碰
  服務），並在備用連接埠上使用純文字權杖，將 `openclaw gateway run` 作為一般前景程序執行。
  此測試工具已驗證階段 3 迴圈，包括真實瀏覽器連線。
- **驗證路徑依用戶端身分而異，而不只取決於認證資訊。** 在線狀態和
  其他操作員讀取使用命令列介面模式的回送用戶端，其認證資訊來自相同的
  設定。權杖驗證閘道需要共用密鑰；SecretRef/none
  閘道可在沒有權杖時回退至受信任的回送驗證。識別為 Control
  UI 的瀏覽器用戶端需要裝置身分，或安全情境的
  回送授權。若探查對提供不同設定的閘道進行驗證
  （請參閱 LaunchAgent 陷阱），將因 "token mismatch" 而失敗——此
  假象曾短暫阻礙階段 3。
- **完成探查**：`runSetupInferenceTest` 將驗證探查限制為
  32 個輸出權杖；自訂提示會略過此限制，並受
  模型自身的 `maxTokens` 約束。推理模型會先以隱藏
  推理消耗該預算——空文字回合通常表示預算已在該處耗盡。
- **代理程式合併需要精確對應最新提交的託管 CI。** 在組織負載下，大型 `CI` 工作流程可能
  不會因推送而排入佇列；維護者的備援方式是在 PR 分支上
  派送發布關卡：

  ```bash
  gh workflow run ci.yml --ref <branch> -f target_ref=<head-sha> -f release_gate=true -f pull_request_number=<pr>
  ```

  此執行必須位於
  分支 ref 上，使 `head_sha` 相符，且標題會變成
  `CI release gate <sha>`，而 `scripts/verify-pr-hosted-gates.mjs`
  會接受該標題。接著照常進行 `scripts/pr` 準備／合併。

- **CI 除重點測試外還會強制執行的關卡**：文件對應表
  （新增任何文件頁面後執行 `pnpm docs:map:gen`）、oxlint（`no-map-spread`、
  `max-lines`——拆分檔案，絕不抑制）、`check:test-types`、knip
  無用程式碼檢查（僅匯出生產環境使用的項目；測試透過公開 API 執行），
  以及即時測試分片分類器
  （`test/scripts/test-live-shard.test.ts` 必須列出所有新的 `*.live.test.ts`）。

## 決策紀錄

- 採用附終止開關的自動掃描，而非先取得同意（階段 1；持續顯示的輸出會
  在掃描前揭露模型和 ClawHub 的使用情況，結果附註也會再次說明）。
- 完整垂直流程，包括節點 `device.apps` 命令（階段 1）。
- 第三方 ClawHub Skills 絕不預先選取，且會標示為
  安裝發布者的程式碼；官方項目可預先勾選
  （階段 1，已推出的安全性態勢）。
- 使用兩張存取卡片，而非三張；同意機制前置整合至選擇中（階段 2）。
- 採用附公告的自動孵化，而非阻塞式按鈕（階段 2/5）。
- 瀏覽器優先：終端退路是備援方案，絕不詢問 "terminal or
  browser?"（階段 3）。
- 管理員會出現在頻道中（召喚 + 復原），而不只限於 Web／命令列介面
  （階段 6）。
- 孵化會在相同對話串中進行，並切換頭像；完成後，
  應用程式會轉換至一般 UI（階段 5）。
- 設定介面保留 "Settings" 名稱；管理員位於其中
  （以及側邊欄中），而非取代它（階段 6）。
- 選項卡片受到限制：2-4 個選項、恰好一個建議選項、永遠
  可略過；相同元件同時用於初始設定和代理程式問題工具
  （階段 4）。
- "Asking OpenClaw…" 是標準委派慣用語；靈魂可增添風格，
  工具敘述則保持平實（階段 5）。
- 向使用者說明弱模型裁減時，文案絕不使用 "code mode"、"tools" 或 "context window"
  （階段 6）。

## 已知缺口與後續工作

- LaunchAgent 標籤未依狀態目錄限定範圍（這是上述的測試陷阱，也是真實的多執行個體產品缺口）。
- 建議項目的僅一次語意與儲存的掃描（第 5 階段）；目前重新執行時會再次提供。
- 瀏覽器交接僅支援 macOS；Linux/Windows 支援仍待完成。
- 工作階段數量的趣味提示僅為定性描述；若要提供數量，需要低成本的工作階段計數介面。
- 瀏覽器交接會進入一般儀表板；新手引導模式管理員的深層連結將於第 4 階段推出。
