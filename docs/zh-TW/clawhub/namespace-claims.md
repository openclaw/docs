# 組織與命名空間申請

ClawHub 使用擁有者代稱、組織代稱、skill slug、外掛套件名稱，以及
套件範圍作為公開命名空間。如果某個命名空間看起來屬於
真實世界的專案、品牌、套件生態系或組織，但在 ClawHub 上已被
申請、保留、造成誤導或存在爭議，請要求工作人員透過
[組織／命名空間申請議題表單](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
進行審查。

此途徑用於公開、非敏感的所有權審查。請勿將產品內回報
或帳號申訴表單用於命名空間申請。

## 何時開啟申請

當你認為 ClawHub 工作人員應審查某個命名空間是否因真實世界所有權而
需要保留、轉移、重新命名、隱藏、隔離、設為別名，
或以其他方式變更時，請開啟命名空間申請。

範例包括：

- 與你的 GitHub 組織、專案、公司或社群相符的組織代稱
- 像 `@example-org/*` 這樣的套件範圍，應只允許對應的
  ClawHub 擁有者發布
- 看起來冒充某個專案的 skill slug 或外掛套件名稱
- 品牌、商標、專案重新命名或套件歷史爭議
- 已刪除、不活躍或無法聯絡的擁有者，阻擋了合法命名空間
  擁有者

如果該列表除了所有權爭議之外還不安全、惡意或具誤導性，
也請遵循相關的管理或安全指引。命名空間申請
表單是用於所有權審查，而不是緊急漏洞揭露。

## 提交前

請先確認你是以符合該命名空間的擁有者身分發布。
對於外掛套件，像 `@example-org/example-plugin` 這樣的範圍名稱必須
以對應的 `example-org` 擁有者發布。

如果你可以管理目前的擁有者，請透過發布、重新命名、轉移、隱藏
或刪除受影響資源，直接修正該命名空間。當你無法管理目前的擁有者，
或需要工作人員解決爭議時，才使用申請。

## 應包含的證據

使用公開、非敏感的證據。有幫助的證明包括：

- GitHub 組織、儲存庫、版本發布或維護者歷史
- 指明該命名空間的官方專案文件
- 網域或官方電子郵件網域證明
- npm、PyPI、crates.io 或其他套件登錄範圍控制
- 可公開討論的商標、品牌或專案所有權證據
- 原始碼儲存庫歷史、套件歷史或公開重新命名通知
- 指向有爭議 ClawHub 擁有者、skill、外掛、套件或議題的連結

說明每個連結證明了什麼。工作人員應能在不需要私人憑證
或秘密的情況下理解其關係。

## 不應包含的內容

請勿在公開 GitHub 議題中放入秘密或私人證明。請勿包含：

- API 權杖、簽署金鑰或憑證
- DNS 驗證權杖
- 私人法律文件或合約
- 個人身分文件
- 私人電子郵件、私人安全報告或機密客戶資料

申請表單會詢問是否需要私人工作人員通道來提交敏感證據。
請使用該選項，而不是公開發布敏感材料。

## 可能結果

根據證據與風險，ClawHub 工作人員可能會保留命名空間、
轉移所有權、重新命名資源、隱藏或隔離既有列表、
新增別名或重新導向、要求更多證明，或拒絕請求。

命名空間審查不保證每個相符名稱都會被轉移。
工作人員會權衡公開證據、既有使用情況、安全風險與使用者影響。
