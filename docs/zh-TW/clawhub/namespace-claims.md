# 組織與命名空間認領

ClawHub 使用擁有者代號、組織代號、Skill slug、外掛套件名稱及
套件 scope 作為公開命名空間。若某個命名空間看似屬於
現實世界中的專案、品牌、套件生態系或組織，但在 ClawHub 上已遭
認領、保留、具有誤導性或存在爭議，請透過
[組織／命名空間認領問題表單](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
請工作人員審查。

此管道適用於公開且不涉及敏感資訊的所有權審查。請勿使用產品內
檢舉功能或帳號申訴表單提出命名空間認領要求。

## 何時提出認領

若你認為 ClawHub 工作人員應審查某個命名空間是否因現實世界中的所有權，
而應予保留、移轉、重新命名、隱藏、隔離、設定別名
或以其他方式變更，請提出命名空間認領。

範例包括：

- 與你的 GitHub 組織、專案、公司或社群相符的組織代號
- 如 `@example-org/*` 的套件 scope，且只應由相符的
  ClawHub 擁有者發布
- 疑似冒充某個專案的 Skill slug 或外掛套件名稱
- 品牌、商標、專案更名或套件歷史爭議
- 已刪除、停止活動或無法聯絡，並阻礙正當命名空間
  擁有者的現有擁有者

若該項目除了所有權爭議外，也有不安全、惡意或誤導情形，
請同時遵循相關的內容管理或安全性指南。命名空間認領
表單用於所有權審查，而非緊急漏洞揭露。

## 提交前

請先確認你是透過與該命名空間相符的擁有者發布。
對於外掛套件，如 `@example-org/example-plugin` 的 scoped 名稱，必須以
相符的 `example-org` 擁有者身分發布。

若你能管理目前的擁有者，請直接透過發布、
重新命名、移轉、隱藏或刪除受影響的資源來修正命名空間。當你
無法管理目前的擁有者，或需要工作人員解決
爭議時，再提出認領。

## 應附上的證據

請使用公開且不涉及敏感資訊的證據。實用的證明包括：

- GitHub 組織、儲存庫、版本發布或維護者歷史
- 載明該命名空間的官方專案文件
- 網域或官方電子郵件網域的證明
- 對 npm、PyPI、crates.io 或其他套件登錄檔 scope 的控制權
- 可安全公開討論的商標、品牌或專案所有權證據
- 原始碼儲存庫歷史、套件歷史或公開的更名公告
- 爭議所涉 ClawHub 擁有者、Skill、外掛、套件或問題的連結

請說明每個連結能證明什麼。工作人員應能理解其中的
關聯，而不需要私人認證資訊或密鑰。

## 不應附上的內容

請勿在公開的 GitHub 問題中放入密鑰或非公開證明。請勿附上：

- API 權杖、簽署金鑰或認證資訊
- DNS 驗證權杖
- 非公開的法律文件或合約
- 個人身分證明文件
- 私人電子郵件、非公開安全性報告或機密客戶資料

認領表單會詢問敏感證據是否需要透過工作人員的私人管道提供。
請使用該選項，而不要公開張貼敏感資料。

## 可能的處理結果

視證據與風險而定，ClawHub 工作人員可能會保留命名空間、
移轉所有權、重新命名資源、隱藏或隔離現有項目、
新增別名或重新導向、要求提供更多證明，或拒絕要求。

命名空間審查不保證所有相符名稱都會獲得移轉。
工作人員會衡量公開證據、現有使用情況、安全風險及對使用者的影響。
