# 可接受使用規範

本頁說明 ClawHub 接受哪些類型的 Skills 與內容，以及它不會託管哪些濫用工作流程。

這些規則刻意保持務實。我們最重視端對端的濫用工作流程，而不只是孤立的關鍵字。如果某個 Skill 是為了規避防護、濫用平台、詐騙他人、侵犯隱私，或促成未經同意的行為而建置，它就不屬於 ClawHub。

## 我們明確接受的近期模式

- 使用真實元件、語意化 token、可存取狀態，以及經測試使用者流程的前端與設計系統工作。
- 使用已安裝原始碼元件、專案別名與文件化變體的 shadcn/ui 組合，而不是一次性標記。
- 保留註解、使用具體 UI5 型別，並讓產生的控制項介面可審查的 UI5 JavaScript 轉 TypeScript。
- 防禦性安全審查、審核工具，以及顯示證據並清楚保留人工核准界線的濫用偵測提示。
- 以同意為基礎、用於個人或團隊帳號的工作流程自動化，並具備明確憑證、透明設定，以及 dry-run 或預覽模式。
- 文件、遷移作業手冊、開發者工具，以及範圍限定於其支援軟體的測試 fixture。

## 不接受

- 安全繞過或未授權存取工作流程。
  - 範例：驗證繞過、帳號接管、CAPTCHA 繞過、Cloudflare 或反機器人規避、速率限制繞過、設計來擊敗防護的隱匿式抓取、即時通話或代理接管、可重複使用的工作階段竊取、為未核准使用者自動核准配對流程。

- 平台濫用與封鎖規避。
  - 範例：遭封鎖後建立隱匿帳號、帳號養號/農場、假互動、karma 或追蹤者培養、多帳號自動化、大量發文、垃圾訊息機器人、為避免偵測而建置的市集或社群自動化。

- 詐欺、詐騙與欺瞞性金融工作流程。
  - 範例：假證書、假發票、欺瞞性付款流程、詐騙外聯、假社會證明、可在沒有清楚人工核准與透明控制下促成花費或收費的工具，或為詐欺建立帳號而建置的合成身分工作流程。

- 侵犯隱私的抓取、資料補強或監控。
  - 範例：大規模抓取聯絡資訊用於垃圾訊息、公開個人隱私資訊、跟蹤、將名單擷取與未邀約外聯結合、祕密監控、未經明確同意使用的人臉搜尋或生物特徵比對，或購買、發布、下載或操作化外洩資料或資料外洩 dump。

- 未經同意的冒充或欺瞞性身分操控。
  - 範例：換臉、數位分身、假人設、複製網紅，或其他用於冒充或誤導的身分操控工具。

- 露骨性內容與停用安全機制的成人內容生成。
  - 範例：NSFW 圖像/影片/內容生成、包裝第三方 API 的成人內容 wrapper，或主要目的為露骨性內容的 Skills。

- 隱藏、不安全或誤導性的執行需求。
  - 範例：混淆過的安裝命令、`curl | sh`、未宣告的秘密需求、未宣告的私密金鑰使用、缺乏清楚可審查性的遠端 `npx @latest` 執行、隱藏 Skill 真正執行需求的誤導性中繼資料。

## 我們明確不接受的近期模式

- 「在市集封鎖後建立隱匿賣家帳號。」
- 「修改 Telegram 配對，讓未核准使用者自動收到配對代碼。」
- 「以不可偵測的自動化培養 Reddit/Twitter 帳號。」
- 「產生可任意使用的專業證書或發票。」
- 「在停用安全檢查的情況下產生 NSFW 內容。」
- 「抓取名單、補強聯絡資訊，並大規模發起冷外聯。」
- 「購買、發布或下載外洩資料或資料外洩 dump。」
- 「使用合成身分或 CAPTCHA 解題大量建立電子郵件或社群帳號。」

## 審查者注意事項

- 脈絡很重要。同一主題在狹義的防禦性或同意式情境中可能是正當的，但若被包裝成濫用工作流程就不可接受。
- 當某個 Skill 明顯針對規避、欺瞞或未經同意的使用進行最佳化時，我們應偏向採取行動。
- 在這些類別中重複上傳，是隱藏內容與封鎖帳號的理由。

## 執行

- 我們可能會隱藏、移除或硬刪除違規 Skills。
- 我們可能會撤銷 token、軟刪除相關內容，並封鎖重複或嚴重違規者。
- 對於明顯濫用，我們不保證會先警告再執行處置。
