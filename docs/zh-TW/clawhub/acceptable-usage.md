# 可接受使用

本頁說明 ClawHub 可接受哪些類型的 Skills 與內容，以及它不會代管哪些濫用工作流程。

這些規則刻意保持實務導向。我們最重視端到端的濫用工作流程，而不只是孤立的關鍵字。如果某項 Skills 是為了規避防護、濫用平台、詐騙他人、侵犯隱私，或促成未經同意的行為而建立，它就不適合出現在 ClawHub 上。

## 我們明確接受的近期模式

- 使用真實元件、語意權杖、可存取狀態，以及經測試使用者流程的前端與設計系統工作。
- 使用已安裝原始碼元件、專案別名與已文件化變體的 shadcn/ui 組合，而不是一次性標記。
- UI5 JavaScript 到 TypeScript 的轉換，保留註解、使用具體 UI5 型別，並讓產生的控制項介面保持可審查。
- 防禦性安全審查、審核工具，以及濫用偵測提示，能呈現證據並清楚維持人類核准界線。
- 針對個人或團隊帳戶、基於同意的工作流程自動化，具備明確憑證、透明設定，以及 dry-run 或預覽模式。
- 文件、遷移操作手冊、開發者公用工具，以及限縮於其支援軟體範圍內的測試夾具。

## 不可接受

- 安全繞過或未授權存取工作流程。
  - 範例：驗證繞過、帳戶接管、CAPTCHA 繞過、Cloudflare 或反機器人規避、速率限制繞過、設計用來擊敗防護的隱蔽爬取、即時通話或代理接管、可重複使用的工作階段竊取、為未核准使用者自動核准配對流程。

- 平台濫用與封禁規避。
  - 範例：封禁後建立隱蔽帳戶、帳戶暖號/養號、假互動、karma 或追蹤者培養、多帳戶自動化、大量發文、垃圾訊息機器人、為避免偵測而建立的市集或社群自動化。

- 詐欺、詐騙與欺瞞性金融工作流程。
  - 範例：假證書、假發票、欺瞞性付款流程、詐騙式外聯、假社會證明、在沒有清楚人類核准與透明控制的情況下促成支出或收費的工具，或為詐欺建立帳戶而設計的合成身分工作流程。

- 侵犯隱私的爬取、強化或監控。
  - 範例：大規模爬取聯絡資料以發送垃圾訊息、公開個資、跟蹤、搭配未經請求外聯的潛在客戶擷取、隱蔽監控、未經明確同意使用的人臉搜尋或生物特徵比對，或購買、發布、下載或操作化外洩資料或資料外洩檔案集。

- 未經同意的冒充或欺瞞性身分操弄。
  - 範例：換臉、數位分身、假人格、複製網紅，或其他用於冒充或誤導的身分操弄工具。

- 明示性內容與停用安全機制的成人內容生成。
  - 範例：NSFW 圖片/影片/內容生成、第三方 API 周邊的成人內容包裝器，或主要目的為明示性內容的 Skills。

- 隱藏、不安全或誤導性的執行需求。
  - 範例：混淆的安裝命令、`curl | sh`、未宣告的祕密需求、未宣告的私鑰使用、缺乏清楚可審查性的遠端 `npx @latest` 執行、隱藏 Skills 真正執行需求的誤導性中繼資料。

## 我們明確不接受的近期模式

- 「在市集封禁後建立隱蔽賣家帳戶。」
- 「修改 Telegram 配對，讓未核准使用者自動收到配對代碼。」
- 「用無法偵測的自動化培養 Reddit/Twitter 帳戶。」
- 「產生可任意使用的專業證書或發票。」
- 「在停用安全檢查的情況下產生 NSFW 內容。」
- 「爬取潛在客戶、強化聯絡人，並大規模啟動冷外聯。」
- 「購買、發布或下載外洩資料或資料外洩檔案集。」
- 「使用合成身分或 CAPTCHA 解題大量建立電子郵件或社群帳戶。」

## 給審查者的注意事項

- 情境很重要。同一主題在狹義的防禦性或基於同意的設定中可能是正當的，但在包裝成濫用工作流程時則不可接受。
- 當某項 Skills 明顯針對規避、欺瞞或未經同意的使用進行最佳化時，我們應偏向採取行動。
- 在這些類別中重複上傳，是隱藏內容並封禁帳戶的理由。

## 執法

- 我們可能會隱藏、移除或硬刪除違規 Skills。
- 我們可能會撤銷權杖、軟刪除相關內容，並封禁重複或嚴重違規者。
- 對於明顯濫用，我們不保證會先警告再執法。
