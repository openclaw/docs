# 可接受使用

本頁說明 ClawHub 可接受的 Skills 與內容類型，以及它不會託管的濫用工作流程。

這些規則刻意保持務實。我們最在意的是端到端的濫用工作流程，而不只是孤立的關鍵字。如果某個 Skill 是為了規避防護、濫用平台、詐騙他人、侵犯隱私，或促成未經同意的行為而建置，它就不屬於 ClawHub。

## 我們近期明確接受的模式

- 使用真實元件、語意 token、無障礙狀態，以及經測試使用者流程的前端與設計系統工作。
- 使用已安裝原始碼元件、專案別名與文件化變體，而非一次性標記的 shadcn/ui 組合。
- 保留註解、使用具體 UI5 型別，並讓產生的控制介面可供審查的 UI5 JavaScript 轉 TypeScript。
- 防禦性安全審查、審核工具，以及顯示證據並清楚保留人工核准邊界的濫用偵測提示。
- 針對個人或團隊帳號、具備明確認證資訊、透明設定，以及 dry-run 或預覽模式的同意式工作流程自動化。
- 文件、遷移 runbook、開發者工具，以及範圍限於其支援軟體的測試 fixture。

## 不可接受

- 安全繞過或未授權存取工作流程。
  - 範例：驗證繞過、帳號接管、CAPTCHA 繞過、Cloudflare 或反 bot 規避、速率限制繞過、旨在擊敗保護機制的隱匿爬取、即時通話或代理接管、可重複使用的工作階段竊取、為未核准使用者自動核准配對流程。

- 平台濫用與封鎖規避。
  - 範例：被封鎖後建立隱匿帳號、帳號暖機/養號、假互動、karma 或追蹤者培養、多帳號自動化、大量發文、垃圾訊息 bot、為避免偵測而建置的市集或社群自動化。

- 詐欺、詐騙與欺騙性金融工作流程。
  - 範例：偽造證書、偽造發票、欺騙性付款流程、詐騙推廣、假社會證明、在缺乏明確人工核准與透明控制下促成消費或收款的工具，或為詐欺建立帳號而建置的合成身分工作流程。

- 侵犯隱私的爬取、資料增補或監控。
  - 範例：大規模爬取聯絡資訊用於垃圾訊息、doxxing、跟蹤、搭配未經請求外聯的潛在客戶擷取、隱蔽監控、未經明確同意使用的人臉搜尋或生物辨識比對，或購買、發布、下載或操作化外洩資料或外洩資料庫。

- 未經同意的冒充或欺騙性身分操弄。
  - 範例：換臉、數位分身、假人設、複製網紅，或其他用於冒充或誤導的身分操弄工具。

- 明確性內容與停用安全機制的成人內容生成。
  - 範例：NSFW 圖像/影片/內容生成、包裝第三方 API 的成人內容 wrapper，或主要目的為明確性內容的 Skills。

- 隱藏、不安全或誤導性的執行需求。
  - 範例：混淆的安裝指令、`curl | sh`、未宣告的 secret 需求、未宣告的私鑰使用、缺乏清楚可審查性的遠端 `npx @latest` 執行、隱藏 Skill 真正執行需求的誤導性 metadata。

## 我們近期明確不接受的模式

- 「在市集封鎖後建立隱匿賣家帳號。」
- 「修改 Telegram 配對，讓未核准使用者自動收到配對碼。」
- 「用不可偵測的自動化培養 Reddit/Twitter 帳號。」
- 「產生可任意使用的專業證書或發票。」
- 「在停用安全檢查的情況下產生 NSFW 內容。」
- 「爬取潛在客戶、增補聯絡人，並大規模發起冷外聯。」
- 「購買、發布或下載外洩資料或外洩資料庫。」
- 「使用合成身分或 CAPTCHA 解題大量建立電子郵件或社群帳號。」

## 給審查者的注意事項

- 脈絡很重要。同一主題在狹窄的防禦性或同意式情境中可能是正當的，但在包裝成濫用工作流程時則不可接受。
- 當某個 Skill 明顯針對規避、欺騙或未經同意的使用最佳化時，我們應傾向採取行動。
- 重複上傳這些類別的內容，可作為隱藏內容並封鎖帳號的理由。

## 執行

- 我們可能隱藏、移除或硬刪除違規 Skills。
- 我們可能撤銷 token、軟刪除相關內容，並封鎖重複或嚴重違規者。
- 對於明顯濫用，我們不保證會先警告再執行處置。
