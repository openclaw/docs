GitHub Copilot — ИИ-помощник GitHub для написания кода. Он предоставляет доступ к моделям Copilot
для вашей учетной записи и тарифного плана GitHub. OpenClaw может использовать Copilot как
поставщика моделей или среду выполнения агентов тремя разными способами.

## Три способа использования Copilot в OpenClaw

<Tabs>
  <Tab title="Встроенный поставщик (github-copilot)">
    Используйте встроенный вход с устройства, чтобы получить токен GitHub, который затем
    при работе OpenClaw обменивается на токены API Copilot. Это **стандартный** и самый простой
    способ, поскольку он не требует VS Code.

    <Steps>
      <Step title="Выполните команду входа">
        ```bash
        openclaw models auth login-github-copilot
        ```

        Вам будет предложено перейти по URL-адресу и ввести одноразовый код. Не закрывайте
        терминал до завершения процесса.
      </Step>
      <Step title="Задайте модель по умолчанию">
        ```bash
        openclaw models set github-copilot/claude-opus-4.7
        ```

        Или в конфигурации:

        ```json5
        {
          agents: {
            defaults: { model: { primary: "github-copilot/claude-opus-4.7" } },
          },
        }
        ```
      </Step>
    </Steps>

  </Tab>

  <Tab title="Plugin среды выполнения Copilot SDK (copilot)">
    Установите внешний plugin `@openclaw/copilot`, если хотите, чтобы низкоуровневым
    циклом агента для выбранных моделей `github-copilot/*` управляли CLI и SDK
    Copilot от GitHub.

    ```bash
    openclaw plugins install @openclaw/copilot
    ```

    Затем включите эту среду выполнения для модели или поставщика:

    ```json5
    {
      agents: {
        defaults: {
          model: "github-copilot/gpt-5.5",
          models: {
            "github-copilot/gpt-5.5": {
              agentRuntime: { id: "copilot" },
            },
          },
        },
      },
    }
    ```

    Выберите этот вариант, если вам нужны собственные сеансы Copilot CLI,
    управляемое SDK состояние потоков и выполняемая Copilot компактизация для этих
    ходов агента. Без явного включения `agentRuntime` модели `github-copilot/*`
    продолжат использовать встроенного поставщика. Полное описание контракта среды
    выполнения см. в разделе [Среда выполнения Copilot SDK](/ru/plugins/copilot).

  </Tab>

  <Tab title="Plugin Copilot Proxy (copilot-proxy)">
    Используйте расширение VS Code **Copilot Proxy** как локальный мост. OpenClaw
    обращается к конечной точке прокси `/v1` (по умолчанию
    `http://localhost:3000/v1`) и использует настроенный вами список моделей.

    Plugin `copilot-proxy` поставляется вместе с OpenClaw и включен по умолчанию.
    Настройте базовый URL-адрес и идентификаторы моделей с помощью команды:

    ```bash
    openclaw models auth login --provider copilot-proxy --set-default
    ```

    <Note>
    Выберите этот вариант, если Copilot Proxy уже запущен в VS Code или вам нужно
    направлять запросы через него. Расширение VS Code должно оставаться запущенным.
    </Note>

  </Tab>
</Tabs>

## GitHub Enterprise (резидентность данных)

Если ваша организация использует клиентскую среду GitHub Enterprise с
резидентностью данных (узел `*.ghe.com`, например `your-org.ghe.com`), Copilot
работает через локальные конечные точки этой среды, а не через общедоступный
`github.com`. В OpenClaw это доступно как отдельный вариант аутентификации,
поэтому вручную редактировать URL-адреса не требуется.

<Steps>
  <Step title="Выберите вариант аутентификации Enterprise">
    При первоначальной настройке или в `openclaw models auth` выберите
    **GitHub Copilot (Enterprise / data residency)**. Вам будет предложено указать
    домен Enterprise (например, `your-org.ghe.com`), после чего вход с устройства
    будет выполнен для этой клиентской среды.

    Указывайте только корневой домен клиентской среды (`your-org.ghe.com`).
    Производные узлы служб, такие как `api.your-org.ghe.com` или
    `copilot-api.your-org.ghe.com`, не принимаются: OpenClaw автоматически
    формирует эти конечные точки из корневого домена клиентской среды.

    ```bash
    openclaw models auth login --provider github-copilot --method device-enterprise
    ```

  </Step>
  <Step title="Домен сохраняется в конфигурации">
    Выбранный узел сохраняется в параметрах поставщика, поэтому при последующих
    обновлениях токена и генерациях запросы автоматически направляются в
    клиентскую среду:

    ```json5
    {
      models: {
        providers: {
          "github-copilot": { params: { githubDomain: "your-org.ghe.com" } },
        },
      },
    }
    ```

  </Step>
</Steps>

Поток входа с устройства, обмен токена и генерации направляются соответственно
на `https://your-org.ghe.com/login/device/code`,
`https://api.your-org.ghe.com/copilot_internal/v2/token` и
`https://copilot-api.your-org.ghe.com`. Токены среды с резидентностью данных
содержат метку клиентской среды и не содержат указания прокси, поэтому в качестве
базового URL-адреса генераций используется узел Copilot клиентской среды, а не
общедоступная конечная точка.

<Note>
При смене домена вход с устройства всегда выполняется повторно. Если у вас уже
сохранен токен Copilot и вы выбираете другой домен (общедоступный `github.com` ↔
клиентская среда `*.ghe.com` либо переходите из одной клиентской среды в другую),
OpenClaw не будет повторно использовать существующий токен — потребуется новый
вход, чтобы область действия токена соответствовала домену, записываемому в
конфигурацию. При повторном входе для *того же* домена по-прежнему предлагается
использовать текущий токен. При возврате к общедоступному `github.com`
сохраненный параметр `githubDomain` удаляется, и конфигурация возвращается к
значению по умолчанию.
</Note>

<Note>
Переменная среды `COPILOT_GITHUB_DOMAIN` переопределяет определенный домен
для всех путей Copilot, которые его используют: входа с устройства Enterprise
(`--method device-enterprise`), отдельной сокращенной команды
`openclaw models auth login-github-copilot`, обновления токена, векторных
представлений и генераций. Для полностью автоматизированных сред или CI укажите
в ней узел `*.ghe.com`. Чтобы использовать общедоступный `github.com`, не
задавайте эту переменную и не указывайте соответствующий параметр в конфигурации.
При входе сохраняется домен, для которого был выпущен токен (а при входе через
общедоступный `github.com` он удаляется), поэтому маршрутизация остается
корректной даже после удаления переменной среды.
</Note>

## Необязательные флаги

| Команда                                                                | Флаг            | Описание                                                        |
| ---------------------------------------------------------------------- | --------------- | --------------------------------------------------------------- |
| `openclaw models auth login-github-copilot`                            | `--yes`         | Перезаписать существующий профиль аутентификации без подтверждения |
| `openclaw models auth login --provider github-copilot --method device` | `--set-default` | Также применить рекомендуемую поставщиком модель по умолчанию    |

```bash
# Пропустить подтверждение повторного входа
openclaw models auth login-github-copilot --yes

# Выполнить вход и задать модель по умолчанию за один шаг
openclaw models auth login --provider github-copilot --method device --set-default
```

## Неинтерактивная первоначальная настройка

Для входа с устройства требуется интерактивный TTY. Для настройки без
интерактивного терминала импортируйте существующий токен доступа GitHub OAuth с
помощью `openclaw onboard --non-interactive`:

```bash
openclaw onboard --non-interactive --accept-risk \
  --auth-choice github-copilot \
  --github-copilot-token "$COPILOT_GITHUB_TOKEN" \
  --skip-channels --skip-health
```

Также можно не указывать `--auth-choice`: при передаче
`--github-copilot-token` вариант аутентификации поставщика GitHub Copilot
определяется автоматически. Если флаг не указан, первоначальная настройка
поочередно проверяет `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, а затем `GITHUB_TOKEN`.
Чтобы вместо открытого текста в `auth-profiles.json` сохранить `tokenRef`,
ссылающийся на переменную среды, используйте `--secret-input-mode ref` при
заданной переменной `COPILOT_GITHUB_TOKEN`.

<AccordionGroup>
  <Accordion title="Требуется интерактивный TTY">
    Для входа с устройства требуется интерактивный TTY. Запускайте его
    непосредственно в терминале, а не в неинтерактивном скрипте или конвейере CI.
  </Accordion>

  <Accordion title="Доступность моделей зависит от вашего тарифного плана">
    Доступность моделей Copilot зависит от вашего тарифного плана GitHub. Если
    модель отклонена, попробуйте другой идентификатор (например,
    `github-copilot/gpt-5.5`). Актуальный список моделей см. в документации GitHub
    о [поддерживаемых моделях для каждого тарифного плана Copilot](https://docs.github.com/en/copilot/reference/ai-models/supported-models#supported-ai-models-per-copilot-plan).
  </Accordion>

  <Accordion title="Оперативное обновление каталога из API Copilot">
    После того как при входе с устройства (или через переменную среды) получен
    токен GitHub, OpenClaw по запросу обновляет каталог моделей из
    `${baseUrl}/models` (той же конечной точки, которую использует Copilot в
    VS Code), чтобы среда выполнения учитывала права конкретной учетной записи и
    точные размеры контекстного окна без постоянных изменений манифеста. Новые
    опубликованные модели Copilot становятся доступны без обновления OpenClaw, а
    размеры контекстного окна соответствуют фактическим ограничениям каждой
    модели (например, 400 тыс. для серии gpt-5.x и 1 млн для внутренних вариантов
    `claude-opus-*-1m`).

    Встроенный статический каталог остается видимым резервным вариантом, если
    обнаружение отключено, у пользователя нет профиля аутентификации GitHub,
    обмен токена завершается ошибкой или HTTPS-вызов `/models` не удается. Чтобы
    отключить обнаружение и полностью полагаться на статический каталог манифеста
    (в автономных или изолированных от сети средах):

    ```json5
    {
      plugins: {
        entries: {
          "github-copilot": {
            config: { discovery: { enabled: false } },
          },
        },
      },
    }
    ```

  </Accordion>

  <Accordion title="Выбор транспорта">
    Для идентификаторов моделей Claude автоматически используется транспорт
    Anthropic Messages. Модели Gemini используют транспорт OpenAI Chat
    Completions, а модели GPT и серии o — транспорт OpenAI Responses. OpenClaw
    выбирает правильный транспорт на основе ссылки на модель.
  </Accordion>

  <Accordion title="Совместимость запросов">
    OpenClaw отправляет в транспортах Copilot заголовки запросов в стиле IDE
    Copilot (версии редактора и plugin VS Code, а также идентификатор интеграции
    `vscode-chat`), помечает последующие ходы после результатов инструментов как
    инициированные агентом и устанавливает заголовок компьютерного зрения
    Copilot, если ход содержит входное изображение.
  </Accordion>

  <Accordion title="Порядок разрешения переменных среды">
    OpenClaw определяет данные аутентификации Copilot из переменных среды в
    следующем порядке приоритета:

    | Приоритет | Переменная              | Примечания                                      |
    | --------- | ----------------------- | ----------------------------------------------- |
    | 1         | `COPILOT_GITHUB_TOKEN`  | Наивысший приоритет, предназначена для Copilot  |
    | 2         | `GH_TOKEN`              | Токен GitHub CLI (резервный вариант)            |
    | 3         | `GITHUB_TOKEN`          | Стандартный токен GitHub (низший приоритет)     |

    Если заданы несколько переменных, OpenClaw использует переменную с наивысшим
    приоритетом. При входе с устройства
    (`openclaw models auth login-github-copilot`) токен сохраняется в хранилище
    профилей аутентификации и имеет приоритет над всеми переменными среды.

  </Accordion>

  <Accordion title="Хранение токена">
    При входе токен GitHub сохраняется в хранилище профилей аутентификации
    (идентификатор профиля `github-copilot:github`) и при работе OpenClaw
    обменивается на краткосрочный токен API Copilot. Управлять токеном вручную не
    требуется.
  </Accordion>
</AccordionGroup>

## Векторные представления для поиска по памяти

GitHub Copilot также можно использовать как поставщика векторных представлений
для [поиска по памяти](/ru/concepts/memory-search). Если у вас есть подписка Copilot
и вы выполнили вход, OpenClaw может использовать его для создания векторных
представлений без отдельного ключа API.

### Конфигурация

Чтобы использовать векторные представления GitHub Copilot, явно задайте
`memorySearch.provider`. Если доступен токен GitHub, OpenClaw получает список
доступных моделей векторных представлений из API Copilot и автоматически
выбирает лучшую.

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "github-copilot",
        // Необязательно: переопределить автоматически обнаруженную модель
        model: "text-embedding-3-small",
      },
    },
  },
}
```

### Как это работает

1. OpenClaw получает ваш токен GitHub (из переменных среды или профиля аутентификации).
2. Обменивает его на краткосрочный токен API Copilot.
3. Запрашивает конечную точку Copilot `/models`, чтобы обнаружить доступные модели векторных представлений.
4. Выбирает лучшую модель (порядок предпочтения: `text-embedding-3-small`,
   `text-embedding-3-large`, `text-embedding-ada-002`).
5. Отправляет запросы векторных представлений в конечную точку Copilot `/embeddings`.

Доступность моделей зависит от вашего тарифного плана GitHub. Если модели
векторных представлений недоступны, OpenClaw пропускает Copilot и пробует
следующего поставщика.
