OpenClaw использует один идентификатор провайдера, `openai`, как для прямой аутентификации по API-ключу, так и для
аутентификации по подписке ChatGPT/Codex. `openai/*` — канонический маршрут модели.
Для встроенных обращений агента, когда политика среды выполнения не задана или имеет значение `auto`, параметры маршрута OpenAI
определяют, может ли OpenClaw неявно выбрать встроенную среду выполнения сервера приложений Codex.
Сам по себе префикс `openai/*` не выбирает среду выполнения.

- **Модели агентов** — `openai/*` через среду выполнения, выбранную явной
  конфигурацией `agentRuntime` или неявной политикой маршрута OpenAI. Для использования подписки ChatGPT/Codex
  войдите с аутентификацией Codex или настройте профиль аутентификации
  по API-ключу, если требуется тарификация на основе ключа.
- **Неагентские API OpenAI** — прямой доступ к OpenAI Platform с оплатой по факту использования
  через `OPENAI_API_KEY` или профиль аутентификации по API-ключу `openai`.
- **Устаревшая конфигурация** — старые ссылки на модели Codex и идентификаторы профилей исправляются на
  `openai/*` с помощью `openclaw doctor --fix`.

OpenAI явно поддерживает использование OAuth подписки во внешних инструментах и
рабочих процессах, таких как OpenClaw.

## Отслеживание использования и затрат

OpenClaw разделяет квоту подписки и оплату API Platform:

- OAuth ChatGPT/Codex показывает план подписки, окна квот и баланс кредитов.
- `OPENAI_ADMIN_KEY` показывает в разделе **Использование** интерфейса Control UI сообщённые провайдером данные о затратах организации и использовании завершений за 30 дней, включая ежедневные расходы, общее количество запросов и токенов, наиболее используемые модели и категории затрат.
- `OPENAI_PROJECT_ID` при необходимости ограничивает историю Admin API одним проектом.
- OpenClaw никогда не отправляет `OPENAI_API_KEY` или профиль вывода `openai` в API организации; эти учётные данные могут принадлежать пользовательским конечным точкам, Azure или локальным конечным точкам агента.

Явно заданный ключ администратора имеет приоритет над OAuth. Сообщённая провайдером история не объединяется с оценочной стоимостью OpenClaw, рассчитанной на основе сеансов; она может включать активность API других клиентов и корректировки тарификации на стороне провайдера.

Документация OpenAI по [панели использования API](https://help.openai.com/en/articles/10478918) описывает требования к владельцу организации и явно предоставленному разрешению Usage Dashboard для доступа к данным об использовании.

Провайдер, модель, среда выполнения и канал — отдельные уровни. Если эти обозначения
смешиваются, прочитайте раздел [Среды выполнения агентов](/ru/concepts/agent-runtimes), прежде чем
изменять конфигурацию.

## Неявная среда выполнения агента

Когда политика `agentRuntime` для провайдера/модели не задана или имеет значение `auto`, принадлежащая OpenAI
политика маршрута выбирает неявную среду выполнения на основе фактической
конечной точки и адаптера:

| Фактические параметры маршрута                                                                                                                                         | Неявная среда выполнения |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| Точная официальная конечная точка Platform HTTPS с `openai-responses` или точная официальная конечная точка ChatGPT HTTPS с `openai-chatgpt-responses`; без пользовательского переопределения запроса | Может быть выбран Codex |
| Пользовательский адаптер `openai-completions`                                                                                                                            | OpenClaw              |
| Пользовательская конечная точка                                                                                                                                        | OpenClaw              |
| Явно заданная точная официальная конечная точка, использующая HTTP                                                                                                      | Отклонено             |
| Маршрут с пользовательским переопределением запроса провайдера/модели                                                                                                   | OpenClaw              |

Явная нестандартная настройка `agentRuntime.id` провайдера/модели остаётся определяющей.
Например, `agentRuntime.id: "openclaw"` сохраняет для маршрута, который в иных условиях допускает Codex,
среду выполнения OpenClaw, тогда как `agentRuntime.id: "codex"` требует Codex и завершает работу
с ошибкой, если фактический маршрут не объявлен совместимым с Codex.
Выбор среды выполнения не изменяет тип учётных данных или тарификацию: аутентификация
по API-ключу Platform и аутентификация по подписке ChatGPT/Codex остаются раздельными.

`openclaw doctor --fix` переносит устаревшие ссылки на модели Codex, устаревшие идентификаторы
профилей аутентификации Codex и устаревшие записи порядка аутентификации Codex на канонический маршрут `openai`.
Используйте `auth.order.openai` для новой конфигурации порядка аутентификации.

<Note>
При новой настройке OpenAI основная модель GPT-5.6 применяется только в том случае, если
основная модель не настроена. Добавление или обновление аутентификации OpenAI сохраняет существующий явный
выбор, включая `openai/gpt-5.5`, если только вы явно не используете
`models auth login --set-default` или `models set`. Используйте профиль аутентификации по API-ключу,
только если для модели агента требуется аутентификация по API-ключу.
</Note>

## Ограниченная предварительная версия GPT-5.6

OpenClaw распознаёт точные идентификаторы моделей `openai/gpt-5.6-sol`,
`openai/gpt-5.6-terra` и `openai/gpt-5.6-luna`. В текущем каталоге все три предоставляют
рассуждение `xhigh` и `max`. OpenAI описывает Sol как
флагманский уровень, Terra — как сбалансированный, а Luna — как быстрый
и более экономичный уровень. См.
[анонс выпуска GPT-5.6](https://openai.com/index/previewing-gpt-5-6-sol/)
и [руководство по доступу](https://help.openai.com/en/articles/20001325-a-preview-of-gpt-5-6-sol-terra-and-luna).

При прямой аутентификации OpenAI по API-ключу идентификатор `openai/gpt-5.6` без уточнений является псевдонимом
Sol и используется по умолчанию при новой настройке. Нативный каталог Codex не применяет
этот псевдоним прямого API на стороне клиента; в зависимости от доступа рабочего пространства он может отображать
точные идентификаторы Sol, Terra и Luna. Поэтому при новой настройке OAuth ChatGPT/Codex
используется `openai/gpt-5.6-sol`. Проверьте текущую учётную запись командой:

```bash
openclaw models list --provider openai
```

Доступ организации API и рабочего пространства Codex может различаться. Если GPT-5.6
недоступна, явно выберите GPT-5.5:

```bash
openclaw models set openai/gpt-5.5
```

OpenClaw отображает исходную ошибку доступа и не заменяет автоматически
выбранную GPT-5.6 на GPT-5.5.

<Note>
Для подходящих точных официальных маршрутов HTTPS может быть выбран встроенный плагин
сервера приложений Codex, если политика среды выполнения не задана или имеет значение `auto`; пользовательские маршруты Completions,
пользовательские конечные точки и переопределения транспорта запросов продолжают использовать OpenClaw. Официальные конечные точки
с незашифрованным HTTP отклоняются. Явная конфигурация среды выполнения для провайдера/модели остаётся
определяющей. Запустите `openclaw doctor --fix`, чтобы исправить устаревшие ссылки на модели Codex,
ссылки `codex-cli/*` или старые привязки сеансов среды выполнения, которые не были заданы
явной конфигурацией среды выполнения.
</Note>

| Эмбеддинги                | провайдер эмбеддингов памяти                                                                     | Да                                                             |

<Note>
Голосовая связь OpenAI Realtime проходит через общедоступный **OpenAI Platform Realtime
API** и требует ключа API платформы. Токены OAuth Codex вместо этого аутентифицируют
серверную часть ChatGPT Codex; они не взаимозаменяемы с ключами API платформы
для общедоступных конечных точек Realtime.

Если при аутентификации по ключу API сообщается об отсутствии платёжного баланса, пополните кредиты платформы на странице
[platform.openai.com/account/billing](https://platform.openai.com/account/billing)
для организации, к которой относятся ваши учётные данные Realtime, когда используется аутентификация по ключу
API. Голосовая связь Realtime принимает профиль аутентификации по ключу API `openai`, созданный
с помощью `openclaw onboard --auth-choice openai-api-key`, ключ API платформы, заданный через
`talk.realtime.providers.openai.apiKey` для разговора в Control UI, либо
`plugins.entries.voice-call.config.realtime.providers.openai.apiKey` для Voice
Call, либо переменную окружения `OPENAI_API_KEY`.
</Note>

## Эмбеддинги памяти

OpenClaw может использовать OpenAI или совместимую с OpenAI конечную точку эмбеддингов для
индексации `memory_search` и эмбеддингов запросов:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
        model: "text-embedding-3-small",
      },
    },
  },
}
```

Для совместимых с OpenAI конечных точек, которым требуются асимметричные метки эмбеддингов, задайте
`queryInputType` и `documentInputType` в `memorySearch`. OpenClaw
передаёт их как специфичные для провайдера поля запроса `input_type`: для эмбеддингов
запросов используется `queryInputType`; для индексируемых фрагментов памяти и пакетной индексации —
`documentInputType`. Полный пример см. в
[справочнике по настройке памяти](/ru/reference/memory-config#provider-specific-config).

    ### Пример конфигурации

    ```json5
    {
      env: { OPENAI_API_KEY: "example-openai-key-not-real" },
      agents: { defaults: { model: { primary: "openai/gpt-5.6" } } },
    }
    ```

    Идентификатор прямого API `gpt-5.6` без уточнений разрешается в уровень Sol. Если эта организация
    API не предоставляет GPT-5.6, явно задайте основной моделью
    `openai/gpt-5.5`.

    Чтобы попробовать текущую модель Instant из ChatGPT через API OpenAI, задайте модель
    `openai/chat-latest`:

    ```json5
    {
      env: { OPENAI_API_KEY: "example-openai-key-not-real" },
      agents: { defaults: { model: { primary: "openai/chat-latest" } } },
    }
    ```

    `chat-latest` — изменяемый псевдоним. При новой настройке ключа API OpenAI вместо него используется
    `openai/gpt-5.6`, идентификатор прямого API без уточнений для которого разрешается в Sol. Существующие
    явно заданные основные модели, включая `openai/gpt-5.5`, остаются без изменений. Псевдоним
    `chat-latest` принимает только уровень детализации текста `medium`; для этой модели OpenClaw принудительно
    заменяет любой другой запрошенный уровень детализации на `medium`.

    <Warning>
    OpenClaw **не** предоставляет `gpt-5.3-codex-spark` через прямой маршрут
    ключа API OpenAI. Он доступен только через записи каталога подписки Codex,
    если он доступен для вашей учётной записи.
    </Warning>

        Для конфигураций без графического интерфейса или с ограничениями обратного вызова добавьте `--device-code`, чтобы
        войти через поток кода устройства ChatGPT вместо обратного вызова
        локального браузера:

        ```bash
        openclaw models auth login --provider openai --device-code
        ```
      </Step>
      <Step title="Используйте канонический маршрут модели OpenAI">
        ```bash
        openclaw config set agents.defaults.model.primary openai/gpt-5.6-sol
        ```

        Для этого точного официального нативного маршрута HTTPS настройка среды выполнения
        не требуется. Он может автоматически выбрать среду выполнения app-server Codex, а
        OpenClaw установит или восстановит встроенный плагин Codex при выборе этой среды выполнения.
      </Step>
      <Step title="Убедитесь, что аутентификация Codex доступна">
        ```bash
        openclaw models list --provider openai
        ```

        После запуска Gateway отправьте в чате `/codex status` или `/codex models`,
        чтобы проверить нативную среду выполнения app-server.
      </Step>
    </Steps>

    ### Сводка маршрутов

    | Ссылка на модель                | Политика среды выполнения или сведения о маршруте                                 | Маршрут                                                    | Аутентификация                                               |
    | ------------------------ | ------------------------------------------------------------- | -------------------------------------------------------- | -------------------------------------------------- |
    | `openai/gpt-5.6-sol`     | не задано/`auto`, точный официальный нативный маршрут HTTPS, без переопределения запроса | Может быть выбран Codex                                    | Вход в Codex или упорядоченный профиль аутентификации `openai` |
    | `openai/gpt-5.6-terra`   | не задано/`auto`, точный официальный нативный маршрут HTTPS, без переопределения запроса | Может быть выбран Codex                                    | Вход в Codex, если каталог предоставляет Terra       |

    </Warning>

    <Note>
    В новой конфигурации порядок аутентификации агента OpenAI следует указывать в `auth.order.openai`;

    </Note>

    ### Пример конфигурации

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
    }
    ```

    При наличии резервного API-ключа оставьте выбранную модель в `openai/*`, а
    порядок аутентификации укажите в `openai`. OpenClaw сначала использует подписку,
    а затем API-ключ, оставаясь в среде Codex:

    ```json5
    {
      plugins: { entries: { codex: { enabled: true } } },
      agents: {
        defaults: {
          model: { primary: "openai/gpt-5.6-sol" },
        },
      },
      auth: {
        order: {
          openai: [
            "openai:user@example.com",
            "openai:api-key-backup",
          ],
        },
      },
    }
    ```

    <Note>
    Первоначальная настройка больше не импортирует данные OAuth из `~/.codex`. Выполните
    вход через OAuth в браузере (по умолчанию) или с помощью описанного выше потока кода устройства;
    OpenClaw сохраняет полученные учётные данные в собственном хранилище аутентификации агента.
    </Note>

    ### Проверка и восстановление маршрутизации Codex OAuth

    ```bash
    openclaw models status
    openclaw models auth list --provider openai
    openclaw config get agents.defaults.model --json
    openclaw config get models.providers.openai.agentRuntime --json
    ```

    Для конкретного агента добавьте `--agent <id>`:

    ```bash
    openclaw models status --agent <id>
    openclaw models auth list --agent <id> --provider openai
    ```

    Если старая конфигурация всё ещё содержит устаревшие ссылки Codex GPT или закреплённую
    устаревшую сессию среды выполнения OpenAI без явной конфигурации среды выполнения,
    исправьте её:

    ```bash
    openclaw doctor --fix
    openclaw config validate
    ```

    Если `models auth list --provider openai` не показывает ни одного пригодного профиля, выполните
    вход повторно:

    ```bash
    openclaw models auth login --provider openai
    openclaw models status --probe --probe-provider openai
    ```

    Используйте `--profile-id` для нескольких входов Codex OAuth в одном агенте, а затем
    управляйте ими с помощью порядка аутентификации или `/model ...@<profileId>`:

    ```bash
    openclaw models auth login --provider openai --profile-id openai:ritsuko
    openclaw models auth login --provider openai --profile-id openai:lain
    ```

    Выполните `openclaw doctor --fix`, чтобы перенести идентификаторы профилей и записи порядка
    со старым префиксом OpenAI Codex, прежде чем полагаться на порядок профилей.

    ### Индикатор состояния

    Команда чата `/status` показывает, какая среда выполнения модели активна в текущей
    сессии. Встроенная среда сервера приложений Codex отображается как
    `Runtime: OpenAI Codex`, когда её выбирает подходящий неявный маршрут или явная
    политика среды выполнения поставщика/модели.

    ### Предупреждение doctor

    Если в конфигурации или состоянии сессии остаются устаревшие ссылки на модели Codex
    или закрепления среды выполнения OpenAI, `openclaw doctor --fix` заменяет их на `openai/*`
    со средой выполнения Codex, если OpenClaw не настроен явно.

    ### Ограничение контекстного окна

    OpenClaw рассматривает метаданные модели и ограничение контекста среды выполнения как
    отдельные значения. Для `openai/gpt-5.5` из каталога Codex OAuth:

    - Нативный `contextWindow`: `400000`
    - Ограничение `contextTokens` среды выполнения по умолчанию: `272000`

    На практике меньшее ограничение по умолчанию обеспечивает лучшие показатели
    задержки и качества. Переопределите его с помощью `contextTokens`:

    ```json5
    {
      models: {
        providers: {
          openai: {
            models: [{ id: "gpt-5.5", contextTokens: 160000 }],
          },
        },
      },
    }
    ```

    <Note>
    Используйте `contextWindow`, чтобы объявить нативные метаданные модели. Используйте `contextTokens`,
    чтобы ограничить бюджет контекста среды выполнения. Прямой маршрут OpenAI с API-ключом
    сообщает о большем нативном `contextWindow` (`1000000`) для `gpt-5.5`; эти два
    маршрута отслеживаются отдельно, поскольку исходные каталоги различаются.
    </Note>

    ### Восстановление каталога

    OpenClaw использует метаданные исходного каталога Codex для `gpt-5.5`, если они
    доступны. Если при активной аутентификации обнаружение Codex не возвращает строку
    `gpt-5.5`, OpenClaw создаёт эту строку модели OAuth, чтобы запуски Cron,
    субагентов и настроенной модели по умолчанию не завершались ошибкой
    `Unknown model`.

## Аутентификация нативного сервера приложений Codex

Нативная среда сервера приложений Codex использует ссылки на модели `openai/*`, когда её
неявно выбирает подходящий точный официальный маршрут HTTPS или когда её явно выбирает
`agentRuntime.id: "codex"` поставщика/модели. Аутентификация по-прежнему
привязана к учётной записи. OpenClaw выбирает способ аутентификации в следующем порядке:

1. Упорядоченные профили аутентификации OpenAI для агента, предпочтительно в
   `auth.order.openai`. Выполните `openclaw doctor --fix`, чтобы перенести устаревшие
   идентификаторы профилей аутентификации Codex и порядок аутентификации.
2. Существующая учётная запись сервера приложений, например локальный
   вход ChatGPT в Codex CLI. Для изолированного домашнего каталога агента по умолчанию
   OpenClaw подключает эту нативную учётную запись CLI к серверу приложений через RPC входа;
   конфигурация, плагины и хранилище цепочек CLI при этом не используются совместно.
3. Только для локальных запусков сервера приложений через stdio и только когда
   сервер приложений сообщает об отсутствии учётной записи: `CODEX_API_KEY`, затем `OPENAI_API_KEY`.

Локальный вход по подписке ChatGPT/Codex не заменяется только потому, что
процесс Gateway также содержит `OPENAI_API_KEY` для прямых моделей OpenAI или
векторных представлений. Резервный API-ключ из переменной окружения применяется только
в локальном пути stdio без учётной записи; он никогда не отправляется через соединения
с сервером приложений по WebSocket. Когда выбран профиль Codex на основе подписки,
OpenClaw также исключает `CODEX_API_KEY` и `OPENAI_API_KEY` из окружения
запускаемого дочернего процесса сервера приложений stdio и вместо этого передаёт
выбранные учётные данные через RPC входа сервера приложений.

Когда этот профиль подписки блокируется из-за ограничения использования Codex,
OpenClaw отмечает профиль заблокированным до указанного Codex времени сброса и
позволяет порядку аутентификации перейти к следующему профилю `openai:*`,
не изменяя выбранную модель и не выходя из среды Codex. После наступления времени
сброса профиль подписки снова становится доступным.

## Генерация изображений

Встроенный плагин `openai` регистрирует генерацию изображений через
инструмент `image_generate`. Он поддерживает генерацию изображений как с API-ключом
OpenAI, так и через Codex OAuth, используя одну и ту же ссылку на модель `openai/gpt-image-2`.

| Возможность               | API-ключ OpenAI                     | Codex OAuth                          |
| ------------------------- | ---------------------------------- | ------------------------------------ |
| Ссылка на модель          | `openai/gpt-image-2`               | `openai/gpt-image-2`                 |
| Аутентификация            | `OPENAI_API_KEY`                   | Вход через OpenAI Codex OAuth         |
| Транспорт                 | OpenAI Images API                  | Серверная часть Codex Responses       |
| Макс. изображений на запрос | 4                                  | 4                                    |
| Режим редактирования      | Включён (до 5 эталонных изображений) | Включён (до 5 эталонных изображений) |
| Переопределение размера   | Поддерживается, включая размеры 2K/4K | Поддерживается, включая размеры 2K/4K |
| Соотношение сторон / разрешение | Не передаётся в OpenAI Images API | По возможности сопоставляется с поддерживаемым размером |

```json5
{
  agents: {
    defaults: {
      imageGenerationModel: { primary: "openai/gpt-image-2" },
    },
  },
}
```

<Note>
Общие параметры инструмента, выбор поставщика и поведение при переключении после сбоя
описаны в разделе [Генерация изображений](/ru/tools/image-generation).
</Note>

`gpt-image-2` используется по умолчанию для генерации изображений по тексту и
редактирования изображений в OpenAI. `gpt-image-1.5`, `gpt-image-1` и `gpt-image-1-mini`
по-прежнему можно использовать как явные переопределения модели. Используйте
`openai/gpt-image-1.5` для вывода PNG/WebP с прозрачным фоном; текущий API
`gpt-image-2` отклоняет `background: "transparent"`.

Для запроса с прозрачным фоном вызовите `image_generate` с
`model: "openai/gpt-image-1.5"`, `outputFormat: "png"` или `"webp"`, а также
`background: "transparent"`; более старый параметр поставщика `openai.background`
по-прежнему принимается. OpenClaw также защищает общедоступные маршруты OpenAI
и OpenAI Codex OAuth, заменяя прозрачные запросы с `openai/gpt-image-2`
по умолчанию на `gpt-image-1.5`; Azure и пользовательские конечные точки,
совместимые с OpenAI, сохраняют настроенные имена развёртываний/моделей.

Та же настройка доступна для запусков CLI без интерфейса:

```bash
openclaw infer image generate \
  --model openai/gpt-image-1.5 \
  --output-format png \
  --background transparent \
  --prompt "Простая наклейка с красным кругом на прозрачном фоне" \
  --json
```

При работе с входным файлом используйте те же флаги `--output-format` и
`--background` вместе с `openclaw infer image edit`.
`--openai-background` остаётся доступным как псевдоним, специфичный для OpenAI. Используйте
`--quality low|medium|high|auto`, чтобы управлять качеством и стоимостью OpenAI Images.
Используйте `--openai-moderation low|auto`, чтобы передать подсказку модерации OpenAI из
`image generate` или `image edit`.

Для установок с ChatGPT/Codex OAuth сохраните ту же ссылку `openai/gpt-image-2`. Когда
настроен профиль OAuth `openai`, OpenClaw получает сохранённый токен доступа OAuth
и отправляет запросы изображений через серверную часть Codex Responses; он не пытается
сначала использовать `OPENAI_API_KEY` и не переключается без уведомления на API-ключ.
Явно настройте `models.providers.openai` с API-ключом, пользовательским базовым
URL или конечной точкой Azure, если вместо этого требуется прямой маршрут OpenAI Images API.
Если эта пользовательская конечная точка изображений находится в доверенной локальной
сети или по частному адресу, также задайте `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork: true`; OpenClaw
блокирует частные и внутренние конечные точки изображений, совместимые с OpenAI,
если это явное разрешение отсутствует.

Генерация:

```
/tool image_generate model=openai/gpt-image-2 prompt="Профессиональный постер запуска OpenClaw в macOS" size=3840x2160 count=1
```

Генерация прозрачного PNG:

```
/tool image_generate model=openai/gpt-image-1.5 prompt="Простая наклейка с красным кругом на прозрачном фоне" outputFormat=png background=transparent
```

Редактирование:

```
/tool image_generate model=openai/gpt-image-2 prompt="Сохранить форму объекта, изменить материал на полупрозрачное стекло" image=/path/to/reference.png size=1024x1536
```

## Генерация видео

Встроенный плагин `openai` регистрирует генерацию видео через
инструмент `video_generate`.

| Возможность       | Значение                                                                           |
| ---------------- | ---------------------------------------------------------------------------------- |
| Модель по умолчанию | `openai/sora-2`                                                                    |
| Режимы           | Текст в видео, изображение в видео, редактирование одного видео                    |
| Входные эталонные материалы | 1 изображение или 1 видео                                                         |
| Переопределение размера | Поддерживается для преобразования текста и изображения в видео                    |
| Соотношение сторон | Преобразуется в ближайший поддерживаемый размер, без передачи исходного значения  |
| Другие переопределения | `resolution`, `audio`, `watermark` не поддерживаются и отбрасываются с предупреждением инструмента |

Запросы OpenAI на преобразование изображения в видео используют `POST /v1/videos` с изображением
`input_reference`. Для редактирования одного видео используется `POST /v1/videos/edits`, а
загруженное видео передаётся в поле `video`.

```json5
{
  agents: {
    defaults: {
      videoGenerationModel: { primary: "openai/sora-2" },
    },
  },
}
```

<Note>
Общие параметры инструмента, выбор провайдера и поведение при переключении после сбоя описаны в разделе [Генерация видео](/ru/tools/video-generation).

Провайдер OpenAI объявляет `supportsSize`, но не `supportsAspectRatio` и
`supportsResolution`. Общий слой нормализации OpenClaw преобразует запрошенное
значение `aspectRatio` в наиболее близкое соответствующее значение OpenAI `size` до того, как
запрос поступит провайдеру, поэтому запросы с соотношением сторон обычно продолжают работать.
Для `resolution` нет резервного значения размера, поэтому параметр отбрасывается, а вызывающей стороне
передаётся `Ignored unsupported overrides for openai/<model>: resolution=<value>`.
</Note>

## Дополнение к промпту GPT-5

OpenClaw добавляет общее дополнение к промпту GPT-5 для моделей семейства GPT-5 у
провайдера `openai` (включая устаревшие ссылки Codex до исправления, которые нормализуются
в `openai/*`). Другие провайдеры, также предоставляющие идентификаторы моделей семейства GPT-5,
например маршруты OpenRouter или opencode, не получают это наложение: оно зависит от
идентификатора провайдера `openai`, а не только от идентификатора модели. Более старые модели GPT-4.x
никогда его не получают.

Нативная обвязка app-server Codex не получает через инструкции разработчика
контракт поведения для персоны и дисциплины использования инструментов или дружественное наложение
стиля взаимодействия; нативный Codex сохраняет собственное базовое поведение Codex, поведение модели и
документации проекта, а OpenClaw отключает встроенную личность Codex для
нативных потоков, чтобы файлы личности в рабочем пространстве агента оставались главным источником.
OpenClaw добавляет в нативные потоки Codex только контекст среды выполнения: доставку
по каналам, динамические инструменты OpenClaw, делегирование ACP, контекст рабочего пространства и
Skills OpenClaw. Текст рекомендаций по Heartbeat из этого же дополнения —
единственное исключение: нативные циклы Heartbeat Codex получают его в виде отдельных
инструкций по совместной работе, а не через общий механизм
дополнения к промпту.

Дополнение GPT-5 добавляет размеченный контракт поведения для сохранения
персоны, безопасности выполнения, дисциплины использования инструментов, формата вывода, проверок
завершения и верификации в соответствующих промптах, сформированных OpenClaw. Зависящее от канала
поведение ответов и беззвучных сообщений остаётся в общем системном промпте OpenClaw
и политике исходящей доставки. Дружественный слой стиля взаимодействия
настраивается отдельно.

| Значение                  | Эффект                                      |
| ---------------------- | ------------------------------------------- |
| `"friendly"` (по умолчанию) | Включить дружественный слой стиля взаимодействия |
| `"on"`                 | Псевдоним для `"friendly"`                      |
| `"off"`                | Отключить только дружественный слой стиля       |

<Tabs>
  <Tab title="Конфигурация">
    ```json5
    {
      agents: {
        defaults: {
          promptOverlays: {
            gpt5: { personality: "friendly" },
          },
        },
      },
    }
    ```
  </Tab>
  <Tab title="CLI">
    ```bash
    openclaw config set agents.defaults.promptOverlays.gpt5.personality off
    ```
  </Tab>
</Tabs>

<Tip>
Регистр значений во время выполнения не учитывается, поэтому и `"Off"`, и `"off"` отключают
дружественный слой стиля.
</Tip>

<Note>
Устаревший параметр `plugins.entries.openai.config.personality` по-прежнему считывается как
резервный вариант совместимости, если общий параметр
`agents.defaults.promptOverlays.gpt5.personality` не задан.
</Note>

    ```json5
        },
      },
    }
    ```

    с помощью ключа API OpenAI Platform; установки только с OAuth могут по-прежнему использовать
    модели чата на базе Codex, но не голосовой ответ OpenAI в реальном времени.
    </Note>

    - Модель по умолчанию: `gpt-4o-transcribe`

    ```json5
    {
      tools: {
        media: {
          audio: {
            models: [
              {
                type: "provider",
                provider: "openai",
                model: "gpt-4o-transcribe",
              },
            ],
          },
        },
      },
    }
    ```

    `ballad`, `coral`, `echo`, `sage`, `shimmer`, `verse`, `marin`, `cedar`.

    отдельный набор, не совпадающий с указанными выше голосами преобразования текста в речь; голос, предназначенный только для TTS,
    например `fable`, `nova` или `onyx`, недопустим для сеансов Realtime.

    **GPT-Live (скоро).** Полнодуплексные модели OpenAI `gpt-live-1` и
    `gpt-live-1-mini` заменили голосовой режим ChatGPT в июле 2026 года; API для
    разработчиков постепенно становится доступен организациям с ранним доступом. OpenClaw

    только через WebRTC, самостоятельно управляют очерёдностью реплик (без VAD) и делегируют работу агента
    через протокол событий передачи, который транспорты Realtime OpenClaw
    пока не реализуют. Настройка модели `gpt-live-*` приводит к безопасному отказу с
    рекомендациями как для моста WebSocket, так и для браузерных сеансов Talk вместо
    неявного подключения звука без доступа к агенту. Доступ к API в период раннего доступа также
    предоставляется отдельно каждой организации OpenAI. Сохраняйте `gpt-realtime-2.1` (значение
    по умолчанию), пока не появится поддержка GPT-Live.
    </Note>

    <Note>
    Серверные мосты OpenAI Realtime используют общедоступный формат сеанса Realtime WebSocket,
    который не принимает `session.temperature`. Развёртывания Azure OpenAI
    остаются доступными через `azureEndpoint` и `azureDeployment` и
    сохраняют совместимый с развёртыванием формат сеанса (включая `temperature`).
    Поддерживаются двунаправленные вызовы инструментов и звук G.711 u-law.
    </Note>

    <Note>
    Голос для режима реального времени выбирается при создании сеанса. OpenAI позволяет позднее
    изменять большинство полей сеанса, но голос нельзя изменить после того, как
    модель выдала аудио в этом сеансе. В настоящее время OpenClaw предоставляет
    идентификаторы встроенных голосов режима реального времени в виде строк.
    </Note>

    <Note>
    Функция Talk в Control UI использует браузерные сеансы OpenAI в режиме реального времени с созданным

    Интерактивная проверка для сопровождающих доступна с помощью
    `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts`;
    этапы OpenAI проверяют как серверный мост WebSocket, так и браузерный

    Передайте `--openai-only`, чтобы выполнить эти два этапа без учётных данных Google.
    </Note>

## Конечные точки Azure OpenAI

Встроенный провайдер `openai` может использовать ресурс Azure OpenAI для генерации
изображений путём переопределения базового URL. На пути генерации изображений OpenClaw
обнаруживает имена хостов Azure в `models.providers.openai.baseUrl` и автоматически переключается на
формат запросов Azure.

<Note>
Голос в режиме реального времени использует отдельный путь конфигурации
(`plugins.entries.voice-call.config.realtime.providers.openai.azureEndpoint`)
и не зависит от `models.providers.openai.baseUrl`. Параметры Azure см. в разделе **Голос
в режиме реального времени** внутри аккордеона [Голос и речь](#voice-and-speech).
</Note>

Используйте Azure OpenAI, если:

- У вас уже есть подписка Azure OpenAI, квота или корпоративное
  соглашение
- Вам нужны региональное хранение данных или средства контроля соответствия требованиям, предоставляемые Azure
- Вы хотите сохранить трафик внутри существующего клиента Azure

### Конфигурация

Для генерации изображений Azure через встроенный провайдер `openai` направьте
`models.providers.openai.baseUrl` на свой ресурс Azure и задайте в `apiKey`
ключ Azure OpenAI (не ключ OpenAI Platform):

```json5
{
  models: {
    providers: {
      openai: {
        baseUrl: "https://<your-resource>.openai.azure.com",
        apiKey: "<azure-openai-api-key>",
      },
    },
  },
}
```

OpenClaw распознаёт следующие суффиксы хостов Azure для маршрута генерации
изображений Azure:

- `*.openai.azure.com`
- `*.services.ai.azure.com`
- `*.cognitiveservices.azure.com`

Для запросов генерации изображений на распознанном хосте Azure OpenClaw:

- Отправляет заголовок `api-key` вместо `Authorization: Bearer`
- Использует пути, привязанные к развёртыванию (`/openai/deployments/{deployment}/...`)
- Добавляет `?api-version=...` к каждому запросу
- Использует тайм-аут запроса по умолчанию 600 с для вызовов генерации изображений Azure.
  Значения `timeoutMs` отдельных вызовов по-прежнему переопределяют это значение по умолчанию.

Другие базовые URL (общедоступный OpenAI, прокси-серверы, совместимые с OpenAI) сохраняют стандартный
формат запросов изображений OpenAI.

<Note>
Для маршрутизации Azure на пути генерации изображений провайдера `openai`
требуется OpenClaw 2026.4.22 или новее. Более ранние версии обрабатывают любой пользовательский
`openai.baseUrl` как общедоступную конечную точку OpenAI, поэтому запросы к развёртываниям изображений Azure
завершаются ошибкой.
</Note>

### Версия API

Задайте `AZURE_OPENAI_API_VERSION`, чтобы закрепить конкретную предварительную или общедоступную версию Azure
для пути генерации изображений Azure:

```bash
export AZURE_OPENAI_API_VERSION="2024-12-01-preview"
```

Если переменная не задана, по умолчанию используется `2024-12-01-preview`.

### Имена моделей — это имена развёртываний

Azure OpenAI связывает модели с развёртываниями. Для запросов генерации изображений Azure,
маршрутизируемых через встроенный провайдер `openai`, поле `model` в OpenClaw
должно содержать **имя развёртывания Azure**, настроенное на портале Azure, а не
идентификатор общедоступной модели OpenAI.

Если вы создадите развёртывание с именем `gpt-image-2-prod`, обслуживающее `gpt-image-2`:

```
/tool image_generate model=openai/gpt-image-2-prod prompt="Чистый плакат" size=1024x1024 count=1
```

То же правило имени развёртывания применяется к любому вызову генерации изображений, маршрутизируемому
через встроенный провайдер `openai`.

### Доступность по регионам

В настоящее время генерация изображений Azure доступна только в некоторых регионах
(например, `eastus2`, `swedencentral`, `polandcentral`, `westus3`,
`uaenorth`). Перед созданием развёртывания проверьте актуальный список регионов Microsoft
и убедитесь, что нужная модель доступна в вашем регионе.

### Различия параметров

Azure OpenAI и общедоступный OpenAI не всегда принимают одинаковые параметры изображений.
Azure может отклонять параметры, разрешённые общедоступным OpenAI (например, некоторые
значения `background` для `gpt-image-2`), или предоставлять их только в определённых версиях
моделей. Эти различия обусловлены Azure и базовой моделью, а не
OpenClaw. Если запрос Azure завершается ошибкой проверки, проверьте на портале
Azure набор параметров, поддерживаемых конкретным развёртыванием и версией API.

<Note>
Azure OpenAI использует нативный транспорт и поведение совместимости, но не получает
скрытые заголовки атрибуции OpenClaw — см. раздел **Нативные и совместимые с OpenAI
маршруты** в аккордеоне [Расширенная конфигурация](#advanced-configuration).

Для трафика чата или Responses в Azure (помимо генерации изображений) используйте
процесс первоначальной настройки или отдельную конфигурацию провайдера Azure; одного `openai.baseUrl`
недостаточно для применения формата API и аутентификации Azure. Существует отдельный
провайдер `azure-openai-responses/*`; см. аккордеон серверного Compaction
ниже.
</Note>

## Расширенная конфигурация

Приведённые ниже примеры `params` для отдельных моделей формируют встроенный запрос
провайдера OpenClaw. Их настройка считается явно заданным поведением запроса, поэтому маршрут
`auto`, который в иных условиях подходит, остаётся в OpenClaw вместо неявного выбора Codex. Нативная
среда app-server Codex управляет собственным транспортом и параметрами запросов; явный
`agentRuntime.id: "codex"` завершается отказом, если действующий маршрут не объявлен
совместимым с Codex.

<AccordionGroup>
  <Accordion title="Транспорт (WebSocket или SSE)">
    OpenClaw использует WebSocket в первую очередь и SSE как резервный вариант (`"auto"`) для `openai/*`.

    В режиме `"auto"` OpenClaw:
    - Повторяет одну раннюю неудачную попытку WebSocket перед переходом на SSE
    - После сбоя помечает WebSocket как деградировавший на 60 секунд и использует SSE
      в период ожидания
    - Добавляет стабильные заголовки идентификации сеанса и хода для повторных попыток и
      переподключений
    - Нормализует счётчики использования (`input_tokens` / `prompt_tokens`) между
      вариантами транспорта

    | Значение                | Поведение                          |
    | ---------------------- | ------------------------------------ |
    | `"auto"` (по умолчанию)   | Сначала WebSocket, затем резервный SSE     |
    | `"sse"`              | Использовать только SSE                    |
    | `"websocket"`        | Использовать только WebSocket              |

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": {
              params: { transport: "auto" },
            },
          },
        },
      },
    }
    ```

    Связанная документация OpenAI:
    - [Realtime API с WebSocket](https://platform.openai.com/docs/guides/realtime-websocket)
    - [Потоковая передача ответов API (SSE)](https://platform.openai.com/docs/guides/streaming-responses)

  </Accordion>

  <Accordion title="Быстрый режим">
    OpenClaw предоставляет общий переключатель быстрого режима для `openai/*`:

    - **Чат/UI:** `/fast status|auto|on|off`
    - **Конфигурация:** `agents.defaults.models["<provider>/<model>"].params.fastMode`

    Когда быстрый режим включён, OpenClaw сопоставляет его с приоритетной обработкой OpenAI
    (`service_tier = "priority"`). Существующие значения `service_tier`
    сохраняются, и быстрый режим не перезаписывает `reasoning` или
    `text.verbosity`. `fastMode: "auto"` запускает новые вызовы модели в быстром режиме до
    автоматического порога, а последующие повторные, резервные вызовы, вызовы с результатами инструментов или
    продолжения запускаются без быстрого режима. По умолчанию порог составляет 60 секунд;
    чтобы изменить его, задайте `params.fastAutoOnSeconds` для активной модели.

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { fastMode: "auto", fastAutoOnSeconds: 30 } },
          },
        },
      },
    }
    ```

    <Note>
    Переопределения сеанса имеют приоритет над конфигурацией. Удаление переопределения сеанса в
    интерфейсе Sessions возвращает сеанс к настроенному значению по умолчанию.
    </Note>

  </Accordion>

  <Accordion title="Приоритетная обработка (service_tier)">
    API OpenAI предоставляет приоритетную обработку через `service_tier`. Задайте её отдельно для
    каждой модели в OpenClaw:

    ```json5
    {
      agents: {
        defaults: {
          models: {
            "openai/gpt-5.5": { params: { serviceTier: "priority" } },
          },
        },
      },
    }
    ```

    Поддерживаемые значения: `auto`, `default`, `flex`, `priority`.

    <Warning>
    `serviceTier` передаётся только нативным конечным точкам OpenAI
    (`api.openai.com`) и нативным конечным точкам Codex (`chatgpt.com/backend-api`).
    Если любой из этих провайдеров маршрутизируется через прокси-сервер, OpenClaw оставляет
    `service_tier` без изменений.
    </Warning>

  </Accordion>

  <Accordion title="Серверный Compaction (Responses API)">
    Для моделей прямого OpenAI Responses (`openai/*` в `api.openai.com`)
    потоковая обёртка OpenClaw плагина OpenAI автоматически включает серверный
    Compaction:

    - Принудительно задаёт `store: true` (если совместимость модели не задаёт `supportsStore: false`)
    - Добавляет `context_management: [{ type: "compaction", compact_threshold: ... }]`
    - Значение `compact_threshold` по умолчанию: 70% от `contextWindow` (или `80000`, если
      оно недоступно)

    Это применяется к пути встроенной среды выполнения OpenClaw и к хукам провайдера OpenAI,
    используемым встроенными запусками. Нативная среда app-server Codex управляет
    собственным контекстом через Codex, и этот параметр на неё не влияет.

    <Tabs>
      <Tab title="Включить явно">
        Полезно для совместимых конечных точек, таких как Azure OpenAI Responses:

        ```json5
        {
          agents: {
            defaults: {
              models: {
                "azure-openai-responses/gpt-5.5": {
                  params: { responsesServerCompaction: true },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="Пользовательский порог">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: {
                    responsesServerCompaction: true,
                    responsesCompactThreshold: 120000,
                  },
                },
              },
            },
          },
        }
        ```
      </Tab>
      <Tab title="Отключить">
        ```json5
        {
          agents: {
            defaults: {
              models: {
                "openai/gpt-5.5": {
                  params: { responsesServerCompaction: false },
                },
              },
            },
          },
        }
        ```
      </Tab>
    </Tabs>

    <Note>
    `responsesServerCompaction` управляет только добавлением `context_management`.
    Прямые модели OpenAI Responses всё равно принудительно задают `store: true`, если совместимость
    не задаёт `supportsStore: false`.
    </Note>

  </Accordion>

  <Accordion title="Строгий агентный режим GPT">
    Для моделей семейства GPT-5 провайдера `openai`, запускаемых через встроенную
    среду выполнения OpenClaw, OpenClaw уже по умолчанию использует более строгий контракт выполнения под названием
    `strict-agentic`. Он автоматически активируется, когда разрешённым провайдером является
    `openai`, а идентификатор модели соответствует семейству GPT-5, если конфигурация
    явно не отключает его:

    ```json5
    {
      agents: {
        defaults: {
          embeddedAgent: { executionContract: "default" },
        },
      },
    }
    ```

    Явная установка `"strict-agentic"` не оказывает никакого эффекта в поддерживаемом режиме (это
    уже значение по умолчанию) и не действует для неподдерживаемых пар провайдер/модель.

    Когда `strict-agentic` активен, OpenClaw:
    - Автоматически включает `update_plan` для существенных задач
    - Повторяет структурно пустые ходы или ходы только с рассуждениями, продолжая
      до получения видимого ответа
    - Использует явные события плана среды выполнения, если выбранная среда
      их предоставляет

    OpenClaw не классифицирует текст ассистента, чтобы определить, является ли ход
    планом, обновлением хода выполнения или окончательным ответом.

    <Note>
    Этот контракт полностью реализован во встроенном средстве запуска агентов OpenClaw. Он
    не применяется к нативной среде app-server Codex, которая самостоятельно управляет
    поведением ходов и планов; для нативных запусков Codex выбор среды выполнения важнее,
    чем настройка контракта выполнения.
    </Note>

  </Accordion>

  <Accordion title="Нативные и OpenAI-совместимые маршруты">
    OpenClaw обрабатывает прямые конечные точки OpenAI, Codex и Azure OpenAI
    иначе, чем универсальные OpenAI-совместимые прокси `/v1`:

    **Нативные маршруты** (`openai/*`, Azure OpenAI):
    - Сохраняют `reasoning: { effort: "none" }` только для моделей, поддерживающих
      уровень рассуждений OpenAI `none`
    - Не передают отключённые рассуждения моделям или прокси, которые отклоняют
      `reasoning.effort: "none"`
    - По умолчанию используют строгий режим для схем инструментов
    - Добавляют скрытые заголовки атрибуции только для проверенных нативных узлов (Azure
      OpenAI не получает эти заголовки, несмотря на то что это нативный маршрут)
    - Сохраняют формирование запросов, специфичное для OpenAI (`service_tier`, `store`,
      совместимость рассуждений, подсказки для кэша промптов)

    **Прокси- и совместимые маршруты:**
    - Используют менее строгое поведение совместимости
    - Удаляют Completions `store` из ненативных полезных данных `openai-completions`
    - Принимают расширенный сквозной JSON `params.extra_body`/`params.extraBody`
      для OpenAI-совместимых прокси Completions
    - Принимают `params.chat_template_kwargs` для OpenAI-совместимых прокси Completions,
      таких как vLLM
    - Не навязывают строгие схемы инструментов или заголовки только для нативных маршрутов

  </Accordion>
</AccordionGroup>
