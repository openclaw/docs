# Архитектура IPC OpenClaw для macOS

Локальный сокет Unix соединяет службу хоста Node с приложением macOS для подтверждения выполнения команд и `system.run`. Для проверок обнаружения и подключения существует отладочный CLI `openclaw-mac` (`apps/macos/Sources/OpenClawMacCLI`); действия агента по-прежнему проходят через WebSocket Gateway и `node.invoke`. Путь `computer.act`, реализуемый через Node, запускает встроенную автоматизацию Peekaboo внутри процесса; автономные клиенты Peekaboo используют PeekabooBridge.

## Цели

- Единственный экземпляр приложения с графическим интерфейсом, отвечающий за всю работу, связанную с TCC (уведомления, запись экрана, микрофон, распознавание речи, AppleScript).
- Компактный интерфейс автоматизации: Gateway и команды Node, внутрипроцессный `computer.act`, а также PeekabooBridge для автономных клиентов автоматизации пользовательского интерфейса.
- Предсказуемые разрешения: всегда один и тот же подписанный идентификатор пакета, запуск через launchd, благодаря чему разрешения TCC сохраняются.

## Принцип работы

### Транспорт Gateway и Node

- Приложение запускает Gateway (в локальном режиме) и подключается к нему как Node.
- Действия агента выполняются через `node.invoke` (например, `system.run`, `system.notify`, `canvas.*`).
- Команды Node включают `canvas.*`, `camera.snap`, `camera.clip`, `screen.snapshot`, `screen.record`, `computer.act`, `system.run` и `system.notify`.
- Node предоставляет карту `permissions`, чтобы агенты могли определить, доступен ли доступ к экрану, камере, микрофону, распознаванию речи, автоматизации или специальным возможностям.

### Служба Node и IPC приложения

- Фоновая служба хоста Node подключается к WebSocket Gateway.
- Запросы `system.run` перенаправляются приложению macOS через локальный сокет Unix (`ExecApprovalsSocket.swift`).
- Приложение выполняет команду в контексте пользовательского интерфейса, при необходимости запрашивает подтверждение и возвращает результат.

Схема (SCI):

```text
Агент -> Gateway -> Служба Node (WS)
                      |  IPC (UDS + токен + HMAC + TTL)
                      v
                  Приложение Mac (UI + TCC + system.run)
```

### PeekabooBridge (автоматизация пользовательского интерфейса)

- Встроенный инструмент агента `computer` **не** использует этот сокет. Связанный Node macOS выполняет `computer.act` в процессе приложения с помощью встроенных служб Peekaboo.
- Для автоматизации пользовательского интерфейса используется отдельный сокет UNIX (`~/Library/Application Support/OpenClaw/<socket>`) и протокол JSON PeekabooBridge.
- Порядок предпочтения хостов (на стороне клиента): Peekaboo.app -> Claude.app -> OpenClaw.app -> локальное выполнение.
- Безопасность: хосты моста требуют TeamID из списка разрешённых (встроенный `PeekabooBridgeHostCoordinator` разрешает фиксированную команду и собственную команду подписи приложения); доступный только в режиме DEBUG механизм для того же UID управляется переменной `PEEKABOO_ALLOW_UNSIGNED_SOCKET_CLIENTS=1` (соглашение Peekaboo).
- Подробнее см. в разделе [Использование PeekabooBridge](/ru/platforms/mac/peekaboo).

## Рабочие процессы

- Перезапуск и пересборка: `scripts/restart-mac.sh` завершает существующие экземпляры, выполняет пересборку с помощью Swift, повторно упаковывает и запускает приложение. Скрипт автоматически определяет доступный идентификатор подписи и при его отсутствии переходит к `--no-sign`; передайте `--sign`, чтобы сделать подпись обязательной (при отсутствии ключа выполнение завершится ошибкой), или `--no-sign`, чтобы принудительно использовать путь без подписи. Переменная `SIGN_IDENTITY`, заданная в окружении, удаляется на пути с подписью, поэтому собственный механизм автоматического определения идентификатора в `scripts/codesign-mac-app.sh` выбирает сертификат.
- Единственный экземпляр: приложение проверяет `NSWorkspace.runningApplications` на наличие дублирующего идентификатора пакета и завершает работу, если обнаружено более одного экземпляра (`isDuplicateInstance()` в `MenuBar.swift`).

## Примечания по усилению защиты

- Для всех привилегированных интерфейсов предпочтительно требовать совпадения TeamID.
- PeekabooBridge: `PEEKABOO_ALLOW_UNSIGNED_SOCKET_CLIENTS=1` (только для DEBUG) может разрешать вызовы от процессов с тем же UID при локальной разработке.
- Вся связь остаётся исключительно локальной; сетевые сокеты не открываются.
- Запросы TCC исходят только от пакета приложения с графическим интерфейсом; сохраняйте стабильный идентификатор подписанного пакета между пересборками.
- Усиление защиты сокета подтверждения выполнения команд: режим файла `0600`, общий токен, проверка UID однорангового процесса (`getpeereid`), механизм запрос-ответ на основе HMAC-SHA256 и короткий TTL запросов.
