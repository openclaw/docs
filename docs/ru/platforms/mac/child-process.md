Приложение macOS по умолчанию управляет Gateway через **launchd** и не
запускает Gateway как дочерний процесс. Сначала оно пытается подключиться к
уже запущенному Gateway на настроенном порту; если он недоступен, приложение
включает службу launchd через внешний CLI `openclaw` (без встроенной среды
выполнения). Это обеспечивает надежный автоматический запуск при входе в
систему и перезапуск после сбоев.

Режим дочернего процесса (Gateway запускается непосредственно приложением)
сейчас **не используется**. Если вам нужна более тесная интеграция с
интерфейсом, запустите Gateway вручную в терминале.

## Поведение по умолчанию (launchd)

- Приложение устанавливает пользовательский LaunchAgent с меткой `ai.openclaw.gateway` (или
  `ai.openclaw.<profile>` при использовании `--profile`/`OPENCLAW_PROFILE`).
- Когда включен локальный режим, приложение проверяет, что LaunchAgent загружен, и
  при необходимости запускает Gateway.
- Журналы записываются по пути журнала Gateway для launchd (он отображается в настройках отладки).

Основные команды:

```bash
launchctl kickstart -k gui/$UID/ai.openclaw.gateway
launchctl bootout gui/$UID/ai.openclaw.gateway
```

При использовании именованного профиля замените метку на `ai.openclaw.<profile>`.

## Неподписанные сборки для разработки

`scripts/restart-mac.sh --no-sign` предназначен для быстрых локальных сборок без ключей
подписания. Чтобы launchd не ссылался на неподписанный исполняемый файл ретранслятора, скрипт создает
`~/.openclaw/disable-launchagent`.

При подписанном запуске `scripts/restart-mac.sh` это переопределение удаляется, если
маркер присутствует. Чтобы сбросить его вручную:

```bash
rm ~/.openclaw/disable-launchagent
```

## Режим только подключения

Чтобы запретить приложению macOS устанавливать launchd или управлять им, запустите его с
`--attach-only` (или `--no-launchd`). При этом создается
`~/.openclaw/disable-launchagent`, поэтому приложение только подключается к уже
запущенному Gateway. Такое же поведение можно переключить в настройках отладки.

## Удаленный режим

В удаленном режиме локальный Gateway никогда не запускается. Приложение создает SSH-туннель к
удаленному узлу и подключается через него.

## Почему мы предпочитаем launchd

- Автоматический запуск при входе в систему.
- Встроенная семантика перезапуска/KeepAlive.
- Предсказуемые журналы и управление процессом.

Если настоящий режим дочернего процесса когда-либо снова понадобится, его следует документировать как
отдельный явный режим только для разработки.
