Этот поток позволяет приложению macOS работать как полноценный удаленный пульт для Gateway OpenClaw, запущенного на другом хосте (настольном компьютере/сервере). Приложение может подключаться напрямую к доверенным URL Gateway в LAN/Tailnet или управлять SSH-туннелем, когда удаленный Gateway доступен только через loopback. Проверки работоспособности, пересылка пробуждения голосом и веб-чат используют одну и ту же удаленную конфигурацию из _Настройки → Общие_.

## Режимы

- **Локально (этот Mac)**: все запускается на ноутбуке. SSH не используется.
- **Удаленно через SSH (по умолчанию)**: команды OpenClaw выполняются на удаленном хосте. Приложение для Mac открывает SSH-подключение с `-o BatchMode`, выбранным удостоверением/ключом и локальной переадресацией порта.
- **Удаленно напрямую (ws/wss)**: без SSH-туннеля. Приложение для Mac подключается к URL Gateway напрямую (например, через LAN, Tailscale, Tailscale Serve или публичный HTTPS reverse proxy).

## Удаленные транспорты

Удаленный режим поддерживает два транспорта:

- **SSH-туннель** (по умолчанию): использует `ssh -N -L ...`, чтобы перенаправить порт Gateway на localhost. Gateway будет видеть IP узла как `127.0.0.1`, потому что туннель является loopback.
- **Напрямую (ws/wss)**: подключается прямо к URL Gateway. Gateway видит реальный IP клиента.

Приложение отключает мультиплексирование SSH-подключений и перевод в фон после аутентификации для SSH-процессов, которыми владеет приложение, чтобы оно могло отслеживать и перезапускать точный процесс, даже если выбранный псевдоним включает `ControlMaster` или `ForkAfterAuthentication`.

Проверка SSH host key по умолчанию строгая, потому что учетные данные Gateway проходят через этот туннель. Для управляемого SSH-псевдонима, поведение доверия которого вы явно хотите использовать, включите это с помощью `openclaw-mac configure-remote --ssh-target <alias> --ssh-host-key-policy openssh` или задайте `gateway.remote.sshHostKeyPolicy` равным `"openssh"`. Это явное включение использует эффективную политику OpenSSH для host key; сначала проверьте псевдоним и любую совпадающую конфигурацию `Host *` или системную конфигурацию. Изменение SSH-цели в приложении или через `configure-remote` сбрасывает политику на `strict`, если вы явно не включите ее снова.

В режиме SSH-туннеля обнаруженные имена хостов LAN/tailnet сохраняются как
`gateway.remote.sshTarget`. Приложение оставляет `gateway.remote.url` на локальной
конечной точке туннеля, например `ws://127.0.0.1:18789`, чтобы CLI, веб-чат и
локальная служба узла-хоста использовали один и тот же безопасный loopback-транспорт.
Когда обнаружение возвращает и необработанные IP Tailnet, и стабильные имена хостов, приложение
предпочитает Tailscale MagicDNS или имена LAN, чтобы удаленные подключения лучше переживали
смену адресов.
Если локальный порт туннеля отличается от удаленного порта Gateway, задайте
`gateway.remote.remotePort` равным порту на удаленном хосте.

Автоматизацией браузера в удаленном режиме владеет узловой хост CLI, а не
нативный узел приложения macOS. Приложение запускает установленную службу узлового хоста, когда
это возможно; если вам нужно управление браузером с этого Mac, установите/запустите ее с помощью
`openclaw node install ...` и `openclaw node start` (или запустите
`openclaw node run ...` на переднем плане), затем нацельтесь на этот узел
с возможностями браузера.

## Предварительные требования на удаленном хосте

1. Установите Node + pnpm и соберите/установите CLI OpenClaw (`pnpm install && pnpm build && pnpm link --global`).
2. Убедитесь, что `openclaw` находится в PATH для неинтерактивных оболочек (при необходимости создайте symlink в `/usr/local/bin` или `/opt/homebrew/bin`).
3. Только для SSH-транспорта: откройте SSH с аутентификацией по ключу. Мы рекомендуем IP **Tailscale** для стабильной доступности вне LAN.

## Настройка приложения macOS

Чтобы предварительно настроить приложение без приветственного потока:

```bash
openclaw-mac configure-remote \
  --ssh-target user@gateway.local \
  --local-port 18789 \
  --remote-port 18789 \
  --token "$OPENCLAW_GATEWAY_TOKEN"
```

Для Gateway, уже доступного в доверенной LAN или Tailnet, полностью пропустите SSH:

```bash
openclaw-mac configure-remote \
  --direct-url ws://192.168.0.202:18789 \
  --token "$OPENCLAW_GATEWAY_TOKEN"
```

Это записывает удаленную конфигурацию, помечает онбординг завершенным и позволяет приложению владеть
выбранным транспортом при запуске.

1. Откройте _Настройки → Общие_.
2. В разделе **OpenClaw запускается** выберите **Удаленно** и задайте:
   - **Транспорт**: **SSH-туннель** или **Напрямую (ws/wss)**.
   - **SSH-цель**: `user@host` (необязательно `:port`).
     - Если Gateway находится в той же LAN и объявляет себя через Bonjour, выберите его из обнаруженного списка, чтобы автоматически заполнить это поле.
   - **URL Gateway** (только напрямую): `wss://gateway.example.ts.net` (или `ws://...` для локального/LAN).
   - **Файл удостоверения** (расширенно): путь к вашему ключу.
   - **Корень проекта** (расширенно): путь к удаленному checkout, используемый для команд.
   - **Путь CLI** (расширенно): необязательный путь к запускаемой точке входа/двоичному файлу `openclaw` (заполняется автоматически, когда объявлен).
3. Нажмите **Проверить удаленное подключение**. Успех означает, что удаленная команда `openclaw status --json` выполняется корректно. Сбои обычно означают проблемы с PATH/CLI; код выхода 127 означает, что CLI не найден удаленно.
4. Проверки работоспособности и веб-чат теперь будут автоматически работать через выбранный транспорт.

## Веб-чат

- **SSH-туннель**: веб-чат подключается к Gateway через перенаправленный управляющий порт WebSocket (по умолчанию 18789).
- **Напрямую (ws/wss)**: веб-чат подключается прямо к настроенному URL Gateway.
- Отдельного HTTP-сервера WebChat больше нет.

## Разрешения

- Удаленному хосту нужны те же одобрения TCC, что и локальному (Automation, Accessibility, Screen Recording, Microphone, Speech Recognition, Notifications). Запустите онбординг на этой машине, чтобы выдать их один раз.
- Узлы объявляют свое состояние разрешений через `node.list` / `node.describe`, чтобы агенты знали, что доступно.

## Заметки по безопасности

- Предпочитайте loopback-привязки на удаленном хосте и подключайтесь через SSH, Tailscale Serve или доверенный прямой URL Tailnet/LAN.
- SSH-туннелирование по умолчанию требует уже доверенный host key. Сначала доверьте host key, чтобы он существовал в настроенном файле known-hosts, или явно выберите `gateway.remote.sshHostKeyPolicy: "openssh"` для управляемого псевдонима, политику доверия OpenSSH которого вы принимаете.
- Если вы привязываете Gateway к интерфейсу, отличному от loopback, требуйте действительную аутентификацию Gateway: token, password или identity-aware reverse proxy с `gateway.auth.mode: "trusted-proxy"`.
- См. [Безопасность](/ru/gateway/security) и [Tailscale](/ru/gateway/tailscale).

## Поток входа WhatsApp (удаленно)

- Запустите `openclaw channels login --verbose` **на удаленном хосте**. Отсканируйте QR с помощью WhatsApp на телефоне.
- Повторно запустите вход на этом хосте, если срок действия аутентификации истечет. Проверка работоспособности покажет проблемы со связью.

## Устранение неполадок

- **exit 127 / не найдено**: `openclaw` не находится в PATH для оболочек без входа в систему. Добавьте его в `/etc/paths`, rc-файл вашей оболочки или создайте symlink в `/usr/local/bin`/`/opt/homebrew/bin`.
- **Проверка работоспособности не удалась**: проверьте доступность SSH, PATH и то, что Baileys вошел в систему (`openclaw status --json`).
- **Веб-чат завис**: подтвердите, что Gateway запущен на удаленном хосте и перенаправленный порт совпадает с WS-портом Gateway; UI требует работоспособного WS-подключения.
- **IP узла показывает 127.0.0.1**: ожидаемо для SSH-туннеля. Переключите **Транспорт** на **Напрямую (ws/wss)**, если хотите, чтобы Gateway видел реальный IP клиента.
- **Панель работает, но возможности Mac offline**: это означает, что операторское/управляющее подключение приложения работоспособно, но соединение companion-узла не подключено или у него отсутствует поверхность команд. Откройте раздел устройства в строке меню и проверьте, находится ли Mac в состоянии `paired · disconnected`. Для конечных точек Tailscale Serve `wss://*.ts.net` приложение обнаруживает устаревшие legacy TLS leaf pins после ротации сертификата, очищает устаревший pin, когда macOS доверяет новому сертификату, и повторяет попытку автоматически. Если сертификат не является системно доверенным или хост не является именем Tailscale Serve, задайте `gateway.remote.tlsFingerprint` равным ожидаемому fingerprint сертификата, проверьте сертификат или переключитесь на **Удаленно через SSH**.
- **Пробуждение голосом**: триггерные фразы автоматически пересылаются в удаленном режиме; отдельный forwarder не нужен.

## Звуки уведомлений

Выбирайте звуки для каждого уведомления из скриптов с `openclaw` и `node.invoke`, например:

```bash
openclaw nodes notify --node <id> --title "Ping" --body "Remote gateway ready" --sound Glass
```

Глобального переключателя "default sound" в приложении больше нет; вызывающие стороны выбирают звук (или его отсутствие) для каждого запроса.
