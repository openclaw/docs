Этот сценарий позволяет использовать приложение macOS как полноценный пульт дистанционного управления для Gateway OpenClaw, работающего на другом хосте (настольном компьютере или сервере). Приложение подключается напрямую к доверенным URL-адресам Gateway в локальной сети или Tailnet либо управляет SSH-туннелем, если удалённый Gateway доступен только через local loopback. Проверки работоспособности, перенаправление Voice Wake и веб-чат используют ту же удалённую конфигурацию из раздела _Настройки -> Основные_.

## Режимы

- **Локальный (этот Mac)**: всё работает на ноутбуке; SSH не используется.
- **Удалённый через SSH (по умолчанию)**: команды OpenClaw выполняются на удалённом хосте. Приложение открывает SSH-соединение с параметром `-o BatchMode`, выбранным идентификатором или ключом и перенаправлением локального порта.
- **Прямое удалённое подключение (ws/wss)**: без SSH-туннеля; приложение подключается непосредственно к URL-адресу Gateway (через локальную сеть, Tailscale, Tailscale Serve или общедоступный обратный прокси-сервер HTTPS).

## Удалённые способы подключения

- **SSH-туннель** (по умолчанию): использует `ssh -N -L ...` для перенаправления порта Gateway на localhost. Gateway видит IP-адрес Node как `127.0.0.1`, поскольку туннель использует local loopback.
- **Прямое подключение (ws/wss)**: подключается непосредственно к URL-адресу Gateway. Gateway видит фактический IP-адрес клиента.

Для собственных процессов SSH приложение отключает мультиплексирование SSH-соединений и переход в фоновый режим после аутентификации, чтобы отслеживать и перезапускать конкретный процесс, даже если для выбранного псевдонима включены `ControlMaster` или `ForkAfterAuthentication`.

По умолчанию проверка ключа SSH-хоста выполняется строго, поскольку через этот туннель передаются учётные данные Gateway. Чтобы использовать собственный механизм доверия управляемого псевдонима SSH, задайте `--ssh-host-key-policy openssh` с помощью `openclaw-mac configure-remote` или напрямую установите для `gateway.remote.sshHostKeyPolicy` значение `"openssh"`. Перед этим проверьте псевдоним и все соответствующие настройки `Host *` или системную конфигурацию. При изменении целевого SSH-хоста (в приложении или через `configure-remote`) политика снова устанавливается в значение `strict`, если вы явно не включите её повторно для нового целевого хоста.

В режиме SSH-туннеля обнаруженные имена хостов локальной сети или Tailnet сохраняются в `gateway.remote.sshTarget`. Приложение сохраняет в `gateway.remote.url` локальную конечную точку туннеля (например, `ws://127.0.0.1:18789`), чтобы CLI, веб-чат и локальная служба хоста Node использовали один и тот же транспорт local loopback. Когда обнаруживаются как необработанные IP-адреса Tailnet, так и стабильные имена хостов, приложение предпочитает имена Tailscale MagicDNS или локальной сети, чтобы соединения лучше сохранялись при изменении адресов. Если локальный порт туннеля отличается от порта удалённого Gateway, задайте в `gateway.remote.remotePort` порт на удалённом хосте.

За автоматизацию браузера в удалённом режиме отвечает хост Node CLI, а не Node нативного приложения macOS. По возможности приложение запускает установленную службу хоста Node. Чтобы включить управление браузером с этого Mac, установите и запустите её с помощью `openclaw node install ...` и `openclaw node start` (либо запустите `openclaw node run ...` в основном режиме), а затем выберите целевой Node с поддержкой браузера.

## Требования к удалённому хосту

1. Установите Node и pnpm, затем соберите или установите CLI OpenClaw (`pnpm install && pnpm build && pnpm link --global`).
2. Убедитесь, что `openclaw` доступен через PATH для неинтерактивных оболочек (при необходимости создайте символическую ссылку в `/usr/local/bin` или `/opt/homebrew/bin`).
3. Для подключения по SSH настройте аутентификацию SSH по ключу. Для стабильного доступа за пределами локальной сети рекомендуется использовать IP-адреса Tailscale.

## Настройка приложения macOS

Чтобы предварительно настроить приложение через SSH без прохождения приветственного сценария:

```bash
openclaw-mac configure-remote \
  --ssh-target user@gateway-host \
  --local-port 18789 \
  --remote-port 18789 \
  --token "$OPENCLAW_GATEWAY_TOKEN"
```

Либо, если Gateway уже доступен в доверенной локальной сети или Tailnet, полностью откажитесь от SSH:

```bash
openclaw-mac configure-remote \
  --direct-url ws://192.168.0.202:18789 \
  --token "$OPENCLAW_GATEWAY_TOKEN"
```

Оба варианта записывают настройки в `~/.openclaw/openclaw.json`, отмечают первоначальную настройку как завершённую и позволяют приложению управлять выбранным способом подключения при следующем запуске. Значение по умолчанию для `--local-port` и `--remote-port` — `18789`. Другие флаги: `--password`, `--identity <path>`, `--ssh-host-key-policy <strict|openssh>`, `--project-root <path>`, `--cli-path <path>`, `--json`. Полный справочник можно вывести командой `openclaw-mac configure-remote --help`.

Чтобы выполнить настройку через пользовательский интерфейс:

1. Откройте _Settings -> General_.
2. В разделе **OpenClaw runs** выберите **Remote** и задайте:
   - **Transport**: **SSH tunnel** или **Direct (ws/wss)**.
   - **SSH target**: `user@host` (необязательно с `:port`). Если Gateway находится в той же локальной сети и объявляет себя через Bonjour, выберите его в списке обнаруженных устройств, чтобы заполнить это поле автоматически.
   - **Gateway URL** (только для прямого подключения): `wss://gateway.example.ts.net` (или `ws://...` для локального подключения или локальной сети).
   - **Identity file** (расширенная настройка): путь к вашему ключу.
   - **Project root** (расширенная настройка): путь к удалённой рабочей копии, используемой для выполнения команд.
   - **CLI path** (расширенная настройка): необязательный путь к исполняемой точке входа или двоичному файлу `openclaw` (заполняется автоматически, если он объявлен).
3. Нажмите **Test remote**. Успешный результат означает, что удалённая команда `openclaw status --json` выполнена корректно. Ошибки обычно связаны с PATH или CLI; код выхода 127 означает, что CLI не найден на удалённом хосте.
4. Теперь проверки работоспособности и веб-чат автоматически используют выбранный способ подключения.

## Веб-чат

- **SSH-туннель**: подключается к Gateway через перенаправленный управляющий порт WebSocket (по умолчанию 18789).
- **Прямое подключение (ws/wss)**: подключается непосредственно к настроенному URL-адресу Gateway.
- Отдельного HTTP-сервера веб-чата нет.

## Разрешения

- Удалённому хосту требуются те же разрешения TCC, что и локальному: автоматизация, универсальный доступ, запись экрана, микрофон, распознавание речи и уведомления. Один раз пройдите первоначальную настройку на этом компьютере, чтобы предоставить их.
- Узлы передают сведения о состоянии своих разрешений через `node.list` и `node.describe`, чтобы агенты знали, какие возможности доступны.

## Примечания по безопасности

- Предпочтительно привязывать службы на удалённом хосте к local loopback и подключаться через SSH, Tailscale Serve либо доверенный прямой URL-адрес Tailnet или локальной сети.
- По умолчанию для туннелирования SSH требуется уже доверенный ключ хоста. Сначала добавьте ключ хоста в настроенный файл известных хостов либо явно задайте `gateway.remote.sshHostKeyPolicy: "openssh"` для управляемого псевдонима, политике доверия OpenSSH которого вы доверяете.
- Если Gateway привязан к интерфейсу, отличному от local loopback, требуйте действительную аутентификацию Gateway: токен, пароль или обратный прокси-сервер с поддержкой идентификации и параметром `gateway.auth.mode: "trusted-proxy"`.
- См. разделы [Безопасность](/ru/gateway/security) и [Tailscale](/ru/gateway/tailscale).

## Процедура входа в WhatsApp (удалённый режим)

- Выполните `openclaw channels login --channel whatsapp --verbose` **на удалённом хосте**. Отсканируйте QR-код с помощью WhatsApp на телефоне.
- Если срок действия аутентификации истёк, повторите вход на этом хосте. Проверка работоспособности сообщает о проблемах с подключением.

| ------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `exit 127` / команда не найдена                   | `openclaw` отсутствует в PATH для оболочек без входа в систему. | Проверка работоспособности завершилась ошибкой    | Проверьте доступность по SSH, PATH и наличие активного входа Baileys (WhatsApp) (`openclaw status --json`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Веб-чат завис                                     | Убедитесь, что Gateway запущен на удалённом узле и перенаправленный порт совпадает с WS-портом Gateway; для работы интерфейса требуется исправное WS-соединение.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| IP-адрес Node отображается как `127.0.0.1`        | Это ожидаемо при использовании SSH-туннеля. Переключите **Транспорт** на **Прямой (ws/wss)**, если хотите, чтобы Gateway видел реальный IP-адрес клиента.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Панель управления работает, но возможности Mac недоступны | Соединение оператора/управления исправно, но соединение со вспомогательным Node не установлено либо в нём отсутствует интерфейс команд. Откройте раздел устройств в строке меню и проверьте, отображается ли состояние Mac как `сопряжён · отключён`. Для конечных точек Tailscale Serve вида `wss://*.ts.net` приложение обнаруживает устаревшие закреплённые отпечатки TLS-сертификатов после их ротации, удаляет устаревший отпечаток, когда macOS начинает доверять новому сертификату, и автоматически повторяет подключение. Если сертификат не является доверенным на уровне системы или имя узла не относится к Tailscale Serve, задайте для `gateway.remote.tlsFingerprint` ожидаемый отпечаток сертификата, проверьте сертификат либо переключитесь на **Удалённое подключение по SSH**. |
| Голосовая активация                               | В удалённом режиме фразы активации пересылаются автоматически; отдельный перенаправитель не требуется.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |

## Звуки уведомлений

Выбирайте звук для каждого уведомления из скриптов с помощью `openclaw nodes notify`, например:

```bash
openclaw nodes notify --node <id> --title "Ping" --body "Remote gateway ready" --sound Glass
```

В приложении нет глобального переключателя звука по умолчанию; вызывающая сторона выбирает звук (или его отсутствие) для каждого запроса.
