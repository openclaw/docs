Доступность: сборки приложения для iPhone распространяются через каналы Apple, когда они включены для релиза. Локальные сборки для разработки также можно запускать из исходного кода.

## Что оно делает

- Подключается к Gateway по WebSocket (LAN или частная сеть Tailscale).
- Предоставляет возможности узла: Canvas, снимок экрана, захват с камеры, местоположение, режим Talk, голосовое пробуждение.
- Получает команды `node.invoke` и сообщает события состояния узла.

## Требования

- Gateway, запущенный на другом устройстве (macOS, Linux или Windows через WSL2).
- Сетевой путь:
  - Та же LAN через Bonjour, **или**
  - Частная сеть Tailscale через unicast DNS-SD (пример домена: `openclaw.internal.`), **или**
  - Ручной хост/порт (резервный вариант).

## Быстрый старт (сопряжение + подключение)

1. Запустите аутентифицированный Gateway с маршрутом, доступным с телефона. Tailscale
   Serve — рекомендуемый удаленный путь:

```bash
openclaw gateway --port 18789 --tailscale serve
```

Для доверенной настройки в одной LAN вместо этого используйте аутентифицированный
`gateway.bind: "lan"`. Привязка по умолчанию к петлевому интерфейсу недоступна
с телефона. Если Gateway еще не настроен, сначала выполните `openclaw onboard`,
чтобы при создании кода настройки был доступен путь аутентификации по токену
или паролю.

2. Откройте [интерфейс управления](/ru/web/control-ui), выберите **Узлы** и нажмите
   **Сопрячь мобильное устройство** в карточке **Устройства**.

3. В приложении iOS откройте **Настройки** → **Gateway**, отсканируйте QR-код
   (или вставьте код настройки) и подключитесь.

4. Официальное приложение подключается автоматически. Если в **Устройствах**
   показан ожидающий запрос, проверьте его роль и области доступа перед одобрением.

Кнопка в интерфейсе управления требует уже сопряженного сеанса с `operator.admin`.
В качестве терминального резервного варианта выберите обнаруженный gateway в
приложении iOS (или включите Manual Host и введите хост/порт), затем одобрите
запрос на хосте Gateway:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

Если приложение повторяет сопряжение с измененными данными аутентификации
(роль/области доступа/публичный ключ), предыдущий ожидающий запрос заменяется
и создается новый `requestId`. Перед одобрением снова выполните
`openclaw devices list`.

Необязательно: если узел iOS всегда подключается из строго контролируемой
подсети, можно явно включить автоматическое одобрение узла при первом
подключении с помощью CIDR или точных IP-адресов:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

По умолчанию это отключено. Это применяется только к новому сопряжению
`role: node` без запрошенных областей доступа. Сопряжение оператора/браузера
и любые изменения роли, области доступа, метаданных или публичного ключа
по-прежнему требуют ручного одобрения.

5. Проверьте подключение:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## Push через relay для официальных сборок

Официальные распространяемые сборки iOS используют внешний push relay вместо публикации исходного
токена APNs в gateway.

Официальные сборки App Store из публичной ветки релизов используют размещенный relay по адресу `https://ios-push-relay.openclaw.ai`.

Пользовательские развертывания relay требуют намеренно отдельного пути сборки/развертывания iOS, где URL relay совпадает с URL relay в gateway. Публичная ветка релизов App Store не принимает переопределения пользовательского URL relay. Если вы используете пользовательскую сборку relay, задайте совпадающий URL relay для gateway:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

Как работает поток:

- Приложение iOS регистрируется в relay с помощью App Attest и JWS транзакции приложения StoreKit.
- Relay возвращает непрозрачный дескриптор relay и разрешение на отправку, ограниченное регистрацией.
- Приложение iOS получает идентичность сопряженного gateway и включает ее в регистрацию relay, поэтому регистрация через relay делегируется этому конкретному gateway.
- Приложение передает эту регистрацию через relay сопряженному gateway с помощью `push.apns.register`.
- Gateway использует сохраненный дескриптор relay для `push.test`, фоновых пробуждений и подсказок пробуждения.
- Пользовательские URL relay gateway должны совпадать с URL relay, встроенным в сборку iOS.
- Если приложение позже подключается к другому gateway или к сборке с другим базовым URL relay, оно обновляет регистрацию relay вместо повторного использования старой привязки.

Что gateway **не** требуется для этого пути:

- Нет токена relay для всего развертывания.
- Нет прямого ключа APNs для официальных отправок через relay из App Store.

Ожидаемый поток оператора:

1. Установите официальное приложение iOS.
2. Необязательно: задайте `gateway.push.apns.relay.baseUrl` на gateway только при использовании намеренно отдельной пользовательской сборки relay.
3. Сопрягите приложение с gateway и дождитесь завершения подключения.
4. Приложение автоматически публикует `push.apns.register` после того, как у него есть токен APNs, сеанс оператора подключен, а регистрация relay успешна.
5. После этого `push.test`, пробуждения для повторного подключения и подсказки пробуждения могут использовать сохраненную регистрацию через relay.

## Фоновые сигналы активности

Когда iOS пробуждает приложение для silent push, фонового обновления или события значимого изменения местоположения, приложение
пытается выполнить короткое повторное подключение узла, а затем вызывает `node.event` с `event: "node.presence.alive"`.
Gateway записывает это как `lastSeenAtMs`/`lastSeenReason` в метаданные сопряженного узла/устройства только
после того, как становится известна аутентифицированная идентичность устройства узла.

Приложение считает фоновое пробуждение успешно записанным только когда ответ gateway содержит
`handled: true`. Более старые gateway могут подтверждать `node.event` с `{ "ok": true }`; такой ответ
совместим, но не считается устойчивым обновлением времени последнего появления.

Примечание о совместимости:

- `OPENCLAW_APNS_RELAY_BASE_URL` по-прежнему работает как временное переопределение env для gateway.
- Публичная ветка релизов App Store отклоняет `OPENCLAW_PUSH_RELAY_BASE_URL` для сборок iOS.

## Поток аутентификации и доверия

Relay существует, чтобы обеспечить два ограничения, которые прямой APNs на gateway не может предоставить для
официальных сборок iOS:

- Только подлинные сборки OpenClaw для iOS, распространяемые через Apple, могут использовать размещенный relay.
- Gateway может отправлять push через relay только для устройств iOS, которые сопряжены с этим конкретным
  gateway.

По шагам:

1. `iOS app -> gateway`
   - Приложение сначала сопрягается с gateway через обычный поток аутентификации Gateway.
   - Это дает приложению аутентифицированный сеанс узла и аутентифицированный сеанс оператора.
   - Сеанс оператора используется для вызова `gateway.identity.get`.

2. `iOS app -> relay`
   - Приложение вызывает конечные точки регистрации relay по HTTPS.
   - Регистрация включает доказательство App Attest и JWS транзакции приложения StoreKit.
   - Relay проверяет bundle ID, доказательство App Attest и доказательство распространения Apple, а также требует
     официальный/производственный путь распространения.
   - Именно это блокирует локальные сборки Xcode/dev от использования размещенного relay. Локальная сборка может быть
     подписана, но она не удовлетворяет доказательству официального распространения Apple, которое ожидает relay.

3. `gateway identity delegation`
   - Перед регистрацией relay приложение получает идентичность сопряженного gateway из
     `gateway.identity.get`.
   - Приложение включает эту идентичность gateway в полезную нагрузку регистрации relay.
   - Relay возвращает дескриптор relay и разрешение на отправку, ограниченное регистрацией, делегированные
     этой идентичности gateway.

4. `gateway -> relay`
   - Gateway сохраняет дескриптор relay и разрешение на отправку из `push.apns.register`.
   - При `push.test`, пробуждениях для повторного подключения и подсказках пробуждения gateway подписывает запрос отправки своей
     собственной идентичностью устройства.
   - Relay проверяет как сохраненное разрешение на отправку, так и подпись gateway относительно делегированной
     идентичности gateway из регистрации.
   - Другой gateway не может повторно использовать эту сохраненную регистрацию, даже если каким-то образом получит дескриптор.

5. `relay -> APNs`
   - Relay владеет производственными учетными данными APNs и исходным токеном APNs для официальной сборки.
   - Gateway никогда не хранит исходный токен APNs для официальных сборок через relay.
   - Relay отправляет итоговый push в APNs от имени сопряженного gateway.

Почему был создан этот дизайн:

- Чтобы не хранить производственные учетные данные APNs на пользовательских gateway.
- Чтобы избежать хранения исходных токенов APNs официальных сборок на gateway.
- Чтобы разрешить использование размещенного relay только для официальных сборок OpenClaw для iOS.
- Чтобы предотвратить отправку push-пробуждений одним gateway на устройства iOS, принадлежащие другому gateway.

Локальные/ручные сборки остаются на прямом APNs. Если вы тестируете такие сборки без relay, gateway
по-прежнему нужны прямые учетные данные APNs:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

Это runtime env vars хоста gateway, а не настройки Fastlane. `apps/ios/fastlane/.env` хранит только
аутентификацию App Store Connect, такую как `APP_STORE_CONNECT_KEY_ID` и
`APP_STORE_CONNECT_ISSUER_ID`; он не настраивает прямую доставку APNs для локальных сборок iOS.

Рекомендуемое хранилище на хосте gateway:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

Не коммитьте файл `.p8` и не размещайте его в рабочей копии репозитория.

## Пути обнаружения

### Bonjour (LAN)

Приложение iOS выполняет обзор `_openclaw-gw._tcp` в `local.` и, когда настроено, в том же
домены обнаружения wide-area DNS-SD. Gateway в той же LAN автоматически появляются из `local.`;
обнаружение между сетями может использовать настроенный wide-area домен без изменения типа beacon.

### Частная сеть Tailscale (межсетевой режим)

Если mDNS заблокирован, используйте зону unicast DNS-SD (выберите домен; пример:
`openclaw.internal.`) и split DNS Tailscale.
См. [Bonjour](/ru/gateway/bonjour) для примера CoreDNS.

### Ручной хост/порт

В настройках включите **Manual Host** и введите хост gateway + порт (по умолчанию `18789`).

## Canvas + A2UI

Узел iOS отображает canvas WKWebView. Используйте `node.invoke` для управления им:

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.navigate --params '{"url":"http://<gateway-host>:18789/__openclaw__/canvas/"}'
```

Примечания:

- Хост canvas Gateway обслуживает `/__openclaw__/canvas/` и `/__openclaw__/a2ui/`.
- Он обслуживается HTTP-сервером Gateway (тот же порт, что и `gateway.port`, по умолчанию `18789`).
- Узел iOS сохраняет встроенный scaffold как подключенный вид по умолчанию. `canvas.a2ui.push` и `canvas.a2ui.reset` используют встроенную страницу A2UI, принадлежащую приложению.
- Удаленные страницы A2UI Gateway на iOS доступны только для отображения; нативные действия кнопок A2UI принимаются только со встроенных страниц, принадлежащих приложению.
- Вернитесь к встроенному scaffold с помощью `canvas.navigate` и `{"url":""}`.

## Связь с Computer Use

Приложение iOS — это мобильная поверхность узла, а не backend Codex Computer Use. Codex
Computer Use и `cua-driver mcp` управляют локальным рабочим столом macOS через инструменты
MCP; приложение iOS предоставляет возможности iPhone через команды узла OpenClaw,
такие как `canvas.*`, `camera.*`, `screen.*`, `location.*` и `talk.*`.

Агенты по-прежнему могут управлять приложением iOS через OpenClaw, вызывая команды
узла, но эти вызовы проходят через протокол узла gateway и подчиняются ограничениям iOS
для переднего/фонового режима. Используйте [Codex Computer Use](/ru/plugins/codex-computer-use)
для локального управления рабочим столом, а эту страницу — для возможностей узла iOS.

### Canvas eval / snapshot

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.eval --params '{"javaScript":"(() => { const {ctx} = window.__openclaw; ctx.clearRect(0,0,innerWidth,innerHeight); ctx.lineWidth=6; ctx.strokeStyle=\"#ff2d55\"; ctx.beginPath(); ctx.moveTo(40,40); ctx.lineTo(innerWidth-40, innerHeight-40); ctx.stroke(); return \"ok\"; })()"}'
```

```bash
openclaw nodes invoke --node "iOS Node" --command canvas.snapshot --params '{"maxWidth":900,"format":"jpeg"}'
```

## Голосовое пробуждение + режим talk

- Голосовое пробуждение и режим разговора доступны в настройках.
- OpenAI realtime Talk использует WebRTC, принадлежащий клиенту, когда `talk.realtime.transport` имеет значение `webrtc`; явная конфигурация `gateway-relay` остается в зоне ответственности Gateway. См. [режим Talk](/ru/nodes/talk).
- Узлы iOS с поддержкой Talk объявляют возможность `talk` и могут объявлять
  `talk.ptt.start`, `talk.ptt.stop`, `talk.ptt.cancel` и `talk.ptt.once`;
  Gateway по умолчанию разрешает эти команды push-to-talk для доверенных
  узлов с поддержкой Talk.
- iOS может приостанавливать фоновое аудио; считайте голосовые функции работающими по мере возможности, когда приложение не активно.

## Распространенные ошибки

- `NODE_BACKGROUND_UNAVAILABLE`: выведите приложение iOS на передний план (это требуется для команд холста, камеры и экрана).
- `A2UI_HOST_UNAVAILABLE`: встроенная страница A2UI была недоступна в WebView приложения; оставьте приложение на переднем плане на вкладке Screen и повторите попытку.
- Запрос на сопряжение не появляется: выполните `openclaw devices list` и подтвердите вручную.
- Повторное подключение не удается после переустановки: токен сопряжения Keychain был очищен; выполните повторное сопряжение узла.
