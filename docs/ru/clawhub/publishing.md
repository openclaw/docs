# Публикация

При публикации папка skill или пакет плагина отправляется в ClawHub от имени выбранного вами владельца. ClawHub проверяет, позволяет ли ваш токен публиковать материалы от имени этого владельца, проверяет метаданные, имя, версию, файлы и сведения об исходном коде, а затем сохраняет выпуск и запускает автоматические проверки безопасности.

Если проверка завершается с ошибкой, ничего не публикуется. Новые выпуски также могут быть недоступны в обычных средствах установки и скачивания до завершения проверки.

## Skills

Проще всего выполнять публикацию через CLI. Войдите в систему, а затем опубликуйте локальную папку skill:

```bash
clawhub login
clawhub skill publish ./my-skill \
  --slug my-skill \
  --name "My Skill" \
  --owner <owner>
```

При публикации от имени организации используйте `--owner <handle>`. Не указывайте этот параметр, чтобы опубликовать материал от имени аутентифицированного пользователя. При публикации неизменившееся содержимое пропускается. Версия нового skill начинается с `1.0.0`, а при последующих изменениях автоматически публикуется следующая версия исправления. Передавайте `--version` только тогда, когда необходимо явно указать версию.

Для репозиториев каталогов используйте многократно используемый рабочий процесс ClawHub
[`skill-publish.yml`](https://github.com/openclaw/clawhub/blob/main/.github/workflows/skill-publish.yml).
Он вызывает `skill publish` для каждой папки skill непосредственно в каталоге `root` (по умолчанию:
`skills`) либо только для папки, указанной в `skill_path`.

```yaml
jobs:
  publish:
    uses: openclaw/clawhub/.github/workflows/skill-publish.yml@main
    with:
      owner: <owner>
      dry_run: false
    secrets:
      clawhub_token: ${{ secrets.CLAWHUB_TOKEN }}
```

Используйте `dry_run: true`, чтобы предварительно просмотреть новые и изменённые Skills без публикации.

## Плагины

Плагины используют имена пакетов в стиле npm. В область имён пакета входит владелец, указанный в первой части имени:

```text
@owner/package-name
```

Область имён должна соответствовать выбранному владельцу публикации. Если пакет называется `@openclaw/dronzer`, его можно опубликовать только от имени `@openclaw`. При публикации от имени `@vintageayu` переименуйте пакет в `@vintageayu/dronzer`.

Это не позволяет пакету претендовать на пространство имён организации, которое не контролируется издателем.

Если вы являетесь законным владельцем организации, бренда, области имён пакета, идентификатора владельца или пространства имён, которое уже занято или зарезервировано в ClawHub, создайте
[запрос на закрепление организации или пространства имён](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml),
приложив общедоступные неконфиденциальные доказательства. Сведения о том, что следует включить и что нельзя публиковать в общедоступных запросах, см. в разделе
[«Закрепление организаций и пространств имён»](/clawhub/namespace-claims).

### Перед публикацией плагина

- Выберите владельца, соответствующего области имён пакета.
- Включите `openclaw.plugin.json`. Для плагинов с кодом также требуется `package.json` с
  `openclaw.compat.pluginApi` и `openclaw.build.openclawVersion`.
- Чтобы отобразить собственный значок карточки плагина, добавьте `icon` в `openclaw.plugin.json`,
  указав любой URL изображения по протоколу HTTPS.
- Включите сведения о репозитории исходного кода и точном коммите либо запустите CLI из рабочей копии,
  связанной с GitHub, чтобы эти сведения были определены автоматически.
- Перед публикацией выполните `clawhub package validate <source>`. Способы устранения проблем с пакетом,
  манифестом, импортом SDK или артефактами см. в разделе
  [«Устранение ошибок проверки плагинов»](/clawhub/plugin-validation-fixes).
- Перед созданием выпуска выполните `clawhub package publish <source> --dry-run`.
- Учитывайте, что новые выпуски могут быть недоступны в общедоступных средствах установки до завершения
  автоматических проверок безопасности и верификации.

### Доверенная публикация пакетов

Настройка доверенной публикации пакетов выполняется в два этапа:

1. Один раз опубликуйте пакет обычным способом вручную или с аутентификацией по токену с помощью
   `clawhub package publish`. При этом создаётся запись пакета и определяются управляющие пакетом,
   которые могут изменять его конфигурацию доверенного издателя.
2. Управляющий пакетом задаёт конфигурацию доверенного издателя GitHub Actions:

```bash
clawhub package trusted-publisher set @owner/package-name \
  --repository owner/repo \
  --workflow-filename package-publish.yml
```

После настройки конфигурации последующие поддерживаемые публикации через GitHub Actions могут использовать OIDC и доверенную публикацию без хранения долгосрочного токена ClawHub в репозитории. Настроенные репозиторий и имя файла рабочего процесса должны соответствовать утверждению OIDC GitHub Actions. Если также передан параметр `--environment <name>`, утверждение о среде GitHub Actions должно в точности соответствовать этому имени.

При настройке конфигурации доверенного издателя ClawHub проверяет указанный репозиторий GitHub. Общедоступные репозитории можно проверить по общедоступным метаданным GitHub. Для частных репозиториев ClawHub должен иметь доступ GitHub к соответствующему репозиторию, например через будущую установку приложения ClawHub GitHub App или другую авторизованную интеграцию GitHub.

Текущий многократно используемый рабочий процесс публикации пакетов поддерживает доверенную публикацию без секретов для публикаций `workflow_dispatch`, если доступно разрешение `id-token: write`. Для реальных публикаций при отправке тегов по-прежнему требуется `clawhub_token`, поэтому оставьте `CLAWHUB_TOKEN` доступным для выпусков по тегам, первых публикаций, недоверенных пакетов и аварийных публикаций.

Просмотреть или удалить конфигурацию можно следующими командами:

```bash
clawhub package trusted-publisher get @owner/package-name
clawhub package trusted-publisher delete @owner/package-name
```

Удаление конфигурации доверенного издателя служит способом отката. Оно отключает выпуск токенов для последующих доверенных публикаций, пока управляющий пакетом не настроит конфигурацию заново.

## Часто задаваемые вопросы

### Область имён пакета должна соответствовать выбранному владельцу

Если область имён пакета не соответствует выбранному владельцу, ClawHub отклоняет публикацию:

```text
Package scope "@openclaw" must match selected owner "@vintageayu".
Publish as "@openclaw" or rename this package to "@vintageayu/dronzer".
```

Чтобы устранить ошибку, выберите владельца, указанного в области имён пакета, либо переименуйте пакет так, чтобы его область имён соответствовала владельцу, от имени которого вы можете публиковать.

Если имя пакета уже содержит правильную область имён, но пакет принадлежит другому издателю, передайте право владения:

```sh
clawhub package transfer @opik/opik-openclaw --to opik
```

Передавайте пакет или skill только при наличии административного доступа как к текущему владельцу, так и к целевому издателю. Передача пакета не позволяет публиковать материалы в области имён, которой вы не можете управлять.

Если у вас нет доступа к текущему владельцу, но вы считаете, что ваша организация, проект или бренд является законным владельцем пространства имён, создайте
[запрос на закрепление организации или пространства имён](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml),
приложив общедоступные неконфиденциальные доказательства для проверки сотрудниками. Перед отправкой запроса ознакомьтесь с разделом
[«Закрепление организаций и пространств имён»](/clawhub/namespace-claims).

Это защищает пространства имён организаций. Пакет с именем `@openclaw/dronzer` претендует на пространство имён
`@openclaw`, поэтому опубликовать его могут только издатели, имеющие доступ к владельцу `@openclaw`.
