# Сбой Node + tsx «\_\_name не является функцией»

## Статус

Устранено. Этот сбой не воспроизводится с текущей версией `tsx`, закреплённой в
`package.json` (`4.22.3`), или с актуальными выпусками Node. Эта страница сохранена на случай,
если будущее обновление `tsx`/esbuild снова приведёт к этой ошибке.

## Исходный симптом

Запуск сценариев разработки OpenClaw через `tsx` завершался ошибкой при запуске:

```text
[openclaw] Failed to start CLI: TypeError: __name is not a function
    at createSubsystemLogger (src/logging/subsystem.ts)
    at <caller> (src/agents/auth-profiles/constants.ts)
```

Номера строк опущены: оба файла изменились после первоначального сбоя,
и конкретные строки больше не совпадают.

Проблема появилась после перехода сценариев разработки с Bun на `tsx` (`2871657e`,
2026-01-06), чтобы сделать Bun необязательным. Эквивалентный путь на основе Bun не приводил к сбою.
Изначально проблема наблюдалась с Node v25.3.0 в macOS; предполагалось, что она также
может затронуть другие платформы, использующие Node 25.

## Причина

`tsx` преобразует TS/ESM с помощью esbuild, жёстко задавая `keepNames: true`
в параметрах преобразования. Этот параметр заставляет esbuild оборачивать именованные объявления
функций и классов в вызов вспомогательной функции `__name`, чтобы `fn.name` сохранялось после минификации
и сборки. Сбой означает, что во время вызова в этом модуле вспомогательная функция отсутствовала
или была перекрыта в затронутой комбинации `tsx`/Node, поэтому `__name(...)`
выбрасывала исключение вместо возврата обёрнутого значения.

## Текущая проверка воспроизведения

```bash
node --version
pnpm install
node --import tsx src/entry.ts status
```

Минимальный изолированный пример воспроизведения (загружает только модуль из исходной трассировки стека):

```bash
node --import tsx scripts/repro/tsx-name-repro.ts
```

Сейчас обе команды завершаются успешно. Если какая-либо из них снова выдаст ошибку `__name is not a
function`, перед отправкой отчёта в вышестоящий проект зафиксируйте точную версию Node, версию `tsx`
(`node_modules/tsx/package.json`) и полную трассировку стека.

## Обходные решения (если сбой вернётся)

- Запускайте сценарии разработки с Bun вместо `node --import tsx`.
- Выполните `pnpm tsgo` для проверки типов, а затем запускайте собранный результат
  вместо исходного кода через `tsx`:

  ```bash
  pnpm tsgo
  node openclaw.mjs status
  ```

- Попробуйте другую версию `tsx` (`pnpm add -D tsx@<version>` изменяет зависимости
  и согласно политике репозитория требует одобрения), чтобы методом деления пополам определить, не вернула ли
  входящая в неё версия esbuild эту ошибку.
- Проверьте другую основную или дополнительную версию Node, чтобы определить,
  зависит ли ошибка от версии.
