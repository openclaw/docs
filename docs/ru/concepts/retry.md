## Цели

- Выполнять повторные попытки для каждого HTTP-запроса, а не для всего многоэтапного процесса.
- Сохранять порядок, повторяя только текущий шаг.
- Избегать дублирования неидемпотентных операций.

## Значения по умолчанию

| Параметр                    | Значение по умолчанию |
| --------------------------- | --------------------- |
| Количество попыток          | 3                     |
| Максимальная задержка       | 30000 мс              |
| Джиттер                     | 0.1 (10%)             |
| Минимальная задержка Telegram | 400 мс              |
| Минимальная задержка Discord  | 500 мс              |

## Поведение

### Провайдеры моделей

- OpenClaw позволяет SDK провайдеров самостоятельно выполнять обычные кратковременные повторные попытки.
- Для SDK на основе Stainless, таких как Anthropic и OpenAI, ответы, допускающие повторную попытку (`408`, `409`, `429` и `5xx`), могут содержать `retry-after-ms` или `retry-after`. Если время ожидания превышает 60 секунд, OpenClaw добавляет `x-should-retry: false`, чтобы SDK немедленно вернул ошибку, а механизм переключения моделей мог перейти к другому профилю аутентификации или резервной модели.
- Измените ограничение с помощью `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS=<seconds>`. Установите значение `0`, `false`, `off`, `none` или `disabled`, чтобы SDK самостоятельно учитывали длительные задержки из `Retry-After`.

### Discord

- Повторные попытки выполняются при ошибках ограничения частоты запросов (HTTP 429), тайм-аутах запросов, ответах HTTP 5xx и временных сбоях транспорта, таких как ошибки DNS-разрешения, сбросы соединения, закрытия сокетов и ошибки получения данных.
- Использует значение Discord `retry_after`, если оно доступно, иначе — экспоненциальную задержку.

### Telegram

- Повторные попытки выполняются при временных ошибках (429, тайм-аут, ошибки подключения, сброса или закрытия соединения, временная недоступность).
- Использует `retry_after`, если оно доступно, иначе — экспоненциальную задержку.
- При ошибках разбора HTML/Markdown повторные попытки не выполняются; при первой попытке используется резервная отправка в виде обычного текста.

## Конфигурация

Настройте политику повторных попыток для каждого провайдера в `~/.openclaw/openclaw.json`:

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
    discord: {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
  },
}
```

## Примечания

- Повторные попытки применяются к каждому запросу (отправка сообщения, загрузка медиафайла, реакция, опрос, стикер).
- В составных процессах завершённые шаги повторно не выполняются.
