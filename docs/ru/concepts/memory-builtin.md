Встроенный движок — это серверная часть памяти по умолчанию. Он хранит индекс памяти в отдельной для каждого агента базе данных SQLite и не требует дополнительных зависимостей для начала работы.

## Возможности

- **Поиск по ключевым словам** с помощью полнотекстового индексирования FTS5 (оценка BM25).
- **Векторный поиск** с помощью эмбеддингов от любого поддерживаемого провайдера.
- **Гибридный поиск**, объединяющий оба подхода для получения наилучших результатов.
- **Поддержка CJK** с помощью триграммной токенизации для китайского, японского и корейского языков.
- **Ускорение с помощью sqlite-vec** для выполнения векторных запросов непосредственно в базе данных (необязательно).

## Начало работы

По умолчанию встроенный движок использует эмбеддинги OpenAI. Если `OPENAI_API_KEY` или `models.providers.openai.apiKey` уже настроен, векторный поиск работает без дополнительной настройки памяти.

Чтобы явно указать провайдера:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
      },
    },
  },
}
```

Без провайдера эмбеддингов доступен только поиск по ключевым словам.

Чтобы принудительно использовать локальные эмбеддинги GGUF, установите официальный Plugin провайдера llama.cpp, а затем укажите в `local.modelPath` путь к файлу GGUF:

```bash
openclaw plugins install @openclaw/llama-cpp-provider
```

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "local",
        fallback: "none",
        local: {
          modelPath: "~/.node-llama-cpp/models/embeddinggemma-300m-qat-Q8_0.gguf",
        },
      },
    },
  },
}
```

## Как работает индексирование

OpenClaw разбивает `MEMORY.md` и `memory/*.md` на фрагменты (по умолчанию 400 токенов с перекрытием в 80 токенов), индексирует их и сохраняет в отдельной для каждого агента базе данных SQLite.

- **Расположение индекса:** база данных соответствующего агента по пути `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`
- **Обслуживание хранилища:** размер вспомогательных WAL-файлов SQLite ограничивается периодическими контрольными точками и контрольными точками при завершении работы.
- **Отслеживание файлов:** изменения файлов памяти запускают повторное индексирование с устранением дребезга (по умолчанию задержка составляет 1,5 с).
- **Автоматическое повторное индексирование:** индекс автоматически перестраивается при изменении провайдера эмбеддингов, модели, конфигурации разбиения на фрагменты, настроенных источников или области действия.
- **Повторное индексирование по запросу:** `openclaw memory index --force`

<Info>
С помощью `memorySearch.extraPaths` также можно индексировать файлы Markdown за пределами рабочего пространства. См. [справочник по конфигурации](/ru/reference/memory-config#additional-memory-paths).
</Info>

## Когда использовать

Встроенный движок подходит большинству пользователей:

- Работает сразу после установки без дополнительных зависимостей.
- Хорошо справляется с поиском по ключевым словам и векторным поиском.
- Поддерживает всех провайдеров эмбеддингов.
- Гибридный поиск объединяет преимущества обоих подходов к извлечению данных.

Рассмотрите переход на [QMD](/ru/concepts/memory-qmd), если вам требуется переранжирование, расширение запросов или индексирование каталогов за пределами рабочего пространства.

Рассмотрите [Honcho](/ru/concepts/memory-honcho), если вам нужна память между сеансами с автоматическим моделированием пользователя.

## Устранение неполадок

**Поиск по памяти отключён?** Проверьте `openclaw memory status`. Если провайдер не обнаружен, укажите его явно или добавьте ключ API.

**Локальный провайдер не обнаружен?** Убедитесь, что локальный путь существует, и выполните:

```bash
openclaw memory status --deep --agent main
openclaw memory index --force --agent main
```

Как отдельные команды CLI, так и Gateway используют один и тот же идентификатор локального провайдера — `local`. Если вы хотите использовать локальные эмбеддинги, задайте `memorySearch.provider: "local"`.

**Результаты устарели?** Выполните `openclaw memory index --force`, чтобы перестроить индекс. В редких пограничных случаях средство отслеживания может пропустить изменения.

**sqlite-vec не загружается?** OpenClaw автоматически переключается на вычисление косинусного сходства внутри процесса. Команда `openclaw memory status --deep` сообщает состояние локального векторного хранилища отдельно от провайдера эмбеддингов: `Vector store: unavailable` указывает на проблему загрузки sqlite-vec, а `Embeddings: unavailable` — на проблему с провайдером, аутентификацией или готовностью модели. Конкретную ошибку загрузки можно найти в журналах.
