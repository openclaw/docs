Когда обычный запрос поступает в момент, когда выполнение в сеансе уже транслируется, а режим очереди — `steer` (по умолчанию, настройка не требуется), OpenClaw пытается передать этот запрос в активную среду выполнения. OpenClaw и нативная обвязка сервера приложений Codex реализуют детали доставки по-разному.

На этой странице описано перенаправление обычных входящих сообщений в режиме очереди `steer`. В режимах `followup` и `collect` обычные сообщения не проходят по этому пути и ожидают завершения активного выполнения. Явная команда `/steer <message>` описана в разделе [Перенаправление](/ru/tools/steer).

## Граница среды выполнения

Перенаправление не прерывает уже выполняющийся вызов инструмента. OpenClaw проверяет наличие сообщений перенаправления в очереди на границах обращений к модели:

1. Ассистент запрашивает вызовы инструментов.
2. OpenClaw выполняет пакет вызовов инструментов из текущего сообщения ассистента.
3. OpenClaw отправляет событие завершения хода.
4. OpenClaw извлекает из очереди сообщения перенаправления.
5. OpenClaw добавляет эти сообщения как пользовательские перед следующим вызовом LLM.

Благодаря этому результаты инструментов остаются связаны с запросившим их сообщением ассистента, а следующий вызов модели получает последние пользовательские данные.

Нативная обвязка сервера приложений Codex предоставляет `turn/steer` вместо внутренней очереди перенаправления среды выполнения OpenClaw. OpenClaw объединяет запросы из очереди в течение настроенного периода ожидания, а затем отправляет единый запрос `turn/steer` со всеми собранными пользовательскими данными в порядке их поступления.

Ходы проверки Codex и ручного Compaction отклоняют перенаправление в рамках того же хода. Если среда выполнения не может принять перенаправление в режиме `steer`, OpenClaw ожидает завершения активного выполнения, прежде чем запустить запрос.

## Режимы

| Режим      | Поведение при активном выполнении                                          | Последующее поведение                                                                                       |
| ----------- | -------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `steer`     | По возможности перенаправляет запрос в активную среду выполнения.          | Если перенаправление недоступно, ожидает завершения активного выполнения.                                   |
| `followup`  | Не перенаправляет запрос.                                                  | Выполняет сообщения из очереди позже, после завершения активного выполнения.                                |
| `collect`   | Не перенаправляет запрос.                                                  | Объединяет совместимые сообщения из очереди в один последующий ход после истечения интервала устранения дребезга. |
| `interrupt` | Прерывает активное выполнение вместо перенаправления запроса.              | После прерывания запускает самое новое сообщение.                                                           |

## Пример серии сообщений

Если четыре пользователя отправляют сообщения, пока агент выполняет вызов инструмента:

- При поведении по умолчанию активная среда выполнения получает все четыре сообщения в порядке поступления до следующего решения модели. OpenClaw извлекает их из очереди на следующей границе обращения к модели; Codex получает их в одном пакетном запросе `turn/steer`.
- При `/queue collect` OpenClaw не выполняет перенаправление. Он ожидает завершения активного выполнения, а затем, после истечения интервала устранения дребезга, создаёт последующий ход с совместимыми сообщениями из очереди.
- При `/queue interrupt` OpenClaw прерывает активное выполнение и запускает самое новое сообщение вместо перенаправления.

## Область действия

Перенаправление всегда нацелено на текущее активное выполнение сеанса. Оно не создаёт новый сеанс, не изменяет политику использования инструментов активного выполнения и не разделяет сообщения по отправителям. Во многопользовательских каналах входящие запросы уже содержат контекст отправителя и маршрута, поэтому следующий вызов модели может определить, кто отправил каждое сообщение.

Используйте `followup` или `collect`, если сообщения по умолчанию должны помещаться в очередь, а не перенаправляться в активное выполнение. Используйте `interrupt`, если самый новый запрос должен заменить активное выполнение.

## Устранение дребезга

`messages.queue.debounceMs` применяется к доставке сообщений из очереди в режимах `followup` и `collect`. В режиме `steer` с нативной обвязкой Codex этот параметр также задаёт период ожидания перед отправкой пакетного запроса `turn/steer`. В OpenClaw активное перенаправление не использует таймер устранения дребезга, поскольку OpenClaw естественным образом объединяет сообщения до следующей границы обращения к модели.
