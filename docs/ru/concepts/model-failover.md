OpenClaw обрабатывает сбои в два этапа:

1. **Ротация профилей аутентификации** в рамках текущего провайдера.
2. **Резервный переход модели** к следующей модели в `agents.defaults.model.fallbacks`.

В этом документе объясняются правила runtime и данные, на которых они основаны.

## Поток runtime

Для обычного текстового запуска OpenClaw оценивает кандидатов в таком порядке:

<Steps>
  <Step title="Определение состояния сессии">
    Определяет активную модель сессии и предпочтение профиля аутентификации.
  </Step>
  <Step title="Построение цепочки кандидатов">
    Строит цепочку моделей-кандидатов на основе текущего выбора модели и политики резервного перехода для источника этого выбора. Настроенные значения по умолчанию, основные модели cron-задач и автоматически выбранные резервные модели могут использовать настроенные резервные варианты; явный пользовательский выбор в сессии является строгим.
  </Step>
  <Step title="Проба текущего провайдера">
    Пробует текущего провайдера с учетом правил ротации и cooldown для профилей аутентификации.
  </Step>
  <Step title="Переход при ошибках, допускающих failover">
    Если этот провайдер исчерпан с ошибкой, допускающей failover, переходит к следующей модели-кандидату.
  </Step>
  <Step title="Сохранение резервного переопределения">
    Сохраняет выбранное резервное переопределение до начала повторной попытки, чтобы другие читатели сессии видели того же провайдера/модель, которые runner собирается использовать. Сохраненное переопределение модели помечается как `modelOverrideSource: "auto"`.
  </Step>
  <Step title="Узкий откат при сбое">
    Если резервный кандидат завершается ошибкой, откатывает только поля переопределения сессии, принадлежащие fallback, если они все еще соответствуют этому неудачному кандидату.
  </Step>
  <Step title="Выброс FallbackSummaryError при исчерпании">
    Если все кандидаты завершились ошибкой, выбрасывает `FallbackSummaryError` с деталями по каждой попытке и ближайшим временем окончания cooldown, если оно известно.
  </Step>
</Steps>

Это намеренно уже, чем «сохранить и восстановить всю сессию». Reply runner сохраняет только поля выбора модели, которыми он владеет для fallback:

- `providerOverride`
- `modelOverride`
- `modelOverrideSource`
- `authProfileOverride`
- `authProfileOverrideSource`
- `authProfileOverrideCompactionCount`

Это предотвращает ситуацию, когда неудачная повторная попытка fallback перезаписывает более новые несвязанные изменения сессии, например ручные изменения `/model` или обновления ротации сессии, произошедшие во время выполнения попытки.

## Политика источника выбора

OpenClaw разделяет выбранного провайдера/модель и причину, по которой они были выбраны. Этот источник управляет тем, разрешена ли цепочка fallback:

- **Настроенное значение по умолчанию**: `agents.defaults.model.primary` использует `agents.defaults.model.fallbacks`.
- **Основная модель агента**: `agents.list[].model` является строгой, если объект модели этого агента не включает собственные `fallbacks`. Используйте `fallbacks: []`, чтобы сделать строгое поведение явным, или укажите непустой список, чтобы включить fallback моделей для этого агента.
- **Автоматическое резервное переопределение**: runtime fallback записывает `providerOverride`, `modelOverride`, `modelOverrideSource: "auto"` и выбранную исходную модель перед повторной попыткой. Это автоматическое переопределение может продолжать идти по настроенной цепочке fallback без проверки основной модели при каждом сообщении, но OpenClaw периодически снова проверяет настроенный источник и очищает автоматическое переопределение, когда он восстанавливается. `/new`, `/reset` и `sessions.reset` также очищают переопределения с источником auto. Запуски Heartbeat без явного `heartbeat.model` очищают прямые автоматические переопределения, когда их источник больше не совпадает с текущим настроенным значением по умолчанию.
- **Пользовательское переопределение сессии**: `/model`, средство выбора модели, `session_status(model=...)` и `sessions.patch` записывают `modelOverrideSource: "user"`. Это точный выбор для сессии. Если выбранный провайдер/модель завершается ошибкой до формирования ответа, OpenClaw сообщает о сбое вместо ответа от несвязанного настроенного fallback.
- **Устаревшее переопределение сессии**: старые записи сессий могут иметь `modelOverride` без `modelOverrideSource`. OpenClaw трактует их как пользовательские переопределения, чтобы явный старый выбор не был незаметно преобразован в поведение fallback.
- **Модель в Cron payload**: `payload.model` / `--model` cron-задачи является основной моделью задания, а не пользовательским переопределением сессии. Она использует настроенные fallback, если задание не предоставляет `payload.fallbacks`; `payload.fallbacks: []` делает cron-запуск строгим.

Интервал проверки основной модели для автоматического fallback составляет пять минут и не настраивается. OpenClaw запоминает недавние проверки по сессии и основной модели, чтобы не повторять попытку с падающей основной моделью на каждом ходе. OpenClaw отправляет видимое уведомление, когда сессия переходит на fallback, и еще одно уведомление, когда она возвращается к выбранной основной модели; он не повторяет уведомление на каждом закрепленном fallback-ходе.

## Кэш пропуска ошибок аутентификации

По умолчанию каждый новый ход сохраняет существующее поведение повторных попыток fallback: OpenClaw
снова попробует каждого настроенного fallback-кандидата, включая неосновных
кандидатов, которые недавно завершились ошибкой `auth` или `auth_permanent`.

Операторы, которые предпочитают подавлять такие повторяющиеся ошибки аутентификации, могут включить это с помощью:

```bash
OPENCLAW_FALLBACK_SKIP_TTL_MS=60000
```

Когда это включено, OpenClaw записывает в памяти маркер пропуска, ограниченный
сессией, для неосновного fallback-кандидата после ошибки класса auth. Маркер
ключуется по id сессии, провайдеру и модели. Основные кандидаты никогда не
пропускаются, поэтому явный пользовательский выбор модели по-прежнему показывает
реальную ошибку аутентификации. Кэш локален для процесса и очищается при
перезапуске Gateway.

Значение является TTL в миллисекундах. `0` или неустановленное значение отключает кэш.
Положительные значения ограничиваются диапазоном от 1 секунды до 10 минут.

## Видимые пользователю уведомления fallback

Когда сессия переходит на автоматически выбранный fallback, OpenClaw отправляет уведомление о статусе в той же поверхности ответа:

```text
↪️ Model Fallback: <fallback> (selected <primary>; <reason>)
```

Когда более поздняя проверка проходит успешно и сессия возвращается к выбранной основной модели, OpenClaw отправляет:

```text
↪️ Model Fallback cleared: <primary> (was <fallback>)
```

Эти уведомления являются операционными сообщениями, а не содержимым assistant. Они доставляются один раз при каждом изменении состояния, включая ходы только с побочными эффектами, когда это возможно, но закрепленные fallback-ходы их не повторяют. Доставка обходит обычное подавление ответа источника, уведомление не занимает первый слот ответа assistant для каналов с потоками и исключается из text-to-speech и извлечения обязательств.

## Хранилище аутентификации (ключи + OAuth)

OpenClaw использует **профили аутентификации** как для API-ключей, так и для токенов OAuth.

- Секреты и состояние маршрутизации аутентификации runtime находятся в `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`.
- Конфиг `auth.profiles` / `auth.order` — это **только metadata + routing** (без секретов).
- Устаревший файл OAuth только для импорта: `~/.openclaw/credentials/oauth.json` (импортируется в хранилище аутентификации конкретного агента при первом использовании).
- Устаревшие файлы `auth-profiles.json`, `auth-state.json` и `auth.json` для отдельных агентов импортируются командой `openclaw doctor --fix`.

Подробнее: [OAuth](/ru/concepts/oauth)

Типы учетных данных:

- `type: "api_key"` → `{ provider, key }`
- `type: "oauth"` → `{ provider, access, refresh, expires, email? }` (+ `projectId`/`enterpriseUrl` для некоторых провайдеров)

## ID профилей

Входы через OAuth создают отдельные профили, чтобы несколько учетных записей могли сосуществовать.

- По умолчанию: `provider:default`, когда email недоступен.
- OAuth с email: `provider:<email>` (например, `google-antigravity:user@gmail.com`).

Профили находятся в хранилище профилей аутентификации `openclaw-agent.sqlite` конкретного агента.

## Порядок ротации

Когда у провайдера есть несколько профилей, OpenClaw выбирает порядок так:

<Steps>
  <Step title="Явная конфигурация">
    `auth.order[provider]` (если задано).
  </Step>
  <Step title="Настроенные профили">
    `auth.profiles`, отфильтрованные по провайдеру.
  </Step>
  <Step title="Сохраненные профили">
    Записи профилей аутентификации провайдера в SQLite конкретного агента.
  </Step>
</Steps>

Если явный порядок не настроен, OpenClaw использует порядок round-robin:

- **Основной ключ:** тип профиля (**OAuth перед API-ключами**).
- **Вторичный ключ:** `usageStats.lastUsed` (самые старые сначала, внутри каждого типа).
- **Профили в cooldown/disabled** перемещаются в конец, упорядоченные по ближайшему окончанию.

### Закрепление в сессии (дружелюбно к кэшу)

OpenClaw **закрепляет выбранный профиль аутентификации за сессией**, чтобы сохранять кэши провайдера прогретыми. Он **не** выполняет ротацию при каждом запросе. Закрепленный профиль используется повторно, пока:

- сессия не сброшена (`/new` / `/reset`)
- не завершится Compaction (счетчик compaction увеличится)
- профиль не находится в cooldown/disabled

Ручной выбор через `/model …@<profileId>` задает **пользовательское переопределение** для этой сессии и не ротируется автоматически до начала новой сессии.

<Note>
Автоматически закрепленные профили (выбранные маршрутизатором сессии) рассматриваются как **предпочтение**: они пробуются первыми, но OpenClaw может перейти к другому профилю при rate limits/timeouts. Когда исходный профиль снова становится доступен, новые запуски могут снова предпочитать его без изменения выбранной модели или runtime. Пользовательски закрепленные профили остаются привязанными к этому профилю; если он завершится ошибкой и настроены fallback моделей, OpenClaw перейдет к следующей модели вместо переключения профилей.
</Note>

### Подписка OpenAI Codex плюс резервный API-ключ

Для агентских моделей OpenAI аутентификация и runtime разделены. `openai/gpt-*` остается на
harness Codex, тогда как аутентификация может ротироваться между профилем подписки Codex и
резервным API-ключом OpenAI.

Используйте `auth.order.openai` для пользовательского порядка:

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

Используйте `openai:*` как для OAuth-профилей ChatGPT/Codex, так и для профилей
API-ключей OpenAI. Когда подписка достигает лимита использования Codex,
OpenClaw записывает точное время сброса, если Codex его предоставляет, пробует следующий
упорядоченный профиль аутентификации и сохраняет запуск внутри harness Codex. После того как время
сброса проходит, профиль подписки снова становится доступным, и следующий автоматический
выбор может вернуться к нему.

Используйте пользовательски закрепленный профиль только тогда, когда хотите принудительно использовать одну учетную запись/ключ для этой
сессии. Пользовательски закрепленные профили намеренно строгие и не переходят незаметно
к другому профилю.

## Cooldowns

Когда профиль завершается ошибкой из-за ошибок аутентификации/rate-limit (или timeout, похожего на rate limiting), OpenClaw помечает его как находящийся в cooldown и переходит к следующему профилю.

<AccordionGroup>
  <Accordion title="Что попадает в корзину rate-limit / timeout">
    Эта корзина rate-limit шире, чем простой `429`: она также включает сообщения провайдеров, такие как `Too many concurrent requests`, `ThrottlingException`, `concurrency limit reached`, `workers_ai ... quota limit exceeded`, `throttled`, `resource exhausted`, и периодические лимиты окон использования, такие как `weekly/monthly limit reached`.

    Ошибки формата/некорректного запроса обычно терминальны, потому что повторная отправка того же payload завершилась бы так же, поэтому OpenClaw показывает их вместо ротации профилей аутентификации. Известные пути retry-repair могут включаться явно: например, ошибки валидации ID вызовов инструментов Cloud Code Assist очищаются и повторяются один раз через политику `allowFormatRetry`. Ошибки stop-reason, совместимые с OpenAI, такие как `Unhandled stop reason: error`, `stop reason: error` и `reason: error`, классифицируются как сигналы timeout/failover.

    Универсальный серверный текст также может попасть в эту корзину timeout, когда источник соответствует известному transient pattern. Например, простое сообщение stream-wrapper runtime модели `An unknown error occurred` считается допускающим failover для каждого провайдера, потому что общий runtime модели выдает его, когда потоки провайдера завершаются с `stopReason: "aborted"` или `stopReason: "error"` без конкретных деталей. JSON payload `api_error` с transient server text, таким как `internal server error`, `unknown error, 520`, `upstream error` или `backend error`, также считаются timeout, допускающими failover.

    Специфичный для OpenRouter универсальный upstream text, такой как простое `Provider returned error`, считается timeout только когда контекст провайдера действительно OpenRouter. Универсальный внутренний fallback text, такой как `LLM request failed with an unknown error.`, остается консервативным и сам по себе не запускает failover.

  </Accordion>
  <Accordion title="Ограничения SDK для retry-after">
    Некоторые SDK поставщиков иначе могут ожидать в течение долгого окна `Retry-After`, прежде чем вернуть управление OpenClaw. Для SDK на основе Stainless, таких как Anthropic и OpenAI, OpenClaw по умолчанию ограничивает внутренние ожидания SDK `retry-after-ms` / `retry-after` 60 секундами и сразу выводит более долгие повторяемые ответы наружу, чтобы этот путь отработки отказа мог выполниться. Настройте или отключите это ограничение с помощью `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS`; см. [Поведение повторных попыток](/ru/concepts/retry).
  </Accordion>
  <Accordion title="Ожидания восстановления на уровне модели">
    Ожидания восстановления после ограничений скорости также могут быть привязаны к модели:

    - OpenClaw записывает `cooldownModel` для сбоев из-за ограничения скорости, когда известен идентификатор модели, на которой произошел сбой.
    - Соседнюю модель у того же поставщика все еще можно попробовать, если ожидание восстановления привязано к другой модели.
    - Окна биллинга/отключения все равно блокируют весь профиль для всех моделей.

  </Accordion>
</AccordionGroup>

Ожидания восстановления используют экспоненциальную задержку:

- 1 минута
- 5 минут
- 25 минут
- 1 час (предел)

Состояние хранится в состоянии аутентификации SQLite для отдельного агента в `usageStats`:

```json
{
  "usageStats": {
    "provider:profile": {
      "lastUsed": 1736160000000,
      "cooldownUntil": 1736160600000,
      "errorCount": 2
    }
  }
}
```

## Отключения из-за биллинга

Сбои биллинга/кредитов (например, "insufficient credits" / "credit balance too low") считаются подходящими для отработки отказа, но обычно они не временные. Вместо короткого ожидания восстановления OpenClaw помечает профиль как **отключенный** (с более долгой задержкой) и переключается на следующий профиль/поставщика.

<Note>
Не каждый ответ, похожий на биллинговый, является `402`, и не каждый HTTP `402` попадает сюда. OpenClaw сохраняет явный биллинговый текст в биллинговом пути, даже когда поставщик вместо этого возвращает `401` или `403`, но специализированные сопоставители поставщика остаются ограничены тем поставщиком, которому они принадлежат (например, OpenRouter `403 Key limit exceeded`).

При этом временные ошибки `402`, связанные с окнами использования и лимитами расходов организации/рабочей области, классифицируются как `rate_limit`, когда сообщение выглядит повторяемым (например, `weekly usage limit exhausted`, `daily limit reached, resets tomorrow` или `organization spending limit exceeded`). Они остаются на пути короткого ожидания восстановления/отработки отказа, а не на долгом пути отключения из-за биллинга.
</Note>

Состояние хранится в состоянии аутентификации SQLite для отдельного агента:

```json
{
  "usageStats": {
    "provider:profile": {
      "disabledUntil": 1736178000000,
      "disabledReason": "billing"
    }
  }
}
```

Значения по умолчанию:

- Задержка биллинга начинается с **5 часов**, удваивается при каждом биллинговом сбое и ограничивается **24 часами**.
- Счетчики задержки сбрасываются, если профиль не давал сбоев в течение **24 часов** (настраивается).
- Повторные попытки при перегрузке допускают **1 ротацию профиля того же поставщика** перед откатом к другой модели.
- Повторные попытки при перегрузке по умолчанию используют задержку **0 мс**.

## Откат к другой модели

Если все профили поставщика дают сбой, OpenClaw переходит к следующей модели в `agents.defaults.model.fallbacks`. Это применяется к сбоям аутентификации, ограничениям скорости и тайм-аутам, которые исчерпали ротацию профилей (другие ошибки не продвигают откат). Ошибки поставщика, которые не раскрывают достаточно подробностей, все равно точно помечаются в состоянии отката: `empty_response` означает, что поставщик не вернул пригодное сообщение или статус, `no_error_details` означает, что поставщик явно вернул `Unknown error (no error details in response)`, а `unclassified` означает, что OpenClaw сохранил необработанный предварительный просмотр, но ни один классификатор пока не сопоставил его.

Ошибки перегрузки и ограничения скорости обрабатываются более агрессивно, чем биллинговые ожидания восстановления. По умолчанию OpenClaw допускает одну повторную попытку с профилем аутентификации того же поставщика, затем без ожидания переключается на следующий настроенный откат модели. Сигналы занятости поставщика, такие как `ModelNotReadyException`, попадают в эту корзину перегрузки. Настройте это с помощью `auth.cooldowns.overloadedProfileRotations`, `auth.cooldowns.overloadedBackoffMs` и `auth.cooldowns.rateLimitedProfileRotations`.

Когда запуск начинается с настроенной основной модели по умолчанию, основной модели задания cron, основной модели агента с явными откатами или автоматически выбранного переопределения отката, OpenClaw может пройти по соответствующей настроенной цепочке отката. Основные модели агента без явных откатов и явные пользовательские выборы (например, `/model ollama/qwen3.5:27b`, выбор модели, `sessions.patch` или разовые переопределения поставщика/модели CLI) являются строгими: если этот поставщик/модель недоступны или дают сбой до создания ответа, OpenClaw сообщает о сбое вместо ответа от несвязанного отката.

### Правила цепочки кандидатов

OpenClaw строит список кандидатов из текущего запрошенного `provider/model` плюс настроенных откатов.

<AccordionGroup>
  <Accordion title="Правила">
    - Запрошенная модель всегда идет первой.
    - Явно настроенные откаты дедуплицируются, но не фильтруются по списку разрешенных моделей. Они считаются явным намерением оператора.
    - Если текущий запуск уже находится на настроенном откате в том же семействе поставщиков, OpenClaw продолжает использовать полную настроенную цепочку.
    - Когда явное переопределение отката не предоставлено, настроенные откаты пробуются перед настроенной основной моделью, даже если запрошенная модель использует другого поставщика.
    - Когда явное переопределение отката не предоставлено исполнителю отката, настроенная основная модель добавляется в конец, чтобы цепочка могла вернуться к обычному значению по умолчанию после исчерпания более ранних кандидатов.
    - Когда вызывающий код предоставляет `fallbacksOverride`, исполнитель использует ровно запрошенную модель плюс этот список переопределения. Пустой список отключает откат к другой модели и не позволяет добавить настроенную основную модель как скрытую цель повторной попытки.

  </Accordion>
</AccordionGroup>

### Какие ошибки продвигают откат

<Tabs>
  <Tab title="Продолжается при">
    - сбоях аутентификации
    - ограничениях скорости и исчерпании ожидания восстановления
    - ошибках перегрузки/занятости поставщика
    - ошибках отработки отказа, похожих на тайм-аут
    - отключениях из-за биллинга
    - `LiveSessionModelSwitchError`, которая нормализуется в путь отработки отказа, чтобы устаревшая сохраненная модель не создавала внешний цикл повторных попыток
    - других нераспознанных ошибках, когда еще остаются кандидаты

  </Tab>
  <Tab title="Не продолжается при">
    - явных прерываниях, которые не похожи на тайм-аут/отработку отказа
    - ошибках переполнения контекста, которые должны оставаться внутри логики Compaction/повторных попыток (например, `request_too_large`, `INVALID_ARGUMENT: input exceeds the maximum number of tokens`, `input token count exceeds the maximum number of input tokens`, `The input is too long for the model` или `ollama error: context length exceeded`)
    - финальной неизвестной ошибке, когда кандидатов больше нет
    - отказах безопасности Claude Fable 5; прямые запросы с API-ключом обрабатывают их на уровне поставщика через серверный откат Anthropic к `claude-opus-4-8` вместо этого (см. [Anthropic](/ru/providers/anthropic#safety-refusal-fallback-claude-fable-5))

  </Tab>
</Tabs>

### Пропуск ожидания восстановления и поведение проб

Когда каждый профиль аутентификации поставщика уже находится в ожидании восстановления, OpenClaw не пропускает этого поставщика автоматически навсегда. Он принимает решение для каждого кандидата:

<AccordionGroup>
  <Accordion title="Решения для каждого кандидата">
    - Постоянные сбои аутентификации сразу пропускают всего поставщика.
    - Отключения из-за биллинга обычно пропускаются, но основной кандидат все еще может проверяться с ограничением частоты, чтобы восстановление было возможным без перезапуска.
    - Основной кандидат может проверяться ближе к истечению ожидания восстановления, с ограничением частоты для каждого поставщика.
    - Соседние откаты того же поставщика можно пробовать несмотря на ожидание восстановления, когда сбой выглядит временным (`rate_limit`, `overloaded` или неизвестный). Это особенно актуально, когда ограничение скорости привязано к модели, а соседняя модель все еще может восстановиться немедленно.
    - Временные пробы ожидания восстановления ограничены одной на поставщика для каждого запуска отката, чтобы один поставщик не задерживал откат между поставщиками.

  </Accordion>
</AccordionGroup>

## Переопределения сеанса и живое переключение модели

Изменения модели сеанса являются общим состоянием. Активный исполнитель, команда `/model`, обновления Compaction/сеанса и согласование живого сеанса читают или записывают части одной и той же записи сеанса.

Это означает, что повторные попытки отката должны координироваться с живым переключением модели:

- Только явные изменения модели, инициированные пользователем, помечают ожидающее живое переключение. Это включает `/model`, `session_status(model=...)` и `sessions.patch`.
- Системные изменения модели, такие как ротация отката, переопределения Heartbeat или Compaction, сами по себе никогда не помечают ожидающее живое переключение.
- Пользовательские переопределения модели считаются точным выбором для политики отката, поэтому недоступный выбранный поставщик показывается как сбой, а не маскируется через `agents.defaults.model.fallbacks`.
- Перед запуском повторной попытки отката исполнитель ответа сохраняет выбранные поля переопределения отката в записи сеанса.
- Автоматические переопределения отката остаются выбранными на последующих ходах, чтобы OpenClaw не проверял известную неисправную основную модель при каждом сообщении. OpenClaw периодически снова проверяет настроенный источник и очищает автоматическое переопределение, когда он восстанавливается; `/new`, `/reset` и `sessions.reset` немедленно очищают переопределения с автоматическим источником.
- Ответы пользователю объявляют о переходах отката и восстановлении после очистки отката один раз на изменение состояния. Ходы с закрепленным откатом не повторяют уведомление.
- `/status` показывает выбранную модель и, когда состояние отката отличается, активную модель отката и причину.
- Согласование живого сеанса предпочитает сохраненные переопределения сеанса устаревшим полям модели времени выполнения.
- Если ошибка живого переключения указывает на более позднего кандидата в активной цепочке отката, OpenClaw переходит напрямую к этой выбранной модели вместо того, чтобы сначала проходить несвязанных кандидатов.
- Если попытка отката завершается сбоем, исполнитель откатывает только те поля переопределения, которые он записал, и только если они все еще совпадают с этим неудачным кандидатом.

Это предотвращает классическую гонку:

<Steps>
  <Step title="Основная модель дает сбой">
    Выбранная основная модель дает сбой.
  </Step>
  <Step title="Откат выбран в памяти">
    Кандидат отката выбран в памяти.
  </Step>
  <Step title="Хранилище сеанса все еще указывает старую основную модель">
    Хранилище сеанса все еще отражает старую основную модель.
  </Step>
  <Step title="Живое согласование читает устаревшее состояние">
    Согласование живого сеанса читает устаревшее состояние сеанса.
  </Step>
  <Step title="Повторная попытка возвращена назад">
    Повторная попытка возвращается к старой модели до начала попытки отката.
  </Step>
</Steps>

Сохраненное переопределение отката закрывает это окно, а узкий откат сохраняет более новые ручные или runtime-изменения сеанса нетронутыми.

## Наблюдаемость и сводки сбоев

`runWithModelFallback(...)` записывает подробности каждой попытки, которые питают журналы и пользовательские сообщения об ожидании восстановления:

- поставщик/модель, которые пробовались
- причина (`rate_limit`, `overloaded`, `billing`, `auth`, `model_not_found` и похожие причины отработки отказа)
- необязательный статус/код
- понятная человеку сводка ошибки

Структурированные журналы `model_fallback_decision` также включают плоские поля `fallbackStep*`, когда кандидат дает сбой, пропускается или более поздний откат успешен. Эти поля делают попытку перехода явной (`fallbackStepFromModel`, `fallbackStepToModel`, `fallbackStepFromFailureReason`, `fallbackStepFromFailureDetail`, `fallbackStepFinalOutcome`), чтобы экспортеры журналов и диагностики могли восстановить сбой основной модели, даже когда терминальный откат тоже завершается сбоем.

Когда все кандидаты дают сбой, OpenClaw выбрасывает `FallbackSummaryError`. Внешний исполнитель ответа может использовать это, чтобы построить более конкретное сообщение, например "все модели временно ограничены по скорости", и включить ближайшее истечение ожидания восстановления, если оно известно.

Эта сводка ожидания восстановления учитывает модель:

- ограничения скорости, привязанные к несвязанным моделям, игнорируются для цепочки поставщика/модели, которая пробовалась
- если оставшаяся блокировка является совпадающим ограничением скорости на уровне модели, OpenClaw сообщает последнее совпадающее истечение, которое все еще блокирует эту модель
