Dreaming — это фоновая система консолидации памяти в `memory-core`. Она помогает OpenClaw переносить сильные краткосрочные сигналы в долговременную память, сохраняя процесс объяснимым и доступным для проверки.

<Note>
Dreaming является **опциональным** и по умолчанию отключен.
</Note>

## Что записывает Dreaming

Dreaming сохраняет два вида вывода:

- **Машинное состояние** в `memory/.dreams/` (хранилище recall, фазовые сигналы, контрольные точки ingestion, блокировки).
- **Вывод для чтения человеком** в `DREAMS.md` (или существующем `dreams.md`) и необязательные файлы фазовых отчетов в `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Долгосрочное продвижение по-прежнему записывает данные только в `MEMORY.md`.

## Модель фаз

Dreaming использует три совместно работающие фазы:

| Фаза  | Назначение                                        | Долговременная запись |
| ----- | ------------------------------------------------- | --------------------- |
| Light | Сортировка и подготовка свежего краткосрочного материала | Нет                   |
| Deep  | Оценка и продвижение долговременных кандидатов    | Да (`MEMORY.md`)      |
| REM   | Анализ тем и повторяющихся идей                   | Нет                   |

Эти фазы являются внутренними деталями реализации, а не отдельными настраиваемыми пользователем «режимами».

<AccordionGroup>
  <Accordion title="Фаза Light">
    Фаза Light принимает свежие ежедневные сигналы памяти и трассы recall, удаляет дубликаты и подготавливает строки-кандидаты.

    - Читает краткосрочное состояние recall, свежие ежедневные файлы памяти и отредактированные стенограммы сессий, когда они доступны.
    - Записывает управляемый блок `## Light Sleep`, когда хранилище включает встроенный вывод.
    - Записывает сигналы подкрепления для последующего ранжирования в фазе Deep.
    - Никогда не записывает в `MEMORY.md`.

  </Accordion>
  <Accordion title="Фаза Deep">
    Фаза Deep решает, что становится долгосрочной памятью.

    - Ранжирует кандидатов с помощью взвешенной оценки и пороговых фильтров.
    - Требует прохождения `minScore`, `minRecallCount` и `minUniqueQueries`.
    - Перед записью повторно загружает фрагменты из актуальных ежедневных файлов, поэтому устаревшие или удаленные фрагменты пропускаются.
    - Добавляет продвинутые записи в `MEMORY.md`.
    - Записывает сводку `## Deep Sleep` в `DREAMS.md` и при необходимости записывает `memory/dreaming/deep/YYYY-MM-DD.md`.

  </Accordion>
  <Accordion title="Фаза REM">
    Фаза REM извлекает паттерны и рефлексивные сигналы.

    - Формирует сводки тем и рефлексий из свежих краткосрочных трасс.
    - Записывает управляемый блок `## REM Sleep`, когда хранилище включает встроенный вывод.
    - Записывает сигналы подкрепления REM, используемые ранжированием Deep.
    - Никогда не записывает в `MEMORY.md`.

  </Accordion>
</AccordionGroup>

## Ingestion стенограмм сессий

Dreaming может принимать отредактированные стенограммы сессий в корпус Dreaming. Когда стенограммы доступны, они передаются в фазу Light вместе с ежедневными сигналами памяти и трассами recall. Персональное и чувствительное содержимое редактируется перед ingestion.

## Дневник снов

Dreaming также ведет повествовательный **Дневник снов** в `DREAMS.md`. После того как каждая фаза получает достаточно материала, `memory-core` запускает best-effort фоновый ход subagent и добавляет короткую запись дневника. Используется модель runtime по умолчанию, если не настроен `dreaming.model`. Если настроенная модель недоступна, Дневник снов один раз повторяет попытку с моделью сессии по умолчанию.

<Note>
Этот дневник предназначен для чтения человеком в интерфейсе Dreams, а не как источник продвижения. Созданные Dreaming артефакты дневников и отчетов исключаются из краткосрочного продвижения. В `MEMORY.md` могут продвигаться только обоснованные фрагменты памяти.
</Note>

Также есть обоснованный исторический канал backfill для проверки и восстановления:

<AccordionGroup>
  <Accordion title="Команды backfill">
    - `memory rem-harness --path ... --grounded` предварительно показывает обоснованный вывод дневника из исторических заметок `YYYY-MM-DD.md`.
    - `memory rem-backfill --path ...` записывает обратимые обоснованные записи дневника в `DREAMS.md`.
    - `memory rem-backfill --path ... --stage-short-term` подготавливает обоснованные долговременные кандидаты в том же краткосрочном хранилище доказательств, которое уже использует обычная фаза Deep.
    - `memory rem-backfill --rollback` и `--rollback-short-term` удаляют эти подготовленные артефакты backfill, не затрагивая обычные записи дневника или активный краткосрочный recall.

  </Accordion>
</AccordionGroup>

Control UI предоставляет тот же поток backfill/reset для дневника, чтобы можно было проверить результаты в сцене Dreams перед решением о том, заслуживают ли обоснованные кандидаты продвижения. Scene также показывает отдельный обоснованный канал, чтобы было видно, какие подготовленные краткосрочные записи поступили из исторического воспроизведения, какие продвинутые элементы были основаны на обоснованных данных, и чтобы можно было очищать только обоснованные подготовленные записи, не затрагивая обычное активное краткосрочное состояние.

## Сигналы ранжирования Deep

Ранжирование Deep использует шесть взвешенных базовых сигналов плюс фазовое подкрепление:

| Сигнал                 | Вес  | Описание                                               |
| ---------------------- | ---- | ------------------------------------------------------ |
| Частота                | 0.24 | Сколько краткосрочных сигналов накопила запись         |
| Релевантность          | 0.30 | Среднее качество retrieval для записи                  |
| Разнообразие запросов  | 0.15 | Разные контексты запроса/дня, в которых она появлялась |
| Актуальность           | 0.15 | Оценка свежести с учетом временного затухания          |
| Консолидация           | 0.10 | Сила повторения в течение нескольких дней              |
| Концептуальная насыщенность | 0.06 | Плотность concept-тегов из фрагмента/пути              |

Попадания фаз Light и REM добавляют небольшой затухающий по времени бонус из `memory/.dreams/phase-signals.json`.

Результаты shadow-trial могут накладываться поверх этой базовой оценки как
сигнал проверки перед любой долговременной записью. Полезный trial дает кандидату небольшой
ограниченный бонус, нейтральный trial оставляет его отложенным, а вредный trial помечает его
как отклоненный для этого прохода оценки. Этот сигнал по-прежнему предназначен только для отчета: он может
изменять порядок кандидатов или метаданные проверки, но не записывает в
`MEMORY.md` и не продвигает кандидата сам по себе.

## Покрытие отчета shadow trial в QA

QA Lab включает сценарий только для отчета, предназначенный для изучения того, как будущий
shadow trial Dreaming мог бы проверять память-кандидат перед продвижением. Сценарий просит
агента сравнить базовый ответ с ответом, который может использовать память-кандидат,
затем записать локальный отчет с вердиктом, причиной и флагами риска.

Это покрытие намеренно ограничено QA. Оно проверяет, что артефакт отчета
остается отдельно от `MEMORY.md` и что агент не утверждает, будто кандидат
был продвинут. Оно не добавляет production-поведение shadow-trial и не меняет
движок продвижения фазы Deep.

Runner shadow-trial в `memory-core` сохраняет тот же контракт «только отчет» для
путей кода, которым нужен стабильный артефакт. Он принимает кандидата, prompt trial,
базовый результат, результат кандидата, вердикт, причину, флаги риска и ссылки
на доказательства, затем записывает отчет с `promotion action: report-only`. Полезные
вердикты отображаются в рекомендацию `promote`, нейтральные вердикты — в `defer`, а
вредные вердикты — в `reject`; ни одна из этих рекомендаций не записывает в
`MEMORY.md` и не применяет продвижение фазы Deep.

## Планирование

Когда включен, `memory-core` автоматически управляет одной cron-задачей для полного прохода Dreaming. Каждый проход запускает фазы по порядку: light → REM → deep.

Проход включает основной workspace runtime и все настроенные workspace агентов, с дедупликацией по пути, поэтому fan-out workspace subagent не исключает `DREAMS.md` и состояние памяти основного агента.

Поведение cadence по умолчанию:

| Настройка            | По умолчанию  |
| -------------------- | ------------- |
| `dreaming.frequency` | `0 3 * * *`   |
| `dreaming.model`     | модель по умолчанию |

## Slash-команда

```
/dreaming status
/dreaming on
/dreaming off
/dreaming help
```

`/dreaming on` и `/dreaming off` изменяют конфигурацию на уровне gateway. Вызывающие
каналы должны быть владельцами, а клиенты Gateway должны иметь `operator.admin`.
`/dreaming status` и `/dreaming help` остаются только для чтения.

## Ключевые значения по умолчанию

Все настройки находятся в `plugins.entries.memory-core.config.dreaming`.

<ParamField path="enabled" type="boolean" default="false">
  Включает или отключает проход Dreaming.
</ParamField>
<ParamField path="frequency" type="string" default="0 3 * * *">
  Cron-cadence для полного прохода Dreaming.
</ParamField>
<ParamField path="model" type="string">
  Необязательное переопределение модели subagent для Дневника снов. Используйте каноническое значение `provider/model`, если также задаете allowlist `allowedModels` для subagent.
</ParamField>
<ParamField path="phases.deep.maxPromotedSnippetTokens" type="number" default="160">
  Максимальное расчетное количество токенов, сохраняемое из каждого краткосрочного фрагмента recall, продвигаемого в `MEMORY.md`. Происхождение ранжирования остается видимым.
</ParamField>

<Warning>
`dreaming.model` требует `plugins.entries.memory-core.subagent.allowModelOverride: true`. Чтобы ограничить его, также задайте `plugins.entries.memory-core.subagent.allowedModels`. Сбои trust или allowlist остаются видимыми вместо тихого fallback; повторная попытка покрывает только ошибки недоступности модели.
</Warning>

<Note>
Большая часть фазовой политики, порогов и поведения хранилища является внутренними деталями реализации. Полный список ключей см. в [справочнике по настройке памяти](/ru/reference/memory-config#dreaming).
</Note>

## Интерфейс Dreams

Когда включен, вкладка **Dreams** в Gateway показывает:

- текущее состояние включения Dreaming
- статус на уровне фаз и наличие управляемого прохода
- счетчики краткосрочных, обоснованных, сигнальных и продвинутых сегодня элементов
- время следующего запланированного запуска
- отдельный обоснованный канал Scene для подготовленных записей исторического воспроизведения
- раскрываемый читатель Дневника снов на основе `doctor.memory.dreamDiary`

## Dreaming никогда не запускается: статус показывает blocked

Если `openclaw memory status` сообщает `Dreaming status: blocked`, управляемый cron существует, но heartbeat агента по умолчанию не срабатывает. Проверьте, что heartbeat включен для агента по умолчанию и что его цель не равна `none`, затем снова выполните `openclaw memory status --deep` после следующего интервала heartbeat.
