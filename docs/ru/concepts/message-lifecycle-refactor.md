## Зачем потребовался этот рефакторинг

Стек каналов вырос из нескольких локальных исправлений: отдельных вспомогательных средств для входящих
сообщений каждого уровня зрелости (`runtime.channel.inbound.run` для простых адаптеров,
`runtime.channel.inbound.runPreparedReply` для расширенных), устаревших вспомогательных
средств диспетчеризации ответов (`dispatchInboundReplyWithBase`, `recordInboundSessionAndDispatchReply`),
потоковой передачи предварительного просмотра для отдельных каналов и обеспечения надёжности
окончательной доставки, добавленного поверх существующих путей полезной нагрузки ответа. Такая
структура привела к чрезмерному количеству общедоступных концепций и мест, где семантика
доставки могла расходиться.

Пробел в надёжности, из-за которого потребовалось перепроектирование:

```text
Telegram polling update acked
  -> assistant final text exists
  -> process restarts before sendMessage succeeds
  -> final response is lost
```

Целевой инвариант: как только ядро решает, что видимое исходящее сообщение должно
существовать, намерение отправки должно быть сохранено до попытки вызова платформы, а
квитанция платформы должна быть зафиксирована после успешного выполнения. По умолчанию это
обеспечивает восстановление с семантикой «хотя бы один раз». Поведение «ровно один раз»
существует только там, где адаптер подтверждает нативную идемпотентность или сверяет
попытку с неизвестным результатом после отправки с состоянием платформы перед повторным
воспроизведением.

источником истины для структуры API, а не приведённые ниже наброски.

### Контекст отправки

`withDurableMessageSendContext` предоставляет коду канала шаги `render`, `previewUpdate`,
`send`, `edit`, `delete`, `commit` и `fail` для одного исходящего
сообщения. `sendDurableMessageBatch` — обёртка для типового случая: отрисовать, отправить,
затем зафиксировать при `sent`/`suppressed` или отметить сбой при ошибке.

`sendDurableMessageBatch` возвращает один дискриминированный результат:

| Состояние        | Значение                                                                          |
| ---------------- | -------------------------------------------------------------------------------- |
| `sent`           | Доставлено хотя бы одно видимое сообщение платформы                              |
| `suppressed`     | Ни одно сообщение платформы не должно считаться отсутствующим (отменено обработчиком, пробный запуск и т. п.) |
| `partial_failed` | Хотя бы одно сообщение доставлено до сбоя последующей полезной нагрузки или побочного эффекта      |
| `failed`         | Квитанция платформы не получена                                                 |

Надёжность принимает одно из значений `required`, `best_effort` или `disabled`
(`MessageDurabilityPolicy` в `src/channels/message/types.ts`). При `required`
операция завершается с ошибкой, если надёжное намерение невозможно записать; при `best_effort`
в случае недоступности постоянного хранилища выполняется непосредственная отправка; `disabled`
сохраняет поведение непосредственной отправки, существовавшее до рефакторинга. Устаревшие
вспомогательные средства совместимости по умолчанию используют `disabled` и не выводят
`required` только из наличия у канала универсального исходящего адаптера.

Опасная граница сохраняется после успешного вызова платформы и до
фиксации квитанции. Если процесс завершается в этот момент, ядро не может определить,
существует ли сообщение на платформе, если адаптер не объявляет `reconcileUnknownSend`.
Этот обработчик классифицирует прерванную отправку как `sent`, `not_sent` или
`unresolved`; повторное воспроизведение допускается только для `not_sent`. Каналы без
сверки переходят в состояние `unknown_after_send` (`src/channels/message/state.ts`,
`src/infra/outbound/delivery-queue-recovery.ts`) и могут выбрать повторное воспроизведение
с семантикой «хотя бы один раз» только в том случае, если дублирование видимых сообщений
является приемлемым и документированным компромиссом для этого канала.

### Контекст получения

`createMessageReceiveContext` отслеживает состояние подтверждения/отклонения каждого входящего события
с идемпотентным `ack()` и явным `nack(error)`. Политика подтверждения
(`ChannelMessageReceiveAckPolicy`) принимает одно из следующих значений:

| Политика               | Когда выполняется подтверждение                                                                                     |
| ---------------------- | --------------------------------------------------------------------------------------------- |
| `after_receive_record` | Ядро сохранило достаточно метаданных входящего события для дедупликации или маршрутизации повторной доставки                           |
| `after_agent_dispatch` | Запуск агента был передан на выполнение                                                             |
| `after_durable_send`   | Надёжная исходящая отправка для этого хода была зафиксирована                                             |
| `manual`               | Вызывающая сторона явно управляет временем подтверждения (значение по умолчанию для адаптеров, не объявляющих политику) |

Опрос Telegram использует это для сохранения водяного знака безопасно завершённого обновления
(`safeCompletedUpdateId` в `extensions/telegram/src/bot-update-tracker.ts`):
grammY по-прежнему наблюдает каждое обновление при его поступлении в цепочку промежуточных обработчиков, но
OpenClaw продвигает сохранённый водяной знак перезапуска только за пределы обновлений,
для которых диспетчеризация завершилась, поэтому обновления со сбоем или всё ещё ожидающие
обработки воспроизводятся после перезапуска. Смещение вышестоящего `getUpdates` Telegram
по-прежнему контролируется grammY; полностью надёжный источник опроса, управляющий
повторной доставкой на уровне платформы за пределами этого водяного знака, не реализован
(см. «Открытые вопросы»).

### Предварительный просмотр в реальном времени

`src/channels/message/live.ts` моделирует предварительный просмотр, редактирование и завершение как единый жизненный цикл:
`createLiveMessageState`, `markLiveMessagePreviewUpdated`,
`markLiveMessageFinalized`, `markLiveMessageCancelled` и
`deliverFinalizableLivePreviewAdapter` (формирует окончательное изменение из черновика, применяет
его и возвращается к обычной отправке, если изменение невозможно или завершается сбоем).
`LiveMessageState.phase` принимает значения `idle | previewing | finalizing | finalized |
cancelled`; `canFinalizeInPlace` определяет, может ли предварительный просмотр стать окончательным
сообщением посредством редактирования вместо новой отправки.

### Надёжные квитанции

`MessageReceipt` (`src/channels/message/types.ts`) нормализует один или несколько
идентификаторов сообщений платформы из одной логической отправки в `platformMessageIds`, а также
сохраняет отдельные `parts` (тип, индекс, идентификатор ветки, идентификатор сообщения, на которое дан ответ).
Основной идентификатор сохраняется для работы с ветками и последующих изменений. Благодаря этому
составные доставки (текст с медиафайлами, разбитый на части текст, резервное представление карточки)
можно повторно воспроизводить и дедуплицировать после перезапуска.

### Сокращение общедоступного SDK

В результате рефакторинга были поглощены или объявлены устаревшими: вспомогательные средства `reply-runtime`, `reply-dispatch-runtime`,
`reply-reference`, `reply-chunking`, `reply-payload`, предоставлявшиеся как общедоступное
API, `inbound-reply-dispatch`, `channel-reply-pipeline` и большинство общедоступных применений
`outbound-runtime`. `src/plugin-sdk/channel-message.ts` теперь представляет собой
устаревший (`@deprecated`) файл повторного экспорта, указывающий на `channel-outbound` /
`channel-inbound`; псевдонимы среды выполнения `channel.turn` удалены, а старая
страница документации `/plugins/sdk-channel-turn` перенаправляет на
[API входящих сообщений каналов](/ru/plugins/sdk-channel-inbound). Новый код Plugin должен
напрямую использовать `channel-outbound` и `channel-inbound`.

## Где реализация отклонилась от первоначального проекта

Приведённый ниже проектный набросок никогда не был реализован буквально в описанном виде.
Он сохранён для исторической точности; не считайте эти имена типов актуальным API.

- **Нет `MessageOrigin` / `shouldDropOpenClawEcho`.** Первоначальный план предусматривал
  тег источника `source: "openclaw"` для сообщений о сбое Gateway, а также общий
  предикат, отбрасывающий помеченные эхо-сообщения, созданные ботом, в общих комнатах
  до авторизации `allowBots`. Эти тип и предикат отсутствуют в
  кодовой базе. Сам `allowBots` является реальным ключом конфигурации для каждого канала (Slack,
  Discord, Google Chat и других), но механизм тегирования происхождения, который должен был
  его защищать, так и не был реализован. Подавление эхо-сообщений о сбое Gateway
  в комнатах с включёнными ботами остаётся открытым пробелом, а не реализованной гарантией.
- **Нет единого пространства имён `core.messages.receive/send/live/state`.**
  Реализованные функции находятся непосредственно в `src/channels/message/*`
  (`withDurableMessageSendContext`, `createMessageReceiveContext`,
  `createLiveMessageState`, `classifyDurableSendRecoveryState`), а не
  за фасадом `core.messages.*`.
- **Нет универсального нормализованного типа сообщения `ChannelMessage` / `MessageTarget` / `MessageRelation`.**
  Ядро по-прежнему передаёт конкретные полезные нагрузки ответа
  (`ReplyPayload`) и контексты отдельных каналов через адаптеры отправки
  вместо единой платформенно-независимой структуры сообщения с отношением `kind: "reply" |
"followup" | "broadcast" | "system"`.
- **Имена политик подтверждения отличаются от наброска.** Реализованы:
  `after_receive_record | after_agent_dispatch | after_durable_send | manual`.
  В первоначальном наброске использовались `immediate | after-record | after-durable-send |
manual` с полем причины тайм-аута Webhook; такая структура не была реализована.
- **Ключи возможностей `DurableFinalDeliveryRequirementMap` заменили предложенный в наброске
  объект `MessageCapabilities`.** Возможности представлены плоскими логическими флагами (`text`,
  `media`, `poll`, `payload`, `silent`, `replyTo`, `thread`, `nativeQuote`,
  `messageSendingHooks`, `batch`, `reconcileUnknownSend`, `afterSendSuccess`,
  `afterCommit`), проверяемыми через `verifyDurableFinalCapabilityProofs`,
  а не вложенной структурой в стиле `text.chunking` / `attachments.voice`.

## Конкретные риски миграции (по-прежнему актуальны)

Эти побочные эффекты, специфичные для каналов, существовали до рефакторинга и должны
продолжать работать через новые пути отправки. Они не гипотетические: каждый из них
реализован и критически важен уже сейчас.

- **iMessage** (`extensions/imessage/src/monitor/echo-cache.ts`,
  `persisted-echo-cache.ts`): после успешной отправки монитор записывает отправленные
  сообщения в кеш эха. Устойчивые финальные отправки должны по-прежнему заполнять этот
  кеш, иначе OpenClaw может повторно принять собственные ответы как входящие сообщения
  пользователя.
- **Tlon** (`extensions/tlon/src/monitor/index.ts`): добавляет необязательную подпись
  модели и регистрирует ветки обсуждений, в которых было принято участие, после ответов
  в группах. Устойчивая доставка не должна обходить эти эффекты.
- **Discord и другие подготовленные диспетчеры** уже самостоятельно управляют прямой
  доставкой и поведением предварительного просмотра. Канал не обеспечивает сквозную
  устойчивость, пока его подготовленный диспетчер явно не направляет финальные
  сообщения через контекст отправки; не следует считать, что одного универсального
  адаптера достаточно.
- **Резервная доставка Telegram без уведомлений** должна доставлять весь массив
  спроецированной полезной нагрузки, а не только первый элемент, после разбиения на
  фрагменты или резервного проецирования.
- **LINE, Zalo, Nostr** и аналогичные вспомогательные пути могут обрабатывать токены
  ответа, проксирование медиафайлов, кеши отправленных сообщений или цели, доступные
  только через обратный вызов. Доставка для них остаётся под управлением канала, пока
  эта семантика не будет представлена адаптером отправки и покрыта тестами.
- **Вспомогательные функции прямых личных сообщений** могут иметь обратный вызов для
  ответа, который является единственной корректной целью транспорта. Универсальный
  механизм исходящей отправки не должен угадывать цель по необработанным полям
  платформы и пропускать этот обратный вызов.

## Классификация сбоев

Адаптеры классифицируют транспортные сбои по закрытым категориям типа
`DeliveryFailureKind` (временный сбой, ограничение частоты запросов, аутентификация,
разрешения, объект не найден, недопустимая полезная нагрузка, конфликт, отмена,
неизвестный сбой). Политика ядра:

- Повторять попытки при временных сбоях и ограничении частоты запросов.
- Не повторять попытки при недопустимой полезной нагрузке, если отсутствует резервный
  вариант рендеринга.
- Не повторять попытки при сбоях аутентификации или разрешений до изменения
  конфигурации.
- Если объект не найден, при финализации в реальном времени разрешить переход от
  редактирования к новой отправке, когда канал объявляет это безопасным.
- При конфликте использовать состояние квитанции и идемпотентности, чтобы определить,
  существует ли сообщение.
- Любая ошибка, возникшая после возможного успешного вызова платформы, но до фиксации
  квитанции, преобразуется в `unknown_after_send`, если адаптер не докажет, что операция
  на платформе не была выполнена.

## Открытые вопросы

- Следует ли Telegram в конечном итоге заменить средство опроса grammY (`1.43.0`) на
  полностью устойчивый источник опроса, который управляет повторной доставкой на уровне
  платформы, а не только сохраняемой OpenClaw отметкой перезапуска
  (`safeCompletedUpdateId`).
- Следует ли хранить состояние предварительного просмотра в реальном времени в одной
  записи с намерением финальной отправки или в отдельном хранилище состояния
  предварительного просмотра.
- Требуется ли для подавления эха при сбое Gateway в общих комнатах с ботами
  изначально запланированный механизм тегирования источника, более простой контракт
  для каждого канала, или это выходит за рамки задачи.
- Какие каналы поддерживают собственные данные об источнике или метаданные для
  подавления эха между ботами, а каким требуется сохраняемый реестр исходящих
  сообщений.
