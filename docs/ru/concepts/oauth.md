OpenClaw поддерживает «аутентификацию по подписке» через OAuth для провайдеров, которые ее предлагают
(в частности, **OpenAI Codex (ChatGPT OAuth)**). Для Anthropic практическое разделение
теперь такое:

- **API-ключ Anthropic**: обычная тарификация Anthropic API
- **Anthropic Claude CLI / аутентификация по подписке внутри OpenClaw**: сотрудники Anthropic
  сообщили нам, что такое использование снова разрешено

OpenAI Codex OAuth явно поддерживается для использования во внешних инструментах, таких как
OpenClaw.

OpenClaw хранит и аутентификацию OpenAI по API-ключу, и ChatGPT/Codex OAuth под
каноническим идентификатором провайдера `openai`. Более старые идентификаторы профилей `openai-codex:*` и
записи `auth.order.openai-codex` являются устаревшим состоянием, которое исправляется
`openclaw doctor --fix`; для новой конфигурации используйте идентификаторы профилей `openai:*` и `auth.order.openai`.

Для Anthropic в production более безопасный рекомендуемый путь — аутентификация по API-ключу.

На этой странице объясняется:

- как работает **обмен токена** OAuth (PKCE)
- где **хранятся** токены (и почему)
- как работать с **несколькими учетными записями** (профили + переопределения для отдельных сеансов)

OpenClaw также поддерживает **Plugin провайдеров**, которые поставляют собственные
потоки OAuth или API-ключей. Запускайте их так:

```bash
openclaw models auth login --provider <id>
```

## Приемник токенов (зачем он нужен)

OAuth-провайдеры часто выпускают **новый refresh token** во время входа или обновления. Некоторые провайдеры (или OAuth-клиенты) могут делать старые refresh tokens недействительными, когда для того же пользователя/приложения выдан новый.

Практический симптом:

- вы входите через OpenClaw _и_ через Claude Code / Codex CLI → позже один из них случайно оказывается «вышедшим из системы»

Чтобы уменьшить вероятность этого, OpenClaw рассматривает `auth-profiles.json` как **приемник токенов**:

- runtime читает учетные данные из **одного места**
- мы можем хранить несколько профилей и маршрутизировать их детерминированно
- повторное использование внешнего CLI зависит от провайдера: Codex CLI может создать пустой
  профиль `openai:default`, но как только у OpenClaw есть локальный OAuth-профиль,
  локальный refresh token становится каноническим. Если этот локальный refresh token отклонен,
  OpenClaw сообщает об управляемом профиле для повторной аутентификации вместо использования
  токенного материала Codex CLI как резервного sibling runtime. Другие интеграции могут
  оставаться внешне управляемыми и перечитывать свое хранилище аутентификации CLI
- пути статуса и запуска, которые уже знают настроенный набор провайдеров, ограничивают область
  обнаружения внешнего CLI этим набором, поэтому нерелевантное хранилище входа CLI не
  проверяется при настройке одного провайдера

## Хранилище (где находятся токены)

Секреты хранятся в хранилищах аутентификации агентов:

- Профили аутентификации (OAuth + API-ключи + необязательные ссылки уровня значения): `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- Устаревший файл совместимости: `~/.openclaw/agents/<agentId>/agent/auth.json`
  (статические записи `api_key` очищаются при обнаружении)

Устаревший файл только для импорта (по-прежнему поддерживается, но не является основным хранилищем):

- `~/.openclaw/credentials/oauth.json` (импортируется в `auth-profiles.json` при первом использовании)

Все перечисленное выше также учитывает `$OPENCLAW_STATE_DIR` (переопределение каталога состояния). Полная справка: [/gateway/configuration](/ru/gateway/configuration-reference#auth-storage)

О статических ссылках на секреты и поведении активации runtime snapshot см. [Управление секретами](/ru/gateway/secrets).

Когда у вторичного агента нет локального профиля аутентификации, OpenClaw использует сквозное
наследование из хранилища агента по умолчанию/основного агента. Он не клонирует
`auth-profiles.json` основного агента при чтении. OAuth refresh tokens особенно
чувствительны: обычные потоки копирования по умолчанию пропускают их, потому что некоторые провайдеры
ротируют или инвалидируют refresh tokens после использования. Настройте отдельный OAuth-вход для
агента, когда ему нужна независимая учетная запись.

## Миграция Anthropic Claude CLI

OpenClaw снова поддерживает повторное использование Anthropic Claude CLI. Если на хосте уже есть локальный
вход Claude, onboarding/configure может использовать его напрямую.

## Обмен OAuth (как работает вход)

Интерактивные потоки входа OpenClaw реализованы в `openclaw/plugin-sdk/llm` и подключены к мастерам/командам.

### Setup-token Anthropic

Форма потока:

1. запустите setup-token Anthropic или paste-token из OpenClaw
2. OpenClaw сохраняет полученные учетные данные Anthropic в профиле аутентификации
3. выбор модели остается на `anthropic/...`
4. существующие профили аутентификации Anthropic остаются доступными для отката/управления порядком

### OpenAI Codex (ChatGPT OAuth)

OpenAI Codex OAuth явно поддерживается для использования вне Codex CLI, включая рабочие процессы OpenClaw.

Команда входа по-прежнему использует канонический идентификатор провайдера OpenAI:

```bash
openclaw models auth login --provider openai
```

Используйте `--profile-id openai:<name>` для нескольких учетных записей ChatGPT/Codex OAuth в
одном агенте. Не используйте `openai-codex:<name>` для новых профилей. Doctor переносит
этот старый префикс в идентификатор профиля `openai:*` без конфликтов; после исправления выполните
`openclaw models auth list --provider openai`, прежде чем копировать
идентификаторы профилей в `auth.order` или `/model ...@<profileId>`.

Форма потока (PKCE):

1. сгенерировать PKCE verifier/challenge + случайный `state`
2. открыть `https://auth.openai.com/oauth/authorize?...`
3. попытаться перехватить callback на `http://127.0.0.1:1455/auth/callback`
4. если callback не удается привязать (или вы работаете удаленно/без UI), вставить URL/код перенаправления
5. выполнить обмен на `https://auth.openai.com/oauth/token`
6. извлечь `accountId` из access token и сохранить `{ access, refresh, expires, accountId }`

Путь в мастере: `openclaw onboard` → выбор аутентификации `openai`.

## Обновление + срок действия

Профили хранят временную метку `expires`.

Во время выполнения:

- если `expires` в будущем → использовать сохраненный access token
- если срок истек → обновить (под файловой блокировкой) и перезаписать сохраненные учетные данные
- если вторичный агент читает унаследованный OAuth-профиль основного агента, обновление
  записывается обратно в хранилище основного агента вместо копирования refresh token в
  хранилище вторичного агента
- исключение: некоторые учетные данные внешнего CLI остаются внешне управляемыми; OpenClaw
  перечитывает эти хранилища аутентификации CLI вместо расходования скопированных refresh tokens.
  Bootstrap Codex CLI намеренно уже: он может заполнить пустой
  `openai:default` или явно запрошенный профиль OpenAI только до того, как OpenClaw
  станет владельцем OAuth для провайдера. После этого обновления, принадлежащие OpenClaw, сохраняют локальные
  профили каноническими, а обнаружение не добавляет аутентификацию Codex CLI ни в один sibling
  slot. Если управляемое обновление завершается с ошибкой, OpenClaw сообщает о затронутом профиле для
  повторной аутентификации вместо возврата токенного материала внешнего CLI.

Поток обновления автоматический; обычно вам не нужно управлять токенами вручную.

## Несколько учетных записей (профили) + маршрутизация

Два шаблона:

### 1) Предпочтительно: отдельные агенты

Если вы хотите, чтобы «личное» и «рабочее» никогда не пересекались, используйте изолированных агентов (отдельные сеансы + учетные данные + workspace):

```bash
openclaw agents add work
openclaw agents add personal
```

Затем настройте аутентификацию для каждого агента (мастер) и маршрутизируйте чаты к нужному агенту.

### 2) Продвинутый вариант: несколько профилей в одном агенте

`auth-profiles.json` поддерживает несколько идентификаторов профилей для одного провайдера.

Выберите, какой профиль используется:

- глобально через порядок в конфигурации (`auth.order`)
- для отдельного сеанса через `/model ...@<profileId>`

Пример (переопределение сеанса):

- `/model Opus@anthropic:work`

Как посмотреть существующие идентификаторы профилей:

- `openclaw channels list --json` (показывает `auth[]`)
