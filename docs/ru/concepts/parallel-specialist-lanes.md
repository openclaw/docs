Параллельные специализированные направления позволяют одному Gateway направлять разные чаты или комнаты
разным агентам, сохраняя высокую скорость взаимодействия для пользователя. Рассматривайте параллелизм как
задачу проектирования с ограниченными ресурсами, а не просто как «больше агентов».

## Основные принципы

Специализированное направление повышает пропускную способность только тогда, когда снижает конкуренцию за
реальные узкие места:

- **Блокировки сеансов**: только один запуск должен изменять конкретный сеанс в каждый момент времени.
- **Общая пропускная способность модели**: все видимые запуски чатов по-прежнему используют общие лимиты провайдера.
- **Пропускная способность инструментов**: работа с оболочкой, браузером, сетью и репозиторием может занимать больше времени,
  чем сам ход модели.
- **Бюджет контекста**: длинные расшифровки замедляют каждый последующий ход и снижают
  его целенаправленность.
- **Неоднозначность ответственности**: агенты, дублирующие одну и ту же работу, впустую расходуют ресурсы.

OpenClaw уже выполняет запуски последовательно в рамках каждого сеанса и ограничивает общий параллелизм
через [очередь команд](/ru/concepts/queue). Специализированные направления добавляют поверх этого правила:
какой агент отвечает за какую работу, что остаётся в чате, а что становится
фоновой задачей.

## Рекомендуемый порядок внедрения

### Этап 1: контракты направлений и тяжёлая работа в фоне

Задайте для каждого направления письменный контракт в его рабочем пространстве и системном промпте:

- **Назначение**: работа, за которую отвечает это направление.
- **Что не входит в задачи**: работа, которую следует передать, а не пытаться выполнить самостоятельно.
- **Бюджет чата**: на быстрые вопросы отвечайте в чате; для длительных задач кратко подтвердите получение,
  а затем запустите фонового субагента или задачу.
- **Правило передачи**: если за работу отвечает другое направление, укажите, куда её следует передать,
  и предоставьте краткую сводку для передачи.
- **Правило риска инструментов**: отдавайте предпочтение минимальному набору инструментов, достаточному для выполнения задачи.

Это самый дешёвый этап, устраняющий большинство перегрузок: одна задача по программированию больше не
превращает исследовательское направление в патоку, а каждый чат сохраняет собственный контекст
чистым.

### Этап 2: управление приоритетами и параллелизмом

Настройте очередь и пропускную способность модели с учётом бизнес-ценности каждого направления:

```json5
{
  agents: {
    defaults: {
      maxConcurrent: 4,
      subagents: { maxConcurrent: 8, delegationMode: "prefer" },
    },
  },
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
    },
  },
}
```

Используйте личные чаты и агентов производственной эксплуатации для высокоприоритетной работы. Переводите
исследования, подготовку черновиков и пакетное программирование в фоновые задачи, когда система
занята.

### Этап 3: координатор / диспетчер трафика

Добавьте простой шаблон координатора, когда активно несколько направлений:

- Отслеживайте активные задачи направлений и ответственных.
- Обнаруживайте дублирующиеся запросы в разных группах.
- Передавайте сводки между направлениями.
- Показывайте только блокирующие проблемы, завершённые результаты и решения, которые должен принять человек.

Не начинайте с этого. Координатор без контрактов направлений лишь координирует хаос.

## Минимальный шаблон контракта направления

```md
# Контракт направления

## Отвечает за

- <job this lane is responsible for>

## Не отвечает за

- <work to hand off>

## Бюджет чата

- Отвечайте на быстрые вопросы напрямую.
- Для многоэтапной, медленной или требующей интенсивного использования инструментов работы: кратко подтвердите получение, запустите работу
  через субагента или в фоновом режиме, а после завершения верните результат.

## Передача

Если за запрос отвечает другое направление, укажите:

- целевое направление
- цель
- релевантный контекст
- точное следующее действие

## Подход к инструментам

Используйте минимальный набор инструментов, достаточный для выполнения задачи. Избегайте широкого использования оболочки или
сети, если это явно не входит в сферу ответственности данного направления.
```
