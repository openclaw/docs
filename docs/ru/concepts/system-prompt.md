OpenClaw формирует собственный системный промпт для каждого запуска агента; системного промпта по умолчанию во время выполнения нет.

Сборка состоит из трех уровней:

- `buildAgentSystemPrompt` формирует промпт из явно заданных входных данных. Он остается чистой функцией рендеринга и не читает глобальную конфигурацию напрямую.
- `resolveAgentSystemPromptConfig` определяет параметры промпта из конфигурации (отображение владельца, подсказки TTS, псевдонимы моделей, режим цитирования памяти, режим делегирования субагентам) для конкретного агента.
- Адаптеры среды выполнения (встроенный, CLI, предварительный просмотр команд/экспорта, Compaction) собирают актуальные сведения (инструменты, состояние песочницы, возможности канала, файлы контекста, дополнения провайдера к промпту) и вызывают настроенный фасад промпта.

Это обеспечивает соответствие экспортируемых и отладочных представлений промпта реальным запускам, не превращая все детали среды выполнения в один монолитный конструктор.

Плагины провайдеров могут добавлять рекомендации с учетом кэширования, не заменяя промпт, принадлежащий OpenClaw. Среда выполнения провайдера может:

- заменять один из трех именованных основных разделов: `interaction_style`, `tool_call_style`, `execution_bias`
- внедрять **стабильный префикс** выше границы кэша промпта
- внедрять **динамический суффикс** ниже границы кэша промпта

Используйте дополнения, принадлежащие провайдеру, для настройки под конкретное семейство моделей. Устаревший хук `before_prompt_build` применяйте только для совместимости или действительно глобальных изменений промпта.

Встроенное наложение для семейства OpenAI/Codex GPT-5 (`resolveGpt5SystemPromptContribution`) использует этот механизм: контракт поведения `stablePrefix` (политика выполнения, дисциплина использования инструментов, контракт вывода, контракт завершения) и необязательное переопределение `interaction_style` для более дружелюбного тона. Оно применяется к любому идентификатору модели `gpt-5*`, направленному через плагины OpenAI или Codex, и управляется параметром `agents.defaults.promptOverlays.gpt5.personality` (`"friendly"`/`"on"` или `"off"`).

## Структура

Промпт компактен и содержит фиксированные разделы:

- **Инструменты**: напоминание о том, что структурированные инструменты являются источником истины, а также рекомендации по их использованию во время выполнения. Когда включен экспериментальный инструмент `update_plan` (`tools.experimental.planTool`), его описание дополнительно предписывает: использовать его только для нетривиальной многоэтапной работы, оставлять не более одного шага в состоянии `in_progress` и не применять его для простой одноэтапной работы.
- **Приоритет выполнения**: выполнять в рамках текущего хода запросы, по которым можно действовать, продолжать до завершения или блокировки, восстанавливаться после слабых результатов инструментов, проверять изменяемое состояние в реальном времени и выполнять проверку перед завершением.
- **Безопасность**: краткое напоминание об ограничениях против стремления к расширению полномочий или обхода надзора.
- **Skills** (при наличии): объясняет модели, как загружать инструкции Skills по мере необходимости.
- **Управление OpenClaw**: отдавать предпочтение инструменту `gateway` для работы с конфигурацией и перезапуском; не выдумывать команды CLI.
- **Самообновление OpenClaw**: безопасно просматривать конфигурацию с помощью `config.schema.lookup`, изменять ее через `config.patch`, полностью заменять через `config.apply` и запускать `update.run` только по явному запросу пользователя. Доступный агенту инструмент `gateway` запрещает перезаписывать `tools.exec.ask` / `tools.exec.security`, включая устаревшие псевдонимы `tools.bash.*`, которые нормализуются в эти защищенные пути.
- **Рабочая область**: рабочий каталог (`agents.defaults.workspace`).
- **Документация**: локальный путь к документации/исходникам и указания о том, когда их следует читать.
- **Файлы рабочей области (внедренные)**: отмечает, что ниже включены файлы начальной загрузки.
- **Песочница** (когда включена): изолированная среда выполнения, пути песочницы и доступность выполнения с повышенными привилегиями.
- **Текущие дата и время**: только часовой пояс (стабилен для кэша; текущее время поступает из `session_status`).
- **Директивы вывода ассистента**: компактный синтаксис вложений, голосовых сообщений и тегов ответа.
- **Heartbeat**: промпт Heartbeat и поведение подтверждения, когда Heartbeat включены для агента по умолчанию.
- **Среда выполнения**: хост, ОС, Node, модель, корень репозитория (если определен), уровень рассуждений (одна строка).
- **Рассуждения**: текущий уровень видимости и подсказка о переключателе `/reasoning`.

Большой стабильный контент (включая **Контекст проекта**) остается выше внутренней границы кэша промпта. Изменчивые разделы для каждого хода (рекомендации по встраиванию интерфейса управления, **Обмен сообщениями**, **Голос**, **Контекст группового чата**, **Реакции**, **Heartbeat**, **Среда выполнения**) добавляются ниже этой границы, чтобы локальные серверные компоненты с кэшированием префиксов могли повторно использовать стабильный префикс рабочей области между ходами в каналах. В описания инструментов не следует встраивать названия текущих каналов, если поддерживаемая схема уже содержит эти данные среды выполнения.

Раздел инструментов также содержит рекомендации для длительных задач:

- использовать cron для будущих последующих действий (`check back later`, напоминаний, повторяющихся задач) вместо циклов ожидания через `exec`, приемов задержки с `yieldMs` или многократного опроса `process`
- использовать `exec` / `process` только для команд, которые запускаются сейчас и продолжают работу в фоновом режиме
- когда включено автоматическое пробуждение по завершении, запускать команду один раз и полагаться на механизм пробуждения через push-уведомление
- использовать `process` для журналов, состояния, ввода или вмешательства в выполняемую команду
- для более крупных задач предпочитать `sessions_spawn`; завершение работы субагента передается через push-механизм и автоматически сообщается инициатору запроса
- не опрашивать `subagents list` / `sessions_list` в цикле только ради ожидания завершения

`agents.defaults.subagents.delegationMode` (по умолчанию `"suggest"`) может усилить это поведение. Значение `"prefer"` добавляет отдельный раздел **Делегирование субагентам**, предписывающий основному агенту действовать как оперативный координатор и направлять через `sessions_spawn` все задачи, которые сложнее непосредственного ответа. Это влияет только на промпт; политика инструментов по-прежнему определяет доступность `sessions_spawn`.

Ограничения безопасности в системном промпте носят рекомендательный характер и не являются механизмом принудительного контроля. Для жесткого контроля используйте политику инструментов, подтверждения выполнения, песочницу и списки разрешенных каналов; операторы могут намеренно отключить ограничения безопасности в промпте.

В каналах с собственными карточками/кнопками подтверждения промпт предписывает агенту в первую очередь полагаться на этот интерфейс и включать ручную команду `/approve` только тогда, когда результат инструмента указывает, что подтверждения в чате недоступны или ручное подтверждение является единственным вариантом.

## Режимы промпта

OpenClaw формирует сокращенные системные промпты для субагентов. Среда выполнения задает `promptMode` для каждого запуска (это не пользовательская конфигурация):

- `full` (по умолчанию): все перечисленные выше разделы.
- `minimal`: используется для субагентов; исключает раздел промпта памяти (включенный как **Вызов из памяти**), **Самообновление OpenClaw**, **Псевдонимы моделей**, **Идентификация пользователя**, **Директивы вывода ассистента**, **Обмен сообщениями**, **Ответы без вывода** и **Heartbeat**. Инструменты, **Безопасность**, **Skills** (если предоставлены), рабочая область, песочница, текущие дата и время (если известны), среда выполнения и внедренный контекст остаются доступными.
- `none`: возвращает только базовую строку идентификации.

При `promptMode=minimal` дополнительные внедренные промпты помечаются как **Контекст субагента**, а не **Контекст группового чата**.

Для запусков автоматических ответов в каналах OpenClaw исключает общий раздел **Ответы без вывода**, когда контекст личной переписки, группы или ответа только через инструмент сообщений уже определяет контракт видимого ответа. Только устаревший автоматический режим группы/канала показывает `NO_REPLY`; в личных чатах и ответах только через инструмент сообщений инструкции по маркеру отсутствия ответа пропускаются.

## Снимки промптов

OpenClaw хранит зафиксированные снимки промптов для основного сценария среды выполнения Codex в `test/fixtures/agents/prompt-snapshots/codex-runtime-happy-path/`. Они отображают выбранные параметры потока/хода сервера приложений и восстановленный стек слоев промпта, передаваемого модели, для личной переписки в Telegram, группы Discord и ходов Heartbeat: закрепленный образец промпта модели Codex `gpt-5.5`, текст инструкций разработчика о разрешениях для основного сценария Codex, инструкции разработчика OpenClaw, инструкции режима совместной работы в рамках хода, когда их предоставляет OpenClaw, пользовательский ввод текущего хода и ссылки на динамические спецификации инструментов.

Обновляйте закрепленный образец промпта модели Codex командой `pnpm prompt:snapshots:sync-codex-model`. По умолчанию она ищет `$CODEX_HOME/models_cache.json`, затем `~/.codex/models_cache.json`, а затем путь, принятый для рабочей копии сопровождающего: `~/code/codex/codex-rs/models-manager/models.json`; если ни один файл не найден, команда завершается без изменения зафиксированного образца. Передайте `--catalog <path>`, чтобы обновить его из конкретного файла `models_cache.json` или `models.json`.

Эти снимки не являются побайтовой копией исходного запроса OpenAI. Codex может добавлять принадлежащий среде выполнения контекст рабочей области (`AGENTS.md`, контекст окружения, воспоминания, инструкции приложений/плагинов, встроенные инструкции режима совместной работы Default) после того, как OpenClaw передаст параметры потока и хода.

Пересоздавайте снимки командой `pnpm prompt:snapshots:gen`; проверяйте расхождения командой `pnpm prompt:snapshots:check`. CI выполняет проверку расхождений вместе с сегментами дополнительных границ, поэтому изменения промптов и обновления снимков включаются в один PR.

## Внедрение начальной загрузки рабочей области

Файлы начальной загрузки определяются из активной рабочей области и направляются в соответствующую их сроку жизни часть промпта:

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md` (только для совершенно новых рабочих областей)
- `MEMORY.md` при наличии

В нативной среде Codex OpenClaw не повторяет стабильные файлы рабочей области в каждом пользовательском ходе. Codex загружает `AGENTS.md` через собственный механизм обнаружения проектной документации. `TOOLS.md` передается как унаследованные инструкции разработчика Codex. `SOUL.md`, `IDENTITY.md` и `USER.md` передаются как инструкции разработчика для режима совместной работы в рамках хода, чтобы нативные субагенты Codex не наследовали их. Содержимое `HEARTBEAT.md` напрямую не внедряется; при наличии непустого файла ходы Heartbeat получают примечание режима совместной работы со ссылкой на него. Содержимое `MEMORY.md` также не вставляется в каждый нативный ход Codex: когда для рабочей области доступны инструменты памяти, ходы Codex получают краткое примечание о памяти рабочей области, направляющее модель к `memory_search` или `memory_get`. Если инструменты отключены, поиск по памяти недоступен или активная рабочая область отличается от рабочей области памяти агента, для `MEMORY.md` используется обычный ограниченный путь контекста хода. `BOOTSTRAP.md` сохраняет обычную роль контекста хода.

В средах, отличных от Codex, файлы начальной загрузки включаются в промпт OpenClaw в соответствии с существующими условиями. `HEARTBEAT.md` исключается при обычных запусках, когда Heartbeat отключены для агента по умолчанию или `agents.defaults.heartbeat.includeSystemPromptSection` имеет значение false. Сохраняйте внедряемые файлы краткими, особенно `MEMORY.md` вне Codex: он должен оставаться тщательно отобранной долгосрочной сводкой, а подробные ежедневные записи следует хранить в `memory/*.md` и извлекать по запросу через `memory_search` / `memory_get`. Чрезмерно большие файлы `MEMORY.md` вне Codex увеличивают расход промпта и могут внедряться лишь частично в соответствии с приведенными ниже ограничениями файлов начальной загрузки.

<Note>
Ежедневные файлы `memory/*.md` **не** входят в обычный Контекст проекта при начальной загрузке. В обычных ходах к ним обращаются по запросу через `memory_search` / `memory_get`, поэтому они не занимают контекстное окно, если модель явно их не читает. Исключение составляют отдельные ходы `/new` и `/reset`: среда выполнения может добавить недавние ежедневные записи памяти в начало первого хода как одноразовый блок начального контекста.
</Note>

Большие файлы усекаются с добавлением маркера:

| Ограничение                                      | Ключ конфигурации                                  | По умолчанию |
| ------------------------------------------------ | -------------------------------------------------- | ------------ |
| Максимальное количество символов на файл         | `agents.defaults.bootstrapMaxChars`                | 20000        |
| Общее количество символов во всех файлах         | `agents.defaults.bootstrapTotalMaxChars`           | 60000        |
| Предупреждение об усечении (`off`\|`once`\|`always`) | `agents.defaults.bootstrapPromptTruncationWarning` | `always`     |

Для отсутствующих файлов внедряется краткий маркер отсутствия. Подробные исходные и внедренные количества остаются в диагностических данных, таких как `/context`, `/status`, doctor и журналы.

Для файлов памяти усечение не означает потерю данных: файл остается на диске без изменений. В нативной среде Codex `MEMORY.md` при наличии инструментов памяти читается по запросу через них, а в остальных случаях используется ограниченный резервный вариант в промпте. В других средах модель видит только сокращенную внедренную копию, пока не прочитает файл или не выполнит прямой поиск по памяти. Если `MEMORY.md` регулярно усекается, преобразуйте его в более краткую устойчивую сводку, перенесите подробную историю в `memory/*.md` или намеренно увеличьте ограничения начальной загрузки.

Сеансы субагентов внедряют только `AGENTS.md` и `TOOLS.md` (остальные файлы начальной загрузки отфильтровываются, чтобы контекст субагента оставался небольшим).

Внутренние хуки могут перехватить этот этап через событие `agent:bootstrap`, чтобы изменить или заменить внедряемые файлы начальной загрузки (например, заменить `SOUL.md` для использования другой персоны).

Чтобы ответы звучали менее шаблонно, начните с [руководства по характеру в SOUL.md](/ru/concepts/soul).

Чтобы узнать, какой объём вносит каждый внедряемый файл (исходный и внедрённый размер, усечение, накладные расходы схем инструментов), используйте `/context list` или `/context detail`. См. раздел [Контекст](/ru/concepts/context).

## Обработка времени

Раздел **Текущие дата и время** отображается только тогда, когда известен часовой пояс пользователя, и содержит только **часовой пояс** (без динамических часов или формата времени), чтобы обеспечить стабильность кэша промпта.

Используйте `session_status`, когда агенту нужно текущее время; карточка состояния этого инструмента содержит строку с меткой времени. Этот же инструмент позволяет при необходимости задать переопределение модели для отдельного сеанса (`model=default` сбрасывает его).

Настройка выполняется с помощью:

- `agents.defaults.userTimezone`
- `agents.defaults.timeFormat` (`auto` | `12` | `24`)

Полное описание поведения см. в разделах [Часовые пояса](/ru/concepts/timezone) и [Дата и время](/ru/date-time).

## Skills

Если имеются доступные Skills, OpenClaw внедряет компактный список `<available_skills>` (`formatSkillsForPrompt`), содержащий **путь к файлу** и сформированный на основе содержимого маркер `<version>sha256:...</version>` для каждого навыка. Промпт предписывает модели использовать `read`, чтобы загрузить SKILL.md из указанного расположения (рабочего пространства, управляемого или встроенного), и повторно читать навык, если его `<version>` отличается от предыдущего хода. Если доступных Skills нет, раздел Skills не добавляется.

Нативные ходы Codex получают этот список в виде ограниченных текущим ходом инструкций разработчика по совместной работе, а не пользовательского ввода для каждого хода, за исключением упрощённых ходов cron, сохраняющих точный запланированный промпт. Другие среды выполнения сохраняют обычный раздел промпта.

Расположение может указывать на вложенный навык, например `skills/personal/foo/SKILL.md`. Вложенность служит только для организации; в промпте используется простое имя навыка из метаданных frontmatter файла `SKILL.md`.

Доступность определяется ограничениями метаданных навыка, проверками среды выполнения и конфигурации, а также действующим списком разрешённых навыков агента, если настроено `agents.defaults.skills` или `agents.list[].skills`. Skills, входящие в комплект Plugin, доступны только тогда, когда включён владеющий ими Plugin, что позволяет Plugin инструментов предоставлять подробные руководства по эксплуатации, не встраивая все эти указания в описание каждого инструмента.

```xml
<available_skills>
  <skill>
    <name>...</name>
    <description>...</description>
    <location>...</location>
    <version>sha256:...</version>
  </skill>
</available_skills>
```

Это позволяет сохранить базовый промпт небольшим, обеспечивая при этом целевое использование Skills. Размером управляет подсистема Skills отдельно от общих ограничений на чтение и внедрение во время выполнения:

| Область   | Бюджет промпта Skills                              | Бюджет фрагментов среды выполнения |
| --------- | ------------------------------------------------- | --------------------------------- |
| Глобально | `skills.limits.maxSkillsPromptChars`              | `agents.defaults.contextLimits.*` |
| Для агента | `agents.list[].skillsLimits.maxSkillsPromptChars` | `agents.list[].contextLimits.*`   |

Бюджет фрагментов среды выполнения распространяется на `memory_get`, актуальные результаты инструментов и обновления `AGENTS.md` после Compaction.

## Документация

Раздел **Документация** указывает на локальную документацию, если она доступна (`docs/` в рабочей копии Git или документация в составе npm-пакета), а в противном случае — на [https://docs.openclaw.ai](https://docs.openclaw.ai). В нём также указывается расположение исходного кода OpenClaw: для рабочих копий Git отображается корневой каталог локального исходного кода, а для установок из пакета — URL исходного кода на GitHub с указанием изучить его там, если документация неполна или устарела.

Промпт представляет документацию как основной источник сведений об OpenClaw до того, как модель определяет, как работает OpenClaw (память и ежедневные заметки, сеансы, инструменты, Gateway, конфигурация, команды, контекст проекта), и предписывает модели рассматривать `AGENTS.md`, контекст проекта, заметки рабочего пространства, профиля и памяти, а также `memory_search` как контекст инструкций или пользовательскую память, а не как сведения об устройстве и реализации OpenClaw. Если документация ничего не говорит по вопросу или устарела, модель должна сообщить об этом и изучить исходный код. Кроме того, модель должна самостоятельно выполнить `openclaw status`, когда это возможно, и обращаться к пользователю только при отсутствии доступа.

Для конфигурации агентам отдельно предлагается сначала использовать действие `config.schema.lookup` инструмента `gateway`, чтобы получить точную документацию и ограничения на уровне отдельных полей, а затем обратиться к `docs/gateway/configuration.md` и `docs/gateway/configuration-reference.md` за более общими указаниями.
