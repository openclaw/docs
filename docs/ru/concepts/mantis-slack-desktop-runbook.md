Mantis Slack для контроля качества настольного интерфейса — это сценарий проверки в реальном интерфейсе для ошибок класса Slack, требующих рабочего стола Linux, аварийного доступа через VNC, Slack Web, настоящего Gateway OpenClaw, снимков экрана, видеозаписей и комментария с доказательствами в PR. Используйте его, когда модульные тесты или безголовый сценарий проверки Slack в реальной среде не могут подтвердить ошибку.

## Модель хранения

Mantis использует три уровня хранения:

- **Образ провайдера** — принадлежит Crabbox и хранится в учётной записи облачного провайдера.
  Содержит возможности машины (Chrome/Chromium, ffmpeg, scrot,
  Node/corepack/pnpm, нативные инструменты сборки) и пустые каталоги кеша.
- **Состояние прогретой аренды** — принадлежит текущему сеансу оператора. Может содержать
  профиль браузера с выполненным входом, `/var/cache/crabbox/pnpm` и подготовленную рабочую копию
  исходного кода, пока аренда активна.
- **Артефакты Mantis** — принадлежат запуску OpenClaw. Находятся в
  `.artifacts/qa-e2e/mantis/...`; GitHub Actions загружает их, а приложение Mantis
  для GitHub публикует встроенные доказательства в комментарии к PR.

Никогда не включайте секреты, файлы cookie браузера, состояние входа в Slack, рабочие копии репозитория,
`node_modules` или `dist/` в образ провайдера.

## Запуск через GitHub

Запустите рабочий процесс из `main`:

```bash
gh workflow run mantis-slack-desktop-smoke.yml \
  --ref main \
  -f candidate_ref=<trusted-ref-or-sha> \
  -f pr_number=<pr-number> \
  -f scenario_id=slack-canary \
  -f crabbox_provider=aws \
  -f keep_vm=false \
  -f hydrate_mode=source
```

Значения `candidate_ref` ограничены, поскольку рабочий процесс использует реальные учётные данные: ссылка
должна разрешаться в текущего предка `main`, тег выпуска или вершину открытого PR в
`openclaw/openclaw`.

Рабочий процесс создаёт:

- загруженный артефакт `mantis-slack-desktop-smoke-<run-id>-<attempt>`
- встроенный комментарий к PR от приложения Mantis для GitHub
- `slack-desktop-smoke.png`, `slack-desktop-smoke.mp4`
- `slack-desktop-smoke-preview.gif`, `slack-desktop-smoke-change.mp4`
- `mantis-slack-desktop-smoke-summary.json`, `mantis-slack-desktop-smoke-report.md`
- удалённые журналы: `slack-desktop-command.log`, `openclaw-gateway.log`, `chrome.log`, `ffmpeg.log`

Комментарий к PR обновляется на месте по скрытому маркеру `<!-- mantis-slack-desktop-smoke -->`.

## Локальный CLI

Проверка исходного кода на холодной машине:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --provider aws \
  --class standard \
  --gateway-setup \
  --credential-source convex \
  --credential-role maintainer \
  --provider-mode live-frontier \
  --model openai/gpt-5.4 \
  --alt-model openai/gpt-5.4 \
  --scenario slack-canary \
  --hydrate-mode source
```

Сохранение виртуальной машины для аварийного доступа через VNC:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --provider aws \
  --class standard \
  --gateway-setup \
  --scenario slack-canary \
  --keep-lease
```

Открытие VNC:

```bash
crabbox vnc --provider aws --id <cbx_id> --open
```

Повторное использование прогретой аренды:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --provider aws \
  --lease-id <cbx_id-or-slug> \
  --gateway-setup \
  --scenario slack-canary \
  --hydrate-mode source
```

Используйте `--hydrate-mode prehydrated` только тогда, когда в повторно используемом удалённом рабочем пространстве уже
есть `node_modules` и собранный каталог `dist/`; в противном случае Mantis завершает работу с отказом.

Проверка нативного интерфейса подтверждения Slack:

```bash
pnpm openclaw qa mantis slack-desktop-smoke \
  --provider aws \
  --class standard \
  --approval-checkpoints \
  --credential-source convex \
  --credential-role maintainer \
  --hydrate-mode source
```

`--approval-checkpoints` несовместим с `--gateway-setup`. Этот флаг запускает
явно включаемые сценарии `slack-approval-exec-native` и `slack-approval-plugin-native`,
если только вы не передадите явный сценарий контрольной точки подтверждения через `--scenario`; другие
сценарии Slack отклоняются до запуска виртуальной машины. Средство контроля качества Slack записывает
каждый JSON-файл контрольной точки из реального сообщения API Slack, которое оно обнаружило, после чего
удалённый наблюдатель отображает это сообщение в
`approval-checkpoints/<scenario>-pending.png` и
`approval-checkpoints/<scenario>-resolved.png`. Запуск завершается с ошибкой, если какой-либо
JSON-файл контрольной точки, доказательство сообщения, JSON подтверждения получения или отображённый снимок экрана отсутствует
либо пуст.

В холодных арендах GitHub Actions нет файлов cookie Slack Web, поэтому при захвате браузера
может отображаться экран входа в Slack. Для подтверждения контрольных точек утверждения используйте
отображённые изображения контрольных точек и артефакты контроля качества Slack, а не
`slack-desktop-smoke.png`. Используйте только сохранённую прогретую аренду с профилем Slack Web,
в который вход выполнен вручную, если сам снимок экрана браузера должен показывать
Slack Web.

## Режимы подготовки

| Режим        | Когда использовать                              | Поведение на удалённой машине                                                        | Компромисс                                                       |
| ------------ | ----------------------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------- |
| `source`     | Обычная проверка PR, холодные машины, CI        | Запускает `pnpm install --frozen-lockfile --prefer-offline` и `pnpm build` внутри ВМ | Самый медленный, наиболее надёжно проверяет рабочую копию исходников |
| `prehydrated` | Вы намеренно подготовили повторно используемую аренду | Требует существующие `node_modules` и `dist/`; пропускает установку и сборку          | Быстро, но допустимо только для контролируемых оператором прогретых аренд |

GitHub Actions всегда подготавливает рабочую копию кандидата перед запуском виртуальной машины. Его
хранилище pnpm кешируется по ОС, версии Node и файлу блокировки. Запуск виртуальной машины в режиме `source`
также повторно использует `/var/cache/crabbox/pnpm`, если он существует.

## Интерпретация времени

`mantis-slack-desktop-smoke-report.md` содержит длительность этапов:

- `crabbox.warmup` — запуск облачного провайдера, готовность рабочего стола и браузера, SSH.
- `crabbox.inspect` — получение метаданных аренды.
- `credentials.prepare` — получение аренды учётных данных Convex.
- `crabbox.remote_run` — синхронизация, запуск браузера, установка и сборка OpenClaw либо
  проверка подготовки, запуск Gateway, создание снимков экрана и запись видео.
- `artifacts.copy` — обратная синхронизация с виртуальной машины через rsync.

Для `crabbox.remote_run` может отображаться `accepted`, когда Crabbox возвращает ненулевой
удалённый статус, но Mantis скопировал метаданные, доказывающие, что либо настройка Gateway OpenClaw
завершилась, либо сама команда контроля качества Slack успешно завершила работу. Считайте
`accepted` успешным результатом с пояснением, а не сбоем сценария.

Если запуск выполняется медленно:

- Преобладает прогрев: предварительно соберите или продвиньте более качественный образ провайдера Crabbox.
- `remote_run` преобладает в режиме `source`: используйте прогретую аренду, улучшите повторное использование хранилища pnpm
  или перенесите необходимые компоненты машины в образ провайдера.
- `remote_run` преобладает в режиме `prehydrated`: удалённое рабочее пространство в действительности не было
  готово либо настройка Gateway, браузера или Slack выполняется медленно.
- Преобладает копирование артефактов: проверьте размер видео и содержимое каталога артефактов.

## Контрольный список доказательств

Хороший комментарий к PR содержит:

- идентификатор сценария и SHA кандидата
- URL запуска GitHub Actions и URL артефакта
- встроенный снимок экрана контрольной точки утверждения или снимок экрана Slack Web из
  прогретой аренды с выполненным входом
- встроенный анимированный предварительный просмотр, если он доступен
- ссылки на полную и обрезанную версии MP4
- статус успеха или сбоя и сводку времени из отчёта

Не добавляйте снимки экрана или видеозаписи в репозиторий. Храните их в артефактах GitHub
Actions или в комментарии к PR.

## Обработка сбоев

Если рабочий процесс завершается с ошибкой до запуска виртуальной машины, сначала проверьте задание Actions.
Типичные причины: недоверенный `candidate_ref`, отсутствие секретов окружения или
ошибка установки либо сборки кандидата.

Если запуск виртуальной машины завершился с ошибкой, но снимки экрана были скопированы обратно, проверьте:

```bash
cat mantis-slack-desktop-smoke-report.md
cat mantis-slack-desktop-smoke-summary.json
cat slack-desktop-command.log
cat openclaw-gateway.log
cat chrome.log
cat ffmpeg.log
```

Если запуск сохранил аренду, откройте VNC с помощью команды `crabbox vnc ...`
из отчёта, а после завершения остановите аренду:

```bash
crabbox stop --provider aws <cbx_id-or-slug>
```

Если срок действия входа в Slack истёк, восстановите его через VNC в сохранённой аренде и перезапустите с
`--lease-id`. Не включайте этот профиль браузера в образ провайдера.
