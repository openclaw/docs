Control UI — это небольшое одностраничное приложение **Vite + Lit**, обслуживаемое Gateway:

- по умолчанию: `http://<host>:18789/`
- необязательный префикс: задайте `gateway.controlUi.basePath` (например, `/openclaw`)

Оно взаимодействует **напрямую с Gateway WebSocket** на том же порту.

## Быстрое открытие (локально)

Если Gateway запущен на том же компьютере, откройте:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/))

Если страница не загружается, сначала запустите Gateway: `openclaw gateway`.

<Note>
При нативных привязках Windows LAN брандмауэр Windows или управляемая организацией групповая политика всё равно могут блокировать объявленный LAN URL, даже если `127.0.0.1` работает на хосте Gateway. Выполните `openclaw gateway status --deep` на хосте Windows; команда сообщит о вероятно заблокированных портах, несоответствиях профилей и локальных правилах брандмауэра, которые политика может игнорировать.
</Note>

Аутентификация передаётся во время WebSocket-рукопожатия через:

- `connect.params.auth.token`
- `connect.params.auth.password`
- заголовки идентификации Tailscale Serve, когда `gateway.auth.allowTailscale: true`
- заголовки идентификации доверенного прокси, когда `gateway.auth.mode: "trusted-proxy"`

Панель настроек дашборда хранит токен для текущей сессии вкладки браузера и выбранного URL Gateway; пароли не сохраняются. Onboarding обычно создаёт токен Gateway для аутентификации с общим секретом при первом подключении, но аутентификация по паролю тоже работает, когда `gateway.auth.mode` равно `"password"`.

## Сопряжение устройства (первое подключение)

Когда вы подключаетесь к Control UI из нового браузера или устройства, Gateway обычно требует **одноразового подтверждения сопряжения**. Это мера безопасности для предотвращения несанкционированного доступа.

**Что вы увидите:** «отключено (1008): требуется сопряжение»

<Steps>
  <Step title="Выведите ожидающие запросы">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="Одобрите по ID запроса">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

Если браузер повторяет попытку сопряжения с изменёнными данными аутентификации (роль/области доступа/открытый ключ), предыдущий ожидающий запрос заменяется и создаётся новый `requestId`. Перед одобрением повторно выполните `openclaw devices list`.

Если браузер уже сопряжён и вы меняете доступ с чтения на запись/администрирование, это считается повышением уровня одобрения, а не тихим переподключением. OpenClaw оставляет старое одобрение активным, блокирует переподключение с более широкими правами и просит явно одобрить новый набор областей доступа.

После одобрения устройство запоминается и не требует повторного одобрения, если только вы не отзовёте его с помощью `openclaw devices revoke --device <id> --role <role>`. См. [CLI устройств](/ru/cli/devices) для ротации и отзыва токенов.

Агенты Paperclip, подключающиеся через адаптер `openclaw_gateway`, используют тот же поток одобрения первого запуска. После первой попытки подключения выполните `openclaw devices approve --latest`, чтобы предварительно просмотреть ожидающий запрос, затем повторно выполните напечатанную команду `openclaw devices approve <requestId>`, чтобы одобрить его. Для удалённого Gateway передайте явные значения `--url` и `--token`. Чтобы одобрения оставались стабильными между перезапусками, настройте постоянный `adapterConfig.devicePrivateKeyPem` в Paperclip вместо того, чтобы позволять ему создавать новую эфемерную идентичность устройства при каждом запуске.

<Note>
- Прямые браузерные подключения local loopback (`127.0.0.1` / `localhost`) одобряются автоматически.
- Tailscale Serve может пропустить цикл сопряжения для операторских сессий Control UI, когда `gateway.auth.allowTailscale: true`, идентичность Tailscale подтверждена, а браузер предъявляет свою идентичность устройства.
- Прямые привязки Tailnet, браузерные подключения по LAN и профили браузера без идентичности устройства всё равно требуют явного одобрения.
- Каждый профиль браузера создаёт уникальный ID устройства, поэтому смена браузера или очистка данных браузера потребуют повторного сопряжения.

</Note>

## Сопряжение мобильного устройства

Уже сопряжённый администратор может создать QR-код подключения iOS/Android без
открытия терминала:

<Steps>
  <Step title="Откройте мобильное сопряжение">
    Выберите **Узлы**, затем нажмите **Сопрячь мобильное устройство** в карточке **Устройства**.
  </Step>
  <Step title="Подключите телефон">
    В мобильном приложении OpenClaw откройте **Настройки** → **Gateway** и отсканируйте QR-
    код. Вместо этого можно скопировать и вставить код настройки.
  </Step>
  <Step title="Подтвердите подключение">
    Официальное приложение iOS/Android подключается автоматически. Если **Устройства** показывают
    ожидающий запрос, проверьте его роль и области доступа перед одобрением.
  </Step>
</Steps>

Для создания кода настройки требуется `operator.admin`; кнопка отключена для
сессий без него. Код настройки содержит краткоживущие bootstrap-учётные данные,
поэтому обращайтесь с QR-кодом и скопированным кодом как с паролем, пока они действительны. Для удалённого
сопряжения Gateway должен разрешаться в `wss://` (например, через Tailscale
Serve/Funnel); обычный `ws://` ограничен loopback и частными LAN-адресами.
См. [Сопряжение](/ru/channels/pairing#pair-from-the-control-ui-recommended) для
полных сведений о безопасности и fallback.

## Личная идентичность (локально в браузере)

Control UI поддерживает личную идентичность на уровне браузера (отображаемое имя и аватар), прикрепляемую к исходящим сообщениям для атрибуции в совместных сессиях. Она хранится в хранилище браузера, привязана к текущему профилю браузера и не синхронизируется с другими устройствами и не сохраняется на сервере, кроме обычных метаданных авторства стенограммы у сообщений, которые вы фактически отправляете. Очистка данных сайта или смена браузера сбрасывает её в пустое состояние.

Та же браузерно-локальная схема применяется к переопределению аватара ассистента. Загруженные аватары ассистента накладываются на идентичность, разрешённую Gateway, только в локальном браузере и никогда не проходят обратно через `config.patch`. Общее поле конфигурации `ui.assistant.avatar` по-прежнему доступно для не-UI клиентов, которые записывают поле напрямую (например, сценарных Gateway или пользовательских дашбордов).

## Эндпоинт runtime-конфигурации

Control UI получает свои runtime-настройки из `/control-ui-config.json`, разрешаемого относительно базового пути Control UI у Gateway (например, `/__openclaw__/control-ui-config.json`, когда UI обслуживается под `/__openclaw__/`). Этот эндпоинт защищён той же аутентификацией Gateway, что и остальная HTTP-поверхность: неаутентифицированные браузеры не могут получить его, а успешное получение требует либо уже действительного токена/пароля Gateway, либо идентичности Tailscale Serve, либо идентичности доверенного прокси.

## Поддержка языков

Control UI может локализоваться при первой загрузке на основе локали вашего браузера. Чтобы переопределить её позже, откройте **Обзор -> Доступ к Gateway -> Язык**. Переключатель локали находится в карточке доступа к Gateway, а не в разделе оформления.

- Поддерживаемые локали: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- Переводы не на английский лениво загружаются в браузере.
- Выбранная локаль сохраняется в хранилище браузера и используется повторно при будущих посещениях.
- Отсутствующие ключи перевода fallback на английский.

Переводы документации создаются для того же набора неанглийских локалей, но встроенный переключатель языков сайта документации Mintlify ограничен кодами локалей, которые принимает Mintlify. Документация на тайском (`th`) и персидском (`fa`) всё равно создаётся в publish-репозитории; она может не появляться в этом переключателе, пока Mintlify не начнёт поддерживать эти коды.

## Темы оформления

Панель оформления сохраняет встроенные темы Claw, Knot и Dash, а также один локальный для браузера слот импорта tweakcn. Чтобы импортировать тему, откройте [редактор tweakcn](https://tweakcn.com/editor/theme), выберите или создайте тему, нажмите **Поделиться** и вставьте скопированную ссылку на тему в оформление. Импортёр также принимает URL реестра `https://tweakcn.com/r/themes/<id>`, URL редактора вроде `https://tweakcn.com/editor/theme?theme=amethyst-haze`, относительные пути `/themes/<id>`, сырые ID тем и имена тем по умолчанию, такие как `amethyst-haze`.

Оформление также включает локальную для браузера настройку размера текста. Настройка хранится вместе с остальными предпочтениями Control UI, применяется к тексту чата, тексту компоновщика, карточкам инструментов и боковым панелям чата, а также удерживает поля ввода текста как минимум на 16px, чтобы мобильный Safari не выполнял автоматическое масштабирование при фокусе.

Импортированные темы хранятся только в текущем профиле браузера. Они не записываются в конфигурацию Gateway и не синхронизируются между устройствами. Замена импортированной темы обновляет один локальный слот; очистка переключает активную тему обратно на Claw, если была выбрана импортированная тема.

## Страница MCP

Выделенная страница MCP — это операторское представление для управляемых OpenClaw MCP-серверов в `mcp.servers`. Она сама не запускает MCP-транспорты; используйте ее для просмотра и редактирования сохраненной конфигурации, затем используйте `openclaw mcp doctor --probe`, когда нужно live-доказательство сервера.

Типичный рабочий процесс:

1. Откройте **MCP** на боковой панели.
2. Проверьте карточки сводки для общего количества серверов, включенных серверов, OAuth и отфильтрованных серверов.
3. Просмотрите строку каждого сервера: транспорт, включение, аутентификация, фильтры, тайм-ауты и подсказки команд.
4. Переключайте включение, когда сервер должен оставаться настроенным, но не участвовать в обнаружении во время выполнения.
5. Редактируйте scoped-раздел конфигурации `mcp` для определений серверов, заголовков, путей TLS/mTLS, метаданных OAuth, фильтров инструментов и метаданных проекции Codex.
6. Используйте **Сохранить** для записи конфигурации или **Сохранить и опубликовать**, когда работающий Gateway должен применить измененную конфигурацию.
7. Запустите `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` или `openclaw mcp reload` из терминала, когда отредактированному процессу нужны статическая диагностика, live-доказательство или сброс кэшированного runtime.

Страница редактирует URL-подобные значения, содержащие учетные данные, перед рендерингом и заключает имена серверов в кавычки во фрагментах команд, чтобы скопированные команды работали с пробелами или метасимволами оболочки. Полная справка по CLI и конфигурации находится в [MCP](/ru/cli/mcp).

## Вкладка Activity

Вкладка Activity — это эфемерный локальный для браузера наблюдатель за live-активностью инструментов. Она выводится из того же потока событий Gateway `session.tool` / tool, который питает карточки инструментов Chat; она не добавляет другое семейство событий Gateway, endpoint, долговременное хранилище активности, ленту метрик или внешний поток наблюдателя.

Записи Activity хранят только очищенные сводки и редактированные, усеченные предпросмотры вывода. Значения аргументов инструментов не сохраняются в состоянии Activity; UI показывает, что аргументы скрыты, и записывает только количество полей аргументов. Список в памяти следует за текущей вкладкой браузера, сохраняется при навигации внутри Control UI и сбрасывается при перезагрузке страницы, переключении сеанса или **Очистке**.

## Поведение чата

<AccordionGroup>
  <Accordion title="Семантика отправки и истории">
    - `chat.send` является **неблокирующим**: он сразу подтверждает получение с `{ runId, status: "started" }`, а ответ передается потоком через события `chat`. Доверенные клиенты Control UI также могут получать необязательные метаданные тайминга ACK для локальной диагностики.
    - Загрузки в чат принимают изображения и не-видео файлы. Изображения сохраняют нативный путь изображения; остальные файлы сохраняются как управляемые медиа и показываются в истории как ссылки на вложения.
    - Повторная отправка с тем же `idempotencyKey` возвращает `{ status: "in_flight" }` во время выполнения и `{ status: "ok" }` после завершения.
    - Ответы `chat.history` ограничены по размеру для безопасности UI. Когда записи транскрипта слишком велики, Gateway может усекать длинные текстовые поля, опускать тяжелые блоки метаданных и заменять слишком большие сообщения заполнителем (`[chat.history omitted: message too large]`).
    - Когда видимое сообщение ассистента было усечено в `chat.history`, боковой читатель может по требованию получить полную нормализованную для отображения запись транскрипта через `chat.message.get` по `sessionKey`, активному `agentId`, когда нужно, и `messageId` транскрипта. Если Gateway все еще не может вернуть больше, читатель показывает явное состояние недоступности вместо молчаливого повторения усеченного предпросмотра.
    - Изображения, созданные ассистентом/генерацией, сохраняются как управляемые медиа-ссылки и отдаются обратно через аутентифицированные URL медиа Gateway, поэтому перезагрузки не зависят от сохранения raw base64-полезных нагрузок изображений в ответе истории чата.
    - При рендеринге `chat.history` Control UI удаляет из видимого текста ассистента inline-теги директив, предназначенные только для отображения (например, `[[reply_to_*]]` и `[[audio_as_voice]]`), plain-text XML-полезные нагрузки вызовов инструментов (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и усеченные блоки вызовов инструментов), а также утекшие ASCII/full-width управляющие токены модели, и опускает записи ассистента, чей весь видимый текст состоит только из точного silent-токена `NO_REPLY` / `no_reply` или токена подтверждения Heartbeat `HEARTBEAT_OK`.
    - Во время активной отправки и финального обновления истории представление чата сохраняет видимыми локальные оптимистичные сообщения пользователя/ассистента, если `chat.history` ненадолго возвращает более старый снимок; канонический транскрипт заменяет эти локальные сообщения, когда история Gateway догоняет.
    - Live-события `chat` являются состоянием доставки, тогда как `chat.history` перестраивается из долговременного транскрипта сеанса. После финальных событий инструмента Control UI перезагружает историю и объединяет только небольшой оптимистичный хвост; граница транскрипта задокументирована в [WebChat](/ru/web/webchat).
    - `chat.inject` добавляет заметку ассистента в транскрипт сеанса и рассылает событие `chat` для обновлений только UI (без запуска агента, без доставки в канал).
    - Боковая панель показывает недавние сеансы с действием New Session, ссылкой All Sessions и кнопкой поиска сеансов, которая открывает полный выбор сеанса (scoped по выбранному агенту, с поиском и пагинацией). При переключении агентов показываются только сеансы, привязанные к этому агенту, с fallback к основному сеансу этого агента, если у него еще нет сохраненных сеансов панели.
    - Каждую строку выбора сеанса можно переименовать, закрепить или архивировать. Активный запуск и основной сеанс агента нельзя архивировать. Архивирование выбранного сейчас сеанса переключает Chat обратно на основной сеанс этого агента.
    - На десктопных ширинах элементы управления чатом остаются в одной компактной строке и сворачиваются при прокрутке вниз по транскрипту; прокрутка вверх, возврат к началу или достижение низа восстанавливают элементы управления.
    - Последовательные повторяющиеся сообщения только с текстом отображаются как один bubble со значком количества. Сообщения, содержащие изображения, вложения, вывод инструментов или предпросмотры canvas, не сворачиваются.
    - Пикеры модели и thinking в заголовке чата сразу изменяют активный сеанс через `sessions.patch`; это постоянные переопределения сеанса, а не send-опции только для одного хода.
    - Если отправить сообщение, пока изменение пикера модели для того же сеанса все еще сохраняется, composer дождется этого patch сеанса перед вызовом `chat.send`, чтобы отправка использовала выбранную модель.
    - Ввод `/new` в Control UI создает и переключает на тот же свежий сеанс панели, что и New Chat, кроме случая, когда настроено `session.dmScope: "main"` и текущий родитель — основной сеанс агента; в этом случае он сбрасывает основной сеанс на месте. Ввод `/reset` сохраняет явный in-place reset Gateway для текущего сеанса.
    - Пикер модели чата запрашивает настроенное представление моделей Gateway. Если присутствует `agents.defaults.models`, этот allowlist управляет пикером, включая записи `provider/*`, которые сохраняют динамичность каталогов в области провайдера. В противном случае пикер показывает явные записи `models.providers.*.models`, а также провайдеров с пригодной аутентификацией. Полный каталог остается доступным через отладочный RPC `models.list` с `view: "all"`.
    - Когда свежие отчеты Gateway об использовании сеанса включают текущие токены контекста, панель инструментов composer чата показывает маленькое кольцо использования контекста с процентом использования; полные сведения о токенах находятся в его подсказке. Кольцо переключается на предупреждающее оформление при высоком давлении контекста и, на рекомендуемых уровнях Compaction, показывает компактную кнопку, запускающую обычный путь Compaction сеанса. Устаревшие снимки токенов скрываются, пока Gateway снова не сообщит свежие данные использования.

  <Accordion title="Режим Talk (браузерное realtime)">
    Режим Talk использует зарегистрированного провайдера realtime-голоса. Настройте OpenAI с `talk.realtime.provider: "openai"` плюс auth-профилем API-ключа `openai`, `talk.realtime.providers.openai.apiKey` или `OPENAI_API_KEY`; OAuth-профили OpenAI не настраивают Realtime-голос. Провайдеры, которые предоставляют только backend realtime bridge, работают через relay-транспорт Gateway, поэтому учетные данные и vendor-сокеты остаются на стороне сервера, а браузерное аудио проходит через аутентифицированные RPC Gateway. Подсказка Realtime-сеанса собирается Gateway; `talk.client.create` не принимает предоставленные вызывающей стороной переопределения инструкций.

Параметры применяются к следующему сеансу разговора и могут переопределять провайдера, транспорт, модель, голос, усилие рассуждения, порог VAD, длительность тишины и отступ префикса. Когда параметр пустой, Gateway использует настроенные значения по умолчанию, если они доступны, или значение по умолчанию провайдера. Выбор ретрансляции Gateway принудительно включает серверный путь ретрансляции; выбор WebRTC оставляет сеанс во владении клиента и приводит к ошибке, а не к незаметному откату на ретрансляцию, если провайдер не может создать браузерный сеанс.

    В компоновщике чата элемент управления разговором — это кнопка с волнами рядом с кнопкой диктовки через микрофон. Когда разговор запускается, строка состояния компоновщика показывает `Connecting Talk...`, затем `Talk live`, пока звук подключен, или `Asking OpenClaw...`, пока realtime-вызов инструмента обращается к настроенной более крупной модели через `talk.client.toolCall`.

    Live smoke для сопровождающих: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` проверяет серверный WebSocket-мост OpenAI, браузерный обмен WebRTC SDP OpenAI, браузерную WebSocket-настройку Google Live с ограниченным числом токенов и браузерный адаптер ретрансляции Gateway с фейковым медиа микрофона. Команда выводит только статус провайдера и не записывает секреты в журнал.

## Установка PWA и Web Push

Интерфейс управления поставляется с `manifest.webmanifest` и service worker, поэтому современные браузеры могут установить его как автономную PWA. Web Push позволяет Gateway пробуждать установленную PWA уведомлениями, даже когда вкладка или окно браузера не открыты.

Если страница показывает **Protocol mismatch** сразу после обновления OpenClaw, сначала снова откройте панель с помощью `openclaw dashboard` и выполните жесткое обновление страницы. Если ошибка сохраняется, очистите данные сайта для origin панели или проверьте в приватном окне браузера; старая вкладка или service-worker-кэш браузера могут продолжать запускать пакет Control UI до обновления против более нового Gateway.

| Поверхность                                            | Что делает                                                         |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | Манифест PWA. Браузеры предлагают "Install app", когда он доступен. |
| `ui/public/sw.js`                                     | Service worker, который обрабатывает события `push` и клики по уведомлениям. |
| `push/vapid-keys.json` (в каталоге состояния OpenClaw) | Автоматически сгенерированная пара ключей VAPID, используемая для подписи полезных нагрузок Web Push. |
| `push/web-push-subscriptions.json`                    | Сохраненные endpoint подписок браузера.                            |

Переопределите пару ключей VAPID через переменные окружения в процессе Gateway, когда нужно закрепить ключи (для развертываний на нескольких хостах, ротации секретов или тестов):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (по умолчанию `https://openclaw.ai`)

Интерфейс управления использует эти методы Gateway с ограничением по scope для регистрации и тестирования браузерных подписок:

- `push.web.vapidPublicKey` — получает активный публичный ключ VAPID.
- `push.web.subscribe` — регистрирует `endpoint` плюс `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` — удаляет зарегистрированный endpoint.
- `push.web.test` — отправляет тестовое уведомление в подписку вызывающей стороны.

<Note>
Web Push не зависит от пути ретрансляции iOS APNS (см. [Конфигурация](/ru/gateway/configuration) для push через ретрансляцию) и существующего метода `push.test`, которые нацелены на нативное мобильное сопряжение.
</Note>

## Размещенные встраивания

Сообщения ассистента могут отображать размещенный веб-контент внутри строки с помощью shortcode `[embed ...]`. Политика песочницы iframe управляется `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    Отключает выполнение скриптов внутри размещенных встраиваний.
  </Tab>
  <Tab title="scripts (default)">
    Разрешает интерактивные встраивания, сохраняя изоляцию origin; это значение по умолчанию и обычно его достаточно для самодостаточных браузерных игр/виджетов.
  </Tab>
  <Tab title="trusted">
    Добавляет `allow-same-origin` поверх `allow-scripts` для документов того же сайта, которым намеренно нужны более сильные привилегии.
  </Tab>
</Tabs>

Пример:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
Используйте `trusted` только тогда, когда встроенному документу действительно требуется поведение same-origin. Для большинства игр и интерактивных canvas, сгенерированных агентом, `scripts` — более безопасный выбор.
</Warning>

Абсолютные внешние URL для встраивания `http(s)` по умолчанию остаются заблокированными. Если вы намеренно хотите, чтобы `[embed url="https://..."]` загружал сторонние страницы, задайте `gateway.controlUi.allowExternalEmbedUrls: true`.

## Ширина сообщений чата

Сгруппированные сообщения чата используют читаемую максимальную ширину по умолчанию. Развертывания с широкими мониторами могут переопределить ее без исправления встроенного CSS, задав `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

Значение проверяется до того, как попадет в браузер. Поддерживаемые значения включают простые длины и проценты, такие как `960px` или `82%`, а также ограниченные выражения ширины `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` и `fit-content(...)`.

## Доступ к tailnet (рекомендуется)

<Tabs>
  <Tab title="Integrated Tailscale Serve (preferred)">
    Оставьте Gateway на local loopback и позвольте Tailscale Serve проксировать его через HTTPS:

    ```bash
    openclaw gateway --tailscale serve
    ```

    Откройте:

    - `https://<magicdns>/` (или настроенный вами `gateway.controlUi.basePath`)

    По умолчанию запросы Serve для интерфейса управления/WebSocket могут проходить аутентификацию через заголовки идентичности Tailscale (`tailscale-user-login`), когда `gateway.auth.allowTailscale` равно `true`. OpenClaw проверяет идентичность, разрешая адрес `x-forwarded-for` через `tailscale whois` и сопоставляя его с заголовком, и принимает такие запросы только когда они приходят на loopback с заголовками Tailscale `x-forwarded-*`. Для операторских сеансов интерфейса управления с идентичностью браузерного устройства этот проверенный путь Serve также пропускает цикл сопряжения устройства; браузеры без устройства и подключения с ролью узла по-прежнему проходят обычные проверки устройства. Задайте `gateway.auth.allowTailscale: false`, если хотите требовать явные учетные данные с общим секретом даже для трафика Serve. Затем используйте `gateway.auth.mode: "token"` или `"password"`.

    Для этого асинхронного пути идентичности Serve неудачные попытки аутентификации для одного и того же IP клиента и scope аутентификации сериализуются перед записью лимитов частоты. Поэтому параллельные неудачные повторы из одного браузера могут показать `retry later` на втором запросе вместо двух обычных несовпадений, соревнующихся параллельно.

    <Warning>
    Аутентификация Serve без токена предполагает, что хост шлюза доверенный. Если на этом хосте может выполняться недоверенный локальный код, требуйте аутентификацию по токену/паролю.
    </Warning>

  </Tab>
  <Tab title="Bind to tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    Затем откройте:

    - `http://<tailscale-ip>:18789/` (или настроенный вами `gateway.controlUi.basePath`)

    Вставьте соответствующий общий секрет в настройки UI (отправляется как `connect.params.auth.token` или `connect.params.auth.password`).

  </Tab>
</Tabs>

## Небезопасный HTTP

Если вы открываете панель через обычный HTTP (`http://<lan-ip>` или `http://<tailscale-ip>`), браузер работает в **небезопасном контексте** и блокирует WebCrypto. По умолчанию OpenClaw **блокирует** подключения интерфейса управления без идентичности устройства.

Документированные исключения:

- совместимость небезопасного HTTP только для localhost с `gateway.controlUi.allowInsecureAuth=true`
- успешная операторская аутентификация интерфейса управления через `gateway.auth.mode: "trusted-proxy"`
- аварийный `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**Рекомендуемое исправление:** используйте HTTPS (Tailscale Serve) или откройте UI локально:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (на хосте Gateway)

<AccordionGroup>
  <Accordion title="Insecure-auth toggle behavior">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` — только локальный переключатель совместимости:

    - Он позволяет сеансам интерфейса управления localhost продолжаться без идентичности устройства в небезопасных HTTP-контекстах.
    - Он не обходит проверки сопряжения.
    - Он не ослабляет требования к идентичности устройства для удаленных (не localhost) подключений.

  </Accordion>
  <Accordion title="Break-glass only">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` отключает проверки идентичности устройства интерфейса управления и является серьезным снижением безопасности. Быстро откатите после аварийного использования.
    </Warning>

  </Accordion>
  <Accordion title="Trusted-proxy note">
    - Успешная аутентификация trusted-proxy может допускать **операторские** сеансы интерфейса управления без идентичности устройства.
    - Это **не** распространяется на сеансы интерфейса управления с ролью узла.
    - Same-host loopback reverse proxies все равно не удовлетворяют аутентификации trusted-proxy; см. [Аутентификация через доверенный прокси](/ru/gateway/trusted-proxy-auth).

  </Accordion>
</AccordionGroup>

См. [Tailscale](/ru/gateway/tailscale) для руководства по настройке HTTPS.

## Политика безопасности контента

Интерфейс управления поставляется со строгой политикой `img-src`: разрешены только ресурсы **same-origin**, URL `data:` и локально сгенерированные URL `blob:`. Удаленные URL изображений `http(s)` и URL изображений с относительным протоколом отклоняются браузером и не инициируют сетевые запросы.

Что это означает на практике:

- Аватары и изображения, отдаваемые по относительным путям (например, `/avatars/<id>`), по-прежнему отображаются, включая аутентифицированные маршруты аватаров, которые UI получает и преобразует в локальные URL `blob:`.
- Встроенные URL `data:image/...` по-прежнему отображаются (полезно для полезных нагрузок внутри протокола).
- Локальные URL `blob:`, созданные интерфейсом управления, по-прежнему отображаются.
- Удаленные URL аватаров, выданные метаданными канала, удаляются в помощниках аватаров интерфейса управления и заменяются встроенным логотипом/значком, поэтому скомпрометированный или вредоносный канал не может принудить браузер оператора выполнять произвольные удаленные загрузки изображений.

Вам не нужно ничего менять, чтобы получить это поведение — оно всегда включено и не настраивается.

## Аутентификация маршрута аватара

Когда аутентификация Gateway настроена, endpoint аватара интерфейса управления требует тот же токен Gateway, что и остальная часть API:

- `GET /avatar/<agentId>` возвращает изображение аватара только аутентифицированным вызывающим сторонам. `GET /avatar/<agentId>?meta=1` возвращает метаданные аватара по тому же правилу.
- Неаутентифицированные запросы к любому из маршрутов отклоняются (как и соседний маршрут assistant-media). Это предотвращает утечку идентичности агента через маршрут аватара на хостах, которые иначе защищены.
- Сам интерфейс управления пересылает токен Gateway как bearer-заголовок при получении аватаров и использует аутентифицированные blob URL, чтобы изображение по-прежнему отображалось на панелях.

Если отключить аутентификацию Gateway (не рекомендуется на общих хостах), маршрут аватара также станет неаутентифицированным, как и остальная часть Gateway.

## Аутентификация маршрута медиа ассистента

Когда аутентификация Gateway настроена, локальные предпросмотры медиа ассистента используют двухэтапный маршрут:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` требует обычной аутентификации оператора Control UI. При проверке доступности браузер отправляет токен Gateway в заголовке bearer.
- Успешные ответы с метаданными включают короткоживущий `mediaTicket`, ограниченный точно этим исходным путем.
- URL изображений, аудио, видео и документов, отображаемых браузером, используют `mediaTicket=<ticket>` вместо активного токена или пароля Gateway. Билет быстро истекает и не может авторизовать другой источник.

Это сохраняет совместимость обычного отображения медиа с нативными медиаэлементами браузера, не помещая переиспользуемые учетные данные Gateway в видимые URL медиа.

## Сборка UI

Gateway раздает статические файлы из `dist/control-ui`. Соберите их с помощью:

```bash
pnpm ui:build
```

Необязательная абсолютная база (если нужны фиксированные URL ресурсов):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

Для локальной разработки (отдельный dev-сервер):

```bash
pnpm ui:dev
```

Затем укажите UI URL WS вашего Gateway (например, `ws://127.0.0.1:18789`).

## Пустая страница Control UI

Если браузер загружает пустую панель, а DevTools не показывает полезной ошибки, расширение или ранний content script мог помешать выполнению JavaScript-модуля приложения. Статическая страница включает простую HTML-панель восстановления, которая появляется, когда `<openclaw-app>` не зарегистрирован после запуска.

Используйте действие **Try again** на панели после изменения среды браузера или перезагрузите вручную после следующих проверок:

- Отключите расширения, которые внедряются во все страницы, особенно расширения с content scripts `<all_urls>`.
- Попробуйте приватное окно, чистый профиль браузера или другой браузер.
- Оставьте Gateway запущенным и проверьте тот же URL панели после изменения браузера.

## Отладка/тестирование: dev-сервер + удаленный Gateway

Control UI — это статические файлы; цель WebSocket настраивается и может отличаться от HTTP origin. Это удобно, когда нужен локальный dev-сервер Vite, но Gateway работает в другом месте.

<Steps>
  <Step title="Start the UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="Open with gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    Необязательная одноразовая аутентификация (если нужна):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Notes">
    - `gatewayUrl` сохраняется в localStorage после загрузки и удаляется из URL.
    - Если вы передаете полный endpoint `ws://` или `wss://` через `gatewayUrl`, URL-кодируйте значение `gatewayUrl`, чтобы браузер правильно разобрал строку запроса.
    - `token` по возможности следует передавать через фрагмент URL (`#token=...`). Фрагменты не отправляются на сервер, что предотвращает утечку через журналы запросов и Referer. Устаревшие query params `?token=` все еще однократно импортируются для совместимости, но только как резервный вариант, и удаляются сразу после bootstrap.
    - `password` хранится только в памяти.
    - Когда задан `gatewayUrl`, UI не использует резервно учетные данные из конфигурации или окружения. Укажите `token` (или `password`) явно. Отсутствие явных учетных данных является ошибкой.
    - Используйте `wss://`, когда Gateway находится за TLS (Tailscale Serve, HTTPS-прокси и т. д.).
    - `gatewayUrl` принимается только в окне верхнего уровня (не во встроенном), чтобы предотвратить clickjacking.
    - Публичные развертывания Control UI не через loopback должны явно задавать `gateway.controlUi.allowedOrigins` (полные origins). Приватные загрузки из той же origin в LAN/Tailnet с loopback, RFC1918/link-local, `.local`, `.ts.net` или хостов Tailscale CGNAT принимаются без включения резервного режима Host-header.
    - При запуске Gateway может добавить локальные origins, такие как `http://localhost:<port>` и `http://127.0.0.1:<port>`, из эффективных runtime bind и порта, но удаленным browser origins все равно нужны явные записи.
    - Не используйте `gateway.controlUi.allowedOrigins: ["*"]`, кроме строго контролируемого локального тестирования. Это означает разрешить любую browser origin, а не «сопоставить любой хост, который я использую».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` включает режим резервного origin по Host-header, но это опасный режим безопасности.

  </Accordion>
</AccordionGroup>

Пример:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

Подробности настройки удаленного доступа: [Удаленный доступ](/ru/gateway/remote).
