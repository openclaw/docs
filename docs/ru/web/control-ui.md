Control UI — это небольшое одностраничное приложение **Vite + Lit**, обслуживаемое Gateway:

- по умолчанию: `http://<host>:18789/`
- необязательный префикс: задайте `gateway.controlUi.basePath` (например, `/openclaw`)

Оно взаимодействует **напрямую с WebSocket Gateway** на том же порту.

## Быстрое открытие (локально)

Если Gateway запущен на том же компьютере, откройте:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/))

Если страница не загружается, сначала запустите Gateway: `openclaw gateway`.

<Note>
При нативных привязках Windows LAN брандмауэр Windows или управляемая организацией групповая политика все еще могут блокировать объявленный URL LAN, даже если `127.0.0.1` работает на хосте Gateway. Выполните `openclaw gateway status --deep` на хосте Windows; команда сообщает о вероятно заблокированных портах, несоответствиях профилей и локальных правилах брандмауэра, которые политика может игнорировать.
</Note>

Аутентификация передается во время рукопожатия WebSocket через:

- `connect.params.auth.token`
- `connect.params.auth.password`
- заголовки идентификации Tailscale Serve, когда `gateway.auth.allowTailscale: true`
- заголовки идентификации доверенного прокси, когда `gateway.auth.mode: "trusted-proxy"`

Панель настроек панели управления хранит токен для текущей сессии вкладки браузера и выбранного URL Gateway; пароли не сохраняются. Онбординг обычно генерирует токен Gateway для аутентификации с общим секретом при первом подключении, но аутентификация по паролю тоже работает, когда `gateway.auth.mode` равно `"password"`.

## Сопряжение устройства (первое подключение)

Когда вы подключаетесь к Control UI из нового браузера или с нового устройства, Gateway обычно требует **одноразового подтверждения сопряжения**. Это мера безопасности для предотвращения несанкционированного доступа.

**Что вы увидите:** "disconnected (1008): pairing required"

<Steps>
  <Step title="Выведите список ожидающих запросов">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="Одобрите по идентификатору запроса">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

Если браузер повторяет попытку сопряжения с измененными данными аутентификации (роль/области доступа/открытый ключ), предыдущий ожидающий запрос заменяется, и создается новый `requestId`. Перед одобрением повторно выполните `openclaw devices list`.

Если браузер уже сопряжен и вы меняете доступ с чтения на запись/администрирование, это считается повышением уровня одобрения, а не тихим переподключением. OpenClaw сохраняет старое одобрение активным, блокирует переподключение с более широкими правами и просит явно одобрить новый набор областей доступа.

После одобрения устройство запоминается и не потребует повторного одобрения, если вы не отзовете его с помощью `openclaw devices revoke --device <id> --role <role>`. См. [CLI устройств](/ru/cli/devices) для ротации и отзыва токенов.

Агенты Paperclip, подключающиеся через адаптер `openclaw_gateway`, используют тот же процесс одобрения первого запуска. После первоначальной попытки подключения выполните `openclaw devices approve --latest`, чтобы предварительно просмотреть ожидающий запрос, затем повторно выполните напечатанную команду `openclaw devices approve <requestId>`, чтобы одобрить его. Для удаленного Gateway передавайте явные значения `--url` и `--token`. Чтобы одобрения оставались стабильными между перезапусками, настройте постоянный `adapterConfig.devicePrivateKeyPem` в Paperclip вместо того, чтобы позволять ему генерировать новую временную идентичность устройства при каждом запуске.

<Note>
- Прямые браузерные подключения через local loopback (`127.0.0.1` / `localhost`) одобряются автоматически.
- Tailscale Serve может пропустить цикл сопряжения для операторских сессий Control UI, когда `gateway.auth.allowTailscale: true`, идентичность Tailscale подтверждена, а браузер предоставляет идентичность своего устройства.
- Прямые привязки Tailnet, подключения браузера через LAN и профили браузера без идентичности устройства по-прежнему требуют явного одобрения.
- Каждый профиль браузера генерирует уникальный идентификатор устройства, поэтому при смене браузера или очистке данных браузера потребуется повторное сопряжение.

</Note>

  ## Личная идентичность (локальная для браузера)

  Интерфейс управления поддерживает личную идентичность для каждого браузера (отображаемое имя и аватар), прикрепляемую к исходящим сообщениям для указания авторства в общих сеансах. Она хранится в хранилище браузера, ограничена текущим профилем браузера и не синхронизируется с другими устройствами и не сохраняется на стороне сервера сверх обычных метаданных авторства транскрипта в сообщениях, которые вы действительно отправляете. Очистка данных сайта или смена браузера сбрасывает ее до пустого состояния.

  Та же локальная для браузера схема применяется к переопределению аватара ассистента. Загруженные аватары ассистента накладываются на идентичность, определенную Gateway, только в локальном браузере и никогда не проходят туда-обратно через `config.patch`. Общее поле конфигурации `ui.assistant.avatar` по-прежнему доступно для клиентов без UI, которые записывают поле напрямую (например, скриптовых gateways или пользовательских панелей управления).

  ## Эндпоинт конфигурации среды выполнения

  Интерфейс управления получает настройки среды выполнения из `/control-ui-config.json`, разрешенного относительно базового пути интерфейса управления Gateway (например, `/__openclaw__/control-ui-config.json`, когда UI обслуживается под `/__openclaw__/`). Этот эндпоинт защищен той же аутентификацией Gateway, что и остальная HTTP-поверхность: неаутентифицированные браузеры не могут получить его, а успешное получение требует либо уже действительный токен/пароль Gateway, идентичность Tailscale Serve, либо идентичность доверенного прокси.

  ## Поддержка языков

  Интерфейс управления может локализовать себя при первой загрузке на основе локали вашего браузера. Чтобы позднее переопределить ее, откройте **Обзор -> Доступ к Gateway -> Язык**. Выбор локали находится в карточке доступа к Gateway, а не в разделе оформления.

  - Поддерживаемые локали: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
  - Переводы не на английский язык загружаются в браузере лениво.
  - Выбранная локаль сохраняется в хранилище браузера и повторно используется при будущих посещениях.
  - Отсутствующие ключи перевода откатываются к английскому.

  Переводы документации генерируются для того же набора неанглийских локалей, но встроенный в сайт документации переключатель языков Mintlify ограничен кодами локалей, которые принимает Mintlify. Документация на тайском (`th`) и персидском (`fa`) по-прежнему генерируется в репозитории публикации; она может не отображаться в этом переключателе, пока Mintlify не начнет поддерживать эти коды.

  ## Темы оформления

  Панель оформления сохраняет встроенные темы Claw, Knot и Dash, а также один локальный для браузера слот импорта tweakcn. Чтобы импортировать тему, откройте [редактор tweakcn](https://tweakcn.com/editor/theme), выберите или создайте тему, нажмите **Поделиться** и вставьте скопированную ссылку на тему в оформление. Импортер также принимает URL реестра `https://tweakcn.com/r/themes/<id>`, URL редактора вроде `https://tweakcn.com/editor/theme?theme=amethyst-haze`, относительные пути `/themes/<id>`, необработанные идентификаторы тем и имена тем по умолчанию, такие как `amethyst-haze`.

  Оформление также включает локальную для браузера настройку размера текста. Эта настройка хранится вместе с остальными предпочтениями интерфейса управления, применяется к тексту чата, тексту компоновщика, карточкам инструментов и боковым панелям чата, а также удерживает текстовые поля ввода на уровне не менее 16px, чтобы мобильный Safari не выполнял автоматическое масштабирование при фокусе.

  Импортированные темы хранятся только в текущем профиле браузера. Они не записываются в конфигурацию Gateway и не синхронизируются между устройствами. Замена импортированной темы обновляет один локальный слот; ее очистка переключает активную тему обратно на Claw, если была выбрана импортированная тема.

## Страница MCP

Выделенная страница MCP — это операторское представление для MCP-серверов, управляемых OpenClaw, в `mcp.servers`. Она сама не запускает MCP-транспорты; используйте ее для просмотра и редактирования сохраненной конфигурации, затем используйте `openclaw mcp doctor --probe`, когда нужно живое подтверждение сервера.

Типичный рабочий процесс:

1. Откройте **MCP** на боковой панели.
2. Проверьте карточки сводки с общим количеством серверов, количеством включенных серверов, OAuth-серверов и отфильтрованных серверов.
3. Просмотрите строку каждого сервера: транспорт, включение, auth, фильтры, таймауты и подсказки команд.
4. Переключайте включение, когда сервер должен оставаться настроенным, но не участвовать в runtime-обнаружении.
5. Отредактируйте scoped-раздел конфигурации `mcp` для определений серверов, заголовков, путей TLS/mTLS, метаданных OAuth, фильтров инструментов и метаданных проекции Codex.
6. Используйте **Сохранить** для записи конфигурации или **Сохранить и опубликовать**, когда работающий Gateway должен применить измененную конфигурацию.
7. Запустите `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` или `openclaw mcp reload` из терминала, когда отредактированному процессу нужны статическая диагностика, живое подтверждение или сброс кэшированного runtime.

Страница редактирует URL-подобные значения, содержащие учетные данные, перед отображением и заключает имена серверов в кавычки в фрагментах команд, чтобы скопированные команды продолжали работать с пробелами или shell-метасимволами. Полная справка по CLI и конфигурации находится в [MCP](/ru/cli/mcp).

## Вкладка Activity

Вкладка Activity — это временный локальный для браузера наблюдатель за живой активностью инструментов. Она формируется из того же потока событий Gateway `session.tool` / tool, который питает карточки инструментов Chat; она не добавляет еще одно семейство событий Gateway, endpoint, долговременное хранилище активности, поток метрик или внешний поток наблюдателя.

Записи Activity хранят только очищенные сводки и отредактированные, усеченные предпросмотры вывода. Значения аргументов инструментов не сохраняются в состоянии Activity; UI показывает, что аргументы скрыты, и записывает только количество полей аргументов. Список в памяти привязан к текущей вкладке браузера, сохраняется при навигации внутри Control UI и сбрасывается при перезагрузке страницы, переключении сеанса или **Очистить**.

## Поведение Chat

<AccordionGroup>
  <Accordion title="Семантика отправки и истории">
    - `chat.send` является **неблокирующим**: он сразу подтверждает запрос через `{ runId, status: "started" }`, а ответ транслируется через события `chat`. Доверенные клиенты Control UI также могут получать необязательные метаданные времени ACK для локальной диагностики.
    - Загрузки Chat принимают изображения и невидеофайлы. Изображения сохраняют нативный путь изображения; остальные файлы сохраняются как управляемые медиа и показываются в истории как ссылки на вложения.
    - Повторная отправка с тем же `idempotencyKey` возвращает `{ status: "in_flight" }` во время выполнения и `{ status: "ok" }` после завершения.
    - Ответы `chat.history` ограничены по размеру для безопасности UI. Когда записи транскрипта слишком велики, Gateway может усекать длинные текстовые поля, опускать тяжелые блоки метаданных и заменять чрезмерно большие сообщения заполнителем (`[chat.history omitted: message too large]`).
    - Когда видимое сообщение ассистента было усечено в `chat.history`, боковой reader может по требованию получить полную display-normalized запись транскрипта через `chat.message.get` по `sessionKey`, активному `agentId` при необходимости и `messageId` транскрипта. Если Gateway по-прежнему не может вернуть больше данных, reader показывает явное недоступное состояние вместо молчаливого повторения усеченного предпросмотра.
    - Изображения, созданные ассистентом/сгенерированные, сохраняются как управляемые media-ссылки и отдаются обратно через аутентифицированные media URL Gateway, поэтому перезагрузки не зависят от сохранения сырых base64 image payloads в ответе истории Chat.
    - При отображении `chat.history` Control UI удаляет из видимого текста ассистента display-only inline-теги директив (например, `[[reply_to_*]]` и `[[audio_as_voice]]`), plain-text XML payloads вызовов инструментов (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и усеченные блоки вызовов инструментов), а также просочившиеся ASCII/full-width управляющие токены модели, и опускает записи ассистента, весь видимый текст которых является только точным silent-токеном `NO_REPLY` / `no_reply` или токеном подтверждения Heartbeat `HEARTBEAT_OK`.
    - Во время активной отправки и финального обновления истории представление Chat оставляет локальные optimistic сообщения пользователя/ассистента видимыми, если `chat.history` ненадолго возвращает более старый snapshot; канонический транскрипт заменяет эти локальные сообщения, когда история Gateway догоняет.
    - Живые события `chat` являются состоянием доставки, тогда как `chat.history` перестраивается из долговременного транскрипта сеанса. После tool-final событий Control UI перезагружает историю и объединяет только небольшой optimistic-хвост; граница транскрипта документирована в [WebChat](/ru/web/webchat).
    - `chat.inject` добавляет заметку ассистента в транскрипт сеанса и рассылает событие `chat` для обновлений только UI (без запуска агента, без доставки в канал).
    - Боковая панель показывает последние сеансы с действием New Session, ссылкой All Sessions и кнопкой поиска сеансов, которая открывает полный выборщик сеансов (scoped выбранным агентом, с поиском и пагинацией). При переключении агентов показываются только сеансы, связанные с этим агентом, с fallback на основной сеанс этого агента, если у него еще нет сохраненных dashboard-сеансов.
    - На desktop-ширинах элементы управления Chat остаются в одной компактной строке и сворачиваются при прокрутке транскрипта вниз; прокрутка вверх, возврат к началу или достижение низа восстанавливает элементы управления.
    - Последовательные дублирующиеся сообщения только с текстом отображаются как один пузырь с бейджем количества. Сообщения с изображениями, вложениями, выводом инструментов или canvas-предпросмотрами не сворачиваются.
    - Пикеры модели и thinking в заголовке Chat сразу патчат активный сеанс через `sessions.patch`; это постоянные переопределения сеанса, а не одноразовые параметры отправки.
    - Если вы отправляете сообщение, пока изменение пикера модели для того же сеанса еще сохраняется, композер ждет этот patch сеанса перед вызовом `chat.send`, чтобы отправка использовала выбранную модель.
    - Ввод `/new` в Control UI создает тот же новый dashboard-сеанс, что и New Chat, и переключается на него, кроме случаев, когда настроено `session.dmScope: "main"` и текущий родитель является основным сеансом агента; в этом случае основной сеанс сбрасывается на месте. Ввод `/reset` сохраняет явный in-place reset Gateway для текущего сеанса.
    - Пикер модели Chat запрашивает настроенное представление моделей Gateway. Если присутствует `agents.defaults.models`, этот allowlist управляет пикером, включая записи `provider/*`, которые оставляют provider-scoped каталоги динамическими. В противном случае пикер показывает явные записи `models.providers.*.models` плюс провайдеров с usable auth. Полный каталог остается доступным через отладочный RPC `models.list` с `view: "all"`.
    - Когда свежие отчеты Gateway об использовании сеанса включают текущие context tokens, панель инструментов композера Chat показывает небольшое context usage ring с процентом использования; полная детализация token находится в его tooltip. Ring переключается на warning-стиль при высокой нагрузке на контекст и, на рекомендуемых уровнях Compaction, показывает компактную кнопку, запускающую обычный путь Compaction сеанса. Устаревшие snapshot токенов скрываются, пока Gateway снова не сообщит свежее использование.

  </Accordion>
  <Accordion title="Режим Talk (браузерный realtime)">
    Режим Talk использует зарегистрированного realtime voice provider. Настройте OpenAI с `talk.realtime.provider: "openai"` плюс auth-профиль API-key `openai`, `talk.realtime.providers.openai.apiKey` или `OPENAI_API_KEY`; OAuth-профили OpenAI не настраивают Realtime voice. Настройте Google с `talk.realtime.provider: "google"` плюс `talk.realtime.providers.google.apiKey`. Браузер никогда не получает стандартный provider API key. OpenAI получает эфемерный Realtime client secret для WebRTC. Google Live получает одноразовый ограниченный Live API auth token для браузерного WebSocket-сеанса, при этом инструкции и объявления инструментов фиксируются в токене Gateway. Провайдеры, которые предоставляют только backend realtime bridge, работают через Gateway relay transport, поэтому учетные данные и vendor sockets остаются на стороне сервера, а браузерное аудио передается через аутентифицированные Gateway RPC. Prompt сеанса Realtime собирается Gateway; `talk.client.create` не принимает переопределения инструкций, предоставленные вызывающей стороной.

    Композер Chat содержит кнопку параметров Talk рядом с кнопкой start/stop Talk. Параметры применяются к следующему сеансу Talk и могут переопределять provider, transport, model, voice, reasoning effort, VAD threshold, silence duration и prefix padding. Когда параметр пуст, Gateway использует настроенные defaults, если они доступны, или default провайдера. Выбор Gateway relay принудительно включает backend relay path; выбор WebRTC оставляет сеанс client-owned и завершается ошибкой вместо молчаливого fallback на relay, если провайдер не может создать браузерный сеанс.

    В композере Chat элемент управления Talk — это кнопка волн рядом с кнопкой диктовки через микрофон. Когда Talk запускается, строка статуса композера показывает `Connecting Talk...`, затем `Talk live`, пока аудио подключено, или `Asking OpenClaw...`, пока realtime tool call обращается к настроенной более крупной модели через `talk.client.toolCall`.

    Maintainer live smoke: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` проверяет OpenAI backend WebSocket bridge, OpenAI browser WebRTC SDP exchange, Google Live constrained-token browser WebSocket setup и Gateway relay browser adapter с fake microphone media. Команда печатает только статус провайдера и не логирует секреты.

## Установка PWA и web push

Control UI поставляется с `manifest.webmanifest` и service worker, поэтому современные браузеры могут установить его как автономное PWA. Web Push позволяет Gateway будить установленное PWA с помощью уведомлений, даже когда вкладка или окно браузера не открыты.

Если страница показывает **несовпадение протокола** сразу после обновления OpenClaw, сначала заново откройте панель управления через `openclaw dashboard` и выполните полное обновление страницы. Если ошибка сохраняется, очистите данные сайта для источника панели управления или проверьте в приватном окне браузера; старая вкладка или кеш браузерного service worker может продолжать запускать дообновленную сборку Control UI с новым Gateway.

| Поверхность                                           | Что делает                                                        |
| ----------------------------------------------------- | ----------------------------------------------------------------- |
| `ui/public/manifest.webmanifest`                      | Манифест PWA. Браузеры предлагают «Установить приложение», когда он доступен. |
| `ui/public/sw.js`                                     | Service worker, который обрабатывает события `push` и нажатия уведомлений. |
| `push/vapid-keys.json` (в каталоге состояния OpenClaw) | Автоматически сгенерированная пара ключей VAPID для подписи полезных нагрузок Web Push. |
| `push/web-push-subscriptions.json`                    | Сохраненные конечные точки подписок браузера.                    |

Переопределяйте пару ключей VAPID через переменные окружения процесса Gateway, когда нужно закрепить ключи (для развертываний на нескольких хостах, ротации секретов или тестов):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (по умолчанию `https://openclaw.ai`)

Control UI использует эти методы Gateway с ограничением по scope для регистрации и тестирования браузерных подписок:

- `push.web.vapidPublicKey` — получает активный публичный ключ VAPID.
- `push.web.subscribe` — регистрирует `endpoint` вместе с `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` — удаляет зарегистрированную конечную точку.
- `push.web.test` — отправляет тестовое уведомление в подписку вызывающей стороны.

<Note>
Web Push независим от пути ретрансляции iOS APNS (см. [Конфигурация](/ru/gateway/configuration) для push через relay) и существующего метода `push.test`, которые предназначены для нативного мобильного связывания.
</Note>

## Размещенные встраивания

Сообщения ассистента могут отображать размещенный веб-контент inline через шорткод `[embed ...]`. Политика sandbox для iframe управляется параметром `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    Отключает выполнение скриптов внутри размещенных встраиваний.
  </Tab>
  <Tab title="scripts (default)">
    Разрешает интерактивные встраивания, сохраняя изоляцию origin; это значение по умолчанию и обычно его достаточно для автономных браузерных игр/виджетов.
  </Tab>
  <Tab title="trusted">
    Добавляет `allow-same-origin` поверх `allow-scripts` для документов того же сайта, которым намеренно нужны более сильные привилегии.
  </Tab>
</Tabs>

Пример:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
Используйте `trusted` только когда встроенному документу действительно нужно поведение same-origin. Для большинства игр и интерактивных canvas, сгенерированных агентом, `scripts` — более безопасный выбор.
</Warning>

Абсолютные внешние URL встраивания `http(s)` по умолчанию остаются заблокированными. Если вы намеренно хотите, чтобы `[embed url="https://..."]` загружал сторонние страницы, задайте `gateway.controlUi.allowExternalEmbedUrls: true`.

## Ширина сообщений чата

Сгруппированные сообщения чата используют удобную для чтения максимальную ширину по умолчанию. Развертывания на широких мониторах могут переопределить ее без исправления встроенного CSS, задав `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

Значение проверяется до попадания в браузер. Поддерживаемые значения включают простые длины и проценты, например `960px` или `82%`, а также ограниченные выражения ширины `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` и `fit-content(...)`.

    </Warning>

## Небезопасный HTTP

Если открыть панель управления по обычному HTTP (`http://<lan-ip>` или `http://<tailscale-ip>`), браузер работает в **небезопасном контексте** и блокирует WebCrypto. По умолчанию OpenClaw **блокирует** подключения Control UI без идентичности устройства.

Задокументированные исключения:

- совместимость небезопасного HTTP только для localhost с `gateway.controlUi.allowInsecureAuth=true`
- успешная операторская аутентификация Control UI через `gateway.auth.mode: "trusted-proxy"`
- аварийный параметр `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**Рекомендуемое исправление:** используйте HTTPS (Tailscale Serve) или откройте UI локально:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (на хосте gateway)

<AccordionGroup>
  <Accordion title="Insecure-auth toggle behavior">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` — только локальный переключатель совместимости:

    - Он позволяет сессиям Control UI на localhost продолжаться без идентичности устройства в небезопасных HTTP-контекстах.
    - Он не обходит проверки связывания.
    - Он не ослабляет требования к идентичности устройства для удаленных (не localhost) подключений.

  </Accordion>
  <Accordion title="Break-glass only">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` отключает проверки идентичности устройства Control UI и является серьезным снижением безопасности. Быстро отмените после аварийного использования.
    </Warning>

  </Accordion>
  <Accordion title="Trusted-proxy note">
    - Успешная аутентификация trusted-proxy может допускать **операторские** сессии Control UI без идентичности устройства.
    - Это **не** распространяется на сессии Control UI с ролью node.
    - Reverse proxy на loopback того же хоста все равно не удовлетворяют аутентификации trusted-proxy; см. [Аутентификация через доверенный proxy](/ru/gateway/trusted-proxy-auth).

  </Accordion>
</AccordionGroup>

См. [Tailscale](/ru/gateway/tailscale) для рекомендаций по настройке HTTPS.

## Политика безопасности контента

Control UI поставляется со строгой политикой `img-src`: разрешены только ресурсы **того же origin**, URL `data:` и локально сгенерированные URL `blob:`. Удаленные URL изображений `http(s)` и protocol-relative отклоняются браузером и не вызывают сетевых запросов.

Что это означает на практике:

- Аватары и изображения, обслуживаемые по относительным путям (например, `/avatars/<id>`), по-прежнему отображаются, включая аутентифицированные маршруты аватаров, которые UI получает и преобразует в локальные URL `blob:`.
- Inline URL `data:image/...` по-прежнему отображаются (полезно для полезных нагрузок внутри протокола).
- Локальные URL `blob:`, созданные Control UI, по-прежнему отображаются.
- Удаленные URL аватаров, выданные метаданными канала, удаляются помощниками аватаров Control UI и заменяются встроенным логотипом/значком, поэтому скомпрометированный или вредоносный канал не может заставить браузер оператора выполнять произвольные удаленные запросы изображений.

Вам не нужно ничего менять, чтобы получить это поведение — оно всегда включено и не настраивается.

## Аутентификация маршрута аватаров

Когда аутентификация gateway настроена, конечная точка аватаров Control UI требует тот же токен gateway, что и остальная часть API:

- `GET /avatar/<agentId>` возвращает изображение аватара только аутентифицированным вызывающим сторонам. `GET /avatar/<agentId>?meta=1` возвращает метаданные аватара по тому же правилу.
- Неаутентифицированные запросы к любому из маршрутов отклоняются (как и у соседнего маршрута assistant-media). Это предотвращает утечку идентичности агента через маршрут аватаров на хостах, которые в остальном защищены.
- Сам Control UI передает токен gateway как bearer-заголовок при получении аватаров и использует аутентифицированные blob URL, чтобы изображение по-прежнему отображалось на панелях управления.

Если отключить аутентификацию gateway (не рекомендуется на общих хостах), маршрут аватаров также становится неаутентифицированным, в соответствии с остальной частью gateway.

## Аутентификация маршрута медиа ассистента

Когда аутентификация gateway настроена, локальные предпросмотры медиа ассистента используют двухэтапный маршрут:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` требует обычную операторскую аутентификацию Control UI. Браузер отправляет токен gateway как bearer-заголовок при проверке доступности.
- Успешные ответы с метаданными включают короткоживущий `mediaTicket`, ограниченный именно этим исходным путем.
- URL изображений, аудио, видео и документов, отображаемых браузером, используют `mediaTicket=<ticket>` вместо активного токена или пароля gateway. Билет быстро истекает и не может авторизовать другой источник.

Это сохраняет совместимость обычного рендеринга медиа с нативными медиаэлементами браузера, не помещая переиспользуемые учетные данные gateway в видимые URL медиа.

## Сборка UI

Gateway обслуживает статические файлы из `dist/control-ui`. Соберите их командой:

```bash
pnpm ui:build
```

Необязательная абсолютная база (когда нужны фиксированные URL ресурсов):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

Для локальной разработки (отдельный dev server):

```bash
pnpm ui:dev
```

Затем укажите UI на WS URL вашего Gateway (например, `ws://127.0.0.1:18789`).

## Пустая страница Control UI

Если браузер загружает пустую панель управления, а DevTools не показывает полезной ошибки, расширение или ранний content script мог помешать выполнению JavaScript-модуля приложения. Статическая страница включает обычную HTML-панель восстановления, которая появляется, когда `<openclaw-app>` не зарегистрирован после запуска.

Используйте действие панели **Попробовать снова** после изменения окружения браузера или перезагрузите вручную после этих проверок:

- Отключите расширения, которые внедряются во все страницы, особенно расширения с content script для `<all_urls>`.
- Попробуйте приватное окно, чистый профиль браузера или другой браузер.
- Оставьте Gateway запущенным и проверьте тот же URL панели управления после изменения браузера.

## Отладка/тестирование: dev server + удаленный Gateway

Control UI — это статические файлы; цель WebSocket настраивается и может отличаться от HTTP origin. Это удобно, когда нужен локальный Vite dev server, а Gateway работает в другом месте.

<Steps>
  <Step title="Start the UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="Open with gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    Необязательная одноразовая аутентификация (если нужна):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Примечания">
    - `gatewayUrl` сохраняется в localStorage после загрузки и удаляется из URL.
    - Если вы передаете полный endpoint `ws://` или `wss://` через `gatewayUrl`, URL-кодируйте значение `gatewayUrl`, чтобы браузер корректно разобрал строку запроса.
    - `token` по возможности следует передавать через URL-фрагмент (`#token=...`). Фрагменты не отправляются на сервер, что предотвращает утечку через журналы запросов и Referer. Устаревшие параметры запроса `?token=` по-прежнему импортируются один раз для совместимости, но только как резервный вариант, и сразу удаляются после начальной загрузки.
    - `password` хранится только в памяти.
    - Когда задан `gatewayUrl`, UI не откатывается к учетным данным из конфигурации или окружения. Передайте `token` (или `password`) явно. Отсутствие явных учетных данных является ошибкой.
    - Используйте `wss://`, когда Gateway находится за TLS (Tailscale Serve, HTTPS-прокси и т. д.).
    - `gatewayUrl` принимается только в окне верхнего уровня (не во встроенном), чтобы предотвратить кликджекинг.
    - Публичные развертывания Control UI не через loopback должны явно задавать `gateway.controlUi.allowedOrigins` (полные origins). Приватные загрузки LAN/Tailnet с тем же origin из loopback, RFC1918/link-local, `.local`, `.ts.net` или хостов Tailscale CGNAT принимаются без включения резервного режима по заголовку Host.
    - Запуск Gateway может заполнить локальные origins, такие как `http://localhost:<port>` и `http://127.0.0.1:<port>`, на основе фактических runtime-привязки и порта, но origins удаленных браузеров все равно требуют явных записей.
    - Не используйте `gateway.controlUi.allowedOrigins: ["*"]`, кроме строго контролируемого локального тестирования. Это означает разрешить любой browser origin, а не «совпадать с любым хостом, который я использую».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` включает резервный режим origin на основе заголовка Host, но это опасный режим безопасности.

  </Accordion>
</AccordionGroup>

Пример:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

Подробности настройки удаленного доступа: [Удаленный доступ](/ru/gateway/remote).
