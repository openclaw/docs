UI управления — это небольшое одностраничное приложение **Vite + Lit**, обслуживаемое Gateway:

- по умолчанию: `http://<host>:18789/`
- необязательный префикс: задайте `gateway.controlUi.basePath` (например, `/openclaw`)

Оно взаимодействует **напрямую с WebSocket Gateway** на том же порту.

## Быстрое открытие (локально)

Если Gateway запущен на том же компьютере, откройте:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/))

Если страница не загружается, сначала запустите Gateway: `openclaw gateway`.

<Note>
При нативных LAN-привязках Windows брандмауэр Windows или управляемая организацией групповая политика могут по-прежнему блокировать объявленный LAN URL, даже когда `127.0.0.1` работает на хосте Gateway. Запустите `openclaw gateway status --deep` на хосте Windows; команда сообщает о вероятно заблокированных портах, несовпадениях профилей и локальных правилах брандмауэра, которые политика может игнорировать.
</Note>

Аутентификация передается во время WebSocket-рукопожатия через:

- `connect.params.auth.token`
- `connect.params.auth.password`
- заголовки идентификации Tailscale Serve, когда `gateway.auth.allowTailscale: true`
- заголовки идентификации доверенного прокси, когда `gateway.auth.mode: "trusted-proxy"`

Панель настроек панели управления хранит токен для текущей сессии вкладки браузера и выбранного URL Gateway; пароли не сохраняются. Онбординг обычно создает токен Gateway для аутентификации по общему секрету при первом подключении, но аутентификация по паролю тоже работает, когда `gateway.auth.mode` равно `"password"`.

## Сопряжение устройства (первое подключение)

Когда вы подключаетесь к UI управления из нового браузера или с нового устройства, Gateway обычно требует **одноразового подтверждения сопряжения**. Это мера безопасности для предотвращения несанкционированного доступа.

**Что вы увидите:** "disconnected (1008): pairing required"

<Steps>
  <Step title="Выведите ожидающие запросы">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="Подтвердите по ID запроса">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

Если браузер повторяет попытку сопряжения с измененными данными аутентификации (роль/области действия/публичный ключ), предыдущий ожидающий запрос заменяется, и создается новый `requestId`. Перед подтверждением повторно выполните `openclaw devices list`.

Если браузер уже сопряжен и вы меняете доступ с чтения на запись/администрирование, это обрабатывается как повышение подтверждения, а не как тихое переподключение. OpenClaw оставляет старое подтверждение активным, блокирует переподключение с более широкими правами и просит явно подтвердить новый набор областей действия.

После подтверждения устройство запоминается и не потребует повторного подтверждения, если только вы не отзовете его с помощью `openclaw devices revoke --device <id> --role <role>`. О ротации и отзыве токенов см. [CLI устройств](/ru/cli/devices).

Агенты Paperclip, которые подключаются через адаптер `openclaw_gateway`, используют тот же поток подтверждения при первом запуске. После первой попытки подключения выполните `openclaw devices approve --latest`, чтобы просмотреть ожидающий запрос, затем повторно выполните напечатанную команду `openclaw devices approve <requestId>`, чтобы подтвердить его. Для удаленного Gateway передайте явные значения `--url` и `--token`. Чтобы подтверждения оставались стабильными между перезапусками, настройте постоянный `adapterConfig.devicePrivateKeyPem` в Paperclip вместо генерации новой эфемерной идентичности устройства при каждом запуске.

<Note>
- Прямые браузерные подключения через local loopback (`127.0.0.1` / `localhost`) подтверждаются автоматически.
- Tailscale Serve может пропустить круговое сопряжение для операторских сессий UI управления, когда `gateway.auth.allowTailscale: true`, идентичность Tailscale проверена, и браузер предъявляет свою идентичность устройства.
- Прямые привязки Tailnet, браузерные подключения по LAN и профили браузера без идентичности устройства по-прежнему требуют явного подтверждения.
- Каждый профиль браузера создает уникальный ID устройства, поэтому переключение браузеров или очистка данных браузера потребует повторного сопряжения.

</Note>

## Личная идентичность (локально в браузере)

UI управления поддерживает персональную идентичность для каждого браузера (отображаемое имя и аватар), которая прикрепляется к исходящим сообщениям для атрибуции в совместных сессиях. Она хранится в хранилище браузера, ограничена текущим профилем браузера и не синхронизируется с другими устройствами и не сохраняется на стороне сервера, кроме обычных метаданных авторства в стенограмме для сообщений, которые вы действительно отправляете. Очистка данных сайта или смена браузера сбрасывает ее до пустого значения.

Та же локальная для браузера схема применяется к переопределению аватара ассистента. Загруженные аватары ассистента накладываются на идентичность, разрешенную Gateway, только в локальном браузере и никогда не проходят туда-обратно через `config.patch`. Общее поле конфигурации `ui.assistant.avatar` по-прежнему доступно для клиентов не из UI, которые записывают поле напрямую (например, скриптовых Gateway или пользовательских панелей управления).

## Эндпоинт конфигурации времени выполнения

UI управления получает свои настройки времени выполнения из `/control-ui-config.json`, разрешаемого относительно базового пути UI управления Gateway (например, `/__openclaw__/control-ui-config.json`, когда UI обслуживается под `/__openclaw__/`). Этот эндпоинт защищен той же аутентификацией Gateway, что и остальная HTTP-поверхность: неаутентифицированные браузеры не могут получить его, а успешное получение требует либо уже действительного токена/пароля Gateway, идентичности Tailscale Serve, либо идентичности доверенного прокси.

## Поддержка языков

UI управления может локализоваться при первой загрузке на основе локали вашего браузера. Чтобы переопределить ее позже, откройте **Обзор -> Доступ к Gateway -> Язык**. Выбор локали находится в карточке «Доступ к Gateway», а не в разделе «Внешний вид».

- Поддерживаемые локали: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- Переводы не на английский язык лениво загружаются в браузере.
- Выбранная локаль сохраняется в хранилище браузера и повторно используется при следующих посещениях.
- Отсутствующие ключи перевода откатываются к английскому.

Переводы документации генерируются для того же набора неанглийских локалей, но встроенный в сайт документации выбор языка Mintlify ограничен кодами локалей, которые принимает Mintlify. Документация на тайском (`th`) и персидском (`fa`) по-прежнему генерируется в репозитории публикации; она может не отображаться в этом выборе, пока Mintlify не начнет поддерживать эти коды.

## Темы внешнего вида

Панель «Внешний вид» сохраняет встроенные темы Claw, Knot и Dash, а также один локальный для браузера слот импорта tweakcn. Чтобы импортировать тему, откройте [редактор tweakcn](https://tweakcn.com/editor/theme), выберите или создайте тему, нажмите **Поделиться** и вставьте скопированную ссылку на тему в «Внешний вид». Импортер также принимает URL реестра `https://tweakcn.com/r/themes/<id>`, URL редактора вроде `https://tweakcn.com/editor/theme?theme=amethyst-haze`, относительные пути `/themes/<id>`, необработанные ID тем и имена тем по умолчанию, такие как `amethyst-haze`.

«Внешний вид» также включает локальную для браузера настройку размера текста. Настройка хранится вместе с остальными предпочтениями UI управления, применяется к тексту чата, тексту компоновщика, карточкам инструментов и боковым панелям чата, а также удерживает текстовые поля ввода не меньше 16px, чтобы мобильный Safari не выполнял автоматическое масштабирование при фокусе.

Импортированные темы хранятся только в текущем профиле браузера. Они не записываются в конфигурацию Gateway и не синхронизируются между устройствами. Замена импортированной темы обновляет один локальный слот; очистка переключает активную тему обратно на Claw, если импортированная тема была выбрана.

## Страница MCP

Выделенная страница MCP — это операторское представление для управляемых OpenClaw серверов MCP в `mcp.servers`. Она сама не запускает транспорты MCP; используйте ее для просмотра и редактирования сохраненной конфигурации, а затем запускайте `openclaw mcp doctor --probe`, когда нужно живое подтверждение сервера.

Типовой рабочий процесс:

1. Откройте **MCP** на боковой панели.
2. Проверьте карточки сводки с общим количеством серверов, а также количеством включенных, OAuth и отфильтрованных серверов.
3. Просмотрите строку каждого сервера: транспорт, включение, авторизация, фильтры, тайм-ауты и подсказки команд.
4. Переключайте включение, когда сервер должен оставаться настроенным, но не участвовать в обнаружении во время выполнения.
5. Отредактируйте область конфигурации `mcp` для определений серверов, заголовков, путей TLS/mTLS, метаданных OAuth, фильтров инструментов и метаданных проекции Codex.
6. Используйте **Сохранить** для записи конфигурации или **Сохранить и опубликовать**, когда работающий Gateway должен применить измененную конфигурацию.
7. Запустите `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` или `openclaw mcp reload` из терминала, когда отредактированному процессу нужны статическая диагностика, живое подтверждение или сброс кэшированного runtime.

Страница скрывает URL-подобные значения с учетными данными перед отображением и заключает имена серверов в кавычки в фрагментах команд, чтобы скопированные команды по-прежнему работали с пробелами или метасимволами оболочки. Полная справка по CLI и конфигурации находится в [MCP](/ru/cli/mcp).

## Вкладка активности

Вкладка активности — это временный локальный для браузера наблюдатель за живой активностью инструментов. Она выводится из того же потока событий Gateway `session.tool` / tool, который питает карточки инструментов в чате; она не добавляет еще одно семейство событий Gateway, endpoint, долговременное хранилище активности, поток метрик или внешний поток наблюдателя.

Записи активности хранят только очищенные сводки и отредактированные, усеченные предпросмотры вывода. Значения аргументов инструментов не сохраняются в состоянии активности; UI показывает, что аргументы скрыты, и записывает только количество полей аргументов. Список в памяти привязан к текущей вкладке браузера, сохраняется при навигации внутри Control UI и сбрасывается при перезагрузке страницы, переключении сеанса или нажатии **Очистить**.

## Поведение чата

<AccordionGroup>
  <Accordion title="Семантика отправки и истории">
    - `chat.send` является **неблокирующим**: он сразу подтверждает запрос через `{ runId, status: "started" }`, а ответ транслируется через события `chat`. Доверенные клиенты Control UI также могут получать необязательные метаданные времени ACK для локальной диагностики.
    - Загрузки в чат принимают изображения и не-видео файлы. Изображения сохраняют исходный путь изображения; остальные файлы сохраняются как управляемые медиа и показываются в истории как ссылки на вложения.
    - Повторная отправка с тем же `idempotencyKey` возвращает `{ status: "in_flight" }` во время выполнения и `{ status: "ok" }` после завершения.
    - Ответы `chat.history` ограничены по размеру для безопасности UI. Когда записи стенограммы слишком велики, Gateway может усекать длинные текстовые поля, пропускать тяжелые блоки метаданных и заменять слишком большие сообщения заполнителем (`[chat.history omitted: message too large]`).
    - Когда видимое сообщение ассистента было усечено в `chat.history`, боковой читатель может по запросу получить полную нормализованную для отображения запись стенограммы через `chat.message.get` по `sessionKey`, активному `agentId` при необходимости и `messageId` стенограммы. Если Gateway все еще не может вернуть больше данных, читатель показывает явное состояние недоступности вместо молчаливого повторения усеченного предпросмотра.
    - Изображения, созданные ассистентом, сохраняются как управляемые медиа-ссылки и отдаются обратно через аутентифицированные медиа-URL Gateway, поэтому перезагрузки не зависят от того, остаются ли сырые base64-полезные нагрузки изображений в ответе истории чата.
    - При отображении `chat.history` Control UI удаляет из видимого текста ассистента inline-теги директив, предназначенные только для отображения (например, `[[reply_to_*]]` и `[[audio_as_voice]]`), текстовые XML-полезные нагрузки вызовов инструментов (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и усеченные блоки вызовов инструментов), а также утекшие ASCII/полноширинные управляющие токены модели, и пропускает записи ассистента, весь видимый текст которых состоит только из точного молчаливого токена `NO_REPLY` / `no_reply` или токена подтверждения Heartbeat `HEARTBEAT_OK`.
    - Во время активной отправки и финального обновления истории представление чата сохраняет видимыми локальные оптимистичные сообщения пользователя/ассистента, если `chat.history` ненадолго возвращает более старый снимок; каноническая стенограмма заменяет эти локальные сообщения, когда история Gateway догоняет.
    - Живые события `chat` являются состоянием доставки, тогда как `chat.history` пересобирается из долговременной стенограммы сеанса. После финальных событий инструментов Control UI перезагружает историю и объединяет только небольшой оптимистичный хвост; граница стенограммы описана в [WebChat](/ru/web/webchat).
    - `chat.inject` добавляет заметку ассистента в стенограмму сеанса и транслирует событие `chat` для обновлений только в UI (без запуска агента и без доставки в канал).
    - Заголовок чата показывает фильтр агента перед выбором сеанса, а выбор сеанса ограничен выбранным агентом. При переключении агентов показываются только сеансы, связанные с этим агентом, и выполняется откат к основному сеансу этого агента, если у него еще нет сохраненных сеансов dashboard.
    - На ширине desktop элементы управления чатом остаются в одной компактной строке и сворачиваются при прокрутке стенограммы вниз; прокрутка вверх, возврат к началу или достижение конца восстанавливает элементы управления.
    - Последовательные дублирующиеся сообщения только с текстом отображаются как один пузырь с бейджем количества. Сообщения с изображениями, вложениями, выводом инструментов или предпросмотрами canvas не сворачиваются.
    - Переключатели модели и thinking в заголовке чата сразу исправляют активный сеанс через `sessions.patch`; это постоянные переопределения сеанса, а не параметры отправки только на один ход.
    - Если вы отправляете сообщение, пока изменение переключателя модели для того же сеанса еще сохраняется, composer ждет этого исправления сеанса перед вызовом `chat.send`, чтобы отправка использовала выбранную модель.
    - Ввод `/new` в Control UI создает и переключает на тот же новый сеанс dashboard, что и New Chat, кроме случая, когда настроено `session.dmScope: "main"` и текущий родитель — основной сеанс агента; тогда основной сеанс сбрасывается на месте. Ввод `/reset` сохраняет явный сброс на месте Gateway для текущего сеанса.
    - Переключатель модели чата запрашивает настроенное представление моделей Gateway. Если присутствует `agents.defaults.models`, этот allowlist управляет переключателем, включая записи `provider/*`, которые сохраняют каталоги в области provider динамическими. В противном случае переключатель показывает явные записи `models.providers.*.models` и providers с пригодной авторизацией. Полный каталог остается доступен через отладочный RPC `models.list` с `view: "all"`.
    - Когда свежие отчеты Gateway об использовании сеанса включают текущие токены контекста, область composer чата показывает компактный индикатор использования контекста. Он переключается на предупреждающий стиль при высокой нагрузке на контекст и, на рекомендуемых уровнях Compaction, показывает компактную кнопку, запускающую обычный путь Compaction сеанса. Устаревшие снимки токенов скрываются, пока Gateway снова не сообщит свежие данные об использовании.

Настройте OpenAI с `talk.realtime.provider: "openai"` и профилем авторизации `openai` с API-ключом, `talk.realtime.providers.openai.apiKey` или `OPENAI_API_KEY`; профили OAuth OpenAI не настраивают голос Realtime. Providers, которые предоставляют только backend-мост реального времени, работают через relay-транспорт Gateway, поэтому учетные данные и vendor-сокеты остаются на стороне сервера, а браузерное аудио передается через аутентифицированные RPC Gateway. Подсказка Realtime-сеанса собирается Gateway; `talk.client.create` не принимает переопределения инструкций, предоставленные вызывающей стороной.

Параметры применяются к следующему сеансу разговора и могут переопределять provider, транспорт, модель, голос, reasoning effort, порог VAD, длительность тишины и отступ префикса. Когда параметр пуст, Gateway использует настроенные значения по умолчанию, если они доступны, или значение по умолчанию provider. Выбор Gateway relay принудительно включает backend-путь relay; выбор WebRTC сохраняет сеанс во владении клиента и приводит к ошибке вместо молчаливого отката к relay, если provider не может создать браузерный сеанс.

    В composer чата элемент управления разговором — это кнопка волн рядом с кнопкой диктовки через микрофон. Когда разговор начинается, строка состояния composer показывает `Connecting Talk...`, затем `Talk live`, пока аудио подключено, или `Asking OpenClaw...`, пока вызов инструмента реального времени обращается к настроенной более крупной модели через `talk.client.toolCall`.

    Живой smoke для сопровождающих: `OPENAI_API_KEY=... GEMINI_API_KEY=... node --import tsx scripts/dev/realtime-talk-live-smoke.ts` проверяет backend WebSocket-мост OpenAI, обмен SDP WebRTC браузера OpenAI, настройку браузерного WebSocket Google Live с ограниченным токеном и браузерный адаптер Gateway relay с поддельным микрофонным медиа. Команда печатает только статус provider и не записывает секреты в лог.

## Установка PWA и web push

Control UI поставляется с `manifest.webmanifest` и service worker, поэтому современные браузеры могут установить его как самостоятельное PWA. Web Push позволяет Gateway будить установленное PWA уведомлениями, даже когда вкладка или окно браузера не открыты.

Если страница показывает **Несоответствие протокола** сразу после обновления OpenClaw, сначала заново откройте dashboard с помощью `openclaw dashboard` и выполните жесткое обновление страницы. Если ошибка остается, очистите данные сайта для origin dashboard или проверьте в приватном окне браузера; старая вкладка или кэш browser service worker может продолжать запускать пакет Control UI до обновления против более нового Gateway.

| Поверхность                                          | Что делает                                                        |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | Манифест PWA. Браузеры предлагают "Установить приложение", когда он становится доступен. |
| `ui/public/sw.js`                                     | Service worker, который обрабатывает события `push` и клики по уведомлениям. |
| `push/vapid-keys.json` (в каталоге состояния OpenClaw) | Автоматически сгенерированная пара ключей VAPID для подписи полезных нагрузок Web Push. |
| `push/web-push-subscriptions.json`                    | Сохраненные конечные точки подписок браузеров.                     |

Переопределите пару ключей VAPID через переменные окружения в процессе Gateway, если нужно закрепить ключи (для развертываний на нескольких хостах, ротации секретов или тестов):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (по умолчанию `https://openclaw.ai`)

Control UI использует эти методы Gateway с ограничением по scope для регистрации и тестирования подписок браузера:

- `push.web.vapidPublicKey` — получает активный публичный ключ VAPID.
- `push.web.subscribe` — регистрирует `endpoint` вместе с `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` — удаляет зарегистрированную конечную точку.
- `push.web.test` — отправляет тестовое уведомление в подписку вызывающего клиента.

<Note>
Web Push не зависит от пути ретрансляции iOS APNS (см. [Конфигурация](/ru/gateway/configuration) для push с ретрансляцией) и существующего метода `push.test`, которые предназначены для нативного мобильного сопряжения.
</Note>

## Встроенный размещенный контент

Сообщения ассистента могут отображать размещенный веб-контент встроенно с помощью shortcode `[embed ...]`. Политика песочницы iframe управляется параметром `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    Отключает выполнение скриптов внутри встроенного размещенного контента.
  </Tab>
  <Tab title="scripts (default)">
    Разрешает интерактивный встроенный контент, сохраняя изоляцию origin; это значение используется по умолчанию и обычно достаточно для автономных браузерных игр/виджетов.
  </Tab>
  <Tab title="trusted">
    Добавляет `allow-same-origin` поверх `allow-scripts` для документов с того же сайта, которым намеренно нужны более широкие привилегии.
  </Tab>
</Tabs>

Пример:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
Используйте `trusted` только тогда, когда встроенному документу действительно нужно поведение same-origin. Для большинства созданных агентом игр и интерактивных canvas более безопасный выбор — `scripts`.
</Warning>

Абсолютные внешние URL для встроенного контента `http(s)` по умолчанию остаются заблокированными. Если вы намеренно хотите, чтобы `[embed url="https://..."]` загружал сторонние страницы, задайте `gateway.controlUi.allowExternalEmbedUrls: true`.

## Ширина сообщения чата

Сгруппированные сообщения чата используют читаемую максимальную ширину по умолчанию. Развертывания с широкими мониторами могут переопределить ее без изменения встроенного CSS, задав `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

Значение проверяется до того, как попадет в браузер. Поддерживаются обычные длины и проценты, такие как `960px` или `82%`, а также ограниченные выражения ширины `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` и `fit-content(...)`.

<Tabs>
  <Tab title="Integrated Tailscale Serve (preferred)">

    ```bash
    openclaw gateway --tailscale serve
    ```

    Откройте:

Для операторских сессий Control UI с идентичностью браузерного устройства этот проверенный путь Serve также пропускает цикл сопряжения устройства; браузеры без устройства и подключения с ролью node по-прежнему проходят обычные проверки устройств. Задайте `gateway.auth.allowTailscale: false`, если хотите требовать явные учетные данные общего секрета даже для трафика Serve. Затем используйте `gateway.auth.mode: "token"` или `"password"`.

Поэтому параллельные неудачные повторные попытки из одного браузера могут показать `retry later` на втором запросе вместо двух обычных несовпадений, выполняющихся параллельно.

    <Warning>
    Аутентификация Serve без токена предполагает, что хост gateway доверенный. Если на этом хосте может выполняться недоверенный локальный код, требуйте аутентификацию по токену/паролю.
    </Warning>

## Небезопасный HTTP

Если открыть панель управления через обычный HTTP (`http://<lan-ip>` или `http://<tailscale-ip>`), браузер работает в **небезопасном контексте** и блокирует WebCrypto. По умолчанию OpenClaw **блокирует** подключения Control UI без идентичности устройства.

Документированные исключения:

- совместимость небезопасного HTTP только для localhost с `gateway.controlUi.allowInsecureAuth=true`
- успешная операторская аутентификация Control UI через `gateway.auth.mode: "trusted-proxy"`
- аварийный параметр `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**Рекомендуемое исправление:** используйте HTTPS (Tailscale Serve) или откройте UI локально:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (на хосте gateway)

<AccordionGroup>
  <Accordion title="Insecure-auth toggle behavior">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` — это только локальный переключатель совместимости:

    - Он позволяет сессиям Control UI на localhost продолжать работу без идентичности устройства в небезопасных HTTP-контекстах.
    - Он не обходит проверки сопряжения.
    - Он не ослабляет требования к идентичности устройства для удаленных (не localhost) подключений.

  </Accordion>
  <Accordion title="Break-glass only">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` отключает проверки идентичности устройства Control UI и является серьезным снижением безопасности. Быстро откатите его после аварийного использования.
    </Warning>

  </Accordion>
  <Accordion title="Trusted-proxy note">
    - Успешная аутентификация trusted-proxy может допускать **операторские** сессии Control UI без идентичности устройства.
    - Это **не** распространяется на сессии Control UI с ролью node.
    - Обратные прокси same-host loopback по-прежнему не удовлетворяют аутентификации trusted-proxy; см. [Аутентификация доверенного прокси](/ru/gateway/trusted-proxy-auth).

  </Accordion>
</AccordionGroup>

См. [Tailscale](/ru/gateway/tailscale) для рекомендаций по настройке HTTPS.

## Политика безопасности контента

Control UI поставляется со строгой политикой `img-src`: разрешены только ресурсы **same-origin**, URL `data:` и локально созданные URL `blob:`. Удаленные URL изображений `http(s)` и URL относительно протокола отклоняются браузером и не создают сетевых запросов.

Что это означает на практике:

- Аватары и изображения, отдаваемые по относительным путям (например, `/avatars/<id>`), по-прежнему отображаются, включая аутентифицированные маршруты аватаров, которые UI получает и преобразует в локальные URL `blob:`.
- Встроенные URL `data:image/...` по-прежнему отображаются (полезно для полезных нагрузок внутри протокола).
- Локальные URL `blob:`, созданные Control UI, по-прежнему отображаются.
- Удаленные URL аватаров, выдаваемые метаданными каналов, удаляются помощниками аватаров Control UI и заменяются встроенным логотипом/значком, поэтому скомпрометированный или вредоносный канал не может заставить браузер оператора выполнять произвольные удаленные запросы изображений.

Вам не нужно ничего менять, чтобы получить это поведение — оно всегда включено и не настраивается.

## Аутентификация маршрута аватара

Когда аутентификация gateway настроена, конечная точка аватара Control UI требует тот же токен gateway, что и остальная часть API:

- `GET /avatar/<agentId>` возвращает изображение аватара только аутентифицированным вызывающим клиентам. `GET /avatar/<agentId>?meta=1` возвращает метаданные аватара по тому же правилу.
- Неаутентифицированные запросы к любому из этих маршрутов отклоняются (как и у соседнего маршрута assistant-media). Это предотвращает утечку идентичности агента через маршрут аватара на хостах, которые в остальном защищены.
- Сам Control UI пересылает токен gateway как bearer-заголовок при получении аватаров и использует аутентифицированные URL blob, чтобы изображение по-прежнему отображалось в панелях управления.

Если вы отключите аутентификацию gateway (не рекомендуется на общих хостах), маршрут аватара также станет неаутентифицированным, как и остальная часть gateway.

## Аутентификация маршрута медиа ассистента

Когда аутентификация gateway настроена, предпросмотры локальных медиа ассистента используют двухэтапный маршрут:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` требует обычной операторской аутентификации Control UI. Браузер отправляет токен gateway как bearer-заголовок при проверке доступности.
- Успешные ответы с метаданными включают короткоживущий `mediaTicket`, ограниченный этим точным исходным путем.
- URL изображений, аудио, видео и документов, отображаемых браузером, используют `mediaTicket=<ticket>` вместо активного токена gateway или пароля. Билет быстро истекает и не может авторизовать другой источник.

Это сохраняет совместимость обычного отображения медиа с нативными медиаэлементами браузера без помещения повторно используемых учетных данных gateway в видимые URL медиа.

## Сборка UI

Gateway отдает статические файлы из `dist/control-ui`. Соберите их с помощью:

```bash
pnpm ui:build
```

Необязательная абсолютная база (если нужны фиксированные URL ресурсов):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

Для локальной разработки (отдельный dev server):

```bash
pnpm ui:dev
```

Затем укажите UI на URL Gateway WS (например, `ws://127.0.0.1:18789`).

## Пустая страница Control UI

Если браузер загружает пустую панель управления, а DevTools не показывает полезной ошибки, расширение или ранний content script могли помешать выполнению JavaScript-модуля приложения. Статическая страница включает простую HTML-панель восстановления, которая появляется, когда `<openclaw-app>` не зарегистрирован после запуска.

Используйте действие **Попробовать снова** на панели после изменения окружения браузера или перезагрузите вручную после этих проверок:

- Отключите расширения, которые внедряются во все страницы, особенно расширения с content scripts `<all_urls>`.
- Попробуйте приватное окно, чистый профиль браузера или другой браузер.
- Оставьте Gateway запущенным и проверьте тот же URL панели управления после изменения браузера.

## Отладка/тестирование: dev server + удаленный Gateway

Control UI — это статические файлы; цель WebSocket настраивается и может отличаться от HTTP origin. Это удобно, когда нужен локальный Vite dev server, а Gateway работает в другом месте.

<Steps>
  <Step title="Start the UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="Open with gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    Необязательная одноразовая аутентификация (если нужна):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Примечания">
    - `gatewayUrl` сохраняется в localStorage после загрузки и удаляется из URL.
    - Если вы передаете полный endpoint `ws://` или `wss://` через `gatewayUrl`, URL-кодируйте значение `gatewayUrl`, чтобы браузер правильно разобрал строку запроса.
    - `token` по возможности следует передавать через фрагмент URL (`#token=...`). Фрагменты не отправляются на сервер, что предотвращает утечки через журналы запросов и Referer. Устаревшие параметры запроса `?token=` по-прежнему однократно импортируются для совместимости, но только как резервный вариант, и удаляются сразу после начальной загрузки.
    - `password` хранится только в памяти.
    - Когда задан `gatewayUrl`, UI не откатывается к учетным данным из конфигурации или окружения. Явно укажите `token` (или `password`). Отсутствие явных учетных данных является ошибкой.
    - Используйте `wss://`, когда Gateway находится за TLS (Tailscale Serve, HTTPS-прокси и т. д.).
    - `gatewayUrl` принимается только в окне верхнего уровня (не во встроенном), чтобы предотвратить кликджекинг.
    - Публичные развертывания Control UI не через loopback должны явно задавать `gateway.controlUi.allowedOrigins` (полные origins). Частные загрузки LAN/Tailnet с тем же origin из loopback, RFC1918/link-local, `.local`, `.ts.net` или хостов Tailscale CGNAT принимаются без включения резервного режима по заголовку Host.
    - При запуске Gateway может заполнить локальные origins, такие как `http://localhost:<port>` и `http://127.0.0.1:<port>`, на основе фактических привязки и порта runtime, но origins удаленного браузера все равно требуют явных записей.
    - Не используйте `gateway.controlUi.allowedOrigins: ["*"]`, кроме строго контролируемого локального тестирования. Это означает разрешить любой browser origin, а не «сопоставить с тем хостом, который я использую».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` включает режим резервного определения origin по заголовку Host, но это опасный режим безопасности.

  </Accordion>
</AccordionGroup>

Пример:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

Подробности настройки удаленного доступа: [Удаленный доступ](/ru/gateway/remote).
