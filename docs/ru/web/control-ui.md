Control UI — это небольшое одностраничное приложение на **Vite + Lit**, обслуживаемое Gateway:

- по умолчанию: `http://<host>:18789/`
- необязательный префикс: задайте `gateway.controlUi.basePath` (например, `/openclaw`)

Оно взаимодействует **напрямую с WebSocket Gateway** через тот же порт.

## Быстрое открытие (локально)

Если Gateway запущен на том же компьютере, откройте [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/)).

Если страница не загружается, сначала запустите Gateway: `openclaw gateway`.

<Note>
При привязке к LAN в нативной среде Windows брандмауэр Windows или управляемая организацией групповая политика всё равно могут блокировать опубликованный LAN-URL, даже если `127.0.0.1` работает на хосте Gateway. Выполните `openclaw gateway status --deep` на хосте Windows; команда сообщает о вероятно заблокированных портах, несоответствиях профилей и локальных правилах брандмауэра, которые политика может игнорировать.
</Note>

Данные аутентификации передаются во время установки соединения WebSocket через:

- `connect.params.auth.token`
- `connect.params.auth.password`
- заголовки идентификации Tailscale Serve, когда `gateway.auth.allowTailscale: true`
- заголовки идентификации доверенного прокси, когда `gateway.auth.mode: "trusted-proxy"`

Панель настроек хранит токен для текущего сеанса вкладки браузера и выбранный URL Gateway; пароли не сохраняются. При первоначальной настройке обычно создаётся токен Gateway для аутентификации с общим секретом при первом подключении, но аутентификация по паролю также работает, если `gateway.auth.mode` имеет значение `"password"`.

## Сопряжение устройства (первое подключение)

Для подключения из нового браузера или с нового устройства обычно требуется **однократное подтверждение сопряжения**, отображаемое как `disconnected (1008): pairing required`.

<Steps>
  <Step title="Выведите список ожидающих запросов">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="Подтвердите по идентификатору запроса">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

Если браузер повторяет попытку сопряжения с изменёнными данными аутентификации (ролью, областями доступа или открытым ключом), предыдущий ожидающий запрос заменяется и создаётся новый `requestId`; перед подтверждением снова выполните `openclaw devices list`.

Переключение уже сопряжённого браузера с доступа на чтение на доступ для записи или административный доступ считается повышением уровня подтверждения, а не автоматическим повторным подключением: OpenClaw сохраняет прежнее подтверждение активным, блокирует повторное подключение с более широкими правами и просит явно подтвердить новый набор областей доступа.

После подтверждения устройство запоминается и не требует повторного подтверждения, пока вы не отзовёте его с помощью `openclaw devices revoke --device <id> --role <role>`. Сведения о ротации и отзыве токенов, а также о процедуре первоначального подтверждения Paperclip / `openclaw_gateway` см. в разделе [CLI устройств](/ru/cli/devices).

<Note>
- Прямые локальные подключения браузера через loopback (`127.0.0.1` / `localhost`) подтверждаются автоматически.
- Tailscale Serve может пропустить цикл сопряжения для операторских сеансов Control UI, если `gateway.auth.allowTailscale: true`, идентификация Tailscale успешно проверена и браузер предоставляет идентификатор устройства. Браузеры без идентификатора устройства и подключения с ролью Node по-прежнему проходят обычные проверки устройств.
- Привязки напрямую к Tailnet, подключения браузера по LAN и профили браузеров без идентификатора устройства по-прежнему требуют явного подтверждения.
- Каждый профиль браузера создаёт уникальный идентификатор устройства, поэтому при смене браузера или очистке его данных требуется повторное сопряжение.

</Note>

## Сопряжение мобильного устройства

Уже сопряжённый администратор может создать QR-код подключения для iOS/Android, не открывая терминал:

<Steps>
  <Step title="Откройте сопряжение мобильного устройства">
    Выберите **Устройства**, затем нажмите **Сопрячь мобильное устройство** в карточке **Устройства**.
  </Step>
  <Step title="Подключите телефон">
    В мобильном приложении OpenClaw откройте **Настройки** → **Gateway** и отсканируйте QR-код. Вместо этого можно скопировать и вставить код настройки.
  </Step>
  <Step title="Подтвердите подключение">
    Официальное приложение для iOS/Android подключается автоматически. Если в разделе **Ожидает подтверждения** отображается запрос, перед подтверждением проверьте его роль и области доступа.
  </Step>
</Steps>

Для создания кода настройки требуется `operator.admin`; в сеансах без него кнопка отключена. Код настройки содержит краткосрочные начальные учётные данные, поэтому, пока они действительны, обращайтесь с QR-кодом и скопированным кодом как с паролем. Для удалённого сопряжения Gateway должен разрешаться в `wss://` (например, через Tailscale Serve/Funnel); обычный `ws://` ограничен адресами loopback и частной LAN. Полные сведения о безопасности и резервных вариантах см. в разделе [Сопряжение](/ru/channels/pairing#pair-from-the-control-ui-recommended).

## Личная идентификация (локально в браузере)

Control UI поддерживает отдельную для каждого браузера личную идентификацию (отображаемое имя и аватар), прикрепляемую к исходящим сообщениям для указания авторства в общих сеансах. Она хранится в хранилище браузера, ограничена текущим профилем браузера, не синхронизируется с другими устройствами и не сохраняется на стороне сервера помимо обычных метаданных авторства в расшифровке отправленных вами сообщений. При очистке данных сайта или смене браузера эти данные сбрасываются.

Переопределение аватара ассистента следует той же локальной для браузера схеме: загруженные переопределения локально накладываются на идентификацию, разрешённую Gateway, и никогда не передаются через `config.patch`. Общее поле конфигурации `ui.assistant.avatar` по-прежнему доступно для клиентов без UI, которые записывают это поле напрямую.

## Конечная точка конфигурации среды выполнения

Control UI получает настройки среды выполнения из `/control-ui-config.json`, путь к которому определяется относительно базового пути Control UI в Gateway (например, `/__openclaw__/control-ui-config.json` при базовом пути `/__openclaw__/`). Эта конечная точка защищена той же аутентификацией Gateway, что и остальная HTTP-поверхность: браузеры без аутентификации не могут получить к ней доступ, а для успешного запроса требуются действительный токен или пароль Gateway, идентификация Tailscale Serve либо идентификация доверенного прокси.

## Состояние хоста Gateway

Откройте **Настройки** в простом представлении, чтобы увидеть карточку **Хост Gateway** с данными о компьютере Gateway, LAN-адресе, операционной системе, среде выполнения, времени работы, загрузке ЦП, памяти и месте на диске тома состояния. Пока карточка видима, она обновляется каждые 10 секунд через RPC Gateway `system.info`, для которого требуется область доступа `operator.read`. В старых версиях Gateway и подключениях без этой области доступа карточка не отображается.

## Поддержка языков

При первой загрузке Control UI выбирает язык на основе локали браузера. Чтобы позднее изменить его, откройте **Settings -> General -> Language** (переключатель находится в карточке быстрых настроек General, а не в разделе Appearance).

- Поддерживаемые локали: `en`, `ar`, `de`, `es`, `fa`, `fr`, `hi`, `id`, `it`, `ja-JP`, `ko`, `nl`, `pl`, `pt-BR`, `ru`, `th`, `tr`, `uk`, `vi`, `zh-CN`, `zh-TW`
- Переводы не на английский язык загружаются в браузере отложенно.
- Выбранная локаль сохраняется в хранилище браузера и используется при следующих посещениях.
- Если ключ перевода отсутствует, используется английский текст.

Переводы документации создаются для того же набора неанглийских локалей, но встроенный переключатель языков Mintlify на сайте документации отображает только коды локалей, принимаемые Mintlify. Документация на тайском (`th`) и персидском (`fa`) языках всё равно создаётся в репозитории публикации; она может не отображаться в этом переключателе, пока Mintlify не начнёт поддерживать эти коды.

## Темы оформления

На панели оформления доступны встроенные темы Claw, Knot и Dash (по умолчанию используется Claw), а также один локальный для браузера слот импорта tweakcn. Чтобы импортировать тему, откройте [редактор tweakcn](https://tweakcn.com/editor/theme), выберите или создайте тему, нажмите **Share** и вставьте скопированную ссылку в раздел оформления. Импорт также принимает URL реестра `https://tweakcn.com/r/themes/<id>`, URL редактора вида `https://tweakcn.com/editor/theme?theme=amethyst-haze`, относительные пути `/themes/<id>`, необработанные идентификаторы тем и имена стандартных тем, например `amethyst-haze`.

Импортированные темы хранятся только в текущем профиле браузера; они не записываются в конфигурацию Gateway и не синхронизируются между устройствами. При замене импортированной темы обновляется единственный локальный слот; если импортированная тема была активна, её удаление переключает оформление обратно на Claw.

В разделе оформления также есть локальная для браузера настройка размера текста, которая хранится вместе с остальными параметрами Control UI. Она применяется к тексту чата, редактора сообщений, карточек инструментов и боковых панелей чата и устанавливает для текстовых полей размер не менее 16px, чтобы мобильный Safari не увеличивал масштаб автоматически при получении фокуса.

## Управление плагинами

Откройте **Плагины** на боковой панели или перейдите по пути `/settings/plugins` относительно
настроенного базового пути Control UI, чтобы просматривать плагины и управлять ими, не покидая
Control UI. Например, для базового пути `/openclaw` используется
`/openclaw/settings/plugins`. Страница доступна всегда, даже если все
необязательные плагины отключены.

Раздел **Плагины** служит центром с четырьмя вкладками: **Установленные** и **Обзор** управляют кодом
плагинов в `/settings/plugins`, **Skills** содержит диспетчер навыков для каждого агента по адресу
`/skills`, а **Мастерская** содержит проверку предложений Skill Workshop по адресу
`/skills/workshop`. Каждая вкладка сохраняет собственный URL, а на боковой панели для всех них
отображается единый пункт **Плагины**.

На вкладке **Установленные** отображается полный локальный список, сгруппированный по категориям, со
сводными счётчиками. Каждая строка открывает подробное представление; её меню дополнительных действий (`…`)
позволяет включить или отключить плагин и предлагает действие **Удалить** для плагинов, установленных извне.
Здесь также перечислены настроенные [серверы MCP](/ru/cli/mcp), которые можно добавлять, отключать
и удалять непосредственно на странице. Вкладка **Обзор** представляет собой магазин: избранные плагины,
включённые в OpenClaw, официальные внешние плагины и подключаемые одним нажатием коннекторы MCP
для популярных сервисов. При вводе текста в поле поиска выполняется встроенный поиск по
[ClawHub](https://clawhub.ai/plugins), после чего добавляется раздел **Из ClawHub**
со счётчиками загрузок и значками проверки источника. Глубокие ссылки могут
открывать магазин напрямую с помощью `/settings/plugins?tab=discover`.

На вкладке **Skills** доступны отчёт о состоянии навыков, переключатели включения и отключения, ввод ключа API
и встроенный поиск навыков ClawHub с областью действия, ограниченной выбранным агентом. На вкладке
**Мастерская** находятся доска Skill Workshop и процесс проверки за сегодня для
[предложений навыков](/ru/tools/skill-workshop).

Включённые в поставку плагины уже присутствуют в Gateway, поэтому для них отображается **Включить** или
**Отключить** вместо **Установить**. Например, Workboard включён в поставку
OpenClaw, но по умолчанию отключён, поэтому для него доступно действие **Включить**. Встроенные плагины
нельзя удалить — только отключить.

Для чтения каталога и поиска в ClawHub требуется `operator.read`. Для установки,
включения, отключения или удаления плагина и изменения серверов MCP требуется
`operator.admin`; для операторов с доступом только для чтения эти действия отключены.

Установка из ClawHub выполняется через Gateway и использует те же проверки доверия, целостности
и политики установки плагинов, что и другие установки, выполняемые посредством Gateway. Для установки
или удаления кода плагина требуется перезапуск Gateway. Включение или отключение
установленного плагина может применяться без перезапуска, если это поддерживают плагин и текущая
среда выполнения Gateway; в противном случае UI сообщает, что требуется
перезапуск. Для коннекторов MCP с OAuth после их добавления требуется однократно выполнить
`openclaw mcp login <name>` в CLI.

Эта страница намеренно сосредоточена на списке, поиске, установке, включении
и удалении. Используйте [`openclaw plugins`](/ru/cli/plugins) для произвольных источников npm, git или
локальных путей, обновлений и расширенной настройки плагинов.

## Навигация на боковой панели

Боковая панель закрепляет навигацию над прокручиваемым списком сеансов. В конфигурациях с несколькими агентами каждый агент отображается как сворачиваемый раздел верхнего уровня; развернув агента, можно просматривать его сеансы, не покидая открытый чат, а у свернутых агентов отображается индикатор непрочитанного. Внутри агента список разделяется на **Закрепленные**, по одному встроенному разделу для каждого подключенного канала (Telegram, Slack, WhatsApp, ...), встроенный раздел **Работа** для сеансов, привязанных к управляемому рабочему дереву или Node выполнения (строки показывают строку `repo ⎇ branch` и хост Node), пользовательские группы (`category` сеанса) и **Чаты** для остальных. Разделы каналов и Работы классифицируют строки автоматически; назначение сеанса пользовательской группе всегда имеет приоритет. При открытии сеанса подсветка выбора перемещается без изменения порядка строк. Сеансы с новой активностью с момента последнего прочтения отмечаются точкой непрочитанного, а при открытии помечаются как прочитанные. У каждой строки сеанса есть контекстное меню (кнопка с многоточием или щелчок правой кнопкой мыши) с пунктами Закрепить/Открепить, Пометить как непрочитанное/прочитанное, Переименовать, Ответвить, Переместить в группу (включая Новая группа и Удалить из группы), Архивировать и Удалить; в сенсорных макетах непосредственные элементы закрепления и меню остаются видимыми. Щелчок с Cmd/Ctrl переключает строки в режим множественного выбора, а щелчок с Shift расширяет выбор в пределах видимого порядка; при открытии меню на выбранной строке предлагаются пакетные действия (Пометить N как непрочитанные/прочитанные, Переместить N в группу, Архивировать N, Удалить N), применяемые ко всем выбранным сеансам, с единым подтверждением пакетного удаления. Перетащите сеанс на пользовательскую группу или **Чаты**, чтобы переместить его. Заголовки пользовательских групп можно сворачивать, разворачивать и перетаскивать для изменения порядка; названия групп и их порядок хранятся в gateway (`sessions.groups.*`), поэтому сохраняются во всех браузерах, а состояние свертывания остается в профиле браузера. У заголовков групп также есть меню (кнопка с многоточием или щелчок правой кнопкой мыши) с пунктами Переименовать группу, Новая группа и Удалить группу; переименование или удаление группы обновляет все входящие в нее сеансы на стороне сервера, включая архивированные, а при удалении группы ее сеансы сохраняются и возвращаются в Чаты. Единственная кнопка **+** в заголовке списка сеансов открывает страницу нового сеанса (см. ниже). Элемент управления сортировкой также содержит переключатель Группировать по: С группировкой (по умолчанию) или Без группировки для единого плоского списка (Закрепленные остаются отдельным разделом); выбор сохраняется в текущем профиле браузера. **Использование**, **Автоматизации** и **Плагины** закреплены по умолчанию; строка **Еще** открывает меню со всеми остальными разделами, включая вкладки, предоставленные плагинами. Выберите в этом меню **Изменить закрепленные элементы** или щелкните правой кнопкой мыши область навигации, чтобы закрепить или открепить разделы либо восстановить значения по умолчанию. Набор закрепленных элементов хранится в текущем профиле браузера и сохраняется после перезагрузки.

## Страница нового сеанса

Кнопка **+** в заголовке списка сеансов боковой панели открывает полноэкранный черновик по адресу `/new`: ничего не создается до отправки первого сообщения. Строка назначения над полем сообщения определяет, где работает сеанс: агент (в конфигурациях с несколькими агентами), место выполнения команд (**Gateway · local** или сопряженный Node, предоставляющий `system.run`; требуется `operator.admin`), папка (по умолчанию рабочее пространство агента; для других абсолютных путей Gateway требуются `operator.admin` и рабочее дерево), а также необязательный переключатель **Рабочее дерево** с выбором базовой ветки (на основе `worktrees.branches`, поэтому получение данных не выполняется) и необязательным названием рабочего дерева (ветка получает имя `openclaw/<name>`). Кнопка обзора в элементе папки открывает встроенное средство выбора каталогов, работающее на основе доступного только администраторам метода `fs.listDir`. На верхнем уровне отображаются Gateway и все известные Node; Node не в сети и Node без поддержки просмотра каталогов остаются видимыми, но недоступными. При выборе Gateway просмотр начинается с текущей папки или домашнего каталога Gateway. При выборе поддерживающего эту возможность Node просматривается файловая система его хоста, выполнение привязывается к нему, а выбранный абсолютный путь Node используется напрямую (управляемые рабочие деревья по-прежнему доступны только в Gateway). Отправка вызывает `sessions.create` с первым сообщением, поэтому запуск начинается в рамках того же обмена запросом и ответом, а интерфейс переходит в чат нового сеанса. Если Gateway создает сеанс, но отклоняет первую отправку, чат сохраняет запрос и ошибку после перезагрузки; кнопка **Повторить** отправляет его через уже созданный сеанс вместо создания еще одного.

В разделе **Настройки** отдельная боковая панель начинается с поля **Поиск настроек** для быстрого поиска разделов настроек.

Поле **Поиск** в верхней части боковой панели открывает палитру команд (⌘K). Щелчок по бренду OpenClaw в заголовке боковой панели открывает чистый начальный экран нового сеанса. Когда что-либо требует внимания — неудачные или просроченные задания Cron, истекающая или истекшая аутентификация модели, — над нижней частью боковой панели появляются компактные индикаторы внимания, ведущие на соответствующую страницу. В нижней части активный агент отображается в виде элемента с аватаром (изображение идентичности или эмодзи), именем, индикатором подключения и обновляемым подзаголовком, а рядом находится кнопка **+** для нового сеанса. Щелчок по этому элементу открывает меню агента: переключатель агентов (в конфигурациях с несколькими агентами), «Что может этот агент?», **Настройки агента**, **Настройки**, сопряжение с мобильным устройством, **Документация**, элемент сборки и переключатель цветового режима. В списках более чем из десяти агентов появляется поле фильтра, а закрепленные агенты отображаются первыми; закреплять и откреплять агентов можно на странице настроек Агентов, при этом набор закрепленных агентов хранится в профиле браузера. Выбор агента ограничивает этим агентом Чат, а также разделы Использование, Автоматизации, Задачи, Рабочая доска и Сеансы. На каждой странице с такой областью действия доступен элемент **Агент** с вариантом **Все агенты** для выхода из ограничения; он расширяет область действия общей страницы, не меняя конкретного агента чата, а прямые ссылки на сеансы по-прежнему открывают целевой сеанс. Страница настроек Агентов сохраняет собственный выбор `?agent=` и не следует общей области действия страницы. Когда gateway запущен из рабочей копии исходного кода в ветке, отличной от `main`, в нижней части также красным отображается название этой ветки, чтобы сразу было видно, что используется нерелизный gateway (в релизных установках оно никогда не отображается). Shift-Command-Comma открывает **Настройки**, не переопределяя браузерное сочетание Command-Comma. В заголовке боковой панели также находится переключатель свертывания (⌘B); при свертывании боковая панель полностью скрывается, освобождая всю ширину рабочего пространства, а плавающий элемент развертывания (или ⌘B) возвращает ее; вместо этого в приложении macOS этот переключатель встроен непосредственно в строку заголовка. На настольных устройствах боковая панель является единственным элементом навигационного интерфейса, без верхней панели. На узких экранах боковая панель заменяется выдвижной панелью за компактной строкой заголовка, содержащей переключатель панели, бренд и поиск по палитре команд; в приложении macOS эта строка заголовка объединяет отступ строки заголовка в единую компактную полосу рядом с элементами управления окном. Навигация использует обычную историю браузера, поэтому по ней можно перемещаться кнопками браузера «Назад» и «Вперед»; приложение macOS добавляет нативный переключатель боковой панели рядом с элементами управления окном и жесты смахивания на трекпаде, кнопки «Назад» и «Вперед» у правого края развернутой боковой панели, а также нативные кнопки поиска (палитры команд) и нового сеанса, когда она свернута.

Когда эта функция отключена или подходящая недорогая модель недоступна, строки сохраняют свои детерминированные подписи и вызов модели не выполняется.
    - Запускайте или отклоняйте временные последующие задачи, предложенные моделью; принятые предложения открывают новый сеанс в управляемом рабочем дереве с предложенным запросом.

Экран входа в автономном режиме обрабатывает случай отсутствия подключения; эта страница изменяет параметры подключения, когда оно установлено.
    - Применение и перезапуск с проверкой (`config.apply`), после чего возобновляется последний активный сеанс.
    - При записи используется проверка хеша основы, предотвращающая перезапись параллельных изменений.
    - Перед записью (`config.set`/`config.apply`/`config.patch`) предварительно проверяется разрешение активных SecretRef для ссылок в отправленной конфигурации; отправленные активные ссылки, которые не удалось разрешить, отклоняются до записи.
    - При сохранении формы отбрасываются устаревшие скрытые заполнители, которые невозможно восстановить из сохранённой конфигурации, но сохраняются скрытые значения, всё ещё соответствующие сохранённым секретам.
    - Схема и отрисовка формы формируются из `config.schema` / `config.schema.lookup`, включая поля `title`/`description`, соответствующие подсказки интерфейса, краткие сводки непосредственных дочерних элементов, метаданные документации для вложенных объектов, подстановочных знаков, массивов и узлов композиции, а также схемы плагинов и каналов, если они доступны. Редактор необработанного JSON доступен только тогда, когда снимок допускает безопасное двустороннее преобразование без потерь; в противном случае Control UI принудительно использует режим формы.
    - Действие «Сбросить до сохранённого» в редакторе необработанного JSON сохраняет исходную структуру необработанного содержимого (форматирование, комментарии, расположение `$include`) вместо повторной отрисовки уплощённого снимка, поэтому внешние изменения сохраняются после сброса, если снимок допускает безопасное двустороннее преобразование.
    - Структурированные объектные значения SecretRef отображаются в текстовых полях формы только для чтения, чтобы предотвратить случайное повреждение при преобразовании объекта в строку.

  </Accordion>
  <Accordion title="Использование">

## Страница MCP

Отдельная страница MCP представляет собой операторское представление серверов MCP, управляемых OpenClaw в разделе `mcp.servers`. Она не запускает транспорты MCP самостоятельно; используйте её для просмотра и изменения сохранённой конфигурации, а когда требуется подтверждение работы сервера в реальном времени, используйте `openclaw mcp doctor --probe`.

Типичный рабочий процесс:

1. Откройте **MCP** на боковой панели.
2. Проверьте сводные карточки с общим количеством серверов и количеством включённых серверов, серверов с OAuth и серверов с фильтрами.
3. Просмотрите в строке каждого сервера сведения о транспорте, включении, аутентификации, фильтрах, тайм-аутах и подсказках по командам.
4. Управляйте серверами (добавляйте, включайте, отключайте и удаляйте) на странице **Плагины**, которая является единственным интерактивным средством записи `mcp.servers`; список строк на этой странице содержит ссылку на неё.
5. Изменяйте соответствующий раздел конфигурации `mcp` для определений серверов, заголовков, путей TLS/mTLS, метаданных OAuth, фильтров инструментов и метаданных проекции Codex.
6. Используйте **Сохранить** для записи конфигурации или **Сохранить и опубликовать**, если работающий Gateway должен применить изменённую конфигурацию.
7. Выполните `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` или `openclaw mcp reload` в терминале для статической диагностики, подтверждения работы в реальном времени или сброса кэшированной среды выполнения.

Перед отображением страница скрывает содержащие учётные данные значения, похожие на URL, и заключает имена серверов в кавычки в фрагментах команд, чтобы скопированные команды продолжали работать с пробелами или метасимволами командной оболочки. Полная справка по CLI и конфигурации: [MCP](/ru/cli/mcp).

## Вкладка активности

Вкладка Activity находится в **Settings › System**, рядом с Logs и Debug. Это временный локальный для браузера инструмент наблюдения за активностью инструментов в реальном времени, использующий тот же поток событий Gateway `session.tool` / инструментов, на основе которого работают карточки инструментов в чате. Он не добавляет отдельное семейство событий Gateway, конечную точку, постоянное хранилище активности, поток метрик или внешний поток наблюдения.

Записи активности содержат только очищенные сводки и отредактированные, усечённые предварительные просмотры вывода. Значения аргументов инструментов не сохраняются в состоянии активности; интерфейс указывает, что аргументы скрыты, и записывает только количество полей аргументов. Список в памяти привязан к текущей вкладке браузера, сохраняется при переходах внутри Control UI и сбрасывается при перезагрузке страницы, смене сеанса или нажатии **Clear**.

## Терминал оператора

Закрепляемый терминал оператора по умолчанию отключён. Чтобы включить его, задайте `gateway.terminal.enabled: true` и перезапустите Gateway. Для работы терминала требуется соединение `operator.admin`; он открывает PTY хоста в рабочем пространстве активного агента. Новые вкладки используют выбранного в данный момент агента чата.

<Warning>
Терминал представляет собой неограниченную оболочку хоста и наследует окружение процесса Gateway. Включайте его только в доверенных развёртываниях для операторов. OpenClaw отклоняет терминальные сеансы для агентов с `sandbox.mode: "all"`; перевод активного агента в этот режим закрывает его существующие и выполняющиеся терминальные сеансы.
</Warning>

Используйте **Ctrl + обратная кавычка**, чтобы показать или скрыть закреплённую панель. Компоновка поддерживает закрепление снизу и справа, изменяет размер вместе с областью просмотра браузера и позволяет держать несколько вкладок оболочки. Описание `gateway.terminal.enabled` и необязательного переопределения `gateway.terminal.shell` см. в разделе [Конфигурация Gateway](/ru/gateway/configuration-reference#gateway).

Сеансы сохраняются при разрыве соединения: при перезагрузке страницы, переходе ноутбука в спящий режим или кратковременном сбое сети сеанс отсоединяется на Gateway, а не завершается; после восстановления соединения та же вкладка браузера повторно подключается к нему с воспроизведением недавнего вывода. Отсоединённые сеансы завершаются через `gateway.terminal.detachedSessionTimeoutSeconds` (по умолчанию 300 секунд; `0` восстанавливает завершение при разрыве соединения). `terminal.list` показывает доступные для подключения сеансы, `terminal.attach` принимает управление одним из них (перехват в стиле tmux), а `terminal.text` считывает недавний вывод сеанса как обычный текст без подключения — это возможность для агента и инструментов.

Терминал также доступен как полноэкранный документ, содержащий только терминал, по адресу `/?view=terminal`. Приложения для iOS и Android встраивают эту страницу в свои экраны Terminal, повторно используя сохранённые учётные данные Gateway; доступность определяется теми же ограничениями `gateway.terminal.enabled` и `operator.admin`, а если подключённый Gateway не предоставляет терминал, на странице отображается уведомление.

## Панель браузера

Control UI включает закрепляемую панель браузера, которая отображает управляемый Gateway браузер — тот же, которым агенты управляют через [инструмент браузера](/ru/tools/browser-control), — в любом обычном веб-браузере без необходимости в нативном webview. Она появляется, когда подключённый Gateway объявляет `browser.request` для соединения `operator.admin`; кнопка с глобусом на боковой панели рабочего пространства сеанса показывает или скрывает её. Панель отображает актуальный снимок страницы с вкладками, редактируемой адресной строкой, кнопками перехода назад, вперёд и перезагрузки, а также кнопкой открытия в вашем браузере; она закрепляется справа или снизу и передаёт удалённой странице щелчки, прокрутку колёсиком и базовый ввод текста.

Два режима захвата формируют контекст страницы для агента:

- **Аннотирование (карандаш)**: рисуйте произвольные пометки поверх страницы. Команда **Send to chat** накладывает штрихи на снимок экрана, прикрепляет изображение к редактору активного чата и предварительно заполняет запрос с описанием URL и заголовка страницы, а также каждой отмеченной области, чтобы агент точно знал, что вы обвели.
- **Инспектирование (указатель)**: наведите указатель, чтобы увидеть элемент под курсором (селектор, доступное имя, роль, размер); щёлкните, чтобы отправить сведения об этом элементе вместе со снимком экрана с подсветкой через тот же процесс редактора. Для инспектирования, прокрутки колёсиком и переходов назад и вперёд требуется `browser.evaluateEnabled` (включено по умолчанию).

Приложение для macOS сохраняет нативную боковую панель браузера ссылок для ссылок, открытых на панели управления; панель браузера также работает в нём и служит способом аннотирования страниц на всех остальных платформах.

## Поведение чата

<AccordionGroup>
  <Accordion title="Семантика отправки и истории">
    - `chat.send` выполняется **неблокирующим образом**: подтверждение с `{ runId, status: "started" }` отправляется немедленно, а ответ передаётся в потоковом режиме через события `chat`. Доверенные клиенты Control UI также могут получать необязательные метаданные времени подтверждения для локальной диагностики.
    - Загрузки в чат принимают изображения и файлы, кроме видео. Для изображений сохраняется собственный путь изображения; остальные файлы хранятся как управляемые медиафайлы и отображаются в истории в виде ссылок на вложения.
    - Повторная отправка с тем же `idempotencyKey` возвращает `{ status: "in_flight" }`, пока операция выполняется, и `{ status: "ok" }` после завершения.
    - Размер ответов `chat.history` ограничен для безопасности пользовательского интерфейса. Если записи расшифровки слишком велики, Gateway может обрезать длинные текстовые поля, опускать объёмные блоки метаданных и заменять слишком большие сообщения заполнителем (`[chat.history omitted: message too large]`).
    - Если видимое сообщение ассистента было обрезано в `chat.history`, боковая панель чтения может по запросу получить полную нормализованную для отображения запись расшифровки через `chat.message.get`, используя `sessionKey`, активный `agentId`, когда это необходимо, и `messageId` расшифровки. Если Gateway по-прежнему не может вернуть больше данных, панель чтения явно показывает состояние недоступности вместо незаметного повторения обрезанного предварительного просмотра.
    - Изображения ассистента и сгенерированные изображения сохраняются как ссылки на управляемые медиафайлы и возвращаются через аутентифицированные URL-адреса медиафайлов Gateway, поэтому после перезагрузки не требуется сохранять необработанные данные изображений в формате base64 в ответе истории чата.
    - При отображении `chat.history` Control UI удаляет из видимого текста ассистента встроенные теги директив, предназначенные только для отображения (например, `[[reply_to_*]]` и `[[audio_as_voice]]`), текстовые XML-данные вызовов инструментов (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и обрезанные блоки вызовов инструментов), а также просочившиеся управляющие токены модели в формате ASCII и полноширинном формате. Записи ассистента опускаются, если весь их видимый текст состоит только из точного токена молчания `NO_REPLY` / `no_reply` или токена подтверждения Heartbeat `HEARTBEAT_OK`.
    - Во время активной отправки и окончательного обновления истории представление чата сохраняет видимыми локальные оптимистичные сообщения пользователя и ассистента, если `chat.history` на короткое время возвращает более старый снимок; каноническая расшифровка заменяет эти локальные сообщения, когда история Gateway синхронизируется.
    - События `chat` в реальном времени отражают состояние доставки, а `chat.history` восстанавливается из сохраняемой расшифровки сеанса. После событий завершения инструмента Control UI перезагружает историю и объединяет только небольшой оптимистичный хвост; граница расшифровки описана в разделе [WebChat](/ru/web/webchat).
    - `chat.inject` добавляет заметку ассистента в расшифровку сеанса и рассылает событие `chat` для обновлений только пользовательского интерфейса (без запуска агента и без доставки в канал).
    - На боковой панели перечислены все загруженные активные сеансы: они сгруппированы по разделам агентов и категориям закреплённых, каналов, рабочих, пользовательских и Chats; единое действие New Session открывает диалог черновика. Открытие видимой строки изменяет только выделение. Пользовательские группы можно сворачивать и переупорядочивать перетаскиванием, а сеансы можно переносить в группу или Chats; названия и порядок групп синхронизируются через Gateway, а состояние свёрнутости сохраняется в браузере. Новый сеанс панели управления асинхронно получает краткий сгенерированный заголовок из первого сообщения, не являющегося командой; явно заданные имена никогда не заменяются. Установите `agents.defaults.utilityModel` (или `agents.list[].utilityModel`), чтобы направить этот отдельный вызов модели к модели с меньшей стоимостью. Разворачивание раздела другого агента позволяет просматривать сеансы этого агента, не покидая открытый чат.
    - Поиск сеансов находится в палитре команд (⌘K или поле Search в верхней части боковой панели): при вводе запроса просматривается ограниченное число страниц с совпадениями по всем агентам, отфильтровываются внутренние дочерние строки и строки Cron, а видимые совпадения отображаются рядом с командами навигации. На странице Sessions остаётся полный список с поиском и фильтрами.
    - В каждой строке боковой панели доступно прямое закрепление и полное контекстное меню для управления состоянием непрочитанности, переименования, создания ответвления, группировки, архивирования и удаления. Для нескольких выбранных строк (Cmd/Ctrl-щелчок, Shift-щелчок для диапазонов) доступно пакетное меню, включающее состояние непрочитанности, группировку, архивирование и удаление; пакетные архивирование и удаление остаются недоступными, если хотя бы один выбранный сеанс нельзя архивировать. Активный запуск и основной сеанс агента нельзя архивировать. При архивировании или удалении выбранного в данный момент сеанса Chat переключается обратно на основной сеанс этого агента.

Пока PR ещё не существует, в строке отображается сама ветка — репозиторий, имя ветки и размер различий +/− относительно базы слияния ветки по умолчанию (зафиксированные и незафиксированные изменения), — а также кнопка Create PR, открывающая страницу создания запроса на включение изменений на GitHub. Данные строки ветки поступают только из локального Git, поэтому она остаётся доступной при ограничении частоты запросов GitHub и показывает такое же предупреждение об устаревшем состоянии, поскольку результат «PR не найден» нельзя считать достоверным до сброса ограничения.

## Потеря соединения и повторное подключение

После установления сеанса разрыв соединения с Gateway не приводит к выходу из системы. Панель управления
остаётся видимой, а под верхней панелью отображается плавающая янтарная капсула «Соединение с Gateway потеряно — повторное подключение…»,
пока клиент автоматически повторяет попытки с увеличивающейся задержкой (от 800 мс до 15 с). Обновления
в реальном времени и действия с сеансом приостанавливаются до восстановления соединения; кнопка **Повторить сейчас**
в капсуле принудительно запускает немедленную попытку. Чат остаётся доступным для редактирования: обычный текст и вложения
сохраняются в привязанном к Gateway и сеансу хранилище браузера текущей вкладки, отображаются как ожидающие
повторного подключения и автоматически отправляются после восстановления Gateway. Элементы управления в реальном времени
и команды с косой чертой остаются недоступными без подключения.

Если в этом браузере уже сохранены учётные данные (настроенный токен или пароль либо одобренный
токен устройства), при первом открытии и перезагрузке во время установления соединения отображается небольшой
анимированный знак OpenClaw вместо кратковременного появления формы входа. Форма входа появляется только тогда,
когда учётные данные ещё не сохранены или Gateway активно отклоняет их (неверный токен или пароль, отозванное сопряжение), —
то есть в состояниях, требующих ваших действий, а не ожидания.

## Установка PWA и веб-push-уведомления

Control UI поставляется с `manifest.webmanifest` и сервис-воркером, поэтому современные браузеры могут устанавливать его как автономное PWA-приложение. Web Push позволяет Gateway активировать установленное PWA-приложение с помощью уведомлений, даже когда вкладка или окно браузера не открыты.

Если сразу после обновления OpenClaw на странице отображается сообщение **Несоответствие протокола**, сначала снова откройте панель управления с помощью `openclaw dashboard` и выполните принудительное обновление. Если ошибка сохраняется, очистите данные сайта для источника панели управления или проверьте её в приватном окне браузера; старая вкладка или браузерный кеш сервис-воркера может продолжать запускать пакет Control UI, существовавший до обновления, с более новой версией Gateway.

| Компонент                                             | Назначение                                                         |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | Манифест PWA. Когда он становится доступен, браузеры предлагают «Установить приложение». |
| `ui/public/sw.js`                                     | Сервис-воркер, обрабатывающий события `push` и нажатия на уведомления. |
| `push/vapid-keys.json` (в каталоге состояния OpenClaw) | Автоматически созданная пара ключей VAPID для подписи полезной нагрузки Web Push. |
| `push/web-push-subscriptions.json`                    | Сохранённые конечные точки браузерных подписок.                     |

Переопределите пару ключей VAPID с помощью переменных среды процесса Gateway, если необходимо закрепить ключи (при развёртывании на нескольких узлах, ротации секретов или тестировании):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (по умолчанию `https://openclaw.ai`)

Control UI использует следующие ограниченные областью действия методы Gateway для регистрации и проверки браузерных подписок:

- `push.web.vapidPublicKey` получает активный открытый ключ VAPID.
- `push.web.subscribe` регистрирует `endpoint` вместе с `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` удаляет зарегистрированную конечную точку.
- `push.web.test` отправляет тестовое уведомление в подписку вызывающей стороны.

<Note>
Web Push не зависит от пути ретрансляции APNS в iOS (см. раздел [Конфигурация](/ru/gateway/configuration) о push-уведомлениях через ретрансляцию) и метода `push.test`, предназначенного для собственного механизма сопряжения мобильных устройств.
</Note>

## Встраивание размещённого содержимого

В сообщениях ассистента можно встроить размещённое веб-содержимое с помощью короткого кода `[embed ...]`. Политика песочницы iframe управляется параметром `gateway.controlUi.embedSandbox`:

Встроенный плагин Canvas также предоставляет [`show_widget`](/ru/tools/show-widget) для отображения автономного SVG или HTML непосредственно из вызова инструмента. Браузер объявляет возможность Gateway `inline-widgets`, а полученный документ Canvas остаётся доступным после повторной загрузки истории чата. Выполнения, инициированные из каналов, не получают этот инструмент.

<Tabs>
  <Tab title="строгий">
    Отключает выполнение скриптов во встроенном размещённом содержимом.
  </Tab>
  <Tab title="скрипты (по умолчанию)">
    Разрешает интерактивное встроенное содержимое, сохраняя изоляцию источника; обычно этого достаточно для автономных браузерных игр и виджетов.
  </Tab>
  <Tab title="доверенный">
    Добавляет `allow-same-origin` поверх `allow-scripts` для документов с того же сайта, которым намеренно требуются расширенные привилегии.
  </Tab>
</Tabs>

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
Используйте `trusted`, только если встроенному документу действительно требуется поведение того же источника. Для большинства созданных агентами игр и интерактивных холстов безопаснее выбрать `scripts`.
</Warning>

Абсолютные внешние URL встраивания `http(s)` по умолчанию остаются заблокированными. Чтобы разрешить `[embed url="https://..."]` загружать сторонние страницы, задайте `gateway.controlUi.allowExternalEmbedUrls: true`.

## Ширина сообщений чата

Расшифровка чата отображается в расположенной по центру удобной для чтения области, выровненной с редактором сообщения. Вывод ассистента и инструментов выравнивается по левому краю, а сообщения пользователя — по правому краю внутри этой области. В развёртываниях с широкими мониторами можно переопределить ширину расшифровки без изменения встроенного CSS, задав `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

Значение проверяется до передачи в браузер. Поддерживаются обычные значения длины и процентов, например `960px` или `82%`, а также ограниченные выражения ширины `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` и `fit-content(...)`.

## Доступ через tailnet (рекомендуется)

<Tabs>
  <Tab title="Встроенный Tailscale Serve (предпочтительно)">
    Оставьте Gateway на интерфейсе обратной петли и разрешите Tailscale Serve проксировать его через HTTPS:

    ```bash
    openclaw gateway --tailscale serve
    ```

    Откройте `https://<magicdns>/` (или настроенный вами `gateway.controlUi.basePath`).

    По умолчанию запросы Control UI/WebSocket через Serve могут проходить аутентификацию с помощью заголовков идентификации Tailscale (`tailscale-user-login`), если `gateway.auth.allowTailscale` имеет значение `true`. OpenClaw проверяет идентификацию, разрешая адрес `x-forwarded-for` с помощью `tailscale whois` и сопоставляя его с заголовком, и принимает такие запросы, только если они поступают через интерфейс обратной петли с заголовками Tailscale `x-forwarded-*`. Для операторских сеансов Control UI с идентификацией браузерного устройства этот проверенный путь Serve также пропускает цикл сопряжения устройства; браузеры без идентификации устройства и подключения с ролью Node по-прежнему проходят обычные проверки устройств. Задайте `gateway.auth.allowTailscale: false`, если хотите требовать явные учётные данные на основе общего секрета даже для трафика Serve, а затем используйте `gateway.auth.mode: "token"` или `"password"`.

    Для этого асинхронного пути идентификации Serve неудачные попытки аутентификации с одного IP-адреса клиента и в одной области аутентификации сериализуются до записи ограничения частоты запросов. Поэтому при параллельных повторных попытках с неверными данными из одного браузера второй запрос может получить `retry later` вместо двух обычных несовпадений, обрабатываемых параллельно.

    <Warning>
    Аутентификация Serve без токена предполагает, что узел Gateway является доверенным. Если на этом узле может выполняться недоверенный локальный код, требуйте аутентификацию по токену или паролю.
    </Warning>

  </Tab>
  <Tab title="Привязка к tailnet + токен">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    Откройте `http://<tailscale-ip>:18789/` (или настроенный вами `gateway.controlUi.basePath`).

    Вставьте соответствующий общий секрет в настройки интерфейса (он отправляется как `connect.params.auth.token` или `connect.params.auth.password`).

  </Tab>
</Tabs>

## Небезопасный HTTP

Если открыть панель управления через обычный HTTP (`http://<lan-ip>` или `http://<tailscale-ip>`), браузер работает в **небезопасном контексте** и блокирует WebCrypto. По умолчанию OpenClaw **блокирует** подключения Control UI без идентификации устройства.

Документированные исключения:

- совместимость с небезопасным HTTP только для localhost через `gateway.controlUi.allowInsecureAuth=true`
- успешная операторская аутентификация Control UI через `gateway.auth.mode: "trusted-proxy"`
- аварийный механизм `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**Рекомендуемое решение:** используйте HTTPS (Tailscale Serve) либо откройте интерфейс локально по адресу `https://<magicdns>/` (Serve) или `http://127.0.0.1:18789/` (на узле Gateway).

<AccordionGroup>
  <Accordion title="Поведение переключателя небезопасной аутентификации">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` — это только локальный переключатель совместимости:

    - Он позволяет локальным сеансам Control UI продолжать работу без идентификации устройства в небезопасных контекстах HTTP.
    - Он не обходит проверки сопряжения.
    - Он не смягчает требования к идентификации удалённых устройств (не с localhost).

  </Accordion>
  <Accordion title="Только для экстренных случаев">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` отключает проверки идентификации устройства в Control UI и существенно снижает безопасность. После экстренного использования как можно скорее отмените это изменение.
    </Warning>

  </Accordion>
  <Accordion title="Примечание о доверенном прокси">
    - Успешная аутентификация через доверенный прокси может разрешить сеансы Control UI с ролью **оператора** без идентификации устройства.
    - Это **не** распространяется на сеансы Control UI с ролью Node.
    - Обратные прокси на том же хосте, использующие loopback, по-прежнему не удовлетворяют требованиям аутентификации через доверенный прокси; см. [Аутентификация через доверенный прокси](/ru/gateway/trusted-proxy-auth).

  </Accordion>
</AccordionGroup>

Рекомендации по настройке HTTPS см. в разделе [Tailscale](/ru/gateway/tailscale).

## Политика безопасности контента

Control UI поставляется со строгой политикой `img-src`: разрешены только ресурсы **того же источника**, URL-адреса `data:` и локально созданные URL-адреса `blob:`. Удалённые URL-адреса `http(s)` и URL-адреса изображений, относительные к протоколу, отклоняются браузером, поэтому сетевые запросы к ним никогда не выполняются.

На практике:

- Аватары и изображения, доступные по относительным путям (например, `/avatars/<id>`), по-прежнему отображаются, включая маршруты аватаров с аутентификацией, содержимое которых интерфейс получает и преобразует в локальные URL-адреса `blob:`.
- Встроенные URL-адреса `data:image/...` по-прежнему отображаются.
- Локальные URL-адреса `blob:`, созданные Control UI, по-прежнему отображаются.
- Gateway получает аватары для предпросмотра ссылок GitHub с фиксированного хоста аватаров GitHub и возвращает их в виде ограниченных URL-адресов `data:`; браузер оператора никогда не обращается к удалённому хосту аватаров.
- Удалённые URL-адреса аватаров из метаданных канала удаляются вспомогательными функциями аватаров Control UI и заменяются встроенным логотипом или значком, поэтому скомпрометированный или вредоносный канал не может заставить браузер оператора произвольно загружать удалённые изображения.

Эта защита включена всегда и не настраивается.

## Аутентификация маршрута аватара

Если настроена аутентификация Gateway, конечная точка аватара Control UI требует тот же токен Gateway, что и остальная часть API:

- `GET /avatar/<agentId>` возвращает изображение аватара только аутентифицированным клиентам. `GET /avatar/<agentId>?meta=1` возвращает метаданные аватара по тому же правилу.
- Неаутентифицированные запросы к обоим маршрутам отклоняются (аналогично соседнему маршруту медиафайлов ассистента), поэтому маршрут аватара не может раскрыть идентификационные данные агента на хостах, защищённых в остальных отношениях.
- При получении аватаров Control UI передаёт токен Gateway в заголовке bearer-аутентификации и использует аутентифицированные blob-URL, чтобы изображение продолжало отображаться на панелях мониторинга.

Если отключить аутентификацию Gateway (не рекомендуется на общих хостах), маршрут аватара, как и остальная часть Gateway, также станет доступен без аутентификации.

## Аутентификация маршрута медиафайлов ассистента

Если настроена аутентификация Gateway, для локального предпросмотра медиафайлов ассистента используется двухэтапный маршрут:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` требует обычной операторской аутентификации Control UI; при проверке доступности браузер отправляет токен Gateway в заголовке bearer-аутентификации.
- Успешные ответы с метаданными содержат краткосрочный `mediaTicket`, область действия которого ограничена этим точным исходным путём.
- URL-адреса изображений, аудио, видео и документов, отображаемых браузером, используют `mediaTicket=<ticket>` вместо активного токена или пароля Gateway. Срок действия билета быстро истекает, и он не может предоставить доступ к другому источнику.

Это обеспечивает совместимость отображения медиафайлов со встроенными медиаэлементами браузера, не помещая повторно используемые учётные данные Gateway в видимые URL-адреса медиафайлов.

## Ссылки для подтверждения

Уведомления оператора о запросах подтверждения могут содержать прямую ссылку на отдельный документ подтверждения, доступный в зарезервированном пространстве имён `${controlUiBasePath}/approve/{approvalId}` (например, `/approve/<approvalId>` или `/openclaw/approve/<approvalId>` при настроенном базовом пути). URL-адрес остаётся неизменным в течение всего срока действия запроса и может безопасно пересылаться между вашими устройствами: он идентифицирует запрос подтверждения, но никогда не авторизует его.

- Односегментное пространство имён `/approve/<approvalId>` зарезервировано Gateway перед HTTP-маршрутами плагинов для **всех** методов HTTP, поэтому маршрут плагина никогда не сможет перекрыть или перехватить документ подтверждения.
- Для открытия документа подтверждения требуется та же аутентификация Gateway, что и для остальной части Control UI (токен/пароль, идентификация Tailscale Serve или идентификация доверенного прокси); учётные данные никогда не включаются в URL-адрес подтверждения.
- Если обслуживание Control UI отключено, запросы к этому пространству имён возвращают `404`, а не передаются обработчикам плагинов.
- Вход в систему в документе подтверждения действует временно и только для этой страницы: он не перезаписывает выбор Gateway или настройки, сохранённые полным интерфейсом Control UI в том же браузере.

Gateway обслуживает статические файлы из `dist/control-ui`:

```bash
pnpm ui:build
```

Необязательный абсолютный базовый путь (фиксированные URL-адреса ресурсов):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

Локальная разработка (отдельный сервер разработки):

```bash
pnpm ui:dev
```

Затем укажите в интерфейсе URL-адрес WS вашего Gateway (например, `ws://127.0.0.1:18789`).

## Пустая страница Control UI

Если браузер загружает пустую панель мониторинга, а DevTools не показывает полезных ошибок, возможно, расширение или ранний скрипт содержимого помешал выполнению приложения на основе модулей JavaScript. Статическая страница содержит простую HTML-панель восстановления, которая появляется, если после запуска не зарегистрирован `<openclaw-app>`.

После изменения среды браузера используйте действие **Try again** на панели или перезагрузите страницу вручную после следующих проверок:

- Отключите расширения, внедряющие код во все страницы, особенно расширения со скриптами содержимого `<all_urls>`.
- Попробуйте открыть страницу в приватном окне, чистом профиле браузера или другом браузере.
- Не останавливайте Gateway и после изменения браузера проверьте тот же URL-адрес панели мониторинга.

## Отладка и тестирование: сервер разработки + удалённый Gateway

Control UI состоит из статических файлов; целевой адрес WebSocket настраивается и может отличаться от источника HTTP. Это удобно, если сервер разработки Vite должен работать локально, а Gateway — на другом узле.

<Steps>
  <Step title="Запустите сервер разработки интерфейса">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="Откройте с параметром gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    Необязательная одноразовая аутентификация (при необходимости):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Примечания">
    - `gatewayUrl` сохраняется в localStorage после загрузки и удаляется из URL-адреса.
    - Если через `gatewayUrl` передаётся полная конечная точка `ws://` или `wss://`, закодируйте значение для URL, чтобы браузер правильно разобрал строку запроса.
    - `token` по возможности следует передавать через фрагмент URL (`#token=...`). Фрагменты не отправляются серверу, что предотвращает утечку через журналы запросов и заголовок Referer. Устаревшие параметры запроса `?token=` по-прежнему однократно импортируются для совместимости, но только в качестве резервного варианта, и удаляются сразу после начальной загрузки.
    - `password` хранится только в памяти.
    - Если задан `gatewayUrl`, интерфейс не использует учётные данные из конфигурации или переменных среды в качестве резервного варианта. Укажите `token` (или `password`) явно; отсутствие явно заданных учётных данных является ошибкой.
    - Используйте `wss://`, если Gateway находится за TLS (Tailscale Serve, прокси HTTPS и т. п.).
    - `gatewayUrl` принимается только в окне верхнего уровня (без встраивания), чтобы предотвратить кликджекинг.
    - Для общедоступных развёртываний Control UI не на loopback необходимо явно задать `gateway.controlUi.allowedOrigins` (полные источники). Частные загрузки из того же источника в LAN/Tailnet с loopback, RFC1918/link-local, `.local`, `.ts.net` или хостов Tailscale CGNAT принимаются без включения резервной проверки заголовка Host.
    - При запуске Gateway может добавить локальные источники, такие как `http://localhost:<port>` и `http://127.0.0.1:<port>`, на основе фактических адреса привязки и порта среды выполнения, но источники удалённых браузеров по-прежнему необходимо указывать явно.
    - Не используйте `gateway.controlUi.allowedOrigins: ["*"]`, кроме строго контролируемого локального тестирования; это означает разрешение любого источника браузера, а не «сопоставить с используемым мной хостом».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` включает режим резервной проверки источника по заголовку Host, но это опасный режим безопасности.

  </Accordion>
</AccordionGroup>

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```
