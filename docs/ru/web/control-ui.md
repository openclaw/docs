Control UI — это небольшое одностраничное приложение **Vite + Lit**, обслуживаемое Gateway:

- по умолчанию: `http://<host>:18789/`
- необязательный префикс: задайте `gateway.controlUi.basePath` (например, `/openclaw`)

Оно взаимодействует **напрямую с Gateway WebSocket** на том же порту.

## Быстрое открытие (локально)

Если Gateway запущен на том же компьютере, откройте:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/))

Если страница не загружается, сначала запустите Gateway: `openclaw gateway`.

<Note>
В нативных LAN-привязках Windows брандмауэр Windows или управляемая организацией групповая политика всё ещё могут блокировать объявленный LAN URL, даже когда `127.0.0.1` работает на хосте Gateway. Выполните `openclaw gateway status --deep` на хосте Windows; команда сообщает о вероятно заблокированных портах, несоответствиях профилей и локальных правилах брандмауэра, которые политика может игнорировать.
</Note>

Аутентификация передаётся во время WebSocket-рукопожатия через:

- `connect.params.auth.token`
- `connect.params.auth.password`
- заголовки идентификации Tailscale Serve, когда `gateway.auth.allowTailscale: true`
- заголовки идентификации доверенного прокси, когда `gateway.auth.mode: "trusted-proxy"`

Панель настроек панели управления хранит токен для текущей сессии вкладки браузера и выбранного URL Gateway; пароли не сохраняются. При первом подключении онбординг обычно создаёт токен Gateway для аутентификации с общим секретом, но аутентификация по паролю тоже работает, когда `gateway.auth.mode` имеет значение `"password"`.

## Сопряжение устройства (первое подключение)

Когда вы подключаетесь к Control UI из нового браузера или с нового устройства, Gateway обычно требует **одноразового подтверждения сопряжения**. Это мера безопасности для предотвращения несанкционированного доступа.

**Что вы увидите:** "disconnected (1008): pairing required"

<Steps>
  <Step title="List pending requests">
    ```bash
    openclaw devices list
    ```
  </Step>
  <Step title="Approve by request ID">
    ```bash
    openclaw devices approve <requestId>
    ```
  </Step>
</Steps>

Если браузер повторяет попытку сопряжения с изменёнными данными аутентификации (роль/области доступа/открытый ключ), предыдущий ожидающий запрос заменяется и создаётся новый `requestId`. Перед подтверждением повторно выполните `openclaw devices list`.

Если браузер уже сопряжён и вы меняете доступ с чтения на доступ записи/администратора, это рассматривается как повышение подтверждения, а не как тихое переподключение. OpenClaw сохраняет старое подтверждение активным, блокирует переподключение с более широкими правами и просит явно подтвердить новый набор областей доступа.

После подтверждения устройство запоминается и не потребует повторного подтверждения, если только вы не отзовёте его с помощью `openclaw devices revoke --device <id> --role <role>`. См. [CLI устройств](/ru/cli/devices) для ротации и отзыва токенов.

Агенты Paperclip, подключающиеся через адаптер `openclaw_gateway`, используют тот же поток подтверждения первого запуска. После первой попытки подключения выполните `openclaw devices approve --latest`, чтобы предварительно просмотреть ожидающий запрос, затем повторно выполните напечатанную команду `openclaw devices approve <requestId>`, чтобы подтвердить его. Для удалённого Gateway передайте явные значения `--url` и `--token`. Чтобы подтверждения оставались стабильными между перезапусками, настройте постоянный `adapterConfig.devicePrivateKeyPem` в Paperclip вместо того, чтобы позволять ему создавать новую эфемерную идентичность устройства при каждом запуске.

<Note>
- Прямые browser-подключения через local loopback (`127.0.0.1` / `localhost`) подтверждаются автоматически.
- Tailscale Serve может пропустить цикл сопряжения для операторских сессий Control UI, когда `gateway.auth.allowTailscale: true`, идентичность Tailscale проверена, а браузер предъявляет свою идентичность устройства.
- Прямые привязки Tailnet, подключения браузера по LAN и профили браузера без идентичности устройства всё равно требуют явного подтверждения.
- Каждый профиль браузера создаёт уникальный ID устройства, поэтому смена браузера или очистка данных браузера потребует повторного сопряжения.

</Note>

## Сопряжение мобильного устройства

Уже сопряжённый администратор может создать QR-код подключения iOS/Android без
открытия терминала:

<Steps>
  <Step title="Open mobile pairing">
    Выберите **Nodes**, затем нажмите **Pair mobile device** в карточке **Devices**.
  </Step>
  <Step title="Connect the phone">
    В мобильном приложении OpenClaw откройте **Settings** → **Gateway** и отсканируйте QR-
    код. Вместо этого можно скопировать и вставить код настройки.
  </Step>
  <Step title="Confirm the connection">
    Официальное приложение iOS/Android подключается автоматически. Если **Devices** показывает
    ожидающий запрос, проверьте его роль и области доступа перед подтверждением.
  </Step>
</Steps>

Для создания кода настройки требуется `operator.admin`; кнопка отключена для
сессий без него. Код настройки содержит кратковременные bootstrap-учётные данные,
поэтому обращайтесь с QR-кодом и скопированным кодом как с паролем, пока они действительны. Для удалённого
сопряжения Gateway должен разрешаться в `wss://` (например, через Tailscale
Serve/Funnel); обычный `ws://` ограничен loopback и частными LAN-адресами.
См. [Сопряжение](/ru/channels/pairing#pair-from-the-control-ui-recommended) для
полных сведений о безопасности и резервных вариантах.

## Личная идентичность (локально в браузере)

Control UI поддерживает личную идентичность для каждого браузера (отображаемое имя и аватар), прикрепляемую к исходящим сообщениям для атрибуции в общих сессиях. Она хранится в хранилище браузера, ограничена текущим профилем браузера и не синхронизируется с другими устройствами и не сохраняется на стороне сервера, кроме обычных метаданных авторства расшифровки для сообщений, которые вы фактически отправляете. Очистка данных сайта или смена браузера сбрасывает её в пустое состояние.

Та же локальная для браузера схема применяется к переопределению аватара ассистента. Загруженные аватары ассистента накладываются на разрешённую Gateway идентичность только в локальном браузере и никогда не проходят туда и обратно через `config.patch`. Общее поле конфигурации `ui.assistant.avatar` по-прежнему доступно для клиентов без UI, которые записывают это поле напрямую (например, скриптовых Gateway или пользовательских панелей управления).

## Эндпоинт конфигурации времени выполнения

Control UI получает свои настройки времени выполнения из `/control-ui-config.json`, разрешаемого относительно базового пути Control UI в Gateway (например, `/__openclaw__/control-ui-config.json`, когда UI обслуживается под `/__openclaw__/`). Этот эндпоинт защищён той же аутентификацией Gateway, что и остальная HTTP-поверхность: неаутентифицированные браузеры не могут получить его, а успешное получение требует либо уже действительного токена/пароля Gateway, идентичности Tailscale Serve, либо идентичности доверенного прокси.

## Поддержка языков

Control UI может локализовать себя при первой загрузке на основе локали вашего браузера. Чтобы переопределить её позже, откройте **Overview -> Gateway Access -> Language**. Выбор локали находится в карточке Gateway Access, а не в Appearance.

- Поддерживаемые локали: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `ar`, `it`, `tr`, `uk`, `id`, `pl`, `th`, `vi`, `nl`, `fa`
- Неанглийские переводы загружаются в браузере лениво.
- Выбранная локаль сохраняется в хранилище браузера и повторно используется при будущих посещениях.
- Отсутствующие ключи перевода откатываются к английскому.

Переводы документации создаются для того же набора неанглийских локалей, но встроенный переключатель языка сайта документации Mintlify ограничен кодами локалей, которые принимает Mintlify. Документация на тайском (`th`) и персидском (`fa`) всё равно создаётся в publish-репозитории; она может не отображаться в этом переключателе, пока Mintlify не поддержит эти коды.

## Темы оформления

Панель Appearance сохраняет встроенные темы Claw, Knot и Dash, а также один локальный для браузера слот импорта tweakcn. Чтобы импортировать тему, откройте [редактор tweakcn](https://tweakcn.com/editor/theme), выберите или создайте тему, нажмите **Share** и вставьте скопированную ссылку на тему в Appearance. Импортёр также принимает URL реестра `https://tweakcn.com/r/themes/<id>`, URL редактора вроде `https://tweakcn.com/editor/theme?theme=amethyst-haze`, относительные пути `/themes/<id>`, сырые ID тем и имена тем по умолчанию, такие как `amethyst-haze`.

Appearance также включает локальную для браузера настройку размера текста. Эта настройка хранится вместе с остальными предпочтениями Control UI, применяется к тексту чата, тексту компоновщика, карточкам инструментов и боковым панелям чата, а также удерживает поля ввода текста на уровне не менее 16px, чтобы мобильный Safari не выполнял автоматическое масштабирование при фокусе.

Импортированные темы хранятся только в текущем профиле браузера. Они не записываются в конфигурацию Gateway и не синхронизируются между устройствами. Замена импортированной темы обновляет один локальный слот; её очистка переключает активную тему обратно на Claw, если импортированная тема была выбрана.

## Страница MCP

Отдельная страница MCP — это операторское представление для управляемых OpenClaw серверов MCP в `mcp.servers`. Она сама не запускает транспорты MCP; используйте ее для просмотра и редактирования сохраненной конфигурации, а затем используйте `openclaw mcp doctor --probe`, когда нужно живое подтверждение сервера.

Типичный рабочий процесс:

1. Откройте **MCP** на боковой панели.
2. Проверьте карточки сводки с общим числом серверов, а также количеством включенных серверов, серверов OAuth и отфильтрованных серверов.
3. Просмотрите строку каждого сервера: транспорт, включение, auth, фильтры, тайм-ауты и подсказки команд.
4. Переключайте включение, когда сервер должен оставаться настроенным, но не участвовать в runtime-обнаружении.
5. Редактируйте раздел конфигурации `mcp` с заданной областью для определений серверов, заголовков, путей TLS/mTLS, метаданных OAuth, фильтров инструментов и метаданных проекции Codex.
6. Используйте **Сохранить** для записи конфигурации или **Сохранить и опубликовать**, когда запущенный Gateway должен применить измененную конфигурацию.
7. Запустите `openclaw mcp status --verbose`, `openclaw mcp doctor --probe` или `openclaw mcp reload` из терминала, когда отредактированному процессу нужны статическая диагностика, живое подтверждение или сброс кэшированного runtime.

Страница редактирует URL-подобные значения, содержащие учетные данные, перед рендерингом и заключает имена серверов в кавычки в фрагментах команд, чтобы скопированные команды продолжали работать с пробелами или метасимволами shell. Полный справочник CLI и конфигурации находится в [MCP](/ru/cli/mcp).

## Вкладка активности

Вкладка активности — это эфемерный локальный для браузера наблюдатель за живой активностью инструментов. Она выводится из того же потока событий Gateway `session.tool` / событий инструментов, который питает карточки инструментов в чате; она не добавляет еще одно семейство событий Gateway, endpoint, долговременное хранилище активности, поток метрик или внешний поток наблюдения.

Записи активности хранят только очищенные сводки и отредактированные, усеченные предпросмотры вывода. Значения аргументов инструментов не сохраняются в состоянии активности; UI показывает, что аргументы скрыты, и записывает только количество полей аргументов. Список в памяти следует за текущей вкладкой браузера, переживает навигацию внутри Control UI и сбрасывается при перезагрузке страницы, переключении сессии или **Очистить**.

## Поведение чата

<AccordionGroup>
  <Accordion title="Семантика отправки и истории">
    - `chat.send` является **неблокирующим**: он сразу подтверждает с `{ runId, status: "started" }`, а ответ передается потоком через события `chat`. Доверенные клиенты Control UI также могут получать необязательные метаданные таймингов ACK для локальной диагностики.
    - Загрузки в чат принимают изображения и невидеофайлы. Изображения сохраняют исходный путь изображения; остальные файлы хранятся как управляемые медиа и показываются в истории как ссылки на вложения.
    - Повторная отправка с тем же `idempotencyKey` возвращает `{ status: "in_flight" }` во время выполнения и `{ status: "ok" }` после завершения.
    - Ответы `chat.history` ограничены по размеру для безопасности UI. Когда записи стенограммы слишком велики, Gateway может усекать длинные текстовые поля, опускать тяжелые блоки метаданных и заменять слишком большие сообщения заполнителем (`[chat.history omitted: message too large]`).
    - Когда видимое сообщение ассистента было усечено в `chat.history`, боковой читатель может по требованию получить полную нормализованную для отображения запись стенограммы через `chat.message.get` по `sessionKey`, активному `agentId`, когда он нужен, и `messageId` стенограммы. Если Gateway все еще не может вернуть больше, читатель показывает явное состояние недоступности вместо молчаливого повторения усеченного предпросмотра.
    - Изображения ассистента/сгенерированные изображения сохраняются как управляемые ссылки на медиа и возвращаются через аутентифицированные URL медиа Gateway, поэтому перезагрузки не зависят от того, остаются ли сырые base64-полезные нагрузки изображений в ответе истории чата.
    - При рендеринге `chat.history` Control UI удаляет из видимого текста ассистента display-only встроенные теги директив (например, `[[reply_to_*]]` и `[[audio_as_voice]]`), plain-text XML-полезные нагрузки вызовов инструментов (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и усеченные блоки вызовов инструментов), а также просочившиеся ASCII/полноширинные управляющие токены модели, и опускает записи ассистента, весь видимый текст которых состоит только из точного молчаливого токена `NO_REPLY` / `no_reply` или токена подтверждения Heartbeat `HEARTBEAT_OK`.
    - Во время активной отправки и финального обновления истории представление чата сохраняет локальные оптимистичные сообщения пользователя/ассистента видимыми, если `chat.history` кратковременно возвращает более старый снимок; каноническая стенограмма заменяет эти локальные сообщения, когда история Gateway догоняет.
    - Живые события `chat` являются состоянием доставки, а `chat.history` перестраивается из долговременной стенограммы сессии. После финальных событий инструментов Control UI перезагружает историю и объединяет только небольшой оптимистичный хвост; граница стенограммы задокументирована в [WebChat](/ru/web/webchat).
    - `chat.inject` добавляет заметку ассистента в стенограмму сессии и транслирует событие `chat` для обновлений только UI (без запуска агента, без доставки в канал).
    - Боковая панель показывает недавние сессии с действием создания новой сессии, ссылкой на все сессии и кнопкой поиска сессии, которая открывает полный выбор сессии (в области выбранного агента, с поиском и пагинацией). Переключение агентов показывает только сессии, связанные с этим агентом, и возвращается к основной сессии этого агента, если у него еще нет сохраненных dashboard-сессий.
    - На десктопной ширине элементы управления чатом остаются в одной компактной строке и сворачиваются при прокрутке стенограммы вниз; прокрутка вверх, возврат к началу или достижение конца восстанавливают элементы управления.
    - Последовательные дублирующиеся текстовые сообщения отображаются как один пузырь со счетчиком. Сообщения с изображениями, вложениями, выводом инструментов или предпросмотрами canvas не сворачиваются.
    - Переключатели модели и мышления в заголовке чата немедленно исправляют активную сессию через `sessions.patch`; это постоянные переопределения сессии, а не параметры отправки только на один ход.
    - Если вы отправляете сообщение, пока изменение переключателя модели для той же сессии еще сохраняется, компоновщик ждет этого исправления сессии перед вызовом `chat.send`, чтобы отправка использовала выбранную модель.
    - Ввод `/new` в Control UI создает и переключает на такую же свежую dashboard-сессию, как новая сессия чата, кроме случая, когда настроено `session.dmScope: "main"` и текущий родитель является основной сессией агента; в этом случае он сбрасывает основную сессию на месте. Ввод `/reset` сохраняет явный сброс Gateway на месте для текущей сессии.
    - Переключатель модели чата запрашивает настроенное представление моделей Gateway. Если присутствует `agents.defaults.models`, этот allowlist управляет переключателем, включая записи `provider/*`, которые сохраняют provider-scoped каталоги динамическими. Иначе переключатель показывает явные записи `models.providers.*.models` плюс провайдеров с usable auth. Полный каталог остается доступен через отладочный RPC `models.list` с `view: "all"`.
    - Когда свежие отчеты Gateway об использовании сессии включают текущие токены контекста, панель инструментов компоновщика чата показывает небольшое кольцо использования контекста с процентом использования; полная детализация токенов находится в его подсказке. При высоком давлении контекста кольцо переключается на предупреждающий стиль, а на рекомендованных уровнях Compaction показывает компактную кнопку, которая запускает обычный путь Compaction сессии. Устаревшие снимки токенов скрываются, пока Gateway снова не сообщит свежее использование.

  </Accordion>
  <Accordion title="Режим разговора (браузерный режим реального времени)">

Настройте OpenAI с `talk.realtime.provider: "openai"` плюс профиль auth с API-ключом `openai`, `talk.realtime.providers.openai.apiKey` или `OPENAI_API_KEY`; профили OAuth OpenAI не настраивают голос Realtime. Провайдеры, которые предоставляют только backend realtime bridge, работают через relay-транспорт Gateway, поэтому учетные данные и vendor-сокеты остаются на стороне сервера, пока браузерное аудио передается через аутентифицированные RPC Gateway. Параметры применяются к следующему сеансу разговора и могут переопределять провайдера, транспорт, модель, голос, усилие рассуждения, порог VAD, длительность тишины и начальное дополнение. Выбор ретрансляции Gateway принудительно включает серверный ретрансляционный путь; выбор WebRTC оставляет сеанс во владении клиента и завершает его ошибкой, а не незаметно откатывается к ретрансляции, если провайдер не может создать браузерный сеанс.

Когда разговор запускается, строка состояния компоновщика показывает `Connecting Talk...`, затем `Talk live`, пока аудио подключено, или `Asking OpenClaw...`, пока вызов инструмента реального времени консультируется с настроенной более крупной моделью через `talk.client.toolCall`.

Команда печатает только статус провайдера и не записывает секреты в журнал.

## Установка PWA и Web Push

Control UI поставляется с `manifest.webmanifest` и service worker, поэтому современные браузеры могут установить его как автономное PWA. Web Push позволяет Gateway пробуждать установленное PWA уведомлениями, даже когда вкладка или окно браузера не открыты.

Если страница показывает **Несоответствие протокола** сразу после обновления OpenClaw, сначала заново откройте панель управления с помощью `openclaw dashboard` и выполните жёсткое обновление страницы. Если ошибка сохраняется, очистите данные сайта для origin панели управления или проверьте в приватном окне браузера; старая вкладка или кэш service worker браузера могут продолжать запускать пакет Control UI до обновления против более нового Gateway.

| Поверхность                                           | Что делает                                                         |
| ----------------------------------------------------- | ------------------------------------------------------------------ |
| `ui/public/manifest.webmanifest`                      | Манифест PWA. Браузеры предлагают «Установить приложение», когда он доступен. |
| `ui/public/sw.js`                                     | Service worker, который обрабатывает события `push` и клики по уведомлениям. |
| `push/vapid-keys.json` (в каталоге состояния OpenClaw) | Автоматически сгенерированная пара ключей VAPID, используемая для подписи полезных нагрузок Web Push. |
| `push/web-push-subscriptions.json`                    | Сохранённые конечные точки браузерных подписок.                    |

Переопределяйте пару ключей VAPID через переменные окружения в процессе Gateway, когда нужно закрепить ключи (для развёртываний на нескольких хостах, ротации секретов или тестов):

- `OPENCLAW_VAPID_PUBLIC_KEY`
- `OPENCLAW_VAPID_PRIVATE_KEY`
- `OPENCLAW_VAPID_SUBJECT` (по умолчанию `https://openclaw.ai`)

Control UI использует эти методы Gateway, ограниченные областью доступа, для регистрации и тестирования браузерных подписок:

- `push.web.vapidPublicKey` — получает активный публичный ключ VAPID.
- `push.web.subscribe` — регистрирует `endpoint` вместе с `keys.p256dh`/`keys.auth`.
- `push.web.unsubscribe` — удаляет зарегистрированную конечную точку.
- `push.web.test` — отправляет тестовое уведомление в подписку вызывающего клиента.

<Note>
Web Push независим от пути ретрансляции iOS APNS (см. [Конфигурация](/ru/gateway/configuration) для push с ретрансляцией) и существующего метода `push.test`, которые нацелены на нативное мобильное сопряжение.
</Note>

## Встраиваемые размещённые элементы

Сообщения ассистента могут отображать размещённый веб-контент прямо в строке с помощью шорткода `[embed ...]`. Политика песочницы iframe управляется `gateway.controlUi.embedSandbox`:

<Tabs>
  <Tab title="strict">
    Отключает выполнение скриптов внутри размещённых встраиваемых элементов.
  </Tab>
  <Tab title="scripts (default)">
    Разрешает интерактивные встраиваемые элементы, сохраняя изоляцию origin; это значение по умолчанию и обычно его достаточно для самодостаточных браузерных игр/виджетов.
  </Tab>
  <Tab title="trusted">
    Добавляет `allow-same-origin` поверх `allow-scripts` для документов того же сайта, которым намеренно нужны более сильные привилегии.
  </Tab>
</Tabs>

Пример:

```json5
{
  gateway: {
    controlUi: {
      embedSandbox: "scripts",
    },
  },
}
```

<Warning>
Используйте `trusted` только тогда, когда встроенному документу действительно нужно поведение same-origin. Для большинства сгенерированных агентом игр и интерактивных холстов `scripts` — более безопасный выбор.
</Warning>

Абсолютные внешние URL встраивания `http(s)` по умолчанию остаются заблокированными. Если вы намеренно хотите, чтобы `[embed url="https://..."]` загружал сторонние страницы, задайте `gateway.controlUi.allowExternalEmbedUrls: true`.

## Ширина сообщений Chat

Сгруппированные сообщения Chat используют читаемую максимальную ширину по умолчанию. Развёртывания на широких мониторах могут переопределить её без исправления встроенного CSS, задав `gateway.controlUi.chatMessageMaxWidth`:

```json5
{
  gateway: {
    controlUi: {
      chatMessageMaxWidth: "min(1280px, 82%)",
    },
  },
}
```

Значение проверяется до того, как попадёт в браузер. Поддерживаемые значения включают простые длины и проценты, такие как `960px` или `82%`, а также ограниченные выражения ширины `min(...)`, `max(...)`, `clamp(...)`, `calc(...)` и `fit-content(...)`.

## Доступ Tailnet (рекомендуется)

<Tabs>
  <Tab title="Integrated Tailscale Serve (preferred)">
    Держите Gateway на local loopback и позвольте Tailscale Serve проксировать его через HTTPS:

    ```bash
    openclaw gateway --tailscale serve
    ```

    Откройте:

    - `https://<magicdns>/` (или ваш настроенный `gateway.controlUi.basePath`)

    По умолчанию запросы Control UI/WebSocket Serve могут проходить аутентификацию через заголовки идентичности Tailscale (`tailscale-user-login`), когда `gateway.auth.allowTailscale` равно `true`. OpenClaw проверяет идентичность, разрешая адрес `x-forwarded-for` с помощью `tailscale whois` и сопоставляя его с заголовком, и принимает их только когда запрос приходит на local loopback с заголовками `x-forwarded-*` от Tailscale. Для операторских сеансов Control UI с браузерной идентичностью устройства этот проверенный путь Serve также пропускает цикл сопряжения устройства; браузеры без устройства и подключения с ролью узла всё ещё проходят обычные проверки устройства. Задайте `gateway.auth.allowTailscale: false`, если хотите требовать явные учётные данные общего секрета даже для трафика Serve. Затем используйте `gateway.auth.mode: "token"` или `"password"`.

    Для этого асинхронного пути идентичности Serve неудачные попытки аутентификации для одного и того же IP клиента и области аутентификации сериализуются перед записью ограничений частоты. Поэтому параллельные неверные повторные попытки из одного браузера могут показать `retry later` на втором запросе вместо двух обычных несоответствий, соревнующихся параллельно.

    <Warning>
    Аутентификация Serve без токена предполагает, что хост Gateway доверенный. Если на этом хосте может выполняться недоверенный локальный код, требуйте аутентификацию токеном/паролем.
    </Warning>

  </Tab>
  <Tab title="Bind to tailnet + token">
    ```bash
    openclaw gateway --bind tailnet --token "$(openssl rand -hex 32)"
    ```

    Затем откройте:

    - `http://<tailscale-ip>:18789/` (или ваш настроенный `gateway.controlUi.basePath`)

    Вставьте соответствующий общий секрет в настройки UI (отправляется как `connect.params.auth.token` или `connect.params.auth.password`).

  </Tab>
</Tabs>

## Небезопасный HTTP

Если вы открываете панель управления через обычный HTTP (`http://<lan-ip>` или `http://<tailscale-ip>`), браузер работает в **небезопасном контексте** и блокирует WebCrypto. По умолчанию OpenClaw **блокирует** подключения Control UI без идентичности устройства.

Задокументированные исключения:

- совместимость небезопасного HTTP только для localhost с `gateway.controlUi.allowInsecureAuth=true`
- успешная аутентификация операторского Control UI через `gateway.auth.mode: "trusted-proxy"`
- аварийный режим `gateway.controlUi.dangerouslyDisableDeviceAuth=true`

**Рекомендуемое исправление:** используйте HTTPS (Tailscale Serve) или откройте UI локально:

- `https://<magicdns>/` (Serve)
- `http://127.0.0.1:18789/` (на хосте Gateway)

<AccordionGroup>
  <Accordion title="Insecure-auth toggle behavior">
    ```json5
    {
      gateway: {
        controlUi: { allowInsecureAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    `allowInsecureAuth` — только локальный переключатель совместимости:

    - Он позволяет сеансам Control UI на localhost продолжать работу без идентичности устройства в небезопасных HTTP-контекстах.
    - Он не обходит проверки сопряжения.
    - Он не ослабляет требования к идентичности устройства для удалённых (не localhost) подключений.

  </Accordion>
  <Accordion title="Break-glass only">
    ```json5
    {
      gateway: {
        controlUi: { dangerouslyDisableDeviceAuth: true },
        bind: "tailnet",
        auth: { mode: "token", token: "replace-me" },
      },
    }
    ```

    <Warning>
    `dangerouslyDisableDeviceAuth` отключает проверки идентичности устройства Control UI и является серьёзным снижением безопасности. Быстро верните настройку после аварийного использования.
    </Warning>

  </Accordion>
  <Accordion title="Trusted-proxy note">
    - Успешная аутентификация trusted-proxy может допускать **операторские** сеансы Control UI без идентичности устройства.
    - Это **не** распространяется на сеансы Control UI с ролью узла.
    - Обратные прокси same-host loopback всё равно не удовлетворяют аутентификации trusted-proxy; см. [Аутентификация через доверенный прокси](/ru/gateway/trusted-proxy-auth).

  </Accordion>
</AccordionGroup>

См. [Tailscale](/ru/gateway/tailscale) для инструкций по настройке HTTPS.

## Политика безопасности контента

Control UI поставляется со строгой политикой `img-src`: разрешены только ресурсы **same-origin**, URL `data:` и локально сгенерированные URL `blob:`. Удалённые URL изображений `http(s)` и protocol-relative отклоняются браузером и не вызывают сетевых запросов.

Что это означает на практике:

- Аватары и изображения, обслуживаемые по относительным путям (например, `/avatars/<id>`), всё ещё отображаются, включая аутентифицированные маршруты аватаров, которые UI запрашивает и преобразует в локальные URL `blob:`.
- Встроенные URL `data:image/...` всё ещё отображаются (полезно для полезных нагрузок внутри протокола).
- Локальные URL `blob:`, созданные Control UI, всё ещё отображаются.
- Удалённые URL аватаров, переданные метаданными канала, удаляются помощниками аватаров Control UI и заменяются встроенным логотипом/значком, поэтому скомпрометированный или вредоносный канал не может принудить браузер оператора выполнять произвольные удалённые запросы изображений.

Вам не нужно ничего менять, чтобы получить это поведение — оно всегда включено и не настраивается.

## Аутентификация маршрута аватара

Когда аутентификация Gateway настроена, конечная точка аватара Control UI требует тот же токен Gateway, что и остальная часть API:

- `GET /avatar/<agentId>` возвращает изображение аватара только аутентифицированным вызывающим клиентам. `GET /avatar/<agentId>?meta=1` возвращает метаданные аватара по тому же правилу.
- Неаутентифицированные запросы к любому из маршрутов отклоняются (как у соседнего маршрута assistant-media). Это предотвращает утечку идентичности агента через маршрут аватара на хостах, которые в остальном защищены.
- Сам Control UI пересылает токен Gateway как bearer-заголовок при получении аватаров и использует аутентифицированные URL blob, чтобы изображение всё равно отображалось в панелях управления.

Если вы отключите аутентификацию Gateway (не рекомендуется на общих хостах), маршрут аватара также станет неаутентифицированным, как и остальная часть Gateway.

## Аутентификация маршрута медиа ассистента

Когда аутентификация Gateway настроена, предварительные просмотры локальных медиа ассистента используют двухэтапный маршрут:

- `GET /__openclaw__/assistant-media?meta=1&source=<path>` требует обычной аутентификации оператора Control UI. Браузер отправляет токен Gateway как bearer-заголовок при проверке доступности.
- Успешные ответы метаданных включают короткоживущий `mediaTicket`, ограниченный этим точным исходным путем.
- URL-адреса изображений, аудио, видео и документов, отображаемых браузером, используют `mediaTicket=<ticket>` вместо активного токена или пароля Gateway. Тикет быстро истекает и не может авторизовать другой источник.

Это сохраняет совместимость обычного отображения медиа с нативными медиаэлементами браузера, не помещая переиспользуемые учетные данные Gateway в видимые URL-адреса медиа.

## Сборка UI

Gateway раздает статические файлы из `dist/control-ui`. Соберите их с помощью:

```bash
pnpm ui:build
```

Необязательная абсолютная база (когда нужны фиксированные URL-адреса ресурсов):

```bash
OPENCLAW_CONTROL_UI_BASE_PATH=/openclaw/ pnpm ui:build
```

Для локальной разработки (отдельный сервер разработки):

```bash
pnpm ui:dev
```

Затем укажите UI URL-адрес WS вашего Gateway (например, `ws://127.0.0.1:18789`).

## Пустая страница Control UI

Если браузер загружает пустую панель мониторинга, а DevTools не показывает полезной ошибки, расширение или ранний скрипт содержимого мог помешать выполнению JavaScript-модуля приложения. Статическая страница включает простую HTML-панель восстановления, которая появляется, когда `<openclaw-app>` не зарегистрирован после запуска.

Используйте действие **Try again** на панели после изменения окружения браузера или перезагрузите вручную после этих проверок:

- Отключите расширения, которые внедряются во все страницы, особенно расширения со скриптами содержимого `<all_urls>`.
- Попробуйте приватное окно, чистый профиль браузера или другой браузер.
- Оставьте Gateway запущенным и проверьте тот же URL-адрес панели мониторинга после изменения браузера.

## Отладка/тестирование: сервер разработки + удаленный Gateway

Control UI состоит из статических файлов; цель WebSocket настраивается и может отличаться от HTTP-origin. Это удобно, когда нужен локальный сервер разработки Vite, а Gateway работает в другом месте.

<Steps>
  <Step title="Start the UI dev server">
    ```bash
    pnpm ui:dev
    ```
  </Step>
  <Step title="Open with gatewayUrl">
    ```text
    http://localhost:5173/?gatewayUrl=ws%3A%2F%2F<gateway-host>%3A18789
    ```

    Необязательная одноразовая аутентификация (если нужна):

    ```text
    http://localhost:5173/?gatewayUrl=wss%3A%2F%2F<gateway-host>%3A18789#token=<gateway-token>
    ```

  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Notes">
    - `gatewayUrl` сохраняется в localStorage после загрузки и удаляется из URL.
    - Если вы передаете полный endpoint `ws://` или `wss://` через `gatewayUrl`, URL-кодируйте значение `gatewayUrl`, чтобы браузер корректно разобрал строку запроса.
    - `token` по возможности следует передавать через фрагмент URL (`#token=...`). Фрагменты не отправляются на сервер, что предотвращает утечку через журналы запросов и Referer. Устаревшие параметры запроса `?token=` все еще импортируются один раз для совместимости, но только как резервный вариант, и сразу удаляются после bootstrap.
    - `password` хранится только в памяти.
    - Когда задан `gatewayUrl`, UI не возвращается к учетным данным из конфигурации или окружения. Укажите `token` (или `password`) явно. Отсутствие явных учетных данных является ошибкой.
    - Используйте `wss://`, когда Gateway находится за TLS (Tailscale Serve, HTTPS-прокси и т. д.).
    - `gatewayUrl` принимается только в окне верхнего уровня (не во встроенном), чтобы предотвратить кликджекинг.
    - Публичные развертывания Control UI не на local loopback должны явно задавать `gateway.controlUi.allowedOrigins` (полные origins). Частные загрузки из той же origin в LAN/Tailnet с local loopback, RFC1918/link-local, `.local`, `.ts.net` или хостов Tailscale CGNAT принимаются без включения резервного режима по заголовку Host.
    - Запуск Gateway может заполнять локальные origins, такие как `http://localhost:<port>` и `http://127.0.0.1:<port>`, на основе фактических runtime-адреса привязки и порта, но origins удаленных браузеров все равно требуют явных записей.
    - Не используйте `gateway.controlUi.allowedOrigins: ["*"]`, кроме строго контролируемого локального тестирования. Это означает разрешить любую browser origin, а не «сопоставить любой хост, который я использую».
    - `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` включает резервный режим origin по заголовку Host, но это опасный режим безопасности.

  </Accordion>
</AccordionGroup>

Пример:

```json5
{
  gateway: {
    controlUi: {
      allowedOrigins: ["http://localhost:5173"],
    },
  },
}
```

Сведения о настройке удаленного доступа: [Удаленный доступ](/ru/gateway/remote).
