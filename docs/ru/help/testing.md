OpenClaw включает три набора тестов Vitest (модульные/интеграционные, e2e и живые), а также средства запуска в Docker. На этой странице описано, что проверяет каждый набор, какую команду запускать для конкретного рабочего процесса, как живые тесты обнаруживают учетные данные и как добавлять регрессионные тесты для реальных ошибок провайдеров и моделей.

<Note>
**Стек контроля качества (qa-lab, qa-channel, живые транспортные сценарии)** описан отдельно:

- [Обзор контроля качества](/ru/concepts/qa-e2e-automation) — архитектура, интерфейс команд и создание сценариев.
- [Матричный контроль качества](/ru/concepts/qa-matrix) — справочник по `pnpm openclaw qa matrix`.
- [Карточка оценки зрелости](/ru/maturity/scorecard) — как свидетельства контроля качества релиза используются при принятии решений о стабильности и LTS.
- [Канал контроля качества](/ru/channels/qa-channel) — синтетический транспортный Plugin, используемый сценариями на основе репозитория.

На этой странице описаны обычные наборы тестов и средства запуска в Docker/Parallels. В разделе [Средства запуска для контроля качества](#qa-specific-runners) ниже перечислены конкретные вызовы `qa` и приведены ссылки на указанные выше материалы.
</Note>

## Быстрый старт

В большинстве случаев:

- Полная проверка (ожидается перед отправкой изменений): `pnpm build && pnpm check && pnpm check:test-types && pnpm test`
- Более быстрый локальный запуск полного набора на мощной машине: `pnpm test:max`
- Непосредственный цикл наблюдения Vitest: `pnpm test:watch`
- Прямое указание файла также обрабатывает пути плагинов и каналов: `pnpm test extensions/discord/src/monitor/message-handler.preflight.test.ts`
- При работе над отдельной ошибкой сначала предпочитайте целевые запуски.
- Сайт контроля качества на базе Docker: `pnpm qa:lab:up`
- Сценарий контроля качества на базе виртуальной машины Linux: `pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline`

При изменении тестов или для дополнительной уверенности:

- Информационный отчет о покрытии V8: `pnpm test:coverage`
- Набор e2e-тестов: `pnpm test:e2e`

## Временные каталоги тестов

Используйте общие вспомогательные функции из `test/helpers/temp-dir.ts` для временных каталогов, принадлежащих тестам, чтобы владелец был указан явно, а очистка выполнялась в рамках жизненного цикла теста:

```ts
import { afterEach } from "vitest";
import { useAutoCleanupTempDirTracker } from "../helpers/temp-dir.js";

const tempDirs = useAutoCleanupTempDirTracker(afterEach);

it("uses a temp workspace", () => {
  const workspace = tempDirs.make("openclaw-example-");
  // use workspace
});
```

`useAutoCleanupTempDirTracker(afterEach)` намеренно не предоставляет метода ручной очистки — Vitest выполняет очистку после каждого теста. Старые низкоуровневые вспомогательные функции (`makeTempDir`, `cleanupTempDirs`, `createTempDirTracker`) по-прежнему существуют для тестов, которые еще не были перенесены; не используйте их в новом коде и не добавляйте прямые вызовы `fs.mkdtemp*`, если только тест явно не проверяет необработанное поведение временных каталогов. Если прямое создание временного каталога действительно необходимо, добавьте проверяемый разрешающий комментарий с указанием причины:

```ts
// openclaw-temp-dir: allow verifies raw fs cleanup behavior
const workspace = fs.mkdtempSync(prefix);
```

`node scripts/report-test-temp-creations.mjs` сообщает о новых прямых созданиях временных каталогов и новом ручном использовании общих вспомогательных функций в добавленных строках различий, не блокируя существующие способы очистки. Скрипт использует ту же классификацию путей тестов, что и `scripts/changed-lanes.mjs`, и пропускает саму реализацию общей вспомогательной функции. `check:changed` запускает этот отчет для измененных путей тестов как сигнал CI только с предупреждением (аннотации предупреждений GitHub, а не ошибки).

## Рабочие процессы с живыми тестами и Docker/Parallels

При отладке реальных провайдеров и моделей (требуются реальные учетные данные):

- Набор живых тестов (модели и проверки инструментов/изображений Gateway): `pnpm test:live`
- Тихий запуск одного файла живых тестов: `pnpm test:live -- src/agents/models.profiles.live.test.ts`
- Отчеты о производительности среды выполнения: запустите `OpenClaw Performance` с
  `live_openai_candidate=true` для реального хода агента `openai/gpt-5.6-luna` или
  `deep_profile=true` для артефактов процессора, кучи и трассировки Kova. Ежедневные запланированные запуски
  публикуют отчеты сценариев с имитацией провайдера, глубоким профилированием и GPT-5.6 Luna в
  `openclaw/clawgrit-reports` из отдельной задачи публикации, потребляющей артефакты;
  отсутствие или недействительность аутентификации издателя приводит к сбою запланированных запусков и
  запусков с `profile=release`. При ручных запусках, не относящихся к релизу, артефакты GitHub
  сохраняются, а публикация отчетов считается рекомендательной. Отчет с имитацией провайдера также
  включает показатели запуска Gateway на уровне исходного кода, памяти, нагрузки плагинов, повторяющегося
  цикла приветствий фиктивной модели и запуска CLI.
- Проверка живых моделей в Docker: `pnpm test:docker:live-models`
  - Для каждой выбранной модели выполняется текстовый ход и небольшая проверка в стиле чтения файла.
    Для моделей, в метаданных которых заявлен ввод `image`, также выполняется небольшой ход с изображением.
    Чтобы изолировать сбои провайдера, отключите дополнительные проверки с помощью `OPENCLAW_LIVE_MODEL_FILE_PROBE=0` или
    `OPENCLAW_LIVE_MODEL_IMAGE_PROBE=0`.
  - Покрытие CI: ежедневный процесс `OpenClaw Scheduled Live And E2E Checks` и запускаемый вручную
    `OpenClaw Release Checks` вызывают повторно используемый рабочий процесс живых/e2e-тестов с
    `include_live_suites: true`, который включает задачи матрицы живых моделей Docker,
    разделенные по провайдерам.
  - Для целевых повторных запусков CI запустите `OpenClaw Live And E2E Checks (Reusable)`
    с `include_live_suites: true` и `live_models_only: true`.
  - Добавляйте новые информативные секреты провайдеров в `scripts/ci-hydrate-live-auth.sh`,
    `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml` и вызывающие его
    запланированные/релизные процессы.
- Проверка привязанного чата нативного Codex: `pnpm test:docker:live-codex-bind`
  - Запускает живой сценарий Docker через путь сервера приложения Codex, привязывает
    синтетическое личное сообщение Slack с помощью `/codex bind`, выполняет `/codex fast` и
    `/codex permissions`, а затем проверяет, что обычный ответ и вложенное изображение
    проходят через привязку нативного плагина, а не через ACP.
- Проверка стенда сервера приложения Codex: `pnpm test:docker:live-codex-harness`
  - Выполняет ходы агента Gateway через принадлежащий плагину стенд сервера приложения Codex,
    проверяет `/codex status` и `/codex models`, а также по умолчанию выполняет проверки
    изображения, MCP для Cron, субагента и Guardian. Чтобы изолировать другие сбои, отключите
    проверку субагента с помощью `OPENCLAW_LIVE_CODEX_HARNESS_SUBAGENT_PROBE=0`. Для целевой
    проверки субагента отключите остальные проверки:
    `OPENCLAW_LIVE_CODEX_HARNESS_IMAGE_PROBE=0 OPENCLAW_LIVE_CODEX_HARNESS_MCP_PROBE=0 OPENCLAW_LIVE_CODEX_HARNESS_GUARDIAN_PROBE=0 OPENCLAW_LIVE_CODEX_HARNESS_SUBAGENT_PROBE=1 pnpm test:docker:live-codex-harness`.
    После проверки субагента процесс завершается, если не задано
    `OPENCLAW_LIVE_CODEX_HARNESS_SUBAGENT_ONLY=0`.
- Проверка установки Codex по требованию: `pnpm test:docker:codex-on-demand`
  - Устанавливает упакованный tarball OpenClaw в Docker, выполняет первоначальную настройку
    с API-ключом OpenAI и проверяет, что Plugin Codex и зависимость `@openai/codex`
    были по требованию загружены в корень управляемого проекта npm.
- Живая проверка зависимости инструмента плагина: `pnpm test:docker:live-plugin-tool`
  - Упаковывает тестовый Plugin с реальной зависимостью `slugify`, устанавливает его
    через `npm-pack:`, проверяет зависимость в корне управляемого проекта npm,
    а затем просит живую модель OpenAI вызвать инструмент плагина и
    вернуть скрытый слаг.
- Проверка команды восстановления Crestodian: `pnpm test:live:crestodian-rescue-channel`
  - Необязательная усиленная проверка интерфейса команды восстановления канала сообщений.
    Выполняет `/crestodian status`, ставит в очередь постоянное изменение модели,
    отвечает `/crestodian yes` и проверяет путь записи аудита и конфигурации.
- Docker-проверка первого запуска Crestodian: `pnpm test:docker:crestodian-first-run`
  - Начинает с пустого каталога состояния OpenClaw и сначала подтверждает, что упакованный
    CLI `openclaw crestodian` безопасно завершается с ошибкой без логического вывода. Затем
    тестирует и активирует фиктивный Claude через упакованный модуль активации.
    Только после этого нечеткий запрос упакованного CLI поступает планировщику и
    преобразуется в типизированную настройку, после чего выполняются одноразовые операции
    с моделью, агентом, Plugin Discord и SecretRef. Проверяются записи конфигурации и аудита.
    Это вспомогательное свидетельство работы шлюза и операций, а не подтверждение интерактивной
    первоначальной настройки или агента, инструмента либо подтверждения Crestodian. Этот же сценарий доступен в QA Lab через
    `pnpm openclaw qa suite --scenario crestodian-ring-zero-setup`.
- Проверка стоимости Moonshot/Kimi: задав `MOONSHOT_API_KEY`, выполните
  `openclaw models list --provider moonshot --json`, затем выполните изолированную команду
  `openclaw agent --local --session-id live-kimi-cost --message 'Reply exactly: KIMI_LIVE_OK' --thinking off --json`
  для `moonshot/kimi-k2.6`. Убедитесь, что JSON указывает Moonshot/K2.6, а
  расшифровка ответа ассистента содержит нормализованное значение `usage.cost`.

<Tip>
Если нужен только один падающий случай, сужайте область живых тестов с помощью переменных окружения со списком разрешений, описанных ниже.
</Tip>

## Средства запуска для контроля качества

Эти команды дополняют основные наборы тестов, когда требуется реалистичность QA Lab.

CI запускает QA Lab в выделенных рабочих процессах. Проверка соответствия агентного поведения вложена в
`QA-Lab - All Lanes` и проверку релиза, а не выделена в отдельный рабочий процесс PR.
Для широкой проверки используйте `Full Release Validation` с
`rerun_group=qa-parity` или группу контроля качества проверок релиза. В стабильных/стандартных проверках релиза
исчерпывающие длительные живые/Docker-тесты включаются через `run_release_soak=true`;
профиль `full` включает их принудительно. `QA-Lab - All Lanes` запускается каждую ночь для `main` и
вручную, выполняя параллельно сценарий соответствия с имитацией, живой сценарий Matrix,
управляемый Convex живой сценарий Telegram и управляемый Convex живой сценарий Discord.
Запланированные проверки качества и проверки релиза явно передают Matrix параметр `--profile fast`,
тогда как CLI Matrix и ввод ручного рабочего процесса по умолчанию используют `all`; ручной запуск
может разделить `all` на задачи `transport`, `media`, `e2ee-smoke`, `e2ee-deep` и `e2ee-cli`.
`OpenClaw Release Checks` выполняет проверку соответствия, а также быстрые сценарии Matrix и Telegram
перед одобрением релиза, используя `mock-openai/gpt-5.6-luna` для релизных проверок транспорта,
чтобы они оставались детерминированными и не требовали обычного запуска плагина провайдера.
Эти Gateway живого транспорта отключают поиск в памяти; поведение памяти по-прежнему проверяется
наборами тестов соответствия контроля качества.

Шарды живой обработки мультимедиа полного релиза используют
`ghcr.io/openclaw/openclaw-live-media-runner:ubuntu-24.04`, где уже установлены
`ffmpeg` и `ffprobe`. Шарды живых моделей и серверных компонентов Docker используют общий образ
`ghcr.io/openclaw/openclaw-live-test:<sha>`, который собирается один раз для выбранного
коммита, а затем загружается с `OPENCLAW_SKIP_DOCKER_BUILD=1` вместо повторной сборки
в каждом шарде.

- `pnpm openclaw qa suite`
  - Запускает сценарии QA из репозитория непосредственно на хосте.
  - Создаёт артефакты верхнего уровня `qa-evidence.json`, `qa-suite-summary.json` и
    `qa-suite-report.md` для выбранного набора сценариев, включая выбор
    смешанных потоков, сценариев Vitest и Playwright.
  - При запуске через `pnpm openclaw qa run --qa-profile <profile>` включает
    оценочную таблицу выбранного профиля таксономии в тот же файл `qa-evidence.json`.
    `smoke-ci` создаёт сокращённые свидетельства (`evidenceMode: "slim"`, без
    `execution` для отдельных записей). `release` охватывает отобранный набор
    проверок готовности к выпуску; `all` выбирает все активные категории зрелости
    и предназначен для явных запусков рабочего процесса QA Profile Evidence,
    когда требуется полный артефакт оценочной таблицы.
  - По умолчанию запускает несколько выбранных сценариев параллельно с
    изолированными рабочими процессами Gateway. Для `qa-channel` по умолчанию
    используется параллелизм 4 (с ограничением по числу выбранных сценариев).
    Используйте `--concurrency <count>`, чтобы настроить число рабочих процессов,
    или `--concurrency 1` для прежнего последовательного режима.
  - Завершается с ненулевым кодом, если какой-либо сценарий не пройден.
    Используйте `--allow-failures`, чтобы получить артефакты без кода завершения,
    указывающего на ошибку.
  - Поддерживает режимы провайдера `live-frontier`, `mock-openai` и `aimock`.
    `aimock` запускает локальный сервер провайдера на базе AIMock для
    экспериментального покрытия фикстур и имитаций протокола, не заменяя
    учитывающий особенности сценариев режим `mock-openai`.
- `pnpm openclaw qa coverage --match <query>`
  - Выполняет поиск по идентификаторам и названиям сценариев, поверхностям,
    идентификаторам покрытия, ссылкам на документацию и код, плагинам и
    требованиям к провайдерам, а затем выводит подходящие цели набора тестов.
  - Используйте эту команду перед запуском QA Lab, если знаете затронутое
    поведение или путь к файлу, но не знаете минимальный подходящий сценарий.
    Результат носит рекомендательный характер — всё равно выбирайте имитационное,
    реальное, Multipass-, Matrix- или транспортное подтверждение в соответствии
    с изменяемым поведением.
- `pnpm test:plugins:kitchen-sink-live`
  - Запускает через QA Lab комплекс реальных испытаний плагина OpenAI Kitchen Sink.
    Устанавливает внешний пакет Kitchen Sink, проверяет перечень поверхностей
    SDK плагина, опрашивает `/healthz` и `/readyz`, записывает свидетельства
    использования CPU/RSS Gateway, выполняет реальный запрос к OpenAI и проверяет
    диагностику в неблагоприятных условиях. Требуется действующая аутентификация
    OpenAI, например `OPENAI_API_KEY`. В подготовленных сеансах Testbox профиль
    реальной аутентификации Testbox подключается автоматически, если доступен
    вспомогательный инструмент `openclaw-testbox-env`.
- `pnpm test:gateway:cpu-scenarios`
  - Запускает тест производительности запуска Gateway и небольшой набор
    имитационных сценариев QA Lab (`channel-chat-baseline`,
    `memory-failure-fallback`, `gateway-restart-inflight-run`), а затем записывает
    сводку наблюдений за CPU в `.artifacts/gateway-cpu-scenarios/`.
  - По умолчанию отмечает только длительные периоды высокой загрузки CPU
    (`--cpu-core-warn`, значение по умолчанию `0.9`; `--hot-wall-warn-ms`,
    значение по умолчанию `30000`), поэтому кратковременные всплески при запуске
    регистрируются как метрики, но не выглядят как регрессия, при которой Gateway
    остаётся полностью загруженным в течение нескольких минут.
  - Работает с собранными артефактами `dist`; если в рабочей копии ещё нет
    актуальных результатов сборки среды выполнения, сначала выполните сборку.
- `pnpm openclaw qa suite --runner multipass`
  - Запускает тот же набор QA внутри одноразовой виртуальной машины Multipass
    с Linux, сохраняя те же флаги выбора сценариев, провайдера и модели, что и
    `qa suite`.
  - При реальных запусках передаёт в гостевую систему доступные входные данные
    аутентификации QA: ключи провайдеров из переменных окружения, путь к
    конфигурации реального провайдера QA и `CODEX_HOME`, если он задан.
  - Каталоги вывода должны находиться внутри корня репозитория, чтобы гостевая
    система могла записывать результаты обратно через подключённую рабочую
    область.
  - Создаёт обычные отчёт и сводку QA, а также журналы Multipass в
    `.artifacts/qa-e2e/...`.
- `pnpm qa:lab:up`
  - Запускает работающий на базе Docker сайт QA для выполнения проверок в
    операторском режиме.

- В GitHub Actions также доступен рабочий процесс `Package Acceptance` для
  дополнительной проверки продукта с одним пакетом-кандидатом. Он принимает
  ссылку Git, опубликованную спецификацию npm, URL HTTPS архива вместе с SHA-256,
  политику доверенных URL или артефакт архива из другого запуска
  (`source=ref|npm|url|trusted-url|artifact`), загружает нормализованный
  `openclaw-current.tgz` как `package-under-test`, а затем запускает существующий
  планировщик Docker E2E с профилями режимов `smoke`, `package`, `product`, `full`
  или `custom`. Задайте `telegram_mode=mock-openai` или `live-frontier`, чтобы
  запустить рабочий процесс QA Telegram с тем же артефактом
  `package-under-test`.
  - Проверка продукта на последней бета-версии:

```bash
gh workflow run package-acceptance.yml --ref main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f telegram_mode=mock-openai
```

- Для проверки по точному URL архива требуется дайджест и применяется политика
  безопасности общедоступных URL:

```bash
gh workflow run package-acceptance.yml --ref main \
  -f source=url \
  -f package_url=https://registry.npmjs.org/openclaw/-/openclaw-VERSION.tgz \
  -f package_sha256=<sha256> \
  -f suite_profile=package
```

- Для корпоративных или закрытых зеркал архивов используется явно заданная
  политика доверенного источника:

```bash
gh workflow run package-acceptance.yml --ref main \
  -f source=trusted-url \
  -f trusted_source_id=enterprise-artifactory \
  -f package_url=https://packages.example.internal:8443/artifactory/openclaw/openclaw-VERSION.tgz \
  -f package_sha256=<sha256> \
  -f suite_profile=package
```

`source=trusted-url` читает `.github/package-trusted-sources.json` из доверенной
ссылки рабочего процесса и не принимает учётные данные в URL или передаваемый
через входные данные рабочего процесса обход ограничений частной сети. Если
именованная политика объявляет аутентификацию с токеном-носителем, настройте
фиксированный секрет `OPENCLAW_TRUSTED_PACKAGE_TOKEN`.

- Проверка артефакта загружает артефакт архива из другого запуска Actions:

```bash
gh workflow run package-acceptance.yml --ref main \
  -f source=artifact \
  -f artifact_run_id=<run-id> \
  -f artifact_name=<artifact-name> \
  -f suite_profile=smoke
```

### Общие учетные данные Telegram через Convex (v1)

Когда для QA живого транспорта включен `--credential-source convex` (или
`OPENCLAW_QA_CREDENTIAL_SOURCE=convex`), лаборатория QA получает эксклюзивную
аренду из пула на базе Convex, отправляет Heartbeat для этой аренды во время
работы контура и освобождает ее при завершении. Название раздела появилось до
поддержки Discord, Slack и WhatsApp; контракт аренды является общим для всех
типов.

Эталонная заготовка проекта Convex: `qa/convex-credential-broker/`

Обязательные переменные окружения:

- `OPENCLAW_QA_CONVEX_SITE_URL` (например, `https://your-deployment.convex.site`)
- Один секрет для выбранной роли:
  - `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER` для `maintainer`
  - `OPENCLAW_QA_CONVEX_SECRET_CI` для `ci`
- Выбор роли учетных данных:
  - CLI: `--credential-role maintainer|ci`
  - Значение по умолчанию из окружения: `OPENCLAW_QA_CREDENTIAL_ROLE` (по
    умолчанию `ci` в CI и `maintainer` в остальных случаях)

Необязательные переменные окружения:

- `OPENCLAW_QA_CREDENTIAL_LEASE_TTL_MS` (по умолчанию `1200000`)
- `OPENCLAW_QA_CREDENTIAL_HEARTBEAT_INTERVAL_MS` (по умолчанию `30000`)
- `OPENCLAW_QA_CREDENTIAL_ACQUIRE_TIMEOUT_MS` (по умолчанию `90000`)
- `OPENCLAW_QA_CREDENTIAL_HTTP_TIMEOUT_MS` (по умолчанию `15000`)
- `OPENCLAW_QA_CONVEX_ENDPOINT_PREFIX` (по умолчанию `/qa-credentials/v1`)
- `OPENCLAW_QA_CREDENTIAL_OWNER_ID` (необязательный идентификатор трассировки)
- `OPENCLAW_QA_ALLOW_INSECURE_HTTP=1` разрешает URL-адреса Convex с local loopback
  `http://` только для локальной разработки.

В обычном режиме работы `OPENCLAW_QA_CONVEX_SITE_URL` должен использовать
`https://`.

Административные команды сопровождающих для добавления, удаления и просмотра
пула требуют именно `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER`.

Вспомогательные команды CLI для сопровождающих:

```bash
pnpm openclaw qa credentials doctor
pnpm openclaw qa credentials add --kind telegram --payload-file qa/telegram-credential.json
pnpm openclaw qa credentials list --kind telegram
pnpm openclaw qa credentials remove --credential-id <credential-id>
```

Перед живыми запусками используйте `doctor`, чтобы проверить URL-адрес сайта
Convex, секреты брокера, префикс конечной точки, тайм-аут HTTP и доступность
административного просмотра списка без вывода значений секретов. Используйте
`--json` для машиночитаемого вывода в скриптах и служебных программах CI.

Контракт конечной точки по умолчанию (`OPENCLAW_QA_CONVEX_SITE_URL` +
`/qa-credentials/v1`). Запросы проходят аутентификацию с помощью заголовка
`Authorization: Bearer <role secret>`; в приведенных ниже телах этот заголовок
опущен:

- `POST /acquire`
  - Запрос: `{ kind, ownerId, actorRole, leaseTtlMs, heartbeatIntervalMs }`
  - Успешный ответ: `{ status: "ok", credentialId, leaseToken, payload, leaseTtlMs?, heartbeatIntervalMs? }`
  - Пул исчерпан или возможен повтор: `{ status: "error", code: "POOL_EXHAUSTED" | "NO_CREDENTIAL_AVAILABLE", ... }`
- `POST /payload-chunk`
  - Запрос: `{ kind, ownerId, actorRole, credentialId, leaseToken, index }`
  - Успешный ответ: `{ status: "ok", index, data }`
- `POST /heartbeat`
  - Запрос: `{ kind, ownerId, actorRole, credentialId, leaseToken, leaseTtlMs }`
  - Успешный ответ: `{ status: "ok" }` (или пустой ответ `2xx`)
- `POST /release`
  - Запрос: `{ kind, ownerId, actorRole, credentialId, leaseToken }`
  - Успешный ответ: `{ status: "ok" }` (или пустой ответ `2xx`)
- `POST /admin/add` (только с секретом сопровождающего)
  - Запрос: `{ kind, actorId, payload, note?, status? }`
  - Успешный ответ: `{ status: "ok", credential }`
- `POST /admin/remove` (только с секретом сопровождающего)
  - Запрос: `{ credentialId, actorId }`
  - Успешный ответ: `{ status: "ok", changed, credential }`
  - Защита активной аренды: `{ status: "error", code: "LEASE_ACTIVE", ... }`
- `POST /admin/list` (только с секретом сопровождающего)
  - Запрос: `{ kind?, status?, includePayload?, limit? }`
  - Успешный ответ: `{ status: "ok", credentials, count }`

Формат полезной нагрузки для типа Telegram:

- `{ groupId: string, driverToken: string, sutToken: string }`
- `groupId` должен быть строкой с числовым идентификатором чата Telegram.
- `admin/add` проверяет этот формат для `kind: "telegram"` и отклоняет
  некорректные полезные нагрузки.

Формат полезной нагрузки для типа реального пользователя Telegram:

- `{ groupId: string, sutToken: string, testerUserId: string, testerUsername: string, telegramApiId: string, telegramApiHash: string, tdlibDatabaseEncryptionKey: string, tdlibArchiveBase64: string, tdlibArchiveSha256: string, desktopTdataArchiveBase64: string, desktopTdataArchiveSha256: string }`
- `groupId`, `testerUserId` и `telegramApiId` должны быть числовыми строками.
- `tdlibArchiveSha256` и `desktopTdataArchiveSha256` должны быть
  шестнадцатеричными строками SHA-256.
- `kind: "telegram-user"` зарезервирован для рабочего процесса подтверждения
  Mantis Telegram Desktop. Универсальные контуры QA Lab не должны получать
  его в аренду.

Проверяемые брокером полезные нагрузки для нескольких каналов:

- Discord: `{ guildId: string, channelId: string, driverBotToken: string, sutBotToken: string, sutApplicationId: string, voiceChannelId?: string }`
- WhatsApp: `{ driverPhoneE164: string, sutPhoneE164: string, driverAuthArchiveBase64: string, sutAuthArchiveBase64: string, groupJid?: string }`

Контуры Slack также могут арендовать данные из пула, но проверка полезной
нагрузки Slack сейчас выполняется в средстве запуска QA для Slack, а не в
брокере. Для строк Slack используйте
`{ channelId: string, driverBotToken: string, sutBotToken: string, sutAppToken: string }`.

- Команда: `pnpm test`

  - Внутрипроцессные интеграционные тесты (аутентификация Gateway,

  - Детерминированные регрессионные тесты известных ошибок
- Ожидания:
  - Запускаются в CI
  - Не требуют реальных ключей

### Стабильность (Gateway)

- Команда: `pnpm test:stability:gateway`
- Конфигурация: `test/vitest/vitest.gateway.config.ts`, `test/vitest/vitest.logging.config.ts` и `test/vitest/vitest.infra.config.ts`, каждая принудительно использует один рабочий процесс
- Область:
  - Запускает реальный Gateway на local loopback с включённой по умолчанию диагностикой
  - Создаёт синтетическую нагрузку сообщениями Gateway, памятью и крупными полезными данными через путь диагностических событий
  - Запрашивает `diagnostics.stability` через WS RPC Gateway
  - Охватывает вспомогательные средства сохранения пакета диагностических данных о стабильности
  - Проверяет, что регистратор остаётся ограниченным, синтетические выборки RSS не превышают бюджет нагрузки, а глубина очередей отдельных сеансов снова снижается до нуля
- Ожидания:
  - Безопасно для CI и не требует ключей
  - Узкий контур для последующей проверки регрессий стабильности, а не замена полного набора тестов Gateway

### E2E (дымовое тестирование Gateway)

- Команда: `pnpm test:e2e:gateway`
- Конфигурация: `test/vitest/vitest.e2e.config.ts`
- Файлы: `src/**/*.e2e.test.ts`, `test/**/*.e2e.test.ts` и E2E-тесты встроенных plugins в `extensions/`
- Параметры среды выполнения по умолчанию:
  - Использует Vitest `threads` с `isolate: false`, как и остальная часть репозитория.
  - Использует адаптивное количество рабочих процессов (CI: до 2, локально: по умолчанию 1).
  - По умолчанию работает в тихом режиме для снижения накладных расходов на консольный ввод-вывод.
- Полезные переопределения:
  - `OPENCLAW_E2E_WORKERS=<n>` для принудительного задания числа рабочих процессов (не более 16).
  - `OPENCLAW_E2E_VERBOSE=1` для повторного включения подробного консольного вывода.
- Область:
  - Сквозное поведение нескольких экземпляров Gateway
  - Поверхности WebSocket/HTTP, сопряжение узлов и более ресурсоёмкая сетевая работа
- Ожидания:
  - Выполняется в CI (если включено в конвейере)
  - Реальные ключи не требуются
  - Больше взаимодействующих компонентов, чем в модульных тестах, поэтому выполнение может быть медленнее

### E2E (браузер Control UI с имитациями)

- Команда: `pnpm test:ui:e2e`
- Конфигурация: `test/vitest/vitest.ui-e2e.config.ts`
- Файлы: `ui/src/**/*.e2e.test.ts`
- Область:
  - Запускает Control UI на базе Vite
  - Управляет реальной страницей Chromium через Playwright
  - Заменяет WebSocket Gateway детерминированными имитациями внутри браузера
- Ожидания:
  - Выполняется в CI как часть `pnpm test:e2e`
  - Реальный Gateway, агенты и ключи поставщиков не требуются
  - Должна присутствовать браузерная зависимость (`pnpm --dir ui exec playwright install chromium`)

### E2E: дымовое тестирование серверной части OpenShell

- Команда: `pnpm test:e2e:openshell`
- Файл: `extensions/openshell/src/backend.e2e.test.ts`
- Область:
  - Повторно использует активный локальный Gateway OpenShell
  - Создаёт песочницу из временного локального Dockerfile
  - Проверяет серверную часть OpenShell в OpenClaw через реальные `sandbox ssh-config` и выполнение по SSH
  - Проверяет каноническое поведение удалённой файловой системы через мост файловой системы песочницы
- Ожидания:
  - Только по явному включению; не входит в стандартный запуск `pnpm test:e2e`
  - Требует локального CLI `openshell` и работающего демона Docker
  - Требует активного локального Gateway OpenShell и источника его конфигурации
  - Использует изолированные `HOME` / `XDG_CONFIG_HOME`, а затем уничтожает тестовую песочницу
- Полезные переопределения:
  - `OPENCLAW_E2E_OPENSHELL=1` для включения теста при ручном запуске более широкого набора e2e
  - `OPENCLAW_E2E_OPENSHELL_COMMAND=/path/to/openshell` для указания нестандартного исполняемого файла CLI или сценария-обёртки
  - `OPENCLAW_E2E_OPENSHELL_CONFIG_HOME=/path/to/config` для предоставления зарегистрированной конфигурации Gateway изолированному тесту
  - `OPENCLAW_E2E_OPENSHELL_HOST_IP=172.18.0.1` для переопределения IP-адреса шлюза Docker, используемого фикстурой политики хоста

### Работа с реальными сервисами (реальные поставщики и реальные модели)

- Команда: `pnpm test:live`
- Конфигурация: `test/vitest/vitest.live.config.ts`
- Файлы: `src/**/*.live.test.ts`, `test/**/*.live.test.ts` и live-тесты встроенных плагинов в `extensions/`
- По умолчанию: **включены** командой `pnpm test:live` (задаёт `OPENCLAW_LIVE_TEST=1`)
- Область проверки:
  - «Действительно ли этот провайдер/модель работает _сегодня_ с реальными учётными данными?»
  - Выявление изменений форматов провайдеров, особенностей вызова инструментов, проблем аутентификации и поведения при ограничении частоты запросов
- Ожидания:
  - По замыслу не гарантируют стабильность в CI (реальные сети, реальные политики провайдеров, квоты, сбои)
  - Требуют денежных затрат / расходуют лимиты запросов
  - Предпочтительно запускать узкие подмножества, а не «всё»
- Live-запуски используют уже экспортированные ключи API и подготовленные профили аутентификации.
- По умолчанию live-запуски по-прежнему изолируют `HOME` и копируют конфигурацию и материалы аутентификации во временный домашний каталог тестов, чтобы модульные фикстуры не могли изменить ваш настоящий `~/.openclaw`.
- Задавайте `OPENCLAW_LIVE_USE_REAL_HOME=1` только тогда, когда намеренно хотите, чтобы live-тесты использовали ваш настоящий домашний каталог.
- По умолчанию `pnpm test:live` работает в более тихом режиме: сохраняет вывод хода выполнения `[live] ...` и подавляет журналы начальной загрузки Gateway и сообщения Bonjour. Задайте `OPENCLAW_LIVE_TEST_QUIET=0`, если хотите снова видеть полные журналы запуска.
- Ротация ключей API (зависит от провайдера): задайте `*_API_KEYS` в формате со значениями, разделёнными запятыми или точками с запятой, либо `*_API_KEY_1`, `*_API_KEY_2` (например, `OPENAI_API_KEYS`, `ANTHROPIC_API_KEYS`, `GEMINI_API_KEYS`), либо переопределение для конкретного live-запуска через `OPENCLAW_LIVE_*_KEY`; при ответах об ограничении частоты запросов тесты повторяют попытку.
- Вывод хода выполнения/Heartbeat:
  - Live-наборы выводят строки хода выполнения в stderr, чтобы во время длительных вызовов провайдера было видно, что процесс активен, даже когда перехват консоли Vitest не выводит сообщений.
  - `test/vitest/vitest.live.config.ts` отключает перехват консоли Vitest, поэтому строки хода выполнения провайдера/Gateway немедленно выводятся во время live-запусков.
  - Настраивайте Heartbeat прямых обращений к моделям с помощью `OPENCLAW_LIVE_HEARTBEAT_MS`.
  - Настраивайте Heartbeat Gateway/проверок с помощью `OPENCLAW_LIVE_GATEWAY_HEARTBEAT_MS`.

## Какой набор тестов следует запустить?

Используйте эту таблицу решений:

- При редактировании логики/тестов: запустите `pnpm test` (и `pnpm test:coverage`, если внесли много изменений)
- При изменении сетевого взаимодействия Gateway / протокола WS / сопряжения: дополнительно запустите `pnpm test:e2e`
- При отладке ситуации «мой бот не работает» / сбоев конкретного провайдера / вызова инструментов: запустите узкий набор через `pnpm test:live`

## Средства запуска Docker (необязательные проверки «работает в Linux»)

Эти средства запуска Docker разделены на две категории:

- Средства запуска live-моделей: `test:docker:live-models` и `test:docker:live-gateway` запускают внутри Docker-образа репозитория только соответствующий live-файл с ключами профилей (`src/agents/models.profiles.live.test.ts` и `src/gateway/gateway-models.profiles.live.test.ts`), подключая локальный каталог конфигурации, рабочее пространство и необязательный файл переменных окружения профиля. Соответствующие локальные точки входа — `test:live:models-profiles` и `test:live:gateway-profiles`.
- Средства live-запуска Docker при необходимости сохраняют собственные практические ограничения:
  `test:docker:live-models` по умолчанию использует тщательно отобранный поддерживаемый набор с высокой диагностической ценностью, а
  `test:docker:live-gateway` по умолчанию задаёт `OPENCLAW_LIVE_GATEWAY_SMOKE=1`,
  `OPENCLAW_LIVE_GATEWAY_MAX_MODELS=8`,
  `OPENCLAW_LIVE_GATEWAY_STEP_TIMEOUT_MS=45000` и
  `OPENCLAW_LIVE_GATEWAY_MODEL_TIMEOUT_MS=90000`. Задайте `OPENCLAW_LIVE_MAX_MODELS`
  или переменные окружения Gateway, если вам явно требуется меньшее ограничение или более широкий охват.
- `test:docker:all` один раз собирает live-образ Docker с помощью `test:docker:live-build`, один раз упаковывает OpenClaw в tar-архив npm с помощью `scripts/package-openclaw-for-docker.mjs`, а затем собирает/повторно использует два образа `scripts/e2e/Dockerfile`. Базовый образ содержит только среду выполнения Node/Git для сценариев установки/обновления/зависимостей плагинов; эти сценарии подключают предварительно собранный tar-архив. Функциональный образ устанавливает тот же tar-архив в `/app` для сценариев проверки функциональности собранного приложения. Определения сценариев Docker находятся в `scripts/lib/docker-e2e-scenarios.mjs`; логика планировщика — в `scripts/lib/docker-e2e-plan.mjs`; `scripts/test-docker-all.mjs` выполняет выбранный план. Совокупный запуск использует взвешенный локальный планировщик: `OPENCLAW_DOCKER_ALL_PARALLELISM` управляет слотами процессов, а ограничения ресурсов не позволяют всем ресурсоёмким live-сценариям, сценариям установки npm и многосервисным сценариям запускаться одновременно. Если один сценарий требует больше ресурсов, чем разрешено активными ограничениями, планировщик всё равно может запустить его при пустом пуле, после чего выполняет его в одиночку до повторного появления доступной ёмкости. Значения по умолчанию: 10 слотов, `OPENCLAW_DOCKER_ALL_LIVE_LIMIT=9`, `OPENCLAW_DOCKER_ALL_NPM_LIMIT=5` и `OPENCLAW_DOCKER_ALL_SERVICE_LIMIT=7`; изменяйте `OPENCLAW_DOCKER_ALL_WEIGHT_LIMIT` или `OPENCLAW_DOCKER_ALL_DOCKER_LIMIT` (и другие переопределения `OPENCLAW_DOCKER_ALL_<RESOURCE>_LIMIT`) только при наличии дополнительного запаса ресурсов на хосте Docker. По умолчанию средство запуска выполняет предварительную проверку Docker, удаляет устаревшие E2E-контейнеры OpenClaw, выводит состояние каждые 30 секунд, сохраняет длительность успешно выполненных сценариев в `.artifacts/docker-tests/lane-timings.json` и использует эти данные, чтобы в последующих запусках первыми запускать более длительные сценарии. Используйте `OPENCLAW_DOCKER_ALL_DRY_RUN=1`, чтобы вывести взвешенный манифест сценариев без сборки или запуска Docker, либо `node scripts/test-docker-all.mjs --plan-json`, чтобы вывести план CI для выбранных сценариев, требования к пакетам/образам и учётным данным.
- `Package Acceptance` — это встроенная в GitHub проверка пакета, отвечающая на вопрос: «Работает ли этот устанавливаемый tar-архив как полноценный продукт?» Она получает один пакет-кандидат из `source=npm`, `source=ref`, `source=url`, `source=trusted-url` или `source=artifact`, загружает его как `package-under-test`, а затем запускает повторно используемые сценарии Docker E2E с этим конкретным tar-архивом вместо повторной упаковки выбранной ссылки. Профили упорядочены по широте охвата: `smoke`, `package`, `product` и `full` (а также `custom` для явно заданного списка сценариев). Контракт пакета/обновления/плагина, матрицу сохранения работоспособности после обновления опубликованных версий, параметры выпуска по умолчанию и разбор сбоев см. в разделе [Тестирование обновлений и плагинов](/ru/help/testing-updates-plugins).
- Проверки сборки и выпуска запускают `scripts/check-cli-bootstrap-imports.mjs` после tsdown. Ограничитель обходит статический граф собранного кода от `dist/entry.js` и `dist/cli/run-main.js` и завершает проверку с ошибкой, если этот загрузочный граф до диспетчеризации команд статически импортирует какой-либо внешний пакет (учитываются Commander, интерфейс приглашений, undici, журналирование и аналогичные тяжёлые при запуске зависимости); он также ограничивает размер собранного фрагмента запуска Gateway значением 70 КБ и запрещает статический импорт из этого фрагмента известных редко используемых путей Gateway (`control-ui-assets`, `diagnostic-stability-bundle`, `onboard-helpers`, `process-respawn`, `restart-sentinel`, `server-close`, `server-reload-handlers`). Отдельно `scripts/release-check.ts` выполняет быструю проверку упакованного CLI с помощью `--help`, `onboard --help`, `doctor --help`, `status --json --timeout 1`, `config schema` и `models list --provider openai`.
- Совместимость прежних версий в Package Acceptance ограничена версией `2026.4.25` (включая `2026.4.25-beta.*`). До этой границы стенд допускает только пробелы в метаданных выпущенного пакета: отсутствие записей закрытого реестра QA, отсутствие `gateway install --wrapper`, отсутствие файлов исправлений в созданной из tar-архива Git-фикстуре, отсутствие сохранённого `update.channel`, прежние расположения записей установки плагинов, отсутствие сохранения записей установки из каталога и миграцию метаданных конфигурации во время `plugins update`. Для пакетов после `2026.4.25` эти случаи считаются строгими ошибками.
- Средства быстрой проверки контейнеров: `test:docker:openwebui`, `test:docker:onboard`, `test:docker:npm-onboard-channel-agent`, `test:docker:release-user-journey`, `test:docker:release-typed-onboarding`, `test:docker:release-media-memory`, `test:docker:release-upgrade-user-journey`, `test:docker:release-plugin-marketplace`, `test:docker:skill-install`, `test:docker:update-channel-switch`, `test:docker:upgrade-survivor`, `test:docker:published-upgrade-survivor`, `test:docker:session-runtime-context`, `test:docker:agents-delete-shared-workspace`, `test:docker:gateway-network`, `test:docker:browser-cdp-snapshot`, `test:docker:mcp-channels`, `test:docker:agent-bundle-mcp-tools`, `test:docker:cron-mcp-cleanup`, `test:docker:plugins`, `test:docker:plugin-update`, `test:docker:plugin-lifecycle-matrix` и `test:docker:config-reload` запускают один или несколько реальных контейнеров и проверяют высокоуровневые пути интеграции.
- Сценарии Docker/Bash E2E, устанавливающие упакованный tar-архив OpenClaw с помощью `scripts/lib/openclaw-e2e-instance.sh`, ограничивают время выполнения `npm install` значением `OPENCLAW_E2E_NPM_INSTALL_TIMEOUT` (по умолчанию `600s`; задайте `0`, чтобы отключить обёртку для отладки).

Средства Docker-запуска live-моделей также подключают только необходимые домашние каталоги аутентификации CLI
(или все поддерживаемые, если запуск не ограничен), а затем перед запуском копируют их в домашний каталог
контейнера, чтобы OAuth внешнего CLI мог обновлять токены,
не изменяя хранилище аутентификации на хосте:

- Прямые модели: `pnpm test:docker:live-models` (сценарий: `scripts/test-live-models-docker.sh`)
- Быстрая проверка привязки ACP: `pnpm test:docker:live-acp-bind` (сценарий: `scripts/test-live-acp-bind-docker.sh`; по умолчанию охватывает Claude, Codex и Gemini, а строгий охват Droid/OpenCode обеспечивается через `pnpm test:docker:live-acp-bind:droid` и `pnpm test:docker:live-acp-bind:opencode`)
- Быстрая проверка серверной части CLI: `pnpm test:docker:live-cli-backend` (сценарий: `scripts/test-live-cli-backend-docker.sh`)
- Быстрая проверка стенда сервера приложений Codex: `pnpm test:docker:live-codex-harness` (сценарий: `scripts/test-live-codex-harness-docker.sh`)
- Gateway + агент разработки: `pnpm test:docker:live-gateway` (сценарий: `scripts/test-live-gateway-models-docker.sh`)
- Быстрые проверки наблюдаемости: `pnpm qa:otel:smoke`, `pnpm qa:prometheus:smoke` и `pnpm qa:observability:smoke` — это закрытые QA-сценарии для рабочей копии исходного кода. Они намеренно не входят в Docker-сценарии выпуска пакета, поскольку tar-архив npm не содержит QA Lab.
- Live-проверка Open WebUI: `pnpm test:docker:openwebui` (сценарий: `scripts/e2e/openwebui-docker.sh`)
- Мастер начальной настройки (TTY, полное формирование структуры): `pnpm test:docker:onboard` (сценарий: `scripts/e2e/onboard-docker.sh`)
- Быстрая проверка начальной настройки/канала/агента из tar-архива npm: `pnpm test:docker:npm-onboard-channel-agent` глобально устанавливает упакованный tar-архив OpenClaw в Docker, по умолчанию настраивает OpenAI через начальную настройку со ссылкой на переменную окружения и Telegram, запускает doctor, а затем выполняет один имитированный ход агента OpenAI. Повторно используйте предварительно собранный tar-архив с помощью `OPENCLAW_CURRENT_PACKAGE_TGZ=/path/to/openclaw-*.tgz`, пропустите повторную сборку на хосте с помощью `OPENCLAW_NPM_ONBOARD_HOST_BUILD=0` или смените канал с помощью `OPENCLAW_NPM_ONBOARD_CHANNEL=discord` либо `OPENCLAW_NPM_ONBOARD_CHANNEL=slack`.

- Дымовая проверка пользовательского сценария релиза: `pnpm test:docker:release-user-journey` глобально устанавливает упакованный tar-архив OpenClaw в чистый домашний каталог Docker, выполняет первоначальную настройку, настраивает имитируемого провайдера OpenAI, выполняет ход агента, устанавливает и удаляет внешние плагины, настраивает ClickClack для локальной фикстуры, проверяет исходящие и входящие сообщения, перезапускает Gateway и запускает doctor.
- Дымовая проверка типизированной первоначальной настройки релиза: `pnpm test:docker:release-typed-onboarding` устанавливает упакованный tar-архив, проводит `openclaw onboard` через настоящий TTY, настраивает OpenAI как провайдера со ссылкой на переменную окружения, проверяет, что необработанный ключ не сохраняется, и выполняет имитируемый ход агента.
- Дымовая проверка мультимедиа и памяти релиза: `pnpm test:docker:release-media-memory` устанавливает упакованный tar-архив и проверяет распознавание изображения из вложения PNG, вывод генерации изображений, совместимый с OpenAI, поиск по памяти и сохранение возможности извлечения после перезапуска Gateway.
- Дымовая проверка пользовательского сценария обновления релиза: `pnpm test:docker:release-upgrade-user-journey` по умолчанию устанавливает новейшую опубликованную базовую версию, которая старше проверяемого tar-архива, настраивает состояние провайдера, плагина и ClickClack в опубликованном пакете, обновляет его до проверяемого tar-архива, а затем повторно выполняет основной сценарий агента, плагина и канала. Если более старой опубликованной базовой версии нет, повторно используется проверяемая версия. Базовую версию можно переопределить с помощью `OPENCLAW_RELEASE_UPGRADE_BASELINE_SPEC=openclaw@<version>`.
- Дымовая проверка каталога плагинов релиза: `pnpm test:docker:release-plugin-marketplace` устанавливает плагин из локальной фикстуры каталога, обновляет и удаляет его, а затем проверяет, что CLI плагина исчезает вместе с очищенными метаданными установки.
- Дымовая проверка установки Skills: `pnpm test:docker:skill-install` глобально устанавливает упакованный tar-архив OpenClaw в Docker, отключает в конфигурации установку загруженных архивов, через поиск определяет идентификатор текущего доступного Skills из ClawHub, устанавливает его командой `openclaw skills install` и проверяет установленный Skills вместе с метаданными происхождения и блокировки `.clawhub`.
- Дымовая проверка переключения канала обновлений: `pnpm test:docker:update-channel-switch` глобально устанавливает упакованный tar-архив OpenClaw в Docker, переключается с пакета `stable` на git-версию `dev`, проверяет сохранённый канал и работу плагина после обновления, затем возвращается к пакету `stable` и проверяет состояние обновления.
- Дымовая проверка сохранности при обновлении: `pnpm test:docker:upgrade-survivor` устанавливает упакованный tar-архив OpenClaw поверх загрязнённой фикстуры старого пользователя с агентами, конфигурацией каналов, списками разрешённых плагинов, устаревшим состоянием зависимостей плагинов и существующими файлами рабочих пространств и сеансов. Проверка выполняет обновление пакета и неинтерактивный запуск doctor без действующих ключей провайдера или канала, затем запускает Gateway через local loopback и проверяет сохранность конфигурации и состояния, а также ограничения времени запуска и получения состояния.
- Дымовая проверка сохранности при обновлении опубликованной версии: `pnpm test:docker:published-upgrade-survivor` по умолчанию устанавливает `openclaw@latest`, создаёт реалистичные файлы существующего пользователя, настраивает эту базовую версию с помощью встроенного набора команд, проверяет полученную конфигурацию, обновляет опубликованную установку до проверяемого tar-архива, запускает doctor в неинтерактивном режиме, записывает `.artifacts/upgrade-survivor/summary.json`, затем запускает Gateway через local loopback и проверяет настроенные намерения, сохранность состояния, запуск, `/healthz`, `/readyz` и ограничения времени получения состояния RPC. Одну базовую версию можно переопределить с помощью `OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPEC`; агрегированному планировщику можно указать развернуть точные локальные базовые версии с помощью `OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPECS`, например `openclaw@2026.5.2 openclaw@2026.4.23 openclaw@2026.4.15`, а фикстуры, соответствующие проблемам, — с помощью `OPENCLAW_UPGRADE_SURVIVOR_SCENARIOS`, например `reported-issues`; набор зарегистрированных проблем включает `configured-plugin-installs` для автоматического восстановления установки внешних плагинов OpenClaw. Приёмочное тестирование пакета предоставляет их как `published_upgrade_survivor_baseline`, `published_upgrade_survivor_baselines` и `published_upgrade_survivor_scenarios`, разрешает метатокены базовых версий, такие как `last-stable-4` или `all-since-2026.4.23`, а полная проверка релиза расширяет пакетный этап продолжительного тестирования релиза до `last-stable-4 2026.4.23 2026.5.2 2026.4.15` с добавлением `reported-issues`.
- Дымовая проверка контекста среды выполнения сеанса: `pnpm test:docker:session-runtime-context` проверяет сохранение скрытого контекста среды выполнения в расшифровке сеанса, а также исправление с помощью doctor затронутых дублирующихся ветвей перезаписи запросов.
- Дымовая проверка глобальной установки Bun: `bash scripts/e2e/bun-global-install-smoke.sh` упаковывает текущее дерево, устанавливает его командой `bun install -g` в изолированном домашнем каталоге и проверяет, что `openclaw infer image providers --json` возвращает встроенных провайдеров изображений, а не зависает. Готовый tar-архив можно повторно использовать с помощью `OPENCLAW_BUN_GLOBAL_SMOKE_PACKAGE_TGZ=/path/to/openclaw-*.tgz`, сборку на хосте можно пропустить с помощью `OPENCLAW_BUN_GLOBAL_SMOKE_HOST_BUILD=0`, а `dist/` можно скопировать из собранного образа Docker с помощью `OPENCLAW_BUN_GLOBAL_SMOKE_DIST_IMAGE=openclaw-dockerfile-smoke:local`.
- Дымовая проверка установщика в Docker: `bash scripts/test-install-sh-docker.sh` использует один кеш npm совместно для контейнеров root, обновления и прямой установки через npm. Для дымовой проверки обновления в качестве стабильной базовой версии по умолчанию используется npm-пакет `latest`, после чего выполняется обновление до проверяемого tar-архива. Локально это можно переопределить с помощью `OPENCLAW_INSTALL_SMOKE_UPDATE_BASELINE=2026.4.22`, а на GitHub — с помощью входного параметра `update_baseline_version` рабочего процесса Install Smoke. Проверки установщика без прав root используют изолированный кеш npm, чтобы принадлежащие root записи кеша не скрывали поведение установки в локальный каталог пользователя. Чтобы повторно использовать кеш root, обновления и прямой установки через npm при локальных повторных запусках, задайте `OPENCLAW_INSTALL_SMOKE_NPM_CACHE_DIR=/path/to/cache`.
- В CI проверка Install Smoke пропускает дублирующее глобальное обновление напрямую через npm с помощью `OPENCLAW_INSTALL_SMOKE_SKIP_NPM_GLOBAL=1`; если требуется покрытие прямой команды `npm install -g`, запускайте скрипт локально без этой переменной окружения.
- Дымовая проверка CLI удаления агентов с общим рабочим пространством: `pnpm test:docker:agents-delete-shared-workspace` (скрипт: `scripts/e2e/agents-delete-shared-workspace-docker.sh`) по умолчанию собирает образ из корневого Dockerfile, создаёт двух агентов с одним рабочим пространством в изолированном домашнем каталоге контейнера, выполняет `agents delete --json` и проверяет корректность JSON, а также сохранение рабочего пространства. Образ проверки установки можно повторно использовать с помощью `OPENCLAW_AGENTS_DELETE_SHARED_WORKSPACE_E2E_IMAGE=openclaw-dockerfile-smoke:local OPENCLAW_AGENTS_DELETE_SHARED_WORKSPACE_E2E_SKIP_BUILD=1`.
- Сетевое взаимодействие Gateway и жизненный цикл хоста: `pnpm test:docker:gateway-network` (скрипт: `scripts/e2e/gateway-network-docker.sh`) сохраняет двухконтейнерную дымовую проверку аутентификации и работоспособности WebSocket в локальной сети, а затем с помощью административного HTTP-интерфейса через local loopback подтверждает блокировку на этапе подготовки, сохранение доступа к управлению, восстановление после возобновления и остановку с последующим запуском подготовленного контейнера без его замены. Проверка перезапуска должна завершиться до истечения исходного срока аренды; она подтверждает, что состояние приостановки является локальным для процесса, а сохранённая конфигурация Gateway и идентификатор контейнера не меняются, и выводит машиночитаемый JSON с длительностью этапов.
- Дымовая проверка снимков CDP браузера: `pnpm test:docker:browser-cdp-snapshot` (скрипт: `scripts/e2e/browser-cdp-snapshot-docker.sh`) собирает исходный образ E2E и слой Chromium, запускает Chromium с необработанным CDP, выполняет `browser doctor --deep` и проверяет, что снимки ролей CDP охватывают URL-адреса ссылок, интерактивные элементы, обнаруженные по форме курсора, ссылки на iframe и метаданные фреймов.
- Регрессионная проверка минимального уровня рассуждений для `web_search` в OpenAI Responses: `pnpm test:docker:openai-web-search-minimal` (скрипт: `scripts/e2e/openai-web-search-minimal-docker.sh`) запускает имитируемый сервер OpenAI через Gateway, проверяет, что `web_search` повышает `reasoning.effort` с `minimal` до `low`, затем принудительно вызывает отклонение схемы провайдером и проверяет, что необработанные подробности присутствуют в журналах Gateway.
- Мост каналов MCP (предварительно заполненный Gateway + мост stdio + дымовая проверка необработанного кадра уведомления Claude): `pnpm test:docker:mcp-channels` (скрипт: `scripts/e2e/mcp-channels-docker.sh`)
- Инструменты MCP в комплекте OpenClaw (настоящий сервер MCP через stdio + дымовая проверка разрешений и запретов встроенного профиля OpenClaw): `pnpm test:docker:agent-bundle-mcp-tools` (скрипт: `scripts/e2e/agent-bundle-mcp-tools-docker.sh`)
- Очистка MCP для Cron и субагентов (настоящий Gateway + завершение дочернего процесса MCP через stdio после изолированных запусков Cron и однократных запусков субагентов): `pnpm test:docker:cron-mcp-cleanup` (скрипт: `scripts/e2e/cron-mcp-cleanup-docker.sh`)
- Плагины (дымовое тестирование установки и обновления для локального пути, `file:`, реестра npm с вынесенными зависимостями, некорректных метаданных пакета npm, перемещающихся ссылок git, комплексной фикстуры ClawHub, обновлений каталога и включения или проверки комплекта Claude): `pnpm test:docker:plugins` (скрипт: `scripts/e2e/plugins-docker.sh`)
  Задайте `OPENCLAW_PLUGINS_E2E_CLAWHUB=0`, чтобы пропустить блок ClawHub, либо переопределите пару пакета и среды выполнения комплексной фикстуры по умолчанию с помощью `OPENCLAW_PLUGINS_E2E_CLAWHUB_SPEC` и `OPENCLAW_PLUGINS_E2E_CLAWHUB_ID`. Если `OPENCLAW_CLAWHUB_URL` или `CLAWHUB_URL` не заданы, тест использует герметичный локальный сервер фикстур ClawHub.
- Дымовая проверка обновления плагина без изменений: `pnpm test:docker:plugin-update` (скрипт: `scripts/e2e/plugin-update-unchanged-docker.sh`)
- Дымовая проверка матрицы жизненного цикла плагина: `pnpm test:docker:plugin-lifecycle-matrix` устанавливает упакованный tar-архив OpenClaw в пустой контейнер, устанавливает плагин npm, переключает его состояния включения и отключения, обновляет и откатывает его через локальный реестр npm, удаляет установленный код, а затем проверяет, что удаление плагина по-прежнему очищает устаревшее состояние, одновременно записывая показатели RSS и ЦП для каждого этапа жизненного цикла.
- Дымовая проверка метаданных перезагрузки конфигурации: `pnpm test:docker:config-reload` (скрипт: `scripts/e2e/config-reload-source-docker.sh`)
- Плагины: `pnpm test:docker:plugins` охватывает дымовое тестирование установки и обновления для локального пути, `file:`, реестра npm с вынесенными зависимостями, перемещающихся ссылок git, фикстур ClawHub, обновлений каталога и включения или проверки комплекта Claude. `pnpm test:docker:plugin-update` проверяет поведение обновления установленных плагинов при отсутствии изменений. `pnpm test:docker:plugin-lifecycle-matrix` охватывает установку, включение, отключение, обновление, откат и удаление npm-плагина при отсутствии кода с отслеживанием ресурсов.

Чтобы вручную предварительно собрать и повторно использовать общий функциональный образ:

```bash
OPENCLAW_DOCKER_E2E_IMAGE=openclaw-docker-e2e-functional:local pnpm test:docker:e2e-build
OPENCLAW_DOCKER_E2E_IMAGE=openclaw-docker-e2e-functional:local OPENCLAW_SKIP_DOCKER_BUILD=1 pnpm test:docker:mcp-channels
```

Переопределения образов для отдельных наборов, например `OPENCLAW_GATEWAY_NETWORK_E2E_IMAGE`, по-прежнему имеют приоритет, если заданы. Если `OPENCLAW_SKIP_DOCKER_BUILD=1` указывает на удалённый общий образ, скрипты загружают его, если он ещё не доступен локально. QR-тесты и тесты установщика в Docker сохраняют собственные Dockerfile, поскольку они проверяют поведение пакета и установки, а не среду выполнения общего собранного приложения.

Средства запуска Docker с действующими моделями также подключают текущую рабочую копию в режиме только для чтения
и помещают её во временный рабочий каталог внутри контейнера. Это позволяет сохранить
компактность образа среды выполнения и при этом запускать Vitest с точными локальными
исходным кодом и конфигурацией. На этапе подготовки пропускаются крупные локальные кеши и результаты
сборки приложений, такие как `.pnpm-store`, `.worktrees`, `__openclaw_vitest__`, а также
локальные для приложения каталоги вывода `.build` или Gradle, чтобы запуски Docker с действующими моделями не
тратили минуты на копирование артефактов, специфичных для конкретной машины. Они также задают
`OPENCLAW_SKIP_CHANNELS=1`, чтобы проверки Gateway с действующими моделями не запускали реальные
обработчики каналов Telegram, Discord и других сервисов внутри контейнера.
`test:docker:live-models` по-прежнему запускает `pnpm test:live`, поэтому при необходимости
сузить или исключить проверку Gateway с действующими моделями из этого этапа Docker также передавайте
`OPENCLAW_LIVE_GATEWAY_*`.

`test:docker:openwebui` — это высокоуровневая проверка совместимости: она запускает контейнер Gateway OpenClaw с включёнными HTTP-эндпоинтами, совместимыми с OpenAI, запускает закреплённую версию контейнера Open WebUI, подключённого к этому Gateway, выполняет вход через Open WebUI, проверяет, что `/api/models` предоставляет `openclaw/default`, а затем отправляет реальный запрос чата через прокси `/api/chat/completions` Open WebUI. Задайте `OPENWEBUI_SMOKE_MODE=models` для проверок CI по пути выпуска, которые должны завершаться после входа в Open WebUI и обнаружения модели, не ожидая завершения запроса к рабочей модели. Первый запуск может быть заметно медленнее, поскольку Docker может потребоваться загрузить образ Open WebUI, а Open WebUI — завершить собственную первоначальную настройку. Для этого набора проверок требуется пригодный к использованию ключ рабочей модели, предоставленный через окружение процесса, подготовленные профили аутентификации или явно заданный `OPENCLAW_PROFILE_FILE`. При успешном выполнении выводится небольшой JSON-объект вида `{ "ok": true, "model": "openclaw/default", ... }`.

`test:docker:mcp-channels` намеренно детерминирован и не требует реальной учётной записи Telegram, Discord или iMessage. Он запускает предварительно заполненный контейнер Gateway, затем второй контейнер, который запускает `openclaw mcp serve`, после чего проверяет обнаружение маршрутизируемых диалогов, чтение расшифровок, метаданные вложений, поведение очереди событий в реальном времени, маршрутизацию исходящей отправки и уведомления о каналах и разрешениях в стиле Claude через реальный stdio-мост MCP. Проверка уведомлений напрямую анализирует необработанные stdio-кадры MCP, поэтому эта проверка подтверждает фактические данные, выдаваемые мостом, а не только те, которые конкретный клиентский SDK случайно предоставляет наружу.

`test:docker:agent-bundle-mcp-tools` детерминирован и не требует ключа рабочей модели. Он собирает Docker-образ репозитория, запускает внутри контейнера реальный пробный stdio-сервер MCP, материализует этот сервер через встроенную среду выполнения MCP комплекта OpenClaw, выполняет инструмент, а затем проверяет, что `coding` и `messaging` сохраняют инструменты `bundle-mcp`, тогда как `minimal` и `tools.deny: ["bundle-mcp"]` отфильтровывают их.

`test:docker:cron-mcp-cleanup` детерминирован и не требует ключа рабочей модели. Он запускает предварительно заполненный Gateway с реальным пробным stdio-сервером MCP, выполняет изолированный проход Cron и однократный дочерний проход `sessions_spawn`, а затем проверяет, что дочерний процесс MCP завершается после каждого запуска.

Ручная проверка ветки ACP с вводом на естественном языке (не для CI):

- `bun scripts/dev/discord-acp-plain-language-smoke.ts --channel <discord-channel-id> ...`
- Сохраните этот скрипт для рабочих процессов регрессионной проверки и отладки. Он может снова потребоваться для проверки маршрутизации веток ACP, поэтому не удаляйте его.

Полезные переменные окружения:

- `OPENCLAW_CONFIG_DIR=...` (по умолчанию: `~/.openclaw`) монтируется в `/home/node/.openclaw`
- `OPENCLAW_WORKSPACE_DIR=...` (по умолчанию: `~/.openclaw/workspace`) монтируется в `/home/node/.openclaw/workspace`
- `OPENCLAW_PROFILE_FILE=...` монтируется и загружается перед запуском тестов
- `OPENCLAW_DOCKER_PROFILE_ENV_ONLY=1` — проверять только переменные окружения, загруженные из `OPENCLAW_PROFILE_FILE`, используя временные каталоги конфигурации и рабочего пространства без внешних подключений аутентификации CLI
- `OPENCLAW_DOCKER_CLI_TOOLS_DIR=...` (по умолчанию: `~/.cache/openclaw/docker-cli-tools`, если запуск ещё не использует каталог подключения, управляемый CI) монтируется в `/home/node/.npm-global` для кэшированных установок CLI внутри Docker
- Внешние каталоги и файлы аутентификации CLI в `$HOME` монтируются в режиме только для чтения в `/host-auth...`, а затем копируются в `/home/node/...` до запуска тестов
  - Каталоги по умолчанию (используются, когда запуск не ограничен конкретными провайдерами): `.factory`, `.gemini`, `.minimax`
  - Файлы по умолчанию: `~/.codex/auth.json`, `~/.codex/config.toml`, `.claude.json`, `~/.claude/.credentials.json`, `~/.claude/settings.json`, `~/.claude/settings.local.json`
  - При запусках, ограниченных провайдерами, монтируются только необходимые каталоги и файлы, определённые по `OPENCLAW_LIVE_PROVIDERS` / `OPENCLAW_LIVE_GATEWAY_PROVIDERS`
  - Переопределите вручную с помощью `OPENCLAW_DOCKER_AUTH_DIRS=all`, `OPENCLAW_DOCKER_AUTH_DIRS=none` или списка через запятую, например `OPENCLAW_DOCKER_AUTH_DIRS=.claude,.codex`
- `OPENCLAW_LIVE_GATEWAY_MODELS=...` / `OPENCLAW_LIVE_MODELS=...` — ограничить запуск
- `OPENCLAW_LIVE_GATEWAY_PROVIDERS=...` / `OPENCLAW_LIVE_PROVIDERS=...` — фильтровать провайдеров внутри контейнера
- `OPENCLAW_SKIP_DOCKER_BUILD=1` — повторно использовать существующий образ `openclaw:local-live` для повторных запусков, не требующих пересборки
- `OPENCLAW_LIVE_REQUIRE_PROFILE_KEYS=1` — обеспечить получение учётных данных из хранилища профилей, а не из окружения
- `OPENCLAW_OPENWEBUI_MODEL=...` — выбрать модель, предоставляемую Gateway для проверки Open WebUI
- `OPENCLAW_OPENWEBUI_PROMPT=...` — переопределить запрос проверки уникального значения, используемый проверкой Open WebUI
- `OPENWEBUI_IMAGE=...` — переопределить тег закреплённого образа Open WebUI

## Проверка документации

После редактирования документации запустите проверки документации: `pnpm check:docs`.
Когда также требуется проверить заголовки внутри страниц, запустите полную проверку якорей Mintlify: `pnpm docs:check-links:anchors`.

## Автономная регрессионная проверка (безопасная для CI)

Это регрессионные проверки «реального конвейера» без реальных провайдеров:

- Вызов инструментов Gateway (имитация OpenAI, реальные Gateway и цикл агента): `src/gateway/gateway.test.ts` (сценарий: "runs a mock OpenAI tool call end-to-end via gateway agent loop")
- Мастер Gateway (WS `wizard.start`/`wizard.next`, записывает конфигурацию и обеспечивает аутентификацию): `src/gateway/gateway.test.ts` (сценарий: "runs wizard over ws and writes auth token config")

## Оценки надёжности агента (Skills)

У нас уже есть несколько безопасных для CI тестов, работающих как «оценки надёжности агента»:

- Имитация вызова инструментов через реальные Gateway и цикл агента (`src/gateway/gateway.test.ts`).
- Сквозные рабочие процессы мастера, проверяющие подключение сеанса и влияние конфигурации (`src/gateway/gateway.test.ts`).

Чего пока не хватает для Skills (см. [Skills](/ru/tools/skills)):

- **Принятие решений:** когда Skills перечислены в запросе, выбирает ли агент подходящий Skill или избегает нерелевантных?
- **Соблюдение требований:** читает ли агент `SKILL.md` перед использованием и выполняет ли обязательные шаги и аргументы?
- **Контракты рабочих процессов:** многоходовые сценарии, проверяющие порядок инструментов, перенос истории сеанса и границы песочницы.

Будущие оценки прежде всего должны оставаться детерминированными:

- Средство выполнения сценариев с имитациями провайдеров для проверки вызовов инструментов и их порядка, чтения файлов Skills и подключения сеансов.
- Небольшой набор сценариев, ориентированных на Skills: использование или отказ от использования, условия допуска, внедрение инструкций.
- Необязательные оценки в реальной среде — только с явным включением и ограничением переменными окружения, после появления безопасного для CI набора.

## Контрактные тесты (структура Plugin и канала)

Контрактные тесты проверяют, что каждый зарегистрированный Plugin и канал соответствует своему контракту интерфейса. Они перебирают все обнаруженные плагины и запускают набор проверок структуры и поведения. Стандартный набор модульных тестов `pnpm test` намеренно пропускает эти общие файлы проверки стыков и базовые проверки; запускайте контрактные команды явно при изменении общих поверхностей каналов или провайдеров.

### Команды

- Все контракты: `pnpm test:contracts`
- Только контракты каналов: `pnpm test:contracts:channels`
- Только контракты провайдеров: `pnpm test:contracts:plugins`

### Контракты каналов

Расположены в `src/channels/plugins/contracts/*.contract.test.ts`. Текущие категории верхнего уровня:

- **channel-catalog** — метаданные записей каталога встроенных каналов и каналов из реестра
- **plugin** (на основе реестра, сегментированный) — базовая структура регистрации Plugin
- **surfaces-only** (на основе реестра, сегментированный) — проверки структуры каждой поверхности для `actions`, `setup`, `status`, `outbound`, `messaging`, `threading`, `directory` и `gateway`
- **session-binding** (на основе реестра) — поведение привязки сеанса
- **outbound-payload** — структура и нормализация полезной нагрузки сообщения
- **group-policy** (резервный вариант) — применение политики групп по умолчанию для каждого канала
- **threading** (на основе реестра, сегментированный) — обработка идентификаторов веток
- **directory** (на основе реестра, сегментированный) — API каталога и списка участников
- **registry** и **plugins-core.\*** — внутреннее устройство реестра плагинов каналов, загрузчика и авторизации записи конфигурации

Вспомогательные средства перехвата входящей диспетчеризации и проверки исходящей полезной нагрузки, используемые этими наборами, доступны для внутреннего использования через `src/plugin-sdk/channel-contract-testing.ts` (исключён из npm и не является общедоступным подпутём SDK); отдельного файла `inbound.contract.test.ts` в этом каталоге нет.

### Контракты провайдеров

Расположены в `src/plugins/contracts/*.contract.test.ts`. Текущие категории включают:

- **shape** — структура манифеста Plugin, API и экспортов среды выполнения
- **plugin-registration** (+ параллельные) — сценарии регистрации манифеста
- **package-manifest** — требования к манифесту пакета
- **loader** — поведение настройки и завершения работы загрузчика плагинов
- **registry** — содержимое и поиск в реестре контрактов плагинов
- **providers** — общее поведение встроенных провайдеров, включая провайдеров веб-поиска
- **auth-choice** — метаданные выбора аутентификации и поведение настройки
- **provider-catalog-deprecation** — метаданные устаревшего каталога провайдеров
- **wizard.choice-resolution**, **wizard.model-picker**, **wizard.setup-options** — контракты мастера настройки провайдера
- **embedding-provider**, **memory-embedding-provider**, **web-fetch-provider**, **tts** — контракты провайдеров для отдельных возможностей
- **session-actions**, **session-attachments**, **session-entry-projection** — принадлежащие Plugin контракты состояния сеанса
- **scheduled-turns** — метаданные запланированных проходов Plugin и границы временных меток
- **host-hooks**, **run-context-lifecycle**, **runtime-import-side-effects**, **runtime-seams** — контракты жизненного цикла хоста и среды выполнения Plugin, а также границ импорта
- **extension-runtime-dependencies** — размещение зависимостей среды выполнения для расширений

### Когда запускать

- После изменения экспортов или подпутей `plugin-sdk`
- После добавления или изменения Plugin канала или провайдера
- После рефакторинга регистрации или обнаружения плагинов

Контрактные тесты выполняются в CI и не требуют реальных ключей API.

## Добавление регрессионных тестов (рекомендации)

При исправлении проблемы провайдера или модели, обнаруженной в реальной среде:

- По возможности добавьте безопасный для CI регрессионный тест: имитацию или заглушку провайдера либо фиксацию точного преобразования структуры запроса
- Если проверка по своей природе возможна только в реальной среде — например, для ограничений частоты или политик аутентификации, — сделайте её узкой и включаемой явно через переменные окружения
- Предпочитайте проверять наименьший уровень, способный обнаружить ошибку:
  - ошибка преобразования или повторного воспроизведения запроса провайдера -> прямой тест моделей
  - ошибка конвейера сеанса, истории или инструментов Gateway -> базовая проверка Gateway в реальной среде или безопасный для CI тест Gateway с имитацией
- Защита обхода SecretRef:
  - `src/secrets/exec-secret-ref-id-parity.test.ts` получает по одной выбранной цели для каждого класса SecretRef из метаданных реестра (`listSecretTargetRegistryEntries()`), а затем проверяет отклонение идентификаторов выполнения с сегментами обхода.
  - Если вы добавляете новое семейство целей SecretRef с `includeInPlan` в `src/secrets/target-registry-data.ts`, обновите `classifyTargetClass` в этом тесте. Тест намеренно завершается с ошибкой для неклассифицированных идентификаторов целей, чтобы новые классы нельзя было незаметно пропустить.
