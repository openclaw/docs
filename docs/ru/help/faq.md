## Первые 60 секунд, если что-то сломалось

1. **Быстрый статус (первая проверка)**

   ```bash
   openclaw status
   ```

   Быстрая локальная сводка: ОС + обновление, доступность gateway/службы, агенты/сессии, конфигурация провайдера + проблемы времени выполнения (когда Gateway доступен).

2. **Отчет для вставки (безопасно делиться)**

   ```bash
   openclaw status --all
   ```

   Диагностика только для чтения с хвостом журнала (токены скрыты).

3. **Состояние демона + порта**

   ```bash
   openclaw gateway status
   ```

   Показывает время выполнения супервизора и доступность RPC, целевой URL проверки и какую конфигурацию, вероятно, использовала служба.

4. **Глубокие проверки**

   ```bash
   openclaw status --deep
   ```

   Запускает живую проверку работоспособности Gateway, включая проверки каналов, когда они поддерживаются
   (требуется доступный Gateway). См. [Работоспособность](/ru/gateway/health).

5. **Отслеживание последнего журнала**

   ```bash
   openclaw logs --follow
   ```

   Если RPC недоступен, используйте запасной вариант:

   ```bash
   tail -f "$(ls -t /tmp/openclaw/openclaw-*.log | head -1)"
   ```

   Файловые журналы отделены от журналов службы; см. [Журналирование](/ru/logging) и [Устранение неполадок](/ru/gateway/troubleshooting).

6. **Запуск doctor (исправления)**

   ```bash
   openclaw doctor
   ```

   Исправляет/мигрирует конфигурацию/состояние + запускает проверки работоспособности. См. [Doctor](/ru/gateway/doctor).

7. **Снимок Gateway**

   ```bash
   openclaw health --json
   openclaw health --verbose   # shows the target URL + config path on errors
   ```

   Запрашивает у работающего Gateway полный снимок (только WS). См. [Работоспособность](/ru/gateway/health).

## Что такое OpenClaw?

<AccordionGroup>
  <Accordion title="Что такое OpenClaw в одном абзаце?">
    OpenClaw — это персональный ИИ-ассистент, который вы запускаете на собственных устройствах. Он отвечает в уже используемых вами мессенджерах и поверхностях обмена сообщениями (WhatsApp, Telegram, Slack, Mattermost, Discord, Google Chat, Signal, iMessage, WebChat и bundled channel plugins, например QQ Bot), а также может поддерживать голос + живой Canvas на поддерживаемых платформах. **Gateway** — это постоянно работающая плоскость управления; ассистент — это продукт.
  </Accordion>

  <Accordion title="Ценностное предложение">
    OpenClaw — это не «просто оболочка для Claude». Это **локальная в первую очередь плоскость управления**, которая позволяет запускать
    способного ассистента на **вашем собственном оборудовании**, доступного из уже используемых вами чат-приложений, с
    сессиями с состоянием, памятью и инструментами — без передачи контроля над вашими рабочими процессами размещенному
    SaaS.

    Главное:

    - **Ваши устройства, ваши данные:** запускайте Gateway где хотите (Mac, Linux, VPS) и храните
      рабочее пространство + историю сессий локально.
    - **Настоящие каналы, а не веб-песочница:** WhatsApp/Telegram/Slack/Discord/Signal/iMessage/и т. д.,
      плюс мобильный голос и Canvas на поддерживаемых платформах.
    - **Не зависит от модели:** используйте Anthropic, OpenAI, MiniMax, OpenRouter и т. д. с маршрутизацией
      и отработкой отказа для каждого агента.
    - **Вариант только локально:** запускайте локальные модели, чтобы **все данные могли оставаться на вашем устройстве**, если хотите.
    - **Маршрутизация нескольких агентов:** отдельные агенты для каждого канала, учетной записи или задачи, каждый со своим
      рабочим пространством и значениями по умолчанию.
    - **Открытый исходный код и возможность доработки:** проверяйте, расширяйте и размещайте самостоятельно без привязки к поставщику.

    Документация: [Gateway](/ru/gateway), [Каналы](/ru/channels), [Несколько агентов](/ru/concepts/multi-agent),
    [Память](/ru/concepts/memory).

## Skills и автоматизация

<AccordionGroup>
  <Accordion title="Как настроить Skills, не оставляя репозиторий грязным?">
    Используйте управляемые переопределения вместо редактирования копии в репозитории. Поместите изменения в `~/.openclaw/skills/<name>/SKILL.md` (или добавьте папку через `skills.load.extraDirs` в `~/.openclaw/openclaw.json`). Приоритет: `<workspace>/skills` → `<workspace>/.agents/skills` → `~/.agents/skills` → `~/.openclaw/skills` → bundled → `skills.load.extraDirs`, поэтому управляемые переопределения все равно имеют преимущество над bundled skills без изменений в git. Если skill нужно установить глобально, но сделать видимым только некоторым агентам, храните общую копию в `~/.openclaw/skills` и управляйте видимостью через `agents.defaults.skills` и `agents.list[].skills`. Только изменения, достойные отправки upstream, должны находиться в репозитории и выходить как PR.
  </Accordion>

  <Accordion title="Можно ли загрузить Skills из пользовательской папки?">
    Да. Добавьте дополнительные каталоги через `skills.load.extraDirs` в `~/.openclaw/openclaw.json` (самый низкий приоритет). Приоритет по умолчанию: `<workspace>/skills` → `<workspace>/.agents/skills` → `~/.agents/skills` → `~/.openclaw/skills` → bundled → `skills.load.extraDirs`. `clawhub` по умолчанию устанавливает в `./skills`, что OpenClaw в следующей сессии рассматривает как `<workspace>/skills`. Если skill должна быть видима только определенным агентам, сочетайте это с `agents.defaults.skills` или `agents.list[].skills`.
  </Accordion>

  <Accordion title="Как использовать разные модели или настройки для разных задач?">
    Сегодня поддерживаются такие шаблоны:

    - **Cron-задания**: изолированные задания могут задавать переопределение `model` для каждого задания.
    - **Агенты**: маршрутизируйте задачи к отдельным агентам с разными моделями по умолчанию, уровнями мышления и параметрами потока.
    - **Переключение по требованию**: используйте `/model`, чтобы в любой момент переключить модель текущей сессии.

    Например, используйте одну и ту же модель с разными настройками для каждого агента:

    ```json5
    {
      agents: {
        list: [
          {
            id: "coder",
            model: "xiaomi/mimo-v2.5-pro",
            thinkingDefault: "high",
            params: { temperature: 0.1 },
          },
          {
            id: "chat",
            model: "xiaomi/mimo-v2.5-pro",
            thinkingDefault: "off",
            params: { temperature: 0.8 },
          },
        ],
      },
    }
    ```

    Поместите общие значения по умолчанию для каждой модели в `agents.defaults.models["provider/model"].params`, затем задайте переопределения для конкретного агента в плоском `agents.list[].params`. Не определяйте отдельные вложенные записи `agents.list[].models["provider/model"].params` для той же модели; `agents.list[].models` предназначен для каталога моделей конкретного агента и переопределений времени выполнения.

    См. [Cron-задания](/ru/automation/cron-jobs), [Маршрутизация нескольких агентов](/ru/concepts/multi-agent), [Конфигурация](/ru/gateway/config-agents) и [Slash-команды](/ru/tools/slash-commands).

  </Accordion>

  <Accordion title="Бот зависает при выполнении тяжелой работы. Как это вынести отдельно?">
    Используйте **субагентов** для долгих или параллельных задач. Субагенты работают в собственной сессии,
    возвращают сводку и сохраняют отзывчивость основного чата.

    Попросите бота «создать субагента для этой задачи» или используйте `/subagents`.
    Используйте `/status` в чате, чтобы увидеть, что Gateway делает прямо сейчас (и занят ли он).

    Совет по токенам: долгие задачи и субагенты оба расходуют токены. Если стоимость важна, задайте
    более дешевую модель для субагентов через `agents.defaults.subagents.model`.

    Документация: [Субагенты](/ru/tools/subagents), [Фоновые задачи](/ru/automation/tasks).

  </Accordion>

  <Accordion title="Как работают привязанные к треду сессии субагентов в Discord?">
    Используйте привязки тредов. Вы можете привязать тред Discord к субагенту или целевой сессии, чтобы последующие сообщения в этом треде оставались в этой привязанной сессии.

    Базовый поток:

    - Создайте через `sessions_spawn` с `thread: true` (и при необходимости `mode: "session"` для постоянных последующих действий).
    - Или привяжите вручную с помощью `/focus <target>`.
    - Используйте `/agents`, чтобы проверить состояние привязки.
    - Используйте `/session idle <duration|off>` и `/session max-age <duration|off>`, чтобы управлять автоматическим снятием фокуса.
    - Используйте `/unfocus`, чтобы отвязать тред.

    Обязательная конфигурация:

    - Глобальные значения по умолчанию: `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
    - Переопределения Discord: `channels.discord.threadBindings.enabled`, `channels.discord.threadBindings.idleHours`, `channels.discord.threadBindings.maxAgeHours`.
    - Автопривязка при создании: `channels.discord.threadBindings.spawnSessions` по умолчанию равно `true`; задайте `false`, чтобы отключить создание сессий, привязанных к треду.

    Документация: [Субагенты](/ru/tools/subagents), [Discord](/ru/channels/discord), [Справочник конфигурации](/ru/gateway/configuration-reference), [Slash-команды](/ru/tools/slash-commands).

  </Accordion>

  <Accordion title="Субагент завершился, но обновление о завершении ушло не туда или вообще не было опубликовано. Что проверить?">
    Сначала проверьте разрешенный маршрут запрашивающей стороны:

    - Доставка субагента в режиме завершения предпочитает любой привязанный тред или маршрут разговора, когда он существует.
    - Если источник завершения содержит только канал, OpenClaw возвращается к сохраненному маршруту сессии запрашивающей стороны (`lastChannel` / `lastTo` / `lastAccountId`), чтобы прямая доставка все еще могла успешно выполниться.
    - Если нет ни привязанного маршрута, ни пригодного сохраненного маршрута, прямая доставка может завершиться неудачей, и результат вместо немедленной публикации в чат вернется к доставке через очередь сессии.
    - Недействительные или устаревшие цели все еще могут принудительно вызвать переход к очереди или окончательный сбой доставки.
    - Если последний видимый ответ ассистента дочерней сессии — точный молчаливый токен `NO_REPLY` / `no_reply` или ровно `ANNOUNCE_SKIP`, OpenClaw намеренно подавляет объявление вместо публикации устаревшего более раннего прогресса.
    - Вывод tool/toolResult не продвигается в текст результата дочерней сессии; результатом является последний видимый ответ ассистента дочерней сессии.

    Отладка:

    ```bash
    openclaw tasks show <runId-or-sessionKey>
    ```

    Документация: [Субагенты](/ru/tools/subagents), [Фоновые задачи](/ru/automation/tasks), [Инструменты сессии](/ru/concepts/session-tool).

  </Accordion>

  <Accordion title="Cron или напоминания не запускаются. Что проверить?">
    Cron выполняется внутри процесса Gateway. Если Gateway не работает непрерывно,
    запланированные задания не будут выполняться.

    Контрольный список:

    - Убедитесь, что cron включен (`cron.enabled`) и `OPENCLAW_SKIP_CRON` не задан.
    - Проверьте, что Gateway работает 24/7 (без сна и перезапусков).
    - Проверьте настройки часового пояса для задания (`--tz` относительно часового пояса хоста).

    Отладка:

    ```bash
    openclaw cron run <jobId>
    openclaw cron runs --id <jobId> --limit 50
    ```

    Документация: [Задания Cron](/ru/automation/cron-jobs), [Автоматизация](/ru/automation).

  </Accordion>

  <Accordion title="Cron сработал, но в канал ничего не отправлено. Почему?">
    Сначала проверьте режим доставки:

    - `--no-deliver` / `delivery.mode: "none"` означает, что резервная отправка runner не ожидается.
    - Отсутствующая или недопустимая цель объявления (`channel` / `to`) означает, что runner пропустил исходящую доставку.
    - Ошибки авторизации канала (`unauthorized`, `Forbidden`) означают, что runner попытался доставить сообщение, но учетные данные это заблокировали.
    - Тихий изолированный результат (только `NO_REPLY` / `no_reply`) считается намеренно недоставляемым, поэтому runner также подавляет резервную доставку из очереди.

    Для изолированных заданий Cron агент все равно может отправлять напрямую через инструмент `message`,
    если доступен маршрут чата. `--announce` управляет только резервным путем runner
    для финального текста, который агент еще не отправил.

    Отладка:

    ```bash
    openclaw cron runs --id <jobId> --limit 50
    openclaw tasks show <runId-or-sessionKey>
    ```

    Документация: [Задания Cron](/ru/automation/cron-jobs), [Фоновые задачи](/ru/automation/tasks).

  </Accordion>

  <Accordion title="Почему изолированный запуск Cron переключил модели или повторился один раз?">
    Обычно это путь переключения живой модели, а не дублирование расписания.

    Изолированный Cron может сохранять передачу модели во время выполнения и повторять запуск, когда активный
    запуск выбрасывает `LiveSessionModelSwitchError`. Повтор сохраняет переключенного
    провайдера/модель, а если переключение передало новое переопределение профиля авторизации, Cron
    также сохраняет его перед повтором.

    Связанные правила выбора:

    - Переопределение модели хуком Gmail имеет приоритет, когда применимо.
    - Затем `model` конкретного задания.
    - Затем любое сохраненное переопределение модели cron-сессии.
    - Затем обычный выбор модели агента/по умолчанию.

    Цикл повторов ограничен. После первоначальной попытки и 2 повторов переключения
    Cron прерывает выполнение, а не зацикливается навсегда.

    Отладка:

    ```bash
    openclaw cron runs --id <jobId> --limit 50
    openclaw tasks show <runId-or-sessionKey>
    ```

    Документация: [Задания Cron](/ru/automation/cron-jobs), [CLI Cron](/ru/cli/cron).

  </Accordion>

  <Accordion title="Как установить Skills в Linux?">
    Используйте встроенные команды `openclaw skills` или поместите Skills в рабочую область. Интерфейс Skills для macOS недоступен в Linux.
    Просматривайте Skills на [https://clawhub.ai](https://clawhub.ai).

    ```bash
    openclaw skills search "calendar"
    openclaw skills search --limit 20
    openclaw skills install @owner/<skill-slug>
    openclaw skills install @owner/<skill-slug> --version <version>
    openclaw skills install @owner/<skill-slug> --force
    openclaw skills install @owner/<skill-slug> --global
    openclaw skills update --all
    openclaw skills update --all --global
    openclaw skills list --eligible
    openclaw skills check
    ```

    Встроенная команда `openclaw skills install` по умолчанию записывает в каталог `skills/`
    активной рабочей области. Добавьте `--global`, чтобы установить в общий управляемый
    каталог Skills для всех локальных агентов. Устанавливайте отдельный CLI `clawhub`
    только если хотите публиковать или синхронизировать собственные Skills. Используйте
    `agents.defaults.skills` или `agents.list[].skills`, если хотите ограничить,
    какие агенты могут видеть общие Skills.

  </Accordion>

  <Accordion title="Может ли OpenClaw запускать задачи по расписанию или непрерывно в фоне?">
    Да. Используйте планировщик Gateway:

    - **Задания Cron** для запланированных или повторяющихся задач (сохраняются между перезапусками).
    - **Heartbeat** для периодических проверок "основной сессии".
    - **Изолированные задания** для автономных агентов, которые публикуют сводки или доставляют сообщения в чаты.

    Документация: [Задания Cron](/ru/automation/cron-jobs), [Автоматизация](/ru/automation),
    [Heartbeat](/ru/gateway/heartbeat).

  </Accordion>

  <Accordion title="Могу ли я запускать Skills только для Apple macOS из Linux?">
    Не напрямую. Skills для macOS ограничиваются `metadata.openclaw.os` и необходимыми бинарными файлами, а Skills появляются в системном промпте только когда они подходят для **хоста Gateway**. В Linux Skills только для `darwin` (например, `apple-notes`, `apple-reminders`, `things-mac`) не загрузятся, если вы не переопределите ограничение.

    Есть три поддерживаемых варианта:

    **Вариант A - запустите Gateway на Mac (самый простой).**
    Запустите Gateway там, где существуют бинарные файлы macOS, затем подключитесь из Linux в [удаленном режиме](#gateway-ports-already-running-and-remote-mode) или через Tailscale. Skills загружаются нормально, потому что хост Gateway работает на macOS.

    **Вариант B - используйте узел macOS (без SSH).**
    Запустите Gateway в Linux, подключите узел macOS (приложение в строке меню) и задайте для **Команд запуска Node** значение "Всегда спрашивать" или "Всегда разрешать" на Mac. OpenClaw может считать Skills только для macOS подходящими, когда необходимые бинарные файлы есть на узле. Агент запускает эти Skills через инструмент `nodes`. Если выбрать "Всегда спрашивать", подтверждение "Всегда разрешать" в промпте добавит эту команду в список разрешенных.

    **Вариант C - проксируйте бинарные файлы macOS через SSH (для опытных пользователей).**
    Оставьте Gateway в Linux, но сделайте так, чтобы необходимые бинарные файлы CLI разрешались в SSH-обертки, которые запускаются на Mac. Затем переопределите Skill, чтобы разрешить Linux и сохранить его доступным.

    1. Создайте SSH-обертку для бинарного файла (пример: `memo` для Apple Notes):

       ```bash
       #!/usr/bin/env bash
       set -euo pipefail
       exec ssh -T user@mac-host /opt/homebrew/bin/memo "$@"
       ```

    2. Поместите обертку в `PATH` на хосте Linux (например, `~/bin/memo`).
    3. Переопределите метаданные Skill (в рабочей области или `~/.openclaw/skills`), чтобы разрешить Linux:

       ```markdown
       ---
       name: apple-notes
       description: Manage Apple Notes via the memo CLI on macOS.
       metadata: { "openclaw": { "os": ["darwin", "linux"], "requires": { "bins": ["memo"] } } }
       ---
       ```

    4. Запустите новую сессию, чтобы снимок Skills обновился.

  </Accordion>

  <Accordion title="Есть ли у вас интеграция с Notion или HeyGen?">
    Сегодня встроенной нет.

    Варианты:

    - **Пользовательский Skill / Plugin:** лучший вариант для надежного доступа к API (у Notion и HeyGen есть API).
    - **Автоматизация браузера:** работает без кода, но медленнее и более хрупкая.

    - Одна страница Notion на клиента (контекст + предпочтения + активная работа).
    - Попросите агента получать эту страницу в начале сессии.

    Если нужна нативная интеграция, откройте запрос функции или создайте Skill,
    работающий с этими API.

    Установите Skills:

    ```bash
    openclaw skills install @owner/<skill-slug>
    openclaw skills update --all
    ```

    Встроенные установки попадают в каталог `skills/` активной рабочей области. Для общих Skills для всех локальных агентов используйте `openclaw skills install @owner/<skill-slug> --global` (или поместите их вручную в `~/.openclaw/skills/<name>/SKILL.md`). Если общую установку должны видеть только некоторые агенты, настройте `agents.defaults.skills` или `agents.list[].skills`. Некоторые Skills ожидают бинарные файлы, установленные через Homebrew; в Linux это означает Linuxbrew (см. запись FAQ Homebrew для Linux выше). См. [Skills](/ru/tools/skills), [Конфигурация Skills](/ru/tools/skills-config) и [ClawHub](/ru/clawhub).

  </Accordion>

  <Accordion title="Как использовать уже авторизованный Chrome с OpenClaw?">
    Используйте встроенный браузерный профиль `user`, который подключается через Chrome DevTools MCP:

    ```bash
    openclaw browser --browser-profile user tabs
    openclaw browser --browser-profile user snapshot
    ```

    Если нужно пользовательское имя, создайте явный профиль MCP:

    ```bash
    openclaw browser create-profile --name chrome-live --driver existing-session
    openclaw browser --browser-profile chrome-live tabs
    ```

    Этот путь может использовать локальный браузер хоста или подключенный браузерный узел. Если Gateway работает в другом месте, либо запустите хост узла на машине с браузером, либо используйте удаленный CDP.

    Текущие ограничения для `existing-session` / `user`:

    - действия управляются ref, а не CSS-селекторами
    - загрузки требуют `ref` / `inputRef` и сейчас поддерживают по одному файлу за раз
    - `responsebody`, экспорт PDF, перехват скачиваний и пакетные действия все еще требуют управляемый браузер или сырой профиль CDP

  <Accordion title="Docker кажется ограниченным - как включить все функции?">
    Образ по умолчанию ориентирован прежде всего на безопасность и запускается от пользователя `node`, поэтому он не
    включает системные пакеты, Homebrew или встроенные браузеры. Для более полной настройки:

    - Сохраняйте `/home/node` с помощью `OPENCLAW_HOME_VOLUME`, чтобы кэши сохранялись.
    - Встраивайте системные зависимости в образ с помощью `OPENCLAW_IMAGE_APT_PACKAGES`.
    - Установите браузеры Playwright через встроенный CLI:
      `node /app/node_modules/playwright-core/cli.js install chromium`
    - Задайте `PLAYWRIGHT_BROWSERS_PATH` и убедитесь, что путь сохраняется.

    Документация: [Docker](/ru/install/docker), [Браузер](/ru/tools/browser).

  </Accordion>

  <Accordion title="Могу ли я оставить личные DM приватными, а группы сделать публичными/изолированными с одним агентом?">
    Да - если ваш приватный трафик это **DM**, а публичный трафик это **группы**.

    Используйте `agents.defaults.sandbox.mode: "non-main"`, чтобы групповые/канальные сессии (ключи не main) запускались в настроенном бэкенде песочницы, а основная DM-сессия оставалась на хосте. Docker является бэкендом по умолчанию, если вы не выберете другой. Затем ограничьте инструменты, доступные в изолированных сессиях, через `tools.sandbox.tools`.

    Пошаговая настройка + пример конфигурации: [Группы: личные DM + публичные группы](/ru/channels/groups#pattern-personal-dms-public-groups-single-agent)

    Справочник по ключевой конфигурации: [Конфигурация Gateway](/ru/gateway/config-agents#agentsdefaultssandbox)

  </Accordion>

  <Accordion title="Как привязать папку хоста к песочнице?">
    Задайте `agents.defaults.sandbox.docker.binds` как `["host:path:mode"]` (например, `"/home/user/src:/src:ro"`). Глобальные привязки и привязки на уровне агента объединяются; привязки на уровне агента игнорируются при `scope: "shared"`. Используйте `:ro` для всего чувствительного и помните, что привязки обходят границы файловой системы песочницы.

    OpenClaw проверяет источники привязок как по нормализованному пути, так и по каноническому пути, разрешенному через самый глубокий существующий предок. Это означает, что выходы через родительские символические ссылки все равно закрываются с отказом, даже если последний сегмент пути еще не существует, а проверки разрешенного корня все равно применяются после разрешения символических ссылок.

    См. [Песочница](/ru/gateway/sandboxing#custom-bind-mounts) и [Песочница vs политика инструментов vs повышенные права](/ru/gateway/sandbox-vs-tool-policy-vs-elevated#bind-mounts-security-quick-check) для примеров и заметок по безопасности.

  </Accordion>

  <Accordion title="Как работает память?">
    Память OpenClaw - это просто Markdown-файлы в рабочей области агента:

    - Ежедневные заметки в `memory/YYYY-MM-DD.md`
    - Курированные долгосрочные заметки в `MEMORY.md` (только основные/приватные сессии)

    OpenClaw также выполняет **тихий сброс памяти перед Compaction**, чтобы напомнить модели
    записать устойчивые заметки перед автоматической Compaction. Это выполняется только когда рабочая область
    доступна для записи (песочницы только для чтения пропускают это). См. [Память](/ru/concepts/memory).

  </Accordion>

  <Accordion title="Память постоянно что-то забывает. Как закрепить информацию?">
    Попросите бота **записать факт в память**. Долгосрочные заметки должны быть в `MEMORY.md`,
    краткосрочный контекст помещается в `memory/YYYY-MM-DD.md`.

    Мы все еще улучшаем эту область. Полезно напомнить модели сохранить воспоминания;
    она поймет, что делать. Если она продолжает забывать, проверьте, что Gateway использует одно и то же
    рабочее пространство при каждом запуске.

    Документация: [Память](/ru/concepts/memory), [Рабочее пространство агента](/ru/concepts/agent-workspace).

  </Accordion>

  <Accordion title="Сохраняется ли память навсегда? Какие есть ограничения?">
    Файлы памяти хранятся на диске и сохраняются, пока вы их не удалите. Ограничение задается вашим
    хранилищем, а не моделью. **Контекст сеанса** по-прежнему ограничен окном контекста модели,
    поэтому длинные диалоги могут сжиматься или усекаться. Поэтому
    существует поиск по памяти - он возвращает в контекст только релевантные части.

    Документация: [Память](/ru/concepts/memory), [Контекст](/ru/concepts/context).

  </Accordion>

  <Accordion title="Требуется ли ключ OpenAI API для семантического поиска по памяти?">
    Только если вы используете **эмбеддинги OpenAI**. Codex OAuth покрывает чат/завершения и
    **не** предоставляет доступ к эмбеддингам, поэтому **вход через Codex (OAuth или
    вход через Codex CLI)** не помогает для семантического поиска по памяти. Для эмбеддингов OpenAI
    все равно нужен настоящий ключ API (`OPENAI_API_KEY` или `models.providers.openai.apiKey`).

    Если явно не задать провайдера, OpenClaw использует эмбеддинги OpenAI. Устаревшие
    конфигурации, где все еще указано `memorySearch.provider = "auto"`, тоже разрешаются в OpenAI.
    Если ключ OpenAI API недоступен, семантический поиск по памяти остается недоступным,
    пока вы не настроите ключ или явно не выберете другого провайдера.

    Если вы предпочитаете оставаться локально, задайте `memorySearch.provider = "local"` (и при необходимости
    `memorySearch.fallback = "none"`). Если нужны эмбеддинги Gemini, задайте
    `memorySearch.provider = "gemini"` и предоставьте `GEMINI_API_KEY` (или
    `memorySearch.remote.apiKey`). Мы поддерживаем модели эмбеддингов **OpenAI, совместимые с OpenAI, Gemini,
    Voyage, Mistral, Bedrock, Ollama, LM Studio, GitHub Copilot, DeepInfra или локальные** -
    подробности настройки см. в разделе [Память](/ru/concepts/memory).

  </Accordion>
</AccordionGroup>

## Где данные находятся на диске

<AccordionGroup>
  <Accordion title="Все ли данные, используемые с OpenClaw, сохраняются локально?">
    Нет - **состояние OpenClaw локально**, но **внешние сервисы по-прежнему видят то, что вы им отправляете**.

    - **Локально по умолчанию:** сеансы, файлы памяти, конфигурация и рабочее пространство находятся на хосте Gateway
      (`~/.openclaw` + каталог вашего рабочего пространства).
    - **Удаленно по необходимости:** сообщения, которые вы отправляете провайдерам моделей (Anthropic/OpenAI/и т. д.), уходят в
      их API, а чат-платформы (WhatsApp/Telegram/Slack/и т. д.) хранят данные сообщений на своих
      серверах.
    - **Вы управляете объемом данных:** использование локальных моделей оставляет промпты на вашей машине, но трафик каналов
      все равно проходит через серверы соответствующего канала.

    Связано: [Рабочее пространство агента](/ru/concepts/agent-workspace), [Память](/ru/concepts/memory).

  </Accordion>

  <Accordion title="Где OpenClaw хранит свои данные?">
    Все находится внутри `$OPENCLAW_STATE_DIR` (по умолчанию: `~/.openclaw`):

    | Путь                                                            | Назначение                                                         |
    | --------------------------------------------------------------- | ------------------------------------------------------------------ |
    | `$OPENCLAW_STATE_DIR/openclaw.json`                             | Основная конфигурация (JSON5)                                      |
    | `$OPENCLAW_STATE_DIR/credentials/oauth.json`                    | Устаревший импорт OAuth (копируется в профили аутентификации при первом использовании) |
    | `$OPENCLAW_STATE_DIR/agents/<agentId>/agent/auth-profiles.json` | Профили аутентификации (OAuth, ключи API и необязательные `keyRef`/`tokenRef`) |
    | `$OPENCLAW_STATE_DIR/secrets.json`                              | Необязательная файловая полезная нагрузка секретов для провайдеров SecretRef `file` |
    | `$OPENCLAW_STATE_DIR/agents/<agentId>/agent/auth.json`          | Устаревший файл совместимости (статические записи `api_key` очищены) |
    | `$OPENCLAW_STATE_DIR/credentials/`                              | Состояние провайдера (например, `whatsapp/<accountId>/creds.json`) |
    | `$OPENCLAW_STATE_DIR/agents/`                                   | Состояние для каждого агента (agentDir + сеансы)                   |
    | `$OPENCLAW_STATE_DIR/agents/<agentId>/sessions/`                | История диалогов и состояние (для каждого агента)                  |
    | `$OPENCLAW_STATE_DIR/agents/<agentId>/sessions/sessions.json`   | Метаданные сеансов (для каждого агента)                            |

    Устаревший путь для одного агента: `~/.openclaw/agent/*` (мигрируется через `openclaw doctor`).

    Ваше **рабочее пространство** (AGENTS.md, файлы памяти, Skills и т. д.) отдельно и настраивается через `agents.defaults.workspace` (по умолчанию: `~/.openclaw/workspace`).

  </Accordion>

  <Accordion title="Где должны находиться AGENTS.md / SOUL.md / USER.md / MEMORY.md?">
    Эти файлы находятся в **рабочем пространстве агента**, а не в `~/.openclaw`.

    - **Рабочее пространство (для каждого агента)**: `AGENTS.md`, `SOUL.md`, `IDENTITY.md`, `USER.md`,
      `MEMORY.md`, `memory/YYYY-MM-DD.md`, необязательный `HEARTBEAT.md`.
      Корневой файл `memory.md` в нижнем регистре является только входом для устаревшего исправления; `openclaw doctor --fix`
      может объединить его с `MEMORY.md`, если оба файла существуют.
    - **Каталог состояния (`~/.openclaw`)**: конфигурация, состояние каналов/провайдеров, профили аутентификации, сеансы, журналы
      и общие Skills (`~/.openclaw/skills`).

    Рабочее пространство по умолчанию: `~/.openclaw/workspace`, настраивается так:

    ```json5
    {
      agents: { defaults: { workspace: "~/.openclaw/workspace" } },
    }
    ```

    Если бот «забывает» после перезапуска, убедитесь, что Gateway использует одно и то же
    рабочее пространство при каждом запуске (и помните: удаленный режим использует
    рабочее пространство **хоста Gateway**, а не вашего локального ноутбука).

    Совет: если вам нужно устойчивое поведение или предпочтение, попросите бота **записать это в
    AGENTS.md или MEMORY.md**, а не полагайтесь на историю чата.

    См. [Рабочее пространство агента](/ru/concepts/agent-workspace) и [Память](/ru/concepts/memory).

  </Accordion>

  <Accordion title="Можно ли увеличить SOUL.md?">
    Да. `SOUL.md` - один из файлов начальной загрузки рабочего пространства, внедряемых в
    контекст агента. Лимит внедрения по умолчанию для одного файла составляет `20000` символов,
    а общий бюджет начальной загрузки по всем файлам - `60000` символов.

    Измените общие значения по умолчанию в конфигурации OpenClaw:

    ```json5
    {
      agents: {
        defaults: {
          bootstrapMaxChars: 50000,
          bootstrapTotalMaxChars: 300000,
        },
      },
    }
    ```

    Или переопределите одного агента:

    ```json5
    {
      agents: {
        list: [
          {
            id: "main",
            bootstrapMaxChars: 50000,
            bootstrapTotalMaxChars: 300000,
          },
        ],
      },
    }
    ```

    Используйте `/context`, чтобы проверить исходные и внедренные размеры, а также было ли усечение.
    Держите `SOUL.md` сфокусированным на голосе, позиции и личности; рабочие правила
    помещайте в `AGENTS.md`, а долговременные факты - в память.

    См. [Контекст](/ru/concepts/context) и [Конфигурация агента](/ru/gateway/config-agents).

  </Accordion>

  <Accordion title="Рекомендуемая стратегия резервного копирования">
    Поместите **рабочее пространство агента** в **приватный** git-репозиторий и сделайте резервную копию где-нибудь
    в приватном месте (например, в приватном GitHub). Это сохранит память + файлы AGENTS/SOUL/USER
    и позволит позже восстановить «разум» ассистента.

    **Не** коммитьте ничего из `~/.openclaw` (учетные данные, сеансы, токены или зашифрованные полезные нагрузки секретов).
    Если требуется полное восстановление, отдельно создайте резервные копии рабочего пространства и каталога состояния
    (см. вопрос о миграции выше).

    Документация: [Рабочее пространство агента](/ru/concepts/agent-workspace).

  </Accordion>

  <Accordion title="Как полностью удалить OpenClaw?">
    См. отдельное руководство: [Удаление](/ru/install/uninstall).
  </Accordion>

  <Accordion title="Могут ли агенты работать вне рабочего пространства?">
    Да. Рабочее пространство - это **cwd по умолчанию** и якорь памяти, а не жесткая песочница.
    Относительные пути разрешаются внутри рабочего пространства, но абсолютные пути могут обращаться к другим
    расположениям на хосте, если песочница не включена. Если нужна изоляция, используйте
    [`agents.defaults.sandbox`](/ru/gateway/sandboxing) или настройки песочницы для отдельного агента. Если вы
    хотите, чтобы репозиторий был рабочим каталогом по умолчанию, укажите корень репозитория в
    `workspace` этого агента. Репозиторий OpenClaw - это просто исходный код; держите
    рабочее пространство отдельно, если вы намеренно не хотите, чтобы агент работал внутри него.

    Пример (репозиторий как cwd по умолчанию):

    ```json5
    {
      agents: {
        defaults: {
          workspace: "~/Projects/my-repo",
        },
      },
    }
    ```

  </Accordion>

  <Accordion title="Удаленный режим: где находится хранилище сеансов?">
    Состоянием сеанса владеет **хост Gateway**. Если вы в удаленном режиме, нужное вам хранилище сеансов находится на удаленной машине, а не на вашем локальном ноутбуке. См. [Управление сеансами](/ru/concepts/session).
  </Accordion>
</AccordionGroup>

## Основы конфигурации

<AccordionGroup>
  <Accordion title="Какой формат у конфигурации? Где она находится?">
    OpenClaw читает необязательную конфигурацию **JSON5** из `$OPENCLAW_CONFIG_PATH` (по умолчанию: `~/.openclaw/openclaw.json`):

    ```
    $OPENCLAW_CONFIG_PATH
    ```

    Если файл отсутствует, используются относительно безопасные значения по умолчанию (включая рабочее пространство по умолчанию `~/.openclaw/workspace`).

  </Accordion>

  <Accordion title='Я задал gateway.bind: "lan" (или "tailnet"), и теперь ничего не слушает / UI говорит, что доступ не авторизован'>
    Привязки не к loopback **требуют действительный путь аутентификации Gateway**. На практике это означает:

    - аутентификация с общим секретом: токен или пароль
    - `gateway.auth.mode: "trusted-proxy"` за корректно настроенным identity-aware обратным прокси

    ```json5
    {
      gateway: {
        bind: "lan",
        auth: {
          mode: "token",
          token: "replace-me",
        },
      },
    }
    ```

    Примечания:

    - `gateway.remote.token` / `.password` сами по себе **не** включают локальную аутентификацию Gateway.
    - Локальные пути вызова могут использовать `gateway.remote.*` как резервный вариант только когда `gateway.auth.*` не задан.
    - Для аутентификации по паролю вместо этого задайте `gateway.auth.mode: "password"` плюс `gateway.auth.password` (или `OPENCLAW_GATEWAY_PASSWORD`).
    - Если `gateway.auth.token` / `gateway.auth.password` явно настроен через SecretRef и не разрешается, разрешение завершается закрыто (без маскировки удаленным резервным вариантом).
    - Настройки Control UI с общим секретом аутентифицируются через `connect.params.auth.token` или `connect.params.auth.password` (хранятся в настройках приложения/UI). Режимы с идентификацией, такие как Tailscale Serve или `trusted-proxy`, вместо этого используют заголовки запросов. Не помещайте общие секреты в URL.
    - При `gateway.auth.mode: "trusted-proxy"` обратные прокси loopback на том же хосте требуют явного `gateway.auth.trustedProxy.allowLoopback = true` и записи loopback в `gateway.trustedProxies`.

  </Accordion>

  <Accordion title="Почему теперь нужен токен на localhost?">
    OpenClaw по умолчанию применяет аутентификацию Gateway, включая loopback. В обычном пути по умолчанию это означает аутентификацию по токену: если явный путь аутентификации не настроен, запуск Gateway разрешается в режим токена и генерирует токен только для времени выполнения этого запуска, поэтому **локальные WS-клиенты должны аутентифицироваться**. Явно настройте `gateway.auth.token`, `gateway.auth.password`, `OPENCLAW_GATEWAY_TOKEN` или `OPENCLAW_GATEWAY_PASSWORD`, если клиентам нужен стабильный секрет между перезапусками. Это блокирует вызовы Gateway другими локальными процессами.

    Если вы предпочитаете другой путь аутентификации, можно явно выбрать режим пароля (или, для identity-aware reverse proxies, `trusted-proxy`). Если вы **действительно** хотите открытый loopback, явно задайте `gateway.auth.mode: "none"` в конфиге. Doctor может сгенерировать токен для вас в любое время: `openclaw doctor --generate-gateway-token`.

  </Accordion>

  <Accordion title="Нужно ли перезапускать после изменения конфига?">
    Gateway отслеживает конфиг и поддерживает горячую перезагрузку:

    - `gateway.reload.mode: "hybrid"` (по умолчанию): безопасные изменения применяются горячо, для критических выполняется перезапуск
    - `hot`, `restart`, `off` также поддерживаются

  </Accordion>

  <Accordion title="Как отключить шутливые слоганы CLI?">
    Задайте `cli.banner.taglineMode` в конфиге:

    ```json5
    {
      cli: {
        banner: {
          taglineMode: "off", // random | default | off
        },
      },
    }
    ```

    - `off`: скрывает текст слогана, но оставляет строку заголовка/версии баннера.
    - `default`: каждый раз использует `All your chats, one OpenClaw.`.
    - `random`: чередующиеся шутливые/сезонные слоганы (поведение по умолчанию).
    - Если вы вообще не хотите показывать баннер, задайте env `OPENCLAW_HIDE_BANNER=1`.

  </Accordion>

  <Accordion title="Как включить веб-поиск (и web fetch)?">
    `web_fetch` работает без API-ключа. `web_search` зависит от выбранного
    провайдера:

    - API-провайдеры, такие как Brave, Exa, Firecrawl, Gemini, Kimi, MiniMax Search, Perplexity и Tavily, требуют обычной настройки API-ключа.
    - Grok может повторно использовать xAI OAuth из аутентификации модели или откатиться к `XAI_API_KEY` / конфигу веб-поиска Plugin.
    - Ollama Web Search не требует ключа, но использует настроенный хост Ollama и требует `ollama signin`.
    - DuckDuckGo не требует ключа, но это неофициальная HTML-интеграция.
    - SearXNG не требует ключа/размещается самостоятельно; настройте `SEARXNG_BASE_URL` или `plugins.entries.searxng.config.webSearch.baseUrl`.

    **Рекомендуется:** выполните `openclaw configure --section web` и выберите провайдера.
    Альтернативы через окружение:

    - Brave: `BRAVE_API_KEY`
    - Exa: `EXA_API_KEY`
    - Firecrawl: `FIRECRAWL_API_KEY`
    - Gemini: `GEMINI_API_KEY`
    - Grok: xAI OAuth, `XAI_API_KEY`
    - Kimi: `KIMI_API_KEY` или `MOONSHOT_API_KEY`
    - MiniMax Search: `MINIMAX_CODE_PLAN_KEY`, `MINIMAX_CODING_API_KEY` или `MINIMAX_API_KEY`
    - Perplexity: `PERPLEXITY_API_KEY` или `OPENROUTER_API_KEY`
    - SearXNG: `SEARXNG_BASE_URL`
    - Tavily: `TAVILY_API_KEY`

    ```json5
    {
      plugins: {
        entries: {
          brave: {
            config: {
              webSearch: {
                apiKey: "BRAVE_API_KEY_HERE",
              },
            },
          },
        },
        },
        tools: {
          web: {
            search: {
              enabled: true,
              provider: "brave",
              maxResults: 5,
            },
            fetch: {
              enabled: true,
              provider: "firecrawl", // optional; omit for auto-detect
            },
          },
        },
    }
    ```

    Конфиг веб-поиска, специфичный для провайдера, теперь находится в `plugins.entries.<plugin>.config.webSearch.*`.
    Устаревшие пути провайдера `tools.web.search.*` временно всё ещё загружаются для совместимости, но их не следует использовать для новых конфигов.
    Конфиг fallback для Firecrawl web-fetch находится в `plugins.entries.firecrawl.config.webFetch.*`.

    Примечания:

    - Если вы используете allowlists, добавьте `web_search`/`web_fetch`/`x_search` или `group:web`.
    - `web_fetch` включён по умолчанию (если явно не отключён).
    - Если `tools.web.fetch.provider` опущен, OpenClaw автоматически определяет первого готового fallback-провайдера fetch по доступным учётным данным. Официальный Plugin Firecrawl предоставляет этот fallback.
    - Демоны читают env vars из `~/.openclaw/.env` (или из окружения сервиса).

    Документация: [Веб-инструменты](/ru/tools/web).

  </Accordion>

  <Accordion title="config.apply стёр мой конфиг. Как восстановиться и избежать этого?">
    `config.apply` заменяет **весь конфиг**. Если отправить частичный объект, всё
    остальное будет удалено.

    Текущий OpenClaw защищает от многих случайных перезаписей:

    - Записи конфига, принадлежащие OpenClaw, проверяют полный конфиг после изменения перед записью.
    - Недопустимые или разрушительные записи, принадлежащие OpenClaw, отклоняются и сохраняются как `openclaw.json.rejected.*`.
    - Если прямое редактирование ломает запуск или горячую перезагрузку, Gateway закрывается с отказом или пропускает перезагрузку; он не переписывает `openclaw.json`.
    - `openclaw doctor --fix` отвечает за восстановление и может вернуть последнюю рабочую версию, сохранив отклонённый файл как `openclaw.json.clobbered.*`.

    Восстановление:

    - Проверьте `openclaw logs --follow` на наличие `Invalid config at`, `Config write rejected:` или `config reload skipped (invalid config)`.
    - Изучите самый новый `openclaw.json.clobbered.*` или `openclaw.json.rejected.*` рядом с активным конфигом.
    - Выполните `openclaw config validate` и `openclaw doctor --fix`.
    - Скопируйте обратно только нужные ключи с помощью `openclaw config set` или `config.patch`.
    - Если у вас нет последней рабочей версии или отклонённой полезной нагрузки, восстановитесь из резервной копии либо повторно выполните `openclaw doctor` и заново настройте каналы/модели.
    - Если это было неожиданно, заведите баг и приложите последний известный конфиг или любую резервную копию.
    - Локальный coding agent часто может восстановить рабочий конфиг по логам или истории.

    Как избежать:

    - Используйте `openclaw config set` для небольших изменений.
    - Используйте `openclaw configure` для интерактивного редактирования.
    - Сначала используйте `config.schema.lookup`, если не уверены в точном пути или форме поля; он возвращает поверхностный узел схемы и краткие сведения о непосредственных дочерних элементах для детализации.
    - Используйте `config.patch` для частичных RPC-изменений; оставляйте `config.apply` только для полной замены конфига.
    - Если вы используете agent-facing инструмент `gateway` из agent run, он всё равно будет отклонять записи в `tools.exec.ask` / `tools.exec.security` (включая устаревшие алиасы `tools.bash.*`, которые нормализуются в те же защищённые exec-пути).

    Документация: [Конфиг](/ru/cli/config), [Настройка](/ru/cli/configure), [Устранение неполадок Gateway](/ru/gateway/troubleshooting#gateway-rejected-invalid-config), [Doctor](/ru/gateway/doctor).

  </Accordion>

  <Accordion title="Как запустить центральный Gateway со специализированными воркерами на разных устройствах?">
    Обычный шаблон — **один Gateway** (например, Raspberry Pi) плюс **узлы** и **агенты**:

    - **Gateway (центральный):** отвечает за каналы (Signal/WhatsApp), маршрутизацию и сессии.
    - **Узлы (устройства):** Macs/iOS/Android подключаются как периферийные устройства и предоставляют локальные инструменты (`system.run`, `canvas`, `camera`).
    - **Агенты (воркеры):** отдельные brains/workspaces для специальных ролей (например, "Hetzner ops", "Personal data").
    - **Суб-агенты:** запускайте фоновую работу из основного агента, когда нужен параллелизм.
    - **TUI:** подключайтесь к Gateway и переключайте агентов/сессии.

    Документация: [Узлы](/ru/nodes), [Удалённый доступ](/ru/gateway/remote), [Маршрутизация нескольких агентов](/ru/concepts/multi-agent), [Суб-агенты](/ru/tools/subagents), [TUI](/ru/web/tui).

  </Accordion>

  <Accordion title="Может ли браузер OpenClaw работать в headless-режиме?">
    Да. Это опция конфига:

    ```json5
    {
      browser: { headless: true },
      agents: {
        defaults: {
          sandbox: { browser: { headless: true } },
        },
      },
    }
    ```

    По умолчанию используется `false` (с графическим интерфейсом). Headless чаще запускает антибот-проверки на некоторых сайтах. См. [Браузер](/ru/tools/browser).

    Headless использует **тот же движок Chromium** и работает для большинства задач автоматизации (формы, клики, scraping, входы в аккаунты). Основные различия:

    - Нет видимого окна браузера (используйте скриншоты, если нужны визуальные данные).
    - Некоторые сайты строже относятся к автоматизации в headless-режиме (CAPTCHA, антибот).
      Например, X/Twitter часто блокирует headless-сессии.

  </Accordion>

  <Accordion title="Как использовать Brave для управления браузером?">
    Задайте `browser.executablePath` на бинарный файл Brave (или любого браузера на основе Chromium) и перезапустите Gateway.
    См. полные примеры конфига в [Браузер](/ru/tools/browser#use-brave-or-another-chromium-based-browser).
  </Accordion>
</AccordionGroup>

## Удалённые Gateway и узлы

<AccordionGroup>
  <Accordion title="Как команды передаются между Telegram, gateway и узлами?">
    Сообщения Telegram обрабатываются **gateway**. Gateway запускает агента и
    только затем вызывает узлы через **Gateway WebSocket**, когда нужен инструмент узла:

    Telegram → Gateway → Agent → `node.*` → Node → Gateway → Telegram

    Узлы не видят входящий трафик провайдера; они получают только RPC-вызовы узла.

  </Accordion>

  <Accordion title="Как мой агент может получить доступ к моему компьютеру, если Gateway размещён удалённо?">
    Короткий ответ: **сопрягите компьютер как узел**. Gateway работает в другом месте, но может
    вызывать инструменты `node.*` (экран, камера, система) на вашей локальной машине через Gateway WebSocket.

    Типичная настройка:

    1. Запустите Gateway на постоянно включённом хосте (VPS/домашний сервер).
    2. Поместите хост Gateway и ваш компьютер в одну tailnet.
    3. Убедитесь, что Gateway WS доступен (привязка tailnet или SSH-туннель).
    4. Откройте приложение macOS локально и подключитесь в режиме **Remote over SSH** (или напрямую через tailnet),
       чтобы оно могло зарегистрироваться как узел.
    5. Одобрите узел на Gateway:

       ```bash
       openclaw devices list
       openclaw devices approve <requestId>
       ```

    Отдельный TCP-мост не требуется; узлы подключаются через Gateway WebSocket.

    Напоминание о безопасности: сопряжение узла macOS позволяет выполнять `system.run` на этой машине. Сопрягайте
    только устройства, которым доверяете, и изучите [Безопасность](/ru/gateway/security).

    Документация: [Узлы](/ru/nodes), [Протокол Gateway](/ru/gateway/protocol), [Удалённый режим macOS](/ru/platforms/mac/remote), [Безопасность](/ru/gateway/security).

  </Accordion>

  <Accordion title="Tailscale подключён, но я не получаю ответов. Что делать?">
    Проверьте базовые вещи:

    - Gateway запущен: `openclaw gateway status`
    - Состояние Gateway: `openclaw status`
    - Состояние канала: `openclaw channels status`

    Затем проверьте аутентификацию и маршрутизацию:

    - Если вы используете Tailscale Serve, убедитесь, что `gateway.auth.allowTailscale` задан корректно.
    - Если подключаетесь через SSH-туннель, подтвердите, что локальный туннель поднят и указывает на правильный порт.
    - Убедитесь, что ваши allowlists (DM или группа) включают ваш аккаунт.

    Документация: [Tailscale](/ru/gateway/tailscale), [Удалённый доступ](/ru/gateway/remote), [Каналы](/ru/channels).

  </Accordion>

  <Accordion title="Могут ли два экземпляра OpenClaw общаться друг с другом (локальный + VPS)?">
    Да. Встроенного моста "bot-to-bot" нет, но это можно настроить несколькими
    надёжными способами:

    **Самый простой:** используйте обычный чат-канал, к которому имеют доступ оба бота (Telegram/Slack/WhatsApp).
    Пусть Bot A отправит сообщение Bot B, а затем Bot B ответит как обычно.

    **CLI-мост (универсальный):** запустите скрипт, который вызывает другой Gateway с
    `openclaw agent --message ... --deliver`, нацеливаясь на чат, где слушает другой бот.
    Если один бот находится на удалённом VPS, направьте CLI на этот удалённый Gateway
    через SSH/Tailscale (см. [Удалённый доступ](/ru/gateway/remote)).

    Пример шаблона (запускайте с машины, которая может достучаться до целевого Gateway):

    ```bash
    openclaw agent --message "Hello from local bot" --deliver --channel telegram --reply-to <chat-id>
    ```

    Совет: добавьте защитное ограничение, чтобы два бота не зациклились бесконечно (только по упоминанию, channel
    allowlists или правило "не отвечать на сообщения ботов").

    Документация: [Удалённый доступ](/ru/gateway/remote), [CLI агента](/ru/cli/agent), [Отправка агентом](/ru/tools/agent-send).

  </Accordion>

  <Accordion title="Нужны ли отдельные VPS для нескольких агентов?">
    Нет. Один Gateway может размещать нескольких агентов, каждый со своим workspace, настройками модели по умолчанию
    и маршрутизацией. Это обычная схема, и она намного дешевле и проще, чем запускать
    отдельный VPS для каждого агента.

    Используйте отдельные VPS только когда нужна жёсткая изоляция (границы безопасности) или очень
    разные конфиги, которыми вы не хотите делиться. В остальных случаях оставьте один Gateway и
    используйте нескольких агентов или суб-агентов.

  </Accordion>

  <Accordion title="Есть ли преимущество в использовании Node на моем личном ноутбуке вместо SSH с VPS?">
    Да — Node являются первоклассным способом доступа к вашему ноутбуку из удаленного Gateway и
    открывают больше возможностей, чем доступ к shell. Gateway работает на macOS/Linux (Windows через WSL2) и
    легковесен (подойдет небольшой VPS или устройство класса Raspberry Pi; 4 ГБ RAM более чем достаточно), поэтому распространенная
    схема — постоянно включенный хост плюс ваш ноутбук как Node.

    - **Входящий SSH не требуется.** Node подключаются наружу к Gateway WebSocket и используют сопряжение устройств.
    - **Более безопасные средства управления выполнением.** `system.run` ограничивается allowlist/одобрениями Node на этом ноутбуке.
    - **Больше инструментов устройства.** Node предоставляют `canvas`, `camera` и `screen` в дополнение к `system.run`.
    - **Локальная автоматизация браузера.** Держите Gateway на VPS, но запускайте Chrome локально через хост Node на ноутбуке или подключайтесь к локальному Chrome на хосте через Chrome MCP.

    SSH подходит для разового доступа к shell, но Node проще для постоянных агентских рабочих процессов и
    автоматизации устройств.

    Документация: [Узлы](/ru/nodes), [CLI узлов](/ru/cli/nodes), [Браузер](/ru/tools/browser).

  </Accordion>

  <Accordion title="Запускают ли Node службу Gateway?">
    Нет. На одном хосте должен работать только **один gateway**, если только вы намеренно не запускаете изолированные профили (см. [Несколько gateway](/ru/gateway/multiple-gateways)). Node — это периферийные устройства, которые подключаются
    к gateway (Node iOS/Android или «режим Node» macOS в приложении строки меню). Для безголовых хостов Node
    и управления через CLI см. [CLI хоста Node](/ru/cli/node).

    Полный перезапуск требуется для изменений `gateway`, `discovery` и поверхности размещаемых plugin.

  </Accordion>

  <Accordion title="Есть ли способ применить конфиг через API / RPC?">
    Да.

    - `config.schema.lookup`: проверить одно поддерево конфига с его неглубоким узлом схемы, совпавшей UI-подсказкой и сводками непосредственных дочерних элементов перед записью
    - `config.get`: получить текущий снимок + хеш
    - `config.patch`: безопасное частичное обновление (предпочтительно для большинства правок через RPC); горячая перезагрузка, когда возможно, и перезапуск, когда требуется
    - `config.apply`: проверить + заменить полный конфиг; горячая перезагрузка, когда возможно, и перезапуск, когда требуется
    - Обращенный к агенту runtime-инструмент `gateway` по-прежнему отказывается переписывать `tools.exec.ask` / `tools.exec.security`; устаревшие алиасы `tools.bash.*` нормализуются в те же защищенные пути exec

  </Accordion>

  <Accordion title="Минимальный разумный конфиг для первой установки">
    ```json5
    {
      agents: { defaults: { workspace: "~/.openclaw/workspace" } },
      channels: { whatsapp: { allowFrom: ["+15555550123"] } },
    }
    ```

    Это задает вашу рабочую область и ограничивает, кто может запускать бота.

  </Accordion>

  <Accordion title="Как настроить Tailscale на VPS и подключиться с моего Mac?">
    Минимальные шаги:

    1. **Установите + войдите на VPS**

       ```bash
       curl -fsSL https://tailscale.com/install.sh | sh
       sudo tailscale up
       ```

    2. **Установите + войдите на Mac**
       - Используйте приложение Tailscale и войдите в ту же tailnet.
    3. **Включите MagicDNS (рекомендуется)**
       - В консоли администратора Tailscale включите MagicDNS, чтобы VPS имел стабильное имя.
    4. **Используйте hostname tailnet**
       - SSH: `ssh user@your-vps.tailnet-xxxx.ts.net`
       - Gateway WS: `ws://your-vps.tailnet-xxxx.ts.net:18789`

    Если вам нужен Control UI без SSH, используйте Tailscale Serve на VPS:

    ```bash
    openclaw gateway --tailscale serve
    ```

    Это оставляет gateway привязанным к loopback и открывает HTTPS через Tailscale. См. [Tailscale](/ru/gateway/tailscale).

  </Accordion>

  <Accordion title="Как подключить Node Mac к удаленному Gateway (Tailscale Serve)?">
    Serve открывает **Gateway Control UI + WS**. Node подключаются через тот же endpoint Gateway WS.

    Рекомендуемая настройка:

    1. **Убедитесь, что VPS + Mac находятся в одной tailnet**.
    2. **Используйте приложение macOS в удаленном режиме** (цель SSH может быть hostname tailnet).
       Приложение создаст туннель к порту Gateway и подключится как Node.
    3. **Одобрите Node** на gateway:

       ```bash
       openclaw devices list
       openclaw devices approve <requestId>
       ```

    Документация: [Протокол Gateway](/ru/gateway/protocol), [Обнаружение](/ru/gateway/discovery), [Удаленный режим macOS](/ru/platforms/mac/remote).

  </Accordion>

  <Accordion title="Стоит ли устанавливать на второй ноутбук или просто добавить Node?">
    Если вам нужны только **локальные инструменты** (screen/camera/exec) на втором ноутбуке, добавьте его как
    **Node**. Это сохраняет единый Gateway и избегает дублирования конфига. Локальные инструменты Node
    сейчас доступны только для macOS, но мы планируем расширить их на другие ОС.

    Устанавливайте второй Gateway только тогда, когда вам нужна **жесткая изоляция** или два полностью отдельных бота.

    Документация: [Узлы](/ru/nodes), [CLI узлов](/ru/cli/nodes), [Несколько gateway](/ru/gateway/multiple-gateways).

  </Accordion>
</AccordionGroup>

## Переменные окружения и загрузка .env

<AccordionGroup>
  <Accordion title="Как OpenClaw загружает переменные окружения?">
    OpenClaw читает переменные окружения из родительского процесса (shell, launchd/systemd, CI и т. д.) и дополнительно загружает:

    - `.env` из текущего рабочего каталога
    - глобальный резервный `.env` из `~/.openclaw/.env` (также известный как `$OPENCLAW_STATE_DIR/.env`)

    Ни один файл `.env` не переопределяет существующие переменные окружения.
    Переменные учетных данных провайдеров являются исключением для workspace `.env`: ключи вроде
    `GEMINI_API_KEY`, `XAI_API_KEY` или `MISTRAL_API_KEY` игнорируются из workspace
    `.env` и должны находиться в окружении процесса, `~/.openclaw/.env` или config `env`.

    Вы также можете определить встроенные переменные окружения в конфиге (применяются только если отсутствуют в окружении процесса):

    ```json5
    {
      env: {
        OPENROUTER_API_KEY: "sk-or-...",
        vars: { GROQ_API_KEY: "gsk-..." },
      },
    }
    ```

    Полный порядок приоритета и источники см. в [/environment](/ru/help/environment).

  </Accordion>

  <Accordion title="Я запустил Gateway через службу, и мои переменные окружения исчезли. Что теперь?">
    Два распространенных исправления:

    1. Поместите отсутствующие ключи в `~/.openclaw/.env`, чтобы они подхватывались даже тогда, когда служба не наследует окружение вашего shell.
    2. Включите импорт shell (удобство по явному включению):

    ```json5
    {
      env: {
        shellEnv: {
          enabled: true,
          timeoutMs: 15000,
        },
      },
    }
    ```

    Это запускает ваш login shell и импортирует только отсутствующие ожидаемые ключи (никогда не переопределяет). Эквиваленты переменных окружения:
    `OPENCLAW_LOAD_SHELL_ENV=1`, `OPENCLAW_SHELL_ENV_TIMEOUT_MS=15000`.

  </Accordion>

  <Accordion title='Я задал COPILOT_GITHUB_TOKEN, но статус моделей показывает "Окружение shell: выкл." Почему?'>
    `openclaw models status` сообщает, включен ли **импорт окружения shell**. "Окружение shell: выкл."
    **не** означает, что ваши переменные окружения отсутствуют — это лишь означает, что OpenClaw не будет
    автоматически загружать ваш login shell.

    Если Gateway работает как служба (launchd/systemd), он не унаследует ваше окружение
    shell. Исправьте это одним из способов:

    1. Поместите токен в `~/.openclaw/.env`:

       ```
       COPILOT_GITHUB_TOKEN=...
       ```

    2. Или включите импорт shell (`env.shellEnv.enabled: true`).
    3. Или добавьте его в блок config `env` (применяется только если отсутствует).

    Затем перезапустите gateway и проверьте снова:

    ```bash
    openclaw models status
    ```

    Токены Copilot читаются из `COPILOT_GITHUB_TOKEN` (также `GH_TOKEN` / `GITHUB_TOKEN`).
    См. [/concepts/model-providers](/ru/concepts/model-providers) и [/environment](/ru/help/environment).

  </Accordion>
</AccordionGroup>

## Сессии и несколько чатов

<AccordionGroup>
  <Accordion title="Как начать новый разговор?">
    Отправьте `/new` или `/reset` как отдельное сообщение. См. [Управление сессиями](/ru/concepts/session).
  </Accordion>

  <Accordion title="Сбрасываются ли сессии автоматически, если я никогда не отправляю /new?">
    Сессии могут истекать после `session.idleMinutes`, но это **отключено по умолчанию** (по умолчанию **0**).
    Установите положительное значение, чтобы включить истечение по простоям. Когда включено, **следующее**
    сообщение после периода простоя начинает новую session id для этого ключа чата.
    Это не удаляет расшифровки — оно просто начинает новую сессию.

    ```json5
    {
      session: {
        idleMinutes: 240,
      },
    }
    ```

  </Accordion>

  <Accordion title="Есть ли способ создать команду экземпляров OpenClaw (один гендиректор и много агентов)?">
    Да, через **маршрутизацию нескольких агентов** и **субагентов**. Вы можете создать одного агента-координатора
    и несколько рабочих агентов с собственными рабочими областями и моделями.

    Тем не менее это лучше воспринимать как **интересный эксперимент**. Он требует много токенов и часто
    менее эффективен, чем один бот с отдельными сессиями. Типичная модель, которую мы
    видим: один бот, с которым вы общаетесь, и разные сессии для параллельной работы. Этот
    бот также может запускать субагентов при необходимости.

    Документация: [Маршрутизация нескольких агентов](/ru/concepts/multi-agent), [Субагенты](/ru/tools/subagents), [CLI агентов](/ru/cli/agents).

  </Accordion>

  <Accordion title="Почему контекст был обрезан посреди задачи? Как это предотвратить?">
    Контекст сессии ограничен окном модели. Длинные чаты, большие выводы инструментов или множество
    файлов могут вызвать Compaction или обрезку.

    Что помогает:

    - Попросите бота резюмировать текущее состояние и записать его в файл.
    - Используйте `/compact` перед длинными задачами и `/new` при смене темы.
    - Держите важный контекст в рабочей области и просите бота прочитать его обратно.
    - Используйте субагентов для долгой или параллельной работы, чтобы основной чат оставался меньше.
    - Выберите модель с более большим окном контекста, если это часто происходит.

  </Accordion>

  <Accordion title="Как полностью сбросить OpenClaw, но оставить его установленным?">
    Используйте команду reset:

    ```bash
    openclaw reset
    ```

    Полный неинтерактивный сброс:

    ```bash
    openclaw reset --scope full --yes --non-interactive
    ```

    Затем снова запустите настройку:

    ```bash
    openclaw onboard --install-daemon
    ```

    Примечания:

    - Onboarding также предлагает **Сброс**, если видит существующий конфиг. См. [Onboarding (CLI)](/ru/start/wizard).
    - Если вы использовали профили (`--profile` / `OPENCLAW_PROFILE`), сбросьте каждый state dir (по умолчанию `~/.openclaw-<profile>`).
    - Сброс для разработки: `openclaw gateway --dev --reset` (только для разработки; стирает dev-конфиг + учетные данные + сессии + рабочую область).

  </Accordion>

  <Accordion title='Я получаю ошибки "контекст слишком большой" — как выполнить сброс или compact?'>
    Используйте один из вариантов:

    - **Compact** (сохраняет разговор, но резюмирует старые ходы):

      ```
      /compact
      ```

      или `/compact <instructions>`, чтобы направить резюме.

    - **Сброс** (новый session ID для того же ключа чата):

      ```
      /new
      /reset
      ```

    Если это продолжается:

    - Включите или настройте **обрезку сессии** (`agents.defaults.contextPruning`), чтобы урезать старый вывод инструментов.
    - Используйте модель с более большим окном контекста.

    Документация: [Compaction](/ru/concepts/compaction), [Обрезка сессии](/ru/concepts/session-pruning), [Управление сессиями](/ru/concepts/session).

  </Accordion>

  <Accordion title='Почему я вижу "LLM request rejected: messages.content.tool_use.input field required"?'>
    Это ошибка валидации провайдера: модель выдала блок `tool_use` без обязательного
    `input`. Обычно это означает, что история сессии устарела или повреждена (часто после длинных тредов
    или изменения инструмента/схемы).

    Исправление: начните новую сессию с `/new` (отдельным сообщением).

  </Accordion>

  <Accordion title="Почему я получаю сообщения Heartbeat каждые 30 минут?">
    Heartbeat по умолчанию запускаются каждые **30m** (**1h** при использовании OAuth auth). Настройте или отключите их:

    ```json5
    {
      agents: {
        defaults: {
          heartbeat: {
            every: "2h", // or "0m" to disable
          },
        },
      },
    }
    ```

    Если `HEARTBEAT.md` существует, но фактически пуст (содержит только пустые строки,
    комментарии Markdown/HTML, заголовки Markdown вроде `# Heading`, маркеры ограждений
    или пустые заготовки чек-листов), OpenClaw пропускает запуск Heartbeat, чтобы сэкономить вызовы API.
    Если файл отсутствует, Heartbeat всё равно запускается, а модель решает, что делать.

    Переопределения для отдельных агентов используют `agents.list[].heartbeat`. Документация: [Heartbeat](/ru/gateway/heartbeat).

  </Accordion>

  <Accordion title='Нужно ли добавлять "аккаунт бота" в группу WhatsApp?'>
    Нет. OpenClaw работает через **ваш собственный аккаунт**, поэтому если вы состоите в группе, OpenClaw может её видеть.
    По умолчанию ответы в группах заблокированы, пока вы не разрешите отправителей (`groupPolicy: "allowlist"`).

    Если вы хотите, чтобы только **вы** могли запускать ответы в группе:

    ```json5
    {
      channels: {
        whatsapp: {
          groupPolicy: "allowlist",
          groupAllowFrom: ["+15551234567"],
        },
      },
    }
    ```

  </Accordion>

  <Accordion title="Как получить JID группы WhatsApp?">
    Вариант 1 (самый быстрый): просматривайте логи в реальном времени и отправьте тестовое сообщение в группе:

    ```bash
    openclaw logs --follow --json
    ```

    Ищите `chatId` (или `from`), заканчивающийся на `@g.us`, например:
    `1234567890-1234567890@g.us`.

    Вариант 2 (если уже настроено/добавлено в список разрешённых): выведите группы из конфигурации:

    ```bash
    openclaw directory groups list --channel whatsapp
    ```

    Документация: [WhatsApp](/ru/channels/whatsapp), [каталог](/ru/cli/directory), [логи](/ru/cli/logs).

  </Accordion>

  <Accordion title="Почему OpenClaw не отвечает в группе?">
    Две распространённые причины:

    - Ограничение по упоминанию включено (по умолчанию). Нужно @упомянуть бота (или совпасть с `mentionPatterns`).
    - Вы настроили `channels.whatsapp.groups` без `"*"`, и группа не добавлена в список разрешённых.

    См. [группы](/ru/channels/groups) и [групповые сообщения](/ru/channels/group-messages).

  </Accordion>

  <Accordion title="Группы/треды используют общий контекст с личными сообщениями?">
    Прямые чаты по умолчанию сворачиваются в основной сеанс. У групп/каналов есть собственные ключи сеансов, а темы Telegram / треды Discord являются отдельными сеансами. См. [группы](/ru/channels/groups) и [групповые сообщения](/ru/channels/group-messages).
  </Accordion>

  <Accordion title="Сколько рабочих пространств и агентов можно создать?">
    Жёстких ограничений нет. Десятки (даже сотни) — нормально, но следите за:

    - **Ростом диска:** сеансы + расшифровки хранятся в `~/.openclaw/agents/<agentId>/sessions/`.
    - **Стоимостью токенов:** больше агентов означает больше одновременного использования моделей.
    - **Операционными накладными расходами:** профили авторизации, рабочие пространства и маршрутизация каналов для каждого агента.

    Советы:

    - Держите одно **активное** рабочее пространство на агента (`agents.defaults.workspace`).
    - Очищайте старые сеансы (удаляйте JSONL или записи хранилища), если диск растёт.
    - Используйте `openclaw doctor`, чтобы находить лишние рабочие пространства и несовпадения профилей.

  </Accordion>

  <Accordion title="Можно ли запускать несколько ботов или чатов одновременно (Slack), и как это настроить?">
    Да. Используйте **маршрутизацию нескольких агентов**, чтобы запускать несколько изолированных агентов и маршрутизировать входящие сообщения по
    каналу/аккаунту/собеседнику. Slack поддерживается как канал и может быть привязан к конкретным агентам.

    Доступ к браузеру мощный, но это не означает "делать всё, что может человек" — защита от ботов, CAPTCHA и MFA
    всё ещё могут блокировать автоматизацию. Для наиболее надёжного управления браузером используйте локальный Chrome MCP на хосте
    или CDP на машине, где фактически запущен браузер.

    Рекомендуемая настройка:

    - Постоянно включённый хост Gateway (VPS/Mac mini).
    - Один агент на роль (привязки).
    - Канал(ы) Slack, привязанные к этим агентам.
    - Локальный браузер через Chrome MCP или узел при необходимости.

    Документация: [маршрутизация нескольких агентов](/ru/concepts/multi-agent), [Slack](/ru/channels/slack),
    [браузер](/ru/tools/browser), [узлы](/ru/nodes).

  </Accordion>
</AccordionGroup>

## Gateway: порты, "уже запущен" и удалённый режим

<AccordionGroup>
  <Accordion title="Какой порт использует Gateway?">
    `gateway.port` управляет единым мультиплексированным портом для WebSocket + HTTP (интерфейс управления, хуки и т. д.).

    Приоритет:

    ```
    --port > OPENCLAW_GATEWAY_PORT > gateway.port > default 18789
    ```

  </Accordion>

  <Accordion title='Почему openclaw gateway status показывает "Runtime: running", но "Connectivity probe: failed"?'>
    Потому что "running" — это представление **супервизора** (launchd/systemd/schtasks). Проверка подключения — это CLI, который фактически подключается к WebSocket Gateway.

    Используйте `openclaw gateway status` и ориентируйтесь на эти строки:

    - `Probe target:` (URL, который фактически использовала проверка)
    - `Listening:` (что фактически привязано к порту)
    - `Last gateway error:` (частая первопричина, когда процесс жив, но порт не слушает)

  </Accordion>

  <Accordion title='Почему openclaw gateway status показывает разные "Config (cli)" и "Config (service)"?'>
    Вы редактируете один файл конфигурации, а сервис использует другой (часто из-за несовпадения `--profile` / `OPENCLAW_STATE_DIR`).

    Исправление:

    ```bash
    openclaw gateway install --force
    ```

    Запустите это из того же `--profile` / окружения, которое должен использовать сервис.

  </Accordion>

  <Accordion title='Что означает "another gateway instance is already listening"?'>
    OpenClaw обеспечивает блокировку среды выполнения, немедленно привязывая слушатель WebSocket при запуске (по умолчанию `ws://127.0.0.1:18789`). Если привязка завершается ошибкой `EADDRINUSE`, он выбрасывает `GatewayLockError`, указывая, что другой экземпляр уже слушает.

    Исправление: остановите другой экземпляр, освободите порт или запустите с `openclaw gateway --port <port>`.

  </Accordion>

  <Accordion title="Как запустить OpenClaw в удалённом режиме (клиент подключается к Gateway в другом месте)?">
    Задайте `gateway.mode: "remote"` и укажите удалённый URL WebSocket, при желании с удалёнными учётными данными с общим секретом:

    ```json5
    {
      gateway: {
        mode: "remote",
        remote: {
          url: "ws://gateway.tailnet:18789",
          token: "your-token",
          password: "your-password",
        },
      },
    }
    ```

    Примечания:

    - `openclaw gateway` запускается только когда `gateway.mode` равен `local` (или вы передали флаг переопределения).
    - Приложение macOS отслеживает файл конфигурации и переключает режимы на лету при изменении этих значений.
    - `gateway.remote.token` / `.password` — это только клиентские удалённые учётные данные; сами по себе они не включают авторизацию локального Gateway.

  </Accordion>

  <Accordion title='Интерфейс управления сообщает "unauthorized" (или постоянно переподключается). Что делать?'>
    Путь авторизации вашего Gateway и метод авторизации интерфейса не совпадают.

    Факты (из кода):

    - Интерфейс управления хранит токен в `sessionStorage` для текущей сессии вкладки браузера и выбранного URL Gateway, поэтому обновления в той же вкладке продолжают работать без восстановления долгоживущего сохранения токена в localStorage.
    - При `AUTH_TOKEN_MISMATCH` доверенные клиенты могут выполнить одну ограниченную повторную попытку с кэшированным токеном устройства, когда Gateway возвращает подсказки повтора (`canRetryWithDeviceToken=true`, `recommendedNextStep=retry_with_device_token`).
    - Эта повторная попытка с кэшированным токеном теперь повторно использует кэшированные утверждённые области доступа, сохранённые вместе с токеном устройства. Вызывающие стороны с явным `deviceToken` / явными `scopes` по-прежнему сохраняют запрошенный набор областей вместо наследования кэшированных областей.
    - Вне этого пути повторной попытки приоритет авторизации подключения такой: сначала явный общий токен/пароль, затем явный `deviceToken`, затем сохранённый токен устройства, затем начальный токен.
    - Встроенная начальная настройка через setup-code возвращает токен устройства узла с `scopes: []` плюс ограниченный токен передачи оператору для доверенного мобильного онбординга. Передача оператору может читать нативную конфигурацию времени настройки, но не выдаёт области для изменения спаривания или `operator.admin`.

    Исправление:

    - Самое быстрое: `openclaw dashboard` (печатает + копирует URL панели мониторинга, пытается открыть; показывает подсказку SSH, если нет графической среды).
    - Если у вас ещё нет токена: `openclaw doctor --generate-gateway-token`.
    - Если удалённо, сначала создайте туннель: `ssh -N -L 18789:127.0.0.1:18789 user@host`, затем откройте `http://127.0.0.1:18789/`.
    - Режим общего секрета: задайте `gateway.auth.token` / `OPENCLAW_GATEWAY_TOKEN` или `gateway.auth.password` / `OPENCLAW_GATEWAY_PASSWORD`, затем вставьте соответствующий секрет в настройках интерфейса управления.
    - Режим Tailscale Serve: убедитесь, что `gateway.auth.allowTailscale` включён и вы открываете URL Serve, а не прямой loopback/tailnet URL, который обходит заголовки идентичности Tailscale.
    - Режим доверенного прокси: убедитесь, что вы проходите через настроенный прокси с учётом идентичности, а не через прямой URL Gateway. Прокси local loopback на том же хосте также требуют `gateway.auth.trustedProxy.allowLoopback = true`.
    - Если несоответствие сохраняется после одной повторной попытки, ротируйте/повторно утвердите токен спаренного устройства:
      - `openclaw devices list`
      - `openclaw devices rotate --device <id> --role operator`
    - Если этот вызов ротации говорит, что он отклонён, проверьте две вещи:
      - сеансы спаренных устройств могут ротировать только своё **собственное** устройство, если у них также нет `operator.admin`
      - явные значения `--scope` не могут превышать текущие области оператора вызывающей стороны
    - Всё ещё не получается? Запустите `openclaw status --all` и следуйте [устранению неполадок](/ru/gateway/troubleshooting). Подробности авторизации см. в [панели мониторинга](/ru/web/dashboard).

  </Accordion>

  <Accordion title="Я задал gateway.bind tailnet, но он не может привязаться и ничего не слушает">
    Привязка `tailnet` выбирает IP Tailscale из сетевых интерфейсов (100.64.0.0/10). Если машина не в Tailscale (или интерфейс выключен), привязываться не к чему.

    Исправление:

    - Запустите Tailscale на этом хосте (чтобы у него был адрес 100.x), или
    - Переключитесь на `gateway.bind: "loopback"` / `"lan"`.

    Примечание: `tailnet` задаётся явно. `auto` предпочитает loopback; используйте `gateway.bind: "tailnet"`, когда нужна привязка только к tailnet.

  </Accordion>

  <Accordion title="Можно ли запускать несколько Gateway на одном хосте?">
    Обычно нет — один Gateway может запускать несколько каналов сообщений и агентов. Используйте несколько Gateway только когда нужна избыточность (например: резервный бот) или жёсткая изоляция.

    Да, но нужно изолировать:

    - `OPENCLAW_CONFIG_PATH` (конфигурация для каждого экземпляра)
    - `OPENCLAW_STATE_DIR` (состояние для каждого экземпляра)
    - `agents.defaults.workspace` (изоляция рабочих пространств)
    - `gateway.port` (уникальные порты)

    Быстрая настройка (рекомендуется):

    - Используйте `openclaw --profile <name> ...` для каждого экземпляра (автоматически создаёт `~/.openclaw-<name>`).
    - Задайте уникальный `gateway.port` в конфигурации каждого профиля (или передайте `--port` для ручных запусков).
    - Установите сервис для каждого профиля: `openclaw --profile <name> gateway install`.

    Профили также добавляют суффиксы к именам сервисов (`ai.openclaw.<profile>`; устаревшие `com.openclaw.*`, `openclaw-gateway-<profile>.service`, `OpenClaw Gateway (<profile>)`).
    Полное руководство: [несколько Gateway](/ru/gateway/multiple-gateways).

  </Accordion>

  <Accordion title='Что означает "invalid handshake" / код 1008?'>
    Gateway — это **сервер WebSocket**, и он ожидает, что самым первым сообщением
    будет кадр `connect`. Если он получает что-либо другое, он закрывает соединение
    с **кодом 1008** (нарушение политики).

    Распространённые причины:

    - Вы открыли **HTTP** URL в браузере (`http://...`) вместо клиента WS.
    - Вы использовали неправильный порт или путь.
    - Прокси или туннель удалил заголовки авторизации или отправил запрос не для Gateway.

    Быстрые исправления:

    1. Используйте URL WS: `ws://<host>:18789` (или `wss://...`, если HTTPS).
    2. Не открывайте порт WS в обычной вкладке браузера.
    3. Если авторизация включена, включите токен/пароль в кадр `connect`.

    Если вы используете CLI или TUI, URL должен выглядеть так:

    ```
    openclaw tui --url ws://<host>:18789 --token <token>
    ```

    Подробности протокола: [протокол Gateway](/ru/gateway/protocol).

  </Accordion>
</AccordionGroup>

## Логирование и отладка

<AccordionGroup>
  <Accordion title="Где находятся логи?">
    Файловые логи (структурированные):

    ```
    /tmp/openclaw/openclaw-YYYY-MM-DD.log
    ```

    Вы можете задать стабильный путь через `logging.file`. Уровень файлового журнала управляется `logging.level`. Подробность консольного вывода управляется `--verbose` и `logging.consoleLevel`.

    Самый быстрый просмотр журнала в реальном времени:

    ```bash
    openclaw logs --follow
    ```

    Журналы службы/супервизора (когда gateway работает через launchd/systemd):

    - stdout macOS launchd: `~/Library/Logs/openclaw/gateway.log` (профили используют `gateway-<profile>.log`; stderr подавляется)
    - Linux: `journalctl --user -u openclaw-gateway[-<profile>].service -n 200 --no-pager`
    - Windows: `schtasks /Query /TN "OpenClaw Gateway (<profile>)" /V /FO LIST`

    Подробнее см. [Устранение неполадок](/ru/gateway/troubleshooting).

  </Accordion>

  <Accordion title="Как запустить/остановить/перезапустить службу Gateway?">
    Используйте вспомогательные команды gateway:

    ```bash
    openclaw gateway status
    openclaw gateway restart
    ```

    Если вы запускаете gateway вручную, `openclaw gateway --force` может вернуть порт. См. [Gateway](/ru/gateway).

  </Accordion>

  <Accordion title="Я закрыл терминал в Windows — как перезапустить OpenClaw?">
    Есть **три режима установки Windows**:

    **1) Локальная настройка Windows Hub:** нативное приложение управляет локальным Gateway в WSL, принадлежащим приложению.

    Откройте **OpenClaw Companion** из меню «Пуск» или трея, затем используйте
    **Gateway Setup** или вкладку Connections.

    **2) Ручной WSL2 Gateway:** Gateway работает внутри Linux.

    Откройте PowerShell, войдите в WSL, затем перезапустите:

    ```powershell
    wsl
    openclaw gateway status
    openclaw gateway restart
    ```

    Если вы никогда не устанавливали службу, запустите ее в foreground:

    ```bash
    openclaw gateway run
    ```

    **3) Нативный Windows CLI/Gateway:** Gateway работает напрямую в Windows.

    Откройте PowerShell и выполните:

    ```powershell
    openclaw gateway status
    openclaw gateway restart
    ```

    Если вы запускаете его вручную (без службы), используйте:

    ```powershell
    openclaw gateway run
    ```

    Документация: [Windows](/ru/platforms/windows), [регламент службы Gateway](/ru/gateway).

  </Accordion>

  <Accordion title="Gateway запущен, но ответы не приходят. Что проверить?">

    openclaw channels status
    openclaw logs --follow

## Медиа и вложения

<AccordionGroup>
  <Accordion title="Мой skill создал изображение/PDF, но ничего не было отправлено">
    Исходящие вложения от агента должны использовать структурированные поля медиа, такие как `media`, `mediaUrl`, `path` или `filePath`. См. [настройку ассистента OpenClaw](/ru/start/openclaw) и [отправку агентом](/ru/tools/agent-send).

    Отправка через CLI:

    ```bash
    openclaw message send --target +15555550123 --message "Here you go" --media /path/to/file.png
    ```

    Также проверьте:

    - Целевой канал поддерживает исходящие медиа и не заблокирован allowlist.
    - Файл находится в пределах лимитов размера провайдера (изображения уменьшаются максимум до 2048px).
    - `tools.fs.workspaceOnly=true` ограничивает отправку локальных путей рабочей областью, temp/media-store и файлами, проверенными песочницей.
    - `tools.fs.workspaceOnly=false` позволяет структурированным отправкам локальных медиа использовать локальные файлы хоста, которые агент уже может читать, но только для медиа и безопасных типов документов (изображения, аудио, видео, PDF, документы Office и проверенные текстовые документы, такие как Markdown/MD, TXT, JSON, YAML и YML). Это не сканер секретов: доступный агенту `secret.txt` или `config.json` может быть прикреплен, если расширение и проверка содержимого совпадают. Храните чувствительные файлы вне путей, доступных агенту, или оставьте `tools.fs.workspaceOnly=true` для более строгой отправки локальных путей.

    См. [Изображения](/ru/nodes/images).

  </Accordion>
</AccordionGroup>

## Безопасность и контроль доступа

<AccordionGroup>
  <Accordion title="Безопасно ли открывать OpenClaw для входящих личных сообщений?">
    Считайте входящие личные сообщения недоверенным вводом. Значения по умолчанию разработаны для снижения риска:

    - Поведение по умолчанию в каналах с поддержкой личных сообщений — **сопряжение**:
      - Неизвестные отправители получают код сопряжения; бот не обрабатывает их сообщение.
      - Подтвердите с помощью: `openclaw pairing approve --channel <channel> [--account <id>] <code>`
      - Ожидающие запросы ограничены **3 на канал**; проверьте `openclaw pairing list --channel <channel> [--account <id>]`, если код не пришел.
    - Публичное открытие личных сообщений требует явного включения (`dmPolicy: "open"` и allowlist `"*"`).

    Запустите `openclaw doctor`, чтобы выявить рискованные политики личных сообщений.

  </Accordion>

  <Accordion title="Prompt injection опасен только для публичных ботов?">
    Нет. Prompt injection связан с **недоверенным содержимым**, а не только с тем, кто может отправлять личные сообщения боту.
    Если ваш ассистент читает внешний контент (веб-поиск/получение страниц, страницы браузера, письма,
    документы, вложения, вставленные журналы), этот контент может содержать инструкции, которые пытаются
    перехватить управление моделью. Это может произойти, даже если **вы единственный отправитель**.

    Наибольший риск возникает, когда включены инструменты: модель можно обманом заставить
    извлечь контекст или вызвать инструменты от вашего имени. Снизьте радиус воздействия:

    - используйте read-only или tool-disabled агента-"читателя" для суммаризации недоверенного содержимого
    - держите `web_search` / `web_fetch` / `browser` выключенными для агентов с включенными инструментами
    - считайте декодированный текст файлов/документов также недоверенным: OpenResponses
      `input_file` и извлечение из медиа-вложений оба оборачивают извлеченный текст
      явными маркерами границы внешнего содержимого вместо передачи сырого текста файла
    - используйте песочницу и строгие allowlist инструментов

    Подробности: [Безопасность](/ru/gateway/security).

  </Accordion>

  <Accordion title="OpenClaw менее безопасен, потому что использует TypeScript/Node вместо Rust/WASM?">
    Язык и runtime имеют значение, но они не являются главным риском для личного
    агента. Практические риски OpenClaw — это открытость gateway, кто может писать
    боту, prompt injection, область действия инструментов, обработка учетных данных, доступ браузера, доступ exec
    и доверие к сторонним skill или plugin.

    Rust и WASM могут обеспечить более сильную изоляцию для некоторых классов кода, но
    они не решают prompt injection, плохие allowlist, публичную открытость gateway,
    чрезмерно широкие инструменты или профиль браузера, который уже вошел в чувствительные
    аккаунты. Считайте это основными средствами контроля:

    - держите Gateway приватным или аутентифицированным
    - используйте сопряжение и allowlist для личных сообщений и групп
    - запрещайте или изолируйте рискованные инструменты для недоверенного ввода
    - устанавливайте только доверенные plugins и skills
    - запускайте `openclaw security audit --deep` после изменений конфигурации

    Подробности: [Безопасность](/ru/gateway/security), [Песочница](/ru/gateway/sandboxing).

  </Accordion>

  <Accordion title="Я видел сообщения об открытых экземплярах OpenClaw. Что проверить?">
    Сначала проверьте свое фактическое развертывание:

    ```bash
    openclaw security audit --deep
    openclaw gateway status
    ```

    Более безопасная базовая конфигурация:

    - Gateway привязан к `loopback` или открыт только через аутентифицированный приватный
      доступ, например tailnet, SSH-туннель, аутентификацию токеном/паролем или корректно
      настроенный доверенный прокси
    - личные сообщения в режиме `pairing` или `allowlist`
    - группы находятся в allowlist и требуют упоминания, если не все участники доверенные
    - инструменты высокого риска (`exec`, `browser`, `gateway`, `cron`) запрещены или строго
      ограничены для агентов, читающих недоверенное содержимое
    - песочница включена там, где выполнению инструментов нужен меньший радиус воздействия

    Публичные привязки без auth, открытые личные сообщения/группы с инструментами и открытое управление
    браузером — находки, которые нужно исправить первыми. Подробности:
    [контрольный список аудита безопасности](/ru/gateway/security#security-audit-checklist).

  </Accordion>

  <Accordion title="Безопасно ли устанавливать Skills из ClawHub и сторонние plugins?">
    Считайте сторонние skills и plugins кодом, которому вы решаете доверять.
    Страницы skills в ClawHub показывают состояние сканирования перед установкой, но сканирование не является
    полноценной границей безопасности. OpenClaw не запускает встроенную локальную
    блокировку опасного кода во время установки/обновления plugin или skill; используйте
    управляемую оператором `security.installPolicy` для локальных решений allow/block.

    Более безопасный шаблон:

    - предпочитайте доверенных авторов и закрепленные версии
    - прочитайте skill или plugin перед включением
    - держите allowlist plugin и skill узкими
    - запускайте workflows с недоверенным вводом в песочнице с минимальным набором инструментов
    - избегайте предоставления стороннему коду широкого доступа к файловой системе, exec, браузеру или секретам

    Подробнее: [Skills](/ru/tools/skills), [Plugins](/ru/tools/plugin),
    [Безопасность](/ru/gateway/security).

  </Accordion>

  <Accordion title="Нужны ли моему боту отдельная электронная почта, аккаунт GitHub или номер телефона?">
    Да, для большинства настроек. Изоляция бота с помощью отдельных аккаунтов и номеров телефона
    уменьшает радиус поражения, если что-то пойдет не так. Это также упрощает ротацию
    учетных данных или отзыв доступа без влияния на ваши личные аккаунты.

    Начинайте с малого. Предоставляйте доступ только к тем инструментам и аккаунтам, которые действительно нужны, и расширяйте
    его позже, если потребуется.

    Документация: [Безопасность](/ru/gateway/security), [Сопряжение](/ru/channels/pairing).

  </Accordion>

  <Accordion title="Можно ли дать ему автономный доступ к моим текстовым сообщениям и безопасно ли это?">
    Мы **не** рекомендуем полную автономность для ваших личных сообщений. Самый безопасный шаблон:

    - Держите личные сообщения в **режиме сопряжения** или строгом списке разрешений.
    - Используйте **отдельный номер или аккаунт**, если хотите, чтобы он отправлял сообщения от вашего имени.
    - Позвольте ему подготовить черновик, затем **подтверждайте перед отправкой**.

    Если хотите экспериментировать, делайте это в выделенном аккаунте и держите его изолированным. См.
    [Безопасность](/ru/gateway/security).

  </Accordion>

  <Accordion title="Можно ли использовать более дешевые модели для задач личного ассистента?">
    Да, **если** агент работает только в чате, а входные данные доверенные. Меньшие уровни
    более подвержены перехвату инструкций, поэтому избегайте их для агентов с включенными инструментами
    или при чтении недоверенного контента. Если вам нужно использовать меньшую модель, жестко ограничьте
    инструменты и запускайте ее в песочнице. См. [Безопасность](/ru/gateway/security).
  </Accordion>

  <Accordion title="Я выполнил /start в Telegram, но не получил код сопряжения">
    Коды сопряжения отправляются **только**, когда неизвестный отправитель пишет боту и
    включен `dmPolicy: "pairing"`. `/start` сам по себе не создает код.

    Проверьте ожидающие запросы:

    ```bash
    openclaw pairing list telegram
    ```

    Если вам нужен немедленный доступ, добавьте свой идентификатор отправителя в список разрешений или задайте `dmPolicy: "open"`
    для этого аккаунта.

  </Accordion>

  <Accordion title="WhatsApp: будет ли он писать моим контактам? Как работает сопряжение?">
    Нет. Политика личных сообщений WhatsApp по умолчанию — **сопряжение**. Неизвестные отправители получают только код сопряжения, а их сообщение **не обрабатывается**. OpenClaw отвечает только в чаты, из которых получает сообщения, или на явные отправки, которые запускаете вы.

    Подтвердите сопряжение с помощью:

    ```bash
    openclaw pairing approve whatsapp <code>
    ```

    Список ожидающих запросов:

    ```bash
    openclaw pairing list whatsapp
    ```

    Запрос номера телефона в мастере: он используется для настройки вашего **списка разрешений/владельца**, чтобы ваши собственные личные сообщения были разрешены. Он не используется для автоматической отправки. Если вы запускаете на своем личном номере WhatsApp, используйте этот номер и включите `channels.whatsapp.selfChatMode`.

  </Accordion>
</AccordionGroup>

## Команды чата, прерывание задач и «он не останавливается»

<AccordionGroup>
  <Accordion title="Как отключить показ внутренних системных сообщений в чате?">
    Большинство внутренних сообщений или сообщений инструментов появляются только тогда, когда для этого сеанса включены
    **verbose**, **trace** или **reasoning**.

    Исправьте это в чате, где вы их видите:

    ```
    /verbose off
    /trace off
    /reasoning off
    ```

    Если шума все еще много, проверьте настройки сеанса в Control UI и задайте для verbose
    значение **inherit**. Также убедитесь, что вы не используете профиль бота, где в конфигурации `verboseDefault` задан
    как `on`.

    Документация: [Thinking и verbose](/ru/tools/thinking), [Безопасность](/ru/gateway/security/index#reasoning-and-verbose-output-in-groups).

  </Accordion>

  <Accordion title="Как остановить/отменить выполняющуюся задачу?">
    Отправьте любой из этих вариантов **отдельным сообщением** (без слеша):

    ```
    stop
    stop action
    stop current action
    stop run
    stop current run
    stop agent
    stop the agent
    stop openclaw
    openclaw stop
    stop don't do anything
    stop do not do anything
    stop doing anything
    please stop
    stop please
    abort
    esc
    wait
    exit
    interrupt
    ```

    Это триггеры прерывания (не слеш-команды).

    Для фоновых процессов (из инструмента exec) можно попросить агента выполнить:

    ```
    process action:kill sessionId:XXX
    ```

    Обзор слеш-команд: см. [Слеш-команды](/ru/tools/slash-commands).

    Большинство команд нужно отправлять как **отдельное** сообщение, начинающееся с `/`, но несколько сокращений (например, `/status`) также работают внутри строки для отправителей из списка разрешений.

  </Accordion>

  <Accordion title='Как отправить сообщение Discord из Telegram? ("Cross-context messaging denied")'>
    OpenClaw по умолчанию блокирует обмен сообщениями **между провайдерами**. Если вызов инструмента привязан
    к Telegram, он не отправит сообщение в Discord, если вы явно это не разрешите.

    Включите обмен сообщениями между провайдерами для агента:

    ```json5
    {
      tools: {
        message: {
          crossContext: {
            allowAcrossProviders: true,
            marker: { enabled: true, prefix: "[from {channel}] " },
          },
        },
      },
    }
    ```

    Перезапустите Gateway после редактирования конфигурации.

  </Accordion>

  <Accordion title='Почему кажется, что бот "игнорирует" быстрые сообщения одно за другим?'>
    Подсказки во время выполнения по умолчанию направляются в активный запуск. Используйте `/queue`, чтобы выбрать поведение активного запуска:

    - `steer` - направлять активный запуск на следующей границе модели
    - `followup` - поставить сообщения в очередь и выполнять их по одному после завершения текущего запуска
    - `collect` - поставить совместимые сообщения в очередь и ответить один раз после завершения текущего запуска
    - `interrupt` - прервать текущий запуск и начать заново

    Режим по умолчанию — `steer`. Для режимов очереди можно добавить параметры вроде `debounce:0.5s cap:25 drop:summarize`. См. [Очередь команд](/ru/concepts/queue) и [Очередь управления](/ru/concepts/queue-steering).

  </Accordion>
</AccordionGroup>

## Разное

<AccordionGroup>
  <Accordion title='Какая модель по умолчанию используется для Anthropic с API-ключом?'>
    В OpenClaw учетные данные и выбор модели разделены. Установка `ANTHROPIC_API_KEY` (или сохранение API-ключа Anthropic в профилях аутентификации) включает аутентификацию, но фактическая модель по умолчанию — та, которую вы настроите в `agents.defaults.model.primary` (например, `anthropic/claude-sonnet-4-6` или `anthropic/claude-opus-4-6`). Если вы видите `No credentials found for profile "anthropic:default"`, это означает, что Gateway не смог найти учетные данные Anthropic в ожидаемом `auth-profiles.json` для выполняющегося агента.
  </Accordion>
</AccordionGroup>
