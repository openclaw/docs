Вспомогательные средства отладки потокового вывода, итеративной разработки Gateway и профилирования запуска.

## Переопределения времени выполнения для отладки

`/debug` задаёт переопределения конфигурации **только для времени выполнения** (в памяти, а не на диске). По умолчанию отключено; включите с помощью `commands.debug: true`.

```text
/debug show
/debug set messages.responsePrefix="[openclaw]"
/debug unset messages.responsePrefix
/debug reset
```

`/debug reset` очищает все переопределения и возвращает конфигурацию, сохранённую на диске.

## Вывод трассировки сеанса

`/trace` показывает принадлежащие плагину строки трассировки и отладки для одного сеанса без включения полного подробного режима. Используйте эту команду для диагностики плагинов, например для отладочных сводок Active Memory; для обычного вывода состояния и инструментов используйте `/verbose`.

```text
/trace
/trace on
/trace off
```

## Трассировка жизненного цикла Plugin

Установите `OPENCLAW_PLUGIN_LIFECYCLE_TRACE=1`, чтобы получить поэтапную разбивку работы с метаданными плагинов, обнаружением, реестром, зеркалом времени выполнения, изменением конфигурации и обновлением. Вывод направляется в stderr, поэтому результат команд в формате JSON остаётся пригодным для разбора.

```bash
OPENCLAW_PLUGIN_LIFECYCLE_TRACE=1 openclaw plugins install tokenjuice --force
```

```text
[plugins:lifecycle] phase="config read" ms=6.83 status=ok command="install"
[plugins:lifecycle] phase="slot selection" ms=94.31 status=ok command="install" pluginId="tokenjuice"
[plugins:lifecycle] phase="registry refresh" ms=51.56 status=ok command="install" reason="source-changed"
```

Используйте это средство, прежде чем прибегать к профилировщику ЦП. При работе с исходным кодом измеряйте собранную среду выполнения командой `node dist/entry.js ...` после `pnpm build`; команда `pnpm openclaw ...` также учитывает накладные расходы средства запуска исходного кода.

## Профилирование запуска CLI и команд

Включённые в репозиторий тесты производительности запуска:

```bash
pnpm test:startup:bench:smoke
pnpm tsx scripts/bench-cli-startup.ts --preset real --case status --runs 3
pnpm tsx scripts/bench-cli-startup.ts --preset real --cpu-prof-dir .artifacts/cli-cpu
```

Для однократного профилирования через обычное средство запуска исходного кода задайте `OPENCLAW_RUN_NODE_CPU_PROF_DIR`:

```bash
OPENCLAW_RUN_NODE_CPU_PROF_DIR=.artifacts/cli-cpu pnpm openclaw status
```

Средство запуска исходного кода добавляет флаги профилирования ЦП Node и записывает для команды файл `.cpuprofile`. Используйте это средство, прежде чем добавлять временное инструментирование в код команды.

Если задержки запуска похожи на результат синхронной работы файловой системы или загрузчика модулей, добавьте флаг трассировки синхронного ввода-вывода Node через средство запуска исходного кода:

```bash
OPENCLAW_TRACE_SYNC_IO=1 pnpm openclaw gateway --force
```

По умолчанию `pnpm gateway:watch` оставляет этот флаг отключённым для отслеживаемого дочернего процесса Gateway; задайте `OPENCLAW_TRACE_SYNC_IO=1`, если вывод трассировки синхронного ввода-вывода также нужен в режиме наблюдения.

## Режим наблюдения Gateway

```bash
pnpm gateway:watch
```

По умолчанию команда запускает или перезапускает сеанс tmux с именем `openclaw-gateway-watch-<profile>` (например, `openclaw-gateway-watch-main`), добавляя суффикс порта, такой как `openclaw-gateway-watch-dev-19001`, только когда `OPENCLAW_GATEWAY_PORT` отличается от порта по умолчанию `18789`. В интерактивных терминалах подключение выполняется автоматически; неинтерактивные оболочки, CI и вызовы выполнения агентов остаются отключёнными и вместо этого выводят инструкции по подключению:

```bash
tmux attach -t openclaw-gateway-watch-main
```

В панели tmux работает исходное средство наблюдения:

```bash
node scripts/watch-node.mjs gateway --force
```

Перед наблюдением за тем же портом остановите установленную службу Gateway:

```bash
pnpm openclaw gateway stop
```

Флаг `--force` средства наблюдения очищает текущий прослушивающий процесс, но не отключает службу под управлением диспетчера. В противном случае служба launchd, systemd или Scheduled Task может перезапуститься и заменить отслеживаемый Gateway.

Режим переднего плана без tmux:

```bash
pnpm gateway:watch:raw
# или
OPENCLAW_GATEWAY_WATCH_TMUX=0 pnpm gateway:watch
```

Чтобы сохранить управление через tmux, но отключить автоматическое подключение:

```bash
OPENCLAW_GATEWAY_WATCH_ATTACH=0 pnpm gateway:watch
```

Для профилирования процессорного времени отслеживаемого Gateway при отладке узких мест запуска или времени выполнения:

```bash
pnpm gateway:watch --benchmark
```

Обёртка наблюдения обрабатывает `--benchmark` перед вызовом Gateway и при каждом завершении дочернего процесса Gateway записывает один файл V8 `.cpuprofile` в `.artifacts/gateway-watch-profiles/`. Остановите или перезапустите отслеживаемый Gateway, чтобы сохранить текущий профиль, затем откройте его с помощью Chrome DevTools или Speedscope:

```bash
npx speedscope .artifacts/gateway-watch-profiles/*.cpuprofile
```

- `--benchmark-dir <path>`: записывать профили в другое место.
- `--benchmark-no-force`: пропустить выполняемую по умолчанию очистку порта с помощью `--force` и немедленно завершиться с ошибкой, если порт Gateway уже используется.

По умолчанию режим тестирования производительности подавляет избыточный вывод трассировки синхронного ввода-вывода. Задайте `OPENCLAW_TRACE_SYNC_IO=1` вместе с `--benchmark`, чтобы получать как профили ЦП, так и трассировки стека синхронного ввода-вывода; в режиме тестирования производительности эти блоки трассировки записываются в `gateway-watch-output.log` в каталоге теста производительности и отфильтровываются из панели терминала, а обычные журналы Gateway остаются видимыми.

Обёртка tmux передаёт в панель распространённые несекретные параметры выбора среды выполнения, включая `OPENCLAW_PROFILE`, `OPENCLAW_CONFIG_PATH`, `OPENCLAW_STATE_DIR`, `OPENCLAW_GATEWAY_PORT` и `OPENCLAW_SKIP_CHANNELS`. Сохраняйте учётные данные провайдера в обычном профиле или конфигурации либо используйте исходный режим переднего плана для разовых временных секретов.

Если отслеживаемый Gateway завершается во время запуска, средство наблюдения один раз выполняет `openclaw doctor --fix --non-interactive` и перезапускает дочерний процесс Gateway. Задайте `OPENCLAW_GATEWAY_WATCH_AUTO_DOCTOR=0`, чтобы увидеть исходную ошибку запуска без предназначенного только для разработки этапа исправления.

Управляемая панель tmux по умолчанию показывает цветные журналы Gateway; чтобы отключить вывод ANSI, задайте `FORCE_COLOR=0` при запуске `pnpm gateway:watch`.

Средство наблюдения перезапускается при изменении влияющих на сборку файлов в `src/`, исходных файлов расширений, метаданных расширений `package.json` и `openclaw.plugin.json`, а также файлов `tsconfig.json`, `package.json` и `tsdown.config.ts`. Изменения метаданных расширений перезапускают Gateway без принудительной пересборки; при изменениях исходного кода и конфигурации сначала по-прежнему пересобирается `dist`.

Добавьте флаги CLI Gateway после `gateway:watch`, и они будут передаваться при каждом перезапуске. Повторный запуск той же команды наблюдения заново создаёт именованную панель tmux; исходное средство наблюдения использует блокировку единственного экземпляра, поэтому дублирующиеся родительские процессы заменяются, а не накапливаются.

## Профиль разработки и Gateway разработки (--dev)

Существуют два **отдельных** флага `--dev`:

- **Глобальный `--dev` (профиль):** изолирует состояние в `~/.openclaw-dev` и задаёт для порта Gateway значение по умолчанию `19001` (производные порты смещаются вместе с ним).
- **`gateway --dev`:** указывает Gateway автоматически создать конфигурацию и рабочее пространство по умолчанию, если они отсутствуют, и пропустить начальную настройку.

Рекомендуемый порядок действий (профиль разработки и начальная настройка разработки):

```bash
pnpm gateway:dev
OPENCLAW_PROFILE=dev openclaw tui
```

Если глобальная установка отсутствует, запускайте CLI через `pnpm openclaw ...`.

В результате выполняется следующее:

1. **Изоляция профиля** (глобальный `--dev`)
   - `OPENCLAW_PROFILE=dev`
   - `OPENCLAW_STATE_DIR=~/.openclaw-dev`
   - `OPENCLAW_CONFIG_PATH=~/.openclaw-dev/openclaw.json`
   - `OPENCLAW_GATEWAY_PORT=19001` (порты браузера и холста соответственно смещаются)

2. **Начальная настройка разработки** (`gateway --dev`)
   - Записывает минимальную конфигурацию, если она отсутствует (`gateway.mode=local`, привязка к loopback).
   - Задаёт для `agents.defaults.workspace` рабочее пространство разработки, а для `agents.defaults.skipBootstrap` — значение `true`.
   - Создаёт в рабочем пространстве отсутствующие файлы: `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`.
   - Идентификатор по умолчанию: **C3-PO** (протокольный дроид).
   - `pnpm gateway:dev` также задаёт `OPENCLAW_SKIP_CHANNELS=1`, чтобы пропустить провайдеры каналов.

Порядок сброса (чистый запуск):

```bash
pnpm gateway:dev:reset
```

<Note>
`--dev` — это **глобальный** флаг профиля, который перехватывают некоторые средства запуска. Если необходимо указать его явно, используйте переменную среды:

```bash
OPENCLAW_PROFILE=dev openclaw gateway --dev --reset
```

</Note>

`--reset` очищает конфигурацию, учётные данные, сеансы и рабочее пространство разработки (перемещает в корзину, а не удаляет), а затем заново создаёт стандартную среду разработки.

<Tip>
Если уже запущен Gateway не для разработки (через launchd или systemd), сначала остановите его:

```bash
openclaw gateway stop
```

</Tip>

## Журналирование исходного потока

OpenClaw может журналировать **исходный поток ассистента** до любой фильтрации или форматирования. Это лучший способ определить, поступают ли рассуждения в виде приращений обычного текста или отдельных блоков размышлений.

Включение через CLI:

```bash
pnpm gateway:watch --raw-stream
```

Необязательное переопределение пути:

```bash
pnpm gateway:watch --raw-stream --raw-stream-path ~/.openclaw/logs/raw-stream.jsonl
```

Эквивалентные переменные среды:

```bash
OPENCLAW_RAW_STREAM=1
OPENCLAW_RAW_STREAM_PATH=~/.openclaw/logs/raw-stream.jsonl
```

Файл по умолчанию: `~/.openclaw/logs/raw-stream.jsonl`

## Примечания по безопасности

- Журналы исходного потока могут содержать полные запросы, вывод инструментов и пользовательские данные.
- Храните журналы локально и удаляйте их после завершения отладки.
- Перед передачей журналов удалите из них секреты и персональные данные.

## Отладка в VSCode

Карты исходного кода обязательны, поскольку при сборке хешируются имена создаваемых файлов. Включённый файл `launch.json` предназначен для службы Gateway:

1. **Rebuild and Debug Gateway** — удаляет `/dist` и перед запуском Gateway выполняет повторную сборку с включённой отладкой.
2. **Debug Gateway** — отлаживает существующую сборку, не изменяя `/dist`.

### Настройка

1. Откройте **Run and Debug** (на панели действий или с помощью `Ctrl`+`Shift`+`D`).
2. Выберите **Rebuild and Debug Gateway** и нажмите **Start Debugging**.

Чтобы вместо этого управлять циклом сборки и отладки вручную:

1. Включите карты исходного кода в терминале:
   - **Linux/macOS**: `export OUTPUT_SOURCE_MAPS=1`
   - **Windows (PowerShell)**: `$env:OUTPUT_SOURCE_MAPS="1"`
   - **Windows (CMD)**: `set OUTPUT_SOURCE_MAPS=1`
2. Выполните повторную сборку: `pnpm clean:dist && pnpm build`
3. Выберите **Debug Gateway** и нажмите **Start Debugging**.

Устанавливайте точки останова в файлах TypeScript в `src/`; отладчик сопоставляет их со скомпилированным JavaScript с помощью карт исходного кода.

### Примечания

- **Rebuild and Debug Gateway** удаляет `/dist` и при каждом запуске выполняет полную команду `pnpm build` с картами исходного кода.
- **Debug Gateway** можно запускать и останавливать, не затрагивая `/dist`, но циклом сборки необходимо управлять в отдельном терминале.
- Измените `args` в `launch.json`, чтобы отлаживать другие подкоманды CLI.
- Чтобы использовать собранный CLI для других задач (например, `dashboard --no-open`, если сеанс отладки создаёт новый токен аутентификации), запустите его из другого терминала: `node ./openclaw.mjs` или через псевдоним, например `alias openclaw-build="node $(pwd)/openclaw.mjs"`.
