Контрольный список для проверки обновления и плагинов: подтвердите, что устанавливаемый пакет может
обновлять реальное состояние пользователя, исправлять устаревшее состояние через `doctor`, а также
по-прежнему устанавливать, загружать, обновлять и удалять плагины из всех поддерживаемых источников.

Общую схему средств запуска тестов см. в разделе [Тестирование](/ru/help/testing). Информацию о ключах
реальных провайдеров и наборах тестов, обращающихся к сети, см. в разделе [Тестирование в реальной среде](/ru/help/testing-live).

## Что мы защищаем

- Архив пакета должен быть полным, содержать корректный файл `dist/postinstall-inventory.json`
  и не зависеть от распакованных файлов репозитория.
- Пользователь должен иметь возможность перейти со старой опубликованной версии пакета на пакет-кандидат
  без потери конфигурации, агентов, сеансов, рабочих пространств, списков разрешённых плагинов и
  конфигурации каналов.
- `openclaw doctor --fix --non-interactive` отвечает за очистку и исправление
  устаревших данных. При запуске не должны появляться скрытые миграции совместимости
  для устаревшего состояния плагинов.
- Установка плагинов должна работать из локальных каталогов, репозиториев Git, пакетов npm и
  через реестр ClawHub.
- Зависимости npm плагина устанавливаются в одном управляемом проекте npm для каждого плагина,
  проверяются перед установлением доверия и удаляются командой `npm uninstall` при
  удалении плагина, чтобы поднятые на верхний уровень зависимости не оставались в системе.
- Если ничего не изменилось, обновление плагина не должно выполнять никаких действий: записи об установке, разрешённый
  источник, структура установленных зависимостей и состояние включения должны оставаться неизменными.

## Локальная проверка во время разработки

Начинайте с узкой области:

```bash
pnpm changed:lanes --json
pnpm check:changed
pnpm test:changed
```

При изменении установки или удаления плагинов, зависимостей либо состава пакета также
запустите целевые тесты, охватывающие изменённую границу:

```bash
pnpm test src/plugins/uninstall.test.ts src/infra/package-dist-inventory.test.ts test/scripts/package-acceptance-workflow.test.ts
```

Прежде чем любой пакетный тест Docker начнёт использовать архив, проверьте артефакт пакета:

```bash
pnpm release:check
```

`release:check` выполняет проверки расхождений конфигурации, документации и API (схема конфигурации, базовый снимок
документации по конфигурации, базовый снимок API и экспорта SDK плагинов, версии и состав плагинов),
создаёт перечень содержимого дистрибутива пакета, выполняет `npm pack --dry-run`, отклоняет запрещённые
файлы в пакете, устанавливает архив во временный префикс, выполняет действия после установки и
проводит дымовую проверку точек входа встроенных каналов.

## Тесты Docker

Тесты Docker обеспечивают проверку на уровне продукта. Они устанавливают или обновляют реальный
пакет внутри контейнеров Linux и проверяют поведение с помощью команд CLI,
запуска Gateway, HTTP-проб, состояния RPC и состояния файловой системы.

Во время итеративной разработки используйте целевые тесты:

```bash
pnpm test:docker:plugins
pnpm test:docker:plugin-lifecycle-matrix
pnpm test:docker:plugin-update
pnpm test:docker:upgrade-survivor
pnpm test:docker:published-upgrade-survivor
pnpm test:docker:update-restart-auth
pnpm test:docker:update-migration
```

Важные тесты:

- `test:docker:plugins` охватывает дымовую проверку установки плагинов, установку из локальных каталогов,
  пропуск обновления локальных каталогов, локальные каталоги с предварительно установленными
  зависимостями, установку пакетов `file:`, установку из Git с выполнением CLI, обновление
  перемещаемых ссылок Git, установку из реестра npm с поднятыми транзитивными
  зависимостями, отсутствие действий при обновлении npm, отклонение некорректных метаданных пакета npm,
  установку из локальной тестовой среды ClawHub и отсутствие действий при обновлении, поведение при обновлении из каталога,
  а также включение и проверку пакета Claude. Задайте `OPENCLAW_PLUGINS_E2E_CLAWHUB=0`, чтобы
  блок ClawHub оставался герметичным и работал без сети.
- `test:docker:plugin-lifecycle-matrix` устанавливает пакет-кандидат в пустом
  контейнере и проводит плагин npm через установку, проверку, отключение, включение,
  явное обновление, явный откат версии и удаление после удаления кода плагина.
  Для каждого этапа записываются метрики RSS и ЦП.
- `test:docker:plugin-update` проверяет, что неизменившийся установленный плагин
  не переустанавливается и не теряет метаданные установки во время `openclaw plugins update`.
- `test:docker:upgrade-survivor` устанавливает архив-кандидат поверх тестового состояния
  старого пользователя с накопленными изменениями, выполняет обновление пакета и запуск doctor без взаимодействия с пользователем, затем запускает
  Gateway через local loopback и проверяет сохранность состояния.
- `test:docker:published-upgrade-survivor` сначала устанавливает опубликованную базовую версию,
  настраивает её с помощью встроенного сценария `openclaw config set`, обновляет до
  архива-кандидата, запускает doctor, проверяет очистку устаревших данных, запускает Gateway и
  проверяет `/healthz`, `/readyz` и состояние RPC.
- `test:docker:update-restart-auth` устанавливает пакет-кандидат, запускает
  управляемый Gateway с аутентификацией по токену, удаляет из окружения вызывающей стороны переменные аутентификации Gateway для
  `openclaw update --yes --json` и требует, чтобы команда обновления кандидата
  перезапустила Gateway до выполнения обычных проверок.
- `test:docker:update-migration` — тест обновления опубликованной версии с усиленной очисткой. Он
  начинает с настроенного пользовательского состояния в стиле Discord/Telegram, запускает doctor
  базовой версии, чтобы зависимости настроенных плагинов могли материализоваться, создаёт
  остатки устаревших зависимостей для настроенного пакетного плагина, обновляет до
  архива-кандидата и требует, чтобы doctor после обновления удалил корневые каталоги устаревших
  зависимостей.

Полезные варианты теста обновления опубликованной версии:

```bash
OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPEC=openclaw@2026.4.23 \
OPENCLAW_UPGRADE_SURVIVOR_SCENARIO=versioned-runtime-deps \
pnpm test:docker:published-upgrade-survivor

OPENCLAW_UPGRADE_SURVIVOR_BASELINE_SPEC=openclaw@latest \
OPENCLAW_UPGRADE_SURVIVOR_SCENARIO=bootstrap-persona \
pnpm test:docker:published-upgrade-survivor
```

Доступные сценарии: `base`, `acpx-openclaw-tools-bridge`, `feishu-channel`,
`bootstrap-persona`, `channel-post-core-restore`, `plugin-deps-cleanup`,
`configured-plugin-installs`, `stale-source-plugin-shadow`, `tilde-log-path`
и `versioned-runtime-deps`. При совокупных запусках `OPENCLAW_UPGRADE_SURVIVOR_SCENARIOS=reported-issues`
(псевдоним `far-reaching`) разворачивается во все сценарии, включая миграцию
установки настроенных плагинов.

Полная миграция обновлений намеренно отделена от полной проверки выпуска в CI. Используйте
запускаемый вручную процесс `Update Migration`, когда вопрос о выпуске звучит так: «может ли каждая
опубликованная стабильная версия, начиная с 2026.4.23, обновиться до этого кандидата и
очистить остатки зависимостей плагинов?»:

```bash
gh workflow run update-migration.yml \
  --ref main \
  -f workflow_ref=main \
  -f package_ref=main \
  -f baselines=all-since-2026.4.23 \
  -f scenarios=plugin-deps-cleanup
```

## Приёмка пакета

Приёмка пакета — это встроенный в GitHub контроль пакета. Он преобразует один пакет-кандидат
в архив `package-under-test`, записывает версию и SHA-256, а затем
запускает повторно используемые сквозные тесты Docker для этого точного архива. Ссылка на
среду процесса отделена от ссылки на источник пакета, поэтому текущая логика тестирования может проверять
старые доверенные выпуски.

Источники кандидата:

- `source=npm`: проверяет `openclaw@extended-stable`, `openclaw@beta`,
  `openclaw@latest` или точную опубликованную версию.
- `source=ref`: упаковывает доверенную ветку, тег или коммит с выбранной текущей
  тестовой средой.
- `source=url`: проверяет общедоступный архив HTTPS с обязательным `package_sha256`.
  Этот путь отклоняет учётные данные в URL, нестандартные порты HTTPS, частные или внутренние
  имена узлов либо результаты DNS/IP, диапазоны IP специального назначения и небезопасные перенаправления.
- `source=trusted-url`: проверяет архив HTTPS с обязательными
  `package_sha256` и `trusted_source_id` по принадлежащей сопровождающим политике
  в `.github/package-trusted-sources.json`. Используйте этот вариант для корпоративных или частных
  зеркал вместо ослабления `source=url` переключателем разрешения частных адресов на уровне входных данных.
  Аутентификация Bearer, если она настроена политикой, использует фиксированный
  секрет `OPENCLAW_TRUSTED_PACKAGE_TOKEN`.
- `source=artifact`: повторно использует архив, загруженный другим запуском Actions.

По умолчанию полная проверка выпуска использует `source=artifact`, собранный из
разрешённого SHA выпуска. Для проверки после публикации передайте
`package_acceptance_package_spec=openclaw@YYYY.M.PATCH`, чтобы та же матрица обновления
проверяла уже выпущенный пакет npm.

Проверки выпуска вызывают приёмку пакета со следующим набором тестов пакета, обновления, перезапуска и плагинов:

```text
doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor root-managed-vps-upgrade update-restart-auth plugins-offline plugin-update plugin-binding-command-escape
```

Если включена выдержка выпуска (принудительно включается для `release_profile=stable` и
`full`), также передаются:

```text
published_upgrade_survivor_baselines=last-stable-4 2026.4.23 2026.5.2 2026.4.15
published_upgrade_survivor_scenarios=reported-issues
telegram_mode=mock-openai
```

Это обеспечивает проверку миграции пакета, переключения канала обновлений, устойчивости к повреждённому управляемому плагину,
очистки устаревших зависимостей плагинов, работы плагинов без сети, поведения при обновлении плагинов
и контроля качества пакета Telegram на одном и том же разрешённом артефакте, не заставляя
стандартный контроль пакета выпуска проходить по всем опубликованным версиям.

`last-stable-4` разрешается в четыре последних стабильных выпуска OpenClaw,
опубликованных в npm. Приёмка пакета выпуска фиксирует `2026.4.23` как первую границу совместимости
обновления плагинов, `2026.5.2` как границу существенных изменений архитектуры плагинов и
`2026.4.15` как более старую базовую версию обновления опубликованного пакета из линейки 2026.4.1x; средство разрешения
удаляет дубликаты фиксированных версий, уже входящих в последние четыре. Для исчерпывающей проверки
миграции обновлений опубликованных версий используйте `all-since-2026.4.23` в отдельном процессе Update
Migration вместо полной проверки выпуска в CI. Вариант `release-history` остаётся
доступным для более широких выборочных ручных проверок, если также нужна устаревшая
опорная версия до указанной даты.

Когда выбрано несколько базовых версий теста обновления опубликованной версии, повторно используемый
процесс Docker выделяет для каждой базовой версии отдельное целевое задание исполнителя. Каждый
сегмент базовой версии по-прежнему запускает выбранный набор сценариев, но журналы и артефакты сохраняются
отдельно для каждой базовой версии, а общее время ограничивается самым медленным сегментом вместо одного большого
последовательного задания.

Для проверки кандидата перед выпуском запустите профиль пакета вручную:

```bash
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=package \
  -f published_upgrade_survivor_baselines="last-stable-4 2026.4.23 2026.5.2 2026.4.15" \
  -f published_upgrade_survivor_scenarios=reported-issues \
  -f telegram_mode=mock-openai
```

Для опубликованной контрольной версии с расширенной стабильностью задайте
`package_spec=openclaw@extended-stable`. Приёмка пакета преобразует этот
селектор в точный архив до запуска тестов Docker.

Используйте `suite_profile=product`, если проверка выпуска включает каналы MCP,
очистку cron и субагентов, веб-поиск OpenAI или OpenWebUI. Используйте `suite_profile=full`
только тогда, когда требуется полное покрытие пути выпуска тестами Docker.

## Стандартная проверка выпуска

Для кандидатов на выпуск стандартный набор проверок выглядит так:

1. `pnpm check:changed` и `pnpm test:changed` для выявления регрессий на уровне исходного кода.
2. `pnpm release:check` для проверки целостности артефакта пакета.
3. Профиль `package` приёмки пакета или пользовательские пакетные тесты проверки выпуска
   для контрактов установки, обновления, перезапуска и плагинов.
4. Межплатформенные проверки выпуска для поведения установщика, первоначальной настройки и платформы,
   зависящего от ОС.
5. Тесты в реальной среде — только если изменённая область затрагивает поведение провайдера или размещённой
   службы.

На компьютерах сопровождающих широкие проверки и проверку продукта через Docker или пакет
следует выполнять в Testbox, если явно не проводится локальная проверка.

## Совместимость с устаревшими версиями

Допуски совместимости узки и ограничены по времени:

- Для пакетов до `2026.4.25` включительно, в том числе `2026.4.25-beta.*`, при приёмке пакета могут допускаться
  пробелы в метаданных уже выпущенных пакетов.
- Для опубликованного пакета `2026.4.26` может выводиться предупреждение о файлах отметки метаданных
  локальной сборки, уже включённых в выпуск.
- Более поздние пакеты должны соответствовать современным контрактам. Такие же несоответствия должны приводить к
  ошибке, а не предупреждению или пропуску проверки.

Не добавляйте новые миграции при запуске для этих старых форматов. Добавьте или расширьте исправление doctor,
а затем подтвердите его с помощью `upgrade-survivor`, `published-upgrade-survivor` или
`update-restart-auth`, если за перезапуск отвечает команда обновления.

## Добавление покрытия

При изменении поведения обновлений или плагинов добавляйте покрытие на самом низком уровне,
который может завершиться ошибкой по нужной причине:

- Чистая логика путей или метаданных: модульный тест рядом с исходным кодом.
- Состав пакета или поведение упакованных файлов: тест `package-dist-inventory` или проверка tarball.
- Поведение установки/обновления через CLI: проверка в Docker-сценарии или фикстура.
- Поведение миграции опубликованного выпуска: сценарий `published-upgrade-survivor`.
- Поведение перезапуска, инициируемого обновлением: `update-restart-auth`.
- Поведение источника реестра/пакета: фикстура `test:docker:plugins` или сервер фикстур ClawHub.
- Поведение структуры или очистки зависимостей: проверяйте как выполнение во время работы, так и границы файловой системы. Зависимости npm могут подниматься внутри управляемого npm-проекта Plugin, поэтому тесты должны подтверждать, что сканируется/очищается именно этот проект, а не исходить из предположения, что обрабатывается только локальное для пакета Plugin дерево `node_modules`.

По умолчанию обеспечивайте герметичность новых Docker-фикстур. Используйте локальные реестры фикстур и фиктивные пакеты, если только целью теста не является проверка поведения реального реестра.

## Разбор сбоев

Начните с идентификации артефакта:

- Сводка `resolve_package` проверки приемки пакета: источник, версия, SHA-256 и имя артефакта.
- Docker-артефакты: `.artifacts/docker-tests/**/summary.json`, `failures.json`, журналы сценариев и команды повторного запуска.
- Сводка проверки сохранения работоспособности после обновления: `.artifacts/upgrade-survivor/summary.json`, включая исходную версию, версию-кандидат, сценарий, длительность этапов и охват рецептов конфигурации.

Предпочитайте повторный запуск конкретного завершившегося сбоем сценария с тем же артефактом пакета повторному запуску всего комплекса проверок выпуска.
