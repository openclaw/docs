  <Accordion title="Я не могу продолжить: как быстрее всего решить проблему">
    Используйте локального ИИ-агента, который может **видеть ваш компьютер**. Большинство случаев «я не могу продолжить»
    вызваны **проблемами локальной конфигурации или окружения**, которые удалённый помощник не может проверить, поэтому этот способ эффективнее,
    чем обращение в Discord.

    - **Claude Code**: [https://www.anthropic.com/claude-code/](https://www.anthropic.com/claude-code/)
    - **OpenAI Codex**: [https://openai.com/codex/](https://openai.com/codex/)

    Предоставьте агенту полный исходный код через изменяемую установку (git), чтобы он мог изучить
    код и документацию и учесть точную используемую вами версию:

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method git
    ```

    Попросите агента пошагово спланировать и проконтролировать исправление, а затем выполнить только
    необходимые команды — небольшие изменения проще проверять.

    При обращении за помощью (в Discord или в задаче GitHub) предоставьте результаты следующих команд:

    | Команда | Что показывает |
    | --- | --- |
    | `openclaw status` | Состояние Gateway/агента и основной снимок конфигурации |
    | `openclaw status --all` | Полная диагностика только для чтения, которую можно вставить в сообщение |
    | `openclaw models status` | Аутентификация провайдера и доступность моделей |
    | `openclaw doctor` | Проверяет и исправляет распространённые проблемы конфигурации и состояния |
    | `openclaw logs --follow` | Поток журналов в реальном времени |
    | `openclaw gateway status --deep` | Углублённая проверка состояния Gateway, конфигурации и плагинов |
    | `openclaw health --verbose` | Подробный отчёт о состоянии |

    Нашли настоящую ошибку или исправление? Создайте задачу или отправьте PR:
    [Задачи](https://github.com/openclaw/openclaw/issues) /
    [Запросы на включение изменений](https://github.com/openclaw/openclaw/pulls).

    Быстрый цикл отладки: [Первые 60 секунд при возникновении неполадки](/ru/help/faq#first-60-seconds-if-something-is-broken).
    Документация по установке: [Установка](/ru/install), [Флаги установщика](/ru/install/installer), [Обновление](/ru/install/updating).

  <Accordion title="Heartbeat постоянно пропускается. Что означают причины пропуска?">
    | Причина пропуска | Значение |
    | --- | --- |
    | `quiet-hours` | За пределами настроенного периода активности |
    | `empty-heartbeat-file` | Файл `HEARTBEAT.md` существует, но содержит только пустые строки, комментарии, заголовки, ограждения блоков или заготовку пустого списка задач |
    | `no-tasks-due` | Режим задач активен, но время выполнения следующего интервала ещё не наступило |
    | `alerts-disabled` | Вся видимость Heartbeat отключена (`showOk`, `showAlerts` и `useIndicator` отключены) |

    В режиме задач сроки следующего выполнения сдвигаются только после завершения фактического запуска Heartbeat.
    Пропущенные запуски не отмечают задачи как выполненные.

    Документация: [Heartbeat](/ru/gateway/heartbeat), [Автоматизация](/ru/automation).

    openclaw onboard --install-daemon

    pnpm build
    pnpm ui:build
    openclaw onboard

  <Accordion title="Как пройти аутентификацию в панели управления локально и удалённо?">
    **Локальный компьютер (тот же компьютер):**

    - Откройте `http://127.0.0.1:18789/`.
    - Если запрашивается аутентификация с общим секретом, вставьте настроенный токен или пароль в настройки Control UI.
    - Источник токена: `gateway.auth.token` (или `OPENCLAW_GATEWAY_TOKEN`).
    - Источник пароля: `gateway.auth.password` (или `OPENCLAW_GATEWAY_PASSWORD`).
    - Общий секрет ещё не настроен? Запустите `openclaw doctor --generate-gateway-token` (или `openclaw doctor --fix --generate-gateway-token`).

    **Не на локальном компьютере:**

    - **Tailscale Serve** (рекомендуется): оставьте привязку к loopback-интерфейсу, запустите `openclaw gateway --tailscale serve`, откройте `https://<magicdns>/`. При `gateway.auth.allowTailscale: true` заголовки идентификации обеспечивают аутентификацию Control UI/WebSocket (вставлять общий секрет не требуется; предполагается доверенный узел Gateway); для HTTP API по-прежнему требуется аутентификация с общим секретом, если только вы намеренно не используете приватный входящий трафик `none` или HTTP-аутентификацию доверенного прокси.
      Одновременные попытки Serve с неверной аутентификацией от одного клиента выполняются последовательно, прежде чем ограничитель неудачной аутентификации зарегистрирует их, поэтому уже при второй неудачной попытке может отображаться `retry later`.
    - **Привязка к tailnet**: запустите `openclaw gateway --bind tailnet --token "<token>"` (или настройте аутентификацию по паролю), откройте `http://<tailscale-ip>:18789/`, вставьте соответствующий общий секрет в настройки панели управления.
    - **Обратный прокси с учётом идентификации**: оставьте Gateway за доверенным прокси, задайте `gateway.auth.mode: "trusted-proxy"`, откройте URL прокси. Для loopback-прокси на том же узле необходимо явно задать `gateway.auth.trustedProxy.allowLoopback: true`.
    - **Туннель SSH**: `ssh -N -L 18789:127.0.0.1:18789 user@gateway-host`, затем откройте `http://127.0.0.1:18789/`. В туннеле по-прежнему применяется аутентификация с общим секретом; при запросе вставьте настроенный токен или пароль.

    Сведения о режимах привязки и аутентификации см. в разделах [Панель управления](/ru/web/dashboard) и [Веб-интерфейсы](/ru/web).

  </Accordion>

  <Accordion title="Почему для подтверждений выполнения в чате существуют две конфигурации?">
    Они управляют разными уровнями:

    - `approvals.exec` — перенаправляет запросы подтверждения в чаты.
    - `channels.<channel>.execApprovals` — делает этот канал нативным клиентом подтверждения выполнения.

    Политика выполнения на узле всё равно остаётся фактическим шлюзом подтверждений; конфигурация чата управляет только тем,
    где появляются запросы и как пользователи отвечают на них.

    Обычно обе конфигурации не нужны:

    - Если чат уже поддерживает команды и ответы, `/approve` в том же чате работает через общий механизм.
    - Когда поддерживаемый нативный канал может безопасно определить подтверждающих пользователей, OpenClaw автоматически включает нативные подтверждения с приоритетом личных сообщений, если `channels.<channel>.execApprovals.enabled` не задано или равно `"auto"`.
    - Когда доступны нативные карточки или кнопки подтверждения, этот интерфейс используется в первую очередь; упоминайте ручную команду `/approve` только в том случае, если результат инструмента сообщает, что подтверждения через чат недоступны.
    - Используйте `approvals.exec` только тогда, когда запросы также должны поступать в другие чаты или явно заданные операционные комнаты.
    - Используйте `channels.<channel>.execApprovals.target: "channel"` или `"both"` только тогда, когда запросы подтверждения нужно публиковать обратно в исходную комнату или тему.
    - Подтверждения плагинов обрабатываются отдельно: по умолчанию используется `/approve` в том же чате, можно включить перенаправление `approvals.plugin`, и лишь некоторые нативные каналы сохраняют нативную обработку и для таких подтверждений.

    Кратко: перенаправление отвечает за маршрутизацию, а конфигурация нативного клиента — за расширенный интерфейс с учётом особенностей канала.
    См. [Подтверждения выполнения](/ru/tools/exec-approvals).

  </Accordion>

    Bun может устанавливать зависимости и запускать скрипты пакетов, но не может запускать CLI OpenClaw или Gateway, поскольку в нём отсутствует `node:sqlite`.

  <Accordion title="Работает ли OpenClaw на Raspberry Pi?">
    Да, но сначала проверьте объём оперативной памяти: оптимальны Pi 5 и Pi 4 (2 GB+); Pi 3B+ (1 GB) работает, но медленно; Pi Zero 2 W (512 MB) не рекомендуется.

    | Модель | Оперативная память | Пригодность |
    | --- | --- | --- |
    | Pi 5 | 4/8 GB | Отлично |
    | Pi 4 | 4 GB | Хорошо |
    | Pi 4 | 2 GB | Приемлемо, добавьте подкачку |
    | Pi 4 | 1 GB | На пределе |
    | Pi 3B+ | 1 GB | Медленно |
    | Pi Zero 2 W | 512 MB | Не рекомендуется |

    Абсолютный минимум: 1 GB оперативной памяти, 1 ядро, 500 MB свободного места на диске, 64-разрядная ОС. Поскольку Pi запускает только
    Gateway (модели обращаются к облачным API), даже устройство Pi со скромными характеристиками справляется с нагрузкой.

    Небольшое устройство Pi или VPS также может размещать только Gateway, а **узлы**
    на ноутбуке или телефоне можно связать с ним для локальной работы с экраном, камерой, холстом или выполнения команд. См. [Узлы](/ru/nodes).

    Полное пошаговое руководство по настройке: [Raspberry Pi](/ru/install/raspberry-pi).

  </Accordion>

  <Accordion title="Есть ли рекомендации по установке на Raspberry Pi?">
    - Используйте **64-разрядную** ОС; не используйте 32-разрядную Raspberry Pi OS.
    - Добавьте подкачку на платах с 2 GB памяти или меньше.
    - Для производительности и долговечности предпочитайте **USB SSD**, а не SD-карту.
    - Предпочитайте изменяемую установку (git), чтобы просматривать журналы и быстро обновляться.
    - Начните без каналов и навыков, затем добавляйте их по одному.
    - Необычные ошибки двоичных файлов («exec format error») обычно означают, что для необязательного инструмента навыка отсутствует сборка ARM64.

    Полное руководство: [Raspberry Pi](/ru/install/raspberry-pi). См. также [Linux](/ru/platforms/linux).

  </Accordion>

  <Accordion title="Процесс завис на «wake up my friend» или первоначальная настройка не завершается. Что делать?">
    Работа этого экрана зависит от доступности Gateway и успешной аутентификации. TUI также автоматически отправляет
    «Проснись, мой друг!» при первом запуске, если настроен провайдер модели. Если
    вы пропустили настройку модели или аутентификации, процесс первоначальной настройки показывает уведомление «Аутентификация модели отсутствует» и открывает
    TUI, ничего не отправляя, — добавьте провайдера с помощью `openclaw configure --section model`.
    Если отображается строка пробуждения, но **ответа нет**, а количество токенов остаётся равным 0, агент не запускался.

    1. Перезапустите Gateway:

    ```bash
    openclaw gateway restart
    ```

    2. Проверьте состояние и аутентификацию:

    ```bash
    openclaw status
    openclaw models status
    openclaw logs --follow
    ```

    3. Всё ещё зависает? Выполните:

    ```bash
    openclaw doctor
    ```

    Если Gateway находится на удалённом компьютере, убедитесь, что туннель или подключение Tailscale активно, а интерфейс
    указывает на правильный Gateway. См. [Удалённый доступ](/ru/gateway/remote).

  </Accordion>

  <Accordion title="Можно ли перенести настройку на новый компьютер без повторного прохождения первоначальной настройки?">
    Да. Скопируйте **каталог состояния** и **рабочее пространство**, затем один раз запустите Doctor:

    1. Установите OpenClaw на новом компьютере.
    2. Скопируйте `$OPENCLAW_STATE_DIR` (по умолчанию: `~/.openclaw`) со старого компьютера.
    3. Скопируйте рабочее пространство (по умолчанию: `~/.openclaw/workspace`).
    4. Запустите `openclaw doctor` и перезапустите службу Gateway.

    При этом сохраняются конфигурация, профили аутентификации, учётные данные WhatsApp, сеансы и память —
    ваш бот остаётся точно таким же при условии, что вы скопировали **оба** расположения. В удалённом режиме
    хост Gateway является владельцем хранилища сеансов и рабочего пространства.

    **Важно:** если вы только фиксируете и отправляете рабочее пространство в GitHub, резервная копия включает
    **память и файлы начальной загрузки**, но не историю сеансов и данные аутентификации. Они находятся в
    `~/.openclaw/` (например, `~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite`).

    Связанные разделы: [Перенос](/ru/install/migrating), [Расположение данных на диске](/ru/help/faq#where-things-live-on-disk),
    [Рабочее пространство агента](/ru/concepts/agent-workspace), [Doctor](/ru/gateway/doctor),
    [Удалённый режим](/ru/gateway/remote).

  <Accordion title="Разница между стабильной и бета-версиями">
    **Стабильная** и **бета-версия** — это **dist-теги npm**, а не отдельные ветки кода:

    - `latest` = стабильная версия
    - `beta` = ранняя сборка для тестирования (используется `latest`, если бета-версия отсутствует или старше текущего стабильного выпуска)

    Стабильный выпуск обычно сначала попадает в канал **beta**, после чего на отдельном этапе продвижения
    эта же версия переносится в `latest` без изменения номера версии. Сопровождающие
    также могут опубликовать выпуск напрямую в `latest`. Поэтому после продвижения бета- и стабильный каналы могут указывать на
    **одну и ту же версию**.

    Список изменений: [CHANGELOG.md](https://github.com/openclaw/openclaw/blob/main/CHANGELOG.md).

    Однострочные команды установки и различия между beta и dev приведены в следующем раскрывающемся разделе.

  </Accordion>

  <Accordion title="Как установить бета-версию и чем beta отличается от dev?">
    **Beta** — это dist-тег npm `beta` (после продвижения может совпадать с `latest`).
    **Dev** — изменяющаяся вершина ветки `main` (git); при публикации в npm используется dist-тег `dev`.

    Однострочные команды (macOS/Linux):

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --beta
    ```

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method git
    ```

    Установщик для Windows (PowerShell): `iwr -useb https://openclaw.ai/install.ps1 | iex`

    Подробнее: [Каналы разработки](/ru/install/development-channels) и [Флаги установщика](/ru/install/installer).

  </Accordion>

  <Accordion title="Как опробовать самые свежие изменения?">
    Есть два варианта:

    1. **Канал dev (существующая установка):**

    ```bash
    openclaw update --channel dev
    ```

  <Accordion title="В выводе exec в Windows китайский текст отображается некорректно — что делать?">
    Обычно причина заключается в несовпадении кодовой страницы консоли в нативных оболочках Windows.

    Признаки: китайский текст в выводе `system.run`/`exec` отображается как нечитаемые символы, но та же команда
    выглядит нормально в другом профиле терминала.

    Обходное решение в PowerShell:

    ```powershell
    chcp 65001
    [Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
    [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
    $OutputEncoding = [System.Text.UTF8Encoding]::new($false)
    ```

    Затем перезапустите Gateway и повторите попытку:

    ```powershell
    openclaw gateway restart
    ```

    Проблема по-прежнему воспроизводится в последней версии OpenClaw? Следите за ней или сообщите подробности: [Задача #30640](https://github.com/openclaw/openclaw/issues/30640).

  <Accordion title="Почему Anthropic возвращает ошибку HTTP 429 rate_limit_error?">
    Ваша **квота Anthropic или ограничение частоты запросов** исчерпаны для текущего периода. При использовании **Claude
    CLI** дождитесь сброса периода или перейдите на более высокий тариф. При использовании **API-ключа Anthropic**
    проверьте использование и оплату в Anthropic Console и при необходимости увеличьте лимиты.

    Если сообщение имеет вид `Extra usage is required for long context requests`,
    запрос пытается использовать контекстное окно Anthropic размером 1M (модель Claude 4.x
    с поддержкой 1M, доступную на уровне GA, либо устаревшую конфигурацию `params.context1m: true`), а ваши текущие учётные данные
    не позволяют оплачивать длинный контекст.

    Настройте **резервную модель**, чтобы OpenClaw продолжал отвечать, пока провайдер ограничивает частоту запросов.
    См. [Модели](/ru/cli/models), [OAuth](/ru/concepts/oauth) и
    [Для длинного контекста при ошибке Anthropic 429 требуется дополнительное использование](/ru/gateway/troubleshooting#anthropic-429-extra-usage-required-for-long-context).

  </Accordion>

  <Accordion title="Поддерживается ли AWS Bedrock?">
    Да. В OpenClaw встроен провайдер **Amazon Bedrock (Converse)**. При наличии переменных

  <Accordion title="Как работает аутентификация Codex?">
    OpenClaw поддерживает **OpenAI Codex** через OAuth (вход в ChatGPT). При новой
    настройке без основной модели используется точно `openai/gpt-5.6-sol` для
    аутентификации по подписке ChatGPT/Codex и выполнения через нативный app-server Codex.
    При повторной аутентификации сохраняется явно заданная модель, включая
    `openai/gpt-5.5`. Если рабочее пространство Codex не предоставляет GPT-5.6, явно выберите
    `openai/gpt-5.5`; OpenClaw не выполняет неявный переход на менее новую модель. Устаревшие
    ссылки на модели с префиксом Codex относятся к устаревшей конфигурации, которую исправляет `openclaw doctor
    --fix`. Прямой доступ по API-ключу OpenAI остаётся доступным для неагентных
    поверхностей API OpenAI, а через упорядоченный профиль API-ключа `openai` — также для агентных
    моделей. См. [Провайдеры моделей](/ru/concepts/model-providers) и
    [Первоначальная настройка (CLI)](/ru/start/wizard).
  </Accordion>

  <Accordion title="Почему OpenClaw по-прежнему упоминает устаревший префикс OpenAI Codex?">
    `openai` — текущий идентификатор провайдера и профиля аутентификации как для API-ключей OpenAI, так и для
    OAuth ChatGPT/Codex: OpenAI Codex включён в него. В старых конфигурациях и предупреждениях
    миграции всё ещё может встречаться устаревший префикс `openai-codex`:

    - `openai/gpt-5.6-sol` = новая настройка подписки ChatGPT/Codex с нативной средой выполнения Codex для обращений агента.
    - `openai/gpt-5.5` = явно поддерживаемый выбор для существующей конфигурации или аккаунтов без доступа к GPT-5.6.
    - Устаревшие ссылки на модели `openai-codex/*` = устаревший маршрут, исправляемый `openclaw doctor --fix`.
    - `openai/gpt-5.5` вместе с упорядоченным профилем API-ключа `openai` = аутентификация по API-ключу для агентной модели OpenAI.
    - Устаревшие идентификаторы профилей аутентификации `openai-codex` = устаревшие идентификаторы, мигрируемые `openclaw doctor --fix`.

    Хотите использовать прямую оплату через OpenAI Platform? Задайте `OPENAI_API_KEY`. Хотите использовать аутентификацию
    по подписке ChatGPT/Codex? Выполните `openclaw models auth login --provider openai`. Храните
    ссылки на модели в каноническом провайдере `openai/*`. При новой настройке подписки
    используется точно `openai/gpt-5.6-sol`; doctor исправляет устаревшие ссылки с префиксом Codex,
    не обновляя явно выбранную модель `openai/gpt-5.5`.

  <Accordion title="Как настроить OAuth Gemini CLI?">
    Gemini CLI использует **процесс аутентификации плагина**, а не идентификатор или секрет клиента в `openclaw.json`.

    1. Установите Gemini CLI локально, чтобы `gemini` находился в `PATH`:
       - Homebrew: `brew install gemini-cli`
       - npm: `npm install -g @google/gemini-cli`
    2. Включите плагин: `openclaw plugins enable google`
    3. Войдите: `openclaw models auth login --provider google-gemini-cli --set-default`
    4. Модель по умолчанию после входа: `google/gemini-3.1-pro-preview` (среда выполнения `google-gemini-cli`)
    5. Запросы завершаются ошибкой после входа? Задайте `GOOGLE_CLOUD_PROJECT` или `GOOGLE_CLOUD_PROJECT_ID` на хосте Gateway и повторите попытку.

    Токены OAuth хранятся в профилях аутентификации на хосте Gateway. Подробнее: [Google](/ru/providers/google), [Провайдеры моделей](/ru/concepts/model-providers).

  </Accordion>

  <Accordion title="Подойдёт ли локальная модель для обычных бесед?">
    Обычно нет. OpenClaw требуется большой контекст и надёжная безопасность; видеокарты с небольшим объёмом памяти обрезают контекст
    и не используют фильтры безопасности на стороне провайдера. Если это необходимо, локально запускайте **самую крупную**
    доступную вам сборку модели (LM Studio) — см. [Локальные модели](/ru/gateway/local-models). Меньшие и квантованные
    модели повышают риск инъекций в промпт — см. [Безопасность](/ru/gateway/security).
  </Accordion>

  <Accordion title="Как сохранить трафик размещённой модели в определённом регионе?">
    Выбирайте конечные точки, привязанные к региону. OpenRouter предоставляет размещённые в США варианты MiniMax, Kimi
    и GLM; выберите вариант, размещённый в США, чтобы данные оставались в этом регионе. Вы по-прежнему можете указать
    Anthropic/OpenAI вместе с ними через `models.mode: "merge"`, чтобы резервные варианты оставались
    доступными при соблюдении региональной привязки выбранного провайдера.
  </Accordion>

  <Accordion title="Нужно ли покупать Mac Mini для установки OpenClaw?">
    Нет. OpenClaw работает в macOS и Linux (в Windows — через WSL2). Mac mini часто выбирают
    в качестве постоянно работающего хоста, но также подойдёт небольшой VPS, домашний сервер или устройство класса Raspberry Pi.

    Mac нужен только **для инструментов, доступных исключительно в macOS**. Для iMessage используйте [iMessage](/ru/channels/imessage)
    с `imsg` на любом Mac, где выполнен вход в Messages. Если Gateway работает в Linux или другой системе,
    задайте для `channels.imessage.cliPath` SSH-обёртку, запускающую `imsg` на этом Mac. Для других
    инструментов, доступных только в macOS, запускайте Gateway на Mac или подключите узел macOS.

    Документация: [iMessage](/ru/channels/imessage), [Узлы](/ru/nodes), [Удалённый режим Mac](/ru/platforms/mac/remote).

  </Accordion>

  <Accordion title="Нужен ли Mac mini для поддержки iMessage?">
    Нужно **любое устройство с macOS**, на котором выполнен вход в Messages, — это не обязательно Mac mini,
    подойдёт любой Mac. Используйте [iMessage](/ru/channels/imessage) с `imsg`; Gateway может работать на этом
    Mac или на другом хосте с SSH-обёрткой `cliPath`.

    Распространённые конфигурации:

    - Gateway работает на Linux/VPS, а `channels.imessage.cliPath` задан как SSH-обёртка, запускающая `imsg` на Mac, где выполнен вход в Messages.
    - Всё работает на одном Mac — самый простой вариант с одним компьютером.

    Документация: [iMessage](/ru/channels/imessage), [Узлы](/ru/nodes), [Удалённый режим Mac](/ru/platforms/mac/remote).

  </Accordion>

  <Accordion title="Если я куплю Mac mini для запуска OpenClaw, смогу ли я подключить его к MacBook Pro?">
    Да. **Mac mini может запускать Gateway**, а MacBook Pro подключается как **узел**
    (сопутствующее устройство). Узлы не запускают Gateway — они добавляют возможности
    экрана, камеры, холста и `system.run` на этом устройстве.

    Распространённая схема: Gateway работает на постоянно включённом Mac mini; на MacBook Pro работает приложение macOS или
    хост узла, сопряжённый с Gateway. Проверьте состояние с помощью `openclaw nodes status` / `openclaw nodes list`.

    Документация: [Узлы](/ru/nodes), [CLI узлов](/ru/cli/nodes).

  </Accordion>

  <Accordion title="Можно ли использовать Bun?">
    Bun можно использовать для установки зависимостей или запуска скриптов пакетов. Для CLI OpenClaw и
    Gateway требуется **Node**, поскольку каноническое хранилище состояния использует `node:sqlite`; Bun
    не предоставляет этот API.
  </Accordion>

  <Accordion title="Telegram: что указывать в allowFrom?">
    `channels.telegram.allowFrom` — это **числовой идентификатор пользователя Telegram, отправляющего сообщения**,
    а не имя пользователя бота. При настройке запрашиваются только числовые идентификаторы пользователей; `openclaw doctor --fix`
    может попытаться разрешить устаревшие записи `@username`.

    Более безопасный вариант без стороннего бота: отправьте личное сообщение своему боту, выполните `openclaw logs --follow` и прочитайте `from.id`.

    Официальный Bot API: отправьте личное сообщение своему боту, вызовите `https://api.telegram.org/bot<bot_token>/getUpdates` и прочитайте `message.from.id`.

    Сторонний вариант с меньшей конфиденциальностью: отправьте личное сообщение `@userinfobot` или `@getidsbot`.

    См. [Управление доступом Telegram](/ru/channels/telegram#access-control-and-activation).

  </Accordion>

  <Accordion title="Могут ли несколько человек использовать один номер WhatsApp с разными экземплярами OpenClaw?">
    Да, с помощью **мультиагентной маршрутизации**. Привяжите личные сообщения WhatsApp каждого отправителя (`peer: { kind: "direct", id: "+15551234567" }`) к отдельному `agentId`, предоставив каждому человеку собственное рабочее пространство и хранилище сеансов. Ответы по-прежнему будут отправляться из **одного аккаунта WhatsApp**; управление доступом к личным сообщениям (`channels.whatsapp.dmPolicy` / `channels.whatsapp.allowFrom`) действует глобально для аккаунта. См. [Мультиагентная маршрутизация](/ru/concepts/multi-agent) и [WhatsApp](/ru/channels/whatsapp).
  </Accordion>

  <Accordion title='Можно ли использовать одного агента для «быстрых бесед», а другого — с «Opus для программирования»?'>
    Да. Используйте мультиагентную маршрутизацию: назначьте каждому агенту собственную модель по умолчанию, а затем привяжите входящие
    маршруты (аккаунт провайдера или конкретных собеседников) к соответствующим агентам. Пример конфигурации:
    [Мультиагентная маршрутизация](/ru/concepts/multi-agent). См. также [Модели](/ru/concepts/models) и
    [Конфигурация](/ru/gateway/configuration).
  </Accordion>

  <Accordion title="Работает ли Homebrew в Linux?">
    Да, через Linuxbrew:

    ```bash
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
    echo 'eval "$(/home/linuxbrew/.linuxbrew/bin/brew shellenv)"' >> ~/.profile
    eval "$(/home/linuxbrew/.linuxbrew/bin/brew shellenv)"
    brew install <formula>
    ```

    При запуске OpenClaw через systemd убедитесь, что PATH службы содержит
    `/home/linuxbrew/.linuxbrew/bin` (или ваш префикс brew), чтобы инструменты, установленные через `brew`,
    разрешались в оболочках без входа в систему. Последние сборки также добавляют в начало PATH распространённые пользовательские каталоги исполняемых файлов в службах
    systemd Linux (например, `~/.local/bin`, `~/.npm-global/bin`,
    `~/.local/share/pnpm`, `~/.bun/bin`) и учитывают `PNPM_HOME`, `NPM_CONFIG_PREFIX`,
    `BUN_INSTALL`, `VOLTA_HOME`, `ASDF_DATA_DIR`, `NVM_DIR` и `FNM_DIR`, если они заданы.

  </Accordion>

  <Accordion title="Разница между изменяемой установкой из git и установкой из npm">
    - **Изменяемая установка (git):** полная копия исходного кода с возможностью редактирования, лучше всего подходит участникам разработки. Вы выполняете сборку локально и можете изменять код и документацию.
    - **Установка из npm:** глобальная установка CLI без репозитория, лучше всего подходит для сценария «просто запустить». Обновления поступают через dist-теги npm.

    Документация: [Начало работы](/ru/start/getting-started), [Обновление](/ru/install/updating).

  </Accordion>

  <Accordion title="Можно ли позже переключаться между установками через npm и git?">
    Да, с помощью `openclaw update --channel ...` для существующей установки. При этом **ваши
    данные не удаляются** — изменяется только способ установки кода OpenClaw. Состояние (`~/.openclaw`) и
    рабочая область (`~/.openclaw/workspace`) остаются без изменений.

    Переход с npm на git:

    ```bash
    openclaw update --channel dev
    ```

    Переход с git на npm:

    ```bash
    openclaw update --channel stable
    ```

    Добавьте `--dry-run`, чтобы сначала просмотреть запланированное переключение режима. Средство обновления выполняет
    последующие действия Doctor, обновляет источники плагинов для целевого канала и перезапускает Gateway,
    если не передан параметр `--no-restart`.

    Установщик также может принудительно выбрать любой из режимов:

    ```bash
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method git
    curl -fsSL --proto '=https' --tlsv1.2 https://openclaw.ai/install.sh | bash -s -- --install-method npm
    ```

    Советы по резервному копированию: [Где данные хранятся на диске](/ru/help/faq#where-things-live-on-disk).

  </Accordion>

  <Accordion title="Где лучше запускать Gateway: на ноутбуке или VPS?">
    Нужна круглосуточная надежность? Используйте **VPS**. Хотите минимальных сложностей и готовы мириться
    с переходом в спящий режим и перезапусками? Запускайте локально.

    **Ноутбук (локальный Gateway)**

    - **Преимущества:** нет расходов на сервер, прямой доступ к локальным файлам, видимое окно браузера.
    - **Недостатки:** переход в спящий режим и обрывы сети приводят к отключению, обновления и перезагрузки ОС прерывают работу, ноутбук должен оставаться активным.

    **VPS / облако**

    - **Преимущества:** постоянная работа, стабильная сеть, нет проблем со спящим режимом ноутбука, проще обеспечить непрерывную работу.
    - **Недостатки:** часто нет графического интерфейса (используйте снимки экрана), только удаленный доступ к файлам, для обновлений требуется SSH.

    WhatsApp/Telegram/Slack/Mattermost/Discord одинаково хорошо работают с VPS — фактический
    компромисс заключается в выборе между браузером без графического интерфейса и видимым окном. См. раздел [Браузер](/ru/tools/browser).

    Рекомендация по умолчанию: используйте VPS, если ранее уже происходили отключения Gateway; локальный запуск отлично подходит,
    когда вы активно используете Mac и вам нужен доступ к локальным файлам либо автоматизация
    пользовательского интерфейса в видимом браузере.

  </Accordion>

  <Accordion title="Насколько важно запускать OpenClaw на выделенном компьютере?">
    Это не обязательно, но рекомендуется для надежности и изоляции.

    - **Выделенный хост (VPS/Mac mini/Raspberry Pi):** постоянная работа, меньше прерываний из-за перехода в спящий режим или перезагрузки, более упорядоченные разрешения, проще обеспечить непрерывную работу.
    - **Общий ноутбук или настольный компьютер:** подходит для тестирования и активного использования, но работа будет приостанавливаться при переходе компьютера в спящий режим или установке обновлений.

    Чтобы получить преимущества обоих вариантов, разместите Gateway на выделенном хосте и подключите ноутбук как
    **Node** для локальных инструментов работы с экраном, камерой и выполнения команд. См. разделы [Узлы](/ru/nodes) и [Безопасность](/ru/gateway/security).

  </Accordion>

  <Accordion title="Каковы минимальные требования к VPS и рекомендуемая ОС?">
    - **Абсолютный минимум:** 1 виртуальный процессор, 1 ГБ ОЗУ, около 500 МБ на диске.
    - **Рекомендуется:** 1–2 виртуальных процессора, не менее 2 ГБ ОЗУ с запасом для журналов, медиафайлов и нескольких каналов. Инструменты Node и автоматизация браузера могут потреблять много ресурсов.

    ОС: **Ubuntu LTS** (или любая современная версия Debian/Ubuntu) — наиболее тщательно протестированный вариант установки в Linux.

    Документация: [Linux](/ru/platforms/linux), [Размещение на VPS](/ru/vps).

  </Accordion>

  <Accordion title="Можно ли запустить OpenClaw в виртуальной машине и каковы требования?">
    Да. Относитесь к виртуальной машине как к VPS: она должна быть постоянно включена, доступна по сети и иметь достаточно ОЗУ
    для Gateway и всех включенных каналов.

    - **Абсолютный минимум:** 1 виртуальный процессор, 1 ГБ ОЗУ.
    - **Рекомендуется:** не менее 2 ГБ ОЗУ для нескольких каналов, автоматизации браузера или инструментов работы с медиафайлами.
    - **ОС:** Ubuntu LTS или другая современная версия Debian/Ubuntu.

    В Windows используйте **Windows Hub** для настройки настольной среды или WSL2 для виртуальной машины Gateway
    в стиле Linux с широкой совместимостью инструментов. См. разделы [Windows](/ru/platforms/windows) и [Размещение на VPS](/ru/vps).
    Запуск macOS в виртуальной машине: см. раздел [Виртуальная машина macOS](/ru/install/macos-vm).

  </Accordion>
