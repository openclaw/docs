Флаги диагностики включают дополнительное журналирование для отдельной подсистемы, не повышая глобально `logging.level`. Флаг не действует, если подсистема его не проверяет.

## Принцип работы

- Флаги — это строки без учёта регистра, получаемые из `diagnostics.flags` в конфигурации с учётом переопределения переменной окружения `OPENCLAW_DIAGNOSTICS`, очищаемые от дубликатов и приводимые к нижнему регистру.
- `name.*` соответствует самому `name` и всему внутри пространства имён `name.` (например, `telegram.*` соответствует `telegram.http`).
- `*` или `all` включает все флаги.
- После изменения `diagnostics.flags` в конфигурации перезапустите Gateway; горячая перезагрузка не поддерживается.

## Известные флаги

| Флаг             | Что включает                                                         |
| ---------------- | --------------------------------------------------------------------- |
| `telegram.http`  | Журналирование ошибок HTTP Telegram Bot API                           |
| `brave.http`     | Журналирование запросов, ответов и кеша Brave Search                   |
| `profiler`       | Профилировщик этапа ответа и профилировщик сервера приложений Codex    |
| `reply.profiler` | Только профилировщик этапа ответа                                      |
| `codex.profiler` | Только профилировщик сервера приложений Codex                          |
| `timeline`       | Структурированный артефакт временной шкалы JSONL (см. ниже)            |

## Включение через конфигурацию

```json
{
  "diagnostics": {
    "flags": ["telegram.http"]
  }
}
```

Несколько флагов:

```json
{
  "diagnostics": {
    "flags": ["telegram.http", "brave.http", "gateway.*"]
  }
}
```

## Переопределение переменной окружения (однократное)

```bash
OPENCLAW_DIAGNOSTICS=telegram.http,brave.http
```

Значения разделяются запятыми или пробельными символами. Специальные значения:

| Значение                    | Действие                                                  |
| --------------------------- | --------------------------------------------------------- |
| `0`, `false`, `off`, `none` | Отключить все флаги, включая заданные в конфигурации       |
| `1`, `true`, `all`, `*`     | Включить все флаги                                        |

`OPENCLAW_DIAGNOSTICS=0` отключает для данного процесса флаги как из переменной окружения, так и из конфигурации. Это полезно, чтобы временно отключить оставленный в конфигурации флаг профилировщика без редактирования файла.

## Флаги профилировщика

Флаги профилировщика управляют упрощёнными интервалами измерения времени; в отключённом состоянии они не создают накладных расходов.

Включить все интервалы, управляемые профилировщиком, для одного запуска Gateway:

```bash
OPENCLAW_DIAGNOSTICS=profiler openclaw gateway run
```

Включить только интервалы профилировщика диспетчеризации ответов:

```bash
OPENCLAW_DIAGNOSTICS=reply.profiler openclaw gateway run
```

Включить только интервалы профилировщика запуска, инструментов и потоков сервера приложений Codex:

```bash
OPENCLAW_DIAGNOSTICS=codex.profiler openclaw gateway run
```

`profiler` включает и профилировщик ответов, и профилировщик Codex; чтобы включить только один из них, используйте соответствующее полное имя флага.

Или задайте их в конфигурации:

```json
{
  "diagnostics": {
    "flags": ["reply.profiler", "codex.profiler"]
  }
}
```

После изменения флагов конфигурации перезапустите Gateway. Чтобы отключить флаг профилировщика, удалите его из `diagnostics.flags` и перезапустите Gateway либо запустите процесс с `OPENCLAW_DIAGNOSTICS=0`, чтобы переопределить все флаги диагностики для этого запуска.

## Артефакты временной шкалы

Флаг `timeline` (псевдоним: `diagnostics.timeline`) записывает структурированные события времени запуска и выполнения в формате JSONL для внешних систем контроля качества:

```bash
OPENCLAW_DIAGNOSTICS=timeline \
OPENCLAW_DIAGNOSTICS_TIMELINE_PATH=/tmp/openclaw-timeline.jsonl \
openclaw gateway run
```

Или включите его в конфигурации:

```json
{
  "diagnostics": {
    "flags": ["timeline"]
  }
}
```

Путь вывода всегда берётся из `OPENCLAW_DIAGNOSTICS_TIMELINE_PATH`, даже если сам флаг задан в конфигурации; ключа конфигурации для этого пути нет. Если `timeline` включён только через конфигурацию, самые ранние интервалы загрузки конфигурации отсутствуют, поскольку OpenClaw к этому моменту ещё не прочитал её; последующие интервалы запуска фиксируются в обычном режиме.

`OPENCLAW_DIAGNOSTICS=1`, `=all` и `=*` также включают временную шкалу, поскольку они включают все флаги. Если вам нужен только артефакт JSONL, а не все остальные флаги диагностики, используйте отдельный флаг `timeline`.

Для выборок задержки цикла событий во временной шкале требуется дополнительное явное включение помимо `timeline`: вместе с включением временной шкалы задайте `OPENCLAW_DIAGNOSTICS_EVENT_LOOP=1` (либо `on`/`true`/`yes`).

Записи временной шкалы используют оболочку `openclaw.diagnostics.v1` и могут содержать идентификаторы процессов, названия фаз, названия интервалов, продолжительность, идентификаторы плагинов, количество зависимостей, выборки задержки цикла событий, названия операций провайдера, состояние завершения дочерних процессов, а также названия и сообщения ошибок запуска. Считайте файлы временной шкалы локальными диагностическими артефактами; проверяйте их перед передачей за пределы своего компьютера.

## Расположение журналов

Флаги записывают сообщения в стандартный файл журнала диагностики. По умолчанию:

```
/tmp/openclaw/openclaw-YYYY-MM-DD.log
```

Если задан `logging.file`, вместо этого используется указанный там путь. Журналы имеют формат JSONL (один объект JSON на строку). Редактирование конфиденциальных данных по-прежнему применяется в соответствии с `logging.redactSensitive`. Полное описание определения пути к журналу, ротации и модели редактирования см. в разделе [Журналирование](/ru/logging).

## Извлечение журналов

Выбрать последний файл журнала:

```bash
ls -t /tmp/openclaw/openclaw-*.log | head -n 1
```

Отфильтровать диагностику HTTP Telegram:

```bash
rg "telegram http error" /tmp/openclaw/openclaw-*.log
```

Отфильтровать диагностику HTTP Brave Search:

```bash
rg "brave http" /tmp/openclaw/openclaw-*.log
```

Или отслеживать журнал во время воспроизведения:

```bash
tail -f /tmp/openclaw/openclaw-$(date +%F).log | rg "telegram http error"
```

Для удалённых экземпляров Gateway используйте вместо этого `openclaw logs --follow` (см. [/cli/logs](/ru/cli/logs)).

## Примечания

- Если для `logging.level` задан уровень выше `warn`, сообщения журналов, управляемые флагами, могут подавляться. Уровень `info` по умолчанию подходит.
- `brave.http` журналирует URL и параметры запросов Brave Search, состояние и время ответов, а также события попадания в кеш, промаха кеша и записи в кеш. Он не журналирует ключ API (передаваемый в заголовке запроса) или тела ответов, однако поисковые запросы могут содержать конфиденциальные данные.
- Флаги можно безопасно оставлять включёнными; они влияют только на объём журналов соответствующей подсистемы.
- Используйте [/logging](/ru/logging), чтобы изменить место хранения журналов, уровни и редактирование конфиденциальных данных.
