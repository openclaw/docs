[сеансы Agent Client Protocol (ACP)](https://agentclientprotocol.com/)
позволяют OpenClaw запускать внешние кодовые harness-среды (например Claude Code,
Cursor, Copilot, Droid, OpenClaw ACP, OpenCode, Gemini CLI и другие
поддерживаемые ACPX harness-среды) через backend Plugin ACP.

Каждый запуск сеанса ACP отслеживается как [фоновая задача](/ru/automation/tasks).

<Note>
**ACP — это путь для внешних harness-сред, а не путь Codex по умолчанию.** 
Нативный Plugin сервера приложений Codex владеет элементами управления `/codex ...` и встроенным runtime по умолчанию
`openai/gpt-*` для ходов агента; ACP владеет
элементами управления `/acp ...` и сеансами `sessions_spawn({ runtime: "acp" })`.

Если вы хотите, чтобы Codex или Claude Code подключались как внешний MCP-клиент
напрямую к существующим беседам каналов OpenClaw, используйте
[`openclaw mcp serve`](/ru/cli/mcp) вместо ACP.
</Note>

## Работает ли это сразу после установки?

Да, после установки официального Plugin runtime ACP:

```bash
openclaw plugins install @openclaw/acpx
openclaw config set plugins.entries.acpx.enabled true
```

Исходные checkout'ы могут использовать локальный workspace Plugin `extensions/acpx` после
`pnpm install`. Запустите `/acp doctor`, чтобы проверить готовность.

OpenClaw обучает агентов запуску ACP только когда ACP **действительно
доступен**: ACP должен быть включен, dispatch не должен быть отключен, текущий
сеанс не должен быть заблокирован sandbox'ом, а backend runtime должен быть
загружен. Если эти условия не выполнены, Skills Plugin ACP и рекомендации
`sessions_spawn` для ACP остаются скрытыми, чтобы агент не предлагал
недоступный backend.

<AccordionGroup>
  <Accordion title="First-run gotchas">
    - Если задан `plugins.allow`, это ограничивающий инвентарь Plugin, и он **обязан** включать `acpx`; иначе установленный backend ACP намеренно блокируется, а `/acp doctor` сообщает об отсутствующей записи allowlist.
    - Адаптер Codex ACP поставляется вместе с Plugin `acpx` и по возможности запускается локально.
    - Codex ACP работает с изолированным `CODEX_HOME`; OpenClaw копирует доверенные записи проекта и безопасную конфигурацию маршрутизации модели/провайдера из конфигурации Codex на хосте, а auth, уведомления и hooks остаются в конфигурации хоста.
    - Другие адаптеры целевых harness-сред всё еще могут загружаться по требованию через `npx` при первом использовании.
    - Vendor auth для этой harness-среды всё равно должен существовать на хосте.
    - Если на хосте нет npm или доступа к сети, первая загрузка адаптера завершится ошибкой, пока кэши не будут заранее прогреты или адаптер не будет установлен другим способом.

  </Accordion>
  <Accordion title="Runtime prerequisites">
    ACP запускает реальный внешний процесс harness-среды. OpenClaw владеет маршрутизацией,
    состоянием фоновых задач, доставкой, привязками и политикой; harness-среда
    владеет своим входом к провайдеру, каталогом моделей, поведением файловой системы и
    нативными инструментами.

    Прежде чем винить OpenClaw, проверьте:

    - `/acp doctor` сообщает о включенном и исправном backend.
    - Целевой id разрешен `acp.allowedAgents`, если этот allowlist задан.
    - Команда harness-среды может запускаться на хосте Gateway.
    - Auth провайдера присутствует для этой harness-среды (`claude`, `codex`, `gemini`, `opencode`, `droid` и т. д.).
    - Выбранная модель существует для этой harness-среды - id моделей не переносимы между harness-средами.
    - Запрошенный `cwd` существует и доступен, либо опустите `cwd` и позвольте backend использовать значение по умолчанию.
    - Режим разрешений соответствует работе. Неинтерактивные сеансы не могут нажимать нативные запросы разрешений, поэтому кодовые запуски с большим количеством записи/exec обычно требуют профиль разрешений ACPX, способный выполняться без участия пользователя.

  </Accordion>
</AccordionGroup>

Инструменты Plugin OpenClaw и встроенные инструменты OpenClaw по умолчанию
**не** предоставляются ACP harness-средам. Включайте явные MCP-мосты в
[ACP-агенты - настройка](/ru/tools/acp-agents-setup) только когда harness-среда
должна вызывать эти инструменты напрямую.

## Поддерживаемые целевые harness-среды

С backend `acpx` используйте эти id harness-сред как цели `/acp spawn <id>`
или `sessions_spawn({ runtime: "acp", agentId: "<id>" })`:

| Id harness-среды | Типичный backend                              | Примечания                                                                        |
| ---------------- | --------------------------------------------- | ---------------------------------------------------------------------------------- |
| `claude`         | Адаптер Claude Code ACP                       | Требует auth Claude Code на хосте.                                                 |
| `codex`          | Адаптер Codex ACP                             | Только явный резервный вариант ACP, когда нативный `/codex` недоступен или запрошен ACP. |
| `copilot`        | Адаптер GitHub Copilot ACP                    | Требует auth Copilot CLI/runtime.                                                  |
| `cursor`         | Cursor CLI ACP (`cursor-agent acp`)           | Переопределите команду acpx, если локальная установка предоставляет другую точку входа ACP. |
| `droid`          | Factory Droid CLI                             | Требует auth Factory/Droid или `FACTORY_API_KEY` в окружении harness-среды.        |
| `gemini`         | Адаптер Gemini CLI ACP                        | Требует auth Gemini CLI или настройки API-ключа.                                   |
| `iflow`          | iFlow CLI                                     | Доступность адаптера и управление моделью зависят от установленного CLI.           |
| `kilocode`       | Kilo Code CLI                                 | Доступность адаптера и управление моделью зависят от установленного CLI.           |
| `kimi`           | Kimi/Moonshot CLI                             | Требует auth Kimi/Moonshot на хосте.                                               |
| `kiro`           | Kiro CLI                                      | Доступность адаптера и управление моделью зависят от установленного CLI.           |
| `opencode`       | Адаптер OpenCode ACP                          | Требует auth OpenCode CLI/провайдера.                                              |
| `openclaw`       | Мост OpenClaw Gateway через `openclaw acp`    | Позволяет ACP-совместимой harness-среде обращаться обратно к сеансу OpenClaw Gateway. |
| `qwen`           | Qwen Code / Qwen CLI                          | Требует Qwen-совместимый auth на хосте.                                            |

Пользовательские псевдонимы агентов acpx можно настроить в самом acpx, но политика OpenClaw
по-прежнему проверяет `acp.allowedAgents` и любое сопоставление
`agents.list[].runtime.acp.agent` перед dispatch.

## Runbook оператора

Быстрый поток `/acp` из чата:

<Steps>
  <Step title="Spawn">
    `/acp spawn claude --bind here`,
    `/acp spawn gemini --mode persistent --thread auto` или явный
    `/acp spawn codex --bind here`.
  </Step>
  <Step title="Work">
    Продолжайте в привязанной беседе или thread (либо явно укажите ключ
    сеанса).
  </Step>
  <Step title="Check state">
    `/acp status`
  </Step>
  <Step title="Tune">
    `/acp model <provider/model>`,
    `/acp permissions <profile>`,
    `/acp timeout <seconds>`.
  </Step>
  <Step title="Steer">
    Без замены контекста: `/acp steer tighten logging and continue`.
  </Step>
  <Step title="Stop">
    `/acp cancel` (текущий ход) или `/acp close` (сеанс + привязки).
  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Lifecycle details">
    - Spawn создает или возобновляет сеанс runtime ACP, записывает метаданные ACP в хранилище сеансов OpenClaw и может создать фоновую задачу, когда запуск принадлежит родителю.
    - Сеансы ACP, принадлежащие родителю, рассматриваются как фоновая работа, даже когда сеанс runtime постоянный; завершение и доставка между surface'ами проходят через notifier родительской задачи, а не ведут себя как обычный пользовательский сеанс чата.
    - Обслуживание задач закрывает terminal или осиротевшие одноразовые сеансы ACP, принадлежащие родителю. Постоянные сеансы ACP сохраняются, пока остается активная привязка беседы; устаревшие постоянные сеансы без активной привязки закрываются, чтобы их нельзя было незаметно возобновить после завершения владеющей задачи или исчезновения ее записи.
    - Привязанные последующие сообщения идут напрямую в сеанс ACP, пока привязка не будет закрыта, снята с фокуса, сброшена или не истечет.
    - Команды Gateway остаются локальными. `/acp ...`, `/status` и `/unfocus` никогда не отправляются как обычный текст prompt в привязанную ACP harness-среду.
    - `cancel` прерывает активный ход, когда backend поддерживает отмену; он не удаляет привязку или метаданные сеанса.
    - `close` завершает сеанс ACP с точки зрения OpenClaw и удаляет привязку. Harness-среда может всё еще сохранять собственную upstream-историю, если поддерживает возобновление.
    - Plugin acpx очищает принадлежащие OpenClaw деревья процессов wrapper'ов и адаптеров после `close`, а также собирает устаревшие принадлежащие OpenClaw ACPX orphans при запуске Gateway.
    - Неактивные workers runtime могут быть очищены после `acp.runtime.ttlMinutes`; сохраненные метаданные сеанса остаются доступны для `/acp sessions`.

  </Accordion>
  <Accordion title="Native Codex routing rules">
    Триггеры на естественном языке, которые должны маршрутизироваться в **нативный Plugin Codex**,
    когда он включен:

    - «Привяжи этот канал Discord к Codex.»
    - «Прикрепи этот чат к thread Codex `<id>`.»
    - «Покажи thread'ы Codex, затем привяжи этот.»

    Нативная привязка беседы Codex является стандартным путем управления чатом.
    Динамические инструменты OpenClaw по-прежнему выполняются через OpenClaw, а
    нативные инструменты Codex, такие как shell/apply-patch, выполняются внутри Codex.
    Для событий нативных инструментов Codex OpenClaw внедряет нативную

    - `/codex ...` - нативное управление беседой Codex.

    - "Запусти это как одноразовый сеанс Claude Code ACP и кратко изложи результат."

    - "Запусти Codex через ACP в фоновом потоке."

    OpenClaw выбирает `runtime: "acp"`, разрешает `agentId` harness,
    привязывается к текущей беседе или потоку, если это поддерживается, и
    маршрутизирует последующие сообщения в этот сеанс до закрытия/истечения срока. Codex

    недоступен для запрошенной операции.

    включен, запрашивающая сторона не находится в песочнице и загружен backend
    среды выполнения ACP. `acp.dispatch.enabled=false` приостанавливает автоматическую

    явно не настроена с `agents.list[].runtime.type="acp"`;
    иначе используйте стандартную среду выполнения субагента. Когда агент OpenClaw
    настроен с `runtime.type="acp"`, OpenClaw использует
    `runtime.acp.agent` как базовый идентификатор harness.

## ACP против субагентов

Используйте ACP, когда вам нужна внешняя среда выполнения harness. Используйте **нативный Codex
app-server** для привязки/управления беседой Codex, когда Plugin `codex`
включен. Используйте **субагентов**, когда вам нужны нативные для OpenClaw
делегированные запуски.

| Область       | Сеанс ACP                             | Запуск субагента                   |
| ------------- | ------------------------------------- | ---------------------------------- |
| Среда выполнения | Backend Plugin ACP (например, acpx) | Нативная среда выполнения субагента OpenClaw |
| Ключ сеанса   | `agent:<agentId>:acp:<uuid>`          | `agent:<agentId>:subagent:<uuid>`  |
| Основные команды | `/acp ...`                         | `/subagents ...`                   |
| Инструмент запуска | `sessions_spawn` с `runtime:"acp"` | `sessions_spawn` (стандартная среда выполнения) |

См. также [Субагенты](/ru/tools/subagents).

## Как ACP запускает Claude Code

Для Claude Code через ACP стек выглядит так:

1. Плоскость управления сеансами ACP OpenClaw.
2. Официальный Plugin среды выполнения `@openclaw/acpx`.
3. Адаптер Claude ACP.
4. Механизмы среды выполнения/сеанса на стороне Claude.

ACP Claude — это **сеанс harness** с элементами управления ACP, возобновлением сеанса,
отслеживанием фоновых задач и необязательной привязкой беседы/потока.

Backend CLI — это отдельные текстовые локальные резервные среды выполнения; см.
[Backend CLI](/ru/gateway/cli-backends).

Для операторов практическое правило такое:

- **Нужны `/acp spawn`, привязываемые сеансы, элементы управления средой выполнения или постоянная работа harness?** Используйте ACP.
- **Нужен простой локальный текстовый резервный путь через raw CLI?** Используйте backend CLI.

## Привязанные сеансы

### Ментальная модель

- **Поверхность чата** - место, где люди продолжают общаться (канал Discord, тема Telegram, чат iMessage).
- **Сеанс ACP** - устойчивое состояние среды выполнения Codex/Claude/Gemini, в которое маршрутизирует OpenClaw.
- **Дочерний поток/тема** - необязательная дополнительная поверхность сообщений, создаваемая только через `--thread ...`.
- **Рабочая область среды выполнения** - расположение в файловой системе (`cwd`, checkout репозитория, рабочая область backend), где запускается harness. Не зависит от поверхности чата.

### Привязки текущей беседы

`/acp spawn <harness> --bind here` закрепляет текущую беседу за
созданным сеансом ACP - без дочернего потока, на той же поверхности чата. OpenClaw продолжает
владеть транспортом, аутентификацией, безопасностью и доставкой. Последующие сообщения в этой
беседе маршрутизируются в тот же сеанс; `/new` и `/reset` сбрасывают
сеанс на месте; `/acp close` удаляет привязку.

Примеры:

```text
/codex bind                                              # нативная привязка Codex, маршрутизировать будущие сообщения сюда
/codex model gpt-5.4                                     # настроить привязанный нативный поток Codex
/codex stop                                              # управлять активным нативным ходом Codex
/acp spawn codex --bind here                             # явный резервный ACP-путь для Codex
/acp spawn codex --thread auto                           # может создать дочерний поток/тему и привязать там
/acp spawn codex --bind here --cwd /workspace/repo       # та же привязка чата, Codex запускается в /workspace/repo
```

<AccordionGroup>
  <Accordion title="Правила привязки и эксклюзивность">
    - `--bind here` и `--thread ...` являются взаимоисключающими.
    - `--bind here` работает только в каналах, которые объявляют привязку текущей беседы; иначе OpenClaw возвращает понятное сообщение о неподдерживаемости. Привязки сохраняются между перезапусками gateway.
    - В Discord `spawnSessions` ограничивает создание дочернего потока для `--thread auto|here` - не для `--bind here`.
    - Если вы запускаете другой ACP-агент без `--cwd`, OpenClaw по умолчанию наследует рабочую область **целевого агента**. Отсутствующие унаследованные пути (`ENOENT`/`ENOTDIR`) откатываются к стандартному значению backend; другие ошибки доступа (например, `EACCES`) отображаются как ошибки запуска.
    - Команды управления Gateway остаются локальными в привязанных беседах - команды `/acp ...` обрабатываются OpenClaw, даже когда обычный последующий текст маршрутизируется в привязанный сеанс ACP; `/status` и `/unfocus` также остаются локальными, когда обработка команд включена для этой поверхности.

  </Accordion>
  <Accordion title="Сеансы, привязанные к потоку">
    Когда привязки потоков включены для адаптера канала:

    - OpenClaw привязывает поток к целевому сеансу ACP.
    - Последующие сообщения в этом потоке маршрутизируются в привязанный сеанс ACP.
    - Вывод ACP доставляется обратно в тот же поток.
    - Unfocus/close/archive/idle-timeout или истечение max-age удаляет привязку.
    - `/acp close`, `/acp cancel`, `/acp status`, `/status` и `/unfocus` — это команды Gateway, а не промпты для ACP harness.

    Обязательные feature flags для ACP, привязанного к потоку:

    - `acp.enabled=true`
    - `acp.dispatch.enabled` включен по умолчанию (установите `false`, чтобы приостановить автоматическую отправку потоков ACP; явные вызовы `sessions_spawn({ runtime: "acp" })` продолжат работать).
    - Запуски сеансов потоков адаптера канала включены (по умолчанию: `true`):
      - Discord: `channels.discord.threadBindings.spawnSessions=true`
      - Telegram: `channels.telegram.threadBindings.spawnSessions=true`

    Поддержка привязки потоков зависит от адаптера. Если активный адаптер канала
    не поддерживает привязки потоков, OpenClaw возвращает понятное
    сообщение о неподдерживаемости/недоступности.

  </Accordion>
  <Accordion title="Каналы с поддержкой потоков">
    - Любой адаптер канала, который предоставляет возможность привязки сеанса/потока.
    - Текущая встроенная поддержка: потоки/каналы **Discord**, темы **Telegram** (темы форума в группах/супергруппах и темы DM).
    - Каналы Plugin могут добавить поддержку через тот же интерфейс привязки.

  </Accordion>
</AccordionGroup>

## Постоянные привязки каналов

Для неэфемерных рабочих процессов настройте постоянные привязки ACP в
записях верхнего уровня `bindings[]`.

### Модель привязки

<ParamField path="bindings[].type" type='"acp"'>
  Помечает постоянную привязку беседы ACP.
</ParamField>
<ParamField path="bindings[].match" type="object">
  Определяет целевую беседу. Формы для отдельных каналов:

- **Канал/поток Discord:** `match.channel="discord"` + `match.peer.id="<channelOrThreadId>"`
- **Канал/DM Slack:** `match.channel="slack"` + `match.peer.id="<channelId|channel:<channelId>|#<channelId>|userId|user:<userId>|slack:<userId>|<@userId>>"`. Предпочитайте стабильные идентификаторы Slack; привязки каналов также сопоставляют ответы внутри потоков этого канала.
- **Тема форума Telegram:** `match.channel="telegram"` + `match.peer.id="<chatId>:topic:<topicId>"`
- **DM/группа WhatsApp:** `match.channel="whatsapp"` + `match.peer.id="<E.164|group JID>"`. Используйте номера E.164, такие как `+15555550123`, для прямых чатов и JID групп WhatsApp, такие как `120363424282127706@g.us`, для групп.
- **DM/группа iMessage:** `match.channel="imessage"` + `match.peer.id="<handle|chat_id:*|chat_guid:*|chat_identifier:*>"`. Предпочитайте `chat_id:*` для стабильных привязок групп.

</ParamField>
<ParamField path="bindings[].agentId" type="string">
  Идентификатор владеющего агента OpenClaw.
</ParamField>
<ParamField path="bindings[].acp.mode" type='"persistent" | "oneshot"'>
  Необязательное переопределение ACP.
</ParamField>
<ParamField path="bindings[].acp.label" type="string">
  Необязательная метка для оператора.
</ParamField>
<ParamField path="bindings[].acp.cwd" type="string">
  Необязательный рабочий каталог среды выполнения.
</ParamField>
<ParamField path="bindings[].acp.backend" type="string">
  Необязательное переопределение backend.
</ParamField>

### Стандартные значения среды выполнения для каждого агента

Используйте `agents.list[].runtime`, чтобы один раз определить стандартные значения ACP для каждого агента:

- `agents.list[].runtime.type="acp"`
- `agents.list[].runtime.acp.agent` (идентификатор harness, например `codex` или `claude`)
- `agents.list[].runtime.acp.backend`
- `agents.list[].runtime.acp.mode`
- `agents.list[].runtime.acp.cwd`

**Приоритет переопределений для привязанных сеансов ACP:**

1. `bindings[].acp.*`
2. `agents.list[].runtime.acp.*`
3. Глобальные стандартные значения ACP (например, `acp.backend`)

### Пример

```json5
{
  agents: {
    list: [
      {
        id: "codex",
        runtime: {
          type: "acp",
          acp: {
            agent: "codex",
            backend: "acpx",
            mode: "persistent",
            cwd: "/workspace/openclaw",
          },
        },
      },
      {
        id: "claude",
        runtime: {
          type: "acp",
          acp: { agent: "claude", backend: "acpx", mode: "persistent" },
        },
      },
    ],
  },
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "discord",
        accountId: "default",
        peer: { kind: "channel", id: "222222222222222222" },
      },
      acp: { label: "codex-main" },
    },
    {
      type: "acp",
      agentId: "claude",
      match: {
        channel: "telegram",
        accountId: "default",
        peer: { kind: "group", id: "-1001234567890:topic:42" },
      },
      acp: { cwd: "/workspace/repo-b" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "discord", accountId: "default" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "telegram", accountId: "default" },
    },
  ],
  channels: {
    discord: {
      guilds: {
        "111111111111111111": {
          channels: {
            "222222222222222222": { requireMention: false },
          },
        },
      },
    },
    telegram: {
      groups: {
        "-1001234567890": {
          topics: { "42": { requireMention: false } },
        },
      },
    },
  },
}
```

### Поведение

- OpenClaw проверяет, что настроенная сессия ACP существует после допуска, специфичного для канала, и перед использованием.
- Сообщения в этом канале, теме или чате маршрутизируются в настроенную сессию ACP.
- Настроенные привязки ACP владеют своим маршрутом сессии. Широковещательное разветвление канала не заменяет настроенную сессию ACP для совпавшей привязки.
- В привязанных диалогах `/new` и `/reset` сбрасывают тот же ключ сессии ACP на месте.
- Временные привязки runtime, например созданные потоками фокуса на треде, по-прежнему применяются там, где присутствуют.
- Для межагентных запусков ACP без явного `cwd` OpenClaw наследует рабочую область целевого агента из конфигурации агента.
- Отсутствующие наследуемые пути рабочей области откатываются к стандартному cwd backend; ошибки доступа для существующих путей отображаются как ошибки запуска.

## Запуск сессий ACP

Два способа запустить сессию ACP:

<Tabs>
  <Tab title="From sessions_spawn">
    Используйте `runtime: "acp"`, чтобы запустить сессию ACP из хода агента или
    вызова инструмента.

    ```json
    {
      "task": "Open the repo and summarize failing tests",
      "runtime": "acp",
      "agentId": "codex",
      "thread": true,
      "mode": "session"
    }
    ```

    <Note>
    По умолчанию `runtime` равен `subagent`, поэтому для сессий ACP явно
    задавайте `runtime: "acp"`. Если `agentId` опущен, OpenClaw использует
    `acp.defaultAgent`, когда он настроен. `mode: "session"` требует
    `thread: true`, чтобы сохранить постоянный привязанный диалог.
    </Note>

  </Tab>
  <Tab title="From /acp command">
    Используйте `/acp spawn` для явного операторского управления из чата.

    ```text
    /acp spawn codex --mode persistent --thread auto
    /acp spawn codex --mode oneshot --thread off
    /acp spawn codex --bind here
    /acp spawn codex --thread here
    ```

    Ключевые флаги:

    - `--mode persistent|oneshot`
    - `--bind here|off`
    - `--thread auto|here|off`
    - `--cwd <absolute-path>`
    - `--label <name>`

    См. [слэш-команды](/ru/tools/slash-commands).

  </Tab>
</Tabs>

### Параметры `sessions_spawn`

<ParamField path="task" type="string" required>
  Начальный prompt, отправляемый в сессию ACP.
</ParamField>
<ParamField path="runtime" type='"acp"' required>
  Для сессий ACP должно быть `"acp"`.
</ParamField>
<ParamField path="agentId" type="string">
  Идентификатор целевого harness ACP. Откатывается к `acp.defaultAgent`, если он задан.
</ParamField>
<ParamField path="thread" type="boolean" default="false">
  Запрашивает поток привязки треда там, где это поддерживается.
</ParamField>
<ParamField path="mode" type='"run" | "session"' default="run">
  `"run"` является одноразовым; `"session"` является постоянным. Если `thread: true`, а
  `mode` опущен, OpenClaw может по умолчанию использовать постоянное поведение в зависимости от
  пути runtime. `mode: "session"` требует `thread: true`.
</ParamField>
<ParamField path="cwd" type="string">
  Запрошенный рабочий каталог runtime, проверяемый политикой backend/runtime.
  Если он опущен, запуск ACP наследует рабочую область целевого агента,
  когда она настроена; отсутствующие наследуемые пути откатываются к стандартным
  значениям backend, а реальные ошибки доступа возвращаются.
</ParamField>
<ParamField path="label" type="string">
  Метка, видимая оператору, которая используется в тексте сессии/баннера.
</ParamField>
<ParamField path="resumeSessionId" type="string">
  Возобновляет существующую сессию ACP вместо создания новой. Агент
  повторно воспроизводит историю диалога через `session/load`. Требует
  `runtime: "acp"`.
</ParamField>
<ParamField path="streamTo" type='"parent"'>
  `"parent"` транслирует сводки прогресса начального запуска ACP обратно в
  сессию запрашивающего как системные события. Принятые ответы включают
  `streamLogPath`, указывающий на ограниченный сессией журнал JSONL
  (`<sessionId>.acp-stream.jsonl`), за которым можно следить для полной истории ретрансляции.
  Потоки прогресса родителя по умолчанию показывают комментарии ассистента и прогресс статуса ACP,
  если только `streaming.progress.commentary=false`. Discord также по умолчанию
  переводит предпросмотры родителя в режим прогресса, когда режим потока не настроен. Прогресс
  статуса по-прежнему соблюдает `acp.stream.tagVisibility`, поэтому теги вроде `plan`
  остаются скрытыми, если они явно не включены.
</ParamField>

Запуски ACP `sessions_spawn` используют `agents.defaults.subagents.runTimeoutSeconds` как
стандартный лимит дочернего хода. Инструмент не принимает переопределения timeout
для отдельных вызовов.

<ParamField path="model" type="string">
  Явное переопределение модели для дочерней сессии ACP. Запуски Codex ACP
  нормализуют ссылки OpenAI, такие как `openai/gpt-5.4`, в стартовую
  конфигурацию Codex ACP перед `session/new`; слэш-формы, такие как `openai/gpt-5.4/high`,
  также задают reasoning effort Codex ACP.
  Когда параметр опущен, `sessions_spawn({ runtime: "acp" })` использует существующие
  стандартные модели subagent (`agents.defaults.subagents.model` или
  `agents.list[].subagents.model`), если они настроены; иначе он позволяет
  harness ACP использовать собственную модель по умолчанию.
  Другие harness должны объявлять ACP `models` и поддерживать
  `session/set_model`; иначе OpenClaw/acpx завершается с понятной ошибкой вместо
  тихого отката к стандартному значению целевого агента.
</ParamField>
<ParamField path="thinking" type="string">
  Явное усилие thinking/reasoning. Для Codex ACP `minimal` соответствует
  низкому усилию, `low`/`medium`/`high`/`xhigh` сопоставляются напрямую, а `off`
  опускает стартовое переопределение reasoning-effort.
  Когда параметр опущен, запуски ACP используют существующие стандартные значения thinking для subagent и
  помодельный `agents.defaults.models["provider/model"].params.thinking`
  для выбранной модели.
</ParamField>

## Режимы привязки и треда при запуске

<Tabs>
  <Tab title="--bind here|off">
    | Режим  | Поведение                                                               |
    | ------ | ---------------------------------------------------------------------- |
    | `here` | Привязать текущий активный диалог на месте; завершиться ошибкой, если активного диалога нет. |
    | `off`  | Не создавать привязку текущего диалога.                          |

    Примечания:

    - `--bind here` — самый простой операторский путь для "сделать этот канал или чат поддерживаемым Codex".
    - `--bind here` не создает дочерний тред.
    - `--bind here` доступен только на каналах, которые предоставляют поддержку привязки текущего диалога.
    - `--bind` и `--thread` нельзя объединять в одном вызове `/acp spawn`.

  </Tab>
  <Tab title="--thread auto|here|off">
    | Режим  | Поведение                                                                                            |
    | ------ | --------------------------------------------------------------------------------------------------- |
    | `auto` | В активном треде: привязать этот тред. Вне треда: создать/привязать дочерний тред, если поддерживается. |
    | `here` | Требовать текущий активный тред; завершиться ошибкой, если вы не в нем.                                                  |
    | `off`  | Без привязки. Сессия запускается непривязанной.                                                                 |

    Примечания:

    - На поверхностях без привязки тредов стандартное поведение фактически равно `off`.
    - Запуск с привязкой к треду требует поддержки политики канала:
      - Discord: `channels.discord.threadBindings.spawnSessions=true`
      - Telegram: `channels.telegram.threadBindings.spawnSessions=true`
    - Используйте `--bind here`, когда нужно закрепить текущий диалог без создания дочернего треда.

  </Tab>
</Tabs>

## Модель доставки

Сессии ACP могут быть как интерактивными рабочими областями, так и фоновой
работой, принадлежащей родителю. Путь доставки зависит от этой формы.

<AccordionGroup>
  <Accordion title="Interactive ACP sessions">
    Интерактивные сессии предназначены для продолжения общения на видимой поверхности чата:

    - `/acp spawn ... --bind here` привязывает текущий диалог к сессии ACP.
    - `/acp spawn ... --thread ...` привязывает тред/тему канала к сессии ACP.
    - Постоянные настроенные `bindings[].type="acp"` маршрутизируют совпадающие диалоги в ту же сессию ACP.

    Последующие сообщения в привязанном диалоге маршрутизируются напрямую в
    сессию ACP, а вывод ACP доставляется обратно в тот же
    канал/тред/тему.

    Что OpenClaw отправляет в harness:

    - Обычные привязанные последующие сообщения отправляются как текст prompt, плюс вложения только тогда, когда harness/backend их поддерживает.
    - Команды управления `/acp` и локальные команды Gateway перехватываются перед отправкой в ACP.
    - События завершения, созданные runtime, материализуются для каждой цели. Агенты OpenClaw получают внутренний envelope runtime-context OpenClaw; внешние harness ACP получают обычный prompt с результатом дочерней задачи и инструкцией. Сырой envelope `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>` никогда не должен отправляться во внешние harness или сохраняться как текст пользовательского транскрипта ACP.
    - Записи транскрипта ACP используют видимый пользователю текст триггера или обычный prompt завершения. Внутренние метаданные событий по возможности остаются структурированными в OpenClaw и не рассматриваются как содержимое чата, написанное пользователем.

  </Accordion>
  <Accordion title="Parent-owned one-shot ACP sessions">
    Одноразовые сессии ACP, запущенные другим агентским запуском, являются фоновыми
    дочерними задачами, похожими на sub-agent:

    - Родитель запрашивает работу через `sessions_spawn({ runtime: "acp", mode: "run" })`.
    - Дочерняя задача выполняется в собственной сессии harness ACP.
    - Дочерние ходы выполняются в той же фоновой линии, которую используют встроенные запуски sub-agent, поэтому медленный harness ACP не блокирует несвязанную работу основной сессии.
    - Отчет о завершении возвращается через путь объявления завершения задачи. OpenClaw преобразует внутренние метаданные завершения в обычный prompt ACP перед отправкой во внешний harness, поэтому harness не видит маркеры runtime context, предназначенные только для OpenClaw.
    - Родитель переписывает результат дочерней задачи обычным голосом ассистента, когда полезен ответ для пользователя.

    **Не** рассматривайте этот путь как peer-to-peer чат между родителем
    и дочерней задачей. У дочерней задачи уже есть канал завершения обратно к
    родителю.

  </Accordion>
  <Accordion title="sessions_send and A2A delivery">
    `sessions_send` может быть нацелен на другую сессию после запуска. Для обычных
    peer-сессий OpenClaw использует путь последующего сообщения agent-to-agent (A2A)
    после внедрения сообщения:

    - Дождаться ответа целевой сессии.
    - При необходимости позволить запрашивающему и цели обменяться ограниченным числом последующих ходов.
    - Попросить цель создать сообщение объявления.
    - Доставить это объявление в видимый канал или тред.

    Этот путь A2A является откатом для peer-отправок, где отправителю нужен
    видимый последующий ответ. Он остается включенным, когда несвязанная сессия может
    видеть цель ACP и отправлять ей сообщения, например при широких
    настройках `tools.sessions.visibility`.

    OpenClaw пропускает последующее действие A2A только когда запрашивающий является
    родителем собственного одноразового дочернего ACP, принадлежащего родителю. В этом случае
    запуск A2A поверх завершения задачи может разбудить родителя с
    результатом дочернего процесса, переслать ответ родителя обратно дочернему процессу и
    создать эхо-цикл родитель/дочерний процесс. Результат `sessions_send` сообщает
    `delivery.status="skipped"` для этого случая с дочерним процессом во владении, потому что
    путь завершения уже отвечает за результат.

  </Accordion>
  <Accordion title="Resume an existing session">
    Используйте `resumeSessionId`, чтобы продолжить предыдущую ACP-сессию вместо
    запуска с нуля. Агент воспроизводит историю разговора через
    `session/load`, поэтому продолжает работу с полным контекстом того, что было раньше.

    ```json
    {
      "task": "Continue where we left off - fix the remaining test failures",
      "runtime": "acp",
      "agentId": "codex",
      "resumeSessionId": "<previous-session-id>"
    }
    ```

    Распространенные сценарии использования:

    - Передайте сессию Codex с ноутбука на телефон — скажите агенту продолжить с того места, где вы остановились.
    - Продолжите сессию кодирования, которую вы начали интерактивно в CLI, теперь в безголовом режиме через своего агента.
    - Возобновите работу, прерванную перезапуском Gateway или тайм-аутом простоя.

    Примечания:

    - `resumeSessionId` применяется только при `runtime: "acp"`; среда выполнения субагента по умолчанию игнорирует это поле только для ACP.
    - `streamTo` применяется только при `runtime: "acp"`; среда выполнения субагента по умолчанию игнорирует это поле только для ACP.
    - `resumeSessionId` — это локальный для хоста ACP/harness идентификатор возобновления, а не ключ сессии канала OpenClaw; OpenClaw по-прежнему проверяет политику запуска ACP и политику целевого агента перед отправкой, а ACP-бэкенд или harness отвечает за авторизацию загрузки этого вышестоящего идентификатора.
    - `resumeSessionId` восстанавливает историю вышестоящего ACP-разговора; `thread` и `mode` по-прежнему применяются обычным образом к новой сессии OpenClaw, которую вы создаете, поэтому `mode: "session"` по-прежнему требует `thread: true`.
    - Целевой агент должен поддерживать `session/load` (Codex и Claude Code поддерживают).
    - Если идентификатор сессии не найден, запуск завершается понятной ошибкой — без тихого отката к новой сессии.

  </Accordion>
  <Accordion title="Post-deploy smoke test">
    После развертывания gateway выполните живую сквозную проверку, а не
    полагайтесь на модульные тесты:

    1. Проверьте развернутую версию gateway и коммит на целевом хосте.
    2. Откройте временную сессию ACPX bridge с живым агентом.
    3. Попросите этого агента вызвать `sessions_spawn` с `runtime: "acp"`, `agentId: "codex"`, `mode: "run"` и задачей `Reply with exactly LIVE-ACP-SPAWN-OK`.
    4. Проверьте `accepted=yes`, реальный `childSessionKey` и отсутствие ошибки валидатора.
    5. Очистите временную bridge-сессию.

    Оставьте gate на `mode: "run"` и пропустите `streamTo: "parent"` —
    привязанный к потоку `mode: "session"` и пути ретрансляции потока относятся к отдельным
    более насыщенным интеграционным проходам.

  </Accordion>
</AccordionGroup>

## Совместимость с песочницей

ACP-сессии сейчас выполняются в среде выполнения хоста, **не** внутри
песочницы OpenClaw.

<Warning>
**Граница безопасности:**

- Внешний harness может читать/писать согласно собственным разрешениям CLI и выбранному `cwd`.
- Политика песочницы OpenClaw **не** оборачивает выполнение ACP harness.
- OpenClaw по-прежнему применяет feature gates ACP, разрешенных агентов, владение сессиями, привязки каналов и политику доставки Gateway.
- Используйте `runtime: "subagent"` для нативной работы OpenClaw с принудительным применением песочницы.

</Warning>

Текущие ограничения:

- Если сессия запрашивающего находится в песочнице, ACP-запуски блокируются как для `sessions_spawn({ runtime: "acp" })`, так и для `/acp spawn`.
- `sessions_spawn` с `runtime: "acp"` не поддерживает `sandbox: "require"`.

## Разрешение целевой сессии

Большинство действий `/acp` принимают необязательную целевую сессию (`session-key`,
`session-id` или `session-label`).

**Порядок разрешения:**

1. Явный целевой аргумент (или `--session` для `/acp steer`)
   - сначала пробует ключ
   - затем UUID-образный идентификатор сессии
   - затем метку
2. Текущая привязка потока (если этот разговор/поток привязан к ACP-сессии).
3. Откат к текущей сессии запрашивающего.

Привязки текущего разговора и привязки потока обе участвуют в
шаге 2.

Если цель не разрешается, OpenClaw возвращает понятную ошибку
(`Unable to resolve session target: ...`).

## Элементы управления ACP

| Команда              | Что делает                                               | Пример                                                        |
| -------------------- | --------------------------------------------------------- | ------------------------------------------------------------- |
| `/acp spawn`         | Создает ACP-сессию; необязательная текущая привязка или привязка потока. | `/acp spawn codex --bind here --cwd /repo`                    |
| `/acp cancel`        | Отменяет выполняющийся ход для целевой сессии.            | `/acp cancel agent:codex:acp:<uuid>`                          |
| `/acp steer`         | Отправляет управляющую инструкцию в выполняющуюся сессию. | `/acp steer --session support inbox prioritize failing tests` |
| `/acp close`         | Закрывает сессию и отвязывает цели потока.                | `/acp close`                                                  |
| `/acp status`        | Показывает бэкенд, режим, состояние, параметры среды выполнения и возможности. | `/acp status`                                                 |
| `/acp set-mode`      | Задает режим среды выполнения для целевой сессии.         | `/acp set-mode plan`                                          |
| `/acp set`           | Записывает универсальный параметр конфигурации среды выполнения. | `/acp set model openai/gpt-5.4`                               |
| `/acp cwd`           | Задает переопределение рабочего каталога среды выполнения. | `/acp cwd /Users/user/Projects/repo`                          |
| `/acp permissions`   | Задает профиль политики утверждений.                      | `/acp permissions strict`                                     |
| `/acp timeout`       | Задает тайм-аут среды выполнения (секунды).               | `/acp timeout 120`                                            |
| `/acp model`         | Задает переопределение модели среды выполнения.           | `/acp model anthropic/claude-opus-4-6`                        |
| `/acp reset-options` | Удаляет переопределения параметров среды выполнения сессии. | `/acp reset-options`                                          |
| `/acp sessions`      | Перечисляет недавние ACP-сессии из хранилища.             | `/acp sessions`                                               |
| `/acp doctor`        | Состояние бэкенда, возможности, применимые исправления.   | `/acp doctor`                                                 |
| `/acp install`       | Печатает детерминированные шаги установки и включения.    | `/acp install`                                                |

Элементы управления средой выполнения (`spawn`, `cancel`, `steer`, `close`, `status`, `set-mode`,
`set`, `cwd`, `permissions`, `timeout`, `model` и `reset-options`) требуют
идентификацию владельца от внешних каналов и `operator.admin` от внутренних клиентов Gateway.
Авторизованные отправители, не являющиеся владельцами, по-прежнему могут использовать `sessions`, `doctor`,
`install` и `help`.

`/acp status` показывает эффективные параметры среды выполнения, а также идентификаторы сессий
на уровне среды выполнения и бэкенда. Ошибки неподдерживаемых элементов управления отображаются
понятно, когда у бэкенда нет возможности. `/acp sessions` читает
хранилище для текущей привязанной сессии или сессии запрашивающего; целевые токены
(`session-key`, `session-id` или `session-label`) разрешаются через
обнаружение gateway-сессий, включая пользовательские корни `session.store` для каждого агента.

### Сопоставление параметров среды выполнения

`/acp` имеет удобные команды и универсальный сеттер. Эквивалентные
операции:

| Команда                      | Сопоставляется с                     | Примечания                                                                                                                                                                                                  |
| ---------------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/acp model <id>`            | ключ конфигурации среды выполнения `model` | Для Codex ACP OpenClaw нормализует `openai/<model>` в идентификатор модели адаптера и сопоставляет суффиксы рассуждения через slash, например `openai/gpt-5.4/high`, с `reasoning_effort`.                 |
| `/acp set thinking <level>`  | канонический параметр `thinking`     | OpenClaw отправляет эквивалент, объявленный бэкендом, когда он есть, предпочитая `thinking`, затем `effort`, `reasoning_effort` или `thought_level`. Для Codex ACP адаптер сопоставляет значения с `reasoning_effort`. |
| `/acp permissions <profile>` | канонический параметр `permissionProfile` | OpenClaw отправляет эквивалент, объявленный бэкендом, когда он есть, например `approval_policy`, `permission_profile`, `permissions` или `permission_mode`.                                              |
| `/acp timeout <seconds>`     | канонический параметр `timeoutSeconds` | OpenClaw отправляет эквивалент, объявленный бэкендом, когда он есть, например `timeout` или `timeout_seconds`.                                                                                              |
| `/acp cwd <path>`            | переопределение cwd среды выполнения | Прямое обновление.                                                                                                                                                                                          |
| `/acp set <key> <value>`     | универсальное                        | `key=cwd` использует путь переопределения cwd.                                                                                                                                                              |
| `/acp reset-options`         | очищает все переопределения среды выполнения | -                                                                                                                                                                                                          |

## Устранение неполадок

| Симптом                                                                     | Вероятная причина                                                                                                           | Исправление                                                                                                                                                                      |
| --------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `ACP runtime backend is not configured`                                     | Бэкенд-плагин отсутствует, отключен или заблокирован `plugins.allow`.                                                       | Установите и включите бэкенд-плагин, добавьте `acpx` в `plugins.allow`, если этот список разрешений задан, затем выполните `/acp doctor`.                                                 |
| `ACP is disabled by policy (acp.enabled=false)`                             | ACP глобально отключен.                                                                                                 | Установите `acp.enabled=true`.                                                                                                                                                  |
| `ACP dispatch is disabled by policy (acp.dispatch.enabled=false)`           | Автоматическая отправка из обычных сообщений потока отключена.                                                               | Установите `acp.dispatch.enabled=true`, чтобы возобновить автоматическую маршрутизацию потоков; явные вызовы `sessions_spawn({ runtime: "acp" })` по-прежнему работают.                                      |
| `ACP agent "<id>" is not allowed by policy`                                 | Агент отсутствует в списке разрешений.                                                                                                | Используйте разрешенный `agentId` или обновите `acp.allowedAgents`.                                                                                                                     |
| `/acp doctor` сообщает, что бэкенд не готов сразу после запуска                 | Бэкенд-плагин отсутствует, отключен, заблокирован политикой разрешения/запрета или его настроенный исполняемый файл недоступен.        | Установите/включите бэкенд-плагин, повторно выполните `/acp doctor` и проверьте ошибку установки бэкенда или политики, если он остается неработоспособным.                                           |
| Команда обвязки не найдена                                                   | CLI адаптера не установлен, внешний плагин отсутствует или первая загрузка `npx` завершилась ошибкой для адаптера не Codex. | Выполните `/acp doctor`, установите/предварительно прогрейте адаптер на хосте Gateway или явно настройте команду агента acpx.                                                      |
| Обвязка сообщает, что модель не найдена                                            | Идентификатор модели действителен для другого провайдера/обвязки, но не для этой цели ACP.                                                | Используйте модель из списка этой обвязки, настройте модель в обвязке или не указывайте переопределение.                                                                            |
| Ошибка аутентификации поставщика от обвязки                                          | OpenClaw исправен, но вход в целевой CLI/провайдер не выполнен.                                                     | Выполните вход или предоставьте требуемый ключ провайдера в окружении хоста Gateway.                                                                                             |
| `Unable to resolve session target: ...`                                     | Некорректный ключ/идентификатор/токен метки.                                                                                                | Выполните `/acp sessions`, скопируйте точный ключ/метку и повторите попытку.                                                                                                                        |
| `--bind here requires running /acp spawn inside an active ... conversation` | `--bind here` использован без активной привязываемой беседы.                                                            | Перейдите в целевой чат/канал и повторите попытку или используйте запуск без привязки.                                                                                                         |
| `Conversation bindings are unavailable for <channel>.`                      | У адаптера нет возможности ACP для привязки текущей беседы.                                                             | Используйте `/acp spawn ... --thread ...`, где это поддерживается, настройте `bindings[]` верхнего уровня или перейдите в поддерживаемый канал.                                                     |
| `--thread here requires running /acp spawn inside an active ... thread`     | `--thread here` использован вне контекста потока.                                                                         | Перейдите в целевой поток или используйте `--thread auto`/`off`.                                                                                                                      |
| `Only <user-id> can rebind this channel/conversation/thread.`               | Другой пользователь владеет активной целью привязки.                                                                           | Выполните повторную привязку от имени владельца или используйте другую беседу либо поток.                                                                                                               |
| `Thread bindings are unavailable for <channel>.`                            | У адаптера нет возможности привязки потоков.                                                                               | Используйте `--thread off` или перейдите в поддерживаемый адаптер/канал.                                                                                                                 |
| `Sandboxed sessions cannot spawn ACP sessions ...`                          | Среда выполнения ACP работает на стороне хоста; запрашивающий сеанс находится в песочнице.                                                              | Используйте `runtime="subagent"` из сеансов в песочнице или запускайте ACP из сеанса без песочницы.                                                                         |
| `sessions_spawn sandbox="require" is unsupported for runtime="acp" ...`     | Для среды выполнения ACP запрошено `sandbox="require"`.                                                                         | Используйте `runtime="subagent"` для обязательной песочницы или используйте ACP с `sandbox="inherit"` из сеанса без песочницы.                                                      |
| `Cannot apply --model ... did not advertise model support`                  | Целевая обвязка не предоставляет универсальное переключение моделей ACP.                                                        | Используйте обвязку, которая объявляет ACP `models`/`session/set_model`, используйте ссылки на модели Codex ACP или настройте модель напрямую в обвязке, если у нее есть собственный флаг запуска. |
| Отсутствуют метаданные ACP для привязанного сеанса                                      | Устаревшие/удаленные метаданные сеанса ACP.                                                                                    | Создайте заново с помощью `/acp spawn`, затем повторно привяжите/сфокусируйте поток.                                                                                                                    |
| `AcpRuntimeError: Permission prompt unavailable in non-interactive mode`    | `permissionMode` блокирует запись/выполнение в неинтерактивном сеансе ACP.                                                    | Установите `plugins.entries.acpx.config.permissionMode` в `approve-all` и перезапустите gateway. См. [Настройка разрешений](/ru/tools/acp-agents-setup#permission-configuration). |
| Сеанс ACP быстро завершается с малым объемом вывода                                  | Запросы разрешений заблокированы `permissionMode`/`nonInteractivePermissions`.                                        | Проверьте журналы gateway на наличие `AcpRuntimeError`. Для полных разрешений установите `permissionMode=approve-all`; для плавной деградации установите `nonInteractivePermissions=deny`.        |
| Сеанс ACP зависает на неопределенное время после завершения работы                       | Процесс обвязки завершился, но сеанс ACP не сообщил о завершении.                                                    | Обновите OpenClaw; текущая очистка acpx при закрытии и запуске Gateway завершает устаревшие процессы обертки и адаптера, принадлежащие OpenClaw.                                             |
| Обвязка видит `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>`                        | Внутренний конверт события просочился через границу ACP.                                                                | Обновите OpenClaw и повторно выполните поток завершения; внешние обвязки должны получать только обычные запросы на завершение.                                                          |

<Note>
`Command blocked by PreToolUse hook: Native hook relay unavailable` относится к
нативному ретранслятору хуков Codex, а не к ACP/acpx. В привязанном чате Codex начните новый
сеанс с `/new` или `/reset`; если это сработало один раз, а затем ошибка вернулась при следующем
нативном вызове инструмента, перезапустите app-server Codex или Gateway OpenClaw вместо
повторного выполнения `/new`. См. [Устранение неполадок обвязки Codex](/ru/plugins/codex-harness#troubleshooting).
</Note>
