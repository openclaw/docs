Субагенты — это фоновые запуски агентов, порождённые из существующего запуска агента.
Каждый из них выполняется в собственном сеансе (`agent:<agentId>:subagent:<uuid>`) и
после завершения **сообщает** свой результат обратно в канал чата запрашивающего агента.
Каждый запуск субагента отслеживается как [фоновая задача](/ru/automation/tasks).

Цели:

- Параллельно выполнять исследования, длительные задачи и медленные операции с инструментами, не блокируя основной запуск.
- По умолчанию изолировать субагентов (раздельные сеансы, необязательная песочница).
- Не допускать опасного использования набора инструментов: по умолчанию субагенты **не** получают инструменты для работы с сеансами или сообщениями.
- Поддерживать настраиваемую глубину вложенности для шаблонов оркестрации.

<Note>
**Примечание о стоимости:** по умолчанию у каждого субагента собственный контекст и
собственное потребление токенов. Для ресурсоёмких или повторяющихся задач назначьте
субагентам более дешёвую модель, а для основного агента оставьте более качественную
с помощью `agents.defaults.subagents.model` или переопределений для отдельных агентов.
Если дочернему агенту действительно нужен текущий диалог запрашивающего агента,
порождайте его с `context: "fork"`. Для сеансов субагентов, привязанных к ветке обсуждения,
по умолчанию используется `context: "fork"`, поскольку они ответвляют текущий разговор
в последующую ветку.
</Note>

## Команда с косой чертой

`/subagents` позволяет просматривать запуски субагентов для **текущего сеанса**:

```text
/subagents list
/subagents log <id|#> [limit] [tools]
/subagents info <id|#>
```

`/subagents info` показывает метаданные запуска (состояние, временные метки, идентификатор
сеанса, путь к расшифровке, очистку). `/subagents log` выводит последние реплики чата
для запуска; добавьте токен `tools`, чтобы включить сообщения о вызовах инструментов и
их результатах (по умолчанию они опущены). Используйте `sessions_history` для ограниченного,
отфильтрованного с учётом безопасности просмотра истории из хода агента либо откройте
путь к расшифровке на диске, чтобы просмотреть полную необработанную расшифровку.

### Управление привязкой к ветке

Эти команды работают в каналах с постоянной привязкой к веткам. См.
[Каналы с поддержкой веток](#thread-supporting-channels) ниже.

```text
/focus <subagent-label|session-key|session-id|session-label>
/unfocus
/agents
/session idle <duration|off>
/session max-age <duration|off>
```

### Поведение при порождении

Агенты запускают фоновых субагентов с помощью инструмента `sessions_spawn`.
Результаты завершения возвращаются как внутренние события родительского сеанса;
родительский или запрашивающий агент решает, требуется ли видимое пользователю обновление.

<AccordionGroup>
  <Accordion title="Неблокирующее завершение с доставкой по событию">
    - `sessions_spawn` не блокирует выполнение и немедленно возвращает идентификатор запуска.
    - После завершения субагент отправляет отчёт обратно в родительский или запрашивающий сеанс.
    - Ходы агента, которым нужны результаты дочерних агентов, должны вызывать `sessions_yield` после порождения необходимой работы. Это завершает текущий ход и позволяет событию завершения поступить как следующее сообщение, видимое модели.
    - Завершение доставляется по событию. После порождения **не** опрашивайте циклически `/subagents list`, `sessions_list` или `sessions_history` только ради ожидания завершения; проверяйте состояние по запросу лишь при отладке.
    - Вывод дочернего агента представляет собой отчёт или доказательства, которые должен обобщить запрашивающий агент. Это не текст инструкций от пользователя, и он не может переопределять системные правила, правила разработчика или пользователя.
    - После завершения OpenClaw по возможности закрывает отслеживаемые вкладки браузера и процессы, открытые сеансом этого субагента, прежде чем продолжить процедуру очистки после объявления.

  </Accordion>
  <Accordion title="Доставка результата завершения">
    - OpenClaw передаёт результаты завершения обратно в запрашивающий сеанс через ход `agent` со стабильным ключом идемпотентности.
    - Если запуск запрашивающего агента ещё активен, OpenClaw сначала пытается пробудить или перенаправить этот запуск, а не создавать второй видимый путь ответа.
    - Если активный запрашивающий агент не может быть пробуждён, OpenClaw передаёт управление запрашивающему агенту с тем же контекстом завершения, а не отбрасывает объявление.
    - Успешная передача родительскому агенту завершает доставку результата субагента, даже если родитель решает, что видимое пользователю обновление не требуется.
    - Встроенные субагенты не получают инструмент сообщений. Они возвращают обычный текст ответа ассистента родительскому или запрашивающему агенту; видимые человеку ответы остаются под управлением стандартной политики доставки родительского или запрашивающего агента.

    - Доставка сохраняет определённый маршрут запрашивающего агента: при наличии приоритет имеют маршруты завершения, привязанные к ветке или разговору. Если источник завершения указывает только канал, OpenClaw заполняет отсутствующую цель или учётную запись из определённого маршрута сеанса запрашивающего агента (`lastChannel` / `lastTo` / `lastAccountId`), чтобы прямая доставка продолжала работать.

    - Инструкцию по окончательному обновлению для случая, когда дальнейшие действия не требуются, написанную обычным голосом ассистента без передачи необработанных внутренних метаданных.

## Режимы контекста

Встроенные субагенты запускаются изолированно, если вызывающая сторона явно не запрашивает
ответвление текущей расшифровки.

| Режим      | Когда использовать                                                                                                                                | Поведение                                                                                 |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `isolated` | Новое исследование, независимая реализация, медленные операции с инструментами или любая задача, которую можно полноценно описать в тексте задания | Создаёт чистую расшифровку дочернего агента. Это режим по умолчанию, снижающий расход токенов. |
| `fork`     | Работа, зависящая от текущего разговора, предыдущих результатов инструментов или нюансов инструкций, уже присутствующих в расшифровке запрашивающего агента | Ответвляет расшифровку запрашивающего агента в дочерний сеанс до запуска дочернего агента. |

Используйте `fork` умеренно. Он предназначен для делегирования с учётом контекста,
а не для замены чётко сформулированного задания.

## Инструмент: `sessions_spawn`

Запускает субагента с `deliver: false` в глобальном канале выполнения `subagent`,
затем выполняет этап объявления и публикует ответ с объявлением в канал чата
запрашивающего агента.

Доступность зависит от фактической политики инструментов вызывающей стороны. Встроенный
профиль `coding` включает `sessions_spawn`; профили `messaging` и `minimal` — нет.
Профиль `full` разрешает все инструменты. Добавьте `tools.alsoAllow: ["sessions_spawn",
"sessions_yield", "subagents"]` или используйте `tools.profile: "coding"` для
агентов с более узким профилем, которым всё же нужно делегировать работу.
Политики разрешения и запрета для каналов, групп, поставщиков, песочниц и отдельных
агентов по-прежнему могут удалить инструмент после применения профиля. Используйте
`/tools` в том же сеансе, чтобы проверить фактический список инструментов.

**Значения по умолчанию:**

- **Модель:** встроенные субагенты наследуют модель вызывающей стороны, если не задано `agents.defaults.subagents.model` (или `agents.list[].subagents.model` для отдельного агента). Запуски среды выполнения ACP используют ту же настроенную модель субагента, если она задана; в противном случае средство ACP сохраняет собственное значение по умолчанию. Явно заданное `sessions_spawn.model` всегда имеет приоритет.
- **Рассуждение:** встроенные субагенты наследуют настройки вызывающей стороны, если не задано `agents.defaults.subagents.thinking` (или `agents.list[].subagents.thinking` для отдельного агента). Запуски среды выполнения ACP также применяют `agents.defaults.models["provider/model"].params.thinking` для выбранной модели. Явно заданное `sessions_spawn.thinking` всегда имеет приоритет.
- **Тайм-аут запуска:** OpenClaw использует `agents.defaults.subagents.runTimeoutSeconds`, если это значение задано; в противном случае применяется `0` (без тайм-аута). `sessions_spawn` не принимает переопределения тайм-аута для отдельных вызовов.
- **Доставка задания:** встроенные субагенты получают делегированное задание в первом видимом сообщении `[Subagent Task]`. Системный запрос субагента содержит правила среды выполнения и контекст маршрутизации, а не скрытую копию задания.

Результаты принятых порождений встроенных субагентов содержат метаданные определённой
модели дочернего агента: `resolvedModel` содержит применённую ссылку на модель, а
`resolvedProvider` — префикс поставщика, если он присутствует в ссылке.

### Режим запроса для делегирования

`agents.defaults.subagents.delegationMode` управляет только рекомендациями в запросе; он не изменяет политику инструментов и не принуждает к делегированию.

- `suggest` (по умолчанию): сохраняет стандартную рекомендацию использовать субагентов для более крупных или медленных задач.
- `prefer`: предписывает основному агенту сохранять отзывчивость и делегировать через `sessions_spawn` всё, что сложнее прямого ответа.

Переопределение для отдельного агента: `agents.list[].subagents.delegationMode`.

```json5
{
  agents: {
    defaults: {
      subagents: {
        delegationMode: "prefer",
        maxConcurrent: 4,
      },
    },
    list: [
      {
        id: "coordinator",
        subagents: { delegationMode: "prefer" },
      },
    ],
  },
}
```

свою последнюю реплику ассистента обратно инициатору запроса; внешняя доставка остаётся
за родительским агентом или агентом-инициатором.
</Warning>

### Имена задач и выбор целей

`taskName` — доступный модели идентификатор для оркестрации, а не ключ сеанса.
Используйте его для стабильных имён дочерних агентов, таких как `review_subagents`,
`linux_validation` или `docs_update`, когда координатору может потребоваться позже
проверить этот дочерний агент.

При разрешении цели принимаются точные совпадения `taskName` и однозначные
префиксы. Поиск ограничен тем же окном активных и недавних целей, которое
используется для нумерованных целей `/subagents`, поэтому устаревший завершённый
дочерний агент не делает повторно используемый идентификатор неоднозначным. Если два
активных или недавних дочерних агента имеют одинаковый `taskName`, цель неоднозначна;
вместо него используйте индекс списка, ключ сеанса или идентификатор запуска.

Зарезервированные цели `last` и `all` недопустимы в качестве значений `taskName`,
поскольку у них уже есть управляющее значение.

## Инструмент: `sessions_yield`

Завершает текущую реплику модели и ожидает событий среды выполнения, прежде всего
событий завершения субагентов, которые поступят в качестве следующего сообщения. Используйте
его после запуска обязательной дочерней работы, если инициатор запроса не может подготовить
окончательный ответ до получения результатов её завершения.

`sessions_yield` — примитив ожидания. Не заменяйте его циклами опроса
`subagents`, `sessions_list`, `sessions_history`, командой оболочки
`sleep` или опросом процессов только ради обнаружения завершения дочернего агента.

Используйте `sessions_yield` только в том случае, если эффективный список инструментов сеанса
содержит его. Некоторые минимальные или пользовательские профили инструментов могут предоставлять
`sessions_spawn` и `subagents`, но не `sessions_yield`; в таком случае не создавайте
цикл опроса только ради ожидания завершения.

При наличии активных дочерних агентов OpenClaw внедряет компактный, сгенерированный средой выполнения
блок подсказки `Active Subagents` в обычные реплики, чтобы инициатор запроса мог видеть
текущие дочерние сеансы, идентификаторы запусков, состояния, метки, задачи и
псевдонимы `taskName` без опроса. Поля задачи и метки в этом
блоке заключены в кавычки как данные, а не инструкции, поскольку они могут происходить
из предоставленных пользователем или моделью аргументов запуска.

## Инструмент: `subagents`

Выводит список запусков субагентов, принадлежащих сеансу инициатора запроса. Область действия
ограничена текущим инициатором запроса; дочерний агент может видеть только собственных подконтрольных дочерних агентов.

Используйте `subagents` для получения состояния по запросу и отладки. Используйте `sessions_yield` для
ожидания событий завершения.

## Сеансы с привязкой к ветке

Когда для канала включены привязки к веткам, субагент может оставаться привязанным
к ветке, чтобы последующие сообщения пользователя в этой ветке продолжали направляться
в тот же сеанс субагента.

### Каналы с поддержкой веток

Канал поддерживает постоянные сеансы субагентов с привязкой к ветке
(`sessions_spawn` с `thread: true`), если он регистрирует адаптер привязки
диалога. Встроенные каналы с такой поддержкой: **Discord**,
**iMessage**, **Matrix** и **Telegram**. Discord и Matrix по умолчанию
создают дочернюю ветку; Telegram и iMessage по умолчанию привязывают
текущий диалог. Используйте параметры конфигурации `threadBindings` соответствующего канала
для включения, настройки времени ожидания и `spawnSessions`.

### Краткая последовательность

<Steps>
  <Step title="Запуск">
    `sessions_spawn` с `thread: true` (и при необходимости `mode: "session"`).
  </Step>
  <Step title="Привязка">
    OpenClaw создаёт или привязывает ветку к целевому сеансу в активном канале.
  </Step>
  <Step title="Маршрутизация последующих сообщений">
    Ответы и последующие сообщения в этой ветке направляются в привязанный сеанс.
  </Step>
  <Step title="Проверка времени ожидания">
    Используйте `/session idle` для просмотра и изменения автоматического снятия фокуса при бездействии и
    `/session max-age` для управления жёстким ограничением.
  </Step>
  <Step title="Отсоединение">
    Используйте `/unfocus` для отсоединения вручную.
  </Step>
</Steps>

### Ручное управление

| Команда             | Действие                                                                                                      |
| ------------------- | ------------------------------------------------------------------------------------------------------------- |
| `/focus <target>`   | Привязать текущую ветку к целевому субагенту или сеансу либо создать для этого ветку                           |
| `/unfocus`          | Удалить привязку текущей привязанной ветки                                                                    |
| `/agents`           | Вывести активные запуски и состояние привязки (`binding:<id>`, `unbound` или `bindings unavailable`)          |
| `/session idle`     | Просмотреть или изменить автоматическое снятие фокуса при бездействии (только для привязанных веток в фокусе) |
| `/session max-age`  | Просмотреть или изменить жёсткое ограничение (только для привязанных веток в фокусе)                          |

### Переключатели конфигурации

- **Глобальные значения по умолчанию:** `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`.
- **Переопределения для каналов и параметры автоматической привязки при запуске** зависят от адаптера. См. раздел [Каналы с поддержкой веток](#thread-supporting-channels) выше.

Актуальные сведения об адаптерах см. в разделах [Справочник по конфигурации](/ru/gateway/configuration-reference) и
[Команды с косой чертой](/ru/tools/slash-commands).

### Список разрешений

<ParamField path="agents.list[].subagents.allowAgents" type="string[]">
  Список идентификаторов настроенных агентов, которые можно явно выбирать через `agentId` (`["*"]` разрешает любую настроенную цель). По умолчанию разрешён только агент-инициатор. Если вы задаёте список и хотите, чтобы инициатор запроса мог запускать самого себя с помощью `agentId`, включите в список идентификатор инициатора запроса.
</ParamField>
<ParamField path="agents.defaults.subagents.allowAgents" type="string[]">
  Список разрешённых целевых настроенных агентов по умолчанию, используемый, когда агент-инициатор не задаёт собственный `subagents.allowAgents`.
</ParamField>
<ParamField path="agents.defaults.subagents.requireAgentId" type="boolean" default="false">
  Блокировать вызовы `sessions_spawn`, в которых отсутствует `agentId` (требует явного выбора профиля). Переопределение для отдельного агента: `agents.list[].subagents.requireAgentId`.
</ParamField>
<ParamField path="agents.defaults.subagents.announceTimeoutMs" type="number" default="120000">
  Время ожидания для каждой попытки доставки объявления `agent` через Gateway. Значения задаются положительным целым числом миллисекунд и ограничиваются безопасным для платформы максимальным значением таймера. Из-за повторных попыток при временных сбоях общее ожидание объявления может превышать одно настроенное время ожидания.
</ParamField>

Если сеанс инициатора запроса работает в песочнице, `sessions_spawn` отклоняет цели,
которые будут выполняться вне песочницы.

### Обнаружение

Используйте `agents_list`, чтобы узнать, какие идентификаторы агентов сейчас разрешены для
`sessions_spawn`. Ответ содержит эффективную модель каждого указанного агента и встроенные
метаданные среды выполнения, чтобы вызывающая сторона могла различать OpenClaw, сервер приложения Codex
и другие настроенные нативные среды выполнения.

Записи `allowAgents` должны указывать на идентификаторы настроенных агентов в `agents.list[]`.
`["*"]` означает любого настроенного целевого агента, а также инициатора запроса. Если конфигурация
агента удалена, но его идентификатор остаётся в `allowAgents`, `sessions_spawn` отклоняет этот
идентификатор, а `agents_list` не выводит его. Запустите `openclaw doctor --fix`, чтобы удалить
устаревшие записи списка разрешений, либо добавьте минимальную запись `agents.list[]`, если цель должна
оставаться доступной для запуска с наследованием значений по умолчанию.

### Автоматическое архивирование

- Сеансы субагентов автоматически архивируются по истечении `agents.defaults.subagents.archiveAfterMinutes` (по умолчанию `60`).
- При архивировании используется `sessions.delete`, а расшифровка переименовывается в `*.deleted.<timestamp>` (в той же папке).
- `cleanup: "delete"` архивирует сеанс сразу после объявления (расшифровка всё равно сохраняется путём переименования).
- Автоматическое архивирование выполняется по мере возможности; ожидающие таймеры теряются при перезапуске Gateway.
- Настроенные ограничения времени выполнения **не** запускают автоматическое архивирование; они только останавливают запуск. Сеанс сохраняется до автоматического архивирования.
- Автоматическое архивирование одинаково применяется к сеансам глубины 1 и 2.
- Очистка браузера выполняется отдельно от очистки архива: отслеживаемые вкладки и процессы браузера по мере возможности закрываются после завершения запуска, даже если расшифровка или запись сеанса сохраняется.

## Вложенные субагенты

По умолчанию субагенты не могут запускать собственных субагентов
(`maxSpawnDepth: 1`). Установите `maxSpawnDepth: 2`, чтобы включить один уровень
вложенности — **шаблон оркестратора**: основной агент → субагент-оркестратор →
субагенты-исполнители второго уровня.

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // разрешить субагентам запускать дочерние агенты (по умолчанию: 1, диапазон 1–5)
        maxChildrenPerAgent: 5, // максимальное число активных дочерних агентов на сеанс агента (по умолчанию: 5, диапазон 1–20)
        maxConcurrent: 8, // глобальное ограничение параллельных запусков (по умолчанию: 8)
        runTimeoutSeconds: 900, // время ожидания по умолчанию для sessions_spawn (0 = без ограничения)
        announceTimeoutMs: 120000, // время ожидания объявления через Gateway для каждого вызова
      },
    },
  },
}
```

### Уровни глубины

| Глубина | Формат ключа сеанса                          | Роль                                               | Может создавать дочерние процессы? |
| ------- | -------------------------------------------- | -------------------------------------------------- | ----------------------------------- |
| 0       | `agent:<id>:main`                            | Основной агент                                     | Всегда                              |
| 1       | `agent:<id>:subagent:<uuid>`                 | Субагент (оркестратор, если разрешена глубина 2)   | Только если `maxSpawnDepth >= 2`    |
| 2       | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | Суб-субагент (конечный исполнитель)                | Никогда                             |

### Цепочка уведомлений

Результаты передаются вверх по цепочке:

1. Исполнитель глубины 2 завершает работу → уведомляет своего родителя (оркестратора глубины 1).
2. Оркестратор глубины 1 получает уведомление, обобщает результаты, завершает работу → уведомляет основной агент.
3. Основной агент получает уведомление и передаёт результат пользователю.

Каждый уровень видит только уведомления от своих непосредственных дочерних агентов.

<Note>
**Рекомендации по эксплуатации:** запускайте дочернюю работу один раз и ожидайте событий завершения вместо построения циклов опроса вокруг `sessions_list`, `sessions_history`, `/subagents list` или команд ожидания `exec`.
`sessions_list` и `/subagents list` сохраняют привязку отношений дочерних сеансов к активной работе: активные дочерние сеансы остаются прикреплёнными, завершённые дочерние сеансы остаются видимыми в течение короткого периода, а устаревшие ссылки на дочерние сеансы, существующие только в хранилище, игнорируются после истечения периода актуальности. Это не позволяет старым метаданным `spawnedBy` /
`parentSessionKey` восстанавливать фантомные дочерние сеансы после перезапуска. Если событие завершения дочернего агента поступило после того, как вы уже отправили окончательный ответ, правильным последующим действием будет точный маркер молчания
`NO_REPLY` / `no_reply`.
</Note>

### Политика инструментов по глубине

- Роль и область управления записываются в метаданные сеанса при создании. Это не позволяет плоским или восстановленным ключам сеансов случайно вновь получить привилегии оркестратора.
- **Глубина 1 (оркестратор, когда `maxSpawnDepth >= 2`):** получает `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history`, чтобы создавать дочерние агенты и проверять их состояние. Остальные сеансовые и системные инструменты остаются запрещёнными.
- **Глубина 1 (конечный исполнитель, когда `maxSpawnDepth == 1`):** без сеансовых инструментов (текущее поведение по умолчанию).
- **Глубина 2 (конечный исполнитель):** без сеансовых инструментов — `sessions_spawn` всегда запрещён на глубине 2. Не может создавать дочерние агенты следующего уровня.

### Ограничение создания дочерних агентов для каждого агента

Каждый сеанс агента (на любой глубине) может одновременно иметь не более `maxChildrenPerAgent`
(по умолчанию `5`) активных дочерних агентов. Это предотвращает неконтролируемое разветвление
из одного оркестратора.

### Каскадная остановка

Остановка оркестратора глубины 1 автоматически останавливает всех его дочерних агентов глубины 2:

- `/stop` в основном чате останавливает всех агентов глубины 1 и каскадно останавливает их дочерние агенты глубины 2.

## Аутентификация

Данные аутентификации субагента определяются по **идентификатору агента**, а не по типу сеанса:

- Ключ сеанса субагента — `agent:<agentId>:subagent:<uuid>`.
- Хранилище данных аутентификации загружается из `agentDir` этого агента.
- Профили аутентификации основного агента объединяются как **резервные**; при конфликтах профили агента переопределяют профили основного агента.

Объединение выполняется с добавлением, поэтому профили основного агента всегда доступны
как резервные. Полностью изолированная аутентификация для каждого агента пока не поддерживается.

## Уведомление

Субагенты сообщают о результатах посредством этапа уведомления:

- Этап уведомления выполняется внутри сеанса субагента (а не сеанса инициатора запроса).
- Если субагент отвечает в точности `ANNOUNCE_SKIP`, ничего не публикуется.
- Если последний текст ассистента представляет собой точный маркер молчания `NO_REPLY` / `no_reply`, вывод уведомления подавляется, даже если ранее отображался видимый ход выполнения.

Доставка зависит от глубины инициатора запроса:

- Сеансы инициаторов верхнего уровня используют последующий вызов `agent` с внешней доставкой (`deliver=true`).
- Вложенные сеансы субагентов-инициаторов получают внутреннюю последующую вставку (`deliver=false`), чтобы оркестратор мог обобщить результаты дочерних агентов внутри сеанса.
- Если вложенный сеанс субагента-инициатора уже отсутствует, OpenClaw при наличии возможности возвращается к инициатору этого сеанса.

Для сеансов инициаторов верхнего уровня прямая доставка в режиме завершения сначала
определяет любой привязанный маршрут беседы/ветки и переопределение обработчика, а затем заполняет
отсутствующие поля канала и адресата из сохранённого маршрута сеанса инициатора.
Благодаря этому результаты доставляются в нужный чат или тему, даже если источник завершения
указывает только канал.

При формировании результатов вложенного завершения агрегация завершений дочерних агентов ограничивается текущим запуском инициатора, что предотвращает попадание устаревших результатов дочерних агентов из предыдущих запусков в текущее уведомление. Ответы уведомлений сохраняют маршрутизацию по ветке/теме, если она доступна в адаптерах каналов.

### Контекст уведомления

Контекст уведомления нормализуется в стабильный внутренний блок события:

| Поле               | Источник                                                                                                             |
| ------------------ | -------------------------------------------------------------------------------------------------------------------- |
| Источник           | `subagent` или `cron`                                                                                                |
| Идентификаторы сеанса | Ключ/идентификатор дочернего сеанса                                                                                |
| Тип                | Тип уведомления + метка задачи                                                                                       |
| Состояние          | Определяется по результату выполнения (`ok`, `error`, `timeout` или `unknown`) — **не** выводится из текста модели   |
| Содержимое результата | Последний видимый текст ассистента от дочернего агента                                                            |
| Последующее действие | Инструкция, описывающая, когда отвечать, а когда сохранять молчание                                                |

Завершившиеся с ошибкой запуски сообщают о состоянии сбоя без повторного вывода
сохранённого текста ответа. Вывод `tool`/`toolResult` не преобразуется в текст результата дочернего агента.

### Строка статистики

Полезная нагрузка уведомлений содержит строку статистики в конце (даже при переносе):

- Время выполнения (например, `runtime 5m12s`).
- Использование токенов (ввод/вывод/всего).
- Ориентировочная стоимость, если настроены цены моделей (`models.providers.*.models[].cost`).
- `sessionKey`, `sessionId` и путь к транскрипту, чтобы основной агент мог получить историю через `sessions_history` или просмотреть файл на диске.

Внутренние метаданные предназначены только для оркестрации; ответы пользователю
следует переписывать обычным языком ассистента.

### Почему следует предпочитать `sessions_history`

`sessions_history` — более безопасный способ оркестрации для чтения транскрипта дочернего агента
в ходе выполнения агента:

- Скрывает текст, похожий на учётные данные или токены, даже когда общее сокрытие конфиденциальных данных в журналах отключено.
- Обрезает длинные текстовые блоки (4000 символов на блок) и удаляет сигнатуры процесса рассуждений, данные повторного воспроизведения рассуждений и встроенные данные изображений.
- Ограничивает размер ответа до 80 КБ; слишком большие строки заменяются на `[sessions_history omitted: message too large]`.
- При наличии `nextOffset` используйте его для постраничного перехода назад к более старым фрагментам транскрипта.
- `sessions_history` **не** удаляет теги рассуждений, служебную структуру `<relevant-memories>` или XML вызовов инструментов из текста сообщений — он возвращает структурированные блоки содержимого, близкие к исходному формату транскрипта, лишь с сокрытием конфиденциальных данных и ограничением размера. `/subagents log` применяет более строгую очистку текста (удаляет теги рассуждений, служебную структуру памяти и XML вызовов инструментов), поскольку отображает обычные строки чата вместо структурированных блоков.
- Непосредственное чтение транскрипта с диска служит резервным вариантом, когда требуется полный транскрипт с точностью до каждого байта.

## Политика инструментов

Субагенты сначала используют тот же профиль и конвейер политик инструментов, что и родительский или
целевой агент. После этого OpenClaw применяет слой ограничений субагента.

Субагенты всегда лишаются `gateway`, `agents_list`, `session_status` и
`cron` независимо от глубины или роли (это системные/интерактивные инструменты либо
инструменты, работу с которыми должен координировать основной агент). Конечные субагенты (поведение по умолчанию для глубины 1
и всегда на глубине 2) дополнительно лишаются `subagents`,
`sessions_list`, `sessions_history` и `sessions_spawn`. Субагенты никогда
не получают инструмент `message` — он отключается при создании, а не фильтруется
этим списком запретов, — а `sessions_send` остаётся запрещённым, поэтому субагенты
обмениваются данными только через цепочку уведомлений.

`sessions_history` и здесь остаётся ограниченным и очищенным представлением истории —
это не выгрузка необработанного транскрипта.

Когда `maxSpawnDepth >= 2`, субагенты-оркестраторы глубины 1 дополнительно
получают `sessions_spawn`, `subagents`, `sessions_list` и
`sessions_history`, чтобы управлять своими дочерними агентами.

### Переопределение через конфигурацию

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // запрет имеет приоритет
        deny: ["gateway", "cron"],
        // если задан allow, разрешаются только перечисленные инструменты (запрет всё равно имеет приоритет)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

`tools.subagents.tools.allow` — окончательный фильтр разрешений. Он может сузить
уже определённый набор инструментов, но не может **вернуть** инструмент, удалённый
параметром `tools.profile`. Например, `tools.profile: "coding"` включает
`web_search`/`web_fetch`, но не инструмент `browser`. Чтобы разрешить
субагентам с профилем разработки использовать автоматизацию браузера, добавьте браузер на
этапе профиля:

```json5
{
  tools: {
    profile: "coding",
    alsoAllow: ["browser"],
  },
}
```

Используйте `agents.list[].tools.alsoAllow: ["browser"]` для конкретного агента, если автоматизацию браузера
должен получить только один агент.

## Параллелизм

Субагенты используют выделенную внутреннюю очередь процесса:

- **Название очереди:** `subagent`
- **Параллелизм:** `agents.defaults.subagents.maxConcurrent` (по умолчанию `8`)

## Контроль активности и восстановление

OpenClaw не считает отсутствие `endedAt` окончательным доказательством того, что
субагент всё ещё активен. Незавершённые запуски старше окна устаревания
(2 часа или настроенное время ожидания запуска плюс небольшой дополнительный период —
в зависимости от того, что больше) перестают считаться активными/ожидающими в `/subagents list`,
сводках состояния, блокировке завершения потомков и проверках
параллелизма для каждого сеанса.

После перезапуска Gateway устаревшие восстановленные незавершённые запуски удаляются, если
их дочерний сеанс не помечен `abortedLastRun: true`. Запуски, прерванные
перезапуском, остаются зарегистрированными для процесса восстановления осиротевших субагентов: устаревшие
запуски завершаются без возобновления, а свежие дочерние сеансы получают
синтетическое сообщение о возобновлении до снятия маркера прерывания.

Автоматическое восстановление после перезапуска ограничено для каждого дочернего сеанса. Если один и тот же
дочерний субагент неоднократно принимается для восстановления осиротевшего состояния в пределах
короткого окна повторного зависания, OpenClaw сохраняет маркер блокировки восстановления в этом
сеансе и прекращает автоматически возобновлять его при последующих перезапусках. Выполните
`openclaw tasks maintenance --apply`, чтобы согласовать запись задачи, или
`openclaw doctor --fix`, чтобы очистить устаревшие флаги прерванного восстановления в
сеансах с маркером блокировки.

<Note>
Если создание субагента завершается ошибкой Gateway `PAIRING_REQUIRED` /
`scope-upgrade`, проверьте вызывающую сторону RPC, прежде чем изменять состояние сопряжения.
Внутренняя координация `sessions_spawn` выполняется внутри процесса, когда
вызывающая сторона уже работает в контексте запроса Gateway, поэтому она не
открывает WebSocket через local loopback и не зависит от базового набора областей доступа сопряжённого устройства CLI.
Вызывающие стороны вне процесса Gateway по-прежнему используют резервное подключение WebSocket
как `client.id: "gateway-client"` с `client.mode: "backend"`
через прямую аутентификацию общим токеном/паролем по local loopback. Удалённые вызывающие стороны, явно заданные
`deviceIdentity`, явно заданные пути токена устройства и клиенты браузера/Node
по-прежнему требуют обычного одобрения устройства для повышения областей доступа.
</Note>

## Остановка

- Отправка `/stop` в чате инициатора прерывает его сеанс и останавливает все активные запуски субагентов, созданные из него, с каскадной остановкой вложенных дочерних агентов.

## Ограничения

- Уведомление субагента выполняется **по возможности**. Если Gateway перезапустится, ожидающие операции обратного уведомления будут потеряны.
- Субагенты по-прежнему совместно используют ресурсы одного процесса Gateway; рассматривайте `maxConcurrent` как предохранительный механизм.
- `sessions_spawn` всегда выполняется без блокировки: он немедленно возвращает `{ status: "accepted", runId, childSessionKey }`.
- В контекст субагента добавляются только `AGENTS.md` и `TOOLS.md` (без `SOUL.md`, `IDENTITY.md`, `USER.md`, `MEMORY.md`, `HEARTBEAT.md` и `BOOTSTRAP.md`). Нативные субагенты Codex следуют тому же ограничению: `TOOLS.md` остаётся в унаследованных инструкциях потока Codex, а файлы персоны, идентичности и пользователя, предназначенные только для родительского агента, добавляются как инструкции по совместной работе в рамках текущего хода, чтобы дочерние агенты их не клонировали.
- Максимальная глубина вложенности — 5 (диапазон `maxSpawnDepth`: 1–5). Для большинства сценариев рекомендуется глубина 2.
- `maxChildrenPerAgent` ограничивает количество активных дочерних агентов на сеанс (по умолчанию `5`, диапазон `1–20`).
