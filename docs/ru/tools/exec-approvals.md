Подтверждения выполнения — это **защитный механизм приложения-компаньона / хоста Node**, позволяющий изолированному агенту выполнять команды на реальном хосте (`gateway` или `node`). Команды выполняются, только когда политика, список разрешений и необязательное подтверждение пользователя согласованы между собой. Подтверждения действуют **поверх** политики инструментов и проверки повышенных привилегий (режим повышенных привилегий `full` их пропускает).

Обзор режимов `deny`, `allowlist`, `ask`, `auto`, `full`, сопоставления Codex Guardian и разрешений среды ACPX см. в разделе [Режимы разрешений](/ru/tools/permission-modes).

<Note>
Эффективная политика выбирается как **более строгая** из `tools.exec.*` и значений подтверждений по умолчанию: подтверждения могут только ужесточать параметры безопасности и запросов, полученные из конфигурации, но никогда не ослаблять их. Если поле подтверждений не задано, используется значение `tools.exec`. Выполнение на хосте также использует локальное состояние подтверждений на этой машине: локальное для хоста значение `ask: "always"` в файле подтверждений хоста выполнения продолжает запрашивать подтверждение, даже если значения сеанса или конфигурации по умолчанию задают `ask: "on-miss"`.
</Note>

## Где это применяется

Подтверждения выполнения применяются локально на хосте выполнения:

- **Хост Gateway** -> процесс `openclaw` на машине Gateway.
- **Хост Node** -> исполнитель Node (приложение-компаньон macOS или безголовый хост Node).

### Модель доверия

- Вызывающие стороны, прошедшие аутентификацию в Gateway, считаются доверенными операторами этого Gateway.
- Сопряжённые узлы распространяют возможности доверенного оператора на хост Node.
- Подтверждения снижают риск случайного выполнения, но **не** являются границей авторизации между пользователями или политикой доступа к файловой системе только для чтения.
- После подтверждения команда может изменять файлы в соответствии с разрешениями файловой системы выбранного хоста или песочницы.
- Подтверждённые запуски на хосте Node привязываются к каноническому контексту выполнения: cwd, точному argv, привязке env при её наличии и закреплённому пути к исполняемому файлу, когда это применимо.
- Для сценариев оболочки и прямых вызовов файлов через интерпретатор или среду выполнения OpenClaw также пытается привязать один конкретный локальный файловый операнд. Если этот файл изменится после подтверждения, но до выполнения, запуск будет отклонён вместо выполнения изменившегося содержимого.
- Привязка файла выполняется по мере возможности и не является полной моделью всех путей загрузки интерпретаторов и сред выполнения. Если невозможно определить ровно один конкретный локальный файл, OpenClaw отказывается создавать запуск на основе подтверждения, а не имитирует полное покрытие.

### Разделение в macOS

- **Служба хоста Node** пересылает `system.run` в **приложение macOS** через локальный IPC.
- **Приложение macOS** применяет подтверждения и выполняет команду в контексте UI.

## Проверка эффективной политики

| Команда                                                           | Что она показывает                                                                                   |
| ----------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `openclaw approvals get` / `--gateway` / `--node <id\|name\|ip>`  | Запрошенную политику, источники политики хоста и эффективный результат.                              |
| `openclaw exec-policy show`                                       | Объединённое представление для локальной машины.                                                     |
| `openclaw exec-policy set` / `preset`                              | Синхронизацию локальной запрошенной политики с локальным файлом подтверждений хоста за один шаг.     |

<Note>
Переопределения `/exec` для отдельных сеансов не включаются. Выполните `/exec` в соответствующем сеансе, чтобы проверить его текущие значения по умолчанию. См. [переопределения сеанса](/ru/tools/exec#session-overrides-exec).
</Note>

Полный справочник CLI (флаги, вывод JSON, добавление и удаление записей списка разрешений): [CLI подтверждений](/ru/cli/approvals).

Когда локальная область запрашивает `host=node`, `exec-policy show` сообщает, что во время выполнения эта область управляется Node, вместо того чтобы считать локальный файл подтверждений источником достоверных данных.

Если UI приложения-компаньона **недоступен**, любой запрос, который обычно требует подтверждения, разрешается согласно **резервной политике запросов** (по умолчанию: `deny`).

<Tip>
Нативные клиенты подтверждений в чатах могут добавлять в ожидающее сообщение о подтверждении специальные элементы управления для конкретного канала. Matrix добавляет быстрые реакции (`✅` разрешить один раз, `♾️` разрешать всегда, `❌` отклонить), при этом сохраняя в сообщении `/approve ...` как резервный вариант.
</Tip>

## Настройки и хранение

Подтверждения хранятся в локальном JSON-файле на хосте выполнения. Если задана переменная `OPENCLAW_STATE_DIR`, файл размещается в указанном каталоге состояния; в противном случае используется стандартный каталог состояния OpenClaw:

```text
$OPENCLAW_STATE_DIR/exec-approvals.json
# в противном случае
~/.openclaw/exec-approvals.json
```

Стандартный сокет подтверждений использует тот же корневой каталог:
`$OPENCLAW_STATE_DIR/exec-approvals.sock` или
`~/.openclaw/exec-approvals.sock`, если переменная не задана.

В выпусках до 2026.6.6 файл всегда хранился в `~/.openclaw`. Если `OPENCLAW_STATE_DIR` указывает на другой каталог, а файл подтверждений всё ещё существует в стандартном каталоге, один раз выполните напрямую `openclaw doctor --fix`, чтобы импортировать его в каталог состояния (исходный файл архивируется с суффиксом `.migrated`). Интерактивный doctor также позволяет предварительно просмотреть и подтвердить импорт. Автоматические обновления и запуски восстановления при наблюдении Gateway никогда не выполняют импорт между каталогами состояния: временный или промежуточный каталог состояния не должен перехватывать подтверждения стандартной установки. Та же граница применяется к импорту устаревших файлов `plugin-binding-approvals.json` в общее состояние SQLite.

Пример схемы:

```json
{
  "version": 1,
  "socket": {
    "path": "~/.openclaw/exec-approvals.sock",
    "token": "base64url-token"
  },
  "defaults": {
    "security": "deny",
    "ask": "on-miss",
    "askFallback": "deny",
    "autoAllowSkills": false
  },
  "agents": {
    "main": {
      "security": "allowlist",
      "ask": "on-miss",
      "askFallback": "deny",
      "autoAllowSkills": true,
      "allowlist": [
        {
          "id": "B0C8C0B3-2C2D-4F8A-9A3C-5A4B3C2D1E0F",
          "pattern": "~/Projects/**/bin/rg",
          "source": "allow-always",
          "lastUsedAt": 1737150000000,
          "lastUsedCommand": "rg -n TODO",
          "lastResolvedPath": "/Users/user/Projects/.../bin/rg"
        }
      ]
    }
  }
}
```

## Параметры политики

### `tools.exec.mode`

`tools.exec.mode` — предпочтительная нормализованная поверхность политики выполнения на хосте:

| Значение    | Поведение                                                                                                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `deny`      | Блокирует выполнение на хосте.                                                                                                                                                                  |
| `allowlist` | Выполняет без запроса только команды из списка разрешений.                                                                                                                                      |
| `ask`       | Использует политику списка разрешений и запрашивает подтверждение при отсутствии совпадения.                                                                                                    |
| `auto`      | Использует политику списка разрешений, напрямую выполняет детерминированные совпадения и передаёт запросы без совпадений встроенному автоматическому рецензенту OpenClaw, прежде чем обращаться к человеку. |
| `full`      | Выполняет команды на хосте без запросов подтверждения.                                                                                                                                          |

Устаревшие `tools.exec.security` / `tools.exec.ask` по-прежнему поддерживаются и применяются везде, где для соответствующей области не задан `mode`.

### `exec.security`

<ParamField path="security" type='"deny" | "allowlist" | "full"'>
  - `deny` — блокировать все запросы выполнения на хосте.
  - `allowlist` — разрешать только команды из списка разрешений.
  - `full` — разрешать всё (эквивалентно повышенным привилегиям).

Для хостов Gateway/Node по умолчанию используется `full`; для хоста `sandbox` вместо этого по умолчанию используется `deny`.
</ParamField>

### `exec.ask`

<ParamField path="ask" type='"off" | "on-miss" | "always"'>
  Настроенная политика запросов подтверждения для выполнения на хосте. Управляет базовым поведением запросов подтверждения, определяемым `tools.exec.ask` и стандартными значениями подтверждений хоста. По умолчанию используется `off`. Параметр инструмента `ask` для отдельного вызова (см. [Инструмент Exec](/ru/tools/exec#parameters)) может только ужесточить эту базовую политику, а вызовы модели из каналов игнорируют его, когда эффективное значение запросов хоста равно `off`.

- `off` — никогда не запрашивать подтверждение.
- `on-miss` — запрашивать подтверждение, только если список разрешений не совпал.
- `always` — запрашивать подтверждение для каждой команды. Постоянное доверие `allow-always` **не** подавляет запросы, когда эффективный режим запросов равен `always`.

</ParamField>

### `askFallback`

<ParamField path="askFallback" type='"deny" | "allowlist" | "full"'>
  Способ разрешения ситуации, когда запрос подтверждения обязателен, но UI недоступен либо истекло время ожидания. Если значение не задано, по умолчанию используется `deny`.

- `deny` — блокировать.
- `allowlist` — разрешить, только если есть совпадение со списком разрешений.
- `full` — разрешить.

</ParamField>

### `tools.exec.strictInlineEval`

<ParamField path="strictInlineEval" type="boolean">
  При значении `true` формы вычисления встроенного кода требуют подтверждения, даже если сам исполняемый файл интерпретатора находится в списке разрешений. Это дополнительный уровень защиты для загрузчиков интерпретаторов, которые невозможно однозначно сопоставить с одним стабильным файловым операндом.
</ParamField>

Примеры, которые перехватывает строгий режим: `python -c`, `node -e`/`--eval`/`-p`,
`ruby -e`, `perl -e`/`-E`, `php -r`, `lua -e`, `osascript -e` (а также встроенные формы `awk`,
`sed`, `make`, `find -exec` и `xargs`).

В строгом режиме этим командам требуется решение рецензента или явное подтверждение. При `tools.exec.mode: "auto"` рецензент может разрешить одно выполнение с низким риском, если команда имеет принудительно исполнимый план; в противном случае OpenClaw запрашивает подтверждение у человека.
Подтверждения команд `Codex app-server`, переданные резервному рецензенту, требуют решения человека, поскольку их запросы подтверждения не содержат принудительно исполнимого разрешённого исполняемого файла.
`allow-always` не сохраняет новые записи списка разрешений для команд вычисления встроенного кода.

### `tools.exec.commandHighlighting`

<ParamField path="commandHighlighting" type="boolean" default="false">
  Только представление: если параметр включён, OpenClaw может прикреплять полученные синтаксическим анализатором диапазоны команд, чтобы веб-запросы подтверждения могли подсвечивать токены команды. Это **не** изменяет `security`, `ask`, сопоставление со списком разрешений, поведение строгого вычисления встроенного кода, пересылку подтверждений или выполнение команд.
</ParamField>

Задайте глобально в `tools.exec.commandHighlighting` или для отдельного агента в
`agents.list[].tools.exec.commandHighlighting`.

## Режим YOLO (без подтверждений)

Чтобы выполнять команды на хосте без запросов подтверждения, откройте **оба** уровня политики:
запрошенную политику выполнения в конфигурации OpenClaw (`tools.exec.*`) **и**
локальную политику подтверждений хоста в файле подтверждений хоста выполнения.

Если `askFallback` не задан, по умолчанию используется `deny`. Явно задайте для хоста `askFallback` значение `full`, если при недоступном UI запрос подтверждения должен разрешать выполнение.

| Уровень               | Настройка YOLO             |
| --------------------- | -------------------------- |
| `tools.exec.security` | `full` для `gateway`/`node` |
| `tools.exec.ask`      | `off`                      |
| `askFallback` хоста   | `full`                     |

<Warning>
**Важные различия:**

- `tools.exec.host=auto` определяет, **где** выполняется команда: в песочнице, если она доступна, иначе на Gateway.
- YOLO определяет, **как** подтверждается выполнение на хосте: `security=full` вместе с `ask=off`.
- YOLO **не** добавляет поверх настроенной политики выполнения на хосте отдельную эвристическую проверку подтверждения обфускации команд или уровень предварительного отклонения сценариев.
- `auto` не превращает маршрутизацию на Gateway в свободно доступное переопределение из изолированного сеанса. Запрос `host=node` для отдельного вызова разрешён из `auto`; `host=gateway` разрешён из `auto`, только если среда выполнения песочницы не активна. Для стабильного значения по умолчанию без `auto` задайте `tools.exec.host` или явно используйте `/exec host=...`.

</Warning>

Провайдеры на базе CLI, предоставляющие собственный неинтерактивный режим разрешений, могут следовать этой политике. Claude CLI добавляет `--permission-mode bypassPermissions`, когда действующая политика выполнения OpenClaw имеет значение YOLO. Для управляемых OpenClaw активных сеансов Claude действующая политика выполнения OpenClaw имеет приоритет над собственным режимом разрешений Claude: при YOLO активные запуски нормализуются до `--permission-mode bypassPermissions`, а при ограничительной действующей политике выполнения — до `--permission-mode default`, даже если необработанные аргументы бэкенда Claude задают другой режим.

Если вам нужна более консервативная конфигурация, ужесточите политику выполнения OpenClaw, вернув значение `allowlist` / `on-miss` или `deny`.

### Постоянная настройка «никогда не запрашивать» на хосте Gateway

<Steps>
  <Step title="Установите требуемую политику конфигурации">
    ```bash
    openclaw config set tools.exec.host gateway
    openclaw config set tools.exec.security full
    openclaw config set tools.exec.ask off
    openclaw gateway restart
    ```
  </Step>
  <Step title="Приведите файл разрешений хоста в соответствие">
    ```bash
    openclaw approvals set --stdin <<'EOF'
    {
      version: 1,
      defaults: {
        security: "full",
        ask: "off",
        askFallback: "full"
      }
    }
    EOF
    ```
  </Step>
</Steps>

### Локальная сокращённая команда

```bash
openclaw exec-policy preset yolo
```

Обновляет как локальные значения `tools.exec.host/security/ask`, так и значения по умолчанию в локальном файле разрешений, включая `askFallback: "full"`. Эта команда намеренно действует только локально. Чтобы удалённо изменить разрешения хоста Gateway или хоста Node, используйте `openclaw approvals set --gateway` или `openclaw approvals set --node
<id|name|ip>`.

Другие встроенные предустановки: `cautious` (`host=gateway`, `security=allowlist`, `ask=on-miss`, `askFallback=deny`) и `deny-all` (`host=gateway`, `security=deny`, `ask=off`, `askFallback=deny`). Применяйте их тем же способом: `openclaw exec-policy preset cautious`.

Чтобы вместо полной предустановки задать отдельные поля, используйте
`openclaw exec-policy set --host <auto|sandbox|gateway|node> --security
<deny|allowlist|full> --ask <off|on-miss|always> --ask-fallback
<deny|allowlist|full>` с любым подмножеством этих флагов.

### Хост Node

Вместо этого примените тот же файл разрешений на Node:

```bash
openclaw approvals set --node <id|name|ip> --stdin <<'EOF'
{
  version: 1,
  defaults: {
    security: "full",
    ask: "off",
    askFallback: "full"
  }
}
EOF
```

<Note>
**Ограничения локального режима:**

- `openclaw exec-policy` не синхронизирует разрешения Node.
- `openclaw exec-policy set --host node` отклоняется.
- Разрешения выполнения Node запрашиваются у Node во время выполнения, поэтому для обновлений, предназначенных для Node, необходимо использовать `openclaw approvals --node ...`.

</Note>

### Сокращённая команда только для сеанса

- `/exec security=full ask=off` изменяет только текущий сеанс.
- `/elevated full` — аварийная сокращённая команда, которая пропускает проверки разрешений выполнения, только когда и запрошенная политика, и файл разрешений хоста разрешаются в `security: "full"` и `ask: "off"`. При более строгом файле хоста, например с `ask:
"always"`, запрос всё равно отображается.

Если файл разрешений хоста остаётся строже конфигурации, более строгая политика хоста по-прежнему имеет приоритет.

## Список разрешённых команд (для каждого агента)

Списки разрешённых команд задаются **для каждого агента**. Если существует несколько агентов, выберите в приложении macOS агента, настройки которого хотите изменить. Шаблоны сопоставляются как glob-шаблоны.

Шаблонами могут быть glob-шаблоны разрешённых путей к исполняемым файлам или glob-шаблоны простых имён команд. Простые имена соответствуют только командам, вызываемым через `PATH`, поэтому `rg` может соответствовать `/opt/homebrew/bin/rg`, если команда — `rg`, но **не** `./rg` или `/tmp/rg`. Чтобы считать доверенным исполняемый файл только в одном конкретном расположении, используйте glob-шаблон пути.

Устаревшие записи `agents.default` при загрузке переносятся в `agents.main`.
В цепочках команд оболочки, таких как `echo ok && pwd`, каждый сегмент верхнего уровня по-прежнему должен соответствовать правилам списка разрешённых команд.

Примеры:

- `rg`
- `~/Projects/**/bin/peekaboo`
- `~/.local/bin/*`
- `/opt/homebrew/bin/rg`

### Ограничение аргументов с помощью argPattern

Добавьте `argPattern`, если запись списка разрешённых команд должна соответствовать исполняемому файлу и определённой структуре аргументов. OpenClaw использует семантику регулярных выражений ECMAScript (JavaScript) на каждом хосте и применяет выражение к разобранным аргументам команды, исключая токен исполняемого файла (`argv[0]`). Для записей, созданных вручную, аргументы объединяются одним пробелом, поэтому используйте якоря шаблона, когда требуется точное совпадение.

```json
{
  "version": 1,
  "agents": {
    "main": {
      "allowlist": [
        {
          "pattern": "python3",
          "argPattern": "^safe\\.py$"
        }
      ]
    }
  }
}
```

Эта запись разрешает `python3 safe.py`; для `python3 other.py` совпадение в списке разрешённых команд отсутствует. Если также присутствует запись только с путём для того же исполняемого файла, несовпадающие аргументы всё ещё могут использовать эту запись в качестве резервного варианта. Не добавляйте запись только с путём, если требуется разрешить для исполняемого файла лишь объявленные аргументы.

Записи, сохранённые через процессы подтверждения, используют внутренний формат разделителей для точного сопоставления `argv`. Вместо ручного редактирования закодированного значения рекомендуется заново создать такие записи через интерфейс или процесс подтверждения. Если OpenClaw не может разобрать `argv` для сегмента команды, записи с `argPattern` не совпадают.

Каждая запись списка разрешённых команд поддерживает:

| Поле               | Значение                                                         |
| ------------------ | ---------------------------------------------------------------- |
| `pattern`          | Glob-шаблон разрешённого пути к исполняемому файлу или имени команды |
| `argPattern`       | Необязательное регулярное выражение ECMAScript для argv; при отсутствии проверяется только путь |
| `id`               | Стабильный непрозрачный идентификатор; при отсутствии создаётся UUID |
| `source`           | Источник записи, например `allow-always`                          |
| `commandText`      | Устаревший ввод в виде обычного текста; отбрасывается при загрузке |
| `lastUsedAt`       | Временная метка последнего использования                         |
| `lastUsedCommand`  | Последняя совпавшая команда                                      |
| `lastResolvedPath` | Последний разрешённый путь к исполняемому файлу                  |

## Автоматическое разрешение CLI из Skills

Когда включено **автоматическое разрешение CLI из Skills** (`autoAllowSkills`), исполняемые файлы, на которые ссылаются известные Skills, считаются включёнными в список разрешённых команд на Node (Node macOS или хосте Node без графического интерфейса). Для получения списка исполняемых файлов Skills используется `skills.bins` через RPC Gateway. Отключите эту функцию, если нужны строгие списки, управляемые вручную.

<Warning>
- Это **неявный список разрешённых команд для удобства**, отдельный от записей с путями, добавленных вручную.
- Он предназначен для доверенных операторских сред, в которых Gateway и Node находятся в одной границе доверия.
- Если требуется строгое явное доверие, оставьте `autoAllowSkills: false` и используйте только добавленные вручную записи с путями.

</Warning>

## Безопасные исполняемые файлы и перенаправление запросов подтверждения

Сведения о безопасных исполняемых файлах (быстром пути только через стандартный ввод), привязке интерпретатора и перенаправлении запросов подтверждения в Slack/Discord/Telegram либо их обработке собственными клиентами подтверждения см. в разделе [Расширенные разрешения выполнения](/ru/tools/exec-approvals-advanced).

## Редактирование в интерфейсе управления

Используйте карточку **Интерфейс управления -> Узлы -> Разрешения выполнения**, чтобы редактировать значения по умолчанию, переопределения для отдельных агентов и списки разрешённых команд. Выберите область действия (значения по умолчанию или агент), измените политику, добавьте или удалите шаблоны списка разрешённых команд, затем нажмите **Сохранить**. Интерфейс показывает метаданные последнего использования для каждого шаблона, что помогает поддерживать список в порядке.

Переключатель цели позволяет выбрать **Gateway** (локальные разрешения) или **Node**.
Node должны объявлять поддержку `system.execApprovals.get/set` (приложение macOS или хост Node без графического интерфейса). Если Node ещё не объявляет поддержку разрешений выполнения, отредактируйте его локальный файл разрешений напрямую.

Некоторые хосты Node, включая вспомогательное приложение Windows, используют другой формат политики подтверждений. Интерфейс управления отображает такие собственные политики хоста только для чтения. Для их редактирования используйте вспомогательное приложение или `openclaw approvals set --node <id|name|ip>` с собственным форматом политики; см. [CLI разрешений](/ru/cli/approvals).

CLI: `openclaw approvals` поддерживает редактирование Gateway или Node — см.
[CLI разрешений](/ru/cli/approvals).

## Процесс подтверждения

Когда требуется запрос, Gateway рассылает событие `exec.approval.requested` операторским клиентам. Интерфейс управления и приложение macOS обрабатывают его через `exec.approval.resolve`, после чего Gateway перенаправляет подтверждённый запрос хосту Node.

Для `host=node` запросы подтверждения включают каноническую полезную нагрузку `systemRunPlan`. Gateway использует этот план как авторитетный контекст команды, `cwd` и сеанса при перенаправлении подтверждённых запросов `system.run`:

- Путь выполнения Node заранее подготавливает один канонический план.
- Запись подтверждения хранит этот план и метаданные его привязки.
- После подтверждения окончательный перенаправленный вызов `system.run` повторно использует сохранённый план, а не доверяет последующим изменениям вызывающей стороны.
- Если после создания запроса подтверждения вызывающая сторона изменяет `command`, `rawCommand`, `cwd`, `agentId` или `sessionKey`, Gateway отклоняет перенаправленный запуск из-за несоответствия подтверждению.

## Системные события и отказы

После того как Node сообщает о завершении, жизненный цикл выполнения публикует в сеансе агента системное сообщение `Выполнение завершено`. OpenClaw также может отправить уведомление о продолжающемся выполнении после выдачи разрешения, когда истечёт `tools.exec.approvalRunningNoticeMs` (по умолчанию `10000`; `0` отключает уведомление). Отклонение разрешения выполнения окончательно прекращает обработку команды хоста: команда не запускается.

- Для асинхронных подтверждений основного агента с исходным сеансом OpenClaw публикует отказ обратно в этот сеанс как внутреннее последующее сообщение, чтобы агент прекратил ожидать асинхронную команду и не инициировал восстановление отсутствующего результата.
- Если сеанс отсутствует или его невозможно возобновить, OpenClaw всё равно может кратко сообщить об отказе оператору или в маршрут прямого чата.
- Отказы для сеансов субагентов и Cron обратно в эти сеансы не публикуются.

Разрешения выполнения на хосте Gateway создают такое же событие жизненного цикла завершения.
Для выполнения, требующего подтверждения, идентификатор подтверждения используется повторно, чтобы связать ожидающий запрос с сообщением о его завершении или отклонении (`Выполнение завершено (gateway
id=...)` / `Выполнение отклонено (gateway id=...)`).

## Последствия

- **`full`** предоставляет широкие возможности; по возможности предпочитайте списки разрешённых команд.
- **`ask`** позволяет контролировать процесс, сохраняя возможность быстро выдавать разрешения.
- Списки разрешённых команд для отдельных агентов предотвращают распространение разрешений одного агента на других.
- Разрешения применяются только к запросам выполнения на хосте от **авторизованных отправителей**. Неавторизованные отправители не могут вызывать `/exec`.
- `/exec security=full` — удобная настройка уровня сеанса для авторизованных операторов, которая намеренно пропускает подтверждения. Чтобы полностью запретить выполнение на хосте, установите для безопасности разрешений значение `deny` или запретите инструмент `exec` через политику инструментов.
