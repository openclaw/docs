Когда агент работает внутри песочницы, его команды `exec` ограничены средой песочницы. **Привилегированный режим** позволяет агенту выйти из неё и выполнять команды за пределами песочницы с настраиваемыми этапами подтверждения.

<Info>
  Привилегированный режим изменяет поведение, только когда агент работает **в песочнице**. Для агентов без песочницы `exec` уже выполняется на хосте.
</Info>

## Директивы

Управляйте привилегированным режимом для каждого сеанса с помощью слеш-команд:

| Директива        | Назначение                                                                                                                                                |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/elevated on`   | Выполнять команды вне песочницы по настроенному пути на хосте, сохраняя подтверждения                                                                     |
| `/elevated ask`  | То же, что и `on` (псевдоним)                                                                                                                             |
| `/elevated full` | Выполнять команды вне песочницы по настроенному пути на хосте и пропускать подтверждения, если политика подтверждений режима и хоста уже разрешает это     |
| `/elevated off`  | Вернуться к выполнению команд в пределах песочницы                                                                                                        |

Также доступны команды `/elev on|off|ask|full`.

Отправьте `/elevated` без аргументов, чтобы увидеть текущий уровень.

## Принцип работы

<Steps>
  <Step title="Проверка доступности">
    Привилегированный режим должен быть включён в конфигурации, а отправитель должен находиться в списке разрешённых:

    ```json5
    {
      tools: {
        elevated: {
          enabled: true,
          allowFrom: {
            discord: ["user-id-123"],
            whatsapp: ["+15555550123"],
          },
        },
      },
    }
    ```

  </Step>

  <Step title="Установка уровня">
    Чтобы задать значение по умолчанию для сеанса, отправьте сообщение, содержащее только директиву:

    ```
    /elevated full
    ```

    Или укажите её в строке сообщения (применяется только к этому сообщению):

    ```
    /elevated on run the deployment script
    ```

  </Step>

  <Step title="Выполнение команд вне песочницы">
    При активном привилегированном режиме вызовы `exec` выполняются вне песочницы. По умолчанию фактическим хостом является
    `gateway`, либо `node`, если настроенной или сеансовой целью выполнения `exec`
    является `node`. В режиме `full` подтверждения `exec` пропускаются, если итоговая
    политика подтверждений режима выполнения и хоста уже полностью разрешающая (безопасность `full`,
    запрос `off`); в противном случае продолжает действовать обычная политика подтверждений. В режимах
    `on`/`ask` настроенные правила подтверждения применяются всегда.
  </Step>
</Steps>

## Порядок разрешения

1. **Встроенная директива** в сообщении (применяется только к этому сообщению)
2. **Переопределение сеанса** (задаётся отправкой сообщения, содержащего только директиву)
3. **Глобальное значение по умолчанию** (`agents.defaults.elevatedDefault` в конфигурации)

## Доступность и списки разрешённых

- **Глобальный шлюз**: `tools.elevated.enabled` (должен иметь значение `true`)
- **Список разрешённых отправителей**: `tools.elevated.allowFrom` со списками для каждого канала
- **Шлюз отдельного агента**: `agents.list[].tools.elevated.enabled` (может только дополнительно ограничивать доступ; и глобальный шлюз, и шлюз агента должны иметь значение `true`)
- **Список разрешённых отдельного агента**: `agents.list[].tools.elevated.allowFrom` (отправитель должен соответствовать как глобальному списку, так и списку агента)
- **Резервный список разрешённых от канала**: плагины каналов могут при необходимости предоставлять резервный список разрешённых через хук адаптера SDK, который используется, если `tools.elevated.allowFrom.<provider>` не настроен. Сейчас ни один встроенный плагин канала не реализует этот хук, поэтому на практике каждому провайдеру требуется явная запись `tools.elevated.allowFrom.<provider>`.
- **Все проверки должны быть пройдены**; в противном случае привилегированный режим считается недоступным

Форматы записей списка разрешённых:

| Префикс                 | Соответствие                                    |
| ----------------------- | ----------------------------------------------- |
| (нет)                   | ID отправителя, номер E.164 или поле From       |
| `name:`                 | Отображаемое имя отправителя                    |
| `username:`             | Имя пользователя отправителя                    |
| `tag:`                  | Тег отправителя                                 |
| `id:`, `from:`, `e164:` | Явное указание идентификатора                   |

## Чем привилегированный режим не управляет

- **Политика инструментов**: если `exec` запрещён политикой инструментов, привилегированный режим не может обойти этот запрет.
- **Политика выбора хоста**: привилегированный режим не превращает `auto` в возможность свободно выбирать любой хост. Он использует настроенные или сеансовые правила выбора цели `exec`, выбирая `node`, только если целью уже является `node`.
- **Не связан с `/exec`**: директива `/exec` изменяет сеансовые значения `exec` по умолчанию (хост, безопасность, подтверждение, Node) для авторизованных отправителей и не требует привилегированного режима.

<Note>
  Чат-команда bash (префикс `!`; псевдоним `/bash`) имеет отдельный шлюз доступа, для которого, помимо собственного флага `tools.bash.enabled`, должен быть включён параметр `tools.elevated`. Отключение привилегированного режима также блокирует команды оболочки `!`.
</Note>
