В распространённой конфигурации с разделением по хостам OpenClaw Gateway работает внутри WSL2, Chrome — в Windows, а управление браузером должно пересекать границу WSL2/Windows. Одновременно могут проявиться несколько независимых проблем (см.
[issue #39369](https://github.com/openclaw/openclaw/issues/39369)): транспорт CDP, безопасность источника Control UI и токен/сопряжение могут отказать независимо друг от друга, при этом выдавая похожие ошибки. Последовательно проверьте описанные ниже уровни, не пытаясь угадать, какой из них неисправен.

## Сначала выберите правильный режим браузера

### Вариант 1: прямое удалённое подключение CDP из WSL2 к Windows

Используйте профиль удалённого браузера, направляющий запросы из WSL2 к конечной точке CDP Chrome в Windows. Выберите этот вариант, если Gateway остаётся внутри WSL2, Chrome работает в Windows, а управление браузером должно пересекать границу WSL2/Windows.

### Вариант 2: локальный для хоста Chrome MCP

Используйте драйвер `existing-session` (профиль `user`) только в том случае, если Gateway работает на том же хосте, что и Chrome, вам нужно локальное состояние браузера с выполненным входом, не требуется межхостовый транспорт браузера и не нужны `responsebody`, экспорт PDF, перехват загрузок или пакетные действия (профили Chrome MCP их не поддерживают).

Для сочетания Gateway в WSL2 и Chrome в Windows используйте прямое удалённое подключение CDP. Chrome MCP работает локально на хосте и не является мостом между WSL2 и Windows.

## Рабочая архитектура

- WSL2 запускает Gateway на `127.0.0.1:18789`
- Windows открывает Control UI в обычном браузере по адресу `http://127.0.0.1:18789/`
- Chrome в Windows предоставляет конечную точку CDP на порту `9222`
- WSL2 может обращаться к этой конечной точке CDP в Windows
- Профиль браузера OpenClaw указывает на адрес, доступный из WSL2

## Критически важное правило для Control UI

При открытии интерфейса из Windows используйте localhost Windows, если у вас нет намеренно настроенной конфигурации HTTPS:

```text
http://127.0.0.1:18789/
```

Не используйте IP-адрес локальной сети по умолчанию. Обычный HTTP на адресе локальной сети или tailnet может вызвать поведение, связанное с небезопасным источником или аутентификацией устройства и не относящееся к самому CDP. См.
[Control UI](/ru/web/control-ui).

## Выполняйте проверку по уровням

Двигайтесь сверху вниз, не пропуская этапы. После исправления одного уровня всё ещё может отображаться другая ошибка с последующего уровня.

### Уровень 1: убедитесь, что Chrome предоставляет CDP в Windows

```powershell
chrome.exe --remote-debugging-port=9222 --user-data-dir="$env:LOCALAPPDATA\OpenClaw\ChromeCDP"
```

Chrome 136 и более поздних версий игнорирует параметры командной строки удалённой отладки для каталога данных Chrome по умолчанию. Используйте отдельный каталог данных, отличный от каталога по умолчанию, как показано выше. См. описание
[изменения безопасности удалённой отладки](https://developer.chrome.com/blog/remote-debugging-port)
в Chrome. Это не делает обычный профиль Chrome с выполненным входом доступным для удалённого управления.

Сначала проверьте сам Chrome из Windows:

```powershell
curl.exe http://127.0.0.1:9222/json/version
curl.exe http://127.0.0.1:9222/json/list
```

Если проверка завершается неудачей, выполните диагностику прослушивающих сокетов Windows, описанную ниже. На этом этапе проблема ещё не связана с OpenClaw.

#### Проверьте IPv4 и IPv6 перед изменением portproxy

Chromium сначала пытается привязать удалённую отладку к `127.0.0.1` и переходит на `[::1]`, только если привязка IPv4 завершается неудачей. Постоянное правило `v4tov4`, прослушивающее `127.0.0.1:9222`, может занять эту конечную точку до запуска Chrome. Тогда Chrome переходит на `[::1]:9222`, а старое правило перенаправляет трафик IPv4 обратно в собственный прослушивающий сокет и возвращает пустой ответ.

Проверьте фактические прослушивающие сокеты и правила прокси-сервера из Windows, а не делайте выводы на основании версии Chrome:

```powershell
netstat -ano | findstr :9222
netsh interface portproxy show all
curl.exe http://127.0.0.1:9222/json/version
curl.exe http://[::1]:9222/json/version
```

Для каждого PID из вывода `netstat` используйте `tasklist /fi "PID eq <PID>"`.

- Если `chrome.exe` отвечает на `127.0.0.1`, удалите все правила portproxy, которые также прослушивают `127.0.0.1:9222`. Перенаправляйте только адрес адаптера Windows, доступный из WSL2, на `127.0.0.1`.
- Если `chrome.exe` отвечает только на `[::1]`, направьте доступный из WSL2 прослушивающий сокет на `::1` с помощью `v4tov6`, а не на неиспользуемый адрес IPv4:

  ```powershell
  netsh interface portproxy add v4tov6 listenaddress=WINDOWS_HOST_OR_IP listenport=9222 connectaddress=::1 connectport=9222
  ```

Привяжите прослушивающий сокет к адресу адаптера, который нужен WSL2. Не открывайте порт CDP на `0.0.0.0`, адресе локальной сети или адресе tailnet: CDP предоставляет управление сеансом браузера.

### Уровень 2: убедитесь, что WSL2 может обращаться к этой конечной точке Windows

Из WSL2 проверьте точный адрес, который планируется использовать в `cdpUrl`:

```bash
curl http://WINDOWS_HOST_OR_IP:9222/json/version
curl http://WINDOWS_HOST_OR_IP:9222/json/list
```

Признаки успешного результата:

- `/json/version` возвращает JSON с метаданными Browser / Protocol-Version
- `/json/list` возвращает JSON (пустой массив допустим, если не открыто ни одной страницы)

Если проверка завершается неудачей, порт Windows ещё не доступен из WSL2, адрес неверен для стороны WSL2 либо отсутствует необходимая настройка брандмауэра, перенаправления портов или проксирования. Исправьте это, прежде чем изменять конфигурацию OpenClaw.

### Уровень 3: настройте правильный профиль браузера

Укажите в OpenClaw адрес, доступный из WSL2:

```json5
{
  browser: {
    enabled: true,
    defaultProfile: "remote",
    profiles: {
      remote: {
        cdpUrl: "http://WINDOWS_HOST_OR_IP:9222",
        attachOnly: true,
        color: "#00AA00",
      },
    },
  },
}
```

Примечания:

- используйте адрес, доступный из WSL2, а не адрес, работающий только в Windows
- сохраняйте `attachOnly: true` для браузеров с внешним управлением
- `cdpUrl` может использовать `http://`, `https://`, `ws://` или `wss://`
- используйте HTTP(S), если OpenClaw должен обнаруживать `/json/version`
- используйте WS(S), только если поставщик браузера предоставляет прямой URL сокета DevTools
- проверьте тот же URL с помощью `curl`, прежде чем ожидать успешной работы OpenClaw

### Уровень 4: отдельно проверьте уровень Control UI

Откройте `http://127.0.0.1:18789/` из Windows, затем убедитесь, что:

- источник страницы соответствует ожиданиям `gateway.controlUi.allowedOrigins`
- аутентификация по токену или сопряжение настроены правильно
- проблема аутентификации Control UI не диагностируется ошибочно как проблема браузера

Полезная страница: [Control UI](/ru/web/control-ui).

### Уровень 5: проверьте сквозное управление браузером

Из WSL2:

```bash
openclaw browser --browser-profile remote open https://example.com
openclaw browser --browser-profile remote tabs
```

Признаки успешного результата:

- вкладка открывается в Chrome в Windows
- `browser tabs` возвращает целевой объект
- последующие действия (`snapshot`, `screenshot`, `navigate`) работают с тем же профилем

## Распространённые вводящие в заблуждение ошибки

| Сообщение                                                                               | Значение                                                                                                                                                                                                |
| --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `control-ui-insecure-auth`                                                              | проблема источника интерфейса или безопасного контекста, а не транспорта CDP                                                                                                                            |
| `token_missing`                                                                         | проблема конфигурации аутентификации                                                                                                                                                                    |
| `pairing required`                                                                      | проблема подтверждения устройства                                                                                                                                                                       |
| `Remote CDP for profile "remote" is not reachable`                                      | WSL2 не может обратиться к настроенному `cdpUrl`                                                                                                                                                         |
| пустой ответ CDP / `other side closed` при использовании portproxy                      | несоответствие прослушивающих сокетов Windows или зацикливание; проверьте оба семейства loopback-адресов и вывод `netsh interface portproxy show all`                                                     |
| `Browser attachOnly is enabled and CDP websocket for profile "remote" is not reachable` | конечная точка HTTP ответила, но не удалось открыть WebSocket DevTools                                                                                                                                   |
| устаревшие параметры области просмотра, тёмного режима, локали или автономного режима после удалённого сеанса | выполните `openclaw browser --browser-profile remote stop`, чтобы закрыть сеанс и освободить кэшированное соединение Playwright/CDP без перезапуска Gateway или внешнего браузера                         |
| истечение времени ожидания около `remoteCdpTimeoutMs` (по умолчанию 1500 мс)             | обычно проблема по-прежнему связана с доступностью CDP либо с медленной или недоступной удалённой конечной точкой                                                                                        |
| `Playwright page enumeration timed out after 3000ms`                                    | подключение к удалённому CDP установлено, но постоянное чтение вкладок остановилось; крайний срок равен большему из значений `remoteCdpTimeoutMs` и `remoteCdpHandshakeTimeoutMs`                          |
| `No Chrome tabs found for profile="user"`                                               | выбран локальный профиль Chrome MCP, но локальные для хоста вкладки недоступны                                                                                                                           |

## Краткий контрольный список диагностики

1. Windows: какой адрес, `127.0.0.1` или `[::1]`, отвечает на `/json/version`, и
   принадлежит ли этот прослушивающий сокет процессу `chrome.exe`?
2. WSL2: работает ли `curl http://WINDOWS_HOST_OR_IP:9222/json/version`?
3. Конфигурация OpenClaw: использует ли `browser.profiles.<name>.cdpUrl` именно
   этот адрес, доступный из WSL2?
4. Control UI: открываете ли вы `http://127.0.0.1:18789/`, а не IP-адрес локальной сети?
5. Не пытаетесь ли вы использовать `existing-session` между WSL2 и Windows
   вместо прямого удалённого подключения CDP?

Сначала локально проверьте конечную точку Chrome в Windows, затем проверьте ту же конечную точку из WSL2 и только после этого диагностируйте конфигурацию OpenClaw или аутентификацию Control UI.
