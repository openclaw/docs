# `openclaw memory`

Управление индексированием семантической памяти, поиском и переносом в `MEMORY.md`.
Предоставляется встроенным плагином `memory-core` и доступно, когда
`plugins.slots.memory` выбирает `memory-core` (по умолчанию). Другие плагины памяти
предоставляют собственные пространства имён CLI.

См. также: концепция [памяти](/ru/concepts/memory), [Dreaming](/ru/concepts/dreaming),
[справочник по конфигурации памяти](/ru/reference/memory-config), [вики памяти](/ru/plugins/memory-wiki),
[вики](/ru/cli/wiki), [плагины](/ru/tools/plugin).

## `memory status`

```bash
openclaw memory status [--agent <id>] [--deep] [--index] [--fix] [--json] [--verbose]
```

Без `--agent` выполняется для каждого агента в `agents.list`; если список агентов
не настроен, используется агент по умолчанию.

| Флаг        | Действие                                                                                                                                                                                                                                                                                                    |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--deep`    | Проверяет готовность векторного хранилища, поставщика эмбеддингов и семантического поиска (предполагает дополнительные обращения к поставщику). Обычный `memory status` выполняется быстро и пропускает эту проверку; неизвестное состояние векторного/семантического поиска означает, что оно не проверялось. Лексический `searchMode: "search"` QMD всегда пропускает семантические векторные проверки, даже с `--deep`. |
| `--index`   | Повторно индексирует, если хранилище содержит незафиксированные изменения. Подразумевает `--deep`.                                                                                                                                                                                                                                                          |
| `--fix`     | Исправляет устаревшие блокировки воспоминаний и нормализует метаданные переноса.                                                                                                                                                                                                                                               |
| `--json`    | Выводит JSON.                                                                                                                                                                                                                                                                                               |
| `--verbose` | Выводит подробные журналы по каждой фазе.                                                                                                                                                                                                                                                                             |

Если строка `Dreaming` остаётся `off` даже с `dreaming.enabled: true` или
запланированные проходы, по-видимому, никогда не выполняются, управляемое задание Cron для Dreaming зависит от
срабатывания Heartbeat агента по умолчанию, которое запускает согласование. Подробности
планирования см. в разделе [Dreaming](/ru/concepts/dreaming).

В статусе также перечисляются дополнительные пути поиска из `agents.defaults.memorySearch.extraPaths`.

## `memory index`

```bash
openclaw memory index [--agent <id>] [--force] [--verbose]
```

Область действия по агентам совпадает с `status`. `--force` запускает полную повторную индексацию вместо
инкрементальной. `--verbose` выводит сведения о поставщике, модели, источниках и
дополнительных путях для каждого агента перед отображением хода индексации.

## `memory search`

```bash
openclaw memory search [query] [--query <text>] [--agent <id>] [--max-results <n>] [--min-score <n>] [--json]
```

- Запрос: позиционный `[query]` или `--query <text>`. Если заданы оба, приоритет имеет `--query`.
  Если не задан ни один, команда завершается с ошибкой.
- `--agent <id>`: по умолчанию используется агент по умолчанию (а не полный список агентов).
- `--max-results <n>`: ограничивает количество результатов (положительное целое число).
- `--min-score <n>`: исключает совпадения с оценкой ниже указанной.

## `memory promote`

Ранжирует краткосрочные кандидаты из `memory/YYYY-MM-DD.md` и при необходимости добавляет
лучшие записи в `MEMORY.md`.

```bash
openclaw memory promote [--agent <id>] [--limit <n>] [--min-score <n>] \
  [--min-recall-count <n>] [--min-unique-queries <n>] [--apply] [--include-promoted] [--json]
```

| Флаг                       | По умолчанию      | Действие                                                            |
| -------------------------- | ------------ | ----------------------------------------------------------------- |
| `--limit <n>`              |              | Максимальное количество возвращаемых/применяемых кандидатов.                                   |
| `--min-score <n>`          | `0.75`       | Минимальная взвешенная оценка переноса.                                 |
| `--min-recall-count <n>`   | `3`          | Минимальное необходимое количество воспоминаний.                                    |
| `--min-unique-queries <n>` | `2`          | Минимальное необходимое количество различных запросов.                            |
| `--apply`                  | только предварительный просмотр | Добавляет выбранных кандидатов в `MEMORY.md` и отмечает их как перенесённых. |
| `--include-promoted`       |              | Включает кандидатов, уже перенесённых в предыдущих циклах.           |
| `--json`                   |              | Выводит JSON.                                                       |

Эти значения CLI по умолчанию отличаются от порогов глубокой фазы запланированного
прохода Dreaming (см. раздел [Dreaming](#dreaming) ниже); чтобы однократный ручной запуск
соответствовал поведению прохода, передайте флаги явно.

Сигналы ранжирования: частота воспоминаний, релевантность извлечения, разнообразие запросов,
временная актуальность, консолидация между днями и насыщенность производными концепциями, полученные
как из обращений к памяти, так и из ежедневных проходов загрузки, а также небольшое усиление
в лёгкой/REM-фазе за повторные обращения Dreaming. Перед записью процесс переноса
повторно читает актуальную ежедневную заметку, поэтому изменения или удаления краткосрочных фрагментов
после ранжирования учитываются и перенос не выполняется из устаревшего снимка.

## `memory promote-explain`

Объясняет составляющие оценки одного кандидата на перенос.

```bash
openclaw memory promote-explain <selector> [--agent <id>] [--include-promoted] [--json]
```

`<selector>` сопоставляется с ключом кандидата (точно или по подстроке), путём или текстом
фрагмента.

## `memory rem-harness`

Предварительно показывает размышления REM, возможные факты и результат переноса глубокой фазы,
ничего не записывая.

```bash
openclaw memory rem-harness [--agent <id>] [--path <file-or-dir>] [--grounded] [--include-promoted] [--json]
```

- `--path <file-or-dir>`: заполняет стенд из исторических ежедневных файлов
  `YYYY-MM-DD.md` вместо текущей рабочей области.
- `--grounded`: также формирует на основе исторических заметок
  предварительный результат `What Happened` / `Reflections` / `Possible Lasting Updates`.

## `memory rem-backfill`

Записывает основанные на исторических данных сводки REM в `DREAMS.md` для проверки в пользовательском интерфейсе.
Обратимо.

```bash
openclaw memory rem-backfill --path <file-or-dir> [--agent <id>] [--stage-short-term] [--json]
openclaw memory rem-backfill --rollback [--rollback-short-term] [--json]
```

- `--path <file-or-dir>`: обязателен, если не задан `--rollback`/`--rollback-short-term`.
  Исторические файлы ежедневной памяти или каталог, из которых выполняется обратное заполнение.
- `--stage-short-term`: также добавляет обоснованных устойчивых кандидатов в текущее
  краткосрочное хранилище переноса, чтобы обычная глубокая фаза могла их ранжировать.
- `--rollback`: удаляет ранее записанные обоснованные дневниковые записи из
  `DREAMS.md`.
- `--rollback-short-term`: удаляет ранее подготовленных обоснованных краткосрочных
  кандидатов.

## Dreaming

Dreaming — фоновая система консолидации памяти с тремя совместно работающими
фазами, которые выполняются по порядку в рамках одного расписания: **лёгкая** (сортировка/подготовка краткосрочного
материала), **REM** (анализ и выявление тем), **глубокая** (перенос устойчивых
фактов в `MEMORY.md`). Только глубокая фаза выполняет запись в `MEMORY.md`.

- Включите с помощью `plugins.entries.memory-core.config.dreaming.enabled: true`
  (по умолчанию `false`); `memory-core` автоматически управляет заданием Cron для прохода, ручная настройка
  `openclaw cron add` не требуется.
- Переключайте из чата с помощью `/dreaming on|off`; проверяйте с помощью `/dreaming status`
  (или `/dreaming`/`/dreaming help`). Для `on`/`off` требуется статус владельца канала
  или `operator.admin` Gateway; `status` и справка доступны всем, кто
  может вызвать команду.
- Понятный человеку результат фаз записывается в `DREAMS.md` (или существующий `dreams.md`).
  По умолчанию (`dreaming.storage.mode: "separate"`) каждая фаза также записывает
  отдельный отчёт в `memory/dreaming/<phase>/YYYY-MM-DD.md`; задайте `mode:
"inline"`, чтобы включать отчёты в файл ежедневной памяти, или `"both"`
  для обоих вариантов.
- Запланированные и ручные запуски `memory promote` используют одинаковые сигналы
  ранжирования глубокой фазы; различаются только пороговые значения по умолчанию (см. таблицу выше и
  запланированные значения по умолчанию ниже).
- Запланированные запуски распределяются по рабочим областям памяти всех настроенных агентов.

Запланированные значения по умолчанию (`plugins.entries.memory-core.config.dreaming`):

| Ключ                                    | По умолчанию     |
| -------------------------------------- | ----------- |
| `frequency`                            | `0 3 * * *` |
| `phases.deep.minScore`                 | `0.8`       |
| `phases.deep.minRecallCount`           | `3`         |
| `phases.deep.minUniqueQueries`         | `3`         |
| `phases.deep.recencyHalfLifeDays`      | `14`        |
| `phases.deep.maxAgeDays`               | `30`        |
| `phases.deep.maxPromotedSnippetTokens` | `160`       |

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Полный список ключей и подробности фаз: [Dreaming](/ru/concepts/dreaming),
[справочник по конфигурации памяти](/ru/reference/memory-config#dreaming).

## Зависимость SecretRef от Gateway

Если поля ключей удалённого API Active Memory настроены как SecretRef, команды `memory`
разрешают их из активного снимка Gateway; если Gateway
недоступен, команда немедленно завершается с ошибкой. Для этого требуется Gateway, поддерживающий
метод `secrets.resolve`; более старые версии Gateway возвращают ошибку неизвестного метода.
