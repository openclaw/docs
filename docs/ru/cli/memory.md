# `openclaw memory`

Управление индексированием семантической памяти, поиском и переносом данных в `MEMORY.md`.
Предоставляется встроенным Plugin `memory-core` и доступно, когда
`plugins.slots.memory` выбирает `memory-core` (по умолчанию). Другие плагины памяти
предоставляют собственные пространства имён CLI.

Связанные материалы: концепция [памяти](/ru/concepts/memory), [Dreaming](/ru/concepts/dreaming),
[справочник по конфигурации памяти](/ru/reference/memory-config), [вики памяти](/ru/plugins/memory-wiki),
[вики](/ru/cli/wiki), [плагины](/ru/tools/plugin).

## `memory status`

```bash
openclaw memory status [--agent <id>] [--deep] [--index] [--fix] [--json] [--verbose]
```

Без `--agent` выполняется для каждого агента из `agents.list`; если список агентов
не настроен, используется агент по умолчанию.

| Флаг        | Действие                                                                                                                                                                                                                                                                                                                                 |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--deep`    | Проверить готовность векторного хранилища, поставщика эмбеддингов и семантического поиска (требует дополнительных обращений к поставщику). Обычная команда `memory status` выполняется быстро и пропускает эту проверку; неизвестное состояние векторного или семантического поиска означает, что проверка не проводилась. Лексический режим QMD `searchMode: "search"` всегда пропускает семантические векторные проверки, даже с `--deep`. |
| `--index`   | Повторно индексировать, если хранилище содержит изменения. Подразумевает `--deep`.                                                                                                                                                                                                                                                         |
| `--fix`     | Исправить устаревшие блокировки извлечения и нормализовать метаданные переноса.                                                                                                                                                                                                                                                            |
| `--json`    | Вывести JSON.                                                                                                                                                                                                                                                                                                                             |
| `--verbose` | Вывести подробные журналы каждого этапа.                                                                                                                                                                                                                                                                                                   |

Если строка `Dreaming` остаётся в состоянии `off` даже при
`dreaming.enabled: true` или запланированные проходы не запускаются, учтите:
управляемая задача Cron для Dreaming зависит от срабатывания Heartbeat агента
по умолчанию, которое запускает синхронизацию. Подробности о расписании см. в
разделе [Dreaming](/ru/concepts/dreaming).

В статусе также перечисляются дополнительные пути поиска из `agents.defaults.memorySearch.extraPaths`.

## `memory index`

```bash
openclaw memory index [--agent <id>] [--force] [--verbose]
```

Область действия по агентам такая же, как у `status`. `--force` запускает полную
переиндексацию вместо инкрементальной. Перед отображением хода индексирования
`--verbose` выводит сведения о поставщике, модели, источниках и дополнительных
путях для каждого агента.

## `memory search`

```bash
openclaw memory search [query] [--query <text>] [--agent <id>] [--max-results <n>] [--min-score <n>] [--json]
```

- Запрос: позиционный аргумент `[query]` или `--query <text>`. Если заданы оба,
  используется `--query`. Если не задан ни один, команда завершается с ошибкой.
- `--agent <id>`: по умолчанию используется агент по умолчанию, а не полный список агентов.
- `--max-results <n>`: ограничить количество результатов положительным целым числом.
- `--min-score <n>`: исключить совпадения с оценкой ниже указанной.

## `memory promote`

Ранжировать кандидатов краткосрочной памяти из `memory/YYYY-MM-DD.md` и при
необходимости добавлять лучшие записи в `MEMORY.md`.

```bash
openclaw memory promote [--agent <id>] [--limit <n>] [--min-score <n>] \
  [--min-recall-count <n>] [--min-unique-queries <n>] [--apply] [--include-promoted] [--json]
```

| Флаг                       | Значение по умолчанию | Действие                                                                      |
| -------------------------- | --------------------- | ----------------------------------------------------------------------------- |
| `--limit <n>`              |                       | Максимальное количество возвращаемых или применяемых кандидатов.              |
| `--min-score <n>`          | `0.75`                | Минимальная взвешенная оценка переноса.                                        |
| `--min-recall-count <n>`   | `3`                   | Минимальное необходимое количество извлечений.                                |
| `--min-unique-queries <n>` | `2`                   | Минимальное необходимое количество различных запросов.                        |
| `--apply`                  | только предварительный просмотр | Добавить выбранных кандидатов в `MEMORY.md` и отметить их как перенесённых. |
| `--include-promoted`       |                       | Включить кандидатов, уже перенесённых в предыдущих циклах.                     |
| `--json`                   |                       | Вывести JSON.                                                                 |

Эти значения CLI по умолчанию отличаются от пороговых значений глубокой фазы
запланированного прохода Dreaming (см. раздел [Dreaming](#dreaming) ниже);
чтобы при разовом ручном запуске воспроизвести поведение прохода, передайте
флаги явно.

Сигналы ранжирования: частота извлечения, релевантность найденного, разнообразие
запросов, временная актуальность, объединение данных за разные дни и богатство
производных концепций. Они формируются как из извлечений памяти, так и из
проходов ежедневного добавления данных, а также включают небольшое усиление
после лёгкой фазы и фазы REM при повторных обращениях Dreaming. Перед записью
процесс переноса заново читает актуальную ежедневную заметку, поэтому изменения
или удаления фрагментов краткосрочной памяти после ранжирования учитываются и
данные не переносятся из устаревшего снимка.

## `memory promote-explain`

Показать структуру оценки одного кандидата на перенос.

```bash
openclaw memory promote-explain <selector> [--agent <id>] [--include-promoted] [--json]
```

`<selector>` сопоставляется с ключом кандидата (полностью или по подстроке),
путём или текстом фрагмента.

## `memory rem-harness`

Предварительно просмотреть размышления REM, потенциально достоверные сведения
и результат переноса глубокой фазы без записи данных.

```bash
openclaw memory rem-harness [--agent <id>] [--path <file-or-dir>] [--grounded] [--include-promoted] [--json]
```

- `--path <file-or-dir>`: заполнить среду историческими ежедневными файлами
  `YYYY-MM-DD.md` вместо данных из текущей рабочей области.
- `--grounded`: также сформировать на основе исторических заметок предварительный
  просмотр разделов `Что произошло` / `Размышления` /
  `Возможные долгосрочные обновления`, опирающийся на фактические данные.

## `memory rem-backfill`

Записать основанные на фактических данных исторические сводки REM в `DREAMS.md`
для просмотра в интерфейсе. Изменения обратимы.

```bash
openclaw memory rem-backfill --path <file-or-dir> [--agent <id>] [--stage-short-term] [--json]
openclaw memory rem-backfill --rollback [--rollback-short-term] [--json]
```

- `--path <file-or-dir>`: обязателен, если не задан `--rollback` или
  `--rollback-short-term`. Исторические файлы ежедневной памяти или каталог,
  из которых нужно восстановить данные.
- `--stage-short-term`: также добавить обоснованных кандидатов для долгосрочного
  хранения в активное хранилище переноса краткосрочной памяти, чтобы их могла
  ранжировать обычная глубокая фаза.
- `--rollback`: удалить из `DREAMS.md` ранее записанные обоснованные записи дневника.
- `--rollback-short-term`: удалить ранее добавленных обоснованных кандидатов
  краткосрочной памяти.

## Dreaming

Dreaming — фоновая система консолидации памяти с тремя совместно работающими
фазами, которые выполняются по порядку в рамках одного расписания:
**лёгкая** (сортировка и подготовка материалов краткосрочной памяти),
**REM** (осмысление и выявление тем), **глубокая** (перенос долговременных
фактов в `MEMORY.md`). Только глубокая фаза записывает данные в `MEMORY.md`.

- Включите с помощью `plugins.entries.memory-core.config.dreaming.enabled: true`
  (по умолчанию `false`); `memory-core` автоматически управляет задачей Cron
  для прохода, поэтому вручную выполнять `openclaw cron add` не требуется.
- Переключайте из чата командой `/dreaming on|off`; проверяйте состояние командой
  `/dreaming status` (либо `/dreaming` или `/dreaming help`). Для `on` и `off`
  требуется статус владельца канала или разрешение Gateway `operator.admin`;
  состояние и справка доступны всем, кто может вызвать команду.
- Понятный человеку результат фаз записывается в `DREAMS.md` (или существующий
  `dreams.md`). По умолчанию (`dreaming.storage.mode: "separate"`) каждая фаза
  также записывает отдельный отчёт в `memory/dreaming/<phase>/YYYY-MM-DD.md`;
  задайте `mode: "inline"`, чтобы включать отчёты непосредственно в ежедневный
  файл памяти, или `"both"`, чтобы использовать оба варианта.
- Запланированные и ручные запуски `memory promote` используют одинаковые
  сигналы ранжирования глубокой фазы; различаются только пороговые значения
  по умолчанию (см. таблицу выше и запланированные значения ниже).
- Запланированные запуски выполняются для рабочих областей памяти всех
  настроенных агентов.

Запланированные значения по умолчанию (`plugins.entries.memory-core.config.dreaming`):

| Ключ                                   | Значение по умолчанию |
| -------------------------------------- | --------------------- |
| `frequency`                            | `0 3 * * *`           |
| `phases.deep.minScore`                 | `0.8`                 |
| `phases.deep.minRecallCount`           | `3`                   |
| `phases.deep.minUniqueQueries`         | `3`                   |
| `phases.deep.recencyHalfLifeDays`      | `14`                  |
| `phases.deep.maxAgeDays`               | `30`                  |
| `phases.deep.maxPromotedSnippetTokens` | `160`                 |

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Полный список ключей и сведения о фазах: [Dreaming](/ru/concepts/dreaming),
[справочник по конфигурации памяти](/ru/reference/memory-config#dreaming).

## Зависимость SecretRef от Gateway

Если поля ключей удалённого API Active Memory настроены как SecretRef, команды
`memory` разрешают их из активного снимка Gateway; если Gateway недоступен,
команда немедленно завершается с ошибкой. Для этого требуется Gateway,
поддерживающий метод `secrets.resolve`; более старые версии Gateway возвращают
ошибку неизвестного метода.
