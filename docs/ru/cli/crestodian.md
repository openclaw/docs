# `openclaw crestodian`

Диалоговый Crestodian — локальный агент OpenClaw для первоначальной настройки, восстановления и конфигурирования. Он запускается только после того, как действующая модель по умолчанию успешно выполнит реальный запрос. При новой установке сначала настраивается инференс; некорректная конфигурация обрабатывается классическим сценарием doctor.

## Когда он запускается

Запуск `openclaw` без подкоманды выбирает сценарий в зависимости от состояния конфигурации:

- Конфигурация отсутствует либо существует, но не содержит заданных пользователем настроек (пуста или содержит только ключи `$schema`/`meta`): запускается пошаговая первоначальная настройка с проверкой через реальный запрос к ИИ.
- Конфигурация существует, но не проходит проверку: запускается классическая первоначальная настройка, которая сообщает о проблемах и предлагает выполнить `openclaw doctor`.
- Конфигурация существует и корректна: открывается обычный TUI агента. Если настроенный Gateway доступен и у его агента по умолчанию задана модель, этот интерфейс открывается сразу, без первоначальной настройки и Crestodian. Чтобы позднее перейти к Crestodian, используйте `/crestodian` в TUI или непосредственно выполните `openclaw crestodian`.

Команда `openclaw crestodian` сначала проверяет настроенную модель по умолчанию реальным запросом. Если запрос выполнен успешно, запускается Crestodian. При ошибке в интерактивном режиме открывается пошаговая настройка инференса, а после успешной проверки подходящего варианта управление передаётся Crestodian. Однократные запросы, JSON и другие неинтерактивные запросы при недоступном инференсе завершаются ошибкой с указанием выполнить `openclaw onboard`. Для `openclaw --help` и `openclaw --version` сохраняются обычные быстрые сценарии.

При неинтерактивном запуске `openclaw` без параметров (без TTY) команда завершается с кратким сообщением вместо вывода корневой справки: для новой или некорректной установки предлагается неинтерактивная первоначальная настройка, а при корректной конфигурации — `openclaw agent --local ...`.

`openclaw onboard --modern` остаётся псевдонимом совместимости для Crestodian, но использует ту же проверку инференса: при рабочем инференсе открывается чат, при ошибках в интерактивном режиме запускается пошаговая настройка инференса, а неинтерактивные ошибки завершают работу с инструкциями по первоначальной настройке. `openclaw onboard --classic` открывает полный пошаговый мастер.

## Что показывает Crestodian

Интерактивный Crestodian открывается в той же оболочке TUI, что и `openclaw tui`, но с серверной частью чата Crestodian. Начальное приветствие содержит:

- сведения о корректности конфигурации и агенте по умолчанию
- проверенную модель, которую использует Crestodian
- сведения о доступности Gateway по результатам первой проверки при запуске
- следующее рекомендуемое действие для диагностики

При запуске он не выводит секреты и не загружает CLI-команды плагинов без необходимости.

Используйте `status` для получения подробной сводки: путь к конфигурации, пути к документации и исходному коду, результаты проверок локального CLI, наличие ключей и токенов, сведения об агентах, модели и Gateway.

Crestodian использует тот же механизм поиска справочных материалов, что и обычные агенты: в рабочей копии Git он указывает на локальный каталог `docs/` и дерево исходного кода; при установке через npm использует встроенную документацию и ссылку на [https://github.com/openclaw/openclaw](https://github.com/openclaw/openclaw), а также рекомендует обратиться к исходному коду, если документации недостаточно.

## Примеры

```bash
openclaw
openclaw crestodian
openclaw crestodian --json
openclaw crestodian --message "models"
openclaw crestodian --message "validate config"
openclaw crestodian --message "setup workspace ~/Projects/work" --yes
openclaw crestodian --message "set default model openai/gpt-5.6" --yes
openclaw onboard --modern
```

В TUI Crestodian:

```text
status

doctor

agents

models

open channel wizard for slack
plugins list
plugins search slack
plugin install clawhub:openclaw-codex-app-server
talk to work agent
talk to agent for ~/Projects/work
audit
quit
```

## Операции и подтверждение

Crestodian использует типизированные операции вместо произвольного редактирования конфигурации.

Операции только для чтения выполняются немедленно: показать обзор, вывести список агентов, вывести список установленных плагинов, найти плагины ClawHub, показать состояние модели и серверной части, выполнить проверки состояния и работоспособности, проверить доступность Gateway, запустить doctor без интерактивного исправления, проверить конфигурацию, показать путь к журналу аудита.

Запуск пошаговой настройки канала (`connect telegram`) также выполняется немедленно. Мастер запрашивает явные ответы и самостоятельно выполняет соответствующие изменения.

Постоянные изменения требуют подтверждения в диалоге (или параметра `--yes` для непосредственной команды): запись конфигурации, `config set`, `config set-ref`, первоначальная настройка и подготовка окружения, изменение модели по умолчанию, запуск, остановка и перезапуск Gateway, создание агентов и установка плагинов.

Исправления doctor недоступны внутри Crestodian, поскольку они могут изменить провайдера, аутентификацию или маршрут инференса агента по умолчанию, от которого зависит текущий сеанс. Выйдите из Crestodian и выполните `openclaw doctor --fix` в терминале. Команда `doctor` в режиме только для чтения остаётся доступной внутри Crestodian.

Новые агенты наследуют маршрут инференса по умолчанию, проверенный реальным запросом. Идентификатор агента `crestodian` зарезервирован для привилегированного виртуального смотрителя и не может использоваться при создании обычного агента.

`config set` и `config set-ref` не могут изменять состояние маршрута инференса, включая учётные данные провайдера инференса, верхнеуровневые параметры `auth.*`, каталоги моделей, серверные части CLI, маршруты моделей по умолчанию и отдельных агентов, параметры и инструменты агентов, а также корневые параметры `tools.*`. Прямые записи в `env.*`, `secrets.*`, `plugins.*` и `$include` также запрещены, поскольку они могут подменить механизм разрешения учётных данных или активацию провайдера. Аутентификация Gateway и каналов остаётся обычной частью конфигурации. Используйте типизированные процессы настройки плагинов и каналов, а для уже настроенного маршрута — `set default model <provider/model>`; перед сохранением маршрут проверяется реальным запросом. Чтобы настроить или восстановить доступ к провайдеру и аутентификацию, выйдите из Crestodian и выполните `openclaw onboard`.

Удаление плагинов внутри Crestodian запрещено, поскольку удаление плагина провайдера может отключить маршрут инференса, обеспечивающий работу текущего сеанса. Выйдите из Crestodian и выполните в терминале `openclaw plugins uninstall <id>`.

Подтверждение можно дать своими словами: однозначные ответы («да», «конечно», «продолжайте», «не сейчас») распознаются по закрытому детерминированному списку. Если настроенный маршрут поддерживает отдельный запрос завершения, другие ответы могут классифицироваться только по вашему сообщению и ожидающему подтверждения предложению — но не самой диалоговой моделью, которая не может подтвердить собственные действия. При нераспознанном или неоднозначном ответе предложение остаётся в ожидании, а система повторно запрашивает подтверждение.

Выполненные изменения записываются в `~/.openclaw/audit/crestodian.jsonl`. Обнаружение не регистрируется в журнале аудита; регистрируются только выполненные операции и записи.

Настройка канала может выполняться в виде размещённого диалога до момента, когда потребуется секрет. Локальный TUI Crestodian не принимает конфиденциальные ответы мастера, поскольку ввод в терминальном чате видим. Он сразу предлагает `open channel wizard`, передавая выбранный канал в терминальный мастер с маскированным вводом; позднее также можно выполнить `openclaw channels add --channel <channel>`.

### Переход к настройке канала с маскированным вводом

Локальный чат может передать управление мастеру настройки канала с маскированным вводом:

```text
open channel wizard for slack
channel info slack
```

Команда `open channel wizard for <channel>` открывает настройку канала с маскированным вводом после закрытия TUI чата. Сначала выполните `channel info <channel>`, чтобы узнать название канала, состояние настройки, краткий список предварительных требований и ссылку на документацию.

Crestodian никогда не изменяет доступ к провайдеру или аутентификацию из собственного сеанса: этот сеанс уже зависит от соответствующего маршрута инференса. Для настройки или восстановления провайдера модели команда `configure model provider` возвращает инструкции по выходу и первоначальной настройке, не запуская мастер и не записывая конфигурацию. Выйдите из Crestodian и выполните `openclaw onboard`; первоначальная настройка подготавливает учётные данные и сохраняет только тот маршрут, который успешно выполняет реальный запрос. После успешной первоначальной настройки снова запустите Crestodian.

## Первоначальная подготовка

Команда `setup` настраивает оставшуюся часть рабочего пространства и состояние Gateway после того, как пошаговая первоначальная настройка уже настроила инференс. Запись выполняется только через типизированные операции с конфигурацией и лишь после запроса подтверждения.

```text
setup
setup workspace ~/Projects/work
```

`setup` сохраняет проверенную действующую модель. Эта команда не настраивает и не заменяет инференс.

Если инференс отсутствует или не проходит проверку реальным запросом, выйдите из Crestodian и выполните `openclaw onboard`. Пошаговая первоначальная настройка обнаруживает настроенные модели, ключи API и аутентифицированные локальные CLI, запрашивает реальный ответ у каждого подходящего варианта и сохраняет только успешно работающий маршрут. Crestodian запускается сразу после прохождения этого этапа и затем может настроить рабочее пространство, Gateway, каналы, агентов, плагины и другие необязательные функции.

Приложение macOS полностью пропускает эту последовательность, если обнаруживает настроенный Gateway, у агента по умолчанию которого уже настроена модель; оно открывает обычный интерфейс агента.
Для нового или не полностью настроенного Gateway приложение выполняет последовательность настройки инференса через методы Gateway `crestodian.setup.detect` и `crestodian.setup.activate`: detect выводит все найденные подходящие серверные части, а activate проверяет одну из них реальным запросом (фактическим завершением «reply with OK») и сохраняет модель, учётные данные и состояние провайдера или среды выполнения, необходимые для этого маршрута, только после успешной проверки. Настройки рабочего пространства и Gateway по умолчанию остаются на стороне Crestodian. Вариант, не прошедший проверку, никогда не изменяет конфигурацию; приложение автоматически переходит к следующему варианту, а в конце предлагает вручную указать ключ или токен на основе активных плагинов провайдеров текстового инференса Gateway. Выбранный провайдер определяет исходную модель и конфигурацию, а учётные данные перед сохранением проверяются тем же способом.

Контроль Codex и другие необязательные возможности плагинов не входят в эту транзакцию активации инференса. Настраивайте их только после того, как инференс заработает и запустится Crestodian; существующие политики плагинов и явные отказы от контроля остаются неизменными во время настройки инференса.

## Диалог с ИИ

Свободный диалог в интерактивном Crestodian работает через тот же цикл агента, что и обычные агенты OpenClaw, но ограничен одним инструментом OpenClaw с полномочиями нулевого кольца — `crestodian`, который служит оболочкой для типизированных операций. Операции чтения выполняются свободно, изменения требуют вашего подтверждения в диалоге именно для этой операции (см. раздел «Операции и подтверждение»), а каждое применённое изменение регистрируется в журнале аудита и повторно проверяется. Сеанс агента сохраняется, поэтому Crestodian обладает полноценной памятью между сообщениями. Если проверенный маршрут инференса позднее перестанет работать, вернитесь к `openclaw onboard` и восстановите его, прежде чем продолжать.

Хост не преобразует запросы на естественном языке в операции. Произвольные сообщения, включая текст, похожий на команды, и вопросы вроде «почему мой gateway остановился?», передаются ИИ, который может сопоставить запрос с типизированной операцией через инструмент `crestodian`.

Когда изменение ожидает подтверждения, без инференса распознаются только однозначные фразы подтверждения или отказа из закрытого списка. Неоднозначное согласие передаётся отдельному настроенному запросу завершения, а при его отсутствии отклоняется по безопасному принципу. Структурированные поля мастера и точная навигация хоста являются элементами управления интерфейса, а не механизмом разбора операций на естественном языке. Особенно важно одно исключение для защиты секретов: точная команда `config set` для конфиденциального пути (токены, ключи, пароли) никогда не передаётся модели. Хост создаёт предложение со скрытым значением, а само значение маскируется в истории, доступной ИИ. Для секретов предпочтительно использовать `config set-ref <path> env <ENV_VAR>`.

Режим восстановления через канал сообщений никогда не использует планировщик с поддержкой модели. Удалённое восстановление остаётся детерминированным, чтобы неисправный или скомпрометированный обычный маршрут агента нельзя было использовать как редактор конфигурации.

### Модель доверия оболочки CLI

Встроенные среды выполнения и оболочка сервера приложений Codex напрямую обеспечивают ограничение нулевого кольца: запуск содержит список разрешённых инструментов OpenClaw, включающий только инструмент `crestodian`. Для Codex OpenClaw также отключает в этом запуске окружения, нативное выполнение, мультиагентный режим, цели, приложения и плагины, Skills/MCP, веб-поиск и интерфейсы `request_user_input`. Codex по-прежнему внедряет свою неактивную нативную утилиту `update_plan`; она может обновлять временный контрольный список модели, но не может записывать файлы или изменять конфигурацию OpenClaw. Оболочки CLI не используют список разрешённых инструментов OpenClaw, поэтому Crestodian допускает только те серверные части, чей собственный контракт выбора инструментов позволяет доказать соблюдение того же ограничения:

- Выбираемые бэкенды, включая Claude Code, запускаются с пустым набором нативных инструментов и одним инструментом MCP — `crestodian`. Сгенерированная Claude конфигурация MCP применяется с флагом `--strict-mcp-config`, поэтому никакие другие серверы MCP не загружаются.
- Бэкенды, не объявляющие нативных инструментов, получают такой же выделенный сервер MCP Crestodian.
- Бэкенды с постоянно активными или неизвестными нативными инструментами блокируются до инференса; они не могут размещать сессию Crestodian.

Сервер MCP `crestodian` получают только сессии Crestodian; обычные запуски агентов никогда не видят этот инструмент. Поэтому выбираемые CLI-бэкенды без нативных инструментов и модели с API-ключами обеспечивают буквальный цикл с единственным инструментом. Модели Codex app-server обеспечивают единственный инструмент полномочий OpenClaw вместе с неактивной нативной утилитой планирования. Во всех трёх случаях операции записи при настройке остаются ограничены проверяемым контрактом подтверждения Crestodian.

Gemini CLI остаётся доступным для обычных агентов, но не может обеспечить пробный запуск без инструментов, требуемый шлюзом инференса, поэтому не может размещать Crestodian.

## Переключение на агента

Используйте селектор на естественном языке, чтобы выйти из Crestodian и открыть обычный TUI:

```text
talk to agent
talk to work agent
switch to main agent
```

`openclaw tui`, `openclaw chat` и `openclaw terminal` напрямую открывают обычный TUI агента; они не запускают Crestodian. После переключения в обычный TUI команда `/crestodian` возвращает в Crestodian, при необходимости с последующим запросом:

```text
/crestodian
/crestodian restart gateway
```

## Режим аварийного восстановления через сообщения

Режим аварийного восстановления через сообщения — это точка входа в Crestodian из канала сообщений: используйте его, когда обычный агент не работает, но доверенный канал (например, WhatsApp) по-прежнему принимает команды.

Это детерминированный обработчик аварийных команд, а не диалоговый агент Crestodian. Он не выполняет первоначальную настройку с нуля и не ослабляет шлюз инференса для чата Crestodian.

Поддерживаемая команда: `/crestodian <request>`. Аварийное восстановление принимает только точную грамматику вводимых команд — естественный язык отклоняется с подсказкой, операция никогда не определяется предположительно, и модель никогда не вызывается.

```text
Вы в доверенном личном сообщении владельца: /crestodian status
OpenClaw: Режим аварийного восстановления Crestodian. Gateway доступен: нет. Конфигурация действительна: нет.
Вы: /crestodian restart gateway
OpenClaw: План: перезапустить Gateway. Ответьте /crestodian yes для применения.
Вы: /crestodian yes
OpenClaw: Применено. Запись аудита создана.
```

Создание агента также можно поставить в локальную очередь или выполнить через аварийное восстановление:

```text
create agent work workspace ~/Projects/work model openai/gpt-5.6-sol
/crestodian create agent work workspace ~/Projects/work
```

При создании агента можно указать только текущую проверенную в рабочей среде модель по умолчанию. Не указывайте модель, чтобы унаследовать этот маршрут.

Удалённое аварийное восстановление является административной поверхностью, и с ним следует обращаться как с удалённым исправлением конфигурации, а не как с обычным чатом.

Контракт безопасности удалённого аварийного восстановления:

- Отключено, когда для агента или сессии активна песочница; Crestodian отказывает в удалённом аварийном восстановлении и предлагает выполнить локальное исправление через CLI.
- Эффективное состояние по умолчанию — `auto`: удалённое аварийное восстановление разрешено только в доверенном режиме YOLO, когда среда выполнения уже имеет локальные полномочия без песочницы (`tools.exec.security` разрешается в `full`, а `tools.exec.ask` — в `off`, при режиме песочницы `off`).
- Требуется явно заданная личность владельца; правила отправителей с подстановочными знаками, открытая политика групп, вебхуки без аутентификации и анонимные каналы запрещены.
- По умолчанию разрешены только личные сообщения владельца; для аварийного восстановления в группах и каналах требуется явное включение.
- Поиск и просмотр списка плагинов доступны только для чтения. Установка плагинов всегда возможна только локально (она заблокирована в режиме аварийного восстановления, даже если включена в других режимах), поскольку при этом загружается исполняемый код. Удаление плагинов запрещено как в локальном Crestodian, так и в режиме аварийного восстановления; выполните `openclaw plugins uninstall <id>` в терминале.
- Удалённое аварийное восстановление не может открыть локальный TUI или переключиться в интерактивную сессию агента; для передачи управления агенту используйте локальную команду `openclaw`.
- Постоянные операции записи по-прежнему требуют подтверждения, даже в режиме аварийного восстановления.
- Каждая применённая операция аварийного восстановления регистрируется в журнале аудита. При восстановлении через канал сообщений записываются метаданные канала, учётной записи, отправителя и адреса источника; для операций, изменяющих конфигурацию, также записываются хеши конфигурации до и после изменения.
- Секреты никогда не выводятся. Проверка SecretRef сообщает о доступности, а не о значениях.
- Если Gateway работает, аварийное восстановление предпочитает типизированные операции Gateway; если он не работает, используется только минимальная локальная поверхность исправления, не зависящая от обычного цикла агента.

Структура конфигурации:

```jsonc
{
  "crestodian": {
    "rescue": {
      "enabled": "auto",
      "ownerDmOnly": true,
      "pendingTtlMinutes": 15,
    },
  },
}
```

- `enabled`: `"auto"` (по умолчанию) разрешает аварийное восстановление, только когда эффективная среда выполнения находится в режиме YOLO и песочница отключена; `false` полностью запрещает аварийное восстановление через канал сообщений; `true` явно разрешает его при успешной проверке владельца и канала, но запрет при активной песочнице сохраняется.
- `ownerDmOnly`: ограничивает аварийное восстановление личными сообщениями владельца. Значение по умолчанию — `true`.
- `pendingTtlMinutes`: срок, в течение которого ожидающая операция записи остаётся доступной для подтверждения командой `/crestodian yes`, прежде чем истечёт её срок действия. Значение по умолчанию — `15`.

Удалённое аварийное восстановление проверяется в Docker-сценарии:

```bash
pnpm test:docker:crestodian-rescue
```

Опциональная дымовая проверка поверхности команд реального канала проверяет `/crestodian status`, а также полный цикл подтверждения постоянной операции через обработчик аварийного восстановления:

```bash
pnpm test:live:crestodian-rescue-channel
```

Одноразовая настройка упакованной версии со шлюзом инференса проверяется командой:

```bash
pnpm test:docker:crestodian-first-run
```

Этот сценарий упакованного CLI запускается с пустым каталогом состояния и подтверждает, что Crestodian блокирует выполнение без инференса. Затем он тестирует и активирует имитацию Claude через упакованный модуль активации. Только после этого нечётко сформулированный запрос поступает планировщику и преобразуется в типизированную настройку, за которой следуют одноразовые команды: они создают дополнительного агента, настраивают Discord путём включения плагина и передачи токена через SecretRef, проверяют конфигурацию и журнал аудита. Этот сценарий предоставляет дополнительные доказательства работы шлюза и операций; он не проверяет интерактивное первоначальное подключение или диалог агента Crestodian с инструментами и подтверждениями. Приведённый ниже сценарий QA Lab перенаправляет на тот же Docker-сценарий:

```bash
pnpm openclaw qa suite --scenario crestodian-ring-zero-setup
```
