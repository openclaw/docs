# `openclaw node`

Запустите **хост Node без графического интерфейса**, который подключается к Gateway WebSocket и предоставляет
`system.run` / `system.which` на этой машине.

## Зачем использовать хост Node?

Используйте хост Node, когда хотите, чтобы агенты **выполняли команды на других машинах** в вашей
сети без установки там полноценного приложения-компаньона для macOS.

Типичные сценарии:

- Выполнение команд на удаленных машинах Linux/Windows (серверы сборки, лабораторные машины, NAS).
- Сохранение **изолированного** выполнения на Gateway, но делегирование одобренных запусков другим хостам.
- Предоставление легкой цели выполнения без графического интерфейса для автоматизации или узлов CI.

Выполнение по-прежнему защищено **одобрениями exec** и списками разрешений для каждого агента на
хосте Node, поэтому доступ к командам остается ограниченным и явным.

## Браузерный прокси (без настройки)

Хосты Node автоматически объявляют браузерный прокси, если `browser.enabled` не
отключен на Node. Это позволяет агенту использовать браузерную автоматизацию на этом Node
без дополнительной настройки.

По умолчанию прокси предоставляет обычную поверхность браузерного профиля Node. Если вы
зададите `nodeHost.browserProxy.allowProfiles`, прокси станет ограничительным:
обращение к профилям вне списка разрешений будет отклоняться, а маршруты
создания/удаления постоянных профилей будут блокироваться через прокси.

При необходимости отключите это на Node:

```json5
{
  nodeHost: {
    browserProxy: {
      enabled: false,
    },
  },
}
```

## Запуск (передний план)

```bash
openclaw node run --host <gateway-host> --port 18789
```

Параметры:

- `--host <host>`: хост Gateway WebSocket (по умолчанию: `127.0.0.1`)
- `--port <port>`: порт Gateway WebSocket (по умолчанию: `18789`)
- `--context-path <path>`: путь контекста Gateway WebSocket (например, `/openclaw-gw`). Добавляется к URL WebSocket.
- `--tls`: использовать TLS для подключения к Gateway
- `--tls-fingerprint <sha256>`: ожидаемый отпечаток TLS-сертификата (sha256)
- `--node-id <id>`: переопределить идентификатор Node (очищает токен сопряжения)
- `--display-name <name>`: переопределить отображаемое имя Node

## Аутентификация Gateway для хоста Node

`openclaw node run` и `openclaw node install` разрешают аутентификацию Gateway из config/env (без флагов `--token`/`--password` в командах Node):

- Сначала проверяются `OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`.
- Затем локальный резервный вариант конфигурации: `gateway.auth.token` / `gateway.auth.password`.
- В локальном режиме хост Node намеренно не наследует `gateway.remote.token` / `gateway.remote.password`.
- Если `gateway.auth.token` / `gateway.auth.password` явно настроены через SecretRef и не разрешены, разрешение аутентификации Node завершается закрытым отказом (без маскирующего удаленного резервного варианта).
- В `gateway.mode=remote` поля удаленного клиента (`gateway.remote.token` / `gateway.remote.password`) также допустимы согласно правилам удаленного приоритета.
- Разрешение аутентификации хоста Node учитывает только переменные окружения `OPENCLAW_GATEWAY_*`.

Для Node, подключающегося к открытому `ws://` Gateway, принимаются loopback, литералы частных IP,
`.local` и хосты Tailnet `*.ts.net`. Для других доверенных имен частного DNS
задайте `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`; без этого запуск Node завершится
закрытым отказом и попросит использовать `wss://`, SSH-туннель или
Tailscale. Это включение на уровне окружения процесса, а не ключ конфигурации
`openclaw.json`.
`openclaw node install` сохраняет его в управляемой службе Node, когда он
присутствует в окружении команды установки.

## Служба (фон)

Установите хост Node без графического интерфейса как пользовательскую службу.

```bash
openclaw node install --host <gateway-host> --port 18789
```

Параметры:

- `--host <host>`: хост Gateway WebSocket (по умолчанию: `127.0.0.1`)
- `--port <port>`: порт Gateway WebSocket (по умолчанию: `18789`)
- `--context-path <path>`: путь контекста Gateway WebSocket (например, `/openclaw-gw`). Добавляется к URL WebSocket.
- `--tls`: использовать TLS для подключения к Gateway
- `--tls-fingerprint <sha256>`: ожидаемый отпечаток TLS-сертификата (sha256)
- `--node-id <id>`: переопределить идентификатор Node (очищает токен сопряжения)
- `--display-name <name>`: переопределить отображаемое имя Node
- `--runtime <runtime>`: среда выполнения службы (`node` или `bun`)
- `--force`: переустановить/перезаписать, если уже установлено

Управление службой:

```bash
openclaw node status
openclaw node start
openclaw node stop
openclaw node restart
openclaw node uninstall
```

Используйте `openclaw node run` для хоста Node на переднем плане (без службы).

Команды службы принимают `--json` для машиночитаемого вывода.

Хост Node повторяет попытки после перезапуска Gateway и сетевых закрытий внутри процесса. Если
Gateway сообщает о терминальной паузе аутентификации по токену/паролю/bootstrap, хост Node
записывает сведения о закрытии в журнал и завершается с ненулевым кодом, чтобы launchd/systemd мог перезапустить его со
свежей конфигурацией и учетными данными. Паузы, требующие сопряжения, остаются в
потоке переднего плана, чтобы ожидающий запрос можно было одобрить.

## Сопряжение

Первое подключение создает ожидающий запрос сопряжения устройства (`role: node`) на Gateway.
Одобрите его через:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

В строго контролируемых сетях Node оператор Gateway может явно включить
автоматическое одобрение первого сопряжения Node из доверенных CIDR:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

По умолчанию это отключено. Это применяется только к новому сопряжению `role: node` без
запрошенных областей доступа. Клиенты оператора/браузера, Control UI, WebChat, а также обновления роли,
области доступа, метаданных или публичного ключа по-прежнему требуют ручного одобрения.

Если Node повторяет попытку сопряжения с измененными деталями аутентификации (роль/области доступа/публичный ключ),
предыдущий ожидающий запрос заменяется и создается новый `requestId`.
Перед одобрением снова выполните `openclaw devices list`.

Хост Node хранит свой идентификатор Node, токен, отображаемое имя и сведения о подключении к Gateway в
`~/.openclaw/node.json`.

## Одобрения exec

`system.run` ограничивается локальными одобрениями exec:

- `$OPENCLAW_STATE_DIR/exec-approvals.json` или
  `~/.openclaw/exec-approvals.json`, когда переменная не задана
- [Одобрения exec](/ru/tools/exec-approvals)
- `openclaw approvals --node <id|name|ip>` (редактирование из Gateway)

Для одобренного асинхронного exec на Node OpenClaw подготавливает канонический `systemRunPlan`
до запроса. Последующая одобренная пересылка `system.run` повторно использует этот сохраненный
план, поэтому правки полей command/cwd/session после создания запроса на одобрение
отклоняются вместо изменения того, что выполнит Node.
