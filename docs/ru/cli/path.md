# `openclaw path`

Доступ из оболочки к схеме адресации `oc://`: единый синтаксис путей с выбором обработчика по типу для просмотра и редактирования адресуемых файлов рабочего пространства (markdown, jsonc, jsonl, yaml/yml/lobster). Пользователи самостоятельного развертывания, авторы плагинов и расширения редакторов используют его для чтения, поиска или обновления конкретного расположения без самостоятельной реализации парсера для каждого типа файлов.

`path` предоставляется встроенным необязательным плагином `oc-path`. Включите его перед первым использованием:

```bash
openclaw plugins enable oc-path
```

Глаголы CLI соответствуют модели адресации:

- `resolve` работает с конкретным путем и единственным совпадением.
- `find` — глагол для нескольких совпадений, используемый с подстановочными знаками, объединениями, предикатами и позиционным развертыванием.
- `set` принимает только конкретные пути или маркеры вставки; шаблоны с подстановочными знаками отклоняются до записи.
- `validate` разбирает путь без доступа к файловой системе.
- `emit` выполняет цикл «разбор + вывод» для файла (диагностика побайтовой точности).

## Зачем это использовать

Состояние OpenClaw распределено между редактируемыми вручную файлами markdown, конфигурацией JSONC с комментариями, журналами JSONL только для добавления и файлами рабочих процессов или спецификаций YAML. Скриптам, хукам и агентам часто требуется одно небольшое значение из таких файлов: ключ frontmatter, настройка плагина, поле записи журнала, шаг YAML или элемент маркированного списка в именованном разделе.

`openclaw path` предоставляет таким вызывающим сторонам стабильный адрес вместо одноразового `grep`, регулярного выражения или отдельного парсера для каждого типа файлов. Один и тот же путь `oc://` можно проверить, разрешить, найти, предварительно выполнить и записать из терминала, благодаря чему узконаправленная автоматизация остается удобной для проверки и повторного выполнения. Остальная часть файла сохраняется, поэтому запись одного конечного значения не затрагивает комментарии, окончания строк или соседнее форматирование.

Используйте эту команду, когда нужный объект имеет логический адрес, но структура файла различается:

- Хук считывает одну настройку из JSONC с комментариями и не теряет комментарии при записи значения обратно.
- Скрипт обслуживания находит каждое совпадающее поле события в журнале JSONL, не загружая весь журнал в специальный парсер.
- Редактор переходит к разделу markdown или элементу маркированного списка по слагу, а затем отображает точную строку, которой соответствует путь.
- Агент предварительно выполняет небольшое изменение рабочего пространства перед его применением, а измененные байты доступны для проверки.

Не используйте `openclaw path` для обычного редактирования файлов целиком, сложных миграций конфигурации или операций записи, относящихся к памяти; для этого следует использовать команду или плагин владельца. `path` предназначен для небольших операций с адресуемыми файлами, когда воспроизводимая команда терминала лучше очередного специализированного парсера.

## Использование

Чтение одного значения из редактируемого вручную файла конфигурации:

```bash
openclaw path resolve 'oc://config.jsonc/plugins/github/enabled'
```

Предварительный просмотр записи без изменения данных на диске:

```bash
openclaw path set 'oc://config.jsonc/plugins/github/enabled' 'true' --dry-run
```

Поиск совпадающих записей в журнале JSONL только для добавления:

```bash
openclaw path find 'oc://session.jsonl/[event=tool_call]/name'
```

Адресация инструкции в markdown по разделу и элементу вместо номера строки:

```bash
openclaw path resolve 'oc://AGENTS.md/runtime-safety/openclaw-gateway'
```

Проверка пути в CI или подготовительном скрипте до чтения или записи данных скриптом:

```bash
openclaw path validate 'oc://AGENTS.md/tools/$last/risk'
```

Эти команды рассчитаны на копирование в скрипты оболочки. Используйте `--json`, когда вызывающей стороне нужен структурированный вывод, и `--human`, когда результат просматривает человек.

## Принцип работы

1. Разбирает адрес `oc://` на слоты: файл, раздел, элемент, поле и необязательный запрос сеанса.
2. Выбирает адаптер типа файла по расширению целевого файла (`.md`, `.jsonc`, `.json`, `.jsonl`, `.ndjson`, `.yaml`, `.yml`, `.lobster`).
3. Разрешает слоты относительно структуры соответствующего типа файла: заголовков и элементов markdown, ключей объектов и индексов массивов JSONC, строковых записей JSONL либо узлов отображений и последовательностей YAML.
4. Для `set` выводит измененные байты через тот же адаптер, чтобы незатронутые части файла сохраняли комментарии, окончания строк и соседнее форматирование там, где это поддерживает соответствующий тип.

`resolve` и `set` требуют одну конкретную цель. `find` предназначен для исследования: он развертывает подстановочные знаки, объединения, предикаты и порядковые номера в конкретные совпадения, которые можно просмотреть перед выбором цели для записи.

## Подкоманды

| Подкоманда              | Назначение                                                                 |
| ----------------------- | -------------------------------------------------------------------------- |
| `resolve <oc-path>`     | Вывести конкретное совпадение по пути (или «не найдено»).                  |
| `find <pattern>`        | Перечислить совпадения для пути с подстановочным знаком, объединением или предикатом. |
| `set <oc-path> <value>` | Записать конечное значение или цель вставки по конкретному пути. Поддерживает `--dry-run`. |
| `validate <oc-path>`    | Только разобрать и вывести структурную декомпозицию (файл / раздел / элемент / поле). |
| `emit <file>`           | Выполнить для файла цикл «разбор + вывод» (диагностика побайтовой точности). |

## Глобальные флаги

| Флаг            | Применяется к                     | Назначение                                                               |
| --------------- | --------------------------------- | ------------------------------------------------------------------------ |
| `--cwd <dir>`   | `resolve`, `find`, `set`, `emit`  | Разрешать слот файла относительно этого каталога (по умолчанию: `process.cwd()`). |
| `--file <path>` | `resolve`, `find`, `set`, `emit`  | Переопределить разрешенный путь слота файла (абсолютный доступ).          |
| `--json`        | ко всем                           | Принудительно использовать вывод JSON (по умолчанию, если stdout не является TTY). |
| `--human`       | ко всем                           | Принудительно использовать человекочитаемый вывод (по умолчанию, если stdout является TTY). |
| `--value-json`  | `set`                             | Разобрать `<value>` как JSON для замены конечного значения JSON/JSONC/JSONL. |
| `--dry-run`     | `set`                             | Вывести байты, которые были бы записаны, без выполнения записи.          |
| `--diff`        | `set` (требует `--dry-run`)       | Вывести унифицированный diff вместо полного набора байтов.                |

`validate` принимает только `--json` / `--human`; эта команда не обращается к файловой системе, поэтому `--cwd` и `--file` к ней не применяются.

## Синтаксис `oc://`

```text
oc://FILE/SECTION/ITEM/FIELD?session=SCOPE
```

Правила слотов: для `field` требуется `item`, а для `item` требуется `section`. Для всех четырех слотов действуют следующие правила:

- **Сегменты в кавычках** — `"a/b.c"` сохраняется без разделения по `/` и `.`. Содержимое интерпретируется побайтово буквально; `"` и `\` внутри кавычек не допускаются. Слот файла также учитывает кавычки: `oc://"skills/email-drafter"/Tools/$last` рассматривает `skills/email-drafter` как единый путь к файлу.
- **Предикаты** — `[k=v]`, `[k!=v]`, `[k<v]`, `[k<=v]`, `[k>v]`, `[k>=v]`. Для числовых операторов обе стороны должны приводиться к конечным числам.
- **Объединения** — `{a,b,c}` соответствует любому из вариантов.
- **Подстановочные знаки** — `*` (один подсегмент) и `**` (ноль или более подсегментов, рекурсивно). `find` принимает их; `resolve` и `set` отклоняют их как неоднозначные.
- **Позиционные обозначения** — `$first` / `$last` разрешаются в первый / последний индекс или объявленный ключ.
- **Порядковое обозначение** — `#N` для N-го совпадения в порядке документа.
- **Маркеры вставки** — `+`, `+key`, `+nnn` для вставки по ключу / индексу (используются с `set`).
- **Область сеанса** — `?session=cron-daily` и аналогичные значения. Не зависит от вложенности слотов. Значения сеанса используются без обработки и не декодируются из процентного кодирования; они не могут содержать управляющие символы или зарезервированные разделители запроса (`?`, `&`, `%`).

Зарезервированные символы (`?`, `&`, `%`) вне сегментов в кавычках, предикатов или объединений отклоняются. Управляющие символы (U+0000–U+001F, U+007F) отклоняются везде, включая значение запроса `session`.

Для канонических путей гарантируется `formatOcPath(parseOcPath(path)) === path`. Неканонические параметры запроса игнорируются, кроме первого непустого значения `session=`.

Жесткие ограничения: длина пути не превышает 4096 байт; допускается не более 4 слотов (файл/раздел/элемент/поле), не более 64 разделенных точками подсегментов на слот и не более 256 уровней вложенного обхода для глубоких путей JSON. Кроме того, входной файл JSONC/JSON размером более 16 МиБ отклоняется с диагностикой разбора вместо выполнения разбора для любой команды, загружающей этот файл.

## Адресация по типу файла

| Тип           | Расширения файлов            | Модель адресации                                                                                    |
| ------------- | ---------------------------- | --------------------------------------------------------------------------------------------------- |
| Markdown      | `.md`                        | Разделы H2 по слагу, элементы маркированного списка по слагу или `#N`, frontmatter через `[frontmatter]`. |
| JSONC/JSON    | `.jsonc`, `.json`            | Ключи объектов и индексы массивов; точки разделяют вложенные подсегменты, если они не заключены в кавычки. |
| JSONL         | `.jsonl`, `.ndjson`          | Адреса строк верхнего уровня (`L1`, `L2`, `$first`, `$last`), затем переход внутри строки в стиле JSONC. |
| YAML/.lobster | `.yaml`, `.yml`, `.lobster`  | Ключи отображений и индексы последовательностей; комментарии и потоковый стиль обрабатываются API документа YAML. |

`resolve` возвращает структурированное совпадение: `root`, `node`, `leaf` или `insertion-point` с номером строки, отсчитываемым от 1. Конечные значения предоставляются как текст вместе с `leafType`, чтобы авторы плагинов могли отображать предварительный просмотр без зависимости от формы AST конкретного типа файла.

## Контракт изменения

`set` записывает одну конкретную цель:

- Значения frontmatter markdown и поля элементов `- key: value` являются строковыми конечными значениями. Вставки markdown добавляют разделы, ключи frontmatter или элементы раздела и формируют каноническую структуру markdown для измененного файла. Тела разделов нельзя записывать целиком через `set`.
- При записи конечных значений JSONC строковое значение приводится к существующему типу конечного значения (`string`, конечное `number`, `true`/`false` или `null`). Используйте `--value-json`, если замена конечного значения JSONC/JSON/JSONL должна разобрать `<value>` как JSON и может изменить структуру, например при замене сокращенной строковой ссылки на секрет объектом. При вставке в объекты и массивы JSONC значение `<value>` разбирается как JSON, а для обычной записи конечных значений используется путь редактирования `jsonc-parser`, сохраняющий комментарии и соседнее форматирование.
- При записи конечных значений JSONL приведение внутри строки выполняется так же, как в JSONC. При замене всей строки и добавлении значение `<value>` разбирается как JSON. Сформированный JSONL сохраняет преобладающее в файле соглашение об окончаниях строк LF/CRLF (по большинству переводов строк в файле, поэтому файл, преимущественно использующий CRLF, сохранит CRLF даже при наличии нескольких случайных LF).
- При записи конечных значений YAML выполняется приведение к существующему скалярному типу (`string`, конечное `number`, `true`/`false` или `null`). Для обновления отображений и последовательностей при вставке YAML используется API документа встроенного пакета `yaml`. Некорректные документы YAML с ошибками парсера отклоняются до изменения с кодом `parse-error`.

Используйте `--dry-run` перед видимыми пользователю операциями записи, когда важны точные байты. Изменения JSONC и YAML накладываются на существующий документ (через `jsonc-parser` или API документа `yaml`), поэтому незатронутые байты обычно сохраняются; при любом изменении markdown файл перестраивается из разобранной структуры, что может нормализовать второстепенное форматирование вне измененного конечного значения. Добавьте `--diff`, если предварительный просмотр нужен в виде компактного патча «до/после», а не полного сформированного файла.

## Примеры

```bash
# Проверка пути (без доступа к файловой системе)
openclaw path validate 'oc://AGENTS.md/Tools/$last/risk'

# Чтение конечного значения
openclaw path resolve 'oc://gateway.jsonc/version'

# Поиск с подстановочным знаком
openclaw path find 'oc://session.jsonl/*/event' --file ./logs/session.jsonl

# Предварительное выполнение записи
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run

# Предварительное выполнение записи в виде унифицированного diff
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run --diff

# Применение записи
openclaw path set 'oc://gateway.jsonc/version' '2.0'

# Цикл с побайтовой точностью (диагностика)
openclaw path emit ./AGENTS.md
```

Дополнительные примеры грамматики:

```bash
# Quote keys containing / or .
openclaw path resolve 'oc://config.jsonc/agents.defaults.models/"anthropic/claude-opus-4-7"/alias'

# Deep JSON/JSONC paths can use slash segments; they normalize to dotted subsegments
openclaw path set 'oc://openclaw.json/agents/list/0/tools/exec/security' 'allowlist' --dry-run

# Replace a JSONC leaf with a parsed object
openclaw path set 'oc://openclaw.json/gateway/auth/token' '{"source":"file","provider":"secrets","id":"/test"}' --value-json --dry-run

# Predicate search over JSONC children
openclaw path find 'oc://config.jsonc/plugins/[enabled=true]/id'

# Insert into a JSONC array
openclaw path set 'oc://config.jsonc/items/+1' '{"id":"new","enabled":true}' --dry-run

# Insert a JSONC object key
openclaw path set 'oc://config.jsonc/plugins/+github' '{"enabled":true}' --dry-run

# Append a JSONL event
openclaw path set 'oc://session.jsonl/+' '{"event":"checkpoint","ok":true}' --file ./logs/session.jsonl

# Resolve the last JSONL value line
openclaw path resolve 'oc://session.jsonl/$last/event' --file ./logs/session.jsonl

# Resolve a YAML workflow step
openclaw path resolve 'oc://workflow.yaml/steps/0/id'

# Update a YAML scalar
openclaw path set 'oc://workflow.yaml/steps/$last/id' 'classify-renamed' --dry-run

# Address markdown frontmatter
openclaw path resolve 'oc://AGENTS.md/[frontmatter]/name'

# Insert markdown frontmatter
openclaw path set 'oc://AGENTS.md/[frontmatter]/+description' 'Agent instructions' --dry-run

# Find markdown item fields
openclaw path find 'oc://SKILL.md/Tools/*/send_email'

# Validate a session-scoped path
openclaw path validate 'oc://AGENTS.md/Tools/$last/risk?session=cron-daily'
```

## Рецепты по типам файлов

Одни и те же пять команд работают со всеми типами файлов; схема адресации выбирает обработчик по расширению файла.

### Markdown

```text
<!-- frontmatter.md -->
---
name: drafter
description: email drafting agent
tier: core
---
## Tools
- gh: GitHub CLI
- curl: HTTP client
- send_email: enabled
```

```bash
$ openclaw path resolve 'oc://x.md/[frontmatter]/tier' --file frontmatter.md --human
leaf @ L4: "core" (string)

$ openclaw path resolve 'oc://x.md/tools/gh/gh' --file frontmatter.md --human
leaf @ L9: "GitHub CLI" (string)

$ openclaw path find 'oc://x.md/tools/*' --file frontmatter.md --human
3 matches for oc://x.md/tools/*:
  oc://x.md/tools/gh           →  node @ L9 [md-item]
  oc://x.md/tools/curl         →  node @ L10 [md-item]
  oc://x.md/tools/send-email   →  node @ L11 [md-item]
```

Предикат `[frontmatter]` адресует блок метаданных YAML; `tools` сопоставляется с заголовком `## Tools` по слагу, а конечные элементы сохраняют форму слага, даже если в исходном тексте используются символы подчёркивания (`send_email` преобразуется в `send-email`).

### JSONC

```text
// config.jsonc
{
  "plugins": {
    "github": {"enabled": true, "role": "vcs"},
    "slack":  {"enabled": false, "role": "chat"}
  }
}
```

```bash
$ openclaw path resolve 'oc://config.jsonc/plugins/github/enabled' --file config.jsonc --human
leaf @ L4: "true" (boolean)

$ openclaw path set 'oc://config.jsonc/plugins/slack/enabled' 'true' --file config.jsonc --dry-run
--dry-run: would write 142 bytes to /…/config.jsonc
{
  "plugins": {
    "github": {"enabled": true, "role": "vcs"},
    "slack":  {"enabled": true, "role": "chat"}
  }
}
```

Изменения JSONC выполняются через `jsonc-parser`, поэтому после `set` комментарии и пробелы сохраняются. Сначала запустите команду с `--dry-run`, чтобы проверить байты перед записью. Для файлов `.json` используются тот же адаптер и тот же путь редактирования, что и для `.jsonc`.

### JSONL

```text
{"event":"start","userId":"u1","ts":1}
{"event":"action","userId":"u1","ts":2}
{"event":"end","userId":"u1","ts":3}
```

```bash
$ openclaw path find 'oc://session.jsonl/[event=action]/userId' --file session.jsonl --human
1 match for oc://session.jsonl/[event=action]/userId:
  oc://session.jsonl/L2/userId  →  leaf @ L2: "u1" (string)

$ openclaw path resolve 'oc://session.jsonl/L2/ts' --file session.jsonl --human
leaf @ L2: "2" (number)
```

Каждая строка представляет собой запись. Используйте предикат (`[event=action]`), если номер строки неизвестен, или канонический сегмент `LN`, если он известен. Для файлов `.ndjson` используется тот же адаптер, что и для `.jsonl`.

### YAML

```text
# workflow.yaml
name: inbox-triage
steps:
  - id: fetch
    command: gmail.search
  - id: classify
    command: openclaw.invoke
```

```bash
$ openclaw path resolve 'oc://workflow.yaml/steps/0/id' --file workflow.yaml --human
leaf @ L3: "fetch" (string)

$ openclaw path set 'oc://workflow.yaml/steps/$last/id' 'classify-renamed' --file workflow.yaml --dry-run
--dry-run: would write 99 bytes to /…/workflow.yaml
name: inbox-triage
steps:
  - id: fetch
    command: gmail.search
  - id: classify-renamed
    command: openclaw.invoke
```

Для YAML используется API `Document` пакета `yaml`, а не самописный синтаксический анализатор, поэтому обычный цикл разбора и генерации сохраняет комментарии и исходную структуру, а разрешённые пути используют ту же модель ключей отображения и индексов последовательности, что и JSONC. Один и тот же адаптер обрабатывает файлы `.yaml`, `.yml` и `.lobster`.

## Справочник подкоманд

### `resolve <oc-path>`

Читает один конечный элемент или узел. Подстановочные знаки не допускаются — для них используйте `find`. Завершается с кодом `0` при совпадении, `1` при корректном отсутствии совпадения и `2` при ошибке разбора или отклонённом шаблоне.

```bash
openclaw path resolve 'oc://AGENTS.md/tools/gh/risk' --human
openclaw path resolve 'oc://gateway.jsonc/server/port' --json
```

### `find <pattern>`

Перечисляет все совпадения для шаблона с подстановочными знаками, предикатом или объединением. Завершается с кодом `0`, если найдено хотя бы одно совпадение, и `1`, если совпадений нет. Подстановочные знаки в позиции файла отклоняются с ошибкой `OC_PATH_FILE_WILDCARD_UNSUPPORTED` — укажите конкретный файл (поддержка поиска по нескольким файлам будет добавлена позднее).

```bash
openclaw path find 'oc://AGENTS.md/tools/**/risk'
openclaw path find 'oc://session.jsonl/[event=action]/userId'
openclaw path find 'oc://config.jsonc/plugins/{github,slack}/enabled'
```

### `set <oc-path> <value>`

Записывает конечный элемент. Используйте вместе с `--dry-run`, чтобы предварительно просмотреть байты, которые были бы записаны, не изменяя файл. Добавьте `--diff`, чтобы просмотреть унифицированную разницу. Завершается с кодом `0` при успешной записи, `1`, если базовый слой отклонил операцию (например, при срабатывании защитного маркера), и `2` при ошибках разбора.

```bash
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run
openclaw path set 'oc://gateway.jsonc/version' '2.0' --dry-run --diff
openclaw path set 'oc://gateway.jsonc/version' '2.0'
openclaw path set 'oc://AGENTS.md/Tools/+gh/risk' 'low'
```

Маркер вставки `+key` создаёт дочерний элемент с указанным именем, если он ещё не существует; `+nnn` и отдельный `+` используются соответственно для вставки по индексу и добавления в конец.

### `validate <oc-path>`

Проверка только разбора. Доступ к файловой системе не выполняется. Полезна, когда нужно убедиться, что путь шаблона сформирован правильно перед подстановкой переменных, или получить структурную разбивку для отладки:

```bash
$ openclaw path validate 'oc://AGENTS.md/tools/gh' --human
valid: oc://AGENTS.md/tools/gh
  file:    AGENTS.md
  section: tools
  item:    gh
```

Завершается с кодом `0` для допустимого пути, `1` для недопустимого пути (со структурированными полями `code` и `message`) и `2` при ошибках аргументов.

### `emit <file>`

Пропускает файл через синтаксический анализатор и генератор для соответствующего типа. Для корректного файла вывод должен побайтно совпадать с вводом; расхождение указывает на ошибку синтаксического анализатора или срабатывание защитного маркера. Полезна для отладки поведения базового слоя на реальных входных данных.

```bash
openclaw path emit ./AGENTS.md
openclaw path emit ./gateway.jsonc --json
```

## Коды завершения

| Код | Значение                                                                                      |
| --- | --------------------------------------------------------------------------------------------- |
| `0` | Успех. (`resolve` / `find`: хотя бы одно совпадение. `set`: запись выполнена успешно.)         |
| `1` | Совпадений нет или `set` отклонён базовым слоем (без ошибки системного уровня).                |
| `2` | Ошибка аргументов или разбора.                                                                |

## Режим вывода

`openclaw path` учитывает TTY: в терминале используется удобочитаемый вывод, а при передаче или перенаправлении стандартного вывода — JSON. Флаги `--json` и `--human` переопределяют автоматическое определение.

## Примечания

- `set` записывает байты через путь генерации базового слоя, который автоматически применяет защиту маркером редактирования. Конечный элемент, содержащий `__OPENCLAW_REDACTED__` (дословно или как подстроку), отклоняется при записи.
- Для разбора JSONC и изменения конечных элементов используется локальная для Plugin зависимость `jsonc-parser`, поэтому при обычной записи конечных элементов комментарии и форматирование сохраняются без использования самописного пути разбора и повторной генерации.
- `path` не учитывает отслеживание или восстановление последней заведомо исправной конфигурации (LKG); этим жизненным циклом управляет другой компонент. Если файл, изменяемый через `path`, также отслеживается как LKG, при следующем чтении конфигурации будет решено, принять его или восстановить; рассматривайте изменение через `path` так же, как любую другую прямую запись в этот файл.
