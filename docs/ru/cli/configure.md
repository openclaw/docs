# `openclaw configure`

Интерактивные запросы для целевых изменений существующей конфигурации: учётных данных, устройств, настроек агента по умолчанию, Gateway, каналов, плагинов, навыков и проверок работоспособности.

Используйте `openclaw onboard` или `openclaw setup` для полного пошагового процесса первого запуска, `openclaw setup --baseline` — только для базовой конфигурации и рабочего пространства, а `openclaw channels add` — если требуется настроить только учётную запись канала.

<Tip>
`openclaw config` без подкоманды открывает тот же мастер. Для неинтерактивного редактирования используйте `openclaw config get|set|unset`.
</Tip>

## Параметры

`--section <section>`: повторяемый фильтр разделов. Доступные разделы:

`workspace`, `model`, `web`, `gateway`, `daemon`, `channels`, `plugins`, `skills`, `health`

```bash
openclaw configure
openclaw configure --section web
openclaw configure --section model --section channels
openclaw configure --section gateway --section daemon
```

При выборе `gateway`, `daemon` или `health` (либо при запуске полного мастера без `--section`) появляется запрос о месте запуска Gateway и обновляется `gateway.mode`. Фильтры разделов, исключающие все три варианта, сразу переходят к запрошенной настройке без запроса режима Gateway. При выборе удалённого режима Gateway записывается удалённая конфигурация, после чего мастер немедленно завершает работу; локальные действия, такие как установка плагинов, не выполняются.

<Note>
`openclaw configure` требует интерактивного терминала (и stdin, и stdout должны быть TTY). При его отсутствии выводятся эквивалентные неинтерактивные команды `openclaw config get|set|patch|validate`, после чего процесс завершается с ошибкой вместо частичного выполнения.
</Note>

## Раздел модели

<Note>
**Модель** включает множественный выбор для списка разрешённых моделей `agents.defaults.models` (определяющего, что отображается в `/model` и средстве выбора модели). Варианты настройки для конкретных провайдеров добавляют выбранные модели в существующий список разрешённых, не заменяя уже настроенные модели других провайдеров.

Повторная авторизация провайдера через configure сохраняет существующее значение `agents.defaults.model.primary`, даже если этап авторизации провайдера возвращает изменение конфигурации с собственной рекомендуемой моделью по умолчанию. Добавление или повторная авторизация провайдера делает его модели доступными, не меняя текущую основную модель. Чтобы намеренно изменить модель по умолчанию, используйте `openclaw models auth login --provider <id> --set-default` или `openclaw models set <model>`.
</Note>

Если configure запускается с выбора авторизации провайдера, средства выбора модели по умолчанию и списка разрешённых моделей автоматически отдают предпочтение этому провайдеру. Для связанных провайдеров, таких как Volcengine и BytePlus, это предпочтение также распространяется на варианты их тарифных планов для программирования (`volcengine-plan/*`, `byteplus-plan/*`). Если фильтр предпочтительного провайдера даёт пустой список, configure вместо пустого средства выбора возвращается к полному каталогу без фильтрации.

## Раздел веб-доступа

`openclaw configure --section web` выбирает провайдера веб-поиска и настраивает его учётные данные. Для некоторых провайдеров отображаются дополнительные параметры:

- **Grok** может предложить необязательную настройку `x_search` с тем же профилем xAI OAuth или ключом API, а также позволить выбрать модель `x_search`.
- **Kimi** может запросить регион Moonshot API (`api.moonshot.ai` или `api.moonshot.cn`) и модель Kimi по умолчанию для веб-поиска.

## Другие примечания

- После записи локальной конфигурации configure устанавливает выбранные загружаемые плагины, если они требуются выбранным способом настройки. Конфигурация удалённого Gateway не устанавливает локальные пакеты плагинов.
- При настройке сервисов, ориентированных на каналы (Slack/Discord/Matrix/Microsoft Teams), запрашиваются списки разрешённых каналов или комнат. Можно вводить имена или идентификаторы; когда это возможно, мастер преобразует имена в идентификаторы.
- Если выполняется этап установки фоновой службы, для аутентификации по токену требуется токен. Если `gateway.auth.token` управляется через SecretRef, configure проверяет SecretRef, но не сохраняет полученное открытое значение токена в метаданных среды службы супервизора; если SecretRef не разрешается, configure блокирует установку фоновой службы и предоставляет практические инструкции по устранению проблемы.
- Если настроены и `gateway.auth.token`, и `gateway.auth.password`, а значение `gateway.auth.mode` не задано, configure блокирует установку фоновой службы, пока режим не будет задан явно.
