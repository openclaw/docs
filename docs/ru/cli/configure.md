# `openclaw configure`

Интерактивные запросы для целевых изменений существующей конфигурации: учётных данных, устройств, параметров агента по умолчанию, Gateway, каналов, плагинов, Skills и проверок работоспособности.

Используйте `openclaw onboard` или `openclaw setup` для полного пошагового процесса первоначальной настройки, `openclaw setup --baseline` — только для базовой конфигурации и рабочего пространства, а `openclaw channels add` — если требуется настроить только учётную запись канала.

<Tip>
Команда `openclaw config` без подкоманды открывает тот же мастер. Для неинтерактивного редактирования используйте `openclaw config get|set|unset`.
</Tip>

## Параметры

`--section <section>`: повторяемый фильтр разделов. Доступные разделы:

`workspace`, `model`, `web`, `gateway`, `daemon`, `channels`, `plugins`, `skills`, `health`

```bash
openclaw configure
openclaw configure --section web
openclaw configure --section model --section channels
openclaw configure --section gateway --section daemon
```

При выборе `gateway`, `daemon` или `health` (либо при запуске полного мастера без `--section`) появляется запрос о том, где работает Gateway, и обновляется `gateway.mode`. Если фильтры разделов исключают все три раздела, мастер сразу переходит к запрошенной настройке, не запрашивая режим Gateway. При выборе удалённого режима Gateway мастер записывает удалённую конфигурацию и немедленно завершает работу; локальные действия, например установку плагинов, он не выполняет.

<Note>
Для `openclaw configure` требуется интерактивный терминал (и stdin, и stdout должны быть TTY). Без него команда выводит эквивалентные неинтерактивные команды `openclaw config get|set|patch|validate` и завершается с ошибкой, не выполняя настройку частично.
</Note>

## Раздел модели

<Note>
Раздел **«Модель»** содержит множественный выбор для списка разрешённых моделей `agents.defaults.models` (он определяет, что отображается в `/model` и средстве выбора модели). Варианты настройки, относящиеся к конкретному провайдеру, добавляют выбранные модели в существующий список разрешённых, а не заменяют уже настроенные модели других провайдеров.

Повторная аутентификация провайдера через мастер настройки сохраняет существующее значение `agents.defaults.model.primary`, даже если этап аутентификации провайдера возвращает исправление конфигурации с собственной рекомендуемой моделью по умолчанию. При добавлении или повторной аутентификации провайдера его модели становятся доступными, но текущая основная модель не изменяется. Чтобы намеренно изменить модель по умолчанию, используйте `openclaw models auth login --provider <id> --set-default` или `openclaw models set <model>`.
</Note>

Если мастер настройки запускается с выбора аутентификации провайдера, средства выбора модели по умолчанию и списка разрешённых автоматически отдают предпочтение этому провайдеру. Для парных провайдеров, таких как Volcengine и BytePlus, это предпочтение также распространяется на варианты их тарифных планов для программирования (`volcengine-plan/*`, `byteplus-plan/*`). Если фильтр предпочтительного провайдера дал бы пустой список, мастер настройки использует каталог без фильтрации, а не отображает пустое средство выбора.

## Раздел веб-поиска

Команда `openclaw configure --section web` позволяет выбрать провайдера веб-поиска и настроить его учётные данные. Для некоторых провайдеров отображаются дополнительные параметры:

- **Grok** может предложить необязательную настройку `x_search` с использованием того же профиля xAI OAuth или ключа API, а также выбор модели `x_search`.
- **Kimi** может запросить регион API Moonshot (`api.moonshot.ai` или `api.moonshot.cn`) и модель веб-поиска Kimi по умолчанию.

## Прочие примечания

- После записи локальной конфигурации мастер устанавливает выбранные загружаемые плагины, если они требуются для выбранного варианта настройки. При настройке удалённого Gateway локальные пакеты плагинов не устанавливаются.
- Сервисы, ориентированные на каналы (Slack/Discord/Matrix/Microsoft Teams), во время настройки запрашивают списки разрешённых каналов или комнат. Можно вводить названия или идентификаторы; по возможности мастер преобразует названия в идентификаторы.
- При выполнении этапа установки фоновой службы для аутентификации по токену требуется токен. Если `gateway.auth.token` управляется через SecretRef, мастер проверяет SecretRef, но не сохраняет полученное открытое значение токена в метаданных окружения службы супервизора; если SecretRef не разрешается, мастер блокирует установку фоновой службы и предоставляет практические рекомендации по устранению проблемы.
- Если настроены и `gateway.auth.token`, и `gateway.auth.password`, а `gateway.auth.mode` не задан, мастер блокирует установку фоновой службы, пока режим не будет указан явно.
