# `openclaw logs`

Отслеживает файловые журналы Gateway через RPC (работает в удаленном режиме).

Связано:

- Обзор журналирования: [Журналирование](/ru/logging)
- CLI Gateway: [gateway](/ru/cli/gateway)

## Параметры

- `--limit <n>`: максимальное количество строк журнала для возврата (по умолчанию `200`)
- `--max-bytes <n>`: максимальное количество байтов для чтения из файла журнала (по умолчанию `250000`)
- `--follow`: следить за потоком журнала
- `--interval <ms>`: интервал опроса при слежении (по умолчанию `1000`)
- `--json`: выводить события JSON, разделенные строками
- `--plain`: вывод обычного текста без стилизованного форматирования
- `--no-color`: отключить цвета ANSI
- `--local-time`: отображать временные метки в вашем локальном часовом поясе (по умолчанию)
- `--utc`: отображать временные метки в UTC

## Общие параметры RPC Gateway

`openclaw logs` также принимает стандартные флаги клиента Gateway:

- `--url <url>`: WebSocket URL Gateway
- `--token <token>`: токен Gateway
- `--timeout <ms>`: тайм-аут в мс (по умолчанию `30000`)
- `--expect-final`: ждать финального ответа, когда вызов Gateway поддерживается агентом

Когда вы передаете `--url`, CLI не применяет автоматически учетные данные из конфигурации или окружения. Укажите `--token` явно, если целевой Gateway требует аутентификацию.

## Примеры

```bash
openclaw logs
openclaw logs --follow
openclaw logs --follow --interval 2000
openclaw logs --limit 500 --max-bytes 500000
openclaw logs --json
openclaw logs --plain
openclaw logs --no-color
openclaw logs --limit 500
openclaw logs --local-time
openclaw logs --utc
openclaw logs --follow --local-time
openclaw logs --url ws://127.0.0.1:18789 --token "$OPENCLAW_GATEWAY_TOKEN"
```

## Примечания

- По умолчанию временные метки отображаются в вашем локальном часовом поясе. Используйте `--utc` для вывода в UTC.
- Если неявный локальный local loopback Gateway запрашивает сопряжение, закрывается во время подключения или истекает по тайм-ауту до ответа `logs.tail`, `openclaw logs` автоматически переключается на настроенный файловый журнал Gateway. Явные цели `--url` не используют этот резервный вариант.
- `openclaw logs --follow` не отслеживает резервные файловые журналы из конфигурации после сбоев неявного локального RPC Gateway. В Linux он использует активный пользовательский журнал systemd Gateway по PID, когда он доступен, и выводит выбранный источник журнала; в противном случае он продолжает повторять попытки подключения к работающему Gateway вместо отслеживания потенциально устаревшего соседнего файла.
- При использовании `--follow` временные отключения gateway (закрытие WebSocket, тайм-аут, обрыв соединения) запускают автоматическое повторное подключение с экспоненциальной задержкой (до 8 повторов, с ограничением 30 с между попытками). При каждой повторной попытке предупреждение выводится в stderr, а после успешного опроса выводится уведомление `[logs] gateway reconnected`. В режиме `--json` и предупреждение о повторной попытке, и переход повторного подключения выводятся как записи `{"type":"notice"}` в stderr. Невосстановимые ошибки (сбой аутентификации, неверная конфигурация) по-прежнему немедленно завершают работу.
- В режиме `--follow --json` переходы источника журнала выводятся как записи `{"type":"meta"}`. Потребителям следует отслеживать курсоры отдельно для каждого `sourceKind`: поток может перейти от файлового вывода Gateway (`sourceKind: "file"`) к локальному резервному журналу (`sourceKind: "journal"`, `localFallback: true`, с `service.pid`/`service.unit`) и обратно к файловому выводу Gateway после восстановления. Не предполагайте один стабильный источник или курсор для всего сеанса слежения и допускайте перекрывающиеся строки, когда восстановление повторно воспроизводит курсор файла Gateway.
