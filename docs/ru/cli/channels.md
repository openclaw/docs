# `openclaw channels`

Управление учётными записями каналов чата и их состоянием выполнения на Gateway.

Связанная документация:

- Руководства по каналам: [Каналы](/ru/channels)
- Настройка Gateway: [Конфигурация](/ru/gateway/configuration)

## Основные команды

```bash
openclaw channels list
openclaw channels list --all
openclaw channels status
openclaw channels capabilities
openclaw channels capabilities --channel discord --target channel:123
openclaw channels resolve --channel slack "#general" "@jane"
openclaw channels logs --channel all
```

`channels list` показывает только каналы чата: по умолчанию — настроенные учётные записи с метками состояния `installed`, `configured` и `enabled` для каждой учётной записи (`--json` для машиночитаемого вывода). Передайте `--all`, чтобы также показать встроенные каналы, для которых ещё не настроена учётная запись, и доступные для установки каналы из каталога, которых ещё нет на диске. Аутентификация провайдеров и использование моделей управляются в других местах: `openclaw models auth list` — для профилей аутентификации провайдеров, `openclaw status` или `openclaw models list` — для данных об использовании и квотах.

## Состояние / возможности / разрешение имён / журналы

- `channels status`: `--channel <name>`, `--probe`, `--timeout <ms>` (по умолчанию `10000`), `--json`
- `channels capabilities`: `--channel <name>`, `--account <id>` (требует `--channel`), `--target <dest>` (требует `--channel`), `--timeout <ms>` (по умолчанию `10000`, максимум `30000`), `--json`
- `channels resolve <entries...>`: `--channel <name>`, `--account <id>`, `--kind <auto|user|group>` (по умолчанию `auto`), `--json`
- `channels logs`: `--channel <name|all>` (по умолчанию `all`), `--lines <n>` (по умолчанию `200`), `--json`

`channels status --probe` использует проверку в реальном времени: если Gateway доступен, для каждой учётной записи выполняются проверки
`probeAccount` и, при наличии, `auditAccount`, поэтому вывод может содержать состояние
транспорта и результаты проверок, например `works`, `probe failed`, `audit ok` или `audit failed`.
Если Gateway недоступен, `channels status` вместо результатов проверки в реальном времени
выводит сводки только на основе конфигурации.

Не используйте `openclaw sessions`, Gateway `sessions.list` или инструмент агента
`sessions_list` как индикатор состояния сокета канала. Эти интерфейсы сообщают
о сохранённых записях разговоров, а не о состоянии выполнения провайдера. После
перезапуска провайдера Discord подключённая, но неактивная учётная запись может быть
исправна, хотя запись сеанса Discord не появится до следующего входящего или исходящего события разговора.

## Добавление и удаление учётных записей

```bash
openclaw channels add --channel telegram --token <bot-token>
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY"
openclaw channels remove --channel telegram --delete
```

<Tip>
`openclaw channels add --help` показывает флаги для каждого канала (токен, закрытый ключ, токен приложения, пути signal-cli и т. д.).
</Tip>

`channels remove` работает только с установленными или настроенными Plugin каналов. Для доступных к установке каналов из каталога сначала используйте `channels add`. Без `--delete` команда предлагает отключить учётную запись и сохраняет её конфигурацию; `--delete` удаляет записи конфигурации без запроса подтверждения.
Для Plugin каналов, работающих через среду выполнения, `channels remove` также просит работающий Gateway остановить выбранную учётную запись перед обновлением конфигурации, поэтому после отключения или удаления учётной записи прежний прослушиватель не остаётся активным до перезапуска.

Общие для каналов флаги неинтерактивного добавления: `--account <id>`, `--name <name>`, `--token`, `--token-file`, `--bot-token`, `--app-token`, `--secret`, `--secret-file`, `--password`, `--cli-path`, `--url`, `--base-url`, `--http-url`, `--auth-dir` и `--use-env` (аутентификация через переменные окружения, только для учётной записи по умолчанию, если поддерживается). Флаги для отдельных каналов:

| Канал       | Флаги                                                                                               |
| ----------- | ---------------------------------------------------------------------------------------------------- |
| Google Chat | `--webhook-path`, `--webhook-url`, `--audience-type`, `--audience`                                   |
| iMessage    | `--cli-path`, `--db-path`, `--service`, `--region`                                                   |
| Matrix      | `--homeserver`, `--user-id`, `--access-token`, `--password`, `--device-name`, `--initial-sync-limit` |
| Nostr       | `--private-key`, `--relay-urls`                                                                      |
| Signal      | `--signal-number`, `--cli-path`, `--http-url`, `--http-host`, `--http-port`                          |
| Tlon        | `--ship`, `--url`, `--code`, `--group-channels`, `--dm-allowlist`, `--auto-discover-channels`        |
| WhatsApp    | `--auth-dir`                                                                                         |

Если при добавлении с помощью флагов требуется установить Plugin канала, OpenClaw использует источник установки канала по умолчанию, не открывая интерактивный запрос установки Plugin.

При запуске `openclaw channels add` без флагов интерактивный мастер может запросить:

- идентификаторы учётных записей для каждого выбранного канала
- необязательные отображаемые имена для этих учётных записей
- `Route these channel accounts to agents now?`

Если подтвердить привязку сейчас, мастер запросит, какой агент должен владеть каждой настроенной учётной записью канала, и запишет привязки маршрутизации для отдельных учётных записей.

Этими же правилами маршрутизации можно управлять позднее с помощью `openclaw agents bindings`, `openclaw agents bind` и `openclaw agents unbind` (см. [агенты](/ru/cli/agents)).

При добавлении учётной записи не по умолчанию в канал, который всё ещё использует настройки верхнего уровня для одной учётной записи, OpenClaw переносит эти значения верхнего уровня в карту учётных записей канала перед записью новой учётной записи. При переносе повторно используется существующая именованная учётная запись, если у канала ровно одна такая запись или если `defaultAccount` указывает на неё; в противном случае значения помещаются в `channels.<channel>.accounts.default`.

Поведение маршрутизации остаётся согласованным:

- Существующие привязки только к каналу (без `accountId`) продолжают соответствовать учётной записи по умолчанию.
- В неинтерактивном режиме `channels add` не создаёт и не перезаписывает привязки автоматически.
- При интерактивной настройке можно дополнительно добавить привязки для отдельных учётных записей.

Если конфигурация уже находилась в смешанном состоянии (присутствовали именованные учётные записи и одновременно были заданы значения верхнего уровня для одной учётной записи), выполните `openclaw doctor --fix`, чтобы перенести значения учётной записи в выбранную для этого канала перенесённую учётную запись.

## Вход и выход (интерактивно)

```bash
openclaw channels login --channel whatsapp
openclaw channels logout --channel whatsapp
```

- `channels login` поддерживает `--account <id>` и `--verbose`; `channels logout` поддерживает `--account <id>`.
- `channels login` и `logout` могут определить канал, если это действие поддерживает только один настроенный канал; если каналов несколько, передайте `--channel`.
- `channels logout` предпочитает путь через работающий Gateway, если он доступен, поэтому при выходе активный прослушиватель останавливается до очистки состояния аутентификации канала. Если локальный Gateway недоступен, команда переходит к локальной очистке аутентификации; при `gateway.mode: "remote"` ошибка Gateway вместо этого приводит к завершению команды с ошибкой.
- После успешного входа CLI просит доступный локальный Gateway запустить учётную запись; в удалённом режиме он сохраняет аутентификационные данные локально и сообщает, что удалённая среда выполнения не была перезапущена.
- Запускайте `channels login` из терминала на узле Gateway. `exec` агента блокирует этот интерактивный процесс входа; для входа из чата следует использовать встроенные в канал инструменты входа агента, например `whatsapp_login`, если они доступны.

## Устранение неполадок

- Выполните `openclaw status --deep` для комплексной проверки.
- Используйте `openclaw doctor` для пошагового исправления проблем.
- Если Gateway недоступен, `openclaw channels status` выводит сводки только на основе конфигурации. Если учётные данные поддерживаемого канала настроены через SecretRef, но недоступны в текущем пути выполнения команды, соответствующая учётная запись отмечается как настроенная с примечаниями об ограниченной работоспособности, а не как ненастроенная.

## Проверка возможностей

Получение сведений о возможностях провайдера (намерениях и областях доступа, если доступны), а также о поддержке статических функций:

```bash
openclaw channels capabilities
openclaw channels capabilities --channel discord --target channel:123
```

Примечания:

- `--channel` необязателен; не указывайте его, чтобы вывести все каналы (включая предоставляемые Plugin).
- `--account` допустим только вместе с `--channel`.
- `--target` принимает `channel:<id>` или необработанный числовой идентификатор канала и применяется только к Discord. Для голосовых каналов Discord проверка разрешений отмечает отсутствие `ViewChannel`, `Connect`, `Speak`, `SendMessages` и `ReadMessageHistory`.
- Проверки зависят от провайдера: идентификатор бота Discord и намерения, а также необязательные разрешения канала; области доступа бота и пользователя Slack; флаги бота Telegram и Webhook; версия демона Signal; токен приложения Microsoft Teams и роли/области доступа Graph (с пояснениями, если известны). Для каналов без проверок выводится `Probe: unavailable`.

## Разрешение имён в идентификаторы

Преобразование имён каналов и пользователей в идентификаторы с помощью каталога провайдера:

```bash
openclaw channels resolve --channel slack "#general" "@jane"
openclaw channels resolve --channel discord "My Server/#support" "@someone"
openclaw channels resolve --channel matrix "Project Room"
```

Примечания:

- Используйте `--kind user|group|auto`, чтобы явно задать тип цели.
- Если несколько записей имеют одинаковое имя, при разрешении предпочтение отдаётся активным совпадениям.
- `channels resolve` работает только для чтения. Если выбранная учётная запись настроена через SecretRef, но эти учётные данные недоступны в текущем пути выполнения команды, команда возвращает неразрешённые результаты с ограниченной работоспособностью и примечаниями, а не прерывает весь запуск.
- `channels resolve` не устанавливает Plugin каналов. Перед разрешением имён для доступного к установке канала из каталога используйте `channels add --channel <name>`.
