# `openclaw update`

Обновление OpenClaw и переключение между каналами stable/extended-stable/beta/dev.

Если установка выполнена через **npm/pnpm/bun** (глобальная установка без метаданных git),
обновления выполняются через процесс менеджера пакетов, описанный в разделе
[Обновление](/ru/install/updating).

## Использование

```bash
openclaw update
openclaw update status
openclaw update repair
openclaw update wizard
openclaw update --channel extended-stable
openclaw update --channel beta
openclaw update --channel dev
openclaw update --tag beta
openclaw update --tag main
openclaw update --dry-run
openclaw update --no-restart
openclaw update --yes
openclaw update --acknowledge-clawhub-risk
openclaw update --json
openclaw --update
```

`openclaw --update` преобразуется в `openclaw update` (это удобно для оболочек и
скриптов запуска).

## Параметры

| Флаг                                             | Описание                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--no-restart`                                   | Не перезапускать службу Gateway после успешного обновления. При обновлениях через менеджер пакетов, предусматривающих перезапуск, перед успешным завершением команды проверяется, что перезапущенная служба сообщает ожидаемую версию.                                                                                                                                                |
| `--channel <stable\|extended-stable\|beta\|dev>` | Задать канал обновления и сохранить его после успешного обновления ядра. Канал extended-stable доступен только для пакетных установок.                                                                                                                                                                                                                                            |
| `--tag <dist-tag\|version\|spec>`                | Переопределить целевой пакет только для этого обновления. Этот параметр нельзя использовать вместе с действующим каналом `extended-stable`, для которого обязательно указание проверенной точной цели. Для других пакетных установок `main` сопоставляется с `github:openclaw/openclaw#main`; спецификации источников GitHub/git упаковываются во временный tar-архив перед поэтапной глобальной установкой через npm. |
| `--dry-run`                                      | Предварительно показать запланированные действия (канал/тег/цель/процесс перезапуска), не изменяя конфигурацию, не устанавливая пакеты, не синхронизируя плагины и не выполняя перезапуск.                                                                                                                                                                                                                |
| `--json`                                         | Вывести машиночитаемый JSON `UpdateRunResult`. Включает `postUpdate.plugins.warnings`, если управляемому плагину требуется восстановление, сведения о резервном выборе плагина для beta-канала и `postUpdate.plugins.integrityDrifts`, если при синхронизации после обновления обнаружено расхождение артефакта npm-плагина.                                                                 |
| `--timeout <seconds>`                            | Тайм-аут каждого этапа. По умолчанию — `1800`.                                                                                                                                                                                                                                                                                                            |
| `--yes`                                          | Пропустить запросы подтверждения (например, подтверждение перехода на более раннюю версию).                                                                                                                                                                                                                                                                              |
| `--acknowledge-clawhub-risk`                     | Разрешить синхронизации плагинов после обновления продолжаться при предупреждениях о доверии к сторонним пакетам ClawHub без интерактивного запроса. Без этого параметра потенциально небезопасные сторонние выпуски пропускаются и остаются без изменений, если OpenClaw не может запросить подтверждение. Официальные пакеты ClawHub и встроенные источники плагинов обходят этот запрос.                                                     |

Флага `--verbose` не существует. Используйте `--dry-run` для предварительного просмотра запланированных действий,
`--json` для машиночитаемых результатов и `openclaw update status --json`
только для сведений о канале и доступности. Подробность вывода в консоль Gateway (`--verbose`) и
уровень журналирования в файл (`logging.level: "debug"`/`"trace"`) настраиваются независимо; см.
[Журналирование Gateway](/ru/gateway/logging).

<Note>
В режиме Nix (`OPENCLAW_NIX_MODE=1`) изменяющие состояние запуски `openclaw update` отключены. Вместо этого обновите источник Nix или входные данные flake для этой установки; для nix-openclaw используйте ориентированное на агента [Краткое руководство](https://github.com/openclaw/nix-openclaw#quick-start). `openclaw update status` и `openclaw update --dry-run` остаются доступными только для чтения.
</Note>

<Warning>
Переход на более раннюю версию требует подтверждения, поскольку старые версии могут нарушить работу конфигурации.
Если установка уже перенесла сеансы в SQLite, перед запуском старой версии
с файловым хранилищем восстановите архивные артефакты прежних расшифровок. См.
[Doctor: переход на более раннюю версию после миграции сеансов в SQLite](/ru/cli/doctor#downgrading-after-session-sqlite-migration).
</Warning>

## `update status`

Показать активный канал обновления, тег/ветку/SHA git (только для установок из исходного кода)
и наличие обновлений.

```bash
openclaw update status
openclaw update status --json
openclaw update status --timeout 10
```

| Флаг                  | По умолчанию | Описание                         |
| --------------------- | ------- | ----------------------------------- |
| `--json`              | `false` | Вывести машиночитаемый JSON состояния. |
| `--timeout <seconds>` | `3`     | Тайм-аут проверок.                 |

Для пакетных установок extended-stable команда состояния использует тот же общедоступный селектор
и ту же проверку точного пакета, что и интерактивное обновление. Она может сообщить
`ahead of extended-stable`, если установленная версия новее. Ошибки JSON
включают `registry.reason` (`selector_missing`, `selector_query_failed`,
`exact_package_mismatch` или `unsupported_git_channel`).

## `update repair`

Повторно выполнить завершение обновления, если основной пакет уже изменён, но последующие
операции восстановления не завершились корректно. Это поддерживаемый способ восстановления, когда
`openclaw update` установил новый основной пакет, но синхронизация плагинов после обновления ядра,
метаданные управляемых npm-плагинов, обновление реестра или восстановление через doctor
не достигли согласованного состояния.

```bash
openclaw update repair
openclaw update repair --channel beta
openclaw update repair --acknowledge-clawhub-risk
openclaw update repair --json
```

| Флаг                                             | Описание                                                                                                                                                                                                                                                         |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--channel <stable\|extended-stable\|beta\|dev>` | Сохранить канал обновления ядра перед восстановлением. Для extended-stable подходящие официальные npm-плагины с пустым/стандартным намерением или намерением `latest` нацеливаются на точную установленную версию ядра. Восстановление extended-stable отклоняется в рабочих копиях Git без изменения конфигурации. |
| `--json`                                         | Вывести машиночитаемый JSON завершения.                                                                                                                                                                                                                           |
| `--timeout <seconds>`                            | Тайм-аут этапов восстановления. По умолчанию — `1800`.                                                                                                                                                                                                                           |
| `--yes`                                          | Пропустить запросы подтверждения.                                                                                                                                                                                                                                          |
| `--acknowledge-clawhub-risk`                     | Поведение аналогично `openclaw update`.                                                                                                                                                                                                                              |
| `--no-restart`                                   | Принимается для единообразия; восстановление никогда не перезапускает Gateway.                                                                                                                                                                                                             |

`update repair` запускает `openclaw doctor --fix`, повторно загружает восстановленную конфигурацию и
записи об установке, синхронизирует отслеживаемые плагины для активного канала обновления, обновляет
установки управляемых npm-плагинов, восстанавливает отсутствующее содержимое настроенных плагинов,
обновляет реестр плагинов и записывает согласованные метаданные записей об установке.
Команда не устанавливает новый основной пакет и не перезапускает Gateway.

## `update wizard`

Интерактивный процесс выбора канала обновления и подтверждения необходимости последующего
перезапуска Gateway (по умолчанию выполняется перезапуск). При выборе `dev` без рабочей
копии git предлагается создать её.

| Флаг                  | По умолчанию | Описание                   |
| --------------------- | ------- | ----------------------------- |
| `--timeout <seconds>` | `1800`  | Тайм-аут каждого этапа обновления. |

## Принцип работы

Явное переключение каналов (`--channel ...`) также обеспечивает соответствие способа
установки:

- `dev` -> обеспечивает наличие рабочей копии git (по умолчанию `~/openclaw` или
  `$OPENCLAW_HOME/openclaw`, если задан `OPENCLAW_HOME`; переопределяется с помощью
  `OPENCLAW_GIT_DIR`), обновляет её и устанавливает глобальный CLI из этой
  рабочей копии.
- `stable` -> выполняет установку из npm с помощью `latest`.
- `extended-stable` -> разрешает общедоступный селектор npm `extended-stable`,
  проверяет точный выбранный пакет и устанавливает именно эту версию. Резервный
  переход к другому селектору не выполняется, а для рабочих копий Git этот вариант отклоняется.
- `beta` -> предпочитает dist-tag npm `beta`, переходя к `latest`, если beta
  отсутствует или старее текущего стабильного выпуска.

### Передача управления при перезапуске

Автоматическое обновление ядра Gateway (если оно включено в конфигурации) запускает процесс
обновления CLI вне активного обработчика запросов Gateway. Обновления через менеджер пакетов
плоскости управления `update.run` и контролируемые обновления рабочих копий git используют
тот же механизм передачи управления управляемой службе вместо замены дерева пакетов или
повторной сборки `dist/` внутри активного процесса Gateway: Gateway запускает
отсоединённый вспомогательный процесс и завершает работу, после чего этот процесс выполняет `openclaw update --yes --json`
вне дерева процессов Gateway. Если передача управления недоступна,
`update.run` возвращает структурированный ответ с безопасной командой оболочки для
ручного запуска.

Сохранённые варианты extended-stable получают при запуске доступные только для чтения подсказки и подсказки об обновлении каждые 24 часа, когда включён `update.checkOnStart`. Эти проверки никогда не применяют обновление, не запускают передачу управления, не перезапускают Gateway, не используют задержку/джиттер стабильного канала и не применяют частоту опроса бета-канала. Явные обновления на переднем плане, обычные обновления на переднем плане с сохранённым `update.channel: "extended-stable"`, получение состояния по запросу и связанная с ними передача управления управляемому Gateway по-прежнему поддерживаются.

Если установлена локальная служба управляемого Gateway и включён перезапуск, обновления через менеджер пакетов и обновления Git checkout останавливают работающую службу перед заменой дерева пакета или изменением checkout/результатов сборки. Затем средство обновления обновляет метаданные службы, перезапускает её и проверяет перезапущенный Gateway, прежде чем сообщить `Gateway: restarted and verified.`.
При обновлениях через менеджер пакетов дополнительно проверяется, что перезапущенный Gateway сообщает ожидаемую версию пакета; при обновлениях Git checkout после повторной сборки проверяются работоспособность Gateway и готовность службы.

В macOS проверка после обновления также подтверждает, что LaunchAgent загружен/запущен для активного профиля и настроенный порт loopback работает. Если plist установлен, но launchd его не контролирует, OpenClaw автоматически повторно инициализирует LaunchAgent и снова выполняет проверки работоспособности/версии/
готовности канала (при новой инициализации задание `RunAtLoad` загружается напрямую,
поэтому восстановление не выполняет немедленно `kickstart -k` для только что запущенного Gateway). Если
Gateway по-прежнему не становится работоспособным, команда завершается с ненулевым кодом и
выводит путь к журналу перезапуска, а также инструкции по перезапуску, переустановке и откату пакета.

Если перезапуск выполнить невозможно, команда выводит `Gateway: restart skipped (...)` или
`Gateway: restart failed: ...` с подсказкой о ручном выполнении `openclaw gateway restart`.
При использовании `--no-restart` замена пакета или повторная сборка Git всё равно выполняется, но
управляемая служба не останавливается и не перезапускается, поэтому работающий Gateway продолжает
использовать старый код, пока вы не перезапустите его вручную.

### Формат ответа плоскости управления

Когда `update.run` выполняется через плоскость управления Gateway для установки через менеджер пакетов
или контролируемого Git checkout, обработчик сообщает об инициировании передачи управления
отдельно от обновления CLI, которое продолжается после завершения работы Gateway:

- `ok: true`, `result.status: "skipped"`,
  `result.reason: "managed-service-handoff-started"` и
  `handoff.status: "started"`: Gateway создал передачу управления управляемой службе
  и запланировал собственный перезапуск, чтобы отсоединённый вспомогательный процесс мог выполнить
  `openclaw update --yes --json` вне процесса работающей службы.
- `ok: false`, `result.reason: "managed-service-handoff-unavailable"` и
  `handoff.status: "unavailable"`: OpenClaw не удалось найти границу контролирующей
  службы и устойчивый идентификатор службы для безопасной передачи управления (например,
  для передачи управления systemd требуется идентификатор модуля `OPENCLAW_SYSTEMD_UNIT`,
  а не только доступные из окружения признаки процесса systemd). Ответ содержит
  `handoff.command` — команду оболочки, которую следует выполнить вне Gateway.
- `ok: false`, `result.reason: "managed-service-handoff-failed"`: Gateway
  попытался создать передачу управления, но не смог запустить отсоединённый вспомогательный процесс.

Полезная нагрузка `sentinel` записывается до завершения работы Gateway, а передача управления
CLI обновляет тот же маркер перезапуска после завершения проверок работоспособности
перезапущенной управляемой службы. Во время передачи управления маркер может содержать
`stats.reason: "restart-health-pending"` без продолжения при успешном результате;
перезапущенный Gateway опрашивает его и запускает продолжение только после того, как CLI
проверит работоспособность службы и перезапишет маркер окончательным результатом `ok`.
`openclaw status` и `openclaw status --all` показывают строку `Update restart`,
пока этот маркер ожидает обработки или содержит ошибку, а `update.status` обновляет
и возвращает актуальный маркер.

## Процесс Git checkout

### Выбор канала

- `stable`: извлекает последний тег, не относящийся к бета-версии, затем выполняет сборку и doctor.
- `beta`: отдаёт предпочтение последнему тегу `-beta`, возвращаясь к последнему стабильному тегу,
  если бета-версия отсутствует или устарела.
- `dev`: извлекает `main`, затем получает изменения и выполняет rebase.
- `extended-stable`: не поддерживается для Git checkout; checkout
  не изменяется.

### Этапы обновления

<Steps>
  <Step title="Проверка чистоты рабочего дерева">
    Требует отсутствия незафиксированных изменений.
  </Step>
  <Step title="Переключение канала">
    Переключает на выбранный канал (тег или ветвь).
  </Step>
  <Step title="Получение изменений из upstream">
    Только для dev.
  </Step>
  <Step title="Предварительная сборка (только dev)">
    Запускает сборку TypeScript во временном рабочем дереве. Если вершина не собирается, последовательно отступает до 10 коммитов назад, чтобы найти самый новый собираемый коммит. Задайте `OPENCLAW_UPDATE_PREFLIGHT_LINT=1`, чтобы также запускать lint во время этой предварительной проверки; lint выполняется в ограниченном последовательном режиме, поскольку пользовательские хосты обновления часто менее производительны, чем исполнители CI.
  </Step>
  <Step title="Rebase">
    Выполняет rebase на выбранный коммит (только dev).
  </Step>
  <Step title="Установка зависимостей">
    Использует менеджер пакетов репозитория. Для checkout с pnpm средство обновления при необходимости инициализирует `pnpm` (сначала через `corepack`, затем с временным резервным вариантом `npm install pnpm@11`) вместо запуска `npm run build` внутри рабочего пространства pnpm. Если инициализация pnpm всё равно завершается с ошибкой, средство обновления прекращает работу раньше с ошибкой, относящейся к менеджеру пакетов, вместо попытки выполнить `npm run build` в checkout.
  </Step>
  <Step title="Сборка Control UI">
    Собирает Gateway и Control UI.
  </Step>
  <Step title="Запуск doctor">
    `openclaw doctor` выполняется как завершающая проверка безопасного обновления.
  </Step>
  <Step title="Синхронизация плагинов">
    Синхронизирует плагины с активным каналом. Dev использует встроенные плагины; stable и beta используют npm. Обновляет отслеживаемые установки плагинов.
  </Step>
</Steps>

### Подробности синхронизации плагинов

В бета-канале отслеживаемые установки плагинов npm и ClawHub, следующие
линии default/latest, сначала пытаются использовать выпуск плагина `@beta`. Если у плагина нет
бета-выпуска, OpenClaw возвращается к записанной спецификации default/latest и
выводит предупреждение. Для плагинов npm OpenClaw также использует резервный вариант, если
бета-пакет существует, но не проходит проверку установки. Эти предупреждения о резервном
варианте не приводят к сбою обновления ядра. Точные версии и явно заданные теги никогда не перезаписываются.

<Warning>
Если обновление плагина npm с точно закреплённой версией разрешается в артефакт, целостность которого отличается от сохранённой записи установки, `openclaw update` прерывает обновление этого артефакта плагина вместо его установки. Переустанавливайте или обновляйте плагин явно только после проверки надёжности нового артефакта.
</Warning>

<Note>
Ошибки синхронизации плагинов после обновления, относящиеся к управляемому плагину и допускающие обход в процессе синхронизации (например, недоступный реестр npm для необязательного плагина), выводятся как предупреждения после успешного обновления ядра. Результат JSON сохраняет верхнеуровневое значение обновления `status: "ok"` и сообщает `postUpdate.plugins.status: "warning"` с рекомендациями `openclaw update repair` и `openclaw plugins inspect <id> --runtime --json`. Непредвиденные исключения средства обновления или синхронизации по-прежнему приводят к сбою обновления. Исправьте ошибку установки или обновления плагина, затем повторно выполните `openclaw update repair`.

После этапа синхронизации каждого плагина `openclaw update` выполняет обязательный проход **согласования после обновления ядра** перед перезапуском Gateway: восстанавливает отсутствующую настроенную полезную нагрузку плагинов, проверяет на диске каждую запись об _активной_ отслеживаемой установке и статически проверяет возможность разбора её `package.json` (а также наличие любого явно объявленного `main`). Ошибки этого прохода, а также недопустимый снимок конфигурации возвращают `postUpdate.plugins.status: "error"` и меняют верхнеуровневое значение обновления `status` на `"error"`, поэтому `openclaw update` завершается с ненулевым кодом, а Gateway _не_ перезапускается с непроверенным набором плагинов. Ошибка содержит структурированные строки `postUpdate.plugins.warnings[].guidance`, указывающие на `openclaw update repair` и `openclaw plugins inspect <id> --runtime --json`. Отключённые записи плагинов и записи, которые не являются связанными с доверенным источником официальными целями синхронизации, здесь пропускаются (в соответствии с политикой `skipDisabledPlugins`, используемой при проверке отсутствующей полезной нагрузки), поэтому устаревшая запись отключённого плагина не может заблокировать в остальном корректное обновление.

При запуске обновлённого Gateway загрузка плагинов выполняется только в режиме проверки: запуск не вызывает менеджеры пакетов и не изменяет деревья зависимостей. Перезапуски менеджера пакетов `update.run` передаются управляемой службе через CLI, поэтому замена пакета происходит вне старого процесса Gateway, а проверки работоспособности службы определяют, можно ли считать обновление завершённым.
</Note>

После успешного обновления ядра extended-stable проверка целостности и
согласование плагинов после обновления ядра нацеливаются на подходящие официальные плагины npm с точной версией
установленного ядра. Для намерения default/`latest` OpenClaw не запрашивает
`@extended-stable` плагина и не возвращается к npm `latest`; версия пакета
определяется по установленному ядру. Явно закреплённые версии, явно заданные теги, отличные от `latest`,
сторонние пакеты и источники, отличные от npm, сохраняют существующее намерение.

Для установок через менеджер пакетов `openclaw update` определяет целевую версию пакета
до вызова менеджера пакетов. Глобальные установки npm используют поэтапную
установку: OpenClaw устанавливает новый пакет во временный префикс npm,
проверяет там состав упакованного `dist`, а затем подменяет этим чистым деревом пакета
настоящее глобальное дерево с соответствующим префиксом. Если проверка завершается с ошибкой, doctor после обновления,
синхронизация плагинов и перезапуск не выполняются из подозрительного дерева. Даже если
установленная версия уже соответствует целевой, команда обновляет
глобальную установку пакета, затем выполняет синхронизацию плагинов, обновление автодополнения
команд ядра и перезапуск. Это поддерживает согласованность упакованных вспомогательных компонентов и принадлежащих каналу
записей плагинов с установленной сборкой OpenClaw, оставляя полную
перестройку автодополнения команд плагинов для явных запусков
`openclaw completion --write-state`.
