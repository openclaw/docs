# `openclaw migrate`

Импортируйте состояние из другой агентной системы через провайдер миграции, принадлежащий Plugin. Встроенные провайдеры поддерживают Claude, Codex CLI и [Hermes](/ru/install/migrating-hermes); плагины могут регистрировать дополнительные провайдеры.

<Tip>
Пошаговые инструкции для пользователей см. в разделах [Миграция с Claude](/ru/install/migrating-claude) и [Миграция с Hermes](/ru/install/migrating-hermes). Все варианты перечислены в [центре миграции](/ru/install/migrating).
</Tip>

## Команды

```bash
openclaw migrate list
openclaw migrate claude --dry-run
openclaw migrate codex --dry-run
openclaw migrate codex --skill gog-vault77-google-workspace
openclaw migrate codex --plugin google-calendar --dry-run
openclaw migrate codex --plugin google-calendar --verify-plugin-apps --dry-run
openclaw migrate hermes --dry-run
openclaw migrate hermes
openclaw migrate apply codex --yes --skill gog-vault77-google-workspace
openclaw migrate apply codex --yes --plugin google-calendar
openclaw migrate apply codex --yes
openclaw migrate apply claude --yes
openclaw migrate apply hermes --yes
openclaw migrate apply hermes --include-secrets --yes
openclaw onboard --flow import
openclaw onboard --import-from claude --import-source ~/.claude
openclaw onboard --import-from hermes --import-source ~/.hermes
```

Запуск `openclaw migrate <provider>` без других флагов формирует и предварительно показывает план, а затем (в TTY) запрашивает подтверждение перед его применением. Команды `openclaw migrate plan <provider>` и `openclaw migrate apply <provider>` разделяют предварительный просмотр и применение на отдельные подкоманды с одинаковыми флагами.

<ParamField path="<provider>" type="string">
  Имя зарегистрированного провайдера миграции, например `hermes`. Чтобы просмотреть установленные провайдеры, выполните `openclaw migrate list`.
</ParamField>
<ParamField path="--dry-run" type="boolean">
  Сформировать план и завершить работу без изменения состояния.
</ParamField>
<ParamField path="--from <path>" type="string">
  Переопределить каталог исходного состояния. По умолчанию Hermes использует `~/.hermes`, Codex — `~/.codex` (или `$CODEX_HOME`), а Claude — `~/.claude`.
</ParamField>
<ParamField path="--include-secrets" type="boolean">
  Импортировать поддерживаемые учётные данные без запроса подтверждения. При интерактивном применении перед импортом обнаруженных учётных данных аутентификации выводится запрос, в котором по умолчанию выбран утвердительный ответ; при неинтерактивном применении с `--yes` для их импорта требуется `--include-secrets`.
</ParamField>
<ParamField path="--no-auth-credentials" type="boolean">
  Пропустить импорт учётных данных аутентификации, включая интерактивный запрос.
</ParamField>
<ParamField path="--overwrite" type="boolean">
  Разрешить при применении замену существующих целевых объектов, если в плане указаны конфликты.
</ParamField>
<ParamField path="--yes" type="boolean">
  Пропустить запрос подтверждения. Обязателен в неинтерактивном режиме.
</ParamField>
<ParamField path="--skill <name>" type="string">
  Выбрать один элемент копирования навыка по имени навыка или идентификатору элемента. Повторите флаг, чтобы перенести несколько навыков. Если флаг не указан, при интерактивной миграции Codex отображается список с флажками, а при неинтерактивной миграции сохраняются все запланированные навыки.
</ParamField>
<ParamField path="--plugin <name>" type="string">
  Выбрать один элемент установки Plugin Codex по имени Plugin или идентификатору элемента. Повторите флаг, чтобы перенести несколько плагинов Codex. Если флаг не указан, при интерактивной миграции Codex отображается встроенный список плагинов Codex с флажками, а при неинтерактивной миграции сохраняются все запланированные плагины. Применимо только к установленным из исходного расположения плагинам Codex из `openai-curated`, обнаруженным по данным инвентаризации сервера приложений Codex.
</ParamField>
<ParamField path="--verify-plugin-apps" type="boolean">
  Только для Codex. Перед планированием активации встроенного Plugin принудительно выполняет новый обход `app/list` исходного сервера приложений Codex. По умолчанию отключено, чтобы планирование миграции выполнялось быстро.
</ParamField>
<ParamField path="--backup-output <path>" type="string">
  Путь к архиву резервной копии перед миграцией или к каталогу для него. Передаётся команде `openclaw backup create`.
</ParamField>
<ParamField path="--no-backup" type="boolean">
  Пропустить резервное копирование перед применением. Если локальное состояние OpenClaw существует, требуется `--force`.
</ParamField>
<ParamField path="--force" type="boolean">
  Требуется вместе с `--no-backup`, если без этого применение откажется пропускать резервное копирование.
</ParamField>
<ParamField path="--json" type="boolean">
  Вывести план или результат применения в формате JSON. Если указать `--json` без `--yes`, команда применения выводит план и не изменяет состояние.
</ParamField>

## Модель безопасности

`openclaw migrate` сначала выполняет предварительный просмотр.

<AccordionGroup>
  <Accordion title="Предварительный просмотр перед применением">
    До внесения любых изменений провайдер возвращает детализированный план, включая конфликты, пропущенные и конфиденциальные элементы. В планах JSON, результатах применения и отчётах о миграции скрываются значения вложенных ключей, похожих на секретные данные, например ключей API, токенов, заголовков авторизации, файлов cookie и паролей.

    `openclaw migrate apply <provider>` показывает предварительный план и запрашивает подтверждение перед изменением состояния, если не указан `--yes`. В неинтерактивном режиме для применения требуется `--yes`.

  </Accordion>
  <Accordion title="Резервные копии">
    Перед применением миграции команда создаёт и проверяет резервную копию OpenClaw. Если локальное состояние OpenClaw ещё не существует, этап резервного копирования пропускается и миграция продолжается. Чтобы пропустить резервное копирование при наличии состояния, укажите одновременно `--no-backup` и `--force`.
  </Accordion>
  <Accordion title="Конфликты">
    Команда применения отказывается продолжать работу при наличии конфликтов в плане. Просмотрите план, а затем повторно запустите команду с `--overwrite`, если существующие целевые объекты нужно заменить намеренно. Провайдеры всё равно могут сохранять резервные копии отдельных перезаписываемых файлов в каталоге отчёта о миграции.
  </Accordion>
  <Accordion title="Секреты">
    При интерактивном применении выводится запрос на импорт обнаруженных учётных данных аутентификации, в котором по умолчанию выбран утвердительный ответ. Используйте `--no-auth-credentials`, чтобы пропустить их, или `--include-secrets`, чтобы импортировать учётные данные без участия пользователя вместе с `--yes`.
  </Accordion>
</AccordionGroup>

## Провайдер Claude

Встроенный провайдер Claude по умолчанию обнаруживает состояние Claude Code в `~/.claude`. Используйте `--from <path>`, чтобы импортировать конкретный домашний каталог или корень проекта Claude Code.

<Tip>
Пошаговую инструкцию для пользователей см. в разделе [Миграция с Claude](/ru/install/migrating-claude).
</Tip>

### Что импортирует Claude

- Проектные файлы `CLAUDE.md` и `.claude/CLAUDE.md` в рабочую область агента OpenClaw (`AGENTS.md`).
- Пользовательский файл `~/.claude/CLAUDE.md`, добавляемый в конец файла `USER.md` рабочей области.
- Определения серверов MCP из проектного файла `.mcp.json`, файла Claude Code `~/.claude.json` (включая записи для отдельных проектов) и файла Claude Desktop `claude_desktop_config.json`.
- Каталоги навыков Claude, содержащие `SKILL.md` (пользовательский `~/.claude/skills` и проектный `.claude/skills`).
- Markdown-файлы команд Claude (пользовательские из `~/.claude/commands` и проектные из `.claude/commands`), преобразуемые в навыки OpenClaw только с ручным вызовом.

### Архивируемое состояние и элементы для ручной проверки

Обработчики Claude, разрешения, значения окружения по умолчанию, проектный файл `CLAUDE.local.md`, каталог `.claude/rules`, пользовательские и проектные каталоги `agents/`, а также история проектов (`projects`, `cache` и `plans` в `~/.claude`) сохраняются в отчёте о миграции или указываются как элементы для ручной проверки. OpenClaw не запускает обработчики, не копирует широкие списки разрешений и не импортирует автоматически состояние учётных данных OAuth или Desktop.

## Провайдер Codex

Встроенный провайдер Codex по умолчанию обнаруживает состояние Codex CLI в `~/.codex` или в `CODEX_HOME`, если задана эта переменная окружения. Используйте `--from <path>`, чтобы получить инвентаризацию конкретного домашнего каталога Codex.

Используйте этот провайдер при переходе на среду выполнения Codex в OpenClaw, если требуется целенаправленно перенести полезные личные ресурсы Codex CLI. Локальные серверы приложений Codex запускаются с отдельным `CODEX_HOME` для каждого агента, поэтому по умолчанию они не читают ваш личный каталог `~/.codex`. Обычное значение `HOME` процесса при этом наследуется, поэтому Codex видит общие навыки и записи каталогов плагинов из `$HOME/.agents/*`, а подпроцессы могут находить конфигурацию и токены из домашнего каталога пользователя.

При запуске `openclaw migrate codex` в интерактивном терминале сначала отображается полный план, затем перед окончательным подтверждением применения открываются списки с флажками. Сначала предлагается выбрать элементы копирования навыков. Для массового выбора используйте `Toggle all on` или `Toggle all off`. Нажмите пробел, чтобы переключить состояние строки, или Enter, чтобы активировать выделенную строку и продолжить. Запланированные навыки изначально отмечены, навыки с конфликтами — не отмечены, а `Skip for now` пропускает копирование навыков в этом запуске, но продолжает выбор плагинов. Если установленные из исходного расположения курируемые плагины Codex доступны для миграции и флаг `--plugin` не указан, затем предлагается выбрать по имени плагины Codex для встроенной активации. Элементы плагинов изначально отмечены, если только в целевой конфигурации Plugin Codex в OpenClaw уже нет соответствующего Plugin. Существующие целевые плагины изначально не отмечены и показывают подсказку о конфликте, например `conflict: plugin exists`; выберите `Toggle all off`, чтобы не переносить встроенные плагины Codex в этом запуске, или `Skip for now`, чтобы остановиться перед применением.

Для автоматизированных запусков или точного выбора явно укажите один или несколько навыков либо плагинов:

```bash
openclaw migrate codex --dry-run --skill gog-vault77-google-workspace
openclaw migrate apply codex --yes --skill gog-vault77-google-workspace
openclaw migrate codex --dry-run --plugin google-calendar
openclaw migrate apply codex --yes --plugin google-calendar
```

### Что импортирует Codex

- Каталоги навыков Codex CLI в `$CODEX_HOME/skills`, кроме системного кеша Codex `.system`.
- Личные AgentSkills из `$HOME/.agents/skills`, копируемые в рабочую область текущего агента OpenClaw для владения на уровне агента.
- Установленные из исходного расположения плагины Codex из `openai-curated`, обнаруженные через `plugin/list` сервера приложений Codex. При планировании для каждого включённого установленного Plugin читаются данные через `plugin/read`.

Для миграции плагинов, использующих приложения, действуют дополнительные проверки:

- Для плагинов, использующих приложения, учётная запись исходного сервера приложений Codex должна быть учётной записью с подпиской ChatGPT. Ответы для учётных записей, не относящихся к ChatGPT, и ответы без данных учётной записи пропускаются с причиной `codex_subscription_required`.
- По умолчанию миграция не вызывает `app/list` исходной системы, поэтому плагины, использующие приложения и прошедшие проверку учётной записи, планируются без проверки доступности приложений в исходной системе, а при транспортных ошибках получения данных учётной записи они пропускаются с причиной `codex_account_unavailable`.
- Укажите `--verify-plugin-apps`, чтобы принудительно получить новый снимок `app/list` исходной системы и перед планированием встроенной активации потребовать, чтобы каждое принадлежащее пользователю приложение существовало, было включено и доступно. В этом режиме при транспортных ошибках получения данных учётной записи выполняется проверка по инвентаризации приложений исходной системы. Снимок хранится в памяти только для текущего процесса; он никогда не записывается в результаты миграции или целевую конфигурацию.

Отключённые плагины, недоступные для чтения сведения о плагинах, исходные учётные записи без требуемой подписки, а также — при установленном `--verify-plugin-apps` — отсутствующие, отключённые или недоступные приложения становятся вручную пропущенными элементами с типизированными причинами вместо записей в целевой конфигурации. Для каждого выбранного подходящего Plugin команда применения вызывает `plugin/install` сервера приложений, даже если целевой сервер приложений уже сообщает, что этот Plugin установлен и включён. Перенесённые плагины Codex доступны только в сеансах, где выбрана встроенная среда выполнения Codex; они недоступны для запусков провайдера OpenClaw, привязок диалогов ACP и других сред выполнения.

### Состояние Codex для ручной проверки

Файл Codex `config.toml`, встроенные обработчики `hooks/hooks.json`, некурируемые каталоги, кешированные пакеты плагинов, которые не являются курируемыми плагинами, установленными из исходного расположения, а также установленные из исходного расположения плагины, не прошедшие проверку подписки исходной системы, не активируются автоматически. Если указан `--verify-plugin-apps`, также пропускаются плагины, не прошедшие проверку по инвентаризации приложений исходной системы. Все эти элементы копируются или перечисляются в отчёте о миграции для ручной проверки.

Для переносимых курируемых плагинов, установленных из исходного расположения, команда применения записывает:

- `plugins.entries.codex.enabled: true`
- `plugins.entries.codex.config.codexPlugins.enabled: true`
- `plugins.entries.codex.config.codexPlugins.allow_destructive_actions: true`
- по одной явной записи Plugin с `marketplaceName: "openai-curated"` и `pluginName` для каждого выбранного Plugin

Миграция никогда не записывает `plugins["*"]` и никогда не сохраняет пути к локальному кешу каталога.

Пропущенные плагины не записываются в целевую конфигурацию. Ошибки подписки на стороне источника указываются для элементов, требующих ручной обработки, с типизированными причинами: `codex_subscription_required`, `codex_account_unavailable`, `plugin_disabled` или `plugin_read_unavailable`. При использовании `--verify-plugin-apps` ошибки инвентаризации приложений источника также могут отображаться как `app_inaccessible`, `app_disabled`, `app_missing` или `app_inventory_unavailable`. Установки на целевой стороне, требующие аутентификации, указываются для соответствующего элемента плагина со значениями `status: "skipped"`, `reason: "auth_required"` и обезличенными идентификаторами приложений; соответствующие явные записи конфигурации записываются отключёнными до повторной авторизации и включения. Другие ошибки установки возвращаются как результаты `error`, относящиеся к конкретным элементам.

Если инвентаризация плагинов сервера приложений Codex недоступна во время планирования, миграция вместо сбоя всего процесса использует резервные информационные элементы из кэшированного пакета.

## Провайдер Hermes

Встроенный провайдер Hermes по умолчанию обнаруживает состояние в `~/.hermes`. Если Hermes находится в другом месте, используйте `--from <path>`.

### Что импортирует Hermes

- Конфигурацию модели по умолчанию из `config.yaml`.
- Настроенные провайдеры моделей и пользовательские конечные точки, совместимые с OpenAI, из `providers` и `custom_providers`.
- Определения серверов MCP из `mcp_servers` или `mcp.servers`.
- `SOUL.md` и `AGENTS.md` в рабочее пространство агента OpenClaw.
- Содержимое `memories/MEMORY.md` и `memories/USER.md`, добавляемое в конец файлов памяти рабочего пространства.
- Значения конфигурации памяти по умолчанию для файловой памяти OpenClaw, а также архивные элементы или элементы для ручной проверки внешних провайдеров памяти, таких как Honcho.
- Skills, содержащие файл `SKILL.md` в каталоге `skills/<name>/`.
- Значения конфигурации отдельных Skills из `skills.config`.
- Учётные данные OpenAI OAuth из файла OpenCode `auth.json`, если при интерактивной миграции разрешён перенос учётных данных или задан параметр `--include-secrets`. Записи OAuth в Hermes `auth.json` являются устаревшим состоянием и помечаются для ручной повторной авторизации OpenAI или исправления с помощью doctor.
- Поддерживаемые ключи API и токены из Hermes `.env` и OpenCode `auth.json`, если при интерактивной миграции разрешён перенос учётных данных или задан параметр `--include-secrets`.

### Поддерживаемые ключи `.env`

### Состояние только для архивирования

Состояние Hermes, которое OpenClaw не может безопасно интерпретировать, копируется в отчёт о миграции для ручной проверки, но не загружается в действующую конфигурацию или учётные данные OpenClaw. Это позволяет сохранить непрозрачное или небезопасное состояние, не создавая ложного впечатления, что OpenClaw может автоматически выполнять его или считать доверенным: `plugins/`, `sessions/`, `logs/`, `cron/`, `mcp-tokens/`, `state.db`.

## Контракт Plugin

Источники миграции представляют собой плагины. Plugin объявляет идентификаторы своих провайдеров в `openclaw.plugin.json`:

```json
{
  "contracts": {
    "migrationProviders": ["hermes"]
  }
}
```

Во время выполнения Plugin вызывает `api.registerMigrationProvider(...)`. Провайдер реализует `detect`, `plan` и `apply`. Ядро отвечает за оркестрацию CLI, политику резервного копирования, запросы пользователю, вывод JSON и предварительную проверку конфликтов. Ядро передаёт проверенный план в `apply(ctx, plan)`, а провайдеры могут повторно сформировать план только в том случае, если этот аргумент отсутствует для обеспечения совместимости.

Плагины провайдеров могут использовать `openclaw/plugin-sdk/migration` для создания элементов и подсчёта итогов, а также `openclaw/plugin-sdk/migration-runtime` для копирования файлов с учётом конфликтов, копирования в отчёт только архивируемых данных, кэшированных обёрток среды выполнения конфигурации и отчётов о миграции.

## Интеграция с первоначальной настройкой

При первоначальной настройке может быть предложена миграция, если провайдер обнаруживает известный источник. И `openclaw onboard --flow import`, и `openclaw setup --wizard --import-from hermes` используют один и тот же провайдер миграции Plugin и перед применением по-прежнему показывают предварительный просмотр.

<Note>
Для импорта при первоначальной настройке требуется новая установка OpenClaw. Если у вас уже есть локальное состояние, сначала сбросьте конфигурацию, учётные данные, сеансы и рабочее пространство. Импорт с резервным копированием и перезаписью или слиянием для существующих установок доступен только при включении соответствующей функции.
</Note>
