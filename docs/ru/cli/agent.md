# `openclaw agent`

Выполняет один ход агента через Gateway. Если запрос к Gateway завершается ошибкой, используется встроенный агент; передайте `--local`, чтобы сразу принудительно запустить встроенное выполнение.

Передайте хотя бы один селектор сеанса: `--to`, `--session-key`, `--session-id` или `--agent`.

См. также: [Инструмент отправки агенту](/ru/tools/agent-send)

## Параметры

- `-m, --message <text>`: текст сообщения
- `--message-file <path>`: прочитать текст сообщения из файла в кодировке UTF-8
- `-t, --to <dest>`: получатель, используемый для формирования ключа сеанса
- `--session-key <key>`: явный ключ сеанса для маршрутизации
- `--session-id <id>`: явный идентификатор сеанса
- `--agent <id>`: идентификатор агента; переопределяет привязки маршрутизации
- `--model <id>`: переопределение модели для этого запуска (`provider/model` или идентификатор модели)
- `--thinking <level>`: уровень рассуждений агента (`off`, `minimal`, `low`, `medium`, `high`, а также поддерживаемые провайдером пользовательские уровни, например `xhigh`, `adaptive` или `max`)
- `--verbose <on|off>`: сохранить уровень подробности для сеанса
- `--channel <channel>`: канал доставки; не указывайте, чтобы использовать основной канал сеанса
- `--reply-to <target>`: переопределение адресата доставки
- `--reply-channel <channel>`: переопределение канала доставки
- `--reply-account <id>`: переопределение учётной записи доставки
- `--local`: запустить встроенного агента напрямую (после предварительной загрузки реестра плагинов)
- `--deliver`: отправить ответ обратно в выбранный канал или выбранному адресату
- `--timeout <seconds>`: переопределить время ожидания агента (по умолчанию 600 или значение `agents.defaults.timeoutSeconds`); `0` отключает ограничение времени
- `--json`: вывести JSON

## Примеры

```bash
openclaw agent --to +15555550123 --message "status update" --deliver
openclaw agent --agent ops --message "Summarize logs"
openclaw agent --agent ops --message-file ./task.md
openclaw agent --agent ops --model openai/gpt-5.4 --message "Summarize logs"
openclaw agent --session-key agent:ops:incident-42 --message "Summarize status"
openclaw agent --agent ops --session-key incident-42 --message "Summarize status"
openclaw agent --session-id 1234 --message "Summarize inbox" --thinking medium
openclaw agent --to +15555550123 --message "Trace logs" --verbose on --json
openclaw agent --agent ops --message "Generate report" --deliver --reply-channel slack --reply-to "#reports"
openclaw agent --agent ops --message "Run locally" --local
```

## Примечания

- Передайте ровно один из параметров: `--message` или `--message-file`. Параметр `--message-file` удаляет начальную метку BOM UTF-8 и сохраняет многострочное содержимое; файлы с недопустимой кодировкой UTF-8 отклоняются.
- Команды с косой чертой (например, `/compact`) нельзя запускать через `--message`. CLI отклоняет их и предлагает вместо этого использовать специализированную команду (`openclaw sessions compact <key>` для Compaction).
- Запуски с `--local` и встроенным резервным режимом являются одноразовыми: входящие в комплект ресурсы локальной обратной связи MCP и прогретые сеансы Claude stdio, открытые для запуска, завершаются после ответа, поэтому запуски из сценариев не оставляют работающих локальных дочерних процессов. При запусках через Gateway принадлежащие Gateway ресурсы локальной обратной связи MCP остаются в работающем процессе Gateway.
- При совместном использовании `--agent`, `--channel` и `--to` маршрутизация сеанса следует каноническому получателю канала и `session.dmScope`. Каналы со стабильным идентификатором получателя, предназначенным только для исходящих сообщений, используют принадлежащий провайдеру сеанс, изолированный от основного сеанса агента. Параметры `--reply-channel` и `--reply-account` влияют только на доставку.
- `--session-key` выбирает явный ключ сеанса. Ключи с префиксом агента должны иметь формат `agent:<agent-id>:<session-key>`, а при одновременном указании `--agent` идентификатор агента должен совпадать с указанным в ключе. Простые ключи, не являющиеся сигнальными значениями, относятся к агенту из `--agent`, если он указан, или к настроенному агенту по умолчанию в противном случае; например, `--agent ops --session-key incident-42` направляет запрос в `agent:ops:incident-42`. Литеральные ключи `global` и `unknown` остаются без области действия, только если параметр `--agent` не указан.
- При использовании `--json` стандартный вывод предназначен для ответа JSON; диагностические сообщения Gateway, плагинов и встроенного резервного режима выводятся в стандартный поток ошибок, чтобы сценарии могли напрямую разбирать стандартный вывод.
- JSON встроенного резервного режима содержит `meta.transport: "embedded"` и `meta.fallbackFrom: "gateway"`, чтобы сценарии могли обнаружить резервный запуск.
- Если Gateway принимает запуск, но время ожидания CLI истекает до получения окончательного ответа, встроенный резервный режим использует новый идентификатор сеанса и запуска `gateway-fallback-*`, а также сообщает `meta.fallbackReason: "gateway_timeout"` и поля резервного сеанса вместо того, чтобы конкурировать за принадлежащую Gateway стенограмму или незаметно заменять исходный сеанс.
- `SIGTERM`/`SIGINT` прерывают ожидающий запрос через Gateway; если Gateway уже принял запуск, CLI перед завершением также отправляет `chat.abort` для идентификатора этого запуска. Запуски с `--local` и встроенным резервным режимом получают тот же сигнал, но не отправляют `chat.abort`. Если для внутреннего ключа дедупликации запусков уже существует активный запуск этого сеанса, ответ сообщает `status: "in_flight"`, а CLI без JSON выводит диагностическое сообщение в стандартный поток ошибок вместо пустого ответа. Для внешних обёрток cron/systemd предусмотрите принудительное завершение, например `timeout -k 60 600 openclaw agent ...`, чтобы управляющий процесс мог завершить процесс, если при остановке не удаётся дождаться завершения обработки.
- Когда эта команда запускает повторное создание `models.json`, управляемые через SecretRef учётные данные провайдера сохраняются в виде маркеров, не содержащих секретов (например, имён переменных среды, `secretref-env:ENV_VAR_NAME` или `secretref-managed`), и никогда не сохраняются как разрешённый открытый текст секрета. Маркеры записываются из активного снимка исходной конфигурации, а не из разрешённых значений секретов среды выполнения.

## Состояние доставки JSON

При использовании `--json --deliver` ответ CLI в формате JSON содержит поле верхнего уровня `deliveryStatus`, чтобы сценарии могли различать успешные, подавленные, частично неудачные и неудачные отправки:

```json
{
  "payloads": [{ "text": "Report ready", "mediaUrl": null }],
  "meta": { "durationMs": 1200 },
  "deliveryStatus": {
    "requested": true,
    "attempted": true,
    "status": "sent",
    "succeeded": true,
    "resultCount": 1
  }
}
```

Ответы CLI через Gateway также сохраняют исходную структуру результата Gateway в `result.deliveryStatus`.

Возможные значения `deliveryStatus.status`:

| Состояние        | Значение                                                                                                                                                        |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `sent`           | Доставка завершена.                                                                                                                                             |
| `suppressed`     | Доставка намеренно не выполнена (например, её отменил обработчик отправки сообщений или видимый результат отсутствовал). Окончательное состояние, без повтора.   |
| `partial_failed` | Хотя бы одна полезная нагрузка была отправлена до сбоя при отправке последующей полезной нагрузки.                                                               |
| `failed`         | Ни одна надёжная отправка не завершилась или завершилась ошибкой предварительная проверка доставки.                                                              |

Общие поля:

- `requested`: всегда `true`, если объект присутствует.
- `attempted`: `true` после запуска пути надёжной отправки; `false` при ошибках предварительной проверки или отсутствии видимых полезных нагрузок.
- `succeeded`: `true`, `false` или `"partial"`; значение `"partial"` используется вместе с `status: "partial_failed"`.
- `reason`: причина в нижнем регистре с разделением слов символами подчёркивания, полученная при надёжной доставке или предварительной проверке. Известные значения включают `cancelled_by_message_sending_hook`, `no_visible_payload`, `no_visible_result`, `channel_resolved_to_internal`, `unknown_channel`, `invalid_delivery_target` и `no_delivery_target`; при сбое надёжной отправки также может указываться этап, на котором произошёл сбой. Рассматривайте неизвестные значения как непрозрачные, поскольку их набор может расширяться.
- `resultCount`: количество результатов отправки в канал, если доступно.
- `sentBeforeError`: `true`, если при частичном сбое до возникновения ошибки была отправлена хотя бы одна полезная нагрузка.
- `error`: `true` для неудачных или частично неудачных отправок.
- `errorMessage`: присутствует, только если удалось получить сообщение о внутренней ошибке доставки. Ошибки предварительной проверки содержат `error`/`reason`, но не `errorMessage`.
- `payloadOutcomes`: необязательные результаты для каждой полезной нагрузки с полями `index`, `status`, `reason`, `resultCount`, `error`, `stage`, `sentBeforeError` или метаданными обработчика, если они доступны.
