OpenClaw сейчас предоставляет три пользовательских канала обновлений:

- stable: существующий продвигаемый канал релизов, который по-прежнему разрешается через
  npm `latest`, пока не будет завершен отдельный этап для CLI/канала
- beta: предварительные теги релизов, публикуемые в npm `beta`
- dev: движущаяся вершина `main`

Отдельно операторы релизов могут публиковать core-пакет за последний завершенный месяц
в npm `extended-stable`, начиная с патча `33`. Обычная финальная линия текущего месяца
продолжает использовать npm `latest`; это операторское разделение публикаций
само по себе не меняет разрешение каналов обновления CLI.

## Именование версий

- Версия ежемесячного npm-релиза extended-stable: `YYYY.M.PATCH`, где `PATCH >= 33`
  - Git-тег: `vYYYY.M.PATCH`
- Версия ежедневного/обычного финального релиза: `YYYY.M.PATCH`, где `PATCH < 33`
  - Git-тег: `vYYYY.M.PATCH`
- Версия обычного корректирующего fallback-релиза: `YYYY.M.PATCH-N`
  - Git-тег: `vYYYY.M.PATCH-N`
- Версия предварительного beta-релиза: `YYYY.M.PATCH-beta.N`
  - Git-тег: `vYYYY.M.PATCH-beta.N`
- Не дополняйте месяц или патч нулями
- Начиная с обновления процесса релизов за июнь 2026 года, третий компонент — это
  последовательный номер ежемесячной релизной ветки, а не календарный день. Релизы stable и beta
  определяют текущую ветку; теги только для alpha не занимают и не
  продвигают номер патча beta/stable. Теги и npm-версии до этого обновления сохраняют
  свои существующие имена и остаются действительными; автоматизация релизов продолжает
  сравнивать их по году, месяцу, патчу, каналу и номеру предварительного релиза или
  корректировки.
- Alpha/nightly-сборки используют следующую невыпущенную патч-ветку и увеличивают только
  `alpha.N` для повторных сборок. Как только у этого патча появляется beta, новые alpha-сборки
  переходят к следующему патчу. Игнорируйте устаревшие теги только для alpha с более высокими
  номерами патчей при выборе ветки beta или stable.
- npm-версии неизменяемы. Если beta-тег уже был опубликован, не
  удаляйте, не публикуйте повторно и не используйте его повторно; создайте следующий номер beta
  или следующий ежемесячный патч. Поскольку `2026.6.5-beta.1` уже был опубликован во время
  перехода, релизные ветки июня 2026 года должны использовать патч `5` или выше. Не
  публикуйте новые ветки stable или beta за июнь 2026 года как `2026.6.2`, `2026.6.3` или
  `2026.6.4`.
- После обычного финального релиза `2026.6.5` следующая новая beta-ветка —
  `2026.6.6-beta.1`, даже
  если уже существуют автоматизированные теги только для alpha с более высокими номерами патчей.
- `latest` продолжает следовать текущей обычной/ежедневной линии npm
- `beta` означает текущую цель установки beta
- `extended-stable` означает поддерживаемый npm-пакет за предыдущий месяц, начиная с патча
  `33`; патч `34` и последующие являются maintenance-релизами в этой ежемесячной линии
- Выделенный ежемесячный путь extended-stable публикует только core npm-пакет. Он
  не публикует plugins, артефакты macOS или Windows, релиз GitHub,
  dist-tags приватного репозитория, Docker-образы, мобильные артефакты или загрузки
  с сайта.

## Частота релизов

- Релизы сначала переходят в beta
- Stable следует только после проверки последней beta
- Maintainers обычно создают релизы из ветки `release/YYYY.M.PATCH`, созданной
  из текущего `main`, чтобы проверка релиза и исправления не блокировали новую
  разработку в `main`
- Если beta-тег был отправлен или опубликован и требует исправления, maintainers создают
  следующий тег `-beta.N`, а не удаляют или пересоздают старый beta-тег
- Подробная процедура релиза, утверждения, учетные данные и заметки по восстановлению
  доступны только maintainers

## Ежемесячная npm-only публикация extended-stable

Это выделенное исключение из обычной процедуры релиза ниже. Для
завершенного месяца `YYYY.M` создайте `extended-stable/YYYY.M.33`; публикуйте `vYYYY.M.33` и
последующие maintenance-патчи из той же ветки. Релизный тег, вершина ветки,
checkout, версия пакета, npm preflight и запуск полной проверки релиза должны
указывать на один и тот же коммит. Защищенный `main` уже должен содержать финальную версию
строго более позднего календарного месяца с патчем ниже `33`; maintenance-патчи остаются
допустимыми после того, как `main` продвинется более чем на один месяц.

Запустите npm preflight и полную проверку релиза из точной ветки extended-stable,
затем сохраните оба ID запусков:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=true \
  -f npm_dist_tag=extended-stable

gh workflow run full-release-validation.yml \
  --ref extended-stable/YYYY.M.33 \
  -f ref=extended-stable/YYYY.M.33 \
  -f release_profile=stable
```

`release_profile=stable` — это существующий профиль глубины проверки; он
отделен от npm dist-tag `extended-stable` и намеренно остается без изменений.

После успешного завершения обоих запусков и готовности npm release environment продвиньте
точный preflight tarball. Патч `P` должен быть `33` или больше:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=false \
  -f npm_dist_tag=extended-stable \
  -f preflight_run_id=<npm-preflight-run-id> \
  -f full_release_validation_run_id=<full-validation-run-id>
```

Для форка или непроизводственной репетиции, которая намеренно не может удовлетворить
месячную политику `.33` или политику защищенного `main`, добавьте
`-f bypass_extended_stable_guard=true` к обоим dispatch для npm preflight и publish. Значение
по умолчанию — `false`. Обход принимается только с `npm_dist_tag=extended-stable` и
записывается в сводку workflow. Он не обходит канонический
workflow ref `extended-stable/YYYY.M.33`, равенство вершины ветки/тега/checkout, синтаксис final-тега,
равенство версии пакета/тега, идентичность указанного запуска и manifest,
происхождение tarball, approval environment, readback registry или доказательство
исправления selector.

Publish workflow проверяет идентичность указанных запусков, подготовленный
digest tarball и оба selector npm registry. Независимо подтвердите
результат после успешного завершения workflow:

```bash
npm view openclaw@YYYY.M.P version --userconfig "$(mktemp)"
npm view openclaw@extended-stable version --userconfig "$(mktemp)"
```

Обе команды должны вернуть `YYYY.M.P`. Если публикация успешна, но selector
readback завершается с ошибкой, не публикуйте неизменяемую версию пакета повторно. Используйте единственную
repair-команду `npm dist-tag add openclaw@YYYY.M.P extended-stable`, напечатанную в
always-run summary неудачного workflow, затем повторите оба независимых
readback. Откат к предыдущему selector — отдельное решение оператора, а не
путь исправления readback.

Обычный checklist ниже продолжает отвечать за beta, `latest`, релиз GitHub,
plugins, macOS, Windows и публикацию на других платформах. Не выполняйте эти шаги
для этого npm-only пути extended-stable.

## Обычный checklist оператора релизов

Этот checklist отражает публичную форму release flow. Приватные учетные данные,
подписание, notarization, восстановление dist-tag и сведения об аварийном откате остаются в
runbook релизов, доступном только maintainers.

1. Начните с текущей ветки `main`: получите последние изменения, подтвердите, что целевой коммит отправлен,
   и убедитесь, что текущий CI для `main` достаточно зеленый, чтобы создавать от нее ветку.
2. Сгенерируйте верхний раздел `CHANGELOG.md` из объединенных PR и всех прямых
   коммитов с момента последнего достижимого тега релиза. Оставляйте записи ориентированными на пользователя,
   удаляйте дублирующиеся записи PR/прямых коммитов, зафиксируйте переписанный файл, отправьте его,
   и еще раз выполните rebase/pull перед созданием ветки.
3. Проверьте записи совместимости релизов в
   `src/plugins/compat/registry.ts` и
   `src/commands/doctor/shared/deprecation-compat.ts`. Удаляйте истекшую
   совместимость только когда путь обновления остается покрытым, или зафиксируйте, почему она
   намеренно сохраняется.
4. Создайте `release/YYYY.M.PATCH` из текущей `main`; не выполняйте обычную релизную работу
   напрямую в `main`.
5. Поднимите версию во всех обязательных местах для целевого тега, затем выполните
   `pnpm release:prep`. Эта команда обновляет версии plugins, инвентарь plugins, схему конфигурации,
   метаданные конфигурации встроенных каналов, базовую линию документации конфигурации, экспорты plugin SDK
   и базовую линию API plugin SDK в правильном порядке. Зафиксируйте любой сгенерированный
   дрейф перед тегированием. Затем выполните локальный детерминированный предварительный прогон:
   `pnpm check:test-types`, `pnpm check:architecture`,
   `pnpm build && pnpm ui:build` и `pnpm release:check`.
6. Запустите `OpenClaw NPM Release` с `preflight_only=true`. Пока тег не существует,
   для предварительной проверки без публикации допускается полный 40-символьный SHA релизной ветки.
   Предварительная проверка генерирует доказательства релиза зависимостей для
   точного извлеченного графа зависимостей и сохраняет их в артефакте предварительной проверки npm.
   Сохраните успешный `preflight_run_id`.
7. Запустите все предрелизные тесты через `Full Release Validation` для
   релизной ветки, тега или полного SHA коммита. Это единственная ручная точка входа
   для четырех больших релизных тестовых блоков: Vitest, Docker, QA Lab и Package.
8. Если валидация завершается с ошибкой, исправьте проблему в релизной ветке и повторно запустите самый маленький отказавший
   файл, дорожку, задание workflow, профиль пакета, провайдера или allowlist моделей, который
   доказывает исправление. Повторно запускайте полный зонтичный workflow только когда измененная поверхность делает
   прежние доказательства устаревшими.
9. Для тегированного бета-кандидата выполните
   `pnpm release:candidate -- --tag vYYYY.M.PATCH-beta.N` из соответствующей
   ветки `release/YYYY.M.PATCH`. Для стабильного релиза также передайте обязательный Windows source
   release:
   `pnpm release:candidate -- --tag vYYYY.M.PATCH --windows-node-tag vX.Y.Z`.
   Вспомогательная команда запускает локальные проверки сгенерированного релиза, dispatch или проверку
   доказательств полной релизной валидации и предварительной проверки npm, запускает Parallels
   fresh/update proof против точно подготовленного tarball плюс доказательство пакета Telegram,
   записывает планы npm и ClawHub для plugins и печатает точную команду
   `OpenClaw Release Publish` только после того, как пакет доказательств становится зеленым.
   `OpenClaw Release Publish` параллельно dispatch выбранных или всех публикуемых пакетов plugins
   в npm и того же набора в ClawHub, а затем продвигает
   подготовленный артефакт предварительной проверки OpenClaw npm с соответствующим dist-tag сразу после
   успешной публикации plugin npm.
   После успешного завершения дочернего задания публикации OpenClaw npm оно создает или обновляет
   соответствующую страницу GitHub release/prerelease из полного соответствующего
   раздела `CHANGELOG.md`. Стабильные релизы, опубликованные в npm `latest`, становятся
   последним релизом GitHub; стабильные maintenance-релизы, оставленные в npm `beta`,
   создаются с GitHub `latest=false`. Workflow также загружает доказательства
   зависимостей из предварительной проверки, манифест полной валидации и доказательства проверки
   реестра после публикации в релиз GitHub для реагирования на пострелизные инциденты.
   Workflow публикации сразу печатает ID дочерних запусков, автоматически подтверждает
   release environment gates, которые токен workflow имеет право подтверждать, суммирует
   отказавшие дочерние задания с хвостами логов, закрывает релиз GitHub и доказательства
   зависимостей сразу после успешной публикации OpenClaw npm, ждет ClawHub всякий раз,
   когда публикуется OpenClaw npm, затем запускает `pnpm release:verify-beta` и
   загружает постпубликационные доказательства для релиза GitHub, пакета npm, выбранных
   plugin npm packages, выбранных пакетов ClawHub, ID запусков дочерних workflow и
   необязательного ID запуска NPM Telegram. Путь ClawHub повторяет попытки при временных сбоях
   установки зависимостей CLI, публикует plugins с успешным preview даже если одна
   preview-ячейка дает сбой, и завершается проверкой реестра для каждой ожидаемой
   версии plugin, чтобы частичные публикации оставались видимыми и доступными для повторной попытки. Затем выполните post-publish
   package acceptance против опубликованного
   пакета `openclaw@YYYY.M.PATCH-beta.N` или
   `openclaw@beta`. Если отправленный или опубликованный prerelease требует исправления,
   создайте следующий соответствующий номер prerelease; не удаляйте и не переписывайте старый
   prerelease.
10. Для стабильного релиза продолжайте только после того, как проверенная beta или release candidate имеет
    обязательные доказательства валидации. Публикация стабильного npm также проходит через
    `OpenClaw Release Publish`, повторно используя успешный артефакт предварительной проверки через
    `preflight_run_id`; готовность стабильного macOS-релиза также требует
    упакованных `.zip`, `.dmg`, `.dSYM.zip` и обновленного `appcast.xml` в `main`.
    Workflow публикации macOS автоматически публикует подписанный appcast в публичную `main`
    после проверки релизных assets; если защита ветки блокирует
    прямую отправку, он открывает или обновляет PR для appcast. Готовность Stable Windows Hub
    требует подписанные assets `OpenClawCompanion-Setup-x64.exe`,
    `OpenClawCompanion-Setup-arm64.exe` и
    `OpenClawCompanion-SHA256SUMS.txt` в релизе OpenClaw GitHub.
    Передайте точный тег релиза подписанного `openclaw/openclaw-windows-node` как
    `windows_node_tag` и его одобренную кандидатом карту digest установщика как
    `windows_node_installer_digests`; `OpenClaw Release Publish` сохраняет
    черновик релиза, dispatch `Windows Node Release` и проверяет все три
    assets перед публикацией.
11. После публикации запустите verifier после публикации npm, необязательный отдельный
    опубликованный-npm Telegram E2E, когда нужны постпубликационные доказательства канала,
    продвижение dist-tag при необходимости, проверьте сгенерированную страницу релиза GitHub,
    выполните шаги объявления релиза, затем завершите [закрытие стабильного релиза в main](#stable-main-closeout), прежде чем считать стабильный релиз завершенным.

## Закрытие стабильного релиза в main

Публикация стабильного релиза не завершена, пока `main` не содержит фактическое отправленное
состояние релиза.

1. Начните со свежей последней `main`. Проведите аудит `release/YYYY.M.PATCH` относительно нее и
   перенесите вперед реальные исправления, отсутствующие в `main`. Не сливайте вслепую
   только релизные адаптеры совместимости, тестов или валидации в более новую `main`.
2. Установите `main` на отправленную стабильную версию, а не на спекулятивный следующий поезд релиза. Запустите
   `pnpm release:prep` после изменения корневой версии, затем
   `pnpm deps:shrinkwrap:generate`.
3. Сделайте раздел `## YYYY.M.PATCH` в `CHANGELOG.md` на `main` точно соответствующим
   тегированной релизной ветке. Включите стабильное обновление `appcast.xml`, когда mac
   release его опубликовал.
4. Не добавляйте `YYYY.M.PATCH+1`, бета-версию или пустой будущий раздел changelog
   в `main`, пока оператор явно не начнет этот релизный поезд.
5. Выполните `pnpm release:generated:check`, `pnpm deps:shrinkwrap:check` и
   `OPENCLAW_TESTBOX=1 pnpm check:changed`. Отправьте изменения, затем убедитесь, что `origin/main`
   содержит отправленную версию и changelog, прежде чем считать стабильный релиз
   завершенным.
6. Поддерживайте переменные репозитория `RELEASE_ROLLBACK_DRILL_ID` и
   `RELEASE_ROLLBACK_DRILL_DATE` актуальными после каждой приватной проверки отката.
   `OpenClaw Stable Main Closeout` стартует от push в `main`, который содержит
   отправленную версию, changelog и appcast после публикации стабильного релиза. Он читает
   неизменяемые постпубликационные доказательства, чтобы связать отправленный тег с его запусками Full Release
   Validation и Publish, затем проверяет состояние stable main, релиз,
   обязательную стабильную выдержку и блокирующие performance-доказательства. Он прикрепляет
   неизменяемый манифест закрытия и checksum к релизу GitHub. Автоматический
   триггер push пропускает legacy-релизы, которые предшествуют неизменяемым постпубликационным
   доказательствам; он никогда не считает такой пропуск завершенным закрытием. Полное
   закрытие требует оба assets и соответствующий checksum. Частичный манифест
   воспроизводит записанные в нем SHA `main` и проверку отката, чтобы заново сгенерировать идентичные
   байты, затем прикрепляет отсутствующий checksum; недопустимая пара или checksum
   без манифеста остаются блокирующими. Запуск по push-триггеру без repository variables
   для проверки отката пропускается без завершения closeout; отсутствующая или
   более чем 90-дневной давности запись проверки все еще блокирует ручной evidence-backed
   closeout. Приватные команды восстановления остаются в runbook только для maintainers.
   Используйте ручной dispatch только для исправления или повтора evidence-backed stable closeout.
   Legacy fallback correction tag может повторно использовать доказательства базового пакета только когда
   correction tag разрешается в тот же исходный коммит, что и базовый стабильный тег.
   Correction с другим исходным кодом должна опубликовать и проверить собственные
   доказательства пакета.

## Предварительная проверка релиза

- Запустите `pnpm check:test-types` перед предварительной проверкой релиза, чтобы тестовый TypeScript оставался
  покрытым вне более быстрого локального этапа `pnpm check`
- Запустите `pnpm check:architecture` перед предварительной проверкой релиза, чтобы более широкие проверки циклов
  импортов и архитектурных границ проходили вне более быстрого локального этапа
- Запустите `pnpm build && pnpm ui:build` перед `pnpm release:check`, чтобы ожидаемые
  артефакты релиза `dist/*` и бандл Control UI существовали для этапа
  проверки пакета
- Запустите `pnpm release:prep` после повышения версии в корне и перед созданием тега. Он
  запускает все детерминированные генераторы релиза, которые обычно расходятся после
  изменения версии/конфигурации/API: версии Plugin, инвентарь Plugin, базовую схему
  конфигурации, метаданные конфигурации встроенного канала, базовую линию документации
  конфигурации, экспорты SDK Plugin и базовую линию API SDK Plugin. `pnpm release:check` повторно запускает эти
  проверки в режиме проверки и сообщает обо всех найденных сбоях сгенерированного расхождения за один
  проход перед запуском проверок релиза пакета.
- Синхронизация версий Plugin по умолчанию обновляет версии пакетов официальных Plugin и существующие
  минимальные значения `openclaw.compat.pluginApi` до версии релиза OpenClaw.
  Рассматривайте это поле как минимальный уровень API SDK/runtime Plugin, а не просто как копию
  версии пакета: для релизов только Plugin, которые намеренно остаются
  совместимыми с более старыми хостами OpenClaw, сохраняйте минимальный уровень на уровне самого старого поддерживаемого
  API хоста и документируйте этот выбор в доказательствах релиза Plugin.
- Запустите ручной рабочий процесс `Full Release Validation` перед утверждением релиза, чтобы
  запустить все предрелизные тестовые среды из одной точки входа. Он принимает ветку,
  тег или полный SHA коммита, запускает ручной `CI` и запускает
  `OpenClaw Release Checks` для install smoke, package acceptance, межплатформенных
  проверок пакета, паритета QA Lab, Matrix и Telegram. Стабильные и полные
  запуски всегда включают исчерпывающие live/E2E и Docker soak для пути релиза;
  `run_release_soak=true` сохранен для явного beta soak. Package
  Acceptance предоставляет канонический package Telegram E2E во время проверки кандидата,
  избегая второго параллельного live poller.
  Укажите `release_package_spec` после публикации beta, чтобы повторно использовать поставленный
  npm-пакет во всех release checks, Package Acceptance и package Telegram
  E2E без пересборки tarball релиза. Укажите
  `npm_telegram_package_spec` только когда Telegram должен использовать другой
  опубликованный пакет, отличный от остальной проверки релиза. Укажите
  `package_acceptance_package_spec`, когда Package Acceptance должен использовать
  другой опубликованный пакет, отличный от спецификации пакета релиза. Укажите
  `evidence_package_spec`, когда отчет с доказательствами релиза должен доказать, что
  проверка соответствует опубликованному npm-пакету, без принудительного Telegram E2E.
  Пример:
  `gh workflow run full-release-validation.yml --ref main -f ref=release/YYYY.M.PATCH`
- Запустите ручной рабочий процесс `Package Acceptance`, когда нужны побочные доказательства
  для кандидата пакета, пока работа над релизом продолжается. Используйте `source=npm` для
  `openclaw@beta`, `openclaw@latest` или точной версии релиза; `source=ref`
  для упаковки доверенной ветки/тега/SHA `package_ref` с текущим
  harness `workflow_ref`; `source=url` для публичного HTTPS tarball с
  обязательным SHA-256 и строгой политикой публичного URL; `source=trusted-url` для
  именованной политики доверенного источника с обязательными `trusted_source_id` и SHA-256; или
  `source=artifact` для tarball, загруженного другим запуском GitHub Actions. Рабочий процесс
  разрешает кандидата в
  `package-under-test`, повторно использует планировщик Docker E2E релиза для этого
  tarball и может запускать Telegram QA для того же tarball с
  `telegram_mode=mock-openai` или `telegram_mode=live-frontier`. Когда
  выбранные Docker-линии включают `published-upgrade-survivor`, артефакт пакета
  является кандидатом, а `published_upgrade_survivor_baseline` выбирает
  опубликованную базовую версию. `update-restart-auth` использует пакет-кандидат как
  установленный CLI и как package-under-test, чтобы проверить управляемый путь перезапуска
  команды обновления кандидата.
  Пример: `gh workflow run package-acceptance.yml --ref main -f workflow_ref=main -f source=npm -f package_spec=openclaw@beta -f suite_profile=product -f published_upgrade_survivor_baseline=openclaw@2026.4.26 -f telegram_mode=mock-openai`
  Общие профили:
  - `smoke`: линии install/channel/agent, gateway network и config reload
  - `package`: нативные для артефакта линии package/update/restart/plugin без OpenWebUI или live ClawHub
  - `product`: профиль package плюс каналы MCP, очистка cron/subagent,
    веб-поиск OpenAI и OpenWebUI
  - `full`: фрагменты Docker release-path с OpenWebUI
  - `custom`: точный выбор `docker_lanes` для сфокусированного повторного запуска
- Запустите ручной рабочий процесс `CI` напрямую, когда вам нужно только детерминированное обычное
  покрытие CI для кандидата релиза. Ручные запуски CI обходят changed
  scoping и принудительно запускают Linux Node shards, bundled-plugin shards, plugin и
  channel contract shards, совместимость с Node 22, `check-*`, `check-additional-*`,
  smoke-проверки собранных артефактов, проверки документации, Python skills, Windows, macOS и
  линии i18n Control UI. Отдельные ручные запуски CI запускают Android только при запуске
  с `include_android=true`; `Full Release Validation` передает этот ввод своему
  дочернему CI.
  Пример с Android: `gh workflow run ci.yml --ref release/YYYY.M.PATCH -f include_android=true`
- Запустите `pnpm qa:otel:smoke` при проверке телеметрии релиза. Он проверяет
  QA-lab через локальный OTLP/HTTP receiver и проверяет экспорт trace, metric и log,
  а также ограниченные атрибуты trace и редактирование содержимого/идентификаторов без
  необходимости в Opik, Langfuse или другом внешнем collector.
- Запустите `pnpm qa:otel:collector-smoke` при проверке совместимости collector.
  Он маршрутизирует тот же OTLP-экспорт QA-lab через реальный Docker-контейнер OpenTelemetry Collector
  перед проверками локального receiver.
- Запустите `pnpm qa:prometheus:smoke` при проверке защищенного Prometheus scraping.
  Он проверяет QA-lab, отклоняет неаутентифицированный scraping и проверяет, что
  критически важные для релиза семейства метрик остаются без содержимого prompt, сырых идентификаторов,
  токенов аутентификации и локальных путей.
- Запустите `pnpm qa:observability:smoke`, когда нужны smoke-линии OpenTelemetry и Prometheus
  из исходного checkout одна за другой.
- Запускайте `pnpm release:check` перед каждым тегированным релизом
- Предварительная проверка `OpenClaw NPM Release` генерирует доказательства релиза зависимостей до
  упаковки npm tarball. Шлюз уязвимостей npm advisory
  блокирует релиз. Отчеты о транзитивном риске манифеста, поверхности владения/установки зависимостей
  и изменениях зависимостей являются только доказательствами релиза. Отчет
  об изменениях зависимостей сравнивает кандидата релиза с предыдущим
  достижимым тегом релиза.
- Предварительная проверка загружает доказательства зависимостей как
  `openclaw-release-dependency-evidence-<tag>`, а также встраивает их в
  `dependency-evidence/` внутри подготовленного артефакта предварительной проверки npm. Реальный
  путь публикации повторно использует этот артефакт предварительной проверки, затем прикрепляет те же доказательства
  к релизу GitHub как `openclaw-<version>-dependency-evidence.zip`.
- Запустите `OpenClaw Release Publish` для изменяющей последовательности публикации после того, как
  тег существует. Запускайте его из `release/YYYY.M.PATCH` (или `main` при публикации
  тега, достижимого из main), передайте тег релиза, успешный OpenClaw npm
  `preflight_run_id` и успешный `full_release_validation_run_id`, и сохраняйте
  область публикации Plugin по умолчанию `all-publishable`, если только вы намеренно
  не запускаете сфокусированное восстановление. Рабочий процесс сериализует публикацию Plugin в npm, публикацию Plugin
  в ClawHub и публикацию OpenClaw в npm, чтобы основной пакет не был опубликован
  раньше своих внешних Plugin.
- Стабильный `OpenClaw Release Publish` требует точный `windows_node_tag` после того, как
  существует соответствующий непререлизный релиз `openclaw/openclaw-windows-node`.
  Он также требует утвержденную кандидатом карту `windows_node_installer_digests`.
  Перед запуском любого дочернего publish он проверяет, что исходный релиз
  опубликован, не является prerelease, содержит обязательные установщики x64/ARM64 и
  все еще совпадает с этой утвержденной картой. Затем он запускает `Windows Node Release`,
  пока релиз OpenClaw все еще является черновиком, передавая закрепленную карту digest установщиков
  без изменений. Дочерний
  рабочий процесс скачивает подписанные установщики Windows Hub из этого точного тега,
  сопоставляет их с закрепленными digest, проверяет, что их подписи Authenticode
  используют ожидаемого подписанта OpenClaw Foundation на Windows runner,
  записывает манифест SHA-256 и загружает установщики плюс манифест в
  канонический релиз OpenClaw на GitHub, затем повторно скачивает продвинутые артефакты и
  проверяет включение в манифест и хэши. Родитель проверяет текущий
  контракт артефактов x64, ARM64 и checksum перед публикацией. Прямое восстановление
  отклоняет неожиданные имена артефактов `OpenClawCompanion-*` перед заменой
  ожидаемых контрактных артефактов закрепленными исходными байтами. Запускайте вручную
  `Windows Node Release` только для восстановления и всегда передавайте точный тег, никогда
  `latest`, плюс явную JSON-карту `expected_installer_digests` из
  утвержденного исходного релиза. Ссылки загрузки на сайте должны указывать на точные URL артефактов релиза OpenClaw
  для текущего стабильного релиза или
  `releases/latest/download/...` только после проверки, что latest redirect GitHub
  указывает на тот же релиз; не ссылайтесь только на страницу релиза companion repo.
- Проверки релиза теперь выполняются в отдельном ручном рабочем процессе:
  `OpenClaw Release Checks`
- `OpenClaw Release Checks` также запускает mock parity линию QA Lab плюс быстрый
  live-профиль Matrix и линию Telegram QA перед утверждением релиза. Live
  линии используют среду `qa-live-shared`; Telegram также использует lease учетных данных Convex CI.
  Запустите ручной рабочий процесс `QA-Lab - All Lanes` с
  `matrix_profile=all` и `matrix_shards=true`, когда нужен полный инвентарь транспорта,
  media и E2EE Matrix в параллели.
- Межплатформенная runtime-проверка установки и обновления является частью публичных
  `OpenClaw Release Checks` и `Full Release Validation`, которые вызывают
  reusable workflow
  `.github/workflows/openclaw-cross-os-release-checks-reusable.yml` напрямую
- Это разделение намеренное: сохраняйте реальный путь npm-релиза коротким,
  детерминированным и сфокусированным на артефактах, а более медленные live-проверки держите в
  собственной линии, чтобы они не задерживали и не блокировали публикацию
- Проверки релиза с секретами следует запускать через `Full Release
Validation` или из workflow ref `main`/release, чтобы логика рабочего процесса и
  секреты оставались контролируемыми
- `OpenClaw Release Checks` принимает ветку, тег или полный SHA коммита, если
  разрешенный коммит достижим из ветки OpenClaw или тега релиза
- Предварительная проверка validation-only `OpenClaw NPM Release` также принимает текущий
  полный 40-символьный SHA коммита ветки workflow без необходимости в отправленном теге
- Этот путь SHA предназначен только для проверки и не может быть продвинут в реальную публикацию
- В режиме SHA рабочий процесс синтезирует `v<package.json version>` только для
  проверки метаданных пакета; реальная публикация по-прежнему требует реальный тег релиза
- Оба рабочих процесса сохраняют реальный путь публикации и продвижения на GitHub-hosted
  runners, тогда как неизменяющий путь проверки может использовать более крупные
  Blacksmith Linux runners
- Этот рабочий процесс запускает
  `OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache`
  с использованием workflow secrets `OPENAI_API_KEY` и `ANTHROPIC_API_KEY`
- Предварительная проверка npm-релиза больше не ждет отдельную линию release checks
- Перед локальным тегированием кандидата релиза запустите
  `RELEASE_TAG=vYYYY.M.PATCH-beta.N pnpm release:fast-pretag-check`. Helper
  запускает быстрые защитные проверки релиза, проверки npm/ClawHub релиза Plugin, build,
  UI build и `release:openclaw:npm:check` в порядке, который выявляет частые
  ошибки, блокирующие утверждение, до запуска workflow публикации GitHub.
- Запустите `RELEASE_TAG=vYYYY.M.PATCH node --import tsx scripts/openclaw-npm-release-check.ts`
  (или соответствующий тег beta/correction) перед утверждением
- После публикации npm запустите
  `node --import tsx scripts/openclaw-npm-postpublish-verify.ts YYYY.M.PATCH`
  (или соответствующую beta/correction-версию), чтобы проверить путь установки
  из опубликованного реестра в свежем временном префиксе
- После публикации beta выполните `OPENCLAW_NPM_TELEGRAM_PACKAGE_SPEC=openclaw@YYYY.M.PATCH-beta.N OPENCLAW_NPM_TELEGRAM_CREDENTIAL_SOURCE=convex OPENCLAW_NPM_TELEGRAM_CREDENTIAL_ROLE=ci pnpm test:docker:npm-telegram-live`,
  чтобы проверить настройку установленного пакета, настройку Telegram и реальный E2E Telegram
  для опубликованного npm-пакета с использованием общего пула арендованных учетных данных Telegram.
  Разовые локальные запуски сопровождающих могут опускать переменные Convex и передавать три
  учетные данные окружения `OPENCLAW_QA_TELEGRAM_*` напрямую.
- Чтобы запустить полную дымовую проверку beta после публикации с машины сопровождающего, используйте `pnpm release:beta-smoke -- --beta betaN`. Вспомогательный скрипт выполняет проверку npm-обновления Parallels и свежей целевой установки, запускает `NPM Telegram Beta E2E`, опрашивает точный запуск workflow, скачивает артефакт и выводит отчет Telegram.

  ручной workflow `NPM Telegram Beta E2E`. Он намеренно доступен только вручную и
  не запускается при каждом merge.
- Автоматизация релизов для сопровождающих теперь использует схему «предварительная проверка, затем продвижение»:
  - реальная публикация npm должна пройти успешный npm `preflight_run_id`
  - реальная публикация npm должна запускаться из той же ветки `main` или
    `release/YYYY.M.PATCH`, что и успешный запуск предварительной проверки
  - стабильные релизы npm по умолчанию используют `beta`

    `npm dist-tag add` по-прежнему требует `NPM_TOKEN`, тогда как исходный репозиторий сохраняет
    публикацию только через OIDC
  - публичный `macOS Release` предназначен только для валидации; когда тег существует только в
    релизной ветке, но workflow запускается из `main`, задайте

  - реальные пути публикации продвигают подготовленные артефакты вместо их повторной сборки

  матрицы тестов расширений, перед одобрением заново сгенерируйте и просмотрите принадлежащие планировщику
  выходные данные матрицы `plugin-prerelease-extension-shard` из
  `.github/workflows/plugin-prerelease.yml`, чтобы примечания к релизу не описывали
  устаревшую структуру CI
- Готовность стабильного релиза macOS также включает поверхности обновления:
  - GitHub-релиз должен в итоге содержать упакованные `.zip`, `.dmg` и `.dSYM.zip`
  - `appcast.xml` в `main` должен указывать на новый стабильный zip после публикации; workflow
    публикации macOS коммитит его автоматически или открывает PR для appcast,
    когда прямой push заблокирован
  - упакованное приложение должно сохранять неотладочный bundle id, непустой URL ленты Sparkle
    и `CFBundleVersion` на уровне или выше канонического минимального номера сборки Sparkle
    для этой версии релиза

## Тестовые боксы релиза

`Full Release Validation` — это способ, которым операторы запускают все предрелизные тесты из
одной точки входа. Для доказательства закрепленного коммита на быстро меняющейся ветке используйте
вспомогательную команду, чтобы каждый дочерний workflow выполнялся из временной ветки, зафиксированной на целевом
SHA:

```bash
pnpm ci:full-release --sha <full-sha>
```

Вспомогательная команда отправляет `release-ci/<sha>-...`, запускает `Full Release Validation`
из этой ветки с `ref=<sha>`, проверяет, что `headSha` каждого дочернего workflow
совпадает с целевым значением, а затем удаляет временную ветку. Это исключает случайное подтверждение
более нового дочернего запуска `main`.

Для валидации релизной ветки или тега запускайте ее из доверенного ref workflow `main`
и передавайте релизную ветку или тег как `ref`:

```bash
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=release/YYYY.M.PATCH \
  -f provider=openai \
  -f mode=both \
  -f release_profile=stable \
  -f evidence_package_spec=openclaw@YYYY.M.PATCH-beta.N
```

Workflow разрешает целевой ref, запускает ручной `CI` с
`target_ref=<release-ref>`, затем запускает `OpenClaw Release Checks`.
`OpenClaw Release Checks` разворачивает install smoke, кросс-OS релизные проверки,
покрытие live/E2E Docker для релизного пути, когда включен soak, Package Acceptance
с каноническим Telegram package E2E, паритет QA Lab, live Matrix и live
Telegram. Полный запуск full/all приемлем только тогда, когда сводка `Full Release Validation`
показывает `normal_ci`, `plugin_prerelease` и `release_checks` как
успешные, если только сфокусированный повторный запуск намеренно не пропустил отдельный дочерний `Plugin
Prerelease`. Используйте отдельный дочерний `npm-telegram` только для сфокусированного
повторного запуска опубликованного пакета с `release_package_spec` или
`npm_telegram_package_spec`. Итоговая сводка
верификатора включает таблицы самых медленных задач для каждого дочернего запуска, чтобы релиз-менеджер
мог видеть текущий критический путь без скачивания логов.
См. [Полная валидация релиза](/ru/reference/full-release-validation) для
полной матрицы этапов, точных имен задач workflow, различий между профилями stable и full,
артефактов и дескрипторов сфокусированных повторных запусков.
Дочерние workflow запускаются из доверенного ref, который выполняет `Full Release
Validation`, обычно `--ref main`, даже когда целевой `ref` указывает на
более старую релизную ветку или тег. Отдельного входного параметра workflow-ref для Full Release Validation
нет; выбирайте доверенную обвязку, выбирая ref запуска workflow.
Не используйте `--ref main -f ref=<sha>` для точного доказательства коммита на подвижном `main`;
raw commit SHA не могут быть refs для workflow dispatch, поэтому используйте
`pnpm ci:full-release --sha <sha>`, чтобы создать закрепленную временную ветку.

Используйте `release_profile`, чтобы выбрать широту live/provider:

- `minimum`: самый быстрый релизно-критичный live и Docker путь OpenAI/core
- `stable`: minimum плюс покрытие стабильных provider/backend для утверждения релиза
- `full`: stable плюс широкое консультативное покрытие provider/media

Валидация stable и full всегда выполняет исчерпывающий live/E2E, Docker
релизный путь и ограниченный sweep выживания опубликованных обновлений перед продвижением.
Используйте `run_release_soak=true`, чтобы запросить тот же sweep для беты. Этот sweep покрывает
последние четыре стабильных пакета плюс закрепленные базовые версии `2026.4.23` и `2026.5.2`
и более старое покрытие `2026.4.15`, с удалением дублирующихся базовых версий и
разбиением каждой базовой версии в отдельную задачу Docker runner.

`OpenClaw Release Checks` использует доверенный ref workflow, чтобы один раз разрешить целевой
ref как `release-package-under-test`, и переиспользует этот артефакт в cross-OS,
Package Acceptance и Docker проверках релизного пути, когда выполняется soak. Это удерживает
все package-facing боксы на одних и тех же байтах и избегает повторных сборок пакета.
После того как бета уже опубликована в npm, задайте `release_package_spec=openclaw@YYYY.M.PATCH-beta.N`,
чтобы релизные проверки один раз скачали поставленный пакет, извлекли SHA его исходника сборки
из `dist/build-info.json` и переиспользовали этот артефакт для cross-OS,
Package Acceptance, Docker релизного пути и package Telegram lanes.
Cross-OS OpenAI install smoke использует `OPENCLAW_CROSS_OS_OPENAI_MODEL`, когда
задана переменная repo/org, иначе `openai/gpt-5.4`, потому что эта lane
подтверждает установку пакета, onboarding, запуск Gateway и один live agent turn,
а не бенчмаркинг самой медленной модели по умолчанию. Более широкая live provider
matrix остается местом для model-specific покрытия.

Используйте эти варианты в зависимости от этапа релиза:

```bash
# Validate an unpublished release candidate branch.
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=release/YYYY.M.PATCH \
  -f provider=openai \
  -f mode=both \
  -f release_profile=stable

# Validate an exact pushed commit.
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=<40-char-sha> \
  -f provider=openai \
  -f mode=both

# After publishing a beta, add published-package Telegram E2E.
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=release/YYYY.M.PATCH \
  -f provider=openai \
  -f mode=both \
  -f release_profile=full \
  -f release_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f evidence_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f npm_telegram_provider_mode=mock-openai
```

Не используйте полный umbrella как первый повторный запуск после сфокусированного исправления. Если один бокс
падает, используйте упавший дочерний workflow, задачу, Docker lane, профиль пакета, model
provider или QA lane для следующего доказательства. Запускайте полный umbrella снова только тогда, когда
исправление изменило общую релизную оркестрацию или сделало более раннее all-box evidence
устаревшим. Итоговый верификатор umbrella повторно проверяет записанные id запусков дочерних workflow,
поэтому после успешного повторного запуска дочернего workflow повторно запускайте только упавшую
родительскую задачу `Verify full validation`.

Для ограниченного восстановления передайте `rerun_group` в umbrella. `all` — настоящий
запуск release-candidate, `ci` запускает только обычный дочерний CI, `plugin-prerelease`
запускает только релизный дочерний plugin, `release-checks` запускает каждый релизный
бокс, а более узкие релизные группы — `install-smoke`, `cross-os`,
`live-e2e`, `package`, `qa`, `qa-parity`, `qa-live` и `npm-telegram`.
Сфокусированные повторные запуски `npm-telegram` требуют `release_package_spec` или
`npm_telegram_package_spec`; запуски full/all используют канонический package Telegram
E2E внутри Package Acceptance. Сфокусированные
cross-OS повторные запуски могут добавить `cross_os_suite_filter=windows/packaged-upgrade` или
другой фильтр OS/suite. Сбои QA release-check блокируют обычную
валидацию релиза, включая обязательный дрейф динамических инструментов OpenClaw в стандартном уровне.
Запуски Tideclaw alpha все еще могут считать non-package-safety release-check lanes
консультативными. Когда `live_suite_filter` явно запрашивает gated QA live lane, такую
как Discord, WhatsApp или Slack, соответствующая переменная repo
`OPENCLAW_RELEASE_QA_*_LIVE_CI_ENABLED` должна быть включена; иначе
захват ввода завершается ошибкой вместо тихого пропуска lane.

### Vitest

Бокс Vitest — это ручной дочерний workflow `CI`. Ручной CI намеренно
обходит changed scoping и принудительно запускает обычный граф тестов для release
candidate: шарды Linux Node, шарды bundled-plugin, шарды контрактов plugin и channel,
совместимость Node 22, `check-*`, `check-additional-*`,
smoke-проверки built-artifact, проверки документации, Python skills, Windows, macOS
и Control UI i18n. Android включается, когда `Full Release Validation` запускает
этот бокс, потому что umbrella передает `include_android=true`; standalone manual CI
требует `include_android=true` для покрытия Android.

Используйте этот бокс, чтобы ответить: "прошло ли дерево исходников полный обычный набор тестов?"
Это не то же самое, что product validation релизного пути. Evidence, которое нужно сохранить:

- сводка `Full Release Validation`, показывающая URL запущенного `CI`
- зеленый запуск `CI` на точном целевом SHA
- имена упавших или медленных шардов из задач CI при расследовании регрессий
- артефакты таймингов Vitest, такие как `.artifacts/vitest-shard-timings.json`, когда
  запуску нужен анализ производительности

Запускайте manual CI напрямую только когда релизу нужен детерминированный обычный CI, но
не Docker, QA Lab, live, cross-OS или package боксы. Используйте первую команду
для прямого CI без Android. Добавьте `include_android=true`, когда прямой
release-candidate CI должен покрывать Android:

```bash
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH -f include_android=true
```

### Docker

Бокс Docker находится в `OpenClaw Release Checks` через
`openclaw-live-and-e2e-checks-reusable.yml`, плюс release-mode
workflow `install-smoke`. Он валидирует release candidate через packaged
Docker environments, а не только через тесты уровня исходников.

Релизное Docker покрытие включает:

- полный install smoke с включенным медленным Bun global install smoke
- подготовку/переиспользование root Dockerfile smoke image по целевому SHA, где QR,
  root/gateway и installer/Bun smoke jobs выполняются как отдельные install-smoke
  шарды
- repository E2E lanes
- Docker чанки релизного пути: `core`, `package-update-openai`,
  `package-update-anthropic`, `package-update-core`, `plugins-runtime-plugins`,
  `plugins-runtime-services`,
  `plugins-runtime-install-a`, `plugins-runtime-install-b`,
  `plugins-runtime-install-c`, `plugins-runtime-install-d`,
  `plugins-runtime-install-e`, `plugins-runtime-install-f`,
  `plugins-runtime-install-g` и `plugins-runtime-install-h`
- покрытие OpenWebUI внутри чанка `plugins-runtime-services`, когда запрошено
- разделенные lanes установки/удаления bundled plugin
  `bundled-plugin-install-uninstall-0` through
  `bundled-plugin-install-uninstall-23`
- live/E2E provider suites и Docker live model coverage, когда release checks
  включают live suites

Используйте Docker артефакты перед повторным запуском. Планировщик релизного пути загружает
`.artifacts/docker-tests/` с логами lanes, `summary.json`, `failures.json`,
таймингами фаз, JSON плана планировщика и командами повторного запуска. Для сфокусированного восстановления
используйте `docker_lanes=<lane[,lane]>` в reusable live/E2E workflow вместо
повторного запуска всех релизных чанков. Сгенерированные команды повторного запуска включают предыдущие
`package_artifact_run_id` и подготовленные входы Docker image, когда доступны, чтобы
упавшая lane могла переиспользовать тот же tarball и GHCR images.

### QA Lab

Бокс QA Lab также является частью `OpenClaw Release Checks`. Это agentic
behavior и channel-level release gate, отдельный от Vitest и механики Docker
package.

Релизное покрытие QA Lab включает:

- mock parity lane, сравнивающую OpenAI candidate lane с базовой Opus 4.6
  с использованием agentic parity pack
- fast live Matrix QA profile с использованием среды `qa-live-shared`
- live Telegram QA lane с использованием leases учетных данных Convex CI
- `pnpm qa:otel:smoke`, `pnpm qa:otel:collector-smoke`,
  `pnpm qa:prometheus:smoke` или
  `pnpm qa:observability:smoke`, когда релизной telemetry нужно явное локальное
  доказательство

Используйте этот бокс, чтобы ответить: "ведет ли себя релиз корректно в QA scenarios и
live channel flows?" Сохраняйте URL артефактов для parity, Matrix и Telegram
lanes при утверждении релиза. Полное покрытие Matrix остается доступным как
ручной шардированный запуск QA-Lab, а не как release-critical lane по умолчанию.

### Package

Бокс Package — это gate installable-product. Он опирается на
`Package Acceptance` и resolver
`scripts/resolve-openclaw-package-candidate.mjs`. Resolver нормализует
candidate в tarball `package-under-test`, потребляемый Docker E2E, валидирует
inventory пакета, записывает версию пакета и SHA-256 и удерживает
workflow harness ref отдельно от package source ref.

Поддерживаемые источники candidate:

- `source=npm`: `openclaw@beta`, `openclaw@latest` или точная версия выпуска OpenClaw
- `source=ref`: упаковывает доверенную ветку `package_ref`, тег или полный SHA коммита
  с выбранной обвязкой `workflow_ref`
- `source=url`: скачивает публичный HTTPS `.tgz` с обязательным `package_sha256`;
  учетные данные в URL, нестандартные HTTPS-порты, частные, внутренние или специальные
  имена хостов либо разрешенные адреса, а также небезопасные перенаправления отклоняются
- `source=trusted-url`: скачивает HTTPS `.tgz` с обязательными
  `package_sha256` и `trusted_source_id` из именованной политики в
  `.github/package-trusted-sources.json`; используйте это для сопровождаемых мейнтейнерами
  корпоративных зеркал или частных репозиториев пакетов вместо добавления обхода
  частной сети на уровне входных данных в `source=url`
- `source=artifact`: повторно использует `.tgz`, загруженный другим запуском GitHub Actions

`OpenClaw Release Checks` запускает Package Acceptance с `source=artifact`,
подготовленным артефактом пакета выпуска, `suite_profile=custom`,
`docker_lanes=doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor update-restart-auth plugins-offline plugin-update`,
`telegram_mode=mock-openai`. Package Acceptance сохраняет проверку миграции, обновления,
перезапуска обновления с настроенной аутентификацией, установки Skills из живого ClawHub, очистки устаревших зависимостей плагинов, автономных
фикстур плагинов, обновления плагинов и пакетной QA для Telegram относительно одного и того же разрешенного
tarball. Блокирующие проверки выпуска используют базовый пакет по умолчанию:
последний опубликованный пакет; бета-профиль с `run_release_soak=true`, `release_profile=stable` или
`release_profile=full` расширяется до каждой стабильной опубликованной в npm базовой версии от
`2026.4.23` до `latest` плюс фикстуры для сообщенных проблем. Используйте
Package Acceptance с `source=npm` для уже выпущенного кандидата,
`source=ref` для локального npm tarball, подкрепленного SHA, перед публикацией,
`source=trusted-url` для сопровождаемого мейнтейнерами корпоративного или частного зеркала либо
`source=artifact` для подготовленного tarball, загруженного другим запуском GitHub Actions.
Это нативная для GitHub
замена большей части покрытия пакетов и обновлений, для которого раньше требовался
Parallels. Кросс-OS проверки выпуска все еще важны для специфичного для OS онбординга,
установщика и поведения платформы, но продуктовая проверка пакетов и обновлений должна
предпочитать Package Acceptance.

Канонический чеклист для проверки обновлений и плагинов:
[Тестирование обновлений и плагинов](/ru/help/testing-updates-plugins). Используйте его при
выборе локальной, Docker, Package Acceptance или release-check дорожки, которая доказывает
установку или обновление плагина, очистку doctor либо изменение миграции опубликованного пакета.
Исчерпывающая миграция опубликованных обновлений из каждого стабильного пакета `2026.4.23+` — это
отдельный ручной workflow `Update Migration`, а не часть Full Release CI.

Устаревшая снисходительность package-acceptance намеренно ограничена по времени. Пакеты до
`2026.4.25` включительно могут использовать путь совместимости для пробелов метаданных, уже опубликованных
в npm: отсутствующие в tarball частные записи QA-инвентаря, отсутствующий
`gateway install --wrapper`, отсутствующие файлы патчей в git-фикстуре, полученной из tarball,
отсутствующий сохраненный `update.channel`, устаревшие расположения install-record
плагинов, отсутствующее сохранение install-record маркетплейса и миграция метаданных
конфигурации во время `plugins update`. Опубликованный пакет `2026.4.26` может предупреждать
о файлах штампов метаданных локальной сборки, которые уже были выпущены. Более поздние пакеты
должны удовлетворять современным контрактам пакетов; те же самые пробелы проваливают проверку
выпуска.

Используйте более широкие профили Package Acceptance, когда вопрос выпуска касается
реального устанавливаемого пакета:

```bash
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f published_upgrade_survivor_baseline=openclaw@2026.4.26
```

Распространенные профили пакетов:

- `smoke`: быстрые дорожки установки пакета, канала, агента, Gateway-сети и
  перезагрузки конфигурации
- `package`: контракты установки, обновления, перезапуска и пакета плагинов плюс доказательство установки Skills из живого ClawHub; это значение по умолчанию для release-check
- `product`: `package` плюс MCP-каналы, очистка cron/subagent, веб-поиск OpenAI
  и OpenWebUI
- `full`: фрагменты Docker release-path с OpenWebUI
- `custom`: точный список `docker_lanes` для точечных повторных запусков

Для доказательства Telegram кандидата пакета включите `telegram_mode=mock-openai` или
`telegram_mode=live-frontier` в Package Acceptance. Workflow передает
разрешенный tarball `package-under-test` в дорожку Telegram; отдельный
workflow Telegram по-прежнему принимает опубликованную npm-спецификацию для проверок после публикации.

## Обычная автоматизация публикации выпуска

Для публикации beta, `latest`, плагинов, GitHub Release и платформ
`OpenClaw Release Publish` является обычной изменяющей точкой входа. Ежемесячный
npm-only путь расширенной стабильности `.33+` не использует этот оркестратор. Обычный workflow
оркестрирует workflow доверенного издателя в порядке, нужном выпуску:

1. Извлечь тег выпуска и разрешить его SHA коммита.
2. Проверить, что тег достижим из `main` или `release/*`.
3. Запустить `pnpm plugins:sync:check`.
4. Запустить `Plugin NPM Release` с `publish_scope=all-publishable` и
   `ref=<release-sha>`.
5. Запустить `Plugin ClawHub Release` с теми же областью и SHA.
6. Запустить `OpenClaw NPM Release` с тегом выпуска, npm dist-tag и
   сохраненным `preflight_run_id` после проверки сохраненного
   `full_release_validation_run_id`.
7. Для стабильных выпусков создать или обновить GitHub release как черновик, запустить
   `Windows Node Release` с явным `windows_node_tag` и
   одобренными для кандидата `windows_node_installer_digests`, а также проверить канонические
   артефакты установщика и контрольных сумм перед публикацией черновика.

Пример публикации beta:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref release/YYYY.M.PATCH \
  -f tag=vYYYY.M.PATCH-beta.N \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f npm_dist_tag=beta
```

Публикация stable в dist-tag beta по умолчанию:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref release/YYYY.M.PATCH \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f npm_dist_tag=beta
```

Прямое продвижение stable в `latest` выполняется явно:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref release/YYYY.M.PATCH \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f npm_dist_tag=latest
```

Используйте низкоуровневые workflow `Plugin NPM Release` и `Plugin ClawHub Release`
только для точечного ремонта или повторной публикации. `OpenClaw Release Publish` отклоняет
`plugin_publish_scope=selected`, когда `publish_openclaw_npm=true`, чтобы основной
пакет не мог быть выпущен без каждого публикуемого официального плагина, включая
`@openclaw/diffs-language-pack`. Для ремонта выбранного плагина задайте
`publish_openclaw_npm=false` с `plugin_publish_scope=selected` и
`plugins=@openclaw/name` либо запустите дочерний workflow напрямую.

## Входные данные NPM workflow

`OpenClaw NPM Release` принимает следующие управляемые оператором входные данные:

- `tag`: обязательный тег выпуска, например `v2026.4.2`, `v2026.4.2-1` или
  `v2026.4.2-beta.1`; когда `preflight_only=true`, это также может быть текущий
  полный 40-символьный SHA коммита ветки workflow для preflight только проверки
- `preflight_only`: `true` только для проверки, сборки и пакета, `false` для
  настоящего пути публикации
- `preflight_run_id`: обязателен на настоящем пути публикации, чтобы workflow повторно использовал
  подготовленный tarball из успешного preflight-запуска
- `full_release_validation_run_id`: обязателен для настоящей ежемесячной extended-stable и обычной
  не-beta публикации, чтобы workflow аутентифицировал точный запуск проверки
- `npm_dist_tag`: целевой npm-тег для пути публикации; принимает `alpha`, `beta`,
  `latest` или `extended-stable` и по умолчанию равен `beta`. Финальный патч `33` и позже должны
  использовать `extended-stable`; по умолчанию `extended-stable` отклоняет более ранние патчи и всегда
  отклоняет нефинальные теги.
- `bypass_extended_stable_guard`: булево значение только для тестирования, по умолчанию `false`; с
  `npm_dist_tag=extended-stable` обходит eligibility ежемесячной extended-stable, сохраняя
  проверки идентичности выпуска, артефакта, одобрения и readback.

`OpenClaw Release Publish` принимает следующие управляемые оператором входные данные:

- `tag`: обязательный тег выпуска; уже должен существовать
- `preflight_run_id`: id успешного preflight-запуска `OpenClaw NPM Release`;
  обязателен, когда `publish_openclaw_npm=true`
- `full_release_validation_run_id`: id успешного запуска `Full Release Validation`;
  обязателен, когда `publish_openclaw_npm=true`
- `windows_node_tag`: точный непререлизный тег выпуска `openclaw/openclaw-windows-node`;
  обязателен для публикации стабильного OpenClaw
- `windows_node_installer_digests`: одобренная для кандидата компактная JSON-карта
  текущих имен установщиков Windows к их закрепленным digest `sha256:`; обязательна
  для публикации стабильного OpenClaw
- `npm_dist_tag`: целевой npm-тег для пакета OpenClaw
- `plugin_publish_scope`: по умолчанию `all-publishable`; используйте `selected` только
  для точечного ремонта только плагинов с `publish_openclaw_npm=false`
- `plugins`: разделенные запятыми имена пакетов `@openclaw/*`, когда
  `plugin_publish_scope=selected`
- `publish_openclaw_npm`: по умолчанию `true`; задавайте `false` только при использовании
  workflow как оркестратора ремонта только плагинов
- `wait_for_clawhub`: по умолчанию `false`, чтобы доступность npm не блокировалась
  сайдкаром ClawHub; задавайте `true` только когда завершение workflow должно включать
  завершение ClawHub

`OpenClaw Release Checks` принимает следующие управляемые оператором входные данные:

- `ref`: ветка, тег или полный SHA коммита для проверки. Проверки с секретами
  требуют, чтобы разрешенный коммит был достижим из ветки OpenClaw или
  тега выпуска.
- `run_release_soak`: включает исчерпывающие live/E2E, Docker release-path и
  all-since upgrade-survivor soak для проверок beta-выпуска. Принудительно включается
  `release_profile=stable` и `release_profile=full`.

Правила:

- Обычные финальные и корректирующие версии ниже патча `33` могут публиковаться либо в
  `beta`, либо в `latest`. Финальные версии на патче `33` или выше должны публиковаться в
  `extended-stable`, а версии с корректирующим суффиксом на этой границе отклоняются.
- Beta prerelease-теги могут публиковаться только в `beta`
- Для `OpenClaw NPM Release` ввод полного SHA коммита разрешен только когда
  `preflight_only=true`
- `OpenClaw Release Checks` и `Full Release Validation` всегда
  выполняют только проверку
- Настоящий путь публикации должен использовать тот же `npm_dist_tag`, что использовался во время preflight;
  workflow проверяет, что метаданные перед публикацией продолжают это подтверждать

## Обычная последовательность beta/latest stable выпуска

Эта устаревшая последовательность предназначена для обычного оркестрированного выпуска, который также владеет
плагинами, GitHub Release, Windows и другой платформенной работой. Это не
ежемесячный npm-only путь extended-stable `.33+`, задокументированный в начале этой страницы.

При подготовке обычного оркестрированного stable выпуска:

1. Запустите `OpenClaw NPM Release` с `preflight_only=true`
   - До появления тега можно использовать текущий полный SHA коммита ветки workflow
     для проверочного dry run только preflight workflow
2. Выберите `npm_dist_tag=beta` для обычного потока сначала beta или `latest` только
   когда вы намеренно хотите прямую публикацию стабильной версии
3. Запустите `Full Release Validation` на release-ветке, release-теге или полном
   SHA коммита, когда вам нужна обычная CI-проверка плюс покрытие live prompt cache, Docker, QA Lab,
   Matrix и Telegram из одного ручного workflow
4. Если вам намеренно нужен только детерминированный обычный граф тестов, вместо этого запустите
   ручной workflow `CI` на release ref
5. Выберите точный не prerelease release-тег `openclaw/openclaw-windows-node`,
   чьи подписанные установщики x64 и ARM64 должны быть выпущены. Сохраните его как
   `windows_node_tag`, а их проверенную карту digest сохраните как
   `windows_node_installer_digests`. Вспомогательный инструмент release-candidate записывает оба
   значения и включает их в сгенерированную команду публикации.
6. Сохраните успешные `preflight_run_id` и `full_release_validation_run_id`
7. Запустите `OpenClaw Release Publish` с тем же `tag`, тем же `npm_dist_tag`,
   выбранным `windows_node_tag`, сохраненным `windows_node_installer_digests`,
   сохраненным `preflight_run_id` и сохраненным `full_release_validation_run_id`;
   он публикует externalized plugins в npm и ClawHub перед продвижением
   npm-пакета OpenClaw
8. Если релиз попал в `beta`, используйте
   workflow `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml`,
   чтобы продвинуть эту стабильную версию из `beta` в `latest`
9. Если релиз намеренно был опубликован напрямую в `latest`, а `beta`
   должна сразу следовать той же стабильной сборке, используйте тот же release
   workflow, чтобы направить оба dist-tags на стабильную версию, или позвольте его запланированной
   самовосстанавливающейся синхронизации переместить `beta` позднее

Мутация dist-tag находится в репозитории release ledger, потому что для нее все еще требуется
`NPM_TOKEN`, тогда как исходный репозиторий сохраняет публикацию только через OIDC.

Так и путь прямой публикации, и путь продвижения сначала beta остаются
задокументированными и видимыми оператору.

Если maintainer должен откатиться к локальной npm-аутентификации, запускайте любые команды 1Password
CLI (`op`) только внутри выделенной tmux-сессии. Не вызывайте `op`
напрямую из основной agent shell; удерживание его внутри tmux делает prompts,
alerts и обработку OTP наблюдаемыми и предотвращает повторяющиеся host alerts.
