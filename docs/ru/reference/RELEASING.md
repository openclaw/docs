В настоящее время OpenClaw предоставляет три пользовательских канала обновлений:

- stable: существующий продвигаемый канал релизов, который по-прежнему разрешается через npm `latest` до завершения отдельного этапа внедрения CLI/каналов
- beta: теги предварительных релизов, публикуемые в npm `beta`
- dev: изменяемая вершина ветки `main`

Кроме того, операторы релизов могут публиковать основной пакет за последний завершённый месяц в npm `extended-stable`, начиная с патча `33`. Обычная финальная линия текущего месяца продолжает публиковаться в npm `latest`; само по себе это разделение публикаций на стороне оператора не изменяет разрешение каналов обновлений CLI.

Альфа-сборки Tideclaw представляют собой отдельную внутреннюю линию предварительных релизов (dist-tag npm `alpha`), описанную в разделах [Входные параметры рабочего процесса NPM](#npm-workflow-inputs) и [Тестовые среды релизов](#release-test-boxes).

## Именование версий

- Версия ежемесячного extended-stable-релиза npm: `YYYY.M.PATCH`, где `PATCH >= 33`, тег git `vYYYY.M.PATCH`
- Версия ежедневного/обычного финального релиза: `YYYY.M.PATCH`, где `PATCH < 33`, тег git `vYYYY.M.PATCH`
- Версия резервного корректирующего обычного релиза: `YYYY.M.PATCH-N`, тег git `vYYYY.M.PATCH-N`
- Версия предварительного beta-релиза: `YYYY.M.PATCH-beta.N`, тег git `vYYYY.M.PATCH-beta.N`
- Версия предварительного alpha-релиза: `YYYY.M.PATCH-alpha.N`, тег git `vYYYY.M.PATCH-alpha.N`
- Никогда не дополняйте номер месяца или патча ведущими нулями
- `PATCH` — это последовательный номер ежемесячной линии релизов, а не календарный день. Обычные финальные и beta-релизы продвигают текущую линию; теги только для alpha никогда не занимают и не увеличивают номер патча beta/обычного релиза, поэтому при выборе линии beta- или обычного релиза игнорируйте устаревшие теги только для alpha с более высокими номерами патчей.
- Сборки alpha/nightly используют следующую ещё не выпущенную линию патча, а при повторных сборках увеличивают только `alpha.N`. После появления beta-релиза для этого патча новые alpha-сборки переходят к следующему патчу.
- Версии npm неизменяемы: никогда не удаляйте, не публикуйте повторно и не используйте повторно опубликованный тег. Вместо этого выпускайте следующий номер предварительного релиза или следующий ежемесячный патч.
- `latest` продолжает указывать на текущую обычную/ежедневную линию npm; `beta` является текущей целью установки beta-версии
- `extended-stable` означает поддерживаемый пакет npm за предыдущий месяц, начиная с патча `33`; патч `34` и последующие являются обслуживающими релизами этой ежемесячной линии
- Обычные финальные и обычные корректирующие релизы по умолчанию публикуются в npm `beta`; операторы релизов могут явно выбрать `latest` или позднее продвинуть проверенную beta-сборку
- Специализированный ежемесячный путь extended-stable публикует основной пакет npm и каждый официальный Plugin, допускающий публикацию в npm, с одной и той же точной версией. Он не публикует плагины в ClawHub и не публикует артефакты macOS или Windows, GitHub Release, dist-tag для закрытых репозиториев, образы Docker, мобильные артефакты или загрузки для сайта.
- Каждый обычный финальный релиз одновременно поставляет пакет npm, приложение macOS, подписанный автономный APK для Android и подписанные установщики Windows Hub. Beta-релизы обычно сначала проверяют и публикуют путь npm/пакета, а сборка, подписание, нотариальное заверение и продвижение нативных приложений выполняются только для обычного финального релиза, если иное не запрошено явно.

## Периодичность релизов

- Релизы сначала проходят через beta; stable следует только после проверки последней beta-версии
- Сопровождающие обычно выпускают релизы из ветки `release/YYYY.M.PATCH`, созданной из текущей `main`, чтобы проверка и исправления релиза не блокировали новую разработку в `main`
- Если тег beta уже отправлен или опубликован и требует исправления, сопровождающие выпускают следующий тег `-beta.N`, а не удаляют или пересоздают старый
- Подробная процедура релиза, согласования, учётные данные и инструкции по восстановлению доступны только сопровождающим

## Ежемесячная публикация extended-stable только в npm

Это отдельное исключение из описанной ниже обычной процедуры релиза. Для завершённого месяца `YYYY.M` создайте `extended-stable/YYYY.M.33`; публикуйте `vYYYY.M.33` и последующие обслуживающие патчи из той же ветки. Тег релиза, вершина ветки, рабочая копия, версия пакета, предварительная проверка npm и запуск полной проверки релиза должны указывать на один и тот же коммит. Защищённая ветка `main` уже должна содержать финальную версию строго более позднего календарного месяца с патчем ниже `33`; обслуживающие патчи остаются допустимыми после того, как `main` продвинется более чем на один месяц.

В точной ветке extended-stable увеличьте версию корневого пакета до `YYYY.M.P`, выполните `pnpm release:prep` и убедитесь, что каждый пакет расширения, допускающий публикацию, имеет ту же версию. Зафиксируйте и отправьте все сгенерированные изменения, создайте и отправьте неизменяемый тег `vYYYY.M.P` на этом коммите и запишите полученный полный SHA. Рабочие процессы используют это подготовленное дерево; они не увеличивают и не синхронизируют версии за вас.

Запустите предварительную проверку npm и полную проверку релиза с точной вершины подготовленной ветки, затем сохраните идентификаторы обоих запусков и номер успешной попытки запуска полной проверки релиза:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=true \
  -f npm_dist_tag=extended-stable

gh workflow run full-release-validation.yml \
  --ref extended-stable/YYYY.M.33 \
  -f ref=extended-stable/YYYY.M.33 \
  -f release_profile=stable
```

`release_profile=stable` — существующий профиль глубины проверки; он не связан с dist-tag npm `extended-stable` и намеренно оставлен без изменений.

После успешного завершения обоих запусков опубликуйте каждый официальный Plugin, допускающий публикацию в npm, из той же точной вершины ветки. Патч `P` должен быть равен `33` или больше. Передайте полный SHA релиза в качестве `ref`, дождитесь завершения всей матрицы и обратной проверки реестра, затем сохраните идентификатор успешного запуска выпуска Plugin в NPM:

```bash
RELEASE_SHA="$(git rev-parse HEAD)"
gh workflow run plugin-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f publish_scope=all-publishable \
  -f ref="$RELEASE_SHA" \
  -f npm_dist_tag=extended-stable
```

Рабочий процесс использует обычный подготовленный перечень пакетов `all-publishable`, включая пакеты, исходный код которых не изменился. Перед успешным завершением он проверяет каждый точный пакет и каждый тег Plugin `extended-stable`. Если частичный запуск завершается с ошибкой, повторно выполните ту же команду: уже опубликованные пакеты будут использованы повторно, отсутствующие или устаревшие теги Plugin будут приведены в соответствие в среде выпуска npm, а итоговая обратная проверка по-прежнему охватит полный набор пакетов.

После успешного завершения рабочего процесса Plugin и готовности среды выпуска npm опубликуйте точный tarball основного пакета из предварительной проверки. Публикация основного пакета проверяет, что указанный запуск Plugin имеет состояние `completed/success` в той же канонической ветке и для точного SHA исходного кода:

```bash
gh workflow run openclaw-npm-release.yml \
  --ref extended-stable/YYYY.M.33 \
  -f tag=vYYYY.M.P \
  -f preflight_only=false \
  -f npm_dist_tag=extended-stable \
  -f preflight_run_id=<npm-preflight-run-id> \
  -f full_release_validation_run_id=<full-validation-run-id> \
  -f full_release_validation_run_attempt=<full-validation-run-attempt> \
  -f plugin_npm_run_id=<plugin-npm-run-id>
```

Для репетиции в форке или непродуктивной среде, которая намеренно не может удовлетворить ежемесячной политике `.33` или политике месяцев защищённой ветки `main`, добавьте `-f bypass_extended_stable_guard=true` как при запуске предварительной проверки npm, так и при запуске публикации. Значение по умолчанию — `false`. Обход принимается только вместе с `npm_dist_tag=extended-stable` и фиксируется в сводке рабочего процесса. Он не обходит требования к канонической ссылке рабочего процесса `extended-stable/YYYY.M.33`, равенству вершины ветки, тега и рабочей копии, синтаксису финального тега, равенству версий пакета и тега, идентичности указанных запусков и манифеста, происхождению tarball, согласованию среды, обратной проверке реестра или подтверждению исправления селектора.

Рабочий процесс публикации проверяет идентичность указанных запусков предварительной проверки, валидации и Plugin, дайджест подготовленного tarball и селекторы реестра основного пакета. После успешного завершения рабочего процесса независимо подтвердите результат:

```bash
npm view openclaw@YYYY.M.P version --userconfig "$(mktemp)"
npm view openclaw@extended-stable version --userconfig "$(mktemp)"
```

Обе команды должны вернуть `YYYY.M.P`. Если публикация завершилась успешно, но обратная проверка селектора не удалась, не публикуйте неизменяемую версию пакета повторно. Используйте единственную команду исправления `npm dist-tag add openclaw@YYYY.M.P extended-stable`, напечатанную в безусловно выполняемой сводке завершившегося с ошибкой рабочего процесса, затем повторите обе независимые обратные проверки. Откат к предыдущему селектору является отдельным решением оператора, а не способом исправления обратной проверки.

В общедоступной документации поддержки Slack, Discord и Codex изначально указаны как поддерживаемые поверхности Plugin для extended-stable. Этот список является заявлением о поддержке, а не списком разрешений в коде релиза: каждый официальный Plugin, допускающий публикацию в npm, следует одному и тому же пути публикации точной версии.

Обычный контрольный список ниже по-прежнему охватывает beta, `latest`, GitHub Release, плагины, macOS, Windows и публикацию на других платформах. Не выполняйте эти шаги для данного пути extended-stable, предназначенного только для npm.

## Контрольный список оператора обычного релиза

Этот контрольный список описывает общедоступную структуру процесса выпуска. Закрытые учётные данные, подписание, нотариальное заверение, восстановление dist-tag и сведения об экстренном откате остаются в доступной только сопровождающим инструкции по выпуску.

1. Начните с актуальной ветки `main`: получите последние изменения, убедитесь, что целевой коммит отправлен в репозиторий, а состояние CI для `main` позволяет создать от неё ветку.
2. Сформируйте верхний раздел `CHANGELOG.md` на основе объединённых PR и всех прямых коммитов после последнего достижимого тега выпуска. Формулируйте записи для пользователей, устраните дублирование между пересекающимися записями PR и прямых коммитов, создайте коммит, отправьте его и ещё раз выполните rebase/получение изменений перед созданием ветки. Если расходящийся выпущенный тег или последующий перенос изменений вперёд повторно связывает уже выпущенные PR, явно передайте этот тег через `--shipped-ref`; средство проверки использует явные строки PR из полных записей о вкладе в нумерованных разделах снимка тега, игнорирует `Unreleased` и фиксирует точный перечень и количество исключённых PR.
3. Проверьте записи о совместимости выпусков в `src/plugins/compat/registry.ts` и `src/commands/doctor/shared/deprecation-compat.ts`. Удаляйте истёкшую совместимость, только если путь обновления остаётся покрытым, либо укажите, почему она намеренно сохраняется.
4. Создайте `release/YYYY.M.PATCH` от актуальной ветки `main`. Не выполняйте обычную подготовку выпуска непосредственно в `main`.
5. Обновите версию во всех обязательных местах для тега, затем выполните `pnpm release:prep`. Команда последовательно обновляет версии плагинов, shrinkwrap-файлы npm, перечень плагинов, базовую схему конфигурации, метаданные конфигурации встроенных каналов, базовую версию документации по конфигурации, экспорты SDK плагинов и базовую версию API SDK плагинов. Перед созданием тега зафиксируйте в коммите все изменения сгенерированных файлов, затем выполните локальную детерминированную предварительную проверку: `pnpm check:test-types`, `pnpm check:architecture`, `pnpm build && pnpm ui:build` и `pnpm release:check`.
6. Запустите `OpenClaw NPM Release` с `preflight_only=true`. До создания тега для предварительной проверки, выполняемой только в целях валидации, допускается полный 40-символьный SHA ветки выпуска. Предварительная проверка формирует подтверждение выпуска зависимостей для точного графа зависимостей извлечённой версии и сохраняет его в артефакте предварительной проверки npm. Сохраните идентификатор успешного запуска `preflight_run_id`.
7. Запустите все предварительные тесты выпуска через `Full Release Validation` для ветки выпуска, тега или полного SHA коммита. Это единая ручная точка входа для четырёх крупных наборов тестов выпуска: Vitest, Docker, QA Lab и Package. Сохраните `full_release_validation_run_id` и точное значение `full_release_validation_run_attempt`; оба параметра обязательны для `OpenClaw NPM Release` и `OpenClaw Release Publish`.
8. Если валидация завершается с ошибкой, исправьте проблему в ветке выпуска и повторно запустите минимальный неуспешный файл, этап, задание рабочего процесса, профиль пакета, провайдера или список разрешённых моделей, достаточный для подтверждения исправления. Повторно запускайте весь общий процесс, только если изменённая область делает предыдущие подтверждения устаревшими.
9. Для помеченного тегом кандидата на бета-выпуск выполните `pnpm release:candidate -- --tag vYYYY.M.PATCH-beta.N` из соответствующей ветки `release/YYYY.M.PATCH`. Для стабильного выпуска также передайте обязательный исходный выпуск Windows: `pnpm release:candidate -- --tag vYYYY.M.PATCH --windows-node-tag vX.Y.Z`. Вспомогательное средство использует доверенную ветку `main` как источник рабочего процесса, при этом каждый рабочий процесс нацелен на точный тег. Оно сохраняет контрольную точку с неизменяемыми идентификаторами кандидата и инструментария, а также идентификаторами запущенных процессов в `.artifacts/release-candidate/<tag>/release-candidate-state.json`; повторный запуск той же команды возобновляет именно эти запуски, а любое расхождение кандидата, инструментария, профиля или параметров приводит к безопасному отказу. Перед запуском полной матрицы валидации вспомогательное средство детерминированно формирует точное описание выпуска GitHub для указанного тега и отклоняет отсутствие заголовка версии, превышающий ограничение размер описания, для которого нельзя применить каноническую компактную форму, либо происхождение базовой и целевой версий записи о вкладе, недостижимое из тега. Оно также проверяет все явно заданные метаданные исключения базовой линии выпущенных версий по указанным накопительным записям тегов. Затем оно запускает локальные проверки сгенерированных материалов выпуска, запускает или проверяет подтверждения полной валидации выпуска и предварительной проверки npm, выполняет в Parallels проверку чистой установки и обновления по точному подготовленному tarball-файлу вместе с проверкой пакета Telegram, фиксирует планы публикации плагинов в npm и ClawHub и выводит точную команду `OpenClaw Release Publish` только после успешной проверки всего набора подтверждений.

   `OpenClaw Release Publish` параллельно отправляет выбранные или все доступные для публикации пакеты плагинов в npm и тот же набор в ClawHub, а затем, после успешной публикации плагинов в npm, продвигает подготовленный артефакт предварительной проверки OpenClaw для npm с соответствующим dist-тегом. Извлечённая версия выпуска остаётся корнем продукта и данных, а планирование и итоговая проверка выполняются из точной доверенной извлечённой версии — источника рабочего процесса, чтобы старый коммит выпуска не мог незаметно использовать устаревший инструментарий выпуска. Перед запуском любого дочернего процесса формируется и кэшируется точное описание выпуска GitHub. Если полный соответствующий раздел `CHANGELOG.md` укладывается в ограничение GitHub в 125 000 символов и соответствующий защитный предел средства формирования в 125 000 байт, страница содержит точный раздел `## YYYY.M.PATCH`, включая его заголовок. Если исходный раздел не помещается, на странице сохраняются точные сгруппированные редакционные примечания, а чрезмерно большая запись о вкладе заменяется стабильной ссылкой на полную запись в привязанном к тегу `CHANGELOG.md`; частичные записи и усечённые пункты никогда не публикуются. Рабочий процесс выбирает полное или компактное описание до добавления `### Проверка выпуска`; если итоговая часть с подтверждениями приведёт к превышению ограничения, он сохраняет каноническое описание и использует вместо неё неизменяемое прикреплённое подтверждение. Стабильные выпуски, опубликованные в npm с тегом `latest`, становятся последним выпуском GitHub, а стабильные сопровождаемые выпуски, оставленные в npm с тегом `beta`, создаются в GitHub с `latest=false`. Рабочий процесс также загружает в выпуск GitHub подтверждение зависимостей из предварительной проверки, манифест полной валидации и подтверждение проверки реестра после публикации для реагирования на инциденты после выпуска. Он немедленно выводит идентификаторы дочерних запусков, автоматически одобряет шлюзы среды выпуска, которые разрешено одобрять токену рабочего процесса, формирует сводку неуспешных дочерних заданий с заключительными фрагментами журналов, заранее создаёт черновик страницы выпуска GitHub и параллельно с публикацией OpenClaw в npm продвигает ресурсы Windows и Android, завершает оформление страницы выпуска и подтверждения зависимостей после успешного завершения этих этапов, ожидает ClawHub, если публикуется OpenClaw в npm, затем запускает средство проверки бета-версии из доверенной ветки `main` и загружает подтверждение после публикации для выпуска GitHub, пакета npm, выбранных пакетов плагинов npm, выбранных пакетов ClawHub, идентификаторов запусков дочерних рабочих процессов и необязательного идентификатора запуска NPM Telegram. Средство проверки начальной загрузки ClawHub требует точные путь и SHA рабочего процесса доверенной ветки `main`, номера попыток запуска производителя и завершающего процесса, SHA выпуска, запрошенный набор пакетов, неизменяемый кортеж артефакта пакета и артефакт итогового чтения из реестра; успешный устаревший запуск по ссылке выпуска не принимается.

   Затем выполните приёмочное тестирование пакета после публикации для опубликованного пакета `openclaw@YYYY.M.PATCH-beta.N` или `openclaw@beta`. Если отправленная или опубликованная предварительная версия требует исправления, создайте следующую предварительную версию с соответствующим номером; никогда не удаляйте и не перезаписывайте предыдущую.

10. Для стабильного выпуска продолжайте только после того, как проверенная бета-версия или кандидат на выпуск получит обязательные подтверждения валидации. Публикация стабильной версии в npm также выполняется через `OpenClaw Release Publish` с повторным использованием успешного артефакта предварительной проверки посредством `preflight_run_id`. Готовность стабильного выпуска macOS также требует наличия упакованных файлов `.zip`, `.dmg`, `.dSYM.zip` и обновлённого `appcast.xml` в `main`; рабочий процесс публикации macOS автоматически публикует подписанный appcast в общедоступную ветку `main` после проверки ресурсов выпуска либо открывает или обновляет PR для appcast, если защита ветки блокирует прямую отправку. Для готовности стабильного Windows Hub требуются подписанные ресурсы `OpenClawCompanion-Setup-x64.exe`, `OpenClawCompanion-Setup-arm64.exe` и `OpenClawCompanion-SHA256SUMS.txt` в выпуске OpenClaw на GitHub. Передайте точный тег подписанного выпуска `openclaw/openclaw-windows-node` как `windows_node_tag`, а одобренную кандидатом карту хешей установщиков — как `windows_node_installer_digests`; `OpenClaw Release Publish` сохраняет выпуск в состоянии черновика, запускает `Windows Node Release` и проверяет все три ресурса перед публикацией.
11. После публикации запустите средство проверки npm после публикации, при необходимости — отдельный сквозной тест Telegram для опубликованного npm-пакета, если требуется подтверждение работы канала после публикации, выполните продвижение dist-тега, если оно необходимо, проверьте сгенерированную страницу выпуска GitHub, выполните шаги объявления о выпуске, а затем завершите процедуру [Завершение стабильного выпуска в main](#stable-main-closeout), прежде чем считать стабильный выпуск завершённым.

## Завершение стабильного выпуска в main

Публикация стабильной версии не завершена, пока ветка `main` не содержит фактическое состояние выпущенной версии.

1. Начните с новой актуальной версии `main`. Сопоставьте `release/YYYY.M.PATCH` с ней и перенесите вперёд реальные исправления, отсутствующие в `main`. Не объединяйте вслепую предназначенные только для выпуска адаптеры совместимости, тестирования или валидации с более новой веткой `main`.
2. Установите в `main` выпущенную стабильную версию, а не предполагаемую версию следующего цикла. После изменения корневой версии выполните `pnpm release:prep`, затем `pnpm deps:shrinkwrap:generate`.
3. Сделайте раздел `## YYYY.M.PATCH` файла `CHANGELOG.md` в `main` полностью идентичным разделу в ветке помеченного тегом выпуска. Включите обновление стабильного `appcast.xml`, если оно было опубликовано в выпуске для Mac.
4. Не добавляйте в `main` версию `YYYY.M.PATCH+1`, бета-версию или пустой раздел будущего журнала изменений, пока оператор явно не начнёт соответствующий цикл выпуска.
5. Выполните `pnpm release:generated:check`, `pnpm deps:shrinkwrap:check` и `OPENCLAW_TESTBOX=1 pnpm check:changed`. Отправьте изменения, затем убедитесь, что `origin/main` содержит выпущенную версию и журнал изменений, прежде чем считать стабильный выпуск завершённым.
6. Поддерживайте переменные репозитория `RELEASE_ROLLBACK_DRILL_ID` и `RELEASE_ROLLBACK_DRILL_DATE` в актуальном состоянии после каждой закрытой проверки отката.

`OpenClaw Stable Main Closeout` запускается после отправки в `main` выпущенной версии, журнала изменений и appcast по завершении публикации стабильного выпуска. Процесс читает неизменяемое подтверждение после публикации, чтобы связать выпущенный тег с соответствующими запусками Full Release Validation и Publish, а затем проверяет состояние стабильной версии в `main`, выпуск, обязательный период наблюдения за стабильной версией и блокирующие подтверждения производительности. Он прикрепляет к выпуску GitHub неизменяемый манифест завершения и его контрольную сумму. Автоматический триггер отправки пропускает устаревшие выпуски, созданные до появления неизменяемого подтверждения после публикации, и никогда не считает такой пропуск завершённой процедурой.

Для полного завершения требуются оба ресурса и соответствующая контрольная сумма. При наличии частичного манифеста процесс повторяет записанные в нём SHA `main` и проверку отката, чтобы повторно сформировать идентичные байты, а затем прикрепляет отсутствующую контрольную сумму; недействительная пара или контрольная сумма без манифеста остаётся блокирующим условием. Запуск по триггеру отправки без переменных репозитория для проверки отката завершается пропуском, не завершая процедуру; отсутствующая или созданная более 90 дней назад запись о проверке по-прежнему блокирует ручное завершение на основе подтверждений. Закрытые команды восстановления остаются в доступном только сопровождающим руководстве. Используйте ручной запуск только для исправления или повторного выполнения подтверждённого стабильного завершения.

Устаревший резервный корректирующий тег может повторно использовать подтверждение базового пакета, только если корректирующий тег разрешается в тот же исходный коммит, что и базовый стабильный тег. Его выпуск Android повторно использует проверенный APK базового тега и добавляет данные о происхождении для корректирующего тега. Корректирующий выпуск с другим исходным кодом должен опубликовать и проверить собственное подтверждение пакета и использовать большее значение Android `versionCode`.

## Предварительная проверка выпуска

- Запустите `pnpm check:test-types` перед предварительной проверкой релиза, чтобы тестовый код TypeScript оставался охвачен проверками за пределами более быстрого локального этапа `pnpm check`.
- Запустите `pnpm check:architecture` перед предварительной проверкой релиза, чтобы расширенные проверки циклических импортов и архитектурных границ успешно проходили за пределами более быстрого локального этапа.
- Запустите `pnpm build && pnpm ui:build` перед `pnpm release:check`, чтобы ожидаемые артефакты релиза `dist/*` и пакет Control UI существовали для этапа проверки пакета.
- Запустите `pnpm release:prep` после повышения версии в корне и перед созданием тега. Эта команда запускает все детерминированные генераторы релиза, результаты которых часто рассинхронизируются после изменения версии, конфигурации или API: версии плагинов, файлы фиксации зависимостей npm, реестр плагинов, базовую схему конфигурации, метаданные конфигурации встроенных каналов, базовый вариант документации по конфигурации, экспорты SDK плагинов и базовый вариант API SDK плагинов. `pnpm release:check` повторно запускает эти проверки в режиме проверки (а также проверку бюджета поверхности SDK плагинов) и за один проход сообщает обо всех ошибках рассинхронизации сгенерированных данных, прежде чем запускать проверки выпуска пакета.
- Синхронизация версий плагинов по умолчанию обновляет публикуемый пакет среды выполнения `@openclaw/ai`, версии пакетов официальных плагинов и существующие минимальные версии `openclaw.compat.pluginApi` до версии релиза OpenClaw. Считайте это поле минимальной версией API SDK/среды выполнения плагинов, а не просто копией версии пакета: для релизов только плагинов, которые намеренно сохраняют совместимость со старыми хостами OpenClaw, оставляйте минимальную версию на уровне самого старого поддерживаемого API хоста и документируйте этот выбор в подтверждении релиза плагина.
- Запустите вручную рабочий процесс `Full Release Validation` перед одобрением релиза, чтобы инициировать все наборы предрелизных тестов из одной точки входа. Он принимает ветку, тег или полный SHA коммита, запускает вручную `CI` и запускает `OpenClaw Release Checks` для быстрой проверки установки, приемочного тестирования пакета, межплатформенных проверок пакета, проверки соответствия QA Lab, а также направлений Matrix и Telegram. Стабильные и полные запуски всегда включают исчерпывающие проверки в реальной среде/E2E и длительное тестирование пути релиза в Docker; `run_release_soak=true` сохраняется для явно заданного длительного тестирования бета-версии. Приемочное тестирование пакета обеспечивает каноническое E2E-тестирование Telegram для пакета во время проверки кандидата, исключая второй параллельный процесс опроса реальной среды.

  Укажите `release_package_spec` после публикации бета-версии, чтобы повторно использовать выпущенный пакет npm в проверках релиза, приемочном тестировании пакета и E2E-тестировании Telegram для пакета без повторной сборки tar-архива релиза. Указывайте `npm_telegram_package_spec` только в том случае, если для Telegram следует использовать опубликованный пакет, отличный от пакета для остальной проверки релиза. Укажите `package_acceptance_package_spec`, если приемочное тестирование пакета должно использовать опубликованный пакет, отличный от спецификации пакета релиза. Укажите `evidence_package_spec`, если отчет с подтверждениями релиза должен доказывать, что проверка соответствует опубликованному пакету npm, не требуя запуска E2E-тестирования Telegram.

  ```bash
  gh workflow run full-release-validation.yml --ref main -f ref=release/YYYY.M.PATCH
  ```

- Запустите вручную рабочий процесс `Package Acceptance`, если вам нужно получить дополнительное подтверждение для кандидата пакета, пока продолжается работа над релизом. Используйте `source=npm` для `openclaw@beta`, `openclaw@latest` или точной версии релиза; `source=ref` — чтобы упаковать доверенную ветку, тег или SHA `package_ref` с текущим набором тестов `workflow_ref`; `source=url` — для общедоступного tar-архива по HTTPS с обязательной контрольной суммой SHA-256 и строгой политикой общедоступных URL; `source=trusted-url` — для именованной политики доверенного источника с обязательными `trusted_source_id` и SHA-256; либо `source=artifact` — для tar-архива, загруженного другим запуском GitHub Actions.

  Рабочий процесс преобразует кандидата в `package-under-test`, повторно использует планировщик E2E-тестирования релиза в Docker для этого tar-архива и может запускать QA Telegram для того же tar-архива с `telegram_mode=mock-openai` или `telegram_mode=live-frontier`. Если выбранные направления Docker включают `published-upgrade-survivor`, артефакт пакета является кандидатом, а `published_upgrade_survivor_baseline` выбирает опубликованную базовую версию. `update-restart-auth` использует пакет-кандидат одновременно как установленный CLI и как тестируемый пакет, чтобы проверить управляемый путь перезапуска команды обновления кандидата.

  Пример:

  ```bash
  gh workflow run package-acceptance.yml --ref main -f workflow_ref=main -f source=npm -f package_spec=openclaw@beta -f suite_profile=product -f published_upgrade_survivor_baseline=openclaw@2026.4.26 -f telegram_mode=mock-openai
  ```

  Распространенные профили:
  - `smoke`: направления установки/канала/агента, сети Gateway и перезагрузки конфигурации
  - `package`: ориентированные на артефакты направления пакета/обновления/перезапуска/плагинов без OpenWebUI или реального ClawHub
  - `product`: профиль пакета плюс каналы MCP, очистка Cron/подагентов, веб-поиск OpenAI и OpenWebUI
  - `full`: части пути релиза в Docker с OpenWebUI
  - `custom`: точный выбор `docker_lanes` для целевого повторного запуска

- Запустите вручную рабочий процесс `CI` напрямую, если вам нужно только детерминированное стандартное покрытие CI для кандидата релиза. Ручные запуски CI обходят ограничение областью изменений и принудительно запускают сегменты Linux Node, сегменты встроенных плагинов, сегменты контрактов плагинов и каналов, совместимость с Node 22, `check-*`, `check-additional-*`, быстрые проверки собранных артефактов, проверки документации, Skills Python, Windows, macOS и направления интернационализации Control UI. Отдельные ручные запуски CI выполняют Android только при запуске с `include_android=true`; `Full Release Validation` передает этот параметр своему дочернему процессу CI.

  ```bash
  gh workflow run ci.yml --ref release/YYYY.M.PATCH -f include_android=true
  ```

- Запустите `pnpm qa:otel:smoke` при проверке телеметрии релиза. Команда проверяет QA Lab через локальный приемник OTLP/HTTP и подтверждает экспорт трассировок, метрик и журналов, а также ограниченность атрибутов трассировки и сокрытие содержимого/идентификаторов без необходимости использовать Opik, Langfuse или другой внешний сборщик.
- Запустите `pnpm qa:otel:collector-smoke` при проверке совместимости со сборщиком. Команда направляет тот же экспорт OTLP из QA Lab через настоящий контейнер Docker с OpenTelemetry Collector перед выполнением проверок локального приемника.
- Запустите `pnpm qa:prometheus:smoke` при проверке защищенного сбора данных Prometheus. Команда проверяет QA Lab, отклоняет запросы сбора без аутентификации и подтверждает, что критически важные для релиза семейства метрик не содержат текст запросов, необработанные идентификаторы, токены аутентификации и локальные пути.
- Запустите `pnpm qa:observability:smoke`, чтобы последовательно выполнить направления быстрых проверок OpenTelemetry и Prometheus для исходного дерева.
- Запускайте `pnpm release:check` перед каждым релизом с тегом.
- Предварительная проверка `OpenClaw NPM Release` создает подтверждения зависимостей релиза перед упаковкой tar-архива npm. Шлюз проверки уязвимостей из бюллетеней npm блокирует релиз при ошибке. Отчеты о рисках транзитивного манифеста, владельцах зависимостей/поверхности установки и изменениях зависимостей служат только подтверждениями релиза. Отчет об изменениях зависимостей сравнивает кандидата релиза с предыдущим доступным тегом релиза. Предварительная проверка загружает подтверждения зависимостей как `openclaw-release-dependency-evidence-<tag>` и также включает их в каталог `dependency-evidence/` внутри подготовленного артефакта предварительной проверки npm. Фактический путь публикации повторно использует этот артефакт предварительной проверки, а затем прикрепляет те же подтверждения к релизу GitHub как `openclaw-<version>-dependency-evidence.zip`.
- Запустите `OpenClaw Release Publish` для последовательности публикации с изменениями после создания тега. Запускайте обычные публикации бета- и стабильных версий из доверенной ветки `main`; тег релиза по-прежнему выбирает точный целевой коммит и может указывать на `release/YYYY.M.PATCH`. Публикации альфа-версий Tideclaw остаются в соответствующих альфа-ветках. Передайте идентификатор успешного запуска предварительной проверки npm OpenClaw `preflight_run_id`, идентификатор успешного запуска `full_release_validation_run_id` и точный номер попытки `full_release_validation_run_attempt`, а также сохраняйте область публикации плагинов по умолчанию `all-publishable`, если вы намеренно не выполняете целевое исправление. Рабочий процесс последовательно выполняет публикацию плагинов в npm, публикацию плагинов в ClawHub и публикацию OpenClaw в npm, чтобы основной пакет не был опубликован раньше вынесенных из него плагинов; продвижение Windows и Android выполняется параллельно с публикацией основного пакета npm относительно черновика страницы релиза. Повторные запуски публикации можно продолжать: если основная версия npm уже опубликована, основной запуск пропускается после того, как рабочий процесс подтвердит соответствие tar-архива в реестре артефакту предварительной проверки для тега, а продвижение Windows/Android пропускается, если релиз уже содержит проверенный контракт ресурсов, поэтому при повторной попытке заново выполняются только завершившиеся с ошибкой этапы. Для целевых исправлений только плагинов требуются `plugin_publish_scope=selected` и непустой список плагинов. Для запусков только плагинов с `all-publishable` требуются полные неизменяемые подтверждения предварительной проверки и Full Release Validation; частичные подтверждения отклоняются.
- Для стабильного запуска `OpenClaw Release Publish` требуется точный `windows_node_tag` после появления соответствующего релиза без признака предварительной версии в `openclaw/openclaw-windows-node`, а также одобренная для кандидата карта `windows_node_installer_digests`. Перед запуском любого дочернего процесса публикации проверяется, что исходный релиз опубликован, не является предварительным, содержит необходимые установщики x64/ARM64 и по-прежнему соответствует этой одобренной карте. Затем запускается `Windows Node Release`, пока релиз OpenClaw еще остается черновиком, причем закрепленная карта контрольных сумм установщиков передается без изменений. Дочерний рабочий процесс загружает подписанные установщики Windows Hub из этого точного тега, сверяет их с закрепленными контрольными суммами, проверяет на исполнителе Windows, что их подписи Authenticode используют ожидаемого подписанта OpenClaw Foundation, создает манифест SHA-256 и загружает установщики вместе с манифестом в канонический релиз OpenClaw на GitHub, после чего повторно загружает продвинутые ресурсы и проверяет их наличие в манифесте и контрольные суммы. Родительский процесс проверяет текущий контракт ресурсов x64, ARM64 и контрольных сумм перед публикацией. Прямое восстановление отклоняет неожиданные имена ресурсов `OpenClawCompanion-*`, прежде чем заменить ожидаемые ресурсы контракта закрепленными байтами источника.

  Запускайте `Windows Node Release` вручную только для восстановления и всегда передавайте точный тег, но никогда не `latest`, а также явную JSON-карту `expected_installer_digests` из одобренного исходного релиза. Ссылки для скачивания на веб-сайте должны указывать на точные URL ресурсов текущего стабильного релиза OpenClaw или на `releases/latest/download/...` только после проверки, что перенаправление GitHub для последней версии указывает на тот же релиз; не давайте ссылку только на страницу релиза сопутствующего репозитория.

- Теперь проверки релиза выполняются в отдельном запускаемом вручную рабочем процессе: `OpenClaw Release Checks`. Перед одобрением релиза он также запускает контур проверки соответствия имитаций QA Lab, быстрый профиль живых проверок Matrix и контур QA для Telegram. Контуры живых проверок используют окружение `qa-live-shared`; Telegram также использует аренду учетных данных Convex CI. Запустите вручную рабочий процесс `QA-Lab - All Lanes` с `matrix_profile=all` и `matrix_shards=true`, если требуется параллельно выполнить полный набор проверок транспорта Matrix, мультимедиа и E2EE.
- Межплатформенная проверка установки, обновления и среды выполнения входит в общедоступные процессы `OpenClaw Release Checks` и `Full Release Validation`, которые напрямую вызывают повторно используемый рабочий процесс `.github/workflows/openclaw-cross-os-release-checks-reusable.yml`. Такое разделение сделано намеренно: реальный путь выпуска в npm остается коротким, детерминированным и ориентированным на артефакты, а более медленные живые проверки выполняются в отдельном контуре, чтобы не задерживать и не блокировать публикацию.
- Проверки релиза, использующие секреты, следует запускать через `Full Release Validation` либо из ссылки на рабочий процесс `main`/релиза, чтобы логика рабочего процесса и секреты оставались под контролем.
- `OpenClaw Release Checks` принимает ветку, тег или полный SHA коммита при условии, что разрешенный коммит достижим из ветки OpenClaw или тега релиза.
- Предварительная проверка `OpenClaw NPM Release`, выполняющая только валидацию, также принимает текущий полный 40-символьный SHA коммита ветки рабочего процесса без обязательного отправленного тега. Путь по SHA предназначен только для валидации и не может быть переведен в реальную публикацию. В режиме SHA рабочий процесс формирует `v<package.json version>` только для проверки метаданных пакета; для реальной публикации по-прежнему требуется настоящий тег релиза.
- Оба рабочих процесса выполняют реальную публикацию и продвижение на раннерах GitHub, а неизменяющий данные путь валидации может использовать более производительные Linux-раннеры Blacksmith.
- Этот рабочий процесс запускает `OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache`, используя секреты рабочего процесса `OPENAI_API_KEY` и `ANTHROPIC_API_KEY`.
- Предварительная проверка релиза npm больше не ожидает завершения отдельного контура проверок релиза.
- Перед локальным созданием тега кандидата на выпуск выполните `RELEASE_TAG=vYYYY.M.PATCH-beta.N pnpm release:fast-pretag-check`. Вспомогательная команда последовательно запускает быстрые защитные проверки релиза, проверки выпуска Plugin в npm/ClawHub, сборку, сборку пользовательского интерфейса и `release:openclaw:npm:check`, чтобы выявить распространенные ошибки, блокирующие одобрение, до запуска рабочего процесса публикации GitHub.
- Перед одобрением выполните `RELEASE_TAG=vYYYY.M.PATCH node --import tsx scripts/openclaw-npm-release-check.ts` (либо команду с соответствующим тегом предварительного или корректирующего выпуска).
- После публикации в npm выполните `node --import tsx scripts/openclaw-npm-postpublish-verify.ts YYYY.M.PATCH` (либо команду с соответствующей бета-версией или корректирующей версией), чтобы проверить путь установки из опубликованного реестра в новом временном префиксе.
- После публикации бета-версии выполните `OPENCLAW_NPM_TELEGRAM_PACKAGE_SPEC=openclaw@YYYY.M.PATCH-beta.N OPENCLAW_NPM_TELEGRAM_CREDENTIAL_SOURCE=convex OPENCLAW_NPM_TELEGRAM_CREDENTIAL_ROLE=ci pnpm test:docker:npm-telegram-live`, чтобы проверить первоначальную настройку установленного пакета, настройку Telegram и реальное сквозное тестирование Telegram для опубликованного пакета npm с использованием общего пула арендуемых учетных данных Telegram. При разовых локальных запусках сопровождающие могут не указывать переменные Convex и передавать три учетных данных окружения `OPENCLAW_QA_TELEGRAM_*` напрямую.
- Чтобы выполнить полную проверку бета-версии после публикации с компьютера сопровождающего, используйте `pnpm release:beta-smoke -- --beta betaN`. Вспомогательная команда запускает проверку обновления npm и новой целевой системы в Parallels, запускает `NPM Telegram Beta E2E`, отслеживает конкретный запуск рабочего процесса, загружает артефакт и выводит отчет Telegram.
- Сопровождающие могут выполнить ту же проверку после публикации через GitHub Actions, используя запускаемый вручную рабочий процесс `NPM Telegram Beta E2E`. Он намеренно запускается только вручную и не выполняется при каждом слиянии.
- Автоматизация релиза для сопровождающих использует схему «предварительная проверка, затем продвижение»:
  - Для реальной публикации в npm требуется успешный `preflight_run_id` предварительной проверки npm.
  - Оркестрация и предварительная проверка публикации обычных бета- и стабильных версий используют доверенную ветку `main` для точного целевого тега. Публикация и предварительная проверка альфа-версии Tideclaw используют соответствующую альфа-ветку.
  - Стабильные релизы npm по умолчанию получают тег `beta`; при публикации стабильного релиза npm можно явно выбрать `latest` через входной параметр рабочего процесса.
  - Изменение dist-тегов npm на основе токена находится в `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml`, поскольку для `npm dist-tag add` по-прежнему требуется `NPM_TOKEN`, а исходный репозиторий использует для публикации только OIDC.
  - Общедоступный процесс `macOS Release` выполняет только валидацию; если тег существует только в ветке релиза, а рабочий процесс запускается из `main`, задайте `public_release_branch=release/YYYY.M.PATCH`.
  - Для реальной публикации macOS требуются успешные `preflight_run_id` и `validate_run_id` macOS.
  - Пути реальной публикации продвигают подготовленные артефакты, а не собирают их повторно.
- Для стабильных корректирующих выпусков наподобие `YYYY.M.PATCH-N` средство проверки после публикации также проверяет тот же путь обновления во временном префиксе с `YYYY.M.PATCH` до `YYYY.M.PATCH-N`, чтобы корректировки релиза не могли незаметно оставить старые глобальные установки на базовом стабильном содержимом.
- Предварительная проверка релиза npm завершается с ошибкой, если tar-архив не содержит одновременно `dist/control-ui/index.html` и непустое содержимое `dist/control-ui/assets/`, чтобы снова не выпустить пустую браузерную панель управления.
- Проверка после публикации также контролирует наличие точек входа опубликованных Plugin и метаданных пакета в структуре установленного пакета из реестра. Релиз без необходимого содержимого среды выполнения Plugin не проходит средство проверки после публикации и не может быть продвинут до `latest`.
- `pnpm test:install:smoke` также проверяет ограничение npm pack `unpackedSize` для tar-архива кандидата на обновление, поэтому сквозная проверка установщика обнаруживает случайное раздувание пакета до начала публикации релиза.
- Если работа над релизом затрагивала планирование CI, манифесты длительности выполнения расширений или матрицы тестов расширений, перед одобрением заново создайте и проверьте управляемые планировщиком выходные данные матрицы `plugin-prerelease-extension-shard` из `.github/workflows/plugin-prerelease.yml`, чтобы примечания к релизу не описывали устаревшую структуру CI.
- Готовность стабильного релиза macOS также охватывает поверхности средства обновления: релиз GitHub должен в итоге содержать упакованные файлы `.zip`, `.dmg` и `.dSYM.zip`; файл `appcast.xml` в `main` после публикации должен указывать на новый стабильный zip-архив (рабочий процесс публикации macOS фиксирует это изменение автоматически либо открывает запрос на включение изменений для appcast, если прямая отправка заблокирована); упакованное приложение должно сохранять идентификатор пакета не для отладки, непустой URL-адрес канала Sparkle и `CFBundleVersion`, не ниже канонического минимального номера сборки Sparkle для этой версии релиза.

## Среды тестирования релиза

`Full Release Validation` — единая точка запуска всех предрелизных тестов операторами. Для подтверждения конкретного коммита в быстро меняющейся ветке используйте вспомогательную команду, чтобы каждый дочерний рабочий процесс запускался из временной ветки, зафиксированной на одном доверенном SHA рабочего процесса `main`, а запрошенный коммит оставался проверяемым кандидатом:

```bash
pnpm ci:full-release --sha <full-sha>
```

Вспомогательная команда получает текущую `origin/main`, отправляет ветку `release-ci/<workflow-sha>-...` на уровне этого доверенного коммита рабочего процесса, запускает `Full Release Validation` из временной ветки с `ref=<target-sha>`, при наличии повторно использует строгие доказательства для точной целевой версии, проверяет соответствие `headSha` каждого дочернего рабочего процесса закрепленному SHA родительского рабочего процесса, а затем удаляет временную ветку. Передайте `-f reuse_evidence=false`, чтобы принудительно выполнить новый запуск, либо `--workflow-sha <trusted-main-sha>`, чтобы закрепить более старый коммит, который все еще достижим из текущей `origin/main`. Сам рабочий процесс никогда не изменяет ссылки репозитория. Это обеспечивает доступность инструментария релиза только из `main` без добавления служебных коммитов в кандидата и предотвращает случайное подтверждение более нового дочернего запуска из `main`.

Для валидации ветки или тега релиза запускайте процесс из доверенной ссылки рабочего процесса `main` и передавайте ветку или тег релиза в качестве `ref`:

```bash
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=release/YYYY.M.PATCH \
  -f provider=openai \
  -f mode=both \
  -f release_profile=stable \
  -f evidence_package_spec=openclaw@YYYY.M.PATCH-beta.N
```

Рабочий процесс разрешает целевую ссылку, запускает вручную `CI` с `target_ref=<release-ref>`, а затем запускает `OpenClaw Release Checks`. `OpenClaw Release Checks` параллельно запускает быструю проверку установки, межплатформенные проверки релиза, живые и сквозные Docker-проверки пути релиза при включенной длительной проверке, приемку пакета с каноническим сквозным тестированием пакета Telegram, проверку соответствия QA Lab, живые проверки Matrix и живые проверки Telegram. Полный запуск со всеми проверками считается приемлемым, только если в сводке `Full Release Validation` процессы `normal_ci`, `plugin_prerelease` и `release_checks` отмечены как успешные, кроме случаев, когда при целевом повторном запуске намеренно пропущен отдельный дочерний процесс `Plugin Prerelease`. Используйте отдельный дочерний процесс `npm-telegram` только для целевого повторного запуска опубликованного пакета с `release_package_spec` или `npm_telegram_package_spec`. Итоговая сводка средства проверки содержит таблицы самых медленных заданий для каждого дочернего запуска, чтобы руководитель релиза мог видеть текущий критический путь без загрузки журналов.

Дочерний процесс проверки производительности продукта в этом пути релиза создает только артефакты. Объединяющий процесс запускает его с `publish_reports=false`, а валидация отклоняется, если защитная проверка режима создания только артефактов не подтверждает, что средство публикации отчетов Clawgrit осталось пропущенным.

Полную матрицу этапов, точные имена заданий рабочих процессов, различия между стабильным и полным профилями, артефакты и параметры целевого повторного запуска см. в разделе [Полная валидация релиза](/ru/reference/full-release-validation).

Дочерние рабочие процессы запускаются из доверенной ссылки, выполняющей `Full Release Validation`, обычно `--ref main`, даже если целевой `ref` указывает на более старую ветку или тег релиза. Каждый дочерний запуск должен использовать точный SHA родительского рабочего процесса; если `main` продвинется до разрешения запуска дочернего процесса, объединяющий процесс завершится с ошибкой. Отдельного входного параметра ссылки рабочего процесса Full Release Validation нет; выбирайте доверенную среду, задавая ссылку запуска рабочего процесса. Не используйте `--ref main -f ref=<sha>` для подтверждения точного коммита в изменяющейся `main`: необработанные SHA коммитов нельзя использовать как ссылки запуска рабочего процесса, поэтому применяйте `pnpm ci:full-release --sha <target-sha>`, чтобы создать временную ветку на доверенной `origin/main`, сохранив целевой SHA в качестве входных данных кандидата.

Используйте `release_profile`, чтобы выбрать охват живых проверок и поставщиков:

- `minimum`: самый быстрый критически важный для релиза путь живых и Docker-проверок OpenAI/ядра
- `stable`: минимальный профиль плюс охват стабильных поставщиков и серверных компонентов для одобрения релиза
- `full`: стабильный профиль плюс широкий рекомендательный охват поставщиков и мультимедиа

При стабильной и полной валидации перед продвижением всегда выполняются исчерпывающие живые и сквозные проверки, Docker-проверки пути релиза и ограниченный полный набор проверок сохранения работоспособности обновлений опубликованных версий. Используйте `run_release_soak=true`, чтобы запросить тот же набор для бета-версии. Он охватывает четыре последних стабильных пакета, закрепленные базовые версии `2026.4.23` и `2026.5.2`, а также более старую версию `2026.4.15`; повторяющиеся базовые версии удаляются, а каждая оставшаяся версия выполняется в отдельном сегментированном задании Docker-раннера.

`OpenClaw Release Checks` использует доверенную ссылку рабочего процесса, чтобы один раз разрешить целевую ссылку в виде `release-package-under-test`, и повторно использует этот артефакт в межплатформенных проверках, приемке пакета и Docker-проверках пути релиза при выполнении длительной проверки. Благодаря этому все среды, работающие с пакетом, используют одни и те же байты, а повторные сборки пакета исключаются. После публикации бета-версии в npm задайте `release_package_spec=openclaw@YYYY.M.PATCH-beta.N`, чтобы проверки релиза один раз загрузили выпущенный пакет, извлекли SHA исходного кода сборки из `dist/build-info.json` и повторно использовали этот артефакт для межплатформенных проверок, приемки пакета, Docker-проверок пути релиза и контуров проверки пакета Telegram.

Межплатформенная быстрая проверка установки OpenAI использует `OPENCLAW_CROSS_OS_OPENAI_MODEL`, если задана переменная репозитория или организации; в противном случае используется `openai/gpt-5.6-luna`, поскольку этот контур подтверждает установку пакета, первоначальную настройку, запуск Gateway и один живой ход агента, а не измеряет производительность самой функциональной модели. Более широкая матрица живых проверок поставщиков остается местом для охвата конкретных моделей.

Используйте следующие варианты в зависимости от этапа релиза:

```bash
# Validate an unpublished release candidate branch.
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=release/YYYY.M.PATCH \
  -f provider=openai \
  -f mode=both \
  -f release_profile=stable

# Validate an exact pushed commit.
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=<40-char-sha> \
  -f provider=openai \
  -f mode=both

# After publishing a beta, add published-package Telegram E2E.
gh workflow run full-release-validation.yml \
  --ref main \
  -f ref=release/YYYY.M.PATCH \
  -f provider=openai \
  -f mode=both \
  -f release_profile=full \
  -f release_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f evidence_package_spec=openclaw@YYYY.M.PATCH-beta.N \
  -f npm_telegram_provider_mode=mock-openai
```

Не используйте полный объединяющий процесс для первого повторного запуска после точечного исправления. Если завершается сбоем один блок, для следующего подтверждения используйте завершившийся сбоем дочерний рабочий процесс, задание, канал Docker, профиль пакета, поставщика модели или канал контроля качества. Повторно запускайте полный объединяющий процесс, только если исправление изменило общую оркестрацию выпуска или сделало устаревшими ранее собранные подтверждения по всем блокам. Итоговый проверяющий этап объединяющего процесса повторно проверяет записанные идентификаторы запусков дочерних рабочих процессов, поэтому после успешного повторного запуска дочернего рабочего процесса перезапустите только завершившееся сбоем родительское задание `Verify full validation`.

`rerun_group=all` может повторно использовать предыдущий успешный запуск объединяющего процесса, только если он проверял в точности тот же целевой SHA, профиль выпуска, фактическую настройку выдержки и входные данные проверки. Это ограниченное восстановление для повторного запуска того же кандидата, а не повторное использование подтверждений между разными SHA. Для изменённого кандидата, включая коммит, изменяющий только журнал изменений или версию, повторно запустите все проверки пакетов, артефактов, установки, Docker или поставщиков, затронутые изменёнными путями или хешами артефактов. Более новые запуски объединяющего процесса для той же ссылки `release/*` и той же группы повторного запуска автоматически заменяют выполняющиеся запуски. Передайте `reuse_evidence=false`, чтобы принудительно выполнить новый полный запуск.

Для ограниченного восстановления передайте объединяющему процессу `rerun_group`. `all` запускает настоящую проверку кандидата на выпуск, `ci` запускает только обычный дочерний процесс CI, `plugin-prerelease` — только дочерний процесс Plugin, предназначенный для выпуска, `release-checks` — все блоки проверки выпуска, а более узкие группы выпуска — `install-smoke`, `cross-os`, `live-e2e`, `package`, `qa`, `qa-parity`, `qa-live` и `npm-telegram`. Для точечных повторных запусков `npm-telegram` требуется `release_package_spec` или `npm_telegram_package_spec`; полные запуски и запуски `all` используют канонический E2E Telegram для пакета внутри Package Acceptance. При точечных повторных запусках для разных ОС можно добавить `cross_os_suite_filter=windows/packaged-upgrade` или другой фильтр ОС/набора. Сбои проверок выпуска для контроля качества блокируют обычную проверку выпуска, включая обязательное выявление расхождений динамических инструментов OpenClaw на стандартном уровне. Альфа-запуски Tideclaw по-прежнему могут считать рекомендательными каналы проверок выпуска, не связанные с безопасностью пакета. При `release_profile=beta` наборы с реальными поставщиками в `Run repo/live E2E validation` имеют рекомендательный характер: они выдают предупреждения, но не блокируют процесс; в профилях stable и full они остаются блокирующими. Если `live_suite_filter` явно запрашивает регулируемый канал проверки качества в реальной среде, например Discord, WhatsApp или Slack, должна быть включена соответствующая переменная репозитория `OPENCLAW_RELEASE_QA_*_LIVE_CI_ENABLED`; иначе сбор входных данных завершается сбоем, а не молча пропускает канал.

### Vitest

Блок Vitest — это запускаемый вручную дочерний рабочий процесс `CI`. Ручной CI намеренно обходит ограничение по изменённым областям и принудительно запускает обычный граф тестов для кандидата на выпуск: сегменты Linux Node, сегменты встроенных Plugin, сегменты контрактов Plugin и каналов, совместимость с Node 22, `check-*`, `check-additional-*`, быстрые проверки собранных артефактов, проверки документации, Python skills, Windows, macOS и интернационализацию Control UI. Android включается, когда блок запускает `Full Release Validation`, поскольку объединяющий процесс передаёт `include_android=true`; для отдельного ручного CI требуется `include_android=true`, чтобы охватить Android.

Используйте этот блок, чтобы ответить на вопрос: «Прошло ли дерево исходного кода полный обычный набор тестов?». Это не то же самое, что проверка продукта по пути выпуска. Необходимо сохранить следующие подтверждения:

- сводку `Full Release Validation` с URL запущенного процесса `CI`
- успешный запуск `CI` для точного целевого SHA
- названия завершившихся сбоем или медленных сегментов из заданий CI при исследовании регрессий
- артефакты длительности Vitest, например `.artifacts/vitest-shard-timings.json`, если требуется анализ производительности запуска

Запускайте ручной CI напрямую, только если выпуску нужен детерминированный обычный CI, но не нужны блоки Docker, QA Lab, реальной среды, разных ОС или пакета. Используйте первую команду для прямого CI без Android. Добавьте `include_android=true`, когда прямой CI кандидата на выпуск должен охватывать Android:

```bash
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH
gh workflow run ci.yml --ref main -f target_ref=release/YYYY.M.PATCH -f include_android=true
```

### Docker

Блок Docker находится в `OpenClaw Release Checks` через `openclaw-live-and-e2e-checks-reusable.yml`, а также в рабочем процессе `install-smoke` в режиме выпуска. Он проверяет кандидата на выпуск в упакованных средах Docker, а не только тестами на уровне исходного кода.

Покрытие Docker для выпуска включает:

- полную быструю проверку установки с включённой медленной быстрой проверкой глобальной установки Bun
- подготовку или повторное использование образа быстрой проверки корневого Dockerfile по целевому SHA, при этом задания быстрой проверки QR, root/Gateway и установщика/Bun выполняются как отдельные сегменты `install-smoke`
- каналы E2E репозитория
- фрагменты Docker для пути выпуска: `core`, `package-update-openai`, `package-update-anthropic`, `package-update-core`, `plugins-runtime-plugins`, `plugins-runtime-services`, от `plugins-runtime-install-a` до `plugins-runtime-install-h` и `openwebui`
- покрытие OpenWebUI на отдельном исполнителе с большим диском, когда оно запрошено
- разделённые каналы установки и удаления встроенных Plugin от `bundled-plugin-install-uninstall-0` до `bundled-plugin-install-uninstall-23`
- наборы поставщиков для реальной среды/E2E и покрытие реальных моделей Docker, когда проверки выпуска включают наборы для реальной среды

Перед повторным запуском используйте артефакты Docker. Планировщик пути выпуска выгружает `.artifacts/docker-tests/` с журналами каналов, `summary.json`, `failures.json`, длительностью фаз, JSON-планом планировщика и командами повторного запуска. Для точечного восстановления используйте `docker_lanes=<lane[,lane]>` в переиспользуемом рабочем процессе реальной среды/E2E вместо повторного запуска всех фрагментов выпуска. Сгенерированные команды повторного запуска включают предыдущий `package_artifact_run_id` и входные данные подготовленного образа Docker, когда они доступны, поэтому завершившийся сбоем канал может повторно использовать тот же tar-архив и образы GHCR.

### QA Lab

Блок QA Lab также входит в `OpenClaw Release Checks`. Это шлюз выпуска для агентного поведения и поведения на уровне каналов, отдельный от Vitest и механики пакетов Docker.

Покрытие QA Lab для выпуска включает:

- канал проверки соответствия с имитацией, сравнивающий кандидатный канал OpenAI с базовой линией `anthropic/claude-opus-4-8` с помощью набора агентного соответствия
- быстрый профиль проверки качества Matrix в реальной среде с использованием окружения `qa-live-shared`
- канал проверки качества Telegram в реальной среде с использованием арендованных учётных данных Convex CI
- `pnpm qa:otel:smoke`, `pnpm qa:otel:collector-smoke`, `pnpm qa:prometheus:smoke` или `pnpm qa:observability:smoke`, когда телеметрии выпуска требуется явное локальное подтверждение

Используйте этот блок, чтобы ответить на вопрос: «Корректно ли выпуск ведёт себя в сценариях контроля качества и потоках каналов в реальной среде?». При одобрении выпуска сохраняйте URL артефактов каналов соответствия, Matrix и Telegram. Полное покрытие Matrix остаётся доступным как запускаемый вручную сегментированный запуск QA Lab, а не как стандартный критический канал выпуска.

### Пакет

Блок Package — это шлюз устанавливаемого продукта. Он обеспечивается `Package Acceptance` и резолвером `scripts/resolve-openclaw-package-candidate.mjs`. Резолвер нормализует кандидата в tar-архив `package-under-test`, используемый Docker E2E, проверяет состав пакета, записывает версию пакета и SHA-256 и сохраняет ссылку рабочего процесса отдельно от ссылки на источник пакета.

Поддерживаемые источники кандидата:

- `source=npm`: `openclaw@beta`, `openclaw@latest` или точная версия выпуска OpenClaw
- `source=ref`: упаковка доверенной ветки `package_ref`, тега или полного SHA коммита с выбранной обвязкой `workflow_ref`
- `source=url`: загрузка общедоступного HTTPS-файла `.tgz` с обязательным `package_sha256`; учётные данные в URL, нестандартные порты HTTPS, частные, внутренние или зарезервированные имена хостов и разрешённые адреса, а также небезопасные перенаправления отклоняются
- `source=trusted-url`: загрузка HTTPS-файла `.tgz` с обязательными `package_sha256` и `trusted_source_id` из именованной политики в `.github/package-trusted-sources.json`; используйте этот вариант для корпоративных зеркал или частных репозиториев пакетов, принадлежащих сопровождающим, вместо добавления обхода ограничений частной сети на уровне входных данных для `source=url`
- `source=artifact`: повторное использование файла `.tgz`, выгруженного другим запуском GitHub Actions

`OpenClaw Release Checks` запускает Package Acceptance с `source=artifact`, подготовленным артефактом пакета выпуска, `suite_profile=custom`, `docker_lanes=doctor-switch update-channel-switch skill-install update-corrupt-plugin upgrade-survivor published-upgrade-survivor root-managed-vps-upgrade update-restart-auth plugins-offline plugin-update plugin-binding-command-escape`, `telegram_mode=mock-openai`. Package Acceptance сохраняет в отношении одного и того же разрешённого tar-архива проверки миграции, обновления, обновления VPS под управлением root, перезапуска после обновления с настроенной аутентификацией, установки skill из ClawHub в реальной среде, очистки устаревших зависимостей Plugin, автономных фикстур Plugin, обновления Plugin, защиты привязки команд Plugin от экранирования и проверки качества пакета Telegram. Блокирующие проверки выпуска используют стандартную базовую линию последнего опубликованного пакета; профиль beta с `run_release_soak=true`, `release_profile=stable` или `release_profile=full` расширяет проверку сохранения работоспособности после обновления опубликованного пакета до `last-stable-4` и закреплённых базовых версий `2026.4.23`, `2026.5.2` и `2026.4.15` со сценариями `reported-issues`. Используйте Package Acceptance с `source=npm` для уже выпущенного кандидата, `source=ref` — для локального tar-архива npm, привязанного к SHA, до публикации, `source=trusted-url` — для корпоративного или частного зеркала, принадлежащего сопровождающим, либо `source=artifact` — для подготовленного tar-архива, выгруженного другим запуском GitHub Actions.

Это встроенная в GitHub замена большей части покрытия пакетов и обновлений, для которого ранее требовался Parallels. Проверки выпуска на разных ОС по-прежнему важны для специфичных для ОС первоначальной настройки, установщика и поведения платформы, однако для проверки продукта в части пакетов и обновлений следует предпочитать Package Acceptance.

Канонический контрольный список для проверки обновлений и Plugin приведён в разделе [Проверка обновлений и Plugin](/ru/help/testing-updates-plugins). Используйте его при выборе локального канала, канала Docker, Package Acceptance или проверки выпуска, подтверждающего изменение установки или обновления Plugin, очистки doctor либо миграции опубликованного пакета. Исчерпывающая проверка миграции обновлений со всех стабильных пакетов `2026.4.23+` выполняется отдельным ручным рабочим процессом `Update Migration` и не входит в Full Release CI.

Послабления для устаревших версий в Package Acceptance намеренно ограничены по времени. Пакеты до версии `2026.4.25` включительно могут использовать путь совместимости для пробелов в метаданных, уже опубликованных в npm: отсутствующих в tar-архиве записей частного состава для контроля качества, отсутствующего `gateway install --wrapper`, отсутствующих файлов исправлений в полученной из tar-архива фикстуре git, отсутствующего сохранённого `update.channel`, устаревших расположений записей установки Plugin, отсутствующего сохранения записей установки из каталога и миграции метаданных конфигурации во время `plugins update`. Опубликованный пакет `2026.4.26` может выдавать предупреждения для файлов отметок метаданных локальной сборки, которые уже были выпущены. Более поздние пакеты должны соответствовать современным контрактам пакетов; те же пробелы приводят к сбою проверки выпуска.

Используйте более широкие профили Package Acceptance, когда вопрос выпуска относится к реальному устанавливаемому пакету:

```bash
gh workflow run package-acceptance.yml \
  --ref main \
  -f workflow_ref=main \
  -f source=npm \
  -f package_spec=openclaw@beta \
  -f suite_profile=product \
  -f published_upgrade_survivor_baseline=openclaw@2026.4.26
```

Распространённые профили пакетов:

- `smoke`: быстрые сценарии установки пакета, канала и агента, проверки сети Gateway и перезагрузки конфигурации
- `package`: контракты установки, обновления и перезапуска пакетов и пакетов плагинов, а также фактическое подтверждение установки Skills из ClawHub; это вариант по умолчанию для проверки релиза
- `product`: `package` плюс каналы MCP, очистка cron и субагентов, веб-поиск OpenAI и OpenWebUI
- `full`: части пути выпуска Docker с OpenWebUI
- `custom`: точный список `docker_lanes` для целевых повторных запусков

Для проверки кандидата пакета с Telegram включите `telegram_mode=mock-openai` или `telegram_mode=live-frontier` в Package Acceptance. Рабочий процесс передаёт определённый tarball `package-under-test` в сценарий Telegram; отдельный рабочий процесс Telegram по-прежнему принимает опубликованную спецификацию npm для проверок после публикации.

## Автоматизация публикации обычного релиза

Для публикации бета-версии, `latest`, плагинов, GitHub Release и платформ
обычной изменяющей точкой входа служит `OpenClaw Release Publish`. Ежемесячный
путь расширенной стабильной версии `.33+` только для npm не использует этот оркестратор. Обычный
рабочий процесс запускает рабочие процессы доверенных издателей в порядке,
необходимом для релиза:

1. Извлечь тег релиза и определить SHA его коммита.
2. Проверить, что тег достижим из `main` или `release/*` (либо из альфа-ветки Tideclaw для предварительных альфа-релизов).
3. Выполнить `pnpm plugins:sync:check`.
4. Запустить `Plugin NPM Release` с `publish_scope=all-publishable` и `ref=<release-sha>`.
5. Запустить `Plugin ClawHub Release` с той же областью и SHA.
6. Запустить `OpenClaw NPM Release` с тегом релиза, dist-tag npm и сохранённым `preflight_run_id` после проверки сохранённых `full_release_validation_run_id` и точной попытки запуска.
7. Для стабильных релизов создать или обновить GitHub Release как черновик, запустить `Windows Node Release` с явно заданным `windows_node_tag` и одобренными для кандидата `windows_node_installer_digests`, а затем проверить канонические ресурсы установщика Windows и контрольных сумм. Также запустить `Android Release`, чтобы собрать подписанный APK для точного тега вместе с контрольной суммой и сведениями о происхождении. Перед публикацией черновика проверить оба контракта нативных ресурсов.

Пример публикации бета-версии:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH-beta.N \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

Публикация стабильной версии с dist-tag `beta` по умолчанию:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=beta
```

Прямое продвижение стабильной версии в `latest` указывается явно:

```bash
gh workflow run openclaw-release-publish.yml \
  --ref main \
  -f tag=vYYYY.M.PATCH \
  -f windows_node_tag=vX.Y.Z \
  -f windows_node_installer_digests='{"OpenClawCompanion-Setup-x64.exe":"sha256:<approved-x64-sha256>","OpenClawCompanion-Setup-arm64.exe":"sha256:<approved-arm64-sha256>"}' \
  -f preflight_run_id=<successful-openclaw-npm-preflight-run-id> \
  -f full_release_validation_run_id=<successful-full-release-validation-run-id> \
  -f full_release_validation_run_attempt=<successful-full-release-validation-run-attempt> \
  -f npm_dist_tag=latest
```

Используйте низкоуровневые рабочие процессы `Plugin NPM Release` и `Plugin ClawHub Release` только для целевого исправления или повторной публикации. `OpenClaw Release Publish` отклоняет `plugin_publish_scope=selected`, когда `publish_openclaw_npm=true`, чтобы основной пакет нельзя было выпустить без всех доступных для публикации официальных плагинов, включая `@openclaw/diffs-language-pack`. Для исправления выбранного плагина задайте `publish_openclaw_npm=false` вместе с `plugin_publish_scope=selected` и `plugins=@openclaw/name` либо запустите дочерний рабочий процесс напрямую.

Первичная публикация в ClawHub является исключением: запустите `Plugin ClawHub New`
из доверенной ветки `main` и передайте полный SHA целевого релиза через `ref`.
Никогда не запускайте сам начальный рабочий процесс из тега или ветки релиза:

```bash
gh workflow run plugin-clawhub-new.yml \
  --ref main \
  -f plugins=@openclaw/name \
  -f ref=<full-40-character-release-sha> \
  -f pretag_validation=true \
  -f dry_run=true
```

Проверка до создания тега требует `dry_run=true`, отклоняет входные данные тега релиза
и родительского запуска и принимает только точную целевую версию, достижимую из `main` или `release/*`.
Она не загружает учётные данные ClawHub, не публикует байты пакета и не изменяет
конфигурацию доверенного издателя. Рабочий процесс по-прежнему определяет актуальный план реестра,
извлекает и упаковывает целевую версию только в задании без секретов, формирует
зафиксированный набор инструментов ClawHub и проверяет неизменяемый артефакт, а также
слаг и идентичность пакета до создания тега релиза. Одобряйте среду
`clawhub-plugin-bootstrap` только после завершения заданий упаковки без секретов;
это защищённое задание проверки не содержит учётных данных или команд изменения.

Одобренный пробный запуск или фактическая начальная публикация после создания тега должны включать точный
тег релиза, а также идентификатор, попытку и ветку родительского запуска `OpenClaw Release Publish`.
Родитель подтверждает SHA собственного рабочего процесса и отдельный точный SHA доверенной
ветки `main` для `Plugin ClawHub New`; дочерний запуск и каждое одобрение защищённой
среды должны соответствовать этому одобренному SHA дочернего процесса. Тег релиза
повторно проверяется перед каждой попыткой публикации и изменением доверенного издателя.

Задание упаковки
загружает один неизменяемый артефакт, имя которого, идентификатор и дайджест артефакта Actions,
запуск и попытка производителя, целевой SHA, а также SHA-256 и размер tarball каждого пакета
передаются заданиям проверки и защищённым заданиям. Защищённое задание извлекает только инструменты из доверенной ветки `main`,
проверяет кортеж артефакта через GitHub API, скачивает его
по точному идентификатору артефакта, повторно вычисляет хеш каждого tarball и проверяет локальные пути TAR и
идентичность пакета по правилам канонизации USTAR закреплённой версии CLI. Затем каждый
кандидат проходит пробную публикацию закреплённой версией CLI, которая завершается до
обращения к реестру или аутентификации. Предварительный фильтр задания с учётными данными ограничивает сжатые ClawPack
размером 120 МиБ, общий объём файлов — 50 МиБ, развёрнутые данные TAR — 64 МиБ, а
число записей TAR — 10 000. Исправление доверенного издателя для существующего пакета остаётся
только настройкой, но оно всё равно упаковывает целевую версию и требует полного совпадения запрошенного тега,
байтов и метаданных реестра перед изменением конфигурации доверенного издателя.
Проверка после публикации скачивает артефакт ClawHub и
требует того же SHA-256 и размера. Восстановление через повторный запуск неудачных заданий может использовать артефакт пакета из предыдущей
попытки, только если конкретное задание-производитель успешно завершилось.
Итоговое подтверждение также связывает зафиксированную версию ClawHub, SHA-256 файла блокировки
и целостность npm. При несовпадении требуется новая версия пакета.

## Входные данные рабочего процесса NPM

`OpenClaw NPM Release` принимает следующие входные данные, управляемые оператором:

- `tag`: обязательный тег релиза, например `v2026.4.2`, `v2026.4.2-1`, `v2026.4.2-beta.1` или `v2026.4.2-alpha.1`; когда `preflight_only=true`, также допускается текущий полный 40-символьный SHA коммита ветки рабочего процесса для предварительной проверки без публикации
- `preflight_only`: `true` только для проверки, сборки и упаковки, `false` для фактической публикации
- `preflight_run_id`: идентификатор существующего успешного предварительного запуска, обязательный для фактической публикации, чтобы рабочий процесс повторно использовал подготовленный tarball вместо его пересборки
- `full_release_validation_run_id`: идентификатор успешного запуска `Full Release Validation` для этого тега или SHA, обязательный для фактической публикации. Публикации бета-версий могут выполняться только с предварительной проверкой и предупреждением, но для продвижения стабильной версии или `latest` он всё равно обязателен.
- `full_release_validation_run_attempt`: точный положительный номер попытки запуска, связанный с `full_release_validation_run_id`; обязателен при указании идентификатора запуска, чтобы повторные запуски не могли изменить подтверждение авторизации во время публикации.
- `release_publish_run_id`: идентификатор одобренного запуска `OpenClaw Release Publish`; обязателен, когда этот рабочий процесс запускается родительским процессом (фактические вызовы публикации от имени бота)
- `plugin_npm_run_id`: идентификатор успешного запуска `Plugin NPM Release` для точной вершины; обязателен для фактической публикации основного пакета `extended-stable`
- `npm_dist_tag`: целевой тег npm для публикации; принимает `alpha`, `beta`, `latest` или `extended-stable`, по умолчанию — `beta`. Финальный патч `33` и последующие должны использовать `extended-stable`; по умолчанию `extended-stable` отклоняет более ранние патчи и всегда отклоняет нефинальные теги.
- `bypass_extended_stable_guard`: логическое значение только для тестирования, по умолчанию `false`; при `npm_dist_tag=extended-stable` обходит проверку соответствия ежемесячной расширенной стабильной версии, сохраняя проверки идентичности релиза, артефакта, одобрения и контрольного чтения.

`Plugin NPM Release` принимает `npm_dist_tag=default` для существующего поведения
релиза или `npm_dist_tag=extended-stable` для защищённого ежемесячного пути.
Вариант расширенной стабильной версии требует `publish_scope=all-publishable`, пустого
входного параметра `plugins`, финального патча не ниже `33` и канонической
ветки `extended-stable/YYYY.M.33` на её точной вершине. Он никогда не перемещает теги плагинов
`latest` или `beta`. Новые версии пакетов атомарно получают `extended-stable`
через доверенную публикацию OIDC (`npm publish --tag extended-stable`); этот
исходный рабочий процесс не использует аутентифицированную токеном команду `npm dist-tag add`. Повторные попытки
пропускают точные версии, уже имеющиеся в npm, а затем завершаются отказом, если полное
контрольное чтение не подтверждает совпадение каждого точного пакета и тега `extended-stable`.

`OpenClaw Release Publish` принимает следующие входные данные, управляемые оператором:

- `tag`: обязательный тег релиза; он уже должен существовать
- `preflight_run_id`: идентификатор успешного предварительного запуска `OpenClaw NPM Release`; обязателен, когда `publish_openclaw_npm=true` или `plugin_publish_scope=all-publishable`
- `full_release_validation_run_id`: идентификатор успешного запуска `Full Release Validation`; обязателен, когда `publish_openclaw_npm=true` или `plugin_publish_scope=all-publishable`
- `full_release_validation_run_attempt`: точный положительный номер попытки, связанный с `full_release_validation_run_id`; обязателен при указании идентификатора запуска
- `windows_node_tag`: точный тег стабильного релиза `openclaw/openclaw-windows-node`; обязателен для публикации стабильной версии OpenClaw
- `windows_node_installer_digests`: одобренная для кандидата компактная карта JSON с текущими именами установщиков Windows и их закреплёнными дайджестами `sha256:`; обязательна для публикации стабильной версии OpenClaw
- `npm_telegram_run_id`: необязательный идентификатор успешного запуска `NPM Telegram Beta E2E` для включения в итоговое подтверждение релиза
- `npm_dist_tag`: целевой тег npm для пакета OpenClaw, один из `alpha`, `beta` или `latest`
- `plugin_publish_scope`: по умолчанию `all-publishable`; используйте `selected` только для целевого исправления исключительно плагинов с `publish_openclaw_npm=false`
- `plugins`: разделённые запятыми имена пакетов `@openclaw/*`, когда `plugin_publish_scope=selected`
- `publish_openclaw_npm`: по умолчанию `true`; задавайте `false` только при использовании рабочего процесса как оркестратора исправления исключительно плагинов
- `release_profile`: профиль охвата релиза, используемый в сводках подтверждений релиза; по умолчанию `from-validation`, который считывает его из манифеста проверки; также можно явно указать `beta`, `stable` или `full`
- `wait_for_clawhub`: по умолчанию `false`, чтобы доступность npm не блокировалась вспомогательным процессом ClawHub; задавайте `true`, только когда завершение рабочего процесса должно включать завершение ClawHub

`OpenClaw Release Checks` принимает следующие входные данные, управляемые оператором:

- `ref`: ветка, тег или полный SHA коммита для проверки. Для проверок с секретами определённый коммит должен быть достижим из ветки OpenClaw или тега релиза.
- `run_release_soak`: включает исчерпывающие фактические и E2E-проверки, путь выпуска Docker и длительную проверку сохранения работоспособности при обновлении со всех предыдущих версий для проверок бета-релиза. Принудительно включается при `release_profile=stable` и `release_profile=full`.

Правила:

- Обычные финальные и корректирующие версии ниже патча `33` могут публиковаться либо в `beta`, либо в `latest`. Финальные версии с патчем `33` или выше должны публиковаться в `extended-stable`, а версии с суффиксом коррекции на этой границе отклоняются.
- Теги предварительных бета-версий могут публиковаться только в `beta`; теги предварительных альфа-версий — только в `alpha`
- Для `OpenClaw NPM Release` полный SHA коммита разрешён только при `preflight_only=true`
- `OpenClaw Release Checks` и `Full Release Validation` всегда предназначены только для проверки
- Фактическая публикация должна использовать тот же `npm_dist_tag`, который применялся при предварительной проверке; рабочий процесс проверяет эти метаданные, прежде чем продолжить публикацию

## Последовательность обычного стабильного выпуска через beta/latest

Эта устаревшая последовательность предназначена для обычного оркестрируемого выпуска, который также охватывает плагины, GitHub Release, Windows и работу для других платформ. Она не относится к описанному в начале этой страницы ежемесячному пути расширенно стабильных выпусков `.33+` только для npm.

При подготовке обычного оркестрируемого стабильного выпуска:

1. Запустите `OpenClaw NPM Release` с `preflight_only=true`. До создания тега для пробного запуска рабочего процесса предварительной проверки без публикации можно использовать полный SHA текущего коммита ветки рабочего процесса.
2. Выберите `npm_dist_tag=beta` для обычного процесса с первоначальной публикацией в beta или `latest`, только если намеренно хотите опубликовать стабильную версию напрямую.
3. Запустите `Full Release Validation` для ветки выпуска, тега выпуска или полного SHA коммита, если хотите одним ручным рабочим процессом выполнить обычный CI, а также проверки кэша запросов в реальной среде, Docker, QA Lab, Matrix и Telegram. Если намеренно требуется только детерминированный обычный граф тестов, вместо этого запустите ручной рабочий процесс `CI` для ссылки выпуска.
4. Выберите точный тег выпуска `openclaw/openclaw-windows-node`, не являющийся тегом предварительной версии, подписанные установщики x64 и ARM64 из которого должны быть выпущены. Сохраните его как `windows_node_tag`, а проверенную карту дайджестов установщиков — как `windows_node_installer_digests`. Вспомогательный инструмент для кандидата на выпуск записывает оба значения и включает их в сгенерированную команду публикации.
5. Сохраните значения успешных запусков `preflight_run_id`, `full_release_validation_run_id` и точное `full_release_validation_run_attempt`.
6. Запустите `OpenClaw Release Publish` из доверенной ветки `main` с тем же `tag`, тем же `npm_dist_tag`, выбранным `windows_node_tag`, сохранённым значением `windows_node_installer_digests`, а также сохранёнными `preflight_run_id`, `full_release_validation_run_id` и `full_release_validation_run_attempt`. Этот процесс публикует вынесенные плагины в npm и ClawHub перед продвижением пакета OpenClaw в npm.
7. Если выпуск был опубликован в `beta`, используйте рабочий процесс `openclaw/releases/.github/workflows/openclaw-npm-dist-tags.yml`, чтобы продвинуть эту стабильную версию из `beta` в `latest`.
8. Если выпуск намеренно был опубликован напрямую в `latest` и `beta` должна сразу указывать на ту же стабильную сборку, используйте тот же рабочий процесс выпуска, чтобы назначить обоим dist-тегам стабильную версию, либо дождитесь, пока запланированная самовосстанавливающаяся синхронизация позже обновит `beta`.

Изменение dist-тегов выполняется в репозитории журнала выпусков, поскольку для него по-прежнему требуется `NPM_TOKEN`, тогда как репозиторий исходного кода использует для публикации только OIDC. Благодаря этому как путь прямой публикации, так и путь продвижения с первоначальной публикацией в beta остаются документированными и видимыми операторам.

Если сопровождающему необходимо вернуться к локальной аутентификации npm, запускайте любые команды CLI 1Password (`op`) только внутри отдельного сеанса tmux. Не вызывайте `op` напрямую из основной оболочки агента: выполнение внутри tmux делает запросы, предупреждения и обработку OTP наблюдаемыми и предотвращает повторные оповещения хоста.
