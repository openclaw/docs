OpenClaw применяет к транскриптам **исправления, специфичные для провайдера**, перед запуском
(при построении контекста модели). Большинство из них представляют собой корректировки
**в памяти**, необходимые для соблюдения строгих требований провайдера. Отдельный этап
исправления файла сессии также может перезаписать сохранённый JSONL до загрузки сессии,
но только для некорректных строк или сохранённых ходов, которые не являются допустимыми
долговременными записями. Доставленные ответы ассистента сохраняются на диске; удаление
предварительного заполнения ассистента, специфичного для провайдера, выполняется только
при формировании исходящих полезных нагрузок.

Когда выполняется исправление, исходный файл записывается во временный соседний файл
`*.bak-<pid>-<ts>` перед атомарной заменой, а после успешной замены удаляется. Резервная
копия сохраняется только в случае сбоя самой очистки; в этом случае возвращается путь
к ней.

Область действия включает:

- Исключение контекста запроса, предназначенного только для среды выполнения, из видимых пользователю ходов транскрипта
- Нормализацию идентификаторов вызовов инструментов
- Проверку входных данных вызовов инструментов
- Исправление сопоставления результатов инструментов
- Проверку и упорядочивание ходов
- Очистку подписей рассуждений
- Очистку подписей процесса мышления
- Нормализацию полезных нагрузок изображений
- Удаление пустых текстовых блоков перед повторной передачей провайдеру
- Удаление незавершённых ходов, содержащих только рассуждения и завершившихся из-за ограничения длины, перед повторной передачей провайдеру
- Маркировку происхождения пользовательского ввода (для запросов, перенаправленных между сессиями)
- Исправление пустых ошибочных ходов ассистента при повторной передаче через Bedrock Converse

Подробности о хранении транскриптов см. в разделе
[Подробный разбор управления сессиями](/ru/reference/session-management-compaction).

## Общее правило: контекст среды выполнения не является пользовательским транскриптом

Контекст среды выполнения или системный контекст может быть добавлен в запрос к модели
для конкретного хода, но он не является содержимым, созданным конечным пользователем.
OpenClaw хранит отдельное тело запроса для транскрипта, используемое в ответах Gateway,
последующих запросах в очереди, ACP, CLI и встроенных запусках OpenClaw. В сохранённых
видимых пользовательских ходах вместо запроса, обогащённого контекстом среды выполнения,
используется это тело транскрипта.

Для устаревших сессий, в которых уже сохранены обёртки среды выполнения, поверхности
истории Gateway применяют проекцию отображения перед возвратом сообщений клиентам
WebChat, TUI, REST или SSE.

## Где это выполняется

Вся обработка транскриптов централизована во встроенном средстве запуска:

- Выбор политики: `src/agents/transcript-policy.ts`
  (`resolveTranscriptPolicy`, с ключами `provider`, `modelApi` и `modelId`)
- Применение нормализации и исправлений: `sanitizeSessionHistory` в
  `src/agents/embedded-agent-runner/replay-history.ts`

Отдельно от обработки транскрипта файлы сессий при необходимости исправляются
перед загрузкой:

- `repairSessionFileIfNeeded` в `src/agents/session-file-repair.ts`
- Вызывается из `src/agents/embedded-agent-runner/run/attempt.ts` и
  `src/agents/embedded-agent-runner/compact.ts`

## Общее правило: нормализация изображений

Полезные нагрузки изображений всегда нормализуются, чтобы предотвратить отклонение
со стороны провайдера из-за ограничений размера (слишком большие изображения в base64
уменьшаются и повторно сжимаются). Это также помогает контролировать нагрузку на токены,
создаваемую изображениями для моделей с поддержкой компьютерного зрения: меньшие
максимальные размеры сокращают расход токенов, а большие сохраняют детализацию.

Реализация:

- `sanitizeSessionMessagesImages` в
  `src/agents/embedded-agent-helpers/images.ts`
- `sanitizeContentBlocksImages` в `src/agents/tool-images.ts`
- Максимальный размер стороны изображения настраивается через `agents.defaults.imageMaxDimensionPx`
  (по умолчанию: `1200`)
- При обходе содержимого для повторной передачи на этом этапе удаляются пустые
  текстовые блоки. Ходы ассистента, ставшие пустыми, исключаются из копии для повторной
  передачи; пользовательские ходы и ходы с результатами инструментов, ставшие пустыми,
  получают непустой заполнитель пропущенного содержимого.

## Общее правило: некорректные вызовы инструментов

Блоки вызовов инструментов ассистента, в которых отсутствуют и `input`, и `arguments`,
удаляются до построения контекста модели. Это предотвращает отклонение со стороны
провайдера частично сохранённых вызовов инструментов (например, после ошибки из-за
ограничения частоты запросов).

Реализация:

- `sanitizeToolCallInputs` в `src/agents/session-transcript-repair.ts`
- Применяется в `sanitizeSessionHistory`
  (`src/agents/embedded-agent-runner/replay-history.ts`)

---

## Общее правило: незавершённые ходы, содержащие только рассуждения

Ходы ассистента, которые достигли ограничения вывода провайдера и содержат только
процесс мышления или отредактированный процесс мышления, исключаются из копии для
повторной передачи в памяти. Такие ходы содержат незавершённое состояние провайдера
и могут включать частичную подпись процесса мышления.

Пустые ходы, завершившиеся из-за ограничения длины, остаются без изменений, как и ходы
с видимым текстом, вызовами инструментов или неизвестными блоками содержимого.
Сохранённые транскрипты не перезаписываются.

Реализация: `normalizeAssistantReplayContent` в
`src/agents/embedded-agent-runner/replay-history.ts`

## Общее правило: происхождение ввода между сессиями

Когда агент отправляет запрос в другую сессию через `sessions_send`
(включая этапы ответа или объявления между агентами), OpenClaw сохраняет
созданный пользовательский ход с `message.provenance.kind = "inter_session"`.

OpenClaw также добавляет перед текстом перенаправленного запроса в том же ходе маркер
`[Inter-session message] ... isUser=false`, чтобы активный вызов модели мог отличить
вывод другой сессии от внешних инструкций конечного пользователя. Когда это возможно,
маркер содержит исходную сессию, канал и инструмент. Для совместимости с провайдером
в транскрипте по-прежнему используется `role: "user"`, но видимый текст и метаданные
происхождения помечают ход как данные из другой сессии.

При повторном построении контекста OpenClaw применяет такой же маркер к ранее сохранённым
пользовательским ходам между сессиями, в которых присутствуют только метаданные
происхождения.

## Матрица провайдеров (текущее поведение)

**OpenAI / OpenAI Codex**

- Только нормализация изображений.
- Удаление несопоставленных подписей рассуждений (отдельных элементов рассуждений,
  за которыми не следует блок содержимого) для транскриптов OpenAI Responses/Codex,
  а также удаление рассуждений OpenAI, пригодных для повторной передачи, после
  переключения маршрута модели.
- Сохранение полезных нагрузок элементов рассуждений OpenAI Responses, пригодных
  для повторной передачи, включая зашифрованные элементы с пустым резюме, чтобы
  при ручной повторной передаче или повторной передаче через WebSocket требуемое
  состояние `rs_*` оставалось сопоставленным с элементами вывода ассистента.
- Нативный ChatGPT Codex Responses сохраняет соответствие протоколу Codex, повторно
  передавая предыдущие полезные нагрузки рассуждений, сообщений и функций Responses
  без идентификаторов предыдущих элементов, сохраняя при этом `prompt_cache_key` сессии.
- Повторная передача семейства OpenAI Responses сохраняет канонические пары рассуждений
  одной модели `call_*|fc_*`, но детерминированно нормализует некорректные или слишком
  длинные идентификаторы `call_id` и элементов вызова функций перед преобразованием
  полезной нагрузки pi-ai.
- Исправление сопоставления результатов инструментов может перемещать реальные
  совпавшие результаты и создавать результаты Codex со значением `aborted`
  для отсутствующих вызовов инструментов.
- Проверка и переупорядочивание ходов не выполняются; подписи рассуждений не удаляются.

**Совместимые с OpenAI Chat Completions**

- Исторические блоки процесса мышления и рассуждений ассистента удаляются перед
  повторной передачей, чтобы локальные и прокси-серверы, совместимые с OpenAI,
  не получали поля рассуждений из предыдущих ходов, такие как `reasoning`
  или `reasoning_content`.
- При продолжении вызовов инструментов в текущем ходе блок рассуждений ассистента
  остаётся прикреплённым к вызову инструмента до повторной передачи результата
  инструмента.
- Пользовательские и самостоятельно размещённые записи моделей с `reasoning: true`
  сохраняют метаданные рассуждений при повторной передаче.
- Исключения, принадлежащие провайдеру, могут отключать это поведение, если их
  протокол требует повторной передачи метаданных рассуждений.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Нормализация идентификаторов вызовов инструментов: только буквы и цифры.
- Исправление сопоставления результатов инструментов и создание синтетических результатов инструментов.
- Проверка ходов (чередование ходов в стиле Gemini).
- Исправление порядка ходов Google (добавление короткого начального пользовательского
  сообщения, если история начинается с ассистента).
- Antigravity Claude: нормализация подписей процесса мышления; удаление неподписанных
  блоков процесса мышления.

**Anthropic / Minimax (совместимые с Anthropic)**

- Исправление сопоставления результатов инструментов и создание синтетических результатов инструментов.
- Проверка ходов (объединение последовательных пользовательских ходов для соблюдения
  строгого чередования).
- Завершающие ходы предварительного заполнения ассистента удаляются из исходящих
  полезных нагрузок Anthropic Messages, когда включён процесс мышления, включая
  маршруты Cloudflare AI Gateway.
- Подписи процесса мышления ассистента, созданные до Compaction, удаляются перед
  повторной передачей провайдеру, если сессия была подвергнута Compaction. Подписи
  процесса мышления криптографически привязываются к префиксу диалога во время
  генерации; после Compaction префикс изменяется (исходное содержимое заменяется
  сводкой), поэтому повторная передача исходных подписей приводит к отклонению
  запроса Anthropic с ошибкой "Invalid signature in thinking block". Текст процесса
  мышления сохраняется как неподписанный блок, после чего обрабатывается следующим
  правилом.
- Блоки процесса мышления с отсутствующими, пустыми или состоящими из пробелов
  подписями для повторной передачи удаляются перед преобразованием для провайдера.
  Если в результате ход ассистента становится пустым, OpenClaw сохраняет структуру
  хода с непустым текстом о пропущенных рассуждениях.
- Более старые ходы ассистента, содержащие только процесс мышления и подлежащие
  удалению, заменяются непустым текстом о пропущенных рассуждениях, чтобы адаптеры
  провайдера не удаляли ход при повторной передаче.

**Amazon Bedrock (Converse API)**

- Пустые ошибочные ходы потока ассистента перед повторной передачей исправляются
  на непустой резервный текстовый блок. Bedrock Converse отклоняет сообщения
  ассистента с `content: []`, поэтому сохранённые ходы ассистента с `stopReason:
"error"` и пустым содержимым также исправляются на диске перед загрузкой.
- Ошибочные ходы потока ассистента, содержащие только пустые текстовые блоки,
  удаляются из копии для повторной передачи в памяти вместо передачи недопустимого
  пустого блока.
- Подписи процесса мышления ассистента, созданные до Compaction, удаляются перед
  повторной передачей Converse, если сессия была подвергнута Compaction, по той же
  причине, что и для Anthropic выше.
- Блоки процесса мышления Claude с отсутствующими, пустыми или состоящими из пробелов
  подписями для повторной передачи удаляются перед повторной передачей Converse.
  Если в результате ход ассистента становится пустым, OpenClaw сохраняет структуру
  хода с непустым текстом о пропущенных рассуждениях.
- Более старые ходы ассистента, содержащие только процесс мышления и подлежащие
  удалению, заменяются непустым текстом о пропущенных рассуждениях, чтобы при повторной
  передаче Converse сохранялась строгая структура ходов.
- При повторной передаче фильтруются ходы ассистента, являющиеся зеркалами доставки
  OpenClaw или добавленные Gateway.
- Нормализация изображений применяется согласно общему правилу.

**Mistral (включая обнаружение по идентификатору модели)**

- Нормализация идентификаторов вызовов инструментов: strict9 (буквы и цифры, длина 9).

**OpenRouter Gemini**

- Очистка подписей рассуждений: удаление значений `thought_signature`, не являющихся
  base64 (значения base64 сохраняются).

**OpenRouter Anthropic**

- Завершающие ходы предварительного заполнения ассистента удаляются из проверенных
  полезных нагрузок моделей Anthropic в OpenRouter, совместимых с OpenAI, когда
  включены рассуждения, в соответствии с поведением повторной передачи для прямого
  Anthropic и Anthropic через Cloudflare.

**Все остальные**

- Только нормализация изображений.

## Историческое поведение (до версии 2026.1.22)

До выпуска версии 2026.1.22 OpenClaw применял несколько уровней обработки транскриптов:

- При каждом построении контекста запускалось **расширение нормализации транскрипта**,
  которое могло:
  - Исправлять сопоставление использования инструментов и их результатов.
  - Нормализовать идентификаторы вызовов инструментов (включая нестрогий режим,
    сохранявший `_`/`-`).
- Средство запуска также выполняло нормализацию, специфичную для провайдера,
  что приводило к дублированию работы.
- Дополнительные изменения выполнялись вне политики провайдера, включая удаление
  тегов `<final>` из текста ассистента перед сохранением, удаление пустых ошибочных
  ходов ассистента и обрезку содержимого ассистента после вызовов инструментов.

Эта сложность вызывала регрессии между провайдерами (в частности, при сопоставлении
`call_id|fc_id` в `openai-responses`). При очистке в версии 2026.1.22 расширение было
удалено, логика централизована в средстве запуска, а для OpenAI установлен режим
**без изменений**, за исключением нормализации изображений.
