Вывод ассистента передает директивы доставки и отображения через несколько специализированных каналов:

- Структурированные поля `mediaUrl` / `mediaUrls` для доставки вложений.
- `[[audio_as_voice]]` для указания способа представления аудио.
- `[[reply_to_current]]` / `[[reply_to:<id>]]` для метаданных ответа.
- `[embed ...]` для расширенного отображения в интерфейсе управления.

Структурированные поля медиафайлов и теги `[[...]]` являются метаданными доставки. `[embed ...]` — это отдельный механизм расширенного отображения, доступный только в веб-интерфейсе; он не является псевдонимом для медиафайлов.

## Медиа-вложения

Удаленные вложения должны быть общедоступными URL-адресами с протоколом `https:`. URL-адреса с `http:`, local loopback, локальные адреса канала, частные и внутренние имена хостов отклоняются в качестве директив вложений; серверные средства получения медиафайлов дополнительно применяют собственные сетевые ограничения.

Для локальных вложений допускаются абсолютные пути, пути относительно рабочего пространства и пути относительно домашнего каталога с префиксом `~/`. Перед доставкой они по-прежнему проходят проверку политикой чтения файлов агента и проверку типа медиафайла.

<Warning>
Не создавайте текстовые команды для вложений из инструментов, плагинов, потоковых блоков, вывода браузера или действий с сообщениями. Вместо этого используйте структурированные поля медиафайлов:

```json
{ "message": "Вот ваше изображение.", "mediaUrl": "/workspace/image.png" }
```

Текст устаревшего формата в окончательном ответе по-прежнему может нормализоваться для совместимости, но это не универсальный протокол плагинов или инструментов.
</Warning>

Обычный синтаксис изображений Markdown (`![alt](url)`) по умолчанию остается текстом. Каналы, которым требуется обрабатывать изображения Markdown как ответы с медиафайлами, включают эту возможность в своем исходящем адаптере; Telegram делает это, поэтому `![alt](url)` преобразуется в медиа-вложение.

При включенной потоковой передаче блоков медиафайлы должны передаваться в структурированных полях полезной нагрузки. Если один и тот же URL-адрес медиафайла встречается в потоковом блоке и затем снова в окончательной полезной нагрузке ассистента, OpenClaw доставляет его один раз и удаляет дубликат из окончательной полезной нагрузки.

## `[embed ...]`

`[embed ...]` — единственный доступный агенту синтаксис расширенного отображения для интерфейса управления. Пример самозакрывающегося элемента:

```text
[embed ref="cv_123" title="Status" /]
```

Правила:

- `[view ...]` больше нельзя использовать в новом выводе.
- Шорткоды встраивания отображаются только в области сообщений ассистента.
- Отображаются только встраиваемые элементы, основанные на URL-адресах; используйте `ref="..."` или `url="..."`.
- Блочные шорткоды встраивания с HTML внутри не отображаются.
- Веб-интерфейс удаляет шорткод из видимого текста и отображает встроенный элемент в тексте.

## Сохраняемая структура отображения

Нормализованный и сохраняемый блок содержимого ассистента представляет собой структурированный элемент `canvas`:

```json
{
  "type": "canvas",
  "preview": {
    "kind": "canvas",
    "surface": "assistant_message",
    "render": "url",
    "viewId": "cv_123",
    "url": "/__openclaw__/canvas/documents/cv_123/index.html",
    "title": "Status",
    "preferredHeight": 320
  }
}
```

`present_view` не распознается; сохраняемые и отображаемые расширенные блоки всегда используют эту структуру `canvas`.
