OpenClaw отслеживает **токены**, а не символы. Токены зависят от модели, но большинство моделей в стиле OpenAI в среднем используют около 4 символов на токен для английского текста.

## Как строится системная инструкция

OpenClaw собирает собственную системную инструкцию при каждом запуске. Она включает:

- Список инструментов + краткие описания
- Список Skills (только метаданные; инструкции загружаются по требованию через `read`).
  Нативные ходы Codex получают компактный блок Skills как ограниченные текущим ходом
  developer-инструкции для совместной работы; другие среды выполнения получают его в обычной
  поверхности инструкции. Он ограничен `skills.limits.maxSkillsPromptChars`, с
  необязательным переопределением для агента в `agents.list[].skillsLimits.maxSkillsPromptChars`.
- Инструкции по самообновлению
- Рабочее пространство + файлы начальной загрузки (`AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md`, когда новые, плюс `MEMORY.md`, когда присутствует). Нативные ходы Codex не вставляют сырой `MEMORY.md` из настроенного рабочего пространства агента, когда для этого рабочего пространства доступны инструменты памяти; они включают небольшой указатель на память в ограниченные текущим ходом developer-инструкции для совместной работы и используют инструменты памяти по требованию. Если инструменты отключены, поиск по памяти недоступен или активное рабочее пространство отличается от рабочего пространства памяти агента, `MEMORY.md` использует обычный ограниченный путь контекста хода. Корневой `memory.md` в нижнем регистре не внедряется; это устаревший вход для исправления через `openclaw doctor --fix` при наличии пары с `MEMORY.md`. Большие внедряемые файлы усекаются по `agents.defaults.bootstrapMaxChars` (по умолчанию: 20000), а общий объем внедрения начальной загрузки ограничен `agents.defaults.bootstrapTotalMaxChars` (по умолчанию: 60000). Ежедневные файлы `memory/*.md` не входят в обычную инструкцию начальной загрузки; в обычных ходах они остаются доступными по требованию через инструменты памяти, но модельные запуски сброса/старта могут предварять первый ход одноразовым блоком стартового контекста с недавней ежедневной памятью. Простые команды чата `/new` и `/reset` подтверждаются без вызова модели. Стартовая преамбула управляется `agents.defaults.startupContext`. Фрагменты AGENTS.md после Compaction являются отдельными и требуют явного включения через `agents.defaults.compaction.postCompactionSections`.
- Время (UTC + часовой пояс пользователя)
- Теги ответа + поведение Heartbeat
- Метаданные среды выполнения (хост/ОС/модель/размышление)

Полную разбивку см. в [Системная инструкция](/ru/concepts/system-prompt).

При документировании учетных данных или фрагментов аутентификации используйте
[Соглашения о заполнителях секретов](/ru/reference/secret-placeholder-conventions), чтобы
избежать ложных срабатываний сканера секретов в изменениях только документации.

## Что учитывается в окне контекста

Все, что получает модель, учитывается в лимите контекста:

- Системная инструкция (все разделы, перечисленные выше)
- История разговора (сообщения пользователя + ассистента)
- Вызовы инструментов и результаты инструментов
- Вложения/расшифровки (изображения, аудио, файлы)
- Сводки Compaction и артефакты прореживания
- Обертки провайдера или заголовки безопасности (не видны, но все равно учитываются)

У некоторых поверхностей с высокой нагрузкой на среду выполнения есть собственные явные лимиты:

- `agents.defaults.contextLimits.memoryGetMaxChars`
- `agents.defaults.contextLimits.memoryGetDefaultLines`
- `agents.defaults.contextLimits.toolResultMaxChars`
- `agents.defaults.contextLimits.postCompactionMaxChars`

Переопределения для агента находятся в `agents.list[].contextLimits`. Эти регуляторы
предназначены для ограниченных фрагментов среды выполнения и внедряемых блоков, которыми владеет среда выполнения. Они
отдельны от лимитов начальной загрузки, лимитов стартового контекста и лимитов инструкции Skills.

`toolResultMaxChars` — расширенный потолок (до `1000000` символов). Если он не задан, OpenClaw выбирает
актуальный лимит результата инструмента из эффективного окна контекста модели: `16000` символов
ниже 100K токенов, `32000` символов при 100K+ токенах и `64000` символов при 200K+
токенах, при этом оставаясь ограниченным защитой доли контекста среды выполнения.

Для изображений OpenClaw уменьшает размер полезной нагрузки изображений из расшифровок/инструментов перед вызовами провайдера.
Используйте `agents.defaults.imageMaxDimensionPx` (по умолчанию: `1200`) для настройки:

- Меньшие значения обычно снижают использование vision-токенов и размер полезной нагрузки.
- Большие значения сохраняют больше визуальных деталей для скриншотов с большим количеством OCR/UI.

Для практической разбивки (по внедренному файлу, инструментам, Skills и размеру системной инструкции) используйте `/context list` или `/context detail`. См. [Контекст](/ru/concepts/context).

## Как посмотреть текущее использование токенов

Используйте это в чате:

- `/status` → **статусная карточка с эмодзи** с моделью сессии, использованием контекста,
  входными/выходными токенами последнего ответа и **оценочной стоимостью**, когда локальные цены
  настроены для активной модели.
- `/usage off|tokens|full` → добавляет **подвал использования для каждого ответа** к каждому ответу.
  - Сохраняется на сессию (хранится как `responseUsage`).
  - `/usage reset` (псевдонимы: `inherit`, `clear`, `default`) — очищает переопределение сессии,
    чтобы сессия снова унаследовала настроенное значение по умолчанию.
  - `/usage tokens` показывает сведения о токенах/кэше хода.
  - `/usage full` показывает компактные сведения о модели/контексте/стоимости; оценочная стоимость появляется
    только когда у OpenClaw есть метаданные использования и локальные цены для активной модели.
    Пользовательские макеты `messages.usageTemplate` могут включать поля токенов/кэша.
- `/usage cost` → показывает локальную сводку стоимости из журналов сессий OpenClaw.

Другие поверхности:

- **TUI/Web TUI:** `/status` + `/usage` поддерживаются.
- **CLI:** `openclaw status --usage` и `openclaw channels list` показывают
  нормализованные окна квот провайдера (`X% left`, а не стоимость каждого ответа).
  Текущие провайдеры окон использования: Anthropic, GitHub Copilot, Gemini CLI,
  OpenAI Codex, MiniMax, Xiaomi и z.ai.

Поверхности использования нормализуют распространенные нативные псевдонимы полей провайдера перед отображением.
Для трафика Responses семейства OpenAI это включает и `input_tokens` /
`output_tokens`, и `prompt_tokens` / `completion_tokens`, поэтому имена полей,
специфичные для транспорта, не меняют `/status`, `/usage` или сводки сессии.
Использование Gemini CLI тоже нормализуется: парсер `stream-json` по умолчанию читает
события `message` ассистента, а `stats.cached` сопоставляется с `cacheRead` с
использованием `stats.input_tokens - stats.cached`, когда CLI не предоставляет явное
поле `stats.input`. Устаревшие JSON-переопределения по-прежнему читают текст ответа из
`response`.
Для нативного трафика Responses семейства OpenAI псевдонимы использования WebSocket/SSE
нормализуются тем же способом, а итоговые значения откатываются к нормализованному вводу + выводу, когда
`total_tokens` отсутствует или равен `0`.
Когда текущий снимок сессии разрежен, `/status` и `session_status` также могут
восстановить счетчики токенов/кэша и метку активной модели среды выполнения из
самого недавнего журнала использования расшифровки. Существующие ненулевые live-значения все равно имеют
приоритет над резервными значениями расшифровки, а более крупные, ориентированные на prompt
итоги расшифровки могут победить, когда сохраненные итоги отсутствуют или меньше.
Аутентификация использования для окон квот провайдера берется из хуков конкретного провайдера, когда
они доступны; иначе OpenClaw откатывается к сопоставлению учетных данных OAuth/API-ключа
из профилей аутентификации, env или конфигурации.
Записи расшифровки ассистента сохраняют ту же нормализованную форму использования, включая
`usage.cost`, когда для активной модели настроены цены и провайдер
возвращает метаданные использования. Это дает `/usage cost` и статусу сессии на основе расшифровки
стабильный источник даже после исчезновения live-состояния среды выполнения.

OpenClaw держит учет использования провайдера отдельно от текущего снимка контекста.
`usage.total` провайдера может включать кэшированный ввод, вывод и несколько
модельных вызовов цикла инструментов, поэтому он полезен для стоимости и телеметрии, но может завышать
живое окно контекста. Отображения и диагностика контекста используют последний снимок prompt
(`promptTokens` или последний вызов модели, когда снимок prompt недоступен)
для `context.used`.

## Оценка стоимости (когда отображается)

Стоимость оценивается из вашей конфигурации цен моделей:

```
models.providers.<provider>.models[].cost
```

Это **USD за 1M токенов** для `input`, `output`, `cacheRead` и
`cacheWrite`. Если цены отсутствуют, `/usage full` опускает стоимость; используйте `/usage tokens`
или пользовательский `messages.usageTemplate`, когда вам нужны сведения о токенах/кэше в каждом
ответе. Отображение стоимости не ограничено аутентификацией по API-ключу: провайдеры без API-ключа,
такие как `aws-sdk`, могут показывать оценочную стоимость, когда их настроенная запись модели включает
локальные цены и провайдер возвращает метаданные использования.

После того как sidecar-процессы и каналы достигают готового пути Gateway, OpenClaw запускает
необязательную фоновую начальную загрузку цен для настроенных ссылок на модели, у которых еще
нет локальных цен. Эта начальная загрузка получает удаленные каталоги цен OpenRouter и LiteLLM.
Установите `models.pricing.enabled: false`, чтобы пропустить получение этих каталогов
в офлайн- или ограниченных сетях; явные записи
`models.providers.*.models[].cost` продолжают управлять локальными оценками стоимости.

## TTL кэша и влияние прореживания

Кэширование prompt у провайдера применяется только в пределах окна TTL кэша. OpenClaw может
необязательно запускать **cache-ttl pruning**: он прореживает сессию после истечения TTL кэша,
затем сбрасывает окно кэша, чтобы последующие запросы могли повторно использовать
свежекэшированный контекст вместо повторного кэширования всей истории. Это снижает
стоимость записи в кэш, когда сессия простаивает дольше TTL.

Настройте это в [Конфигурация Gateway](/ru/gateway/configuration) и смотрите
подробности поведения в [Прореживание сессии](/ru/concepts/session-pruning).

Heartbeat может поддерживать кэш **теплым** во время простоев. Если TTL кэша вашей модели
равен `1h`, установка интервала Heartbeat чуть меньше этого значения (например, `55m`) может избежать
повторного кэширования полного prompt, снижая стоимость записи в кэш.

В многоагентных настройках можно держать одну общую конфигурацию модели и настраивать поведение кэша
для каждого агента через `agents.list[].params.cacheRetention`.

Полное руководство по каждому регулятору см. в [Кэширование prompt](/ru/reference/prompt-caching).

Для цен Anthropic API чтения из кэша значительно дешевле, чем входные
токены, а записи в кэш тарифицируются с более высоким множителем. Последние тарифы и множители TTL см. в ценах Anthropic
на кэширование prompt:
[https://docs.anthropic.com/docs/build-with-claude/prompt-caching](https://docs.anthropic.com/docs/build-with-claude/prompt-caching)

### Пример: поддерживать кэш 1h теплым с помощью Heartbeat

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
    heartbeat:
      every: "55m"
```

### Пример: смешанный трафик со стратегией кэша для каждого агента

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long" # default baseline for most agents
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m" # keep long cache warm for deep sessions
    - id: "alerts"
      params:
        cacheRetention: "none" # avoid cache writes for bursty notifications
```

`agents.list[].params` объединяется поверх `params` выбранной модели, поэтому вы можете
переопределить только `cacheRetention` и унаследовать остальные значения модели по умолчанию без изменений.

### Контекст Anthropic 1M

OpenClaw задает размер для GA-совместимых моделей Claude 4.x, таких как Opus 4.8, Opus 4.7, Opus 4.6 и
Sonnet 4.6, с окном контекста Anthropic 1M. Вам не нужен
`params.context1m: true` для этих моделей.

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        alias: opus
```

Старые конфигурации могут сохранять `context1m: true`, но OpenClaw больше не отправляет
устаревший beta-заголовок Anthropic `context-1m-2025-08-07` для этой настройки и
не расширяет неподдерживаемые более старые модели Claude до 1M.

Требование: учетные данные должны иметь право на использование длинного контекста. Если нет,
Anthropic отвечает ошибкой ограничения скорости на стороне провайдера для этого запроса.

Если вы аутентифицируете Anthropic с помощью токенов OAuth/подписки (`sk-ant-oat-*`),
OpenClaw сохраняет обязательные для OAuth beta-заголовки Anthropic, одновременно удаляя
устаревший beta `context-1m-*`, если он остается в старой конфигурации.

## Советы по снижению токенной нагрузки

- Используйте `/compact`, чтобы резюмировать длинные сеансы.
- Обрезайте большие выводы инструментов в своих рабочих процессах.
- Уменьшите `agents.defaults.imageMaxDimensionPx` для сеансов с большим количеством скриншотов.
- Делайте описания Skills короткими (список Skills внедряется в промпт).
- Предпочитайте модели меньшего размера для многословной исследовательской работы.

См. [Skills](/ru/tools/skills), где приведена точная формула накладных расходов списка Skills.
