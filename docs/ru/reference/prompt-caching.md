Кэширование промптов означает, что провайдер модели может повторно использовать неизмененные префиксы промпта (обычно системные/разработческие инструкции и другой стабильный контекст) между ходами вместо того, чтобы каждый раз обрабатывать их заново. OpenClaw нормализует использование провайдеров в `cacheRead` и `cacheWrite`, когда вышестоящий API напрямую предоставляет эти счетчики.

Поверхности статуса также могут восстанавливать счетчики кэша из самого свежего журнала
использования транскрипта, когда они отсутствуют в текущем снимке сессии, поэтому `/status` может продолжать
показывать строку кэша после частичной потери метаданных сессии. Существующие ненулевые текущие
значения кэша по-прежнему имеют приоритет над резервными значениями из транскрипта.

Почему это важно: ниже стоимость токенов, быстрее ответы и более предсказуемая производительность для долгоживущих сессий. Без кэширования повторяющиеся промпты оплачивают полную стоимость промпта на каждом ходе, даже когда большая часть ввода не изменилась.

Разделы ниже охватывают каждый связанный с кэшем параметр, который влияет на повторное использование промпта и стоимость токенов.

## Основные параметры

### `cacheRetention` (глобальное значение по умолчанию, модель и отдельный агент)

Задайте удержание кэша как глобальное значение по умолчанию для всех моделей:

```yaml
agents:
  defaults:
    params:
      cacheRetention: "long" # none | short | long
```

Переопределение для отдельной модели:

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "short" # none | short | long
```

Переопределение для отдельного агента:

```yaml
agents:
  list:
    - id: "alerts"
      params:
        cacheRetention: "none"
```

Порядок слияния конфигурации:

1. `agents.defaults.params` (глобальное значение по умолчанию — применяется ко всем моделям)
2. `agents.defaults.models["provider/model"].params` (переопределение для отдельной модели)
3. `agents.list[].params` (совпадающий идентификатор агента; переопределяет по ключу)

### `contextPruning.mode: "cache-ttl"`

Удаляет старый контекст результатов инструментов после окон TTL кэша, чтобы запросы после простоя не кэшировали заново чрезмерно большую историю.

```yaml
agents:
  defaults:
    contextPruning:
      mode: "cache-ttl"
      ttl: "1h"
```

Полное поведение см. в [Обрезка сессий](/ru/concepts/session-pruning).

### Поддержание прогрева Heartbeat

Heartbeat может поддерживать окна кэша прогретыми и уменьшать повторные записи кэша после периодов простоя.

```yaml
agents:
  defaults:
    heartbeat:
      every: "55m"
```

Heartbeat для отдельного агента поддерживается в `agents.list[].heartbeat`.

## Поведение провайдеров

### Anthropic (прямой API)

- `cacheRetention` поддерживается.
- Для профилей авторизации Anthropic с API-ключом OpenClaw задает `cacheRetention: "short"` для ссылок на модели Anthropic, если значение не указано.
- Нативные ответы Messages Anthropic предоставляют и `cache_read_input_tokens`, и `cache_creation_input_tokens`, поэтому OpenClaw может показывать и `cacheRead`, и `cacheWrite`.
- Для нативных запросов Anthropic `cacheRetention: "short"` соответствует стандартному 5-минутному эфемерному кэшу, а `cacheRetention: "long"` повышает TTL до 1 часа только на прямых хостах `api.anthropic.com`.

### OpenAI (прямой API)

- Кэширование промптов выполняется автоматически на поддерживаемых новых моделях. OpenClaw не нужно вставлять маркеры кэша на уровне блоков.
- OpenClaw использует `prompt_cache_key`, чтобы маршрутизация кэша оставалась стабильной между ходами. Прямые хосты OpenAI используют `prompt_cache_retention: "24h"`, когда выбран `cacheRetention: "long"`.
- OpenAI-совместимые провайдеры Completions получают `prompt_cache_key` только когда их конфигурация модели явно задает `compat.supportsPromptCacheKey: true`. Передача долгого удержания — отдельная возможность: явный `cacheRetention: "long"` отправляет `prompt_cache_retention: "24h"` только когда эта запись compat также поддерживает долгое удержание кэша. Провайдеры вроде Mistral могут включить ключи кэша, задав `compat.supportsLongCacheRetention: false`, чтобы подавить поле долгого удержания. `cacheRetention: "none"` подавляет оба поля.
- Ответы OpenAI предоставляют кэшированные токены промпта через `usage.prompt_tokens_details.cached_tokens` (или `input_tokens_details.cached_tokens` в событиях Responses API). OpenClaw сопоставляет это с `cacheRead`.
- Использование Responses GPT-5.6 также может предоставлять `input_tokens_details.cache_write_tokens`. OpenClaw сопоставляет это с `cacheWrite` и оценивает по тарифу записи в кэш модели; Responses, где поле отсутствует, оставляют `cacheWrite` равным `0`.
- OpenAI возвращает полезные заголовки трассировки и лимитов скорости, такие как `x-request-id`, `openai-processing-ms` и `x-ratelimit-*`, но учет попаданий в кэш должен поступать из полезной нагрузки usage, а не из заголовков.
- На практике OpenAI часто ведет себя как кэш начального префикса, а не как повторное использование полной движущейся истории в стиле Anthropic. Стабильные текстовые ходы с длинным префиксом могут достигать плато около `4864` кэшированных токенов в текущих live-пробах, тогда как насыщенные инструментами или MCP-подобные транскрипты часто выходят на плато около `4608` кэшированных токенов даже при точных повторах.

### Anthropic Vertex

- Модели Anthropic на Vertex AI (`anthropic-vertex/*`) поддерживают `cacheRetention` так же, как прямой Anthropic.
- `cacheRetention: "long"` соответствует реальному 1-часовому TTL кэша промптов на конечных точках Vertex AI.
- Удержание кэша по умолчанию для `anthropic-vertex` совпадает с прямыми значениями по умолчанию Anthropic.
- Запросы Vertex маршрутизируются через формирование кэша с учетом границ, чтобы повторное использование кэша оставалось согласованным с тем, что провайдеры фактически получают.

### Amazon Bedrock

- Ссылки на модели Anthropic Claude (`amazon-bedrock/*anthropic.claude*`) поддерживают явную сквозную передачу `cacheRetention`.
- Модели Bedrock не от Anthropic принудительно получают `cacheRetention: "none"` во время выполнения.

### Модели OpenRouter

Для ссылок на модели `openrouter/anthropic/*` OpenClaw вставляет Anthropic
`cache_control` в системные/разработческие блоки промпта, чтобы улучшить повторное
использование кэша промптов, только когда запрос все еще нацелен на проверенный маршрут OpenRouter
(`openrouter` на его стандартной конечной точке или любой провайдер/базовый URL, который разрешается
в `openrouter.ai`).

Для ссылок на модели `openrouter/deepseek/*`, `openrouter/moonshot*/*` и `openrouter/zai/*`
разрешен `contextPruning.mode: "cache-ttl"`, потому что OpenRouter
автоматически обрабатывает кэширование промптов на стороне провайдера. OpenClaw не вставляет
маркеры Anthropic `cache_control` в эти запросы.

Построение кэша DeepSeek выполняется по мере возможности и может занимать несколько секунд. Немедленный
последующий запрос все еще может показывать `cached_tokens: 0`; проверяйте повторным
запросом с тем же префиксом после короткой задержки и используйте `usage.prompt_tokens_details.cached_tokens`
как сигнал попадания в кэш.

Если вы перенаправите модель на произвольный OpenAI-совместимый URL прокси, OpenClaw
перестанет вставлять эти специфичные для OpenRouter маркеры кэша Anthropic.

### Другие провайдеры

Если провайдер не поддерживает этот режим кэша, `cacheRetention` не имеет эффекта.

### Прямой API Google Gemini

- Прямой транспорт Gemini (`api: "google-generative-ai"`) сообщает о попаданиях в кэш
  через вышестоящий `cachedContentTokenCount`; OpenClaw сопоставляет это с `cacheRead`.
- Когда `cacheRetention` задан для прямой модели Gemini, OpenClaw автоматически
  создает, повторно использует и обновляет ресурсы `cachedContents` для системных промптов
  при запусках Google AI Studio. Это означает, что вам больше не нужно заранее создавать
  дескриптор кэшированного содержимого вручную.
- Вы все еще можете передать уже существующий дескриптор кэшированного содержимого Gemini как
  `params.cachedContent` (или устаревший `params.cached_content`) в настроенной
  модели.
- Это отдельно от кэширования префиксов промптов Anthropic/OpenAI. Для Gemini
  OpenClaw управляет нативным для провайдера ресурсом `cachedContents`, а не
  вставляет маркеры кэша в запрос.

### Использование Gemini CLI

- Вывод Gemini CLI `stream-json` может показывать попадания в кэш через `stats.cached`;
  OpenClaw сопоставляет это с `cacheRead`. Устаревшие переопределения `--output-format json` используют
  ту же нормализацию использования.
- Если CLI не передает прямое значение `stats.input`, OpenClaw выводит входные токены
  из `stats.input_tokens - stats.cached`.
- Это только нормализация использования. Это не означает, что OpenClaw создает
  маркеры кэша промптов в стиле Anthropic/OpenAI для Gemini CLI.

## Граница кэша системного промпта

OpenClaw разделяет системный промпт на **стабильный префикс** и **изменчивый
суффикс**, разделенные внутренней границей префикса кэша. Содержимое выше
границы (определения инструментов, метаданные Skills, файлы рабочей области и другой
относительно статичный контекст) упорядочивается так, чтобы оно оставалось побайтно идентичным между ходами.
Содержимому ниже границы (например, `HEARTBEAT.md`, временным меткам времени выполнения и
другим метаданным отдельного хода) разрешено изменяться без инвалидирования кэшированного
префикса.

Ключевые проектные решения:

- Стабильные файлы проектного контекста рабочей области упорядочиваются перед `HEARTBEAT.md`, чтобы
  изменения Heartbeat не разрушали стабильный префикс.
- Граница применяется при формировании транспорта семейства Anthropic, семейства OpenAI, Google и
  CLI, чтобы все поддерживаемые провайдеры получали пользу от той же стабильности префикса.
- Запросы Codex Responses и Anthropic Vertex маршрутизируются через
  формирование кэша с учетом границ, чтобы повторное использование кэша оставалось согласованным с тем, что провайдеры
  фактически получают.
- Отпечатки системных промптов нормализуются (пробелы, окончания строк,
  контекст, добавленный хуками, порядок возможностей времени выполнения), чтобы семантически неизмененные
  промпты совместно использовали KV/кэш между ходами.

Если вы видите неожиданные всплески `cacheWrite` после изменения конфигурации или рабочей области,
проверьте, попадает ли изменение выше или ниже границы кэша. Перемещение
изменчивого содержимого ниже границы (или его стабилизация) часто решает
проблему.

## Защитные механизмы стабильности кэша OpenClaw

OpenClaw также сохраняет несколько чувствительных к кэшу форм полезной нагрузки детерминированными до того,
как запрос достигнет провайдера:

- Каталоги инструментов Bundle MCP детерминированно сортируются перед
  регистрацией инструментов, поэтому изменения порядка `listTools()` не меняют блок инструментов и
  не разрушают префиксы кэша промптов.
- Устаревшие сессии с сохраненными блоками изображений сохраняют **3 самых последних
  завершенных хода** без изменений; более старые уже обработанные блоки изображений могут быть
  заменены маркером, чтобы последующие запросы с большим количеством изображений не продолжали повторно отправлять крупные
  устаревшие полезные нагрузки.

## Шаблоны настройки

### Смешанный трафик (рекомендуемое значение по умолчанию)

Сохраняйте долгоживущую базовую линию на основном агенте, отключайте кэширование на всплесковых агентах-уведомителях:

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m"
    - id: "alerts"
      params:
        cacheRetention: "none"
```

### Базовая линия с приоритетом стоимости

- Задайте базовое `cacheRetention: "short"`.
- Включите `contextPruning.mode: "cache-ttl"`.
- Держите Heartbeat ниже вашего TTL только для агентов, которым полезны прогретые кэши.

## Диагностика кэша

OpenClaw предоставляет выделенную диагностику трассировки кэша для встроенных запусков агентов.

Для обычной пользовательской диагностики `/status` и другие сводки использования могут использовать
последнюю запись использования транскрипта как резервный источник для `cacheRead` /
`cacheWrite`, когда в текущей записи сессии нет этих счетчиков.

## Live-регрессионные тесты

OpenClaw поддерживает один объединенный live-регрессионный контроль кэша для повторяющихся префиксов, ходов с инструментами, ходов с изображениями, MCP-подобных транскриптов инструментов и контрольного случая Anthropic без кэша.

- `src/agents/live-cache-regression.live.test.ts`
- `src/agents/live-cache-regression-baseline.ts`

Запустите узкий live-контроль с помощью:

```sh
OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache
```

Базовый файл хранит последние наблюдавшиеся live-числа, а также специфичные для провайдера нижние пороги регрессии, используемые тестом.
Runner также использует свежие ID сессий и пространства имен промптов для каждого запуска, чтобы предыдущее состояние кеша не загрязняло текущую выборку регрессии.

Эти тесты намеренно не используют одинаковые критерии успеха для всех провайдеров.

### Live-ожидания Anthropic

- Ожидаются явные warmup-записи через `cacheWrite`.
- Ожидается почти полное повторное использование истории на повторных ходах, потому что управление кешем Anthropic продвигает контрольную точку кеша по разговору.
- Текущие live-проверки по-прежнему используют высокие пороги доли попаданий для стабильных, инструментальных и графических путей.

### Live-ожидания OpenAI

- Ожидается только `cacheRead`. `cacheWrite` остается `0`.
- Рассматривайте повторное использование кеша на повторных ходах как специфичное для провайдера плато, а не как Anthropic-подобное подвижное повторное использование полной истории.
- Текущие live-проверки используют консервативные нижние пороги, выведенные из наблюдавшегося live-поведения на `gpt-5.4-mini`:
  - стабильный префикс: `cacheRead >= 4608`, доля попаданий `>= 0.90`
  - транскрипт инструмента: `cacheRead >= 4096`, доля попаданий `>= 0.85`
  - транскрипт изображения: `cacheRead >= 3840`, доля попаданий `>= 0.82`
  - транскрипт в стиле MCP: `cacheRead >= 4096`, доля попаданий `>= 0.85`

Свежая комбинированная live-верификация от 2026-04-04 дала:

- стабильный префикс: `cacheRead=4864`, доля попаданий `0.966`
- транскрипт инструмента: `cacheRead=4608`, доля попаданий `0.896`
- транскрипт изображения: `cacheRead=4864`, доля попаданий `0.954`
- транскрипт в стиле MCP: `cacheRead=4608`, доля попаданий `0.891`

Недавнее локальное время выполнения комбинированного gate по wall-clock составляло около `88s`.

Почему проверки отличаются:

- Anthropic предоставляет явные контрольные точки кеша и подвижное повторное использование истории разговора.
- Кеширование промптов OpenAI по-прежнему чувствительно к точному префиксу, но эффективный переиспользуемый префикс в live-трафике Responses может выходить на плато раньше, чем полный промпт.
- Поэтому сравнение Anthropic и OpenAI по единому межпровайдерному процентному порогу создает ложные регрессии.

### Конфигурация `diagnostics.cacheTrace`

```yaml
diagnostics:
  cacheTrace:
    enabled: true
    filePath: "~/.openclaw/logs/cache-trace.jsonl" # optional
    includeMessages: false # default true
    includePrompt: false # default true
    includeSystem: false # default true
```

Значения по умолчанию:

- `filePath`: `$OPENCLAW_STATE_DIR/logs/cache-trace.jsonl`
- `includeMessages`: `true`
- `includePrompt`: `true`
- `includeSystem`: `true`

### Переключатели окружения (разовая отладка)

- `OPENCLAW_CACHE_TRACE=1` включает трассировку кеша.
- `OPENCLAW_CACHE_TRACE_FILE=/path/to/cache-trace.jsonl` переопределяет путь вывода.
- `OPENCLAW_CACHE_TRACE_MESSAGES=0|1` переключает захват полного payload сообщений.
- `OPENCLAW_CACHE_TRACE_PROMPT=0|1` переключает захват текста промпта.
- `OPENCLAW_CACHE_TRACE_SYSTEM=0|1` переключает захват системного промпта.

### Что проверять

- События трассировки кеша имеют формат JSONL и включают staged-снимки вроде `session:loaded`, `prompt:before`, `stream:context` и `session:after`.
- Влияние кешированных токенов за ход видно в обычных поверхностях использования через `cacheRead` и `cacheWrite` (например, `/usage tokens`, `/status`, сводки использования сессии и пользовательские макеты `messages.usageTemplate`).
- Для Anthropic ожидайте и `cacheRead`, и `cacheWrite`, когда кеширование активно.
- Для OpenAI ожидайте `cacheRead` при попаданиях в кеш. GPT-5.6 Responses также может сообщать `cacheWrite` при записи сегментов промпта; другие payload Responses, которые не содержат счетчик записи, оставляют его равным `0`.
- Если нужна трассировка запросов, логируйте ID запросов и заголовки rate-limit отдельно от метрик кеша. Текущий вывод cache-trace в OpenClaw сосредоточен на форме промпта/сессии и нормализованном использовании токенов, а не на сырых заголовках ответов провайдера.

## Быстрое устранение неполадок

- Высокий `cacheWrite` на большинстве ходов: проверьте изменчивые входные данные системного промпта и убедитесь, что модель/провайдер поддерживает ваши настройки кеша.
- Высокий `cacheWrite` на Anthropic: часто означает, что контрольная точка кеша попадает на содержимое, которое меняется при каждом запросе.
- Низкий `cacheRead` OpenAI: убедитесь, что стабильный префикс находится в начале, повторяющийся префикс содержит не менее 1024 токенов, а один и тот же `prompt_cache_key` повторно используется для ходов, которые должны разделять кеш.
- Нет эффекта от `cacheRetention`: подтвердите, что ключ модели совпадает с `agents.defaults.models["provider/model"]`.
- Запросы Bedrock Nova/Mistral с настройками кеша: ожидаемое runtime-принудительное значение `none`.
