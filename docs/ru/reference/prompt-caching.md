Кэширование промптов означает, что провайдер модели может повторно использовать неизменившиеся префиксы промпта (обычно системные/разработческие инструкции и другой стабильный контекст) между ходами вместо повторной обработки каждый раз. OpenClaw нормализует usage провайдера в `cacheRead` и `cacheWrite`, когда upstream API напрямую предоставляет эти счетчики.

Поверхности статуса также могут восстанавливать счетчики кэша из самого свежего журнала usage в transcript, когда они отсутствуют в live-снимке сессии, поэтому `/status` может продолжать показывать строку кэша после частичной потери метаданных сессии. Существующие ненулевые live-значения кэша по-прежнему имеют приоритет над fallback-значениями из transcript.

Почему это важно: ниже стоимость токенов, быстрее ответы и более предсказуемая производительность для длительных сессий. Без кэширования повторяющиеся промпты оплачивают полную стоимость промпта на каждом ходе, даже если большая часть ввода не изменилась.

Разделы ниже описывают все параметры, связанные с кэшем, которые влияют на повторное использование промптов и стоимость токенов.

Справки провайдеров:

- Кэширование промптов Anthropic: [https://platform.claude.com/docs/en/build-with-claude/prompt-caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)
- Кэширование промптов OpenAI: [https://developers.openai.com/api/docs/guides/prompt-caching](https://developers.openai.com/api/docs/guides/prompt-caching)
- Заголовки OpenAI API и идентификаторы запросов: [https://developers.openai.com/api/reference/overview](https://developers.openai.com/api/reference/overview)
- Идентификаторы запросов и ошибки Anthropic: [https://platform.claude.com/docs/en/api/errors](https://platform.claude.com/docs/en/api/errors)

## Основные параметры

### `cacheRetention` (глобальное значение по умолчанию, модель и отдельный агент)

Задайте удержание кэша как глобальное значение по умолчанию для всех моделей:

```yaml
agents:
  defaults:
    params:
      cacheRetention: "long" # none | short | long
```

Переопределение для модели:

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "short" # none | short | long
```

Переопределение для отдельного агента:

```yaml
agents:
  list:
    - id: "alerts"
      params:
        cacheRetention: "none"
```

Порядок слияния конфигурации:

1. `agents.defaults.params` (глобальное значение по умолчанию — применяется ко всем моделям)
2. `agents.defaults.models["provider/model"].params` (переопределение для модели)
3. `agents.list[].params` (соответствующий id агента; переопределяет по ключу)

### `contextPruning.mode: "cache-ttl"`

Обрезает старый контекст результатов инструментов после окон TTL кэша, чтобы запросы после простоя не кэшировали повторно чрезмерно большую историю.

```yaml
agents:
  defaults:
    contextPruning:
      mode: "cache-ttl"
      ttl: "1h"
```

Полное поведение см. в [Обрезка сессий](/ru/concepts/session-pruning).

### Поддержание прогретого Heartbeat

Heartbeat может поддерживать окна кэша прогретыми и уменьшать повторные записи кэша после периодов простоя.

```yaml
agents:
  defaults:
    heartbeat:
      every: "55m"
```

Heartbeat для отдельного агента поддерживается в `agents.list[].heartbeat`.

## Поведение провайдеров

### Anthropic (прямой API)

- `cacheRetention` поддерживается.
- Для профилей аутентификации Anthropic с API-ключом OpenClaw задает `cacheRetention: "short"` для ссылок на модели Anthropic, если значение не задано.
- Нативные ответы Anthropic Messages предоставляют и `cache_read_input_tokens`, и `cache_creation_input_tokens`, поэтому OpenClaw может показывать и `cacheRead`, и `cacheWrite`.
- Для нативных запросов Anthropic `cacheRetention: "short"` соответствует стандартному 5-минутному ephemeral-кэшу, а `cacheRetention: "long"` повышает TTL до 1 часа только на прямых хостах `api.anthropic.com`.

### OpenAI (прямой API)

- Кэширование промптов выполняется автоматически на поддерживаемых актуальных моделях. OpenClaw не нужно внедрять маркеры кэша на уровне блоков.
- OpenClaw использует `prompt_cache_key`, чтобы сохранять стабильную маршрутизацию кэша между ходами. Прямые хосты OpenAI используют `prompt_cache_retention: "24h"`, когда выбран `cacheRetention: "long"`.
- OpenAI-совместимые провайдеры Completions получают `prompt_cache_key` только когда их конфигурация модели явно задает `compat.supportsPromptCacheKey: true`. Проброс долгого удержания — отдельная возможность: явный `cacheRetention: "long"` отправляет `prompt_cache_retention: "24h"` только когда эта запись compat также поддерживает долгое удержание кэша. Провайдеры вроде Mistral могут включить ключи кэша, установив `compat.supportsLongCacheRetention: false`, чтобы подавить поле долгого удержания. `cacheRetention: "none"` подавляет оба поля.
- Ответы OpenAI предоставляют кэшированные токены промпта через `usage.prompt_tokens_details.cached_tokens` (или `input_tokens_details.cached_tokens` в событиях Responses API). OpenClaw сопоставляет это с `cacheRead`.
- Usage GPT-5.6 Responses также может предоставлять `input_tokens_details.cache_write_tokens`. OpenClaw сопоставляет это с `cacheWrite` и оценивает по тарифу записи кэша модели; Responses, которые пропускают поле, оставляют `cacheWrite` равным `0`.
- OpenAI возвращает полезные заголовки трассировки и лимитов скорости, такие как `x-request-id`, `openai-processing-ms` и `x-ratelimit-*`, но учет cache-hit должен браться из payload usage, а не из заголовков.
- На практике OpenAI часто ведет себя как кэш начального префикса, а не как Anthropic-подобное скользящее повторное использование всей истории. Стабильные ходы с длинным префиксом текста могут выходить примерно на плато `4864` кэшированных токенов в текущих live-пробах, тогда как transcript с большим количеством инструментов или в MCP-стиле часто выходят на плато около `4608` кэшированных токенов даже при точных повторах.

### Anthropic Vertex

- Модели Anthropic в Vertex AI (`anthropic-vertex/*`) поддерживают `cacheRetention` так же, как прямой Anthropic.
- `cacheRetention: "long"` соответствует реальному 1-часовому TTL кэша промпта на endpoint Vertex AI.
- Удержание кэша по умолчанию для `anthropic-vertex` совпадает со значениями по умолчанию прямого Anthropic.
- Запросы Vertex маршрутизируются через формирование кэша с учетом границ, чтобы повторное использование кэша оставалось согласованным с тем, что провайдеры фактически получают.

### Amazon Bedrock

- Ссылки на модели Anthropic Claude (`amazon-bedrock/*anthropic.claude*`) поддерживают явный сквозной проброс `cacheRetention`.
- Не-Anthropic модели Bedrock принудительно получают `cacheRetention: "none"` во время выполнения.

### Модели OpenRouter

Для ссылок на модели `openrouter/anthropic/*` OpenClaw внедряет Anthropic `cache_control` в системные/разработческие блоки промпта, чтобы улучшить повторное использование кэша промпта, только когда запрос все еще нацелен на проверенный маршрут OpenRouter (`openrouter` на его endpoint по умолчанию или любой провайдер/base URL, который разрешается в `openrouter.ai`).

Для ссылок на модели `openrouter/deepseek/*`, `openrouter/moonshot*/*` и `openrouter/zai/*` разрешен `contextPruning.mode: "cache-ttl"`, потому что OpenRouter автоматически обрабатывает кэширование промптов на стороне провайдера. OpenClaw не внедряет Anthropic-маркеры `cache_control` в эти запросы.

Построение кэша DeepSeek выполняется по возможности и может занимать несколько секунд. Немедленный следующий запрос все еще может показать `cached_tokens: 0`; проверяйте повторным запросом с тем же префиксом после небольшой задержки и используйте `usage.prompt_tokens_details.cached_tokens` как сигнал cache-hit.

Если вы перенаправляете модель на произвольный OpenAI-совместимый proxy URL, OpenClaw прекращает внедрять эти OpenRouter-специфичные маркеры кэша Anthropic.

### Другие провайдеры

Если провайдер не поддерживает этот режим кэша, `cacheRetention` не имеет эффекта.

### Прямой API Google Gemini

- Прямой транспорт Gemini (`api: "google-generative-ai"`) сообщает попадания в кэш через upstream `cachedContentTokenCount`; OpenClaw сопоставляет это с `cacheRead`.
- Когда `cacheRetention` задан для прямой модели Gemini, OpenClaw автоматически создает, повторно использует и обновляет ресурсы `cachedContents` для системных промптов в запусках Google AI Studio. Это означает, что вам больше не нужно заранее вручную создавать handle cached-content.
- Вы все еще можете передать уже существующий handle cached-content Gemini как `params.cachedContent` (или устаревший `params.cached_content`) в настроенной модели.
- Это отдельно от кэширования префикса промпта Anthropic/OpenAI. Для Gemini OpenClaw управляет provider-native ресурсом `cachedContents`, а не внедряет маркеры кэша в запрос.

### Использование Gemini CLI

- Вывод Gemini CLI `stream-json` может показывать попадания в кэш через `stats.cached`; OpenClaw сопоставляет это с `cacheRead`. Устаревшие переопределения `--output-format json` используют ту же нормализацию usage.
- Если CLI пропускает прямое значение `stats.input`, OpenClaw выводит входные токены из `stats.input_tokens - stats.cached`.
- Это только нормализация usage. Это не означает, что OpenClaw создает Anthropic/OpenAI-подобные маркеры кэша промпта для Gemini CLI.

## Граница кэша системного промпта

OpenClaw разделяет системный промпт на **стабильный префикс** и **изменчивый суффикс**, разделенные внутренней границей cache-prefix. Содержимое выше границы (определения инструментов, метаданные Skills, файлы рабочей области и другой относительно статичный контекст) упорядочивается так, чтобы оставаться побайтно идентичным между ходами. Содержимому ниже границы (например, `HEARTBEAT.md`, runtime-временные метки и другие метаданные каждого хода) разрешено изменяться без инвалидирования кэшированного префикса.

Ключевые проектные решения:

- Стабильные файлы project-context рабочей области упорядочиваются перед `HEARTBEAT.md`, чтобы изменения heartbeat не разрушали стабильный префикс.
- Граница применяется при формировании транспорта для семейств Anthropic, OpenAI, Google и CLI, чтобы все поддерживаемые провайдеры получали пользу от той же стабильности префикса.
- Запросы Codex Responses и Anthropic Vertex маршрутизируются через формирование кэша с учетом границ, чтобы повторное использование кэша оставалось согласованным с тем, что провайдеры фактически получают.
- Отпечатки системного промпта нормализуются (пробелы, окончания строк, контекст, добавленный hook, порядок runtime-возможностей), чтобы семантически неизменившиеся промпты разделяли KV/кэш между ходами.

Если вы видите неожиданные всплески `cacheWrite` после изменения конфигурации или рабочей области, проверьте, попадает ли изменение выше или ниже границы кэша. Перенос изменчивого содержимого ниже границы (или его стабилизация) часто решает проблему.

## Защиты стабильности кэша OpenClaw

OpenClaw также поддерживает несколько чувствительных к кэшу форм payload детерминированными до того, как запрос достигнет провайдера:

- Каталоги инструментов Bundle MCP детерминированно сортируются перед регистрацией инструментов, поэтому изменения порядка `listTools()` не меняют блок инструментов и не разрушают префиксы кэша промпта.
- Устаревшие сессии с сохраненными блоками изображений сохраняют **3 самых последних завершенных хода** без изменений; более старые уже обработанные блоки изображений могут заменяться маркером, чтобы follow-up с большим количеством изображений не продолжали повторно отправлять крупные устаревшие payload.

## Паттерны настройки

### Смешанный трафик (рекомендуемое значение по умолчанию)

Сохраняйте долгоживущую базовую линию на основном агенте, отключайте кэширование на всплесковых агентах-уведомителях:

```yaml
agents:
  defaults:
    model:
      primary: "anthropic/claude-opus-4-6"
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "long"
  list:
    - id: "research"
      default: true
      heartbeat:
        every: "55m"
    - id: "alerts"
      params:
        cacheRetention: "none"
```

### Базовая линия с приоритетом стоимости

- Задайте базовый `cacheRetention: "short"`.
- Включите `contextPruning.mode: "cache-ttl"`.
- Держите Heartbeat ниже вашего TTL только для агентов, которым полезны прогретые кэши.

## Диагностика кэша

OpenClaw предоставляет специализированную диагностику cache-trace для встроенных запусков агентов.

Для обычной пользовательской диагностики `/status` и другие сводки usage могут использовать последнюю запись usage в transcript как fallback-источник для `cacheRead` / `cacheWrite`, когда в live-записи сессии нет этих счетчиков.

## Live-регрессионные тесты

OpenClaw поддерживает один объединенный live-регрессионный gate кэша для повторяющихся префиксов, ходов с инструментами, ходов с изображениями, transcript инструментов в MCP-стиле и Anthropic-контроля без кэша.

- `src/agents/live-cache-regression.live.test.ts`
- `src/agents/live-cache-regression-baseline.ts`

Запустите узкий live gate командой:

```sh
OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache
```

Файл baseline хранит самые последние наблюдавшиеся live-числа, а также специфичные для провайдера минимальные пороги регрессии, используемые тестом.
Runner также использует свежие идентификаторы сессий и пространства имен prompt для каждого запуска, чтобы предыдущее состояние кэша не загрязняло текущую регрессионную выборку.

Эти тесты намеренно не используют одинаковые критерии успешности для всех провайдеров.

### Live-ожидания Anthropic

- Ожидаются явные warmup-записи через `cacheWrite`.
- Ожидается почти полное повторное использование истории при повторных ходах, потому что управление кэшем Anthropic продвигает точку останова кэша по разговору.
- Текущие live-утверждения по-прежнему используют высокие пороги hit rate для стабильных путей, путей с инструментами и изображениями.

### Live-ожидания OpenAI

- Ожидается только `cacheRead`. `cacheWrite` остается `0`.
- Повторное использование кэша при повторных ходах рассматривается как специфичное для провайдера плато, а не как движущееся повторное использование полной истории в стиле Anthropic.
- Текущие live-утверждения используют консервативные нижние пороги, выведенные из наблюдаемого live-поведения на `gpt-5.4-mini`:
  - стабильный префикс: `cacheRead >= 4608`, hit rate `>= 0.90`
  - transcript инструмента: `cacheRead >= 4096`, hit rate `>= 0.85`
  - transcript изображения: `cacheRead >= 3840`, hit rate `>= 0.82`
  - transcript в стиле MCP: `cacheRead >= 4096`, hit rate `>= 0.85`

Свежая комбинированная live-проверка 2026-04-04 дала:

- стабильный префикс: `cacheRead=4864`, hit rate `0.966`
- transcript инструмента: `cacheRead=4608`, hit rate `0.896`
- transcript изображения: `cacheRead=4864`, hit rate `0.954`
- transcript в стиле MCP: `cacheRead=4608`, hit rate `0.891`

Недавнее локальное wall-clock время для комбинированного gate составляло около `88s`.

Почему утверждения различаются:

- Anthropic предоставляет явные точки останова кэша и движущееся повторное использование истории разговора.
- Кэширование prompt в OpenAI по-прежнему чувствительно к точному префиксу, но эффективный повторно используемый префикс в live-трафике Responses может выходить на плато раньше, чем полный prompt.
- Поэтому сравнение Anthropic и OpenAI по одному межпровайдерному процентному порогу создает ложные регрессии.

### Конфигурация `diagnostics.cacheTrace`

```yaml
diagnostics:
  cacheTrace:
    enabled: true
    filePath: "~/.openclaw/logs/cache-trace.jsonl" # optional
    includeMessages: false # default true
    includePrompt: false # default true
    includeSystem: false # default true
```

Значения по умолчанию:

- `filePath`: `$OPENCLAW_STATE_DIR/logs/cache-trace.jsonl`
- `includeMessages`: `true`
- `includePrompt`: `true`
- `includeSystem`: `true`

### Переключатели env для разовой отладки

- `OPENCLAW_CACHE_TRACE=1` включает трассировку кэша.
- `OPENCLAW_CACHE_TRACE_FILE=/path/to/cache-trace.jsonl` переопределяет путь вывода.
- `OPENCLAW_CACHE_TRACE_MESSAGES=0|1` переключает захват полного payload сообщений.
- `OPENCLAW_CACHE_TRACE_PROMPT=0|1` переключает захват текста prompt.
- `OPENCLAW_CACHE_TRACE_SYSTEM=0|1` переключает захват системного prompt.

### Что проверять

- События трассировки кэша имеют формат JSONL и включают staged-снимки, такие как `session:loaded`, `prompt:before`, `stream:context` и `session:after`.
- Влияние токенов кэша за ход видно в обычных поверхностях использования через `cacheRead` и `cacheWrite` (например, `/usage full` и сводки использования сессии).
- Для Anthropic ожидаются и `cacheRead`, и `cacheWrite`, когда кэширование активно.
- Для OpenAI ожидается `cacheRead` при попаданиях в кэш. GPT-5.6 Responses также может сообщать `cacheWrite`, пока записываются сегменты prompt; другие payload Responses, которые не включают счетчик записи, оставляют его равным `0`.
- Если вам нужна трассировка запросов, логируйте идентификаторы запросов и заголовки rate-limit отдельно от метрик кэша. Текущий вывод cache-trace в OpenClaw сосредоточен на форме prompt/сессии и нормализованном использовании токенов, а не на сырых заголовках ответа провайдера.

## Быстрое устранение неполадок

- Высокий `cacheWrite` на большинстве ходов: проверьте изменчивые входные данные системного prompt и убедитесь, что модель/провайдер поддерживает ваши настройки кэша.
- Высокий `cacheWrite` на Anthropic: часто означает, что точка останова кэша попадает на контент, который меняется при каждом запросе.
- Низкий `cacheRead` OpenAI: убедитесь, что стабильный префикс находится в начале, повторяемый префикс содержит не менее 1024 токенов, а один и тот же `prompt_cache_key` повторно используется для ходов, которые должны совместно использовать кэш.
- Нет эффекта от `cacheRetention`: подтвердите, что ключ модели соответствует `agents.defaults.models["provider/model"]`.
- Запросы Bedrock Nova/Mistral с настройками кэша: ожидаемое принудительное runtime-значение `none`.
