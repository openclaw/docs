Если не указано иное, все настройки поиска по памяти находятся в `agents.defaults.memorySearch` файла `openclaw.json` (либо в переопределении `agents.list[].memorySearch` для отдельного агента).

<Note>
Если вам нужен переключатель функции **Active Memory** и конфигурация подагента, они находятся в `plugins.entries.active-memory`, а не в `memorySearch`.

Active Memory использует модель с двумя условиями:

1. Plugin должен быть включён и нацелен на идентификатор текущего агента;
2. запрос должен относиться к подходящему интерактивному постоянному сеансу чата.

Описание модели активации, конфигурации, принадлежащей Plugin, сохранения расшифровок и безопасной схемы развёртывания см. в разделе [Active Memory](/ru/concepts/active-memory).
</Note>

## Выбор провайдера

| Ключ      | Тип       | По умолчанию         | Описание                                                                                                                                                                                                                                                                                                                             |
| ---------- | --------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `enabled`  | `boolean` | `true`               | Включает или отключает поиск по памяти                                                                                                                                                                                                                                                                                               |
| `provider` | `string`  | `"openai"`           | Идентификатор адаптера эмбеддингов, например `bedrock`, `deepinfra`, `gemini`, `github-copilot`, `local`, `mistral`, `ollama`, `openai`, `openai-compatible` или `voyage`; также можно указать настроенный `models.providers.<id>`, чей `api` ссылается на адаптер эмбеддингов памяти или совместимый с OpenAI API модели |
| `model`    | `string`  | значение провайдера  | Имя модели эмбеддингов                                                                                                                                                                                                                                                                                                               |
| `fallback` | `string`  | `"none"`             | Идентификатор резервного адаптера на случай сбоя основного                                                                                                                                                                                                                                                                            |

Если `provider` не задан, OpenClaw использует эмбеддинги OpenAI. Задайте `provider`
явно, чтобы использовать Bedrock, DeepInfra, Gemini, GitHub Copilot, Mistral, Ollama,
Voyage, локальную модель GGUF или совместимую с OpenAI конечную точку `/v1/embeddings`.
Устаревшие конфигурации, в которых по-прежнему указано `provider: "auto"`, разрешаются в `openai`.

<Warning>
Изменение провайдера или модели эмбеддингов, настроек провайдера, источников, области действия,
разбиения на фрагменты или токенизатора может сделать существующий векторный индекс SQLite несовместимым.
OpenClaw приостанавливает векторный поиск и сообщает предупреждение о несовпадении идентичности индекса,
вместо того чтобы автоматически повторно создавать эмбеддинги для всех данных. Когда будете готовы,
перестройте индекс с помощью `openclaw memory status --index --agent <id>` или
`openclaw memory index --force --agent <id>`.
</Warning>

Если `provider` не задан, присутствует устаревшее значение `provider: "auto"` или
`provider: "none"` намеренно выбирает режим только с FTS, извлечение данных из памяти всё равно
может использовать лексическое ранжирование FTS, когда эмбеддинги недоступны.

Явно заданные нелокальные провайдеры завершают работу с ошибкой при недоступности. Если для
`memorySearch.provider` задан конкретный удалённый провайдер, например Bedrock, DeepInfra,
Gemini, GitHub Copilot, LM Studio, Mistral, Ollama, OpenAI, Voyage или пользовательский
провайдер, совместимый с OpenAI, и он недоступен во время выполнения, `memory_search`
возвращает результат о недоступности вместо неявного перехода к извлечению только через FTS.
Исправьте конфигурацию провайдера или аутентификации, переключитесь на доступного провайдера
либо задайте `provider: "none"`, если хотите намеренно использовать извлечение только через FTS.

### Пользовательские идентификаторы провайдеров

`memorySearch.provider` может ссылаться на пользовательскую запись `models.providers.<id>` для специализированных адаптеров провайдеров памяти, таких как `ollama`, или для совместимых с OpenAI API моделей, таких как `openai-responses` / `openai-completions`. OpenClaw определяет владельца `api` этого провайдера для адаптера эмбеддингов, сохраняя пользовательский идентификатор провайдера для обработки конечной точки, аутентификации и префикса модели. Это позволяет конфигурациям с несколькими GPU или узлами выделить для эмбеддингов памяти определённую локальную конечную точку:

```json5
{
  models: {
    providers: {
      "ollama-5080": {
        api: "ollama",
        baseUrl: "http://gpu-box.local:11435",
        apiKey: "ollama-local",
        models: [{ id: "qwen3-embedding:0.6b", name: "Qwen3 Embedding 0.6B" }],
      },
    },
  },
  agents: {
    defaults: {
      memorySearch: {
        provider: "ollama-5080",
        model: "qwen3-embedding:0.6b",
      },
    },
  },
}
```

### Определение ключа API

Для удалённых эмбеддингов требуется ключ API. Вместо него Bedrock использует стандартную цепочку учётных данных AWS SDK (роли экземпляров, SSO, ключи доступа или ключ API Bedrock).

| Провайдер      | Переменная окружения                                 | Ключ конфигурации                    |
| -------------- | ---------------------------------------------------- | ------------------------------------ |
| Bedrock        | Цепочка учётных данных AWS или `AWS_BEARER_TOKEN_BEDROCK` | Ключ API не требуется                |
| DeepInfra      | `DEEPINFRA_API_KEY`                                  | `models.providers.deepinfra.apiKey`  |
| Gemini         | `GEMINI_API_KEY`                                     | `models.providers.google.apiKey`     |
| GitHub Copilot | `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, `GITHUB_TOKEN`   | Профиль аутентификации через вход с устройства |
| Mistral        | `MISTRAL_API_KEY`                                    | `models.providers.mistral.apiKey`    |
| Ollama         | `OLLAMA_API_KEY` (заполнитель)                       | --                                   |
| OpenAI         | `OPENAI_API_KEY`                                     | `models.providers.openai.apiKey`     |
| Voyage         | `VOYAGE_API_KEY`                                     | `models.providers.voyage.apiKey`     |

<Note>
OAuth Codex распространяется только на чат и дополнения и не подходит для запросов эмбеддингов.
</Note>

## Конфигурация удалённой конечной точки

Используйте `provider: "openai-compatible"` для универсального совместимого с OpenAI
сервера `/v1/embeddings`, который не должен наследовать глобальные учётные данные чата OpenAI.

<ParamField path="remote.baseUrl" type="string">
  Пользовательский базовый URL API.
</ParamField>
<ParamField path="remote.apiKey" type="string">
  Переопределение ключа API.
</ParamField>
<ParamField path="remote.headers" type="object">
  Дополнительные заголовки HTTP (объединяются со значениями провайдера по умолчанию).
</ParamField>

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai-compatible",
        model: "text-embedding-3-small",
        remote: {
          baseUrl: "https://api.example.com/v1/",
          apiKey: "YOUR_KEY",
        },
      },
    },
  },
}
```

## Конфигурация для отдельных провайдеров

<AccordionGroup>
  <Accordion title="Gemini">
    | Ключ                   | Тип      | По умолчанию          | Описание                                      |
    | ---------------------- | -------- | --------------------- | --------------------------------------------- |
    | `model`                | `string` | `gemini-embedding-001` | Также поддерживает `gemini-embedding-2-preview` |
    | `outputDimensionality` | `number` | `3072`                | Для Embedding 2: 768, 1536 или 3072           |

    <Warning>
    Изменение модели или `outputDimensionality` меняет идентичность индекса. OpenClaw
    приостанавливает векторный поиск, пока вы явно не перестроите индекс памяти.
    </Warning>

  </Accordion>
  <Accordion title="Типы входных данных, совместимые с OpenAI">
    Совместимые с OpenAI конечные точки эмбеддингов могут использовать специфичные для провайдера поля запроса `input_type`. Это полезно для асимметричных моделей эмбеддингов, которым требуются разные метки для эмбеддингов запросов и документов.

    | Ключ                | Тип      | По умолчанию | Описание                                                       |
    | ------------------- | -------- | ------------ | -------------------------------------------------------------- |
    | `inputType`         | `string` | не задано    | Общее значение `input_type` для эмбеддингов запросов и документов |
    | `queryInputType`    | `string` | не задано    | Значение `input_type` при запросе; переопределяет `inputType`   |
    | `documentInputType` | `string` | не задано    | Значение `input_type` для индекса или документа; переопределяет `inputType` |

    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            provider: "openai-compatible",
            remote: {
              baseUrl: "https://embeddings.example/v1",
              apiKey: "${EMBEDDINGS_API_KEY}",
            },
            model: "asymmetric-embedder",
            queryInputType: "query",
            documentInputType: "passage",
          },
        },
      },
    }
    ```

    Изменение этих значений влияет на идентичность кэша эмбеддингов при пакетной индексации провайдером; если вышестоящая модель обрабатывает эти метки по-разному, после изменения следует переиндексировать память.

  </Accordion>

    **Аутентификация:** OpenClaw сначала проверяет наличие `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` или `AWS_BEARER_TOKEN_BEDROCK`, а затем переходит к стандартной цепочке поставщиков учётных данных AWS SDK:

    1. Переменные окружения (`AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY`), если также не задана `AWS_PROFILE`

    4. Процесс получения учётных данных (`credential_process` в файле конфигурации AWS)
    5. Учётные данные на основе токена веб-идентификации
    6. Учётные данные из метаданных экземпляра ECS или EC2

    ```

    ```

### Тайм-аут встроенного создания эмбеддингов

<ParamField path="sync.embeddingBatchTimeoutSeconds" type="number">
  Переопределяет тайм-аут для встроенных пакетов создания эмбеддингов при индексировании памяти.

Если значение не задано, используется значение по умолчанию поставщика: 600 секунд для локальных или самостоятельно размещённых поставщиков, таких как `local`, `ollama` и `lmstudio`, и 120 секунд для размещённых поставщиков. Увеличьте это значение, если локальные пакеты создания эмбеддингов с нагрузкой на ЦП работают корректно, но медленно.
</ParamField>

## Поведение индексирования

Все параметры находятся в `memorySearch.sync`, если не указано иное:

| Ключ                           | Тип       | Значение по умолчанию | Описание                                                                         |
| ------------------------------ | --------- | --------------------- | -------------------------------------------------------------------------------- |
| `onSessionStart`               | `boolean` | `true`                | Синхронизировать индекс памяти при запуске сеанса                                 |
| `onSearch`                     | `boolean` | `true`                | Отложенно синхронизировать при поиске после обнаружения изменений содержимого    |
| `watch`                        | `boolean` | `true`                | Отслеживать файлы памяти (chokidar) и планировать переиндексацию при изменениях   |
| `watchDebounceMs`              | `number`  | `1500`                | Окно задержки для объединения частых событий отслеживания файлов                  |
| `intervalMinutes`              | `number`  | `0`                   | Интервал периодической переиндексации в минутах (`0` отключает её)                |
| `sessions.postCompactionForce` | `boolean` | `true`                | Принудительно переиндексировать сеанс после обновлений расшифровки, вызванных Compaction |

<ParamField path="chunking.tokens" type="number">
  Размер фрагмента в токенах, используемый при разделении источников памяти перед созданием эмбеддингов (по умолчанию: 400).
</ParamField>
<ParamField path="chunking.overlap" type="number">
  Перекрытие соседних фрагментов в токенах для сохранения контекста вблизи границ разделения (по умолчанию: 80).
</ParamField>

<Note>
Изменение `chunking.tokens` или `chunking.overlap` изменяет границы фрагментов и делает недействительным идентификатор существующего индекса (см. предупреждение в разделе «Выбор провайдера»).
</Note>

## Настройка гибридного поиска

Все параметры находятся в `memorySearch.query`:

| Ключ         | Тип      | По умолчанию | Описание                                                   |
| ------------ | -------- | ------------ | ---------------------------------------------------------- |
| `maxResults` | `number` | `6`          | Максимальное количество результатов поиска в памяти перед внедрением |
| `minScore`   | `number` | `0.35`       | Минимальная оценка релевантности для включения результата   |

А следующие параметры находятся в `memorySearch.query.hybrid`:

| Ключ                  | Тип       | По умолчанию | Описание                                      |
| --------------------- | --------- | ------------ | --------------------------------------------- |
| `enabled`             | `boolean` | `true`       | Включить гибридный векторный поиск с BM25     |
| `vectorWeight`        | `number`  | `0.7`        | Вес оценок векторного поиска (0–1)            |
| `textWeight`          | `number`  | `0.3`        | Вес оценок BM25 (0–1)                         |
| `candidateMultiplier` | `number`  | `4`          | Множитель размера пула кандидатов             |

<Tabs>
  <Tab title="MMR (разнообразие)">
    | Ключ          | Тип       | По умолчанию | Описание                                         |
    | ------------- | --------- | ------------ | ------------------------------------------------ |
    | `mmr.enabled` | `boolean` | `false`      | Включить повторное ранжирование с помощью MMR    |
    | `mmr.lambda`  | `number`  | `0.7`        | 0 = максимальное разнообразие, 1 = максимальная релевантность |
  </Tab>
  <Tab title="Временное затухание (актуальность)">
    | Ключ                         | Тип       | По умолчанию | Описание                              |
    | ---------------------------- | --------- | ------------ | ------------------------------------- |
    | `temporalDecay.enabled`      | `boolean` | `false`      | Включить повышение оценки за актуальность |
    | `temporalDecay.halfLifeDays` | `number`  | `30`         | Оценка уменьшается вдвое каждые N дней |

    Оценки постоянно актуальных файлов (`MEMORY.md` и файлов без даты в `memory/`) никогда не снижаются.

  </Tab>
</Tabs>

### Полный пример

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        query: {
          maxResults: 6,
          minScore: 0.35,
          hybrid: {
            vectorWeight: 0.7,
            textWeight: 0.3,
            mmr: { enabled: true, lambda: 0.7 },
            temporalDecay: { enabled: true, halfLifeDays: 30 },
          },
        },
      },
    },
  },
}
```

## Дополнительные пути к памяти

| Ключ         | Тип        | Описание                                      |
| ------------ | ---------- | --------------------------------------------- |
| `extraPaths` | `string[]` | Дополнительные каталоги или файлы для индексации |

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        extraPaths: ["../team-docs", "/srv/shared-notes"],
      },
    },
  },
}
```

Пути могут быть абсолютными или относительными к рабочему пространству. Каталоги рекурсивно сканируются на наличие файлов `.md`. Обработка символических ссылок зависит от активного бэкенда: встроенный движок пропускает символические ссылки, а QMD использует поведение базового сканера QMD.

Для поиска по транскриптам других агентов в области конкретного агента используйте `agents.list[].memorySearch.qmd.extraCollections` вместо `memory.qmd.paths`. Эти дополнительные коллекции имеют ту же структуру `{ path, name, pattern? }`, но объединяются отдельно для каждого агента и могут сохранять явно заданные общие имена, если путь указывает за пределы текущего рабочего пространства. Если один и тот же разрешённый путь указан и в `memory.qmd.paths`, и в `memorySearch.qmd.extraCollections`, QMD сохраняет первую запись и пропускает дубликат.

## Кэш эмбеддингов

| Ключ               | Тип       | По умолчанию | Описание                                        |

## Пакетная индексация

| Ключ                          | Тип       | По умолчанию | Описание                              |
| ----------------------------- | --------- | ------------ | ------------------------------------- |
| `remote.nonBatchConcurrency`  | `number`  | `4`          | Параллельные непосредственные эмбеддинги |
| `remote.batch.enabled`        | `boolean` | `false`      | Включить API пакетного создания эмбеддингов |
| `remote.batch.concurrency`    | `number`  | `2`          | Параллельные пакетные задания         |

Доступно для `gemini`, `openai` и `voyage`. Пакетная обработка OpenAI обычно обеспечивает максимальную скорость и минимальную стоимость при больших объёмах повторного заполнения.

## Поиск по памяти сеансов (экспериментальная функция)

Индексируйте расшифровки сеансов и предоставляйте их через `memory_search`:

| Ключ                          | Тип        | По умолчанию | Описание                                      |
| ----------------------------- | ---------- | ------------ | --------------------------------------------- |
| `experimental.sessionMemory`  | `boolean`  | `false`      | Включить индексацию сеансов                   |
| `sources`                     | `string[]` | `["memory"]` | Добавьте `"sessions"`, чтобы включить расшифровки |
| `sync.sessions.deltaBytes`    | `number`   | `100000`     | Порог переиндексации в байтах                 |
| `sync.sessions.deltaMessages` | `number`   | `50`         | Порог переиндексации по количеству сообщений |

<Warning>
Индексация сеансов включается явно и выполняется асинхронно. Результаты могут быть немного устаревшими. Журналы сеансов хранятся на диске, поэтому считайте доступ к файловой системе границей доверия.
</Warning>

Результаты поиска по расшифровкам сеансов также подчиняются параметру
[`tools.sessions.visibility`](/ru/gateway/config-tools#toolssessions). Значение видимости
`tree` по умолчанию предоставляет доступ только к текущему сеансу и созданным им сеансам. Чтобы
из другого сеанса получить данные несвязанного сеанса того же агента, отправленного через
Gateway, например личного сообщения, явно расширьте видимость до `agent` (или до `all`, только
если также требуется доступ между агентами и это разрешено политикой взаимодействия между ними).

В приведённых ниже примерах эти параметры размещены в `agents.defaults`. Можно также
применить эквивалентные параметры `memorySearch` в переопределении для отдельного агента, если
индексировать расшифровки сеансов и выполнять по ним поиск должен только один агент.

Для получения данных из Gateway в личном сообщении в рамках одного агента:

<Tabs>
  <Tab title="Встроенный бэкенд">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
  <Tab title="Бэкенд QMD">
    ```json5
    {
      agents: {
        defaults: {
          memorySearch: {
            experimental: { sessionMemory: true },
            sources: ["memory", "sessions"],
          },
        },
      },
      memory: {
        backend: "qmd",
        qmd: {
          sessions: { enabled: true },
        },
      },
      tools: {
        sessions: { visibility: "agent" },
      },
    }
    ```
  </Tab>
</Tabs>

При использовании QMD параметры `agents.defaults.memorySearch.experimental.sessionMemory` и
`sources: ["sessions"]` сами по себе не экспортируют расшифровки в QMD. Также задайте
`memory.qmd.sessions.enabled: true`.

  ## Конфигурация бэкенда QMD

  Для включения задайте `memory.backend = "qmd"`. Все настройки QMD находятся в разделе `memory.qmd`:

  | Ключ                     | Тип       | По умолчанию | Описание                                                                                                  |
  | ------------------------ | --------- | ------------ | --------------------------------------------------------------------------------------------------------- |
  | `command`                | `string`  | `qmd`        | Путь к исполняемому файлу QMD; укажите абсолютный путь, если служебная переменная `PATH` отличается от используемой в вашей оболочке |
  | `searchMode`             | `string`  | `search`     | Команда поиска: `search`, `vsearch`, `query`                                                              |
  | `rerank`                 | `boolean` | --           | Задайте `false` при `searchMode: "query"` и QMD 2.1+, чтобы пропустить повторное ранжирование QMD           |
  | `includeDefaultMemory`   | `boolean` | `true`       | Автоматически индексировать `MEMORY.md` и `memory/**/*.md`                                                |
  | `paths[]`                | `array`   | --           | Дополнительные пути: `{ name, path, pattern? }`                                                           |
  | `sessions.enabled`       | `boolean` | `false`      | Экспортировать расшифровки сеансов в QMD                                                                  |
  | `sessions.retentionDays` | `number`  | --           | Срок хранения расшифровок                                                                                 |
  | `sessions.exportDir`     | `string`  | --           | Каталог экспорта                                                                                           |

  Режим `searchMode: "search"` использует только лексический поиск/BM25. В этом режиме OpenClaw не выполняет проверки готовности семантических векторов или обслуживание эмбеддингов QMD, в том числе при выполнении `memory status --deep`; режимам `vsearch` и `query` по-прежнему требуются готовность векторов QMD и эмбеддинги.

  Параметр `rerank: false` влияет только на режим QMD `query` и требует QMD 2.1 или новее. В режиме прямого CLI OpenClaw передаёт `--no-rerank`; в режиме MCP через mcporter он передаёт `rerank: false` унифицированному инструменту запросов QMD. Не задавайте этот параметр, чтобы использовать стандартное поведение QMD для повторного ранжирования запросов.

  OpenClaw предпочитает актуальные форматы коллекций и запросов MCP в QMD, но сохраняет работоспособность старых выпусков QMD, при необходимости пробуя совместимые флаги шаблонов коллекций и прежние имена инструментов MCP. Когда QMD заявляет поддержку нескольких фильтров коллекций, коллекции из одного источника обрабатываются одним процессом QMD; старые сборки QMD продолжают использовать совместимый путь с отдельной обработкой каждой коллекции. Один источник означает, что коллекции долговременной памяти — стандартные файлы памяти и пользовательские пути — группируются вместе, а коллекции расшифровок сеансов остаются отдельной группой, чтобы диверсификация источников по-прежнему использовала оба типа входных данных.

  <Note>
  Переопределения моделей QMD задаются на стороне QMD, а не в конфигурации OpenClaw. Если необходимо глобально переопределить модели QMD, задайте переменные окружения, например `QMD_EMBED_MODEL`, `QMD_RERANK_MODEL` и `QMD_GENERATE_MODEL`, в среде выполнения Gateway.
  </Note>

  ### Интеграция с mcporter

  Все параметры находятся в разделе `memory.qmd.mcporter`. Поисковые запросы QMD направляются через долгоживущий демон MCP `mcporter` вместо запуска `qmd` для каждого запроса, что снижает накладные расходы холодного запуска для крупных моделей.

  | Ключ          | Тип       | По умолчанию | Описание                                                                       |
  | ------------- | --------- | ------------ | ------------------------------------------------------------------------------ |
  | `enabled`     | `boolean` | `false`      | Направлять вызовы QMD через mcporter вместо запуска `qmd` для каждого запроса  |
  | `serverName`  | `string`  | `qmd`        | Имя сервера mcporter, запускающего `qmd mcp` с `lifecycle: keep-alive`          |
  | `startDaemon` | `boolean` | `true`       | Автоматически запускать демон mcporter, когда `enabled` имеет значение `true`  |

  Требуется установить `mcporter`, обеспечить его доступность через PATH и настроить сервер mcporter, запускающий `qmd mcp`. Оставьте эту функцию отключённой для более простых локальных конфигураций, где допустимы затраты на запуск отдельного процесса для каждого запроса.

  <AccordionGroup>
  <Accordion title="Расписание обновлений">
    | Ключ                      | Тип       | По умолчанию | Описание                                                                                             |
    | ------------------------- | --------- | ------------ | ---------------------------------------------------------------------------------------------------- |
    | `update.interval`         | `string`  | `5m`         | Интервал обновления                                                                                  |
    | `update.debounceMs`       | `number`  | `15000`      | Задержка обработки изменений файлов                                                                 |
    | `update.onBoot`           | `boolean` | `true`       | Обновлять при открытии долгоживущего менеджера QMD; задайте `false`, чтобы пропустить немедленное обновление при запуске |
    | `update.startup`          | `string`  | `off`        | Необязательная инициализация QMD при запуске Gateway: `off`, `idle` или `immediate`                  |
    | `update.startupDelayMs`   | `number`  | `120000`     | Задержка перед обновлением с `startup: "idle"`                                                       |
    | `update.waitForBootSync`  | `boolean` | `false`      | Блокировать открытие менеджера до завершения первоначального обновления                              |
    | `update.embedInterval`    | `string`  | `60m`        | Отдельный интервал создания эмбеддингов                                                              |
    | `update.commandTimeoutMs` | `number`  | `30000`      | Тайм-аут команд обслуживания QMD (просмотр/добавление коллекций)                                     |
    | `update.updateTimeoutMs`  | `number`  | `120000`     | Тайм-аут каждого цикла `qmd update`                                                                  |
    | `update.embedTimeoutMs`   | `number`  | `120000`     | Тайм-аут каждого цикла `qmd embed`                                                                   |
  </Accordion>
  <Accordion title="Ограничения">
    | Ключ                      | Тип      | По умолчанию | Описание                                      |
    | ------------------------- | -------- | ------------ | --------------------------------------------- |
    | `limits.maxResults`       | `number` | `4`          | Максимальное количество результатов поиска   |
    | `limits.maxSnippetChars`  | `number` | `450`        | Ограничение длины фрагмента                   |
    | `limits.maxInjectedChars` | `number` | `2200`       | Ограничение общего числа внедряемых символов  |
    | `limits.timeoutMs`        | `number` | `4000`       | Тайм-аут поиска                               |
  </Accordion>
  <Accordion title="Область действия">
    Определяет, какие сеансы могут получать результаты поиска QMD. Схема совпадает с [`session.sendPolicy`](/ru/gateway/config-agents#session):

    ```json5
    {
      memory: {
        qmd: {
          scope: {
            default: "deny",
            rules: [{ action: "allow", match: { chatType: "direct" } }],
          },
        },
      },
    }
    ```

    Поставляемая конфигурация по умолчанию разрешает только личные/прямые сообщения, запрещая группы и другие типы каналов. `match.keyPrefix` сопоставляется с нормализованным ключом сеанса; `match.rawKeyPrefix` — с исходным ключом, включая `agent:<id>:`.

  </Accordion>
  <Accordion title="Цитаты">
    `memory.citations` применяется ко всем бэкендам:

    | Значение         | Поведение                                                        |
    | ------------------ | ------------------------------------------------------ |
    | `auto` (по умолчанию) | Добавлять нижний колонтитул `Source: <path#line>` во фрагменты |
    | `on`             | Всегда добавлять нижний колонтитул                               |
    | `off`            | Не добавлять нижний колонтитул (путь по-прежнему передаётся агенту внутренним образом) |

  </Accordion>
</AccordionGroup>

Когда инициализация QMD при запуске Gateway включена, OpenClaw запускает QMD только для подходящих агентов. Если `update.onBoot` имеет значение true и интервальное обслуживание обновления или встраивания не настроено, при запуске используется одноразовый менеджер для начального обновления, после чего он закрывается. Если настроен интервал обновления или встраивания, при запуске открывается долгоживущий менеджер QMD, который управляет наблюдателем и таймерами интервалов; `update.onBoot: false` пропускает только немедленное начальное обновление.

```json5
{
  memory: {
    backend: "qmd",
    citations: "auto",

      limits: { maxResults: 4, timeoutMs: 4000 },
      scope: {
        default: "deny",
        rules: [{ action: "allow", match: { chatType: "direct" } }],
      },
      paths: [{ name: "docs", path: "~/notes", pattern: "**/*.md" }],
    },
  },
}
```

## Dreaming

Dreaming настраивается в `plugins.entries.memory-core.config.dreaming`, а не в `agents.defaults.memorySearch`.

Dreaming выполняется как один запланированный проход, а внутренние фазы light/deep/REM являются деталью реализации.

Описание концептуального поведения и команд с косой чертой см. в разделе [Dreaming](/ru/concepts/dreaming).

### Пользовательские настройки

| Ключ                                   | Тип       | Значение по умолчанию | Описание                                                                                                                      |
| -------------------------------------- | --------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `enabled`                              | `boolean` | `false`       | Полностью включить или отключить Dreaming                                                                                              |
| `frequency`                            | `string`  | `0 3 * * *`   | Необязательная периодичность Cron для полного прохода Dreaming                                                                                |
| `model`                                | `string`  | модель по умолчанию | Необязательное переопределение модели субагента Dream Diary                                                                                     |
| `phases.deep.maxPromotedSnippetTokens` | `number`  | `160`         | Максимальное расчётное количество токенов, сохраняемых из каждого фрагмента кратковременного воспоминания, перенесённого в `MEMORY.md`; метаданные происхождения остаются видимыми |

### Пример

```json5
{
  plugins: {
    entries: {
      "memory-core": {
        subagent: {
          allowModelOverride: true,
          allowedModels: ["anthropic/claude-sonnet-4-6"],
        },
        config: {
          dreaming: {
            enabled: true,
            frequency: "0 3 * * *",
            model: "anthropic/claude-sonnet-4-6",
          },
        },
      },
    },
  },
}
```

<Note>
- Dreaming записывает машинное состояние в `memory/.dreams/`.
- Dreaming записывает удобный для чтения повествовательный результат в `DREAMS.md` (или существующий `dreams.md`).
- `dreaming.model` использует существующий шлюз доверия субагентов Plugin; перед включением задайте `plugins.entries.memory-core.subagent.allowModelOverride: true`.
- Dream Diary повторяет попытку один раз с моделью сеанса по умолчанию, если настроенная модель недоступна. Ошибки доверия или списка разрешённых моделей регистрируются в журнале и не приводят к скрытой повторной попытке.
- Политика и пороговые значения фаз light/deep/REM являются внутренним поведением, а не пользовательской конфигурацией.

</Note>
