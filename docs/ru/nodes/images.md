Канал WhatsApp работает на базе Baileys Web. На этой странице описаны правила обработки медиафайлов при отправке, работе Gateway и ответах агента.

## Цели

- Отправлять медиафайлы с необязательной подписью через `openclaw message send --media`.
- Разрешить автоматическим ответам из веб-папки «Входящие» включать медиафайлы вместе с текстом.
- Обеспечить разумные и предсказуемые ограничения для каждого типа.

## Интерфейс CLI

`openclaw message send --target <dest> --media <path-or-url> [--message <caption>]`

- `--media <path-or-url>` — прикрепляет медиафайл (изображение, аудио, видео или документ); принимает локальные пути или URL. Необязательный параметр; для отправки только медиафайла подпись может быть пустой.
- `--gif-playback` — воспроизводит видео как GIF-анимацию (только WhatsApp).
- `--force-document` — отправляет медиафайл как документ, чтобы избежать сжатия каналом (Telegram, WhatsApp); применяется к изображениям, GIF-анимациям и видео.
- `--reply-to <id>`, `--thread-id <id>`, `--pin`, `--silent` — параметры доставки и веток, общие с отправкой только текста.
- `--dry-run` — выводит сформированную полезную нагрузку без отправки.
- `--json` — выводит результат в формате JSON: `{ action, channel, dryRun, handledBy, messageId?, payload }` (`payload` содержит специфичный для канала результат отправки, включая ссылку на медиафайл, если она есть).

## Поведение веб-канала WhatsApp

- Входные данные: путь к локальному файлу **или** URL HTTP(S).
- Процесс: загрузка в буфер, определение типа медиафайла и построение исходящей полезной нагрузки в зависимости от типа:
  - **Изображения:** оптимизируются так, чтобы размер не превышал `channels.whatsapp.mediaMaxMb` (по умолчанию 50 МБ). Непрозрачные изображения повторно сжимаются в JPEG (стандартная последовательность размеров сторон начинается с 2048 пикселей и уменьшается при повторных превышениях размера); изображения с прозрачностью сохраняются в формате PNG. Если исходный файл уже является допустимым JPEG, PNG или WebP и укладывается в ограничения по размеру файла и длине стороны, исходные байты сохраняются без изменений вместо повторного сжатия. Анимированные GIF-файлы никогда не перекодируются, а только проверяются на соответствие ограничению размера.
  - **Аудио и голосовые сообщения:** если исходный файл ещё не является аудио в нативном формате голосовых сообщений (`.ogg`/`.opus` или `audio/ogg`/`audio/opus`), перед отправкой в качестве голосового сообщения (`ptt: true`) исходящее аудио перекодируется через `ffmpeg` в Opus/OGG (48 кГц, моно, 64 кбит/с, продолжительность не более 20 минут).
  - **Видео:** передаётся без изменений, если размер не превышает 16 МБ.
  - **Документы:** все остальные файлы размером до 100 МБ; имя файла сохраняется, если оно доступно.
- Воспроизведение в стиле GIF в WhatsApp: отправьте MP4 с `gifPlayback: true` (CLI: `--gif-playback`), чтобы мобильные клиенты циклически воспроизводили его непосредственно в сообщении.
- При определении MIME-типа сначала проверяются сигнатурные байты, затем расширение файла и после этого заголовки ответа; общий тип контейнера, определённый по сигнатуре (`application/octet-stream`, `zip`), никогда не заменяет более точное сопоставление по расширению (например, XLSX на ZIP).
- Подпись берётся из `--message` или `reply.text`; допускается пустая подпись.
- Журналирование: в обычном режиме отображаются `↩️`/`✅`; в подробном режиме также указываются размер и исходный путь или URL.

<Note>
Указанные выше значения 16 МБ для аудио и видео и 100 МБ для документов — это общие стандартные ограничения по типам медиафайлов, применяемые, когда явное ограничение в байтах не задано. При отправке через WhatsApp явно задаётся ограничение из `channels.whatsapp.mediaMaxMb` (по умолчанию 50 МБ), которое одинаково применяется ко всем типам для этой учётной записи.
</Note>

## Конвейер автоматических ответов

- `getReplyFromConfig` возвращает полезную нагрузку ответа (или массив полезных нагрузок), которая среди прочих полей содержит `text?`, `mediaUrl?` и `mediaUrls?`.
- При наличии медиафайлов веб-отправитель обрабатывает локальные пути или URL с помощью того же конвейера, что и `openclaw message send`.
- Если указано несколько медиафайлов, они отправляются последовательно.

## Передача входящих медиафайлов командам

- Когда входящие веб-сообщения содержат медиафайлы, OpenClaw загружает их во временные файлы и предоставляет переменные шаблонизации:
  - `{{MediaUrl}}` — псевдо-URL входящего медиафайла.
  - `{{MediaPath}}` — локальный временный путь, записанный перед выполнением команды.
- Если включена отдельная для каждого сеанса песочница Docker, входящий медиафайл копируется в рабочую область песочницы, а `MediaPath`/`MediaUrl` заменяются относительным путём внутри песочницы, например `media/inbound/<filename>`.
- Анализ медиафайлов (настраиваемый через `tools.media.*` или общие `tools.media.models`) выполняется до шаблонизации и может вставлять в `Body` блоки `[Image]`, `[Audio]` и `[Video]`.
  - Для аудио задаётся `{{Transcript}}`, а расшифровка используется при разборе команд, чтобы команды с косой чертой продолжали работать.
  - Описания видео и изображений сохраняют текст подписи для разбора команд.
  - Если активная основная модель уже имеет встроенную поддержку изображений, OpenClaw пропускает блок сводки `[Image]` и вместо него передаёт модели исходное изображение.
- По умолчанию обрабатывается только первое подходящее вложение с изображением, аудио или видео; чтобы обрабатывать несколько вложений, задайте `tools.media.<capability>.attachments`.

## Ограничения и ошибки

**Ограничения исходящей отправки (веб-отправка WhatsApp)**

- Изображения: после оптимизации — до `channels.whatsapp.mediaMaxMb` (по умолчанию 50 МБ).
- Аудио и видео: ограничение 16 МБ (общее значение по умолчанию; при отправке через WhatsApp заменяется значением `mediaMaxMb`).
- Документы: ограничение 100 МБ (общее значение по умолчанию; при отправке через WhatsApp заменяется значением `mediaMaxMb`).
- Слишком большой или нечитаемый медиафайл приводит к появлению понятного сообщения об ошибке в журналах, а ответ пропускается.

**Ограничения анализа медиафайлов (расшифровка и описание)**

- Изображение: по умолчанию 10 МБ (`tools.media.image.maxBytes`).
- Аудио: по умолчанию 20 МБ (`tools.media.audio.maxBytes`).
- Видео: по умолчанию 50 МБ (`tools.media.video.maxBytes`).
- Для слишком больших медиафайлов анализ пропускается, но ответ всё равно отправляется с исходным содержимым.

## Примечания по тестированию

- Охватите процессы отправки и ответа для изображений, аудио и документов.
- Проверьте ограничения размера после оптимизации изображений и флаг голосового сообщения для аудио.
- Убедитесь, что ответы с несколькими медиафайлами разворачиваются в последовательные отправки.
