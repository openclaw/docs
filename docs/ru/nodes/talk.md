Режим разговора поддерживает пять вариантов среды выполнения:

- **Нативный режим разговора в macOS/iOS/Android**: локальное распознавание речи, чат через Gateway и синтез речи `talk.speak`. Узлы объявляют возможность `talk` и указывают, какие команды `talk.*` они поддерживают.
- **Режим разговора в iOS (реального времени)**: управляемый клиентом WebRTC для конфигураций OpenAI реального времени, в которых выбран транспорт `webrtc` или транспорт не указан. Явно заданные конфигурации реального времени `gateway-relay`, `provider-websocket` и конфигурации не от OpenAI продолжают использовать ретранслятор под управлением Gateway; конфигурации не в реальном времени используют нативный речевой цикл.
- **Режим разговора в браузере**: `talk.client.create` для управляемых клиентом сеансов `webrtc`/`provider-websocket` или `talk.session.create` для управляемых Gateway сеансов `gateway-relay`. `managed-room` зарезервирован для передачи управления Gateway и комнат в режиме рации.
- **Режим разговора в Android (реального времени)**: включается с помощью `talk.realtime.mode: "realtime"` и `talk.realtime.transport: "gateway-relay"`. В противном случае Android продолжает использовать нативное распознавание речи, чат через Gateway и `talk.speak`.
- **Клиенты только для транскрибирования**: `talk.session.create({ mode: "transcription", transport: "gateway-relay", brain: "none" })`, затем `talk.session.appendAudio`, `talk.session.cancelTurn` и `talk.session.close` для субтитров и диктовки без голосового ответа ассистента. Однократно загружаемые голосовые заметки по-прежнему обрабатываются через аудиоканал [анализа мультимедиа](/ru/nodes/media-understanding).

Нативный режим разговора представляет собой непрерывный цикл: прослушать речь, отправить транскрипцию модели через активный сеанс, дождаться ответа, а затем озвучить его через настроенного поставщика режима разговора (`talk.speak`).

Управляемый клиентом режим разговора реального времени перенаправляет вызовы инструментов поставщика через `talk.client.toolCall`, а не вызывает `chat.send` напрямую. Пока активна консультация в реальном времени, клиенты могут вызывать `talk.client.steer` или `talk.session.steer`, чтобы классифицировать голосовой ввод как `status`, `steer`, `cancel` или `followup`. Принятое управляющее сообщение ставится в очередь активного встроенного запуска; при отклонении возвращается причина, например `no_active_run`, `not_streaming` или `compacting`.

Режим разговора только для транскрибирования отправляет ту же оболочку событий режима разговора, что и сеансы реального времени и STT/TTS, но использует `mode: "transcription"` и `brain: "none"`. Все сеансы режима разговора транслируют события в канале `talk.event`; клиенты подписываются на него для получения промежуточных и окончательных обновлений транскрипции (`transcript.delta`/`transcript.done`), а также других телеметрических данных сеанса.

## Поведение (macOS)

- Постоянно отображаемая панель, пока включён режим разговора.
- Переходы между фазами **Прослушивание &rarr; Обработка &rarr; Озвучивание**.
- После короткой паузы (интервала тишины) отправляется текущая транскрипция.
- Ответы записываются в WebChat (как при вводе с клавиатуры).
- **Прерывание при обнаружении речи** (включено по умолчанию): если пользователь говорит во время озвучивания ответа ассистентом, воспроизведение останавливается, а временная метка прерывания сохраняется для следующего запроса.

## Голосовые директивы в ответах

Ассистент может добавить в начало ответа одну строку JSON для управления голосом:

```json
{ "voice": "<voice-id>", "once": true }
```

Правила:

- Учитывается только первая непустая строка; строка JSON удаляется перед воспроизведением TTS.
- Неизвестные ключи игнорируются.
- `once: true` применяется только к текущему ответу; без него голос становится новым голосом по умолчанию для режима разговора.

Поддерживаемые ключи: `voice` / `voice_id` / `voiceId`, `model` / `model_id` / `modelId`, `speed`, `rate` (слов в минуту), `stability`, `similarity`, `style`, `speakerBoost`, `seed`, `normalize`, `lang`, `output_format`, `latency_tier`, `once`.

## Конфигурация (`~/.openclaw/openclaw.json`)

```json5
{
  talk: {
    provider: "elevenlabs",
    providers: {
      elevenlabs: {
        voiceId: "elevenlabs_voice_id",
        modelId: "eleven_v3",
        outputFormat: "mp3_44100_128",
        apiKey: "elevenlabs_api_key",
      },
      mlx: {
        modelId: "mlx-community/Soprano-80M-bf16",
      },
      system: {},
    },
    speechLocale: "ru-RU",
    silenceTimeoutMs: 1500,
    interruptOnSpeech: true,
    realtime: {
      provider: "openai",
      providers: {
        openai: {
          apiKey: "openai_api_key",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
        },
      },
      instructions: "Говори доброжелательно и отвечай кратко.",
      mode: "realtime",
      transport: "webrtc",
      brain: "agent-consult",
    },
  },
}
```

| `provider`                               | -                                          | Активный поставщик TTS для режима разговора. Используйте `elevenlabs`, `mlx` или `system` для локальных путей воспроизведения в macOS.                                                                                                                                                                                                        |

`talk.catalog` предоставляет канонические идентификаторы провайдеров и псевдонимы реестра, допустимые для каждого провайдера режимы, транспорты, стратегии обработки, форматы аудио в реальном времени, флаги возможностей, а также выбранный средой выполнения результат готовности. Официальные клиенты Talk должны получать эти данные из каталога, а не поддерживать псевдонимы провайдеров локально; если более старая версия Gateway не возвращает готовность группы, считайте её непроверенной, а не однозначно ненастроенной. Провайдеры потоковой транскрипции обнаруживаются через `talk.catalog.transcription`; текущий ретранслятор Gateway использует конфигурацию провайдера потоковой передачи Voice Call до появления отдельной конфигурации транскрипции Talk.

## Интерфейс macOS

- Переключатель в строке меню: **Talk**
- Вкладка конфигурации: группа **Talk Mode** (идентификатор голоса и переключатель прерывания)
- Наложение: сфера отображает универсальную звуковую волну Talk (общую для iOS, watchOS и Android). В режиме прослушивания она следует за текущим уровнем микрофона, в режиме речи — за фактической огибающей воспроизведения TTS, а в режиме обработки мягко пульсирует. Нажмите сферу, чтобы приостановить или возобновить работу, дважды нажмите, чтобы остановить речь, либо нажмите X, чтобы выйти из режима Talk.

## Интерфейс Android

- Переключатель на вкладке голоса: **Talk**
- Ручные режимы **Mic** и **Talk** являются взаимоисключающими режимами захвата.
- Ручной режим Mic и Talk в реальном времени предпочитают микрофон подключённой гарнитуры Bluetooth Classic или BLE; при её отключении приложение запрашивает другой вход гарнитуры или переключается на микрофон по умолчанию, восстанавливая выбор по умолчанию после прекращения захвата.
- Ручной режим Mic прекращает работу, когда приложение уходит из активного режима или пользователь покидает вкладку голоса.
- Режим Talk продолжает работать, пока его не отключат переключателем или пока Node не отсоединится; во время работы используется тип фоновой службы Android для микрофона.
- Android поддерживает выходные форматы `pcm_16000`, `pcm_22050`, `pcm_24000` и `pcm_44100` для потоковой передачи через `AudioTrack` с низкой задержкой.

## Примечания

- Требуются разрешения на распознавание речи и доступ к микрофону.
- Нативный режим Talk использует активный сеанс Gateway и переходит к опросу истории только при недоступности событий ответа.
- Gateway обрабатывает воспроизведение Talk через `talk.speak`, используя активного провайдера Talk. Android переходит на локальный системный TTS, только если этот RPC недоступен.
- Локальное воспроизведение MLX в macOS использует встроенный вспомогательный компонент `openclaw-mlx-tts`, если он доступен, либо исполняемый файл из `PATH`. Во время разработки задайте `OPENCLAW_MLX_TTS_BIN`, указав пользовательский исполняемый файл вспомогательного компонента.
- Диапазоны значений голосовых директив (ElevenLabs): `stability`, `similarity` и `style` принимают значения `0..1`; `speed` — `0.5..2`; `latency_tier` — `0..4`.
