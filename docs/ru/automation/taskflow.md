Поток задач — это слой оркестрации потоков, расположенный над [фоновыми задачами](/ru/automation/tasks). Он управляет устойчивыми многошаговыми потоками с собственным состоянием, отслеживанием ревизий и семантикой синхронизации, при этом отдельные задачи остаются единицей отделенной работы.

## Когда использовать поток задач

Используйте поток задач, когда работа охватывает несколько последовательных или ветвящихся шагов и вам нужно устойчивое отслеживание прогресса при перезапусках gateway. Для одиночных фоновых операций достаточно обычной [задачи](/ru/automation/tasks).

| Сценарий                              | Использование                  |
| ------------------------------------- | ------------------------------ |
| Одиночное фоновое задание             | Обычная задача                 |
| Многошаговый конвейер (A, затем B, затем C) | Поток задач (управляемый) |
| Наблюдение за внешне созданными задачами | Поток задач (зеркальный) |
| Одноразовое напоминание               | Задание Cron                   |

## Надежный шаблон запланированного рабочего процесса

Для повторяющихся рабочих процессов, таких как сводки рыночной аналитики, рассматривайте расписание, оркестрацию и проверки надежности как отдельные слои:

1. Используйте [запланированные задачи](/ru/automation/cron-jobs) для выбора времени.
2. Храните предыдущий контекст в собственных файлах, базе данных или состоянии инструментов рабочего процесса.
3. Используйте [Lobster](/ru/tools/lobster) для детерминированных шагов, шлюзов утверждения и токенов возобновления.
4. Используйте поток задач, чтобы отслеживать многошаговый запуск между дочерними задачами, ожиданиями, повторными попытками и перезапусками gateway.

Пример формы cron:

```bash
openclaw cron add \
  --name "Market intelligence brief" \
  --cron "0 7 * * 1-5" \
  --tz "America/New_York" \
  --session session:market-intel \
  --message "Run the market-intel Lobster workflow. Verify source freshness before summarizing." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"
```

Используйте `session:<id>`, когда задание должно быть направлено в известный чат или сессию для контекста доставки либо безопасного начального заполнения предпочтений. Cron все равно выполняет каждый запуск в отделенной сессии, поэтому помещайте сводки предыдущих запусков и постоянное состояние рабочего процесса в явное хранилище, которое задание может читать.

Внутри рабочего процесса размещайте проверки надежности перед шагом сводки LLM:

```yaml
name: market-intel-brief
steps:
  - id: preflight
    command: market-intel check --json
  - id: collect
    command: market-intel collect --json
    stdin: $preflight.json
  - id: summarize
    command: market-intel summarize --json
    stdin: $collect.json
  - id: approve
    command: market-intel deliver --preview
    stdin: $summarize.json
    approval: required
  - id: deliver
    command: market-intel deliver --execute
    stdin: $summarize.json
    condition: $approve.approved
```

Рекомендуемые предварительные проверки:

- Доступность браузера и выбор профиля, например `openclaw` для управляемого состояния или `user`, когда требуется авторизованная сессия Chrome. См. [Браузер](/ru/tools/browser).
- Учетные данные API и квота для каждого источника.
- Сетевая доступность требуемых конечных точек.
- Требуемые инструменты включены для агента, например `lobster`, `browser` и `llm-task`.
- Назначение для сбоев настроено для cron, чтобы ошибки предварительной проверки были видимы. См. [запланированные задачи](/ru/automation/cron-jobs#delivery-and-output).

Рекомендуемые поля происхождения данных для каждого собранного элемента:

```json
{
  "sourceUrl": "https://example.com/report",
  "retrievedAt": "2026-04-24T12:00:00Z",
  "asOf": "2026-04-24",
  "title": "Example report",
  "content": "..."
}
```

Рабочий процесс должен отклонять или помечать устаревшие элементы перед созданием сводки. Шаг LLM должен получать только структурированный JSON, и его следует попросить сохранять `sourceUrl`, `retrievedAt` и `asOf` в выводе. Используйте [задачу LLM](/ru/tools/llm-task), когда внутри рабочего процесса нужен шаг модели с проверкой по схеме.

Для повторно используемых командных или общественных рабочих процессов упакуйте CLI, файлы `.lobster` и любые заметки по настройке как навык или plugin и опубликуйте через [ClawHub](/clawhub). Держите специфичные для рабочего процесса защитные ограничения в этом пакете, если только API plugin не лишен нужной универсальной возможности.

## Режимы синхронизации

### Управляемый режим

Поток задач владеет жизненным циклом от начала до конца. Он создает задачи как шаги потока, доводит их до завершения и автоматически продвигает состояние потока.

Пример: поток еженедельного отчета, который (1) собирает данные, (2) создает отчет и (3) доставляет его. Поток задач создает каждый шаг как фоновую задачу, ожидает завершения, затем переходит к следующему шагу.

```
Flow: weekly-report
  Step 1: gather-data     → task created → succeeded
  Step 2: generate-report → task created → succeeded
  Step 3: deliver         → task created → running
```

### Зеркальный режим

Поток задач наблюдает за внешне созданными задачами и поддерживает состояние потока в синхронизации, не беря на себя владение созданием задач. Это полезно, когда задачи поступают из заданий cron, команд CLI или других источников, а вам нужен единый вид их прогресса как потока.

Пример: три независимых задания cron, которые вместе образуют утреннюю операционную процедуру. Зеркальный поток отслеживает их общий прогресс, не управляя тем, когда и как они выполняются.

## Устойчивое состояние и отслеживание ревизий

Каждый поток сохраняет собственное состояние и отслеживает ревизии, чтобы прогресс переживал перезапуски gateway. Отслеживание ревизий позволяет обнаруживать конфликты, когда несколько источников пытаются одновременно продвинуть один и тот же поток.
Реестр потоков использует SQLite с ограниченным обслуживанием журнала предварительной записи, включая
периодические контрольные точки и контрольные точки при завершении, поэтому долго работающие gateway не сохраняют
неограниченные побочные файлы `registry.sqlite-wal`.

## Поведение отмены

`openclaw tasks flow cancel` задает для потока закрепленное намерение отмены. Активные задачи внутри потока отменяются, и новые шаги не запускаются. Намерение отмены сохраняется между перезапусками, поэтому отмененный поток остается отмененным, даже если gateway перезапускается до завершения всех дочерних задач.

## Команды CLI

```bash
# List active and recent flows
openclaw tasks flow list

# Show details for a specific flow
openclaw tasks flow show <lookup>

# Cancel a running flow and its active tasks
openclaw tasks flow cancel <lookup>
```

| Команда                           | Описание                                   |
| --------------------------------- | ------------------------------------------ |
| `openclaw tasks flow list`        | Показывает отслеживаемые потоки со статусом и режимом синхронизации |
| `openclaw tasks flow show <id>`   | Просмотреть один поток по идентификатору потока или ключу поиска |
| `openclaw tasks flow cancel <id>` | Отменить выполняющийся поток и его активные задачи |

## Как потоки связаны с задачами

Потоки координируют задачи, а не заменяют их. Один поток может управлять несколькими фоновыми задачами за время своего существования. Используйте `openclaw tasks`, чтобы просматривать записи отдельных задач, и `openclaw tasks flow`, чтобы просматривать оркестрирующий поток.
