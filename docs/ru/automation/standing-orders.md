Постоянные распоряжения предоставляют вашему агенту **постоянные полномочия на выполнение** определённых программ. Вместо того чтобы давать агенту указание для каждой задачи, вы определяете программы с чёткой областью действия, триггерами и правилами эскалации, а агент автономно выполняет их в установленных границах: «Ты отвечаешь за еженедельный отчёт. Составляй и отправляй его каждую пятницу, а эскалируй только в том случае, если что-то выглядит не так».

## Зачем нужны постоянные распоряжения

**Без постоянных распоряжений:** вы даёте агенту указание для каждой задачи, рутинная работа забывается или откладывается, а вы становитесь узким местом.

**С постоянными распоряжениями:** агент автономно действует в установленных границах, рутинная работа выполняется по расписанию, а вы подключаетесь только при исключениях и необходимости согласования.

## Как они работают

Постоянные распоряжения определяются в файлах [рабочего пространства агента](/ru/concepts/agent-workspace). Рекомендуется включать их непосредственно в `AGENTS.md` (который автоматически добавляется в каждый сеанс), чтобы они всегда находились в контексте агента. Для более крупных конфигураций их также можно поместить в отдельный файл, например `standing-orders.md`, и сослаться на него из `AGENTS.md`.

Каждая программа определяет:

1. **Область полномочий** — что агенту разрешено делать
2. **Триггеры** — когда выполнять программу (по расписанию, событию или условию)
3. **Точки согласования** — какие действия требуют предварительного одобрения человеком
4. **Правила эскалации** — когда следует остановиться и обратиться за помощью

Агент загружает эти инструкции в каждом сеансе через файлы начальной загрузки рабочего пространства (полный список автоматически добавляемых файлов см. в разделе [Рабочее пространство агента](/ru/concepts/agent-workspace)) и выполняет их совместно с [заданиями Cron](/ru/automation/cron-jobs), обеспечивающими запуск по времени.

<Tip>
Поместите постоянные распоряжения в `AGENTS.md`, чтобы гарантировать их загрузку в каждом сеансе. При начальной загрузке рабочего пространства автоматически добавляются `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md` и `MEMORY.md`, но не произвольные файлы из подкаталогов.
</Tip>

## Структура постоянного распоряжения

```markdown
## Программа: Еженедельный отчёт о состоянии

**Полномочия:** Собирать данные, формировать отчёт, доставлять его заинтересованным сторонам
**Триггер:** Каждую пятницу в 16:00 (обеспечивается заданием Cron)
**Точка согласования:** Для стандартных отчётов отсутствует. Аномалии следует отмечать для проверки человеком.
**Эскалация:** Если источник данных недоступен или показатели выглядят необычно (>2σ от нормы)

### Этапы выполнения

1. Получить показатели из настроенных источников
2. Сравнить их с предыдущей неделей и целевыми значениями
3. Сформировать отчёт в Reports/weekly/YYYY-MM-DD.md
4. Отправить сводку через настроенный канал
5. Записать сведения о завершении в Agent/Logs/

### Что НЕ следует делать

- Не отправлять отчёты внешним сторонам
- Не изменять исходные данные
- Не пропускать отправку, если показатели выглядят плохо, — отражать их точно
```

## Постоянные распоряжения и задания Cron

Постоянные распоряжения определяют, **что** агенту разрешено делать. [Задания Cron](/ru/automation/cron-jobs) определяют, **когда** это происходит. Они работают совместно:

```text
Постоянное распоряжение: «Ты отвечаешь за ежедневный разбор входящих сообщений»
    ↓
Задание Cron (ежедневно в 8:00): «Выполни разбор входящих сообщений согласно постоянным распоряжениям»
    ↓
Агент: Читает постоянные распоряжения → выполняет этапы → сообщает результаты
```

В запросе задания Cron следует ссылаться на постоянное распоряжение, а не дублировать его:

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel imessage \
  --to "+1XXXXXXXXXX" \
  --message "Выполни ежедневный разбор входящих сообщений согласно постоянным распоряжениям. Проверь почту на наличие новых оповещений. Разбери, классифицируй и сохрани каждый элемент. Отправь владельцу сводку. Эскалируй неизвестные случаи."
```

## Примеры

### Пример 1: контент и социальные сети (еженедельный цикл)

```markdown
## Программа: Контент и социальные сети

**Полномочия:** Готовить черновики контента, планировать публикации, составлять отчёты о вовлечённости
**Точка согласования:** В течение первых 30 дней все публикации требуют проверки владельцем, после чего действует постоянное разрешение
**Триггер:** Еженедельный цикл (проверка в понедельник → черновики в середине недели → краткий отчёт в пятницу)

### Еженедельный цикл

- **Понедельник:** Проверить показатели платформ и вовлечённость аудитории
- **Вторник–четверг:** Подготовить черновики публикаций для социальных сетей и материалы для блога
- **Пятница:** Составить еженедельный маркетинговый отчёт → передать владельцу

### Правила работы с контентом

- Стиль должен соответствовать бренду (см. SOUL.md или руководство по стилю бренда)
- Никогда не представляться ИИ в публичном контенте
- Включать показатели, когда они доступны
- Сосредоточиться на пользе для аудитории, а не на саморекламе
```

### Пример 2: финансовые операции (по событию)

```markdown
## Программа: Обработка финансовых данных

**Полномочия:** Обрабатывать данные о транзакциях, формировать отчёты, отправлять сводки
**Точка согласования:** Для анализа отсутствует. Рекомендации требуют одобрения владельца.
**Триггер:** Обнаружен новый файл данных ИЛИ наступил запланированный ежемесячный цикл

### При поступлении новых данных

1. Обнаружить новый файл в назначенном входном каталоге
2. Разобрать и классифицировать все транзакции
3. Сравнить их с целевыми показателями бюджета
4. Отметить необычные позиции, превышения пороговых значений и новые регулярные списания
5. Сформировать отчёт в назначенном выходном каталоге
6. Отправить владельцу сводку через настроенный канал

### Правила эскалации

- Отдельная позиция > $500: немедленное оповещение
- Категория превышает бюджет на 20%: отметить в отчёте
- Нераспознанная транзакция: запросить у владельца категорию
- Ошибка обработки после 2 повторных попыток: сообщить о сбое, не строить догадок
```

### Пример 3: мониторинг и оповещения (непрерывно)

```markdown
## Программа: Мониторинг системы

**Полномочия:** Проверять состояние системы, перезапускать службы, отправлять оповещения
**Точка согласования:** Автоматически перезапускать службы. Эскалировать, если перезапуск дважды завершился неудачей.
**Триггер:** Каждый цикл Heartbeat

### Проверки

- Конечные точки состояния служб отвечают
- Свободное место на диске превышает пороговое значение
- Ожидающие задачи не устарели (>24 часов)
- Каналы доставки работоспособны

### Матрица реагирования

| Условие                 | Действие                                      | Эскалировать?                      |
| ----------------------- | --------------------------------------------- | ---------------------------------- |
| Служба недоступна       | Автоматически перезапустить                   | Только если 2 перезапуска неудачны |
| Свободное место < 10%   | Оповестить владельца                          | Да                                 |
| Задача устарела > 24 ч. | Напомнить владельцу                           | Нет                                |
| Канал недоступен        | Записать в журнал и повторить в следующем цикле | Если недоступен > 2 часов          |
```

## Шаблон «выполнить — проверить — сообщить»

Постоянные распоряжения работают лучше всего в сочетании со строгой дисциплиной выполнения. Каждая задача в постоянном распоряжении должна проходить следующий цикл:

1. **Выполнить** — фактически выполнить работу, а не просто подтвердить получение указания
2. **Проверить** — убедиться в правильности результата: файл существует, сообщение доставлено, данные разобраны
3. **Сообщить** — рассказать владельцу, что было сделано и что было проверено

```markdown
### Правила выполнения

- Каждая задача проходит цикл «Выполнить — проверить — сообщить». Без исключений.
- «Я это сделаю» не является выполнением. Сначала выполните, затем сообщите.
- «Готово» без проверки неприемлемо. Предоставьте доказательство.
- Если выполнение завершилось неудачей: повторите попытку один раз, скорректировав подход.
- Если повторная попытка также не удалась: сообщите о сбое и укажите причину. Никогда не скрывайте сбой.
- Никогда не повторяйте попытки бесконечно: максимум 3 попытки, затем эскалация.
```

Этот шаблон предотвращает наиболее распространённый сбой агента: подтверждение задачи без её выполнения.

## Архитектура с несколькими программами

Для агентов, управляющих несколькими направлениями, организуйте постоянные распоряжения как отдельные программы с чёткими границами:

```markdown
## Программа 1: [Область A] (Еженедельно)

...

## Программа 2: [Область B] (Ежемесячно + по запросу)

...

## Программа 3: [Область C] (По необходимости)

...

## Правила эскалации (Все программы)

- [Общие критерии эскалации]
- [Точки согласования, применимые ко всем программам]
```

Каждая программа должна иметь:

- Собственную **периодичность запуска** (еженедельно, ежемесячно, по событию или непрерывно)
- Собственные **точки согласования** (некоторым программам требуется более строгий контроль)
- Чёткие **границы** (агент должен понимать, где заканчивается одна программа и начинается другая)

## Рекомендации

### Следует

- Начинать с узких полномочий и расширять их по мере роста доверия
- Определять явные точки согласования для действий с высоким риском
- Включать разделы «Что НЕ следует делать» — границы так же важны, как и разрешения
- Сочетать постоянные распоряжения с заданиями Cron для надёжного выполнения по времени
- Еженедельно проверять журналы агента, чтобы убедиться в соблюдении постоянных распоряжений
- Обновлять постоянные распоряжения по мере изменения потребностей — это живые документы

### Не следует

- Предоставлять широкие полномочия с первого дня («делай всё, что считаешь нужным»)
- Пропускать правила эскалации — каждой программе нужен пункт, определяющий, когда следует остановиться и обратиться за помощью
- Полагаться на то, что агент запомнит устные инструкции, — внесите всё в файл
- Смешивать разные направления в одной программе — создавайте отдельные программы для отдельных областей
- Забывать об обеспечении запуска с помощью заданий Cron — постоянные распоряжения без триггеров превращаются в рекомендации
