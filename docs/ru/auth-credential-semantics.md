Эта семантика обеспечивает согласованность поведения аутентификации на этапе выбора и во время выполнения. Она используется совместно в:

- `resolveAuthProfileOrder` (упорядочивание профилей)
- `resolveApiKeyForProfile` (разрешение учетных данных во время выполнения)
- `openclaw models status --probe`
- проверках аутентификации `openclaw doctor` (`doctor-auth`)

## Стабильные коды причин проверки

Результаты проверки содержат категорию `status` (`ok`, `auth`, `rate_limit`, `billing`, `timeout`, `format`, `unknown`, `no_model`), а также стабильный `reasonCode`, если проверка не дошла до вызова модели:

| `reasonCode`             | Значение                                                                                         |
| ------------------------ | ------------------------------------------------------------------------------------------------ |
| `excluded_by_auth_order` | Профиль исключен из явно заданного порядка аутентификации для своего провайдера.                  |
| `missing_credential`     | Не настроены ни встроенные учетные данные, ни SecretRef.                                         |
| `expired`                | Срок действия токена `expires` уже истек.                                                        |
| `invalid_expires`        | `expires` не является допустимой положительной меткой времени Unix в миллисекундах.               |
| `unresolved_ref`         | Не удалось разрешить настроенный SecretRef.                                                      |
| `ineligible_profile`     | Профиль несовместим с конфигурацией провайдера, включая некорректно заданный ключ.                 |
| `no_model`               | Учетные данные существуют, но не удалось определить модель-кандидат, доступную для проверки.      |

Проверки пригодности возвращают код причины `ok` для пригодных учетных данных.

## Учетные данные токена

Учетные данные токена (`type: "token"`) поддерживают встроенный `token` и/или `tokenRef`.

### Правила пригодности

1. Профиль токена непригоден, если отсутствуют и `token`, и `tokenRef` (`missing_credential`).
2. `expires` необязателен. Если он указан, то должен быть конечным числом миллисекунд с начала эпохи Unix, большим `0` и не превышающим максимальную метку времени JavaScript `Date` (8640000000000000).
3. Если `expires` недопустим (неверный тип, `NaN`, `0`, отрицательное или неконечное значение либо значение выше указанного максимума), профиль считается непригодным с кодом `invalid_expires`.
4. Если `expires` находится в прошлом, профиль считается непригодным с кодом `expired`.
5. `tokenRef` не позволяет обойти проверку `expires`.

### Правила разрешения

1. Семантика средства разрешения для `expires` совпадает с семантикой проверки пригодности.
2. Для пригодных профилей значение токена может быть разрешено из встроенного значения или `tokenRef`.
3. Неразрешимые ссылки приводят к коду `unresolved_ref` в выводе `models status --probe`.

## Переносимость при копировании агента

Наследование аутентификации агентом выполняется сквозным чтением. Если у агента нет локального профиля, во время выполнения он разрешает профили из хранилища агента по умолчанию/главного агента, не копируя секретные данные в собственное хранилище учетных данных (`agents/<agentId>/agent/openclaw-agent.sqlite`).

Явные операции копирования, например `openclaw agents add`, используют следующую политику переносимости:

- Профили `api_key` и `token` переносимы, если не задано `copyToAgents: false`.
- Профили `oauth` по умолчанию непереносимы, поскольку токены обновления могут быть одноразовыми или зависеть от ротации.
- OAuth-потоки, принадлежащие провайдеру, могут включить перенос с помощью `copyToAgents: true`, только если известно, что копирование данных обновления между агентами безопасно; это разрешение применяется только к профилям со встроенными данными доступа/обновления.

Непереносимые профили остаются доступными через наследование со сквозным чтением, если целевой агент не выполнит отдельный вход и не создаст собственный локальный профиль.

## Маршруты аутентификации только в конфигурации

Записи `auth.profiles` с `mode: "aws-sdk"` являются метаданными маршрутизации, а не сохраненными учетными данными. Они допустимы, когда целевой провайдер использует `models.providers.<id>.auth: "aws-sdk"` — этот маршрут записывается настройкой Amazon Bedrock, принадлежащей Plugin. Идентификаторы таких профилей могут присутствовать в `auth.order` и переопределениях сеанса, даже если в хранилище учетных данных нет соответствующей записи.

Не записывайте `type: "aws-sdk"` в хранилище учетных данных; сохраненные учетные данные могут иметь только тип `api_key`, `token` или `oauth`. Если устаревший файл `auth-profiles.json` содержит такой маркер, `openclaw doctor --fix` переносит его в `auth.profiles` и удаляет маркер из хранилища.

## Фильтрация по явно заданному порядку аутентификации

- Если для провайдера задано `auth.order.<provider>` или переопределение порядка в хранилище аутентификации, `models status --probe` проверяет только идентификаторы профилей, оставшиеся в разрешенном порядке аутентификации для этого провайдера. Сохраненное переопределение имеет приоритет над конфигурацией `auth.order`.
- Сохраненный профиль этого провайдера, исключенный из явно заданного порядка, не проверяется неявно позднее. В выводе проверки для него указывается `reasonCode: excluded_by_auth_order` и пояснение `Excluded by auth.order for this provider.`

## Разрешение цели проверки

- Цели проверки могут поступать из профилей аутентификации, учетных данных окружения или `models.json` (`source` результата: `profile`, `env`, `models.json`).
- Если у провайдера есть учетные данные, но OpenClaw не может определить для него модель-кандидат, доступную для проверки, `models status --probe` возвращает `status: no_model` с `reasonCode: no_model`.

## Обнаружение учетных данных внешних CLI

- Учетные данные только для времени выполнения, принадлежащие внешним CLI (Claude CLI для `claude-cli`, Codex CLI для `openai`, MiniMax CLI для `minimax-portal`), обнаруживаются только тогда, когда провайдер, среда выполнения или профиль аутентификации входят в область текущей операции либо когда сохраненный локальный профиль для этого внешнего источника уже существует.
- Вызывающие стороны хранилища аутентификации выбирают явный режим обнаружения внешнего CLI: `none` — только для сохраненной аутентификации/аутентификации Plugin, `existing` — для обновления уже сохраненных профилей внешних CLI или `scoped` — для конкретного набора провайдеров/профилей.
- Пути только для чтения/получения состояния передают `allowKeychainPrompt: false`; они используют только учетные данные внешних CLI, сохраненные в файлах, и не читают и не используют повторно результаты macOS Keychain.

## Защитная политика OAuth SecretRef

Входные данные SecretRef предназначены только для статических учетных данных. Учетные данные OAuth изменяемы во время выполнения (потоки обновления сохраняют ротированные токены), поэтому использование OAuth-данных через SecretRef разделило бы изменяемое состояние между хранилищами.

- Если учетные данные профиля имеют `type: "oauth"`, объекты SecretRef отклоняются во всех полях данных учетных данных этого профиля.
- Если `auth.profiles.<id>.mode` имеет значение `"oauth"`, входные данные `keyRef`/`tokenRef` на основе SecretRef для этого профиля отклоняются.
- Нарушения приводят к жестким сбоям (выбрасываемым ошибкам) в путях подготовки секретов при запуске/перезагрузке и разрешения профилей.

## Сообщения с поддержкой устаревших сценариев

Для совместимости со скриптами первая строка ошибок проверки остается неизменной:

`Auth profile credentials are missing or expired.`

На следующих строках выводятся понятное пользователю пояснение и стабильный код причины в формате `↳ Auth reason [code]: ...`.
