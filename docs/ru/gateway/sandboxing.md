OpenClaw может выполнять инструменты внутри изолированной среды, чтобы уменьшить масштаб возможного ущерба. Изоляция по умолчанию отключена и управляется параметром `agents.defaults.sandbox` (глобально) или `agents.list[].sandbox` (для каждого агента). Процесс Gateway всегда остаётся на хосте; при включённой изоляции в изолированную среду переносится только выполнение инструментов.

<Note>
Это не идеальная граница безопасности, но она существенно ограничивает доступ к файловой системе и процессам, когда модель совершает неразумные действия.
</Note>

## Что выполняется в изолированной среде

- Выполнение инструментов: `exec`, `read`, `write`, `edit`, `apply_patch`, `process` и т. д.
- Необязательный изолированный браузер (`agents.defaults.sandbox.browser`).

Не выполняются в изолированной среде:

- Сам процесс Gateway.
- Любой инструмент, которому через `tools.elevated` явно разрешено работать вне изолированной среды. Выполнение с повышенными привилегиями обходит изоляцию и использует настроенный путь выхода (`gateway` по умолчанию или `node`, когда целью выполнения является `node`). Если изоляция отключена, `tools.elevated` ничего не меняет, поскольку выполнение уже происходит на хосте. См. [Режим повышенных привилегий](/ru/tools/elevated).

## Режимы, область и бэкенд

Поведение изолированной среды определяется тремя независимыми настройками:

| Настройка | Ключ                              | Значения                     | По умолчанию |
| --------- | --------------------------------- | ---------------------------- | ------------ |
| Режим     | `agents.defaults.sandbox.mode`    | `off`, `non-main`, `all`     | `off`        |
| Область   | `agents.defaults.sandbox.scope`   | `agent`, `session`, `shared` | `agent`      |
| Бэкенд    | `agents.defaults.sandbox.backend` | `docker`, `ssh`, `openshell` | `docker`     |

**Режим** определяет, когда применяется изоляция:

- `off`: без изоляции.
- `non-main`: изолировать каждый сеанс, кроме основного сеанса агента. Ключ основного сеанса всегда имеет вид `agent:<agentId>:main` (или `global`, когда `session.scope` равен `"global"`); его нельзя настроить. У групповых сеансов и сеансов каналов собственные ключи, поэтому они всегда считаются неосновными и выполняются в изолированной среде.
- `all`: каждый сеанс выполняется в изолированной среде.

**Область** определяет количество создаваемых контейнеров или окружений:

- `agent`: один контейнер на агента.
- `session`: один контейнер на сеанс.
- `shared`: один контейнер, общий для всех изолированных сеансов (при этой области переопределения `docker`/`ssh`/`browser` для отдельных агентов игнорируются).

**Бэкенд** определяет, какая среда выполнения запускает изолированные инструменты. Настройки SSH находятся в `agents.defaults.sandbox.ssh`, а настройки OpenShell — в `plugins.entries.openshell.config`.

|                       | Docker                              | SSH                                     | OpenShell                                                   |
| --------------------- | ----------------------------------- | --------------------------------------- | ----------------------------------------------------------- |
| **Где выполняется**   | Локальный контейнер                 | Любой хост с доступом по SSH             | Изолированная среда под управлением OpenShell               |
| **Настройка**         | `scripts/sandbox-setup.sh`          | Ключ SSH и целевой хост                  | Включённый Plugin OpenShell                                  |
| **Модель рабочей области** | Подключение каталога или копирование | Удалённая каноническая копия (однократное заполнение) | `mirror` или `remote`                              |
| **Управление сетью**  | `docker.network` (по умолчанию: нет)| Зависит от удалённого хоста              | Зависит от OpenShell                                        |
| **Изолированный браузер** | Поддерживается                  | Не поддерживается                        | Пока не поддерживается                                      |
| **Подключения каталогов** | `docker.binds`                   | Н/Д                                     | Н/Д                                                         |
| **Лучше всего подходит для** | Локальной разработки, полной изоляции | Переноса нагрузки на удалённую машину | Управляемых удалённых изолированных сред с необязательной двусторонней синхронизацией |

## Бэкенд Docker

После включения изоляции Docker используется как бэкенд по умолчанию. Он локально запускает инструменты и изолированные браузеры через сокет демона Docker (`/var/run/docker.sock`); изоляцию обеспечивают пространства имён Docker.

Значения по умолчанию: `network: "none"` (без исходящих подключений), `readOnlyRoot: true`, `capDrop: ["ALL"]`, образ `openclaw-sandbox:bookworm-slim`.

Чтобы предоставить доступ к графическим процессорам хоста, задайте для `agents.defaults.sandbox.docker.gpus` (или переопределения для отдельного агента) значение наподобие `"all"` или `"device=GPU-uuid"`. Оно передаётся флагу Docker `--gpus` и требует совместимой среды выполнения на хосте, например NVIDIA Container Toolkit.

<Warning>
**Ограничения Docker-out-of-Docker (DooD)**

Если сам Gateway OpenClaw развёрнут в контейнере Docker, он управляет соседними изолированными контейнерами через сокет Docker хоста (DooD). Это накладывает ограничение на сопоставление путей:

- **В конфигурации необходимы пути хоста**: параметр `workspace` в `openclaw.json` должен содержать **абсолютный путь на хосте** (например, `/home/user/.openclaw/workspaces`), а не внутренний путь контейнера Gateway. Демон Docker интерпретирует пути относительно пространства имён ОС хоста, а не пространства имён самого Gateway.
- **Необходимо идентичное сопоставление тома**: процесс Gateway также записывает файлы Heartbeat и моста по этому пути `workspace`. Предоставьте контейнеру Gateway идентичное сопоставление тома (`-v /home/user/.openclaw:/home/user/.openclaw`), чтобы тот же путь хоста правильно разрешался и внутри контейнера Gateway. Несовпадающие сопоставления приводят к ошибке `EACCES`, когда Gateway пытается записать файл Heartbeat.
- **Режим кода Codex**: когда активна изолированная среда OpenClaw, для этого обращения OpenClaw отключает встроенный Code Mode сервера приложений Codex, пользовательские серверы MCP и выполнение Plugin на базе приложений (они запускаются процессом сервера приложений на хосте Gateway, а не бэкендом изолированной среды OpenClaw), если политика инструментов изолированной среды не предоставляет необходимые инструменты и вы не включили экспериментальный путь сервера выполнения изолированной среды. После этого доступ к оболочке направляется через инструменты OpenClaw, работающие в изолированной среде, например `sandbox_exec` и `sandbox_process`. Не подключайте сокет Docker хоста к контейнерам изолированной среды агентов или пользовательским изолированным средам Codex. Полное описание поведения см. в разделе [Среда Codex](/ru/plugins/codex-harness).

На хостах Ubuntu/AppArmor с включённым режимом изоляции Docker для выполнения команд оболочки Codex app-server с `workspace-write` необходимы непривилегированные пользовательские пространства имён внутри изолированного контейнера; запуск может завершиться ошибкой ещё до запуска оболочки, если пользователь службы не может их создать. Когда исходящие подключения из изолированной среды Docker отключены (`network: "none"`, значение по умолчанию), также требуется непривилегированное сетевое пространство имён. Типичные признаки: `bwrap: setting up uid map: Permission denied` и `bwrap: loopback: Failed RTM_NEWADDR: Operation not permitted`. Запустите `openclaw doctor`; если он сообщает об ошибке проверки пространства имён Codex bwrap, предпочтительно использовать профиль AppArmor, предоставляющий процессу службы OpenClaw необходимые пространства имён. Параметр `kernel.apparmor_restrict_unprivileged_userns=0` служит запасным вариантом для всего хоста, но снижает безопасность; используйте его только в том случае, если это допустимо для модели безопасности данного хоста.
</Warning>

### Изолированный браузер

- Изолированный браузер запускается автоматически (чтобы обеспечить доступность CDP), когда он требуется инструменту браузера. Настройка выполняется через `agents.defaults.sandbox.browser.autoStart` (по умолчанию `true`) и `autoStartTimeoutMs` (по умолчанию 12 с).
- Контейнеры изолированного браузера используют выделенную сеть Docker (`openclaw-sandbox-browser`) вместо глобальной сети `bridge`. Настройте её с помощью `agents.defaults.sandbox.browser.network`.
- `agents.defaults.sandbox.browser.cdpSourceRange` ограничивает входящий доступ CDP на границе контейнера с помощью списка разрешённых CIDR (например, `172.21.0.1/32`).
- Доступ наблюдателя noVNC по умолчанию защищён паролем; OpenClaw создаёт URL с краткосрочным токеном, который открывает локальную загрузочную страницу и запускает noVNC с паролем во фрагменте URL (а не в строке запроса или журналах заголовков).
- `agents.defaults.sandbox.browser.allowHostControl` (по умолчанию `false`) позволяет изолированным сеансам явно обращаться к браузеру хоста.
- Необязательные списки разрешений ограничивают `target: "custom"`: `allowedControlUrls`, `allowedControlHosts`, `allowedControlPorts`.

## Бэкенд SSH

Используйте `backend: "ssh"`, чтобы выполнять `exec`, файловые инструменты и чтение мультимедиа в изолированной среде на произвольной машине с доступом по SSH.

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "ssh",
        scope: "session",
        workspaceAccess: "rw",
        ssh: {
          target: "user@gateway-host:22",
          workspaceRoot: "/tmp/openclaw-sandboxes",
          strictHostKeyChecking: true,
          updateHostKeys: true,
          identityFile: "~/.ssh/id_ed25519",
          certificateFile: "~/.ssh/id_ed25519-cert.pub",
          knownHostsFile: "~/.ssh/known_hosts",
          // Или используйте SecretRefs / встроенное содержимое вместо локальных файлов:
          // identityData: { source: "env", provider: "default", id: "SSH_IDENTITY" },
          // certificateData: { source: "env", provider: "default", id: "SSH_CERTIFICATE" },
          // knownHostsData: { source: "env", provider: "default", id: "SSH_KNOWN_HOSTS" },
        },
      },
    },
  },
}
```

Значения по умолчанию: `command: "ssh"`, `workspaceRoot: "/tmp/openclaw-sandboxes"`, `strictHostKeyChecking: true`, `updateHostKeys: true`.

- **Жизненный цикл**: OpenClaw создаёт удалённый корневой каталог для каждой области в `sandbox.ssh.workspaceRoot`. При первом использовании после создания или повторного создания удалённая рабочая область однократно заполняется из локальной рабочей области. После этого `exec`, `read`, `write`, `edit`, `apply_patch`, чтение мультимедиа для запросов и подготовка входящих мультимедиа выполняются непосредственно в удалённой рабочей области через SSH. OpenClaw не синхронизирует удалённые изменения обратно в локальную рабочую область автоматически.
- **Материалы аутентификации**: `identityFile`/`certificateFile`/`knownHostsFile` ссылаются на существующие локальные файлы. `identityData`/`certificateData`/`knownHostsData` принимают встроенные строки или SecretRefs, разрешаемые через обычный снимок среды выполнения секретов, записываются во временные файлы с режимом `0600` и удаляются после завершения сеанса SSH. Если для одного элемента заданы варианты `*File` и `*Data`, в этом сеансе приоритет имеет `*Data`.
- **Последствия удалённой канонической модели**: после первоначального заполнения удалённая рабочая область SSH становится фактическим состоянием изолированной среды. Локальные изменения на хосте, внесённые вне OpenClaw после этапа заполнения, не видны удалённо до повторного создания изолированной среды. Команда `openclaw sandbox recreate` удаляет удалённый корневой каталог соответствующей области, после чего при следующем использовании он снова заполняется из локальной рабочей области. Этот бэкенд не поддерживает изоляцию браузера, а настройки `sandbox.docker.*` к нему не применяются.

## Бэкенд OpenShell

Используйте `backend: "openshell"`, чтобы выполнять инструменты в удалённой среде под управлением OpenShell. OpenShell использует тот же транспорт SSH и мост удалённой файловой системы, что и универсальный бэкенд SSH, а также добавляет управление жизненным циклом OpenShell (`sandbox create/get/delete/ssh-config`) и необязательный режим синхронизации рабочей области `mirror`.

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "openshell",
        scope: "session",
        workspaceAccess: "rw",
      },
    },
  },
  plugins: {
    entries: {
      openshell: {
        enabled: true,
        config: {
          from: "openclaw",
          mode: "remote", // mirror | remote
        },
      },
    },
  },
}
```

`mode: "mirror"` (по умолчанию) сохраняет локальную рабочую область канонической: OpenClaw синхронизирует локальные данные с изолированной средой перед `exec`, а после выполнения синхронизирует изменения обратно. `mode: "remote"` однократно заполняет удалённую рабочую область локальными данными, после чего `exec`/`read`/`write`/`edit`/`apply_patch` выполняются непосредственно в удалённой рабочей области без обратной синхронизации; локальные изменения после заполнения не видны, пока вы не выполните `openclaw sandbox recreate`. При `scope: "agent"` или `scope: "shared"` удалённая рабочая область совместно используется в пределах той же области. Текущие ограничения: изолированный браузер пока не поддерживается, а `sandbox.docker.binds` к этому бэкенду не применяется.

Команды `openclaw sandbox list`/`recreate`/prune обрабатывают среды выполнения OpenShell так же, как среды выполнения Docker; логика очистки учитывает бэкенд.

Полные требования, справочник по конфигурации, сравнение режимов рабочей области и подробности жизненного цикла см. в разделе [OpenShell](/ru/gateway/openshell).

## Доступ к рабочей области

`agents.defaults.sandbox.workspaceAccess` определяет, что доступно изолированной среде:

| Значение         | Поведение                                                                                      |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| `none` (по умолчанию) | Инструменты видят изолированное рабочее пространство песочницы в `~/.openclaw/sandboxes`. |
| `ro`             | Монтирует рабочее пространство агента только для чтения в `/agent` (отключает `write`/`edit`/`apply_patch`). |
| `rw`             | Монтирует рабочее пространство агента для чтения и записи в `/workspace`.                      |

При использовании бэкенда OpenShell режим `mirror` по-прежнему использует локальное рабочее пространство как канонический источник между вызовами exec, режим `remote` после первоначального заполнения использует как каноническое удалённое рабочее пространство OpenShell, а `workspaceAccess: "ro"`/`"none"` по-прежнему ограничивает запись тем же образом.

Входящие медиафайлы копируются в активное рабочее пространство песочницы (`media/inbound/*`).

<Note>
**Skills**: корнем инструмента `read` является песочница. При `workspaceAccess: "none"` OpenClaw зеркалирует подходящие Skills в рабочее пространство песочницы (`.../skills`), чтобы их можно было читать. При `"rw"` Skills рабочего пространства доступны для чтения из `/workspace/skills`, а подходящие управляемые, встроенные или предоставленные плагинами Skills материализуются в созданном пути только для чтения `/workspace/.openclaw/sandbox-skills/skills`.
</Note>

## Пользовательские bind-монтирования

`agents.defaults.sandbox.docker.binds` монтирует дополнительные каталоги хоста в контейнер. Формат: `хост:контейнер:режим` (например, `"/home/user/source:/source:rw"`).

Глобальные и заданные для отдельных агентов bind-монтирования объединяются (а не заменяются). При `scope: "shared"` bind-монтирования отдельных агентов игнорируются.

`agents.defaults.sandbox.browser.binds` монтирует дополнительные каталоги хоста только в контейнер **браузера песочницы**. Если этот параметр задан (включая `[]`), он заменяет `docker.binds` для контейнера браузера; если он не указан, контейнер браузера использует `docker.binds`.

```json5
{
  agents: {
    defaults: {
      sandbox: {
        docker: {
          binds: ["/home/user/source:/source:ro", "/var/data/myapp:/data:ro"],
        },
      },
    },
    list: [
      {
        id: "build",
        sandbox: {
          docker: {
            binds: ["/mnt/cache:/cache:rw"],
          },
        },
      },
    ],
  },
}
```

<Warning>
**Безопасность bind-монтирований**

- Bind-монтирования обходят файловую систему песочницы: они предоставляют доступ к путям хоста в заданном вами режиме (`:ro` или `:rw`).
- По умолчанию OpenClaw блокирует опасные источники bind-монтирования: системные пути (`/etc`, `/proc`, `/sys`, `/dev`, `/root`, `/boot`), каталоги сокетов Docker (`/run`, `/var/run` и находящиеся в них варианты `docker.sock`), а также распространённые корневые каталоги учётных данных в домашнем каталоге (`~/.aws`, `~/.cargo`, `~/.config`, `~/.docker`, `~/.gnupg`, `~/.netrc`, `~/.npm`, `~/.ssh`).
- При проверке исходный путь нормализуется, затем повторно разрешается через самого глубокого существующего предка, после чего снова проверяются заблокированные пути и разрешённые корни. Поэтому попытки выхода через родительскую символьную ссылку блокируются даже тогда, когда конечный элемент ещё не существует (например, `/workspace/run-link/new-file` всё равно разрешается как `/var/run/...`, если `run-link` указывает туда).
- Целевые пути bind-монтирования, перекрывающие зарезервированные точки монтирования контейнера (`/workspace`, `/agent`), также по умолчанию блокируются; это можно переопределить с помощью `agents.defaults.sandbox.docker.dangerouslyAllowReservedContainerTargets: true`.
- Источники bind-монтирования за пределами разрешённых корней рабочего пространства или рабочего пространства агента по умолчанию блокируются; это можно переопределить с помощью `agents.defaults.sandbox.docker.dangerouslyAllowExternalBindSources: true`. Разрешённые корни канонизируются таким же образом, поэтому путь, который выглядит находящимся внутри разрешённого списка только до разрешения символьных ссылок, всё равно отклоняется как находящийся за пределами разрешённых корней.
- Конфиденциальные монтирования (секреты, ключи SSH, учётные данные служб) следует выполнять в режиме `:ro`, если запись не является абсолютно необходимой.
- Если нужен только доступ к рабочему пространству для чтения, используйте совместно с `workspaceAccess: "ro"`; режимы bind-монтирований остаются независимыми.
- О взаимодействии bind-монтирований с политикой инструментов и привилегированным выполнением см. в разделе [Песочница, политика инструментов и привилегированное выполнение](/ru/gateway/sandbox-vs-tool-policy-vs-elevated).

</Warning>

## Образы и настройка

Образ Docker по умолчанию: `openclaw-sandbox:bookworm-slim`

<Note>
**Исходный репозиторий и установка через npm**

Вспомогательные сценарии `scripts/sandbox-setup.sh`, `scripts/sandbox-common-setup.sh` и `scripts/sandbox-browser-setup.sh` доступны только при запуске из [рабочей копии исходного репозитория](https://github.com/openclaw/openclaw). Они не входят в пакет npm.

Если вы установили OpenClaw с помощью `npm install -g openclaw`, используйте приведённые ниже встроенные команды `docker build`.
</Note>

<Steps>
  <Step title="Сборка образа по умолчанию">
    Из рабочей копии исходного репозитория:

    ```bash
    scripts/sandbox-setup.sh
    ```

    При установке через npm (рабочая копия исходного репозитория не требуется):

    ```bash
    docker build -t openclaw-sandbox:bookworm-slim - <<'DOCKERFILE'
    FROM debian:bookworm-slim
    ENV DEBIAN_FRONTEND=noninteractive
    RUN apt-get update && apt-get install -y --no-install-recommends \
      bash ca-certificates curl git jq python3 ripgrep \
      && rm -rf /var/lib/apt/lists/*
    RUN useradd --create-home --shell /bin/bash sandbox
    USER sandbox
    WORKDIR /home/sandbox
    CMD ["sleep", "infinity"]
    DOCKERFILE
    ```

    Образ по умолчанию **не** содержит Node. Если для Skills требуется Node (или другие среды выполнения), создайте собственный образ с ними или установите их через `sandbox.docker.setupCommand` (требуются исходящий доступ к сети, доступная для записи корневая файловая система и пользователь root).

    Если `openclaw-sandbox:bookworm-slim` отсутствует, OpenClaw не подменяет его незаметно обычным `debian:bookworm-slim`. Запуски песочницы, использующие образ по умолчанию, немедленно завершаются с инструкцией по сборке, пока вы не соберёте его, поскольку встроенный образ содержит `python3`, необходимый вспомогательным инструментам записи и редактирования в песочнице.

  </Step>
  <Step title="Необязательно: сборка общего образа">
    Чтобы получить более функциональный образ песочницы с распространёнными инструментами (например, `curl`, `jq`, Node 24, pnpm, `python3` и `git`):

    Из рабочей копии исходного репозитория:

    ```bash
    scripts/sandbox-common-setup.sh
    ```

    При установке через npm сначала соберите образ по умолчанию (см. выше), а затем на его основе соберите общий образ, используя файл [`scripts/docker/sandbox/Dockerfile.common`](https://github.com/openclaw/openclaw/blob/main/scripts/docker/sandbox/Dockerfile.common) из репозитория.

    Затем задайте для `agents.defaults.sandbox.docker.image` значение `openclaw-sandbox-common:bookworm-slim`.

  </Step>
  <Step title="Необязательно: сборка образа браузера песочницы">
    Из рабочей копии исходного репозитория:

    ```bash
    scripts/sandbox-browser-setup.sh
    ```

    При установке через npm выполните сборку с помощью файла [`scripts/docker/sandbox/Dockerfile.browser`](https://github.com/openclaw/openclaw/blob/main/scripts/docker/sandbox/Dockerfile.browser) из репозитория.

  </Step>
</Steps>

## setupCommand (однократная настройка контейнера)

`setupCommand` выполняется **один раз** после создания контейнера песочницы (а не при каждом запуске). Команда выполняется внутри контейнера через `sh -lc`.

Пути:

- Глобальный: `agents.defaults.sandbox.docker.setupCommand`
- Для отдельного агента: `agents.list[].sandbox.docker.setupCommand`

<AccordionGroup>
  <Accordion title="Распространённые ошибки">
    - Значение `docker.network` по умолчанию — `"none"` (без исходящего доступа), поэтому установка пакетов завершится ошибкой.
    - Для `docker.network: "container:<id>"` требуется `dangerouslyAllowContainerNamespaceJoin: true`; этот режим предназначен только для аварийного использования.
    - `readOnlyRoot: true` запрещает запись; задайте `readOnlyRoot: false` или создайте собственный образ.
    - Для установки пакетов `user` должен быть пользователем root (не указывайте `user` или задайте `user: "0:0"`).
    - Выполнение команд в песочнице **не** наследует `process.env` хоста. Для API-ключей Skills используйте `agents.defaults.sandbox.docker.env` (или собственный образ).
    - Значения из `agents.defaults.sandbox.docker.env` передаются как явно заданные переменные среды контейнера Docker. Любой пользователь с доступом к демону Docker может просмотреть их с помощью команд метаданных Docker, например `docker inspect`. Если такое раскрытие через метаданные неприемлемо, используйте собственный образ, смонтированный файл с секретом или другой способ доставки секретов.

  </Accordion>
</AccordionGroup>

## Политика инструментов и способы обхода ограничений

Политики разрешения и запрета инструментов применяются до правил песочницы. Если инструмент запрещён глобально или для отдельного агента, песочница не вернёт к нему доступ.

`tools.elevated` — это явный способ обхода ограничений, который запускает `exec` вне песочницы (по умолчанию в `gateway`, либо в `node`, если целью выполнения является `node`). Директивы `/exec` применяются только к авторизованным отправителям и сохраняются в рамках сеанса; чтобы полностью отключить `exec`, запретите его в политике инструментов (см. [Песочница, политика инструментов и привилегированное выполнение](/ru/gateway/sandbox-vs-tool-policy-vs-elevated)).

Диагностика:

- `openclaw sandbox list` показывает контейнеры песочницы, их состояние, соответствие образу, возраст, время простоя и связанный сеанс или агент.
- `openclaw sandbox explain [--session <key>] [--agent <id>]` проверяет фактический режим песочницы, рабочее пространство хоста, рабочий каталог среды выполнения, монтирования Docker, политику инструментов и ключи конфигурации для исправления. Поле `workspaceRoot` остаётся настроенным корнем песочницы; `effectiveHostWorkspaceRoot` показывает фактическое расположение активного рабочего пространства.
- `openclaw sandbox recreate [--all | --session <key> | --agent <id>] [--browser] [--force]` удаляет контейнеры и среды, чтобы при следующем использовании они были созданы заново с текущей конфигурацией.
- Модель для понимания причин блокировки описана в разделе [Песочница, политика инструментов и привилегированное выполнение](/ru/gateway/sandbox-vs-tool-policy-vs-elevated).

## Переопределения для нескольких агентов

Каждый агент может переопределять настройки песочницы и инструментов: `agents.list[].sandbox` и `agents.list[].tools` (а также `agents.list[].tools.sandbox.tools` для политики инструментов песочницы). Порядок приоритетов см. в разделе [Песочница и инструменты для нескольких агентов](/ru/tools/multi-agent-sandbox-tools).

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        scope: "session",
        workspaceAccess: "none",
      },
    },
  },
}
```
