# Журналирование

Обзор для пользователей (CLI + Control UI + конфигурация) см. в разделе [/logging](/ru/logging).

OpenClaw предоставляет два вида журналов:

- **Вывод в консоль** — то, что отображается в терминале или интерфейсе отладки.
- **Файловые журналы** — строки JSON, записываемые регистратором Gateway.

При запуске Gateway записывает в журнал разрешённую модель агента по умолчанию, а также режимы по умолчанию, влияющие на новые сеансы:

```text
agent model: openai/gpt-5.6-sol (thinking=medium, fast=on)
```

Значение `thinking` берётся из настроек агента по умолчанию, параметров модели или глобальных настроек агента по умолчанию; если оно не задано, отображается `medium`. Значение `fast` берётся из настроек агента по умолчанию или параметров `fastMode` модели.

## Файловый регистратор

- По умолчанию ротируемый файл журнала находится в `/tmp/openclaw/` (один файл в день): `openclaw-YYYY-MM-DD.log`; дата определяется по локальному часовому поясу узла Gateway. Если этот каталог небезопасен или недоступен для записи (неверный владелец, запись разрешена всем или каталог является символической ссылкой), OpenClaw вместо него использует пользовательский путь `os.tmpdir()/openclaw-<uid>`; в Windows всегда используется этот резервный путь во временном каталоге ОС.
- Активные файлы журналов ротируются при достижении размера `logging.maxFileBytes` (по умолчанию 100 МБ). Сохраняется до пяти нумерованных архивов (от `.1` до `.5`), после чего запись продолжается в новый активный файл.
- Путь к файлу журнала и уровень журналирования настраиваются в `~/.openclaw/openclaw.json`: `logging.file`, `logging.level`.
- Формат файла — один объект JSON в каждой строке.

Пути кода для разговоров, голосовой связи в реальном времени и управляемых комнат используют общий файловый регистратор для ограниченных записей жизненного цикла, предназначенных для оперативной отладки и экспорта журналов OTLP. Текст расшифровки, звуковые данные, идентификаторы реплик, вызовов и элементов провайдера никогда не копируются в запись журнала.

Вкладка журналов в Control UI отслеживает этот файл через Gateway (`logs.tail`). CLI делает то же самое:

```bash
openclaw logs --follow
```

### Подробный режим и уровни журналирования

- **Файловые журналы** управляются исключительно параметром `logging.level`.
- `--verbose` влияет только на **подробность вывода в консоль** (и стиль журналов WS) — он **не** повышает уровень файлового журнала.
- Чтобы сохранять в файловых журналах сведения, доступные только в подробном режиме, установите для `logging.level` значение `debug` или `trace`.
- Трассировочное журналирование также включает диагностические сводки времени выполнения для некоторых критических путей, например подготовки фабрики инструментов Plugin. См. [/tools/plugin#slow-plugin-tool-setup](/ru/tools/plugin#slow-plugin-tool-setup).

## Перехват консольного вывода

CLI перехватывает `console.log/info/warn/error/debug/trace`, записывает сообщения в файловые журналы и одновременно выводит их в stdout/stderr.

Подробность вывода в консоль настраивается независимо:

- `logging.consoleLevel` (по умолчанию `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`; по умолчанию `pretty` в TTY и `compact` в остальных случаях)

## Маскирование

OpenClaw маскирует конфиденциальные токены до того, как журналы или расшифровки покидают процесс. Эта политика применяется к консоли, файловым журналам, записям журналов OTLP и текстовым хранилищам расшифровок сеансов, поэтому совпадающие секретные значения маскируются до записи строк JSONL или сообщений на диск.

- `logging.redactSensitive`: `off` | `tools` (по умолчанию `tools`)
- `logging.redactPatterns`: массив строк с регулярными выражениями (переопределяет значения по умолчанию)
  - Используйте необрамлённые строки регулярных выражений (флаги `gi` добавляются автоматически) или `/pattern/flags` для пользовательских флагов.
  - Совпадения маскируются с сохранением первых 6 и последних 4 символов (для значений длиной не менее 18 символов); более короткие значения заменяются на `***`.
  - Шаблоны по умолчанию охватывают распространённые присваивания ключей, флаги CLI, поля JSON, заголовки Bearer, блоки PEM, популярные префиксы токенов поставщиков и названия полей платёжных реквизитов (номер карты, CVC/CVV, общий платёжный токен, платёжные учётные данные).

Некоторые границы безопасности всегда применяют маскирование независимо от `logging.redactSensitive`: события вызовов инструментов Control UI, вывод инструмента `sessions_history`, экспорт диагностических данных для службы поддержки, наблюдения за ошибками провайдера, отображение команд при подтверждении выполнения и журналы протокола WebSocket Gateway. На этих поверхностях `logging.redactPatterns` по-прежнему учитывается как набор дополнительных шаблонов, однако `redactSensitive: "off"` не разрешает вывод необработанных секретов.

## Журналы WebSocket Gateway

Gateway выводит журналы протокола WebSocket в двух режимах:

- **Обычный режим (без `--verbose`)**: выводятся только «значимые» результаты RPC — ошибки (`ok=false`), медленные вызовы (порог по умолчанию: `>= 50ms`) и ошибки разбора.
- **Подробный режим (`--verbose`)**: выводится весь трафик запросов и ответов WS.

### Стиль журналов WS

`openclaw gateway` поддерживает выбор стиля для каждого Gateway:

- `--ws-log auto` (по умолчанию): обычный режим оптимизирован, а в подробном режиме используется компактный вывод.
- `--ws-log compact`: компактный вывод (парные запросы и ответы) в подробном режиме.
- `--ws-log full`: полный вывод каждого кадра в подробном режиме.
- `--compact`: псевдоним для `--ws-log compact`.

```bash
# оптимизированный режим (только ошибки и медленные вызовы)
openclaw gateway

# показать весь трафик WS (парные запросы и ответы)
openclaw gateway --verbose --ws-log compact

# показать весь трафик WS (полные метаданные)
openclaw gateway --verbose --ws-log full
```

## Форматирование консоли (журналирование подсистем)

Средство форматирования консоли **учитывает TTY** и выводит единообразные строки с префиксами. Регистраторы подсистем группируют вывод и упрощают его просмотр:

- **Префиксы подсистем** в каждой строке (например, `[gateway]`, `[canvas]`, `[tailscale]`).
- **Цвета подсистем** (стабильные для каждой подсистемы и хешируемые из её имени), а также цветовая маркировка уровней.
- **Цветной вывод, если используется TTY** или окружение похоже на терминал с расширенными возможностями (`TERM`/`COLORTERM`/`TERM_PROGRAM`); учитываются `NO_COLOR` и `FORCE_COLOR`.
- **Сокращённые префиксы подсистем**: начальный сегмент `gateway/`, `channels/` или `providers/` удаляется, после чего сохраняются не более двух последних оставшихся сегментов (например, `channels/turn/kernel` отображается как `turn/kernel`). Известные подсистемы каналов (`telegram`, `whatsapp`, `slack` и т. д.) всегда сокращаются до имени канала.
- **Вложенные регистраторы по подсистемам** (автоматический префикс + структурированное поле `{ subsystem }`).
- **`logRaw()`** для вывода QR-кодов и пользовательского интерфейса (без префикса и форматирования).
- **Стили консоли**: `pretty` | `compact` | `json`.
- **Уровень журналирования консоли** не зависит от уровня файлового журнала (файловый журнал сохраняет полную детализацию, если для `logging.level` задано `debug` или `trace`).
- **Текст сообщений WhatsApp** записывается на уровне `debug` (используйте `--verbose`, чтобы увидеть его).

Это обеспечивает стабильность файловых журналов и удобство просмотра интерактивного вывода.
