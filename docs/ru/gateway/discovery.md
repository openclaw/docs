У OpenClaw есть две связанные, но разные задачи обнаружения:

1. **Удаленное управление оператором**: приложение в строке меню macOS управляет Gateway, запущенным на другом устройстве.
2. **Сопряжение Node**: iOS/Android (и будущие Node) находят Gateway и безопасно выполняют сопряжение.

Все сетевое обнаружение и анонсирование выполняются в **Node Gateway**
(`openclaw gateway`); клиенты (приложение для Mac, iOS) только используют их.

## Термины

- **Gateway**: единый долгоживущий процесс, который управляет состоянием (сеансами,
  сопряжением, реестром Node) и запускает каналы. В большинстве конфигураций используется один процесс на хост;
  также возможны изолированные конфигурации с несколькими Gateway.
- **Gateway WS (плоскость управления)**: конечная точка WebSocket на `127.0.0.1:18789`
  по умолчанию; привяжите ее к локальной сети/tailnet с помощью `gateway.bind`.
- **Прямой транспорт WS**: конечная точка Gateway WS, доступная в локальной сети/tailnet (без SSH).
- **Транспорт SSH (резервный вариант)**: удаленное управление посредством перенаправления
  `127.0.0.1:18789` через SSH.
- **Устаревший мост TCP (удален)**: прежний транспорт Node (см.
  [Протокол моста](/ru/gateway/bridge-protocol)); он больше не анонсируется для
  обнаружения и не входит в текущие сборки.

Сведения о протоколах: [Протокол Gateway](/ru/gateway/protocol),
[Протокол моста (устаревший)](/ru/gateway/bridge-protocol).

## Зачем нужны и прямое подключение, и SSH

- **Прямой WS** обеспечивает наиболее удобную работу в одной сети и внутри tailnet: автоматическое
  обнаружение в локальной сети через Bonjour, токены сопряжения и ACL под управлением Gateway,
  а также отсутствие необходимости в доступе к командной оболочке.
- **SSH** — универсальный резервный вариант: работает везде, где есть доступ по SSH, даже
  между несвязанными сетями, не зависит от проблем с многоадресной рассылкой/mDNS и не требует
  нового входящего порта, кроме SSH.

## Источники обнаружения

### 1) Bonjour / DNS-SD

Многоадресное обнаружение Bonjour работает по мере возможности и не выходит за пределы сети. OpenClaw также
поддерживает поиск того же маяка Gateway через настроенный домен глобального DNS-SD,
поэтому обнаружение может охватывать как `local.` в той же локальной сети, так и настроенный
одноадресный домен DNS-SD для обнаружения между сетями.

**Gateway** анонсирует свою конечную точку WS через Bonjour, когда включен встроенный
Plugin `bonjour`; клиенты выполняют поиск и отображают список «выберите Gateway»,
а затем сохраняют выбранную конечную точку.

Устранение неполадок и сведения о маяке: [Bonjour](/ru/gateway/bonjour).

#### Сведения о служебном маяке

- Тип службы: `_openclaw-gw._tcp` (маяк транспорта Gateway).
- Ключи TXT (несекретные):

  | Ключ                        | Примечания                                                                                                                                                                                       |
  | --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
  | `role=gateway`              | Присутствует всегда.                                                                                                                                                                             |
  | `transport=gateway`         | Присутствует всегда.                                                                                                                                                                             |
  | `displayName=<name>`        | Отображаемое имя, настроенное оператором.                                                                                                                                                         |
  | `lanHost=<hostname>.local`  | Только для анонсирования mDNS в локальной сети; не записывается глобальным DNS-SD.                                                                                                                |
  | `gatewayPort=18789`         | Порт Gateway WS и HTTP.                                                                                                                                                                          |
  | `gatewayTls=1`              | Только при включенном TLS.                                                                                                                                                                       |
  | `gatewayTlsSha256=<sha256>` | Только при включенном TLS и наличии отпечатка.                                                                                                                                                    |
  | `tailnetDns=<magicdns>`     | Необязательная подсказка; определяется автоматически, когда доступен Tailscale.                                                                                                                   |
  | `sshPort=<port>`            | Присутствует только при `discovery.mdns.mode="full"`; в режиме по умолчанию `"minimal"` отсутствует (порт SSH по умолчанию — `22`) как в маяке локальной сети, так и в глобальном DNS-SD.           |
  | `cliPath=<path>`            | Подчиняется тому же условию `discovery.mdns.mode="full"`, что и `sshPort`; подсказка пути к CLI для удаленной установки.                                                                           |

  Ключ TXT `canvasPort` определен в контракте обнаружения Plugin для будущего
  порта хоста холста, но текущие пути выполнения не задают значение, поэтому сейчас
  он никогда не передается.

Примечания по безопасности:

- TXT-записи Bonjour/mDNS **не аутентифицированы**. Клиенты должны рассматривать значения TXT
  только как подсказки для интерфейса.
- При маршрутизации (хост/порт) следует отдавать предпочтение **разрешенной конечной точке службы**
  (SRV + A/AAAA), а не значениям `lanHost`, `tailnetDns` или `gatewayPort` из TXT.
- При закреплении TLS нельзя допускать, чтобы анонсированный `gatewayTlsSha256` заменял
  ранее сохранленный отпечаток.
- Перед сохранением отпечатка при первом подключении Node iOS/Android должны требовать явного
  подтверждения «доверять этому отпечатку» (проверка по внешнему каналу),
  если выбранный маршрут является защищенным или использует TLS.

Включение, отключение и переопределение:

- `openclaw plugins enable bonjour` включает многоадресное анонсирование в локальной сети.
- `discovery.mdns.mode` в `openclaw.json` управляет вещанием mDNS:
  `"minimal"` (по умолчанию), `"full"` (добавляет `cliPath`/`sshPort` как в маяк локальной
  сети, так и в любую глобальную зону DNS-SD) или `"off"` (отключает mDNS).
- `OPENCLAW_DISABLE_BONJOUR=1` принудительно отключает анонсирование; `discovery.mdns.mode="off"`
  отключает его независимо. `OPENCLAW_DISABLE_BONJOUR=0` явно включает его,
  переопределяя автоматическое отключение Plugin внутри обнаруженного контейнера
  (Docker, containerd, Kubernetes, LXC); это не переопределяет
  `discovery.mdns.mode="off"`. Встроенный Plugin `bonjour` автоматически запускается на
  хостах macOS (`enabledByDefaultOnPlatforms: ["darwin"]`) и автоматически отключается
  внутри обнаруженных контейнеров; в Linux, Windows и других контейнерных
  развертываниях требуется явно выполнить `plugins enable bonjour`.
- `gateway.bind` в `~/.openclaw/openclaw.json` управляет режимом привязки Gateway.
- `OPENCLAW_SSH_PORT` переопределяет анонсируемый порт SSH (действует только
  при `discovery.mdns.mode="full"`).
- `OPENCLAW_TAILNET_DNS` публикует подсказку `tailnetDns` (MagicDNS).
- `OPENCLAW_CLI_PATH` переопределяет анонсируемый путь к CLI.

### 2) Tailnet (между сетями)

Для Gateway в разных физических сетях Bonjour не поможет.
Рекомендуемая цель прямого подключения — имя Tailscale MagicDNS (предпочтительно) или
стабильный IP-адрес tailnet.

Если Gateway обнаруживает, что работает под управлением Tailscale, он публикует
`tailnetDns` как необязательную подсказку для клиентов (в том числе в глобальных маяках).
Для обнаружения Gateway приложение macOS предпочитает имена MagicDNS необработанным
IP-адресам Tailscale, что обеспечивает надежность при изменении IP-адресов tailnet
(перезапуске Node, переназначении CGNAT), поскольку MagicDNS автоматически разрешается
в текущий IP-адрес.

При сопряжении мобильного Node подсказки обнаружения никогда не ослабляют безопасность транспорта
на маршрутах tailnet/публичной сети:

- Для iOS/Android по-прежнему требуется безопасный маршрут первого подключения через tailnet/публичную сеть
  (`wss://` или Tailscale Serve/Funnel).
- Обнаруженный необработанный IP-адрес tailnet — это подсказка для маршрутизации, а не разрешение
  использовать незашифрованный удаленный `ws://`.
- Прямое подключение по `ws://` в частной локальной сети остается поддерживаемым.
- Для самого простого подключения мобильных Node через Tailscale используйте Tailscale Serve, чтобы
  и обнаружение, и настройка использовали одну и ту же защищенную конечную точку MagicDNS.

### 3) Цель вручную / через SSH

Если прямой маршрут отсутствует (или прямое подключение отключено), клиенты всегда могут
подключиться через SSH, перенаправив порт Gateway на local loopback. См.
[Удаленный доступ](/ru/gateway/remote).

## Выбор транспорта (политика клиента)

1. Если настроена и доступна сопряженная конечная точка прямого подключения, используйте ее.
2. В противном случае, если при обнаружении найден Gateway в `local.` или настроенном глобальном
   домене, предложите одним нажатием выбрать «Использовать этот Gateway» и сохраните его как
   конечную точку прямого подключения.
3. В противном случае, если настроен DNS/IP tailnet, попробуйте прямое подключение. Для мобильных Node на
   маршрутах tailnet/публичной сети прямое подключение означает защищенную конечную точку, а не незашифрованный
   удаленный `ws://`.
4. В противном случае используйте SSH как резервный вариант.

## Сопряжение и аутентификация (прямой транспорт)

Gateway является источником истины для допуска Node/клиентов:

- Запросы на сопряжение создаются, одобряются или отклоняются в Gateway (см.
  [Сопряжение Gateway](/ru/gateway/pairing)).
- Gateway обеспечивает аутентификацию (токен/пара ключей), области доступа/ACL (это не обычный
  прокси ко всем методам) и ограничения частоты запросов.

## Ответственность компонентов

- **Gateway**: анонсирует маяки обнаружения, управляет решениями о сопряжении и размещает
  конечную точку WS.
- **Приложение macOS**: помогает выбрать Gateway, отображает запросы на сопряжение и использует SSH
  только как резервный вариант.
- **Node iOS/Android**: для удобства выполняют поиск через Bonjour и подключаются к
  сопряженному Gateway WS.
