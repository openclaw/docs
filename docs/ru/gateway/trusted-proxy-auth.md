<Warning>
**Функция, критичная для безопасности.** В этом режиме аутентификация полностью делегируется обратному прокси. Ошибка конфигурации может открыть несанкционированный доступ к вашему Gateway. Внимательно прочитайте эту страницу перед включением.
</Warning>

## Когда использовать

- OpenClaw работает за **прокси с поддержкой идентификации** (Pomerium, Caddy + OAuth, nginx + oauth2-proxy, Traefik + forward auth).
- Прокси выполняет всю аутентификацию и передаёт идентификатор пользователя через заголовки.
- Вы используете Kubernetes или контейнерную среду, где прокси — единственный путь к Gateway.
- Вы сталкиваетесь с ошибками WebSocket `1008 unauthorized`, поскольку браузеры не могут передавать токены в полезной нагрузке WS.

## Когда НЕ использовать

- Ваш прокси не аутентифицирует пользователей, а только завершает TLS или балансирует нагрузку.
- Существует любой путь к Gateway в обход прокси (открытые порты межсетевого экрана, доступ из внутренней сети).
- Вы не уверены, что прокси правильно удаляет или перезаписывает перенаправляемые заголовки.
- Вам нужен только персональный однопользовательский доступ (вместо этого рассмотрите Tailscale Serve + loopback).

## Как это работает

<Steps>
  <Step title="Прокси аутентифицирует пользователя">
    Обратный прокси аутентифицирует пользователей (OAuth, OIDC, SAML и т. д.).
  </Step>
  <Step title="Прокси добавляет заголовок идентификации">
    Прокси добавляет заголовок с идентификатором аутентифицированного пользователя (например, `x-forwarded-user: nick@example.com`).
  </Step>
  <Step title="Gateway проверяет доверенный источник">
    OpenClaw проверяет, что запрос поступил с **IP-адреса доверенного прокси** (`gateway.trustedProxies`), а не с собственного loopback-адреса Gateway или адреса его локального интерфейса.
  </Step>
  <Step title="Gateway извлекает идентификатор">
    OpenClaw считывает обязательные заголовки, а затем идентификатор пользователя из настроенного заголовка.
  </Step>
  <Step title="Авторизация">
    Если все проверки пройдены и пользователь соответствует `allowUsers` (если параметр задан), запрос авторизуется.
  </Step>
</Steps>

## Конфигурация

```json5
{
  gateway: {
    // Аутентификация через доверенный прокси по умолчанию ожидает, что исходный IP-адрес прокси не является loopback-адресом
    bind: "lan",

    // КРИТИЧЕСКИ ВАЖНО: добавляйте сюда только IP-адреса вашего прокси
    trustedProxies: ["10.0.0.1", "172.17.0.1"],

    auth: {
      mode: "trusted-proxy",
      trustedProxy: {
        // Заголовок, содержащий идентификатор аутентифицированного пользователя (обязательно)
        userHeader: "x-forwarded-user",

        // Необязательно: заголовки, которые ОБЯЗАТЕЛЬНО должны присутствовать (проверка прокси)
        requiredHeaders: ["x-forwarded-proto", "x-forwarded-host"],

        // Необязательно: ограничить доступ указанными пользователями (пустой список = разрешить всем)
        allowUsers: ["nick@example.com", "admin@company.org"],

        // Необязательно: разрешить loopback-прокси на том же узле после явного согласия
        allowLoopback: false,
      },
    },
  },
}
```

<Warning>
**Правила среды выполнения в порядке проверки**

1. Исходный IP-адрес запроса должен соответствовать `gateway.trustedProxies` с учётом CIDR, иначе запрос отклоняется (`trusted_proxy_untrusted_source`).
2. Запросы из loopback (`127.0.0.1`, `::1`) отклоняются, если только не задано `gateway.auth.trustedProxy.allowLoopback = true` и loopback-адрес также не включён в `trustedProxies` (`trusted_proxy_loopback_source`). Эта проверка выполняется до проверки заголовков, поэтому запрос из loopback отклоняется именно по этой причине, даже если обязательные заголовки также отсутствуют.
3. Источники не из loopback, соответствующие одному из адресов собственных локальных сетевых интерфейсов узла Gateway, отклоняются для защиты от подмены (`trusted_proxy_local_interface_source`). Если определить интерфейсы не удаётся, запрос также отклоняется (`trusted_proxy_local_interface_check_failed`).
4. `requiredHeaders` и `userHeader` должны присутствовать и содержать непустые значения.
5. Если `allowUsers` не пуст, извлечённый пользователь должен входить в этот список.

**Наличие перенаправляемых заголовков отменяет локальность loopback для локального прямого резервного доступа.** Если запрос поступает через loopback, но содержит заголовок `Forwarded`, любой заголовок `X-Forwarded-*` или `X-Real-IP`, это исключает возможность локального прямого резервного входа по паролю и проверки идентификатора устройства, хотя аутентификация через доверенный прокси всё равно завершается ошибкой из-за источника loopback.

`allowLoopback` предоставляет локальным процессам на узле Gateway тот же уровень доверия, что и обратному прокси. Включайте этот параметр только в том случае, если Gateway по-прежнему защищён межсетевым экраном от прямого удалённого доступа, а локальный прокси удаляет или перезаписывает предоставленные клиентом заголовки идентификации.

Внутренним клиентам Gateway, трафик которых не проходит через обратный прокси, следует использовать `gateway.auth.password` / `OPENCLAW_GATEWAY_PASSWORD`, а не заголовки идентификации доверенного прокси. Для развёртываний Control UI не через loopback по-прежнему требуется явно задать `gateway.controlUi.allowedOrigins`.
</Warning>

### Справочник по конфигурации

<ParamField path="gateway.trustedProxies" type="string[]" required>
  Массив доверенных IP-адресов прокси или диапазонов CIDR. Запросы с других IP-адресов отклоняются.
</ParamField>
<ParamField path="gateway.auth.mode" type="string" required>
  Должно быть задано значение `"trusted-proxy"`.
</ParamField>
<ParamField path="gateway.auth.trustedProxy.userHeader" type="string" required>
  Имя заголовка, содержащего идентификатор аутентифицированного пользователя.
</ParamField>
<ParamField path="gateway.auth.trustedProxy.requiredHeaders" type="string[]">
  Дополнительные заголовки, которые должны присутствовать, чтобы запрос считался доверенным.
</ParamField>
<ParamField path="gateway.auth.trustedProxy.allowUsers" type="string[]">
  Список разрешённых идентификаторов пользователей. Пустой список разрешает доступ всем аутентифицированным пользователям.
</ParamField>
<ParamField path="gateway.auth.trustedProxy.allowLoopback" type="boolean" default="false">
  Явно включаемая поддержка обратных прокси на том же узле, использующих loopback.
</ParamField>

<Warning>
Включайте `allowLoopback` только в том случае, если локальный обратный прокси является предполагаемой границей доверия. Любой локальный процесс, способный подключиться к Gateway, может попытаться отправить заголовки идентификации прокси, поэтому разрешайте прямой доступ к Gateway только с самого узла и требуйте заголовки, добавляемые прокси, например `x-forwarded-proto`, либо подписанный заголовок утверждения, если ваш прокси его поддерживает.
</Warning>

## Поведение сопряжения Control UI

Когда активен режим `gateway.auth.mode = "trusted-proxy"` и запрос проходит проверки доверенного прокси, сеансы WebSocket Control UI могут подключаться без идентификатора сопряжённого устройства.

Последствия для областей разрешений:

- Сеансы WebSocket Control UI без устройства подключаются, но по умолчанию не получают областей разрешений оператора. OpenClaw очищает список запрошенных областей до `[]`, чтобы сеанс, не привязанный к одобренному сопряжённому устройству или токену, не мог самостоятельно объявить разрешения.
- Если после успешного подключения WebSocket методы завершаются ошибкой `missing scope`, используйте HTTPS, чтобы браузер мог создать идентификатор устройства и завершить сопряжение. См. [небезопасный HTTP для Control UI](/ru/web/control-ui#insecure-http).
- Только для аварийного доступа: `gateway.controlUi.dangerouslyDisableDeviceAuth=true` сохраняет запрошенные области разрешений даже без идентификатора устройства. Это серьёзное снижение уровня безопасности; как можно скорее отмените эту настройку. См. [небезопасный HTTP для Control UI](/ru/web/control-ui#insecure-http).

Ограничение областей разрешений обратным прокси: если прокси передаёт `x-openclaw-scopes` в запросе на переключение протокола WebSocket для Control UI, OpenClaw ограничивает области разрешений сеанса пересечением запрошенных и объявленных областей. Этот заголовок не предоставляет области разрешений, а только сужает их набор для сеанса.

Последствия:

- В этом режиме сопряжение больше не является основным средством контроля доступа к Control UI.
- Политика аутентификации обратного прокси и `allowUsers` становятся фактическими средствами контроля доступа.
- Разрешайте входящий трафик Gateway только с IP-адресов доверенных прокси (`gateway.trustedProxies` + межсетевой экран).

Пользовательские клиенты WebSocket не являются сеансами Control UI. `gateway.controlUi.dangerouslyDisableDeviceAuth` не предоставляет области разрешений произвольным клиентам с `client.mode: "backend"` или клиентам, оформленным как CLI. Пользовательской автоматизации следует использовать идентификатор устройства и сопряжение, зарезервированный внутренний путь помощника для прямого локального доступа с `client.id: "gateway-client"` либо [Plugin административного HTTP RPC](/ru/plugins/admin-http-rpc), если интерфейс HTTP «запрос — ответ» подходит лучше.

## Заголовок областей разрешений оператора

Аутентификация через доверенный прокси — это HTTP-режим, **несущий идентификационные данные**, поэтому вызывающие стороны могут при необходимости объявлять области разрешений оператора с помощью `x-openclaw-scopes` в запросах HTTP API.

Примечание: области разрешений WebSocket определяются рукопожатием протокола Gateway и привязкой идентификатора устройства. В запросах на переключение протокола WebSocket для Control UI `x-openclaw-scopes` только ограничивает согласованные области разрешений сеанса, но не предоставляет их. См. [поведение сопряжения Control UI](#control-ui-pairing-behavior).

Примеры:

- `x-openclaw-scopes: operator.read`
- `x-openclaw-scopes: operator.read,operator.write`
- `x-openclaw-scopes: operator.admin,operator.write`

Поведение:

- Если заголовок присутствует, OpenClaw учитывает объявленный набор областей разрешений.
- Если заголовок присутствует, но пуст, запрос объявляет отсутствие областей разрешений оператора.
- Если заголовок отсутствует, обычные HTTP API, несущие идентификационные данные, используют стандартный набор областей разрешений оператора по умолчанию (`operator.admin`, `operator.read`, `operator.write`, `operator.approvals`, `operator.pairing`, `operator.talk.secrets`).
- **HTTP-маршруты Plugin**, использующие аутентификацию Gateway, по умолчанию имеют более узкие права: если `x-openclaw-scopes` отсутствует, их область разрешений среды выполнения ограничивается только `operator.write`.
- HTTP-запросы из браузера даже после успешной аутентификации через доверенный прокси должны пройти проверку `gateway.controlUi.allowedOrigins` либо преднамеренно включённого резервного режима на основе заголовка Host.

Практическое правило: передавайте `x-openclaw-scopes` явно, если хотите ограничить запрос через доверенный прокси сильнее, чем позволяют значения по умолчанию, либо если маршруту Plugin с аутентификацией Gateway нужны права выше области записи.

## Завершение TLS и HSTS

Используйте одну точку завершения TLS и применяйте HSTS в ней.

<Tabs>
  <Tab title="Завершение TLS на прокси (рекомендуется)">
    Если обратный прокси обслуживает HTTPS для `https://control.example.com`, задайте `Strict-Transport-Security` на прокси для этого домена.

    - Хорошо подходит для развёртываний с доступом из интернета.
    - Хранит сертификат и политику усиления безопасности HTTP в одном месте.
    - OpenClaw может продолжать работать по HTTP через loopback за прокси.

    Пример значения заголовка:

    ```text
    Strict-Transport-Security: max-age=31536000; includeSubDomains
    ```

  </Tab>
  <Tab title="Завершение TLS на Gateway">
    Если OpenClaw самостоятельно обслуживает HTTPS напрямую, без прокси, завершающего TLS, задайте:

    ```json5
    {
      gateway: {
        tls: { enabled: true },
        http: {
          securityHeaders: {
            strictTransportSecurity: "max-age=31536000; includeSubDomains",
          },
        },
      },
    }
    ```

    `strictTransportSecurity` принимает строковое значение заголовка либо `false` для явного отключения.

  </Tab>
</Tabs>

### Рекомендации по внедрению

- Сначала задайте небольшой срок действия, например `max-age=300`, пока проверяете трафик.
- Увеличивайте срок до длительных значений, например `max-age=31536000`, только после достижения высокой уверенности.
- Добавляйте `includeSubDomains`, только если каждый поддомен готов к работе через HTTPS.
- Используйте предварительную загрузку только в том случае, если намеренно выполняете её требования для всего набора доменов.
- Локальная разработка только через loopback не получает преимуществ от HSTS.

## Примеры настройки прокси

<AccordionGroup>
  <Accordion title="Pomerium">
    Pomerium передаёт идентификатор в `x-pomerium-claim-email` или других заголовках утверждений, а JWT — в `x-pomerium-jwt-assertion`.

    ```json5
    {
      gateway: {
        bind: "lan",
        trustedProxies: ["10.0.0.1"], // IP-адрес Pomerium
        auth: {
          mode: "trusted-proxy",
          trustedProxy: {
            userHeader: "x-pomerium-claim-email",
            requiredHeaders: ["x-pomerium-jwt-assertion"],
          },
        },
      },
    }
    ```

    Фрагмент конфигурации Pomerium:

    ```yaml
    routes:
      - from: https://openclaw.example.com
        to: http://openclaw-gateway:18789
        policy:
          - allow:
              or:
                - email:
                    is: nick@example.com
        pass_identity_headers: true
    ```

  </Accordion>
  <Accordion title="Caddy с OAuth">
    Caddy с Plugin `caddy-security` может аутентифицировать пользователей и передавать заголовки идентификации.

    ```json5
    {
      gateway: {
        bind: "lan",
        trustedProxies: ["10.0.0.1"], // IP-адрес прокси Caddy/sidecar
        auth: {
          mode: "trusted-proxy",
          trustedProxy: {
            userHeader: "x-forwarded-user",
          },
        },
      },
    }
    ```

    Фрагмент Caddyfile:

    ```caddy
    openclaw.example.com {
        authenticate with oauth2_provider
        authorize with policy1

        reverse_proxy openclaw:18789 {
            header_up X-Forwarded-User {http.auth.user.email}
        }
    }
    ```

  </Accordion>
  <Accordion title="nginx + oauth2-proxy">
    oauth2-proxy аутентифицирует пользователей и передаёт идентификатор в `x-auth-request-email`.

    ```json5
    {
      gateway: {
        bind: "lan",
        trustedProxies: ["10.0.0.1"], // IP-адрес nginx/oauth2-proxy
        auth: {
          mode: "trusted-proxy",
          trustedProxy: {
            userHeader: "x-auth-request-email",
          },
        },
      },
    }
    ```

    Фрагмент конфигурации nginx:

    ```nginx
    location / {
        auth_request /oauth2/auth;
        auth_request_set $user $upstream_http_x_auth_request_email;

        proxy_pass http://openclaw:18789;
        proxy_set_header X-Auth-Request-Email $user;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
    ```

## Конфигурация со смешанным использованием токенов

Gateway не запускается с аутентификацией через доверенный прокси, если одновременно настроен общий токен (`gateway.auth.token` или `OPENCLAW_GATEWAY_TOKEN`). Эти варианты взаимоисключающие, поскольку общий токен позволил бы вызывающим процессам на том же хосте проходить аутентификацию совершенно иным способом, в обход проверенного прокси идентификатора, который должен обеспечиваться этим режимом.

Если при запуске возникает ошибка наподобие `gateway auth mode is trusted-proxy, but a shared token is also configured`:

- Удалите общий токен при использовании режима доверенного прокси или
- Измените `gateway.auth.mode` на `"token"`, если планируете использовать аутентификацию по токену.

Заголовки идентификации доверенного прокси из local loopback по-прежнему отклоняются по принципу безопасного отказа: вызывающие процессы на том же хосте не аутентифицируются неявно как пользователи прокси. Внутренние вызывающие процессы OpenClaw, обходящие прокси, могут вместо этого проходить аутентификацию с помощью `gateway.auth.password` / `OPENCLAW_GATEWAY_PASSWORD`. Резервная аутентификация по токену намеренно не поддерживается в режиме доверенного прокси.

## Контрольный список безопасности

Перед включением аутентификации через доверенный прокси убедитесь, что:

- [ ] **Прокси — единственный путь**: порт Gateway закрыт межсетевым экраном для всех, кроме вашего прокси.
- [ ] **Список trustedProxies минимален**: указаны только фактические IP-адреса ваших прокси, а не целые подсети.
- [ ] **Источник прокси через local loopback выбран осознанно**: аутентификация через доверенный прокси отклоняет по принципу безопасного отказа запросы, поступающие через local loopback, если `gateway.auth.trustedProxy.allowLoopback` не включён явно для прокси на том же хосте.
- [ ] **Прокси удаляет заголовки**: ваш прокси перезаписывает, а не дополняет заголовки `x-forwarded-*`, полученные от клиентов.
- [ ] **Завершение TLS**: ваш прокси обрабатывает TLS; пользователи подключаются по HTTPS.
- [ ] **allowedOrigins задан явно**: для Control UI не через local loopback явно настроен `gateway.controlUi.allowedOrigins`.
- [ ] **allowUsers задан** (рекомендуется): разрешён доступ только известным пользователям, а не всем прошедшим аутентификацию.
- [ ] **Нет смешанной конфигурации токенов**: не задавайте одновременно `gateway.auth.token` и `gateway.auth.mode: "trusted-proxy"`.
- [ ] **Локальная резервная аутентификация по паролю недоступна извне**: если для внутренних прямых вызывающих процессов настроен `gateway.auth.password`, закройте порт Gateway межсетевым экраном, чтобы удалённые клиенты, не использующие прокси, не могли подключиться к нему напрямую.

## Аудит безопасности

`openclaw security audit` помечает аутентификацию через доверенный прокси как проблему **критического** уровня серьёзности. Это сделано намеренно: предупреждение напоминает, что безопасность делегирована конфигурации вашего прокси.

Аудит проверяет:

- Базовое предупреждение или критическое напоминание `gateway.trusted_proxy_auth`.
- Отсутствие конфигурации `trustedProxies`.
- Отсутствие конфигурации `userHeader`.
- Пустой список `allowUsers` (разрешает доступ любому аутентифицированному пользователю).
- Включённый параметр `allowLoopback` для источников прокси на том же хосте.

Отдельные проверки, не относящиеся исключительно к доверенному прокси, также применяются при открытом доступе к Control UI: отсутствующий или содержащий подстановочный знак `gateway.controlUi.allowedOrigins`, а также резервное определение источника по заголовку Host.

## Устранение неполадок

<AccordionGroup>
  <Accordion title="trusted_proxy_untrusted_source">
    Запрос поступил с IP-адреса, отсутствующего в `gateway.trustedProxies`. Проверьте:

    - Правильно ли указан IP-адрес прокси? (IP-адреса контейнеров Docker могут изменяться.)
    - Есть ли перед вашим прокси балансировщик нагрузки?
    - Используйте `docker inspect` или `kubectl get pods -o wide`, чтобы определить фактические IP-адреса.

  </Accordion>
  <Accordion title="trusted_proxy_loopback_source">
    OpenClaw отклонил запрос доверенного прокси, поступивший через local loopback.

    Проверьте:

    - Подключается ли прокси с адреса `127.0.0.1` / `::1`?
    - Пытаетесь ли вы использовать аутентификацию через доверенный прокси с обратным прокси на том же хосте, работающим через local loopback?

    Исправление:

    - Для внутренних клиентов на том же хосте, которые не проходят через прокси, предпочтительно используйте аутентификацию по токену или паролю либо
    - Направьте трафик через адрес доверенного прокси, не относящийся к local loopback, и оставьте этот IP-адрес в `gateway.trustedProxies` либо
    - Для намеренно настроенного обратного прокси на том же хосте задайте `gateway.auth.trustedProxy.allowLoopback = true`, оставьте адрес local loopback в `gateway.trustedProxies` и убедитесь, что прокси удаляет или перезаписывает заголовки идентификации.

  </Accordion>
  <Accordion title="trusted_proxy_local_interface_source / trusted_proxy_local_interface_check_failed">
    IP-адрес источника запроса совпал с одним из собственных адресов сетевых интерфейсов хоста Gateway, не относящихся к local loopback (а не с адресом прокси). Эта защита предотвращает подделку трафика с того же хоста в tailnet или мостовых сетях Docker. `..._check_failed` означает, что при обнаружении интерфейсов произошла ошибка, поэтому OpenClaw применяет принцип безопасного отказа.

    Проверьте:

    - Отправляет ли процесс на самом хосте Gateway заголовки идентификации напрямую, обходя прокси?
    - Работает ли прокси в том же сетевом пространстве имён, что и Gateway, с IP-адресом, который также отображается как локальный интерфейс?

    Исправление: направьте трафик прокси через адрес, который не привязан локально к хосту Gateway, либо используйте `allowLoopback` только для настоящей конфигурации прокси на том же хосте.

  </Accordion>
  <Accordion title="trusted_proxy_user_missing">
    Заголовок пользователя отсутствовал или был пуст. Проверьте:

    - Настроен ли ваш прокси на передачу заголовков идентификации?
    - Правильно ли указано имя заголовка? (регистр не учитывается, но написание имеет значение)
    - Действительно ли пользователь прошёл аутентификацию на прокси?

  </Accordion>
  <Accordion title="trusted_proxy_missing_header_*">
    Обязательный заголовок отсутствовал. Проверьте:

    - Конфигурацию прокси для этих конкретных заголовков.
    - Не удаляются ли заголовки где-либо в цепочке.

  </Accordion>
  <Accordion title="trusted_proxy_user_not_allowed">
    Пользователь аутентифицирован, но отсутствует в `allowUsers`. Добавьте его или удалите список разрешённых пользователей.
  </Accordion>
  <Accordion title="trusted_proxy_no_proxies_configured / trusted_proxy_config_missing">
    Для `gateway.auth.mode` задано значение `"trusted-proxy"`, но `gateway.trustedProxies` пуст либо отсутствует сам параметр `gateway.auth.trustedProxy`. Все запросы будут отклоняться, пока не будут заданы оба параметра.
  </Accordion>
  <Accordion title="trusted_proxy_origin_not_allowed">
    Аутентификация через доверенный прокси прошла успешно, но заголовок браузера `Origin` не прошёл проверку источника Control UI.

    Проверьте:

    - `gateway.controlUi.allowedOrigins` содержит точный источник браузера.
    - Вы не используете источники с подстановочными знаками, если только намеренно не хотите разрешить все источники.
    - Если намеренно используется резервный режим на основе заголовка Host, параметр `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true` задан осознанно.

  </Accordion>
  <Accordion title="Соединение устанавливается, но методы сообщают об отсутствующей области доступа">
    Соединение WebSocket устанавливается, но вызовы `chat.history`, `sessions.list` или
    `models.list` завершаются ошибкой `missing scope: operator.read`.

    Распространённые причины:

    - Сеанс Control UI без устройства: аутентификация через доверенный прокси может разрешить соединение WebSocket без идентификатора устройства, но OpenClaw намеренно очищает области доступа для сеансов без устройства.
    - Пользовательский серверный клиент: `gateway.controlUi.dangerouslyDisableDeviceAuth` действует только в рамках Control UI и не предоставляет области доступа произвольным серверным клиентам или WebSocket-клиентам в формате CLI.
    - Чрезмерно ограниченный `x-openclaw-scopes`: если прокси добавляет этот заголовок в запрос на повышение соединения WebSocket для Control UI, области доступа сеанса ограничиваются указанным набором. Пустое значение заголовка означает отсутствие областей доступа.

    Исправление:

    - Для Control UI используйте HTTPS, чтобы браузер мог создать идентификатор устройства и завершить сопряжение.
    - Для пользовательской автоматизации используйте идентификатор устройства и сопряжение, зарезервированный для прямых локальных подключений вспомогательный серверный путь `gateway-client` или [административный HTTP RPC](/ru/plugins/admin-http-rpc).
    - Используйте `gateway.controlUi.dangerouslyDisableDeviceAuth: true` только как временный аварийный обходной путь для Control UI.

  </Accordion>
  <Accordion title="WebSocket по-прежнему не работает">
    Убедитесь, что ваш прокси:

    - Поддерживает повышение соединений WebSocket (`Upgrade: websocket`, `Connection: upgrade`).
    - Передаёт заголовки идентификации в запросах на повышение соединения WebSocket, а не только в HTTP-запросах.
    - Не использует отдельный путь аутентификации для соединений WebSocket.

  </Accordion>
</AccordionGroup>

## Переход с аутентификации по токену

<Steps>
  <Step title="Настройте прокси">
    Настройте прокси для аутентификации пользователей и передачи заголовков.
  </Step>
  <Step title="Проверьте прокси отдельно">
    Проверьте конфигурацию прокси отдельно (`curl` с заголовками).
  </Step>
  <Step title="Обновите конфигурацию OpenClaw">
    Добавьте в конфигурацию OpenClaw аутентификацию через доверенный прокси.
  </Step>
  <Step title="Перезапустите Gateway">
    Перезапустите Gateway.
  </Step>
  <Step title="Проверьте WebSocket">
    Проверьте соединения WebSocket из Control UI.
  </Step>
  <Step title="Выполните аудит">
    Выполните `openclaw security audit` и изучите результаты.
  </Step>
</Steps>
