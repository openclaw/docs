<Note>
На этой странице описана аутентификация у **провайдера моделей** (API-ключи, OAuth, повторное использование Claude CLI, токен настройки Anthropic). Об аутентификации для **подключения к Gateway** (токен, пароль, доверенный прокси) см. в разделах [Конфигурация](/ru/gateway/configuration) и [Аутентификация через доверенный прокси](/ru/gateway/trusted-proxy-auth).
</Note>

OpenClaw поддерживает OAuth и API-ключи провайдеров моделей. Для постоянно работающего хоста Gateway наиболее предсказуемым вариантом является API-ключ; подписка и OAuth также подходят, если они соответствуют модели учетной записи вашего провайдера.

- Полный процесс OAuth и структура хранения: [/concepts/oauth](/ru/concepts/oauth)
- Аутентификация на основе SecretRef (провайдеры `env`/`file`/`exec`): [Управление секретами](/ru/gateway/secrets)
- Коды пригодности и причин для учетных данных, используемые командой `models status --probe`: [Семантика учетных данных аутентификации](/ru/auth-credential-semantics)

## Рекомендуемая настройка: API-ключ (любой провайдер)

1. Создайте API-ключ в консоли провайдера.
2. Разместите его на **хосте Gateway** (компьютере, на котором выполняется `openclaw gateway`):

```bash
export <PROVIDER>_API_KEY="..."
openclaw models status
```

3. Если Gateway работает под управлением systemd/launchd, поместите ключ в `~/.openclaw/.env`, чтобы демон мог его прочитать:

```bash
cat >> ~/.openclaw/.env <<'EOF'
<PROVIDER>_API_KEY=...
EOF
```

4. Перезапустите процесс Gateway (или демон), затем повторите проверку:

```bash
openclaw models status
openclaw doctor
```

`openclaw onboard` также может сохранять API-ключи для использования демоном, если вы не хотите самостоятельно управлять переменными окружения. Полный порядок загрузки переменных окружения (`env.shellEnv`, `~/.openclaw/.env`, systemd/launchd) см. в разделе [Переменные окружения](/ru/help/environment).

## Anthropic: повторное использование Claude CLI

Аутентификация с помощью токена настройки Anthropic остается поддерживаемым вариантом. Повторное использование Claude CLI (вызовы в стиле `claude -p`) также официально поддерживается этой интеграцией; если на хосте доступен вход через Claude CLI, этот вариант предпочтителен для локального или настольного использования. Для долговременно работающих хостов Gateway наиболее предсказуемым выбором по-прежнему остается API-ключ Anthropic с явным серверным контролем оплаты.

Настройка повторного использования Claude CLI на хосте:

```bash
# Выполните на хосте Gateway
claude auth login
claude auth status --text
openclaw models auth login --provider anthropic --method cli --set-default
```

Процесс состоит из двух шагов: войдите в Anthropic через Claude Code на хосте, затем укажите OpenClaw направлять выбор моделей Anthropic через локальный бэкенд `claude-cli` и сохранять соответствующий профиль аутентификации OpenClaw.

Если `claude` отсутствует в `PATH`, установите Claude Code или задайте для `agents.defaults.cliBackends.claude-cli.command` путь к исполняемому файлу.

## Ввод токена вручную

Работает с любым провайдером; записывает данные в хранилище аутентификации SQLite для отдельного агента и обновляет конфигурацию:

```bash
openclaw models auth paste-token --provider openrouter
```

OpenClaw считывает профили аутентификации из файла `openclaw-agent.sqlite` каждого агента. Параметры конечной точки (`baseUrl`, `api`, идентификаторы моделей, заголовки, тайм-ауты) должны находиться в `models.providers.<id>` в `openclaw.json` или `models.json`, а не в профилях аутентификации.

Если в старой установке по-прежнему имеются `auth-profiles.json`, `auth-state.json` или плоская структура вида `{ "openrouter": { "apiKey": "..." } }`, выполните `openclaw doctor --fix`, чтобы импортировать данные в SQLite; doctor сохранит резервные копии с временными метками рядом с исходными JSON-файлами.

Внешние маршруты аутентификации, такие как `auth: "aws-sdk"` для Bedrock, не являются учетными данными. Для именованного маршрута Bedrock задайте `auth.profiles.<id>.mode: "aws-sdk"` в `openclaw.json` — не записывайте `type: "aws-sdk"` в хранилище профилей аутентификации. `openclaw doctor --fix` переносит устаревшие маркеры AWS SDK из хранилища учетных данных в метаданные конфигурации.

### Учетные данные на основе SecretRef

- Учетные данные `api_key` могут использовать `keyRef: { source, provider, id }`
- Учетные данные `token` могут использовать `tokenRef: { source, provider, id }`
- Профили в режиме OAuth отклоняют учетные данные SecretRef: если значение `auth.profiles.<id>.mode` равно `"oauth"`, привязанный к SecretRef параметр `keyRef`/`tokenRef` для этого профиля отклоняется.

## Проверка состояния аутентификации моделей

```bash
openclaw models status
openclaw doctor
```

Проверка для автоматизации: код завершения `1`, если срок действия истек или данные отсутствуют, и `2`, если срок действия скоро истечет:

```bash
openclaw models status --check
```

Интерактивные проверки аутентификации (добавьте `--probe-provider`, `--probe-profile`, `--probe-timeout`, `--probe-concurrency` или `--probe-max-tokens`, чтобы сузить область проверки):

```bash
openclaw models status --probe
```

Примечания:

- Строки результатов проверки могут формироваться из профилей аутентификации, учетных данных из переменных окружения или `models.json`.
- Если в `auth.order.<provider>` отсутствует сохраненный профиль, проверка сообщает для него `excluded_by_auth_order`, а не пытается его использовать.
- Если данные аутентификации существуют, но OpenClaw не удается определить доступную для проверки модель этого провайдера, проверка сообщает `status: no_model`.
- Периоды ожидания после ограничения частоты запросов могут применяться к отдельным моделям: профиль, временно недоступный для одной модели, по-прежнему может обслуживать другую модель того же провайдера.

Необязательные эксплуатационные скрипты (systemd/Termux): [Скрипты мониторинга аутентификации](/ru/help/scripts#auth-monitoring-scripts).

## Ротация API-ключей (Gateway)

Некоторые провайдеры повторяют запрос с другим настроенным ключом, если вызов сталкивается с ограничением частоты запросов провайдера.

Порядок приоритета ключей для каждого провайдера:

1. `OPENCLAW_LIVE_<PROVIDER>_KEY` (единственное переопределение, фиксирующее один ключ)
2. `<PROVIDER>_API_KEYS` (список, разделенный запятыми, пробелами или точками с запятой)
3. `<PROVIDER>_API_KEY`
4. `<PROVIDER>_API_KEY_*` (любая переменная окружения с этим префиксом)

Провайдеры Google (`google`, `google-vertex`) дополнительно используют `GOOGLE_API_KEY` в качестве резервного варианта. Перед использованием из объединенного списка удаляются дубликаты.

OpenClaw переходит к следующему ключу, только если сообщение об ошибке соответствует одному из значений: `rate_limit`, `rate limit`, `429`, `quota exceeded`/`quota_exceeded`, `resource exhausted`/`resource_exhausted` или `too many requests`. При других ошибках запросы с альтернативными ключами не повторяются. Если все ключи завершаются ошибкой, возвращается последняя ошибка последней попытки.

<Note>
Специфичные для провайдера фразы, такие как `ThrottlingException`, `concurrency limit reached` или `workers_ai ... quota limit exceeded`, определяют **классификацию переключения/повторной попытки** (переключение модели или провайдера при повторяющихся сбоях) — это отдельный механизм, не связанный с описанной выше ротацией API-ключей.
</Note>

Удаление сохраненных данных аутентификации не отзывает ключ у провайдера — если необходимо сделать его недействительным на стороне провайдера, смените или отзовите ключ в панели управления провайдера.

## Удаление аутентификации провайдера во время работы Gateway

При удалении аутентификации провайдера через плоскость управления Gateway OpenClaw удаляет сохраненные профили аутентификации этого провайдера и прерывает активные запуски чатов и агентов, выбранная модель которых принадлежит удаленному провайдеру. Прерванные запуски создают стандартные события отмены и жизненного цикла со значением `stopReason: "auth-revoked"`, поэтому подключенные клиенты могут показать, что запуск остановлен из-за удаления учетных данных.

## Управление выбором учетных данных

### OpenAI и устаревшие идентификаторы `openai-codex`

Профили OpenAI с API-ключами и профили OAuth ChatGPT/Codex используют канонический идентификатор провайдера `openai`. В новой конфигурации используйте идентификаторы профилей `openai:*` и `auth.order.openai`.

Если в старой конфигурации, идентификаторах профилей аутентификации или `auth.order.openai-codex` встречается `openai-codex`, считайте его устаревшими входными данными для миграции — не создавайте новые профили `openai-codex`. Выполните:

```bash
openclaw doctor --fix
openclaw models auth list --provider openai
```

Doctor преобразует устаревшие идентификаторы профилей `openai-codex:*` и записи `auth.order.openai-codex` в канонический маршрут `openai`. Сведения о маршрутизации моделей и среды выполнения, относящейся к OpenAI, см. в разделе [OpenAI](/ru/providers/openai).

### Во время входа (CLI)

```bash
openclaw models auth login --provider openai --profile-id openai:ritsuko
openclaw models auth login --provider openai --profile-id openai:lain
```

Параметр `--profile-id` позволяет разделить несколько входов OAuth для одного провайдера в рамках одного агента.

Параметр `--force` удаляет сохраненные профили аутентификации этого провайдера в каталоге выбранного агента, а затем повторно запускает тот же процесс аутентификации. Используйте его, если сохраненный профиль завис, истек или связан не с той учетной записью. Этот параметр не отзывает учетные данные у провайдера.

```bash
openclaw models auth login --provider anthropic --force
```

### Для отдельного сеанса (команда чата)

- `/model <alias-or-id>@<profileId>` фиксирует конкретные учетные данные провайдера для текущего сеанса (примеры идентификаторов профилей: `anthropic:default`, `anthropic:work`).
- `/model` (или `/model list`) отображает компактное средство выбора; `/model status` показывает полное представление (кандидаты и следующий профиль аутентификации, а также сведения о конечной точке провайдера, если она настроена).

Если вы изменили порядок аутентификации или закрепление профиля для уже запущенного чата, отправьте `/new` или `/reset`, чтобы начать новый сеанс — существующие сеансы сохраняют текущий выбор модели и профиля до сброса.

### Для отдельного агента (переопределение через CLI)

Переопределения порядка аутентификации хранятся в состоянии аутентификации SQLite этого агента:

```bash
openclaw models auth order get --provider anthropic
openclaw models auth order set --provider anthropic anthropic:default
openclaw models auth order clear --provider anthropic
```

Используйте `--agent <id>`, чтобы выбрать конкретного агента; не указывайте его, чтобы использовать настроенного агента по умолчанию. `openclaw models status --probe` показывает пропущенные сохраненные профили как `excluded_by_auth_order`, а не молча пропускает их.

## Устранение неполадок

### «Учетные данные не найдены»

Настройте API-ключ Anthropic на **хосте Gateway** или настройте вариант с токеном настройки Anthropic, затем повторите проверку:

```bash
openclaw models status
```

### Срок действия токена скоро истечет или уже истек

Выполните `openclaw models status`, чтобы узнать, срок действия какого профиля истекает. Если профиль токена Anthropic отсутствует или срок его действия истек, обновите его с помощью токена настройки или перейдите на API-ключ Anthropic.
