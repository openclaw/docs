Сопряжение Node состоит из двух уровней, оба хранятся в записи сопряжённого устройства в базе данных состояния SQLite Gateway:

- **Сопряжение устройства** (роль `node`) управляет доступом к рукопожатию `connect`. См.
  [Автоматическое одобрение устройства по доверенному CIDR](#trusted-cidr-device-auto-approval)
  ниже и [Сопряжение канала](/ru/channels/pairing).
- **Одобрение возможностей Node** (`node.pair.*`) определяет, какие заявленные
  возможности/команды может предоставлять подключённый Node. Gateway является
  источником истины; интерфейсы (приложение macOS, Control UI) служат клиентскими
  интерфейсами для одобрения или отклонения ожидающих запросов.

Прежнее отдельное хранилище сопряжений Node (`nodes/paired.json` с отдельным
токеном для каждого Node, исключённое из пути подключения в январе 2026 года)
удалено: при запуске Gateway однократно переносит все оставшиеся записи в записи
устройств и архивирует устаревшие файлы с суффиксом `.migrated`. Поддержка
устаревшего моста TCP удалена.

## Как работает одобрение возможностей

1. Node подключается к WS Gateway (этот этап контролируется сопряжением устройства).
2. Gateway сравнивает заявленный набор возможностей/команд с одобренным;
   новые или расширенные наборы сохраняются как **ожидающий запрос** в записи
   устройства и вызывают событие `node.pair.requested`.
3. Вы одобряете или отклоняете запрос (через CLI или пользовательский интерфейс).
4. До одобрения команды Node остаются отфильтрованными; после одобрения
   заявленный набор становится доступен с учётом обычной политики команд.

Ожидающие запросы автоматически истекают **через 5 минут после последней
повторной попытки Node** — активно переподключающийся Node сохраняет один
ожидающий запрос активным вместо создания нового запроса (и запроса на
одобрение) при каждой попытке.

## Рабочий процесс CLI (подходит для работы без графического интерфейса)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes remove --node <id|name|ip>
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

`nodes status` показывает сопряжённые/подключённые Node и их возможности.

## Поверхность API (протокол Gateway)

События:

- `node.pair.requested` — вызывается при создании нового ожидающего запроса.
- `node.pair.resolved` — вызывается при одобрении, отклонении или истечении
  срока действия запроса.

Методы:

- `node.pair.list` — вывод ожидающих и сопряжённых Node (`operator.pairing`).
- `node.pair.approve` — одобрение ожидающего запроса.
- `node.pair.reject` — отклонение ожидающего запроса.
- `node.pair.remove` — удаление сопряжённого Node. При этом роль `node`
  устройства отзывается в хранилище сопряжённых устройств, вместе с ней
  удаляется одобренный набор возможностей Node, а сеансы этого устройства
  с ролью Node аннулируются и отключаются. Устройство **со смешанными ролями**
  (например, также имеющее роль `operator`) сохраняет свою запись и теряет
  только роль `node`; запись устройства только с ролью Node удаляется.
  Авторизация: `operator.pairing` позволяет удалять записи Node без роли
  оператора; вызывающей стороне с токеном устройства, отзывающей **собственную**
  роль Node у устройства со смешанными ролями, дополнительно требуется
  `operator.admin`.
- `node.rename` — изменение отображаемого для оператора имени сопряжённого Node.

Удалены в версии 2026.7: `node.pair.request` и `node.pair.verify`. Ожидающие
запросы создаются самим Gateway при подключении Node, а отдельного токена Node,
который они обслуживали, больше не существует; для аутентификации Node
используется токен сопряжения устройства.

Примечания:

- При повторных подключениях с неизменившимся набором используется тот же
  ожидающий запрос; повторные запросы обновляют сохранённые метаданные Node
  и последний снимок заявленных команд из списка разрешений для отображения оператору.
- Уровни областей доступа оператора и проверки во время одобрения кратко описаны
  в разделе [Области доступа оператора](/ru/gateway/operator-scopes).
- `node.pair.approve` использует заявленные команды ожидающего запроса, чтобы
  требовать дополнительные области доступа для одобрения:
  - запрос без команд: `operator.pairing`
  - запрос команд без выполнения: `operator.pairing` + `operator.write`
  - запрос `system.run` / `system.run.prepare` / `system.which`:
    `operator.pairing` + `operator.admin`

<Warning>
Одобрение сопряжения Node фиксирует доверенный набор возможностей. Оно **не** закрепляет текущий набор команд отдельно для каждого Node.

- Текущие команды Node определяются тем, что Node заявляет при подключении,
  и фильтруются глобальной политикой команд Node в Gateway
  (`gateway.nodes.allowCommands` и `denyCommands`).
- Политика разрешений и запросов `system.run` для каждого Node хранится на
  самом Node в `exec.approvals.node.*`, а не в записи сопряжения.

</Warning>

## Ограничение команд Node (2026.3.31+)

<Warning>
**Критическое изменение:** начиная с версии `2026.3.31`, команды Node отключены до одобрения сопряжения Node. Одного сопряжения устройства больше недостаточно для предоставления заявленных команд Node.
</Warning>

При первом подключении Node запрос на сопряжение создаётся автоматически.
До одобрения этого запроса все ожидающие команды от этого Node фильтруются и
не выполняются. После одобрения сопряжения заявленные команды Node становятся
доступны с учётом обычной политики команд.

Это означает:

- Node, которые ранее полагались только на сопряжение устройства для
  предоставления команд, теперь также должны завершить сопряжение Node.
- Команды, поставленные в очередь до одобрения сопряжения, отбрасываются,
  а не откладываются.

## Границы доверия событий Node (2026.3.31+)

<Warning>
**Критическое изменение:** запуски, инициированные Node, теперь ограничены сокращённой доверенной поверхностью.
</Warning>

Сводки, инициированные Node, и связанные события сеансов ограничены
предусмотренной доверенной поверхностью. Потоки, запускаемые уведомлениями
или Node, которые ранее полагались на более широкий доступ к инструментам
хоста или сеанса, могут потребовать корректировки. Это усиление защиты не
позволяет событиям Node повышать уровень доступа до инструментов хоста за
пределами, разрешёнными границей доверия Node.

Долговременные обновления присутствия Node следуют той же границе идентичности:
событие `node.presence.alive` принимается только от аутентифицированных сеансов
устройства Node и обновляет метаданные сопряжения только в том случае, если
идентичность устройства/Node уже сопряжена. Самостоятельно заявленного значения
`client.id` недостаточно для записи состояния последней активности.

## Автоматическое одобрение устройства с проверкой по SSH (по умолчанию)

Первичное сопряжение устройства с `role: node` с частного адреса/адреса CGNAT
одобряется автоматически, когда Gateway может **подтвердить владение машиной
через SSH**: он подключается обратно к узлу сопряжения (`BatchMode`,
`StrictHostKeyChecking=yes`), выполняет там `openclaw node identity --json`
и одобряет запрос, только если идентификатор удалённого устройства и открытый
ключ в точности совпадают с ожидающим запросом. Безопасность обеспечивает
именно совпадение ключа: одной доступности недостаточно для одобрения, поэтому
другие арендаторы за NAT, другие пользователи общего хоста и подмена в LAN
всегда приводят к обычному запросу подтверждения.

Включено по умолчанию. Условия срабатывания:

- Пользователь процесса Gateway (или `sshVerify.user`) может подключиться по
  SSH к хосту Node без интерактивного ввода (ключи/агент; Tailscale SSH также
  поддерживается), а ключ хоста уже является доверенным.
- Команда `openclaw` доступна в удалённой переменной `PATH` для
  неинтерактивного запуска `sh -lc`.
- IP-адрес подключения является прямым (без прокси и не local loopback)
  частным адресом, адресом ULA, link-local или CGNAT либо соответствует
  `sshVerify.cidrs`, если этот параметр задан.
- Применяется тот же минимальный порог соответствия требованиям, что и для
  одобрения по доверенному CIDR: только новое сопряжение Node без областей
  доступа; обновления, браузеры, Control UI и WebChat всегда требуют
  подтверждения.

Пока выполняется проверка, клиенту Node предписывается продолжать повторные
попытки (`wait_then_retry`), а не приостанавливаться в ожидании ручного
одобрения; если проверка завершается неудачно, следующая попытка возвращается
к обычному процессу запроса подтверждения. Для не прошедших проверку целей
действует короткая задержка (5 минут после несовпадения ключей).

Для одобренных устройств записывается `approvedVia: "ssh-verified"`, а их первый
заявленный набор возможностей одобряется на том же этапе — совпадение ключа уже
подтверждает, что Node работает под учётной записью оператора на принадлежащей
ему машине, то есть подтверждает то же утверждение, что и ручное одобрение
возможностей. Последующие расширения набора по-прежнему требуют подтверждения.

Усиление защиты или отключение:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        // Полностью отключить:
        sshVerify: false,
        // ...или ограничить/настроить проверку:
        // sshVerify: { user: "me", identity: "~/.ssh/probe", timeoutMs: 7000, cidrs: ["10.0.0.0/8"] },
      },
    },
  },
}
```

## Автоматическое одобрение (приложение macOS)

Приложение macOS может попытаться выполнить **тихое одобрение** запросов
возможностей Node, если:

- запрос помечен как `silent` (Gateway помечает первый набор возможностей как
  тихий, если сопряжение устройства было одобрено без интерактивного ввода), и
- приложение может проверить SSH-подключение к хосту Gateway с использованием
  того же пользователя.

Если тихое одобрение завершается неудачно, используется обычный запрос
Approve/Reject.

## Автоматическое одобрение устройства по доверенному CIDR

Сопряжение устройства WS для `role: node` по умолчанию остаётся ручным.
Для частных сетей Node, в которых Gateway уже доверяет сетевому пути,
операторы могут явно включить эту возможность с помощью CIDR или точных
IP-адресов:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Граница безопасности:

- Отключено, если `gateway.nodes.pairing.autoApproveCidrs` не задан.
- Универсального режима автоматического одобрения для LAN или частной сети не
  существует; автоматическое одобрение с проверкой по SSH (выше) требует
  криптографического совпадения ключа устройства, а не только сетевой близости.
- Подходит только новый запрос на сопряжение устройства с `role: node` без
  запрошенных областей доступа.
- Клиенты оператора, браузера, Control UI и WebChat остаются в ручном режиме.
- Обновления роли, области доступа, метаданных и открытого ключа остаются ручными.
- Пути заголовков доверенного прокси через local loopback на том же хосте не
  подходят, поскольку локальные вызывающие стороны могут подделать такой путь.

## Очистка заменённых тихих сопряжений

При неинтерактивных одобрениях их происхождение записывается в строке
сопряжённого устройства: одобрения локальной политикой на том же хосте — как
`silent`, одобрения Node по доверенному CIDR — как `trusted-cidr`, одобрения
Node с проверкой по SSH — как `ssh-verified`. Клиенты с эфемерным каталогом
состояния (временные домашние каталоги, контейнеры, отдельные песочницы для
каждого запуска) создают новую пару ключей устройства при каждом запуске, и
каждый запуск тихо сопрягается заново как совершенно новое устройство — без
очистки список сопряжённых устройств увеличивался бы на одну устаревшую запись
при каждом запуске.

Когда Gateway тихо одобряет **локальное** сопряжение устройства, он выводит из
эксплуатации более старые одобренные как `silent` записи, которые относятся к
тому же кластеру клиентов (совпадают `clientId`, `clientMode` и отображаемое
имя) и в данный момент не подключены. Локальные клиенты работают на самом хосте
Gateway, поэтому ключ кластера не может совпасть с другой машиной. Токены
выведенных из эксплуатации записей немедленно становятся недействительными;
все соответствующие устаревшие записи сопряжения Node удаляются, а событие
удаления `node.pair.resolved` передаётся всем клиентам.

Границы:

- Подходят только записи, последнее одобрение которых было выполнено локально
  на том же хосте (`silent`), как в качестве инициатора, так и в качестве цели.
  Сопряжения по доверенному CIDR и с проверкой по SSH охватывают разные хосты,
  где отображаемые метаданные не являются идентичностью машины, поэтому они
  никогда не удаляются автоматически — для них используйте очистку в
  Control UI или `openclaw nodes remove`.
- Сопряжения, одобренные владельцем, а также сопряжения с помощью QR-кода/кода
  настройки (начальной загрузки) никогда не удаляются автоматически. Записи,
  одобренные до появления данных о происхождении, остаются защищёнными даже
  после последующего тихого повторного одобрения того же идентификатора устройства.
- Подключённые в данный момент устройства пропускаются, поэтому параллельные
  локальные сеансы с разными каталогами состояния сохраняют свои токены, пока
  активны. Записи, одобренные в течение последней минуты, также пропускаются,
  чтобы одновременные рукопожатия сопряжения не могли вывести друг друга из
  эксплуатации до регистрации соответствующих подключений.
- Затронутые клиенты по определению являются локальными, поэтому при следующем
  подключении они снова сопрягаются в тихом режиме.

## Автоматическое одобрение обновления метаданных

Когда уже сопряжённое устройство переподключается только с изменениями
неконфиденциальных метаданных (например, отображаемого имени или подсказок о
платформе клиента), OpenClaw рассматривает это как `metadata-upgrade`. Тихое
автоматическое одобрение имеет узкую область применения: оно действует только
для доверенных локальных переподключений не из браузера, которые уже подтвердили
владение локальными или общими учётными данными, включая переподключения
нативного приложения на том же хосте после изменения метаданных версии ОС.
Клиенты браузера/Control UI и удалённые клиенты по-прежнему проходят явный
процесс повторного одобрения. Расширения областей доступа (с чтения до
записи/администрирования) и изменения открытого ключа **не** допускают
автоматического одобрения как обновления метаданных; для них сохраняются явные
запросы повторного одобрения.

## Вспомогательные средства сопряжения по QR-коду

`/pair qr` отображает данные сопряжения в виде структурированного медиаконтента, чтобы мобильные и браузерные клиенты могли сканировать их напрямую.

При удалении устройства также удаляются все устаревшие ожидающие запросы на сопряжение для идентификатора этого устройства, поэтому после отзыва доступа команда `nodes pending` не показывает потерявшие связь строки.

## Локальность и перенаправленные заголовки

При сопряжении Gateway считает подключение выполненным через local loopback, только если это подтверждается как необработанным сокетом, так и всеми данными от вышестоящего прокси. Если запрос поступает через local loopback, но содержит данные заголовка `Forwarded`, любого заголовка `X-Forwarded-*` или `X-Real-IP`, эти данные перенаправленных заголовков исключают возможность считать подключение локальным, и для сопряжения требуется явное одобрение вместо неявной обработки запроса как подключения с того же хоста. Эквивалентное правило для аутентификации оператора см. в разделе [Аутентификация через доверенный прокси](/ru/gateway/trusted-proxy-auth).

## Хранение (локальное, конфиденциальное)

Состояние сопряжения хранится в записях сопряжённых устройств в общей базе данных состояния SQLite в каталоге состояния Gateway (по умолчанию `~/.openclaw`):

- `~/.openclaw/state/openclaw.sqlite` (сопряжённые устройства с данными аутентификации, одобренные интерфейсы узлов, ожидающие запросы на доступ к интерфейсам, ожидающие запросы на сопряжение устройств и токены начальной настройки)

Если переопределить `OPENCLAW_STATE_DIR`, база данных переместится вместе с ним. При обновлении Gateway с выпусков, использовавших хранилища JSON, данные из них импортируются при запуске, а файлы `devices/*.json.migrated` и `nodes/*.json.migrated` сохраняются в качестве архивов.

Примечания по безопасности:

- Токены устройств являются секретами; считайте базу данных состояния конфиденциальной.
- Для ротации токена устройства используются `openclaw devices rotate` / `device.token.rotate`.

## Поведение транспорта

- Транспорт **не хранит состояние**; он не сохраняет сведения о членстве.
- Если Gateway недоступен или сопряжение отключено, узлы не могут выполнить сопряжение.
- В удалённом режиме сопряжение выполняется с хранилищем удалённого Gateway.
