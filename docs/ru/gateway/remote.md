OpenClaw запускает один Gateway (главный) на хосте и подключает к нему все клиенты. Gateway управляет сеансами, профилями аутентификации, каналами и состоянием; всё остальное является клиентом.

- **Операторы** (вы или приложение macOS): прямое подключение по WebSocket через LAN/Tailnet — самый простой вариант, когда Gateway доступен; SSH-туннелирование — универсальный резервный вариант.
- **Узлы** (iOS/Android и другие устройства): подключаются к **WebSocket** Gateway (через LAN/tailnet или SSH-туннель).

## Основная идея

По умолчанию WebSocket Gateway привязывается к **local loopback** на порту `18789` (`gateway.port`). Для удалённого использования предоставьте доступ через Tailscale Serve / доверенную привязку LAN-Tailnet либо перенаправьте порт local loopback через SSH.

Для конфигураций с постоянно включённым хостом и ноутбуком рекомендуется сохранять `gateway.bind: "loopback"` и использовать **Tailscale Serve** для интерфейса управления либо доверенную привязку LAN/Tailnet с `gateway.remote.transport: "direct"`. SSH-туннель — резервный вариант, работающий с любой машины.

## Поток команд (что где выполняется)

Один Gateway управляет состоянием и каналами; узлы являются периферийными устройствами. Пример (сообщение Telegram направляется инструменту узла):

1. Сообщение Telegram поступает в **Gateway**.
2. Gateway запускает **агента**, который решает, следует ли вызвать инструмент узла.
3. Gateway вызывает **узел** через WebSocket Gateway (`node.invoke` RPC).
4. Узел возвращает результат; Gateway отвечает в Telegram.

Узлы не запускают службу Gateway. На одном хосте должен работать только один Gateway, если только вы намеренно не используете изолированные профили (см. [Несколько Gateway](/ru/gateway/multiple-gateways)). «Режим узла» приложения macOS — это всего лишь клиент узла, работающий через WebSocket Gateway.

## SSH-туннель (CLI и инструменты)

```bash
ssh -N -L 18789:127.0.0.1:18789 user@gateway-host
```

При активном туннеле команды `openclaw health` и `openclaw status --deep` обращаются к удалённому Gateway через `ws://127.0.0.1:18789`. Команды `openclaw gateway status`, `openclaw gateway health`, `openclaw gateway probe` и `openclaw gateway call` также могут обращаться к перенаправленному URL с помощью `--url`.

<Note>
Замените `18789` на настроенное значение `gateway.port` (или `--port` / `OPENCLAW_GATEWAY_PORT`).
</Note>

<Warning>
При использовании `--url` учётные данные из конфигурации или переменных среды никогда не применяются как резервные. Явно передайте `--token` или `--password`; без них клиент не отправляет учётные данные, и подключение завершится ошибкой, если целевой Gateway требует аутентификацию.
</Warning>

## Удалённые настройки CLI по умолчанию

Сохраните удалённую цель, чтобы команды CLI использовали её по умолчанию:

```json5
{
  gateway: {
    mode: "remote",
    remote: {
      url: "ws://127.0.0.1:18789",
      token: "your-token",
    },
  },
}
```

Если Gateway доступен только через local loopback, оставьте URL `ws://127.0.0.1:18789` и сначала откройте SSH-туннель. При использовании транспорта через SSH-туннель в приложении macOS обнаруженное имя хоста Gateway указывается в `gateway.remote.sshTarget` (`user@host` или `user@host:port`); `gateway.remote.url` остаётся локальным URL туннеля. Если удалённый порт отличается от локального, задайте `gateway.remote.remotePort`.

По умолчанию используется строгая проверка ключа хоста (`gateway.remote.sshHostKeyPolicy: "strict"`). Установите значение `"openssh"`, чтобы делегировать проверку действующей конфигурации OpenSSH; перед включением проверьте пользовательские и системные настройки SSH.

Для Gateway, уже доступного в доверенной LAN или Tailnet, используйте прямой режим:

```json5
{
  gateway: {
    mode: "remote",
    remote: {
      transport: "direct",
      url: "ws://192.168.0.202:18789",
      token: "your-token",
    },
  },
}
```

## Приоритет учётных данных

Разрешение учётных данных Gateway следует единому контракту для путей вызова, проверки и состояния, а также для мониторинга подтверждения выполнения команд Discord. Хост узла использует тот же контракт с одним исключением для локального режима (он игнорирует `gateway.remote.*`).

- Явно заданные учётные данные (`--token`, `--password` или `gatewayToken` инструмента) всегда имеют приоритет в путях вызова, принимающих явные данные аутентификации.
- Безопасность переопределения URL:
  - CLI-флаг `--url` никогда не использует неявные учётные данные из конфигурации или переменных среды.
  - Переменная среды `OPENCLAW_GATEWAY_URL` может использовать только учётные данные из переменных среды (`OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`).
- Настройки локального режима по умолчанию:
  - токен: `OPENCLAW_GATEWAY_TOKEN` -> `gateway.auth.token` -> `gateway.remote.token` (удалённое резервное значение используется, только если локальный токен не задан)
  - пароль: `OPENCLAW_GATEWAY_PASSWORD` -> `gateway.auth.password` -> `gateway.remote.password` (удалённое резервное значение используется, только если локальный пароль не задан)
- Настройки удалённого режима по умолчанию:
  - токен: `gateway.remote.token` -> `OPENCLAW_GATEWAY_TOKEN` -> `gateway.auth.token`
  - пароль: `OPENCLAW_GATEWAY_PASSWORD` -> `gateway.remote.password` -> `gateway.auth.password`
- Исключение локального режима хоста узла: `gateway.remote.token` / `gateway.remote.password` игнорируются.
- По умолчанию проверки токена для удалённых запросов проверки и состояния строгие: при работе в удалённом режиме они используют только `gateway.remote.token` (без резервного перехода к локальному токену).
- Переопределения Gateway через переменные среды используют только `OPENCLAW_GATEWAY_*`.

## Удалённый доступ к интерфейсу чата

У WebChat нет отдельного HTTP-порта; интерфейс чата SwiftUI подключается непосредственно к WebSocket Gateway.

- Перенаправьте `18789` через SSH (см. выше), затем подключите клиенты к `ws://127.0.0.1:18789`.
- В прямом режиме LAN/Tailnet подключите клиенты к настроенному приватному URL `ws://` или защищённому URL `wss://`.
- В macOS удалённый режим приложения автоматически управляет выбранным транспортом.

## Удалённый режим приложения macOS

Приложение в строке меню macOS обеспечивает всю настройку от начала до конца: удалённые проверки состояния, WebChat и перенаправление Voice Wake. Инструкция: [удалённый доступ в macOS](/ru/platforms/mac/remote).

## Правила безопасности (удалённый доступ/VPN)

Оставляйте Gateway привязанным **только к local loopback**, если не уверены, что вам нужна другая привязка.

- **Local loopback + SSH/Tailscale Serve** — самый безопасный вариант по умолчанию (без публичного доступа).
- Незашифрованный `ws://` разрешён для local loopback, частных адресов/LAN (RFC 1918), локальных адресов канала, CGNAT, а также хостов `.local` и `.ts.net`. Публичные удалённые хосты должны использовать `wss://`.
- **Привязки не к local loopback** (`lan`/`tailnet`/`custom` или `auto`, когда local loopback недоступен) должны использовать аутентификацию Gateway: токен, пароль или обратный прокси с поддержкой идентификации и `gateway.auth.mode: "trusted-proxy"`.
- `gateway.remote.token` / `.password` являются источниками учётных данных клиента; сами по себе они не настраивают аутентификацию сервера.
- Локальные пути вызова могут использовать `gateway.remote.*` как резервный вариант, только если `gateway.auth.*` не задан.
- Если `gateway.auth.token` / `gateway.auth.password` явно настроен через SecretRef, но не разрешается, разрешение завершается с отказом (без маскирующего резервного перехода к удалённым данным).
- `gateway.remote.tlsFingerprint` закрепляет удалённый сертификат TLS для `wss://`, включая прямой режим macOS. Без сохранённого отпечатка macOS закрепляет сертификат при первом использовании только после успешной стандартной системной проверки доверия; для Gateway с самоподписанным сертификатом или сертификатом частного центра сертификации требуется явно заданный отпечаток либо удалённое подключение через SSH.
- **Tailscale Serve** может аутентифицировать трафик интерфейса управления/WebSocket с помощью заголовков идентификации при `gateway.auth.allowTailscale: true`. Конечные точки HTTP API не используют аутентификацию по этим заголовкам и вместо этого следуют обычному режиму HTTP-аутентификации Gateway. Этот поток без токена предполагает, что хост Gateway является доверенным; установите значение `false`, чтобы везде использовать аутентификацию с общим секретом.
- Аутентификация через **доверенный прокси** по умолчанию предполагает прокси с поддержкой идентификации, не привязанный к local loopback. Для обратных прокси на том же хосте, привязанных к local loopback, необходимо явно задать `gateway.auth.trustedProxy.allowLoopback = true`.
- Относитесь к управлению через браузер как к операторскому доступу: только через tailnet и с явным сопряжением узлов.

Подробнее: [Безопасность](/ru/gateway/security).

### macOS: постоянный SSH-туннель через LaunchAgent

Для клиентов macOS проще всего создать постоянную конфигурацию с записью SSH `LocalForward` и LaunchAgent, который поддерживает туннель после перезагрузок и сбоев.

#### Шаг 1: добавьте конфигурацию SSH

Измените `~/.ssh/config`:

```ssh
Host remote-gateway
    HostName <REMOTE_IP>
    User <REMOTE_USER>
    LocalForward 18789 127.0.0.1:18789
    IdentityFile ~/.ssh/id_rsa
```

Замените `<REMOTE_IP>` и `<REMOTE_USER>` своими значениями.

#### Шаг 2: скопируйте ключ SSH (однократно)

```bash
ssh-copy-id -i ~/.ssh/id_rsa <REMOTE_USER>@<REMOTE_IP>
```

#### Шаг 3: настройте токен Gateway

```bash
openclaw config set gateway.remote.token "<your-token>"
```

Если удалённый Gateway использует аутентификацию по паролю, укажите вместо этого `gateway.remote.password`. `OPENCLAW_GATEWAY_TOKEN` по-прежнему можно использовать как переопределение на уровне оболочки, но для постоянной настройки удалённого клиента следует применять `gateway.remote.token` / `gateway.remote.password`.

#### Шаг 4: создайте LaunchAgent

Сохраните файл как `~/Library/LaunchAgents/ai.openclaw.ssh-tunnel.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>ai.openclaw.ssh-tunnel</string>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/bin/ssh</string>
        <string>-N</string>
        <string>remote-gateway</string>
    </array>
    <key>KeepAlive</key>
    <true/>
    <key>RunAtLoad</key>
    <true/>
</dict>
</plist>
```

#### Шаг 5: загрузите LaunchAgent

```bash
launchctl bootstrap gui/$UID ~/Library/LaunchAgents/ai.openclaw.ssh-tunnel.plist
```

Туннель автоматически запускается при входе в систему, перезапускается после сбоя и поддерживает перенаправленный порт активным.

<Note>
Если после старой конфигурации у вас остался LaunchAgent `com.openclaw.ssh-tunnel`, выгрузите и удалите его.
</Note>

#### Устранение неполадок

```bash
# Проверить, работает ли туннель
ps aux | grep "ssh -N remote-gateway" | grep -v grep
lsof -i :18789

# Перезапустить туннель
launchctl kickstart -k gui/$UID/ai.openclaw.ssh-tunnel

# Остановить туннель
launchctl bootout gui/$UID/ai.openclaw.ssh-tunnel
```

| Запись конфигурации                   | Назначение                                                                |
| ------------------------------------ | ------------------------------------------------------------------------- |
| `LocalForward 18789 127.0.0.1:18789` | Перенаправляет локальный порт 18789 на удалённый порт 18789               |
| `ssh -N`                             | Запускает SSH без выполнения удалённых команд (только перенаправление портов) |
| `KeepAlive`                          | Автоматически перезапускает туннель после сбоя                            |
| `RunAtLoad`                          | Запускает туннель при загрузке LaunchAgent во время входа в систему       |
