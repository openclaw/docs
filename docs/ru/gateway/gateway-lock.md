## Зачем

- Только один процесс Gateway на хосте должен использовать заданную конфигурацию и порт; дополнительные процессы Gateway следует запускать с изолированными профилями и уникальными портами.
- Переживать сбои и `SIGKILL`, не оставляя устаревших файлов блокировки.
- Быстро завершаться с понятной ошибкой, если порт уже занят другим процессом Gateway.

## Два уровня

При запуске единоличное владение экземпляром обеспечивается двумя независимыми последовательными этапами:

1. **Файловая блокировка** захватывает отдельный для каждой конфигурации файл блокировки в каталоге блокировок состояния. При захвате блокировки процесс запуска проверяет настроенный порт на наличие активного слушателя, чтобы обнаружить устаревшую блокировку, оставшуюся после сбоя процесса-владельца.
2. **Привязка сокета** привязывает слушатель HTTP/WebSocket (по умолчанию `ws://127.0.0.1:18789`) как эксклюзивный слушатель TCP.

Каждый уровень может завершиться ошибкой независимо от другого и генерирует собственную ошибку `GatewayLockError`.

### Файловая блокировка

- Если файл блокировки отсутствует, зарегистрированный процесс-владелец завершился или проверка его порта не обнаруживает активного слушателя, процесс запуска освобождает блокировку и продолжает работу.
- Если блокировка активно удерживается и ни одно из перечисленных выше условий не выполняется, процесс запуска повторяет попытки в течение не более 5 секунд (по умолчанию), после чего прекращает их:

  ```text
  GatewayLockError("gateway already running (pid <pid>); lock timeout after <ms>ms")
  ```

### Привязка сокета

- При `EADDRINUSE` процесс запуска повторяет попытку привязки до 20 раз с интервалом 500 мс (в общей сложности около 10 секунд), чтобы дождаться завершения состояния `TIME_WAIT` после недавно завершившегося процесса.
- Если после повторных попыток порт по-прежнему занят:

  ```text
  GatewayLockError("another gateway instance is already listening on ws://127.0.0.1:<port>")
  ```

- Другие ошибки привязки:

  ```text
  GatewayLockError("failed to bind gateway socket on ws://127.0.0.1:<port>: <cause>")
  ```

При завершении работы Gateway закрывает сервер HTTP/WebSocket и удаляет файл блокировки.

## Примечания по эксплуатации

- Если порт занят другим процессом, не являющимся Gateway, ошибка будет такой же; освободите порт или выберите другой с помощью `openclaw gateway --port <port>`.
- При работе под управлением диспетчера служб новый процесс Gateway, столкнувшийся с одной из указанных выше ошибок, сначала проверяет `/healthz` существующего процесса. Если этот процесс исправен, новый процесс оставляет управление за ним вместо завершения с ошибкой. В systemd он завершается с кодом `78`; параметр модуля `RestartPreventExitStatus=78` не позволяет `Restart=always` зациклить перезапуски при конфликте блокировки или `EADDRINUSE`. Если существующий процесс так и не становится исправным, повторные проверки состояния ограничены по времени, после чего запуск завершается с указанной выше ошибкой блокировки вместо бесконечного цикла.
- Приложение macOS использует собственную упрощённую проверку PID перед запуском Gateway; фактическое ограничение во время выполнения обеспечивают описанные выше файловая блокировка и привязка сокета.
