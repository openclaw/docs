Статус: готово к эксплуатации через WhatsApp Web (Baileys). Gateway управляет связанными сеансами.

## Установка (по требованию)

- Онбординг (`openclaw onboard`) и `openclaw channels add --channel whatsapp`
  предлагают установить WhatsApp plugin при первом выборе.
- `openclaw channels login --channel whatsapp` также предлагает поток установки, если
  plugin еще отсутствует.
- Dev-канал + git checkout: по умолчанию используется локальный путь plugin.
- Stable/Beta: сначала устанавливает официальный plugin `@openclaw/whatsapp` из ClawHub,
  с npm в качестве резервного варианта.
- Среда выполнения WhatsApp распространяется вне основного npm-пакета OpenClaw, чтобы
  специфичные для WhatsApp зависимости среды выполнения оставались во внешнем plugin.

Ручная установка остается доступной:

```bash
openclaw plugins install clawhub:@openclaw/whatsapp
```

Используйте простой npm-пакет (`@openclaw/whatsapp`) только когда нужен резервный вариант
через registry. Закрепляйте точную версию только когда нужна воспроизводимая установка.

## Быстрая настройка

<Steps>
  <Step title="Настройте политику доступа WhatsApp">

```json5
{
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      allowFrom: ["+15551234567"],
      groupPolicy: "allowlist",
      groupAllowFrom: ["+15551234567"],
    },
  },
}
```

  </Step>

  <Step title="Свяжите WhatsApp (QR)">

```bash
openclaw channels login --channel whatsapp
```

    Текущий вход выполняется по QR-коду. В удаленных средах или средах без графического интерфейса убедитесь, что у вас
    есть надежный способ доставить текущий QR-код на телефон, который будет его сканировать,
    до запуска входа.

    Для конкретной учетной записи:

```bash
openclaw channels login --channel whatsapp --account work
```

    Чтобы подключить существующий/пользовательский каталог аутентификации WhatsApp Web перед входом:

```bash
openclaw channels add --channel whatsapp --account work --auth-dir /path/to/wa-auth
openclaw channels login --channel whatsapp --account work
```

  </Step>

  <Step title="Запустите gateway">

```bash
openclaw gateway
```

  </Step>

  <Step title="Одобрите первый запрос сопряжения (если используется режим сопряжения)">

```bash
openclaw pairing list whatsapp
openclaw pairing approve whatsapp <CODE>
```

    Запросы сопряжения истекают через 1 час. Число ожидающих запросов ограничено 3 на канал.

  </Step>
</Steps>

<Note>
OpenClaw рекомендует по возможности запускать WhatsApp на отдельном номере. (Метаданные канала и поток настройки оптимизированы для такой схемы, но схемы с личным номером также поддерживаются.)
</Note>

<Warning>
Текущий поток настройки WhatsApp поддерживает только QR. QR-коды, отрисованные в терминале, снимки экрана,
PDF-файлы или вложения в чатах могут истечь или стать нечитаемыми при передаче
с удаленной машины. Для удаленных хостов или хостов без графического интерфейса предпочитайте прямой способ
передачи изображения QR вместо ручного захвата из терминала.
</Warning>

## Вызовите текущего запрашивающего через MeowCaller (экспериментально)

WhatsApp plugin может предоставлять `whatsapp_call` в агентских ходах, начатых из WhatsApp. Инструмент
использует [MeowCaller](https://github.com/purpshell/meowcaller), чтобы выполнить голосовой вызов WhatsApp
текущему авторизованному запрашивающему и воспроизвести сообщение OpenClaw TTS после ответа. Инструмент
не принимает номер назначения, поэтому промпт не может перенаправить вызов третьей стороне.
Эта экспериментальная возможность по умолчанию отключена.

<Warning>
MeowCaller экспериментален, не имеет отмеченного релиза и использует отдельно сопряженный сеанс связанного устройства whatsmeow. Он не может повторно использовать учетные данные Baileys из WhatsApp plugin. Сопряжение добавляет
еще одно связанное устройство к той же учетной записи WhatsApp. Сканируйте с идентификатором WhatsApp, используемым
OpenClaw. Режим личного номера/чата с самим собой не может звонить самому себе; используйте выделенный номер OpenClaw,
чтобы звонить на личный номер.
</Warning>

<Steps>
  <Step title="Включите экспериментальные вызовы">

    Добавьте `actions.calls: true` в канал WhatsApp в `openclaw.json`:

```json
{
  "channels": {
    "whatsapp": {
      "actions": {
        "calls": true
      }
    }
  }
}
```

    Объедините это с существующей конфигурацией WhatsApp, затем перезапустите gateway. Когда
    параметр отсутствует или равен `false`, OpenClaw не предоставляет инструмент `whatsapp_call` агенту.

  </Step>

  <Step title="Установите проверенный MeowCaller CLI">

    Адаптер ожидает исполняемый файл с именем `meowcaller` в `PATH` хоста gateway.
    Пока [MeowCaller PR #7](https://github.com/purpshell/meowcaller/pull/7) не объединен, соберите
    проверенную ветку на коммите `752050471fc2bf7a8cdfbf7dbd3cd4e865d85d3f`:

```bash
git clone --branch feat/send-only-notify https://github.com/steipete/meowcaller.git
cd meowcaller
git checkout 752050471fc2bf7a8cdfbf7dbd3cd4e865d85d3f
mkdir -p "$HOME/.local/bin"
go build -o "$HOME/.local/bin/meowcaller" ./cmd/meowcaller
```

    Убедитесь, что `$HOME/.local/bin` также находится в `PATH` службы gateway. Эта ревизия предоставляет
    явные команды `pair` и send-only `notify`. `notify` не открывает микрофон, динамик,
    видеоустройство, приемник входящего аудио или диагностический захват. Не заменяйте ее командой
    `play` из примерного CLI.

  </Step>

  <Step title="Сопрягите связанное устройство MeowCaller">

    Попросите агента WhatsApp проверить настройку вызовов. Действие статуса `whatsapp_call` сообщает
    каталог состояния для конкретной учетной записи и команду сопряжения. Для учетной записи по умолчанию:

```bash
state_dir="$HOME/.openclaw/credentials/whatsapp-calls/default"
mkdir -p "$state_dir"
chmod 700 "$state_dir"
meowcaller pair --store "$state_dir/wa-voip.db"
```

    Запустите команду в интерактивном терминале. Сканируйте ее QR из **WhatsApp > Связанные устройства**
    и дождитесь `MeowCaller linked device ready`. Затем команда завершится. Держите `wa-voip.db`
    приватным; это сеанс связанного устройства MeowCaller. Действие статуса `whatsapp_call`
    возвращает команду и оболочку для конкретной учетной записи, когда используется учетная запись не по умолчанию. В
    Windows запустите ее команду PowerShell; MeowCaller создаст каталог хранилища.

  </Step>

  <Step title="Настройте TTS и вызов из WhatsApp">

    Настройте [поставщика TTS](/ru/tools/tts) с поддержкой телефонии, перезапустите gateway, затем отправьте
    запрос WhatsApp, например `Call me and say the build finished.` Инструмент определяет отправителя
    из доверенного входящего контекста, синтезирует временный приватный WAV-файл, запускает MeowCaller на
    ограниченное окно вызова и затем удаляет аудиофайл. OpenClaw явно передает хранилище
    учетной записи, ожидает нулевой код выхода после ответа, воспроизведения и завершения вызова, и считает
    тайм-аут или ненулевой код выхода неудачным вызовом инструмента.

  </Step>
</Steps>

Текущие ограничения:

- только исходящие аудиовызовы один-на-один
- нет произвольных номеров назначения
- нет общей аутентификации с чат-соединением
- нет вызовов самому себе из режима личного номера/чата с самим собой
- синтезированное аудио ограничено 60 секундами
- нет подтверждения слышимости на стороне телефона, кроме завершения ответа/воспроизведения/завершения вызова в MeowCaller
- OpenClaw останавливает сопутствующий процесс после ограниченного окна 115–175 секунд, включая
  фазы подключения, ответа, воспроизведения и завершения MeowCaller

## Модель среды выполнения

- Gateway управляет сокетом WhatsApp и циклом повторного подключения.
- Watchdog повторного подключения использует активность транспорта WhatsApp Web, а не только объем входящих сообщений приложения, поэтому тихий сеанс связанного устройства не перезапускается только потому, что недавно никто не отправлял сообщения. Более длинный предел тишины приложения все равно принудительно выполняет повторное подключение, если транспортные фреймы продолжают поступать, но сообщения приложения не обрабатываются в течение окна watchdog; после временного повторного подключения для недавно активного сеанса эта проверка тишины приложения использует обычный тайм-аут сообщений для первого окна восстановления.
- Тайминги сокета Baileys явно задаются в `web.whatsapp.*`: `keepAliveIntervalMs` управляет ping-сообщениями приложения WhatsApp Web, `connectTimeoutMs` управляет тайм-аутом начального рукопожатия, а `defaultQueryTimeoutMs` управляет ожиданиями запросов Baileys плюс локальными границами операций OpenClaw для исходящей отправки/присутствия и входящих подтверждений прочтения.
- Исходящие отправки требуют активного слушателя WhatsApp для целевой учетной записи.
- Групповые отправки добавляют нативные метаданные упоминаний для токенов `@+<digits>` и `@<digits>` в тексте и подписях к медиа, когда токен соответствует текущим метаданным участника WhatsApp, включая группы на основе LID.
- Чаты статусов и рассылок игнорируются (`@status`, `@broadcast`).
- Watchdog повторного подключения следует активности транспорта WhatsApp Web, а не только объему входящих сообщений приложения: тихие сеансы связанного устройства остаются активными, пока транспортные фреймы продолжаются, но остановка транспорта принудительно вызывает повторное подключение задолго до более позднего пути удаленного отключения.
- Прямые чаты используют правила сеансов DM (`session.dmScope`; значение по умолчанию `main` сворачивает DM в основной сеанс агента).
- Групповые сеансы изолированы (`agent:<agentId>:whatsapp:group:<jid>`).
- WhatsApp Channels/Newsletters могут быть явными исходящими целями с их нативным JID `@newsletter`. Исходящие отправки в newsletter используют метаданные сеанса канала (`agent:<agentId>:whatsapp:channel:<jid>`), а не семантику сеанса DM.
- Транспорт WhatsApp Web учитывает стандартные переменные окружения прокси на хосте gateway (`HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY` / варианты в нижнем регистре). Предпочитайте прокси-конфигурацию на уровне хоста вместо специфичных для канала настроек прокси WhatsApp.
- Когда включено `messages.removeAckAfterReply`, OpenClaw очищает реакцию подтверждения WhatsApp после доставки видимого ответа.

## Запросы одобрения

WhatsApp может отображать запросы одобрения exec и plugin с реакциями `👍` / `👎`. Доставка
управляется конфигурацией пересылки одобрений верхнего уровня:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "session",
    },
    plugin: {
      enabled: true,
      mode: "targets",
      targets: [{ channel: "whatsapp", to: "+15551234567" }],
    },
  },
}
```

`approvals.exec` и `approvals.plugin` независимы. Включение WhatsApp как канала только связывает
транспорт; оно не отправляет запросы одобрения, если соответствующее семейство одобрений не включено
и не маршрутизируется в WhatsApp. Режим сеанса доставляет нативные emoji-одобрения только для одобрений,
которые исходят из WhatsApp. Режим цели использует общий pipeline пересылки для явных целей WhatsApp
и не создает отдельную рассылку approver-DM.

Реакции одобрения WhatsApp требуют явных approver WhatsApp из `allowFrom` или `"*"`.
`defaultTo` управляет обычными целями сообщений по умолчанию; это не approver одобрений. Ручные
команды `/approve` все равно проходят через обычный путь авторизации отправителя WhatsApp перед
разрешением одобрения.

## Хуки plugin и приватность

Входящие сообщения WhatsApp могут содержать личное содержимое сообщений, номера телефонов,
идентификаторы групп, имена отправителей и поля корреляции сеансов. Поэтому
WhatsApp не транслирует входящие полезные нагрузки хуков `message_received` в plugins,
если вы явно не включите это:

```json5
{
  channels: {
    whatsapp: {
      pluginHooks: {
        messageReceived: true,
      },
    },
  },
}
```

Вы можете ограничить включение одной учетной записью:

```json5
{
  channels: {
    whatsapp: {
      accounts: {
        work: {
          pluginHooks: {
            messageReceived: true,
          },
        },
      },
    },
  },
}
```

Включайте это только для plugins, которым вы доверяете получение содержимого
и идентификаторов входящих сообщений WhatsApp.

## Контроль доступа и активация

<Tabs>
  <Tab title="Политика DM">
    `channels.whatsapp.dmPolicy` управляет доступом к прямым чатам:

    - `pairing` (по умолчанию)
    - `allowlist`
    - `open` (требует, чтобы `allowFrom` включал `"*"`)
    - `disabled`

    `allowFrom` принимает номера в стиле E.164 (нормализуются внутренне).

    `allowFrom` — это список контроля доступа отправителей DM. Он не ограничивает явные исходящие отправки в групповые JID WhatsApp или JID каналов `@newsletter`.

    Переопределение для нескольких учетных записей: `channels.whatsapp.accounts.<id>.dmPolicy` (и `allowFrom`) имеют приоритет над значениями по умолчанию уровня канала для этой учетной записи.

    Подробности поведения во время выполнения:

    - привязки сохраняются в хранилище разрешений канала и объединяются с настроенным `allowFrom`
    - запланированная автоматизация и резервный получатель Heartbeat используют явные цели доставки или настроенный `allowFrom`; подтверждения привязки DM не являются неявными получателями Cron или Heartbeat
    - если список разрешений не настроен, связанный собственный номер разрешен по умолчанию
    - OpenClaw никогда автоматически не привязывает исходящие DM `fromMe` (сообщения, которые вы отправляете себе со связанного устройства)

  </Tab>

  <Tab title="Политика групп + списки разрешений">
    Доступ к группам имеет два уровня:

    1. **Список разрешенных групповых членств** (`channels.whatsapp.groups`)
       - если `groups` опущен, допустимы все группы
       - если `groups` присутствует, он действует как список разрешенных групп (разрешено `"*"`)

    2. **Политика групповых отправителей** (`channels.whatsapp.groupPolicy` + `groupAllowFrom`)
       - `open`: список разрешений отправителей обходится
       - `allowlist`: отправитель должен соответствовать `groupAllowFrom` (или `*`)
       - `disabled`: блокировать все входящие групповые сообщения

    Резервный список разрешений отправителей:

    - если `groupAllowFrom` не задан, среда выполнения откатывается к `allowFrom`, когда он доступен
    - списки разрешений отправителей оцениваются до активации по упоминанию/ответу

    Примечание: если блок `channels.whatsapp` полностью отсутствует, резервная политика групп во время выполнения — `allowlist` (с предупреждающим логом), даже если задан `channels.defaults.groupPolicy`.

  </Tab>

  <Tab title="Упоминания + /activation">
    Ответы в группах по умолчанию требуют упоминания.

    Обнаружение упоминаний включает:

    - явные упоминания WhatsApp идентичности бота
    - настроенные regex-шаблоны упоминаний (`agents.list[].groupChat.mentionPatterns`, резервный вариант `messages.groupChat.mentionPatterns`)
    - расшифровки входящих голосовых заметок для авторизованных групповых сообщений
    - неявное обнаружение ответа боту (отправитель ответа совпадает с идентичностью бота)

    Примечание по безопасности:

    - цитата/ответ удовлетворяет только проверку упоминания; это **не** предоставляет авторизацию отправителю
    - с `groupPolicy: "allowlist"` отправители вне списка разрешений все равно блокируются, даже если они отвечают на сообщение пользователя из списка разрешений

    Команда активации уровня сеанса:

    - `/activation mention`
    - `/activation always`

    `activation` обновляет состояние сеанса (не глобальную конфигурацию). Оно ограничено владельцем.

  </Tab>
</Tabs>

## Настроенные привязки ACP

WhatsApp поддерживает постоянные привязки ACP с записями верхнего уровня `bindings[]`:

```json5
{
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "direct", id: "+15555550123" },
      },
    },
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "whatsapp",
        accountId: "work",
        peer: { kind: "group", id: "120363424282127706@g.us" },
      },
    },
  ],
}
```

- Прямые чаты сопоставляются с номерами E.164, например `+15555550123`.
- Группы сопоставляются с групповыми JID WhatsApp, например `120363424282127706@g.us`.
- Списки разрешений групп, политика отправителей и проверка упоминания или активации выполняются до того, как OpenClaw гарантирует существование настроенного сеанса ACP.
- Сопоставленная настроенная привязка ACP владеет маршрутом. Группы рассылки WhatsApp не разветвляют этот ход на обычные сеансы WhatsApp.

## Поведение личного номера и чата с самим собой

Когда связанный собственный номер также присутствует в `allowFrom`, активируются меры защиты чата с самим собой в WhatsApp:

- пропускать уведомления о прочтении для ходов чата с самим собой
- игнорировать поведение автозапуска по JID упоминания, которое иначе отправило бы вам уведомление
- если `messages.responsePrefix` не задан, ответы в чате с самим собой по умолчанию используют `[{identity.name}]` или `[openclaw]`

## Доставка, разбиение на фрагменты и медиа

<AccordionGroup>
  <Accordion title="Разбиение текста на фрагменты">
    - лимит фрагмента по умолчанию: `channels.whatsapp.textChunkLimit = 4000`
    - `channels.whatsapp.chunkMode = "length" | "newline"`
    - режим `newline` предпочитает границы абзацев (пустые строки), затем откатывается к безопасному по длине разбиению

  </Accordion>

  <Accordion title="Поведение исходящих медиа">
    - поддерживает полезные нагрузки изображений, видео, аудио (голосовая заметка PTT) и документов
    - аудиомедиа отправляется через полезную нагрузку Baileys `audio` с `ptt: true`, поэтому клиенты WhatsApp отображают его как голосовую заметку push-to-talk
    - полезные нагрузки ответов сохраняют `audioAsVoice`; вывод голосовой заметки TTS для WhatsApp остается на этом пути PTT, даже когда провайдер возвращает MP3 или WebM
    - нативное аудио Ogg/Opus отправляется как `audio/ogg; codecs=opus` для совместимости с голосовыми заметками

    - `forceDocument` / `asDocument` отправляет исходящие изображения, GIF и видео через полезную нагрузку документа Baileys, чтобы избежать сжатия медиа WhatsApp, сохраняя разрешенное имя файла и MIME-тип
    - подписи применяются к первому медиаэлементу при отправке полезных нагрузок ответа с несколькими медиа, кроме голосовых заметок PTT: они отправляют аудио первым, а видимый текст отдельно, потому что клиенты WhatsApp не отображают подписи голосовых заметок стабильно
    - источник медиа может быть HTTP(S), `file://` или локальными путями

  </Accordion>

  <Accordion title="Ограничения размера медиа и резервное поведение">

    - лимит отправки исходящих медиа: `channels.whatsapp.mediaMaxMb` (по умолчанию `50`)

    - при сбое отправки медиа резервное поведение для первого элемента отправляет текстовое предупреждение вместо тихого отбрасывания ответа

  </Accordion>

## Цитирование ответов

WhatsApp поддерживает нативное цитирование ответов, при котором исходящие ответы визуально цитируют входящее сообщение. Управляйте этим через `channels.whatsapp.replyToMode`.

| Значение    | Поведение                                                             |
| ----------- | --------------------------------------------------------------------- |
| `"off"`     | Никогда не цитировать; отправлять как обычное сообщение               |
| `"first"`   | Цитировать только первый фрагмент исходящего ответа                   |
| `"all"`     | Цитировать каждый фрагмент исходящего ответа                          |
| `"batched"` | Цитировать поставленные в очередь пакетные ответы, оставляя немедленные ответы без цитирования |

По умолчанию используется `"off"`. Переопределения для учетной записи используют `channels.whatsapp.accounts.<id>.replyToMode`.

```json5
{
  channels: {
    whatsapp: {
      replyToMode: "first",
    },
  },
}
```

## Уровень реакций

`channels.whatsapp.reactionLevel` управляет тем, насколько широко агент использует emoji-реакции в WhatsApp:

| Уровень       | Реакции подтверждения | Реакции, инициированные агентом | Описание                                                  |
| ------------- | --------------------- | ------------------------------- | --------------------------------------------------------- |
| `"off"`       | Нет                   | Нет                             | Реакций нет вообще                                        |
| `"ack"`       | Да                    | Нет                             | Только реакции подтверждения (получение перед ответом)   |
| `"minimal"`   | Да                    | Да (консервативно)              | Подтверждение + реакции агента с консервативными указаниями |
| `"extensive"` | Да                    | Да (поощряется)                 | Подтверждение + реакции агента с поощряющими указаниями   |

По умолчанию: `"minimal"`.

Переопределения для учетной записи используют `channels.whatsapp.accounts.<id>.reactionLevel`.

```json5
{
  channels: {
    whatsapp: {
      reactionLevel: "ack",
    },
  },
}
```

## Реакции подтверждения

WhatsApp поддерживает немедленные реакции подтверждения при получении входящего сообщения через `channels.whatsapp.ackReaction`.
Реакции подтверждения ограничиваются `reactionLevel` — они подавляются, когда `reactionLevel` равен `"off"`.

```json5
{
  channels: {
    whatsapp: {
      ackReaction: {
        emoji: "👀",
        direct: true,
        group: "mentions", // always | mentions | never
      },
    },
  },
}
```

Примечания к поведению:

- отправляется сразу после принятия входящего сообщения (до ответа)
- если `ackReaction` указан без `emoji`, WhatsApp использует эмодзи идентичности маршрутизированного агента, с откатом к "👀"; опустите `ackReaction` или задайте `emoji: ""`, чтобы не отправлять реакцию подтверждения
- сбои логируются, но не блокируют обычную доставку ответа
- групповой режим `mentions` реагирует на ходы, запущенные упоминанием; групповая активация `always` действует как обход этой проверки
- WhatsApp использует `channels.whatsapp.ackReaction` (устаревший `messages.ackReaction` здесь не используется)

## Реакции статуса жизненного цикла

Задайте `messages.statusReactions.enabled: true`, чтобы WhatsApp заменял реакцию подтверждения во время хода вместо того, чтобы оставлять статичный эмодзи квитанции. Когда это включено, OpenClaw использует тот же слот реакции входящего сообщения для состояний жизненного цикла, таких как поставлено в очередь, обдумывание, активность инструмента, Compaction, завершено и ошибка.

```json5
{
  messages: {
    statusReactions: {
      enabled: true,
      emojis: {
        deploy: "🛫",
        build: "🏗️",
        concierge: "💁",
      },
    },
  },
}
```

Примечания к поведению:

- `channels.whatsapp.ackReaction` по-прежнему управляет тем, доступны ли реакции статуса для личных сообщений и групп.
- Реакция статуса очереди использует тот же эффективный эмодзи подтверждения, что и обычные реакции подтверждения.
- В WhatsApp есть один слот реакции бота на сообщение, поэтому обновления жизненного цикла заменяют текущую реакцию на месте.
- `messages.removeAckAfterReply: true` очищает финальную реакцию статуса после настроенной задержки для завершения/ошибки.
- Категории эмодзи инструментов включают `tool`, `coding`, `web`, `deploy`, `build` и `concierge`.

## Устранение неполадок

<AccordionGroup>
  <Accordion title="Не связано (требуется QR)">
    Симптом: статус канала сообщает, что он не связан.

    Исправление:

    ```bash
    openclaw channels login --channel whatsapp
    openclaw channels status
    ```

  </Accordion>

  <Accordion title="Связано, но отключено / цикл повторного подключения">
    Симптом: связанная учетная запись с повторяющимися отключениями или попытками повторного подключения.

    Тихие учетные записи могут оставаться подключенными дольше обычного тайм-аута сообщений; сторожевой механизм
    перезапускается, когда прекращается активность транспорта WhatsApp Web, закрывается сокет или
    активность на уровне приложения остается бездействующей дольше расширенного защитного окна.

    Если логи показывают повторяющееся `status=408 Request Time-out Connection was lost`, настройте
    тайминги сокета Baileys в `web.whatsapp`. Начните с уменьшения
    `keepAliveIntervalMs` ниже тайм-аута простоя вашей сети и увеличения
    `connectTimeoutMs` на медленных или нестабильных соединениях:

    ```json5
    {
      web: {
        whatsapp: {
          keepAliveIntervalMs: 15000,
          connectTimeoutMs: 60000,
          defaultQueryTimeoutMs: 60000,
        },
      },
    }
    ```

    Исправление:

    ```bash
    openclaw channels status --probe
    openclaw doctor
    openclaw logs --follow
    openclaw gateway status
    ```

    Если цикл сохраняется после исправления подключения хоста и таймингов, создайте резервную копию
    каталога авторизации учетной записи и заново свяжите эту учетную запись:

    ```bash
    cp -a ~/.openclaw/credentials/whatsapp/<accountId> \
      ~/.openclaw/credentials/whatsapp/<accountId>.bak
    openclaw channels logout --channel whatsapp --account <accountId>
    openclaw channels login --channel whatsapp --account <accountId>
    ```

    Если `~/.openclaw/logs/whatsapp-health.log` сообщает `Gateway inactive`, но
    `openclaw gateway status` и `openclaw channels status --probe` показывают, что
    Gateway и WhatsApp исправны, запустите `openclaw doctor`. В Linux doctor
    предупреждает об устаревших записях crontab, которые все еще вызывают
    `~/.openclaw/bin/ensure-whatsapp.sh`; удалите эти устаревшие записи через
    `crontab -e`, потому что cron может не иметь окружения пользовательской шины systemd и
    из-за этого старый скрипт может неверно сообщать о состоянии Gateway.

    При необходимости заново свяжите через `channels login`.

  </Accordion>

  <Accordion title="Вход по QR завершается тайм-аутом за прокси">
    Симптом: `openclaw channels login --channel whatsapp` завершается с ошибкой до показа пригодного QR-кода с `status=408 Request Time-out` или отключением TLS-сокета.

    Вход WhatsApp Web использует стандартное прокси-окружение хоста Gateway (`HTTPS_PROXY`, `HTTP_PROXY`, варианты в нижнем регистре и `NO_PROXY`). Проверьте, что процесс Gateway наследует переменные окружения прокси и что `NO_PROXY` не совпадает с `mmg.whatsapp.net`.

  </Accordion>

  <Accordion title="Нет активного слушателя при отправке">
    Исходящие отправки быстро завершаются ошибкой, когда для целевой учетной записи нет активного слушателя Gateway.

    Убедитесь, что Gateway запущен и учетная запись связана.

  </Accordion>

  <Accordion title="Ответ отображается в транскрипте, но не в WhatsApp">
    Строки транскрипта записывают то, что сгенерировал агент. Доставка WhatsApp проверяется отдельно: OpenClaw считает автоответ отправленным только после того, как Baileys возвращает идентификатор исходящего сообщения хотя бы для одной видимой отправки текста или медиа.

    Реакции подтверждения являются независимыми квитанциями до ответа. Успешная реакция не доказывает, что последующий текстовый или медиаответ был принят WhatsApp.

    Проверьте логи Gateway на `auto-reply delivery failed` или `auto-reply was not accepted by WhatsApp provider`.

  </Accordion>

  <Accordion title="Групповые сообщения неожиданно игнорируются">
    Проверяйте в таком порядке:

    - `groupPolicy`
    - `groupAllowFrom` / `allowFrom`
    - записи списка разрешенных в `groups`
    - ограничение по упоминанию (`requireMention` + шаблоны упоминаний)
    - повторяющиеся ключи в `openclaw.json` (JSON5): более поздние записи переопределяют более ранние, поэтому оставляйте один `groupPolicy` на область

    Если `channels.whatsapp.groups` присутствует, WhatsApp все еще может наблюдать сообщения из других групп, но OpenClaw отбрасывает их до маршрутизации сессии. Добавьте JID группы в `channels.whatsapp.groups` или добавьте `groups["*"]`, чтобы допустить все группы, сохраняя авторизацию отправителей через `groupPolicy` и `groupAllowFrom`.

  </Accordion>

  <Accordion title="Предупреждение среды выполнения Bun">
    Среда выполнения Gateway WhatsApp должна использовать Node. Bun помечен как несовместимый для стабильной работы Gateway WhatsApp/Telegram.
  </Accordion>
</AccordionGroup>

## Системные промпты

WhatsApp поддерживает системные промпты в стиле Telegram для групп и личных чатов через карты `groups` и `direct`.

Иерархия разрешения для групповых сообщений:

Эффективная карта `groups` определяется первой: если учетная запись определяет собственную `groups`, она полностью заменяет корневую карту `groups` (без глубокого слияния). Затем поиск промпта выполняется по получившейся единственной карте:

1. **Системный промпт конкретной группы** (`groups["<groupId>"].systemPrompt`): используется, когда запись конкретной группы существует в карте **и** ее ключ `systemPrompt` определен. Если `systemPrompt` является пустой строкой (`""`), подстановочный вариант подавляется и системный промпт не применяется.
2. **Подстановочный системный промпт группы** (`groups["*"].systemPrompt`): используется, когда запись конкретной группы полностью отсутствует в карте или когда она существует, но не определяет ключ `systemPrompt`.

Иерархия разрешения для личных сообщений:

Эффективная карта `direct` определяется первой: если учетная запись определяет собственную `direct`, она полностью заменяет корневую карту `direct` (без глубокого слияния). Затем поиск промпта выполняется по получившейся единственной карте:

1. **Системный промпт конкретного личного чата** (`direct["<peerId>"].systemPrompt`): используется, когда запись конкретного собеседника существует в карте **и** ее ключ `systemPrompt` определен. Если `systemPrompt` является пустой строкой (`""`), подстановочный вариант подавляется и системный промпт не применяется.
2. **Подстановочный системный промпт личного чата** (`direct["*"].systemPrompt`): используется, когда запись конкретного собеседника полностью отсутствует в карте или когда она существует, но не определяет ключ `systemPrompt`.

<Note>
`dms` остается легким контейнером переопределения истории для каждого личного сообщения (`dms.<id>.historyLimit`). Переопределения промптов находятся в `direct`.
</Note>

**Отличие от поведения Telegram с несколькими учетными записями:** В Telegram корневой `groups` намеренно подавляется для всех учетных записей в конфигурации с несколькими учетными записями — даже для учетных записей, которые не определяют собственный `groups`, — чтобы бот не получал групповые сообщения для групп, к которым он не принадлежит. WhatsApp не применяет эту защиту: корневые `groups` и корневой `direct` всегда наследуются учетными записями, которые не определяют переопределение на уровне учетной записи, независимо от того, сколько учетных записей настроено. В конфигурации WhatsApp с несколькими учетными записями, если вам нужны групповые или личные промпты для каждой учетной записи, явно определяйте полную карту под каждой учетной записью, а не полагайтесь на значения по умолчанию корневого уровня.

Важное поведение:

- `channels.whatsapp.groups` является одновременно картой конфигурации для каждой группы и списком разрешенных групп на уровне чата. В корневой области или области учетной записи `groups["*"]` означает "допущены все группы" для этой области.
- Добавляйте подстановочный групповой `systemPrompt` только тогда, когда вы уже хотите, чтобы эта область допускала все группы. Если вы все еще хотите, чтобы подходил только фиксированный набор идентификаторов групп, не используйте `groups["*"]` для значения промпта по умолчанию. Вместо этого повторите промпт в каждой явно разрешенной записи группы.
- Допуск группы и авторизация отправителя являются отдельными проверками. `groups["*"]` расширяет набор групп, которые могут попасть в обработку групп, но сам по себе не авторизует каждого отправителя в этих группах. Доступ отправителей по-прежнему отдельно управляется `channels.whatsapp.groupPolicy` и `channels.whatsapp.groupAllowFrom`.
- `channels.whatsapp.direct` не имеет такого же побочного эффекта для личных сообщений. `direct["*"]` только предоставляет конфигурацию личного чата по умолчанию после того, как личное сообщение уже допущено через `dmPolicy` плюс `allowFrom` или правила хранилища сопряжений.

Пример:

```json5
{
  channels: {
    whatsapp: {
      groups: {
        // Use only if all groups should be admitted at the root scope.
        // Applies to all accounts that do not define their own groups map.
        "*": { systemPrompt: "Default prompt for all groups." },
      },
      direct: {
        // Applies to all accounts that do not define their own direct map.
        "*": { systemPrompt: "Default prompt for all direct chats." },
      },
      accounts: {
        work: {
          groups: {
            // This account defines its own groups, so root groups are fully
            // replaced. To keep a wildcard, define "*" explicitly here too.
            "120363406415684625@g.us": {
              requireMention: false,
              systemPrompt: "Focus on project management.",
            },
            // Use only if all groups should be admitted in this account.
            "*": { systemPrompt: "Default prompt for work groups." },
          },
          direct: {
            // This account defines its own direct map, so root direct entries are
            // fully replaced. To keep a wildcard, define "*" explicitly here too.
            "+15551234567": { systemPrompt: "Prompt for a specific work direct chat." },
            "*": { systemPrompt: "Default prompt for work direct chats." },
          },
        },
      },
    },
  },
}
```
