Группы доступа — это именованные списки отправителей, которые вы один раз определяете в `accessGroups`, а затем указываете в списках разрешений каналов как `accessGroup:<name>`.

Используйте их, когда одним и тем же людям нужно разрешить доступ в нескольких каналах сообщений или когда один доверенный набор должен применяться как к личным сообщениям, так и к авторизации отправителей в группах.

Сама по себе группа ничего не разрешает. Она учитывается только там, где на неё ссылается поле списка разрешений.

## Статические группы отправителей сообщений

Статические группы отправителей используют `type: "message.senders"`. Ключами `members` служат идентификаторы каналов сообщений, а ключ `"*"` содержит записи, общие для всех каналов:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        "*": ["global-owner-id"],
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
}
```

| Ключ                       | Значение                                                                                 |
| -------------------------- | ---------------------------------------------------------------------------------------- |
| `"*"`                      | Общие записи, проверяемые для каждого канала сообщений, который ссылается на эту группу. |
| `discord`, `telegram`, ... | Записи, проверяемые только при сопоставлении со списком разрешений соответствующего канала. |

Записи сопоставляются по обычным правилам `allowFrom` целевого канала. OpenClaw не преобразует идентификаторы отправителей между каналами: если у Алисы есть идентификаторы Telegram и Discord, укажите оба идентификатора под соответствующими ключами каналов.

## Ссылки на группы в списках разрешений

Указывайте группу как `accessGroup:<name>` везде, где путь канала сообщений поддерживает списки разрешённых отправителей.

Пример списка разрешений для личных сообщений:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
      },
    },
  },
  channels: {
    discord: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:operators"],
    },
    telegram: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:operators"],
    },
  },
}
```

Пример списка разрешённых отправителей для групп:

```json5
{
  accessGroups: {
    oncall: {
      type: "message.senders",
      members: {
        whatsapp: ["+15551234567"],
        googlechat: ["users/1234567890"],
      },
    },
  },
  channels: {
    whatsapp: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["accessGroup:oncall"],
    },
    googlechat: {
      groups: {
        "spaces/AAA": {
          users: ["accessGroup:oncall"],
        },
      },
    },
  },
}
```

Группы можно сочетать с непосредственными записями:

```json5
{
  channels: {
    discord: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:operators", "discord:123456789012345678"],
    },
  },
}
```

## Поддерживаемые пути каналов сообщений

Группы доступа работают в общих путях авторизации каналов сообщений:

- списках разрешённых отправителей личных сообщений, например `channels.<channel>.allowFrom`;
- списках разрешённых отправителей групповых сообщений, например `channels.<channel>.groupAllowFrom`;
- специфичных для каналов списках разрешённых отправителей для отдельных комнат, использующих те же правила сопоставления отправителей (например, `groups.<space>.users` в Google Chat);
- путях авторизации команд, повторно использующих списки разрешённых отправителей каналов сообщений.

Поддержка зависит от того, подключён ли канал к общим вспомогательным функциям OpenClaw для авторизации отправителей. В настоящее время встроенная поддержка включает ClickClack, Discord, Feishu, Google Chat, iMessage, IRC, LINE, Mattermost, Microsoft Teams, Nextcloud Talk, Nostr, QQ Bot, Signal, Slack, SMS, Telegram, WhatsApp, Zalo и Zalo Personal. Статические группы `message.senders` не зависят от канала, поэтому новые каналы сообщений получают их поддержку при использовании общих вспомогательных функций входящих сообщений из SDK плагинов вместо собственной логики раскрытия списков разрешений.

## Аудитории каналов Discord

Discord также поддерживает динамический тип группы доступа:

```json5
{
  accessGroups: {
    maintainers: {
      type: "discord.channelAudience",
      guildId: "1456350064065904867",
      channelId: "1456744319972282449",
      membership: "canViewChannel",
    },
  },
  channels: {
    discord: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:maintainers"],
    },
  },
}
```

`discord.channelAudience` означает «разрешить личные сообщения от пользователей Discord, которые в данный момент могут просматривать этот канал сервера». OpenClaw проверяет отправителя через Discord во время авторизации и применяет правила разрешения Discord `ViewChannel`. Поле `membership` необязательно; по умолчанию используется `canViewChannel`.

Используйте этот вариант, когда канал Discord уже является источником достоверных данных о составе команды, например `#maintainers` или `#on-call`.

Требования и поведение при ошибках:

- Боту необходим доступ к серверу и каналу.
- Боту требуется **Server Members Intent** в Discord Developer Portal.
- Группа доступа запрещает доступ при ошибке, если Discord возвращает `Missing Access`, отправителя невозможно определить как участника сервера или канал принадлежит другому серверу.

Дополнительные примеры для Discord: [Управление доступом в Discord](/ru/channels/discord#access-control-and-routing)

## Диагностика плагинов

Авторы плагинов могут проверять структурированное состояние групп доступа, не раскрывая его обратно в плоский список разрешений:

```typescript
import { resolveAccessGroupAllowFromState } from "openclaw/plugin-sdk/access-groups";

const state = await resolveAccessGroupAllowFromState({
  accessGroups: cfg.accessGroups,
  allowFrom: channelConfig.allowFrom,
  channel: "my-channel",
  accountId: "default",
  senderId,
  isSenderAllowed,
});
```

Результат содержит сведения о группах, на которые имеются ссылки, а также о совпавших, отсутствующих, неподдерживаемых группах и группах, обработка которых завершилась ошибкой. Используйте его для диагностики или тестов на соответствие требованиям. Используйте `expandAllowFromWithAccessGroups(...)` только в путях совместимости, которые по-прежнему ожидают плоский массив `allowFrom`.

## Примечания по безопасности

- Группы доступа — это псевдонимы списков разрешений, а не роли. Сами по себе они не назначают владельцев, не одобряют запросы на сопряжение и не предоставляют разрешения на использование инструментов.
- Для `dmPolicy: "open"` по-прежнему требуется `"*"` в итоговом списке разрешений для личных сообщений. Ссылка на группу доступа не равнозначна общедоступному доступу.
- Отсутствующие имена групп приводят к запрету доступа. Если `allowFrom` содержит `accessGroup:operators`, а `accessGroups.operators` отсутствует, эта запись никого не авторизует.
- Обеспечьте стабильность идентификаторов каналов. Если канал поддерживает и идентификаторы, и отображаемые имена, предпочитайте числовые идентификаторы или идентификаторы пользователей.

## Устранение неполадок

Если отправитель должен соответствовать правилам, но заблокирован:

1. Убедитесь, что поле списка разрешений содержит точную ссылку `accessGroup:<name>`.
2. Убедитесь, что значение `accessGroups.<name>.type` указано правильно.
3. Убедитесь, что идентификатор отправителя указан под ключом соответствующего канала или под ключом `"*"`.
4. Убедитесь, что запись использует обычный синтаксис списка разрешений этого канала.
5. Для аудиторий каналов Discord убедитесь, что бот видит канал сервера и что **Server Members Intent** включён.

После изменения конфигурации управления доступом выполните `openclaw doctor`. Эта команда выявляет многие недопустимые сочетания списков разрешений и политик до запуска.
