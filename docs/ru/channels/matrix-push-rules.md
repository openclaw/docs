Когда для `channels.matrix.streaming` задано значение `"quiet"`, OpenClaw передаёт ответ потоково, редактируя одно событие предварительного просмотра на месте. Предварительные просмотры отправляются как события `m.notice` без уведомлений, а окончательная правка помечается с помощью `content["com.openclaw.finalized_preview"] = true`. Клиенты Matrix уведомляют об этой окончательной правке, только если пользовательское правило push-уведомлений соответствует маркеру. Эта страница предназначена для операторов, которые самостоятельно размещают Matrix и хотят установить это правило для каждой учётной записи получателя.

При `streaming: "progress"` черновики завершаются тем же способом, поэтому это же правило срабатывает и для окончательных правок в режиме прогресса.

Если вам нужно только стандартное поведение уведомлений Matrix, используйте `streaming: "partial"` или отключите потоковую передачу. См. [Настройка канала Matrix](/ru/channels/matrix#streaming-previews).

## Предварительные требования

- пользователь-получатель — человек, который должен получить уведомление
- пользователь-бот — учётная запись OpenClaw в Matrix, которая отправляет ответ
- для приведённых ниже вызовов API используйте токен доступа пользователя-получателя
- в правиле push-уведомлений сопоставляйте `sender` с полным MXID пользователя-бота
- для учётной записи получателя уже должны быть настроены работающие обработчики push-уведомлений; правила тихого предварительного просмотра работают только при исправной обычной доставке push-уведомлений Matrix

## Шаги

<Steps>
  <Step title="Настройте тихие предварительные просмотры">

```json5
{
  channels: {
    matrix: {
      streaming: "quiet",
    },
  },
}
```

  </Step>

  <Step title="Получите токен доступа получателя">
    По возможности повторно используйте существующий токен клиентского сеанса. Чтобы выпустить новый:

```bash
curl -sS -X POST \
  "https://matrix.example.org/_matrix/client/v3/login" \
  -H "Content-Type: application/json" \
  --data '{
    "type": "m.login.password",
    "identifier": { "type": "m.id.user", "user": "@alice:example.org" },
    "password": "REDACTED"
  }'
```

  </Step>

  <Step title="Убедитесь, что обработчики push-уведомлений существуют">

```bash
curl -sS \
  -H "Authorization: Bearer $USER_ACCESS_TOKEN" \
  "https://matrix.example.org/_matrix/client/v3/pushers"
```

Если обработчики push-уведомлений отсутствуют в ответе, прежде чем продолжить, исправьте обычную доставку push-уведомлений Matrix для этой учётной записи.

  </Step>

  <Step title="Установите переопределяющее правило push-уведомлений">
    Установите правило, которое сопоставляет маркер завершённого предварительного просмотра и MXID бота в качестве отправителя:

```bash
curl -sS -X PUT \
  "https://matrix.example.org/_matrix/client/v3/pushrules/global/override/openclaw-finalized-preview-botname" \
  -H "Authorization: Bearer $USER_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{
    "conditions": [
      { "kind": "event_match", "key": "type", "pattern": "m.room.message" },
      {
        "kind": "event_property_is",
        "key": "content.m\\.relates_to.rel_type",
        "value": "m.replace"
      },
      {
        "kind": "event_property_is",
        "key": "content.com\\.openclaw\\.finalized_preview",
        "value": true
      },
      { "kind": "event_match", "key": "sender", "pattern": "@bot:example.org" }
    ],
    "actions": [
      "notify",
      { "set_tweak": "sound", "value": "default" },
      { "set_tweak": "highlight", "value": false }
    ]
  }'
```

    Перед запуском замените:

    - `https://matrix.example.org`: базовый URL вашего домашнего сервера
    - `$USER_ACCESS_TOKEN`: токен доступа пользователя-получателя
    - `openclaw-finalized-preview-botname`: идентификатор правила, уникальный для каждой пары «бот — получатель» (шаблон: `openclaw-finalized-preview-<botname>`)
    - `@bot:example.org`: MXID вашего бота OpenClaw, а не получателя

  </Step>

  <Step title="Проверьте">

```bash
curl -sS \
  -H "Authorization: Bearer $USER_ACCESS_TOKEN" \
  "https://matrix.example.org/_matrix/client/v3/pushrules/global/override/openclaw-finalized-preview-botname"
```

Затем протестируйте потоковый ответ. В тихом режиме в комнате отображается тихий предварительный просмотр черновика, а уведомление отправляется один раз после завершения блока или хода.

  </Step>
</Steps>

Чтобы позднее удалить правило, отправьте запрос `DELETE` на тот же URL правила с токеном получателя.

## Примечания для нескольких ботов

Правила push-уведомлений индексируются по `ruleId`: повторный вызов `PUT` с тем же идентификатором обновляет одно правило. Чтобы несколько ботов OpenClaw уведомляли одного получателя, создайте отдельное правило для каждого бота с отдельным условием сопоставления отправителя.

Новые пользовательские правила `override` вставляются перед стандартными серверными правилами подавления, поэтому дополнительный параметр порядка не требуется. Правило влияет только на правки текстовых предварительных просмотров, которые можно завершить на месте; ответы с медиафайлами, резервные варианты для устаревших предварительных просмотров и окончательные тексты, которые активировали бы упоминания Matrix, вместо этого доставляются как обычные сообщения с уведомлениями.

## Примечания для домашнего сервера

<AccordionGroup>
  <Accordion title="Synapse">
    Специальные изменения в `homeserver.yaml` не требуются. Если обычные уведомления Matrix уже доходят до этого пользователя, основной этап настройки — вызов `pushrules` с приведённым выше токеном получателя.

    Если Synapse работает за обратным прокси-сервером или с рабочими процессами, убедитесь, что запросы к `/_matrix/client/.../pushrules/` корректно достигают Synapse. Доставку push-уведомлений обрабатывает основной процесс либо `synapse.app.pusher` / настроенные рабочие процессы push-уведомлений — убедитесь в их исправности.

    Правило использует условие правила push-уведомлений `event_property_is` (MSC3758, правило push-уведомлений версии 1.10), которое было добавлено в Synapse в 2023 году. Более старые выпуски Synapse принимают вызов `PUT pushrules/...`, но условие никогда не срабатывает без предупреждения — обновите Synapse, если при окончательной правке предварительного просмотра уведомление не приходит.

  </Accordion>

  <Accordion title="Tuwunel">
    Используется тот же процесс, что и для Synapse; для маркера завершённого предварительного просмотра не требуется конфигурация, специфичная для Tuwunel.

    Если уведомления исчезают, пока пользователь активен на другом устройстве, проверьте, включён ли параметр `suppress_push_when_active`. Tuwunel добавил этот параметр в версии 1.4.2 (сентябрь 2025 года), и он может намеренно подавлять push-уведомления на другие устройства, пока активно одно устройство.

  </Accordion>
</AccordionGroup>
