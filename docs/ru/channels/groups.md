OpenClaw применяет одинаковые правила групп ко всем каналам с поддержкой групп, включая Discord, iMessage, Matrix, Microsoft Teams, QQBot, Signal, Slack, Telegram, WhatsApp и Zalo.

Для постоянно активных комнат, которые должны предоставлять фоновый контекст, не создавая шума, если агент явно не отправляет видимое сообщение, см. [Фоновые события комнаты](/ru/channels/ambient-room-events).

## Введение для начинающих (2 минуты)

OpenClaw «живет» в ваших собственных учетных записях мессенджеров. Отдельного пользователя-бота WhatsApp нет: если **вы** состоите в группе, OpenClaw может видеть эту группу и отвечать в ней.

Поведение по умолчанию:

- Доступ к группам ограничен (`groupPolicy: "allowlist"`); отправители в группах блокируются, пока их не добавят в список разрешенных.
- Для ответов требуется упоминание, если вы не отключите проверку упоминаний для группы.
- Текст окончательного ответа автоматически публикуется в комнате (`visibleReplies: "automatic"`).

Иными словами, отправители из списка разрешенных могут активировать OpenClaw, упомянув его.

<Note>
**Кратко**

- **Доступ к личным сообщениям** управляется параметром `*.allowFrom`.
- **Доступ к группам** управляется параметром `*.groupPolicy` и списками разрешенных (`*.groups`, `*.groupAllowFrom`).
- **Запуск ответа** управляется проверкой упоминаний (`requireMention`, `/activation`).

</Note>

Краткая схема обработки группового сообщения:

```text
groupPolicy? disabled -> отбросить
groupPolicy? allowlist -> группа разрешена? нет -> отбросить
requireMention? да -> есть упоминание? нет -> сохранить только для контекста
упоминание/ответ/команда/личное сообщение -> запрос пользователя
переписка постоянно активной группы -> запрос пользователя или событие комнаты, если настроено
```

## Видимые ответы

Для обычных запросов в группах и каналах OpenClaw по умолчанию использует `messages.groupChat.visibleReplies: "automatic"`: окончательный текст ассистента публикуется в комнате как видимый ответ.

Используйте `messages.groupChat.visibleReplies: "message_tool"`, если в общей комнате агент должен сам решать, когда говорить, вызывая `message(action=send)`. Лучше всего этот режим работает с моделями, надежно использующими инструменты (например, GPT-5.6 Sol). Если модель не вызовет инструмент и вернет содержательный окончательный текст, OpenClaw сохранит этот текст приватным и не опубликует его в комнате.

Используйте `"automatic"` для моделей и сред выполнения, которые ненадежно соблюдают требование доставки только через инструмент: обычные окончательные текстовые ответы публикуются непосредственно в комнате, а агент по-прежнему может вызывать `message(action=send)` для файлов, изображений и других вложений, которые нельзя передать вместе с окончательным текстом.

Если инструмент сообщений недоступен согласно действующей политике инструментов, OpenClaw переключается на автоматические видимые ответы вместо того, чтобы без уведомления подавлять ответ. `openclaw doctor` предупреждает об этом несоответствии.

Для личных чатов и любых других исходных событий параметр `messages.visibleReplies: "message_tool"` применяет такое же поведение с обязательным использованием инструмента глобально; `messages.groupChat.visibleReplies` остается более специфичным переопределением для групповых и канальных комнат. Во внутренних личных диалогах WebChat окончательные ответы по умолчанию доставляются автоматически, чтобы Pi и Codex использовали одинаковый контракт видимых ответов.

Режим доставки только через инструмент заменяет прежний подход, при котором модель принудительно отвечала `NO_REPLY` в большинстве случаев пассивного наблюдения. В этом режиме запрос не определяет контракт `NO_REPLY`: отсутствие видимых действий означает лишь то, что инструмент сообщений не был вызван.

Привязки бесед, принадлежащие Plugin, являются исключением. После того как Plugin привязывает ветку и принимает входящий запрос, возвращенный Plugin ответ становится видимым ответом привязки; для него не требуется `message(action=send)`. Этот ответ является результатом среды выполнения Plugin, а не приватным окончательным текстом модели.

Индикаторы набора текста по-прежнему отправляются для прямых групповых запросов. Фоновые события постоянно активных комнат, если они включены, работают строго и незаметно, пока агент не вызовет инструмент сообщений.

По умолчанию в сеансах подавляются подробные сводки об инструментах и ходе выполнения. Используйте `/verbose on` (или `/verbose full`), чтобы показывать их в текущем сеансе во время отладки, и `/verbose off`, чтобы вернуться к отображению только окончательных ответов. Состояние подробного режима задается отдельно для каждого сеанса и одинаково работает в личных чатах, группах, каналах и темах форумов.

Чтобы отправлять переписку постоянно активной группы без упоминаний как ненавязчивый контекст комнаты, а не как запросы пользователя, используйте [Фоновые события комнаты](/ru/channels/ambient-room-events):

```json5
{
  messages: {
    groupChat: {
      unmentionedInbound: "room_event",
    },
  },
}
```

Значение по умолчанию — `unmentionedInbound: "user_request"`. Сообщения с упоминаниями, команды, запросы на прерывание и личные сообщения остаются запросами пользователя.

Чтобы для видимого вывода в ответ на запросы групп и каналов требовалось использование инструмента сообщений:

```json5
{
  messages: {
    groupChat: {
      visibleReplies: "message_tool",
    },
  },
}
```

Чтобы требовать этого для каждого исходного чата:

```json5
{
  messages: {
    visibleReplies: "message_tool",
  },
}
```

После сохранения файла Gateway применяет изменения конфигурации `messages` без перезапуска. Перезапуск требуется только при отключенной перезагрузке конфигурации (`gateway.reload.mode: "off"`).

Командные запросы обходят `visibleReplies: "message_tool"` и всегда получают видимый ответ: как встроенные команды с косой чертой (в Discord, Telegram и других интерфейсах со встроенной поддержкой команд), так и авторизованные текстовые команды `/...` публикуют ответ в исходном чате. Неавторизованные текстовые запросы `/...` в группах по-прежнему требуют инструмента сообщений; обычные запросы чата следуют настроенному поведению по умолчанию.

## Видимость контекста и списки разрешенных

За безопасность групп отвечают два разных механизма:

- **Авторизация запуска**: кто может активировать агента (`groupPolicy`, `groups`, `groupAllowFrom`, списки разрешенных для отдельных каналов).
- **Видимость контекста**: какой дополнительный контекст передается модели (текст ответа или цитаты, история ветки, метаданные пересылки).

По умолчанию OpenClaw сохраняет контекст в полученном виде: списки разрешенных определяют, кто может запускать действия, а не то, какие цитаты или фрагменты истории видит модель. Чтобы также фильтровать дополнительный контекст, задайте `contextVisibility`:

| Режим               | Поведение                                                                                                   |
| ------------------- | ----------------------------------------------------------------------------------------------------------- |
| `"all"` (по умолчанию) | Сохранять дополнительный контекст в полученном виде.                                                     |
| `"allowlist"`       | Передавать историю, ветки, цитаты и пересланный контекст только от отправителей из списка разрешенных.      |
| `"allowlist_quote"` | Как `allowlist`, но также сохранять явно процитированное сообщение или сообщение, на которое дан ответ, от любого отправителя. |

Задайте его для канала (`channels.<channel>.contextVisibility`), учетной записи (`channels.<channel>.accounts.<accountId>.contextVisibility`) или глобально (`channels.defaults.contextVisibility`). Каналы, получающие дополнительный контекст (Discord, Feishu, iMessage, Matrix, Microsoft Teams, Signal, Slack, Telegram, WhatsApp), применяют эту политику при формировании входящего контекста; при неизвестных сочетаниях политик они безопасно отказывают в доступе и исключают контекст.

## Ключи сеансов

- Групповые сеансы используют ключи вида `agent:<agentId>:<channel>:group:<id>` (комнаты и каналы используют `agent:<agentId>:<channel>:channel:<id>`).
- Темы форумов Telegram добавляют `:topic:<threadId>` к идентификатору группы, поэтому каждая тема получает собственный сеанс.
- Личные чаты используют основной сеанс (или отдельные сеансы для каждого отправителя, если настроен `session.dmScope`).
- Heartbeat выполняются в настроенном сеансе Heartbeat (по умолчанию — в основном сеансе агента); групповые сеансы не запускают собственные Heartbeat.

## Схема: личные сообщения + публичные группы (один агент)

Да — это хорошо работает, если ваш «личный» трафик состоит из **личных сообщений**, а «публичный» — из **групп**.

Причина: в режиме одного агента личные сообщения обычно попадают в ключ **основного** сеанса (`agent:main:main`), тогда как группы всегда используют ключи **неосновных** сеансов (`agent:main:<channel>:group:<id>`). Если включить песочницу с `mode: "non-main"`, эти групповые сеансы будут выполняться в настроенной среде песочницы, а основной сеанс личных сообщений останется на основной системе. Если вы не выберете другую среду, по умолчанию используется Docker.

Так вы получаете один «мозг» агента (общее рабочее пространство и память), но два режима выполнения:

- **Личные сообщения**: полный набор инструментов (основная система)
- **Группы**: песочница и ограниченный набор инструментов

<Note>
Если вам нужны полностью раздельные рабочие пространства или роли («личное» и «публичное» никогда не должны смешиваться), используйте второго агента и привязки. См. [Маршрутизация между несколькими агентами](/ru/concepts/multi-agent).
</Note>

## Отображаемые метки

- Метки интерфейса используют `displayName`, если оно доступно, в формате `<channel>:<token>`.
- `#room` зарезервировано для комнат и каналов; групповые чаты используют `g-<slug>` (нижний регистр, пробелы -> `-`, символы `#@+._-` сохраняются). Очень длинные непрозрачные идентификаторы сокращаются до стабильного токена, чтобы полные идентификаторы маршрутов не отображались в интерфейсе.

## Политика групп

Управляйте обработкой сообщений групп и комнат отдельно для каждого канала:

```json5
{
  channels: {
    whatsapp: {
      groupPolicy: "disabled", // "open" | "disabled" | "allowlist"
      groupAllowFrom: ["+15551234567"],
    },
    telegram: {
      groupPolicy: "disabled",
      groupAllowFrom: ["123456789"], // числовой идентификатор пользователя Telegram (при настройке разрешается @username)
    },
    signal: {
      groupPolicy: "disabled",
      groupAllowFrom: ["+15551234567"],
    },
    imessage: {
      groupPolicy: "disabled",
      groupAllowFrom: ["chat_id:123"],
    },
    msteams: {
      groupPolicy: "disabled",
      groupAllowFrom: ["user@org.com"],
    },
    discord: {
      groupPolicy: "allowlist",
      guilds: {
        GUILD_ID: { channels: { help: { enabled: true } } },
      },
    },
    slack: {
      groupPolicy: "allowlist",
      channels: { "#general": { enabled: true } },
    },
    matrix: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["@owner:example.org"],
      groups: {
        "!roomId:example.org": { enabled: true },
        "#alias:example.org": { enabled: true },
      },
    },
  },
}
```

| Политика       | Поведение                                                                        |
| -------------- | -------------------------------------------------------------------------------- |
| `"open"`       | Группы обходят списки разрешений; проверка упоминаний по-прежнему применяется.    |
| `"disabled"`   | Полностью блокирует все групповые сообщения.                                     |
| `"allowlist"`  | Разрешает только группы/комнаты, соответствующие настроенному списку разрешений.  |

## Проверка упоминаний (по умолчанию)

Для групповых сообщений требуется упоминание, если это не переопределено для конкретной группы. Значения по умолчанию для каждой подсистемы находятся в `*.groups."*"`.

Ответ на сообщение бота считается неявным упоминанием, когда канал предоставляет метаданные ответа; цитирование сообщения бота также может считаться упоминанием в каналах, предоставляющих метаданные цитирования. Текущие встроенные случаи: Discord, Microsoft Teams, QQBot, Slack, Telegram, WhatsApp и личный канал Zalo.

```json5
{
  channels: {
    whatsapp: {
      groups: {
        "*": { requireMention: true },
        "123@g.us": { requireMention: false },
      },
    },
    telegram: {
      groups: {
        "*": { requireMention: true },
        "123456789": { requireMention: false },
      },
    },
    imessage: {
      groups: {
        "*": { requireMention: true },
        "123": { requireMention: false },
      },
    },
  },
  agents: {
    list: [
      {
        id: "main",
        groupChat: {
          mentionPatterns: ["@openclaw", "openclaw", "\\+15555550123"],
          historyLimit: 50,
        },
      },
    ],
  },
}
```

## Область действия настроенных шаблонов упоминаний

Настроенные `mentionPatterns` — это резервные триггеры на основе регулярных выражений. Используйте их, когда платформа не предоставляет нативное упоминание бота или когда обычный текст, например `openclaw:`, должен считаться упоминанием. Нативные упоминания платформы обрабатываются отдельно: если Discord, Slack, Telegram, Matrix или другой канал может подтвердить, что в сообщении явно упомянут бот, такое нативное упоминание по-прежнему срабатывает, даже если настроенные шаблоны регулярных выражений запрещены.

По умолчанию настроенные шаблоны упоминаний применяются везде, где канал передаёт сведения о провайдере и беседе в механизм обнаружения упоминаний. Чтобы широкие шаблоны не активировали агента в каждой группе, ограничьте область их действия для каждого канала с помощью `channels.<channel>.mentionPatterns`.

Используйте `mode: "deny"`, когда шаблоны упоминаний на основе регулярных выражений должны быть по умолчанию отключены для канала, а затем включите их для конкретных комнат с помощью `allowIn`:

```json5
{
  messages: {
    groupChat: {
      mentionPatterns: ["\\bopenclaw\\b", "\\bops bot\\b"],
    },
  },
  channels: {
    slack: {
      mentionPatterns: {
        mode: "deny",
        allowIn: ["C0123OPS"],
      },
    },
  },
}
```

Используйте стандартный `mode: "allow"` (или не указывайте `mode`), когда шаблоны упоминаний на основе регулярных выражений должны применяться широко, а затем отключите их в шумных комнатах с помощью `denyIn`:

```json5
{
  messages: {
    groupChat: {
      mentionPatterns: ["\\bopenclaw\\b"],
    },
  },
  channels: {
    telegram: {
      mentionPatterns: {
        denyIn: ["-1001234567890", "-1001234567890:topic:42"],
      },
    },
  },
}
```

Разрешение политики:

| Поле            | Действие                                                                                                                                               |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `mode: "allow"` | Шаблоны упоминаний на основе регулярных выражений включены, если идентификатор беседы отсутствует в `denyIn`. Это значение по умолчанию.                 |
| `mode: "deny"`  | Шаблоны упоминаний на основе регулярных выражений отключены, если идентификатор беседы отсутствует в `allowIn`.                                         |
| `allowIn`       | Идентификаторы бесед, в которых шаблоны упоминаний на основе регулярных выражений включены в режиме запрета.                                            |
| `denyIn`        | Идентификаторы бесед, в которых шаблоны упоминаний на основе регулярных выражений отключены. Если один идентификатор есть в обоих списках, побеждает `denyIn`. |

В настоящее время поддерживается следующая политика области действия регулярных выражений:

| Канал    | Идентификаторы, используемые в `allowIn` / `denyIn`                         |
| -------- | -------------------------------------------------------------------------- |
| Discord  | Идентификаторы каналов Discord.                                             |
| Matrix   | Идентификаторы комнат Matrix.                                               |
| Slack    | Идентификаторы каналов Slack.                                               |
| Telegram | Идентификаторы групповых чатов или `chatId:topic:threadId` для тем форума. |
| WhatsApp | Идентификаторы бесед WhatsApp, например `123@g.us`.                        |

Конфигурации каналов на уровне учётной записи могут задавать ту же политику в `channels.<channel>.accounts.<accountId>.mentionPatterns`, если канал поддерживает несколько учётных записей. Политика учётной записи имеет приоритет над политикой верхнего уровня канала для этой учётной записи.

<AccordionGroup>
  <Accordion title="Примечания о проверке упоминаний">
    - `mentionPatterns` — это безопасные регулярные выражения без учёта регистра; недопустимые шаблоны и небезопасные формы с вложенными повторениями игнорируются (с предупреждением).
    - Приоритет шаблонов: `agents.list[].groupChat.mentionPatterns` (полезно, когда несколько агентов находятся в одной группе) переопределяет `messages.groupChat.mentionPatterns`; если ни один из них не задан, шаблоны выводятся из имени/эмодзи идентичности агента.
    - Проверка упоминаний применяется только тогда, когда обнаружение упоминаний возможно (доступны нативные упоминания или настроены `mentionPatterns`).
    - Добавление группы или отправителя в список разрешений не отключает проверку упоминаний; задайте для `requireMention` этой группы значение `false`, если все сообщения должны запускать агента.
    - Автоматический контекст запроса группового чата при каждом обращении содержит определённую инструкцию о беззвучном ответе; файлы рабочего пространства не должны дублировать механику `NO_REPLY`.
    - В группах, где разрешены автоматические беззвучные ответы, пустые результаты модели или результаты, содержащие только рассуждения, считаются беззвучными и эквивалентны `NO_REPLY`. Личные чаты никогда не получают указания `NO_REPLY`, а групповые ответы, использующие только инструмент сообщений, остаются беззвучными, если не вызывают `message(action=send)`.
    - Для постоянно активного фонового общения в группах по умолчанию используется семантика пользовательского запроса. Задайте `messages.groupChat.unmentionedInbound: "room_event"`, чтобы вместо этого отправлять его как тихий контекст. Примеры настройки см. в разделе [Фоновые события комнат](/ru/channels/ambient-room-events).
    - События комнат не сохраняются как фиктивные пользовательские запросы, а приватный текст ассистента из событий комнат без инструмента сообщений не воспроизводится в истории чата.
    - Значения Discord по умолчанию находятся в `channels.discord.guilds."*"` (их можно переопределить для каждой гильдии/канала).
    - Контекст истории групп оформляется единообразно во всех каналах. Группы с проверкой упоминаний сохраняют ожидающие пропущенные сообщения; постоянно активные группы также могут сохранять недавние обработанные сообщения комнаты, если канал это поддерживает. Используйте `messages.groupChat.historyLimit` для глобального значения по умолчанию и `channels.<channel>.historyLimit` (или `channels.<channel>.accounts.*.historyLimit`) для переопределений. Задайте `0`, чтобы отключить.

## Ограничения инструментов для групп/каналов (необязательно)

Некоторые конфигурации каналов позволяют ограничивать доступность инструментов **внутри конкретной группы/комнаты/канала**.

- `tools`: разрешает/запрещает инструменты для всей группы (`allow`, `alsoAllow`, `deny`; запрет имеет приоритет).
- `toolsBySender`: переопределения для отдельных отправителей внутри группы. Используйте явные префиксы ключей: `channel:<channelId>:<senderId>`, `id:<senderId>`, `e164:<phone>`, `username:<handle>`, `name:<displayName>` и подстановочный знак `"*"`. Идентификаторы каналов используют канонические идентификаторы каналов OpenClaw; псевдонимы, например `teams`, нормализуются в `msteams`. Устаревшие ключи без префиксов по-прежнему принимаются, сопоставляются только как `id:` и вызывают запись предупреждения об устаревании в журнал.

Порядок разрешения (наиболее конкретное правило имеет приоритет):

<Steps>
  <Step title="Групповой toolsBySender">
    Совпадение `toolsBySender` группы/канала.
  </Step>
  <Step title="Групповой tools">
    `tools` группы/канала.
  </Step>
  <Step title="Стандартный toolsBySender">
    Совпадение стандартного (`"*"`) `toolsBySender`.
  </Step>
  <Step title="Стандартный tools">
    Стандартный (`"*"`) `tools`.
  </Step>
</Steps>

Пример (Telegram):

```json5
{
  channels: {
    telegram: {
      groups: {
        "*": { tools: { deny: ["exec"] } },
        "-1001234567890": {
          tools: { deny: ["exec", "read", "write"] },
          toolsBySender: {
            "id:123456789": { alsoAllow: ["exec"] },
          },
        },
      },
    },
  },
}
```

<Note>
Ограничения инструментов для групп/каналов применяются в дополнение к глобальной политике инструментов или политике агента (запрет по-прежнему имеет приоритет). Некоторые каналы используют другую вложенность для комнат/каналов (например, Discord — `guilds.*.channels.*`, Slack — `channels.*`, Microsoft Teams — `teams.*.channels.*`).
</Note>

## Групповые списки разрешений

Если настроены `channels.whatsapp.groups`, `channels.telegram.groups` или `channels.imessage.groups`, их ключи действуют как список разрешённых групп. Используйте `"*"`, чтобы разрешить все группы, сохранив при этом стандартное поведение упоминаний.

<Warning>
Распространённая ошибка: одобрение сопряжения в личных сообщениях — не то же самое, что авторизация в группе. Для каналов, поддерживающих сопряжение в личных сообщениях, хранилище сопряжений разрешает только личные сообщения. Для групповых команд по-прежнему требуется явная авторизация отправителя в группе через списки разрешённых значений в конфигурации, например `groupAllowFrom`, или документированную резервную настройку конфигурации для этого канала.
</Warning>

Типовые задачи (можно скопировать и вставить):

<Tabs>
  <Tab title="Отключить все ответы в группах">
    ```json5
    {
      channels: { whatsapp: { groupPolicy: "disabled" } },
    }
    ```
  </Tab>
  <Tab title="Разрешить только определённые группы (WhatsApp)">
    ```json5
    {
      channels: {
        whatsapp: {
          groups: {
            "123@g.us": { requireMention: true },
            "456@g.us": { requireMention: false },
          },
        },
      },
    }
    ```
  </Tab>
  <Tab title="Разрешить все группы, но требовать упоминание">
    ```json5
    {
      channels: {
        whatsapp: {
          groups: { "*": { requireMention: true } },
        },
      },
    }
    ```
  </Tab>
  <Tab title="Активация только владельцем (WhatsApp)">
    ```json5
    {
      channels: {
        whatsapp: {
          groupPolicy: "allowlist",
          groupAllowFrom: ["+15551234567"],
          groups: { "*": { requireMention: true } },
        },
      },
    }
    ```
  </Tab>
</Tabs>

## Активация (только владельцем)

Владельцы групп могут переключать режим активации для каждой группы отдельным сообщением:

- `/activation mention`
- `/activation always`

`/activation` — основная команда, доступная только владельцу и применимая исключительно в групповых чатах. Владельцем считается отправитель, соответствующий `allowFrom` / `commands.ownerAllowFrom` канала (если список разрешённых значений не настроен, владельцем считается учётная запись с собственным идентификатором). Сохранённый режим переопределяет параметр `requireMention` этой группы в каналах, которые его учитывают (Google Chat, QQBot, Telegram, WhatsApp), а вводная часть системного запроса для группы везде отражает активный режим.

## Поля контекста

Входящие данные группы задают:

- `ChatType=group`
- `GroupSubject` (если известна)
- `GroupMembers` (если известны)
- `WasMentioned` (результат проверки требования упоминания)
- Темы форумов Telegram также содержат `MessageThreadId` и `IsForum`.

Системный запрос агента содержит вводную часть для группы на первом ходе нового группового сеанса (и после изменений `/activation`). Она напоминает модели отвечать как человек, сводить количество пустых строк к минимуму, соблюдать обычные интервалы в чате и не вводить буквальные последовательности `\n`. В группах, кроме Telegram, также не рекомендуется использовать таблицы Markdown; рекомендации по форматированному тексту Telegram задаются запросом канала Telegram. Полученные от канала названия групп и подписи участников отображаются как недоверенные метаданные в блоках кода, а не как встроенные системные инструкции.

## Особенности iMessage

- Для маршрутизации или добавления в список разрешённых значений предпочтительно использовать `chat_id:<id>`.
- Вывести список чатов: `imsg chats --limit 20`.
- Ответы в группе всегда отправляются обратно в тот же `chat_id`.
