"Сопряжение" — это явный шаг подтверждения доступа в OpenClaw.
Оно используется в двух местах:

1. **Сопряжение DM** (кому разрешено общаться с ботом)
2. **Сопряжение Node** (каким устройствам/узлам разрешено подключаться к сети Gateway)

Контекст безопасности: [Безопасность](/ru/gateway/security)

## 1) Сопряжение DM (доступ входящего чата)

Когда канал настроен с политикой DM `pairing`, неизвестные отправители получают короткий код, а их сообщение **не обрабатывается**, пока вы не подтвердите доступ.

Политики DM по умолчанию документированы здесь: [Безопасность](/ru/gateway/security)

`dmPolicy: "open"` является публичной только когда эффективный список разрешенных DM включает `"*"`.
Настройка и проверка требуют этот wildcard для публично-открытых конфигураций. Если существующее
состояние содержит `open` с конкретными записями `allowFrom`, во время выполнения по-прежнему допускаются
только эти отправители, а подтверждения из хранилища сопряжений не расширяют доступ `open`.

Коды сопряжения:

- 8 символов, в верхнем регистре, без неоднозначных символов (`0O1I`).
- **Истекают через 1 час**. Бот отправляет сообщение о сопряжении только при создании нового запроса (примерно один раз в час на отправителя).
- Ожидающие запросы сопряжения DM по умолчанию ограничены **3 на канал**; дополнительные запросы игнорируются, пока один из них не истечет или не будет подтвержден.

### Подтвердить отправителя

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

Если владелец команд еще не настроен, подтверждение кода сопряжения DM также инициализирует
`commands.ownerAllowFrom` подтвержденным отправителем, например `telegram:123456789`.
Это дает первоначальной настройке явного владельца для привилегированных команд и запросов подтверждения
exec. После появления владельца последующие подтверждения сопряжения предоставляют только доступ
DM; они не добавляют новых владельцев.

Поддерживаемые каналы: `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### Повторно используемые группы отправителей

Используйте верхнеуровневые `accessGroups`, когда один и тот же набор доверенных отправителей должен применяться к
нескольким каналам сообщений или одновременно к спискам разрешенных DM и групп.

Статические группы используют `type: "message.senders"` и ссылаются через
`accessGroup:<name>` из списков разрешенных каналов:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

Группы доступа подробно документированы здесь: [Группы доступа](/ru/channels/access-groups)

### Где хранится состояние

Хранится в `~/.openclaw/credentials/`:

- Ожидающие запросы: `<channel>-pairing.json`
- Хранилище подтвержденного списка разрешенных:
  - Учетная запись по умолчанию: `<channel>-allowFrom.json`
  - Учетная запись не по умолчанию: `<channel>-<accountId>-allowFrom.json`

Поведение области действия учетной записи:

- Учетные записи не по умолчанию читают/записывают только свой scoped-файл списка разрешенных.
- Учетная запись по умолчанию использует channel-scoped файл списка разрешенных без области.

Считайте эти данные чувствительными (они управляют доступом к вашему ассистенту).

<Note>
Хранилище списка разрешенных сопряжения предназначено для доступа DM. Авторизация групп выполняется отдельно.
Подтверждение кода сопряжения DM не разрешает этому отправителю автоматически выполнять групповые
команды или управлять ботом в группах. Инициализация первого владельца — это отдельное состояние конфигурации
в `commands.ownerAllowFrom`, а доставка группового чата по-прежнему следует
спискам разрешенных групп канала (например, `groupAllowFrom`, `groups` или переопределениям
для отдельных групп или тем, в зависимости от канала).
</Note>

## 2) Сопряжение устройства Node (iOS/Android/macOS/headless-узлы)

Узлы подключаются к Gateway как **устройства** с `role: node`. Gateway
создает запрос сопряжения устройства, который нужно подтвердить.

### Сопряжение из Control UI (рекомендуется)

Используйте уже подключенную сессию Control UI с доступом `operator.admin`:

1. Откройте Control UI и выберите **Узлы**.
2. В разделе **Устройства** нажмите **Сопрячь мобильное устройство**.
3. На телефоне откройте приложение OpenClaw → **Настройки** → **Gateway**.
4. Отсканируйте QR-код или вставьте код настройки, затем подключитесь.

Официальные приложения OpenClaw для iOS и Android подтверждаются автоматически, когда их
метаданные кода настройки совпадают. Если в **Устройствах** показан ожидающий запрос (например,
для неофициального клиента или при несовпадающих метаданных), проверьте его роль и
области доступа перед подтверждением.

Кнопка отключена, если текущая сессия Control UI не имеет
администраторского доступа. В таком случае используйте поток подтверждения CLI ниже с хоста Gateway.

### Сопряжение через Telegram

Если вы используете Plugin `device-pair`, первое сопряжение устройства можно выполнить полностью из Telegram:

1. В Telegram отправьте боту сообщение: `/pair`
2. Бот отвечает двумя сообщениями: сообщением с инструкцией и отдельным сообщением с **кодом настройки** (его легко скопировать/вставить в Telegram).
3. На телефоне откройте приложение OpenClaw для iOS → Настройки → Gateway.
4. Отсканируйте QR-код или вставьте код настройки и подключитесь.
5. Официальное мобильное приложение подключается автоматически. Если `/pair pending` показывает
   запрос, проверьте его роль и области доступа перед подтверждением.

Код настройки — это JSON-полезная нагрузка в кодировке base64, которая содержит:

- `url`: URL WebSocket Gateway (`ws://...` или `wss://...`)
- `bootstrapToken`: краткоживущий bootstrap-токен для одного устройства, используемый для первоначального рукопожатия сопряжения

Этот bootstrap-токен несет встроенный bootstrap-профиль сопряжения:

- встроенный профиль настройки разрешает только базовый вариант свежего QR/кода настройки:
  `node` плюс ограниченную передачу `operator`
- переданный токен `node` остается `scopes: []`
- переданный токен `operator` ограничен `operator.approvals`,
  `operator.read`, `operator.talk.secrets` и `operator.write`
- `operator.admin` не предоставляется bootstrap через QR/код настройки; для него требуется
  отдельное подтвержденное сопряжение operator или поток токена
- последующие ротация/отзыв токенов остаются ограничены как подтвержденным
  контрактом роли устройства, так и operator-областями сессии вызывающего

Считайте код настройки паролем, пока он действителен.

Для сопряжения мобильного устройства через Tailscale, публичную сеть или другой удаленный доступ используйте Tailscale Serve/Funnel
или другой URL Gateway `wss://`. Открытые текстовые коды настройки `ws://` принимаются только
для loopback, частных LAN-адресов, хостов Bonjour `.local` и хоста эмулятора Android.
Адреса CGNAT tailnet, имена `.ts.net` и публичные хосты по-прежнему
fail-closed до выпуска QR/кода настройки.

### Подтвердить устройство Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

Когда явное подтверждение отклоняется из-за того, что сессия сопряженного устройства,
выполняющая подтверждение, была открыта только с областью сопряжения, CLI повторяет тот же запрос с
`operator.admin`. Это позволяет существующему сопряженному устройству с правами администратора восстановить новое
сопряжение Control UI/браузера без ручного редактирования `devices/paired.json`. Gateway
по-прежнему проверяет повторное подключение; токены, которые не могут пройти аутентификацию
с `operator.admin`, остаются заблокированными.

Если то же устройство повторяет попытку с другими данными аутентификации (например, другой
ролью/областями доступа/публичным ключом), предыдущий ожидающий запрос заменяется и создается новый
`requestId`.

<Note>
Уже сопряженное устройство не получает более широкий доступ незаметно. Если оно переподключается с запросом дополнительных областей доступа или более широкой роли, OpenClaw сохраняет существующее подтверждение как есть и создает новый ожидающий запрос на повышение доступа. Используйте `openclaw devices list`, чтобы сравнить текущий подтвержденный доступ с новым запрошенным доступом перед подтверждением.
</Note>

### Необязательное автоодобрение Node по доверенным CIDR

По умолчанию сопряжение устройств остается ручным. Для строго контролируемых сетей узлов
можно включить автоодобрение первого сопряжения Node с явными CIDR или точными IP-адресами:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Это применяется только к новым запросам сопряжения `role: node` без запрошенных
областей доступа. Клиенты Operator, браузера, Control UI и WebChat по-прежнему требуют ручного
подтверждения. Изменения роли, области доступа, метаданных и публичного ключа также требуют ручного
подтверждения.

### Хранилище состояния сопряжения Node

Хранится в `~/.openclaw/devices/`:

- `pending.json` (краткоживущее; ожидающие запросы истекают)
- `paired.json` (сопряженные устройства + токены)

### Примечания

- Устаревший API `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) — это
  отдельное хранилище сопряжений, принадлежащее gateway. Узлы WS по-прежнему требуют сопряжения устройств.
- Запись сопряжения является долговечным источником истины для подтвержденных ролей. Активные
  токены устройств остаются ограничены этим подтвержденным набором ролей; случайная запись токена
  вне подтвержденных ролей не создает новый доступ.
