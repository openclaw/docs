«Сопряжение» — явный шаг подтверждения доступа в OpenClaw.
Оно используется в двух местах:

1. **Сопряжение DM** (кому разрешено общаться с ботом)
2. **Сопряжение Node** (каким устройствам/узлам разрешено подключаться к сети Gateway)

Контекст безопасности: [Безопасность](/ru/gateway/security)

## 1) Сопряжение DM (доступ входящего чата)

Когда канал настроен с политикой DM `pairing`, неизвестные отправители получают короткий код, а их сообщение **не обрабатывается**, пока вы не подтвердите доступ.

Политики DM по умолчанию описаны здесь: [Безопасность](/ru/gateway/security)

`dmPolicy: "open"` является публичной только тогда, когда эффективный список разрешений DM включает `"*"`.
Настройка и проверка требуют этот шаблон для публично открытых конфигураций. Если существующее
состояние содержит `open` с конкретными записями `allowFrom`, среда выполнения по-прежнему допускает
только этих отправителей, а подтверждения в хранилище сопряжений не расширяют доступ `open`.

Коды сопряжения:

- 8 символов, в верхнем регистре, без неоднозначных символов (`0O1I`).
- **Истекают через 1 час**. Бот отправляет сообщение о сопряжении только при создании нового запроса (примерно раз в час для каждого отправителя).
- Ожидающие запросы на сопряжение DM по умолчанию ограничены **3 на канал**; дополнительные запросы игнорируются, пока один из них не истечет или не будет подтвержден.

### Подтвердить отправителя

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

Если владелец команд еще не настроен, подтверждение кода сопряжения DM также инициализирует
`commands.ownerAllowFrom` утвержденным отправителем, например `telegram:123456789`.
Это дает первоначальным настройкам явного владельца для привилегированных команд и запросов
подтверждения выполнения. После появления владельца последующие подтверждения сопряжения дают только
доступ DM; они не добавляют новых владельцев.

Поддерживаемые каналы: `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### Переиспользуемые группы отправителей

Используйте верхнеуровневые `accessGroups`, когда один и тот же набор доверенных отправителей должен применяться к
нескольким каналам сообщений или одновременно к спискам разрешений DM и групп.

Статические группы используют `type: "message.senders"` и указываются как
`accessGroup:<name>` в списках разрешений канала:

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
        whatsapp: ["+15551234567"],
      },
    },
  },
  channels: {
    telegram: { dmPolicy: "allowlist", allowFrom: ["accessGroup:operators"] },
    whatsapp: { groupPolicy: "allowlist", groupAllowFrom: ["accessGroup:operators"] },
  },
}
```

Группы доступа подробно описаны здесь: [Группы доступа](/ru/channels/access-groups)

### Где хранится состояние

Хранится в `~/.openclaw/credentials/`:

- Ожидающие запросы: `<channel>-pairing.json`
- Хранилище утвержденного списка разрешений:
  - Учетная запись по умолчанию: `<channel>-allowFrom.json`
  - Учетная запись не по умолчанию: `<channel>-<accountId>-allowFrom.json`

Поведение области действия учетных записей:

- Учетные записи не по умолчанию читают/записывают только свой файл списка разрешений с областью действия.
- Учетная запись по умолчанию использует файл списка разрешений канала без области действия.

Считайте эти данные чувствительными (они ограничивают доступ к вашему ассистенту).

<Note>
Хранилище списка разрешений сопряжения предназначено для доступа DM. Авторизация групп отделена.
Подтверждение кода сопряжения DM не дает этому отправителю автоматического разрешения выполнять групповые
команды или управлять ботом в группах. Первичная инициализация владельца — отдельное состояние конфигурации
в `commands.ownerAllowFrom`, а доставка групповых чатов по-прежнему следует групповым спискам разрешений
канала (например `groupAllowFrom`, `groups` или переопределениям для отдельных групп
или тем, в зависимости от канала).
</Note>

## 2) Сопряжение устройств Node (iOS/Android/macOS/узлы без интерфейса)

Node подключаются к Gateway как **устройства** с `role: node`. Gateway
создает запрос на сопряжение устройства, который нужно подтвердить.

### Сопряжение через Telegram (рекомендуется для iOS)

Если вы используете Plugin `device-pair`, первичное сопряжение устройства можно полностью выполнить из Telegram:

1. В Telegram отправьте сообщение вашему боту: `/pair`
2. Бот отвечает двумя сообщениями: сообщением с инструкциями и отдельным сообщением с **кодом настройки** (его удобно копировать/вставлять в Telegram).
3. На телефоне откройте приложение OpenClaw для iOS → Settings → Gateway.
4. Отсканируйте QR-код или вставьте код настройки и подключитесь.
5. Вернитесь в Telegram: `/pair pending` (проверьте идентификаторы запросов, роль и области доступа), затем подтвердите.

Код настройки — это JSON-полезная нагрузка в кодировке base64, которая содержит:

- `url`: URL WebSocket для Gateway (`ws://...` или `wss://...`)
- `bootstrapToken`: короткоживущий токен инициализации для одного устройства, используемый для первоначального рукопожатия сопряжения

Этот токен инициализации несет встроенный профиль инициализации сопряжения:

- встроенный профиль настройки разрешает только базовый вариант свежего QR-кода/кода настройки:
  `node` плюс ограниченную передачу `operator`
- переданный токен `node` остается `scopes: []`
- переданный токен `operator` ограничен `operator.approvals`,
  `operator.read`, `operator.talk.secrets` и `operator.write`
- `operator.admin` не выдается через инициализацию QR-кодом/кодом настройки; для него требуется
  отдельное утвержденное сопряжение оператора или поток токена
- последующая ротация/отзыв токенов остаются ограничены как утвержденным
  контрактом роли устройства, так и областями доступа оператора в сеансе вызывающего

Считайте код настройки паролем, пока он действителен.

Для Tailscale, публичного или другого удаленного мобильного сопряжения используйте Tailscale Serve/Funnel
или другой URL Gateway с `wss://`. Коды настройки в открытом виде `ws://` принимаются только
для loopback, адресов частной LAN, хостов Bonjour `.local` и хоста эмулятора Android.
Адреса Tailnet CGNAT, имена `.ts.net` и публичные хосты по-прежнему закрыто отклоняются до
выдачи QR-кода/кода настройки.

### Подтвердить устройство Node

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
```

Когда явное подтверждение отклоняется из-за того, что сеанс подтверждающего сопряженного устройства
был открыт только с областью доступа сопряжения, CLI повторяет тот же запрос с
`operator.admin`. Это позволяет существующему сопряженному устройству с правами администратора восстановить новое
сопряжение Control UI/браузера без ручного редактирования `devices/paired.json`.
Gateway по-прежнему проверяет повторное подключение; токены, которые не могут пройти аутентификацию
с `operator.admin`, остаются заблокированными.

Если то же устройство повторяет попытку с другими данными аутентификации (например другой
ролью/областями доступа/публичным ключом), предыдущий ожидающий запрос заменяется и создается новый
`requestId`.

<Note>
Уже сопряженное устройство не получает более широкий доступ неявно. Если оно повторно подключается, запрашивая больше областей доступа или более широкую роль, OpenClaw оставляет существующее подтверждение без изменений и создает новый ожидающий запрос на повышение доступа. Используйте `openclaw devices list`, чтобы сравнить текущий утвержденный доступ с вновь запрошенным доступом перед подтверждением.
</Note>

### Необязательное автоматическое подтверждение Node по доверенному CIDR

Сопряжение устройств по умолчанию остается ручным. Для строго контролируемых сетей Node
можно явно включить автоматическое подтверждение первого сопряжения Node с явными CIDR или точными IP-адресами:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24"],
      },
    },
  },
}
```

Это применяется только к свежим запросам сопряжения `role: node` без запрошенных
областей доступа. Клиенты Operator, браузера, Control UI и WebChat по-прежнему требуют ручного
подтверждения. Изменения роли, области доступа, метаданных и публичного ключа также требуют ручного
подтверждения.

### Хранение состояния сопряжения Node

Хранится в `~/.openclaw/devices/`:

- `pending.json` (короткоживущий; ожидающие запросы истекают)
- `paired.json` (сопряженные устройства и токены)

### Примечания

- Устаревший API `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|remove|rename`) — это
  отдельное хранилище сопряжений, принадлежащее gateway. WS-узлы по-прежнему требуют сопряжения устройства.
- Запись сопряжения — устойчивый источник истины для утвержденных ролей. Активные
  токены устройств остаются ограничены этим утвержденным набором ролей; случайная запись токена
  вне утвержденных ролей не создает новый доступ.
