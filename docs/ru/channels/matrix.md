Matrix — это загружаемый Plugin канала для OpenClaw.
Он использует официальный `matrix-js-sdk` и поддерживает личные сообщения, комнаты, ветки, медиа, реакции, опросы, геолокацию и E2EE.

## Установка

Установите Matrix из ClawHub перед настройкой канала:

```bash
openclaw plugins install @openclaw/matrix
```

Спецификации Plugin без префикса сначала пробуют ClawHub, затем резервно npm. Чтобы принудительно выбрать источник реестра, используйте `openclaw plugins install clawhub:@openclaw/matrix` или `openclaw plugins install npm:@openclaw/matrix`.

Из локального checkout:

```bash
openclaw plugins install ./path/to/local/matrix-plugin
```

`plugins install` регистрирует и включает Plugin, поэтому отдельный шаг `openclaw plugins enable matrix` не нужен. Plugin все равно ничего не делает, пока вы не настроите канал ниже. Общие сведения о поведении Plugin и правилах установки см. в разделе [Plugins](/ru/tools/plugin).

## Настройка

1. Создайте учетную запись Matrix на своем homeserver.
2. Настройте `channels.matrix` с `homeserver` + `accessToken` либо с `homeserver` + `userId` + `password`.
3. Перезапустите Gateway.
4. Начните личную переписку с ботом или пригласите его в комнату (см. [автоприсоединение](#auto-join) - новые приглашения принимаются только тогда, когда это разрешает `autoJoin`).

### Интерактивная настройка

```bash
openclaw channels add
openclaw configure --section channels
```

Мастер запрашивает: URL homeserver, метод аутентификации (access token или пароль), ID пользователя (только для аутентификации по паролю), необязательное имя устройства, включать ли E2EE, а также настраивать ли доступ к комнатам и автоприсоединение.

Если соответствующие переменные окружения `MATRIX_*` уже существуют, а у выбранной учетной записи нет сохраненной аутентификации, мастер предложит быстрый вариант с env-var. Чтобы разрешить имена комнат перед сохранением allowlist, выполните `openclaw channels resolve --channel matrix "Project Room"`. Когда E2EE включено, мастер записывает конфигурацию и запускает тот же bootstrap, что и [`openclaw matrix encryption setup`](#encryption-and-verification).

### Минимальная конфигурация

На основе токена:

```json5
{
  channels: {
    matrix: {
      enabled: true,
      homeserver: "https://matrix.example.org",
      accessToken: "syt_xxx",
      dm: { policy: "pairing" },
    },
  },
}
```

На основе пароля (токен кэшируется после первого входа):

```json5
{
  channels: {
    matrix: {
      enabled: true,
      homeserver: "https://matrix.example.org",
      userId: "@bot:example.org",
      password: "replace-me", // pragma: allowlist secret
      deviceName: "OpenClaw Gateway",
    },
  },
}
```

### Автоприсоединение

`channels.matrix.autoJoin` по умолчанию имеет значение `off`. При значении по умолчанию бот не появится в новых комнатах или личных сообщениях из новых приглашений, пока вы не присоединитесь вручную.

OpenClaw не может в момент приглашения определить, является ли приглашенная комната личной перепиской или группой, поэтому все приглашения, включая приглашения в стиле личной переписки, сначала проходят через `autoJoin`. `dm.policy` применяется только позже, после того как бот присоединился и комната была классифицирована.

<Warning>
Задайте `autoJoin: "allowlist"` вместе с `autoJoinAllowlist`, чтобы ограничить, какие приглашения принимает бот, или `autoJoin: "always"`, чтобы принимать каждое приглашение.

`autoJoinAllowlist` принимает только стабильные цели: `!roomId:server`, `#alias:server` или `*`. Простые имена комнат отклоняются; записи alias разрешаются относительно homeserver, а не относительно состояния, заявленного приглашенной комнатой.
</Warning>

```json5
{
  channels: {
    matrix: {
      autoJoin: "allowlist",
      autoJoinAllowlist: ["!ops:example.org", "#support:example.org"],
      groups: {
        "!ops:example.org": { requireMention: true },
      },
    },
  },
}
```

Чтобы принимать каждое приглашение, используйте `autoJoin: "always"`.

### Форматы целей allowlist

Allowlist для личных сообщений и комнат лучше заполнять стабильными ID:

- Личные сообщения (`dm.allowFrom`, `groupAllowFrom`, `groups.<room>.users`): используйте `@user:server`. Отображаемые имена по умолчанию игнорируются, потому что они изменяемы; задавайте `dangerouslyAllowNameMatching: true` только тогда, когда вам явно нужна совместимость с записями отображаемых имен.
- Ключи allowlist комнат (`groups`, устаревшие `rooms`): используйте `!room:server` или `#alias:server`. Простые имена комнат по умолчанию игнорируются; задавайте `dangerouslyAllowNameMatching: true` только тогда, когда вам явно нужна совместимость с поиском имени присоединенной комнаты.
- Allowlist приглашений (`autoJoinAllowlist`): используйте `!room:server`, `#alias:server` или `*`. Простые имена комнат отклоняются.

### Нормализация ID учетной записи

Мастер преобразует понятное имя в нормализованный ID учетной записи. Например, `Ops Bot` становится `ops-bot`. Пунктуация экранируется в scoped env-var names, чтобы две учетные записи не могли конфликтовать: `-` → `_X2D_`, поэтому `ops-prod` сопоставляется с `MATRIX_OPS_X2D_PROD_*`.

### Кэшированные учетные данные

Matrix хранит кэшированные учетные данные в `~/.openclaw/credentials/matrix/`:

- учетная запись по умолчанию: `credentials.json`
- именованные учетные записи: `credentials-<account>.json`

Когда кэшированные учетные данные там существуют, OpenClaw считает Matrix настроенным, даже если access token отсутствует в файле конфигурации - это покрывает настройку, `openclaw doctor` и проверки статуса канала.

### Переменные окружения

Используются, когда эквивалентный ключ конфигурации не задан. Учетная запись по умолчанию использует имена без префикса; именованные учетные записи используют ID учетной записи, вставленный перед суффиксом.

| Учетная запись по умолчанию | Именованная учетная запись (`<ID>` — нормализованный ID учетной записи) |
| --------------------- | --------------------------------------------------- |
| `MATRIX_HOMESERVER`   | `MATRIX_<ID>_HOMESERVER`                            |
| `MATRIX_ACCESS_TOKEN` | `MATRIX_<ID>_ACCESS_TOKEN`                          |
| `MATRIX_USER_ID`      | `MATRIX_<ID>_USER_ID`                               |
| `MATRIX_PASSWORD`     | `MATRIX_<ID>_PASSWORD`                              |
| `MATRIX_DEVICE_ID`    | `MATRIX_<ID>_DEVICE_ID`                             |
| `MATRIX_DEVICE_NAME`  | `MATRIX_<ID>_DEVICE_NAME`                           |
| `MATRIX_RECOVERY_KEY` | `MATRIX_<ID>_RECOVERY_KEY`                          |

Для учетной записи `ops` имена становятся `MATRIX_OPS_HOMESERVER`, `MATRIX_OPS_ACCESS_TOKEN` и так далее. Переменные окружения recovery-key читаются CLI-потоками, поддерживающими recovery (`verify backup restore`, `verify device`, `verify bootstrap`), когда вы передаете ключ через `--recovery-key-stdin`.

`MATRIX_HOMESERVER` нельзя задать из workspace `.env`; см. [файлы workspace `.env`](/ru/gateway/security).

## Пример конфигурации

Практичная базовая конфигурация с сопряжением личных сообщений, allowlist комнат и E2EE:

```json5
{
  channels: {
    matrix: {
      enabled: true,
      homeserver: "https://matrix.example.org",
      accessToken: "syt_xxx",
      encryption: true,

      dm: {
        policy: "pairing",
        sessionScope: "per-room",
        threadReplies: "off",
      },

      groupPolicy: "allowlist",
      groupAllowFrom: ["@admin:example.org"],
      groups: {
        "!roomid:example.org": { requireMention: true },
      },

      autoJoin: "allowlist",
      autoJoinAllowlist: ["!roomid:example.org"],
      threadReplies: "inbound",
      replyToMode: "off",
      streaming: "partial",
    },
  },
}
```

## Потоковые превью

Потоковая передача ответов Matrix включается явно. `streaming` управляет тем, как OpenClaw доставляет текущий ответ ассистента; `blockStreaming` управляет тем, сохраняется ли каждый завершенный блок как отдельное сообщение Matrix.

```json5
{
  channels: {
    matrix: {
      streaming: "partial",
    },
  },
}
```

Чтобы сохранить live-превью ответа, но скрыть промежуточные строки инструментов/прогресса, используйте объектную
форму:

```json5
{
  channels: {
    matrix: {
      streaming: {
        mode: "partial",
        preview: {
          toolProgress: false,
        },
      },
    },
  },
}
```

Полная объектная форма принимает `{ mode, preview, progress }`:

```json5
{
  channels: {
    matrix: {
      streaming: {
        mode: "progress",
        progress: {
          label: "auto", // pick from configured or built-in labels (false to hide)
          labels: ["Thinking", "Writing", "Searching"], // candidates for label: "auto"
          maxLines: 8, // max rolling progress lines (default: 8)
          maxLineChars: 120, // max chars per line before truncation (default: 120)
          toolProgress: true, // show tool/progress activity (default: true)
        },
      },
    },
  },
}
```

- `progress.label`: пользовательская метка, `"auto"` или не задано для выбора из настроенных или встроенных меток, либо `false`, чтобы скрыть строку метки.
- `progress.labels`: метки-кандидаты, используемые только когда `label` равно `"auto"` или не задано. Оставьте не заданным для встроенных значений по умолчанию.
- `progress.maxLines`: максимальное количество прокручиваемых строк прогресса, сохраняемых в черновике. После этого лимита старые строки обрезаются.
- `progress.maxLineChars`: максимальное количество символов в компактной строке прогресса перед усечением.
- `progress.toolProgress`: когда `true` (по умолчанию), live-активность инструментов/прогресса появляется в черновике.

| `streaming`       | Поведение                                                                                                                                                            |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `"off"` (по умолчанию) | Ждать полный ответ, отправить один раз. `true` ↔ `"partial"`, `false` ↔ `"off"`.                                                                                        |
| `"partial"`       | Редактировать одно обычное текстовое сообщение на месте по мере того, как модель пишет текущий блок. Стандартные клиенты Matrix могут уведомить при первом превью, а не при финальном редактировании.              |
| `"quiet"`         | То же, что `"partial"`, но сообщение является notice без уведомления. Получатели получают уведомление только после того, как per-user push rule совпадет с финальным редактированием (см. ниже). |
| `"progress"`      | Отправляет отдельные компактные строки прогресса с использованием черновика прогресса.                                                                                                     |

`blockStreaming` не зависит от `streaming`:

| `streaming`             | `blockStreaming: true`                                              | `blockStreaming: false` (по умолчанию)                    |
| ----------------------- | ------------------------------------------------------------------- | ---------------------------------------------------- |
| `"partial"` / `"quiet"` | Live-черновик для текущего блока, завершенные блоки сохраняются как сообщения | Live-черновик для текущего блока, финализируется на месте |
| `"off"`                 | Одно уведомляющее сообщение Matrix на каждый завершенный блок                     | Одно уведомляющее сообщение Matrix для полного ответа      |

Примечания:

- Если превью превышает лимит размера одного события Matrix, OpenClaw останавливает потоковое превью и возвращается к доставке только финального ответа.
- Медиаответы всегда отправляют вложения обычным способом. Если устаревшее превью больше нельзя безопасно использовать повторно, OpenClaw редактирует его перед отправкой финального медиаответа.
- Обновления превью прогресса инструментов включены по умолчанию, когда активно потоковое превью Matrix. Задайте `streaming.preview.toolProgress: false`, чтобы оставить редактирование превью для текста ответа, но оставить прогресс инструментов в обычном пути доставки.
- Редактирование превью требует дополнительных вызовов Matrix API. Оставьте `streaming: "off"`, если вам нужен самый консервативный профиль rate-limit.

## Голосовые сообщения

Входящие голосовые заметки Matrix транскрибируются до проверки упоминания комнаты. Это позволяет голосовой заметке, в которой произносится имя бота, запустить агента в комнате с `requireMention: true`, и дает агенту расшифровку вместо одного только placeholder аудиовложения.

Matrix использует общий провайдер аудиомедиа, настроенный в `tools.media.audio`, например OpenAI `gpt-4o-mini-transcribe`. См. [обзор медиаинструментов](/ru/tools/media-overview) для настройки провайдера и ограничений.

Подробности поведения:

- События `m.audio` и события `m.file` с MIME-типом `audio/*` подходят.
- В зашифрованных комнатах OpenClaw расшифровывает вложение через существующий путь Matrix media перед транскрипцией.
- Транскрипт помечается в промпте агента как машинно сгенерированный и недоверенный.
- Вложение помечается как уже транскрибированное, чтобы последующие медиа-инструменты не транскрибировали ту же голосовую заметку повторно.
- Задайте `tools.media.audio.enabled: false`, чтобы отключить транскрипцию аудио глобально.

## Метаданные подтверждения

Нативные запросы подтверждения Matrix — это обычные события `m.room.message` с пользовательским содержимым события, специфичным для OpenClaw, в `com.openclaw.approval`. Matrix разрешает пользовательские ключи содержимого события, поэтому стандартные клиенты по-прежнему отображают текстовое тело, а клиенты, поддерживающие OpenClaw, могут читать структурированный идентификатор подтверждения, тип, состояние, доступные решения и сведения об exec/plugin.

Когда запрос подтверждения слишком длинный для одного события Matrix, OpenClaw разбивает видимый текст на части и прикрепляет `com.openclaw.approval` только к первой части. Реакции для решений разрешить/запретить привязываются к этому первому событию, поэтому длинные запросы сохраняют ту же цель подтверждения, что и запросы из одного события.

### Правила push для самостоятельного хостинга тихих финализированных предпросмотров

`streaming: "quiet"` уведомляет получателей только после финализации блока или хода — пользовательское правило push должно совпадать с маркером финализированного предпросмотра. Полный рецепт см. в разделе [Правила push Matrix для тихих предпросмотров](/ru/channels/matrix-push-rules) (токен получателя, проверка pusher, установка правила, заметки по отдельным домашним серверам).

## Комнаты бот-бот

По умолчанию сообщения Matrix от других настроенных учетных записей Matrix OpenClaw игнорируются.

Используйте `allowBots`, когда намеренно хотите разрешить межагентский трафик Matrix:

```json5
{
  channels: {
    matrix: {
      allowBots: "mentions", // true | "mentions"
      groups: {
        "!roomid:example.org": {
          requireMention: true,
        },
      },
    },
  },
}
```

- `allowBots: true` принимает сообщения от других настроенных учетных записей ботов Matrix в разрешенных комнатах и личных сообщениях.
- `allowBots: "mentions"` принимает эти сообщения только когда они явно упоминают этого бота в комнатах. Личные сообщения по-прежнему разрешены.
- `groups.<room>.allowBots` переопределяет настройку уровня учетной записи для одной комнаты.
- Принятые сообщения от настроенных ботов используют общую [защиту от циклов ботов](/ru/channels/bot-loop-protection). Настройте `channels.defaults.botLoopProtection`, затем переопределите через `channels.matrix.botLoopProtection` или `channels.matrix.groups.<room>.botLoopProtection`, когда для одной комнаты нужен другой бюджет.
- OpenClaw по-прежнему игнорирует сообщения от того же идентификатора пользователя Matrix, чтобы избежать циклов самоответов.
- Matrix не предоставляет здесь нативный флаг бота; OpenClaw трактует «создано ботом» как «отправлено другой настроенной учетной записью Matrix на этом Gateway OpenClaw».

Используйте строгие списки разрешенных комнат и требования упоминаний при включении трафика бот-бот в общих комнатах.

## Шифрование и проверка

В зашифрованных комнатах (E2EE) исходящие события изображений используют `thumbnail_file`, поэтому предпросмотры изображений шифруются вместе с полным вложением. Незашифрованные комнаты по-прежнему используют простой `thumbnail_url`. Настройка не требуется — plugin автоматически определяет состояние E2EE.

Все команды `openclaw matrix` принимают `--verbose` (полная диагностика), `--json` (машиночитаемый вывод) и `--account <id>` (настройки с несколькими учетными записями). По умолчанию вывод краткий, с тихим внутренним логированием SDK. Примеры ниже показывают каноническую форму; добавляйте флаги по мере необходимости.

### Включение шифрования

```bash
openclaw matrix encryption setup
```

Инициализирует хранилище секретов и кросс-подпись, при необходимости создает резервную копию ключей комнаты, затем печатает статус и следующие шаги. Полезные флаги:

- `--recovery-key <key>` применить ключ восстановления перед инициализацией (предпочитайте форму через stdin, документированную ниже)
- `--force-reset-cross-signing` отбросить текущую идентичность кросс-подписи и создать новую (используйте только намеренно)

Для новой учетной записи включайте E2EE при создании:

```bash
openclaw matrix account add \
  --homeserver https://matrix.example.org \
  --access-token syt_xxx \
  --enable-e2ee
```

`--encryption` — это псевдоним для `--enable-e2ee`.

Эквивалент ручной конфигурации:

```json5
{
  channels: {
    matrix: {
      enabled: true,
      homeserver: "https://matrix.example.org",
      accessToken: "syt_xxx",
      encryption: true,
      dm: { policy: "pairing" },
    },
  },
}
```

### Статус и сигналы доверия

```bash
openclaw matrix verify status
openclaw matrix verify status --include-recovery-key --json
```

`verify status` сообщает три независимых сигнала доверия (`--verbose` показывает их все):

- `Locally trusted`: доверено только этим клиентом
- `Cross-signing verified`: SDK сообщает о проверке через кросс-подпись
- `Signed by owner`: подписано вашим собственным ключом самоподписи (только диагностика)

`Verified by owner` становится `yes` только когда `Cross-signing verified` равно `yes`. Одного локального доверия или подписи владельца недостаточно.

`--allow-degraded-local-state` возвращает диагностику по мере возможности без предварительной подготовки учетной записи Matrix; полезно для офлайн-проверок или частично настроенных проверок.

### Проверка этого устройства с помощью ключа восстановления

Ключ восстановления чувствителен — передавайте его через stdin вместо передачи в командной строке. Задайте `MATRIX_RECOVERY_KEY` (или `MATRIX_<ID>_RECOVERY_KEY` для именованной учетной записи):

```bash
printf '%s\n' "$MATRIX_RECOVERY_KEY" | openclaw matrix verify device --recovery-key-stdin
```

Команда сообщает три состояния:

- `Recovery key accepted`: Matrix принял ключ для хранилища секретов или доверия устройству.
- `Backup usable`: резервная копия ключей комнаты может быть загружена с доверенным материалом восстановления.
- `Device verified by owner`: это устройство имеет полное доверие идентичности кросс-подписи Matrix.

Она завершается с ненулевым кодом, когда полное доверие идентичности не завершено, даже если ключ восстановления разблокировал материал резервной копии. В этом случае завершите самопроверку из другого клиента Matrix:

```bash
openclaw matrix verify self
```

`verify self` ожидает `Cross-signing verified: yes`, прежде чем успешно завершиться. Используйте `--timeout-ms <ms>`, чтобы настроить ожидание.

Форма с буквальным ключом `openclaw matrix verify device "<recovery-key>"` также поддерживается, но ключ попадает в историю вашей оболочки.

### Инициализация или восстановление кросс-подписи

```bash
openclaw matrix verify bootstrap
```

`verify bootstrap` — это команда восстановления и настройки для зашифрованных учетных записей. По порядку она:

- инициализирует хранилище секретов, по возможности повторно используя существующий ключ восстановления
- инициализирует кросс-подпись и загружает недостающие публичные ключи
- помечает и кросс-подписывает текущее устройство
- создает серверную резервную копию ключей комнаты, если она еще не существует

Если домашний сервер требует UIA для загрузки ключей кросс-подписи, OpenClaw сначала пробует без аутентификации, затем `m.login.dummy`, затем `m.login.password` (требует `channels.matrix.password`).

Полезные флаги:

- `--recovery-key-stdin` (используйте вместе с `printf '%s\n' "$MATRIX_RECOVERY_KEY" | …`) или `--recovery-key <key>`
- `--force-reset-cross-signing`, чтобы отбросить текущую идентичность кросс-подписи (только намеренно; требует, чтобы активный ключ восстановления был сохранен или предоставлен с `--recovery-key-stdin`)

### Резервная копия ключей комнаты

```bash
openclaw matrix verify backup status
printf '%s\n' "$MATRIX_RECOVERY_KEY" | openclaw matrix verify backup restore --recovery-key-stdin
```

`backup status` показывает, существует ли серверная резервная копия и может ли это устройство ее расшифровать. `backup restore` импортирует сохраненные в резервной копии ключи комнаты в локальное криптохранилище; если ключ восстановления уже есть на диске, `--recovery-key-stdin` можно опустить.

Чтобы заменить поврежденную резервную копию новой базовой версией (принимая потерю невосстановимой старой истории; также может пересоздать хранилище секретов, если текущий секрет резервной копии невозможно загрузить):

```bash
openclaw matrix verify backup reset --yes
```

Добавляйте `--rotate-recovery-key` только когда намеренно хотите, чтобы предыдущий ключ восстановления перестал разблокировать новую базовую резервную копию.

### Просмотр, запрос и ответ на проверки

```bash
openclaw matrix verify list
```

Перечисляет ожидающие запросы проверки для выбранной учетной записи.

```bash
openclaw matrix verify request --own-user
openclaw matrix verify request --user-id @ops:example.org --device-id ABCDEF
```

Отправляет запрос проверки из этой учетной записи OpenClaw. `--own-user` запрашивает самопроверку (вы принимаете запрос в другом клиенте Matrix того же пользователя); `--user-id`/`--device-id`/`--room-id` нацелены на другого пользователя. `--own-user` нельзя сочетать с другими флагами выбора цели.

Для более низкоуровневой обработки жизненного цикла — обычно при сопровождении входящих запросов из другого клиента — эти команды действуют на конкретный запрос `<id>` (печатается `verify list` и `verify request`):

| Команда                                    | Назначение                                                          |
| ------------------------------------------ | ------------------------------------------------------------------- |
| `openclaw matrix verify accept <id>`       | Принять входящий запрос                                             |
| `openclaw matrix verify start <id>`        | Запустить поток SAS                                                 |
| `openclaw matrix verify sas <id>`          | Напечатать эмодзи или десятичные числа SAS                          |
| `openclaw matrix verify confirm-sas <id>`  | Подтвердить, что SAS совпадает с тем, что показывает другой клиент  |
| `openclaw matrix verify mismatch-sas <id>` | Отклонить SAS, когда эмодзи или десятичные числа не совпадают       |
| `openclaw matrix verify cancel <id>`       | Отменить; принимает необязательные `--reason <text>` и `--code <matrix-code>` |

`accept`, `start`, `sas`, `confirm-sas`, `mismatch-sas` и `cancel` принимают `--user-id` и `--room-id` как подсказки для последующего личного сообщения, когда проверка привязана к конкретной комнате прямых сообщений.

## Треды

Matrix поддерживает нативные треды Matrix как для автоматических ответов, так и для отправок через инструмент сообщений. Поведение управляется двумя независимыми переключателями:

### Маршрутизация сессий (`sessionScope`)

`dm.sessionScope` определяет, как DM-комнаты Matrix сопоставляются с сессиями OpenClaw:

- `"per-user"` (по умолчанию): все DM-комнаты с одним и тем же маршрутизированным собеседником используют одну сессию.
- `"per-room"`: каждая DM-комната Matrix получает собственный ключ сессии, даже если собеседник тот же.

Явные привязки бесед всегда имеют приоритет над `sessionScope`, поэтому привязанные комнаты и треды сохраняют выбранную целевую сессию.

### Ответы в тредах (`threadReplies`)

`threadReplies` определяет, куда бот публикует свой ответ:

- `"off"`: ответы являются верхнеуровневыми. Входящие сообщения из тредов остаются в родительской сессии.
- `"inbound"`: отвечать внутри треда только когда входящее сообщение уже было в этом треде.
- `"always"`: отвечать внутри треда, корнем которого является вызвавшее сообщение; эта беседа маршрутизируется через соответствующую сессию с областью треда, начиная с первого триггера.

`dm.threadReplies` переопределяет это только для DM - например, чтобы изолировать треды комнат, оставляя DM плоскими.

### Наследование тредов и slash-команды

- Входящие сообщения из тредов включают корневое сообщение треда как дополнительный контекст агента.
- Отправки через инструмент сообщений автоматически наследуют текущий тред Matrix при отправке в ту же комнату (или тому же целевому пользователю DM), если не указан явный `threadId`.
- Повторное использование целевого пользователя DM включается только когда метаданные текущей сессии доказывают того же DM-собеседника в том же аккаунте Matrix; иначе OpenClaw возвращается к обычной маршрутизации с областью пользователя.
- `/focus`, `/unfocus`, `/agents`, `/session idle`, `/session max-age` и привязанный к треду `/acp spawn` работают в комнатах Matrix и DM.
- Верхнеуровневый `/focus` создает новый тред Matrix и привязывает его к целевой сессии, когда включен `threadBindings.spawnSessions`.
- Запуск `/focus` или `/acp spawn --thread here` внутри существующего треда Matrix привязывает этот тред на месте.

Когда OpenClaw обнаруживает, что DM-комната Matrix конфликтует с другой DM-комнатой в той же общей сессии, он публикует однократное `m.notice` в этой комнате, указывая на обходной путь `/focus` и предлагая изменить `dm.sessionScope`. Уведомление появляется только когда включены привязки тредов.

## Привязки бесед ACP

Комнаты Matrix, DM и существующие треды Matrix можно превратить в долговременные рабочие пространства ACP без изменения поверхности чата.

Быстрый операторский поток:

- Запустите `/acp spawn codex --bind here` внутри DM Matrix, комнаты или существующего треда, который хотите продолжать использовать.
- В верхнеуровневом DM Matrix или комнате текущий DM/комната остается поверхностью чата, а будущие сообщения маршрутизируются в созданную ACP-сессию.
- Внутри существующего треда Matrix `--bind here` привязывает текущий тред на месте.
- `/new` и `/reset` сбрасывают ту же привязанную ACP-сессию на месте.
- `/acp close` закрывает ACP-сессию и удаляет привязку.

Примечания:

- `--bind here` не создает дочерний тред Matrix.
- `threadBindings.spawnSessions` управляет `/acp spawn --thread auto|here`, где OpenClaw должен создать или привязать дочерний тред Matrix.

### Конфигурация привязки тредов

Matrix наследует глобальные значения по умолчанию из `session.threadBindings`, а также поддерживает переопределения для конкретного канала:

- `threadBindings.enabled`
- `threadBindings.idleHours`
- `threadBindings.maxAgeHours`
- `threadBindings.spawnSessions`
- `threadBindings.defaultSpawnContext`

Создание сессий, привязанных к тредам Matrix, по умолчанию включено:

- Установите `threadBindings.spawnSessions: false`, чтобы запретить верхнеуровневому `/focus` и `/acp spawn --thread auto|here` создавать/привязывать треды Matrix.
- Установите `threadBindings.defaultSpawnContext: "isolated"`, когда нативные создания тредов подагентов не должны форкать родительскую расшифровку.

## Реакции

Matrix поддерживает исходящие реакции, входящие уведомления о реакциях и ack-реакции.

Инструменты исходящих реакций управляются `channels.matrix.actions.reactions`:

- `react` добавляет реакцию к событию Matrix.
- `reactions` выводит текущую сводку реакций для события Matrix.
- `emoji=""` удаляет собственные реакции бота на этом событии.
- `remove: true` удаляет только указанную emoji-реакцию от бота.

**Порядок разрешения** (побеждает первое заданное значение):

| Настройка               | Порядок                                                                          |
| ----------------------- | -------------------------------------------------------------------------------- |
| `ackReaction`           | для аккаунта → канал → `messages.ackReaction` → резервный emoji идентичности агента |
| `ackReactionScope`      | для аккаунта → канал → `messages.ackReactionScope` → по умолчанию `"group-mentions"` |
| `reactionNotifications` | для аккаунта → канал → по умолчанию `"own"`                                      |

`reactionNotifications: "own"` пересылает добавленные события `m.reaction`, когда они нацелены на сообщения Matrix, созданные ботом; `"off"` отключает системные события реакций. Удаления реакций не синтезируются в системные события, потому что Matrix представляет их как редактирования, а не как отдельные удаления `m.reaction`.

## Контекст истории

- `channels.matrix.historyLimit` управляет тем, сколько недавних сообщений комнаты включается как `InboundHistory`, когда сообщение в комнате Matrix запускает агента. Выполняется откат к `messages.groupChat.historyLimit`; если оба значения не заданы, эффективное значение по умолчанию равно `0`. Установите `0`, чтобы отключить.
- История комнаты Matrix относится только к комнате. DM продолжают использовать обычную историю сессии.
- История комнаты Matrix является pending-only: OpenClaw буферизует сообщения комнаты, которые еще не вызвали ответ, затем создает снимок этого окна, когда приходит упоминание или другой триггер.
- Текущее сообщение-триггер не включается в `InboundHistory`; оно остается в основном входящем теле для этого хода.
- Повторные попытки того же события Matrix повторно используют исходный снимок истории, а не смещаются вперед к более новым сообщениям комнаты.

## Видимость контекста

Matrix поддерживает общий элемент управления `contextVisibility` для дополнительного контекста комнаты, такого как полученный текст ответа, корни тредов и ожидающая история.

- `contextVisibility: "all"` используется по умолчанию. Дополнительный контекст сохраняется в полученном виде.
- `contextVisibility: "allowlist"` фильтрует дополнительный контекст до отправителей, разрешенных активными проверками allowlist комнаты/пользователя.
- `contextVisibility: "allowlist_quote"` ведет себя как `allowlist`, но все равно сохраняет один явный процитированный ответ.

Эта настройка влияет на видимость дополнительного контекста, а не на то, может ли само входящее сообщение вызвать ответ.
Авторизация триггера по-прежнему поступает из `groupPolicy`, `groups`, `groupAllowFrom` и настроек политики DM.

## Политика DM и комнат

```json5
{
  channels: {
    matrix: {
      dm: {
        policy: "allowlist",
        allowFrom: ["@admin:example.org"],
        threadReplies: "off",
      },
      groupPolicy: "allowlist",
      groupAllowFrom: ["@admin:example.org"],
      groups: {
        "!roomid:example.org": { requireMention: true },
      },
    },
  },
}
```

Чтобы полностью заглушить DM, сохранив работу комнат, установите `dm.enabled: false`:

```json5
{
  channels: {
    matrix: {
      dm: { enabled: false },
      groupPolicy: "allowlist",
      groupAllowFrom: ["@admin:example.org"],
    },
  },
}
```

См. [Группы](/ru/channels/groups) для поведения gating по упоминаниям и allowlist.

Пример pairing для DM Matrix:

```bash
openclaw pairing list matrix
openclaw pairing approve matrix <CODE>
```

Если неподтвержденный пользователь Matrix продолжает писать вам до подтверждения, OpenClaw повторно использует тот же ожидающий код pairing и может отправить ответ-напоминание после короткого cooldown вместо выпуска нового кода.

См. [Pairing](/ru/channels/pairing) для общего потока pairing DM и схемы хранения.

## Ремонт прямых комнат

Если состояние прямых сообщений рассинхронизируется, OpenClaw может получить устаревшие сопоставления `m.direct`, указывающие на старые solo-комнаты вместо активного DM. Проверьте текущее сопоставление для собеседника:

```bash
openclaw matrix direct inspect --user-id @alice:example.org
```

Восстановите его:

```bash
openclaw matrix direct repair --user-id @alice:example.org
```

Обе команды принимают `--account <id>` для настроек с несколькими аккаунтами. Поток восстановления:

- предпочитает строгий DM 1:1, уже сопоставленный в `m.direct`
- откатывается к любому текущему присоединенному строгому DM 1:1 с этим пользователем
- создает свежую прямую комнату и переписывает `m.direct`, если исправного DM нет

Он не удаляет старые комнаты автоматически. Он выбирает исправный DM и обновляет сопоставление, чтобы будущие отправки Matrix, уведомления о проверке и другие потоки прямых сообщений были направлены в правильную комнату.

## Подтверждения exec

Matrix может выступать как нативный клиент подтверждений. Настройте в `channels.matrix.execApprovals` (или `channels.matrix.accounts.<account>.execApprovals` для переопределения на уровне аккаунта):

- `enabled`: доставлять подтверждения через нативные prompts Matrix. Если не задано или равно `"auto"`, Matrix автоматически включается, как только можно разрешить хотя бы одного подтверждающего. Установите `false`, чтобы явно отключить.
- `approvers`: ID пользователей Matrix (`@owner:example.org`), которым разрешено подтверждать exec-запросы. Необязательно - выполняется откат к `channels.matrix.dm.allowFrom`.
- `target`: куда отправляются prompts. `"dm"` (по умолчанию) отправляет в DM подтверждающих; `"channel"` отправляет в исходную комнату Matrix или DM; `"both"` отправляет в оба места.
- `agentFilter` / `sessionFilter`: необязательные allowlist для того, какие агенты/сессии запускают доставку Matrix.

Авторизация немного отличается для разных видов подтверждений:

- **Exec-подтверждения** используют `execApprovals.approvers` с откатом к `dm.allowFrom`.
- **Plugin-подтверждения** авторизуются только через `dm.allowFrom`.

Оба вида совместно используют быстрые реакции Matrix и обновления сообщений. Подтверждающие видят быстрые реакции на основном сообщении подтверждения:

- `✅` разрешить один раз
- `❌` запретить
- `♾️` разрешать всегда (когда это допускает эффективная exec-политика)

Резервные slash-команды: `/approve <id> allow-once`, `/approve <id> allow-always`, `/approve <id> deny`.

Только разрешенные согласующие могут подтверждать или отклонять. Доставка подтверждений exec в канал включает текст команды — включайте `channel` или `both` только в доверенных комнатах.

См. также: [Подтверждения exec](/ru/tools/exec-approvals).

## Slash-команды

Slash-команды (`/new`, `/reset`, `/model`, `/focus`, `/unfocus`, `/agents`, `/session`, `/acp`, `/approve` и т. д.) работают напрямую в личных сообщениях. В комнатах OpenClaw также распознает команды с префиксом собственного упоминания Matrix-бота, поэтому `@bot:server /new` запускает путь команды без пользовательского regex для упоминаний. Это сохраняет отзывчивость бота для сообщений комнатного вида `@mention /command`, которые Element и похожие клиенты отправляют, когда пользователь дополняет имя бота клавишей Tab перед вводом команды.

Правила авторизации по-прежнему применяются: отправители команд должны удовлетворять тем же политикам allowlist/владельца для личных сообщений или комнат, что и обычные сообщения.

## Несколько аккаунтов

```json5
{
  channels: {
    matrix: {
      enabled: true,
      defaultAccount: "assistant",
      dm: { policy: "pairing" },
      accounts: {
        assistant: {
          homeserver: "https://matrix.example.org",
          accessToken: "syt_assistant_xxx",
          encryption: true,
        },
        alerts: {
          homeserver: "https://matrix.example.org",
          accessToken: "syt_alerts_xxx",
          dm: {
            policy: "allowlist",
            allowFrom: ["@ops:example.org"],
            threadReplies: "off",
          },
        },
      },
    },
  },
}
```

**Наследование:**

- Значения верхнего уровня `channels.matrix` действуют как значения по умолчанию для именованных аккаунтов, если аккаунт не переопределяет их.
- Ограничьте унаследованную запись комнаты конкретным аккаунтом с помощью `groups.<room>.account`. Записи без `account` общие для всех аккаунтов; `account: "default"` по-прежнему работает, когда аккаунт по умолчанию настроен на верхнем уровне.

**Выбор аккаунта по умолчанию:**

- Задайте `defaultAccount`, чтобы выбрать именованный аккаунт, который предпочитают неявная маршрутизация, проверки и команды CLI.
- Если у вас несколько аккаунтов и один буквально называется `default`, OpenClaw использует его неявно, даже когда `defaultAccount` не задан.
- Если у вас несколько именованных аккаунтов и аккаунт по умолчанию не выбран, команды CLI отказываются угадывать — задайте `defaultAccount` или передайте `--account <id>`.
- Блок верхнего уровня `channels.matrix.*` считается неявным аккаунтом `default` только когда его auth завершен (`homeserver` + `accessToken` или `homeserver` + `userId` + `password`). Именованные аккаунты остаются обнаруживаемыми по `homeserver` + `userId`, когда кэшированные учетные данные покрывают auth.

**Продвижение:**

- Когда OpenClaw продвигает конфигурацию с одним аккаунтом до конфигурации с несколькими аккаунтами во время исправления или настройки, он сохраняет существующий именованный аккаунт, если он есть или `defaultAccount` уже указывает на него. В продвинутый аккаунт перемещаются только ключи auth/bootstrap Matrix; общие ключи политики доставки остаются на верхнем уровне.

Общий шаблон нескольких аккаунтов см. в [справочнике по конфигурации](/ru/gateway/config-channels#multi-account-all-channels).

## Частные/LAN-домашние серверы

По умолчанию OpenClaw блокирует частные/внутренние домашние серверы Matrix для защиты от SSRF, если вы
явно не включите это для аккаунта.

Если ваш домашний сервер работает на localhost, LAN/Tailscale IP или внутреннем имени хоста, включите
`network.dangerouslyAllowPrivateNetwork` для этого аккаунта Matrix:

```json5
{
  channels: {
    matrix: {
      homeserver: "http://matrix-synapse:8008",
      network: {
        dangerouslyAllowPrivateNetwork: true,
      },
      accessToken: "syt_internal_xxx",
    },
  },
}
```

Пример настройки CLI:

```bash
openclaw matrix account add \
  --account ops \
  --homeserver http://matrix-synapse:8008 \
  --allow-private-network \
  --access-token syt_ops_xxx
```

Это явное включение разрешает только доверенные частные/внутренние цели. Публичные домашние серверы без шифрования, такие как
`http://matrix.example.org:8008`, остаются заблокированными. По возможности предпочитайте `https://`.

## Проксирование трафика Matrix

Если вашему развертыванию Matrix требуется явный исходящий HTTP(S)-прокси, задайте `channels.matrix.proxy`:

```json5
{
  channels: {
    matrix: {
      homeserver: "https://matrix.example.org",
      accessToken: "syt_bot_xxx",
      proxy: "http://127.0.0.1:7890",
    },
  },
}
```

Именованные аккаунты могут переопределять значение по умолчанию верхнего уровня через `channels.matrix.accounts.<id>.proxy`.
OpenClaw использует одну и ту же настройку прокси для runtime-трафика Matrix и проверок статуса аккаунта.

## Разрешение целей

Matrix принимает эти формы целей везде, где OpenClaw запрашивает цель комнаты или пользователя:

- Пользователи: `@user:server`, `user:@user:server` или `matrix:user:@user:server`
- Комнаты: `!room:server`, `room:!room:server` или `matrix:room:!room:server`
- Алиасы: `#alias:server`, `channel:#alias:server` или `matrix:channel:#alias:server`

ID комнат Matrix чувствительны к регистру. Используйте точный регистр ID комнаты из Matrix
при настройке явных целей доставки, заданий cron, привязок или allowlist.
OpenClaw сохраняет внутренние ключи сессий в каноническом виде для хранения, поэтому эти ключи в нижнем регистре
не являются надежным источником ID доставки Matrix.

Живой поиск в каталоге использует вошедший аккаунт Matrix:

- Поиск пользователей запрашивает каталог пользователей Matrix на этом домашнем сервере.
- Поиск комнат напрямую принимает явные ID комнат и алиасы. Поиск по имени присоединенной комнаты выполняется по возможности и применяется только к runtime-allowlist комнат, когда задано `dangerouslyAllowNameMatching: true`.
- Если имя комнаты нельзя разрешить в ID или алиас, оно игнорируется при runtime-разрешении allowlist.

## Справочник по конфигурации

Пользовательские поля в стиле allowlist (`groupAllowFrom`, `dm.allowFrom`, `groups.<room>.users`) принимают полные ID пользователей Matrix (самый безопасный вариант). Записи пользователей, не являющиеся ID, по умолчанию игнорируются. Если задать `dangerouslyAllowNameMatching: true`, точные совпадения отображаемых имен в каталоге Matrix разрешаются при запуске и каждый раз, когда allowlist меняется во время работы монитора; записи, которые не удалось разрешить, игнорируются во время выполнения.

Ключи allowlist комнат (`groups`, устаревший `rooms`) должны быть ID комнат или алиасами. Обычные ключи с именами комнат по умолчанию игнорируются; `dangerouslyAllowNameMatching: true` восстанавливает поиск по возможности среди имен присоединенных комнат.

### Аккаунт и подключение

- `enabled`: включает или отключает канал.
- `name`: необязательная отображаемая метка для аккаунта.
- `defaultAccount`: предпочитаемый ID аккаунта, когда настроено несколько аккаунтов Matrix.
- `accounts`: именованные переопределения для отдельных аккаунтов. Значения верхнего уровня `channels.matrix` наследуются как значения по умолчанию.
- `homeserver`: URL домашнего сервера, например `https://matrix.example.org`.
- `network.dangerouslyAllowPrivateNetwork`: разрешает этому аккаунту подключаться к `localhost`, LAN/Tailscale IP или внутренним именам хостов.
- `proxy`: необязательный URL HTTP(S)-прокси для трафика Matrix. Поддерживается переопределение для отдельного аккаунта.
- `userId`: полный ID пользователя Matrix (`@bot:example.org`).
- `accessToken`: токен доступа для auth на основе токена. Поддерживаются значения в открытом виде и SecretRef через поставщиков env/file/exec ([Управление секретами](/ru/gateway/secrets)).
- `password`: пароль для входа по паролю. Поддерживаются значения в открытом виде и SecretRef.
- `deviceId`: явный ID устройства Matrix.
- `deviceName`: отображаемое имя устройства, используемое во время входа по паролю.
- `avatarUrl`: сохраненный URL собственного аватара для синхронизации профиля и обновлений `profile set`.
- `initialSyncLimit`: максимальное количество событий, получаемых во время синхронизации при запуске.

### Шифрование

- `encryption`: включает E2EE. По умолчанию: `false`.
- `startupVerification`: `"if-unverified"` (по умолчанию, когда E2EE включено) или `"off"`. Автоматически запрашивает самопроверку при запуске, когда это устройство не проверено.
- `startupVerificationCooldownHours`: период ожидания перед следующим автоматическим запросом при запуске. По умолчанию: `24`.

### Доступ и политика

- `groupPolicy`: `"open"`, `"allowlist"` или `"disabled"`. По умолчанию: `"allowlist"`.
- `groupAllowFrom`: allowlist ID пользователей для трафика комнат.
- `mentionPatterns`: scoped regex-шаблоны для упоминаний в комнатах. Объект с `{ mode: "allow"|"deny", allowIn: [roomId, ...], denyIn: [roomId, ...] }`. Управляет тем, применяются ли настроенные `agents.list[].groupChat.mentionPatterns` для каждой комнаты.
- `dm.enabled`: когда `false`, игнорировать все личные сообщения. По умолчанию: `true`.
- `dm.policy`: `"pairing"` (по умолчанию), `"allowlist"`, `"open"` или `"disabled"`. Применяется после того, как бот присоединился и классифицировал комнату как личное сообщение; не влияет на обработку приглашений.
- `dm.allowFrom`: allowlist ID пользователей для трафика личных сообщений.
- `dm.sessionScope`: `"per-user"` (по умолчанию) или `"per-room"`.
- `dm.threadReplies`: переопределение только для личных сообщений для потоков ответов (`"off"`, `"inbound"`, `"always"`).
- `allowBots`: принимать сообщения от других настроенных аккаунтов ботов Matrix (`true` или `"mentions"`).
- `allowlistOnly`: когда `true`, принудительно переводит все активные политики личных сообщений (кроме `"disabled"`) и групповые политики `"open"` в `"allowlist"`. Не изменяет политики `"disabled"`.
- `dangerouslyAllowNameMatching`: когда `true`, разрешает поиск отображаемых имен Matrix в каталоге для записей пользовательского allowlist и поиск имен присоединенных комнат для ключей allowlist комнат. Предпочитайте полные ID `@user:server` и ID комнат или алиасы.
- `autoJoin`: `"always"`, `"allowlist"` или `"off"`. По умолчанию: `"off"`. Применяется к каждому приглашению Matrix, включая приглашения в стиле личных сообщений.
- `autoJoinAllowlist`: комнаты/алиасы, разрешенные, когда `autoJoin` равно `"allowlist"`. Записи алиасов разрешаются относительно домашнего сервера, а не относительно состояния, заявленного приглашенной комнатой.
- `contextVisibility`: дополнительная видимость контекста (`"all"` по умолчанию, `"allowlist"`, `"allowlist_quote"`).

### Поведение ответов

- `replyToMode`: `"off"`, `"first"`, `"all"` или `"batched"`.
- `threadReplies`: `"off"`, `"inbound"` или `"always"`.
- `threadBindings`: переопределения для каждого канала для маршрутизации и жизненного цикла сессий, привязанных к потоку.
- `streaming`: `"off"` (по умолчанию), `"partial"`, `"quiet"`, `"progress"` или объектная форма `{ mode, preview: { toolProgress }, progress: { label, labels, maxLines, maxLineChars, toolProgress } }`. `true` ↔ `"partial"`, `false` ↔ `"off"`.
- `blockStreaming`: когда `true`, завершенные блоки ассистента сохраняются как отдельные сообщения прогресса.
- `markdown`: необязательная конфигурация рендеринга Markdown для исходящего текста.
- `responsePrefix`: необязательная строка, добавляемая перед исходящими ответами.
- `textChunkLimit`: размер исходящего фрагмента в символах, когда `chunkMode: "length"`. По умолчанию: `4000`.
- `chunkMode`: `"length"` (по умолчанию, разделяет по количеству символов) или `"newline"` (разделяет по границам строк).
- `historyLimit`: количество последних сообщений комнаты, включаемых как `InboundHistory`, когда сообщение комнаты запускает агента. Возвращается к `messages.groupChat.historyLimit`; эффективное значение по умолчанию `0` (отключено).
- `mediaMaxMb`: ограничение размера медиа в MB для исходящих отправок и входящей обработки.

### Настройки реакций

- `ackReaction`: переопределение ack-реакции для этого канала/аккаунта.
- `ackReactionScope`: переопределение области (`"group-mentions"` по умолчанию, `"group-all"`, `"direct"`, `"all"`, `"none"`, `"off"`).
- `reactionNotifications`: режим уведомлений о входящих реакциях (`"own"` по умолчанию, `"off"`).

### Инструменты и переопределения для отдельных комнат

- `actions`: управление доступом к инструментам для каждого действия (`messages`, `reactions`, `pins`, `profile`, `memberInfo`, `channelInfo`, `verification`).
- `groups`: карта политик для каждой комнаты. Идентификатор сеанса использует стабильный ID комнаты после разрешения. (`rooms` — устаревший псевдоним.)
  - `groups.<room>.account`: ограничивает одну унаследованную запись комнаты конкретной учетной записью.
  - `groups.<room>.enabled`: переключатель для каждой комнаты. Когда `false`, комната игнорируется так, как если бы ее не было в карте.
  - `groups.<room>.requireMention`: переопределение требования упоминания уровня канала для каждой комнаты.
  - `groups.<room>.allowBots`: переопределение настройки уровня канала для каждой комнаты (`true` или `"mentions"`).
  - `groups.<room>.botLoopProtection`: переопределение бюджета защиты от циклов bot-to-bot для каждой комнаты.
  - `groups.<room>.users`: allowlist отправителей для каждой комнаты.
  - `groups.<room>.tools`: переопределения разрешений/запретов инструментов для каждой комнаты.
  - `groups.<room>.autoReply`: переопределение управления упоминаниями для каждой комнаты. `true` отключает требования упоминаний для этой комнаты; `false` принудительно включает их снова.
  - `groups.<room>.skills`: фильтр Skills для каждой комнаты.
  - `groups.<room>.systemPrompt`: фрагмент системного промпта для каждой комнаты.

### Настройки подтверждения exec

- `execApprovals.enabled`: доставляет подтверждения exec через встроенные подсказки Matrix.
- `execApprovals.approvers`: ID пользователей Matrix, которым разрешено подтверждать. Использует `dm.allowFrom` как резервный вариант.
- `execApprovals.target`: `"dm"` (по умолчанию), `"channel"` или `"both"`.
- `execApprovals.agentFilter` / `execApprovals.sessionFilter`: необязательные allowlist агентов/сеансов для доставки.
