## Обзор

Группы рассылки запускают **несколько агентов** для одного входящего сообщения. Каждый агент обрабатывает сообщение в собственной изолированной сессии и публикует свой ответ, поэтому один номер WhatsApp может обслуживать команду специализированных агентов в одном групповом чате или личной переписке.

Группы рассылки проверяются после списков разрешённых каналов и правил активации групп. В группах WhatsApp рассылка выполняется, когда OpenClaw в обычной ситуации должен был бы ответить (например, при упоминании — в зависимости от настроек группы). Она изменяет только то, **какие агенты запускаются**, но не определяет, допускается ли сообщение к обработке.

Рабочий контур контроля качества WhatsApp включает `whatsapp-broadcast-group-fanout`, который проверяет, что одно сообщение с упоминанием в группе может привести к появлению отдельных видимых ответов от двух настроенных агентов.

## Конфигурация

### Базовая настройка

Добавьте раздел верхнего уровня `broadcast` (рядом с `bindings`). Ключи — идентификаторы собеседников WhatsApp, значения — массивы идентификаторов агентов:

- групповые чаты: JID группы (например, `120363403215116621@g.us`)
- личные сообщения: номер телефона отправителя в формате E.164 (например, `+15551234567`)

```json
{
  "broadcast": {
    "120363403215116621@g.us": ["alfred", "baerbel", "assistant3"]
  }
}
```

**Результат:** когда OpenClaw должен ответить в этом чате, он запускает всех трёх агентов.

Каждый указанный идентификатор агента должен присутствовать в `agents.list`: при неизвестных идентификаторах проверка конфигурации сообщает об ошибке, а среда выполнения пропускает их с предупреждением `Broadcast agent <id> not found in agents.list; skipping`.

### Стратегия обработки

`broadcast.strategy` определяет, как агенты обрабатывают сообщение:

| Стратегия              | Поведение                                                                    |
| ---------------------- | --------------------------------------------------------------------------- |
| `parallel` (по умолчанию) | Все агенты обрабатывают сообщение одновременно; ответы приходят в любом порядке. |
| `sequential`           | Агенты обрабатывают сообщение в порядке массива; каждый ждёт завершения предыдущего. |

```json
{
  "broadcast": {
    "strategy": "sequential",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

## Как это работает

### Поток обработки сообщения

<Steps>
  <Step title="Получение входящего сообщения">
    Поступает сообщение из группы WhatsApp или личной переписки.
  </Step>
  <Step title="Маршрутизация и допуск">
    OpenClaw применяет списки разрешённых каналов, правила активации групп и настроенное владение привязками ACP.
  </Step>
  <Step title="Проверка рассылки">
    Если маршрутом не владеет ни одна настроенная привязка ACP, OpenClaw проверяет наличие идентификатора собеседника в `broadcast`.
  </Step>
  <Step title="Если применяется рассылка">
    - Все указанные агенты обрабатывают сообщение.
    - У каждого агента есть собственный ключ сессии и изолированный контекст.
    - Агенты работают параллельно (по умолчанию) или последовательно.
    - Аудиовложения транскрибируются один раз до распределения, поэтому агенты используют общую расшифровку вместо отдельных вызовов STT.

  </Step>
  <Step title="Если рассылка не применяется">
    OpenClaw передаёт сообщение по обычному маршруту или по маршруту настроенной сессии ACP, выбранному во время маршрутизации.
  </Step>
</Steps>

<Note>
Группы рассылки не обходят списки разрешённых каналов или правила активации групп (упоминания, команды и т. д.). Они изменяют только то, _какие агенты запускаются_, когда сообщение допускается к обработке.
</Note>

### Изоляция сессий

Каждый агент в группе рассылки имеет полностью отдельные:

- **Ключи сессий** (`agent:alfred:whatsapp:group:120363...` и `agent:baerbel:whatsapp:group:120363...`)
- **Историю переписки** (агент не видит ответы других агентов)
- **Рабочее пространство** (отдельные песочницы, если они настроены)
- **Доступ к инструментам** (разные списки разрешений и запретов)
- **Память и контекст** (отдельные `IDENTITY.md`, `SOUL.md` и т. д.)

Есть одно намеренно сделанное исключение: **буфер контекста группы** (последние сообщения группы, используемые как контекст) является общим для каждого собеседника, поэтому при запуске все агенты рассылки видят один и тот же контекст. После завершения распределённой обработки он очищается один раз.

Это позволяет назначать агентам разные характеры, модели, навыки и права доступа к инструментам (например, только чтение или чтение и запись).

### Пример: изолированные сессии

В группе `120363403215116621@g.us` с агентами `["alfred", "baerbel"]`:

## Сценарии использования

- **Команды специализированных агентов**: группа разработчиков, в которой `code-reviewer`, `security-auditor`, `test-generator` и `docs-checker` отвечают на одно и то же сообщение каждый со своей точки зрения.
- **Многоязычная поддержка**: один чат поддержки с агентами `support-en`, `support-de` и `support-es`, отвечающими на соответствующих языках.
- **Контроль качества**: `support-agent` отвечает, а `qa-agent` проверяет ответ и реагирует только при обнаружении проблем.
- **Автоматизация задач**: `task-tracker`, `time-logger` и `report-generator` обрабатывают одно и то же обновление статуса.

    Назначайте каждому агенту одну чёткую обязанность (`formatter`, `linter`, `tester`) вместо одного универсального агента «dev-helper».
  </Accordion>
  <Accordion title="2. Используйте понятные идентификаторы и имена">
    ```json
    {
      "agents": {
        "list": [
          { "id": "security-scanner", "name": "Security Scanner" },
          { "id": "code-formatter", "name": "Code Formatter" },
          { "id": "test-generator", "name": "Test Generator" }
        ]
      }
    }
    ```
  </Accordion>
  <Accordion title="3. Настройте разные права доступа к инструментам">
    ```json
    {
      "agents": {
        "list": [
          { "id": "reviewer", "tools": { "allow": ["read", "exec"] } },
          { "id": "fixer", "tools": { "allow": ["read", "write", "edit", "exec"] } }
        ]
      }
    }
    ```

    `reviewer` имеет доступ только для чтения. `fixer` может читать и записывать данные.

  </Accordion>
  <Accordion title="4. Контролируйте производительность">
    При большом количестве агентов используйте `"strategy": "parallel"` (по умолчанию), ограничивайте группы рассылки несколькими агентами и выбирайте более быстрые модели для простых агентов.
  </Accordion>
  <Accordion title="5. Сбои остаются изолированными">
    Сбои агентов происходят независимо. Ошибка одного агента записывается в журнал (`Broadcast agent <id> failed: ...`) и не блокирует остальных.
  </Accordion>
</AccordionGroup>

## Совместимость

### Поставщики

В настоящее время группы рассылки реализованы только для WhatsApp (веб-канала). Другие каналы игнорируют конфигурацию `broadcast`.

### Маршрутизация

Группы рассылки работают совместно с существующей маршрутизацией:

```json
{
  "bindings": [
    {
      "match": { "channel": "whatsapp", "peer": { "kind": "group", "id": "GROUP_A" } },
      "agentId": "alfred"
    }
  ],
  "broadcast": {
    "GROUP_B": ["agent1", "agent2"]
  }
}
```

- `GROUP_A`: отвечает только alfred (обычная маршрутизация).
- `GROUP_B`: отвечают agent1 И agent2 (рассылка).

<Note>
**Приоритет:** `broadcast` имеет приоритет над обычными привязками маршрутов. Настроенные привязки ACP (`bindings[].type="acp"`) являются исключительными: при совпадении OpenClaw передаёт сообщение настроенной сессии ACP вместо распределённой рассылки.
</Note>

## Устранение неполадок

<AccordionGroup>
  <Accordion title="Агенты не отвечают">
    **Проверьте следующее:**

    1. Идентификаторы агентов присутствуют в `agents.list` (проверка конфигурации отклоняет неизвестные идентификаторы).
    2. Идентификатор собеседника имеет правильный формат (JID группы, например `120363403215116621@g.us`, или E.164, например `+15551234567`, для личных сообщений).
    3. Сообщение прошло обычные проверки допуска (правила упоминаний и активации продолжают действовать).

    **Отладка:**

    ```bash
    openclaw logs --follow | grep -i broadcast
    ```

    При успешном распределении в журнал записывается `Broadcasting message to <n> agents (<strategy>)`.

  </Accordion>
  <Accordion title="Отвечает только один агент">
    **Причина:** идентификатор собеседника может присутствовать в обычных привязках маршрутов, но отсутствовать в `broadcast`, либо он может соответствовать исключительной настроенной привязке ACP.

    **Решение:** добавьте собеседников, привязанных к обычным маршрутам, в конфигурацию рассылки или удалите либо измените настроенную привязку ACP, если требуется распределённая рассылка.

  </Accordion>
  <Accordion title="Проблемы с производительностью">
    Если при большом количестве агентов обработка выполняется медленно, сократите число агентов в группе, используйте более лёгкие модели и проверьте время запуска песочницы.
  </Accordion>
</AccordionGroup>

## Справочник API

### Схема конфигурации

```typescript
interface OpenClawConfig {
  broadcast?: {
    strategy?: "parallel" | "sequential";
    [peerId: string]: string[];
  };
}
```

### Поля

<ParamField path="strategy" type='"parallel" | "sequential"' default='"parallel"'>
  Способ обработки агентами. `parallel` запускает всех агентов одновременно; `sequential` запускает их в порядке массива.
</ParamField>
<ParamField path="[peerId]" type="string[]">
  JID группы WhatsApp или номер телефона в формате E.164. Значение представляет собой массив идентификаторов агентов, каждый из которых должен обрабатывать сообщения от этого собеседника.
</ParamField>

## Ограничения

1. **Максимальное количество агентов:** жёсткого ограничения нет, но большое количество агентов (10 и более) может замедлить работу.
2. **Общий контекст:** агенты не видят ответы друг друга (так задумано).
3. **Порядок сообщений:** при параллельной обработке ответы могут приходить в любом порядке.
4. **Ограничения частоты:** все ответы отправляются из одной учётной записи WhatsApp, поэтому ответ каждого агента учитывается в общих ограничениях частоты WhatsApp.
