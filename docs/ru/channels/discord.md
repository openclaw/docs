## Быстрая настройка

Создайте приложение Discord с ботом, добавьте бота на свой сервер и сопрягите его с OpenClaw. По возможности используйте частный сервер; при необходимости сначала [создайте его](https://support.discord.com/hc/en-us/articles/204849977-How-do-I-create-a-server) (**Create My Own > For me and my friends**).

<Steps>
  <Step title="Создайте приложение Discord и бота">
    На [портале разработчика Discord](https://discord.com/developers/applications) нажмите **New Application** и задайте приложению имя (например, «OpenClaw»).

    Откройте **Bot** на боковой панели и задайте в поле **Username** имя вашего агента.

  </Step>

  <Step title="Включите привилегированные намерения">
    На странице **Bot** в разделе **Privileged Gateway Intents** включите:

    - **Message Content Intent** (обязательно)
    - **Server Members Intent** (рекомендуется; обязательно для списков разрешённых ролей, сопоставления имён с идентификаторами и групп доступа к аудитории каналов)
    - **Presence Intent** (необязательно; только для обновлений статуса присутствия)

  </Step>

  <Step title="Скопируйте токен бота">
    На странице **Bot** нажмите **Reset Token** и скопируйте токен.

    <Note>
    Несмотря на название, это создаёт ваш первый токен — ничего не «сбрасывается».
    </Note>

  </Step>

  <Step title="Создайте URL приглашения и добавьте бота на сервер">
    Откройте **OAuth2** на боковой панели. В разделе **OAuth2 URL Generator** включите области доступа:

    - `bot`
    - `applications.commands`

    В появившемся разделе **Bot Permissions** включите как минимум:

    **General Permissions**
      - View Channels

    **Text Permissions**
      - Send Messages
      - Read Message History
      - Embed Links
      - Attach Files
      - Add Reactions (необязательно)

    Это базовый набор для обычных текстовых каналов. Если бот будет публиковать сообщения в ветках, включая процессы в каналах-форумах или медиаканалах, которые создают или продолжают ветку, также включите **Send Messages in Threads**.

    Скопируйте созданный URL, откройте его в браузере, выберите свой сервер и нажмите **Continue**. Теперь бот должен появиться на вашем сервере.

  </Step>

  <Step title="Включите режим разработчика и соберите идентификаторы">
    В приложении Discord включите режим разработчика, чтобы можно было копировать идентификаторы:

    1. **User Settings** (значок шестерёнки) → **Developer** → включите **Developer Mode**
       *(на мобильном устройстве: **App Settings** → **Advanced**)*
    2. Нажмите правой кнопкой мыши значок своего сервера → **Copy Server ID**
    3. Нажмите правой кнопкой мыши свой аватар → **Copy User ID**

    Сохраните идентификаторы сервера и пользователя вместе с токеном бота: далее понадобятся все три значения.

  </Step>

  <Step title="Разрешите личные сообщения от участников сервера">
    Для работы сопряжения Discord должен разрешать боту отправлять вам личные сообщения. Нажмите правой кнопкой мыши значок своего сервера → **Privacy Settings** → включите **Direct Messages**.

    Не отключайте этот параметр, если используете личные сообщения Discord с OpenClaw. Если вы используете только каналы сервера, после сопряжения его можно отключить.

  </Step>

  <Step title="Безопасно задайте токен бота (не отправляйте его в чат)">
    Токен бота является секретом. Задайте его на машине, где работает OpenClaw, прежде чем отправлять сообщение агенту:

```bash
export DISCORD_BOT_TOKEN="YOUR_BOT_TOKEN"
cat > discord.patch.json5 <<'JSON5'
{
  channels: {
    discord: {
      enabled: true,
      token: { source: "env", provider: "default", id: "DISCORD_BOT_TOKEN" },
    },
  },
}
JSON5
openclaw config patch --file ./discord.patch.json5 --dry-run
openclaw config patch --file ./discord.patch.json5
openclaw gateway
```

    Если OpenClaw уже работает как фоновая служба, перезапустите её через приложение OpenClaw для Mac либо остановите и снова запустите процесс `openclaw gateway run`.
    При установке в качестве управляемой службы выполните `openclaw gateway install` в оболочке, где задана переменная `DISCORD_BOT_TOKEN`, либо сохраните переменную в `~/.openclaw/.env`, чтобы после перезапуска служба могла разрешить ссылку на секрет из переменной окружения.
    Если Discord блокирует или ограничивает частоту выполняемого при запуске запроса сведений о приложении для вашего узла, задайте идентификатор приложения/клиента с портала разработчика, чтобы при запуске пропустить этот REST-вызов: `channels.discord.applicationId` для учётной записи по умолчанию или `channels.discord.accounts.<accountId>.applicationId` для каждого бота.

  </Step>

  <Step title="Настройте OpenClaw и выполните сопряжение">

    <Tabs>
      <Tab title="Попросите агента">
        Начните чат с агентом OpenClaw в уже существующем канале (например, Telegram) и сообщите ему нужные данные. Если Discord — ваш первый канал, используйте вместо этого вкладку CLI / конфигурации.

        > «Я уже задал токен бота Discord в конфигурации. Заверши настройку Discord с идентификатором пользователя `<user_id>` и идентификатором сервера `<server_id>`».
      </Tab>
      <Tab title="CLI / конфигурация">
        Конфигурация на основе файла:

```json5
{
  channels: {
    discord: {
      enabled: true,
      token: {
        source: "env",
        provider: "default",
        id: "DISCORD_BOT_TOKEN",
      },
    },
  },
}
```

        Резервное значение из переменной окружения для учётной записи по умолчанию:

```bash
DISCORD_BOT_TOKEN=...
```

        Для сценарной или удалённой настройки запишите тот же блок JSON5 с помощью `openclaw config patch --file ./discord.patch.json5 --dry-run`, а затем повторите команду без `--dry-run`. Также поддерживаются строки `token` в открытом виде и значения SecretRef для `channels.discord.token` у поставщиков env/file/exec. См. раздел [Управление секретами](/ru/gateway/secrets).

        При использовании нескольких ботов Discord храните токен и идентификатор приложения каждого бота в его учётной записи. Значение верхнего уровня `channels.discord.applicationId` наследуется учётными записями, поэтому задавайте его там только в том случае, если все учётные записи используют один и тот же идентификатор приложения.

```json5
{
  channels: {
    discord: {
      enabled: true,
      accounts: {
        personal: {
          token: { source: "env", provider: "default", id: "DISCORD_PERSONAL_TOKEN" },
          applicationId: "111111111111111111",
        },
        work: {
          token: { source: "env", provider: "default", id: "DISCORD_WORK_TOKEN" },
          applicationId: "222222222222222222",
        },
      },
    },
  },
}
```

      </Tab>
    </Tabs>

  </Step>

  <Step title="Подтвердите первое сопряжение через личные сообщения">
    После запуска Gateway отправьте боту личное сообщение в Discord. В ответ он пришлёт код сопряжения.

    <Tabs>
      <Tab title="Попросите агента">
        Отправьте код сопряжения агенту через существующий канал:

        > «Подтверди этот код сопряжения Discord: `<CODE>`»
      </Tab>
      <Tab title="CLI">

```bash
openclaw pairing list discord
openclaw pairing approve discord <CODE>
```

      </Tab>
    </Tabs>

    Срок действия кодов сопряжения — 1 час. После подтверждения общайтесь с агентом в личных сообщениях Discord.

  </Step>
</Steps>

<Note>
Разрешение токенов учитывает учётную запись. Значения токенов из конфигурации имеют приоритет над резервным значением из переменной окружения, а `DISCORD_BOT_TOKEN` используется только для учётной записи по умолчанию.
Если для двух включённых учётных записей Discord разрешается один и тот же токен бота, OpenClaw запускает для этого токена только один монитор Gateway: токен из конфигурации имеет приоритет над резервным значением из переменной окружения; в остальных случаях используется первая включённая учётная запись, а дублирующая учётная запись помечается как отключённая с причиной `duplicate bot token`.
Для расширенных исходящих вызовов (инструмент сообщений/действия канала) явно указанный для отдельного вызова `token` используется именно в этом вызове. Это относится к отправке и действиям чтения/проверки (чтение, поиск, получение, ветки, закреплённые сообщения, разрешения). Политика учётной записи и параметры повторных попыток по-прежнему берутся из выбранной учётной записи в активном снимке среды выполнения.
</Note>

## Рекомендуется: настройте рабочее пространство сервера

После того как заработают личные сообщения, сервер можно превратить в полноценное рабочее пространство, где каждый канал получает отдельный сеанс агента с собственным контекстом. Рекомендуется для частных серверов, где находитесь только вы и ваш бот.

<Steps>
  <Step title="Добавьте сервер в список разрешённых серверов">
    Это позволит агенту отвечать в любом канале вашего сервера, а не только в личных сообщениях.

    <Tabs>
      <Tab title="Попросите агента">
        > «Добавь идентификатор моего сервера Discord `<server_id>` в список разрешённых серверов»
      </Tab>
      <Tab title="Конфигурация">

```json5
{
  channels: {
    discord: {
      groupPolicy: "allowlist",
      guilds: {
        YOUR_SERVER_ID: {
          requireMention: true,
          users: ["YOUR_USER_ID"],
        },
      },
    },
  },
}
```

      </Tab>
    </Tabs>

  </Step>

  <Step title="Разрешите ответы без @упоминания">
    По умолчанию агент отвечает в каналах сервера только при @упоминании. На частном сервере, вероятно, удобнее отвечать на каждое сообщение.

    По умолчанию обычные ответы в каналах сервера публикуются автоматически. Для общих постоянно активных комнат включите `messages.groupChat.visibleReplies: "message_tool"`, чтобы агент мог молча следить за разговором и публиковать сообщения только тогда, когда сочтёт ответ в канале полезным. Лучше всего это работает с моделями последнего поколения, надёжно использующими инструменты, например GPT-5.6 Sol. Фоновые события комнаты не вызывают публикаций, если инструмент ничего не отправляет. Полную конфигурацию режима наблюдения см. в разделе [Фоновые события комнаты](/ru/channels/ambient-room-events).

    Если Discord показывает индикатор набора текста и в журналах отображается расход токенов, но сообщение не публикуется, проверьте, был ли ход настроен как фоновое событие комнаты или включены ли видимые ответы через инструмент сообщений.

    <Tabs>
      <Tab title="Попросите агента">
        > «Разреши моему агенту отвечать на этом сервере без обязательного @упоминания»
      </Tab>
      <Tab title="Конфигурация">
        Задайте `requireMention: false` в конфигурации сервера:

```json5
{
  channels: {
    discord: {
      guilds: {
        YOUR_SERVER_ID: {
          requireMention: false,
        },
      },
    },
  },
}
```

        Чтобы для видимых ответов в группах и каналах требовалась отправка через инструмент сообщений, задайте `messages.groupChat.visibleReplies: "message_tool"`.

      </Tab>
    </Tabs>

  </Step>

  <Step title="Спланируйте использование памяти в каналах сервера">
    Долговременная память (`MEMORY.md`) автоматически загружается только в сеансах личных сообщений; в каналах сервера она не загружается.

    <Tabs>
      <Tab title="Попросите агента">
        > «Когда я задаю вопросы в каналах Discord, используй memory_search или memory_get, если тебе нужен долгосрочный контекст из MEMORY.md».
      </Tab>
      <Tab title="Вручную">
        Для общего контекста во всех каналах поместите постоянные инструкции в `AGENTS.md` или `USER.md` (они внедряются в каждый сеанс). Храните долгосрочные заметки в `MEMORY.md` и обращайтесь к ним по мере необходимости с помощью инструментов памяти.
      </Tab>
    </Tabs>

  </Step>
</Steps>

Теперь создайте каналы и начинайте общение. Агент видит название канала, а каждый канал представляет собой изолированный сеанс — создайте `#coding`, `#home`, `#research` или любые другие каналы, подходящие для вашего рабочего процесса.

## Модель среды выполнения

- Gateway управляет подключением к Discord.
- Маршрутизация ответов детерминирована: ответы на входящие сообщения Discord возвращаются в Discord.
- Метаданные сервера и канала Discord добавляются в запрос к модели как недоверенный контекст, а не как видимый пользователю префикс ответа. Если модель скопирует эту оболочку в ответ, OpenClaw удалит скопированные метаданные из исходящих ответов и будущего контекста повторного воспроизведения.
- По умолчанию (`session.dmScope=main`) личные чаты используют общий основной сеанс агента (`agent:main:main`).
- Каналы сервера имеют изолированные ключи сеансов (`agent:<agentId>:discord:channel:<channelId>`).
- Групповые личные сообщения по умолчанию игнорируются (`channels.discord.dm.groupEnabled=false`).
- Встроенные команды с косой чертой выполняются в изолированных сеансах команд (`agent:<agentId>:discord:slash:<userId>`), но при этом передают `CommandTargetSessionKey` в сеанс разговора, выбранный маршрутизацией.
- При доставке в Discord текстовых объявлений Cron/Heartbeat отправляется только окончательный видимый ответ ассистента, причём однократно. Медиаданные и структурированные полезные нагрузки компонентов по-прежнему отправляются несколькими сообщениями, если агент создаёт несколько полезных нагрузок для доставки.

## Каналы-форумы

Каналы-форумы и медиаканалы Discord принимают публикации только в ветках. OpenClaw поддерживает два способа их создания:

- Отправьте сообщение в родительский канал форума (`channel:<forumId>`), чтобы автоматически создать ветку. Заголовком ветки станет первая непустая строка сообщения (обрезанная до установленного Discord ограничения в 100 символов для имени ветки).
- Используйте `openclaw message thread create`, чтобы создать ветку напрямую. Для каналов форума не передавайте `--message-id`.

Отправьте сообщение в родительский канал форума, чтобы создать ветку:

```bash
openclaw message send --channel discord --target channel:<forumId> \
  --message "Topic title\nBody of the post"
```

Создайте ветку форума явно:

```bash
openclaw message thread create --channel discord --target channel:<forumId> \
  --thread-name "Topic title" --message "Body of the post"
```

Родительские каналы форума не принимают компоненты Discord. Если нужны компоненты, отправляйте сообщение в саму ветку (`channel:<threadId>`).

## Интерактивные компоненты

OpenClaw поддерживает контейнеры компонентов Discord v2 для сообщений агента. Используйте инструмент сообщений с полезной нагрузкой `components`. Результаты взаимодействий возвращаются агенту как обычные входящие сообщения и обрабатываются в соответствии с существующими настройками Discord `replyToMode`.

Поддерживаемые блоки:

- `text`, `section`, `separator`, `actions`, `media-gallery`, `file`
- Строки действий могут содержать до 5 кнопок или одно меню выбора
- Типы выбора: `string`, `user`, `role`, `mentionable`, `channel`

По умолчанию компоненты можно использовать только один раз. Установите `components.reusable=true`, чтобы кнопки, меню выбора и формы можно было использовать многократно до истечения срока их действия.

Чтобы ограничить круг пользователей, которые могут нажать кнопку, задайте для неё `allowedUsers` (идентификаторы или теги пользователей Discord либо `*`). Остальные пользователи получат эфемерное сообщение об отказе.

По умолчанию срок действия обратных вызовов компонентов истекает через 30 минут. Чтобы изменить время хранения реестра обратных вызовов для учётной записи по умолчанию, задайте `channels.discord.agentComponents.ttlMs`, а для отдельной учётной записи — `channels.discord.accounts.<accountId>.agentComponents.ttlMs`. Значение задаётся в миллисекундах, должно быть положительным целым числом и ограничено максимумом `86400000` (24 часа). Более длительный срок действия подходит для процессов проверки и утверждения, в которых кнопки должны оставаться доступными, но увеличивает период, в течение которого старое сообщение Discord всё ещё может инициировать действие. Выбирайте минимально достаточный срок действия и сохраняйте значение по умолчанию, если срабатывание устаревших обратных вызовов может оказаться неожиданным.

Slash-команды `/model` и `/models` открывают интерактивное средство выбора модели с раскрывающимися списками поставщика, модели и совместимой среды выполнения, а также этапом Submit. Команда `/models add` устарела и вместо регистрации моделей из чата возвращает сообщение об устаревании. Ответ средства выбора является эфемерным и доступен только вызвавшему его пользователю. Меню выбора Discord ограничены 25 вариантами, поэтому добавьте записи `provider/*` в `agents.defaults.models`, если хотите, чтобы средство выбора показывало динамически обнаруженные модели только для выбранных поставщиков, например `openai` или `vllm`.

Вложения файлов:

- Блоки `file` должны ссылаться на вложение (`attachment://<filename>`)
- Передайте вложение через `media`/`path`/`filePath` (один файл); для нескольких файлов используйте `media-gallery`
- Используйте `filename`, чтобы переопределить имя загружаемого файла, когда оно должно совпадать со ссылкой на вложение

Модальные формы:

- Типы полей: `text`, `checkbox`, `radio`, `select`, `role-select`, `user-select`
- OpenClaw автоматически добавляет кнопку запуска

## Управление доступом и маршрутизация

<Tabs>
  <Tab title="Политика личных сообщений">
    `channels.discord.dmPolicy` управляет доступом к личным сообщениям. `channels.discord.allowFrom` — канонический список разрешённых отправителей личных сообщений.

    - `pairing` (по умолчанию)
    - `allowlist` (требуется хотя бы один отправитель в `allowFrom`)
    - `open` (требуется, чтобы `channels.discord.allowFrom` содержал `"*"`)
    - `disabled`

    Если политика личных сообщений не открыта, неизвестные пользователи блокируются (либо им предлагается выполнить сопряжение в режиме `pairing`).

    Приоритет настроек для нескольких учётных записей:

    - `channels.discord.accounts.default.allowFrom` применяется только к учётной записи `default`.
    - Для одной учётной записи `allowFrom` имеет приоритет над устаревшим `dm.allowFrom`.
    - Именованные учётные записи наследуют `channels.discord.allowFrom`, если у них не заданы собственные `allowFrom` и устаревший `dm.allowFrom`.
    - Именованные учётные записи не наследуют `channels.discord.accounts.default.allowFrom`.

    Устаревшие `channels.discord.dm.policy` и `channels.discord.dm.allowFrom` по-прежнему считываются для совместимости. `openclaw doctor --fix` переносит их в `dmPolicy` и `allowFrom`, если это можно сделать без изменения доступа.

    Формат цели личного сообщения для доставки:

    - `user:<id>`
    - упоминание `<@id>`

    Простые числовые идентификаторы обычно интерпретируются как идентификаторы каналов, когда активно значение канала по умолчанию, однако идентификаторы, перечисленные в действующем для учётной записи списке `allowFrom` личных сообщений, для совместимости рассматриваются как цели личных сообщений пользователям.

  </Tab>

  <Tab title="Группы доступа">
    Для авторизации личных сообщений и текстовых команд Discord можно использовать динамические записи `accessGroup:<name>` в `channels.discord.allowFrom`.

    Имена групп доступа являются общими для всех каналов сообщений. Используйте `type: "message.senders"` для статической группы, участники которой задаются в обычном синтаксисе `allowFrom` каждого канала, или `type: "discord.channelAudience"`, когда членство должно динамически определяться текущей аудиторией `ViewChannel` канала Discord. Общее поведение групп доступа: [Группы доступа](/ru/channels/access-groups).

```json5
{
  accessGroups: {
    operators: {
      type: "message.senders",
      members: {
        "*": ["global-owner-id"],
        discord: ["discord:123456789012345678"],
        telegram: ["987654321"],
      },
    },
  },
  channels: {
    discord: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:operators"],
    },
  },
}
```

    У текстового канала Discord нет отдельного списка участников. `type: "discord.channelAudience"` моделирует членство следующим образом: отправитель личного сообщения является участником настроенного сервера и после применения ролей и переопределений канала в данный момент имеет действующее разрешение `ViewChannel` для настроенного канала.

    Пример: разрешить всем, кто видит `#maintainers`, отправлять личные сообщения боту, сохранив личные сообщения закрытыми для остальных.

```json5
{
  accessGroups: {
    maintainers: {
      type: "discord.channelAudience",
      guildId: "1456350064065904867",
      channelId: "1456744319972282449",
      membership: "canViewChannel",
    },
  },
  channels: {
    discord: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:maintainers"],
    },
  },
}
```

    Динамические и статические записи можно сочетать:

```json5
{
  accessGroups: {
    maintainers: {
      type: "discord.channelAudience",
      guildId: "1456350064065904867",
      channelId: "1456744319972282449",
    },
  },
  channels: {
    discord: {
      dmPolicy: "allowlist",
      allowFrom: ["accessGroup:maintainers", "discord:123456789012345678"],
    },
  },
}
```

    При ошибке поиска доступ запрещается. Если Discord возвращает `Missing Access`, не удаётся найти участника или канал принадлежит другому серверу, отправитель личного сообщения считается неавторизованным.

    При использовании групп доступа на основе аудитории канала включите **Server Members Intent** на портале разработчика Discord. Личные сообщения не содержат состояние участника сервера, поэтому во время авторизации OpenClaw получает сведения об участнике через Discord REST.

  </Tab>

  <Tab title="Политика сервера">
    Обработка серверов управляется параметром `channels.discord.groupPolicy`:

    - `open`
    - `allowlist`
    - `disabled`

    Безопасное базовое значение при наличии `channels.discord` — `allowlist`.

    Поведение `allowlist`:

    - сервер должен соответствовать `channels.discord.guilds` (предпочтителен `id`, также принимается краткое имя)
    - необязательные списки разрешённых отправителей: `users` (рекомендуются стабильные идентификаторы) и `roles` (только идентификаторы ролей); если настроен хотя бы один из них, отправители допускаются при совпадении с `users` ИЛИ `roles`
    - прямое сопоставление имён и тегов по умолчанию отключено; включайте `channels.discord.dangerouslyAllowNameMatching: true` только как аварийный режим совместимости
    - для `users` поддерживаются имена и теги, однако идентификаторы безопаснее; `openclaw security audit` предупреждает об использовании записей с именами или тегами
    - если для сервера настроен блок `channels`, доступ к неуказанным каналам запрещается
    - если у сервера нет блока `channels`, разрешаются все каналы этого сервера из списка разрешённых

    Пример:

```json5
{
  channels: {
    discord: {
      groupPolicy: "allowlist",
      guilds: {
        "123456789012345678": {
          requireMention: true,
          ignoreOtherMentions: true,
          users: ["987654321098765432"],
          roles: ["123456789012345678"],
          channels: {
            general: { enabled: true },
            help: { enabled: true, requireMention: true },
          },
        },
      },
    },
  },
}
```

    Устаревший ключ `allow` для отдельных каналов переносится в `enabled` командой `openclaw doctor --fix`.

    Если задан только `DISCORD_BOT_TOKEN`, а блок `channels.discord` не создан, резервным значением среды выполнения будет `groupPolicy="allowlist"` (с предупреждением в журналах), даже если `channels.defaults.groupPolicy` имеет значение `open`.

  </Tab>

  <Tab title="Упоминания и групповые личные сообщения">
    По умолчанию сообщения сервера обрабатываются только при наличии упоминания.

    Обнаружение упоминаний включает:

    - явное упоминание бота
    - настроенные шаблоны упоминаний (`agents.list[].groupChat.mentionPatterns`, с резервным использованием `messages.groupChat.mentionPatterns`)
    - неявное поведение ответа боту в поддерживаемых случаях

    При создании исходящих сообщений Discord используйте канонический синтаксис упоминаний: `<@USER_ID>` для пользователей, `<#CHANNEL_ID>` для каналов и `<@&ROLE_ID>` для ролей. Не используйте устаревшую форму упоминания по псевдониму `<@!USER_ID>`.

    `requireMention` настраивается для каждого сервера или канала (`channels.discord.guilds...`).
    `ignoreOtherMentions` позволяет отбрасывать сообщения, в которых упомянут другой пользователь или роль, но не бот (за исключением @everyone/@here).

    Групповые личные сообщения:

    - по умолчанию игнорируются (`dm.groupEnabled=false`)
    - необязательный список разрешённых через `dm.groupChannels` (идентификаторы или краткие имена каналов)

  </Tab>
</Tabs>

### Маршрутизация агентов на основе ролей

Используйте `bindings[].match.roles`, чтобы направлять участников сервера Discord к разным агентам по идентификатору роли. Привязки на основе ролей принимают только идентификаторы ролей и проверяются после привязок по собеседнику или родительскому собеседнику, но до привязок только по серверу. Если в привязке также заданы другие поля сопоставления (например, `peer` + `guildId` + `roles`), должны совпасть все настроенные поля.

```json5
{
  bindings: [
    {
      agentId: "opus",
      match: {
        channel: "discord",
        guildId: "123456789012345678",
        roles: ["111111111111111111"],
      },
    },
    {
      agentId: "sonnet",
      match: {
        channel: "discord",
        guildId: "123456789012345678",
      },
    },
  ],
}
```

## Встроенные команды и авторизация команд

- `commands.native` по умолчанию имеет значение `"auto"` и включён для Discord.
- Переопределение для отдельного канала: `channels.discord.commands.native`.
- `commands.native=false` пропускает регистрацию и очистку слеш-команд Discord при запуске. Ранее зарегистрированные команды могут оставаться видимыми в Discord, пока вы не удалите их из приложения Discord.
- Авторизация встроенных команд использует те же списки разрешений и политики Discord, что и обычная обработка сообщений.
- Команды могут оставаться видимыми в интерфейсе Discord для неавторизованных пользователей; при выполнении OpenClaw проверяет авторизацию и отвечает «нет авторизации».
- Настройки слеш-команд по умолчанию: `ephemeral: true` (`channels.discord.slashCommand.ephemeral`).

Каталог команд и описание их поведения см. в разделе [Слеш-команды](/ru/tools/slash-commands).

## Подробности функций

<AccordionGroup>
  <Accordion title="Теги ответов и встроенные ответы">
    Discord поддерживает теги ответов в выводе агента:

    - `[[reply_to_current]]`
    - `[[reply_to:<id>]]`

    Управляется параметром `channels.discord.replyToMode`:

    - `off` (по умолчанию): без неявной привязки ответов к ветке; явные теги `[[reply_to_*]]` по-прежнему учитываются
    - `first`: добавляет неявную встроенную ссылку на исходное сообщение к первому исходящему сообщению Discord в рамках хода
    - `all`: добавляет её к каждому исходящему сообщению
    - `batched`: добавляет её только тогда, когда входящее событие представляло собой объединённый после задержки пакет из нескольких сообщений — это удобно, если встроенные ответы нужны преимущественно для неоднозначных чатов с потоком быстрых сообщений, а не для каждого хода с одним сообщением

    Идентификаторы сообщений передаются в контекст и историю, чтобы агенты могли адресовать ответы конкретным сообщениям.

  </Accordion>

  <Accordion title="Предпросмотр ссылок">
    По умолчанию Discord создаёт для URL расширенные встроенные карточки ссылок. OpenClaw по умолчанию отключает эти автоматически создаваемые карточки в исходящих сообщениях Discord, поэтому отправленные агентом URL остаются обычными ссылками, если вы явно не включите их:

```json5
{
  channels: {
    discord: {
      suppressEmbeds: false,
    },
  },
}
```

    Чтобы переопределить настройку для одной учётной записи, задайте `channels.discord.accounts.<id>.suppressEmbeds`. При отправке через инструмент сообщений агент также может передать `suppressEmbeds: false` для отдельного сообщения. Явно заданные полезные нагрузки Discord `embeds` не отключаются настройкой предпросмотра ссылок по умолчанию.

  </Accordion>

  <Accordion title="Предпросмотр потокового ответа в реальном времени">
    OpenClaw может передавать черновик ответа в потоковом режиме, отправляя временное сообщение и редактируя его по мере поступления текста. `channels.discord.streaming.mode` принимает значения `off` | `partial` | `block` | `progress` (по умолчанию, если не задан ключ `streaming` или устаревший `streamMode`). `streamMode` — устаревший псевдоним; выполните `openclaw doctor --fix`, чтобы преобразовать сохранённую конфигурацию в каноническую вложенную структуру `streaming`.

```json5
{
  channels: {
    discord: {
      streaming: {
        mode: "progress",
        progress: {
          label: "auto",
          maxLines: 8,
          maxLineChars: 120,
          toolProgress: true,
          commentary: false,
        },
      },
    },
  },
}
```

    - `off` отключает редактирование предпросмотра в Discord.
    - `partial` редактирует одно сообщение предпросмотра по мере поступления токенов.
    - `block` выводит фрагменты размером с черновик; размер и точки разрыва настраиваются через `streaming.preview.chunk` (`minChars`, `maxChars`, `breakPreference`) с ограничением до `textChunkLimit`. Если блочная потоковая передача включена явно, OpenClaw пропускает поток предпросмотра, чтобы избежать дублирования потоковой передачи.
    - `progress` сохраняет один редактируемый черновик состояния и обновляет его сведениями о ходе работы инструментов до окончательной доставки; общая начальная метка представляет собой прокручиваемую строку, поэтому после появления достаточного объёма данных она уходит за пределы видимой области вместе с остальным содержимым.
    - Окончательные ответы с медиафайлами, ошибками или явными ссылками на исходные сообщения отменяют ожидающие изменения предпросмотра.
    - `streaming.preview.toolProgress` (по умолчанию `true`) определяет, используют ли обновления инструментов и хода выполнения повторно сообщение предпросмотра.
    - Строки инструментов и хода выполнения отображаются в компактном виде: эмодзи, заголовок и, при наличии, подробности, например `🛠️ Bash: run tests` или `🔎 Web Search: for "query"`.
    - `streaming.progress.commentary` (по умолчанию `false`) включает пояснения и вводный текст ассистента во временном черновике хода выполнения. Перед отображением пояснения очищаются, остаются временными и не изменяют доставку окончательного ответа.
    - `streaming.progress.maxLineChars` задаёт лимит символов на строку в предпросмотре хода выполнения. Обычный текст сокращается по границам слов; в сведениях о командах и путях сохраняются полезные окончания.
    - `streaming.preview.commandText` / `streaming.progress.commandText` определяет детализацию команд и их выполнения в компактных строках хода выполнения: `raw` (по умолчанию) или `status` (только метка инструмента).

    Чтобы скрыть исходный текст команд и их выполнения, сохранив компактные строки хода выполнения:

    ```json
    {
      "channels": {
        "discord": {
          "streaming": {
            "mode": "progress",
            "progress": {
              "toolProgress": true,
              "commandText": "status"
            }
          }
        }
      }
    }
    ```

    Потоковый предпросмотр поддерживает только текст; ответы с медиафайлами доставляются обычным способом.

  </Accordion>

  <Accordion title="История, контекст и поведение веток">
    Контекст истории сервера:

    - значение `channels.discord.historyLimit` по умолчанию — `20`
    - резервное значение: `messages.groupChat.historyLimit`
    - `0` отключает историю

    Управление историей личных сообщений:

    - `channels.discord.dmHistoryLimit`
    - `channels.discord.dms["<user_id>"].historyLimit`

    Поведение веток:

    - Ветки Discord маршрутизируются как сеансы каналов и наследуют конфигурацию родительского канала, если она не переопределена.
    - Сеансы веток наследуют выбранную на уровне сеанса родительского канала модель `/model` только как резервный вариант; выбор `/model` в самой ветке имеет приоритет, а история расшифровки родительского канала не копируется, если не включено наследование расшифровки.
    - `channels.discord.thread.inheritParent` (по умолчанию `false`) включает заполнение новых автоматических веток данными из расшифровки родительского канала. Переопределение для отдельной учётной записи: `channels.discord.accounts.<id>.thread.inheritParent`.
    - Реакции, отправляемые через инструмент сообщений, могут разрешать цели личных сообщений вида `user:<id>`.
    - `guilds.<guild>.channels.<channel>.requireMention: false` сохраняется при резервной активации на этапе ответа.

    Темы каналов добавляются как **ненадёжный** контекст. Списки разрешений определяют, кто может активировать агента, но не являются полноценной границей очистки дополнительного контекста.

  </Accordion>

  <Accordion title="Привязанные к веткам сеансы субагентов">
    Discord может привязать ветку к целевому сеансу, чтобы последующие сообщения в этой ветке продолжали направляться в тот же сеанс, включая сеансы субагентов.

    Команды:

    - `/focus <target>` — привязать текущую или новую ветку к целевому субагенту или сеансу
    - `/unfocus` — удалить привязку текущей ветки
    - `/agents` — показать активные запуски и состояние привязки
    - `/session idle <duration|off>` — просмотреть или изменить автоматическое снятие привязки при бездействии для сфокусированных привязок
    - `/session max-age <duration|off>` — просмотреть или изменить предельный срок действия сфокусированных привязок

    Конфигурация:

```json5
{
  session: {
    threadBindings: {
      enabled: true,
      idleHours: 24,
      maxAgeHours: 0,
    },
  },
  channels: {
    discord: {
      threadBindings: {
        enabled: true,
        idleHours: 24,
        maxAgeHours: 0,
        spawnSessions: true,
        defaultSpawnContext: "fork",
      },
    },
  },
}
```

    Примечания:

    - `session.threadBindings.*` задаёт глобальные значения по умолчанию; `channels.discord.threadBindings.*` переопределяет поведение Discord.
    - `spawnSessions` управляет автоматическим созданием и привязкой веток для `sessions_spawn({ thread: true })` и запуска веток ACP. Значение по умолчанию: `true`.
    - `defaultSpawnContext` управляет контекстом встроенного субагента для запусков, привязанных к веткам. Значение по умолчанию: `"fork"`.
    - Устаревшие ключи `spawnSubagentSessions`/`spawnAcpSessions` переносятся командой `openclaw doctor --fix`.
    - Если привязки веток отключены для учётной записи, `/focus` и связанные операции привязки веток недоступны.

    См. разделы [Субагенты](/ru/tools/subagents), [Агенты ACP](/ru/tools/acp-agents) и [Справочник по конфигурации](/ru/gateway/configuration-reference).

  <Accordion title="Approvals in Discord">
    Discord поддерживает обработку подтверждений с помощью кнопок в личных сообщениях и при необходимости может публиковать запросы на подтверждение в исходном канале.

    Пути конфигурации:

    - `channels.discord.execApprovals.enabled`
    - `channels.discord.execApprovals.approvers` (необязательно; по возможности используется резервное значение `commands.ownerAllowFrom`)
    - `channels.discord.execApprovals.target` (`dm` | `channel` | `both`, по умолчанию: `dm`)
    - `agentFilter`, `sessionFilter`, `cleanupAfterResolve`

    Discord автоматически включает встроенные подтверждения выполнения, если `enabled` не задано или имеет значение `"auto"` и удаётся определить хотя бы одного утверждающего — из `execApprovals.approvers` либо из `commands.ownerAllowFrom`. Discord не определяет утверждающих для выполнения по значению канала `allowFrom`, устаревшему `dm.allowFrom` или `defaultTo` для личных сообщений. Чтобы явно отключить Discord как встроенный клиент подтверждений, задайте `enabled: false`.

    Для конфиденциальных групповых команд, доступных только владельцу, таких как `/diagnostics` и `/export-trajectory`, OpenClaw отправляет запросы на подтверждение и окончательные результаты приватно. Сначала предпринимается попытка отправки личного сообщения Discord, если у владельца, вызвавшего команду, есть маршрут владельца Discord; в противном случае используется первый доступный маршрут владельца из `commands.ownerAllowFrom`, например Telegram.

    Если `target` имеет значение `channel` или `both`, запрос на подтверждение виден в канале. Использовать кнопки могут только определённые утверждающие; остальные пользователи получают временное сообщение об отказе. Запросы на подтверждение содержат текст команды, поэтому включайте доставку в канал только для доверенных каналов. Если идентификатор канала нельзя получить из ключа сеанса, OpenClaw использует доставку в личные сообщения.

    Discord отображает общие кнопки подтверждения, используемые другими каналами чата; встроенный адаптер Discord в основном добавляет маршрутизацию личных сообщений утверждающим и рассылку по каналам. При наличии этих кнопок они служат основным интерфейсом подтверждения; OpenClaw должен включать команду ручного подтверждения `/approve` только тогда, когда результат инструмента указывает, что подтверждения в чате недоступны или ручное подтверждение является единственным вариантом. Если встроенная среда выполнения подтверждений Discord неактивна, OpenClaw оставляет видимым локальный детерминированный запрос `/approve <id> <decision>`. Если среда выполнения активна, но встроенную карточку невозможно доставить ни одной цели, OpenClaw отправляет в тот же чат резервное уведомление с точной командой `/approve` из ожидающего подтверждения.

    Аутентификация Gateway и разрешение подтверждений соответствуют общему контракту клиента Gateway (идентификаторы `plugin:` разрешаются через `plugin.approval.resolve`, остальные — через `exec.approval.resolve`). По умолчанию срок действия подтверждений истекает через 30 минут.

    См. [Подтверждения выполнения](/ru/tools/exec-approvals).

## Интерфейс компонентов v2

OpenClaw использует компоненты Discord v2 для подтверждений выполнения и маркеров межконтекстного взаимодействия. Действия с сообщениями Discord также могут принимать `components` для пользовательского интерфейса (расширенная возможность; требуется сформировать полезную нагрузку компонента с помощью инструмента Discord), а устаревшие `embeds` по-прежнему доступны, но не рекомендуются.

- `channels.discord.ui.components.accentColor` задаёт цвет акцента, используемый контейнерами компонентов Discord (шестнадцатеричное значение). Для отдельной учётной записи: `channels.discord.accounts.<id>.ui.components.accentColor`.
- `channels.discord.agentComponents.ttlMs` определяет, как долго обратные вызовы отправленных компонентов Discord остаются зарегистрированными (по умолчанию `1800000`, максимум `86400000`). Для отдельной учётной записи: `channels.discord.accounts.<id>.agentComponents.ttlMs`.
- `embeds` игнорируются при наличии компонентов v2.
- Предварительный просмотр обычных URL по умолчанию подавляется. Укажите `suppressEmbeds: false` в действии с сообщением, если требуется разворачивать одну исходящую ссылку.

Пример:

```json5
{
  channels: {
    discord: {
      ui: {
        components: {
          accentColor: "#5865F2",
        },
      },
    },
  },
}
```

## Голосовая связь

В Discord есть две отдельные возможности голосовой связи: **голосовые каналы** в реальном времени (непрерывные разговоры) и **вложения с голосовыми сообщениями** (формат предварительного просмотра с осциллограммой). Gateway поддерживает обе возможности.

### Голосовые каналы

Контрольный список настройки:

1. Включите Message Content Intent в Discord Developer Portal.
2. Включите Server Members Intent, если используются списки разрешённых ролей или пользователей.
3. Пригласите бота с областями доступа `bot` и `applications.commands`.
4. Предоставьте разрешения Connect, Speak, Send Messages и Read Message History в целевом голосовом канале.
5. Включите встроенные команды (`commands.native` или `channels.discord.commands.native`).
6. Настройте `channels.discord.voice`.

Используйте `/vc join|leave|status` для управления сеансами. Команда использует агента учётной записи по умолчанию и подчиняется тем же правилам списков разрешений и групповой политики, что и другие команды Discord.

```bash
/vc join channel:<voice-channel-id>
/vc status
/vc leave
```

Чтобы перед подключением проверить фактические разрешения бота:

```bash
openclaw channels capabilities --channel discord --target channel:<voice-channel-id>
```

Пример автоматического подключения:

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        model: "openai/gpt-5.6-sol",
        autoJoin: [
          {
            guildId: "123456789012345678",
            channelId: "234567890123456789",
          },
        ],
        allowedChannels: [
          {
            guildId: "123456789012345678",
            channelId: "234567890123456789",
          },
        ],
        daveEncryption: true,
        decryptionFailureTolerance: 24,
        connectTimeoutMs: 30000,
        reconnectGraceMs: 15000,
        realtime: {
          provider: "openai",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
        },
      },
    },
  },
}
```

Примечания:

- Голосовые функции Discord являются опциональными для конфигураций, предназначенных только для текста; задайте `channels.discord.voice.enabled=true` (либо сохраните существующий блок `channels.discord.voice`), чтобы включить команды `/vc`, голосовую среду выполнения и намерение Gateway `GuildVoiceStates`. Параметр `channels.discord.intents.voiceStates` позволяет явно переопределить подписку на намерение; не задавайте его, чтобы она соответствовала фактическому состоянию включения голосовых функций.
- `voice.mode` определяет схему ведения разговора. По умолчанию используется `agent-proxy`: голосовой интерфейс реального времени управляет очередностью реплик, прерываниями и воспроизведением, делегирует содержательную работу маршрутизированному агенту OpenClaw через `openclaw_agent_consult` и обрабатывает результат как текстовый запрос Discord от этого говорящего. Режим `stt-tts` сохраняет прежнюю пакетную схему STT с последующим TTS. Режим `bidi` позволяет модели реального времени вести разговор напрямую, предоставляя `openclaw_agent_consult` для обращения к интеллектуальному ядру OpenClaw.
- `voice.agentSession` определяет, в какой диалог OpenClaw поступают голосовые реплики. Не задавайте этот параметр, чтобы использовать собственный сеанс голосового канала, либо укажите `{ mode: "target", target: "channel:<text-channel-id>" }`, чтобы голосовой канал служил расширением с микрофоном и динамиком для существующего сеанса текстового канала Discord, например `#maintainers`.
- `voice.model` переопределяет интеллектуальную модель агента OpenClaw для голосовых ответов Discord и консультаций в реальном времени. Не задавайте этот параметр, чтобы наследовать модель маршрутизированного агента. Он не связан с `voice.realtime.model`.
- `voice.followUsers` позволяет боту входить в голосовые каналы Discord вместе с выбранными пользователями, переходить вслед за ними и выходить. См. раздел [Следование за пользователями в голосовых каналах](#follow-users-in-voice).
- В режиме `agent-proxy` речь маршрутизируется через `discord-voice`, который сохраняет обычную авторизацию владельца и инструментов для говорящего и целевого сеанса, но скрывает инструмент агента `tts`, поскольку воспроизведением управляет голосовая подсистема Discord. По умолчанию `agent-proxy` предоставляет консультации полный, эквивалентный владельцу доступ к инструментам для говорящих-владельцев (`voice.realtime.toolPolicy: "owner"`) и настоятельно предпочитает обращаться к агенту OpenClaw перед содержательными ответами (`voice.realtime.consultPolicy: "always"`). В этом стандартном режиме `always` слой реального времени не произносит автоматически заполняющую паузу фразу перед ответом консультации; он записывает и транскрибирует речь, а затем озвучивает ответ маршрутизированного агента OpenClaw. Если несколько обязательных ответов консультации завершаются, пока Discord ещё воспроизводит первый ответ, последующие ответы с точной формулировкой речи помещаются в очередь до окончания воспроизведения, а не заменяют речь посреди предложения.
- В режиме `stt-tts` для STT используется `tools.media.audio`; параметр `voice.model` не влияет на транскрипцию.
- В режимах реального времени параметры `voice.realtime.provider`, `voice.realtime.model` и `voice.realtime.speakerVoice` настраивают аудиосеанс реального времени. Для OpenAI Realtime 2.1 с интеллектуальным ядром Codex используйте `voice.realtime.model: "gpt-realtime-2.1"` и `voice.model: "openai/gpt-5.6-sol"`.
- По умолчанию голосовые режимы реального времени включают небольшие файлы профилей `IDENTITY.md`, `USER.md` и `SOUL.md` в инструкции поставщику реального времени, чтобы быстрые прямые реплики сохраняли ту же идентичность, привязку к пользователю и образ, что и маршрутизированный агент OpenClaw. Задайте в `voice.realtime.bootstrapContextFiles` подмножество файлов, чтобы настроить это поведение, или `[]`, чтобы отключить его. Поддерживаются только эти файлы профилей; `AGENTS.md` остаётся в обычном контексте агента. Внедрённый контекст профиля не заменяет `openclaw_agent_consult` при работе с рабочей областью, актуальными сведениями, поиске в памяти или действиях с использованием инструментов.
- В режиме реального времени OpenAI `agent-proxy` задайте `voice.realtime.requireWakeName: true`, чтобы голосовая подсистема реального времени Discord не отвечала, пока транскрипция не начинается или не заканчивается именем активации. Настроенные имена активации должны состоять из одного или двух слов. Если `voice.realtime.wakeNames` не задан, OpenClaw использует `name` маршрутизированного агента вместе с `OpenClaw`, а при его отсутствии — идентификатор агента вместе с `OpenClaw`. Фильтрация по имени активации отключает автоматические ответы поставщика реального времени, направляет принятые реплики через путь консультации с агентом OpenClaw и выдаёт короткое голосовое подтверждение, когда ведущее имя активации распознано по частичной транскрипции до получения окончательной.
- Поставщик OpenAI для работы в реальном времени принимает актуальные имена событий Realtime 2 и устаревшие псевдонимы, совместимые с Codex, для событий выходного звука и транскрипции, поэтому версии совместимых поставщиков могут различаться без потери звука ответов ассистента.
- `voice.realtime.bargeIn` определяет, прерывают ли события начала речи в Discord активное воспроизведение в реальном времени. Если параметр не задан, используется настройка прерывания по входному аудио поставщика реального времени.
- `voice.realtime.minBargeInAudioEndMs` задаёт минимальную продолжительность воспроизведения ответа ассистента, после которой вмешательство в разговор в реальном времени OpenAI обрезает звук. По умолчанию: `250`. Задайте `0` для немедленного прерывания в помещениях со слабым эхом или увеличьте значение для акустических систем в помещениях с сильным эхом.
- `voice.tts` переопределяет `messages.tts` только для голосового воспроизведения в режиме `stt-tts`; режимы реального времени вместо этого используют `voice.realtime.speakerVoice`. Чтобы использовать голос OpenAI при воспроизведении в Discord, задайте `voice.tts.provider: "openai"` и выберите голос синтеза речи в `voice.tts.providers.openai.speakerVoice`. `cedar` — хороший вариант с мужским звучанием для текущей модели TTS OpenAI.
- Переопределения `systemPrompt` для отдельных каналов Discord применяются к голосовым репликам, транскрибированным в соответствующем голосовом канале.
- Для транскрибированных голосовых реплик статус владельца определяется на основе Discord `allowFrom` (или `dm.allowFrom`) для команд и действий канала, доступных только владельцу. Доступность инструментов агента определяется настроенной политикой инструментов маршрутизированного сеанса.
- Если `voice.autoJoin` содержит несколько записей для одной гильдии, OpenClaw подключается к последнему настроенному каналу этой гильдии.
- `voice.allowedChannels` — необязательный список разрешённых каналов пребывания. Не задавайте его, чтобы разрешить `/vc join` для любого авторизованного голосового канала Discord. Если список задан, `/vc join`, автоматическое подключение при запуске и перемещения голосового состояния бота ограничиваются перечисленными записями `{ guildId, channelId }`. Задайте пустой массив, чтобы запретить любые подключения к голосовым каналам Discord. Если Discord перемещает бота за пределы списка разрешённых каналов, OpenClaw покидает этот канал и повторно подключается к настроенному целевому каналу автоматического подключения, если он доступен.
- `voice.daveEncryption` и `voice.decryptionFailureTolerance` передаются в параметры подключения `@discordjs/voice`; значения вышестоящего проекта по умолчанию — `daveEncryption=true` и `decryptionFailureTolerance=24`.
- OpenClaw использует встроенный кодек `libopus-wasm` для приёма голоса Discord и воспроизведения необработанного PCM в реальном времени. В поставку входит закреплённая версия libopus для WebAssembly, поэтому нативные дополнения opus не требуются.
- `voice.connectTimeoutMs` задаёт время первоначального ожидания состояния `@discordjs/voice` Ready для `/vc join` и попыток автоматического подключения. По умолчанию: `30000`.
- `voice.reconnectGraceMs` определяет, как долго OpenClaw ожидает начала повторного подключения отключённого голосового сеанса, прежде чем уничтожить его. По умолчанию: `15000`.
- В режиме `stt-tts` голосовое воспроизведение не останавливается только потому, что другой пользователь начал говорить. Чтобы избежать циклов обратной связи, OpenClaw игнорирует новый захват голоса во время воспроизведения TTS; для следующей реплики говорите после завершения воспроизведения. Режимы реального времени передают начало речи как сигнал вмешательства в разговор поставщику реального времени.
- В режимах реального времени эхо от динамиков, попадающее в открытый микрофон, может восприниматься как вмешательство в разговор и прерывать воспроизведение. Для помещений Discord с сильным эхом задайте `voice.realtime.providers.openai.interruptResponseOnInputAudio: false`, чтобы OpenAI не прерывал ответ автоматически при поступлении входного аудио. Добавьте `voice.realtime.bargeIn: true`, если события начала речи в Discord всё равно должны прерывать активное воспроизведение. Мост реального времени OpenAI считает обрезания воспроизведения короче `voice.realtime.minBargeInAudioEndMs` вероятным эхом или шумом, игнорирует их и записывает в журнал как пропущенные, вместо того чтобы очищать воспроизведение Discord.
- `voice.captureSilenceGraceMs` определяет, как долго OpenClaw ожидает после сообщения Discord о прекращении речи, прежде чем завершить этот аудиосегмент для STT. По умолчанию: `2000`; увеличьте значение, если Discord разбивает обычные паузы на прерывистые частичные транскрипции.
- Если в качестве поставщика TTS выбран ElevenLabs, голосовое воспроизведение Discord использует потоковый TTS и начинается непосредственно из потока ответа поставщика. Для поставщиков без поддержки потоковой передачи используется резервный путь через синтезированный временный файл.
- OpenClaw отслеживает ошибки расшифровки принимаемого звука и автоматически восстанавливает работу, покидая голосовой канал и повторно подключаясь к нему после нескольких ошибок за короткий период.
- Если после обновления в журналах приёма неоднократно появляется `DecryptionFailed(UnencryptedWhenPassthroughDisabled)`, соберите отчёт о зависимостях и журналы. Встроенная версия `@discordjs/voice` содержит вышестоящее исправление заполнения из PR discord.js № 11449, закрывшее проблему discord.js № 11419.
- События приёма `The operation was aborted` ожидаемы, когда OpenClaw завершает захваченный сегмент речи; это подробные диагностические сообщения, а не предупреждения.
- Подробные журналы голосовой подсистемы Discord содержат ограниченный однострочный предварительный просмотр транскрипции STT для каждого принятого сегмента речи, поэтому при отладке видны как реплика пользователя, так и ответ агента без вывода неограниченного объёма текста транскрипции.
- В режиме `agent-proxy` резервная обязательная консультация пропускает вероятно незавершённые фрагменты транскрипции, например текст, заканчивающийся на `...` или союз вроде «и», а также очевидные завершения разговора, не требующие действий, например «сейчас вернусь» или «пока». Когда это предотвращает выдачу устаревшего ответа из очереди, в журнале появляется `forced agent consult skipped reason=...`.

### Следование за пользователями в голосовых каналах

Используйте `voice.followUsers`, если голосовой бот Discord должен оставаться с одним или несколькими известными пользователями Discord, а не подключаться к фиксированному каналу при запуске или ожидать `/vc join`.

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        followUsersEnabled: true,
        followUsers: ["discord:123456789012345678"],
        allowedChannels: [
          {
            guildId: "123456789012345678",
            channelId: "234567890123456789",
          },
        ],
      },
    },
  },
}
```

Поведение:

- `followUsers` принимает необработанные идентификаторы пользователей Discord и значения `discord:<id>`. OpenClaw нормализует обе формы перед сопоставлением событий голосового состояния.
- Когда настроен `followUsers`, параметр `followUsersEnabled` по умолчанию равен `true`. Задайте `false`, чтобы сохранить список, но прекратить автоматическое следование в голосовых каналах.
- Когда отслеживаемый пользователь подключается к разрешённому голосовому каналу, OpenClaw входит в этот канал. Когда пользователь перемещается, OpenClaw следует за ним. Когда активный отслеживаемый пользователь отключается, OpenClaw выходит.
- Если в одной гильдии находятся несколько отслеживаемых пользователей и активный отслеживаемый пользователь выходит, OpenClaw переходит в канал другого отслеживаемого пользователя, прежде чем покинуть гильдию. Если несколько отслеживаемых пользователей перемещаются одновременно, приоритет имеет последнее наблюдавшееся событие голосового состояния.
- Ограничение `allowedChannels` продолжает действовать. Отслеживаемый пользователь в запрещённом канале игнорируется, а сеанс, принадлежащий функции следования, переходит к другому отслеживаемому пользователю или завершается.
- OpenClaw согласовывает пропущенные события голосового состояния при запуске и через ограниченные интервалы времени. При согласовании опрашиваются настроенные гильдии, а число запросов REST за один проход ограничено, поэтому для очень больших списков `followUsers` может потребоваться несколько интервалов до полной синхронизации.
- Если Discord или администратор перемещает бота во время следования за пользователем, OpenClaw перестраивает голосовой сеанс и сохраняет принадлежность функции следования, если целевой канал разрешён. Если бот перемещён за пределы `allowedChannels`, OpenClaw выходит и повторно подключается к настроенному целевому каналу, если он существует.
- При восстановлении приёма DAVE бот может покинуть канал и повторно войти в него после нескольких ошибок расшифровки. Сеансы, принадлежащие функции следования, сохраняют эту принадлежность во время восстановления, поэтому при последующем отключении отслеживаемого пользователя бот всё равно покинет канал.

Выбор режима подключения:

- Используйте `followUsers` для личных или операторских конфигураций, в которых бот должен автоматически присутствовать в голосовом канале вместе с вами.
- Используйте `autoJoin` для ботов в фиксированных комнатах, которые должны присутствовать там, даже когда ни один отслеживаемый пользователь не находится в голосовом канале.
- Используйте `/vc join` для разовых подключений или комнат, где автоматическое присутствие в голосовом канале было бы неожиданным.

Голосовой кодек Discord:

- В журналах приёма голоса отображается `discord voice: opus decoder: libopus-wasm`.
- При воспроизведении в реальном времени необработанный стереофонический PCM с частотой 48 кГц кодируется в Opus с помощью того же встроенного пакета `libopus-wasm`, после чего пакеты передаются в `@discordjs/voice`.
- При воспроизведении файлов и потоков поставщика данные перекодируются с помощью ffmpeg в необработанный стереофонический PCM с частотой 48 кГц, после чего `libopus-wasm` формирует поток пакетов Opus для отправки в Discord.

Конвейер STT с последующим TTS:

- Захват PCM-аудио в Discord преобразуется во временный WAV-файл.
- `tools.media.audio` выполняет распознавание речи, например с помощью `openai/gpt-4o-mini-transcribe`.
- Транскрипция передаётся через входящий поток и маршрутизацию Discord, а LLM ответа работает с политикой голосового вывода, которая скрывает от агента инструмент `tts` и запрашивает возврат текста, поскольку окончательное воспроизведение TTS выполняет голосовой канал Discord.
- Если параметр `voice.model` задан, он переопределяет только LLM ответа для этого обращения в голосовом канале.
- `voice.tts` объединяется поверх `messages.tts`; провайдеры с поддержкой потоковой передачи направляют данные непосредственно проигрывателю, иначе полученный аудиофайл воспроизводится в подключённом канале.

Пример сеанса голосового канала с проксированием агента по умолчанию:

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        model: "openai/gpt-5.6-sol",
        followUsersEnabled: true,
        followUsers: ["123456789012345678"],
        realtime: {
          provider: "openai",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
        },
      },
    },
  },
}
```

Если блок `voice.agentSession` отсутствует, каждый голосовой канал получает отдельный маршрутизируемый сеанс OpenClaw. Например, команда `/vc join channel:234567890123456789` обращается к сеансу соответствующего голосового канала Discord. Модель реального времени служит только голосовым интерфейсом; содержательные запросы передаются настроенному агенту OpenClaw. Если модель реального времени создаёт окончательную транскрипцию, не вызывая инструмент консультации, OpenClaw принудительно запускает консультацию как запасной вариант, чтобы поведение по умолчанию по-прежнему соответствовало разговору с агентом.

Пример устаревшего режима распознавания и синтеза речи:

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        mode: "stt-tts",
        model: "openai/gpt-5.4-mini",
        tts: {
          provider: "openai",
          providers: {
            openai: {
              model: "gpt-4o-mini-tts",
              speakerVoice: "cedar",
            },
          },
        },
      },
    },
  },
}
```

Пример двунаправленного режима реального времени:

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        mode: "bidi",
        model: "openai/gpt-5.6-sol",
        realtime: {
          provider: "openai",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
          toolPolicy: "safe-read-only",
          consultPolicy: "always",
        },
      },
    },
  },
}
```

Голосовая связь как расширение существующего сеанса канала Discord:

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        mode: "agent-proxy",
        model: "openai/gpt-5.6-sol",
        agentSession: {
          mode: "target",
          target: "channel:123456789012345678",
        },
        realtime: {
          provider: "openai",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
        },
      },
    },
  },
}
```

В режиме `agent-proxy` бот подключается к настроенному голосовому каналу, но обращения к агенту OpenClaw используют обычный маршрутизируемый сеанс и агента целевого канала. Голосовой сеанс реального времени озвучивает возвращённый результат в голосовом канале. Агент-супервизор по-прежнему может использовать обычные инструменты для сообщений в соответствии со своей политикой инструментов, в том числе отправлять отдельное сообщение в Discord, если это уместное действие.

Пока выполняется делегированный запуск OpenClaw, новые голосовые транскрипции Discord рассматриваются как команды управления активным запуском, а не как начало нового обращения к агенту. Такие фразы, как «статус», «отмени это», «используй более компактное исправление» или «когда закончишь, также проверь тесты», классифицируются как запрос статуса, отмена, корректирующая инструкция или последующий запрос для активного сеанса. Результаты запроса статуса, отмены, принятой корректирующей инструкции и последующего запроса озвучиваются в голосовом канале, чтобы вызывающий пользователь знал, обработал ли OpenClaw запрос.

Полезные форматы цели:

- `target: "channel:123456789012345678"` выполняет маршрутизацию через сеанс текстового канала Discord.
- `target: "123456789012345678"` интерпретируется как цель-канал.
- `target: "dm:123456789012345678"` или `target: "user:123456789012345678"` выполняет маршрутизацию через соответствующий сеанс личных сообщений.

Пример OpenAI Realtime для среды с сильным эхом:

```json5
{
  channels: {
    discord: {
      voice: {
        enabled: true,
        mode: "bidi",
        model: "openai/gpt-5.6-sol",
        realtime: {
          provider: "openai",
          model: "gpt-realtime-2.1",
          speakerVoice: "cedar",
          bargeIn: true,
          minBargeInAudioEndMs: 500,
          consultPolicy: "always",
          providers: {
            openai: {
              interruptResponseOnInputAudio: false,
            },
          },
        },
      },
    },
  },
}
```

Используйте эту конфигурацию, когда модель слышит собственное воспроизведение Discord через открытый микрофон, но вы всё равно хотите иметь возможность прерывать её голосом. OpenClaw не позволяет OpenAI автоматически прерывать ответ из-за необработанного входного аудио, а параметр `bargeIn: true` позволяет событиям начала речи в Discord и аудио уже активного говорящего отменять активные ответы реального времени до того, как следующий захваченный фрагмент речи поступит в OpenAI. Слишком ранние сигналы прерывания со значением `audioEndMs` ниже `minBargeInAudioEndMs` считаются вероятным эхом или шумом и игнорируются, чтобы модель не обрывала ответ на первом кадре воспроизведения.

Ожидаемые журналы голосовой связи:

- При подключении: `discord voice: joining ... voiceSession=... supervisorSession=... agentSessionMode=... voiceModel=... realtimeModel=...`
- При запуске режима реального времени: `discord voice: realtime bridge starting ... autoRespond=false interruptResponse=false bargeIn=false minBargeInAudioEndMs=...`
- При поступлении аудио говорящего: `discord voice: realtime speaker turn opened ...`, `discord voice: realtime input audio started ... outputAudioMs=... outputActive=...` и `discord voice: realtime speaker turn closed ... chunks=... discordBytes=... realtimeBytes=... interruptedPlayback=...`
- При пропуске устаревшей речи: `discord voice: realtime forced agent consult skipped reason=incomplete-transcript ...` или `reason=non-actionable-closing ...`
- При завершении ответа реального времени: `discord voice: realtime audio playback finishing reason=response.done ... audioMs=... chunks=...`
- При остановке или сбросе воспроизведения: `discord voice: realtime audio playback stopped reason=... audioMs=... elapsedMs=... chunks=...`
- При консультации в режиме реального времени: `discord voice: realtime consult requested ... voiceSession=... supervisorSession=... question=...`
- При ответе агента: `discord voice: agent turn answer ...`
- При постановке точной фразы в очередь: `discord voice: realtime exact speech queued ... queued=... outputAudioMs=... outputActive=...`, а затем `discord voice: realtime exact speech dequeued reason=player-idle ...`
- При обнаружении голосового прерывания: `discord voice: realtime barge-in detected source=speaker-start ...` или `discord voice: realtime barge-in detected source=active-speaker-audio ...`, а затем `discord voice: realtime barge-in requested reason=... outputAudioMs=... outputActive=...`
- При прерывании ответа реального времени: `discord voice: realtime model interrupt requested client:response.cancel reason=barge-in`, а затем либо `discord voice: realtime model audio truncated client:conversation.item.truncate reason=barge-in audioEndMs=...`, либо `discord voice: realtime model interrupt confirmed server:response.done status=cancelled ...`
- При игнорировании эха или шума: `discord voice: realtime model interrupt ignored client:conversation.item.truncate.skipped reason=barge-in audioEndMs=0 minAudioEndMs=250`
- При отключённом голосовом прерывании: `discord voice: realtime capture ignored during playback (barge-in disabled) ...`
- При отсутствии активного воспроизведения: `discord voice: realtime barge-in ignored reason=... outputActive=false ... playbackChunks=0`

Чтобы диагностировать обрыв аудио, рассматривайте журналы голосовой связи реального времени как временную шкалу:

1. `realtime audio playback started` означает, что Discord начал воспроизводить аудио ассистента. С этого момента мост начинает подсчитывать фрагменты выходного аудио ассистента, байты PCM Discord, байты данных реального времени провайдера и длительность синтезированного аудио.
2. `realtime speaker turn opened` отмечает момент, когда говорящий в Discord становится активным. Если воспроизведение уже активно и параметр `bargeIn` включён, далее может появиться `barge-in detected source=speaker-start`.
3. `realtime input audio started` отмечает первый фактический аудиокадр, полученный в рамках этого фрагмента речи. Значение `outputActive=true` или ненулевое значение `outputAudioMs` в этой строке означает, что микрофон отправляет входное аудио, пока воспроизведение ответа ассистента ещё активно.
4. `barge-in detected source=active-speaker-audio` означает, что OpenClaw обнаружил аудио активного говорящего во время воспроизведения ответа ассистента. Это помогает отличить реальное прерывание от события начала речи Discord, за которым не последовало полезного аудио.
5. `barge-in requested reason=...` означает, что OpenClaw запросил у провайдера реального времени отмену или усечение активного ответа. Строка содержит `outputAudioMs`, `outputActive` и `playbackChunks`, позволяя определить, сколько аудио ассистента фактически было воспроизведено до прерывания.
6. `realtime audio playback stopped reason=...` обозначает точку локального сброса воспроизведения Discord. Причина указывает, что остановило воспроизведение: `barge-in`, `player-idle`, `provider-clear-audio`, `forced-agent-consult`, `stream-close` или `session-close`.
7. `realtime speaker turn closed` содержит сводку захваченного фрагмента входной речи. Значение `chunks=0` или `hasAudio=false` означает, что фрагмент речи был открыт, но полезное аудио не поступило в мост реального времени. Значение `interruptedPlayback=true` означает, что этот фрагмент входной речи пересёкся с выводом ассистента и запустил логику голосового прерывания.

Полезные поля:

- `outputAudioMs`: длительность аудио ассистента, созданного провайдером реального времени до указанной строки журнала.
- `audioMs`: длительность аудио ассистента, подсчитанная OpenClaw до остановки воспроизведения.
- `elapsedMs`: фактическое время между открытием и закрытием потока воспроизведения или фрагмента речи.
- `discordBytes`: байты стереофонического PCM-аудио с частотой 48 кГц, отправленные в голосовой канал Discord или полученные из него.
- `realtimeBytes`: байты PCM-аудио в формате провайдера, отправленные провайдеру реального времени или полученные от него.
- `playbackChunks`: фрагменты аудио ассистента, переданные в Discord для активного ответа.
- `sinceLastAudioMs`: интервал между последним захваченным аудиокадром говорящего и закрытием фрагмента речи.

Распространённые ситуации:

- Немедленный обрыв при `source=active-speaker-audio`, малом значении `outputAudioMs` и нахождении того же пользователя рядом обычно указывает на попадание эха динамиков в микрофон. Увеличьте `voice.realtime.minBargeInAudioEndMs`, уменьшите громкость динамиков, используйте наушники или задайте `voice.realtime.providers.openai.interruptResponseOnInputAudio: false`.
- `source=speaker-start`, после которого следует `speaker turn closed ... hasAudio=false`, означает, что Discord сообщил о начале речи, но аудио не поступило в OpenClaw. Причиной может быть кратковременное событие голосовой связи Discord, работа шумового порога или короткое включение микрофона клиентом.
- `audio playback stopped reason=stream-close` без близкого по времени голосового прерывания или события `provider-clear-audio` означает, что локальный поток воспроизведения Discord неожиданно завершился. Проверьте предшествующие журналы провайдера и проигрывателя Discord.
- `capture ignored during playback (barge-in disabled)` означает, что OpenClaw намеренно отбросил входное аудио во время активного воспроизведения ответа ассистента. Включите `voice.realtime.bargeIn`, если хотите, чтобы речь прерывала воспроизведение.
- `barge-in ignored ... outputActive=false` означает, что Discord или система обнаружения голосовой активности провайдера обнаружили речь, но в OpenClaw не было активного воспроизведения, которое можно было бы прервать. Это не должно приводить к обрыву аудио.

Учётные данные разрешаются отдельно для каждого компонента: аутентификация маршрута LLM для `voice.model`, аутентификация распознавания речи для `tools.media.audio`, аутентификация синтеза речи для `messages.tts`/`voice.tts` и аутентификация провайдера реального времени для `voice.realtime.providers` или обычной конфигурации аутентификации провайдера.

### Голосовые сообщения

Голосовые сообщения Discord отображают предварительный просмотр звуковой волны и требуют аудио в формате OGG/Opus. OpenClaw создаёт звуковую волну автоматически, но для анализа и преобразования на хосте Gateway должны быть установлены `ffmpeg` и `ffprobe`.

- Укажите **путь к локальному файлу** (URL-адреса отклоняются).
- Не указывайте текстовое содержимое (Discord отклоняет сочетание текста и голосового сообщения в одной полезной нагрузке).
- Поддерживается любой аудиоформат; при необходимости OpenClaw преобразует его в OGG/Opus.

```bash
message(action="send", channel="discord", target="channel:123", path="/path/to/audio.mp3", asVoice=true)
```

## Устранение неполадок

  <Accordion title="Используются запрещённые намерения или бот не видит сообщения сервера">

    - включите Message Content Intent

    - проверьте список разрешенных гильдий в `channels.discord.guilds`
    - если существует карта `channels` гильдии, разрешены только перечисленные каналы
    - проверьте поведение `requireMention` и шаблоны упоминаний

    Полезные проверки:

```bash
openclaw doctor
openclaw channels status --probe
openclaw logs --follow
```

  </Accordion>

  <Accordion title="Упоминание не требуется, но сообщения по-прежнему блокируются">
    Распространенные причины:

    - `groupPolicy="allowlist"` без соответствующего списка разрешенных гильдий или каналов
    - `requireMention` настроен не в том месте (он должен находиться в `channels.discord.guilds` или в записи канала)
    - отправитель заблокирован списком разрешенных `users` гильдии или канала

  </Accordion>

  <Accordion title="Длительная обработка запросов Discord или повторяющиеся ответы">

    Типичные записи журнала:

    - `Slow listener detected ...`
    - `stuck session: sessionKey=agent:...:discord:... state=processing ...`

    Параметры очереди Gateway Discord:

    - одна учетная запись: `channels.discord.eventQueue.listenerTimeout`
    - несколько учетных записей: `channels.discord.accounts.<accountId>.eventQueue.listenerTimeout`
    - этот параметр управляет только работой обработчика событий Gateway Discord, а не временем выполнения хода агента

    Discord не применяет тайм-аут, принадлежащий каналу, к поставленным в очередь ходам агента. Обработчики сообщений немедленно передают работу дальше, а поставленные в очередь запуски Discord сохраняют порядок в рамках сеанса, пока жизненный цикл сеанса, инструмента или среды выполнения не завершит или не прервет работу.

```json5
{
  channels: {
    discord: {
      accounts: {
        default: {
          eventQueue: {
            listenerTimeout: 120000,
          },
        },
      },
    },
  },
}
```

  </Accordion>

  <Accordion title="Предупреждения о тайм-ауте получения метаданных Gateway">
    Перед подключением OpenClaw получает метаданные Discord `/gateway/bot`. При временных сбоях используется стандартный URL Gateway Discord, а частота записей в журнале ограничивается.

    Параметры тайм-аута метаданных:

    - одна учетная запись: `channels.discord.gatewayInfoTimeoutMs`
    - несколько учетных записей: `channels.discord.accounts.<accountId>.gatewayInfoTimeoutMs`
    - резервная переменная окружения, если параметр конфигурации не задан: `OPENCLAW_DISCORD_GATEWAY_INFO_TIMEOUT_MS`
    - значение по умолчанию: `30000` (30 секунд), максимум: `120000`

  </Accordion>

  <Accordion title="Перезапуски из-за тайм-аута READY Gateway">
    При запуске и после повторных подключений среды выполнения OpenClaw ожидает событие `READY` от Gateway Discord. В конфигурациях с несколькими учетными записями и поэтапным запуском может потребоваться более длительный период ожидания READY при запуске, чем задано по умолчанию.

    Параметры тайм-аута READY:

    - запуск с одной учетной записью: `channels.discord.gatewayReadyTimeoutMs`
    - запуск с несколькими учетными записями: `channels.discord.accounts.<accountId>.gatewayReadyTimeoutMs`
    - резервная переменная окружения при запуске, если параметр конфигурации не задан: `OPENCLAW_DISCORD_READY_TIMEOUT_MS`
    - значение при запуске по умолчанию: `15000` (15 секунд), максимум: `120000`
    - среда выполнения с одной учетной записью: `channels.discord.gatewayRuntimeReadyTimeoutMs`
    - среда выполнения с несколькими учетными записями: `channels.discord.accounts.<accountId>.gatewayRuntimeReadyTimeoutMs`
    - резервная переменная окружения среды выполнения, если параметр конфигурации не задан: `OPENCLAW_DISCORD_RUNTIME_READY_TIMEOUT_MS`
    - значение среды выполнения по умолчанию: `30000` (30 секунд), максимум: `120000`

  </Accordion>

  <Accordion title="Несоответствия при аудите разрешений">
    Проверка разрешений с помощью `channels status --probe` работает только для числовых идентификаторов каналов.

    Если вы используете ключи-слаги, сопоставление во время выполнения по-прежнему может работать, однако проверка не сможет полностью подтвердить разрешения.

  </Accordion>

  <Accordion title="Проблемы с личными сообщениями и сопряжением">

    - личные сообщения отключены: `channels.discord.dm.enabled=false`
    - политика личных сообщений отключена: `channels.discord.dmPolicy="disabled"` (устаревший параметр: `channels.discord.dm.policy`)
    - в режиме `pairing` ожидается подтверждение сопряжения

  </Accordion>

  <Accordion title="Зацикливание взаимодействия между ботами">
    По умолчанию сообщения, отправленные ботами, игнорируются.

    Если вы задаете `channels.discord.allowBots=true`, используйте строгие правила упоминаний и списков разрешенных, чтобы избежать зацикливания.
    Предпочтительно использовать `channels.discord.allowBots="mentions"`, чтобы принимать только сообщения ботов, в которых упомянут бот.

    OpenClaw также поставляется с общей [защитой от зацикливания ботов](/ru/channels/bot-loop-protection). Когда `allowBots` позволяет сообщениям ботов поступать на диспетчеризацию, Discord преобразует входящее событие в данные `(учетная запись, канал, пара ботов)`, а общий механизм защиты пары подавляет ее после превышения настроенного бюджета событий. Этот механизм предотвращает неконтролируемые циклы между двумя ботами, которые ранее приходилось останавливать с помощью ограничений частоты запросов Discord; он не влияет на развертывания с одним ботом или однократные ответы ботов, не превышающие бюджет.

    Настройки по умолчанию (активны, когда задан `allowBots`):

    - `maxEventsPerWindow: 20` -- пара ботов может обменяться 20 сообщениями в пределах скользящего окна
    - `windowSeconds: 60` -- длительность скользящего окна
    - `cooldownSeconds: 60` -- после исчерпания бюджета каждое последующее сообщение между ботами в любом направлении отбрасывается в течение одной минуты

    Один раз настройте общее значение по умолчанию в `channels.defaults.botLoopProtection`, а затем переопределите его для Discord, если допустимому рабочему процессу требуется больший запас. Порядок приоритета:

    - `channels.discord.accounts.<account>.botLoopProtection`
    - `channels.discord.botLoopProtection`
    - `channels.defaults.botLoopProtection`
    - встроенные значения по умолчанию

    Discord использует общие ключи `maxEventsPerWindow`, `windowSeconds` и `cooldownSeconds`.

```json5
{
  channels: {
    defaults: {
      botLoopProtection: {
        maxEventsPerWindow: 20,
        windowSeconds: 60,
        cooldownSeconds: 60,
      },
    },
    discord: {
      // Optional Discord-wide override. Account blocks override individual
      // fields and inherit omitted fields from here.
      botLoopProtection: {
        maxEventsPerWindow: 4,
      },
      accounts: {
        alpha: {
          // Alpha listens to other bots only when they mention it.
          allowBots: "mentions",
        },
        bravo: {
          // Bravo listens to all bot-authored Discord messages.
          allowBots: true,
          mentionAliases: {
            // Lets Bravo write an Alpha Discord mention with the configured user id.
            Alpha: "ALPHA_DISCORD_USER_ID",
          },
          botLoopProtection: {
            // Allow up to five messages per minute before suppressing the pair.
            maxEventsPerWindow: 5,
            windowSeconds: 60,
            cooldownSeconds: 90,
          },
        },
      },
    },
  },
}
```

  </Accordion>

  <Accordion title="Пропуски голосового STT с ошибкой DecryptionFailed(...)">

    - поддерживайте OpenClaw в актуальном состоянии (`openclaw update`), чтобы была доступна логика восстановления приема голосовых данных Discord
    - убедитесь, что задано `channels.discord.voice.daveEncryption=true` (значение по умолчанию)
    - начните со значения `channels.discord.voice.decryptionFailureTolerance=24` (значение по умолчанию в вышестоящем проекте) и изменяйте его только при необходимости
    - отслеживайте в журналах:
      - `discord voice: DAVE decrypt failures detected`
      - `discord voice: repeated decrypt failures; attempting rejoin`
    - если сбои продолжаются после автоматического повторного подключения, соберите журналы и сравните их с историей приема DAVE в вышестоящем проекте: [discord.js #11419](https://github.com/discordjs/discord.js/issues/11419) и [discord.js #11449](https://github.com/discordjs/discord.js/pull/11449)

  </Accordion>
</AccordionGroup>

- ответы и история: `replyToMode`, `historyLimit`, `dmHistoryLimit`, `dms.*.historyLimit`
- доставка: `textChunkLimit` (по умолчанию `2000`), `maxLinesPerMessage` (по умолчанию `17`)
- потоковая передача: `streaming.mode`, `streaming.chunkMode`, `streaming.preview.*`, `streaming.progress.*`, `streaming.block.*` (устаревшие плоские ключи `streamMode`, `draftChunk`, `blockStreaming`, `blockStreamingCoalesce`, `chunkMode` переносятся в `streaming.*` командой `openclaw doctor --fix`)
- медиафайлы и повторные попытки: `mediaMaxMb` (ограничивает размер исходящих загрузок в Discord, по умолчанию `100`), `retry`
- действия: `actions.*`
- присутствие: `activity`, `status`, `activityType`, `activityUrl`, `autoPresence.*`
- интерфейс: `ui.components.accentColor`
- возможности: `threadBindings`, верхнеуровневый `bindings[]` (`type: "acp"`), `pluralkit`, `execApprovals`, `intents`, `agentComponents.enabled`, `agentComponents.ttlMs`, `heartbeat`, `responsePrefix`

</Accordion>

## Безопасность и эксплуатация

- Считайте токены ботов секретами (в управляемых средах предпочтительно использовать `DISCORD_BOT_TOKEN`).
- Предоставляйте Discord минимально необходимые разрешения.
- Если развертывание или состояние команд устарело, перезапустите Gateway и повторно проверьте его командой `openclaw channels status --probe`.
