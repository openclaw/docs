Raft подключает агента OpenClaw к внешнему агенту Raft через локальный
CLI Raft. Raft отправляет аутентифицированные сигналы пробуждения в Gateway; затем агент
использует CLI Raft для проверки и отправки сообщений. Поддерживаются только личные чаты (без групп).

## Установка

Raft — официальный внешний Plugin. Установите его на узле Gateway:

```bash
openclaw plugins install @openclaw/raft
openclaw gateway restart
```

Подробности: [Плагины](/ru/tools/plugin)

## Предварительные требования

- Рабочее пространство Raft с внешним агентом.
- CLI Raft, установленный на том же узле, что и Gateway OpenClaw, и доступный
  в `PATH` службы.
- Профиль CLI Raft, в котором уже выполнен вход и который связан с этим
  внешним агентом.

Plugin не хранит учетные данные Raft; CLI Raft хранит данные
аутентификации в собственном профиле.

## Настройка

Укажите профиль в конфигурации:

```json5
{
  channels: {
    raft: {
      enabled: true,
      profile: "openclaw",
    },
  },
}
```

Для учетной записи по умолчанию вместо этого можно задать `RAFT_PROFILE` в окружении
Gateway:

```bash
RAFT_PROFILE=openclaw
```

Используйте именованные учетные записи, если один Gateway подключается более чем к одному внешнему агенту Raft:

```json5
{
  channels: {
    raft: {
      accounts: {
        support: {
          profile: "support-agent",
        },
        engineering: {
          profile: "engineering-agent",
        },
      },
    },
  },
}
```

При интерактивной настройке сохраняется тот же профиль:

```bash
openclaw channels add --channel raft
```

## Принцип работы

При запуске Gateway Plugin:

1. Открывает HTTP-конечную точку пробуждения, доступную только через local loopback, на временном порту.
2. Запускает `raft --profile <profile> agent bridge` с этой конечной точкой и
   отдельным токеном для каждого процесса.
3. Принимает от локального моста только аутентифицированные сигналы пробуждения без содержимого
   с идентификатором для защиты от повторного воспроизведения.
4. Требует наличия одного из полей `eventId`, `attemptId`, `messageId`, `delivery_id`,
   `wake_id` или `id` в каждой полезной нагрузке пробуждения.
5. В течение 24 часов устраняет дубликаты повторных доставок сигналов пробуждения по идентификатору события моста,
   в том числе после перезапусков Gateway.
6. Возвращает стабильный сеанс среды выполнения для текущего моста и пустой
   пакет выгрузки активности для протокола CLI Raft.
7. Запускает по одному последовательному ходу агента OpenClaw для каждого принятого сигнала пробуждения.

Мост управляет повторными попытками доставки и переподключениями Raft. Ход OpenClaw
получает только уведомление о пробуждении, а не копию тела сообщения Raft. Для чтения
ожидающих сообщений и отправки ответа он использует CLI:

```bash
raft --profile openclaw message check
raft --profile openclaw message send
```

<Note>
Raft не является транспортом push-сообщений. OpenClaw не отправляет автоматически итоговый текст модели обратно через мост, поэтому после обработки сигнала пробуждения агент должен использовать CLI Raft.
</Note>

## Проверка

Убедитесь, что OpenClaw может найти CLI и что профиль настроен:

```bash
openclaw channels status --probe
openclaw plugins inspect raft --runtime --json
```

Затем отправьте сообщение внешнему агенту Raft. В журнале Gateway сначала должен появиться
запуск моста Raft, а затем — входящий сигнал пробуждения. Агент должен использовать
настроенный профиль Raft для проверки ожидающих сообщений.

## Устранение неполадок

<AccordionGroup>
  <Accordion title="CLI Raft отсутствует">
    Установите CLI Raft на узле Gateway и обеспечьте доступность `raft` в
    `PATH` службы. Проверьте его командой `raft --help`, затем перезапустите Gateway.
  </Accordion>
  <Accordion title="Мост сразу завершает работу">
    Убедитесь, что в настроенном профиле выполнен вход и что профиль принадлежит нужному
    внешнему агенту Raft. Запустите `raft --profile <profile> agent bridge` напрямую,
    чтобы просмотреть диагностическое сообщение CLI.
  </Accordion>
  <Accordion title="Сигнал пробуждения поступает, но ответ Raft не отправляется">
    Это ожидаемо, если агент не вызывает CLI Raft. Мост
    пробуждения не передает тела сообщений и не отправляет итоговые ответы автоматически. Проверьте
    политику инструментов агента и убедитесь, что он может выполнять `raft --profile <profile>
    message check` и `message send`.
  </Accordion>
</AccordionGroup>
