Nostr — это загружаемый Plugin канала (`@openclaw/nostr`), который позволяет OpenClaw получать зашифрованные личные сообщения NIP-04 через ретрансляторы Nostr и отвечать на них. Одна учётная запись на Gateway; только личные сообщения.

## Установка

```bash
openclaw plugins install @openclaw/nostr
```

Используйте спецификацию пакета без версии, чтобы получать текущий официальный тег выпуска. Фиксируйте точную версию только тогда, когда требуется воспроизводимая установка.

Из локальной рабочей копии (для процессов разработки):

```bash
openclaw plugins install --link <path-to-local-nostr-plugin>
```

Перезапустите Gateway после установки или включения плагинов. После установки Plugin Nostr становится доступен при первоначальной настройке (`openclaw onboard`) и выполнении `openclaw channels add` через общий каталог каналов.

### Неинтерактивная настройка

```bash
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY"
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY" --relay-urls "wss://relay.damus.io,wss://relay.primal.net"
```

Используйте `--use-env`, чтобы хранить `NOSTR_PRIVATE_KEY` в окружении, а не сохранять ключ в конфигурации (только для учётной записи по умолчанию).

## Быстрая настройка

1. Создайте пару ключей Nostr (при необходимости):

```bash
# С помощью nak
nak key generate
```

2. Добавьте в конфигурацию:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
    },
  },
}
```

3. Экспортируйте ключ:

```bash
export NOSTR_PRIVATE_KEY="nsec1..."
```

4. Перезапустите Gateway.

## Справочник по конфигурации

| Ключ         | Тип      | Значение по умолчанию                        | Описание                                                        |
| ------------ | -------- | -------------------------------------------- | --------------------------------------------------------------- |
| `privateKey` | строка   | обязательно                                  | Закрытый ключ в формате `nsec` или hex; разрешены ссылки на секреты |
| `relays`     | строка[] | `['wss://relay.damus.io', 'wss://nos.lol']`  | URL-адреса ретрансляторов (WebSocket)                            |
| `dmPolicy`   | строка   | `pairing`                                    | Политика доступа к личным сообщениям                            |
| `allowFrom`  | строка[] | `[]`                                         | Разрешённые открытые ключи отправителей                         |
| `enabled`    | логический | `true`                                     | Включение или отключение канала                                 |
| `name`       | строка   | -                                            | Отображаемое имя                                                |
| `profile`    | объект   | -                                            | Метаданные профиля NIP-01                                       |

## Метаданные профиля

Данные профиля публикуются как событие NIP-01 `kind:0`. Ими можно управлять в интерфейсе управления (Channels -> Nostr -> Profile) или задать их непосредственно в конфигурации.

Пример:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      profile: {
        name: "openclaw",
        displayName: "OpenClaw",
        about: "Бот личных сообщений — персональный помощник",
        picture: "https://example.com/avatar.png",
        banner: "https://example.com/banner.png",
        website: "https://example.com",
        nip05: "openclaw@example.com",
        lud16: "openclaw@example.com",
      },
    },
  },
}
```

Примечания:

- URL-адреса профиля должны использовать `https://`.
- При импорте из ретрансляторов поля объединяются, а локальные переопределения сохраняются.

## Управление доступом

### Политики личных сообщений

- **pairing** (по умолчанию): неизвестные отправители получают код сопряжения.
- **allowlist**: личные сообщения могут отправлять только открытые ключи из `allowFrom`.
- **open**: общедоступные входящие личные сообщения (требуется `allowFrom: ["*"]`).
- **disabled**: игнорировать входящие личные сообщения.

Примечания о применении политик:

- Подписи входящих событий проверяются до применения политики отправителей и расшифровки NIP-04, поэтому поддельные события отклоняются на раннем этапе.
- Ответы с кодом сопряжения отправляются без расшифровки или обработки исходного содержимого личного сообщения.
- Частота входящих личных сообщений ограничивается как глобально, так и для каждого отправителя; слишком большие полезные данные отбрасываются до расшифровки.

### Пример списка разрешённых отправителей

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      dmPolicy: "allowlist",
      allowFrom: ["npub1abc...", "npub1xyz..."],
    },
  },
}
```

## Форматы ключей

Допустимые форматы:

- **Закрытый ключ:** `nsec...` или 64-символьная шестнадцатеричная строка
- **Открытые ключи (`allowFrom`):** `npub...` или шестнадцатеричная строка

## Ретрансляторы

По умолчанию: `relay.damus.io` и `nos.lol`.

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["wss://relay.damus.io", "wss://relay.primal.net", "wss://nostr.wine"],
    },
  },
}
```

Советы:

- Для резервирования используйте 2–3 ретранслятора.
- Не используйте слишком много ретрансляторов из-за задержек и дублирования.
- Платные ретрансляторы могут повысить надёжность.
- Локальные ретрансляторы подходят для тестирования (`ws://localhost:7777`).

## Поддержка протоколов

| NIP    | Состояние       | Описание                                      |
| ------ | --------------- | --------------------------------------------- |
| NIP-01 | Поддерживается  | Базовый формат событий и метаданные профиля   |
| NIP-04 | Поддерживается  | Зашифрованные личные сообщения (`kind:4`)     |
| NIP-17 | Запланировано   | Личные сообщения в защищённой обёртке         |
| NIP-44 | Запланировано   | Версионируемое шифрование                     |

## Тестирование

### Локальный ретранслятор

```bash
# Запуск strfry
docker run -p 7777:7777 ghcr.io/hoytech/strfry
```

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["ws://localhost:7777"],
    },
  },
}
```

### Ручная проверка

1. Найдите открытый ключ бота в журналах Gateway или с помощью `openclaw channels status` (в шестнадцатеричном формате; при необходимости преобразуйте его в npub в своём клиенте).
2. Откройте клиент Nostr (Amethyst, Damus и т. п.).
3. Отправьте личное сообщение на открытый ключ бота.
4. Убедитесь, что получен ответ.

## Устранение неполадок

### Сообщения не поступают

- Убедитесь, что закрытый ключ действителен.
- Убедитесь, что URL-адреса ретрансляторов доступны и используют `wss://` (или `ws://` для локального подключения).
- Убедитесь, что для `enabled` не задано значение `false`.
- Проверьте журналы Gateway на наличие ошибок подключения к ретрансляторам.

### Ответы не отправляются

- Убедитесь, что ретранслятор принимает записи.
- Проверьте исходящее подключение.
- Учитывайте ограничения частоты запросов ретранслятора.

### Дублирующиеся ответы

- Это ожидаемое поведение при использовании нескольких ретрансляторов.
- Сообщения дедуплицируются по идентификатору события; ответ инициирует только первая доставка.

## Безопасность

- Никогда не добавляйте закрытые ключи в репозиторий.
- Используйте переменные окружения для хранения ключей.
- Для рабочих ботов рекомендуется использовать `allowlist`.
- Подписи проверяются до применения политики отправителей, а политика отправителей применяется до расшифровки, поэтому поддельные события отклоняются на раннем этапе, а неизвестные отправители не могут инициировать полный объём криптографических операций.

## Ограничения (MVP)

- Только личные сообщения (без групповых чатов).
- Нет вложений мультимедиа.
- Только NIP-04 (поддержка защищённой обёртки NIP-17 запланирована).
