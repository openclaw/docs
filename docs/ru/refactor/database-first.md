# Рефакторинг состояния с приоритетом базы данных

## Решение

Использовать двухуровневую структуру SQLite:

- Глобальная база данных: `~/.openclaw/state/openclaw.sqlite`
- База данных агента: одна база данных SQLite на каждого агента для рабочей области, принадлежащей агенту,
  транскрипта, VFS, артефактов и большого состояния runtime на уровне агента
- Конфигурация остается файловой: `openclaw.json` остается вне
  базы данных. Runtime-профили аутентификации переносятся в SQLite; файлы учетных данных внешнего провайдера или CLI
  остаются управляемыми владельцем вне базы данных OpenClaw.

Глобальная база данных является базой данных control-plane. Она владеет обнаружением агентов,
общим состоянием Gateway, сопряжением, состоянием устройств/узлов, журналами задач и flow, состоянием Plugin,
runtime-состоянием планировщика, метаданными резервных копий и состоянием миграций.

База данных агента является базой данных data-plane. Она владеет метаданными сессии агента,
потоком событий транскрипта, рабочей областью VFS или scratch-пространством имен, артефактами инструментов,
артефактами запусков и доступными для поиска/индексации локальными cache-данными агента.

Это дает одно долговечное глобальное представление без принудительного помещения больших рабочих областей агентов,
транскриптов и бинарных scratch-данных в общую полосу записи Gateway.

## Жесткий контракт

У этой миграции есть одна каноническая runtime-форма:

- Строки сессий сохраняют только метаданные сессии. Они не должны сохранять
  `transcriptLocator`, пути к файлам транскриптов, sibling JSONL-пути, пути блокировок,
  метаданные pruning или указатели совместимости файловой эпохи.
- Идентичность транскрипта всегда является SQLite-идентичностью: `{agentId, sessionId}` плюс
  необязательные метаданные topic там, где они нужны протоколу.
- `sqlite-transcript://...` не является runtime- или протокольной идентичностью. Новый код не должен
  выводить, сохранять, передавать, парсить или мигрировать локаторы транскриптов. Runtime и
  тесты вообще не должны содержать псевдолокаторов; документация может упоминать эту строку
  только чтобы запретить ее.
- Устаревшие `sessions.json`, JSONL транскриптов, `.jsonl.lock`, pruning, truncation
  и старая логика путей сессий относятся только к пути миграции/импорта doctor.
- Устаревшие alias конфигурации сессий относятся только к миграции doctor. Runtime не
  интерпретирует `session.idleMinutes`, `session.resetByType.dm` или
  cross-agent alias основной сессии `agent:main:*` для другого настроенного агента.
- Идентичность маршрутизации сессий является типизированным реляционным состоянием. Горячие runtime- и UI-пути
  должны читать `sessions.session_scope`, `sessions.account_id`,
  `sessions.primary_conversation_id`, `conversations` и
  `session_conversations`; они не должны парсить `session_key` или извлекать
  `session_entries.entry_json` для идентичности провайдера, кроме как в виде compatibility-shadow,
  пока старые call site удаляются.
- Маркеры direct-message на уровне канала, такие как `dm` и `direct`, являются словарем маршрутизации,
  а не локаторами транскриптов или дескрипторами совместимости file-store.
- Устаревшая конфигурация hook handler относится только к поверхностям предупреждений/миграции doctor.
  Runtime не должен загружать `hooks.internal.handlers`; hooks запускаются только через обнаруженные
  директории hook и метаданные `HOOK.md`.
- Запуск runtime, горячие пути ответов, Compaction, reset, восстановление, диагностика,
  TTS, memory hooks, subagents, маршрутизация команд Plugin, границы протокола и
  hooks должны передавать `{agentId, sessionId}` через runtime.
- Тесты должны seed и assert строки SQLite-транскриптов через
  `{agentId, sessionId}`. Тесты, которые доказывают только передачу JSONL-пути,
  сохранение caller-supplied locator или совместимость transcript-file, должны
  быть удалены, если они не покрывают импорт doctor, non-session support/debug
  materialization или форму протокола.
- `runEmbeddedPiAgent(...)`, подготовленные worker-запуски и внутренняя embedded
  попытка не должны принимать локаторы транскриптов. Они открывают SQLite transcript
  manager по `{agentId, sessionId}` и передают этот manager во внутреннюю
  PI-совместимую сессию агента, чтобы устаревшие вызывающие стороны не могли заставить runner писать
  JSON/JSONL-транскрипты.
- Диагностика runner должна хранить runtime/cache/payload trace-записи в SQLite.
  Runtime-диагностика не должна раскрывать параметры переопределения JSONL-файлов или общие
  helpers экспорта JSONL-транскриптов; пользовательские экспорты могут materialize явные
  артефакты из строк базы данных, не передавая имена файлов обратно в runtime.
- Сырое stream-логирование использует `OPENCLAW_RAW_STREAM=1` плюс SQLite diagnostic rows.
  Старый контракт pi-mono file logger `PI_RAW_STREAM`, `PI_RAW_STREAM_PATH` и
  `raw-openai-completions.jsonl` не является частью runtime или тестов OpenClaw.
- Индексация памяти QMD не должна экспортировать SQLite-транскрипты в markdown-файлы.
  QMD индексирует только настроенные файлы памяти; поиск по транскриптам сессий остается
  основанным на SQLite.
- Подпуть QMD SDK предназначен только для QMD в новом коде. Helpers индексации SQLite-транскриптов сессий
  живут в `memory-core-host-engine-session-transcripts`; любой QMD re-export является только совместимостью
  и не должен использоваться runtime-кодом.
- Встроенные memory indexes живут в базе данных агента-владельца. Runtime config и
  resolved runtime contracts не должны раскрывать `memorySearch.store.path`; doctor
  удаляет этот устаревший ключ конфигурации, а текущий код передает агентский
  `databasePath` внутренне.

Работа по реализации должна продолжать удалять код, пока эти утверждения не станут истинными
без исключений вне границ doctor/import/export/debug.

## Целевое состояние и прогресс

### Жесткая цель

- Одна глобальная база данных SQLite владеет состоянием control-plane:
  `state/openclaw.sqlite`.
- Одна база данных SQLite на каждого агента владеет состоянием data-plane:
  `agents/<agentId>/agent/openclaw-agent.sqlite`.
- Конфигурация остается файловой. `openclaw.json` не является частью этого
  рефакторинга базы данных.
- Устаревшие файлы являются только входными данными миграции doctor.
- Runtime никогда не пишет и не читает JSONL сессий или транскриптов как активное состояние.

### Целевые состояния

- `not-started`: runtime-код файловой эпохи все еще пишет активное состояние.
- `migrating`: код doctor/import может переносить файловые данные в SQLite.
- `dual-read`: временный мост читает и SQLite, и устаревшие файлы. Это состояние
  запрещено для этого рефакторинга, если оно явно не задокументировано как
  doctor-only.
- `sqlite-runtime`: runtime читает и пишет только SQLite.
- `clean`: устаревшие runtime API и тесты удалены, а guard предотвращает
  регрессии.
- `done`: документация, тесты, backup, миграция doctor и changed checks доказывают
  clean-состояние.

### Текущее состояние

- Сессии: `clean` для runtime. Строки сессий живут в базе данных на уровне агента,
  runtime API используют `{agentId, sessionId}` или `{agentId, sessionKey}`, а
  `sessions.json` является только устаревшим входом doctor.
- Транскрипты: `clean` для runtime. События транскриптов, идентичности, snapshots
  и trajectory runtime events живут в базе данных на уровне агента. Runtime больше
  не принимает локаторы транскриптов или JSONL-пути транскриптов.
- PI embedded runner: `clean`. Embedded PI-запуски, подготовленные workers, Compaction
  и retry loops используют SQLite session scope и отклоняют устаревшие дескрипторы транскриптов.
- Cron: `clean` для runtime. Runtime использует `cron_jobs` и `cron_run_logs`;
  runtime-тесты используют именование SQLite `storeKey`, а файловые пути Cron остаются
  только в устаревших тестах миграции doctor.
- Реестр задач: `clean`. Runtime-строки Task и Task Flow живут в
  `state/openclaw.sqlite`; unshipped sidecar SQLite importers удалены.
- Состояние Plugin: `clean`. Строки состояния/blob Plugin живут в общей глобальной
  базе данных; старые sidecar SQLite helpers для plugin-state защищены guard.
- Память: `sqlite-runtime` для встроенной памяти и индексации транскриптов сессий.
  Таблицы memory index живут в базе данных на уровне агента, состояние памяти Plugin использует
  общие строки plugin-state, а устаревшие файлы памяти являются входами миграции doctor
  или содержимым пользовательской рабочей области.
- Backup: `sqlite-runtime`. Этапы backup compact SQLite snapshots, опускают live
  WAL/SHM sidecars, проверяют целостность SQLite и записывают backup-запуски в
  глобальную базу данных.
- Миграция doctor: `migrating`, намеренно. Doctor импортирует устаревшие JSON,
  JSONL и retired sidecar stores в SQLite, записывает migration runs/sources
  и удаляет успешно обработанные sources.
- E2E-скрипты: `clean` для runtime-покрытия. Docker MCP seeding пишет SQLite
  rows. Docker-скрипт runtime-context создает устаревший JSONL только внутри
  seed миграции doctor и явно называет устаревший путь индекса сессий.

### Не допускать регрессий

- Никаких локаторов транскриптов.
- Никаких активных файлов сессий.
- Никаких фальшивых JSONL test fixtures, кроме устаревших тестов миграции doctor.
- Никакого сырого доступа к SQLite там, где ожидается Kysely.
- Никаких новых устаревших DB migrations. Эта структура не shipped; держать версию схемы
  на `1`, если нет сильной причины.

## Предположения чтения кода

Нет последующих продуктовых решений, которые блокируют этот план. Реализация должна
продолжаться с этими предположениями:

- Используйте `node:sqlite` напрямую и требуйте среду выполнения Node 22+ для этого пути
  хранения.
- Оставьте ровно один обычный файл конфигурации. Не переносите конфигурацию, манифесты plugin
  или рабочие области Git в SQLite в рамках этого рефакторинга.
- Файлы совместимости времени выполнения не требуются. Устаревшие файлы JSON и JSONL являются
  только входными данными миграции. Локальные для ветки боковые файлы SQLite никогда не поставлялись и
  удаляются вместо импорта.
- `openclaw doctor --fix` владеет шагом миграции устаревших файлов в базу данных.
  Запуск времени выполнения и `openclaw migrate` не должны нести устаревшие пути
  обновления базы данных OpenClaw.
- Совместимость учетных данных следует тому же правилу: учетные данные времени выполнения живут в
  SQLite. Старые файлы `auth-profiles.json`, поагентные `auth.json` и общие
  `credentials/oauth.json` являются входными данными миграции doctor, затем удаляются
  после импорта.
- Состояние сгенерированного каталога моделей поддерживается базой данных. Код времени выполнения не должен записывать
  `agents/<agentId>/agent/models.json`; существующие файлы `models.json` являются устаревшими
  входными данными doctor и удаляются после импорта в `agent_model_catalogs`.
- Время выполнения не должно мигрировать, нормализовать или связывать локаторы транскриптов. Активная
  идентичность транскрипта — это `{agentId, sessionId}` в SQLite. Пути к файлам являются
  только устаревшими входными данными doctor, а `sqlite-transcript://...` должен исчезнуть из
  поверхностей времени выполнения, протокола, хуков и plugin вместо того, чтобы рассматриваться как
  граничный дескриптор.
- Чтения SQLite-транскриптов во времени выполнения не выполняют старые миграции формы записей JSONL и
  не переписывают транскрипты целиком ради совместимости. Устаревшая нормализация записей остается в
  явных утилитах doctor/импорта. Doctor нормализует устаревшие файлы транскриптов JSONL
  перед вставкой строк SQLite; текущие строки времени выполнения уже записаны в текущей
  схеме транскриптов. Экспорт траекторий/сессий читает эти строки как есть и не должен
  выполнять устаревшие миграции во время экспорта.
- Устаревшие помощники разбора/миграции JSONL-транскриптов предназначены только для doctor. Код формата
  транскриптов времени выполнения строит только текущий контекст SQLite-транскрипта; doctor
  владеет обновлениями старых записей JSONL перед вставкой строк.
- Старый принадлежащий времени выполнения помощник потоковой передачи JSONL-транскриптов был удален. Код
  импорта doctor владеет явным чтением устаревших файлов; история сессий времени выполнения читает
  строки SQLite.
- Привязки app-server Codex используют `sessionId` OpenClaw как канонический
  ключ в пространстве имен состояния plugin Codex. `sessionKey` — это метаданные для
  маршрутизации/отображения, и он не должен заменять долговечный идентификатор сессии или воскрешать
  идентичность файла транскрипта.
- Движки контекста получают текущий контракт времени выполнения напрямую. Реестр
  не должен оборачивать движки прокладками повторных попыток, которые удаляют `sessionKey`,
  `transcriptScope` или `prompt`; движки, которые не могут принять текущие
  database-first параметры, должны явно завершаться ошибкой вместо того, чтобы связываться мостом.
- Вывод резервной копии должен оставаться одним архивным файлом. Содержимое базы данных должно попадать
  в этот архив как компактные снимки SQLite, а не как сырые активные боковые файлы WAL.
- Поиск по транскриптам полезен, но не требуется для первого database-first
  среза. Спроектируйте схему так, чтобы FTS можно было добавить позже.
- Выполнение worker должно оставаться экспериментальным за настройками, пока граница базы данных
  стабилизируется.

## Результаты чтения кода

Текущая ветка уже вышла за стадию proof-of-concept. Общая
база данных существует, Node `node:sqlite` подключен через небольшой помощник времени выполнения, а
бывшие хранилища теперь записывают в `state/openclaw.sqlite` или во владеющую
базу данных `openclaw-agent.sqlite`.

Оставшаяся работа — не выбор SQLite; она в том, чтобы держать новую границу чистой
и удалить все интерфейсы, похожие на совместимость, которые все еще выглядят как старый
файловый мир:

- Session `storePath` больше не является идентичностью времени выполнения, формой тестовой фикстуры или
  полем полезной нагрузки статуса. Тесты времени выполнения и bridge больше не содержат
  имя контракта `storePath`; код doctor/миграции владеет этим устаревшим словарем.
- Записи сессий больше не проходят через старую внутрипроцессную очередь `store-writer.ts`.
  Записи патчей SQLite вместо этого используют обнаружение конфликтов и ограниченные повторные попытки.
- Устаревшее обнаружение путей все еще имеет допустимые сценарии миграции, но код времени выполнения должен
  перестать рассматривать `sessions.json` и файлы транскриптов JSONL как возможные цели записи.
- Таблицы, принадлежащие агенту, живут в поагентных базах данных SQLite. Глобальная БД хранит
  строки реестра/плоскости управления; идентичность транскрипта — это `{agentId, sessionId}` в
  поагентных строках транскриптов. Код времени выполнения не должен сохранять пути к файлам транскриптов
  или мигрировать локаторы транскриптов.
- Doctor уже импортирует несколько устаревших файлов. Очистка заключается в том, чтобы сделать это
  одной явной реализацией миграции, которую вызывает doctor, с долговечным
  отчетом о миграции.

Дополнительных продуктовых вопросов, блокирующих реализацию, нет.

## Текущая форма кода

Ветка уже имеет настоящую общую базу SQLite:

- Минимальная версия среды выполнения теперь Node 22+: `package.json`, runtime-guard CLI,
  настройки установщика по умолчанию, локатор среды выполнения macOS, CI и публичная
  документация по установке согласованы. Старый канал совместимости с Node 22 удален.
- `src/state/openclaw-state-db.ts` открывает `openclaw.sqlite`, задает WAL,
  `synchronous=NORMAL`, `busy_timeout=30000`, `foreign_keys=ON` и применяет
  сгенерированный модуль схемы, полученный из
  `src/state/openclaw-state-schema.sql`.
- Типы таблиц Kysely и runtime-модули схемы генерируются из одноразовых
  баз данных SQLite, созданных из закоммиченных файлов `.sql`; runtime-код больше
  не хранит скопированные вручную строки схем для глобальных, агентских или proxy
  capture-баз данных.
- Runtime-хранилища выводят типы выбранных и вставленных строк из этих
  сгенерированных интерфейсов Kysely `DB`, а не вручную дублируют формы строк
  SQLite. Raw SQL остается ограниченным применением схем, pragmas и DDL только
  для миграций.
- Схемы SQLite сведены к `user_version = 1`, потому что этот макет базы данных
  еще не поставлялся. Runtime-открыватели создают только текущую схему;
  импорт из файлов в базу данных остается в коде doctor, а локальные для ветки
  помощники обновления базы данных удалены.
- Реляционное владение принудительно обеспечивается там, где граница владения
  канонична: строки исходных миграций каскадно удаляются из `migration_runs`,
  состояние доставки задач каскадно удаляется из `task_runs`, а строки
  идентичности транскрипта каскадно удаляются из событий транскрипта.
- Текущие общие таблицы включают `agent_databases`,
  `auth_profile_stores`, `auth_profile_state`,
  `plugin_state_entries`, `plugin_blob_entries`, `media_blobs`,
  `skill_uploads`, `capture_sessions`, `capture_events`, `capture_blobs`,
  `sandbox_registry_entries`, `cron_run_logs`, `cron_jobs`, `commitments`,
  `delivery_queue_entries`, `model_capability_cache`,
  `workspace_setup_state`, `native_hook_relay_bridges`,
  `current_conversation_bindings`, `plugin_binding_approvals`,
  `tui_last_sessions`, `acp_sessions`, `acp_replay_sessions`,
  `acp_replay_events`, `task_runs`, `task_delivery_state`, `flow_runs`,
  `subagent_runs`, `migration_runs` и `backup_runs`.
- Произвольное состояние, принадлежащее плагинам, не получает типизированные
  таблицы, принадлежащие хосту. Установленные плагины используют
  `plugin_state_entries` для версионированных JSON-полезных нагрузок и
  `plugin_blob_entries` для байтов, с владением namespace/key, очисткой TTL,
  резервным копированием и записями миграций плагинов. Состояние оркестрации
  плагинов, принадлежащее хосту, по-прежнему может иметь типизированные таблицы,
  когда хост владеет контрактом запросов, например `plugin_binding_approvals`.
- Миграции Plugin — это миграции данных в namespace, принадлежащих плагинам,
  а не миграции схемы хоста. Плагин может мигрировать собственные
  версионированные записи состояния/blob через provider миграций, а хост
  записывает статус источника/запуска в обычный журнал миграций. Новые установки
  плагинов не требуют изменения `openclaw-state-schema.sql`, если только сам
  хост не берет на себя владение новым межплагиновым контрактом.
- `src/state/openclaw-agent-db.ts` открывает
  `agents/<agentId>/agent/openclaw-agent.sqlite`, регистрирует базу данных в
  глобальной БД и владеет агентскими таблицами сеансов, транскриптов, VFS,
  артефактов, кэша и индекса памяти. Общее runtime-обнаружение теперь читает
  сгенерированный типизированный реестр `agent_databases`, а не реализует этот
  запрос заново в каждой точке вызова.
- Глобальные и агентские базы данных записывают строку `schema_meta` с ролью
  базы данных, версией схемы, временными метками и id агента для агентских баз
  данных. Макет по-прежнему остается на `user_version = 1`, потому что эта
  схема SQLite еще не поставлялась.
- Идентичность агентских сеансов теперь имеет каноническую корневую таблицу
  `sessions` с ключом `session_id`, где `session_key`, `session_scope`,
  `account_id`, `primary_conversation_id`, временные метки, отображаемые поля,
  метаданные модели, id harness и связь parent/spawn являются запрашиваемыми
  столбцами. `session_routes` — это уникальный активный индекс маршрута от
  `session_key` к текущему `session_id`, поэтому ключ маршрута может перейти
  к новому долговечному сеансу, не заставляя горячие чтения выбирать между
  дублирующимися строками `sessions.session_key`. Старый payload совместимой
  формы `session_entries.entry_json` привязан к долговечному корню `session_id`
  внешним ключом; он больше не является единственным представлением сеанса
  на уровне схемы.
- Агентская идентичность внешних разговоров тоже реляционная:
  `conversations` хранит нормализованную идентичность provider/account/conversation,
  а `session_conversations` связывает один сеанс OpenClaw с одним или несколькими
  внешними разговорами. Это покрывает shared-main DM-сеансы, где несколько
  участников могут намеренно сопоставляться с одним сеансом без искажения
  `session_key`. SQLite также обеспечивает уникальность естественной
  идентичности provider, чтобы один и тот же кортеж channel/account/kind/peer/thread
  не мог разветвиться на разные id разговоров.
  Shared-main direct peers связываются с ролью `participant`, поэтому один
  сеанс OpenClaw может представлять несколько внешних DM-участников, не понижая
  более старых участников до расплывчатых связанных строк. `sessions.primary_conversation_id`
  по-прежнему указывает на текущую типизированную цель доставки. Закрытые
  столбцы маршрутизации/статуса обеспечиваются ограничениями SQLite `CHECK`,
  а не только TypeScript unions.
  Runtime-проекция сеанса очищает совместимые тени маршрутизации из
  `session_entries.entry_json` перед применением типизированных столбцов
  session/conversation, поэтому устаревшие JSON-полезные нагрузки не могут
  воскресить цели доставки.
  Маршрутизация объявлений subagent также требует типизированный контекст
  доставки SQLite; она больше не откатывается к совместимым полям маршрута
  `SessionEntry`.
  Явное наследование доставки Gateway `chat.send` читает типизированный контекст
  доставки SQLite вместо совместимых полей `origin`/`last*`.
  `tools.effective` аналогично выводит контекст provider/account/thread из
  типизированных строк доставки/маршрутизации SQLite, а не из устаревших теней
  `last*` в session-entry.
  Контекст prompt для системных событий восстанавливает поля channel/to/account/thread
  из типизированных полей доставки вместо теней `origin`.
  Общий помощник `deliveryContextFromSession` и mapper session-to-conversation
  теперь полностью игнорируют `SessionEntry.origin`; только типизированные поля
  доставки и реляционные строки разговоров могут создавать горячую идентичность
  маршрута.
  Runtime-нормализация записи сеанса удаляет `origin` перед сохранением или
  проекцией `entry_json`, а входящие метаданные записывают типизированные поля
  channel/chat плюс реляционные строки разговоров вместо создания новых теней
  origin.
- События транскриптов, снимки транскриптов и runtime-события траектории теперь
  ссылаются на канонический агентский корень `sessions` и каскадно удаляются
  при удалении сеанса. Строки идентичности/идемпотентности транскрипта
  продолжают каскадно удаляться от точной строки события транскрипта.
- Индексы memory-core теперь используют явные таблицы агентской базы данных
  `memory_index_meta`, `memory_index_sources`, `memory_index_chunks` и
  `memory_embedding_cache`, а `memory_index_state` отслеживает изменения
  revision. Необязательные побочные индексы FTS/vector называются
  `memory_index_chunks_fts` и `memory_index_chunks_vec` вместо общих таблиц
  `meta`, `files`, `chunks`, `chunks_fts` или `chunks_vec`. Канонические имена
  сохраняют текущую форму строк path/source и совместимость сериализованных
  embeddings. Эти таблицы являются производным/search-кэшем, а не каноническим
  хранилищем транскриптов; их можно удалить и перестроить из файлов рабочей
  области памяти и настроенных источников.
  Открытие поставлявшегося индекса памяти с общими именами мигрирует его
  метаданные, источники, chunks и кэш embeddings в канонические таблицы;
  производные таблицы FTS/vector перестраиваются под каноническими именами.
- Состояние восстановления запусков subagent теперь живет в типизированных
  общих строках `subagent_runs` с индексированными ключами сеансов child,
  requester и controller. Старый файл `subagents/runs.json` является только
  входом миграции doctor.
- Текущие привязки разговоров теперь живут в типизированных общих строках
  `current_conversation_bindings` с ключом по нормализованному id разговора,
  где целевые столбцы agent/session, kind разговора, статус, срок действия и
  метаданные хранятся как реляционные столбцы вместо дублированной непрозрачной
  записи привязки.
  Долговечный ключ привязки включает нормализованный kind разговора, поэтому
  ссылки direct/group/channel не могут конфликтовать, а SQLite отклоняет
  недопустимые значения kind/status привязки. Старый файл
  `bindings/current-conversations.json` является только входом миграции doctor.
- Восстановление очереди доставки теперь накладывает типизированные столбцы
  очереди для channel, target, account, session, retry, error, platform-send и
  recovery state поверх JSON воспроизведения. `entry_json` сохраняет payload
  воспроизведения, hooks и payload форматирования, но типизированные столбцы
  являются авторитетными для горячей маршрутизации/состояния очереди.
- Указатели восстановления последнего сеанса TUI теперь живут в типизированных
  общих строках `tui_last_sessions` с ключом по хэшу scope подключения/сеанса
  TUI. Старый JSON-файл TUI является только входом миграции doctor.
- Предпочтения TTS по умолчанию теперь живут в общих строках SQLite состояния
  плагина с ключом под плагином `speech-core`. Старый файл `settings/tts.json`
  является только входом миграции doctor; runtime больше не читает и не пишет
  JSON-файлы предпочтений TTS, а resolver устаревшего пути живет в модуле
  миграции doctor.
- Метаданные secret target теперь говорят о хранилищах, а не притворяются, что
  каждая credential target является конфигурационным файлом. `openclaw.json`
  остается хранилищем конфигурации; цели auth-profile используют типизированные
  строки SQLite `auth_profile_stores`, где credentials в форме provider
  хранятся как JSON-payload.
- Аудит секретов больше не сканирует устаревшие агентские файлы `auth.json`.
  Doctor владеет предупреждением об этом устаревшем файле, его импортом и
  удалением.
- Устаревшие помощники путей auth profile теперь живут в legacy-коде doctor.
  Помощники путей core auth profile раскрывают идентичность и отображаемые
  расположения SQLite auth-store, а не runtime-пути `auth-profiles.json` или
  `auth-state.json`.
- Runtime-модули восстановления запусков subagent и кэша возможностей моделей
  OpenRouter теперь держат SQLite snapshot readers/writers отдельно от
  doctor-only помощников импорта legacy JSON. Возможности OpenRouter используют
  типизированные общие строки `model_capability_cache` под
  `provider_id = "openrouter"` вместо одного непрозрачного blob кэша или
  provider-specific таблицы хоста. `taskName` запуска subagent хранится в
  типизированном столбце `subagent_runs.task_name`; копия `payload_json` — это
  данные replay/debug, а не источник горячих полей отображения или поиска.
- `src/agents/filesystem/virtual-agent-fs.sqlite.ts` реализует SQLite VFS
  поверх таблицы агентской базы данных `vfs_entries`. Чтение каталогов,
  рекурсивный экспорт, удаления и переименования используют индексированные
  prefix-диапазоны `(namespace, path)` вместо сканирования всего namespace или
  зависимости от сопоставления пути через `LIKE`.
- `src/agents/runtime-worker.entry.ts` создает для каждого запуска SQLite VFS,
  хранилища артефактов tool, артефактов запуска и scoped cache для workers.
- Маркеры завершения workspace bootstrap теперь живут в типизированных общих
  строках `workspace_setup_state` с ключом по resolved workspace path вместо
  `.openclaw/workspace-state.json`; runtime больше не читает и не перезаписывает
  legacy-маркер workspace, а helper API больше не передают фиктивный путь
  `.openclaw/setup-state` только для вывода идентичности хранилища.
- Exec approvals теперь живут в singleton-строке типизированной общей SQLite
  `exec_approvals_config`. Doctor импортирует legacy
  `~/.openclaw/exec-approvals.json`; runtime-записи больше не создают, не
  перезаписывают и не сообщают этот файл как активное расположение хранилища.
  macOS companion читает и пишет ту же строку таблицы `state/openclaw.sqlite`;
  на диске он держит только Unix prompt socket, потому что это IPC, а не
  долговечное runtime-состояние.
- Runtime-модули device identity, device auth и bootstrap теперь держат свои
  SQLite snapshot readers/writers отдельно от doctor-only помощников импорта
  legacy JSON. Device identity использует типизированные строки
  `device_identities`, а device auth tokens используют типизированные строки
  `device_auth_tokens`. Записи device auth согласуют строки по device/role
  вместо усечения таблицы tokens, и runtime больше не маршрутизирует обновления
  single-token через старый whole-store adapter. Устаревший
  Полезные нагрузки JSON версии 1 существуют только как формы импорта/экспорта doctor.
- Кэш обмена токенов GitHub Copilot использует общую таблицу состояния Plugin в SQLite
  по пути `github-copilot/token-cache/default`. Это состояние кэша принадлежит провайдеру,
  поэтому оно намеренно не добавляет таблицу схемы хоста.
- Compaction GitHub Copilot больше не записывает боковые файлы рабочей области
  `openclaw-compaction-*.json`. Среда вызывает RPC Compaction истории SDK для
  отслеживаемой сессии SDK, а OpenClaw хранит долговечное состояние сессии/транскрипта в
  SQLite вместо маркерных файлов совместимости.
- Общая среда выполнения Swift (`OpenClawKit`) использует те же строки
  `state/openclaw.sqlite` для идентичности устройства и авторизации устройства. Помощники приложения macOS
  импортируют общие помощники SQLite вместо владения вторым JSON- или
  SQLite-путем. Оставшийся устаревший `identity/device.json` блокирует создание идентичности,
  пока doctor не импортирует его в SQLite, что соответствует стартовому шлюзу TypeScript и Android.
- Идентичность устройства Android использует тот же совместимый с TypeScript ключевой материал,
  сохраненный в типизированных строках `state/openclaw.sqlite#table/device_identities`. Она никогда
  не читает и не записывает `openclaw/identity/device.json`; оставшийся устаревший файл блокирует
  запуск, пока doctor не импортирует его в SQLite.
- Кэшированные токены авторизации устройства Android также используют типизированные
  строки `state/openclaw.sqlite#table/device_auth_tokens` и разделяют ту же
  семантику токенов версии 1, что TypeScript и Swift. Среда выполнения больше не читает ключи совместимости `SecurePrefs`
  `gateway.deviceToken*`; они относятся только к логике миграции/doctor.
- История недавних пакетов уведомлений Android использует типизированные
  строки `android_notification_recent_packages`. Среда выполнения больше не мигрирует и не
  читает старые CSV-ключи SharedPreferences.
- Создание идентичности устройства завершается закрытым отказом, когда существует устаревший `identity/device.json`,
  когда строка идентичности SQLite недействительна или когда хранилище идентичности SQLite
  невозможно открыть. Doctor сначала импортирует и удаляет этот файл, поэтому запуск среды выполнения
  не может молча сменить идентичность сопряжения до миграции.
- Выбор идентичности устройства — это ключ строки SQLite, а не локатор JSON-файла. Тесты
  и помощники Gateway передают явные ключи идентичности; только миграция doctor и
  стартовый шлюз с закрытым отказом знают имя удаленного файла `identity/device.json`.
- Совместимость сброса сессии теперь находится в миграции конфигурации doctor:
  `session.idleMinutes` переносится в `session.reset.idleMinutes`,
  `session.resetByType.dm` переносится в `session.resetByType.direct`, а
  политика сброса среды выполнения читает только канонические ключи сброса.
- Совместимость устаревшей конфигурации теперь находится в `src/commands/doctor/`. Обычная
  валидация `readConfigFileSnapshot()` не импортирует устаревшие детекторы doctor
  и не аннотирует устаревшие проблемы; `runDoctorConfigPreflight()` добавляет эти проблемы для
  исправления/отчетности doctor. Поток конфигурации doctor импортирует
  `src/commands/doctor/legacy-config.ts`, а исправление старого OAuth profile-id находится
  в
  `src/commands/doctor/legacy/oauth-profile-ids.ts`.
- Команды не-doctor не запускают автоматически исправление устаревшей конфигурации. Например,
  `openclaw update --channel` теперь завершается ошибкой при недействительной устаревшей конфигурации и просит
  пользователя запустить doctor, вместо того чтобы молча импортировать код миграции doctor.
- Web push, APNs, Voice Wake, проверки обновлений и здоровье конфигурации теперь используют типизированные общие таблицы SQLite
  для подписок, ключей VAPID, регистраций узлов, строк триггеров,
  строк маршрутизации, состояния уведомлений об обновлениях и записей здоровья конфигурации вместо
  целых непрозрачных JSON-блобов. Записи снимков Web push и APNs теперь согласуют
  подписки/регистрации по первичному ключу вместо очистки их таблиц;
  здоровье конфигурации делает то же самое по пути конфигурации.
  Их модули среды выполнения держат читатели/писатели снимков SQLite отдельно от
  помощников импорта устаревшего JSON, предназначенных только для doctor.
- Конфигурация Node-хоста теперь использует типизированную singleton-строку в общей базе данных SQLite;
  doctor импортирует старый файл `node.json` перед обычным использованием среды выполнения.
- Сопряжение устройства/узла, сопряжение канала, allowlist каналов и состояние bootstrap
  теперь используют типизированные строки SQLite вместо целых непрозрачных JSON-блобов. Одобрения привязок Plugin
  и состояние заданий Cron следуют тому же разделению: модули среды выполнения предоставляют
  операции на базе SQLite и нейтральные помощники снимков, а записи снимков сопряжения/bootstrap
  плюс одобрений привязок Plugin согласуют строки по первичному ключу
  вместо усечения таблиц, тогда как doctor импортирует/удаляет старые JSON-файлы через
  модули `src/commands/doctor/legacy/*`.
- Записи установленных Plugin теперь находятся в SQLite-индексе установленных Plugin.
  Чтение/запись конфигурации среды выполнения больше не мигрирует и не сохраняет старые
  данные авторской конфигурации `plugins.installs`; doctor импортирует эту устаревшую форму конфигурации
  в SQLite перед обычным использованием среды выполнения.
- Снимки восстановления учетных данных QQBot теперь находятся в состоянии Plugin SQLite по пути
  `qqbot/credential-backups`. Среда выполнения больше не записывает
  `qqbot/data/credential-backup*.json`; контракт doctor QQBot импортирует и
  архивирует эти устаревшие файлы резервных копий из активного каталога состояния.
- Планирование перезагрузки Gateway сравнивает снимки SQLite-индекса установленных Plugin в
  внутреннем пространстве различий `installedPluginIndex.installRecords.*`. Решения о перезагрузке среды выполнения
  больше не оборачивают эти строки в фиктивные объекты конфигурации `plugins.installs`.
- Обновление учетных данных именованной учетной записи Matrix больше не происходит во время чтения
  среды выполнения. Doctor владеет переименованием старого верхнеуровневого
  `credentials/matrix/credentials.json`, когда можно разрешить единственную/default учетную запись Matrix.
- Основные модули среды выполнения сопряжения и Cron больше не экспортируют построители
  устаревших JSON-путей. Устаревшие модули, принадлежащие doctor, строят исходные пути
  `pending.json`, `paired.json`, `bootstrap.json` и `cron/jobs.json` только для тестов импорта и
  миграции. Нормализация устаревшей формы заданий Cron и импорт журнала запусков Cron
  находятся в `src/commands/doctor/legacy/cron*.ts`.
- `src/commands/doctor/legacy/runtime-state.ts` импортирует устаревшие JSON-файлы состояния,
  включая конфигурацию хоста узла, в SQLite из doctor. Новые импортеры устаревших файлов
  остаются в `src/commands/doctor/legacy/`.
- `src/commands/doctor/state-migrations.ts` импортирует устаревшие `sessions.json` и
  транскрипты `*.jsonl` напрямую в SQLite и удаляет успешно импортированные источники. Он
  больше не подготавливает корневые устаревшие транскрипты через
  `agents/<agentId>/sessions/*.jsonl` и не создает каноническую цель JSONL перед
  импортом.
- Проверки целостности состояния doctor больше не сканируют устаревшие каталоги сессий и
  не предлагают удаление сиротских JSONL. Устаревшие файлы транскриптов являются только входными данными миграции,
  а шаг миграции владеет импортом и удалением источников.
- Импорт устаревшего реестра песочниц находится в
  `src/commands/doctor/legacy/sandbox-registry.ts`; активные чтения и записи реестра песочниц
  остаются только SQLite.
- Устаревшее исправление здоровья/импорта транскриптов сессий находится в
  `src/commands/doctor/legacy/session-transcript-health.ts`; командные модули среды выполнения
  больше не содержат парсинг транскриптов JSONL или код исправления активной ветки.

Основные моменты завершенной консолидации и удаления:

- Состояние Plugin теперь использует общую базу данных `state/openclaw.sqlite`. Старый
  импортер бокового файла `plugin-state/state.sqlite`, локального для ветки, удален, потому что
  эта структура SQLite никогда не поставлялась. Вспомогательные средства проб и тестов сообщают общий
  `databasePath` вместо раскрытия пути SQLite, специфичного для состояния Plugin.
- Таблицы среды выполнения задач и потоков задач теперь находятся в общей
  базе данных `state/openclaw.sqlite` вместо `tasks/runs.sqlite` и
  `tasks/flows/registry.sqlite`; старые импортеры боковых файлов удалены по той же
  причине: такая структура никогда не поставлялась.
- `src/config/sessions/store.ts` больше не требует `storePath` для входящих
  метаданных, обновлений маршрутов или чтения updated-at. Сохранение команд, очистка
  сессий CLI, глубина subagent, переопределения auth и идентичность сессии транскрипта
  используют API строк agent/session. Записи применяются как патчи строк SQLite
  с оптимистичной повторной попыткой при конфликте.
- Разрешение целей сессии теперь предоставляет цели баз данных для каждого агента, а не устаревшие
  пути `sessions.json`. Общий Gateway, метаданные ACP, исправление маршрутов doctor и
  `openclaw sessions` перечисляют `agent_databases` плюс настроенных агентов.
- Маршрутизация сессий Gateway теперь использует `resolveGatewaySessionDatabaseTarget`; возвращаемая
  цель несет `databasePath` и возможные ключи строк SQLite вместо устаревшего пути к файлу хранилища сессий.
- Типы среды выполнения сессий канала теперь предоставляют `{agentId, sessionKey}` для
  чтения updated-at, входящих метаданных и обновлений последнего маршрута. Старый
  совместимый тип `saveSessionStore(storePath, store)` удален.
- Среда выполнения Plugin, API расширений и barrel-поверхности `config/sessions` теперь направляют
  код Plugin к вспомогательным средствам строк сессий на базе SQLite. Экспорты совместимости
  корневой библиотеки (`loadSessionStore`, `saveSessionStore`, `resolveStorePath`) остаются
  как устаревшие shim-обертки для существующих потребителей. Старый
  вспомогательный метод `resolveLegacySessionStorePath` удален; построение устаревшего пути
  `sessions.json` теперь локально для миграции и тестовых фикстур.
- `src/config/sessions/session-entries.sqlite.ts` теперь хранит канонические записи сессий
  в базе данных каждого агента и поддерживает чтение/upsert/delete-патчи на уровне строк.
  Runtime upsert/patch/delete больше не сканирует варианты регистра и не удаляет устаревшие
  alias-ключи; doctor отвечает за каноникализацию. Отдельный вспомогательный метод импорта JSON
  удален, а миграция объединяет upsert более новых строк вместо замены всей таблицы сессий.
  Публичные вспомогательные методы read/list/load проецируют горячие метаданные сессий из типизированных
  строк `sessions` и `conversations`; `entry_json` является совместимой/отладочной тенью и может быть устаревшим
  или недействительным без потери типизированной идентичности сессии или контекста доставки.
- `src/config/sessions/delivery-info.ts` теперь разрешает контекст доставки из
  типизированных строк `sessions` + `conversations` + `session_conversations` для каждого агента.
  Он больше не реконструирует идентичность доставки runtime из
  `session_entries.entry_json`; отсутствующая типизированная строка беседы является проблемой
  миграции/исправления doctor, а не fallback среды выполнения.
- Решения сброса сохраненных сессий теперь предпочитают типизированные метаданные `sessions.session_scope`,
  `sessions.chat_type` и `sessions.channel`. Разбор `sessionKey`
  остается только для явных суффиксов thread/topic в целях команд; классификация сброса group vs
  direct больше не берется из формы ключа.
- Классификация отображения списка/статуса сессий теперь использует типизированные метаданные чата и
  вид сессии Gateway. Она больше не считает подстроки `:group:` или `:channel:`
  внутри `session_key` устойчивой истиной group/direct.
- Выбор политики silent-reply теперь использует только явный тип беседы или метаданные поверхности.
  Он больше не угадывает политику direct/group по подстрокам
  `session_key`.
- Разрешение модели отображения сессии теперь получает id агента из цели базы данных сессий SQLite,
  вместо извлечения его из `session_key`.
- Гидратация цели объявления agent-to-agent теперь использует только типизированный `deliveryContext`
  `sessions.list`. Она больше не восстанавливает маршрутизацию channel/account/thread
  из устаревшего `origin`, зеркальных полей `last*` или формы `session_key`.
- Отклонение thread-целей `sessions_send` теперь читает типизированные метаданные маршрутизации SQLite.
  Оно больше не отклоняет и не принимает цели путем разбора суффиксов thread
  из целевого ключа.
- Проверка политики инструментов в области группы теперь читает типизированную маршрутизацию бесед SQLite
  для текущей или созданной сессии. Она больше не доверяет идентичности group/channel
  через декодирование `sessionKey`; предоставленные вызывающим кодом id групп отбрасываются, когда
  нет типизированной строки сессии, которая их подтверждает.
- Сопоставление переопределений модели канала теперь использует явные метаданные группы и родительской
  беседы. Оно больше не декодирует id родительской беседы из
  `parentSessionKey`.
- Наследование сохраненных переопределений модели теперь требует явного ключа родительской сессии
  из типизированного контекста сессии. Оно больше не выводит родительские переопределения из
  суффиксов `:thread:` или `:topic:` в `sessionKey`.
- Старые обертка thread-info сессии и parser thread загруженного Plugin удалены;
  ни один runtime-код не импортирует `config/sessions/thread-info`.
- Вспомогательный метод бесед канала больше не предоставляет мосты разбора полного ключа сессии.
  Core по-прежнему нормализует принадлежащие провайдеру сырые id бесед через
  `resolveSessionConversation(...)`, но не реконструирует факты маршрута
  из `sessionKey`.
- Доставка завершений, политика отправки и обслуживание задач больше не выводят тип чата
  из формы `session_key`. Старый parser ключа типа чата удален;
  эти пути требуют типизированных метаданных сессии, типизированного контекста доставки или
  явного словаря целей доставки.
- Список/статус сессий, диагностика, привязка аккаунтов approval, фильтрация Heartbeat
  TUI и сводки использования больше не извлекают маршрутизацию provider/account/thread/display
  из `SessionEntry.origin`. Единственные оставшиеся чтения runtime
  `origin` относятся к несессионным концепциям или объектам доставки текущего хода.
- Нативный поиск беседы approval-request теперь читает типизированные строки маршрутизации сессий
  для каждого агента. Он больше не разбирает идентичность беседы channel/group/thread
  из `sessionKey`; отсутствующие типизированные метаданные являются проблемой миграции/исправления.
- Полезные нагрузки событий Gateway session changed/chat/session больше не дублируют
  `SessionEntry.origin` или теневые маршруты `last*`; клиенты получают типизированные
  `channel`, `chatType` и `deliveryContext`.
- Разрешение доставки Heartbeat теперь может получать типизированный SQLite
  `deliveryContext` напрямую, а runtime Heartbeat передает строку доставки сессии для каждого агента
  вместо опоры на совместимые тени `session_entries`
  для текущей маршрутизации.
- Разрешение цели доставки изолированного агента Cron также гидратирует свой текущий
  маршрут из типизированной строки доставки сессии для каждого агента перед fallback к
  совместимой полезной нагрузке записи.
- Разрешение origin объявления subagent теперь протягивает типизированный контекст доставки сессии requester
  через `loadRequesterSessionEntry` и предпочитает эту строку
  совместимым теням `last*`/`deliveryContext`.
- Обновления входящих метаданных сессии теперь сначала объединяются с типизированной строкой доставки
  для каждого агента; старые поля доставки `SessionEntry` используются только как fallback,
  когда типизированной строки беседы нет.
- Извлечение доставки restart/update теперь дает типизированному SQLite delivery
  `threadId` приоритет над фрагментами topic/thread, разобранными из `sessionKey`; разбор
  является только fallback для устаревших ключей в форме thread.
- Id каналов контекста hook-агента теперь предпочитают типизированную идентичность беседы SQLite,
  затем явные метаданные сообщения. Они больше не разбирают фрагменты provider/group/channel
  из `sessionKey`.
- Наследование внешнего маршрута Gateway `chat.send` теперь читает типизированные метаданные маршрутизации сессий SQLite
  вместо вывода области channel/direct/group из
  частей `sessionKey`. Сессии в области канала наследуют только когда типизированные
  канал сессии и тип чата совпадают с сохраненным контекстом доставки; сессии shared-main
  сохраняют свое более строгое правило CLI/no-client-metadata.
- Пробуждение restart-sentinel и маршрутизация продолжения теперь читают типизированные строки
  delivery/routing SQLite перед постановкой в очередь пробуждений Heartbeat или маршрутизированных продолжений agent-turn.
  Они больше не реконструируют контекст доставки из
  JSON-тени session-entry.
- Разрешение контекста Gateway `tools.effective` теперь читает типизированные строки
  delivery/routing SQLite для входных данных provider, account, target, thread и reply-mode.
  Оно больше не восстанавливает эти горячие поля маршрутизации из устаревших
  теней origin `session_entries.entry_json`.
- Маршрутизация realtime voice consult теперь разрешает доставку parent/call из типизированных
  строк сессий SQLite для каждого агента. Она больше не возвращается к совместимым
  теням `SessionEntry.deliveryContext` при выборе маршрута сообщения встроенного агента.
- Ретрансляция ACP spawn Heartbeat и маршрутизация parent-stream теперь читают родительскую доставку
  из типизированных строк сессий SQLite. Они больше не реконструируют родительский контекст доставки
  из совместимых теней session-entry.
- Сохранение маршрута доставки сессии теперь следует типизированным метаданным чата и
  сохраненным столбцам доставки. Оно больше не извлекает подсказки канала, маркеры direct/main
  или форму thread из `sessionKey`; внутренние маршруты webchat наследуют внешнюю цель только
  когда SQLite уже имеет типизированную/сохраненную идентичность доставки для сессии.
- Общее извлечение доставки сессии теперь читает только точную типизированную строку доставки
  сессии SQLite. Оно больше не разбирает суффиксы thread/topic и не откатывается
  от ключа в форме thread к базовому ключу сессии.
- Диспетчеризация ответов, восстановление restart sentinel и маршрутизация realtime voice consult
  теперь используют точные типизированные строки сессий/бесед SQLite для маршрутизации thread. Они
  больше не восстанавливают id thread или контекст доставки базовой сессии путем разбора
  ключей сессий в форме thread.
- Ограничение истории встроенного PI теперь использует типизированную проекцию маршрутизации сессий SQLite
  (`sessions` + primary `conversations`) для provider, типа чата
  и идентичности peer. Оно больше не разбирает provider, DM, group или форму thread
  из `sessionKey`.
- Вывод доставки инструментов Cron теперь использует только явную доставку или текущий типизированный
  контекст доставки. Он больше не декодирует цели channel, peer, account или thread
  из `agentSessionKey`.
- Строки runtime-сессий больше не несут старый alias маршрута `lastProvider`.
  Вспомогательные средства и тесты используют типизированные поля `lastChannel` и `deliveryContext`;
  миграция doctor — единственное место, которое должно переводить старые alias маршрутов
  или сохраненные тени `origin`.
- События транскрипта, строки VFS и строки артефактов инструментов теперь пишутся в базу данных
  каждого агента. Непоставлявшаяся глобальная таблица сопоставления файлов транскрипта удалена; doctor
  вместо этого записывает устаревшие исходные пути в устойчивые строки миграции.
- Поиск транскриптов runtime больше не сканирует byte offsets JSONL и не проверяет устаревшие
  файлы транскриптов. Пути Gateway chat/media/history читают строки транскриптов из
  SQLite; session JSONL теперь является только устаревшим входом doctor, а не состоянием runtime
  или форматом экспорта.

## Форма миграции doctor

Doctor должен вызывать один явный шаг миграции, который можно включать в отчет и безопасно
запускать повторно:

```bash
openclaw doctor --fix
```

`openclaw doctor --fix` вызывает реализацию миграции состояния после
обычной предварительной проверки конфигурации и создает проверенную резервную копию перед импортом. Запуск среды выполнения
и `openclaw migrate` не должны импортировать устаревшие файлы состояния OpenClaw.

Свойства миграции:

- Один проход миграции обнаруживает все устаревшие файловые источники и формирует план
  перед любыми изменениями.
- Doctor создает проверенный архив резервной копии перед миграцией до импорта
  устаревших файлов.
- Импорты идемпотентны и привязаны к пути источника, mtime, размеру, хэшу и целевой
  таблице.
- Успешно обработанные исходные файлы удаляются или архивируются после того, как целевая база данных
  зафиксировала изменения.
- Неудачные импорты оставляют источник нетронутым и записывают предупреждение в
  `migration_runs`.
- Код среды выполнения читает только SQLite после появления миграции.
- Путь отката/экспорта обратно в файлы среды выполнения не требуется.

## Инвентаризация миграции

Перенесите это в глобальную базу данных:

- Записи реестра задач среды выполнения теперь используют общую базу данных; не выпускавшийся
  сопутствующий импортёр `tasks/runs.sqlite` удалён. Сохранения снимков вставляют или обновляют по id задачи
  и удаляют только отсутствующие строки задач/доставок.
- Записи среды выполнения Task Flow теперь используют общую базу данных; не выпускавшийся
  сопутствующий импортёр `tasks/flows/registry.sqlite` удалён. Сохранения снимков
  вставляют или обновляют по id потока и удаляют только отсутствующие строки потоков.
- Записи состояния Plugin среды выполнения теперь используют общую базу данных; не выпускавшийся

- Очереди исходящей доставки и доставки сеансов теперь совместно используют глобальную таблицу SQLite
  `delivery_queue_entries` под отдельными именами очередей
  (`outbound-delivery`, `session-delivery`) вместо долговечных
  файлов `delivery-queue/*.json`, `delivery-queue/failed/*.json` и
  `session-delivery-queue/*.json`. Шаг doctor legacy-state импортирует
  ожидающие и неудавшиеся строки, удаляет устаревшие маркеры доставленных сообщений и удаляет старые
  файлы JSON после импорта. Поля горячей маршрутизации и повторов являются типизированными столбцами; полезная нагрузка
  JSON сохраняется только для повторного воспроизведения/отладки.
- Аренды процессов ACPX теперь используют состояние Plugin SQLite в `acpx/process-leases`
  вместо `process-leases.json`.
- Метаданные запусков резервного копирования и миграции

Перенести это в базы данных агентов:

- Корни сеансов агентов и payload session-entry в форме совместимости. Выполнено для
  записей среды выполнения: горячие метаданные сеанса доступны для запросов в `sessions`, а
  полный payload `SessionEntry` в устаревшей форме остаётся в `session_entries`.
- События транскриптов агентов. Выполнено для записей среды выполнения.
- Контрольные точки Compaction и снимки транскриптов. Выполнено для записей среды выполнения:
  копии транскриптов контрольных точек являются строками транскриптов SQLite, а метаданные контрольных точек
  записываются в `transcript_snapshots`. Помощники контрольных точек Gateway
  теперь называют эти значения снимками транскриптов, а не исходными файлами.
- Scratch/workspace пространства имён VFS агента. Выполнено для записей VFS среды выполнения.
- Payload вложений субагентов. Выполнено для записей среды выполнения: это seed-записи VFS SQLite,
  а не долговечные файлы рабочего пространства.
- Артефакты инструментов. Выполнено для записей среды выполнения.
- Артефакты запусков. Выполнено для записей среды выполнения worker через таблицу
  `run_artifacts` конкретного агента.
- Локальные кэши среды выполнения агента. Выполнено для записей scoped-кэша среды выполнения worker через
  таблицу `cache_entries` конкретного агента. Общие для Gateway кэши моделей остаются в
  глобальной базе данных, если не станут специфичными для агента.
- Журналы родительских потоков ACP. Выполнено для записей среды выполнения.
- Сеансы журнала replay ACP. Выполнено для записей среды выполнения через
  `acp_replay_sessions` и `acp_replay_events`; устаревший `acp/event-ledger.json`
  остаётся только как входные данные doctor.
- Метаданные сеансов ACP. Выполнено для записей среды выполнения через `acp_sessions`; устаревшие
  блоки `entry.acp` в `sessions.json` являются только входными данными миграции doctor.
- Сопутствующие файлы траекторий, когда они не являются явными файлами экспорта. Выполнено для записей среды выполнения:
  захват траектории записывает строки `trajectory_runtime_events` в базу данных агента
  и зеркалирует артефакты, ограниченные запуском, в SQLite. Устаревшие сопутствующие файлы являются только
  входными данными импорта doctor; экспорт может материализовать новые выходные JSONL для support bundle,
  но не читает и не мигрирует старые сопутствующие файлы траекторий/транскриптов во время выполнения.
  Захват траектории среды выполнения раскрывает область SQLite; помощники путей JSONL
  изолированы для поддержки экспорта/отладки и не реэкспортируются из модуля среды выполнения.
  Метаданные траектории встроенного runner записывают идентичность `{agentId, sessionId, sessionKey}`
  вместо сохранения локатора транскрипта.

Пока оставить файловыми:

- `openclaw.json`
- файлы учётных данных provider или CLI
- манифесты Plugin/пакетов
- пользовательские рабочие пространства и Git-репозитории, когда выбран дисковый режим
- журналы, предназначенные для отслеживания оператором, если конкретная поверхность журналирования не перенесена

## План миграции

### Фаза 0: Зафиксировать границу

Сделать границу долговечного состояния явной перед переносом дополнительных строк:

- Добавить таблицу `migration_runs` в глобальную базу данных.
  Выполнено для отчётов о выполнении миграции legacy-state.
- Добавить единую, принадлежащую doctor службу миграции состояния для импорта из файлов в базу данных.
  Выполнено: `openclaw doctor --fix` использует реализацию миграции legacy-state.
- Сделать `plan` доступным только для чтения, а `apply` должен создавать резервную копию, импортировать, проверять и
  затем удалять или помещать старые файлы в карантин.
  Выполнено: doctor создаёт проверенную резервную копию перед миграцией, передаёт путь резервной копии
  в `migration_runs` и повторно использует пути импортёра/удаления.
- Добавить статические запреты, чтобы новый код среды выполнения не мог записывать устаревшие файлы состояния, пока
  код миграции и тесты всё ещё могут их создавать/читать.
  Выполнено для текущих мигрированных устаревших хранилищ; проверка также сканирует вложенные
  тесты на запрещённые контракты локатора транскриптов среды выполнения.

### Фаза 1: Завершить глобальную плоскость управления

Держать общее координационное состояние в `state/openclaw.sqlite`:

- Агенты и реестр баз данных агентов
- Реестры задач и Task Flow
- Состояние Plugin
- Реестр sandbox-контейнеров/браузеров
- История запусков Cron/планировщика
- Сопряжение, устройство, push, проверка обновлений, TUI, кэши OpenRouter/моделей и другое
  небольшое состояние среды выполнения, ограниченное Gateway
- Метаданные резервного копирования и миграции
- Байты медиа-вложений Gateway. Выполнено для записей среды выполнения; прямые файловые пути
  являются временными материализациями для совместимости с отправителями каналов и staging sandbox.
  allowlist среды выполнения принимает пути материализации SQLite, а не устаревшие корни media state/config.
  Doctor импортирует устаревшие медиафайлы в `media_blobs` и удаляет исходные файлы после
  успешной записи строк.
- Сеансы захвата debug proxy, события и blob payload. Выполнено: захваты находятся
  в общей БД состояния и открываются через bootstrap общей БД состояния, схему,
  WAL и настройки busy-timeout. Байты payload сжаты gzip в
  `capture_blobs.data`; нет переопределения sidecar DB среды выполнения debug proxy,
  каталога blob или отдельной сгенерированной схемы/цели codegen только для proxy-capture.
  Миграция doctor/startup импортирует строки выпущенного `debug-proxy/capture.sqlite`
  и связанные blob payload, включая активные устаревшие переопределения окружения DB/blob,
  затем архивирует эти источники, оставляя сертификаты CA нетронутыми.

Эта фаза также удаляет дублирующие открыватели sidecar, помощники разрешений, настройку WAL,
очистку файловой системы и писатели совместимости из этих подсистем.

### Фаза 2: Ввести базы данных для каждого агента

Создать одну базу данных на агента и зарегистрировать её из глобальной БД:

```text
~/.openclaw/state/openclaw.sqlite
~/.openclaw/agents/<agentId>/agent/openclaw-agent.sqlite
```

Глобальная строка `agent_databases` хранит путь, версию схемы, отметку времени
последнего появления и базовые метаданные размера/целостности. Код среды выполнения запрашивает у реестра
БД агента вместо прямого вывода файловых путей.

БД агента владеет:

- `sessions` как канонический корень сессий, с `session_entries` как таблицей
  полезной нагрузки в форме совместимости, привязанной к этому корню, и
  `session_routes` как уникальным поиском активного `session_key`
- `conversations` и `session_conversations` как нормализованная идентичность
  маршрутизации провайдера, привязанная к сессиям
- `transcript_events`
- снимки транскриптов и контрольные точки Compaction. Готово для записей среды выполнения.
- `vfs_entries`
- `tool_artifacts` и артефакты запусков
- локальные для агента строки среды выполнения/кэша. Готово для кэшей в области worker.
- события родительского потока ACP
- события среды выполнения траектории, когда они не являются явными артефактами экспорта

### Фаза 3: заменить API хранилища сессий

Готово для среды выполнения. Поверхность хранилища сессий в файловой форме не является активным
контрактом среды выполнения:

- Среда выполнения больше не вызывает `loadSessionStore(storePath)` и не считает `storePath`
  идентичностью сессии.
- Операции среды выполнения со строками: `getSessionEntry`, `upsertSessionEntry`,
  `patchSessionEntry`, `deleteSessionEntry` и `listSessionEntries`.
- Помощники перезаписи всего хранилища, файловые писатели, тесты очередей, очистка алиасов и
  параметры удаления устаревших ключей удалены из среды выполнения.
- Устаревшие экспорты совместимости корневого пакета по-прежнему адаптируют канонические
  пути `sessions.json` к API строк SQLite.
- Разбор `sessions.json` остается только в коде миграции/импорта doctor и
  тестах doctor.
- Резервное чтение жизненного цикла среды выполнения читает заголовки транскриптов SQLite, а не первые
  строки JSONL.

Продолжайте удалять все, что повторно вводит параметры файловых блокировок,
словарь очистки/усечения как файлового обслуживания, идентичность пути хранилища или тесты,
единственная проверка которых — сохранение JSON.

### Фаза 4: перенести транскрипты, потоки ACP, траектории и VFS

Сделайте каждый поток данных агента нативным для базы данных:

- Записи добавления транскрипта проходят через одну транзакцию SQLite, которая обеспечивает
  заголовок сессии, проверяет идемпотентность сообщений, выбирает родительский хвост, вставляет
  в `transcript_events` и записывает запрашиваемые метаданные идентичности в
  `transcript_event_identities`. Готово для прямых добавлений сообщений транскрипта и
  обычных сохраненных добавлений `TranscriptSessionManager`; явные операции с ветками
  сохраняют свой явный выбор родителя и все еще записывают строки SQLite
  без вывода какого-либо файлового локатора.
- Журналы родительских потоков ACP становятся строками, а не файлами `.acp-stream.jsonl`. Готово.
- Настройка порождения ACP больше не сохраняет пути JSONL транскриптов. Готово.
- Захват траекторий среды выполнения записывает строки событий/артефакты напрямую. Явная
  команда поддержки/экспорта все еще может создавать артефакты JSONL для пакета поддержки как
  формат экспорта, но экспорт сессии не пересоздает JSONL сессии. Готово.
- Дисковые рабочие пространства остаются на диске, когда настроен дисковый режим.
- Черновые VFS и экспериментальный режим рабочего пространства только VFS используют базу данных агента.

Миграция однократно импортирует старые файлы JSONL, записывает счетчики/хэши в
`migration_runs` и удаляет импортированные файлы после проверок целостности.

### Фаза 5: резервное копирование, восстановление, Vacuum и проверка

Резервные копии остаются одним архивным файлом:

- Создать контрольную точку для каждой глобальной базы данных и базы данных агента.
- Снять снимок каждой БД с семантикой резервного копирования SQLite или `VACUUM INTO`.
- Архивировать компактные снимки БД, конфигурацию, внешние учетные данные и запрошенные
  экспорты рабочих пространств.
- Не включать сырые live-файлы `*.sqlite-wal` и `*.sqlite-shm`.
- Проверять, открывая каждый снимок БД и выполняя `PRAGMA integrity_check`.
  `openclaw backup create` выполняет эту проверку архива по умолчанию;
  `--no-verify` пропускает только проход проверки архива после записи, а не проверку
  целостности при создании снимка.
- Восстановление копирует снимки обратно в их целевые пути. Эта ветка сбрасывает
  не выпущенную раскладку SQLite до `user_version = 1`; будущие выпущенные изменения схемы
  смогут добавить явные миграции, когда они понадобятся.

### Фаза 6: среда выполнения Worker

Сохраняйте режим worker экспериментальным, пока внедряется разделение базы данных:

- Workers получают идентификатор агента, идентификатор запуска, режим файловой системы и идентичность реестра БД.
- Каждый worker открывает собственное соединение SQLite.
- Родитель сохраняет полномочия доставки в каналы, approvals, конфигурации и отмены.
- Начните с одного worker на активный запуск; добавляйте пул только после того, как жизненный цикл и владение
  соединениями БД станут стабильными.

### Фаза 7: удалить старый мир

Готово для управления сессиями среды выполнения. Старый мир разрешен только как явный
ввод doctor или вывод поддержки/экспорта:

- Нет записей `sessions.json`, JSONL транскриптов, JSON реестра песочницы, sidecar SQLite задач
  или sidecar SQLite состояния Plugin в среде выполнения.
- Нет очистки JSON/файлов сессий, усечения файлов транскриптов, файловых блокировок сессий
  или тестов сессий в форме блокировок.
- Нет экспортов совместимости среды выполнения, цель которых — поддерживать старые файлы сессий
  актуальными.
- Явные экспорты поддержки остаются запрашиваемыми пользователем форматами архива/материализации
  и не должны передавать имена файлов обратно в идентичность среды выполнения.

## Резервное копирование и восстановление

Резервные копии должны быть одним архивным файлом, но захват базы данных должен быть
нативным для SQLite:

1. Остановите долгие операции записи или войдите в короткий барьер резервного копирования.
2. Для каждой глобальной базы данных и базы данных агента выполните контрольную точку.
3. Снимите снимок каждой базы данных с семантикой резервного копирования SQLite или `VACUUM INTO` во
   временный каталог резервной копии.
4. Архивируйте компактные снимки баз данных, файл конфигурации, каталог учетных данных,
   выбранные рабочие пространства и манифест.
5. Проверьте архив, открыв каждый включенный снимок SQLite и выполнив
   `PRAGMA integrity_check`.
   `openclaw backup create` делает это по умолчанию; `--no-verify` предназначен только для
   намеренного пропуска прохода проверки архива после записи.

Не полагайтесь на сырые live-копии `*.sqlite`, `*.sqlite-wal` и `*.sqlite-shm` как
основной формат резервной копии. Манифест архива должен записывать роль базы данных,
идентификатор агента, версию схемы, исходный путь, путь снимка, размер в байтах и статус
целостности.

Восстановление должно заново создавать глобальную базу данных и файлы баз данных агентов из
снимков архива. Поскольку раскладка SQLite еще не была выпущена, этот рефакторинг
сохраняет только схему версии 1 плюс импорт из файлов в базу данных через doctor. Команда
восстановления сначала проверяет архив, затем заменяет каждый ресурс манифеста из
проверенной извлеченной полезной нагрузки.

## План рефакторинга среды выполнения

1. Добавить API реестра базы данных.
   - Разрешать пути глобальной БД и БД на каждого агента.
   - Сохранять не выпущенные схемы на `user_version = 1`; не добавлять код запуска
     миграций схемы, пока выпущенной схеме это не понадобится.
   - Добавить помощники закрытия/контрольной точки/целостности, используемые тестами, резервным копированием и doctor.

2. Свернуть sidecar-хранилища SQLite.
   - Перенести таблицы состояния Plugin в глобальную базу данных. Готово для записей среды выполнения;
     не выпущенный устаревший sidecar-импортер удален.
   - Перенести таблицы реестра задач в глобальную базу данных. Готово для записей среды выполнения;
     не выпущенный устаревший sidecar-импортер удален.
   - Перенести таблицы TaskFlow в глобальную базу данных. Готово для записей среды выполнения;
     не выпущенный устаревший sidecar-импортер удален.
   - Перенести встроенные таблицы поиска по памяти в базу данных каждого агента. Готово; явный
     пользовательский `memorySearch.store.path` теперь удаляется миграцией конфигурации doctor.
     Полная переиндексация выполняется на месте только по таблицам памяти; старый путь замены всего файла
     и помощник замены sidecar-индекса удалены.
   - Удалить дублирующиеся открыватели базы данных, настройку WAL, помощники разрешений и
     пути закрытия из этих подсистем.

3. Перенести таблицы, принадлежащие агенту, в базы данных на каждого агента.
   - Создавать БД агента по требованию через глобальный реестр базы данных. Готово.
   - Перенести записи сессий среды выполнения, события транскриптов, строки VFS и
     артефакты инструментов в БД агентов. Готово.
   - Не мигрировать локальные для ветки записи сессий общей БД, события транскриптов,
     строки VFS или артефакты инструментов; эта раскладка никогда не выпускалась. Сохранять только устаревший
     импорт из файлов в базу данных в doctor.

## Правила производительности

- Одно соединение на поток/процесс допустимо; не разделяйте handles между
  workers.
- Используйте WAL, `foreign_keys=ON`, 30-секундный busy timeout и короткие write-транзакции `BEGIN IMMEDIATE`.
- Оставляйте helper-функции write-транзакций синхронными, пока/если async transaction
  API не добавит явную семантику mutex/backpressure.
- Делайте записи parent delivery небольшими и транзакционными.
- Избегайте перезаписи всего хранилища; используйте upsert/delete на уровне строк.
- Добавьте индексы для путей list-by-agent, list-by-session, updated-at, run id и
  expiration перед переносом горячего кода.
- Храните крупные артефакты, медиа и векторы как BLOB или строки chunked BLOB, а не
  base64 или JSON с числовыми массивами.
- Держите непрозрачные записи plugin-state небольшими и ограниченными областью.
- Добавьте SQL cleanup для TTL/expiration вместо очистки файловой системы.
  Выполнено для runtime-хранилищ, которыми владеет база данных: media, plugin state, plugin blobs,
  persistent dedupe и agent cache теперь истекают через строки SQLite. Оставшаяся
  очистка файловой системы ограничена временными материализациями или явными
  командами удаления.

читать, импортировать и удалять устаревшие файловые источники. Невыпущенные SQLite-сайдкары остаются запрещенными
и не получают разрешений на импорт через `doctor`.

## Критерии готовности

- Записи runtime-данных и кэша идут в глобальную или агентскую базу данных SQLite.
- Runtime больше не записывает индексы сеансов, JSONL транскриптов, JSON реестра песочницы, сайдкар SQLite задач или сайдкар SQLite состояния Plugin. Импортеры невыпущенных сайдкаров SQLite для задач и состояния Plugin удалены.
- Импорт устаревших файлов выполняется только через `doctor`.
- Резервное копирование создает один архив с компактными снимками SQLite и доказательством целостности.
- Рабочие процессы агента могут запускаться с диском, черновым VFS или экспериментальным хранилищем только на VFS.
- Конфигурация и явные файлы учетных данных остаются единственными ожидаемыми постоянными управляющими файлами вне базы данных.
- Проверки репозитория предотвращают повторное введение устаревших runtime-файловых хранилищ.
