OpenClaw поддерживает три способа миграции: импорт из другой агентной системы, перенос существующей установки на новый компьютер и обновление плагина на месте.

## Импорт из другой агентной системы

Встроенные провайдеры миграции переносят в OpenClaw инструкции, серверы MCP, навыки, конфигурацию моделей и, при явном согласии, ключи API. Перед внесением изменений отображается предварительный план, секреты в отчётах скрываются, а применение изменений сопровождается созданием проверенной резервной копии.

Точка входа CLI — [`openclaw migrate`](/ru/cli/migrate). Первоначальная настройка также может предложить миграцию при обнаружении известного источника (`openclaw onboard --flow import`).

## Перенос OpenClaw на новый компьютер

Скопируйте **каталог состояния** (по умолчанию `~/.openclaw/`) и **рабочее пространство**, чтобы сохранить:

- **Конфигурацию** — `openclaw.json` и все настройки Gateway.
- **Аутентификацию** — отдельные для каждого агента `auth-profiles.json` (ключи API и OAuth), а также любое состояние каналов или провайдеров в `credentials/`.
- **Сеансы** — историю разговоров и состояние агентов.
- **Состояние каналов** — данные входа в WhatsApp, сеанс Telegram и аналогичные данные.
- **Файлы рабочего пространства** — `MEMORY.md`, `USER.md`, навыки и промпты.

<Tip>
Выполните `openclaw status` на старом компьютере, чтобы уточнить путь к каталогу состояния. Пользовательские профили используют `~/.openclaw-<profile>/` или путь, заданный через `OPENCLAW_STATE_DIR`.
</Tip>

### Этапы миграции

<Steps>
  <Step title="Остановите Gateway и создайте резервную копию">
    На **старом** компьютере остановите Gateway, чтобы файлы не изменялись во время копирования, затем создайте архив:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    Если вы используете несколько профилей (например, `~/.openclaw-work`), создайте отдельный архив для каждого из них.

  </Step>

  <Step title="Установите OpenClaw на новый компьютер">
    [Установите](/ru/install) CLI (и Node, если требуется) на новый компьютер. Ничего страшного, если первоначальная настройка создаст новый `~/.openclaw/` — на следующем этапе вы его перезапишете.
  </Step>

  <Step title="Скопируйте каталог состояния и рабочее пространство">
    Передайте архив через `scp`, `rsync -a` или на внешнем накопителе, затем распакуйте его:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    Убедитесь, что скрытые каталоги были включены, а владельцем файлов является пользователь, от имени которого будет запускаться Gateway.

  </Step>

  <Step title="Запустите Doctor и выполните проверку">
    На новом компьютере запустите [Doctor](/ru/gateway/doctor), чтобы применить миграции конфигурации и восстановить службы:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```

  </Step>
</Steps>

Если Telegram или Discord использует резервное значение из переменной среды по умолчанию (`TELEGRAM_BOT_TOKEN` или `DISCORD_BOT_TOKEN`), убедитесь, что файл `.env` в перенесённом каталоге состояния содержит эти ключи, не выводя значения секретов:

```bash
awk -F= '/^(TELEGRAM_BOT_TOKEN|DISCORD_BOT_TOKEN)=/ { print $1 "=present" }' ~/.openclaw/.env
```

`openclaw doctor` также выводит предупреждение, если у включённой учётной записи Telegram или Discord по умолчанию не настроен токен, а соответствующая переменная среды недоступна процессу Doctor.

### Распространённые ошибки

<AccordionGroup>
  <Accordion title="Несоответствие профиля или каталога состояния">
    Если старый Gateway использовал `--profile` или `OPENCLAW_STATE_DIR`, а новый их не использует, каналы будут отображаться как вышедшие из системы, а сеансы окажутся пустыми. Запустите Gateway с **тем же** профилем или каталогом состояния, который вы перенесли, а затем повторно выполните `openclaw doctor`.
  </Accordion>

  <Accordion title="Копирование только openclaw.json">
    Одного файла конфигурации недостаточно. Профили аутентификации моделей хранятся в `agents/<agentId>/agent/auth-profiles.json`, а состояние каналов и провайдеров — в `credentials/`. Всегда переносите **весь** каталог состояния.
  </Accordion>

  <Accordion title="Разрешения и владелец">
    Если вы выполняли копирование от имени root или сменили пользователя, Gateway может не получить доступ к учётным данным. Убедитесь, что каталог состояния и рабочее пространство принадлежат пользователю, от имени которого запускается Gateway.
  </Accordion>

  <Accordion title="Удалённый режим">
    Если ваш интерфейс подключён к **удалённому** Gateway, сеансы и рабочее пространство принадлежат удалённому узлу. Перенесите сам узел Gateway, а не локальный ноутбук. См. [часто задаваемые вопросы](/ru/help/faq#where-things-live-on-disk).
  </Accordion>

  <Accordion title="Секреты в резервных копиях">
    Каталог состояния содержит профили аутентификации, учётные данные каналов и другое состояние провайдеров. Храните резервные копии в зашифрованном виде, избегайте небезопасных каналов передачи и замените ключи, если подозреваете их раскрытие.
  </Accordion>
</AccordionGroup>

### Контрольный список проверки

На новом компьютере убедитесь, что:

- [ ] `openclaw status` показывает, что Gateway работает.
- [ ] Каналы по-прежнему подключены (повторное сопряжение не требуется).
- [ ] Панель управления открывается и показывает существующие сеансы.
- [ ] Файлы рабочего пространства (память, конфигурации) присутствуют.

## Обновление плагина на месте

При обновлении плагина на месте сохраняются его идентификатор и ключи конфигурации, но состояние на диске может быть перемещено в текущую структуру. Руководства по обновлению конкретных плагинов находятся рядом с документацией соответствующих каналов:

- [Миграция Matrix](/ru/channels/matrix-migration): ограничения восстановления зашифрованного состояния, поведение автоматических снимков состояния и команды ручного восстановления.
