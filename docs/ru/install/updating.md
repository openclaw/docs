Поддерживайте OpenClaw в актуальном состоянии.

Сведения о замене образов Docker, Podman и Kubernetes см. в разделе
[Обновление образов контейнеров](/ru/install/docker#upgrading-container-images). Перед переходом в состояние готовности
Gateway выполняет безопасные при запуске операции обновления и завершает работу, если
подключённое состояние требует ручного восстановления.

## Рекомендуемый способ: `openclaw update`

Определяет тип установки (npm или git), получает последнюю версию, запускает `openclaw doctor` и перезапускает Gateway.

```bash
openclaw update
```

Переключите канал или укажите определённую версию:

```bash
openclaw update --channel beta
openclaw update --channel extended-stable
openclaw update --channel dev
openclaw update --dry-run   # предварительный просмотр без применения
```

У `openclaw update` нет флага `--verbose` (он есть у установщика). Для диагностики используйте
`--dry-run`, чтобы предварительно просмотреть запланированные действия, `--json` для структурированных результатов или
`openclaw update status --json`, чтобы проверить канал и состояние доступности.

`--channel beta` предпочитает npm dist-tag `beta`, но возвращается к `stable`/`latest`,
если тег `beta` отсутствует или его версия старше последнего стабильного
выпуска. Для разового обновления пакета, закреплённого за исходным npm
dist-tag `beta`, используйте вместо этого `--tag beta`.

`--channel extended-stable` работает только с пакетами, а установка по-прежнему
выполняется только в основном процессе. OpenClaw считывает общедоступный селектор npm `extended-stable`,
проверяет выбранный точный пакет и устанавливает именно эту версию. При отсутствии
или несогласованности данных реестра операция безопасно завершается с ошибкой и никогда не возвращается к `latest`.
Если выбранная версия старше установленной, по-прежнему применяется обычное
подтверждение понижения версии. После успешного обновления ядра CLI сохраняет канал;
прямой вызов `npm install -g openclaw@extended-stable`
не обновляет `update.channel`.
После замены ядра подходящие официальные npm-плагины с пустым намерением, намерением по умолчанию или
`latest` синхронизируются с этой точной версией ядра. Точные закрепления и явно заданные
теги, отличные от `latest`, сторонние плагины и источники не из npm остаются без изменений.
Установки из каталога, созданные текущими версиями OpenClaw, сохраняют это намерение
по умолчанию. Более старые записи, содержащие только точную версию, остаются закреплёнными, поскольку
OpenClaw не может безопасно отличить старое автоматическое закрепление от пользовательского; один раз выполните
`openclaw plugins update @openclaw/name` на канале extended-stable,
чтобы снова включить для этого плагина отслеживание точной версии ядра.

`--channel dev` создаёт постоянную обновляемую рабочую копию ветки GitHub `main`. Для разового
обновления пакета `--tag main` сопоставляется со спецификацией пакета `github:openclaw/openclaw#main`
и устанавливает её напрямую через целевой менеджер пакетов (npm/pnpm/bun).

Для управляемых плагинов отсутствие бета-выпуска вызывает предупреждение, а не ошибку:
обновление ядра всё равно может завершиться успешно, а плагин вернётся к записанному
выпуску по умолчанию/`latest`.

Описание семантики каналов см. в разделе [Каналы выпусков](/ru/install/development-channels).

## Переключение между установками npm и git

Используйте каналы для изменения типа установки. Средство обновления сохраняет состояние, конфигурацию,
учётные данные и рабочую область в `~/.openclaw`; изменяется только установка кода OpenClaw,
которую используют CLI и Gateway.

```bash
# установка пакета npm -> редактируемая рабочая копия git
openclaw update --channel dev

# рабочая копия git -> установка пакета npm
openclaw update --channel stable
```

Сначала предварительно просмотрите переключение режима установки:

```bash
openclaw update --channel dev --dry-run
openclaw update --channel stable --dry-run
```

`dev` обеспечивает наличие рабочей копии git, собирает её и устанавливает глобальный CLI из этой
рабочей копии. Каналы `stable`, `extended-stable` и `beta` используют установку
пакетов. Канал extended-stable отклоняется для рабочей копии git без её изменения или
преобразования. Если Gateway уже установлен, `openclaw update` обновляет
метаданные службы и перезапускает её, если не передан флаг `--no-restart`.

Для пакетных установок с управляемой службой Gateway команда `openclaw update` использует
корень пакета этой службы. Если команда оболочки `openclaw`
относится к другой установке, средство обновления выводит оба корня и путь к Node
управляемой службы, а также проверяет соответствие этой версии Node требованию `engines.node`
целевого выпуска перед заменой пакета.

## Альтернативный способ: повторный запуск установщика

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
```

Добавьте `--no-onboard`, чтобы пропустить первоначальную настройку. Чтобы принудительно выбрать определённый тип установки, передайте
`--install-method git --no-onboard` или `--install-method npm --no-onboard`.

Если `openclaw update` завершится ошибкой после этапа установки пакета npm, повторно запустите
установщик. Он не вызывает средство обновления, а выполняет установку глобального пакета
напрямую и может восстановить частично обновлённую установку npm.

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method npm
```

Закрепите восстановление за определённой версией или dist-tag с помощью `--version`:

```bash
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --install-method npm --version <version-or-dist-tag>
```

## Альтернативный способ: ручное обновление через npm, pnpm или bun

```bash
npm i -g openclaw@latest
```

Для контролируемых установок предпочтительнее использовать `openclaw update`: эта команда может координировать замену
пакета с работающей службой Gateway. Если вы вручную обновляете контролируемую
установку, сначала остановите управляемый Gateway. Менеджеры пакетов заменяют файлы
на месте, и в противном случае работающий Gateway может попытаться загрузить файлы ядра или плагинов
в процессе замены. После завершения работы менеджера пакетов перезапустите Gateway, чтобы он использовал
новую установку.

Если для общесистемной установки Linux, принадлежащей пользователю root, команда `openclaw update` завершается
ошибкой `EACCES`, выполните восстановление с помощью системного npm, не запуская Gateway во время
ручной замены. Используйте те же флаги профиля и переменные окружения, которые обычно применяются для
этого Gateway. Замените `/usr/bin/npm` на системный npm, которому принадлежит
глобальный префикс root на вашем узле:

```bash
openclaw gateway stop
sudo /usr/bin/npm i -g openclaw@latest
openclaw gateway install --force
openclaw gateway restart
```

Затем выполните проверку:

```bash
openclaw --version
curl -fsS http://127.0.0.1:18789/readyz
openclaw plugins list --json
openclaw gateway status --deep --json
openclaw doctor --lint --json
```

Когда `openclaw update` управляет глобальной установкой npm, целевая версия сначала устанавливается
во временный префикс npm, затем проверяется состав упакованного каталога `dist`, после чего
чистое дерево пакета подменяет дерево в реальном глобальном префиксе — это предотвращает
наложение нового пакета npm на устаревшие файлы старого пакета. Если команда установки
завершается ошибкой, OpenClaw повторяет попытку один раз с флагом `--omit=optional`, что помогает на узлах,
где не удаётся скомпилировать необязательные нативные зависимости.

Команды обновления npm и плагинов, управляемые OpenClaw, также отключают для дочернего
процесса npm карантин цепочки поставок `min-release-age` (или более старый ключ конфигурации `before`).
Эта политика обеспечивает общую защиту, но явное обновление OpenClaw означает:
«установить выбранный выпуск сейчас».

```bash
pnpm add -g openclaw@latest
```

```bash
bun add -g openclaw@latest
```

### Расширенные темы установки npm

<AccordionGroup>
  <Accordion title="Дерево пакетов только для чтения">
    OpenClaw рассматривает упакованные глобальные установки во время выполнения как доступные только для чтения, даже если текущий пользователь может записывать данные в глобальный каталог пакета. Пакеты плагинов устанавливаются в принадлежащие OpenClaw корни npm/git в каталоге пользовательской конфигурации, а запуск Gateway не изменяет дерево пакета OpenClaw.

    В некоторых конфигурациях npm для Linux глобальные пакеты устанавливаются в принадлежащие пользователю root каталоги, например `/usr/lib/node_modules/openclaw`. OpenClaw поддерживает такую структуру, поскольку команды установки и обновления плагинов записывают данные за пределами этого глобального каталога пакета.

  </Accordion>
  <Accordion title="Усиленно защищённые модули systemd">
    Предоставьте OpenClaw доступ на запись к корням конфигурации и состояния, чтобы явные установки плагинов, обновления плагинов и очистка с помощью doctor могли сохранять изменения:

    ```ini
    ReadWritePaths=/var/lib/openclaw /home/openclaw/.openclaw /tmp
    ```

  </Accordion>
  <Accordion title="Предварительная проверка места на диске">
    Перед обновлением пакетов и явной установкой плагинов OpenClaw по возможности проверяет свободное место на целевом томе. При недостатке места выводится предупреждение с проверенным путём, но обновление не блокируется, поскольку квоты файловой системы, снимки и сетевые тома могут измениться после проверки. Определяющими остаются фактическая установка менеджером пакетов и проверка после установки.
  </Accordion>
</AccordionGroup>

## Автоматическое обновление

По умолчанию отключено. Включите его в `~/.openclaw/openclaw.json`:

```json5
{
  update: {
    channel: "stable",
    auto: {
      enabled: true,
      stableDelayHours: 6,
      stableJitterHours: 12,
      betaCheckIntervalHours: 1,
    },
  },
}
```

| Канал             | Поведение                                                                                                                                                               |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `stable`          | Ожидает `stableDelayHours` (по умолчанию: 6), затем применяет обновление с детерминированным разбросом в пределах `stableJitterHours` (по умолчанию: 12) для поэтапного развёртывания. |
| `extended-stable` | Проверяет наличие подсказки об обновлении только для чтения при запуске и каждые 24 часа, если включён `checkOnStart`. Никогда не применяет обновления автоматически.       |
| `beta`            | Проверяет каждые `betaCheckIntervalHours` (по умолчанию: 1) и применяет обновление немедленно.                                                                            |
| `dev`             | Автоматическое применение отсутствует. Используйте `openclaw update` вручную.                                                                                            |

Gateway также записывает в журнал подсказку об обновлении при запуске (отключается параметром
`update.checkOnStart: false`). Сохранённые варианты extended-stable используют этот
путь подсказок только для чтения и существующий 24-часовой интервал подсказок, но никогда не запускают
автоматическую установку, передачу управления, перезапуск, задержку/разброс stable или опрос beta.
Для понижения версии или восстановления после инцидента задайте `OPENCLAW_NO_AUTO_UPDATE=1` в окружении Gateway, чтобы заблокировать автоматическое применение, даже если настроен параметр `update.auto.enabled`. Подсказки об обновлении при запуске могут продолжать выполняться, если также не отключён `update.checkOnStart`.

Обновления через менеджер пакетов, запрошенные посредством действующей плоскости управления Gateway
(`update.run`), не заменяют дерево пакета внутри работающего процесса Gateway.
В установках с управляемой службой Gateway запускает отсоединённую передачу управления,
завершает работу и позволяет обычному пути CLI `openclaw update --yes --json` остановить
службу, заменить пакет, обновить метаданные службы, перезапустить её, проверить
версию и доступность Gateway, а также по возможности восстановить установленный, но не загруженный
LaunchAgent macOS. Если Gateway не может безопасно выполнить такую передачу управления,
`update.run` сообщает безопасную команду оболочки вместо запуска менеджера
пакетов внутри процесса.

Карточка обновления на боковой панели Control UI запускает тот же процесс `update.run`. В
подписанном приложении macOS карточка сначала обновляет приложение через Sparkle; после повторного запуска
приложение приводит управляемый локальный Gateway к соответствующей версии.

## После обновления

<Steps>

### Запустите doctor

```bash
openclaw doctor
```

Переносит конфигурацию, проверяет политики личных сообщений и работоспособность Gateway. Подробнее: [Doctor](/ru/gateway/doctor)

### Перезапустите Gateway

```bash
openclaw gateway restart
```

### Выполните проверку

```bash
openclaw health
```

</Steps>

## Откат

### Закрепление версии (npm)

```bash
npm i -g openclaw@<version>
openclaw doctor
openclaw gateway restart
```

<Tip>
`npm view openclaw version` показывает текущую опубликованную версию.
</Tip>

### Закрепление коммита (исходный код)

```bash
git fetch origin
git checkout "$(git rev-list -n 1 --before=\"2026-01-01\" origin/main)"
pnpm install && pnpm build
openclaw gateway restart
```

Чтобы вернуться к последней версии, выполните `git checkout main && git pull`.

## Если вы столкнулись с проблемой

- Снова выполните `openclaw doctor` и внимательно прочитайте вывод.
- Для `openclaw update --channel dev` в рабочих копиях исходного кода средство обновления при необходимости автоматически устанавливает `pnpm`. Если появляется ошибка первоначальной настройки pnpm/corepack, установите `pnpm` вручную (или снова включите `corepack`) и повторите обновление.
- См.: [Устранение неполадок](/ru/gateway/troubleshooting)
- Обратитесь за помощью в Discord: [https://discord.gg/clawd](https://discord.gg/clawd)
