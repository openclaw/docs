# Руководство по документации

Этот каталог содержит правила создания документации, правила ссылок Mintlify и политику интернационализации документации.

## Правила Mintlify

- Документация размещена на Mintlify (`https://docs.openclaw.ai`).
- Внутренние ссылки документации в `docs/**/*.md` должны оставаться относительными от корня и не иметь суффикса `.md` или `.mdx` (пример: `[Конфигурация](/gateway/configuration)`).
- Перекрёстные ссылки на разделы должны использовать якоря в путях, относительных от корня (пример: `[Хуки](/gateway/configuration-reference#hooks)`).
- В заголовках документации следует избегать длинных тире и апострофов, поскольку генерация якорей в Mintlify в таких случаях ненадёжна.
- В README и другой документации, отображаемой на GitHub, следует сохранять абсолютные URL-адреса документации, чтобы ссылки работали вне Mintlify.
- Содержимое документации должно оставаться универсальным: без личных имён устройств, имён хостов и локальных путей; используйте заполнители, например `user@gateway-host`.

## Правила содержимого документации

- В документации, тексте интерфейса и списках выбора располагайте сервисы и поставщиков в алфавитном порядке, если раздел явно не описывает порядок выполнения или порядок автоматического обнаружения.
- Соблюдайте единообразие названий встроенных плагинов в соответствии с общими для репозитория правилами терминологии плагинов в корневом файле `AGENTS.md`.
- Никогда не редактируйте созданную автоматически документацию вручную: `docs/plugins/reference/**`, `docs/plugins/reference.md` и `docs/plugins/plugin-inventory.md` создаются командой `pnpm plugins:inventory:gen`; `docs/docs_map.md` — командой `pnpm docs:map:gen`; `docs/maturity/**` — командой `pnpm maturity:render`.

## Внутренняя документация

- Долгосрочная закрытая документация для операторов должна находиться в `~/Projects/manager/docs/`.
- Локальные для репозитория внутренние черновики и зеркала документации могут находиться в игнорируемом каталоге `docs/internal/`.
- Никогда не добавляйте страницы `docs/internal/**` в навигацию `docs/docs.json` и не ссылайтесь на них из общедоступной документации.
- `scripts/docs-sync-publish.mjs` исключает и удаляет `docs/internal/**` из общедоступного репозитория публикации `openclaw/docs`, если позднее страница будет добавлена принудительно.
- Во внутренней документации можно упоминать пути репозитория, названия закрытых приложений, названия элементов 1Password и инструкции по эксплуатации, но нельзя включать значения секретов.

## Редактирование карты оценки зрелости

`taxonomy.yaml` и `qa/maturity-scores.yaml` являются исходными данными; создаваемая автоматически документация о зрелости в `docs/maturity/` представляет собой их проекцию, и её не следует редактировать вручную для изменения оценок, LTS, таксономии, профиля контроля качества или таблиц подтверждающих данных.
За генерацию отвечает `scripts/qa/render-maturity-docs.ts`; используйте `pnpm maturity:render`, чтобы обновить зафиксированную в репозитории документацию, и `pnpm maturity:check`, чтобы проверить её.
`.github/workflows/maturity-scorecard.yml` создаёт предварительные версии артефактов и может открывать запросы на включение изменений с автоматически созданной документацией; `.github/workflows/openclaw-release-checks.yml` запускает этот процесс для контроля качества выпуска.
Храните детерминированные данные `qa-evidence.json.scorecard` в артефактах GitHub Actions, если сопровождающий явно не запросил очищенную проекцию для фиксации в репозитории.
Ручные переопределения должны изменять исходное состояние в запросе на включение изменений и объяснять причину, а также содержать общедоступные или отредактированные подтверждающие данные.

## Интернационализация документации

- Документация на иностранных языках в этом репозитории не поддерживается. Созданные результаты публикации находятся в отдельном репозитории `openclaw/docs` (который локально часто клонируется как `../openclaw-docs`).
- Не добавляйте и не редактируйте здесь локализованную документацию в `docs/<locale>/**`.
- Считайте английскую документацию в этом репозитории и файлы глоссария источником истины.
- Конвейер: обновите здесь английскую документацию, при необходимости обновите `docs/.i18n/glossary.<locale>.json`, а затем дождитесь синхронизации репозитория публикации и запуска `scripts/docs-i18n` в `openclaw/docs`.
- Перед повторным запуском `scripts/docs-i18n` добавьте в глоссарий все новые технические термины, заголовки страниц и краткие метки навигации, которые должны оставаться на английском языке или иметь фиксированный перевод.
- `pnpm docs:check-i18n-glossary` служит проверкой изменённых заголовков английской документации и кратких меток внутренней документации.
- Память переводов хранится в автоматически созданных файлах `docs/.i18n/*.tm.jsonl` в репозитории публикации.
- См. `docs/.i18n/README.md`.
