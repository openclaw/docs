## Статус

Реализовано для поверхностей общего агента, CLI, возможностей плагинов и исходящей доставки:

- `ReplyPayload.presentation` содержит семантический интерфейс сообщения.
- `ReplyPayload.delivery.pin` содержит запросы на закрепление отправленного сообщения.
- Общие действия с сообщениями предоставляют `presentation`, `delivery` и `pin` вместо специфичных для провайдеров `components`, `blocks`, `buttons` или `card`.
- Ядро отображает представление или автоматически упрощает его с учётом объявленных плагином возможностей исходящей доставки.
- Средства отображения Discord, Slack, Telegram, Mattermost, MS Teams и Feishu используют общий контракт.
- Код плоскости управления каналом Discord больше не импортирует контейнеры интерфейса на основе Carbon.

Каноническая документация теперь находится в разделе [Представление сообщений](/ru/plugins/message-presentation).
Сохраните этот план как исторический контекст реализации; изменения контракта,
средств отображения или резервного поведения вносите в каноническое руководство.

## Проблема

В настоящее время интерфейс каналов разделён между несколькими несовместимыми поверхностями:

- Ядро владеет ориентированной на Discord межконтекстной точкой расширения средства отображения через `buildCrossContextComponents`.
- `channel.ts` Discord может импортировать нативный интерфейс Carbon через `DiscordUiContainer`, из-за чего зависимости интерфейса среды выполнения попадают в плоскость управления плагина канала.
- Агент и CLI предоставляют обходные пути для нативной полезной нагрузки, например `components` Discord, `blocks` Slack, `buttons` Telegram или Mattermost, а также `card` Teams или Feishu.
- `ReplyPayload.channelData` содержит как транспортные подсказки, так и нативные оболочки интерфейса.
- Общая модель `interactive` существует, но она уже, чем более развитые макеты, которые используются в Discord, Slack, Teams, Feishu, LINE, Telegram и Mattermost.

Из-за этого ядро знает о формах нативного интерфейса, ослабляется отложенная загрузка среды выполнения плагинов, а агенты получают слишком много специфичных для провайдеров способов выразить один и тот же замысел сообщения.

## Цели

- Ядро выбирает наилучшее семантическое представление сообщения на основе объявленных возможностей.
- Расширения объявляют возможности и преобразуют семантическое представление в нативную транспортную полезную нагрузку.
- Веб-интерфейс управления остаётся отделённым от нативного интерфейса чатов.
- Нативная полезная нагрузка каналов не предоставляется через общую поверхность сообщений агента или CLI.
- Неподдерживаемые возможности представления автоматически упрощаются до наилучшего текстового варианта.
- Поведение доставки, например закрепление отправленного сообщения, является общими метаданными доставки, а не представлением.

## Не является целью

- Отсутствует прослойка обратной совместимости для `buildCrossContextComponents`.
- Отсутствуют публичные обходные пути для нативных `components`, `blocks`, `buttons` или `card`.
- Ядро не импортирует библиотеки нативного интерфейса каналов.
- Отсутствуют специфичные для провайдеров точки расширения SDK для встроенных каналов.

## Целевая модель

Добавьте принадлежащее ядру поле `presentation` в `ReplyPayload`.

```ts
type MessagePresentationTone = "neutral" | "info" | "success" | "warning" | "danger";

type MessagePresentation = {
  tone?: MessagePresentationTone;
  title?: string;
  blocks: MessagePresentationBlock[];
};

type MessagePresentationBlock =
  | { type: "text"; text: string }
  | { type: "context"; text: string }
  | { type: "divider" }
  | { type: "buttons"; buttons: MessagePresentationButton[] }
  | { type: "select"; placeholder?: string; options: MessagePresentationOption[] };

type MessagePresentationButton = {
  label: string;
  value?: string;
  url?: string;
  style?: "primary" | "secondary" | "success" | "danger";
};

type MessagePresentationOption = {
  label: string;
  value: string;
};
```

Во время миграции `interactive` становится подмножеством `presentation`:

- Текстовый блок `interactive` сопоставляется с `presentation.blocks[].type = "text"`.
- Блок кнопок `interactive` сопоставляется с `presentation.blocks[].type = "buttons"`.
- Блок выбора `interactive` сопоставляется с `presentation.blocks[].type = "select"`.

Внешние схемы агента и CLI теперь используют `presentation`; `interactive` остаётся внутренним устаревшим вспомогательным средством разбора и отображения для существующих производителей ответов.
Публичный API для производителей считает `interactive` устаревшим. Поддержка
среды выполнения сохраняется, чтобы существующие вспомогательные средства подтверждения и старые плагины продолжали
работать, пока новый код формирует `presentation`.

## Метаданные доставки

Добавьте принадлежащее ядру поле `delivery` для поведения отправки, не относящегося к интерфейсу.

```ts
type ReplyPayloadDelivery = {
  pin?:
    | boolean
    | {
        enabled: boolean;
        notify?: boolean;
        required?: boolean;
      };
};
```

Семантика:

- `delivery.pin = true` означает закрепление первого успешно доставленного сообщения.
- По умолчанию `notify` имеет значение `false`.
- По умолчанию `required` имеет значение `false`; для неподдерживаемых каналов или при ошибке закрепления выполняется автоматическое упрощение с продолжением доставки.
- Ручные действия с сообщениями `pin`, `unpin` и `list-pins` сохраняются для существующих сообщений.

Текущую привязку темы ACP в Telegram следует перенести с `channelData.telegram.pin = true` на `delivery.pin = true`.

## Контракт возможностей среды выполнения

Добавьте точки расширения отображения представления и доставки в адаптер исходящей доставки среды выполнения, а не в плагин канала плоскости управления.

```ts
type ChannelPresentationCapabilities = {
  supported: boolean;
  buttons?: boolean;
  selects?: boolean;
  context?: boolean;
  divider?: boolean;
  tones?: MessagePresentationTone[];
  limits?: {
    actions?: {
      maxActions?: number;
      maxActionsPerRow?: number;
      maxRows?: number;
      maxLabelLength?: number;
      maxValueBytes?: number;
      supportsStyles?: boolean;
      supportsDisabled?: boolean;
      supportsLayoutHints?: boolean;
    };
    selects?: {
      maxOptions?: number;
      maxLabelLength?: number;
      maxValueBytes?: number;
    };
    text?: {
      maxLength?: number;
      encoding?: "characters" | "utf8-bytes" | "utf16-units";
      markdownDialect?: "plain" | "markdown" | "html" | "slack-mrkdwn" | "discord-markdown";
      supportsEdit?: boolean;
    };
  };
};

type ChannelDeliveryCapabilities = {
  pinSentMessage?: boolean;
};

type ChannelOutboundAdapter = {
  presentationCapabilities?: ChannelPresentationCapabilities;

  renderPresentation?: (params: {
    payload: ReplyPayload;
    presentation: MessagePresentation;
    ctx: ChannelOutboundSendContext;
  }) => ReplyPayload | null;

  deliveryCapabilities?: ChannelDeliveryCapabilities;

  pinDeliveredMessage?: (params: {
    cfg: OpenClawConfig;
    accountId?: string | null;
    to: string;
    threadId?: string | number | null;
    messageId: string;
    notify: boolean;
  }) => Promise<void>;
};
```

Поведение ядра:

- Определить целевой канал и адаптер среды выполнения.
- Запросить возможности представления.
- Упростить неподдерживаемые блоки и применить общие ограничения возможностей перед
  отображением.
- Вызвать `renderPresentation`.
- Если средство отображения отсутствует, преобразовать представление в резервный текстовый вариант.
- После успешной отправки вызвать `pinDeliveredMessage`, если запрошено и поддерживается `delivery.pin`.

## Сопоставление каналов

Discord:

- Преобразовывать `presentation` в компоненты v2 и контейнеры Carbon в модулях, используемых только средой выполнения.
- Сохранить вспомогательные средства цветов акцента в лёгких модулях.
- Удалить импорты `DiscordUiContainer` из кода плоскости управления плагина канала.

Slack:

- Преобразовывать `presentation` в Block Kit.
- Удалить ввод `blocks` из агента и CLI.

Telegram:

- Отображать текст, контекст и разделители как текст.
- Отображать действия и выбор как встроенные клавиатуры, если они настроены и разрешены для целевой поверхности.
- Использовать резервный текстовый вариант, если встроенные кнопки отключены.
- Перенести закрепление темы ACP в `delivery.pin`.

Mattermost:

- Отображать действия как интерактивные кнопки, если они настроены.
- Отображать другие блоки с помощью резервного текстового варианта.

MS Teams:

- Преобразовывать `presentation` в Adaptive Cards.
- Сохранить ручные действия закрепления, открепления и просмотра закреплённых сообщений.
- При необходимости реализовать `pinDeliveredMessage`, если поддержка Graph надёжна для целевого диалога.

Feishu:

- Преобразовывать `presentation` в интерактивные карточки.
- Сохранить ручные действия закрепления, открепления и просмотра закреплённых сообщений.
- При необходимости реализовать `pinDeliveredMessage` для закрепления отправленных сообщений, если поведение API надёжно.

LINE:

- По возможности преобразовывать `presentation` в сообщения Flex или шаблонные сообщения.
- Использовать резервный текстовый вариант для неподдерживаемых блоков.
- Удалить полезную нагрузку интерфейса LINE из `channelData`.

Простые каналы или каналы с ограниченными возможностями:

- Преобразовывать представление в текст с консервативным форматированием.

## Этапы рефакторинга

1. Повторно применить исправление выпуска Discord, которое отделяет `ui-colors.ts` от интерфейса на основе Carbon и удаляет `DiscordUiContainer` из `extensions/discord/src/channel.ts`.
2. Добавить `presentation` и `delivery` в `ReplyPayload`, нормализацию исходящей полезной нагрузки, сводки доставки и полезную нагрузку точек расширения.
3. Добавить схему `MessagePresentation` и вспомогательные средства разбора в узком подпути SDK или среды выполнения.
4. Заменить возможности сообщений `buttons`, `cards`, `components` и `blocks` семантическими возможностями представления.
5. Добавить в адаптер исходящей доставки среды выполнения точки расширения для отображения представления и закрепления при доставке.
6. Заменить создание межконтекстных компонентов на `buildCrossContextPresentation`.
7. Удалить `src/infra/outbound/channel-adapters.ts` и убрать `buildCrossContextComponents` из типов плагинов каналов.
8. Изменить `maybeApplyCrossContextMarker`, чтобы он добавлял `presentation` вместо нативных параметров.
9. Обновить пути отправки диспетчеризации плагинов, чтобы они использовали только семантическое представление и метаданные доставки.
10. Удалить из агента и CLI параметры нативной полезной нагрузки: `components`, `blocks`, `buttons` и `card`.
11. Удалить вспомогательные средства SDK, создающие нативные схемы инструментов сообщений, и заменить их вспомогательными средствами схемы представления.
12. Удалить интерфейсные и нативные оболочки из `channelData`; сохранять только транспортные метаданные, пока не будет проверено каждое оставшееся поле.
13. Мигрировать средства отображения Discord, Slack, Telegram, Mattermost, MS Teams, Feishu и LINE.
14. Обновить документацию по CLI сообщений, страницам каналов, SDK плагинов и справочнику возможностей.
15. Выполнить профилирование разветвления импортов для Discord и затронутых точек входа каналов.

Этапы 1–11 и 13–14 реализованы в этом рефакторинге для контрактов общего агента, CLI, возможностей плагинов и адаптера исходящей доставки. Этап 12 остаётся более глубокой внутренней очисткой специфичных для провайдера транспортных оболочек `channelData`. Этап 15 остаётся последующей проверкой, если помимо проверки типов и тестов потребуются количественные показатели разветвления импортов.
