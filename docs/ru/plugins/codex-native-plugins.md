Поддержка нативных плагинов Codex позволяет агенту OpenClaw в режиме Codex использовать собственные возможности приложений и плагинов app-server Codex в том же потоке Codex, который обрабатывает ход OpenClaw. Вызовы плагинов остаются в нативной расшифровке Codex; app-server Codex управляет выполнением MCP через приложения. OpenClaw не преобразует плагины Codex в синтетические динамические инструменты OpenClaw `codex_plugin_*`.

Используйте эту страницу после того, как базовая [среда выполнения Codex](/ru/plugins/codex-harness) заработает.

## Требования

- Средой выполнения агента должна быть нативная среда выполнения Codex.
- `plugins.entries.codex.enabled` должно иметь значение `true`.
- `plugins.entries.codex.config.codexPlugins.enabled` должно иметь значение `true`.
- Целевой app-server Codex должен видеть ожидаемый набор маркетплейсов, плагинов и приложений.
- Миграция поддерживает только плагины `openai-curated`, которые были установлены из исходного кода в исходном домашнем каталоге Codex.
- Для вручную настроенных плагинов `workspace-directory` требуется app-server Codex, в котором `plugin/list` принимает `marketplaceKinds`, а сводки рабочих пространств без пути содержат `remotePluginId`. Плагин должен быть уже установлен и включён, а принадлежащие ему приложения должны быть доступны через `app/list`.

`codexPlugins` не влияет на запуски через провайдер OpenClaw, привязки диалогов ACP или другие среды выполнения, поскольку эти пути никогда не создают потоки app-server Codex с нативной конфигурацией `apps`.

Учётная запись Codex на стороне OpenAI, доступность приложений и управление приложениями и плагинами рабочего пространства определяются учётной записью Codex, в которую выполнен вход. Модель учётных записей и администрирования OpenAI описана в статье [Использование Codex с вашим тарифным планом ChatGPT](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan).

## Быстрый старт

Предварительно просмотрите миграцию из исходного домашнего каталога Codex:

```bash
openclaw migrate codex --dry-run
```

Добавьте `--verify-plugin-apps`, чтобы при миграции вызывался исходный `app/list`, а перед планированием нативной активации проверялось наличие, включённость и доступность каждого принадлежащего плагину приложения:

```bash
openclaw migrate codex --dry-run --verify-plugin-apps
```

Примените миграцию, когда план будет выглядеть правильно:

```bash
openclaw migrate apply codex --yes
```

Миграция записывает явные элементы `codexPlugins` для подходящих плагинов и вызывает `plugin/install` app-server Codex для выбранных плагинов. Мигрированная конфигурация выглядит так:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

Миграция по-прежнему ограничена `openai-curated`. Чтобы использовать существующий плагин `workspace-directory`, добавьте его вручную, указав точный идентификатор `summary.id` с квалификатором маркетплейса, возвращённый `plugin/list`. Например, если Codex возвращает `example-plugin@workspace-directory`, укажите это полное значение, а не его отображаемое имя:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            plugins: {
              "example-plugin": {
                enabled: true,
                marketplaceName: "workspace-directory",
                pluginName: "example-plugin@workspace-directory",
              },
            },
          },
        },
      },
    },
  },
}
```

OpenClaw не вызывает `plugin/install` и не запускает аутентификацию для плагина `workspace-directory`. Установите, включите и аутентифицируйте его в Codex, прежде чем добавлять или включать политику OpenClaw. OpenClaw не показывает приложения, если в ответе отсутствуют точные сведения о маркетплейсе, идентификаторе плагина, идентификаторе подробной записи или готовности приложения. Если Codex отклоняет явный запрос `plugin/list` для рабочего пространства, OpenClaw сообщает `marketplace_missing` для каждого включённого плагина рабочего пространства и сохраняет доступность независимо обнаруженных курируемых плагинов.

После изменения `codexPlugins` новые диалоги Codex автоматически получают обновлённый набор приложений. Выполните `/new` или `/reset`, чтобы обновить текущий диалог. Для изменений включения или отключения плагинов перезапуск Gateway не требуется.

## Управление плагинами из чата

`/codex plugins` позволяет просматривать или изменять настроенные нативные плагины Codex из того же чата, в котором вы работаете со средой выполнения Codex:

```text
/codex plugins
/codex plugins list
/codex plugins disable google-calendar
/codex plugins enable google-calendar
```

`/codex plugins` является псевдонимом для `/codex plugins list`. В списке для каждого настроенного плагина отображаются ключ, состояние включения, имя плагина Codex и маркетплейс из `plugins.entries.codex.config.codexPlugins.plugins`.

`enable`/`disable` записывают изменения только в `~/.openclaw/openclaw.json`; они никогда не изменяют `~/.codex/config.toml` и не устанавливают новые плагины Codex. Выполнять эти команды может только владелец или клиент Gateway с областью доступа `operator.admin`.

При включении настроенного плагина также включается глобальный переключатель `codexPlugins.enabled`. Если курируемый плагин был записан отключённым из-за того, что миграция вернула `auth_required`, повторно авторизуйте приложение в Codex перед его включением в OpenClaw. Для элемента `workspace-directory` включение здесь изменяет только политику OpenClaw; плагин и приложение уже должны быть активны в Codex.

## Как работает нативная настройка плагинов

Интеграция отслеживает три состояния:

| Состояние    | Значение                                                                                                                                              |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| Установлен   | Codex содержит комплект плагина в среде выполнения целевого app-server.                                                                               |
| Включён      | Codex сообщает, что плагин включён, а конфигурация OpenClaw разрешает его для ходов среды выполнения Codex.                                            |
| Доступен     | App-server Codex подтверждает, что элементы приложения плагина доступны активной учётной записи и соответствуют настроенному идентификатору плагина.   |

Для плагинов `openai-curated` миграция является постоянным этапом установки и проверки соответствия требованиям:

- Во время планирования OpenClaw считывает исходные сведения Codex из `plugin/read` и проверяет, что учётная запись исходного app-server Codex является учётной записью с подпиской ChatGPT. Если учётная запись отсутствует или не является учётной записью ChatGPT, плагины с поддержкой приложений пропускаются с кодом `codex_subscription_required`.
- По умолчанию миграция пропускает исходный вызов `app/list`: исходные плагины с поддержкой приложений, прошедшие проверку учётной записи, планируются без проверки доступности исходных приложений, а при транспортных ошибках получения учётной записи плагины пропускаются с кодом `codex_account_unavailable`.
- С параметром `--verify-plugin-apps` миграция получает свежий снимок исходного `app/list` и перед планированием нативной активации требует, чтобы каждое принадлежащее плагину приложение присутствовало, было включено и доступно. В этом случае транспортные ошибки получения учётной записи передаются на этап проверки исходного набора приложений вместо немедленного пропуска.

Для плагинов `workspace-directory` настройка выполняется вне OpenClaw. OpenClaw запрашивает этот маркетплейс, только если настроен хотя бы один включённый элемент рабочего пространства, разрешает каждый плагин по точному `summary.id` и повторно использует существующие проверки принадлежности через `plugin/read` и готовности через `app/list`. Неустановленный, отключённый, недоступный или неаутентифицированный плагин не предоставляет приложений; OpenClaw не пытается выполнить установку или аутентификацию.

Набор приложений среды выполнения служит проверкой доступности целевого сеанса как для мигрированных курируемых плагинов, так и для настроенных вручную плагинов рабочего пространства. При настройке сеанса среды выполнения Codex вычисляется ограничительная конфигурация приложений потока на основе включённых и доступных приложений плагинов; она не пересчитывается при каждом ходе, поэтому `/codex plugins enable`/`disable` влияют только на новые диалоги Codex. Используйте `/new` или `/reset`, чтобы применить изменение к текущему диалогу.

## Границы поддержки V1

- Для миграции подходят только плагины `openai-curated`, уже установленные в исходном наборе app-server Codex.
- Среда выполнения также поддерживает явные элементы `workspace-directory` в сборках app-server, где `plugin/list` реализует `marketplaceKinds` и возвращает `remotePluginId` для сводок рабочего пространства без пути. Эти элементы должны использовать точный `summary.id` с квалификатором маркетплейса и уже быть установленными, включёнными и доступными приложениям. Отклонённый запрос списка рабочего пространства приводит к существующей диагностике `marketplace_missing` для каждого плагина; при отсутствии сведений о маркетплейсе, плагине, подробной записи или приложении приложение рабочего пространства не предоставляется. Курируемый набор из запроса списка по умолчанию остаётся доступным.
- Исходные плагины с поддержкой приложений должны пройти проверку подписки во время миграции. `--verify-plugin-apps` добавляет проверку исходного набора приложений. Учётные записи, не прошедшие проверку подписки, а в режиме проверки также недоступные, отключённые или отсутствующие исходные приложения и ошибки обновления набора приложений, отображаются как пропущенные элементы, требующие ручной обработки, а не как включённые элементы конфигурации. Плагины с нечитаемыми подробными сведениями пропускаются до проверки набора приложений.
- Миграция записывает явные идентификаторы плагинов (`marketplaceName` и `pluginName`); локальные пути кеша `marketplacePath` не записываются.
- `codexPlugins.enabled` — единственный глобальный переключатель включения; подстановочного знака `plugins["*"]` или ключа конфигурации, предоставляющего произвольные полномочия на установку, не существует.
- Некурируемые маркетплейсы, кешированные комплекты плагинов, перехватчики и файлы конфигурации Codex сохраняются в отчёте о миграции для ручной проверки, но не активируются автоматически. Среда выполнения принимает вручную настроенные элементы `workspace-directory`; другие маркетплейсы не поддерживаются.

## Набор приложений и принадлежность

OpenClaw считывает набор приложений Codex через `app/list`, кеширует его в памяти на один час и асинхронно обновляет устаревшие или отсутствующие записи. Кеш является локальным для процесса; перезапуск CLI или Gateway очищает его, после чего OpenClaw заново формирует его при следующем чтении `app/list`.

Миграция и среда выполнения используют разные ключи кеша:

- При проверке исходной миграции используются исходный домашний каталог Codex и параметры запуска. Она выполняется только с `--verify-plugin-apps` и принудительно осуществляет новый обход исходного `app/list` для этого запуска планирования.
- При настройке целевой среды выполнения используется идентификатор app-server Codex целевого агента для построения конфигурации приложений потока. Активация курируемого плагина делает недействительным соответствующий ключ целевого кеша, а затем принудительно обновляет его после `plugin/install`. Настройка `workspace-directory` никогда не запускает этот путь активации.

Приложение плагина предоставляется только тогда, когда OpenClaw может сопоставить его с настроенным плагином по устойчивым данным о принадлежности: точному идентификатору приложения из подробных сведений о плагине, известному имени сервера MCP или уникальным стабильным метаданным. Принадлежность, основанная только на отображаемом имени или неоднозначных данных, исключается до тех пор, пока следующее обновление набора не подтвердит её.

## Приложения подключённой учётной записи

Агенты, управляемые владельцем, могут разрешить все приложения, уже подключённые к их учётной записи Codex, без требования соответствующего пакета плагина:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_all_plugins: true,
            allow_destructive_actions: "auto",
          },
        },
      },
    },
  },
}
```

`allow_all_plugins: true` получает полный снимок `app/list` при создании нового нативного потока Codex и допускает только приложения, отмеченные как доступные для этой учётной записи. Этот параметр не устанавливает, не аутентифицирует и не включает приложения глобально. Существующие потоки сохраняют ранее записанный набор приложений; используйте `/new`, `/reset` или перезапустите Gateway, чтобы применить изменения после подключения или отзыва приложений.

Приложения учётной записи наследуют глобальное значение `codexPlugins.allow_destructive_actions`, которое принимает `true`, `false`, `"auto"` или `"ask"`. Явная политика отдельного плагина переопределяет глобальную политику для пересекающихся идентификаторов приложений. При ошибках получения набора доступ блокируется, а не заменяется неограниченным значением по умолчанию.

## Конфигурация приложений потока

OpenClaw внедряет ограничивающий патч `config.apps` для потока Codex:
`_default` отключён, и включены только приложения, принадлежащие включённым настроенным Plugin, или доступные приложения учётной записи, разрешённые параметром `allow_all_plugins`.

Значение `destructive_enabled` для каждого приложения определяется действующей глобальной или заданной для конкретного Plugin политикой `allow_destructive_actions`; значения `true`, `"auto"` и `"ask"` устанавливают `destructive_enabled: true`, а `false` устанавливает его в `false`. Codex по-прежнему применяет метаданные деструктивных инструментов из собственных аннотаций инструментов приложений.
`_default` отключён с `open_world_enabled: false`; включённые приложения Plugin получают `open_world_enabled: true`. OpenClaw не предоставляет отдельного параметра политики открытого мира на уровне Plugin и не ведёт для отдельных Plugin списки запрещённых имён деструктивных инструментов.

Для разрешённых приложений режим подтверждения инструментов по умолчанию автоматический, поэтому недеструктивные инструменты чтения выполняются без запроса подтверждения в том же потоке. Деструктивные инструменты остаются под управлением политики `destructive_enabled` каждого приложения.

## Политика деструктивных действий

Для настроенных Plugin Codex запросы на деструктивные действия Plugin разрешены по умолчанию, тогда как небезопасные схемы и неоднозначная принадлежность приводят к безопасному отказу:

- Глобальное значение `allow_destructive_actions` по умолчанию равно `true`.
- Значение `allow_destructive_actions` для отдельного Plugin переопределяет глобальную политику для этого Plugin.
- `false`: OpenClaw возвращает детерминированный отказ.
- `true`: OpenClaw автоматически принимает только безопасные схемы, которые можно преобразовать в ответ подтверждения, например поле подтверждения с логическим значением.
- `"auto"`: OpenClaw предоставляет Codex деструктивные действия Plugin, а затем преобразует запросы подтверждения MCP с доказанной принадлежностью в подтверждения Plugin OpenClaw, прежде чем вернуть ответ подтверждения Codex.
- `"ask"`: OpenClaw применяет ту же блокировку операций записи и деструктивных действий Codex, что и для `"auto"`, перед запуском потока очищает постоянные переопределения подтверждений Codex для отдельных инструментов приложения и предлагает только однократное подтверждение или отказ, чтобы постоянные подтверждения не могли подавлять последующие запросы для действий записи. Для каждого разрешённого приложения, использующего `"ask"`, OpenClaw выбирает рецензента подтверждений с участием человека в Codex, чтобы Codex направлял запросы подтверждения этого приложения в OpenClaw; другие приложения и подтверждения потока, не относящиеся к приложениям, сохраняют настроенного рецензента и политику.
- Отсутствующий идентификатор Plugin, неоднозначная принадлежность, отсутствующий или несовпадающий идентификатор хода либо небезопасная схема запроса приводят к отказу без запроса подтверждения.

## Устранение неполадок

| Код                                               | Значение                                                                                                                                                 | Исправление                                                                                                                               |
| ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `auth_required`                                   | Миграция установила Plugin, но одному из его приложений по-прежнему требуется аутентификация. Запись остаётся отключённой до повторной авторизации.       | Повторно авторизуйте приложение в Codex, затем включите Plugin в OpenClaw.                                                                |
| `app_inaccessible`, `app_disabled`, `app_missing` | При использовании `--verify-plugin-apps` исходный реестр приложений Codex не показал все принадлежащие Plugin приложения как присутствующие, включённые и доступные. | Повторно авторизуйте или включите приложение в Codex, затем повторите миграцию с `--verify-plugin-apps`.                                   |
| `app_inventory_unavailable`                       | Была запрошена строгая проверка исходных приложений, но обновить исходный реестр приложений Codex не удалось.                                             | Исправьте доступ к исходному серверу приложений Codex или повторите попытку без `--verify-plugin-apps`, чтобы принять более быстрый план с проверкой учётной записи. |
| `codex_subscription_required`                     | Учётная запись исходного сервера приложений Codex не являлась учётной записью с подпиской ChatGPT.                                                        | Войдите в приложение Codex с помощью аутентификации по подписке, затем повторите миграцию.                                                 |
| `codex_account_unavailable`                       | Не удалось прочитать учётную запись исходного сервера приложений Codex.                                                                                  | Исправьте аутентификацию исходного сервера приложений Codex или повторите запуск с `--verify-plugin-apps`, чтобы пригодность определялась исходным реестром приложений. |
| `marketplace_missing`, `plugin_missing`           | Маркетплейс или конкретный Plugin недоступен; явный запрос каталога рабочего пространства мог быть отклонён; для приложений рабочего пространства применяется безопасный отказ. | Проверьте описанные ниже совместимый контракт сервера приложений и точный идентификатор.                                                   |
| `plugin_detail_unavailable`                       | OpenClaw не удалось прочитать сведения о принадлежности Plugin.                                                                                          | Проверьте ответы `plugin/list` и `plugin/read` целевого сервера приложений.                                                               |
| `plugin_disabled`                                 | Codex сообщает, что Plugin установлен, но отключён.                                                                                                      | Активация из курируемого каталога может исправить проблему; перед повторной попыткой включите Plugin рабочего пространства в Codex.       |
| `plugin_activation_failed`                        | Не удалось завершить активацию Plugin.                                                                                                                   | Используйте приложенную диагностику, чтобы различить сбои маркетплейса, аутентификации, обновления или готовности рабочего пространства.   |
| `app_inventory_missing`, `app_inventory_stale`    | Состояние готовности приложения получено из пустого или устаревшего кэша.                                                                                | OpenClaw автоматически планирует асинхронное обновление; приложения Plugin остаются исключёнными, пока не будут установлены их принадлежность и готовность. |
| `app_ownership_ambiguous`                         | Приложение в реестре совпало только по отображаемому имени.                                                                                              | Приложение остаётся скрытым от потока Codex, пока последующее обновление не подтвердит его принадлежность.                                |

**Plugin рабочего пространства установлен, но не отображается:** убедитесь, что результат `plugin/list` для рабочего пространства сообщает точный настроенный идентификатор как установленный и включённый, а затем убедитесь, что `app/list` сообщает о доступности каждого принадлежащего Plugin приложения для той же учётной записи Codex. OpenClaw может включить доступное приложение для потока, даже если реестр учётной записи сейчас сообщает, что оно отключено. Если вы изменили это состояние после того, как Gateway закэшировал реестр приложений, дождитесь обновления кэша через час или перезапустите Gateway, затем используйте `/new` или `/reset`. OpenClaw не восстанавливает и не аутентифицирует Plugin рабочего пространства.
Если явный запрос списка рабочего пространства отклонён, каждая включённая запись рабочего пространства сообщает `marketplace_missing`; обработка несвязанных записей курируемого каталога продолжается на основе ответа списка по умолчанию.

В случае `plugin_detail_unavailable` сводка рабочего пространства без пути должна содержать `remotePluginId`; OpenClaw оставляет принадлежащие Plugin приложения скрытыми, если этот селектор или последующий результат `plugin/read` недоступен. В случае `plugin_activation_failed` Plugin из курируемого каталога могут сообщать о сбое маркетплейса, аутентификации или обновления после установки. Plugin рабочего пространства сообщает этот код, если он ещё не активен; установите, включите и аутентифицируйте его вне OpenClaw.

**Конфигурация изменена, но агент не видит Plugin:** выполните `/codex plugins
list`, чтобы проверить настроенное состояние, затем `/new` или `/reset`. Существующие привязки потоков Codex сохраняют конфигурацию приложений, с которой они были запущены, пока OpenClaw не создаст новый сеанс среды выполнения или не заменит устаревшую привязку.

**Деструктивное действие отклонено:** проверьте глобальное и заданное для конкретного Plugin значения `allow_destructive_actions`. Даже при `true`, `"auto"` или `"ask"` небезопасные схемы запросов и неоднозначный идентификатор Plugin по-прежнему приводят к безопасному отказу.
