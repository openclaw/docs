Плагины каналов предоставляют поведение исходящих сообщений через
`openclaw/plugin-sdk/channel-outbound`. Используйте
`openclaw/plugin-sdk/channel-inbound` для оркестрации
получения, контекста и диспетчеризации.

Ядро отвечает за очереди, надёжное хранение, общую политику повторных попыток, хуки, квитанции и
общий инструмент `message`. Plugin отвечает за нативные вызовы отправки, редактирования и удаления,
нормализацию адресата, ветки обсуждений платформы, выбранные цитаты, флаги
уведомлений, состояние учётной записи и побочные эффекты, специфичные для платформы.

## Адаптер

Большинство плагинов определяют один адаптер `message`:

```ts
import {
  defineChannelMessageAdapter,
  createMessageReceiptFromOutboundResults,
} from "openclaw/plugin-sdk/channel-outbound";

export const demoMessageAdapter = defineChannelMessageAdapter({
  id: "demo",
  durableFinal: {
    capabilities: {
      text: true,
      replyTo: true,
      thread: true,
      messageSendingHooks: true,
    },
  },
  send: {
    text: async ({ cfg, to, text, accountId, replyToId, threadId, signal }) => {
      const sent = await sendDemoMessage({
        cfg,
        to,
        text,
        accountId: accountId ?? undefined,
        replyToId: replyToId ?? undefined,
        threadId: threadId == null ? undefined : String(threadId),
        signal,
      });

      return {
        receipt: createMessageReceiptFromOutboundResults({
          results: [{ channel: "demo", messageId: sent.id, conversationId: to }],
          kind: "text",
          threadId: threadId == null ? undefined : String(threadId),
          replyToId: replyToId ?? undefined,
        }),
      };
    },
  },
});
```

Объявляйте только те возможности, которые нативный транспорт действительно сохраняет. Покройте
каждую объявленную возможность отправки, квитанций, предварительного просмотра в реальном времени и подтверждения получения
контрактными вспомогательными функциями, экспортируемыми из этого подпути.

## Очистка обычного текста

Используйте `sanitizeForPlainText(...)`, когда адаптеру исходящих сообщений требуется преобразовать
поддерживаемые теги форматирования HTML в облегчённую текстовую разметку. По умолчанию сохраняются
существующие маркеры жирного и зачёркнутого текста в стиле чата. Передавайте
`{ style: "markdown" }` только тогда, когда канал повторно анализирует результат как Markdown:

```ts
import { sanitizeForPlainText } from "openclaw/plugin-sdk/channel-outbound";

const chatText = sanitizeForPlainText(text);
const markdownText = sanitizeForPlainText(text, { style: "markdown" });
```

Стиль Markdown использует `**bold**` и `~~strikethrough~~`; курсив и встроенный
код сохраняют маркеры `_italic_` и обратные кавычки в обоих стилях. Выбирайте стиль на
границе канала, а не переписывайте текст маркеров после очистки.

## Подтверждение доставки

`MessageReceipt` фиксирует результат, возвращённый адаптером канала. Конкретные
идентификаторы сообщений платформы показывают, что путь отправки платформы принял
сообщение; они не доказывают, что устройство получателя отобразило или прочитало его.
Квитанции без идентификаторов сообщений платформы являются лишь локальными метаданными квитанции.
Каналы с уведомлениями о прочтении или состоянием доставки на устройство должны отслеживать эти факты
через отдельный путь, специфичный для канала.

Если адаптер канала может доказать, что повторная попытка после сбоя не способна продублировать
видимую получателю отправку и ни один вызов, способный выполнить финализацию, не начался, выбросьте
`new PlatformMessageNotDispatchedError("...", { cause: error })` из
`openclaw/plugin-sdk/error-runtime`. Тогда ядро сможет удалить устаревшие данные о попытке
отправки и безопасно повторить поставленное в очередь намерение. Только адаптер, которому принадлежит
граница окончательной отправки, вправе делать такое утверждение. Никогда не используйте этот маркер после
начала вызова финализации или отправки либо после получения им неоднозначного результата; ошибочная маркировка может
привести к дублированию сообщений.

## Существующие адаптеры исходящих сообщений

Если у канала уже есть совместимый адаптер `outbound`, создайте на его основе
адаптер сообщений вместо дублирования кода отправки:

```ts
import { createChannelMessageAdapterFromOutbound } from "openclaw/plugin-sdk/channel-outbound";

export const messageAdapter = createChannelMessageAdapterFromOutbound({
  id: "demo",
  outbound,
  durableFinal: {
    capabilities: {
      text: true,
      media: true,
    },
  },
});
```

## Надёжные отправки

Вспомогательные функции отправки среды выполнения также находятся в `channel-outbound`:

- `sendDurableMessageBatch(...)`
- `withDurableMessageSendContext(...)`
- `deliverInboundReplyWithMessageSendContext(...)`
- вспомогательные функции потоковой передачи черновика и прогресса, например `resolveChannelDraftStreamingChunking(...)`

`sendDurableMessageBatch(...)` возвращает один явно определённый результат:

| Результат        | Значение                                                                                |
| ---------------- | --------------------------------------------------------------------------------------- |
| `sent`           | путь отправки платформы принял хотя бы одно видимое сообщение платформы                 |
| `suppressed`     | ни одно сообщение платформы не следует считать отсутствующим                            |
| `partial_failed` | хотя бы одно сообщение платформы было принято до сбоя последующей полезной нагрузки или побочного эффекта |
| `failed`         | квитанция платформы не была создана                                                      |

Используйте `payloadOutcomes`, когда пакет содержит одновременно отправленные, подавленные и неудачные
полезные нагрузки. Не делайте вывод об отмене хуком из пустого устаревшего
результата прямой доставки.

## Допуск отложенной доставки

Используйте `message.durableFinal.admitDeferredDelivery(...)`, когда определённая учётная запись
не может безопасно принимать управляемую ядром исходящую или отложенную доставку. Ядро вызывает
этот хук синхронно перед выполнением исходящей операции в реальном времени, включая пути, которые пропускают
сохранение в очереди, а также повторно перед воспроизведением восстановленного намерения. Контекст
включает `cfg`, `channel`, `to`, `accountId` и `phase` со значением `live` или
`recovery`.

Верните `{ status: "allowed" }`, чтобы продолжить. Верните
`{ status: "permanent_rejection", reason }`, если доставку нельзя
сохранять, отправлять напрямую или воспроизводить повторно. Отклонение в реальном времени приводит к сбою до создания
очереди, вызова хуков сообщений или выполнения операций платформы. Отклонение при восстановлении помечает
запись в очереди как неудачную и пропускает согласование и повторное воспроизведение. Отсутствие хука
означает разрешение.

Хук представляет собой синхронное решение о допуске, а не путь отправки. Читайте только
уже загруженную конфигурацию или состояние среды выполнения; не выполняйте сетевой, файловый или
другой асинхронный ввод-вывод. Контрактные тесты должны проверять обе фазы и оба
варианта результата через `ChannelMessageDurableFinalAdapter` из
`openclaw/plugin-sdk/channel-outbound`.

## Диспетчеризация совместимости

Формируйте диспетчеризацию входящих ответов через `dispatchChannelInboundReply(...)`
из `channel-inbound`. Оставляйте доставку платформы в адаптере доставки; используйте
`channel-outbound` для адаптеров сообщений, надёжных отправок, квитанций, предварительного просмотра в реальном
времени и параметров конвейера ответов.
