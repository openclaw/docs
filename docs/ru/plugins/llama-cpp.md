`llama-cpp` — официальный внешний Plugin-провайдер для локальных эмбеддингов GGUF. Он регистрирует идентификатор провайдера эмбеддингов `local` и управляет зависимостью среды выполнения `node-llama-cpp`, используемой параметром `memorySearch.provider: "local"`.

Установите его перед использованием локальных эмбеддингов памяти:

```bash
openclaw plugins install @openclaw/llama-cpp-provider
```

Основной npm-пакет `openclaw` не включает `node-llama-cpp`. Хранение нативной зависимости в этом Plugin предотвращает удаление вручную установленной среды выполнения из каталога пакета OpenClaw при обычных обновлениях OpenClaw через npm.

## Конфигурация

Задайте для `memorySearch.provider` значение `local`:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "local",
        local: {
          modelPath: "hf:ggml-org/embeddinggemma-300m-qat-q8_0-GGUF/embeddinggemma-300m-qat-Q8_0.gguf",
        },
      },
    },
  },
}
```

По умолчанию `local.modelPath` использует указанный выше URI `hf:` (`embeddinggemma-300m-qat-Q8_0.gguf`). Чтобы использовать другую модель, укажите другой URI `hf:` или локальный файл `.gguf`. Параметр `local.modelCacheDir` переопределяет каталог кэширования загруженных моделей (по умолчанию: `~/.node-llama-cpp/models`), а `local.contextSize` принимает целое число или `"auto"`.

Если `local.contextSize` задан числом, провайдер также передаёт это требование механизму автоматического распределения слоёв по GPU в node-llama-cpp. Это позволяет node-llama-cpp совместно разместить модель и контекст эмбеддингов, сохраняя проверки безопасности использования памяти. При значении `"auto"` node-llama-cpp использует обычное автоматическое распределение.

## Нативная среда выполнения

Для наиболее беспроблемной нативной установки используйте Node 24. В исходных копиях, использующих pnpm, может потребоваться разрешить и заново собрать нативную зависимость:

```bash
pnpm approve-builds
pnpm rebuild node-llama-cpp
```

## Диагностика среды выполнения

После загрузки провайдера выполните `openclaw memory status --deep`, чтобы просмотреть выбранный бэкенд и сборку, имена устройств, количество слоёв, выгруженных на GPU, запрошенный размер контекста, а также последний зафиксированный снимок VRAM или объединённой памяти. Значения VRAM содержат временную метку наблюдения, поскольку пассивное чтение состояния не перезагружает модель и не опрашивает устройство.

Те же последние известные данные могут отображаться в `openclaw doctor`, если работающий Gateway уже использовал локальный провайдер. Обычная команда просмотра состояния или диагностики не загружает модель только ради сбора диагностических данных.

## Устранение неполадок

Если `node-llama-cpp` отсутствует или не загружается, OpenClaw сообщает об ошибке и предлагает:

1. Установить Plugin: `openclaw plugins install @openclaw/llama-cpp-provider`.
2. Использовать Node 24 для нативных установок и обновлений.
3. В исходной копии с pnpm выполнить `pnpm approve-builds`, а затем `pnpm rebuild node-llama-cpp`.

Чтобы использовать локальные эмбеддинги без сложностей, связанных с этапом нативной сборки, вместо этого задайте для `memorySearch.provider` удалённый провайдер эмбеддингов, например `lmstudio`, `ollama`, `openai` или `voyage`.
