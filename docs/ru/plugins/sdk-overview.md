Plugin SDK — это типизированный контракт между плагинами и ядром. На этой странице
приведена справочная информация о том, **что импортировать** и **что можно регистрировать**.

<Note>
  Эта страница предназначена для авторов плагинов, использующих `openclaw/plugin-sdk/*`
  внутри OpenClaw. Для внешних приложений, скриптов, панелей мониторинга, заданий CI
  и расширений IDE, которым требуется запускать агентов через Gateway, используйте
  вместо этого раздел [Интеграции Gateway для внешних приложений](/ru/gateway/external-apps).
</Note>

<Tip>
Ищете практическое руководство? Начните с раздела [Создание плагинов](/ru/plugins/building-plugins). Для каналов используйте [Плагины каналов](/ru/plugins/sdk-channel-plugins), для поставщиков моделей — [Плагины поставщиков](/ru/plugins/sdk-provider-plugins), для локальных серверных компонентов CLI ИИ — [Плагины серверных компонентов CLI](/ru/plugins/cli-backend-plugins), для нативных исполнителей агентов — [Плагины среды выполнения агентов](/ru/plugins/sdk-agent-harness), а для хуков инструментов или жизненного цикла — [Хуки плагинов](/ru/plugins/hooks).
</Tip>

## Соглашение об импорте

Всегда импортируйте из конкретного подпути:

```typescript
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { defineChannelPluginEntry } from "openclaw/plugin-sdk/channel-core";
```

Каждый подпуть представляет собой небольшой автономный модуль. Это ускоряет запуск и
предотвращает проблемы с циклическими зависимостями. Для вспомогательных функций
точки входа и сборки, относящихся к каналам, предпочитайте
`openclaw/plugin-sdk/channel-core`; используйте `openclaw/plugin-sdk/core` для
более широкой объединяющей поверхности и общих вспомогательных функций, таких как
`buildChannelConfigSchema`.

Для конфигурации канала публикуйте принадлежащую каналу схему JSON через
`openclaw.plugin.json#channelConfigs`. Подпуть `plugin-sdk/channel-config-schema`
предназначен для общих примитивов схемы и универсального построителя. Встроенные
плагины OpenClaw используют `plugin-sdk/bundled-channel-config-schema` для
сохраняемых схем встроенных каналов. Устаревшие экспорты совместимости остаются в
`plugin-sdk/channel-config-schema-legacy`; ни один из подпутей встроенных схем не
следует использовать как образец для новых плагинов.

<Warning>
  Не импортируйте вспомогательные интерфейсы, привязанные к конкретному поставщику
  или каналу (например, `openclaw/plugin-sdk/slack`, `.../discord`, `.../signal`,
  `.../whatsapp`). Встроенные плагины объединяют универсальные подпути SDK внутри
  собственных файлов-агрегаторов `api.ts` / `runtime-api.ts`; потребителям ядра
  следует либо использовать эти локальные для плагина файлы-агрегаторы, либо
  добавлять узкий универсальный контракт SDK, если потребность действительно
  распространяется на несколько каналов.

Небольшой набор вспомогательных интерфейсов встроенных плагинов по-прежнему
присутствует в создаваемой карте экспортов, если зафиксировано их использование
владельцем. Они существуют только для сопровождения встроенных плагинов и не
рекомендуются как пути импорта для новых сторонних плагинов.

`openclaw/plugin-sdk/discord` и `openclaw/plugin-sdk/telegram-account` также
сохраняются как устаревшие фасады совместимости для зафиксированного использования
владельцем. Не копируйте эти пути импорта в новые плагины; вместо этого используйте
внедряемые вспомогательные функции среды выполнения и универсальные подпути SDK
каналов.
</Warning>

## Справочник подпутей

Plugin SDK предоставляется в виде набора узких подпутей, сгруппированных по областям
(точка входа плагина, канал, поставщик, аутентификация, среда выполнения, возможности,
память и зарезервированные вспомогательные функции встроенных плагинов). Полный
каталог с группировкой и ссылками см. в разделе
[Подпути Plugin SDK](/ru/plugins/sdk-subpaths).

Перечень точек входа компилятора находится в
`scripts/lib/plugin-sdk-entrypoints.json`; экспорты пакета создаются из
общедоступного подмножества после исключения локальных для репозитория тестовых и
внутренних подпутей, перечисленных в
`scripts/lib/plugin-sdk-private-local-only-subpaths.json`. Выполните
`pnpm plugin-sdk:surface`, чтобы проверить количество общедоступных экспортов.
Устаревшие общедоступные подпути, которые существуют достаточно давно и не
используются в рабочем коде встроенных расширений, отслеживаются в
`scripts/lib/plugin-sdk-deprecated-public-subpaths.json`; широкие устаревшие
файлы-агрегаторы повторных экспортов отслеживаются в
`scripts/lib/plugin-sdk-deprecated-barrel-subpaths.json`.

## API регистрации

Функция обратного вызова `register(api)` получает объект `OpenClawPluginApi`
со следующими методами:

### Регистрация возможностей

| Метод                                            | Что он регистрирует                                                                                   |
| ------------------------------------------------ | ----------------------------------------------------------------------------------------------------- |
| `api.registerProvider(...)`                      | Генерация текста (LLM)                                                                                |
| `api.registerWorkerProvider(...)`                | Аренды жизненного цикла облачных рабочих процессов                                                    |
| `api.registerModelCatalogProvider(...)`          | Строки каталога моделей для генерации текста и медиаданных                                             |
| `api.registerAgentHarness(...)`                  | [Экспериментальный](/ru/plugins/sdk-agent-harness) нативный исполнитель агентов (Codex, Copilot)          |
| `api.registerCliBackend(...)`                    | Локальный серверный компонент CLI для генерации                                                       |
| `api.registerChannel(...)`                       | Канал обмена сообщениями                                                                              |
| `api.registerEmbeddingProvider(...)`             | Повторно используемый поставщик векторных представлений                                                |
| `api.registerSpeechProvider(...)`                | Синтез речи из текста / STT                                                                           |
| `api.registerRealtimeTranscriptionProvider(...)` | Потоковая расшифровка в реальном времени                                                               |
| `api.registerRealtimeVoiceProvider(...)`         | Дуплексные голосовые сеансы в реальном времени                                                         |
| `api.registerMediaUnderstandingProvider(...)`    | Анализ изображений, аудио и видео                                                                      |
| `api.registerTranscriptSourceProvider(...)`      | Источник прямых или импортированных расшифровок встреч                                                 |
| `api.registerImageGenerationProvider(...)`       | Генерация изображений                                                                                  |
| `api.registerMusicGenerationProvider(...)`       | Генерация музыки                                                                                       |
| `api.registerVideoGenerationProvider(...)`       | Генерация видео                                                                                        |
| `api.registerWebFetchProvider(...)`              | Поставщик получения веб-страниц / извлечения данных                                                    |
| `api.registerWebSearchProvider(...)`             | Поиск в интернете                                                                                      |
| `api.registerCompactionProvider(...)`            | Подключаемый серверный компонент Compaction расшифровок                                                |

Поставщики рабочих процессов также должны объявлять свой идентификатор в `contracts.workerProviders`.
Ядро сохраняет устойчивое намерение до вызова `provision(profile, operationId)`. Поставщики проверяют настройки до выделения внешних ресурсов и создают исключение `WorkerProviderError`, если профиль отклонён окончательно. При повторении идентификатора операции `provision` должен использовать ту же аренду.
Ядро сохраняет проверенные настройки профиля вместе с арендой и передаёт этот снимок в `destroy({ leaseId, profile })`, который должен быть идемпотентным, и в `inspect({ leaseId, profile })`, который возвращает `active`, `destroyed` или `unknown`. Это позволяет поставщикам направлять вызовы жизненного цикла после перезапуска Gateway или удаления именованного профиля. Конечные точки SSH используют `SecretRef` для `keyRef`, никогда не содержат ключевой материал непосредственно и включают `hostKey` из доверенного результата подготовки ресурсов строго в формате `algorithm base64`, без имени хоста или комментария. Ядро закрепляет `hostKey` и никогда не доверяет ключу, полученному при первом подключении. Поставщик, создающий динамический `keyRef`, может реализовать `resolveSshIdentity({ leaseId, profile, keyRef })`; при наличии эта функция разрешения является авторитетным источником, а поставщики без неё используют настроенную универсальную функцию разрешения секретов.
Поставщики с возобновляемыми арендами также могут реализовать `renew(leaseId)`.
`inspect` должен создавать исключение при временных или неопределённых сбоях; возвращайте `unknown` только при достоверно установленном отсутствии. Ядро помечает активную локальную запись как потерявшую владельца или, если ранее был сохранён запрос на уничтожение, считает отсутствие завершением удаления ресурсов.

Поставщики векторных представлений, зарегистрированные с помощью
`api.registerEmbeddingProvider(...)`, также должны быть перечислены в
`contracts.embeddingProviders` в манифесте плагина. Это универсальный интерфейс
векторных представлений для повторно используемой генерации векторов. Поиск по памяти
может использовать этот универсальный интерфейс поставщика. Прежний интерфейс
`api.registerMemoryEmbeddingProvider(...)` и
`contracts.memoryEmbeddingProviders` является устаревшим интерфейсом совместимости
на время миграции существующих поставщиков, относящихся к памяти.

Поставщики, относящиеся к памяти и всё ещё предоставляющие во время выполнения
`batchEmbed(...)`, сохраняют существующий контракт пакетной обработки отдельных
файлов, если только их среда выполнения явно не задаёт
`sourceWideBatchEmbed: true`. Это явное включение позволяет узлу памяти передавать
фрагменты из нескольких изменённых файлов памяти и включённых источников за один
вызов `batchEmbed(...)` в пределах ограничений размера пакета узла. Пакетные адаптеры,
загружающие файлы запросов JSONL, должны разделять задания поставщика как до достижения
ограничения размера загружаемых данных, так и до достижения ограничения количества
запросов. Поставщик должен возвращать по одному векторному представлению для каждого
входного фрагмента в том же порядке, что и `batch.chunks`; не указывайте этот флаг,
если поставщик ожидает пакеты в пределах одного файла или не может сохранить порядок
входных данных в более крупном задании, охватывающем весь источник.

### Инструменты и команды

Используйте [`defineToolPlugin`](/ru/plugins/tool-plugins) для простых плагинов,
содержащих только инструменты с фиксированными именами. Используйте
`api.registerTool(...)` напрямую для смешанных плагинов или полностью динамической
регистрации инструментов.

| Метод                                  | Что он регистрирует                                                                                                                                                                   |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `api.registerTool(tool, opts?)`        | Инструмент агента (обязательный или `{ optional: true }`)                                                                                                                             |
| `api.registerCommand(def)`             | Пользовательская команда (в обход LLM)                                                                                                                                                |
| `api.registerNodeHostCommand(command)` | Команда, обрабатываемая `openclaw node run`; необязательные метаданные `agentTool` могут предоставлять её как видимый агенту инструмент, пока Node подключён |

Команды плагинов могут задавать `agentPromptGuidance`, когда агенту нужна короткая
подсказка по маршрутизации, принадлежащая команде. Этот текст должен относиться к
самой команде; не добавляйте в построители подсказок ядра правила, относящиеся к
конкретному поставщику или плагину.

Элементы указаний могут быть устаревшими строками, применяемыми ко всем поверхностям
подсказок, или структурированными элементами:

```ts
agentPromptGuidance: [
  "Global command hint.",
  { text: "Only show this in the main OpenClaw prompt.", surfaces: ["openclaw_main"] },
];
```

Структурированный список `surfaces` может включать `openclaw_main`,
`codex_app_server`, `cli_backend`, `acp_backend` или `subagent`. `pi_main`
остаётся устаревшим псевдонимом для `openclaw_main`. Не указывайте `surfaces`,
если указание намеренно предназначено для всех поверхностей. Не передавайте пустой
массив `surfaces`; он отклоняется, чтобы случайная потеря области действия не
привела к превращению текста в глобальную подсказку.

Инструкции разработчика нативного сервера приложений Codex строже, чем на других
поверхностях подсказок: в эту полосу с более высоким приоритетом продвигаются только
указания, область действия которых явно ограничена `codex_app_server`. Устаревшие
строковые указания и структурированные указания без указанной области действия
остаются доступными поверхностям подсказок, не относящимся к Codex, для обеспечения
совместимости.

Команды хоста Node выполняются на подключённом хосте Node, а не внутри процесса Gateway. Если присутствует `agentTool`, Node публикует дескриптор после успешного подключения к Gateway; Gateway предоставляет его запускам агента, только пока этот Node подключён и только если `command` дескриптора входит в одобренный набор команд Node. Укажите `agentTool.defaultPlatforms`, чтобы включить безопасную команду в стандартный список разрешённых команд Node; в противном случае требуется явное указание `gateway.nodes.allowCommands` или политика вызова Node. Значение `agentTool.name` должно быть безопасным для провайдера: начинаться с буквы, содержать только буквы, цифры, знаки подчёркивания или дефисы и не превышать 64 символа. Инструменты Node на базе MCP могут задавать метаданные `agentTool.mcp`, чтобы в каталоге и интерфейсах поиска инструментов отображались идентификаторы удалённого сервера и инструмента MCP, но выполнение всё равно происходит через объявленную команду Node.

### Инфраструктура

| Метод                                           | Что он регистрирует                                                                 |
| ----------------------------------------------- | ----------------------------------------------------------------------------------- |
| `api.registerHook(events, handler, opts?)`      | Обработчик событий                                                                  |
| `api.registerHttpRoute(params)`                 | Конечная точка HTTP Gateway                                                         |
| `api.registerGatewayMethod(name, handler)`      | Метод RPC Gateway                                                                   |
| `api.registerGatewayDiscoveryService(service)`  | Объявление локальной службы обнаружения Gateway                                     |
| `api.registerCli(registrar, opts?)`             | Подкоманда CLI                                                                      |
| `api.registerNodeCliFeature(registrar, opts?)`  | Функция CLI Node в разделе `openclaw nodes`                                         |
| `api.registerService(service)`                  | Фоновая служба                                                                      |
| `api.registerInteractiveHandler(registration)`  | Интерактивный обработчик                                                            |
| `api.registerAgentToolResultMiddleware(...)`    | Промежуточный обработчик результатов инструментов среды выполнения                  |
| `api.registerMemoryPromptSupplement(builder)`   | Дополнительный раздел запроса, связанный с памятью                                  |
| `api.registerMemoryCorpusSupplement(adapter)`   | Дополнительный корпус для поиска и чтения памяти                                    |
| `api.registerHostedMediaResolver(resolver)`     | Сопоставитель размещённых URL медиафайлов браузерного типа                          |
| `api.registerTextTransforms(transforms)`        | Преобразования текста запросов и сообщений для совместимости, принадлежащие Plugin  |
| `api.registerConfigMigration(migrate)`          | Упрощённая миграция конфигурации перед загрузкой среды выполнения Plugin            |
| `api.registerMigrationProvider(provider)`       | Импортёр для `openclaw migrate`                                                     |
| `api.registerAutoEnableProbe(probe)`            | Проверка конфигурации, способная автоматически включить этот Plugin                 |
| `api.registerReload(registration)`              | Политика префиксов конфигурации для перезапуска, горячей перезагрузки или бездействия |
| `api.registerNodeHostCommand(command)`          | Обработчик команд, доступный сопряжённым Node                                       |
| `api.registerNodeInvokePolicy(policy)`          | Политика списка разрешений и подтверждения для команд, вызываемых Node              |
| `api.registerSecurityAuditCollector(collector)` | Сборщик результатов для `openclaw security audit`                                   |

Построители дополнений к запросу памяти получают необязательный контекст `agentId`, `agentSessionKey` и `sandboxed`. Вызовы `search` и `get` дополнения к корпусу памяти получают необязательный контекст `agentId` и `sandboxed`. Plugins с хранилищем, принадлежащим агенту, должны определять это хранилище для каждого вызова, а не фиксировать один глобальный путь при регистрации. Если идентификатор агента обязателен, но отсутствует в операции с несколькими агентами, следует безопасно завершить операцию отказом, а не выбирать произвольного агента.

Интерактивные обработчики Telegram могут возвращать `{ submitText }`, чтобы после успешного завершения обработчика направить текст через обычный входящий путь агента Telegram. OpenClaw сохраняет кнопку обратного вызова, если входная политика пропускает текст или обработка завершается ошибкой, чтобы пользователь мог повторить попытку после устранения блокирующего условия. Это поле результата специфично для Telegram; другие каналы сохраняют собственные контракты интерактивных результатов.

### Хуки хоста для Plugins рабочих процессов

Хуки хоста — это точки расширения SDK для Plugins, которым требуется участвовать в жизненном цикле хоста, а не только добавлять провайдера, канал или инструмент. Это универсальные контракты; их может использовать режим планирования, а также рабочие процессы подтверждения, шлюзы политик рабочей области, фоновые мониторы, мастера настройки и сопутствующие Plugins пользовательского интерфейса.

| Метод                                                                                | Контракт, которым он управляет                                                                                                                                                                             |
| ------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `api.session.state.registerSessionExtension(...)`                                    | Принадлежащее Plugin совместимое с JSON состояние сеанса, проецируемое через сеансы Gateway                                                                                                                 |
| `api.session.workflow.enqueueNextTurnInjection(...)`                                 | Устойчивый контекст с семантикой строго однократной доставки, внедряемый в следующий ход агента для одного сеанса                                                                                           |
| `api.registerTrustedToolPolicy(...)`                                                 | Доверенная политика инструментов до запуска Plugin, доступная только через манифест и способная блокировать или изменять параметры инструмента                                                             |
| `api.registerToolMetadata(...)`                                                      | Метаданные отображения каталога инструментов без изменения реализации инструмента                                                                                                                          |
| `api.registerCommand(...)`                                                           | Команды Plugin с ограниченной областью действия; результаты команд могут задавать `continueAgent: true` или `suppressReply: true`; встроенные команды Discord поддерживают `descriptionLocalizations`         |
| `api.session.controls.registerControlUiDescriptor(...)`                              | Дескрипторы расширений пользовательского интерфейса управления для поверхностей сеанса, инструмента, запуска, настроек или вкладок                                                                          |
| `api.lifecycle.registerRuntimeLifecycle(...)`                                        | Обратные вызовы очистки ресурсов среды выполнения, принадлежащих Plugin, для путей сброса, удаления и перезагрузки                                                                                          |
| `api.agent.events.registerAgentEventSubscription(...)`                               | Очищенные подписки на события для состояния рабочего процесса и мониторов                                                                                                                                   |
| `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`  | Временное состояние Plugin для отдельного запуска, очищаемое при завершении жизненного цикла запуска                                                                                                       |
| `api.session.workflow.registerSessionSchedulerJob(...)`                              | Метаданные очистки для заданий планировщика, принадлежащих Plugin; не планирует работу и не создаёт записи задач                                                                                            |
| `api.session.workflow.sendSessionAttachment(...)`                                    | Доступная только встроенным Plugins доставка вложенного файла через хост в активный маршрут прямой исходящей отправки сеанса                                                                                |
| `api.session.workflow.scheduleSessionTurn(...)` / `unscheduleSessionTurnsByTag(...)` | Доступные только встроенным Plugins запланированные ходы сеанса на базе Cron и очистка по тегам                                                                                                             |
| `api.session.controls.registerSessionAction(...)`                                    | Типизированные действия сеанса, которые клиенты могут отправлять через Gateway                                                                                                                              |

Дескриптор `surface: "tab"` добавляет вкладку на боковую панель пользовательского интерфейса управления. Дескрипторы вкладок активных Plugins объявляются клиентам панели управления в приветственном сообщении Gateway (`controlUiTabs`), поэтому вкладка отображается, только пока Plugin включён. Встроенные Plugins могут предоставлять полноценное представление панели управления для своей вкладки; другие Plugins могут задать в `path` HTTP-маршрут Plugin (см. `api.registerHttpRoute(...)`), который панель управления отображает в изолированном фрейме. `icon` — рекомендуемое имя значка панели управления, `group` выбирает раздел боковой панели (`control` или `agent`), `order` определяет порядок среди вкладок Plugins, а `requiredScopes` скрывает вкладку от подключений без соответствующих областей полномочий оператора:

```typescript
api.session.controls.registerControlUiDescriptor({
  surface: "tab",
  id: "logbook",
  label: "Logbook",
  description: "Your day as a timeline, built from screen snapshots.",
  icon: "sun",
  group: "control",
  requiredScopes: ["operator.write"],
});
```

Для нового кода Plugin используйте сгруппированные пространства имён:

- `api.session.state.registerSessionExtension(...)`
- `api.session.workflow.enqueueNextTurnInjection(...)`
- `api.session.workflow.registerSessionSchedulerJob(...)`
- `api.session.workflow.sendSessionAttachment(...)`
- `api.session.workflow.scheduleSessionTurn(...)`
- `api.session.workflow.unscheduleSessionTurnsByTag(...)`
- `api.session.controls.registerSessionAction(...)`
- `api.session.controls.registerControlUiDescriptor(...)`
- `api.agent.events.registerAgentEventSubscription(...)`
- `api.agent.events.emitAgentEvent(...)`
- `api.runContext.setRunContext(...)` / `getRunContext(...)` / `clearRunContext(...)`
- `api.lifecycle.registerRuntimeLifecycle(...)`

Эквивалентные плоские методы остаются доступными как устаревшие псевдонимы совместимости для существующих Plugins. Не добавляйте новый код Plugin, который напрямую вызывает `api.registerSessionExtension`, `api.enqueueNextTurnInjection`, `api.registerControlUiDescriptor`, `api.registerRuntimeLifecycle`, `api.registerAgentEventSubscription`, `api.emitAgentEvent`, `api.setRunContext`, `api.getRunContext`, `api.clearRunContext`, `api.registerSessionSchedulerJob`, `api.registerSessionAction`, `api.sendSessionAttachment`, `api.scheduleSessionTurn` или `api.unscheduleSessionTurnsByTag`.

`scheduleSessionTurn(...)` — это привязанная к сеансу удобная обёртка над планировщиком Cron Gateway. Cron управляет временем и создаёт запись фоновой задачи при выполнении хода; SDK Plugin лишь ограничивает целевой сеанс, принадлежащие Plugin имена и очистку. Используйте `api.runtime.tasks.managedFlows` внутри запланированного хода, когда самой работе требуется устойчивое многошаговое состояние Task Flow.

Контракты намеренно разделяют полномочия:

- Внешние Plugins могут владеть расширениями сеансов, дескрипторами пользовательского интерфейса, командами, метаданными инструментов, внедрениями в следующий ход и обычными хуками.
- Доверенные политики инструментов выполняются до обычных хуков `before_tool_call` и пользуются доверием хоста. Встроенные политики выполняются первыми; политики установленных Plugins требуют явного включения и указания их локальных идентификаторов в `contracts.trustedToolPolicies`, после чего выполняются в порядке загрузки Plugins. Идентификаторы политик ограничены областью регистрирующего Plugin.
- Зарезервированные команды могут принадлежать только встроенным Plugins. Внешние Plugins должны использовать собственные имена команд или псевдонимы.
- `allowPromptInjection=false` отключает хуки, изменяющие запрос, включая `agent_turn_prepare`, `before_prompt_build`, `heartbeat_prompt_contribution`, поля запроса из устаревшего `before_agent_start` и `enqueueNextTurnInjection`.

Примеры потребителей вне режима планирования:

| Архетип Plugin                    | Используемые хуки                                                                                                                                                                |
| --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Рабочий процесс утверждения       | Расширение сеанса, продолжение команды, внедрение в следующий ход, дескриптор пользовательского интерфейса                                                                        |
| Шлюз политики бюджета/рабочей области | Политика доверенных инструментов, метаданные инструментов, проекция сеанса                                                                                                    |
| Фоновый монитор жизненного цикла  | Очистка жизненного цикла среды выполнения, подписка на события агента, владение планировщиком сеанса и его очистка, вклад в запрос Heartbeat, дескриптор пользовательского интерфейса |
| Мастер настройки или подключения | Расширение сеанса, команды с ограниченной областью действия, дескриптор Control UI                                                                                                |

<Note>
  Зарезервированные основные пространства имён администрирования (`config.*`, `exec.approvals.*`, `wizard.*`,
  `update.*`) всегда сохраняют область `operator.admin`, даже если Plugin пытается назначить
  методу Gateway более узкую область. Для методов, принадлежащих Plugin,
  предпочтительно использовать специфичные для него префиксы.
</Note>

<Accordion title="Когда использовать промежуточное ПО для результатов инструментов">
  Встроенные Plugins и явно включённые установленные Plugins с соответствующими
  контрактами манифеста могут использовать `api.registerAgentToolResultMiddleware(...)`,
  когда им требуется переписать результат инструмента после выполнения, но до того,
  как среда выполнения передаст этот результат обратно модели. Это доверенная,
  нейтральная к среде выполнения точка расширения для асинхронных обработчиков вывода,
  таких как tokenjuice.

Plugins должны объявлять `contracts.agentToolResultMiddleware` для каждой целевой
среды выполнения, например `["openclaw", "codex"]`. Установленные Plugins без этого
контракта или без явного включения не могут регистрировать это промежуточное ПО;
для работы, не требующей обработки результата инструмента до передачи модели,
используйте обычные хуки Plugin OpenClaw. Старый путь регистрации фабрики расширений,
предназначенный только для встроенного исполнителя, удалён.
</Accordion>

### Регистрация обнаружения Gateway

`api.registerGatewayDiscoveryService(...)` позволяет Plugin объявлять активный
Gateway в локальном транспорте обнаружения, например mDNS/Bonjour. OpenClaw вызывает
службу при запуске Gateway, когда локальное обнаружение включено, передаёт текущие
порты Gateway и несекретные данные подсказок TXT, а при завершении работы Gateway
вызывает возвращённый обработчик `stop`.

```typescript
api.registerGatewayDiscoveryService({
  id: "my-discovery",
  async advertise(ctx) {
    const handle = await startMyAdvertiser({
      gatewayPort: ctx.gatewayPort,
      tls: ctx.gatewayTlsEnabled,
      displayName: ctx.machineDisplayName,
    });
    return { stop: () => handle.stop() };
  },
});
```

Plugins обнаружения Gateway не должны считать объявляемые значения TXT секретами или
данными аутентификации. Обнаружение служит подсказкой для маршрутизации; за доверие
по-прежнему отвечают аутентификация Gateway и привязка TLS.

### Метаданные регистрации CLI

`api.registerCli(registrar, opts?)` принимает два вида метаданных команд:

- `commands`: явные имена команд, принадлежащих регистратору
- `descriptors`: дескрипторы команд времени разбора, используемые для справки CLI,
  маршрутизации и отложенной регистрации CLI Plugin
- `parentPath`: необязательный путь родительской команды для вложенных групп команд,
  например `["nodes"]`

Для функций сопряжённых узлов предпочтительно использовать
`api.registerNodeCliFeature(registrar, opts?)`. Это небольшая обёртка над
`api.registerCli(..., { parentPath: ["nodes"] })`, которая явно определяет такие
команды, как `openclaw nodes canvas`, в качестве принадлежащих Plugin функций узлов.

Если команда Plugin должна загружаться отложенно в обычном корневом пути CLI,
укажите `descriptors`, охватывающие каждый корень команды верхнего уровня,
предоставляемый этим регистратором.

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerMatrixCli } = await import("./src/cli.js");
    registerMatrixCli({ program });
  },
  {
    descriptors: [
      {
        name: "matrix",
        description: "Manage Matrix accounts, verification, devices, and profile state",
        hasSubcommands: true,
      },
    ],
  },
);
```

Вложенные команды получают разрешённую родительскую команду как `program`:

```typescript
api.registerCli(
  async ({ program }) => {
    const { registerNodesCanvasCommands } = await import("./src/cli.js");
    registerNodesCanvasCommands(program);
  },
  {
    parentPath: ["nodes"],
    descriptors: [
      {
        name: "canvas",
        description: "Capture or render canvas content from a paired node",
        hasSubcommands: true,
      },
    ],
  },
);
```

Используйте только `commands`, если отложенная регистрация корневого CLI не нужна.
Этот путь совместимости с немедленной загрузкой по-прежнему поддерживается, но он
не устанавливает заполнители на основе дескрипторов для отложенной загрузки во
время разбора.

### Регистрация серверной части CLI

`api.registerCliBackend(...)` позволяет Plugin владеть конфигурацией по умолчанию
для локальной серверной части CLI ИИ, например `claude-cli` или `my-cli`.

- `id` серверной части становится префиксом поставщика в ссылках на модели, таких как `my-cli/gpt-5`.
- `config` серверной части использует ту же структуру, что и `agents.defaults.cliBackends.<id>`.
- Пользовательская конфигурация по-прежнему имеет приоритет. Перед запуском CLI OpenClaw
  накладывает `agents.defaults.cliBackends.<id>` поверх значения Plugin по умолчанию.
- Используйте `normalizeConfig`, когда серверной части после слияния требуется
  преобразование для совместимости (например, нормализация старых форм флагов).
- Используйте `resolveExecutionArgs` для зависящего от запроса преобразования argv,
  относящегося к диалекту CLI, например для сопоставления уровней рассуждения OpenClaw
  с нативным флагом интенсивности. Хук получает `ctx.executionMode`; используйте
  `"side-question"`, чтобы добавлять нативные для серверной части флаги изоляции
  для временных вызовов `/btw`. Если эти флаги надёжно отключают нативные инструменты
  для CLI, где они иначе всегда включены, также объявите
  `sideQuestionToolMode: "disabled"`.
- Серверные части, способные отключать все нативные инструменты для конкретного запуска,
  могут объявить `nativeToolMode: "selectable"`. Ограниченные вызовы передают пустой
  кортеж `ctx.toolAvailability.native` вместе с точным списком разрешений MCP,
  изолированным хостом; `resolveExecutionArgs` должен обеспечить соблюдение обоих
  ограничений в итоговом argv нового или возобновлённого запуска.
  Если серверная часть не может этого сделать, OpenClaw завершает операцию с запретом.

Полное руководство по созданию см. в разделе
[Plugins серверной части CLI](/ru/plugins/cli-backend-plugins).

### Семантика решений хуков

`before_install` — это хук жизненного цикла среды выполнения Plugin, а не поверхность
политики установки оператора. Используйте `security.installPolicy`, когда решение
о разрешении или блокировке должно охватывать пути установки или обновления через
CLI и Gateway.

- `before_tool_call`: возврат `{ block: true }` завершает обработку. Как только любой обработчик устанавливает это значение, обработчики с более низким приоритетом пропускаются.
- `before_tool_call`: возврат `{ block: false }` считается отсутствием решения (как и отсутствие `block`), а не переопределением.
- `before_install`: возврат `{ block: true }` завершает обработку. Как только любой обработчик устанавливает это значение, обработчики с более низким приоритетом пропускаются.
- `before_install`: возврат `{ block: false }` считается отсутствием решения (как и отсутствие `block`), а не переопределением.
- `reply_dispatch`: возврат `{ handled: true, ... }` завершает обработку. Как только любой обработчик берёт диспетчеризацию на себя, обработчики с более низким приоритетом и стандартный путь диспетчеризации модели пропускаются.
- `message_sending`: возврат `{ cancel: true }` завершает обработку. Как только любой обработчик устанавливает это значение, обработчики с более низким приоритетом пропускаются.
- `message_sending`: возврат `{ cancel: false }` считается отсутствием решения (как и отсутствие `cancel`), а не переопределением.
- `message_received`: используйте типизированное поле `threadId`, когда требуется маршрутизация входящих сообщений по веткам или темам. Поле `metadata` оставляйте для дополнительных данных, специфичных для канала.
- `message_sending`: прежде чем прибегать к специфичному для канала полю `metadata`, используйте типизированные поля маршрутизации `replyToId` / `threadId`.
- `gateway_start`: для состояния запуска, принадлежащего Gateway, используйте `ctx.config`, `ctx.workspaceDir` и `ctx.getCron?.()` вместо внутренних хуков `gateway:startup`. На этом этапе Cron всё ещё может загружаться.
- `cron_reconciled`: перестраивайте полную внешнюю проекцию Cron после запуска или перезагрузки планировщика. Она включает `reason` и фактическое состояние `enabled`, в том числе `enabled: false`, а `ctx.getCron?.()` возвращает точный согласованный планировщик. Передавайте `ctx.abortSignal` в операции с сохраняемой проекцией; сигнал прерывается, когда этот снимок планировщика заменяется более новым или Gateway закрывается.
- `cron_changed`: отслеживайте изменения жизненного цикла Cron, принадлежащего Gateway. События `scheduled` и `removed` — это подсказки для согласования после фиксации, а не упорядоченный журнал изменений. У запланированного события поле `event.nextRunAtMs` отсутствует, если для задания не предусмотрено следующее пробуждение; событие удаления по-прежнему содержит снимок удалённого задания.

Внешним планировщикам пробуждений следует применять задержку или объединять события `cron_changed`,
а затем повторно считывать полное сохраняемое представление из планировщика, последним полученного через
`cron_reconciled`. Не перенимайте планировщик из контекста `cron_changed`: отдельная подсказка
от старого планировщика может пересечься с более поздней перезагрузкой.

Используйте `cron_reconciled` как триггер полного снимка для сохраняемого состояния, загружаемого при
запуске Gateway или замене планировщика. При горячей перезагрузке только плагина это событие повторно
не воспроизводится. Обработчики наблюдения выполняются параллельно, а диспетчеризации
без ожидания результата могут пересекаться, поэтому потребители не должны зависеть от порядка завершения событий.
OpenClaw должен оставаться источником истины для проверок срока выполнения и запуска заданий.

Пример адаптера с единственным параллельным выполнением, сохраняемой заменой, повторными попытками и задержкой,
а также корректным завершением работы см. в разделе [Безопасная внешняя проекция Cron](/ru/plugins/hooks#safe-external-cron-projection).

### Поля объекта API

| Поле                     | Тип                       | Описание                                                                                    |
| ------------------------ | ------------------------- | ------------------------------------------------------------------------------------------- |
| `api.id`                 | `string`                  | Идентификатор плагина                                                                       |
| `api.name`               | `string`                  | Отображаемое имя                                                                            |
| `api.version`            | `string?`                 | Версия плагина (необязательно)                                                              |
| `api.description`        | `string?`                 | Описание плагина (необязательно)                                                            |
| `api.source`             | `string`                  | Путь к исходному коду плагина                                                               |
| `api.rootDir`            | `string?`                 | Корневой каталог плагина (необязательно)                                                    |
| `api.config`             | `OpenClawConfig`          | Текущий снимок конфигурации (при наличии — активный снимок среды выполнения в памяти)        |
| `api.pluginConfig`       | `Record<string, unknown>` | Конфигурация плагина из `plugins.entries.<id>.config`                                        |
| `api.runtime`            | `PluginRuntime`           | [Вспомогательные средства среды выполнения](/ru/plugins/sdk-runtime)                            |
| `api.logger`             | `PluginLogger`            | Журнал с ограниченной областью (`debug`, `info`, `warn`, `error`)                            |
| `api.registrationMode`   | `PluginRegistrationMode`  | Текущий режим загрузки; `"setup-runtime"` — облегчённое окно запуска и настройки до загрузки полной точки входа |
| `api.resolvePath(input)` | `(string) => string`      | Разрешает путь относительно корня плагина                                                    |

## Соглашение о внутренних модулях

Внутри плагина используйте локальные файлы экспорта для внутренних импортов:

```text
my-plugin/
  api.ts            # Открытые экспорты для внешних потребителей
  runtime-api.ts    # Экспорты среды выполнения только для внутреннего использования
  index.ts          # Точка входа плагина
  setup-entry.ts    # Облегчённая точка входа только для настройки (необязательно)
```

<Warning>
  Никогда не импортируйте собственный плагин через `openclaw/plugin-sdk/<your-plugin>`
  в рабочем коде. Направляйте внутренние импорты через `./api.ts` или
  `./runtime-api.ts`. Путь SDK предназначен только для внешнего контракта.
</Warning>

Открытые поверхности встроенных плагинов, загружаемых через фасад (`api.ts`, `runtime-api.ts`,
`index.ts`, `setup-entry.ts` и аналогичные открытые файлы точек входа), предпочитают
активный снимок конфигурации среды выполнения, если OpenClaw уже запущен. Если снимка среды
выполнения ещё нет, они используют разрешённый файл конфигурации на диске.
Фасады упакованных встроенных плагинов следует загружать через фасадные загрузчики плагинов
OpenClaw; прямые импорты из `dist/extensions/...` обходят проверки манифеста
и вспомогательного состояния среды выполнения, используемые упакованными установками для кода, принадлежащего плагину.

Плагины провайдеров могут предоставлять узкую локальную точку контракта плагина, если
вспомогательное средство намеренно зависит от конкретного провайдера и пока не относится к универсальному
подпути SDK. Примеры встроенных плагинов:

- **Anthropic**: открытая граница `api.ts` / `contract-api.ts` для вспомогательных средств
  потоковой передачи Claude, связанных с заголовками бета-версии и `service_tier`.
- **`@openclaw/openai-provider`**: `api.ts` экспортирует конструкторы провайдера,
  вспомогательные средства модели по умолчанию и конструкторы провайдера реального времени.
- **`@openclaw/openrouter-provider`**: `api.ts` экспортирует конструктор провайдера
  и вспомогательные средства подключения и конфигурации.

<Warning>
  Рабочему коду расширения также следует избегать импортов `openclaw/plugin-sdk/<other-plugin>`.
  Если вспомогательное средство действительно общее, перенесите его в нейтральный подпуть SDK,
  например `openclaw/plugin-sdk/speech`, `.../provider-model-shared` или другую
  поверхность, ориентированную на возможности, вместо связывания двух плагинов.
</Warning>
