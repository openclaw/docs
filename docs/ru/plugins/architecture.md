## Общедоступная модель возможностей

Возможности — это общедоступная модель **нативных плагинов** в OpenClaw. Каждый нативный плагин OpenClaw регистрируется для одного или нескольких типов возможностей:

| Возможность                        | Метод регистрации                               | Примеры плагинов                         |
| ---------------------------------- | ------------------------------------------------ | ---------------------------------------- |
| Текстовый инференс                 | `api.registerProvider(...)`                      | `anthropic`, `openai`                    |
| Серверная часть инференса для CLI  | `api.registerCliBackend(...)`                    | `anthropic`, `openai`                    |
| Векторные представления            | `api.registerEmbeddingProvider(...)`             | Векторные плагины поставщиков            |
| Речь                               | `api.registerSpeechProvider(...)`                | `elevenlabs`, `microsoft`                |
| Транскрибирование в реальном времени | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                                 |
| Голосовая связь в реальном времени | `api.registerRealtimeVoiceProvider(...)`         | `google`, `openai`                       |
| Анализ медиаданных                 | `api.registerMediaUnderstandingProvider(...)`    | `google`, `openai`                       |
| Источник транскриптов              | `api.registerTranscriptSourceProvider(...)`      | `discord`                                |
| Генерация изображений              | `api.registerImageGenerationProvider(...)`       | `fal`, `google`, `openai`                |
| Генерация музыки                   | `api.registerMusicGenerationProvider(...)`       | `fal`, `google`, `minimax`               |
| Генерация видео                    | `api.registerVideoGenerationProvider(...)`       | `fal`, `google`, `qwen`                  |
| Получение веб-страниц              | `api.registerWebFetchProvider(...)`              | `firecrawl`                              |
| Веб-поиск                          | `api.registerWebSearchProvider(...)`             | `brave`, `firecrawl`, `google`           |
| Канал / обмен сообщениями          | `api.registerChannel(...)`                       | `matrix`, `msteams`                      |
| Обнаружение Gateway                | `api.registerGatewayDiscoveryService(...)`       | `bonjour`                                |

<Note>
Плагин, который не регистрирует ни одной возможности, но предоставляет перехватчики, инструменты, службы обнаружения или фоновые службы, считается **устаревшим плагином только с перехватчиками**. Такой шаблон по-прежнему полностью поддерживается.
</Note>

### Подход к внешней совместимости

Модель возможностей уже реализована в ядре и сейчас используется встроенными и нативными плагинами, но для совместимости внешних плагинов требуется более строгий критерий, чем «экспортировано — значит зафиксировано».

| Ситуация с плагином                                      | Рекомендация                                                                                                                     |
| -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Существующие внешние плагины                             | Сохраняйте работоспособность интеграций на основе перехватчиков: это базовый уровень совместимости.                               |
| Новые встроенные и нативные плагины                      | Предпочитайте явную регистрацию возможностей обращениям к внутренним компонентам конкретного поставщика и новым схемам только с перехватчиками. |
| Внешние плагины, переходящие на регистрацию возможностей | Допускается, но вспомогательные интерфейсы отдельных возможностей следует считать развивающимися, если документация не объявляет их стабильными. |

Регистрация возможностей — целевое направление развития. В переходный период устаревшие перехватчики остаются самым безопасным способом избежать поломок внешних плагинов. Экспортируемые вспомогательные подпути не равноценны — предпочитайте узкие документированные контракты случайным вспомогательным экспортам.

### Формы плагинов

OpenClaw классифицирует каждый загруженный плагин по форме на основе его фактического поведения при регистрации, а не только статических метаданных:

<AccordionGroup>
  <Accordion title="одна возможность">
    Регистрирует ровно один тип возможности (например, плагин только поставщика, такой как `arcee` или `chutes`).
  </Accordion>
  <Accordion title="несколько возможностей">
    Регистрирует несколько типов возможностей (например, `openai` отвечает за текстовый инференс, речь, анализ медиаданных и генерацию изображений).
  </Accordion>
  <Accordion title="только перехватчики">
    Регистрирует только перехватчики (типизированные или пользовательские), без возможностей, инструментов, команд и служб.
  </Accordion>
  <Accordion title="без возможностей">
    Регистрирует инструменты, команды, службы или маршруты, но не возможности.
  </Accordion>
</AccordionGroup>

Чтобы просмотреть форму плагина и разбивку по возможностям, используйте `openclaw plugins inspect <id>`. Подробности см. в [справочнике CLI](/ru/cli/plugins#inspect).

### Устаревшие перехватчики

Перехватчик `before_agent_start` по-прежнему поддерживается как путь совместимости для плагинов только с перехватчиками. От него всё ещё зависят реально используемые устаревшие плагины.

Направление развития:

- сохранять его работоспособность
- обозначать его в документации как устаревший
- для переопределения модели или поставщика предпочитать `before_model_resolve`
- для изменения подсказки предпочитать `before_prompt_build`
- удалять его только после снижения реального использования и подтверждения безопасности миграции тестовыми фикстурами

### Сигналы совместимости

`openclaw doctor`, `openclaw plugins inspect <id>`, `openclaw status --all` и `openclaw plugins doctor` выводят следующие уведомления о совместимости:

| Сигнал                                            | Значение                                                                                                                         |
| ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **конфигурация допустима**                         | Конфигурация успешно разбирается, а плагины разрешаются                                                                           |
| **только перехватчики** (информация)               | Плагин регистрирует только перехватчики; этот путь поддерживается, но ещё не перенесён на регистрацию возможностей                |
| **устаревший `before_agent_start`** (предупреждение) | Плагин использует устаревший перехватчик `before_agent_start` вместо `before_model_resolve`/`before_prompt_build`                  |
| **устаревший API векторных представлений памяти** (предупреждение) | Невстроенный плагин использует старый специализированный API поставщика векторных представлений памяти вместо `registerEmbeddingProvider` |
| **критическая ошибка**                             | Конфигурация недопустима или плагин не удалось загрузить                                                                           |

Сейчас ни информационные сигналы, ни предупреждения не нарушают работу вашего плагина. Эти сигналы также отображаются в `openclaw status --all` и `openclaw plugins doctor`.

## Обзор архитектуры

Система плагинов OpenClaw состоит из четырёх уровней:

<Steps>
  <Step title="Манифест и обнаружение">
    OpenClaw находит потенциальные плагины в настроенных путях, корневых каталогах рабочих пространств, глобальных корневых каталогах плагинов и среди встроенных плагинов. При обнаружении сначала считываются нативные манифесты `openclaw.plugin.json`, а также поддерживаемые манифесты пакетов.
  </Step>
  <Step title="Включение и проверка">
    Ядро определяет, включён ли обнаруженный плагин, отключён, заблокирован или выбран для эксклюзивной позиции, например памяти.
  </Step>
  <Step title="Загрузка среды выполнения">
    Нативные плагины OpenClaw загружаются внутри процесса и регистрируют возможности в центральном реестре. Упакованный JavaScript загружается через нативный `require`; локальный исходный код сторонних разработчиков на TypeScript в крайнем случае загружается через Jiti. Совместимые пакеты нормализуются в записи реестра без импорта кода среды выполнения.
  </Step>
  <Step title="Использование интерфейсов">
    Остальные компоненты OpenClaw считывают реестр, чтобы предоставлять инструменты, каналы, настройку поставщиков, перехватчики, маршруты HTTP, команды CLI и службы.
  </Step>
</Steps>

В частности, для CLI плагинов обнаружение корневых команд разделено на два этапа:

- метаданные на этапе разбора поступают из `registerCli(..., { descriptors: [...] })`
- фактический модуль CLI плагина может оставаться ленивым и регистрироваться при первом вызове

Благодаря этому принадлежащий плагину код CLI остаётся внутри плагина, а OpenClaw может зарезервировать имена корневых команд до разбора.

Важная архитектурная граница:

- проверка манифеста и конфигурации должна работать на основе **метаданных манифеста и схемы** без выполнения кода плагина
- при обнаружении нативных возможностей может загружаться доверенный входной код плагина для построения неактивирующего снимка реестра
- нативное поведение среды выполнения обеспечивается путём `register(api)` модуля плагина при `api.registrationMode === "full"`

Такое разделение позволяет OpenClaw проверять конфигурацию, объяснять отсутствие или отключение плагинов и формировать подсказки для интерфейса и схемы до активации полной среды выполнения.

### Снимок метаданных плагинов и таблица поиска

При запуске Gateway создаётся один `PluginMetadataSnapshot` для текущего снимка конфигурации. Снимок содержит только метаданные: индекс установленных плагинов, реестр манифестов, диагностические данные манифестов, карты владельцев, нормализатор идентификаторов плагинов и записи манифестов. Он не содержит загруженных модулей плагинов, SDK поставщиков, содержимого пакетов или экспортов среды выполнения.

Проверка конфигурации с учётом плагинов, автоматическое включение при запуске и начальная загрузка плагинов Gateway используют этот снимок вместо независимого повторного построения метаданных манифестов и индекса. `PluginLookUpTable` создаётся из того же снимка и добавляет план запуска плагинов для текущей конфигурации среды выполнения.

После запуска Gateway сохраняет текущий снимок метаданных как заменяемый продукт среды выполнения. Повторное обнаружение поставщиков во время работы может использовать этот снимок вместо повторного построения индекса установленных плагинов и реестра манифестов при каждом проходе по каталогу поставщиков. Снимок очищается или заменяется при завершении работы Gateway, изменениях конфигурации или перечня плагинов и записи индекса установленных плагинов; если совместимого текущего снимка нет, вызывающий код возвращается к холодному пути манифеста и индекса. Проверки совместимости должны учитывать корневые каталоги обнаружения плагинов, например `plugins.load.paths`, и рабочее пространство агента по умолчанию, поскольку плагины рабочего пространства входят в область метаданных.

Снимок и таблица поиска обеспечивают быстрый путь для повторяющихся решений при запуске:

- владение каналами
- отложенный запуск каналов
- идентификаторы запускаемых плагинов
- владение поставщиками и серверными частями CLI
- владение поставщиком настройки, псевдонимом команды, поставщиком каталога моделей и контрактом манифеста
- проверка схемы конфигурации плагинов и схемы конфигурации каналов
- решения об автоматическом включении при запуске

Границей безопасности служит замена снимка, а не его изменение. Перестраивайте снимок при изменении конфигурации, перечня плагинов, записей об установке или политики сохраняемого индекса. Не используйте его как обширный изменяемый глобальный реестр и не храните неограниченное количество исторических снимков. Загрузка плагинов среды выполнения остаётся отделённой от снимков метаданных, чтобы устаревшее состояние среды выполнения нельзя было скрыть за кешем метаданных.

Правило кеширования описано в разделе [Внутреннее устройство архитектуры плагинов](/ru/plugins/architecture-internals#plugin-cache-boundary): метаданные манифестов и обнаружения актуальны, если вызывающий код не хранит явный снимок, таблицу поиска или реестр манифестов для текущего потока. Скрытые кеши метаданных и сроки действия по настенным часам не входят в механизм загрузки плагинов. После фактической загрузки кода или установленных артефактов могут сохраняться только кеши загрузчика среды выполнения, модулей и артефактов зависимостей.

Некоторые вызывающие компоненты холодного пути всё ещё напрямую восстанавливают реестры манифестов из сохранённого индекса установленных плагинов вместо получения `PluginLookUpTable` от Gateway. Теперь этот путь восстанавливает реестр по требованию; если у вызывающего кода уже есть текущая таблица поиска или явный реестр манифестов, предпочтительно передавать их по потокам среды выполнения.

### Планирование активации

Планирование активации является частью плоскости управления. Вызывающая сторона может запросить, какие плагины относятся к конкретной команде, провайдеру, каналу, маршруту, среде выполнения агента или возможности, прежде чем загружать более широкие реестры среды выполнения.

Планировщик сохраняет совместимость с текущим поведением манифеста:

- поля `activation.*` являются явными подсказками для планировщика
- `providers`, `channels`, `commandAliases`, `setup.providers`, `contracts.tools` и хуки остаются резервным механизмом на основе владения, указанного в манифесте
- API планировщика, возвращающий только идентификаторы, остаётся доступным для существующих вызывающих сторон
- API плана возвращает метки причин, чтобы диагностика могла отличать явные подсказки от резервного механизма на основе владения

<Warning>
Не рассматривайте `activation` как хук жизненного цикла или замену `register(...)`. Это метаданные, используемые для ограничения области загрузки. Отдавайте предпочтение полям владения, если они уже описывают взаимосвязь; используйте `activation` только для дополнительных подсказок планировщику.
</Warning>

### Плагины каналов и общий инструмент сообщений

Плагинам каналов не нужно регистрировать отдельный инструмент отправки, редактирования или реакции для обычных действий в чате. OpenClaw сохраняет в ядре единый общий инструмент `message`, а плагины каналов отвечают за специфичные для каналов обнаружение и выполнение, стоящие за ним.

Текущая граница ответственности:

- ядро отвечает за хост общего инструмента `message`, подключение к промпту, учёт сеансов и веток, а также диспетчеризацию выполнения
- плагины каналов отвечают за обнаружение действий в заданной области, обнаружение возможностей и любые специфичные для канала фрагменты схемы
- плагины каналов отвечают за специфичную для провайдера грамматику диалогов сеанса, например за то, как идентификаторы диалогов кодируют идентификаторы веток или наследуются от родительских диалогов
- плагины каналов выполняют итоговое действие через свой адаптер действий

Для плагинов каналов поверхностью SDK является `ChannelMessageActionAdapter.describeMessageTool(...)`. Этот единый вызов обнаружения позволяет плагину вместе вернуть видимые действия, возможности и дополнения к схеме, чтобы эти части не расходились.

Когда специфичный для канала параметр инструмента сообщений содержит источник медиа, например локальный путь или удалённый URL медиафайла, плагин также должен возвращать `mediaSourceParams` из `describeMessageTool(...)`. Ядро использует этот явный список, чтобы применять нормализацию путей песочницы и подсказки для исходящего доступа к медиафайлам без жёстко заданных имён параметров, принадлежащих плагину. Здесь отдавайте предпочтение картам, ограниченным областью действия, а не единому плоскому списку для всего канала, чтобы параметр медиафайла, относящийся только к профилю, не нормализовался для несвязанных действий, таких как `send`.

Ядро передаёт область среды выполнения на этот этап обнаружения. Важные поля:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- доверенный входящий `requesterSenderId`

Это важно для контекстно-зависимых плагинов. Канал может скрывать или показывать действия с сообщениями в зависимости от активной учётной записи, текущей комнаты, ветки или сообщения либо доверенной личности инициатора запроса, не задавая в ядре жёстко специфичные для канала ветви инструмента `message`.

Именно поэтому изменения маршрутизации встроенного средства выполнения по-прежнему относятся к работе над плагином: средство выполнения отвечает за передачу идентичности текущего чата и сеанса на границу обнаружения плагина, чтобы общий инструмент `message` предоставлял для текущего хода правильную поверхность, принадлежащую каналу.

Для вспомогательных средств выполнения, принадлежащих каналам, встроенные плагины должны хранить среду выполнения внутри собственных модулей плагина. Ядро больше не отвечает за среды выполнения действий с сообщениями Discord, Slack, Telegram или WhatsApp в `src/agents/tools`. Мы не публикуем отдельные подпути `plugin-sdk/*-action-runtime`, а встроенные плагины должны импортировать собственный локальный код среды выполнения непосредственно из принадлежащих им модулей.

Та же граница применяется в целом к именованным по провайдеру интерфейсам SDK: ядро не должно импортировать специфичные для канала вспомогательные точки экспорта для Discord, Signal, Slack, WhatsApp или аналогичных плагинов. Если ядру требуется какое-либо поведение, оно должно либо использовать собственную точку экспорта `api.ts` / `runtime-api.ts` встроенного плагина, либо оформить эту потребность как узкую универсальную возможность в общем SDK.

Встроенные плагины следуют тому же правилу. `runtime-api.ts` встроенного плагина не должен повторно экспортировать собственный брендированный фасад `openclaw/plugin-sdk/<plugin-id>`. Такие брендированные фасады остаются прослойками совместимости для внешних плагинов и прежних потребителей, однако встроенные плагины должны использовать локальные экспорты и узкие универсальные подпути SDK, такие как `openclaw/plugin-sdk/channel-policy`, `openclaw/plugin-sdk/runtime-store` или `openclaw/plugin-sdk/webhook-ingress`. Новый код не должен добавлять специфичные для идентификатора плагина фасады SDK, если этого не требует граница совместимости существующей внешней экосистемы.

Для опросов предусмотрены два пути выполнения:

- `outbound.sendPoll` — общая основа для каналов, соответствующих универсальной модели опросов
- `actions.handleAction("poll")` — предпочтительный путь для специфичной для канала семантики опросов или дополнительных параметров опроса

Теперь ядро откладывает общий разбор опроса до тех пор, пока диспетчеризация опроса в плагин не отклонит действие, поэтому принадлежащие плагинам обработчики опросов могут принимать специфичные для канала поля опроса, не блокируясь предварительно универсальным анализатором опросов.

Полная последовательность запуска описана в разделе [Внутреннее устройство архитектуры плагинов](/ru/plugins/architecture-internals).

## Модель владения возможностями

OpenClaw рассматривает нативный плагин как границу владения **компанией** или **функцией**, а не как набор несвязанных интеграций.

Это означает:

- плагин компании обычно должен владеть всеми поверхностями этой компании, обращёнными к OpenClaw
- плагин функции обычно должен владеть всей поверхностью вводимой им функции
- каналы должны использовать общие возможности ядра вместо ситуативной повторной реализации поведения провайдера

<AccordionGroup>
  <Accordion title="Провайдер с несколькими возможностями">
    `google` отвечает за генерацию текста, серверную часть CLI, эмбеддинги, речь, голос в реальном времени, понимание медиа, генерацию изображений, музыки и видео, а также веб-поиск. `openai` отвечает за генерацию текста, эмбеддинги, речь, транскрибирование в реальном времени, голос в реальном времени, понимание медиа и генерацию изображений и видео. `minimax` отвечает за генерацию текста, а также за понимание медиа, речь, генерацию изображений, музыки и видео и веб-поиск.
  </Accordion>
  <Accordion title="Провайдер с одной возможностью">
    `arcee` и `chutes` отвечают только за генерацию текста; `microsoft` отвечает только за речь. Плагин провайдера может оставаться настолько узким, пока ему не потребуется охватить бо́льшую часть поверхности этого провайдера.
  </Accordion>
  <Accordion title="Плагин функции">
    `voice-call` отвечает за транспорт звонков, инструменты, CLI, маршруты и связку с медиапотоками Twilio, но использует общие возможности речи, транскрибирования в реальном времени и голоса в реальном времени вместо прямого импорта плагинов провайдеров.
  </Accordion>
</AccordionGroup>

Целевое конечное состояние:

- поверхность провайдера, обращённая к OpenClaw, находится в одном плагине, даже если она охватывает текстовые модели, речь, изображения и видео
- другие провайдеры могут поступать так же со своей областью ответственности
- каналам неважно, какой плагин провайдера владеет провайдером; они используют общий контракт возможности, предоставляемый ядром

Ключевое различие:

- **плагин** = граница владения
- **возможность** = контракт ядра, который могут реализовывать или использовать несколько плагинов

Поэтому, если OpenClaw добавляет новую область, например видео, первый вопрос должен звучать не «какой провайдер должен содержать жёстко заданную обработку видео?», а «каким должен быть контракт возможности видео в ядре?». После появления этого контракта плагины провайдеров смогут регистрировать свои реализации, а плагины каналов и функций — использовать его.

Если возможность ещё не существует, обычно следует:

<Steps>
  <Step title="Определить возможность">
    Определить недостающую возможность в ядре.
  </Step>
  <Step title="Предоставить через SDK">
    Предоставить её через API и среду выполнения плагинов с типизацией.
  </Step>
  <Step title="Подключить потребителей">
    Подключить каналы и функции к этой возможности.
  </Step>
  <Step title="Реализации провайдеров">
    Позволить плагинам провайдеров регистрировать реализации.
  </Step>
</Steps>

Так владение остаётся явным, а в ядре не появляется поведение, зависящее от единственного провайдера или разового специфичного для плагина пути кода.

### Уровни возможностей

При выборе места для кода используйте следующую ментальную модель:

<Tabs>
  <Tab title="Уровень возможностей ядра">
    Общая оркестрация, политика, резервное поведение, правила объединения конфигурации, семантика доставки и типизированные контракты.
  </Tab>
  <Tab title="Уровень плагина провайдера">
    Специфичные для провайдера API, аутентификация, каталоги моделей, синтез речи, генерация изображений, серверные части видео и конечные точки использования.
  </Tab>
  <Tab title="Уровень плагина канала или функции">
    Интеграция Discord, Slack, `voice-call` и других компонентов, которая использует возможности ядра и предоставляет их на своей поверхности.
  </Tab>
</Tabs>

Например, TTS следует этой структуре:

- ядро отвечает за политику TTS при ответе, порядок резервных вариантов, настройки и доставку в канал
- `elevenlabs`, `google`, `microsoft` и `openai` отвечают за реализации синтеза
- `voice-call` использует вспомогательное средство среды выполнения TTS для телефонии

Для будущих возможностей следует отдавать предпочтение тому же шаблону.

### Пример плагина компании с несколькими возможностями

Плагин компании должен восприниматься извне как единое целое. Если OpenClaw предоставляет общие контракты для моделей, речи, транскрибирования в реальном времени, голоса в реальном времени, понимания медиа, генерации изображений, генерации видео, веб-запросов и веб-поиска, провайдер может владеть всеми своими поверхностями в одном месте:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";
import { createPluginBackedWebSearchProvider } from "openclaw/plugin-sdk/provider-web-search";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          ...req,
          provider: "exampleai",
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          ...req,
          provider: "exampleai",
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

Важны не точные имена вспомогательных функций, а структура:

- один плагин владеет поверхностью провайдера
- ядро по-прежнему владеет контрактами возможностей
- каналы и плагины функций используют вспомогательные средства `api.runtime.*`, а не код провайдера
- тесты контрактов могут проверять, что плагин зарегистрировал возможности, которыми он заявляет владение

### Пример возможности: понимание видео

OpenClaw уже рассматривает понимание изображений, аудио и видео как одну общую возможность. Здесь применяется та же модель владения:

<Steps>
  <Step title="Ядро определяет контракт">
    Ядро определяет контракт понимания медиа.
  </Step>
  <Step title="Плагины провайдеров регистрируют реализации">
    Плагины провайдеров регистрируют `describeImage`, `transcribeAudio` и, где применимо, `describeVideo`.
  </Step>
  <Step title="Потребители используют общее поведение">
    Каналы и плагины функций используют общее поведение ядра вместо прямого подключения к коду провайдера.
  </Step>
</Steps>

Это предотвращает встраивание предположений одного провайдера о видео в ядро. Плагин владеет поверхностью провайдера; ядро владеет контрактом возможности и резервным поведением.

Генерация видео уже использует ту же последовательность: ядро владеет типизированным контрактом возможности и вспомогательным средством среды выполнения, а плагины провайдеров регистрируют для него реализации `api.registerVideoGenerationProvider(...)`.

Нужен конкретный контрольный список внедрения? См. [Руководство по возможностям](/ru/plugins/adding-capabilities).

## Контракты и контроль соблюдения

Поверхность API плагинов намеренно типизирована и централизована в `OpenClawPluginApi`. Этот контракт определяет поддерживаемые точки регистрации и вспомогательные средства среды выполнения, на которые может полагаться плагин.

Почему это важно:

- авторы плагинов получают единый стабильный внутренний стандарт
- ядро может отклонять дублирование владения, например когда два плагина регистрируют один и тот же идентификатор провайдера
- при запуске могут выводиться полезные для устранения проблем диагностические сообщения о некорректной регистрации
- контрактные тесты могут обеспечивать соблюдение границ владения встроенных плагинов и предотвращать незаметное расхождение

Предусмотрено два уровня контроля:

<AccordionGroup>
  <Accordion title="Контроль регистрации во время выполнения">
    Реестр плагинов проверяет регистрации по мере загрузки плагинов. Например, дублирующиеся идентификаторы провайдеров, дублирующиеся идентификаторы провайдеров синтеза речи и некорректные регистрации создают диагностические сообщения плагина вместо неопределённого поведения.
  </Accordion>
  <Accordion title="Контрактные тесты">
    Во время выполнения тестов встроенные плагины фиксируются в контрактных реестрах, чтобы OpenClaw мог явно проверять владение. Сейчас это используется для провайдеров моделей, провайдеров синтеза речи, провайдеров веб-поиска и владения регистрациями встроенных плагинов.
  </Accordion>
</AccordionGroup>

На практике это означает, что OpenClaw заранее знает, какой плагин владеет каждой поверхностью. Благодаря этому ядро и каналы могут бесшовно взаимодействовать, поскольку владение объявлено, типизировано и доступно для тестирования, а не задано неявно.

### Что должно входить в контракт

<Tabs>
  <Tab title="Хорошие контракты">
    - типизированы
    - компактны
    - относятся к конкретной возможности
    - принадлежат ядру
    - могут повторно использоваться несколькими плагинами
    - могут использоваться каналами и функциями без знаний о конкретном поставщике

  </Tab>
  <Tab title="Плохие контракты">
    - политика конкретного поставщика, скрытая в ядре
    - одноразовые обходные механизмы плагинов, минующие реестр
    - прямой доступ кода канала к реализации конкретного поставщика
    - специализированные объекты среды выполнения, не являющиеся частью `OpenClawPluginApi` или `api.runtime`

  </Tab>
</Tabs>

В случае сомнений повысьте уровень абстракции: сначала определите возможность, а затем позвольте плагинам подключаться к ней.

## Модель выполнения

Нативные плагины OpenClaw выполняются **в процессе** вместе с Gateway. Они не изолированы в песочнице. Загруженный нативный плагин находится в тех же границах доверия на уровне процесса, что и код ядра.

<Warning>
Следствия использования нативных плагинов: плагин может регистрировать инструменты, сетевые обработчики, перехватчики и службы; ошибка в плагине может вызвать сбой или нарушить стабильность Gateway; вредоносный нативный плагин эквивалентен выполнению произвольного кода внутри процесса OpenClaw.
</Warning>

Совместимые комплекты по умолчанию безопаснее, поскольку сейчас OpenClaw рассматривает их как пакеты метаданных и содержимого. В текущих выпусках это преимущественно означает встроенные Skills.

Для невстроенных плагинов используйте списки разрешённых элементов и явные пути установки и загрузки. Рассматривайте плагины рабочего пространства как код для разработки, а не как настройки по умолчанию для рабочей среды.

Для имён встроенных пакетов рабочего пространства сохраняйте привязку идентификатора плагина к имени npm: по умолчанию `@openclaw/<id>` либо утверждённый типизированный суффикс, например `-provider`, `-plugin`, `-speech`, `-sandbox` или `-media-understanding`, если пакет намеренно предоставляет более узкую роль плагина.

<Note>
**Примечание о доверии:** `plugins.allow` доверяет **идентификаторам плагинов**, а не происхождению исходного кода. Плагин рабочего пространства с тем же идентификатором, что и у встроенного плагина, намеренно замещает встроенную копию, если этот плагин рабочего пространства включён или добавлен в список разрешённых. Это нормально и полезно для локальной разработки, тестирования исправлений и срочных исправлений. Доверие к встроенному плагину определяется по снимку исходного кода — манифесту и коду на диске во время загрузки, — а не по метаданным установки. Повреждённая или подменённая запись об установке не может незаметно расширить поверхность доверия встроенного плагина сверх того, что заявлено в фактическом исходном коде.
</Note>

## Граница экспорта

OpenClaw экспортирует возможности, а не удобные детали реализации.

Оставляйте регистрацию возможностей общедоступной. Удаляйте экспорты вспомогательных средств, не входящих в контракт:

- вспомогательные подпути, относящиеся к конкретным встроенным плагинам
- подпути инфраструктуры среды выполнения, не предназначенные для использования в качестве общедоступного API
- вспомогательные средства для удобства, относящиеся к конкретному поставщику
- вспомогательные средства настройки и первоначальной настройки, являющиеся деталями реализации

Зарезервированные вспомогательные подпути встроенных плагинов удалены из создаваемой карты экспорта SDK. Храните вспомогательные средства, относящиеся к конкретному владельцу, внутри пакета соответствующего плагина; переносите в общие контракты SDK только многократно используемое поведение основной системы, например `plugin-sdk/gateway-runtime`, `plugin-sdk/security-runtime` и `plugin-sdk/plugin-config-runtime`.
